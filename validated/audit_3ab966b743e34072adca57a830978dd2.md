### Title
ERC-20 Tokens Permanently Burned Without Refund When `ft_transfer` Fails in `ExitToNear` Precompile — (File: `engine-precompiles/src/native.rs`)

---

### Summary

When the `error_refund` compile-time feature is not enabled (the opt-in default), the `ExitToNear` precompile burns ERC-20 tokens from the EVM and creates an `ft_transfer` promise to the NEP-141 contract. If that promise fails, no callback is attached and no refund is issued. The ERC-20 tokens are permanently destroyed while the NEP-141 tokens remain locked inside Aurora's account, inaccessible to the user.

---

### Finding Description

**Step 1 — Callback args construction with `refund: None` by default**

In `engine-precompiles/src/native.rs`, the `ExitToNear::run()` method constructs the callback arguments. The `refund` field is only populated when the `error_refund` feature flag is compiled in; otherwise it is hardcoded to `None`: [1](#0-0) 

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,
    transfer_near: transfer_near_args,
};
```

**Step 2 — No callback attached when `refund` and `transfer_near` are both `None`**

Immediately after, the code decides whether to attach a callback at all. When `callback_args` equals the default (`refund: None, transfer_near: None`), the promise is created as a bare `Create` with **no callback**: [2](#0-1) 

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // ← no callback attached
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs { ... })
};
```

For a standard ERC-20 exit (not wNEAR unwrap), `transfer_near` is `None`. Without `error_refund`, both fields are `None`, so the equality holds and **no callback is ever scheduled**.

**Step 3 — Even when a callback is attached, failure is silently swallowed**

For the wNEAR-unwrap path (`transfer_near: Some(...)`), a callback *is* attached. But inside `exit_to_near_precompile_callback`, when the base promise fails and `args.refund` is `None`, the function silently returns `Ok(None)` — no refund, no error: [3](#0-2) 

```rust
let maybe_result = if let Some(PromiseResult::Successful(_)) = handler.promise_result(0) {
    ...
    None
} else if let Some(args) = args.refund {
    // refund path — only reachable with `error_refund` feature
    let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;
    ...
} else {
    None   // ← silent no-op on failure without `error_refund`
};
```

**Step 4 — The ERC-20 burn is irreversible**

The ERC-20 burn happens inside the EVM execution *before* the NEAR promise is dispatched. By the time the `ft_transfer` promise fails, the ERC-20 balance is already gone. The `refund_on_error` function (which would re-mint the tokens) is never called: [4](#0-3) 

**Step 5 — The test suite explicitly acknowledges the permanent loss**

The workspace test `test_exit_to_near_refund` documents that without `error_refund`, the ERC-20 balance is permanently reduced: [5](#0-4) 

```rust
#[cfg(feature = "error_refund")]
let balance = FT_TRANSFER_AMOUNT.into();
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
```

---

### Impact Explanation

- ERC-20 tokens are burned from the EVM — this is irreversible.
- The NEP-141 `ft_transfer` fails (e.g., recipient not registered with the NEP-141 contract).
- Without `error_refund`, no callback is attached (or the callback is a no-op on failure).
- The NEP-141 tokens remain locked inside Aurora's account with no user-accessible recovery path.
- The user permanently loses both their ERC-20 balance and the corresponding NEP-141 value.

**Impact: Critical — Permanent freezing (and effective destruction) of user funds.**

---

### Likelihood Explanation

- The `error_refund` feature is opt-in; it is **not** a default Cargo feature, so any build that does not explicitly enable it is vulnerable.
- The trigger condition is straightforward: call `ExitToNear` with a NEAR recipient account that is not registered with the NEP-141 token contract. This is a common user mistake (e.g., sending to a fresh account that has never interacted with the token).
- No privileged access is required. Any EVM user holding ERC-20 tokens can trigger this path.

---

### Recommendation

Remove the `#[cfg(not(feature = "error_refund"))] refund: None` branch and always populate `refund` with the appropriate `RefundCallArgs`. The refund callback should be unconditionally attached whenever ERC-20 tokens are burned, regardless of compile-time feature flags. The `error_refund` feature gate should be removed or made a default feature so that the safe path is the only path.

---

### Proof of Concept

1. Deploy Aurora Engine **without** the `error_refund` feature.
2. Bridge a NEP-141 token into Aurora as an ERC-20 (standard `ft_transfer_call` → `ft_on_transfer` flow).
3. From an EVM address holding the ERC-20, call the `ExitToNear` precompile specifying a NEAR recipient account that has **never registered** with the NEP-141 contract (storage deposit not paid).
4. Observe: the ERC-20 balance is immediately reduced (tokens burned in EVM).
5. The `ft_transfer` promise to the NEP-141 contract fails with "account not registered."
6. Because `callback_args == ExitToNearPrecompileCallbackArgs::default()`, no callback was attached — there is no `exit_to_near_precompile_callback` invocation.
7. The NEP-141 tokens remain in Aurora's account. The user's ERC-20 tokens are gone. No recovery is possible without admin intervention.

This is the direct analog of the `CyanPeerPlan.revive()` bug: a multi-step process (ERC-20 burn → NEP-141 transfer) reaches its "completion" state (burn committed) but the associated locked value (NEP-141 balance in Aurora) is never released back to the user when the second step fails.
