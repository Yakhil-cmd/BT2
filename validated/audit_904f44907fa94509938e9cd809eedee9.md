### Title
Zero-gas nop for `Module::deserialize` failure under `fix_contract_loading_error=false` (protocol version ≤85) — (`runtime/near-vm-runner/src/wasmtime_runner/mod.rs`)

---

### Summary

When `Module::deserialize` fails inside `with_compiled_and_loaded` and `fix_contract_loading_error=false` (protocol version ≤85), the error escapes the memory-cache closure as `Err(VMRunnerError::LoadingError(...))` **before** `before_loading_executable` is ever called. The caller `execute_function_call` converts this into `VMOutcome::nop_outcome(...)`, which carries `burnt_gas = 0`. The attacker therefore pays zero gas for a failed execution that forced the node to deserialize a compiled artifact.

---

### Finding Description

**Step 1 — `Module::deserialize` failure path.**

Inside `with_compiled_and_loaded`, the `Module::deserialize` call is nested inside the `cache.memory_cache().try_lookup()` closure: [1](#0-0) 

When `fix_contract_loading_error` is `false`, the `Err` arm returns `Err(VMRunnerError::LoadingError(...))` directly from the closure. The `?` operator at the end of `try_lookup` propagates this error out of `with_compiled_and_loaded` immediately.

**Step 2 — Gas charging is skipped.**

`before_loading_executable` (which charges `contract_loading_base + wasm_bytes × contract_loading_bytes`) is called **after** the `try_lookup` call: [2](#0-1) 

Because the error propagates via `?` at line 812, this line is never reached. No loading fee is charged.

**Step 3 — `VMRunnerError::LoadingError` becomes a zero-gas nop.**

In `execute_function_call`: [3](#0-2) 

`VMOutcome::nop_outcome` is constructed without a `result_state` / gas counter, so `burnt_gas = 0` and `used_gas = 0`. The caller then adds this to `result.gas_burnt`: [4](#0-3) 

Adding zero means `gas_burnt` is not updated, and the attacker's account is not charged.

**Step 4 — Protocol version gating.**

`fix_contract_loading_error` is `false` for protocol version ≤85: [5](#0-4) 

`MIN_SUPPORTED_PROTOCOL_VERSION = 84`, so the current binary can still process blocks at version 84 and 85: [6](#0-5) 

The `fix_contract_loading_error` field is documented explicitly in `Config`: [7](#0-6) 

**Step 5 — Triggering `Module::deserialize` failure.**

`MAX_CORE_INSTANCE_SIZE = 1 << 20` (1 MiB) is the pooling-allocator cap on the VMContext slot per instance: [8](#0-7) 

Wasmtime checks this limit at `Module::deserialize` time (not at instantiation). A contract with ~100 k mutable globals (each occupying 8 bytes in the VMContext) can exceed this cap. NEAR's `prepare_contract` has no explicit global-count limit (no `max_globals` parameter exists in `LimitConfig`): [9](#0-8) 

So such a contract passes preparation and compilation (`precompile_module` succeeds) but fails deserialization at execution time.

---

### Impact Explanation

The exact corrupted value is `outcome.burnt_gas`, which should equal `contract_loading_base + wasm_bytes × contract_loading_bytes` but is instead `0`. This means:

- The attacker's account is not charged for the loading work performed by the node.
- `result.gas_burnt` in the action result is understated by the full loading fee.
- The node expends real CPU/memory deserializing the compiled artifact and receives no gas compensation.

---

### Likelihood Explanation

- Requires protocol version ≤85, which is still within `MIN_SUPPORTED_PROTOCOL_VERSION = 84`. During the epoch-transition window from version 85 to 86, this is exploitable by any unprivileged account that can submit a `FunctionCall` transaction.
- Deploying a contract with many globals is a standard `DeployContract` action requiring no special privileges.
- The fix (`FixContractLoadingError`, version 86) is the current stable version, so the window is narrow but real during the upgrade epoch.

---

### Recommendation

The fix is already present in the codebase at protocol version 86 via `FixContractLoadingError`: [10](#0-9) 

When `fix_contract_loading_error=true`, the `Module::deserialize` error is wrapped as a `FunctionCallError::LoadingError` and returned as `Ok(...)` from the closure, allowing the normal gas-charging path (`before_loading_executable` / `after_loading_executable`) to execute before the abort is finalized. No additional code change is needed beyond ensuring the network upgrades to version 86.

---

### Proof of Concept

1. Construct a WASM binary with ~100,000 mutable globals (no NEAR preparation limit blocks this).
2. Deploy it via `DeployContract` on a network running protocol version 85.
3. Call any method on the contract.
4. Observe: `Module::deserialize` fails (VMContext size > 1 MiB), `VMRunnerError::LoadingError` is returned, `execute_function_call` converts it to `VMOutcome::nop_outcome`, and `outcome.burnt_gas == 0`.
5. Repeat under protocol version 86: `Module::deserialize` failure is now caught inside the closure, wrapped as `FunctionCallError::LoadingError`, and the loading fee is charged before the abort, so `outcome.burnt_gas == contract_loading_base + wasm_bytes × contract_loading_bytes`.
