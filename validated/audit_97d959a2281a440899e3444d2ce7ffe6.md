### Title
Unbounded Loop in `calculate_rewards` Over Delegator Balance Trace Enables Permanent Freezing of Unclaimed Yield - (File: `src/pool/pool.cairo`)

---

### Summary

The `calculate_rewards` function in `src/pool/pool.cairo` contains an explicitly acknowledged unbounded loop that iterates over every entry in a delegator's `pool_member_epoch_balance` trace. Because an unprivileged pool member can grow this trace by one entry per epoch through normal balance-changing operations (`add_to_delegation_pool`, `exit_delegation_pool_intent`), a delegator who accumulates enough trace entries without claiming rewards will eventually be unable to claim — permanently freezing their unclaimed yield.

---

### Finding Description

**Root cause — unbounded loop in `calculate_rewards`:** [1](#0-0) 

The code itself acknowledges the issue:

```cairo
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
    if pool_member_checkpoint.epoch() >= until_epoch {
        break;
    }
    let to_sigma = self.find_sigma(pool_member_checkpoint, curr_epoch: until_epoch);
    ...
    entry_to_claim_from += 1;
}
```

Each iteration performs a storage read (`pool_member_trace.at(...)`) and a `find_sigma` call that itself reads from `cumulative_rewards_trace`. The cost per iteration is non-trivial.

**How the trace grows — one entry per epoch:**

`set_member_balance` inserts into the trace keyed by `epoch + K`: [2](#0-1) 

The underlying `insert` implementation only updates the last entry if the key matches; otherwise it appends a new checkpoint: [3](#0-2) 

So each balance change in a distinct epoch appends one new entry. Multiple changes within the same epoch collapse into one entry.

**Unprivileged entry points that grow the trace:**

1. `add_to_delegation_pool` — callable by the pool member or their reward address, calls `increase_member_balance` → `set_member_balance`: [4](#0-3) 

2. `exit_delegation_pool_intent` — callable by the pool member, calls `set_member_balance`: [5](#0-4) 

**`entry_to_claim_from` is only advanced on `claim_rewards`:**

After a successful `claim_rewards`, `entry_to_claim_from` is updated to skip already-processed entries: [6](#0-5) 

If a delegator never calls `claim_rewards`, `entry_to_claim_from` stays at its initial value and the loop must traverse every accumulated entry from the beginning.

**`pool_member_info_v1` (view) also calls `calculate_rewards` without advancing the pointer:** [7](#0-6) 

This means even a read-only query will fail once the trace is large enough.

---

### Impact Explanation

A delegator who changes their balance once per epoch (minimum 1 wei via `add_to_delegation_pool`) without ever calling `claim_rewards` accumulates one trace entry per epoch. After a sufficient number of epochs (the exact threshold depends on Starknet's per-transaction gas limit, but on the order of thousands of epochs), both `claim_rewards` and `pool_member_info_v1` will revert with an out-of-gas error. The delegator's accumulated yield becomes permanently unclaimable — matching the allowed impact **"Permanent freezing of unclaimed yield"**.

---

### Likelihood Explanation

The protocol is designed to run indefinitely across many epochs. A long-term delegator who actively manages their position (adding stake each epoch to compound, or partially exiting and re-entering) will naturally accumulate trace entries. The minimum cost per epoch is 1 wei of token plus transaction gas. No privileged access is required. The protocol's own comment acknowledges the loop is unbounded and relies on an informal assumption that gas limits will not be exceeded — an assumption that breaks down over a long enough time horizon.

---

### Recommendation

1. **Enforce a maximum trace length per pool member.** Reject `add_to_delegation_pool` and `exit_delegation_pool_intent` if the trace already has more than `MAX_TRACE_ENTRIES` entries (e.g., 1000).
2. **Alternatively, checkpoint rewards lazily on every balance change.** Compute and store `unclaimed_rewards` at the time of each balance change so that `calculate_rewards` only needs to process the delta since the last checkpoint, bounding the loop to a small constant (at most K entries).
3. **Remove the informal "unlikely to exceed gas limits" assumption** and replace it with an enforced invariant.

---

### Proof of Concept

```
1. Delegator D enters pool with amount = min_delegation.
2. For epoch i = 1 to N (e.g., N = 5000):
     a. Advance epoch (staker attests).
     b. D calls add_to_delegation_pool(pool_member: D, amount: 1).
        → set_member_balance inserts a new checkpoint at epoch i + K.
        → pool_member_epoch_balance trace length = i.
3. D calls claim_rewards(pool_member: D).
   → calculate_rewards loops from entry_to_claim_from=0 to trace_length=N.
   → Each iteration reads storage and calls find_sigma.
   → Transaction runs out of gas before completing.
   → D's accumulated rewards are permanently frozen.
```

The same out-of-gas failure occurs on `pool_member_info_v1(pool_member: D)`, making the frozen state unobservable as well.
