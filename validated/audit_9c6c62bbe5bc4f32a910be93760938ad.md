Audit Report

## Title
Missing `http2_max_header_list_size` on SOCKS Proxy Client Enables Adapter Crash via Oversized HTTP/2 Response Headers — (`rs/https_outcalls/adapter/src/rpc_server.rs`)

## Summary
The HTTPS outcalls adapter builds two hyper clients: a direct client protected by `.http2_max_header_list_size(MAX_HEADER_LIST_SIZE)` (52 KB), and a SOCKS proxy client that omits this call entirely, leaving h2's enforcement limit at `u32::MAX`. An unprivileged canister can route a request through the SOCKS path by targeting an IPv4 address it controls, then have its server respond with an HTTP/2 HEADERS frame exceeding hyper's internal `HeaderMap` capacity. Because `abort_on_panic()` is installed unconditionally at adapter startup, the resulting hyper panic terminates the entire adapter process, dropping all in-flight HTTP outcalls on that node.

## Finding Description
In `CanisterHttp::new` (L88–90), the direct client is constructed with:
```rust
let client = Client::builder(TokioExecutor::new())
    .http2_max_header_list_size(MAX_HEADER_LIST_SIZE)  // 52 * 1024
    .build::<_, Full<Bytes>>(direct_https_connector);
```
This causes the h2 library to enforce the limit on the client receive side: if the server sends a HEADERS frame exceeding 52 KB, h2 rejects it with a connection error before passing any headers to hyper's `HeaderMap`.

In `create_socks_proxy_client` (L119–127), the SOCKS client is constructed without this call:
```rust
Client::builder(TokioExecutor::new()).build::<_, Full<Bytes>>(
    builder.enable_all_versions().wrap_connector(SocksConnector { ... })
)
```
With no limit set, h2 defaults to `u32::MAX` and passes all received headers to hyper. The developers' own comment at L38–40 states: *"hyper panics if we try to allocate more headers"* and *"hyper sometimes grows the map by doubling the entries."*

The SOCKS path is reached via the `.or_else` fallback at L339–364: when the direct connection fails (e.g., IPv6-only node cannot reach an IPv4 destination), `do_https_outcall_socks_proxy` is invoked automatically. The post-receive header size check at L402–417 runs only after `http_resp` is already returned from hyper — after h2 has decoded and hyper has inserted all headers — providing no protection against the panic. `validate_headers` (L483–524) only validates outgoing request headers from the canister, not incoming response headers from the server.

`abort_on_panic()` at `main.rs:23` converts any thread panic into a process abort with core dump.

## Impact Explanation
A crash of the adapter process drops every in-flight HTTP outcall on the affected replica node. Canisters relying on HTTP outcalls receive transient errors for all pending requests. The adapter is a per-node process; crashing it on multiple nodes simultaneously (by issuing concurrent requests from multiple canisters) degrades subnet-wide HTTP outcall availability until the adapter is restarted by systemd. This matches the allowed impact: **High — Application/platform-level DoS, crash, or subnet availability impact not based on raw volumetric DDoS** ($2,000–$10,000).

## Likelihood Explanation
The SOCKS path is the normal production fallback for IPv4 destinations, since IC nodes are IPv6-only. Any canister can issue an `http_request` to an IPv4 HTTPS endpoint it controls. No privileged access, key material, or network-level attack is required. The attacker controls the destination server entirely and can craft any HTTP/2 HEADERS frame. The attack is repeatable: after systemd restarts the adapter, the same request can be reissued. Concurrent requests from multiple canisters can target multiple nodes simultaneously.

## Recommendation
Add `.http2_max_header_list_size(MAX_HEADER_LIST_SIZE)` to the `Client::builder` call inside `create_socks_proxy_client`, mirroring the direct client:
```rust
Client::builder(TokioExecutor::new())
    .http2_max_header_list_size(MAX_HEADER_LIST_SIZE)  // add this line
    .build::<_, Full<Bytes>>(
        builder.enable_all_versions().wrap_connector(SocksConnector { ... })
    )
```

## Proof of Concept
1. Deploy a canister that calls `http_request` targeting `https://<attacker-IPv4>/`.
2. The adapter's direct connection fails (node has no IPv4 route); `do_https_outcall_socks_proxy` is invoked via the `.or_else` fallback at L339.
3. The attacker's server, on receiving the TLS-tunnelled HTTP/2 connection via the SOCKS proxy, responds with a HEADERS frame containing either >32,768 individual header fields or a total header block exceeding 52 KB.
4. Because `create_socks_proxy_client` omits `http2_max_header_list_size`, h2 does not reject the frame at the protocol level; hyper attempts to insert all headers into its `HeaderMap`.
5. `HeaderMap` panics; `abort_on_panic()` kills the adapter process.
6. Differential test: send the identical oversized response to the direct client — h2 rejects it with a connection error before hyper touches the `HeaderMap`, returning a graceful error to the canister. The SOCKS client panics instead.
7. A deterministic integration test can be written using a local HTTP/2 server (e.g., with `h2` or `hyper` in test mode) that sends a crafted HEADERS frame exceeding the limit, routed through a local SOCKS5 proxy, to confirm the panic vs. graceful-error differential.
