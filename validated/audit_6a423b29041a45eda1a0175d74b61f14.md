### Title
Missing Blocked-Sender-Address Check in Gateway Stateless Validation Allows Any Address to Submit Transactions — (`File: crates/apollo_gateway/src/stateless_transaction_validator.rs`)

### Summary
`StatelessTransactionValidator::validate()` contains an explicit, unimplemented TODO acknowledging that no mechanism exists to reject transactions from blocked sender addresses. Every transaction type (Invoke, Declare, DeployAccount) passes through this function as the first admission gate, yet no blocked-address check is performed at any point in the stateless path, the stateful path, or the mempool admission path. Any address the operator intends to block can freely submit transactions that are admitted to the mempool.

### Finding Description
In `StatelessTransactionValidator::validate()`, the very first lines of the function body are:

```rust
// TODO(Arni, 1/5/2024): Add a mechanism that validate the sender address is not blocked.
// TODO(Arni, 1/5/2024): Validate transaction version.
```

The function then proceeds to check contract address validity, empty fields, resource bounds, calldata/signature sizes, DA modes, and Sierra version — but never checks whether the sender address is on any blocklist. [1](#0-0) 

A search across the entire codebase confirms there is no blocked-address enforcement anywhere in the gateway, stateful validator, or mempool layers. The stateful validator (`extract_state_nonce_and_run_validations`) only checks nonce validity and resource bounds against the previous block's gas price. [2](#0-1) 

The `run_pre_validation_checks` path calls `validate_state_preconditions` (nonce + resource bounds) and `validate_by_mempool`, neither of which performs any sender identity check. [3](#0-2) 

The full `add_tx_inner` flow in the gateway confirms the sequence: stateless validate → convert → stateful validate → mempool add, with no blocked-address gate at any step. [4](#0-3) 

### Impact Explanation
The gateway admission invariant — "blocked addresses must not be able to submit transactions" — is entirely absent. A blocked sender address can submit Invoke, Declare, or DeployAccount transactions that pass all existing checks (valid contract address format, valid nonce, sufficient resource bounds) and are admitted to the mempool, propagated via P2P to peer sequencers, and ultimately sequenced into blocks. This matches the **High** impact: "Mempool/gateway/RPC admission accepts invalid transactions before sequencing."

### Likelihood Explanation
Exploitation requires only that an attacker know their address is intended to be blocked. Since the check is completely absent (not merely bypassable), any blocked address trivially bypasses the control by submitting a well-formed transaction. No special privileges, timing, or state manipulation are required.

### Recommendation
Implement the blocked-address check that the TODO acknowledges is missing. Add a configurable blocklist (e.g., a `HashSet<ContractAddress>` in `StatelessTransactionValidatorConfig`) and check the sender address against it at the top of `StatelessTransactionValidator::validate()`, before any other checks, analogous to how `check_declare_permissions` gates Declare transactions:

```rust
pub fn validate(&self, tx: &RpcTransaction) -> StatelessTransactionValidatorResult<()> {
    Self::validate_sender_not_blocked(&self.config.blocked_addresses, tx)?;
    // ... existing checks
}
```

For `DeployAccount` transactions (which have no `sender_address` field in the RPC form), the deployer address should be computed and checked before admission.

### Proof of Concept
1. Operator configures a blocklist containing address `0xDEAD`.
2. Attacker controls address `0xDEAD` and submits an `RpcInvokeTransaction::V3` with `sender_address = 0xDEAD`, valid nonce, and sufficient resource bounds.
3. `StatelessTransactionValidator::validate()` is called. It checks contract address validity (`0xDEAD` is a valid felt), empty fields, resource bounds, sizes, and DA modes — all pass. [5](#0-4) 
4. The transaction proceeds to `convert_rpc_tx_to_internal_and_executable_txs`, then `extract_state_nonce_and_run_validations` (nonce and gas price checks pass), then `mempool_client.add_tx`. [6](#0-5) 
5. The transaction is admitted to the mempool and propagated to peer sequencers, bypassing the intended block entirely.

    pub fn validate(&self, tx: &RpcTransaction) -> StatelessTransactionValidatorResult<()> {
        // TODO(Arni, 1/5/2024): Add a mechanism that validate the sender address is not blocked.
        // TODO(Arni, 1/5/2024): Validate transaction version.

        Self::validate_contract_address(tx)?;
        Self::validate_empty_account_deployment_data(tx)?;
        Self::validate_empty_paymaster_data(tx)?;
        self.validate_resource_bounds(tx)?;
        self.validate_tx_size(tx)?;
        self.validate_nonce_data_availability_mode(tx)?;
        self.validate_fee_data_availability_mode(tx)?;

        if let RpcTransaction::Invoke(invoke_tx) = tx {
            self.validate_client_side_proving_allowed(invoke_tx)?;
            self.validate_proof_facts_and_proof_consistency(invoke_tx)?;
        }

        if let RpcTransaction::Declare(declare_tx) = tx {
            self.validate_declare_tx(declare_tx)?;
        }
        Ok(())
    }
