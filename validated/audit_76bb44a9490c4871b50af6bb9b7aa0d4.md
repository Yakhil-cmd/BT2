### Title
Unrestricted `domainSeparationTag` in `VerifySignatureFromRuntime` Enables Cross-Context Transaction Signature Reuse - (File: `fvm/crypto/crypto.go`)

---

### Summary

`VerifySignatureFromRuntime` — the FVM function backing Cadence's `PublicKey.verify()` — accepts any arbitrary `domainSeparationTag` string from user-controlled Cadence code with no restriction. Because the Flow transaction domain tag `"FLOW-V0.0-transaction"` is not blocked, a malicious Cadence contract author can craft a contract that asks a user to sign a transaction envelope message under that tag. The resulting off-chain signature is cryptographically identical to a valid Flow transaction envelope signature, allowing the attacker to execute an arbitrary transaction on behalf of the victim.

---

### Finding Description

**Root cause — `fvm/crypto/crypto.go` lines 132–199:**

`VerifySignatureFromRuntime` receives `tag string` directly from the Cadence runtime without any allowlist or blocklist: [1](#0-0) 

For ECDSA the tag is forwarded to `NewPrefixedHashing`, which prepends the 32-byte padded tag to every message before hashing: [2](#0-1) 

For BLS there are explicitly **no tag constraints**: [3](#0-2) 

**How transaction verification works:**

`ValidateExtensionDataAndReconstructMessage` (plain scheme) reconstructs the verification message as `TransactionDomainTag || payload`: [4](#0-3) 

`VerifySignatureFromTransaction` then verifies the signature over that concatenated message using a **prefix-free** hasher (`NewHashing`): [5](#0-4) 

`TransactionDomainTag` is the 32-byte padded form of `"FLOW-V0.0-transaction"`: [6](#0-5) 

**The collision:**

When a Cadence script calls `PublicKey.verify(sig, envelopeMessage, "FLOW-V0.0-transaction", SHA2_256)`, the FVM constructs `NewPrefixedHashing(SHA2_256, "FLOW-V0.0-transaction")` and verifies:

```
hash( paddedTag("FLOW-V0.0-transaction") || envelopeMessage )
= hash( TransactionDomainTag || envelopeMessage )
```

This is byte-for-byte identical to what `VerifySignatureFromTransaction` checks for a real transaction envelope signature. There is no code path that prevents a Cadence contract from using `"FLOW-V0.0-transaction"` as its tag. [7](#0-6) 

The `paddedDomainTag` function confirms that tags shorter than 32 bytes are zero-padded, so `"FLOW-V0.0-transaction"` and `TransactionDomainTag[:]` produce the same 32-byte prefix: [8](#0-7) 

---

### Impact Explanation

An attacker who controls a Cadence contract can trick a victim into providing a signature that is valid as a Flow transaction envelope signature. Once obtained, the attacker submits the victim's signature as the `EnvelopeSignature` of a pre-crafted transaction (e.g., one that transfers tokens, adds a malicious key, or deploys a contract). The FVM's `signatureContinuation.verify()` will accept it: [9](#0-8) 

This constitutes **unauthorized transaction authorization** — the victim's account key is used to authorize a transaction the victim never intended to sign.

---

### Likelihood Explanation

- **Entry path is unprivileged**: any Cadence contract author can deploy the malicious contract.
- **Off-chain signing is common**: wallets and SDKs routinely expose raw signing APIs (e.g., `SignUserMessage`) that accept arbitrary tags and messages.
- **Attacker can determine the envelope message**: the target transaction's `envelopeMessage` is deterministic given the script, arguments, reference block ID, and the victim's current sequence number — all of which are observable on-chain.
- **No protocol-level guard**: `VerifySignatureFromRuntime` imposes no restriction on the tag value.

Likelihood is **medium**: it requires a social-engineering step (convincing the victim to sign via a malicious dApp), but the cryptographic preconditions are fully satisfied by the existing code.

---

### Recommendation

In `VerifySignatureFromRuntime` (`fvm/crypto/crypto.go`), reject any `domainSeparationTag` that matches a reserved Flow protocol tag before constructing the hasher. At minimum, block `"FLOW-V0.0-transaction"` (and its padded form) and all tags matching the `"FLOW-"` prefix pattern used by `module/signature/signing_tags.go`: [10](#0-9) 

Alternatively, enforce a mandatory application-level prefix (e.g., `"user-"`) for all tags accepted through the Cadence runtime path, mirroring the fix described in the external report.

---

### Proof of Concept

```
1. Attacker observes victim's account: address A, sequence number N,
   public key K (ECDSA-P256 / SHA2-256).

2. Attacker crafts transaction T:
     script:    <drain A's tokens>
     payer:     A
     proposer:  A, keyIndex=0, seqNum=N
     authorizers: [A]

3. Attacker computes envelopeMessage(T) = RLP(T.PayloadCanonicalForm(),
                                               T.PayloadSignatures)

4. Attacker deploys malicious Cadence contract C:
     "Sign this 32-byte nonce to claim your airdrop."
     // The "nonce" is actually envelopeMessage(T).
     // The contract calls PublicKey.verify(...,
     //   domainSeparationTag: "FLOW-V0.0-transaction", ...)

5. Victim's wallet signs envelopeMessage(T) off-chain with K under
   tag "FLOW-V0.0-transaction" and returns sig S.

6. Attacker constructs a Flow transaction submission for T with
   EnvelopeSignature = { address: A, keyIndex: 0, sig: S }.

7. FVM verifies:
     message = TransactionDomainTag || envelopeMessage(T)
     hash(message) verified against S under K  →  VALID

8. Transaction T executes; victim's funds are drained.
```

// paddedDomainTag converts a string into a padded byte array.
func paddedDomainTag(s string) ([tagLength]byte, error) {
	var tag [tagLength]byte
	if len(s) > tagLength {
		return tag, fmt.Errorf("domain tag cannot be longer than %d characters, got %s", tagLength, s)
	}
	copy(tag[:], s)
	return tag, nil
}

func (entry *signatureContinuation) verify() errors.CodedError {
	if entry.invokedVerify {
		return entry.verifyErr
	}

	entry.invokedVerify = true

	valid, message := entry.ValidateExtensionDataAndReconstructMessage(entry.payload)
	if !valid {
		entry.verifyErr = entry.newError(fmt.Errorf("signature extension data is not valid"))
		return entry.verifyErr
	}

	valid, err := crypto.VerifySignatureFromTransaction(
		entry.Signature,
		message,
		entry.accountKey.PublicKey,
		entry.accountKey.HashAlgo,
	)
	if err != nil {
		entry.verifyErr = entry.newError(err)
	} else if !valid {
		entry.verifyErr = entry.newError(fmt.Errorf("signature is not valid"))
	}

	return entry.verifyErr
}

// Flow protocol prefix
const protocolPrefix = "FLOW-"

// Flow protocol version
const protocolVersion = "-V00-"

// Ciphersuite index
// Only one ciphersuite is used in Flow protocol
const cipherSuiteIndex = "CS00-"

// an example of domain tag output is :
// FLOW-CERTAIN_DOMAIN-V00-CS00-with-cipherSuite
// where cipherSuite is fixed by the Flow crypto library
// (only one ciphersuite is provided for BLS signatures by the crypto library
// and therefore it's not possible to choose one)
func tag(domain string) string {
	return protocolPrefix + domain + protocolVersion + cipherSuiteIndex + "with-"
}
