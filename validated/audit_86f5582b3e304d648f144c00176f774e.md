### Title
Signature Validation Bypass via `skip_stateful_validations` Allows Unsigned Invoke Transactions to Enter Mempool — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the gateway's stateful transaction validator skips the `__validate__` entry point (which performs signature verification) for any invoke transaction with `nonce=1` when the on-chain account nonce is `0` and `account_tx_in_pool_or_recent_block` returns `true`. Because `account_tx_in_pool_or_recent_block` returns `true` for **any** transaction in the pool for that address — not exclusively a `deploy_account` — an attacker who observes a victim's pending `deploy_account` can immediately submit an invoke with `nonce=1` for the victim's address carrying an arbitrary/forged signature. The gateway accepts it without calling `__validate__`, injecting an unsigned transaction into the mempool. When the batcher later executes it with `validate=true` (via `AccountTransaction::new_for_sequencing`), `__validate__` fails, the transaction is reverted, and the fee is charged from the victim's newly-deployed account.

---

### Finding Description

**Root cause — `skip_stateful_validations`:** [1](#0-0) 

The function returns `true` (skip validation) when all three conditions hold:
1. The transaction is an `Invoke` with `nonce == 1`.
2. The on-chain `account_nonce == 0` (account not yet deployed).
3. `account_tx_in_pool_or_recent_block(sender_address)` returns `true`.

The inline comment claims this is safe because the account "either has a deploy_account transaction or transactions with future nonces that passed validations." This reasoning is broken.

**`account_tx_in_pool_or_recent_block` is not deploy-account-specific:** [2](#0-1) 

It returns `true` if the address appears in `self.state` (recent-block history) **or** `self.tx_pool` (any pending transaction). A victim's `deploy_account` with `nonce=0` is sufficient to satisfy this check.

**Gateway sets `validate=false` and skips `__validate__`:** [3](#0-2) 

When `skip_validate=true`, `execution_flags.validate = !skip_validate = false`. The blockifier's `validate_tx` then short-circuits: [4](#0-3) 

No `__validate__` entry point is called. The transaction is forwarded to the mempool without any signature check.

**Batcher re-executes with `validate=true`:**

The batcher constructs transactions via `AccountTransaction::new_for_sequencing`: [5](#0-4) 

This sets `validate: true` and `strict_nonce_check: true`. When the batcher executes the attacker's invoke, `__validate__` is called and fails (forged signature), causing a revert — but the fee is still charged from the victim's account.

**Pre-validation checks do not catch the attack:**

`validate_state_preconditions` allows `nonce=1` when `account_nonce=0` (gap of 1 is within `max_allowed_nonce_gap`): [6](#0-5) 

`validate_by_mempool` checks only nonce ordering and fee escalation — not the signature: [7](#0-6) 

**End-to-end exploit flow:**

1. Victim submits `deploy_account` (nonce=0) → enters mempool → `account_tx_in_pool_or_recent_block(victim_addr)` returns `true`.
2. Attacker submits `Invoke(nonce=1, sender=victim_addr, calldata=<arbitrary>, signature=<forged>)`.
3. Gateway: `tx_nonce=1`, `account_nonce=0`, `account_tx_in_pool_or_recent_block` → `true` → `skip_validate=true` → `validate=false` → `__validate__` skipped → transaction accepted into mempool.
4. Attacker's transaction occupies the `(victim_addr, nonce=1)` slot. Victim's legitimate nonce=1 invoke is rejected as a duplicate or requires fee escalation to displace it.
5. Batcher executes `deploy_account` (victim's account deployed), then executes attacker's invoke with `validate=true` → `__validate__` fails → revert, fee charged from victim's account.

---

### Impact Explanation

**Primary (High — Mempool admission):** An invalid, unsigned invoke transaction is accepted by the gateway and enters the mempool, violating the invariant that all admitted transactions must pass account signature verification. This matches: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

**Secondary (Critical — Unauthorized transaction accepted):** The unsigned transaction is bound to the victim's account address and nonce, constituting an unauthorized transaction accepted through account validation/signature logic. The attacker occupies the victim's first post-deploy nonce slot, forcing fee escalation or causing a fee-charging revert on the victim's account. This matches: *"Invalid or unauthorized Starknet transaction accepted through account validation, signature, nonce… logic."*

---

### Likelihood Explanation

- **Precondition:** The victim must have a pending `deploy_account` transaction in the mempool. This is a common, observable event (new account onboarding).
- **Attacker capability:** Any unprivileged user can submit an RPC transaction to the gateway. No keys, admin access, or special privileges are required.
- **Timing:** The attacker must submit before the `deploy_account` is committed to a block. The mempool window is typically multiple seconds to minutes, giving ample time.
- **Detectability:** The attack is silent at the gateway level; the victim only discovers it when their own nonce=1 invoke is rejected or when the fee is charged for the failed transaction.

---

### Recommendation

Replace the coarse `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a **`deploy_account` transaction** is pending for the sender address. The mempool should expose a dedicated query such as `has_pending_deploy_account(address) -> bool` that inspects the transaction type, not merely the address presence. Alternatively, the gateway can inspect the `InternalRpcTransaction` type of the pending transaction before granting the skip.

A minimal fix in `skip_stateful_validations`:

```rust
// Instead of:
return mempool_client.account_tx_in_pool_or_recent_block(tx.sender_address()).await ...;

// Use:
return mempool_client.has_pending_deploy_account(tx.sender_address()).await ...;
```

where `has_pending_deploy_account` only returns `true` if the pooled transaction for that address at nonce=0 is a `DeployAccount` variant.

---

### Proof of Concept

**Setup:** Starknet devnet or integration test environment. Victim account class: standard ECDSA account (e.g., `account_with_real_validate`).

**Steps:**

1. Fund a fresh address `A` (victim's intended account address).
2. Victim submits `deploy_account` for address `A` (nonce=0, valid signature). Confirm it is in the mempool via `account_tx_in_pool_or_recent_block(A) == true`.
3. Attacker submits:
   ```
   RpcInvokeTransaction::V3 {
       sender_address: A,
       nonce: 1,
       calldata: [<transfer_to_attacker_calldata>],
       signature: [0x0, 0x0],   // forged / empty
       resource_bounds: <valid bounds>,
       ...
   }
   ```
4. **Expected (broken) behavior:** Gateway accepts the transaction. `skip_stateful_validations` returns `true` because `account_tx_in_pool_or_recent_block(A)` is `true` (victim's deploy_account is in pool). `__validate__` is not called. Transaction enters mempool at `(A, nonce=1)`.
5. Victim attempts to submit their own invoke at nonce=1 → rejected with `DuplicateNonce` or requires fee escalation.
6. Batcher executes `deploy_account` (account deployed), then executes attacker's invoke with `validate=true` → `__validate__` panics on invalid ECDSA signature → transaction reverted, fee deducted from `A`.

**Verification:** Confirm via `mempool_snapshot` that the attacker's transaction (with forged signature) is present at `(A, nonce=1)` after step 4, before the block is committed.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
