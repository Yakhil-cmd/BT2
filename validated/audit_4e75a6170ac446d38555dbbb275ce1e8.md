### Title
`PresignOutput` Not Consumed by `rerandomize_presign` Enables Presignature Reuse and Private Key Recovery — (`src/ecdsa/ot_based_ecdsa/mod.rs`, `src/ecdsa/robust_ecdsa/mod.rs`)

---

### Summary

Both the OT-based and Robust ECDSA schemes expose a `rerandomize_presign` function that accepts the `PresignOutput` by shared reference (`&PresignOutput`) rather than by value. This means the caller retains ownership of the original presignature after rerandomization and can call `rerandomize_presign` again with different arguments — producing two distinct `RerandomizedPresignOutput` values from the same underlying nonce material. Using both in separate signing sessions yields signatures with multiplicatively related nonces, which is sufficient for full private key recovery via standard ECDSA nonce-reuse techniques. The library documents this as a critical invariant ("never reuse a presignature") but does not enforce it at the type level.

---

### Finding Description

**Root cause — OT-based ECDSA:**

```rust
// src/ecdsa/ot_based_ecdsa/mod.rs
pub fn rerandomize_presign(
    presignature: &PresignOutput,          // ← shared reference; caller keeps ownership
    args: &RerandomizationArguments,
) -> Result<Self, ProtocolError>
``` [1](#0-0) 

**Root cause — Robust ECDSA:**

```rust
// src/ecdsa/robust_ecdsa/mod.rs
pub fn rerandomize_presign(
    presignature: &PresignOutput,          // ← shared reference; caller keeps ownership
    args: &RerandomizationArguments,
) -> Result<Self, ProtocolError>
``` [2](#0-1) 

Additionally, `PresignOutput` derives `Clone`, so even if the API were changed to consume by value, a caller could trivially clone before passing: [3](#0-2) 

**Exploit path:**

1. A library caller (e.g., a malicious coordinator or any party holding a `PresignOutput`) calls `rerandomize_presign(&presign_out, &args1)` with `(msg_hash1, entropy1, tweak1)` → obtains `RerandomizedPresignOutput1`.
2. The same caller calls `rerandomize_presign(&presign_out, &args2)` with `(msg_hash2, entropy2, tweak2)` → obtains `RerandomizedPresignOutput2`.
3. Both rerandomized outputs are used in two separate signing sessions, producing signatures `(R1, s1)` and `(R2, s2)`.
4. Because both derive from the same underlying nonce `k`, the nonces satisfy `k1 = k · δ1⁻¹` and `k2 = k · δ2⁻¹`, i.e., `k1 = k2 · (δ2/δ1)` — a multiplicative relationship.
5. Given two signatures with multiplicatively related nonces and known `δ1`, `δ2` (which are public, derived from public HKDF inputs), the private key `x` is recoverable algebraically.

The security documentation explicitly acknowledges this attack vector:

> "If different subsets of size at least 2t+1 sign different (h, ε) values using shares derived from the same presignature, the resulting signatures use multiplicatively related nonces and the secret key can be recovered using standard ECDSA nonce-reuse attacks." [4](#0-3) 

The orchestration documentation also states:

> "It's **critical** that the output is then destroyed, so that no other group of parties attempts to re-use that output for another phase." [5](#0-4) 

Despite this documented requirement, the API provides no enforcement mechanism. The `PresignOutput` is never invalidated, zeroed, or moved-out after rerandomization.

---

### Impact Explanation

**Critical — Extraction of aggregate secret key material.**

Two signing sessions using the same `PresignOutput` with different `(msg_hash, tweak, entropy)` tuples produce signatures `(R1, s1)` and `(R2, s2)` where the underlying nonces are multiplicatively related by the publicly known ratio `δ2/δ1`. This is sufficient to recover the full aggregate private key `x` using standard ECDSA nonce-reuse algebra. Once `x` is recovered, the attacker can forge valid threshold signatures for any message without participation of any honest party.

---

### Likelihood Explanation

**High.** Any library caller that stores `PresignOutput` values (e.g., in a database for later use, or for retry after a failed signing session) will naturally reuse them. The API signature actively invites this: `&PresignOutput` communicates "borrow, don't consume." A malicious coordinator can deliberately trigger two signing sessions from the same presignature by initiating a first session, aborting it, and initiating a second with a different message hash — a realistic operational scenario explicitly called out in the security notes ("even across failed, aborted, or partially completed signing sessions"). [6](#0-5) 

---

### Recommendation

Change both `rerandomize_presign` functions to consume the `PresignOutput` by value, enforcing single-use at the type level:

```rust
// Before (both schemes):
pub fn rerandomize_presign(
    presignature: &PresignOutput,
    args: &RerandomizationArguments,
) -> Result<Self, ProtocolError>

// After:
pub fn rerandomize_presign(
    presignature: PresignOutput,   // consumed; caller cannot reuse
    args: &RerandomizationArguments,
) -> Result<Self, ProtocolError>
```

Additionally, remove the `Clone` derive from `PresignOutput` in both schemes, or at minimum gate it behind a `#[cfg(test)]` attribute, so that production code cannot trivially circumvent the ownership-based protection. [3](#0-2) [7](#0-6) 

---

### Proof of Concept

```rust
// Both calls succeed; presign_out is never invalidated
let rerand1 = RerandomizedPresignOutput::rerandomize_presign(
    &presign_out,
    &RerandomizationArguments::new(pk, tweak1, msg_hash1, big_r, participants.clone(), entropy1),
).unwrap();

let rerand2 = RerandomizedPresignOutput::rerandomize_presign(
    &presign_out,   // same presignature reused
    &RerandomizationArguments::new(pk, tweak2, msg_hash2, big_r, participants.clone(), entropy2),
).unwrap();

// Run two signing sessions → (R1, s1) and (R2, s2)
// δ1, δ2 are public (HKDF of public inputs)
// k1 = k·δ1⁻¹,  k2 = k·δ2⁻¹  →  k1/k2 = δ2/δ1  (known)
// Solve for x using standard nonce-reuse equations on (R1,s1,h1) and (R2,s2,h2)
```

The analog to the original report is exact: `failedMessages[origin][recipient][nonce]` was never deleted after a successful retry, allowing the same amount to be minted repeatedly. Here, `PresignOutput` is never consumed after `rerandomize_presign`, allowing the same nonce material to be used for multiple signatures — with equivalent catastrophic consequence (private key recovery instead of unlimited minting).
