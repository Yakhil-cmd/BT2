### Title
WalletConnect `chia_sendTransaction` Confirmation Dialog Hardcodes `mojo-to-xch` Unit Display While Hiding `wallet_id`, Enabling Signing-Context Confusion for CAT Spends - (File: packages/gui/src/electron/commands/Commands.ts)

### Summary

The `chia_wallet.send_transaction` WalletConnect command schema unconditionally applies `humanize: 'mojo-to-xch'` to the `amount` parameter in the user-facing confirmation dialog, while simultaneously hiding the `wallet_id` field from that same dialog. A malicious dapp that has been granted `chia_sendTransaction` permission can pass a CAT wallet's `wallet_id` instead of the default XCH wallet, causing the user to see an XCH-denominated amount in the confirmation dialog while actually approving a CAT token spend of a completely different human-readable magnitude.

### Finding Description

The `Commands.ts` file defines the WalletConnect command schema for `chia_wallet.send_transaction`. The `amount` parameter is always humanized as `mojo-to-xch`:

```ts
// packages/gui/src/electron/commands/Commands.ts
'chia_wallet.send_transaction': {
  params: [
    { name: 'amount', type: 'bigint', humanize: 'mojo-to-xch' },  // always XCH
    { name: 'wallet_id', type: 'number', hide: true },             // hidden from user
  ],
  dapp: [{ command: 'chia_sendTransaction', defaults: { wallet_id: 1 } }],
}
``` [1](#0-0) 

The `humanize: 'mojo-to-xch'` annotation routes through `humanizeParamValue` → `formatMojoXch`, which unconditionally calls `mojoToChiaLocaleString` regardless of which wallet is actually targeted:

```ts
function formatMojoXch(amount: unknown, networkPrefix?: string): string {
  const formatted = mojoToChiaLocaleString(amount as string | number);
  return networkPrefix ? `${formatted} ${networkPrefix.toUpperCase()}` : formatted;
}
``` [2](#0-1) 

The conversion ratio is 1 XCH = 1,000,000,000,000 mojos, while 1 CAT = 1,000 mojos: [3](#0-2) 

Because `wallet_id` is `hide: true`, the user never sees which wallet is being spent from in the confirmation dialog. The `defaults: { wallet_id: 1 }` only applies when the dapp omits the field; a dapp can freely override it with any wallet ID.

The separate `chia_wallet.cat_spend` command correctly uses `humanize: 'mojo-to-cat'` for its amount, confirming the schema is aware of the distinction — but `send_transaction` does not apply this awareness dynamically based on the actual `wallet_id` value: [4](#0-3) 

### Impact Explanation

A dapp granted `chia_sendTransaction` permission calls it with `wallet_id` pointing to a CAT wallet and `amount = 5_000_000` (5,000,000 mojos). The confirmation dialog shows:

- **Amount**: `0.000005 XCH` (mojo-to-xch conversion)
- **wallet_id**: hidden

The user approves, believing they are sending 0.000005 XCH. In reality, 5,000,000 mojos of CAT = **5,000 CAT** (since 1 CAT = 1,000 mojos). The unit confusion produces a magnitude difference of 1,000,000,000× between what the user sees and what they sign. This is a direct signing-context confusion causing approval of the wrong asset and wrong amount — matching the "High: Corruption, spoofing, or unsafe trust of WalletConnect state that causes a user to approve the wrong asset, amount" impact class.

### Likelihood Explanation

Any dapp that has been granted `chia_sendTransaction` permission (a spending-class permission) can exploit this without any additional privileges. The `wallet_id` override requires no special capability — it is a standard parameter the dapp controls. The user has no visible signal in the confirmation dialog that the targeted wallet is not the XCH wallet.

### Recommendation

1. **Show `wallet_id` in the confirmation dialog** (remove `hide: true`) so users can verify which wallet is being spent.
2. **Dynamically select the humanize function** based on the resolved wallet type at confirmation time. If `wallet_id` resolves to a CAT wallet, apply `mojo-to-cat`; if it resolves to the standard wallet, apply `mojo-to-xch`. This mirrors the correct per-type logic already used in `useWalletHumanValue`: [5](#0-4) 

3. **Validate `wallet_id` against the expected wallet type** for the command before presenting the confirmation dialog, and reject or warn if the type does not match.

### Proof of Concept

1. Dapp is granted `chia_sendTransaction` permission via WalletConnect.
2. Dapp calls:
   ```json
   {
     "command": "chia_sendTransaction",
     "params": {
       "wallet_id": <CAT_WALLET_ID>,
       "amount": "5000000",
       "fee": "0",
       "address": "<attacker_address>"
     }
   }
   ```
3. The GUI confirmation dialog renders `amount` via `formatMojoXch(5000000n)` → displays **"0.000005 XCH"**.
4. `wallet_id` is hidden; the user sees no indication this is a CAT wallet.
5. User clicks **Send**.
6. The daemon executes `send_transaction` against the CAT wallet, transferring **5,000 CAT** to the attacker address.
7. The user's CAT balance is drained; they believed they sent a negligible XCH amount.
