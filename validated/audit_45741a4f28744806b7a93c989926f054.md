### Title
`skip_stateful_validations` Bypasses Signature Verification for Invoke Transactions via Non-`deploy_account` Mempool Presence - (File: crates/apollo_gateway/src/stateful_transaction_validator.rs)

### Summary

The `skip_stateful_validations` function in the gateway's stateful transaction validator is designed to skip the `__validate__` entry point (signature verification) for the deploy-account + invoke UX flow. However, the guard condition it uses — `account_tx_in_pool_or_recent_block` — returns `true` for **any** transaction type in the mempool, not exclusively `deploy_account` transactions. An attacker controlling an account deployed via an L1→L2 message (nonce=0 in state, code already present) can first admit a valid nonce=0 invoke transaction, then submit a nonce=1 invoke transaction with an **invalid signature** that bypasses `__validate__` entirely and is admitted to the mempool.

---

### Finding Description

`skip_stateful_validations` (lines 429–461) fires when three conditions are simultaneously true:

1. The incoming transaction is an `Invoke` with `tx.nonce() == Nonce(Felt::ONE)`
2. The on-chain account nonce is `Nonce(Felt::ZERO)`
3. `account_tx_in_pool_or_recent_block(sender_address)` returns `true` [1](#0-0) 

When all three hold, `skip_stateful_validations` returns `true`, which propagates to `run_validate_entry_point` as `skip_validate = true`: [2](#0-1) 

Inside `StatefulValidator::perform_validations`, when `execution_flags.validate == false`, the function returns immediately **before** calling `__validate__`: [3](#0-2) 

The guard `account_tx_in_pool_or_recent_block` is implemented as: [4](#0-3) 

It checks `self.state.contains_account` (staged/committed nonces) **or** `self.tx_pool.contains_account` (any pooled transaction). Neither check inspects the **type** of the pooled transaction. The inline comment claims this is sufficient because the account "either has a deploy_account transaction or transactions with future nonces that passed validations," but this reasoning is incorrect for accounts that already have code at nonce=0. [5](#0-4) 

**Root cause:** An account deployed via an L1→L2 message (L1Handler syscall) has its class hash set and code deployed, but its nonce remains `0` because L1Handler transactions do not increment the account nonce. Such an account can receive a valid nonce=0 invoke transaction that passes `run_validate_entry_point` (the account exists, `__validate__` succeeds). Once that nonce=0 invoke is in the pool, `tx_pool.contains_account` returns `true` for the address. A subsequent nonce=1 invoke with an **arbitrary/invalid signature** then satisfies all three conditions and is admitted without any signature check.

The mempool's `validate_tx` path does not inspect signatures either — it only checks nonce ordering and fee escalation: [6](#0-5) 

---

### Impact Explanation

The gateway admits an invoke transaction carrying an invalid (or entirely absent) signature to the mempool. This violates the admission invariant: **"the gateway must verify the account's `__validate__` entry point before accepting any transaction."** The corrupted admission decision is the exact analog of the VRF bug — a pending-request context (the nonce=0 invoke) is mistaken for a deploy-account context, causing the fulfillment step (`__validate__`) to be skipped for the wrong request.

The admitted invalid transaction will fail during batcher execution when `__validate__` is called, but it has already consumed mempool capacity and passed the gateway's authoritative admission gate. This matches the allowed impact: **High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

---

### Likelihood Explanation

Accounts deployed via L1→L2 messages are a standard Starknet pattern (e.g., bridged accounts, factory-deployed wallets). Any such account with nonce=0 that has at least one pending invoke transaction in the mempool is vulnerable. The attacker only needs to:

1. Observe or create such an account.
2. Submit a valid nonce=0 invoke (or wait for one to appear).
3. Submit a nonce=1 invoke with a forged/empty signature.

No privileged access is required; the attack is fully unprivileged and reachable through the public gateway endpoint.

---

### Recommendation

Replace the type-agnostic `account_tx_in_pool_or_recent_block` check with a check that specifically verifies the presence of a **`deploy_account`** transaction for the sender address in the mempool. Concretely:

- Add a `has_deploy_account_in_pool(address: ContractAddress) -> bool` method to the mempool that inspects the transaction type stored in `tx_pool`.
- Use this narrower predicate inside `skip_stateful_validations` instead of `account_tx_in_pool_or_recent_block`.

This ensures the skip is only triggered in the intended UX scenario (deploy_account + invoke submitted together) and not for already-deployed accounts with pending invoke transactions.

---

### Proof of Concept

```
1. Deploy account A via an L1→L2 message (L1Handler).
   → A has class_hash set, code deployed, nonce = 0 in blockchain state.

2. Submit InvokeV3 tx T0:
     sender_address = A, nonce = 0, valid_signature
   → validate_nonce: 0 <= 0 <= max_gap  ✓
   → validate_by_mempool: nonce 0 >= account_nonce 0  ✓
   → run_validate_entry_point: __validate__ called, succeeds  ✓
   → T0 added to mempool pool.
   → tx_pool.contains_account(A) = true

3. Submit InvokeV3 tx T1:
     sender_address = A, nonce = 1, INVALID_SIGNATURE (e.g., all zeros)
   → validate_nonce: 0 <= 1 <= max_gap  ✓
   → validate_by_mempool: nonce 1 >= account_nonce 0  ✓
   → skip_stateful_validations:
       tx.nonce() == 1  ✓
       account_nonce == 0  ✓
       account_tx_in_pool_or_recent_block(A) == true  ✓  (T0 is in pool)
       → returns true (skip validation)
   → run_validate_entry_point called with validate=false
       → StatefulValidator::perform_validations returns Ok(()) immediately
       → __validate__ is NEVER called
   → T1 admitted to mempool with invalid signature.

4. Batcher calls get_txs, receives T1.
   During execution __validate__ is called → tx reverts.
   But T1 was admitted past the gateway's authoritative admission gate.
``` [7](#0-6) [8](#0-7) [4](#0-3)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
