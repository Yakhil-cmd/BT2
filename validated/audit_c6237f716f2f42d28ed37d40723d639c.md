### Title
`native_per_pubdata` Computed with Floor Division While `native_per_gas` Uses Ceiling Division, Causing Systematic Pubdata Resource Underpayment - (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

In ZKsync OS's transaction validation, `native_per_pubdata` is computed using `wrapping_div` (truncating/floor division), while `native_per_gas` is computed using `div_ceil` (ceiling division). This rounding direction inconsistency means every transaction that generates pubdata pays slightly fewer native resources per pubdata byte than the actual cost, allowing users to systematically underpay for pubdata publication.

---

### Finding Description

In `validate_and_compute_fee_for_transaction` in `validation_impl.rs`, two critical ratios are computed:

```rust
// native_per_gas: uses CEILING division — favors the protocol
let native_per_gas = u256_try_to_u64(&gas_price.div_ceil(native_price))...

// native_per_pubdata: uses FLOOR division — favors the user
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
``` [1](#0-0) 

`native_per_gas` uses `div_ceil` to ensure the user pays at least the full cost per gas unit. However, `native_per_pubdata` uses `wrapping_div` (Rust integer division, which truncates toward zero), meaning the user pays `floor(pubdata_price / native_price)` native units per pubdata byte instead of `ceil(pubdata_price / native_price)`.

The same inconsistency is present in the public API helper `validate_l2_tx_intrinsic_native_resources`:

```rust
// native_per_gas = ceil(gas_price / native_price)
let native_per_gas = u256_try_to_u64(&gas_price.div_ceil(native_price)).ok_or(())?;

// native_per_pubdata = pubdata_price / native_price  ← floor, no ceiling
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
``` [2](#0-1) 

The `native_per_pubdata` value flows directly into `get_resources_to_charge_for_pubdata`, which computes the actual native charge as `current_pubdata_spent * native_per_pubdata`: [3](#0-2) 

Because `native_per_pubdata` is truncated, every pubdata byte costs the user up to `(native_price - 1)` fewer native units than the true cost.

---

### Impact Explanation

The native resource model is designed so that `nativePerPubdata := pubdata_price / native_price` represents the native cost per pubdata byte. When this is rounded DOWN:

- Users are charged fewer native resources per pubdata byte than the actual cost.
- For a transaction generating `N` pubdata bytes, the total underpayment is up to `(native_price - 1) * N` native units.
- Users can afford to generate slightly more pubdata than their native resource budget should allow, since the per-byte charge is understated.
- The operator/protocol receives less compensation for the actual cost of publishing pubdata on-chain.

This is a resource accounting bug: the protocol is designed to use ceiling for `native_per_gas` to protect itself, but uses floor for `native_per_pubdata`, breaking the invariant that the protocol should never be underpaid.

---

### Likelihood Explanation

This affects **every L2 transaction** that generates pubdata when `pubdata_price` is not an exact multiple of `native_price`. Since `pubdata_price` and `native_price` are operator-set block-level parameters with no alignment requirement, the truncation error occurs in virtually all real-world blocks. The entry path is any unprivileged user submitting a standard EVM transaction that writes to storage (generating pubdata).

---

### Recommendation

Replace `wrapping_div` with `div_ceil` for the `native_per_pubdata` calculation in both locations, consistent with how `native_per_gas` is computed:

```rust
// In validation_impl.rs
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;

// In api/src/helpers.rs
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price)).ok_or(())?;
```

This ensures the protocol is never underpaid for pubdata, consistent with the ceiling rounding already applied to `native_per_gas`.

---

### Proof of Concept

**Concrete example:**
- `native_price = 10`, `pubdata_price = 19`
- Current code: `native_per_pubdata = floor(19 / 10) = 1`
- Correct value: `native_per_pubdata = ceil(19 / 10) = 2`
- For a transaction generating 1,000 pubdata bytes:
  - User pays: `1,000 * 1 = 1,000` native units
  - Should pay: `1,000 * 2 = 2,000` native units
  - Underpayment: **1,000 native units** (50% discount on pubdata cost)

Any unprivileged user submitting a transaction that writes to storage (e.g., an ERC-20 transfer) triggers this path. The attacker-controlled entry is `validate_and_compute_fee_for_transaction` called during L2 transaction validation. [4](#0-3) [5](#0-4)
