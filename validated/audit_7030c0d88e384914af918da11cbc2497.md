### Title
Sequencer-Controlled `mix_hash` (prevRandao) Is Unconstrained by ZK Public Input, Enabling Randomness Manipulation — (`zk_ee/src/system/metadata/zk_metadata.rs`, `basic_bootloader/src/bootloader/block_flow/zk/post_tx_op/public_input.rs`)

---

### Summary

The `mix_hash` field in `BlockMetadataFromOracle` is the designated source of randomness (prevRandao) exposed to EVM contracts via the `DIFFICULTY`/`PREVRANDAO` opcode. It is set entirely by the sequencer-controlled oracle and is **never committed to in the ZK proof's public input** (`ChainStateCommitment` or `BatchOutput`). This means the sequencer can freely choose any `mix_hash` value per block without the proof system detecting or constraining it, enabling manipulation of any on-chain randomness that relies on `block.prevrandao`.

---

### Finding Description

This value is returned to EVM contracts via the `difficulty()` instruction handler: [2](#0-1)

Now examine what is actually committed to in the ZK public input. `ChainStateCommitment` — the structure whose hash forms `state_before`/`state_after` in `BatchPublicInput` — contains only: [4](#0-3) 

`mix_hash` is absent. `BatchOutput`, which forms the `batch_output` field of `BatchPublicInput`, also omits `mix_hash`: [5](#0-4) 

The final `BatchPublicInput` hash — the value verified on the settlement layer — is derived solely from `state_before`, `state_after`, and `batch_output`: [6](#0-5) 

This is confirmed in both the single-block and multi-block proving paths: [7](#0-6) [8](#0-7) 

The L1 integration documentation confirms the public input structure and does not mention `mix_hash` anywhere: [9](#0-8) 

The oracle documentation itself acknowledges that oracle responses are untrusted and must be validated by calling code — but no such validation exists for `mix_hash`: [10](#0-9) 

---

### Impact Explanation

Any EVM smart contract deployed on ZKsync OS that uses `block.prevrandao` (or `block.difficulty`) as a source of randomness — lotteries, NFT trait generation, random selection, commit-reveal schemes — is fully manipulable by the sequencer. The sequencer can iterate over candidate `mix_hash` values off-chain, compute the resulting `PREVRANDAO` output, and submit only the block whose `mix_hash` produces the desired outcome. Because `mix_hash` is not part of the ZK proof's public input, the settlement layer cannot detect or reject this manipulation. The proof remains valid regardless of what `mix_hash` value was used.

---

### Likelihood Explanation

The sequencer has unconditional, unconstrained control over `mix_hash` for every block. No additional compromise, key leak, or governance attack is required beyond the sequencer acting in its own interest. Any block containing a transaction that reads `PREVRANDAO` is a candidate for exploitation. The attack requires only off-chain simulation (compute `keccak256(mix_hash_candidate)` until the desired contract outcome is reached, then submit that block), which is trivially cheap.

---

### Recommendation

Include `mix_hash` in the `ChainStateCommitment` structure so that it is committed to in the ZK proof's public input and verified on the settlement layer. This binds the sequencer to a specific `mix_hash` value per block that cannot be changed after the proof is generated. Alternatively, derive `mix_hash` deterministically from values already committed to in the public input (e.g., a hash of the previous block hash and block number), removing sequencer discretion entirely. [4](#0-3)

### Proof of Concept

```
// Off-chain attacker loop (pseudocode):
// Attacker controls the sequencer and wants to win a lottery contract
// that uses block.prevrandao % N == target_index

for candidate_mix_hash in 0..u256::MAX {
    // Simulate block execution with this mix_hash
    let prevrandao_seen_by_contract = candidate_mix_hash;
    if prevrandao_seen_by_contract % N == target_index {
        // Submit block with this mix_hash to the oracle
        submit_block(BlockMetadataFromOracle {
            mix_hash: candidate_mix_hash,
            // ... other fields
        });
        // The ZK proof will be valid regardless of mix_hash value
        // because mix_hash is not in ChainStateCommitment or BatchOutput
        break;
    }
}
```

pub struct ChainStateCommitment {
    pub state_root: Bytes32,
    pub next_free_slot: u64,
    pub block_number: u64,
    pub last_256_block_hashes_blake: Bytes32,
    pub last_block_timestamp: u64,
}

impl ChainStateCommitment {
    ///
    /// Calculate blake2s hash of chain state commitment.
    ///
    /// We are using proving friendly blake2s because this commitment will be generated and opened during proving,
    /// but we don't need to open it on the settlement layer.
    ///
    pub fn hash(&self) -> [u8; 32] {
        let mut hasher = crypto::blake2s::Blake2s256::new();
        hasher.update(self.state_root.as_u8_ref());
        hasher.update(&self.next_free_slot.to_be_bytes());
        hasher.update(&self.block_number.to_be_bytes());
        hasher.update(self.last_256_block_hashes_blake.as_u8_ref());
        hasher.update(&self.last_block_timestamp.to_be_bytes());
        hasher.finalize()
    }
