### Title
Missing Zero-Address Validation for `refund_recipient` in L1→L2 Transactions Causes Permanent Loss of Gas Refund Funds - (`basic_bootloader/src/bootloader/transaction/abi_encoded/mod.rs`)

---

### Summary

The `validate_structure` function for ABI-encoded L1→L2 and upgrade transactions contains an explicit `// TODO: validate address?` comment at the `reserved[1]` (refund recipient) field and performs no zero-address check. When `refund_recipient` is `address(0)` — which is the default when a user omits the field — the bootloader unconditionally mints the gas refund to `address(0)`, permanently burning those funds.

---

### Finding Description

In `basic_bootloader/src/bootloader/transaction/abi_encoded/mod.rs`, the `validate_structure` function validates several fields of ABI-encoded L1 transactions but explicitly skips validation of the `reserved[1]` (refund recipient) field:

```rust
// reserved[1] = refund recipient for l1 to l2 and upgrade txs
match tx_type {
    Self::L1_L2_TX_TYPE | Self::UPGRADE_TX_TYPE => {
        // TODO: validate address?   // <-- no check performed
    }
    _ => unreachable!(),
}
``` [1](#0-0) 

Downstream, in `process_l1_transaction.rs`, the refund recipient is read directly from `transaction.reserved[1]` without any zero-address guard, and the refund amount is unconditionally minted to it:

```rust
if to_refund_recipient > U256::ZERO {
    let refund_recipient = u256_to_b160_checked(transaction.reserved[1].read());
    mint_base_token::<S, Config>(
        system, ..., &to_refund_recipient, &refund_recipient, ...
    )?;
}
``` [2](#0-1) 

The `L1TxBuilder` used throughout the codebase defaults `refund_recipient` to `Address::default()` (i.e., `address(0)`) when the caller does not explicitly set it:

```rust
refund_recipient: self.refund_recipient.unwrap_or_default(),
``` [3](#0-2) 

The `ZKsyncL1Tx` struct itself also declares `refund_recipient: Address` with no default enforcement: [4](#0-3) 

---

### Impact Explanation

When `refund_recipient == address(0)`, `mint_base_token` transfers the unused-gas refund amount from the treasury to `address(0)`. Because `address(0)` is an uncontrolled burn address on ZKsync OS, those tokens are permanently inaccessible. The treasury balance is correctly debited, but the refund is lost rather than returned to the user. This constitutes a direct, irreversible loss of user funds proportional to the unused gas in the transaction. [5](#0-4) 

---

### Likelihood Explanation

The default value of `refund_recipient` in `L1TxBuilder` is `address(0)`. Any L1→L2 transaction submitted without an explicit refund recipient — a common pattern in tooling and bridge integrations — will silently burn the gas refund. The `// TODO: validate address?` comment in `validate_structure` confirms this gap was identified but never closed. The entry path requires only a standard L1→L2 transaction submission, which is fully unprivileged. [6](#0-5) 

---

### Recommendation

Add a zero-address check inside `validate_structure` for the `reserved[1]` field of L1 and upgrade transactions, replacing the `// TODO` comment:

```rust
Self::L1_L2_TX_TYPE | Self::UPGRADE_TX_TYPE => {
    let refund_recipient_raw = self.reserved[1].read();
    if u256_to_b160_checked(refund_recipient_raw) == B160::ZERO {
        return Err(());
    }
}
```

Alternatively, if `address(0)` is intentionally allowed (e.g., to redirect refunds to the sender), document this explicitly in natspec/comments and add a fallback in `process_l1_transaction` to redirect the refund to `transaction.from()` when `refund_recipient` is zero.

---

### Proof of Concept

1. Submit an L1→L2 transaction using `L1TxBuilder::new()` without calling `.refund_recipient(...)`. The builder encodes `refund_recipient = address(0)` into `reserved[1]`.
2. `validate_structure` accepts the transaction (the `// TODO` branch does nothing).
3. The transaction executes with `gas_limit > gas_used`, producing a non-zero `to_refund_recipient`.
4. `process_l1_transaction` reads `refund_recipient = address(0)` from `reserved[1]` and calls `mint_base_token(..., &to_refund_recipient, &B160::ZERO, ...)`.
5. `transfer_from_treasury` debits the treasury and credits `address(0)` — the refund is permanently burned.

The existing regression test `test_treasury_based_token_distribution_regression` in `tests/instances/transactions/src/lib.rs` already demonstrates this exact scenario: it sets `refund_recipient = address(0)` and asserts that `address(0)` receives the refund amount, confirming the funds are sent there rather than being rejected. [7](#0-6)

/// Transfers [value] from the treasury account to address [to].
///
/// Returns `TreasuryTransferFailed` if:
/// - Treasury has insufficient balance
/// - Balance overflow occurs
pub fn transfer_from_treasury<'a, S: EthereumLikeTypes + 'a>(
    system: &mut System<S>,
    nominal_token_value: &U256,
    to: &B160,
    resources: &mut S::Resources,
    fee_payment_in_simulation: bool,
) -> Result<(), BootloaderSubsystemError>
where
    S::IO: IOSubsystemExt,
{
    system_log!(
        system,
        "Transferring {nominal_token_value:?} tokens from treasury to {to:?}\n"
    );

    let treasury_address = &system_hooks::addresses_constants::BASE_TOKEN_HOLDER_ADDRESS;

    let _ = system
        .io
        .update_account_nominal_token_balance(
            zk_ee::execution_environment_type::ExecutionEnvironmentType::EVM,
            resources,
            treasury_address,
            nominal_token_value,
            true, // true = subtract from balance
            fee_payment_in_simulation,
        )
        .map_err(|e| -> BootloaderSubsystemError {
            match e {
                SubsystemError::LeafUsage(balance_error) => {
                    system_log!(system, "Treasury transfer failed: {balance_error:?}");
                    interface_error!(BootloaderInterfaceError::TreasuryTransferFailed)
                }
                _ => wrap_error!(e),
            }
        })?;

    let _ = system
        .io
        .update_account_nominal_token_balance(
            zk_ee::execution_environment_type::ExecutionEnvironmentType::EVM,
            resources,
            to,
            nominal_token_value,
            false, // false = add to balance
            fee_payment_in_simulation,
        )
        .map_err(|e| -> BootloaderSubsystemError {
            match e {
                SubsystemError::LeafUsage(balance_error) => {
                    system_log!(system, "Error while minting: {balance_error:?}");
                    interface_error!(BootloaderInterfaceError::MintingBalanceOverflow)
                }
                _ => wrap_error!(e),
            }
        })?;

    Ok(())
}
