### Title
Missing `app_pk` Identity-Element Validation in CKD Allows Direct Disclosure of Confidential Derived Key — (`src/confidential_key_derivation/protocol.rs`)

### Summary

The `ckd()` function in the Confidential Key Derivation protocol accepts a caller-supplied `app_pk: PublicKey` (a `blstrs::G1Projective` point) without validating that it is not the group identity element. If an attacker submits `app_pk = G1::identity()`, the ElGamal masking term `y * A` collapses to zero, causing the coordinator's output `C` to equal `msk * H(pk, app_id)` in the clear. Because `(Y, C)` is broadcast on-chain, the confidential derived key is directly disclosed to any observer.

### Finding Description

The CKD protocol is designed as an ElGamal encryption of the BLS signature `msk * H(pk, app_id)`:

- Each node computes `C_i = S_i + y_i * A` where `S_i = x_i * H(pk, app_id)` and `A` is the caller-supplied `app_pk`.
- The coordinator sums the Lagrange-weighted shares to produce `C = msk * H(pk, app_id) + a * Y`.
- Confidentiality holds because only the holder of the private key `a` (where `A = a * G1`) can compute `C - a * Y = msk * H(pk, app_id)`.

The root cause is in `compute_signature_share`:

```rust
// C <- S + y . A
let big_c = big_s + app_pk * y.0;   // line 174 — no check that app_pk != identity
``` [1](#0-0) 

If `app_pk` is the identity element of G1, then `app_pk * y.0 = G1::identity()`, so `big_c = big_s = hash_point * private_share`. After Lagrange-weighted summation by the coordinator:

```
C = Σ λ_i * C_i = Σ λ_i * S_i = msk * H(pk, app_id)
```

The output `(Y, C)` is sent on-chain. `C` is now the raw confidential key with no masking. Any observer of the on-chain state can read it directly.

The `ckd()` entry point performs no validation of `app_pk`:

```rust
pub fn ckd(
    participants: &[Participant],
    coordinator: Participant,
    me: Participant,
    key_pair: KeygenOutput,
    app_id: impl Into<AppId>,
    app_pk: PublicKey,          // ← accepted without any check
    rng: impl CryptoRngCore + Send + 'static,
) -> Result<impl Protocol<Output = CKDOutputOption>, InitializationError> {
    // only checks: participant count, duplicates, self-presence, coordinator-presence
    // NO check: app_pk is not identity, not low-order, is on curve
``` [2](#0-1) 

The documentation explicitly states the MPC contract does not validate `app_pk`; that responsibility is delegated to the developer contract:

> "Notice the MPC contract does not verify remote attestation, nor does it impose how the authentication of the TEE app is enforced." [3](#0-2) 

The library itself is therefore the necessary vulnerable step: it provides no defense-in-depth against a malformed `app_pk`.

### Impact Explanation

**Critical — Extraction/disclosure of a confidential derived secret.**

An attacker who submits `app_pk = G1::identity()` receives `(Y, C)` on-chain where `C = msk * H(pk, app_id)` in the clear. This is the exact value the CKD protocol is designed to keep secret. The attacker can then derive the application secret `s = HKDF(C)` without ever possessing the private key `a`. The master secret key `msk` itself is not directly exposed, but the per-app derived secret — the primary output the protocol is designed to protect — is fully disclosed.

This maps directly to the allowed impact: *"Extraction, reconstruction, or disclosure of... confidential derived secrets."*

### Likelihood Explanation

Any party that can submit a CKD request to the MPC contract controls `app_pk`. The `app_id` is authenticated by the blockchain (it equals the caller's account ID), but `app_pk` is a free parameter. A malicious developer contract, or any legitimate developer who understands the protocol, can submit `G1Projective::identity()` as `app_pk`. No privileged access, leaked keys, or cryptographic breaks are required. The `blstrs::G1Projective` type natively represents the identity element; it is a valid Rust value that compiles and runs without error.

### Recommendation

Add an explicit check in `ckd()` (or at the top of `compute_signature_share`) that rejects `app_pk` if it is the identity element or fails a subgroup/curve-membership check:

```rust
use elliptic_curve::Group;

pub fn ckd(
    ...
    app_pk: PublicKey,
    ...
) -> Result<impl Protocol<Output = CKDOutputOption>, InitializationError> {
    // Reject identity and low-order points
    if bool::from(app_pk.is_identity()) {
        return Err(InitializationError::BadParameters(
            "app_pk must not be the identity element".to_string(),
        ));
    }
    ...
}
```

Additionally, consider checking `app_pk.is_on_curve()` and `app_pk.is_torsion_free()` (as is already done for the verifying key in `verify_signature`) to prevent small-subgroup attacks. [4](#0-3) 

### Proof of Concept

1. Attacker registers a developer contract with `app_id = "attacker_app"`.
2. Attacker calls `gen_app_private_key(A = G1::identity())` on the MPC contract.
3. Each MPC node calls `ckd(..., app_pk = G1::identity(), ...)`.
4. In `compute_signature_share`, `big_c = big_s + identity * y = big_s = x_i * H(pk, app_id)`.
5. Coordinator sums: `C = msk * H(pk, app_id)` — the unmasked confidential key.
6. `(Y, C)` is posted on-chain. Attacker reads `C` directly.
7. Attacker computes `s = HKDF(C)` — the application secret — without knowing any private key.

/// BLS signature verification
/// following the standard in <https://www.ietf.org/archive/id/draft-irtf-cfrg-bls-signature-05.html#name-coreverify>
pub fn verify_signature(
    verifying_key: &VerifyingKey,
    msg: &[u8],
    signature: &Signature,
) -> Result<(), frost_core::Error<BLS12381SHA256>> {
    let element1: G1Affine = signature.into();
    if (!element1.is_on_curve() | !element1.is_torsion_free() | element1.is_identity()).into() {
        return Err(frost_core::Error::InvalidSignature);
    }
    let element2: G2Affine = verifying_key.to_element().into();
    if (!element2.is_on_curve() | !element2.is_torsion_free() | element2.is_identity()).into() {
        return Err(frost_core::Error::MalformedVerifyingKey);
    }

    // Concatenate the master public key (96 bytes) in the hash computation
    // H(pk || app_id) when H is a random oracle
    let base1 = hash_app_id_with_pk(verifying_key, msg).into();
    let base2 =
        <<BLS12381SHA256 as frost_core::Ciphersuite>::Group as frost_core::Group>::generator()
            .into();

    if blstrs::pairing(&base1, &element2).eq(&blstrs::pairing(&element1, &base2)) {
        Ok(())
    } else {
        Err(frost_core::Error::InvalidSignature)
    }
}
