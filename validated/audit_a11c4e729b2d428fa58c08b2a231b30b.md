### Title
Signature Verification Bypass via `skip_stateful_validations` Allows Unauthorized Invoke Transactions into Mempool — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator unconditionally skips the `__validate__` entry point (account signature verification) for any invoke transaction with `nonce=1` when `account_tx_in_pool_or_recent_block` returns `true` for the sender address. Because this check returns `true` whenever **any** transaction from that address is in the mempool — including a victim's pending `deploy_account` — an unprivileged attacker can submit an invoke with an arbitrary/invalid signature for any account that has a pending deploy-account transaction, bypassing signature verification entirely and getting the invalid transaction admitted to the mempool.

---

### Finding Description

**Root cause — `skip_stateful_validations`:** [1](#0-0) 

The function returns `true` (skip validation) when all three conditions hold:
1. The transaction is `Invoke` with `nonce == Nonce(Felt::ONE)`
2. The on-chain `account_nonce == Nonce(Felt::ZERO)` (account not yet deployed)
3. `account_tx_in_pool_or_recent_block(sender_address)` returns `true`

**Condition 3 is satisfied by any transaction in the pool for that address**, including a victim's `deploy_account`: [2](#0-1) 

`tx_pool.contains_account(address)` returns `true` as soon as the victim's `deploy_account` enters the pool.

**Effect of `skip_validate = true`:**

`run_validate_entry_point` sets `execution_flags.validate = !skip_validate = false`: [3](#0-2) 

Inside `StatefulValidator::perform_validations`, when `execution_flags.validate` is `false`, the function returns immediately after `perform_pre_validation_stage` without ever calling `__validate__`: [4](#0-3) 

The account's `__validate__` entry point — which performs signature verification — is **never executed** at the gateway level.

**Remaining checks do not catch an invalid signature:**

`perform_pre_validation_stage` only checks nonce (with `strict_nonce_check=false`, so `0 ≤ 1` passes) and fee bounds (passes if the account is funded, which it must be for the deploy-account to succeed): [5](#0-4) 

The stateless validator checks signature **size** but not signature **validity**: [6](#0-5) 

The mempool's `validate_tx` only checks for duplicate hashes and fee escalation — not signature correctness: [7](#0-6) 

**Ordering in `run_pre_validation_checks`:**

`validate_by_mempool` runs before `skip_stateful_validations`, so if no prior tx with `nonce=1` exists for the target address, the attacker's invalid invoke passes the duplicate check and then skips `__validate__`: [8](#0-7) 

---

### Impact Explanation

An unprivileged attacker can submit an invoke transaction with an **invalid/arbitrary signature** for any account that has a pending `deploy_account` transaction in the mempool. The invalid transaction is admitted to the mempool without signature verification. This:

- **Blocks the victim's legitimate `invoke(nonce=1)`** with a `DuplicateNonce` error (or forces the victim to pay a higher fee to replace it via fee escalation).
- **Wastes block space** when the invalid invoke is eventually executed and fails (since `__validate__` is called during block execution with `validate=true`).
- Enables a **persistent griefing attack**: the attacker can repeatedly replace the victim's valid invoke with an invalid one, indefinitely delaying the victim's post-deploy transaction.

Matching impact: **High — Mempool/gateway admission accepts invalid transactions (unverified signatures) before sequencing.**

---

### Likelihood Explanation

The attack requires only:
1. Observing a `deploy_account` transaction for address X in the mempool (public information).
2. Submitting an `invoke(nonce=1, sender=X)` with an arbitrary signature before the victim's invoke.

No privileged access, special keys, or complex setup is needed. Any network participant can execute this attack against any account in the deploy-account + invoke UX flow.

---

### Recommendation

1. **Restrict the skip condition to deploy-account transactions only**: In `skip_stateful_validations`, instead of checking `account_tx_in_pool_or_recent_block` (which matches any transaction type), verify that the account specifically has a **pending `deploy_account` transaction** in the mempool. This requires a new mempool query such as `has_pending_deploy_account(address)`.

2. **Alternatively, always run `__validate__` at the gateway**: Remove the `skip_stateful_validations` bypass entirely and rely on the blockifier's `strict_nonce_check=false` to handle the deploy-account + invoke UX case. The invoke will be re-validated during block execution anyway.

3. **At minimum, add a `max_nonce_for_validation_skip` guard in the Rust gateway**: The Python interface (`PyValidator`) already uses `max_nonce_for_validation_skip` to bound the skip window, but the Rust gateway's `skip_stateful_validations` hardcodes `nonce == Nonce(Felt::ONE)` with no configurable limit.

    sender_address: X,
    nonce: Nonce(Felt::ONE),
    signature: TransactionSignature(vec![Felt::from(0xdeadbeef_u64)].into()), // invalid
    resource_bounds: /* valid, above threshold */,
    calldata: /* arbitrary */,
    ..
};

// Gateway flow:
// 1. stateless_tx_validator.validate(&attacker_invoke) -> Ok(())  [size checks pass]
// 2. extract_state_nonce_and_run_validations:
//    - account_nonce = get_nonce_from_state(X) = 0  [X not deployed]
//    - run_pre_validation_checks:
//      - validate_state_preconditions: nonce 0<=1 passes, resource bounds pass
//      - validate_by_mempool: no duplicate nonce=1 for X yet -> Ok(())
//      - skip_stateful_validations:
//          nonce==1 && account_nonce==0 && account_tx_in_pool_or_recent_block(X)==true
//          -> returns true (SKIP __validate__)
//    - run_validate_entry_point(skip_validate=true):
//          execution_flags.validate = false
//          perform_pre_validation_stage: nonce ok, fee bounds ok (X is funded)
//          __validate__ is NOT called  <-- signature never verified
// 3. mempool.add_tx(attacker_invoke) -> Ok(())

// Result: attacker's invalid-signature invoke is in the mempool.
// Victim's invoke(nonce=1) is now rejected with DuplicateNonce.
// During block execution, attacker's invoke fails (__validate__ called with validate=true).
// Victim must resubmit (possibly with higher fee to replace via fee escalation).

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}

        state: &mut S,
