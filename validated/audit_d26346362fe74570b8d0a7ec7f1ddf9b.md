### Title
Missing Time Limit in COA Ownership Proof Signature Allows Indefinite Replay - (File: `fvm/evm/stdlib/contract.cdc`)

---

### Summary

`EVM.validateCOAOwnershipProof` in `fvm/evm/stdlib/contract.cdc` verifies Cadence account signatures over an opaque `signedData: [UInt8]` blob with no timestamp, block-height, or nonce enforcement. The `COAOwnershipProof` struct (`fvm/evm/types/proof.go`) likewise carries no expiry field. A proof signed once is cryptographically valid forever, giving any EVM contract that calls the `verifyCOAOwnershipProof` precompile (or the COA's ERC-1271 `isValidSignature`) the ability to replay that proof indefinitely.

---

### Finding Description

`validateCOAOwnershipProof` accepts five caller-supplied arguments and verifies signatures over the raw `signedData` bytes:

```cadence
fun validateCOAOwnershipProof(
    address: Address,
    path: PublicPath,
    signedData: [UInt8],      // opaque; no timestamp enforced
    keyIndices: [UInt64],
    signatures: [[UInt8]],
    evmAddress: [UInt8; 20]
): ValidationResult
```

The only checks performed are:
1. Array-length parity between `keyIndices` and `signatures`.
2. Key revocation status.
3. Cryptographic signature validity over `signedData`.
4. EVM address match.

No block height, wall-clock timestamp, or nonce is required inside `signedData`. The `COAOwnershipProof` struct has no expiry field:

```go
type COAOwnershipProof struct {
    KeyIndices     KeyIndices
    Address        FlowAddress
    CapabilityPath PublicPath
    Signatures     Signatures   // no Deadline / Nonce
}
```

The ERC-1271 entry point in `coa.sol` passes the caller-supplied `_hash` directly as `signedData`:

```solidity
function isValidSignature(bytes32 _hash, bytes memory _sig)
    external view virtual returns (bytes4) {
    (bool ok, bytes memory data) = cadenceArch.staticcall(
        abi.encodeWithSignature(
            "verifyCOAOwnershipProof(address,bytes32,bytes)",
            address(this), _hash, _sig));
    ...
}
```

The protocol's own comment acknowledges the absence of replay protection:

> *"Note: this function does not enforce that `signedData` includes `evmAddress`. In principle, a signature produced for one purpose could be replayed here against a different COA owned by the same Cadence account."*

The same structural gap applies to time: nothing prevents a proof signed at block N from being accepted at block N+1,000,000.

---

### Impact Explanation

Any EVM contract that relies on ERC-1271 (e.g., a lending protocol, an NFT marketplace, a permit-based DEX) can call `isValidSignature` on a COA at any future time. If the `_hash` the user originally signed does not embed a deadline, the proof is permanently valid. Concretely:

- A user signs a COA ownership proof authorising a specific on-chain action (e.g., a limit order, a collateral pledge, a permit).
- Market conditions change; the user wishes to revoke that authorisation.
- Because `validateCOAOwnershipProof` imposes no expiry, the only revocation mechanism is revoking the entire Flow account key — a drastic, account-wide action.
- An adversary (or a malicious/compromised EVM contract) can replay the captured proof to execute the stale action, potentially draining assets from the COA or triggering unfavourable financial terms.

Impact class: **cross-VM asset loss / unauthorised account action** via a stale, indefinitely-valid signature.

---

### Likelihood Explanation

- COAs are explicitly designed to act as ERC-1271 signers in EVM DeFi protocols.
- The `verifyCOAOwnershipProof` precompile is reachable by any unprivileged EVM transaction or script — no staked node or admin key is required.
- Off-chain proof capture is trivial: the proof is submitted in a public EVM transaction.
- DeFi protocols that do not embed a deadline in their EIP-712 hash (a common omission) are directly affected.

Likelihood: **Medium** — depends on COA adoption in DeFi, but the attack path is fully permissionless once a proof is captured.

---

### Recommendation

1. **Add a `deadline` (block height or timestamp) field to `COAOwnershipProof`** and include it in the signed payload.
2. **Enforce the deadline inside `validateCOAOwnershipProof`** before accepting the proof:

```cadence
// Proposed addition to validateCOAOwnershipProof
pre {
    getCurrentBlock().height <= proof.deadline :
        "EVM.validateCOAOwnershipProof(): Proof has expired."
}
```

3. **Add a per-account nonce** (analogous to `nonceOf[msg.sender]` in the Solidity recommendation) so that each proof can only be used once, preventing replay even within the validity window.

---

### Proof of Concept

**Setup**

1. Alice owns a Flow account with key index 0 and a COA stored at `/public/coa`.
2. A DeFi lending protocol on Flow EVM uses ERC-1271 to authorise collateral pledges. Its `_hash` is `keccak256(abi.encode(collateralAmount, borrowAmount, aliceCOA))` — no deadline.

**Attack**

1. Alice signs a COA ownership proof over `_hash` (representing today's favourable terms) and submits it to the protocol.
2. Collateral values drop; Alice decides not to proceed and does not submit a follow-up transaction.
3. An attacker captures Alice's proof from the mempool/chain history.
4. Six months later the attacker calls `lendingProtocol.executeLoan(aliceCOA, _hash, aliceProof)`.
5. The protocol calls `aliceCOA.isValidSignature(_hash, aliceProof)`.
6. `validateCOAOwnershipProof` is invoked; Alice's key is still active (not revoked); the proof is cryptographically valid; `isValid = true` is returned.
7. The protocol executes the loan under the stale, now-unfavourable terms, transferring assets out of Alice's COA.
