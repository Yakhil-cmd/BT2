### Title
Missing Lower-Bound Check on `threshold` in FROST Signing Validation Allows Threshold=1 — (`src/frost/mod.rs`)

---

### Summary

`assert_sign_inputs` and `presign` in `src/frost/mod.rs` accept any `threshold >= 1` without enforcing the same `threshold >= 2` lower bound that `assert_key_invariants` in `src/dkg.rs` enforces. A caller can pass `threshold = 1` to the FROST signing path, causing the protocol to run with a single-participant reconstruction, producing a cryptographically invalid (unusable) signature that honest parties accept as the protocol output.

---

### Finding Description

`assert_key_invariants` in `src/dkg.rs` explicitly rejects `threshold < 2`:

```rust
if threshold < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
}
``` [1](#0-0) 

The same lower-bound guard appears in `validate_triple_inputs` for OT-based triple generation:

```rust
if threshold_value < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold: threshold_value, min: 2 });
}
``` [2](#0-1) 

However, `assert_sign_inputs` — the validation entry point for FROST (EdDSA/RedJubjub) signing — only checks the upper bound:

```rust
if threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
``` [3](#0-2) 

There is no corresponding `threshold < 2` rejection. The `presign` function for FROST has the same gap:

```rust
if args.threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
``` [4](#0-3) 

`ReconstructionLowerBound` is a plain `usize` wrapper with no minimum enforced at the type level:

```rust
pub struct ReconstructionLowerBound(usize);
``` [5](#0-4) 

So `ReconstructionLowerBound::from(1usize)` is a valid value that passes all type-level and runtime checks in the FROST signing path.

---

### Impact Explanation

In FROST, the signing threshold determines how many participant shares are Lagrange-interpolated to reconstruct the group nonce and produce the aggregate signature. Keys are always generated with `threshold >= 2` (degree-≥1 polynomial). If a signing session is initiated with `threshold = 1`, the Lagrange interpolation uses only one share evaluated at a single point, reconstructing the wrong scalar. The resulting signature fails standard Ed25519/RedJubjub verification. Honest participants complete the protocol and receive this output as the final signature — a **corrupted, unusable signing output** that they cannot distinguish from a valid one at the protocol layer.

**Impact class:** High — Corruption of sign outputs so honest parties accept unusable cryptographic outputs.

---

### Likelihood Explanation

The entry path is a direct library API call. Any caller of the FROST signing or presigning functions (e.g., a malicious coordinator or a misconfigured integration) can pass `threshold = 1` as a `usize` that converts to `ReconstructionLowerBound` via the `From` impl. No special privileges, leaked keys, or cryptographic breaks are required. The inconsistency with DKG's own `threshold >= 2` guard makes this an easy mistake for integrators to miss.

---

### Recommendation

Add the same lower-bound guard to `assert_sign_inputs` and `presign` in `src/frost/mod.rs`:

```rust
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

This mirrors the existing check in `assert_key_invariants` and `validate_triple_inputs`, making the invariant consistent across all protocol entry points. Alternatively, encode the minimum at the `ReconstructionLowerBound` type level (e.g., a constructor that rejects values < 2) so the invariant cannot be violated anywhere.

---

### Proof of Concept

1. Run DKG with `participants = [P0, P1]`, `threshold = 2` — succeeds, produces valid key shares.
2. Call `assert_sign_inputs(&[P0, P1], 1usize, P0, P0)` — **passes** (1 ≤ 2, no lower-bound check).
3. Initiate FROST signing with `threshold = 1` and only P0's share.
4. The Lagrange coefficient for a single point is 1, so P0's share is used directly as the reconstructed secret — wrong value (the actual secret is split across a degree-1 polynomial).
5. The aggregate signature is produced and returned to all honest parties as the protocol output.
6. Verification against the public key fails — the output is a corrupted, unusable signature. [6](#0-5) [7](#0-6)
