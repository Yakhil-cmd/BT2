### Title
Silently Discarded `Result` from EIP-7685 Request Processors Produces Incorrect `requests_hash` in Block Header — (`basic_bootloader/src/bootloader/block_flow/ethereum/post_tx_op_sequencing.rs`)

---

### Summary

In `post_tx_op_sequencing.rs`, the return values of three critical block-finalization functions are unconditionally discarded with `let _ =`. If any of these functions fail **after** they have already written a partial prefix into the shared `requests_hasher`, the SHA-256 state is left in a corrupted intermediate state. The final `requests_hash` committed to the sealed block header is then incorrect, causing a state-transition divergence between the ZKsync OS execution trace and the expected block commitment.

---

### Finding Description

In the `post_op` function of `EthereumPostOp`, three EIP-7685 request-processing functions are called and their `Result` values are silently dropped:

```rust
// Environment may have no such contracts predeployed for tests or sequencing purposes
let _ = eip6110_events_parser(&system, &mut requests_hasher);
let _ = eip7002_system_part(&mut system, &mut requests_hasher);
let _ = eip7251_system_part(&mut system, &mut requests_hasher);

let requests_hash = Bytes32::from_array(requests_hasher.finalize().into());
``` [1](#0-0) 

All three functions share the same `requests_hasher: &mut impl crypto::sha256::Digest` and return `Result<bool, SystemError>`. The critical failure mode is **partial hasher mutation before an error return**:

**`eip7002_system_part`**: After confirming the contract is deployed, it writes the type byte `WITHDRAWAL_REQUEST_EIP_7685_TYPE` (= `0x01`) into the hasher unconditionally before reading individual queue slots:

```rust
requests_hasher.update([WITHDRAWAL_REQUEST_EIP_7685_TYPE]);
// ... then reads per-slot storage, each of which can return Err via `?`
``` [2](#0-1) 

**`eip7251_system_part`**: Identical pattern — writes `CONSOLIDATION_REQUEST_EIP_7685_TYPE` (= `0x02`) before per-slot reads that can fail:

```rust
requests_hasher.update([CONSOLIDATION_REQUEST_EIP_7685_TYPE]);
// ... then reads per-slot storage, each of which can return Err via `?`
``` [3](#0-2) 

If a storage read fails mid-loop (returning `Err` via `?`), the function exits with the hasher in a partially-updated state. The `let _ =` in the caller discards the `Err`, and `requests_hasher.finalize()` is called on the corrupted state. The resulting `requests_hash` is then committed to the block header:

```rust
result_keeper.record_sealed_block(metadata.block_level.header);
``` [4](#0-3) 

---

### Impact Explanation

The `requests_hash` field in the sealed block header is a SHA-256 commitment over all EIP-7685 requests (deposits, withdrawals, consolidations) processed in the block. An incorrect `requests_hash` means:

1. **State-transition divergence**: The committed block header does not match the actual requests processed. Any verifier or consensus layer checking the `requests_hash` against independently computed values will detect a mismatch.
2. **Forward/proving divergence**: The ZKsync OS proof system must prove the correctness of the block header. A `requests_hash` derived from a partially-updated SHA-256 state is unprovable against the correct expected value, causing proof generation failure or an invalid proof.
3. **Silent corruption**: Because the error is swallowed, no error is surfaced to the block producer. The block is sealed and submitted with a silently wrong header field.

---

### Likelihood Explanation

The predeploy contracts for EIP-7002 and EIP-7251 are expected to be deployed in production Ethereum-compatible environments. Once deployed, any storage read failure (e.g., oracle I/O error, internal system error) occurring after the type byte has been written to the hasher but before all queue slots are read will trigger this path. Additionally, any user transaction that enqueues a withdrawal or consolidation request (by calling the predeploy contracts) increases the queue depth, increasing the number of per-slot storage reads and thus the window for a mid-loop failure. The attacker-controlled entry path is: submit a transaction that calls the EIP-7002 or EIP-7251 predeploy contract to enqueue a request, then rely on any system-level I/O error during block finalization to trigger the partial-hash corruption.

---

### Recommendation

Replace the `let _ =` pattern with proper error propagation. If the intent is to tolerate missing predeploy contracts, distinguish that specific error case explicitly and propagate all other errors:

```rust
// Only ignore "contract not deployed" errors; propagate all others
match eip7002_system_part(&mut system, &mut requests_hasher) {
    Ok(_) => {},
    Err(SystemError::LeafDefect(_)) => {}, // contract not deployed, expected in some envs
    Err(e) => return Err(wrap_error!(e)),
}
```

This ensures that partial hasher mutations caused by unexpected errors are never silently committed to the block header.

---

### Proof of Concept

1. Deploy the EIP-7002 withdrawal predeploy contract at `WITHDRAWAL_REQUEST_PREDEPLOY_ADDRESS` with nonce=1 and non-empty bytecode.
2. Submit a transaction that calls the predeploy contract to enqueue at least one withdrawal request (populating queue storage slots).
3. Arrange for a storage read to fail during `eip7002_system_part` after the type byte `0x01` has been written to `requests_hasher` (e.g., by triggering an oracle I/O error on the slot read at line 188–195 of `eip_7002_withdrawal_contract/mod.rs`).
4. Observe that `post_op` completes without error, `requests_hash` is finalized from a SHA-256 state containing only the `0x01` prefix byte (not the full request data), and `record_sealed_block` commits this incorrect hash to the block header.
5. Any independent computation of `requests_hash` from the actual queue contents will produce a different value, demonstrating the divergence. [1](#0-0) [5](#0-4) [6](#0-5)

```rust
pub fn eip7002_system_part<S: EthereumLikeTypes>(
    system: &mut System<S>,
    requests_hasher: &mut impl crypto::sha256::Digest,
) -> Result<bool, SystemError>
where
    S::IO: IOSubsystemExt,
{
    let mut resources = S::Resources::from_native(
        <S::Resources as Resources>::Native::from_computational(u64::MAX),
    );

    let props = resources.with_infinite_ergs(|resources| {
        system.io.read_account_properties(
            ExecutionEnvironmentType::NoEE,
            resources,
            &WITHDRAWAL_REQUEST_PREDEPLOY_ADDRESS,
            AccountDataRequest::empty()
                .with_nonce()
                .with_observable_bytecode_len(),
        )
    })?;

    let is_contract = props.nonce.0 == 1 && props.observable_bytecode_len.0 > 0;
    if is_contract == false {
        return Err(SystemError::LeafDefect(internal_error!(
            "EIP-7002 withdrawal contract is not deployed"
        )));
    }
```
