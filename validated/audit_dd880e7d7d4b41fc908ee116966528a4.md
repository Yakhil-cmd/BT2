### Title
`MsgTurnBridge` Handler Is a No-Op, Permanently Disabling the Bridge Emergency-Disable Mechanism - (File: `x/cronos/keeper/msg_server.go`)

### Summary
The `TurnBridge` message handler in `x/cronos/keeper/msg_server.go` returns `nil, nil` without performing any state change or permission check. The bridge emergency-disable mechanism is completely non-functional: any caller receives a success response, no bridge-disabled state is ever written, and the bridge cannot be halted under any circumstances.

### Finding Description
`MsgTurnBridge` is a proto-defined transaction surface intended to allow a permissioned address to enable or disable the Cronos gravity bridge in an emergency. The CLI command `CmdTurnBridge` and the proto definition both exist and are wired up correctly. However, the actual server-side implementation is:

```go
// TurnBridge implements the grpc method
func (k msgServer) TurnBridge(goCtx context.Context, msg *types.MsgTurnBridge) (*types.MsgTurnBridgeResponse, error) {
    return nil, nil
}
``` [1](#0-0) 

Two failures are present simultaneously:

1. **No permission check.** Every other privileged handler in the same file checks `HasPermission` or compares `msg.From` against the admin. `TurnBridge` skips this entirely, so any unprivileged address can submit the message and receive a success response. [2](#0-1) 

2. **No state write.** The function returns immediately without writing a bridge-enabled/disabled flag to the KV store. The `CanTurnBridge` permission constant exists in the permissions system, and the integration test explicitly expects that calling `turn_bridge("false")` from an unpermissioned address fails and that bridge operations subsequently fail — but neither outcome is achievable because the handler is a no-op. [3](#0-2) 

The EVM hooks registered in `app.go` (`SendToIbcHandler`, `SendToIbcV2Handler`, `SendCroToIbcHandler`, `SendToAccountHandler`) have no bridge-enabled guard because there is no bridge-enabled state to read. [4](#0-3) 

### Impact Explanation
**High — Bypass of bridge authorization/control.** The bridge emergency-disable mechanism is permanently broken. In an emergency (e.g., an exploit draining CRC20/CRC21 assets through the bridge), the `EMERGENCY_ADMIN` or any permissioned address cannot halt bridge operations via `MsgTurnBridge`. Bridge-related asset movements (IBC transfers, gravity bridge sends) continue unimpeded regardless of any administrative action. This directly matches the allowed High impact: *"Bypass of Cronos admin, governance authority, permission, token-mapping, bridge, block-list, or module-account authorization checks."*

### Likelihood Explanation
Certain. The no-op is unconditional — every invocation of `MsgTurnBridge` silently succeeds and changes nothing. There is no code path through which the bridge can be disabled.

### Recommendation
Implement `TurnBridge` to:
1. Verify the caller holds `CanTurnBridge` permission via `k.HasPermission`.
2. Persist the bridge-enabled flag to the KV store (e.g., under a dedicated key).
3. Add a bridge-enabled guard at the top of each EVM log handler (`SendToIbcHandler`, `SendToIbcV2Handler`, `SendCroToIbcHandler`, `SendToAccountHandler`) that reads the flag and returns an error when the bridge is disabled.

### Proof of Concept
1. Attacker begins draining funds via `send_to_evm_chain` / `send_to_ibc` on a CRC21 contract.
2. Admin submits `MsgTurnBridge{sender: admin, enable: false}`.
3. The handler returns `nil, nil` — success — without writing any state.
4. The EVM hooks have no bridge-disabled flag to check; all bridge operations continue normally.
5. The attacker continues draining assets unimpeded. [1](#0-0) [5](#0-4)
