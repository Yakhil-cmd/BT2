### Title
Gateway L2 gas price admission check uses stale `l2_gas_price` instead of `next_l2_gas_price`, causing admission/execution inconsistency — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The gateway's `validate_resource_bounds` validates a transaction's `max_l2_gas_price` against the **current block's** `l2_gas_price`, but the batcher executes transactions using the **next block's** gas price (stored as `next_l2_gas_price` in the block header and tracked as `context.l2_gas_price` in the consensus orchestrator). This is the direct Sequencer analog of the WLFI `getVotes`/`getPastVotes` inconsistency: the admission check uses a stale snapshot value while execution uses the live EIP-1559-adjusted value. When gas prices are rising, transactions are admitted that will fail at execution; when gas prices are falling, valid transactions are rejected at the gate.

---

### Finding Description

In `StatefulTransactionValidator::validate_resource_bounds`, the gateway reads `l2_gas_price` from the previous block's `BlockInfo`:

```rust
// TODO(Arni): getnext_l2_gas_price from the block header.
let previous_block_l2_gas_price = self
    .gateway_fixed_block_state_reader
    .get_block_info()
    .await?
    .gas_prices
    .strk_gas_prices
    .l2_gas_price;
self.validate_tx_l2_gas_price_within_threshold(
    executable_tx.resource_bounds(),
    previous_block_l2_gas_price,
)?;
``` [1](#0-0) 

The `GatewayFixedBlockSyncStateClient::get_block_info_from_sync_client` reads `block_header.l2_gas_price` but **silently drops** `block_header.next_l2_gas_price`: [2](#0-1) 

The block header carries both fields. `next_l2_gas_price` is the EIP-1559-computed price for the **next** block, stored in every committed block header: [3](#0-2) 

On the execution side, the consensus orchestrator maintains `context.l2_gas_price`, which is updated after every block via `update_l2_gas_price` → `calculate_next_l2_gas_price_for_fin`. This value equals `previous_block.next_l2_gas_price`: [4](#0-3) 

When the batcher proposes a block, it passes `args.l2_gas_price` (= `context.l2_gas_price`) as the block's gas price: [5](#0-4) 

Inside the batcher, `AccountTransaction::check_fee_bounds` enforces `tx.max_l2_gas_price >= block_info.gas_prices.l2_gas_price`, where `block_info.gas_prices.l2_gas_price` is the batcher's `context.l2_gas_price` = `previous_block.next_l2_gas_price`: [6](#0-5) 

The two values diverge whenever the EIP-1559 fee market adjusts the gas price (up to ±12.5% per block). The TODO comment in the gateway code explicitly acknowledges the bug: `// TODO(Arni): getnext_l2_gas_price from the block header.`

---

### Impact Explanation

**When gas prices are rising** (`next_l2_gas_price > l2_gas_price`):

A transaction with `max_l2_gas_price` in the range `[l2_gas_price, next_l2_gas_price)` passes the gateway's admission check but fails the batcher's `check_fee_bounds` at execution time with `MaxGasPriceTooLow`. The transaction is admitted to the mempool and consumes a nonce slot, but will never be successfully executed. This matches **"Mempool/gateway/RPC admission accepts invalid transactions before sequencing"** (High).

**When gas prices are falling** (`next_l2_gas_price < l2_gas_price`):

A transaction with `max_l2_gas_price` in the range `[next_l2_gas_price, l2_gas_price)` is rejected by the gateway even though it would succeed at execution. This matches **"Mempool/gateway/RPC admission rejects valid transactions before sequencing"** (High).

---

### Likelihood Explanation

The EIP-1559 fee market adjusts gas prices after every block based on gas usage. In a live network with variable load, `next_l2_gas_price` will differ from `l2_gas_price` in almost every block. The maximum divergence is ~12.5% per block (EIP-1559 cap). This is a persistent, unprivileged condition: any user submitting a transaction with `max_l2_gas_price` in the gap between the two values will trigger the inconsistency. No special privileges or adversarial setup are required.

---

### Recommendation

1. Extend `GatewayFixedBlockSyncStateClient::get_block_info_from_sync_client` to read `block_header.next_l2_gas_price` and expose it (either as a new field on `BlockInfo` or via a dedicated accessor on `GatewayFixedBlockStateReader`).
2. In `validate_resource_bounds`, replace `previous_block_l2_gas_price` with `previous_block.next_l2_gas_price` as the threshold. This aligns the gateway's admission check with the actual gas price the batcher will use for execution, closing the snapshot/live-value gap.

---
