### Title
Missing Refund Callback in `ExitToEthereum` Precompile on Failed `withdraw` Promise — (`engine-precompiles/src/native.rs`)

---

### Summary

The `ExitToEthereum` precompile burns user ERC-20 tokens (or locks ETH at the precompile address) and then fires a `withdraw` promise to the ETH connector. Unlike the `ExitToNear` precompile, which attaches a callback (`exit_to_near_precompile_callback`) to handle promise failure and re-mint burned tokens, `ExitToEthereum` uses a bare `PromiseArgs::Create` with no error-handling callback. If the `withdraw` promise fails for any reason, the user's funds are permanently destroyed with no refund path.

---

### Finding Description

**`ExitToNear` — has a refund callback:**

In `ExitToNear::run`, after building the transfer promise, the code constructs `ExitToNearPrecompileCallbackArgs` (which includes `RefundCallArgs`) and wraps the promise in a `PromiseArgs::Callback` pointing to `exit_to_near_precompile_callback`:

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs {
        base: transfer_promise,
        callback: PromiseCreateArgs {
            ...
            method: "exit_to_near_precompile_callback".to_string(),
            ...
        },
    })
};
``` [1](#0-0) 

The callback `exit_to_near_precompile_callback` checks whether the base promise succeeded; on failure it calls `engine::refund_on_error`, which either re-mints burned ERC-20 tokens or transfers ETH back from the precompile address to the user. [2](#0-1) 

**`ExitToEthereum` — no refund callback:**

In `ExitToEthereum::run`, after burning the user's ERC-20 tokens (or receiving ETH via `apparent_value`), the code builds only a bare `PromiseArgs::Create`:

```rust
let withdraw_promise = PromiseCreateArgs {
    target_account_id: nep141_address,
    method: "withdraw".to_string(),
    ...
};

let promise = borsh::to_vec(&PromiseArgs::Create(withdraw_promise)).unwrap();
``` [3](#0-2) 

There is no `PromiseArgs::Callback` wrapping this promise, no `exit_to_ethereum_precompile_callback` function exists anywhere in the codebase, and no `RefundCallArgs` is ever constructed for the Ethereum exit path. A grep for `exit_to_ethereum.*callback` returns zero results.

The ERC-20 burn happens inside `EvmErc20.withdrawToEthereum` / `EvmErc20V2.withdrawToEthereum` before the precompile is called:

```solidity
function withdrawToEthereum(address recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);   // <-- tokens destroyed here
    ...
    assembly {
        let res := call(gas(), 0xb0bd02f6..., 0, ...);  // ExitToEthereum precompile
    }
}
``` [4](#0-3) [5](#0-4) 

---

### Impact Explanation

**Critical — Permanent freezing / destruction of funds.**

- **ERC-20 path**: Tokens are burned on Aurora before the `withdraw` promise executes. If `withdraw` fails (connector paused, storage exhausted, connector-side error), the tokens are gone from Aurora's EVM state and never arrive on Ethereum. The user suffers a total, unrecoverable loss of the bridged amount.
- **ETH (base token) path**: ETH is transferred to the `exit_to_ethereum` precompile address (`0xb0bd02f6…`) during EVM execution. If `withdraw` fails, that ETH is permanently locked at the precompile address with no mechanism to recover it, because only `refund_on_error` (called from a callback) can move ETH out of that address. [6](#0-5) 

---

### Likelihood Explanation

The `withdraw` promise targets the ETH connector contract. Realistic failure conditions include:

1. **Connector paused**: An admin pauses the connector between the EVM burn and the async promise execution (NEAR's async model means these are separate receipts).
2. **Insufficient attached deposit**: The `withdraw` call attaches only 1 yoctoNEAR; if the connector's storage requirements change or the call panics for any reason, the promise fails.
3. **Connector upgrade / migration**: During a connector upgrade window, `withdraw` calls may be rejected.

Any of these conditions — which are realistic operational events — cause permanent fund loss for any user who called `withdrawToEthereum` in that window. The user has no way to know in advance that the promise will fail, and no recovery path exists after the fact.

---

### Recommendation

Mirror the `ExitToNear` pattern for `ExitToEthereum`:

1. Add a `RefundCallArgs` to the `ExitToEthereum` precompile output (analogous to `ExitToNearPrecompileCallbackArgs`).
2. Wrap the `withdraw_promise` in a `PromiseArgs::Callback` pointing to a new `exit_to_ethereum_precompile_callback` contract method.
3. In that callback, check `handler.promise_result(0)`; on failure, call `engine::refund_on_error` with the appropriate `RefundCallArgs` (re-mint ERC-20 tokens or return ETH from the precompile address to the original sender). [7](#0-6) [8](#0-7) 

---

### Proof of Concept

1. User holds 1000 units of a bridged ERC-20 token on Aurora.
2. User calls `EvmErc20.withdrawToEthereum(ethRecipient, 1000)`.
3. `_burn(user, 1000)` executes — ERC-20 balance is now 0.
4. The `ExitToEthereum` precompile fires `PromiseArgs::Create(withdraw_promise)` targeting the ETH connector's `withdraw` method.
5. Between step 3 and step 4's receipt execution, the ETH connector is paused (or any other failure occurs).
6. The `withdraw` promise fails. No callback exists to detect this failure.
7. User's 1000 ERC-20 tokens are permanently destroyed; no ETH arrives on Ethereum; no refund is issued on Aurora.
8. Contrast: the same scenario with `ExitToNear` would trigger `exit_to_near_precompile_callback`, detect the failure, and re-mint the tokens to the user. [9](#0-8) [10](#0-9)

pub fn exit_to_near_precompile_callback<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        env.assert_private_call()?;

        // This function should only be called as the callback of
        // exactly one promise.
        if handler.promise_results_count() != 1 {
            return Err(errors::ERR_PROMISE_COUNT.into());
        }

        let args: ExitToNearPrecompileCallbackArgs = io.read_input_borsh()?;

        let maybe_result = if let Some(PromiseResult::Successful(_)) = handler.promise_result(0) {
            if let Some(args) = args.transfer_near {
                let action = PromiseAction::Transfer {
                    amount: Yocto::new(args.amount),
                };
                let promise = PromiseBatchAction {
                    target_account_id: args.target_account_id,
                    actions: vec![action],
                };

                // Safety: this call is safe because it comes from the exit to near precompile, not users.
                // The call is to transfer the unwrapped wNEAR tokens.
                let promise_id = handler.promise_create_batch(&promise);
                handler.promise_return(promise_id);
            }

            None
        } else if let Some(args) = args.refund {
            // Exit call failed; need to refund tokens
            let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;

            if !refund_result.status.is_ok() {
                return Err(errors::ERR_REFUND_FAILURE.into());
            }

            Some(refund_result)
        } else {
            None
        };

        Ok(maybe_result)
    })
