### Title
Malicious Participant Can Corrupt CKD Output by Using Arbitrary `app_id` Without Cross-Participant Verification — (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

The CKD protocol's coordinator aggregates participant contributions without any mechanism to verify that each participant computed their share using the agreed-upon `app_id`. A single malicious participant can silently substitute a different `app_id` when computing their share, causing the coordinator to produce a corrupted `CKDOutput` that honest parties accept as valid protocol output.

---

### Finding Description

In `src/confidential_key_derivation/protocol.rs`, each participant independently computes their blinded signature share using a locally supplied `app_id`:

```
S_i = x_i · H(pk, app_id)
C_i = S_i + y_i · A
```

They then send `(λ_i · Y_i, λ_i · C_i)` privately to the coordinator. The coordinator in `do_ckd_coordinator` aggregates all received contributions:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
``` [1](#0-0) 

There is no commitment, zero-knowledge proof, or cross-participant broadcast that binds each participant's contribution to the session's agreed-upon `app_id`. The coordinator knows `app_id` and can compute `H(pk, app_id)`, but cannot verify that `S_i = x_i · H(pk, app_id)` without knowing `x_i`. The protocol provides no mechanism to detect a participant who computed `S_i' = x_i · H(pk, app_id')` for an attacker-chosen `app_id' ≠ app_id`.

The `compute_signature_share` function that each participant runs locally:

```rust
fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);
    let big_s = hash_point * private_share.to_scalar();
    let big_c = big_s + app_pk * y.0;
    ...
}
``` [2](#0-1) 

is called with the participant's own local `app_id` argument, with no subsequent proof sent to the coordinator that the correct `app_id` was used.

The `ckd()` public entry point accepts `app_id` as a caller-supplied parameter with no cross-participant consistency enforcement: [3](#0-2) 

---

### Impact Explanation

A malicious participant substitutes `app_id'` for the session's `app_id`. Their contribution `λ_j · C_j'` is computed over `H(pk, app_id')` instead of `H(pk, app_id)`. The coordinator sums all contributions:

```
C = Σ λ_i · C_i  (honest)  +  λ_j · C_j'  (malicious)
  = msk · H(pk, app_id) + (λ_j · x_j)(H(pk, app_id') − H(pk, app_id)) + a · Y
```

The resulting `CKDOutput` is not a valid ElGamal encryption of `msk · H(pk, app_id)`. When the TEE app unmasks it with `sig = C − a · Y`, it obtains a point that is not a valid BLS signature for `app_id` under `pk`. The CKD session is permanently corrupted for the requesting application.

This maps to the allowed impact: **High — Corruption of CKD outputs so honest parties accept inconsistent or unusable cryptographic outputs.**

---

### Likelihood Explanation

Any single malicious MPC node participating in a CKD session can trigger this. The attack requires only that the node call `ckd()` with a different `app_id` than the one agreed upon for the session. No cryptographic capability, leaked key, or external assumption is needed. The network layer authenticates the sender's identity but does not authenticate the content of the contribution relative to the session's `app_id`. [4](#0-3) 

The security model explicitly states MPC nodes are trusted to execute honestly, but the library provides no enforcement mechanism to detect deviation on `app_id`: [5](#0-4) 

---

### Recommendation

Introduce a session-binding commitment to `app_id` before participants send their contributions:

1. **Broadcast `app_id` hash at session start**: Each participant broadcasts `H(session_id, app_id)` in an initial round. The coordinator verifies all participants committed to the same `app_id` before accepting contributions.
2. **ZK proof of correct share**: Each participant sends a proof that their `(Y_i, C_i)` was computed using the committed `app_id` and their registered public share `x_i · G_2`.
3. **Alternatively**, treat `app_id` as a public session parameter that is reliably broadcast (via the existing `echo_broadcast` mechanism) to all participants before the computation round, so any deviation is detectable.

---

### Proof of Concept

**Setup**: 3 participants `{P1, P2, P3}`, threshold 2, coordinator = P1. Agreed `app_id = b"Near App"`.

**Attack**:
- P2 (malicious) calls `ckd(..., AppId::try_from(b"Evil App").unwrap(), app_pk, ...)` instead of `b"Near App"`.
- P2 computes `S_2' = x_2 · H(pk, b"Evil App")` and sends `(λ_2 · Y_2, λ_2 · C_2')` to P1.
- P1 and P3 compute honest contributions for `b"Near App"`.
- P1 aggregates: `C = λ_1·C_1 + λ_2·C_2' + λ_3·C_3`.
- P1 returns `CKDOutput { big_y: Y, big_c: C }`.

**Result**: The TEE app computes `sig = C − a·Y`. This is not `msk · H(pk, b"Near App")`. `verify_signature(&pk, b"Near App", &sig)` returns `Err(InvalidSignature)`. [6](#0-5) 

The CKD session for `b"Near App"` is permanently denied for the requesting application, achievable by any single malicious participant with no additional capability.

pub fn ckd(
    participants: &[Participant],
    coordinator: Participant,
    me: Participant,
    key_pair: KeygenOutput,
    app_id: impl Into<AppId>,
    app_pk: PublicKey,
    rng: impl CryptoRngCore + Send + 'static,
) -> Result<impl Protocol<Output = CKDOutputOption>, InitializationError> {
    // not enough participants
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }

    // kick out duplicates
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }

    let comms = Comms::new();
    let chan = comms.shared_channel();

    let fut = run_ckd_protocol(
        chan,
        coordinator,
        me,
        participants,
        key_pair,
        app_id.into(),
        app_pk,
        rng,
    );
    Ok(make_protocol(comms, fut))

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}

pub fn verify_signature(
    verifying_key: &VerifyingKey,
    msg: &[u8],
    signature: &Signature,
) -> Result<(), frost_core::Error<BLS12381SHA256>> {
    let element1: G1Affine = signature.into();
    if (!element1.is_on_curve() | !element1.is_torsion_free() | element1.is_identity()).into() {
        return Err(frost_core::Error::InvalidSignature);
    }
    let element2: G2Affine = verifying_key.to_element().into();
    if (!element2.is_on_curve() | !element2.is_torsion_free() | element2.is_identity()).into() {
        return Err(frost_core::Error::MalformedVerifyingKey);
    }

    // Concatenate the master public key (96 bytes) in the hash computation
    // H(pk || app_id) when H is a random oracle
    let base1 = hash_app_id_with_pk(verifying_key, msg).into();
    let base2 =
        <<BLS12381SHA256 as frost_core::Ciphersuite>::Group as frost_core::Group>::generator()
            .into();

    if blstrs::pairing(&base1, &element2).eq(&blstrs::pairing(&element1, &base2)) {
        Ok(())
    } else {
        Err(frost_core::Error::InvalidSignature)
    }
