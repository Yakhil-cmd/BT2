### Title
Missing Access Control on `update_rewards()` Allows Any Caller to Suppress Block Rewards - (File: `src/staking/staking.cairo`)

### Summary

`update_rewards()` is specified to be callable only by the Starkware sequencer, but the implementation contains no such access control check. Any unprivileged caller can invoke it with `disable_rewards: true`, which writes `last_reward_block` to the current block number without distributing any rewards. Because `last_reward_block` is a global singleton, this permanently blocks all reward distribution for that block.

### Finding Description

The protocol specification explicitly restricts `update_rewards` to the Starkware sequencer:

> **access control**: Only starkware sequencer. [1](#0-0) 

The implementation, however, only calls `general_prerequisites()`, which checks for pause state and a non-zero caller — no sequencer identity check is performed: [2](#0-1) 

After validating that the staker exists and is active, the function unconditionally writes `last_reward_block` to the current block number **before** checking `disable_rewards`: [3](#0-2) 

`last_reward_block` is a global storage variable (not per-staker). Once it is set to the current block, the guard at line 1454–1458 causes every subsequent call in the same block to revert with `REWARDS_ALREADY_UPDATED`, regardless of who calls it or what `disable_rewards` value they pass. [4](#0-3) 

### Impact Explanation

An attacker calls `update_rewards(any_valid_active_staker, true)` once per block. This:

1. Passes all validation (staker exists, has non-zero balance, block not yet processed).
2. Writes `last_reward_block = current_block`.
3. Returns immediately without distributing any rewards (because `disable_rewards = true`).

The legitimate sequencer call for the same block then reverts with `REWARDS_ALREADY_UPDATED`. All stakers and delegators receive **zero rewards** for that block. Repeated every block, this permanently freezes all unclaimed yield accrual for the entire protocol.

This matches the allowed impact: **Permanent freezing of unclaimed yield or unclaimed royalties**.

### Likelihood Explanation

The entry point is fully public — any non-zero address can call it. The only prerequisite is supplying a valid, active staker address with non-zero balance, which is trivially discoverable from on-chain events (`NewStaker`). The attack costs one transaction per block and requires no capital. It can be automated to run continuously.

### Recommendation

Add a sequencer-only access control check at the top of `update_rewards`, consistent with the specification. For example, using the existing roles component:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.roles.only_sequencer(); // enforce spec: "Only starkware sequencer"
    self.general_prerequisites();
    ...
}
```

Alternatively, restrict the caller to a known sequencer address stored in contract storage, similar to how `assert_caller_is_attestation_contract` is implemented for `update_rewards_from_attestation_contract`. [5](#0-4) 

### Proof of Concept

1. Staker A stakes and becomes active (K epochs pass so balance is non-zero).
2. Attacker (any address) calls `update_rewards(staker_A_address, true)` at block N.
3. `last_reward_block` is set to N; function returns without distributing rewards.
4. Sequencer attempts `update_rewards(staker_A_address, false)` at block N → reverts with `REWARDS_ALREADY_UPDATED`.
5. Staker A and all delegators earn zero rewards for block N.
6. Attacker repeats step 2 at every subsequent block.

The test suite itself demonstrates this exact behavior — calling `update_rewards` with `disable_rewards: true` followed by a second call in the same block panics with `REWARDS_ALREADY_UPDATED`: [6](#0-5)
