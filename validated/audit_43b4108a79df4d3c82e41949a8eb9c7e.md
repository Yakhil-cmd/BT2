### Title
Missing `range_check96`, `add_mod`, `mul_mod` Builtin Handling Causes `UnknownBuiltin` Error in RPC Simulation/Trace - (`crates/apollo_rpc_execution/src/objects.rs`)

---

### Summary

`vm_resources_to_execution_resources` in `objects.rs` does not handle the `range_check96`, `add_mod`, or `mul_mod` builtins. Any contract that uses these builtins will cause `starknet_simulateTransactions` and `starknet_traceTransaction` to return `ExecutionError::UnknownBuiltin` even though the underlying blockifier execution succeeded.

---

### Finding Description

The function `vm_resources_to_execution_resources` converts a blockifier `VmExecutionResources` into the RPC `ExecutionResources` type. Its `match` arm for builtin names explicitly handles eight builtins and silently skips `output`, but falls through to a catch-all that returns an error for anything else: [1](#0-0) 

The developer left a TODO comment acknowledging the gap:

```rust
// TODO(DanB): what about the following?
// BuiltinName::range_check96 => todo!(),
// BuiltinName::add_mod => todo!(),
// BuiltinName::mul_mod => todo!(),
_ => {
    return Err(ExecutionError::UnknownBuiltin { builtin_name });
}
```

These three builtins are fully supported by the Starknet VM and the blockifier. The `starknet_api` `Builtin` enum already defines `AddMod`, `MulMod`, and `RangeCheck96` variants: [2](#0-1) 

The blockifier also accounts for them in bouncer and versioned-constants logic, confirming they are live, executable builtins.

The error propagates through the full simulation call chain:

1. `simulate_transactions` (lib.rs:981) calls `execute_transactions`, which succeeds.
2. It then calls `trace_constructor(tx_execution_output.execution_info)` for each result. [3](#0-2) 

3. `get_trace_constructor` returns a closure that calls `execution_info.try_into()?`, which eventually calls `FunctionInvocation::try_from`. [4](#0-3) 

4. `FunctionInvocation::try_from` calls `vm_resources_to_execution_resources(call_info.resources.vm_resources, gas_vector)?`. [5](#0-4) 

5. If any call in the trace used `range_check96`, `add_mod`, or `mul_mod`, the `?` propagates `Err(ExecutionError::UnknownBuiltin)` all the way back to the RPC caller.

---

### Impact Explanation

`starknet_simulateTransactions` and `starknet_traceTransaction` return an error for any transaction whose execution trace includes a call that used `range_check96`, `add_mod`, or `mul_mod` builtins. The transaction itself executes correctly on-chain; only the RPC simulation/trace path is broken. This matches the **High** impact category: *RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value.*

---

### Likelihood Explanation

`range_check96` is used by the modular arithmetic opcodes introduced in Sierra ≥ 1.4 / CASM with `u96` range checks. Any contract compiled with a recent Sierra compiler that uses integer arithmetic or the `add_mod`/`mul_mod` opcodes will trigger this. The trigger requires no privileges — any user can deploy such a contract and call `starknet_simulateTransactions`.

---

### Recommendation

Add the three missing arms to the `match` in `vm_resources_to_execution_resources`:

```rust
BuiltinName::range_check96 => {
    builtin_instance_counter.insert(Builtin::RangeCheck96, count)
}
BuiltinName::add_mod => {
    builtin_instance_counter.insert(Builtin::AddMod, count)
}
BuiltinName::mul_mod => {
    builtin_instance_counter.insert(Builtin::MulMod, count)
}
```

The target `Builtin` variants already exist in `starknet_api::execution_resources::Builtin`. [2](#0-1) 

---

### Proof of Concept

```rust
#[test]
fn test_vm_resources_to_execution_resources_range_check96() {
    use cairo_vm::vm::runners::cairo_runner::ExecutionResources as VmExecutionResources;
    use cairo_vm::types::builtin_name::BuiltinName;
    use starknet_api::execution_resources::GasVector;
    use std::collections::HashMap;

    let mut counter = HashMap::new();
    counter.insert(BuiltinName::range_check96, 5usize);

    let vm_resources = VmExecutionResources {
        n_steps: 100,
        n_memory_holes: 0,
        builtin_instance_counter: counter,
    };

    // This currently returns Err(UnknownBuiltin { range_check96 })
    // but should return Ok with RangeCheck96 => 5 in the counter.
    let result = vm_resources_to_execution_resources(
        vm_resources,
        GasVector::default(),
    );
    assert!(result.is_ok(), "Expected Ok but got: {:?}", result);
    let resources = result.unwrap();
    assert_eq!(
        resources.builtin_instance_counter
            .get(&starknet_api::execution_resources::Builtin::RangeCheck96),
        Some(&5u64)
    );
}
```
