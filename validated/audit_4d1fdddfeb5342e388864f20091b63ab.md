### Title
Unbounded Loop in `calculate_rewards` Enables Permanent Freezing of Delegator Unclaimed Yield - (File: `src/pool/pool.cairo`)

---

### Summary

The `calculate_rewards` internal function in `src/pool/pool.cairo` contains an unbounded `while` loop that iterates over every entry in a pool member's `pool_member_epoch_balance` trace. The code itself acknowledges this: *"This loop is unbounded but unlikely to exceed gas limits."* A delegator who repeatedly modifies their delegation balance across many epochs will accumulate an arbitrarily large trace, causing any future `claim_rewards` call to run out of gas and permanently freeze their unclaimed yield.

---

### Finding Description

In `src/pool/pool.cairo`, the internal function `calculate_rewards` is responsible for computing a delegator's accumulated rewards between two checkpoints. It does so by iterating over every balance-change entry in the delegator's `pool_member_epoch_balance` trace: [1](#0-0) 

The loop has no upper bound on iterations — it runs once per entry in `pool_member_trace` up to `pool_member_trace_length`. Each call to `add_to_delegation_pool` (or any other operation that writes a new epoch-balance checkpoint for the pool member) appends one entry to this trace. Because there is no cap on how many times a delegator may change their balance, the trace can grow without limit.

When `claim_rewards` is eventually called, it invokes `calculate_rewards` with the full trace. If the trace is large enough, the transaction exhausts the Starknet gas limit and reverts. Because the checkpoint state is not advanced on a failed claim, every subsequent `claim_rewards` attempt will also revert, permanently locking the delegator's accumulated yield.

---

### Impact Explanation

**Permanent freezing of unclaimed yield** for the affected delegator. Once the trace is large enough that a single `claim_rewards` transaction cannot iterate through it within the block gas limit, the delegator has no alternative code path to recover their rewards. The funds remain in the pool contract but are inaccessible.

This matches the allowed impact: *"Permanent freezing of unclaimed yield or unclaimed royalties."*

---

### Likelihood Explanation

Any unprivileged delegator can trigger this condition on themselves (or an adversary can grief a victim delegator by front-running or social-engineering repeated small top-ups). The operations that grow the trace — `add_to_delegation_pool`, `enter_delegation_pool`, partial exits followed by re-entry — are all publicly callable with no rate limit. Starknet's per-transaction gas ceiling is finite and well-defined, so a sufficiently large trace will deterministically cause an out-of-gas revert.

---

### Recommendation

Replace the linear iteration with a checkpoint-based or paginated claim mechanism, or enforce a maximum number of unclaimed balance-change entries before a claim must be made. Alternatively, store a running cumulative reward value that is updated incrementally at each balance-change event, eliminating the need to replay the entire history at claim time. This mirrors the mathematical-formula fix suggested in the original ThrottledExit report: avoid iterating over a potentially unbounded range by computing the result directly from stored aggregates.

---

### Proof of Concept

1. Delegator calls `add_to_delegation_pool` (or `enter_delegation_pool`) once per epoch for `N` epochs without ever calling `claim_rewards`. Each call appends one entry to `pool_member_epoch_balance`.
2. After `N` epochs, the delegator calls `claim_rewards`.
3. `claim_rewards` calls `calculate_rewards`, which enters the `while entry_to_claim_from < pool_member_trace_length` loop.
4. The loop executes `N` iterations, each performing storage reads and arithmetic.
5. For sufficiently large `N` (bounded by Starknet's gas limit per transaction), the transaction reverts with out-of-gas.
6. All subsequent `claim_rewards` calls also revert because the trace has not shrunk and the checkpoint has not advanced.
7. The delegator's accumulated yield is permanently frozen in the pool contract. [2](#0-1)

        fn calculate_rewards(
            self: @ContractState,
            pool_member: ContractAddress,
            from_checkpoint: PoolMemberCheckpoint,
            until_checkpoint: PoolMemberCheckpoint,
            mut entry_to_claim_from: VecIndex,
        ) -> (Amount, VecIndex) {
            let pool_member_trace = self.pool_member_epoch_balance.entry(pool_member);
            // Note: `until_epoch` is the current epoch.
            let until_epoch = until_checkpoint.epoch();

            let mut rewards = 0;

            let pool_member_trace_length = pool_member_trace.length();

            let mut from_sigma = self.find_sigma(from_checkpoint, curr_epoch: until_epoch);
            let mut from_balance = from_checkpoint.balance();

            let base_value = self.staking_rewards_base_value.read();

            // **Note**: The loop iterates over the balance changes in the pool member's balance
            // trace. This loop is unbounded but unlikely to exceed gas limits.
            while entry_to_claim_from < pool_member_trace_length {
                let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
                // If the balance change is after `until_epoch` (and therefore does not affect
                // the current reward computation), exit the loop.
                if pool_member_checkpoint.epoch() >= until_epoch {
                    break;
                }

                // Compute rewards from (inclusive) the previous balance change (or from
                // `from_checkpoint`) to (exclusive) the current entry.
                let to_sigma = self.find_sigma(pool_member_checkpoint, curr_epoch: until_epoch);
                rewards +=
                    compute_rewards_rounded_down(
                        amount: from_balance, interest: to_sigma - from_sigma, :base_value,
                    );
                from_sigma = to_sigma;
                from_balance = pool_member_checkpoint.balance();
                entry_to_claim_from += 1;
            }

            // Compute the remaining rewards from (inclusive) the last visited balance change in
            // `pool_member_trace` (or from `from_checkpoint`) to (exclusive) `until_checkpoint`.
            let to_sigma = self.find_sigma(until_checkpoint, curr_epoch: until_epoch);
            rewards +=
                compute_rewards_rounded_down(
                    amount: from_balance, interest: to_sigma - from_sigma, :base_value,
                );

            (rewards, entry_to_claim_from)
        }
