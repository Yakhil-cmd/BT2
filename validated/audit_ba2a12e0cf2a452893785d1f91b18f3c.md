### Title
Incorrect Multi-NFT Royalty Calculation in WalletConnect Signing Confirmation Understates Total Spend - (File: packages/gui/src/electron/commands/parseCommandDisplay.ts)

### Summary
The `formatAmountWithRoyalties` function in the WalletConnect command display pipeline incorrectly calculates the total amount including royalties when an offer involves multiple NFTs. It divides the fungible payment amount equally among NFTs before applying each royalty percentage, producing a displayed total that is materially lower than what the wallet daemon will actually charge. A user shown this confirmation dialog approves a spend they believe costs less than it does.

### Finding Description

In `parseCommandDisplay.ts`, `formatAmountWithRoyalties` computes the royalty-inclusive total shown to the user in the WalletConnect signing confirmation:

```typescript
const splitAmount = amount / BigInt(royaltyPercentages.length);   // ← wrong
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
const totalAmount = amount + royaltyAmount;
``` [1](#0-0) 

The code first divides `amount` by the count of NFTs (`royaltyPercentages.length`), then applies each NFT's royalty percentage to that reduced `splitAmount`. The correct formula is to apply each NFT's royalty percentage to the **full** `amount`, because the Chia protocol's `nft_calculate_royalties` (called by the daemon) charges each NFT's royalty against the entire fungible payment, not a fractional share of it. [2](#0-1) 

The regular offer builder (`OfferBuilderProvider`) correctly delegates to the daemon's `nft_calculate_royalties` RPC for its display, so the bug is isolated to the WalletConnect signing confirmation path. [3](#0-2) 

### Impact Explanation

**Concrete numeric example** (matching the existing test fixture):

| | Displayed (buggy) | Actual on-chain |
|---|---|---|
| Amount | 0.0001 XCH | 0.0001 XCH |
| NFT-A royalty (5%) | 0.0000025 XCH | 0.000005 XCH |
| NFT-B royalty (0.1%) | 0.000000005 XCH | 0.00000001 XCH |
| **Total shown** | **≈ 0.00010255 XCH** | **≈ 0.00010510 XCH** | [4](#0-3) 

The understatement scales with the number of NFTs and their royalty percentages. For two NFTs each carrying a 10% royalty and a 1 XCH payment, the dialog shows 1.1 XCH while the daemon charges 1.2 XCH — a 0.1 XCH discrepancy. The user approves based on the lower figure; the daemon then executes the spend at the correct (higher) amount.

This satisfies: **High — Corruption of WalletConnect state that causes a user to approve the wrong amount.**

### Likelihood Explanation

Any WalletConnect-connected dApp can trigger this by issuing a `take_offer` or `create_offer_for_ids` command that bundles two or more NFTs with differing royalty percentages. No special privileges are required beyond an active WalletConnect session, which is a routine user action. The discrepancy grows with the number of NFTs and their royalty rates, making it easy to construct offers where the understatement is economically meaningful. [5](#0-4) 

### Recommendation

Replace the split-then-multiply formula with the correct per-NFT calculation applied to the full amount:

```typescript
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (amount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
```

This matches the semantics of the daemon's `nft_calculate_royalties` endpoint and the `OfferBuilderProvider` display path.

### Proof of Concept

1. Connect a WalletConnect dApp to the Chia GUI wallet.
2. From the dApp, issue a `chia_wallet.take_offer` command for an offer that bundles two NFTs — NFT-A with `royalty_percentage = 1000` (10%) and NFT-B with `royalty_percentage = 1000` (10%) — requesting 1 XCH.
3. Observe the WalletConnect signing confirmation dialog: `amountWithRoyalties` displays **1.1 XCH** (split: 0.5 XCH × 10% × 2 = 0.1 XCH royalty).
4. Approve the transaction.
5. The daemon charges **1.2 XCH** (1 XCH × 10% + 1 XCH × 10% = 0.2 XCH royalty), debiting 0.1 XCH more than the user was shown. [1](#0-0) [6](#0-5)

function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],
): string | undefined {
  if (royaltyPercentages.length === 0 || line.kind === 'nft') {
    return undefined;
  }

  const splitAmount = amount / BigInt(royaltyPercentages.length);
  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;

  if (line.kind === 'xch') {
    return mojoToChiaLocaleString(totalAmount);
  }

  return mojoToCATLocaleString(totalAmount);
}

function withRoyaltyTotals(
  items: DisplayWalletDeltaItemWithKey[],
  amounts: Record<string, bigint>,
  oppositeSideLines: DisplayWalletDeltaItem[],
): DisplayWalletDeltaItem[] {
  const royaltyPercentages = royaltyPercentagesForSide(oppositeSideLines);

  return items.map(({ key, line }) => {
    if (line.kind === 'nft') {
      return line;
    }

    const amount = amounts[key];
    const amountWithRoyalties = amount ? formatAmountWithRoyalties(line, amount, royaltyPercentages) : undefined;

    return amountWithRoyalties ? { ...line, amountWithRoyalties } : line;
  });
}
