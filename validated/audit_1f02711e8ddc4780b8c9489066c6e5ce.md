### Title
Disabled Cheater Detection in FROST Signing Enables Unattributable Permanent Denial of Signing - (File: `src/frost/eddsa/sign.rs`, `src/frost/redjubjub/sign.rs`)

---

### Summary

The codebase uses a forked, commit-pinned version of the `reddsa` crate (from `https://github.com/near/reddsa`) specifically to disable the upstream "cheater-detection" feature. As a direct consequence, both the EdDSA and RedJubjub FROST signing coordinators call `aggregate()` with an empty `verifying_shares` map, bypassing per-share verification. A malicious participant can submit any invalid signature share, causing signing to fail with no attribution. Because the coordinator cannot identify the offending party, the malicious participant can repeat this attack indefinitely across every signing round in which they are included, permanently denying signing to honest parties.

---

### Finding Description

**Forked, pinned dependency with disabled security feature:**

`Cargo.toml` declares `reddsa` as a git dependency pinned to a single commit:

```toml
# This project has been forked due to incompatibility problems with cheater detection feature activated on the original Zcash repo
reddsa = { git = "https://github.com/near/reddsa", rev = "c7cd92a55f7399d8d7f8c0ac386445b5f898f197", ... }
``` [1](#0-0) 

The fork was created explicitly to remove the cheater-detection feature present in the upstream Zcash `reddsa` crate. The fork is pinned to a single commit and will not receive upstream security patches.

**Empty verifying-shares map in EdDSA aggregate (v1 and v2):**

In `do_sign_coordinator_v1` and `do_sign_coordinator_v2`, the `PublicKeyPackage` passed to `aggregate()` is constructed with an empty `BTreeMap`, skipping all per-share verification:

```rust
// We supply empty map as `verifying_shares` because we have disabled "cheater-detection" feature flag.
let public_key_package = PublicKeyPackage::new(BTreeMap::new(), vk_package);
let signature = aggregate(&signing_package, &signature_shares, &public_key_package)
    .map_err(|e| ProtocolError::AssertionFailed(e.to_string()))?;
``` [2](#0-1) [3](#0-2) 

**Same pattern in RedJubjub aggregate:**

```rust
// We use empty BTreeMap because "cheater-detection" feature is disabled
let pk_package = PublicKeyPackage::new(BTreeMap::new(), keygen_output.public_key);
let signature = aggregate(&signing_package, &signature_shares, &pk_package, &randomized_params)
    .map_err(|_| ProtocolError::ErrorFrostAggregation)?;
``` [4](#0-3) 

**Known advisory suppressed:**

`deny.toml` suppresses `RUSTSEC-2023-0089` (deprecated `atomic-polyfill`) introduced transitively through the pinned `reddsa` fork, confirming the fork is not being kept current with upstream: [5](#0-4)

### Impact Explanation

In FROST, `aggregate()` with a populated `verifying_shares` map verifies each participant's signature share individually against their known verifying share before combining them. With an empty map, this per-share check is entirely skipped. The only verification that remains is the final check of the aggregated signature against the group public key.

This means:
- A malicious participant can submit any value as their signature share (zero, random, or adversarially crafted).
- The coordinator collects all shares and calls `aggregate()`.
- The aggregate fails the final group-key check and returns an error.
- The coordinator receives no information about which participant's share was invalid.
- The malicious participant cannot be identified or excluded.
- Every subsequent signing round that includes the malicious participant will fail identically.

This constitutes **permanent denial of signing** for all honest parties in any signing configuration that includes the malicious participant. The developers' comment that cheater detection "doesn't bring any additional guarantees" is incorrect: it removes the only mechanism by which a misbehaving participant can be identified and excluded from future rounds.

**Mapped impact:** High — Permanent denial of signing for honest parties under valid protocol inputs.

---

### Likelihood Explanation

Any participant in a FROST signing session can trigger this. No cryptographic knowledge, leaked keys, or external compromise is required. The attacker only needs to be a legitimate participant (i.e., hold a valid key share from DKG) and send an arbitrary byte string as their round-2 signature share. The attack is repeatable at zero marginal cost per round and is undetectable by the coordinator.

---

### Recommendation

1. **Re-enable cheater detection:** Populate the `verifying_shares` map in `PublicKeyPackage` with the actual per-participant verifying shares derived from the DKG output. This allows `aggregate()` to identify and report the specific participant whose share is invalid.

2. **Rebase the `reddsa` fork:** The fork was created to work around an incompatibility with cheater detection. Resolving the incompatibility at the protocol level (recommendation 1) removes the need for the fork. The dependency should be updated to track the upstream Zcash `reddsa` crate to receive security patches.

3. **Implement exclusion logic:** Once cheater detection is re-enabled, the coordinator should use the identified malicious participant's identity to exclude them from future signing rounds and alert the application layer.

---

### Proof of Concept

1. A threshold signing session is initiated with participants `[P1, P2, ..., Pn]` where `Pmalicious` is one of the participants.
2. DKG completes normally; all participants hold valid key shares.
3. A signing round begins. The coordinator (`do_sign_coordinator_v1` or `do_sign_coordinator_v2`) collects commitments and sends the signing package.
4. `Pmalicious` receives the signing package but instead of computing `round2::sign(...)`, sends a random 32-byte value as its `SignatureShare`.
5. The coordinator collects all shares including the malformed one from `Pmalicious`.
6. `aggregate()` is called with `PublicKeyPackage::new(BTreeMap::new(), vk_package)` — per-share verification is skipped.
7. The aggregated value fails the final group-key check; `aggregate()` returns `Err(...)`.
8. The coordinator maps this to `ProtocolError::AssertionFailed` or `ProtocolError::ErrorFrostAggregation` with no participant attribution.
9. The signing round fails. The coordinator has no basis to exclude `Pmalicious`.
10. Steps 3–9 repeat for every subsequent signing attempt that includes `Pmalicious`, permanently blocking signing for all honest parties.
