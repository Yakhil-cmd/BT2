### Title
Missing Zero-Address Validation for `reward_address` Allows Permanent Loss of Unclaimed Yield - (File: src/staking/staking.cairo, src/pool/pool.cairo)

---

### Summary

Both `stake()` and `change_reward_address()` in `staking.cairo`, and `enter_delegation_pool()` and `change_reward_address()` in `pool.cairo`, accept `ContractAddress` zero (`0x0`) as a valid `reward_address` without any zero-address guard. When `claim_rewards()` is subsequently invoked, accumulated yield is transferred to the null address, permanently burning it.

---

### Finding Description

The external report's vulnerability class is **missing zero-address validation for a critical address parameter at initialization/mutation time**, causing the contract to become non-functional or funds to be irrecoverably lost.

The direct analog in this codebase is the `reward_address` parameter accepted by four entry points, none of which assert `reward_address != 0`:

**`stake()` — `src/staking/staking.cairo` lines 288–317**

```cairo
fn stake(
    ref self: ContractState,
    reward_address: ContractAddress,   // ← no zero check
    operational_address: ContractAddress,
    amount: Amount,

    self.general_prerequisites();
    ...
    assert!(
        !self.does_token_exist(token_address: reward_address),
        "{}",
        GenericError::REWARD_ADDRESS_IS_TOKEN,   // only token-address check, no zero check
    );
    ...
}
```

Once `reward_address` is stored as zero, `claim_rewards()` in `pool.cairo` (line 366) executes:

```cairo
reward_token.checked_transfer(recipient: reward_address, amount: rewards.into());
```

with `reward_address == 0x0`, sending all accumulated yield to the null address.

---

### Impact Explanation

**High — Permanent freezing / loss of unclaimed yield.**

When `claim_rewards()` is called with a zero `reward_address` stored:

- If the STRK ERC-20 `transfer` to `0x0` succeeds (no revert on zero recipient), all accumulated rewards are burned and irrecoverable.
- If the STRK ERC-20 reverts on a zero recipient, `claim_rewards()` will revert on every call, permanently freezing the yield until the staker/pool member calls `change_reward_address()` again — but by then any rewards already transferred in a prior successful call are gone.

Both outcomes map directly to **"Permanent freezing of unclaimed yield"** (High) in the allowed impact scope.

---

### Likelihood Explanation

**Low-to-Medium.** A staker or pool member can reach this state by:

1. Accidentally passing `0x0` as `reward_address` during `stake()` or `enter_delegation_pool()` (e.g., an off-by-one in a script, an uninitialized variable in a front-end integration).
2. Calling `change_reward_address(0x0)` intending to "clear" the field or during a migration script error.

No privileged role is required. Any unprivileged staker or delegator can trigger this on their own account. The protocol provides no on-chain guard to prevent it.

---

### Recommendation

Add an explicit zero-address assertion in every function that writes `reward_address`:

```cairo
assert!(reward_address.is_non_zero(), "Reward address cannot be zero");
```

This guard should be added to:
- `stake()` in `src/staking/staking.cairo`
- `change_reward_address()` in `src/staking/staking.cairo`
- `enter_delegation_pool()` in `src/pool/pool.cairo`
- `change_reward_address()` in `src/pool/pool.cairo`

---

### Proof of Concept

1. Deploy the system normally.
2. Call `stake(reward_address: 0x0, operational_address: <valid>, amount: <min_stake>)` from a funded account.
3. Advance epochs so rewards accrue.
4. Call `claim_rewards(staker_address: <staker>)`.
5. Observe that the STRK reward amount is transferred to `0x0` (burned) or the call reverts permanently, with no way for the protocol to recover the yield.

The same path applies to a pool member via `enter_delegation_pool(reward_address: 0x0, ...)` followed by `claim_rewards(pool_member: <member>)`.
