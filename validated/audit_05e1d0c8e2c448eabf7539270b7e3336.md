### Title
Unbounded Loop in `calculate_rewards` Can Permanently Freeze Pool Member's Unclaimed Yield — (File: `src/pool/pool.cairo`)

---

### Summary
The `calculate_rewards` function in `src/pool/pool.cairo` contains an explicitly acknowledged unbounded loop that iterates over a pool member's entire `pool_member_epoch_balance` trace since their last reward claim. A pool member who repeatedly changes their delegation balance across many epochs without claiming rewards will accumulate an unbounded number of trace entries. When they eventually call `claim_rewards`, the loop will iterate over all accumulated entries, potentially exceeding the Starknet block gas limit and permanently freezing their unclaimed yield.

---

### Finding Description

In `src/pool/pool.cairo`, the internal function `calculate_rewards` iterates over the `pool_member_epoch_balance` trace:

```cairo
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
    if pool_member_checkpoint.epoch() >= until_epoch {
        break;
    }
    let to_sigma = self.find_sigma(pool_member_checkpoint, curr_epoch: until_epoch);
    ...
    entry_to_claim_from += 1;
}
``` [1](#0-0) 

The loop starts at `entry_to_claim_from`, which is stored in `InternalPoolMemberInfoV1` and is only advanced when `claim_rewards` is successfully called. [2](#0-1) 

The trace grows by one entry per unique epoch whenever a pool member calls:
- `add_to_delegation_pool` → `increase_member_balance` → `set_member_balance` → `trace.insert(key: self.get_epoch_plus_k(), ...)`
- `exit_delegation_pool_intent` → `set_member_balance`
- `enter_delegation_pool` → `set_member_balance` [3](#0-2) 

The `insert` function in the trace only appends a new checkpoint when the epoch key differs from the last entry's key, meaning at most one new entry is added per epoch per pool member. [4](#0-3) 

If a pool member changes their balance once per epoch for N epochs without ever calling `claim_rewards`, the trace accumulates N entries. The next call to `claim_rewards` (or the view function `pool_member_info_v1`) must iterate over all N entries in a single transaction.

`claim_rewards` calls `calculate_rewards` and then updates `entry_to_claim_from`: [5](#0-4) 

`pool_member_info_v1` also calls `calculate_rewards` (read-only, but still gas-consuming): [6](#0-5) 

Each loop iteration performs multiple storage reads (`pool_member_trace.at(...)` and `find_sigma` which reads `cumulative_rewards_trace`), making each iteration expensive in gas.

---

### Impact Explanation

If the trace grows large enough that a single `claim_rewards` call exceeds the Starknet block gas limit, the transaction will always revert. Since `entry_to_claim_from` is only updated on a successful `claim_rewards` call, there is no way to partially claim rewards or reset the counter. The pool member's entire accumulated unclaimed yield becomes permanently unclaimable — a permanent freeze of unclaimed yield.

This matches the allowed impact: **Permanent freezing of unclaimed yield or unclaimed royalties (High)**.

---

### Likelihood Explanation

The attack is self-inflicted: only the pool member or their reward address can call `add_to_delegation_pool`, and only the pool member can call `exit_delegation_pool_intent`. A pool member who:
1. Actively adjusts their delegation balance every epoch (e.g., compounding by adding small amounts), AND
2. Does not claim rewards for an extended period (months to years)

will organically accumulate enough trace entries to trigger this condition. Given that Starknet epochs are on the order of days and the protocol is designed for long-term staking, this is a realistic scenario for active delegators. The developers themselves acknowledge the loop is unbounded in a code comment.

---

### Recommendation

1. **Enforce periodic reward claiming**: Require or incentivize pool members to claim rewards before balance changes accumulate beyond a safe threshold.
2. **Limit trace growth**: Cap the number of balance-change entries per pool member, or compact/prune old entries when rewards are claimed.
3. **Paginated claiming**: Introduce a `claim_rewards_partial(max_entries: u64)` function that advances `entry_to_claim_from` by at most `max_entries` per call, allowing incremental reward collection even when the trace is large.

---

### Proof of Concept

1. Staker deploys a delegation pool with `pool_enabled: true`.
2. Attacker (pool member) calls `enter_delegation_pool` with a minimum amount.
3. For each of N epochs (e.g., N = 5000), the pool member calls `add_to_delegation_pool` with 1 token. Each call inserts one new entry into `pool_member_epoch_balance` at `current_epoch + K`.
4. The pool member never calls `claim_rewards` during this period.
5. After N epochs, `pool_member_epoch_balance.length()` ≈ N and `entry_to_claim_from` = 0.
6. Pool member calls `claim_rewards`. The `calculate_rewards` loop must iterate over all N entries, each requiring multiple storage reads via `pool_member_trace.at(i)` and `find_sigma(...)`.
7. The transaction exceeds the Starknet block gas limit and reverts.
8. Every subsequent `claim_rewards` call also reverts. The pool member's accumulated unclaimed yield is permanently frozen with no recovery path. [7](#0-6) [8](#0-7)

    fn insert(
        self: StoragePath<Mutable<PoolMemberBalanceTrace>>, key: Epoch, value: PoolMemberBalance,
    ) -> (PoolMemberBalance, PoolMemberBalance) {
        let checkpoints = self.checkpoints;

        let len = checkpoints.len();
        if len == Zero::zero() {
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
            return (Zero::zero(), value);
        }

        // Update or append new checkpoint.
        let mut last = checkpoints[len - 1].read();
        let prev = last.value;
        if last.key == key {
            last.value = value;
            checkpoints[len - 1].write(last);
        } else {
            // Checkpoint keys must be non-decreasing.
            assert!(last.key < key, "{}", TraceErrors::UNORDERED_INSERTION);
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
        }
        (prev, value)
    }
