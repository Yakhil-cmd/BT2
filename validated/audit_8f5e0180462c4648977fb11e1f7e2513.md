### Title
Integer Division Truncation in `native_per_pubdata` Silently Zeroes Pubdata Resource Charges - (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

When computing `native_per_pubdata` for L2 transaction validation, the bootloader uses truncating integer division (`wrapping_div`) instead of ceiling division (`div_ceil`). When `pubdata_price < native_price`, the result truncates to zero, making all pubdata consumption completely free in native resources. This is the direct analog of the VotingEscrow bias-truncation bug: a small numerator divided by a larger denominator silently yields zero, causing the accounting to be incorrect.

---

### Finding Description

In `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`, the two per-unit native resource rates are computed as follows:

```rust
// native_per_gas uses ceiling division — conservative, user pays more
let native_per_gas = u256_try_to_u64(&gas_price.div_ceil(native_price))...

// native_per_pubdata uses FLOOR division — user may pay nothing
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
``` [1](#0-0) 

`native_per_gas` uses `div_ceil` (rounds up, conservative), while `native_per_pubdata` uses `wrapping_div` (truncates, floors). When `pubdata_price < native_price` — for example `pubdata_price = 999`, `native_price = 1000` — the result is `999 / 1000 = 0`.

The same asymmetry exists in the public API helper:

```rust
// native_per_gas = ceil(gas_price / native_price)
let native_per_gas = u256_try_to_u64(&gas_price.div_ceil(native_price)).ok_or(())?;

// native_per_pubdata = pubdata_price / native_price  ← floor, not ceil
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
``` [2](#0-1) 

Once `native_per_pubdata = 0`, it propagates into two downstream charging sites:

**1. Intrinsic pubdata overhead in `create_resources_for_tx`:**
```rust
let intrinsic_pubdata_overhead = native_per_pubdata_byte.saturating_mul(intrinsic_pubdata);
// → 0 * intrinsic_pubdata = 0, no native deducted for intrinsic pubdata
``` [3](#0-2) 

**2. Execution-time pubdata charging in `get_resources_to_charge_for_pubdata`:**
```rust
let native = current_pubdata_spent
    .checked_mul(native_per_pubdata)  // → 0 when native_per_pubdata == 0
    .ok_or(out_of_native_resources!())?;
``` [4](#0-3) 

Both the intrinsic pubdata cost and the per-byte execution pubdata cost are silently zeroed out.

---

### Impact Explanation

When `0 < pubdata_price < native_price`, every L2 transaction pays **zero native resources** for all pubdata it generates (storage writes, events, L1 messages). The native resource budget allocated from gas is entirely available for computation, and pubdata consumption does not reduce it at all. This breaks the resource accounting invariant: pubdata has a non-zero price (`pubdata_price > 0`) but is charged as if it were free. Users can write to many storage slots — generating significant pubdata — without incurring the intended native resource cost. The operator/sequencer bears the DA cost without receiving the corresponding native resource compensation from the transaction.

This matches the external report's impact class: *contract fails to deliver promised returns* (the fee model promises to charge for pubdata but does not) and *resource accounting incorrectness* (pubdata bytes consumed are not reflected in native charges).

---

### Likelihood Explanation

The block context fields `pubdata_price` and `native_price` are independent values set by the operator/sequencer oracle. The condition `pubdata_price < native_price` is realistic: during low-congestion periods, pubdata (DA) costs can be low while the native resource price (computational cost denominator) remains higher. Any unprivileged L2 transaction sender can trigger this path — no special role or key is required. The sender only needs to submit a normal EIP-1559 or ZKsync-type L2 transaction that performs storage writes.

---

### Recommendation

Replace `wrapping_div` with `div_ceil` for `native_per_pubdata`, matching the treatment of `native_per_gas`:

```rust
// Before (truncates to 0 when pubdata_price < native_price):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;

// After (rounds up, consistent with native_per_gas):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

Apply the same fix in `api/src/helpers.rs` line 427. This ensures that even when `pubdata_price` is just 1 wei below `native_price`, the user is charged at least 1 native unit per pubdata byte rather than 0.

---

### Proof of Concept

Set block context with `native_price = 1000`, `pubdata_price = 999`. Submit an L2 EIP-1559 transaction that writes to 10 storage slots (generating ~320 bytes of pubdata).

**Expected**: `native_per_pubdata = ceil(999/1000) = 1`; the transaction is charged `320 * 1 = 320` native units for pubdata.

**Actual**: `native_per_pubdata = floor(999/1000) = 0`; the transaction is charged `320 * 0 = 0` native units for pubdata. The full native budget from gas remains available for computation, and pubdata is effectively free despite `pubdata_price > 0`.
