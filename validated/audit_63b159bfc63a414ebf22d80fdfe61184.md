### Title
Pool Admin Bypasses `maxAdminSpreadFeeE6` Cap via Uncapped `setPoolBinAdditionalFees` — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary
`setPoolAdminFees` enforces `maxAdminSpreadFeeE6` on the global admin spread component, but `setPoolBinAdditionalFees` passes `addFeeBuyE6`/`addFeeSellE6` directly to the pool with **no cap check**, allowing the pool admin to set per-bin fees up to `uint16` max (65535 = 6.5535%) on every bin, bypassing the configured admin fee ceiling entirely.

### Finding Description

`setPoolAdminFees` correctly enforces the cap: [1](#0-0) 

`setPoolBinAdditionalFees` performs **no equivalent check** — it forwards the raw `uint16` values directly: [2](#0-1) 

`setBinAdditionalFees` on the pool only validates the bin index, not the fee magnitude: [3](#0-2) 

During every swap, the per-bin fee is added unconditionally on top of `baseFeeX64` (which already encodes the oracle spread): [4](#0-3) 

The same uncapped addition applies to all four swap directions (`buyToken0InBinSpecifiedOut`, `buyToken0InBinSpecifiedIn`, `buyToken1InBinSpecifiedOut`, `buyToken1InBinSpecifiedIn`). [5](#0-4) 

`HARD_MAX_SPREAD_FEE_E6 = 200_000` (20%) is the absolute ceiling for `maxAdminSpreadFeeE6`: [6](#0-5) 

A `uint16` max of 65535 in E6 units = **6.5535%**. If `maxAdminSpreadFeeE6` is configured at, e.g., 1000 (0.1%), the pool admin can still write 65535 per bin — 65× the intended cap — with no revert.

### Impact Explanation
Every swap through a bin with `addFeeBuyE6 = 65535` pays 6.5535% additional fee on top of the global spread fee. For a multi-bin swap traversing N bins, the excess fee compounds per bin. Traders lose principal directly and measurably above Sherlock thresholds. The excess fee accrues to LPs (not the admin), so it is not a direct admin extraction, but it is a direct trader loss caused by the admin exceeding their configured bounds — an admin-boundary break explicitly listed as an allowed impact.

### Likelihood Explanation
The pool admin is a semi-trusted role. The Smart Audit Pivots explicitly flag `setBinAdditionalFees` as a bypass surface. The call requires only `onlyPoolAdmin(pool)` — no timelock, no second step, no owner co-signature. A single transaction sets the fee on any bin in `[LOWEST_BIN, HIGHEST_BIN]`.

### Recommendation
Add a cap check in `setPoolBinAdditionalFees` (or inside `setBinAdditionalFees`) that enforces `addFeeBuyE6 <= maxAdminSpreadFeeE6` and `addFeeSellE6 <= maxAdminSpreadFeeE6` (or a dedicated per-bin cap), mirroring the guard already present in `setPoolAdminFees`.

### Proof of Concept
1. Deploy pool with `maxAdminSpreadFeeE6 = 1000` (0.1%).
2. Pool admin calls `factory.setPoolBinAdditionalFees(pool, bin, 65535, 65535)` for every bin in `[LOWEST_BIN, HIGHEST_BIN]` — no revert.
3. Trader executes a multi-bin swap; each bin charges 6.5535% additional fee on top of the global spread.
4. Assert trader output is far below what `maxAdminSpreadFeeE6 = 1000` would imply; the delta is direct principal loss.

  function setBinAdditionalFees(int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external
    onlyFactory
    nonReentrant(PoolActions.SET_BIN_ADDITIONAL_FEES)
  {
    if (bin < LOWEST_BIN || bin > HIGHEST_BIN) revert InvalidBinIndex(bin);
    BinState storage s = _binStates[bin];
    s.addFeeBuyE6 = addFeeBuyE6;
    s.addFeeSellE6 = addFeeSellE6;
    emit BinAdditionalFeesUpdated(bin, addFeeBuyE6, addFeeSellE6);
  }
