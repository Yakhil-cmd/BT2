### Title
Signature Verification Bypass via `skip_stateful_validations` Allows Unsigned Invoke Transactions to Enter Mempool — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

`skip_stateful_validations` skips the `__validate__` entry point (account signature check) for any invoke transaction with `nonce=1` when `account_tx_in_pool_or_recent_block` returns `true` for the sender. The check is not restricted to confirming a `deploy_account` transaction exists — it returns `true` for **any** transaction in the pool or recent block history for that address. An attacker who observes a victim's `deploy_account` transaction in the mempool can immediately submit an invoke transaction with `nonce=1` for the victim's address carrying an invalid or absent signature, and it will be admitted to the mempool without signature verification.

### Finding Description

`skip_stateful_validations` is designed to improve UX for the `deploy_account + invoke` flow: when a user deploys a new account and immediately sends a first invoke, the invoke can be admitted before the deploy is confirmed. [1](#0-0) 

The skip condition fires when:
1. The transaction is an `Invoke` with `nonce == 1`
2. The on-chain account nonce is `0` (account not yet deployed)
3. `account_tx_in_pool_or_recent_block(sender_address)` returns `true` [2](#0-1) 

When `skip_validate = true`, `run_validate_entry_point` sets `execution_flags.validate = false`: [3](#0-2) 

Inside `StatefulValidator::perform_validations`, the `__validate__` call is guarded by this flag and is entirely skipped: [4](#0-3) 

The code comment justifying the skip reads:

> "It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction **or transactions with future nonces that passed validations**."

The second branch of this disjunction is circular: future-nonce invoke transactions that themselves skipped validation are not a sound basis for trust. More critically, the check `account_tx_in_pool_or_recent_block` is not restricted to `deploy_account` transactions — it returns `true` for **any** transaction associated with the address in the pool or recent commit history. [5](#0-4) 

`perform_pre_validation_stage` is still called (nonce, fee, balance checks), but the account's `__validate__` entry point — the only place the account contract verifies the caller's signature — is not invoked: [6](#0-5) 

### Impact Explanation

An attacker who observes a victim's `deploy_account` transaction in the mempool (via P2P gossip or RPC) can:

1. Submit an invoke transaction with `nonce=1` for the victim's address carrying an arbitrary or empty signature.
2. The gateway skips `__validate__` because `account_tx_in_pool_or_recent_block` returns `true` (the victim's `deploy_account` is in the pool).
3. The unsigned invoke transaction is admitted to the mempool, occupying the `nonce=1` slot for the victim's address.
4. The victim's own legitimate `nonce=1` invoke transaction is blocked (duplicate nonce) or must pay a higher fee to replace the attacker's transaction.
5. When the batcher executes the block, the attacker's transaction calls `__validate__` and fails, but the fee is charged to the victim's newly-deployed account.

This matches: **High — Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.**

### Likelihood Explanation

- The attack requires only that the victim's `deploy_account` transaction be visible in the mempool, which is the normal state for any new account deployment.
- No privileged access is required; any unprivileged actor can submit transactions to the gateway.
- The attack window is the time between the `deploy_account` transaction entering the mempool and being committed to a block.
- The attacker needs only to know the victim's contract address (derivable from the `deploy_account` transaction itself).

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a `deploy_account` transaction exists for the sender address in the mempool. The mempool should expose a dedicated `deploy_account_tx_in_pool(address)` query that only returns `true` when a `deploy_account` transaction (not any transaction) is pending for that address.

Alternatively, restrict the skip to cases where the mempool can positively confirm the transaction type is `DeployAccount`, preventing any other transaction type from satisfying the skip condition.

### Proof of Concept

1. Victim calls gateway: `add_transaction(deploy_account_tx)` for address `V` with valid signature. Transaction enters mempool.
2. Attacker calls gateway: `add_transaction(invoke_tx { sender: V, nonce: 1, calldata: [transfer_all_tokens], signature: [] })`.
3. Gateway evaluates `skip_stateful_validations`: `nonce == 1` ✓, `account_nonce == 0` ✓, `account_tx_in_pool_or_recent_block(V)` → `true` (victim's deploy_account is in pool) ✓.
4. `skip_validate = true` → `__validate__` is not called → attacker's transaction is admitted to mempool.
5. Victim submits their own `invoke_tx { sender: V, nonce: 1, calldata: [...], signature: [valid] }` → rejected or must outbid attacker's transaction.
6. Batcher executes block: `deploy_account(V)` succeeds, then attacker's `invoke(V, nonce=1)` calls `__validate__` → fails (invalid signature) → reverted, fee charged to `V`.
7. Victim's legitimate transaction is gone from the mempool; victim must resubmit. [7](#0-6) [8](#0-7)

/// Check if validation of an invoke transaction should be skipped due to deploy_account not being
/// processed yet. This feature is used to improve UX for users sending deploy_account + invoke at
/// once.
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}

        &self,
        state: &mut S,

        Self::handle_nonce(state, tx_info, self.execution_flags.strict_nonce_check)?;
