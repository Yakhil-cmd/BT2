### Title
Invoke Transaction with Nonce=1 Bypasses Gateway Signature Validation via `skip_stateful_validations` — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator unconditionally skips the account's `__validate__` entry point for any invoke transaction with `nonce=1` targeting an account whose on-chain nonce is `0`, provided that account has *any* transaction in the mempool or a recent block. Because the check does not verify the caller's identity or the transaction's signature before granting this bypass, an unprivileged attacker can submit an invoke transaction with an empty or forged signature for any account that has a pending `deploy_account` in the mempool, and the gateway will admit it without signature verification.

---

### Finding Description

`skip_stateful_validations` is designed as a UX convenience: when a user submits a `deploy_account` (nonce=0) and an `invoke` (nonce=1) simultaneously, the account does not yet exist on-chain, so the `__validate__` entry point cannot be called. The function therefore skips validation when the conditions are met. [1](#0-0) 

The bypass condition is:

```
tx is Invoke  AND  tx.nonce == 1  AND  account_nonce == 0
AND  account_tx_in_pool_or_recent_block(sender_address) == true
```

When `skip_validate` is `true`, `run_validate_entry_point` constructs `ExecutionFlags` with `validate: false`: [2](#0-1) 

This means the account's `__validate__` function is **never called** at the gateway layer for this transaction. The gateway then proceeds to add the transaction to the mempool via `mempool_client.add_tx`. [3](#0-2) 

The critical flaw is that `account_tx_in_pool_or_recent_block` is a **public mempool query** — any observer of the mempool can see which accounts have pending `deploy_account` transactions. An attacker can then submit an invoke with `nonce=1` for any such account with an arbitrary (invalid) signature, and the gateway will admit it because the bypass condition is satisfied by the *victim's* deploy_account transaction, not the attacker's.

The `validate_by_mempool` call that precedes `skip_stateful_validations` performs nonce-gap and fee-level checks but does **not** perform signature verification: [4](#0-3) 

---

### Impact Explanation

**High — Mempool/gateway admission accepts invalid transactions before sequencing.**

An attacker can inject unsigned or maliciously-crafted invoke transactions (nonce=1) for any account that has a pending `deploy_account` in the mempool. These transactions pass all gateway checks and are stored in the mempool. Consequences include:

1. **Mempool pollution / DoS**: The attacker can flood the mempool with zero-cost invalid transactions targeting every newly-deploying account, consuming mempool slots and batcher resources.
2. **Transaction displacement**: If the mempool uses nonce-based replacement semantics, the attacker's invalid invoke (nonce=1) may displace the victim's legitimate invoke (nonce=1), causing the victim's transaction to be silently dropped. The victim's deploy_account succeeds but their intended invoke never executes.
3. **Wasted block space**: The batcher will include the invalid transaction in a block proposal; the blockifier will call `__validate__` during actual execution (via `new_for_sequencing` which always sets `validate: true`), the transaction will revert, and the sequencer earns no fee because the account has no funds at nonce=1. [5](#0-4) 

---

### Likelihood Explanation

**Medium.** The attacker only needs to monitor the public mempool for `deploy_account` transactions (nonce=0, account_nonce=0). This is observable without any privileged access. The window is open from the moment the `deploy_account` enters the mempool until the block containing it is finalized and the state reader reflects nonce=1. During high-traffic periods (e.g., a popular contract launch causing many simultaneous account deployments), the attack surface is large.

---

### Recommendation

1. **Narrow the bypass condition**: Instead of checking `account_tx_in_pool_or_recent_block` (which returns true for *any* transaction type), add a dedicated mempool query `deploy_account_in_pool(sender_address)` that returns true only if the pending transaction is specifically a `DeployAccount`. This closes the window where a third-party's deploy_account enables bypass for an attacker's invoke.

2. **Authenticate the bypass**: Before skipping `__validate__`, verify that the invoke transaction's sender matches the deployer in the pending `deploy_account` transaction, and that the invoke's `tx_hash` is signed by the same key used in the `deploy_account` (or at minimum that the signature field is non-empty and structurally valid).

3. **Rate-limit or deduplicate**: Enforce that only one nonce=1 invoke per account address can be in the mempool at a time when the account is in the "pending deploy" state.

---

### Proof of Concept

```
1. Victim broadcasts:
     deploy_account(account=A, nonce=0, valid_sig)   → enters mempool
     invoke(account=A, nonce=1, calldata=X, valid_sig) → enters mempool (skip_validate=true, UX path)

2. Attacker observes deploy_account for account A in the mempool.

3. Attacker broadcasts:
     invoke(account=A, nonce=1, calldata=DRAIN_FUNDS, sig=[])

4. Gateway stateful validator evaluates skip_stateful_validations:
     - tx is Invoke ✓
     - tx.nonce == 1 ✓
     - account_nonce(A) == 0 ✓  (A not yet on-chain)
     - account_tx_in_pool_or_recent_block(A) == true ✓  (victim's deploy_account)
     → skip_validate = true

5. run_validate_entry_point called with validate=false.
   __validate__ is NOT called. Signature is NOT checked.

6. Attacker's invoke is admitted to the mempool.

7. If mempool replaces victim's nonce=1 invoke with attacker's:
     - Block executes: deploy_account(A) succeeds → A is deployed.
     - Block executes: attacker's invoke → blockifier calls __validate__ → fails (bad sig).
     - Victim's invoke was displaced; it never executes.
     - Attacker pays nothing (no valid sig, fee transfer fails on revert).
``` [1](#0-0) [6](#0-5)

    async fn run_validate_entry_point(
        &mut self,
        executable_tx: &ExecutableTransaction,
        skip_validate: bool,
    ) -> StatefulTransactionValidatorResult<()> {
        let only_query = false;
        let charge_fee = enforce_fee(executable_tx, only_query);
        let strict_nonce_check = false;
        let execution_flags =
            ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };

        let account_tx = AccountTransaction { tx: executable_tx.clone(), execution_flags };

        // Build block context.
        let mut versioned_constants = VersionedConstants::get_versioned_constants(
            self.config.versioned_constants_overrides.clone(),
        );
        // The validation of a transaction is not affected by the casm hash migration.
        versioned_constants.disable_casm_hash_migration();

        let mut block_info = self.gateway_fixed_block_state_reader.get_block_info().await?;
        block_info.block_number = block_info.block_number.unchecked_next();
        let block_context = BlockContext::new(
            block_info,
            self.chain_info.clone(),
            versioned_constants,
            BouncerConfig::max(),
        );

        // Move state into the blocking task and run CPU-heavy validation.
        let state_reader_and_contract_manager = self.take_state_reader_and_contract_manager();

        let cur_span = Span::current();
        #[allow(clippy::result_large_err)]
        tokio::task::spawn_blocking(move || {
            cur_span.in_scope(|| {
                let state = CachedState::new(state_reader_and_contract_manager);
                let mut blockifier_validator = StatefulValidator::create(state, block_context);
                blockifier_validator.validate(account_tx)
            })
        })
        .await
        .map_err(|e| StarknetError {
            code: StarknetErrorCode::UnknownErrorCode(
                "StarknetErrorCode.InternalError".to_string(),
            ),
            message: format!("Blocking task join error when running the validate entry point: {e}"),
        })?
        .map_err(|e| StarknetError {
            code: StarknetErrorCode::KnownErrorCode(KnownStarknetErrorCode::ValidateFailure),
            message: e.to_string(),
        })?;
        Ok(())
    }

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
