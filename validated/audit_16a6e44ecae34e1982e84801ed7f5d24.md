### Title
`claim_rewards` Passes Accounting Check But May Fail on Actual Transfer Due to Pending L1 Minting - (File: src/reward_supplier/reward_supplier.cairo)

### Summary
The `claim_rewards` function in `L2RewardSupplier` validates only the accounting invariant (`amount <= unclaimed_rewards`) but does not verify the contract's actual token balance. Because the system relies on asynchronous L1 minting to fund the reward supplier, the real token balance can be less than `unclaimed_rewards` while L1 minting is in-flight. Any staker or delegator who triggers a reward claim during this window will have their transaction revert, temporarily freezing their unclaimed yield.

### Finding Description
`claim_rewards` in `reward_supplier.cairo` performs a single guard:

```cairo
let unclaimed_rewards = self.unclaimed_rewards.read();
assert!(amount <= unclaimed_rewards, "{}", GenericError::AMOUNT_TOO_HIGH);
// ...
token_dispatcher.checked_transfer(recipient: staking_contract, amount: amount.into());
``` [1](#0-0) 

The `request_funds` helper maintains the invariant:

```
balance + l1_pending_requested_amount >= unclaimed_rewards + threshold
``` [2](#0-1) 

This invariant does **not** guarantee `balance >= unclaimed_rewards`. Whenever `l1_pending_requested_amount > threshold`, the actual token balance is strictly less than `unclaimed_rewards`. The accounting check at line 214 passes, but the `checked_transfer` at line 219 reverts because the contract does not hold enough tokens.

The staking contract's `send_rewards_to_staker` calls `claim_from_reward_supplier` which routes directly to this function: [3](#0-2) 

So any call to `claim_rewards` on the staking contract, `unstake_action`, or `exit_delegation_pool_action` that triggers reward distribution will revert if the reward supplier's real balance is below the owed amount.

### Impact Explanation
Stakers and delegators are temporarily unable to claim their accrued rewards. `unstake_action` and `exit_delegation_pool_action` also internally claim rewards, so those flows revert too, blocking principal withdrawal. This constitutes **temporary freezing of unclaimed yield and of funds** until the pending L1 mint messages are processed and tokens arrive via StarkGate.

### Likelihood Explanation
L1→L2 message delivery on Starknet is asynchronous and can be delayed by hours or longer during congestion. The reward supplier is designed to operate with `balance < unclaimed_rewards` as a normal state (covered by `l1_pending`). Any epoch with active attestations increases `unclaimed_rewards` and triggers L1 mint requests. If a staker or delegator calls `claim_rewards` before the minted tokens arrive, the revert is deterministic. This is a medium-likelihood condition that occurs in every deployment cycle between attestation and L1 token delivery.

### Recommendation
Add an explicit balance check inside `claim_rewards` before attempting the transfer:

```cairo
let balance: Amount = token_dispatcher
    .balance_of(account: get_contract_address())
    .try_into()
    .expect_with_err(InternalError::BALANCE_ISNT_AMOUNT_TYPE);
assert!(amount <= balance, "{}", Error::INSUFFICIENT_BALANCE);
```

Alternatively, make `claim_rewards` a soft-fail that returns the actually-transferred amount (capped at `min(amount, balance)`), and let the staking contract handle partial reward delivery gracefully. The `request_funds` threshold should also be tuned so that `balance` is always kept above the maximum single-epoch reward claim.

### Proof of Concept

1. Protocol is live; reward supplier holds 0 STRK (all funds are `l1_pending`).
2. Staker attests → `update_unclaimed_rewards_from_staking_contract(rewards)` is called → `unclaimed_rewards` = `rewards`, `request_funds` sends L1 mint messages, `l1_pending` = `rewards + threshold`.
3. L1 mint has not yet been processed; actual token balance of reward supplier = 0.
4. Staker calls `claim_rewards(staker_address)` on the staking contract.
5. Staking contract calls `claim_from_reward_supplier(amount = rewards)`.
6. Reward supplier: `assert!(rewards <= unclaimed_rewards)` → passes.
7. `token_dispatcher.checked_transfer(staking_contract, rewards)` → **reverts** (balance = 0 < rewards).
8. Entire transaction reverts; staker cannot claim rewards or unstake until L1 minting completes. [4](#0-3) [5](#0-4)

            assert!(
                get_caller_address() == staking_contract,

            let token_dispatcher = self.token_dispatcher.read();
            token_dispatcher.checked_transfer(recipient: staking_contract, amount: amount.into());

        fn request_funds(ref self: ContractState, unclaimed_rewards: Amount) {
            // Read current balance.
            let token_dispatcher = self.token_dispatcher.read();
            let balance: Amount = token_dispatcher
                .balance_of(account: get_contract_address())
                .try_into()
                .expect_with_err(InternalError::BALANCE_ISNT_AMOUNT_TYPE);

            // Calculate credit, which is the contract's balance plus the amount already requested
            // from L1, and the debit, which is the unclaimed rewards.
            let mut l1_pending_requested_amount = self.l1_pending_requested_amount.read();
            let credit = balance + l1_pending_requested_amount;
            let debit = unclaimed_rewards;

            // If there isn't enough credit to cover the debit + threshold, request funds.
            let base_mint_amount = self.base_mint_amount.read();
            let threshold = compute_threshold(base_mint_amount);
            if credit < debit + threshold {
                let diff = debit + threshold - credit;
                let num_msgs = ceil_of_division(dividend: diff, divisor: base_mint_amount);
                let total_amount = num_msgs * base_mint_amount;
                for _ in 0..num_msgs {
                    self.send_mint_request_to_l1_reward_supplier();
                }
                self.emit(Events::MintRequest { total_amount, num_msgs });
                l1_pending_requested_amount += total_amount;
            }

            // Commit to storage the requested amount, which is now part of the credit.
            self.l1_pending_requested_amount.write(l1_pending_requested_amount);
        }
