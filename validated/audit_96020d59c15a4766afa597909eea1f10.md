### Title
Malicious Participant Sends Unverified CKD Shares Corrupting the Coordinator's Derived Key Output — (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

The CKD coordinator aggregates `(big_y, big_c)` contributions from every participant with no zero-knowledge proof or commitment binding those values to the participant's actual private share. A single malicious participant can substitute arbitrary group elements, causing the coordinator to output a corrupted `CKDOutput` that honest parties accept as the legitimate confidential derived key.

---

### Finding Description

The external report's vulnerability class is **unverified spot-value substitution**: a value fetched from an untrusted source is used directly in a security-critical computation without any proof of correctness. The analog here is that each participant's `(big_y, big_c)` pair is the "spot value" — it is received over the network and used directly with no binding proof.

**Protocol intent** (`compute_signature_share`, lines 148–181):

Each participant `i` is supposed to compute:
```
y_i  ← random scalar
big_y_i = y_i · G
big_s_i = x_i · H(pk ‖ app_id)          // x_i = private share
big_c_i = big_s_i + y_i · app_pk        // ElGamal encryption
norm_big_y = λ_i · big_y_i
norm_big_c = λ_i · big_c_i
```
and send `(norm_big_y, norm_big_c)` to the coordinator.

**Coordinator aggregation** (`do_ckd_coordinator`, lines 50–55):

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
```

The coordinator sums every received pair and immediately constructs the output:

```rust
let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
Ok(Some(ckd_output))
```

There is **no proof of knowledge** that `big_c` was formed as `x_i · H(pk ‖ app_id) + y_i · app_pk` for the participant's actual `x_i`, and **no commitment** binding `big_y` to the same `y_i` used in `big_c`. Any participant can send any pair of group elements.

**Corruption arithmetic**:

If malicious participant `m` sends `(big_y', big_c')` instead of the correct `(norm_big_y_m, norm_big_c_m)`, the coordinator computes:

```
Y'  = Y_correct − norm_big_y_m + big_y'
C'  = C_correct − norm_big_c_m + big_c'
```

Unmasking with `app_sk`:

```
C' − app_sk · Y'
  = msk · H(pk ‖ app_id)
    + (big_c' − app_sk · big_y')
    − λ_m · x_m · H(pk ‖ app_id)
```

Setting `big_y' = 0, big_c' = 0` (the simplest attack) yields:

```
(msk − λ_m · x_m) · H(pk ‖ app_id)
```

which is a deterministically wrong derived key. The coordinator has no way to detect the substitution.

---

### Impact Explanation

The coordinator outputs a `CKDOutput` whose `unmask` produces an incorrect confidential derived key. Every honest party that relies on this output — including TEE nodes — accepts the corrupted value as legitimate. This is a **corruption of CKD output** matching the allowed High impact: *"Corruption of DKG, reshare, refresh, presign, sign, or CKD outputs so honest parties accept inconsistent public keys, participant sets, transcripts, or unusable cryptographic outputs."*

Depending on how the derived key is used downstream, the corruption can render the key unusable (permanent denial of the CKD service for that session) or cause the TEE to operate under an attacker-shifted key.

---

### Likelihood Explanation

Any single participant in the CKD protocol can trigger this. The entry path is direct: call `ckd()` as a malicious participant and, in the single protocol round, send crafted `(big_y', big_c')` values to the coordinator. No special privilege, no leaked key material, and no external oracle is required. The protocol has exactly one round of participant-to-coordinator communication and no subsequent verification step.

---

### Recommendation

Require each participant to accompany their `(big_y, big_c)` submission with a Sigma/Schnorr proof of correct formation — specifically a proof of knowledge of `y_i` such that:

```
big_y = λ_i · y_i · G
big_c = λ_i · (x_i · H(pk ‖ app_id) + y_i · app_pk)
```

where `x_i` is bound to the participant's public verification share (derivable from the DKG output). The coordinator must verify this proof before including the contribution in the sum. Alternatively, adopt a commit-then-reveal pattern analogous to the DKG's `commitment_hash` round (`src/dkg.rs` lines 409–426) so that participants cannot adaptively choose their contribution after seeing others'.

---

### Proof of Concept

1. Run `ckd()` with `N = 3` participants and one coordinator.
2. The malicious participant intercepts the `waitpoint` send in `do_ckd_participant` (line 30) and instead sends `(ElementG1::identity(), ElementG1::identity())` — i.e., both group elements set to the identity.
3. The coordinator sums the three contributions; the malicious participant's `(0, 0)` replaces `(λ_m · y_m · G, λ_m · (x_m · H + y_m · app_pk))`.
4. The resulting `CKDOutput` unmasked with `app_sk` yields `(msk − λ_m · x_m) · H(pk ‖ app_id)` instead of `msk · H(pk ‖ app_id)`.
5. The coordinator returns `Some(ckd_output)` with no error; honest parties accept the corrupted key.

```rust
fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
```
