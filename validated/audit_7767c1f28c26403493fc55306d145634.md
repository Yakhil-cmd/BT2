### Title
`protocolUnpausePool` Calls `setPause(1)` Instead of `setPause(0)`, Leaving Pool Permanently Paused After Protocol Unpause — (`File: metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary

`protocolUnpausePool` is the protocol-owner's counterpart to `protocolPausePool`. It is expected to restore a pool to the active state (`pauseLevel = 0`). Instead, it calls `setPause(1)`, which sets the pool to admin-paused — a state that still blocks all swaps. The pool remains unusable even after the protocol owner explicitly "unpauses" it.

### Finding Description

The factory defines three pause levels:

> `0 = active, 1 = paused by admin, 2 = paused by protocol. Transitions enforced by factory.` [1](#0-0) 

`protocolPausePool` can pause from level 0 **or** level 1 to level 2: [2](#0-1) 

`protocolUnpausePool` then transitions from level 2 to level **1** — not level 0: [3](#0-2) 

The pool's `_checkNotPaused` guard rejects any non-zero `pauseLevel`: [4](#0-3) 

So after `protocolUnpausePool`, `pauseLevel == 1`, and every call to `swap()` (which carries `whenNotPaused`) still reverts with `PoolPaused`. [5](#0-4) 

The admin-side `unpausePool` correctly calls `setPause(0)`: [6](#0-5) 

The wrong value (`1` vs `0`) in `protocolUnpausePool` is the direct analog of the M-01 pattern: a function that should update a state variable to the correct value calls the wrong variant, leaving the variable in a broken state.

### Impact Explanation

When the protocol owner pauses an **active** pool (`pauseLevel 0 → 2`) and then calls `protocolUnpausePool`, the pool lands at `pauseLevel = 1`. All swaps remain blocked. LPs cannot trade; the pool is functionally dead until the pool admin separately calls `unpausePool`. If the admin key is unavailable or the admin is unresponsive, the pool is stuck indefinitely. This is broken core swap functionality and matches the "unusable swap/liquidity flows" impact gate.

### Likelihood Explanation

The protocol owner pausing and then unpausing a pool that was originally active is the primary use-case for `protocolUnpausePool`. Every such invocation silently leaves the pool paused. No special conditions or attacker involvement are required — the bug fires on every normal protocol-pause/unpause cycle of an active pool.

### Recommendation

Change `protocolUnpausePool` to call `setPause(0)` instead of `setPause(1)`:

```solidity
function protocolUnpausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 2) revert InvalidPauseTransition(cur, 0);
-   IMetricOmmPoolFactoryActions(pool).setPause(1);
+   IMetricOmmPoolFactoryActions(pool).setPause(0);
}
```

If the intended design is a two-step unpause (protocol removes level-2, admin must then remove level-1), the factory must store the pre-pause level and restore it, and the function must be renamed and documented to reflect that it does **not** fully unpause the pool.

### Proof of Concept

1. Pool is deployed and active (`pauseLevel = 0`).
2. Protocol owner calls `protocolPausePool(pool)` → `pauseLevel = 2`. All swaps revert.
3. Emergency resolved. Protocol owner calls `protocolUnpausePool(pool)`.
4. `setPause(1)` executes → `pauseLevel = 1`. Pool is still paused.
5. Any call to `swap()` reverts with `PoolPaused` despite the protocol owner having "unpaused" the pool.
6. Only the pool admin calling `unpausePool(pool)` can restore `pauseLevel = 0`. [3](#0-2) [7](#0-6) [8](#0-7)
