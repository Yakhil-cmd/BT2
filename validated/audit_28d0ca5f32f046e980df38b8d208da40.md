### Title
`addLiquidity()` Callable on Paused Pool Exposes New LP Deposits to Compromised Price State — (`File: metric-core/contracts/MetricOmmPool.sol`)

### Summary

`MetricOmmPool.addLiquidity()` lacks the `whenNotPaused` guard that protects `swap()`. When a pool is paused due to a security concern (e.g., a bad or manipulated price provider), users can still deposit principal into the pool. Once the pool is unpaused, those funds are immediately exposed to the condition that triggered the pause.

### Finding Description

The pool exposes a `pauseLevel` state variable (0 = active, 1 = admin-paused, 2 = protocol-paused). The `swap()` function is correctly gated:

```solidity
function swap(...) external whenNotPaused nonReentrant(PoolActions.SWAP) ...
``` [1](#0-0) 

However, `addLiquidity()` carries no equivalent guard:

```solidity
function addLiquidity(...) external nonReentrant(PoolActions.ADD_LIQUIDITY) ...
``` [2](#0-1) 

The `_checkNotPaused()` helper exists and is used by `swap()`, but is never invoked in the liquidity path:

```solidity
function _checkNotPaused() internal view {
    if (pauseLevel != 0) revert PoolPaused();
}
``` [3](#0-2) 

The factory's pause transitions confirm that pausing is a security response, not a routine operational state: [4](#0-3) 

### Impact Explanation

A pool is paused precisely because its current state is unsafe for trading — most commonly because the active price provider is returning stale, inverted, or manipulated prices. While swaps are blocked, `addLiquidity()` remains open. Any LP who deposits during the pause window has their principal locked into bins priced against the compromised oracle. The moment the pool is unpaused, arbitrageurs can execute swaps at the bad price and drain those bins. The LP suffers a direct loss of deposited token0/token1 principal with no recourse.

### Likelihood Explanation

The trigger is a pool pause event, which is an expected operational action (pool admin at level 1, protocol owner at level 2). Users observing a paused pool have no on-chain signal explaining *why* it is paused. A user who believes the pause is routine (e.g., a fee update) and adds liquidity in anticipation of resumption is the realistic victim. The pool admin path (`pausePool` → `unpausePool`) requires no timelock, so the window between pause and unpause can be arbitrarily short, making front-running of the unpause transaction feasible.

### Recommendation

Add `whenNotPaused` to `addLiquidity()`:

```solidity
function addLiquidity(...) external whenNotPaused nonReentrant(PoolActions.ADD_LIQUIDITY) ...
```

`removeLiquidity()` should intentionally remain ungated (LPs must always be able to exit), but new deposits should be blocked whenever the pool is in a non-zero pause state.

### Proof of Concept

1. Pool is deployed with a mutable price provider (timelock < `type(uint256).max`).
2. The price provider begins returning a manipulated ask price far below fair value.
3. Protocol owner calls `protocolPausePool(pool)` → `pauseLevel = 2`. Swaps revert.
4. Victim LP calls `addLiquidity(...)` with 1000 USDC + 1 WETH. The call succeeds; tokens are transferred into the pool and credited to bins at the current (manipulated) bin position.
5. Protocol owner calls `protocolUnpausePool(pool)` → `pauseLevel = 1`. Pool admin calls `unpausePool(pool)` → `pauseLevel = 0`.
6. Attacker immediately calls `swap(zeroForOne=false, ...)` buying token0 at the manipulated low ask price, draining the victim's token0 balance from the bins.
7. Victim calls `removeLiquidity(...)` and receives far less than deposited. [5](#0-4) [6](#0-5)
