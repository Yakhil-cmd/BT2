Audit Report

## Title
Unbounded Loop via Attacker-Controlled `ingress_start`/`ingress_end` Causes OOM/Infinite-Loop DoS in ICP Rosetta `/construction/payloads` — (`rs/rosetta-api/icp/src/request_handler/construction_payloads.rs`)

## Summary

The ICP Rosetta `/construction/payloads` handler accepts user-supplied `ingress_start` and `ingress_end` values with no range validation, then iterates a `while now < ingress_end` loop that pushes one `u64` per 120-second interval. Supplying `ingress_start=0` and `ingress_end=u64::MAX` causes ~153 billion iterations (≈1.2 TB heap allocation, OOM), and if OOM does not terminate the process first, a truncating `as u64` cast in `Time::from_duration` wraps `now` back to a small value, producing an infinite loop. Either outcome renders the Rosetta node completely unavailable.

## Finding Description

**Unbounded loop with no guard:** [1](#0-0) 

The loop variable `now` starts at attacker-controlled `ingress_start` and increments by `interval` (120 s = 120,000,000,000 ns) each iteration until `now >= ingress_end`. With `ingress_start=0` and `ingress_end=u64::MAX`, this is `u64::MAX / 120_000_000_000 ≈ 153,722,867,280` iterations, each pushing a `u64` onto `ingress_expiries` — approximately 1.2 TB of heap allocation.

**Interval computation:** [2](#0-1) 

`MAX_INGRESS_TTL (300 s) − PERMITTED_DRIFT (60 s) − 120 s = 120 s`. [3](#0-2) 

**Inputs are fully attacker-controlled with no validation:** [4](#0-3) 

`ingress_start` and `ingress_end` are `Option<u64>` fields deserialized directly from JSON: [5](#0-4) 

**No guard in the ICP handler.** The ICRC1 Rosetta handler explicitly rejects bad ranges: [6](#0-5) 

The ICP handler has no equivalent check.

**Wrap-around infinite loop via truncating cast:** [7](#0-6) 

`AddAssign` calls `Time::from_duration`, which performs: [8](#0-7) 

`t.as_nanos()` returns `u128`. When `now` is near `u64::MAX` and `interval = 120_000_000_000`, the sum exceeds `u64::MAX` in `u128`, and the `as u64` cast silently truncates — wrapping `now` back to a small value (e.g., ~10,448,384 ns ≈ 10 ms). Since `ingress_end = u64::MAX`, the condition `now < ingress_end` is immediately true again, producing an infinite loop.

## Impact Explanation

This is a **High** severity DoS: "Application/platform-level DoS, crash" and "Significant Rosetta … infrastructure security impact with concrete user or protocol harm." The ICP Rosetta node is a financial integration component used by exchanges and custodians to submit ICP ledger transactions. A single malicious request causes either:
- **Phase 1 (OOM):** ~153 billion `u64` pushes ≈ 1.2 TB heap allocation, exhausting memory and crashing the process.
- **Phase 2 (infinite loop):** If OOM does not kill the process, the wrap-around makes the loop infinite, consuming 100% of one CPU core indefinitely.

Either outcome makes the Rosetta node completely unresponsive to all subsequent requests, blocking all exchange integrations and ICP transfers routed through it.

## Likelihood Explanation

The `/construction/payloads` endpoint is unauthenticated HTTP. Any external client can POST a single crafted JSON request with `metadata.ingress_start=0` and `metadata.ingress_end=18446744073709551615`. No credentials, no privileged role, no network position, and no victim interaction are required. The attack is trivially repeatable and requires no special tooling beyond a standard HTTP client.

## Recommendation

Add an explicit upper-bound check on the ingress window before the loop, mirroring the ICRC1 Rosetta implementation:

```rust
let max_window = Duration::from_secs(24 * 3600);
if ingress_end.saturating_duration_since(ingress_start) > max_window {
    return Err(ApiError::invalid_request(
        "ingress_end - ingress_start exceeds maximum allowed window",
    ));
}
```

Additionally, replace the unchecked `as u64` cast in `Time::from_duration` with a checked or saturating conversion to prevent silent wrap-around on overflow:

```rust
fn from_duration(t: Duration) -> Self {
    Time(u64::try_from(t.as_nanos()).unwrap_or(u64::MAX))
}
```

## Proof of Concept

```
POST /construction/payloads HTTP/1.1
Content-Type: application/json

{
  "network_identifier": { "blockchain": "Internet Computer", "network": "<ledger_canister_id>" },
  "operations": [/* any valid transfer operation */],
  "public_keys": [/* any valid public key */],
  "metadata": {
    "ingress_start": 0,
    "ingress_end": 18446744073709551615
  }
}
```

The server thread handling this request enters the unbounded loop at `construction_payloads.rs:101`, allocating memory until OOM kills the process, or looping infinitely after the `as u64` wrap-around in `time.rs:104`. A deterministic unit test can confirm the wrap-around by constructing `Time::from_nanos_since_unix_epoch(u64::MAX - 60_000_000_000)`, calling `+= Duration::from_secs(120)`, and asserting the result is less than the original value.
