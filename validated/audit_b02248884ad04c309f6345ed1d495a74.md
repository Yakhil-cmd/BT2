### Title
`DepositAllowlistExtension` gates on `owner` (LP share recipient) instead of `sender` (actual depositor), allowing any unprivileged caller to bypass the deposit allowlist — (File: metric-periphery/contracts/extensions/DepositAllowlistExtension.sol)

---

### Summary

`DepositAllowlistExtension.beforeAddLiquidity` is the production deposit-access guard for Metric OMM pools. It receives both `sender` (the address that called `pool.addLiquidity`, i.e., the payer) and `owner` (the address that will receive LP shares). The hook silently discards `sender` and checks only `owner` against the allowlist. Because `pool.addLiquidity` accepts any arbitrary `owner` with no requirement that `msg.sender == owner`, an address that is **not** on the allowlist can call `addLiquidity` directly, supply an allowlisted address as `owner`, pass the extension check, and execute a deposit — violating the pool admin's access-control invariant.

---

### Finding Description

`MetricOmmPool.addLiquidity` is an operator-pattern function: the caller (`msg.sender`) pays via callback, but the LP shares are credited to the caller-supplied `owner`. [1](#0-0) 

The pool passes both identities to the extension hook: [2](#0-1) 

`DepositAllowlistExtension.beforeAddLiquidity` receives `sender` as its first argument but names it `_` (discarded). The allowlist lookup is keyed on `owner` only: [3](#0-2) 

Contrast with `SwapAllowlistExtension.beforeSwap`, which correctly checks `sender` (the actual caller): [4](#0-3) 

Because `owner` is a free parameter with no `msg.sender == owner` enforcement anywhere in `addLiquidity`, an attacker who is **not** on the allowlist can call:

```solidity
pool.addLiquidity(allowlistedAddress, salt, deltas, callbackData, extensionData)
```

The extension evaluates `allowedDepositor[pool][allowlistedAddress]` → `true`, the hook returns its selector, and the deposit proceeds. The attacker's tokens are pulled via callback; LP shares are minted to `allowlistedAddress`. The pool admin's intent — that only allowlisted addresses may deposit — is broken.

The `MetricOmmPoolLiquidityAdder` path has the same exposure: `_validateOwner` only rejects `address(0)`, not mismatched callers. [5](#0-4) 

---

### Impact Explanation

The deposit allowlist — the sole on-chain mechanism for restricting who may add liquidity to a pool — is rendered ineffective. Any address can deposit into an allowlist-gated pool by nominating an allowlisted address as `owner`. The pool admin's access-control boundary (e.g., KYC/regulatory gating, curated LP sets) is bypassed by an unprivileged path with no special privileges required. LP shares are minted to the nominated allowlisted address, not the attacker, so the attacker does not directly profit, but the invariant "only allowlisted addresses may hold newly minted LP shares from this pool" is violated and the allowlist provides no real protection.

---

### Likelihood Explanation

The attack requires only: (1) knowing any allowlisted address (observable on-chain via `AllowedToDepositSet` events or direct storage reads), and (2) calling `pool.addLiquidity` directly with that address as `owner`. No special role, no flash loan, no oracle manipulation. Any EOA or contract can execute this in a single transaction.

---

### Recommendation

Change `DepositAllowlistExtension.beforeAddLiquidity` to check `sender` (the actual depositor/payer) instead of `owner` (the LP share recipient), mirroring the correct pattern in `SwapAllowlistExtension`:

```solidity
function beforeAddLiquidity(address sender, address, uint80, LiquidityDelta calldata, bytes calldata)
    external
    view
    override
    returns (bytes4)
{
    if (!allowAllDepositors[msg.sender] && !allowedDepositor[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToDeposit();
    }
    return IMetricOmmExtensions.beforeAddLiquidity.selector;
}
```

If the intended semantics is to gate the LP share recipient (`owner`), the NatDoc and admin tooling must be updated to reflect that, and the `sender` identity must be separately gated if caller restriction is also desired.

---

### Proof of Concept

1. Pool admin deploys a pool with `DepositAllowlistExtension` attached to `beforeAddLiquidity`. Only `alice` is allowlisted (`allowedDepositor[pool][alice] = true`).
2. `bob` (not allowlisted) calls `pool.addLiquidity(alice, salt, deltas, callbackData, "")` directly.
3. `DepositAllowlistExtension.beforeAddLiquidity` is invoked with `sender = bob`, `owner = alice`. The hook discards `sender` and checks `allowedDepositor[pool][alice]` → `true`. Hook returns its selector.
4. `LiquidityLib.addLiquidity` mints shares to `(alice, salt)`. `bob`'s tokens are pulled via callback.
5. `bob` has deposited into an allowlist-gated pool without being on the allowlist. The pool admin's access-control invariant is violated. [3](#0-2) [1](#0-0)
