### Title
Missing Zero-Address Validation for `admin` in `AdminControlled` Constructor Permanently Freezes `EvmErc20`/`EvmErc20V2` Token Minting - (File: etc/eth-contracts/contracts/AdminControlled.sol)

### Summary
The `AdminControlled` constructor accepts an `_admin` address parameter and assigns it directly without validating it is non-zero. The developers explicitly suppressed the static-analysis warning with `// slither-disable-next-line missing-zero-check`. Both `EvmErc20` and `EvmErc20V2` inherit from `AdminControlled` and forward the `admin` constructor argument without any zero-address guard. If `admin` is set to `address(0)` at deployment, every `onlyAdmin`-gated function â€” including `mint()` â€” becomes permanently inaccessible, permanently freezing the ability to issue ERC-20 tokens on the EVM side of the bridge.

### Finding Description
`AdminControlled.sol` constructor:

```solidity
constructor(address _admin, uint flags) {
    // slither-disable-next-line missing-zero-check
    admin = _admin;
    paused = flags;
}
```

`EvmErc20.sol` and `EvmErc20V2.sol` both forward the `admin` argument directly:

```solidity
constructor (string memory metadata_name, string memory metadata_symbol,
             uint8 metadata_decimals, address admin)
    ERC20(metadata_name, metadata_symbol)
    AdminControlled(admin, 0)   // no zero-check before this call
```

The `onlyAdmin` modifier is:

```solidity
modifier onlyAdmin {
    require(msg.sender == admin);
    _;
}
```

If `admin == address(0)`, `msg.sender` is never `address(0)` in any reachable EVM call, so `onlyAdmin` permanently reverts. This makes the following functions permanently inaccessible:

- `mint(address account, uint256 amount)` â€” the sole mechanism for crediting ERC-20 tokens when bridging NEP-141 assets from NEAR to EVM.
- `setMetadata(...)` â€” metadata update.
- `adminPause(uint flags)` â€” pause control.
- `adminSstore(...)`, `adminSendEth(...)`, `adminDelegatecall(...)` â€” low-level admin operations.

### Impact Explanation
`mint()` in `EvmErc20`/`EvmErc20V2` is the on-chain entry point called by the Aurora Engine to credit a user's EVM address when NEP-141 tokens are bridged from NEAR. If `admin` is `address(0)` at construction, `mint()` is permanently blocked. Any NEP-141 tokens locked on the NEAR side for bridging can never be represented on the EVM side; the corresponding ERC-20 tokens can never be minted. This constitutes a **permanent freeze of bridged funds** for the affected token contract. The `adminPause` path is also permanently inaccessible, removing the emergency-stop capability.

### Likelihood Explanation
When the Aurora Engine deploys `EvmErc20` via `deploy_erc20_token`, it passes its own account address as `admin`, so the engine itself would not trigger this in normal operation. However, the constructor provides zero protection against a deployment error or a direct EVM-side deployment with `admin = address(0)`.
