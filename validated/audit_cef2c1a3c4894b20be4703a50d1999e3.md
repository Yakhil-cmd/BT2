### Title
Pool Admin Bypasses `maxAdminSpreadFeeE6` Cap via Uncapped Per-Bin Additional Fees - (File: metric-core/contracts/MetricOmmPoolFactory.sol)

---

### Summary

The factory enforces `maxAdminSpreadFeeE6` to bound the total spread fee a pool admin may impose on traders. However, `setPoolBinAdditionalFees` passes `addFeeBuyE6` / `addFeeSellE6` directly to the pool with **no cap check**, allowing the pool admin to layer additional spread on top of the already-capped base fee and exceed the factory owner's intended limit on every affected bin.

---

### Finding Description

The factory maintains four fee caps enforced by the owner: [1](#0-0) 

`setPoolAdminFees` correctly validates the base admin spread component against `maxAdminSpreadFeeE6`: [2](#0-1) 

`setPoolBinAdditionalFees`, however, performs **no such validation** — it forwards the caller-supplied `uint16` values straight to the pool: [3](#0-2) 

The pool's `setBinAdditionalFees` only checks the bin index range, not the fee magnitudes: [4](#0-3) 

During every swap, the per-bin additional fee is **added on top of** the base spread fee before computing the effective bid/ask: [5](#0-4) [6](#0-5) 

`addFeeBuyE6` / `addFeeSellE6` are `uint16` fields, so their maximum value is **65,535** (≈ 6.55 % in E6 units): [7](#0-6) 

The effective spread seen by a trader in a given bin is therefore:

```
effectiveSpread = spreadFeeE6 (base, capped) + addFeeBuyE6/addFeeSellE6 (uncapped)
```

The factory owner can lower `maxAdminSpreadFeeE6` to any value (e.g., 10,000 = 1 %) to protect users, but the pool admin can still set per-bin additional fees up to 65,535 (6.55 %) on every bin, making the real effective spread up to **7.55 %** — far above the owner-imposed cap.

---

### Impact Explanation

The factory owner's `maxAdminSpreadFeeE6` cap is the sole on-chain mechanism preventing a pool admin from imposing excessive spread costs on traders. Because `setPoolBinAdditionalFees` is uncapped, the pool admin can silently exceed that limit on any or all bins. Traders receive less output than the factory-owner-sanctioned fee schedule permits, constituting a direct loss of user principal on every swap routed through the affected bins. The surplus from the inflated spread accrues to LPs; because the pool admin controls the pool and may also be an LP, this creates an indirect extraction path beyond the admin's authorized boundary.

---

### Likelihood Explanation

The pool admin is a semi-trusted role that can call `setPoolBinAdditionalFees` at any time with no timelock and no on-chain approval from the factory owner. The call requires only `onlyPoolAdmin(pool)` access. Any pool admin who wishes to extract additional value from traders, or who is simply misconfigured, can trigger this immediately after pool deployment.

---

### Recommendation

Add a cap check inside `setPoolBinAdditionalFees` (or inside `setBinAdditionalFees` on the pool) that enforces `addFeeBuyE6 + addFeeSellE6 <= maxAdminSpreadFeeE6` (or a dedicated `maxBinAdditionalFeeE6` cap). For example, in `MetricOmmPoolFactory.setPoolBinAdditionalFees`:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (addFeeBuyE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

Alternatively, introduce a dedicated `maxBinAdditionalFeeE6` cap that the factory owner can configure independently.

---

### Proof of Concept

1. Factory owner deploys factory and sets `maxAdminSpreadFeeE6 = 10_000` (1 %) to protect users.
2. Pool admin calls `setPoolAdminFees(pool, 10_000, 0)` — accepted, at the cap.
3. Pool admin calls `setPoolBinAdditionalFees(pool, 0, 65_535, 65_535)` — **no revert**; bin 0 now carries an additional 6.55 % buy and sell fee.
4. A trader swaps through bin 0. The effective spread applied is `10_000 + 65_535 = 75_535` E6 units ≈ **7.55 %**, more than 7× the factory owner's intended 1 % cap.
5. The excess spread surplus (≈ 6.55 % of notional) accrues to LPs in the pool, bypassing the factory owner's fee-protection boundary with no on-chain guard.
