Audit Report

## Title
Unbounded Neuron Iteration in SNS `distribute_rewards` Causes Permanent Instruction-Limit Trap, Blocking Reward Distribution - (File: `rs/sns/governance/src/governance.rs`)

## Summary

SNS governance's `distribute_rewards` function iterates over all ballots of all considered proposals and all voting neurons in a single synchronous execution with no instruction-limit guard or batching mechanism. The IC's 5-billion-instruction message limit can be exceeded when the neuron population and proposal backlog are large. Because the heartbeat is the only mechanism that calls `distribute_rewards` and no batched-timer fallback exists, a trap permanently halts reward distribution. NNS governance fixed this exact class of bug; SNS governance has no equivalent protection.

## Finding Description

**Root cause — `distribute_rewards` (no batching, no instruction guard):**

`rs/sns/governance/src/governance.rs` lines 5892–5997 contain a nested loop over all `considered_proposals` and, for each proposal, over all `proposal.ballots`. For each ballot entry the code performs `NeuronId::from_str`, a `HashMap` entry operation, and `Decimal` arithmetic. After the loop, a second pass iterates over every rewarded neuron to mutate `maturity_e8s_equivalent`. There is no call to any instruction-limit check, no checkpoint, and no continuation mechanism. [1](#0-0) 

`run_periodic_tasks` calls `self.distribute_rewards(supply)` synchronously after a single `await` for the ledger supply. The entire O(proposals × neurons) computation must complete within that one resumed message. [2](#0-1) 

**Worst-case bounds are governance-enforced ceilings:**

- `MAX_NUMBER_OF_NEURONS_CEILING` = 200,000
- `MAX_NUMBER_OF_PROPOSALS_WITH_BALLOTS_CEILING` = 700

Worst-case ballot entries: 700 × 200,000 = 140 million. At even 50 instructions per entry (HashMap insert + Decimal add + string parse), that is 7 billion instructions — 40% over the 5-billion limit. [3](#0-2) 

**Contrast with NNS governance (fixed):**

NNS governance's `continue_processing` in `rs/nns/governance/src/reward/distribution.rs` checks `is_over_instructions_limit()` after each neuron update and breaks, resuming in the next timer invocation. [4](#0-3) 

A grep for `is_message_over_threshold`, `is_over_instructions_limit`, and `over_threshold` across all of `rs/sns/governance/**` returns zero matches, confirming no equivalent guard exists in SNS governance.

**Trap is permanent:**

When the heartbeat traps on instruction exhaustion, the IC rolls back all state changes. `should_distribute_rewards` therefore still returns `true` on the next heartbeat invocation, which triggers the same unbounded computation, traps again, and so on indefinitely. The reward round is never marked complete; no neuron ever receives its maturity increment for that period or any subsequent period.

## Impact Explanation

This is an application/platform-level DoS on SNS governance with concrete, permanent user harm: all voting neurons permanently stop accruing maturity (voting rewards). This matches the allowed High impact: *"Significant SNS security impact with concrete user or protocol harm"* and *"Application/platform-level DoS … not based on raw volumetric DDoS."* Severity: **High ($2,000–$10,000)**.

## Likelihood Explanation

Staking SNS tokens and claiming neurons via `manage_neuron → ClaimOrRefresh` is an unprivileged, standard user action. No attacker coordination is required; organic growth of a successful SNS toward its configured `max_number_of_neurons` ceiling is sufficient. The condition worsens monotonically and is irreversible once triggered. Any SNS that configures a high `max_number_of_neurons` (up to the 200,000 ceiling) and accumulates normal proposal activity is at risk.

## Recommendation

1. Move SNS `distribute_rewards` out of the synchronous `run_periodic_tasks` path and into a dedicated periodic timer task, mirroring the NNS fix documented in `rs/nns/governance/CHANGELOG.md`.
2. Add an instruction-counter guard (equivalent to `is_over_instructions_limit`) inside the reward-distribution loop so that work is broken across multiple messages, resuming from a stable checkpoint each time.
3. Apply a similar guard or snapshot-based approach to `compute_ballots_for_new_proposal` (lines 5255–5280), which also iterates over all neurons synchronously during proposal submission.

## Proof of Concept

1. Deploy an SNS with `max_number_of_neurons` set to 200,000 (within the allowed ceiling).
2. Have users stake SNS tokens and call `manage_neuron → ClaimOrRefresh` until the neuron count approaches 200,000. This is an unprivileged operation.
3. Submit and allow 700 proposals to reach `ReadyToSettle` state. Each proposal records a ballot for every eligible neuron at creation time.
4. When the reward round ends, `run_periodic_tasks` calls `distribute_rewards`. The function must iterate over 700 × 200,000 = 140 million ballot entries plus update maturity for every rewarded neuron — exceeding the 5-billion-instruction message limit.
5. The heartbeat traps; state is rolled back; `should_distribute_rewards` remains `true`; the trap recurs on every subsequent heartbeat. Reward distribution is permanently frozen.

The NNS canbench result of 2,450,000 instructions for `compute_ballots_for_new_proposal_with_stable_neurons` (implying ~24,500 instructions per neuron for ballot computation alone) confirms that per-neuron instruction costs are non-trivial, and `distribute_rewards` performs additional work (Decimal arithmetic, map operations, neuron mutation) per ballot entry beyond what ballot creation requires.
