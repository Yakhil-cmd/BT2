### Title
Stale updater permissions persist after `transferProviderOwnership`, allowing unauthorized `confidenceParam` manipulation — (File: smart-contracts-poc/contracts/PriceProviderFactory.sol)

### Summary
`PriceProviderFactory.transferProviderOwnership` and `PriceProviderFactoryL2.transferProviderOwnership` transfer provider ownership without clearing any `isUpdater[provider][...]` entries granted by the previous owner. Every address the previous owner authorized as an updater retains the ability to call `setConfidence` and manipulate `confidenceParam`, which directly controls the bid/ask spread delivered to every pool that uses that price provider.

### Finding Description
`transferProviderOwnership` updates `providerOwner[provider]` and adjusts the `_providersByCreator` enumerable sets, but never touches the `isUpdater` mapping: [1](#0-0) 

The authorization check `_requireUpdater` passes for any address where `isUpdater[provider][addr] == true`, regardless of whether the *current* owner granted that permission: [2](#0-1) 

Because `isUpdater` is a plain nested mapping with no enumerable counterpart, the new owner has no on-chain mechanism to discover which addresses were granted by the previous owner. They cannot revoke stale permissions without knowing each address individually.

The identical pattern exists in `PriceProviderFactoryL2`: [3](#0-2) [4](#0-3) 

`setConfidence` calls `setConfidenceParam` on the provider, which updates `confidenceParam`: [5](#0-4) [6](#0-5) 

`confidenceParam` is a direct multiplier on the oracle spread that feeds into every bid/ask computation: [7](#0-6) 

### Impact Explanation
A stale updater can:

1. **Widen the spread** — set `confidenceParam` to `CONFIDENCE_MAX` (1,000,000). With a typical oracle spread of 500 bps, the adjusted spread becomes 500 × 1,000,000 = 500,000,000, widening the bid/ask by up to ~5% of mid-price. Every swap through a pool using this provider executes at a materially worse price than the oracle intends — a direct bad-price execution impact on traders.

2. **Narrow the spread to zero** — set `confidenceParam` to 0. For pools whose price provider was deployed with a negative `marginStep`, this collapses the confidence-derived spread below the step inversion threshold, causing `getBidAndAskPrice` to revert with `FeedStalled` and making the pool completely unusable for swaps (broken core pool functionality).

### Likelihood Explanation
Medium. Provider ownership transfers are an expected operational event (e.g., handing a provider from a deployer to a DAO multisig). Any updater bots or addresses authorized by the previous owner automatically carry over. The new owner has no enumeration path to discover them, so stale permissions persist indefinitely unless the previous owner proactively discloses every address it granted.

### Recommendation
Maintain an `EnumerableSet.AddressSet` of updaters per provider alongside the `isUpdater` mapping. On `transferProviderOwnership`, iterate the set and delete every `isUpdater[provider][addr]` entry before clearing the set

    function setConfidenceParam(uint256 newValue) external {
        require(msg.sender == factory, OnlyFactory());
        if (newValue > CONFIDENCE_MAX) {
            revert ConfidenceParamOutOfBounds();
        }
        if (block.timestamp < lastConfidenceUpdate + CONFIDENCE_COOLDOWN) {
            revert CooldownNotElapsed();
        }

        confidenceParam = newValue;
        lastConfidenceUpdate = block.timestamp;
        emit ConfidenceParamSet(newValue);
    }
