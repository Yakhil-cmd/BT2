### Title
Missing Binding Between Signature and COA Identity in `validateCOAOwnershipProof` Enables Cross-COA Ownership Proof Forgery - (File: `fvm/evm/stdlib/contract.cdc`)

---

### Summary

`EVM.validateCOAOwnershipProof` verifies that a Flow account's keys signed `signedData`, then separately borrows a `CadenceOwnedAccount` capability at a user-supplied `path` and checks its EVM address against a user-supplied `evmAddress`. Neither `path` nor `evmAddress` is included in the signed payload. This is the direct analog of the original report's missing `cap.namespace_addr == manage_cap.namespace_addr` check: two identity-bearing objects (the signature credential and the COA capability path) are accepted together without verifying they are bound to the same namespace/identity.

---

### Finding Description

`EVM.validateCOAOwnershipProof` is declared `access(all)` and accepts six parameters:

```
address: Address        // Flow account whose keys are checked
path: PublicPath        // user-supplied path to borrow the COA capability
signedData: [UInt8]     // the data that was signed
keyIndices: [UInt64]
signatures: [[UInt8]]
evmAddress: [UInt8; 20] // user-supplied EVM address to match against
```

The function:
1. Verifies the signatures over `signedData` using keys from `address`.
2. Borrows `acc.capabilities.borrow<&EVM.CadenceOwnedAccount>(path)` using the user-supplied `path`.
3. Checks that the borrowed COA's EVM address equals `evmAddress`.

The signature covers only `signedData`. It does **not** commit to `path` or `evmAddress`. The code itself documents this gap:

> *"Note: this function does not enforce that `signedData` includes `evmAddress`. In principle, a signature produced for one purpose could be replayed here against a different COA owned by the same Cadence account."*

Because `validateCOAOwnershipProof` is `access(all)`, any Cadence script or transaction can call it with arbitrary `path` and `evmAddress` values. An attacker who possesses any valid `"FLOW-V0.0-user"`-tagged signature from a victim's account can:

1. Set `address` to the victim's Flow address.
2. Set `path` to any public path on the victim's account that exposes a `CadenceOwnedAccount` capability.
3. Set `evmAddress` to the EVM address of the COA at that path.
4. Supply the victim's previously obtained signature over any `signedData`.

The function will return `ValidationResult(isValid: true)` for the COA at `path`, even though the victim never signed for that specific COA.

This is structurally identical to the original report: the original missing check was `cap.namespace_addr == manage_cap.namespace_addr` — two objects accepted together without verifying they belong to the same namespace. Here, the missing check is that `signedData` must commit to `evmAddress` (and `path`) — the signature credential and the COA identity are accepted together without verifying they are bound to each other. ### Impact Explanation

`validateCOAOwnershipProof` is the Cadence-side implementation invoked by the `verifyCOAOwnershipProof` EVM precompile, which is the backbone of ERC-1271 (`isValidSignature`) for COA accounts. Any EVM smart contract that calls `isValidSignature` on a COA — for example to authorize a multisig action, a permit, or a governance vote — relies on this function returning the correct result.

Because `validateCOAOwnershipProof` is `access(all)`, it is also directly callable from Cadence scripts and transactions. Any Cadence-layer protocol or contract that calls it and trusts the `ValidationResult` is equally affected.

An attacker who forges a `ValidationResult(isValid: true)` for a victim's COA can impersonate the victim in any EVM or Cadence context that relies on COA ownership verification, potentially authorizing on-chain asset transfers, governance actions, or bridge operations that require the victim's COA signature. ### Likelihood Explanation

The attacker needs a valid `"FLOW-V0.0-user"`-tagged signature from the victim's account. This is achievable in two realistic ways:

1. **Cross-COA replay**: A victim with multiple COAs signs a legitimate ownership proof for COA_A. The attacker intercepts or observes the encoded proof (which is transmitted as calldata in an EVM transaction and is therefore public), extracts the signature, and replays it against COA_B owned by the same victim by substituting `path` and `evmAddress`.

2. **Cross-purpose replay**: Flow transaction envelopes are signed with the same `"FLOW-V0.0-user"` domain separation tag. If the 32-byte `signedData` field in a COA proof happens to match bytes from a past transaction payload signed by the victim, the signature is reusable. While this requires a hash collision in practice, the domain tag provides no isolation between transaction signatures and COA proof signatures.

The EVM precompile path (`verifyCOAOwnershipProof`) partially mitigates this by fixing `evmAddress` to `address(this)` (the calling contract's EVM address), which prevents an attacker from substituting a different `evmAddress` in that specific call path. However, the direct Cadence call path — which is `access(all)` — has no such constraint, and any Cadence consumer of `validateCOAOwnershipProof` is fully exposed. [6](#0-5) [7](#0-6) 

---

### Recommendation

Bind the signature to the specific COA being claimed by requiring `signedData` to encode `evmAddress` (and optionally `path`). The `validateCOAOwnershipProof` function should enforce:

```cadence
// Require that signedData encodes evmAddress as its first 20 bytes
// (or via a structured encoding), so the signature is COA-specific.
pre {
    signedData.length >= 20:
        "signedData must encode evmAddress"
    // verify first 20 bytes of signedData match evmAddress
}
```

Alternatively, add a `pre` condition that rejects calls where `signedData` does not include the `evmAddress`, and document this as a hard requirement rather than a caller advisory. [8](#0-7)

### Proof of Concept

**Setup**: Victim account `0xVICTIM` has two COAs:
- COA_A stored at `/storage/coaA`, published at `/public/coaA`, EVM address `0xAAAA...`
- COA_B stored at `/storage/coaB`, published at `/public/coaB`, EVM address `0xBBBB...`

**Step 1**: Victim legitimately signs a COA ownership proof for COA_A. The encoded proof contains `{KeyIndices: [0], Address: 0xVICTIM, CapabilityPath: "coaA", Signatures: [sig]}` where `sig` is a valid signature over `signedData = H` using the victim's key.

**Step 2**: Attacker observes the proof (it is public calldata in the EVM transaction).

**Step 3**: Attacker constructs a forged proof: `{KeyIndices: [0], Address: 0xVICTIM, CapabilityPath: "coaB", Signatures: [sig]}` — identical except `CapabilityPath` is changed to `"coaB"`.

**Step 4**: Attacker calls directly from Cadence:
```cadence
import EVM from 0x...
let result = EVM.validateCOAOwnershipProof(
    address: 0xVICTIM,
    path: /public/coaB,   // ← substituted
    signedData: H,
    keyIndices: [0],
    signatures: [sig],    // ← victim's original signature, unchanged
    evmAddress: [0xBB, 0xBB, ...]  // ← COA_B's address
)
// result.isValid == true  ← forged
```

**Step 5**: The function verifies `sig` over `H` using the victim's key (valid), borrows the COA at `/public/coaB`, checks its EVM address matches `0xBBBB...` (matches), and returns `ValidationResult(isValid: true)` — falsely certifying that the victim signed `H` for COA_B. [9](#0-8) [10](#0-9)
