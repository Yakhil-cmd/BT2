### Title
Presigning Participant-Set Not Validated Against Triple Generation Participants, Enabling Nonce Reuse and Key Extraction — (`src/ecdsa/ot_based_ecdsa/presign.rs`)

---

### Summary

The OT-based ECDSA `presign()` function explicitly omits the check that the presigning participant set matches the participant set recorded in `TriplePub.participants`. A malicious coordinator can present the same Beaver triple to two different presigning sessions with different (overlapping) participant subsets. Because threshold-sharing guarantees that any valid subset reconstructs the same underlying nonce scalar `k`, both sessions produce presignatures with the same `big_r`. If the coordinator then rerandomizes both presignatures with the same entropy, the final signatures share an identical nonce, enabling full private-key extraction via standard ECDSA nonce-reuse algebra.

---

### Finding Description

**Root cause — explicit omission in `presign()`:** [1](#0-0) 

```rust
// NOTE: We omit the check that the new participant set was present for
// the triple generation, because presumably they need to have been present
// in order to have shares.

// Also check that we have enough participants to reconstruct shares.
if args.threshold != args.triple0.1.threshold || args.threshold != args.triple1.1.threshold {
    return Err(InitializationError::BadParameters(
        "New threshold must match the threshold of both triples".to_string(),
    ));
}
```

The code checks that the threshold matches, but never reads `TriplePub.participants`: [2](#0-1) 

```rust
pub struct TriplePub {
    pub big_a: AffinePoint,
    pub big_b: AffinePoint,
    pub big_c: AffinePoint,
    /// The participants in generating this triple.
    pub participants: Vec<Participant>,
    /// The threshold which will be able to reconstruct it.
    pub threshold: ReconstructionLowerBound,
}
```

The `participants` field is populated during triple generation: [3](#0-2) 

but is never validated against the presigning participant list anywhere in `presign.rs` or `do_presign`.

**Why this produces identical nonces:**

Suppose triples are generated by `{P1, P2, P3}` with threshold 2. The shares `(k_1, k_2, k_3)` satisfy the Shamir property: any 2-subset reconstructs the same secret `k`.

In presigning session A with `{P1, P2}`: [4](#0-3) 

```rust
let lambda_me = participants.lagrange::<Secp256>(me)?;
let k_prime_i = lambda_me * k_i;
```

`k_A = λ₁({P1,P2})·k₁ + λ₂({P1,P2})·k₂ = k`

In presigning session B with `{P1, P3}` using the **same triple**:

`k_B = λ₁({P1,P3})·k₁ + λ₃({P1,P3})·k₃ = k`

Both sessions produce `big_r = k⁻¹·G` — the same nonce commitment.

**Rerandomization does not save you:**

The rerandomization step multiplies `big_r` by a `delta` derived via HKDF from `(participants, big_r, entropy, tweak, msg_hash)`: [5](#0-4) 

If the coordinator supplies the same `entropy` to both sessions (which they control), `delta_A = delta_B`, and the rerandomized nonces are identical — direct nonce reuse. Even with different entropy, the coordinator knows both `delta` values (all inputs are public or coordinator-supplied), so the multiplicative relationship `k_A_rerandomized / k_B_rerandomized = delta_B / delta_A` is known, which is sufficient for standard related-nonce key recovery.

**Orchestration documentation acknowledges the risk but does not enforce it in code:** [6](#0-5) 

> It's **critical** that the output is then destroyed, so that no other group of parties attempts to re-use that output for another phase. … Otherwise, you might have two independent groups of parties trying to use the same output, which is bad.

This is documented as a caller obligation but is not enforced by any code-level check.

---

### Impact Explanation

A malicious coordinator can extract the aggregate private signing key `x` by:

1. Generating triples with `{P1, P2, P3}` (threshold 2).
2. Running presigning session A with `{P1, P2}` → presignature with nonce `k`.
3. Running presigning session B with `{P1, P3}` using the **same triple** → presignature with the same nonce `k`.
4. Rerandomizing both with the same entropy → identical rerandomized nonces.
5. Requesting signatures on two different messages → two ECDSA signatures `(R, s₁)` and `(R, s₂)` with the same `R`.
6. Applying standard nonce-reuse recovery: `x = (s₁·h₂ - s₂·h₁) / (s₂ - s₁) · R_x⁻¹`.

This maps to the **Critical** allowed impact: *Extraction, reconstruction, or disclosure of private signing shares or aggregate secret material.*

---

### Likelihood Explanation

The malicious coordinator role is explicitly within scope per the library's trust model. The coordinator controls which triples are assigned to which presigning session and which participants are invited. No out-of-band agreement or privileged key material is required beyond the coordinator's normal orchestration authority. The `TriplePub` is public, so the coordinator can freely present it to multiple sessions. The attack requires only two presigning sessions and two signing calls.

---

### Recommendation

1. **Enforce participant-set binding at presign initialization.** Add a check that the presigning `participants` list exactly matches `args.triple0.1.participants` and `args.triple1.1.participants`. Remove the comment that excuses the omission.

```rust
// Validate that presigning participants match triple generation participants
if participants.as_slice() != args.triple0.1.participants.as_slice()
    || participants.as_slice() != args.triple1.1.participants.as_slice()
{
    return Err(InitializationError::BadParameters(
        "Presigning participants must exactly match triple generation participants".to_string(),
    ));
}
```

2. **Bind the triple identity into the presigning transcript.** Include `TriplePub.big_a`, `TriplePub.big_b`, `TriplePub.big_c` in the rerandomization HKDF input so that any attempt to reuse the same triple across sessions with different participant sets produces a detectable divergence.

3. **Document and enforce single-use at the API level.** The orchestration docs already state triples must be destroyed after use; this should be enforced by consuming `TripleShare` by value (it already is) and by making `TriplePub` non-`Clone` or wrapping it in a single-use token type.

### Proof of Concept

```
Setup:
  - Participants: P1 (id=1), P2 (id=2), P3 (id=3), threshold=2
  - Run triple generation with {P1, P2, P3} → (triple_share_1, triple_share_2, triple_share_3, triple_pub)
  - triple_pub.participants = [P1, P2, P3]

Attack:
  Session A:
    presign(participants=[P1, P2], me=P1,
            args=PresignArguments { triple0=(triple_share_1, triple_pub), ... })
    presign(participants=[P1, P2], me=P2,
            args=PresignArguments { triple0=(triple_share_2, triple_pub), ... })
    → presign_output_A.big_r = R

  Session B (same triple, different subset):
    presign(participants=[P1, P3], me=P1,
            args=PresignArguments { triple0=(triple_share_1, triple_pub), ... })
    presign(participants=[P1, P3], me=P3,
            args=PresignArguments { triple0=(triple_share_3, triple_pub), ... })
    → presign_output_B.big_r = R  ← identical to session A

  Rerandomize both with same entropy:
    rerandomize_presign(presign_output_A, args_with_same_entropy) → rerandomized_A
    rerandomize_presign(presign_output_B, args_with_same_entropy) → rerandomized_B
    → rerandomized_A.big_r == rerandomized_B.big_r

  Sign two different messages:
    sign(..., presignature=rerandomized_A, msg_hash=h1) → (R', s1)
    sign(..., presignature=rerandomized_B, msg_hash=h2) → (R', s2)

  Key recovery:
    x = (s1*h2 - s2*h1) / ((s2 - s1) * R'_x)
```

```rust
    pub fn rerandomize_presign(
        presignature: &PresignOutput,
        args: &RerandomizationArguments,
    ) -> Result<Self, ProtocolError> {
        if presignature.big_r != args.big_r {
            return Err(ProtocolError::IncompatibleRerandomizationInputs);
        }
        let delta = args.derive_randomness()?;
        if delta.is_zero().into() {
            return Err(ProtocolError::ZeroScalar);
        }

        // cannot be zero due to the previous check
        let inv_delta = delta.invert().unwrap();

        // delta . R
        let rerandomized_big_r = presignature.big_r * delta;

        //  (sigma + tweak * k) * delta^{-1}
        let rerandomized_sigma =
            (presignature.sigma + args.tweak.value() * presignature.k) * inv_delta;

        // k * delta^{-1}
        let rerandomized_k = presignature.k * inv_delta;

        Ok(Self {
            big_r: rerandomized_big_r.into(),
            k: rerandomized_k,
            sigma: rerandomized_sigma,
        })
    }
```
