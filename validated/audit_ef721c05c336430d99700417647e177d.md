### Title
`OracleValueStopLossExtension.proposeOracleStopLossTimelock` Lacks Upper Bound, Enabling uint32 Overflow to Bypass All Future Timelocks - (File: metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol)

### Summary
`proposeOracleStopLossTimelock` accepts any `uint32` value for `newTimelock` with no upper-bound check. Setting `newTimelock = type(uint32).max` causes `_afterTimelock` to overflow its `uint32` return, producing an `executeAfter` timestamp already in the past. Once this timelock is committed, every subsequent `proposeOracleStopLoss*` call computes an `executeAfter` that is already elapsed, allowing the pool admin to immediately execute drawdown, decay, and watermark changes with zero LP reaction window.

### Finding Description

`proposeOracleStopLossTimelock` stores the caller-supplied value without any validation:

```solidity
// OracleValueStopLossExtension.sol:78-84
function proposeOracleStopLossTimelock(address pool_, uint32 newTimelock) external onlyPoolAdmin(pool_) {
    PoolStopLossSchedule storage sched = _initializedSchedule(pool_);
    uint32 executeAfter = _afterTimelock(pool_);
    sched.pendingTimelock = newTimelock;
    sched.pendingTimelockExecuteAfter = executeAfter;
    emit OracleStopLossTimelockProposed(pool_, newTimelock, executeAfter);
}
``` [1](#0-0) 

`_afterTimelock` computes the deadline as a `uint32` cast of a `uint256` sum:

```solidity
// OracleValueStopLossExtension.sol:297-299
function _afterTimelock(address pool_) private view returns (uint32) {
    return uint32(block.timestamp + oracleStopLossConfig[pool_].timelock);
}
``` [2](#0-1) 

The overflow arithmetic when `timelock = type(uint32).max`:

| Variable | Value |
|---|---|
| `block.timestamp` (July 2026) | ≈ 1,753,000,000 |
| `type(uint32).max` | 4,294,967,295 |
| Sum (uint256) | 6,047,967,295 |
| `uint32(sum)` | **1,752,999,999** (≈ `block.timestamp − 1`) |

`_requireElapsed` then passes immediately because `block.timestamp ≥ 1,752,999,999`:

```solidity
// OracleValueStopLossExtension.sol:301-303
function _requireElapsed(uint32 executeAfter) private view {
    if (block.timestamp < executeAfter) revert OracleStopLossTimelockNotElapsed(executeAfter, block.timestamp);
}
``` [3](#0-2) 

After the overflowed timelock is committed via `executeOracleStopLossTimelock`, every future call to `proposeOracleStopLossDrawdown`, `proposeOracleStopLossDecay`, and `proposeOracleStopLossHighWatermarks` computes `executeAfter` via the same `_afterTimelock`, which now always returns a past timestamp. The admin can call `execute*` in the same transaction as `propose*`.

The same absence of validation exists in `initialize`, where `timelock` is decoded from `data` without any bound check:

```solidity
// OracleValueStopLossExtension.sol:56-62
(uint32 drawdownE6, uint32 decayPerSecondE8, uint32 timelock) = abi.decode(data, (uint32, uint32, uint32));
_validateDrawdown(drawdownE6);
_validateDecay(decayPerSecondE8);
// timelock: no validation
``` [4](#0-3) 

Note that `drawdownE6` and `decayPerSecondE8` both have explicit validators (`_validateDrawdown`, `_validateDecay`), but `timelock` has none. [5](#0-4) 

### Impact Explanation

Once the overflowed timelock is active, the pool admin can:

1. **Disable stop-loss immediately** — call `proposeOracleStopLossDrawdown(pool, 0)` then `executeOracleStopLossDrawdown(pool)` in the same block. `drawdownE6 = 0` causes `_afterSwapOracleStopLoss` to return early (`if (drawdown == 0) return`), silently removing all LP value-loss protection with no warning window.

2. **Brick the pool** — call `proposeOracleStopLossHighWatermarks(pool, bin, 0, 0)` then `executeOracleStopLossHighWatermarks(pool)` immediately. Watermarks of 0 cause every subsequent swap to revert with `OracleStopLossTriggered`, making the pool permanently unusable for swaps while LP funds remain locked. [6](#0-5) 

The stop-loss extension's entire security model rests on the timelock giving LPs time to observe parameter changes and exit. Bypassing it removes the only on-chain protection against a malicious or compromised pool admin silently degrading LP safety guarantees.

### Likelihood Explanation

The pool admin is semi-trusted within defined caps. The exploit requires:
1. Waiting through one current-timelock period to commit `type(uint32).max` (if the current timelock is non-zero).
2. If the pool was initialized with `timelock = 0` (valid, no validation prevents it), the entire exploit is executable atomically in a single block.

The trigger is a valid, permissioned call path — no external oracle manipulation or flash loan is needed.

### Recommendation

Add an upper-bound check in both `proposeOracleStopLossTimelock` and `initialize`:

```solidity
uint32 public constant MAX_STOP_LOSS_TIMELOCK = 30 days;

function proposeOracleStopLossTimelock(address pool_, uint32 newTimelock) external onlyPoolAdmin(pool_) {
    if (newTimelock > MAX_STOP_LOSS_TIMELOCK) revert TimelockTooLarge(newTimelock);
    ...
}
```

Apply the same guard in `initialize` before storing `timelock`. This mirrors the pattern already used for `drawdownE6` and `decayPerSecondE8`.

### Proof of Concept

```
// Pool initialized with timelock = 0 (no validation prevents this)
// Step 1: Admin proposes type(uint32).max timelock
//   executeAfter = uint32(block.timestamp + 0) = block.timestamp → immediately elapsed
proposeOracleStopLossTimelock(pool, type(uint32).max)
executeOracleStopLossTimelock(pool)
// oracleStopLossConfig[pool].timelock = 4,294,967,295

// Step 2: Admin proposes drawdown = 0 (disable stop-loss)
//   executeAfter = uint32(1_753_000_000 + 4_294_967_295) = uint32(6_047_967_295) = 1_752_999_999
//   block.timestamp (1_753_000_000) >= 1_752_999_999 → _requireElapsed passes
proposeOracleStopLossDrawdown(pool, 0)
executeOracleStopLossDrawdown(pool)
// drawdownE6 = 0 → stop-loss silently disabled, no LP reaction window
