### Title
Silo Whitelist Bypass via `call` Entrypoint — (`engine/src/contract_methods/evm_transactions.rs`)

### Summary
The Aurora Engine's Silo mode enforces an address/account whitelist on the `submit` entrypoint via `assert_access`, but the `call` entrypoint executes EVM transactions with no whitelist check at all. Any NEAR account — regardless of whitelist status — can invoke `call` to execute arbitrary EVM logic, completely circumventing the Silo access-control system.

### Finding Description

The `submit` and `submit_with_args` entrypoints both invoke `engine::submit_with_alt_modexp`, which calls `assert_access` before executing any EVM code:

```rust
// engine/src/engine.rs
fn assert_access<I: IO + Copy, E: Env>(
    io: &I,
    env: &E,
    transaction: &NormalizedEthTransaction,
) -> Result<(), EngineError> {
    let allowed = if transaction.to.is_some() {
        silo::is_allow_submit(io, &env.predecessor_account_id(), &transaction.address)
    } else {
        silo::is_allow_deploy(io, &env.predecessor_account_id(), &transaction.address)
    };
    if !allowed {
        return Err(EngineError { kind: EngineErrorKind::NotAllowed, gas_used: 0 });
    }
    Ok(())
}
```

`is_allow_submit` checks both the NEAR predecessor account (Account whitelist) and the EVM address (Address whitelist):

```rust
// engine/src/contract_methods/silo/mod.rs
pub fn is_allow_submit<I: IO + Copy>(io: &I, account: &AccountId, address: &Address) -> bool {
    is_address_allowed(io, address) && is_account_allowed(io, account)
}
```

However, the `call` entrypoint in `evm_transactions.rs` performs **no such check**:

```rust
// engine/src/contract_methods/evm_transactions.rs
pub fn call<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I, env: &E, handler: &mut H,
) -> Result<SubmitResult, ContractError> {
    with_logs_hashchain(io, env, function_name!(), |mut io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        // ← No assert_access / whitelist check here
        let args = CallArgs::deserialize(&bytes).ok_or(errors::ERR_BORSH_DESERIALIZE)?;
        let mut engine = Engine::new_with_state(
            state,
            predecessor_address(&predecessor_account_id), // EVM origin from NEAR account
            current_account_id, io, env,
        );
        let result = engine.call_with_args(args, handler)?;
        ...
    })
}
```

The EVM origin for `call` is `predecessor_address(&predecessor_account_id)` — a deterministic EVM address derived from the NEAR caller's account ID. This means any NEAR account can call any EVM contract as that derived address, with no whitelist enforcement.

The same omission applies to `deploy_code`, which also skips `assert_access`.

### Impact Explanation

In Silo mode, the whitelist is the primary mechanism for restricting which NEAR accounts and EVM addresses may interact with the EVM. The `call` bypass renders both the Account whitelist and the Address whitelist completely ineffective for any caller using the `call` entrypoint.

Concrete impact:
- An unauthorized NEAR account can call any EVM contract (token transfers, DeFi interactions, etc.) inside the Silo.
- If the attacker's derived EVM address holds balances (e.g., received via `ft_on_transfer` from a whitelisted party), they can transfer those funds out without being whitelisted.
- Unauthorized contract deployment is also possible via `deploy_code`.

This constitutes a **High** severity whitelist bypass with a realistic path to fund theft (theft of tokens held at the attacker's derived EVM address) and unauthorized EVM state mutation.

### Likelihood Explanation

The `call` entrypoint is a standard, publicly documented NEAR method on the Aurora contract. Any NEAR account can invoke it with 1 yoctoNEAR. No special privileges, leaked keys, or social engineering are required. The only precondition is that the Silo operator has enabled the whitelist (which is the intended production configuration for Silo deployments). Likelihood is **High**.

### Recommendation

Apply the same `assert_access` guard to the `call` (and `deploy_code`) entrypoints as is applied inside `submit_with_alt_modexp`. Specifically, derive the EVM address from `predecessor_address(&env.predecessor_account_id())` and construct a synthetic `NormalizedEthTransaction`-equivalent check, or refactor `assert_access` to accept the NEAR account and EVM address directly:

```rust
// In call():
let predecessor_account_id = env.predecessor_account_id();
let evm_address = predecessor_address(&predecessor_account_id);
if !silo::is_allow_submit(&io, &predecessor_account_id, &evm_address) {
    return Err(errors::ERR_NOT_ALLOWED.into());
}
```

### Proof of Concept

1. Silo operator enables whitelists: Account whitelist and Address whitelist are both active.
2. Attacker controls NEAR account `attacker.near` (not in Account whitelist). Its derived EVM address `predecessor_address("attacker.near")` is not in the Address whitelist.
3. Attacker calls `submit` with a signed Ethereum transaction → rejected by `assert_access` with `NotAllowed`.
4. Attacker calls `call` with `CallArgs { contract: <token_contract>, input: <transfer_calldata> }` → **succeeds**, because `call` never invokes `assert_access`.
5. The EVM executes the call with origin = `predecessor_address("attacker.near")`, bypassing all Silo whitelist restrictions.

pub fn call<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<SubmitResult, ContractError> {
    with_logs_hashchain(io, env, function_name!(), |mut io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        let bytes = io.read_input().to_vec();
        let args = CallArgs::deserialize(&bytes).ok_or(errors::ERR_BORSH_DESERIALIZE)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();

        let mut engine: Engine<_, E, AuroraModExp> = Engine::new_with_state(
            state,
            predecessor_address(&predecessor_account_id),
            current_account_id,
            io,
            env,
        );
        let result = engine.call_with_args(args, handler)?;
        let result_bytes = borsh::to_vec(&result).map_err(|_| errors::ERR_SERIALIZE)?;
        io.return_output(&result_bytes);
        Ok(result)
    })
}
