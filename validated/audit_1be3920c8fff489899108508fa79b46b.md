### Title
Missing Participant-Set Validation in OT-Based ECDSA Presigning Allows Cross-Context Triple Injection — (`src/ecdsa/ot_based_ecdsa/presign.rs`)

---

### Summary

The `presign()` function in `src/ecdsa/ot_based_ecdsa/presign.rs` accepts `PresignArguments` containing two Beaver triples (`triple0`, `triple1`), each of which carries a `TriplePub` with a `participants` field recording which parties generated the triple. The function validates only that the threshold matches, but **explicitly omits** the check that the triple's participant set matches the presigning participant set. A malicious participant can inject triples generated by a different participant set into a presigning session, causing the protocol to produce corrupted outputs or abort for all honest parties.

---

### Finding Description

`TriplePub` records the participant set that generated the triple: [1](#0-0) 

`PresignArguments` bundles two such triples with a `KeygenOutput` and a threshold: [2](#0-1) 

Inside `presign()`, the only cross-field validation performed is a threshold equality check. The participant-set check is **explicitly skipped** with a comment: [3](#0-2) 

Inside `do_presign`, Lagrange coefficients are computed from the presigning `participants` list — not from the triple's embedded `participants` field: [4](#0-3) 

The triple shares (`k_i`, `e_i`, `a_i`, `b_i`, `c_i`) are polynomial evaluations at each participant's index, evaluated under the polynomial that was secret-shared for the **triple's** participant set. When a triple from participant set T is used in a presigning session with participant set P ≠ T, the Lagrange coefficients computed for P are wrong for shares generated under T. The resulting linearized values will not satisfy the public commitment check: [5](#0-4) 

Additionally, there is no check that `triple0.1.participants == triple1.1.participants`, so the two triples can themselves come from different participant sets, compounding the inconsistency.

The `TriplePub.participants` field is populated at triple generation time: [6](#0-5) 

but is **never read or validated** anywhere in `do_presign`.

---

### Impact Explanation

A malicious participant who has triple shares from a prior session with participant set T can inject those triples into a presigning session with participant set P ≠ T. Because the Lagrange linearization uses wrong coefficients, the verification check `alpha·G == K + A` and `beta·G == X + B` will fail, causing `ProtocolError::AssertionFailed` for all honest parties in the presigning session. The malicious participant can repeat this across every presigning attempt, permanently denying the ability to produce presignatures — and therefore signatures — for honest parties.

This maps to: **High: Permanent denial of signing for honest parties under valid protocol inputs.**

---

### Likelihood Explanation

Any participant who has participated in triple generation with one participant set (a common scenario given that triples are generated offline in advance, potentially with varying subsets) can trivially supply those triples to a presigning session with a different participant set. No cryptographic break is required. The only prerequisite is holding triple shares from any prior session — a normal operational artifact.

---

### Recommendation

Add an explicit check in `presign()` that the participant sets embedded in both triples match the presigning participant set, and that the two triples share the same participant set:

```rust
// After constructing `participants` (ParticipantList):
let triple0_participants: Vec<Participant> = args.triple0.1.participants.clone();
let triple1_participants: Vec<Participant> = args.triple1.1.participants.clone();

if triple0_participants != triple1_participants {
    return Err(InitializationError::BadParameters(
        "triple0 and triple1 must have been generated by the same participant set".to_string(),
    ));
}

// Verify the triple participant set matches the presigning participant set
let triple_participant_list = ParticipantList::new(&triple0_participants)
    .ok_or(InitializationError::DuplicateParticipants)?;
if triple_participant_list != participants {
    return Err(InitializationError::BadParameters(
        "triples must have been generated by the same participant set as the presigning session".to_string(),
    ));
}
```

This mirrors the fix recommended in the external report: validate that the entity passed in (triple / term) belongs to the same context (participant set / market) as the session it is being used in.

---

### Proof of Concept

**Setup:**
- Participant sets: T = {P0, P1, P2} and P = {P0, P1, P3}
- Triples are generated offline by T, producing `(TripleShare_T, TriplePub_T)` where `TriplePub_T.participants = [P0, P1, P2]`
- A presigning session is initiated with participant set P = {P0, P1, P3}

**Attack:**
- Malicious participant P0 (present in both sets) supplies `PresignArguments { triple0: (share_T, pub_T), triple1: (share_T2, pub_T2), ... }` — triples from T — into the presigning session for P
- `presign()` accepts this: threshold matches, participant-set check is omitted
- Inside `do_presign`, Lagrange coefficients are computed for P = {P0, P1, P3}, but the triple shares are evaluations of polynomials secret-shared for T = {P0, P1, P2}
- The linearized sums `alpha` and `beta` do not satisfy `alpha·G == K + A` or `beta·G == X + B`
- All honest parties receive `ProtocolError::AssertionFailed("received incorrect shares of additive triple phase.")`
- The presigning session is aborted; no presignature is produced
- P0 can repeat this for every presigning attempt, permanently blocking signing [3](#0-2) [1](#0-0)
