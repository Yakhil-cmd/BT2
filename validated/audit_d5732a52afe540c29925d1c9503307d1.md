### Title
Unprivileged Overwrite of Any Account's Registered Encryption Key — (`x/e2ee/keeper/keeper.go`, `x/e2ee/types/msg.go`)

### Summary

`MsgRegisterEncryptionKey` contains an `Address` field that is fully attacker-controlled. Neither `ValidateBasic` nor the keeper's `RegisterEncryptionKey` handler ever verifies that `msg.Address` equals the transaction signer. Any account can permanently overwrite any other account's registered encryption key.

---

### Finding Description

`ValidateBasic` in `msg.go` performs exactly two checks:

1. `msg.Address` is a valid bech32 string
2. `msg.Key` is a valid age X25519 recipient key [1](#0-0) 

There is no `GetSigners()` method and no `cosmos.msg.v1.signer` proto annotation anywhere in the e2ee module, so no SDK-level signer-equality check is enforced either. [2](#0-1) 

The keeper's message handler directly uses `req.Address` as the storage key without extracting or comparing the transaction signer:

```go
func (k Keeper) RegisterEncryptionKey(
    ctx context.Context,
    req *types.MsgRegisterEncryptionKey,
) (*types.MsgRegisterEncryptionKeyResponse, error) {
    if err := k.registerEncryptionKey(ctx, req.Address, []byte(req.Key)); err != nil {
        return nil, err
    }
    return &types.MsgRegisterEncryptionKeyResponse{}, nil
}
``` [3](#0-2) 

The underlying `registerEncryptionKey` unconditionally writes `key` to the store slot for `address`: [4](#0-3) 

---

### Impact Explanation

An attacker submits:

```
MsgRegisterEncryptionKey{
    Address: victimAddress,   // any valid bech32 address
    Key:     attackerAgeKey,  // attacker-controlled age public key
}
```

After the tx is included:
- `Key(victimAddress)` returns `attackerAgeKey`
- Any sender using the on-chain key registry to encrypt a message to the victim will encrypt it to the attacker's key — the attacker can decrypt it, the victim cannot
- The victim's legitimate key is permanently overwritten; they cannot restore it without submitting another tx, which the attacker can immediately overwrite again in a griefing loop

This is **corruption of e2ee key/message state with direct security impact** (High per scope rules).

---

### Likelihood Explanation

The attack requires only a funded account and a single transaction. No special privileges, leaked keys, or external assumptions are needed. The path is fully on-chain and reproducible.

---

### Recommendation

In `RegisterEncryptionKey`, extract the signer from the SDK context and assert it equals `req.Address`:

```go
func (k Keeper) RegisterEncryptionKey(
    ctx context.Context,
    req *types.MsgRegisterEncryptionKey,
) (*types.MsgRegisterEncryptionKeyResponse, error) {
    sdkCtx := sdk.UnwrapSDKContext(ctx)
    signers := sdkCtx.TxSigner() // or extract via msg server context
    // assert req.Address == signer bech32
    ...
}
```

Alternatively, add the `cosmos.msg.v1.signer` proto annotation to the `MsgRegisterEncryptionKey` proto definition pointing to the `address` field, and enforce the check in the keeper.

---

### Proof of Concept

```go
// accountA signs and broadcasts:
msg := &types.MsgRegisterEncryptionKey{
    Address: accountB.Address,  // victim's address
    Key:     attackerAgePublicKey,
}
// tx signed by accountA, fee paid by accountA

// query after inclusion:
resp, _ := queryClient.Key(ctx, &types.KeyRequest{Address: accountB.Address})
assert(resp.Key == attackerAgePublicKey)  // victim's key is now attacker's key
``` [3](#0-2) [1](#0-0)

func (k Keeper) registerEncryptionKey(

	address string,
	key []byte,
) error {
	bz, err := k.addressCodec.StringToBytes(address)
	if err != nil {
		return err
	}
	sdk.UnwrapSDKContext(ctx).KVStore(k.storeKey).Set(types.KeyPrefix(bz), key)
	return nil
}
