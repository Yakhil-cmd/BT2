### Title
Presignature Reuse Enabled by `Clone` on `PresignOutput` Allows Private Key Extraction - (File: `src/ecdsa/ot_based_ecdsa/mod.rs`, `src/ecdsa/robust_ecdsa/mod.rs`)

### Summary

Both `PresignOutput` and `RerandomizedPresignOutput` structs derive `Clone`, allowing a malicious coordinator to reuse the same presignature material across multiple signing sessions with different message hashes. Because the underlying nonce `k` is shared across sessions (only scaled by different `delta^{-1}` values), the resulting signatures have multiplicatively related nonces, enabling full private key extraction via standard ECDSA nonce-reuse algebra.

### Finding Description

The library explicitly documents that presignatures must be used exactly once:

> "Never reuse a presignature, even across failed, aborted, or partially completed signing sessions." [1](#0-0) 

Both READMEs reinforce this:

> "Each presignature is consumed **exactly once** (one-time use)." [2](#0-1) [3](#0-2) 

However, the library enforces this requirement only at the documentation level. The `PresignOutput` struct in both schemes derives `Clone`: [4](#0-3) 

And `RerandomizedPresignOutput` also derives `Clone`: [5](#0-4) 

The `sign()` function takes `presignature: RerandomizedPresignOutput` by value (consuming it), which appears to enforce one-time use. However, because `PresignOutput` is `Clone`, a malicious coordinator can clone it before calling `rerandomize_presign`, then call `rerandomize_presign` a second time with a different `(h, ε)` pair, producing two distinct `RerandomizedPresignOutput` values from the same underlying nonce material. [6](#0-5) 

The rerandomization step computes `delta = HKDF(Y, ε, h, R, ρ)` and transforms:
- `R → delta * R`
- `k_i → k_i * delta^{-1}`

When the same presignature is rerandomized twice with different `(h1, ε1)` and `(h2, ε2)`, the effective nonces in the two signing sessions are `k * delta1^{-1}` and `k * delta2^{-1}`. The ratio `delta2 / delta1` is fully computable from public inputs, so the nonces are multiplicatively related — a known sufficient condition for ECDSA private key recovery.

The security documentation itself confirms this:

> "If different subsets of size at least 2t+1 sign different (h, ε) values using shares derived from the same presignature, the resulting signatures use multiplicatively related nonces and the secret key can be recovered using standard ECDSA nonce-reuse attacks." [7](#0-6) 

The library enforces the `N1 = N2 = 2t+1` constraint and the `msg_hash != 0` constraint in code: [8](#0-7) 

But it does **not** enforce the equally critical presignature one-time-use constraint. The `Clone` derivation on `PresignOutput` is the missing enforcement gap.

### Impact Explanation

A malicious coordinator who holds a `PresignOutput` can:
1. Clone it: `let presign2 = presign_output.clone()`
2. Call `rerandomize_presign(&presign_output, &args_h1)` → session 1 with message hash `h1`
3. Call `rerandomize_presign(&presign2, &args_h2)` → session 2 with message hash `h2`
4. Run both signing sessions to completion, obtaining two valid ECDSA signatures `(R1, s1)` and `(R2, s2)` with multiplicatively related nonces
5. Recover the aggregate private key using standard nonce-reuse algebra

This constitutes **Critical** impact: full extraction of the aggregate secret key material from honest participants who each contributed their private share.

### Likelihood Explanation

The coordinator role is a standard participant in every signing session — it is not a privileged external role. Any participant designated as coordinator can execute this attack without any leaked keys or external assumptions. The only precondition is holding a `PresignOutput`, which is the normal output of the presigning protocol. The `Clone` trait is derived automatically and is reachable by any library caller.

### Recommendation

1. **Remove `Clone` from `PresignOutput` and `RerandomizedPresignOutput`** in both `src/ecdsa/ot_based_ecdsa/mod.rs` and `src/ecdsa/robust_ecdsa/mod.rs`. Rust's ownership system will then statically enforce one-time use: once the value is moved into `rerandomize_presign` or `sign`, it cannot be reused.
2. If `Clone` is needed for serialization/storage use cases, replace it with an explicit `consume()` or `take()` pattern that zeroes the original after producing the clone, and document that callers must never call it more than once.
3. Consider wrapping `PresignOutput` in a newtype that tracks usage state, returning an error on any second use.

// => nonce1 / nonce2 = delta2 / delta1  (known)

impl RerandomizedPresignOutput {
    pub fn rerandomize_presign(
        presignature: &PresignOutput,
        args: &RerandomizationArguments,
    ) -> Result<Self, ProtocolError> {
        if presignature.big_r != args.big_r {
            return Err(ProtocolError::IncompatibleRerandomizationInputs);
        }
        let delta = args.derive_randomness()?;
        if delta.is_zero().into() {
            return Err(ProtocolError::ZeroScalar);
        }

        // cannot be zero due to the previous check
        let inv_delta = delta.invert().unwrap();

        // delta . R
        let rerandomized_big_r = presignature.big_r * delta;

        //  (sigma + tweak * k) * delta^{-1}
        let rerandomized_sigma =
            (presignature.sigma + args.tweak.value() * presignature.k) * inv_delta;

        // k * delta^{-1}
        let rerandomized_k = presignature.k * inv_delta;

        Ok(Self {
            big_r: rerandomized_big_r.into(),
            k: rerandomized_k,
            sigma: rerandomized_sigma,
        })
    }
