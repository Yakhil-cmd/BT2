### Title
Missing Sequencer Access Control on `update_rewards` Allows Any Caller to Permanently Suppress Block Rewards — (`src/staking/staking.cairo`)

---

### Summary

The spec mandates "Only starkware sequencer" access control for `update_rewards`, but the implementation enforces no such check. Any unprivileged address can call `update_rewards(staker_address, disable_rewards: true)` first in any block, which atomically sets the **global** `last_reward_block` to the current block and skips reward distribution. All subsequent calls to `update_rewards` in that block — for any staker — revert with `REWARDS_ALREADY_UPDATED`. The rewards for that block are permanently lost.

---

### Finding Description

The spec at `docs/spec.md:1645` states:

> **access control**: Only starkware sequencer.

The implementation at `src/staking/staking.cairo:1449-1507` is:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();   // only checks: not paused, caller != zero
    let current_block_number = starknet::get_block_number();
    assert!(
        current_block_number > self.last_reward_block.read(),
        "{}", Error::REWARDS_ALREADY_UPDATED,
    );
    ...
    self.last_reward_block.write(current_block_number);  // global write

    if disable_rewards || self.is_pre_consensus() {
        return;   // no rewards distributed, but last_reward_block already updated
    }
    ...
}
```

`general_prerequisites()` at line 1794 only asserts the contract is unpaused and the caller is non-zero:

```cairo
fn general_prerequisites(ref self: ContractState) {
    self.assert_is_unpaused();
    assert_caller_is_not_zero();
}
```

There is no sequencer role check anywhere in the function or its prerequisites.

`last_reward_block` is declared as a **global** (contract-wide, not per-staker) `BlockNumber` in storage at line 187:

```cairo
/// Last block number for which rewards were distributed.
last_reward_block: BlockNumber,
```

The gate `current_block_number > self.last_reward_block.read()` is therefore a **single global slot** shared across all stakers. Whoever calls `update_rewards` first in a block consumes this slot for the entire block.

---

### Impact Explanation

An attacker calls `update_rewards(any_valid_active_staker, disable_rewards: true)` as the first transaction in any block. This:

1. Passes `general_prerequisites()` (not paused, caller non-zero).
2. Passes the `last_reward_block` gate (new block).
3. Writes `last_reward_block = current_block_number` — consuming the global slot.
4. Returns early at line 1487 without distributing any rewards.

Every subsequent call to `update_rewards` for any staker in that block hits `REWARDS_ALREADY_UPDATED` and reverts. The block's rewards are permanently gone — there is no catch-up mechanism; `last_reward_block` is never rolled back.

Repeating this every block permanently zeroes all consensus block rewards for all stakers and their delegators. The attacker gains nothing financially; the cost is only gas per block.

**Scoped impact**: Permanent freezing of unclaimed yield (High).

---

### Likelihood Explanation

- No privilege, stake, or capital required — any EOA with gas can execute this.
- The attack is repeatable every block at negligible cost.
- On Starknet, transaction ordering within a block is controlled by the sequencer, but the sequencer is the very entity that *should* be the only caller. An attacker front-running the sequencer's own call is the exact threat the spec's access control was meant to prevent.

---

### Recommendation

Add a sequencer role assertion at the top of `update_rewards`, consistent with the spec:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
    self.only_sequencer();   // enforce "Only starkware sequencer"
    ...
}
```

Where `only_sequencer()` checks `get_caller_address() == sequencer_address` (stored at construction or via a role in the `RolesComponent` already embedded in the contract).

---

### Proof of Concept

1. Deploy the staking contract with two active stakers, `S1` and `S2`, both past the K-epoch activation window and with consensus rewards enabled.
2. At the start of block `N`, attacker calls:
   ```
   update_rewards(staker_address: S1, disable_rewards: true)
   ```
3. `last_reward_block` is now `N`. No rewards are distributed to `S1`.
4. The legitimate sequencer (or `S2`) attempts:
   ```
   update_rewards(staker_address: S2, disable_rewards: false)
   ```
   → reverts with `REWARDS_ALREADY_UPDATED`.
5. Repeat for every block. Both `S1` and `S2` accumulate zero consensus block rewards indefinitely.
6. Compare `unclaimed_rewards_own` against a baseline run without the attacker — the difference equals all suppressed yield.
