### Title
Zero `priceProviderTimelock` Allows Pool Admin to Atomically Swap Oracle and Drain LP Funds - (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary
`_validatePoolParameters` does not enforce any minimum value for `priceProviderTimelock`. A pool admin who sets this to `0` at creation can, in the same block, propose and immediately execute a replacement of the pool's price provider with a malicious one that returns extreme bid/ask prices, enabling complete drainage of LP funds.

### Finding Description
`createPool` stores `params.priceProviderTimelock` directly into `priceProviderTimelock[pool]` without any lower-bound check. [1](#0-0) 

The only special-cased value is `type(uint256).max`, which activates immutable mode. Every other value — including `0` — is accepted silently. [2](#0-1) 

When `priceProviderTimelock[pool] == 0`, `proposePoolPriceProvider` computes:

```
executeAfter = block.timestamp + 0  →  executeAfter == block.timestamp
``` [3](#0-2) 

`executePoolPriceProviderUpdate` then checks `block.timestamp < execAfter`, which is `false` immediately (same block, same timestamp), so the update proceeds without any delay. [4](#0-3) 

The only validation applied to the incoming provider is that it is non-zero and returns the correct `token0()`/`token1()` pair — it does not constrain the bid/ask values the provider may return. [5](#0-4) 

A malicious provider trivially satisfies these checks while returning `bidPrice = type(uint128).max - 1` and `askPrice = type(uint128).max`, or `bidPrice = 1` and `askPrice = 2`, making every swap settle at an extreme rate that transfers all pool reserves to the swapper.

The factory enforces hard caps on admin fees (`maxAdminSpreadFeeE6`, `maxAdminNotionalFeeE8`) but applies no analogous floor to the timelock, creating an inconsistency in the security model: the pool admin is "semi-trusted only inside caps and timelocks," yet the timelock cap is unenforced. [6](#0-5) 

### Impact Explanation
A pool admin who creates a pool with `priceProviderTimelock = 0` can, after attracting LP deposits, atomically (within a single block or transaction):

1. Deploy a malicious `IPriceProvider` that returns the correct `token0`/`token1` but extreme bid/ask prices.
2. Call `proposePoolPriceProvider` → `executePoolPriceProviderUpdate` with zero elapsed time.
3. Execute a swap through the pool at the manipulated price, receiving all token0 or token1 reserves.

All LP principal is lost. The pool's `setPriceProvider` path writes the new provider directly into mutable storage with no further guard. [7](#0-6) 

### Likelihood Explanation
The trigger requires the pool admin to be malicious and to have set `priceProviderTimelock = 0` at pool creation. Because `createPool` is permissionless (any caller may deploy a pool), a malicious actor can create a pool with this configuration, market it to LPs, and execute the attack. The factory provides no on-chain signal distinguishing a zero-timelock pool from a properly configured one, so LPs relying solely on on-chain state cannot detect the risk without reading the raw `priceProviderTimelock` mapping.

### Recommendation
Add a minimum timelock floor in `_validatePoolParameters` (or as a factory-level constant), for example:

```solidity
uint256 internal constant MIN_PRICE_PROVIDER_TIMELOCK = 48 hours;

// inside _validatePoolParameters:
if (
    params.priceProviderTimelock != type(uint256).max &&
    params.priceProviderTimelock < MIN_PRICE_PROVIDER_TIMELOCK
) revert PriceProviderTimelockTooShort();
```

This mirrors the existing pattern of hard caps on fees and ensures the timelock always provides a meaningful observation window for LPs.

// Malicious price provider: correct tokens, extreme prices
contract MaliciousPriceProvider {
    address public token0;
    address public token1;
    constructor(address t0, address t1) { token0 = t0; token1 = t1; }
    function getBidAndAskPrice() external pure returns (uint128, uint128) {
        // bid ≈ 0: attacker sells token1 and receives almost all token0 for free
        return (1, 2);
    }
}

// Attack sequence (Foundry test sketch):
function test_zeroTimelockDrain() public {
    // 1. Pool admin creates pool with priceProviderTimelock = 0
    PoolParameters memory params = _defaultPoolParams();
    params.priceProviderTimelock = 0;          // <-- zero timelock accepted
    address pool = factory.createPool(params);

    // 2. LPs deposit liquidity
    _addLiquidity(pool, 1_000_000e18, 1_000_000e18);

    // 3. Admin deploys malicious provider (correct tokens, extreme prices)
    MaliciousPriceProvider evil = new MaliciousPriceProvider(token0, token1);

    // 4. Propose + execute in the same block (no warp needed)
    vm.prank(admin);
    factory.proposePoolPriceProvider(pool, address(evil));
    // executeAfter == block.timestamp → passes immediately
    vm.prank(admin);
    factory.executePoolPriceProviderUpdate(pool);

    // 5. Swap at extreme price — attacker receives all token0 reserves
    pool.swap(attacker, /*zeroForOne=*/false, int128(1), type(uint128).max, "");
    // attacker now holds nearly all token0 from the pool
}
