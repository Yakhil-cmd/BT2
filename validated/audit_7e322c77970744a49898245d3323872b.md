## Title
Unchecked Integer Overflow in SNS Governance `distribute_rewards` Silently Destroys Neuron Maturity - (File: rs/sns/governance/src/governance.rs)

## Summary
The SNS governance canister's reward distribution loop uses bare `+` and `+=` operators on `u64` maturity fields when crediting voting rewards to neurons. In Rust compiled to WebAssembly in release mode, integer overflow wraps silently rather than panicking. A neuron whose accumulated maturity approaches `u64::MAX` will have its maturity silently zeroed on the next reward distribution, permanently destroying earned governance rewards. The NNS equivalent path uses `saturating_add` throughout, confirming this is an oversight in the SNS path.

## Finding Description
In `rs/sns/governance/src/governance.rs` at lines 5989–5996, the reward distribution loop credits each voting neuron using unchecked arithmetic:

```rust
if neuron.auto_stake_maturity.unwrap_or(false) {
    neuron.staked_maturity_e8s_equivalent = Some(
        neuron.staked_maturity_e8s_equivalent.unwrap_or(0) + neuron_reward_e8s,
    );
} else {
    neuron.maturity_e8s_equivalent += neuron_reward_e8s;
}
distributed_e8s_equivalent += neuron_reward_e8s;
```

All three additions are unchecked `u64` arithmetic. In Rust release builds targeting WebAssembly (the production canister compilation target), integer overflow wraps silently — there is no trap, no panic, and no error. The `debug_assert!` at line 6004 that checks `distributed_e8s_equivalent <= rewards_purse_e8s` is a no-op in production release builds.

The NNS equivalent in `rs/nns/governance/src/reward/distribution.rs` at lines 162–171 uses `saturating_add` for both `staked_maturity_e8s_equivalent` and `maturity_e8s_equivalent`. The SNS path is the only reward-crediting path in the codebase that omits this protection.

The rewards purse itself has an overflow guard (lines 5878–5890) that aborts distribution if the purse overflows `u64`, but this guard does not protect the per-neuron maturity field accumulation.

Exploit flow:
1. A neuron accumulates `maturity_e8s_equivalent` (or `staked_maturity_e8s_equivalent`) near `u64::MAX` over many reward periods without disbursing.
2. The governance canister's periodic `distribute_rewards` heartbeat fires.
3. `neuron_reward_e8s` is computed as a valid `u64` fraction of the rewards purse.
4. The bare `+` addition wraps: `(u64::MAX - k) + (k + 1)` → `0`.
5. The neuron's maturity is silently set to a near-zero value; `distributed_e8s_equivalent` also wraps, masking the accounting error in the emitted `RewardEvent`.

## Impact Explanation
A neuron's `maturity_e8s_equivalent` or `staked_maturity_e8s_equivalent` silently wraps to near-zero. Because maturity is the precursor to minting new SNS tokens via `disburse_maturity` or `merge_maturity`, this constitutes permanent, irrecoverable loss of earned governance rewards for the neuron holder. No error is logged, no revert occurs, and the `RewardEvent` records a wrapped `distributed_e8s_equivalent` that masks the accounting discrepancy. This maps to: **High — significant SNS governance security impact with concrete user funds harm** (loss of neuron maturity/earned rewards without any error or revert).

## Likelihood Explanation
Low. Triggering the overflow requires a neuron's accumulated maturity to approach `u64::MAX` (≈ 1.84 × 10¹⁹ e8s). In practice, SNS token supplies are far below this bound, and maturity is bounded by total rewards ever distributed. However, no attacker action is required — the overflow is triggered automatically by the governance canister's periodic reward heartbeat, reachable by any neuron holder who votes. The structural inconsistency with the NNS path (which uses `saturating_add` throughout) confirms this is an unintentional omission rather than a deliberate design choice.

## Recommendation
Replace the three unchecked additions with `saturating_add`, consistent with the NNS reward distribution path in `rs/nns/governance/src/reward/distribution.rs`:

```rust
if neuron.auto_stake_maturity.unwrap_or(false) {
    neuron.staked_maturity_e8s_equivalent = Some(
        neuron.staked_maturity_e8s_equivalent
            .unwrap_or(0)
            .saturating_add(neuron_reward_e8s),
    );
} else {
    neuron.maturity_e8s_equivalent =
        neuron.maturity_e8s_equivalent.saturating_add(neuron_reward_e8s);
}
distributed_e8s_equivalent =
    distributed_e8s_equivalent.saturating_add(neuron_reward_e8s);
```

Alternatively, use `checked_add` and log an error without panicking if overflow would occur, mirroring the pattern already used for the rewards purse overflow check at lines 5878–5890.
