### Title
Pool Member Reward Address Validation Omits STRK Token, Permanently Freezing Unclaimed Yield — (File: `src/pool/pool.cairo`)

---

### Summary

The `Pool` contract's `enter_delegation_pool` and `change_reward_address` functions validate that the supplied `reward_address` is not equal to the pool's own staking token (e.g., BTC). However, they do not validate against the STRK token address — the token in which all pool rewards are actually paid. A pool member in a BTC delegation pool can therefore set their `reward_address` to `STRK_TOKEN_ADDRESS`. When `claim_rewards` is subsequently called, the STRK reward transfer is directed to the STRK token contract itself, permanently freezing those funds with no recovery path.

---

### Finding Description

The `Pool` contract stores a single `token_dispatcher` for the pool's staking token (BTC in multi-token pools). Both entry-point functions that accept a `reward_address` guard only against that one token:

**`enter_delegation_pool`** (`src/pool/pool.cairo`, lines 192–195):
```cairo
let token_address = token_dispatcher.contract_address;
assert!(token_address != pool_member, "{}", Error::POOL_MEMBER_IS_TOKEN);
assert!(token_address != reward_address, "{}", GenericError::REWARD_ADDRESS_IS_TOKEN);
```

**`change_reward_address`** (`src/pool/pool.cairo`, lines 506–510):
```cairo
assert!(
    self.token_dispatcher.contract_address.read() != reward_address,
    "{}",
    GenericError::REWARD_ADDRESS_IS_TOKEN,
);
```

Neither check tests `reward_address` against `STRK_TOKEN_ADDRESS`. Yet `claim_rewards` always pays out in STRK regardless of which token the pool uses:

**`claim_rewards`** (`src/pool/pool.cairo`, lines 365–366):
```cairo
let reward_token = IERC20Dispatcher { contract_address: STRK_TOKEN_ADDRESS };
reward_token.checked_transfer(recipient: reward_address, amount: rewards.into());
```

By contrast, the `Staking` contract's `stake` and `change_reward_address` functions call `does_token_exist`, which checks against **all** registered tokens (STRK and BTC):

**`stake`** (`src/staking/staking.cairo`, lines 307–311):
```cairo
assert!(
    !self.does_token_exist(token_address: reward_address),
    "{}",
    GenericError::REWARD_ADDRESS_IS_TOKEN,
);
```

The pool contract's validation is therefore incomplete relative to the staking contract's own standard, and the gap is exploitable.

---

### Impact Explanation

A pool member in a BTC delegation pool who sets `reward_address = STRK_TOKEN_ADDRESS` (either by mistake or deliberately) will have all future STRK reward claims transferred into the STRK token contract's own balance. The STRK token contract has no mechanism to redistribute or recover tokens credited to itself; those tokens are permanently frozen. This constitutes **permanent freezing of unclaimed yield**, matching the High-severity impact category.

---

### Likelihood Explanation

The entry path is fully unprivileged: any pool member can call `enter_delegation_pool` or `change_reward_address` with `reward_address = STRK_TOKEN_ADDRESS`. No special role, leaked key, or external dependency is required. The scenario is most likely to occur through user error (e.g., a developer testing with a token address, or a UI bug), but the protocol provides no guard against it. Because the staking contract already enforces the correct check, users may reasonably assume the pool contract does too.

---

### Recommendation

Add a check against `STRK_TOKEN_ADDRESS` in both `enter_delegation_pool` and `change_reward_address` in `src/pool/pool.cairo`, mirroring the staking contract's `does_token_exist` pattern:

```cairo
assert!(
    reward_address != STRK_TOKEN_ADDRESS,
    "{}",
    GenericError::REWARD_ADDRESS_IS_TOKEN,
);
assert!(
    token_address != reward_address,
    "{}",
    GenericError::REWARD_ADDRESS_IS_TOKEN,
);
```

Alternatively, expose a helper that checks against all registered tokens and call it from both the staking and pool contracts.

---

### Proof of Concept

1. A staker opens a BTC delegation pool via `set_open_for_delegation(btc_token_address)`.
2. A pool member calls `enter_delegation_pool(reward_address: STRK_TOKEN_ADDRESS, amount: X)` on the BTC pool.
   - The check at line 195 passes because `btc_token_address != STRK_TOKEN_ADDRESS`.
3. Over several epochs the pool member accrues STRK rewards tracked in `cumulative_rewards_trace`.
4. The pool member (or anyone) calls `claim_rewards(pool_member)`.
5. At line 366, `reward_token.checked_transfer(recipient: STRK_TOKEN_ADDRESS, amount: rewards)` executes successfully — STRK tokens are credited to the STRK token contract's own balance.
6. No function in the STRK token contract allows recovery of these tokens; they are permanently frozen.
