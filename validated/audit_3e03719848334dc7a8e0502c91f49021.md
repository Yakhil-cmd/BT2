### Title
Div-Before-Mul Precision Loss in `native_per_pubdata` Calculation Causes Systematic Pubdata Underpayment — (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

The `native_per_pubdata` rate used to charge users for pubdata consumption is computed by dividing `pubdata_price` by `native_price` before multiplying by the number of pubdata bytes. This div-before-mul ordering causes systematic truncation, meaning every transaction that generates pubdata pays slightly less than the mathematically correct amount.

---

### Finding Description

In `validation_impl.rs`, the per-pubdata-byte native cost is computed as:

```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
``` [1](#0-0) 

This `native_per_pubdata` value is then used in `get_resources_to_charge_for_pubdata` to compute the total native charge:

```rust
let native = current_pubdata_spent
    .checked_mul(native_per_pubdata)
    .ok_or(out_of_native_resources!())?;
``` [2](#0-1) 

The effective formula is:

```
native_charged = pubdata_bytes × floor(pubdata_price / native_price)
```

The mathematically correct formula (avoiding intermediate truncation) is:

```
native_charged = floor(pubdata_bytes × pubdata_price / native_price)
```

Because `floor(pubdata_bytes × pubdata_price / native_price) ≥ pubdata_bytes × floor(pubdata_price / native_price)`, the current code always charges **less than or equal to** the correct amount. The maximum underpayment per transaction is:

```
error ≤ pubdata_bytes × (native_price − 1) / native_price  <  pubdata_bytes  native units
```

The same pattern appears in the API helper used for off-chain validation:

```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
``` [3](#0-2) 

A secondary instance exists in `compute_gas_refund`, where `delta_gas` is computed as `native_used / native_per_gas` — another div-before-div that underestimates the gas adjustment, causing users to receive slightly larger refunds than they should:

```rust
(native_used / native_per_gas) as i64 - (gas_used as i64)
``` [4](#0-3) 

The documentation confirms the intended formula is `deltaGas := (nativeUsed / nativePerGas) − gasUsed`, but the intermediate `nativePerGas` is itself a truncated integer, compounding the loss. [5](#0-4) 

---

### Impact Explanation

Every L2 transaction that writes pubdata pays slightly less native resource than the correct amount. The operator (sequencer/prover) is systematically underpaid for pubdata costs. For a transaction using `P` pubdata bytes with `pubdata_price = Q` and `native_price = N`, the underpayment is up to `P × (N−1) / N` native units. At high pubdata volumes (e.g., large storage writes), this is a consistent, non-negligible resource accounting shortfall that accumulates across all transactions in a block.

---

### Likelihood Explanation

This affects every L2 transaction that generates pubdata whenever `pubdata_price` is not an exact multiple of `native_price`. Since `pubdata_price` and `native_price` are independent operator-set parameters, non-divisibility is the common case. Any unprivileged user submitting a transaction with storage writes triggers this path.

---

### Recommendation

Defer the division to after the multiplication. Instead of:

```rust
let native_per_pubdata = pubdata_price / native_price;
// later:
let native = pubdata_bytes * native_per_pubdata;
```

Compute directly:

```rust
let native = pubdata_bytes * pubdata_price / native_price;
```

Care must be taken to use a wider integer type (e.g., `U256`) for the intermediate product before truncating to `u64` to avoid overflow. The same fix applies to the `delta_gas` calculation in `compute_gas_refund`: replace `native_used / native_per_gas` with `native_used * native_price / gas_price` (using `U256` arithmetic).

---

### Proof of Concept

Concrete example:
- `pubdata_price = 100`, `native_price = 7`, `pubdata_bytes = 1000`
- Current code: `native_per_pubdata = 100 / 7 = 14`; charged = `1000 × 14 = 14000`
- Correct: `floor(1000 × 100 / 7) = floor(100000 / 7) = 14285`
- Underpayment: **285 native units** per 1000 bytes of pubdata

For a block with many pubdata-heavy transactions, this underpayment accumulates. A user can maximize the effect by submitting transactions that write many storage slots (maximizing `pubdata_bytes`) under block parameters where `pubdata_price % native_price` is large (close to `native_price − 1`).
