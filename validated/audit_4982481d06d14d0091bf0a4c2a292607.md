### Title
Unbounded Loop in `calculate_rewards` Can Permanently Freeze Pool Member's Unclaimed Yield - (File: src/pool/pool.cairo)

### Summary
The `calculate_rewards` internal function in `pool.cairo` contains an unbounded loop that iterates over every balance-change entry in a pool member's `pool_member_epoch_balance` trace since their last reward claim. A pool member who repeatedly changes their delegation balance across many epochs without ever calling `claim_rewards` can grow this trace without bound. When `claim_rewards` is eventually invoked, the loop may exhaust Starknet's execution-step budget, causing the transaction to revert and permanently freezing the pool member's accumulated unclaimed yield.

### Finding Description

`calculate_rewards` is called inside `claim_rewards` (and the view `pool_member_info_v1`) in `src/pool/pool.cairo`. It iterates from the stored `entry_to_claim_from` index up to the current length of the per-member balance trace:

```cairo
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
    if pool_member_checkpoint.epoch() >= until_epoch {
        break;
    }
    let to_sigma = self.find_sigma(pool_member_checkpoint, curr_epoch: until_epoch);
    ...
    entry_to_claim_from += 1;
}
``` [1](#0-0) 

The developers themselves annotate this loop as **"unbounded"**. Each iteration performs at least two storage reads: one for `pool_member_trace.at(entry_to_claim_from)` and one inside `find_sigma` against `cumulative_rewards_trace_vec`. Storage reads are among the most expensive operations in Starknet's step-counting model.

The trace grows by one checkpoint per epoch in which the pool member changes their balance. The `entry_to_claim_from` cursor is only advanced when `claim_rewards` succeeds and writes back the updated value:

```cairo
pool_member_info.entry_to_claim_from = updated_entry_to_claim_from;
pool_member_info.reward_checkpoint = until_checkpoint;
``` [2](#0-1) 

If a pool member never calls `claim_rewards`, `entry_to_claim_from` stays at zero and the loop must traverse every balance-change entry ever recorded for that member. A delegator who actively adjusts their stake across hundreds of epochs without claiming will accumulate a trace of that length. When `claim_rewards` is finally attempted — by the pool member or by their separately-configured `reward_address` — the transaction will revert once the step budget is exhausted, and the cursor is never advanced. Every subsequent attempt faces the same (or larger) trace, making the freeze permanent.

The `InternalPoolMemberInfoV1` struct stores `entry_to_claim_from` and `reward_checkpoint` per member: [3](#0-2) 

The pool member balance trace itself is an append-only `Vec`-backed structure; entries are never pruned: [4](#0-3) 

### Impact Explanation
A pool member whose `claim_rewards` call permanently reverts loses all accumulated yield that has not yet been claimed. This matches the allowed impact **"High: Permanent freezing of unclaimed yield"**. The frozen STRK rewards remain in the pool contract and cannot be transferred to the member or their reward address.

### Likelihood Explanation
Any unprivileged pool member can reach this state by:
1. Delegating to a pool.
2. Repeatedly calling `add_to_delegation_pool` / `remove_from_delegation_pool_intent` across many epochs (each epoch adds at most one new checkpoint).
3. Deferring `claim_rewards` indefinitely.

No privileged access, bridge compromise, or external dependency is required. The entry point is the public `claim_rewards` function: [5](#0-4) 

Access control allows either the pool member or their reward address to call it, so a yield-aggregator contract set as `reward_address` is equally affected if the underlying pool member has grown a large trace.

### Recommendation
1. **Paginated claiming**: Accept an optional `max_entries` parameter in `claim_rewards` so callers can process the trace in bounded chunks, advancing `entry_to_claim_from` after each partial call.
2. **Epoch-based cursor**: Store a per-member "last claimed epoch" and binary-search the trace to the relevant window rather than iterating linearly from index zero.
3. **Enforce periodic claiming**: Require or incentivise pool members to claim at least once per N epochs, bounding the maximum trace length that must be traversed in a single call.

### Proof of Concept
1. Deploy the staking system and open a delegation pool.
2. Have `pool_member` call `add_to_delegation_pool` once per epoch for a large number of epochs (e.g., 500) without ever calling `claim_rewards`. Each epoch adds one entry to `pool_member_epoch_balance`.
3. After 500 epochs, call `claim_rewards(pool_member)`. The loop in `calculate_rewards` must now iterate over all 500 entries, each requiring two storage reads. Measure the step count; it will scale linearly with the number of entries.
4. At a sufficiently large entry count the transaction reverts with an out-of-resources error. `entry_to_claim_from` is never updated, so every future call faces the same (or larger) trace — the yield is permanently frozen.

The developers' own comment confirms the unbounded nature of the loop: [6](#0-5)

            // Asserts.
            let mut pool_member_info = self.internal_pool_member_info(:pool_member);

            // Calculate rewards and update entry_to_claim_from.

    fn insert(
        self: StoragePath<Mutable<PoolMemberBalanceTrace>>, key: Epoch, value: PoolMemberBalance,
    ) -> (PoolMemberBalance, PoolMemberBalance) {
        let checkpoints = self.checkpoints;

        let len = checkpoints.len();
        if len == Zero::zero() {
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
            return (Zero::zero(), value);
        }

        // Update or append new checkpoint.
        let mut last = checkpoints[len - 1].read();
        let prev = last.value;
        if last.key == key {
            last.value = value;
            checkpoints[len - 1].write(last);
        } else {
            // Checkpoint keys must be non-decreasing.
            assert!(last.key < key, "{}", TraceErrors::UNORDERED_INSERTION);
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
        }
        (prev, value)
    }
