### Title
Single Malicious Participant Can Permanently Abort DKG/Reshare/Refresh by Broadcasting False Success - (File: src/dkg.rs)

### Summary
The `broadcast_success` function in `src/dkg.rs` (Round 5.4/5.5 of the PedPop+ DKG) requires **every** participant to broadcast `(true, session_id)`. A single malicious participant within the documented n/3 BFT tolerance can broadcast `(false, session_id)` with no legitimate reason, causing all honest parties to abort with `ProtocolError::AssertionFailed`. Because the attack can be repeated on every DKG attempt, it constitutes a permanent denial of key generation, reshare, and refresh for all honest parties.

### Finding Description
In `do_keyshare` — the shared core of `do_keygen`, `do_reshare`, and key refresh — the final step is:

```rust
// Step 5.4 and Step 5.5
broadcast_success(&mut chan, &participants, me, session_id).await?;
```

`broadcast_success` uses the reliable broadcast protocol to collect `(bool, session_id)` from every participant, then enforces a unanimous-success gate:

```rust
if !vote_list.iter().all(|&(boolean, _)| boolean) {
    return Err(ProtocolError::AssertionFailed(
        "A participant\n                seems to have failed its checks. Aborting Protocol!"
            .to_string(),
    ));
}
```

This check requires **all** participants to broadcast `true`. A malicious participant can broadcast `(false, session_id)` — a valid, well-formed message — without any legitimate reason. The reliable broadcast protocol guarantees all honest parties receive this `false` value, causing all of them to abort.

**Attack flow:**
1. All honest parties complete all five rounds of DKG successfully (session IDs, commitments, proofs, secret shares all verified).
2. In Round 5.4, the malicious participant broadcasts `(false, session_id)` instead of `(true, session_id)`.
3. The reliable broadcast delivers `false` to all honest parties.
4. All honest parties fail the `all(|&(boolean, _)| boolean)` check and return `Err(ProtocolError::AssertionFailed(...))`.
5. The malicious participant repeats this on every subsequent DKG attempt.

The analog to the external report is exact: just as a malicious depositor changes the vault's `status` from `Open` to `Deposit` to block every other user's call, a malicious DKG participant changes the collective "success status" from `true` to `false` to block every honest party's key generation.

### Impact Explanation
Permanent denial of key generation, reshare, and refresh for all honest parties. Because `do_keyshare` is the shared implementation called by `do_keygen`, `do_reshare`, and (by extension) key refresh, all three operations are affected. The malicious participant can repeat this attack indefinitely, making it impossible for honest parties to ever complete key generation or update their key shares. Without a valid key share, no threshold signing, presigning, or CKD operation can proceed.

### Likelihood Explanation
The attack requires only a single malicious participant within the n/3 BFT tolerance — the very adversary model the protocol claims to handle. The attack is trivial: broadcast `false` instead of `true` in the final round. No cryptographic knowledge, leaked secrets, or special capabilities are required. The attack is repeatable at zero marginal cost on every DKG retry.

### Recommendation
Replace the unanimous-success gate with a threshold-success gate consistent with the BFT tolerance. Specifically, require at least `n - floor((n-1)/3)` participants to broadcast `true` before proceeding, rather than requiring all `n`. This matches the liveness guarantee already provided by the reliable broadcast channel and prevents a single malicious participant from aborting the protocol.

### Proof of Concept
With n=4 participants (P1, P2, P3 honest; P4 malicious):

1. All four participants complete Rounds 1–5 of DKG successfully.
2. In Round 5.4, P4 calls `do_broadcast` with `(false, session_id)` instead of `(true, session_id)`.
3. The reliable broadcast delivers `(false, session_id)` to P1, P2, P3.
4. Each honest party executes:
   ```rust
   if !vote_list.iter().all(|&(boolean, _)| boolean) { ... }
   ```
   P4's entry is `(false, session_id)` → the `all(...)` predicate is `false` → `Err(AssertionFailed(...))` is returned.
5. P1, P2, P3 all abort. No `KeygenOutput` is produced.

async fn broadcast_success(
    chan: &mut SharedChannel,
    participants: &ParticipantList,
    me: Participant,
    session_id: HashOutput,
) -> Result<(), ProtocolError> {
    // broadcast node me succeded
    let vote_list = do_broadcast(chan, participants, me, (true, session_id)).await?;
    // unwrap here would never fail as the broadcast protocol ends only when the map is full
    let vote_list = vote_list
        .into_vec_or_none()
        .ok_or_else(|| ProtocolError::AssertionFailed("vote_list is empty".to_string()))?;
    // go through all the list of votes and check if any is fail or some does not contain the session id

    if !vote_list.iter().all(|(_, ref sid)| sid == &session_id) {
        return Err(ProtocolError::AssertionFailed(
            "A participant
                broadcast the wrong session id. Aborting Protocol!"
                .to_string(),
        ));
    }

    if !vote_list.iter().all(|&(boolean, _)| boolean) {
        return Err(ProtocolError::AssertionFailed(
            "A participant
                seems to have failed its checks. Aborting Protocol!"
                .to_string(),
        ));
    }
    // Wait for all the tasks to complete
    Ok(())
}

/// reshares the keyshares between the parties and allows changing the threshold
#[allow(clippy::too_many_arguments)]
pub async fn do_reshare<C: Ciphersuite>(
    chan: SharedChannel,
    participants: ParticipantList,
    me: Participant,
    threshold: impl Into<ReconstructionLowerBound>,
    old_signing_key: Option<SigningShare<C>>,
    old_public_key: VerifyingKey<C>,
    old_participants: ParticipantList,
    mut rng: impl CryptoRngCore,
) -> Result<KeygenOutput<C>, ProtocolError> {
    let threshold = threshold.into();
    let intersection = old_participants.intersection(&participants);
    // either extract the share and linearize it or set it to zero
    let secret = old_signing_key
        .map(|x_i| {
            intersection
                .lagrange::<C>(me)
                .map(|lambda| lambda * x_i.to_scalar())
        })
        .transpose()?
        .unwrap_or_else(<C::Group as Group>::Field::zero);

    let old_reshare_package = Some((old_public_key, old_participants));
    let keygen_output = do_keyshare::<C>(
        chan,
        participants,
        me,
        threshold,
        secret,
        old_reshare_package,
        &mut rng,
    )
    .await?;

    Ok(keygen_output)
