### Title
Anyone Can Permanently Freeze Staker Yield by Calling `update_rewards` with `disable_rewards=true` — (File: src/staking/staking.cairo)

---

### Summary

The `update_rewards` function in the Staking contract is publicly callable with no role-based access control. It accepts a `disable_rewards: bool` parameter. When called with `disable_rewards=true`, it writes the current block number into the global `last_reward_block` storage slot **without distributing any rewards**. Because the function enforces `current_block_number > last_reward_block`, any subsequent legitimate call for the same block reverts with `REWARDS_ALREADY_UPDATED`. The yield for that block is permanently lost. An unprivileged attacker can automate this call every block to permanently freeze all stakers' consensus-era unclaimed yield.

---

### Finding Description

`update_rewards` is exposed through `IStakingRewardsManager` and is gated only by `general_prerequisites()`, which checks that the contract is not paused and the caller is non-zero — no role check whatsoever. [1](#0-0) 

The function writes `last_reward_block` to the current block **before** the `disable_rewards` branch: [2](#0-1) 

After that write, the early-return path is taken when `disable_rewards == true`: [3](#0-2) 

`last_reward_block` is a single global slot shared across all stakers: [4](#0-3) 

The uniqueness guard at the top of the function means only one call per block can succeed: [5](#0-4) 

Consequence: if an attacker calls `update_rewards(any_valid_active_staker, true)` first in a block, `last_reward_block` is stamped with that block number and no rewards are ever distributed for it. The consensus block rewards (`block_rewards`) are fixed per epoch and are not accumulated or carried forward; they are simply never claimed from the reward supplier for that block. [6](#0-5) 

---

### Impact Explanation

**High — Permanent freezing of unclaimed yield.**

Consensus-era block rewards are calculated once per epoch and divided across blocks. A block that passes without `_update_rewards` being executed loses its share of yield permanently — it is never re-queued or accumulated. An attacker who calls `update_rewards(valid_staker, true)` every block causes every staker and every delegator in every pool to receive zero consensus rewards indefinitely. The damage scales linearly with the duration of the attack and the total staked value.

---

### Likelihood Explanation

**Medium.**

The attacker requires:
1. Knowledge of any one active staker address with non-zero balance (trivially obtained from on-chain events).
2. One Starknet transaction per block.

Starknet transaction fees are low and the call is simple. The attack is fully automatable with a bot. The attacker has no profit motive but causes severe, compounding damage to all protocol participants.

---

### Recommendation

Add a role check to `update_rewards` so that only the authorized consensus reward distributor (or a dedicated `REWARD_MANAGER` role) may call it. The `disable_rewards` parameter should not be exposed to arbitrary callers; if it is needed internally, it should be a private/internal function or restricted to a trusted caller.

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
+   self.roles.only_reward_manager(); // or equivalent role check
    ...
}
```

### Proof of Concept

```
// Attacker bot, runs every block:
loop {
    staking_contract.update_rewards(
        staker_address = <any known active staker>,
        disable_rewards = true,
    );
    // last_reward_block is now == current_block_number
    // No rewards distributed; slot consumed.
    // Any legitimate call this block reverts with REWARDS_ALREADY_UPDATED.
    wait_for_next_block();
}
```

Step-by-step:
1. Block N begins. Attacker calls `update_rewards(valid_staker, true)`.
2. `last_reward_block` is written to `N`; function returns early — no rewards distributed.
3. Consensus mechanism (or any honest caller) attempts `update_rewards(valid_staker, false)` in the same block → reverts: `current_block_number (N) > last_reward_block (N)` is false.
4. Block N's yield is permanently lost for all stakers and delegators.
5. Repeat for every block.
