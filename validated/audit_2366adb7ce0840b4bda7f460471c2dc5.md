### Title
Unbound `app_pk` Parameter Allows Malicious Participant to Corrupt CKD Output — (`File: src/confidential_key_derivation/protocol.rs`)

---

### Summary

In the Confidential Key Derivation (CKD) protocol, the `app_pk` (application public key) parameter is never committed to, broadcast, or cross-verified between participants. Each participant independently uses their own locally-supplied `app_pk` to compute their share contribution. A malicious participant can silently substitute an arbitrary `app_pk`, causing the coordinator to aggregate an inconsistent set of contributions and produce a permanently unusable CKD output that honest parties accept without any error.

---

### Finding Description

The CKD protocol is implemented across two functions in `src/confidential_key_derivation/protocol.rs`.

`compute_signature_share` (lines 148–182) is called by every participant (including the coordinator) to produce their individual contribution `(norm_big_y, norm_big_c)`:

```
big_s = private_share * H(pk || app_id)
big_c = big_s + app_pk * y          // app_pk used here, unchecked
big_y = y * G
``` [1](#0-0) 

`do_ckd_participant` (lines 17–33) then sends only the raw `(norm_big_y, norm_big_c)` tuple to the coordinator — **`app_pk` is never included in the message**:

```rust
chan.send_private(waitpoint, coordinator, &(norm_big_y, norm_big_c))?;
``` [2](#0-1) 

`do_ckd_coordinator` (lines 35–58) blindly sums all received contributions with no verification that participants used the same `app_pk`:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
``` [3](#0-2) 

There is no broadcast round, no commitment, and no consistency check for `app_pk` anywhere in the protocol. The `ckd()` entry point also performs no validation of `app_pk`: [4](#0-3) 

The correct CKD output requires every participant to use the **same** `app_pk`, because the final unmasking step is:

```
unmask(app_sk) = big_c − app_sk · big_y  =  msk · H(pk ∥ app_id)
``` [5](#0-4) 

If participant `m` uses `app_pk_m ≠ app_pk_honest`, the aggregated `big_c` becomes:

```
big_c = Σ_{i≠m} (big_s_i + app_pk_honest · y_i)  +  (big_s_m + app_pk_m · y_m)
```

Then `unmask(app_sk_honest) = msk·H(pk‖app_id) + (app_pk_m − app_pk_honest)·λ_m·y_m`, which is a random, non-zero error term. The output is permanently corrupted and the confidential key is unrecoverable.

---

### Impact Explanation

**High — Corruption of CKD output so honest parties accept an inconsistent, unusable cryptographic output.**

The honest coordinator receives and accepts a `CKDOutput` whose `big_c` is polluted by the malicious participant's mismatched `app_pk`. The coordinator has no way to detect the manipulation because `app_pk` is never transmitted or committed to in any protocol message. The derived confidential key is irrecoverably wrong, and the protocol gives no error signal. This matches the allowed impact: *"Corruption of CKD outputs so honest parties accept inconsistent public keys, participant sets, transcripts, or unusable cryptographic outputs."*

---

### Likelihood Explanation

Any single participant in the CKD session can trigger this. The attacker needs only to call `ckd()` with a chosen `app_pk` value (e.g., the identity element, a random point, or their own public key). No cryptographic break, no leaked key, and no external assumption is required. The attack is silent — the coordinator receives a well-formed `CKDOutput` struct and has no mechanism to distinguish it from a correct one.

---

### Recommendation

Include `app_pk` in a broadcast commitment round before participants compute their shares, analogous to how `app_id` is already domain-separated into the hash. Concretely:

1. Add a broadcast round at the start of `run_ckd_protocol` where every participant broadcasts a commitment to their `app_pk` (e.g., `H(app_pk)`).
2. Each participant verifies that all received commitments match their own `app_pk` before proceeding to `compute_signature_share`.
3. Alternatively, include `app_pk` in the session transcript hash so that any divergence causes a session-ID mismatch and protocol abort.

This mirrors the fix applied in the referenced audit: including the previously-unconstrained parameter (`airdropAmount` / `app_pk`) in the authenticated payload so it cannot be freely substituted by a participant.

### Proof of Concept

```
Setup: 3-of-3 CKD session. Participants P1 (honest), P2 (honest), P3 (malicious).
       Legitimate app_pk = A·G  (app_sk = A).

Step 1: P1 and P2 call ckd(..., app_pk = A·G, ...) — honest.
Step 2: P3 calls ckd(..., app_pk = B·G, ...) — malicious substitution.

Step 3: P3's compute_signature_share produces:
          big_c_3 = big_s_3 + (B·G) · y_3   ← uses wrong app_pk

Step 4: P3 sends (norm_big_y_3, norm_big_c_3) to coordinator.
        Message contains only the curve points — no app_pk field.
        [src/confidential_key_derivation/protocol.rs line 30]

Step 5: Coordinator sums all contributions:
          big_c = big_c_1 + big_c_2 + big_c_3
                = (big_s_1 + A·G·y_1) + (big_s_2 + A·G·y_2) + (big_s_3 + B·G·y_3)

Step 6: unmask(A) = big_c − A · big_y
                  = msk·H(pk‖app_id) + (B·G − A·G)·λ_3·y_3
                  ≠ msk·H(pk‖app_id)

Result: CKDOutput is accepted by the coordinator but is permanently corrupted.
        The confidential key is unrecoverable. No error is raised.
```

pub fn ckd(
    participants: &[Participant],
    coordinator: Participant,
    me: Participant,
    key_pair: KeygenOutput,
    app_id: impl Into<AppId>,
    app_pk: PublicKey,
    rng: impl CryptoRngCore + Send + 'static,
) -> Result<impl Protocol<Output = CKDOutputOption>, InitializationError> {
    // not enough participants
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }

    // kick out duplicates
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }

    let comms = Comms::new();
    let chan = comms.shared_channel();

    let fut = run_ckd_protocol(
        chan,
        coordinator,
        me,
        participants,
        key_pair,
        app_id.into(),
        app_pk,
        rng,
    );
    Ok(make_protocol(comms, fut))
}
