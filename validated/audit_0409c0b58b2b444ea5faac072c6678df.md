After thorough analysis of the codebase, I found a concrete analog.

The `StatefulTransactionValidatorConfig` defines `max_nonce_for_validation_skip` (default: `Nonce(Felt::ONE)`, documented as "Maximum nonce for which the validation is skipped"), but the production `skip_stateful_validations` function in the Rust gateway **never reads this field** — it hardcodes `tx.nonce() == Nonce(Felt::ONE)`. The Python-path validator (`PyValidator::should_run_stateful_validations`) correctly consults `self.max_nonce_for_validation_skip`. This is structurally identical to the Venus bug: a precondition check that is too narrow (hardcoded to one specific state) ignores a configurable bound, causing valid transactions to be rejected.

### Title
`skip_stateful_validations` Ignores `max_nonce_for_validation_skip` Config, Causing Valid Invoke Transactions to Be Rejected at Gateway Admission — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The Rust gateway's `skip_stateful_validations` function hardcodes the nonce equality check to `Nonce(Felt::ONE)` and never reads the `max_nonce_for_validation_skip` field that exists in `StatefulTransactionValidatorConfig`. When that config value is set above 1, any invoke transaction with `1 < nonce ≤ max_nonce_for_validation_skip` submitted alongside a pending `deploy_account` will not receive the validation-skip, will be sent to the blockifier for `__validate__` against a not-yet-deployed account, and will be rejected. The Python-path validator (`PyValidator`) correctly implements the same feature using the same config field.

---

### Finding Description

`StatefulTransactionValidatorConfig` declares:

```rust
pub max_nonce_for_validation_skip: Nonce,
```

with default `Nonce(Felt::ONE)` and the schema description "Maximum nonce for which the validation is skipped." [1](#0-0) 

The production Rust gateway function `skip_stateful_validations` is a free function that receives no config argument and hardcodes the nonce check:

```rust
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
``` [2](#0-1) 

`run_pre_validation_checks` calls this free function but never passes `self.config.max_nonce_for_validation_skip`:

```rust
let skip_validate =
    skip_stateful_validations(executable_tx, account_nonce, mempool_client.clone()).await?;
``` [3](#0-2) 

By contrast, the Python-path validator correctly uses the field:

```rust
let nonce_small_enough_to_qualify_for_validation_skip =
    tx_nonce <= self.max_nonce_for_validation_skip;
let skip_validate = deploy_account_not_processed
    && is_post_deploy_nonce
    && nonce_small_enough_to_qualify_for_validation_skip;
``` [4](#0-3) 

The `StatefulValidator::perform_validations` path for `Invoke` runs `perform_pre_validation_stage` and then the `__validate__` entry point when `execution_flags.validate == true`. For an account that has not yet been deployed (nonce=0 on-chain), this call will fail because the contract does not exist. [5](#0-4) 

The config schema confirms the field is public and operator-configurable with a default of `0x1`: [6](#0-5) 

---

### Impact Explanation

**High — Gateway rejects valid transactions before sequencing.**

When `max_nonce_for_validation_skip` is set to any value `N > 1`, a user who submits `deploy_account` + `invoke(nonce=1)` + `invoke(nonce=2)` … `invoke(nonce=N)` simultaneously will have all invokes with `nonce > 1` rejected at the gateway with a `ValidateFailure` error, even though those transactions are semantically valid and the config explicitly declares they should be admitted. The `deploy_account` and `invoke(nonce=1)` are accepted; every subsequent invoke in the batch is dropped. No funds are at risk, but the gateway's admission decision is wrong relative to its own declared configuration.

---

### Likelihood Explanation

The default value of `max_nonce_for_validation_skip` is `0x1`, so the bug is silent with the default configuration. It activates the moment an operator raises the value above 1 — a documented, public, operator-facing knob. The existence of the field in the config schema and its correct use in the Python validator strongly imply the intent was for the Rust gateway to honour it as well.

---

### Recommendation

Convert `skip_stateful_validations` from a free function into a method on `StatefulTransactionValidator` (or pass `max_nonce_for_validation_skip` as an explicit parameter) and replace the hardcoded equality with the range check used by `PyValidator`:

```rust
// inside run_pre_validation_checks, replace the free-function call with:
let skip_validate = skip_stateful_validations(
    executable_tx,
    account_nonce,
    self.config.max_nonce_for_validation_skip,
    mempool_client.clone(),
).await?;

// inside skip_stateful_validations:
if tx.nonce() >= Nonce(Felt::ONE)
    && tx.nonce() <= max_nonce_for_validation_skip
    && account_nonce == Nonce(Felt::ZERO)
{
    // existing mempool check …
}
```

### Proof of Concept

1. Configure the node with `max_nonce_for_validation_skip = 0x3`.
2. Submit three transactions from an undeployed account in one batch:
   - `deploy_account` (nonce 0)
   - `invoke` (nonce 1)
   - `invoke` (nonce 2)
3. **Observed**: `deploy_account` and `invoke(nonce=1)` are accepted; `invoke(nonce=2)` is rejected with `ValidateFailure` because `skip_stateful_validations` returns `false` (hardcoded `== Nonce(Felt::ONE)` fails for nonce 2), so the blockifier attempts `__validate__` on a not-yet-deployed account.
4. **Expected**: all three transactions should be admitted, because `nonce=2 ≤ max_nonce_for_validation_skip=3` and a `deploy_account` is present in the mempool.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}

    pub fn perform_validations(&mut self, tx: AccountTransaction) -> StatefulValidatorResult<()> {
        // Deploy account transaction should be fully executed, since the constructor must run
        // before `__validate_deploy__`. The execution already includes all necessary validations,
        // so they are skipped here.
        // Declare transaction should also be fully executed - otherwise, if we only go through
        // the validate phase, we would miss the check that the class was not declared before.
        match tx.tx {
            ApiTransaction::DeployAccount(_) | ApiTransaction::Declare(_) => self.execute(tx),
            ApiTransaction::Invoke(_) => {
                let tx_context = Arc::new(self.tx_executor.block_context.to_tx_context(&tx));
                tx.perform_pre_validation_stage(self.state(), &tx_context)?;
                if !tx.execution_flags.validate {
                    return Ok(());
                }

                // `__validate__` call.
                let (_optional_call_info, actual_cost) = self.validate(&tx, tx_context.clone())?;

                // Post validations.
                PostValidationReport::verify(
                    &tx_context,
                    &actual_cost,
                    tx.execution_flags.charge_fee,
                )?;

                Ok(())
            }
        }
    }
