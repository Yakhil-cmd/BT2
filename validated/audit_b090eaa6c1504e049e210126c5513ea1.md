### Title
Unguarded `InteropRootAdded` Event Hook Allows Unprivileged Injection of Fake Interop Roots into Batch Public Input - (File: `system_hooks/src/event_hooks/interop_root_reporter.rs`)

---

### Summary

`interop_root_reporter_event_hook` fires on any `InteropRootAdded` event emitted by **any contract**, not only by the legitimate `L2_INTEROP_ROOT_STORAGE_ADDRESS`. An unprivileged L2 user can deploy a contract that emits this event with attacker-controlled `chain_id`, `block_or_batch_number`, and `root` values. The hook unconditionally calls `system.io.add_interop_root()`, appending the fake root to `InteropRootStorage`. This corrupts the `interop_roots_rolling_hash` committed in the batch public input, causing the batch to be rejected on the settlement layer or enabling cross-chain message forgery.

---

### Finding Description

`interop_root_reporter_event_hook` performs no check on the address of the contract that emitted the event. The only guards are:

- Event signature matches `InteropRootAdded`
- `data.len() == 96`
- ABI offset equals 32
- Exactly one side (`len == 1`)
- Exactly 3 topics [1](#0-0) 

The `_caller_ee` parameter (execution environment type) is explicitly ignored: [2](#0-1) 

There is no `address` parameter in the hook signature at all. Any contract emitting `InteropRootAdded(uint256 indexed chainId, uint256 indexed blockOrBatchNumber, bytes32[] sides)` with one side passes all checks and causes the hook to call: [3](#0-2) 

`add_interop_root` in the IO subsystem performs no source validation and delegates directly to `push_root`: [4](#0-3) 

`push_root` appends unconditionally with no uniqueness or origin check: [5](#0-4) 

At batch finalization, `calculate_interop_roots_rolling_hash` iterates over **all** accumulated roots (including attacker-injected ones) to produce the `interop_roots_rolling_hash` committed in `BatchOutput`: [6](#0-5) 

This hash is included in `BatchOutput.interop_roots_rolling_hash`: [7](#0-6) 

And hashed into the final `BatchPublicInput` that the settlement layer verifies: [8](#0-7) 

The same injection path exists in the multiblock batch accumulator: [9](#0-8) 

---

### Impact Explanation

**Vulnerability class:** Oracle IO mismatch / state-transition bug — the system accepts interop root data from untrusted sources.

1. **Batch rejection / chain halt (DoS):** The settlement layer verifies `interop_roots_rolling_hash` against the roots it knows were legitimately imported via service transactions. A fake root injected by an attacker changes the rolling hash, causing the batch proof to be invalid or the settlement layer to reject the batch. Since every block in the batch is affected, this can halt the chain's ability to settle.

2. **Cross-chain message forgery (funds loss):** If the settlement layer accepts the batch public input as authoritative (trusting the ZK proof), an attacker-controlled `root` value is treated as a valid cross-chain Merkle root. This could allow the attacker to forge cross-chain messages and drain assets bridged via the interop system.

---

### Likelihood Explanation

Any unprivileged L2 user who can deploy a contract and pay gas can trigger this. The attacker-controlled inputs (`chain_id`, `block_or_batch_number`, `root`) are fully controllable via the emitted event topics and data. No special role, key, or oracle manipulation is required. The attack is executable in a single L2 transaction.

---

### Recommendation

Add an emitting-address check in `interop_root_reporter_event_hook`. The hook should only process events emitted by `L2_INTEROP_ROOT_STORAGE_ADDRESS`:

```rust
// At the top of interop_root_reporter_event_hook, add:
// (requires passing the emitting address into the hook, or checking caller_address)
if emitting_address != L2_INTEROP_ROOT_STORAGE_ADDRESS {
    return Ok(());
}
```

Alternatively, pass the emitting contract address as a parameter to the event hook dispatch and reject events from non-whitelisted addresses before any parsing occurs.

---

### Proof of Concept

1. Deploy the following contract on L2:

```solidity
contract FakeInteropRootEmitter {
    // InteropRootAdded(uint256,uint256,bytes32[])
    event InteropRootAdded(
        uint256 indexed chainId,
        uint256 indexed blockOrBatchNumber,
        bytes32[] sides
    );

    function inject(uint256 chainId, uint256 blockNum, bytes32 fakeRoot) external {
        bytes32[] memory sides = new bytes32[](1);
        sides[0] = fakeRoot;
        emit InteropRootAdded(chainId, blockNum, sides);
    }
}
```

2. Call `inject(1, 100, 0xdeadbeef...)` from any L2 EOA.

3. The `interop_root_reporter_event_hook` fires (event signature matches, `data.len() == 96`, offset == 32, len == 1, topics.len() == 3), calls `add_interop_root` with the attacker-supplied values, and appends the fake `InteropRoot` to `InteropRootStorage`.

4. At block/batch finalization, `calculate_interop_roots_rolling_hash` includes the fake root, producing a `interop_roots_rolling_hash` that diverges from the value the settlement layer expects, causing batch rejection or enabling cross-chain message forgery.

```rust
pub fn calculate_interop_roots_rolling_hash<'a>(
    old_rolling_hash: Bytes32,
    roots: impl Iterator<Item = &'a InteropRoot>,
    hasher: &mut crypto::sha3::Keccak256,
) -> Bytes32 {
    let mut data = [0u8; 96];

    let mut rolling_hash = old_rolling_hash;
    for root in roots {
        data[0..32].copy_from_slice(&rolling_hash.as_u8_ref());
        data[32..64].copy_from_slice(&root.chain_id.to_be_bytes::<{ U256::BYTES }>());
        data[64..96].copy_from_slice(&root.block_or_batch_number.to_be_bytes::<{ U256::BYTES }>());
        hasher.update(data);

        // Note: now we have only one side
        hasher.update(root.root.as_u8_ref());

        rolling_hash = hasher.finalize_reset().into()
    }

    rolling_hash
}
```
