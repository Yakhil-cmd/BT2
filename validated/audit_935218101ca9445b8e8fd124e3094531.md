### Title
Pool Admin Bypasses `maxAdminSpreadFeeE6` Cap via Uncapped `setPoolBinAdditionalFees` — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`setPoolAdminFees` enforces `maxAdminSpreadFeeE6` on the global admin spread component, but `setPoolBinAdditionalFees` passes `addFeeBuyE6`/`addFeeSellE6` directly to the pool with **no factory-side cap check**, allowing a pool admin to set per-bin additional fees up to `uint16` max (65535 ≈ 6.55% in E6 units) on any bin, bypassing the protocol-enforced fee ceiling and extracting excess fees from traders.

---

### Finding Description

`setPoolAdminFees` in `MetricOmmPoolFactory` explicitly guards against exceeding the protocol cap: [1](#0-0) 

```solidity
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (newAdminNotionalFeeE8 > maxAdminNotionalFeeE8) revert AdminFeeTooHigh();
```

By contrast, `setPoolBinAdditionalFees` performs **no validation** on the fee values before forwarding them to the pool: [2](#0-1) 

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
  external override nonReentrant onlyPoolAdmin(pool)
{
  IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

The pool's `setBinAdditionalFees` also performs no cap check — it stores the raw values directly: [3](#0-2) 

These per-bin fees are then added directly to `baseFeeX64` in every swap through the affected bin: [4](#0-3) 

```solidity
params.baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6),
```

The same additive pattern appears in `getSellAndBuyPrices`: [5](#0-4) 

`uint16` max is 65535, which in E6 units equals ~6.55%. This is additive on top of the global spread fee and notional fee, with no ceiling enforced by the factory.

---

### Impact Explanation

A pool admin can call `setPoolBinAdditionalFees(pool, 0, 65535, 65535)` to set per-bin additional fees to ~6.55% on any bin. Every swap routed through that bin pays this fee on top of the global spread and notional fees. The excess fee accrues to LPs (not the admin directly), but the trader suffers direct principal loss far beyond any disclosed or protocol-enforced cap. This is a clear admin-boundary break: the pool admin exceeds the cap system that `maxAdminSpreadFeeE6` is meant to enforce.

**Impact: High** — direct loss of trader principal on every swap through the affected bin, with no protocol-enforced upper bound.

---

### Likelihood Explanation

Any pool admin (a role that can be held by a third-party deployer, not just the protocol) can call this at any time with no timelock. The call is a single transaction with no preconditions beyond holding the `poolAdmin` role.

**Likelihood: Medium** — requires a malicious or compromised pool admin, but the role is semi-trusted and the bypass is trivially executable.

---

### Recommendation

Add a cap check in `setPoolBinAdditionalFees` analogous to the one in `setPoolAdminFees`:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
  external override nonReentrant onlyPoolAdmin(pool)
{
  if (addFeeBuyE6 > maxAdminBinFeeE6) revert AdminFeeTooHigh();
  if (addFeeSellE6 > maxAdminBinFeeE6) revert AdminFeeTooHigh();
  IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

A separate `maxAdminBinFeeE6` cap (or reuse of `maxAdminSpreadFeeE6`) should be introduced and enforced at the factory level, consistent with how global admin fees are bounded.

---

### Proof of Concept

1. Deploy a pool with a standard configuration and a pool admin controlled by the attacker.
2. Call `factory.setPoolBinAdditionalFees(pool, 0, 65535, 65535)` — succeeds with no revert.
3. A trader swaps through bin 0. The effective buy fee is `baseFeeX64 + (65535 / 1e6) * ONE_X64`, i.e., ~6.55% additional on top of the global spread.
4. Assert the trader receives significantly less than the oracle mid-price permits, far exceeding `maxAdminSpreadFeeE6`. [2](#0-1) [3](#0-2)
