### Title
`skip_stateful_validations` ignores `max_nonce_for_validation_skip` config, hardcoding `Nonce(Felt::ONE)` — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

`StatefulTransactionValidatorConfig` exposes a documented, serialized, deployment-configured field `max_nonce_for_validation_skip` that is supposed to control the upper nonce bound for which gateway-level `__validate__` is skipped on invoke transactions from newly-deployed accounts. The Rust gateway's `skip_stateful_validations` function ignores this field entirely and hardcodes `Nonce(Felt::ONE)` instead. The Python-path validator (`PyValidator::should_run_stateful_validations`) correctly reads `self.max_nonce_for_validation_skip`. The inconsistency means the config knob is dead code in the Rust gateway, and any operator-set value other than the default `0x1` silently produces the wrong admission decision.

### Finding Description

`StatefulTransactionValidatorConfig` declares:

```rust
pub max_nonce_for_validation_skip: Nonce,
```

with default `Nonce(Felt::ONE)`, documented as *"Maximum nonce for which the validation is skipped"*, serialized in `config_schema.json` and `gateway_config.json`. [1](#0-0) 

The Rust gateway's `skip_stateful_validations` function — a free function that does not receive `self` — hardcodes the bound:

```rust
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
``` [2](#0-1) 

`run_pre_validation_checks` calls it without forwarding the config:

```rust
let skip_validate =
    skip_stateful_validations(executable_tx, account_nonce, mempool_client.clone()).await?;
``` [3](#0-2) 

The Python-path validator (`PyValidator`) correctly uses the configurable bound:

```rust
let nonce_small_enough_to_qualify_for_validation_skip =
    tx_nonce <= self.max_nonce_for_validation_skip;
let skip_validate = deploy_account_not_processed
    && is_post_deploy_nonce
    && nonce_small_enough_to_qualify_for_validation_skip;
``` [4](#0-3) 

A grep for `max_nonce_for_validation_skip` inside `crates/apollo_gateway/` returns zero matches — the field is completely dead in the Rust gateway path.

**Divergence table:**

| Config value | Python validator skips? | Rust gateway skips? | Effect |
|---|---|---|---|
| `0x0` (disable feature) | No | **Yes** (hardcoded `== 1`) | Invalid-signature invoke admitted |
| `0x1` (default) | Yes | Yes | Correct |
| `0x2` (extend range) | Yes (nonces 1–2) | **No** (only nonce 1) | Valid invoke rejected |

### Impact Explanation

When an operator sets `max_nonce_for_validation_skip = 0x0` to disable the validation-skip UX feature, the Rust gateway still skips the blockifier `__validate__` entry-point call for any invoke transaction with `nonce == 1` from an account whose on-chain nonce is `0` and that has a `deploy_account` in the mempool. The transaction is admitted to the mempool without signature verification. This satisfies **High: Mempool/gateway/RPC admission accepts invalid transactions before sequencing**. [5](#0-4) 

### Likelihood Explanation

The field is present in the production deployment config (`gateway_config.json` line 18: `"0x1"`) and in `config_schema.json`. Any operator who changes it to `0x0` to harden the gateway, or to `0x2` to extend the UX window, will silently get the wrong behavior. The default value of `0x1` happens to coincide with the hardcoded value, masking the bug under normal operation. [6](#0-5) 

### Recommendation

Pass `self.config.max_nonce_for_validation_skip` into `skip_stateful_validations` (or make it a method on `StatefulTransactionValidator`) and replace the hardcoded equality check with the same range check used by `PyValidator`:

```rust
// Instead of:
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {

// Use:
if tx.nonce() >= Nonce(Felt::ONE)
    && tx.nonce() <= max_nonce_for_validation_skip
    && account_nonce == Nonce(Felt::ZERO) {
```

This aligns the Rust gateway with the Python validator's logic and makes the config field effective.

### Proof of Concept

1. Deploy a sequencer node with `stateful_tx_validator_config.max_nonce_for_validation_skip = 0x0` (operator disables the skip feature).
2. Submit a `deploy_account` transaction for account `A` — it enters the mempool.
3. Submit an `invoke` transaction from `A` with `nonce = 1` and a **deliberately invalid signature**.
4. Gateway calls `skip_stateful_validations`: `tx.nonce() == Nonce(Felt::ONE)` is `true`, `account_nonce == Nonce(Felt::ZERO)` is `true`, `account_tx_in_pool_or_recent_block` returns `true` → returns `true` (skip validation).
5. `run_validate_entry_point` is called with `validate: false` — the `__validate__` entry point is never executed.
6. The invalid-signature invoke transaction is forwarded to the mempool and accepted, violating the operator's intent and the gateway admission invariant. [7](#0-6)

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
