### Title
Single `tick()` Call Exhausts Weekly `PERIOD_MINT_CAP`, Permanently Blocking Remaining Pending Mint Requests for Up to One Week — (`L1/starkware/solidity/stake/RewardSupplier.sol`)

### Summary
`MAX_MESSAGES_TO_PROCESS_PER_TICK (5) × TOKENS_PER_MINT_REQUEST (1,300,000e18) = PERIOD_MINT_CAP (6,500,000e18)` exactly. A single permissionless `tick()` call when ≥5 messages are pending exhausts the entire weekly minting quota. Any additional pending messages are frozen until the next period (up to 7 days).

### Finding Description
`tick()` is an unrestricted external function. `requiredMinting()` caps the number of messages consumed per call at `MAX_MESSAGES_TO_PROCESS_PER_TICK = 5`. Each message requests `TOKENS_PER_MINT_REQUEST = 1,300,000e18` tokens. Therefore one `tick()` call mints at most `5 × 1,300,000e18 = 6,500,000e18` tokens — which is precisely `PERIOD_MINT_CAP`.

`MintManager.mintRequest` calls `checkAndUpdatePeriodicalQuota`, which reverts with `"EXCEED_PERIOD_MINTING"` if the cumulative minted amount in the current period would exceed `PERIOD_MINT_CAP`. After one full `tick()`, `mintedThisPeriodAfter == PERIOD_MINT_CAP`, so any subsequent `tick()` call in the same period — even for a single remaining message — will revert because `mintedThisPeriodAfter = PERIOD_MINT_CAP + 1,300,000e18 > PERIOD_MINT_CAP`.

The L2 `request_funds` function can queue more than 5 messages in a single invocation when the accumulated reward deficit is large enough (e.g., after a long idle period or a large stake increase). This is a normal, expected operational scenario, not an edge case.

### Impact Explanation
Any pending mint requests beyond the first 5 are frozen for up to `MINTING_PERIOD_DURATION = 1 week`. During this window, the L2 `RewardSupplier` does not receive the expected token deposits via StarkGate, and stakers/pool members attempting to claim rewards that depend on those deposits will be unable to do so. This constitutes **temporary freezing of unclaimed yield** for up to one week.

### Likelihood Explanation
- `tick()` requires no role, no stake, no special state — any EOA can call it.
- The L2 side can legitimately queue >5 messages (e.g., `num_msgs = 10` when a large deficit accumulates).
- The attacker needs only to observe the pending message count (publicly readable from the Starknet messaging contract) and call `tick()` at the right moment.
- No profit motive is required; the cost is only the gas + `msg.value` for the bridge fee.

### Recommendation
Decouple the per-tick message cap from the period mint cap. Either:
1. Raise `PERIOD_MINT_CAP` to be a multiple of `MAX_MESSAGES_TO_PROCESS_PER_TICK × TOKENS_PER_MINT_REQUEST` (e.g., allow at least 2 full ticks per period), or
2. In `tick()`, check the remaining period quota before consuming messages and only consume as many messages as the remaining quota allows, or
3. Add a pre-check in `tick()` that reverts gracefully (rather than mid-execution) if the quota is already exhausted, so callers can retry next period without wasting gas.

### Proof of Concept
```
// Setup: 10 L2->L1 mint request messages are pending in the messaging contract.
// Period is fresh (mintedThisPeriod == 0).

// Step 1: Anyone calls tick() with sufficient msg.value for bridge fees.
rewardSupplier.tick{value: 2 ether}();
// → requiredMinting() returns (6_500_000e18, 5)  [capped at MAX_MESSAGES_TO_PROCESS_PER_TICK]
// → 5 messages consumed, 6_500_000e18 minted
// → checkAndUpdatePeriodicalQuota: mintedThisPeriodAfter = 6_500_000e18 == PERIOD_MINT_CAP ✓

// Step 2: 5 messages still pending. Call tick() again in the same period.
rewardSupplier.tick{value: 2 ether}();
// → requiredMinting() returns (6_500_000e18, 5)  [5 messages still pending]
// → mintRequest called with 6_500_000e18
// → checkAndUpdatePeriodicalQuota: 6_500_000e18 + 6_500_000e18 > PERIOD_MINT_CAP → REVERT "EXCEED_PERIOD_MINTING"

// Assert: 5 messages remain pending, all tick() calls revert for up to 1 week.
assert(messagingContract.l2ToL1Messages(msgHash) == 5);
```
