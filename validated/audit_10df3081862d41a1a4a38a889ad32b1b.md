### Title
Missing Zero-Address Validation in `change_reward_address` Allows Permanent Loss of Unclaimed Yield - (File: `src/staking/staking.cairo`, `src/pool/pool.cairo`)

---

### Summary
Both the `Staking` and `Pool` contracts allow a staker or pool member to set their `reward_address` to the zero address (`0x0`) via `change_reward_address`. Neither function validates against the zero address. When `claim_rewards` is subsequently called, accumulated STRK rewards are transferred to address(0) via `checked_transfer`, permanently destroying them.

---

### Finding Description

**Root cause — `Staking::change_reward_address`** (`src/staking/staking.cairo`, lines 517–540):

```cairo
fn change_reward_address(ref self: ContractState, reward_address: ContractAddress) {
    self.general_prerequisites();
    assert!(
        !self.does_token_exist(token_address: reward_address),
        "{}",
        GenericError::REWARD_ADDRESS_IS_TOKEN,
    );
    let staker_address = get_caller_address();
    let mut staker_info = self.internal_staker_info(:staker_address);
    // ...
    staker_info.reward_address = reward_address;  // no zero-address check
    self.write_staker_info(:staker_address, :staker_info);
```

The only guard is `REWARD_ADDRESS_IS_TOKEN` — there is no check that `reward_address.is_non_zero()`. A staker can freely call `change_reward_address(0)`. [1](#0-0) 

**Root cause — `Pool::change_reward_address`** (`src/pool/pool.cairo`, lines 505–526):

```cairo
fn change_reward_address(ref self: ContractState, reward_address: ContractAddress) {
    assert!(
        self.token_dispatcher.contract_address.read() != reward_address,
        "{}",
        GenericError::REWARD_ADDRESS_IS_TOKEN,
    );
    let pool_member = get_caller_address();
    let mut pool_member_info = self.internal_pool_member_info(:pool_member);
    // ...
    pool_member_info.reward_address = reward_address;  // no zero-address check
```

Same pattern — only the token-address guard, no zero-address guard. [2](#0-1) 

**Reward transfer sinks** — once `reward_address` is zero, the next `claim_rewards` call routes tokens there:

- Staking contract internal helper `send_rewards_to_staker` (line 1625):
  ```cairo
  token_dispatcher.checked_transfer(recipient: reward_address, amount: amount.into());
  ``` [3](#0-2) 

- Pool contract `claim_rewards` (line 366):
  ```cairo
  reward_token.checked_transfer(recipient: reward_address, amount: rewards.into());
  ``` [4](#0-3) 

**Access control does not prevent the loss.** `Pool::claim_rewards` checks:
```cairo
assert!(
    caller_address == pool_member || caller_address == reward_address, ...
);
```
When `reward_address == 0`, the `pool_member` branch still passes, so the pool member can trigger the transfer to address(0) themselves. [5](#0-4) 

---

### Impact Explanation
Any staker or pool member who calls `change_reward_address(0)` — whether by mistake, UI bug, or deliberate self-sabotage — will have all future (and already-accrued) STRK rewards permanently sent to address(0) on the next `claim_rewards` call. The tokens are irrecoverable. This constitutes **permanent freezing of unclaimed yield**, which is within the allowed High-severity impact scope.

---

### Likelihood Explanation
**Low-to-Medium.** The call requires the staker/pool member themselves to set address(0). This can happen via:
- A front-end bug that passes an empty/null address
- A user scripting an address update with a zero value
- A malicious actor who has obtained the staker's key

The protocol has no recovery mechanism once rewards are sent to address(0).

---

### Recommendation
Add a non-zero check in both `change_reward_address` implementations:

```cairo
assert!(reward_address.is_non_zero(), "Reward address cannot be zero");
```

Apply this in:
- `src/staking/staking.cairo` — `Staking::change_reward_address` (after line 519)
- `src/pool/pool.cairo` — `Pool::change_reward_address` (after line 506)

Optionally, also validate the initial `reward_address` supplied to `enter_delegation_pool` and `stake`.

---

### Proof of Concept

**Pool path (analogous to the original report):**

1. Pool member calls `pool.enter_delegation_pool(reward_address: VALID_ADDR, amount: X)` — normal delegation.
2. Epochs pass; rewards accrue in the pool's cumulative trace.
3. Pool member calls `pool.change_reward_address(reward_address: 0x0)` — succeeds because only `REWARD_ADDRESS_IS_TOKEN` is checked.
4. Pool member (or anyone who triggers it) calls `pool.claim_rewards(pool_member: MEMBER_ADDR)` — the `caller_address == pool_member` branch passes.
5. `checked_transfer(recipient: 0x0, amount: rewards)` executes — STRK rewards are permanently lost.

**Staking path:**

1. Staker calls `staking.stake(...)` with a valid `reward_address`.
2. Staker attests; `unclaimed_rewards_own` accumulates.
3. Staker calls `staking.change_reward_address(reward_address: 0x0)` — succeeds.
4. Staker calls `staking.claim_rewards(staker_address: STAKER_ADDR)`.
5. `send_rewards_to_staker` fires `checked_transfer(recipient: 0x0, amount: rewards)` — STRK rewards are permanently lost. [6](#0-5) [1](#0-0)
