### Title
Unprivileged Caller Can Permanently Grief Consensus Block Reward Distribution via `update_rewards(disable_rewards: true)` — (File: `src/staking/staking.cairo`)

---

### Summary

The `update_rewards` function in the Staking contract is callable by any non-zero address and accepts an attacker-controlled `disable_rewards` boolean. Because `last_reward_block` is a single global variable that gates the function to one execution per block, any unprivileged caller can front-run legitimate reward-distribution calls with `disable_rewards: true`, consuming the block's reward slot while distributing nothing. Sustained at the cost of only gas, this permanently denies all stakers their consensus block rewards.

---

### Finding Description

`update_rewards` is exposed as a public function with no role check beyond `general_prerequisites` (not-paused + caller-not-zero): [1](#0-0) 

The function enforces a single-execution-per-block invariant through a global `last_reward_block` storage variable: [2](#0-1) 

After passing all staker-validity checks, it unconditionally writes the current block number to `last_reward_block`: [3](#0-2) 

When `disable_rewards` is `true`, the function returns immediately after that write, distributing zero rewards: [4](#0-3) 

Because `last_reward_block` is global (not per-staker), any subsequent call to `update_rewards` in the same block — regardless of which staker it targets — will revert with `REWARDS_ALREADY_UPDATED`. The attacker therefore needs only one cheap transaction per block to nullify the entire epoch's block-reward distribution for all stakers.

The `general_prerequisites` guard that is the only barrier to entry: [5](#0-4) 

---

### Impact Explanation

After consensus rewards are activated (`!is_pre_consensus()`), every block's STRK and BTC rewards are distributed through a single `update_rewards` call. An attacker who occupies that slot with `disable_rewards: true` every block causes:

- **All stakers** to accrue zero consensus block rewards indefinitely.
- No funds already held are moved, but all future yield is permanently suppressed — matching the **Medium: Griefing with no profit motive but damage to users or protocol** impact category.

The attacker gains nothing; the protocol and every staker lose all consensus-era yield.

---

### Likelihood Explanation

- The entry point is fully permissionless; any EOA or contract qualifies.
- The attacker only needs to supply a valid active staker address (readable from public events or `get_stakers`).
- The gas cost on Starknet is low, making sustained per-block griefing economically viable.
- No privileged role, leaked key, or external dependency is required.
- The attack is analogous to the Basin "no-fee sandwich" DoS: zero protocol-level cost to the attacker, full disruption to legitimate users.

---

### Recommendation

1. **Restrict `disable_rewards`** to a privileged role (e.g., `security_agent`) so only trusted actors can suppress reward distribution.
2. **Alternatively**, remove the `disable_rewards` parameter entirely and handle the disable case through the existing `pause` mechanism, which already requires `security_agent` privileges.
3. **Or**, make `last_reward_block` per-staker so one attacker call cannot block all other stakers in the same block.

---

### Proof of Concept

1. Consensus rewards are active (`consensus_rewards_first_epoch` has passed).
2. Attacker reads any active staker address `S` from on-chain state.
3. At the start of every block, attacker submits:
   ```
   staking.update_rewards(staker_address: S, disable_rewards: true)
   ```
4. `last_reward_block` is set to the current block; no rewards are distributed.
5. Any legitimate `update_rewards` call in the same block reverts with `REWARDS_ALREADY_UPDATED`.
6. Repeated every block: all stakers receive zero consensus block rewards indefinitely.
7. Attacker's only cost is gas per block; no capital at risk, no profit required.
