Audit Report

## Title
Cache Key Includes Unvalidated `ingress_expiry` for Anonymous Queries, Enabling Cache Pollution — (`rs/boundary_node/ic_boundary/src/routes.rs`)

## Summary
The boundary node query response cache uses `RequestContext` as its cache key, and both `Hash` and `PartialEq` for `RequestContext` include `ingress_expiry`. For anonymous query requests, `ingress_expiry` is never validated at the boundary node — it is stored directly from the CBOR body with no range check. An attacker can send a stream of anonymous `QueryV2` requests with distinct `ingress_expiry` values, each producing a unique cache entry, evicting legitimate entries from the finite cache and forcing all subsequent legitimate queries to be cache misses.

## Finding Description
**Cache key includes `ingress_expiry`:** In `rs/boundary_node/ic_boundary/src/routes.rs` lines 94–125, both `Hash` and `PartialEq` for `RequestContext` include `self.ingress_expiry`. Two requests identical in `canister_id`, `sender`, `method_name`, and `arg` but differing only in `ingress_expiry` are treated as entirely different cache entries.

**No boundary-node-level validation of `ingress_expiry` for anonymous queries:** In `rs/boundary_node/ic_boundary/src/http/middleware/process.rs` lines 149–159, `ingress_expiry` is stored directly from `content.ingress_expiry` with no range check. The only place `ingress_expiry` is range-validated is in `rs/validator/src/ingress_validation.rs` lines 140–142, where `validate_ingress_expiry` is explicitly skipped for anonymous senders. This validation runs at the replica, after the cache layer has already processed the request.

**Cache bypasser does not guard against `ingress_expiry` variation:** In `rs/boundary_node/ic_boundary/src/http/middleware/cache.rs` lines 63–87, `BypasserIC::bypass` only bypasses for non-query request types, presence of a nonce, or non-anonymous senders. There is no check that normalizes or rejects requests varying only in `ingress_expiry`.

**Cache key extractor uses the full `RequestContext`:** In `rs/boundary_node/ic_boundary/src/http/middleware/cache.rs` lines 33–46, `KeyExtractorContext` returns `Arc<RequestContext>` directly as the cache key, so all fields included in `Hash`/`PartialEq` — including `ingress_expiry` — differentiate cache entries.

**Exploit flow:** Attacker sends anonymous `QueryV2` requests to a fixed `canister_id`/`method_name`/`arg`, each with a distinct `ingress_expiry` (e.g., incrementing integers). Since anonymous queries skip expiry validation entirely, all values are accepted. Each request inserts a new entry into the finite moka/LRU cache. Once the cache is full, new entries evict the oldest legitimate entries. Legitimate clients making repeated identical queries never achieve `CacheStatus::Hit`.

## Impact Explanation
This is a targeted cache pollution attack exploiting a specific logic flaw, not raw volumetric DDoS. The attacker does not need to overwhelm network bandwidth — they only need to maintain a stream of requests sufficient to keep the finite cache filled with unique entries. The result is elimination of the caching benefit for all legitimate query traffic on the affected boundary node and proportional increase in replica load. This matches the allowed High impact: **Application/platform-level DoS, or boundary/API security impact with concrete user or protocol harm, not based on raw volumetric DDoS.**

## Likelihood Explanation
- No authentication required: anonymous identity, no keys, no registration.
- No boundary-node-side guard detects or throttles requests varying only in `ingress_expiry`.
- IP-based rate limiting (`rate_limiting/mod.rs`) constrains per-IP request rate but does not prevent cache pollution — an attacker operating within the rate limit can still slowly evict all legitimate entries, and multiple source IPs trivially bypass the per-IP limit.
- Trivially scriptable: a simple loop incrementing `ingress_expiry` in a CBOR-encoded query body is sufficient.
- Continuously exploitable with low cost.

## Recommendation
Remove `ingress_expiry` from the `Hash` and `PartialEq` implementations of `RequestContext` in `rs/boundary_node/ic_boundary/src/routes.rs`. `ingress_expiry` is semantically irrelevant to query execution results and should not differentiate cache entries. The cache key for query responses should be based only on `canister_id`, `sender`, `method_name`, and `arg` (or `http_request`).

## Proof of Concept
The test harness in `rs/boundary_node/ic_boundary/src/http/middleware/cache.rs` already provides `gen_request_with_params` accepting an `ingress_expiry` parameter. A minimal reproducing test:

```rust
// Set cache_size = 10
// Flood with N > cache_size anonymous QueryV2 requests, each with distinct ingress_expiry
for i in 0..(cache_size + 100) {
    let req = gen_request_with_params(
        CANISTER_1, RequestType::QueryV2, false, DEFAULT_SIZE,
        i as u64, /*anonymous=*/true, StatusCode::OK,
    );
    app.call(req).await.unwrap();
}
// Now send the "legitimate" repeated query (ingress_expiry=0)
let req = gen_request_with_params(
    CANISTER_1, RequestType::QueryV2, false, DEFAULT_SIZE,
    0, true, StatusCode::OK,
);
let res = app.call(req).await.unwrap();
let cs = res.extensions().get::<CacheStatus<BypassReasonIC>>().cloned().unwrap();
// Asserts Miss — the entry was evicted by the flood
assert!(matches!(cs, CacheStatus::Miss(_)));
```
