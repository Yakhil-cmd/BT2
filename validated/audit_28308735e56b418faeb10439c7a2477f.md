### Title
Coordinator Accepts Unverified Participant Contributions in CKD Protocol, Allowing Malicious Participant to Corrupt Derived Key Output - (`File: src/confidential_key_derivation/protocol.rs`)

### Summary

The Confidential Key Derivation (CKD) coordinator aggregates participant-supplied `(norm_big_y, norm_big_c)` shares with no zero-knowledge proof that each participant used their actual key share. A single malicious participant can substitute arbitrary group elements, causing the coordinator to output a corrupted `CKDOutput` that does not correspond to the honest aggregate secret.

### Finding Description

`do_ckd_coordinator` receives one `CKDOutput` per participant and blindly adds the components together: [1](#0-0) 

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
```

The honest computation each participant is supposed to perform is: [2](#0-1) 

```
big_s  = private_share * H(pk || app_id)   // key-share contribution
big_c  = big_s + y * app_pk               // ElGamal ciphertext
norm_big_y = λ_i · (y · G)
norm_big_c = λ_i · big_c
```

There is no accompanying proof that `norm_big_c` was formed with the participant's actual `private_share`. The coordinator has no mechanism to distinguish a correct contribution from an arbitrary pair of G1 points.

The codebase already contains a discrete-log-equality proof (`src/crypto/proofs/dlogeq.rs`) that proves knowledge of a scalar `x` satisfying `public0 = x·G` and `public1 = x·H` simultaneously — exactly the relation needed to bind a participant's G2 verification share to their G1 key-share contribution. That infrastructure is not invoked in the CKD path. [3](#0-2) 

### Impact Explanation

A malicious participant sends `(norm_big_y', norm_big_c')` where `norm_big_c'` is an arbitrary G1 point unrelated to their key share. The coordinator computes:

```
C_final = Σ norm_big_c_i  (includes the attacker's arbitrary point)
Y_final = Σ norm_big_y_i
```

The resulting `CKDOutput` no longer decrypts to `H(pk, app_id) · msk`. Any consumer of the output (e.g., a TEE that calls `ckd_output.unmask(app_sk)`) receives a wrong derived key. This maps directly to:

> **High: Corruption of CKD outputs so honest parties accept inconsistent … or unusable cryptographic outputs.**

### Likelihood Explanation

Any single participant in the CKD protocol can trigger this. The attacker-controlled entry path is the `send_private` call in `do_ckd_participant`: [4](#0-3) 

```rust
chan.send_private(waitpoint, coordinator, &(norm_big_y, norm_big_c))?;
```

No authentication or proof is attached to the message. The coordinator's receive loop performs no validation before accumulating the values. [5](#0-4) 

### Recommendation

Require each participant to attach a zero-knowledge proof that their `norm_big_c` was formed using the key share committed to during DKG. Concretely, the participant should prove:

- They know `x_i` such that `vk_share_i = x_i · G2` (from DKG output) **and** `big_s_i = x_i · H(pk, app_id)` (in G1).

This is a cross-group DLEQ relation; a pairing-based proof (e.g., using the BLS12-381 pairing already available in the ciphersuite) can bind the G2 verification share to the G1 contribution. The coordinator must verify this proof before accumulating each participant's `(norm_big_y, norm_big_c)`.

### Proof of Concept

1. Honest participants run `ckd(...)` normally.
2. Attacker participant intercepts the protocol at `do_ckd_participant` and replaces the outgoing message with `(G1::generator(), G1::generator())` (arbitrary points).
3. Coordinator accumulates the attacker's arbitrary point into `norm_big_c`.
4. `CKDOutput::new(norm_big_y, norm_big_c)` is returned to the caller.
5. Caller invokes `ckd_output.unmask(app_sk)` and receives `C - app_sk·Y ≠ H(pk, app_id)·msk`.
6. The derived confidential key is wrong; any downstream use (e.g., TEE secret derivation) silently operates on an attacker-influenced value.

pub fn verify<C: Ciphersuite>(
    transcript: &mut Transcript,
    statement: Statement<'_, C>,
    proof: &Proof<C>,
) -> Result<bool, ProtocolError>
where
    Element<C>: ConstantTimeEq,
{
    if statement.generator1.ct_eq(&C::Group::identity()).into() {
        return Err(ProtocolError::IdentityElement);
    }

    transcript.message(NEAR_DLOGEQ_STATEMENT_LABEL, &statement.encode()?);

    let (phi0, phi1) = statement.phi(&proof.s.0);
    let big_k0 = phi0 - *statement.public0 * proof.e.0;
    let big_k1 = phi1 - *statement.public1 * proof.e.0;

    let enc = encode_two_points::<C>(&big_k0, &big_k1)?;

    transcript.message(NEAR_DLOGEQ_COMMITMENT_LABEL, &enc);
    let mut rng = transcript.challenge_then_build_rng(NEAR_DLOGEQ_CHALLENGE_LABEL);
    let e = frost_core::random_nonzero::<C, _>(&mut rng);

    Ok(e == proof.e.0)
