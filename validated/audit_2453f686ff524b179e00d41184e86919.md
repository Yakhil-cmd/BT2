### Title
`skip_stateful_validations` Bypasses `__validate__` Signature Check for Nonce-1 Invokes via Overly Broad Mempool Presence Check - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The gateway's UX shortcut for the deploy-account + invoke flow (`skip_stateful_validations`) uses `account_tx_in_pool_or_recent_block` as a proxy for "a deploy_account is pending." That check returns `true` for **any** transaction from the account in the mempool — including a regular invoke. An unprivileged attacker who observes that a victim account has nonce=0 on-chain and any pending transaction in the mempool can submit an invoke with nonce=1 and an **arbitrary wrong signature**. The gateway skips the `__validate__` entry point entirely, and the invalid transaction is admitted to the mempool without signature verification.

---

### Finding Description

**Broken invariant:** Every invoke transaction admitted to the mempool must have its signature verified by the account's `__validate__` entry point.

**Root cause — `skip_stateful_validations`:** [1](#0-0) 

The function returns `true` (skip validation) when all three conditions hold:
1. The transaction is an Invoke.
2. `tx.nonce() == Nonce(Felt::ONE)` (hardcoded).
3. `account_nonce == Nonce(Felt::ZERO)` (on-chain nonce is 0).
4. `account_tx_in_pool_or_recent_block(sender_address)` returns `true`.

The code comment claims condition 4 proves "either it has a deploy_account transaction or transactions with future nonces that passed validations." The second branch of that disjunction is the flaw: if the account already has a committed nonce=0 on-chain (i.e., it was deployed and its first invoke is pending), `account_tx_in_pool_or_recent_block` returns `true` via `tx_pool.contains_account`, yet no deploy_account is pending. [2](#0-1) 

**How the skip propagates to execution:**

`run_validate_entry_point` sets `validate: !skip_validate` in `ExecutionFlags`: [3](#0-2) 

`StatefulValidator::perform_validations` then short-circuits before calling `__validate__`: [4](#0-3) 

The `__validate__` entry point — the only place where the account contract verifies the ECDSA signature — is never reached: [5](#0-4) 

**Why no other guard catches it:**

`validate_by_mempool` only checks nonce ordering, duplicate tx_hash, and fee escalation — no signature field exists in `ValidationArgs`: [6](#0-5) 

`perform_pre_validation_stage` checks nonce increment, fee bounds, and balance — not signatures: [7](#0-6) 

---

### Impact Explanation

An attacker can inject an invoke transaction with an **arbitrary wrong signature** into the mempool for any victim account that satisfies:
- On-chain nonce = 0 (newly deployed account, before its first committed invoke).
- Any transaction from that account is currently in the mempool.

The invalid transaction passes all gateway checks and is forwarded to the mempool. The batcher will later attempt to execute it; it will fail at the `__validate__` entry point during block building. However, the transaction was **admitted** — violating the admission invariant. Consequences include: mempool slot consumption, batcher CPU waste, and potential ordering interference with the victim's legitimate nonce-1 transaction.

This matches: **High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.**

---

### Likelihood Explanation

The preconditions are observable from public state:
- On-chain nonce=0 is visible via `starknet_getNonce`.
- Pending transactions are visible via the mempool P2P layer or the gateway's own RPC.

No privileged access is required. Any newly deployed account that submits its first invoke is transiently vulnerable until that invoke is committed and the on-chain nonce advances past 0.

---

### Recommendation

Replace the overly broad `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a **deploy_account** transaction is pending for the sender address. Alternatively, check whether the account's class hash is unset on-chain (indicating the account has not yet been deployed), which is the true precondition that makes skipping `__validate__` safe. If the account already exists on-chain (class hash is set), the `__validate__` entry point must always be called regardless of mempool state.

---

### Proof of Concept

```
1. Alice deploys an account at address A. On-chain nonce(A) = 0.

2. Alice submits a valid invoke(nonce=0, sig=valid_sig) from A.
   → Gateway: stateless OK, stateful OK (account exists, __validate__ passes).
   → Mempool: tx admitted. account_tx_in_pool_or_recent_block(A) now returns true.

3. Eve submits invoke(nonce=1, sig=[0xDEAD, 0xBEEF]) from A (wrong signature).
   → Stateless: valid address, valid resource bounds, valid DA modes → OK.
   → Stateful:
       account_nonce = get_nonce_from_state(A) = 0
       validate_nonce: 0 ≤ 1 ≤ 200 → OK
       validate_by_mempool: no duplicate hash, nonce=1 > account_nonce=0 → OK
       skip_stateful_validations:
           tx.nonce() == 1  ✓
           account_nonce == 0  ✓
           account_tx_in_pool_or_recent_block(A) == true  ✓  (Alice's nonce=0 invoke)
           → returns true (SKIP __validate__)
       run_validate_entry_point: ExecutionFlags { validate: false }
           → perform_validations returns Ok(()) without calling __validate__
   → Eve's invalid invoke is admitted to the mempool.

4. Batcher picks up Eve's nonce=1 invoke, executes it.
   __validate__ fails (wrong signature). Transaction reverts.
   Alice's legitimate nonce=1 invoke (if any) is delayed or displaced.
``` [8](#0-7)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
