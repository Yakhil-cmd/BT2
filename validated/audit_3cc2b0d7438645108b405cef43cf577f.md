### Title
Incorrect Scalar Computation in `from_be_bytes_mod_order` Due to Spurious Multiplication — (`src/confidential_key_derivation/scalar_wrapper.rs`)

### Summary
The `from_be_bytes_mod_order` function in `ScalarWrapper` contains an operator error on the remainder-handling branch: it writes `res * res.shl(count * 8)` instead of `res.shl(count * 8)`. This is structurally identical to the external report's pattern — a computation whose result is wrong because one operand is incorrectly included — and would produce a corrupted BLS12-381 scalar whenever the input byte length is not a multiple of 8.

### Finding Description

In `src/confidential_key_derivation/scalar_wrapper.rs`, the function `from_be_bytes_mod_order` converts an arbitrary byte slice into a `blstrs::Scalar` by processing 8 bytes at a time:

```rust
// Main loop — correct
res = res.shl(64) + blstrs::Scalar::from(remainder);   // line 41
```

After the main loop, any leftover bytes (`count > 0`) are handled at line 47:

```rust
// Remainder branch — BUGGY
res = res * res.shl(count * 8) + blstrs::Scalar::from(remainder);  // line 47
```

The intended operation is a left-shift of the accumulated value followed by adding the remaining bytes, i.e.:

```rust
res = res.shl(count * 8) + blstrs::Scalar::from(remainder);
```

Instead, `res` is **multiplied by its own shifted self** (`res * res.shl(count * 8)`), which is a quadratic expression in `res` rather than a linear shift. This produces a completely wrong scalar value whenever `count > 0`. [1](#0-0) 

The `from_be_bytes_mod_order` function is the sole implementation backing the `FromOkm` trait for `ScalarWrapper`:

```rust
fn from_okm(okm: &GenericArray<u8, Self::Length>) -> Self {
    Self::from_be_bytes_mod_order(okm)
}
``` [2](#0-1) 

`FromOkm` is the standard `hash_to_field` expansion interface used by `elliptic_curve::hash2curve`. In the CKD ciphersuite, this is the mechanism by which protocol inputs are hashed into BLS12-381 scalars for threshold secret derivation. [3](#0-2) 

### Impact Explanation

**Impact: High** — Corruption of CKD outputs.

If the buggy branch executes, the derived BLS12-381 scalar is wrong. Every honest participant would independently compute the same wrong scalar (since the bug is deterministic), so the protocol would not abort — it would silently produce an incorrect derived secret. The CKD output would be a deterministic but cryptographically unrelated value, meaning the derived key material is corrupted and unusable for its intended purpose, or — worse — it could be predictable if an attacker can characterize the quadratic residue structure of the error.

### Likelihood Explanation

**Likelihood: Low.**

The `FromOkm` implementation fixes `Length = U48` (48 bytes). Since 48 is exactly divisible by 8, the main loop consumes all bytes and `count` is 0 after the loop, so the `if count > 0` branch at line 46 is **never entered** in the current production call path. The bug is latent: it exists in the code and is exercised by the stress test at line 74 (which calls `from_be_bytes_mod_order` with arbitrary lengths), but the single production caller avoids it. If a future caller passes a non-multiple-of-8 byte slice, or if `Length` is changed, the bug activates immediately. [4](#0-3) [5](#0-4) 

### Recommendation

Replace the spurious multiplication with the correct shift:

```diff
 if count > 0 {
-    res = res * res.shl(count * 8) + blstrs::Scalar::from(remainder);
+    res = res.shl(count * 8) + blstrs::Scalar::from(remainder);
 }
```

This mirrors the pattern used in the main loop at line 41 and matches the reference implementation the comment cites.

### Proof of Concept

For any byte slice whose length is not a multiple of 8, `from_be_bytes_mod_order` returns a wrong scalar. Concretely, for a 1-byte input `[0x01]`:

- **Expected:** `res = 0.shl(8) + 1 = 1`
- **Actual:** `res = 0 * 0.shl(8) + 1 = 0 * 0 + 1 = 1` — coincidentally correct here because `res = 0` initially.

For a 9-byte input `[0x01, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00, 0x00, 0x02]`:

- After the main loop (8 bytes): `res = 0x0100000000000000`, `count = 1`, `remainder = 0x02`
- **Expected:** `res = 0x0100000000000000.shl(8) + 2 = 0x010000000000000002`
- **Actual:** `res = 0x0100000000000000 * 0x0100000000000000.shl(8) + 2` — a quadratic product, completely wrong.

Any future extension of `from_be_bytes_mod_order` to non-48-byte inputs (e.g., a different `FromOkm` length or a direct call from another CKD sub-protocol) would immediately produce corrupted scalars, silently corrupting all derived CKD secrets. [6](#0-5)

    fn from_be_bytes_mod_order(bytes: &[u8]) -> Self {
        let mut res = blstrs::Scalar::ZERO;

        let mut count = 0;
        let mut remainder = 0;
        for byte in bytes {
            remainder = (remainder << 8) + u64::from(*byte);
            count += 1;
            if count == 8 {
                res = res.shl(64) + blstrs::Scalar::from(remainder);
                remainder = 0;
                count = 0;
            }
        }
        if count > 0 {
            res = res * res.shl(count * 8) + blstrs::Scalar::from(remainder);
        }
        Self(res)
    }
}
