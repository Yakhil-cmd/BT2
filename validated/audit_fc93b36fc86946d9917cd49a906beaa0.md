Based on my research, I found a strong analog to the `liquidateFrom` public-visibility vulnerability in the Starknet Staking codebase.

The key pattern: `update_rewards` in `IStakingRewardsManager` is publicly callable with no caller restriction, while `update_rewards_from_attestation_contract` in `IStakingAttestation` is the intended entry point (called only after attestation validation). An attacker can call `update_rewards` directly with `disable_rewards: true`, updating `last_reward_block` without distributing rewards, causing the attestation contract's subsequent call to fail with `REWARDS_ALREADY_UPDATED`.

---

### Title
`update_rewards` Lacks Caller Restriction, Allowing Griefing of Staker Reward Distribution - (File: src/staking/staking.cairo)

### Summary
`IStakingRewardsManager.update_rewards` is embedded in the public ABI with no check that the caller is the attestation contract. An unprivileged attacker can call it directly with `disable_rewards: true` in the same block as a staker's attestation, consuming the per-block reward slot without distributing any rewards. The attestation contract's subsequent call to `update_rewards_from_attestation_contract` (which internally calls `update_rewards`) then reverts with `REWARDS_ALREADY_UPDATED`, permanently denying the staker their epoch reward.

### Finding Description

`IStakingAttestation.update_rewards_from_attestation_contract` is the intended, gated entry point for reward distribution. Its interface comment states it is called only after the attestation contract validates correct attestation: [1](#0-0) 

The inner workhorse, `update_rewards`, is separately exposed as a full public ABI entry point via `#[abi(embed_v0)]`: [2](#0-1) 

Its body performs only generic guards — pause check, block-number deduplication, staker existence, and non-zero balance — but **no check that the caller is the attestation contract**: [3](#0-2) 

The deduplication guard is the critical line: [4](#0-3) 

Once `last_reward_block` is written to the current block number, any subsequent call in the same block — including the legitimate call from the attestation contract — reverts with `REWARDS_ALREADY_UPDATED`. The early-return path when `disable_rewards: true` is passed means no rewards are ever calculated or sent: [5](#0-4) 

This is structurally identical to the `liquidateFrom` pattern: a public wrapper (`update_rewards_from_attestation_contract`) enforces the important precondition (valid attestation), but the inner function (`update_rewards`) is also public and can be called directly, bypassing that precondition entirely.

### Impact Explanation

An attacker who front-runs a staker's attestation transaction by calling `update_rewards(staker_address, disable_rewards: true)` causes:

1. `last_reward_block` is set to the current block with zero rewards distributed.
2. The attestation contract's call to `update_rewards_from_attestation_contract` reverts.
3. The staker receives **no rewards** for that epoch despite having attested correctly.

Repeated across every attestation epoch, this permanently freezes the targeted staker's unclaimed yield. This matches the **High** impact category: *Permanent freezing of unclaimed yield*.

### Likelihood Explanation

The attack requires only:
- Monitoring the mempool for attestation transactions (or predicting the attestation block from on-chain epoch data).
- Submitting a higher-gas call to `update_rewards` with `disable_rewards: true` in the same block.

No privileged role, leaked key, or external dependency is needed. Any unprivileged address can execute this. Starknet's sequencer model means front-running is feasible for a sequencer or any party with sequencer access.

### Recommendation

Add a caller check to `update_rewards` (or remove it from the public ABI and make it `pub(crate)` / internal), analogous to the fix recommended in the original report:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    // Add: assert caller is attestation contract
    assert!(
        get_caller_address() == self.attestation_contract_address.read(),
        "{}",
        GenericError::CALLER_IS_NOT_ATTESTATION_CONTRACT,
    );
    self.general_prerequisites();
    // ... rest of function
}
```

Alternatively, remove `update_rewards` from `IStakingRewardsManager` as a standalone public interface and only expose it through `update_rewards_from_attestation_contract`.

### Proof of Concept

1. Staker `S` submits an attestation transaction targeting block `B`.
2. Attacker observes the pending attestation and submits `update_rewards(S, disable_rewards: true)` with higher priority in block `B`.
3. Attacker's call executes first: `last_reward_block` is set to `B`, no rewards distributed.
4. Attestation contract's call to `update_rewards_from_attestation_contract(S)` executes next in block `B`, internally calls `update_rewards`, hits `assert!(current_block_number > self.last_reward_block.read())` → reverts with `REWARDS_ALREADY_UPDATED`.
5. Staker `S` receives zero rewards for epoch despite valid attestation.
6. Attacker repeats each epoch at negligible cost (only gas). [6](#0-5)

    #[abi(embed_v0)]
    impl StakingRewardsManagerImpl of IStakingRewardsManager<ContractState> {
        fn update_rewards(
            ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
        ) {
            self.general_prerequisites();
            let current_block_number = starknet::get_block_number();
            assert!(
                current_block_number > self.last_reward_block.read(),
                "{}",
                Error::REWARDS_ALREADY_UPDATED,
            );

            // Assert staker exists and active.
            // Staker is considered to exist from the moment of `stake` (when `InternalStakerInfo`
            // struct is created) until the calling to `unstake_action` (when `InternalStakerInfo`
            // struct is deleted).
            // Staker remains active until the intent period begins, i.e. K epochs after
            // `unstake_intent` is called.
            let staker_info = self.internal_staker_info(:staker_address);
            let curr_epoch = self.get_current_epoch();
            assert!(
                self.is_staker_active(:staker_address, epoch_id: curr_epoch),
                "{}",
                Error::INVALID_STAKER,
            );

            let staker_pool_info = self.staker_pool_info.entry(staker_address).as_non_mut();
            let (staker_total_strk_balance, staker_total_btc_balance) = self
                .get_staker_total_strk_btc_balance_at_epoch(
                    :staker_address, :staker_pool_info, epoch_id: curr_epoch,
                );
            // Assert staker has non-zero balance.
            // Staker exists with zero balance for the first K epochs after `stake`, then the stake
            // becomes effective.
            assert!(staker_total_strk_balance.is_non_zero(), "{}", Error::INVALID_STAKER);

            // Update last block rewards.
            self.last_reward_block.write(current_block_number);

            if disable_rewards || self.is_pre_consensus() {
                return;
            }

            // Get current block data and update rewards.
            let reward_supplier_dispatcher = self.reward_supplier_dispatcher.read();
            let (strk_block_rewards, btc_block_rewards) = self
                .calculate_block_rewards(:reward_supplier_dispatcher, :curr_epoch);
            self
                ._update_rewards(
                    :staker_address,
                    strk_total_rewards: strk_block_rewards,
                    btc_total_rewards: btc_block_rewards,
                    strk_total_stake: staker_total_strk_balance,
                    btc_total_stake: staker_total_btc_balance,
                    :staker_info,
                    :staker_pool_info,
                    :reward_supplier_dispatcher,
                    :curr_epoch,
                );
        }
