### Title
`uint32` Overflow in `_afterTimelock` Lets Pool Admin Bypass Stop-Loss Timelock and Instantly Disable LP Protection - (File: `metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

### Summary

`_afterTimelock` computes the execution deadline as `uint32(block.timestamp + timelock)`. When `timelock` is set to any value that causes `block.timestamp + timelock > type(uint32).max`, the cast silently wraps to a timestamp in the past. `_requireElapsed` then passes immediately, letting the pool admin execute drawdown/decay/watermark changes with zero delay — defeating the entire LP-protection guarantee of the extension.

### Finding Description

`_afterTimelock` is the single source of truth for every timelocked proposal in the extension:

```solidity
// OracleValueStopLossExtension.sol line 297-299
function _afterTimelock(address pool_) private view returns (uint32) {
    return uint32(block.timestamp + oracleStopLossConfig[pool_].timelock);
}
``` [1](#0-0) 

`block.timestamp` is `uint256`; `timelock` is `uint32`. The addition is performed in `uint256` space, then truncated to `uint32`. If the sum exceeds `type(uint32).max = 4,294,967,295`, the result wraps to a value **smaller than `block.timestamp`**.

The guard that enforces the delay:

```solidity
// line 301-303
function _requireElapsed(uint32 executeAfter) private view {
    if (block.timestamp < executeAfter) revert OracleStopLossTimelockNotElapsed(executeAfter, block.timestamp);
}
``` [2](#0-1) 

`executeAfter` (a `uint32`) is implicitly widened to `uint256` for the comparison. If it has wrapped to a past timestamp, `block.timestamp < executeAfter` is `false` and the revert never fires.

`initialize` accepts any `uint32` timelock with no upper-bound validation:

```solidity
// line 56-58
(uint32 drawdownE6, uint32 decayPerSecondE8, uint32 timelock) = abi.decode(data, (uint32, uint32, uint32));
_validateDrawdown(drawdownE6);
_validateDecay(decayPerSecondE8);
// timelock: no validation
``` [3](#0-2) 

`_afterTimelock` is called identically for every timelocked proposal — drawdown, decay, and high-watermarks: [4](#0-3) [5](#0-4) [6](#0-5) 

The `PoolStopLossSchedule` stores all pending deadlines as `uint32`: [7](#0-6) 

### Impact Explanation

The extension's own NatSpec states: *"Drawdown and decay changes are timelocked so LPs can react."* [8](#0-7) 

With the overflow, the pool admin can:

1. **Disable stop-loss entirely** — set `drawdownE6 = 0`; `_afterSwapOracleStopLoss` short-circuits at `if (drawdown == 0) return` and never checks watermarks again.
2. **Maximize decay** — set `decayPerSecondE8 = 1e8` (100 %/s); watermarks collapse to zero within one second, making the stop-loss permanently non-triggering.
3. **Zero out watermarks** — set per-bin watermarks to 0; the ratchet then sets them from the next live metric, erasing any accumulated protection.

All three actions can be proposed and executed in the same block, giving LPs zero time to react and withdraw.

### Likelihood Explanation

The pool admin controls `extensionInitData` at `createPool` time. Setting `timelock = type(uint32).max` requires no special privilege beyond being the pool admin. To an LP reading on-chain state, `type(uint32).max` seconds looks like a ~136-year timelock — the longest possible — when in fact it wraps to a timestamp one second in the past. The admin can also reach the overflow state post-deployment by first proposing a timelock change to any value `> type(uint32).max - block.timestamp` (currently ~80 years) and executing it after the current timelock elapses.

### Recommendation

1. **Validate timelock at initialization**: add `if (uint256(block.timestamp) + timelock > type(uint32).max) revert TimelockOverflow()` in `initialize`.
2. **Use `uint256` for deadline storage**: change `pendingTimelockExecuteAfter`, `pendingDrawdownExecuteAfter`, `pendingDecayExecuteAfter`, and `PendingHighWatermarks.executeAfter` to `uint256`, and change `_afterTimelock` to return `uint256`. This eliminates the truncation entirely.
3. **Cap the maximum timelock**: enforce a reasonable upper bound (e.g., 365 days) so the addition can never overflow even `uint256`.
