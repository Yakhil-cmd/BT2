### Title
Missing Sequencer-Only Access Control on `update_rewards` Allows Any Caller to Permanently Freeze Staker Yield - (File: `src/staking/staking.cairo`)

---

### Summary

The `update_rewards` function in `StakingRewardsManagerImpl` is specified to be callable only by the Starkware sequencer, but the implementation contains no caller check. Any unprivileged address can call it with `disable_rewards: true`, consuming the per-block reward slot and permanently denying yield for that block to all stakers and pool members.

---

### Finding Description

The specification at `docs/spec.md` line 1645 explicitly states:

> **access control**: Only starkware sequencer.

However, the implementation in `src/staking/staking.cairo` at lines 1449–1507 contains no such check:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();                          // only checks pause
    let current_block_number = starknet::get_block_number();
    assert!(
        current_block_number > self.last_reward_block.read(),
        "{}",
        Error::REWARDS_ALREADY_UPDATED,
    );
    // ... staker existence checks ...
    self.last_reward_block.write(current_block_number);   // consumes the block slot
    if disable_rewards || self.is_pre_consensus() {
        return;                                           // exits without distributing
    }
    // ... reward distribution ...
}
```

`last_reward_block` is a **global** storage variable shared across all stakers. [1](#0-0) 

Once `last_reward_block` is written to the current block number, the guard `current_block_number > self.last_reward_block.read()` will revert any subsequent call in the same block with `REWARDS_ALREADY_UPDATED`. [2](#0-1) 

The spec mandates this function is "Only starkware sequencer" but no `get_caller_address()` check exists anywhere in the function body. [3](#0-2) 

---

### Impact Explanation

An attacker calls `update_rewards(any_valid_staker_address, disable_rewards: true)` at the beginning of every block. This:

1. Writes `current_block_number` into `last_reward_block`.
2. Returns immediately without distributing any rewards (the `disable_rewards || self.is_pre_consensus()` branch).
3. Causes every subsequent legitimate sequencer call in that block to revert with `REWARDS_ALREADY_UPDATED`.

Stakers and pool members permanently lose the yield that would have been distributed for each blocked block. Because the block slot is consumed and can never be reclaimed, the yield loss is **permanent** — not deferred. This matches the allowed impact: **High: Permanent freezing of unclaimed yield**.

---

### Likelihood Explanation

- No special role, stake, or token balance is required.
- The attacker only needs to know any currently-active staker address (publicly readable from events or `staker_info_v1`).
- The call is cheap (no token transfer, no approval).
- The attacker can automate this every block to continuously suppress all reward distribution across the entire protocol.

---

### Recommendation

Add a caller check at the top of `update_rewards` that asserts `get_caller_address()` equals the registered sequencer address (stored in contract storage), mirroring the pattern used in `update_rewards_from_attestation_contract` which correctly asserts `CALLER_IS_NOT_ATTESTATION_CONTRACT`. [4](#0-3) 

---

### Proof of Concept

```cairo
// Attacker calls this every block before the sequencer does:
fn grief_rewards(staking_contract: ContractAddress, any_active_staker: ContractAddress) {
    // No role or stake required — any address can call this.
    let staking_rewards = IStakingRewardsManagerDispatcher {
        contract_address: staking_contract,
    };
    // Consumes the block's reward slot; disable_rewards=true skips distribution.
    staking_rewards.update_rewards(
        staker_address: any_active_staker,
        disable_rewards: true,
    );
    // Sequencer's subsequent update_rewards call now reverts with REWARDS_ALREADY_UPDATED.
    // Stakers and pool members receive zero yield for this block — permanently.
}
```

The `last_reward_block` guard at line 1454 ensures the sequencer cannot retry in the same block. [5](#0-4)
