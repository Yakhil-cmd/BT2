### Title
Pool Admin Bypasses `maxAdminSpreadFeeE6` Cap via Uncapped `setPoolBinAdditionalFees` - (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary

`MetricOmmPoolFactory.setPoolBinAdditionalFees` forwards `addFeeBuyE6` and `addFeeSellE6` directly to the pool with no cap validation, while the parallel path `setPoolAdminFees` enforces `maxAdminSpreadFeeE6`. A pool admin can set per-bin additional fees up to `uint16.max` (65,535 E6 = 6.5535%) on any bin, bypassing the factory's fee-cap system entirely and extracting fees from swappers beyond the protocol-enforced ceiling.

### Finding Description

The factory maintains four fee caps enforced by `setFeeCaps` and checked on every admin fee mutation:

```solidity
// MetricOmmPoolFactory.sol L414-415
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (newAdminNotionalFeeE8 > maxAdminNotionalFeeE8) revert AdminFeeTooHigh();
``` [1](#0-0) 

However, the sibling function `setPoolBinAdditionalFees` performs no such check:

```solidity
// MetricOmmPoolFactory.sol L450-457
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
``` [2](#0-1) 

These per-bin fees are added directly on top of the oracle-derived base spread in every swap iteration:

```solidity
// MetricOmmPool.sol L999
params.baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6),
``` [3](#0-2) 

```solidity
// MetricOmmPool.sol L1088
params.baseFeeX64 + Math.mulDiv(binState.addFeeSellE6, ONE_X64, 1e6),
``` [4](#0-3) 

The `addFeeBuyE6`/`addFeeSellE6` fields are `uint16`, so the unchecked maximum is 65,535 (6.5535% in E6 units). This is additive to the oracle spread and directly increases the token cost paid by swappers, independent of and unconstrained by `maxAdminSpreadFeeE6`. [5](#0-4) 

The hard cap constant is:

```solidity
uint24 internal constant HARD_MAX_SPREAD_FEE_E6 = 200_000; // 20%
``` [6](#0-5) 

The factory owner can lower `maxAdminSpreadFeeE6` to 0 to prevent the admin from charging any spread fee via `setPoolAdminFees`, yet the admin retains the ability to set `addFeeBuyE6 = 65535` on every bin through `setPoolBinAdditionalFees`, completely defeating the cap.

### Impact Explanation

Swappers pay an effective per-bin fee of `baseFee + addFee`, where `addFee` is uncapped at up to 6.5535%. This is a direct loss of user principal on every swap routed through an affected bin. The fee cap system — the primary admin-boundary control protecting users from a rogue or compromised pool admin — is rendered ineffective for the bin-additional-fee dimension. The factory owner's ability to lower `maxAdminSpreadFeeE6` is also silently bypassed.

### Likelihood Explanation

The pool admin is a semi-trusted role that is explicitly expected to operate only within caps. The call path is a single transaction by the pool admin with no timelock. Any pool admin (including a compromised multisig key) can exploit this immediately after pool deployment. The Smart Audit Pivots explicitly flag `setBinAdditionalFees` as a bypass surface.

### Recommendation

Add a cap check in `setPoolBinAdditionalFees` before forwarding to the pool. The simplest fix is to bound the sum of the bin additional fee and the current admin spread fee against `maxAdminSpreadFeeE6`, or at minimum enforce `addFeeBuyE6 <= maxAdminSpreadFeeE6` and `addFeeSellE6 <= maxAdminSpreadFeeE6`:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (addFeeBuyE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

Optionally, also validate the initial bin `addFeeBuyE6`/`addFeeSellE6` values in `_unpackAndValidateBinStates` at pool creation time. [7](#0-6) 

### Proof of Concept

1. Factory owner deploys factory with `maxAdminSpreadFeeE6 = 0` (or later calls `setFeeCaps(0, 0, ...)`) to enforce zero admin spread fee.
2. Pool admin calls `setPoolAdminFees(pool, 1, 0)` → reverts `AdminFeeTooHigh`. Cap is enforced.
3. Pool admin calls `setPoolBinAdditionalFees(pool, 0, 65535, 65535)` → succeeds. No cap check.
4. A swapper calls `swap(...)` routing through bin 0. The effective fee applied is `baseFeeX64 + mulDiv(65535, ONE_X64, 1e6)` — an additional 6.5535% on top of the oracle spread — despite the factory owner having set the admin spread cap to 0.
5. The swapper pays 6.5535% more than the oracle spread alone would require, with the excess accruing as LP fees to the pool (benefiting the admin-controlled LP positions). [8](#0-7)
