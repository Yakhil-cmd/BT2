### Summary

The `formatAmountWithRoyalties` function in the WalletConnect confirmation pipeline incorrectly divides the payment amount equally among all NFTs before computing royalties. When an offer contains N NFTs with royalties, the displayed "Total Amount with Royalties" is understated by approximately a factor of N relative to what the blockchain actually charges. A user approving a `chia_wallet.take_offer` via WalletConnect sees a materially lower total cost than they will actually spend.

---

### Finding Description

In `parseCommandDisplay.ts`, `formatAmountWithRoyalties` computes the royalty display total as:

```
splitAmount = amount / N          // integer division, truncates
royaltyAmount = ÎŁ (splitAmount Ă— royaltyPercentage_i / 10_000)
``` [1](#0-0) 

The correct formula â€” matching what the Chia wallet backend actually charges â€” is:

```
royaltyAmount = ÎŁ (amount Ă— royaltyPercentage_i / 10_000)
```

Because `splitAmount = amount / N`, the displayed royalty is `1/N` of the real royalty for each NFT. The existing test fixture encodes this bug as the expected value:

- `amount = 100,000,000` mojos (0.0001 XCH), two NFTs with royalties 500 (5%) and 10 (0.1%)
- **Displayed** `amountWithRoyalties`: `0.00010255` XCH (royalty = 2,550,000 mojos)
- **Actual** charge: `0.0001051` XCH (royalty = 5,100,000 mojos â€” each NFT royalty is computed on the full amount) [2](#0-1) 

The understated total is rendered in the WalletConnect confirmation dialog under "Total Amount with Royalties": [3](#0-2) 

The `formatAmountWithRoyalties` result flows through `withRoyaltyTotals` â†’ `walletDeltaToDisplay` â†’ the `Confirm` component, which is the sole source of truth for the user's approval decision in the WalletConnect flow. [4](#0-3)

### Impact Explanation

A user approving a `chia_wallet.take_offer` via WalletConnect is shown a "Total Amount with Royalties" that is materially lower than what the blockchain will actually deduct. The discrepancy scales with the number of NFTs: for N NFTs, the displayed royalty is `1/N` of the real royalty. For a 2-NFT offer with 5% + 0.1% royalties, the user is shown a 2.55% total royalty cost but actually pays 5.1%. The user approves based on the understated figure; the wallet backend charges the correct (higher) amount.

This matches the allowed High impact: *"WalletConnect state that causes a user to approve â€¦ the wrong â€¦ amount."*

---

### Likelihood Explanation

Any WalletConnect-connected dapp can call `chia_wallet.take_offer` with an offer blob containing multiple NFTs with royalties. The dapp does not need to forge royalty percentages â€” those are on-chain in the NFT. The dapp only needs to bundle two or more royalty-bearing NFTs into a single offer. No special privileges are required; the user's WalletConnect session approval is the only prerequisite.

---

### Recommendation

Replace the split-amount logic with per-NFT royalty computation on the full amount:

```typescript
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (amount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
```

Remove the `splitAmount` variable entirely. Update the test fixture to assert the corrected value (`0.0001051` XCH for the two-NFT case).

### Proof of Concept

```

offered:   { NFT_A: "1", NFT_B: "1" }
NFT_A royalty_percentage: 500     // 5%
NFT_B royalty_percentage: 10      // 0.1%
```

**Current display** (buggy):
- `splitAmount = 100000000 / 2 = 50000000`

- Shown to user: **0.00010255 XCH**

**Actual blockchain charge** (correct):
- royalty_A = `100000000 Ă— 500 / 10000 = 5,000,000`
- royalty_B = `100000000 Ă— 10 / 10000 = 100,000`
- Actually spent: **0.0001051 XCH**

The user approves believing they spend `0.00010255 XCH`; the wallet deducts `0.0001051 XCH` â€” a 2.5Ă— understatement of the royalty cost. The gap grows linearly with the number of NFTs in the offer.
