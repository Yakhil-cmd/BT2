### Title
Missing `me` Membership Validation in `generate_triple` / `generate_triple_many` Allows Silent Protocol Corruption - (File: `src/ecdsa/ot_based_ecdsa/triples/generation.rs`)

---

### Summary

`generate_triple` and `generate_triple_many` — the public entry points for OT-based Beaver triple generation — delegate all input validation to `validate_triple_inputs`, which checks participant count, threshold bounds, and duplicates, but **never checks that `me` is a member of the participant list**. Every other protocol initialization function in the library performs this check. The missing guard allows the protocol to be initialized and run with an invalid local identity, producing a corrupted or non-terminating triple generation session.

---

### Finding Description

Every other protocol initialization function in the codebase explicitly verifies that `me` is present in the participant list before proceeding:

- `ot_based_ecdsa::presign` — [1](#0-0) 
- `ot_based_ecdsa::sign` — [2](#0-1) 
- `robust_ecdsa::presign` — [3](#0-2) 
- `robust_ecdsa::sign` — [4](#0-3) 
- `ckd` — [5](#0-4) 
- `frost::assert_sign_inputs` — [6](#0-5) 

However, `generate_triple` and `generate_triple_many` only call `validate_triple_inputs`, which performs no such check:

```rust
fn validate_triple_inputs(
    participants: &[Participant],
    threshold: impl Into<ReconstructionLowerBound>,
) -> Result<(ParticipantList, ReconstructionLowerBound), InitializationError> {
    // checks: len < 2, threshold bounds, duplicates
    // NO check: participants.contains(me)
    ...
}
``` [7](#0-6) 

Both public functions then pass the unvalidated `me` directly into the async protocol future:

```rust
pub fn generate_triple(...) -> Result<...> {
    let (participants, threshold) = validate_triple_inputs(participants, threshold)?;
    let ctx = Comms::new();
    let fut = do_generation(ctx.clone(), participants, me, threshold, rng);
    Ok(make_protocol(ctx, fut))
}
``` [8](#0-7) [9](#0-8) 

Inside `do_generation_many`, `me` is used in two critical ways:

1. **`recv_from_others(&chan, ..., &participants, me)`** — filters `me` out of the participant list to determine who to receive from. If `me` is not in the list, no participant is filtered, so the party waits for messages from all `N` participants (including the one whose identity it should hold), causing a protocol hang.
2. **`big_l.eval_at_participant(me)`** — evaluates the committed polynomial at `me`'s scalar. If `me` is outside the participant set, this produces a share at a non-participant evaluation point, yielding a cryptographically invalid `TripleShare`. [10](#0-9) 

---

### Impact Explanation

Beaver triples are the foundational prerequisite for OT-based ECDSA presigning. A corrupted or non-terminating triple generation session directly prevents presigning and signing from completing. Specifically:

- If `me` is not in the participant list, `recv_from_others` waits for all `N` participants (including the one whose identity `me` should hold). That participant's instance will not send to the caller (since the caller is not in the list), causing a **permanent hang** in triple generation.
- If `me` is set to a participant identity that is in the list but does not correspond to the caller's actual role, the resulting `TripleShare` is evaluated at the wrong polynomial point, producing **cryptographically inconsistent output** that will cause presigning to fail or produce an invalid presignature.

This matches: **High — Corruption of presign outputs so honest parties accept inconsistent transcripts or unusable cryptographic outputs**, and **High — Permanent denial of signing for honest parties under valid protocol inputs**.

---

### Likelihood Explanation

The entry point is the public API (`generate_triple`, `generate_triple_many`). Any library caller — including a misconfigured honest party or a malicious participant controlling their own node — can trigger this by passing a `me` value not present in `participants`. The function returns `Ok(...)` with no error, so the caller has no indication that the protocol is misconfigured until it hangs or produces a bad triple. All peer protocol functions enforce this check, making the omission here a clear inconsistency that is easy to trigger accidentally or deliberately.

---

### Recommendation

Add the `me` membership check inside `validate_triple_inputs` (or directly in `generate_triple` / `generate_triple_many`) to match the pattern used by every other protocol initialization function:

```rust
fn validate_triple_inputs(
    participants: &[Participant],
    me: Participant,
    threshold: impl Into<ReconstructionLowerBound>,
) -> Result<(ParticipantList, ReconstructionLowerBound), InitializationError> {
    // ... existing checks ...
    let participants =
        ParticipantList::new(participants).ok_or(InitializationError::DuplicateParticipants)?;

+   if !participants.contains(me) {
+       return Err(InitializationError::MissingParticipant {
+           role: "self",
+           participant: me,
+       });
+   }

    Ok((participants, threshold))
}
``` [7](#0-6)

```rust
fn validate_triple_inputs(
    participants: &[Participant],
    threshold: impl Into<ReconstructionLowerBound>,
) -> Result<(ParticipantList, ReconstructionLowerBound), InitializationError> {
    let threshold = threshold.into();
    let threshold_value = threshold.value();
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }
    // Spec 1.1
    if threshold_value > participants.len() {
        return Err(InitializationError::ThresholdTooLarge {
            threshold: threshold_value,
            max: participants.len(),
        });
    }
    if threshold_value < 2 {
        return Err(InitializationError::ThresholdTooSmall {
            threshold: threshold_value,
            min: 2,
        });
    }
    let participants =
        ParticipantList::new(participants).ok_or(InitializationError::DuplicateParticipants)?;
    Ok((participants, threshold))
}
