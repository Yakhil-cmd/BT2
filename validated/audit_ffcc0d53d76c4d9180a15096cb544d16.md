### Title
Unguarded `deploy_erc20_token` Allows Any Caller to Register Arbitrary NEP-141 Accounts, Enabling Unbacked ERC-20 Minting - (File: `engine/src/contract_methods/connector.rs`)

---

### Summary

The public NEAR entrypoint `deploy_erc20_token` in the Aurora Engine performs no caller access control. Any NEAR account can call it with an arbitrary NEP-141 account ID. Because `ft_on_transfer` uses `predecessor_account_id` as the authoritative token identifier and mints ERC-20 tokens based solely on the caller-supplied `amount` field, an attacker who registers a malicious NEP-141 contract can mint an unlimited number of unbacked ERC-20 tokens on Aurora.

---

### Impact Explanation

**Critical — Direct theft of user funds / insolvency.**

An attacker mints an arbitrary quantity of ERC-20 tokens on Aurora that are not backed by any real NEP-141 deposit. These tokens are indistinguishable from legitimately bridged tokens within the Aurora EVM. The attacker can swap them against real assets in any Aurora DeFi protocol (AMMs, lending markets), draining real user funds and causing insolvency of the bridge accounting.

---

### Likelihood Explanation

**High.** The entry point is a standard public NEAR function callable by any NEAR account with no deposit or special key requirement. Deploying a malicious NEP-141 contract on NEAR is trivial and cheap. The two-step attack (register + call `ft_on_transfer`) requires no privileged access and can be executed permissionlessly on mainnet.

---

### Recommendation

Add `require_owner_only` (or a dedicated allowlist) to `deploy_erc20_token`, mirroring the pattern already used in `mirror_erc20_token`:

```rust
pub fn deploy_erc20_token<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I, env: &E, handler: &mut H,
) -> Result<PromiseOrValue<Address>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
+       require_owner_only(&state, &env.predecessor_account_id())?;
        ...
    })
}
```

Alternatively, maintain an explicit allowlist of trusted NEP-141 bridge contracts that are permitted to be registered.

---

### Proof of Concept

1. Attacker deploys a malicious NEAR contract `evil.near` that implements the NEP-141 interface but does not enforce real token balances.

2. Attacker calls `deploy_erc20_token` on the Aurora contract with `DeployErc20TokenArgs::Legacy("evil.near")`. This succeeds because there is no access control — only `require_running` is checked. [6](#0-5) 

3. Aurora deploys a new `EvmErc20` contract and registers the bidirectional mapping `evil.near ↔ erc20_address` via `register_token`. [7](#0-6) 

4. `evil.near` calls `ft_on_transfer` on Aurora directly (a public NEAR function), passing `amount = 1_000_000_000_000` and `msg = <attacker_evm_address_hex>`. The `predecessor_account_id` is `evil.near`. [8](#0-7) 

5. `ft_on_transfer` routes to `receive_erc20_tokens("evil.near", ...)`, finds the registered ERC-20, and calls `mint(attacker_evm_address, 1_000_000_000_000)` on it. [9](#0-8) 

6. Attacker now holds 1 trillion unbacked ERC-20 tokens on Aurora and can swap them for real bridged assets in any Aurora DeFi protocol.

    pub fn receive_erc20_tokens<P: PromiseHandler>(
        &mut self,
        token: &AccountId,
        args: &FtOnTransferArgs,
        current_account_id: &AccountId,
        handler: &mut P,
    ) -> Result<Option<SubmitResult>, ContractError> {
        let amount = args.amount.as_u128();
        // Parse message to determine recipient
        let mut recipient = {
            // The message should contain the recipient EOA address.
            let message = args.msg.strip_prefix("0x").unwrap_or(&args.msg);
            // Recipient - 40 characters (Address in hex without '0x' prefix)
            if message.len() < 40 {
                return Err(ParseOnTransferMessageError::WrongMessageFormat.into());
            }
            let mut address_bytes = [0; 20];
            hex::decode_to_slice(&message[..40], &mut address_bytes)
                .map_err(|_| ParseOnTransferMessageError::WrongMessageFormat)?;
            Address::from_array(address_bytes)
        };

        if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
            && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
        {
            recipient = fallback_address;
        }

        let erc20_token = get_erc20_from_nep141(&self.io, token)?;
        let erc20_admin_address = current_address(current_account_id);
        let result = self
            .call(
                &erc20_admin_address,
                &erc20_token,
                Wei::zero(),
                setup_receive_erc20_tokens_input(&recipient, amount),
                u64::MAX,
                Vec::new(), // TODO: are there values we should put here?
                Vec::new(),
                handler,
            )
            .and_then(submit_result_or_err)?;
