### Title
Hardcoded Zero Echo-Broadcast Thresholds for n≤3 Enable Equivocation During DKG — (File: src/protocol/echo_broadcast.rs)

---

### Summary

`echo_ready_thresholds` in `src/protocol/echo_broadcast.rs` returns hardcoded `(0, 0)` for any participant count `n ≤ 3`. This eliminates all Byzantine fault-tolerance guarantees from the echo-broadcast primitive for small participant sets. Because the DKG (`src/dkg.rs`) uses `do_broadcast` — which internally calls `reliable_broadcast_receive_all` — to distribute polynomial commitments, a single malicious participant in a 3-party DKG can equivocate: sending different commitments to each honest party. With zero echo/ready thresholds, each honest party immediately delivers the value it received without waiting for corroborating echoes from peers, so the two honest parties finish DKG holding different public keys.

---

### Finding Description

`echo_ready_thresholds` contains a hardcoded early-return for small groups:

```rust
fn echo_ready_thresholds(n: usize) -> (usize, usize) {
    if n <= 3 {
        return (0, 0);   // ← hardcoded assumption: no malicious parties
    }
    let broadcast_threshold = (n - 1) / 3;
    let echo_threshold = usize::midpoint(n, broadcast_threshold);
    (echo_threshold, broadcast_threshold)
}
``` [1](#0-0) 

The comment documents the intent: "no malicious parties are assumed" for n ≤ 3. However, the library's public API imposes no such restriction — `do_keyshare` / `reshare` accept as few as 2 participants, and a caller-supplied malicious participant is indistinguishable from an honest one at the API boundary. [2](#0-1) 

Inside `reliable_broadcast_receive_all`, the echo and ready phase transitions are gated on `count > echo_t` and `count > ready_t` respectively: [3](#0-2) [4](#0-3) 

When both thresholds are 0, the condition `count > 0` is satisfied by the **simulated self-echo** that is injected immediately after the Send phase (lines 207–210). This means each honest party transitions through all phases based solely on its own simulated message, before any network echo from peers is received. The `finish_echo = true` flag is set at that point, causing all subsequent echoes from other parties to be silently dropped (`continue`). [5](#0-4) 

The DKG imports and calls `do_broadcast` to distribute polynomial commitments to all participants: [6](#0-5) 

---

### Impact Explanation

**High — Corruption of DKG outputs so honest parties accept inconsistent public keys.**

Attack scenario with n=3 (parties B, C honest; M malicious):

1. M sends `commitment_1` (Send message) to B and `commitment_2` to C.
2. B's echo-broadcast instance immediately self-echoes `commitment_1`, satisfies `count > 0`, transitions to Ready, and delivers `commitment_1` — all before receiving C's echo.
3. C does the same with `commitment_2`.
4. M sends secret shares to B and C that are each individually consistent with the commitment the respective party holds.
5. Both B and C pass share-verification against their respective (different) commitments.
6. The final public key is the sum of all participants' first commitment coefficients. B computes `PK_B = f(commitment_1, ...)` and C computes `PK_C = f(commitment_2, ...)`. These are different group elements.

Honest parties B and C now hold incompatible key material: any signature produced by B's view of the key will fail verification under C's public key, and vice versa. This permanently corrupts the DKG output for the session.

---

### Likelihood Explanation

**Medium.** The attack requires only that a single participant in a 3-party (or 2-party) DKG behaves maliciously. The library explicitly supports n=2 and n=3 participant sets with no API-level warning that Byzantine security is absent. A malicious coordinator or participant who controls one node in a small deployment can trigger this deterministically with no cryptographic assumptions.

---

### Recommendation

1. **Remove the hardcoded `n ≤ 3` early-return.** Apply the general formula for all n ≥ 2. For n=2 the formula yields `broadcast_threshold = 0`, `echo_threshold = 1`, which still requires at least one peer echo before delivery — sufficient for 2-party honest operation and safe against equivocation.
2. **Alternatively**, enforce a minimum participant count of 4 at the DKG initialization layer if the protocol's security model genuinely requires n ≥ 3f+1 with f ≥ 1.
3. **Document** in the public API that n ≤ 3 currently provides no Byzantine fault tolerance, so callers can make an informed decision.

---

### Proof of Concept

```

echo_ready_thresholds(3) → (0, 0)   ← hardcoded path

Step 1: M sends Send(commit_1) to B, Send(commit_2) to C via network.

Step 2 (B's view of M's broadcast session):
  - B receives Send(commit_1) from M
  - B sends Echo(commit_1) to all, simulates own echo immediately
  - data_echo[commit_1] = 1; 1 > echo_t(=0) → TRUE
  - B sends Ready(commit_1), finish_echo = true
  - All subsequent echoes (including C's Echo(commit_2)) are dropped
  - data_ready[commit_1] = 1; 1 > ready_t(=0) → amplify
  - data_ready[commit_1] = 2; 2 > 2*ready_t(=0) → DELIVER commit_1

Step 3 (C's view of M's broadcast session):
  - Symmetric: C delivers commit_2

Step 4: M sends share_B consistent with commit_1 to B → verification passes
        M sends share_C consistent with commit_2 to C → verification passes

Step 5: B computes public_key using commit_1; C computes public_key using commit_2
        → inconsistent public keys accepted by both honest parties

fn echo_ready_thresholds(n: usize) -> (usize, usize) {
    // case where no malicious parties are assumed: when n <= 3/
    // In this case the echo and ready thresholds are both 0
    // later we compare if we have collected more votes than these thresholds
    if n <= 3 {
        return (0, 0);
    }
    // we should always have n >= 3*threshold + 1
    let broadcast_threshold = (n - 1) / 3;
    let echo_threshold = usize::midpoint(n, broadcast_threshold);
    (echo_threshold, broadcast_threshold)
}
