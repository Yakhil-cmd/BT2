### Title
`estimate_fee` hardcodes `charge_fee = false`, bypassing balance and resource-bound pre-validation, returning an authoritative-looking fee estimate for transactions that would fail in actual execution — (File: crates/apollo_rpc_execution/src/lib.rs)

---

### Summary

The `estimate_fee` function in `crates/apollo_rpc_execution/src/lib.rs` unconditionally passes `charge_fee = false` to `execute_transactions`. This causes `perform_pre_validation_stage` in the blockifier to skip both `check_fee_bounds` and `verify_can_pay_committed_bounds`. As a result, `starknet_estimateFee` returns a successful fee estimate for transactions that would fail at pre-validation in actual execution — for example, because the sender has zero balance or because `max_fee`/resource bounds are below the protocol minimum. This is the direct Sequencer analog of `BunniQuoter` not calling hooklet functions: the simulation path omits a class of checks that actual execution enforces, producing an authoritative-looking but incorrect result.

---

### Finding Description

**Root cause — hardcoded `charge_fee = false` in `estimate_fee`:** [1](#0-0) 

The `false` literal at line 646 is the only place `charge_fee` is set for the fee-estimation path. There is no caller-controlled flag; `simulate_transactions` (line 990–1008) correctly threads `charge_fee` through, but `estimate_fee` does not.

**Guard that is skipped — `perform_pre_validation_stage`:** [2](#0-1) 

When `charge_fee = false`, the entire `if self.execution_flags.charge_fee { … }` block is bypassed. This block contains:
- `check_fee_bounds` — verifies `max_fee` / resource bounds are at or above the protocol minimum.
- `verify_can_pay_committed_bounds` — verifies the sender's on-chain balance covers the committed maximum fee.

**RPC entry point that exposes this to any caller:** [3](#0-2) 

Any unprivileged user can call `starknet_estimateFee`. The `validate` flag is user-controlled via `SimulationFlag::SkipValidate`, but `charge_fee` is never exposed — it is always `false` inside `exec_estimate_fee`.

**Contrast with `simulate_transactions`, which correctly threads `charge_fee`:** [4](#0-3) 

`simulate_transactions` derives `charge_fee` from the caller-supplied `SimulationFlag::SkipFeeCharge`. `estimate_fee` has no equivalent path.

**Contrast with gateway stateful validation, which enforces the balance check:** [5](#0-4) 

The gateway calls `enforce_fee(executable_tx, only_query)` to set `charge_fee`, so `verify_can_pay_committed_bounds` runs for real submissions. The RPC simulation path never reaches this code.

---

### Impact Explanation

The Starknet RPC spec for `starknet_estimateFee` states: *"If one of the transactions reverts or fails due to any reason (e.g. validation failure or an internal error), a TRANSACTION_EXECUTION_ERROR is returned."* A transaction whose sender has zero balance, or whose `max_fee` is below the protocol minimum, **will** fail at pre-validation when actually submitted. `estimate_fee` does not surface this failure — it returns a non-zero fee estimate, implying the transaction would succeed. This is an authoritative-looking wrong value from the RPC layer.

Concrete wrong-value scenario:
- Sender balance = 0 STRK; `max_fee` = 1 (below minimum).
- `starknet_estimateFee` → returns `{ overall_fee: X, … }` (success).
- `starknet_addInvokeTransaction` → gateway stateful validation calls `verify_can_pay_committed_bounds` → fails with `GasBoundsExceedBalance` / `MaxFeeExceedsBalance`.

The estimate is wrong in the sense that it asserts executability for a transaction the protocol will reject.

---

### Likelihood Explanation

Any caller can trigger this with a zero-balance account or a transaction whose `max_fee` is set to 1. No special privilege is required. The discrepancy is permanent (hardcoded literal), not configuration-dependent.

---

### Recommendation

Pass `charge_fee = true` (or derive it from the transaction's resource bounds, as `simulate_transactions` does) inside `estimate_fee`, so that `check_fee_bounds` and `verify_can_pay_committed_bounds` run during fee estimation. If the intent is to allow estimation without a balance check, the function should at minimum return a `TRANSACTION_EXECUTION_ERROR` when the pre-validation checks that *would* run with `charge_fee = true` would fail, matching the spec's stated behavior.

---

### Proof of Concept

1. Deploy an account contract at address `A` with zero STRK balance.
2. Call `starknet_estimateFee` with an `INVOKE` V3 transaction from `A`, `max_fee = 1`, valid calldata.
3. Observe: the RPC returns `{ overall_fee: <non-zero>, … }` — no error.
4. Call `starknet_addInvokeTransaction` with the same parameters.
5. Observe: the gateway's stateful validator calls `verify_can_pay_committed_bounds` → returns `GasBoundsExceedBalance` / `MaxFeeExceedsBalance` error.

The fee estimate in step 3 is an authoritative-looking wrong value: it implies the transaction is executable, but step 5 proves it is not.

    pub fn perform_pre_validation_stage<S: State + StateReader>(
        &self,
        state: &mut S,
        tx_context: &TransactionContext,
    ) -> TransactionPreValidationResult<()> {
        let tx_info = &tx_context.tx_info;
        Self::handle_nonce(state, tx_info, self.execution_flags.strict_nonce_check)?;

        if self.execution_flags.charge_fee {
            self.check_fee_bounds(tx_context)?;

            verify_can_pay_committed_bounds(state, tx_context).map_err(Box::new)?;
        }

        self.validate_proof_facts(&tx_context.block_context, state)?;

        Ok(())
    }

            transactions.into_iter().map(|tx| tx.try_into()).collect::<Result<_, _>>()?;

        let block_number = get_accepted_block_number(&storage_txn, block_id)?;
        let block_not_reverted_validator =
            BlockNotRevertedValidator::new(block_number, &storage_txn)?;
        drop(storage_txn);
        let state_number = StateNumber::unchecked_right_after_block(block_number);
        let execution_config = self.execution_config;

                    TransactionExecutionError {
                        transaction_index: reverted_tx.index,
                        execution_error: reverted_tx.revert_reason,
                    },
                )))
            }
            Err(err) => Err(internal_server_error(err)),
        }
    }
