### Title
Pool Admin Bypasses `maxAdminSpreadFeeE6` Cap via Uncapped `setPoolBinAdditionalFees` — (File: metric-core/contracts/MetricOmmPoolFactory.sol)

---

### Summary

`MetricOmmPoolFactory.setPoolAdminFees` enforces `maxAdminSpreadFeeE6` on the admin spread component, but `setPoolBinAdditionalFees` — also callable by the pool admin — forwards per-bin additional fees to the pool with **no cap check at all**. Because the swap engine adds `addFeeBuyE6` / `addFeeSellE6` directly on top of the base spread fee, the pool admin can charge traders more than the protocol-enforced ceiling allows.

---

### Finding Description

The factory maintains a configurable cap `maxAdminSpreadFeeE6` (hard-limited to `HARD_MAX_SPREAD_FEE_E6 = 200_000`, i.e. 20%) that the factory owner sets via `setFeeCaps`. Every path that writes the admin spread component respects this cap:

- `setPoolAdminFees` (line 414–415): reverts `AdminFeeTooHigh` if `newAdminSpreadFeeE6 > maxAdminSpreadFeeE6`
- `_validatePoolParameters` (line 557–558): same check at pool creation [1](#0-0) 

However, `setPoolBinAdditionalFees` — also gated only by `onlyPoolAdmin` — passes `addFeeBuyE6` and `addFeeSellE6` straight through to the pool with **zero cap enforcement**: [2](#0-1) 

On the pool side, `setBinAdditionalFees` only validates the bin index range, not the fee magnitude: [3](#0-2) 

During every swap, the per-bin additional fee is added **on top of** `baseFeeX64` (which already embeds the oracle spread plus `adminSpreadFeeE6`): [4](#0-3) [5](#0-4) [6](#0-5) 

`addFeeBuyE6` and `addFeeSellE6` are `uint16`, so the pool admin can set them to at most 65,535 (≈ 6.55% in E6 units) per bin. Because the factory owner cannot cap this value — there is no `maxAdminBinFeeE6` storage variable and no check in `setPoolBinAdditionalFees` — the pool admin can silently push the effective swap fee above `maxAdminSpreadFeeE6` on every bin.

The resulting surplus is collected via `collectFees` and split between admin and protocol proportionally to their stored spread components, so the admin directly profits from the excess. [7](#0-6) 

---

### Impact Explanation

Traders pay a higher effective spread fee than the protocol-enforced cap permits. With `maxAdminSpreadFeeE6 = 0` (factory owner fully suppresses admin spread) the pool admin can still impose up to 6.55% additional fee per bin via `setPoolBinAdditionalFees`. The excess accrues as pool surplus and is partially swept to the admin on the next `collectFees` call. This is a direct loss of trader principal above the cap the protocol guarantees.

---

### Likelihood Explanation

The pool admin is semi-trusted only within caps and timelocks. The bypass requires no special timing, no external oracle manipulation, and no privileged factory-owner cooperation — the pool admin calls `setPoolBinAdditionalFees` unilaterally at any time. The factory owner has no mechanism to prevent or detect it before trades execute.

---

### Recommendation

Add a cap check inside `setPoolBinAdditionalFees` before forwarding to the pool, analogous to the check in `setPoolAdminFees`:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    // NEW: enforce a per-bin additional fee ceiling
    if (addFeeBuyE6 > maxAdminBinFeeE6) revert AdminFeeTooHigh();
    if (addFeeSellE6 > maxAdminBinFeeE6) revert AdminFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

Introduce a `maxAdminBinFeeE6` storage variable (settable by the factory owner via `setFeeCaps`) and apply the same hard-limit guard used for `maxAdminSpreadFeeE6`. Alternatively, enforce that `adminSpreadFeeE6 + addFeeBuyE6 ≤ maxAdminSpreadFeeE6` to keep the total admin-controlled spread within the cap.

---

### Proof of Concept

1. Factory owner sets `maxAdminSpreadFeeE6 = 0` (zero admin spread allowed).
2. Pool admin calls `setPoolAdminFees(pool, 0, 0)` — passes the cap check.
3. Pool admin calls `setPoolBinAdditionalFees(pool, 0, 65535, 65535)` — **no revert**; bin 0 now carries ≈ 6.55% additional buy and sell fee.
4. A trader swaps through bin 0. The effective fee is `baseFeeX64 + mulDiv(65535, ONE_X64, 1e6)`, i.e. oracle spread + 6.55%, despite the factory owner having set the admin spread cap to 0.
5. On the next `collectFees` call the surplus (including the 6.55% excess) is split between admin and protocol, with the admin receiving its proportional share. [2](#0-1) [8](#0-7)
