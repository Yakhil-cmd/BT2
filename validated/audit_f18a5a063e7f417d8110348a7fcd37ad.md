### Title
`protocolUnpausePool` Always Transitions to Admin-Paused (Level 1) Instead of Active (Level 0), Permanently Blocking Swaps After Protocol Unpause - (File: metric-core/contracts/MetricOmmPoolFactory.sol)

---

### Summary

`protocolUnpausePool` hardcodes the target pause level to `1` (admin-paused) regardless of the pool's state before the protocol pause. When the protocol pauses a pool that was fully active (level 0) and later unpauses it, the pool is left at level 1 (admin-paused), where `whenNotPaused` still reverts. Swaps remain permanently blocked until the pool admin separately calls `unpausePool`. The protocol owner has no path to restore level 0 directly.

---

### Finding Description

The pause state machine has three levels:

| Level | Meaning |
|---|---|
| 0 | Active |
| 1 | Paused by admin |
| 2 | Paused by protocol |

`protocolPausePool` accepts pools at **either** level 0 or level 1 as valid starting states before escalating to level 2: [1](#0-0) 

`protocolUnpausePool` then unconditionally targets level 1, never level 0: [2](#0-1) 

The pool's `whenNotPaused` modifier rejects any non-zero pause level: [3](#0-2) 

This modifier guards every `swap` call: [4](#0-3) 

The only path from level 1 → 0 is `unpausePool`, which is gated to the pool admin: [5](#0-4) 

The protocol owner has no function that sets level 0 directly. After `protocolUnpausePool`, the protocol owner cannot complete the restoration — only the pool admin can.

---

### Impact Explanation

Every pool that was at level 0 (active) when `protocolPausePool` is called will be left at level 1 (admin-paused) after `protocolUnpausePool`. All `swap` calls revert with `PoolPaused`. Liquidity providers cannot trade out of positions; routers and integrators that call `swap` receive hard reverts. The pool is functionally broken for its core purpose until the pool admin intervenes. If the admin is unresponsive, the pool is permanently stuck. This matches the "broken core pool functionality causing loss of funds or unusable swap flows" impact gate.

---

### Likelihood Explanation

The trigger is a normal, privileged-but-valid protocol owner action: pause then unpause a pool. No malicious setup is required. The condition that causes the bug (pool was at level 0 before the protocol pause) is the common case — most pools spend the majority of their lifetime at level 0. Every protocol pause/unpause cycle on an active pool reproduces the issue deterministically.

---

### Recommendation

`protocolUnpausePool` must restore the pool to the level it held before the protocol pause. The simplest fix is to set level 0 directly (since the protocol can pause from both 0 and 1, and the intent of "unpause" is to restore operability):

```solidity
function protocolUnpausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 2) revert InvalidPauseTransition(cur, 0);
    IMetricOmmPoolFactoryActions(pool).setPause(0); // restore to active
}
```

Alternatively, store the pre-pause level in factory storage when `protocolPausePool` is called and restore it on unpause.

---

### Proof of Concept

```
State: pool at pauseLevel = 0 (active), swaps work normally.

Step 1 — Protocol owner calls protocolPausePool(pool):
  cur = 0 → check passes (cur != 0 && cur != 1 is false)
  setPause(2) → pauseLevel = 2

Step 2 — Protocol owner calls protocolUnpausePool(pool):
  cur = 2 → check passes (cur != 2 is false)
  setPause(1) → pauseLevel = 1   ← BUG: should be 0

Step 3 — Any user calls swap(...):
  whenNotPaused → _checkNotPaused() → pauseLevel (1) != 0 → revert PoolPaused()

Step 4 — Protocol owner has no function to set level 0.
         Only pool admin can call unpausePool() to reach level 0.
         If admin is absent, pool is permanently swap-disabled.
```
