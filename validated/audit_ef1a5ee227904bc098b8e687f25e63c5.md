Audit Report

## Title
Lossy `f64` Conversion in `convert_tokens_to_string_representation` Breaks ICRC-21 Consent Message Uniqueness - (File: `packages/icrc-ledger-types/src/icrc21/responses.rs`)

## Summary
The private helper `convert_tokens_to_string_representation` converts an arbitrary-precision `Nat` token amount to `f64` before formatting it for display in ICRC-21 `GenericDisplayMessage` consent messages. Because `f64` has only 53 bits of mantissa, distinct large token amounts that differ only in low-order bits are rounded to the same floating-point value and produce identical rendered strings. This breaks the uniqueness guarantee of ICRC-21 consent messages, allowing a malicious dapp to present a user with a consent screen that matches a different (slightly larger) transaction than the one actually being signed.

## Finding Description
At `packages/icrc-ledger-types/src/icrc21/responses.rs` lines 318–327, `convert_tokens_to_string_representation` calls `tokens.0.to_f64()` on the inner `BigUint` of a `Nat`, then divides by `10_f64.pow(decimals)`:

```rust
fn convert_tokens_to_string_representation(
    tokens: Nat,
    decimals: u8,
) -> Result<String, Icrc21Error> {
    let tokens = tokens.0.to_f64().ok_or(...)?;
    Ok(format!("{}", tokens / 10_f64.pow(decimals)))
}
```

This function is called for every amount field rendered in the `GenericDisplayMessage` path:
- `add_amount` (line 113)
- `add_fee` (line 141)
- `add_allowance` (line 186)
- `add_existing_allowance` (lines 211–212)

`f64` represents integers exactly only up to 2^53 = 9,007,199,254,740,992. For any integer above this threshold, `to_f64()` rounds to the nearest representable value. For a token with 8 decimals (ICP), 2^53 e8s corresponds to ~90,071,992.547 ICP. The integers 9,007,199,254,740,992 and 9,007,199,254,740,993 both convert to `9007199254740992.0_f64` (the latter is not representable), so both produce the rendered string `"90071992.54740992"`. The `FieldsDisplayMessage` path is unaffected because it stores the raw integer via `nat_to_u64` and delegates rendering to the wallet.

The exploit path is:
1. A malicious dapp prepares a transaction with Amount B (e.g., 9,007,199,254,740,993 e8s).
2. The dapp calls `icrc21_consent_message` with Amount B's Candid-encoded arguments.
3. The ICP or ICRC-1 ledger canister returns a `GenericDisplayMessage` containing `"90071992.54740992 ICP"` — identical to what Amount A (9,007,199,254,740,992 e8s) would produce.
4. A hardware wallet (e.g., Ledger ICP app) displays this consent message to the user.
5. The user approves, believing the amount matches their expectation.
6. The hardware wallet signs and submits the transaction for Amount B.

No existing guard prevents this: the `ok_or` error path only fires if `to_f64()` returns `None` (i.e., the value is infinite/NaN), which does not happen for any finite `Nat`.

## Impact Explanation
The ICRC-21 consent message is the sole user-visible authorization surface for hardware wallet flows. A violation of its rendering uniqueness means a user cannot give informed consent for large-amount transactions. A malicious dapp can exploit the collision to get a user to sign a transaction for a slightly different amount than displayed. The financial difference per collision is small (1 e8 at the 2^53 boundary, growing at higher magnitudes), but the security model of the consent mechanism is concretely broken for any user with a balance above ~90 million ICP or equivalent. This constitutes a moderate user-funds and ledger security impact with a concrete, demonstrable exploit path, matching the Medium severity tier.

## Likelihood Explanation
The `icrc21_consent_message` endpoint is a public query endpoint callable by any unprivileged principal on both the ICP ledger and ICRC-1 ledger. No special role or governance majority is required. The affected path (`GenericDisplayMessage`) is the default when no `device_spec` is provided and is the path used by hardware wallet integrations. The precondition — a user balance above ~90 million ICP — limits the affected population but is realistic for exchanges, treasury accounts, and large institutional holders. The attack requires a malicious dapp but no other special capability.

## Recommendation
Replace the lossy `f64` conversion with exact arbitrary-precision integer arithmetic, following the pattern already used by `format_amount` in `rs/bitcoin/ckbtc/minter/src/updates/icrc21.rs` (lines 211–222):

```rust
fn convert_tokens_to_string_representation(
    tokens: Nat,
    decimals: u8,
) -> Result<String, Icrc21Error> {
    let divisor = BigUint::from(10_u64).pow(decimals as u32);
    let whole = &tokens.0 / &divisor;
    let frac = &tokens.0 % &divisor;
    if frac.is_zero() {
        Ok(format!("{whole}"))
    } else {
        let frac_str = format!("{frac:0>width$}", width = decimals as usize);
        let trimmed = frac_str.trim_end_matches('0');
        Ok(format!("{whole}.{trimmed}"))
    }
}
```

This ensures every distinct `Nat` amount produces a distinct rendered string, restoring the uniqueness guarantee required by ICRC-21.

## Proof of Concept
In a unit test or local PocketIC integration test against the ICP or ICRC-1 ledger:

1. Call `icrc21_consent_message` with a Candid-encoded `icrc1_transfer` argument where `amount = Nat::from(9_007_199_254_740_992_u64)` (= 2^53 e8s) and `GenericDisplay` device spec.
2. Call `icrc21_consent_message` again with `amount = Nat::from(9_007_199_254_740_993_u64)` (= 2^53 + 1 e8s).
3. Assert that both returned `GenericDisplayMessage` strings contain `"90071992.54740992 ICP"` — they will be identical, confirming the collision.
4. Assert that the two strings are different — this assertion will fail, proving the bug.

The same test can be written as a direct unit test of `convert_tokens_to_string_representation` without any canister infrastructure.
