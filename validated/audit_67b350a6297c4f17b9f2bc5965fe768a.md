### Title
Stale `eligible_operational_addresses` Declaration Not Cleared on Operational Address Change — (`src/staking/staking.cairo`)

---

### Summary

When a staker changes their operational address via `change_operational_address`, the `eligible_operational_addresses` mapping entry for the **old** operational address is never cleared. This stale "declaration" (the analog of an uncleared approval) persists indefinitely. A staker can later exploit this stale entry to re-assign a previously-used operational address as their own — without the address owner's current consent — locking that address out of the protocol until the staker voluntarily releases it.

---

### Finding Description

The Starknet Staking contract implements a two-step consent mechanism for assigning operational addresses:

1. **`declare_operational_address(staker_address)`** — the prospective operational address calls this to record consent: `eligible_operational_addresses[caller] = staker_address`.
2. **`change_operational_address(operational_address)`** — the staker calls this to activate the declared address, after verifying `eligible_operational_addresses[operational_address] == staker_address`.

In `change_operational_address`, when the staker switches from `old_address` to `new_address`, the code correctly clears the active-use mapping:

```cairo
self.operational_address_to_staker_address.write(old_address, Zero::zero());
```

But it **never clears** the declaration mapping for the old address:

```cairo
// eligible_operational_addresses[old_address] is NOT cleared here
staker_info.operational_address = operational_address;
self.write_staker_info(:staker_address, :staker_info);
self.operational_address_to_staker_address.write(operational_address, staker_address);
```

The same omission exists in `remove_staker`: `operational_address_to_staker_address[operational_address]` is zeroed, but `eligible_operational_addresses[operational_address]` is left intact.

This means `eligible_operational_addresses[old_address]` permanently retains the stale consent for the original staker, even after the staker has moved on.

---

### Impact Explanation

**Attack path:**

1. `O1` calls `declare_operational_address(S1)` → `eligible_operational_addresses[O1] = S1`.
2. `S1` calls `change_operational_address(O1)` → `O1` becomes S1's active operational address.
3. `S1` later calls `change_operational_address(O2)` (after `O2` declares for `S1`):
   - `operational_address_to_staker_address[O1]` → zero (cleared ✓)
   - `eligible_operational_addresses[O1]` → still `S1` (NOT cleared ✗)
4. `O1` is now free. A new staker `S3` stakes directly with `operational_address = O1` (via `stake()`), which writes `operational_address_to_staker_address[O1] = S3` but **never touches** `eligible_operational_addresses[O1]`, which remains `S1`.
5. `S3` unstakes → `operational_address_to_staker_address[O1]` → zero again.
6. `S1` now calls `change_operational_address(O1)`:
   - `operational_address_to_staker_address[O1].is_zero()` → true ✓
   - `eligible_operational_addresses[O1] == S1` → true ✓ (stale!)
   - **`S1` successfully hijacks `O1` without `O1`'s current consent.**

Once `O1` is assigned to `S1`, `O1` is locked: `declare_operational_address` checks `operational_address_to_staker_address[O1].is_zero()` and reverts with `OPERATIONAL_IN_USE` if non-zero. `O1` cannot revoke or re-declare for any other staker until `S1` voluntarily releases them. `O1` is effectively frozen from participating as an operational address for any other staker for an indefinite period.

**Impact category:** Medium — Griefing with no profit motive but damage to users. `O1` is locked out of the operational address role without their current consent, preventing them from serving as an operational address for any other staker.

---

### Likelihood Explanation

Any staker who has ever changed their operational address retains a stale declaration for the old address. The exploit requires only that the old address is temporarily free (not currently in use by another staker). This is a realistic condition: stakers regularly rotate operational addresses, and the old address may be idle. No privileged access is required — only the staker's own address calling `change_operational_address`.

---

### Recommendation

In `change_operational_address`, after clearing `operational_address_to_staker_address[old_address]`, also clear the stale declaration:

```cairo
self.operational_address_to_staker_address.write(old_address, Zero::zero());
self.eligible_operational_addresses.write(old_address, Zero::zero()); // add this
```

Similarly, in `remove_staker`, clear the declaration for the staker's final operational address:

```cairo
self.operational_address_to_staker_address.write(operational_address, Zero::zero());
self.eligible_operational_addresses.write(operational_address, Zero::zero()); // add this
```
