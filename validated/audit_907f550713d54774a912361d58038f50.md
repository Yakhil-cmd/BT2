### Title
Unconditional Zero-Value BTC Transfer in `transfer_to_pools_when_unstake` Permanently Freezes Staker's STRK Stake — (File: `src/staking/staking.cairo`)

---

### Summary

`transfer_to_pools_when_unstake` in `src/staking/staking.cairo` unconditionally transfers the delegated token balance to every pool contract during `unstake_action`, including when that balance is zero. If the BTC pool token reverts on zero-value transfers — a documented behavior of certain ERC20 tokens — `unstake_action` will always revert, permanently trapping the staker's STRK principal in the contract with no recovery path.

---

### Finding Description

When a staker calls `unstake_action`, the internal helper `transfer_to_pools_when_unstake` iterates over every pool the staker owns and transfers the current delegated balance to each pool contract: [1](#0-0) 

```cairo
fn transfer_to_pools_when_unstake(...) {
    for (pool_contract, token_address) in staker_pool_info.pools {
        let pool_balance = self.get_delegated_balance(:staker_address, :pool_contract);
        let token_dispatcher = IERC20Dispatcher { contract_address: token_address };
        ...
        let decimals = self.get_token_decimals(:token_address);
        token_dispatcher
            .checked_transfer(
                recipient: pool_contract,
                amount: pool_balance.to_native_amount(:decimals).into(),
            );
    }
}
```

There is **no guard** checking whether `pool_balance` is zero before calling `checked_transfer`. If `pool_balance` is zero and the pool's token (a BTC-variant ERC20) reverts on zero-value transfers, the entire `unstake_action` call reverts.

`unstake_action` is the only way for a staker to recover their STRK principal: [2](#0-1) 

After `unstake_intent` is called, `staker_info.unstake_time` is set to `Some(...)`. Any subsequent call to `unstake_intent` is blocked: [3](#0-2) 

And `add_stake_from_pool` rejects new delegations once an unstake intent is in progress: [4](#0-3) 

So once the staker is in the unstake-intent state with an empty BTC pool, there is no escape: `unstake_action` always reverts, and no new delegators can join to make the balance non-zero.

The BTC token is any ERC20 with 5–18 decimals added by the token admin: [5](#0-4) 

The protocol explicitly supports multiple BTC-variant tokens. Some real-world ERC20 tokens (e.g., LEND, certain USDT deployments, and others catalogued at [weird-erc20](https://github.com/d-xo/weird-erc20#revert-on-zero-value-transfers)) revert on zero-value transfers.

---

### Impact Explanation

**Permanent freezing of the staker's STRK principal.** The staker's entire self-staked STRK amount is locked in the staking contract forever. There is no admin rescue function, no alternative withdrawal path, and no way to re-enter a state where `unstake_action` can succeed. This matches the allowed impact: *Temporary/Permanent freezing of funds* (High).

---

### Likelihood Explanation

**Medium.** The scenario requires two conditions that are both realistic:

1. **A BTC pool with zero delegated balance at the time of `unstake_action`.** This is a normal lifecycle event: a staker opens a BTC pool, all delegators exit via `exit_delegation_pool_action` (which transfers BTC back to the pool contract and zeroes the staking contract's delegated balance), and then the staker attempts to unstake. The staker has no way to force new delegators to join after `unstake_intent` is called.

2. **The BTC token reverts on zero-value transfers.** This is a documented, non-exotic ERC20 behavior. The token admin may add such a token without being aware of this edge case.

Neither condition requires a malicious actor; both arise from ordinary protocol usage.

---

### Recommendation

Add a zero-check before the transfer in `transfer_to_pools_when_unstake`:

```cairo
let native_amount = pool_balance.to_native_amount(:decimals);
if native_amount.is_non_zero() {
    token_dispatcher
        .checked_transfer(recipient: pool_contract, amount: native_amount.into());
}
```

This mirrors the fix applied in the referenced H-04 report: condition the transfer on the amount being positive.

---

### Proof of Concept

1. Token admin calls `add_token` and `enable_token` for a BTC-variant ERC20 that reverts on zero-value transfers.
2. Staker calls `stake(...)` and then `set_open_for_delegation(btc_token_address)` to open a BTC pool.
3. A delegator calls `enter_delegation_pool(...)` and later `exit_delegation_pool_intent` + `exit_delegation_pool_action`, fully withdrawing their BTC. The staking contract's delegated balance for the BTC pool is now zero.
4. Staker calls `unstake_intent()`. `staker_info.unstake_time` is set; the staker is locked into the exit flow.
5. After the exit window, staker (or anyone) calls `unstake_action(staker_address)`.
6. Inside `unstake_action`, `transfer_to_pools_when_unstake` is reached. It calls `checked_transfer(recipient: btc_pool, amount: 0)` on the BTC token.
7. The BTC token reverts on the zero-value transfer. The entire `unstake_action` reverts.
8. The staker cannot call `unstake_intent` again (blocked by `Error::UNSTAKE_IN_PROGRESS`). No new delegators can join (blocked by `Error::UNSTAKE_IN_PROGRESS` in `add_stake_from_pool`). The staker's STRK is permanently frozen. [6](#0-5) [7](#0-6)

        fn transfer_to_pools_when_unstake(
            ref self: ContractState,
            staker_address: ContractAddress,
            staker_pool_info: StoragePath<InternalStakerPoolInfoV2>,
        ) {
            for (pool_contract, token_address) in staker_pool_info.pools {
                let pool_balance = self.get_delegated_balance(:staker_address, :pool_contract);
                let token_dispatcher = IERC20Dispatcher { contract_address: token_address };
                let pool_dispatcher = IPoolDispatcher { contract_address: pool_contract };
                pool_dispatcher.set_staker_removed();
                self

                        :staker_address, :pool_contract, delegated_balance: Zero::zero(),

                let decimals = self.get_token_decimals(:token_address);
                token_dispatcher
                    .checked_transfer(
                        recipient: pool_contract,
                        amount: pool_balance.to_native_amount(:decimals).into(),
                    );
