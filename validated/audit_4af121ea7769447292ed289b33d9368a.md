### Title
Missing Access Control on `update_rewards` Allows Any Caller to Permanently Suppress Staker Block Rewards — (File: `src/staking/staking.cairo`)

---

### Summary
The `update_rewards` function in `IStakingRewardsManager` is publicly callable by any address and accepts a caller-controlled `disable_rewards` boolean. An unprivileged attacker can call `update_rewards(victim_staker, disable_rewards: true)` every block, consuming the global `last_reward_block` slot and permanently suppressing the victim's block rewards with no bypass available to the staker.

---

### Finding Description
`update_rewards` is exposed as a public ABI function with no role check — `general_prerequisites()` only asserts the contract is unpaused and the caller is non-zero. [1](#0-0) 

The function writes `last_reward_block` to the current block number **before** evaluating the `disable_rewards` flag: [2](#0-1) 

When `disable_rewards: true` is passed, execution returns immediately after updating `last_reward_block`, skipping the entire reward calculation and distribution path. Because `last_reward_block` is a **global** (not per-staker) storage slot, this single call also blocks every other staker from receiving rewards in the same block — the next legitimate call will revert with `REWARDS_ALREADY_UPDATED`. [3](#0-2) 

The `IStakingRewardsManager` interface exposes this with no caller restriction: [4](#0-3) 

---

### Impact Explanation
Block rewards that are skipped are **permanently lost** — they are never added to `staker_info.unclaimed_rewards_own` and never transferred. An attacker who front-runs the legitimate `update_rewards` call every block causes the victim staker to accrue zero consensus-phase block rewards indefinitely. This constitutes **permanent freezing of unclaimed yield** for the targeted staker, and also denies rewards to every other staker in each griefed block.

**Allowed impact matched:** *High — Permanent freezing of unclaimed yield.*

---

### Likelihood Explanation
High. The function is permissionlessly callable by any EOA or contract. Starknet transaction fees are low. The attacker needs only to submit one transaction per block targeting any valid, active staker. No privileged key, leaked secret, or external dependency is required. The attack is fully self-contained within the staking contract.

---

### Recommendation
Restrict `update_rewards` to an authorized caller — for example, the designated block-proposer contract or a whitelisted consensus contract — using a role check analogous to `assert_caller_is_attestation_contract`. Alternatively, if `disable_rewards: true` is a legitimate signal (e.g., staker missed a block), gate that flag behind a privileged caller while allowing permissionless calls only with `disable_rewards: false`.

---

### Proof of Concept

1. Consensus rewards are active (`is_pre_consensus()` returns `false`).
2. Attacker monitors the chain. At the start of block `N`, before the legitimate block-proposer transaction lands, attacker submits:
   ```
   staking.update_rewards(victim_staker_address, disable_rewards: true)
   ```
3. Inside `update_rewards`:
   - `current_block_number > last_reward_block` passes (first call this block). [3](#0-2) 
   - `last_reward_block` is written to block `N`. [5](#0-4) 
   - `disable_rewards` is `true` → early return, no rewards distributed. [6](#0-5) 
4. The legitimate `update_rewards(victim_staker, disable_rewards: false)` call in the same block reverts: `current_block_number > last_reward_block` is now `false`.
5. `victim_staker.unclaimed_rewards_own` is unchanged; block `N` rewards are permanently lost.
6. Attacker repeats every block → victim accrues zero block rewards indefinitely.
