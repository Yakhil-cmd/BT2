### Title
Overly Broad `skip_stateful_validations` Check Allows Signature-Bypass Invoke Admission into Mempool - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function is designed to improve UX for users sending `deploy_account` + `invoke` together. It skips the `__validate__` entry point (signature verification) for an invoke with `nonce=1` when the account's committed nonce is `0` and the account has a transaction in the mempool. However, the mempool check `account_tx_in_pool_or_recent_block` is satisfied by **any** transaction for that account — not only a pending `deploy_account`. An attacker who observes a `deploy_account` transaction for account A in the mempool can immediately inject a malicious invoke with `nonce=1` and an arbitrary/invalid signature, which the gateway admits without calling `__validate__`.

---

### Finding Description

`skip_stateful_validations` fires when two conditions hold simultaneously:

```
tx.nonce() == Nonce(Felt::ONE)  &&  account_nonce == Nonce(Felt::ZERO)
``` [1](#0-0) 

When both are true, the function calls `account_tx_in_pool_or_recent_block` and returns its result as the `skip_validate` flag. [2](#0-1) 

The mempool implements this check as:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [3](#0-2) 

`tx_pool.contains_account` returns `true` for **any** transaction type in the pool — `deploy_account`, `invoke`, or `declare`. The code comment claims this is sufficient because "it means that either it has a deploy_account transaction or transactions with future nonces that passed validations," but this reasoning is incorrect: a valid `invoke` with `nonce=0` from the legitimate owner also satisfies the check, as does any other transaction type.

When `skip_validate` is `true`, `run_validate_entry_point` sets `execution_flags.validate = false`:

```rust
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [4](#0-3) 

Inside `StatefulValidator::perform_validations`, when `validate == false` the `__validate__` entry point is never called:

```rust
if !tx.execution_flags.validate {
    return Ok(());
}
// `__validate__` call.
let (_optional_call_info, actual_cost) = self.validate(&tx, tx_context.clone())?;
``` [5](#0-4) 

The gateway's `add_tx_inner` flow is: stateless validation → conversion → stateful validation (nonce + resource bounds + mempool validate + skip_validate check + validate entry point). None of the earlier steps verify the cryptographic signature independently; that is entirely delegated to the `__validate__` entry point. [6](#0-5) 

---

### Impact Explanation

An attacker can inject an invoke transaction carrying an **arbitrary or invalid signature** into the mempool for any account whose committed nonce is `0` and that has any pending transaction in the pool. The malicious transaction occupies the `nonce=1` slot. When the legitimate user subsequently submits their own `nonce=1` invoke, the mempool rejects it as a duplicate nonce, effectively **blocking the account's first post-deployment invoke**. This matches the allowed impact: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

---

### Likelihood Explanation

The attack is unprivileged and requires only:
1. Observing a `deploy_account` (or any `nonce=0` invoke) for a target account enter the mempool — publicly visible.
2. Submitting a crafted invoke with `nonce=1` and a garbage signature before the legitimate `nonce=1` invoke arrives.

This is a straightforward race condition with no special access required. The window is the time between the `deploy_account` being accepted and the user's `invoke` being submitted, which in the intended UX flow (`create_deploy_account_tx_and_invoke_tx`) is non-zero because the gateway processes them sequentially. [7](#0-6) 

---

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a **pending `deploy_account` transaction** exists for the sender address. The mempool should expose a dedicated query such as `has_pending_deploy_account(address) -> bool` that inspects the transaction type stored in the pool, rather than returning `true` for any transaction type. Alternatively, restrict the skip-validate path to only fire when the mempool confirms the pending transaction for that address is of type `DeployAccount`.

---

### Proof of Concept

```
1. Account A: committed nonce = 0, no contract deployed yet.

2. Legitimate user submits deploy_account tx (nonce=0) for account A.
   → Gateway accepts it; mempool pool now contains A's deploy_account.
   → account_tx_in_pool_or_recent_block(A) == true.

3. Attacker observes deploy_account in mempool (public).
   Attacker crafts: Invoke { sender=A, nonce=1, signature=[0xdead, 0xbeef] }.

4. Gateway stateless validation: passes (signature length ≤ max, resource bounds OK).

5. Gateway stateful validation:
   a. get_nonce_from_state(A) → Nonce(0)          [committed state]
   b. validate_nonce: 0 ≤ 1 ≤ max_allowed_nonce_gap  → OK
   c. validate_by_mempool: no duplicate nonce=1 for A → OK
   d. skip_stateful_validations:
        tx.nonce()==1 && account_nonce==0  → true
        account_tx_in_pool_or_recent_block(A) → true  (deploy_account is in pool)
        returns true  →  skip_validate = true
   e. run_validate_entry_point(skip_validate=true):
        execution_flags.validate = false
        __validate__ is NOT called → signature never verified

6. Malicious invoke (nonce=1, garbage signature) is forwarded to mempool.add_tx → accepted.

7. Legitimate user now submits their own invoke (nonce=1, valid signature).
   → mempool.validate_tx: DuplicateNonce { nonce=1 } → REJECTED.

Result: attacker's invalid transaction occupies nonce=1 slot; legitimate user's
        first post-deployment invoke is permanently blocked until the malicious
        tx is evicted or expires.
``` [8](#0-7) [3](#0-2)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
