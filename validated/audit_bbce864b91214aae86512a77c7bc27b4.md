### Title
Incorrect `HISTORY_SERVE_WINDOW` Constant Causes EIP-2935 Block Hash Storage at Wrong Slots - (`File: basic_bootloader/src/bootloader/block_flow/ethereum/eip_2935_historical_block_hash/mod.rs`)

---

### Summary

ZKsync OS implements EIP-2935 (historical block hash storage) using a ring-buffer constant `HISTORY_SERVE_WINDOW = 8191`, but the EIP-2935 specification mandates `HISTORY_SERVE_WINDOW = 8192`. This off-by-one error causes every parent block hash to be written to the wrong storage slot once block numbers exceed 8191, permanently diverging from Ethereum's EIP-2935 state layout and breaking any contract that reads from the history contract using the standard slot formula.

---

### Finding Description

EIP-2935 defines a system contract at a fixed address that stores the last `HISTORY_SERVE_WINDOW` parent block hashes in a ring buffer. The storage slot for block `N`'s hash is `N % HISTORY_SERVE_WINDOW`. The EIP specification fixes this window at **8192**.

In ZKsync OS, the `eip2935_system_part` function is called automatically at the start of every block (in `EthereumPreOp::pre_op`) and writes the parent block hash to storage:

```rust
const HISTORY_SERVE_WINDOW: u64 = 8191;   // ← should be 8192

let slot_idx = (block_number - 1) % HISTORY_SERVE_WINDOW;
``` [1](#0-0) [2](#0-1) 

The write path is invoked unconditionally before every block's transaction loop: [3](#0-2) 

For all block numbers where `(block_number - 1) % 8191 ≠ (block_number - 1) % 8192` — which first diverges at `block_number = 8192` — the hash is written to a slot that no standard EIP-2935-compliant reader will ever query. Any EVM contract that calls the history contract using the canonical formula `block_number % 8192` will read `bytes32(0)` instead of the actual hash.

This is structurally identical to the VotingEscrow bug: a periodic/modular accounting system uses an incorrect constant for its ring-buffer period, causing state to accumulate at wrong indices and breaking all downstream consumers of that state.

---

### Impact Explanation

1. **EVM semantic mismatch**: The ZKsync OS EIP-2935 state layout diverges from Ethereum's after block 8191. Any contract that relies on EIP-2935 for historical block hash verification (e.g., cross-chain bridges, light-client verifiers, rollup fraud proofs) will receive `bytes32(0)` instead of the correct hash, causing silent verification failures or incorrect security decisions.

2. **Forward/proving divergence**: If the prover or any external verifier uses the correct EIP-2935 constant (8192), the storage state it expects will not match what the sequencer produced, causing proof failures or state root mismatches.

3. **Permanent state corruption**: Because the ring buffer wraps at 8191 instead of 8192, every 8191 blocks the write pointer is one slot ahead of where readers expect it. This is not a transient error — it accumulates permanently across the chain's lifetime.

---

### Likelihood Explanation

- **Trigger**: Automatic. `eip2935_system_part` is called at the start of every block with no user interaction required.
- **Onset**: The divergence first manifests at block 8192 (when `(block_number - 1) % 8191` first differs from `(block_number - 1) % 8192`).
- **No privileged access required**: Any block execution triggers the write. An unprivileged transaction sender simply needs to submit a transaction in any block ≥ 8192.
- **Detectability**: Silent — the write succeeds, no error is raised, but the stored slot is wrong.

---

### Recommendation

Change the constant to match the EIP-2935 specification:

```rust
// Before (incorrect):
const HISTORY_SERVE_WINDOW: u64 = 8191;

// After (correct):
const HISTORY_SERVE_WINDOW: u64 = 8192;
``` [1](#0-0) 

Add a test that verifies the slot written for block 8193 matches `(8193 - 1) % 8192 = 1`, not `(8193 - 1) % 8191 = 2`.

---

### Proof of Concept

The divergence is deterministic and can be demonstrated by inspection:

- At `block_number = 8192`: slot written = `(8192 - 1) % 8191 = 8191 % 8191 = 0`. Correct slot per EIP = `8191 % 8192 = 8191`. **Mismatch.**
- At `block_number = 8193`: slot written = `8192 % 8191 = 1`. Correct slot = `8192 % 8192 = 0`. **Mismatch.**

A contract calling the EIP-2935 history contract at address `HISTORY_STORAGE_ADDRESS` to retrieve the hash of block 8191 would compute slot `8191 % 8192 = 8191` and read `bytes32(0)`, because ZKsync OS wrote it to slot `8191 % 8191 = 0`. [4](#0-3) [5](#0-4)
