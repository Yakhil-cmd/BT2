### Title
Unchecked Exit Precompile Return Value After `_burn()` Causes Permanent Token Loss — (`etc/eth-contracts/contracts/EvmErc20.sol`, `etc/eth-contracts/contracts/EvmErc20V2.sol`)

---

### Summary

`withdrawToNear()` and `withdrawToEthereum()` in `EvmErc20.sol` and `EvmErc20V2.sol` irrevocably burn the caller's ERC-20 tokens **before** calling the exit precompile, and then **never check the return value** of that call. If the precompile call fails for any reason — including a user-supplied `recipient` that causes the input to exceed the precompile's hard `MAX_INPUT_SIZE` limit — the tokens are permanently destroyed on Aurora with no corresponding release on the destination chain and no refund path.

---

### Finding Description

Both `EvmErc20.sol` and `EvmErc20V2.sol` implement the same withdrawal pattern:

```solidity
// EvmErc20.sol lines 53-63
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // ← tokens destroyed here, unconditionally

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        // res is assigned but NEVER checked — no revert on failure
    }
}
``` [1](#0-0) 

The identical pattern appears in `EvmErc20V2.sol`: [2](#0-1) 

And in `withdrawToEthereum` in both contracts: [3](#0-2) 

The `ExitToNear` precompile enforces a hard input-size ceiling:

```rust
const MAX_INPUT_SIZE: usize = 1_024;
``` [4](#0-3) 

Input validation rejects anything outside the allowed range:

```rust
fn validate_input_size(input: &[u8], min: usize, max: usize) -> Result<(), ExitError> {
    if input.len() < min || input.len() > max {
        return Err(ExitError::Other(Cow::from("ERR_INVALID_INPUT")));
    }
    Ok(())
}
``` [5](#0-4) 

For `EvmErc20.sol`, the precompile input is `1 + 32 + recipient.length` bytes. When `recipient.length > 991`, the total exceeds 1 024 bytes and the precompile returns `ExitError`, causing the low-level `call` to return `0`. Because `res` is never inspected and there is no `revert`, the outer transaction succeeds — with the tokens already burned.

For `EvmErc20V2.sol` the threshold is `recipient.length > 971` (input layout is `1 + 20 + 32 + recipient.length`). [6](#0-5) 

The `ExitToNear` precompile also fails when called in a static context or via `delegatecall`: [7](#0-6) 

In every failure case the ERC-20 tokens are gone from Aurora's EVM state, the NEP-141 tokens remain locked in Aurora's NEAR account, and there is no refund mechanism in either contract.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

When the precompile call fails silently:
- The caller's ERC-20 balance is decremented by `amount` (burned).
- No NEP-141 `ft_transfer` promise is ever scheduled.
- The NEP-141 tokens remain in Aurora's NEAR account indefinitely with no on-chain recovery path.
- The user has permanently lost their bridged assets.

---

### Likelihood Explanation

**Medium.** The trigger is fully unprivileged and user-controlled:

1. `withdrawToNear` accepts an arbitrary `bytes memory recipient` with no length validation in Solidity.
2. Any caller can supply a `recipient` longer than 991 bytes (EvmErc20) or 971 bytes (EvmErc20V2), which is trivially constructable.
3. No admin action, governance capture, or external oracle is required.
4. The function signature is part of the public `IExit` interface, so any EOA or contract can call it directly. [8](#0-7) 

---

### Recommendation

1. **Check the precompile return value and revert on failure.** Replace the unchecked assembly block with:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

2. **Validate `recipient` length before burning.** Add an explicit upper-bound check (e.g., `require(recipient.length <= 991)`) so the transaction reverts before any tokens are destroyed.

3. **Apply the same fix to `withdrawToEthereum`** in both `EvmErc20.sol` and `EvmErc20V2.sol`.

---

### Proof of Concept

1. Attacker (or unsuspecting user) holds 1 000 units of a bridged ERC-20 token (`EvmErc20`).
2. Calls `withdrawToNear(recipient, 1000)` where `recipient` is a 992-byte byte array (any content).
3. `_burn(msg.sender, 1000)` executes — ERC-20 balance drops to 0.
4. Input to precompile is `1 + 32 + 992 = 1025` bytes, exceeding `MAX_INPUT_SIZE = 1024`.
5. `ExitToNear.run()` returns `Err(ExitError::Other("ERR_INVALID_INPUT"))`. [5](#0-4) 

6. The EVM `call` returns `0`; `res` is never checked; no `revert` is issued.
7. Transaction succeeds. Caller's ERC-20 balance is 0. No `ft_transfer` promise was created. NEP-141 tokens remain locked in Aurora's NEAR account forever. [1](#0-0)
