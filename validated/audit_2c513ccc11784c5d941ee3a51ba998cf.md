### Title
Compile-time hardcoded EVM `CHAIN_ID` in wallet contract enables cross-fork replay of ETH-implicit account transactions — (File: `runtime/near-wallet-contract/implementation/wallet-contract/src/internal.rs`)

---

### Summary

The EVM chain ID used to authenticate Ethereum-style transactions in the NEAR wallet contract is baked into the WASM binary at compile time as a Rust `const`. If NEAR undergoes a hard fork (network split), both resulting chains would run the same deployed wallet contract WASM with the same EVM chain ID (397 for mainnet). A signed EVM transaction from one chain can then be replayed on the other chain, bypassing the only chain-domain guard and executing unauthorized actions — including fund transfers — on ETH-implicit accounts.

---

### Finding Description

In `internal.rs`, the EVM chain ID is defined as a compile-time constant read from a source file via `std::include!`:

```rust
pub const CHAIN_ID: u64 = std::include!("CHAIN_ID");
```

The file `src/CHAIN_ID` contains the literal integer `397` (mainnet) and is written by `build.rs` before the Docker WASM build, then restored. The resulting value is frozen into the WASM binary at build time. [1](#0-0) [2](#0-1) 

The sole chain-domain check in `validate_tx_relayer_data` compares the transaction's `chain_id` field against this frozen constant:

```rust
if tx.chain_id != Some(CHAIN_ID) {
    return Err(Error::Relayer(RelayerError::InvalidChainId));
}
``` [3](#0-2) 

This function is the only place where cross-chain domain separation is enforced for EVM transactions submitted to `rlp_execute`. There is no runtime lookup of the current chain's identity — the constant cannot change without recompiling and redeploying the WASM. [4](#0-3) 

The wallet contract WASM is embedded as a static byte array in `near-wallet-contract/src/lib.rs` and served as a global contract to all ETH-implicit accounts on the chain: [5](#0-4) 

If NEAR undergoes a hard fork producing two live chains, both chains share the identical deployed WASM (chain ID 397). A transaction signed by a user on chain A carries `chain_id = 397`. When submitted to chain B, `validate_tx_relayer_data` evaluates `397 == Some(397)` — the check passes — and the transaction executes on chain B without the user's intent.

---

### Impact Explanation

An attacker who observes a signed EVM transaction on one fork (all transaction bytes are public on-chain) can submit the identical RLP-encoded bytes to the wallet contract on the sibling fork. Because the only cross-chain discriminator (`CHAIN_ID`) is identical on both forks, the authorization check passes. The wallet contract then increments the nonce and dispatches the embedded Near action (transfer, function call, add/delete key) on the attacker's chosen chain. Concretely:

- **Unauthorized fund transfers**: a `Transfer` action drains the ETH-implicit account's balance on the replayed chain.
- **Unauthorized key management**: an `AddKey` or `DeleteKey` action modifies access control on the replayed chain.

The corrupted value is the authorization decision in `validate_tx_relayer_data`: a transaction that is valid only for chain A is incorrectly accepted as valid for chain B.

---

### Likelihood Explanation

The precondition is a NEAR network split (hard fork). NEAR's BFT-based protocol upgrade mechanism makes accidental splits unlikely, but deliberate or contentious forks are not impossible — the Golom judge's reasoning applies equally here: "hard forks can and do happen." The EVM chain IDs 397/398 are registered on chainlist.org and are not rotated on protocol upgrades, so both fork chains would carry the same numeric ID. No special privilege is required to replay a transaction; the attacker only needs the publicly available RLP bytes from the original chain.

---

### Recommendation

Replace the compile-time constant with a runtime lookup of the current chain's identity. The NEAR SDK already exposes the genesis chain ID string via `env::chain_id()` (demonstrated in the `ext_chain_id` host function). The wallet contract should:

1. Call `env::chain_id()` (or an equivalent host function returning the numeric EVM chain ID) at the start of `rlp_execute` or `validate_tx_relayer_data`.
2. Compare `tx.chain_id` against the dynamically obtained value rather than the frozen `CHAIN_ID` constant.

If a direct numeric EVM chain ID host function is not yet available, it should be added to the NEAR VM host interface so the wallet contract can always reflect the live chain's identity rather than the identity at compile time.

---

### Proof of Concept

1. User signs an EVM transaction on NEAR mainnet (chain ID 397):
   ```
   Transaction2930 { chain_id: 397, nonce: 5, to: <receiver>, value: 1 ETH-equiv, ... }
   ```
2. NEAR undergoes a hard fork; two chains now exist, both running wallet contract WASM with `CHAIN_ID = 397`.
3. Attacker reads the RLP-encoded signed transaction from chain A's history (public data).
4. Attacker calls `rlp_execute(target, <replayed_tx_bytes_b64>)` on chain B.
5. `validate_tx_relayer_data` evaluates:
   - `tx.address == current_address` ✓ (same key, same address on both forks)
   - `tx.chain_id == Some(397)` ✓ (frozen constant matches)
   - `tx.nonce == expected_nonce` ✓ (nonce state is identical at fork point)
6. The wallet contract accepts the transaction, increments the nonce, and dispatches the transfer on chain B — draining the user's ETH-implicit account without consent. [4](#0-3) [6](#0-5)

```rust
fn validate_tx_relayer_data<'a>(
    tx: &NormalizedEthTransaction,
    target: &'a AccountId,
    context: &ExecutionContext,
    expected_nonce: u64,
) -> Result<TargetKind<'a>, Error> {
    if tx.address.raw() != context.current_address {
        return Err(Error::Relayer(RelayerError::InvalidSender));
    }

    if tx.chain_id != Some(CHAIN_ID) {
        return Err(Error::Relayer(RelayerError::InvalidChainId));
    }

    let to = tx.to.ok_or(Error::User(UserError::EvmDeployDisallowed))?.raw();

    let target_kind = parse_target(target, context.current_address);

    // valid targets satisfy `to == target` or `to == hash(target)`
    let is_valid_target = match target_kind {
        TargetKind::CurrentAccount if to == context.current_address => {
            target == &context.current_account_id
        }
        TargetKind::EthImplicit(address) if to == address => {
            target.as_str()
                == format!("0x{}{}", hex::encode(address), context.current_account_suffix())
        }
        _ => to == account_id_to_address(target),
    };

    if !is_valid_target {
        return Err(Error::Relayer(RelayerError::InvalidTarget));
    }

    let nonce = if tx.nonce <= U64_MAX {
        tx.nonce.low_u64()
    } else {
        return Err(Error::Relayer(RelayerError::InvalidNonce));
    };
    if nonce != expected_nonce {
        return Err(Error::Relayer(RelayerError::InvalidNonce));
    }

    // Relayers must attach at least as much gas as the user requested.
    let gas_limit = if tx.gas_limit < U64_MAX { tx.gas_limit.as_u64() } else { u64::MAX };
    if env::prepaid_gas().as_gas() < gas_limit.saturating_mul(GAS_MULTIPLIER) {
        return Err(Error::Relayer(RelayerError::InsufficientGas));
    }

    Ok(target_kind)
}
```
