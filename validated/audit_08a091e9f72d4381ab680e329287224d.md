### Title
`withdraw_wnear_to_router` Reads wNEAR Address from Storage at Callback Execution Time, Allowing `factory_set_wnear_address()` to Break In-Flight XCC Operations and Freeze NEAR Funds - (File: `engine/src/contract_methods/xcc.rs`)

### Summary
The `withdraw_wnear_to_router` callback reads the wNEAR ERC-20 address fresh from on-chain storage at execution time rather than using the address baked into the promise arguments at construction time. If the owner calls `factory_set_wnear_address()` between when the XCC precompile fires and when the `withdraw_wnear_to_router` callback executes, the callback uses the new (wrong) wNEAR address. The user has not approved the new contract, so the EVM `transferFrom` reverts, `withdraw_wnear_to_router` returns `ERR_WITHDRAW_FAILED`, and the entire XCC promise chain fails. When a new router sub-account was being created as part of this XCC flow, the `STORAGE_AMOUNT` (2 NEAR) transferred from the engine to the router cannot be refunded because `send_refund` requires no failed promises, leaving it frozen in the router sub-account with no direct recovery path.

### Finding Description

**Root cause — stale wNEAR address read at callback time:**

In `engine/src/contract_methods/xcc.rs`, `withdraw_wnear_to_router` reads the wNEAR address from storage at the moment the callback executes:

```rust
let wnear_address = aurora_engine_precompiles::xcc::state::get_wnear_address(&io);
``` [1](#0-0) 

The `WithdrawWnearToRouterArgs` struct carries only `target` (the router sub-account address) and `amount`; the wNEAR address is never baked into the promise arguments at construction time: [2](#0-1) 

**The XCC promise chain constructed in `handle_precompile_promise`:** [3](#0-2) 

The chain is:
1. *(Optional)* Deploy/upgrade router batch → `factory_update_address_version` callback
2. `withdraw_wnear_to_router` self-call (reads wNEAR address from storage **at execution time**)
3. *(Optional, only when `create_needed`)* `send_refund` callback
4. User's actual cross-contract call

**The state-changing function — `factory_set_wnear_address`:** [4](#0-3) 

This function can be called by the owner at any time with no check for in-flight XCC operations. It immediately overwrites the stored wNEAR address: [5](#0-4) 

**Why `send_refund` cannot recover the stuck NEAR:**

When a new router is created, `STORAGE_AMOUNT` (2 NEAR) is transferred from the engine to the router: [6](#0-5) 

The `send_refund` callback is supposed to return this 2 NEAR to the engine, but it calls `require_no_failed_promises()` and panics if any prior promise in the chain failed: [7](#0-6) [8](#0-7) 

Since `withdraw_wnear_to_router` failed, `send_refund` panics, and the 2 NEAR remains in the router sub-account. There is no other on-chain recovery path without deploying a new router version containing a recovery function.

### Impact Explanation

The 2 NEAR (`STORAGE_AMOUNT`) transferred to a newly created router sub-account is frozen with no direct recovery mechanism. The engine's NEAR balance (which represents user-deposited funds) is reduced by 2 NEAR per affected router. Recovery requires the owner to deploy a new router version with an explicit withdrawal function. If a wNEAR migration coincides with many first-time XCC users (new router creations), multiple routers can be simultaneously affected, multiplying the frozen amount. This meets **High: Temporary freezing of funds**.

### Likelihood Explanation

The owner must call `factory_set_wnear_address` during the narrow inter-block window between XCC precompile execution and `withdraw_wnear_to_router` callback execution. This is most plausible during a planned wNEAR contract migration. During such a migration, multiple in-flight XCC operations from first-time users could be affected simultaneously. Likelihood is **Low**, but the scenario is operationally realistic.

### Recommendation

Pass the wNEAR address as an explicit field in `WithdrawWnearToRouterArgs` at promise construction time (inside `handle_precompile_promise`), rather than reading it from storage inside the callback. This makes the callback's behavior independent of any subsequent `factory_set_wnear_address` calls:

```rust
// In handle_precompile_promise, when building withdraw_call_args:
let wnear_address = get_wnear_address(io); // read once at construction time
let withdraw_call_args = WithdrawWnearToRouterArgs {
    target: sender,
    amount: required_near,
    wnear_address,  // bake it in
};
```

Then in `withdraw_wnear_to_router`, use `args.wnear_address` directly instead of calling `get_wnear_address(&io)`.

### Proof of Concept

1. User Alice calls the XCC precompile with `required_near > 0` and no existing router sub-account (`create_needed = true`).
2. `handle_precompile_promise` (engine/src/xcc.rs) constructs the promise chain:
   - Deploy batch: creates router, transfers 2 NEAR (`STORAGE_AMOUNT`) from engine to router.
   - `factory_update_address_version` callback.
   - `withdraw_wnear_to_router` callback with args `{target: alice_addr, amount: required_near}` — **wNEAR address NOT included**.
   - `send_refund` callback (because `create_needed = true`).
   - Alice's actual cross-contract call.
3. Owner calls `factory_set_wnear_address(new_wnear_addr)` — overwrites the stored wNEAR address.
4. `withdraw_wnear_to_router` callback executes; reads `new_wnear_addr` from storage (line 43 of `engine/src/contract_methods/xcc.rs`).
5. EVM call to `new_wnear_addr.withdrawToNear(alice_router, required_near)` fails — Alice approved only the old wNEAR contract.
6. `withdraw_wnear_to_router` returns `ERR_WITHDRAW_FAILED`.
7. `send_refund` executes but panics at `require_no_failed_promises()` — the 2 NEAR is not returned to the engine.
8. Alice's actual call does not execute.
9. **Result**: 2 NEAR is frozen in Alice's router sub-account with no direct recovery path.

pub fn factory_set_wnear_address<I: IO + Copy, E: Env>(

#[allow(clippy::too_many_lines)]
pub fn handle_precompile_promise<I, P>(
    io: &I,
    handler: &mut P,
    base_id: Option<PromiseId>,
    promise: &PromiseCreateArgs,
    required_near: Yocto,
    current_account_id: &AccountId,
) -> PromiseId
where
    P: PromiseHandler,
    I: IO + Copy,
{
    let target_account: &str = promise.target_account_id.as_ref();
    let sender = Address::decode(&target_account[0..40]).expect(ERR_INVALID_ACCOUNT);

    // Confirm target_account is of the form `{address}.{aurora}`
    // Address prefix parsed above, so only need to check `.{aurora}`
    assert_eq!(&target_account[40..41], ".", "{ERR_INVALID_ACCOUNT}");
    assert_eq!(
        &target_account[41..],
        current_account_id.as_ref(),
        "{ERR_INVALID_ACCOUNT}"
    );
    // Confirm there is 0 NEAR attached to the promise
    // (the precompile should not drain the engine's balance).
    assert_eq!(promise.attached_balance, ZERO_YOCTO, "{ERR_ATTACHED_NEAR}");

    let latest_code_version = get_latest_code_version(io);
    let sender_code_version = get_code_version_of_address(io, &sender);
    let deploy_needed = AddressVersionStatus::new(io, latest_code_version, sender_code_version);
    // 1. If the router contract account does not exist or is out of date then we start
    //    with a batch transaction to deploy the router. This batch also has an attached
    //    callback to update the engine's storage with the new version of that router account.
    let setup_id = match &deploy_needed {
        AddressVersionStatus::DeployNeeded { create_needed } => {
            let mut promise_actions = Vec::with_capacity(4);
            let code = get_router_code(io).0.into_owned();
            // After the deployment we will call the contract's initialize function
            let wnear_address = get_wnear_address(io);
            let wnear_account = crate::engine::nep141_erc20_map(*io)
                .lookup_right(&crate::engine::ERC20Address(wnear_address))
                .expect("wnear account not found");
            let init_args = format!(
                r#"{{"wnear_account": "{}", "must_register": {}}}"#,
                wnear_account.0.as_ref(),
                create_needed,
            );
            if *create_needed {
                promise_actions.push(PromiseAction::CreateAccount);
                promise_actions.push(PromiseAction::Transfer {
                    amount: STORAGE_AMOUNT,
                });
                promise_actions.push(PromiseAction::DeployContract { code });
                promise_actions.push(PromiseAction::FunctionCall {
                    name: "initialize".into(),
                    args: init_args.into_bytes(),
                    attached_yocto: ZERO_YOCTO,
                    gas: INITIALIZE_GAS,
                });
            } else {
                let deploy_args = DeployUpgradeParams {
                    code,
                    initialize_args: init_args.into_bytes(),
                };
                promise_actions.push(PromiseAction::FunctionCall {
                    name: "deploy_upgrade".into(),
                    args: borsh::to_vec(&deploy_args).expect(ERR_UPGRADE_ARG_SERIALIZATION),
                    attached_yocto: ZERO_YOCTO,
                    gas: UPGRADE_GAS + INITIALIZE_GAS,
                });
            }

            let batch = PromiseBatchAction {
                target_account_id: promise.target_account_id.clone(),
                actions: promise_actions,
            };
            // Safety: This batch creation is safe because it only acts on the router sub-account
            // (not the main engine account), and the actions performed are only (1) create it
            // for the first time and/or (2) deploy the code from our storage (i.e. the deployed
            // code is controlled by us, not the user).
            let promise_id = match base_id {
                Some(id) => handler.promise_attach_batch_callback(id, &batch),
                None => handler.promise_create_batch(&batch),
            };
            // Add a callback here to update the version of the account
            let args = AddressVersionUpdateArgs {
                address: sender,
                version: latest_code_version,
            };
            let callback = PromiseCreateArgs {
                target_account_id: current_account_id.clone(),
                method: "factory_update_address_version".into(),
                args: borsh::to_vec(&args).unwrap(),
                attached_balance: ZERO_YOCTO,
                attached_gas: VERSION_UPDATE_GAS,
            };

            // Safety: A call from the engine to the engine's `factory_update_address_version`
            // method is safe because that method only writes the specific router sub-account
            // metadata that has just been deployed above.
            Some(handler.promise_attach_callback(promise_id, &callback))
        }
        AddressVersionStatus::UpToDate => base_id,
    };
    // 2. If some NEAR is required for this call (from storage staking for a new account
    //    and/or attached NEAR to the call the user wants to make), then we need to have the
    //    engine withdraw that amount of wNEAR to the router account and then have the router
    //    unwrap it into actual NEAR. In the case of storage staking, the engine contract
    //    covered the cost initially (see setup batch above), so the unwrapping also sends
    //    a refund back to the engine.
    let withdraw_id = if required_near == ZERO_YOCTO {
        setup_id
    } else {
        let withdraw_call_args = WithdrawWnearToRouterArgs {
            target: sender,
            amount: required_near,
        };
        let withdraw_call = PromiseCreateArgs {
            target_account_id: current_account_id.clone(),
            method: "withdraw_wnear_to_router".into(),
            args: borsh::to_vec(&withdraw_call_args).unwrap(),
            attached_balance: ZERO_YOCTO,
            attached_gas: WITHDRAW_GAS,
        };
        // Safety: This promise is safe. Even though this is a call from the engine account to
        // itself invoking the `call` method (which could be dangerous), the argument to `call`
        // is controlled entirely by us (not any user). This call will only execute the wnear
        // exit precompile, and only for the necessary amount. Note that this amount will always
        // be present, otherwise the user's call to the xcc precompile would have failed.
        let id = match setup_id {
            None => handler.promise_create_call(&withdraw_call),
            Some(setup_id) => handler.promise_attach_callback(setup_id, &withdraw_call),
        };
        let refund_needed = match deploy_needed {
            AddressVersionStatus::DeployNeeded { create_needed } => create_needed,
            AddressVersionStatus::UpToDate => false,
        };
        if refund_needed {
            let refund_call = PromiseCreateArgs {
                target_account_id: promise.target_account_id.clone(),
                method: "send_refund".into(),
                args: Vec::new(),
                attached_balance: ZERO_YOCTO,
                attached_gas: REFUND_GAS,
            };
            // Safety: This call is safe because the router's `send_refund` method
            // does not violate any security invariants. It only sends NEAR back to this contract.
            Some(handler.promise_attach_callback(id, &refund_call))
        } else {
            Some(id)
        }
    };
    // 3. Finally we can do the call the user wanted to do.

    // Safety: this call is safe because the promise comes from the XCC precompile, not the
    // user directly. The XCC precompile will only construct promises that target the `execute`
    // and `schedule` methods of the user's router contract. Therefore, the user cannot have
    // the engine make arbitrary calls.
    match withdraw_id {
        None => handler.promise_create_call(promise),
        Some(withdraw_id) => handler.promise_attach_callback(withdraw_id, promise),
    }
}

    pub fn send_refund(&self) -> Promise {
        let parent = self.get_parent().unwrap_or_else(env_panic);

        require_caller(&parent)
            .and_then(|_| require_no_failed_promises())
            .unwrap_or_else(env_panic);

        Promise::new(parent).transfer(REFUND_AMOUNT)

```rust
fn require_no_failed_promises() -> Result<(), Error> {
    let num_promises = env::promise_results_count();
    for index in 0..num_promises {
        // We can use deprecated `promise_result` rather than `promise_result_checked` safely here,
        // because the promise result could be received from the Aurora Engine itself,
        // and we can be sure that the len of the promise result is within bounds.
        #[allow(deprecated)]
        if env::promise_result(index) == PromiseResult::Failed {
            return Err(Error::CallbackOfFailedPromise);
        }
    }
    Ok(())
}
