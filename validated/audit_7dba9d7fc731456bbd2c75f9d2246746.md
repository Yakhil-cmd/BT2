Audit Report

## Title
Unbounded `IngressQueue` with Bypassed Capacity Guard on System Subnets Enables Ingress-Flooding DoS - (File: `rs/replicated_state/src/canister_state/queues/queue.rs`, `rs/messaging/src/scheduling/valid_set_rule.rs`)

## Summary
`IngressQueue` explicitly documents and implements no upper bound on stored messages. The sole protocol-level guard against unbounded induction — the `ingress_history_max_messages` check in `ValidSetRule::enqueue()` — is unconditionally skipped for `SubnetType::System` subnets (including the NNS). An unprivileged attacker can continuously induct messages into the NNS ingress queue, growing it without bound and delaying or starving legitimate governance, ledger, and management canister calls.

## Finding Description
`IngressQueue` at `rs/replicated_state/src/canister_state/queues/queue.rs` lines 294–295 explicitly states no upper bound exists. Its `push()` method (lines 333–349) never returns an error and unconditionally appends every message.

The only protocol-level count guard is in `ValidSetRule::enqueue()` at `rs/messaging/src/scheduling/valid_set_rule.rs` lines 250–256:

```rust
if state.metadata.own_subnet_type != SubnetType::System
    && state.metadata.ingress_history.len() >= self.ingress_history_max_messages
{
    return Err(IngressInductionError::IngressHistoryFull { ... });
}
```

The `!= SubnetType::System` condition causes the entire guard to be skipped for the NNS subnet. Execution falls through unconditionally to `state.push_ingress(ingress)` (lines 290, 334), which calls `canister.push_ingress()` → `CanisterQueues::push_ingress()` → `IngressQueue::push()` with no capacity check at any layer.

The upstream defense at `rs/http_endpoints/public/src/call.rs` lines 229–236 (`exceeds_threshold()`) is a single global pool-level soft limit with no per-sender or per-principal quota. The per-peer `exceeds_limit()` check in `rs/ingress_manager/src/ingress_handler.rs` lines 74–76 is keyed on `originator_id` (a `NodeId` for P2P gossip peers) and does not apply to direct HTTP submissions from user principals. `INGRESS_HISTORY_MAX_MESSAGES = 600,000` (`rs/limits/src/lib.rs` line 37) is the intended ceiling for application subnets; System subnets have no equivalent.

## Impact Explanation
An attacker flooding the NNS subnet can grow the inducted `IngressQueue` and ingress history without bound. Because `IngressQueue` schedules canisters in round-robin order, a large attacker-controlled backlog directly delays execution of legitimate governance proposals, ICP ledger transfers, and management canister calls. Memory for the queue is bounded only by available heap, not by any protocol-enforced message count. This matches the allowed High bounty impact: **Application/platform-level DoS or subnet availability impact not based on raw volumetric DDoS** — the exploit leverages a missing protocol-level capacity check, not raw bandwidth flooding.

## Likelihood Explanation
Medium. The attacker requires no special privileges — only the ability to submit valid signed ingress messages via the public `/api/v2/canister/{id}/call` endpoint. The global `exceeds_threshold()` soft limit can be circumvented by distributing submissions across multiple boundary nodes or timing bursts below the per-node threshold. At `MAX_INGRESS_MESSAGES_PER_BLOCK = 1000` and one block per second (`rs/limits/src/lib.rs` line 78), up to 1,000 messages/second can be inducted. For subnet methods (management canister calls), induction cost is `Free`, removing the cycles economic barrier entirely. Sustaining this for minutes produces a backlog that delays legitimate users by an unbounded number of rounds.

## Recommendation
- **Short term:** Remove the `!= SubnetType::System` exclusion from the `ingress_history_max_messages` guard in `ValidSetRule::enqueue()`, or introduce a separate per-canister ingress queue depth limit enforced inside `IngressQueue::push()` that returns an error on overflow.
- **Long term:** Introduce per-sender (principal-level) rate limiting at the HTTP ingress endpoint, analogous to the per-peer `exceeds_limit()` check that already exists for P2P gossip, to prevent a single identity or coordinated set of identities from monopolizing the ingress queue.

## Proof of Concept
1. Generate N `SignedIngress` messages targeting any canister (or the management canister for free induction) on the NNS subnet, each with a valid expiry within `MAX_INGRESS_TTL`.
2. Submit them via `/api/v2/canister/{id}/call` distributed across multiple boundary nodes to stay below the per-node `exceeds_threshold()` soft limit.
3. Each consensus round includes up to 1,000 messages. `ValidSetRule::induct_messages()` calls `enqueue()` for each.
4. Inside `enqueue()`, the `ingress_history_max_messages` branch is skipped because `own_subnet_type == SubnetType::System`.
5. `state.push_ingress()` → `canister.push_ingress()` → `CanisterQueues::push_ingress()` → `IngressQueue::push()` — no capacity check, message enqueued unconditionally.
6. A legitimate user submitting a governance proposal enters the round-robin schedule behind the attacker's backlog and waits an unbounded number of rounds.

A deterministic integration test can confirm this by constructing a `ReplicatedState` with `own_subnet_type = SubnetType::System`, inducting more than `INGRESS_HISTORY_MAX_MESSAGES` messages via `ValidSetRuleImpl::induct_messages()`, and asserting that `ingress_history.len()` exceeds the cap and `IngressQueue::size()` grows without rejection.
