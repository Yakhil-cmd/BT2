### Title
Inconsistent `processCallbackTransaction` Failure Handling Between Execution and Verification Nodes Causes Permanent Chunk Verification Failure - (`module/chunks/chunkVerifier.go`)

### Summary
The Execution Node's `blockComputer.executeProcessCallback` and the Verification Node's `ChunkVerifier.createSystemChunk` handle a failed `processCallbackTransaction` output error in opposite ways. The Execution Node silently continues and produces a sealed block containing the failed process-callback transaction; the Verification Node treats the same failure as a hard error and aborts chunk verification entirely. When the `FlowTransactionScheduler.process` transaction fails at runtime, the Execution Node produces a valid (from its perspective) block that the Verification Node can never successfully verify, causing the block to remain unsealed indefinitely — the direct Flow analog of the Cairo `assert`-vs-`return false` inconsistency.

---

### Finding Description

**Execution Node path** — `engine/execution/computation/computer/computer.go`, `executeProcessCallback`:

```go
if txn.Output().Err != nil {
    // if the process transaction fails we log the critical error but don't return an error
    // so that block execution continues and only the scheduled transactions halt
    callbackLogger.Error().
        Err(txn.Output().Err).
        Bool("critical_error", true).
        ...
        Msg("system process transaction output error")

    return nil, txnIndex, nil   // ← continues; no callbacks, no error
}
```

The Execution Node records the failed process-callback transaction in the block result (with a non-empty `ErrorMessage`) and then appends only the system-chunk transaction. Block execution succeeds and an Execution Receipt is produced.

**Verification Node path** — `module/chunks/chunkVerifier.go`, `createSystemChunk`:

```go
executionSnapshot, processOutput, err := fcv.vm.Run(callbackCtx, processTx, *snapshotTree)
if err != nil {
    return nil, nil, fmt.Errorf("failed to execute process callback transaction: %w", err)
}
if processOutput.Err != nil {
    return nil, nil, fmt.Errorf("process callback transaction failed: %w", processOutput.Err)
}
```

When the Verification Node re-executes the same process-callback transaction against the same ledger state and receives the same `processOutput.Err`, it returns a hard error from `createSystemChunk`. This propagates up through `verifyTransactionsInContext` and `Verify` as an unrecoverable error, so the Verification Node never produces a result approval for the chunk.

The two components execute the identical transaction on the identical state and reach opposite conclusions about whether execution should proceed — one continues, the other aborts.

---

### Impact Explanation

1. **Permanent block-sealing stall.** A block whose system chunk contains a failed `processCallbackTransaction` will never accumulate the required Verification Node approvals. The block is stuck in an unsealed state indefinitely, exactly mirroring the Cairo bug where cross-chain transactions remain `Pending` forever.

2. **Protocol-level correctness violation.** The Execution Node and Verification Node disagree on the validity of the same chunk. The Execution Node's receipt is correct by its own logic; the Verification Node's inability to verify it is a false negative. This breaks the fundamental invariant that honest nodes must agree on chunk validity.

3. **Cascading liveness failure.** Because Flow's sealing pipeline is sequential (each block must be sealed before the next epoch boundary can be crossed), a permanently unsealed block can stall epoch transitions and all downstream protocol operations.

4. **Scheduled transactions permanently stuck.** Any `FlowTransactionScheduler` callbacks that were pending at the height of the failing block will never be executed or marked as failed — they remain in a limbo state with no recovery path, identical in character to the Cairo `Pending` state described in the external report.

---

### Likelihood Explanation

The `FlowTransactionScheduler.process` system transaction can fail at runtime if:
- The `FlowTransactionScheduler` Cadence contract encounters an unexpected state (e.g., storage corruption, a contract upgrade that introduces a regression, or a computation-limit overrun caused by a large backlog of scheduled callbacks).
- A malicious or buggy contract author schedules a callback that causes the scheduler's `process` function to panic or exceed its gas limit, which is a realistic scenario given that any account can schedule callbacks.

The trigger is not purely theoretical: the test suite explicitly covers the `processOutputError` case and confirms the Execution Node's "log and continue" behavior, while the Verification Node's divergent hard-error path is untested for this scenario.

---

### Recommendation

Update `ChunkVerifier.createSystemChunk` in `module/chunks/chunkVerifier.go` to mirror the Execution Node's behavior when `processOutput.Err != nil`:

- Record the process-callback transaction result as `Failed: true`.
- Emit no callback-execution transactions (empty list).
- Continue to build and return the transaction list containing only `[processTx, systemTx]`.
- Do **not** return an error.

This makes the Verification Node's reconstruction of the system chunk deterministically identical to what the Execution Node produced, restoring the agreement required for chunk verification to succeed.

---

### Proof of Concept

**Step 1 — Execution Node behavior (confirmed by test).**

`engine/execution/computation/computer/computer_test.go` line 1424–1427 explicitly tests the `processOutputError` path:

```go
t.Run("process callback transaction output error", func(t *testing.T) {
    processCallbackError := fvmErrors.NewInvalidAddressErrorf(flow.EmptyAddress, "process callback output error")
    testScheduledTransactionsWithError(t, chain, []cadence.Event{}, 2, nil, processCallbackError)
})
```

Lines 1611–1628 confirm the Execution Node logs a `critical_error` and continues, producing a block with 2 transactions (process-callback + system chunk):

```go
if processOutputError != nil {
    require.NoError(t, err)
    require.Truef(t,
        testLogger.HasLogWithField("system process transaction output error", "critical_error", true), ...)
    require.Len(t, result.AllTransactionResults(), expectedTransactionCount)
    processCallbackResult := result.AllTransactionResults()[0]
    require.NotEmpty(t, processCallbackResult.ErrorMessage, ...)
    systemChunkResult := result.AllTransactionResults()[1]
    require.Empty(t, systemChunkResult.ErrorMessage, ...)
    return
}
```

**Step 2 — Verification Node behavior (hard error).**

`module/chunks/chunkVerifier.go` lines 480–486:

```go
executionSnapshot, processOutput, err := fcv.vm.Run(callbackCtx, processTx, *snapshotTree)
if err != nil {
    return nil, nil, fmt.Errorf("failed to execute process callback transaction: %w", err)
}
if processOutput.Err != nil {
    return nil, nil, fmt.Errorf("process callback transaction failed: %w", processOutput.Err)
}
```

When the Verification Node re-executes the same failing process-callback transaction, it hits line 485 and returns an error. `verifyTransactionsInContext` (line 224–226) propagates this error, and `Verify` returns it to the caller. No result approval is ever produced for the chunk.

**Step 3 — Divergence.**

| Component | `processOutput.Err != nil` | Outcome |
|---|---|---|
| Execution Node (`computer.go:653`) | Log + `return nil, txnIndex, nil` | Block sealed with failed process-callback tx |
| Verification Node (`chunkVerifier.go:484`) | `return nil, nil, error` | Chunk verification aborted; no approval |
