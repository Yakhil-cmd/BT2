### Title
Unprivileged Caller Can Suppress All Stakers' Consensus Rewards by Calling `update_rewards` with `disable_rewards=true` - (File: src/staking/staking.cairo)

### Summary
The `update_rewards` function in the Staking contract is publicly callable with no access control and accepts a caller-controlled `disable_rewards: bool` parameter. The global `last_reward_block` storage variable is written **before** the `disable_rewards` guard, so any unprivileged caller can consume a block's reward slot without distributing any rewards. Because `last_reward_block` is a single contract-wide variable, one such call per block silences consensus rewards for every staker simultaneously.

### Finding Description
`update_rewards` is defined in `StakingRewardsManagerImpl` with no role check: [1](#0-0) 

The function writes `last_reward_block` unconditionally, then branches on `disable_rewards`: [2](#0-1) 

`last_reward_block` is a single global field, not per-staker: [3](#0-2) 

The subsequent guard that prevents a second call in the same block compares strictly against this global value: [4](#0-3) 

Because the write happens before the `disable_rewards` branch, calling `update_rewards(any_active_staker, true)` at block N:
1. Sets `last_reward_block = N`.
2. Returns immediately — no rewards are calculated or credited.
3. Any subsequent call in block N (with `disable_rewards = false`) reverts on `REWARDS_ALREADY_UPDATED`.

The attacker needs only one valid staker address (publicly readable from the `stakers` vector) and enough gas to call this once per block.

### Impact Explanation
Every block in which the attacker fires this call, **all** stakers and their delegators earn zero consensus rewards. Sustained over time this constitutes a permanent (or indefinitely prolonged) freeze of unclaimed yield for the entire protocol. The `unclaimed_rewards` counter in `RewardSupplier` is never incremented for those blocks, so the tokens are never minted and never distributed — the yield is destroyed, not merely delayed. [5](#0-4) 

### Likelihood Explanation
The attack requires no privileged role, no leaked key, and no external dependency. Any address can call `update_rewards` at any time. The cost is one transaction per block (~seconds on Starknet). A motivated griefer (e.g., a competing protocol, a short-seller of STRK, or a censorship attacker) can sustain this indefinitely at negligible cost relative to the yield destroyed across all stakers.

### Recommendation
Move the `last_reward_block` write to **after** the `disable_rewards` guard so that a no-op call does not consume the block slot:

```cairo
// Only mark the block as processed when rewards are actually computed.
if disable_rewards || self.is_pre_consensus() {
    return;
}
self.last_reward_block.write(current_block_number);
// ... reward calculation continues
```

Alternatively, add access control (e.g., `only_staking_contract` or `only_attestation_contract`) to `update_rewards`, or make `last_reward_block` a per-staker mapping so one call cannot block others.

### Proof of Concept
```
// Attacker script — run once per block
loop {
    wait_for_new_block();
    let victim_staker = staking.stakers[0]; // any active staker
    staking.update_rewards(victim_staker, disable_rewards: true);
    // last_reward_block is now current_block; no rewards credited
    // Any legitimate update_rewards call this block reverts with REWARDS_ALREADY_UPDATED
}
