### Title
Missing Caller Restriction on `update_rewards` Allows Any Address to Permanently Suppress Block Rewards — (`src/staking/staking.cairo::StakingRewardsManagerImpl::update_rewards`)

### Summary

The `update_rewards` function is documented as restricted to "Only starkware sequencer" but the implementation enforces no such check. Any unprivileged address can call it with `disable_rewards: true`, consuming the global `last_reward_block` slot for the current block and permanently discarding that block's rewards for the targeted staker.

---

### Finding Description

The spec explicitly states the access control for `update_rewards` is "Only starkware sequencer": [1](#0-0) 

The implementation, however, only calls `general_prerequisites()` (which checks only the pause flag) and performs no caller identity check: [2](#0-1) 

The function writes to the **global** `last_reward_block` storage variable unconditionally — before the `disable_rewards` branch — at line 1485: [3](#0-2) 

Because `last_reward_block` is a single global slot (not per-staker), once it is written for block N, the guard at line 1454–1458 causes every subsequent call in the same block to revert with `REWARDS_ALREADY_UPDATED`. If the attacker's call used `disable_rewards: true`, the early return at line 1487–1488 fires and no rewards are distributed. The block's reward opportunity is permanently gone.

The interface definition confirms there is no access-control annotation on the trait itself: [4](#0-3) 

---

### Impact Explanation

For every block where an attacker front-runs the sequencer's legitimate `update_rewards` call with `disable_rewards: true`, the targeted staker (and their delegators) permanently lose that block's STRK and BTC consensus rewards. The rewards are never minted/transferred; they are simply skipped. This matches **High — Permanent freezing of unclaimed yield**.

---

### Likelihood Explanation

On Starknet, transaction ordering within a block is controlled by the sequencer, but the mempool is public. An attacker can submit a `update_rewards(victim, disable_rewards: true)` transaction at the start of every block. Because the gas cost is low and the call requires no stake or special role, a motivated attacker can sustain this indefinitely against any active validator.

---

### Recommendation

Add a sequencer-only (or a designated role) guard at the top of `update_rewards`, consistent with the spec:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
    self.roles.only_sequencer(); // enforce the documented access control
    ...
}
```

Alternatively, restrict the `IStakingRewardsManager` interface so it is only callable from a whitelisted address (e.g., the Starknet sequencer address or a dedicated rewards-manager role).

---

### Proof of Concept

1. Deploy the staking contract and register two stakers, `victim` and `attacker_staker` (attacker needs a valid staker only to satisfy the `INVALID_STAKER` guard; the actual exploit caller is an arbitrary EOA).
2. Advance past the consensus rewards start epoch.
3. At the beginning of each new block, the attacker EOA calls:
   ```
   update_rewards(staker_address: victim, disable_rewards: true)
   ```
   This succeeds (no caller check), writes `last_reward_block = current_block`, and returns without distributing rewards.
4. When the sequencer later tries to call `update_rewards(victim, disable_rewards: false)` in the same block, it reverts with `REWARDS_ALREADY_UPDATED`.
5. After N blocks, `victim.unclaimed_rewards_own == 0` while the model predicts `N * block_reward`. The attacker spent only gas; the victim's yield is permanently lost.

The existing test `test_update_rewards_assertions_already_consensus` (lines 3957–3973) already demonstrates that a second call in the same block always reverts regardless of the `disable_rewards` flag — confirming the one-slot-per-block nature of the gate: [5](#0-4)
