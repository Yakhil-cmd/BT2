### Title
Pool Admin Bypasses `maxAdminSpreadFeeE6` Cap via Uncapped `setPoolBinAdditionalFees` — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`MetricOmmPoolFactory.setPoolBinAdditionalFees` allows the pool admin to write arbitrary `uint16` per-bin additional fees directly to the pool with **no validation against `maxAdminSpreadFeeE6`**. The factory's fee-cap system is therefore bypassable: a pool admin can charge users a higher effective spread than the factory owner ever intended to permit.

---

### Finding Description

The factory enforces a two-layer fee cap hierarchy:

- **Hard absolute ceiling** — `HARD_MAX_SPREAD_FEE_E6 = 200_000` (20 %) is an immutable constant.
- **Configurable cap** — `maxAdminSpreadFeeE6` (set by the factory owner via `setFeeCaps`) is the runtime ceiling that the factory owner can lower at any time to restrict pool admins.

`setPoolAdminFees` correctly enforces this cap:

```solidity
// MetricOmmPoolFactory.sol L414-415
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (newAdminNotionalFeeE8 > maxAdminNotionalFeeE8) revert AdminFeeTooHigh();
``` [1](#0-0) 

However, `setPoolBinAdditionalFees` passes the caller-supplied values straight through to the pool with **no cap check at all**:

```solidity
// MetricOmmPoolFactory.sol L450-457
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
``` [2](#0-1) 

The pool's `setBinAdditionalFees` only validates the bin index, never the fee magnitudes:

```solidity
// MetricOmmPool.sol L464-474
function setBinAdditionalFees(int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external onlyFactory nonReentrant(PoolActions.SET_BIN_ADDITIONAL_FEES)
{
    if (bin < LOWEST_BIN || bin > HIGHEST_BIN) revert InvalidBinIndex(bin);
    BinState storage s = _binStates[bin];
    s.addFeeBuyE6 = addFeeBuyE6;
    s.addFeeSellE6 = addFeeSellE6;
    ...
}
``` [3](#0-2) 

During every swap these per-bin fees are **added on top of** the oracle-derived base spread fee:

```solidity
// MetricOmmPool.sol L540-541
uint256 buyFeeX64  = baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6,  ONE_X64, 1e6);
uint256 sellFeeX64 = baseFeeX64 + Math.mulDiv(binState.addFeeSellE6, ONE_X64, 1e6);
``` [4](#0-3) 

The same additive path is taken in every swap branch (e.g. `buyToken0InBinSpecifiedOut` at line 910): [5](#0-4) 

`uint16` allows values up to **65 535**, which in E6 units equals **6.5535 %**. If the factory owner has tightened `maxAdminSpreadFeeE6` to, say, 500 (0.05 %), the pool admin can still write 65 535 to any bin — a **131× overshoot** of the intended cap — and every swap touching that bin silently pays the inflated fee.

---

### Impact Explanation

Every user who swaps through an affected bin pays a higher effective spread than the factory's fee-cap system permits. The excess fee is extracted from the user's output token and credited to the LP/protocol split, constituting a **direct loss of user principal** on each swap. Because the fee is applied silently inside the swap math, users have no on-chain mechanism to detect or reject it at execution time (the `priceLimitX64` guard only stops execution at a price level, it does not bound the fee rate itself).

---

### Likelihood Explanation

The trigger requires only the pool admin — a semi-trusted role — to call `setPoolBinAdditionalFees` with a `uint16` value above the current `maxAdminSpreadFeeE6`. No external oracle manipulation, no flash loan, and no privileged factory-owner action is needed. The factory owner's ability to lower `maxAdminSpreadFeeE6` is the primary protection mechanism, and this path completely bypasses it. Any pool whose admin is compromised, malicious, or simply unaware of the cap semantics can exploit this.

---

### Recommendation

Add the same cap guard in `setPoolBinAdditionalFees` that exists in `setPoolAdminFees`:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (addFeeBuyE6  > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

Alternatively, enforce the cap inside `MetricOmmPool.setBinAdditionalFees` by reading the factory's `maxAdminSpreadFeeE6` via an interface call, so the invariant is upheld even if the factory is upgraded.

---

### Proof of Concept

1. Factory owner calls `setFeeCaps(500, 500, ...)` — capping admin spread fees at 0.05 %.
2. Pool admin calls `setPoolBinAdditionalFees(pool, 0, 65535, 65535)` — no revert, no cap check.
3. A user calls `swap(...)` routing through bin 0. Inside `_swapToken1ForToken0SpecifiedOutput`, the effective buy fee becomes `baseFeeX64 + mulDiv(65535, ONE_X64, 1e6)` — approximately 6.55 % additional fee on top of the oracle spread.
4. The user receives ~6.55 % fewer tokens than the oracle price and the 0.05 % cap would imply, with the excess silently retained in the pool and later collectible by the admin via `collectPoolFees`. [6](#0-5) [2](#0-1) [3](#0-2)
