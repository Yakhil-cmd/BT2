### Title
V1 `EvmErc20.withdrawToNear` Input Format Incompatibility Causes Permanent ERC-20 Token Loss on Failed Exit — (`etc/eth-contracts/contracts/EvmErc20.sol`, `engine-precompiles/src/native.rs`, `engine/src/engine.rs`)

---

### Summary

When the `error_refund` feature is enabled, the `ExitToNear` precompile's `parse_input` function expects the caller to embed a 20-byte `refund_address` (the sender) immediately after the 1-byte flag. The V2 contract (`EvmErc20V2`) was updated to include this field. The V1 contract (`EvmErc20`) was not. When a V1 ERC-20 token holder calls `withdrawToNear`, the precompile silently misreads the first 20 bytes of the big-endian `amount` field as the `refund_address`. For any practical token amount (< 2¹⁶⁰), those 20 bytes are all zeros — i.e., `address(0)`. If the downstream `ft_transfer` promise then fails, `refund_on_error` attempts to re-mint the burned tokens to `address(0)`. OpenZeppelin's `ERC20._mint` unconditionally reverts on a zero-address recipient, so the refund fails, `exit_to_near_precompile_callback` returns `ERR_REFUND_FAILURE`, and the user's tokens are permanently destroyed with no recovery path.

---

### Finding Description

**V1 contract — no `sender` field in precompile input:**

`EvmErc20.withdrawToNear` burns the caller's tokens and then calls the `ExitToNear` precompile with:

```
input = \x01 | amount_b (32 bytes) | recipient (variable)
``` [1](#0-0) 

**V2 contract — adds `sender` as `refund_address`:**

`EvmErc20V2.withdrawToNear` was updated to prepend the 20-byte `sender` address between the flag and the amount:

```
input = \x01 | sender (20 bytes) | amount_b (32 bytes) | recipient (variable)
``` [2](#0-1) 

**Precompile input parsing — reads 20 bytes after flag as `refund_address`:**

When `error_refund` is compiled in, `ExitToNearParams::try_from` calls `parse_input`, which strips the flag byte and reads the next 20 bytes as `refund_address`. The test fixture confirms this layout: [3](#0-2) 

For a V1 call, those 20 bytes are the high-order bytes of `amount_b`. Because token amounts are always less than 2¹⁶⁰, those bytes are `0x00…00` — i.e., `address(0)`.

**`refund_call_args` propagates the zero address into `RefundCallArgs`:** [4](#0-3) 

**`refund_on_error` calls `_mint(address(0), amount)` on the ERC-20:** [5](#0-4) 

`setup_refund_on_error_input` encodes `mint(address(0), amount)`: [6](#0-5) 

**OZ `_mint` reverts on zero address.** Both `EvmErc20` and `EvmErc20V2` inherit from OpenZeppelin's `ERC20` and call `_mint` directly: [7](#0-6) 

OpenZeppelin's `_mint` contains `require(account != address(0), "ERC20: mint to the zero address")`, so the re-mint reverts.

**Callback treats refund revert as fatal:** [8](#0-7) 

`ERR_REFUND_FAILURE` propagates to `sdk_unwrap()` in `lib.rs`, panicking the NEAR callback: [9](#0-8) 

---

### Impact Explanation

The V1 ERC-20 burn is irreversible — it executes before the precompile call and is not rolled back when the precompile misparses the input. The downstream `ft_transfer` is scheduled with a corrupted recipient and amount (the misread bytes), so it is likely to fail. When it does, the refund path attempts `mint(address(0), …)`, which reverts. The callback panics. The user's ERC-20 tokens are permanently destroyed with no NEP-141 credit and no on-chain recovery mechanism. This is a **permanent freezing / destruction of user funds**.

---

### Likelihood Explanation

- V1 `EvmErc20` contracts are present in the production repository alongside V2 and may still be deployed for legacy bridged tokens.
- The `error_refund` feature is a compile-time flag; if enabled in the production WASM build, every V1 `withdrawToNear` call is affected.
- `ft_transfer` failures are realistic: the NEAR recipient account may be unregistered with the NEP-141 contract, have insufficient storage deposit, or the contract may be paused.
- No special attacker capability is required — any ordinary token holder calling `withdrawToNear` on a V1 ERC-20 contract triggers the bug if the exit fails.

---

### Recommendation

1. **Immediate**: In `refund_on_error`, guard against `recipient_address == Address::zero()` before calling `engine.call`. If zero, skip the EVM mint and instead log an error or use an alternative recovery path.
2. **Structural**: Deprecate and migrate all V1 `EvmErc20` deployments to `EvmErc20V2` before enabling `error_refund` in any production build.
3. **Defensive**: Add a validation in `ExitToNearParams::try_from` (or `parse_input`) that rejects inputs whose parsed `refund_address` is `address(0)` when the `error_refund` feature is active, returning an explicit error rather than silently proceeding with a zero refund address.

---

### Proof of Concept

1. Deploy a V1 `EvmErc20` token (bridged NEP-141) on Aurora with `error_refund` enabled.
2. Mint tokens to `alice` (EVM address).
3. `alice` calls `EvmErc20.withdrawToNear("unregistered.near", 100)`.
4. V1 burns 100 tokens from `alice`. Precompile receives `\x01 | 0x0000…0064 (32 bytes) | "unregistered.near"`.
5. `parse_input` reads `refund_address = 0x0000000000000000000000000000000000000000`.
6. `ft_transfer("unregistered.near", …)` fails (account not registered).
7. `exit_to_near_precompile_callback` calls `refund_on_error` with `recipient_address = address(0)`.
8. `setup_refund_on_error_input` encodes `mint(0x0000…, amount)`.
9. OZ `_mint` reverts → `refund_result.status` is `Revert` → `ERR_REFUND_FAILURE` → callback panics.
10. `alice` has lost 100 tokens permanently: no ERC-20 balance, no NEP-141 credit.
