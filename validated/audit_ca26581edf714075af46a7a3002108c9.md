### Title
Malicious Signer Can Permanently Abort Signing by Submitting an Arbitrary Scalar Share — (`src/ecdsa/robust_ecdsa/sign.rs`)

### Summary
The signing phase of the robust ECDSA scheme accepts signature shares from participants as bare scalars with no proof of correct computation. A single malicious participant among the required 2t+1 signers can submit any arbitrary scalar (e.g., one computed with `-alpha_i` instead of `alpha_i`), causing the coordinator's final `sig.verify()` check to fail and permanently aborting the signing session. Because presignatures cannot be reused and the coordinator cannot identify which participant submitted the wrong share, the malicious participant can indefinitely deny signing.

---

### Finding Description

In `do_sign_coordinator`, the coordinator collects one scalar per participant and sums them blindly: [1](#0-0) 

Each participant's share is computed in `compute_signature_share` as:

```
s_me = msg_hash * presignature.alpha + beta
linearized_s_me = s_me * lambda_i
``` [2](#0-1) 

No commitment, zero-knowledge proof, or any other binding between the submitted scalar and the participant's actual presignature components (`alpha`, `beta`, `e`) is required or checked. The coordinator only performs a final aggregate check: [3](#0-2) 

If a malicious participant submits `s_i` computed with `-alpha_i` instead of `alpha_i`, the aggregated `s` is off by `2 * lambda_i * msg_hash * alpha_i`, causing `sig.verify()` to fail. The protocol returns `ProtocolError::AssertionFailed("signature failed to verify")` with no indication of which participant is responsible.

The protocol specification confirms there is no per-share validation step — the coordinator simply sums and checks the aggregate: [4](#0-3) 

---

### Impact Explanation

- Signing permanently fails for the current attempt.
- Presignatures **cannot be reused** (documented constraint), so a new presignature must be generated for any retry.
- The coordinator has no mechanism to identify which participant submitted the wrong share, so the malicious participant cannot be excluded from future signing sets without out-of-band detection.
- As long as the malicious participant is included in the 2t+1 signing set, they can abort every signing attempt indefinitely.

Impact: **High — Permanent denial of signing for honest parties.**

---

### Likelihood Explanation

- The attacker only needs to be one of the 2t+1 required signers — a threshold that is enforced to be exactly `2*max_malicious+1` with no redundancy.
- The malicious action (submitting any wrong scalar) requires no cryptographic capability beyond basic scalar arithmetic.
- The attack is completely indistinguishable from an honest computation error at the protocol level.
- The protocol explicitly requires exactly `N2 = 2t+1` participants with no fallback or redundancy: [5](#0-4) 

---

### Recommendation

Add per-share verification before aggregation. Concretely, require each participant to attach a proof of correct share computation (e.g., a Schnorr proof that `s_i` is consistent with the public presignature commitment `W_i = R^{a_i}` established during presigning). Alternatively, implement cheater-detection by having each participant broadcast their share to all others (not just the coordinator), allowing honest parties to identify and exclude the malicious one. The presign phase already establishes the public commitment `W_i`: [6](#0-5) 

This commitment can be used to verify that each submitted `s_i` is consistent with the participant's committed presignature material.
