### Title
Missing Vector Length Validation on Network-Received Slices Causes Panic in Triple Generation — (`File: src/ecdsa/ot_based_ecdsa/triples/generation.rs`)

### Summary
In `do_generation_many`, every protocol round that receives a `Vec<_>` from a peer immediately indexes into it with `for i in 0..N { ... vec[i] ... }` without first verifying that the received vector has at least `N` elements. A single malicious participant can send a short (e.g., empty) vector in any of these rounds, triggering an index-out-of-bounds **panic** that aborts the honest party's triple-generation task and permanently blocks signing.

### Finding Description

`do_generation_many<const N: usize>` is the core of both `generate_triple` (N=1) and `generate_triple_many<N>`. Throughout the function, messages received over the network are deserialized into `Vec<_>` and then immediately indexed with the compile-time constant `N` — but the actual runtime length of the received vector is never validated.

### Impact Explanation

A Rust index-out-of-bounds access (`vec[i]` where `i >= vec.len()`) is an unconditional **panic**. In an async context this unwinds the task, causing the honest party's `Protocol::poke()` loop to terminate with an error. Because triple generation is a prerequisite for both `presign` and `sign` in the OT-based ECDSA scheme, a malicious participant who triggers this panic in every session permanently denies the honest party the ability to produce signatures.

**Impact: High** — Permanent denial of signing for honest parties.

### Likelihood Explanation

Any participant admitted to a triple-generation session can craft and inject a short vector. The serialization layer (`rmp_serde`) faithfully deserializes whatever length the sender encodes; there is no schema enforcement on `Vec` length. The attack requires no cryptographic capability — only the ability to send a single malformed message in round 1. Because `generate_triple` (N=1) is the common case, even the simplest deployment is affected.

**Likelihood: High** — Trivially exploitable by any protocol participant with no special privileges.

### Recommendation

Before the `for i in 0..N` loop, validate that every received `Vec` has exactly `N` elements and return a `ProtocolError` (not a panic) if the check fails. For example, in the round-1 commitment loop:

```rust
let (from, commitments): (_, Vec<_>) = chan.recv(wait0).await?;
if commitments.len() != N {
    return Err(ProtocolError::AssertionFailed(format!(
        "expected {N} commitments from {from:?}, got {}", commitments.len()
    )));
}
for i in 0..N {
    all_commitments_vec[i].put(from, commitments[i]);
}
```

Apply the same guard to every other site that indexes a network-received `Vec` with `0..N`: the `PolynomialCommitmentsMessageMany` fields, the private-share vectors, the `big_c_j_v`/`their_phi_proofs` pairs, and the `hat_big_c_i_points`/`their_phi_proofs` pairs.

### Proof of Concept

1. Instantiate a triple-generation session with participants `[A, B]` and `N = 1` (i.e., call `generate_triple`).
2. Participant `B` (malicious) intercepts round 0 (`wait0`) and instead of sending `vec![commitment]` (length 1), sends `vec![]` (length 0).
3. Participant `A`'s executor reaches `commitments[0]` inside the `while` loop at line 182 and panics.
4. `A`'s `Protocol::poke()` returns an error; triple generation is aborted.
5. Because `B` can repeat this in every session, `A` can never complete triple generation and therefore can never sign.
