### Title
Reverted DeclareV0/DeclareV1 Simulation Incorrectly Populates `deprecated_declared_classes` in `induced_state_diff` — (`crates/apollo_rpc_execution/src/lib.rs`)

---

### Summary

`execute_transactions` unconditionally passes `deprecated_declared_class_hash = Some(class_hash)` to `induced_state_diff` for every `DeclareV0`/`DeclareV1` transaction, regardless of whether blockifier reverted the transaction. When a DeclareV1 reverts (e.g., post-execution fee check fails due to insufficient `max_fee`), the class is never written to the state, but the simulation output's `ThinStateDiff.deprecated_declared_classes` still contains the attacker-supplied class hash.

---

### Finding Description

In `execute_transactions`, the `deprecated_declared_class_hash` is extracted from the transaction struct **before** execution: [1](#0-0) 

It is then passed unconditionally to `induced_state_diff` after execution, with no check on whether `tx_execution_info_result` contains a revert: [2](#0-1) 

Inside `induced_state_diff`, the field is populated purely from the parameter, not from the actual blockifier state diff: [3](#0-2) 

When blockifier's post-execution fee check fails for a DeclareV1 (e.g., `max_fee` is too low), it calls `execution_state.abort()` to roll back the class declaration and returns `Ok(TransactionExecutionInfo { revert_error: Some(...), ... })`: [4](#0-3) 

Because `tx_execution_info_result` is `Ok(...)` (not `Err`), the error-propagation at line 755 does not short-circuit. The `state_diff` already computed at line 752 — with the phantom class hash — is bundled into `TransactionExecutionOutput.induced_state_diff` and returned to the RPC caller: [5](#0-4) 

The `simulate_transactions` public function then wraps this directly into `TransactionSimulationOutput.induced_state_diff`: [6](#0-5) 

Which is served to the caller of `starknet_simulateTransactions`: [7](#0-6) 

The existing test for `induced_state_diff` explicitly notes `// TODO(yair): Add a reverted transaction.` and is marked `#[ignore]`, confirming this revert path has never been validated: [8](#0-7) 

---

### Impact Explanation

Any caller of `starknet_simulateTransactions` receives a `state_diff` that claims a deprecated class was declared when it was not. This is a **High** impact: RPC simulation returns an authoritative-looking wrong value. Downstream tooling (wallets, explorers, SDKs) that relies on the simulated state diff to determine whether a class was declared will be misled. This does not corrupt on-chain state — it is scoped to the simulation output.

---

### Likelihood Explanation

Triggering the revert requires only that the caller submit a DeclareV1 with a `max_fee` lower than the actual execution cost. This is a fully unprivileged, zero-cost operation against the public `starknet_simulateTransactions` RPC endpoint. No special access is required.

---

### Recommendation

After calling `blockifier_tx.execute()`, inspect the result before computing `induced_state_diff`. If the result is `Ok` but `revert_error` is `Some(...)`, pass `None` as `deprecated_declared_class_hash` to `induced_state_diff` so that `deprecated_declared_classes` remains empty — matching the actual (rolled-back) state:

```rust

    blockifier_tx.execute(&mut transactional_state, &block_context);

// Only report the class as declared if the transaction was NOT reverted.
let effective_deprecated_class_hash = match &tx_execution_info_result {
    Ok(info) if info.revert_error.is_none() => deprecated_declared_class_hash,
    _ => None,
};

let state_diff =
    induced_state_diff(&mut transactional_state, effective_deprecated_class_hash)?;
```
