### Title
Gateway Stateful Validator Uses Stale `l2_gas_price` Instead of `next_l2_gas_price` for Admission Threshold, Causing Valid Transactions to Be Rejected and Invalid Transactions to Be Admitted - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

`StatefulTransactionValidator::validate_resource_bounds` reads `block_header.l2_gas_price` (the price used for transactions in the **current** committed block) as the reference for the L2 gas price admission threshold. The correct reference is `block_header.next_l2_gas_price` (the EIP-1559-derived price that will govern the **next** block). The code itself acknowledges this with an explicit TODO. Because the two values diverge every block, the gateway either rejects valid transactions (when price is falling) or admits transactions that will fail during block building (when price is rising).

### Finding Description

`GatewayFixedBlockSyncStateClient::get_block_info_from_sync_client` constructs a `BlockInfo` by reading `block_header.l2_gas_price.price_in_fri` and placing it in `strk_gas_prices.l2_gas_price`. It never reads `block_header.next_l2_gas_price`. [1](#0-0) 

`validate_resource_bounds` then calls `get_block_info()` and extracts `strk_gas_prices.l2_gas_price`, labelling it `previous_block_l2_gas_price`. The TODO comment explicitly flags that `next_l2_gas_price` should be used instead: [2](#0-1) 

This value is passed to `validate_tx_l2_gas_price_within_threshold`, which rejects any `AllResources` transaction whose `l2_gas.max_price_per_unit` is below `min_gas_price_percentage% × previous_block_l2_gas_price`: [3](#0-2) 

The `next_l2_gas_price` is computed by the consensus orchestrator via EIP-1559 logic (`calculate_next_base_gas_price`) and stored in `BlockHeaderWithoutHash.next_l2_gas_price`: [4](#0-3) [5](#0-4) 

When block utilization is below the gas target, `next_l2_gas_price < l2_gas_price`. When above, `next_l2_gas_price > l2_gas_price`. The gateway's threshold is anchored to the wrong value in both directions.

The batcher's block builder receives the actual `next_l2_gas_price` via `ProposalInit` and constructs the real `BlockInfo` from it: [6](#0-5) 

The blockifier's `check_fee_bounds` then compares the transaction's `max_price_per_unit` against this real price: [7](#0-6) 

### Impact Explanation

**Scenario A — price falling (`next_l2_gas_price < l2_gas_price`, utilization below target):**

A user who correctly prices their transaction at `next_l2_gas_price` (the price they will actually pay) is rejected at the gateway with `GAS_PRICE_TOO_LOW` because the threshold is computed from the higher `l2_gas_price`. The transaction is valid and would succeed in the next block, but it never reaches the mempool. This is the direct analog of the external bug: a hardcoded/stale reference value causes valid operations to be rejected.

**Scenario B — price rising (`next_l2_gas_price > l2_gas_price`, utilization above target):**

A user who prices their transaction at `l2_gas_price` passes the gateway threshold (which is `min_gas_price_percentage% × l2_gas_price`). The gateway's own blockifier validation also passes because `run_validate_entry_point` reuses the same stale `block_info`: [8](#0-7) 

The transaction enters the mempool. When the batcher attempts to include it, `check_fee_bounds` compares `tx.max_price_per_unit` against the real `next_l2_gas_price` and rejects it with `MaxGasPriceTooLow`. The transaction occupies mempool space indefinitely and is never sequenced.

Both scenarios match the "High" impact category: **gateway admission rejects valid transactions or accepts transactions that will fail before sequencing**.

### Likelihood Explanation

The L2 gas price changes every block. With the default `min_gas_price_percentage = 100`, any single-block decrease in L2 gas price (which occurs whenever utilization is below the gas target) causes Scenario A. Any single-block increase causes Scenario B. The EIP-1559 mechanism guarantees these fluctuations are continuous and normal. The divergence between `l2_gas_price` and `next_l2_gas_price` is not a rare edge case; it is the expected steady-state behavior of the fee market. [9](#0-8) 

### Recommendation

In `validate_resource_bounds`, replace the read of `block_header.l2_gas_price` with `block_header.next_l2_gas_price`. The `GatewayFixedBlockStateReader` trait and its `GatewayFixedBlockSyncStateClient` implementation should expose `next_l2_gas_price` from `BlockHeaderWithoutHash` so the stateful validator can use the correct forward-looking price as the admission reference. This directly resolves the acknowledged TODO.

### Proof of Concept

1. Block N is committed with `l2_gas_price = 1000 fri` and `next_l2_gas_price = 900 fri` (utilization was below gas target, so EIP-1559 decreased the price).
2. A user submits an `AllResources` invoke transaction with `l2_gas.max_price_per_unit = 900 fri`.
3. The gateway calls `get_block_info()`, which returns `strk_gas_prices.l2_gas_price = 1000 fri` (from `block_header.l2_gas_price`).
4. With `min_gas_price_percentage = 100`, the threshold is `100% × 1000 = 1000 fri`.
5. `900 < 1000` → `validate_tx_l2_gas_price_within_threshold` returns `Err(GAS_PRICE_TOO_LOW)`.
6. The transaction is rejected at the gateway.
7. Block N+1 is built with `l2_gas_price = 900 fri` (= the `next_l2_gas_price` from block N). The transaction would have passed `check_fee_bounds` (`900 >= 900`) and been included successfully.

The user's valid transaction is permanently rejected by the gateway despite being correctly priced for the next block, mirroring the `FloorFromChainlink` pattern where a stale reference value causes valid operations to be silently dropped.

    // TODO(Arni): Consider running this validation for all gas prices.
    fn validate_tx_l2_gas_price_within_threshold(
        &self,
        tx_resource_bounds: ValidResourceBounds,
        previous_block_l2_gas_price: NonzeroGasPrice,
    ) -> StatefulTransactionValidatorResult<()> {
        match tx_resource_bounds {
            ValidResourceBounds::AllResources(tx_resource_bounds) => {
                let tx_l2_gas_price = tx_resource_bounds.l2_gas.max_price_per_unit;
                let gas_price_threshold_multiplier =
                    Ratio::new(self.config.min_gas_price_percentage.into(), 100_u128);
                let threshold = (gas_price_threshold_multiplier
                    * previous_block_l2_gas_price.get().0)
                    .to_integer();
                if tx_l2_gas_price.0 < threshold {
                    return Err(StarknetError {
                        // We didn't have this kind of an error.
                        code: StarknetErrorCode::UnknownErrorCode(
                            "StarknetErrorCode.GAS_PRICE_TOO_LOW".to_string(),
                        ),
                        message: format!(
                            "Transaction L2 gas price {tx_l2_gas_price} is below the required \
                             threshold {threshold}.",
                        ),
                    });
                }
            }
            ValidResourceBounds::L1Gas(_) => {
                // No validation required for legacy transactions.
            }
        }
        Ok(())
    }

pub fn calculate_next_base_gas_price(
    price: GasPrice,
    gas_used: GasAmount,
    gas_target: GasAmount,
    min_gas_price: GasPrice,
) -> GasPrice {
    let versioned_constants = VersionedConstants::latest_constants();
    assert!(
        gas_target < versioned_constants.max_block_size,
        "Gas target must be lower than max block size."
    );
    assert!(gas_target.0 > 0, "Gas target must be greater than zero.");
    assert!(
        versioned_constants.gas_price_max_change_denominator > 0,
        "Denominator constant must be greater than zero."
    );

    // If the current price is below the minimum, apply a gradual adjustment and return early.
    // This allows the price to increase by at most 1/MIN_GAS_PRICE_INCREASE_DENOMINATOR per block.
    if price < min_gas_price {
        let max_increase = price.0 / MIN_GAS_PRICE_INCREASE_DENOMINATOR;
        let adjusted = price.0 + max_increase;
        // Cap at min_gas_price to avoid overshooting
        let adjusted_price = adjusted.min(min_gas_price.0);
        info!(
            "Fee Market: Price {} below minimum gas price {}, adjusted price: {} )",
            price.0, min_gas_price.0, adjusted_price
        );
        return GasPrice(adjusted_price);
    }

    // Use U256 to avoid overflow, as multiplying a u128 by a u64 remains within U256 bounds.
    let gas_delta = U256::from(gas_used.0.abs_diff(gas_target.0));
    let gas_target_u256 = U256::from(gas_target.0);
    let price_u256 = U256::from(price.0);

    // Calculate price change by multiplying first, then dividing. This avoids the precision loss
    // that occurs when dividing before multiplying.
    let denominator =
        gas_target_u256 * U256::from(versioned_constants.gas_price_max_change_denominator);
    let price_change = (price_u256 * gas_delta) / denominator;

    let adjusted_price_u256 =
        if gas_used > gas_target { price_u256 + price_change } else { price_u256 - price_change };

    // Sanity check: ensure direction of change is correct
    assert!(
        gas_used > gas_target && adjusted_price_u256 >= price_u256
            || gas_used <= gas_target && adjusted_price_u256 <= price_u256
    );

    // Price should not realistically exceed u128::MAX, bound to avoid theoretical overflow.
    let adjusted_price = u128::try_from(adjusted_price_u256).unwrap_or(u128::MAX);
    GasPrice(max(adjusted_price, min_gas_price.0))
}
