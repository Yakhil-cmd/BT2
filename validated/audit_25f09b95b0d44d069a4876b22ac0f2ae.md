### Title
Multi-NFT Royalty Amount Under-Calculated in WalletConnect Offer Confirmation Dialog - (`File: packages/gui/src/electron/commands/parseCommandDisplay.ts`)

---

### Summary

`formatAmountWithRoyalties` in `parseCommandDisplay.ts` incorrectly divides the fungible amount by the number of NFTs before applying each royalty percentage. This causes the "Total Amount with Royalties" shown in the WalletConnect approval dialog to be under-reported by a factor of N (number of NFTs), causing a user to approve a `take_offer` or `create_offer_for_ids` command believing they will spend less than the transaction actually deducts.

---

### Finding Description

In `formatAmountWithRoyalties`:

```ts
const splitAmount = amount / BigInt(royaltyPercentages.length); // divides by NFT count
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
const totalAmount = amount + royaltyAmount;
``` [1](#0-0) 

The intent is to compute the total fungible cost including all NFT creator royalties. However, the code first divides `amount` by the number of NFTs (`royaltyPercentages.length`), then applies each royalty to that split amount. The correct formula is to apply each royalty to the **full** amount:

```
correct:  royaltyAmount = Σ (amount × royaltyPct_i / 10000)
actual:   royaltyAmount = Σ ((amount / N) × royaltyPct_i / 10000)
                        = (1/N) × correct
```

For N=2 NFTs with royalties 500 (5%) and 10 (0.1%) on 0.0001 XCH:
- **Displayed** (buggy): `0.00010255 XCH` (royalty = 2.55% of amount)
- **Actual on-chain**: `0.0001051 XCH` (royalty = 5.1% of amount)

This is confirmed by the existing test, which encodes the buggy behavior as the expected output: [2](#0-1) 

The `amountWithRoyalties` value is rendered in the WalletConnect confirmation dialog as "Total Amount with Royalties": [3](#0-2) 

`parseCommandDisplay` is called directly in the WalletConnect approval flow in `main.tsx` before the user sees the confirmation dialog: [4](#0-3) 

It is also called in the regular (non-WalletConnect) command approval flow: [5](#0-4) 

---

### Impact Explanation

A user approving a multi-NFT offer via WalletConnect sees a "Total Amount with Royalties" that is under-reported by a factor of N. The actual transaction deducts the correct (higher) amount from the user's wallet. The user is caused to approve a transaction they would not have approved had the correct total been displayed. This is a direct approval-confusion impact on XCH/CAT balances via WalletConnect state spoofing.

---

### Likelihood Explanation

Any WalletConnect-connected DApp can send a `take_offer` or `create_offer_for_ids` command containing two or more NFTs with royalties. No special privileges are required beyond an established WalletConnect session, which is a normal user action. The discrepancy grows linearly with the number of NFTs, making it easy to construct a scenario where the displayed amount is materially lower than the actual deduction.

---

### Recommendation

Remove the `splitAmount` division. Apply each royalty percentage to the full `amount`:

```ts
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (amount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
const totalAmount = amount + royaltyAmount;
``` [6](#0-5) 

---

### Proof of Concept

**Setup:** WalletConnect DApp sends `chia_wallet.take_offer` for an offer where the user receives 2 NFTs (royalties 500 and 10) in exchange for 100,000,000 mojos (0.0001 XCH).

**Buggy display in confirmation dialog:**
- `amount`: `0.0001 XCH`
- `amountWithRoyalties`: `0.00010255 XCH` (splitAmount = 50,000,000; royalty = 2,500,000 + 50,000 = 2,550,000 mojos)

**Actual on-chain deduction:**
- Royalty for NFT1: 100,000,000 × 500 / 10,000 = 5,000,000 mojos
- Royalty for NFT2: 100,000,000 × 10 / 10,000 = 100,000 mojos
- Total: 105,100,000 mojos = **0.0001051 XCH**

The user approves seeing `0.00010255 XCH` but loses `0.0001051 XCH` — a ~2.3% discrepancy that scales with the number of NFTs and their royalty rates. [6](#0-5) [7](#0-6)

function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],
): string | undefined {
  if (royaltyPercentages.length === 0 || line.kind === 'nft') {
    return undefined;
  }

  const splitAmount = amount / BigInt(royaltyPercentages.length);
  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;

  if (line.kind === 'xch') {
    return mojoToChiaLocaleString(totalAmount);
  }

  return mojoToCATLocaleString(totalAmount);
}

function withRoyaltyTotals(
  items: DisplayWalletDeltaItemWithKey[],
  amounts: Record<string, bigint>,
  oppositeSideLines: DisplayWalletDeltaItem[],
): DisplayWalletDeltaItem[] {
  const royaltyPercentages = royaltyPercentagesForSide(oppositeSideLines);

  return items.map(({ key, line }) => {
    if (line.kind === 'nft') {
      return line;
    }

    const amount = amounts[key];
    const amountWithRoyalties = amount ? formatAmountWithRoyalties(line, amount, royaltyPercentages) : undefined;

    return amountWithRoyalties ? { ...line, amountWithRoyalties } : line;
  });
}
