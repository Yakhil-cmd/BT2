### Title
Gateway `skip_stateful_validations` Bypasses `__validate__` Signature Check via Overly Broad Mempool Presence Check — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary
The `skip_stateful_validations` function in the gateway's stateful validator is designed to skip the account's `__validate__` entry point for an invoke transaction with nonce=1 when a deploy_account is pending (UX feature). However, the guard condition uses `account_tx_in_pool_or_recent_block`, which returns `true` for **any** pending transaction for the account — not specifically a deploy_account. An unprivileged attacker can exploit this to submit an invoke transaction with an invalid signature that bypasses the `__validate__` entry point at the gateway level and is admitted to the mempool.

### Finding Description

The `skip_stateful_validations` function fires when:
1. The incoming transaction is an `Invoke`
2. `tx.nonce() == Nonce(Felt::ONE)`
3. `account_nonce == Nonce(Felt::ZERO)`
4. `mempool_client.account_tx_in_pool_or_recent_block(sender_address)` returns `true` [1](#0-0) 

When all four conditions hold, the function returns `true` (skip), which propagates into `run_validate_entry_point`: [2](#0-1) 

The `execution_flags.validate` is set to `!skip_validate`, so when `skip_validate=true`, the blockifier's `StatefulValidator::perform_validations` returns early without calling `__validate__`: [3](#0-2) 

The intended invariant is: "the account has a deploy_account pending, so `__validate__` cannot run yet." The code comment acknowledges the broader interpretation: *"it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."* [4](#0-3) 

However, `account_tx_in_pool_or_recent_block` returns `true` for **any** account transaction in the pool or a recently committed block — not only deploy_account transactions: [5](#0-4) 

**Attack scenario:**

1. Account `X` exists on-chain with `nonce=0` (freshly deployed, or any account with nonce=0).
2. The legitimate owner submits `invoke(nonce=0)` for account `X`; it passes `__validate__` normally and enters the mempool.
3. An attacker observes account `X` in the mempool and submits `invoke(nonce=1)` for account `X` with an **arbitrary/invalid signature**.
4. Gateway stateful validation: `account_nonce=0`, `tx_nonce=1`, `account_tx_in_pool_or_recent_block=true` → `skip_stateful_validations` returns `true`.
5. `run_validate_entry_point` is called with `execution_flags.validate=false`; the blockifier's `perform_validations` returns early without running `__validate__`.
6. The invalid invoke(nonce=1) is admitted to the mempool without any signature verification.

The stateless validator only checks signature **length**, not cryptographic validity: [6](#0-5) 

The `validate_by_mempool` call (which runs before `skip_stateful_validations`) only checks for duplicate hashes and fee escalation — not signature validity: [7](#0-6) 

### Impact Explanation

An unprivileged attacker can submit invoke transactions with invalid signatures that are admitted to the mempool, bypassing the account's `__validate__` entry point at the gateway level. This matches the **High** impact category: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

Concrete consequences:
- **Nonce slot squatting**: The attacker's invalid invoke(nonce=1) occupies the nonce=1 slot for account `X`. The legitimate owner's invoke(nonce=1) would be rejected as a duplicate nonce (unless fee escalation applies), disrupting the account's transaction ordering.
- **Mempool pollution**: Invalid transactions fill mempool capacity, potentially evicting legitimate transactions.
- **Fee drain on execution**: During block execution, the blockifier runs `__validate__` with `strict_nonce_check=true` and `charge_fee=true`. The invalid transaction fails `__validate__`, but the fee (up to `max_fee`) is charged to account `X`'s balance — allowing the attacker to drain the victim's fee token balance without controlling the account's private key.

### Likelihood Explanation

The condition is easy to trigger: any account with a pending transaction at nonce=0 and no pending transaction at nonce=1 is vulnerable. This is a common state for active accounts. The attacker needs no privileged access — only the ability to submit transactions to the public gateway endpoint.

### Recommendation

Replace the overly broad `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a **deploy_account** transaction is pending for the sender address. The mempool should expose a dedicated query such as `has_pending_deploy_account(address)` that only returns `true` when a `DeployAccount` transaction is in the pool or was recently committed. This preserves the intended UX feature while closing the bypass for accounts that have only invoke transactions pending.

```rust
// Current (too broad):
return mempool_client
    .account_tx_in_pool_or_recent_block(tx.sender_address())
    .await ...

// Recommended:
return mempool_client
    .has_pending_deploy_account(tx.sender_address())
    .await ...
```

### Proof of Concept

1. Deploy account `X` on-chain (nonce=0). Fund it with STRK.
2. Submit `invoke_v3(sender=X, nonce=0, calldata=[...], signature=[valid])` to the gateway. It passes `__validate__` and enters the mempool.
3. Construct `invoke_v3(sender=X, nonce=1, calldata=[arbitrary], signature=[0x1337, 0x1337])` (invalid signature).
4. Submit the malformed transaction to the gateway.
5. Gateway stateful validation: `account_nonce=0`, `tx_nonce=1`, `account_tx_in_pool_or_recent_block(X)=true` → `skip_stateful_validations` returns `true` → `execution_flags.validate=false` → `__validate__` is not called.
6. The malformed transaction is accepted by the gateway and added to the mempool.
7. Observe: the mempool now contains an invoke(nonce=1) for account `X` with an invalid signature, admitted without any cryptographic signature check.
8. During block execution, the transaction fails `__validate__` and the fee is charged to account `X`. [1](#0-0) [8](#0-7) [9](#0-8)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}

    pub fn perform_validations(&mut self, tx: AccountTransaction) -> StatefulValidatorResult<()> {
        // Deploy account transaction should be fully executed, since the constructor must run
        // before `__validate_deploy__`. The execution already includes all necessary validations,
        // so they are skipped here.
        // Declare transaction should also be fully executed - otherwise, if we only go through
        // the validate phase, we would miss the check that the class was not declared before.
        match tx.tx {
            ApiTransaction::DeployAccount(_) | ApiTransaction::Declare(_) => self.execute(tx),
            ApiTransaction::Invoke(_) => {
                let tx_context = Arc::new(self.tx_executor.block_context.to_tx_context(&tx));
                tx.perform_pre_validation_stage(self.state(), &tx_context)?;
                if !tx.execution_flags.validate {
                    return Ok(());
                }

                // `__validate__` call.
                let (_optional_call_info, actual_cost) = self.validate(&tx, tx_context.clone())?;

                // Post validations.
                PostValidationReport::verify(
                    &tx_context,
                    &actual_cost,
                    tx.execution_flags.charge_fee,
                )?;

                Ok(())
            }
        }
    }
