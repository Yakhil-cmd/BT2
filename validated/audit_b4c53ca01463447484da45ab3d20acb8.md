### Title
`run_validate_entry_point` Collapses All `StatefulValidatorError` Variants Into `ValidateFailure`, Masking Infrastructure Errors as Validation Failures — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

`run_validate_entry_point` uses a single blanket `.map_err` to convert every possible `StatefulValidatorError` variant — including `StateError` (storage/infrastructure failures) and `TransactionPreValidationError` (fee/nonce failures) — into `KnownStarknetErrorCode::ValidateFailure`. This is the direct Sequencer analog of the Solidity `catch Error(string memory)` pattern: just as that catch clause silently swallows custom errors and arithmetic panics, this single `map_err` arm swallows infrastructure errors and pre-validation errors by relabelling them all as `ValidateFailure`. Clients that receive `ValidateFailure` conclude their transaction's `__validate__` entry point failed and do not retry; valid transactions that were rejected only because of a transient storage error are therefore permanently lost from the client's perspective.

### Finding Description

In `extract_state_nonce_and_run_validations`, after the gateway-level nonce/fee pre-checks pass, `run_validate_entry_point` is called. Inside it, `blockifier_validator.validate(account_tx)` returns a `StatefulValidatorResult<()>` whose error type is `StatefulValidatorError`:

```rust
// crates/blockifier/src/blockifier/stateful_validator.rs:30-40
pub enum StatefulValidatorError {
    StateError(#[from] StateError),                          // storage / infra
    TransactionExecutionError(#[from] TransactionExecutionError),
    TransactionExecutorError(#[from] TransactionExecutorError), // infra
    TransactionPreValidationError(#[from] TransactionPreValidationError), // nonce/fee
}
```

The gateway converts this result with a single, undiscriminating `map_err`:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs:351-354
.map_err(|e| StarknetError {
    code: StarknetErrorCode::KnownErrorCode(KnownStarknetErrorCode::ValidateFailure),
    message: e.to_string(),
})?;
```

Every variant — regardless of whether it represents a genuine `__validate__` failure, a storage read error, or a fee/nonce pre-validation failure — is collapsed into `ValidateFailure`.

**Concrete error-type misclassifications:**

| Actual error | Correct code | Emitted code |
|---|---|---|
| `StateError` (DB/storage unavailable) | `InternalError` | `ValidateFailure` |
| `TransactionExecutorError` (bouncer/state) | `InternalError` | `ValidateFailure` |
| `TransactionPreValidationError::TransactionFeeError` | `InsufficientMaxFee` / `InsufficientResourceBounds` | `ValidateFailure` |
| `TransactionPreValidationError::InvalidNonce` | `InvalidTransactionNonce` | `ValidateFailure` |

The `TransactionPreValidationError` path is reachable inside `run_validate_entry_point` because `perform_validations` calls `perform_pre_validation_stage` for Invoke transactions, and calls the full `execute` path (which also runs `perform_pre_validation_stage`) for DeployAccount and Declare. The gateway's own pre-checks use the *previous* block's L2 gas price; the blockifier inside `run_validate_entry_point` uses the *current* block's gas price (via `block_info.block_number.unchecked_next()`). A transaction whose gas price passes the gateway check but fails the blockifier check will be rejected with `ValidateFailure` instead of the correct fee-related code.

The `StateError` path is reachable whenever the storage backend returns an error during class-hash lookup, nonce read, or compiled-class fetch inside the blocking task.

### Impact Explanation

The RPC layer in `write_api_error.rs` maps `ValidateFailure` to a `validation_failure` JSON-RPC error, while unknown/internal codes map to `unexpected_error`. Clients and SDKs treat `validation_failure` as a permanent, non-retryable rejection of the transaction itself. When a `StateError` is misclassified as `ValidateFailure`, a valid transaction that failed only because of a transient storage error is permanently abandoned by the client. This is a gateway admission path that rejects valid transactions before sequencing, matching the **High** impact criterion.

### Likelihood Explanation

- **`StateError` path**: triggered by any transient storage unavailability during the blocking validation task; not attacker-controlled but occurs in production under load or during storage hiccups.
- **`TransactionPreValidationError` path**: triggered by any user whose transaction's L2 gas price passes the gateway's previous-block threshold but falls below the blockifier's current-block threshold — a normal occurrence when gas prices rise between blocks.

### Recommendation

Replace the single blanket `map_err` with a discriminating match that preserves the semantic meaning of each error variant:

```rust
.map_err(|e| match e {
    StatefulValidatorError::StateError(_)
    | StatefulValidatorError::TransactionExecutorError(_) => {
        StarknetError::internal_with_signature_logging(
            "Internal error during validate entry point",
            &executable_tx.signature(),
            e,
        )
    }
    StatefulValidatorError::TransactionPreValidationError(ref pre_val_err) => {
        // map to the same codes used in run_pre_validation_checks
        pre_validation_error_to_starknet_error(pre_val_err)
    }
    StatefulValidatorError::TransactionExecutionError(_) => StarknetError {
        code: StarknetErrorCode::KnownErrorCode(KnownStarknetErrorCode::ValidateFailure),
        message: e.to_string(),
    },
})?;
```

### Proof of Concept

1. Arrange for the storage backend to return a `StateError` on the first class-hash read inside the blocking task (e.g., by injecting a fault into the state reader).
2. Submit a valid Invoke transaction through the gateway.
3. `run_validate_entry_point` spawns the blocking task; `blockifier_validator.validate` calls `state.get_class_hash_at(storage_address)`, which returns `StateError`.
4. `StatefulValidatorError::StateError` propagates out of the blocking task.
5. The blanket `map_err` converts it to `StarknetError { code: ValidateFailure, ... }`.
6. The gateway returns HTTP 400 with `StarknetErrorCode.VALIDATE_FAILURE` to the client.
7. The client's SDK treats this as a permanent validation failure and does not retry.
8. The valid transaction is never sequenced.

The same sequence applies to the fee-price mismatch case: submit a transaction with `max_price_per_unit` equal to the previous block's L2 gas price (passes `validate_resource_bounds`) but below the current block's price (fails `check_fee_bounds` inside `perform_pre_validation_stage`).     pub fn perform_validations(&mut self, tx: AccountTransaction) -> StatefulValidatorResult<()> {
        // Deploy account transaction should be fully executed, since the constructor must run
        // before `__validate_deploy__`. The execution already includes all necessary validations,
        // so they are skipped here.
        // Declare transaction should also be fully executed - otherwise, if we only go through
        // the validate phase, we would miss the check that the class was not declared before.
        match tx.tx {
            ApiTransaction::DeployAccount(_) | ApiTransaction::Declare(_) => self.execute(tx),
            ApiTransaction::Invoke(_) => {
                let tx_context = Arc::new(self.tx_executor.block_context.to_tx_context(&tx));
                tx.perform_pre_validation_stage(self.state(), &tx_context)?;
                if !tx.execution_flags.validate {
                    return Ok(());
                }

                // `__validate__` call.
                let (_optional_call_info, actual_cost) = self.validate(&tx, tx_context.clone())?;

                // Post validations.
                PostValidationReport::verify(
                    &tx_context,
                    &actual_cost,
                    tx.execution_flags.charge_fee,
                )?;

                Ok(())
            }
        }
    }

pub(crate) fn starknet_error_to_invoke_error(error: StarknetError) -> JsonRpcError<String> {
    let StarknetErrorCode::KnownErrorCode(known_error_code) = error.code else {
        return unexpected_error(error.message);
    };
    match known_error_code {
        KnownStarknetErrorCode::DuplicatedTransaction => DUPLICATE_TX,
        // EntryPointNotFoundInContract is not thrown in __execute__ since that is
        // considered as a reverted transaction. It is also not thrown in __validate__ since
        // every error there is considered a ValidateFailure. This means that if
        // EntryPointNotFoundInContract is thrown then it failed because it couldn't find
        // __validate__ or __execute__ and that means the contract is not an account contract.
        KnownStarknetErrorCode::EntryPointNotFoundInContract => NON_ACCOUNT,
        KnownStarknetErrorCode::InsufficientAccountBalance => INSUFFICIENT_ACCOUNT_BALANCE,
        KnownStarknetErrorCode::InsufficientMaxFee => INSUFFICIENT_MAX_FEE,
        KnownStarknetErrorCode::InvalidTransactionNonce => INVALID_TRANSACTION_NONCE,
        KnownStarknetErrorCode::InvalidTransactionVersion => UNSUPPORTED_TX_VERSION,
        KnownStarknetErrorCode::ValidateFailure => validation_failure(error.message),
        _ => unexpected_error(error.message),
    }
}
