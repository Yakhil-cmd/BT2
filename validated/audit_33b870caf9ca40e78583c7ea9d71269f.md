### Title
`protocolUnpausePool` Sets Pause Level to 1 Instead of 0, Leaving Pool Permanently Paused After Protocol Unpause - (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`MetricOmmPoolFactory.protocolUnpausePool` calls `setPause(1)` instead of `setPause(0)`. Because `_checkNotPaused()` reverts on any non-zero pause level, the pool's `swap` function remains blocked after the protocol owner "unpauses" it. The pool is stuck at admin-pause level (1) indefinitely until the pool admin separately calls `unpausePool`.

---

### Finding Description

The pool uses a three-level pause system:

| Level | Meaning |
|---|---|
| 0 | Active (swaps allowed) |
| 1 | Paused by admin |
| 2 | Paused by protocol |

`_checkNotPaused` blocks swaps at any non-zero level:

```solidity
function _checkNotPaused() internal view {
    if (pauseLevel != 0) revert PoolPaused();
}
```

The protocol owner's unpause function is:

```solidity
function protocolUnpausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 2) revert InvalidPauseTransition(cur, 1);
    IMetricOmmPoolFactoryActions(pool).setPause(1);   // ← sets to 1, not 0
}
```

It sets the pool to level **1** (admin-paused), not level **0** (active). The pool remains paused after the call. This is the direct analog of the external bug: the function does the opposite of what its name and intent describe.

Compare with the correct admin-side pair:

```solidity
function pausePool(...)   { ... setPause(1); }   // 0 → 1  ✓
function unpausePool(...) { ... setPause(0); }   // 1 → 0  ✓
```

The protocol-side pair is broken:

```solidity
function protocolPausePool(...)   { ... setPause(2); }   // 0|1 → 2  ✓
function protocolUnpausePool(...) { ... setPause(1); }   // 2 → 1  ✗ (should be 0)
```

---

### Impact Explanation

After `protocolUnpausePool` is called on a pool that was active (level 0) before being protocol-paused:

1. Pool was at level 0 → protocol pauses → level 2.
2. Protocol owner calls `protocolUnpausePool` → level 1.
3. `swap` still reverts with `PoolPaused` because `pauseLevel != 0`.
4. The pool admin never paused the pool, yet the pool is now stuck at admin-pause level. The admin must call `unpausePool` to restore level 0.

If the pool admin is unaware, or is unavailable, swaps remain permanently blocked. This is broken core pool functionality: the primary revenue-generating operation (`swap`) is disabled even after the authorized protocol owner explicitly attempted to restore it.

---

### Likelihood Explanation

Triggered every time the protocol owner calls `protocolUnpausePool` on a pool that was at level 0 before being protocol-paused (the common case). No special conditions, no attacker required — the protocol owner's own action produces the broken state.

---

### Recommendation

Change `setPause(1)` to `setPause(0)` in `protocolUnpausePool`:

```solidity
function protocolUnpausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 2) revert InvalidPauseTransition(cur, 0);
-   IMetricOmmPoolFactoryActions(pool).setPause(1);
+   IMetricOmmPoolFactoryActions(pool).setPause(0);
}
```

If the design intent is to restore the pool to admin-paused state (level 1) when the protocol unpauses, a separate `protocolUnpauseToAdminPause` function should be added and the existing function must still set level 0 to match its name.

---

### Proof of Concept

1. Pool is deployed and active (`pauseLevel == 0`).
2. Protocol owner calls `protocolPausePool(pool)` → `pauseLevel` becomes 2.
3. Protocol owner calls `protocolUnpausePool(pool)` → `pauseLevel` becomes **1** (not 0).
4. Any user calls `swap(...)` → `_checkNotPaused()` fires → `PoolPaused` revert.
5. Pool is permanently paused until the pool admin calls `unpausePool`, which the admin may not know is required. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
