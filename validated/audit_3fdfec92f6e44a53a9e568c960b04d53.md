### Title
Missing Lower-Bound Validation of `threshold` in `assert_sign_inputs` Allows Signing with Cryptographically Insufficient Threshold — (`src/frost/mod.rs`)

---

### Summary

`assert_sign_inputs` in `src/frost/mod.rs` validates only the upper bound of `threshold` (must not exceed participant count) but omits the lower-bound check (`threshold >= 2`) that is correctly enforced in `assert_key_invariants` in `src/dkg.rs`. A malicious coordinator can pass `threshold = 1` (or `0`) to FROST signing without triggering any initialization error. The protocol then proceeds with a cryptographically insufficient threshold, producing an unusable or incorrect signature that honest parties receive as the protocol output.

---

### Finding Description

`assert_key_invariants` in `src/dkg.rs` enforces:

```rust
if threshold < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
}
``` [1](#0-0) 

`assert_sign_inputs` in `src/frost/mod.rs` performs no equivalent lower-bound check. It only validates the upper bound:

```rust
if threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
``` [2](#0-1) 

The full validation block in `assert_sign_inputs` is: [3](#0-2) 

`MaxMalicious` and `ReconstructionLowerBound` are plain newtypes over `usize` with no invariant enforcement at construction time: [4](#0-3) 

Because `derive_more::From` is derived, `ReconstructionLowerBound::from(1usize)` is valid and passes directly into `assert_sign_inputs` without rejection.

---

### Impact Explanation

When `threshold = 1` is supplied to FROST signing:

- Only one participant's nonce and signature share are required.
- Lagrange interpolation at zero with a single point returns that participant's share directly (coefficient = 1).
- The aggregated `s` value equals a single participant's `s_i = nonce_i + challenge * key_share_i`.
- The resulting signature `(R, s)` satisfies `s·G = R_i + challenge·public_share_i`, **not** `R + challenge·public_key` (which is the sum of all shares).
- The signature is cryptographically invalid and unusable.

Unlike the robust ECDSA coordinator path, which internally calls `sig.verify` and aborts on failure: [5](#0-4) 

the FROST signing path has no equivalent internal verification step visible in the codebase, meaning the invalid output is returned to honest callers as the protocol result. This constitutes **corruption of signing outputs — honest parties receive an unusable cryptographic output** — matching the High impact tier.

---

### Likelihood Explanation

The coordinator is the party that constructs and passes signing arguments, including `threshold`. A malicious coordinator (or a library caller who misconfigures the parameter) can trivially pass `threshold = 1`. No cryptographic capability or key material is required. The missing check is a single missing guard that is already present in the DKG path, making accidental misconfiguration equally plausible alongside deliberate abuse.

---

### Recommendation

Add the same lower-bound guard to `assert_sign_inputs` that already exists in `assert_key_invariants`:

```rust
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

This should be inserted immediately after the participant-count check in `assert_sign_inputs`. [6](#0-5) 

---

### Proof of Concept

```rust
pub fn assert_sign_inputs(
    participants: &[Participant],
    threshold: impl Into<ReconstructionLowerBound>,
    me: Participant,
    coordinator: Participant,
) -> Result<ParticipantList, InitializationError> {
    let threshold = threshold.into();
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // validate threshold
    if threshold.value() > participants.len() {
        return Err(InitializationError::ThresholdTooLarge {
            threshold: threshold.value(),
            max: participants.len(),
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }
    Ok(participants)
}
```
