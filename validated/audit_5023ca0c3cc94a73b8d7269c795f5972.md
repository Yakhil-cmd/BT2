### Title
Pool Admin Bypasses Global Spread Fee Cap via Unconstrained `setPoolBinAdditionalFees` - (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary

`MetricOmmPoolFactory.setPoolBinAdditionalFees` forwards `addFeeBuyE6` and `addFeeSellE6` directly to the pool with no upper-bound check, while every other pool-admin fee setter is explicitly capped. A pool admin can set per-bin additional fees to `uint16.max = 65535` (≈ 6.55% in E6 units), which is additive to the global spread fee in swap math, allowing the effective per-bin fee to exceed the protocol's hard 20% cap (`HARD_MAX_SPREAD_FEE_E6 = 200_000`).

### Finding Description

The factory enforces hard caps on all global spread and notional fees:

- `setPoolAdminFees` checks `newAdminSpreadFeeE6 > maxAdminSpreadFeeE6` and reverts with `AdminFeeTooHigh`.
- `setFeeCaps` checks against `HARD_MAX_SPREAD_FEE_E6 = 200_000` (20%) and `HARD_MAX_NOTIONAL_FEE_E8 = 1_000_000` (1%). [1](#0-0) [2](#0-1) 

However, `setPoolBinAdditionalFees` performs **no cap check** at all:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external
    override
    nonReentrant
    onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
``` [3](#0-2) 

The pool's `setBinAdditionalFees` also performs no cap check — it only validates the bin index range: [4](#0-3) 

In swap math, the bin additional fee is **added on top of** the oracle-derived base spread fee:

```solidity
uint256 buyFeeX64 = baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6);
uint256 sellFeeX64 = baseFeeX64 + Math.mulDiv(binState.addFeeSellE6, ONE_X64, 1e6);
``` [5](#0-4) 

This `currBinBuyFeeX64` / `currBinSellFeeX64` is then used directly in all four swap-in-bin functions to compute the fee charged to the trader: [6](#0-5) 

### Impact Explanation

`uint16.max = 65535` in E6 units equals **6.5535%**. Added on top of a global spread fee already at the 20% hard cap, the effective per-bin fee reaches **26.5535%**. Traders swapping through that bin pay this inflated fee; the excess accrues to LPs in the bin (not to the protocol's tracked fee split). This is a direct loss of trader principal above the protocol-guaranteed cap, and it is not visible in the `poolFeeConfig` split that `collectPoolFees` uses — the extra fee silently stays in the bin balances as LP surplus.

### Likelihood Explanation

The pool admin is semi-trusted and has a direct, single-call path to trigger this: `setPoolBinAdditionalFees(pool, curBinIdx, 65535, 65535)`. No timelock, no co-signer, no factory-owner approval is required. The README explicitly states the finding is valid if "the Pool admin can bypass the caps … and it qualifies for Medium or higher severity." [7](#0-6) 

### Recommendation

Add a cap check in `setPoolBinAdditionalFees` analogous to the check in `setPoolAdminFees`. A reasonable bound is `maxAdminSpreadFeeE6` (already enforced for the global admin spread component), or introduce a dedicated `maxBinAdditionalFeeE6` constant:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (addFeeBuyE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

### Proof of Concept

1. Factory is deployed with `HARD_MAX_SPREAD_FEE_E6 = 200_000` (20%).
2. Pool admin calls `factory.setPoolAdminFees(pool, 200_000, 0)` — sets global admin spread to the 20% cap (passes the cap check).
3. Pool admin calls `factory.setPoolBinAdditionalFees(pool, 0, 65535, 65535)` — no revert, sets per-bin additional fees to `uint16.max` (6.5535%).
4. A trader calls `swap` on bin 0. Inside `MetricOmmPool.swap`, the effective buy fee is `baseFeeX64 + mulDiv(65535, ONE_X64, 1e6)` — approximately 26.5% total, exceeding the 20% hard cap.
5. The trader's input is inflated by the uncapped bin fee; the excess stays in the bin as LP surplus, bypassing the protocol's fee accounting in `poolFeeConfig`. [8](#0-7) [3](#0-2) [5](#0-4)
