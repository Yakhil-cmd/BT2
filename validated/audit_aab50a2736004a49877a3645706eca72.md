### Title
ERC-20 Tokens Burned Without Refund on Omni `ft_transfer_call` Rejection When `error_refund` Is Disabled — (`engine-precompiles/src/native.rs`)

---

### Summary

When the `error_refund` feature is not compiled in (which is the case in the standard production build), the Omni path of `exit_erc20_token_to_near` schedules a bare `PromiseArgs::Create` (no callback) for `ft_transfer_call`. If the downstream NEP-141 contract's `ft_on_transfer` rejects the transfer and returns the tokens to aurora, the ERC-20 tokens that were already burned in the EVM are never re-minted. The user permanently loses their ERC-20 balance while the NEP-141 tokens are stranded in aurora's account.

---

### Finding Description

**Step 1 — Production build does not include `error_refund`.**

`Makefile.toml` sets:

```
CARGO_FEATURES_BUILD = "contract"
``` [1](#0-0) 

Neither `engine/Cargo.toml` nor `engine-precompiles/Cargo.toml` list `error_refund` as a default feature. [2](#0-1) [3](#0-2) 

**Step 2 — Omni path sets `transfer_near_args = None`.**

In `exit_erc20_token_to_near`, the `Some(Message::Omni(msg))` branch explicitly returns `None` for `transfer_near_args`: [4](#0-3) 

**Step 3 — `callback_args` collapses to `default()`, so no callback is scheduled.**

Back in `ExitToNear::run`, when `error_refund` is absent, `refund` is hardcoded to `None`: [5](#0-4) 

`ExitToNearPrecompileCallbackArgs` derives `Default` with both fields as `Option`, so `default()` is `{refund: None, transfer_near: None}`: [6](#0-5) 

The guard at line 470 therefore selects `PromiseArgs::Create` — no callback: [7](#0-6) 

**Step 4 — No refund path exists in the callback even if it were scheduled.**

`exit_to_near_precompile_callback` only re-mints tokens when `args.refund` is `Some(...)`: [8](#0-7) 

Since `error_refund` is disabled, `args.refund` is always `None`, so the `else if` branch is never taken.

---

### Impact Explanation

The ERC-20 burn happens inside the EVM transaction (before any NEAR promise executes). Once the precompile returns successfully, the ERC-20 balance is reduced. If `ft_transfer_call` on the NEP-141 contract fires and `ft_on_transfer` rejects (returning the full amount to aurora), the NEP-141 tokens land back in aurora's account with no mechanism to re-mint the ERC-20 tokens. The user's ERC-20 balance is permanently zeroed while the equivalent NEP-141 tokens are stranded in aurora's account — a permanent freezing of funds.

---

### Likelihood Explanation

- The production WASM is built with `contract` only; `error_refund` is never compiled in.
- The Omni path is a supported, documented production path (used by Omni Bridge integrations).
- `ft_on_transfer` rejection is a normal NEP-141 outcome (slippage exceeded, insufficient liquidity, invalid message format, contract paused, etc.) — it does not require any attacker capability beyond submitting a transaction whose downstream call fails.
- Any user of the Omni exit path is exposed; no privileged access is required.

---

### Recommendation

1. **Short-term:** Always schedule the `exit_to_near_precompile_callback` for `ft_transfer_call` paths, regardless of the `error_refund` feature flag, so that a failed promise result can be detected and the ERC-20 tokens re-minted.
2. **Long-term:** Make `error_refund` a default feature, or restructure the callback-scheduling logic so that the Omni `ft_transfer_call` path unconditionally attaches a callback with enough information to perform a re-mint on failure.

---

### Proof of Concept

Call sequence (no admin access required):

1. User holds a bridged ERC-20 token (e.g., `usdc.near` mirrored as an ERC-20 on Aurora).
2. User (or attacker controlling the user's EVM key) calls the ERC-20 contract's `withdrawToNear` with input `flag=0x1 | amount | receiver_account_id:{"some":"omni_msg"}`.
3. The ERC-20 contract burns the tokens and calls the `ExitToNear` precompile.
4. `exit_erc20_token_to_near` matches `Some(Message::Omni(...))`, returns `transfer_near_args = None`.
5. `callback_args == default()` → `PromiseArgs::Create(ft_transfer_call)` is emitted with no callback.
6. The NEAR runtime executes `ft_transfer_call` on the NEP-141 contract; `ft_on_transfer` rejects (returns full amount to aurora).
7. No callback fires; no re-mint occurs.
8. Assert: user's ERC-20 balance is 0; NEP-141 balance of aurora is unchanged (tokens returned to it); user has no recovery path.
