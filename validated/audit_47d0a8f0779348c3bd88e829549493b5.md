### Title
Gateway `skip_stateful_validations` Bypasses `__validate__` Signature Check Based on Any Mempool Entry, Not Specifically a `deploy_account` — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The gateway's `skip_stateful_validations` function skips the `__validate__` entry-point call (which performs signature verification) for an invoke transaction with `nonce=1` whenever `account_tx_in_pool_or_recent_block` returns `true` for the sender address. The check is too broad: it returns `true` for **any** transaction from that address in the mempool, not exclusively a `deploy_account`. An unprivileged attacker who observes a pending `deploy_account` for address A can submit a forged invoke (nonce=1, arbitrary calldata, invalid signature) for A, have it admitted to the mempool without signature verification, and thereby either inject a fee-free reverting transaction into a block or front-run and displace the legitimate user's nonce-1 invoke.

---

When this returns `true`, `run_validate_entry_point` is called with `skip_validate = true`, which sets `ExecutionFlags { validate: false }` and therefore the blockifier's `validate_tx` returns `Ok(None)` immediately without executing `__validate__`: [2](#0-1) [3](#0-2) 

The mempool's `account_tx_in_pool_or_recent_block` returns `true` for **any** transaction from the address, not only a `deploy_account`:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [4](#0-3) 

The code comment acknowledges the ambiguity — "it means that either it has a deploy_account transaction **or transactions with future nonces that passed validations**" — but does not account for the case where an attacker deliberately places a transaction in the pool to trigger the skip for a subsequent forged invoke.

The `validate_by_mempool` call that precedes `skip_stateful_validations` only checks nonce validity and duplicate hashes; it does not verify signatures: [5](#0-4) 

---

### Impact Explanation

**Impact: High — Mempool/gateway admission accepts an invalid transaction before sequencing.**

A forged invoke transaction (arbitrary calldata, invalid/absent signature) with `nonce=1` is admitted to the mempool without any signature check. At block-execution time the blockifier does call `__validate__` with the correct flags, so the forged invoke reverts. However:

1. **Nonce slot poisoning**: The forged invoke occupies the nonce-1 slot for address A. The legitimate user's real invoke with nonce=1 is subsequently rejected by the mempool as a duplicate nonce, blocking the user's transaction.
2. **Fee-free block pollution**: Because the account does not yet exist on-chain (the `deploy_account` has not been executed), the fee-transfer step after revert has no balance to draw from. The attacker injects a reverting transaction into a block at zero cost.

---

### Likelihood Explanation

The mempool is observable (transactions are gossiped). Any attacker watching for `deploy_account` submissions can immediately race to submit a forged nonce-1 invoke for the same address. No privileged access, special key material, or prior on-chain state is required. The only prerequisite is that the target address has a pending transaction in the mempool and its on-chain nonce is 0, which is the exact state that exists between a user submitting `deploy_account` and it being included in a block.

---

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a check that specifically confirms a **`deploy_account`** transaction is pending for the address. The mempool should expose a dedicated query such as `deploy_account_in_pool_or_recent_block(address)` that inspects the transaction type, rather than returning `true` for any transaction type. Alternatively, the gateway can inspect the transaction type of the pending mempool entry before deciding to skip `__validate__`.

---

### Proof of Concept

**Setup**: Address A has no on-chain contract (account_nonce = 0 in state). A legitimate user submits a `deploy_account` transaction for A; it is accepted into the mempool.

**Attack**:

1. Attacker observes the pending `deploy_account` for address A in the mempool.
2. Attacker crafts an `RpcInvokeTransactionV3` for address A with `nonce = 1`, arbitrary `calldata`, and a random/invalid `signature`.
3. Attacker submits the forged invoke to the gateway.
4. Gateway stateful validation path:
   - `get_nonce_from_state(A)` → `Nonce(0)` (account not deployed yet).
   - `validate_state_preconditions`: nonce check passes (`0 ≤ 1 ≤ 0 + max_allowed_nonce_gap`).
   - `validate_by_mempool`: passes (no duplicate nonce=1 exists yet).
   - `skip_stateful_validations`: `tx.nonce() == 1 && account_nonce == 0` → calls `account_tx_in_pool_or_recent_block(A)` → **returns `true`** (the deploy_account is in the pool) → returns `true` (skip).
   - `run_validate_entry_point` called with `skip_validate = true` → `ExecutionFlags { validate: false }` → `__validate__` is **never called**.
5. Forged invoke is admitted to the mempool.
6. Legitimate user's real invoke with nonce=1 is now rejected by the mempool as `DuplicateNonce`.
7. When the block is built: `deploy_account` executes (account deployed), then the forged invoke executes — `__validate__` is called, signature fails, transaction reverts. No fee is collected because the account had no pre-funded balance. [6](#0-5) [7](#0-6) [4](#0-3)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
