### Title
`skip_stateful_validations` Admits Invoke Transactions Without Signature Verification When Any Account Transaction Is in the Mempool - (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validation path skips the `__validate__` entry-point call (which performs signature verification) for any Invoke transaction with nonce 1 when `account_tx_in_pool_or_recent_block` returns `true`. That helper returns `true` for **any** transaction in the pool for the account — not exclusively a `DeployAccount` transaction. An attacker can therefore submit an Invoke transaction with nonce 1 carrying an invalid or unauthorized signature, have it admitted to the mempool, and bypass the account's custom validation logic at the gateway layer.

### Finding Description

`skip_stateful_validations` is designed to improve UX for the deploy-account + invoke flow: when a user submits both transactions simultaneously, the invoke cannot be validated because the account does not yet exist. The function therefore skips `__validate__` for an Invoke with nonce 1 when the account has a pending transaction.

The condition is:

```rust

    return mempool_client
        .account_tx_in_pool_or_recent_block(tx.sender_address())
        .await
        ...
}

`account_tx_in_pool_or_recent_block` is implemented as:

```rust
// crates/apollo_mempool/src/mempool.rs  lines 697-700
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [2](#0-1) 

This returns `true` for **any** transaction type in the pool (Invoke nonce 0, Declare nonce 0, or DeployAccount nonce 0). The comment in the code claims this is sufficient because "it means that either it has a deploy_account transaction or transactions with future nonces that passed validations." However, the check does not distinguish between transaction types, so a valid Invoke nonce 0 (or Declare nonce 0) in the pool is enough to trigger the skip for a nonce 1 Invoke with an entirely invalid signature.

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` sets `execution_flags.validate = false`:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs  lines 310-312
let strict_nonce_check = false;
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [3](#0-2) 

Inside `StatefulValidator::perform_validations`, when `!tx.execution_flags.validate` is true, execution returns immediately after `perform_pre_validation_stage` without ever calling `__validate__`:

```rust
// crates/blockifier/src/blockifier/stateful_validator.rs  lines 78-81

if !tx.execution_flags.validate {
    return Ok(());
}

`perform_pre_validation_stage` checks nonce, fee bounds, and proof facts — but **not the signature**:

```rust
// crates/blockifier/src/transaction/account_transaction.rs  lines 355-372

    &self, state: &mut S, tx_context: &TransactionContext,

    self.validate_proof_facts(&tx_context.block_context, state)?;

}

### Impact Explanation

An attacker can submit an Invoke transaction with nonce 1 carrying an invalid or unauthorized signature and have it admitted to the mempool without any signature check. The gateway's admission decision is authoritative for mempool inclusion. The transaction will eventually fail during block execution when the blockifier calls `__validate__`, but it has already been accepted as a valid transaction by the gateway, wasting sequencer resources and polluting the mempool. An attacker can repeat this pattern (one deploy_account tx unlocks unlimited invalid nonce-1 invokes) to mount a sustained mempool-spam attack.

This matches the impact category: **High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

### Likelihood Explanation

The trigger is unprivileged and requires only:
1. Submitting a valid `DeployAccount` (or any valid nonce-0 transaction for an existing account) to place the account in the mempool.
2. Submitting an Invoke with nonce 1 and an invalid/unauthorized signature.

Both steps are available to any external user via the public RPC endpoint. No special privileges or insider access are required.

### Recommendation

Change `account_tx_in_pool_or_recent_block` to a type-specific check, or add a separate helper that returns `true` only when a `DeployAccount` transaction for the given address is pending in the pool. For example:

```rust
async fn skip_stateful_validations(...) -> ... {
    if let ExecutableTransaction::Invoke(...) = tx {
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            // Only skip if a DeployAccount is specifically pending.
            return mempool_client
                .deploy_account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                ...
        }
    }
