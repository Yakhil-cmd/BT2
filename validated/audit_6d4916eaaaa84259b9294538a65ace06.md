### Title
Silo Mode Whitelist Activation Silently Redirects In-Flight ERC-20 Bridge Deposits to Fallback Address, Causing Permanent User Fund Loss - (`engine/src/engine.rs`)

---

### Summary

In Silo mode, when the Address whitelist is enabled or a user's EVM address is removed from the whitelist, any concurrent `ft_on_transfer` callback for that address silently mints ERC-20 tokens to the `erc20_fallback_address` instead of the intended recipient. Because `ft_on_transfer` returns `"0"` (success) in this case, the NEP-141 contract does not refund the tokens to the sender, causing permanent fund loss with no error signal to the user.

---

### Finding Description

The `receive_erc20_tokens` function in `engine/src/engine.rs` checks whether the intended recipient is in the Address whitelist. If not, it silently redirects the mint to the `erc20_fallback_address`:

```rust
if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
    && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
{
    recipient = fallback_address;
}
``` [1](#0-0) 

`is_allow_receive_erc20_tokens` delegates to `is_address_allowed`, which returns `true` only when the Address whitelist is disabled OR the address is explicitly listed:

```rust
pub fn is_allow_receive_erc20_tokens<I: IO + Copy>(io: &I, address: &Address) -> bool {
    is_address_allowed(io, address)
}

fn is_address_allowed<I: IO + Copy>(io: &I, address: &Address) -> bool {
    let list = Whitelist::init(io, WhitelistKind::Address);
    !list.is_enabled() || list.is_exist(address)
}
``` [2](#0-1) 

The `ft_on_transfer` callback returns `"0"` (success) whenever `receive_erc20_tokens` does not return an `Err`. A silent redirect to the fallback address is not an error — it returns `Ok(Some(result))` — so the NEP-141 contract is told the transfer succeeded and does **not** refund the tokens to the sender:

```rust
let amount_to_return = if let Err(_err) = &result {
    args.amount.as_u128()
} else {
    // Everything is ok, so return 0.
    0
};
``` [3](#0-2) 

The whitelist status is a mutable, owner-controlled parameter. The owner can enable the Address whitelist at any time via `set_whitelist_status` or remove a specific address via `remove_entry_from_whitelist`: [4](#0-3) 

In NEAR Protocol, `ft_transfer_call` is asynchronous: the user's call on the NEP-141 contract and the resulting `ft_on_transfer` callback on Aurora are processed in separate receipts. There is always a window between the two. If the owner enables the whitelist or removes the user's address during this window, the user's tokens are silently minted to the fallback address instead of their intended EVM address.

The `erc20_fallback_address` is itself a mutable parameter set via `set_erc20_fallback_address`. In the test suite it is initialized to `Address::zero()`: [5](#0-4) 

If the fallback address is the zero address or any uncontrolled address, the redirected tokens are permanently unrecoverable.

---

### Impact Explanation

A user who initiates a legitimate NEP-141 → ERC-20 bridge transfer loses their tokens permanently. The tokens are minted to the `erc20_fallback_address` with no error returned to the user or the originating NEP-141 contract. The user's ERC-20 balance at their intended address remains zero. Recovery requires the fallback address owner to voluntarily return the tokens; if the fallback is the zero address, recovery is impossible. This constitutes **direct theft of user funds** (Critical).

---

### Likelihood Explanation

Silo mode is a documented production feature. The Address whitelist is designed to be toggled by the owner for access-control purposes — enabling it is a routine, legitimate administrative action. NEAR's asynchronous receipt model guarantees a non-zero window between `ft_transfer_call` and `ft_on_transfer`. In any active Silo deployment with ongoing bridge activity, enabling the whitelist or removing an address will race with in-flight transfers. The likelihood is **medium**: it requires no attacker, only a routine owner action during normal bridge activity.

---

### Recommendation

When the intended recipient is not whitelisted, `receive_erc20_tokens` should return an error (causing `ft_on_transfer` to return the full token amount to the sender) rather than silently redirecting to the fallback address. This mirrors the NEP-141 standard's refund mechanism and ensures users never lose funds due to a whitelist configuration change. The fallback redirect, if retained at all, should only apply to cases where the user explicitly opts in or where the operator has a documented recovery process.

---

### Proof of Concept

1. Deploy Aurora Engine in Silo mode; set `erc20_fallback_address = Address::zero()`.
2. Deploy a NEP-141 token and its ERC-20 mirror on Aurora via `deploy_erc20_token`.
3. Enable the Address whitelist (`set_whitelist_status { kind: Address, active: true }`); add `user_addr` to it.
4. User calls `ft_transfer_call` on the NEP-141 contract with `msg = hex(user_addr)` and `receiver_id = aurora`.
5. Before the `ft_on_transfer` receipt is processed, owner calls `remove_entry_from_whitelist` for `user_addr` (or calls `set_whitelist_status { active: true }` if the whitelist was previously disabled).
6. `ft_on_transfer` is processed on Aurora → `receive_erc20_tokens` evaluates `is_allow_receive_erc20_tokens(&io, &user_addr)` → returns `false` → `recipient = Address::zero()`.
7. ERC-20 tokens are minted to `Address::zero()`; `ft_on_transfer` returns `"0"` (success).
8. NEP-141 contract does not refund; user's NEP-141 balance is reduced and their ERC-20 balance at `user_addr` is `0`.
9. Tokens are permanently burned. [6](#0-5) [7](#0-6) [8](#0-7)

    pub fn receive_erc20_tokens<P: PromiseHandler>(
        &mut self,
        token: &AccountId,
        args: &FtOnTransferArgs,
        current_account_id: &AccountId,
        handler: &mut P,
    ) -> Result<Option<SubmitResult>, ContractError> {
        let amount = args.amount.as_u128();
        // Parse message to determine recipient
        let mut recipient = {
            // The message should contain the recipient EOA address.
            let message = args.msg.strip_prefix("0x").unwrap_or(&args.msg);
            // Recipient - 40 characters (Address in hex without '0x' prefix)
            if message.len() < 40 {
                return Err(ParseOnTransferMessageError::WrongMessageFormat.into());
            }
            let mut address_bytes = [0; 20];
            hex::decode_to_slice(&message[..40], &mut address_bytes)
                .map_err(|_| ParseOnTransferMessageError::WrongMessageFormat)?;
            Address::from_array(address_bytes)
        };

        if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
            && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
        {
            recipient = fallback_address;
        }

        let erc20_token = get_erc20_from_nep141(&self.io, token)?;
        let erc20_admin_address = current_address(current_account_id);
        let result = self
            .call(
                &erc20_admin_address,
                &erc20_token,
                Wei::zero(),
                setup_receive_erc20_tokens_input(&recipient, amount),
                u64::MAX,
                Vec::new(), // TODO: are there values we should put here?
                Vec::new(),
                handler,
            )
            .and_then(submit_result_or_err)?;

        sdk::log!("Mint {amount} ERC-20 tokens for: {}", recipient.encode());

        // Return SubmitResult so that it can be accessed in standalone engine.
        // This is used to help with the indexing of bridge transactions.
        Ok(Some(result))
    }

pub fn set_erc20_fallback_address<I: IO>(io: &mut I, address: Option<Address>) {
    let key = erc20_fallback_address_key();

        io.remove_storage(&key);

#[named]
pub fn ft_on_transfer<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();
        let mut engine: Engine<_, _> = Engine::new(
            predecessor_address(&predecessor_account_id),
            current_account_id.clone(),
            io,
            env,
        )?;

        sdk::log!("Call ft_on_transfer");

        let args: FtOnTransferArgs = read_json_args(&io)?;
        let result = if predecessor_account_id == get_connector_account_id(&io)? {
            engine.receive_base_tokens(&args)
        } else {
            engine.receive_erc20_tokens(
                &predecessor_account_id,
                &args,
                &current_account_id,
                handler,
            )
        };

        #[allow(clippy::used_underscore_binding)]
        let amount_to_return = if let Err(_err) = &result {
            sdk::log!("Error in ft_on_transfer: {_err:?}");
            // An error occurred, so we need to return the amount of tokens to the sender.
            args.amount.as_u128()
        } else {
            // Everything is ok, so return 0.
            0
        };

        let output = crate::prelude::format!("\"{amount_to_return}\"");
        io.return_output(output.as_bytes());

        // In case of an error, we just return Ok(None) to avoid a panic in the contract. It's ok
        // because in case of an error, we already returned the amount of tokens to the sender.
        Ok(result.unwrap_or(None))
    })
}
