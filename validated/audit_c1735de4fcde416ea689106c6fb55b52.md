### Title
Missing Access Control on `update_rewards` Allows Any Caller to Permanently Freeze Block Rewards for All Stakers - (File: src/staking/staking.cairo)

### Summary
`update_rewards` in the `StakingRewardsManagerImpl` is intended to be callable only by the Starknet sequencer (per the protocol spec), but the implementation contains no access control check. Any public caller can invoke it with `disable_rewards: true` to consume the global `last_reward_block` slot without distributing any rewards, permanently destroying that block's yield for every staker in the protocol.

### Finding Description
The `IStakingRewardsManager::update_rewards` function is documented in `docs/spec.md` as restricted to "Only starkware sequencer": [1](#0-0) 

However, the implementation in `src/staking/staking.cairo` only calls `general_prerequisites()`, which checks that the contract is not paused and the caller is not the zero address — no sequencer or role check is present: [2](#0-1) 

The function uses a single **global** `last_reward_block` storage variable to gate reward distribution — one call per block for the entire contract: [3](#0-2) 

When `disable_rewards` is `true`, the function updates `last_reward_block` to the current block and returns immediately without distributing any rewards: [4](#0-3) 

Because `last_reward_block` is a single global value (not per-staker), a single call with `disable_rewards: true` for **any** staker address blocks all subsequent `update_rewards` calls for every staker in that block: [5](#0-4) 

### Impact Explanation
Block rewards that are skipped are permanently lost — the block number can never be revisited. An attacker calling `update_rewards(any_active_staker, disable_rewards: true)` at the start of every block causes every staker and every delegation pool to receive zero consensus-era block rewards indefinitely. This constitutes **permanent freezing of unclaimed yield** for all stakers and delegators in the protocol.

This maps to the allowed impact: **High — Permanent freezing of unclaimed yield or unclaimed royalties**.

### Likelihood Explanation
The function is public, requires no tokens, no stake, and no privileged role. The only cost is the gas for one transaction per block. Any motivated attacker (e.g., a competing validator, a griefing party) can sustain this attack indefinitely at negligible cost. The attack is also invisible to stakers until they notice their `unclaimed_rewards_own` is not growing.

### Recommendation
Add an access control check to `update_rewards` that restricts callers to the authorized sequencer address (or a designated role), consistent with the specification:

```rust
fn update_rewards(...) {
    self.roles.only_sequencer(); // or equivalent role check
    ...
}
```

Alternatively, if the sequencer address is not stored as a role, add a dedicated storage field and assert `get_caller_address() == self.sequencer_address.read()`.

### Proof of Concept
1. Consensus rewards are active (`!self.is_pre_consensus()`).
2. Staker A is active with non-zero balance.
3. Attacker (any address, no stake required) calls:
   ```
   update_rewards(staker_address: staker_A, disable_rewards: true)
   ```
   at the first transaction of every block.
4. `last_reward_block` is set to the current block number.
5. Any legitimate sequencer call to `update_rewards` for any staker in the same block reverts with `REWARDS_ALREADY_UPDATED`.
6. All stakers receive zero block rewards for that block. Repeated every block → permanent yield freeze.

The `last_reward_block` global storage variable and the missing role check are the two necessary vulnerable components: [6](#0-5) [7](#0-6)
