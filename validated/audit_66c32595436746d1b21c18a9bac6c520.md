### Title
Staker Can Frontrun Delegator's Pool Entry to Increase Commission With No Slippage Protection - (File: `src/pool/pool.cairo`, `src/staking/staking.cairo`)

### Summary

`enter_delegation_pool` and `add_to_delegation_pool` accept no `max_commission` parameter. A staker holding an active `commission_commitment` can call `set_commission` to raise their commission up to `max_commission` in the same block as a delegator's pool-entry transaction, causing the delegator to receive permanently less yield than they observed on-chain before submitting.

### Finding Description

`set_commission_commitment` lets a staker set a `max_commission` that is **greater than or equal to** the current commission:

```cairo
// src/staking/staking.cairo:769-770
let current_commission = staker_pool_info.commission();
assert!(current_commission <= max_commission, "{}", Error::MAX_COMMISSION_TOO_LOW);
``` [1](#0-0) 

While the commitment is active, `update_commission` only requires the new value to be `<= max_commission` and `!= old_commission` — it may be **higher** than the current commission:

```cairo
// src/staking/staking.cairo:1583-1589
if self.is_commission_commitment_active(:commission_commitment) {
    assert!(
        commission <= commission_commitment.max_commission, ...
    );
    assert!(commission != old_commission, ...);
``` [2](#0-1) 

The developers themselves note this gap in a comment directly above `set_commission_commitment`:

```cairo
// src/staking/staking.cairo:745-746
/// **Note**: Current commission increase safeguards still allow for sudden commission
/// changes.
``` [3](#0-2) 

Neither `enter_delegation_pool` nor `add_to_delegation_pool` accept a `max_commission` guard:

```cairo
// src/pool/interface.cairo:30-32
fn enter_delegation_pool(
    ref self: TContractState, reward_address: ContractAddress, amount: Amount,
);
``` [4](#0-3) 

```cairo
// src/pool/interface.cairo:58-60
fn add_to_delegation_pool(
    ref self: TContractState, pool_member: ContractAddress, amount: Amount,
) -> Amount;
``` [5](#0-4) 

### Impact Explanation

A delegator who observes a 5% commission and submits `enter_delegation_pool` can have the staker frontrun that transaction with `set_commission(max_commission)` — e.g., 50% — in the same block. The delegator's funds are locked at the higher commission rate immediately. All future rewards are silently reduced by the difference. Because the exit window imposes a mandatory waiting period before funds can be recovered, the delegator suffers theft of unclaimed yield for the entire lock-up duration. This maps directly to the **High: Theft of unclaimed yield** impact category.

### Likelihood Explanation

The attack requires only that the staker has a currently active `commission_commitment` with `max_commission > current_commission`. This is a normal, publicly visible on-chain state. Any staker who has ever called `set_commission_commitment` to signal trustworthiness to delegators simultaneously gains the ability to execute this frontrun. No privileged access, leaked key, or external dependency is needed — only the staker's own address and a pending delegator transaction in the mempool.

### Recommendation

Add a `max_commission: Commission` parameter to both `enter_delegation_pool` and `add_to_delegation_pool`. At the start of each function, assert that the current commission fetched from the staking contract does not exceed the caller-supplied bound:

```cairo
let current_commission = self.get_commission_from_staking_contract();
assert!(current_commission <= max_commission, "commission exceeds max_commission");
```

This mirrors the `maxFeePct` guard in the referenced report and ensures delegators cannot be silently subjected to a higher commission than they accepted.

### Proof of Concept

1. Staker stakes and calls `set_commission_commitment(max_commission: 5000, expiration_epoch: current + 52)` while current commission is 200 (2%).
2. Delegator reads the pool, sees 2% commission, and submits `enter_delegation_pool(reward_address, amount)`.
3. Staker observes the pending transaction and frontruns it with `set_commission(5000)` (50%).
4. Delegator's transaction executes: funds are accepted, but the commission stored in the pool contract is now 50%.
5. Every subsequent epoch, `update_rewards_from_staking_contract` distributes rewards after deducting 50% commission instead of 2%, permanently reducing the delegator's yield.
6. The delegator must wait through the full exit window before recovering principal, during which all accrued rewards are subject to the inflated commission. [6](#0-5) [7](#0-6)

            // Prerequisites and asserts.
            self.general_prerequisites();

                    );
            } else {
