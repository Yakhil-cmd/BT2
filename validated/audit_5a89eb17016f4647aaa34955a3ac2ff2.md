### Title
Unchecked `transferFrom` Boolean Return Value in XCC Precompile Allows Free Cross-Contract Calls — (`engine-precompiles/src/xcc.rs`)

---

### Summary

The Cross-Contract Call (XCC) precompile calls `transferFrom` on the wNEAR ERC-20 contract to collect payment from the caller before scheduling a NEAR cross-contract call. The code checks the EVM-level exit reason (`Succeed`/`Revert`/`Error`/`Fatal`) but never decodes or validates the boolean return value of `transferFrom`. If the wNEAR contract is a non-reverting ERC-20 that returns `false` on failure instead of reverting, the precompile silently proceeds to schedule the NEAR cross-contract call without any wNEAR having been transferred, draining the engine's NEAR balance without compensation.

---

### Finding Description

In `engine-precompiles/src/xcc.rs`, the `run_with_handle` function of `CrossContractCall` performs an EVM subcall to the wNEAR ERC-20 contract to execute `transferFrom(sender, engine_implicit_address, required_near)`: [1](#0-0) 

The variable `return_value` holds the ABI-encoded return data of the `transferFrom` call — which for a standard ERC-20 is `abi.encode(bool)`. The code matches on `exit_reason` and, upon `ExitReason::Succeed(_)`, does nothing (`()`). It never decodes `return_value` to check whether the boolean is `true`. [2](#0-1) 

A non-reverting ERC-20 wNEAR implementation (one that returns `false` on insufficient balance or allowance instead of reverting) would cause the EVM subcall to exit with `Succeed` while the actual token transfer did not occur. The precompile then continues to emit the promise log that schedules the NEAR cross-contract call: [3](#0-2) 

The wNEAR address is read from storage and is configurable: [4](#0-3) 

---

### Impact Explanation

**Critical / High — Theft of engine NEAR funds / Insolvency.**

When `required_near != ZERO_YOCTO`, the engine's implicit NEAR account is expected to fund the cross-contract call only after receiving wNEAR from the caller. If the `transferFrom` silently returns `false`, the engine's NEAR balance is consumed for the cross-contract call without the caller having paid. An attacker with zero wNEAR balance (or zero allowance) can repeatedly invoke the XCC precompile, each time triggering a NEAR cross-contract call at the engine's expense. This leads to direct, unbounded drainage of the engine's NEAR balance — insolvency of the engine's operational funds.

---

### Likelihood Explanation

The currently deployed wNEAR on Aurora is an OpenZeppelin ERC-20 derivative that reverts on failure, so the bug is not immediately exploitable on mainnet with the canonical wNEAR. However:

1. The wNEAR address is set via `factory_set_wnear_address` and can be changed.
2. The code makes no compile-time or runtime assumption that wNEAR is a reverting ERC-20.
3. Any wNEAR upgrade or replacement with a non-reverting implementation (including WETH9-style contracts that return `false`) immediately opens this attack vector to any unprivileged EVM user.

Likelihood is **medium** given the dependency on wNEAR behavior, but the impact is critical if triggered.

---

### Recommendation

After the `Succeed` branch, decode `return_value` and verify the boolean is `true`. Reject the call if it is `false`:

```rust
aurora_evm::ExitReason::Succeed(_) => {
    // Decode the bool return value of transferFrom
    let transfer_ok = return_value
        .get(31)  // last byte of the 32-byte ABI-encoded bool
        .copied()
        .unwrap_or(0) == 1;
    if !transfer_ok {
        return Err(PrecompileFailure::Revert {
            exit_status: aurora_evm::ExitRevert::Reverted,
            output: b"ERR_WNEAR_TRANSFER_FAILED".to_vec(),
        });
    }
}
```

This mirrors the `SafeERC20.safeTransferFrom` pattern from OpenZeppelin, which checks both the call success and the returned boolean.

---

### Proof of Concept

1. Deploy a non-reverting wNEAR ERC-20 on Aurora that returns `false` from `transferFrom` when the caller has insufficient allowance, instead of reverting.
2. Call `factory_set_wnear_address` to point the engine at this contract (or assume it is already set to such a contract).
3. As an unprivileged EVM user with zero wNEAR balance, send a transaction calling the XCC precompile at `0x516cded1d16af10cad47d6d49128e2eb7d27b372` with a valid `CrossContractCallArgs::Eager` payload specifying `attached_near > 0`.
4. The precompile computes `required_near > 0`, calls `transferFrom(attacker, engine_implicit, required_near)` on the non-reverting wNEAR, which returns `false` with `ExitReason::Succeed`.
5. The `Succeed` arm at line 203 does nothing; execution continues to line 219.
6. A promise log is emitted scheduling the NEAR cross-contract call, funded by the engine's NEAR balance.
7. Repeat to drain the engine's NEAR balance at no cost to the attacker. [5](#0-4)
