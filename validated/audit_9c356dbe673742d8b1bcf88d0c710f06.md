### Title
Unguarded `update_rewards` Allows Any Caller to Permanently Suppress Consensus Block Rewards via `disable_rewards=true` — (File: `src/staking/staking.cairo`)

---

### Summary

The `update_rewards` function in the `Staking` contract has no access control and updates the global `last_reward_block` state **before** checking the `disable_rewards` flag. Any unprivileged caller can invoke `update_rewards(valid_staker, disable_rewards=true)` on every block, consuming the per-block reward slot without distributing any rewards, permanently freezing unclaimed consensus yield for all stakers.

---

### Finding Description

`update_rewards` is the entry point for distributing per-block consensus rewards (V3 reward regime). It enforces a single-call-per-block invariant via the global `last_reward_block` storage variable.

The critical sequence is:

```
// 1. Validates staker is active and has non-zero balance
let staker_info = self.internal_staker_info(:staker_address);
assert!(self.is_staker_active(...), Error::INVALID_STAKER);
assert!(staker_total_strk_balance.is_non_zero(), Error::INVALID_STAKER);

// 2. *** Unconditionally marks the block as processed ***
self.last_reward_block.write(current_block_number);   // line 1485

// 3. Only THEN checks disable_rewards — but the slot is already consumed
if disable_rewards || self.is_pre_consensus() {
    return;                                            // line 1487-1489
}
``` [1](#0-0) 

The function begins with only `general_prerequisites()`, which checks the pause flag and that the caller is non-zero — there is no role check or caller restriction. [2](#0-1) 

`last_reward_block` is a single global field, not per-staker: [3](#0-2) 

Once `last_reward_block` is written to the current block number, any subsequent call in the same block — including the legitimate consensus-layer call with `disable_rewards=false` — will revert with `REWARDS_ALREADY_UPDATED`: [4](#0-3) 

The state-transition bug is that the "block processed" state transition (`last_reward_block` update) is decoupled from the "rewards distributed" state transition. An attacker can drive the system into the "block processed, no rewards distributed" state on every block.

---

### Impact Explanation

**High — Permanent freezing of unclaimed consensus yield.**

In the V3 consensus rewards regime, `update_rewards` is the sole mechanism for distributing per-block STRK rewards to stakers and their delegation pools. If an attacker calls `update_rewards(any_valid_staker, disable_rewards=true)` on every block:

- `last_reward_block` is set to the current block number.
- No rewards are calculated or credited to any staker.
- The legitimate consensus-layer call for that block is permanently blocked.
- All stakers and pool members lose their entire consensus reward stream indefinitely.

This constitutes permanent freezing of unclaimed yield for all protocol participants.

---

### Likelihood Explanation

**High.** The attack requires:
1. Knowledge of any active staker address with non-zero balance — trivially obtained from on-chain `NewStaker` events.
2. Calling `update_rewards(staker_address, disable_rewards=true)` once per block.

There is no economic barrier, no privileged access required, and no way for the protocol to prevent it short of a contract upgrade. The attacker pays only gas costs.

---

### Recommendation

1. **Add access control**: Restrict `update_rewards` to a trusted caller (e.g., the consensus layer address or a designated role), analogous to how `update_rewards_from_attestation_contract` is restricted to the attestation contract via `assert_caller_is_attestation_contract`. [5](#0-4) 

2. **Reorder state update**: Move `self.last_reward_block.write(current_block_number)` to after the `disable_rewards` guard, or only write it when rewards are actually distributed, so the "block processed" state is not set unless the full reward distribution transition completes.

---

### Proof of Concept

1. Attacker observes any active staker address `S` from on-chain events (e.g., `NewStaker`).
2. On every new block `B`, attacker calls:
   ```
   staking.update_rewards(staker_address=S, disable_rewards=true)
   ```
3. Inside `update_rewards`:
   - `general_prerequisites()` passes (contract not paused, caller non-zero). [6](#0-5) 
   - `current_block_number > last_reward_block` passes (new block). [4](#0-3) 
   - Staker `S` is active with non-zero balance — passes. [7](#0-6) 
   - `last_reward_block` is written to block `B`. [8](#0-7) 
   - `disable_rewards=true` → early return, zero rewards distributed. [9](#0-8) 
4. The consensus layer's legitimate call to `update_rewards(S, disable_rewards=false)` in block `B` reverts with `REWARDS_ALREADY_UPDATED`.
5. Repeated every block → all consensus rewards are permanently frozen.

        fn update_rewards(
            ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
        ) {
            self.general_prerequisites();
            let current_block_number = starknet::get_block_number();
            assert!(
                current_block_number > self.last_reward_block.read(),
                "{}",
                Error::REWARDS_ALREADY_UPDATED,
            );

            // Assert staker exists and active.
            // Staker is considered to exist from the moment of `stake` (when `InternalStakerInfo`
            // struct is created) until the calling to `unstake_action` (when `InternalStakerInfo`
            // struct is deleted).
            // Staker remains active until the intent period begins, i.e. K epochs after
            // `unstake_intent` is called.
            let staker_info = self.internal_staker_info(:staker_address);
            let curr_epoch = self.get_current_epoch();
            assert!(
                self.is_staker_active(:staker_address, epoch_id: curr_epoch),
                "{}",
                Error::INVALID_STAKER,
            );

            let staker_pool_info = self.staker_pool_info.entry(staker_address).as_non_mut();
            let (staker_total_strk_balance, staker_total_btc_balance) = self
                .get_staker_total_strk_btc_balance_at_epoch(
                    :staker_address, :staker_pool_info, epoch_id: curr_epoch,
                );
            // Assert staker has non-zero balance.
            // Staker exists with zero balance for the first K epochs after `stake`, then the stake
            // becomes effective.
            assert!(staker_total_strk_balance.is_non_zero(), "{}", Error::INVALID_STAKER);

            // Update last block rewards.
            self.last_reward_block.write(current_block_number);

            if disable_rewards || self.is_pre_consensus() {
                return;
            }
```
