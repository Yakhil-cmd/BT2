Audit Report

## Title
Unbounded O(n) Per-Round Scan in `purge_expired_ingress_messages` Enables Replica CPU DoS via Ingress Queue Flooding — (File: `rs/execution_environment/src/scheduler.rs`)

## Summary

`IngressQueue` carries no upper bound on messages per canister. The scheduler's `purge_expired_ingress_messages`, called unconditionally at the start of every consensus round, performs an O(n) linear scan over the subnet's own ingress queue with no pre-check guard, and an O(n) `all_ingress_messages` scan over every hot canister's queue even when no messages are expired. An attacker controlling a canister with cycles can accumulate up to ~300,000 messages (1,000/block × 300 blocks at 5-minute TTL), causing O(n) native-Rust CPU work in the round-preparation phase every round for the entire TTL window, with a second O(n) burst of `set_status` writes at expiry.

## Finding Description

**Root cause — no per-canister ingress queue bound:**

`IngressQueue` is explicitly documented as unbounded:

> "Representation of the Ingress queue. There is no upper bound on the number of messages it can store." [1](#0-0) 

`IngressQueue::push()` performs no capacity check before inserting: [2](#0-1) 

**O(n) unconditional scan — subnet queue:**

`purge_expired_ingress_messages` calls `subnet_queues_retain_ingress_messages` unconditionally every round with no `all_ingress_messages` pre-check: [3](#0-2) 

The underlying `subnet_queues_retain_ingress_messages` delegates directly to `retain_messages` with no guard: [4](#0-3) 

`retain_messages` iterates over every message in every per-canister `VecDeque` unconditionally: [5](#0-4) 

**O(n) scan — per-canister hot queues:**

For each hot canister, `all_ingress_messages(not_expired)` is called as a guard before `retain_ingress_messages`. However, `all_ingress_messages` itself is O(n) — it iterates all messages to verify the predicate holds: [6](#0-5) [7](#0-6) 

When all N messages are set to expire in the future, `all_ingress_messages` scans all N messages and returns `true` — O(N) work per hot canister per round, every round, for the entire TTL window.

**Canisters with pending ingress stay in the hot pool:**

`CanisterState::is_cold()` returns `false` when `has_input()` is true (which includes pending ingress messages), so flooded canisters are never demoted to cold and are scanned every round: [8](#0-7) 

**Call site — unconditional every round:** [9](#0-8) 

**System subnet bypass of ingress history cap:**

On system subnets (NNS), the `ingress_history_max_messages` check is skipped entirely, removing the only indirect cap on queue depth: [10](#0-9) 

**Practical accumulation limits:**

`MAX_INGRESS_MESSAGES_PER_BLOCK = 1000` and `MAX_INGRESS_TTL = 5 minutes` allow up to ~300,000 messages to accumulate: [11](#0-10) [12](#0-11) 

**Note on "free" flooding claim:** The report's assertion that subnet management calls are universally free to unprivileged users is partially inaccurate. For most subnet methods (e.g., `canister_status`), the `effective_canister_id` is non-null and the target canister pays the induction cost. `raw_rand` is explicitly rejected for ingress messages at the canister manager level. The attack therefore requires the attacker to control a canister with cycles, or to target a canister with sufficient cycles. This is a meaningful but not prohibitive constraint. The O(n) scan vulnerability is real regardless of whether flooding is free.

## Impact Explanation

The O(n) work occurs in the replica's native Rust execution path, entirely outside any Wasm instruction budget or per-round instruction limit. With 300,000 messages accumulated, `retain_messages` on the subnet queue iterates 300,000 entries per round; `all_ingress_messages` iterates all N messages per hot canister per round. If this scan consumes enough wall-clock time to push the round past its time budget (~1–2 seconds on application subnets), block production stalls and the subnet falls behind consensus. At expiry, a burst of up to 300,000 `set_status` BTreeMap writes amplifies the cost further. This matches the allowed impact: **High — Application/platform-level DoS, consensus blocking, or subnet availability impact not based on raw volumetric DDoS**.

## Likelihood Explanation

An attacker controlling any canister with cycles can submit up to 1,000 ingress messages per block to the subnet management canister (targeting their own canister), paying induction costs from the target canister's balance. Sustaining this for 5 minutes fills the queue to its practical maximum. No privileged access, key material, or governance majority is required. The `IngressPoolThrottler` is per-node and does not prevent messages from being included in blocks by other nodes or submitted via multiple boundary nodes.

## Recommendation

1. **Add a per-canister ingress queue depth limit** in `IngressQueue::push()`, analogous to `DEFAULT_QUEUE_CAPACITY = 500` used for canister-to-canister queues. [13](#0-12) 

2. **Guard `subnet_queues_retain_ingress_messages` with an `all_ingress_messages` pre-check** (as is done for per-canister queues at scheduler.rs line 782) to avoid the unconditional O(n) `retain_messages` call when no messages are expired.

3. **Add a global cap on the subnet ingress queue depth** (separate from the per-canister ingress history limit) to bound the worst-case scan cost regardless of subnet type.

## Proof of Concept

```
Setup:
  Attacker controls canister C with sufficient cycles.

Attack (sustained for 5 minutes):
  for block in 0..300:
    for i in 0..1000:
      submit ingress to IC_00 (e.g., canister_status for C)
      with expiry_time = now + MAX_INGRESS_TTL (5 min)
      // induction cost charged to C; subnet queue grows unbounded

Effect per round (every ~1 second for 5 minutes):
  purge_expired_ingress_messages():
    subnet_queues_retain_ingress_messages(not_expired)
      -> retain_messages() iterates all ~300,000 messages  // O(300K) unconditional
    for each hot canister:
      all_ingress_messages(not_expired)
        -> iterates all N messages per canister             // O(N) per canister

Effect at T+5min (one-time burst):
  retain_messages() removes 300,000 entries from VecDeque  // O(300K)
  set_status() called 300,000 times -> BTreeMap writes     // O(300K * log(600K))

Verification: deterministic integration test using StateMachine / PocketIC:
  1. Create a canister with cycles.
  2. Induct 300,000 ingress messages into the subnet queue via state.subnet_queues_mut().push_ingress().
  3. Measure wall-clock time of purge_expired_ingress_messages() per round.
  4. Confirm linear growth in round-preparation time with queue depth.
```

    pub(super) fn retain_messages<F>(&mut self, mut predicate: F) -> Vec<Arc<Ingress>>
    where
        F: FnMut(&Ingress) -> bool,
    {
        let mut filtered_messages = vec![];
        for canister_ingress_queue in self.queues.values_mut() {
            canister_ingress_queue.retain_mut(|item| {
                if predicate(item) {
                    return true;
                }
                // Empty `canister_ingress_queues` and their corresponding schedule entry
                // are pruned below.
                filtered_messages.push(Arc::clone(item));
                self.size_bytes -= Self::ingress_size_bytes(&(*item));
                self.total_ingress_count -= 1;
                false
            });
        }

        self.schedule
            .retain_mut(|canister_id| match self.queues.entry(*canister_id) {
                Entry::Occupied(entry) if entry.get().is_empty() => {
                    entry.remove();
                    self.size_bytes -= Self::PER_CANISTER_QUEUE_OVERHEAD_BYTES;
                    false
                }
                Entry::Occupied(_) => true,
                Entry::Vacant(_) => unreachable!(),
            });

        filtered_messages
    }
