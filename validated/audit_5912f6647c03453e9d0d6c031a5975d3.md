### Title
Staking Contract Pause Blocks Attestation, Causing Permanent Loss of Epoch Rewards - (File: src/attestation/attestation.cairo)

### Summary
`Attestation::attest` is not pausable, but it calls `IStakingAttestation::update_rewards_from_attestation_contract` on the staking contract, which is gated by `general_prerequisites()` and reverts when the staking contract is paused. Because each staker's attestation window is a narrow, epoch-specific block range, a pause that overlaps with that window permanently prevents the staker from earning rewards for that epoch.

### Finding Description
`Attestation::attest` in `src/attestation/attestation.cairo` has no pause guard of its own. After validating the attestation and marking it as done, it calls into the staking contract:

```cairo
// src/attestation/attestation.cairo lines 116-134
fn attest(ref self: ContractState, block_hash: felt252) {
    ...
    self._validate_attestation(:block_hash, :staking_attestation_info);
    self._mark_attestation_is_done(
        staker_address: staking_attestation_info.staker_address(),
        current_epoch: staking_attestation_info.epoch_id(),
    );
    staking_dispatcher
        .update_rewards_from_attestation_contract(
            staker_address: staking_attestation_info.staker_address(),
        );
}
```

`update_rewards_from_attestation_contract` in `src/staking/staking.cairo` immediately calls `general_prerequisites()`:

```cairo
// src/staking/staking.cairo lines 1394-1399
fn update_rewards_from_attestation_contract(
    ref self: ContractState, staker_address: ContractAddress,
) {
    self.general_prerequisites();   // <-- reverts if paused
    ...
}
```

`general_prerequisites` calls `assert_is_unpaused`, which panics with `"Contract is paused"`:

```cairo
// src/staking/staking.cairo lines 1793-1797
fn general_prerequisites(ref self: ContractState) {
    self.assert_is_unpaused();
    assert_caller_is_not_zero();
}
```

Because the entire `attest` transaction reverts, `_mark_attestation_is_done` is also rolled back, so the staker's attestation slot for the epoch is not consumed. However, the attestation window is a narrow, deterministic block range per epoch enforced by `_assert_attest_in_window`:

```cairo
// src/attestation/attestation.cairo lines 241-250
fn _assert_attest_in_window(self: @ContractState, target_attestation_block: BlockNumber) {
    let attestation_window = self.attestation_window.read();
    let current_block_number = get_block_number();
    let min_block = target_attestation_block + MIN_ATTESTATION_WINDOW.into();
    let max_block = target_attestation_block + attestation_window.into();
    assert!(
        min_block <= current_block_number && current_block_number <= max_block, ...
    );
}
```

If the staking contract remains paused for the duration of a staker's window, the staker can never successfully call `attest` for that epoch, and the epoch's rewards are permanently lost.

### Impact Explanation
Every epoch, each staker has a unique, time-bounded attestation window. If the staking contract is paused for the entirety of that window, the staker permanently forfeits the unclaimed yield for that epoch. This matches **High: Permanent freezing of unclaimed yield**.

### Likelihood Explanation
The security agent can pause the staking contract at any time (a single privileged call). Pauses are expected during security incidents or upgrades. The attestation window is a small fraction of an epoch, so even a brief pause that happens to coincide with a staker's window causes permanent reward loss. With many stakers, each having a different window, any non-trivial pause duration will affect at least some stakers.

### Recommendation
Decouple the attestation proof from the reward distribution. The attestation contract should record the successful attestation unconditionally (even when the staking contract is paused), and the reward distribution should be claimable separately once the staking contract is unpaused. Alternatively, add a mechanism to retroactively credit rewards for epochs where attestation was proven but reward distribution was blocked by a pause.

### Proof of Concept
1. Staker S has an attestation window for epoch E covering blocks `[W_min, W_max]`.
2. Security agent calls `staking_contract.pause()` before block `W_min`.
3. During `[W_min, W_max]`, staker S calls `attestation_contract.attest(block_hash)`.
4. `attest` calls `staking_dispatcher.update_rewards_from_attestation_contract(S)`.
5. `update_rewards_from_attestation_contract` calls `general_prerequisites()` → `assert_is_unpaused()` → panics with `"Contract is paused"`.
6. The entire transaction reverts; `_mark_attestation_is_done` is rolled back.
7. The pause continues past block `W_max`; the window expires.
8. Staker S permanently loses epoch E rewards. The staking contract is later unpaused, but epoch E's reward opportunity is gone. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
