### Title
Hardcoded Gas Limit for Native Token Deposit/Withdraw Direct Calls Causes Cross-VM Asset Transfer Failure for Smart Contract Recipients - (File: `fvm/evm/types/call.go`)

---

### Summary

The Flow EVM bridge uses hardcoded gas limits of `23_300` (21,000 + 2,300) for all native FLOW token deposit and withdraw direct calls. When the recipient EVM address is a smart contract with a `receive()` or `fallback()` function that requires more than 2,300 gas, the deposit call will revert with `ErrExecutionReverted`, causing the cross-VM asset transfer to fail. This is confirmed by an existing test that explicitly demonstrates the revert behavior.

---

### Finding Description

In `fvm/evm/types/call.go`, the constants `DepositCallGasLimit` and `WithdrawCallGasLimit` are both set to `DefaultGasLimitForTokenTransfer`, which equals `IntrinsicFeeForTokenTransfer + 2_300` = `21_000 + 2_300 = 23_300`:

```go
// fvm/evm/types/call.go lines 29-37
IntrinsicFeeForTokenTransfer    = gethParams.TxGas          // 21_000
DefaultGasLimitForTokenTransfer = IntrinsicFeeForTokenTransfer + 2_300  // 23_300
DepositCallGasLimit  = DefaultGasLimitForTokenTransfer
WithdrawCallGasLimit = DefaultGasLimitForTokenTransfer
```

These constants are used unconditionally in `NewDepositCall` and `NewWithdrawCall`:

```go
// fvm/evm/types/call.go lines 193-229
func NewDepositCall(...) *DirectCall {
    return &DirectCall{
        ...
        GasLimit: DepositCallGasLimit,  // always 23_300
    }
}
func NewWithdrawCall(...) *DirectCall {
    return &DirectCall{
        ...
        GasLimit: WithdrawCallGasLimit,  // always 23_300
    }
}
```

The `Deposit` method in `fvm/evm/handler/handler.go` calls `NewDepositCall` with this hardcoded limit:

```go
// fvm/evm/handler/handler.go lines 965-974
res, err := a.fch.executeAndHandleCall(
    types.NewDepositCall(
        bridge,
        a.address,
        v.Balance(),
        bridgeAccount.Nonce(),
    ),
    ...
)
panicOnErrorOrInvalidOrFailedState(res, err)
```

The code comment in `call.go` itself acknowledges the fragility:

> "Note that these gas values might need to change if we change the transaction (e.g. add access list), then it has to be updated to use Intrinsic function to calculate the minimum gas needed to run the transaction."

The existing test at `fvm/evm/emulator/emulator_test.go` lines 83–110 explicitly confirms that depositing to a smart contract that requires more than 2,300 gas in its `receive()`/`fallback()` results in `ErrExecutionReverted`:

```go
call := types.NewDepositCall(bridgeAccount, testContract, types.MakeBigIntInFlow(1), 0)
res, err := blk.DirectCall(call)
require.Equal(t, res.VMError, gethVM.ErrExecutionReverted)
```

The 2,300 gas stipend is the classic Ethereum `transfer()`/`send()` restriction — it is intentionally too small for any smart contract that does non-trivial work in its `receive()` function (e.g., emitting events, updating storage, or calling other contracts). Any COA or EVM smart contract wallet that implements a non-trivial `receive()` function will be permanently unable to receive FLOW deposits via the bridge.

---

### Impact Explanation

**Cross-VM asset loss / bridge escrow mis-accounting.** When a user calls `EVM.EVMAddress.deposit()` or `CadenceOwnedAccount.deposit()` targeting an EVM smart contract address whose `receive()` or `fallback()` function consumes more than 2,300 gas, the EVM execution reverts. The handler calls `panicOnErrorOrInvalidOrFailedState`, which panics and rolls back the Cadence transaction. The FLOW tokens are not credited to the EVM address, and the user's vault is consumed (destroyed) by the Cadence runtime before the panic, resulting in permanent loss of the bridged tokens. Any EVM smart contract wallet (e.g., a multisig, a vault with event emission, or a contract using ERC-777 `tokensReceived`) deployed on Flow EVM is permanently unable to receive FLOW via the native bridge.

---

### Likelihood Explanation

**Medium.** Any unprivileged user can trigger this by calling `EVM.EVMAddress.deposit()` or `CadenceOwnedAccount.deposit()` targeting an EVM smart contract address. Smart contract wallets, multisigs, and ERC-777 token receivers are common patterns. The COA contract itself implements `receive()` (confirmed by `fvm/evm/handler/coa/coa_abi.json`), but other user-deployed contracts with heavier `receive()` logic are directly affected. No special privileges are required — any Flow account can submit such a transaction.

---

### Recommendation

Replace the hardcoded `DepositCallGasLimit` and `WithdrawCallGasLimit` constants with a dynamic gas estimation mechanism, or allow callers to specify a gas limit for deposit/withdraw direct calls. At minimum, the gas limit should be configurable per-call rather than a compile-time constant. The comment in the code already acknowledges this debt. A reasonable approach is to use the EVM `estimateGas` path (analogous to `dryRun`) before constructing the `DirectCall`, or to expose a higher default that accounts for non-trivial `receive()` implementations.

**Attack path:**
1. Attacker or user deploys an EVM smart contract at address `X` on Flow EVM whose `receive()` function uses >2,300 gas (e.g., emits an event or updates storage).
2. User submits a Cadence transaction calling `EVM.EVMAddress(bytes: X_bytes).deposit(from: <-vault)` or `coa.deposit(from: <-vault)` where the COA address is `X`.
3. The FVM constructs a `NewDepositCall` with `GasLimit = 23_300`.
4. The EVM executes the deposit; the `receive()` function runs out of gas and reverts.
5. `panicOnErrorOrInvalidOrFailedState` panics, rolling back the Cadence transaction.
6. The FLOW tokens are not credited to the EVM address. Depending on vault lifecycle, tokens may be lost.
