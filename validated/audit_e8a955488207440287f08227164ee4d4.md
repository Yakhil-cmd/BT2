### Title
Missing vector-length validation before indexing in `do_generation_many` causes panic on malicious empty `PolynomialCommitmentsMessageMany` — (`src/ecdsa/ot_based_ecdsa/triples/generation.rs`)

### Summary

A malicious participant can send a `PolynomialCommitmentsMessageMany` message in round 2 of triple generation with all inner vectors set to length 0. Because `do_generation_many` indexes into those vectors with `[i]` (Rust's panicking index operator) before performing any length or commitment check, every honest party's protocol future panics. Since `make_protocol` / `ProtocolExecutor::poke` does not wrap the future poll in `catch_unwind`, the panic propagates to the caller, permanently aborting triple generation for all honest parties.

### Finding Description

**Step 1 — Attacker-controlled entry point.**

`recv_from_others` in `src/protocol/helpers.rs` deserializes any `T: DeserializeOwned` from the wire without any structural validation beyond what `rmp_serde` requires. An empty `Vec` is perfectly valid msgpack, so a `PolynomialCommitmentsMessageMany` with all six fields set to `[]` deserializes successfully. [1](#0-0) 

**Step 2 — Unchecked indexing.**

After receiving the message, `do_generation_many` immediately indexes into the deserialized vectors with `[i]` for `i in 0..N`. `N` is always ≥ 1 (enforced by `assert!(N > 0)` at line 116; the only call site uses `N = 1`). With empty vectors, `their.big_e_v[0]` panics. [2](#0-1) 

**Step 3 — Commitment check is unreachable.**

The commitment check (`Commitment::check`) and the degree check that would catch a wrong-length polynomial both appear *after* the panicking index operations at lines 337–342. They cannot protect against this attack. [3](#0-2) 

**Step 4 — Panic is not caught by the protocol executor.**

`ProtocolExecutor::poke` polls the future at line 505 with no `catch_unwind`. A panic inside the future propagates directly through `poke()` to the caller. [4](#0-3) 

The `PolynomialCommitmentsMessageMany` struct has no invariant-enforcing deserializer; all six `Vec` fields accept any length. [5](#0-4) 

### Impact Explanation

Triple generation is the prerequisite for all OT-based ECDSA presigning and signing. A single malicious participant who sends empty vectors in round 2 causes every honest party's `poke()` call to panic, aborting the session. Because the attacker can repeat this on every new session, triple generation — and therefore all downstream ECDSA operations — is permanently denied for honest parties as long as the attacker remains a participant.

**Impact: High — Permanent denial of triple generation (and therefore signing) for honest parties.**

### Likelihood Explanation

The attack requires only that the adversary be a registered participant (the normal trust assumption for threshold protocols). No cryptographic assumption needs to be broken. The malicious message is trivially constructable: serialize a `PolynomialCommitmentsMessageMany` with all six fields as empty arrays. The attacker does not even need to send a valid round-1 commitment — any 32-byte value passes the `Commitment` deserialization.

### Recommendation

Add an explicit length check on all six inner vectors immediately after deserialization, before any indexing:

```rust
// In the recv_from_others loop, before `for i in 0..N`:
if their.big_e_v.len() != N
    || their.big_f_v.len() != N
    || their.big_l_v.len() != N
    || their.randomizer_v.len() != N
    || their.phi_proof0_v.len() != N
    || their.phi_proof1_v.len() != N
{
    return Err(ProtocolError::AssertionFailed(format!(
        "message from {from:?} has wrong number of polynomial commitments"
    )));
}
```

This converts the panic into a graceful `ProtocolError`, allowing honest parties to abort cleanly and identify the malicious sender.
