### Title
Hardcoded `NearGas` constants in `ExitToNear`/`ExitToEthereum` precompiles can permanently freeze bridged funds - (File: `engine-precompiles/src/native.rs`)

---

### Summary

The `ExitToNear` and `ExitToEthereum` bridge precompiles attach hardcoded, compile-time `const` NEAR gas values to every outbound NEAR promise. Because these values cannot be updated without a full contract upgrade, any NEAR protocol gas repricing that makes the hardcoded amounts insufficient will cause the NEAR-side promise to fail silently after the EVM-side tokens have already been burned — permanently freezing user funds with no on-chain recovery path.

---

### Finding Description

In `engine-precompiles/src/native.rs`, the `costs` module defines four Rust compile-time constants used as `attached_gas` for every NEAR promise emitted by the two bridge exit precompiles:

```rust
// engine-precompiles/src/native.rs  lines 42-62
mod costs {
    // TODO(#483): Determine the correct amount of gas
    pub(super) const EXIT_TO_NEAR_GAS: EthGas = EthGas::new(0);
    // TODO(#483): Determine the correct amount of gas
    pub(super) const EXIT_TO_ETHEREUM_GAS: EthGas = EthGas::new(0);
    /// Value determined experimentally based on tests and mainnet data.
    pub(super) const FT_TRANSFER_GAS: NearGas = NearGas::new(10_000_000_000_000);
    pub(super) const FT_TRANSFER_CALL_GAS: NearGas = NearGas::new(70_000_000_000_000);
    /// Value determined experimentally based on tests.
    pub(super) const EXIT_TO_NEAR_CALLBACK_GAS: NearGas = NearGas::new(10_000_000_000_000);
    // TODO(#332): Determine the correct amount of gas
    pub(super) const WITHDRAWAL_GAS: NearGas = NearGas::new(100_000_000_000_000);
}
``` [1](#0-0) 

These constants are baked into the compiled WASM binary. There is no storage-backed, admin-settable mechanism to update them at runtime.

**`ExitToEthereum` — no refund path at all.**
When a user withdraws ETH or an ERC-20 token to Ethereum, the precompile burns the tokens on the EVM side and then emits a single `PromiseArgs::Create` wrapping a `withdraw` call with `attached_gas: costs::WITHDRAWAL_GAS`:

```rust
// engine-precompiles/src/native.rs  lines 977-985
let withdraw_promise = PromiseCreateArgs {
    target_account_id: nep141_address,
    method: "withdraw".to_string(),
    args: serialized_args,
    attached_balance: Yocto::new(1),
    attached_gas: costs::WITHDRAWAL_GAS,   // ← hardcoded 100 TGas
};
let promise = borsh::to_vec(&PromiseArgs::Create(withdraw_promise)).unwrap();
``` [2](#0-1) 

There is no callback and no refund branch. If the NEAR promise fails (e.g., because 100 TGas is no longer sufficient after a protocol repricing), the burned tokens are irrecoverable.

**`ExitToNear` — same root cause, partial mitigation only under a compile-time feature.**
The `ft_transfer` path attaches `FT_TRANSFER_GAS` (10 TGas) and the `ft_transfer_call` path attaches `FT_TRANSFER_CALL_GAS` (70 TGas):

```rust
// engine-precompiles/src/native.rs  lines 456-468
let attached_gas = if method == "ft_transfer_call" {
    costs::FT_TRANSFER_CALL_GAS
} else {
    costs::FT_TRANSFER_GAS
};
let transfer_promise = PromiseCreateArgs {
    ...
    attached_gas,
};
``` [3](#0-2) 

A refund callback is only appended when the `error_refund` compile-time feature is active **and** the callback itself uses the equally hardcoded `EXIT_TO_NEAR_CALLBACK_GAS` (10 TGas):

```rust
// engine-precompiles/src/native.rs  lines 473-482
PromiseArgs::Callback(PromiseWithCallbackArgs {
    base: transfer_promise,
    callback: PromiseCreateArgs {
        ...
        attached_gas: costs::EXIT_TO_NEAR_CALLBACK_GAS,  // ← hardcoded 10 TGas
    },
})
``` [4](#0-3) 

If the callback itself runs out of gas, the refund also fails, leaving funds frozen.

The `WITHDRAWAL_GAS` constant carries an explicit `TODO(#332): Determine the correct amount of gas` comment, confirming the value was never formally validated. [5](#0-4) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

The EVM-side burn is committed atomically within the Aurora transaction. The NEAR promise is a separate, asynchronous action. If the promise fails due to insufficient `attached_gas`, the EVM burn is not rolled back. For `ExitToEthereum` there is no recovery path whatsoever. For `ExitToNear` without the `error_refund` feature, the same applies. Even with `error_refund`, the refund callback uses a second hardcoded gas value that can equally fail. All in-flight withdrawals submitted before a gas repricing event would be affected simultaneously.

---

### Likelihood Explanation

**Medium.**

NEAR Protocol has repriced gas operations in past protocol upgrades (e.g., storage-read/write costs changed with the introduction of flat storage). The `WITHDRAWAL_GAS` constant is explicitly marked with a TODO indicating its value was never formally determined. Any future NEAR protocol upgrade that increases the cost of `ft_transfer`, `ft_transfer_call`, or `withdraw` operations on NEP-141 contracts — or any upgrade to the eth-connector contract itself — could push actual gas consumption above the hardcoded ceiling. The trigger is a NEAR protocol-level event, not an attacker action, but the resulting fund loss is permanent and affects all concurrent withdrawals.

---

### Recommendation

Replace the compile-time `const` gas values with storage-backed, admin-updatable fields (analogous to how `SiloParams` are stored and updated). Expose a privileged method (callable only by the engine owner) to update each gas limit independently. This mirrors the recommended fix in the external report: make the gas limit mutable so it can be adjusted in response to protocol changes without requiring a full WASM upgrade and without affecting already-in-flight promises.

---

### Proof of Concept

1. User calls the `ExitToEthereum` precompile (`0xb0bd02f6a392af548bdf1cfaee5dfa0eefcc8eab`) with an ERC-20 withdrawal.
2. The precompile burns the ERC-20 tokens on the Aurora EVM (state change committed).
3. The precompile emits a `PromiseArgs::Create` log containing a `withdraw` call with `attached_gas = WITHDRAWAL_GAS = 100_000_000_000_000` (100 TGas). [6](#0-5) 

4. NEAR Protocol reprices storage or cross-contract call operations such that the `withdraw` method on the eth-connector now requires > 100 TGas.
5. The NEAR runtime executes the promise and fails with `GasExceeded`.
6. No callback exists to refund the user; the EVM burn is not reversed.
7. The user's tokens are permanently frozen — they no longer exist on Aurora and the NEAR-side transfer never completed.

The same scenario applies to `ExitToNear` with `FT_TRANSFER_GAS = 10_000_000_000_000` (10 TGas) for `ft_transfer` calls. [7](#0-6)
