### Title
Missing Lower-Bound Validation on `threshold` in FROST Sign and Presign Initialization — (`File: src/frost/mod.rs`)

---

### Summary

`assert_sign_inputs` (used by every FROST EdDSA and RedJubjub signing entry-point) and the FROST `presign` initializer both validate only the **upper** bound of the caller-supplied `threshold` parameter. Neither enforces the documented minimum of `threshold ≥ 2`. A caller — including a malicious coordinator — can supply `threshold = 1` (or `0`), bypassing the threshold security property and causing the signing protocol to run with an inconsistent threshold, corrupting the produced signature or allowing a single participant to aggregate a signing session.

---

### Finding Description

`assert_sign_inputs` in `src/frost/mod.rs` checks:

```rust
if threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
``` [1](#0-0) 

There is **no corresponding lower-bound check** (`threshold < 2`). Compare with `assert_key_invariants` in `src/dkg.rs`, which explicitly enforces the minimum:

```rust
if threshold < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
}
``` [2](#0-1) 

And `validate_triple_inputs` in triple generation, which also enforces it:

```rust
if threshold_value < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold: threshold_value, min: 2 });
}
``` [3](#0-2) 

The protocol specification in `docs/dkg.md` explicitly states the invariant `1 < threshold < N` must hold at round 1.1. DKG enforces it; signing does not.

`assert_sign_inputs` is the sole validation gate for all three FROST signing entry-points:

- `sign_v1` / `sign_v2` in `src/frost/eddsa/sign.rs` [4](#0-3) 
- `sign` in `src/frost/redjubjub/sign.rs` [5](#0-4) 

The FROST `presign` initializer in `src/frost/mod.rs` has the same gap — it checks only the upper bound:

```rust
if args.threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
``` [6](#0-5) 

No lower-bound check exists there either.

---

### Impact Explanation

When `threshold = 1` is supplied:

1. `assert_sign_inputs` passes without error.
2. The validated `threshold` value is forwarded into the FROST signing future (`fut_wrapper_v1`, `fut_wrapper_v2`, redjubjub `fut_wrapper`), where it is used to construct a `KeyPackage` with `min_signers = 1`.
3. The FROST `aggregate` function then accepts a signing package containing only **one** signature share (the coordinator's own), producing a signature that is either cryptographically invalid (because the key shares were generated under a higher threshold during DKG) or — in degenerate configurations — a signature that verifies against the public key with a single participant's contribution.
4. In either case, honest participants who contributed their shares receive no indication that the threshold was violated; the coordinator silently produces a corrupted or unauthorized output.

This maps to **High: Corruption of sign outputs so honest parties accept inconsistent transcripts or unusable cryptographic outputs**, and potentially **Critical: Unauthorized creation of a valid threshold signature** if the FROST library constructs a `KeyPackage` with `min_signers = 1` and the coordinator's linearized share happens to satisfy the verification equation. [7](#0-6) 

---

### Likelihood Explanation

The `threshold` parameter is a plain `usize` wrapped in `ReconstructionLowerBound`, which has no internal invariant enforcement:

```rust
pub struct ReconstructionLowerBound(usize);
impl ReconstructionLowerBound {
    pub fn value(self) -> usize { self.0 }
}
``` [8](#0-7) 

Any caller of `sign_v1`, `sign_v2`, or redjubjub `sign` can pass `ReconstructionLowerBound::from(1usize)` or `ReconstructionLowerBound::from(0usize)`. No privilege beyond being a protocol participant is required. A malicious coordinator orchestrating a signing session controls which `threshold` value is passed to their own local call and can present a different threshold to different participants, creating an inconsistent view of the session.

---

### Recommendation

Add the same lower-bound guard that `assert_key_invariants` and `validate_triple_inputs` already apply, inside `assert_sign_inputs` and the FROST `presign` initializer:

```rust
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

This should be inserted immediately after the `threshold.into()` conversion in both `assert_sign_inputs` (line 126) and `presign` (after line 47 in `src/frost/mod.rs`).

pub fn assert_sign_inputs(
    participants: &[Participant],
    threshold: impl Into<ReconstructionLowerBound>,
    me: Participant,
    coordinator: Participant,
) -> Result<ParticipantList, InitializationError> {
    let threshold = threshold.into();
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // validate threshold
    if threshold.value() > participants.len() {
        return Err(InitializationError::ThresholdTooLarge {
            threshold: threshold.value(),
            max: participants.len(),
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }
    Ok(participants)
}
```
