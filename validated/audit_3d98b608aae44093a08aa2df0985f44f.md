### Title
Missing Zero-Address Validation for `reward_address` Allows Permanent Freezing of Unclaimed Yield - (File: src/staking/staking.cairo, src/pool/pool.cairo)

### Summary
Neither `stake()` nor `change_reward_address()` in the staking contract, nor `enter_delegation_pool()` nor `change_reward_address()` in the pool contract, validate that the supplied `reward_address` is non-zero. A staker or pool member who registers with `reward_address = 0` will find that every subsequent call to `claim_rewards()` reverts, permanently freezing their accrued yield.

### Finding Description
The `stake()` function in `src/staking/staking.cairo` validates that `reward_address` is not a registered token address, but performs no zero-address check: [1](#0-0) 

The only guard is `REWARD_ADDRESS_IS_TOKEN`. A caller may freely pass `reward_address = 0`.

The same gap exists in `change_reward_address()` in the staking contract: [2](#0-1) 

And in `enter_delegation_pool()` in the pool contract: [3](#0-2) 

And in `change_reward_address()` in the pool contract: [4](#0-3) 

When `claim_rewards()` is later invoked, the staking contract's internal helper `send_rewards_to_staker` executes: [5](#0-4) 

`checked_transfer` to address `0` will revert because the STRK token (OpenZeppelin ERC20 on Starknet) rejects transfers to the zero address. The pool contract's `claim_rewards` has the identical problem: [6](#0-5) 

Additionally, `unstake_action()` calls `send_rewards_to_staker` before returning the principal, so a staker with `reward_address = 0` also cannot complete the unstake flow until the address is corrected: [7](#0-6) 

### Impact Explanation
Any staker or pool member whose `reward_address` is the zero address will have their unclaimed yield permanently frozen: every call to `claim_rewards()` reverts. If the account is a smart contract that cannot invoke `change_reward_address()` (e.g., a contract with no such method), the freeze is irrecoverable. This matches the **High** impact category: *Permanent freezing of unclaimed yield*.

### Likelihood Explanation
Smart-contract stakers and delegators are a realistic and growing user class. An uninitialized `ContractAddress` in Cairo defaults to `0`. A contract that calls `stake()` or `enter_delegation_pool()` without explicitly setting `reward_address` will silently register the zero address. The protocol provides no on-chain guard to prevent this, making accidental misconfiguration likely.

### Recommendation
Add a non-zero assertion for `reward_address` in all four entry points:

- `stake()` in `src/staking/staking.cairo`
- `change_reward_address()` in `src/staking/staking.cairo`
- `enter_delegation_pool()` in `src/pool/pool.cairo`
- `change_reward_address()` in `src/pool/pool.cairo`

The `ZERO_ADDRESS` error string is already defined in the protocol's error catalog: [8](#0-7) 

Use it with an assertion such as:
```cairo
assert!(reward_address.is_non_zero(), "{}", GenericError::ZERO_ADDRESS);
```

### Proof of Concept
1. Deploy the system normally.
2. Call `stake(reward_address: 0, operational_address: <valid>, amount: <min_stake>)` from a staker address.
3. Advance epochs and trigger reward accrual via attestation.
4. Call `claim_rewards(staker_address: <staker>)` from the staker address.
5. Observe the transaction reverts because `checked_transfer(recipient: 0, amount: <rewards>)` fails inside `send_rewards_to_staker`.
6. The staker's `unclaimed_rewards_own` is never cleared; yield is permanently frozen unless the staker calls `change_reward_address` with a valid address (impossible for contracts lacking that capability).
