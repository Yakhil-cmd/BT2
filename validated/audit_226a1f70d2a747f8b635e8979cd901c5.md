### Title
Single Coordinator as Sole Signature Aggregator Enables Permanent Denial of Signing — (`File: src/ecdsa/ot_based_ecdsa/sign.rs`, `src/ecdsa/robust_ecdsa/sign.rs`)

---

### Summary

Both the OT-based and Robust ECDSA signing protocols designate a single coordinator as the only entity capable of aggregating signature shares and producing the final signature. Non-coordinator participants send their shares exclusively to the coordinator and immediately return `Ok(None)`, with no fallback path. A malicious coordinator can collect all shares and refuse to produce the signature, permanently denying signing for honest parties — directly analogous to the oracle single-point-of-failure described in the external report.

---

### Finding Description

In `do_sign_participant` (both OT-based and Robust ECDSA), every non-coordinator participant computes its signature share, sends it **only** to the coordinator via `chan.send_private`, and immediately returns `Ok(None)`: [1](#0-0) [2](#0-1) 

The coordinator is the only party that aggregates shares and produces the final `Signature`: [3](#0-2) [4](#0-3) 

The routing decision is made in `fut_wrapper`, which unconditionally delegates signature production to the coordinator path: [5](#0-4) 

There is no reserve aggregator, no broadcast of the final signature back to participants for independent verification, and no mechanism for honest participants to detect or recover from coordinator silence after shares have been submitted.

---

### Impact Explanation

A malicious coordinator can:
1. Participate in presigning to obtain a valid `PresignOutput`.
2. Be designated as coordinator for the signing round.
3. Receive all `n-1` signature shares from honest participants (who immediately return `Ok(None)` after sending).
4. Silently discard them and produce no signature.

Honest participants have already consumed their presignature material and returned. Because presignatures must never be reused (reuse leaks the private key), the entire presign phase must be re-executed. The malicious coordinator can repeat this indefinitely whenever it is selected, causing **permanent denial of signing** for honest parties. The Robust ECDSA scheme explicitly tolerates up to `max_malicious` malicious participants, yet the signing phase provides no tolerance for a malicious coordinator — a single compromised coordinator defeats the entire scheme's liveness guarantee.

**Impact class:** High — Permanent denial of signing for honest parties under valid protocol inputs.

---

### Likelihood Explanation

The coordinator is an ordinary `Participant` with no additional trust requirement enforced by the library. Any participant designated as coordinator can trivially execute this attack by simply not calling the aggregation logic. The attack requires no cryptographic capability, no key material beyond what the coordinator legitimately holds, and leaves no detectable evidence to other participants.

---

### Recommendation

- Require the coordinator to **broadcast** the final aggregated signature (or a commitment to it) to all participants, so honest parties can independently verify completion and detect silence.
- Alternatively, implement a **threshold aggregation** path where any `t` participants can reconstruct the signature without a designated coordinator, eliminating the single point of failure.
- At minimum, document the coordinator as a **trusted role** and enforce that it cannot be one of the `max_malicious` counted adversaries, so the liveness assumption is explicit.

---

### Proof of Concept

1. Run presigning with participants `[P1, P2, P3, P4, P5]`, `max_malicious = 2`.
2. Designate `P1` (malicious) as coordinator in the signing round.
3. `P2`–`P5` each call `do_sign_participant`, compute their share, send it to `P1` via `chan.send_private`, and return `Ok(None)`.
4. `P1` receives all four shares in `recv_from_others` but never calls `sig.verify` or returns `Ok(Some(sig))` — it simply drops the future.
5. No signature is produced. The presignature is consumed. Honest parties have no recourse and must re-run presigning, where `P1` can repeat the attack. [6](#0-5) [7](#0-6)

async fn do_sign_coordinator(
    mut chan: SharedChannel,
    participants: ParticipantList,
    me: Participant,
    public_key: AffinePoint,
    presignature: RerandomizedPresignOutput,
    msg_hash: Scalar,
) -> Result<SignatureOption, ProtocolError> {
    let mut s = compute_signature_share(&presignature, msg_hash, &participants, me)?.0;
    let wait_round = chan.next_waitpoint();

    for (_, s_i) in
        recv_from_others::<SerializableScalar<C>>(&chan, wait_round, &participants, me).await?
    {
        // Sum the linearized shares
        s += s_i.0;
    }

    // raise error if s is zero
    if s.is_zero().into() {
        return Err(ProtocolError::AssertionFailed(
            "signature part s cannot be zero".to_string(),
        ));
    }
    // Normalize s
    s.conditional_assign(&(-s), s.is_high());

    let sig = Signature {
        big_r: presignature.big_r,
        s,
    };

    if !sig.verify(&public_key, &msg_hash) {
        return Err(ProtocolError::AssertionFailed(
            "signature failed to verify".to_string(),
        ));
    }

    Ok(Some(sig))
}
