Audit Report

## Title
Missing Intra-Payload Duplicate `callback_id` Check in Bitcoin Payload Validation Allows Byzantine Proposer to Inject Multiple Responses — (`rs/bitcoin/consensus/src/payload_builder.rs`)

## Summary
`validate_payload` in `BitcoinPayloadBuilder` decodes the `SelfValidatingPayload` and delegates to `validate_self_validating_payload_impl`, which only checks whether the payload is empty. No check is performed for duplicate `callback_id` values within the same payload. Additionally, the `delivered_ids` set computed from past payloads is never used to filter the current payload's callback IDs. A Byzantine block proposer can craft a payload containing N copies of the same `BitcoinAdapterResponse`, which passes validation on all honest nodes, causing N `ConsensusResponse` entries to be pushed into `consensus_queue` for a single pending request and subsequently processed by the execution environment.

## Finding Description
**Root cause — `validate_self_validating_payload_impl` performs no semantic validation.** The function at `rs/bitcoin/consensus/src/payload_builder.rs` lines 234–251 returns `Ok(size)` for any non-empty payload without inspecting the contents. `validate_payload` (lines 357–397) decodes the payload into a `Vec<BitcoinAdapterResponse>` and calls this function, but never iterates the decoded vector to check for duplicate `callback_id` values.

**`delivered_ids` is computed and discarded.** At line 369, `parse::parse_past_payload_ids(past_payloads, &self.log)` produces a `BTreeSet<u64>` of already-delivered callback IDs. This set is assigned to `delivered_ids` and never referenced again; it is not used to reject callback IDs already present in past payloads, nor to detect intra-payload duplicates.

**`push_response` is not idempotent.** In `rs/replicated_state/src/bitcoin.rs`:
- For `GetSuccessorsResponse` (lines 28–62): the context is fetched with `get_mut` (not `remove`), so it survives the first call. A second call with the same `callback_id` succeeds again, pushes a second `ConsensusResponse`, and overwrites `bitcoin_get_successors_follow_up_responses`.
- For `GetSuccessorsReject`, `SendTransactionResponse`, and `SendTransactionReject` (lines 64–103): no context lookup is performed at all; each call unconditionally pushes a new `ConsensusResponse` to `consensus_queue`.

**`DemuxImpl::process_payload` iterates without deduplication.** In `rs/messaging/src/routing/demux.rs` lines 86–94, every entry in `batch_messages.bitcoin_adapter_responses` is passed to `state.push_response_bitcoin(response)` with errors only logged, not used to halt processing.

**Execution layer processes all entries.** In `rs/execution_environment/src/scheduler.rs` lines 1299–1325, the scheduler drains `consensus_queue` with `while let Some(response) = state.consensus_queue.pop()` and calls `execute_subnet_message` for every entry without deduplication. The `consensus_queue` is a plain `Vec<ConsensusResponse>` with no duplicate-detection logic.

## Impact Explanation
A finalized block containing N copies of the same `callback_id` causes N `ConsensusResponse` entries to be pushed into `consensus_queue` and processed by the execution environment in the same round. The bitcoin canister's Wasm execution receives N responses to a single `bitcoin_get_successors` or `bitcoin_send_transaction` callback. The first response is processed normally; subsequent responses arrive after the callback has been consumed, causing the bitcoin canister's state machine to receive spurious responses or errors for already-handled callbacks. This constitutes concrete disruption of the Bitcoin integration canister's operation — a significant Chain Fusion / financial integration component — matching the allowed impact: **High ($2,000–$10,000): Significant Chain Fusion, ck-token, ledger, or infrastructure security impact with concrete user or protocol harm.** The double-minting claim for ckBTC is not directly proven by the evidence here, as it depends on the bitcoin canister's internal UTXO deduplication logic, which is not analyzed in this report.

## Likelihood Explanation
Requires a single Byzantine subnet node to act as block proposer in a round where at least one `bitcoin_get_successors_contexts` or `bitcoin_send_transaction_internal_contexts` entry is pending. This condition is routinely satisfied on the Bitcoin-enabled subnet. No threshold majority, no key material, and no privileged access are required. The attacker is a below-threshold Byzantine protocol peer, which is within the standard ICP attacker model.

## Recommendation
In `validate_payload`, after decoding the payload, collect all `callback_id` values into a `BTreeSet` and reject the payload if any duplicate is found:
```rust
let mut seen_ids = BTreeSet::new();
for response in &payload {
    if !seen_ids.insert(response.callback_id) {
        return Err(ValidationError::InvalidArtifact(
            consensus::InvalidPayloadReason::InvalidSelfValidatingPayload(
                InvalidSelfValidatingPayloadReason::DuplicateCallbackId,
            ),
        ));
    }
}
```
Additionally, use the already-computed `delivered_ids` to reject callback IDs that appear in past payloads. For `GetSuccessorsResponse` in `push_response`, replace `get_mut` with `remove` so the context is consumed on first use, making the function idempotent by construction.

## Proof of Concept
1. Push one `BitcoinGetSuccessorsContext` into `subnet_call_context_manager` (callback_id = 0).
2. Serialize two identical `BitcoinAdapterResponse { callback_id: 0, response: GetSuccessorsReject(...) }` entries using `iterator_to_bytes`.
3. Call `BitcoinPayloadBuilder::validate_payload` with this crafted byte slice and an empty `past_payloads` slice.
4. Assert the call returns `Ok(())` — demonstrating the missing guard.
5. Call `state.push_response_bitcoin` twice with the same `GetSuccessorsReject` response (callback_id = 0).
6. Assert `state.consensus_queue.len() == 2` — demonstrating non-idempotency.
