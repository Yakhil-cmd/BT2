### Title
Contract Pause Blocks `withdraw()` and `exit_to_near_precompile_callback()`, Causing Temporary and Potentially Permanent Fund Freeze - (File: engine/src/contract_methods/connector.rs)

### Summary
The Aurora Engine's `pause_contract` mechanism applies `require_running()` to all user-facing fund-movement functions, including `withdraw()` and `exit_to_near_precompile_callback()`. When the contract is paused, users cannot withdraw their ETH or ERC-20 tokens from Aurora. More critically, if a user's `ExitToNear` precompile call is in-flight when the contract is paused and the underlying `ft_transfer` fails, the refund path inside `exit_to_near_precompile_callback()` is also blocked by `require_running()`, causing permanent loss of the user's tokens.

### Finding Description
The `require_running()` guard is applied uniformly to every state-mutating function in the engine, including functions that are critical for users to recover their own funds.

**`pause_contract` sets `is_paused = true`:** [2](#0-1) 

**`withdraw()` is blocked when paused:** [3](#0-2) 

**`exit_to_near_precompile_callback()` is also blocked when paused — before the refund path is reached:** [4](#0-3) 

The refund branch at line 231 (`else if let Some(args) = args.refund`) is only reachable if `require_running()` at line 203 succeeds. If the contract is paused, the function returns `ERR_PAUSED` before any refund logic executes.

Additionally, `ft_transfer()`, `ft_transfer_call()`, `ft_on_transfer()`, `submit()`, `call()`, and `withdraw_wnear_to_router()` are all gated by `require_running()`: [5](#0-4) [6](#0-5) [7](#0-6) 

### Impact Explanation
**Temporary fund freeze (High):** While the contract is paused, any user holding ETH or ERC-20 tokens inside Aurora cannot call `withdraw()` to retrieve them. All EVM execution paths (`submit`, `call`) are also blocked, so users cannot interact with their own EVM contracts to move funds.

**Permanent fund freeze (Critical):** The `ExitToNear` precompile burns ERC-20 tokens in the EVM and schedules a NEAR promise (`ft_transfer` on the NEP-141 contract). If the contract is paused between the EVM execution and the callback, and the `ft_transfer` fails for any reason (e.g., the NEP-141 contract is also paused via `set_paused_flags`, or has a transient error), the NEAR runtime invokes `exit_to_near_precompile_callback` with a failed promise result. Because `require_running()` is checked first, the refund branch (`engine::refund_on_error`) is never reached. The tokens were already burned in the EVM and are permanently unrecoverable.

### Likelihood Explanation
Low. Protocol pauses are rare emergency events. However, the Aurora Engine has two independent pause mechanisms — `pause_contract` (engine-level) and `set_paused_flags` / `pa_pause_feature` (eth-connector level) — making the overlap scenario (engine paused while an in-flight exit fails) more plausible than in a single-pause system. The permanent loss scenario requires the coincidence of a pause and a failed `ft_transfer`, but both are independently possible.

### Recommendation
Remove `require_running()` from `withdraw()` and `exit_to_near_precompile_callback()` so that users can always retrieve their funds and in-flight exits can always be refunded, even during a pause. The pause should protect against new deposits and EVM execution, not against fund recovery:

### Proof of Concept
**Scenario 1 — Temporary freeze of `withdraw()`:**
1. User deposits ETH into Aurora; balance is credited in the EVM.
2. Owner calls `pause_contract`.
3. User calls `withdraw()` to retrieve their ETH.
4. `require_running()` returns `ERR_PAUSED`; the call reverts.
5. User's ETH is frozen inside Aurora for the duration of the pause.

**Scenario 2 — Permanent loss via `exit_to_near_precompile_callback()`:**
1. User submits an EVM transaction calling the `ExitToNear` precompile for 100 ERC-20 tokens.
2. EVM execution succeeds: 100 tokens are burned in the EVM; a NEAR promise to call `ft_transfer` on the NEP-141 contract is scheduled.
3. Owner calls `pause_contract`.
4. The NEP-141 `ft_transfer` fails (e.g., the eth-connector's `engine_withdraw` feature is paused via `pa_pause_feature`).
5. NEAR runtime invokes `exit_to_near_precompile_callback` with a failed promise result.
6. `require_running()` at line 203 returns `ERR_PAUSED`; the callback panics/reverts.
7. The refund branch (`engine::refund_on_error`) is never reached.
8. The 100 tokens are permanently lost: burned in the EVM, never transferred to NEAR, never refunded.

pub fn exit_to_near_precompile_callback<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        env.assert_private_call()?;

        // This function should only be called as the callback of
        // exactly one promise.
        if handler.promise_results_count() != 1 {
            return Err(errors::ERR_PROMISE_COUNT.into());
        }

        let args: ExitToNearPrecompileCallbackArgs = io.read_input_borsh()?;

        let maybe_result = if let Some(PromiseResult::Successful(_)) = handler.promise_result(0) {
            if let Some(args) = args.transfer_near {
                let action = PromiseAction::Transfer {
                    amount: Yocto::new(args.amount),
                };
                let promise = PromiseBatchAction {
                    target_account_id: args.target_account_id,
                    actions: vec![action],
                };

                // Safety: this call is safe because it comes from the exit to near precompile, not users.
                // The call is to transfer the unwrapped wNEAR tokens.
                let promise_id = handler.promise_create_batch(&promise);
                handler.promise_return(promise_id);
            }

            None
        } else if let Some(args) = args.refund {
            // Exit call failed; need to refund tokens
            let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;

            if !refund_result.status.is_ok() {
                return Err(errors::ERR_REFUND_FAILURE.into());
            }

            Some(refund_result)
        } else {
            None
        };

        Ok(maybe_result)
    })
}

pub fn withdraw_wnear_to_router<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<SubmitResult, ContractError> {
    with_logs_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        env.assert_private_call()?;
        if matches!(handler.promise_result_check(), Some(false)) {
            return Err(b"ERR_CALLBACK_OF_FAILED_PROMISE".into());
        }
        let args: WithdrawWnearToRouterArgs = io.read_input_borsh()?;
        let current_account_id = env.current_account_id();
        let recipient = AccountId::try_from(format!(
            "{}.{}",
            args.target.encode(),
            current_account_id.as_ref()
        ))?;
        let wnear_address = aurora_engine_precompiles::xcc::state::get_wnear_address(&io);
        let mut engine: Engine<_, E, AuroraModExp> = Engine::new_with_state(
            state,
            predecessor_address(&current_account_id),
            current_account_id,
            io,
            env,
        );
        let (result, ids) = xcc::withdraw_wnear_to_router(
            &recipient,
            args.amount,
            wnear_address,
            &mut engine,
            handler,
        )?;
        if !result.status.is_ok() {
            return Err(b"ERR_WITHDRAW_FAILED".into());
        }
        let id = ids.last().ok_or(b"ERR_NO_PROMISE_CREATED")?;
        handler.promise_return(*id);
        Ok(result)
    })
}
