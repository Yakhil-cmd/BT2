### Title
Permanent Token Loss When `near_withdraw` Fails in wNEAR Unwrap Exit Path — (`engine-precompiles/src/native.rs`)

---

### Summary

When a user exits wNEAR ERC-20 tokens to NEAR via the `ExitToNear` precompile with the `:unwrap` suffix, the ERC-20 tokens are burned before the `near_withdraw` NEAR promise executes. If `near_withdraw` fails and the `error_refund` compile-time feature is not enabled, the `exit_to_near_precompile_callback` has no refund path (`refund: None`), resulting in permanent loss of the burned tokens with no recovery mechanism.

---

### Finding Description

The `ExitToNear` precompile's `run` method constructs `ExitToNearPrecompileCallbackArgs` with a conditional refund field:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,
    transfer_near: transfer_near_args,
};
``` [1](#0-0) 

When `error_refund` is not enabled, `refund` is hardcoded to `None`. The callback `exit_to_near_precompile_callback` then handles the two outcomes:

```rust
let maybe_result = if let Some(PromiseResult::Successful(_)) = handler.promise_result(0) {
    if let Some(args) = args.transfer_near {
        // Transfer NEAR to target_account_id — only reached on success
        ...
    }
    None
} else if let Some(args) = args.refund {
    // Refund path — args.refund is None when error_refund is disabled
    let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;
    ...
} else {
    None  // ← silent no-op: tokens already burned, nothing returned
};
``` [2](#0-1) 

The wNEAR unwrap path in `exit_erc20_token_to_near` burns the ERC-20 tokens (via the ERC-20 contract calling the precompile) and then schedules `near_withdraw` on the wNEAR NEP-141 contract:

```rust
Some(Message::UnwrapWnear) if erc20_address == get_wnear_address(io).raw() => {
    (
        nep141_account_id,
        format!(r#"{{"amount":"{}"}}"#, exit_params.amount.as_u128()),
        "near_withdraw",
        Some(TransferNearArgs {
            target_account_id: exit_params.receiver_account_id.clone(),
            amount: exit_params.amount.as_u128(),
        }),
        ...
    )
}
``` [3](#0-2) 

The ERC-20 burn is irreversible and occurs before the cross-contract `near_withdraw` call. If `near_withdraw` fails (e.g., wNEAR contract paused, insufficient NEP-141 balance on the Aurora contract side, or any other rejection), the callback receives a non-successful promise result, `args.refund` is `None`, and the `else { None }` branch executes silently — no NEAR is delivered and no ERC-20 tokens are restored.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

The user's wNEAR ERC-20 tokens are burned from their EVM balance before the NEAR-side promise executes. If the NEAR promise fails and `error_refund` is not compiled in, the tokens are permanently destroyed with no recovery path. The user loses the full exit amount.

---

### Likelihood Explanation

**Medium.** The `near_withdraw` call targets the wNEAR NEP-141 contract. Failure conditions include: the wNEAR contract being paused, the Aurora engine contract not holding sufficient NEP-141 balance, or any transient NEAR runtime error. The `error_refund` feature being opt-in (not the default) means production deployments that omit it are fully exposed. Any user calling the wNEAR unwrap exit path is at risk.

---

### Recommendation

1. Enable the `error_refund` feature in all production builds, or make the refund path unconditional (not gated by a compile-time feature).
2. Alternatively, restructure the exit flow so that the ERC-20 burn only occurs after confirmation that the NEAR-side promise succeeded (i.e., use a two-phase commit pattern).
3. At minimum, document clearly that without `error_refund`, failed exits result in permanent token loss.

---

### Proof of Concept

1. User holds wNEAR ERC-20 tokens on Aurora.
2. User calls the `ExitToNear` precompile with flag `0x1`, amount, and receiver account with `:unwrap` suffix.
3. The ERC-20 contract burns the user's wNEAR ERC-20 tokens and calls the precompile.
4. The precompile schedules `near_withdraw` on the wNEAR NEP-141 contract with `transfer_near` args and `refund: None` (because `error_refund` is not enabled).
5. `near_withdraw` fails (e.g., wNEAR contract is paused).
6. `exit_to_near_precompile_callback` is invoked: `promise_result(0)` is not `Successful`, `args.refund` is `None`, the `else { None }` branch executes.
7. No NEAR is transferred to the user. No ERC-20 tokens are restored. Funds are permanently lost. [4](#0-3) [2](#0-1)
