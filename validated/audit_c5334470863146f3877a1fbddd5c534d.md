### Title
Concurrent `session_request` Events Open Multiple Confirmation Dialogs, Enabling WalletConnect Approval Bypass — (`packages/gui/src/electron/utils/dispatchPairRequest.ts`, `packages/gui/src/electron/main.tsx`)

---

### Summary

`dispatchPairRequest` has no concurrency guard. A malicious dApp can send two simultaneous `session_request` events for the same pair and command, causing two independent confirmation dialog windows to open. A user who dismisses one dialog believing they rejected the command leaves the second dialog open; approving it executes the command, bypassing the user's intended rejection.

---

### Finding Description

**Entry point — renderer (`WalletConnectProvider.tsx` line 297–390)**

Each `session_request` event fires `processSessionRequest` independently. There is no per-pair or per-command in-flight tracking: [1](#0-0) 

Two concurrent events both reach `currentProcess(pairTopic, method, commandParams, ...)` without any serialization.

**IPC handler — main process (`main.tsx` line 262–373)**

`ipcMainHandle` is a bare `ipcMain.handle` wrapper with no queuing: [2](#0-1) 

Electron's `ipcMain.handle` allows concurrent invocations of the same channel. Both calls proceed independently into `dispatchPairRequest`.

**`dispatchPairRequest` — no mutex (`dispatchPairRequest.ts` line 14–66)**

The function performs authorization checks and then calls `confirm()` with no guard against a concurrent in-flight call for the same `(topic, command)` pair: [3](#0-2) 

**`openReactDialog` — creates a new `BrowserWindow` per call (`openReactDialog.tsx` line 266–291)**

Each call to `confirm()` calls `openReactDialog`, which unconditionally creates a new `BrowserWindow` dialog. There is no check for an already-open dialog for the same pair or command: [4](#0-3) 

Two concurrent calls produce two independent dialog windows, each with its own `resolveChannelId`/`rejectChannelId` and its own promise. Resolving or rejecting one has no effect on the other.

---

### Impact Explanation

A user who sees two identical `chia_sendTransaction` dialogs and dismisses one (intending to reject the transaction) leaves the second dialog alive. If the user then approves the second dialog — or is socially engineered into doing so — the transaction is submitted. This is a concrete bypass of the WalletConnect approval gate: the user's rejection of one dialog does not cancel the other, and the command executes against their intent.

---

### Likelihood Explanation

Any connected dApp can send two `session_request` messages in rapid succession over the WalletConnect relay. The WalletConnect SDK imposes no rate limit on concurrent requests per session. No privilege beyond an established pair is required. The attack is trivially scriptable.

---

### Recommendation

Serialize `DISPATCH_AS_PAIR` calls per `(topic, command)` pair using a per-key promise chain or mutex. Before opening a confirmation dialog, check whether one is already pending for the same `(topic, command)` and either queue the second request or reject it immediately with `USER_REJECTED`. The existing `limit` utility in `packages/gui/src/util/limit.ts` provides a concurrency-limiting primitive that could be adapted for this purpose. [5](#0-4) 

---

### Proof of Concept

1. Establish a WalletConnect pair with `chia_sendTransaction` granted and no bypass.
2. From the dApp, send two `session_request` events for `chia_sendTransaction` with identical parameters simultaneously (e.g., `Promise.all([client.request(...), client.request(...)])`).
3. Observe that two confirmation dialog windows open in the Chia GUI.
4. Dismiss (cancel) the first dialog.
5. Observe that the second dialog remains open and functional.
6. Click "Approve" on the second dialog.
7. Observe that the transaction is submitted — despite the user having dismissed the first dialog with the intent to reject.

export default function limit(concurrency: number) {
  const queue: {
    func: Function;
    resolve: (value: any) => void;
    reject: (error: Error) => void;
  }[] = [];
  let active = 0;

  async function execute() {
    if (!queue.length || active >= concurrency) {
      return;
    }

    active++;

    const item = queue.shift();
    if (!item) {
      return;
    }

    const { func, resolve, reject } = item;
    try {
      const result = await func();
      resolve(result);
    } catch (e) {
      reject(e as Error);
    } finally {
      active--;
      execute();
    }
  }

  function add<T>(func: Function) {
    return new Promise<T>((resolve, reject) => {
      queue.push({ func, resolve, reject });
      execute();
    });
  }

  return add;
}
