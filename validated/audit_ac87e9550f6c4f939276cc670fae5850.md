`max_nonce_for_validation_skip` Config Field Never Read in Rust Gateway — `__validate__` Skip Boundary Is Hardcoded and Uncontrollable - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

`StatefulTransactionValidatorConfig` declares a field `max_nonce_for_validation_skip` documented as "Maximum nonce for which the validation is skipped." The Rust gateway's `skip_stateful_validations` free function ignores this field entirely and hardcodes the boundary to `Nonce(Felt::ONE)`. Any invoke transaction with `tx.nonce == 1` and `account_nonce == 0` whose sender has any transaction in the mempool or a recent block will have its `__validate__` entry-point call suppressed at the gateway, regardless of how the operator configures `max_nonce_for_validation_skip`. An attacker who controls the deploy-account submission for a fresh address can therefore inject an invoke with an arbitrary (invalid) signature into the mempool without triggering account-level signature verification.

---

### Finding Description

`StatefulTransactionValidatorConfig` in `crates/apollo_gateway_config/src/config.rs` defines:

```rust
pub max_nonce_for_validation_skip: Nonce,   // default: Nonce(Felt::ONE)
```

with the serialised description *"Maximum nonce for which the validation is skipped."* [1](#0-0) 

The legacy Python-gateway path (`PyValidator::should_run_stateful_validations`) correctly reads this field:

```rust
let nonce_small_enough_to_qualify_for_validation_skip =
    tx_nonce <= self.max_nonce_for_validation_skip;
let skip_validate = deploy_account_not_processed
    && is_post_deploy_nonce
    && nonce_small_enough_to_qualify_for_validation_skip;
``` [2](#0-1) 

The production Rust gateway path calls the free function `skip_stateful_validations`, which **never receives `self.config`** and hardcodes the boundary:

```rust
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
    return mempool_client
        .account_tx_in_pool_or_recent_block(tx.sender_address())
        .await …;
}
``` [3](#0-2) 

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` sets `validate: false` in `ExecutionFlags`, so the blockifier's `StatefulValidator::validate` path is never reached for that transaction:

```rust
ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [4](#0-3) 

The `__validate__` entry point — which is where account contracts verify the caller's ECDSA signature — is therefore never executed at the gateway for these transactions. [5](#0-4) 

The grep corpus confirms `max_nonce_for_validation_skip` appears **zero times** inside `crates/apollo_gateway/src/stateful_transaction_validator.rs`; it is dead configuration in the production gateway.

### Impact Explanation

**Matching impact bucket:** *High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.*

An invoke transaction whose `__validate__` entry point would reject it (wrong signature, wrong calldata hash, any account-defined guard) is admitted to the mempool and forwarded to the batcher. The batcher will eventually reject it during block building, but:

1. The invalid transaction occupies a mempool slot and consumes sequencer CPU during block-building validation.
2. Because the transaction fails in `__validate__` during execution, **no fee is charged** to the sender — the attacker bears zero cost for the spam.
3. An operator who sets `max_nonce_for_validation_skip = 0` to harden the gateway (e.g., to prevent this exact UX bypass) receives no protection; the hardcoded `Nonce(Felt::ONE)` check fires regardless.

The corrupted admission decision is: *a transaction that should be rejected at the gateway (invalid signature, `__validate__` would fail) is instead accepted and queued.*

---

### Likelihood Explanation

Exploitation requires only standard, unprivileged gateway access:

1. Attacker generates a fresh key pair and computes the corresponding Starknet account address.
2. Attacker submits a valid `DeployAccount` transaction for that address (this passes all checks because the attacker controls the key).
3. Attacker immediately submits an `Invoke` with `nonce = 1` and a **garbage signature** from the same address.
4. `skip_stateful_validations` fires: `tx.nonce() == 1`, `account_nonce == 0`, `account_tx_in_pool_or_recent_block` returns `true` (the deploy-account is in the pool).
5. The invoke is admitted to the mempool with `validate: false`; no signature check is performed.

No privileged access, no special contract, no prior on-chain state is required. The attacker can repeat this indefinitely at the cost of one deploy-account transaction per fresh address.

---

### Recommendation

Convert `skip_stateful_validations` from a free function into a method on `StatefulTransactionValidator` (or pass the config explicitly) and apply the configured bound:

```rust
async fn skip_stateful_validations(
    &self,                          // ← add self
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        if tx.nonce() == Nonce(Felt::ONE)
            && account_nonce == Nonce(Felt::ZERO)
            && tx.nonce() <= self.config.max_nonce_for_validation_skip  // ← enforce config
        {
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await …;
        }
    }
    Ok(false)
}
```

Setting `max_nonce_for_validation_skip = Nonce(Felt::ZERO)` in the deployment config should then fully disable the UX bypass and restore mandatory `__validate__` execution for all incoming invoke transactions.

### Proof of Concept

```
# Step 1 – generate a fresh account address (off-chain, any Starknet SDK)
PRIV=0xdeadbeef...
ADDR=$(starknet_address $PRIV $CLASS_HASH $SALT)

# Step 2 – submit a valid DeployAccount (nonce=0, correct signature)
curl -X POST $GATEWAY/add_transaction \
  -d '{"type":"DEPLOY_ACCOUNT","nonce":"0x0","signature":["<valid_r>","<valid_s>"],...}'
# → accepted, deploy-account enters mempool

# Step 3 – submit an Invoke with nonce=1 and a WRONG signature
curl -X POST $GATEWAY/add_transaction \
  -d '{"type":"INVOKE","nonce":"0x1","signature":["0x1","0x2"],...}'
# → skip_stateful_validations fires (nonce==1, account_nonce==0,
#   account_tx_in_pool_or_recent_block==true)
# → run_validate_entry_point called with validate=false
# → __validate__ is NEVER executed
# → transaction admitted to mempool with invalid signature
```

The invalid invoke will be rejected by the blockifier during block building (no fee charged), confirming zero-cost mempool pollution. Repeating steps 1–3 with fresh addresses scales the attack linearly.
