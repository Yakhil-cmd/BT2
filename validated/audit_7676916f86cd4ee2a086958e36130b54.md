### Title
Truncating Division in `native_per_pubdata` Computation Causes Systematic Undercharging for Pubdata - (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

The ZKsync OS bootloader computes `native_per_pubdata` — the native resource cost per byte of pubdata — using integer floor division (`wrapping_div`) of `pubdata_price / native_price`. This truncation is asymmetric with the ceiling division used for `native_per_gas`. The result is that the native resource budget allocated for pubdata is systematically lower than the actual cost the operator incurs, allowing a transaction sender to consume more pubdata than they pay for in native resources.

---

### Finding Description

In `validate_and_compute_fee_for_transaction` in `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`, the two key per-unit native costs are computed as:

```rust
// native_per_gas uses ceiling division
let native_per_gas = u256_try_to_u64(&gas_price.div_ceil(native_price))...

// native_per_pubdata uses FLOOR (truncating) division
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))...
``` [1](#0-0) 

`native_per_gas` uses `div_ceil` (ceiling division), which ensures the user is charged at least enough native per gas unit. However, `native_per_pubdata` uses `wrapping_div`, which is floor division. This means:

- If `pubdata_price = 10` and `native_price = 3`, then `native_per_pubdata = 3` (floor), but the true ratio is `3.33...`.
- The user's native budget for pubdata is computed as `pubdata_bytes * 3` instead of `pubdata_bytes * ceil(10/3) = pubdata_bytes * 4`.
- The user is undercharged by up to `(native_price - 1)` native units per pubdata byte.

This `native_per_pubdata` value is then used in two critical places:

1. **Intrinsic pubdata overhead deduction** in `create_resources_for_tx`:
   ```rust
   let intrinsic_pubdata_overhead = native_per_pubdata_byte.saturating_mul(intrinsic_pubdata);
   let native_limit = native_limit.checked_sub(intrinsic_pubdata_overhead)...
   ``` [2](#0-1) 

2. **Post-execution pubdata charging** in `get_resources_to_charge_for_pubdata`:
   ```rust
   let native = current_pubdata_spent.checked_mul(native_per_pubdata)...
   ``` [3](#0-2) 

Both use the truncated `native_per_pubdata`, so the user's native budget is set too high (less is deducted for pubdata), and the post-execution pubdata charge is too low.

The same truncation is replicated in the public API helper `validate_l2_tx_intrinsic_native_resources`:
```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
``` [4](#0-3) 

---

### Impact Explanation

The truncation error is bounded by `(native_price - 1)` native units per pubdata byte. For each pubdata byte consumed, the user pays up to `(native_price - 1)` fewer native units than the true cost. Over a transaction that writes many pubdata bytes (e.g., many storage slots), this accumulates:

- A transaction writing `N` pubdata bytes underpays by up to `N * (native_price - 1)` native units.
- The `delta_gas` adjustment in `compute_gas_refund` only compensates when native consumption exceeds gas consumption in the computational dimension; it does not retroactively correct the pubdata undercharge because pubdata native is charged separately from computational native.
- The operator receives less fee than the actual cost of publishing the pubdata on L1, creating a systematic loss per transaction that writes pubdata.

An attacker can craft transactions that maximize pubdata output (e.g., writing to many new storage slots) to maximize the per-transaction underpayment. The attacker pays less in native resources than the operator must spend to publish the data. [5](#0-4) 

---

### Likelihood Explanation

- The condition is triggered on every L2 transaction where `pubdata_price` is not an exact multiple of `native_price`, which is the common case in production.
- No special privileges are required; any unprivileged transaction sender submitting a transaction that writes storage (pubdata) triggers this path.
- The operator sets `native_price` and `pubdata_price` independently; the truncation error is maximized when `native_price` is large (e.g., close to `pubdata_price`).
- The entry path is the standard L2 transaction validation flow, reachable by any user.

---

### Recommendation

Replace `wrapping_div` with `div_ceil` for `native_per_pubdata` in `validation_impl.rs` to match the rounding direction used for `native_per_gas`:

```rust
// Before (floor division — undercharges pubdata):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))...

// After (ceiling division — consistent with native_per_gas):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))...
```

Apply the same fix to `api/src/helpers.rs` in `validate_l2_tx_intrinsic_native_resources`. [6](#0-5) 

---

### Proof of Concept

**Setup:**
- `native_price = 3`
- `pubdata_price = 10`
- Transaction writes 1000 pubdata bytes

**Current behavior (floor division):**
- `native_per_pubdata = 10 / 3 = 3` (truncated)
- Native charged for pubdata = `1000 * 3 = 3000`

**Correct behavior (ceiling division):**
- `native_per_pubdata = ceil(10 / 3) = 4`
- Native charged for pubdata = `1000 * 4 = 4000`

**Underpayment:** 1000 native units per transaction. The user's `fee_to_prepay` is computed from `gas_price * gas_limit` (in tokens), but the native budget for pubdata is derived from the truncated `native_per_pubdata`. The user can consume pubdata that costs the operator 4000 native units while only being charged for 3000, a 25% discount on pubdata costs. At scale (many transactions, large pubdata), this represents a material loss to the operator/protocol. [7](#0-6) [8](#0-7)

```rust
pub fn get_resources_to_charge_for_pubdata<S: EthereumLikeTypes>(
    system: &mut System<S>,
    native_per_pubdata: u64,
    base_pubdata: Option<u64>,
) -> Result<(u64, S::Resources), SystemError> {
    let current_pubdata_spent = system
        .net_pubdata_used()?
        .saturating_sub(base_pubdata.unwrap_or(0));
    let native = current_pubdata_spent
        .checked_mul(native_per_pubdata)
        .ok_or(out_of_native_resources!())?;
    let native = <S::Resources as zk_ee::system::Resources>::Native::from_computational(native);
    Ok((current_pubdata_spent, S::Resources::from_native(native)))
}
