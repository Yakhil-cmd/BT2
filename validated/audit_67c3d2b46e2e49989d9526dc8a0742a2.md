### Title
Gateway Admits Invoke Transactions with Unverified Signatures via Overly Broad `skip_stateful_validations` Check - (File: crates/apollo_gateway/src/stateful_transaction_validator.rs)

### Summary

The `skip_stateful_validations` function bypasses the account's `__validate__` entry point for invoke transactions with nonce=1 when `account_tx_in_pool_or_recent_block` returns `true`. That helper returns `true` for **any** transaction from the address in the pool — not specifically a `deploy_account` transaction. An attacker who observes a pending `deploy_account` in the mempool can submit an invoke with nonce=1 carrying an **invalid signature**, and the gateway will admit it without ever calling the account's authorization entry point.

### Finding Description

`skip_stateful_validations` in `crates/apollo_gateway/src/stateful_transaction_validator.rs` is the UX shortcut that lets a user broadcast `deploy_account` + `invoke(nonce=1)` simultaneously. When the condition `tx.nonce() == 1 && account_nonce == 0` is met, the function delegates the "is a deploy_account pending?" question to `account_tx_in_pool_or_recent_block`:

```rust

    return mempool_client
        .account_tx_in_pool_or_recent_block(tx.sender_address())
        .await
        ...
}
``` [1](#0-0) 

The mempool implementation of that helper is:

```rust
// lines 697-700
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [2](#0-1) 

`tx_pool.contains_account` returns `true` for **any** transaction type stored for that address — `Invoke`, `Declare`, or `DeployAccount`. There is no type filter. The code comment claims this is safe because the account "either has a deploy_account transaction or transactions with future nonces that passed validations," but that reasoning is circular: those "future-nonce transactions that passed validations" were themselves admitted via the same skip path.

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` sets `execution_flags.validate = false`:

```rust
// lines 311-312
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [3](#0-2) 

`StatefulValidator::perform_validations` then short-circuits before calling `__validate__`:

```rust
// lines 79-81
if !tx.execution_flags.validate {
    return Ok(());
}
``` [4](#0-3) 

The only checks that still run are the pre-validation stage (nonce increment, fee-bound check, balance check) — none of which verify the cryptographic signature. `validate_by_mempool`, called just before, only checks for duplicate hashes and nonce ordering:

```rust
// lines 402-408
pub fn validate_tx(&mut self, args: ValidationArgs) -> MempoolResult<()> {
    let tx_reference = (&args).into();
    self.validate_incoming_tx(tx_reference, args.account_nonce)?;
    self.validate_fee_escalation(tx_reference)?;
    Ok(())
}
``` [5](#0-4) 

No step in the gateway pipeline verifies the invoke's signature when `skip_validate` is `true`.

### Impact Explanation

An attacker who observes a victim's `deploy_account` transaction in the public mempool can immediately submit an `invoke(nonce=1)` for the same address carrying an **arbitrary or invalid signature**. The gateway admits it because `account_tx_in_pool_or_recent_block` returns `true` (the `deploy_account` is in the pool) and `__validate__` is never called. The transaction enters the mempool as a seemingly valid pending transaction. When the batcher later executes the block, `deploy_account` succeeds, then the invoke runs with `execution_flags.validate = true` (the batcher uses default flags), `__validate__` is called, it fails, the transaction reverts, and the fee is charged from the newly-deployed account's balance. This satisfies **"High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing"**: an unauthorized transaction (wrong signer) is accepted through the gateway admission path.

### Likelihood Explanation

The mempool is publicly observable. Any `deploy_account` broadcast by a user is visible before it is included in a block. The attacker only needs to race-submit an `invoke(nonce=1)` with a crafted (invalid) signature for the same sender address. No privileged access is required. The window is the entire time the `deploy_account` sits in the mempool waiting to be sequenced.

### Recommendation

Replace the generic `account_tx_in_pool_or_recent_block` check with a type-specific query that returns `true` only when a `DeployAccount` transaction for the address is present in the pool or was recently committed. Expose a dedicated mempool method such as `has_pending_deploy_account(address: ContractAddress) -> bool` that inspects the transaction type stored in `tx_pool` for the address, and use that in `skip_stateful_validations` instead of the current broad check.

### Proof of Concept

1. Victim pre-funds counterfactual address `A` and submits `deploy_account(A)` → accepted, visible in mempool.
2. Attacker calls `POST /gateway/add_transaction` with `invoke { sender_address: A, nonce: 1, signature: [0xdead, 0xbeef] }` (invalid signature).
3. Gateway stateful validator: `account_nonce = 0`, `tx.nonce() = 1` → enters `skip_stateful_validations` → `account_tx_in_pool_or_recent_block(A)` returns `true` (deploy_account is in pool) → returns `true` (skip).
4. `run_validate_entry_point` sets `validate = false`; `StatefulValidator::perform_validations` returns `Ok(())` after pre-validation without calling `__validate__`. [6](#0-5) 
5. Transaction is forwarded to the mempool and admitted. [7](#0-6) 
6. Batcher sequences the block: `deploy_account(A)` executes successfully; `invoke(A, nonce=1, bad_sig)` executes with `validate=true`, `__validate__` panics/reverts, fee is charged from `A`'s balance.

The invalid transaction passed gateway admission — the broken invariant is that every admitted invoke must have its account's `__validate__` entry point verified or the account must not yet exist with no pending deploy.
