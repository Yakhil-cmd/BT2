### Title
Stale Cadence Program Cache Not Invalidated on VersionBeacon Cadence Feature-Flag Activation — (File: `module/state_synchronization/indexer/indexer_core.go`)

---

### Summary

Access/Observer nodes running local script execution with the experimental program cache enabled (`--program-cache-size > 0`) do not flush cached Cadence programs when a `VersionBeacon` service event changes the `MinimumRequiredVersion` (which controls Cadence feature flags). Programs compiled and type-checked under old, potentially more permissive feature flags remain in cache after a Cadence version upgrade, bypassing new restrictions introduced by the flag change — the direct analog of the Solana executor cache inconsistency.

---

### Finding Description

Flow's `DerivedChainData` / `DerivedBlockData` cache stores compiled Cadence `*runtime.Program` objects across blocks. Each `*runtime.Program` is the result of parsing **and type-checking** a contract under the Cadence feature-flag state active at compilation time. That feature-flag state is determined by `MinimumRequiredVersion`, which is derived from the `VersionBeacon` service event. [1](#0-0) 

The `IndexerCore.updateProgramCache` function — the sole cache-maintenance path for Access/Observer nodes — invalidates cached programs only when:

1. A contract was updated (`accountContractUpdated` events via `findContractUpdates`), or
2. A service-account-authorized transaction appeared in the block (`hasAuthorizedTransaction`). [2](#0-1) 

A `VersionBeacon` service event is **neither** an `accountContractUpdated` event nor a service-account-authorized transaction. Consequently, when a `VersionBeacon` advances the `MinimumRequiredVersion` at block height H, the `programInvalidator` is constructed with `invalidateAll = false` and an empty `invalidated` map, leaving every cached program intact. [3](#0-2) 

The `DerivedChainData.GetOrCreateDerivedBlockData` propagates the stale cache forward by inheriting all parent-block programs into the child block: [4](#0-3) 

The same gap exists in the Execution-Node path: `DerivedDataInvalidator.ExecutionParametersUpdated` is set only when meter-state registers are written, not when a `VersionBeacon` changes the protocol-level `MinimumRequiredVersion`. [5](#0-4) 

---

### Impact Explanation

Cadence feature flags gated by `MinimumRequiredVersion` can introduce stricter type-checking, new capability/entitlement restrictions, or tighter access-control rules. A `*runtime.Program` cached before the flag transition was type-checked under the old, more permissive rules. After the transition, scripts executed against that cached program on an Access/Observer node bypass the new restrictions — the program is never re-type-checked. An Access API caller can therefore obtain script results that should be rejected or restricted under the new Cadence rules, constituting private data exposure or entitlement bypass via API validation. Because script execution on Access/Observer nodes has no downstream verification step (unlike Execution Node results, which are checked by Verification Nodes), the incorrect results are served silently.

---

### Likelihood Explanation

Low-to-medium. The program cache is experimental and **disabled by default** (`--program-cache-size 0`), so only operators who explicitly opt in are exposed. [6](#0-5) 

`VersionBeacon` transitions are infrequent (they occur during coordinated Cadence upgrades), but the window of inconsistency persists for every block after the transition until the node restarts or the cache is otherwise cleared. Any Access API caller who knows a version transition has occurred can exploit the inconsistency by querying a cached node versus a cold node.

---

### Recommendation

In `IndexerCore.updateProgramCache`, detect `VersionBeacon` service events in the block's events (or compare the current `MinimumRequiredVersion` against the parent block's version). When a version change is detected, set `invalidateAll = true` on both the `programInvalidator` and `executionParametersInvalidator`, mirroring the existing `hasAuthorizedTransaction` path.

Similarly, `DerivedDataInvalidator` should be extended with a `CadenceVersionUpdated` flag (analogous to `ExecutionParametersUpdated`) that is set whenever the `VersionBeacon`-derived `MinimumRequiredVersion` changes between the parent and current block, triggering full program cache invalidation. [7](#0-6) 

---

### Proof of Concept

1. Start an Access/Observer node with `--program-cache-size 100` (experimental cache enabled).
2. Deploy a Cadence contract `C` that is valid under the current `MinimumRequiredVersion` (feature flags off).
3. Execute a script that imports `C`; the `*runtime.Program` for `C` is type-checked under the old flags and stored in `DerivedBlockData`.
4. Submit a governance transaction that emits a `VersionBeacon` service event advancing `MinimumRequiredVersion` to a version that enables a stricter Cadence feature flag (e.g., one that would reject `C`'s type usage).
5. Wait for the `VersionBeacon` to be sealed and indexed by `IndexerCore.IndexBlockData`. Observe that `updateProgramCache` does **not** set `invalidateAll = true` because no `accountContractUpdated` event was emitted and no service-account-authorized transaction appeared.
6. Execute the same script again. The Access node serves the cached `*runtime.Program` for `C` (type-checked under the old, permissive flags) rather than recompiling under the new flags. The script succeeds and returns data that should be restricted under the new Cadence rules, while a cold Access node (no cache) correctly rejects or restricts the same script.

// MinimumRequiredVersion The returned cadence version can be used by cadence runtime for supporting feature flag.
// The feature flag in cadence allows ENs to produce consistent results even if running with
// different cadence versions at the same height, which is useful for rolling out cadence
// upgrade without all ENs restarting all together.
// For instance, we would like to grade cadence from v1 to v3, where v3 has a new cadence feature.
// We first make a cadence v2 that has feature flag only turned on when the MinimumRequiredVersion()
// method returns v2 or above.
// So cadence v2 with the feature flag turned off will produce the same result as v1 which doesn't have the feature.
// And cadence v2 with the feature flag turned on will also produce the same result as v3 which has the feature.
// The feature flag allows us to roll out cadence v2 to all ENs which was running v1.
// And we use the MinimumRequiredVersion to control when the feature flag should be switched from off to on.
// And the switching should happen at the same height for all ENs.
//
// The height-based switch over can be done by using VersionBeacon, however, the VersionBeacon only
// defines the flow-go version, not cadence version.
// So we first read the current minimum required flow-go version from the VersionBeacon control,
// and map it to the cadence version to be used by cadence to decide feature flag status.
//
// For instance, let’s say all ENs are running flow-go v0.37.0 with cadence v1.
// We first create a version mapping entry for flow-go v0.37.1 to cadence v2, and roll out v0.37.1 to all ENs.Z
// v0.37.1 ENs will produce the same result as v0.37.0 ENs, because the current version beacon still returns v0.37.0,
// which maps zero cadence version, and cadence will keep the feature flag off.
//
// After all ENs have upgraded to v0.37.1, we send out a version beacon to switch to v0.37.1 at a future height,
// let’s say height 1000.
// Then what happens is that:
//  1. ENs running v0.37.0 will crash after height 999, until upgrade to higher version
//  2. ENs running v0.37.1 will execute with cadence v2 with feature flag off up until height 999, and from height 1000,
//     the feature flag will be on, which means all v0.37.1 ENs will again produce consistent results for blocks above 1000.
//
// After height 1000 have been sealed, we can roll out v0.37.2 to all ENs with cadence v3, and it will produce the consistent
// result as v0.37.1.
func (c minimumCadenceRequiredVersion) MinimumRequiredVersion() (string, error) {
	executionVersion, err := c.executionVersionProvider.ExecutionVersion()
	if err != nil {
		return "", errors.NewExecutionVersionProviderFailure(err)
	}

	// map the minimum required flow-go version to a minimum required cadence version
	cadenceVersion := mapToCadenceVersion(executionVersion, minimumFvmToMinimumCadenceVersionMapping)

	return cadenceVersion.String(), nil

func (c *IndexerCore) updateProgramCache(header *flow.Header, events []flow.Event, collections []*flow.Collection) error {
	if c.derivedChainData == nil {
		return nil
	}

	derivedBlockData := c.derivedChainData.GetOrCreateDerivedBlockData(
		header.ID(),
		header.ParentID,
	)

	// get a list of all contracts that were updated in this block
	updatedContracts, err := findContractUpdates(events)
	if err != nil {
		return fmt.Errorf("could not find contract updates for block %d: %w", header.Height, err)
	}

	// invalidate cache entries for all modified programs
	tx, err := derivedBlockData.NewDerivedTransactionData(0, 0)
	if err != nil {
		return fmt.Errorf("could not create derived transaction data for block %d: %w", header.Height, err)
	}

	tx.AddInvalidator(&accessInvalidator{
		programs: &programInvalidator{
			invalidated:   updatedContracts,
			invalidateAll: hasAuthorizedTransaction(collections, c.serviceAddress),
		},
		executionParameters: &executionParametersInvalidator{
			invalidateAll: hasAuthorizedTransaction(collections, c.serviceAddress),
		},
	})

	err = tx.Commit()
	if err != nil {
		return fmt.Errorf("could not commit derived transaction data for block %d: %w", header.Height, err)
	}

	return nil
}

func (chain *DerivedChainData) GetOrCreateDerivedBlockData(
	currentBlockId flow.Identifier,
	parentBlockId flow.Identifier,
) *DerivedBlockData {
	chain.mutex.Lock()
	defer chain.mutex.Unlock()

	currentEntry := chain.unsafeGet(currentBlockId)
	if currentEntry != nil {
		return currentEntry
	}

	var current *DerivedBlockData
	parentEntry, ok := chain.lru.Get(parentBlockId)
	if ok {
		current = parentEntry.NewChildDerivedBlockData()
	} else {
		current = NewEmptyDerivedBlockData(0)
	}

	chain.lru.Add(currentBlockId, current)
	return current
}

// executionParametersUpdated returns true if the meter param overrides have been updated
// this is done by checking if the registers needed to compute the meter param overrides
// have been touched in the execution snapshot
func executionParametersUpdated(
	executionSnapshot *snapshot.ExecutionSnapshot,
	meterStateRead *snapshot.ExecutionSnapshot,
) bool {
	if len(executionSnapshot.WriteSet) > len(meterStateRead.ReadSet) {
		for registerId := range meterStateRead.ReadSet {
			_, ok := executionSnapshot.WriteSet[registerId]
			if ok {
				return true
			}
		}
	} else {
		for registerId := range executionSnapshot.WriteSet {
			_, ok := meterStateRead.ReadSet[registerId]
			if ok {
				return true
			}
		}
	}

	return false
}
