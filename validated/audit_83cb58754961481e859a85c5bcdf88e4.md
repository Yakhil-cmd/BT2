### Title
ETH Sent to a Same-Transaction SELFDESTRUCT Contract After Destruction Is Permanently Burned — (`basic_system/src/system_implementation/flat_storage_model/account_cache.rs`, `basic_system/src/system_implementation/ethereum_storage_model/caches/account_cache.rs`)

---

### Summary

In ZKsync OS, when a contract created in the same transaction calls `SELFDESTRUCT`, the balance is **immediately** transferred to the beneficiary and the contract is flagged `is_marked_for_deconstruction`. If any ETH is subsequently sent to that contract (still within the same transaction), `finish_tx` silently zeros the contract's balance at transaction end, permanently burning the ETH instead of forwarding it to the beneficiary. This diverges from Ethereum EIP-6780 semantics and causes irreversible loss of user funds.

---

### Finding Description

**Ethereum EIP-6780 semantics (correct behavior):**
Under EIP-6780, `SELFDESTRUCT` on a same-transaction contract schedules both the balance transfer *and* the contract deletion for the **end** of the transaction. Any ETH received by the contract after `SELFDESTRUCT` (but still within the same transaction) is therefore included in the final balance transfer to the beneficiary.

**ZKsync OS behavior (divergent):**

`mark_for_deconstruction` is called immediately when the `SELFDESTRUCT` opcode executes:

```
evm_interpreter/src/instructions/host.rs:268-276
```

Inside `mark_for_deconstruction` (flat storage model):

1. The balance is **immediately** transferred to the beneficiary (or zeroed if `same_address`).
2. The contract is flagged `is_marked_for_deconstruction = true`. [1](#0-0) 

At the end of the transaction, `finish_tx` iterates over all pending changes and, for any account with `is_marked_for_deconstruction = true`, resets the entire account to `AccountProperties::TRIVIAL_VALUE` — which includes zeroing the balance: [2](#0-1) 

The same pattern exists in the Ethereum storage model: [3](#0-2) 

**The gap:** Between the immediate balance transfer in `mark_for_deconstruction` and the final zeroing in `finish_tx`, the contract can still receive ETH (its code is still present; no guard prevents a `CALL` with value to it). Any ETH received in this window is silently burned by `finish_tx` — it is neither forwarded to the beneficiary nor returned to the sender.

The `transfer_nominal_token_value` path that handles incoming ETH does not check whether the recipient is `is_marked_for_deconstruction`: [4](#0-3) 

---

### Impact Explanation

Any ETH sent to a same-transaction selfdestructed contract after the `SELFDESTRUCT` opcode executes is permanently burned. The beneficiary address receives less ETH than it should (only the balance at the moment of `SELFDESTRUCT`, not the balance at end-of-transaction). This is a **permanent, irreversible loss of user funds** — the analog of "permanent freezing of unclaimed yield" from the reference report.

The `docs/not-a-bug.md` file does not document this divergence from EIP-6780 as intentional. [5](#0-4) 

---

### Likelihood Explanation

The scenario requires:
1. A contract deployed within the same transaction (e.g., via `CREATE`/`CREATE2` in a factory pattern).
2. That contract calls `SELFDESTRUCT` with a beneficiary.
3. The calling contract (or another contract in the call chain) subsequently sends ETH to the selfdestructed address.

This pattern arises naturally in DeFi factory contracts, proxy deployment patterns, and flash-loan-style atomic operations. An attacker can craft a transaction to trigger this deliberately, causing ETH to be burned rather than delivered to the intended beneficiary. The entry path is a standard unprivileged EVM transaction — no privileged role is required.

---

### Recommendation

Align with Ethereum EIP-6780 semantics: defer the balance transfer to the beneficiary until `finish_tx`, rather than performing it immediately in `mark_for_deconstruction`. Specifically:

- In `mark_for_deconstruction`, record the beneficiary address alongside the `is_marked_for_deconstruction` flag instead of transferring the balance immediately.
- In `finish_tx`, read the current balance (which may have grown since `SELFDESTRUCT` was called), transfer it to the stored beneficiary, then zero the account.

This ensures that any ETH received after `SELFDESTRUCT` (but within the same transaction) is correctly forwarded to the beneficiary, matching Ethereum behavior.

---

### Proof of Concept

```
Transaction T:
  1. Deployer contract C is called.
  2. C deploys contract A (CREATE), funding it with 1 ETH.
     → A.balance = 1 ETH
  3. C calls A.execute(), which runs SELFDESTRUCT(beneficiary=B).
     → mark_for_deconstruction: transfers 1 ETH to B immediately.
     → A.balance = 0, B.balance += 1 ETH, A.is_marked_for_deconstruction = true
  4. C sends 0.5 ETH to A via CALL (A's code still present, no guard).
     → A.balance = 0.5 ETH
  5. Transaction ends; finish_tx runs.
     → A.balance is zeroed (AccountProperties::TRIVIAL_VALUE).
     → The 0.5 ETH is permanently burned.

Expected (Ethereum EIP-6780): B.balance = 1.5 ETH
Actual (ZKsync OS):            B.balance = 1.0 ETH, 0.5 ETH burned

                        .0

# Not a Bug

This page lists ZKsync OS behavior that is intentional, protocol-correct, or already documented, but is commonly misreported as a vulnerability.

Use this page as a triage aid, not as a substitute for checking the code. If an observed behavior differs from what is described here, investigate it normally.

## EVM / Cancun Behavior

### Cancun is the supported EVM hardfork

ZKsync OS currently targets Cancun EVM semantics. Reports that assume later hardfork behavior are not valid unless they identify an issue under the supported hardfork.

See [EVM Execution Environment](./execution_environments/evm.md).

### EIP-4844 transactions are disabled in production

EIP-4844 type `0x03` transactions are implemented behind the `basic_bootloader/eip-4844` feature, but this feature is not enabled by the production feature sets. It is enabled for test and Ethereum test-runner configurations.

See [Transaction formats](./bootloader/transaction_format.md).

### `BLOBHASH` returns `0`

This is expected in production. Since EIP-4844 transactions are disabled, transactions do not have blob versioned hashes. Cancun/EIP-4844 specifies that `BLOBHASH(index)` returns a zeroed `bytes32` when `index` is outside `tx.blob_versioned_hashes`.

### `BLOBBASEFEE` returns `1`

This is expected in production. There are no EIP-4844 transactions in production history, so `excess_blob_gas` remains `0`. Cancun/EIP-4844 defines the minimum blob base fee as `1`, and EIP-7516 makes `BLOBBASEFEE` return the current block's blob base fee.

### Opcode `0x44` returns `1`

Opcode `0x44` is `PREVRANDAO` under Cancun semantics, historically named `DIFFICULTY`. ZKsync OS uses the block `mix_hash` value for this opcode. In production, `mix_hash` is mocked to `1`.

This does not contradict the block header `difficulty` field being `0` after the Merge.

See [Bootloader block header](./bootloader/bootloader.md#block-header).

### `SUB` operand order

ZKsync OS follows the standard EVM operand order for `SUB`. The top stack item is the first operand, and the item below it is the second operand. For example, `PUSH1 0x03 PUSH1 0x05 SUB` leaves `0x02` on the stack.

This is not a reversed subtraction bug.

    pub fn selfdestruct(
        &mut self,
        system: &mut System<S>,
        tracer: &mut impl Tracer<S>,
    ) -> InstructionResult {
        self.gas
            .spend_gas_and_native(gas_constants::SELFDESTRUCT, SELFDESTRUCT_NATIVE_COST)?;

        if self.is_static_frame() {
            return Err(EvmError::StateChangeDuringStaticCall.into());
        }

        let beneficiary = u256_to_b160(self.stack.pop_1()?);

        let amount_transferred = system
            .io
            .mark_for_deconstruction(
                THIS_EE_TYPE,
                self.gas.resources_mut(),
                &self.address,
                &beneficiary,
            )
            .map_err(wrap_error!())?;

        tracer.evm_tracer().on_selfdestruct(
            beneficiary,
            amount_transferred,
            &InterpreterExternal::new_from(&self, system),
        );

        Err(ExitCode::SelfDestruct)
    }
