### Title
Missing Lower-Bound Validation on `threshold` in FROST Signing Entry Points Allows Protocol-Breaking Values — (`File: src/frost/mod.rs`)

---

### Summary

`assert_sign_inputs` and `presign` in `src/frost/mod.rs` accept a `threshold` parameter but only enforce an upper bound (`threshold <= participants.len()`). They do **not** enforce the minimum `threshold >= 2` that the DKG entry point (`assert_key_invariants` in `src/dkg.rs`) enforces. A malicious coordinator or caller can supply `threshold = 0` or `threshold = 1` to the FROST signing protocol, causing signing to produce cryptographically invalid (unusable) signatures, permanently denying signing to honest parties.

---

### Finding Description

`ReconstructionLowerBound` is a plain `usize` newtype with no built-in invariants: [1](#0-0) 

Any `usize` value — including `0` and `1` — is a valid `ReconstructionLowerBound`.

The DKG entry point correctly enforces a minimum of 2: [2](#0-1) 

The triple-generation entry point also enforces a minimum of 2: [3](#0-2) 

However, the FROST signing validation function `assert_sign_inputs` only checks the upper bound: [4](#0-3) 

There is no corresponding `threshold < 2` rejection. The same gap exists in `presign`: [5](#0-4) 

Both functions accept `threshold = 0` or `threshold = 1` without error.

---

### Impact Explanation

In FROST, the `threshold` passed at signing time determines how many signature shares are aggregated to reconstruct the final signature. The key was generated with `threshold >= 2` (enforced by `assert_key_invariants`). If signing is invoked with `threshold = 1` (or `0`), the aggregation uses fewer shares than the secret-sharing polynomial degree requires. The resulting signature is cryptographically invalid and will not verify against the public key produced during DKG. Honest parties who attempt to use the output receive an unusable cryptographic artifact — a corrupted signing output — matching the **High: Corruption of sign outputs so honest parties accept unusable cryptographic outputs** impact category.

---

### Likelihood Explanation

The coordinator role in FROST is a participant who orchestrates the signing session and supplies the `threshold` argument. A malicious coordinator — an explicitly documented adversary in the threshold-signatures threat model — can trivially pass `threshold = 1` to `assert_sign_inputs` or `presign`. No cryptographic capability is required; it is a single integer substitution at the API call site. The inconsistency between the DKG validation (which blocks `threshold < 2`) and the signing validation (which does not) makes this easy to trigger and hard to detect without auditing both call sites.

---

### Recommendation

Add the same lower-bound guard that `assert_key_invariants` and `validate_triple_inputs` already use:

```rust
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

This check should be added to both `assert_sign_inputs` and `presign` in `src/frost/mod.rs`, immediately alongside the existing upper-bound check, to make the invariant consistent across all protocol entry points.

---

### Proof of Concept

1. Run DKG with `participants = [P1, P2, P3]`, `threshold = 2` — succeeds, key generated with a degree-1 polynomial.
2. Call `assert_sign_inputs(&[P1, P2, P3], /*threshold=*/ 1usize, P1, P1)` — passes all checks (1 ≤ 3, participants valid, coordinator valid).
3. Call `presign` with `args.threshold = ReconstructionLowerBound(1)` — passes the only threshold check (`1 > 3` is false).
4. Proceed to FROST signing aggregation with `threshold = 1`; only one share is combined.
5. The resulting signature does not verify against the public key (which requires degree-1 reconstruction, i.e., 2 shares). All honest verifiers reject it.
6. Signing is permanently broken for this session; the presignature is consumed and cannot be reused. [6](#0-5) [7](#0-6) [8](#0-7)
