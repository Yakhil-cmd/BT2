### Title
Unprivileged Caller Can Permanently Suppress Block Rewards for All Stakers via `update_rewards(disable_rewards: true)` - (File: src/staking/staking.cairo)

---

### Summary

`StakingRewardsManagerImpl::update_rewards` is publicly callable with no access control. It unconditionally advances the global `last_reward_block` checkpoint before checking the `disable_rewards` flag. Any unprivileged caller can invoke it with `disable_rewards: true`, permanently consuming the per-block reward slot for the entire protocol without distributing any yield.

---

### Finding Description

`update_rewards` is exposed as a public ABI function under `IStakingRewardsManager`. Its only caller-side gate is `general_prerequisites()`, which checks only that the contract is unpaused and the caller is non-zero. [1](#0-0) 

The function writes `last_reward_block` to the current block number **before** branching on `disable_rewards`: [2](#0-1) 

`last_reward_block` is a single global storage slot: [3](#0-2) 

The guard at the top of the function enforces that only one call per block can succeed: [4](#0-3) 

Because the checkpoint write happens before the `disable_rewards` branch, a front-running call with `disable_rewards: true` permanently exhausts the reward slot for that block. Any subsequent legitimate call in the same block reverts with `REWARDS_ALREADY_UPDATED`. The rewards that should have been minted and distributed for that block are permanently unclaimable.

---

### Impact Explanation

**High — Permanent freezing of unclaimed yield.**

Block rewards are calculated per-block and gated by `last_reward_block`. Once that slot is consumed with `disable_rewards: true`, the corresponding block's yield is irrecoverably lost for every staker in the protocol. An attacker who front-runs `update_rewards` every block can suppress 100 % of consensus-era block rewards indefinitely, with zero cost beyond gas.

---

### Likelihood Explanation

**High.** The entry path requires no privilege, no stake, and no special state. Any EOA can call `update_rewards(any_valid_staker_address, disable_rewards: true)` at the start of every block. The only precondition is that the contract is unpaused and the supplied `staker_address` passes the active-staker checks (lines 1466–1482), which is trivially satisfied by using any live staker's address obtained from on-chain events. [5](#0-4) 

---

### Recommendation

1. **Restrict the caller.** Add an access-control check so that only the designated consensus/sequencer role (or the staker themselves) may call `update_rewards`. Alternatively, remove the `disable_rewards` parameter from the public interface and handle that logic internally.
2. **Move the checkpoint write after the reward branch.** Only advance `last_reward_block` when rewards are actually distributed, so a no-op call does not consume the slot.

---

### Proof of Concept

```
// Attacker script (pseudocode, executed at the start of every block)
let victim = any_active_staker_address();   // obtained from NewStaker events
staking_contract.update_rewards(victim, disable_rewards: true);
// last_reward_block is now == current_block_number
// No rewards distributed; slot permanently consumed for this block
// Any legitimate update_rewards call in this block reverts with REWARDS_ALREADY_UPDATED
```

Repeat each block → all stakers receive zero consensus block rewards indefinitely.
