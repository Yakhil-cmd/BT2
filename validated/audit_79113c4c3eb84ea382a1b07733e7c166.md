### Title
Unbounded Message Buffer Allows Malicious Participant to Cause Unbounded Memory Growth — (`File: src/protocol/internal.rs`)

### Summary

The `MessageBuffer` inside `Comms` uses `futures::channel::mpsc::unbounded()` channels with no cap on the number of distinct `MessageHeader` keys or messages per key. Any participant in a protocol session can call `Protocol::message()` with crafted messages bearing arbitrary headers, creating new unbounded-channel entries in the `HashMap<MessageHeader, SubMessageQueue>` indefinitely. The developers themselves wrote a test that proves the attack works, naming it `attacker_can_fill_message_buffer_with_unused_waitpoints`.

---

### Finding Description

`SubMessageQueue` is backed by `futures::channel::mpsc::unbounded()`: [1](#0-0) 

`MessageBuffer::push()` inserts a new `SubMessageQueue` for every distinct `MessageHeader` key it has not seen before, with no size limit on the map: [2](#0-1) 

`Comms::push_message()` accepts any message whose first 40 bytes parse as a valid `MessageHeader` and forwards it unconditionally: [3](#0-2) 

The public `Protocol::message()` implementation calls `push_message` with no validation or rate-limiting: [4](#0-3) 

A `MessageHeader` is a 32-byte `ChannelTag` plus an 8-byte `Waitpoint` (u64). An attacker can vary the waitpoint across the full u64 range, producing 2^64 distinct headers. Each unique header creates a new `HashMap` entry holding an `UnboundedSender`/`UnboundedReceiver` pair. Messages queued to headers the honest protocol never polls accumulate forever.

The developers confirmed this attack surface in a test: [5](#0-4) 

The test asserts that 512 injected messages with distinct waitpoints produce 512 distinct map entries — demonstrating unbounded growth with no mitigation in place.

---

### Impact Explanation

A malicious participant in any DKG, reshare, refresh, presign, sign, or CKD session can flood the honest party's `MessageBuffer` with messages addressed to headers the honest code never consumes. Each injected message grows the `HashMap` and the per-header `UnboundedReceiver` queue. With no cap, this causes unbounded heap growth, eventually exhausting the process's memory and crashing or severely degrading the honest node. This matches the allowed **Medium** impact: *"Griefing or resource-exhaustion by a malicious caller or participant causing unbounded CPU, memory, bandwidth, or non-terminating work beyond documented behavior."*

---

### Likelihood Explanation

Every protocol in the library (DKG, ECDSA triple generation, presign, sign, EdDSA sign, CKD) is driven through the same `Protocol::message()` entry point. Any participant who is legitimately part of a session can call it. No privilege beyond being a registered participant is required. The attack requires only sending crafted byte strings with varying 8-byte waitpoint fields — trivially achievable by any malicious peer.

---

### Recommendation

1. **Cap the `HashMap` size**: reject `push_message` calls once the number of distinct headers exceeds a protocol-specific bound (e.g., `n_participants × max_rounds × max_channels`).
2. **Cap per-header queue depth**: replace `futures::channel::mpsc::unbounded()` with `futures::channel::mpsc::channel(BOUND)` and drop messages that exceed the bound.
3. **Validate headers against expected waitpoints**: track which waitpoints are currently live and silently discard messages for unknown or future waitpoints.

---

### Proof of Concept

The existing test in `src/protocol/internal.rs` at lines 532–554 is a self-contained proof of concept. It shows that injecting 512 messages with distinct waitpoints (`1_000_000 + i` for `i` in `0..512`) creates 512 separate `HashMap` entries. Scaling `attack_count` to `u64::MAX` iterations (or simply sending messages in a tight loop during a live DKG session) would exhaust available memory on the honest party's machine. [5](#0-4)

    fn attacker_can_fill_message_buffer_with_unused_waitpoints() {
        let comms = Comms::new();
        let attacker = Participant::from(99_u32);
        let attack_count = 512_u64;

        for i in 0..attack_count {
            let header =
                MessageHeader::new(ChannelTag::root_shared()).with_waitpoint(1_000_000 + i);
            let mut message = header.to_bytes().to_vec();
            message.extend_from_slice(&i.to_le_bytes());

            // Attacker injects messages for waitpoints the honest code never polls.
            comms.push_message(attacker, message);
        }

        let messages = comms
            .incoming
            .messages
            .lock()
            .expect("lock should not fail");

        assert!(messages.len() == usize::try_from(attack_count).unwrap());
    }
