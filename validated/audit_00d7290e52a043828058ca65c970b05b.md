### Title
Simulation Trace Unconditionally Reports Zero `gas_consumed` and Transaction-Level `da_gas_consumed` for Every `FunctionInvocation` Frame — (`crates/apollo_rpc_execution/src/objects.rs`)

---

### Summary

`vm_resources_to_execution_resources` hardcodes `gas_consumed` to `StarknetApiGasVector::default()` (zero) and sets `da_gas_consumed` to the transaction-level `receipt.da_gas` for every `FunctionInvocation` node in the simulation trace — including validate, execute, fee_transfer, and all recursive inner calls. This is unconditional and affects every transaction type.

---

### Finding Description

The conversion function `vm_resources_to_execution_resources` is the sole path that populates `ExecutionResources` for every `FunctionInvocation` in the RPC simulation trace:

```rust
// objects.rs:394-400
Ok(ExecutionResources {
    steps: u64_from_usize(vm_resources.n_steps),
    builtin_instance_counter,
    memory_holes: u64_from_usize(vm_resources.n_memory_holes),
    da_gas_consumed: StarknetApiGasVector { l1_gas, l2_gas, l1_data_gas },
    gas_consumed: StarknetApiGasVector::default(),   // ← always zero, hardcoded
})
``` [1](#0-0) 

The `gas_vector` argument destructured into `da_gas_consumed` is not per-frame gas — it is `receipt.da_gas`, the transaction-level state DA cost, passed identically to every frame:

```rust
// objects.rs:135, 145, 152
(execute_call_info, transaction_execution_info.receipt.da_gas).try_into()?
(call_info,         transaction_execution_info.receipt.da_gas).try_into()?
(call_info,         transaction_execution_info.receipt.da_gas).try_into()?
``` [2](#0-1) 

And inner calls propagate the same value recursively:

```rust
// objects.rs:329-334
calls: call_info
    .inner_calls
    .into_iter()
    .map(|call_info| (call_info, gas_vector))   // same gas_vector for every child
    .map(Self::try_from)
    .collect::<Result<_, _>>()?,
``` [3](#0-2) 

`receipt.da_gas` is the transaction-level state DA gas vector, computed from state changes for data availability — not from per-frame execution gas:

```rust
// blockifier/src/fee/receipt.rs:121-124
let da_gas = tx_resources
    .starknet_resources
    .state
    .da_gas_vector(tx_context.block_context.block_info.use_kzg_da);
``` [4](#0-3) 

The result is that every `FunctionInvocation` node in the simulation trace — regardless of transaction type, version, or actual per-frame gas usage — reports:
- `execution_resources.gas_consumed = {l1_gas: 0, l1_data_gas: 0, l2_gas: 0}` (hardcoded zero)
- `execution_resources.da_gas_consumed = receipt.da_gas` (the same transaction-level DA gas vector, duplicated into every frame including validate, execute, fee_transfer, and all inner calls)

The `v0_8` RPC layer then sums these per-frame values to produce the top-level `execution_resources` for the trace response:

```rust
// apollo_rpc/src/v0_8/execution.rs:177-179
execution_resources: validate_execution_resources.unwrap_or_default()
    + execute_execution_resources
    + fee_transfer_execution_resources.unwrap_or_default(),
``` [5](#0-4) 

This means the top-level `execution_resources.da_gas_consumed` in the trace is `N × receipt.da_gas` where N is the number of non-None invocation frames (2 or 3), further inflating the wrong value.

---

### Impact Explanation

Every call to `starknet_simulateTransactions` or `starknet_traceTransaction` returns authoritative-looking but systematically wrong values for `execution_resources` in every `FunctionInvocation` node:

- `gas_consumed` is always zero regardless of actual L1/L2/L1-data gas consumed per frame.
- `da_gas_consumed` is the transaction-level DA gas duplicated into every frame, not the per-frame DA cost.
- The top-level `execution_resources` in the trace is a multiple of `receipt.da_gas`, not the true total.

Actual fee charging and state changes are unaffected — `FeeEstimation` is computed from `receipt.gas` directly:

```rust
// objects.rs:172-178
let gas_vector = tx_execution_output.execution_info.receipt.gas;
Ok(FeeEstimation {
    gas_consumed: gas_vector.l1_gas.0.into(),
    ...
    data_gas_consumed: gas_vector.l1_data_gas.0.into(),
``` [6](#0-5) 

The economic impact is zero. The impact is confined to simulation/trace RPC responses returning wrong per-frame and aggregate execution resource values.

---

### Likelihood Explanation

This is triggered unconditionally by any `starknet_simulateTransactions` or `starknet_traceTransaction` call for any transaction type (Invoke, Declare, DeployAccount, L1Handler). No special attacker input is required. The question's framing about "attacker-controlled v3 Invoke with non-zero L2 gas" is technically sufficient but not necessary — the behavior is identical for all transactions.

---

### Recommendation

In `vm_resources_to_execution_resources`, replace the hardcoded `gas_consumed: StarknetApiGasVector::default()` with the actual per-frame gas consumed from `call_info.execution.gas_consumed` (available in blockifier's `CallInfo`). The `da_gas_consumed` field should reflect per-frame DA gas, not the transaction-level `receipt.da_gas`. The `gas_vector` parameter should be removed or replaced with per-frame values sourced from `CallInfo` rather than propagated from the transaction receipt.

---

### Proof of Concept

The bug is statically visible without execution. The hardcoded line:

```rust
gas_consumed: StarknetApiGasVector::default(),
``` [7](#0-6) 

...is unconditional. A unit test asserting `gas_consumed == GasVector::default()` for every `FunctionInvocation` in any simulated transaction would pass trivially. Similarly, asserting `da_gas_consumed == receipt.da_gas` for every frame (including inner calls) would pass for any transaction with non-empty call trees.
