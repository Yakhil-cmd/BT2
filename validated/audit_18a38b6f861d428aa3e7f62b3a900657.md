### Title
Incomplete Gas Accounting in `deployAt` Omits Constructor Execution Gas, Undercharging COA Contract Deployments — (File: `fvm/evm/emulator/emulator.go`)

---

### Summary

The `deployAt` function in `fvm/evm/emulator/emulator.go` computes `GasConsumed` using only the code-deposit cost (`len(ret) * CreateDataGas`), completely omitting the gas consumed by the constructor execution itself. This is the same class of bug as the Arbitrum Stylus report: a wrong unit/value is passed to a cost model, causing the reported cost to be systematically lower than the true cost. Any unprivileged user who owns a Cadence-Owned Account (COA) can trigger this path via `COA.deploy()`.

---

### Finding Description

In `fvm/evm/emulator/emulator.go`, the `deployAt` function runs the constructor bytecode through the EVM interpreter and then sets `GasConsumed` to only the EIP-161 code-deposit fee:

```go
ret, err := proc.evm.Run(contract, nil, false)
gasCost := uint64(len(ret)) * gethParams.CreateDataGas
res.GasConsumed = gasCost
``` [1](#0-0) 

The `gethVM.NewContract` call above it allocates the full `call.GasLimit` to the contract object:

```go
contract := gethVM.NewContract(
    callerCommon, addr, castedValue, call.GasLimit, nil,
)
``` [2](#0-1) 

After `proc.evm.Run` returns, the remaining gas is held in `contract.Gas`. The gas actually consumed by the constructor is `call.GasLimit - contract.Gas`, but this quantity is never added to `gasCost`. Only the storage-deposit portion (`len(ret) * CreateDataGas`) is recorded.

By contrast, the standard EVM path (`proc.run` → `gethCore.ApplyMessage`) correctly captures all gas:

```go
res.GasConsumed = execResult.UsedGas
``` [3](#0-2) 

The comment in `deployAt` acknowledges the simplification but does not flag it as intentionally free:

> "in the future we might optimize this method accepting deploy-ready byte codes and skip interpreter call, gas calculations and many checks." [4](#0-3) 

The entry path from Cadence is:

1. `COA.deploy()` in `fvm/evm/stdlib/contract.cdc` calls `InternalEVM.deploy(...)`. [5](#0-4) 

2. `newInternalEVMTypeDeployFunction` in `fvm/evm/impl/impl.go` calls `account.Deploy(code, gasLimit, amount)`. [6](#0-5) 

3. `Account.Deploy` in `fvm/evm/handler/handler.go` issues a `NewDeployCall` direct call. [7](#0-6) 

4. The emulator dispatches to `deployAt` for that call subtype. [8](#0-7) 

---

### Impact Explanation

`GasConsumed` from `deployAt` is the value placed in the EVM `Result` struct and propagated as the gas used for the direct call. Flow's EVM handler converts EVM gas into Cadence computation units (via `ComputationKindEVMGasUsage`). Because constructor execution gas is silently dropped, any COA deployment whose constructor performs non-trivial work (storage writes, loops, sub-calls) is charged far less computation than it actually consumed. This directly lowers the Flow transaction fee paid by the deployer, constituting an undercharge on on-chain asset consumption — the same class of financial impact as the Arbitrum Stylus report.

---

### Likelihood Explanation

The entry path requires only that the caller own a COA, which any Flow account can create permissionlessly via `EVM.createCadenceOwnedAccount()`. No privileged role, staked node, or admin key is needed. The bug fires for every COA deployment whose constructor uses any gas beyond the trivial minimum, which is the common case for real Solidity contracts.

---

### Recommendation

After `proc.evm.Run` returns, add the constructor execution gas to the deposit cost:

```go
constructorGasUsed := call.GasLimit - contract.Gas
gasCost := uint64(len(ret))*gethParams.CreateDataGas + constructorGasUsed
res.GasConsumed = gasCost
```

This mirrors how `gethCore.ApplyMessage` accounts for gas in the standard `run` path. Additionally, add a test that deploys a contract with a gas-consuming constructor via a COA and asserts that `GasConsumed` equals the sum of constructor gas and deposit gas.

---

### Proof of Concept

1. Create a COA via `EVM.createCadenceOwnedAccount()`.
2. Deploy a Solidity contract whose constructor performs 10 000 SSTORE operations (each costs 20 000 gas), so constructor gas ≈ 200 000 000 gas.
3. Observe that `result.GasConsumed` equals only `len(deployedBytecode) * 200` (the deposit cost), not `200_000_000 + deposit_cost`.
4. The Flow transaction fee is computed from the underreported value, so the deployer pays a fraction of the true cost.

func (a *Account) Deploy(code types.Code, gaslimit types.GasLimit, balance types.Balance) *types.ResultSummary {

	res, err := a.executeAndHandleAuthorizedCall(
