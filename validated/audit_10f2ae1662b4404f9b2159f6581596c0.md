### Title
Cross-Address COA Ownership Proof Replay via Missing `evmAddress` Binding in `signedData` - (File: `fvm/evm/stdlib/contract.cdc`)

---

### Summary

`EVM.validateCOAOwnershipProof()` verifies that a Cadence account's keys signed `signedData`, and that the COA resource at the given `CapabilityPath` matches the provided `evmAddress`. However, the function does **not** enforce that `signedData` encodes `evmAddress` or `CapabilityPath`. A signature produced to prove ownership of one COA can be replayed to forge a valid proof for a different COA owned by the same Cadence account.

---

### Finding Description

`EVM.validateCOAOwnershipProof()` performs two checks:

1. Verifies signatures over the caller-supplied `signedData` using the `"FLOW-V0.0-user"` domain tag.
2. Checks that the COA resource at `path` in account `address` has an EVM address matching `evmAddress`. [1](#0-0) 

The function itself documents the gap: [2](#0-1) 

Because `signedData` is an arbitrary byte array with no enforced binding to `evmAddress` or `CapabilityPath`, a signature `sig = Sign(D, "FLOW-V0.0-user", aliceKey)` is equally valid for any COA path in alice's account. An attacker who obtains a valid proof for COA_A (at `/public/coaA`) can construct a structurally different proof pointing to COA_B (at `/public/coaB`) using the same `sig` and `D`, and `validateCOAOwnershipProof` will return `isValid: true` for COA_B.

The `COAOwnershipProof` struct encodes `CapabilityPath` but the signatures are **not** over the path — they are only over `signedData`: [3](#0-2) 

The `proofVerifier` precompile in `arch.go` calls the validator with the decoded proof as-is, with no additional binding check: [4](#0-3) 

The COA Solidity contract's ERC-1271 `isValidSignature` calls `verifyCOAOwnershipProof(address(this), _hash, _sig)`, where `address(this)` is the calling COA's EVM address: [5](#0-4) 

---

### Impact Explanation

If a Cadence account holds two COAs at different public paths (e.g., `/public/coaA` → `0xAAA` and `/public/coaB` → `0xBBB`), an attacker who obtains a valid proof for COA_A can forge a proof for COA_B:

1. Victim signs `D` to prove ownership of COA_A; proof = `{CapabilityPath: "coaA", Signatures: [sig], ...}`.
2. Attacker constructs a new proof: `{CapabilityPath: "coaB", Signatures: [sig], ...}` (same `sig`, different path).
3. Attacker calls `COA_B.isValidSignature(D, attacker_proof)`.
4. The precompile invokes `validateCOAOwnershipProof(alice, /public/coaB, D, [0], [sig], 0xBBB)`.
5. Signature check passes — `sig` is a valid signature over `D` with `"FLOW-V0.0-user"` using alice's key.
6. COA address check passes — `/public/coaB` resolves to COA_B whose address is `0xBBB`.
7. `isValidSignature` returns `ValidERC1271Signature` (0x1626ba7e) for a hash the victim never authorized for COA_B.

This allows unauthorized ERC-1271 signature forgery for COA_B, enabling unauthorized asset transfers or privileged contract interactions that rely on ERC-1271 from COA_B.

The same attack applies to any Cadence transaction or script that calls `EVM.validateCOAOwnershipProof` directly, since the function is `access(all)`. [6](#0-5) 

---

### Likelihood Explanation

The precondition is that the victim Cadence account holds at least two COAs at different public paths. The code comment acknowledges this is currently uncommon ("Flow wallets historically create at most one COA per account"), but:

- The protocol places no restriction on creating multiple COAs per account.
- As COA usage grows (e.g., DAO multi-sig setups, protocol-owned accounts), multi-COA accounts become more likely.
- The attacker only needs to observe a valid proof for one COA (e.g., from a public on-chain transaction or off-chain authentication flow) and can replay it immediately.

The entry path is fully unprivileged: any EVM transaction can call `COA.isValidSignature`, and any Cadence script can call `EVM.validateCOAOwnershipProof` directly.

---

### Recommendation

Enforce that `signedData` commits to both `evmAddress` and `CapabilityPath` inside `validateCOAOwnershipProof`. The function should verify that the decoded `signedData` encodes the `evmAddress` argument (and ideally the `path`), rejecting proofs where `signedData` does not bind to the specific COA being proven. Alternatively, the function should internally construct the signed message as `hash(evmAddress || capabilityPath || callerData)` rather than accepting an arbitrary `signedData` blob.

---

### Proof of Concept

```
// Setup: alice has two COAs
// /public/coaA → EVM address 0xAAA
// /public/coaB → EVM address 0xBBB

// Step 1: alice legitimately signs D for coaA
sig = Sign(D, "FLOW-V0.0-user", aliceKey)
proof_A = {KeyIndices:[0], Address:alice, CapabilityPath:"coaA", Signatures:[sig]}

// Step 2: attacker constructs a forged proof for coaB using the same sig
proof_B = {KeyIndices:[0], Address:alice, CapabilityPath:"coaB", Signatures:[sig]}
encodedProof_B = RLP.encode(proof_B)

// Step 3: attacker calls COA_B.isValidSignature(D, encodedProof_B)
// Precompile invokes: validateCOAOwnershipProof(alice, /public/coaB, D, [0], [sig], 0xBBB)
// → signature check: sig valid over D with "FLOW-V0.0-user" → PASS
// → COA address check: /public/coaB → 0xBBB == 0xBBB → PASS
// → returns isValid: true  ← FORGED
``` [7](#0-6) [8](#0-7)
