### Title
Pool Admin Bypasses `maxAdminSpreadFeeE6` Cap via Uncapped `setPoolBinAdditionalFees` — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary
`MetricOmmPoolFactory.setPoolBinAdditionalFees` forwards `addFeeBuyE6`/`addFeeSellE6` to the pool with no validation against `maxAdminSpreadFeeE6`, while the parallel path `setPoolAdminFees` enforces that cap. A pool admin can set per-bin fees up to `uint16` max (65535 in E6 = 6.5535%) regardless of the factory-enforced ceiling, causing traders to pay excess fees on every swap through the affected bin.

### Finding Description
`setPoolAdminFees` correctly enforces the factory cap:

```solidity
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
``` [1](#0-0) 

`setPoolBinAdditionalFees` performs no such check — it passes the caller-supplied values straight through:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
``` [2](#0-1) 

`setBinAdditionalFees` on the pool only validates the bin index, then writes the values unconditionally:

```solidity
if (bin < LOWEST_BIN || bin > HIGHEST_BIN) revert InvalidBinIndex(bin);
BinState storage s = _binStates[bin];
s.addFeeBuyE6 = addFeeBuyE6;
s.addFeeSellE6 = addFeeSellE6;
``` [3](#0-2) 

During every swap the bin additional fee is added directly to the effective fee applied to the trade:

```solidity
params.baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6),
``` [4](#0-3) 

The `BinState` struct stores `addFeeBuyE6` and `addFeeSellE6` as unconstrained `uint16` fields: [5](#0-4) 

### Impact Explanation
Traders swapping through any bin where the pool admin has set `addFeeBuyE6`/`addFeeSellE6` above the factory cap pay a higher effective spread than the protocol intends to permit. At `uint16` max (65535), the additional fee alone is 6.5535%, far exceeding the hard ceiling of 20% (`HARD_MAX_SPREAD_FEE_E6 = 200_000`) that the factory owner is themselves bound by, and completely unconstrained relative to `maxAdminSpreadFeeE6` (which can be 0). The extra fee accrues into the LP surplus pool, from which the admin collects a proportional share via `collectFees`, so the admin indirectly extracts value beyond the cap. Traders receive less output than the protocol's fee model guarantees. [6](#0-5) 

### Likelihood Explanation
Any pool admin — a semi-trusted role explicitly constrained by `maxAdminSpreadFeeE6` — can call this at any time after pool creation. No special precondition is required beyond holding the `poolAdmin[pool]` role. The bypass is a single transaction.

### Recommendation
Add the same cap check to `setPoolBinAdditionalFees` that exists in `setPoolAdminFees`. The simplest fix is to enforce that each per-bin fee value does not exceed `maxAdminSpreadFeeE6` before forwarding to the pool:

```solidity
if (addFeeBuyE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
```
