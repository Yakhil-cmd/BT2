Looking at the codebase, I need to find an analog to the Chainlink "stale data used without freshness validation" pattern. The core invariant is: **data used for admission/fee decisions must reflect the value at execution time, not a stale snapshot**.

Let me trace the gateway stateful validation path carefully.

---

### Title
Gateway Stateful Validator Uses Stale Previous-Block L2 Gas Price for Resource Bounds Admission — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

`StatefulTransactionValidator::validate_resource_bounds` reads the L2 gas price from the **already-committed previous block** via `get_block_info()` and uses it as the reference price to admit or reject incoming transactions. Because the next block's L2 gas price is computed by the EIP-1559 fee market (`calculate_next_base_gas_price`) and can differ from the previous block's price by up to `price / gas_price_max_change_denominator`, the gateway systematically validates against a stale price — the direct sequencer analog of calling Chainlink's `latestAnswer` instead of `latestRoundData`.

The embedded TODO explicitly acknowledges that the correct reference is the **next** block's L2 gas price, not the previous block's. The next block's price is produced by `calculate_next_base_gas_price` in the fee market:

```rust
let price_change = (price_u256 * gas_delta) / denominator;
let adjusted_price_u256 =
    if gas_used > gas_target { price_u256 + price_change } else { price_u256 - price_change };
``` [2](#0-1) 

The denominator is `gas_target * gas_price_max_change_denominator`, so the price can shift by up to `1/gas_price_max_change_denominator` per block — a normal, per-block operational change.

Meanwhile, the blockifier's `check_fee_bounds` (called during actual execution) compares the transaction's `max_price_per_unit` against the **current block's** gas price from `block_info.gas_prices`:

```rust
if resource_bounds.max_price_per_unit < actual_gas_price.get() {
    insufficiencies_resource.push(
        ResourceBoundsError::MaxGasPriceTooLow { ... }
    );
}
``` [3](#0-2) 

The gateway and blockifier therefore use **different reference prices** for the same invariant.

### Impact Explanation

**Accepts invalid transactions (High):** When the L2 gas price rises between blocks (high utilization), a transaction with `max_price_per_unit` that satisfies the gateway's threshold against `previous_block_l2_gas_price` may fall below the blockifier's `actual_gas_price` for the next block. The gateway admits the transaction to the mempool; the blockifier rejects it at `perform_pre_validation_stage`. The transaction occupies a mempool slot and wastes a sequencer execution slot. [4](#0-3) 

**Rejects valid transactions (High):** When the L2 gas price falls (low utilization), a transaction whose `max_price_per_unit` is below the previous (higher) block's threshold is rejected by the gateway even though it would satisfy the blockifier's check against the lower current price. Valid transactions are incorrectly denied admission.

### Likelihood Explanation

The L2 gas price changes every block as a function of gas utilization. This is not an edge case — it is the normal operation of the EIP-1559 fee market. The discrepancy is active whenever `config.validate_resource_bounds` is `true` (the production mode; the comment "Skip this validation during the systems bootstrap phase" confirms it is enabled post-bootstrap). Any user submitting a transaction near the price boundary triggers the mismatch without any special privilege.

### Recommendation

Replace the `get_block_info()` call with a source that returns the **next** block's projected L2 gas price, consistent with what `calculate_next_base_gas_price` will produce for the block being built. The TODO comment already identifies this as the correct fix. Until then, the gateway and blockifier operate on divergent price references for the same fee invariant.

### Proof of Concept

1. Configure the sequencer with `validate_resource_bounds: true` (production mode).
2. Observe the previous block's `l2_gas_price = P` (e.g., `1_000_000_000`).
3. The next block's price rises to `P' = P + P/denominator` due to a full block.
4. Submit a transaction with `max_price_per_unit = P * threshold` (passes gateway check against `P`).
5. The gateway calls `validate_tx_l2_gas_price_within_threshold` → **passes** (price ≥ `P * threshold`).
6. The transaction enters the mempool and reaches the blockifier.
7. The blockifier calls `check_fee_bounds` with `actual_gas_price = P'` → **fails** (`max_price_per_unit < P'`).
8. The transaction is rejected at execution time despite having been admitted by the gateway. [5](#0-4) [6](#0-5)
