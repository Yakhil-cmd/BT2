### Title
`protocolUnpausePool()` Sets Wrong Pause Level, Leaving Pool Permanently Admin-Paused After Protocol Unpause — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`protocolUnpausePool()` calls `setPause(1)` instead of `setPause(0)`, so after the protocol owner unpauses a pool that was originally active, the pool is left in the admin-paused state. Swaps remain blocked indefinitely until the pool admin separately calls `unpausePool()`.

---

### Finding Description

The protocol defines three pause levels:

| Level | Meaning |
|---|---|
| `0` | Active — swaps allowed |
| `1` | Admin-paused |
| `2` | Protocol-paused |

`_checkNotPaused()` blocks swaps for **any non-zero** pause level:

```solidity
function _checkNotPaused() internal view {
    if (pauseLevel != 0) revert PoolPaused();
}
``` [1](#0-0) 

`protocolPausePool()` correctly accepts pools at level `0` (active) **or** level `1` (admin-paused) and moves them to level `2`:

```solidity
function protocolPausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 0 && cur != 1) revert InvalidPauseTransition(cur, 2);
    IMetricOmmPoolFactoryActions(pool).setPause(2);
}
``` [2](#0-1) 

`protocolUnpausePool()` then unconditionally calls `setPause(1)` — the **admin-paused** level — instead of `setPause(0)`:

```solidity
function protocolUnpausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 2) revert InvalidPauseTransition(cur, 1);
    IMetricOmmPoolFactoryActions(pool).setPause(1);   // ← wrong: should be 0
}
``` [3](#0-2) 

Because `protocolPausePool` accepts pools at level `0`, the common path is:

```
Active (0) → protocolPausePool → Protocol-paused (2)
           → protocolUnpausePool → Admin-paused (1)   ← pool never returns to active
```

After `protocolUnpausePool`, `pauseLevel == 1 != 0`, so `_checkNotPaused()` still reverts on every swap call. The pool is stuck in admin-paused state even though the admin never paused it.

---

### Impact Explanation

Swaps are the core revenue-generating operation of the pool. After a protocol unpause of an originally-active pool, all swap calls revert with `PoolPaused`. LPs cannot earn fees, traders cannot execute, and the pool is functionally dead until the pool admin separately calls `unpausePool()`. The pool admin may not be monitoring for this condition, and there is no on-chain signal distinguishing "admin intentionally paused" from "protocol unpause left residual pause level." This is broken core pool functionality per the allowed impact gate.

---

### Likelihood Explanation

The trigger is a routine, valid protocol-owner action: pause a live pool for an emergency, then unpause it. No malicious setup is required. The protocol owner is semi-trusted and this is an expected operational flow. Every such cycle on an originally-active pool reproduces the bug deterministically.

---

### Recommendation

Change `protocolUnpausePool` to restore the pool to fully active (`0`), not admin-paused (`1`):

```solidity
function protocolUnpausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 2) revert InvalidPauseTransition(cur, 0);
-   IMetricOmmPoolFactoryActions(pool).setPause(1);
+   IMetricOmmPoolFactoryActions(pool).setPause(0);
}
```

If the design intent is to preserve a pre-existing admin pause, the factory must record the pause level before the protocol pause and restore it on unpause.

---

### Proof of Concept

1. Pool is deployed and active (`pauseLevel == 0`).
2. Protocol owner calls `protocolPausePool(pool)` → `pauseLevel` becomes `2`.
3. Protocol owner calls `protocolUnpausePool(pool)` → `setPause(1)` is called → `pauseLevel` becomes `1`.
4. Any user calls `swap(...)` → `_checkNotPaused()` fires → `PoolPaused` revert.
5. Pool admin must call `unpausePool(pool)` to restore `pauseLevel` to `0`, even though the admin never paused the pool.
