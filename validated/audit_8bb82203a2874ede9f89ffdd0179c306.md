### Title
Missing Authorization Check for Contract Updates Allows Bypass of Deployment Restriction - (File: `fvm/environment/contract_updater.go`)

---

### Summary

The `SetContract` function in `fvm/environment/contract_updater.go` explicitly skips the global authorized-deployment-list check when a contract already exists. When `RestrictedDeploymentEnabled = true`, any account that holds an existing contract can update it to arbitrary code without being in the authorized list, directly analogous to the ERC1967Factory pattern where the admin check was absent for the upgrade path.

---

### Finding Description

`SetContract` is the single FVM function that handles both initial contract deployment and contract updates. Its authorization logic is:

```go
// Initial contract deployments must be authorized by signing accounts.
//
// Contract updates are always allowed.
exists, err := updater.accounts.ContractExists(location.Name, flow.Address(location.Address))
...
if !exists && !updater.isAuthorizedForDeployment(signingAccounts) {
    return fmt.Errorf("deploying contract failed: ...")
}
// No check when exists == true
updater.draftUpdates[location] = ContractUpdate{...}
```

The comment is explicit: **"Contract updates are always allowed."** The `isAuthorizedForDeployment` guard fires only when `!exists`. When the contract already exists, the function unconditionally queues the update with zero authorization verification against the global allowlist.

`isAuthorizedForDeployment` reads the service-account-stored allowlist (`ContractDeploymentAuthorizedAddressesPath`) and returns `true` only if a signing account appears in it — or returns `true` unconditionally when `RestrictedDeploymentEnabled() == false`. For removals, `RemoveContract` symmetrically calls `isAuthorizedForRemoval`. The update path has no equivalent guard.

`UpdateAccountContractCode` (the Cadence runtime entry point) simply delegates to `SetContract` with `updater.signingAccounts` and performs no additional allowlist check:

```go
func (updater *ContractUpdaterImpl) UpdateAccountContractCode(
    location common.AddressLocation,
    code []byte,
) error {
    ...
    err = updater.SetContract(location, code, updater.signingAccounts)
    ...
}
```

---

### Impact Explanation

When the network operator enables `RestrictedDeploymentEnabled = true` (the default, `DefaultContractUpdaterParams()` sets `RestrictContractDeployment: true`) to lock down who may deploy contracts — for example during a security incident or a controlled upgrade window — the restriction is enforced only for *new* deployments. Any account that already holds a contract (deployed before the restriction was activated, or deployed by the service account on behalf of a user) can submit a transaction calling `account.contracts.update()` and replace the contract bytecode with arbitrary Cadence code. The FVM will accept the update because `exists == true` causes the allowlist check to be skipped entirely.

Concrete consequences:
- A previously-authorized account that was subsequently removed from the allowlist retains full ability to mutate its contracts.
- Any account that deployed contracts during an unrestricted period can continue to update them after the restriction is enabled, defeating the governance intent.
- Malicious contract code can be injected into accounts whose contracts are imported by other contracts or dApps, enabling downstream exploitation (e.g., stealing resources, emitting false events, corrupting shared state).

---

### Likelihood Explanation

The `RestrictedDeploymentEnabled` flag is read from on-chain state (`IsContractDeploymentRestrictedPath` in the service account) and falls back to the context-level `RestrictContractDeployment` boolean, which defaults to `true`. On any network where this restriction is active and accounts with pre-existing contracts exist (which is true of mainnet), the condition for exploitation is already met. An attacker needs only to own an account with a deployed contract and submit a standard signed transaction — no special privileges, no staked node role, no key compromise beyond their own account key.

---

### Recommendation

Apply the same `isAuthorizedForDeployment` check to the update path inside `SetContract`, or introduce a dedicated `isAuthorizedForUpdate` check that consults the same (or a separate) allowlist. The asymmetry between deployment (checked), removal (checked), and update (unchecked) is the root cause.

---

### Proof of Concept

**Setup**: Network has `RestrictedDeploymentEnabled = true`. Account `0xALICE` deployed `contract Foo {}` before the restriction was enabled. `0xALICE` is **not** in `ContractDeploymentAuthorizedAddressesPath`.

**Attack transaction** (signed only by `0xALICE`):
```cadence
transaction {
    prepare(signer: auth(UpdateContract) &Account) {
        // Replaces Foo with malicious code — no allowlist check fires
        signer.contracts.update(
            name: "Foo",
            code: "<malicious Cadence bytecode>".decodeHex()
        )
    }
}
```

**FVM execution path**:
1. Cadence runtime calls `UpdateAccountContractCode(location{Foo, 0xALICE}, maliciousCode)`.
2. `UpdateAccountContractCode` calls `SetContract(location, maliciousCode, [0xALICE])`.
3. `ContractExists("Foo", 0xALICE)` returns `true` → `exists = true`.
4. The guard `if !exists && !updater.isAuthorizedForDeployment(...)` evaluates to `false` (short-circuits on `!exists`).
5. `draftUpdates[location] = ContractUpdate{..., Code: maliciousCode}` is queued.
6. `Commit()` writes the malicious code to the ledger with no further authorization check.

The malicious contract is now live on `0xALICE`'s account, bypassing the deployment restriction entirely. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

```go
) error {

		trace.FVMEnvUpdateAccountContractCode).End()

	err := updater.meter.MeterComputation(
		common.ComputationUsage{
			Kind:      ComputationKindUpdateAccountContractCode,
			Intensity: 1,
		},
	)
	if err != nil {
		return fmt.Errorf("update account contract code failed: %w", err)
	}

	err = updater.SetContract(
		location,
		code,
		updater.signingAccounts)
	if err != nil {
		return fmt.Errorf("updating account contract code failed: %w", err)
	}

	return nil
}
```

```go
func (updater *ContractUpdaterImpl) SetContract(
	location common.AddressLocation,
	code []byte,
	signingAccounts []flow.Address,
) error {
	// Initial contract deployments must be authorized by signing accounts.
	//
	// Contract updates are always allowed.
	exists, err := updater.accounts.ContractExists(location.Name, flow.Address(location.Address))
	if err != nil {
		return err
	}

	if !exists && !updater.isAuthorizedForDeployment(signingAccounts) {
		return fmt.Errorf(
			"deploying contract failed: %w",
			errors.NewOperationAuthorizationErrorf(
				"SetContract",
				"deploying contracts requires authorization from specific "+
					"accounts"))

	}

	updater.draftUpdates[location] = ContractUpdate{
		Location: location,
		Code:     code,
	}

	return nil
}
```

```go
func (updater *ContractUpdaterImpl) isAuthorizedForDeployment(
	signingAccounts []flow.Address,
) bool {
	if updater.RestrictedDeploymentEnabled() {
		return updater.isAuthorized(
			signingAccounts,
			blueprints.ContractDeploymentAuthorizedAddressesPath)
	}
	return true
}
```
