### Title
`estimate_fee` and `simulate_transactions` hardcode `only_query = false`, producing a wrong transaction hash for query-version transactions — (`crates/apollo_rpc/src/v0_8/api/mod.rs`)

### Summary

The `TryFrom<BroadcastedTransaction> for ExecutableTransactionInput` conversion, which feeds both `starknet_estimateFee` and `starknet_simulateTransactions`, unconditionally sets `only_query = false` for every `Invoke` and `DeployAccount` transaction. The Starknet spec requires that simulation/estimation calls use the query-version bit (high bit of the version field), which changes the transaction hash used inside `__validate__`. Because the bit is silently dropped, the hash seen by the account contract during simulation differs from the hash the user signed, making the simulation return an authoritative-looking wrong result.

### Finding Description

`BroadcastedTransaction::try_into()` is the sole conversion path used by both `estimate_fee` and `simulate_transactions`:

```rust
// crates/apollo_rpc/src/v0_8/api/mod.rs  lines 329-339
impl TryFrom<BroadcastedTransaction> for ExecutableTransactionInput {
    type Error = ErrorObjectOwned;
    fn try_from(value: BroadcastedTransaction) -> Result<Self, Self::Error> {
        // TODO(yair): pass the right value for only_query field.
        match value {
            BroadcastedTransaction::Declare(tx) => Ok(tx.try_into()?),
            BroadcastedTransaction::DeployAccount(tx) => Ok(Self::DeployAccount(tx.into(), false)),
            BroadcastedTransaction::Invoke(tx) => Ok(Self::Invoke(tx.into(), false)),
        }
    }
}
```

The `false` literal is the `OnlyQuery` parameter. It is forwarded unchanged through `to_blockifier_tx` into `ExecutionFlags::only_query`, which is then read by `TransactionInfo::signed_version()`:

```rust
// crates/blockifier/src/transaction/objects.rs  lines 101-103
pub fn signed_version(&self) -> TransactionVersion {
    signed_tx_version(&self.version(), &TransactionOptions { only_query: self.only_query() })
}
```

`signed_version` is the version field hashed into the transaction hash. When `only_query = true` the high bit is set (e.g. `0x100000000000000000000000000000001` for V1), producing a different hash. Because the conversion always passes `false`, the hash computed during simulation is the *regular* hash, not the query hash.

A caller who follows the spec and signs the query hash (as wallets and SDKs are expected to do for simulation calls) will have their signature validated against the wrong hash. With `validate = true` (the default for `estimate_fee`) the `__validate__` entry point will reject the signature and the RPC call returns a validation-failure error instead of a fee estimate. With `validate = false` (`SkipValidate` flag) the simulation runs to completion but the account contract's `__execute__` receives a `tx_info.version` without the query bit, so any account that branches on the query bit (e.g. to skip replay-protection checks during simulation) will execute the wrong code path, and the returned `fee_estimation` reflects that wrong execution.

The same conversion is used by `simulate_transactions`:

```rust
// crates/apollo_rpc/src/v0_8/api/api_impl.rs  lines 1074-1075
let executable_txns =
    transactions.into_iter().map(|tx| tx.try_into()).collect::<Result<_, _>>()?;
```

### Impact Explanation

- **`estimate_fee` with `validate = true`**: any caller that correctly signs the query-version hash receives a `VALIDATION_FAILURE` error instead of a fee estimate. The RPC endpoint returns an authoritative-looking error for a transaction that would succeed on-chain.
- **`simulate_transactions` with `SkipValidate`**: the account contract's `__execute__` sees the non-query version. Accounts that gate behaviour on the query bit execute the wrong branch; the returned `fee_estimation` and `transaction_trace` reflect that wrong execution, misleading callers about actual on-chain cost and state changes.

Both cases match: *"High. RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value."*

### Likelihood Explanation

Any unprivileged caller of `starknet_estimateFee` or `starknet_simulateTransactions` who passes a transaction with the query bit set in the version field (as the Starknet spec permits and as conforming SDKs produce) will trigger the bug. No special privilege or unusual state is required.

### Recommendation

Replace the hardcoded `false` literals with the actual query bit extracted from the incoming transaction version:

```rust
BroadcastedTransaction::DeployAccount(tx) => {
    let only_query = tx.is_query();   // extract high bit from version
    Ok(Self::DeployAccount(tx.into(), only_query))
},
BroadcastedTransaction::Invoke(tx) => {
    let only_query = tx.is_query();
    Ok(Self::Invoke(tx.into(), only_query))
},
```

The same fix is needed in `TryFrom<BroadcastedDeclareTransaction>` where the same `false` literal appears with the same TODO comment.

### Proof of Concept

1. Construct an `InvokeTransaction::Version1` with `version = 0x100000000000000000000000000000001` (V1 + query bit).
2. Sign the transaction hash computed with that query version.
3. Call `starknet_estimateFee` (without `SKIP_VALIDATE`) with this transaction.
4. The conversion sets `only_query = false`; `signed_version()` returns `0x1`; the hash is recomputed without the query bit; `__validate__` receives a hash that does not match the signature → the RPC returns `VALIDATION_FAILURE` instead of a fee estimate.
5. Repeat with `starknet_simulateTransactions` + `SKIP_VALIDATE`: the call succeeds but `tx_info.version` inside `__execute__` is `0x1` instead of `0x100000000000000000000000000000001`; any account that checks the query bit takes the wrong branch; the returned `fee_estimation` is wrong.
