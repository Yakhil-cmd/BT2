### Title
Stale Pending Block Synthesized with Hardcoded `Calldata` DA Mode Corrupts `use_kzg_da` in `estimate_fee` and `simulate_transactions` — (`crates/apollo_rpc/src/v0_8/api/api_impl.rs`)

### Summary

When the in-memory pending data is stale (its `parent_block_hash` does not match the latest committed block hash), `read_pending_data` synthesizes a fallback `DeprecatedPendingBlock`. Because `DeprecatedPendingBlock` has no `l1_da_mode` field, its `l1_da_mode()` accessor unconditionally returns `L1DataAvailabilityMode::Calldata`. The actual `l1_da_mode` stored in the latest committed block header is available but is never read. This corrupts the `use_kzg_da` flag in the `BlockContext` used by `estimate_fee` and `simulate_transactions` when called with `block_id = Tag(Pending)` on a BLOB DA network.

### Impact Explanation

On a BLOB DA network (`l1_da_mode = Blob`), the `BlockContext` built during the stale window has `use_kzg_da = false`. This changes how data gas is priced: blob DA and calldata DA have different cost models. Any `estimate_fee` or `simulate_transactions` call with `block_id = Tag(Pending)` during the stale window returns an authoritative-looking but incorrect fee estimate for transactions with nonzero L1 data gas. The error is not a minor rounding difference — it is a categorical DA-mode mismatch.

### Likelihood Explanation

The stale window is a normal operational state. It occurs every time a new block is committed to storage before the pending data `RwLock` is updated by the sync loop. On a live network this window recurs at every block. Any unprivileged user who calls `estimate_fee` or `simulate_transactions` with `block_id = Tag(Pending)` during this window is affected. No special privileges are required.

### Recommendation

In the stale branch of `read_pending_data`, read `l1_da_mode` from `latest_header.block_header_without_hash.l1_da_mode` and use `PendingBlockOrDeprecated::Current(PendingBlock { l1_da_mode: latest_header.block_header_without_hash.l1_da_mode, l1_data_gas_price: latest_header.block_header_without_hash.l1_data_gas_price, ... })` instead of `DeprecatedPendingBlock`. This ensures the synthesized fallback block carries the correct DA mode (and also the correct `l1_data_gas_price`, which is similarly zeroed out for `Deprecated` blocks). [7](#0-6) 

### Proof of Concept

1. Commit a block header to storage with `l1_da_mode = Blob`.
2. Set `pending_data.block.parent_block_hash` to any hash that does not match the committed block's hash (simulating the stale window).
3. Call `estimate_fee` with `block_id = Tag(Pending)` for a transaction that touches storage (nonzero L1 data gas).
4. Observe that `read_pending_data` enters the `else` branch, synthesizes a `DeprecatedPendingBlock`, and `l1_da_mode()` returns `Calldata`.
5. Assert that `use_kzg_da` in the resulting `BlockContext` is `false` — contradicting the committed block's `Blob` DA mode — and that the returned data gas fee differs from the correct BLOB-priced estimate.
