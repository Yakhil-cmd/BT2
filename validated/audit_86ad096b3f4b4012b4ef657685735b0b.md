### Title
Unbounded Loop Over `pool_member_epoch_balance` Trace in `calculate_rewards` Freezes Delegator's Unclaimed Yield - (File: src/pool/pool.cairo)

### Summary
The `calculate_rewards` function in `Pool` iterates over the entire `pool_member_epoch_balance` trace for a pool member since their last claim checkpoint. Because a delegator can grow this trace without bound by making balance changes across many epochs, the loop can eventually exceed the Starknet block gas limit, permanently preventing the delegator from claiming their accumulated rewards. The code itself acknowledges this with the comment: *"This loop is unbounded but unlikely to exceed gas limits."*

### Finding Description

**Root cause — unbounded loop over a user-grown trace:**

In `src/pool/pool.cairo`, `calculate_rewards` (lines 837–888) iterates over every entry in `pool_member_epoch_balance` for a given pool member between their last-claimed checkpoint and the current epoch:

```cairo
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
    if pool_member_checkpoint.epoch() >= until_epoch {
        break;
    }
    ...
    entry_to_claim_from += 1;
}
``` [1](#0-0) 

The variable `entry_to_claim_from` is initialized from the pool member's stored checkpoint index (the index at the time of their last successful `claim_rewards` call). Every balance-changing operation a delegator performs in a new epoch — `add_to_delegation_pool`, `exit_delegation_pool_intent`, or `switch_delegation_pool` — appends a new entry to this trace. If the delegator never claims (or claims infrequently) while repeatedly changing their balance across epochs, the number of entries between their last checkpoint and the current epoch grows without bound.

**How the trace grows:**

Each call to `add_to_delegation_pool` or `exit_delegation_pool_intent` in a new epoch inserts a new `PoolMemberCheckpoint` into `pool_member_epoch_balance`. There is no cap on the number of entries per pool member. [2](#0-1) 

**Where `calculate_rewards` is invoked:**

`calculate_rewards` is called during the reward-claiming flow. When the trace is large enough, the transaction reverts due to gas exhaustion, and the delegator's accumulated rewards become permanently inaccessible.

**Attacker-controlled entry path:**

1. An unprivileged delegator calls `add_to_delegation_pool` (or `exit_delegation_pool_intent`) once per epoch over N epochs without ever calling `claim_rewards`.
2. Each call in a new epoch appends one entry to `pool_member_epoch_balance`.
3. After N epochs, the delegator's trace has N entries since their last checkpoint.
4. When `claim_rewards` is finally called, `calculate_rewards` must iterate all N entries.
5. For sufficiently large N, the transaction exceeds the block gas limit and reverts.
6. The delegator can never claim their rewards; the yield is permanently frozen.

### Impact Explanation

**Impact: High — Permanent freezing of unclaimed yield.**

A delegator who has accumulated rewards over many epochs with frequent balance changes will find their `claim_rewards` transaction permanently reverting. The rewards are locked in the contract with no recovery path, since every attempt to claim triggers the same unbounded loop. This matches the allowed impact: *"Permanent freezing of unclaimed yield or unclaimed royalties."* [3](#0-2) 

### Likelihood Explanation

**Likelihood: Medium.**

- No privileged access is required. Any delegator can trigger this by making balance changes across many epochs.
- The protocol is designed for long-term participation; a delegator active for hundreds of epochs who adjusts their delegation periodically (e.g., topping up or partially withdrawing) will naturally accumulate many trace entries.
- The code itself acknowledges the risk ("unbounded but unlikely to exceed gas limits"), indicating the developers are aware but have not mitigated it.
- Starknet's current block gas limit bounds the maximum safe trace length; a delegator with a few hundred balance-change epochs without claiming could realistically hit this limit.

### Recommendation

1. **Checkpoint on every claim**: Store the `entry_to_claim_from` index after each successful `claim_rewards` call so subsequent claims only iterate over new entries. (This appears to be the intent, but verify the checkpoint is always persisted correctly.)
2. **Cap trace entries per claim call**: Introduce a `max_entries_per_claim` parameter. If the trace has more entries than the cap, process only up to the cap and update the stored checkpoint, requiring the delegator to call `claim_rewards` multiple times (analogous to the post-audit fix in the reference report).
3. **Enforce a minimum epoch gap between balance changes**: Prevent a delegator from adding a new trace entry more than once per epoch (already partially enforced by epoch-keyed traces, but verify no bypass exists).

### Proof of Concept

```
1. Deploy the staking system with a pool.
2. Delegator D calls `add_to_delegation_pool(amount: MIN_DELEGATION)` in epoch 1.
   → pool_member_epoch_balance[D] has 1 entry.
3. For each epoch i from 2 to N (e.g., N = 500):
   a. Advance to epoch i.
   b. D calls `add_to_delegation_pool(amount: 1)` or `exit_delegation_pool_intent(amount: 1)`.
   → pool_member_epoch_balance[D] now has i entries.
   (D never calls claim_rewards.)
4. After epoch N, D calls `claim_rewards`.
   → calculate_rewards iterates from entry_to_claim_from=0 to pool_member_trace_length=N.
   → For large N, the transaction exceeds the block gas limit and reverts.
5. D retries claim_rewards — same result. Rewards are permanently frozen.
``` [4](#0-3)

```text
        fn calculate_rewards(
            self: @ContractState,
            pool_member: ContractAddress,
            from_checkpoint: PoolMemberCheckpoint,
            until_checkpoint: PoolMemberCheckpoint,
            mut entry_to_claim_from: VecIndex,
        ) -> (Amount, VecIndex) {
            let pool_member_trace = self.pool_member_epoch_balance.entry(pool_member);
            // Note: `until_epoch` is the current epoch.
            let until_epoch = until_checkpoint.epoch();

            let mut rewards = 0;

            let pool_member_trace_length = pool_member_trace.length();

            let mut from_sigma = self.find_sigma(from_checkpoint, curr_epoch: until_epoch);
            let mut from_balance = from_checkpoint.balance();

            let base_value = self.staking_rewards_base_value.read();

            // **Note**: The loop iterates over the balance changes in the pool member's balance
            // trace. This loop is unbounded but unlikely to exceed gas limits.
            while entry_to_claim_from < pool_member_trace_length {
                let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
                // If the balance change is after `until_epoch` (and therefore does not affect
                // the current reward computation), exit the loop.
                if pool_member_checkpoint.epoch() >= until_epoch {
                    break;
                }

                // Compute rewards from (inclusive) the previous balance change (or from
                // `from_checkpoint`) to (exclusive) the current entry.
                let to_sigma = self.find_sigma(pool_member_checkpoint, curr_epoch: until_epoch);
                rewards +=
                    compute_rewards_rounded_down(
                        amount: from_balance, interest: to_sigma - from_sigma, :base_value,
                    );
                from_sigma = to_sigma;
                from_balance = pool_member_checkpoint.balance();
                entry_to_claim_from += 1;
            }

            // Compute the remaining rewards from (inclusive) the last visited balance change in
            // `pool_member_trace` (or from `from_checkpoint`) to (exclusive) `until_checkpoint`.
            let to_sigma = self.find_sigma(until_checkpoint, curr_epoch: until_epoch);
            rewards +=
                compute_rewards_rounded_down(
                    amount: from_balance, interest: to_sigma - from_sigma, :base_value,
                );

            (rewards, entry_to_claim_from)
        }
