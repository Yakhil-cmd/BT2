### Title
Unauthenticated `LedgerService.Set` RPC Allows Any Network-Reachable Caller to Corrupt Execution State - (File: `cmd/ledger/main.go`, `ledger/remote/service.go`)

### Summary
The standalone `flow-ledger-service` binary (`cmd/ledger/`) exposes a gRPC `LedgerService` that includes a `Set` RPC method for writing arbitrary key-value pairs into the execution-node ledger (MTrie). When the service is started with a TCP listener (`-ledger-service-tcp 0.0.0.0:9000`), the gRPC server is created with **no authentication, no TLS, and no authorization interceptor**. Any network-reachable caller can invoke `LedgerService.Set` and write arbitrary register values into the ledger, producing a new state commitment that the execution node will treat as authoritative.

### Finding Description
In `cmd/ledger/main.go`, the gRPC server is constructed with only message-size options and no security credentials:

```go
grpcServer := grpc.NewServer(
    grpc.MaxRecvMsgSize(int(*maxRequestSize)),
    grpc.MaxSendMsgSize(int(*maxResponseSize)),
)
ledgerService := remote.NewService(ledgerStorage, logger)
ledgerpb.RegisterLedgerServiceServer(grpcServer, ledgerService)
```

The TCP listener is bound to whatever address the operator provides (e.g., `0.0.0.0:9000`):

```go
lis, err := net.Listen("tcp", *ledgerServiceTCP)
```

The `Service.Set` handler in `ledger/remote/service.go` performs only structural validation (non-nil state hash, non-empty keys, matching key/value lengths) and then directly calls `s.ledger.Set(update)`, which mutates the underlying MTrie and persists the change to the WAL. There is no caller identity check, no token, no mTLS, and no gRPC interceptor that would restrict who may call `Set`.

The analog to the reported Solidity bug is exact: a privileged state-mutation function (`sfnProcessTradeDispute` / `LedgerService.Set`) is callable by any external party because the access-control guard (Solidity modifier / gRPC auth interceptor + TLS) is absent.

### Impact Explanation
An attacker who can reach the TCP port can:

1. Call `LedgerService.Set` with an arbitrary `(state, keys[], values[])` tuple to write any register value at any key under any existing state root.
2. The execution node's `CommitDelta` path calls `ldg.Set(update)` on the same ledger object; if the ledger is the remote `Client`, the forged state root is returned and stored as the new execution state commitment.
3. This allows direct corruption of account balances, contract bytecode, or any other on-chain register, producing a fraudulent state commitment that the execution node signs and submits as an execution receipt.
4. Downstream, Verification Nodes re-execute chunks against the real transactions and will produce a different result, triggering a slashing/dispute path — but the corrupted state has already been committed and broadcast.

The impact is **unauthorized mutation of execution state / integrity failure in the state transition model**, directly analogous to the price-override impact in the original report.

### Likelihood Explanation
The TCP bind address is operator-controlled. The README explicitly documents and encourages TCP deployment (`-ledger-service-tcp 0.0.0.0:9000`). In any deployment where the ledger service port is reachable from outside the execution node's localhost (e.g., containerized environments, cloud VMs with open security groups, or multi-host deployments), any network peer can call `Set` without credentials. No brute force or key compromise is required — a single valid gRPC call suffices.

### Recommendation
1. **Require mutual TLS** on the gRPC server when TCP is used. Add a `grpc.Creds(credentials.NewTLS(cfg))` server option and refuse to start with a TCP listener unless TLS credentials are provided.
2. **Add a gRPC unary interceptor** that validates the caller identity (e.g., via peer certificate CN or a shared secret) before dispatching any mutating RPC (`Set`).
3. **Separate read and write interfaces** at the proto level so that read-only callers (e.g., scripts) cannot accidentally be granted write access.
4. At minimum, **document that TCP mode must never be exposed to untrusted networks** and enforce `localhost`-only binding by default.

// Set updates keys with new values at a specific state and returns the new state
func (s *Service) Set(ctx context.Context, req *ledgerpb.SetRequest) (*ledgerpb.SetResponse, error) {
	if req.State == nil || len(req.State.Hash) != len(ledger.State{}) {
		return nil, status.Error(codes.InvalidArgument, "invalid state")
	}

	if len(req.Keys) == 0 {
		return nil, status.Error(codes.InvalidArgument, "keys cannot be empty")
	}

	if len(req.Keys) != len(req.Values) {
		return nil, status.Error(codes.InvalidArgument, "keys and values length mismatch")
	}

	var state ledger.State
	copy(state[:], req.State.Hash)

	keys := make([]ledger.Key, len(req.Keys))
	for i, protoKey := range req.Keys {
		key, err := protoKeyToLedgerKey(protoKey)
		if err != nil {
			return nil, err // protoKeyToLedgerKey already returns status.Error
		}
		keys[i] = key
	}

	values := make([]ledger.Value, len(req.Values))
	for i, protoValue := range req.Values {
		var value ledger.Value
		// Reconstruct the original value type using is_nil flag
		// This preserves the distinction between nil and []byte{} that protobuf loses
		if len(protoValue.Data) == 0 {
			if protoValue.IsNil {
				// Original value was nil
				value = nil
			} else {
				// Original value was []byte{} (empty slice)
				value = ledger.Value([]byte{})
			}
		} else {
			// Non-empty value, use data as-is
			value = ledger.Value(protoValue.Data)
		}
		values[i] = value
	}

	update, err := ledger.NewUpdate(state, keys, values)
	if err != nil {
		return nil, status.Error(codes.InvalidArgument, err.Error())
	}

	newState, trieUpdate, err := s.ledger.Set(update)
