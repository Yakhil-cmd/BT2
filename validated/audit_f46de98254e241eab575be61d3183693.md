### Title
Off-by-One in `SliceVec::resize` Causes Spurious `OutOfGas` on EVM Heap Expansion to Exact Backing-Buffer Capacity - (`File: zk_ee/src/memory/slice_vec.rs`)

---

### Summary

`SliceVec::resize` uses a strict `>=` guard instead of `>` when checking whether the requested new length exceeds the backing buffer. This means a resize to exactly the backing-buffer capacity always returns `Err(())`. In the EVM interpreter, that error is unconditionally mapped to `OutOfGas`, so any EVM memory-expansion opcode that would grow the heap to exactly the last available byte of the backing slice silently aborts the transaction with an out-of-gas error even though sufficient gas was provided and the access is within the declared memory limit.

---

### Finding Description

**Root cause — `zk_ee/src/memory/slice_vec.rs`, line 78:**

```rust
pub fn resize(&mut self, new_length: usize, padding: T) -> Result<(), ()> {
    if new_length >= self.memory.len() {   // ← should be `>`
        return Err(());
    }
    …
}
```

`self.memory` is the entire backing `&mut [MaybeUninit<T>]` slice (capacity). A resize to `new_length == self.memory.len()` is perfectly valid — it fills the slice to capacity — but the `>=` guard rejects it and returns `Err(())`.

**Propagation path — `evm_interpreter/src/utils.rs`, lines 72–93:**

```rust
pub(crate) fn resize_heap_implementation<'a>(
    heap: &mut SliceVec<'a, u8>,
    gas: &mut Gas<S>,
    offset: usize,
    len: usize,
) -> Result<(), ExitCode> {
    let max_offset = offset.saturating_add(len);
    let new_heap_size = …max_offset.next_multiple_of(32)…;
    let current_heap_size = heap.len();
    if new_heap_size > current_heap_size {
        gas.pay_for_memory_growth(current_heap_size, new_heap_size)?;  // gas charged here

        heap.resize(new_heap_size, 0)
            .map_err(|_| ExitCode::EvmError(EvmError::OutOfGas))?;    // ← always Err when new_heap_size == memory.len()
    }
    Ok(())
}
```

When `new_heap_size == heap.memory.len()`:
1. Gas is successfully charged via `pay_for_memory_growth`.
2. `heap.resize(new_heap_size, 0)` returns `Err(())` due to the off-by-one.
3. The error is mapped to `EvmError::OutOfGas` and the transaction is aborted.

The heap backing buffer is allocated in the bootloader:

```rust
pub const MAX_HEAP_BUFFER_SIZE: usize = 1 << 27; // 128 MB
let mut heaps = Box::new_uninit_slice_in(MAX_HEAP_BUFFER_SIZE, system.get_allocator());
```

The most-recent EVM frame's `SliceVec` is given a slice of this buffer whose length equals the remaining free space. In deeply nested call chains, that remaining space can be far smaller than 128 MB, making the exact-capacity boundary reachable at modest memory sizes.

---

### Impact Explanation

Any EVM opcode that triggers `resize_heap_implementation` — `MLOAD`, `MSTORE`, `MSTORE8`, `CALLDATACOPY`, `CODECOPY`, `RETURNDATACOPY`, `EXTCODECOPY`, `MCOPY`, `SHA3`, `LOG*`, `CALL`/`CREATE` return-data handling — will fail with a spurious `OutOfGas` when the rounded-up new heap size equals the backing-buffer capacity. This is an **EVM semantic mismatch**: Ethereum mainnet would succeed (the access is within the memory limit and gas was available), but ZKsync OS aborts the transaction. The state transition produced by the sequencer and prover diverges from the canonical EVM result, which can cause incorrect block outputs and break EVM equivalence guarantees.

---

### Likelihood Explanation

The trigger condition is `new_heap_size == heap.memory.len()`. In a deeply nested call chain the remaining heap buffer shrinks with each frame; a crafted sequence of nested calls can arrange for the innermost frame's backing slice to be exactly `N` bytes, then issue a memory access that rounds up to exactly `N`. This is attacker-controllable via ordinary EVM bytecode with no privileged access required. The attacker only needs to know (or brute-force) the remaining heap budget, which is deterministic given the call depth and prior heap usage.

---

### Recommendation

Change the guard in `SliceVec::resize` from `>=` to `>`:

```rust
// zk_ee/src/memory/slice_vec.rs, line 78
- if new_length >= self.memory.len() {
+ if new_length > self.memory.len() {
```

This allows the slice to be resized to exactly its backing capacity, consistent with the behaviour of `try_extend` (which correctly uses `if idx == cap { return Err(()) }`, permitting writes up to and including the last slot).

---

### Proof of Concept

1. Deploy a contract whose constructor or fallback executes a sequence of `CALL`s to exhaust the heap buffer to a known remaining size `R` (a multiple of 32).
2. In the innermost call frame, execute `MSTORE` at offset `R - 32` (length 32), so `new_heap_size = R`.
3. Because `R == heap.memory.len()`, `SliceVec::resize` returns `Err(())`.
4. `resize_heap_implementation` maps this to `OutOfGas` and aborts the innermost call.
5. On Ethereum mainnet the same transaction succeeds (gas was sufficient, memory limit not exceeded).
6. The ZKsync OS state transition records a failed inner call where mainnet records success, producing a divergent state root. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
