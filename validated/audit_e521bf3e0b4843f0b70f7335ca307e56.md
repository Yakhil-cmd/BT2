### Title
Missing Minimum Threshold Validation in FROST Signing Allows `threshold=1` to Corrupt Signing Outputs - (File: `src/frost/mod.rs`)

---

### Summary

`assert_key_invariants` enforces a minimum threshold of 2 for DKG, but the analogous validation function `assert_sign_inputs` (and the `presign` entry point) in `src/frost/mod.rs` omit this same lower-bound check. A malicious coordinator or caller can supply `threshold=1` to the FROST signing protocol without triggering any error, causing honest participants to produce partial signatures that combine into an invalid, unusable signature — which they accept as a legitimate protocol output.

---

### Finding Description

`assert_key_invariants` in `src/dkg.rs` enforces two threshold bounds:

```rust
// src/dkg.rs lines 573-582
if threshold > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
if threshold < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
}
``` [1](#0-0) 

The FROST signing entry-point `assert_sign_inputs` in `src/frost/mod.rs` only enforces the upper bound:

```rust
// src/frost/mod.rs lines 144-150
if threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
// ← no lower-bound check; threshold=1 passes silently
``` [2](#0-1) 

The `presign` entry-point in the same file has the identical gap:

```rust
// src/frost/mod.rs lines 71-77
if args.threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
// ← no lower-bound check
``` [3](#0-2) 

This is structurally identical to the reported BPT bug: one validation function (`assert_key_invariants` / `ConvexStakingMixin._isInvalidRewardToken`) has the critical guard, while the parallel function (`assert_sign_inputs` / `AuraStakingMixin._isInvalidRewardToken`) is missing it.

---

### Impact Explanation

When `threshold=1` is accepted by `assert_sign_inputs`, the FROST signing protocol computes Lagrange coefficients for a single-participant reconstruction. Because the key was generated under a minimum threshold of 2 (enforced by DKG), the Lagrange interpolation at signing time is inconsistent with the key's polynomial degree. The combined signature produced by honest participants will fail external verification — an unusable cryptographic output that honest parties nonetheless accept as a completed protocol run.

This maps to the allowed impact: **High — Corruption of sign outputs so honest parties accept unusable cryptographic outputs.**

---

### Likelihood Explanation

The `assert_sign_inputs` function is the public-facing validation gate for FROST signing. Any caller — including a malicious coordinator who controls the `threshold` argument passed to the signing entry point — can supply `threshold=1`. No privilege beyond being a protocol initiator is required. The `ReconstructionLowerBound` type does not self-enforce a minimum of 2; that enforcement is left entirely to the caller-facing validation functions. [4](#0-3) 

---

### Recommendation

Add the same lower-bound guard that `assert_key_invariants` already enforces to both `assert_sign_inputs` and the `presign` entry-point in `src/frost/mod.rs`:

```rust
// In assert_sign_inputs and presign:
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
``` [5](#0-4) 

---

### Proof of Concept

1. Run DKG with `participants = [P1, P2, P3]`, `threshold = 2` — succeeds normally.
2. Call `assert_sign_inputs(&[P1, P2, P3], /*threshold=*/1u8, P1, P1)` — returns `Ok(participants)` with no error, because the `threshold < 2` guard is absent.
3. Proceed with FROST signing using `threshold=1`. Each participant computes a partial signature with a Lagrange coefficient derived from a single-point reconstruction, inconsistent with the degree-1 polynomial used during DKG.
4. The aggregated signature fails verification against the public key — honest parties have accepted and completed a signing round that produced an unusable output. [6](#0-5) [7](#0-6)

pub fn assert_sign_inputs(
    participants: &[Participant],
    threshold: impl Into<ReconstructionLowerBound>,
    me: Participant,
    coordinator: Participant,
) -> Result<ParticipantList, InitializationError> {
    let threshold = threshold.into();
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // validate threshold
    if threshold.value() > participants.len() {
        return Err(InitializationError::ThresholdTooLarge {
            threshold: threshold.value(),
            max: participants.len(),
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }
    Ok(participants)
}
