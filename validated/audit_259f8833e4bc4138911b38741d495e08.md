### Title
Unchecked Precompile Return Value After `_burn` in `withdrawToNear`/`withdrawToEthereum` Causes Permanent Token Loss - (File: `etc/eth-contracts/contracts/EvmErc20.sol`, `etc/eth-contracts/contracts/EvmErc20V2.sol`)

---

### Summary

Both `EvmErc20.sol` and `EvmErc20V2.sol` call `_burn()` on the user's ERC-20 balance **before** invoking the exit precompile via inline assembly, and **never check the return value** of that precompile call. If the precompile call fails for any reason, the tokens are permanently destroyed with no corresponding NEAR-side transfer — an exact structural analog of the reported "transfer before check" vulnerability class.

---

### Finding Description

In `EvmErc20.sol`, both `withdrawToNear` and `withdrawToEthereum` follow this pattern:

```solidity
// EvmErc20.sol lines 53–63
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // ← tokens destroyed HERE

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        // res is NEVER checked
    }
}
``` [1](#0-0) 

```solidity
// EvmErc20.sol lines 65–76

    _burn(_msgSender(), amount);          // ← tokens destroyed HERE
    ...
    assembly {
        let res := call(gas(), 0xb0bd02f6a392af548bdf1cfaee5dfa0eefcc8eab, 0, add(input, 32), input_size, 0, 32)

``` [2](#0-1) 

The identical pattern exists in `EvmErc20V2.sol`: [3](#0-2) [4](#0-3) 

The `ExitToNear` precompile (`engine-precompiles/src/native.rs`) can return an `ExitError` (causing the EVM `call` opcode to return `0`) in multiple reachable conditions:

- Out of gas (`required_gas > target_gas`)
- Called in static context (`ERR_INVALID_IN_STATIC`)
- Called via delegate call (`ERR_INVALID_IN_DELEGATE`)
- Invalid or malformed input (parsing failures)
- Invalid NEAR recipient account ID (fails `AccountId` validation) [5](#0-4) 

When the EVM `call` to the precompile returns `0`, the calling contract's execution continues normally — the `_burn` is **not** reverted, because it occurred in the calling contract's own execution frame. The precompile's side effect (scheduling the NEAR `ft_transfer` promise) never happens. The user's ERC-20 tokens are gone with no corresponding NEAR-side credit.

---

### Impact Explanation

**Critical — Permanent freezing/destruction of user funds.**

A user who calls `withdrawToNear` or `withdrawToEthereum` with any input that causes the precompile to fail will have their ERC-20 tokens permanently burned. There is no recovery path: the tokens are destroyed on the EVM side, and no NEAR-side transfer is ever scheduled. The total supply of the bridged token decreases without any corresponding release of the underlying NEP-141 asset, causing permanent insolvency between the ERC-20 mirror and the NEP-141 backing.

---

### Likelihood Explanation

**Medium-High.** The `recipient` parameter in `withdrawToNear` is a raw `bytes memory` value with no Solidity-level validation. Any caller can supply a byte sequence that is not a valid NEAR account ID. The `ExitToNear` precompile will reject it, return failure, but the `_burn` has already executed. This is directly user-triggerable with no special privileges. Additionally, out-of-gas conditions on the precompile call (e.g., if the caller forwards insufficient gas) produce the same outcome.

---

### Recommendation

Move the `_burn` call to **after** a successful precompile invocation, or add an explicit revert on precompile failure:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    uint res;
    assembly {
        res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    }
    require(res != 0, "ERR_EXIT_TO_NEAR_FAILED");

    _burn(_msgSender(), amount);   // only burn after confirmed success
}
```

Apply the same fix to `withdrawToEthereum` in both `EvmErc20.sol` and `EvmErc20V2.sol`.

---

### Proof of Concept

1. Deploy `EvmErc20` mapped to a NEP-141 token. Mint 1000 tokens to `alice`.
2. `alice` calls `withdrawToNear(bytes("!!invalid-near-account!!"), 1000)`.
3. `_burn(alice, 1000)` executes — alice's balance drops to 0, total supply decreases by 1000.
4. The assembly `call` to the ExitToNear precompile at `0xe9217bc70b7ed1f598ddd3199e80b093fa71124f` fails because `!!invalid-near-account!!` is not a valid NEAR `AccountId`. The precompile returns `ExitError`, so `res = 0`.
5. `res` is never checked. The function returns normally.
6. No NEAR promise is scheduled. The underlying NEP-141 tokens remain locked in the Aurora engine contract forever.
7. Alice has lost 1000 tokens permanently. The ERC-20 mirror is now insolvent relative to its NEP-141 backing.
