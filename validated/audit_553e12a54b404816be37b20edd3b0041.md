### Title
`eligible_operational_addresses` Not Cleared After `change_operational_address` Allows Staker to Permanently Capture Operational Address Without Consent - (File: src/staking/staking.cairo)

### Summary
`change_operational_address` consumes the `eligible_operational_addresses` declaration but never clears it. After a staker switches away from operational address X, the stale eligibility entry persists, allowing the staker to re-claim X at any time without X's re-consent. A malicious staker can exploit this to permanently block X from being used by any other staker, disrupting attestation for other participants.

### Finding Description
The two-step operational-address change flow works as follows:

1. An operational address calls `declare_operational_address(staker_A)`, writing `eligible_operational_addresses[X] = staker_A`.
2. Staker_A calls `change_operational_address(X)`, which verifies the eligibility entry and sets `operational_address_to_staker_address[X] = staker_A`.

The critical omission is in `change_operational_address`: after consuming the eligibility declaration, it **never clears** `eligible_operational_addresses[X]`. [1](#0-0) 

After step 2, `eligible_operational_addresses[X]` still equals `staker_A`. If staker_A later switches to a different operational address Y: [2](#0-1) 

`operational_address_to_staker_address[X]` is zeroed (X is freed), but `eligible_operational_addresses[X]` remains `staker_A`. Staker_A can immediately call `change_operational_address(X)` again — the eligibility check at line 684 passes because the stale entry was never cleared: [3](#0-2) 

Meanwhile, X cannot re-declare for a different staker while it is in use (`OPERATIONAL_IN_USE` guard in `declare_operational_address`): [4](#0-3) 

This creates a permanent capture loop: staker_A frees X, X attempts to re-declare for staker_B, staker_A front-runs and re-claims X, X is back in use and cannot re-declare. The `eligible_operational_addresses` storage map is defined at: [5](#0-4) 

### Impact Explanation
A malicious staker can permanently prevent a specific operational address from serving any other staker. Because the operational address is the only address authorized to submit `attest()` calls on behalf of a staker, blocking it from being assigned to staker_B means staker_B cannot earn attestation rewards. This constitutes **griefing with no profit motive but concrete damage to users and the protocol** (staker_B's attestation rewards are frozen for as long as the attack continues, and the operational address operator is denied the ability to serve their intended client).

### Likelihood Explanation
Any active staker who has previously used an operational address retains the stale eligibility entry indefinitely. No special privilege is required — only a prior `declare_operational_address` + `change_operational_address` sequence, which is a normal part of the staker lifecycle. The attack requires monitoring when X is freed and submitting `change_operational_address(X)` before X's re-declaration, which is straightforward on Starknet given deterministic transaction ordering.

### Recommendation
Clear the `eligible_operational_addresses` entry for the newly adopted operational address inside `change_operational_address`, immediately after the eligibility check passes:

```cairo
// After the eligibility assert, before updating staker_info:
self.eligible_operational_addresses.write(operational_address, Zero::zero());
```

This mirrors the correct pattern used for `operational_address_to_staker_address`, which is explicitly zeroed for the old address on every change. [6](#0-5) 

### Proof of Concept

```
1. X calls declare_operational_address(staker_A)
   → eligible_operational_addresses[X] = staker_A

2. staker_A calls change_operational_address(X)
   → operational_address_to_staker_address[X] = staker_A
   → eligible_operational_addresses[X] = staker_A  ← NOT cleared

3. staker_A calls change_operational_address(Y)  (Y previously declared for staker_A)
   → operational_address_to_staker_address[X] = 0  (X freed)
   → eligible_operational_addresses[X] = staker_A  ← still stale

4. X calls declare_operational_address(staker_B)  ← staker_A front-runs this

5. staker_A calls change_operational_address(X)
   → check: operational_address_to_staker_address[X].is_zero() → PASS (X is free)
   → check: eligible_operational_addresses[X] == staker_A → PASS (stale entry)
   → operational_address_to_staker_address[X] = staker_A  ← X captured again

6. X cannot call declare_operational_address(staker_B) because OPERATIONAL_IN_USE fires.

Repeat steps 3–6 indefinitely to permanently block X from serving staker_B.
```
