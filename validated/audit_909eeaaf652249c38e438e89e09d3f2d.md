### Title
Unauthorized Burn of Victim's Bank Balance via Missing Caller Check in `BankContract.Run` — (`x/cronos/keeper/precompiles/bank.go`)

### Summary

`BankContract.Run` handles both `mint` and `burn` in a shared branch. For `burn`, it derives the **source address** from `args[0]` (caller-supplied), but never verifies that this address equals `contract.Caller()`. Any EVM contract can therefore burn tokens from an arbitrary victim's account.

### Finding Description

In the shared `MintMethodName, BurnMethodName` branch:

```
recipient := args[0].(common.Address)          // attacker-controlled
addr      := sdk.AccAddress(recipient.Bytes()) // victim's SDK address
denom     := EVMDenom(contract.Caller())       // evm/<attackerContract>
``` [1](#0-0) 

For `burn`, `addr` is then passed directly as the **sender** to `SendCoinsFromAccountToModule`:

```go
bc.bankKeeper.SendCoinsFromAccountToModule(ctx, addr, types.ModuleName, sdk.NewCoins(amt))
``` [2](#0-1) 

There is no guard of the form `if recipient != contract.Caller() { return error }`. The only checks present are:

- `amount.Sign() <= 0` — rejects zero/negative amounts [3](#0-2) 
- `checkBlockedAddr` — only rejects module accounts, not arbitrary victim addresses [4](#0-3) 

Compare this with the `exec` helper in `utils.go`, which **does** enforce caller authentication for other precompile paths:

```go
if caller != e.caller {
    return nil, fmt.Errorf("caller is not authenticated: ...")
}
``` [5](#0-4) 

The `BankContract.Run` burn path has no equivalent check.

**The same flaw exists in `TransferMethodName`**: `sender` (args[0]) is used as the `from` address in `SendCoins` without verifying it equals `contract.Caller()`, enabling unauthorized token theft. [6](#0-5) 

### Impact Explanation

An attacker deploys a contract (`attackerContract`) and calls `bank.burn(victim, amount)`. The precompile:

1. Sets `denom = evm/<attackerContract>`
2. Sets source = `victim`
3. Calls `SendCoinsFromAccountToModule(victim, evm_module, evm/<attackerContract> coins)`

Any victim holding a non-zero balance of `evm/<attackerContract>` tokens has those tokens permanently destroyed without consent. This is **unauthorized burn of native bank balances**, a Critical-tier accounting invariant violation.

### Likelihood Explanation

The victim must hold `evm/<attackerContract>` tokens. This is realistic: the attacker can first call `mint(victim, amount)` (which has the same missing-caller-check for the recipient side, but minting to others is not harmful by itself), establishing a balance, then immediately call `burn(victim, amount)` to destroy it. Alternatively, any legitimate token contract using the bank precompile creates victims organically.

### Recommendation

Add a caller check at the start of the `BurnMethodName` branch:

```go
if recipient != contract.Caller() {
    return nil, errors.New("burn: recipient must be the caller")
}
```

Similarly, add the equivalent check in `TransferMethodName` for `sender`.

### Proof of Concept

1. Deploy `AttackerContract` at address `0xATK`.
2. Call `bank.mint(victim, 1000)` from `AttackerContract` — victim now holds 1000 `evm/0xATK`.
3. Call `bank.burn(victim, 1000)` from `AttackerContract`.
4. `BankContract.Run` executes `SendCoinsFromAccountToModule(victim, evm_module, 1000 evm/0xATK)` then `BurnCoins`.
5. Victim's balance is destroyed; no signature or consent from victim was required.

Unit test: call `BankContract.Run` with `contract.Caller() = 0xATK`, `args[0] = victim`, assert the call **succeeds** and victim's balance is zeroed — demonstrating the missing guard.
