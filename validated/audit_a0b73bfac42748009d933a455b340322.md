### Title
Admin Fee Destination Change Without Prior Fee Collection Redirects Accumulated Fees to New Recipient — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`setPoolAdminFeeDestination` updates `poolAdminFeeDestination[pool]` without first calling `collectFees`, causing all fees accumulated under the old destination (both notional and spread surplus) to be paid to the new destination when collection next occurs.

---

### Finding Description

Every other fee-configuration mutator in the factory flushes accumulated fees at the current rates before making any change:

- `setPoolAdminFees` calls `collectFees` before updating `poolFeeConfig` [1](#0-0) 
- `setPoolProtocolFee` calls `collectFees` before updating `poolFeeConfig` [2](#0-1) 

`setPoolAdminFeeDestination` does neither — it simply overwrites the destination and returns:

```solidity
function setPoolAdminFeeDestination(address pool, address newAdminFeeDestination)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (newAdminFeeDestination == address(0)) revert InvalidAdminFeeDestination();
    poolAdminFeeDestination[pool] = newAdminFeeDestination;   // ← no collectFees call
    emit PoolAdminFeeDestinationUpdated(pool, newAdminFeeDestination);
}
``` [3](#0-2) 

The pool holds two categories of accumulated fees at any moment:

1. **Notional fees** — tracked explicitly in `notionalFeeToken0Scaled` / `notionalFeeToken1Scaled`, reset to zero only inside `collectFees`. [4](#0-3) 
2. **Spread surplus** — computed on-the-fly as `balance * scaleMultiplier − binTotals.scaledToken − notionalFeeScaled`; it is never stored separately and is therefore always attributed to whoever `poolAdminFeeDestination[pool]` points to at collection time. [5](#0-4) 

When `collectFees` is eventually called (by anyone via `collectPoolFees`, or implicitly by the next fee-rate change), the admin share of both fee categories is sent to `poolAdminFeeDestination[pool]` — which now points to the new address. [6](#0-5) 

---

### Impact Explanation

The old admin fee destination loses all fees it earned before the destination was changed. Those fees are instead paid to the new destination. This is a direct, quantifiable loss of owed protocol/admin fee revenue. The magnitude equals the full admin share of every swap's spread and notional fees that accrued since the last `collectFees` call.

---

### Likelihood Explanation

Any pool admin can call `setPoolAdminFeeDestination` at any time. The pool admin and the admin fee destination are separate addresses (set independently at pool creation), so the old destination is a distinct party that has no ability to protect itself. The trigger requires only a single valid transaction from the pool admin — no special market conditions are needed.

---

### Recommendation

Mirror the pattern used by `setPoolAdminFees` and `setPoolProtocolFee`: call `collectFees` with the current `poolFeeConfig` and the **current** `poolAdminFeeDestination` before overwriting the destination.

```solidity
function setPoolAdminFeeDestination(address pool, address newAdminFeeDestination)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (newAdminFeeDestination == address(0)) revert InvalidAdminFeeDestination();
    // Flush fees owed to the old destination first
    PoolFeeConfig memory c = poolFeeConfig[pool];
    IMetricOmmPoolCollectFees(pool).collectFees(
        c.protocolSpreadFeeE6, c.adminSpreadFeeE6,
        c.protocolNotionalFeeE8, c.adminNotionalFeeE8,
        poolAdminFeeDestination[pool]   // old destination
    );
    poolAdminFeeDestination[pool] = newAdminFeeDestination;
    emit PoolAdminFeeDestinationUpdated(pool, newAdminFeeDestination);
}
```

---

### Proof of Concept

1. Pool is deployed; `adminFeeDestination = Alice`.
2. Swaps occur; `notionalFeeToken0Scaled` accumulates 1 000 units; spread surplus grows to 500 units.
3. Pool admin calls `setPoolAdminFeeDestination(pool, Bob)`. No `collectFees` is triggered.
4. Anyone calls `collectPoolFees(pool)`. `collectFees` runs with `adminFeeDestination_ = Bob`.
5. Bob receives the admin share of both the 1 000-unit notional fee and the 500-unit spread surplus.
6. Alice receives nothing, despite having earned those fees during steps 1–2.
