### Title
Unauthorized Native Token Transfer via Bank Precompile `transfer` — Missing Caller-Sender Authorization Check - (File: `x/cronos/keeper/precompiles/bank.go`)

### Summary

The bank precompile's `transfer` method accepts an arbitrary `sender` address from calldata and executes `bankKeeper.SendCoins(ctx, from, to, ...)` without verifying that `contract.Caller()` equals `sender`. Any EVM contract can therefore drain the native-bank balance (denom `"evm/<contractAddress>"`) of any victim address to an attacker-controlled recipient, with no victim consent required.

### Finding Description

In `x/cronos/keeper/precompiles/bank.go`, the `TransferMethodName` case unpacks `sender` directly from ABI-encoded calldata:

```go
sender := args[0].(common.Address)   // attacker-supplied
recipient := args[1].(common.Address)
// ...
from := sdk.AccAddress(sender.Bytes())
to   := sdk.AccAddress(recipient.Bytes())
denom := EVMDenom(contract.Caller())  // "evm/<callerAddress>"
// ...
bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt))
``` [1](#0-0) 

The denom is scoped to `contract.Caller()`, so the precompile correctly restricts which token namespace is affected. However, there is **no check** that `contract.Caller() == sender`. The precompile blindly transfers tokens from the caller-supplied `from` address to the caller-supplied `to` address.

Compare this with the `exec` helper used by the relayer/ICA precompile, which explicitly enforces that the message signer matches the EVM caller:

```go
caller := common.BytesToAddress(signers[0])
if caller != e.caller {
    return nil, fmt.Errorf("caller is not authenticated: ...")
}
``` [2](#0-1) 

No equivalent guard exists in the bank precompile's `transfer` path.

The same structural flaw exists in the `burn` case: `args[0]` is an arbitrary address from which tokens are burned, with no check that the caller owns or has been approved to burn from that address. [3](#0-2) 

### Impact Explanation

**Critical — Unauthorized transfer of CRC20/CRC21-backed native assets.**

A user who converts their CRC20 tokens to native bank tokens (denom `"evm/<contractAddress>"`) via `bank.mint` holds a Cosmos-side balance. A malicious contract at that same address can call `bank.transfer(victim, attacker, fullBalance)` at any time, moving the victim's entire native balance to the attacker with no victim signature or approval. This constitutes an unauthorized balance change for precompile-controlled assets, matching the Critical impact tier.

### Likelihood Explanation

Any unprivileged actor can deploy an EVM contract and call the bank precompile. The only precondition is that a victim holds a native balance of denom `"evm/<maliciousContractAddress>"`, which occurs whenever a user calls the contract's `moveToNative`-style function. The attack is fully reachable by an unprivileged external attacker who deploys a contract and lures users into converting tokens.

### Recommendation

In the `TransferMethodName` case, enforce that the supplied `sender` equals `contract.Caller()`:

```go
if sender != contract.Caller() {
    return nil, errors.New("sender must be the calling contract")
}
```

Apply the same guard to the `BurnMethodName` case (verify `recipient == contract.Caller()`). This mirrors the authorization model already used in `utils.go` for the relayer precompile.

### Proof of Concept

1. Attacker deploys `MaliciousToken` (a CRC20 contract) at address `0xMAL`.
2. Victim acquires `MaliciousToken` ERC20 tokens and calls `MaliciousToken.moveToNative(amount)`, which internally calls `bank.mint(victim, amount)`. Victim now holds `amount` of denom `"evm/0xMAL"` in the Cosmos bank.
3. Attacker calls a function on `MaliciousToken` that executes:
   ```solidity
   IBankModule(0x64).transfer(victimAddress, attackerAddress, amount);
   ``` [4](#0-3) 
4. The bank precompile sets `denom = "evm/0xMAL"` (from `contract.Caller()`), `from = victim`, `to = attacker`, and calls `bankKeeper.SendCoins(ctx, victim, attacker, coins)` — no victim authorization checked.
5. Victim's entire native balance of `"evm/0xMAL"` is transferred to the attacker. The victim's CRC20 tokens were already burned in step 2, so the loss is permanent.

The `TestBank.sol` integration contract demonstrates the intended call pattern (`bank.transfer(msg.sender, recipient, amount)`), confirming the precompile was designed to be called by the token contract on behalf of users — but the precompile itself never enforces this constraint. [5](#0-4)
