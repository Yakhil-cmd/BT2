### Title
Stop-loss timelock increase does not retroactively protect pending proposals, allowing pool admin to bypass the new longer timelock - (File: metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol)

---

### Summary

`OracleValueStopLossExtension` computes each proposal's `executeAfter` once at proposal time using the **current** timelock value. When the timelock is subsequently increased, all previously-queued proposals (drawdown, decay, watermarks) retain their old, shorter `executeAfter` timestamps and can be executed immediately after the timelock increase — bypassing the protection the new longer timelock was meant to provide.

---

### Finding Description

Every `propose*` function in `OracleValueStopLossExtension` calls `_afterTimelock`:

```solidity
// line 297-299
function _afterTimelock(address pool_) private view returns (uint32) {
    return uint32(block.timestamp + oracleStopLossConfig[pool_].timelock);
}
```

This snapshots `block.timestamp + currentTimelock` into `pendingXxxExecuteAfter` at proposal time. [1](#0-0) 

For example, `proposeOracleStopLossDrawdown` stores the result:

```solidity
// lines 106-108
uint32 executeAfter = _afterTimelock(pool_);
sched.pendingDrawdownE6 = uint32(newMaxDrawdownE6);
sched.pendingDrawdownExecuteAfter = executeAfter;
``` [2](#0-1) 

The corresponding `execute*` function only checks the stored `executeAfter`, never re-reading the current timelock:

```solidity
// lines 114-115
if (sched.pendingDrawdownExecuteAfter == 0) revert OracleStopLossNoPendingDrawdown(pool_);
_requireElapsed(sched.pendingDrawdownExecuteAfter);
``` [3](#0-2) 

The same pattern applies to `proposeOracleStopLossDecay` / `executeOracleStopLossDecay` and `proposeOracleStopLossHighWatermarks` / `executeOracleStopLossHighWatermarks`. [4](#0-3) 

The timelock itself is also changed through the same propose-execute pattern, meaning the timelock increase goes through the **old** (shorter) delay before taking effect. [5](#0-4) 

**Attack sequence:**

1. Pool is initialized with `timelock = 1 day`, `drawdownE6 = 100_000` (10% — tight LP protection).
2. Admin calls `proposeOracleStopLossDrawdown(pool, 900_000)` (90% — nearly no protection). `pendingDrawdownExecuteAfter = T₀ + 1 day`.
3. Admin calls `proposeOracleStopLossTimelock(pool, 30 days)`. `pendingTimelockExecuteAfter = T₀ + 1 day`.
4. After 1 day, admin calls `executeOracleStopLossTimelock` → timelock is now 30 days. LPs observe the 30-day timelock and believe they are safe.
5. Admin immediately calls `executeOracleStopLossDrawdown` — `pendingDrawdownExecuteAfter = T₀ + 1 day` has already elapsed, so it succeeds. Drawdown is now 90%.
6. LPs received only 1 day of notice instead of 30 days. The stop-loss is now nearly disabled.

---

### Impact Explanation

The `OracleValueStopLossExtension` is the primary on-chain mechanism protecting LP principal from value extraction through manipulated swaps. The drawdown parameter directly controls the maximum tolerated per-bin value loss before swaps are halted. [6](#0-5) 

By bypassing the timelock, the admin can silently weaken the drawdown threshold (or disable decay) without giving LPs the full timelock window to exit. This is a direct loss of LP principal protection and constitutes an admin-boundary break: the pool admin is semi-trusted only within caps and timelocks, and this path circumvents the timelock constraint.

---

### Likelihood Explanation

The pool admin is a semi-trusted role. The attack requires the admin to pre-stage a drawdown proposal before increasing the timelock — a deliberate two-step action. No external actors or special conditions are needed beyond the admin's own transactions. Any pool using `OracleValueStopLossExtension` with a mutable timelock is affected.

---

### Recommendation

When `executeOracleStopLossTimelock` increases the timelock, re-stamp all pending proposals' `executeAfter` to `max(current executeAfter, block.timestamp + newTimelock)`. Alternatively, on each `execute*` call, enforce that the elapsed time since proposal is at least the **current** timelock (not just the stored `executeAfter`):

```solidity
function executeOracleStopLossDrawdown(address pool_) external onlyPoolAdmin(pool_) {
    PoolStopLossSchedule storage sched = _initializedSchedule(pool_);
    if (sched.pendingDrawdownExecuteAfter == 0) revert OracleStopLossNoPendingDrawdown(pool_);
    // Enforce both the stored executeAfter AND the current timelock
    uint32 currentDeadline = uint32(
        Math.max(sched.pendingDrawdownExecuteAfter,
                 block.timestamp - oracleStopLossConfig[pool_].timelock + sched.pendingDrawdownExecuteAfter)
    );
    _requireElapsed(currentDeadline);
    ...
}
```

The simplest correct fix: when executing any pending proposal, require `block.timestamp >= pendingXxxExecuteAfter` **and** `block.timestamp >= proposalTimestamp + currentTimelock`. Since `proposalTimestamp` is not stored, the cleanest approach is to re-stamp `pendingXxxExecuteAfter = max(pendingXxxExecuteAfter, block.timestamp + newTimelock)` inside `executeOracleStopLossTimelock` whenever the new timelock is larger than the old one.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.35;

// Scenario (pseudo-test):
// 1. Pool initialized: timelock=1 day, drawdown=100_000 (10%)
// 2. T=0: admin proposes drawdown=900_000 → pendingDrawdownExecuteAfter = 1 day
// 3. T=0: admin proposes timelock=30 days → pendingTimelockExecuteAfter = 1 day
// 4. T=1 day: admin executes timelock → oracleStopLossConfig[pool].timelock = 30 days
//             LPs see 30-day timelock, believe they are protected
// 5. T=1 day: admin executes drawdown → succeeds (pendingDrawdownExecuteAfter elapsed)
//             drawdown is now 900_000 (90%) — stop-loss nearly disabled
//             LPs had only 1 day notice, not 30 days

function test_timelockBypass() public {
    // Initialize with 1-day timelock, 10% drawdown
    extension.initialize(pool, abi.encode(uint32(100_000), uint32(0), uint32(1 days)));

    vm.startPrank(admin);
    // Stage the malicious drawdown proposal under the short timelock
    extension.proposeOracleStopLossDrawdown(pool, 900_000);
    // Stage the timelock increase under the same short timelock
    extension.proposeOracleStopLossTimelock(pool, 30 days);
    vm.stopPrank();

    // Advance only 1 day
    vm.warp(block.timestamp + 1 days);

    vm.startPrank(admin);
    // Execute timelock increase — now timelock = 30 days
    extension.executeOracleStopLossTimelock(pool);
    // Execute drawdown change — succeeds despite 30-day timelock now active
    extension.executeOracleStopLossDrawdown(pool); // should revert but does NOT
    vm.stopPrank();

    (uint32 dd,,,) = extension.oracleStopLossConfig(pool);
    assertEq(dd, 900_000); // drawdown weakened with only 1-day notice
}
```

  function proposeOracleStopLossTimelock(address pool_, uint32 newTimelock) external onlyPoolAdmin(pool_) {
    PoolStopLossSchedule storage sched = _initializedSchedule(pool_);
    uint32 executeAfter = _afterTimelock(pool_);
    sched.pendingTimelock = newTimelock;
    sched.pendingTimelockExecuteAfter = executeAfter;
    emit OracleStopLossTimelockProposed(pool_, newTimelock, executeAfter);
  }

  function executeOracleStopLossTimelock(address pool_) external onlyPoolAdmin(pool_) {
    PoolStopLossSchedule storage sched = _initializedSchedule(pool_);
    if (sched.pendingTimelockExecuteAfter == 0) revert OracleStopLossNoPendingTimelock(pool_);
    _requireElapsed(sched.pendingTimelockExecuteAfter);
    uint32 timelock = sched.pendingTimelock;
    oracleStopLossConfig[pool_].timelock = timelock;
    (sched.pendingTimelock, sched.pendingTimelockExecuteAfter) = (0, 0);
    emit OracleStopLossTimelockSet(pool_, timelock);
  }
