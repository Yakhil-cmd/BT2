### Title
Admin Fee Destination Change Does Not Flush Accrued Fees, Sending Them to the Wrong Recipient — (File: metric-core/contracts/MetricOmmPoolFactory.sol)

---

### Summary

`setPoolAdminFeeDestination` updates `poolAdminFeeDestination[pool]` without first collecting accrued fees to the old destination. Any spread surplus and notional fees that accrued while the old destination was active are subsequently paid to the new destination, permanently depriving the old recipient of funds they are owed.

---

### Finding Description

`setPoolAdminFees` explicitly collects fees before mutating the fee config: [1](#0-0) 

This pattern is intentional: fees accrued under the old rates must be settled before the split changes. The same invariant applies to the fee *destination* — fees accrued while `oldTreasury` was the destination should be paid to `oldTreasury`, not to whatever address replaces it.

`setPoolAdminFeeDestination`, however, performs no such flush: [2](#0-1) 

After this call, the next invocation of `collectPoolFees`, `setPoolAdminFees`, or `setPoolProtocolFee` will call `collectFees` with the **new** destination: [3](#0-2) 

Inside `collectFees`, both the spread surplus and the notional accumulator are paid out to `adminFeeDestination_`: [4](#0-3) 

All fees that accrued before the destination change are therefore sent to the new address, not the old one.

---

### Impact Explanation

The old admin fee destination — which may be a separate DAO treasury, multisig, or revenue-sharing contract — permanently loses all spread and notional fees that accrued before the destination was changed. The new destination receives funds it was never owed. This is a direct, irreversible loss of owed protocol revenue with no recovery path.

---

### Likelihood Explanation

The pool admin is a semi-trusted role that can call `setPoolAdminFeeDestination` at any time without a timelock. The old and new destinations are frequently distinct entities (e.g., a treasury rotation). Any destination change after meaningful swap volume has occurred triggers the loss. No special preconditions beyond normal pool operation are required.

---

### Recommendation

Mirror the pattern used in `setPoolAdminFees`: collect fees to the current destination before overwriting it.

```solidity
function setPoolAdminFeeDestination(address pool, address newAdminFeeDestination)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (newAdminFeeDestination == address(0)) revert InvalidAdminFeeDestination();

    // Flush accrued fees to the OLD destination before rotating.
    PoolFeeConfig memory c = poolFeeConfig[pool];
    IMetricOmmPoolCollectFees(pool).collectFees(
        c.protocolSpreadFeeE6,
        c.adminSpreadFeeE6,
        c.protocolNotionalFeeE8,
        c.adminNotionalFeeE8,
        poolAdminFeeDestination[pool]   // old destination
    );

    poolAdminFeeDestination[pool] = newAdminFeeDestination;
    emit PoolAdminFeeDestinationUpdated(pool, newAdminFeeDestination);
}
```

---

### Proof of Concept

1. Pool is deployed; `poolAdminFeeDestination[pool] = oldTreasury`.
2. Traders execute swaps; spread surplus accumulates in the pool balance and `notionalFeeToken0Scaled` / `notionalFeeToken1Scaled` grow.
3. Pool admin calls `setPoolAdminFeeDestination(pool, newTreasury)` — no `collectFees` is triggered.
4. Anyone calls `collectPoolFees(pool)`:
   - `collectFees` is invoked with `adminFeeDestination_ = newTreasury`.
   - All accrued admin-share tokens are transferred to `newTreasury`.
5. `oldTreasury` receives nothing despite being the entitled recipient for the entire accrual period. [2](#0-1) [3](#0-2) [5](#0-4)
