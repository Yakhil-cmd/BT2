### Title
Gateway Skips `__validate__` Signature Verification for Invoke Transactions Paired with Pending Deploy-Account — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The gateway's stateful validator contains a UX feature (`skip_stateful_validations`) that unconditionally bypasses the account's `__validate__` entry-point — the only place where the transaction signature is cryptographically verified — for any Invoke transaction with `nonce == 1` whose sender address already appears in the mempool or a recent block. An unprivileged attacker who observes a pending `DeployAccount` transaction in the mempool can immediately submit a crafted Invoke transaction carrying an arbitrary (invalid) signature for the same address, and the gateway will admit it without ever verifying the signature.

---

### Finding Description

**Root cause — `skip_stateful_validations`**

In `crates/apollo_gateway/src/stateful_transaction_validator.rs` the function `skip_stateful_validations` returns `true` (skip) when all three conditions hold:

1. The transaction is an `Invoke`.
2. `tx.nonce() == Nonce(Felt::ONE)`.
3. `account_nonce == Nonce(Felt::ZERO)` (account not yet on-chain).
4. `mempool_client.account_tx_in_pool_or_recent_block(sender)` returns `true`. [1](#0-0) 

The boolean is then forwarded directly to `run_validate_entry_point`, which sets `execution_flags.validate = !skip_validate`: [2](#0-1) 

Inside `StatefulValidator::perform_validations`, when `execution_flags.validate == false` the code returns `Ok(())` before ever calling `tx.validate_tx` (the `__validate__` entry-point): [3](#0-2) 

`validate_tx` itself also short-circuits on the same flag: [4](#0-3) 

**What the stateless validator does NOT do**

`StatelessTransactionValidator::validate` checks signature *length* but never verifies the cryptographic content: [5](#0-4) 

**What the mempool does NOT do**

`validate_by_mempool` only checks nonce ordering; it does not inspect the signature: [6](#0-5) 

**Attack path**

1. Alice broadcasts a `DeployAccount` for address `X` (valid signature, nonce 0). It enters the mempool; `account_tx_in_pool_or_recent_block(X)` now returns `true`.
2. Attacker submits `Invoke { sender: X, nonce: 1, signature: [0xdead, 0xbeef] }` — any garbage signature.
3. Stateless validator: signature length ≤ max → passes.
4. Stateful validator: nonce=1, account_nonce=0, address in mempool → `skip_validate = true` → `__validate__` never called → transaction admitted.
5. The attacker's Invoke is now in the mempool alongside Alice's `DeployAccount`.

**Displacement sub-attack**

If Alice also submitted a legitimate Invoke (nonce=1) for the UX flow, the attacker can submit a higher-tip fake Invoke for the same (address, nonce=1) slot. The mempool's fee-escalation logic will evict Alice's legitimate transaction in favour of the attacker's invalid one. The attacker's transaction will fail during block execution (signature invalid), and Alice must resubmit. [7](#0-6) 

---

### Impact Explanation

**Broken invariant:** Every transaction admitted to the mempool must carry a signature that the account's `__validate__` entry-point would accept.

**Concrete wrong value:** The admission decision for an Invoke with an invalid signature is `Ok(account_nonce)` (accepted) instead of `Err(ValidateFailure)` (rejected).

**Matching impact tier:** *High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.*

Secondary effects:
- Mempool pollution with transactions that will always fail during block execution, wasting sequencer resources.
- Griefing of the deploy-account + invoke UX flow by displacing legitimate nonce-1 transactions via fee escalation.

---

### Likelihood Explanation

- `DeployAccount` transactions are publicly visible in the mempool.
- The attacker needs only to observe a pending `DeployAccount`, craft an Invoke with `nonce=1` and any signature bytes, and submit it. No privileged access, no special knowledge, no on-chain funds required.
- The condition `nonce == 1 && account_nonce == 0 && in_mempool` is reliably satisfied for every new account deployment.

Likelihood: **High**.

---

### Recommendation

1. **Verify the signature even when skipping the entry-point call.** Before setting `validate = false`, perform a lightweight off-chain ECDSA check against the stored public key (or the constructor calldata of the pending `DeployAccount`) so that at minimum the signature is well-formed for the expected signer.

2. **Narrow the skip condition.** Instead of checking `account_tx_in_pool_or_recent_block` (which matches *any* transaction type), explicitly verify that the pending mempool transaction for the address is a `DeployAccount`. This prevents the skip from being triggered by unrelated future-nonce transactions.

3. **Align with the external-report fix pattern.** Analogously to the Freeverse fix (requiring the seller to sign the `paymentId`), require the submitter to provide a signature over a minimal commitment (e.g., the transaction hash) that can be verified without executing the not-yet-deployed contract, and verify it in the gateway before admission.

---

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
