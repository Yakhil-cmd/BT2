### Title
Malicious CKD Coordinator Can Unilaterally Corrupt the Confidential Key Derivation Output — (`File: src/confidential_key_derivation/protocol.rs`)

---

### Summary

The CKD protocol designates a single coordinator who aggregates all participants' partial shares and is the **sole** recipient of the final `CKDOutput`. Non-coordinator participants send their shares privately to the coordinator and immediately return `Ok(None)` with no further protocol steps. There is no mechanism for participants to verify that the coordinator correctly aggregated their contributions. A malicious coordinator can silently corrupt `big_y` and `big_c` before returning the output, causing the TEE application to receive an incorrect confidential key. This is a direct analog to the single-EOA-owner pattern: one designated party holds unilateral, unverifiable control over a critical protocol output.

---

### Finding Description

In `do_ckd_participant`, each non-coordinator participant computes its share `(norm_big_y, norm_big_c)` and sends it privately to the coordinator, then terminates:

```rust
// src/confidential_key_derivation/protocol.rs lines 27-32
let (norm_big_y, norm_big_c) =
    compute_signature_share(participants, me, key_pair, app_id, app_pk, rng)?;
let waitpoint = chan.next_waitpoint();
chan.send_private(waitpoint, coordinator, &(norm_big_y, norm_big_c))?;
Ok(None)
``` [1](#0-0) 

The coordinator in `do_ckd_coordinator` receives all shares, sums them, and constructs the final `CKDOutput` with no integrity check:

```rust
// src/confidential_key_derivation/protocol.rs lines 44-57
let (mut norm_big_y, mut norm_big_c) =
    compute_signature_share(&participants, me, key_pair, app_id, app_pk, rng)?;
let waitpoint = chan.next_waitpoint();
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
Ok(Some(ckd_output))
``` [2](#0-1) 

The coordinator is the **only** party that receives `Some(CKDOutput)`. The type alias makes this explicit:

```rust
// src/confidential_key_derivation/mod.rs line 60
pub type CKDOutputOption = Option<CKDOutput>;
``` [3](#0-2) 

The `unmask` function on the output computes `big_c − a·big_y = msk·H(pk‖app_id)`:

```rust
// src/confidential_key_derivation/mod.rs lines 54-56
pub fn unmask(&self, secret_scalar: Scalar) -> Signature {
    self.big_c - self.big_y * secret_scalar
}
``` [4](#0-3) 

A malicious coordinator can replace the accumulated `norm_big_y` and `norm_big_c` with arbitrary group elements before constructing `CKDOutput`. Because participants have already terminated with `Ok(None)`, they have no opportunity to detect or reject this substitution. The TEE application receives a `CKDOutput` that decrypts to a wrong or attacker-chosen group element instead of `msk·H(pk‖app_id)`.

Contrast this with the ECDSA signing coordinator, which performs a self-verifying step — it checks `sig.verify(&public_key, &msg_hash)` before returning — so a corrupt aggregation is caught locally:

```rust
// src/ecdsa/robust_ecdsa/sign.rs lines 159-163
if !sig.verify(&public_key, &msg_hash) {
    return Err(ProtocolError::AssertionFailed(
        "signature failed to verify".to_string(),
    ));
}
``` [5](#0-4) 

No equivalent self-verification exists in `do_ckd_coordinator`. The CKD output is an ElGamal ciphertext; its correctness can only be checked externally by the TEE app using the BLS pairing after the protocol has already completed.

---

### Impact Explanation

**High — Corruption of CKD outputs so honest parties accept unusable or incorrect cryptographic outputs.**

A malicious coordinator can produce a `CKDOutput` whose `unmask` result is an arbitrary group element rather than `msk·H(pk‖app_id)`. The TEE application will either:
- Receive a key that fails BLS pairing verification (detectable but denial-of-service), or
- Receive a key that the coordinator has chosen to be a specific value (if the coordinator can solve the discrete log, which is computationally infeasible, but the coordinator can still produce a key that is inconsistent with the master secret key).

In either case, honest participants have completed the protocol believing it succeeded, while the coordinator has unilaterally determined the output. This is a single point of failure: compromise or malice of one designated party fully controls the CKD result for all participants.

---

### Likelihood Explanation

**Medium-High.** The coordinator is a single designated participant chosen at protocol initialization. The attack requires no cryptographic break, no external key leakage, and no side-channel — only that the coordinator deviates from the aggregation step in `do_ckd_coordinator`. The coordinator's deviation is undetectable by other participants during the protocol run. The only post-hoc check is the TEE app's BLS pairing verification, which occurs outside the library's scope.

---

### Recommendation

1. **Short term:** After aggregation, the coordinator should broadcast the final `(big_y, big_c)` to all participants so each can independently verify the sum equals the expected linear combination of their own contribution and the received commitments.
2. **Short term:** Participants should commit to their `(norm_big_y, norm_big_c)` values (e.g., via a hash broadcast) before sending them privately to the coordinator, enabling post-hoc auditability.
3. **Long term:** Redesign the CKD aggregation phase to be symmetric (all-to-all broadcast of shares) so no single coordinator controls the final output, analogous to how the DKG protocol uses reliable broadcast for all critical values.

---

### Proof of Concept

1. Honest participants `P_1, P_2, P_3` run `ckd(...)` with `P_1` as coordinator.
2. `P_2` and `P_3` each compute `(norm_big_y_i, norm_big_c_i)` and send them privately to `P_1` via `chan.send_private`.
3. `P_2` and `P_3` return `Ok(None)` — protocol complete from their perspective.
4. `P_1` (malicious coordinator) in `do_ckd_coordinator` ignores the received values and instead constructs:
   ```rust
   let ckd_output = CKDOutput::new(
       ElementG1::generator() * Scalar::random(rng),  // arbitrary big_y
       ElementG1::generator() * Scalar::random(rng),  // arbitrary big_c
   );
   Ok(Some(ckd_output))
   ```
5. The TEE application calls `ckd_output.unmask(app_sk)` and receives a random group element instead of `msk·H(pk‖app_id)`.
6. No participant other than `P_1` observes any protocol error; `P_2` and `P_3` have already exited with `Ok(None)`. [6](#0-5)
