### Title
`skip_stateful_validations` Bypasses `__validate__` for Invoke Transactions When Any Non-Deploy-Account Tx Exists in Mempool — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

`skip_stateful_validations` is designed to skip the `__validate__` entry-point call for an invoke with `nonce=1` when the account's on-chain nonce is `0`, provided a `deploy_account` transaction is pending in the mempool (UX flow: deploy + first invoke submitted together). The check it uses — `account_tx_in_pool_or_recent_block(address)` — is keyed only on **address**, not on **transaction type**. It returns `true` for any transaction type in the pool. An attacker who observes that a deployed account (class hash ≠ 0, on-chain nonce = 0) has a pending nonce-0 invoke in the mempool can submit a nonce-1 invoke with an arbitrary payload and invalid signature; the gateway skips `__validate__`, accepts the transaction, and forwards it to the mempool.

---

### Finding Description

**Invariant broken:** The skip-validate path must only activate when a `DeployAccount` transaction for the sender is pending, because that is the only case where the account contract does not yet exist on-chain and `__validate__` cannot run.

**Root cause — single-key lookup instead of double-key (address + tx-type):**

`skip_stateful_validations` (lines 429–461) fires when:
- incoming tx is `Invoke`
- `tx.nonce() == 1`
- `account_nonce == 0`
- `account_tx_in_pool_or_recent_block(sender) == true` [1](#0-0) 

`account_tx_in_pool_or_recent_block` checks only the address key:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [2](#0-1) 

It returns `true` for **any** transaction type — `Invoke`, `Declare`, or `DeployAccount`. The code comment claims this is sufficient because the account "either has a deploy_account transaction or transactions with future nonces that passed validations," but it omits the third case: a **nonce-0 invoke** from an already-deployed account. A nonce-0 invoke passes full `__validate__` at the gateway (nonce-0 does not trigger the skip path), lands in the pool, and then causes `account_tx_in_pool_or_recent_block` to return `true` for any subsequent nonce-1 submission. [3](#0-2) 

When `skip_validate = true`, the gateway sets `execution_flags.validate = false`:

```rust
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [4](#0-3) 

Inside `StatefulValidator::perform_validations`, the `__validate__` call is gated on this flag:

```rust
if !tx.execution_flags.validate {
    return Ok(());
}
// `__validate__` call.
``` [5](#0-4) 

The public `validate` method on `StatefulValidatorTrait` delegates directly to `perform_validations`:

```rust
fn validate(&mut self, account_tx: AccountTransaction) -> StatefulValidatorResult<()> {
    self.perform_validations(account_tx)
}
``` [6](#0-5) 

`validate_state_preconditions` only checks resource bounds and nonce range — it does not verify the account is undeployed: [7](#0-6) 

The nonce check for a non-declare, non-deploy-account tx with `tx_nonce=1`, `account_nonce=0` passes trivially (`0 ≤ 1 ≤ max_allowed_nonce`): [8](#0-7) 

**Attack path:**

1. Alice's account is deployed on-chain (class hash ≠ 0, nonce = 0 in state).
2. Alice submits an invoke with nonce = 0; it passes full `__validate__` and enters the mempool.
3. Attacker submits an invoke for Alice's address with nonce = 1, arbitrary calldata, and an invalid/empty signature.
4. Gateway: `account_nonce = 0`, `tx.nonce() = 1`, `account_tx_in_pool_or_recent_block = true` (Alice's nonce-0 invoke) → `skip_validate = true` → `__validate__` is **not called** → transaction is accepted.
5. Mempool: nonce-1 is within the allowed gap → transaction is stored.
6. Batcher: retrieves the attacker's tx, creates a fresh `AccountTransaction` with `validate = true`, calls `__validate__` → fails → transaction reverts (no fee charged, no state change).

---

### Impact Explanation

**High — Mempool/gateway admission accepts invalid transactions.**

The gateway accepts an invoke transaction whose signature has never been verified by the account contract. The mempool is polluted with attacker-controlled transactions. Because the attacker's nonce-1 slot is occupied, Alice's own legitimate nonce-1 invoke is rejected with `DuplicateNonce` until the attacker's transaction is processed and evicted. This is a repeatable, fee-free griefing vector against any deployed account that has a pending nonce-0 invoke in the mempool.

---

### Likelihood Explanation

Any deployed account with on-chain nonce = 0 and a pending nonce-0 invoke in the mempool is vulnerable. This is a common state immediately after account deployment. The attack requires no privileged access — only the ability to observe the mempool (public) and submit a transaction to the gateway (public RPC endpoint). The attacker pays no fee because the transaction reverts at `__validate__` during execution.

---

### Recommendation

Replace the type-blind `account_tx_in_pool_or_recent_block` check with a type-aware query that returns `true` only when a `DeployAccount` transaction for the address is pending. Concretely, expose a `deploy_account_tx_in_pool(address) -> bool` method on the mempool that inspects the transaction type stored in `tx_pool`, and use it in `skip_stateful_validations` instead of the current address-only lookup.

---

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
