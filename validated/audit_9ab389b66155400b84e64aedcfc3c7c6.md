### Title
Gateway L2 Gas Price Validation Uses Stale `l2_gas_price` Instead of `next_l2_gas_price`, Causing Incorrect Transaction Admission and Rejection — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

`StatefulTransactionValidator::validate_resource_bounds` validates a transaction's `max_price_per_unit` against the **current** block's `l2_gas_price` read from the latest committed block header. However, the transaction will be executed in the **next** block at `next_l2_gas_price`, which is the EIP-1559-adjusted price already computed and stored in that same block header. Because `GatewayFixedBlockSyncStateClient` never surfaces `next_l2_gas_price` through `BlockInfo`, the gateway uses a stale reference price, causing it to admit transactions that will fail during blockifier execution (price rising) and to reject transactions that would succeed (price falling). A developer TODO in the code explicitly acknowledges the gap.

### Finding Description

When block N is finalized, `SequencerConsensusContext::finalize_decision` calls `update_l2_gas_price(height, l2_gas_used)`, which runs the EIP-1559 formula and stores the result in `self.l2_gas_price`. That updated value is then written into block N's header as `next_l2_gas_price`:

```rust
// sequencer_consensus_context.rs
let block_header_without_hash = BlockHeaderWithoutHash {
    l2_gas_price,                          // block N's price
    next_l2_gas_price: self.l2_gas_price,  // block N+1's price (just computed)
    ...
};
``` [1](#0-0) 

When the gateway later validates a new transaction, `GatewayFixedBlockSyncStateClient::get_block_info_from_sync_client` reads block N's header but maps only `block_header.l2_gas_price.price_in_fri` into `BlockInfo`. The `next_l2_gas_price` field is silently dropped:

```rust
// gateway_fixed_block_state_reader.rs
strk_gas_prices: GasPriceVector {
    l2_gas_price: block_header.l2_gas_price.price_in_fri.try_into()?,
    // next_l2_gas_price is never read
    ...
},
``` [2](#0-1) 

`validate_resource_bounds` then uses this stale price as the validation reference, with a developer TODO acknowledging the problem:

```rust
// stateful_transaction_validator.rs
// TODO(Arni): getnext_l2_gas_price from the block header.
let previous_block_l2_gas_price = self
    .gateway_fixed_block_state_reader
    .get_block_info()
    .await?
    .gas_prices
    .strk_gas_prices
    .l2_gas_price;
self.validate_tx_l2_gas_price_within_threshold(
    executable_tx.resource_bounds(),
    previous_block_l2_gas_price,
)?;
``` [3](#0-2) 

The threshold is `l2_gas_price × min_gas_price_percentage / 100` (default 100%), but the blockifier will execute the transaction against `next_l2_gas_price`. The two values diverge every block where gas utilization differs from the gas target. [4](#0-3) 

### Impact Explanation

**High.** Two distinct failure modes arise:

1. **Price rising** (`next_l2_gas_price > l2_gas_price`): A transaction with `max_price_per_unit` in the range `[l2_gas_price, next_l2_gas_price)` passes the gateway threshold check but fails `check_fee_bounds` inside `AccountTransaction::perform_pre_validation_stage` during blockifier execution. The gateway incorrectly admits a transaction that will be rejected at execution time.

2. **Price falling** (`next_l2_gas_price < l2_gas_price`): A transaction with `max_price_per_unit` in the range `[next_l2_gas_price, l2_gas_price)` is rejected by the gateway with `GAS_PRICE_TOO_LOW` even though it would satisfy the actual next-block price and succeed during execution. The gateway incorrectly rejects a valid transaction.

Both outcomes match the allowed impact: *"High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."* [5](#0-4) 

### Likelihood Explanation

High. The EIP-1559 L2 gas price is recalculated every block based on actual gas consumption versus the gas target. Any block whose utilization differs from the target — which is the common case in production — produces `next_l2_gas_price ≠ l2_gas_price`. The discrepancy is bounded per block by `gas_price_max_change_denominator`, but it is non-zero on virtually every block. No special attacker capability is required; any ordinary user submitting a transaction at the boundary price triggers the incorrect admission or rejection. [6](#0-5) 

### Recommendation

1. Extend `GatewayFixedBlockStateReader::get_block_info` (or add a dedicated method) to return `next_l2_gas_price` from `BlockHeaderWithoutHash`.
2. In `GatewayFixedBlockSyncStateClient::get_block_info_from_sync_client`, read `block_header.next_l2_gas_price` and expose it.
3. Replace the stale `l2_gas_price` reference in `validate_resource_bounds` with the exposed `next_l2_gas_price`, resolving the existing TODO.

### Proof of Concept

**Scenario A — false positive admission (price rising):**

1. Block N finalizes with `l2_gas_price = 100 fri` and full utilization; EIP-1559 computes `next_l2_gas_price = 110 fri`, stored in block N's header.
2. User submits an invoke V3 transaction with `l2_gas.max_price_per_unit = 105 fri`.
3. Gateway reads block N's header: `l2_gas_price = 100`, `next_l2_gas_price` is not surfaced.
4. `validate_tx_l2_gas_price_within_threshold`: `105 ≥ 100 × 100% = 100` → **passes**; transaction enters mempool.
5. Block N+1 is built with `l2_gas_price = 110 fri`.
6. Blockifier `check_fee_bounds`: `105 < 110` → `MaxGasPriceTooLow` → **execution fails**.
7. Result: gateway admitted an invalid transaction; mempool slot and execution resources wasted.

**Scenario B — false negative rejection (price falling):**

1. Block N finalizes with `l2_gas_price = 100 fri` and low utilization; EIP-1559 computes `next_l2_gas_price = 90 fri`.
2. User submits a transaction with `l2_gas.max_price_per_unit = 95 fri`.
3. Gateway: `95 < 100 × 100% = 100` → **`GAS_PRICE_TOO_LOW`; transaction rejected**.
4. Block N+1 would execute at `l2_gas_price = 90 fri`; `95 ≥ 90` → the transaction would have **succeeded**.
5. Result: a valid transaction is incorrectly rejected; user must resubmit with an artificially inflated price. [7](#0-6) [8](#0-7) [9](#0-8)

    fn validate_tx_l2_gas_price_within_threshold(
        &self,
        tx_resource_bounds: ValidResourceBounds,
        previous_block_l2_gas_price: NonzeroGasPrice,
    ) -> StatefulTransactionValidatorResult<()> {
        match tx_resource_bounds {
            ValidResourceBounds::AllResources(tx_resource_bounds) => {
                let tx_l2_gas_price = tx_resource_bounds.l2_gas.max_price_per_unit;
                let gas_price_threshold_multiplier =
                    Ratio::new(self.config.min_gas_price_percentage.into(), 100_u128);
                let threshold = (gas_price_threshold_multiplier
                    * previous_block_l2_gas_price.get().0)
                    .to_integer();
                if tx_l2_gas_price.0 < threshold {
                    return Err(StarknetError {
                        // We didn't have this kind of an error.
                        code: StarknetErrorCode::UnknownErrorCode(
                            "StarknetErrorCode.GAS_PRICE_TOO_LOW".to_string(),
                        ),
                        message: format!(
                            "Transaction L2 gas price {tx_l2_gas_price} is below the required \
                             threshold {threshold}.",
                        ),
                    });
                }
            }
            ValidResourceBounds::L1Gas(_) => {
                // No validation required for legacy transactions.
            }
        }
        Ok(())
    }

```rust
pub fn calculate_next_base_gas_price(
    price: GasPrice,
    gas_used: GasAmount,
    gas_target: GasAmount,
    min_gas_price: GasPrice,
) -> GasPrice {
    let versioned_constants = VersionedConstants::latest_constants();
    assert!(
        gas_target < versioned_constants.max_block_size,
        "Gas target must be lower than max block size."
    );
    assert!(gas_target.0 > 0, "Gas target must be greater than zero.");
    assert!(
        versioned_constants.gas_price_max_change_denominator > 0,
        "Denominator constant must be greater than zero."
    );

    // If the current price is below the minimum, apply a gradual adjustment and return early.
    // This allows the price to increase by at most 1/MIN_GAS_PRICE_INCREASE_DENOMINATOR per block.
    if price < min_gas_price {
        let max_increase = price.0 / MIN_GAS_PRICE_INCREASE_DENOMINATOR;
        let adjusted = price.0 + max_increase;
        // Cap at min_gas_price to avoid overshooting
        let adjusted_price = adjusted.min(min_gas_price.0);
        info!(
            "Fee Market: Price {} below minimum gas price {}, adjusted price: {} )",
            price.0, min_gas_price.0, adjusted_price
        );
        return GasPrice(adjusted_price);
    }

    // Use U256 to avoid overflow, as multiplying a u128 by a u64 remains within U256 bounds.
    let gas_delta = U256::from(gas_used.0.abs_diff(gas_target.0));
    let gas_target_u256 = U256::from(gas_target.0);
    let price_u256 = U256::from(price.0);

    // Calculate price change by multiplying first, then dividing. This avoids the precision loss
    // that occurs when dividing before multiplying.
    let denominator =
        gas_target_u256 * U256::from(versioned_constants.gas_price_max_change_denominator);
    let price_change = (price_u256 * gas_delta) / denominator;

    let adjusted_price_u256 =
        if gas_used > gas_target { price_u256 + price_change } else { price_u256 - price_change };

    // Sanity check: ensure direction of change is correct
    assert!(
        gas_used > gas_target && adjusted_price_u256 >= price_u256
            || gas_used <= gas_target && adjusted_price_u256 <= price_u256
    );

    // Price should not realistically exceed u128::MAX, bound to avoid theoretical overflow.
    let adjusted_price = u128::try_from(adjusted_price_u256).unwrap_or(u128::MAX);
    GasPrice(max(adjusted_price, min_gas_price.0))
}
