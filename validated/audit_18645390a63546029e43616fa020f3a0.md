### Title
WalletConnect `chia_cancelOffer` Confirmation Shows Only Raw `trade_id`, Enabling Targeted Offer Cancellation Without Meaningful User Consent - (File: packages/gui/src/electron/commands/Commands.ts)

---

### Summary

The WalletConnect `chia_cancelOffer` dapp command presents a confirmation dialog that displays only a raw hex `trade_id` string and a fee amount. It omits all human-readable offer details (assets offered, assets requested, offer status). A malicious dapp that has been granted `chia_cancelOffer` permission can silently enumerate the user's pending offers via the no-confirmation-required `chia_getOffer` command, identify a high-value pending trade, and then invoke `chia_cancelOffer` targeting that specific offer. The user is shown only an opaque hex string and cannot determine which offer is being cancelled, making meaningful consent impossible.

---

### Finding Description

The `chia_cancelOffer` dapp command is defined in `Commands.ts` with the following confirmation params exposed to the user:

```
params: [
  { name: 'trade_id', type: 'string' },   // raw hex, e.g. "0x3f4a..."
  { name: 'fee',      type: 'bigint', humanize: 'mojo-to-xch' },
]
```

The `secure` parameter is hidden from the user and hardcoded to `true` via `defaults: { secure: true }`. [1](#0-0) 

The confirmation dialog title is "Confirm Cancel Offer" and the message is "Please carefully review and confirm this offer cancellation." — but the only reviewable data is the raw `trade_id` hex string. No offer summary (offered assets, requested assets, counterparty, status) is resolved or displayed. [2](#0-1) 

Separately, `chia_wallet.get_offer`, `chia_wallet.get_offer_summary`, and `chia_wallet.check_offer_validity` are all listed in `DAPP_ALLOWED_COMMANDS` — meaning a paired dapp can call them **without any user confirmation dialog** to silently enumerate all pending offers and their `trade_id` values. [3](#0-2) 

The attack chain:

1. Dapp is paired and granted `chia_cancelOffer` (requires user approval at pairing, but is a bypassable spending command the user may grant to a "trading assistant" dapp).
2. Dapp silently calls `chia_getOffer` / `chia_getAllOffers` (no confirmation required) to enumerate all pending offers and identify a high-value one.
3. Dapp calls `chia_cancelOffer` with the targeted `trade_id`.
4. User sees only the raw hex `trade_id` and fee in the confirmation dialog — no offer details.
5. User approves (cannot distinguish this from a legitimate cancellation request).
6. The pending offer — locking XCH, CAT, or NFT — is cancelled on-chain with `secure: true`, permanently invalidating it.

---

### Impact Explanation

**High.** A malicious dapp with `chia_cancelOffer` permission can cause the user to approve cancellation of a specific high-value pending offer (XCH, CAT, NFT) by presenting an opaque `trade_id`. The user has no way to verify which offer is being cancelled from the confirmation dialog. The result is permanent, on-chain cancellation of the offer (with `secure: true`), destroying the pending trade and spending a fee. This is WalletConnect state that causes a user to approve the wrong action on the wrong asset — matching the "Corruption, spoofing, or unsafe trust of WalletConnect state that causes a user to approve... the wrong asset, identity, amount, destination, or status" criterion.

---

### Likelihood Explanation

Medium. The attacker must have already been granted `chia_cancelOffer` permission by the user. However, a dapp marketed as a "trading assistant" or "offer manager" would naturally request this permission. Once granted, the dapp can silently enumerate offers (no confirmation required) and target any pending offer. The user has no defense at the confirmation step because the dialog provides no offer context.

---

### Recommendation

The `chia_cancelOffer` WalletConnect confirmation dialog should resolve the `trade_id` to human-readable offer details before presenting the confirmation. Specifically:

1. Before showing the confirmation, call `get_offer` with the provided `trade_id` to retrieve the offer summary.
2. Display the offered assets, requested assets, and offer status in the confirmation dialog — analogous to how the native `ConfirmOfferCancellation` component is used in the GUI after the user selects an offer from their own list.
3. If the `trade_id` cannot be resolved (offer not found or not owned by the current wallet), reject the request before showing any confirmation. [4](#0-3) 

---

### Proof of Concept

**Setup**: User has a pending offer to sell 1 NFT for 10 XCH (`trade_id = "0xabc123..."`). User has also granted a "trading assistant" dapp `chia_cancelOffer` permission.

**Attack**:

```javascript
// Step 1: Silently enumerate offers (no confirmation required)
const offers = await walletConnect.request({
  method: 'chia_getAllOffers',
  params: { fingerprint: 123456 }
});
// Identify the high-value NFT offer: trade_id = "0xabc123..."

// Step 2: Request cancellation of the targeted offer
await walletConnect.request({
  method: 'chia_cancelOffer',
  params: {
    fingerprint: 123456,
    trade_id: '0xabc123...',  // the NFT offer
    fee: 50000000             // 0.00005 XCH
  }
});
```

**User sees in confirmation dialog**:
- Trade Id: `0xabc123...`
- Fee: `0.00005 XCH`

The user has no way to know this is their NFT offer. If they click "Proceed", the offer is cancelled on-chain (`secure: true`), the NFT is unlocked but the trade is permanently lost, and the fee is spent. [5](#0-4) [3](#0-2) [6](#0-5)
