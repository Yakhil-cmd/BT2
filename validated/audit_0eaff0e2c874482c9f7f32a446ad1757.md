### Title
`StateDB.Commit()` Skips Account Recreation for Addresses That Are Both Self-Destructed and Created in the Same Transaction, Permanently Destroying Balance — (`fvm/evm/emulator/state/stateDB.go`)

### Summary

`StateDB.Commit()` contains a logic flaw: when an account is both flagged as `HasSelfDestructed` **and** `IsCreated` within the same transaction (the "recreated" path, triggered by deploying a contract to a pre-existing address), `Commit()` deletes the account and then unconditionally skips the recreation step via `continue`. The carried-over balance is permanently lost from the base view.

### Finding Description

**Root cause — `DeltaView.CreateAccount()` sets both `toBeDestructed` and `created` for a pre-existing address:**

When `CreateAccount(X)` is called on an address X that already exists (e.g., during a CREATE2 deployment to a pre-funded address), `DeltaView.CreateAccount()` internally calls `SelfDestruct(X)` to reset the account, then marks X in both `toBeDestructed` and `created`, and carries the original balance forward in `d.balances[addr]`: [1](#0-0) 

After this call:
- `toBeDestructed[X]` = original balance B (set by the internal `SelfDestruct` call)
- `created[X]` = set
- `balances[X]` = B (carried over, overriding the zero set by `SelfDestruct`)
- `recreated[X]` = set (the `DeltaView` even has a dedicated map for this case)

The `recreated` map comment explicitly acknowledges this state: *"flag used to track accounts that has been flagged for destruction but recreated later"*. [2](#0-1) 

**Root cause — `StateDB.Commit()` never checks `IsCreated` when `HasSelfDestructed` is true:** [3](#0-2) 

The `if deleted { continue }` at line 486-488 unconditionally skips the `IsCreated` branch at lines 495-511. For a recreated account, both `HasSelfDestructed` and `IsCreated` are true, but only the deletion is executed. `baseView.CreateAccount(X, bal, ...)` is never called, so the carried-over balance B is never written to persistent storage.

The slot-update loop at lines 542-558 also skips slots for self-destructed addresses, compounding the loss: [4](#0-3) 

### Impact Explanation

An attacker can permanently destroy the balance of any EVM account at a CREATE2-computable address by deploying a contract to it. The balance is zeroed in the delta view by `SelfDestruct`, the carried-over value is never flushed to `baseView`, and `DeleteAccount` removes the account record entirely. After `Commit()` + `Finalize()`, the address has zero balance and does not exist — the funds are unrecoverable.

This violates the invariant that an unprivileged EVM transaction cannot destroy another account's balance.

### Likelihood Explanation

The attack is straightforward and requires no special privileges:

1. Compute a CREATE2 target address X = `keccak256(0xff ++ deployer ++ salt ++ initcode_hash)[12:]`
2. Ensure X has a non-zero balance (either by sending funds to it, or targeting an address that already received funds — a common pattern in counterfactual deployment)
3. Submit an EVM transaction that deploys a contract to X via CREATE2

The EVM will call `CreateAccount(X)` during deployment (standard Geth behavior when deploying to an existing address), triggering the buggy path. The attack is deterministic and locally reproducible on the Flow emulator.

### Recommendation

In `StateDB.Commit()`, after deleting a self-destructed account, check whether the address is also marked as `IsCreated` (i.e., it was recreated in the same transaction). If so, proceed to call `baseView.CreateAccount(...)` with the current balance/nonce/code rather than skipping via `continue`:

```go
if db.HasSelfDestructed(addr) {
    err = db.baseView.DeleteAccount(addr)
    // ...
    deleted = true
}
// If the account was recreated after self-destruct, fall through to IsCreated
if deleted && !db.IsCreated(addr) {
    continue
}
```

The `recreated` map already exists in `DeltaView` for exactly this purpose — `Commit()` should consult it (or equivalently use `IsCreated`) to distinguish a pure self-destruct from a self-destruct-then-recreate.

### Proof of Concept

```
1. Fund address X = CREATE2(deployer, salt, initcode) with 1 ETH via a Cadence EVM transaction.
2. In a new block, submit an EVM transaction from `deployer` that executes CREATE2 with the same salt/initcode, targeting X.
3. The EVM calls StateDB.CreateAccount(X):
   - DeltaView.SelfDestruct(X) sets toBeDestructed[X]=1ETH, balances[X]=0
   - DeltaView carries over: balances[X]=1ETH, created[X]=set, recreated[X]=set
4. StateDB.Commit() is called:
   - HasSelfDestructed(X) → true → baseView.DeleteAccount(X) → deleted=true → continue
   - IsCreated(X) branch is never reached
   - baseView.CreateAccount(X, 1ETH, ...) is never called
5. After Finalize(), query X:
   - Exist(X) → false
   - GetBalance(X) → 0
   - GetCode(X) → nil
   - The 1 ETH is permanently destroyed.
``` [5](#0-4) [6](#0-5)

func (d *DeltaView) CreateAccount(addr gethCommon.Address) error {
	// if is already created return
	if d.IsCreated(addr) {
		return nil
	}
	exist, err := d.Exist(addr)
	if err != nil {
		return err
	}
	if exist {
		// check if already destructed
		destructed, balance := d.HasSelfDestructed(addr)
		if !destructed {
			balance, err = d.GetBalance(addr)
			if err != nil {
				return err
			}
			err = d.SelfDestruct(addr)
			if err != nil {
				return err
			}
		}

		d.nonces[addr] = 0
		d.codes[addr] = nil
		d.codeHashes[addr] = gethTypes.EmptyCodeHash
		// carrying over the balance. (legacy behavior of the Geth stateDB)
		d.balances[addr] = balance

		// flag addr as recreated, this flag helps with postponing deletion of slabs
		// otherwise we have to iterate over all slabs of this account and set the to nil
		d.recreated[addr] = struct{}{}

		// remove slabs from cache related to this account
		for k := range d.slots {
			if k.Address == addr {
				delete(d.slots, k)
			}
		}
	}
	d.dirtyAddresses[addr] = struct{}{}
	d.created[addr] = struct{}{}
	return nil
}
