Looking at the CKD protocol in `src/confidential_key_derivation/protocol.rs`, I need to examine how the coordinator aggregates participant contributions and whether any component of that aggregation is missing from a security-critical check — the direct analog of the original "bid price not accounted for" bug class.

### Title
Unverified Participant Contributions in CKD Coordinator Allow Malicious Participant to Corrupt Confidential Key Derivation Output - (File: src/confidential_key_derivation/protocol.rs)

---

### Summary

In `do_ckd_coordinator`, the coordinator sums `(norm_big_y, norm_big_c)` shares received from every participant with no cryptographic check that each contribution is correctly formed with respect to the sender's actual key share. A single malicious participant can inject arbitrary group elements, causing the coordinator to produce and accept a corrupted `CKDOutput` that the application will unmask to a wrong confidential key.

---

### Finding Description

**Root cause — missing component in the aggregation check**

Each honest participant is supposed to compute (in `compute_signature_share`, lines 148–182):

```
big_y  = y_i · G₁
big_c  = x_i · H(pk ‖ app_id) + y_i · app_pk
norm_big_y = λ_i · big_y
norm_big_c = λ_i · big_c
```

so that after the coordinator sums all shares:

```
Y = Σ λ_i · y_i · G₁
C = msk · H(pk ‖ app_id) + Y · app_sk
```

The coordinator's aggregation loop in `do_ckd_coordinator` (lines 50–55) is:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
```

There is **no verification** that the received `(norm_big_y, norm_big_c)` satisfies the required relationship

```
norm_big_c − norm_big_y · app_pk  =  λ_i · x_i · H(pk ‖ app_id)
```

The coordinator blindly adds whatever group elements it receives. This is the direct analog of the original bug: the check accounts for only part of the relevant data (the received points) and ignores the component that would bind those points to the participant's actual secret share.

**Attack path**

A malicious participant `P_m` replaces its honest output with arbitrary group elements `(Δ_Y, Δ_C)` before sending to the coordinator. The coordinator computes:

```
Y_out = Y_honest + Δ_Y
C_out = C_honest + Δ_C
```

The application then unmasks:

```
C_out − Y_out · app_sk
  = msk · H(pk ‖ app_id) + (Δ_C − Δ_Y · app_sk)
```

The result is a corrupted confidential key. Because `P_m` does not know `app_sk`, it cannot target a specific output, but it can make the output wrong in an undetectable way. The coordinator and all honest parties accept this corrupted `CKDOutput` as valid — there is no subsequent integrity check.

---

### Impact Explanation

A single malicious participant (no special privilege beyond holding a valid DKG share) can cause every honest party, including the coordinator, to accept a `CKDOutput` whose confidential key is cryptographically wrong. The application will silently derive an incorrect key, making the CKD output permanently unusable. This matches the allowed **High** impact: *"Corruption of CKD outputs so honest parties accept inconsistent… unusable cryptographic outputs."*

---

### Likelihood Explanation

Any participant enrolled in the CKD session can execute this attack. No cryptographic assumption needs to be broken; the attacker only needs to deviate from the protocol by sending two arbitrary `G₁` points instead of the honestly computed share. The attack is single-round, requires no coordination, and leaves no detectable trace in the coordinator's output.

---

### Recommendation

Each participant should accompany its `(norm_big_y, norm_big_c)` with a Maurer/Schnorr ZK proof of knowledge of `(x_i, y_i)` satisfying:

```
norm_big_y = λ_i · y_i · G₁
norm_big_c − norm_big_y · app_pk = λ_i · x_i · H(pk ‖ app_id)
vk_share_i = x_i · G₂          (participant's public key share from DKG)
```

The coordinator must verify this proof before adding the contribution to the running sum. This requires passing individual public key shares (`vk_share_i`) into the CKD protocol alongside the aggregate `public_key`. The existing `dlogeq` proof infrastructure in `src/crypto/proofs/dlogeq.rs` can be reused for this purpose.

---

### Proof of Concept

1. Run DKG with participants `[P1, P2, P3]`; `P3` is the coordinator.
2. `P1` is malicious. Instead of calling `compute_signature_share`, `P1` constructs `CKDOutput::new(G₁, 2·G₁)` (arbitrary points) and sends it privately to `P3`.
3. `P3` receives `P1`'s contribution and `P2`'s honest contribution, sums them, and returns `CKDOutput::new(norm_big_y, norm_big_c)`.
4. The application calls `ckd_output.unmask(app_sk)` and obtains `msk · H(pk ‖ app_id) + (2·G₁ − G₁ · app_sk)`, which is not the correct confidential key.
5. Any downstream operation that depends on the correct confidential key (e.g., decryption, re-encryption) silently fails or produces wrong results.
