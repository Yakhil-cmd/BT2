### Title
`Account.Call()` Does Not Enforce EVM Execution Result Checking, Enabling Silent Cross-VM Asset Loss - (File: `fvm/evm/handler/handler.go`)

### Summary

`Account.Call()` in `fvm/evm/handler/handler.go` returns a `*types.ResultSummary` to the Cadence caller without panicking on a failed EVM execution. Unlike `Deposit`, `Withdraw`, and `Transfer` — which all call `panicOnErrorOrInvalidOrFailedState` and revert the Cadence transaction on any EVM failure — `Call` only calls `panicOnError(err)`, which only catches fatal/backend errors. A failed EVM execution (e.g., ERC-20 `transfer()` reverts, contract is destroyed, or runs out of gas) silently returns `Status = StatusFailed` to the Cadence caller. If the caller does not check `res.status`, the Cadence transaction proceeds as if the EVM call succeeded, enabling cross-VM asset loss.

### Finding Description

`Account.Call()` is the Go-level implementation backing `coa.call()` in Cadence. It executes an EVM call via `executeAndHandleAuthorizedCall` and then only calls `panicOnError(err)`:

```go
func (a *Account) Call(...) *types.ResultSummary {
    res, err := a.executeAndHandleAuthorizedCall(...)
    panicOnError(err)          // only panics on fatal/backend errors
    return res.ResultSummary() // returns failed result silently
}
``` [1](#0-0) 

By contrast, `Deposit`, `Withdraw`, and `Transfer` all call `panicOnErrorOrInvalidOrFailedState(res, err)`, which panics — and thus reverts the Cadence transaction — on both invalid and failed EVM execution:

```go
func (a *Account) Deposit(v *types.FLOWTokenVault) {
    res, err := a.fch.executeAndHandleCall(...)
    panicOnErrorOrInvalidOrFailedState(res, err) // panics on failed EVM state
}
``` [2](#0-1) [3](#0-2) [4](#0-3) 

`panicOnErrorOrInvalidOrFailedState` explicitly panics on `res.Failed()` (i.e., `VMError != nil`): [5](#0-4) 

The EVM `Status` enum has three non-unknown states: `invalid`, `failed`, and `successful`. A `failed` status means the EVM executed the transaction but the VM reported an error (e.g., revert, out of gas). The EVM block still records the transaction and the nonce is consumed: [6](#0-5) 

The `EVM.Result` struct returned to Cadence carries `status`, `errorCode`, and `errorMessage`, but there is no protocol-level enforcement that the Cadence caller checks these fields before proceeding: [7](#0-6) 

### Impact Explanation

Any Cadence transaction or contract that uses `coa.call()` to invoke an EVM token transfer (e.g., ERC-20 `transfer()`, ERC-721 `safeTransferFrom()`) without checking `res.status == EVM.Status.successful` will silently proceed even when the EVM call fails. If the Cadence side releases escrowed assets, mints tokens, or records a balance change contingent on the EVM call succeeding, the result is **cross-VM asset loss**: Cadence-side assets are released while the EVM-side transfer never occurred.

This is the direct analog to the Solidity report: just as `safeTransfer` using a low-level call returns `success = true` when the target contract does not exist, `coa.call()` returns a `Result` struct without reverting the Cadence transaction when the EVM call fails — the caller must manually check the result, and if it does not, the failure is silent.

### Likelihood Explanation

The likelihood is **medium**. The `coa.call()` API is the primary mechanism for Cadence contracts to interact with EVM contracts. The Flow VM bridge contracts (deployed via `setupVMBridge`) use `coa.call()` internally to invoke ERC-20/ERC-721 operations on the EVM side. Any bridge contract or user-authored Cadence contract that omits a `res.status` check after `coa.call()` is vulnerable. The inconsistency between `Call`/`Deploy` (no panic on failure) and `Deposit`/`Withdraw`/`Transfer` (panic on failure) makes this easy to miss. [8](#0-7) 

### Recommendation

**Short term:** Apply `panicOnErrorOrInvalidOrFailedState(res, err)` in `Account.Call()` (and `Account.Deploy()`) instead of `panicOnError(err)`, consistent with `Deposit`, `Withdraw`, and `Transfer`. This ensures that any EVM call failure immediately reverts the enclosing Cadence transaction, preventing silent cross-VM asset loss.

**Long term:** Audit all Cadence contracts (including bridge contracts) that invoke `coa.call()` to verify they check `res.status == EVM.Status.successful` before proceeding with Cadence-side state changes. Add a protocol-level post-condition or wrapper that enforces result checking for asset-transfer calls.

### Proof of Concept

A Cadence transaction that silently loses assets:

```cadence
import EVM from <EVM_ADDRESS>
import SomeToken from <TOKEN_ADDRESS>

transaction {
    prepare(signer: auth(Storage) &Account) {
        let coa = signer.storage.borrow<auth(EVM.Call) &EVM.CadenceOwnedAccount>(
            from: /storage/evmCOA
        )!

        // ERC-20 transfer calldata: transfer(address,uint256)
        let calldata: [UInt8] = [...] // encodes transfer(victim, 1000)

        // coa.call() returns Result with status=failed if ERC-20 reverts
        // (e.g., contract destroyed, insufficient balance, or any revert)
        let res = coa.call(
            to: EVM.EVMAddress(bytes: erc20ContractAddress),
            data: calldata,
            gasLimit: 100_000,
            value: EVM.Balance(attoflow: 0)
        )
        // res.status may be EVM.Status.failed here, but NO CHECK IS PERFORMED

        // Cadence-side asset release proceeds regardless of EVM call outcome
        // e.g., unlock escrowed Cadence tokens, record balance, etc.
        releaseEscrowedAssets() // executes even though EVM transfer failed
    }
}
```

The root cause is at `fvm/evm/handler/handler.go:1057` where `panicOnError(err)` is used instead of `panicOnErrorOrInvalidOrFailedState(res, err)`, allowing `res.ResultSummary()` with `Status = StatusFailed` to be returned to the Cadence caller without reverting the transaction. [9](#0-8)

func panicOnErrorOrInvalidOrFailedState(res *types.Result, err error) {

	if res != nil && res.Invalid() {
		panic(fvmErrors.NewEVMError(res.ValidationError))
	}

	if res != nil && res.Failed() {
		panic(fvmErrors.NewEVMError(res.VMError))
	}

	// this should never happen
	if err == nil && res == nil {
		panic(fvmErrors.NewEVMError(types.ErrUnexpectedEmptyResult))
	}

	panicOnError(err)
}

    access(all) struct Result {

        access(all) let errorCode: UInt64

        /// error message
        access(all) let errorMessage: String

        /// returns the amount of gas metered during
        /// evm execution
        access(all) let gasUsed: UInt64

        /// returns the data that is returned from
        /// the evm for the call. For coa.deploy
        /// calls it returns the code deployed to
        /// the address provided in the contractAddress field.
        /// in case of revert, the smart contract custom error message
        /// is also returned here (see EIP-140 for more details).
        access(all) let data: [UInt8]

        /// returns the newly deployed contract address
        /// if the transaction caused such a deployment
        /// otherwise the value is nil.
        access(all) let deployedContract: EVMAddress?

        init(
            status: Status,
            errorCode: UInt64,
            errorMessage: String,
            gasUsed: UInt64,
            data: [UInt8],
            contractAddress: [UInt8; 20]?
        ) {
            self.status = status
            self.errorCode = errorCode
            self.errorMessage = errorMessage
            self.gasUsed = gasUsed
            self.data = data

                self.deployedContract = EVMAddress(bytes: addressBytes)
            } else {
