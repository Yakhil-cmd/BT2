### Title
Pool admin bypasses global fee cap via uncapped `setPoolBinAdditionalFees` — (File: metric-core/contracts/MetricOmmPoolFactory.sol)

### Summary
`MetricOmmPoolFactory.setPoolBinAdditionalFees` imposes no upper-bound check on `addFeeBuyE6` / `addFeeSellE6`, while the parallel global-fee setter `setPoolAdminFees` is strictly capped at `maxAdminSpreadFeeE6`. A pool admin can therefore set per-bin additional fees up to `type(uint16).max` = 65 535 (≈ 6.55 % in E6 units) on any bin, silently exceeding whatever fee ceiling the factory owner intended to enforce.

### Finding Description
The factory enforces a two-layer fee cap system:

**Layer 1 – global admin spread fee (capped):** [1](#0-0) 

```solidity
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (newAdminNotionalFeeE8 > maxAdminNotionalFeeE8) revert AdminFeeTooHigh();
```

**Layer 2 – per-bin additional fee (no cap):** [2](#0-1) 

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

The pool-level handler also performs no cap check — it only validates the bin index: [3](#0-2) 

The per-bin additional fee is added directly to the base fee inside every swap path: [4](#0-3) 

```solidity
params.baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6)
```

The same pattern appears for the sell direction: [5](#0-4) 

Because `addFeeBuyE6` / `addFeeSellE6` are `uint16`, the maximum settable value is 65 535, equivalent to ≈ 6.55 % in E6 notation. This is applied **on top of** the global spread fee and is **not counted** against `maxAdminSpreadFeeE6`. The factory owner can set `maxAdminSpreadFeeE6 = 0` to prohibit any admin spread fee, yet the pool admin can still charge 6.55 % per bin via this path.

### Impact Explanation
**High impact, low likelihood.** Any trader routing through an affected bin pays the uncapped per-bin fee in addition to the global spread fee. Because the fee is deducted from the swap input (exact-in) or added to the required input (exact-out), the trader receives fewer output tokens than the fee cap system was designed to guarantee. This is a direct, quantifiable loss of user principal on every swap touching the manipulated bin. The factory owner's ability to enforce fee ceilings — the primary protection for traders against pool-admin overreach — is silently nullified for the per-bin dimension.

### Likelihood Explanation
Low. Exploitation requires a malicious or compromised pool admin. Pool admins are semi-trusted and are expected to operate within the cap system. However, the cap system itself is broken for this code path, so no external attacker capability is needed — the pool admin alone can trigger the loss.

### Recommendation
Add the same cap guard to `setPoolBinAdditionalFees` that exists in `setPoolAdminFees`. The simplest fix is to enforce that `addFeeBuyE6` and `addFeeSellE6` do not exceed `maxAdminSpreadFeeE6` (cast to `uint16` with an overflow check), or introduce a dedicated `maxBinAdditionalFeeE6` constant. The pool-level `setBinAdditionalFees` should also validate the cap so that the invariant is preserved even if the factory is upgraded or bypassed.

```solidity
// In MetricOmmPoolFactory.setPoolBinAdditionalFees:
if (addFeeBuyE6  > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
```

### Proof of Concept
1. Factory owner deploys factory with `maxAdminSpreadFeeE6 = 1 000` (0.1 % cap).
2. Pool is created; pool admin calls `setPoolAdminFees(pool, 1_000, 0)` — passes cap check.
3. Pool admin calls `setPoolBinAdditionalFees(pool, 0, 65_535, 65_535)` — **no revert**.
4. A trader executes an exact-in swap through bin 0. The effective fee applied is:
   - `baseFeeX64` (oracle spread) + `1 000 / 1e6` (global admin) + `65 535 / 1e6` (per-bin) ≈ oracle spread + **6.65 %**
5. The factory owner's intended 0.1 % admin ceiling is bypassed by a factor of ~65×, and the trader loses the excess fee amount with no on-chain guard preventing it.
