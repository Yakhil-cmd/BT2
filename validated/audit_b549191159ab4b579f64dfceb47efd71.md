### Title
Missing Zero-Address Check in `receive_base_tokens` Permanently Freezes Deposited ETH - (File: `engine/src/engine.rs`)

---

### Summary

`Engine::receive_base_tokens` mints Aurora ETH directly to whatever address is decoded from the `ft_on_transfer` message without checking whether that address is `Address::zero()`. Any unprivileged user who calls `ft_transfer_call` on the eth-connector with `msg = "0000000000000000000000000000000000000000"` will have their NEP-141 ETH tokens consumed while the equivalent Aurora ETH is credited to the zero address — permanently inaccessible.

---

### Finding Description

`receive_base_tokens` is invoked from `ft_on_transfer` whenever the predecessor is the eth-connector account (i.e., a base-token deposit into Aurora). [1](#0-0) 

Inside `receive_base_tokens`, the recipient is decoded from the caller-supplied `args.msg` string via `FtTransferMessageData::try_from`: [2](#0-1) 

`FtTransferMessageData::try_from` accepts any 40-hex-character string and decodes it into an `Address` with no zero-address guard: [3](#0-2) 

After parsing, `receive_base_tokens` immediately credits the balance to whatever address was decoded — including `Address::zero()`: [4](#0-3) 

No check of the form `if recipient == Address::zero() { return Err(...) }` exists anywhere in this path.

---

### Impact Explanation

When `recipient` is `Address::zero()`, `set_balance` writes the minted ETH to the zero-address slot in Aurora's EVM state. No private key controls `0x0000000000000000000000000000000000000000`, so the ETH is permanently inaccessible. The user's NEP-141 ETH tokens are simultaneously consumed by the eth-connector. This constitutes **permanent freezing of funds**.

---

### Likelihood Explanation

The `msg` field of `ft_transfer_call` is a plain user-supplied string. A user who accidentally passes an all-zero hex string (e.g., copy-paste error, uninitialized buffer in a script, or a contract that forgot to set the recipient) triggers the loss with no further preconditions. No admin access or special role is required — any token holder can call `ft_transfer_call` on the eth-connector. This is directly analogous to the Gearbox liquidator calling with `to = address(0)` by mistake.

---

### Recommendation

Add a zero-address guard in `receive_base_tokens` before crediting the balance:

```rust
pub fn receive_base_tokens(
    &mut self,
    args: &FtOnTransferArgs,
) -> Result<Option<SubmitResult>, ContractError> {
    let message_data = FtTransferMessageData::try_from(args.msg.as_str())?;
    let receipient = message_data.recipient;

    // ADD: reject the zero address to prevent permanent fund loss
    if receipient == Address::zero() {
        return Err(errors::ERR_ZERO_ADDRESS_RECIPIENT.into());
    }

    let amount = Wei::new_u128(args.amount.as_u128());
    ...
}
```

Optionally, the same guard can be added inside `FtTransferMessageData::try_from` so all callers benefit automatically.

---

### Proof of Concept

1. User holds NEP-141 ETH tokens on the eth-connector contract.
2. User calls `ft_transfer_call` on the eth-connector:
   ```json
   {
     "receiver_id": "aurora",
     "amount": "1000000000000000000",
     "msg": "0000000000000000000000000000000000000000"
   }
   ```
3. The eth-connector calls `ft_on_transfer` on Aurora with the same `msg`.
4. `ft_on_transfer` routes to `engine.receive_base_tokens` because `predecessor == eth_connector_account_id`. [1](#0-0) 
5. `receive_base_tokens` decodes `recipient = Address::zero()` and calls `set_balance(&mut self.io, &Address::zero(), &new_balance)`. [4](#0-3) 
6. The function returns `Ok(None)`, the eth-connector considers the transfer successful (0 tokens returned), and the user's NEP-141 ETH is gone. The 1 ETH now sits at `0x0000000000000000000000000000000000000000` in Aurora's EVM state with no way to recover it.
