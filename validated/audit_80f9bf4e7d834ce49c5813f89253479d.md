### Title
Gateway Skips `__validate__` Signature Check for Invoke Transactions with Nonce=1 When Victim Has Pending Deploy-Account — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

`skip_stateful_validations` in the gateway's stateful validator unconditionally skips the `__validate__` entry-point (i.e., signature verification) for any Invoke transaction whose nonce is exactly 1 and whose sender address appears in the mempool. An attacker who observes a victim's pending `deploy_account` transaction can submit a crafted Invoke transaction (nonce=1, arbitrary calldata, garbage signature) from the victim's address. The gateway admits it to the mempool without verifying the signature. When the batcher later executes the block, `__validate__` is called and fails, the transaction reverts, and the victim is charged fees for a transaction they never signed.

### Finding Description

`skip_stateful_validations` is the Sequencer analog of the "founder-only maturity check" pattern: a restriction (signature/`__validate__` verification) is enforced for all transactions **except** one specific class (Invoke with nonce=1 from an address with a pending deploy-account), which is allowed to bypass it entirely.

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` sets `validate: false` in the `ExecutionFlags`, causing `StatefulValidator::perform_validations` to return early before calling `__validate__`: [2](#0-1) [3](#0-2) 

The `account_tx_in_pool_or_recent_block` check only verifies that the sender address has **any** transaction in the mempool state or a recent committed block — it does not verify that the pending transaction is a `deploy_account`, nor does it verify that the Invoke transaction being admitted is signed by the account owner: [4](#0-3) 

The pre-validation checks that **do** run (`handle_nonce`, `check_fee_bounds`, `verify_can_pay_committed_bounds`) do not verify the transaction signature: [5](#0-4) 

### Impact Explanation

An attacker can submit an Invoke transaction with:
- `sender_address` = any victim address that has a pending `deploy_account` in the mempool
- `nonce = 1`
- `signature = [garbage]`
- `calldata` = arbitrary (e.g., a call to drain funds)

The gateway admits this transaction to the mempool without signature verification. When the batcher executes the block, the victim's `deploy_account` (nonce=0) runs first, deploying the account. Then the attacker's Invoke (nonce=1) runs, `__validate__` is called with the deployed account code, the garbage signature fails, the transaction reverts, and the victim is charged fees for a transaction they never authorized.

The `max_nonce_for_validation_skip` config field (`0x1` in production) is only consumed by the Python `PyValidator` path; the Rust gateway's `skip_stateful_validations` hardcodes the nonce=1 check and does not consult this config: [6](#0-5) [7](#0-6) 

**Impact category:** High — Mempool/gateway admission accepts an invalid (unauthorized) transaction before sequencing. The victim pays fees for a transaction they never signed.

### Likelihood Explanation

- The attacker only needs to monitor the public mempool for `deploy_account` transactions (the sender address is visible in the transaction).
- The attack window is the time between the victim's `deploy_account` being submitted and being included in a block.
- No funds or special privileges are required by the attacker.
- The attack is limited to one unauthorized transaction per victim per deploy-account event (only nonce=1 is skipped).

### Recommendation

The skip condition should additionally verify that the transaction carries a plausible signature before bypassing `__validate__`. One approach: perform a lightweight signature-format check (e.g., non-empty, correct length) before granting the skip. A stronger fix is to require that the pending mempool entry for the sender is specifically a `deploy_account` transaction, not just any transaction. The `account_tx_in_pool_or_recent_block` check should be replaced with a `deploy_account_in_pool` check that inspects the transaction type.

### Proof of Concept

1. Alice submits `deploy_account` (nonce=0, pre-funded address). It enters the mempool.
2. Attacker observes Alice's address from the mempool.
3. Attacker submits `Invoke { sender_address: Alice, nonce: 1, signature: [0xdead], calldata: [...] }`.
4. Gateway stateful validation:
   - `validate_nonce`: `0 <= 1 <= 200` → passes (falls into `_` arm).
   - `validate_by_mempool`: nonce not too old → passes.
   - `skip_stateful_validations`: `nonce==1 && account_nonce==0 && account_in_mempool==true` → returns `true`.
   - `run_validate_entry_point` with `validate=false` → `__validate__` skipped → tx admitted.
5. Batcher executes block: Alice's `deploy_account` runs (nonce=0) → Alice deployed. Attacker's Invoke runs (nonce=1) → `__validate__` called → garbage signature fails → tx reverts → Alice charged fees. [8](#0-7)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
