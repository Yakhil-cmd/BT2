Audit Report

## Title
Malicious ICRC-1 Ledger Canister Can Permanently Halt Index-NG Canister via Balance Overflow/Underflow Trap - (File: `rs/ledger_suite/icrc1/index-ng/src/main.rs`)

## Summary

The `credit()` and `debit()` functions in the index-ng canister call `ic_cdk::trap()` on attacker-controlled arithmetic overflow/underflow. Because `is_build_index_running = true` is committed to state before the first `await` in `build_index()`, and because `ic_cdk::trap()` terminates Wasm execution without running Rust destructors (including the `scopeguard` that resets the flag), a malicious ICRC-1 ledger canister can permanently halt the index-ng canister's indexing loop with a single crafted block.

## Finding Description

**`credit()` and `debit()` trap on attacker-controlled values:**

Both functions call `ic_cdk::trap()` when arithmetic overflows or underflows. The amounts come directly from blocks returned by the configured ledger canister — an externally controlled source. [1](#0-0) 

**State committed before the trap:**

`build_index()` sets `is_build_index_running = true` at line 711, installs a `scopeguard` at line 713, then immediately hits `find_get_blocks_method().await` at line 718. In IC's execution model, state mutations before an `await` are committed when the inter-canister call is dispatched. The `is_build_index_running = true` mutation is permanently committed at that point. [2](#0-1) 

**Scopeguard destructor never runs on trap:**

`ic_cdk::trap()` calls `ic0.trap()`, a Wasm host function that terminates execution immediately without stack unwinding. Rust destructors — including the `scopeguard` guard — are never invoked. The flag `is_build_index_running` remains `true` permanently in committed state. [3](#0-2) 

**Timer is never rescheduled:**

`set_build_index_timer` is only called after `num_indexed` resolves (lines 729 and 736). A trap inside `fetch_blocks_via_get_blocks()` or `fetch_blocks_via_icrc3()` prevents this code from ever executing. [4](#0-3) 

**`process_balance_changes()` does not return a `Result`:**

`append_block()` calls `process_balance_changes()` at line 939, which returns `()`. Any trap inside it propagates as a Wasm trap rather than a `Result::Err`, bypassing the `SyncError` error-handling path that would otherwise allow graceful recovery. [5](#0-4) 

**Full call chain:**

`build_index()` → `fetch_blocks_via_get_blocks()` → `get_blocks_from_ledger().await` → (callback) `append_blocks()` → `append_block()` → `process_balance_changes()` → `credit()` / `debit()` → `ic_cdk::trap()`. [6](#0-5) [7](#0-6) 

**Every future invocation is permanently blocked:**

After the trap, every future call to `build_index()` returns `None` immediately at line 707-709 because `is_build_index_running` is `true` and is never reset. [8](#0-7) 

## Impact Explanation

The index-ng canister permanently stops indexing blocks. All user-facing queries — `get_account_transactions`, `icrc1_balance_of`, `list_subaccounts` — return stale or empty data indefinitely. Recovery requires an operator-initiated canister upgrade to reset `is_build_index_running`. This constitutes an **application/platform-level DoS** of the index-ng canister, matching the **High ($2,000–$10,000)** impact tier: "Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."

## Likelihood Explanation

The attacker must control the ledger canister that the target index-ng canister is configured to poll. This is a realistic constraint: any canister developer can deploy a malicious ICRC-1-compatible ledger on the IC. The crafted block is trivial to construct (e.g., a `Mint` of `Tokens::MAX` to an account with any nonzero balance, or a `Burn` exceeding any account's balance). The attack is one-shot, requires no privileged access, no consensus corruption, and no key compromise. It is irreversible without an operator upgrade.

## Recommendation

1. **Replace `ic_cdk::trap()` with `Result` propagation** in `credit()` and `debit()`. Return `Result<Tokens, SyncError>` and propagate errors up through `process_balance_changes()` → `append_block()` → `append_blocks()` as a non-retriable `SyncError`, so the indexing timer stops gracefully without corrupting `is_build_index_running`.

2. **Reset `is_build_index_running` unconditionally** in the timer callback wrapper rather than via a `scopeguard`, so the flag is reset even if the async body traps. For example, use a separate update message or a post-callback hook that unconditionally clears the flag.

3. **Validate block amounts before processing**: reject blocks where `amount > Tokens::MAX - current_balance` before calling `credit()`, returning a non-retriable `SyncError` rather than trapping.

## Proof of Concept

1. Deploy a malicious ICRC-1 ledger canister `M` on the IC (or a local PocketIC instance).
2. Deploy an index-ng canister `I` with `ledger_id = M`.
3. `M` initially returns one valid `Mint` block crediting `account_A` with balance `1`.
4. `I` indexes normally; `account_A` now has balance `1` in `I`'s state.
5. `M` then returns a second `Mint` block: `{ to: account_A, amount: Tokens::MAX }`.
6. `I`'s timer fires → `build_index()` sets `is_build_index_running = true` → `find_get_blocks_method().await` commits state.
7. Callback fetches blocks from `M` → `append_block()` → `process_balance_changes()` → `credit(account_A, Tokens::MAX)` → `1 + Tokens::MAX` overflows → `ic_cdk::trap()`.
8. Trap rolls back callback changes; `is_build_index_running = true` remains committed; scopeguard never runs; timer never rescheduled.
9. All future `build_index()` calls return `None` immediately. `I` is permanently halted.
10. Verify: query `icrc1_balance_of` on `I` — it returns stale data indefinitely; no new blocks are ever indexed.

fn debit(block_index: BlockIndex64, account: Account, amount: Tokens) {
    change_balance(account, |balance| {
        balance.checked_sub(&amount).unwrap_or_else(|| {
            ic_cdk::trap(format!("Block {block_index} caused an underflow for account {account} when calculating balance {balance} - amount {amount}"));
        })
    })
}

fn credit(block_index: BlockIndex64, account: Account, amount: Tokens) {
    change_balance(account, |balance| {
        balance.checked_add(&amount).unwrap_or_else(|| {
            ic_cdk::trap(format!("Block {block_index} caused an overflow for account {account} when calculating balance {balance} + amount {amount}"))
        })
    });
}
