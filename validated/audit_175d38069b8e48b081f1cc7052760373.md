### Title
`ExitToEthereum` Precompile Creates Fire-and-Forget Promise With No Failure Callback, Permanently Locking NEP-141 Tokens When `withdraw` Fails - (File: `engine-precompiles/src/native.rs`)

### Summary

The `ExitToEthereum` precompile burns ERC-20 tokens on the EVM side (irreversible), then creates a `PromiseArgs::Create` (fire-and-forget) promise to call `withdraw` on the eth-connector. There is no callback to handle promise failure. If the `withdraw` call fails for any reason, the ERC-20 tokens are permanently burned while the corresponding NEP-141 tokens remain locked in the connector contract with no recovery path.

### Finding Description

In `ExitToEthereum::run()`, after the ERC-20 contract has already burned the user's tokens, the precompile constructs a withdrawal promise:

```rust
let withdraw_promise = PromiseCreateArgs {
    target_account_id: nep141_address,
    method: "withdraw".to_string(),
    args: serialized_args,
    attached_balance: Yocto::new(1),
    attached_gas: costs::WITHDRAWAL_GAS,
};

let promise = borsh::to_vec(&PromiseArgs::Create(withdraw_promise)).unwrap();
``` [1](#0-0) 

This is unconditionally `PromiseArgs::Create` — a fire-and-forget call with **no callback**. If the `withdraw` call on the eth-connector fails (e.g., connector paused, gas exhaustion, or any runtime error), NEAR silently discards the failure. The ERC-20 burn has already been committed to EVM state, so the user's ERC-20 tokens are gone, while the NEP-141 tokens remain locked in the connector with no refund mechanism.

Contrast this with `ExitToNear`, which conditionally wraps the promise in a `PromiseArgs::Callback` pointing to `exit_to_near_precompile_callback` when the `error_refund` feature is enabled:

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs {
        base: transfer_promise,
        callback: PromiseCreateArgs {
            ...
            method: "exit_to_near_precompile_callback".to_string(),
            ...
        },
    })
};
``` [2](#0-1) 

`ExitToEthereum` has **no equivalent callback path at all**, regardless of feature flags.

The ERC-20 contract (`EvmErc20V2.sol`) confirms the burn is unconditional and precedes the precompile call:

```solidity
function withdrawToEthereum(address recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);
    ...
    assembly {
        let res := call(gas(), 0xb0bd02f6a392af548bdf1cfaee5dfa0eefcc8eab, ...)
    }
}
``` [3](#0-2)

### Impact Explanation

**Critical — Permanent freezing of funds.**

When the `withdraw` promise fails:
- ERC-20 tokens are already burned on the EVM side (irreversible, committed to EVM state)
- NEP-141 tokens remain locked in the eth-connector contract
- No withdrawal receipt is created for the Rainbow Bridge
- No refund callback exists to re-mint ERC-20 tokens or return NEP-141 tokens
- User funds are permanently frozen with zero recovery path

### Likelihood Explanation

**Medium.** The eth-connector supports pausability (confirmed by `test_withdraw_from_near_pausability` in `engine-tests-connector/src/connector.rs`). [5](#0-4) 

A user calling `withdrawToEthereum` while the connector's `withdraw` method is paused — or during any transient failure of the eth-connector — will have their ERC-20 tokens burned with no recourse. The user has no way to know in advance whether the downstream promise will succeed, and the EVM transaction itself succeeds regardless.

### Recommendation

Add a callback to `ExitToEthereum` analogous to `exit_to_near_precompile_callback`. The callback should detect promise failure and re-mint the burned ERC-20 tokens to the original sender (or credit the equivalent NEP-141 balance), mirroring the `refund_on_error` pattern already used in the `ExitToNear` path. [6](#0-5) 

### Proof of Concept

1. User holds 1000 units of an ERC-20 token on Aurora (backed by NEP-141 tokens in the eth-connector).
2. User calls `withdrawToEthereum(recipient, 1000)` on the ERC-20 contract.
3. ERC-20 contract executes `_burn(msg.sender, 1000)` — EVM state committed, tokens gone.
4. ERC-20 contract calls `ExitToEthereum` precompile, which schedules `PromiseArgs::Create(withdraw_promise)` targeting the eth-connector.
5. The eth-connector's `withdraw` method is paused (or fails for any reason).
6. NEAR runtime marks the promise as failed; no callback exists to handle this.
7. Result: 1000 ERC-20 tokens are permanently burned; 1000 NEP-141 tokens remain locked in the connector. The user has lost their funds with no recovery mechanism.

pub fn exit_to_near_precompile_callback<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        env.assert_private_call()?;

        // This function should only be called as the callback of
        // exactly one promise.
        if handler.promise_results_count() != 1 {
            return Err(errors::ERR_PROMISE_COUNT.into());
        }

        let args: ExitToNearPrecompileCallbackArgs = io.read_input_borsh()?;

        let maybe_result = if let Some(PromiseResult::Successful(_)) = handler.promise_result(0) {
            if let Some(args) = args.transfer_near {
                let action = PromiseAction::Transfer {
                    amount: Yocto::new(args.amount),
                };
                let promise = PromiseBatchAction {
                    target_account_id: args.target_account_id,
                    actions: vec![action],
                };

                // Safety: this call is safe because it comes from the exit to near precompile, not users.
                // The call is to transfer the unwrapped wNEAR tokens.
                let promise_id = handler.promise_create_batch(&promise);
                handler.promise_return(promise_id);
            }

            None
        } else if let Some(args) = args.refund {
            // Exit call failed; need to refund tokens
            let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;

            if !refund_result.status.is_ok() {
                return Err(errors::ERR_REFUND_FAILURE.into());
            }

            Some(refund_result)
        } else {
            None
        };

        Ok(maybe_result)
    })
}
