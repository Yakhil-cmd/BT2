### Title
Malicious CKD Participant Can Corrupt Aggregated Derived-Key Output Without Detection — (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

The Confidential Key Derivation (CKD) coordinator blindly sums every participant's `(norm_big_y, norm_big_c)` contribution with no proof of well-formedness. A single malicious participant can substitute arbitrary group elements, permanently biasing the aggregated output so the application (TEE) receives a silently wrong derived key. This is the direct analog of the TWAP-manipulation class: just as a manipulator can steer a time-weighted average by controlling inputs within an accepted range, a malicious CKD participant can steer the aggregated ElGamal ciphertext by sending any group elements that pass the only "check" — which is none.

---

### Finding Description

`do_ckd_coordinator` in `src/confidential_key_derivation/protocol.rs` collects one `CKDOutput` from every other participant and accumulates it:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
```

The honest computation each participant is supposed to perform is:

```
big_y  = y_i · G
big_s  = x_i · H(pk ‖ app_id)
big_c  = big_s + y_i · app_pk
norm_big_y = λ_i · big_y
norm_big_c = λ_i · big_c
```

There is **no zero-knowledge proof, no consistency check, and no range constraint** that the received `(norm_big_y, norm_big_c)` was produced this way. The coordinator accepts any two `G1` points.

The correct aggregate satisfies:

```
C − a · Y  =  msk · H(pk ‖ app_id)
```

If participant `P_m` sends `(norm_big_y + Δ_Y, norm_big_c + Δ_C)` for arbitrary `Δ_Y, Δ_C ∈ G1`, the coordinator computes:

```
Y' = Y + Δ_Y
C' = C + Δ_C
```

The TEE's unmask call `C' − a · Y'` yields:

```
msk · H(pk ‖ app_id)  +  Δ_C − a · Δ_Y
```

which is wrong for any `(Δ_Y, Δ_C)` not satisfying `Δ_C = a · Δ_Y` — and `P_m` does not know `a` (the application's secret key), so it cannot cancel the error. The TEE has no way to detect the corruption.

---

### Impact Explanation

**High — Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.**

The TEE silently receives a wrong derived key. Any downstream operation (decryption, signing, key agreement) that relies on `msk · H(pk ‖ app_id)` will produce incorrect or attacker-influenced results. Because the CKD protocol has no threshold structure — every participant's contribution is summed — a **single** malicious participant is sufficient to corrupt the output for every honest party.

---

### Likelihood Explanation

**Medium.** The attacker must be a registered participant in the CKD session. No external resources, leaked keys, or cryptographic breaks are required. The attack is a one-shot, single-message deviation (send wrong `big_c`/`big_y`) that is undetectable by the coordinator or the TEE. Any participant who wishes to deny service or deliver a wrong key to a target application can do so trivially.

---

### Recommendation

Require each participant to accompany their `(norm_big_y, norm_big_c)` with a Sigma-protocol (DLEQ / Chaum-Pedersen) proof that:

1. `norm_big_y = λ_i · y_i · G` for some `y_i` they know, and
2. `norm_big_c = λ_i · (x_i · H(pk ‖ app_id) + y_i · app_pk)` using the same `y_i` and the committed `x_i` (verifiable against the public key share from DKG).

The coordinator must verify all proofs before aggregating. This is the standard approach used in verifiable threshold ElGamal encryption (e.g., Pedersen's VSS, Schoenmakers' PVSS).

---

### Proof of Concept

```
Setup: 3 participants P1, P2, P3; coordinator = P1.
       Honest msk = x1+x2+x3 (Lagrange-weighted).

Attack (P3 is malicious):
  1. P3 computes legitimate (norm_big_y3, norm_big_c3) per compute_signature_share.
  2. P3 instead sends (norm_big_y3 + Δ_Y, norm_big_c3 + Δ_C)
     where Δ_Y, Δ_C are arbitrary non-zero G1 points.
  3. P1 (coordinator) runs:
       norm_big_y += big_y(P2);  norm_big_y += big_y(P3)+Δ_Y;
       norm_big_c += big_c(P2);  norm_big_c += big_c(P3)+Δ_C;
     → CKDOutput { big_y: Y+Δ_Y, big_c: C+Δ_C }
  4. TEE calls unmask(app_sk):
       (C+Δ_C) − app_sk·(Y+Δ_Y)
       = msk·H(pk‖app_id) + Δ_C − app_sk·Δ_Y   ← wrong key
  5. No error is raised anywhere in the protocol.
     The TEE uses the wrong derived key silently.
```

The entry path is entirely within `do_ckd_coordinator` [1](#0-0)  and `compute_signature_share` [2](#0-1) , with no validation between the two. [3](#0-2)

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}
