### Title
`skip_stateful_validations` Bypasses `__validate__` Signature Check via Overly Broad `account_tx_in_pool_or_recent_block` Guard - (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the gateway's stateful transaction validator is designed to skip the `__validate__` entry point (which performs cryptographic signature verification) for invoke transactions with nonce=1 when the account nonce is 0 on-chain and a deploy_account transaction is pending. The guard used to authorize this skip — `account_tx_in_pool_or_recent_block` — returns `true` for **any** transaction from the account, not just deploy_account transactions. An unprivileged attacker can exploit this to submit an invoke transaction with an invalid signature that is admitted to the mempool without any signature verification.

---

### Finding Description

`skip_stateful_validations` at lines 429–461 of `crates/apollo_gateway/src/stateful_transaction_validator.rs` implements a UX feature: when a user simultaneously submits a deploy_account + invoke pair, the invoke (nonce=1) is admitted even though the account does not yet exist on-chain. The function skips the blockifier's `__validate__` entry point under these conditions:

1. Transaction type is Invoke
2. Transaction nonce == 1
3. Account nonce on-chain == 0
4. `account_tx_in_pool_or_recent_block(sender_address)` returns `true` [1](#0-0) 

The authorization guard in condition 4 is implemented as:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [2](#0-1) 

This returns `true` for **any** transaction type in the pool — invoke, declare, or deploy_account. The code comment claims this is sufficient because the account "either has a deploy_account transaction or transactions with future nonces that passed validations." This reasoning is flawed: an account deployed via the `deploy` syscall has nonce=0 on-chain and can have a valid invoke (nonce=0) in the mempool. In that case, `account_tx_in_pool_or_recent_block` returns `true`, and a subsequent invoke with nonce=1 and an **invalid signature** bypasses `__validate__`.

When `skip_validate=true`, `run_validate_entry_point` sets `validate=false` in `ExecutionFlags`:

```rust
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [3](#0-2) 

`StatefulValidator::perform_validations` then returns `Ok(())` after `perform_pre_validation_stage` without ever calling `__validate__`:

```rust
if !tx.execution_flags.validate {
    return Ok(());
}
``` [4](#0-3) 

`perform_pre_validation_stage` only checks nonce, fee bounds, balance, and proof facts — it does not verify the signature: [5](#0-4) 

The `validate_by_mempool` call that precedes `skip_stateful_validations` also does not check the signature. The `ValidationArgs` struct contains only address, nonce, tx_hash, tip, and gas price — no signature field: [6](#0-5) 

And `Mempool::validate_tx` only checks for duplicate tx hash, nonce ordering, and fee escalation: [7](#0-6) 

---

### Impact Explanation

An unprivileged attacker can submit an invoke transaction with an **invalid signature** (e.g., all-zero bytes) that is admitted to the mempool without any cryptographic signature verification. This breaks the invariant that every transaction admitted to the mempool must have a valid signature verified by the account's `__validate__` entry point.

The admitted invalid transaction will fail at execution time when the batcher calls `__validate__`, but it was already accepted by the gateway and occupies mempool space. This enables:

- Mempool pollution with signature-invalid transactions
- Batcher resource waste on transactions that will always fail validation
- Repeated admission across blocks until the transaction's TTL expires

This matches: **High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

---

### Likelihood Explanation

Any unprivileged user can trigger this:

1. Create a new account address `A` (not deployed, nonce=0 on-chain)
2. Submit a valid deploy_account transaction for `A` → admitted to mempool
3. `account_tx_in_pool_or_recent_block(A)` now returns `true`
4. Submit an invoke transaction for `A` with nonce=1 and an invalid signature

No special privileges are required. The only cost is the fee for the deploy_account transaction, which is a normal user operation. The attack is deterministic and reproducible.

---

### Recommendation

Replace the `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a **deploy_account** transaction exists for the account. The mempool should expose a dedicated `has_pending_deploy_account_tx(address)` query, or the existing check should filter by transaction type before returning `true`. The current check conflates "account is known to the mempool" with "account has a pending deploy_account transaction," which are not equivalent.

---

### Proof of Concept

```
1. Account A: not deployed, nonce=0 on-chain.

2. Submit deploy_account tx for A → mempool admits it.
   account_tx_in_pool_or_recent_block(A) == true.

3. Submit invoke tx for A:
     nonce = 1
     signature = [0x0, 0x0]  // invalid

4. Gateway stateful validation:
   - validate_nonce: account_nonce=0, tx_nonce=1, gap=1 ≤ 200 → OK
   - validate_by_mempool: checks hash/nonce/fee only → OK
   - skip_stateful_validations:
       tx.nonce() == Nonce(1)          → true
       account_nonce == Nonce(0)       → true
       account_tx_in_pool_or_recent_block(A) → true (deploy_account is in pool)
       returns true (skip __validate__)

5. run_validate_entry_point:
   ExecutionFlags { validate: false, ... }
   StatefulValidator::perform_validations returns Ok(()) without calling __validate__

6. Invalid-signature invoke tx is admitted to the mempool. ✓

7. Batcher executes deploy_account → A is deployed.
   Batcher executes invoke (nonce=1) → __validate__ runs → signature invalid → tx rejected.
   Invalid tx occupied mempool slot for its full TTL.
``` [8](#0-7) [2](#0-1) [9](#0-8)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
