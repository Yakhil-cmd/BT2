### Title
Hardcoded NEAR-to-EVM Gas Conversion Ratio in XCC Precompile Causes Permanent Fee-Accounting Insolvency When NEAR Gas Costs Change - (File: engine-precompiles/src/xcc.rs)

### Summary

The `CrossContractCall` precompile uses a hardcoded constant `CROSS_CONTRACT_CALL_NEAR_GAS = 175_000_000` to convert NEAR gas units into EVM gas charges. This ratio was derived from a one-time empirical measurement and is compiled into the contract with no mechanism to update it. NEAR Protocol's gas pricing has changed historically and is expected to continue evolving. When the actual NEAR-to-EVM gas ratio diverges from the hardcoded value, the XCC precompile systematically undercharges or overcharges every user who calls it, leading to either insolvency of the engine's economic model or permanent economic unviability of the XCC feature.

### Finding Description

In `engine-precompiles/src/xcc.rs`, the `costs` module defines:

```rust
pub const CROSS_CONTRACT_CALL_NEAR_GAS: u64 = 175_000_000;
``` [1](#0-0) 

The comment explicitly acknowledges this value was derived from a single gas report (`https://hackmd.io/@birchmd/Sy4piXQ29`) at a fixed point in time. It is used in the precompile's `run_with_handle` to charge EVM gas to the caller:

```rust
cost += EthGas::new(promise.attached_gas.as_u64() / costs::CROSS_CONTRACT_CALL_NEAR_GAS);
``` [2](#0-1) 

This single division is the only place where the NEAR gas cost of the XCC promise is translated into an EVM gas charge. There is no on-chain storage slot, no admin setter, and no upgrade path for this ratio short of a full contract redeployment.

The same constant is referenced in the test suite to validate gas accounting: [3](#0-2) 

The `STORAGE_AMOUNT` constant (2 NEAR, hardcoded in the same `state` module) exhibits the same pattern and is used to charge users for router account creation: [4](#0-3) [5](#0-4) 

The XCC router contract also hardcodes a matching `REFUND_AMOUNT` with a comment explicitly noting it must stay in sync: [6](#0-5) 

### Impact Explanation

**Scenario A — NEAR gas becomes more expensive (ratio too high):**
Every XCC call charges the user fewer EVM gas units than the NEAR gas actually consumed. The relayer/engine subsidizes the difference from its own NEAR balance. Because any EVM user can call the XCC precompile with arbitrarily large `attached_gas`, a user can repeatedly make XCC calls that consume far more NEAR gas than they pay for in EVM fees. Over time this drains the engine's NEAR economic reserves — **insolvency**.

**Scenario B — NEAR gas becomes cheaper (ratio too low):**
Every XCC call charges the user more EVM gas than the actual NEAR cost. XCC becomes economically unviable; users stop using it. The XCC feature is effectively frozen — **temporary freezing of functionality**.

Both scenarios are in the allowed impact scope. Scenario A is the critical path.

### Likelihood Explanation

NEAR Protocol has changed gas costs through protocol upgrades in the past. The ratio `0.175 T(NEAR_gas) / k(EVM_gas)` is not a protocol invariant — it is an empirical measurement that will drift as NEAR's runtime evolves. Because the constant is compiled into the WASM binary with no setter, any drift requires a full engine upgrade to correct, during which the mispricing is actively exploitable by any EVM user who can call the XCC precompile address `0x516cded1d16af10cad47d6d49128e2eb7d27b372`. [7](#0-6) 

### Recommendation

1. Store `CROSS_CONTRACT_CALL_NEAR_GAS` and `STORAGE_AMOUNT` in on-chain engine state (alongside existing engine state fields) rather than as compiled constants.
2. Add an owner-only setter (analogous to `factory_set_wnear_address`) that allows updating these values without a full contract upgrade.
3. Emit an event whenever the ratio is updated so off-chain tooling can track changes.

### Proof of Concept

1. At deployment, `CROSS_CONTRACT_CALL_NEAR_GAS = 175_000_000` is baked into the WASM.
2. NEAR Protocol upgrades change the actual gas cost such that the true ratio is now `350_000_000` (2× more expensive).
3. An EVM user constructs an XCC call with `attached_gas = 100_000_000_000_000` (100 TGas).
4. The precompile charges: `100_000_000_000_000 / 175_000_000 = 571_428` EVM gas — half what it should charge.
5. The engine spends 100 TGas of NEAR but only collects EVM fees for 50 TGas worth.
6. Repeated across many users, the engine's NEAR balance is drained at twice the expected rate, with no on-chain mechanism to correct the ratio until a full redeployment.
