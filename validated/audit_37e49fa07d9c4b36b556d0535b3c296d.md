### Title
ZK Transaction Balance Check Omits Blob Gas Fee, Triggering `unreachable!()` Panic in `precharge_fee` - (File: `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

The ZK transaction validation path checks the sender's balance using `transaction.required_balance()`, which computes only `max_fee_per_gas * gas_limit + value`. It does not include the blob gas fee (`fee_for_blob_gas`). However, the actual fee deducted in `precharge_fee` is `gas_price * gas_limit + fee_for_blob_gas`. When a sender's balance is sufficient to pass the balance check but insufficient to cover the blob gas component of the actual fee, `precharge_fee` hits an `unreachable!()` branch that asserts the balance was pre-verified — causing a panic that halts block processing.

---

### Finding Description

**Step 1 — Balance check (ZK path) omits blob gas.**

In `basic_bootloader/src/bootloader/transaction/abi_encoded/mod.rs`, `required_balance()` is:

```rust
pub fn required_balance(&self) -> Option<U256> {
    let fee_amount = self
        .max_fee_per_gas
        .read()
        .checked_mul(U256::from(self.gas_limit.read()))?;
    self.value.read().checked_add(U256::from(fee_amount))
}
```

This returns `max_fee_per_gas * gas_limit + value` — no blob gas term.

In `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`, the balance check calls this function:

```rust
let Some(total_required_balance) = transaction.required_balance() else { ... };
if total_required_balance > originator_account_data.nominal_token_balance.0 {
    return Err(TxError::Validation(InvalidTransaction::LackOfFundForMaxFee { ... }));
}
```

**Step 2 — `fee_to_prepay` includes blob gas.**

Immediately after the balance check, the same function computes the actual fee to charge:

```rust
let num_blobs = system.metadata.num_blobs();
let blob_gas_used = num_blobs as u64 * GAS_PER_BLOB;
let fee_for_blob_gas = if blob_gas_used > 0 {
    system.get_blob_base_fee_per_gas().checked_mul(U256::from(blob_gas_used))...
} else { U256::ZERO };
let fee_to_prepay = gas_fee_amount
    .checked_add(fee_for_blob_gas)
    .ok_or(internal_error!("gfa+ffbg"))?;
```

`fee_to_prepay = gas_price * gas_limit + fee_for_blob_gas` is stored in the transaction context.

**Step 3 — `precharge_fee` asserts balance was pre-verified.**

In `basic_bootloader/src/bootloader/transaction_flow/zk/mod.rs`, `precharge_fee` deducts `context.fee_to_prepay` from the sender's balance and treats any `InsufficientBalance` error as impossible:

```rust
.map_err(|e| match e {
    SubsystemError::LeafUsage(interface_error) => {
        unreachable!(
            "balance should be pre-verified, but received error {:?}",
            interface_error
        );
    }
    ...
```

**Step 4 — Contrast with Ethereum path (correct).**

In `basic_bootloader/src/bootloader/transaction_flow/ethereum/validation_impl.rs`, the Ethereum balance check correctly adds blob gas:

```rust
let mut total_required_balance = tx_value.checked_add(worst_case_fee_amount)...;
total_required_balance = total_required_balance.checked_add(fee_for_blob_gas)...;
```

The ZK path is missing this addition.

---

### Impact Explanation

A sender whose balance satisfies `max_fee_per_gas * gas_limit + value ≤ balance < gas_price * gas_limit + fee_for_blob_gas` will:

1. Pass the ZK balance check (blob gas not included).
2. Reach `precharge_fee`, which attempts to deduct `gas_price * gas_limit + fee_for_blob_gas`.
3. Fail with `InsufficientBalance`.
4. Hit `unreachable!()`, causing a Rust panic.

In the forward (sequencer) execution mode this crashes the sequencer process. In the proving (RISC-V) mode the panic terminates the RISC-V binary, making the block unprovable. Either outcome is a **state-transition denial-of-service**: the block containing the offending transaction cannot be finalized.

The simplest triggering condition (no priority fee, no value):
- `max_fee_per_gas = gas_price`, `value = 0`
- `balance = gas_price * gas_limit` (exactly covers gas, passes balance check)
- Transaction carries ≥1 blob → `fee_for_blob_gas > 0` → fee deduction fails → panic.

---

### Likelihood Explanation

Any unprivileged user can submit a ZK blob transaction (EIP-4844 type) with a balance calibrated to exactly `gas_price * gas_limit`. The ZK validation path accepts it; `precharge_fee` panics. No privileged access, no oracle manipulation, and no governance action is required. The attacker only needs to know the current blob base fee and set their balance accordingly — a straightforward on-chain measurement.

---

### Recommendation

In `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`, add the blob gas fee to the balance check, mirroring the Ethereum path:

```rust
// After computing fee_for_blob_gas (lines 469-488):
let Some(total_required_balance) = transaction.required_balance() else {
    return Err(TxError::Validation(InvalidTransaction::OverflowPaymentInTransaction));
};
// Add blob gas to the required balance check
let total_required_balance = total_required_balance
    .checked_add(fee_for_blob_gas)
    .ok_or(internal_error!("required_balance + blob gas"))?;
if total_required_balance > originator_account_data.nominal_token_balance.0 {
    return Err(TxError::Validation(InvalidTransaction::LackOfFundForMaxFee { ... }));
}
```

Alternatively, update `required_balance()` in `abi_encoded/mod.rs` to accept a blob gas parameter, or move the balance check to after `fee_for_blob_gas` is computed (as the Ethereum path does).

---

### Proof of Concept

1. Deploy on a ZKsync OS chain with EIP-4844 enabled.
2. Fund account `Alice` with exactly `gas_price * gas_limit` tokens (e.g., `gas_price = 1000`, `gas_limit = 21000` → balance = `21_000_000`).
3. Submit a ZK EIP-4844 transaction from Alice: `max_fee_per_gas = 1000`, `gas_limit = 21000`, `value = 0`, 1 blob attached.
4. `required_balance()` returns `1000 * 21000 + 0 = 21_000_000` ≤ Alice's balance → balance check passes.
5. `fee_for_blob_gas = blob_base_fee_per_gas * GAS_PER_BLOB > 0`.
6. `fee_to_prepay = 21_000_000 + fee_for_blob_gas > 21_000_000 = Alice's balance`.
7. `precharge_fee` attempts to deduct `fee_to_prepay`, gets `InsufficientBalance`, hits `unreachable!()` → panic → block processing halts. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

        system: &mut System<S>,
        transaction: &Transaction<<S as SystemTypes>::Allocator>,
        context: &mut Self::TransactionContext,
        _tracer: &mut impl Tracer<S>,
    ) -> Result<(), TxError> {
        let from = transaction.from();
        let fee = context.fee_to_prepay;

        system_log!(
            system,
            "Will precharge {:?} native tokens for transaction\n",
            &fee
        );

        // ARCHITECTURE NOTE: Fee payment is split into two phases:
        // 1. Deduct full fee from sender at transaction start (here)
        // 2. Transfer actual payment to operator after execution (in refund_transaction_and_pay_operator)
        // This ensures sender has sufficient funds before execution begins
        context
            .intrinsic_resources
            .with_infinite_ergs(|resources| {
                system.io.update_account_nominal_token_balance(
                    ExecutionEnvironmentType::NoEE,
                    resources,
                    &from,
                    &fee,
                    true,
                    Config::SIMULATION,
                )
            })
            .map_err(|e| match e {
                SubsystemError::LeafUsage(interface_error) => {
                    unreachable!(
                        "balance should be pre-verified, but received error {:?}",
                        interface_error
                    );
                }
                SubsystemError::LeafDefect(internal_error) => internal_error.into(),
                // shouldn't be reachable as we are using infinite resources
                SubsystemError::LeafRuntime(runtime_error) => match runtime_error {
                    RuntimeError::FatalRuntimeError(_) => {
                        TxError::oon_as_validation(out_of_native_resources!().into())
                    }
                    RuntimeError::OutOfErgs(_) => {
                        TxError::Validation(InvalidTransaction::OutOfGasDuringValidation)
                    }
                },
                SubsystemError::Cascaded(cascaded_error) => match cascaded_error {},
            })?;
        Ok(())
