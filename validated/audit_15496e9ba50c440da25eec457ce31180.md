### Title
`DACommitmentScheme::PubdataKeccak256` Accepted by Oracle Parser but Unimplemented in Generator Causes Proving Panic — (`basic_bootloader/src/bootloader/block_flow/zk/post_tx_op/da_commitment_generator/mod.rs`)

---

### Summary

The `DACommitmentScheme` enum declares `PubdataKeccak256` (ID=2) and `None` (ID=0) as valid, oracle-parseable variants. Both are successfully deserialized from prover oracle input without error. However, `da_commitment_generator_from_scheme()` does not implement either variant and returns `Err(internal_error!("Unsupported DA commitment scheme"))` for them. Both proving paths — singleblock and multiblock batch — call `.unwrap()` on this result unconditionally, causing an abort/panic in the RISC-V proving environment. Any block whose oracle input carries DA scheme ID 0 or 2 cannot be proven.

---

### Finding Description

**Step 1 — Scheme is declared valid and parsed without error.**

`DACommitmentScheme` in `zk_ee/src/common_structs/da_commitment_scheme.rs` defines five variants:

```
None = 0, EmptyNoDA = 1, PubdataKeccak256 = 2,
BlobsAndPubdataKeccak256 = 3, BlobsZKsyncOS = 4
```

`TryFrom<u8>` maps ID 0 → `None` and ID 2 → `PubdataKeccak256` without returning an error. `DACommitmentScheme::try_from_oracle()` calls this conversion and propagates the result into `FullIO::init_from_oracle()` (proving mode only, `PROOF_ENV = true`), storing it in `io.da_commitment_scheme`. [1](#0-0) 

**Step 2 — Generator dispatch does not handle `PubdataKeccak256` or `None`.**

`da_commitment_generator_from_scheme()` matches only three variants and falls through to an error for everything else:

```rust
_ => Err(internal_error!("Unsupported DA commitment scheme")),
``` [2](#0-1) 

**Step 3 — Both proving paths `.unwrap()` the result unconditionally.**

Singleblock batch:
```rust
let mut da_commitment_generator =
    da_commitment_generator_from_scheme(io.da_commitment_scheme.unwrap(), A::default())
        .unwrap();   // ← panics for PubdataKeccak256 / None
``` [3](#0-2) 

Multiblock batch:
```rust
batch_data.da_commitment_generator = Some(
    da_commitment_generator_from_scheme(da_commitment_scheme, A::default()).unwrap(),
);  // ← panics for PubdataKeccak256 / None
``` [4](#0-3) 

**Step 4 — Documentation confirms the mismatch.**

The official documentation explicitly marks `PubdataKeccak256` as "Currently not supported" while still listing it as a named scheme with a use-case description (third-party DA layers: Celestia, Avail, etc.), creating the false impression that it is a valid operational choice. [5](#0-4) 

---

### Impact Explanation

When the oracle/prover input carries DA commitment scheme ID 2 (`PubdataKeccak256`) or ID 0 (`None`), the RISC-V proving binary aborts at the `.unwrap()` call during the post-transaction block-finalization phase. No valid proof output is produced. This is a **valid-execution unprovability** bug: the forward execution may succeed, but the proving run cannot complete, making it impossible to settle the corresponding batch on L1. The chain halts for any deployment that configures either of these DA schemes.

---

### Likelihood Explanation

`PubdataKeccak256` is the scheme intended for third-party DA layers (Celestia, Avail). As ZKsync OS is designed to support multiple DA backends, operators deploying on such layers would naturally select this scheme. The scheme ID is defined in the L1 contracts (`L2DACommitmentScheme`) and is a valid wire value. An operator or prover who provides ID=2 via the oracle — either intentionally (misconfiguration) or as part of a future integration — will trigger the panic. The entry point is the prover/forward execution oracle input, which is explicitly listed in the Immunefi scope as an attacker-reachable surface.

---

### Recommendation

1. **Reject unsupported schemes at oracle parse time.** In `DACommitmentScheme::try_from_oracle()`, return `Err(internal_error!(...))` immediately for `None` and `PubdataKeccak256` rather than allowing them to propagate into the generator dispatch.
2. **Replace `.unwrap()` with explicit error propagation** in both `post_tx_op_proving_singleblock_batch.rs` and `post_tx_op_proving_multiblock_batch.rs` so that an unsupported scheme produces a recoverable `BootloaderSubsystemError` rather than a panic.
3. **Implement `PubdataKeccak256`** if third-party DA layer support is planned, or remove the variant from `TryFrom<u8>` until it is implemented.

---

### Proof of Concept

1. Construct a prover oracle that returns `2u8` for the `DA_COMMITMENT_SCHEME_QUERY_ID` query.
2. Run the ZKsync OS singleblock or multiblock batch proving binary against any block.
3. `FullIO::init_from_oracle()` stores `DACommitmentScheme::PubdataKeccak256` in `io.da_commitment_scheme` without error.
4. At block finalization, `da_commitment_generator_from_scheme(PubdataKeccak256, ...)` returns `Err(internal_error!("Unsupported DA commitment scheme"))`.
5. `.unwrap()` panics; the RISC-V binary aborts; no proof output is produced.
6. The batch cannot be settled on L1 — proving is permanently impossible for this block. [6](#0-5) [7](#0-6)

impl TryFrom<u8> for DACommitmentScheme {
    type Error = ();

    fn try_from(value: u8) -> Result<Self, Self::Error> {
        match value {
            0 => Ok(DACommitmentScheme::None),
            1 => Ok(DACommitmentScheme::EmptyNoDA),
            2 => Ok(DACommitmentScheme::PubdataKeccak256),
            3 => Ok(DACommitmentScheme::BlobsAndPubdataKeccak256),
            4 => Ok(DACommitmentScheme::BlobsZKsyncOS),
            _ => Err(()),
        }
    }
}

impl DACommitmentScheme {
    pub fn try_from_oracle<O: IOOracle>(oracle: &mut O) -> Result<Self, InternalError> {
        let da_commitment_scheme_id_raw: u8 =
            oracle.query_with_empty_input(DA_COMMITMENT_SCHEME_QUERY_ID)?;
        DACommitmentScheme::try_from(da_commitment_scheme_id_raw)
            .map_err(|_| internal_error!("Invalid DA commitment scheme ID"))
    }
```
