### Title
Malicious Coordinator Can Present Inconsistent `(msg_hash, ε)` Parameters to Different Signers, Enabling Private Key Extraction — (`src/ecdsa/robust_ecdsa/sign.rs`)

---

### Summary

The robust ECDSA `sign()` function accepts `msg_hash` and a `RerandomizedPresignOutput` (which embeds the rerandomization tweak `ε`) as caller-supplied parameters with no cross-participant agreement enforcement. A malicious coordinator — an explicitly modeled attacker role — can present different `(msg_hash, ε)` pairs to different subsets of honest signers across one or more signing sessions that share the same underlying presignature. The resulting signatures have multiplicatively related nonces, enabling full private key extraction via standard ECDSA nonce-reuse algebra. The library's own security documentation names this attack and mandates enforcement, but the production signing code provides no mechanism to satisfy that mandate.

---

### Finding Description

**Root cause — no parameter-agreement enforcement in `sign()`:**

`src/ecdsa/robust_ecdsa/sign.rs` `sign()` validates only structural constraints (participant count equals `2t+1`, coordinator membership, `msg_hash ≠ 0`). It performs no check that all participants in a session received the same `msg_hash` or the same rerandomization tweak `ε` embedded in their `RerandomizedPresignOutput`. [1](#0-0) 

Each participant independently calls `sign()` with their own `presignature: RerandomizedPresignOutput` and their own `msg_hash: Scalar`. Both values are fully attacker-controlled when the coordinator is malicious: the coordinator orchestrates which `ε` each participant uses during rerandomization and which `msg_hash` each participant is told to sign.

**Security documentation explicitly names the missing constraint:**

The library's own specification states:

> "Ensure all participants agree on (h, ε) and the signing set. The coordinator must not be able to present different message hashes, tweaks, or participant lists to different signers." [2](#0-1) 

And further:

> "If different subsets of size at least 2t+1 sign different (h, ε) values using shares derived from the same presignature, the resulting signatures use multiplicatively related nonces and the secret key can be recovered using standard ECDSA nonce-reuse attacks." [3](#0-2) 

**Exploit path:**

1. Coordinator runs presigning with `N = 2t+1` participants, producing a shared `PresignOutput` (nonce `R`, shares `α_i, β_i, c_i, e_i`).
2. Coordinator initiates **two signing sessions** using the same presignature:
   - Session A: tells all `2t+1` participants to rerandomize with tweak `ε_A` and sign `h_A`.
   - Session B: tells all `2t+1` participants to rerandomize with tweak `ε_B ≠ ε_A` and sign `h_B ≠ h_A`.
3. Each participant calls `sign()` with their locally rerandomized presignature and the coordinator-supplied `msg_hash`. No participant can detect the inconsistency because `sign()` enforces no cross-participant agreement.
4. The coordinator collects signature shares from both sessions and aggregates two valid ECDSA signatures `(R_A, s_A)` and `(R_B, s_B)` where `R_A` and `R_B` are multiplicatively related (derived from the same nonce `k`).
5. Standard ECDSA nonce-reuse equations yield the master secret key `x`.

The novel linearization-based variant documented in the spec reduces the required sessions to two and the required participants to `2t+2`, making the attack cheaper than the classical form. [4](#0-3) 

**Analog to the SeaDrop finding:** In SeaDrop, the mapping stores only a `bool` for allowed signers, placing no constraints on the parameters (`feeBps`, `feeRecipient`) those signers may use. Here, the library stores no constraints on what `(msg_hash, ε)` a coordinator may present to participants — the coordinator is "allowed" by virtue of being a participant, and the signing API imposes no bound on the parameters it can supply to different honest parties.

---

### Impact Explanation

**Critical — Extraction of the aggregate private signing key.**

A malicious coordinator can recover the full threshold secret key `x` from two signing sessions that reuse the same presignature with different `(msg_hash, ε)` inputs. This satisfies the allowed impact: *"Extraction, reconstruction, or disclosure of private signing shares, aggregate secret material, presign secrets, nonce material, or confidential derived secrets."*

All honest participants contribute their shares in good faith; none can individually detect the attack because `sign()` provides no cross-participant consistency check.

---

### Likelihood Explanation

The coordinator is an explicitly modeled participant role in the robust ECDSA protocol. Any party that controls the coordinator role — including a compromised or malicious MPC node designated as coordinator — can execute this attack with two signing sessions and standard algebraic computation. No external assumptions, leaked keys, or cryptographic breaks are required. The attack is described in the library's own documentation as a known, concrete threat.

---

### Recommendation

Enforce cross-participant agreement on `(msg_hash, ε, participants)` at the protocol level before participants contribute their signature shares. Concretely:

1. **Commit-then-reveal for signing parameters:** Before the signing round, have each participant broadcast a hash commitment to `(msg_hash, ε, participant_set)`. After all commitments are collected, reveal and verify that all committed values are identical. Abort if any mismatch is detected.
2. **Bind `ε` to the presignature identifier:** Derive `ε` deterministically from a shared, authenticated presignature identifier and the agreed `msg_hash` (e.g., via a PRF keyed on a value established during presigning), so no participant can be given a different `ε` without detection.
3. **Never reuse a presignature:** Enforce single-use at the library level by consuming (zeroing) the `PresignOutput` on first use, preventing the coordinator from initiating a second session with the same nonce material.

### Proof of Concept

```
Setup: t=1, N=3 participants P1, P2, P3. Coordinator = P1.

Step 1 — Presigning:
  All three run presign(), producing shared nonce k and shares (α_i, β_i, c_i, e_i).
  R = k·G is fixed.

Step 2 — Session A (ε_A, h_A):
  Coordinator tells P1, P2, P3: rerandomize with ε_A, sign h_A.
  Each calls sign(..., presignature_rerandomized_with_ε_A, h_A).
  sign() passes all checks (|participants|=3=2·1+1, h_A≠0).
  Coordinator aggregates: s_A = Σ λ_i·s_i,A → signature (R_A, s_A).

Step 3 — Session B (ε_B ≠ ε_A, h_B ≠ h_A), same PresignOutput:
  Coordinator tells P1, P2, P3: rerandomize with ε_B, sign h_B.
  Each calls sign(..., presignature_rerandomized_with_ε_B, h_B).
  sign() passes all checks identically.
  Coordinator aggregates: s_B = Σ λ_i·s_i,B → signature (R_B, s_B).

Step 4 — Key extraction:
  R_A and R_B are derived from the same k (with known ε_A, ε_B offsets).
  Using (R_A, s_A, h_A) and (R_B, s_B, h_B) with known nonce relationship:
    x = (s_A·h_B - s_B·h_A) / (s_B·Rx_A - s_A·Rx_B)  [standard nonce-reuse formula]
  Full private key x recovered. All honest participants' key shares are compromised.

pub fn sign(
    participants: &[Participant],
    coordinator: Participant,
    max_malicious: impl Into<MaxMalicious>,
    me: Participant,
    public_key: AffinePoint,
    presignature: RerandomizedPresignOutput,
    msg_hash: Scalar,
) -> Result<impl Protocol<Output = SignatureOption>, InitializationError> {
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }

    let participants =
        ParticipantList::new(participants).ok_or(InitializationError::DuplicateParticipants)?;

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }

    // ensure number of participants during the signing phase is >= 2 * max_malicious + 1
    let robust_ecdsa_threshold = max_malicious
        .into()
        .value()
        .checked_mul(2)
        .and_then(|v| v.checked_add(1))
        .ok_or_else(|| {
            InitializationError::BadParameters(
                "2*threshold+1 must be less than usize::MAX".to_string(),
            )
        })?;
    if robust_ecdsa_threshold > participants.len() {
        return Err(InitializationError::BadParameters(
            "2*max_malicious+1 must be less than or equals to participant count".to_string(),
        ));
    }

    // The next two conditions prevent split-view attacks
    // documented in docs/ecdsa/robust_ecdsa/signing.md
    if participants.len() != robust_ecdsa_threshold {
        return Err(InitializationError::BadParameters(
            "the number of participants during signing must be exactly 2*max_malicious+1 to avoid split view attacks".to_string(),
        ));
    }
    if bool::from(msg_hash.is_zero()) {
        return Err(InitializationError::BadParameters(
            "msg_hash cannot be 0 to avoid potential split view attacks".to_string(),
