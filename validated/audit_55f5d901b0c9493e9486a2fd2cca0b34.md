### Title
Missing Verification of Participant Contributions in CKD Coordinator Allows Malicious Participant to Corrupt CKD Output - (File: src/confidential_key_derivation/protocol.rs)

### Summary
The `do_ckd_coordinator()` function in the Confidential Key Derivation protocol blindly aggregates `(λ_i·Y_i, λ_i·C_i)` contributions from all participants without any proof of correctness. A single malicious MPC participant can substitute arbitrary group elements, causing the coordinator to output a corrupted `(Y, C)` pair that is cryptographically unusable by the requesting application.

### Finding Description
The external report's vulnerability class is **missing context/domain validation**: `executeProposal()` processes a VAA without verifying the chain ID or receiver address, enabling cross-chain replay. The direct analog here is that `do_ckd_coordinator()` processes participant contributions without verifying that each contribution was computed for the correct `app_id`, `app_pk`, or even that the values are consistent with the participant's public key share.

In `do_ckd_coordinator()` (lines 35–57 of `src/confidential_key_derivation/protocol.rs`):

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
Ok(Some(ckd_output))
```

The coordinator:
1. Receives `(λ_i·Y_i, λ_i·C_i)` from every participant.
2. Sums them unconditionally — the sender identity is discarded (`_`).
3. Emits the aggregate as the final CKD output.

There is no check that:
- `Y_i` is a non-identity element of G1.
- `C_i` is a non-identity element of G1.
- The pair `(Y_i, C_i)` is consistent with the participant's public key share and the agreed `app_id`/`app_pk` (i.e., no ZKP that `C_i = x_i·H(pk, app_id) + y_i·A` and `Y_i = y_i·G`).

The correct per-participant computation is performed in `compute_signature_share()` (lines 148–181), but nothing prevents a malicious participant from sending values that were computed for a different `app_id`, for a different `app_pk`, or that are entirely arbitrary. [1](#0-0) [2](#0-1) 

### Impact Explanation
The coordinator is the sole entity that assembles and outputs the CKD result `(Y, C)`. When it emits a corrupted aggregate, the honest output of the protocol is an **unusable cryptographic object**: the application computes `sig = C − a·Y`, which will not satisfy the BLS pairing check `e(sig, G₂) = e(H(pk, app_id), pk)`. The application cannot derive its confidential key `s`. This matches the allowed High impact: *"Corruption of … CKD outputs so honest parties accept … unusable cryptographic outputs."* [3](#0-2) 

### Likelihood Explanation
Any single MPC node that is a non-coordinator participant can mount this attack. The node participates in the protocol normally up to the contribution round, then sends arbitrary `(Y_i', C_i')` instead of the correctly computed values. No cryptographic capability beyond participation in the protocol is required. The communication layer (`recv_from_others`) authenticates the sender identity but does not validate the content of the message. [4](#0-3) 

### Recommendation
Require each participant to accompany their `(λ_i·Y_i, λ_i·C_i)` contribution with a non-interactive zero-knowledge proof (e.g., a Sigma protocol / Chaum–Pedersen proof) demonstrating that:
- `Y_i = y_i · G₁` for some scalar `y_i`, and
- `C_i = x_i · H(pk, app_id) + y_i · A` using the same `y_i` and the participant's committed public key share `x_i · G₂`.

The coordinator must verify all proofs before aggregating. This is the standard approach used in verifiable ElGamal encryption schemes and prevents any participant from injecting malformed contributions. [3](#0-2) 

### Proof of Concept
1. All `n` participants begin the CKD protocol for `(app_id, app_pk = A)`.
2. Honest participants compute and send `(λ_i·Y_i, λ_i·C_i)` correctly.
3. Malicious participant `P_m` instead sends `(λ_m·G₁, λ_m·G₁)` — two copies of the generator — to the coordinator.
4. The coordinator sums all contributions: `Y' = Σ_{i≠m} λ_i·Y_i + λ_m·G₁`, `C' = Σ_{i≠m} λ_i·C_i + λ_m·G₁`.
5. The coordinator emits `CKDOutput::new(Y', C')` as the protocol result.
6. The application computes `sig' = C' − a·Y'`, which equals `msk·H(pk, app_id) + λ_m·G₁ − a·λ_m·G₁` — a value that does not satisfy the BLS verification equation.
7. The application's BLS check fails; it cannot derive `s` and the CKD session is rendered unusable. [5](#0-4) [6](#0-5)

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
