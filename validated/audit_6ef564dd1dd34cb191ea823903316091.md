### Title
Missing Identity-Point Validation for `app_pk` Allows Confidential Derived Key Disclosure - (File: `src/confidential_key_derivation/protocol.rs`)

### Summary
The `ckd()` function in `src/confidential_key_derivation/protocol.rs` accepts an `app_pk` (application public key) parameter without validating that it is not the group identity element. If `app_pk` is the identity point of G1, the ElGamal masking term vanishes, and the coordinator's CKD output directly exposes the confidential derived key `msk · H(pk, app_id)` in plaintext inside `big_c`, defeating the entire confidentiality guarantee of the protocol.

### Finding Description
The CKD protocol is designed as a threshold ElGamal encryption scheme. Each participant computes:

```
big_s  = private_share · H(pk, app_id)
big_c  = big_s + app_pk · y          // ElGamal ciphertext
big_y  = y · G                        // ephemeral public key
```

After aggregation by the coordinator, the output is `(big_y, big_c)` where:

```
big_c = msk · H(pk, app_id) + app_pk · y
```

The client recovers the confidential key via `unmask(app_sk)`:

```
big_c - app_sk · big_y = msk · H(pk, app_id)
```

The masking term `app_pk · y` is the sole mechanism that hides `msk · H(pk, app_id)` from the coordinator and any observer of the protocol output.

If a caller passes `app_pk = G1::identity()` (the zero point), then `app_pk · y = identity`, so:

```
big_c = msk · H(pk, app_id) + identity = msk · H(pk, app_id)
```

The coordinator's output `big_c` is now the confidential derived key itself, in the clear. No `app_sk` is needed to recover it.

The `ckd()` function performs several input validations (participant count, duplicates, self-membership, coordinator membership) but performs **no validation** that `app_pk` is not the identity point: [1](#0-0) 

The masking computation in `compute_signature_share` unconditionally multiplies `app_pk` by the ephemeral scalar `y`: [2](#0-1) 

Specifically, line 174 `let big_c = big_s + app_pk * y.0;` produces `big_c = big_s` when `app_pk` is the identity, leaving the secret unmasked. [3](#0-2) 

The `CKDOutput::unmask` function then trivially returns the correct confidential key when called with `app_sk = 0` (the only scalar for which `app_sk · G = identity`): [4](#0-3) 

This is structurally identical to the reported `rfqOrderSigner == address(0)` pattern: a zero/identity value in a security-critical field causes a verification or masking operation to trivially pass or collapse, exposing protected material.

### Impact Explanation
The confidential derived key `msk · H(pk, app_id)` is directly readable from the coordinator's CKD output `big_c`. Any party that receives or observes the `CKDOutput` — including the coordinator itself, which is a participant — can extract the confidential derived secret without possessing `app_sk`. This constitutes **disclosure of a confidential derived secret**, matching the Critical impact tier: *"Extraction, reconstruction, or disclosure of... confidential derived secrets."*

### Likelihood Explanation
The `app_pk` parameter is caller-supplied with no type-level or runtime constraint preventing the identity point. A malicious application client, or any party that controls the `app_pk` argument passed to `ckd()`, can trivially trigger this by passing `G1Projective::identity()`. The identity point is a valid, representable value of type `ElementG1 = blstrs::G1Projective`. No cryptographic break, key leakage, or trusted-party compromise is required.

### Recommendation
Add an explicit check in `ckd()` that rejects `app_pk` equal to the group identity before the protocol begins:

```rust
if app_pk == ElementG1::identity() {
    return Err(InitializationError::BadParameters(
        "app_pk must not be the identity point".to_string(),
    ));
}
```

This mirrors the pattern already used in `dlogeq::prove_with_nonce` and `dlogeq::verify`, which both guard against an identity `generator1`: [5](#0-4) [6](#0-5) 

### Proof of Concept
1. Caller invokes `ckd(participants, coordinator, me, key_pair, app_id, G1Projective::identity(), rng)`.
2. `ckd()` passes all existing validation checks (lines 76–101) because none inspect `app_pk`.
3. Each participant's `compute_signature_share` computes `big_c = big_s + identity * y = big_s = private_share_i · H(pk, app_id)`.
4. The coordinator aggregates: `big_c = Σ (λ_i · private_share_i · H(pk, app_id)) = msk · H(pk, app_id)`.
5. The coordinator's `CKDOutput` contains `big_c = msk · H(pk, app_id)` — the confidential derived key — in plaintext.
6. Any observer reads `ckd_output.big_c()` directly to obtain the confidential derived key, with no knowledge of `app_sk` required.

pub fn ckd(
    participants: &[Participant],
    coordinator: Participant,
    me: Participant,
    key_pair: KeygenOutput,
    app_id: impl Into<AppId>,
    app_pk: PublicKey,
    rng: impl CryptoRngCore + Send + 'static,
) -> Result<impl Protocol<Output = CKDOutputOption>, InitializationError> {
    // not enough participants
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }

    // kick out duplicates
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}
