### Title
Missing `native_price` Validation in ZK Block Metadata Initialization Causes All Transactions to Fail Fatally - (File: `basic_bootloader/src/bootloader/block_flow/zk/metadata_op.rs`)

### Summary
`ZkMetadata::metadata_op` deserializes oracle-provided `BlockMetadataFromOracle` and only validates `block_gas_limit` and `individual_tx_gas_limit`. It does not validate that `native_price != 0`. When a zero `native_price` is provided by the oracle, the block metadata is accepted and the system enters an inconsistent initialized state. Every subsequent L2 transaction then hits a fatal `internal_error!` during validation, making the entire block unprocessable.

### Finding Description
In `basic_bootloader/src/bootloader/block_flow/zk/metadata_op.rs`, the `metadata_op` function reads `BlockMetadataFromOracle` from the oracle and performs only two range checks before accepting the metadata as valid block state: [1](#0-0) 

The `native_price` field â€” a critical divisor used in every L2 transaction's resource accounting â€” is accepted without any non-zero check. The `BlockMetadataFromOracle::from_iter` deserializer also performs no semantic validation: [2](#0-1) 

Later, during per-transaction validation in `validate_and_compute_fee_for_transaction`, the zero check is deferred to transaction time: [3](#0-2) 

This check returns `internal_error!(...)` converted into a `TxError`, which is a fatal internal error (not a graceful `InvalidTransaction` rejection). The block metadata is already committed to an inconsistent state at this point.

### Impact Explanation
A zero `native_price` in the oracle-provided `BlockMetadataFromOracle` causes every L2 transaction in the block to fail with a fatal internal error at validation time. The block is initialized successfully but is rendered entirely unprocessable â€” no transaction can be included. This is a state-transition bug: the system accepts an invalid block configuration and then fails on every transaction rather than rejecting the block at initialization.

### Likelihood Explanation
The oracle input in the ZK proving path is provided by the prover/sequencer, which is listed as an attacker-controlled entry point in the Immunefi scope ("prover/forward execution input", "oracle-data-influencing caller"). A malicious or buggy oracle providing `native_price = 0` triggers this path. The `BlockMetadataFromOracle` struct is deserialized directly from oracle bytes with no semantic validation at the initialization boundary.

### Recommendation
Add a `native_price != 0` check in `ZkMetadata::metadata_op` immediately after deserializing the oracle metadata, analogous to the existing gas limit checks:

```rust
if metadata.block_gas_limit() > MAX_BLOCK_GAS_LIMIT
    || metadata.individual_tx_gas_limit() > MAX_TX_GAS_LIMIT
{
    return Err(internal_error!("block or tx gas limit is too high"));
}
// Add:
if metadata.block_level.native_price.is_zero() {
    return Err(internal_error!("native price cannot be zero"));
}
```

This mirrors the pattern already applied to gas limits and catches the invalid state at block initialization rather than deferring it to per-transaction processing.

### Proof of Concept
1. Oracle provides `BlockMetadataFromOracle` with `native_price = U256::ZERO`.
2. `ZkMetadata::metadata_op` deserializes it via `oracle.query_with_empty_input(BLOCK_METADATA_QUERY_ID)`.
3. The only checks performed are `block_gas_limit > MAX_BLOCK_GAS_LIMIT` and `individual_tx_gas_limit > MAX_TX_GAS_LIMIT` â€” both pass for a zero `native_price`.
4. The metadata is stored as valid block state.
5. For every L2 transaction, `validate_and_compute_fee_for_transaction` calls `system.get_native_price()`, gets `U256::ZERO`, and hits `return Err(internal_error!("Native price cannot be 0").into())`.
6. All transactions fail with a fatal internal error; the block cannot be finalized. [4](#0-3) [5](#0-4)
