### Title
Missing Participant-Set Consistency Check Between Triple Generation and Presigning Causes Corrupted/Unusable Presignatures - (File: `src/ecdsa/ot_based_ecdsa/presign.rs`)

---

### Summary

The OT-based ECDSA `presign` function explicitly omits validating that the presigning participant set matches the participant set recorded in `TriplePub.participants`. Because Lagrange linearization of triple shares is computed relative to the presigning participant set rather than the triple-generation participant set, any mismatch between the two sets produces cryptographically incorrect linearized shares. The resulting presignature is unusable, and honest parties cannot complete signing.

---

### Finding Description

`TriplePub` records the participant set that generated the triple:

```rust
pub struct TriplePub {
    pub big_a: AffinePoint,
    pub big_b: AffinePoint,
    pub big_c: AffinePoint,
    /// The participants in generating this triple.
    pub participants: Vec<Participant>,
    /// The threshold which will be able to reconstruct it.
    pub threshold: ReconstructionLowerBound,
}
``` [1](#0-0) 

The `presign` entry-point checks that the threshold matches the triples, but explicitly skips checking that the presigning participant set matches `TriplePub.participants`:

```rust
// NOTE: We omit the check that the new participant set was present for
// the triple generation, because presumably they need to have been present
// in order to have shares.

// Also check that we have enough participants to reconstruct shares.
if args.threshold != args.triple0.1.threshold || args.threshold != args.triple1.1.threshold {
    return Err(InitializationError::BadParameters(
        "New threshold must match the threshold of both triples".to_string(),
    ));
}
``` [2](#0-1) 

Inside `do_presign`, the Lagrange coefficient is computed relative to the **presigning** participant list, not the triple-generation participant list:

```rust
let lambda_me = participants.lagrange::<Secp256>(me)?;

let k_prime_i = lambda_me * k_i;
let e_i: Scalar = lambda_me * e_i;

let a_prime_i = lambda_me * a_i;
let b_prime_i = lambda_me * b_i;
...
let x_prime_i = lambda_me * private_share;
``` [3](#0-2) 

Triple shares `k_i`, `e_i`, `a_i`, `b_i` are polynomial evaluations `f(i)` generated during triple generation. Correct Lagrange reconstruction of `f(0)` requires that the Lagrange coefficients be computed relative to the **same** participant set used during triple generation. If the presigning participant set differs from `TriplePub.participants`, then:

```
sum_i lambda_i(presign_set) * f(i)  ≠  f(0)
```

The consistency check that follows immediately detects this:

```rust
if big_e != (ProjectivePoint::GENERATOR * e).to_affine() {
    return Err(ProtocolError::AssertionFailed(
        "received incorrect shares of kd".to_string(),
    ));
}
``` [4](#0-3) 

And the second check reinforces this:

```rust
if (ProjectivePoint::GENERATOR * alpha != big_k + big_a)
    || (ProjectivePoint::GENERATOR * beta != big_x + big_b)
{
    return Err(ProtocolError::AssertionFailed(
        "received incorrect shares of additive triple phase.".to_string(),
    ));
}
``` [5](#0-4) 

Both checks will fail whenever the participant sets differ, causing the presigning protocol to abort and producing no usable presignature.

---

### Impact Explanation

**High: Corruption of presign outputs so honest parties produce unusable cryptographic outputs.**

When the presigning participant set diverges from the triple-generation participant set — a realistic scenario after a key reshare — every participant's linearized shares are computed with wrong Lagrange coefficients. The protocol aborts at the consistency checks, and no valid presignature is produced. Honest parties cannot complete signing. A malicious coordinator can deliberately trigger this by instructing participants to reuse pre-reshare triples with the post-reshare participant set, causing indefinite denial of signing without any cryptographic break.

---

### Likelihood Explanation

The scenario arises naturally in any deployment that:
1. Pre-generates a pool of triples (the documented offline phase),
2. Subsequently performs a key reshare (adding/removing participants or changing threshold), and
3. Attempts to consume pre-generated triples with the updated participant set.

The `TriplePub.participants` field exists precisely to carry the information needed to detect this mismatch, yet `presign` never reads it. The omission is acknowledged in a code comment, indicating the library relies on callers to enforce consistency — but provides no enforcement mechanism and no documented invariant warning callers of the requirement.

---

### Recommendation

In `presign` (`src/ecdsa/ot_based_ecdsa/presign.rs`), after constructing the `ParticipantList`, validate that the presigning participant set is consistent with both triples' recorded participant sets:

```rust
// Validate participant set matches triple generation participant set
let presign_participants_sorted: Vec<Participant> = participants.participants().to_vec();
if args.triple0.1.participants != presign_participants_sorted
    || args.triple1.1.participants != presign_participants_sorted
{
    return Err(InitializationError::BadParameters(
        "Presigning participant set must match the participant set used during triple generation".to_string(),
    ));
}
```

This makes the invariant explicit and enforced at the API boundary rather than relying on a silent assumption that callers will never mix participant sets across a reshare.

### Proof of Concept

**Setup:**
- 5 participants `{P1, P2, P3, P4, P5}`, threshold = 3.
- Triples generated with all 5 participants → `TriplePub.participants = [P1, P2, P3, P4, P5]`.
- Key reshare removes `P5`, new set = `{P1, P2, P3, P4}`.

**Attack:**
- Coordinator instructs `{P1, P2, P3, P4}` to run `presign` using the old triples (still valid shares for `P1–P4`) but with the new 4-participant set.
- Each participant computes `lambda_me` relative to `{P1, P2, P3, P4}` instead of `{P1, P2, P3, P4, P5}`.
- The linearized `e = sum_i lambda_i({P1..P4}) * c_i ≠ c = f_c(0)`.
- The check `big_e != e*G` fails → protocol aborts for all honest parties.
- No presignature is produced; signing is denied.
- The coordinator can repeat this with any remaining pre-reshare triples, exhausting the triple pool and permanently blocking signing until new triples are generated with the new participant set. [6](#0-5) [1](#0-0)

    me: Participant,
    args: PresignArguments,

        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }
    // Spec 1.1
    if args.threshold.value() > participants.len() {
        return Err(InitializationError::ThresholdTooLarge {
            threshold: args.threshold.value(),
            max: participants.len(),
        });
    }

    // NOTE: We omit the check that the new participant set was present for
    // the triple generation, because presumably they need to have been present
    // in order to have shares.

    // Also check that we have enough participants to reconstruct shares.
    if args.threshold != args.triple0.1.threshold || args.threshold != args.triple1.1.threshold {
        return Err(InitializationError::BadParameters(
            "New threshold must match the threshold of both triples".to_string(),
        ));
    }

    let participants =
        ParticipantList::new(participants).ok_or(InitializationError::DuplicateParticipants)?;

        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    let ctx = Comms::new();
    let fut = do_presign(ctx.shared_channel(), participants, me, args);
    Ok(make_protocol(ctx, fut))
