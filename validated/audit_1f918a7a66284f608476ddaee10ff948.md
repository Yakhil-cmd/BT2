### Title
Missing `validate_no_value_attached_to_precompile` in All BLS12-381 Precompiles Allows Permanent ETH Freezing — (`engine-precompiles/src/bls12_381/g1_add.rs`, `g2_add.rs`, `g1_msm.rs`, `g2_msm.rs`, `pairing_check.rs`, `map_fp_to_g1.rs`, `map_fp2_to_g2.rs`)

---

### Summary

All seven BLS12-381 precompiles introduced under the Prague/Osaka hard forks ignore the `context.apparent_value` field entirely and never call `utils::validate_no_value_attached_to_precompile`. Because the underlying EVM executor (`aurora-evm`/SputnikVM) transfers value from the caller to the target address as part of the call state transition before executing the precompile, a successful precompile call with `value > 0` permanently credits ETH to the precompile address. That address has no private key and no spendable code path, so the ETH is irrecoverably frozen.

---

### Finding Description

`BlsG1Add::run` (and all sibling BLS12-381 precompiles) accept the `_context: &Context` parameter but never inspect `context.apparent_value`: [1](#0-0) 

By contrast, every Aurora-specific precompile that can receive value explicitly rejects it: [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) 

The guard function is: [6](#0-5) 

The EVM call path is:

1. `engine.call()` invokes `executor.transact_call(origin, contract=0xB, value, input, ...)`.
2. SputnikVM transfers `value` from the caller's balance to address `0xB` in the EVM state before dispatching the precompile.
3. `BlsG1Add::run` is called; with a valid 256-byte input it returns `Ok(PrecompileOutput)`.
4. Because the call succeeded, `executor.into_state().deconstruct()` yields `Apply::Modify` items reflecting the balance change (caller −value, address `0xB` +value).
5. `self.apply(values, ...)` commits those changes to storage. [7](#0-6) [8](#0-7) 

The accounting invariant check only panics on a **net gain** of ETH (i.e., ETH created from nothing). A caller→precompile transfer is net-zero from the accounting module's perspective, so it passes silently: [9](#0-8) 

Address `0xB` has no private key and no deployed code; the ETH balance stored there can never be spent, transferred, or withdrawn.

The same defect exists in all seven BLS12-381 precompiles:

|---|---|---|
| `BlsG1Add` | `0xB` | `g1_add.rs` |
| `BlsG1Msm` | `0xC` | `g1_msm.rs` |
| `BlsG2Add` | `0xD` | `g2_add.rs` |
| `BlsG2Msm` | `0xE` | `g2_msm.rs` |

| `BlsMapFpToG1` | `0x10` | `map_fp_to_g1.rs` |
| `BlsMapFp2ToG2` | `0x11` | `map_fp2_to_g2.rs` | [10](#0-9) 

---

### Impact Explanation

Any ETH sent to a BLS12-381 precompile address with a valid input is permanently frozen. The caller's balance is debited, the precompile address's balance is credited, and there is no mechanism to recover those funds. Over time, repeated calls accumulate an ever-growing pool of permanently inaccessible ETH, constituting a **permanent freezing of user funds** and contributing to **insolvency** of the engine's ETH supply accounting (the connector's total supply diverges from the sum of spendable balances).

---

### Likelihood Explanation

- The BLS12-381 precompiles are active in the Prague and Osaka hard fork configurations already present in the codebase.
- The attack requires only a standard EVM `CALL` to address `0xB` with `value > 0` and a valid 256-byte input — no special privileges, no admin access, no external dependencies.
- Any EVM user or contract can trigger this; it is trivially scriptable.

---

### Recommendation

Add `utils::validate_no_value_attached_to_precompile(context.apparent_value)?;` as the first statement in the `run` method of every BLS12-381 precompile, mirroring the pattern used by `PredecessorAccount`, `CurrentAccount`, `PrepaidGas`, `RandomSeed`, `PromiseResult`, and `CrossContractCall`.
