### Title
Unrestricted `update_rewards` with Caller-Controlled `disable_rewards` Flag Allows Permanent Freezing of Consensus Rewards — (File: src/staking/staking.cairo)

### Summary
The `update_rewards` function in the staking contract has no caller access control and accepts a caller-supplied `disable_rewards` boolean. Any unprivileged address can call it with `disable_rewards: true` every block, consuming the single per-block reward slot while distributing nothing, permanently starving all stakers of their consensus-era unclaimed yield.

### Finding Description
`update_rewards` is exposed as a fully public function under `IStakingRewardsManager`. Its only gate is `general_prerequisites`, which checks only that the contract is unpaused and the caller is non-zero — no role check, no allowlist, no trusted-caller assertion. [1](#0-0) 

The function enforces a single-call-per-block invariant through a single global `last_reward_block` storage variable: [2](#0-1) 

After passing the block-number check, it unconditionally writes the current block to `last_reward_block`, then branches on `disable_rewards`: [3](#0-2) 

When `disable_rewards` is `true`, the function returns immediately after updating `last_reward_block` — no rewards are calculated or transferred. Because `last_reward_block` is a **single global slot** (not per-staker), any subsequent call to `update_rewards` in the same block fails with `REWARDS_ALREADY_UPDATED`. An attacker who calls `update_rewards(any_valid_staker, true)` in every block permanently occupies the reward slot without distributing anything.

The `disable_rewards` parameter has no validation against who is supplying it: [4](#0-3) 

Compare with `update_rewards_from_attestation_contract`, which correctly restricts its caller: [5](#0-4) 

`update_rewards` has no equivalent guard.

### Impact Explanation
All stakers in the consensus-rewards phase (`current_epoch >= consensus_rewards_first_epoch`) lose their unclaimed yield for every block the attacker occupies. Because the attacker can sustain this indefinitely at the cost of one transaction per block, this constitutes **permanent freezing of unclaimed yield** — a High-severity impact under the allowed scope.

### Likelihood Explanation
The entry path requires no privilege: any non-zero EOA or contract can call `update_rewards`. The only requirement is a valid (active, non-zero-balance) staker address to pass the staker-existence checks, which is trivially satisfied by reading any existing staker from on-chain events. The cost is one Starknet transaction per block. No leaked keys, bridge compromise, or external dependency is involved.

### Recommendation
Add a trusted-caller check to `update_rewards`, analogous to the guard already present in `update_rewards_from_attestation_contract`:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
+   self.assert_caller_is_attestation_contract(); // or a dedicated rewards-manager role
    ...
}
```

Alternatively, remove the `disable_rewards` parameter from the public interface and handle the "no-reward" case internally based on on-chain attestation state.

### Proof of Concept

1. Wait for `consensus_rewards_first_epoch` to be set (consensus rewards phase begins).
2. Read any active staker address `S` from on-chain `NewStaker` events.
3. In every Starknet block, submit: `staking_contract.update_rewards(S, disable_rewards: true)`.
4. Each call passes all checks (unpaused, non-zero caller, valid staker, non-zero balance, new block) and writes `last_reward_block = current_block`.
5. No rewards are calculated or transferred (`disable_rewards` branch returns early).
6. Any legitimate call to `update_rewards` in the same block reverts with `REWARDS_ALREADY_UPDATED`.
7. All stakers accumulate zero consensus rewards indefinitely. [6](#0-5)
