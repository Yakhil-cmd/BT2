### Title
`proposePoolPriceProvider` allows admin to re-propose without expiry, halving the effective LP-protection timelock and enabling opportunistic oracle switching — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`proposePoolPriceProvider` unconditionally overwrites the pending proposal with no guard against an existing one, and `executePoolPriceProviderUpdate` has no expiry on the pending proposal. A pool admin can (a) keep re-proposing every `timelock/2` seconds to maintain a perpetually "warm" proposal while the effective LP exit window is only `timelock/2`, and (b) execute a stale, already-elapsed proposal at any strategically chosen future moment. Both paths let the admin bypass the timelock's LP-protection intent.

---

### Finding Description

`proposePoolPriceProvider` in `MetricOmmPoolFactory` unconditionally writes to `pendingPriceProvider[pool]` and `pendingPriceProviderExecuteAfter[pool]`:

```solidity
// metric-core/contracts/MetricOmmPoolFactory.sol lines 487-490
uint256 executeAfter = block.timestamp + timelock;
pendingPriceProvider[pool] = newPriceProvider;
pendingPriceProviderExecuteAfter[pool] = executeAfter;
```

There is no check for an existing pending proposal, and there is no `cancelPoolPriceProvider` function in the interface (only `cancelPoolAdminTransfer` exists). [1](#0-0) 

`executePoolPriceProviderUpdate` enforces only a lower-bound check (`block.timestamp < execAfter`) with no upper-bound expiry: [2](#0-1) 

The interface confirms there is no cancel function for the price provider proposal: [3](#0-2) 

This creates two exploitable paths:

**Path A — Timelock halving via re-proposal (direct analog to external bug):**
1. Admin proposes `providerB` at `T=0` → `executeAfter = T + L` (where `L = priceProviderTimelock`).
2. At `T = L/2`, admin re-proposes `providerB` → `executeAfter` resets to `T + L/2 + L`.
3. Repeat every `L/2` seconds indefinitely.
4. The admin always has a "ready to execute" proposal pending, but LPs only ever see `L/2` of real notice before the change can land.

**Path B — Stale proposal executed at opportunistic moment:**
1. Admin proposes `providerB` at `T=0`.
2. Timelock elapses at `T=L`. LPs observe no execution and re-enter the pool.
3. Months later, when the pool holds maximum liquidity or the current oracle is giving LP-favorable prices, the admin executes the stale proposal in a single transaction, switching to a degraded oracle with no further warning.

---

### Impact Explanation

The `priceProviderTimelock` is the sole on-chain mechanism protecting LPs from an admin-controlled oracle switch. Bypassing it via re-proposal or stale execution allows the admin to switch the pool to a manipulated, stale, or adversarial `IPriceProvider` at a moment of their choosing. Because the oracle directly determines the bid/ask prices used in every swap, a degraded provider causes bad-price execution: traders receive more than the correct oracle permits, or LPs receive less than owed, resulting in direct LP principal loss and pool insolvency risk. This matches the "bad-price execution" and "admin-boundary break: bypasses timelocks" allowed impact categories.

---

### Likelihood Explanation

The trigger requires only the pool admin (semi-trusted per contest scope). No external conditions, special tokens, or off-chain data are needed. The re-proposal path costs only gas and can be scripted. The stale-execution path requires no repeated action at all — a single proposal followed by a delayed execute suffices. Likelihood is **Medium**: it requires a malicious or compromised admin, but the timelock is the only LP safeguard and the bypass is trivially executable.

---

### Recommendation

1. **Reject re-proposals while one is pending**: add `if (pendingPriceProvider[pool] != address(0)) revert PriceProviderChangePending();` at the top of `proposePoolPriceProvider`.
2. **Add a proposal expiry**: store a `pendingPriceProviderExpireAfter` alongside `pendingPriceProviderExecuteAfter` and revert in `executePoolPriceProviderUpdate` if `block.timestamp > expireAfter`. A reasonable expiry is `executeAfter + timelock`.
3. **Add a `cancelPoolPriceProvider` function** so the admin can explicitly withdraw a proposal (emitting an event LPs can monitor), rather than silently overwriting it.

### Proof of Concept

```solidity
// Demonstrates Path A: effective timelock = L/2
uint256 L = factory.priceProviderTimelock(pool); // e.g. 7 days

// T=0: propose providerB
vm.prank(admin);
factory.proposePoolPriceProvider(pool, address(providerB));
// executeAfter = block.timestamp + 7 days

// T=3.5 days: re-propose before timelock elapses — resets the clock
vm.warp(block.timestamp + L / 2);
vm.prank(admin);
factory.proposePoolPriceProvider(pool, address(providerB));
// executeAfter = block.timestamp + 7 days (LPs only had 3.5 days of real notice)

// T=10.5 days: execute — LPs had at most 3.5 days of actual warning
vm.warp(block.timestamp + L);
vm.prank(admin);
factory.executePoolPriceProviderUpdate(pool);
// Pool now uses providerB; LPs had half the intended notice period

// Demonstrates Path B: stale execution
vm.prank(admin);
factory.proposePoolPriceProvider(pool, address(providerB));
vm.warp(block.timestamp + L); // timelock elapses; LPs see no execution and re-enter
// ... 30 days later, pool has maximum TVL ...
vm.warp(block.timestamp + 30 days);
vm.prank(admin);
factory.executePoolPriceProviderUpdate(pool); // executes with no additional warning
```

```text
  function executePoolPriceProviderUpdate(address pool) external override nonReentrant onlyPoolAdmin(pool) {
    address pending = pendingPriceProvider[pool];
    if (pending == address(0)) revert NoPriceProviderChangeProposed();
    uint256 execAfter = pendingPriceProviderExecuteAfter[pool];
    // forge-lint: disable-next-line(block-timestamp) -- timelock enforcement legitimately relies on `block.timestamp`.
    if (block.timestamp < execAfter) revert PriceProviderTimelockNotElapsed(execAfter, block.timestamp);
    PoolImmutables memory p = IMetricOmmPool(pool).getImmutables();
    if (p.immutablePriceProvider != address(0)) revert PriceProviderImmutable();
    _validatePriceProvider(p.token0, p.token1, pending);
    IMetricOmmPoolFactoryActions(pool).setPriceProvider(pending);
    delete pendingPriceProvider[pool];
    delete pendingPriceProviderExecuteAfter[pool];
    emit PoolPriceProviderUpdated(pool, pending);
  }
```
