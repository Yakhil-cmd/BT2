### Title
Missing Threshold Validation in CKD Protocol Allows Coordinator to Produce Corrupted Derived Keys - (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

The `ckd()` function does not accept or validate a threshold parameter. A malicious coordinator can invoke the CKD protocol with fewer participants than the threshold required to correctly reconstruct the master secret key, causing all honest participants to contribute valid shares toward a cryptographically incorrect derived key that the coordinator accepts as legitimate output.

---

### Finding Description

The `ckd()` function performs four input validations: [1](#0-0) 

It checks that at least 2 participants are present, no duplicates exist, `me` is in the list, and the coordinator is in the list. However, **no threshold parameter is accepted or validated**. The threshold is a security parameter established during DKG, but it is absent from the `ckd()` signature and is not stored in `KeygenOutput`.

Inside `compute_signature_share()`, each participant computes their normalized share as `lambda_i * big_c_i`, where `lambda_i` is the Lagrange coefficient relative to the **current** `participants` list: [2](#0-1) 

The coordinator then sums all normalized shares: [3](#0-2) 

The aggregated `norm_big_c` equals `H(pk, app_id) * msk + ...` **only when** the Lagrange interpolation over the participant set correctly reconstructs the master secret key `msk`. This reconstruction is mathematically correct only when `participants.len() >= threshold`. If `participants.len() < threshold`, the Lagrange basis polynomials evaluated at zero produce a value different from `msk`, and the CKD output is a corrupted derived key.

Honest participants have no way to detect this: they only compute their own share relative to the provided participant list and send it to the coordinator. The `KeygenOutput` struct carries only `private_share` and `public_key`, not the threshold: [4](#0-3) 

---

### Impact Explanation

A malicious coordinator can call `ckd()` with fewer participants than the keygen threshold. All honest participants will compute and submit cryptographically valid (but wrong) shares. The coordinator aggregates them into a CKD output that is silently incorrect — a derived key that does not correspond to `H(pk, app_id) * msk`. Any downstream consumer of this output (e.g., a TEE decrypting with the expected derived key) will fail or receive a wrong key without any protocol-level error.

This matches the **High** impact: *Corruption of CKD outputs so honest parties accept inconsistent or unusable cryptographic outputs.*

---

### Likelihood Explanation

The coordinator is an explicit adversarial role in the threshold model. The `ckd()` public API provides no mechanism to enforce the threshold, and `KeygenOutput` does not carry it. A malicious coordinator needs only to omit participants from the call — a trivial, single-step action with no cryptographic barrier.

---

### Recommendation

Add a `threshold: impl Into<ReconstructionLowerBound>` parameter to `ckd()` and validate:

```rust
if participants.len() < threshold.value() {
    return Err(InitializationError::NotEnoughParticipantsForThreshold {
        threshold: threshold.value(),
        participants: participants.len(),
    });
}
```

Alternatively, store the threshold inside `KeygenOutput` so it can be validated internally without requiring callers to pass it separately. The pattern used in `assert_sign_inputs` in `src/frost/mod.rs` (lines 144–150) is the correct model to follow: [5](#0-4) 

---

### Proof of Concept

1. Run DKG with 3 participants and threshold = 3, producing `KeygenOutput` for each.
2. Call `ckd(participants = [p0, p1], coordinator = p0, me = p0, key_pair, app_id, app_pk, rng)` — only 2 participants, below threshold 3.
3. `ckd()` passes all four existing checks (≥2 participants, no duplicates, `me` present, coordinator present).
4. Each of `p0`, `p1` computes `lambda_i` relative to the 2-participant list — these Lagrange coefficients do **not** reconstruct `msk` from a 3-of-3 sharing.
5. The coordinator aggregates the shares into `CKDOutput` containing a wrong derived key.
6. No error is returned; honest parties accept the corrupted output. [6](#0-5)

pub fn ckd(
    participants: &[Participant],
    coordinator: Participant,
    me: Participant,
    key_pair: KeygenOutput,
    app_id: impl Into<AppId>,
    app_pk: PublicKey,
    rng: impl CryptoRngCore + Send + 'static,
) -> Result<impl Protocol<Output = CKDOutputOption>, InitializationError> {
    // not enough participants
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }

    // kick out duplicates
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }

    let comms = Comms::new();
    let chan = comms.shared_channel();

    let fut = run_ckd_protocol(
        chan,
        coordinator,
        me,
        participants,
        key_pair,
        app_id.into(),
        app_pk,
        rng,
    );
    Ok(make_protocol(comms, fut))
