Audit Report

## Title
Malicious Canister Can Displace Legitimate Best-Effort Messages via Size-Biased Load Shedding - (File: `rs/replicated_state/src/replicated_state.rs`)

## Summary
The IC's best-effort messaging system enforces a 5 GiB subnet-wide memory cap and sheds the **largest** best-effort messages when the cap is exceeded. Because `ic0_call_with_best_effort_response` enforces only an upper bound on timeout (no lower bound), an attacker can flood a target canister with many small best-effort requests carrying a 1-second timeout. When the cap is exceeded, `enforce_best_effort_message_limit` sheds the largest messages — the legitimate ones — while the attacker's small messages expire naturally in the next round at minimal additional cost.

## Finding Description

**Root cause 1 — No minimum timeout:**

`rs/embedders/src/wasmtime_embedder/system_api.rs` lines 4196–4199 apply only an upper bound:

```rust
Some(request) => {
    let bounded_timeout =
        std::cmp::min(timeout_seconds, MAX_CALL_TIMEOUT_SECONDS);
    request.set_timeout(bounded_timeout);
```

`MAX_CALL_TIMEOUT_SECONDS = 300` (line 65). There is no `max(timeout_seconds, MIN_CALL_TIMEOUT_SECONDS)` lower bound. Any canister can pass `timeout_seconds = 1`, producing a deadline of `current_time + 1 second`.

**Root cause 2 — Load shedding is purely size-ordered:**

`rs/replicated_state/src/canister_state/queues/message_pool.rs` line 396 defines the shedding priority queue keyed by `(usize, Id)` — byte size ascending:

```rust
size_queue: MutableIntMap<(usize, Id), ()>,
```

`shed_largest_message` (lines 674–688) pops `self.size_queue.max_key()`, unconditionally removing the globally largest best-effort message from the pool, regardless of deadline proximity.

**Root cause 3 — Shedding targets the highest-memory canister:**

`rs/replicated_state/src/replicated_state.rs` lines 1333–1334 pop the canister with the largest best-effort memory footprint:

```rust
while memory_usage > limit && !priority_queue.is_empty() {
    let (memory_usage_before, canister_id) = priority_queue.pop_last().unwrap();
```

If the attacker floods a single target canister with many small messages, that canister accumulates the highest memory usage and becomes the repeated shedding target. Its largest messages — the legitimate ones — are shed first.

**Exploit flow:**

1. Attacker deploys N canisters; each calls `ic0_call_with_best_effort_response(1)` and `ic0_call_perform` to send many small (e.g., 5 KB) best-effort requests to a target canister.
2. The target canister's best-effort memory usage grows, eventually pushing the subnet total past `SUBNET_BEST_EFFORT_MESSAGE_MEMORY_CAPACITY = 5 GiB` (`rs/config/src/execution_environment.rs` line 62).
3. At end-of-round, `state_after_stream_builder.enforce_best_effort_message_limit(...)` fires (`rs/messaging/src/state_machine.rs` lines 267–270).
4. The target canister has the highest memory usage; `shed_largest_message` repeatedly removes its largest messages — the legitimate large ones.
5. In the next round, `expire_messages` removes the attacker's 1-second-deadline requests. The attacker's memory footprint returns to zero; the legitimate messages are permanently gone (converted to `SYS_UNKNOWN` reject responses).

**Why existing checks are insufficient:**

- The `size_queue` ordering has no deadline-proximity component; a message expiring in 1 second is treated identically to one expiring in 300 seconds.
- There is no per-sender rate limit on best-effort message induction that would throttle the attacker's flood.
- The shedding loop does not distinguish attacker-originated messages from legitimate ones.

## Impact Explanation

Legitimate best-effort inter-canister messages are permanently dropped and their senders receive `SYS_UNKNOWN` reject responses. Workflows that rely on best-effort messaging — including cross-subnet calls, SNS downstream governance calls, and ckBTC/ckETH minter interactions — can be selectively disrupted. In-flight cycles attached to shed messages are lost. Because the 5 GiB cap is subnet-wide, the attack simultaneously affects all canisters on the targeted subnet.

This matches **High ($2,000–$10,000): Application/platform-level DoS or subnet availability impact not based on raw volumetric DDoS**, as the attacker exploits a protocol-level algorithmic bias to selectively evict legitimate traffic.

## Likelihood Explanation

Any deployed canister can call `ic0_call_with_best_effort_response(1)` without special privileges. The attacker's messages expire after one round, so the cycles cost per wave is bounded by the number of small messages needed to push the subnet over the 5 GiB cap — which is far less than 5 GiB if the subnet already carries substantial legitimate best-effort traffic. Multiple attacker canisters amplify throughput. The attack is repeatable every round (~1 second) with no recovery cost between waves.

## Recommendation

1. **Enforce a minimum timeout** in `ic0_call_with_best_effort_response` (`rs/embedders/src/wasmtime_embedder/system_api.rs`):
   ```rust
   const MIN_CALL_TIMEOUT_SECONDS: u32 = 10;
   let bounded_timeout = timeout_seconds
       .max(MIN_CALL_TIMEOUT_SECONDS)
       .min(MAX_CALL_TIMEOUT_SECONDS);
   ```
   This forces attacker messages to occupy memory for at least 10 rounds, raising the sustained cycles cost of the attack.

2. **Incorporate deadline proximity into the shedding order.** Prefer shedding messages whose deadlines are nearest to expiry before shedding by size. Messages about to expire naturally are the cheapest to shed and the least disruptive to legitimate traffic.

3. **Rate-limit best-effort message induction per sender.** Track per-sender best-effort byte counts in the induction layer and reject or throttle senders that exceed a per-round quota, preventing a single attacker from dominating a target canister's memory footprint.

## Proof of Concept

```
Setup:
  Subnet best-effort memory usage: 4.8 GiB (legitimate traffic).
  LegitCanister holds: 50 × 4 MB best-effort requests (outbound, deadline = now+200s).

Round N:
  Attacker canisters A1..A50 each send 1,000 × 5 KB best-effort requests
  (timeout=1) to LegitCanister.
  Total attacker contribution: 50 × 1,000 × 5 KB = 250 MB.
  Subnet total: 4.8 GiB + 250 MB > 5 GiB → cap exceeded.

  enforce_best_effort_message_limit fires:
    LegitCanister has highest memory usage.
    shed_largest_message() removes the 4 MB legitimate requests one by one.
    All 50 legitimate requests are shed → SYS_UNKNOWN reject to their senders.

Round N+1:
  expire_messages() removes all attacker messages (deadline < now).
  Attacker memory footprint: 0.
  Legitimate messages: permanently gone.

Reproducible test: write an integration test using PocketIC that:
  1. Creates a subnet with a reduced best_effort_message_memory_capacity (e.g., 10 MB).
  2. Installs a "legitimate sender" canister that enqueues several large (1 MB)
     best-effort requests to a target canister.
  3. Installs an "attacker" canister that sends many small (1 KB, timeout=1)
     best-effort requests to the same target canister, pushing total usage over 10 MB.
  4. Advances one round and asserts that the legitimate sender receives
     SYS_UNKNOWN reject responses while the attacker's messages have expired cleanly.
```
