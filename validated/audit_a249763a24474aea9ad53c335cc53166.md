### Title
Incorrect NFT Seller Net Proceeds Calculation Misleads Users Into Accepting Offers With Inflated Displayed Amounts - (File: packages/gui/src/components/offers/utils.ts)

### Summary
In `calculateNFTRoyalties`, `nftSellerNetAmount` is hardcoded to the gross `amount` instead of `amount - royaltyAmount`. This incorrect value propagates into both the NFT offer editor ("You will receive") and the offer viewer ("Net Proceeds" and the primary "You will receive" override), causing the NFT seller to see an inflated figure when reviewing an imported offer before accepting it.

### Finding Description
In `packages/gui/src/components/offers/utils.ts`, the `calculateNFTRoyalties` function computes `nftSellerNetAmount` as the raw input `amount`, with the correct subtraction of royalties and maker fee commented out:

```ts
const nftSellerNetAmount: number = amount;
// : parseFloat(
//     (amount - parseFloat(royaltyAmountString) - makerFee).toFixed(12),
//   );
``` [1](#0-0) 

The correct value should be `amount - royaltyAmount` (for the NFTForToken case) or `amount - royaltyAmount` (for the TokenForNFT case where royalties are deducted from the seller's proceeds). The commented-out block was the intended implementation.

This incorrect `nftSellerNetAmount` is then consumed in two critical display paths:

**1. NFTOfferViewer — `overrideNFTSellerAmount`** (the primary "You will receive" display when accepting an imported offer):
```ts
const overrideNFTSellerAmount =
  exchangeType === NFTOfferExchangeType.TokenForNFT
    ? assetType === OfferAsset.CHIA
      ? chiaToMojo(nftSaleInfo?.nftSellerNetAmount ?? 0)
      : catToMojo(nftSaleInfo?.nftSellerNetAmount ?? 0)
    : undefined;
``` [2](#0-1) 

This `overrideNFTSellerAmount` is passed to `NFTOfferSummary` as the amount shown under the prominent "You will receive" label. [3](#0-2) 

**2. NFTOfferViewer — "Net Proceeds" label** (explicitly described in its tooltip as "the asking price, minus any associated creator fees"):
```tsx
<FormatLargeNumber value={new BigNumber(nftSaleInfo?.nftSellerNetAmount ?? 0)} /> {displayName}
``` [4](#0-3) 

The tooltip for this field explicitly states: *"The net proceeds include the asking price, minus any associated creator fees."* [5](#0-4) 

**3. NFTOfferEditor — "You will receive" / "They will receive"** (offer creation preview): [6](#0-5) 

Additionally, the `showNegativeAmountWarning` guard (`nftSellerNetAmount < 0`) can never trigger because `nftSellerNetAmount` is always equal to the positive input `amount`, even when the actual net would be negative. [7](#0-6) 

### Impact Explanation
In the `TokenForNFT` exchange type, the NFT seller's actual on-chain proceeds equal `offered_amount - royalty_amount`. For example, if a buyer offers 100 XCH for an NFT with a 10% royalty, the NFT seller receives 90 XCH and the NFT creator receives 10 XCH. However, the GUI displays "Net Proceeds: 100 XCH" and "You will receive: 100 XCH" — the full gross amount — causing the seller to accept an offer under a materially false understanding of what they will receive. The actual blockchain transaction correctly deducts royalties, so the seller receives 10% less than the GUI promised.

This meets the **High** impact criterion: *"Corruption, spoofing, or unsafe trust of... offer... state that causes a user to... display the wrong... amount... or status"* — specifically causing the user to accept an offer based on a wrong displayed amount.

### Likelihood Explanation
Any NFT with a non-zero `royaltyPercentage` triggers this bug. An attacker (token buyer) simply creates a standard `TokenForNFT` offer for any royalty-bearing NFT and shares the offer file. No special privileges, leaked keys, or social engineering beyond sharing a valid offer file are required. The NFT seller opens the offer in the GUI, sees the inflated "Net Proceeds" and "You will receive" figures, and accepts.

### Recommendation
Restore the commented-out net amount calculation in `calculateNFTRoyalties`:

```ts
// In packages/gui/src/components/offers/utils.ts
const nftSellerNetAmount: number = parseFloat(
  (amount - parseFloat(royaltyAmountString)).toFixed(12),
);
```

The `makerFee` subtraction in the original commented-out code should be reviewed separately — in the `TokenForNFT` case the maker fee is paid by the buyer on top of the offered amount, not deducted from the seller's proceeds, so only `royaltyAmount` should be subtracted from `nftSellerNetAmount`.

### Proof of Concept
1. Mint an NFT with `royalty_percentage = 1000` (10%).
2. As the token buyer (attacker), create a `TokenForNFT` offer: offer 100 XCH for the NFT.
3. Share the offer file with the NFT seller (victim).
4. Victim opens the offer in the Chia GUI (`NFTOfferViewer`).
5. GUI displays:
   - "You will receive: **100 XCH**" (via `overrideNFTSellerAmount`)
   - "NFT Purchase Price: 100 XCH"
   - "Creator Fee (10%): 10 XCH"
   - "Net Proceeds: **100 XCH**" (should be 90 XCH)
6. Victim accepts the offer expecting 100 XCH.
7. On-chain settlement: victim receives **90 XCH**; 10 XCH goes to the NFT creator.

The root cause is `nftSellerNetAmount = amount` at line 314 of `utils.ts` instead of `amount - royaltyAmount`. [8](#0-7)

export function calculateNFTRoyalties(
  amount: number,
  makerFee: number,
  royaltyPercentage: number,
  exchangeType: NFTOfferExchangeType,
): CalculateNFTRoyaltiesResult {
  const royaltyAmount: number = royaltyPercentage ? (royaltyPercentage / 100) * amount : 0;
  const royaltyAmountString: string = formatAmount(royaltyAmount);
  const nftSellerNetAmount: number = amount;
  // : parseFloat(
  //     (amount - parseFloat(royaltyAmountString) - makerFee).toFixed(12),
  //   );
  const totalAmount: number =
    exchangeType === NFTOfferExchangeType.NFTForToken ? amount + royaltyAmount : amount + makerFee + royaltyAmount;
  const totalAmountString: string = formatAmount(totalAmount);

  return {
    royaltyAmount,
    royaltyAmountString,
    nftSellerNetAmount,
    totalAmount,
    totalAmountString,
  };
}
