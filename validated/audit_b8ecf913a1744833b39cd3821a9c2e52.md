### Title
Malicious CKD Participant Can Corrupt Derived Key Output Without Detection - (File: `src/confidential_key_derivation/protocol.rs`)

### Summary
The CKD coordinator unconditionally aggregates `(norm_big_y, norm_big_c)` contributions from every participant with no proof of correct computation. A single malicious participant can send arbitrary group elements that pass the type check but are not derived from their actual secret share, silently corrupting the final `CKDOutput` that honest parties accept.

### Finding Description
In `do_ckd_coordinator`, the coordinator receives each participant's `(norm_big_y, norm_big_c)` pair and adds them directly into the running aggregate:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
``` [1](#0-0) 

The correct computation each participant is supposed to perform is:

```
big_s  = x_i · H(pk ‖ app_id)
big_c  = big_s + y_i · app_pk
norm_big_y = λ_i · y_i · G
norm_big_c = λ_i · big_c
``` [2](#0-1) 

No zero-knowledge proof, commitment, or consistency check is attached to the message sent by `do_ckd_participant`:

```rust
chan.send_private(waitpoint, coordinator, &(norm_big_y, norm_big_c))?;
``` [3](#0-2) 

Because the coordinator performs no verification, a malicious participant can substitute any pair of valid `G1` points. The coordinator cannot distinguish a correctly-computed share from a fabricated one.

This is structurally identical to the reported Rubicon bug: just as the malicious `targetPool` implements the `IBathToken` interface and returns values that pass every `require` check while performing no real token transfer, a malicious CKD participant returns valid `ElementG1` values that satisfy the type system while encoding no relationship to their actual secret share.

### Impact Explanation
**High — Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.**

The final `CKDOutput` is:

```
big_Y = Σ λ_i · y_i · G
big_C = Σ λ_i · (x_i · H(pk‖app_id) + y_i · app_pk)
```

When unmasked by the application: `big_C − app_sk · big_Y = msk · H(pk‖app_id)`.

If one participant substitutes `(0, 0)` or any arbitrary `(Y', C')`, the aggregate deviates from the correct value. The coordinator and all honest parties receive and accept a `CKDOutput` that does not equal `msk · H(pk‖app_id)`. The derived confidential key is permanently wrong for that invocation, and the honest parties have no way to detect the corruption. [4](#0-3) 

### Likelihood Explanation
Any single participant in the protocol can trigger this. The attacker needs only to be a legitimate member of the `participants` list — no special privilege is required. The malicious message is indistinguishable from an honest one at the network layer. Because the protocol is one-shot (participants send once and the coordinator aggregates once), there is no retry or detection mechanism. [5](#0-4) 

### Recommendation
Require each participant to attach a non-interactive zero-knowledge proof of correct computation alongside their `(norm_big_y, norm_big_c)` contribution. Concretely, each participant should prove in zero knowledge that:

1. `norm_big_y = λ_i · y_i · G` for a known `y_i`
2. `norm_big_c = λ_i · (x_i · H(pk‖app_id) + y_i · app_pk)` for the same `y_i` and for `x_i` consistent with the participant's committed public share `x_i · G2`

The coordinator must verify all proofs before aggregating. This mirrors the proof-of-knowledge pattern already used in the DKG round to bind each participant's polynomial commitment to their secret. [6](#0-5) 

### Proof of Concept

1. Honest participants `P_1 … P_{n-1}` and malicious participant `P_n` all call `ckd(…)`.
2. `P_n` intercepts the `chan.send_private` call and instead sends `(G1::identity(), G1::identity())` (the additive identity) to the coordinator.
3. The coordinator executes:
   ```rust
   norm_big_y += participant_output.big_y();  // += identity → no change
   norm_big_c += participant_output.big_c();  // += identity → no change
   ```
   effectively dropping `P_n`'s legitimate share from the aggregate.
4. The coordinator returns a `CKDOutput` where `big_C − app_sk · big_Y ≠ msk · H(pk‖app_id)`.
5. The application calls `ckd_output.unmask(app_sk)` and receives a wrong confidential key with no error. [7](#0-6) [4](#0-3)

fn proof_of_knowledge<C: Ciphersuite>(
    session_id: &HashOutput,
    domain_separator: &mut DomainSeparator,
    me: Participant,
    coefficients: &Polynomial<C>,
    coefficient_commitment: &PolynomialCommitment<C>,
    rng: &mut impl CryptoRngCore,
) -> Result<Signature<C>, ProtocolError> {
    // creates an identifier for the participant
    let id = me.scalar::<C>();
    let vk_share = coefficient_commitment.eval_at_zero()?;

    // pick a random k_i and compute R_id = g^{k_id},
    // Step 2.5
    let (k, big_r) = <C>::generate_nonce(rng);

    // Step 2.6
    // compute H(domain_separator, id, me, g^{a_0}, R_id) as a scalar
    let hash = challenge::<C>(domain_separator, session_id, id, &vk_share, &big_r)?;
    let a_0 = coefficients.eval_at_zero()?.0;
    // Step 2.7
    let mu = k + a_0 * hash.to_scalar();
    Ok(Signature::new(big_r, mu))
}
