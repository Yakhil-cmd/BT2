### Title
`Staking.change_reward_address()` does not validate that `reward_address` is non-zero, permanently freezing staker yield — (File: src/staking/staking.cairo)

---

### Summary
`Staking.change_reward_address()` accepts any `ContractAddress` as the new reward address and only checks that it is not a registered token address. There is no guard against the zero address. A staker who mistakenly passes `0` will have all future reward claims permanently routed to the zero address, with no recovery path.

---

### Finding Description
`change_reward_address` in `src/staking/staking.cairo` performs exactly one validation on the incoming address — that it is not a known token contract — and then unconditionally writes it to storage:

```cairo
fn change_reward_address(ref self: ContractState, reward_address: ContractAddress) {
    self.general_prerequisites();
    assert!(
        !self.does_token_exist(token_address: reward_address),
        "{}",
        GenericError::REWARD_ADDRESS_IS_TOKEN,
    );
    let staker_address = get_caller_address();
    let mut staker_info = self.internal_staker_info(:staker_address);
    let old_address = staker_info.reward_address;

    staker_info.reward_address = reward_address;   // ← zero is accepted here
    self.write_staker_info(:staker_address, :staker_info);
    ...
}
``` [1](#0-0) 

There is no `assert!(reward_address.is_non_zero(), ...)` guard anywhere in this function. The `stake()` entry point has the same omission for the initial `reward_address` parameter. [2](#0-1) 

Once `reward_address` is set to zero, every subsequent `claim_rewards` call will transfer the staker's accumulated yield to `0x0`. Because the staking contract has no admin override to forcibly correct a staker's `reward_address`, the loss is permanent.

---

### Impact Explanation
**High — Permanent freezing of unclaimed yield.**

All epoch rewards that accrue after the zero-address assignment are transferred to the zero address and are unrecoverable. The staker retains their principal (it is not at risk), but every wei of yield is permanently destroyed. This matches the allowed impact: *"Permanent freezing of unclaimed yield or unclaimed royalties."*

---

### Likelihood Explanation
Any active staker can call `change_reward_address` at any time — it is a fully public, permissionless entry point for the staker role. A single off-by-one error in a script, a UI bug that submits an empty field, or a copy-paste mistake is sufficient to trigger the condition. No privileged access, no bridge compromise, and no third-party dependency is required.

---

### Recommendation
Add a non-zero guard at the top of `change_reward_address` (and symmetrically in `stake()`):

```cairo
assert!(reward_address.is_non_zero(), "{}", GenericError::REWARD_ADDRESS_IS_ZERO);
```

Apply the same guard to the pool contract's `change_reward_address` for pool members.

---

### Proof of Concept

1. Staker `A` is active with `reward_address = 0xABC`.
2. Staker `A` calls `staking.change_reward_address(reward_address: 0)`.
3. The only check — `does_token_exist(0)` — returns `false`, so the call succeeds.
4. `staker_info.reward_address` is now `0x0` in storage.
5. Any call to `claim_rewards` (by the staker or anyone) transfers accumulated yield to `0x0`.
6. There is no function in the contract that allows the staker to recover those tokens or for an admin to override the zero reward address on behalf of the staker.
7. All future yield for staker `A` is permanently frozen at the zero address. [3](#0-2)
