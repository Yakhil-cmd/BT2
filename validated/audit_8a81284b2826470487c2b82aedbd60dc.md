### Title
Unchecked Precompile Return Value Causes Silent Token Burn Without Exit — (`etc/eth-contracts/contracts/EvmErc20.sol`)

---

### Summary

Both `withdrawToNear` and `withdrawToEthereum` in `EvmErc20.sol` call `_burn` **before** invoking the exit precompile via inline assembly. The return value `res` of the `call` opcode is stored but **never checked**. Under standard EVM semantics, a failed sub-call returns 0 without reverting the caller's state. If the exit precompile fails for any reason, the burn is committed and the user's tokens are permanently destroyed with no corresponding NEP-141 release on NEAR.

---

### Finding Description

In `EvmErc20.sol`, both withdraw functions follow the same pattern:

```solidity
// withdrawToNear (lines 53-63)
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // ← state change committed here

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        // res is NEVER CHECKED — silent failure
    }
}
``` [1](#0-0) 

The same pattern exists in `withdrawToEthereum`: [2](#0-1) 

The exit precompile (`ExitToNear` at `0xe9217bc70b7ed1f598ddd3199e80b093fa71124f`) can fail with `PrecompileFailure::Error` (not `Fatal`) in several concrete conditions:

**Condition 1 — Invalid NEAR account ID as recipient:**
`ExitToNearParams::try_from` calls `parse_recipient`, which calls `AccountId::try_from` on the raw bytes. If the bytes are not a valid NEAR account ID (empty, too long, invalid characters), this returns `ExitError::Other(...)`. [3](#0-2) 

**Condition 2 — Amount exceeds `u128::MAX`:**
`parse_amount` rejects amounts that overflow `u128`, but `_burn` accepts any `uint256`. A user holding a balance > `u128::MAX` (theoretically possible) would have their tokens burned with the precompile rejecting the call.

**Condition 3 — ERC-20 not registered in the NEP-141 map:**
`get_nep141_from_erc20` returns `ERR_TARGET_TOKEN_NOT_FOUND` if the ERC-20 address has no corresponding NEP-141 entry. [4](#0-3) 

In all three cases, `process_precompile` maps the `ExitError` to `PrecompileFailure::Error`: [5](#0-4) 

`PrecompileFailure::Error` causes the sub-call to return 0 **without reverting the caller's state**. The `_burn` in `EvmErc20.sol` has already executed. The `res` variable is never inspected, so the function returns normally. No promise log is emitted, so `filter_promises_from_logs` schedules nothing: [6](#0-5) 

The promise log and exit event log are only produced inside `Ok(PrecompileOutput { ... })` — they are absent when the precompile returns `Err`: [7](#0-6) 

**Note on `PrecompileFailure::Fatal` (paused precompile):** The `is_paused` path returns `Fatal`, which propagates to `ExitFatal` and causes the entire NEAR transaction to panic — state is not committed. This path does **not** trigger the vulnerability. [8](#0-7) 

---

### Impact Explanation

When the precompile fails with `PrecompileFailure::Error`:
- The ERC-20 `Transfer` event to `address(0)` is emitted (burn is committed)
- No exit event and no promise log are produced
- The EVM transaction status is `Succeed`
- The NEP-141 tokens remain locked in Aurora's NEAR contract indefinitely
- The user has no recourse — there is no re-entry path, no refund mechanism, and no admin recovery function for this case

The result is **permanent freezing of user funds** (Critical per scope). This is not Insolvency — Aurora ends up over-collateralized (more NEP-141 locked than ERC-20 outstanding). The claimed "Insolvency" impact in the question is incorrect; the correct Critical impact is **permanent freezing of funds**.

---

### Likelihood Explanation

The most realistic trigger is Condition 1: a user or integrating contract passes an invalid NEAR account ID (e.g., empty `bytes`, a string exceeding 64 characters, or bytes containing characters outside `[a-z0-9_\-.]`). This is a plausible user error, especially for integrators constructing the `recipient` argument programmatically. Condition 3 is also reachable if a custom contract inheriting `EvmErc20` is deployed outside the engine's factory flow.

---

### Recommendation

Check the return value of the assembly `call` and revert if it is 0:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

Apply the same fix to `withdrawToEthereum`. Alternatively, validate the recipient before burning (e.g., enforce non-empty bytes and a maximum length matching NEAR account ID rules).
