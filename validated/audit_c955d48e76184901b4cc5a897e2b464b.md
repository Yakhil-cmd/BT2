### Title
Attacker-Controlled `puzzleHash` in Blockchain Notification Redirects Counter-Offer Notification to Attacker Address — (`packages/gui/src/components/offers2/OfferIncomingTable.tsx`)

### Summary

An attacker can craft an on-chain notification with an arbitrary `ph` (puzzleHash) field. The GUI trusts this value without any validation, converts it to a bech32m address, and pre-populates it as the **disabled, non-editable** notification destination in `NotificationSendDialog`. When the victim clicks "Counter" and then "Send Notification," their counter-offer notification (including the offer URL) is sent to the attacker's address instead of the legitimate offer creator's address.

---

### Finding Description

**Step 1 — Attacker-controlled entry point: `useBlockchainNotifications.tsx`**

The `prepareNotifications` callback decodes the on-chain message and extracts `puzzleHash` directly from the `ph` field with no validation — no check that it matches the actual coin sender, no cryptographic binding: [1](#0-0) 

The attacker simply broadcasts a coin spend with a message payload of `{"v":1,"t":1,"d":{"u":"<legitimate_offer_url>","ph":"<attacker_puzzle_hash>"}}` targeting the victim's address.

**Step 2 — No validation in `useValidNotifications`**

`useValidNotifications` only filters notifications by whether the associated offer is valid (`offerState.offer?.valid`). It performs no validation of the `puzzleHash` field: [2](#0-1) 

**Step 3 — `handleCounterOffer` converts attacker's puzzle hash to address**

When the user clicks "Counter" in `OfferIncomingTable`, `handleCounterOffer` extracts `puzzleHash` from the notification and calls `toBech32m` on it, producing the attacker's bech32m address. This address is placed into React Router navigation state: [3](#0-2) 

**Step 4 — `CreateOffer` passes attacker address to `OfferShareDialog`**

`CreateOffer` reads `locationState.address` (the attacker's address) and passes it to `CreateOfferBuilder`, which passes it back through `onOfferCreated`. `handleOfferCreated` then passes it to `OfferShareDialog`: [4](#0-3) 

**Step 5 — `OfferShareDialog` uses attacker address as `notificationDestination`**

Each sharing service dialog (Dexie, Spacescan, Offerpool) sets `notificationDestination = address || nftId`, where `address` is the attacker's address: [5](#0-4) 

**Step 6 — `NotificationSendDialog` locks the attacker address in a disabled field**

`NotificationSendDialog` receives the attacker's address as `destination`, sets it as the default form value, and renders the address `TextField` with `disabled`, preventing the user from correcting it: [6](#0-5) [7](#0-6) 

**Step 7 — `sendNotification` targets the attacker's puzzle hash**

On submit, `fromBech32m(address)` converts the attacker's address back to their puzzle hash, which is used as the `target` of the on-chain `sendNotification` RPC call: [8](#0-7) 

---

### Impact Explanation

The user's counter-offer notification — containing the counter-offer URL and spending ~0.0001 XCH in notification fees — is sent to the attacker's address instead of the legitimate offer creator's address. The address field is `disabled`, so the user has no opportunity to correct it. The legitimate offer creator never receives the counter-offer notification. The attacker receives the counter-offer URL and can use it to monitor, front-run, or disrupt the negotiation. This fits the High impact category: unsafe trust of notification state causes a user to **send** to the **wrong destination**.

---

### Likelihood Explanation

The Chia on-chain notification mechanism is a documented, production feature. Any unprivileged actor can broadcast a coin spend with an arbitrary `ph` field targeting any victim address. No special access is required. The victim only needs to click "Counter" on the incoming notification and then "Send Notification" after sharing the counter-offer — a normal user workflow.

---

### Recommendation

In `useBlockchainNotifications.tsx`, do not trust the `ph` field as the counter-offer reply address without verification. At minimum, display the decoded puzzle hash to the user and require explicit confirmation before using it as a notification destination. In `NotificationSendDialog`, the address field should be **editable** (not `disabled`) so the user can correct a pre-populated destination before sending. Ideally, the GUI should warn the user that the reply address comes from the notification sender and cannot be cryptographically verified.

---

### Proof of Concept

1. Attacker generates their own Chia wallet address and obtains their puzzle hash `<attacker_ph>`.
2. Attacker crafts a notification message: `{"v":1,"t":1,"d":{"u":"https://dexie.space/offers/<legitimate_offer>","ph":"<attacker_ph>"}}`, hex-encodes it, and broadcasts a coin spend targeting the victim's puzzle hash with this message.
3. Victim opens the GUI; the notification appears in `OfferIncomingTable` with a "Counter" button enabled (because `puzzleHash` is non-empty).
4. Victim clicks "Counter" → `handleCounterOffer` calls `toBech32m("<attacker_ph>", "xch")` → navigates to `/dashboard/offers/builder` with `state.address = "<attacker_xch_address>"`.
5. Victim builds and submits the counter-offer → `OfferShareDialog` opens with `address = "<attacker_xch_address>"`.
6. Victim shares on Dexie → clicks "Send Notification" → `NotificationSendDialog` opens with the attacker's address pre-filled in a **disabled** field.
7. Victim clicks send → `sendNotification({ target: fromBech32m("<attacker_xch_address>"), ... })` is called → counter-offer notification (with offer URL) is delivered to the attacker, not the legitimate offer creator.
