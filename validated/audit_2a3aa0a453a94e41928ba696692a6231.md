### Title
`poolAdminFeeDestination` Not Reset on Pool Admin Transfer Allows Previous Admin to Permanently Redirect Admin Fees - (File: metric-core/contracts/MetricOmmPoolFactory.sol)

---

### Summary

When pool admin ownership is transferred via `acceptPoolAdmin`, the factory's `poolAdminFeeDestination[pool]` mapping is not cleared. An outgoing admin can set the fee destination to their own address immediately before initiating the transfer. After the new admin accepts, the permissionless `collectPoolFees` path continues to send all admin fees to the previous admin's address indefinitely.

---

### Finding Description

`acceptPoolAdmin` in `MetricOmmPoolFactory` performs only two state mutations:

```solidity
// MetricOmmPoolFactory.sol lines 518-526
function acceptPoolAdmin(address pool) external override nonReentrant {
    address pending = pendingPoolAdmin[pool];
    if (pending == address(0)) revert NoPendingPoolAdminTransfer();
    if (msg.sender != pending) revert NotPendingPoolAdmin(pool, msg.sender, pending);
    address previousAdmin = poolAdmin[pool];
    poolAdmin[pool] = pending;          // ← only this changes
    delete pendingPoolAdmin[pool];      // ← and this
    emit PoolAdminTransferred(pool, previousAdmin, pending);
}
``` [1](#0-0) 

It does **not** reset `poolAdminFeeDestination[pool]`. The outgoing admin can call `setPoolAdminFeeDestination(pool, selfAddress)` at any point before the transfer, which is a fully valid admin action within caps:

```solidity
// MetricOmmPoolFactory.sol lines 438-447
function setPoolAdminFeeDestination(address pool, address newAdminFeeDestination)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (newAdminFeeDestination == address(0)) revert InvalidAdminFeeDestination();
    poolAdminFeeDestination[pool] = newAdminFeeDestination;
    ...
}
``` [2](#0-1) 

After the transfer, `collectPoolFees` is **permissionless** and reads the stale destination:

```solidity
// MetricOmmPoolFactory.sol lines 379-389
function collectPoolFees(address pool) external override nonReentrant {
    PoolFeeConfig memory c = poolFeeConfig[pool];
    IMetricOmmPoolCollectFees(pool).collectFees(
        c.protocolSpreadFeeE6, c.adminSpreadFeeE6,
        c.protocolNotionalFeeE8, c.adminNotionalFeeE8,
        poolAdminFeeDestination[pool]   // ← still points to old admin
    );
}
``` [3](#0-2) 

The pool's `collectFees` then transfers both token0 and token1 admin shares directly to that stale address: [4](#0-3) 

A second stale-state issue also exists: `pendingPriceProvider[pool]` and `pendingPriceProviderExecuteAfter[pool]` are also not cleared on admin transfer. If the outgoing admin had an in-flight provider proposal, the new admin inherits it silently and could unknowingly execute it via `executePoolPriceProviderUpdate` (which is `onlyPoolAdmin`). This is lower severity since execution requires an active call by the new admin, but it is a hidden state trap. [5](#0-4) 

---

### Impact Explanation

All admin fee revenue (spread and notional components) for the pool is permanently diverted to the previous admin's address until the new admin discovers the issue and calls `setPoolAdminFeeDestination`. Because `collectPoolFees` is permissionless, the old admin or any keeper can trigger collection at any time, draining fees before the new admin can react. This is a direct, ongoing loss of protocol fees owed to the new admin.

---

### Likelihood Explanation

The outgoing admin must act maliciously before the transfer. However, the action (`setPoolAdminFeeDestination`) is entirely within their normal, uncapped authority and leaves no on-chain signal visible to the incoming admin at accept time. The new admin has no mechanism to enumerate or audit the fee destination atomically with `acceptPoolAdmin`. The scenario is realistic in any adversarial or competitive admin handover.

---

### Recommendation

In `acceptPoolAdmin`, reset the fee destination to the new admin's address (or `address(0)` to force the new admin to set it explicitly), and clear any pending price provider state:

```solidity
function acceptPoolAdmin(address pool) external override nonReentrant {
    address pending = pendingPoolAdmin[pool];
    if (pending == address(0)) revert NoPendingPoolAdminTransfer();
    if (msg.sender != pending) revert NotPendingPoolAdmin(pool, msg.sender, pending);
    address previousAdmin = poolAdmin[pool];
    poolAdmin[pool] = pending;
    delete pendingPoolAdmin[pool];
    // Reset stale state from previous admin
    delete poolAdminFeeDestination[pool];          // force new admin to set destination
    delete pendingPriceProvider[pool];             // clear any in-flight oracle proposal
    delete pendingPriceProviderExecuteAfter[pool];
    emit PoolAdminTransferred(pool, previousAdmin, pending);
}
```

Alternatively, require the incoming admin to supply a `newFeeDestination` parameter when calling `acceptPoolAdmin`.

---

### Proof of Concept

```
1. Pool is live; oldAdmin is poolAdmin[pool].

2. oldAdmin calls:
     factory.setPoolAdminFeeDestination(pool, address(oldAdmin))
   → poolAdminFeeDestination[pool] = oldAdmin  (valid admin action, no cap)

3. oldAdmin calls:
     factory.proposePoolAdminTransfer(pool, newAdmin)

4. newAdmin calls:
     factory.acceptPoolAdmin(pool)
   → poolAdmin[pool] = newAdmin
   → poolAdminFeeDestination[pool] still = oldAdmin  ← BUG

5. Swaps accrue fees on the pool.

6. Anyone calls:
     factory.collectPoolFees(pool)
   → pool.collectFees(..., adminFeeDestination = oldAdmin)
   → token0/token1 admin shares transferred to oldAdmin

7. newAdmin receives zero admin fees until they discover and fix the destination.
   oldAdmin can front-run any fix by calling collectPoolFees before newAdmin's
   setPoolAdminFeeDestination transaction confirms.
```
