### Title
EIP-7702 Authorization List Ordering Manipulation Allows Transaction Sender to Control Final Delegation State - (`File: basic_bootloader/src/bootloader/transaction/authorization_list.rs`)

### Summary

`parse_authorization_list_and_apply_delegations` processes EIP-7702 authorization entries strictly in the order they appear in the transaction, with no deduplication or ordering enforcement for entries sharing the same authority address. Because the transaction sender (an unprivileged third party) controls the ordering of pre-signed authorization entries, they can choose the permutation that produces the most damaging final delegation state for the authority — a direct analog to the PermitC unordered-nonce MEV risk.

### Finding Description

`parse_authorization_list_and_apply_delegations` iterates over every entry in the authorization list and calls `validate_and_apply_delegation` for each one in sequence. [1](#0-0) 

Inside `validate_and_apply_delegation`, step 6 checks that the authority's current nonce equals the entry's `auth_nonce`: [2](#0-1) 

If the check passes, step 8 sets the delegation and step 9 increments the authority's nonce by 1: [3](#0-2) 

Because each successful entry bumps the nonce by exactly 1, a transaction sender who holds two valid pre-signed entries for the same authority — `Auth_A` (nonce=N, delegate=X) and `Auth_B` (nonce=N+1, delegate=Y) — can choose either ordering:

| Auth list order | Auth_A result | Auth_B result | Final delegation |
|---|---|---|---|
| `[A, B]` | succeeds (nonce N→N+1) | succeeds (nonce N+1→N+2) | Y |
| `[B, A]` | — | fails (nonce mismatch) | X |
| `[A]` only | succeeds | omitted | X |

The transaction sender picks whichever permutation (or subset) produces the outcome most beneficial to them.

### Impact Explanation

**Vulnerability class:** State-transition bug / MEV risk from signature-ordering manipulation.

An authority (Alice) who signs two authorization entries — e.g., `Auth_A` to delegate to a trusted contract and `Auth_B` to subsequently revoke — and hands both to a transaction sender cannot guarantee the final delegation state. A malicious transaction sender can:

1. Include only `Auth_A` → Alice remains delegated to the contract, revocation never applied.
2. Include `[Auth_B, Auth_A]` → `Auth_B` fails (nonce mismatch), `Auth_A` succeeds → Alice ends up delegated to the contract, not revoked.
3. Front-run Alice's own revocation transaction by submitting a transaction containing only `Auth_A` first, consuming nonce N; Alice's subsequent revocation (`Auth_B` at nonce N+1) then succeeds but the window of exposure already occurred.

The most severe scenario is keeping an authority delegated to a malicious or compromised contract when the authority intended to revoke, enabling the malicious contract to act on behalf of the authority's EOA (executing arbitrary calls with `msg.sender = authority`).

### Likelihood Explanation

**Medium.** The attack requires:
1. An authority to sign two or more authorization entries (a realistic pattern for "set then revoke" or "upgrade delegation" workflows).
2. A transaction sender who is distinct from the authority and is willing to choose a damaging ordering.

Both conditions are realistic in production: EIP-7702 is explicitly designed for third-party transaction senders to include authorization entries on behalf of authorities. The authorization entries are valid as soon as they are signed and can be observed in any mempool or off-chain channel.

### Recommendation

1. **Enforce last-entry-wins deduplication within a single transaction**: after processing all entries, if the same authority appears more than once, only the last successful entry's delegation should be committed. This matches the spirit of EIP-7702 while removing the ordering-dependent intermediate states.
2. **Alternatively, document clearly** that authorities must never sign more than one authorization entry per transaction context, and that the transaction sender controls ordering.
3. Consider adding a check in `parse_authorization_list_and_apply_delegations` that detects duplicate authority addresses and either rejects the transaction or processes only the last entry for each authority.

### Proof of Concept

```
Setup:
  Alice's current nonce = 0
  Alice signs Auth_A: { chain_id, nonce=0, delegate=ContractX, sig_A }
  Alice signs Auth_B: { chain_id, nonce=1, delegate=0x0 (revoke), sig_B }
  Alice gives both to Bob (transaction sender) expecting revocation.

Attack (Bob is malicious):
  Bob submits EIP-7702 tx with authorization_list = [Auth_A]
  (omits Auth_B entirely)

Execution in validate_and_apply_delegation:
  Entry Auth_A: authority.nonce(0) == auth_nonce(0) → PASS
    → set_delegation(Alice, ContractX)   [line 186-190]
    → increment_nonce(Alice, 1)          [line 191-205]

Result:
  Alice.nonce = 1
  Alice.code  = 0xef0100 || ContractX   ← delegation active

Alice's subsequent revocation attempt (Auth_B, nonce=1) would now
succeed if submitted, but the window of exposure has already occurred:
ContractX can act as Alice between Bob's tx and Alice's revocation tx.

Alternatively, Bob submits [Auth_B, Auth_A]:
  Auth_B: authority.nonce(0) != auth_nonce(1) → FAIL (skipped)
  Auth_A: authority.nonce(0) == auth_nonce(0) → PASS → delegates to X
  Alice ends up delegated to X, revocation never applied.

```rust
pub fn parse_authorization_list_and_apply_delegations<S: EthereumLikeTypes>(
    system: &mut System<S>,
    resources: &mut S::Resources,
    auth_list: AuthorizationList<'_>,
) -> Result<(), TxError>
where
    S::IO: IOSubsystemExt,
{
    use crate::bootloader::transaction::rlp_encoded::AuthorizationEntry;
    let mut hasher = crypto::sha3::Keccak256::new();

    for entry in auth_list.iter() {
        let AuthorizationEntry {
            chain_id,
            address,
            nonce,
            y_parity,
            r,
            s,
        } = entry;
        let success = validate_and_apply_delegation(
            system,
            resources,
            &chain_id,
            nonce,
            address,
            (y_parity, r, s),
            &mut hasher,
        )?;
        system_log!(system, "Delegation success: {success}\n");

        if !success {}
    }
    Ok(())
}
