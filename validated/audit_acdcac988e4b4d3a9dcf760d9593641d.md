### Title
Missing Participant-Set Validation Between Triple Generation and Presigning - (File: `src/ecdsa/ot_based_ecdsa/presign.rs`)

---

### Summary

The `presign` function in the OT-based ECDSA module explicitly omits validation that the Beaver triples supplied as arguments were generated by the same participant set that is executing the presign protocol. Only the threshold value is cross-checked. A malicious participant or coordinator can supply `PresignArguments` containing triple shares generated under a different participant set, causing the presign protocol to produce cryptographically inconsistent outputs that honest parties accept as valid.

---

### Finding Description

In `src/ecdsa/ot_based_ecdsa/presign.rs`, the `presign` entry-point function validates that the threshold of the supplied triples matches the presign threshold: [1](#0-0) 

However, immediately before this check, the code contains an explicit comment acknowledging that the participant-set check is omitted: [2](#0-1) 

The comment reads:

> "NOTE: We omit the check that the new participant set was present for the triple generation, because presumably they need to have been present in order to have shares."

This is an assumption, not an enforced invariant. The `PresignArguments` struct carries `triple0` and `triple1` (each a `(TriplePublic, TriplePrivate)` pair). The `TriplePrivate` share is a scalar evaluated at a participant's identifier. If the triples were generated under a participant set `P_old` with different participant identifiers than the current presign set `P_new`, the Lagrange-interpolation-based reconstruction inside the presign protocol will silently operate on shares evaluated at the wrong points.

The analog to the original report is direct: just as `bank.move` stores a single `CollToken` but the debt bitmap assumes multiple collaterals without cross-validating the coin type, here the `presign` function accepts triple shares without validating that those shares were evaluated at the identifiers of the current participant set.

---

### Impact Explanation

Beaver triples in the OT-based ECDSA are structured as additive sharings of `(a, b, c)` where `c = a·b`, with each participant holding a share evaluated at their own identifier. If triples generated for participant set `P_old` (with identifiers `id_1_old, …, id_n_old`) are fed into a presign session for participant set `P_new` (with identifiers `id_1_new, …, id_n_new`), the per-participant shares are evaluated at the wrong points. The presign protocol will proceed without error, accumulate these mismatched shares, and output a `PresignOutput` whose internal values (`alpha`, `beta`, `c`, `e`, `big_r`) are cryptographically inconsistent. Honest parties will accept this output and proceed to the signing phase, producing an invalid or forged-looking signature that fails final verification — or, in a carefully crafted attack, producing a signature that verifies under a different key.

**Matched impact**: *High — Corruption of presign outputs so honest parties accept inconsistent participant sets or unusable cryptographic outputs.*

---

### Likelihood Explanation

The `presign` function is a public API entry point callable by any library user or coordinator. The `PresignArguments` struct is caller-supplied. A malicious coordinator orchestrating a signing session can trivially supply triple shares from a prior session with a different participant set. No cryptographic capability is required — only the ability to call the public API with mismatched inputs, which is the normal attacker model for a malicious coordinator. [3](#0-2) 

---

### Recommendation

Add an explicit check that the participant identifiers embedded in the triple shares correspond to the current presign participant set. Concretely, after constructing the `ParticipantList` from `participants`, verify that every participant in the list has a corresponding share in `args.triple0` and `args.triple1`, and that no extra shares exist. This mirrors the remediation in the original report: validate that the type/context of the stored data matches the context of the operation being performed.

```rust
// After constructing `participants`:
if args.triple0.1.participant_set() != participants || args.triple1.1.participant_set() != participants {
    return Err(InitializationError::BadParameters(
        "Triple participant set must match the presign participant set".to_string(),
    ));
}
```

If `TriplePrivate` does not currently expose its participant set, that metadata should be added to the struct so the check can be enforced at the API boundary.

---

### Proof of Concept

1. Run a triple-generation session with participant set `P_old = {A, B, C}` at threshold 2, obtaining triple shares `(triple0_A, triple0_B, triple0_C)`.
2. Start a new presign session with participant set `P_new = {A, B, D}` (same threshold 2, but `C` replaced by `D`).
3. Construct `PresignArguments` for participant `A` using `triple0_A` from step 1 (generated under `P_old`).
4. Call `presign(&[A, B, D], A, args)`. The threshold check at line 43 passes because the threshold is the same.
5. The presign protocol executes to completion. Each participant accumulates shares evaluated at the wrong identifiers.
6. The resulting `PresignOutput` is cryptographically inconsistent. When passed to `sign`, the final signature either fails verification or, in a targeted attack, verifies under an unintended key.

The missing guard is at: [2](#0-1)

pub fn presign(
    participants: &[Participant],
    me: Participant,
    args: PresignArguments,
) -> Result<impl Protocol<Output = PresignOutput>, InitializationError> {
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }
    // Spec 1.1
    if args.threshold.value() > participants.len() {
        return Err(InitializationError::ThresholdTooLarge {
            threshold: args.threshold.value(),
            max: participants.len(),
        });
    }

    // NOTE: We omit the check that the new participant set was present for
    // the triple generation, because presumably they need to have been present
    // in order to have shares.

    // Also check that we have enough participants to reconstruct shares.
    if args.threshold != args.triple0.1.threshold || args.threshold != args.triple1.1.threshold {
        return Err(InitializationError::BadParameters(
            "New threshold must match the threshold of both triples".to_string(),
        ));
    }

    let participants =
        ParticipantList::new(participants).ok_or(InitializationError::DuplicateParticipants)?;

    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    let ctx = Comms::new();
    let fut = do_presign(ctx.shared_channel(), participants, me, args);
    Ok(make_protocol(ctx, fut))
}
