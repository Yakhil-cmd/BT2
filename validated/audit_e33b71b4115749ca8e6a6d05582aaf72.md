Now I have a complete picture of the codebase. Let me synthesize the findings.

**Key observation from SwapMath.sol:** The `spreadFeeE6` parameter passed to each bin swap function is the *total* spread fee (protocol + admin). The protocol fee extracted per bin is `feeAmountScaled * spreadFeeE6 / 1e6`, and the remainder stays in the bin as LP fee. The surplus (actual balance − binTotals − notionalFees) is what `collectFees` splits between protocol and admin destinations.

**Critical asymmetry found:** Both `setPoolAdminFees` and `setPoolProtocolFee` call `collectFees` to the *current* `poolAdminFeeDestination[pool]` before updating any fee parameter. `setPoolAdminFeeDestination` changes that destination directly with no prior collection.

### Title
Pool admin silently redirects all accrued admin fees by changing fee destination without prior collection — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary
`setPoolAdminFeeDestination` updates `poolAdminFeeDestination[pool]` without first flushing accrued admin-fee surplus to the old destination. Every other fee-parameter mutator (`setPoolAdminFees`, `setPoolProtocolFee`) calls `collectFees` against the old destination before writing new state. The missing flush lets the pool admin silently redirect all previously accrued, uncollected admin fees to an address they control.

### Finding Description

`setPoolAdminFees` (line 408) and `setPoolProtocolFee` (line 318) both follow the same pattern: collect accrued fees to the *current* `poolAdminFeeDestination[pool]` first, then update the fee configuration. [1](#0-0) [2](#0-1) 

`setPoolAdminFeeDestination` (line 438) performs no such flush: [3](#0-2) 

The admin-fee surplus lives in the pool as the gap between actual token balances and tracked `binTotals + notionalFees`. It is only realized when `collectPoolFees` (or any `collectFees` call) is executed, and at that moment the factory reads the *current* `poolAdminFeeDestination[pool]`: [4](#0-3) 

So any surplus that accrued while the old destination was set is silently redirected to the new destination the next time fees are collected.

The `collectFees` function in the pool confirms that `adminFeeDestination_` is a caller-supplied parameter — the pool itself has no memory of the "correct" destination: [5](#0-4) 

### Impact Explanation

The admin fee destination is explicitly decoupled from the pool admin address. It can be a DAO treasury, a revenue-sharing contract, or any third-party beneficiary set at pool creation via `params.adminFeeDestination`. A pool admin who is distinct from that beneficiary can:

1. Allow admin fees to accumulate (spread surplus grows with every swap).
2. Call `setPoolAdminFeeDestination(pool, attackerWallet)` — zero fees collected, destination silently updated.
3. Call `collectPoolFees(pool)` — entire accumulated admin-fee surplus is transferred to `attackerWallet`.

The old beneficiary receives nothing for the entire accrual period. The loss scales linearly with time between collections and pool volume. Protocol fees (sent to `FACTORY`) are unaffected; only the admin-fee share is redirected.

### Likelihood Explanation

The pool admin is semi-trusted but has a direct financial incentive. The attack requires a single permissioned call (`setPoolAdminFeeDestination`) followed by a permissionless call (`collectPoolFees`). No timelock, no quorum, no cap check guards this path. The asymmetry is invisible to the old fee destination because no event signals that a collection was skipped.

### Recommendation

Mirror the pattern used by `setPoolAdminFees` and `setPoolProtocolFee`: flush accrued fees to the old destination before writing the new one.

```solidity
function setPoolAdminFeeDestination(address pool, address newAdminFeeDestination)
    external
    override
    nonReentrant
    onlyPoolAdmin(pool)
{
    if (newAdminFeeDestination == address(0)) revert InvalidAdminFeeDestination();
    // Flush accrued fees to old destination before changing it
    PoolFeeConfig memory c = poolFeeConfig[pool];
    IMetricOmmPoolCollectFees(pool).collectFees(
        c.protocolSpreadFeeE6,
        c.adminSpreadFeeE6,
        c.protocolNotionalFeeE8,
        c.adminNotionalFeeE8,
        poolAdminFeeDestination[pool]   // old destination
    );
    poolAdminFeeDestination[pool] = newAdminFeeDestination;
    emit PoolAdminFeeDestinationUpdated(pool, newAdminFeeDestination);
}
```

### Proof of Concept

```
Setup:
  pool deployed with adminFeeDestination = treasury (0xTREASURY)
  poolAdmin = 0xADMIN (separate from treasury)

Step 1 – fees accrue:
  Many swaps execute; spread surplus builds up in pool.
  collectPoolFees is NOT called (admin withholds collection).

Step 2 – destination swap (single tx by 0xADMIN):
  factory.setPoolAdminFeeDestination(pool, 0xATTACKER)
  // No collectFees call; poolAdminFeeDestination[pool] = 0xATTACKER

Step 3 – drain (permissionless):
  factory.collectPoolFees(pool)
  // collectFees(..., poolAdminFeeDestination[pool])
  //   = collectFees(..., 0xATTACKER)
  // All accrued admin-fee surplus transferred to 0xATTACKER

Result:
  0xTREASURY receives 0 for the entire accrual window.
  0xATTACKER receives the full admin-fee surplus.
  Protocol fees (→ FACTORY) are unaffected.
```

The root cause is the missing `collectFees` call in `setPoolAdminFeeDestination` [3](#0-2) 
while both peer mutators [6](#0-5) [7](#0-6) 
enforce the flush-before-update invariant.
