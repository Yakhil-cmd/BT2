### Title
Missing Lower-Bound Threshold Validation in `assert_sign_inputs` Allows Signing with Cryptographically Insufficient Threshold - (File: `src/frost/mod.rs`)

---

### Summary

The `assert_sign_inputs` function in `src/frost/mod.rs` validates only the upper bound of the `threshold` parameter (`threshold > participants.len()`), but omits the lower-bound check (`threshold ≥ 2`) that is consistently enforced in `assert_key_invariants` in `src/dkg.rs`. Because `ReconstructionLowerBound` is a plain `usize` newtype with no constructor-level minimum enforcement, a caller can pass `threshold = 0` or `threshold = 1` to FROST signing functions. The signing protocol then proceeds with a cryptographically insufficient threshold, producing corrupted or unusable signature outputs.

---

### Finding Description

`ReconstructionLowerBound` in `src/thresholds.rs` is defined as a newtype over `usize` with a `#[derive(From)]` blanket conversion: [1](#0-0) 

This means `ReconstructionLowerBound::from(0usize)` and `ReconstructionLowerBound::from(1usize)` are both valid at the type level with no runtime rejection.

`assert_key_invariants` in `src/dkg.rs` correctly enforces a minimum of 2: [2](#0-1) 

However, `assert_sign_inputs` in `src/frost/mod.rs` only checks the upper bound and is entirely missing the symmetric lower-bound guard: [3](#0-2) 

The same gap exists in the OT-based ECDSA `sign` function, which checks `participants.len() < threshold` but never checks `threshold < 2`: [4](#0-3) 

The FROST `presign` function has the identical omission: [5](#0-4) 

---

### Impact Explanation

When `threshold = 0` or `threshold = 1` is accepted by `assert_sign_inputs`, the FROST signing protocol proceeds with a threshold that is inconsistent with the one used during key generation (which enforced `threshold ≥ 2`). The Lagrange interpolation during signing uses the participant identifiers and the caller-supplied threshold to reconstruct the signing contribution. With `threshold = 1`, only a single participant's share is used with a Lagrange coefficient of 1 (trivially), which does not correctly reconstruct the secret key polynomial's constant term that was generated under a degree-`(t-1)` polynomial with `t ≥ 2`. The resulting aggregate signature is cryptographically invalid and will fail verification — corrupting the sign output for all honest participants who accepted the protocol run.

**Allowed impact matched**: High — Corruption of sign outputs so honest parties accept inconsistent or unusable cryptographic outputs.

---

### Likelihood Explanation

The entry path is a direct library API call. Any unprivileged library caller who invokes the FROST `sign` or `presign` functions (e.g., `eddsa::sign`, `redjubjub::sign`) controls the `threshold` argument. A misconfigured integrator or a malicious participant acting as coordinator can trivially pass `threshold = 1` or `threshold = 0`. The `ReconstructionLowerBound` type provides no compile-time or runtime barrier against this. The inconsistency with `assert_key_invariants` (which does enforce the minimum) makes accidental misconfiguration likely in real deployments.

---

### Recommendation

1. Add a lower-bound check in `assert_sign_inputs` mirroring `assert_key_invariants`:

```rust
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

2. Apply the same guard to the FROST `presign` function and the OT-based ECDSA `sign` function.

3. Consider adding a validated constructor to `ReconstructionLowerBound` (e.g., `ReconstructionLowerBound::new(v: usize) -> Result<Self, _>`) that enforces `v ≥ 2` at the type level, removing the reliance on call-site checks.

---
