Audit Report

## Title
Sandbox RSS-Based Eviction Permanently Suppressed for Canisters with Large Stable Memory — (`rs/canister_sandbox/src/replica_controller/sandboxed_execution_controller.rs`)

## Summary
In `monitor_and_evict_sandbox_processes`, only anonymous RSS (`get_anon_rss`) is pushed into `sandbox_processes_rss` and subsequently stored in `stats.rss` per backend. The page-allocator RSS (`get_page_allocator_rss`), which accounts for canister and stable memory held in memfd or file-backed regions, is collected for metrics only and never stored. Because `total_sandboxes_rss` sums only `stats.rss` (anon-only), the conjunctive gate in `trigger_sandbox_eviction` — which requires `total_sandboxes_rss > max_sandboxes_rss` — is permanently false for any canister holding significant stable memory, disabling the RSS-based eviction path entirely for such canisters even under real node memory pressure.

## Finding Description
In `monitor_and_evict_sandbox_processes` (Linux path, lines 1398–1417):

```rust
if let Ok(kib) = process_os_metrics::get_anon_rss(pid) {
    // ...
    sandbox_processes_rss.push((*canister_id, bytes));  // stored for eviction
}
if let Ok(kib) = process_os_metrics::get_page_allocator_rss(pid) {
    total_memfd_rss += kib;
    process_rss += kib;
    metrics.sandboxed_execution_subprocess_memfd_rss.observe(kib as f64);
    // NOT pushed to sandbox_processes_rss
}
```

`update_sandbox_processes_rss` (lines 2009–2019) writes only the anon-only value into `stats.rss`. `total_sandboxes_rss` (lines 2022–2030) sums only `Backend::Active { stats, .. } => stats.rss`, which is anon-only. `trigger_sandbox_eviction` (lines 1542–1560) gates the RSS eviction branch on:

```rust
total_sandboxes_rss > self.max_sandboxes_rss()   // anon-only → always false for stable-memory canisters
&& available_memory().unwrap_or_default() < DEFAULT_MIN_MEM_AVAILABLE_TO_EVICT_SANDBOXES
```

For a canister with large stable memory: anon RSS ≈ 0, page-allocator RSS ≈ hundreds of GiB. Condition 1 is permanently false. The AND short-circuits. No RSS-based eviction fires. The monitor thread's own eviction call (lines 1482–1490) passes `max_sandboxes_rss = u64::MAX`, so it also performs no RSS-based eviction. The count-based and idle-time paths remain functional but are blind to per-canister memory magnitude.

## Impact Explanation
A canister with large stable memory can accumulate hundreds of GiB of physical RAM in memfd/file-backed regions on a node without triggering the RSS eviction mechanism. If the count of active sandboxes remains below `max_sandbox_count` and the canister is not idle, neither the count-based nor idle-time eviction paths engage. The node's available memory falls below the 250 GiB threshold (condition 2 becomes true), but condition 1 remains false, so the eviction gate never opens. The node is left to rely on the OS OOM killer rather than the designed IC eviction mechanism, risking replica process termination and loss of subnet availability. This matches the allowed High impact: **Application/platform-level DoS or subnet availability impact not based on raw volumetric DDoS**.

## Likelihood Explanation
Any unprivileged canister developer can call `stable_grow` and `stable_write` via the public IC System API to allocate and populate stable memory up to the protocol limit. No governance majority, subnet-majority corruption, privileged access, or infrastructure compromise is required. The condition is deterministic: anon RSS will always be near zero for a canister that stores data exclusively in stable memory, so condition 1 of the eviction gate is structurally false. The scenario is reproducible in a local replica environment and on mainnet application subnets.

## Recommendation
Accumulate both `get_anon_rss` and `get_page_allocator_rss` before pushing to `sandbox_processes_rss`. In `monitor_and_evict_sandbox_processes`, replace the two separate pushes with a single combined push:

```rust
let anon_kib = process_os_metrics::get_anon_rss(pid).unwrap_or(0);
let page_alloc_kib = process_os_metrics::get_page_allocator_rss(pid).unwrap_or(0);
let total_kib = anon_kib + page_alloc_kib;
sandbox_processes_rss.push((*canister_id, NumBytes::new(total_kib * 1024)));
```

This ensures `stats.rss` and `total_sandboxes_rss` reflect actual sandbox memory consumption including stable memory, allowing the eviction gate to fire correctly under memory pressure.

## Proof of Concept
1. Start a local replica with `default_subnet_heap_delta_capacity` configured so that `max_sandboxes_rss` (= capacity / 3) is, e.g., 100 MiB.
2. Deploy a canister that calls `stable_grow` to allocate ~200 MiB of stable memory and `stable_write` to fault in all pages (forcing physical RSS).
3. Keep Wasm heap allocation minimal (near-zero anon RSS).
4. Inspect `/proc/<sandbox_pid>/smaps` and confirm ~200 MiB of memfd or `.mem (deleted)` RSS with near-zero `RssAnon`.
5. Confirm via the `sandboxed_execution_subprocess_anon_rss` metric that `stats.rss` for the backend is near zero.
6. Confirm `total_sandboxes_rss` is below `max_sandboxes_rss` (condition 1 = false).
7. Mock `available_memory` to return a value below `DEFAULT_MIN_MEM_AVAILABLE_TO_EVICT_SANDBOXES` (condition 2 = true).
8. Call `trigger_sandbox_eviction` and assert no eviction occurs despite actual physical memory exceeding the configured limit. The existing unit test `sandbox_eviction_is_triggered_by_available_memory` can be extended with a backend whose `stats.rss` is set to anon-only (near zero) while the actual process holds large page-allocator RSS, demonstrating the gate failure.
