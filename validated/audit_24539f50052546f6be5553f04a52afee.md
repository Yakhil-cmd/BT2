### Title
Missing Caller Restriction on `update_rewards` Allows Any Address to Suppress Per-Block Rewards — (`src/staking/staking.cairo`)

### Summary

`update_rewards` is documented as "Only starkware sequencer" in the spec but the implementation enforces no such check. Any non-zero address can call it with `disable_rewards: true`, consuming the global `last_reward_block` slot for that block and permanently discarding all block rewards that would otherwise have been distributed.

### Finding Description

`StakingRewardsManagerImpl::update_rewards` is a public ABI entry point. [1](#0-0) 

Its only gate is `general_prerequisites`, which checks only that the contract is unpaused and the caller is non-zero: [2](#0-1) 

The spec explicitly states access should be restricted to the Starkware sequencer: [3](#0-2) 

After passing the `REWARDS_ALREADY_UPDATED` block-number guard, the function unconditionally writes `current_block_number` into the global `last_reward_block` storage variable: [4](#0-3) 

It then checks `disable_rewards || self.is_pre_consensus()` and returns early without distributing anything: [5](#0-4) 

Because `last_reward_block` is a single global value (not per-staker), this single early-return call permanently exhausts the reward slot for the entire block for every staker: [6](#0-5) 

Any subsequent call in the same block — including the legitimate sequencer call — reverts with `REWARDS_ALREADY_UPDATED`: [7](#0-6) 

### Impact Explanation

Block rewards are computed per-block and are never retroactively credited. A block whose `last_reward_block` slot is consumed with `disable_rewards: true` loses its rewards permanently. Repeating this every block causes **permanent freezing of unclaimed yield** for all stakers and delegators. This matches the High impact category in the allowed scope.

### Likelihood Explanation

The call requires only a valid (active, non-zero-balance) staker address as argument — trivially obtained from on-chain events or `get_stakers`. The attacker pays only gas. On Starknet the sequencer controls ordering, but the sequencer is not guaranteed to place its own `update_rewards` invocation as the first transaction of every block; any user transaction included before it in a block is sufficient to trigger the suppression. A sustained griefing campaign requires one cheap transaction per block.

### Recommendation

Add a sequencer-only caller check at the top of `update_rewards`, consistent with the spec. For example, assert `get_caller_address() == sequencer_address` where `sequencer_address` is a stored, governance-controlled value, or reuse whatever role-based guard is used for other privileged operations in the contract.

### Proof of Concept

1. Deploy the staking contract and advance past the consensus rewards start epoch.
2. Stake as `staker_A` and wait K epochs for balance to become effective.
3. From an arbitrary EOA, call `update_rewards(staker_address: staker_A, disable_rewards: true)`.
4. Observe: call succeeds, `last_reward_block` is set to the current block, `staker_A.unclaimed_rewards_own` is unchanged.
5. Now call `update_rewards(staker_address: staker_A, disable_rewards: false)` (simulating the sequencer) — it reverts with `REWARDS_ALREADY_UPDATED`.
6. Advance one block and repeat step 3 every block indefinitely.
7. After N blocks, `staker_A.unclaimed_rewards_own` remains zero despite N blocks of eligible consensus rewards having elapsed.

This is directly exercised by the existing test pattern at: [8](#0-7) 

which already demonstrates that a `disable_rewards: true` call followed by a `disable_rewards: false` call in the same block reverts — confirming the suppression mechanism is fully reachable.
