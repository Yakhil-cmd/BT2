### Title
Unrestricted `update_rewards` with `disable_rewards: true` Enables Permanent Freezing of Staker Unclaimed Yield — (File: `src/staking/staking.cairo`)

---

### Summary

The `IStakingRewardsManager::update_rewards` function in the Staking contract carries no role-based access control. Any unprivileged caller can invoke it with `disable_rewards: true`, which consumes the single per-block "reward slot" (by writing `last_reward_block`) without distributing any rewards. Because the function enforces a strict one-call-per-block invariant, a sustained attacker can permanently prevent all stakers from accumulating consensus-phase rewards.

---

### Finding Description

`update_rewards` is the entry point through which per-block consensus rewards are distributed to stakers and their delegation pools. Its only gate is `general_prerequisites()`, which checks two things: the contract is not paused, and the caller is not the zero address. [1](#0-0) 

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
    let current_block_number = starknet::get_block_number();
    assert!(
        current_block_number > self.last_reward_block.read(),
        "{}",
        Error::REWARDS_ALREADY_UPDATED,
    );
``` [2](#0-1) 

`general_prerequisites` only enforces:

```cairo
fn general_prerequisites(ref self: ContractState) {
    self.assert_is_unpaused();
    assert_caller_is_not_zero();
}
```

No role check, no allowlist, no caller-must-be-sequencer guard.

The function then writes the current block number into the **global** `last_reward_block` field before checking `disable_rewards`: [3](#0-2) 

```cairo
// Update last block rewards.
self.last_reward_block.write(current_block_number);

if disable_rewards || self.is_pre_consensus() {
    return;
}
```

Because `last_reward_block` is global (not per-staker), a single call with `disable_rewards: true` blocks **all** reward distribution for that block. Any subsequent call in the same block fails with `REWARDS_ALREADY_UPDATED`.

The `disable_rewards` parameter is part of the public `IStakingRewardsManager` interface with no access restriction: [4](#0-3) 

```cairo
#[starknet::interface]
pub trait IStakingRewardsManager<TContractState> {
    /// Update current block rewards for the given `staker_address`.
    /// Distribute rewards only if `disable_rewards` is `false` and consensus rewards already
    /// started.
    fn update_rewards(
        ref self: TContractState, staker_address: ContractAddress, disable_rewards: bool,
    );
}
```

The contrast with `update_rewards_from_attestation_contract`, which **does** enforce `assert_caller_is_attestation_contract()`, confirms that the missing guard on `update_rewards` is an oversight, not intentional design: [5](#0-4) 

---

### Impact Explanation

**High — Permanent freezing of unclaimed yield.**

Per-block rewards are calculated from the current epoch's `block_rewards` value and are not retroactively accumulated for missed blocks. Each block where `update_rewards` is called with `disable_rewards: true` results in permanently lost rewards for every staker and every delegation pool member. If an attacker sustains the attack across all blocks, stakers and pool members can never accumulate any consensus-phase yield. [6](#0-5) 

---

### Likelihood Explanation

**Medium.**

The attacker must supply a valid, active staker address with non-zero balance (easily obtained from on-chain events) and call `update_rewards` before the legitimate block-proposer call in every block. On Starknet the sequencer controls ordering, which is a partial mitigation, but the function is unconditionally public with no on-chain enforcement. A single missed block by the sequencer's ordering logic, or a sequencer that is indifferent to ordering, is sufficient for the attack to succeed. The gas cost per call on Starknet is low, making sustained execution economically viable.

---

### Recommendation

1. **Add a role guard** to `update_rewards` restricting callers to a designated `REWARDS_MANAGER` role (analogous to how `update_rewards_from_attestation_contract` restricts to the attestation contract).
2. **Remove `disable_rewards` from the public interface** or move it behind a privileged function so that only an authorized operator can suppress reward distribution.
3. Consider making `last_reward_block` per-staker rather than global, so that a call for one staker cannot block reward distribution for all others.

---

### Proof of Concept

1. Attacker reads any active staker address from `NewStaker` events.
2. In every block, attacker submits `update_rewards(valid_staker, disable_rewards: true)`.
3. `last_reward_block` is set to the current block number; no rewards are distributed.
4. Any legitimate call to `update_rewards` in the same block reverts with `REWARDS_ALREADY_UPDATED`.
5. Stakers' `unclaimed_rewards_own` and pool members' cumulative reward index are never updated.
6. `claim_rewards` and `Pool::claim_rewards` return zero for all participants indefinitely.
