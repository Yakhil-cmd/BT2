### Title
`SendToAccountHandler` Bypasses `IsSendEnabledCoins` Governance Check That `BankContract.Transfer` Enforces — (`x/cronos/keeper/evmhandlers/send_to_account.go`)

---

### Summary

`SendToAccountHandler.Handle` calls `bankKeeper.SendCoins` directly without first calling `IsSendEnabledCoins`, while `BankContract.Transfer` (the bank precompile) explicitly calls `IsSendEnabledCoins` before `SendCoins`. Because the Cosmos SDK's `SendCoins` does **not** internally enforce `IsSendEnabledCoins`, the `__CronosSendToAccount` EVM log path bypasses the governance-controlled send-enabled restriction entirely.

---

**Path 2 — `SendToAccountHandler.Handle` (no check):**

`x/cronos/keeper/evmhandlers/send_to_account.go` line 84 calls `SendCoins` with no `IsSendEnabledCoins` guard:

```go
err = h.bankKeeper.SendCoins(ctx, contractAddr, recipient, coins)
``` [2](#0-1) 

**Structural reason the check is absent:**

The `types.BankKeeper` interface used by `SendToAccountHandler` (and the entire `x/cronos` keeper layer) does not include `IsSendEnabledCoins` at all: [3](#0-2) 

`IsSendEnabledCoins` only appears in the ethermint `evmtypes.BankKeeper` interface, which is what `BankContract` uses. The handler structurally cannot call it.

**Why `SendCoins` alone does not enforce the restriction:**

In the Cosmos SDK bank keeper, `SendCoins` does not call `IsSendEnabledCoins` internally. That check lives at the message-server level (`MsgSend` handler). Calling `SendCoins` directly bypasses it entirely.

---

### Impact Explanation

If governance sets `send_enabled = false` for a denom (e.g., to freeze a compromised or paused token):

- `BankContract.Transfer` → **blocked** (returns error at `IsSendEnabledCoins`)
- `__CronosSendToAccount` EVM log → **succeeds** (skips the check, `SendCoins` proceeds)

Any user holding CRC20 tokens mapped to a frozen native denom can call the CRC20 contract, trigger the `__CronosSendToAccount` log, and receive native tokens despite the governance freeze. This is a direct bypass of governance authority over token send restrictions.

Impact category: **High — Bypass of governance authority / permission check** over native token transfer paths.

---

### Likelihood Explanation

- The `send_enabled` parameter is a standard Cosmos SDK governance mechanism. Cronos governance can set it at any time (e.g., emergency token freeze).
- The attacker only needs to call a registered CRC20 contract (no special privileges). Any contract registered via `SetExternalContractForDenom` or `SetAutoContractForDenom` qualifies.
- The bypass is unconditional — no race condition, no timing dependency.

---

### Recommendation

Add `IsSendEnabledCoins` to the `types.BankKeeper` interface in `x/cronos/types/interfaces.go`, then call it in `SendToAccountHandler.Handle` before invoking `bankKeeper.SendCoins`, mirroring the pattern in `BankContract.Transfer`.

The same gap exists in `SendToIbcHandler` and `SendCroToIbcHandler` (both call `bankKeeper.SendCoins` without `IsSendEnabledCoins`), and should be fixed consistently across all EVM log handlers.

---

### Proof of Concept

1. Deploy a CRC20 contract and register it for a native denom via `SetExternalContractForDenom`.
2. Fund the contract address with native tokens of that denom.
3. Via governance, set `send_enabled = false` for that denom.
4. Attempt `BankContract.Transfer` for that denom → expect failure (`send_enabled` check fires).
5. Call the CRC20 contract to emit `__CronosSendToAccount(recipient, amount)`.
6. Observe that `SendToAccountHandler.Handle` succeeds and the recipient receives native tokens despite the governance freeze.
7. Assert: step 4 fails, step 6 succeeds — the invariant is violated. [4](#0-3) [5](#0-4)
