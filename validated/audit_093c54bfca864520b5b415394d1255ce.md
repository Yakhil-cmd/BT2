### Title
`skip_stateful_validations` Bypasses Signature Verification for Invoke Transactions, Allowing Admission of Unsigned Transactions to the Mempool - (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` UX feature in the gateway's stateful validator unconditionally skips the `__validate__` entry-point call (which performs signature verification) for any invoke transaction with `nonce == 1` sent from an address that already appears in the mempool. An attacker who observes a legitimate `deploy_account` transaction in the mempool can immediately submit a second invoke transaction from the same address with a garbage signature, and the gateway will admit it without any signature check. If the attacker's transaction carries a higher tip, it replaces the victim's legitimate invoke via fee escalation, causing the victim's transaction to be lost and their account to be charged fees for a reverted transaction.

### Finding Description

In `crates/apollo_gateway/src/stateful_transaction_validator.rs`, the function `skip_stateful_validations` (lines 429–461) implements a UX shortcut: when an invoke transaction has `nonce == 1` and the on-chain account nonce is `0`, it queries the mempool for whether the sender address is already known (`account_tx_in_pool_or_recent_block`). If the address is present, the function returns `true` (skip validation). [1](#0-0) 

This return value is consumed by `run_validate_entry_point`, which sets `execution_flags.validate = !skip_validate`. When `skip_validate == true`, the `ExecutionFlags` are constructed with `validate = false`: [2](#0-1) 

With `validate = false`, the blockifier's `validate_tx` immediately returns `Ok(None)` without calling the account contract's `__validate__` entry point: [3](#0-2) 

The `account_tx_in_pool_or_recent_block` check in the mempool is a simple address-presence test — it does **not** verify that the matching transaction is specifically a `deploy_account`: [4](#0-3) 

The `validate_by_mempool` call that precedes `skip_stateful_validations` only checks for duplicate hashes and nonce ordering — it performs no signature verification: [5](#0-4) 

**Attack sequence:**

1. Alice submits `deploy_account(address=A, nonce=0)` + `invoke(address=A, nonce=1, calldata=legitimate, tip=T)`.
2. The gateway processes `deploy_account` first; it passes all checks and is added to the mempool. Address `A` is now tracked in `tx_pool.contains_account`.
3. The attacker observes this and immediately submits `invoke(address=A, nonce=1, calldata=malicious, signature=garbage, tip=T+1)`.
4. The gateway processes the attacker's invoke:
   - `validate_nonce`: `account_nonce=0`, `tx_nonce=1`, within `max_allowed_nonce_gap` → passes.
   - `validate_by_mempool`: no duplicate hash, nonce not too old → passes.
   - `skip_stateful_validations`: `tx.nonce()==1 && account_nonce==0 && A in mempool` → returns `true`.
   - `run_validate_entry_point`: `validate=false` → `__validate__` is **never called**; garbage signature is never checked.
5. The attacker's invoke is admitted to the mempool. Because `tip=T+1 > T`, fee escalation replaces Alice's legitimate invoke with the attacker's.
6. During block execution, `deploy_account(A)` executes first. Then the attacker's invoke executes with `execution_flags.validate=true` (default for batcher execution), `__validate__` is called, the garbage signature fails, and the transaction **reverts**.
7. Alice's legitimate invoke is gone from the mempool; her account is charged fees for the reverted transaction she never authorized.

### Impact Explanation

This matches the **High** impact category: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

The gateway's invariant — that every admitted invoke transaction has passed cryptographic signature verification via `__validate__` — is broken. An unprivileged attacker can inject unsigned/wrongly-signed invoke transactions into the mempool for any address that has a pending `deploy_account`. This enables:

- **Displacement of legitimate transactions** via fee escalation (victim's invoke is lost).
- **Forced fee drain** on the victim's newly deployed account (fees charged for the reverted attacker transaction).
- **Denial of service** for the deploy-account + invoke UX flow.

### Likelihood Explanation

The trigger is fully unprivileged and requires only observing the public mempool. The window is open from the moment a `deploy_account` transaction is admitted until the block is committed. The attacker needs only to know the target address (deterministic from `class_hash`, `contract_address_salt`, `constructor_calldata`) and submit a higher-tip invoke. No special access or privileged position is required.

### Recommendation

The `skip_stateful_validations` check must verify that the account address has a **`deploy_account` transaction** specifically in the mempool, not merely any transaction. The mempool should expose a dedicated query such as `has_pending_deploy_account(address)` that inspects the transaction type, rather than the generic `account_tx_in_pool_or_recent_block` which returns `true` for any transaction type from that address.

Alternatively, the skip-validate path should be restricted to cases where the gateway can cryptographically confirm the invoke's signature will be valid once the account is deployed (e.g., by running a lightweight off-chain signature pre-check against the declared account class).

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
