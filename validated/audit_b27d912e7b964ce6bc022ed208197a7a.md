### Title
Unbounded Growth of `stakers` Vector Causes DoS on `get_stakers()` via Repeated Stake/Unstake Cycles - (File: src/staking/staking.cairo)

### Summary
The `stakers` storage vector in `Staking.cairo` grows permanently with every new staker registration and is **never pruned** when stakers exit. The `get_stakers()` function iterates over the entire vector on every call. An unprivileged attacker can inflate this vector to an arbitrary size by repeatedly staking with fresh addresses and then unstaking (recovering their principal), causing `get_stakers()` to consume unbounded gas and eventually become uncallable.

### Finding Description

The `stakers` storage field is declared with an explicit note that entries are never removed: [1](#0-0) 

Every call to `stake()` unconditionally appends the caller's address to this vector: [2](#0-1) 

`get_stakers()` iterates over the **full range** of the vector on every invocation, checking each entry for activity and staking power: [3](#0-2) 

When a staker completes `unstake_action()`, `remove_staker()` clears the staker's `staker_info` and balance traces but does **not** remove the address from the `stakers` vector: [4](#0-3) 

`assert_staker_address_not_reused` prevents the same address from re-staking, but the attacker uses a **different** address each cycle, so this guard is irrelevant.

The attack loop:
1. Attacker stakes `min_stake` STRK from address `A_1` → `A_1` appended to `stakers`.
2. Attacker calls `unstake_intent()` from `A_1`, waits the exit window (default 1 week), calls `unstake_action()` → principal returned, `A_1` remains in `stakers` forever.
3. Repeat with `A_2`, `A_3`, … `A_N`.

After N cycles the attacker has spent only gas fees (principal is always recovered). `get_stakers()` must now read and evaluate N storage slots on every call.

### Impact Explanation

`get_stakers()` is the consensus-layer entry point that the sequencer/consensus node calls to build the validator set for each epoch: [5](#0-4) 

On Starknet, all execution — including view calls — is bounded by the execution-resource limit. Once the `stakers` vector is large enough, `get_stakers()` will exceed the resource limit and revert on every invocation, making it impossible for the consensus layer to retrieve the active validator set. This is a **permanent, protocol-level Denial of Service** on the consensus mechanism, matching the allowed impact: **Unbounded gas consumption / griefing with damage to the protocol**.

### Likelihood Explanation

- Entry path is fully permissionless: any address holding `min_stake` STRK can execute the attack.
- The attacker recovers their principal after each cycle; the only ongoing cost is gas fees.
- The attack is incremental and irreversible — each cycle permanently increases the iteration cost of `get_stakers()`.
- No privileged role, leaked key, or external dependency is required.

### Recommendation

1. **Short term:** Track a separate `active_stakers_count` and skip the full-range iteration by maintaining a compact set of currently-active staker addresses (e.g., a doubly-linked list or a separate `active_stakers` map that is updated on `stake` and `unstake_action`).
2. **Long term:** Remove exited stakers from the `stakers` vector (or replace it with a data structure that supports O(1) deletion), so the iteration cost is bounded by the number of *currently active* stakers rather than the cumulative historical count.

### Proof of Concept

```
// Pseudocode — repeat N times with fresh addresses
for i in 0..N:
    address = fresh_address(i)
    STRK.approve(staking, min_stake)
    staking.stake(reward=address, operational=fresh_op(i), amount=min_stake)  // appends to stakers[]
    staking.unstake_intent()                                                   // from address
    advance_time(exit_wait_window)
    staking.unstake_action(address)                                            // principal returned; address stays in stakers[]

// After N iterations:
staking.get_stakers(current_epoch)  // iterates N entries → OOG / resource-limit exceeded
```

The `stakers` vector now contains N permanently dead entries. Each call to `get_stakers()` must read and evaluate all N slots. Once N is large enough to exhaust Starknet's per-call execution budget, the function reverts unconditionally, blocking the consensus layer from obtaining the validator set.
