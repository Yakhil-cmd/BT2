### Title
`protocolUnpausePool` Always Transitions to Admin-Paused State (Level 1) Instead of Restoring Active State — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary

`MetricOmmPoolFactory.protocolUnpausePool` hardcodes the target pause level to `1` (admin-paused) instead of `0` (active). When the protocol pauses a pool that was previously active (`pauseLevel == 0`) and then unpauses it, the pool is left in the admin-paused state, keeping swaps permanently blocked until the pool admin separately intervenes.

### Finding Description

The factory defines three pause levels:

- `0` = active (swaps allowed)
- `1` = paused by pool admin
- `2` = paused by protocol

The pause state machine is: [1](#0-0) 

```
pausePool (admin):         cur == 0  → 1
unpausePool (admin):       cur == 1  → 0
protocolPausePool (owner): cur == 0 or 1 → 2
protocolUnpausePool (owner): cur == 2 → 1   ← BUG
```

The critical path is: [2](#0-1) 

`protocolUnpausePool` unconditionally calls `setPause(1)`. It does not distinguish between a pool that was active (`0`) before the protocol pause and one that was already admin-paused (`1`). In both cases the pool ends up at level `1` after the protocol unpause.

The `swap` function enforces `whenNotPaused`, which reverts on any non-zero pause level: [3](#0-2) 

So after `protocolUnpausePool`, swaps remain blocked. The pool admin must separately call `unpausePool` (which requires `cur == 1` → `0`) to restore the pool to active. The factory stores no record of the pre-protocol-pause state, so the protocol owner cannot determine whether the pool was originally active or admin-paused. [4](#0-3) 

### Impact Explanation

After the protocol owner calls `protocolUnpausePool` on a pool that was active before the protocol pause, the pool remains at `pauseLevel == 1`. All `swap` calls revert with `PoolPaused`. The pool's primary function is broken. `addLiquidity` and `removeLiquidity` are not gated by `whenNotPaused` so LP principal is not directly locked, but the pool is unusable for trading and LPs cannot earn fees. If the pool admin is a slow multisig, unresponsive, or unaware that a secondary unpause is required, the pool can remain stuck in a paused state indefinitely after the protocol has already signaled its intent to restore it.

This matches the allowed impact: **broken core pool functionality causing unusable swap flows**.

### Likelihood Explanation

The scenario requires:
1. A pool that is active (`pauseLevel == 0`) — the normal operating state for any live pool.
2. The protocol owner calling `protocolPausePool` — a legitimate emergency action.
3. The protocol owner calling `protocolUnpausePool` — the natural follow-up.

All three steps are routine, expected operations. No malicious setup or non-standard tokens are required. The trigger is a valid semi-trusted action by the protocol owner.

### Recommendation

`protocolUnpausePool` should transition to `0` (active) rather than `1` (admin-paused), since the protocol's intent is to restore the pool to a usable state:

```solidity
function protocolUnpausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 2) revert InvalidPauseTransition(cur, 2);
-   IMetricOmmPoolFactoryActions(pool).setPause(1);
+   IMetricOmmPoolFactoryActions(pool).setPause(0);
}
```

Alternatively, if the design intent is to preserve the pre-protocol-pause state, the factory should store the pre-pause level in a mapping (e.g., `mapping(address => uint8) public prePausePauseLevel`) when `protocolPausePool` is called and restore it on `protocolUnpausePool`.

### Proof of Concept

```
1. Pool is deployed and active: pauseLevel == 0.
2. Protocol owner calls protocolPausePool(pool).
   - Precondition: cur == 0 → passes (cur != 0 && cur != 1 is false).
   - Pool transitions: pauseLevel = 2.
3. Protocol owner calls protocolUnpausePool(pool).
   - Precondition: cur == 2 → passes.
   - Pool transitions: pauseLevel = 1  ← still paused.
4. Any user calls pool.swap(...).
   - _checkNotPaused(): pauseLevel != 0 → revert PoolPaused().
5. Pool admin must separately call unpausePool(pool) to reach pauseLevel = 0.
   - If pool admin is a slow multisig or unresponsive, pool remains paused
     indefinitely despite the protocol having already "unpaused" it.
```
