### Title
Unauthenticated `SendCroToIbc` EVM Hook Allows Any Contract to Drain CRO Balances — (File: `app/upgrades.go`)

### Summary
Before the v1.8 upgrade sets `CroBridgeContractAddresses`, the `SendCroToIbc` EVM hook processes `__CronosSendCroToIbc` events emitted by **any** contract without verifying the emitting contract is an authorized CroBridge contract. An unprivileged attacker can deploy a contract, emit the event, and drain CRO balances.

### Finding Description
The upgrade handler in `app/upgrades.go` explicitly acknowledges the root cause and impact:

> "Set CroBridgeContractAddresses to authorize the canonical CroBridge contracts for the SendCroToIbc hook. This closes the **unauthenticated-drain vulnerability** where any contract emitting `__CronosSendCroToIbc` could drain CRO balances."

The fix is gated behind the v1.8 upgrade execution. Before the upgrade handler runs, `CroBridgeContractAddresses` in the cronos module params is empty. The EVM hook for `SendCroToIbc` reads this parameter to decide which contracts are authorized to trigger the CRO-to-IBC bridge path. With an empty allowlist, the hook falls through to permitting all emitters — any deployed contract can emit `__CronosSendCroToIbc` and cause the hook to initiate an IBC transfer of CRO from the module account on behalf of the event.

This is the direct Cronos analog of the external report's vulnerability class: a privileged bridge-configuration/authorization check that is absent at runtime (not just during initialization), allowing any unprivileged actor to invoke a critical state-changing path. The external report's `setAgent`-callable-at-any-time pattern maps here to the hook's missing per-caller guard that should have been set at genesis/upgrade time.

### Impact Explanation
**Critical — Unauthorized transfer/drain of CRO (native asset).**
Any unprivileged EVM contract can emit `__CronosSendCroToIbc` and cause the `SendCroToIbc` EVM hook to bridge CRO balances out of the module account to an attacker-controlled IBC destination, constituting an unauthorized cross-chain transfer of the chain's native asset.

### Likelihood Explanation
High. The attacker requires only the ability to deploy a contract and call it — no admin key, governance vote, or validator access is needed. The attack is executable in a single transaction before the upgrade height.

### Recommendation
1. Execute the v1.8 upgrade at the earliest opportunity so `CroBridgeContractAddresses` is populated with the two authorized contract addresses (`0x6b1b50c2223eb31E0d4683b046ea9C6CB0D0ea4F`, `0xCE13a6F3d4167CE958f4764D423e6D62a114c751`).
2. Harden the EVM hook to treat an empty `CroBridgeContractAddresses` as "deny all" rather than "allow all," so the pre-upgrade window is safe by default.
3. Add a genesis-time invariant check that rejects a chain start with an empty `CroBridgeContractAddresses` if the `SendCroToIbc` hook is enabled.

### Proof of Concept
1. Before the v1.8 upgrade executes, `CroBridgeContractAddresses` is `[]` in cronos module params.
2. Attacker deploys a minimal Solidity contract on Cronos that emits `__CronosSendCroToIbc(address recipient, uint256 amount, string ibcDest)` with attacker-controlled parameters.
3. Attacker calls the contract; the EVM hook fires on the emitted log.
4. Because `CroBridgeContractAddresses` is empty (no allowlist enforced), the hook processes the event and initiates an IBC transfer of CRO to the attacker's IBC address.
5. CRO is drained from the module account to the attacker's cross-chain address. [1](#0-0) [2](#0-1)
