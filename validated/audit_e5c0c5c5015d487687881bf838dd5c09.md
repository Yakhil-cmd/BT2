### Title
Gateway L2 Gas Price Threshold Check Uses Stale `l2_gas_price` Instead of `next_l2_gas_price`, Causing Invalid Admission and False Rejection - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The gateway's stateful `validate_resource_bounds` check compares a transaction's `max_price_per_unit` against the **current block's** `l2_gas_price`, but the transaction will be executed in the **next block** whose gas price is `next_l2_gas_price`. A developer TODO comment in the code explicitly acknowledges this is wrong. The mismatch causes the gateway to admit transactions that will fail at blockifier execution time (when gas price is rising) and to reject valid transactions (when gas price is falling).

### Finding Description

In `extract_state_nonce_and_run_validations`, the gateway reads the nonce and runs pre-validation checks before calling `run_validate_entry_point`: [1](#0-0) 

The pre-validation path calls `validate_resource_bounds`: [2](#0-1) 

Inside `validate_resource_bounds`, the code reads the **current block's** `l2_gas_price` from the block header and uses it as the threshold reference. A developer TODO comment explicitly acknowledges the wrong field is being read: [3](#0-2) 

The `GatewayFixedBlockSyncStateClient` populates `BlockInfo` using `block_header.l2_gas_price` (the price of the **current** block), not `block_header.next_l2_gas_price` (the price that will govern the **next** block): [4](#0-3) 

The block header carries both fields. The `next_l2_gas_price` is written into every committed block header by the consensus orchestrator: [5](#0-4) 

The `run_validate_entry_point` also constructs its `BlockContext` from the same stale `block_info` (only incrementing the block number, not updating gas prices): [6](#0-5) 

So both the gateway's admission check and its blockifier simulation use the old gas price. The transaction passes both, enters the mempool, and is later handed to the batcher, which builds the actual next block using `next_l2_gas_price`. At that point, `check_fee_bounds` in `perform_pre_validation_stage` compares the transaction's `max_price_per_unit` against the real next-block gas price: [7](#0-6) 

The invariant broken: **the gateway's admission check must use the same gas price that the blockifier will enforce at execution time**.

### Impact Explanation

**When `next_l2_gas_price > l2_gas_price` (gas price rising):**
A transaction with `threshold * l2_gas_price ≤ tx.max_price < next_l2_gas_price` passes the gateway's threshold check and the gateway's blockifier simulation (both use old prices), is admitted to the mempool, and then fails at batcher execution time with `MaxGasPriceTooLow`. The gateway has admitted an invalid transaction.

**When `next_l2_gas_price < l2_gas_price` (gas price falling):**
A transaction with `next_l2_gas_price ≤ tx.max_price < threshold * l2_gas_price` is rejected by the gateway even though it would succeed in the next block. Valid transactions are incorrectly rejected.

Both outcomes match the **High** impact scope: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

### Likelihood Explanation

The L2 gas price is updated every block via an EIP-1559-style algorithm. The price can change by approximately `current_price / 333` per block (~0.3%). The discrepancy is small per block but is always present whenever the gas price is not exactly flat. Any user submitting a transaction near the threshold boundary is affected. The TODO comment in the production code confirms the developers are aware the wrong field is being used.

### Recommendation

In `validate_resource_bounds`, read `next_l2_gas_price` from the block header instead of `l2_gas_price`. This requires either:
1. Extending the `GatewayFixedBlockStateReader` trait to expose `next_l2_gas_price` separately, or
2. Returning the full `BlockHeaderWithoutHash` (which already carries `next_l2_gas_price`) from the reader.

The `run_validate_entry_point` should similarly use `next_l2_gas_price` when constructing the `BlockContext` gas prices, so the gateway's blockifier simulation matches the actual execution environment.

### Proof of Concept

1. Observe the current L2 gas price from the latest committed block header: `P_current`.
2. Compute `P_next = calculate_next_base_gas_price(P_current, ...)` — this will be slightly above `P_current` if the block was above the gas target.
3. Submit an `InvokeV3` transaction with `l2_gas.max_price_per_unit = P_current` (i.e., exactly equal to the current block price, which satisfies `threshold * P_current` when `min_gas_price_percentage = 100`).
4. The gateway's `validate_resource_bounds` computes `threshold = 100% * P_current = P_current` and accepts the transaction since `tx.price >= threshold`.
5. The gateway's `run_validate_entry_point` also uses `P_current` in its `BlockContext`, so `check_fee_bounds` passes.
6. The transaction enters the mempool.
7. The batcher builds the next block with gas price `P_next > P_current`. The blockifier's `check_fee_bounds` now checks `tx.max_price (= P_current) < P_next` and returns `MaxGasPriceTooLow`, reverting or dropping the transaction.

The gateway admitted a transaction that the blockifier rejects — a direct consequence of using `l2_gas_price` instead of `next_l2_gas_price` for the admission threshold.
