### Title
Gateway Signature Validation Bypass via Overly Broad `skip_stateful_validations` Admission Check — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator skips the `__validate__` entry-point call (signature verification) for any invoke transaction with nonce=1 when the account's on-chain nonce is 0 and `account_tx_in_pool_or_recent_block` returns `true`. The check is intended to confirm a pending `deploy_account` transaction, but it returns `true` for **any** transaction in the pool for that address — including a regular nonce-0 invoke. An attacker who observes a valid nonce-0 invoke in the mempool for account A can submit a nonce-1 invoke with an **invalid signature** for account A, and the gateway will admit it without running `__validate__`.

### Finding Description

`skip_stateful_validations` implements the deploy-account + invoke UX shortcut:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs:437-456
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
    return mempool_client
        .account_tx_in_pool_or_recent_block(tx.sender_address())
        .await
        ...
}
``` [1](#0-0) 

The comment claims: *"It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."*

`account_tx_in_pool_or_recent_block` is implemented as:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [2](#0-1) 

This returns `true` for **any** transaction in the pool for that address — not specifically a `deploy_account`. A regular invoke with nonce=0 for an account that already exists on-chain (nonce=0 in state) passes the gateway's own `validate_nonce` check:

```rust
// account_nonce=0, incoming_tx_nonce=0 → 0 <= 0 <= max_allowed_nonce → valid
_ => {
    let max_allowed_nonce =
        Nonce(account_nonce.0 + Felt::from(self.config.max_allowed_nonce_gap));
    if !(account_nonce <= incoming_tx_nonce && incoming_tx_nonce <= max_allowed_nonce) {
        return Err(...)
    }
}
``` [3](#0-2) 

So once a legitimate nonce-0 invoke from account A is in the pool, `account_tx_in_pool_or_recent_block(A)` returns `true`. An attacker can then submit a nonce-1 invoke for account A with an **arbitrary/invalid signature**. The gateway calls `run_validate_entry_point` with `skip_validate=true`, setting `execution_flags.validate = false`:

```rust
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [4](#0-3) 

Inside `StatefulValidator::perform_validations`, when `validate=false`, the `__validate__` call is skipped entirely:

```rust
if !tx.execution_flags.validate {
    return Ok(());
}
``` [5](#0-4) 

The invalid nonce-1 invoke is admitted to the mempool without any signature check.

### Impact Explanation

When the batcher later executes the invalid nonce-1 invoke:
1. `perform_pre_validation_stage` increments the account nonce (from 1 to 2) before `__validate__` runs.
2. `validate_tx` calls `__validate__`, which fails due to the invalid signature.
3. The transaction reverts, but the nonce increment and fee charge are applied to the account owner.

The account owner (victim) pays fees for a transaction they never signed, and their nonce is advanced by 1, potentially breaking the ordering of their own pending transactions.

This matches: **High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

### Likelihood Explanation

The precondition — a valid nonce-0 invoke in the mempool for the target account — is common. Any account actively submitting transactions will have pending nonces. The attacker needs no special privilege: they only need to observe the mempool (or submit a transaction to trigger the condition) and then submit a crafted nonce-1 invoke. The attack is cheap (no on-chain cost to the attacker; fees are charged to the victim).

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a specific check that confirms a `deploy_account` transaction is pending for the address. The mempool should expose a dedicated `has_pending_deploy_account(address)` query, or the pool should be filtered to only match `DeployAccount` transaction types when deciding to skip validation.

Alternatively, the skip should only trigger when the account does **not** exist on-chain at all (i.e., `get_class_hash_at(address)` returns zero), which is the true precondition for the deploy-account UX shortcut.

### Proof of Concept

1. Account A is deployed on-chain with nonce=0 (class hash set, contract exists).
2. Account owner submits a valid invoke with nonce=0 for account A → passes `run_validate_entry_point` → admitted to mempool. Now `account_tx_in_pool_or_recent_block(A)` returns `true`.
3. Attacker submits an invoke with nonce=1, sender=A, **invalid signature** (e.g., all-zero signature).
4. Gateway calls `extract_state_nonce_and_run_validations`:
   - `account_nonce = 0` (from state)
   - `validate_nonce`: `0 <= 1 <= max_allowed_nonce` → passes
   - `skip_stateful_validations`: `tx.nonce()==1 && account_nonce==0 && account_in_pool==true` → returns `true`
   - `run_validate_entry_point(tx, skip_validate=true)` → `validate=false` → `__validate__` not called
5. Invalid transaction is admitted to the mempool.
6. Batcher executes nonce-0 invoke → account nonce becomes 1.
7. Batcher executes nonce-1 invoke → `__validate__` runs → fails (invalid signature) → reverts.
8. Account A's nonce is now 2; account A is charged fees for the failed transaction.
