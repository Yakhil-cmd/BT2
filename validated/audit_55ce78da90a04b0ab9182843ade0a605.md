### Title
Missing `ExecutionMode::Validate` Guard on `replace_class` Allows Class-Hash Mutation During Validation, Causing Validate/Execute Class Mismatch and Persistent State Corruption — (`crates/blockifier/src/execution/syscalls/syscall_base.rs`)

---

### Summary

`SyscallHandlerBase::replace_class` has no execution-mode guard. Every other state-mutating or sensitive syscall (`meta_tx_v0`, `get_class_hash_at`, `deploy`, `call_contract`) explicitly calls `reject_syscall_in_validate_mode` when `context.execution_mode == ExecutionMode::Validate`. `replace_class` does not. An account whose `__validate__` calls `replace_class(class_hash_B)` will:

1. Pass the retdata check using the **old** `class_hash_A` (captured before validate runs).
2. Have its `__execute__` dispatched against the **new** `class_hash_B` (re-read from state after validate mutates it).
3. Permanently commit the class-hash change to the block state **even if `__execute__` is later reverted**, because validate writes directly to the outer `TransactionalState` while execute writes to a nested `TransactionalState` that can be independently aborted.

---

### Impact Explanation

Two concrete wrong-state effects:

1. **Validate/execute class mismatch**: The retdata check at line 1042 uses `class_hash_A` (the pre-validate snapshot), while `__execute__` dispatches against `class_hash_B` (the post-validate state). The class that was validated is not the class that executes.

2. **Permanent class-hash mutation from validate phase**: `replace_class` called during `__validate__` writes `class_hash_B` to the outer `TransactionalState`. This write is committed to the block state even if `__execute__` fails and its `TransactionalState` is aborted. The account's class hash in the committed state diff is `class_hash_B`, not `class_hash_A`, even for a reverted transaction. This is a wrong class hash in the committed state from blockifier execution logic.

The `replace_class` guard `!is_cairo1` (line 373) prevents downgrading to Cairo0, so `class_hash_B` must be Cairo1, but it can be any declared Cairo1 class — including one with a trivially-accepting `__validate__` for all future transactions. [10](#0-9) 

---

### Likelihood Explanation

Any account contract can call `replace_class` from `__validate__`. No privileged access is required. The attacker deploys an account with a `__validate__` that calls `replace_class(class_hash_B)` before returning `VALID`. This is a single unprivileged transaction from any sender address.

---

### Recommendation

Add an execution-mode guard to `SyscallHandlerBase::replace_class`, consistent with the pattern used by `get_class_hash_at`, `meta_tx_v0`, and `deploy`:

```rust
pub fn replace_class(&mut self, class_hash: ClassHash) -> SyscallResult<()> {
    if self.context.execution_mode == ExecutionMode::Validate {
        return Err(self.reject_syscall_in_validate_mode("replace_class")
            .map_err(SyscallExecutionError::from)?);
    }
    let compiled_class = self.state.get_compiled_class(class_hash)?;
    if !is_cairo1(&compiled_class) {
        return Err(SyscallExecutionError::ForbiddenClassReplacement { class_hash });
    }
    self.state.set_class_hash_at(self.call.storage_address, class_hash)?;
    Ok(())
}
```

The same guard must be applied to the deprecated syscall handler's `replace_class` in `hint_processor.rs`: [11](#0-10)

### Proof of Concept

**Step-by-step:**

1. Declare `class_hash_B`: a Cairo1 account class whose `__validate__` always returns `VALID` (no signature check) and whose `__execute__` does arbitrary work.

2. Declare `class_hash_A`: a Cairo1 account class whose `__validate__` calls `replace_class(class_hash_B)` then returns `VALID`.

3. Deploy an account at address `sender` with `class_hash_A`.

4. Send an Invoke V3 transaction from `sender`.

**Execution trace:**

- `validate_tx` line 1017: `class_hash = state.get_class_hash_at(sender)` → `class_hash_A`
- `validate_call.execute(state, ...)` runs `class_hash_A.__validate__`:
  - calls `replace_class(class_hash_B)` → `state.set_class_hash_at(sender, class_hash_B)`
  - returns `VALID`
- `validate_tx` line 1042: `state.get_compiled_class(class_hash_A)` → Cairo1 → checks retdata → passes
- `run_execute` line 313: `state.get_class_hash_at(sender)` → `class_hash_B`
- `CallEntryPoint::execute` line 192: `class_hash: None` → resolves to `class_hash_B`
- `__execute__` of `class_hash_B` runs

**Assertions that demonstrate the bug:**

```
assert state.get_class_hash_at(sender) == class_hash_B   // state mutated during validate
assert validate_call_info.call.class_hash == class_hash_A // validated with A
assert execute_call_info.call.class_hash == class_hash_B  // executed with B
// class_hash_A != class_hash_B  → invariant violated
```

**Revert-persistence sub-case:** If `class_hash_B.__execute__` panics, `execution_state.abort()` reverts execute writes, but `state` (which holds the `replace_class` write from validate) is still committed by `transactional_state.commit()`. The block state diff contains `sender → class_hash_B` even though the transaction is marked reverted.
