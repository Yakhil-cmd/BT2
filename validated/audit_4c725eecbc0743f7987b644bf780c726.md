### Title
Missing Promise-Result Existence Check in `ExitToNear` Precompile Callback Leads to Permanent Fund Freeze When `error_refund` Feature Is Disabled - (File: engine-precompiles/src/native.rs)

### Summary

When the `error_refund` compile-time feature is not enabled, the `ExitToNear` precompile burns ERC-20 tokens on the EVM side and schedules a NEAR `ft_transfer` promise, but the callback (`exit_to_near_precompile_callback`) performs no refund when that promise fails. Because `args.refund` is unconditionally `None` without `error_refund`, a failed NEAR-side transfer silently discards the burned tokens, permanently freezing user funds. This is the direct Aurora analog of the Timelock missing-existence-check bug: a critical action (token burn) is committed before verifying the target is reachable, and success of the EVM transaction does not guarantee the NEAR-side effect occurred.

---

### Finding Description

The `ExitToNear` precompile constructs `ExitToNearPrecompileCallbackArgs` with the `refund` field set conditionally at compile time:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,          // ← always None without the feature
    transfer_near: transfer_near_args,
};
``` [1](#0-0) 

The callback `exit_to_near_precompile_callback` then checks the NEAR promise result:

```rust
let maybe_result = if let Some(PromiseResult::Successful(_)) = handler.promise_result(0) {
    // success path
    None
} else if let Some(args) = args.refund {   // ← None without error_refund
    // refund path — never reached
    ...
} else {
    None   // ← silent no-op on failure
};
``` [2](#0-1) 

Without `error_refund`, the `else if` branch is dead code. A failed `ft_transfer` (e.g., target NEAR account not registered with the NEP-141 contract) causes the callback to return `Ok(None)` — no refund, no error, no recovery. The ERC-20 tokens were already burned before the promise was dispatched.

`error_refund` is **not** a default feature in `engine/Cargo.toml`:

```toml
[features]
default = ["std"]
...
error_refund = ["aurora-engine-precompiles/error_refund"]
``` [3](#0-2) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

When a user calls `exitToNear` (or an ERC-20 contract calls the precompile via `withdrawToNear`) with a NEAR account ID that is syntactically valid but not registered with the NEP-141 token contract, the sequence is:

1. ERC-20 tokens are burned on the EVM side (irreversible within the EVM).
2. A NEAR `ft_transfer` promise is scheduled to the NEP-141 contract.
3. The NEP-141 `ft_transfer` panics because the receiver is not registered → promise fails.
4. `exit_to_near_precompile_callback` is invoked; `args.refund` is `None`; callback returns `Ok(None)`.
5. No ERC-20 tokens are re-minted; no NEP-141 tokens are transferred. Funds are gone.

The EVM transaction reports `TransactionStatus::Succeed`, so the user and any monitoring tooling observes success while the funds are permanently lost. [4](#0-3) 

---

### Likelihood Explanation

**High.** Any unprivileged EVM user can trigger this by supplying a NEAR account ID that is not registered with the NEP-141 contract. NEAR's fungible token standard (NEP-141) requires explicit `storage_deposit` registration before receiving tokens; unregistered accounts are common. The `exitToNear` path is a core, publicly reachable bridge function. The `error_refund` feature is not in the default feature set, so any deployment built without explicitly enabling it is vulnerable.

---

### Recommendation

**Short term:** Enable the `error_refund` feature in all production builds of the Aurora Engine WASM artifact. Verify the Makefile/CI build flags explicitly include `--features error_refund`.

**Long term:** Analogous to the Timelock fix (check target code existence before executing), add a pre-flight check in the `ExitToNear` precompile that verifies the target NEAR account is registered with the NEP-141 contract before burning ERC-20 tokens. Alternatively, make `error_refund` a default feature so the refund path is always compiled in, removing the dependency on a build-time flag for a critical safety property.

---
