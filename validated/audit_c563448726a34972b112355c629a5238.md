### Title
Gateway `skip_stateful_validations` uses attacker-controlled `sender_address` to bypass `__validate__` — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful transaction validator decides whether to skip the account's `__validate__` entry point by looking up the attacker-supplied `sender_address` in the mempool. Because the lookup key is taken directly from the transaction being submitted — not from any authenticated source — an attacker can set `sender_address` to any address that already has a pending deploy-account transaction in the mempool, causing the gateway to admit an invoke transaction that carries an arbitrary, invalid signature.

### Finding Description

`skip_stateful_validations` (lines 429–461) implements the "deploy-account + invoke UX skip":

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs  lines 429-461
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();          // ← attacker-controlled
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())  // ← same field
                .await ...;
        }
    }
    Ok(false)
}

**Analog to the `setConfig` bug.** The original bug validates `creator == _creator` — a stored value against a caller-supplied parameter — so the attacker simply passes the correct value. Here the gateway validates `account_nonce == 0 && account_tx_in_pool_or_recent_block(sender_address)` — both sides of the check are derived from the attacker-supplied `sender_address`. The attacker picks a `sender_address` for which both conditions are already true (any address with a pending deploy-account), satisfying the check without owning the key.

**Concrete attack path:**

1. Alice broadcasts a `deploy_account` for address Y; it sits in the mempool (Y's on-chain nonce = 0).
2. Attacker observes the mempool (public), learns Y has a pending deploy-account.
3. Attacker pre-funds Y with enough STRK to pass `verify_can_pay_committed_bounds`.
4. Attacker submits `Invoke { sender_address: Y, nonce: 1, calldata: <anything>, signature: <garbage> }`.
5. Gateway fetches nonce for Y → 0; checks `account_tx_in_pool_or_recent_block(Y)` → `true`; sets `skip_validate = true`.
6. `perform_pre_validation_stage` runs: nonce check passes (1 ≥ 0), balance check passes (attacker funded Y).
7. `__validate__` is **not** called. Transaction is forwarded to the mempool.
8. If fee escalation is enabled and the attacker pays a higher tip/gas price, the attacker's transaction **replaces** Alice's legitimate nonce-1 invoke already in the mempool.
9. Batcher executes the block: deploy-account for Y succeeds; attacker's invoke runs `__validate__` → reverts (bad signature); Y's nonce advances to 2.
10. Alice's legitimate nonce-1 invoke is now permanently rejected (`NonceTooOld`). [5](#0-4) 

### Impact Explanation

The gateway admits an invoke transaction that carries an invalid signature for the claimed `sender_address`. This satisfies the "High" impact criterion: **mempool/gateway admission accepts invalid transactions before sequencing**. Concretely:

- An attacker can permanently block any account's first post-deploy invoke by occupying or replacing the `(address, nonce=1)` slot with an unsigned transaction.
- With fee escalation enabled the replacement is unconditional (attacker just pays more).
- The reverted transaction still increments the victim's nonce and drains the pre-funded balance, leaving the victim unable to use nonce 1.

### Likelihood Explanation

- The mempool is public; any observer can enumerate addresses with pending deploy-accounts.
- Pre-funding the target address costs the attacker only the fee amount (small).
- The race window is wide: the attacker can act any time between Alice's deploy-account entering the mempool and Alice's invoke entering the mempool.
- Fee escalation (enabled in production config) removes the race entirely — the attacker can always outbid Alice's invoke.

### Recommendation

The skip decision must be tied to the **authenticated identity** of the submitter, not to a field the submitter controls. Two concrete options:

1. **Verify the deploy-account key matches the invoke signature.** When `skip_validate = true`, verify that the invoke's signature is consistent with the class hash and constructor calldata declared in the pending deploy-account (i.e., perform a lightweight off-chain signature check against the expected public key).
2. **Bind the skip to the deploy-account transaction hash.** Require the invoke to include the hash of the pending deploy-account as an additional field, and verify that hash exists in the mempool for the same address. An attacker who does not control the deploy-account cannot produce a valid hash.

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
