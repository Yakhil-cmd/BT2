### Title
`validateCOAOwnershipProof` Does Not Bind `signedData` to `evmAddress`, Enabling Cross-COA ERC-1271 Signature Forgery - (File: `fvm/evm/stdlib/contract.cdc`)

---

### Summary

`EVM.validateCOAOwnershipProof` verifies that (a) the provided signatures are valid over `signedData`, and (b) the COA stored at `path` has an EVM address matching `evmAddress`. It does **not** verify that `signedData` encodes `evmAddress`. This is the direct analog of the reported hashlock bug: the check is present but insufficient — it validates the signature and the address independently, but never binds them together. An attacker who observes any valid COA ownership proof for COA-A can modify the `CapabilityPath` field (which is not signed) to point to COA-B and replay the same signature to forge an ERC-1271 `isValidSignature` response for COA-B.

---

### Finding Description

`EVM.validateCOAOwnershipProof` in `fvm/evm/stdlib/contract.cdc` performs two independent checks:

1. Verifies the signatures are valid over `signedData` (lines 1082–1086).
2. Verifies the COA resource at `path` has an EVM address matching `evmAddress` (lines 1095–1105).

The function explicitly documents the gap at lines 1003–1009:

> *"Note: this function does not enforce that `signedData` includes `evmAddress`. In principle, a signature produced for one purpose could be replayed here against a different COA owned by the same Cadence account."*

The `COAOwnershipProof` struct (in `fvm/evm/types/proof.go`) contains `CapabilityPath`, `KeyIndices`, and `Signatures`. The signatures are over `signedData` only — **not** over the proof struct itself. This means an attacker can freely substitute the `CapabilityPath` field in an observed proof without invalidating the signatures.

The EVM-side entry point is `coa.sol`'s `isValidSignature` (line 118), which calls the `verifyCOAOwnershipProof` precompile with `address(this)` as the EVM address and the caller-supplied `_hash` as `signedData`. The precompile (`fvm/evm/precompiles/arch.go`, `DecodeABIEncodedProof`) constructs a `COAOwnershipProofInContext` where `EVMAddress` is the calling COA's address and `SignedData` is the hash. The attacker-controlled `_sig` bytes contain the `COAOwnershipProof` struct including `CapabilityPath`.

**Attack steps:**

1. Account `A` owns two COAs: COA-X at `/public/coaX` and COA-Y at `/public/coaY`.
2. Account `A` legitimately signs hash `H` and produces a valid proof `P_X = {Address: A, CapabilityPath: "coaX", KeyIndices: [0], Signatures: [sig(H)]}`.
3. Attacker observes `P_X` (e.g., from a public transaction).
4. Attacker crafts `P_Y = {Address: A, CapabilityPath: "coaY", KeyIndices: [0], Signatures: [sig(H)]}` — only `CapabilityPath` changed; `sig(H)` is unchanged and still valid.
5. Attacker calls `COA-Y.isValidSignature(H, encode(P_Y))`.
6. `validateCOAOwnershipProof` is invoked with `signedData=H`, `path=/public/coaY`, `evmAddress=COA-Y's address`.
7. Step 1: `sig(H)` is valid over `H` → passes.
8. Step 2: COA at `/public/coaY` has address == COA-Y's address → passes.
9. Returns `isValid: true` — COA-Y's `isValidSignature` returns `0x1626ba7e` (valid).

---

### Impact Explanation

An attacker can forge ERC-1271 `isValidSignature` responses for any COA owned by an account that also owns at least one other COA, provided the attacker has observed a single valid ownership proof for any COA of that account. ERC-1271 is used by DeFi protocols for permit-style approvals, NFT marketplace approvals, and multi-sig schemes. A forged `isValidSignature` response allows the attacker to authorize token transfers, approve ERC-20 permits, or satisfy any ERC-1271-gated operation on behalf of COA-Y without the account owner's consent — constituting unauthorized cross-VM asset loss.

---

### Likelihood Explanation

The precondition is that the target Cadence account holds multiple COAs. While historically rare, the Flow ecosystem is growing and multi-COA accounts are a supported and documented pattern. A valid proof for any one COA of the account is sufficient; such proofs are broadcast in public transactions. The attacker needs only to observe one such proof and modify a single field (`CapabilityPath`) in the RLP-encoded struct. No privileged access, leaked keys, or staked node control is required. The attack is executable by any unprivileged EVM transaction sender.

---

### Recommendation

Enforce that `signedData` encodes `evmAddress` before accepting the proof as valid. Concretely, inside `validateCOAOwnershipProof`, after verifying the signatures, add a check that the first 20 bytes (or a well-defined prefix) of `signedData` equal `evmAddress`. Alternatively, define a canonical signed payload format that commits to both the hash and the target COA address, and enforce it here. The comment at line 1003 already identifies the fix direction; it should be promoted from a documentation note to an enforced invariant.

func DecodeABIEncodedProof(input []byte) (*types.COAOwnershipProofInContext, error) {
	index := 0
	caller, err := ReadAddress(input, index)
	index += FixedSizeUnitDataReadSize
	if err != nil {
		return nil, err
	}

	hash, err := ReadBytes32(input, index)
	index += Bytes32DataReadSize
	if err != nil {
		return nil, err
	}

	encodedProof, err := ReadBytes(input, index)
	if err != nil {
		return nil, err
	}

	return types.NewCOAOwnershipProofInContext(
		hash,
		types.Address(caller),
		encodedProof,
	)
}

// COAOwnershipProofInContext contains all the data
// needed to verify a COAOwnership proof.
// The proof is verified by checking the signatures over the
// input signed data (SignedData), then loading the resource
// capability from the provided path in the proof, and
// at last checking if the EVMAddress of the resource matches
// the provided one.
type COAOwnershipProofInContext struct {
	COAOwnershipProof
	SignedData SignedData
	EVMAddress Address
}

func NewCOAOwnershipProofInContext(sd []byte, addr Address, encodedProof []byte) (*COAOwnershipProofInContext, error) {
	proof, err := COAOwnershipProofFromEncoded(encodedProof)
	if err != nil {
		return nil, err
	}
	return &COAOwnershipProofInContext{
		COAOwnershipProof: *proof,
		SignedData:        sd,
		EVMAddress:        addr,
	}, nil
}

func (proof *COAOwnershipProofInContext) ToCadenceValues() []cadence.Value {
	return []cadence.Value{
		proof.Address.ToCadenceValue(),
		proof.CapabilityPath.ToCadenceValue(),
		proof.SignedData.ToCadenceValue(),
		proof.KeyIndices.ToCadenceValue(),
		proof.Signatures.ToCadenceValue(),
		proof.EVMAddress.ToCadenceValue(),
	}
}

// COAOwnershipProof is a proof that a flow account
// controls a COA resource. To do so, the flow
// account (Address is address of this account)
// provides signatures (with proper total weights) over an arbitrary data input
// set by proof requester. KeyIndices captures,
// which account keys has been used for signatures.
// Beside signatures, it provides the CapabilityPath
// where the resource EVMAddress capability is stored.
type COAOwnershipProof struct {
	KeyIndices     KeyIndices
	Address        FlowAddress
	CapabilityPath PublicPath
	Signatures     Signatures
}
