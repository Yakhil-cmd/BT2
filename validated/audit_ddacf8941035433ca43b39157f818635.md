After thorough analysis of the codebase, I've found a valid analog to the StRSR era-desynchronization bug. The StRSR bug's core invariant is: two state counters that should be synchronized diverge, causing records created against the old state to be silently accepted and then lost when the second counter advances. The Sequencer analog is a two-state desynchronization between the **mempool's account-presence state** and the **gateway's on-chain nonce state**, which together gate whether the `__validate__` entry point (signature check) is called.

---

### Title
Gateway Admits Invoke Transactions with Invalid Signatures via `skip_stateful_validations` Bypass — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

`skip_stateful_validations` skips the blockifier `__validate__` entry-point call (the only place the account's signature is verified) for any Invoke with `nonce == 1` when `account_nonce == 0` and `account_tx_in_pool_or_recent_block` returns `true`. The pool-presence check is satisfied by **any** transaction for the target address — not specifically a `deploy_account`. An unprivileged attacker who observes a pending `deploy_account` for address X in the public mempool can immediately submit an Invoke for X with a completely fabricated signature, and the gateway will admit it without ever calling `__validate__`.

### Finding Description

The gateway stateful-validation path in `extract_state_nonce_and_run_validations` is:

1. Read `account_nonce` from the latest committed block state.
2. Call `run_pre_validation_checks` → `validate_state_preconditions` (nonce range, resource bounds) + `validate_by_mempool` (nonce-too-old check only) + `skip_stateful_validations`.
3. Call `run_validate_entry_point(executable_tx, skip_validate)`. [1](#0-0) 

Inside `run_validate_entry_point`, when `skip_validate == true` the `ExecutionFlags` are built with `validate: false`: [2](#0-1) 

`StatefulValidator::perform_validations` then returns early before the `__validate__` call: [3](#0-2) 

`skip_stateful_validations` sets `skip_validate = true` whenever:
- the transaction is an Invoke with `nonce == 1`,
- `account_nonce == 0`, and
- `account_tx_in_pool_or_recent_block(sender)` returns `true`. [4](#0-3) 

The pool-presence predicate is: [5](#0-4) 

which returns `true` if the address appears in `staged`, `committed`, or `tx_pool` — for **any** transaction type, not only `deploy_account`. The code comment claims this implies a `deploy_account` exists, but that is not enforced: [6](#0-5) 

`MempoolState::contains_account` checks only key presence in two maps: [7](#0-6) 

**The desynchronization:** The gateway reads `account_nonce` from the committed on-chain state (one "era"), while the mempool tracks account presence across `staged`/`committed`/`tx_pool` (a second, independent "era"). When the on-chain state says `nonce == 0` (account not deployed) but the mempool says the address is known (because a `deploy_account` was staged or committed), the two states diverge. The gateway treats this divergence as a signal to skip `__validate__`, but the mempool's presence record does not guarantee the pending transaction is a `deploy_account`, nor that it has not already been evicted or rewound.

### Impact Explanation

An attacker can submit an Invoke for any address X that has a pending `deploy_account` in the public mempool, using a completely fabricated signature. The gateway admits it without calling `__validate__`. The transaction enters the mempool and competes for block inclusion. Concretely:

- **Invalid transaction admitted to mempool** — matches "High. Mempool/gateway/RPC admission accepts invalid transactions before sequencing."
- The attacker can front-run the legitimate user's nonce-1 Invoke: if the attacker's invalid Invoke is admitted first with a higher tip, the legitimate user's Invoke must pay a higher fee to replace it (fee-escalation griefing), or is queued behind it.
- The invalid Invoke will fail at batcher execution time (the batcher sets `validate=true`), wasting block space and potentially causing the legitimate user's deploy-and-invoke flow to stall.

### Likelihood Explanation

- Requires no privilege: `deploy_account` transactions are publicly visible in the mempool.
- The attacker only needs to craft an Invoke with `nonce=1`, `sender_address=X`, and arbitrary bytes as the signature.
- The default `max_nonce_for_validation_skip` is `0x1`, so the window is always open for the first post-deploy Invoke. [8](#0-7) 

### Recommendation

Replace the coarse `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a `deploy_account` transaction for address X is present in the pool (not merely any transaction). Alternatively, restrict the skip to cases where the mempool can confirm the pending transaction is of type `DeployAccount`, preventing the bypass from being triggered by an attacker-controlled pool entry.

### Proof of Concept

1. Legitimate user submits `deploy_account` for address X → admitted to mempool; `tx_pool.contains_account(X)` becomes `true`.
2. Attacker submits `Invoke { sender: X, nonce: 1, signature: [0xde, 0xad, ...] }` (garbage signature).
3. Gateway calls `skip_stateful_validations`:
   - `tx.nonce() == Nonce(1)` ✓
   - `account_nonce == Nonce(0)` ✓ (X not yet deployed on-chain)
   - `account_tx_in_pool_or_recent_block(X)` → `true` (deploy_account is in pool) ✓
   - Returns `true` → `skip_validate = true`.
4. `run_validate_entry_point` builds `ExecutionFlags { validate: false, ... }` and calls `blockifier_validator.validate(account_tx)`.
5. `StatefulValidator::perform_validations` hits `if !tx.execution_flags.validate { return Ok(()); }` and returns without calling `__validate__`.
6. The Invoke with the fabricated signature is forwarded to the mempool via `add_tx`.
7. The attacker's Invoke now occupies nonce-1 for address X. The legitimate user's Invoke must escalate fees to displace it, or the invalid Invoke is sequenced and fails at batcher execution time, wasting block space. [9](#0-8) [10](#0-9) [3](#0-2)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
