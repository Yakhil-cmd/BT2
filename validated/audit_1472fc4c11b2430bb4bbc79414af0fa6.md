### Title
Non-Deterministic In-Memory Blocklist in `ProcessProposalHandler` Causes Consensus Divergence and Chain Halt - (`app/proposal.go`)

### Summary

`ProcessProposalHandler` makes consensus-critical ACCEPT/REJECT decisions based on an in-memory blocklist (`h.blocklist`) that is populated by decrypting an on-chain encrypted blob using each validator's private `age.Identity` key. Because decryption is validator-local and `SetBlockList` silently no-ops when `Identity == nil` or decryption fails, different validators can hold different in-memory blocklist states at the same consensus height. When a proposer constructs a block under one blocklist state and validators verify it under a different state, `ProcessProposal` returns divergent results across the validator set, causing a consensus failure and chain halt.

### Finding Description

The `ProposalHandler` struct holds an in-memory map `h.blocklist` that is the sole source of truth for blocklist enforcement during both `PrepareProposal` and `ProcessProposal`. [1](#0-0) 

This map is populated by `SetBlockList`, which decrypts an encrypted blob using the validator's own `age.Identity`: [2](#0-1) 

Critically, if `Identity` is `nil`, the function returns immediately without modifying `h.blocklist`: [3](#0-2) 

`ProcessProposalHandler` then uses this in-memory map to make a binding consensus decision: [4](#0-3) 

The fast-path at line 340–343 means a validator with an empty blocklist unconditionally accepts every block, while a validator with a populated blocklist may reject the same block. This is the exact double-verification divergence: the proposer filters transactions in `PrepareProposal` using its own blocklist state, and each validator independently re-verifies in `ProcessProposal` using its own (potentially different) blocklist state. [5](#0-4) 

The analog to the external bug is direct:

| External (TemporalGovernor) | Cronos |
|---|---|
| VAA verified at queue time (Wormhole state S1) | Txs filtered at `PrepareProposal` (proposer's blocklist B1) |
| VAA re-verified at execute time (Wormhole state S2) | Txs re-verified at `ProcessProposal` (each validator's blocklist Bₙ) |
| Guardian set change between phases → second check fails | Blocklist state diverges across validators → `ProcessProposal` returns REJECT on some, ACCEPT on others |
| Proposal permanently stuck | Consensus divergence → chain halt |

### Impact Explanation

When the admin submits a `MsgStoreBlockList` update, validators that successfully decrypt the new blob update `h.blocklist`; validators whose `Identity` is `nil` or whose decryption fails retain the old (or empty) blocklist. A proposer operating under the old state includes a transaction from a newly-blocked address. Validators with the updated blocklist return `REJECT`; validators with the old/empty blocklist return `ACCEPT`. CometBFT requires a 2/3+ supermajority agreement on `ProcessProposal`; a split result causes a consensus failure and halts the chain.

This matches the Critical impact category: **chain halt triggered by a mempool/proposal interaction**.

### Likelihood Explanation

The trigger is a routine, legitimate admin action (`MsgStoreBlockList`). Any validator that runs without a configured `age.Identity` (the `noneIdentity` stub is explicitly provided in the codebase) will always have an empty blocklist: [6](#0-5) 

In a heterogeneous validator set where some nodes are configured with an identity and others are not, every blocklist update creates a divergence window. No attacker action is required beyond having a pending transaction in the mempool at the time of the update.

### Recommendation

`ProcessProposal` must be deterministic. The blocklist enforcement decision must be derived from **committed, on-chain state** rather than from each validator's independently-decrypted in-memory map. Options:

1. **Store the plaintext blocklist on-chain** (in a module KVStore) so `ProcessProposal` reads from the same deterministic state on every validator.
2. **Remove blocklist enforcement from `ProcessProposal`** entirely and enforce it only in `PrepareProposal` (proposer-side filtering) and the `AnteHandler` (`BlockAddressesDecorator`), both of which are already present and do not affect consensus safety.
3. If encrypted storage is required, derive a deterministic commitment (e.g., a hash of the decrypted set) that is stored on-chain and verified in `ProcessProposal`, with the full decryption happening only in `PrepareProposal`.

### Proof of Concept

1. Validator A is configured with a valid `age.Identity`; Validator B has `Identity == nil` (uses `noneIdentity`).
2. Admin submits `MsgStoreBlockList` adding address `X` to the blocklist. Block N commits.
3. Validator A calls `SetBlockList` successfully; `h.blocklist = {"X": {}}`.
4. Validator B calls `SetBlockList`; hits the `Identity == nil` guard and returns immediately; `h.blocklist = {}`.
5. Validator B is the proposer for block N+1. Its `PrepareProposal` passes the fast-path (`len(h.blocklist) == 0`) and includes a transaction from address `X`.
6. Validator A runs `ProcessProposalHandler` for block N+1. `len(h.blocklist) != 0`, so it checks the tx from `X`, finds it blocked, and returns `REJECT`.
7. Validator B (and any other validator with an empty blocklist) returns `ACCEPT` for the same block.
8. The validator set is split; CometBFT cannot reach 2/3+ agreement → **chain halt**. [4](#0-3) [7](#0-6)

// SetBlockList don't fail if the identity is not set or the block list is empty.
func (h *ProposalHandler) SetBlockList(blob []byte) error {
	if h.Identity == nil {
		return nil
	}

	if bytes.Equal(h.lastBlockList, blob) {
		return nil
	}
	h.lastBlockList = make([]byte, len(blob))
	copy(h.lastBlockList, blob)

	if len(blob) == 0 {
		h.blocklist = make(map[string]struct{})
		return nil
	}

	reader, err := age.Decrypt(bytes.NewBuffer(blob), h.Identity)

func (h *ProposalHandler) ProcessProposalHandler() sdk.ProcessProposalHandler {
	return func(ctx sdk.Context, req *abci.RequestProcessProposal) (*abci.ResponseProcessProposal, error) {
		if len(h.blocklist) == 0 {
			// fast path, accept all txs
			return &abci.ResponseProcessProposal{Status: abci.ResponseProcessProposal_ACCEPT}, nil
		}

		for _, txBz := range req.Txs {
			if err := h.ValidateTransaction(nil, txBz); err != nil {
				return &abci.ResponseProcessProposal{Status: abci.ResponseProcessProposal_REJECT}, nil
			}
		}

		return &abci.ResponseProcessProposal{Status: abci.ResponseProcessProposal_ACCEPT}, nil
	}
}
