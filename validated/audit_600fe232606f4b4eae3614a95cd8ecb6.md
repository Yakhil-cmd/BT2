### Title
Missing Role-Based Authorization on Chunk Data Pack Requests Allows Any Staked Node to Exfiltrate Execution State - (File: `engine/execution/provider/engine.go`)

### Summary

The Execution Node's provider engine responds to `ChunkDataRequest` messages by delivering a `ChunkDataResponse` (containing the full `ChunkDataPack`, including the transaction collection) to **any staked, non-ejected node** that sends a request, regardless of whether the requester is a Verification Node. The networking layer only checks that the sender is a known, non-ejected participant; no role check is performed at the application layer before the chunk data pack is retrieved from storage and unicast back to the requester. This is the direct analog of `window.postMessage(this.eventMessage, '*')`: sensitive execution-state data is dispatched to any origin without restricting the recipient to the intended role.

### Finding Description

In `engine/execution/provider/engine.go`, `onChunkDataRequest` is the handler that processes every inbound `ChunkDataRequest`:

```go
// TODO(ramtin): we might add a future logic to do extra checks on the origin of the request
// currently the networking layer checks that the requested is a valid node operator
// that has not been ejected.
e.metrics.ChunkDataPackRequestProcessed()
chunkDataPack, err := e.execState.ChunkDataPackByChunkID(request.ChunkId)
...
e.deliverChunkDataResponse(chunkDataPack, request.RequesterId)
```

The comment explicitly acknowledges that **no role check is performed**. The only guard is the generic networking-layer check (`AuthorizedSenderValidator`) that verifies the sender is a staked, non-ejected node — it does not restrict the message to Verification Nodes.

The `Engine` struct holds `checkAuthorizedAtBlock` (used only in `BroadcastExecutionReceipt` to check whether the *local* execution node is authorized to broadcast), but this function is **never called** inside `onChunkDataRequest` to validate the *requester's* role.

The channel `channels.ProvideChunks` is registered for this engine. Any staked node (e.g., a Collection Node, Access Node, or another Execution Node) that can send a `ChunkDataRequest` on that channel will receive the full `ChunkDataPack` in response, including the embedded `Collection` (the raw transaction bodies for that chunk).

The attacker-controlled entry path:
1. Attacker operates any staked, non-ejected Flow node (e.g., a Collection Node — a role reachable without privileged assumptions).
2. Attacker sends a `flow.ChunkDataRequest{ChunkID: <target_chunk_id>}` on `channels.RequestChunks` / `channels.ProvideChunks` to a target Execution Node.
3. `onChunkDataRequest` retrieves the chunk data pack from storage and calls `deliverChunkDataResponse`, unicasting the full `ChunkDataPack` back to the attacker's node ID.
4. The attacker receives the `ChunkDataResponse` containing the `Collection` (all transaction bodies for that chunk) and the `StartState` commitment.

### Impact Explanation

A staked node of any role can request and receive chunk data packs — which contain the full transaction collections (raw transaction bodies) for any executed block — from Execution Nodes. This constitutes **private data exposure**: transaction-level execution state that is only supposed to be served to authorized Verification Nodes is disclosed to any staked participant. An attacker operating a legitimately staked Collection Node (the cheapest staking tier) can enumerate chunk IDs from public execution receipts and systematically harvest all transaction collections from Execution Nodes, bypassing the intended access control boundary.

Impact: 3 — Sensitive execution-state data (transaction collections embedded in chunk data packs) is exposed to any staked node, not just Verification Nodes.

### Likelihood Explanation

Likelihood: 3 — Staking a Collection Node is a realistic, documented path for network participants. Chunk IDs are derivable from publicly broadcast execution receipts. The attack requires no privileged keys, no social engineering, and no cryptographic break. The TODO comment in the code confirms the missing check is a known, deferred gap.

### Recommendation

In `onChunkDataRequest`, before retrieving and delivering the chunk data pack, validate that `request.RequesterId` is an authorized Verification Node at the block referenced by the chunk:

```go
authorized := e.isAuthorizedVerificationNodeAtBlock(request.RequesterId, chunkBlockID)
if !authorized {
    lg.Warn().Msg("rejecting chunk data request from unauthorized requester")
    return
}
```

This mirrors the check already implemented on the receiving side in `engine/verification/fetcher/engine.go` (`validateAuthorizedExecutionNodeAtBlockID`), which validates that chunk data pack *responses* come from authorized Execution Nodes. The same pattern should be applied on the *sending* side to restrict who may *request* chunk data packs.

```go
// validateAuthorizedExecutionNodeAtBlockID validates sender ID of a chunk data pack response as an authorized
// execution node at the given block ID.
func (e Engine) validateAuthorizedExecutionNodeAtBlockID(senderID flow.Identifier, blockID flow.Identifier) bool {
	snapshot := e.state.AtBlockID(blockID)
	valid, err := protocol.IsNodeAuthorizedWithRoleAt(snapshot, senderID, flow.RoleExecution)

	if err != nil {
		e.log.Fatal().
			Err(err).
			Hex("block_id", logging.ID(blockID)).
			Hex("sender_id", logging.ID(senderID)).
			Msg("could not validate sender identity at specified block ID snapshot as execution node")
	}

	return valid
}
```
