Let me examine the exact index computation and how shard layouts change across resharding boundaries.

### Title
Stale `target_shard_index` in `GetExecutionOutcome` produces wrong `outcome_root_proof` across resharding epoch boundary — (`chain/client/src/view_client_actor.rs`)

### Summary

`GetExecutionOutcome` computes `target_shard_index` once from the **pre-resharding** epoch's shard layout, then uses that index to select the Merkle path from the **post-resharding** block's `outcome_roots` array. After a dynamic resharding split, every shard whose position in the layout comes **after** the split shard has its index incremented by 1 in the new layout. The stale index therefore selects the wrong `prev_outcome_root` entry, producing an `outcome_root_proof` that fails `verify_path` against the block header's `outcome_root`.

### Finding Description

In `view_client_actor.rs`, the handler for `GetExecutionOutcome`:

```
epoch_id        ← epoch of the block where the outcome was recorded  (pre-resharding)
shard_layout    ← pre-resharding layout
target_shard_id ← account_id_to_shard_id(..., &epoch_id)            (pre-resharding shard)
target_shard_index ← shard_layout.get_shard_index(target_shard_id)  (pre-resharding index)

(h, target_shard_id) ← get_next_block_hash_with_new_chunk(...)      (post-resharding block)

outcome_roots ← block(h).chunks().map(|c| c.prev_outcome_root())    (post-resharding order)
outcome_root_proof ← merklize(&outcome_roots).1[target_shard_index] (WRONG index!)
``` [1](#0-0) 

`target_shard_index` is never recomputed after `get_next_block_hash_with_new_chunk` returns. The returned block `h` lives in the new epoch; its chunks are ordered by the **new** shard layout.

**How shard indices shift after a split** — `derive_impl` in `ShardLayoutV3` splices the parent shard out of `shard_ids` at position `new_boundary_idx` and inserts two children in its place:

```rust
let [last_split] = shard_ids
    .splice(new_boundary_idx..new_boundary_idx + 1, new_shards.clone())
    ...;
``` [2](#0-1) 

Every shard that was at index `> new_boundary_idx` in the old layout is now at index `old_index + 1` in the new layout. The test data confirms this concretely:

- Old layout: `[1, 3, 4]` — shard 3 at index 1, shard 4 at index 2
- Shard 1 splits → new layout: `[5, 6, 3, 4]` — shard 3 now at index **2**, shard 4 now at index **3** [3](#0-2) 

For a receipt executed in shard 3 (old index 1):
- `target_shard_index` = 1 (stale)
- `outcome_roots[1]` = `prev_outcome_root` of the chunk for shard 6 (not shard 3)
- The returned `outcome_root_proof` is the Merkle path for the **wrong** shard

For non-split shards, `get_children_shards_ids(s)` returns `[s]` (trivial child), so `get_next_block_hash_with_new_chunk` returns `(block_hash, s)` — the same shard ID — but the index mismatch is still present. [4](#0-3) 

For the **parent shard** case specifically asked in the question: the first child C1 is inserted at the parent's old index, so if C1 is returned the index is accidentally correct. The mismatch only occurs if C2 is returned (requires C1's chunk to be missing), which is not directly user-controllable. The **non-split shard** case is unconditionally broken for any shard positioned after the split shard.

### Impact Explanation

The exact corrupted value is `outcome_root_proof = merklize(&outcome_roots).1[old_index]` where `old_index` is off by 1 (or more) relative to the shard's true position in the post-resharding block. The three-part light client proof assembled by `light_client_proof_local` will fail `verify_path` at the `outcome_root_proof` step:

```python
block_outcome_root = compute_root(sha256(borsh(shard_outcome_root)), outcome_root_proof)
# block_outcome_root ≠ block_header_lite.inner_lite.outcome_root  → proof invalid
``` [5](#0-4) 

Any application relying on `EXPERIMENTAL_light_client_proof` (cross-chain bridges, light clients) will reject proofs for transactions in shards that follow the split shard in the layout.

### Likelihood Explanation

Triggered automatically whenever dynamic resharding (V3) splits a shard that is not the last shard in the layout. Any user with an account in a shard positioned after the split shard can observe the failure by submitting a transaction before the epoch boundary and calling `GetExecutionOutcome` afterward. No special privileges are required.

### Recommendation

After `get_next_block_hash_with_new_chunk` returns `(h, target_shard_id)`, recompute `target_shard_index` using the epoch of block `h`:

```rust
if let Some((h, target_shard_id)) = res {
    let new_epoch_id = *self.chain.get_block(&h)?.header().epoch_id();
    let new_shard_layout = self.epoch_manager.get_shard_layout(&new_epoch_id).into_chain_error()?;
    let target_shard_index = new_shard_layout
        .get_shard_index(target_shard_id)
        .map_err(Into::into)
        .into_chain_error()?;
    // ... rest of handler
``` [6](#0-5) 

### Proof of Concept

1. Configure a test-loop with dynamic resharding enabled and a layout where shard 1 (at index 0) will split, leaving shards 3 and 4 (at indices 1 and 2) to shift.
2. Submit a transaction from an account in shard 3 in the epoch before the resharding boundary.
3. After the boundary, call `GetExecutionOutcome` for that transaction.
4. Assert that `verify_path(block_outcome_root, &outcome_root_proof, &chunk_outcome_root)` returns `false` — the proof selects `outcome_roots[1]` (shard 6's root) instead of `outcome_roots[2]` (shard 3's root).

The existing test harness in `test-loop-tests/src/tests/light_client.rs` (`check_outcome_proofs`) already performs exactly this `verify_path` assertion and would catch the failure if run across a resharding boundary with a non-terminal split shard. [7](#0-6)
