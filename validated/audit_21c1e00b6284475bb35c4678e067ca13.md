### Title
`compute_gas_refund` floors `delta_gas` when converting native resource usage to gas, causing gas estimation to underestimate the required gas limit — (`basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`)

---

### Summary

`compute_gas_refund` uses integer floor division when computing `delta_gas = native_used / native_per_gas`. This causes the reported `gas_used` (and therefore any `eth_estimateGas` result) to be at most 1 gas unit too low. When a caller re-submits a transaction using exactly that gas limit, the derived native limit `native_per_gas × gas_used` is strictly less than `native_used` whenever `native_per_gas` does not evenly divide `native_used`, causing the transaction to run out of native resources and revert.

---

### Finding Description

`compute_gas_refund` adjusts the EVM gas-used figure upward to account for native resource consumption:

```rust
// basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs  lines 69-79
let delta_gas = if native_per_gas == 0 {
    0
} else {
    (native_used / native_per_gas) as i64 - (gas_used as i64)   // ← floor division
};

if delta_gas > 0 {
    gas_used += delta_gas as u64;
}
```

The final `gas_used` therefore equals `max(gas_used_from_ergs, ⌊native_used / native_per_gas⌋)`.

The design intent (documented in `docs/double_resource_accounting.md` line 48) is that `delta_gas` bridges the gap between EVM gas consumed and the gas equivalent of native resources consumed, so that a subsequent execution with `gas_limit = gas_used` will have a native budget large enough to cover the same workload:

```
nativeLimit = native_per_gas × gas_limit
            = native_per_gas × ⌊native_used / native_per_gas⌋
            ≤ native_used                                        ← strict inequality when remainder ≠ 0
```

Because `native_per_gas × ⌊N / native_per_gas⌋ < N` whenever `native_per_gas ∤ N`, the native budget for the re-submitted transaction is insufficient to reproduce the same execution, and the transaction reverts with out-of-native-resources.

The parallel with the external report is exact: `compute_offer_amount` floors the required input amount so the resulting swap yields less than the requested output; `compute_gas_refund` floors the required gas so the resulting execution runs out of native resources.

`native_per_gas` itself is computed with **ceiling** division (`gas_price.div_ceil(native_price)`) in both the ZK and Ethereum validation paths, which makes the native budget per gas unit slightly larger than the true ratio. This does not compensate for the floor in `delta_gas`; it only reduces the magnitude of the shortfall.

---

### Impact Explanation

Any caller that:
1. Simulates a transaction (e.g. via `eth_estimateGas` / `forward_system` simulation), and
2. Re-submits with `gas_limit` set to the returned `gas_used`,

will have their transaction revert with out-of-native-resources whenever `native_per_gas ∤ native_used`. This is the common case: `native_per_gas` is typically a small integer (e.g. 10 when `gas_price = 1000` and `native_price = 100`), and `native_used` is an arbitrary counter that is almost never a multiple of `native_per_gas`.

The shortfall is at most `native_per_gas − 1` native units, which translates to exactly 1 gas unit of underestimation. The user must add 1 gas unit to the estimate to guarantee success — but standard tooling (wallets, SDKs) that trusts `eth_estimateGas` will not do this, leading to systematic unexpected reverts.

---

### Likelihood Explanation

The condition `native_per_gas ∤ native_used` holds for the vast majority of transactions. `native_per_gas` is derived from `gas_price / native_price` (ceiling), and `native_used` accumulates from many independent RISC-V cycle counts. The probability that the two are exactly divisible is `1 / native_per_gas`, so for `native_per_gas = 10` the bug triggers ~90 % of the time. Any unprivileged user submitting a transaction after a standard gas estimation call can trigger this.

---

### Recommendation

Replace the floor division with ceiling division in the `delta_gas` calculation:

```rust
// basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs
let delta_gas = if native_per_gas == 0 {
    0
} else {
-   (native_used / native_per_gas) as i64 - (gas_used as i64)
+   native_used.div_ceil(native_per_gas) as i64 - (gas_used as i64)
};
```

This guarantees `native_per_gas × gas_used ≥ native_used`, so a re-submitted transaction with `gas_limit = gas_used` always has a sufficient native budget.

---

### Proof of Concept

Concrete parameters:
- `gas_price = 1000`, `native_price = 100`
- `native_per_gas = ⌈1000 / 100⌉ = 10`
- `gas_limit = 200`, `native_limit = 10 × 200 = 2000`
- After execution: `native_used = 1999`, `gas_used_from_ergs = 150`
- `delta_gas = ⌊1999 / 10⌋ − 150 = 199 − 150 = 49`
- Reported `gas_used = 150 + 49 = 199`

Re-submit with `gas_limit = 199`:
- `native_limit = 10 × 199 = 1990`
- Execution needs `native_used = 1999 > 1990` → **out-of-native-resources revert**

With the fix (`⌈1999 / 10⌉ = 200`):
- `delta_gas = 200 − 150 = 50`, `gas_used = 200`
- Re-submit with `gas_limit = 200`: `native_limit = 2000 ≥ 1999` → **succeeds**
