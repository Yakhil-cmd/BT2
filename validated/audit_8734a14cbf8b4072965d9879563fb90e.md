### Title
Gateway Stateful Validator Admits Transactions Below Configured Gas-Price Threshold Due to Integer Truncation in Ratio Comparison — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

`validate_tx_l2_gas_price_within_threshold` computes the minimum acceptable L2 gas price by multiplying the previous block's gas price by a configured percentage and then calling `Ratio::to_integer()`, which **truncates (floors)** the result. The subsequent `<` comparison against the truncated integer allows transactions whose true ratio is below the configured threshold to pass admission. This is the direct Sequencer analog of the external "imprecise ratio comparison via division" bug.

---

### Finding Description

The vulnerable function is `validate_tx_l2_gas_price_within_threshold` in `crates/apollo_gateway/src/stateful_transaction_validator.rs`:

```rust
let gas_price_threshold_multiplier =
    Ratio::new(self.config.min_gas_price_percentage.into(), 100_u128);
let threshold = (gas_price_threshold_multiplier
    * previous_block_l2_gas_price.get().0)
    .to_integer();
if tx_l2_gas_price.0 < threshold {
    return Err(...)
}
```

`Ratio::to_integer()` from `num_rational` returns the **floor** of the rational value. The threshold is therefore:

```
threshold = floor(min_gas_price_percentage × previous_block_l2_gas_price / 100)
```

The admission check is `tx_l2_gas_price >= threshold`. Because the threshold is floored, a transaction whose true ratio `tx_l2_gas_price / previous_block_l2_gas_price` is strictly less than `min_gas_price_percentage / 100` can still satisfy `tx_l2_gas_price >= floor(...)` and be admitted.

**Concrete example** (`min_gas_price_percentage = 90`, `previous_block_l2_gas_price = 101`):

| Value | Computation |
|---|---|
| True threshold | 90 × 101 / 100 = 90.9 |
| Computed threshold | floor(90.9) = **90** |
| Attacker's gas price | **90** |
| Check `90 >= 90` | **passes** |
| True ratio | 90 / 101 ≈ **89.1%** < 90% |

The transaction is admitted even though its gas price is only 89.1% of the previous block price, violating the 90% threshold invariant.

The correct comparison avoids division entirely:

```rust
// Equivalent to: tx_price / prev_price >= percentage / 100
// Cross-multiply: tx_price * 100 >= percentage * prev_price
if tx_l2_gas_price.0.checked_mul(100)
    < Some(u128::from(self.config.min_gas_price_percentage)
        .checked_mul(previous_block_l2_gas_price.get().0)?)
{
    return Err(...)
}
``` [1](#0-0) 

The configuration field `min_gas_price_percentage` is a `u8` that can be set to any value from 0 to 255; the default is 100, but the deployed configs and documentation explicitly support values like 80 or 90. [2](#0-1) 

The existing test suite only exercises cases where `previous_block_l2_gas_price` is a round multiple of 100 (e.g., 100 with 50% → threshold 50), so the truncation error is never triggered in tests. [3](#0-2) 

---

### Impact Explanation

The gateway stateful path is the last admission gate before a transaction enters the mempool. A transaction that passes this check with a gas price below the true threshold is admitted to the mempool and will be sequenced. This violates the invariant that every admitted transaction's `max_price_per_unit` is at least `min_gas_price_percentage%` of the previous block's L2 gas price — the exact invariant the check is designed to enforce.

**Impact category:** High — Mempool/gateway admission accepts an invalid transaction (one whose gas price is below the configured threshold) before sequencing.

---

### Likelihood Explanation

- The bug is triggered whenever `min_gas_price_percentage × previous_block_l2_gas_price` is not divisible by 100, which is the common case for any non-round gas price.
- Any unprivileged user can observe the previous block's L2 gas price (it is public), compute the floored threshold, and submit a transaction at exactly that value.
- The window is at most `min_gas_price_percentage / 100` units below the true threshold — small in absolute terms but deterministic and always exploitable.
- The default `min_gas_price_percentage = 100` produces no error (floor(100 × P / 100) = P exactly), but any sub-100 configuration (e.g., 80 or 90, which are explicitly documented as example values) is vulnerable.

---

### Recommendation

Replace the division-based threshold computation with a cross-multiplication comparison, eliminating truncation entirely:

```rust
// Current (buggy):
let threshold = (Ratio::new(percentage as u128, 100_u128)
    * previous_block_l2_gas_price.get().0)
    .to_integer();
if tx_l2_gas_price.0 < threshold { ... }

// Fixed (exact, no division):
let lhs = tx_l2_gas_price.0.saturating_mul(100);
let rhs = (self.config.min_gas_price_percentage as u128)
    .saturating_mul(previous_block_l2_gas_price.get().0);
if lhs < rhs { ... }
```

This is the direct analog of the external report's recommended fix (`quote1 * base2 >= quote2 * base1`).

---

### Proof of Concept

```rust
// min_gas_price_percentage = 90, previous_block_l2_gas_price = 101
let percentage: u128 = 90;
let prev_price: u128 = 101;

// Current code:
let threshold = (percentage * prev_price) / 100; // = 9090 / 100 = 90 (truncated)
let tx_price: u128 = 90;
assert!(tx_price >= threshold); // 90 >= 90 → PASSES (admitted)

// True ratio check:
// tx_price / prev_price = 90 / 101 ≈ 89.1% < 90% → should FAIL

// Fixed cross-multiplication:
assert!(tx_price * 100 < percentage * prev_price); // 9000 < 9090 → correctly REJECTED
``` [4](#0-3)

    fn validate_tx_l2_gas_price_within_threshold(
        &self,

        match tx_resource_bounds {

                let tx_l2_gas_price = tx_resource_bounds.l2_gas.max_price_per_unit;
