### Title
Pool Admin Can Permanently Block `setPoolProtocolFee` by Setting `adminFeeDestination` to a USDC-Blacklisted Address — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`setPoolProtocolFee` (factory-owner-only) and `setPoolAdminFees` (pool-admin-only) both call `collectFees` on the pool **before** updating fee rates. `collectFees` uses a push pattern: it transfers pool tokens directly to `poolAdminFeeDestination[pool]` before clearing the fee accumulators. Because the pool admin controls `adminFeeDestination` via `setPoolAdminFeeDestination`, a semi-trusted pool admin can set that destination to a USDC-blacklisted address, causing every `collectFees` call to revert and permanently blocking the factory owner from updating protocol fees on that pool.

---

### Finding Description

**Push transfer in `collectFees`:**

In `MetricOmmPool.sol`, `collectFees` transfers tokens to `adminFeeDestination_` before clearing `notionalFeeToken0Scaled` / `notionalFeeToken1Scaled`: [1](#0-0) 

The state-clearing write happens only after all four transfers succeed: [2](#0-1) 

**`setPoolProtocolFee` embeds `collectFees` as a mandatory pre-step:** [3](#0-2) 

The call at lines 328–335 passes `poolAdminFeeDestination[pool]` as the admin fee destination. If that transfer reverts, the entire `setPoolProtocolFee` transaction reverts and the fee-rate update never lands.

**`setPoolAdminFees` has the same structure:** [4](#0-3) 

**Pool admin controls `adminFeeDestination` without restriction beyond non-zero:** [5](#0-4) 

Any non-zero address is accepted, including a USDC-blacklisted address.

**`collectPoolFees` (permissionless) is also blocked:** [6](#0-5) 

---

### Impact Explanation

Once the pool admin sets `adminFeeDestination` to a USDC-blacklisted address and admin fees are non-zero with accrued balances:

1. `collectPoolFees` (permissionless) reverts → protocol spread and notional fees are permanently stuck in the pool and cannot be swept to the factory treasury.
2. `setPoolProtocolFee` (factory owner) reverts → the factory owner loses the ability to adjust protocol fee rates on that pool for as long as the destination remains blacklisted.
3. `setPoolAdminFees` (pool admin) reverts → the pool admin also loses the ability to update their own fee rates (self-inflicted, but it locks the fee configuration).

The factory owner has no unilateral path to fix this: `setPoolAdminFeeDestination` is a pool-admin-only function, and the two-step admin transfer (`proposePoolAdminTransfer` / `acceptPoolAdmin`) requires the current admin's cooperation. The factory owner cannot force a destination change or bypass the `collectFees` pre-step.

This is an **admin-boundary break**: a semi-trusted pool admin uses their authority over `adminFeeDestination` to permanently block a higher-privileged factory-owner action, and to trap accrued protocol fees inside the pool.

---

### Likelihood Explanation

- Requires USDC (or USDT) as `token0` or `token1` — explicitly in scope per the contest rules.
- Requires a pool admin willing to act adversarially (semi-trusted, not fully trusted).
- Requires non-zero admin fee rates and at least some accrued fee balance so the transfer amount is `> 0`.
- No timelock or cap prevents the pool admin from setting any non-zero `adminFeeDestination`.
- The attack is permanent until the pool admin cooperates, making recovery difficult.

---

### Recommendation

Decouple fee collection from fee-rate updates. Two options:

1. **Pull pattern for admin fees:** Do not transfer to `adminFeeDestination` inside `collectFees`. Instead, credit a claimable balance mapping and let the admin pull their share separately. This mirrors the M-4 recommendation exactly.

2. **Skip `collectFees` in rate-update paths:** Allow `setPoolProtocolFee` and `setPoolAdminFees` to update rates without first collecting accrued fees. Collect fees lazily or in a separate permissionless call that cannot block governance actions.

---

### Proof of Concept

```
1. Pool is deployed with USDC as token0 and non-zero adminSpreadFeeE6.
2. Pool admin calls:
     factory.setPoolAdminFeeDestination(pool, USDC_BLACKLISTED_ADDR)
3. Swaps occur; spread surplus and notional fees accrue in the pool.
4. Factory owner calls:
     factory.setPoolProtocolFee(pool, newProtocolSpreadFeeE6, newProtocolNotionalFeeE8)
5. Internally:
     pool.collectFees(c.protocolSpreadFeeE6, c.adminSpreadFeeE6, ..., USDC_BLACKLISTED_ADDR)
       → transferToken0(USDC_BLACKLISTED_ADDR, totalFee0ToAdmin)   // USDC reverts
6. setPoolProtocolFee reverts. Protocol fee rates cannot be updated.
7. collectPoolFees(pool) also reverts for any caller.
8. notionalFeeToken0Scaled / notionalFeeToken1Scaled are never cleared;
   accrued fees remain locked in the pool indefinitely.
```
