### Title
Gateway `skip_stateful_validations` admits invoke transactions with arbitrary signatures without `__validate__` verification — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator unconditionally skips the account's `__validate__` entry point for any invoke transaction with nonce=1 when the account's on-chain nonce is 0 and the account has any transaction in the mempool. Because the `__validate__` entry point is the sole mechanism by which an account contract verifies the caller's signature, this allows an unprivileged attacker to inject an invoke transaction carrying an arbitrary (invalid) signature into the mempool for any victim account that has a pending deploy_account transaction.

### Finding Description

In `skip_stateful_validations` (lines 429–461 of `crates/apollo_gateway/src/stateful_transaction_validator.rs`), the gateway checks three conditions to decide whether to skip the blockifier's `__validate__` call:

1. The transaction is an Invoke.
2. `tx.nonce() == Nonce(Felt::ONE)`.
3. `account_nonce == Nonce(Felt::ZERO)` (account not yet deployed on-chain).
4. `mempool_client.account_tx_in_pool_or_recent_block(sender_address)` returns `true`. [1](#0-0) 

When all four conditions hold, `skip_stateful_validations` returns `true`. This value is passed directly to `run_validate_entry_point`, which sets `execution_flags.validate = !skip_validate = false`: [2](#0-1) 

With `validate = false`, `StatefulValidator::perform_validations` calls `perform_pre_validation_stage` (nonce/fee/balance/proof-facts checks) but then immediately returns `Ok(())` without ever calling `validate_tx` / `__validate__`: [3](#0-2) 

And `validate_tx` itself short-circuits on the flag: [4](#0-3) 

The `account_tx_in_pool_or_recent_block` check is not specific to deploy_account transactions; it returns `true` for any account that has any transaction in the pool or is known from a recent committed block: [5](#0-4) 

**Attack path:**

1. Victim submits a valid `deploy_account` for address X (pre-funded with STRK). The deploy_account enters the mempool.
2. Attacker observes address X in the mempool (public mempool state).
3. Attacker submits `Invoke(sender=X, nonce=1, calldata=<arbitrary>, signature=<garbage>)`.
4. Gateway stateless checks pass (signature length, resource bounds, etc. — no cryptographic check).
5. `validate_nonce` passes: nonce=1 ≥ account_nonce=0, within `max_allowed_nonce_gap`.
6. `validate_by_mempool` passes: mempool checks nonce ordering and fee escalation only — no signature check.
7. `skip_stateful_validations` returns `true` because `account_tx_in_pool_or_recent_block(X)` is `true` (victim's deploy_account is there).
8. `run_validate_entry_point` sets `validate=false`; `__validate__` is never called.
9. The attacker's transaction is accepted into the mempool.

The mempool enforces `DuplicateNonce` for the same `(address, nonce)` pair: [6](#0-5) 

Therefore, if the attacker's transaction occupies nonce=1 first, the victim's legitimate invoke(nonce=1) is rejected with `MempoolError::DuplicateNonce` until the attacker's transaction is executed (and reverts) or replaced via fee escalation.

During batcher execution, `new_for_sequencing` sets `validate: true`, so `__validate__` IS called and the attacker's transaction reverts. However, the validation-phase fee is still charged from the victim's pre-funded account: [7](#0-6) 

### Impact Explanation

**High — Mempool/gateway admission accepts invalid transactions before sequencing.**

Concrete harms:
- **Nonce squatting / DoS**: An attacker can front-run any victim's first post-deploy invoke by occupying nonce=1 with an invalid-signature transaction. The victim's legitimate invoke is rejected by the mempool until the attacker's transaction is processed and the nonce advances.
- **Fee drain**: The reverted transaction charges the victim's pre-funded account for the validation-phase gas, draining funds the victim deposited to cover deployment costs.
- **Mempool pollution**: Any account with a pending deploy_account is a target; the attacker needs only to observe the mempool and submit a single transaction per victim.

### Likelihood Explanation

The trigger is straightforward and requires no privileged access:
- The attacker only needs to observe the public mempool for pending deploy_account transactions.
- The attacker submits a single invoke with nonce=1 and any signature.
- No special knowledge of the victim's private key or account class is required.
- The condition (`nonce=1`, `account_nonce=0`, `account_in_pool=true`) is reliably satisfied during the deploy_account + invoke UX window.

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a **deploy_account** transaction exists for the sender address in the mempool. Alternatively, add a separate mempool API `deploy_account_tx_in_pool(address) -> bool` that only returns `true` when the pending transaction for that address is of type `DeployAccount`. This preserves the UX feature while ensuring that the skip-validate path is only triggered when the account's own deploy_account is pending — not when an attacker has injected an arbitrary transaction for the address.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
