### Title
Malicious Participant Can Corrupt CKD Output via Unvalidated Share Contributions — (`File: src/confidential_key_derivation/protocol.rs`)

---

### Summary
The CKD coordinator in `do_ckd_coordinator` blindly accumulates `(big_y, big_c)` values received from all participants with no proof of correct computation. A single malicious participant can inject arbitrary elliptic-curve points, causing the coordinator to output a cryptographically wrong derived key that honest parties will silently accept as valid.

---

### Finding Description

In `do_ckd_coordinator`, the coordinator receives each participant's `CKDOutput` and unconditionally adds it to the running sum:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
``` [1](#0-0) 

Each honest participant is supposed to compute and send `(λ_i · y_i · G, λ_i · (x_i · H(pk, app_id) + y_i · A))`. The coordinator has no mechanism to verify that any received pair was honestly computed. There is no zero-knowledge proof, no commitment scheme, and no consistency check on the received points before they are folded into the final output. [2](#0-1) 

The honest protocol invariant is:

```
C_final - app_sk · Y_final  =  msk · H(pk, app_id)
```

A malicious participant who sends `(big_y + δ_Y, big_c + δ_C)` for arbitrary curve points `δ_Y, δ_C` breaks this invariant, producing:

```
C_final - app_sk · Y_final  =  msk · H(pk, app_id) + δ_C - app_sk · δ_Y
```

The coordinator has no way to detect the deviation and returns the corrupted `CKDOutput` as if it were correct. [3](#0-2) 

The analog to the external report is direct: just as the token-logo endpoint blindly forwarded every HTTP header from an untrusted external server into the API response (allowing `Content-Type: text/html` injection), the CKD coordinator blindly forwards every received curve point from an untrusted participant into the final output (allowing arbitrary point injection).

---

### Impact Explanation

The coordinator outputs a `CKDOutput` whose `unmask(app_sk)` value is not `msk · H(pk, app_id)` but an attacker-controlled offset of it. Any downstream consumer of the derived key — e.g., a wallet or application that uses the CKD result to derive a child key — silently receives a wrong key with no indication of failure. This constitutes **corruption of a CKD output so honest parties accept an unusable or attacker-biased cryptographic output**, matching the High impact tier. [4](#0-3) 

---

### Likelihood Explanation

Any single participant in the CKD session is a sufficient attacker. The participant role requires no special privilege beyond being listed in the `participants` array passed to `ckd()`. The attack requires only sending two arbitrary `G1` points instead of the honestly computed ones — a trivial deviation from the protocol that requires no cryptographic capability. [5](#0-4) 

---

### Recommendation

Require each participant to accompany their `(big_y, big_c)` contribution with a non-interactive zero-knowledge proof of correct computation (e.g., a Schnorr-style proof that `big_y = y · G` and `big_c = x_i · H(pk, app_id) + y · A` for the same witness `y` and the committed public share `x_i · G2`). The coordinator must verify all proofs before accumulating any contribution. Alternatively, use a verifiable secret-sharing approach where each participant's contribution can be checked against their public key share. [6](#0-5) 

---

### Proof of Concept

1. Honest participants `P_1, …, P_{n-1}` run `compute_signature_share` correctly and send `(norm_big_y_i, norm_big_c_i)` to the coordinator.
2. Malicious participant `P_n` instead sends `(δ_Y, δ_C)` for arbitrary non-identity `G1` points `δ_Y, δ_C` that satisfy `δ_C ≠ app_sk · δ_Y` (attacker does not know `app_sk`).
3. The coordinator loop at lines 50–55 of `protocol.rs` adds `δ_Y` and `δ_C` to the running sums without any check.
4. The returned `CKDOutput` satisfies `ckd_output.unmask(app_sk) ≠ msk · H(pk, app_id)`.
5. The honest coordinator and all downstream consumers accept this wrong value as the legitimate derived key, with no error returned. [2](#0-1)

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}
