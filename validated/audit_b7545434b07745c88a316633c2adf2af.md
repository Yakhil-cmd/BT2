### Title
Gateway `skip_stateful_validations` Admits Invalid-Signature Invoke Transactions to Mempool — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` UX feature, designed to allow a `deploy_account + invoke` pair to be submitted atomically, unconditionally skips the blockifier `__validate__` entry-point call for any Invoke V3 transaction with `nonce == 1` when the sender address already appears in the mempool. Because neither the stateless validator nor the mempool checks cryptographic signature validity, an attacker can submit a valid `deploy_account` followed by an Invoke with a completely forged signature; the gateway accepts the forged Invoke into the mempool without ever running `__validate__`.

### Finding Description

**Exact code path:**

`StatelessTransactionValidator::validate` checks only signature *length*, not validity: [1](#0-0) 

`run_pre_validation_checks` calls `skip_stateful_validations` after the mempool nonce/fee check: [2](#0-1) 

`skip_stateful_validations` returns `true` (skip) whenever `tx.nonce() == 1 && account_nonce == 0 && account_tx_in_pool_or_recent_block(sender) == true`: [3](#0-2) 

When `skip_validate == true`, `run_validate_entry_point` sets `execution_flags.validate = false` and the blockifier's `perform_validations` returns `Ok(())` immediately without calling `__validate__`: [4](#0-3) [5](#0-4) 

`validate_by_mempool` (called before the skip decision) only checks nonce ordering and fee-escalation rules — no signature field exists in `ValidationArgs`: [6](#0-5) [7](#0-6) 

**Attack steps (unprivileged):**

1. Compute a deterministic account address (class hash + salt + constructor calldata).
2. Submit a valid `deploy_account` transaction — passes full execution validation, enters the mempool, and causes `account_tx_in_pool_or_recent_block` to return `true`.
3. Submit an Invoke V3 with `nonce = 1` and an **arbitrary/forged signature** (length ≤ `max_signature_length`).
4. Gateway path: stateless length check passes → `convert_rpc_tx_to_internal` computes tx hash → stateful validator reads `account_nonce = 0` from state → `validate_by_mempool` passes (nonce 1 ≥ 0, no duplicate) → `skip_stateful_validations` returns `true` → `run_validate_entry_point` skips `__validate__` → transaction is forwarded to the mempool via `mempool_client.add_tx`. [8](#0-7) 

The forged-signature Invoke is now a live entry in the mempool.

**Why the comment's reasoning is insufficient:**

The inline comment states: *"It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."*

This reasoning only guarantees that *some* prior transaction from the address passed validation — it says nothing about the *current* Invoke's signature. The prior transaction (the `deploy_account`) was validated with a completely different key-pair check (constructor execution), so its validity provides no assurance about the nonce-1 Invoke's signature.

### Impact Explanation

An invalid (forged-signature) Invoke transaction is admitted to the mempool — a direct violation of the invariant that every mempool entry must carry a signature that will pass `__validate__` at execution time. This matches the **High** impact category: *"Mempool/gateway/RPC admission accepts invalid transactions … before sequencing."*

Concrete consequences:
- The forged Invoke occupies a mempool slot and is returned by `get_txs` to the batcher.
- The batcher executes the `deploy_account` (account is deployed), then attempts the Invoke; `__validate__` fails with an invalid-signature revert. The transaction is included in the block as a reverted entry and the fee is charged to the account.
- An attacker who controls the account's private key can repeatedly replace the nonce-1 slot via fee escalation, keeping a permanently invalid transaction in the mempool and forcing the batcher to waste execution resources on each block.
- The same forged Invoke is propagated to peer sequencers via P2P, causing each peer's gateway to re-admit it (the same skip logic applies on every node). [9](#0-8) 

### Likelihood Explanation

The trigger is fully unprivileged: any external user can craft a `deploy_account` + forged-signature Invoke pair and submit them to the public HTTP endpoint. No special role, key material, or prior state is required beyond the ability to compute a valid account address and pay the `deploy_account` fee. The condition `nonce == 1 && account_nonce == 0 && account_in_mempool` is trivially satisfied by the attacker's own submission.

### Recommendation

Before returning `skip_validate = true`, perform a lightweight off-chain signature pre-check against the account's expected public key (derivable from the `deploy_account` constructor calldata already in the mempool), or restrict the skip to only the *exact* transaction hash that was paired with the `deploy_account` at submission time. At minimum, document that the skip window intentionally accepts unverified signatures and add a batcher-side metric to track reverts caused by `__validate__` failures on nonce-1 transactions.

### Proof of Concept

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
