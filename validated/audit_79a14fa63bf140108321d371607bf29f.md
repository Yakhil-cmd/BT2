### Title
Recreated Account Silently Deleted in `StateDB.Commit` Due to Unconditional `continue` After `HasSelfDestructed` Check — (`fvm/evm/emulator/state/stateDB.go`)

---

### Summary

`StateDB.Commit` checks `HasSelfDestructed(addr)` first and unconditionally executes `continue` when true, permanently skipping the `IsCreated(addr)` branch. Because `DeltaView.CreateAccount` internally calls `SelfDestruct` (setting `toBeDestructed[addr]`) and then sets `created[addr]` in the same call, both flags are simultaneously true after a contract is deployed to an address that already carries a balance. `Commit` therefore calls `baseView.DeleteAccount` and skips `baseView.CreateAccount`, silently destroying the account and its carried-over balance while recording only a `DeleteAccount` in the `UpdateCommitter`.

---

### Finding Description

**Root cause — `StateDB.Commit` (stateDB.go lines 475–488):**

```go
if db.HasSelfDestructed(addr) {
    err = db.baseView.DeleteAccount(addr)
    ...
    deleted = true
}
if deleted {
    continue          // ← unconditionally skips IsCreated check
}
...
if db.IsCreated(addr) {   // ← never reached when deleted == true
    err = db.baseView.CreateAccount(addr, bal, nonce, code, codeHash)
    ...
}
``` [1](#0-0) 

**How both flags become simultaneously true — `DeltaView.CreateAccount` (delta.go lines 119–161):**

When `CreateAccount` is called on an address that already exists (e.g., it holds a pre-funded balance) and has not yet been self-destructed:

1. `IsCreated(addr)` → false → does not return early.
2. `Exist(addr)` → true.
3. `HasSelfDestructed(addr)` → false → enters the `if !destructed` branch.
4. Calls `d.SelfDestruct(addr)` → writes `toBeDestructed[addr]` and zeroes `balances[addr]`.
5. Restores `balances[addr]` to the carried-over balance.
6. Sets `recreated[addr]` and `created[addr]`. [2](#0-1) 

After this call: `toBeDestructed[addr]` is set **and** `created[addr]` is set.

**`HasSelfDestructed` is blind to `recreated` (delta.go lines 195–201):**

```go
func (d *DeltaView) HasSelfDestructed(addr gethCommon.Address) (bool, *uint256.Int) {
    bal, found := d.toBeDestructed[addr]
    if found {
        return true, bal   // ← no check of d.recreated
    }
    return d.parent.HasSelfDestructed(addr)
}
``` [3](#0-2) 

`toBeDestructed` is never cleared by `CreateAccount`, so `HasSelfDestructed` returns `true` even after the account has been recreated.

---

### Impact Explanation

For every address where a contract is deployed on top of a pre-existing balance (a standard EVM pattern — send ETH to a CREATE2 address before deploying):

| Step | Expected | Actual |
|---|---|---|
| `baseView.DeleteAccount(addr)` | called | called ✓ |
| `baseView.CreateAccount(addr, bal, …)` | called | **never called** ✗ |
| `updateCommitter.CreateAccount(…)` | called | **never called** ✗ |

Consequences:
- **Carried-over balance is permanently destroyed** — `DeleteAccount` removes the pre-funded balance from `baseView`; `CreateAccount` is never called to restore it.
- **Contract code is not persisted** — `baseView` has no record of the new contract after `Finalize`.
- **Commitment diverges** — `UpdateCommitter` records only `DeleteAccount`, omitting the `CreateAccount`, so the commitment hash does not reflect the actual intended post-state. [4](#0-3) 

---

### Likelihood Explanation

The trigger is a routine EVM operation: deploying a contract (via CREATE or CREATE2) to an address that already holds a balance. This is explicitly anticipated by the comment in `DeltaView.CreateAccount`:

> `// carrying over the balance. (legacy behavior of the Geth stateDB)` [5](#0-4) 

No special attacker privileges are required. Any EVM transaction that deploys a contract to a pre-funded address triggers the bug. The `recreated` map exists precisely to track this case for storage-slot handling, but `Commit` never consults it. [6](#0-5) 

---

### Recommendation

In `StateDB.Commit`, replace the unconditional `continue` with a combined check: if `HasSelfDestructed(addr)` **and** `IsCreated(addr)` are both true, call `baseView.DeleteAccount` followed immediately by `baseView.CreateAccount` (with the current balance/nonce/code), and record both operations in `UpdateCommitter`. Only skip to the next address when `HasSelfDestructed` is true **and** `IsCreated` is false.

Alternatively, clear `toBeDestructed[addr]` inside `DeltaView.CreateAccount` when the account is being recreated (after the internal `SelfDestruct` call), so that `HasSelfDestructed` correctly returns false for recreated accounts.

---

### Proof of Concept

```
1. In a prior transaction, send 1 ETH to address A
   (A has balance=1 ETH, nonce=0, no code in baseView).

2. In a new transaction, deploy a contract to A via CREATE2.
   EVM calls: CreateAccount(A) → internally SelfDestruct(A) + created[A]=true
   EVM calls: CreateContract(A), SetCode(A, bytecode)

3. Call StateDB.Commit(true).

4. Observe:
   - baseView.DeleteAccount(A) was called (balance destroyed).
   - baseView.CreateAccount(A, ...) was never called.
   - db.Exist(A) == false after Commit.
   - db.GetBalance(A) == 0 (1 ETH lost).
   - UpdateCommitter commitment contains only DeleteAccount(A),
     diverging from a reference implementation that records
     DeleteAccount(A) + CreateAccount(A, 1 ETH, 0, bytecode).
```
