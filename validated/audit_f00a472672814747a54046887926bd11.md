Audit Report

## Title
Unbounded tokio task spawning via crafted `StrippedBlockProposal` ingress/IDkg ID vectors — (`rs/p2p/artifact_downloader/src/fetch_stripped_artifact/assembler.rs`)

## Summary

A Byzantine peer below the fault threshold can serve a crafted `StrippedBlockProposal` whose `unstripped_consensus_message_id` matches a legitimately advertised block ID while carrying an arbitrarily large `ingress_messages` or `stripped_idkg_dealings` vector. `assemble_message` spawns one unbounded tokio task per entry before any size validation, and each task runs an infinite retry loop. The bouncer only cancels tasks after they are already live, and the attack is repeatable every consensus round, sustaining memory pressure that can degrade or halt consensus on the targeted replica.

## Finding Description

**Root cause — no vector length cap at deserialization or before task spawning.**

`TryFrom<pb::StrippedBlockProposal> for StrippedBlockProposal` iterates over `value.ingress_messages` and `value.stripped_idkg_dealings` with no length guard: [1](#0-0) 

The axum router serving these artifacts explicitly disables body size limits: [2](#0-1) 

**ID check is insufficient — only `unstripped_consensus_message_id` is validated.**

`download_artifact` in `fetch_artifact/download.rs` accepts a downloaded `MaybeStrippedConsensusMessage` if `message.id() == id`: [3](#0-2) 

For a `StrippedBlockProposal`, `id()` is derived solely from `unstripped_consensus_message_id`, not from the ingress vector: [4](#0-3) 

A Byzantine peer sets `unstripped_consensus_message_id` to a legitimately advertised block hash and inflates `ingress_messages` with N fake entries. The ID check passes; the vector length is never checked.

**One task spawned per entry, no cap.**

`assemble_message` calls `missing_stripped_messages()` and immediately spawns one `JoinSet` task per returned ID: [5](#0-4) 

`missing_stripped_messages()` returns every entry from both vectors unconditionally: [6](#0-5) 

The only comment bounding the vector size is a performance note ("at most 1000 elements") at line 440 — it is not enforced anywhere. [7](#0-6) 

**Each spawned task runs an infinite retry loop.**

`download_stripped_message` uses `with_max_elapsed_time(None)` and loops forever with no internal cancellation: [8](#0-7) [9](#0-8) 

**Bouncer cancellation is post-hoc.**

The bouncer check runs inside the join loop, after all N tasks are already spawned: [10](#0-9) 

The bouncer refresh period is 3 seconds: [11](#0-10) 

During that window all N tasks are live, each holding its own async stack frame, cloned `StrippedMessageId`, `ConsensusMessageId`, `Arc<dyn Transport>`, logger, metrics arc, and peer receiver. Dropping the `JoinSet` on bouncer fire aborts them, but the attack is repeatable every block interval.

**Final hash validation is post-hoc.**

`try_assemble` validates the reconstructed block hash only after all tasks complete, catching the fraud too late to prevent resource damage: [12](#0-11) 

## Impact Explanation

A single Byzantine replica below the f-fault threshold can force the victim to spawn O(N) tokio tasks per consensus round. At N = 100,000 fake ingress IDs, heap consumption is in the hundreds of MB range; at N = 1,000,000 it is plausible to trigger OOM on typical replica hardware. Because tasks loop indefinitely until the JoinSet is dropped, and the attack is repeatable every block interval (faster than the 3-second bouncer refresh), memory pressure is sustained. This matches the allowed High impact: **application/platform-level DoS, consensus blocking, or subnet availability impact not based on raw volumetric DDoS**.

## Likelihood Explanation

The attacker needs only a single Byzantine node in the same subnet — below the f-fault threshold. No key material, governance majority, or network-level capability is required. The crafted payload is a valid protobuf message with a legitimate `unstripped_consensus_message_id` and an inflated ID vector. The attack is repeatable every block interval (~1–2 seconds), faster than the 3-second bouncer refresh window.

## Recommendation

1. **Cap vector length before spawning**: After `missing_stripped_messages()` returns, assert the count is within a protocol-defined bound (e.g., `MAX_INGRESS_MESSAGES_PER_BLOCK + MAX_IDKG_DEALINGS_PER_BLOCK`). Return `AssembleResult::Unwanted` if exceeded.
2. **Enforce the cap at deserialization**: Add a length check inside `TryFrom<pb::StrippedBlockProposal>` so malformed payloads are rejected before reaching `assemble_message`.
3. **Re-enable body size limits**: Remove or scope `DefaultBodyLimit::disable()` to avoid accepting arbitrarily large artifact payloads over the wire.
