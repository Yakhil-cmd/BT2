### Title
Reentrancy in `claim_rewards` Allows Staker to Steal Unclaimed Yield via Malicious `reward_address` — (File: `src/staking/staking.cairo`)

---

### Summary

The `claim_rewards` function in `Staking.cairo` writes the updated staker info to storage **after** transferring rewards to the user-supplied `reward_address`. A staker who sets their `reward_address` to a malicious re-entrant contract can call `claim_rewards` repeatedly within a single transaction, draining the `RewardSupplier`'s balance far beyond their entitled rewards.

---

### Finding Description

In `claim_rewards` the execution order is:

1. Read `staker_info` from storage (including `unclaimed_rewards_own = X`)
2. Call `send_rewards_to_staker`, which:
   - Calls `claim_from_reward_supplier(amount = X)` — pulls X tokens from `RewardSupplier` into the staking contract
   - Calls `token_dispatcher.checked_transfer(recipient: reward_address, amount: X)` — **external call to attacker-controlled address**
   - Sets `staker_info.unclaimed_rewards_own = 0` in the **local variable only**
3. Call `write_staker_info` — writes the zeroed value to storage [1](#0-0) [2](#0-1) 

Because `write_staker_info` is called **after** the external transfer, storage still holds `unclaimed_rewards_own = X` during the transfer. A malicious `reward_address` contract can re-enter `claim_rewards(staker_address)` at that point. The re-entrant call:

- Reads `staker_info` from storage → `unclaimed_rewards_own = X` (unchanged)
- Passes the authorization check because `get_caller_address() == staker_info.reward_address` ✓
- Pulls another X from `RewardSupplier` and transfers it again

This can be repeated until the `RewardSupplier`'s `unclaimed_rewards` balance is exhausted.

**Contrast with `unstake_action`**, where the developers explicitly protected against this pattern with the comment *"This is done here to avoid re-entrancy"* and call `write_staker_info` **before** any token transfers: [3](#0-2) 

**Contrast with `Pool.claim_rewards`**, which also correctly writes state before transferring: [4](#0-3) 

The `reward_address` is freely settable by any staker via `change_reward_address`, with no whitelist or contract-type restriction beyond "not a token address": [5](#0-4) 

---

### Impact Explanation

**High — Theft of unclaimed yield.**

An attacker can drain the `RewardSupplier`'s `unclaimed_rewards` balance by repeatedly claiming the same staker rewards within one transaction. The stolen amount is bounded only by the `RewardSupplier`'s balance, not by the staker's legitimate entitlement. Other stakers and delegators who have not yet claimed may find the `RewardSupplier` insolvent, permanently freezing their unclaimed yield.

---

### Likelihood Explanation

**Medium.** The attacker only needs to:
1. Be a registered staker (unprivileged, permissionless action).
2. Call `change_reward_address` to point to a malicious re-entrant contract (unprivileged, no restrictions beyond "not a token address").
3. Call `claim_rewards` once to trigger the cascade.

No privileged role, leaked key, or external dependency compromise is required. The `RewardSupplier` routinely holds rewards for all active stakers, so it will almost always have a balance large enough to make the attack profitable.

---

### Recommendation

Move `write_staker_info` to **before** the call to `send_rewards_to_staker`, mirroring the pattern already used in `unstake_action` and `Pool.claim_rewards`:

```cairo
fn claim_rewards(ref self: ContractState, staker_address: ContractAddress) -> Amount {
    self.general_prerequisites();
    let mut staker_info = self.internal_staker_info(:staker_address);
    let caller_address = get_caller_address();
    let reward_address = staker_info.reward_address;
    assert!(
        caller_address == staker_address || caller_address == reward_address,
        "{}",
        Error::CLAIM_REWARDS_FROM_UNAUTHORIZED_ADDRESS,
    );
    let amount = staker_info.unclaimed_rewards_own;
    let token_dispatcher = strk_token_dispatcher();
    // Zero out rewards in storage BEFORE the external transfer.
    staker_info.unclaimed_rewards_own = Zero::zero();
    self.write_staker_info(:staker_address, :staker_info);
    // Now safe to call external contracts.
    self.send_rewards_to_staker(:staker_address, ref :staker_info, :token_dispatcher);
    amount
}
```

---

### Proof of Concept

1. Attacker deploys `MaliciousReward` contract:
   ```
   fn on_erc20_receive(...) {
       // Re-enter claim_rewards with the same staker_address
       IStaking(staking).claim_rewards(staker_address: VICTIM_STAKER);
   }
   ```
2. Attacker (as staker) calls `change_reward_address(reward_address: MaliciousReward)`.
3. Attacker calls `claim_rewards(staker_address: attacker_staker)`.
4. Execution flow:
   - Staking reads `unclaimed_rewards_own = X` from storage.
   - `claim_from_reward_supplier(X)` — RewardSupplier sends X to staking contract.
   - `checked_transfer(MaliciousReward, X)` — triggers `on_erc20_receive`.
   - **Re-entrant call**: storage still shows `unclaimed_rewards_own = X`; authorization passes; another X is pulled from RewardSupplier and sent to `MaliciousReward`.
   - Re-entrant call writes `unclaimed_rewards_own = 0` to storage.
   - Outer call writes `unclaimed_rewards_own = 0` (already 0, no harm).
5. Attacker receives `2X` instead of `X`. Repeat nesting for `N*X` total, limited by RewardSupplier balance.

        }
