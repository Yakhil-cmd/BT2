### Title
Gateway L2 Gas Price Admission Check Uses Current Block Price Instead of Next Block Price - (File: crates/apollo_gateway/src/stateful_transaction_validator.rs)

### Summary

`StatefulTransactionValidator::validate_resource_bounds` validates incoming transactions against `l2_gas_price` from the latest committed block header, but transactions will be executed in the **next** block at `next_l2_gas_price`. The block header stores both fields separately. Using the wrong field causes the gateway to admit transactions that will fail at execution (when the next price is higher) or reject transactions that would succeed (when the next price is lower).

### Finding Description

`BlockHeaderWithoutHash` contains two distinct L2 gas price fields:

- `l2_gas_price`: the price used for transactions **in that block**
- `next_l2_gas_price`: the EIP-1559-adjusted price for the **following block** [1](#0-0) 

`GatewayFixedBlockSyncStateClient::get_block_info_from_sync_client` populates `BlockInfo.gas_prices.strk_gas_prices.l2_gas_price` from `block_header.l2_gas_price.price_in_fri`, discarding `block_header.next_l2_gas_price` entirely: [2](#0-1) 

`validate_resource_bounds` then reads this field and names it `previous_block_l2_gas_price`, with a TODO comment acknowledging the bug: [3](#0-2) 

The threshold check compares the transaction's `max_price_per_unit` against a percentage of this wrong price: [4](#0-3) 

The blockifier's `check_fee_bounds`, which runs at actual execution time, compares `max_price_per_unit` against the block's real gas price (which is `next_l2_gas_price` from the previous block): [5](#0-4) 

### Impact Explanation

**Case 1 — next price higher than current (block over gas target):** A transaction with `max_price_per_unit` in the range `[threshold(l2_gas_price), next_l2_gas_price)` passes gateway admission but fails at blockifier execution with `InsufficientResourceBounds`. The gateway admits invalid transactions.

**Case 2 — next price lower than current (block under gas target):** A transaction with `max_price_per_unit` in the range `[threshold(next_l2_gas_price), threshold(l2_gas_price))` is rejected by the gateway even though it would succeed at execution. The gateway rejects valid transactions.

Both cases match the **High** impact: *Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.*

### Likelihood Explanation

The EIP-1559 mechanism adjusts `next_l2_gas_price` every block based on gas usage relative to `gas_target`. Any block that deviates from the target produces a `next_l2_gas_price ≠ l2_gas_price`. Under normal network load this occurs continuously. The discrepancy is bounded by `price / gas_price_max_change_denominator` per block but is always present when the block is not exactly at target. The `validate_resource_bounds` flag must be enabled (non-bootstrap phase), which is the production configuration.

### Recommendation

In `validate_resource_bounds`, read `next_l2_gas_price` from the block header rather than `l2_gas_price`. This requires either:

1. Extending `GatewayFixedBlockStateReader::get_block_info` to return `next_l2_gas_price` alongside the existing `BlockInfo`, or
2. Adding a dedicated `get_next_l2_gas_price` method to `GatewayFixedBlockStateReader`.

The TODO comment at line 229 already identifies this exact fix: `// TODO(Arni): getnext_l2_gas_price from the block header.` [6](#0-5) 

### Proof of Concept

1. Observe the latest committed block has `l2_gas_price = P` and `next_l2_gas_price = P'` where `P' > P` (block was over gas target).
2. Set `min_gas_price_percentage = 100` (full enforcement).
3. Submit an invoke transaction with `l2_gas.max_price_per_unit = P` (equals current block price, below next block price `P'`).
4. Gateway calls `validate_resource_bounds`: threshold = `100% * P = P`; transaction price `P >= P` → **admitted**.
5. Blockifier executes the transaction in the next block at price `P'`; `check_fee_bounds` finds `P < P'` → `InsufficientResourceBounds` → transaction reverts.
6. The gateway admitted a transaction that the blockifier rejects, confirming the invariant violation. [7](#0-6) [8](#0-7)
