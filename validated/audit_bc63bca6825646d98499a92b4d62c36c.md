### Title
Double-Counting of Notional Fees in `collectFees` Drains LP Principal When Both Spread and Notional Fees Are Active — (File: `metric-core/contracts/MetricOmmPool.sol`)

---

### Summary

`collectFees` uses `surplus0Scaled` / `surplus1Scaled` — the pool's total token surplus over LP bin balances, which already includes accumulated notional fees — as the full base for spread-fee distribution, and then distributes `notionalFeeToken0Scaled` / `notionalFeeToken1Scaled` a second time on top. The combined payout exceeds the actual surplus, and the shortfall is drawn from LP principal. The protocol's own test `test_collectProtocolFees_math_overallocates_whenSpreadAndNotionalBothActive` explicitly asserts this over-allocation.

---

### Finding Description

In `MetricOmmPool.sol`, the `collectFees` function computes the surplus as:

```
surplus0Scaled = balance0() * TOKEN0_SCALE_MULTIPLIER − totalScaledToken0InBins
```

This surplus contains **both** accumulated spread (LP) fees **and** accumulated notional fees (stored in `notionalFeeToken0Scaled`).

The function then distributes:

```solidity
// Spread leg — uses the FULL surplus as base
uint256 spreadFee0ToAdminScaled    = (surplus0Scaled * adminSpreadFeeE6_)    / spreadSumE6;
uint256 spreadFee0ToProtocolScaled = (surplus0Scaled * protocolSpreadFeeE6_) / spreadSumE6;
```

Because `adminSpreadFeeE6_ + protocolSpreadFeeE6_ == spreadSumE6`, these two lines together consume **100 % of `surplus0Scaled`**.

```solidity
// Notional leg — distributes notional fees AGAIN on top
uint256 notionalFee0ToAdminScaled    = (notionalFee0AmountScaled * adminNotionalFeeE8_) / notionalSumE8;
uint256 notionalFee0ToProtocolScaled = notionalFee0AmountScaled − notionalFee0ToAdminScaled;
```

Total payout = `surplus0Scaled + notionalFee0AmountScaled`.  
Available funds = `surplus0Scaled` (which already contains `notionalFee0AmountScaled`).  
**Over-allocation = `notionalFee0AmountScaled`** — drawn from LP bin balances. [1](#0-0) 

The correct base for the spread leg should be `surplus0Scaled − notionalFee0AmountScaled` (pure spread surplus only), so that the two legs are disjoint and their sum equals `surplus0Scaled`.

---

### Impact Explanation

Every call to `collectFees` when both `spreadFeeE6 > 0` and `notionalFeeE8 > 0` transfers `notionalFee0AmountScaled + notionalFee1AmountScaled` extra tokens out of the pool. These tokens come from LP bin balances (`totalScaledToken0InBins`), directly reducing the assets backing LP positions. Repeated fee collections progressively drain LP principal, making the pool insolvent: LP withdrawals will eventually revert or return less than owed.

---

### Likelihood Explanation

Any caller who can invoke `collectFees` (factory owner or anyone if the factory exposes a public path) triggers the bug whenever both fee types are non-zero. The default deployment sets both spread and notional fees, so the condition is met in normal operation. The protocol's own test suite confirms the over-allocation is reachable and measurable. [2](#0-1) 

---

### Recommendation

Subtract the notional fee accumulator from the surplus before computing the spread leg:

```solidity
uint256 spreadOnly0Scaled = surplus0Scaled > notionalFee0AmountScaled
    ? surplus0Scaled - notionalFee0AmountScaled : 0;
uint256 spreadOnly1Scaled = surplus1Scaled > notionalFee1AmountScaled
    ? surplus1Scaled - notionalFee1AmountScaled : 0;

uint256 spreadFee0ToAdminScaled    = spreadSumE6 == 0 ? 0 : (spreadOnly0Scaled * adminSpreadFeeE6_)    / spreadSumE6;
uint256 spreadFee0ToProtocolScaled = spreadSumE6 == 0 ? 0 : (spreadOnly0Scaled * protocolSpreadFeeE6_) / spreadSumE6;
// ... same for token1 and notional legs unchanged
```

This ensures `spreadLeg + notionalLeg == surplus`, preserving pool solvency.

---

### Proof of Concept

The protocol's own test already demonstrates the bug:

```solidity
// metric-core/test/MetricOmmPool.notionalFee.t.sol
function test_collectProtocolFees_math_overallocates_whenSpreadAndNotionalBothActive() public {
    ...
    uint256 total0Attempted = spread0ToAdmin + spread0ToProtocol + notional0ToAdmin + notional0ToProtocol;
    uint256 total1Attempted = spread1ToAdmin + spread1ToProtocol + notional1ToAdmin + notional1ToProtocol;

    assertGt(total0Attempted, surplus0Scaled, "token0 attempted payout exceeds computed surplus");
    assertGt(total1Attempted, surplus1Scaled, "token1 attempted payout exceeds computed surplus");
}
``` [3](#0-2) 

The excess `notionalFee0AmountScaled` is transferred from LP bin reserves, reducing the assets backing LP claims by exactly the notional fee amount on every `collectFees` call.
