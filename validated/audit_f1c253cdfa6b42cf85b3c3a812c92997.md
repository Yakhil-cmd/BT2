### Title
Arbitrary Address Injection via Custom Event Fields Pollutes Victim Account Transaction History — (`module/state_synchronization/indexer/extended/account_transactions.go`)

---

### Summary

`AccountTransactions.extractAddresses` blindly extracts every `cadence.Address`-typed field from any event emitted in a transaction, with no filtering by event type and no verification that the address corresponds to an account that actually authorized or was modified by the transaction. Any user who can deploy a Cadence contract and submit a transaction can inject an arbitrary victim address into the access node's account-transaction index, causing the victim's transaction history to include the attacker's transaction under the `interacted` role.

---

### Finding Description

`ProcessBlockData` iterates over all events for each transaction and calls `extractAddresses` on each one: [1](#0-0) 

`extractAddresses` decodes the event payload and returns every field whose runtime type is `cadence.Address` or `cadence.Optional[cadence.Address]`: [2](#0-1) 

The only guard applied to the returned addresses before indexing is a chain-validity check: [3](#0-2) 

There is no check on:
- the event type (user-defined vs. protocol-defined),
- whether the address field semantically represents an account that participated in the transaction,
- or whether the address authorized, paid for, or was modified by the transaction.

The `TransactionRoleInteracted` role is defined precisely for this purpose — "Account is referenced by an event in the transaction" — but the design assumes that only meaningful protocol events (FT transfers, NFT transfers, account creation, etc.) will contain address fields. That assumption is not enforced anywhere in the indexer. [4](#0-3) 

---

### Impact Explanation

An attacker can:
1. Deploy a Cadence contract that defines an event with an `Address` field (e.g., `pub event Poison(victim: Address)`).
2. Submit a transaction that emits `Poison(victim: <victim_address>)`.
3. After the block is indexed, query the victim's transaction history via the access API.
4. The attacker's transaction appears in the victim's history with role `interacted`.

The victim's account-transaction index entry is permanently mutated without the victim's knowledge or consent. This enables false attribution of malicious or embarrassing transactions to arbitrary accounts.

---

### Likelihood Explanation

- Requires only the ability to deploy a contract and submit a transaction — both are permissionless on Flow mainnet.
- No special keys, staked nodes, or operator access are needed.
- The exploit is deterministic and reproducible on any localnet or emulator running the extended indexer.
- The attacker can target any valid on-chain address.

---

### Recommendation

`extractAddresses` must not treat all `cadence.Address` fields in arbitrary user-defined events as meaningful account references. Mitigations include:

1. **Allowlist event types**: Only extract addresses from known protocol events (e.g., `flow.AccountCreated`, `flow.AccountUpdated`, FT/NFT standard events). Reject or ignore user-defined event types entirely for the purpose of address extraction.
2. **Restrict to authorizer/payer/proposer set**: Only index an address from an event if it is already present in the transaction's authorizer, payer, or proposer set — i.e., use events only to annotate roles for already-known participants, not to discover new ones.
3. **Validate event source**: Check that the event type is emitted by a known, trusted contract (e.g., a core contract deployed at a well-known address) before extracting addresses from it.

```go
func (a *AccountTransactions) extractAddresses(event flow.Event) ([]flow.Address, error) {
	cadenceEvent, err := events.DecodePayload(event)
	if err != nil {
		return nil, fmt.Errorf("failed to decode event payload: %w", err)
	}

	addresses := make([]flow.Address, 0)

	fields := cadence.FieldsMappedByName(cadenceEvent)
	for _, field := range fields {
		switch v := field.(type) {
		case cadence.Address:
			addresses = append(addresses, flow.Address(v))
		case cadence.Optional:
			if v.Value == nil {
				continue
			}

			addr, ok := v.Value.(cadence.Address)
			if !ok {
				continue
			}

			addresses = append(addresses, flow.Address(addr))
		}
	}
	return addresses, nil
```
