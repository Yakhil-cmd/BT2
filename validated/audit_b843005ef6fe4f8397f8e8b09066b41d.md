Audit Report

## Title
Per-Destination-Canister Round-Robin Enables Targeted Ingress Starvation at Near-Zero Sender Cost - (File: `rs/ingress_manager/src/ingress_selector.rs`)

## Summary
The ingress selector's `get_ingress_payload` groups pending messages by destination canister and iterates them in a randomized round-robin. Because induction fees are charged to the receiving canister and no per-sender-principal quota exists anywhere in the block-building path, an attacker can maintain messages to N distinct destination canisters in the pool at ~30 submissions/second, probabilistically filling every 1000-message block and starving a victim canister's messages until they expire after `MAX_INGRESS_TTL` (5 minutes). The attack is sustainable indefinitely at near-zero direct cost to the attacker.

## Finding Description

**Root cause 1 — Round-robin is keyed on destination canister, not sender.**

`get_ingress_payload` builds a `canister_queues` map keyed on `artifact.msg.signed_ingress.canister_id()` (the destination). [1](#0-0)  The outer loop iterates through all destination canisters in a randomized order, including at least one message per canister per pass. [2](#0-1)  With N attacker-controlled destination canisters and 1 victim canister, the victim's slot is filled only if it falls within the first 1000 of N+1 randomly ordered canisters, giving inclusion probability ≈ 1000/(N+1). At N=9000 this is ~11%, meaning ~89% of blocks exclude the victim.

**Root cause 2 — Induction cost is charged to the receiving canister; sender pays nothing.**

`ingress_induction_cost` explicitly sets `payer = ingress.canister_id()` for all non-subnet messages. [3](#0-2)  The only financial gate in `validate_ingress` checks whether the *receiving* canister can afford the fee. [4](#0-3)  The submitting principal is never charged, so the attacker's direct cost is zero.

**Root cause 3 — `exceeds_limit` is keyed on P2P `NodeId`, not user principal.**

The ingress pool throttle checks per-`NodeId` counters. [5](#0-4)  All user-submitted messages arrive via the local replica node and share the same `NodeId`, so the per-node limit is a global ceiling on all user traffic combined, not a per-principal cap. Additionally, `ArtifactPoolTomlConfig::new()` initialises `ingress_pool_max_count` to `usize::MAX`, meaning the pool has no effective count ceiling by default. [6](#0-5) 

**Attack flow:**

1. Attacker identifies N=9000 canister IDs on the target subnet with sufficient cycles (publicly discoverable via the IC dashboard/registry).
2. Attacker submits one ingress message to each of those N canisters. Each receiving canister is charged `ingress_message_reception_fee + ingress_byte_reception_fee`; the attacker pays nothing.
3. With `MAX_INGRESS_TTL = 5 minutes` and N=9000 messages needed in the pool, the attacker must submit only ~30 new messages/second to maintain the attack state — well below any realistic boundary-node rate limit.
4. `get_ingress_payload` fills the 1000-message cap with attacker messages ~89% of the time, excluding the victim canister.
5. After 5 minutes, victim messages expire and are purged. The victim receives no error response. [7](#0-6) [8](#0-7) 

## Impact Explanation

This is a **High** severity finding matching the allowed impact: *"Application/platform-level DoS, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."*

The attack is structural, not volumetric: it exploits the per-destination round-robin scheduling to achieve proportional block-space capture at ~30 req/s. A targeted victim canister's messages are excluded from ~89% of blocks and silently expire after 5 minutes, producing a permanent, renewable denial-of-service. Unlike a transaction revert, the victim receives no error, making the impact worse from a user perspective. The attack can be sustained indefinitely and can be directed at any canister or user on the subnet.

## Likelihood Explanation

An unprivileged principal with no special access can execute this attack. Prerequisites are: (a) knowledge of N canister IDs with sufficient cycles (publicly available), and (b) the ability to submit ~30 HTTP `/api/v2/canister/{id}/call` requests per second. Boundary-node IP-based rate limits are unlikely to trigger at 30 req/s and can be further bypassed by distributing across a small number of IPs. The attack is repeatable, requires no victim interaction, and has no on-chain cost to the attacker.

## Recommendation

1. **Introduce per-sender-principal limits in the ingress selector.** Track how many messages from each sender principal are included per block and cap any single principal's contribution to a fraction of `MAX_INGRESS_MESSAGES_PER_BLOCK`.
2. **Charge the sender a small submission fee.** Even a nominal fee at the HTTP handler level (e.g., a fraction of the induction cost) would make sustained flooding economically costly.
3. **Add per-principal rate limiting at the replica HTTP endpoint**, independent of boundary-node controls, so a single principal cannot maintain thousands of in-flight messages regardless of how many destination canisters are targeted.
4. **Consider keying `exceeds_limit` on the submitting user principal** rather than (or in addition to) the P2P `NodeId`, so that a single user cannot monopolise the ingress pool.

## Proof of Concept

```bash
# Maintain 9000 messages in the pool (~30 new submissions/second to replace expiring ones)
# All target canisters must be Running with sufficient cycles (publicly discoverable)

CANISTERS=$(ic-admin --nns-url https://ic0.app get-subnet-canisters <SUBNET_ID> | head -9000)

while true; do
  EXPIRED_SLOTS=$(( 9000 / 300 ))  # ~30 per second to replace TTL-expired messages
  COUNT=0
  for CANISTER_ID in $CANISTERS; do
    [ $COUNT -ge $EXPIRED_SLOTS ] && break
    curl -s -X POST "https://ic0.app/api/v2/canister/$CANISTER_ID/call" \
      -H "Content-Type: application/cbor" \
      --data-binary @minimal_signed_ingress.cbor &
    COUNT=$((COUNT + 1))
  done
  wait
  sleep 1
done
# Victim's messages to any canister on the subnet are excluded from ~89% of blocks
# and expire silently after MAX_INGRESS_TTL = 5 minutes.
```

A deterministic integration test can be written using `PocketIC` or the replica test harness: populate the validated ingress pool with 9000 messages to distinct destination canisters, then call `get_ingress_payload` and assert that a message to a 9001st (victim) canister is absent from the returned payload in the expected proportion of trials.
