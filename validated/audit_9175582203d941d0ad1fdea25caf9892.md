### Title
STRK Reward Token Not Protected in BTC Pool Reward Address Validation — (File: `src/pool/pool.cairo`)

---

### Summary

In BTC delegation pools, `enter_delegation_pool` and `change_reward_address` only guard against the pool's own staking token (`token_dispatcher.contract_address`, i.e. BTC) being used as a reward address. They do not guard against `STRK_TOKEN_ADDRESS`. Because pool rewards are **always disbursed in STRK** (hardcoded at line 365), a pool member who sets `reward_address = STRK_TOKEN_ADDRESS` will have all their STRK rewards transferred into the STRK token contract itself — permanently freezing them.

---

### Finding Description

`pool.cairo` contains two entry points that accept a `reward_address`:

**`enter_delegation_pool`** (line 194–195):
```cairo
let token_address = token_dispatcher.contract_address;
assert!(token_address != pool_member, "{}", Error::POOL_MEMBER_IS_TOKEN);
assert!(token_address != reward_address, "{}", GenericError::REWARD_ADDRESS_IS_TOKEN);
```

**`change_reward_address`** (line 506–510):
```cairo
assert!(
    self.token_dispatcher.contract_address.read() != reward_address,
    "{}",
    GenericError::REWARD_ADDRESS_IS_TOKEN,
);
```

Both checks compare only against `token_dispatcher.contract_address` — the pool's staking token. For a BTC pool, `token_dispatcher` holds the BTC token address. The check therefore only rejects `reward_address == BTC_TOKEN_ADDRESS`.

However, `claim_rewards` always pays out in STRK, hardcoded:

```cairo
// line 365–366
let reward_token = IERC20Dispatcher { contract_address: STRK_TOKEN_ADDRESS };
reward_token.checked_transfer(recipient: reward_address, amount: rewards.into());
```

So for a BTC pool, `reward_address = STRK_TOKEN_ADDRESS` passes the guard (`BTC != STRK`), is stored, and every subsequent `claim_rewards` call sends STRK to the STRK token contract — an address with no recovery mechanism.

By contrast, the staking contract's own `change_reward_address` correctly uses `does_token_exist`, which checks all registered tokens including STRK:

```cairo
// staking.cairo line 520–524
assert!(
    !self.does_token_exist(token_address: reward_address),
    "{}",
    GenericError::REWARD_ADDRESS_IS_TOKEN,
);
```

The pool contract applies only a single-token guard where a multi-token guard (covering at minimum both the staking token and `STRK_TOKEN_ADDRESS`) is required.

---

### Impact Explanation

When a BTC pool member sets `reward_address = STRK_TOKEN_ADDRESS` (either accidentally or via a misconfigured integration), every call to `claim_rewards` transfers their accrued STRK yield into the STRK token contract. The STRK token contract has no sweep or recovery function for tokens sent to it. The rewards are permanently frozen.

This matches the allowed impact: **Permanent freezing of unclaimed yield (High)**.

---

### Likelihood Explanation

The BTC multi-token staking feature is new (V3). Integrators writing scripts or UIs that automate pool entry may pass a token address as `reward_address` by mistake — particularly `STRK_TOKEN_ADDRESS`, since STRK is the reward currency and is the most prominent address in the system. The staking contract's own `change_reward_address` already defends against this exact mistake, confirming the protocol authors are aware of the risk; the pool contract simply has an incomplete guard. Likelihood is **Low-Medium**: it requires a user error or integration bug, but the error is natural given the dual-token context.

---

### Recommendation

In `pool.cairo`, replace the single-token check in both `enter_delegation_pool` and `change_reward_address` with a check that also rejects `STRK_TOKEN_ADDRESS`:

```cairo
// enter_delegation_pool
assert!(token_address != reward_address, "{}", GenericError::REWARD_ADDRESS_IS_TOKEN);
assert!(STRK_TOKEN_ADDRESS != reward_address, "{}", GenericError::REWARD_ADDRESS_IS_TOKEN);

// change_reward_address
assert!(
    self.token_dispatcher.contract_address.read() != reward_address,
    "{}",
    GenericError::REWARD_ADDRESS_IS_TOKEN,
);
assert!(STRK_TOKEN_ADDRESS != reward_address, "{}", GenericError::REWARD_ADDRESS_IS_TOKEN);
```

Alternatively, mirror the staking contract's pattern and call a helper that checks all protocol-registered token addresses.

---

### Proof of Concept

1. A BTC pool is deployed (staker calls `set_open_for_delegation(btc_token_address)`).
2. An unprivileged pool member calls `enter_delegation_pool(reward_address: STRK_TOKEN_ADDRESS, amount: X)`.
3. The guard at line 195 evaluates `btc_token_address != STRK_TOKEN_ADDRESS` → **passes**.
4. The member's `reward_address` is stored as `STRK_TOKEN_ADDRESS`.
5. Over several epochs the member accrues STRK rewards.
6. Anyone calls `claim_rewards(pool_member)`.
7. Line 365–366 executes: `STRK.transfer(recipient: STRK_TOKEN_ADDRESS, amount: rewards)`.
8. STRK rewards are sent to the STRK token contract and are permanently unrecoverable.
