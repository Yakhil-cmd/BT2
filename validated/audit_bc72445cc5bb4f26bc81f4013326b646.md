### Title
Blocklist Enforcement is Optional and Absent During Block Execution, Enabling Blocked-Address Bypass - (File: `app/block_address.go`, `app/proposal.go`)

### Summary
Cronos implements two separate blocklist enforcement mechanisms. The `BlockAddressesDecorator` ante handler only enforces the static (node-config) blacklist during `CheckTx` and is entirely skipped during `FinalizeBlock`. The `ProposalHandler` enforces the on-chain blocklist (`MsgStoreBlockList`) only during `PrepareProposal`/`ProcessProposal`, and only for validators that have a valid e2ee `age.Identity` key. Validators without this key silently skip all on-chain blocklist enforcement. The combination means a blocked address can have its transaction included in a block and executed with no blocklist check applied at execution time.

### Finding Description

**Layer 1 — `BlockAddressesDecorator` (static blacklist, CheckTx-only)**

`BlockAddressesDecorator.AnteHandle` wraps all its checks inside `if ctx.IsCheckTx()`:

```go
func (bad BlockAddressesDecorator) AnteHandle(ctx sdk.Context, tx sdk.Tx, simulate bool, next sdk.AnteHandler) (newCtx sdk.Context, err error) {
    if ctx.IsCheckTx() {
        // ... all blocklist checks ...
    }
    return next(ctx, tx, simulate)   // always passes during FinalizeBlock
}
```

During `FinalizeBlock`, `ctx.IsCheckTx()` is `false`, so the entire body is skipped and `next` is called unconditionally. [1](#0-0) 

**Layer 2 — `ProposalHandler` (on-chain blocklist, identity-gated)**

`SetBlockList` silently no-ops when `h.Identity == nil` (non-validator nodes) and returns an error when `h.Identity` is `noneIdentity{}` (validator nodes without a configured e2ee key), leaving `h.blocklist` permanently empty:

```go
func (h *ProposalHandler) SetBlockList(blob []byte) error {
    if h.Identity == nil {
        return nil   // silent no-op; blocklist never populated
    }
    ...
    reader, err := age.Decrypt(bytes.NewBuffer(blob), h.Identity)
    if err != nil {
        return err   // noneIdentity always fails here; blocklist stays empty
    }
``` [2](#0-1) 

`ProcessProposalHandler` fast-paths to `ACCEPT` when `h.blocklist` is empty, meaning validators without the identity key accept every block regardless of whether it contains transactions from blocked addresses:

```go

**Identity setup — validator nodes without e2ee key get `noneIdentity{}`**

In `app.go`, if a validator node cannot find or parse its e2ee key, it is assigned `noneIdentity{}` rather than a real identity. `noneIdentity.Unwrap` always returns `age.ErrIncorrectIdentity`, so `age.Decrypt` always fails for these nodes: [4](#0-3) 

The `noneIdentity` type is defined in `app/proposal.go`: [5](#0-4) 

### Impact Explanation

A blocked address (e.g., a sanctioned wallet added to the on-chain blocklist via `MsgStoreBlockList`) can bypass the blocklist entirely and execute transactions — including CRO/IBC-voucher/CRC20/CRC21 transfers, bridge interactions, and arbitrary EVM calls — if any validator without the e2ee identity key proposes a block containing the blocked address's transaction. Other validators without the identity key will accept the block in `ProcessProposal` (fast-path ACCEPT), and during `FinalizeBlock` no blocklist check is applied at all.

This is a **High** impact: bypass of the Cronos block-list authorization check, directly matching the allowed impact scope ("Bypass of Cronos admin, governance authority, permission, token-mapping, bridge, block-list, or module-account authorization checks").

### Likelihood Explanation

The e2ee identity key is an optional, per-validator configuration step. Any validator that has not run `e2ee keygen` and registered their key will operate with `noneIdentity{}`. In a network where even a minority of validators lack the key, a blocked address can route its transaction to one of those validators' mempools, have it included in a proposed block, and have that block accepted by the non-enforcing majority. The on-chain blocklist provides no protection at the execution layer regardless of validator configuration.

### Recommendation

1. **Enforce the on-chain blocklist during `FinalizeBlock`**: Remove the `if ctx.IsCheckTx()` guard in `BlockAddressesDecorator.AnteHandle`, or add a separate ante decorator that reads the on-chain blocklist from the KV store and rejects blocked-address transactions unconditionally during both `CheckTx` and `FinalizeBlock`.
2. **Do not silently no-op `SetBlockList`**: When `h.Identity == nil` or decryption fails, log a clear warning and consider whether the node should refuse to propose/accept blocks until the identity is configured, rather than silently accepting all transactions.
3. **Treat the blocklist as a consensus-enforced rule**, not a validator-side best-effort filter, given that it is stored on-chain via `MsgStoreBlockList` and is clearly intended as a security control.

### Proof of Concept

1. Admin calls `MsgStoreBlockList` with attacker address `A` encrypted to all validator keys. The blob is stored on-chain.
2. Validator `V` has no e2ee key configured; `identity = noneIdentity{}`. `SetBlockList` is called with the blob, `age.Decrypt` fails, `h.blocklist` remains empty.
3. Attacker submits a `MsgEthereumTx` (e.g., CRC20 transfer) from address `A` to validator `V`'s node. `BlockAddressesDecorator` checks `ctx.IsCheckTx() == true` but `A` is not in the **static** blacklist (only in the on-chain blocklist), so the tx is admitted to the mempool.
4. `V` calls `PrepareProposal`. `ExtTxSelector.validateTx` → `ProposalHandler.ValidateTransaction` → `h.blocklist` is empty → fast-path returns `nil` (no error) → tx is included in the proposed block.
5. Other validators without the identity key call `ProcessProposalHandler` → `h.blocklist` is empty → fast-path `ACCEPT`.
6. `FinalizeBlock` runs. `BlockAddressesDecorator.AnteHandle` is called with `ctx.IsCheckTx() == false` → the entire check body is skipped → `next` is called → the EVM transaction executes, transferring tokens from the blocked address.

func (bad BlockAddressesDecorator) AnteHandle(ctx sdk.Context, tx sdk.Tx, simulate bool, next sdk.AnteHandler) (newCtx sdk.Context, err error) {
	if ctx.IsCheckTx() {
		if sigTx, ok := tx.(signing.SigVerifiableTx); ok {
			signers, err := sigTx.GetSigners()
			if err != nil {
				return ctx, err
			}
			for _, signer := range signers {
				if _, ok := bad.blockedMap[sdk.AccAddress(signer).String()]; ok {
					return ctx, fmt.Errorf("signer is blocked: %s", sdk.AccAddress(signer).String())
				}
			}
		}

		for _, msg := range tx.GetMsgs() {
			msgEthTx, ok := msg.(*evmtypes.MsgEthereumTx)
			if ok {
				ethTx := msgEthTx.AsTransaction()
				// check the destination address
				if ethTx.To() != nil {
					if _, ok := bad.blockedMap[sdk.AccAddress(ethTx.To().Bytes()).String()]; ok {
						return ctx, fmt.Errorf("destination address is blocked: %s", sdk.AccAddress(ethTx.To().Bytes()).String())
					}
				}
				// check EIP-7702 authorisation list
				if ethTx.SetCodeAuthorizations() != nil {
					for _, auth := range ethTx.SetCodeAuthorizations() {
						addr, err := auth.Authority()
						if err == nil {
							if _, ok := bad.blockedMap[sdk.AccAddress(addr.Bytes()).String()]; ok {
								return ctx, fmt.Errorf("signer is blocked: %s", addr.String())
							}
						}
						// check the target address
						if _, ok := bad.blockedMap[sdk.AccAddress(auth.Address.Bytes()).String()]; ok {
							return ctx, fmt.Errorf("authorisation address is blocked: %s", sdk.AccAddress(auth.Address.Bytes()).String())
						}
					}
				}
			}
		}

		admin := bad.getParams(ctx).CronosAdmin
		for _, msg := range tx.GetMsgs() {
			if blocklistMsg, ok := msg.(*types.MsgStoreBlockList); ok {
				if admin != blocklistMsg.From {
					return ctx, errors.Wrap(sdkerrors.ErrUnauthorized, "msg sender is not authorized")
				}
			}
		}
	}
	return next(ctx, tx, simulate)
