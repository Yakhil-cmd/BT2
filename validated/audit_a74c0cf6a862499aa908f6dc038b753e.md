### Title
Malicious CKD Participant Injects Unverified Contributions to Corrupt Derived Key Output — (File: src/confidential_key_derivation/protocol.rs)

---

### Summary
The CKD coordinator aggregates `(big_y, big_c)` contributions from all participants with no proof of correctness. Any single malicious participant can send arbitrary group elements, causing the coordinator to output a corrupted `CKDOutput`. The application then unmasks a wrong derived key, silently accepting a cryptographically invalid result.

---

### Finding Description

In `do_ckd_coordinator`, the coordinator receives each participant's `CKDOutput` and unconditionally adds it to the running sum:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
``` [1](#0-0) 

The honest computation each participant is supposed to perform is:

```
big_y  = y_i · G
big_s  = x_i · H(pk ‖ app_id)
big_c  = big_s + y_i · app_pk
norm_big_y = λ_i · big_y
norm_big_c = λ_i · big_c
``` [2](#0-1) 

Nothing in the protocol enforces that the received `(big_y, big_c)` satisfies this relation. A malicious participant controls their own protocol execution and can call `chan.send_private` with any two `ElementG1` values instead of running `compute_signature_share`. [3](#0-2) 

The coordinator has no way to distinguish a correct contribution from a forged one because:
- There is no zero-knowledge proof binding `big_c` to the participant's actual signing share `x_i`.
- There is no consistency check between `big_y` and `big_c` (e.g., a DLEQ proof that `big_c - big_s = y_i · app_pk` for the same `y_i` used in `big_y = y_i · G`).
- The `CKDOutput` struct is a plain pair of group elements with no authentication. [4](#0-3) 

The final unmask operation performed by the application is:

```
derived_key = big_C − app_sk · big_Y
``` [5](#0-4) 

If participant `j` sends `(big_y'_j, big_c'_j)` instead of their correct values, the sum becomes:

```
big_Y = Σ_{i≠j} λ_i·y_i·G  +  λ_j·big_y'_j
big_C = Σ_{i≠j} λ_i·(x_i·H + y_i·app_pk)  +  λ_j·big_c'_j
```

Unmask:
```
big_C − app_sk·big_Y
  = msk·H(pk‖app_id)  +  λ_j·(big_c'_j − app_sk·big_y'_j)
```

The additive term `λ_j·(big_c'_j − app_sk·big_y'_j)` is attacker-controlled (the malicious participant freely chooses `big_c'_j` and `big_y'_j`). The application receives and accepts a derived key that is not `msk·H(pk‖app_id)`.

---

### Impact Explanation

**High — Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.**

The application silently receives a wrong derived key. Because the CKD protocol produces no output that the application can independently verify against the master public key, the corruption is undetectable at the protocol level. Any downstream use of the derived key (e.g., sealing secrets to a TEE, deriving encryption keys) will silently fail or produce incorrect results. The honest coordinator and all honest participants complete the protocol without error.

---

### Likelihood Explanation

**High.** The attack requires only one malicious participant among the `n` CKD participants. No cryptographic knowledge is needed — the attacker simply sends two arbitrary `G1` points. The attack is single-round, requires no interaction beyond the normal protocol message, and leaves no detectable trace in the protocol transcript. Any participant who is compromised or adversarial can execute this trivially.

---

### Recommendation

Each participant must accompany their `(norm_big_y, norm_big_c)` with a non-interactive zero-knowledge proof of correct formation. Concretely, a DLEQ proof is needed to demonstrate:

1. `big_y = y_i · G` and `big_c − big_s = y_i · app_pk` share the same discrete log `y_i` (proving the ElGamal ciphertext is well-formed).
2. `big_s = x_i · H(pk ‖ app_id)` where `x_i` is consistent with the participant's public verification share from the DKG output (proving the BLS share is correctly applied).

The coordinator must verify all proofs before aggregating. This is the standard approach used in threshold BLS and verifiable secret sharing aggregation protocols.

---

### Proof of Concept

A malicious participant `j` replaces `do_ckd_participant`'s honest send with:

```rust
// Instead of calling compute_signature_share, send (0, 0)
let waitpoint = chan.next_waitpoint();
let zero = ElementG1::identity();
chan.send_private(waitpoint, coordinator, &(zero, zero))?;
```

The coordinator sums all contributions. Participant `j`'s Lagrange-weighted share `λ_j · x_j · H(pk ‖ app_id)` is missing from `big_C`. The application unmasks:

```
big_C − app_sk · big_Y  =  msk·H(pk‖app_id) − λ_j·x_j·H(pk‖app_id)
                         =  (msk − λ_j·x_j)·H(pk‖app_id)
```

This is not the correct derived key. The application accepts it without error because `do_ckd_coordinator` returns `Ok(Some(ckd_output))` unconditionally after aggregation. [6](#0-5)

fn do_ckd_participant(
    mut chan: SharedChannel,
    participants: &ParticipantList,
    coordinator: Participant,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,

    let (norm_big_y, norm_big_c) =
        compute_signature_share(participants, me, key_pair, app_id, app_pk, rng)?;
    let waitpoint = chan.next_waitpoint();
    chan.send_private(waitpoint, coordinator, &(norm_big_y, norm_big_c))?;

    Ok(None)
}

#[derive(Debug, Clone, Deserialize, Serialize)]
pub struct CKDOutput {
    big_y: ElementG1,
    big_c: ElementG1,
}

impl CKDOutput {
    pub fn new(big_y: ElementG1, big_c: ElementG1) -> Self {
        Self { big_y, big_c }
    }

    /// Outputs `big_y`
    pub fn big_y(&self) -> ElementG1 {
        self.big_y
    }

    /// Outputs `big_c`
    pub fn big_c(&self) -> ElementG1 {
        self.big_c
    }

    /// Takes a secret scalar and returns
    /// s <- C − a ⋅ Y = msk ⋅ H ( `app_id` )
    pub fn unmask(&self, secret_scalar: Scalar) -> Signature {
        self.big_c - self.big_y * secret_scalar
    }
}
