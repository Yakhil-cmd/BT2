### Title
Missing `msg_hash == 0` Rejection in OT-Based ECDSA Signing Allows Message-Independent Signature Creation - (File: `src/ecdsa/ot_based_ecdsa/sign.rs`)

---

### Summary

The OT-based ECDSA `sign` function accepts a zero `msg_hash` scalar without validation and proceeds through the full threshold signing protocol, producing a valid ECDSA signature. The robust ECDSA variant explicitly rejects `msg_hash == 0` as a documented security requirement. When `msg_hash = 0`, the message-dependent term in each participant's signature share vanishes entirely (`s_i = 0 * k_i + r * sigma_i = r * sigma_i`), yielding a signature that does not commit to any message and directly exposes the product `k * x` (nonce times secret key) in the aggregated `s` value.

---

### Finding Description

**Root cause:** The `sign` function in `src/ecdsa/ot_based_ecdsa/sign.rs` performs several input validations (participant count, threshold, membership) but contains no check for `msg_hash.is_zero()`. [1](#0-0) 

The sibling robust ECDSA `sign` function in `src/ecdsa/robust_ecdsa/sign.rs` explicitly guards against this at lines 91–94: [2](#0-1) 

**Algebraic consequence:** Each participant's share is computed in `compute_signature_share` as:

```
s_i = msg_hash * k_i + r * sigma_i
``` [3](#0-2) 

When `msg_hash = 0`, this collapses to `s_i = r * sigma_i`. After aggregation by the coordinator, the final `s = r * sigma` where `sigma = k * x` (nonce times secret key). The signature `(R, s)` is a valid ECDSA signature for `h = 0` — the coordinator's final `sig.verify()` check passes — but it carries no message binding whatsoever.

**Attacker-controlled entry path:** The `msg_hash` parameter is caller-supplied. A malicious coordinator or application layer passes `Scalar::zero()` as `msg_hash` to all participants. Participants perform no independent validation of `msg_hash`; they simply compute and send their share. The protocol completes normally and the coordinator returns a valid `Signature`.

**The security documentation for the codebase itself identifies this as a prohibited input.** `docs/ecdsa/robust_ecdsa/signing.md` states:

> "Do not sign with h = 0 (the zero message hash). This input enables a related algebraic split-view attack in the modified scheme when N1 > 2t+1." [4](#0-3) 

The OT-based ECDSA scheme shares the same structural modification (signature-share linearization, outsourced message hash, coordinator aggregation) that makes `h = 0` dangerous, yet the guard is absent.

---

### Impact Explanation

**Impact: Critical — Unauthorized creation of a valid threshold signature for attacker-chosen inputs.**

With `msg_hash = 0`:

1. The aggregated `s = r * k * x`. Since `r` is the public x-coordinate of `R`, an observer who obtains `(R, s)` can compute `k * x = s / r`. If the same presignature is reused (or if a second signing session with the same presignature and a non-zero `h'` is obtained), the secret key `x` is directly recoverable: from `s' = k^{-1}(h' + r*x)` and `k*x = s/r`, one can solve for `x`.

2. Even without presignature reuse, the produced signature `(R, s)` is valid under the public key for `h = 0`, meaning the threshold signing ceremony has been completed for an attacker-chosen degenerate input that the protocol is explicitly documented to prohibit.

3. The OT-based ECDSA `sign` allows `participants.len() >= threshold` (not the strict `= 2t+1` of robust ECDSA), so the split-view attack surface is broader: a malicious coordinator can run parallel sessions with overlapping participant subsets, some with `h = 0` and some with a real `h`, enabling key extraction via the nonce-reuse relationship.

---

### Likelihood Explanation

The `msg_hash` parameter is a raw `Scalar` accepted directly from the caller with no domain enforcement. Any application that:
- Passes a pre-hashed value without verifying it is non-zero, or
- Is controlled by a malicious coordinator who deliberately supplies `Scalar::zero()`

will trigger this path. The OT-based ECDSA `sign` function is a public API entry point. The robust ECDSA variant already demonstrates that the library authors are aware of this risk and have mitigated it there; the omission in the OT-based path is an inconsistency that a malicious caller can exploit directly.

---

### Recommendation

Add the same zero-hash guard present in the robust ECDSA `sign` function to the OT-based ECDSA `sign` function, immediately after the participant-count checks:

```rust
if bool::from(msg_hash.is_zero()) {
    return Err(InitializationError::BadParameters(
        "msg_hash cannot be 0".to_string(),
    ));
}
``` [5](#0-4) 

This mirrors the existing guard in `src/ecdsa/robust_ecdsa/sign.rs` lines 91–94 and aligns with the security constraint documented in `docs/ecdsa/robust_ecdsa/signing.md`.

---

### Proof of Concept

**Setup:** Two participants, threshold = 2, a valid presignature `(R, k_i, sigma_i)` produced by the normal presign protocol.

**Attack:**
```rust
// Attacker (malicious coordinator or application) passes msg_hash = Scalar::zero()
let msg_hash = Scalar::ZERO;

let protocol = ot_based_ecdsa::sign::sign(
    &participants,
    coordinator,
    threshold,
    me,
    public_key,
    rerandomized_presignature,
    msg_hash,   // <-- zero, not rejected
).unwrap();    // succeeds: no zero-check exists
```

**Result:** The protocol runs to completion. Each participant computes `s_i = 0 * k_i + r * sigma_i = r * sigma_i` and sends it to the coordinator. The coordinator sums them to `s = r * sigma = r * k * x`. The coordinator's `sig.verify(&public_key, &msg_hash)` call passes because the ECDSA verification equation holds for `h = 0`:

```
s^{-1} * 0 * G + s^{-1} * r * X = (r/s) * X = (r / (r*k*x)) * x*G = k^{-1} * G = R  ✓
``` [6](#0-5) 

A valid `Signature { big_r: R, s: r*k*x }` is returned. From `s` and the public `r`, the attacker computes `k*x = s/r`. Combined with any second signing session using the same presignature (or via the split-view technique), the full secret key `x` is recoverable.

    participants: &[Participant],
    coordinator: Participant,
    threshold: impl Into<ReconstructionLowerBound>,
    me: Participant,
    public_key: AffinePoint,
    presignature: RerandomizedPresignOutput,
    msg_hash: Scalar,
) -> Result<impl Protocol<Output = SignatureOption>, InitializationError> {
    let threshold = usize::from(threshold.into());
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }

    let participants =
        ParticipantList::new(participants).ok_or(InitializationError::DuplicateParticipants)?;

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }

    // ensure number of participants during the signing phase is >= threshold
    if participants.len() < threshold {
        return Err(InitializationError::NotEnoughParticipantsForThreshold {
            threshold,
            participants: participants.len(),
        });
    }

    let ctx = Comms::new();
    let fut = fut_wrapper(
        ctx.shared_channel(),
        participants,
        coordinator,
        me,
        public_key,
        presignature,
        msg_hash,
    );
    Ok(make_protocol(ctx, fut))
}
