### Title
Missing Actual Token Balance Check in `RewardSupplier.claim_rewards` Causes Temporary Freezing of Staker Funds - (File: src/reward_supplier/reward_supplier.cairo)

### Summary
`RewardSupplier.claim_rewards` validates `amount <= unclaimed_rewards` (an accounting variable) but never checks the contract's actual STRK token balance before calling `checked_transfer`. Because `unclaimed_rewards` routinely exceeds the actual on-chain balance during the L1→L2 bridge delay window, the transfer reverts, causing `unstake_action` and `claim_rewards` in the staking contract to fail and temporarily freezing staker funds.

### Finding Description
The `RewardSupplier` contract tracks owed rewards in the storage variable `unclaimed_rewards` and requests real STRK tokens from L1 asynchronously through `request_funds`. The two quantities are intentionally decoupled: `request_funds` sends L1 mint messages but the tokens only arrive later via the StarkGate bridge.

`claim_rewards` (lines 205–220) performs only the accounting guard:

```cairo
let unclaimed_rewards = self.unclaimed_rewards.read();
assert!(amount <= unclaimed_rewards, "{}", GenericError::AMOUNT_TOO_HIGH);

self.unclaimed_rewards.write(unclaimed_rewards - amount);
let token_dispatcher = self.token_dispatcher.read();
token_dispatcher.checked_transfer(recipient: staking_contract, amount: amount.into());
``` [1](#0-0) 

There is no guard of the form `assert!(amount <= actual_balance)`. The `request_funds` function explicitly shows that `balance` can be zero while `l1_pending_requested_amount` covers the gap:

```cairo
let credit = balance + l1_pending_requested_amount;
let debit = unclaimed_rewards;
if credit < debit + threshold { ... }
``` [2](#0-1) 

When `l1_pending_requested_amount > 0` and the bridge has not yet delivered the tokens, `balance` can be zero (or less than `amount`) while `amount <= unclaimed_rewards` still passes. The subsequent `checked_transfer` then reverts from the ERC20 contract — the exact "throw from another contract" pattern described in the external report.

The staking contract's `send_rewards_to_staker` calls `claim_from_reward_supplier` unconditionally before transferring the principal back to the staker:

```cairo
claim_from_reward_supplier(:reward_supplier_dispatcher, :amount, :token_dispatcher);
token_dispatcher.checked_transfer(recipient: reward_address, amount: amount.into());
``` [3](#0-2) 

This is called inside `unstake_action` before the principal transfer:

```cairo
self.send_rewards_to_staker(:staker_address, ref :staker_info, :token_dispatcher);
// ...
token_dispatcher.checked_transfer(recipient: staker_address, amount: staker_amount.into());
``` [4](#0-3) 

If `send_rewards_to_staker` reverts, the entire `unstake_action` reverts, and the staker cannot retrieve their principal.

### Impact Explanation
Any staker who has accumulated `unclaimed_rewards_own > 0` and calls `unstake_action` (or `claim_rewards`) while the reward supplier's actual STRK balance is less than their owed amount will have the transaction revert. Because `unstake_action` bundles the reward claim and the principal return in a single atomic transaction, the staker is blocked from withdrawing their staked principal until L1 bridge funds arrive. This constitutes **temporary freezing of funds** (principal locked in the staking contract) and **temporary freezing of unclaimed yield**.

### Likelihood Explanation
The condition `actual_balance < unclaimed_rewards` is a normal, recurring operational state: every time `update_unclaimed_rewards_from_staking_contract` is called, `unclaimed_rewards` grows immediately while the corresponding STRK tokens only arrive after the L1→L2 bridge processes the mint request (typically hours). Any staker whose `unclaimed_rewards_own` exceeds the reward supplier's current balance — which can be zero — will hit this revert. No special privileges or attack setup are required; a staker simply calls `unstake_action` or `claim_rewards` during the bridge-delay window.

### Recommendation
Add an actual balance check inside `RewardSupplier.claim_rewards` before the transfer, mirroring the pattern already used in `request_funds`:

```cairo
fn claim_rewards(ref self: ContractState, amount: Amount) {
    let staking_contract = self.staking_contract.read();
    assert!(
        get_caller_address() == staking_contract,
        "{}",
        GenericError::CALLER_IS_NOT_STAKING_CONTRACT,
    );
    let unclaimed_rewards = self.unclaimed_rewards.read();
    assert!(amount <= unclaimed_rewards, "{}", GenericError::AMOUNT_TOO_HIGH);

    // --- ADD THIS ---
    let token_dispatcher = self.token_dispatcher.read();
    let balance: Amount = token_dispatcher
        .balance_of(account: get_contract_address())
        .try_into()
        .expect_with_err(InternalError::BALANCE_ISNT_AMOUNT_TYPE);
    assert!(amount <= balance, "{}", Error::INSUFFICIENT_BALANCE);
    // ----------------

    self.unclaimed_rewards.write(unclaimed_rewards - amount);
    token_dispatcher.checked_transfer(recipient: staking_contract, amount: amount.into());
}
```

This surfaces a clear, actionable error instead of an opaque ERC20 revert, and prevents `unstake_action` from being silently blocked. Longer-term, consider decoupling the reward claim from `unstake_action` so that a temporarily-unfunded reward supplier does not block principal withdrawal.

### Proof of Concept
1. Staker stakes and accumulates `unclaimed_rewards_own = R` over several epochs (via `_update_rewards` / `update_rewards_from_attestation_contract`).
2. Each reward update calls `update_unclaimed_rewards_from_staking_contract(rewards)`, increasing `unclaimed_rewards` in the reward supplier and sending L1 mint requests. The reward supplier's actual STRK balance remains 0 (bridge not yet delivered).
3. Staker calls `unstake_intent`, waits for the exit window, then calls `unstake_action`.
4. `unstake_action` → `send_rewards_to_staker` → `claim_from_reward_supplier(amount: R)` → `reward_supplier.claim_rewards(R)`.
5. Guard `R <= unclaimed_rewards` passes (accounting is correct).
6. `token_dispatcher.checked_transfer(staking_contract, R)` reverts — reward supplier balance is 0.
7. Entire `unstake_action` reverts. Staker's principal remains locked in the staking contract.
8. Staker must wait for L1 bridge delivery before retrying — potentially hours or longer.
