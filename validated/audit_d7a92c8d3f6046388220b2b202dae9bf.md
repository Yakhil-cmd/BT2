### Title
Unprivileged Caller Can Permanently Freeze Unclaimed Yield by Calling `update_rewards` with `disable_rewards = true` — (File: `src/staking/staking.cairo`)

---

### Summary

The `update_rewards` function in the Staking contract is callable by any unprivileged address and accepts a `disable_rewards` boolean parameter. When set to `true`, the function updates the global `last_reward_block` state variable and returns early — without distributing any rewards. Because the global guard `last_reward_block` is written **before** the `disable_rewards` early-return check, an attacker can front-run every legitimate reward-distribution call, permanently freezing unclaimed yield for all stakers.

---

### Finding Description

`update_rewards` is exposed as a fully public function under `StakingRewardsManagerImpl`. Its only gate is `general_prerequisites()`, which checks only that the contract is not paused and the caller is non-zero — no identity or role check is performed. [1](#0-0) 

The function accepts a caller-controlled `disable_rewards` flag: [2](#0-1) 

The critical ordering flaw: `last_reward_block` is written to storage **before** the `disable_rewards` branch is evaluated: [3](#0-2) 

Once `last_reward_block` is set to the current block number, the guard at the top of the function: [4](#0-3) 

…will cause every subsequent call in the same block to revert with `REWARDS_ALREADY_UPDATED`. The rewards for that block are permanently lost — they are never credited to `unclaimed_rewards_own` or forwarded to delegation pools.

The `general_prerequisites` function confirms there is no caller authorization: [5](#0-4) 

---

### Impact Explanation

**High — Permanent freezing of unclaimed yield.**

Each block, exactly one call to `update_rewards` can succeed (enforced by `last_reward_block`). If an attacker occupies that slot with `disable_rewards = true`, the block's rewards are never computed or distributed. Stakers cannot recover rewards for a missed block by calling in a later block — each block's reward window is independent. A persistent attacker running a bot that front-runs every block can permanently deny all stakers and delegators their yield, causing protocol-wide insolvency of the reward stream.

---

### Likelihood Explanation

**High.** The attack requires only:
1. A valid, active staker address (trivially obtained from on-chain events or `get_stakers`).
2. One transaction per block with `disable_rewards = true`.

Gas costs on Starknet are low. The attack is fully automatable and requires no capital, no privileged access, and no external dependencies.

---

### Recommendation

1. **Restrict `disable_rewards = true` to authorized callers** (e.g., the staker's own address or their registered operational address). Add an assertion before the `last_reward_block` write:
   ```cairo
   if disable_rewards {
       let caller = get_caller_address();
       assert!(
           caller == staker_address || caller == staker_info.operational_address,
           "UNAUTHORIZED_DISABLE_REWARDS"
       );
   }
   ```
2. **Alternatively**, move the `last_reward_block.write(...)` to **after** the `disable_rewards` guard so that a no-op call does not consume the block's reward slot.
3. Consider whether `disable_rewards` should be a public parameter at all, or whether it should be removed in favor of an internal-only mechanism.

---

### Proof of Concept

```
1. Attacker identifies any valid, active staker address `S` with non-zero STRK balance.
2. In every block B:
   a. Attacker calls: update_rewards(staker_address: S, disable_rewards: true)
   b. last_reward_block is written to B.
   c. Function returns early — no rewards distributed.
3. S's node (or anyone) calls: update_rewards(staker_address: S, disable_rewards: false)
   → Reverts: REWARDS_ALREADY_UPDATED
4. Block B's rewards are permanently lost for all stakers and delegators.
5. Repeated every block → all stakers permanently receive zero yield.
```

The root cause mirrors the external report exactly: a state-transition (`last_reward_block`) is committed without verifying that the caller is authorized to trigger the no-reward path, allowing an unprivileged actor to occupy the one-per-block slot and freeze yield for the entire protocol.
