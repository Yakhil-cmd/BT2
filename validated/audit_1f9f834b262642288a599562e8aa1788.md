### Title
Deprecated `.transfer()` / `.send()` ETH Transfer Causes DoS for Smart Contract Callers — (`contracts/src/WSEI.sol`, `contracts/src/MultiSender.sol`)

---

### Summary

Two production contracts in the Sei chain repository use the deprecated `.transfer()` and `.send()` primitives for ETH transfers. Post-Istanbul (EIP-1884), these methods forward only a 2300-gas stipend, which is insufficient for any recipient contract whose `receive()` or `fallback()` function performs even a single SLOAD (now 800 gas). This can permanently prevent smart-contract callers from withdrawing their WSEI, and allows a malicious recipient to grief batch ETH sends.

---

### Finding Description

**Location 1 — `contracts/src/WSEI.sol`, `withdraw()`, line 30:**

```solidity
function withdraw(uint wad) public {
    require(balanceOf[msg.sender] >= wad);
    balanceOf[msg.sender] -= wad;
    payable(msg.sender).transfer(wad);   // ← 2300-gas stipend
    emit Withdrawal(msg.sender, wad);
}
``` [1](#0-0) 

`payable(msg.sender).transfer(wad)` hard-caps the forwarded gas at 2300. Any smart-contract caller (multisig, DeFi vault, EIP-4337 account) whose `receive()` function reads a storage slot will exceed this stipend and cause the call to revert. Because the balance decrement precedes the transfer, the revert rolls back the decrement — funds are not permanently lost — but the contract can **never** successfully call `withdraw()`, effectively freezing its WSEI-backed ETH for as long as it holds the token.

**Location 2 — `contracts/src/MultiSender.sol`, `batchTransferEqualAmount()` / `batchTransfer()`, lines 19 and 41:**

```solidity
bool success = payable(recipients[i]).send(amount);
require(success, "Failed to send Ether");
``` [2](#0-1) [3](#0-2) 

`.send()` also forwards only 2300 gas. A malicious actor who is one of the recipients can deploy a contract whose `receive()` deliberately consumes > 2300 gas, causing the `require(success)` to revert the entire batch and denying ETH to all other recipients.

`EVMCompatibilityTester.sol` line 164 also uses `.transfer()`, but its `sendEther()` is owner-gated (`require(msg.sender == owner)`), so it is not an unprivileged attacker path and is excluded. [4](#0-3) 

---

### Impact Explanation

- **WSEI.sol**: Any smart-contract wallet or protocol that accumulates WSEI and calls `withdraw()` will have its ETH permanently unwithdrawable through that path. A workaround exists (transfer WSEI to an EOA first), but the contract itself cannot retrieve its ETH directly. This maps to **fund freeze** in the Sei bounty scope.
- **MultiSender.sol**: A single malicious recipient can grief an entire batch transfer, causing a DoS on the batch-send functionality. Severity is lower since no funds are lost.

Scope classification: **Low** (fund freeze with a viable workaround; DoS on a utility contract).

---

### Likelihood Explanation

Smart-contract callers of WSEI are common in DeFi (vaults, routers, multisigs). EIP-1884 has been live since the Istanbul hard fork (2019), so any contract with a non-trivial `receive()` is already affected. The trigger requires no special privileges — any unprivileged contract that holds WSEI and calls `withdraw()` hits this path.

---

### Recommendation

Replace `.transfer()` and `.send()` with `.call{value: ...}("")` and check the return value:

```solidity
// WSEI.sol withdraw()
(bool success, ) = payable(msg.sender).call{value: wad}("");
require(success, "ETH transfer failed");

// MultiSender.sol
(bool success, ) = payable(recipients[i]).call{value: amounts[i]}("");
require(success, "Failed to send Ether");
```

Alternatively, use OpenZeppelin's `Address.sendValue()` which wraps this pattern with a descriptive revert.

---

### Proof of Concept

1. Deploy a contract `Victim` with a `receive()` that reads one storage slot (costs 800+ gas post-Istanbul, total > 2300).
2. Have `Victim` call `WSEI.deposit{value: 1 ether}()` — succeeds.
3. Have `Victim` call `WSEI.withdraw(1 ether)` — the inner `payable(msg.sender).transfer(wad)` forwards 2300 gas; `Victim.receive()` exceeds the stipend; the call reverts.
4. `Victim`'s ETH is now unwithdrawable via `WSEI.withdraw()` directly, demonstrating the freeze.
