### Title
Delegator Cannot Specify Maximum Commission Bound When Entering or Switching Delegation Pool - (File: src/pool/pool.cairo)

### Summary
`enter_delegation_pool` and `switch_delegation_pool` accept no `max_commission` parameter, so a delegator cannot enforce a commission upper bound at entry time. A staker holding an active `CommissionCommitment` can front-run the delegator's transaction and raise the commission to `max_commission` before the delegator's funds are committed, causing the delegator to earn materially less yield than expected.

### Finding Description
`enter_delegation_pool` accepts only `reward_address` and `amount`:

```
fn enter_delegation_pool(
    ref self: ContractState, reward_address: ContractAddress, amount: Amount,
)
```

`switch_delegation_pool` accepts only `to_staker`, `to_pool`, and `amount`:

```
fn switch_delegation_pool(
    ref self: ContractState,
    to_staker: ContractAddress,
    to_pool: ContractAddress,
    amount: Amount,
) -> Amount
```

Neither function lets the caller specify a maximum acceptable commission. The commission that will govern the delegator's yield is read live from the staking contract at reward-distribution time via `split_rewards_with_commission`, using whatever value the staker has set.

The `set_commission_commitment` mechanism explicitly allows a staker to increase commission up to `max_commission` while a commitment is active:

```
fn update_commission(...) {
    if let Option::Some(commission_commitment) = ... {
        if self.is_commission_commitment_active(:commission_commitment) {
            assert!(
                commission <= commission_commitment.max_commission, ...
            );
            // commission can be HIGHER than old_commission
```

The code comment at the call site even acknowledges this: *"Current commission increase safeguards still allow for sudden commission changes."*

**Attack path:**
1. Staker sets commission to 2% (attractive).
2. Staker calls `set_commission_commitment(max_commission: 50%, expiration_epoch: current+10)` — publicly visible on-chain.
3. Delegator observes 2% commission and submits `enter_delegation_pool` (or `switch_delegation_pool`).
4. Staker front-runs the delegator's transaction with `set_commission(50%)`, which is valid under the active commitment.
5. Delegator's funds land in a pool charging 50% commission instead of 2%.

### Impact Explanation
The delegator's share of pool rewards is computed as:

```
pool_rewards = rewards_including_commission - commission_rewards
// where commission_rewards = rewards * commission / 10000
```

A commission jump from 2% to 50% reduces the delegator's yield by ~49 percentage points of the gross pool reward. This constitutes ongoing theft of unclaimed yield: every future reward distribution transfers to the staker commission that the delegator never consented to. The delegator's principal is not at risk, but their yield stream is permanently impaired for as long as they remain in the pool.

**Impact: High — Theft of unclaimed yield.**

### Likelihood Explanation
- The `CommissionCommitment` mechanism is a first-class protocol feature, so any staker can set one.
- The commitment and its `max_commission` are publicly readable on-chain, so a delegator can observe the risk — but the protocol provides no in-transaction enforcement.
- On Starknet, sequencer-level transaction ordering is controlled by the sequencer; a staker who also controls or can influence transaction ordering (or simply submits with a higher fee) can reliably front-run delegator entries.
- The attack requires no privileged role beyond being a registered staker with an active commitment.

**Likelihood: Medium.**

### Recommendation
Add an optional `max_commission: Option<Commission>` parameter to both `enter_delegation_pool` and `switch_delegation_pool`. Before transferring funds, assert:

```
if let Option::Some(max_comm) = max_commission {
    let current_commission = self.get_commission_from_staking_contract();
    assert!(current_commission <= max_comm, Error::COMMISSION_TOO_HIGH);
}
```

This mirrors the slippage-protection pattern recommended in the external report: the user provides an upper bound on the key parameter that governs their economic outcome, and the transaction reverts atomically if the bound is violated.

### Proof of Concept
1. Deploy the system; staker stakes with 2% commission and opens a STRK pool.
2. Staker calls `set_commission_commitment(max_commission: 5000, expiration_epoch: current+5)`.
3. Delegator approves the pool contract and submits `enter_delegation_pool(reward_address, amount)`.
4. Before the delegator's tx is included, staker calls `set_commission(5000)` — succeeds because commitment is active (see `update_commission`, `src/staking/staking.cairo` lines 1580–1597).
5. Delegator's tx executes; `pool_member_info_v1` now shows `commission: 5000`.
6. After the next attestation/reward cycle, `split_rewards_with_commission` (src/staking/utils.cairo lines 68–76) distributes 50% of the delegator's gross pool reward to the staker as commission — 25× more than the 2% the delegator expected.             // Asserts.

            assert!(

                GenericError::MISSING_UNDELEGATE_INTENT,
            );
            assert!(amount <= pool_member_info.unpool_amount, "{}", GenericError::AMOUNT_TOO_HIGH);
            let reward_address = pool_member_info.reward_address;

            // Update pool_member_info and write to storage.
            pool_member_info.unpool_amount -= amount;
            if pool_member_info.unpool_amount.is_zero() {
                // unpool_amount is zero, clear unpool_time.
                pool_member_info.unpool_time = Option::None;
            }
            self.write_pool_member_info(:pool_member, :pool_member_info);

            // Serialize the switch pool data and invoke the staking contract to switch pool.
            let switch_pool_data = SwitchPoolData { pool_member, reward_address };
            let mut serialized_data = array![];
            switch_pool_data.serialize(ref output: serialized_data);
            self
                .staking_pool_dispatcher
                .read()
                .switch_staking_delegation_pool(
                    :to_staker,
                    :to_pool,
                    switched_amount: amount,
                    data: serialized_data.span(),
                    identifier: pool_member.into(),
                );

            // Emit event.
            self
                .emit(
                    Events::SwitchDelegationPool {
                        pool_member, new_delegation_pool: to_pool, amount,
                    },
                );

        }

pub(crate) fn split_rewards_with_commission(
    rewards_including_commission: Amount, commission: Commission,
) -> (Amount, Amount) {
    let commission_rewards = compute_commission_amount_rounded_down(
        :rewards_including_commission, :commission,
    );
    let pool_rewards = rewards_including_commission - commission_rewards;
    (commission_rewards, pool_rewards)
}
