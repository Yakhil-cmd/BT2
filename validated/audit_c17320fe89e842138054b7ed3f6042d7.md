### Title
Malicious Coordinator Can Substitute `app_pk` to Extract Confidential Derived Key — (`File: src/confidential_key_derivation/protocol.rs`)

---

### Summary

The CKD protocol accepts `app_pk` (the application's ElGamal public key) as a caller-supplied parameter with no protocol-level validation or participant agreement. A malicious coordinator who controls the distribution of `app_pk` to all participants can substitute their own controlled key, then unmask the aggregated output to recover the confidential derived secret `msk · H(pk, app_id)`.

---

### Finding Description

In `src/confidential_key_derivation/protocol.rs`, the public entry point `ckd()` accepts `app_pk: PublicKey` as a raw caller-supplied argument: [1](#0-0) 

The function validates participant membership and coordinator membership, but performs **no validation on `app_pk`**. There is no check that `app_pk` is a non-identity element, no check that it corresponds to the `app_id`, and — critically — no protocol step where participants broadcast and reach consensus on `app_pk` before using it.

Each participant independently calls `ckd()` with whatever `app_pk` their caller supplies. Inside `compute_signature_share`, `app_pk` is used directly to mask the secret contribution: [2](#0-1) 

The masking term is `big_c = big_s + app_pk * y`, where `big_s = x_i · H(pk, app_id)`. The entire confidentiality guarantee of CKD rests on `app_pk` being a key whose discrete log is unknown to the coordinator. The library provides no mechanism to enforce this.

---

### Impact Explanation

If a malicious coordinator controls the channel through which `app_pk` is distributed to all participants (a common hub-and-spoke deployment pattern), they substitute `app_pk_evil = app_sk_evil · G`:

1. All participants compute shares using `app_pk_evil`.
2. Coordinator aggregates: `C = msk · H(pk, app_id) + Y · app_pk_evil`
3. Coordinator unmasks: `C − app_sk_evil · Y = msk · H(pk, app_id)`

The coordinator recovers the confidential derived key `msk · H(pk, app_id)` — the exact secret the CKD protocol is designed to keep hidden from MPC participants.

This matches the allowed Critical impact: **Extraction or disclosure of confidential derived secrets**.

---

### Likelihood Explanation

The attack is realistic in any deployment where the coordinator is also responsible for distributing protocol parameters (including `app_pk`) to other participants before the protocol run. This is a natural deployment topology for a coordinator-driven MPC service. The coordinator is already a trusted participant in the protocol, and the library provides no in-protocol mechanism to detect or prevent `app_pk` substitution.

---

### Recommendation

Include `app_pk` in the protocol transcript and add a broadcast/agreement round before shares are computed. Specifically:

1. Each participant should broadcast their received `app_pk` at the start of the protocol (analogous to how `sid_i` is broadcast in DKG).
2. All participants must verify that every other participant's `app_pk` matches their own before proceeding.
3. Validate that `app_pk` is not the identity element of G1 at the entry point of `ckd()`.

This mirrors the pattern already used in `do_keyshare` where session identifiers are reliably broadcast before any secret material is used. [3](#0-2) 

---

### Proof of Concept

```
Setup:
  - 3 participants: P1 (coordinator/malicious), P2, P3
  - Legitimate app_pk = app_sk * G  (application holds app_sk)
  - Malicious coordinator generates: app_pk_evil = app_sk_evil * G

Attack:
  - Coordinator distributes app_pk_evil to P2 and P3 instead of app_pk
  - P1 calls ckd(..., app_pk_evil, ...)
  - P2 calls ckd(..., app_pk_evil, ...)  ← fed by coordinator
  - P3 calls ckd(..., app_pk_evil, ...)  ← fed by coordinator

  Each participant i computes:
    norm_big_y_i = λ_i · y_i · G
    norm_big_c_i = λ_i · (x_i · H(pk, app_id) + y_i · app_pk_evil)

  Coordinator aggregates:
    Y = Σ norm_big_y_i
    C = msk · H(pk, app_id) + Y · app_pk_evil

  Coordinator unmasks:
    C − app_sk_evil · Y = msk · H(pk, app_id)   ← confidential derived key extracted
```

The root cause is the absence of any validation or in-protocol agreement on `app_pk` in `src/confidential_key_derivation/protocol.rs`. [4](#0-3)

            participants: participants.len(),
        });
    }

    // kick out duplicates

        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // ensure the coordinator is a participant

        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }

    let chan = comms.shared_channel();

        chan,
        coordinator,
        me,
        participants,
        key_pair,
        app_id.into(),
        app_pk,
        rng,
    );
    Ok(make_protocol(comms, fut))

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
