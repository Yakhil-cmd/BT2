### Title
COA Ownership Proof Signature Replay Across Multiple COAs Owned by the Same Cadence Account - (File: `fvm/evm/stdlib/contract.cdc`)

### Summary
`EVM.validateCOAOwnershipProof` verifies that supplied signatures are valid over `signedData` using the Cadence account's keys, then separately checks that the COA resource at the given `path` has an EVM address matching `evmAddress`. Because `signedData` is never required to encode `evmAddress`, a signature produced to prove ownership of one COA can be replayed verbatim to prove ownership of any other COA belonging to the same Cadence account.

### Finding Description
`validateCOAOwnershipProof` in `fvm/evm/stdlib/contract.cdc` performs two independent checks:

1. Cryptographic: `keyList.verify(signatureSet, signedData, "FLOW-V0.0-user")` — verifies the signatures over the caller-supplied `signedData` using the Cadence account's public keys.
2. Structural: borrows the `CadenceOwnedAccount` capability at `path` and compares its `.address().bytes` against the caller-supplied `evmAddress`. [1](#0-0) [2](#0-1) 

These two checks are entirely decoupled: the function never asserts that `signedData` encodes `evmAddress`. The code itself documents this gap: [3](#0-2) 

**Attack scenario (Cadence account with two COAs):**

1. Victim's Cadence account owns `COA_A` (EVM address `0xEVM_A`) published at `/public/coaA`, and `COA_B` (EVM address `0xEVM_B`) published at `/public/coaB`.
2. Victim legitimately signs `signedData = H` to prove ownership of `COA_A` for some application. The signature does not encode `0xEVM_A`.
3. Attacker constructs a proof: `{address: victim_cadence_addr, path: /public/coaB, keyIndices: [...], signatures: [victim_sig_over_H]}`.
4. Attacker calls `COA_B.isValidSignature(H, attacker_proof)` via the ERC-1271 interface on the COA contract. [4](#0-3) 

5. `isValidSignature` calls the Cadence Arch precompile `verifyCOAOwnershipProof(address(this), H, attacker_proof)`, which invokes `validateCOAOwnershipProof`. [5](#0-4) 

6. Check 1 passes: victim's keys validly signed `H`. Check 2 passes: the COA at `/public/coaB` has EVM address `0xEVM_B` = `address(this)`. Result: `isValid = true`.
7. The attacker has forged a valid ERC-1271 ownership proof for `COA_B` without the victim ever authorizing it.

The Go-side validator simply forwards the proof to the Cadence function without adding any binding: [6](#0-5) 

### Impact Explanation
Any EVM-side protocol that uses ERC-1271 `isValidSignature` on a COA for authentication (e.g., DEX order signing, smart-contract wallet authorization, NFT marketplace approvals) can be bypassed. An attacker who observes a victim's valid COA ownership proof for `COA_A` can replay it to impersonate the victim as the owner of `COA_B`, gaining unauthorized access to any on-chain asset or permission gated by `COA_B`'s ERC-1271 identity. This is a cross-VM asset loss / unauthorized account mutation class impact.

### Likelihood Explanation
The precondition is that the victim Cadence account holds more than one COA. While the comment notes this is historically rare, the Flow EVM bridge and ecosystem actively encourage COA creation, and nothing in the protocol prevents multiple COAs per account. The attacker entry path — calling `isValidSignature` on any deployed COA contract — is fully permissionless and requires no special privileges. Any EVM EOA or contract can trigger it.

### Recommendation
Enforce that `signedData` commits to `evmAddress` inside `validateCOAOwnershipProof` itself, rather than relying on callers to do so. Concretely, replace the bare `signedData` verification with a check over `evmAddress || signedData` (or a structured hash thereof), so that a signature produced for one COA's EVM address cannot validate for a different COA's EVM address:

```cadence
// Inside validateCOAOwnershipProof, before keyList.verify:
let boundData = evmAddress.toBytes().concat(signedData)
let isValid = keyList.verify(
    signatureSet: signatureSet,
    signedData: boundData,
    domainSeparationTag: "FLOW-V0.0-user"
)
```

This mirrors the fix applied in the referenced Augur PR #5564, which bound the contract address into the signed hash to prevent cross-instance replay.

### Proof of Concept

```
1. Deploy two COAs for the same Cadence account:
   - COA_A at /public/coaA  → EVM address 0xAAAA...
   - COA_B at /public/coaB  → EVM address 0xBBBB...

2. Victim signs H = keccak256("some message") with their Cadence key
   using tag "FLOW-V0.0-user" to prove ownership of COA_A.

3. Attacker constructs proof_B:
   {
     address:        victim_cadence_address,
     path:           /public/coaB,       // ← changed to COA_B's path
     keyIndices:     [0],
     signatures:     [victim_sig_over_H] // ← same signature, no re-signing
   }

4. Attacker calls from any EVM EOA:
   COA_B.isValidSignature(H, encode(proof_B))
   → verifyCOAOwnershipProof(0xBBBB..., H, encode(proof_B))
   → validateCOAOwnershipProof(victim, /public/coaB, H, [0], [sig], 0xBBBB...)

5. Signature check: victim's key signed H → PASS
   Address check:  COA at /public/coaB has address 0xBBBB... → PASS
   Result: isValid = true  ← unauthorized proof for COA_B accepted
``` [7](#0-6) [1](#0-0) [8](#0-7)
