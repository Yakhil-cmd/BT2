### Title
Pool Admin Can Set `timelock = type(uint32).max` in `OracleValueStopLossExtension`, Causing `_afterTimelock` uint32 Truncation Overflow That Bypasses All Future LP-Protection Timelocks - (File: metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol)

---

### Summary

`proposeOracleStopLossTimelock` accepts any `uint32 newTimelock` with no upper-bound validation. When the pool admin sets `newTimelock = type(uint32).max`, the helper `_afterTimelock` computes `uint32(block.timestamp + type(uint32).max)`, which silently truncates to a timestamp already in the past. Every subsequent `_requireElapsed` check passes immediately, giving LPs zero reaction time before the admin disables or weakens the stop-loss.

---

### Finding Description

`_afterTimelock` is the single function that schedules all timelocked parameter changes (drawdown, decay, high watermarks, and the timelock itself):

```solidity
function _afterTimelock(address pool_) private view returns (uint32) {
    return uint32(block.timestamp + oracleStopLossConfig[pool_].timelock);
}
``` [1](#0-0) 

The addition is performed in `uint256` (no revert), then explicitly cast to `uint32`, which silently truncates. With `block.timestamp ≈ 1,753,000,000` (July 2026) and `timelock = type(uint32).max = 4,294,967,295`:

```
1,753,000,000 + 4,294,967,295 = 6,047,967,295
uint32(6,047,967,295) = 6,047,967,295 mod 2^32 = 1,752,999,999   ← past timestamp
```

`_requireElapsed` then compares `block.timestamp < executeAfter`:

```solidity
function _requireElapsed(uint32 executeAfter) private view {
    if (block.timestamp < executeAfter) revert OracleStopLossTimelockNotElapsed(...);
}
``` [2](#0-1) 

`1,753,000,000 < 1,752,999,999` is `false`, so the guard never reverts. Every pending change becomes immediately executable.

`proposeOracleStopLossTimelock` accepts the unbounded value without any cap check:

```solidity
function proposeOracleStopLossTimelock(address pool_, uint32 newTimelock) external onlyPoolAdmin(pool_) {
    PoolStopLossSchedule storage sched = _initializedSchedule(pool_);
    uint32 executeAfter = _afterTimelock(pool_);
    sched.pendingTimelock = newTimelock;
    sched.pendingTimelockExecuteAfter = executeAfter;
    ...
}
``` [3](#0-2) 

By contrast, `drawdownE6` and `decayPerSecondE8` are both validated at proposal time, but `newTimelock` is not: [4](#0-3) 

The same absence of validation exists in `initialize`, where the initial `timelock` is decoded and stored without any bound check: [5](#0-4) 

---

### Impact Explanation

The `OracleValueStopLossExtension` is the sole on-chain mechanism protecting LP principal from value drain. Its NatSpec explicitly states: *"Drawdown and decay changes are timelocked so LPs can react."* [6](#0-5) 

Once the pool admin installs `timelock = type(uint32).max`, they can in the same block:

1. Call `proposeOracleStopLossDrawdown(pool, 0)` — `executeAfter` wraps to a past value.
2. Call `executeOracleStopLossDrawdown(pool)` — passes immediately, sets `drawdownE6 = 0`.
3. With `drawdownE6 == 0`, `_afterSwapOracleStopLoss` returns early and never checks watermarks. [7](#0-6) 

LPs have no on-chain window to observe the pending change and exit before the stop-loss is silently disabled. Subsequent swaps can drain LP value without triggering the guard, causing direct loss of LP principal.

---

### Likelihood Explanation

The pool admin is semi-trusted and is explicitly constrained to operate *within caps and timelocks*. The two-step path (propose → wait current timelock → execute → overflow active) is reachable by any pool admin without any external dependency. No special token behavior, oracle manipulation, or privileged factory-owner action is required. The only prerequisite is that the pool uses `OracleValueStopLossExtension`, which is a first-party periphery extension.

---

### Recommendation

Add an explicit upper-bound cap on `newTimelock` in both `initialize` and `proposeOracleStopLossTimelock`. A reasonable maximum (e.g., 30 days = `2_592_000`) prevents both the overflow path and indefinitely long timelocks:

```solidity
uint32 private constant MAX_TIMELOCK = 30 days;

function _validateTimelock(uint32 timelock) private pure {
    if (timelock > MAX_TIMELOCK) revert OracleStopLossTimelockTooLarge(timelock);
}
```

Apply `_validateTimelock` in `initialize` alongside `_validateDrawdown`/`_validateDecay`, and at the top of `proposeOracleStopLossTimelock` before writing to storage.

---

### Proof of Concept

```solidity
// Assume pool is deployed with OracleValueStopLossExtension, timelock = 3 days, drawdownE6 = 50_000.

// Step 1: Admin proposes timelock = type(uint32).max (no revert — no validation).
vm.prank(admin);
extension.proposeOracleStopLossTimelock(pool, type(uint32).max);

// Step 2: Wait out the current 3-day timelock.
vm.warp(block.timestamp + 3 days);
vm.prank(admin);
extension.executeOracleStopLossTimelock(pool);
// oracleStopLossConfig[pool].timelock == type(uint32).max

// Step 3: In the same block, propose drawdown = 0.
vm.prank(admin);
extension.proposeOracleStopLossDrawdown(pool, 0);
// _afterTimelock returns uint32(block.timestamp + type(uint32).max) → past timestamp

// Step 4: Immediately execute — _requireElapsed passes because executeAfter < block.timestamp.
vm.prank(admin);
extension.executeOracleStopLossDrawdown(pool);
// drawdownE6 == 0 → stop-loss permanently disabled, zero LP reaction time.

// Step 5: Subsequent swaps skip all watermark checks; LP value can drain freely.
(uint32 dd,,,) = extension.oracleStopLossConfig(pool);
assertEq(dd, 0); // stop-loss silently disabled
```
