### Title
Negative `delta_gas` Silently Discarded in `compute_gas_refund` Causes Permanent User Fund Loss - (File: `basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`)

---

### Summary

In `compute_gas_refund`, the double-accounting reconciliation between EVM gas consumption and native resource consumption is one-sided: when native resource usage *exceeds* EVM gas usage (`delta_gas > 0`), extra gas is charged from the user. But when EVM gas usage *exceeds* native resource usage (`delta_gas < 0`), the negative delta is silently discarded. The user is overcharged — they paid for native resources they did not consume — and the excess fee is permanently lost to them. A developer `// TODO` comment at the exact line acknowledges this gap.

---

### Finding Description

`compute_gas_refund` in `basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs` implements the ZKsync OS double-accounting reconciliation described in `docs/double_resource_accounting.md`. After execution, it computes:

```
delta_gas = (native_used / native_per_gas) - gas_used
``` [1](#0-0) 

The code only handles the `delta_gas > 0` branch (native overrun → charge extra gas). The `delta_gas < 0` branch (native underrun → user overpaid for native resources) is explicitly left unhandled, with a developer `// TODO: return delta_gas to gas_used?` comment at line 80 acknowledging the gap. [2](#0-1) 

The resulting `gas_used` (inflated relative to actual native consumption) is then used to compute the user's refund and the operator's fee payment:

- For L2 ZK transactions: `token_to_refund = gas_price * (tx_gas_limit - gas_used)` and `token_to_pay_operator = gas_used * gas_price_for_operator` [3](#0-2) [4](#0-3) 

- For L1→L2 transactions: `pay_to_operator = gas_used * gas_price` [5](#0-4) 

Because `gas_used` is not reduced when `delta_gas < 0`, the user receives a smaller refund than they are owed. The difference `|delta_gas| * gas_price` is permanently transferred to the operator (or burned under `burn_base_fee`) and is irrecoverable by the user.

The documentation itself confirms the asymmetry: it only describes the `delta_gas > 0` case and says nothing about the negative case. [6](#0-5) 

---

### Impact Explanation

**Direct financial loss to users.** Any transaction where EVM gas consumption exceeds native resource consumption (i.e., compute-heavy but pubdata-light transactions) results in the user being overcharged. The overcharge amount is:

```
overcharge = |delta_gas| * gas_price
           = (gas_used - native_used / native_per_gas) * gas_price
```

This amount is permanently transferred to the operator as excess fee and cannot be recovered. The user's balance is reduced by more than the actual cost of the transaction. This is a resource accounting bug with a direct, irreversible funds-loss path for every affected transaction.

---

### Likelihood Explanation

**High.** The `delta_gas < 0` condition occurs whenever a transaction consumes more EVM gas than its proportional native resource cost. This is a routine scenario for:
- Transactions with significant EVM computation but minimal storage writes (low pubdata)
- Any transaction where the EVM gas price is high relative to the native price, making `native_per_gas` large and `native_used / native_per_gas` small

This is not an edge case — it is a normal operating condition for many transaction types. Every such transaction silently overcharges the user.

---

### Recommendation

In `compute_gas_refund`, handle the `delta_gas < 0` case symmetrically with the `delta_gas > 0` case. When `delta_gas < 0`, reduce `gas_used` by `|delta_gas|` (clamped to `minimal_gas_used` to avoid underflow below the minimum charge):

```rust
if delta_gas > 0 {
    gas_used += delta_gas as u64;
} else if delta_gas < 0 {
    // Native underrun: user overpaid for native resources, return the excess
    let reduction = (-delta_gas) as u64;
    gas_used = gas_used.saturating_sub(reduction).max(minimal_gas_used);
}
```

This ensures the user is refunded the full amount corresponding to unused native resources, consistent with the documented intent of the double-accounting model.

---

### Proof of Concept

**Setup:**
- `gas_limit = 100_000`
- `gas_price = 1_000` (wei)
- `native_price = 1`, `native_per_gas = 1_000` (i.e., `gas_price / native_price`)
- Transaction uses `gas_used_evm = 50_000` EVM gas (from ergs)
- Transaction uses `native_used = 10_000_000` native units (low pubdata, compute-light)

**Computation in `compute_gas_refund`:**
1. `gas_used = 50_000` (from ergs remaining)
2. `full_native_limit = 100_000 * 1_000 = 100_000_000`
3. `native_used = 100_000_000 - remaining_native` → suppose `native_used = 10_000_000`
4. `delta_gas = (10_000_000 / 1_000) - 50_000 = 10_000 - 50_000 = -40_000`
5. Since `delta_gas < 0`, **nothing happens** — `gas_used` stays at `50_000`

**Result:**
- User is refunded `(100_000 - 50_000) * 1_000 = 50_000_000` wei
- **Correct refund** should be `(100_000 - 10_000) * 1_000 = 90_000_000` wei (since native only consumed 10,000 gas-equivalents)
- **User permanently loses** `40_000 * 1_000 = 40_000_000` wei to the operator

The `// TODO: return delta_gas to gas_used?` comment at line 80 of `refund_calculation.rs` is a direct developer acknowledgment that this case is unhandled. [7](#0-6)

pub(crate) fn compute_gas_refund<S: EthereumLikeTypes>(
    system: &mut System<S>,
    to_charge_for_pubdata: S::Resources,
    gas_limit: u64,
    minimal_gas_used: u64,
    native_per_gas: u64,
    resources: &mut S::Resources,
) -> Result<RefundInfo, InternalError> {
    // Already checked
    resources.charge_unchecked(&to_charge_for_pubdata);

    let mut gas_used = gas_limit
        .checked_sub(resources.ergs().0.div_floor(ERGS_PER_GAS))
        .ok_or(internal_error!("gas remaining > gas limit"))?;
    resources.exhaust_ergs();

    system_log!(system, "Gas used before refund calculations: {gas_used}\n");

    // Following EIP-3529, refunds are capped to 1/5 of the gas used
    let evm_refund = {
        let full_refund_ergs = system.io.get_refund_counter().ergs();
        let full_refund_gas = full_refund_ergs.0.div_floor(ERGS_PER_GAS);
        let max_refund = gas_used / 5;
        core::cmp::min(full_refund_gas, max_refund)
    };

    system_log!(system, "Gas refund from refund counters = {evm_refund}\n");

    gas_used -= evm_refund;

    system_log!(
        system,
        "Minimal gas used from validation = {minimal_gas_used}\n"
    );

    #[allow(unused_mut)]
    let mut gas_used = core::cmp::max(gas_used, minimal_gas_used);

    // Note: for zero gas price, we use "unlimited native"
    let full_native_limit = if cfg!(feature = "unlimited_native") || native_per_gas == 0 {
        u64::MAX - 1
    } else {
        gas_limit.saturating_mul(native_per_gas)
    };
    let native_used = full_native_limit.saturating_sub(resources.native().remaining().as_u64());

    #[cfg(not(feature = "unlimited_native"))]
    {
        // Adjust gas_used with difference with used native
        let delta_gas = if native_per_gas == 0 {
            0
        } else {
            (native_used / native_per_gas) as i64 - (gas_used as i64)
        };

        if delta_gas > 0 {
            // In this case, the native resource consumption is more than the
            // gas consumption accounted for. Consume extra gas.
            gas_used += delta_gas as u64;
        }
        // TODO: return delta_gas to gas_used?
    }

    let total_gas_refund = gas_limit - gas_used;
    system_log!(system, "Refund after accounting for unused gas, refund counters and native cost: {total_gas_refund}\n");
    require_internal!(
        total_gas_refund <= gas_limit,
        "Gas refund greater than gas limit",
        system
    )?;
    let refund_info = RefundInfo {
        gas_used,
        evm_refund,
        native_used,
    };
    system_log!(system, "Final gas used: {gas_used}\n");
    Ok(refund_info)
}

Then we compute the difference between the implicit gas used derived from native resource consumption and the gas used by EEs from the ergs used. We call this value `deltaGas`.
  `deltaGas := (nativeUsed / nativePerGas) - gasUsed`

If `deltaGas > 0`, we add it to `gasUsed` and charge it from ergs. This ensures that gas estimation will include additional gas to cover for native resources using just base fee. We expect the base fee to be enough to cover most transactions without the need of additional gas.
