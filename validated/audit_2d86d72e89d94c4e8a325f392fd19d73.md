### Title
Unprivileged Caller Can Permanently Suppress All Consensus Block Rewards via `update_rewards(disable_rewards: true)` — (File: `src/staking/staking.cairo`)

---

### Summary

The `update_rewards` function in the Staking contract is publicly callable with no access control beyond a non-zero caller check. Any address can invoke it with `disable_rewards: true`, which advances the global `last_reward_block` sentinel without distributing any rewards. Because only one `update_rewards` call is permitted per block (enforced by that sentinel), an attacker can permanently suppress all consensus block rewards for every staker by calling this function once per block.

---

### Finding Description

`update_rewards` is exposed as part of `IStakingRewardsManager` with `#[abi(embed_v0)]`, making it a fully public entry point. [1](#0-0) 

The only gate is `general_prerequisites()`, which checks that the contract is not paused and the caller is non-zero — no role or identity check. [2](#0-1) 

Inside the function, after validating that the staker exists and is active, `last_reward_block` is unconditionally written to the current block number: [3](#0-2) 

When `disable_rewards` is `true`, the function returns immediately after that write — no rewards are computed or distributed. The guard at the top of the function ensures no second call can succeed in the same block: [4](#0-3) 

`last_reward_block` is a global singleton with no admin reset path. Rewards for a block that was "consumed" by a `disable_rewards: true` call are permanently unrecoverable. [5](#0-4) 

---

### Impact Explanation

**High — Permanent freezing of unclaimed yield.**

An attacker who calls `update_rewards(any_valid_staker, disable_rewards: true)` in every block prevents all consensus block rewards from ever being distributed to any staker or delegator. The yield is not deferred — it is destroyed. There is no mechanism to retroactively credit missed blocks.

---

### Likelihood Explanation

**High.** The attack requires:
- No stake, no tokens, no privileged role.
- Any non-zero address suffices.
- The only cost is one Starknet transaction per block.

A motivated competitor, a griefing actor, or an automated bot can sustain this indefinitely. The attack becomes active as soon as `consensus_rewards_first_epoch` is reached. [6](#0-5) 

---

### Recommendation

Restrict `update_rewards` to a trusted caller (e.g., the attestation contract, a designated operator role, or the staker's own operational address). Alternatively, remove the `disable_rewards` parameter entirely and derive the disable condition from on-chain protocol state rather than a caller-supplied flag.

---

### Proof of Concept

1. Wait until `consensus_rewards_first_epoch` is set and the current epoch reaches it (i.e., `is_pre_consensus()` returns `false`).
2. Identify any currently active staker address `S` with non-zero balance.
3. In every Starknet block, submit: `staking_contract.update_rewards(staker_address: S, disable_rewards: true)`.
4. Each call passes all validity checks, writes `last_reward_block = current_block`, and returns at line 1488 without entering `_update_rewards`.
5. Any legitimate call to `update_rewards` in the same block fails at the `last_reward_block` guard (line 1454–1458).
6. Block rewards for that block are permanently lost. Repeating every block permanently zeroes all staker and delegator yield. [7](#0-6)
