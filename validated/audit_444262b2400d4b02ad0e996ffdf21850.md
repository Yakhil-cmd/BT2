The proposed attack path is concrete and reachable. Let me trace it precisely.

**Step 1 — Can a `PolynomialCommitment` with an identity constant term be constructed and deserialized?**

`PolynomialCommitment::new()` strips only *trailing* identity coefficients from the high-degree end: [1](#0-0) 

`rposition` finds the last non-identity coefficient. For a vector `[identity, non_identity_point]`, it returns `Some(1)`, so `count = 2` and the polynomial is accepted with the identity as its constant term. The deserialization path calls `Self::new()` identically: [2](#0-1) 

So a `PolynomialCommitment` with `coefficients[0] == identity` and at least one non-identity higher-degree coefficient passes both construction and deserialization.

**Step 2 — Does the malicious polynomial pass the Round 3 checks?**

In `do_generation_many`, after receiving `PolynomialCommitmentsMessageMany`:

1. **Degree check** — only verifies `their_big_e.degree() == threshold - 1`, not the constant term: [3](#0-2) 

2. **Commitment check** — verifies the hash commitment opened in Round 1. The attacker committed to the malicious polynomial in Round 1, so this passes: [4](#0-3) 

**Step 3 — Where does the fatal error occur?**

After both checks pass, the verifier constructs a `dlog::Statement` with the identity as `public`: [5](#0-4) 

Inside `dlog::verify()`, `statement.encode()` is called first, which calls `C::Group::serialize(self.public)` on the identity point. On secp256k1 the point at infinity is not serializable, so this returns `Err`, mapped to `ProtocolError::PointSerialization`: [6](#0-5) 

This error propagates via `?` through `dlog::verify()` → the `parallel_to_multiplication_task` closure → `try_join` → `do_generation_many`, aborting the honest verifier's entire triple generation session.

**Note on the question's file attribution:** The crash occurs in `dlog.rs` (not `dlogeq.rs`). The `dlogeq::verify()` path at line 483 is never reached because `dlog::verify()` at line 368 fails first. The mechanism is identical (`element_into` vs `C::Group::serialize` on an identity point), but the file is `src/crypto/proofs/dlog.rs`.

**Conclusion:**

The attack is concrete and locally testable:
- Craft `big_e` = `PolynomialCommitment([identity, G])` (degree 1, identity constant term)
- Commit to it in Round 1 (binding commitment passes)
- Reveal it in Round 2 (degree check passes, commitment check passes)
- Honest verifier's `dlog::verify()` aborts with `ProtocolError::PointSerialization`

A persistent malicious participant can repeat this in every session, permanently denying triple generation for all honest parties.

### Title
Malicious participant can abort triple generation by sending a `PolynomialCommitment` with an identity constant term — (`src/ecdsa/ot_based_ecdsa/triples/generation.rs`)

### Summary
A malicious participant in OT-based ECDSA triple generation can craft a `PolynomialCommitment` whose constant term is the group identity, commit to it in Round 1, and reveal it in Round 2. All degree and hash-commitment checks pass, but the subsequent `dlog::verify()` call fails fatally when attempting to serialize the identity point, aborting the honest verifier's session.

### Finding Description
`PolynomialCommitment::new()` strips only trailing identity coefficients from the high-degree end. A polynomial `[identity, non_identity_point]` passes construction and deserialization. In `do_generation_many`, after receiving `PolynomialCommitmentsMessageMany`, the verifier checks degree and hash commitment but does not check whether `their_big_e.eval_at_zero().value()` is the identity. It then passes this identity point directly as `dlog::Statement::public` to `dlog::verify()`. Inside `dlog::verify()`, `statement.encode()` calls `C::Group::serialize()` on the identity, which fails on secp256k1, returning `Err(ProtocolError::PointSerialization)`. This propagates via `?` and terminates the session.

### Impact Explanation
Every honest participant running `do_generation_many` (or `do_generation`) will have their triple generation session aborted. Since triples are required for OT-based ECDSA signing, a persistent attacker can deny all signing operations by aborting every triple generation session.

### Likelihood Explanation
Any participant in the triple generation protocol can execute this attack with no special privileges. The attacker only needs to craft a specific polynomial commitment and commit to it in Round 1. The attack is deterministic and repeatable.

### Recommendation
After the commitment check and before calling `dlog::verify()`, add an explicit check:
```rust
if their_big_e.eval_at_zero()?.value() == C::Group::identity() {
    return Err(ProtocolError::IdentityElement);
}
```
Apply the same check for `their_big_f.eval_at_zero()?.value()`. Alternatively, add a non-identity check to `PolynomialCommitment::eval_at_zero()` or enforce it at the `dlog::Statement` construction site.
