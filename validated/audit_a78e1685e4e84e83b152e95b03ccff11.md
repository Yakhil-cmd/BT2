Audit Report

## Title
Unbounded Prometheus Label Cardinality via Attacker-Controlled `network_identifier.network` — (`rs/rosetta-api/common/rosetta_core/src/metrics.rs`)

## Summary
`RosettaMetricsMiddleware` extracts `network_identifier.network` from every incoming POST request body and uses it verbatim as the `token_display_name` Prometheus label with no validation, allowlist, or length cap. Each unique attacker-supplied value permanently allocates new time-series in the global Prometheus registry and a new entry in the `CANISTER_DISPLAY_NAMES` HashMap, both of which are never evicted, enabling a single unprivileged HTTP client to exhaust Rosetta process heap memory and crash the service.

## Finding Description

**Root cause — no validation of the extracted label value.**

`extract_canister_id` reads the full request body with `to_bytes(body, usize::MAX)` and extracts `network_identifier.network` as a raw, unvalidated string: [1](#0-0) 

No format check, no allowlist, no length cap is applied to the returned string.

**Attacker string flows directly into Prometheus label.**

In `RosettaMetricsMiddleware::call`, when a `canister_id` is extracted, `get_display_name_from_canister_id` is called. For any string not already in `CANISTER_DISPLAY_NAMES`, it returns the raw input unchanged: [2](#0-1) 

That string is then passed to `RosettaMetrics::new`, which inserts it into the unbounded `CANISTER_DISPLAY_NAMES` HashMap: [3](#0-2) 

**New Prometheus time-series permanently allocated per unique value.**

`inc_api_status_count` and `observe_request_duration` both call `with_label_values` on the global `ENDPOINTS_METRICS` vectors using the attacker-controlled `token_display_name`: [4](#0-3) [5](#0-4) 

Both `rosetta_api_status_total` (IntCounterVec) and `rosetta_http_request_duration_seconds` (HistogramVec) are registered as `lazy_static` singletons in the global Prometheus registry: [6](#0-5) 

Each call to `with_label_values` with a new label combination permanently allocates a new time-series. For the HistogramVec, each unique `token_display_name` creates one counter per histogram bucket plus sum/count — all retained forever.

**No rate limiting or body-size guard exists.**

A grep across all Rosetta API Rust sources confirms zero occurrences of `rate_limit`, `RateLimit`, or any body-size cap. The `to_bytes(body, usize::MAX)` call imposes no upper bound on body size.

## Impact Explanation

This is a **High** severity finding matching: *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS"* and *"Significant Rosetta … infrastructure security impact with concrete user or protocol harm."*

The attack is not raw volumetric DDoS — it exploits a specific code flaw where each request with a unique label value permanently allocates heap memory regardless of request rate. With M unique values, memory grows as O(M × histogram_buckets × label_dimensions). At sufficient scale the Rosetta process exhausts heap, crashing ledger sync, balance queries, and transfer submissions for all users of the affected Rosetta instance.

## Likelihood Explanation

The attack requires only an HTTP client and no credentials. All Rosetta POST endpoints (e.g., `/network/list`, `/account/balance`) are publicly reachable. The exploit is deterministic, requires no timing, and is trivially scriptable. A single attacker can drive the process to OOM without any coordination.

## Recommendation

- Validate `network_identifier.network` against a static allowlist of known canister IDs at the middleware layer before using it as a label value; fall back to `default_metrics` for unknown values.
- Apply a hard cap on `CANISTER_DISPLAY_NAMES` map size (e.g., reject or evict entries beyond a fixed limit).
- Replace `usize::MAX` in `extract_canister_id` with a reasonable body-size cap (e.g., 64 KiB).
- Add per-IP or global request rate limiting at the HTTP server layer.

## Proof of Concept

```rust
// Sends 100_000 POST requests each with a unique UUID as network_identifier.network
// and asserts that Prometheus registry memory stays bounded.
use uuid::Uuid;

#[tokio::test]
async fn test_label_cardinality_explosion() {
    let client = reqwest::Client::new();
    for _ in 0..100_000 {
        let uid = Uuid::new_v4().to_string();
        let body = serde_json::json!({
            "network_identifier": { "blockchain": "Internet Computer", "network": uid }
        });
        let _ = client
            .post("http://localhost:8080/network/list")
            .json(&body)
            .send()
            .await;
    }
    let families = prometheus::default_registry().gather();
    let total_series: usize = families.iter()
        .map(|f| f.get_metric().len())
        .sum();
    // Will far exceed any reasonable bound; process will OOM before this assertion
    assert!(total_series < 10_000, "cardinality explosion: {} series", total_series);
}
```

async fn extract_canister_id(
    body: Body,
) -> Result<(Option<String>, Bytes), Box<dyn std::error::Error + Send + Sync>> {
    // Read body bytes
    let bytes = to_bytes(body, usize::MAX).await?;

    // Don't attempt to parse if empty
    if bytes.is_empty() {
        return Ok((None, bytes));
    }

    // Try to parse as JSON
    let canister_id = match serde_json::from_slice::<Value>(&bytes) {
        Ok(json) => json
            .get("network_identifier")
            .and_then(|ni| ni.get("network"))
            .and_then(|n| n.as_str())
            .map(|s| s.to_string()),
        Err(_) => None,
    };

    Ok((canister_id, bytes))
