I need to trace the full wNEAR unwrap promise chain and check whether a failure of the NEAR transfer in the callback has any recovery path.

### Title
Missing Callback for Failed NEAR Transfer After Successful `near_withdraw` in wNEAR Unwrap Path — (`engine-precompiles/src/native.rs`, `engine/src/contract_methods/connector.rs`)

---

### Summary

The wNEAR ERC-20 unwrap path (`withdrawToNear` with `:unwrap` suffix) constructs a two-step promise chain: `near_withdraw` → `exit_to_near_precompile_callback`. Inside the callback, if `near_withdraw` succeeded, a NEAR `Transfer` action is dispatched to `target_account_id` via `promise_return`. There is no third-level callback to handle the case where this NEAR transfer fails. If the transfer fails (e.g., `target_account_id` does not exist), the wNEAR ERC-20 tokens are already burned, the wNEAR NEP-141 has already been unwrapped, and the NEAR is permanently stranded in Aurora's NEAR account with no recovery path.

---

### Finding Description

**Step 1 — ERC-20 burn and precompile invocation.**
`withdrawToNear(bytes recipient, uint256 amount)` on the wNEAR ERC-20 contract burns the caller's tokens and calls the `ExitToNear` precompile with flag `0x01`, the amount, and a recipient of the form `"target.near:unwrap"`.

Inside `exit_erc20_token_to_near`, the guard `erc20_address == get_wnear_address(io)` passes and `Message::UnwrapWnear` is matched, producing:
- `method = "near_withdraw"`
- `transfer_near_args = Some(TransferNearArgs { target_account_id: "target.near", amount })` [1](#0-0) 

**Step 2 — Promise chain construction.**
Back in the precompile `run()`, `callback_args` is built with `refund: None` (the `error_refund` feature gate sets `refund` only for the `near_withdraw`-failure branch, not for the subsequent transfer failure) and `transfer_near: Some(...)`. Because `callback_args != default()`, a `PromiseWithCallbackArgs` is emitted:
- Base: `near_withdraw` on the wNEAR NEP-141 contract
- Callback: `exit_to_near_precompile_callback` on the Aurora contract [2](#0-1) 

**Step 3 — Callback execution.**
`exit_to_near_precompile_callback` checks `handler.promise_result(0)`. If `near_withdraw` succeeded, it creates a `PromiseBatchAction` with a single `Transfer` action to `args.target_account_id` and calls `handler.promise_return(promise_id)`. The function then returns `Ok(None)` — no further callback is registered on this transfer promise. [3](#0-2) 

**Step 4 — The gap.**
`promise_return` in NEAR means the result of the current receipt is the result of the returned promise. However, the state changes from receipts 1–3 (ERC-20 burn, `near_withdraw`, callback execution) are already committed and are not rolled back if receipt 4 (the NEAR transfer) fails. There is no fourth receipt / callback to re-wrap the NEAR or refund the ERC-20 tokens.

The `refund` branch in the callback only fires when `near_withdraw` itself fails: [4](#0-3) 

The `ExitToNearPrecompileCallbackArgs` struct has no field to carry a re-wrap address for the transfer-failure case: [5](#0-4) 

---

### Impact Explanation

When `near_withdraw` succeeds but the NEAR `Transfer` to `target_account_id` fails:

1. The wNEAR ERC-20 tokens are permanently burned (EVM state committed in receipt 1).
2. The wNEAR NEP-141 balance of Aurora's account is permanently reduced (receipt 2 committed).
3. The equivalent NEAR yocto amount is now in Aurora's NEAR account with no corresponding wNEAR ERC-20 supply and no mechanism to recover it.

The NEAR is permanently frozen in Aurora's account. The user loses their entire position. This satisfies **Critical — Permanent freezing of funds** and also constitutes **Critical — Insolvency** (the wNEAR ERC-20 supply is reduced without the NEAR being delivered, creating an unrecoverable accounting discrepancy in Aurora's NEAR balance).

---

### Likelihood Explanation

The trigger is straightforward and requires no privilege: any wNEAR ERC-20 holder can call `withdrawToNear` with a non-existent NEAR account as the target (e.g., a typo, a freshly-generated implicit account that has never been funded, or a deliberately crafted account name). The NEAR protocol will fail the `Transfer` action to a non-existent named account. No admin access, no key compromise, and no external oracle is required. The path is reachable on any Aurora deployment where `factory_set_wnear_address` has been called.

---

### Recommendation

Add a third-level callback on the NEAR transfer promise. This callback should detect a failed transfer and re-wrap the NEAR back into wNEAR ERC-20 for the original caller (analogous to how `refund_on_error` re-mints ERC-20 tokens when `ft_transfer` fails). Concretely:

1. Extend `ExitToNearPrecompileCallbackArgs` (or introduce a new args struct) to carry the original ERC-20 caller address and wNEAR ERC-20 contract address.
2. In `exit_to_near_precompile_callback`, instead of a bare `promise_return`, attach a second callback (`exit_to_near_transfer_callback` or similar) to the NEAR transfer promise.
3. In that second callback, if the transfer failed, call `near_deposit` on the wNEAR contract to re-wrap the NEAR, then mint the equivalent wNEAR ERC-20 back to the original caller.

---

### Proof of Concept

```
1. Deploy Aurora with wNEAR bridge configured (factory_set_wnear_address set).
2. Bridge N wNEAR into Aurora as wNEAR ERC-20 for account `attacker`.
3. From `attacker`, call:
     wNEAR_ERC20.withdrawToNear(
         abi.encode("nonexistent-account-xyz.near:unwrap"),
         N
     )
4. Observe receipt chain:
   - Receipt A: EVM executes, wNEAR ERC-20 burned, ExitToNear precompile emits
     PromiseWithCallbackArgs(near_withdraw, exit_to_near_precompile_callback).
   - Receipt B: near_withdraw on wNEAR contract succeeds; Aurora's wNEAR NEP-141
     balance decreases by N.
   - Receipt C: exit_to_near_precompile_callback fires (promise_result = Successful),
     dispatches Transfer(N yocto) to "nonexistent-account-xyz.near", returns promise.
   - Receipt D: Transfer fails (account does not exist). No further receipt scheduled.
5. Assert:
   - attacker's wNEAR ERC-20 balance == 0  (burned, not refunded)
   - "nonexistent-account-xyz.near" NEAR balance == 0  (transfer failed)
   - Aurora's wNEAR NEP-141 balance decreased by N  (near_withdraw committed)
   - Aurora's NEAR balance increased by N  (NEAR unwrapped, never forwarded)
   => N yocto NEAR permanently stranded in Aurora's account.
```
