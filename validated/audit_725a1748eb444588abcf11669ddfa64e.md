### Title
Signature Validation Bypass via `skip_stateful_validations` Admits Invoke Transactions with Invalid Signatures — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The gateway's UX shortcut for the `deploy_account + invoke` flow unconditionally skips the `__validate__` entry-point call for any invoke transaction whose nonce is exactly `1` and whose sender's on-chain nonce is `0`, provided the mempool reports any transaction from that address. An unprivileged attacker who observes a victim's `deploy_account` transaction in the mempool can inject an invoke transaction carrying an arbitrary, invalid signature that is admitted to the mempool without signature verification.

---

The three conditions that trigger the skip are:
1. Transaction type is `Invoke`.
2. Transaction nonce is exactly `Nonce(Felt::ONE)`.
3. On-chain account nonce is `Nonce(Felt::ZERO)` (account not yet deployed).
4. `account_tx_in_pool_or_recent_block(sender)` returns `true`.

Condition 4 is satisfied as soon as **any** transaction from that address is in the pool — including a `deploy_account` submitted by the victim. The check does not verify that the pooled transaction is specifically a `deploy_account`, nor does it verify that the caller of `skip_stateful_validations` is the same party who submitted the `deploy_account`.

The `validate_state_preconditions` call (nonce range, resource-bound threshold) and `validate_by_mempool` (duplicate hash, fee escalation) still run, but neither checks the cryptographic signature. The signature is only verified inside `__validate__`, which is the step that is skipped.

---

### Impact Explanation

An invoke transaction with a completely invalid (e.g., all-zero) signature is admitted to the mempool. This violates the invariant that every transaction in the mempool has passed account-level signature verification. Concretely:

- **Admission of invalid transactions**: The gateway accepts and forwards to the mempool an invoke transaction whose `__validate__` would fail, satisfying the High impact criterion "Mempool/gateway/RPC admission accepts invalid transactions … before sequencing."
- **Resource exhaustion without economic penalty**: The invalid transaction occupies a mempool slot. When the sequencer attempts to execute it, `__validate__` fails (account not yet deployed or signature invalid), the transaction reverts, and because the account has no deployed contract the fee transfer also fails — the attacker pays nothing. This is the direct Sequencer analog of the oracle DoS described in the external report.
- **Victim interference**: An attacker who observes a victim's `deploy_account` can pre-occupy nonce 1 with a garbage invoke, forcing the victim's legitimate nonce-1 invoke to either be rejected (duplicate nonce) or require a fee-escalation replacement.

---

### Likelihood Explanation

- **Unprivileged trigger**: The attacker needs only to observe a `deploy_account` transaction in the public mempool (or P2P gossip) and submit an invoke with nonce=1 from the same address. No private key knowledge is required.
- **Pre-funding is common**: Starknet accounts must be pre-funded before `deploy_account` is submitted, so `verify_can_pay_committed_bounds` passes for the attacker's transaction.
- **Stateless checks pass**: The stateless validator enforces non-zero resource bounds and valid calldata length, but does not check the signature at all — that is the sole responsibility of `__validate__`.
- **Window of opportunity**: The window is the time between the victim's `deploy_account` entering the mempool and being included in a block. This window is typically several seconds to minutes.

---

### Recommendation

1. **Restrict the skip to the submitter's own transaction pair.** The skip should only apply when the gateway can confirm that the invoke and the deploy_account were submitted together in the same request (e.g., by tracking a session token or requiring the invoke to be submitted atomically with the deploy_account).

2. **Alternatively, run `__validate__` even when skipping the nonce-strict check.** The original motivation is that the account contract does not exist yet, so `__validate__` would fail. A safer approach is to run `__validate__` against the *pending* state that includes the deploy_account's constructor output, or to defer the invoke until the deploy_account is executed.

3. **At minimum, verify that the pooled transaction for the address is specifically a `deploy_account`.** The current check `account_tx_in_pool_or_recent_block` returns `true` for any transaction type from that address.

### Proof of Concept

```
Precondition: Victim has pre-funded address A and submitted a valid deploy_account tx.
              The deploy_account tx is visible in the mempool.

Step 1 – Attacker constructs an invoke tx:
  sender_address = A
  nonce          = 1
  calldata       = [arbitrary]
  signature      = [0x0, 0x0]   // invalid / all-zero
  resource_bounds = { l2_gas: { max_amount: 1, max_price_per_unit: 1 } }  // non-zero to pass stateless check

Step 2 – Attacker submits the invoke tx to the gateway.

Step 3 – Gateway stateless validation passes (non-zero bounds, valid address, etc.).

Step 4 – Gateway stateful validation:
  account_nonce = get_nonce(A) = 0          // A not yet deployed
  validate_state_preconditions: nonce 1 in [0, 0+max_gap] ✓, resource bounds ✓
  validate_by_mempool: no duplicate, no escalation needed ✓
  skip_stateful_validations:
    tx.nonce() == 1 ✓
    account_nonce == 0 ✓
    account_tx_in_pool_or_recent_block(A) == true ✓  (victim's deploy_account is in pool)
    → returns true
  run_validate_entry_point(skip_validate=true):
    execution_flags.validate = false
    StatefulValidator::perform_validations → returns Ok() without calling __validate__

Step 5 – Attacker's invoke tx with invalid signature is forwarded to the mempool and admitted.

Step 6 – When the sequencer executes the block:
  deploy_account executes → A is deployed
  attacker's invoke executes → __validate__ called → signature check fails → tx reverts
  fee transfer fails (or succeeds but attacker controls A's balance) → no economic cost to attacker

Result: Invalid transaction admitted; victim's nonce-1 slot occupied; no fee paid by attacker.
```

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
