### Title
`StatelessTransactionValidator::validate_resource_bounds` rejects valid client-side proving transactions with zero prices - (File: `crates/apollo_gateway/src/stateless_transaction_validator.rs`)

### Summary

`StatelessTransactionValidator::validate_resource_bounds` unconditionally enforces that `max_possible_fee(Tip::ZERO) > 0` and that `l2_gas.max_price_per_unit >= min_gas_price` (default 8 GWei). Client-side proving invoke transactions are required by the prover to carry **zero** `max_price_per_unit` across all resource dimensions. Because `validate_resource_bounds` executes before `validate_client_side_proving_allowed`, every client-side proving transaction is rejected at the gateway stateless-validation stage even when `allow_client_side_proving = true`, which is the production default.

### Finding Description

The gateway's stateless validator runs checks in this order:

```rust
// crates/apollo_gateway/src/stateless_transaction_validator.rs  lines 33-53
pub fn validate(&self, tx: &RpcTransaction) -> StatelessTransactionValidatorResult<()> {
    Self::validate_contract_address(tx)?;
    Self::validate_empty_account_deployment_data(tx)?;
    Self::validate_empty_paymaster_data(tx)?;
    self.validate_resource_bounds(tx)?;          // ← runs first, unconditionally
    self.validate_tx_size(tx)?;
    ...
    if let RpcTransaction::Invoke(invoke_tx) = tx {
        self.validate_client_side_proving_allowed(invoke_tx)?;   // ← runs later
        self.validate_proof_facts_and_proof_consistency(invoke_tx)?;
    }
    ...
}
```

`validate_resource_bounds` contains two sub-checks that both fire on a zero-price transaction:

```rust
// lines 56-88
fn validate_resource_bounds(&self, tx: &RpcTransaction) -> ... {
    if !self.config.validate_resource_bounds { return Ok(()); }

    let resource_bounds = *tx.resource_bounds();
    // Check 1 – rejects when all prices are zero
    if ValidResourceBounds::AllResources(resource_bounds).max_possible_fee(Tip::ZERO) == Fee(0) {
        return Err(StatelessTransactionValidatorError::ZeroResourceBounds { resource_bounds });
    }
    // Check 2 – rejects when l2_gas price < 8 GWei (production default)
    if resource_bounds.l2_gas.max_price_per_unit.0 < self.config.min_gas_price {
        return Err(StatelessTransactionValidatorError::MaxGasPriceTooLow { ... });
    }
    ...
}
```

The prover (`virtual_snos_prover.rs`) explicitly mandates that client-side proving transactions carry zero prices:

```rust
// crates/starknet_transaction_prover/src/proving/virtual_snos_prover.rs  lines 392-445
/// Since proving is client-side, no fees are charged. All `max_price_per_unit` fields and `tip`
/// must be zero. The `max_amount` fields have different semantics:
/// - `l2_gas.max_amount`: determines the gas limit the OS enforces on the transaction. Must be non-zero.
fn validate_zero_fee_resource_bounds(tx: &RpcInvokeTransactionV3) -> Result<...> {
    if bounds.l1_gas.max_price_per_unit   != GasPrice(0) { violations.push(...); }
    if bounds.l2_gas.max_price_per_unit   != GasPrice(0) { violations.push(...); }
    if bounds.l1_data_gas.max_price_per_unit != GasPrice(0) { violations.push(...); }
    if tx.tip != Tip(0) { violations.push(...); }
    ...
    if bounds.l2_gas.max_amount == GasAmount(0) { return Err(...); }  // amount must be non-zero
}
```

With all prices = 0 and `Tip::ZERO`, `max_possible_fee(Tip::ZERO)` evaluates to:

```
l1_gas.amount × 0  +  l2_gas.amount × (0 + 0)  +  l1_data_gas.amount × 0  =  0
```

So Check 1 fires (`ZeroResourceBounds`). Even if Check 1 were removed, Check 2 fires because `0 < 8_000_000_000`.

Both `validate_resource_bounds = true` and `allow_client_side_proving = true` are the production defaults:

```rust
// crates/apollo_gateway_config/src/config.rs  lines 188-203
impl Default for StatelessTransactionValidatorConfig {
    fn default() -> Self {
        StatelessTransactionValidatorConfig {
            validate_resource_bounds: true,
            min_gas_price: 8_000_000_000,
            ...

            ...
        }
    }
}
```

There is no code path that skips `validate_resource_bounds` for client-side proving transactions. The `validate_client_side_proving_allowed` guard that would permit them is never reached.

### Impact Explanation

Every client-side proving invoke transaction submitted to the gateway is rejected with `ZeroResourceBounds` (or `MaxGasPriceTooLow`) before it can be admitted to the mempool or sequenced. The `allow_client_side_proving = true` configuration flag is rendered completely inoperative. This matches the **High** impact category: *Mempool/gateway/RPC admission rejects valid transactions before sequencing.*

### Likelihood Explanation

The bug is triggered by any user who follows the prover's documented API (zero prices, non-zero `l2_gas.max_amount`). Both conflicting flags are `true` in the production default config and in the deployed `gateway_config.json`. No special attacker capability is required; any ordinary user attempting to submit a client-side proving transaction will hit this rejection deterministically.

### Recommendation

In `StatelessTransactionValidator::validate_resource_bounds`, detect client-side proving invoke transactions (those with non-empty `proof_facts` or `proof`) and skip the fee-positivity and minimum-gas-price sub-checks for them:

```rust
fn validate_resource_bounds(&self, tx: &RpcTransaction) -> StatelessTransactionValidatorResult<()> {
    if !self.config.validate_resource_bounds { return Ok(()); }

    // Client-side proving transactions carry zero prices by design; fee checks do not apply.
    if let RpcTransaction::Invoke(RpcInvokeTransaction::V3(invoke)) = tx {
        if !invoke.proof_facts.is_empty() || !invoke.proof.is_empty() {
            return Ok(());
        }
    }

    let resource_bounds = *tx.resource_bounds();
    if ValidResourceBounds::AllResources(resource_bounds).max_possible_fee(Tip::ZERO) == Fee(0) {
        return Err(StatelessTransactionValidatorError::ZeroResourceBounds { resource_bounds });
    }
    if resource_bounds.l2_gas.max_price_per_unit.0 < self.config.min_gas_price {
        return Err(StatelessTransactionValidatorError::MaxGasPriceTooLow { ... });
    }
    ...
}
```

Alternatively, reorder the validation steps so that `validate_client_side_proving_allowed` and `validate_proof_facts_and_proof_consistency` run before `validate_resource_bounds`, and then branch on the result.

### Proof of Concept

1. Construct a valid client-side proving invoke V3 transaction following the prover's requirements:
   - `proof_facts`: non-empty
   - `proof`: non-empty
   - `resource_bounds.l1_gas.max_price_per_unit = 0`
   - `resource_bounds.l2_gas.max_price_per_unit = 0`
   - `resource_bounds.l1_data_gas.max_price_per_unit = 0`
   - `resource_bounds.l2_gas.max_amount = 100_000_000` (non-zero, as required)
   - `tip = 0`

2. Submit to the gateway (`POST /add_transaction`).

3. `StatelessTransactionValidator::validate` calls `validate_resource_bounds` first.

4. `max_possible_fee(Tip::ZERO)` = `0 × 0 + 100_000_000 × (0+0) + 0 × 0` = `0`.

5. The validator returns `Err(ZeroResourceBounds { ... })` — the transaction is rejected before `validate_client_side_proving_allowed` is ever reached, despite `allow_client_side_proving = true`. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

/// Validates resource bounds for proving, collecting all violations into a single error.
///
/// Since proving is client-side, no fees are charged. All `max_price_per_unit` fields and `tip`
/// must be zero. The `max_amount` fields have different semantics:
/// - `l2_gas.max_amount`: determines the gas limit the OS enforces on the transaction. Must be
///   non-zero. Set this to the value returned by `starknet_estimateFee`, or use a safe upper bound
///   like 100,000,000 (sufficient for ~1 million Cairo steps).
/// - `l1_gas.max_amount` and `l1_data_gas.max_amount`: do not affect OS execution and can be any
///   value.
fn validate_zero_fee_resource_bounds(
    tx: &RpcInvokeTransactionV3,
) -> Result<(), VirtualSnosProverError> {
    let bounds = &tx.resource_bounds;
    let mut violations = Vec::new();

    if bounds.l1_gas.max_price_per_unit != GasPrice(0) {
        violations
            .push(format!("l1_gas.max_price_per_unit = {}", bounds.l1_gas.max_price_per_unit.0));
    }
    if bounds.l2_gas.max_price_per_unit != GasPrice(0) {
        violations
            .push(format!("l2_gas.max_price_per_unit = {}", bounds.l2_gas.max_price_per_unit.0));
    }
    if bounds.l1_data_gas.max_price_per_unit != GasPrice(0) {
        violations.push(format!(
            "l1_data_gas.max_price_per_unit = {}",
            bounds.l1_data_gas.max_price_per_unit.0
        ));
    }
    if tx.tip != Tip(0) {
        violations.push(format!("tip = {}", tx.tip.0));
    }

    if !violations.is_empty() {
        return Err(VirtualSnosProverError::InvalidTransactionInput(format!(
            "Proving is client-side — no fees are charged. The following fields must be zero but \
             were not: [{}]. Set all max_price_per_unit fields and tip to 0x0. Note: max_amount \
             fields are fine to set — l2_gas.max_amount controls the gas limit enforced by the OS \
             (use the value from starknet_estimateFee, or 100000000 as a safe upper bound). \
             l1_gas.max_amount and l1_data_gas.max_amount do not affect OS execution.",
            violations.join(", ")
        )));
    }

    if bounds.l2_gas.max_amount == GasAmount(0) {
        return Err(VirtualSnosProverError::InvalidTransactionInput(
            "l2_gas.max_amount must be non-zero — it is the gas limit enforced by the OS on the \
             transaction. Set this to the value returned by starknet_estimateFee, or use \
             100000000 (0x5f5e100) as a safe upper bound (sufficient for ~1 million Cairo steps)."
                .to_string(),
        ));
    }

    Ok(())
