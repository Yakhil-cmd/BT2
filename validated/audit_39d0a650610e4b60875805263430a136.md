### Title
`setPoolAdminFeeDestination` Redirects All Accumulated Spread Fees to New Destination Without Prior Collection, Causing Loss to Previous Fee Destination — (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`setPoolAdminFeeDestination` updates `poolAdminFeeDestination[pool]` without first flushing accumulated spread-fee surplus to the old destination. Every other fee-parameter mutator (`setPoolAdminFees`, `setPoolProtocolFee`) calls `collectFees` with the old configuration before writing new values. The destination setter does not, so all spread surplus that accrued while the old destination was active is retroactively redirected to the new address on the next `collectPoolFees` call.

---

### Finding Description

The spread-fee surplus is not tracked per-period. It is computed lazily at collection time as:

```
surplus0Scaled = balance0() × TOKEN_0_SCALE_MULTIPLIER
                 − binTotals.scaledToken0
                 − notionalFeeToken0Scaled
``` [1](#0-0) 

This single number represents **all** spread fees earned since the last collection, regardless of when they were earned or what destination was active at the time. When `collectPoolFees` is called, the admin share is sent to `poolAdminFeeDestination[pool]` — the **current** destination: [2](#0-1) 

`setPoolAdminFees` and `setPoolProtocolFee` both guard against this by calling `collectFees` with the **old** config before writing new values: [3](#0-2) [4](#0-3) 

`setPoolAdminFeeDestination` performs no such flush: [5](#0-4) 

The result is structurally identical to the external bug: a parameter that should be evaluated at the time fees were **earned** (the destination in effect during accumulation) is instead evaluated at the time fees are **collected** (the current destination after the change).

---

### Impact Explanation

Any address set as `adminFeeDestination` at pool creation or via a prior `setPoolAdminFeeDestination` call has a legitimate claim to the spread surplus that accrued during its tenure. After the destination is changed without a prior flush, the next `collectPoolFees` call — which is **permissionless** and can be triggered by anyone — transfers that entire historical surplus to the new address. The previous destination receives nothing for the period it was active.

Concretely: if 500 token0 of spread surplus accumulated while Destination A was set, and the admin then changes to Destination B before collection, Destination A loses 500 token0 and Destination B receives them without having earned them. The loss is bounded only by how long fees went uncollected and the pool's trading volume.

---

### Likelihood Explanation

The pool admin is semi-trusted and can call `setPoolAdminFeeDestination` at any time with no timelock. The permissionless `collectPoolFees` means the admin does not even need to trigger collection themselves — a keeper or any third party can do so immediately after the destination change. The scenario is reachable in normal operations (e.g., admin rotates treasury address, admin role is transferred and new admin redirects fees, or admin separates the fee-recipient from the admin key).

---

### Recommendation

Add a `collectFees` call with the current configuration before updating the destination, mirroring the pattern in `setPoolAdminFees` and `setPoolProtocolFee`:

```solidity
function setPoolAdminFeeDestination(address pool, address newAdminFeeDestination)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (newAdminFeeDestination == address(0)) revert InvalidAdminFeeDestination();

    // Flush surplus earned under the old destination first
    PoolFeeConfig memory c = poolFeeConfig[pool];
    IMetricOmmPoolCollectFees(pool).collectFees(
        c.protocolSpreadFeeE6,
        c.adminSpreadFeeE6,
        c.protocolNotionalFeeE8,
        c.adminNotionalFeeE8,
        poolAdminFeeDestination[pool]   // old destination
    );

    poolAdminFeeDestination[pool] = newAdminFeeDestination;
    emit PoolAdminFeeDestinationUpdated(pool, newAdminFeeDestination);
}
```

---

### Proof of Concept

1. Pool is created; `adminFeeDestination = TreasuryA`. Protocol and admin spread fees are both non-zero.
2. The pool trades actively for 30 days. Spread surplus of 1 000 token0 accumulates in the pool (balance exceeds `binTotals.scaledToken0 + notionalFeeToken0Scaled`).
3. Pool admin calls `setPoolAdminFeeDestination(pool, TreasuryB)`. No collection occurs; `poolAdminFeeDestination[pool]` is overwritten.
4. Any address calls `collectPoolFees(pool)`. Inside `collectFees`, `surplus0Scaled` is computed from the current balance — it includes all 1 000 token0 earned while TreasuryA was active. The admin share (say 50 %) = 500 token0 is transferred to **TreasuryB**.
5. TreasuryA receives 0 token0 for the 30-day period it was the designated recipient. TreasuryB receives 500 token0 it did not earn.

The corrupted value is `adminFeeDestination_` passed to `transferToken0/1` at lines 417–420 of `MetricOmmPool.sol`: [6](#0-5) 

It reflects the **current** destination at collection time rather than the destination that was active when the surplus was generated, directly mirroring the external bug's use of current investor country instead of the country at issuance time.
