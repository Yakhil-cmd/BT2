### Title
Invoke Transaction with Invalid Signature Admitted to Mempool via `skip_stateful_validations` UX Bypass — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The gateway's `skip_stateful_validations` function unconditionally skips the blockifier `__validate__` entry-point call (i.e., account signature verification) for any Invoke transaction with `nonce == 1` sent from an address whose on-chain nonce is `0`, provided `account_tx_in_pool_or_recent_block` returns `true` for that address. Because `account_tx_in_pool_or_recent_block` checks only whether *any* transaction from that address is in the pool or recently committed — not whether the caller controls the address — an unprivileged attacker who observes a `DeployAccount` transaction for address A in the mempool can immediately submit a crafted Invoke from A with an arbitrary (invalid) signature and have it admitted to the mempool without any signature check.

`account_tx_in_pool_or_recent_block` is:

`contains_account` checks the `staged` and `committed` maps, and `tx_pool.contains_account` checks whether any transaction from that address is pooled — it does **not** verify that the pooled transaction is a `DeployAccount` or that the submitter controls the private key. [3](#0-2) 

When `skip_validate = true`, `run_validate_entry_point` sets `validate: !skip_validate = false`: [4](#0-3) 

Inside `StatefulValidator::perform_validations`, when `validate == false` the function returns `Ok(())` immediately after `perform_pre_validation_stage`, never calling `tx.validate_tx(…)` (the `__validate__` entry point): [5](#0-4) 

### Impact Explanation

An attacker who monitors the public mempool can:

1. Observe a `DeployAccount` for address A submitted by a legitimate user (A has on-chain nonce 0, pre-funded by the user).
2. Craft `Invoke(sender=A, nonce=1, calldata=arbitrary, signature=0x0, tip=high)`.
3. Submit to the gateway. The gateway reads on-chain nonce = 0, sees A in the pool via `account_tx_in_pool_or_recent_block`, sets `skip_validate = true`, and admits the transaction **without calling `__validate__`**.
4. The mempool accepts it (fee-escalation logic may evict the legitimate user's Invoke if the attacker's tip is higher).
5. The legitimate user's `DeployAccount` executes, but their Invoke is gone from the mempool; the attacker's Invoke reverts on execution (invalid signature), but the damage — displacement of the legitimate transaction — is done.

This satisfies: **High — Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.**

### Likelihood Explanation

- Requires only passive mempool observation (public) and a single crafted RPC call.
- No privileged access, no special contract, no dependency on external systems.
- The attack window is the time between the `DeployAccount` entering the mempool and being committed — typically one block, but sufficient for a mempool-watching bot.
- The attacker does not need to control address A's private key.

### Recommendation

Replace the bare `account_tx_in_pool_or_recent_block` check with a stricter guard that verifies a **`DeployAccount` transaction specifically** (not just any transaction) is present in the pool for address A. Alternatively, require that the Invoke transaction carry a valid signature even when the account is not yet deployed, deferring the UX convenience to a client-side retry mechanism rather than a gateway-level signature bypass.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
