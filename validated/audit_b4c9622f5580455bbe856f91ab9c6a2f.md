The race condition is real and fully traceable through the production code. Here is the analysis:

---

**The `logOut` function is an explicit no-op at the daemon level.** [1](#0-0) 

```typescript
const handleLogOut = useCallback(async () => {
  // do nothing until backend change API,
  // user is still logged in and syncing with previously selected fingerprint
}, []);
```

`LogoutButton.tsx` calls `await logOut()` then `navigate('/')` — the daemon fingerprint is never cleared. [2](#0-1) 

---

**The `requiresSync` path in `useWalletConnectCommand` re-checks the fingerprint post-sync by querying the daemon directly with `forceRefetch: true`.** [3](#0-2) 

Because `logOut` is a no-op, the daemon is still logged in with the original fingerprint when `getLoggedInFingerprint` is re-fetched. The check at lines 67–74 (`fingerprintAfterSync !== pair.fingerprint`) therefore **passes**, and `dispatchAsPair` executes the spend. [4](#0-3) 

---

**`waitForWalletSync` polls the daemon every second with no logout-abort signal.** [5](#0-4) 

There is no cancellation token, no logout flag, and no mechanism to interrupt the polling loop when the user navigates away.

---

### Title
Logout no-op allows in-flight WalletConnect `requiresSync` command to complete unauthorized XCH spend — (`packages/gui/src/components/app/LogoutButton.tsx`, `packages/core/src/components/Auth/AuthProvider.tsx`, `packages/gui/src/hooks/useWalletConnectCommand.tsx`)

### Summary
`handleLogOut` in `AuthProvider` is explicitly a no-op. A WalletConnect dApp that dispatches a `requiresSync` command (e.g. `chia_sendTransaction`) while the wallet is syncing can have that command execute successfully even after the user clicks logout, because the post-sync fingerprint re-check queries a daemon that is still logged in.

### Finding Description
The attack sequence:

1. Unprivileged WalletConnect dApp sends `chia_sendTransaction` (marked `requiresSync: true`).
2. `handleProcess` in `useWalletConnectCommand` passes the initial fingerprint guard (lines 38–46) because the wallet is currently logged in.
3. `await waitForWalletSync()` begins polling the daemon every 1 second (no abort signal).
4. User clicks the logout button. `handleLogout` calls `await logOut()` — which is a no-op — then `navigate('/')`.
5. The daemon remains logged in with the same fingerprint.
6. Sync completes; `waitForWalletSync()` resolves.
7. `getLoggedInFingerprint` is re-fetched with `forceRefetch: true`. The daemon returns the original fingerprint.
8. The post-sync guard at lines 67–74 passes (`fingerprintAfterSync === pair.fingerprint`).
9. `dispatchAsPair` executes the spend. XCH leaves the wallet.

### Impact Explanation
Critical. An unprivileged WalletConnect dApp can cause an XCH spend to complete after the user has clicked logout and navigated away from the wallet UI, violating the user's reasonable expectation that logout terminates all in-flight privileged operations.

### Likelihood Explanation
The window requires the wallet to be syncing at the moment the dApp sends the command, and the user to click logout during that window. This is a narrow but realistic race (sync can take minutes on first launch or after downtime). A malicious dApp can deliberately time its request to coincide with a known sync period.

### Recommendation
- Introduce a logout signal (e.g. an `AbortController` or a shared ref) that is set when the user initiates logout.
- `waitForWalletSync` should accept and respect an `AbortSignal`; on abort, reject the promise.
- `handleProcess` should check the abort signal before and after `waitForWalletSync` resolves, and throw before calling `dispatchAsPair`.
- Alternatively, `logOut` must actually call the daemon logout RPC so that the post-sync fingerprint re-check returns `null`, causing the existing guard to throw.

### Proof of Concept
Integration test plan:
1. Mock `waitForWalletSync` to delay resolution by 2 seconds.
2. Dispatch `chia_sendTransaction` via WalletConnect.
3. After 500 ms, simulate user clicking logout (call `handleLogout`).
4. Assert that `dispatchAsPair` is **never called** after logout.
5. With the current code, `dispatchAsPair` **is called** — confirming the vulnerability.

```typescript
export default async function waitForWalletSync(maxWait = HOUR): Promise<void> {
  const startTime = Date.now();
  await new Promise((resolve, reject) => {
    const interval = setInterval(async () => {
      const resultPromise = store.dispatch(api.endpoints.getSyncStatus.initiate());
      const { data } = await resultPromise;
      const walletState = getWalletSyncingStatus(data);

      resultPromise.unsubscribe();

      if (walletState === SyncingStatus.SYNCED) {
        clearInterval(interval);
        resolve();
      }

      if (Date.now() - startTime > maxWait) {
        clearInterval(interval);
        reject(new Error('Timed out waiting for wallet sync'));
      }
    }, 1000);
  });
```
