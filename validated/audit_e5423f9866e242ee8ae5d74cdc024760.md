### Title
Incorrect Royalty Amount Displayed in WalletConnect/Wallet Confirmation Dialog for Multi-NFT Offers — (`File: packages/gui/src/electron/commands/parseCommandDisplay.ts`)

### Summary

`formatAmountWithRoyalties` in `parseCommandDisplay.ts` divides the payment amount by the number of NFTs before computing each royalty, producing a displayed total that is `1/N` of the correct value (where N = number of NFTs with royalties). The confirmation dialog shown to the user before they approve a `take_offer` or `create_offer_for_ids` command therefore understates the true cost, causing the user to approve a transaction under a false understanding of the total amount they will spend.

### Finding Description

`formatAmountWithRoyalties` is called from `withRoyaltyTotals`, which is called from `walletDeltaToDisplay`, which is called from `parseCommandDisplay` for both `chia_wallet.take_offer` and `chia_wallet.create_offer_for_ids`. The result is placed in the `display` field of the confirmation dialog shown to the user before they approve the command. [1](#0-0) 

The erroneous line is:

```typescript
const splitAmount = amount / BigInt(royaltyPercentages.length);
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
```

`splitAmount` divides the full payment amount by the number of NFTs before multiplying by each royalty percentage. The correct formula is to multiply the **full** amount by each royalty percentage:

```typescript
// correct
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (amount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
```

The existing test at line 251–334 of `parseCommandDisplay.test.ts` encodes and asserts the **wrong** value (`0.00010255` XCH), confirming the bug is present and tested as-is. [2](#0-1) 

The `display` object produced by `parseCommandDisplay` is passed directly into the `Confirm` dialog at two call sites in `main.tsx`:

- Line 329: WalletConnect `DISPATCH_AS_PAIR` handler
- Line 825: direct wallet command confirmation handler [3](#0-2) [4](#0-3) 

### Impact Explanation

The user sees `amountWithRoyalties` in the confirmation dialog that is lower than the amount that will actually be deducted from their wallet. For N NFTs each with royalty percentage R and a payment of A mojos:

- **Displayed**: `A/N * R/10000 * N = A * R/10000` — coincidentally correct only when all royalties are equal; in the general case the per-NFT split distorts the sum.
- **Actual on-chain**: `A * R₁/10000 + A * R₂/10000 + … = A * ΣRᵢ/10000`

Concrete example (from the test): 2 NFTs, royalties 5% and 0.1%, payment 0.0001 XCH:
- Displayed total: **0.00010255 XCH**
- Correct total: **0.000105100 XCH** (~2.4% understatement)

With higher royalties or more NFTs the gap widens substantially. A user who would have rejected the offer at the correct total cost is induced to approve it by the understated display.

This matches the allowed High impact: *"Corruption, spoofing, or unsafe trust of … WalletConnect state that causes a user to approve … the wrong … amount."*

### Likelihood Explanation

Any dApp that has established a WalletConnect session can send `chia_wallet.take_offer` with a multi-NFT offer. No special privileges are required beyond a normal WalletConnect pairing. Multi-NFT offers with royalties are a supported and documented use case. The bug fires automatically whenever `royaltyPercentages.length > 1`.

### Recommendation

Remove the division by `royaltyPercentages.length`. Each NFT's royalty is owed on the **full** fungible payment amount, not a fractional share of it:

```typescript
function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],
): string | undefined {
  if (royaltyPercentages.length === 0 || line.kind === 'nft') {
    return undefined;
  }

  // Each royalty is calculated on the full amount, not a split share
  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (amount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;

  if (line.kind === 'xch') {
    return mojoToChiaLocaleString(totalAmount);
  }
  return mojoToCATLocaleString(totalAmount);
}
```

Update the corresponding test assertion from `'0.00010255'` to `'0.000105100'` (or the locale-formatted equivalent).

### Proof of Concept

1. Attacker dApp establishes a WalletConnect session with the victim.
2. Attacker constructs an offer containing 2 NFTs, each with a 5% royalty, requesting 1 XCH.
3. Attacker sends `chia_wallet.take_offer` via WalletConnect.
4. `parseCommandDisplay` calls `formatAmountWithRoyalties` with `amount = 1_000_000_000_000n` and `royaltyPercentages = [500, 500]`.
5. `splitAmount = 1_000_000_000_000n / 2n = 500_000_000_000n`
6. `royaltyAmount = (500_000_000_000n * 500n) / 10_000n + (500_000_000_000n * 500n) / 10_000n = 25_000_000_000n + 25_000_000_000n = 50_000_000_000n`
7. Displayed total: `1_050_000_000_000n` mojos = **1.05 XCH**
8. Correct total: `(1_000_000_000_000n * 500n / 10_000n) * 2 = 100_000_000_000n` royalty → **1.10 XCH**
9. The confirmation dialog shows **1.05 XCH**; the user approves; the wallet actually spends **1.10 XCH**.
