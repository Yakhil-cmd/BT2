Audit Report

## Title
Byzantine Peer Permanently Stalls State Sync via Persistent HTTP 429 Without Eviction — (`rs/p2p/state_sync_manager/src/ongoing.rs`)

## Summary
A Byzantine subnet peer that always responds with HTTP 429 is never removed from `active_downloads` because `DownloadChunkError::Overloaded` has no eviction path and no retry limit. If the Byzantine peer is the sole advertiser of the target state, the `OngoingStateSync` loop runs indefinitely, the victim replica never completes state sync, and its ability to rejoin consensus is permanently blocked for that sync attempt.

## Finding Description

**Root cause — asymmetric error handling in `handle_downloaded_chunk_result`.**

`parse_chunk_handler_response` maps HTTP 429 to `DownloadChunkError::Overloaded` ( [1](#0-0) ). The `Overloaded | Timeout | Cancelled` arm in `handle_downloaded_chunk_result` only re-queues the chunk via `download_failed`; it never calls `active_downloads.remove(&peer_id)` and never decrements `allowed_downloads` ( [2](#0-1) ). By contrast, `NoContent` and `RequestError` both remove the peer and decrement the budget ( [3](#0-2) ).

**Exploit flow.**

1. Byzantine peer is the sole (or first) advertiser of the target state; it is inserted into `active_downloads` with a budget of `PARALLEL_CHUNK_DOWNLOADS = 10`.
2. Every chunk download returns HTTP 429 → `Overloaded`. The peer's counter is decremented to 0 via `saturating_sub` but the entry is never removed ( [4](#0-3) ).
3. `spawn_chunk_downloads` selects the peer again (weight = `max - 0 + 1`), spawns new download tasks, and the cycle repeats.
4. The only self-termination path is `active_downloads.is_empty()` ( [5](#0-4) ), which is never satisfied while the Byzantine peer remains in the map.
5. `StateSyncManager::run` in `lib.rs` has no independent timeout for an ongoing sync; it only cleans up when `ongoing.shutdown.completed()` or `cancel_if_running` returns true ( [6](#0-5) ). Neither condition is triggered while the inner loop is spinning.

**Why existing guards are insufficient.**

The `CHUNK_DOWNLOAD_TIMEOUT` of 10 seconds applies only to transport-level timeouts; a peer that responds immediately with 429 bypasses it entirely. There is no per-peer consecutive-failure counter, no exponential backoff, and no circuit breaker anywhere in the codebase. The existing `test_cancel_if_running` test uses a mock that always returns `TOO_MANY_REQUESTS` and only asserts that external cancellation works — it does not assert peer eviction, confirming the design intent is "retry forever" ( [7](#0-6) ).

## Impact Explanation
A lagging replica performing state sync is permanently stalled for the duration of the sync attempt when the Byzantine peer is the sole advertiser. The replica cannot advance its certified state height, cannot participate in consensus, and reduces the effective fault tolerance of the subnet. This matches the allowed impact: **application/platform-level DoS and subnet availability impact not based on raw volumetric DDoS** (High, $2,000–$10,000). The constraint that the Byzantine peer must be the sole or dominant advertiser limits severity to the lower end of the High band.

## Likelihood Explanation
Any node operator running a malicious replica can implement this by returning HTTP 429 for all `/state-sync/chunk` requests — no threshold corruption, key compromise, or privileged access is required. The attacker only needs to be the first peer to advertise the target state to the victim replica, which is achievable by timing the advert broadcast. The attack is repeatable across every state sync attempt the victim makes. The primary constraint is that honest peers advertising the same state will eventually be added to `active_downloads` via `new_peers_rx`, at which point downloads from them can succeed; however, the `try_send` in `handle_advert` can silently drop honest-peer adverts under channel pressure, and the Byzantine peer can exploit the window before honest peers are added.

## Recommendation
In the `Overloaded` arm of `handle_downloaded_chunk_result`, maintain a per-peer consecutive-overload counter (e.g., in a `HashMap<NodeId, u32>` alongside `active_downloads`). After a configurable threshold of consecutive `Overloaded` responses (e.g., `3 * PARALLEL_CHUNK_DOWNLOADS`), treat the peer identically to `RequestError`: call `active_downloads.remove(&peer_id)` and decrement `allowed_downloads`. Alternatively, introduce an exponential backoff that delays re-queuing chunks for the offending peer, effectively starving it of download slots over time. Either approach preserves the transient-retry semantics for genuinely overloaded honest peers while bounding the damage from a Byzantine peer.

## Proof of Concept
Construct a state-machine unit test in `ongoing.rs`:
1. Create a `MockTransport` whose `rpc` always returns `StatusCode::TOO_MANY_REQUESTS`.
2. Create a `MockChunkable` whose `chunks_to_download` always yields `ChunkId::from(1)`.
3. Call `start_ongoing_state_sync`, send `NODE_1` as the sole peer.
4. After `N * PARALLEL_CHUNK_DOWNLOADS` download cycles (tracked via a shared atomic counter on the mock), assert either: (a) `NODE_1` is absent from `active_downloads`, or (b) the sync has terminated without external cancellation.
Under the current code, neither assertion holds — the loop runs indefinitely until `ongoing.shutdown.shutdown()` is called externally, exactly as `test_cancel_if_running` demonstrates.
