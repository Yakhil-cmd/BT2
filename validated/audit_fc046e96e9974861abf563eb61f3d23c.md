### Title
FROST `PresignOutput` Derives `Clone` With No Reuse Guard, Enabling Nonce Reuse and Private Key Share Extraction — (File: `src/frost/mod.rs`)

---

### Summary

The `PresignOutput<C>` struct in `src/frost/mod.rs` derives `Clone` and `PartialEq`, and the library provides no mechanism — no consumption-by-move, no used-flag, no stored state — to prevent a caller or malicious coordinator from supplying the same presign output (and therefore the same secret nonces) to multiple signing invocations. This is the direct analog of the external report's pattern: a caller-supplied argument that should be validated against previously stored state is accepted without any such check. In FROST, reusing the same `(d_i, e_i)` nonce pair across two signing sessions with different messages breaks the security proof and enables private key share extraction.

---

### Finding Description

`PresignOutput<C>` is defined as:

```rust
#[derive(Debug, Clone, Serialize, Deserialize, Eq, PartialEq)]
pub struct PresignOutput<C: Ciphersuite + Send + 'static> {
    pub nonces: SigningNonces<C>,          // secret nonce pair (d_i, e_i)
    pub commitments_map: BTreeMap<Identifier<C>, SigningCommitments<C>>,
}
``` [1](#0-0) 

`SigningNonces<C>` holds the two secret scalars `d_i` (hiding nonce) and `e_i` (binding nonce) generated during the presign round. [2](#0-1) 

Because `Clone` is derived, any caller can trivially duplicate the struct before passing it to a signing function. The library stores no "last-used presign" state and performs no check that the nonces inside have not been consumed. The `presign` function itself generates fresh nonces via `commit(&signing_share, &mut rng)` and returns them in the struct, but once returned, the library has no further control over their lifecycle. [2](#0-1) 

The same structural gap exists in the OT-based ECDSA path, where the code comment explicitly acknowledges the danger but provides no enforcement:

> "This work does depend on the private key though, and it's crucial that a presignature is never reused." [3](#0-2)

### Impact Explanation

**Critical — Extraction of private signing shares.**

In FROST, each participant's signature share is:

```
z_i = (d_i + e_i * ρ_i) + λ_i * s_i * c
```

where `ρ_i = H(i, M, B)` is the binding factor (message-dependent and public), `c = H(R, M, pk)` is the challenge, and `s_i` is the secret key share.

If the same `(d_i, e_i)` are used for two messages `M1` and `M2`:

```
z_i^(1) = d_i + e_i*ρ_i^(1) + λ_i * s_i * c^(1)
z_i^(2) = d_i + e_i*ρ_i^(2) + λ_i * s_i * c^(2)
```

Subtracting:

```
z_i^(1) - z_i^(2) = e_i*(ρ_i^(1) - ρ_i^(2)) + λ_i * s_i * (c^(1) - c^(2))
```

All quantities except `e_i` and `s_i` are public. A coordinator who observes both signature shares has one equation in two unknowns. However, the coordinator also observes `E_i = e_i * G` from the commitment map. With a second reuse (three signing sessions with the same nonces), the system becomes over-determined and `s_i` is directly solvable, leaking the participant's private key share. Even with two sessions, the binding factor's dependence on the commitment list `B` (which is fixed for a given presign output) means `ρ_i^(1)` and `ρ_i^(2)` differ only by the message, giving the attacker a linear system that can be solved with one additional piece of information the coordinator controls (e.g., choosing `M2` such that `c^(1) - c^(2)` is a known value).

For OT-based ECDSA, presignature reuse is even more directly catastrophic: reusing the same nonce `k` for two messages `m1`, `m2` gives `s1 - s2 = (m1 - m2)/k`, immediately recovering `k`, and then `x = (s1*k - m1)/r`. [3](#0-2) 

---

### Likelihood Explanation

**Medium.** A malicious coordinator is an explicitly modeled adversary in threshold signature protocols. The coordinator drives the signing protocol and chooses which presign output to supply to each signing invocation. Because `PresignOutput` is `Clone`, the coordinator (or any orchestration layer) can trivially retain a copy and initiate a second signing session with the same nonces and a different message. No cryptographic capability is required — only the ability to call the signing API twice with the same struct. An honest-but-buggy caller (e.g., retrying a failed signing attempt without generating a fresh presignature) triggers the same condition accidentally, matching the exploit scenario in the external report exactly.

---

### Recommendation

**Short term:** Remove the `Clone` derive from `PresignOutput<C>` and ensure the signing function takes it by value (consuming it), so that reuse requires an explicit and deliberate `clone()` call at the call site. This mirrors the fix recommended in the external report: store the last-used state and reject duplicate use.

**Long term:** Document a state-machine specification for the presign → sign lifecycle, analogous to the external report's recommendation. Enforce single-use semantics at the type level (e.g., a `PhantomData<*const ()>` marker to make the type non-`Send`/non-`Clone` by default, or a wrapper that zeroizes nonces on drop after first use).

### Proof of Concept

1. Honest participants run `presign(...)`, each receiving a `PresignOutput` containing their secret nonces `(d_i, e_i)` and the shared `commitments_map`.
2. The coordinator clones each participant's `PresignOutput` (trivially possible because `Clone` is derived).
3. The coordinator initiates `sign_v2` for message `M1` using the original `PresignOutput` values. Each participant computes and sends their signature share `z_i^(1)`.
4. The coordinator initiates a second `sign_v2` for a different message `M2` using the **cloned** `PresignOutput` values (same nonces). Each participant computes and sends `z_i^(2)`.
5. The coordinator now holds two equations per participant. Using the public binding factors `ρ_i^(1)`, `ρ_i^(2)` (computable from the public commitment map and messages) and the public challenges `c^(1)`, `c^(2)`, the coordinator solves for `s_i`, extracting each participant's private key share.
6. With threshold-many key shares recovered, the coordinator reconstructs the full private key. [1](#0-0) [2](#0-1)
