### Title
Unrestricted `execute_scheduled` Allows Any Caller to Grief Scheduled XCC Promises, Permanently Destroying User wNEAR - (`etc/xcc-router/src/lib.rs`)

---

### Summary

The XCC router's `execute_scheduled` function has no access control and can be called by any NEAR account. Because the function removes the promise from persistent storage before the asynchronous execution occurs, an attacker who calls it at a moment when the target promise would fail causes the promise to be permanently erased with no recovery path. The user's wNEAR, which was already burned by the XCC precompile to fund the call, is permanently lost.

---

### Finding Description

When a user submits a `CrossContractCallArgs::Delayed` XCC transaction, the Aurora Engine:

1. Burns the user's wNEAR ERC-20 tokens via `transferFrom` inside the XCC precompile (`engine-precompiles/src/xcc.rs`, lines 184–216).
2. Calls `withdraw_wnear_to_router` to convert that wNEAR into real NEAR and send it to the router sub-account.
3. Calls `schedule` on the router, which stores the `PromiseArgs` under a sequential nonce and emits a public log: `"Promise scheduled at nonce {}"`. [1](#0-0) 

The nonce is therefore publicly observable by any on-chain observer.

Later, anyone can call `execute_scheduled`: [2](#0-1) 

The function:
- **Line 151**: calls `self.scheduled_promises.remove(&nonce.0)` — this state mutation is committed to storage the moment the function returns successfully (NEAR only rolls back state on a panic, not on a downstream promise failure).
- **Line 154**: calls `Self::promise_create(promise)` — schedules the promise for asynchronous execution.
- **Line 155**: returns the promise.

If the asynchronous promise execution subsequently fails (e.g., the target NEP-141 contract rejects the call because the recipient is not registered, the contract is paused, or any other runtime rejection), NEAR does **not** roll back the `remove` on line 151. The promise is permanently gone from storage. There is no failure callback attached, so there is no recovery path. [3](#0-2) 

The developer comment claims safety because "it can only act on promises that were created via `schedule`." This is true for content integrity, but it ignores the timing dimension: an attacker controls *when* the promise is executed, and can choose a moment when execution is guaranteed to fail.

The XCC precompile's wNEAR burn is irreversible — there is no `error_refund` path for the `Delayed` (scheduled) XCC flow, unlike the `ExitToNear` precompile which has an `exit_to_near_precompile_callback` refund path: [4](#0-3) 

No equivalent refund callback exists for failed scheduled XCC promises.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

The user's wNEAR ERC-20 balance is burned at the time the XCC precompile runs. If an attacker calls `execute_scheduled` at a moment when the downstream promise fails, the user permanently loses the wNEAR they paid. The NEAR that was sent to the router sub-account is refunded to the router on promise failure, but the user has no mechanism to reclaim it — the router only accepts calls from its parent (the Aurora Engine), and there is no engine method to return stranded NEAR to a specific EVM user. The attack can be repeated for every scheduled promise across all router sub-accounts.

---

### Likelihood Explanation

**Medium-High.**

- The nonce of every scheduled promise is emitted as a public NEAR log, making it trivially discoverable.
- Many realistic conditions cause a downstream promise to fail: the recipient account is not registered with a NEP-141 token, the target contract is temporarily paused, the target method does not exist, or the attached NEAR is insufficient for storage deposit.
- An attacker can monitor the NEAR chain for `schedule` calls, identify the nonce, observe a window where the target call would fail (e.g., during a token contract migration or pause), and call `execute_scheduled` in that window.
- No special privilege or capital is required — only a NEAR account and gas.

---

### Recommendation

Add an access control check to `execute_scheduled` so that only the parent Aurora Engine account (or the EVM address that owns the router sub-account) can trigger execution. The `schedule` function already enforces `assert_preconditions()` (caller must be parent); `execute_scheduled` should do the same, or at minimum restrict callers to the parent:

```rust
pub fn execute_scheduled(&mut self, nonce: U64) {
    // Add: only the parent (Aurora Engine) may trigger execution
    self.assert_preconditions();   // <-- add this line
    let Some(promise) = self.scheduled_promises.remove(&nonce.0) else {
        env::panic_str("ERR_PROMISE_NOT_FOUND")
    };
    let promise_id = Self::promise_create(promise);
    env::promise_return(promise_id);
}
```

Alternatively, attach a failure callback that re-inserts the promise into storage (or refunds the user's wNEAR) if the downstream execution fails, so that the promise is not permanently lost on failure.

---

### Proof of Concept

1. User calls the XCC precompile with `CrossContractCallArgs::Delayed(promise)` where `promise` targets a NEP-141 `ft_transfer` with `attached_balance: Yocto::new(1)`. The user's wNEAR is burned; the router stores the promise at nonce `N`; the log `"Promise scheduled at nonce N"` is emitted. [5](#0-4) 

2. Attacker observes the log, learns nonce `N` and the target NEP-141 contract.

3. Attacker waits for (or engineers) a moment when the target NEP-141 contract would reject the `ft_transfer` — for example, the recipient account is not yet registered with the token contract.

4. Attacker calls `execute_scheduled({"nonce": "N"})` on the router sub-account. The function succeeds: the promise is removed from storage and scheduled. [6](#0-5) 

5. The asynchronous `ft_transfer` fails. NEAR does not roll back the `remove`. The promise is permanently gone.

6. The user's wNEAR (burned in step 1) is permanently lost. There is no refund callback and no engine method to recover it. [7](#0-6)
