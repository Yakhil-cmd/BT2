### Title
Incorrect Multi-NFT Royalty Calculation in WalletConnect Confirmation Causes User to Approve Wrong Amount - (File: `packages/gui/src/electron/commands/parseCommandDisplay.ts`)

### Summary
The `formatAmountWithRoyalties` function in `parseCommandDisplay.ts` contains an arithmetic accounting bug analogous to the external report: when multiple NFTs are present in a WalletConnect offer, the function divides the payment amount equally among NFTs before applying each royalty percentage, instead of applying each royalty to the full amount. This causes the `amountWithRoyalties` shown in the WalletConnect confirmation dialog to be significantly lower than what the user will actually pay, causing them to approve a transaction under a false understanding of its cost.

### Finding Description

In `formatAmountWithRoyalties`:

```typescript
const splitAmount = amount / BigInt(royaltyPercentages.length);  // ← wrong: divides first
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
``` [1](#0-0) 

When N NFTs are present, the code computes `splitAmount = amount / N` (integer division), then applies each royalty to `splitAmount` rather than to `amount`. This underestimates total royalties by a factor of N.

The correct formula is:
```typescript
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (amount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
```

This `amountWithRoyalties` value is the primary cost figure shown to the user in the WalletConnect confirmation dialog before they approve a `take_offer` or `create_offer_for_ids` command. [2](#0-1) 

### Impact Explanation

The WalletConnect confirmation dialog shows `amountWithRoyalties` as the total cost the user will pay. Because the displayed value is incorrectly lower than the actual on-chain cost (enforced by the blockchain regardless of what the GUI shows), the user approves a transaction believing they will spend less XCH or CAT than they actually will. This is WalletConnect state that causes a user to approve the wrong amount — a High-impact finding per the allowed scope.

### Likelihood Explanation

Any WalletConnect-connected dApp can craft an offer involving two or more NFTs with royalties. No special privileges are required. The attacker only needs to present a multi-NFT offer through WalletConnect; the GUI's own arithmetic then produces the misleading confirmation value.

### Recommendation

Remove the `splitAmount` division. Apply each royalty percentage directly to the full `amount`:

```typescript
function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],
): string | undefined {
  if (royaltyPercentages.length === 0 || line.kind === 'nft') {
    return undefined;
  }

  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (amount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;
  ...
}
```

### Proof of Concept

The existing test in `parseCommandDisplay.test.ts` (lines 251–335) demonstrates the bug:

- **Offer**: 100,000,000 mojos (0.0001 XCH) for two NFTs with royalties 500 (5%) and 10 (0.1%)
- **Displayed** (`amountWithRoyalties`): `splitAmount = 50,000,000`; royalty = `2,500,000 + 50,000 = 2,550,000`; total = **0.00010255 XCH**
- **Correct**: royalty = `5,000,000 + 100,000 = 5,100,000`; total = **0.0001051 XCH**
- **Discrepancy**: user sees 0.00010255 XCH but the blockchain enforces 0.0001051 XCH — a ~2× understatement of royalties [3](#0-2) 

The user approves the WalletConnect command believing they will spend 0.00010255 XCH, but the actual deduction is 0.0001051 XCH. The root cause is the same class as the external report: an edge-case arithmetic error in a fee/royalty calculation that produces an incorrect accounting value used in a security-sensitive approval flow.

function withRoyaltyTotals(
  items: DisplayWalletDeltaItemWithKey[],
  amounts: Record<string, bigint>,
  oppositeSideLines: DisplayWalletDeltaItem[],
): DisplayWalletDeltaItem[] {
  const royaltyPercentages = royaltyPercentagesForSide(oppositeSideLines);

  return items.map(({ key, line }) => {
    if (line.kind === 'nft') {
      return line;
    }

    const amount = amounts[key];
    const amountWithRoyalties = amount ? formatAmountWithRoyalties(line, amount, royaltyPercentages) : undefined;

    return amountWithRoyalties ? { ...line, amountWithRoyalties } : line;
  });
}
