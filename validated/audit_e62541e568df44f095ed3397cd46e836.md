### Title
Missing Minimum Threshold Validation in FROST Signing Allows Sub-Threshold Signature Creation - (File: src/frost/mod.rs)

### Summary
The `assert_sign_inputs` function, which guards all FROST EdDSA and RedJubjub signing entry points, validates that `threshold <= participants.len()` but never enforces a minimum value of `threshold >= 2`. The DKG entry point (`assert_key_invariants`) explicitly rejects `threshold < 2`, but this lower-bound guard is absent from the signing path. A caller can pass `threshold = 1` (or even `0`) to any FROST signing function, causing the protocol to proceed with a degenerate threshold that violates the security guarantees of the scheme.

### Finding Description

`assert_key_invariants` in `src/dkg.rs` enforces a hard lower bound:

```rust
if threshold < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
}
``` [1](#0-0) 

`assert_sign_inputs` in `src/frost/mod.rs`, which is the sole validation gate for `sign_v1`, `sign_v2` (EdDSA), and `sign` (RedJubjub), only checks the upper bound:

```rust
if threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
``` [2](#0-1) 

There is no corresponding `ThresholdTooSmall` guard. A `threshold` of `0` or `1` passes all checks and is forwarded into `construct_key_package`, which converts it directly to a `u16` and hands it to the `frost_ed25519` / `reddsa` library:

```rust
u16::try_from(threshold.value()).map_err(|_| { ... })?
``` [3](#0-2) 

All three public signing entry points delegate to `assert_sign_inputs` without any additional threshold floor check: [4](#0-3) [5](#0-4) 

### Impact Explanation

With `threshold = 1` accepted by `assert_sign_inputs`, the `KeyPackage` is constructed with `min_signers = 1`. The FROST aggregation step (`aggregate()`) then requires only a single signature share to produce a valid, verifiable group signature. The coordinator, who always contributes their own share, can complete the protocol unilaterally — producing a cryptographically valid threshold signature without the participation of any other key-holder. This directly violates the `t-of-n` security property that the DKG established.

**Matched impact**: *High — Corruption of sign outputs so honest parties accept unusable cryptographic outputs* (a signature produced below the agreed threshold is a protocol-level forgery relative to the key's security parameters).

### Likelihood Explanation

The threshold value is an externally supplied parameter at every call site. Any application layer that:
- Misconfigures the threshold (e.g., passes `min_signers` instead of `threshold`, or passes a raw `1`), or
- Is controlled by a malicious coordinator who deliberately supplies `threshold = 1` to their own signing invocation

will silently bypass the intended security level. Because `assert_key_invariants` already demonstrates the library authors know this check is required, its absence from `assert_sign_inputs` is an oversight rather than an intentional design choice.

### Recommendation

Add the same lower-bound guard that exists in `assert_key_invariants` to `assert_sign_inputs`:

```rust
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

This should be inserted in `src/frost/mod.rs` inside `assert_sign_inputs`, immediately after the upper-bound check, mirroring the pattern at `src/dkg.rs` lines 580–582.

### Proof of Concept

1. Call `sign_v1` (EdDSA) with `threshold = 1` and any two participants.
2. `assert_sign_inputs` passes: `1 <= 2` (upper bound satisfied), no lower bound check exists.
3. `construct_key_package` builds a `KeyPackage` with `min_signers = 1`.
4. The coordinator generates their own commitment and signature share in Round 1 and Round 2.
5. `aggregate()` is called with a `signature_shares` map containing only the coordinator's share; the FROST library accepts it because `min_signers = 1`.
6. A fully valid, verifiable Ed25519 threshold signature is returned — produced by a single party against a key that was generated with a threshold of ≥ 2.
