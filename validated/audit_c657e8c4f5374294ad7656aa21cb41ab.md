### Title
Malicious Participant Can Corrupt CKD Output Without Detection Due to Missing Share Proof Verification - (File: src/confidential_key_derivation/protocol.rs)

### Summary
The CKD coordinator unconditionally sums participant-supplied `(Y_i, C_i)` shares with no proof that each share is correctly formed. A single malicious participant can send an arbitrary group element pair, causing the coordinator to output a corrupted confidential derived key that honest parties will accept as valid.

### Finding Description
In `do_ckd_coordinator` the coordinator collects each participant's `(norm_big_y, norm_big_c)` pair and adds them together:

```rust
// src/confidential_key_derivation/protocol.rs  lines 50-56
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
``` [1](#0-0) 

Each honest participant computes their share in `compute_signature_share` as:

```
norm_big_y = λ_i · (y_i · G)
norm_big_c = λ_i · (x_i · H(pk, app_id) + y_i · app_pk)
``` [2](#0-1) 

The coordinator never verifies that the received `(Y_i, C_i)` satisfies this relation. There is no zero-knowledge proof (e.g., a Schnorr-style proof of knowledge of `x_i` and `y_i` such that `C_i = x_i · H(pk, app_id) + y_i · app_pk` and `Y_i = y_i · G`) attached to each participant's message.

A malicious participant `j` can instead send:

```
Y_j' = Y_j + Δ_Y   (arbitrary additive offset)
C_j' = C_j + Δ_C   (arbitrary additive offset)
```

The coordinator then computes:

```
Y_final = Y_honest + Δ_Y
C_final = C_honest + Δ_C
```

When the application calls `unmask(app_sk)`:

```
C_final - app_sk · Y_final
  = (msk · H(pk, app_id) + y · app_pk) + Δ_C
    - app_sk · (y · G + Δ_Y)
  = msk · H(pk, app_id) + Δ_C - app_sk · Δ_Y
```

The derived confidential key is shifted by `Δ_C - app_sk · Δ_Y`, which is attacker-controlled. The coordinator has no way to detect this corruption.

### Impact Explanation
The coordinator and any downstream consumer of `CKDOutput` will accept a corrupted confidential derived key as if it were the legitimate output of the protocol. Because the `unmask` step is performed outside the MPC (by the application holding `app_sk`), the corruption is invisible at the protocol layer. This maps directly to: **High — Corruption of CKD outputs so honest parties accept inconsistent or unusable cryptographic outputs.**

### Likelihood Explanation
Any single participant in the CKD session is sufficient to trigger this. The participant's deviation is a simple additive offset to two group elements — no cryptographic capability beyond participation in the protocol is required. The malicious participant does not need to know `msk`, `app_sk`, or any other secret. The attack is reachable by any unprivileged library caller who holds a valid key share and participates in a CKD session.

### Recommendation
Each participant should attach a zero-knowledge proof of correct share formation alongside their `(Y_i, C_i)` message. Concretely, a Chaum-Pedersen / sigma-protocol proof demonstrating knowledge of `(x_i, y_i)` such that:

- `Y_i = y_i · G`
- `C_i = x_i · H(pk, app_id) + y_i · app_pk`
- `x_i` is consistent with the participant's public verification share from DKG

should be generated by `compute_signature_share` and verified by the coordinator before accumulating each participant's contribution. This is the direct analog of setting a `quoteAssetLimit` in the AMM: it constrains what each participant can contribute to the output.

### Proof of Concept

1. Honest participants `{P1, P2, P3}` run `ckd(...)` with coordinator `P1`.
2. Malicious `P2` intercepts the protocol at the `chan.send_private(waitpoint, coordinator, ...)` step and instead sends `(Y_2 + Δ_Y, C_2 + Δ_C)` for arbitrary non-zero `Δ_Y, Δ_C ∈ G1`.
3. `P1` (coordinator) executes `do_ckd_coordinator`, sums all shares including `P2`'s corrupted pair, and returns `CKDOutput::new(Y_honest + Δ_Y, C_honest + Δ_C)`.
4. The application calls `ckd_output.unmask(app_sk)` and obtains `msk · H(pk, app_id) + Δ_C - app_sk · Δ_Y` — a key that differs from the intended `msk · H(pk, app_id)` by an attacker-chosen offset.
5. No error is raised anywhere in the protocol; the corrupted key is silently accepted. [3](#0-2) [4](#0-3)

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}
