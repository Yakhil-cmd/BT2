### Title
Missing Signer Authorization in `MsgRegisterEncryptionKey` Allows Arbitrary e2ee Key State Corruption — (File: x/e2ee/keeper/keeper.go)

---

### Summary

The `x/e2ee` module's `RegisterEncryptionKey` message handler stores an encryption key for an arbitrary `Address` field without verifying that the transaction signer owns that address. Any unprivileged user can overwrite any other user's registered encryption key, directly corrupting e2ee key state and enabling silent interception of encrypted messages.

---

### Finding Description

In `x/e2ee/keeper/keeper.go`, `RegisterEncryptionKey` unconditionally writes `req.Key` into the KV store under `req.Address`:

```go
func (k Keeper) RegisterEncryptionKey(
    ctx context.Context,
    req *types.MsgRegisterEncryptionKey,
) (*types.MsgRegisterEncryptionKeyResponse, error) {
    if err := k.registerEncryptionKey(ctx, req.Address, []byte(req.Key)); err != nil {
        return nil, err
    }
    return &types.MsgRegisterEncryptionKeyResponse{}, nil
}
``` [1](#0-0) 

The internal `registerEncryptionKey` simply calls `KVStore.Set` with no authorization check:

```go
sdk.UnwrapSDKContext(ctx).KVStore(k.storeKey).Set(types.KeyPrefix(bz), key)
``` [2](#0-1) 

`ValidateBasic()` in `x/e2ee/types/msg.go` only validates bech32 format and age X25519 key format — it performs no signer-vs-address check:

```go
func (m *MsgRegisterEncryptionKey) ValidateBasic() error {
    if _, err := sdk.AccAddressFromBech32(m.Address); err != nil {
        return fmt.Errorf("invalid address: %w", err)
    }
    return ValidateRecipientKey(m.Key)
}
``` [3](#0-2) 

The `MsgRegisterEncryptionKey` proto message carries both `Address` and `Key` as free-form fields with no binding to the actual transaction signer: [4](#0-3) 

There is no place in the call chain — neither in `ValidateBasic`, the keeper handler, nor any middleware — that asserts `msg.Address == tx.signer`.

---

### Impact Explanation

An attacker who calls `MsgRegisterEncryptionKey{Address: victim, Key: attacker_controlled_age_key}` silently replaces the victim's registered public encryption key with one the attacker controls. Any party that subsequently queries the victim's key via `KeyRequest` and encrypts a message to it will encrypt to the attacker's key. The attacker can decrypt all such messages. This is direct, permanent corruption of e2ee key/message state with concrete security impact: confidentiality of the e2ee system is broken for any targeted address.

Additionally, because each distinct address maps to a separate KV entry and there is no rate limit or per-address write cost beyond the base transaction fee, an attacker can register keys for an unbounded number of addresses, bloating the e2ee KV store at near-zero marginal cost per entry — the direct analog to the Existential Deposit bloat in the external report. [5](#0-4) 

---

### Likelihood Explanation

The entry point is a standard, publicly accessible Cosmos SDK message (`/e2ee.Msg/RegisterEncryptionKey`) reachable by any unprivileged account. No special role, key, or privilege is required. The attack requires only a valid bech32 address and a syntactically valid age X25519 public key, both trivially constructable.

---

### Recommendation

Add a signer authorization check in `RegisterEncryptionKey` to assert that `req.Address` matches the transaction signer before writing to the store:

```go
func (k Keeper) RegisterEncryptionKey(
    ctx context.Context,
    req *types.MsgRegisterEncryptionKey,
) (*types.MsgRegisterEncryptionKeyResponse, error) {
    signers := sdk.UnwrapSDKContext(ctx).TxSigner() // or equivalent signer extraction
    if req.Address != signers.String() {
        return nil, sdkerrors.ErrUnauthorized.Wrap("signer must match address")
    }
    if err := k.registerEncryptionKey(ctx, req.Address, []byte(req.Key)); err != nil {
        return nil, err
    }
    return &types.MsgRegisterEncryptionKeyResponse{}, nil
}
```

The `MsgRegisterEncryptionKey` proto should also declare `Address` as the signer field via the `cosmos.msg.v1.signer` annotation so the framework enforces it automatically.

---

### Proof of Concept

1. Victim `V` has previously registered their legitimate age public key via `MsgRegisterEncryptionKey{Address: V, Key: V_pubkey}`.
2. Attacker `A` generates a fresh age X25519 keypair: `(A_pubkey, A_privkey)`.
3. Attacker broadcasts `MsgRegisterEncryptionKey{Address: V, Key: A_pubkey}` signed by `A`.
4. The keeper writes `A_pubkey` into the KV store under `V`'s address with no error.
5. Any sender that queries `KeyRequest{Address: V}` now receives `A_pubkey` and encrypts messages to it.
6. Attacker decrypts all such messages using `A_privkey`.
7. Victim `V` can no longer decrypt messages intended for them, and has no on-chain indication the key was replaced. [1](#0-0) [6](#0-5)
