### Title
Silo Mode ERC-20 Fallback Permanently Redirects User Tokens When Whitelist Is Enabled — (`engine/src/contract_methods/silo/mod.rs`)

---

### Summary

In Silo mode, when the `Address` whitelist is enabled and a recipient's EVM address is absent from it, incoming ERC-20 tokens are silently redirected to the admin-configured `erc20_fallback_address` instead of the intended recipient. An admin enabling the whitelist (or removing a user from it) after users have established token flows causes those users' incoming ERC-20 tokens to be permanently lost with no recourse — a direct structural analog to the reported vulnerability.

---

### Finding Description

`SiloParamsArgs` carries two coupled fields:

```rust
pub struct SiloParamsArgs {
    pub fixed_gas: EthGas,
    /// EVM address, which is used for withdrawing ERC-20 base tokens in case
    /// a recipient of the tokens is not in the silo white list.
    /// Note: the logic described above works only if the fallback address
    /// is set by `set_silo_params` function. In other words, in Silo mode.
    pub erc20_fallback_address: Address,
}
``` [1](#0-0) 

The function `is_allow_receive_erc20_tokens` gates whether a recipient address may receive ERC-20 tokens:

```rust
pub fn is_allow_receive_erc20_tokens<I: IO + Copy>(io: &I, address: &Address) -> bool {
    is_address_allowed(io, address)
}
``` [2](#0-1) 

`is_address_allowed` checks the `WhitelistKind::Address` list:

```rust
fn is_address_allowed<I: IO + Copy>(io: &I, address: &Address) -> bool {
    let list = Whitelist::init(io, WhitelistKind::Address);
    !list.is_enabled() || list.is_exist(address)
}
``` [3](#0-2) 

When this check fails, the documented behavior is that tokens are minted to `erc20_fallback_address` rather than the intended recipient. The admin controls two levers that can trigger this outcome for any existing user:

1. **Enable the whitelist** — `set_whitelist_status` with `active: true` for `WhitelistKind::Address`.
2. **Remove a user** — `remove_entry_from_whitelist` targeting the user's address. [4](#0-3) [5](#0-4) 

After either action, any ERC-20 tokens sent to that user's Aurora address via `ft_on_transfer` are permanently redirected to the fallback address. The `ft_on_transfer` handler returns `0` (success) in this path, so the NEP-141 contract does **not** refund the sender — the tokens are gone from the user's perspective. [6](#0-5) 

This is structurally identical to the reported vulnerability: just as `AuraLocker.isShutdown` makes the penalty-free `lock()` path always revert, enabling the Silo whitelist makes the normal token-receipt path unavailable, silently routing tokens to the fallback address with no user recourse.

---

### Impact Explanation

Users lose **100%** of incoming ERC-20 tokens when their address is absent from an enabled whitelist. The tokens are minted to the admin-controlled fallback address and are unrecoverable by the original recipient. This constitutes either **permanent freezing of funds** (if the fallback is a dead address) or **direct theft of user funds** (if the fallback is admin-controlled). Impact: **Critical**.

---

### Likelihood Explanation

Silo mode with whitelists is a production feature of Aurora Engine, actively used by operators running permissioned Aurora instances. Any operator can enable the `Address` whitelist at any time. Users with automated token flows (e.g., bridge integrations, scheduled transfers) would receive no warning before their tokens are redirected. The whitelist can be toggled without notice, making this a realistic operational risk.

---

### Recommendation

When `is_allow_receive_erc20_tokens` returns `false`, the `ft_on_transfer` handler should treat the call as an error and return the full token amount to the sender (the NEP-141 refund path), rather than silently minting to the fallback address. This ensures users never lose tokens due to a whitelist configuration change they cannot control. If the fallback-redirect behavior is intentional, users must be able to query their whitelist status before initiating a transfer, and the bridge UI must surface this check.

---

### Proof of Concept

1. Admin deploys Aurora in Silo mode: `set_silo_params(Some(SiloParamsArgs { fixed_gas: ..., erc20_fallback_address: ADMIN_ADDRESS }))`.
2. User's EVM address is initially in the `Address` whitelist; they receive ERC-20 tokens normally.
3. Admin calls `set_whitelist_status({ kind: Address, active: true })` to enable the whitelist, then `remove_entry_from_whitelist(WhitelistAddressArgs { kind: Address, address: USER_ADDRESS })`.
4. An automated bridge or counterparty sends ERC-20 tokens to `USER_ADDRESS` via `ft_on_transfer`.
5. `is_allow_receive_erc20_tokens` returns `false` for `USER_ADDRESS`.
6. Tokens are minted to `ADMIN_ADDRESS` (the fallback) instead of `USER_ADDRESS`.
7. `ft_on_transfer` returns `"0"` (success) — the NEP-141 contract does not refund the sender.
8. User loses 100% of their tokens with no recourse; the admin action that caused this was a legitimate whitelist management operation. [1](#0-0) [7](#0-6) [8](#0-7)

#[named]
pub fn ft_on_transfer<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();
        let mut engine: Engine<_, _> = Engine::new(
            predecessor_address(&predecessor_account_id),
            current_account_id.clone(),
            io,
            env,
        )?;

        sdk::log!("Call ft_on_transfer");

        let args: FtOnTransferArgs = read_json_args(&io)?;
        let result = if predecessor_account_id == get_connector_account_id(&io)? {
            engine.receive_base_tokens(&args)
        } else {
            engine.receive_erc20_tokens(
                &predecessor_account_id,
                &args,
                &current_account_id,
                handler,
            )
        };

        #[allow(clippy::used_underscore_binding)]
        let amount_to_return = if let Err(_err) = &result {
            sdk::log!("Error in ft_on_transfer: {_err:?}");
            // An error occurred, so we need to return the amount of tokens to the sender.
            args.amount.as_u128()
        } else {
            // Everything is ok, so return 0.
            0
        };

        let output = crate::prelude::format!("\"{amount_to_return}\"");
        io.return_output(output.as_bytes());

        // In case of an error, we just return Ok(None) to avoid a panic in the contract. It's ok
        // because in case of an error, we already returned the amount of tokens to the sender.
        Ok(result.unwrap_or(None))
    })
