### Title
Unvalidated Zero `reward_address` Allows Permanent Freezing of Unclaimed Staker and Pool Member Yield - (File: `src/staking/staking.cairo`, `src/pool/pool.cairo`)

---

### Summary
Both `Staking.change_reward_address()` and `Pool.change_reward_address()` (as well as the initial `stake()` and `enter_delegation_pool()` entry points) accept a `reward_address` parameter without validating it is non-zero. A staker or pool member who sets their reward address to `0x0` will have all future claimed rewards transferred to the zero address, permanently freezing their unclaimed yield.

---

### Finding Description

In `src/staking/staking.cairo`, the `change_reward_address` function at line 517 accepts the new `reward_address` and performs only one check — that the address is not a registered token contract — but never asserts `reward_address.is_non_zero()`:

```cairo
fn change_reward_address(ref self: ContractState, reward_address: ContractAddress) {
    self.general_prerequisites();
    assert!(
        !self.does_token_exist(token_address: reward_address),
        "{}",
        GenericError::REWARD_ADDRESS_IS_TOKEN,
    );
    // No zero-address check here
    let staker_address = get_caller_address();
    let mut staker_info = self.internal_staker_info(:staker_address);
    ...
    staker_info.reward_address = reward_address;
    self.write_staker_info(:staker_address, :staker_info);
``` [1](#0-0) 

The same omission exists in the initial `stake()` call, where `reward_address` is stored without a zero check: [2](#0-1) 

In `src/pool/pool.cairo`, `Pool.change_reward_address()` similarly only checks that the reward address is not the pool's token contract, with no zero-address guard: [3](#0-2) 

And `enter_delegation_pool()` has the same gap: [4](#0-3) 

When rewards are later claimed, `send_rewards_to_staker` in `staking.cairo` unconditionally transfers to whatever `reward_address` is stored:

```cairo
fn send_rewards_to_staker(...) {
    let reward_address = staker_info.reward_address;
    let amount = staker_info.unclaimed_rewards_own;
    ...
    token_dispatcher.checked_transfer(recipient: reward_address, amount: amount.into());
``` [5](#0-4) 

Likewise, `Pool.claim_rewards()` transfers directly to the stored `reward_address`: [6](#0-5) 

If `reward_address` is `0x0`, the ERC-20 `checked_transfer` call sends tokens to the zero address. On Starknet, address `0x0` is a valid felt252 value and ERC-20 transfers to it succeed — the tokens are irrecoverably lost.

---

### Impact Explanation

**High — Permanent freezing of unclaimed yield.**

Any staker or pool member who sets (or initially registers with) a zero reward address will have all accumulated STRK rewards permanently sent to address `0x0` upon the next `claim_rewards` call. The tokens cannot be recovered. This matches the allowed impact: *"Permanent freezing of unclaimed yield or unclaimed royalties."*

---

### Likelihood Explanation

**Medium.** The entry path requires only an unprivileged staker or pool member to call `change_reward_address(0)` or `stake(..., reward_address: 0, ...)`. No privileged role, leaked key, or external dependency is needed. The scenario is reachable by:
- A user making a mistake (passing zero by accident in a script or UI bug).
- A griefing actor deliberately burning their own yield (damaging protocol TVL metrics and trust).

The `general_prerequisites()` guard only checks that the **caller** is non-zero, not the `reward_address` argument: [7](#0-6) 

---

### Recommendation

Add a non-zero assertion for `reward_address` in all four entry points:

1. `Staking::stake()` — before writing `staker_info`
2. `Staking::change_reward_address()` — at the top of the function
3. `Pool::enter_delegation_pool()` — before writing `pool_member_info`
4. `Pool::change_reward_address()` — at the top of the function

Example fix for `Staking::change_reward_address()`:
```cairo
fn change_reward_address(ref self: ContractState, reward_address: ContractAddress) {
    self.general_prerequisites();
    assert!(reward_address.is_non_zero(), "{}", GenericError::ZERO_ADDRESS);
    assert!(
        !self.does_token_exist(token_address: reward_address),
        "{}",
        GenericError::REWARD_ADDRESS_IS_TOKEN,
    );
    ...
}
```

Note that `GenericError::ZERO_ADDRESS` already exists in the codebase (used in `add_token`): [8](#0-7) 

---

### Proof of Concept

1. Staker Alice calls `Staking::stake(reward_address: 0x0, operational_address: <valid>, amount: <min_stake>)`. The zero-address check is absent; the call succeeds and `staker_info.reward_address = 0x0` is stored. [9](#0-8) 

2. Alternatively, Alice already has a valid reward address but calls `Staking::change_reward_address(reward_address: 0x0)`. The only guard (`does_token_exist`) passes for address zero; the zero address is written to storage. [1](#0-0) 

3. After epochs pass and rewards accrue, anyone calls `Staking::claim_rewards(staker_address: Alice)`.

4. `send_rewards_to_staker` reads `reward_address = 0x0` and executes `token_dispatcher.checked_transfer(recipient: 0x0, amount: rewards)`. The transfer succeeds; STRK rewards are sent to the zero address and permanently frozen. [10](#0-9) 

5. The same flow applies to pool members via `Pool::enter_delegation_pool` / `Pool::change_reward_address` → `Pool::claim_rewards`. [11](#0-10)
