### Title
Useless Post-Subtraction Guard Always True for Unsigned `gas_used` — (`File: basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`)

### Summary
`compute_gas_refund` performs a plain (non-checked) subtraction `gas_limit - gas_used` and then asserts `total_gas_refund <= gas_limit`. Because both operands are `u64`, the assertion is a tautology whenever the subtraction does not underflow, and it is placed *after* the subtraction that would already panic (debug) or silently wrap (release) if `gas_used > gas_limit`. The guard therefore provides no protection and no informative error message — a direct structural analog of the reported `require(_balanceFrom.sub(_amount) >= 0)` pattern.

### Finding Description

In `compute_gas_refund`:

```rust
// basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs

let total_gas_refund = gas_limit - gas_used;          // line 83 — plain subtraction
// ...
require_internal!(
    total_gas_refund <= gas_limit,                     // line 86 — always true
    "Gas refund greater than gas limit",
    system
)?;
```

`total_gas_refund = gas_limit - gas_used`. For any `u64` value of `gas_used` where the subtraction does not underflow, `total_gas_refund ≤ gas_limit` is equivalent to `gas_used ≥ 0` — trivially true for an unsigned integer. The `require_internal!` check therefore **never fires** in the normal execution path.

The only scenario where it could fire is when `gas_used > gas_limit`, causing the subtraction on line 83 to underflow. In **debug/test builds** the subtraction panics before the check is ever reached, producing no informative error. In **release builds** the value wraps to `u64::MAX - (gas_used - gas_limit) + 1`, which is larger than `gas_limit`, so the check fires — but only as a side-effect of wrapping arithmetic, not as the intended guard.

`gas_used` can exceed `gas_limit` via the `minimal_gas_used` path:

```rust
let mut gas_used = core::cmp::max(gas_used, minimal_gas_used);   // line 56
```

`minimal_gas_used` is the gas consumed during validation and is passed in from callers in `validation_impl.rs` and `process_l1_transaction.rs`. If a caller supplies a `minimal_gas_used` value that exceeds `gas_limit` (e.g., due to a separate accounting bug or an edge case in L1 transaction handling), `gas_used` becomes `minimal_gas_used > gas_limit`, the subtraction on line 83 underflows, and the guard on line 86 is either unreachable (debug) or fires only because of wrapping (release). ### Impact Explanation

If `gas_used` exceeds `gas_limit` (reachable via `minimal_gas_used` or a future delta-gas edge case), the refund calculation silently wraps in release builds, producing a `total_gas_refund` near `u64::MAX`. This value is then used to credit the transaction sender with a near-maximal gas refund, constituting a **public funds-loss path** (over-refund of ETH/native token to the transaction originator). The `require_internal!` guard that was intended to prevent this is logically inert in the normal case and misplaced in the error case. [1](#0-0) 

### Likelihood Explanation

The subtraction underflow requires `gas_used > gas_limit`. The `delta_gas` path is bounded by the native limit and cannot push `gas_used` above `gas_limit` under normal conditions. However, the `minimal_gas_used` path (line 56) is an externally-supplied parameter from callers in `validation_impl.rs` and `process_l1_transaction.rs`. An attacker who can influence validation gas accounting — for example, by crafting a transaction whose validation phase consumes gas in a way that is mis-accounted — could trigger the underflow. The useless guard means there is no safety net and no informative error message to detect the condition. [3](#0-2) 

### Recommendation

Replace the post-subtraction tautology with a pre-subtraction guard that checks the actual invariant and uses `checked_sub`:

```rust
// Before the subtraction:
require_internal!(
    gas_used <= gas_limit,
    "Gas used exceeds gas limit in refund calculation",
    system
)?;
let total_gas_refund = gas_limit - gas_used;
// Remove the now-redundant post-subtraction check.
```

This mirrors the recommendation in the original report: check the meaningful precondition (`gas_used <= gas_limit`) rather than the always-true postcondition (`total_gas_refund <= gas_limit`), and do so *before* the arithmetic that would underflow. [1](#0-0) 

### Proof of Concept

1. `gas_used` is initialized safely: `gas_limit - remaining_ergs_as_gas` (line 31–33, checked).
2. `gas_used` is then reduced by `evm_refund ≤ gas_used/5` (line 48) — still `≤ gas_limit`.
3. `gas_used = max(gas_used, minimal_gas_used)` (line 56). If `minimal_gas_used > gas_limit`, `gas_used` now exceeds `gas_limit`.
4. Line 83: `gas_limit - gas_used` underflows. In release mode, wraps to `~u64::MAX`.
5. Line 86: `require_internal!(total_gas_refund <= gas_limit)` — `u64::MAX <= gas_limit` is false, so the function returns an `InternalError`. **But** the check is logically equivalent to `gas_used >= 0` in the non-underflow case, so it never fires as a meaningful guard and provides no informative error message in debug builds where the panic occurs at line 83 instead. pub(crate) fn compute_gas_refund<S: EthereumLikeTypes>(
    system: &mut System<S>,
    to_charge_for_pubdata: S::Resources,
    gas_limit: u64,
    minimal_gas_used: u64,
    native_per_gas: u64,
    resources: &mut S::Resources,
) -> Result<RefundInfo, InternalError> {
    // Already checked
    resources.charge_unchecked(&to_charge_for_pubdata);

    let mut gas_used = gas_limit
        .checked_sub(resources.ergs().0.div_floor(ERGS_PER_GAS))
        .ok_or(internal_error!("gas remaining > gas limit"))?;
    resources.exhaust_ergs();

    system_log!(system, "Gas used before refund calculations: {gas_used}\n");

    // Following EIP-3529, refunds are capped to 1/5 of the gas used
    let evm_refund = {
        let full_refund_ergs = system.io.get_refund_counter().ergs();
        let full_refund_gas = full_refund_ergs.0.div_floor(ERGS_PER_GAS);
        let max_refund = gas_used / 5;
        core::cmp::min(full_refund_gas, max_refund)
    };

    system_log!(system, "Gas refund from refund counters = {evm_refund}\n");

    gas_used -= evm_refund;

    system_log!(
        system,
        "Minimal gas used from validation = {minimal_gas_used}\n"
    );

    #[allow(unused_mut)]
    let mut gas_used = core::cmp::max(gas_used, minimal_gas_used);
