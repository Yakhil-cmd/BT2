### Title
Malicious CKD Participant Can Corrupt Coordinator's Derived Secret Output Without Detection - (File: src/confidential_key_derivation/protocol.rs)

### Summary

In `do_ckd_coordinator`, participant-provided `CKDOutput` values (`big_y`, `big_c`) are summed directly without any proof of correctness or commitment binding. A single malicious participant can inject arbitrary group elements, causing the coordinator to compute a corrupted CKD output that does not correspond to the actual threshold secret. No honest party can detect the corruption before accepting the result.

### Finding Description

The CKD protocol is designed so that each participant `i` computes and sends:

- `norm_big_y = λ_i · y_i · G`
- `norm_big_c = λ_i · (x_i · H(pk ‖ app_id) + y_i · app_pk)`

The coordinator aggregates these by summing all contributions. The final output satisfies `C − y · app_pk = msk · H(pk ‖ app_id)`, which is the confidential key.

The vulnerable aggregation in `do_ckd_coordinator` is:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
``` [1](#0-0) 

There is **no zero-knowledge proof, commitment scheme, or consistency check** verifying that the received `(big_y, big_c)` pair was honestly computed from the participant's actual key share `x_i` and a valid nonce `y_i`. The coordinator blindly adds whatever group elements it receives.

Compare this to the DKG protocol, which requires each participant to supply a proof of knowledge of their secret coefficient before their commitment is accepted: [2](#0-1) 

No analogous verification exists in the CKD aggregation path.

**Attacker-controlled entry path:**

1. Attacker is a valid participant in a CKD session (no privilege required beyond being in the participant list).
2. Attacker calls `ckd(...)` and participates honestly until the contribution round.
3. Instead of sending the correctly computed `(norm_big_y, norm_big_c)`, the attacker sends `do_ckd_participant` with a crafted message — any arbitrary pair of `ElementG1` values — directly to the coordinator via `chan.send_private(waitpoint, coordinator, &(crafted_big_y, crafted_big_c))`. [3](#0-2) 
4. The coordinator receives this via `recv_from_others` and adds it unconditionally. [4](#0-3) 
5. The resulting `CKDOutput` is corrupted. When the application calls `ckd_output.unmask(app_sk)`, it recovers a wrong value instead of `msk · H(pk ‖ app_id)`.

The analog to the external report is direct: just as `contractCallData` was accepted and rendered without sanitization, here participant-provided `CKDOutput` is accepted and aggregated without any proof of correctness, allowing a malicious participant to inject arbitrary values into a critical cryptographic computation.

### Impact Explanation

**High — Corruption of CKD output so honest parties accept an unusable or attacker-biased cryptographic output.**

The coordinator, which is the sole recipient of the final `CKDOutput`, will compute and return a value that does not equal `msk · H(pk ‖ app_id)`. Any downstream application relying on this derived secret (e.g., a TEE application using the confidential key) will silently receive a wrong key. The corruption is undetectable by the coordinator without out-of-band verification. A single malicious participant is sufficient to trigger this.

### Likelihood Explanation

Any participant in a CKD session can exploit this. The attacker needs only to be a valid member of the participant list — no privileged access, no leaked keys, no external assumptions. The attack requires sending one crafted private message to the coordinator, which is a normal protocol action. The likelihood is **high** given that the CKD protocol is explicitly designed for multi-party settings where not all participants are trusted.

### Recommendation

Add a zero-knowledge proof of correct computation to each participant's contribution, binding `big_c` and `big_y` to the participant's committed public key share and a committed nonce. Specifically:

1. Each participant should commit to their nonce `y_i` before revealing `big_y = λ_i · y_i · G`.
2. Each participant should provide a proof that `big_c` was computed as `λ_i · (x_i · H(pk ‖ app_id) + y_i · app_pk)` using their actual key share `x_i` (whose public counterpart `x_i · G` is known from DKG output) and the same `y_i`.
3. The coordinator must verify all proofs before aggregating, analogous to how `verify_proof_of_knowledge` is called for every participant in the DKG protocol before their commitment is accepted. [2](#0-1) 

### Proof of Concept

```
Setup:
  participants = [P0 (coordinator), P1, P2 (malicious)]
  All participants run ckd() with the same (app_id, app_pk, keygen_out)

Honest execution:
  P1 computes (norm_big_y_1, norm_big_c_1) correctly and sends to P0
  P2 computes (norm_big_y_2, norm_big_c_2) correctly and sends to P0
  P0 aggregates: Y = Y_0 + Y_1 + Y_2, C = C_0 + C_1 + C_2
  Result: C - y*app_pk = msk * H(pk, app_id)  ✓

Malicious execution (P2 is attacker):
  P1 computes (norm_big_y_1, norm_big_c_1) correctly and sends to P0
  P2 sends (ElementG1::identity(), ElementG1::generator())  ← arbitrary crafted values
  P0 aggregates: Y = Y_0 + Y_1 + 0, C = C_0 + C_1 + G
  Result: C - y*app_pk ≠ msk * H(pk, app_id)  ✗ (corrupted)
  ckd_output.unmask(app_sk) returns wrong key, undetected
```

The coordinator has no mechanism to distinguish the honest from the malicious execution. [5](#0-4)

async fn do_ckd_coordinator(
    mut chan: SharedChannel,
    participants: ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<CKDOutputOption, ProtocolError> {
    let (mut norm_big_y, mut norm_big_c) =
        compute_signature_share(&participants, me, key_pair, app_id, app_pk, rng)?;

    // Receive everyone's inputs and add them together
    let waitpoint = chan.next_waitpoint();

    for (_, participant_output) in
        recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
    {
        norm_big_y += participant_output.big_y();
        norm_big_c += participant_output.big_c();
    }
    let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
    Ok(Some(ckd_output))
