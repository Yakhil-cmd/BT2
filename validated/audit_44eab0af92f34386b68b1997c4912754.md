### Title
WalletConnect Offer Confirmation Understates Total Royalty Cost for Multi-NFT Offers — (File: `packages/gui/src/electron/commands/parseCommandDisplay.ts`)

### Summary
`formatAmountWithRoyalties` in `parseCommandDisplay.ts` incorrectly divides the fungible amount by the number of NFTs before computing each royalty. The "Total Amount with Royalties" shown in the WalletConnect signing confirmation dialog is therefore systematically understated whenever an offer involves more than one NFT with royalties, causing the user to approve a spend larger than what was displayed.

### Finding Description
`formatAmountWithRoyalties` (lines 363–367) splits the fungible amount equally across all NFTs before applying each royalty percentage:

```typescript
const splitAmount = amount / BigInt(royaltyPercentages.length);   // line 363
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
```

For N NFTs each royalty is computed on `amount / N` instead of the full `amount`. The rest of the GUI uses the `nft_calculate_royalties` RPC (called in `OfferBuilderProvider.tsx`) which sends the full fungible amount per royalty asset — the correct on-chain behaviour.

**Concrete example** (taken directly from the existing multi-NFT test, lines 251–335 of `parseCommandDisplay.test.ts`):

| | GUI (split) | Correct (full amount) |
|---|---|---|
| amount | 100 000 000 mojos | 100 000 000 mojos |
| splitAmount | 50 000 000 | — |
| royalty NFT1 (500 bp) | 2 500 000 | 5 000 000 |
| royalty NFT2 (10 bp) | 50 000 | 100 000 |
| **total royalty** | **2 550 000** | **5 100 000** |
| **displayed total** | **0.00010255 XCH** | **0.0001051 XCH** |

The confirmation dialog understates the total by **2 550 000 mojos** — exactly 50 % of the true royalty in this case. The understatement scales with the number of NFTs and their royalty percentages.

The `amountWithRoyalties` field produced here is rendered verbatim in the WalletConnect signing dialog (`Confirm.tsx`, lines 109–114) as "Total Amount with Royalties" — the primary cost signal the user relies on before clicking Approve.

### Impact Explanation
A user who receives a WalletConnect `take_offer` or `create_offer_for_ids` request for a multi-NFT offer sees a lower total cost than what the blockchain will actually deduct. They approve the transaction believing they are spending X, but the wallet spends Y > X. This is a direct instance of the High-impact class: "WalletConnect state that causes a user to approve the wrong amount." The gap is proportional to `(N-1)/N × Σ(royalty_i × amount)` and can be substantial for offers with several high-royalty NFTs.

### Likelihood Explanation
Any WalletConnect-paired dapp can issue `chia_wallet.take_offer` or `chia_wallet.create_offer_for_ids` with a crafted multi-NFT payload. No privileged access, leaked keys, or social engineering beyond the existing WalletConnect pairing is required. The trigger condition (≥ 2 NFTs with royalties in one offer) is a normal product feature, not an exotic edge case.

### Recommendation
Remove the per-NFT split and apply each royalty to the full fungible amount, matching the `nft_calculate_royalties` RPC behaviour:

```typescript
// packages/gui/src/electron/commands/parseCommandDisplay.ts
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (amount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
const totalAmount = amount + royaltyAmount;
```

### Proof of Concept
1. Pair a dapp to the user's wallet via WalletConnect.
2. Send `chia_wallet.take_offer` for an offer containing 2 NFTs (royalty_percentage 500 and 10) requesting 1 000 000 000 000 mojos (1 XCH).
3. The confirmation dialog renders `amountWithRoyalties` = **1.0255 XCH** (split calculation).
4. The blockchain deducts the correct royalties: `(1T × 500/10000) + (1T × 10/10000)` = 51 000 000 000 mojos → actual total **1.051 XCH**.
5. The user approved a spend of 1.051 XCH while the dialog showed 1.0255 XCH — a silent 0.0255 XCH overcharge. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
