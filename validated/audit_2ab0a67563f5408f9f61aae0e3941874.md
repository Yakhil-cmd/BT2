Audit Report

## Title
Missing Per-Canister Quota Allows Single Canister to Exhaust Subnet-Wide HTTP Outcall Pool - (`rs/execution_environment/src/execution_environment.rs`)

## Summary
The subnet-wide canister HTTP outcall in-flight pool (`canister_http_request_contexts`) is bounded by `MAX_CANISTER_HTTP_REQUESTS_IN_FLIGHT = 3000` but enforces no per-canister sub-limit. Any single canister with sufficient cycles can occupy all 3000 slots, causing every other canister on the same subnet to receive `CanisterRejectedMessage` for all HTTP outcall attempts for as long as the attack is sustained. The attack is repeatable every 60 seconds as timed-out slots free up.

## Finding Description
In `try_add_http_context_to_replicated_state` (`rs/execution_environment/src/execution_environment.rs`, lines 2148–2162), the sole admission guard is:

```rust
if state
    .metadata
    .subnet_call_context_manager
    .canister_http_request_contexts
    .len()
    >= self.config.max_canister_http_requests_in_flight
{
    return Err(UserError::new(
        ErrorCode::CanisterRejectedMessage,
        format!(
            "max number ({}) of http requests in-flight reached.",
            self.config.max_canister_http_requests_in_flight
        ),
    ));
}
```

`canister_http_request_contexts` is a flat `BTreeMap<CallbackId, CanisterHttpRequestContext>` (`rs/replicated_state/src/metadata_state/subnet_call_context_manager.rs`, line 219). There is no per-sender count, no per-canister quota field, and no fairness mechanism anywhere in the insertion path. The constant is set to 3000 (`rs/config/src/execution_environment.rs`, line 206). Requests time out after 60 seconds (`rs/types/types/src/canister_http.rs`, lines 78–79), after which the attacker re-fills the pool.

The existing test `http_request_bound_holds` (`rs/execution_environment/src/execution_environment/tests.rs`, lines 2319–2381) uses a single `caller_canister` to fill the entire pool to its configured maximum of 10, confirming that one sender can exhaust the full quota with no per-canister distinction enforced.

The contrast with the general inter-canister callback system is instructive: that system has both a `subnet_callback_soft_limit` and a `canister_guaranteed_callback_quota` (`rs/config/src/execution_environment.rs`, lines 266–275) to prevent exactly this class of starvation. No equivalent mechanism exists for HTTP outcalls.

## Impact Explanation
This is a **High** severity finding matching the allowed impact: *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."*

All canisters on the targeted subnet that invoke `ic00::HttpRequest` receive `CanisterRejectedMessage: "max number (3000) of http requests in-flight reached."` for the duration of the attack. Any protocol or application relying on canister HTTP outcalls — oracle price feeds, cross-chain bridges, HTTPS API integrations — is completely blocked. This is a sustained, renewable availability failure of the HTTP outcall feature for the entire subnet, exploiting a protocol-level resource exhaustion flaw rather than raw volumetric flooding.

## Likelihood Explanation
Any canister deployed on the target subnet can execute this attack without any privileged access, governance majority, or threshold corruption. The only cost is cycles for the non-refundable base HTTP request fee multiplied by 3000 slots, renewed every 60 seconds. The economic cost is bounded and predictable. For a motivated attacker targeting a high-value subnet running DeFi protocols dependent on price oracles via HTTP outcalls, the cost-to-impact ratio is highly favorable. No victim mistakes, phishing, DNS hijack, or external compromise is required.

## Recommendation
Introduce a per-canister limit on simultaneously in-flight HTTP outcall requests. Track the count of in-flight requests per `sender` canister ID within `canister_http_request_contexts` and reject new requests from a canister that has already reached its individual quota (e.g., `MAX_CANISTER_HTTP_REQUESTS_IN_FLIGHT / expected_active_canisters` or a fixed per-canister cap). This mirrors the existing `canister_guaranteed_callback_quota` pattern already used for general inter-canister callbacks (`rs/config/src/execution_environment.rs`, lines 266–275) and is the natural extension of that fairness mechanism to the HTTP outcall subsystem.

## Proof of Concept
1. Deploy a canister `attacker` on any application subnet with HTTP outcalls enabled.
2. In a loop, have `attacker` call `ic00::HttpRequest` with a URL pointing to a slow or non-existent endpoint, attaching the minimum required cycles.
3. After 3000 such calls are in flight, any other canister on the same subnet calling `ic00::HttpRequest` receives `CanisterRejectedMessage: "max number (3000) of http requests in-flight reached."`.
4. After 60 seconds the attacker's requests time out; the attacker immediately re-submits 3000 new requests, maintaining the DoS indefinitely.
5. To reproduce safely in a unit test: adapt `http_request_bound_holds` (`rs/execution_environment/src/execution_environment/tests.rs`, lines 2319–2381) to use two distinct canister IDs — fill the pool entirely from `canister_A`, then assert that `canister_B`'s request is rejected with `CanisterRejectedMessage`, demonstrating the absence of any per-canister fairness.
