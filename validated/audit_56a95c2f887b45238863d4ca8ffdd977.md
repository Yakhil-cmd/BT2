### Title
Missing Zero-Address Validation in `change_reward_address` and `enter_delegation_pool` Enables Permanent Freezing of Unclaimed Yield - (File: `src/pool/pool.cairo`, `src/staking/staking.cairo`)

---

### Summary

Both `change_reward_address` functions (in `staking.cairo` and `pool.cairo`) and `enter_delegation_pool` in `pool.cairo` accept a `reward_address: ContractAddress` parameter without validating it is non-zero. A staker or pool member can set their reward address to `0x0`, causing all subsequent `claim_rewards` calls to revert (because the underlying ERC20 panics on transfer to the zero address), permanently freezing their unclaimed yield unless they recover by calling `change_reward_address` again. In the staking contract, this also causes `unstake_action` to revert, temporarily freezing the staker's principal in the exit window.

---

### Finding Description

**Root cause — `staking.cairo` `change_reward_address` (line 517–540):**

The only guard present is a check that the address is not a registered token address. There is no zero-address check:

```cairo
fn change_reward_address(ref self: ContractState, reward_address: ContractAddress) {
    self.general_prerequisites();
    assert!(
        !self.does_token_exist(token_address: reward_address),
        "{}",
        GenericError::REWARD_ADDRESS_IS_TOKEN,
    );
    let staker_address = get_caller_address();
    let mut staker_info = self.internal_staker_info(:staker_address);
    staker_info.reward_address = reward_address;   // written with no zero check
    self.write_staker_info(:staker_address, :staker_info);
    ...
}

**Downstream effect — `send_rewards_to_staker` (staking, line 1614–1629):**

When `claim_rewards` or `unstake_action` is executed, the staking contract calls `checked_transfer(recipient: reward_address, ...)`. If `reward_address` is zero, the OpenZeppelin ERC20 panics, reverting the entire transaction:

```cairo

``` [4](#0-3) 

**Downstream effect — `pool.cairo` `claim_rewards` (line 335–377):**

``` [5](#0-4)

---

### Impact Explanation

**Permanent freezing of unclaimed yield (High):** A staker or pool member who sets their `reward_address` to `0x0` (accidentally or via a buggy front-end) will find that every subsequent `claim_rewards` call reverts. Rewards continue to accrue in `unclaimed_rewards_own` (staker) or the pool's cumulative trace (pool member) but can never be transferred out. If the user loses access to their signing key before calling `change_reward_address` to correct the mistake, the accumulated yield is permanently frozen in the contract.

**Temporary freezing of staked principal (High):** For stakers, `unstake_action` internally calls `send_rewards_to_staker` before returning the principal. With a zero `reward_address`, `unstake_action` always reverts, trapping the staker's principal in the exit window. The staker must first call `change_reward_address` with a valid address before `unstake_action` can succeed. During this window the principal is inaccessible. [7](#0-6) 

---

### Likelihood Explanation

Likelihood is **Low**. The affected user must be the staker or pool member themselves (the functions are caller-gated). The most realistic trigger is a front-end or integration bug that passes an uninitialized/zero address, or a user who copies a zero-padded value. The `change_reward_address` recovery path exists and is unrestricted (no `unstake_time` guard), so the freeze is self-recoverable in most cases — but not if the user loses key access after the bad write. [8](#0-7) 

---

### Recommendation

Add a non-zero assertion to all three entry points, mirroring the existing `GenericError::ZERO_ADDRESS` error that is already defined in the shared error module:

```cairo
// staking.cairo — change_reward_address
assert!(reward_address.is_non_zero(), "{}", GenericError::ZERO_ADDRESS);

// pool.cairo — change_reward_address
assert!(reward_address.is_non_zero(), "{}", GenericError::ZERO_ADDRESS);

// pool.cairo — enter_delegation_pool
assert!(reward_address.is_non_zero(), "{}", GenericError::ZERO_ADDRESS);
``` [9](#0-8) 

---

### Proof of Concept

1. Deploy the system normally.
2. As a registered staker, call `staking.change_reward_address(reward_address: 0x0)`. The call succeeds — no revert.
3. Advance epochs so rewards accrue.
4. Call `staking.claim_rewards(staker_address)` — the call reverts because `checked_transfer(recipient: 0x0, ...)` panics inside `send_rewards_to_staker`. Rewards remain frozen in `unclaimed_rewards_own`.
5. Call `staking.unstake_intent()` — succeeds.
6. After the exit window, call `staking.unstake_action(staker_address)` — reverts for the same reason, trapping the principal.
7. Repeat steps 4 and 6 indefinitely; both always revert until `change_reward_address` is called with a valid address.

The same sequence applies to pool members via `pool.change_reward_address(0x0)` or `pool.enter_delegation_pool(reward_address: 0x0, amount: X)`. [10](#0-9) [11](#0-10)
