### Title
Unverified Participant Shares in CKD Protocol Enable Replay Attack and CKD Output Corruption — (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

The CKD coordinator in `do_ckd_coordinator` aggregates `(norm_big_y, norm_big_c)` tuples from participants with no proof of correctness and no binding to the current session (`app_id`, `app_pk`). A malicious participant can replay shares from a prior CKD session or send arbitrary values; the coordinator accepts them unconditionally, corrupting the derived confidential key that honest parties receive.

---

### Finding Description

`compute_signature_share` (lines 148–182) produces two values per participant:

- `norm_big_y = λᵢ · y · G` — ElGamal randomness
- `norm_big_c = λᵢ · (xᵢ · H(pk ‖ app_id) + y · app_pk)` — ElGamal ciphertext [1](#0-0) 

These values are sent privately to the coordinator. In `do_ckd_coordinator`, the coordinator receives them and blindly sums them:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
``` [2](#0-1) 

There is **no proof of knowledge** attached to the shares, and **no session binding** to the current `app_id` or `app_pk`. Specifically, the coordinator never verifies:

1. That the sender knows `y` such that `norm_big_y = λᵢ · y · G`.
2. That the same `y` was used in both `norm_big_y` and `norm_big_c`.
3. That the sender's actual secret share `xᵢ` was used in `norm_big_c`.
4. That the shares were freshly computed for the current `(app_id, app_pk)` pair rather than replayed from a prior session.

This is the direct analog of the zkSync `proveBlocks` bug: just as that contract verified a proof without checking it was bound to the committed blocks, the CKD coordinator verifies nothing about the shares it aggregates — the "proof" step is entirely absent rather than merely commented out. [3](#0-2) 

---

### Impact Explanation

A malicious participant can corrupt the aggregated `CKDOutput`. The requester then calls `unmask(app_sk)` on the corrupted output:

```
corrupted_big_c − app_sk · corrupted_big_y  ≠  msk · H(pk ‖ app_id)
```

Honest parties accept this corrupted output as the protocol result, deriving an incorrect or unusable confidential key. This matches the allowed impact: **High — Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.** [4](#0-3) 

---

### Likelihood Explanation

Any single malicious participant among the threshold set can trigger this. No special privilege beyond being a protocol participant is required. The attack is trivially executable: the participant simply sends stale shares from a previous session or arbitrary group elements. The coordinator has no mechanism to detect or reject them. [5](#0-4) 

---

### Recommendation

1. **Add a Chaum–Pedersen / DLogEq proof** alongside each share, proving that the same scalar `y` was used in both `norm_big_y = λᵢ · y · G` and the `y · app_pk` term inside `norm_big_c`, and that `xᵢ` is consistent with the public key commitment.
2. **Bind shares to the session** by including `app_id`, `app_pk`, and a fresh session nonce in the proof transcript, preventing cross-session replay.
3. The coordinator must verify each participant's proof before aggregating, rejecting any participant whose proof fails.

The library already has the necessary primitives: `DLogEq` proofs in `src/crypto/proofs/dlogeq.rs` and a Fiat-Shamir transcript in `src/crypto/proofs/strobe_transcript.rs` that supports domain separation and session binding. [6](#0-5) [7](#0-6) 

---

### Proof of Concept

```
Setup:
  Participants: {P_honest_1, P_honest_2, P_malicious}
  Threshold: 2-of-3

Step 1 — Session A (app_id = "app_A"):
  P_malicious runs compute_signature_share for app_A.
  It records the output: (norm_big_y_A, norm_big_c_A).

Step 2 — Session B (app_id = "app_B"):
  P_malicious, instead of calling compute_signature_share for app_B,
  sends the previously recorded (norm_big_y_A, norm_big_c_A) to the coordinator.

Step 3 — Coordinator aggregation (do_ckd_coordinator):
  coordinator receives (norm_big_y_A, norm_big_c_A) from P_malicious.
  No verification is performed.
  Coordinator sums: total_big_y += norm_big_y_A; total_big_c += norm_big_c_A.

Step 4 — Output:
  CKDOutput { big_y: total_big_y, big_c: total_big_c } is returned.
  unmask(app_sk) yields:
    total_big_c − app_sk · total_big_y
    ≠ msk · H(pk ‖ app_B)
  Honest parties accept this corrupted key as the CKD result for app_B.
``` [8](#0-7)

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}

    }
