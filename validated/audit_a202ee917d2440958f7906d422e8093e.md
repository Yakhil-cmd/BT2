### Title
Access Node Subscription Penalty Bypassed Due to Wrong Condition in `computeAppSpecificScore` - (File: `network/p2p/scoring/registry.go`)

### Summary
In `computeAppSpecificScore`, the subscription penalty check is gated by `if stakingScore >= 0`. For access nodes, `stakingScore` returns `MinAppSpecificPenalty` (-1), which is negative. This causes the subscription penalty check to be completely skipped for access nodes. A malicious staked access node can therefore subscribe to unauthorized topics without receiving the `InvalidSubscriptionPenalty` (-100) that would push its score below the `GraylistThreshold` (-99) and cause it to be graylisted.

### Finding Description
`computeAppSpecificScore` in `GossipSubAppSpecificScoreRegistry` computes the application-specific score of a peer in five steps. Step 3 — the subscription penalty — is gated by the condition `if stakingScore >= 0`:

```go
// (2) staking score
stakingScore, flowId, role := r.stakingScore(pid)
if stakingScore < 0 {
    appSpecificScore += stakingScore
}

if stakingScore >= 0 {
    // (3) subscription penalty
    subscriptionPenalty := r.subscriptionPenalty(pid, flowId, role)
    if subscriptionPenalty < 0 {
        appSpecificScore += subscriptionPenalty
    }
}
```

`stakingScore` for access nodes is set to `MinAppSpecificPenalty` (-1) — a negative value — to push them to the edges of the network:

```go
if flowId.Role == flow.RoleAccess {
    return r.appSpecificScoreParams.MinAppSpecificPenalty, flowId.NodeID, flowId.Role
}
```

Because `MinAppSpecificPenalty = -1 < 0`, the condition `stakingScore >= 0` is **false** for every access node. The subscription penalty check (`subscriptionPenalty`) is therefore **never reached** for access nodes, regardless of what topics they subscribe to.

The `subscriptionPenalty` function calls `CheckSubscribedToAllowedTopics(pid, role)`, which returns an error for any topic not allowed for the peer's role. The penalty for a violation is `InvalidSubscriptionPenalty = -100`. This penalty is designed to push the peer's total score below `GraylistThreshold = -99`, causing it to be graylisted (all incoming/outgoing GossipSub RPCs rejected). For access nodes, this entire enforcement path is dead code.

The code comment at step 3 reads: *"Note: subscription penalty can be considered only for staked peers, for non-staked peers, we cannot determine the role of the peer."* Access nodes **are** staked peers with a known role — the comment's intent does not justify skipping the check for them. The condition conflates "unknown identity" (score = -100) with "access node" (score = -1), treating both as ineligible for subscription enforcement.

### Impact Explanation
A staked access node that subscribes to topics outside its authorized set (e.g., consensus-committee, execution-receipts, or verification-specific channels) receives a final app-specific score of `-1` instead of `-1 + (-100) = -101`. Since `-1 > GraylistThreshold (-99)`, the node is **not graylisted**. It remains connected to the GossipSub mesh and continues to receive messages on topics it is not authorized to observe. The subscription enforcement mechanism — which the `SubscriptionValidator` and `InvalidSubscriptionPenalty` are explicitly designed to enforce — is entirely bypassed for the access-node role. The test suite confirms that every role, including access nodes, has unauthorized topics (`TestSubscriptionValidator_InvalidSubscriptions`).

### Likelihood Explanation
Any operator of a staked access node can trigger this by simply subscribing to additional topics beyond those allowed for `flow.RoleAccess`. No key compromise, quorum manipulation, or privileged node type is required. Access nodes are the most numerous and least-trusted node type in the Flow network, making this a realistic attack surface.

### Recommendation
Replace the condition `if stakingScore >= 0` with a check that distinguishes between "unknown identity" (where the role is genuinely unknown) and "access node" (where the role is known but the staking score is negative by design). For example:

```go
_, hasValidIdentity := r.idProvider.ByPeerID(pid)
if hasValidIdentity {
    subscriptionPenalty := r.subscriptionPenalty(pid, flowId, role)
    if subscriptionPenalty < 0 {
        appSpecificScore += subscriptionPenalty
    }
}
```

Alternatively, return a sentinel value from `stakingScore` for unknown peers that is distinct from `MinAppSpecificPenalty`, and gate the subscription check on the absence of that sentinel.

### Proof of Concept

**Step 1 — Root cause:** `stakingScore` for `flow.RoleAccess` returns `MinAppSpecificPenalty = -1`. [1](#0-0) 

**Step 2 — Gate condition:** The subscription penalty block is entered only when `stakingScore >= 0`, which is never true for access nodes. [2](#0-1) 

**Step 3 — Penalty that is never applied:** `subscriptionPenalty` returns `InvalidSubscriptionPenalty = -100` for any unauthorized topic subscription. [3](#0-2) 

**Step 4 — Penalty value and graylist threshold:** `InvalidSubscriptionPenalty = -100` equals `MaxAppSpecificPenalty`, which is set one point below `GraylistThreshold = -99`, so a legitimate penalty would graylist the peer. [4](#0-3) 

**Step 5 — Confirmed unauthorized topics exist for access nodes:** The test `TestSubscriptionValidator_InvalidSubscriptions` iterates over all roles including `flow.RoleAccess` and asserts that unauthorized topics exist for each. [5](#0-4) 

**Concrete scenario:** An attacker runs a staked access node and subscribes it to `channels.SyncCommittee` or `channels.ConsensusCommittee` (topics restricted to consensus nodes). Every peer that evaluates this access node's score via `computeAppSpecificScore` skips the subscription check, assigns score `-1`, and does not graylist the node. The access node remains in the GossipSub mesh for those topics and receives all messages propagated on them, bypassing the role-based subscription enforcement entirely.

```go
func (r *GossipSubAppSpecificScoreRegistry) subscriptionPenalty(pid peer.ID, flowId flow.Identifier, role flow.Role) float64 {
	// checks if peer has any subscription violation.
	if err := r.validator.CheckSubscribedToAllowedTopics(pid, role); err != nil {
		r.logger.Warn().
			Err(err).
			Str("peer_id", p2plogging.PeerId(pid)).
			Hex("flow_id", logging.ID(flowId)).
			Bool(logging.KeySuspicious, true).
			Msg("invalid subscription detected, penalizing peer")
		return r.appSpecificScoreParams.InvalidSubscriptionPenalty
	}

	return 0
}
```

```go
func TestSubscriptionValidator_InvalidSubscriptions(t *testing.T) {
	sp := mockp2p.NewSubscriptionProvider(t)
	sv := scoring.NewSubscriptionValidator(unittest.Logger(), sp)

	for _, role := range flow.Roles() {
		peerId := unittest.PeerIdFixture(t)
		unauthorizedChannels := channels.Channels(). // all channels
								ExcludeChannels(channels.ChannelsByRole(role)). // excluding the channels for the role
								ExcludePattern(regexp.MustCompile("^(test).*")) // excluding the test channels.
		sporkID := unittest.IdentifierFixture()
		unauthorizedTopics := make([]string, 0, len(unauthorizedChannels))
		for _, channel := range unauthorizedChannels {
			unauthorizedTopics = append(unauthorizedTopics, channels.TopicFromChannel(channel, sporkID).String())
		}

		// peer should NOT pass subscription validator as it has subscribed to any subset of its unauthorized topics,
		// regardless of the role.
		for i := range unauthorizedTopics {
			sp.On("GetSubscribedTopics", peerId).Return(unauthorizedTopics[:i+1])
			err := sv.CheckSubscribedToAllowedTopics(peerId, role)
			require.Error(t, err, role)
			require.True(t, p2p.IsInvalidSubscriptionError(err), role)
		}
	}
}
```
