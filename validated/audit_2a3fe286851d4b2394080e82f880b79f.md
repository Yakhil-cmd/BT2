### Title
`skip_stateful_validations` ignores `max_nonce_for_validation_skip` config, unconditionally skipping blockifier `__validate__` for invoke nonce=1 regardless of operator configuration — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function hardcodes the nonce=1 skip condition and never consults `StatefulTransactionValidatorConfig::max_nonce_for_validation_skip`. An operator who sets `max_nonce_for_validation_skip = Nonce(Felt::ZERO)` to disable the UX skip entirely has no effect on the gateway path. Any invoke transaction with nonce=1 submitted while account_nonce=0 and a deploy_account is in the mempool will bypass blockifier `__validate__` (signature and account validation) unconditionally.

### Finding Description

`StatefulTransactionValidatorConfig` declares `max_nonce_for_validation_skip: Nonce` at line 283 of `crates/apollo_gateway_config/src/config.rs`: [1](#0-0) 

The field is documented as "Maximum nonce for which the validation is skipped." The default is `Nonce(Felt::ONE)`. [2](#0-1) 

However, `skip_stateful_validations` — the function that decides whether to skip blockifier `__validate__` — takes no config parameter and hardcodes the nonce=1 check: [3](#0-2) 

The call site in `run_pre_validation_checks` passes no config to it: [4](#0-3) 

A grep of all `crates/apollo_gateway/src/*.rs` files returns **zero matches** for `max_nonce_for_validation_skip` — the field is entirely dead in the gateway path.

By contrast, the Python/native-blockifier path (`crates/native_blockifier/src/py_validator.rs`) correctly stores and applies `max_nonce_for_validation_skip` in `should_run_stateful_validations`: [5](#0-4) 

The two paths are inconsistent. The Rust gateway path ignores the config entirely.

### Impact Explanation

An attacker who has submitted a valid `deploy_account` transaction (nonce=0) for a fresh account can immediately submit a second invoke transaction (nonce=1) with an **invalid signature**. Because `account_tx_in_pool_or_recent_block` returns `true` (the deploy_account is in the mempool), `skip_stateful_validations` returns `true` and `run_validate_entry_point` is called with `skip_validate=true`, bypassing blockifier `__validate__`. The invoke with the invalid signature is admitted to the mempool.

This matches the **High** impact scope: *Mempool/gateway/RPC admission accepts invalid transactions before sequencing.*

An operator who explicitly sets `max_nonce_for_validation_skip = 0x0` to disable this UX skip (e.g., for a security-sensitive deployment) gets no protection — the config value is silently ignored.

### Likelihood Explanation

The trigger conditions are fully attacker-controlled from unprivileged public inputs:
1. Submit a valid `deploy_account` (nonce=0) — passes all validation normally.
2. Submit an invoke (nonce=1) with an invalid/arbitrary signature — `skip_stateful_validations` returns `true`, blockifier `__validate__` is skipped, transaction is admitted.

No operator/admin privilege is required. The only prerequisite is a valid deploy_account in the mempool, which the attacker controls.

### Recommendation

Pass `self.config.max_nonce_for_validation_skip` into `skip_stateful_validations` and gate the skip on `tx.nonce() <= max_nonce_for_validation_skip` (mirroring the Python path logic at `py_validator.rs` lines 113–118). When `max_nonce_for_validation_skip = Nonce(Felt::ZERO)`, the condition `Nonce(Felt::ONE) <= Nonce(Felt::ZERO)` is false and the skip never fires.

### Proof of Concept

        ..Default::default()
    },
    ...
};

// Invoke with nonce=1, account_nonce=0, mempool has deploy_account → should NOT skip
let skip = stateful_validator
    .run_pre_validation_checks(
        &executable_invoke_tx(invoke_tx_args!(nonce: nonce!(1))),
        nonce!(0),
        mempool_client_returning_true,
    )
    .await
    .unwrap();

// Actual:   skip == true  (validation bypassed — config ignored)
