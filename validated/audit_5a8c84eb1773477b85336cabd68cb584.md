### Title
Loss of Treasury Funds When `coinbase` Is `B160::ZERO` in L1→L2 Transaction Fee Distribution — (`basic_bootloader/src/bootloader/transaction_flow/zk/process_l1_transaction.rs`)

---

### Summary

In `process_l1_transaction`, the operator fee (`pay_to_operator`) is unconditionally transferred from the treasury to `system.get_coinbase()`. No validation is performed to ensure `coinbase != B160::ZERO`. When the block's coinbase is the zero address — which is the **default value** in both `BlockMetadataFromOracle` and `BlockContext` — every L1→L2 transaction permanently burns operator fees out of the treasury into `address(0)`.

---

### Finding Description

In `process_l1_transaction.rs`, after computing `pay_to_operator`, the code calls:

```rust
let coinbase = system.get_coinbase();
mint_base_token::<S, Config>(
    system, system_functions, memories.reborrow(),
    &pay_to_operator,
    &coinbase,          // ← no zero-address guard
    l1_chain_id,
    &mut inf_resources,
    tracer, validator,
)?;
```

`mint_base_token` calls `transfer_from_treasury`, which subtracts `pay_to_operator` from the treasury balance and adds it to `to` (coinbase):

```rust
pub fn transfer_from_treasury<'a, S: EthereumLikeTypes + 'a>(
    system: &mut System<S>,
    nominal_token_value: &U256,
    to: &B160,          // ← no zero-address guard
    ...
```

Neither `process_l1_transaction` nor `transfer_from_treasury` checks `to != B160::ZERO`.

The default `BlockMetadataFromOracle::new_for_test()` explicitly sets `coinbase: B160::ZERO`, and `BlockContext::default()` sets `coinbase: B160::default()` which is also `B160::ZERO`. There is no enforcement anywhere in the oracle deserialization or block-metadata pipeline that prevents a zero coinbase from reaching fee distribution.

The analog to the external report is exact: just as `feeCollector = address(0)` causes `claimFees()` to burn the fee collector's share, `coinbase = B160::ZERO` causes every L1 transaction's operator fee to be burned from the treasury.

---

### Impact Explanation

For every L1→L2 priority transaction processed while `coinbase = B160::ZERO`:

- `pay_to_operator = gas_used × gas_price` is deducted from the treasury (`BASE_TOKEN_HOLDER_ADDRESS`)
- The same amount is credited to `address(0)` — permanently inaccessible
- The treasury balance is irreversibly reduced; no operator receives compensation

Because the treasury is a finite pre-funded pool used for all L1 deposit distributions, repeated loss drains it, eventually causing `TreasuryTransferFailed` errors that halt L1 transaction processing entirely.

**Impact: High** — permanent, irreversible loss of protocol treasury funds.

---

### Likelihood Explanation

- The default `BlockContext` and `BlockMetadataFromOracle::new_for_test()` both initialize `coinbase` to `B160::ZERO`.
- A sequencer that omits or forgets to set a non-zero coinbase will silently burn all L1 operator fees.
- No on-chain or in-protocol guard prevents a zero coinbase from being submitted via the oracle.
- Any unprivileged user submitting an L1 transaction with `gas_price > 0` triggers the loss.

**Likelihood: Low** — requires the operator to use a zero coinbase (default or misconfiguration), but the default being zero makes this a realistic operational risk.

---

### Recommendation

Add a zero-address guard before transferring operator fees in `process_l1_transaction.rs`:

```rust
let coinbase = system.get_coinbase();
if coinbase != B160::ZERO && !pay_to_operator.is_zero() {
    mint_base_token::<S, Config>(
        system, system_functions, memories.reborrow(),
        &pay_to_operator, &coinbase, l1_chain_id,
        &mut inf_resources, tracer, validator,
    )?;
}
```

Alternatively, validate `coinbase != B160::ZERO` at block-metadata ingestion time (oracle deserialization) and reject blocks with a zero coinbase. The same guard should be applied in `transfer_from_treasury` as a defense-in-depth measure.

---

### Proof of Concept

The following trace demonstrates the loss path:

1. Block is produced with `coinbase = B160::ZERO` (the default `BlockContext`).
2. An unprivileged user submits an L1→L2 transaction with `gas_price = 1000`, `gas_limit = 100_000`.
3. `process_l1_transaction` computes `pay_to_operator = gas_used × 1000 > 0`.
4. `mint_base_token(..., &pay_to_operator, &B160::ZERO, ...)` is called.
5. `transfer_from_treasury` subtracts `pay_to_operator` from `BASE_TOKEN_HOLDER_ADDRESS` and adds it to `B160::ZERO`.
6. Treasury balance decreases; `address(0)` balance increases by the same amount — funds are permanently inaccessible.

This is confirmed by the existing regression test `test_treasury_based_token_distribution_regression` (line 1843), which explicitly uses `refund_recipient = address(0)` and asserts that the treasury decreases by the full amount including the refund sent to `address(0)` — demonstrating the system performs no zero-address filtering on any of its treasury transfer destinations.

```rust
pub fn transfer_from_treasury<'a, S: EthereumLikeTypes + 'a>(
    system: &mut System<S>,
    nominal_token_value: &U256,
    to: &B160,
    resources: &mut S::Resources,
    fee_payment_in_simulation: bool,
) -> Result<(), BootloaderSubsystemError>
where
    S::IO: IOSubsystemExt,
{
    system_log!(
        system,
        "Transferring {nominal_token_value:?} tokens from treasury to {to:?}\n"
    );

    let treasury_address = &system_hooks::addresses_constants::BASE_TOKEN_HOLDER_ADDRESS;

    let _ = system
        .io
        .update_account_nominal_token_balance(
            zk_ee::execution_environment_type::ExecutionEnvironmentType::EVM,
            resources,
            treasury_address,
            nominal_token_value,
            true, // true = subtract from balance
            fee_payment_in_simulation,
        )
        .map_err(|e| -> BootloaderSubsystemError {
            match e {
                SubsystemError::LeafUsage(balance_error) => {
                    system_log!(system, "Treasury transfer failed: {balance_error:?}");
                    interface_error!(BootloaderInterfaceError::TreasuryTransferFailed)
                }
                _ => wrap_error!(e),
            }
        })?;

    let _ = system
        .io
        .update_account_nominal_token_balance(
            zk_ee::execution_environment_type::ExecutionEnvironmentType::EVM,
            resources,
            to,
            nominal_token_value,
            false, // false = add to balance
            fee_payment_in_simulation,
        )
        .map_err(|e| -> BootloaderSubsystemError {
            match e {
                SubsystemError::LeafUsage(balance_error) => {
                    system_log!(system, "Error while minting: {balance_error:?}");
                    interface_error!(BootloaderInterfaceError::MintingBalanceOverflow)
                }
                _ => wrap_error!(e),
            }
        })?;
```
