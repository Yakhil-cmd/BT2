### Title
`PresignOutput` Derives `Clone` and `rerandomize_presign` Takes `&PresignOutput` by Reference, Providing No Type-System Enforcement of the Critical Single-Use Requirement — (`File: src/ecdsa/ot_based_ecdsa/mod.rs`, `src/ecdsa/robust_ecdsa/mod.rs`)

---

### Summary

Both the OT-based ECDSA and Robust ECDSA schemes document that presignatures are **one-time-use** and must be destroyed after consumption. However, the library's type system provides no enforcement of this requirement: `PresignOutput` and `RerandomizedPresignOutput` both derive `Clone`, and `rerandomize_presign` accepts `&PresignOutput` by shared reference rather than consuming the value. A malicious coordinator or any caller holding a presignature can silently rerandomize and sign with the same presignature under different `(h, ε)` pairs, producing signatures with multiplicatively related nonces from which the aggregate secret key can be extracted.

---

### Finding Description

The library explicitly documents the single-use requirement in multiple places:

- `docs/ecdsa/ot_based_ecdsa/orchestration.md` states: *"It's **critical** that the output is then destroyed, so that no other group of parties attempts to re-use that output for another phase."*
- `docs/ecdsa/robust_ecdsa/signing.md` states: *"**Never reuse a presignature**, even across failed, aborted, or partially completed signing sessions."*
- Both `presign.rs` docstrings state: *"it's crucial that a presignature is never reused."*

Despite this, the type definitions make reuse trivially possible:

**OT-based ECDSA** (`src/ecdsa/ot_based_ecdsa/mod.rs`):

```rust
#[derive(Debug, Clone, Serialize, Deserialize, Eq, PartialEq, ZeroizeOnDrop)]
pub struct PresignOutput { ... }

#[derive(Debug, Clone, Serialize, Deserialize, ZeroizeOnDrop)]
pub struct RerandomizedPresignOutput { ... }

pub fn rerandomize_presign(
    presignature: &PresignOutput,          // ← shared reference, NOT consumed
    args: &RerandomizationArguments,
) -> Result<Self, ProtocolError> { ... }
```

**Robust ECDSA** (`src/ecdsa/robust_ecdsa/mod.rs`):

```rust
#[derive(Debug, Clone, Serialize, Deserialize, PartialEq, Eq, ZeroizeOnDrop)]
pub struct PresignOutput { ... }

#[derive(Debug, Clone, Serialize, Deserialize, ZeroizeOnDrop)]
pub struct RerandomizedPresignOutput { ... }

pub fn rerandomize_presign(
    presignature: &PresignOutput,          // ← shared reference, NOT consumed
    args: &RerandomizationArguments,
) -> Result<Self, ProtocolError> { ... }
```

The `Clone` derive means any caller can duplicate a `PresignOutput` or `RerandomizedPresignOutput` before passing it to a consuming function. The `&PresignOutput` reference in `rerandomize_presign` means the original is never consumed at all — the same `PresignOutput` can be passed to `rerandomize_presign` an unlimited number of times with different `RerandomizationArguments` (different message hashes `h` and tweaks `ε`).

This is the direct analog to the `selfdestruct()` issue: the "destroy after use" mechanism is documented as critical but is never enforced by the code. The Escher contracts relied on `selfdestruct()` to prevent post-cancel purchases; this library relies on caller discipline to prevent presignature reuse — and provides no language-level mechanism to enforce it.

---

### Impact Explanation

The security consequence is explicitly stated in `docs/ecdsa/robust_ecdsa/signing.md`:

> *"If different subsets of size at least 2t+1 sign different (h, ε) values using shares derived from the same presignature, the resulting signatures use multiplicatively related nonces and the secret key can be recovered using standard ECDSA nonce-reuse attacks."*
>
> *"a novel split-view attack exists that can extract the secret key using as few as 2t+2 presigning participants, with as few as two signing sessions."*

Two signatures `(R, s₁)` and `(R, s₂)` produced from the same presignature (same nonce `k`) under different message hashes `h₁ ≠ h₂` satisfy:

```
s₁ = k·h₁ + Rₓ·x    (mod q)
s₂ = k·h₂ + Rₓ·x    (mod q)
```

Subtracting: `s₁ - s₂ = k·(h₁ - h₂)`, so `k = (s₁ - s₂)·(h₁ - h₂)⁻¹`, and then `x = (s₁ - k·h₁)·Rₓ⁻¹`. The aggregate secret key is fully recovered.

**Impact classification: Critical** — Extraction of the aggregate secret signing key.

---

### Likelihood Explanation

The attacker-controlled entry path requires only that a coordinator (or any party that stores presignatures) calls `rerandomize_presign` twice on the same `PresignOutput` with two different `RerandomizationArguments`. Because `rerandomize_presign` takes `&PresignOutput`, this requires zero cloning — the same reference is passed twice. The `Clone` derive additionally allows any storage layer (database, cache, retry logic) to inadvertently duplicate a presignature and feed it into a second signing session.

A malicious coordinator is an explicitly documented threat model for this library (the coordinator role is distinct from participants in both signing protocols). A malicious coordinator can:

1. Run presigning with honest participants to obtain `PresignOutput`.
2. Call `rerandomize_presign(&presign_output, &args_for_msg1)` → `rerandomized1`.
3. Call `rerandomize_presign(&presign_output, &args_for_msg2)` → `rerandomized2`.
4. Run two signing sessions, collecting `s₁` and `s₂` from honest participants.
5. Recover the secret key offline.

No participant can detect this because each signing session appears individually valid.

---

### Recommendation

Enforce single-use at the type level by consuming `PresignOutput` in `rerandomize_presign`:

```rust
pub fn rerandomize_presign(
    presignature: PresignOutput,           // ← consume by value
    args: &RerandomizationArguments,
) -> Result<Self, ProtocolError> { ... }
```

Additionally, remove the `Clone` derive from both `PresignOutput` and `RerandomizedPresignOutput` in both schemes. `ZeroizeOnDrop` is already present and correct; removing `Clone` makes the Rust ownership system enforce the one-time-use invariant that the documentation requires but currently cannot guarantee.

### Proof of Concept

```rust
// Attacker (coordinator) holds a single PresignOutput from an honest presigning run.
let presign_output: PresignOutput = /* result of honest presigning */;

// rerandomize_presign takes &PresignOutput — original is NOT consumed.
let rerandomized1 = RerandomizedPresignOutput::rerandomize_presign(
    &presign_output,
    &args_for_message_hash_h1,
).unwrap();

// Same presignature, different message hash — trivially possible.
let rerandomized2 = RerandomizedPresignOutput::rerandomize_presign(
    &presign_output,
    &args_for_message_hash_h2,
).unwrap();

// Run two signing sessions → two signatures (R, s1) and (R, s2) with the same R.
// k = (s1 - s2) * (h1 - h2)^{-1}  (mod q)
// x = (s1 - k*h1) * Rx^{-1}        (mod q)
// Secret key x is fully recovered.
