### Title
Gateway Stateful Validator Skips `__validate__` for Invoke Transactions Based on Overly Broad Mempool Presence Check — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

`skip_stateful_validations` uses `account_tx_in_pool_or_recent_block` as a proxy to determine whether a `deploy_account` transaction exists in the mempool before skipping the `__validate__` entry point for a nonce-1 invoke. The check is too broad: it returns `true` for **any** transaction from the address, not specifically a `deploy_account`. An unprivileged attacker can exploit this to admit an invoke transaction with an invalid signature to the mempool by first submitting any valid transaction from the same fresh address.

---

### Finding Description

`skip_stateful_validations` is the sequencer's analog of the donation-attack root cause: it relies on a manipulatable external value (`account_tx_in_pool_or_recent_block`) instead of an authoritative internal check (verifying the specific transaction type), and the resulting wrong admission decision is the invariant break.

**Vulnerable function — `skip_stateful_validations`:** [1](#0-0) 

The function fires when three conditions hold simultaneously:

1. The transaction is an `Invoke`.
2. `tx.nonce() == Nonce(Felt::ONE)`.
3. `account_nonce == Nonce(Felt::ZERO)` (account not yet deployed in committed state).

When all three hold it calls:

```rust
return mempool_client
    .account_tx_in_pool_or_recent_block(tx.sender_address())
    .await
    ...
``` [2](#0-1) 

`account_tx_in_pool_or_recent_block` is implemented as:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [3](#0-2) 

`contains_account` in `MempoolState` returns `true` if the address appears in `staged` or `committed` — populated by **any** transaction type, not only `deploy_account`: [4](#0-3) 

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` sets `validate = false`:

```rust
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [5](#0-4) 

This means `blockifier_validator.validate(account_tx)` is called with `validate = false`, so the account's `__validate__` entry point — the only place the signature is verified — is **never executed** at gateway admission time. [6](#0-5) 

**Attack steps:**

1. Attacker generates a fresh address `A` (never deployed, `account_nonce = 0` in state).
2. Attacker submits a valid `deploy_account` for `A` → passes all gateway checks, enters `tx_pool`. Now `tx_pool.contains_account(A) = true`.
3. Attacker submits an `Invoke` with `nonce = 1` from `A` carrying an **invalid signature** (e.g., all-zero felts).
4. Gateway stateful path:
   - `validate_nonce`: `0 ≤ 1 ≤ 200` → passes.
   - `validate_by_mempool` → `mempool.validate_tx` checks duplicate hash and nonce-too-old only, no signature check.
   - `skip_stateful_validations`: `nonce == 1`, `account_nonce == 0`, `account_tx_in_pool_or_recent_block(A) == true` → returns `true`.
5. `run_validate_entry_point` is called with `skip_validate = true` → `__validate__` is **not called**.
6. Invalid invoke is admitted to the mempool without any signature verification.

The code comment itself acknowledges the assumption that is violated: *"It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."* The check does not enforce this — it accepts any transaction type.

The `max_nonce_for_validation_skip` config (default `Nonce(Felt::ONE)`) limits the skip to nonce-1 invokes, but the attacker only needs one invalid invoke per `deploy_account` submission. [7](#0-6) 

---

### Impact Explanation

An unprivileged attacker can admit invoke transactions with invalid (or forged) signatures to the mempool. The gateway's admission invariant — that every accepted transaction carries a signature verified by the account's `__validate__` entry point — is broken. The invalid transaction will fail at blockifier execution time (when `__validate__` is called with `validate = true` by the batcher), but it was admitted to the mempool and can be included in a block proposal, wasting block space and potentially enabling targeted griefing of specific accounts.

This matches the allowed impact: **"High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."**

---

### Likelihood Explanation

The preconditions are trivially satisfiable by any unprivileged user:

- Generate a fresh key pair → fresh address `A`.
- Submit a valid `deploy_account` (no special privilege required).
- Submit an invoke with nonce=1 and an all-zero signature.

No privileged access, no leaked secrets, no special network position is required. The only constraint is the `max_nonce_for_validation_skip` default of `Nonce(Felt::ONE)`, which limits the bypass to one invalid invoke per `deploy_account`.

---

### Recommendation

Replace the overly broad `account_tx_in_pool_or_recent_block` check with a check that specifically verifies the presence of a `deploy_account` transaction for the address. The mempool should expose a dedicated method such as `deploy_account_in_pool(address) -> bool` that inspects transaction types, not just address presence. Alternatively, the gateway should filter the pool result to confirm the pooled transaction is of type `DeployAccount` before granting the skip.

---

### Proof of Concept

```
// Precondition: fresh address A, account_nonce(A) = 0 in state.

// Step 1: submit valid deploy_account for A.
// → tx_pool.contains_account(A) = true

// Step 2: submit invoke(nonce=1, sender=A, signature=[0x0, 0x0]).
// Gateway path:
//   validate_nonce(nonce=1, account_nonce=0) → OK (0 ≤ 1 ≤ 200)
//   validate_by_mempool → OK (no duplicate, nonce not too old)
//   skip_stateful_validations:
//     tx.nonce() == 1 ✓
//     account_nonce == 0 ✓
//     account_tx_in_pool_or_recent_block(A) == true ✓  ← deploy_account in pool
//     → returns true (skip __validate__)
//   run_validate_entry_point(skip_validate=true):
//     execution_flags.validate = false
//     __validate__ NOT called
//     → Ok(())
// → Invalid invoke admitted to mempool.

// Step 3 (batcher execution):
//   deploy_account executes → A deployed, nonce = 1
//   invoke executes → __validate__ called → signature invalid → tx reverted
```

The root cause is in `skip_stateful_validations` at: [8](#0-7) 

using `account_tx_in_pool_or_recent_block` (manipulatable by submitting any transaction) instead of a type-specific `deploy_account_in_pool` check — the direct sequencer analog of the external report's use of `balanceOf` (manipulatable by direct token transfer) instead of internal accounting.

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
