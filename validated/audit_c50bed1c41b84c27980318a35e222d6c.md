### Title
Pool Admin Can Set Uncapped Per-Bin Additional Fees, Bypassing the Fee Cap System - (File: metric-core/contracts/MetricOmmPoolFactory.sol)

### Summary

The factory's `setPoolBinAdditionalFees` function forwards `addFeeBuyE6` and `addFeeSellE6` directly to the pool with no cap validation, while every other admin fee setter enforces `maxAdminSpreadFeeE6` / `maxAdminNotionalFeeE8`. A pool admin can set per-bin additional fees to `uint16.max` (65 535 in E6 = 6.55%) on any bin, silently charging traders more than the documented hard cap allows.

### Finding Description

The factory enforces fee caps on the global admin spread and notional components:

```solidity
// setPoolAdminFees — caps enforced
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (newAdminNotionalFeeE8 > maxAdminNotionalFeeE8) revert AdminFeeTooHigh();
``` [1](#0-0) 

But `setPoolBinAdditionalFees` passes the values straight through with no cap check:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
``` [2](#0-1) 

The pool's `setBinAdditionalFees` also performs no cap check — it only validates the bin index: [3](#0-2) 

During every swap, the per-bin additional fee is added directly to the oracle-derived base fee before computing the trader's required input:

```solidity
params.baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6),
``` [4](#0-3) [5](#0-4) [6](#0-5) [7](#0-6) 

The hard cap on the global spread fee is `HARD_MAX_SPREAD_FEE_E6 = 200_000` (20%): [8](#0-7) 

`addFeeBuyE6` / `addFeeSellE6` are `uint16`, so they can reach `65 535` (6.55%) with no on-chain limit. The total effective fee for a swap in the targeted bin becomes `spreadFeeE6 (≤ 20%) + addFeeE6 (≤ 6.55%)`, exceeding the documented hard cap by up to 6.55 percentage points.

### Impact Explanation

Every swap that crosses the manipulated bin pays the inflated fee. The excess fee amount is deducted from the trader's input token before the net amount is credited to the bin, constituting a direct loss of trader principal. The protocol/admin split (`spreadFeeE6`) is then applied to the inflated total fee, so both the LP and the protocol/admin receive more than the cap permits. Traders have no on-chain mechanism to detect the per-bin override before execution.

### Likelihood Explanation

The pool admin is a semi-trusted role that is explicitly constrained to operate "inside caps." The missing cap check is reachable by any address holding `poolAdmin[pool]` — a single transaction suffices, with no timelock. The active bin (where most swaps execute) is the highest-value target; setting `addFeeBuyE6 = 65535` on bin 0 silently inflates every buy-token0 swap.

### Recommendation

Add a cap check in `setPoolBinAdditionalFees` before forwarding to the pool, mirroring the pattern used in `setPoolAdminFees`:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (addFeeBuyE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

Alternatively, introduce a dedicated `maxBinAdditionalFeeE6` cap that the factory owner can configure, and enforce it here. The same cap should be applied at pool creation time inside `_unpackAndValidateBinStates` where `buyFee` and `sellFee` are unpacked from the packed bin arrays without any range check. [9](#0-8) 

### Proof of Concept

1. Pool admin calls `factory.setPoolBinAdditionalFees(pool, 0, 65535, 65535)`.
2. No revert — `65535 > maxAdminSpreadFeeE6` is never checked.
3. `_binStates[0].addFeeBuyE6 = 65535` and `addFeeSellE6 = 65535` are stored.
4. A trader calls `swap(...)` with a buy-token0 direction. The swap loop enters bin 0 and computes:
   ```
   currBinBuyFeeX64 = baseFeeX64 + mulDiv(65535, ONE_X64, 1e6)
   ```
   The trader's required token1 input is inflated by ≈6.55% above what the oracle spread alone would require.
5. The trader receives the correct token0 output but pays ≈6.55% more token1 than the hard-capped fee system permits, with no on-chain warning.
