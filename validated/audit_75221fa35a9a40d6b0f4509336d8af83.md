Audit Report

## Title
Unbounded Iteration in SNS Governance `distribute_rewards` Over All Proposals × Neuron Ballots Without Instruction-Limit Guard - (File: rs/sns/governance/src/governance.rs)

## Summary
`SNS Governance::distribute_rewards` is a synchronous function called from the periodic timer that iterates over all `ReadyToSettle` proposals and all neuron ballots within each, then over all voting neurons to apply maturity, with no instruction-limit guard at any point. With ceiling parameters of up to 700 proposals × 200,000 neurons, the function can exhaust the IC per-message instruction limit (~40 billion instructions), causing the timer callback to trap on every subsequent invocation and permanently breaking SNS reward distribution.

## Finding Description
`distribute_rewards` at [1](#0-0)  is called unconditionally from `run_periodic_tasks` at [2](#0-1)  whenever `should_distribute_rewards` is true. The function contains four unbounded loops with no instruction-limit check:

1. **Round accumulation loop** at [3](#0-2)  — iterates over all missed reward rounds performing `Decimal` arithmetic.

2. **Ballot aggregation double-loop** at [4](#0-3)  — iterates `proposals × neurons` (worst case 700 × 200,000 = 140,000,000 iterations), each performing `Decimal` arithmetic and `HashMap` insertion.

3. **Maturity distribution loop** at [5](#0-4)  — iterates over all voting neurons (up to 200,000), performing `Decimal` division and neuron mutation.

4. **Proposal settlement loop** at [6](#0-5)  — calls `process_proposal` for each settled proposal.

A search across all of `rs/sns/governance/**/*.rs` confirms zero use of `is_message_over_threshold`, `instruction_counter`, or any equivalent guard. The ceiling constants bounding the worst-case iteration count are confirmed at [7](#0-6) .

By contrast, NNS governance's `continue_processing` in `rs/nns/governance/src/reward/distribution.rs` explicitly checks `is_over_instructions_limit()` after each neuron and breaks out of the loop to resume in the next timer tick: [8](#0-7) 

When the IC instruction limit is exceeded, the runtime traps the message and rolls back all state changes. Because `distribute_rewards` is called inside the async `run_periodic_tasks` timer callback, the trap causes the entire periodic task to fail. The timer reschedules and traps again with identical state, creating a permanent failure loop.

## Impact Explanation
This matches the allowed High impact: **"Significant SNS security impact with concrete user or protocol harm."** The concrete harms are: (1) voting reward maturity is never credited to any neuron; (2) proposals are never moved from `ReadyToSettle` to `Settled`, so ballots are never cleared; (3) since `max_number_of_proposals_with_ballots` is enforced, the accumulation of unsettled proposals eventually blocks all new proposal submissions, rendering SNS governance non-functional. This is a governance availability impact affecting all participants of the SNS.

## Likelihood Explanation
Any unprivileged principal holding SNS tokens can stake neurons. The `max_number_of_neurons` governance parameter defaults to and is capped at 200,000 per [9](#0-8) . A large active SNS with tens of thousands of neurons and many proposals settling in the same reward round can reach the instruction limit organically. A motivated actor can accelerate this by staking many small neurons and submitting many proposals up to the `max_number_of_proposals_with_ballots` ceiling of 700. No special privileges are required.

## Recommendation
Apply the same batched-processing pattern already used in NNS governance (`rs/nns/governance/src/reward/distribution.rs`):
1. Introduce a resumable `RewardsDistributionInProgress` state machine for SNS governance, persisted to stable storage.
2. Check `is_message_over_threshold` (or equivalent IC system API) after each neuron is processed in the maturity distribution loop and break out, saving remaining work.
3. Re-run distribution in subsequent timer ticks until all neurons are credited before settling proposals.
4. As a minimum short-term mitigation, impose a hard cap on neurons processed per invocation and document the maximum safe `max_number_of_neurons × max_number_of_proposals_with_ballots` product.

## Proof of Concept
A deterministic integration test using PocketIC can reproduce this:
1. Deploy an SNS governance canister with `max_number_of_neurons = N` (e.g., 50,000) and `max_number_of_proposals_with_ballots = P` (e.g., 700).
2. Stake `N` neurons and submit `P` proposals, allowing all to reach `ReadyToSettle`.
3. Advance the PocketIC clock past the reward round boundary to trigger `should_distribute_rewards = true`.
4. Call `run_periodic_tasks` and observe the canister trap due to instruction limit exhaustion.
5. Confirm that subsequent calls to `run_periodic_tasks` continue to trap and that no `RewardEvent` is ever recorded, permanently blocking reward distribution.
