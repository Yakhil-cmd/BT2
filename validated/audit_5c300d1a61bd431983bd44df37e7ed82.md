### Title
Incomplete `REWARD_ADDRESS_IS_TOKEN` Validation in Pool Contract Allows Reward Address to Be Set to a Cross-Token Contract Address - (File: `src/pool/pool.cairo`)

---

### Summary

The `Pool` contract's `change_reward_address` (and `enter_delegation_pool`) only validates the reward address against the pool's own single token, not against all registered token addresses. This mirrors the M-7 pattern exactly: a protection check exists but is incomplete, allowing a pool member to set their reward address to a different token contract (e.g., the STRK token address in a BTC pool), causing all future STRK reward claims to be permanently sent to that token contract and frozen.

---

### Finding Description

The staking contract's `change_reward_address` uses `does_token_exist`, which checks ALL registered tokens (STRK + all BTC tokens):

```cairo
// staking.cairo:520-524
assert!(
    !self.does_token_exist(token_address: reward_address),
    "{}",
    GenericError::REWARD_ADDRESS_IS_TOKEN,
);
```

Where `does_token_exist` is:

```cairo
// staking.cairo:2227-2229
fn does_token_exist(self: @ContractState, token_address: ContractAddress) -> bool {
    token_address == STRK_TOKEN_ADDRESS || self.btc_tokens.read(token_address).is_some()
}
```

However, the pool contract's `change_reward_address` only checks against the pool's own single token:

```cairo
// pool.cairo:505-510
fn change_reward_address(ref self: ContractState, reward_address: ContractAddress) {
    assert!(
        self.token_dispatcher.contract_address.read() != reward_address,
        "{}",
        GenericError::REWARD_ADDRESS_IS_TOKEN,
    );
```

The same incomplete check exists in `enter_delegation_pool`:

```cairo
// pool.cairo:193-195
let token_address = token_dispatcher.contract_address;
assert!(token_address != pool_member, "{}", Error::POOL_MEMBER_IS_TOKEN);
assert!(token_address != reward_address, "{}", GenericError::REWARD_ADDRESS_IS_TOKEN);
```

For a BTC pool, `token_dispatcher.contract_address` is the BTC token address. The check therefore only blocks the BTC token address as a reward address — it does **not** block the STRK token address or any other registered token address. A pool member in a BTC pool can freely set `reward_address = STRK_TOKEN_ADDRESS`.

When `claim_rewards` is later called, rewards are always paid in STRK regardless of pool type:

```cairo
// pool.cairo:365-366
let reward_token = IERC20Dispatcher { contract_address: STRK_TOKEN_ADDRESS };
reward_token.checked_transfer(recipient: reward_address, amount: rewards.into());
```

STRK tokens transferred to the STRK token contract itself (or any other token contract) have no recovery path and are permanently frozen.

---

### Impact Explanation

A pool member whose reward address is set to a token contract address will have all their accumulated STRK rewards permanently sent to that token contract on every `claim_rewards` call. The STRK token contract has no mechanism to recover or forward these tokens. This constitutes **permanent freezing of unclaimed yield**, matching the allowed High impact.

---

### Likelihood Explanation

Any pool member in a BTC pool (an unprivileged role) can call `change_reward_address` with `STRK_TOKEN_ADDRESS` as the argument. The pool's guard passes because `STRK_TOKEN_ADDRESS != BTC_TOKEN_ADDRESS`. No special access, leaked keys, or external dependencies are required. The same applies to a STRK pool member setting their reward address to a BTC token address. The action is directly reachable from a normal user flow.

---

### Recommendation

Replace the single-token check in both `enter_delegation_pool` and `change_reward_address` in `pool.cairo` with a call to the staking contract's `does_token_exist` (or an equivalent cross-token check). The pool already holds a reference to the staking contract via `staking_pool_dispatcher`, so it can query whether the proposed reward address is any registered token:

```cairo
// pool.cairo change_reward_address - proposed fix
fn change_reward_address(ref self: ContractState, reward_address: ContractAddress) {
    // Check against ALL registered tokens, not just the pool's own token
    let staking_dispatcher = IStakingDispatcher {
        contract_address: self.staking_pool_dispatcher.contract_address.read()
    };
    assert!(
        !staking_dispatcher.does_token_exist(token_address: reward_address),
        "{}",
        GenericError::REWARD_ADDRESS_IS_TOKEN,
    );
    ...
}
```

Apply the same fix to the `reward_address` check in `enter_delegation_pool`.

---

### Proof of Concept

1. Deploy the system with STRK and a BTC token registered.
2. A staker opens a BTC delegation pool via `set_open_for_delegation(btc_token_address)`.
3. A pool member calls `enter_delegation_pool(reward_address: STRK_TOKEN_ADDRESS, amount: X)` on the BTC pool.
   - The check at pool.cairo:195 evaluates `btc_token_address != STRK_TOKEN_ADDRESS` → `true` → passes.
   - The pool member record is created with `reward_address = STRK_TOKEN_ADDRESS`.
4. After epochs pass, the pool member calls `claim_rewards(pool_member)`.
   - pool.cairo:366 executes `STRK.transfer(recipient: STRK_TOKEN_ADDRESS, amount: rewards)`.
   - STRK rewards are sent to the STRK token contract itself and are permanently frozen.
5. The pool member has lost all their yield with no recovery path. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
