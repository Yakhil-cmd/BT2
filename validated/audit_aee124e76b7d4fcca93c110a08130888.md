### Title
FLOW Tokens Permanently Locked in EVM State When `CadenceOwnedAccount` Is Destroyed With Non-Zero Balance - (File: fvm/evm/stdlib/contract.cdc)

### Summary
The `EVM.CadenceOwnedAccount` resource in `fvm/evm/stdlib/contract.cdc` can be destroyed while holding a non-zero EVM balance. Because COA addresses have no associated private key, the locked FLOW tokens cannot be recovered via EVM transactions, resulting in permanent cross-VM asset loss. This is the direct Flow analog of the external report's pattern: funds are credited to an address that lacks the mechanism to retrieve them.

### Finding Description
`EVM.CadenceOwnedAccount` is the **only** Cadence-side mechanism to withdraw FLOW tokens from a COA's EVM address back to Cadence. The `withdraw` function is gated behind the `Owner | Withdraw` entitlement and is only callable through the resource itself. [1](#0-0) 

The resource has **no `destroy` function** that checks or handles a non-zero EVM balance before destruction. When a COA resource is destroyed with a non-zero balance, the FLOW tokens remain in the EVM state but become permanently inaccessible because:

1. The COA resource — the only Cadence-side withdrawal mechanism — is gone.
2. COA addresses have no private key, so EVM transactions cannot be signed for that address. [2](#0-1) 

The `EVMAddress` struct's `deposit()` function is `access(all)`, meaning any transaction can deposit FLOW tokens into any EVM address, including a COA address, without any guarantee that the controlling resource still exists. [3](#0-2) 

The Go-level `Withdraw` implementation confirms that withdrawal is an authorized call that requires the COA resource to be present: [4](#0-3) 

### Impact Explanation
**Cross-VM asset loss.** FLOW tokens deposited into a COA's EVM address become permanently locked when the COA resource is destroyed. The EVM balance remains in the EVM state (visible via `EVMAddress.balance()`), but no Cadence transaction can withdraw it and no EVM transaction can sign for the address. The tokens are effectively burned without reducing the total supply, creating an accounting discrepancy.

**Impact: Medium** — funds are permanently lost with no recovery path.

### Likelihood Explanation
**Likelihood: High** — the pattern is trivially reachable by any unprivileged transaction sender. A user can:
- Create a COA, deposit FLOW tokens, and destroy the COA in a single transaction.
- A Cadence contract author can create a contract that accepts user funds, creates a COA, deposits the funds, and destroys the COA (intentionally or through a logic error).

The existing test suite already demonstrates this exact pattern without flagging it as a problem: [5](#0-4) [6](#0-5) 

Both tests create a COA, deposit FLOW tokens, and then `destroy` the resource without withdrawing the balance first.

### Recommendation
Add a `destroy` function to `EVM.CadenceOwnedAccount` in `fvm/evm/stdlib/contract.cdc` that either:
1. **Panics if the EVM balance is non-zero**, preventing accidental destruction with locked funds.
2. **Automatically withdraws the EVM balance** to a caller-specified Cadence `FlowToken.Vault` receiver before destruction.

Option 1 example:
```cadence
destroy() {
    pre {
        self.balance().isZero():
            "EVM.CadenceOwnedAccount: Cannot destroy a COA with a non-zero EVM balance. Withdraw funds first."
    }
}
```

### Proof of Concept
Any unprivileged user can submit the following transaction to permanently lock FLOW tokens:

```cadence
import EVM from <EVM_ADDRESS>
import FungibleToken from <FT_ADDRESS>
import FlowToken from <FLOW_TOKEN_ADDRESS>

transaction(amount: UFix64) {
    prepare(account: auth(BorrowValue) &Account) {
        let vaultRef = account.storage
            .borrow<auth(FungibleToken.Withdraw) &FlowToken.Vault>(from: /storage/flowTokenVault)
            ?? panic("No vault")

        // Step 1: Create a COA
        let coa <- EVM.createCadenceOwnedAccount()

        // Step 2: Deposit FLOW tokens into the COA's EVM address
        let vault <- vaultRef.withdraw(amount: amount) as! @FlowToken.Vault
        coa.deposit(from: <-vault)

        // Step 3: Destroy the COA resource — FLOW tokens are now permanently locked.
        // No private key exists for the COA address, so EVM transactions cannot recover them.
        // No Cadence resource controls the address anymore.
        destroy coa
    }
}
```

After execution, `EVM.EVMAddress(bytes: <coa_address_bytes>).balance()` will show the deposited amount, but no transaction can ever withdraw it. [7](#0-6)

func (a *Account) Withdraw(b types.Balance) *types.FLOWTokenVault {
	defer a.fch.backend.StartChildSpan(trace.FVMEVMWithdraw).End()

	res, err := a.executeAndHandleAuthorizedCall(
		types.NewWithdrawCall(
			a.fch.addressAllocator.NativeTokenBridgeAddress(),
			a.address,
			b,
			a.Nonce(),
		),
		b,
		true,
	)
	panicOnErrorOrInvalidOrFailedState(res, err)

	return types.NewFlowTokenVault(b)
}
