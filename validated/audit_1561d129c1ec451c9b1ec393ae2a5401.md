### Title
Unprivileged Caller Can Permanently Suppress Consensus Block Rewards via `update_rewards(disable_rewards: true)` - (File: src/staking/staking.cairo)

### Summary
The `IStakingRewardsManager::update_rewards` function accepts a caller-controlled `disable_rewards` parameter with no access control. Any unprivileged address can call it every block with `disable_rewards: true`, consuming the single per-block reward slot without distributing rewards, permanently suppressing consensus yield for all stakers.

### Finding Description
`update_rewards` enforces a global, single-use-per-block gate via `last_reward_block`: [1](#0-0) 

After passing that gate, the function unconditionally writes the current block number to `last_reward_block`: [2](#0-1) 

Then it checks the caller-supplied `disable_rewards` flag and returns early if it is `true`, skipping all reward computation and distribution: [3](#0-2) 

There is no role check anywhere in the function — `general_prerequisites()` only asserts the contract is unpaused and the caller is non-zero: [4](#0-3) 

The interface definition confirms no access restriction is documented or enforced: [5](#0-4) 

Because `last_reward_block` is a single global field (not per-staker), one successful call per block exhausts the slot for every staker. An attacker who calls `update_rewards(any_valid_staker, disable_rewards: true)` at every block permanently prevents the legitimate block-proposer call (`disable_rewards: false`) from ever succeeding in that block.

The `disable_rewards` parameter is the direct analog of the external report's "unsigned/unvalidated parameter" — just as the `m` parameter was never signed and could be replayed with different values, `disable_rewards` is never authenticated and can be set adversarially on every block.

### Impact Explanation
Consensus block rewards are calculated and distributed once per block. When the slot is consumed with `disable_rewards: true`, the rewards for that block are never credited to any staker and are permanently lost — the reward supplier retains the funds but the stakers' yield is irreversibly suppressed for every attacked block. This constitutes **permanent freezing of unclaimed yield** (High impact).

### Likelihood Explanation
- Requires no special role, no stake, and no funds beyond gas.
- Any externally-owned account or contract can execute the attack.
- One transaction per block is sufficient to sustain it indefinitely.
- The attacker has a clear griefing motive (e.g., a competing validator set or protocol adversary).

### Recommendation
Add an authorized-caller check to `update_rewards`. The simplest fix is to restrict it to a designated role (e.g., `only_app_governor` or a dedicated `REWARDS_MANAGER` role). Alternatively, remove the `disable_rewards` parameter entirely and handle the pre-consensus guard internally, since `is_pre_consensus()` is already checked on line 1487.

### Proof of Concept
1. Consensus rewards are active (`consensus_rewards_first_epoch` has passed).
2. At block `N`, attacker calls `update_rewards(any_active_staker, disable_rewards: true)`.
3. The gate check passes (`N > last_reward_block`); `last_reward_block` is written to `N`; function returns early — no rewards distributed.
4. The legitimate block-proposer calls `update_rewards(staker, disable_rewards: false)` at block `N` — reverts with `REWARDS_ALREADY_UPDATED` because `N > N` is false.
5. Attacker repeats at block `N+1`, `N+2`, … indefinitely.
6. All stakers receive zero consensus block rewards for the entire duration of the attack.
