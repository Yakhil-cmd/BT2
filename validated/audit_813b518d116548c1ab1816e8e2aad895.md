Audit Report

## Title
Stale `dapp_canister_ids` Entry After Canister Deletion Causes Permanent Panic in `set_dapp_controllers`, Blocking SNS Swap Finalization - (File: `rs/sns/root/src/lib.rs`)

## Summary

`SnsRootCanister::set_dapp_controllers` performs a pre-flight `canister_status` check on every entry in `dapp_canister_ids`. If any registered dapp canister has been deleted (e.g., ran out of cycles), the management canister returns an error and the code unconditionally panics. Because `dapp_canister_ids` is persisted proto state and the stale entry is never cleaned up on panic, every subsequent call to `set_dapp_controllers` — including the swap finalization path — hits the same panic permanently, locking all surviving dapp canisters under SNS root control with no recovery path.

## Finding Description

In `rs/sns/root/src/lib.rs` at lines 796–826, `set_dapp_controllers` iterates over `dapp_canister_ids` and calls `management_canister_client.canister_status(...)` on each entry as a pre-flight ownership check. On any error — including `CanisterNotFound` returned when a canister has been deleted — the `Err(_)` arm unconditionally panics:

```rust
// lines 807–812
Err(_) => {
    // TODO(NNS1-1993): Remove this panic and return an error type instead.
    panic!(
        "Could not get the status of canister: {dapp_canister_id}.  Root may not be a controller."
    )
}
```

The stale entry cleanup at lines 872–879 (`swap_remove_if` on `dapp_canister_ids`) is only reached after a successful `update_settings` call in the second loop — it is never reached when the pre-flight panic fires. The `dapp_canister_ids` field is a `prost(message, repeated, tag = "3")` field on `SnsRootCanister` (persisted proto state), so the stale `PrincipalId` survives canister upgrades. Every future invocation of `set_dapp_controllers` re-enters the same pre-flight loop and hits the same panic. There is no administrative escape hatch: neither the swap canister nor governance can bypass the pre-flight check.

The existing `TODO(NNS1-1993)` comment in the code itself acknowledges the panic should be replaced with a structured error, confirming the bug is known but unresolved.

## Impact Explanation

`set_dapp_controllers` is the sole mechanism by which (1) the swap canister returns dapp canisters to original owners after SNS token swap completion or failure, and (2) governance executes `DeregisterDappCanisters` proposals. A single deleted dapp canister permanently bricks both paths for the entire SNS instance. Surviving dapp canisters remain locked under SNS root control indefinitely with no on-chain recovery. This is a concrete, permanent application-level DoS against SNS swap finalization and SNS governance operations, matching the allowed High impact: *"Significant SNS security impact with concrete user or protocol harm"* and *"Application/platform-level DoS... not based on raw volumetric DDoS"* ($2,000–$10,000).

## Likelihood Explanation

Dapp canisters registered with SNS root can run out of cycles through ordinary operation (insufficient top-up, high traffic, neglect). The IC automatically deletes canisters that remain below the freezing threshold for a grace period — this is a standard, non-adversarial lifecycle event. No privileged access is required to trigger the condition. An adversary who can call any cycles-consuming public method on a dapp canister can also accelerate depletion. The condition is realistic, repeatable, and requires no special attacker capability beyond the ability to register a dapp canister with an SNS.

## Recommendation

In the pre-flight loop of `set_dapp_controllers` (lines 803–814), handle `canister_status` errors gracefully instead of panicking:

1. **Prune stale entries**: On a `CanisterNotFound` error (or any error indicating the canister no longer exists), remove the entry from `dapp_canister_ids` and continue processing remaining canisters.
2. **Return a structured error**: As the existing `TODO(NNS1-1993)` already acknowledges, replace the `panic!` with a typed error return so callers can observe and handle the failure without trapping the canister.
3. **Expose a recovery method**: Add a governance-callable method to explicitly deregister a canister ID from `dapp_canister_ids` without requiring `canister_status` to succeed, enabling recovery from the stuck state.

## Proof of Concept

1. Deploy an SNS. Register dapp canister `D` via `register_dapp_canisters`. `D` is appended to `dapp_canister_ids` (line 743).
2. Canister `D` runs out of cycles and is deleted by the IC (or explicitly deleted). `dapp_canister_ids` still contains `D`'s `PrincipalId` in persisted proto state.
3. The swap canister calls `set_dapp_controllers` (with `canister_ids: None`) to return all dapp canisters to original owners after swap finalization.
4. The pre-flight loop (line 796) reaches `D`, calls `management_canister_client.canister_status(D)`.
5. The management canister returns `CanisterNotFound`. The `Err(_)` arm fires at line 807: `panic!("Could not get the status of canister: D. Root may not be a controller.")`.
6. SNS root traps. No controller changes are made. `dapp_canister_ids` is unchanged.
7. Every retry of `set_dapp_controllers` hits the same panic. The SNS swap cannot finalize. All surviving dapp canisters remain permanently locked under SNS root.

A deterministic integration test or PocketIC test can reproduce this by: registering a dapp canister, mocking `canister_status` to return `Err(CanisterNotFound)` for that canister ID, calling `set_dapp_controllers`, and asserting the canister traps and `dapp_canister_ids` is unchanged.
