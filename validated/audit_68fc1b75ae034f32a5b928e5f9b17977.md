### Title
TOCTOU Race in WalletConnect Disconnect Allows `chia_sendTransaction` After Pair Revocation — (`packages/gui/src/electron/utils/dispatchPairRequest.ts`, `packages/gui/src/components/walletConnect/WalletConnectProvider.tsx`)

---

### Summary

A dApp that has been granted a bypass for `chia_sendTransaction` can send a burst of concurrent `session_request` events such that some requests pass the `findPair` authorization check and enter `process(context)` before `revokePair` removes the pair from `pairStore`. Because there is no re-check of pair validity after `process(context)` begins, those in-flight requests complete — submitting real transactions to the Chia daemon — after the user has initiated disconnect.

---

### Finding Description

**Disconnect flow** (`WalletConnectProvider.tsx`, `disconnectPair`, lines 145–169):

```
1. await window.permissionsAPI.revokePair(topic)   → removePair() in pairStore
2. disconnect all WC SDK sessions
3. disconnect the pairing
```

**Request execution flow** for a bypassed command (`dispatchPairRequest.ts`, lines 14–67):

```
1. await Promise.all([getLoggedInFingerprint(), isMainnet()])  ← yields to event loop
2. findPair(topic)                                             ← synchronous in-memory read
3. pair.bypass.includes(command) → true
4. return process(context)                                     ← async, yields inside sendCommand
```

The Electron main process is single-threaded; IPC handlers interleave at every `await`. The race window is:

| Step | Event |
|------|-------|
| T0 | 50 `DISPATCH_AS_PAIR` IPC calls arrive; each starts `dispatchPairRequest` and yields at `await Promise.all(...)` |
| T1 | First N calls resolve `Promise.all`, call `findPair` (pair still present), pass all synchronous checks, enter `process(context)`, yield at `await sendCommand(...)` |
| T2 | `REVOKE_PAIR` IPC arrives; `removePair(topic)` executes synchronously — pair gone from pairStore |
| T3 | The N in-flight `sendCommand` calls resume and submit transactions to the Chia daemon |

The critical gap: for bypassed commands, `dispatchPairRequest` has **no second `findPair` check** between the bypass branch and `process(context)`. [1](#0-0) 

The `confirm` callback does contain a second `findPair` guard, but that path is never reached for bypassed commands. [2](#0-1) 

`removePair` is a synchronous in-memory write followed by a YAML flush; it completes in one event-loop tick. [3](#0-2) 

`sendCommand` involves network I/O to the Chia daemon and takes far longer, so requests already inside it at T2 complete after T2. [4](#0-3) 

---

### Impact Explanation

Transactions (`chia_sendTransaction`) are submitted to the Chia daemon after the user has revoked the pair. The user's explicit revocation intent is violated: they believe no further commands can execute the moment they click "Disconnect," but in-flight requests that cleared the authorization gate before `removePair` ran will still transfer XCH/CATs.

---

### Likelihood Explanation

Requires a pre-existing pair where the user previously granted bypass for `chia_sendTransaction`. Given that condition, the attacker fully controls request timing and concurrency. Sending 20–50 concurrent `session_request` events is trivial from a dApp. The race window (duration of `sendCommand` network I/O) is reliably tens to hundreds of milliseconds — wide enough to be hit deterministically with concurrent requests.

---

### Recommendation

After `process(context)` is entered (or immediately before `sendCommand` is called), re-check `findPair(topic)` and abort if the pair no longer exists. Alternatively, hold a per-topic mutex that `disconnectPair` acquires exclusively, preventing any new `process(context)` calls from starting and waiting for in-flight ones to drain before `removePair` is called.

---

### Proof of Concept

1. Establish a WalletConnect pair; grant bypass for `chia_sendTransaction`.
2. From the dApp, fire 50 concurrent `session_request` events for `chia_sendTransaction` with valid params.
3. Immediately after firing, trigger `handleDisconnectApp` from the Settings UI (or call `window.permissionsAPI.revokePair` directly).
4. Observe: some `chia_sendTransaction` calls complete (daemon returns success) after `removePair` has executed (verifiable by checking `dapp-pairs.yaml` is already empty at the time the daemon responds).
5. Assert: zero successful daemon responses should occur after `removePair` completes; in practice, N > 0 responses will be observed. [5](#0-4) [6](#0-5) [7](#0-6)

async function disconnectPair(client: Client, topic: string) {
  try {
    // revoke pair from main process
    await window.permissionsAPI.revokePair(topic);

    // get all sessions for the pairing
    const sessions = client.session.getAll();
    const relatedSessions = sessions.filter((s) => s.pairingTopic === topic);

    // disconnect all related sessions
    await Promise.all(
      relatedSessions.map((session) =>
        client.disconnect({
          topic: session.topic,
          reason: getSdkError('USER_DISCONNECTED'),
        }),
      ),
    );

    // disconnect the pairing
    await client.core.pairing.disconnect({ topic });
  } catch (e) {
    processError(e as Error);
  }
}
