### Title
Systematic Operator Undercompensation Due to Truncating Integer Division in `delta_gas` Calculation — (`basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`)

---

### Summary

In `compute_gas_refund`, the adjustment of `gas_used` to account for native resource consumption uses truncating (floor) integer division. This causes a per-transaction rounding loss where the operator is systematically paid slightly less than the true native resource cost. The discrepancy accumulates across all transactions, analogous to the LOB `calcCommissions` dust-accumulation pattern.

---

### Finding Description

`compute_gas_refund` in `refund_calculation.rs` implements ZKsync OS's double-resource accounting reconciliation. After execution, it computes `delta_gas` to bridge the gap between EVM gas consumed and native resource consumed:

```rust
// Line 69-79
let delta_gas = if native_per_gas == 0 {
    0
} else {
    (native_used / native_per_gas) as i64 - (gas_used as i64)
};

if delta_gas > 0 {
    gas_used += delta_gas as u64;
}
// TODO: return delta_gas to gas_used?
```

The expression `native_used / native_per_gas` is Rust integer division — it truncates toward zero (floor). The true gas-equivalent of `native_used` native units is `native_used / native_per_gas` (real number), but the code charges only `⌊native_used / native_per_gas⌋`. The fractional remainder `(native_used % native_per_gas)` native units — worth up to `(native_per_gas − 1)` native units per transaction — is silently discarded.

`native_per_gas` is itself computed with ceiling division at validation time:

```rust
// validation_impl.rs line 135
u256_try_to_u64(&gas_price.div_ceil(native_price))
```

So `native_per_gas = ⌈gas_price / native_price⌉`. The ceiling here is intended to ensure the operator is not undercompensated for native costs, but the subsequent floor division in `delta_gas` partially or fully negates this protection.

The final fee split is:
- **Operator receives**: `gas_used × gas_price` (where `gas_used` is rounded down by up to 1 gas unit)
- **User receives refund**: `(gas_limit − gas_used) × gas_price` (correspondingly rounded up)

The total `gas_limit × gas_price` is conserved — no funds are stuck — but the operator is systematically underpaid by up to `gas_price` tokens per transaction (the value of 1 gas unit).

The `// TODO: return delta_gas to gas_used?` comment at line 80 confirms the developers are aware of an unresolved accounting asymmetry in this function.

---

### Impact Explanation

**Type**: Resource accounting bug — systematic operator undercompensation.

Per transaction, the operator receives up to `gas_price` fewer tokens than the true native resource cost warrants. Over the lifetime of the chain with millions of transactions, this accumulates into a material shortfall in operator compensation. The operator subsidizes the rounding error on every transaction where `delta_gas > 0` (i.e., native resource consumption exceeds EVM gas consumption — the common case for compute-heavy or pubdata-heavy transactions).

At `gas_price = 1000` and `native_per_gas = 2`, the per-transaction loss is up to 500 tokens. Across 10 million such transactions, the cumulative undercompensation reaches 5 × 10⁹ tokens.

---

### Likelihood Explanation

**High likelihood of triggering**: Every L2 transaction that consumes native resources disproportionately to EVM gas (i.e., any transaction where `delta_gas > 0`) triggers the rounding loss. This includes any transaction with significant pubdata or native-heavy opcodes. No special attacker action is required — ordinary users submitting ordinary transactions cause the accumulation.

---

### Recommendation

Replace the truncating division with ceiling division in the `delta_gas` computation so the operator is charged the full gas-equivalent of native consumption:

```rust
// Current (truncates, undercharges operator):
(native_used / native_per_gas) as i64 - (gas_used as i64)

// Recommended (ceiling, ensures full native cost is recovered):
(native_used.div_ceil(native_per_gas)) as i64 - (gas_used as i64)
```

Alternatively, following the LOB report's recommendation pattern, compute the operator's gas-equivalent as `total_native_gas_equivalent − already_accounted_gas` to avoid independent rounding on each component.

### Proof of Concept

**Setup**: `gas_price = 1001`, `native_price = 1000`, `gas_limit = 100_000`

1. `native_per_gas = ⌈1001 / 1000⌉ = 2`
2. Transaction executes; EVM uses 80,000 gas (`gas_used_from_ergs = 80_000`)
3. Native consumed: `native_used = 200_001`
4. `delta_gas = ⌊200_001 / 2⌋ − 80_000 = 100_000 − 80_000 = 20_000`
5. `gas_used = 80_000 + 20_000 = 100_000`

**True gas-equivalent of native**: `200_001 / 2 = 100_000.5` gas

**Operator receives**: `100_000 × 1001 = 100_100_000` tokens
**Operator should receive**: `100_000.5 × 1001 ≈ 100_100_500` tokens
**Per-transaction shortfall**: ~500 tokens (the `0.5 × gas_price` rounding loss)

The rounding loss is bounded by `(native_per_gas − 1) / native_per_gas × gas_price < gas_price` per transaction, and accumulates linearly with transaction count. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
