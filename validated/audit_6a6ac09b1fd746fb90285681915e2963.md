### Title
Unvalidated `app_pk` Identity Element in CKD Protocol Bypasses ElGamal Masking, Disclosing Confidential Derived Key to Coordinator — (File: `src/confidential_key_derivation/protocol.rs`)

### Summary

The `ckd` function in `src/confidential_key_derivation/protocol.rs` accepts a caller-supplied `app_pk: PublicKey` (a `G1Projective` point) with no check that it is not the group identity element. When `app_pk = G1::identity()`, the ElGamal masking term `app_pk * y` collapses to the identity, so each participant's ciphertext component `big_c_i` reduces directly to the unmasked secret share `S_i = hash_point * private_share_i`. The coordinator then aggregates these into the full confidential derived key `S = msk * H(pk || app_id)` in plaintext, without needing `app_sk` to unmask it.

### Finding Description

In `compute_signature_share` (called by both coordinator and participant paths):

```rust
// C <- S + y . A
let big_c = big_s + app_pk * y.0;
``` [1](#0-0) 

`big_s = hash_point * private_share` is the participant's secret contribution. The term `app_pk * y` is the ElGamal blinding factor — it hides `big_s` from the coordinator. If `app_pk = G1Projective::identity()`, then `app_pk * y = identity`, so `big_c = big_s` with no masking.

Each participant sends `norm_big_c = lambda_i * big_c` to the coordinator:

```rust
let norm_big_c = big_c * lambda_i;
``` [2](#0-1) 

The coordinator sums all contributions:

```rust
norm_big_c += participant_output.big_c();
``` [3](#0-2) 

When `app_pk = identity` for all participants, the sum is:

```
big_c = Σ lambda_i * S_i = msk * H(pk || app_id) = S
```

This is exactly the confidential derived key `S`, now sitting in `ckd_output.big_c` in plaintext — no `app_sk` needed to unmask it.

The `ckd` public entry point performs no validation on `app_pk`:

```rust
pub fn ckd(
    participants: &[Participant],
    coordinator: Participant,
    me: Participant,
    key_pair: KeygenOutput,
    app_id: impl Into<AppId>,
    app_pk: PublicKey,   // ← no identity check
    rng: impl CryptoRngCore + Send + 'static,
) -> Result<impl Protocol<Output = CKDOutputOption>, InitializationError> {
    if participants.len() < 2 { ... }
    let Some(participants) = ParticipantList::new(participants) else { ... };
    if !participants.contains(me) { ... }
    if !participants.contains(coordinator) { ... }
    // no check: app_pk != G1::identity()
``` [4](#0-3) 

### Impact Explanation

**Critical — Extraction/disclosure of confidential derived secrets.**

The CKD protocol's security guarantee is that the coordinator learns only the ElGamal ciphertext `(Y, C)` and cannot recover `S = msk * H(pk || app_id)` without the client's `app_sk`. Passing `app_pk = G1::identity()` completely voids this guarantee: the coordinator receives `S` directly in `ckd_output.big_c`. This matches the allowed impact: *"Critical: Extraction, reconstruction, or disclosure of… confidential derived secrets."*

### Likelihood Explanation

A malicious coordinator controls the distribution of protocol parameters to other participants. In a realistic deployment, the coordinator receives `app_pk` from the client and relays it to all MPC nodes. A malicious coordinator can substitute `G1::identity()` for the real `app_pk` before distributing it. Since there is no validation at any participant's `ckd` call site, all participants silently accept the degenerate value and the attack succeeds in a single protocol run.

### Recommendation

Add an explicit identity-element check on `app_pk` at the start of `ckd`, before the protocol is launched:

```rust
if bool::from(app_pk.is_identity()) {
    return Err(InitializationError::BadParameters(
        "app_pk must not be the group identity element".to_string(),
    ));
}
```

This mirrors the existing pattern used elsewhere in the library (e.g., the `delta.is_zero()` check in `rerandomize_presign` and the `w.is_zero()` check in robust presigning). [5](#0-4) [6](#0-5) 

### Proof of Concept

**Setup:** 3 participants, threshold 2. Coordinator is `P0`.

1. Client generates `app_sk`, computes real `app_pk = app_sk * G1::generator()`.
2. Malicious `P0` (coordinator) intercepts `app_pk` and distributes `app_pk = G1Projective::identity()` to all participants.
3. All three participants call `ckd(..., app_pk = G1::identity(), ...)`.
4. Each participant computes:
   - `big_c_i = hash_point * private_share_i + identity * y_i = hash_point * private_share_i`
   - `norm_big_c_i = lambda_i * hash_point * private_share_i`
5. Coordinator sums: `ckd_output.big_c = Σ norm_big_c_i = hash_point * msk = S`
6. Coordinator reads `S` directly from `ckd_output.big_c` — the confidential derived key is fully disclosed without knowing `app_sk`.

pub fn ckd(
    participants: &[Participant],
    coordinator: Participant,
    me: Participant,
    key_pair: KeygenOutput,
    app_id: impl Into<AppId>,
    app_pk: PublicKey,
    rng: impl CryptoRngCore + Send + 'static,

    // not enough participants
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }

    // kick out duplicates
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }
