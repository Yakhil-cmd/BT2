### Title
Gateway stateful validator computes L2 gas price admission threshold with floor rounding, admitting transactions below the intended minimum — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

`StatefulTransactionValidator::validate_tx_l2_gas_price_within_threshold` computes the minimum acceptable L2 gas price using `Ratio::to_integer()`, which performs **floor (truncating) division**. Because the threshold is a lower bound that the transaction's price must meet, it must be rounded **up** (ceiling). Rounding it down produces a threshold that is strictly less than the true mathematical minimum, allowing transactions whose gas price is below the intended floor to pass gateway admission.

### Finding Description

In `crates/apollo_gateway/src/stateful_transaction_validator.rs` lines 367–371:

```rust
let gas_price_threshold_multiplier =
    Ratio::new(self.config.min_gas_price_percentage.into(), 100_u128);
let threshold = (gas_price_threshold_multiplier
    * previous_block_l2_gas_price.get().0)
    .to_integer();          // ← floor division
if tx_l2_gas_price.0 < threshold {
    return Err(...);
}
```

`Ratio::to_integer()` in `num_rational` returns `numer / denom` — integer division, which truncates toward zero (floor) for positive values. The threshold is intended to be the minimum price a transaction must offer: `ceil(min_gas_price_percentage * previous_block_l2_gas_price / 100)`. Using floor instead of ceiling produces a threshold that is 1 FRI lower than the true minimum whenever the product is not exactly divisible by 100.

**Concrete example:**
- `previous_block_l2_gas_price = 101 FRI`
- `min_gas_price_percentage = 10` (10%)
- True threshold = `101 × 10 / 100 = 10.1 FRI` → should require `≥ 11 FRI`
- Computed threshold = `floor(10.1) = 10 FRI`
- A transaction with `tx_l2_gas_price = 10` satisfies `10 >= 10` and is **admitted**, even though `10 < 10.1` — below the intended minimum.

The default production value of `min_gas_price_percentage = 100` (line 296 of `crates/apollo_gateway_config/src/config.rs`) produces an exact integer and is unaffected. Any non-100 percentage (e.g., 80, 50, or any operator-configured value) triggers the rounding error for the majority of gas price values. [1](#0-0) [2](#0-1) 

### Impact Explanation

**High — Mempool/gateway admission accepts invalid transactions before sequencing.**

Transactions whose `max_price_per_unit` for L2 gas is mathematically below the configured minimum threshold (but at or above the floored threshold) bypass the gateway's gas price floor check and are admitted to the mempool. This breaks the invariant that the gateway enforces a minimum gas price equal to `min_gas_price_percentage%` of the previous block's L2 gas price. The sequencer then sequences transactions that should have been rejected at the gateway boundary.

### Likelihood Explanation

The bug is triggered whenever `min_gas_price_percentage × previous_block_l2_gas_price` is not divisible by 100. With `min_gas_price_percentage = 100` (the current default), the product is always divisible by 100 and no rounding error occurs. However, any operator-configured non-100 percentage activates the flaw for the vast majority of gas price values. An attacker who knows the previous block's gas price can craft a transaction with `tx_l2_gas_price = floor(pct × price / 100)` to reliably bypass the check.

### Recommendation

Replace `to_integer()` (floor) with ceiling division. Using `num_rational`'s `ceil()` method:

```rust
let threshold = (gas_price_threshold_multiplier
    * previous_block_l2_gas_price.get().0)
    .ceil()
    .to_integer();   // ceiling, consistent with all other threshold computations
```

This is consistent with the pattern used throughout the codebase for minimum-bound computations, e.g., `sierra_gas_to_l1_gas_amount_round_up` and `convert_l1_to_l2_gas_price_round_up`. [3](#0-2) [4](#0-3) 

### Proof of Concept

1. Configure `min_gas_price_percentage = 10`.
2. Previous block L2 gas price = `101 FRI` (a value not divisible by 10).
3. True threshold = `ceil(101 × 10 / 100)` = `ceil(10.1)` = `11 FRI`.
4. Computed threshold = `floor(10.1)` = `10 FRI`.
5. Submit a transaction with `l2_gas.max_price_per_unit = 10 FRI`.
6. The check `10 < 10` is `false` → transaction is **admitted** despite being below the intended 10% floor.
7. With the fix (`ceil`), the check `10 < 11` is `true` → transaction is **correctly rejected**. [5](#0-4)

    fn validate_tx_l2_gas_price_within_threshold(
        &self,
        tx_resource_bounds: ValidResourceBounds,
        previous_block_l2_gas_price: NonzeroGasPrice,
    ) -> StatefulTransactionValidatorResult<()> {
        match tx_resource_bounds {
            ValidResourceBounds::AllResources(tx_resource_bounds) => {
                let tx_l2_gas_price = tx_resource_bounds.l2_gas.max_price_per_unit;
                let gas_price_threshold_multiplier =
                    Ratio::new(self.config.min_gas_price_percentage.into(), 100_u128);
                let threshold = (gas_price_threshold_multiplier
                    * previous_block_l2_gas_price.get().0)
                    .to_integer();
                if tx_l2_gas_price.0 < threshold {
                    return Err(StarknetError {
                        // We didn't have this kind of an error.
                        code: StarknetErrorCode::UnknownErrorCode(
                            "StarknetErrorCode.GAS_PRICE_TOO_LOW".to_string(),
                        ),
                        message: format!(
                            "Transaction L2 gas price {tx_l2_gas_price} is below the required \
                             threshold {threshold}.",
                        ),
                    });
                }
            }
            ValidResourceBounds::L1Gas(_) => {
                // No validation required for legacy transactions.
            }
        }
        Ok(())
    }
