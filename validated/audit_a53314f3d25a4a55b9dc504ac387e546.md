### Title
Gateway Skips `__validate__` Signature Verification for Nonce-1 Invoke When Deploy-Account Is In Mempool — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator unconditionally bypasses the `__validate__` entry point — the account contract's signature-verification hook — for any invoke transaction whose nonce equals 1 when a deploy-account transaction for the same address is already in the mempool. An attacker who controls any account can exploit this to inject an invoke with an arbitrary or forged signature into the mempool without the gateway ever verifying it.

---

### Finding Description

`skip_stateful_validations` returns `true` (skip validation) when all four conditions hold simultaneously:

```
tx is Invoke
tx.nonce() == Nonce(Felt::ONE)          // nonce = 1
account_nonce == Nonce(Felt::ZERO)      // account not yet deployed on-chain
account_tx_in_pool_or_recent_block(sender) == true
``` [1](#0-0) 

When `skip_stateful_validations` returns `true`, `run_pre_validation_checks` propagates that value back to `extract_state_nonce_and_run_validations`, which passes it to `run_validate_entry_point`. [2](#0-1) 

Inside `run_validate_entry_point`, the flag is translated directly into `execution_flags.validate = false`: [3](#0-2) 

`StatefulValidator::perform_validations` then short-circuits before the `__validate__` call:

```rust
ApiTransaction::Invoke(_) => {
    tx.perform_pre_validation_stage(self.state(), &tx_context)?;
    if !tx.execution_flags.validate {
        return Ok(());   // ← exits here; __validate__ never runs
    }
    let (_optional_call_info, actual_cost) = self.validate(&tx, tx_context.clone())?;
    ...
}
``` [4](#0-3) 

`perform_pre_validation_stage` — the only check that still runs — verifies nonce ordering, fee bounds, balance, and proof facts, but **never touches the signature**: [5](#0-4) 

The fourth condition (`account_tx_in_pool_or_recent_block`) is satisfied by the presence of **any** transaction from the sender in the mempool pool or staged queue — not specifically a deploy-account: [6](#0-5) 

The code comment justifying the skip reads: *"It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."* This reasoning is circular: future-nonce transactions that "passed validations" may themselves have passed only because of this same skip.

---

### Impact Explanation

The gateway admits an invoke transaction carrying an invalid or forged signature into the mempool without any signature check. This directly satisfies:

> **High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.**

The corrupted value is the `signature` field of the admitted `InternalRpcTransaction`: it is stored in the mempool and forwarded to the batcher as if it had been validated, when it has not.

---

### Likelihood Explanation

The trigger is unprivileged. Any Starknet user can:
1. Deploy any account class (including one whose `__validate_deploy__` always succeeds).
2. Submit a deploy-account transaction to the gateway (which passes full validation).
3. Immediately submit an invoke with `nonce=1` and an arbitrary signature.

Step 3 requires no knowledge of the account's private key. The attacker only needs the deploy-account to be in the mempool at the moment the invoke arrives, which is trivially arranged by submitting both in the same batch.

---

### Recommendation

Remove the unconditional skip of `__validate__` for nonce-1 invokes. If the UX goal of accepting a paired deploy-account + invoke before the deploy-account is confirmed is required, the gateway should at minimum perform a stateless ECDSA/Stark-curve signature check against the transaction hash before bypassing the full entry-point validation. Alternatively, the skip should be gated on the deploy-account transaction being present in the mempool **with the same sender address and class hash**, and the invoke's signature should still be verified against the not-yet-deployed account's public key extracted from the deploy-account calldata.

---

### Proof of Concept

```
1. Attacker declares a contract class C whose __validate_deploy__ always returns VALID.
2. Attacker submits DeployAccount(class=C, salt=S) → address A.
   Gateway runs __validate_deploy__ → passes. A is now in the mempool.

3. Attacker submits Invoke(sender=A, nonce=1, calldata=<anything>, signature=[0xff, 0xff]).
   Gateway calls skip_stateful_validations:
     - tx is Invoke                          ✓
     - tx.nonce() == 1                       ✓
     - account_nonce == 0 (A not on-chain)   ✓
     - account_tx_in_pool_or_recent_block(A) ✓  (deploy-account is in pool)
   → returns true → execution_flags.validate = false
   → __validate__ is never called
   → Invoke with signature=[0xff,0xff] is admitted to the mempool.

4. Batcher picks up both transactions.
   DeployAccount executes → A is deployed.
   Invoke executes → __validate__ is called → signature check fails → tx reverts.
   But the mempool accepted and forwarded an invalid transaction.
``` [7](#0-6) [8](#0-7)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
