### Title
Malicious Participant Can Corrupt CKD Output Without Detection — (File: `src/confidential_key_derivation/protocol.rs`)

### Summary
The CKD coordinator blindly aggregates participant contributions without any proof of correct computation. A single malicious participant can send arbitrary `ElementG1` values instead of correctly computed shares, silently corrupting the derived confidential key for all honest parties.

### Finding Description
In `do_ckd_coordinator`, the coordinator receives `(norm_big_y, norm_big_c)` from every other participant and unconditionally adds them to its running sum: [1](#0-0) 

No proof of correct computation accompanies these values. The correct computation, performed in `compute_signature_share`, requires each participant to:
1. Sample a fresh blinding scalar `y_i`
2. Compute `big_y = y_i * G`
3. Compute `big_s = x_i * H(pk || app_id)` using their private share `x_i`
4. Compute `big_c = big_s + y_i * app_pk`
5. Scale both by their Lagrange coefficient `lambda_i` [2](#0-1) 

Nothing in the protocol forces a participant to follow these steps. The coordinator receives raw `CKDOutput` structs containing two `ElementG1` fields and adds them without any algebraic consistency check, zero-knowledge proof, or commitment binding the contribution to the participant's public key share. [3](#0-2) 

The participant path is equally unchecked — it simply sends whatever it computes (or fabricates) directly to the coordinator: [4](#0-3) 

### Impact Explanation
The intended invariant is that the coordinator's final output satisfies:

```
C − a·Y  =  Σ λᵢ·xᵢ · H(pk ∥ app_id)  =  msk · H(pk ∥ app_id)
```

A malicious participant who sends `(big_y′, big_c′)` of their choice breaks this invariant. The coordinator returns a `CKDOutput` whose `unmask(app_sk)` call yields an arbitrary group element unrelated to `msk`. Every honest party that relies on this output — including the application calling `ckd_output.unmask` — accepts a cryptographically invalid derived key. [5](#0-4) 

This maps to **High: Corruption of CKD outputs so honest parties accept inconsistent or unusable cryptographic outputs**.

### Likelihood Explanation
The attack requires only that one participant in the protocol is malicious. No cryptographic capability, leaked key, or external assumption is needed. The attacker simply substitutes arbitrary `ElementG1` values for their correctly computed share. Because the coordinator performs no verification, the attack is undetectable and always succeeds.

### Recommendation
Each participant should accompany their `(norm_big_y, norm_big_c)` with a zero-knowledge proof of correct formation — specifically, a proof that:
- `norm_big_y = lambda_i * y_i * G` for some `y_i`
- `norm_big_c = lambda_i * (x_i * H(pk ∥ app_id) + y_i * app_pk)` where `x_i` is consistent with the participant's public key share

A standard sigma protocol (e.g., a Chaum–Pedersen proof of discrete-log equality) can establish the relationship between `norm_big_y`, `norm_big_c`, and the participant's committed public key share without revealing `x_i` or `y_i`.

### Proof of Concept
1. Honest participants run `ckd(participants, coordinator, me, key_pair, app_id, app_pk, rng)`.
2. Malicious participant `P_bad` intercepts the protocol at `do_ckd_participant` and instead of calling `compute_signature_share`, sends `(ElementG1::generator(), ElementG1::generator())` to the coordinator.
3. The coordinator executes the loop at lines 50–55 of `protocol.rs`, adding the fabricated values to its running sum.
4. The returned `CKDOutput` satisfies `big_c ≠ msk·H(pk ∥ app_id) + a·big_y`.
5. The application calls `ckd_output.unmask(app_sk)` and receives a value that is not `msk·H(pk ∥ app_id)`, rendering the confidential derived key unusable. [6](#0-5)

async fn do_ckd_coordinator(
    mut chan: SharedChannel,
    participants: ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<CKDOutputOption, ProtocolError> {
    let (mut norm_big_y, mut norm_big_c) =
        compute_signature_share(&participants, me, key_pair, app_id, app_pk, rng)?;

    // Receive everyone's inputs and add them together
    let waitpoint = chan.next_waitpoint();

    for (_, participant_output) in
        recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
    {
        norm_big_y += participant_output.big_y();
        norm_big_c += participant_output.big_c();
    }
    let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
    Ok(Some(ckd_output))
