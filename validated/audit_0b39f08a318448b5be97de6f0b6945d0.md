### Title
`addLiquidity` Not Blocked When Pool Is Paused, Allowing LP Deposits Into a Compromised Pool — (File: `metric-core/contracts/MetricOmmPool.sol`)

---

### Summary

The `addLiquidity` function in `MetricOmmPool` lacks the `whenNotPaused` modifier, allowing liquidity providers to deposit tokens into a pool even when it has been paused by the admin or protocol. Only `swap` enforces the pause guard. This is the direct Metric OMM analog of the ERC-4626 `maxDeposit`-when-paused bug: a deposit-path function ignores the disabled-state signal entirely.

---

### Finding Description

`MetricOmmPool` defines a `whenNotPaused` modifier backed by `_checkNotPaused()`, which reverts when `pauseLevel != 0`. [1](#0-0) 

The `swap` function correctly applies this modifier: [2](#0-1) 

However, `addLiquidity` carries only `nonReentrant` — the `whenNotPaused` guard is absent: [3](#0-2) 

The pool supports two pause levels: `1` (admin-initiated) and `2` (protocol-initiated). Both are intended to halt pool activity during emergencies such as a compromised or stale price provider. [4](#0-3) 

The factory's `pausePool` and `protocolPausePool` paths enforce valid pause transitions but have no mechanism to prevent `addLiquidity` from succeeding after the pause is set: [5](#0-4) 

---

### Impact Explanation

When a pool is paused — typically because the price provider is suspected to be compromised, stale, or manipulated — swaps are correctly blocked. However, an LP (or a front-running bot) can still call `addLiquidity` and deposit real tokens into the pool. Those tokens are immediately credited to bins at the current (potentially corrupted) bin position. When the pool is later unpaused, the deposited funds are exposed to the same exploit that triggered the pause in the first place, resulting in direct loss of LP principal. The impact is **Medium**: fund loss is conditional on the pool being unpaused after a security event, but the deposit path is fully reachable by any unprivileged caller with no special setup.

---

### Likelihood Explanation

**Medium.** Any pool admin or protocol owner can pause a pool. The missing guard is reachable by any address on every paused pool. The scenario (pause due to oracle issue → LP deposits → unpause → exploit) is a realistic operational sequence, not a contrived edge case.

---

### Recommendation

Add the `whenNotPaused` modifier to `addLiquidity`, consistent with how `swap` is protected:

```solidity
function addLiquidity(
    address owner,
    uint80 salt,
    LiquidityDelta calldata deltas,
    bytes calldata callbackData,
    bytes calldata extensionData
) external whenNotPaused nonReentrant(PoolActions.ADD_LIQUIDITY) returns (...) {
```

`removeLiquidity` intentionally omitting `whenNotPaused` is defensible (LPs must be able to exit), but `addLiquidity` has no analogous justification for bypassing the pause.

---

### Proof of Concept

1. Pool is deployed with a mutable price provider (timelock > 0).
2. The price provider begins returning a manipulated price. The pool admin calls `factory.pausePool(pool)`, setting `pauseLevel = 1`.
3. An LP (or attacker-controlled account) calls `pool.addLiquidity(...)` with a large token deposit. The call succeeds — no revert — because `addLiquidity` has no `whenNotPaused` check.
4. The LP's tokens are credited to bins at the current (manipulated) bin position.
5. The admin investigates, concludes the issue is resolved, and calls `factory.unpausePool(pool)`, setting `pauseLevel = 0`.
6. An attacker immediately calls `pool.swap(...)` against the manipulated price, draining the LP's newly deposited tokens.

The root cause is solely in `MetricOmmPool.addLiquidity` at line 188 — no privileged action, no malicious token, no off-chain dependency required beyond the normal pause/unpause lifecycle. [3](#0-2)
