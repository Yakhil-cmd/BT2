### Title
`skip_stateful_validations` admits invoke transactions with invalid signatures when any pending transaction exists for the sender — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The gateway's `skip_stateful_validations` function is intended to skip `__validate__` for an invoke with nonce=1 only when a `deploy_account` is pending for that account. However, the check it uses — `account_tx_in_pool_or_recent_block` — returns `true` for **any** transaction from the sender address in the pool, not exclusively a `deploy_account`. This allows an attacker to submit an invoke transaction with an **invalid signature** from any account that has nonce=0 on-chain and any pending transaction in the mempool, bypassing the gateway's signature verification entirely.

### Finding Description

The gateway stateful validation path in `extract_state_nonce_and_run_validations` calls `run_pre_validation_checks`, which calls `skip_stateful_validations`:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs:429-461
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            // ...
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                ...
        }
    }
    Ok(false)
}
``` [1](#0-0) 

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` sets `validate: false` in the execution flags, meaning `__validate__` (the account's signature-checking entry point) is **never called**:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs:310-312
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [2](#0-1) 

The condition for skipping is: incoming invoke has `nonce == 1`, on-chain account nonce is `0`, and `account_tx_in_pool_or_recent_block` returns `true`. The mempool implements this check as:

```rust
// crates/apollo_mempool/src/mempool.rs:697-700
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [3](#0-2) 

`tx_pool.contains_account` returns `true` for **any** transaction from that address — including a regular invoke with nonce=0 — not only a `deploy_account`. The code comment claims "it means that either it has a deploy_account transaction or transactions with future nonces that passed validations," but a pending invoke with nonce=0 is neither. [4](#0-3) 

The mempool's `validate_tx` (called via `validate_by_mempool` before `skip_stateful_validations`) only checks nonce ordering, duplicate hashes, and fee escalation — it does **not** verify the signature:

```rust
// crates/apollo_mempool/src/mempool.rs:402-408
pub fn validate_tx(&mut self, args: ValidationArgs) -> MempoolResult<()> {
    let tx_reference = (&args).into();
    self.validate_incoming_tx(tx_reference, args.account_nonce)?;
    self.validate_fee_escalation(tx_reference)?;
    Ok(())
}
``` [5](#0-4) 

`ValidationArgs` carries only `address`, `account_nonce`, `tx_hash`, `tx_nonce`, `tip`, and `max_l2_gas_price` — no signature field: [6](#0-5) 

**Attack scenario:**

1. Alice has a deployed account with on-chain nonce=0 (never sent a transaction).
2. Alice submits a valid invoke with nonce=0 to the gateway; it passes all checks and enters the mempool.
3. Attacker submits an invoke from Alice's address with nonce=1 and an **arbitrary/invalid signature**.
4. Gateway: `validate_nonce` passes (0 ≤ 1 ≤ max\_allowed\_nonce\_gap).
5. Gateway: `validate_by_mempool` passes (nonce=1 is not a duplicate, not too old).
6. Gateway: `skip_stateful_validations` returns `true` because `tx_pool.contains_account(Alice)` is `true` (Alice's nonce=0 tx is in the pool).
7. Gateway: `run_validate_entry_point` runs with `validate: false` — `__validate__` is **not called**.
8. The attacker's transaction with invalid signature is admitted to the mempool.

### Impact Explanation

An unprivileged attacker can inject transactions with invalid signatures into the mempool for any account that has nonce=0 on-chain and any pending transaction in the pool. This satisfies the **High** impact category: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

Concretely:
- The mempool is polluted with signature-invalid transactions that consume capacity (`capacity_in_bytes`), potentially causing legitimate transactions to be evicted or rejected.
- The attacker pays no fees (transactions are rejected during blockifier execution, not reverted, so no fee is charged).
- The attack is repeatable at low cost: the attacker only needs to observe the public mempool for eligible accounts.

### Likelihood Explanation

The condition is easily met: any newly deployed account (nonce=0) that submits its first transaction creates a window during which an attacker can inject a nonce=1 transaction with an invalid signature. This is a common pattern (e.g., a user deploying an account and immediately sending a transaction). The mempool is public, so eligible accounts are trivially discoverable.

### Recommendation

In `skip_stateful_validations`, replace the generic `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a **`deploy_account` transaction** exists for the sender address in the mempool. Alternatively, add a dedicated `deploy_account_in_pool(address)` query to the mempool API that only returns `true` when a `DeployAccount` transaction is present for that address.

### Proof of Concept

```
1. Deploy account A on-chain (nonce becomes 0 after deployment, or use a pre-funded
   counterfactual address with nonce=0).
2. Submit a valid invoke from A with nonce=0 to the gateway. It is admitted.
3. Craft an invoke from A with nonce=1, arbitrary calldata, and a random/invalid
   signature (e.g., all-zero signature bytes).
4. Submit the crafted invoke to the gateway.
   - validate_nonce: passes (0 <= 1 <= max_allowed_nonce_gap)
   - validate_by_mempool: passes (nonce=1 not duplicate, not too old)
   - skip_stateful_validations: returns true (tx_pool.contains_account(A) == true)
   - run_validate_entry_point: validate=false, __validate__ NOT called
   - Transaction admitted to mempool.
5. Repeat for every account observed in the mempool with nonce=0 on-chain.
   Each iteration consumes mempool capacity at zero cost to the attacker.
``` [7](#0-6) [1](#0-0)

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
