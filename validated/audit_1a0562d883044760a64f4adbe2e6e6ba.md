### Title
Missing Lower-Bound Validation on `threshold` in FROST Signing Allows Corrupted Signature Output - (`File: src/frost/mod.rs`)

---

### Summary

`assert_sign_inputs` in `src/frost/mod.rs` validates that `threshold ≤ participants.len()` but never checks that `threshold ≥ 2`. A caller (or malicious coordinator) can pass `threshold = 1` (or `0`) to `sign_v1` / `sign_v2` in `src/frost/eddsa/sign.rs`, which is accepted without error. The signing protocol then runs to completion but produces a cryptographically invalid signature, permanently denying honest parties a usable signing output.

---

### Finding Description

`assert_key_invariants` in `src/dkg.rs` enforces `threshold ≥ 2` at key-generation time:

```rust
if threshold < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
}
``` [1](#0-0) 

`validate_triple_inputs` in the OT-based triple generation enforces the same lower bound:

```rust
if threshold_value < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold: threshold_value, min: 2 });
}
``` [2](#0-1) 

However, `assert_sign_inputs` — the shared validation gate for all FROST signing entry points — only checks the upper bound:

```rust
// validate threshold
if threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
``` [3](#0-2) 

There is **no** corresponding `threshold < 2` check. `sign_v1` and `sign_v2` both delegate entirely to `assert_sign_inputs` for threshold validation: [4](#0-3) [5](#0-4) 

The accepted `threshold` value is then forwarded directly into `construct_key_package`, which converts it to `u16` and passes it as `min_signers` to the underlying `frost_core::KeyPackage`:

```rust
u16::try_from(threshold.value()).map_err(|_| { ... })
``` [6](#0-5) 

The same gap exists in `src/frost/mod.rs`'s `presign` function, which also only checks the upper bound on `args.threshold`: [7](#0-6) 

And in `src/ecdsa/ot_based_ecdsa/sign.rs`, which checks `participants.len() < threshold` but never `threshold < 2`: [8](#0-7) 

---

### Impact Explanation

When `threshold = 1` is supplied to `sign_v1` or `sign_v2`:

1. `assert_sign_inputs` passes without error.
2. Each participant constructs a `KeyPackage` with `min_signers = 1`.
3. The FROST `aggregate` call accepts a signing package containing only one signature share.
4. Lagrange interpolation with a single share evaluates to that share directly (coefficient = 1), reconstructing `x_i` rather than the actual aggregate secret key `x`.
5. The resulting signature is computed over the wrong key and **fails verification** against the public key established during DKG.

The signing protocol completes without returning any error, but every honest party that relies on the output receives an unusable, invalid signature. This constitutes **corruption of signing output** and **permanent denial of signing** for honest parties under valid protocol inputs.

**Allowed impact matched:** *High: Corruption of sign outputs so honest parties accept inconsistent or unusable cryptographic outputs.*

---

### Likelihood Explanation

The `threshold` parameter is caller-supplied at every signing invocation. A malicious coordinator who controls the `threshold` argument to `sign_v1` / `sign_v2` can trivially pass `1` (or `0`). Because the library documents that the coordinator role is trusted only for liveness, not for correctness of outputs, a coordinator that corrupts the threshold silently breaks signing for all participants without any detectable protocol-level error. The entry path requires no special privilege beyond being a library caller.

---

### Recommendation

Add the same lower-bound guard that `assert_key_invariants` and `validate_triple_inputs` already enforce, inside `assert_sign_inputs` in `src/frost/mod.rs`:

```rust
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

Apply the equivalent check to `presign` in `src/frost/mod.rs` and to `sign` in `src/ecdsa/ot_based_ecdsa/sign.rs`.

---

### Proof of Concept

```
1. Run DKG with participants = [P1, P2, P3], threshold = 2.
   → Each party holds a degree-1 polynomial share x_i.

2. Call sign_v1(
       participants = [P1, P2, P3],
       threshold    = 1,          // ← below minimum, not rejected
       me           = P1,
       coordinator  = P1,
       keygen_output = <output from step 1>,
       message      = b"hello",
       rng          = ...,
   )
   → assert_sign_inputs passes (1 ≤ 3, no lower-bound check).
   → construct_key_package builds KeyPackage with min_signers = 1.
   → aggregate runs with 1 share, Lagrange coefficient = 1.
   → Signature produced uses x_1 as the secret key, not x.
   → Signature verification against the DKG public key fails.
   → Honest parties receive an unusable signature with no protocol error.
```
