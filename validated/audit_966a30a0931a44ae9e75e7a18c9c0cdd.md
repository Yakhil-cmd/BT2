### Title
Missing Access Control on `update_rewards` Allows Any Caller to Permanently Freeze Consensus Block Rewards - (File: src/staking/staking.cairo)

### Summary

The `update_rewards` function in the Staking contract is intended to be callable only by the Starkware sequencer (per the protocol specification), but the implementation contains no such access control check. Any unprivileged address can call `update_rewards` with `disable_rewards: true` to consume the single allowed reward-update slot per block without distributing any rewards, permanently denying consensus block rewards to all stakers.

### Finding Description

The protocol specification at `docs/spec.md` lines 1644–1645 explicitly states:

```
#### access control
Only starkware sequencer.
```

However, the implementation of `StakingRewardsManagerImpl::update_rewards` in `src/staking/staking.cairo` performs no such check:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();   // only checks: not paused, caller != zero
    let current_block_number = starknet::get_block_number();
    assert!(
        current_block_number > self.last_reward_block.read(),
        "{}",
        Error::REWARDS_ALREADY_UPDATED,
    );
    ...
    // Update last block rewards — happens unconditionally.
    self.last_reward_block.write(current_block_number);

    if disable_rewards || self.is_pre_consensus() {
        return;   // exits without distributing any rewards
    }
    ...
}
``` [1](#0-0) 

`general_prerequisites()` only asserts the contract is not paused and the caller is non-zero — it does **not** restrict the caller to the sequencer. [2](#0-1) 

The global `last_reward_block` is written **before** the `disable_rewards` branch, meaning a call with `disable_rewards: true` still consumes the block's single update slot while distributing nothing. [3](#0-2) 

The spec documents this one-slot-per-block invariant as a precondition:

```
#### pre-condition
Rewards did not distribute for the current block yet.
``` [4](#0-3) 

### Impact Explanation

An attacker calls `update_rewards(any_valid_active_staker, disable_rewards: true)` once per block. This:

1. Writes `last_reward_block = current_block_number` — consuming the slot.
2. Returns immediately without distributing any STRK or BTC block rewards.
3. Causes every subsequent call in the same block (including the legitimate sequencer call) to revert with `REWARDS_ALREADY_UPDATED`.

Repeated across every block, this permanently freezes all consensus block rewards for every staker and delegator in the protocol. No staker can ever accumulate `unclaimed_rewards_own` via the consensus path, and no pool ever receives `update_rewards_from_staking_contract`. This constitutes **permanent freezing of unclaimed yield** — a High-severity impact under the allowed scope. [5](#0-4) 

### Likelihood Explanation

The entry point is a public, permissionless function. The attacker needs only a valid (active, non-zero-balance) staker address — trivially obtained from on-chain events — and enough gas to submit one transaction per block. There is no financial cost beyond gas. The attack is sustainable indefinitely and requires no privileged access, leaked keys, or external dependencies.

### Recommendation

Add a sequencer-only access control guard at the top of `update_rewards`, consistent with the specification. For example, introduce a stored `sequencer_address` (or reuse an existing role) and assert:

```cairo
assert!(
    get_caller_address() == self.sequencer_address.read(),
    "{}",
    Error::CALLER_IS_NOT_SEQUENCER,
);
```

Alternatively, if the function must remain callable by multiple parties, move `self.last_reward_block.write(current_block_number)` to **after** the `disable_rewards` guard so that a no-op call does not consume the block's reward slot.

### Proof of Concept

```
Block N:
  Attacker tx:  update_rewards(alice_staker, disable_rewards=true)
    → last_reward_block := N
    → returns without distributing rewards

  Sequencer tx: update_rewards(alice_staker, disable_rewards=false)
    → assert!(N > N)  ← FAILS with REWARDS_ALREADY_UPDATED

Block N+1:
  Attacker repeats → same outcome

Result: stakers accumulate zero consensus rewards indefinitely.
```

The flow test at `src/flow_test/test.cairo` lines 2882–2895 already demonstrates that calling with `disable_rewards: true` produces zero rewards and that a subsequent call in the same block panics with `REWARDS_ALREADY_UPDATED` — confirming the mechanism. The missing piece (no access control) means any address, not just the sequencer, can trigger this path. [6](#0-5) [7](#0-6)
