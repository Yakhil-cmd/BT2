### Title
Missing NEP-141 Registry Validation in `handle_precompile_promise` Panics and Blocks All New-User XCC Calls — (`engine/src/xcc.rs`)

---

### Summary

`handle_precompile_promise` in `engine/src/xcc.rs` calls `.expect("wnear account not found")` when looking up the NEP-141 account ID for the stored wNEAR ERC-20 address. If `factory_set_wnear_address` was called to set the wNEAR ERC-20 address but that address was never registered in the `nep141_erc20_map` (via `register_token`), the lookup returns `None` and the NEAR contract panics. Because this code path is reached during post-EVM log processing for every first-time XCC caller (any EVM address whose router has not yet been deployed), the entire XCC precompile is permanently blocked for new users until an admin corrects the registry state.

---

### Finding Description

`factory_set_wnear_address` stores a raw ERC-20 address with no requirement that the address already has a corresponding NEP-141 entry in the bidirectional `nep141_erc20_map`: [1](#0-0) 

Later, when any EVM user calls the XCC precompile for the first time (triggering router deployment), `filter_promises_from_logs` in `engine/src/engine.rs` calls `handle_precompile_promise`: [2](#0-1) 

Inside `handle_precompile_promise`, when `AddressVersionStatus::DeployNeeded` is matched (i.e., the sender has no deployed router), the code reads the stored wNEAR address and then performs a registry lookup with a hard `.expect`: [3](#0-2) 

If the wNEAR ERC-20 address is set but not present in the `nep141_erc20_map`, `lookup_right` returns `None` and `.expect("wnear account not found")` panics the NEAR contract.

The sibling function `fund_xcc_sub_account` handles the identical lookup gracefully with `.ok_or(FundXccError::MissingWNearAddress)?`, proving the panic is not intentional: [4](#0-3) 

The `nep141_erc20_map` lookup itself is a safe `Option`-returning call: [5](#0-4) 

`factory_set_wnear_address` imposes no precondition that the address is already registered: [6](#0-5) 

---

### Impact Explanation

When `handle_precompile_promise` panics, the NEAR runtime rolls back the entire transaction. Every EVM user whose XCC router has not yet been deployed will trigger this code path on their first XCC call. The result is that **all new XCC callers are permanently blocked** from using the cross-contract call precompile until an admin corrects the registry. Existing users with already-deployed routers (`AddressVersionStatus::UpToDate`) are unaffected, but the XCC feature is effectively frozen for any new participant.

Impact: **High — Temporary freezing of funds/functionality** (XCC precompile DoS for all new users).

---

### Likelihood Explanation

The triggering condition — wNEAR ERC-20 address set but not registered in the NEP-141 map — can arise from:

1. An operator calling `factory_set_wnear_address` before `deploy_erc20_from_nep141` completes the registration.
2. A new Aurora Engine instance configured with `factory_set_wnear_address` without the corresponding `register_token` step.
3. Any future migration or upgrade that resets the NEP-141 map without resetting the wNEAR address pointer.

The entry point is fully unprivileged: any EVM address can call the XCC precompile at `0x516cded1d16af10cad47d6d49128e2eb7d27b372`. [7](#0-6) 

Likelihood: **Medium** — requires a specific misconfiguration, but the misconfiguration is easy to introduce and the attack surface is open to all EVM users.

---

### Recommendation

Mirror the graceful error handling already present in `fund_xcc_sub_account`. Replace the `.expect` in `handle_precompile_promise` with a recoverable error path, and enforce in `factory_set_wnear_address` that the supplied address already has a NEP-141 entry in the map before storing it — analogous to the original report's recommendation to validate oracle presence before adding a lender.

---

### Proof of Concept

1. Deploy Aurora Engine and call `factory_update` to set XCC router bytecode.
2. Deploy a wNEAR ERC-20 contract but **skip** calling `register_token` / `deploy_erc20_from_nep141` (so the NEP-141 map has no entry for the wNEAR ERC-20 address).
3. Call `factory_set_wnear_address` with the wNEAR ERC-20 address. This succeeds because no validation is performed.
4. From any EVM address that has never used XCC before, submit an EVM transaction that calls the XCC precompile (`CrossContractCallArgs::Eager` or `Delayed`).
5. The XCC precompile's `run_with_handle` succeeds and emits a log with `AMOUNT_TOPIC`.
6. `filter_promises_from_logs` calls `handle_precompile_promise`.
7. `AddressVersionStatus::DeployNeeded { create_needed: true }` is matched (no router exists for this sender).
8. `nep141_erc20_map(*io).lookup_right(&ERC20Address(wnear_address))` returns `None`.
9. `.expect("wnear account not found")` panics — the NEAR transaction is rolled back and the XCC call fails.
10. Every subsequent first-time XCC caller hits the same panic. The XCC feature is blocked for all new users. [8](#0-7) [9](#0-8)
