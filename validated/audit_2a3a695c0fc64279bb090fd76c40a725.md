Audit Report

## Title
ckERC20 Withdrawal with Insufficient `max_transaction_fee` Permanently Blocks All Subsequent Withdrawal Finalizations - (`rs/ethereum/cketh/minter/src/state/transactions/mod.rs`)

## Summary

When a ckERC20 withdrawal transaction's user-supplied `max_transaction_fee` cap is exceeded during gas-price-driven resubmission, `create_resubmit_transactions` pushes an `InsufficientTransactionFee` error and immediately returns, leaving the transaction permanently stuck in `sent_tx`. Because Ethereum enforces sequential nonce ordering and `sent_transactions_to_finalize` only considers nonces strictly below the on-chain finalized count, every subsequent withdrawal (nonces N+1, N+2, …) is blocked from finalization until gas prices fall below the stuck cap or a canister upgrade is performed. Any unprivileged user can trigger this condition by submitting a ckERC20 withdrawal with a `max_transaction_fee` that is valid at submission time but insufficient during a subsequent gas spike.

## Finding Description

**Root cause — early return in `create_resubmit_transactions`:**
In `rs/ethereum/cketh/minter/src/state/transactions/mod.rs` at lines 618–630, when `last_signed_tx.resubmit(current_gas_fee)` returns `Err(InsufficientTransactionFee)` for nonce N, the function pushes the error and immediately `return`s. Nonces N+1, N+2, … are never processed. The code comment at lines 589–592 explicitly acknowledges this: *"We stop on the first error since if a transaction with nonce n could not be resubmitted… then the next transactions with nonces n+1, n+2, … are blocked anyway."*

**ckERC20-specific fee cap — `GuaranteeEthAmount` strategy:**
In `record_created_transaction` (lines 534–536), ckERC20 withdrawals are assigned `ResubmissionStrategy::GuaranteeEthAmount { allowed_max_transaction_fee: ckerc20.max_transaction_fee }`. In `SignedTransactionRequest::resubmit` (`rs/ethereum/cketh/minter/src/tx.rs`, lines 169–173), if `new_tx_price.max_transaction_fee() > self.resubmission.allowed_max_transaction_fee()`, the function returns `Err(InsufficientTransactionFee)`. Unlike ckETH withdrawals (which use `ReduceEthAmount` and can always absorb higher fees by reducing the sent amount), ckERC20 withdrawals are hard-capped at the user-supplied value.

**Caller silently discards the error:**
`resubmit_transactions_batch` in `rs/ethereum/cketh/minter/src/withdraw.rs` (lines 242–244) matches the `Err(e)` arm with only `log!(INFO, "Failed to resubmit transaction: {e:?}")` and no further action. The stuck transaction remains in `sent_tx` with its original low fee cap indefinitely.

**Finalization gated on sequential nonce advancement:**
`sent_transactions_to_finalize` (lines 661–664) filters `sent_tx` to only nonces `< finalized_transaction_count`. Because Ethereum will not mine nonce N+1 before nonce N, the on-chain finalized count never advances past N. Every subsequent withdrawal in `sent_tx` is permanently excluded from finalization.

**No cancel/quarantine path exists:**
The `reschedule_withdrawal_request` mechanism (line 290 of `withdraw.rs`) only applies at the `create_transaction` stage, before a transaction is sent. Once a transaction is in `sent_tx`, there is no admin endpoint, no cancel function, and no quarantine path. Resolution requires a canister upgrade.

## Impact Explanation

This is a **High** impact: application/platform-level DoS on the ckETH/ckERC20 withdrawal system with concrete user harm. A single malicious or unlucky ckERC20 withdrawal permanently halts finalization of all subsequent withdrawals for all users until gas prices recover or an out-of-band canister upgrade is deployed. Users whose withdrawals are queued behind the stuck nonce cannot receive their ETH or ERC-20 tokens for an indefinite period. This matches the allowed impact class: *"Significant Chain Fusion, ck-token… security impact with concrete user or protocol harm"* and *"Application/platform-level DoS… not based on raw volumetric DDoS."*

## Likelihood Explanation

Any unprivileged user can call `withdraw_erc20` with a `max_transaction_fee` set just above the current gas estimate. No special privileges, no threshold corruption, and no external oracle manipulation are required. Ethereum gas prices are historically volatile (5–10× spikes within hours are common). An adversary can deliberately set a minimal `max_transaction_fee` (just enough to pass the initial `create_transaction` check) and wait for natural congestion or time the submission to coincide with a known high-activity event. The attack is cheap, repeatable, and requires no victim interaction beyond the normal withdrawal flow.

## Recommendation

1. **Add a cancel-and-reimburse path for stuck nonces.** When `create_resubmit_transactions` returns `InsufficientTransactionFee` for nonce N, the minter should burn the nonce by sending a zero-value self-transfer at the current gas price (funded from the minter's ETH reserve), remove the entry from `sent_tx`, and reimburse the user's ckERC20 tokens. This unblocks nonces N+1, N+2, ….
2. **Expose an operator-callable `cancel_withdrawal` endpoint** that performs the same nonce-burn and reimbursement under NNS governance control.
3. **Enforce a minimum `max_transaction_fee` multiplier** at `withdraw_erc20` submission time (e.g., a configurable multiple of the current gas estimate) to reduce the probability of a fee cap being breached during normal volatility.

## Proof of Concept

1. Ethereum mainnet gas price is 10 gwei. User calls `withdraw_erc20` with `max_transaction_fee` = 650 000 gwei (barely covers 10 gwei × 65 000 gas limit). The minter creates a transaction with nonce N via `record_created_transaction` (lines 521–526), assigns `GuaranteeEthAmount { allowed_max_transaction_fee: 650_000 gwei }`, and sends it.
2. Gas price spikes to 15 gwei. The minter's timer calls `resubmit_transactions_batch` → `create_resubmit_transactions(latest_tx_count, new_gas_fee)`. For nonce N, `SignedTransactionRequest::resubmit` computes `new_tx_price.max_transaction_fee()` = 975 000 gwei > 650 000 gwei = `allowed_max_transaction_fee`, returns `Err(InsufficientTransactionFee)` (tx.rs lines 169–173). `create_resubmit_transactions` pushes the error and `return`s (mod.rs lines 622–630).
3. `resubmit_transactions_batch` logs the error and does nothing (withdraw.rs lines 242–244). Nonce N remains in `sent_tx` with the original 10-gwei fee cap.
4. All subsequent nonces N+1, N+2, … are in `sent_tx` but `sent_transactions_to_finalize` (mod.rs lines 661–664) filters them out because `finalized_transaction_count` is still N.
5. Every subsequent timer tick re-runs `resubmit_transactions_batch`, hits the same error for nonce N, and exits. No subsequent withdrawal is ever finalized.

**Reproducible test plan:** Write an integration test using PocketIC that (a) submits two ckERC20 withdrawals, (b) advances the simulated gas price above the first withdrawal's `max_transaction_fee`, (c) triggers the resubmit timer, and (d) asserts that the second withdrawal's `sent_transactions_to_finalize` result is empty despite the second withdrawal having a sufficient fee cap.
