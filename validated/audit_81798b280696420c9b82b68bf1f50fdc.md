### Title
Malicious CKD Participant Can Corrupt Coordinator's Confidential Key Output Without Detection — (File: `src/confidential_key_derivation/protocol.rs`)

### Summary

The CKD coordinator aggregates participant shares without any cryptographic verification of their correctness. A single malicious participant can send arbitrary `(big_y, big_c)` values, causing the coordinator to compute and return a corrupted `CKDOutput` as `Ok(Some(...))` — indistinguishable from a valid result. This is the direct analog of the THORChain event-emission bug: the protocol reports success and produces output even when the underlying computation has been silently corrupted by a malicious party.

### Finding Description

In `do_ckd_coordinator` (`src/confidential_key_derivation/protocol.rs`, lines 35–57), the coordinator:

1. Computes its own share `(norm_big_y, norm_big_c)` correctly via `compute_signature_share`.
2. Receives `CKDOutput` shares from all other participants via `recv_from_others` (line 50–55).
3. Adds each received `big_y()` and `big_c()` into its running totals **without any verification**.
4. Constructs `CKDOutput::new(norm_big_y, norm_big_c)` and returns `Ok(Some(ckd_output))`. [1](#0-0) 

There is no ZK proof, commitment check, or consistency check on the received `(big_y, big_c)` pairs. Compare this to `compute_signature_share` (lines 148–182), which shows what a correct share looks like:

- `big_y = λ_i · y_i · G`
- `big_c = λ_i · (x_i · H(pk, app_id) + y_i · app_pk)` [2](#0-1) 

A malicious participant sends `(big_y + Δ_y, big_c + Δ_c)` for any chosen `Δ_y, Δ_c`. The coordinator computes:

```
Y'  = Y  + Δ_y
C'  = C  + Δ_c
```

After `unmask(app_sk)`, the derived key becomes:

```
C' − app_sk · Y' = msk · H(pk, app_id) + (Δ_c − app_sk · Δ_y)
```

which is wrong for any `Δ_c ≠ app_sk · Δ_y`. The coordinator has no way to detect this and returns `Ok(Some(ckd_output))` as if the protocol succeeded.

Contrast this with the DKG protocol (`src/dkg.rs`), which performs explicit `verify_proof_of_knowledge` and `validate_received_share` checks before accepting any participant contribution: [3](#0-2) 

No equivalent verification exists in the CKD path.

### Impact Explanation

A single malicious participant causes the coordinator to silently produce a corrupted `CKDOutput`. The coordinator returns `Ok(Some(...))` — a success result — with a wrong confidential key. Any downstream consumer of the CKD output (e.g., an application decrypting data or deriving child keys) will use the wrong key, making the output cryptographically unusable or attacker-biased. This matches:

> **High: Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.**

### Likelihood Explanation

Any single participant in the CKD protocol can trigger this. No privileged access, leaked keys, or external assumptions are required. The attacker simply deviates from the protocol by sending crafted `(big_y, big_c)` values. The coordinator has no mechanism to distinguish a malicious share from an honest one.

### Recommendation

Add a zero-knowledge proof of correct share formation. Each participant must prove that their submitted `(big_y, big_c)` satisfies:

- `big_y = y_i · G` for some `y_i`
- `big_c = x_i · H(pk, app_id) + y_i · app_pk` for the participant's committed secret share `x_i`

This is a standard ElGamal correctness proof (a Chaum-Pedersen / dlog-equality proof over the pair `(G, app_pk)` with public values `(big_y, big_c − x_i · H(pk, app_id))`). The coordinator should verify each proof before adding the share into the running aggregate, and abort if any proof fails — analogous to how `verify_proof_of_knowledge` gates share acceptance in the DKG protocol. [4](#0-3) 

### Proof of Concept

A malicious participant replaces the honest `chan.send_private(waitpoint, coordinator, &(norm_big_y, norm_big_c))` call with:

```rust
// Δ is any non-zero element of G1
let delta = ElementG1::generator(); // attacker-chosen corruption
chan.send_private(waitpoint, coordinator, &(norm_big_y, norm_big_c + delta))?;
```

The coordinator receives this, adds `delta` into `norm_big_c`, and returns:

```rust
CKDOutput::new(Y, C + delta)  // wrong, returned as Ok(Some(...))
```

After `unmask(app_sk)`, the caller obtains `msk · H(pk, app_id) + delta` instead of `msk · H(pk, app_id)`. The coordinator reports success; the corruption is undetectable. [5](#0-4)

async fn do_ckd_coordinator(
    mut chan: SharedChannel,
    participants: ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<CKDOutputOption, ProtocolError> {
    let (mut norm_big_y, mut norm_big_c) =
        compute_signature_share(&participants, me, key_pair, app_id, app_pk, rng)?;

    // Receive everyone's inputs and add them together
    let waitpoint = chan.next_waitpoint();

    for (_, participant_output) in
        recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
    {
        norm_big_y += participant_output.big_y();
        norm_big_c += participant_output.big_c();
    }
    let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
    Ok(Some(ckd_output))
