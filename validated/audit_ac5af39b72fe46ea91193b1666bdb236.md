### Title
Pool Contract `claim_rewards` and `change_reward_address` Bypass Staking Pause Mechanism - (`src/pool/pool.cairo`)

### Summary
The staking contract implements a pause mechanism that guards all its state-changing functions via `general_prerequisites()`. However, the pool contract (`src/pool/pool.cairo`) has no awareness of the staking contract's pause state. Pool members can call `claim_rewards` and `change_reward_address` on the pool contract even when the staking contract is paused, allowing reward drainage and state mutation during an emergency freeze.

### Finding Description

The staking contract stores a pause flag in `is_paused: bool` storage and exposes it via `IStakingPause`. Every state-changing function in the staking contract calls `self.general_prerequisites()` at its entry point, which enforces the pause. [1](#0-0) [2](#0-1) 

The pool contract, however, contains **no pause check whatsoever**. Specifically:

**`Pool::claim_rewards`** (lines 335–377 of `pool.cairo`) calculates accumulated rewards from the internal `cumulative_rewards_trace` and then directly transfers STRK to the reward address via `IERC20Dispatcher::checked_transfer`. It never queries `IStaking::is_paused()` and never calls back to the staking contract for the transfer itself. [3](#0-2) 

The transfer is issued directly against the STRK ERC-20 contract: [4](#0-3) 

**`Pool::change_reward_address`** (lines 505–526) mutates the pool member's reward address in storage with no pause check. [5](#0-4) 

Neither function calls `assert_staker_is_active()` (which only checks `staker_removed`, not the staking pause) nor queries the staking contract's pause state. [6](#0-5) 

### Impact Explanation

**High — Theft of unclaimed yield.**

If the staking contract is paused due to an emergency (e.g., a reward calculation bug that inflated `cumulative_rewards_trace` entries), pool members can still call `Pool::claim_rewards` to drain the incorrectly accumulated STRK rewards before the protocol team can assess and correct the state. The pause is intended to freeze all fund movements, but the pool contract's direct ERC-20 transfer path is entirely outside the pause guard. Any pool member or their reward address can exploit this window.

### Likelihood Explanation

**Medium.** The attack requires the staking contract to be paused — an emergency scenario. However, once paused, the path is trivially reachable by any unprivileged pool member calling `Pool::claim_rewards` with no additional preconditions. The pool contract is a public, deployed contract with no access control on `claim_rewards` beyond membership.

### Recommendation

Add a pause check at the entry of all state-changing pool functions that move funds or alter member state. The pool contract should query `IStaking::is_paused()` (via the already-stored `staking_pool_dispatcher` address) and revert if the staking contract is paused:

```cairo
fn assert_staking_not_paused(self: @ContractState) {
    let staking_dispatcher = IStakingDispatcher {
        contract_address: self.staking_pool_dispatcher.contract_address.read(),
    };
    assert!(!staking_dispatcher.is_paused(), "Staking contract is paused");
}
```

Apply this guard at the top of `claim_rewards`, `change_reward_address`, `exit_delegation_pool_intent`, `exit_delegation_pool_action`, `enter_delegation_pool`, `add_to_delegation_pool`, and `switch_delegation_pool`.

### Proof of Concept

1. Admin calls `IStakingPause::pause()` on the staking contract due to a reward calculation emergency.
2. `staking.is_paused` is set to `true`. All `IStaking` functions now revert via `general_prerequisites()`.
3. Attacker (a pool member) calls `Pool::claim_rewards(pool_member)` directly on the pool contract.
4. The pool contract computes rewards from `cumulative_rewards_trace` (which may contain inflated values from the buggy epoch) and calls `IERC20Dispatcher::checked_transfer` to send STRK to the reward address.
5. The staking contract's pause is never consulted. The transfer succeeds.
6. Incorrect/inflated yield is extracted before the protocol can recover. [3](#0-2) [7](#0-6)

        fn claim_rewards(ref self: ContractState, pool_member: ContractAddress) -> Amount {
            // Asserts.
            let mut pool_member_info = self.internal_pool_member_info(:pool_member);
            let caller_address = get_caller_address();
            let reward_address = pool_member_info.reward_address;
            assert!(
                caller_address == pool_member || caller_address == reward_address,
                "{}",
                Error::POOL_CLAIM_REWARDS_FROM_UNAUTHORIZED_ADDRESS,
            );

            let until_checkpoint = self.get_current_checkpoint(:pool_member);

            // Calculate rewards and update entry_to_claim_from.
            let (mut rewards, updated_entry_to_claim_from) = self
                .calculate_rewards(
                    :pool_member,
                    from_checkpoint: pool_member_info.reward_checkpoint,
                    :until_checkpoint,
                    entry_to_claim_from: pool_member_info.entry_to_claim_from,
                );
            rewards += pool_member_info._unclaimed_rewards_from_v0;
            pool_member_info._unclaimed_rewards_from_v0 = Zero::zero();
            pool_member_info.entry_to_claim_from = updated_entry_to_claim_from;
            pool_member_info.reward_checkpoint = until_checkpoint;

            // Write the updated pool member info to storage.
            self.write_pool_member_info(:pool_member, :pool_member_info);

            // Transfer rewards to the pool member.
            let reward_token = IERC20Dispatcher { contract_address: STRK_TOKEN_ADDRESS };
            reward_token.checked_transfer(recipient: reward_address, amount: rewards.into());

            // Emit event.
            self
                .emit(
                    Events::PoolMemberRewardClaimed {
                        pool_member, reward_address, amount: rewards,
                    },
                );

            rewards
        }
