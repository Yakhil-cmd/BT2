### Title
Unprivileged Caller Can Invoke `update_rewards` With `disable_rewards: true` to Permanently Freeze Staker Yield - (File: src/staking/staking.cairo)

### Summary
The `update_rewards` function in the Staking contract is publicly callable with no access control. Any unprivileged caller can supply any valid `staker_address` and set `disable_rewards: true`, which advances the global `last_reward_block` checkpoint without distributing rewards. By repeating this at every block, an attacker permanently prevents all stakers from accruing consensus-era yield.

### Finding Description
`update_rewards` is exposed as a public ABI entry point via `StakingRewardsManagerImpl`: [1](#0-0) 

The function's only gate is `general_prerequisites()`, which checks the contract is not paused and the caller is non-zero: [2](#0-1) 

There is no check that `get_caller_address()` equals `staker_address`, the staker's operational address, or any privileged role.

The function then unconditionally writes the current block number to the **global** `last_reward_block` storage slot: [3](#0-2) 

After that write, if `disable_rewards` is `true`, the function returns immediately without distributing any rewards: [4](#0-3) 

Because `last_reward_block` is a single global variable shared across all stakers: [5](#0-4) 

any subsequent legitimate call to `update_rewards` in the same block is rejected by: [6](#0-5) 

The attacker only needs to supply any currently-active staker address with non-zero balance (trivially discoverable on-chain via the `stakers` vector and public view functions).

### Impact Explanation
An attacker calling `update_rewards(any_active_staker, true)` once per block:
- Marks every block as "already processed" without distributing rewards.
- Prevents the legitimate consensus mechanism from ever calling `update_rewards` with `disable_rewards: false`.
- Freezes the accrual of `unclaimed_rewards_own` for every staker and the forwarding of pool rewards to every delegation pool.

This constitutes **permanent freezing of unclaimed yield** for all stakers and delegators across the entire protocol.

### Likelihood Explanation
- The function is publicly callable; no special role or token ownership is required.
- The attacker only needs a valid, active staker address, which is publicly readable from the `stakers` vector.
- Starknet transaction costs are low, making sustained per-block griefing economically feasible.
- No front-running or timing precision is required; the attacker simply needs to submit one transaction per block.

### Recommendation
Restrict `update_rewards` to a single authorized caller. The most natural choice is the attestation contract (already used for the pre-consensus reward path) or a dedicated consensus rewards distributor role. For example:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
    // Add: assert caller is the authorized rewards distributor
    assert!(
        get_caller_address() == self.attestation_contract.read(),
        "{}",
        Error::CALLER_IS_NOT_ATTESTATION_CONTRACT,
    );
    ...
}
```

Alternatively, introduce a dedicated `rewards_manager` role and gate the function on it.

### Proof of Concept
1. Observe any active staker address `S` with non-zero STRK balance (readable via `get_stakers()` or `staker_info_v3()`).
2. At every new block `N`, call:
   ```
   staking_contract.update_rewards(S, disable_rewards=true)
   ```
3. The call passes all checks: contract is unpaused, caller is non-zero, `N > last_reward_block`, staker `S` is active with non-zero balance.
4. `last_reward_block` is set to `N`; no rewards are distributed.
5. Any subsequent call to `update_rewards` in block `N` (by the legitimate consensus mechanism) reverts with `REWARDS_ALREADY_UPDATED`.
6. Repeating step 2 at every block permanently prevents all stakers from accruing consensus-era yield.
