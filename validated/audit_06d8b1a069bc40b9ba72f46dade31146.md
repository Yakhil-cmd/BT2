### Title
Cross-Address COA Ownership Proof Replay via Missing `evmAddress` Binding in `signedData` - (File: `fvm/evm/stdlib/contract.cdc`)

### Summary
`EVM.validateCOAOwnershipProof` verifies Cadence account key signatures over an arbitrary `signedData` blob but does not enforce that `signedData` commits to the `evmAddress` parameter. A signature produced to prove ownership of one COA (Cadence-Owned Account) can be replayed by an unprivileged attacker to pass ownership verification for a *different* COA belonging to the same Cadence account, bypassing ERC-1271 authorization checks for EVM contracts.

### Finding Description
`EVM.validateCOAOwnershipProof` in `fvm/evm/stdlib/contract.cdc` accepts six parameters: `address`, `path`, `signedData`, `keyIndices`, `signatures`, and `evmAddress`. The function verifies that the provided signatures are valid over `signedData` using the Cadence account's keys, then checks that the COA resource at `path` has an EVM address matching `evmAddress`.

The critical gap: the cryptographic verification step (`keyList.verify`) operates only over `signedData` with the fixed domain tag `"FLOW-V0.0-user"`. The `evmAddress` parameter is checked *after* signature verification purely as a structural match against the COA resource — it is never included in the signed message. The code itself documents this:

> "Note: this function does not enforce that `signedData` includes `evmAddress`. In principle, a signature produced for one purpose could be replayed here against a different COA owned by the same Cadence account."

Because `validateCOAOwnershipProof` is `access(all)`, any unprivileged caller can invoke it from a Cadence script or transaction with attacker-controlled arguments. An attacker who observes a valid `(signedData, signatures)` pair used to prove ownership of COA_A can substitute a different `path` and `evmAddress` (pointing to COA_B, also owned by the same Cadence account) and obtain a `ValidationResult { isValid: true }` for COA_B without ever holding the account's private key. ### Impact Explanation
The `validateCOAOwnershipProof` function is the backend for the EVM precompile `verifyCOAOwnershipProof`, which is called by the COA's ERC-1271 `isValidSignature` implementation in `coa.sol`. EVM contracts that rely on ERC-1271 to authorize privileged operations (token transfers, governance votes, bridge withdrawals, etc.) on behalf of a COA will accept a replayed proof as valid authorization for a COA the attacker does not control.

Concretely: if Alice holds two COAs (COA_A and COA_B) under the same Cadence account, and an EVM contract asks Alice to prove ownership of COA_B via ERC-1271, an attacker who observed Alice's prior proof for COA_A can replay it to impersonate Alice's authorization for COA_B. This constitutes unauthorized EVM contract authorization and can lead to unauthorized asset movement or privileged state changes in EVM contracts that gate actions on `isValidSignature`. ### Likelihood Explanation
The function is `access(all)` and callable from any Cadence script or transaction submitted by an unprivileged user. The attacker only needs to observe a prior on-chain proof (available from transaction history or the mempool) and submit a new script with the substituted `path`/`evmAddress`. The precondition — a Cadence account with multiple COAs — is explicitly anticipated by the protocol (the `path` parameter exists precisely to support multiple COAs per account). The code comment acknowledges the replay risk and defers the fix to callers, leaving the protocol-level function exploitable. [5](#0-4) [6](#0-5) 

### Recommendation
Enforce inside `validateCOAOwnershipProof` that `signedData` cryptographically commits to `evmAddress` (e.g., require `signedData` to equal `SHA3_256(evmAddress || applicationData)` or include `evmAddress` as a mandatory prefix). Alternatively, construct the message to verify as `hash(evmAddress || signedData)` internally before passing it to `keyList.verify`, so the binding is protocol-enforced rather than caller-convention. This mirrors the EIP-712 domain separator pattern recommended in the external report. [2](#0-1) 

### Proof of Concept
1. Alice holds Cadence account `0xALICE` with two COAs: COA_A at `/public/coaA` (EVM address `0xEVM_A`) and COA_B at `/public/coaB` (EVM address `0xEVM_B`).
2. Alice legitimately proves ownership of COA_A by signing `signedData = <some_bytes>` with her account key, producing `sig`. This proof is submitted on-chain and is publicly observable.
3. An EVM contract asks Alice to prove ownership of COA_B via ERC-1271 (`isValidSignature(hash, proof)`), which internally calls `verifyCOAOwnershipProof(0xEVM_B, hash, encodedProof)` → `validateCOAOwnershipProof(0xALICE, /public/coaB, hash, [keyIdx], [sig], 0xEVM_B)`.
4. The attacker, without Alice's private key, submits a Cadence script calling `EVM.validateCOAOwnershipProof(address: 0xALICE, path: /public/coaB, signedData: <same_bytes_as_step_2>, keyIndices: [keyIdx], signatures: [sig], evmAddress: 0xEVM_B)`.
5. `keyList.verify` succeeds because `sig` is valid over `signedData` regardless of which COA is targeted. The subsequent EVM address check passes because the attacker correctly supplied `0xEVM_B`. The function returns `isValid: true`.
6. The EVM contract accepts this as valid authorization for COA_B, granting the attacker unauthorized control over COA_B's EVM contract interactions. [7](#0-6) [8](#0-7)
