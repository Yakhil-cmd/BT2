### Title
Missing Participant-Set Consistency Check Between Presign Participants and Triple Participants - (File: src/ecdsa/ot_based_ecdsa/presign.rs)

---

### Summary

The `presign()` function accepts a `participants` argument and two triples (`triple0`, `triple1`), each of which embeds its own `participants` field in `TriplePub`. The function checks that the threshold values match, but **explicitly omits** any check that the presign `participants` list matches the participant sets stored inside the triples. This is a direct analog to the external report: just as `setLiquidationPath` fails to verify that tokens match the pair, `presign()` fails to verify that the participant set matches the one used during triple generation. The result is a silently corrupted presignature that causes signing to fail.

---

### Finding Description

In `src/ecdsa/ot_based_ecdsa/presign.rs`, the public entry point `presign()` performs the following checks:

```
// Spec 1.1
if args.threshold.value() > participants.len() { ... }

if args.threshold != args.triple0.1.threshold || args.threshold != args.triple1.1.threshold {
    return Err(InitializationError::BadParameters(
        "New threshold must match the threshold of both triples".to_string(),
    ));
}
``` [1](#0-0) 

The threshold is validated against both triples. However, the function then contains this explicit comment:

```
// NOTE: We omit the check that the new participant set was present for
// the triple generation, because presumably they need to have been present
// in order to have shares.
``` [2](#0-1) 

The `TriplePub` struct (produced by triple generation) stores its own `participants` field:

```rust
TriplePub {
    big_a,
    big_b,
    big_c,
    participants: participants.clone().into(),
    threshold,
},
``` [3](#0-2) 

Inside `do_presign()`, the Lagrange coefficient is computed using the **presign** `participants` argument, not the triple's embedded participant set:

```rust
let lambda_me = participants.lagrange::<Secp256>(me)?;
let k_prime_i = lambda_me * k_i;
let e_i: Scalar = lambda_me * e_i;
let a_prime_i = lambda_me * a_i;
let b_prime_i = lambda_me * b_i;
let x_prime_i = lambda_me * private_share;
``` [4](#0-3) 

The triple shares `k_i`, `e_i`, `a_i`, `b_i`, `c_i` are Shamir shares valid for the **triple's** participant set. Applying Lagrange coefficients derived from a **different** participant set produces arithmetically wrong linearized values. The presignature is then computed from these wrong values and returned as `Ok(PresignOutput { ... })` â€” no error is raised.

The root cause is the missing cross-reference check:
```rust
// MISSING:
if participants != args.triple0.1.participants || participants != args.triple1.1.participants {
    return Err(InitializationError::BadParameters(...));
}
```

---

### Impact Explanation

**High â€” Corruption of presign outputs so honest parties accept inconsistent/unusable cryptographic outputs.**

When `presign()` is called with a `participants` list that differs from the triple's embedded participant set, the function silently returns a `PresignOutput` with cryptographically invalid `k` and `sigma` values. Honest parties have no way to detect this at the presign stage. When the corrupted presignature is later used in the signing phase, the aggregated signature will be invalid, causing signing to fail. A malicious coordinator who controls the distribution of `participants` and `args` to honest participants can trigger this condition for all honest parties simultaneously, permanently blocking signing for any message until new triples are generated.

---

### Likelihood Explanation

**Medium.** The `presign()` function is a library-level API. In a real deployment, a coordinator or orchestration layer assembles the `participants` list and `PresignArguments` and distributes them to each participant. A malicious coordinator, or a coordinator that has been compromised or misconfigured, can trivially supply a `participants` list that does not match the triple's embedded participant set. The library provides no defense: the comment at line 38â€“40 explicitly acknowledges the omission and relies on an assumption ("presumably") rather than an enforced invariant.

---

### Recommendation

Add an explicit check in `presign()` that the presign `participants` list matches the participant sets stored in both triples before proceeding:

```rust
if participants.as_slice() != args.triple0.1.participants.as_slice()
    || participants.as_slice() != args.triple1.1.participants.as_slice()
{
    return Err(InitializationError::BadParameters(
        "Presign participant set must match the participant set of both triples".to_string(),
    ));
}
```

This mirrors the existing threshold check and closes the structural consistency gap analogous to the external report.

---

### Proof of Concept

1. Generate two triples with participant set `[P1, P2, P3]` and threshold 2.
2. Call `presign()` with `participants = [P1, P2, P4]` (substituting `P4` for `P3`) and the triples from step 1.
3. Observe that `presign()` returns `Ok(PresignOutput { ... })` without error.
4. The returned `k` and `sigma` values are computed with Lagrange coefficients for `[P1, P2, P4]` applied to shares that are valid only for `[P1, P2, P3]`.
5. Pass the corrupted presignature into the signing phase; the aggregated signature fails verification, demonstrating denial of signing for all honest participants who received the mismatched inputs. [5](#0-4)

pub fn presign(
    participants: &[Participant],
    me: Participant,
    args: PresignArguments,
) -> Result<impl Protocol<Output = PresignOutput>, InitializationError> {
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }
    // Spec 1.1
    if args.threshold.value() > participants.len() {
        return Err(InitializationError::ThresholdTooLarge {
            threshold: args.threshold.value(),
            max: participants.len(),
        });
    }

    // NOTE: We omit the check that the new participant set was present for
    // the triple generation, because presumably they need to have been present
    // in order to have shares.

    // Also check that we have enough participants to reconstruct shares.
    if args.threshold != args.triple0.1.threshold || args.threshold != args.triple1.1.threshold {
        return Err(InitializationError::BadParameters(
            "New threshold must match the threshold of both triples".to_string(),
        ));
    }

    let participants =
        ParticipantList::new(participants).ok_or(InitializationError::DuplicateParticipants)?;

    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    let ctx = Comms::new();
    let fut = do_presign(ctx.shared_channel(), participants, me, args);
    Ok(make_protocol(ctx, fut))
}
