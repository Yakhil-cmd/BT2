### Title
Missing Access Control on `update_rewards` Allows Any Caller to Permanently Freeze Staker Yield - (`src/staking/staking.cairo`)

### Summary

The `update_rewards` function in the Staking contract is specified to be callable only by the Starkware sequencer, but the implementation enforces no such restriction. Any unprivileged address can call `update_rewards` with `disable_rewards: true`, which updates `last_reward_block` without distributing rewards. This permanently prevents the sequencer from distributing rewards for that block to the targeted staker, since the `REWARDS_ALREADY_UPDATED` guard fires on the sequencer's subsequent call.

### Finding Description

The protocol specification explicitly states:

> **access control**: Only starkware sequencer. [1](#0-0) 

However, the implementation of `update_rewards` only calls `general_prerequisites()`: [2](#0-1) 

`general_prerequisites` only checks that the contract is unpaused and the caller is non-zero — there is no sequencer role check: [3](#0-2) 

Critically, `last_reward_block` is written **before** the `disable_rewards` guard is evaluated: [4](#0-3) 

This means an attacker who calls `update_rewards(target_staker, disable_rewards: true)` will:
1. Advance `last_reward_block` to the current block number.
2. Return immediately without distributing any rewards.

When the sequencer subsequently attempts to call `update_rewards(target_staker, disable_rewards: false)` for the same block, it hits:

```cairo
assert!(
    current_block_number > self.last_reward_block.read(),
    "{}",
    Error::REWARDS_ALREADY_UPDATED,
);
```

and reverts. The staker's rewards for that block are permanently lost — there is no mechanism to retroactively distribute rewards for a missed block.

### Impact Explanation

This is **permanent freezing of unclaimed yield**. Each block for which an attacker front-runs the sequencer with `disable_rewards: true` results in the targeted staker (and their delegators) permanently losing the block reward they were entitled to. The loss compounds with each block targeted. The rewards are never minted/transferred; they are simply skipped.

### Likelihood Explanation

The entry point is fully public — any non-zero address can call `update_rewards`. The attacker only needs to submit a transaction calling `update_rewards(victim, disable_rewards: true)` in a block before the sequencer's own reward-distribution transaction. On Starknet, users submit transactions to the sequencer's mempool; a malicious user can submit such a transaction targeting any active staker at any time. No capital, stake, or privileged access is required.

### Recommendation

Add a sequencer-only access control check at the top of `update_rewards`, consistent with the spec and analogous to how other privileged functions use role checks (e.g., `self.roles.only_token_admin()`):

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
    self.roles.only_sequencer(); // <-- add this
    ...
}
```

Alternatively, move the `last_reward_block.write(current_block_number)` to after the `disable_rewards` branch so that a call with `disable_rewards: true` does not consume the block's reward slot.

### Proof of Concept

```
1. Staker S is active and eligible for consensus block rewards.
2. Attacker A (any address) calls:
       staking.update_rewards(staker_address: S, disable_rewards: true)
   - last_reward_block is set to current_block_number.
   - No rewards are distributed (early return).
3. Sequencer calls:
       staking.update_rewards(staker_address: S, disable_rewards: false)
   - Panics: REWARDS_ALREADY_UPDATED (current_block_number == last_reward_block).
4. Staker S and all delegators permanently lose block rewards for this block.
5. Attacker repeats every block to continuously deny all rewards to S.
```

The relevant guard that makes the loss permanent: [5](#0-4)
