Audit Report

## Title
Unbounded DKG Unvalidated Pool Growth via Byzantine P2P Peer — (`rs/artifact_pool/src/dkg_pool.rs`)

## Summary
The DKG artifact pipeline has no per-peer slot limit (`SLOT_TABLE_NO_LIMIT = usize::MAX`), no capacity guard in `DkgPoolImpl::insert`, and a `DkgBouncer` that accepts any message at the current DKG interval start height without checking dealer membership or signature validity. A single Byzantine subnet node can flood every connected honest replica's unvalidated pool with unique, height-correct DKG messages, exhausting heap memory before the asynchronous validator can purge them, and causing OOM-kill of the replica process.

## Finding Description

**Root cause 1 — No slot limit for DKG.**
`SLOT_TABLE_NO_LIMIT = usize::MAX` is passed to `abortable_broadcast_channel` for DKG at `rs/replica/setup_ic_network/src/lib.rs:279`. [1](#0-0) [2](#0-1) 

The guard in `receiver.rs` at line 373 (`peer_slot_table_len < self.slot_limit`) is effectively disabled because `self.slot_limit` is `usize::MAX`. A Byzantine peer can advertise an unbounded number of distinct slot numbers, each carrying a unique artifact ID. [3](#0-2) 

**Root cause 2 — `DkgBouncer` accepts any message at the current interval height.**
`DkgBouncer::new_bouncer` returns `BouncerValue::Wants` for any `DkgMessageId` whose `height == start_height`. It does not check dealer membership against the DKG summary, does not verify the `BasicSignature`, and does not inspect the dealing payload. [4](#0-3) 

**Root cause 3 — `DkgPoolImpl::insert` is unconditional.**
The insert is a direct `BTreeMap` insertion with no capacity check, no per-peer quota, and no rejection path. [5](#0-4) 

**Root cause 4 — `DkgMessageId` is a full-message crypto hash.**
Each distinct dealing payload produces a unique `DkgMessageId`, so every flooded message occupies a distinct pool entry and a distinct P2P slot. [6](#0-5) 

**Root cause 5 — Asynchronous validation processes only one message per `(dealer, DKG ID)` per cycle.**
`on_state_change` groups unvalidated messages by `(dealer, DKG ID)` and calls `validate_dealings_for_dealer` on only the first message in each group. If the Byzantine node sends N messages under the same `(dealer, DKG ID)` key, only one is removed per validation cycle. Messages at the current interval height are never purged by `purge` (which only removes entries with `id.height < height`), so they accumulate until the next summary block. [7](#0-6) [8](#0-7) 

**Why existing checks fail:**
`crypto_validate_dealing` does check dealer membership and signature validity, but this check runs asynchronously after pool insertion. The P2P layer (bouncer + slot table) is the only pre-insertion gate, and it is entirely bypassed by the missing slot limit and the height-only bouncer. [9](#0-8) 

## Impact Explanation

A Byzantine subnet node can exhaust the heap of every honest replica it is connected to. `NiDkgDealing` messages contain large cryptographic blobs (forward-secure ciphertexts for all receivers). Sending 10^4–10^5 unique messages at the current DKG interval start height will grow the unvalidated pool to gigabytes before the next summary block triggers a purge. The replica process is killed by the OS OOM killer, halting its participation in consensus. If enough replicas on a subnet are affected simultaneously, the subnet stalls.

This matches the **High ($2,000–$10,000)** impact class: *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."* The attack exploits a specific protocol design flaw, not raw bandwidth flooding.

## Likelihood Explanation

The attacker must be a registered subnet node with a valid TLS certificate — one Byzantine node within the f-out-of-(3f+1) fault model. No special preconditions are required beyond being a P2P peer during an active DKG interval. The `DkgBouncer` height check is trivially satisfied by using the correct `start_block_height` from the public chain state. The slot table imposes no limit. The attack is mechanically straightforward and repeatable across every DKG interval.

## Recommendation

1. **Add a per-peer slot limit for DKG** analogous to `SLOT_TABLE_LIMIT_INGRESS = 50_000`. Replace `SLOT_TABLE_NO_LIMIT` with a registry-derived constant (e.g., `num_dealers * num_active_dkg_configs * small_multiplier`) passed to `abortable_broadcast_channel` for the DKG channel.

2. **Add a capacity check in `DkgPoolImpl::insert`** that rejects insertions beyond a computed bound (e.g., `num_subnet_nodes * num_active_dkg_configs * MAX_DEALINGS_PER_DEALER`).

3. **Extend `DkgBouncer` to check dealer membership** against the current DKG summary before returning `BouncerValue::Wants`, so messages from non-dealers are dropped at the P2P layer before pool insertion.

## Proof of Concept

A local integration test using `PocketIC` or a single-subnet `ic-starter` replica:

```rust
// Byzantine node: generate 100_000 unique DKG messages at the current interval height
let start_height = dkg_pool.get_current_start_height(); // read from public chain state
for i in 0..100_000u64 {
    // Vary the dealing bytes to produce a unique DkgMessageId per message
    let dealing = NiDkgDealing { internal_dealing: craft_unique_dealing(i) };
    let content = DealingContent::new(dealing, NiDkgId {
        start_block_height: start_height, // satisfies DkgBouncer: Wants
        ..current_dkg_id
    });
    let msg = sign_with_byzantine_node_key(content); // valid TLS peer, own key
    p2p_send_slot_update(slot_number: i, msg); // SLOT_TABLE_NO_LIMIT → accepted
    // DkgPoolImpl::insert → no capacity check → pool grows by one entry
}
// Monitor replica RSS; expect OOM kill before next summary block
```

Each message passes the `DkgBouncer` (height == `start_height` → `Wants`), bypasses the slot table (limit is `usize::MAX`), and is unconditionally inserted into the `BTreeMap`. The asynchronous validator processes at most one message per `(dealer, DKG ID)` group per `on_state_change` cycle, and `purge` never removes current-interval messages, so the pool grows unboundedly until the replica is OOM-killed.

impl<Pool: DkgPool> BouncerFactory<DkgMessageId, Pool> for DkgBouncer {
    fn new_bouncer(&self, dkg_pool: &Pool) -> Bouncer<DkgMessageId> {
        let _timer = self.metrics.update_duration.start_timer();

        let start_height = dkg_pool.get_current_start_height();
        Box::new(move |id| {
            use std::cmp::Ordering;
            match id.height.cmp(&start_height) {
                Ordering::Equal => BouncerValue::Wants,
                Ordering::Greater => BouncerValue::MaybeWantsLater,
                Ordering::Less => BouncerValue::Unwanted,
            }
        })
    }

    fn refresh_period(&self) -> std::time::Duration {
        std::time::Duration::from_secs(3)
    }
}
