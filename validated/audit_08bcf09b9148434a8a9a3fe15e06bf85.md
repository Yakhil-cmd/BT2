### Title
`compute_rewards_per_unit` Returns Zero for Sub-Threshold Pool Balance, Permanently Freezing Forwarded Rewards — (`File: src/pool/pool.cairo`)

---

### Summary

`Pool::compute_rewards_per_unit` silently returns zero when the total delegated balance is below `min_delegation_for_rewards`. Because the staking contract unconditionally transfers STRK reward tokens to the pool contract *before* calling `update_rewards_from_staking_contract`, those tokens are permanently locked in the pool with no mechanism for delegators to claim them.

---

### Finding Description

`compute_rewards_per_unit` in `src/pool/pool.cairo` contains an early-return guard:

```cairo
fn compute_rewards_per_unit(
    self: @ContractState, staking_rewards: Amount, total_stake: Amount,
) -> Index {
    // Return zero if the total stake is too small, to avoid overflow below.
    if total_stake < self.min_delegation_for_rewards.read() {
        return Zero::zero();  // ← returns 0, discarding real rewards
    }
    mul_wide_and_div(
        lhs: staking_rewards, rhs: self.staking_rewards_base_value.read(), div: total_stake,
    )
        .expect_with_err(err: InternalError::REWARDS_COMPUTATION_OVERFLOW)
}
``` [1](#0-0) 

For a STRK pool, `min_delegation_for_rewards = 10^18` (1 STRK), set at construction time from `STRK_CONFIG`. [2](#0-1) 

The caller of `compute_rewards_per_unit` is `update_rewards_from_staking_contract`:

```cairo
fn update_rewards_from_staking_contract(
    ref self: ContractState, rewards: Amount, pool_balance: Amount,
) {
    self.assert_caller_is_staking_contract();
    let (_, last) = self.cumulative_rewards_trace.last().unwrap();
    self.cumulative_rewards_trace.insert(
        key: self.get_current_epoch(),
        value: last + self.compute_rewards_per_unit(
            staking_rewards: rewards, total_stake: pool_balance,
        ),
    );
}
``` [3](#0-2) 

When `compute_rewards_per_unit` returns zero, the cumulative sigma index does not advance. All subsequent calls to `calculate_rewards` for any pool member will compute `to_sigma - from_sigma == 0`, yielding zero claimable rewards regardless of how much STRK was forwarded.

The staking contract's `update_pool_rewards` transfers STRK tokens to the pool **before** calling `update_rewards_from_staking_contract`:

```cairo
self.send_rewards_to_delegation_pool(
    :staker_address, pool_address: pool_contract,
    amount: pool_rewards, token_dispatcher: strk_token_dispatcher,
);
// ...
pool_dispatcher.update_rewards_from_staking_contract(
    rewards: pool_rewards,
    pool_balance: pool_balance.to_native_amount(:decimals),
);
``` [4](#0-3) 

The code's own documentation acknowledges this:

> **Note**: Delegation rewards lost when pool balance is less than `min_delegation_for_rewards`. The staking contract continues to forward `pool_rewards` to the pool contract even in this case. [5](#0-4) 

The tokens are transferred to the pool contract's address but the sigma index is never updated, so no pool member can ever claim them. There is no recovery path.

---

### Impact Explanation

**High — Permanent freezing of unclaimed yield.**

Any STRK rewards forwarded to a pool whose total delegated balance is below `min_delegation_for_rewards` (1 STRK for STRK pools) are permanently locked in the pool contract. The `cumulative_rewards_trace` is never updated for those reward events, so `calculate_rewards` always returns zero for the affected epochs. The tokens cannot be swept, refunded, or redistributed.

---

### Likelihood Explanation

**Medium.** There is no minimum delegation amount enforced in `enter_delegation_pool` beyond `amount.is_non_zero()`. [6](#0-5) 

A delegator can legitimately enter with any sub-threshold amount (e.g., 0.5 STRK). If the total pool balance stays below 1 STRK across all members, every reward distribution event permanently freezes the forwarded STRK. This is also reachable in the early life of a new pool before significant delegation accumulates.

---

### Recommendation

The staking contract should not forward rewards to a pool when the pool balance is below `min_delegation_for_rewards`. The guard should be moved upstream into `calculate_staker_pools_rewards` so that `pool_rewards` is set to zero before `send_rewards_to_delegation_pool` is called, preventing the token transfer entirely. Alternatively, the pool contract should revert or return the tokens when `total_stake < min_delegation_for_rewards` rather than silently discarding the accounting update after accepting the transfer.

---

### Proof of Concept

1. Staker stakes and opens a STRK delegation pool.
2. Delegator calls `enter_delegation_pool` with `amount = 5 * 10^17` (0.5 STRK < `min_delegation_for_rewards`).
3. Staking contract records `pool_balance_curr_epoch = 5 * 10^17` (normalized).
4. On the next `update_rewards` / attestation, `calculate_staker_pools_rewards` computes `pool_rewards_including_commission = total_rewards * 5e17 / total_stake`. With `total_rewards = 10^18` and `total_stake = 10^21`, this yields `5 * 10^14` — non-zero, so the pool is included in `pool_rewards_array`.
5. `update_pool_rewards` calls `send_rewards_to_delegation_pool`, transferring `5 * 10^14` STRK to the pool contract.
6. `update_rewards_from_staking_contract` is called with `pool_balance = 5 * 10^17`.
7. `compute_rewards_per_unit(staking_rewards: 5e14, total_stake: 5e17)` → `5e17 < 10^18 = min_delegation_for_rewards` → returns `0`.
8. `cumulative_rewards_trace` is updated with `last + 0 = last` (no change).
9. Delegator calls `claim_rewards`: `calculate_rewards` computes `to_sigma - from_sigma = 0` → returns `0`.
10. The `5 * 10^14` STRK tokens are permanently frozen in the pool contract.
