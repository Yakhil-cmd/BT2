### Title
Weak Timestamp Monotonicity Check Allows Equal Block Timestamps in ZK Proving Path — (File: `basic_bootloader/src/bootloader/block_flow/zk/post_tx_op/post_tx_op_proving_singleblock_batch.rs` and `post_tx_op_proving_multiblock_batch.rs`)

---

### Summary

The ZK proving path enforces block timestamp monotonicity using a non-strict `>=` comparison, while the authoritative `ProofData` struct documentation explicitly states the check must be a **strict** `>` ("greater than"). This discrepancy allows a prover/operator to submit blocks whose timestamp equals the previous block's timestamp, effectively freezing on-chain time for L2 contracts.

---

### Finding Description

In both single-block and multi-block batch proving post-tx-op handlers, the timestamp check reads:

```rust
// validate that timestamp didn't decrease
assert!(metadata.block_timestamp() >= last_block_timestamp);
``` [1](#0-0) [2](#0-1) 

The `ProofData` struct documentation — which defines the contract for this exact validation — explicitly states:

> "We'll validate reads/apply writes against `state_root_view` and validate that block timestamp is **greater than** `last_block_timestamp`." [3](#0-2) 

The documented intent is a **strict** inequality (`>`), but the implementation uses `>=` (non-strict). This is the same class of incorrect comparison operator bug as the external report: the wrong relational operator is used, causing the guard to be weaker than specified.

`last_block_timestamp` is read from `ProofData` (oracle-supplied), which is derived from the on-chain `ChainStateCommitment`. The `ChainStateCommitment` includes `last_block_timestamp` specifically "to ensure that block timestamps are not decreasing." [4](#0-3) 

With `>=`, a prover can satisfy the constraint by supplying `block_timestamp == last_block_timestamp`, which the strict-`>` intent would have rejected.

Additionally, in the multiblock batch path, `ZKBatchDataKeeper::apply_block` performs **no timestamp monotonicity check** between consecutive blocks within the same batch: [5](#0-4) 

Only the first block of a batch is compared against the previous batch's `last_block_timestamp`. Subsequent blocks within the same batch have no timestamp ordering enforced at all.

---

### Impact Explanation

A malicious prover/operator can submit a valid ZK proof for a block (or sequence of blocks within a batch) where `block_timestamp == last_block_timestamp`. The settlement layer will accept this proof because the `ChainStateCommitment` hash will be consistent. The result is that L2 `block.timestamp` appears frozen:

- Time-locked contracts (e.g., vesting, escrow) cannot release funds because the timestamp never advances.
- Deadline-based DeFi protocols (e.g., options, liquidations) cannot expire positions.
- Any contract using `block.timestamp` for ordering or rate-limiting is manipulable.

This is a state-transition correctness bug: the committed `chain_state_commitment_after` encodes a `last_block_timestamp` equal to the previous block's, which the settlement layer stores as the new canonical state, permanently anchoring the chain's time reference.

---

### Likelihood Explanation

The attacker is the prover/operator who controls `metadata.block_timestamp()` (the new block's timestamp). This role is explicitly listed in the Immunefi scope as a valid attacker entry point ("prover/forward execution input"). The operator sets the block timestamp as part of block metadata; choosing `block_timestamp == last_block_timestamp` is a trivial, deliberate act requiring no special capability beyond normal operator access.

---

### Recommendation

Change both occurrences to use strict inequality, matching the documented intent in `ProofData`:

```rust
// validate that timestamp is strictly greater than last block timestamp
assert!(metadata.block_timestamp() > last_block_timestamp);
```

Additionally, add a monotonicity check between consecutive blocks within a multiblock batch in `ZKBatchDataKeeper::apply_block`:

```rust
} else {
    // existing checks ...
    assert!(block_timestamp > self.current_block_timestamp.unwrap(),
        "block timestamp must be strictly increasing within a batch");
    self.current_block_timestamp = Some(block_timestamp);
```

---

### Proof of Concept

1. Previous batch ends with `last_block_timestamp = T`, committed on the settlement layer.
2. Prover constructs a new block with `metadata.block_timestamp() = T` (identical value).
3. `ProofData` oracle returns `last_block_timestamp = T` (read from on-chain commitment).
4. The check `assert!(T >= T)` passes — proof generation succeeds.
5. The resulting `chain_state_commitment_after` encodes `last_block_timestamp = T`.
6. Settlement layer accepts the proof; the new canonical state has `last_block_timestamp = T`.
7. All L2 contracts querying `block.timestamp` observe no time progression.
8. Repeat indefinitely: the prover can keep submitting blocks with `block_timestamp = T`, freezing L2 time while the chain continues to process transactions.
