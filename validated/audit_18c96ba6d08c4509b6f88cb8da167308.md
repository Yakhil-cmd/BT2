### Title
WalletConnect `take_offer` Approval Dialog Understates Total Spend Due to Incorrect Multi-NFT Royalty Accumulation — (File: `packages/gui/src/electron/commands/parseCommandDisplay.ts`)

---

### Summary

The `formatAmountWithRoyalties` function in the WalletConnect command-display pipeline incorrectly calculates the total royalty cost when an offer involves multiple NFTs. It divides the base amount equally among NFTs before applying each royalty rate, producing a displayed `amountWithRoyalties` that is `1/n` of the correct value (where `n` is the number of NFTs). A user approving a `take_offer` WalletConnect request sees a materially understated total spend and approves a transaction that deducts significantly more XCH or CAT than shown.

---

### Finding Description

`formatAmountWithRoyalties` is called by `withRoyaltyTotals` → `walletDeltaToDisplay` → `parseCommandDisplay` for every `chia_wallet.take_offer` WalletConnect request. It is responsible for computing the "Total Amount with Royalties" line shown in the approval dialog. [1](#0-0) 

The flawed logic:

```typescript
const splitAmount = amount / BigInt(royaltyPercentages.length);   // ← divides first
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
```

This computes:

```
displayed_royalty = (amount / n) × Σ(royaltyPercentage_i) / 10_000
                  = amount × Σ(royaltyPercentage_i) / (n × 10_000)
```

The correct formula is:

```
correct_royalty = Σ(amount × royaltyPercentage_i / 10_000)
               = amount × Σ(royaltyPercentage_i) / 10_000
```

The displayed royalty is exactly `1/n` of the correct royalty. The project's own test fixture confirms the wrong expected value: [2](#0-1) 

With 2 NFTs (royalties 500 bp and 10 bp) and 100,000,000 mojos:
- **Displayed**: `splitAmount = 50,000,000` → royalty = 2,550,000 → total = **102,550,000 mojos (0.00010255 XCH)**
- **Correct**: royalty = 5,000,000 + 100,000 = 5,100,000 → total = **105,100,000 mojos (0.0001051 XCH)**

The `amountWithRoyalties` field is propagated into the WalletConnect approval dialog via `walletDeltaToDisplay`: [3](#0-2) 

The royalty percentages for `take_offer` are sourced from the offer's `infos` field (returned by the node's `get_offer_summary` RPC), which accurately reflects on-chain NFT royalty data: [4](#0-3) 

The actual coin spends embedded in the offer pay the full per-NFT royalty to each royalty address. The GUI display is the only place the undercount occurs.

---

### Impact Explanation

A user approving a `take_offer` WalletConnect request sees `amountWithRoyalties` as the total XCH or CAT they will spend. Because the displayed value is `1/n` of the correct royalty total, the user consents to a transaction believing they are spending materially less than is actually deducted. With 3 NFTs each carrying a 30% royalty and a 1 XCH base price, the dialog shows **1.3 XCH** while the wallet pays **1.9 XCH** — a 46% understatement. This satisfies the High impact criterion: WalletConnect state causes a user to approve the wrong amount.

---

### Likelihood Explanation

Any WalletConnect-connected dApp that has been granted `take_offer` permission can send a crafted offer containing multiple NFTs with royalties. No leaked keys, host compromise, or cryptographic break is required. The user must have an active WalletConnect session with `take_offer` in the approved command list, which is a standard permission for NFT-trading dApps. [5](#0-4) 

---

### Recommendation

Remove the erroneous split and apply each royalty percentage to the full `amount`:

```typescript
// Before (wrong):
const splitAmount = amount / BigInt(royaltyPercentages.length);
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);

// After (correct):
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (amount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
```

Update the corresponding test expectations in `parseCommandDisplay.test.ts` to reflect the correct totals.

---

### Proof of Concept

1. Attacker operates a WalletConnect dApp that the user has granted `take_offer` permission.
2. Attacker crafts an offer: user pays 1 XCH, receives 3 NFTs each with 20% royalty (2,000 bp). The offer's coin spends include three royalty outputs of 0.2 XCH each (total royalty 0.6 XCH, correct total 1.6 XCH).
3. Dapp sends `chia_wallet.take_offer` via WalletConnect.
4. `parseCommandDisplay` calls `formatAmountWithRoyalties` with `royaltyPercentages = [2000, 2000, 2000]` and `amount = 1_000_000_000_000n` mojos.
5. `splitAmount = 333_333_333_333n`; displayed `royaltyAmount = 3 × (333_333_333_333 × 2000 / 10_000) = 200_000_000_000` mojos (0.2 XCH).
6. Approval dialog shows **Total: 1.2 XCH**.
7. User approves. Actual deduction: **1.6 XCH** (the three 0.2 XCH royalty coin spends embedded in the offer).
8. User's wallet is short by 0.4 XCH relative to what they consented to. [6](#0-5)

```typescript
function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],
): string | undefined {
  if (royaltyPercentages.length === 0 || line.kind === 'nft') {
    return undefined;
  }

  const splitAmount = amount / BigInt(royaltyPercentages.length);
  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;

  if (line.kind === 'xch') {
    return mojoToChiaLocaleString(totalAmount);
  }

  return mojoToCATLocaleString(totalAmount);
}
```
