### Title
Unimplemented `TurnBridge` Handler Bypasses Authorization and Permanently Disables Bridge Emergency Stop - (File: x/cronos/keeper/msg_server.go)

### Summary
The `TurnBridge` message handler at line 84 of `x/cronos/keeper/msg_server.go` is a stub that unconditionally returns `nil, nil` without performing any permission check or state mutation. Any unprivileged address can submit a `MsgTurnBridge` transaction that succeeds on-chain, and no actor — including the designated admin — can ever actually disable the gravity bridge.

### Finding Description
The `TurnBridge` gRPC handler is registered in the Cronos `MsgServer` interface and is reachable via a live proto-defined transaction surface: [1](#0-0) 

The body is a two-line stub: `return nil, nil`. It performs no authorization check, reads no state, and writes no state. Compare this to every other privileged handler in the same file:

- `UpdateTokenMapping` calls `k.HasPermission(ctx, msg.GetSigners(), CanChangeTokenMapping)` [2](#0-1) 
- `UpdatePermissions` and `StoreBlockList` compare `msg.From` against the stored `CronosAdmin` [3](#0-2) 

`TurnBridge` has none of these guards. The `ValidateBasic` for `MsgTurnBridge` only validates that `Sender` is a valid bech32 address: [4](#0-3) 

The proto definition confirms `TurnBridge` is a live, publicly-routable RPC: [5](#0-4) 

The CLI description for `UpdatePermissions` explicitly names `CanTurnBridge` (value `2`) as a distinct permission bit: [6](#0-5) 

This permission is never checked in the handler, and the handler never calls any keeper method to persist the `Enable` field from `MsgTurnBridge`.

### Impact Explanation
Two distinct impacts arise:

**1. Authorization bypass (High):** Any unprivileged address can submit `MsgTurnBridge` and receive a success response (`code == 0`). The `CanTurnBridge` permission system is entirely bypassed. The integration test at `integration_tests/test_gravity.py` line 661–662 explicitly asserts that a community (non-admin) caller must be rejected: [7](#0-6) 

With the current stub, both the community and validator calls succeed, violating the intended access control.

**2. Permanent inability to disable the bridge (High):** Because the handler writes no state, the bridge enabled/disabled flag can never be changed by anyone. The gravity bridge emergency-stop mechanism — which the integration test confirms is expected to block `send_to_evm_chain` after `TurnBridge(false)` — is permanently non-functional: [8](#0-7) 

If a bridge exploit occurs, operators have no on-chain mechanism to halt outbound bridge flows.

### Likelihood Explanation
The entry path is a standard signed Cosmos SDK transaction requiring only a valid bech32 sender address and gas. No special privilege, key, or condition is needed. The transaction is accepted by `ValidateBasic` and routed to the stub handler, which always returns success. Likelihood is high.

### Recommendation
Implement the `TurnBridge` handler body:
1. Add a `HasPermission(ctx, []sdk.AccAddress{signer}, CanTurnBridge)` check (mirroring `UpdateTokenMapping`).
2. Persist `msg.Enable` to a keeper-managed store key (e.g., a `BridgeEnabled` param or dedicated KV entry).
3. Enforce the stored flag in all gravity bridge outbound paths (e.g., `send_to_evm_chain` EVM hook handler).

### Proof of Concept
```
# Any unprivileged address submits MsgTurnBridge — tx succeeds, no state changes
cronosd tx cronos turn-bridge false --from community --gas 200000 --gas-prices 5000000000000basetcro -y

# Expected: code != 0 (unauthorized)
# Actual:   code == 0 (stub returns nil, nil — authorization bypassed, bridge state unchanged)
```

The stub at `x/cronos/keeper/msg_server.go:85–86` is the sole root cause. No downstream validation compensates for the missing permission check or missing state write. [1](#0-0)
