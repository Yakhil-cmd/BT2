### Title
Missing Validation of `app_pk` in `ckd()` Allows Malicious Coordinator to Extract Confidential Derived Key — (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

The `ckd()` entry point accepts an `app_pk: PublicKey` (a `blstrs::G1Projective` element) without validating that it is a non-identity point. A malicious coordinator can supply the G1 identity element as `app_pk` to all participants. Because the masking term `y · app_pk` collapses to zero, each participant's `big_c` output equals their raw signature share `S_i = x_i · H(pk ∥ app_id)`. The coordinator aggregates these shares and directly recovers the confidential derived key `msk · H(pk ∥ app_id)` — the very secret the ElGamal masking layer is designed to protect.

---

### Finding Description

In `src/confidential_key_derivation/protocol.rs`, the public entry point `ckd()` performs the following input checks: [1](#0-0) 

It verifies participant count, uniqueness, self-membership, and coordinator membership — but it never checks whether `app_pk` is the group identity. The value flows unchanged into `compute_signature_share()`: [2](#0-1) 

The masking step is:

```
big_c = big_s + app_pk * y
```

where `big_s = x_i · H(pk ∥ app_id)` is the participant's raw secret contribution. When `app_pk = G1::identity()`:

```
big_c = big_s + identity * y = big_s
```

The masking term vanishes entirely. Each participant then sends `(λ_i · Y_i, λ_i · big_s_i)` to the coordinator. The coordinator sums the `norm_big_c` values: [3](#0-2) 

```
Σ λ_i · big_s_i = Σ λ_i · x_i · H(pk ∥ app_id) = msk · H(pk ∥ app_id)
```

This is the confidential derived key, returned directly in `CKDOutput::big_c` with no masking.

By contrast, the library's own `verify_signature` function in `src/confidential_key_derivation/ciphersuite.rs` explicitly rejects identity elements for both the signature and the verifying key: [4](#0-3) 

The `BLS12381G1Group::deserialize` implementation also rejects the identity on deserialization: [5](#0-4) 

However, `app_pk` is passed as a native Rust `ElementG1` value — never deserialized through that path — so neither guard applies.

---

### Impact Explanation

When `app_pk = G1::identity()`, the coordinator receives `CKDOutput` where `big_c = msk · H(pk ∥ app_id)`. Calling `CKDOutput::unmask(Scalar::ZERO)` returns `big_c` directly: [6](#0-5) 

This is the confidential derived key — the secret that the entire CKD protocol is designed to keep hidden from the coordinator. The impact is **extraction/disclosure of a confidential derived secret**, matching the Critical allowed impact tier.

---

### Likelihood Explanation

In any real deployment the coordinator communicates the CKD request parameters — including `app_pk` — to participants out-of-band before they call `ckd()`. A malicious coordinator simply advertises `app_pk = G1Projective::identity()`. Participants have no in-library defense: `ckd()` accepts the value without error, and `compute_signature_share()` silently produces unmasked output. No cryptographic capability or key material beyond coordinator role is required.

---

### Recommendation

Add an identity-point check for `app_pk` inside `ckd()` before constructing the protocol, mirroring the guard already present in `verify_signature` and `BLS12381G1Group::deserialize`:

```rust
// In ckd(), after the coordinator membership check:
if bool::from(app_pk.is_identity()) {
    return Err(InitializationError::BadParameters(
        "app_pk must not be the identity element".to_string(),
    ));
}
```

The same guard should be added at the top of `compute_signature_share()` as a defence-in-depth measure.

---

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}
