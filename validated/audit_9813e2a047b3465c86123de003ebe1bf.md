### Title
Asymmetric `delta_gas` Adjustment in `compute_gas_refund` Causes Users to Systematically Overpay Gas Fees — (File: `basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`)

---

### Summary

In `compute_gas_refund`, the native-resource-to-gas reconciliation (`delta_gas`) is applied **one-sidedly**: when native resource consumption implies more gas than EVM gas used (`delta_gas > 0`), `gas_used` is increased and the user pays more. When native resource consumption implies *less* gas than EVM gas used (`delta_gas < 0`), `gas_used` is **not decreased**. The user's prepaid fee (stake) is therefore not fully returned: the portion corresponding to the negative delta is silently forfeited, going to the operator or being burned, rather than being refunded to the sender.

---

### Finding Description

`compute_gas_refund` is the single function that determines the final `gas_used` value for every L2 transaction (ZK flow) and every L1→L2 transaction. Its output directly controls how much of the pre-charged fee is returned to the sender.

The function computes `delta_gas` as:

```
delta_gas = (native_used / native_per_gas) as i64 - (gas_used as i64)
``` [1](#0-0) 

Only the positive branch is acted upon:

```rust
if delta_gas > 0 {
    gas_used += delta_gas as u64;
}
// TODO: return delta_gas to gas_used?
``` [2](#0-1) 

The `// TODO: return delta_gas to gas_used?` comment is an explicit developer acknowledgement that the negative case is unhandled.

The final refund to the sender is:

```
token_to_refund = gas_price * (tx_gas_limit - gas_used)
``` [3](#0-2) 

Because `gas_used` is not reduced when `delta_gas < 0`, `token_to_refund` is smaller than it should be by exactly `|delta_gas| * gas_price` tokens. Those tokens were deducted from the sender at pre-charge time: [4](#0-3) 

but are never returned. Instead they flow to the operator via `token_to_pay_operator = gas_used * gas_price_for_operator`: [5](#0-4) 

or are burned when the `burn_base_fee` feature is active.

The same `compute_gas_refund` is called for L1→L2 priority transactions, where the refund recipient receives `prepaid_fee - pay_to_operator`: [6](#0-5) 

and `pay_to_operator = gas_used * gas_price`: [7](#0-6) 

so the same shortfall applies to L1→L2 deposits.

The design intent, documented explicitly, is that only the `delta_gas > 0` direction is currently implemented: [8](#0-7) 

---

### Impact Explanation

Every transaction where EVM gas consumption exceeds what native resource consumption implies (`delta_gas < 0`) results in the sender receiving a smaller refund than they are entitled to. The shortfall is `|delta_gas| * gas_price` native tokens per transaction. These tokens are not burned in a protocol-defined sense — they are transferred to the operator (or burned under `burn_base_fee`) without the user's consent and without any protocol justification. This is a direct, systematic financial loss to every affected sender, analogous to a fee being silently extracted from a stake that was supposed to be returned in full.

---

### Likelihood Explanation

The condition `delta_gas < 0` (i.e., `native_used / native_per_gas < gas_used`) is reached whenever a transaction's EVM opcode cost exceeds its native proving cost. This is a routine occurrence: EVM gas prices for opcodes like `SLOAD` (warm, 100 gas), arithmetic, and memory operations are calibrated to Ethereum's cost model, while ZKsync OS native resources reflect RISC-V proving cycles. Any transaction whose EVM gas consumption is dominated by opcodes that are cheap to prove will trigger this path. No special attacker action is required — any ordinary user submitting a standard EVM transaction can be affected.

---

### Recommendation

Apply the `delta_gas` adjustment symmetrically. When `delta_gas < 0`, reduce `gas_used` by `|delta_gas|`, subject to the existing `minimal_gas_used` floor:

```rust
if delta_gas > 0 {
    gas_used += delta_gas as u64;
} else if delta_gas < 0 {
    // Return the unused native budget as a gas refund, floored at minimal_gas_used
    let reduction = (-delta_gas) as u64;
    gas_used = gas_used.saturating_sub(reduction).max(minimal_gas_used);
}
```

This ensures the sender's prepaid fee is fully returned when native resources are cheaper than EVM gas implies, consistent with the stated design goal that "any remaining gas left is refunded as usual."

---

### Proof of Concept

**Setup** (L2 ZK transaction, `burn_base_fee` disabled):

| Parameter | Value |
|---|---|
| `gas_limit` | 100,000 |
| `gas_price` | 1,000 |
| `native_per_gas` | 50 |
| EVM gas consumed (from ergs) | 60,000 |
| Native consumed | 1,500,000 |

**Computation**:

1. `gas_used` (from ergs) = 60,000
2. `native_used` = `full_native_limit - remaining_native` = 1,500,000 (hypothetical)
3. `delta_gas = (1,500,000 / 50) - 60,000 = 30,000 - 60,000 = -30,000`
4. Since `delta_gas < 0`, the branch is skipped. `gas_used` stays at 60,000.
5. Refund = `(100,000 - 60,000) * 1,000 = 40,000,000` tokens
6. **Correct refund** = `(100,000 - 30,000) * 1,000 = 70,000,000` tokens
7. **User loses** `30,000 * 1,000 = 30,000,000` tokens that should have been returned.

The operator receives `60,000 * 1,000 = 60,000,000` tokens instead of the correct `30,000 * 1,000 = 30,000,000`, extracting `30,000,000` tokens from the user's prepaid stake without protocol justification.
