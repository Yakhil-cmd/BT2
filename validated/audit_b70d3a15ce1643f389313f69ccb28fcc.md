Audit Report

## Title
f64 Precision Loss in `convert_tokens_to_string_representation` Causes Consent Message to Display Wrong Token Amount — (`packages/icrc-ledger-types/src/icrc21/responses.rs`)

## Summary
`convert_tokens_to_string_representation` converts a `Nat` (arbitrary-precision integer) to `f64` via `to_f64()` before formatting the token amount in ICRC-21 consent messages. IEEE 754 double precision has a 53-bit mantissa, so any integer above 2^53 that is not a power of two is rounded to the nearest representable `f64`. Because the ICRC-21 consent message is the sole mechanism wallets use to show users what they are about to sign, a malicious dApp can craft a `Nat` amount above 2^53 to produce a consent message displaying a different value than the amount that will actually be transferred or approved.

## Finding Description
The vulnerable function is confirmed at `packages/icrc-ledger-types/src/icrc21/responses.rs` lines 318–327:

```rust
fn convert_tokens_to_string_representation(
    tokens: Nat,
    decimals: u8,
) -> Result<String, Icrc21Error> {
    let tokens = tokens.0.to_f64().ok_or(Icrc21Error::GenericError { ... })?;
    Ok(format!("{}", tokens / 10_f64.pow(decimals)))
}
```

`tokens.0` is a `num_bigint::BigUint`. `to_f64()` (from `num_traits::ToPrimitive`) is a lossy conversion: any integer requiring more than 53 significant bits is silently rounded to the nearest representable `f64`. The rounded value is then formatted and embedded in the `GenericDisplayMessage` string.

This function is called from three places, all on the `GenericDisplayMessage` branch:
- `add_amount` (L113) — displays the transfer/approval amount
- `add_fee` (L141) — displays the transaction fee
- `add_allowance` (L186) — displays the requested allowance for `icrc2_approve`

The `FieldsDisplayMessage` branch uses `nat_to_u64` (L120, L156, L195), which is exact but rejects values above `u64::MAX`.

Full call chain from an unprivileged ingress call:
1. Any caller → `icrc21_consent_message` endpoint (public, no authentication)
2. → `build_icrc21_consent_info` (lib.rs L324) decodes the `arg` bytes and extracts the `Nat` amount
3. → `ConsentMessageBuilder::build()` → `add_amount` / `add_allowance` / `add_fee`
4. → `convert_tokens_to_string_representation` → `to_f64()` → precision loss

No existing guard prevents a `Nat` value above 2^53 from reaching `to_f64()`. The only guard in `build_icrc21_consent_info` is a size check on `arg` bytes (L333), which does not constrain the numeric value of the encoded `Nat`.

## Impact Explanation
The ICRC-21 consent message is the only information a wallet presents to the user before they authorize a transaction. If the displayed amount differs from the amount encoded in `arg`, the user signs a transaction for a different value than what they saw. The actual on-chain transfer uses the `arg` bytes directly — the display string has no effect on execution — so the ledger processes the true amount while the user believed they approved a different one.

Concretely, with `decimals = 0` and `amount = 9007199254740993` (2^53 + 1):
- Displayed: `"9007199254740992"`
- Actual transfer: `9007199254740993`

The discrepancy grows for amounts further above 2^53: at 2^63 the ULP is 1024, meaning a malicious dApp can craft an amount that displays as N but transfers N + 1024. This constitutes a significant ICRC ledger security impact with concrete user harm — the consent message integrity guarantee is broken — falling under the High bounty category.

## Likelihood Explanation
Triggering this requires only a valid `ConsentMessageRequest` with a `Nat` amount whose `BigUint` value exceeds 2^53. The `icrc21_consent_message` endpoint is public and unauthenticated. No privileged role, key, or governance action is needed. The `GenericDisplayMessage` display type is the default when no `device_spec` is provided, making it the common path. The only practical constraint is that the token's total supply must allow such a balance, which is true for any token with low decimals or high supply.

## Recommendation
Replace the lossy `f64` conversion with exact integer arithmetic using `num_bigint::BigUint` division and modulo:

```rust
fn convert_tokens_to_string_representation(
    tokens: Nat,
    decimals: u8,
) -> Result<String, Icrc21Error> {
    let divisor = num_bigint::BigUint::from(10u64).pow(decimals as u32);
    let whole = &tokens.0 / &divisor;
    let remainder = &tokens.0 % &divisor;
    if decimals == 0 || remainder.is_zero() {
        Ok(whole.to_string())
    } else {
        Ok(format!("{}.{:0>width$}", whole, remainder, width = decimals as usize))
    }
}
```

This preserves full precision for arbitrarily large `Nat` values and produces a correctly rounded decimal string.

## Proof of Concept

```rust
#[test]
fn test_f64_precision_loss() {
    use candid::Nat;
    use num_bigint::BigUint;

    // 2^53 + 1: first integer not exactly representable as f64
    let amount = Nat(BigUint::from(9007199254740993u64));
    let result = convert_tokens_to_string_representation(amount, 0).unwrap();
    // Fails: result is "9007199254740992", not "9007199254740993"
    assert_eq!(result, "9007199254740993");
}
```

The assertion fails because `9007199254740993u64 as f64` rounds to `9007199254740992.0`. The consent message displays the wrong amount. This test can be added directly to `packages/icrc-ledger-types/src/icrc21/responses.rs` and run with `cargo test`.
