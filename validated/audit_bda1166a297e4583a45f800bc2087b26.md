### Title
Unpermissioned `update_rewards` Caller Allows Any Address to Disable Staker Rewards — (`File: src/staking/staking.cairo`)

### Summary

`IStakingRewardsManager::update_rewards` in `src/staking/staking.cairo` performs no caller validation. Any unprivileged address can invoke it with `disable_rewards: true`, consuming the single allowed call per block and preventing the staker from receiving consensus-epoch rewards for that block. Repeated across every block, this permanently freezes a staker's unclaimed yield.

### Finding Description

The analog vulnerability class from the external report is **missing authorization on a state-mutating entry point**. In the Django context, `@csrf_exempt` removed the authentication gate from POST endpoints. Here, `update_rewards` is a reward-state-mutating function that carries no `get_caller_address()` check whatsoever.

`update_rewards` is gated only by a per-block idempotency guard: [1](#0-0) 

```rust
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
    let current_block_number = starknet::get_block_number();
    assert!(
        current_block_number > self.last_reward_block.read(),
        "{}",
        Error::REWARDS_ALREADY_UPDATED,
    );
```

There is no assertion of the form `assert!(get_caller_address() == <trusted_address>)`. The `disable_rewards: bool` parameter is fully attacker-controlled.

The flow tests confirm the semantics: when `disable_rewards: true` is passed during an active consensus-rewards epoch, the staker receives zero rewards for that block: [2](#0-1) 

Because `REWARDS_ALREADY_UPDATED` prevents a second call in the same block, a single attacker call with `disable_rewards: true` exhausts the reward slot for that block. No legitimate caller can recover it.

### Impact Explanation

**High — Permanent freezing of unclaimed yield.**

Once consensus rewards are active, an attacker who front-runs `update_rewards` every block with `disable_rewards: true` causes the targeted staker to accrue zero rewards indefinitely. The staker's `unclaimed_rewards_own` is never incremented; pool members of that staker also receive no rewards (pool rewards are distributed inside the same call). The funds are not stolen outright but are permanently withheld from the staker and delegators.

### Likelihood Explanation

- The function is public with no access control.
- The attacker needs only to submit a transaction before the legitimate `update_rewards` call each block — a straightforward mempool front-run or a bot that monitors the chain.
- No privileged role, leaked key, or external dependency is required.
- The cost to the attacker is only gas; the damage to the victim is total loss of staking yield.

### Recommendation

Add an explicit caller restriction to `update_rewards`. The legitimate callers are either a designated keeper address or the staker/operational address itself. For example:

```rust
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
    // Add: only the staker's operational address (or a whitelisted keeper) may call this.
    let caller = get_caller_address();
    let staker_info = self.internal_staker_info(:staker_address);
    assert!(
        caller == staker_address || caller == staker_info.operational_address,
        "{}",
        Error::CALLER_CANNOT_UPDATE_REWARDS,
    );
    ...
}
```

Alternatively, remove the `disable_rewards` parameter entirely and derive the value deterministically from on-chain state, eliminating the attacker-controlled input.

### Proof of Concept

1. Staker registers and consensus rewards become active.
2. Each block, attacker calls:
   ```
   staking_rewards_dispatcher.update_rewards(
       staker_address: victim_staker,
       disable_rewards: true,
   )
   ```
   before any legitimate call arrives.
3. `REWARDS_ALREADY_UPDATED` blocks any subsequent call in the same block.
4. `staker_info.unclaimed_rewards_own` remains zero every epoch.
5. Staker calls `claim_rewards` and receives nothing.

The per-block guard at `last_reward_block` is the mechanism that makes the attack permanent: one attacker call per block is sufficient to zero out all rewards for the victim indefinitely. [3](#0-2) [4](#0-3)
