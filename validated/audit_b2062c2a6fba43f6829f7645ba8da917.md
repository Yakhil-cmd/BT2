### Title
Unprivileged EVM Contract Can Transfer Native Bank Tokens From Any Address via Bank Precompile `transfer` - (File: `x/cronos/keeper/precompiles/bank.go`)

### Summary

The `transfer` method of the Bank precompile (`0x0000000000000000000000000000000000000064`) accepts the `sender` address as a caller-supplied ABI argument and performs `bankKeeper.SendCoins(ctx, from, to, ...)` without verifying that `sender == contract.Caller()`. Any EVM contract can therefore drain `evm/<itself>` native bank tokens from any victim address that holds them, without the victim's consent.

### Finding Description

In `x/cronos/keeper/precompiles/bank.go`, the `TransferMethodName` case unpacks `sender` directly from the ABI-encoded input:

```go
sender := args[0].(common.Address)
recipient := args[1].(common.Address)
amount := args[2].(*big.Int)
...
from := sdk.AccAddress(sender.Bytes())
to := sdk.AccAddress(recipient.Bytes())
...
denom := EVMDenom(contract.Caller())   // evm/<calling_contract>
...
bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt))
``` [1](#0-0) 

The `denom` is correctly tied to `contract.Caller()` (the calling contract's address), but the `from` account is taken verbatim from the ABI argument with no check that `sender == contract.Caller()`. There is no guard equivalent to `require(sender == msg.sender)`.

The same structural flaw exists in the `BurnMethodName` branch: the `recipient` argument (the address to burn from) is also caller-supplied with no caller-identity check. [2](#0-1) 

### Impact Explanation

**Critical — Unauthorized transfer of native bank tokens.**

Any EVM contract can call `bank.transfer(victim, attacker, amount)` at the bank precompile. The precompile will execute `bankKeeper.SendCoins(ctx, victim, attacker, evm/<contract>_coins)`, moving the victim's `evm/<contract>` native bank tokens to the attacker with no approval from the victim. This is a direct, unprivileged, unauthorized balance change for EVM-denominated assets managed by the Cosmos bank module.

The `burn` variant allows any contract to destroy another account's `evm/<contract>` tokens without consent.

### Likelihood Explanation

Any user who holds `evm/<contract>` tokens (obtained via `bank.mint` or by receiving them) is at risk the moment the contract's deployer (or any caller of that contract) invokes `bank.transfer(victim, attacker, amount)`. No special role or privilege is required — any deployed EVM contract can exploit this against any holder of its associated native denom.

### Recommendation

In the `TransferMethodName` case, enforce that the `sender` argument equals the EVM caller:

```go
if sender != contract.Caller() {
    return nil, errors.New("sender must be the caller")
}
```

Similarly, in the `BurnMethodName` case, enforce that the address being burned from equals `contract.Caller()`. This mirrors the fix recommended in the external report: the first parameter of the transfer must be `msg.sender`.

### Proof of Concept

1. Attacker deploys `MaliciousContract` at address `0xDEAD`.
2. Victim receives `evm/0xDEAD` tokens (e.g., via `bank.mint(victim, 1000)` called from `MaliciousContract`).
3. Attacker calls `MaliciousContract`, which calls the bank precompile at `0x64`:
   ```solidity
   IBankModule(0x64).transfer(victim, attacker, 1000);
   ```
4. Inside `BankContract.Run`, `contract.Caller()` is `0xDEAD`, so `denom = "evm/0xDEAD"`. `sender = victim` (from ABI args). No caller check is performed.
5. `bankKeeper.SendCoins(ctx, victim, attacker, [{denom:"evm/0xDEAD", amount:1000}])` executes — victim's tokens are transferred to attacker without consent. [3](#0-2)
