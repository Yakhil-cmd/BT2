### Title
Cross-Network Replay of COA Ownership Proof Signatures via Network-Agnostic Domain Separation Tag - (File: `fvm/evm/stdlib/contract.cdc`)

### Summary
`EVM.validateCOAOwnershipProof` verifies Cadence account signatures over caller-supplied `signedData` using the hardcoded domain separation tag `"FLOW-V0.0-user"`. This tag is identical across all Flow networks (Mainnet, Testnet, Previewnet, etc.). Because the tag contains no network identifier and the function does not enforce that `signedData` encodes any chain-specific context, a valid COA ownership proof signature produced on one Flow network is cryptographically valid on every other Flow network where the same account and key exist.

### Finding Description
`validateCOAOwnershipProof` in `fvm/evm/stdlib/contract.cdc` verifies a set of Cadence account signatures over an arbitrary `signedData` byte array:

```cadence
let isValid = keyList.verify(
    signatureSet: signatureSet,
    signedData: signedData,
    domainSeparationTag: "FLOW-V0.0-user"   // ← hardcoded, network-agnostic
)
``` [1](#0-0) 

The tag `"FLOW-V0.0-user"` carries no network identity. Flow account addresses are deterministic and the same address can exist on Mainnet and Testnet with the same key material. A signature over an arbitrary `signedData` blob produced on Mainnet is therefore byte-for-byte valid when submitted on Testnet (or any other Flow network), because the cryptographic verification is identical.

The function's own NatSpec comment acknowledges a cross-address replay risk but is silent on cross-network replay:

> "Callers building off-chain authentication flows on top of this function should ensure `signedData` encodes `evmAddress` to prevent cross-address replay." [2](#0-1) 

No analogous guidance is given for network replay, and the protocol itself does not enforce it.

The EVM-side entry point is the `verifyCOAOwnershipProof(address,bytes32,bytes)` precompile in the Cadence Arch contract, callable by any EVM contract or EOA: [3](#0-2) 

`DecodeABIEncodedProof` extracts the `signedData` (the `bytes32` argument) and the encoded proof, then passes them directly to `validateCOAOwnershipProof` with no network-binding check: [4](#0-3) 

The `COAOwnershipProofInContext` struct treats `SignedData` as an opaque byte slice with no chain-ID field: [5](#0-4) 

### Impact Explanation
Any EVM contract or off-chain authentication flow that calls `verifyCOAOwnershipProof` to gate access (e.g., to authorize a bridge withdrawal, a governance vote, or an asset transfer) can be bypassed by an attacker who replays a legitimate COA ownership proof signature that the victim produced on a different Flow network. The attacker gains the ability to impersonate the victim's COA ownership on the target network without possessing the victim's private key, leading to unauthorized account mutation or unauthorized asset access in any protocol built on top of this primitive.

### Likelihood Explanation
Flow account addresses are deterministic and the same address commonly exists across Mainnet, Testnet, and Previewnet. Users routinely sign messages on Testnet during development and testing. Any such signature is immediately replayable on Mainnet if the account exists there with the same key. The attack requires only that the attacker observe a valid proof (e.g., from a public transaction on Testnet) and submit it on Mainnet — no privileged access, no key compromise, and no brute force is needed.

### Recommendation
Include the Flow chain ID in the domain separation tag used for signature verification, making proofs network-specific:

```cadence
// Derive a network-scoped tag, e.g. "FLOW-V0.0-user-flow-mainnet"
let networkTag = "FLOW-V0.0-user-".concat(self.chainID)

let isValid = keyList.verify(
    signatureSet: signatureSet,
    signedData: signedData,
    domainSeparationTag: networkTag
)
```

Alternatively, require that `signedData` encodes the chain ID (and the `evmAddress`) as part of the protocol specification, and enforce this in `validateCOAOwnershipProof` before performing signature verification.

### Proof of Concept
1. **Setup**: Alice holds Flow account `0xABCD` on both Mainnet and Testnet with the same ECDSA key.
2. **Victim action (Testnet)**: Alice signs `signedData = keccak256("authorize_bridge_withdrawal")` on Testnet to prove COA ownership to a bridge contract. The signature is produced with `domainSeparationTag = "FLOW-V0.0-user"`.
3. **Attacker observes**: The attacker reads Alice's `signedData`, `keyIndices`, `signatures`, and `CapabilityPath` from the public Testnet transaction.
4. **Replay on Mainnet**: The attacker calls the Mainnet EVM bridge contract, which internally calls `verifyCOAOwnershipProof(0xABCD_evm, signedData, encodedProof)`.
5. **Precompile execution**: `DecodeABIEncodedProof` reconstructs the `COAOwnershipProofInContext` and calls `validateCOAOwnershipProof` on the Mainnet EVM contract.
6. **Verification passes**: `keyList.verify(..., domainSeparationTag: "FLOW-V0.0-user")` succeeds because the tag is identical on Mainnet, Alice's key exists at `0xABCD` on Mainnet, and the signature is cryptographically valid.
7. **Result**: The attacker is granted Alice's COA ownership rights on Mainnet without her consent.

func DecodeABIEncodedProof(input []byte) (*types.COAOwnershipProofInContext, error) {
	index := 0
	caller, err := ReadAddress(input, index)
	index += FixedSizeUnitDataReadSize
	if err != nil {
		return nil, err
	}

	hash, err := ReadBytes32(input, index)
	index += Bytes32DataReadSize
	if err != nil {
		return nil, err
	}

	encodedProof, err := ReadBytes(input, index)
	if err != nil {
		return nil, err
	}

	return types.NewCOAOwnershipProofInContext(
		hash,
		types.Address(caller),
		encodedProof,
	)
}
