### Title
Unauthenticated Invoke Transaction Bypasses Signature Verification via `skip_stateful_validations` Race Window - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator allows any invoke transaction with nonce=1 to bypass `__validate__` (signature verification) when the target account has **any** pending transaction in the mempool and an on-chain nonce of zero. An unprivileged attacker who observes a victim's `deploy_account` in the mempool can inject an invoke with an invalid signature that is admitted without signature verification, displaces the victim's legitimate invoke via fee escalation, and corrupts the victim's account nonce upon execution.

### Finding Description

**Root cause — `skip_stateful_validations` does not verify the pooled transaction is a `deploy_account`**

`skip_stateful_validations` (lines 429–461) returns `true` (skip `__validate__`) when three conditions hold simultaneously:

1. The incoming transaction is an `Invoke` with `nonce == 1`
2. The on-chain account nonce is `0`
3. `account_tx_in_pool_or_recent_block(sender)` returns `true` [1](#0-0) 

`account_tx_in_pool_or_recent_block` is implemented as:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [2](#0-1) 

`tx_pool.contains_account` returns `true` if **any** transaction from that address is in the pool — it does not check whether the pooled transaction is specifically a `deploy_account`. The code comment claims this is sufficient because "it means that either it has a deploy_account transaction or transactions with future nonces that passed validations," but this reasoning does not prevent an attacker from exploiting the window.

**How the bypass propagates**

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` sets `execution_flags.validate = false`:

```rust
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [3](#0-2) 

The blockifier's `StatefulValidator::perform_validations` then skips the `__validate__` entry-point call entirely for the Invoke path:

```rust
ApiTransaction::Invoke(_) => {
    tx.perform_pre_validation_stage(self.state(), &tx_context)?;
    if !tx.execution_flags.validate {
        return Ok(());
    }
    // `__validate__` call.
``` [4](#0-3) 

`perform_pre_validation_stage` uses `strict_nonce_check = false`, so nonce=1 is accepted when account_nonce=0:

```rust
let valid_nonce = if strict {
    account_nonce == incoming_tx_nonce
} else {
    account_nonce <= incoming_tx_nonce
};
``` [5](#0-4) 

The transaction is then forwarded to the mempool via `add_tx` with no signature check having occurred.

**Attack flow**

1. Victim submits `deploy_account` for account `A` (address is deterministic from class hash + salt + constructor calldata — publicly computable).
2. Attacker observes the `deploy_account` in the mempool; `tx_pool.contains_account(A)` is now `true`.
3. Attacker submits `Invoke(sender=A, nonce=1, signature=[0xdead, 0xbeef])` with a fee higher than the victim's legitimate invoke.
4. Gateway stateful validation:
   - `validate_nonce`: passes — `account_nonce=0 ≤ tx_nonce=1 ≤ max_allowed_nonce_gap` [6](#0-5) 
   - `validate_by_mempool`: passes — mempool nonce check: `tx_nonce(1) ≥ resolve_nonce(0)` [7](#0-6) 
   - `skip_stateful_validations`: returns `true` — `tx_nonce==1`, `account_nonce==0`, `account_tx_in_pool_or_recent_block==true` [8](#0-7) 
   - `run_validate_entry_point(skip_validate=true)`: `__validate__` is **never called**. [9](#0-8) 
5. Attacker's invalid invoke is admitted to the mempool. Because its fee is higher, the mempool's fee-escalation logic replaces the victim's legitimate invoke(nonce=1).
6. Batcher executes `deploy_account` → account `A` deployed, on-chain nonce becomes 1.
7. Batcher executes attacker's invoke with `execution_flags.validate=true` (default for sequencing): [10](#0-9) 
   - `perform_pre_validation_stage`: nonce=1 matches, nonce incremented to **2**.
   - `__validate__` runs and **fails** (invalid signature).
   - Transaction reverts; on-chain nonce is now **2**.
8. Victim's legitimate invoke(nonce=1) is permanently rejected: nonce too old.

### Impact Explanation

An unprivileged attacker can inject an invoke transaction carrying an arbitrary (invalid) signature into the mempool, bypassing the gateway's only signature-verification gate. This directly satisfies **"High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing."** Secondary consequences include:

- The victim's legitimate invoke is evicted from the mempool via fee escalation.
- The victim's account nonce is permanently advanced to 2 by the reverted attacker transaction, making all previously-queued nonce=1 transactions permanently invalid.
- The attack is repeatable for every new account deployment observed in the mempool.

### Likelihood Explanation

- The mempool is publicly observable (P2P gossip, RPC `starknet_pendingTransactions`).
- The victim's account address is deterministically computable from the `deploy_account` fields.
- The attacker only needs to submit a single transaction with a fee slightly above the victim's invoke fee.
- No privileged access, special keys, or on-chain state is required.
- The attack window is the entire period between the `deploy_account` entering the mempool and the batcher committing the block containing it — typically multiple seconds to minutes.

### Recommendation

Replace the generic `account_tx_in_pool_or_recent_block` check with a type-specific check that confirms the pooled transaction is a `deploy_account`:

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            // Only skip if a deploy_account specifically is pending, not any tx.
            return mempool_client
                .deploy_account_in_pool_or_recent_block(tx.sender_address())
                .await
                ...
        }
    }
    Ok(false)
}
```

Alternatively, the mempool can expose a `has_pending_deploy_account(address)` query that checks the transaction type of the pooled entry for the given address at nonce=0.

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
