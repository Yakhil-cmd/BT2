### Title
Cross-Address COA Ownership Proof Replay Enables Unauthorized ERC-1271 Authentication - (File: `fvm/evm/stdlib/contract.cdc`)

### Summary

`EVM.validateCOAOwnershipProof` does not enforce that `signedData` encodes the `evmAddress` being authenticated. A signature produced by a Cadence account to prove ownership of one COA (Cadence-Owned Account) can be replayed by an attacker to prove ownership of a *different* COA owned by the same Cadence account, bypassing ERC-1271 authentication on the second COA.

### Finding Description

`EVM.validateCOAOwnershipProof` accepts six parameters: a Cadence `address`, a `path` to a public COA capability, `signedData`, `keyIndices`, `signatures`, and an `evmAddress`. The function verifies that the signatures over `signedData` are valid for the account at `address`, then checks that the COA resource at `path` has the EVM address `evmAddress`. Critically, it does **not** verify that `signedData` encodes `evmAddress`. [1](#0-0) 

The code itself documents this gap:

> "Note: this function does not enforce that `signedData` includes `evmAddress`. In principle, a signature produced for one purpose could be replayed here against a different COA owned by the same Cadence account." [2](#0-1) 

The COA's EVM-side `isValidSignature` (ERC-1271) calls the `verifyCOAOwnershipProof` precompile with `address(this)` as the EVM address argument: [3](#0-2) 

This means the `evmAddress` is correctly bound to the calling contract's address, but the `_hash` (the `signedData`) is not required to include that address.

**Attack path:**

1. Alice owns two COAs: `COA_A` (at `/public/coa_a`) and `COA_B` (at `/public/coa_b`), both published as public capabilities.
2. Alice signs data `D` (e.g., a login challenge or ERC-1271 hash) for a protocol that uses `COA_A`. The signed data `D` does not include `COA_A`'s EVM address (which is the normal, undocumented-as-dangerous usage).
3. An attacker obtains Alice's signature over `D` (e.g., from a public transaction or off-chain interaction).
4. The attacker constructs a proof: `{address: alice_cadence_addr, path: /public/coa_b, signatures: [alice_sig_over_D], keyIndices: [...]}`.
5. The attacker calls `COA_B.isValidSignature(D, encoded_proof)`.
6. `verifyCOAOwnershipProof(COA_B_address, D, encoded_proof)` is invoked on the Cadence Arch precompile.
7. `validateCOAOwnershipProof` verifies Alice's signatures over `D` — **valid**, because Alice did sign `D`.
8. It checks that the COA at `/public/coa_b` has address `COA_B_address` — **valid**.
9. Returns `isValid: true`.

The attacker has now impersonated Alice as the ERC-1271 owner of `COA_B` using a signature Alice produced for `COA_A`. [4](#0-3) 

The `validateCOAOwnershipProof` function is `access(all)`, callable by any Cadence script or transaction, and the `verifyCOAOwnershipProof` precompile is callable from any EVM contract. [5](#0-4) 

### Impact Explanation

An attacker who obtains any valid signature from a Cadence account over arbitrary data can replay it to pass ERC-1271 authentication (`isValidSignature`) for any other COA owned by the same Cadence account. If an EVM protocol uses ERC-1271 to authorize asset transfers, withdrawals, or privileged operations on a COA, the attacker can drain or manipulate that COA's assets without the owner's consent. The `deposit` function on `EVMAddress` is `access(all)` and requires no ownership, so assets can be present in COA_B independently. [6](#0-5) 

### Likelihood Explanation

The attack requires:
1. The victim to own multiple COAs with public capabilities — a realistic pattern as DeFi usage grows.
2. The victim to have signed data that does not include the COA EVM address — this is the default behavior since no enforcement exists and the warning is only in a code comment, not surfaced to users or wallet developers.
3. The attacker to know the victim's Cadence address and the capability path of the second COA — both are publicly discoverable on-chain.

The risk is low today because most accounts have one COA, but the code comment itself acknowledges this assumption may not hold: "Flow wallets historically create at most one COA per account." As multi-COA patterns emerge (e.g., separate COAs for different DeFi protocols), this becomes exploitable. [7](#0-6) 

### Recommendation

Enforce that `signedData` encodes `evmAddress` inside `validateCOAOwnershipProof`. After verifying the signatures, check that `evmAddress` appears within `signedData` (e.g., as a prefix or via a domain-separation scheme). Alternatively, derive `signedData` from a canonical encoding that includes `evmAddress` at the protocol level, removing the burden from callers. At minimum, add a strong `@param` warning in the Cadence NatSpec and in the EVM-side `isValidSignature` implementation that `_hash` MUST encode `address(this)` to prevent cross-COA replay.

### Proof of Concept

```cadence
// Alice owns COA_A at /public/coa_a and COA_B at /public/coa_b.
// Alice previously signed `challengeData` for COA_A (without including COA_A's address).
// Attacker replays that signature against COA_B:

import EVM from 0xEVMAddress

access(all) fun main(
    aliceCadenceAddr: Address,
    challengeData: [UInt8],
    keyIndices: [UInt64],
    aliceSignatures: [[UInt8]],
    coaBAddress: [UInt8; 20]
): EVM.ValidationResult {
    // Path points to COA_B, not COA_A
    return EVM.validateCOAOwnershipProof(
        address: aliceCadenceAddr,
        path: /public/coa_b,       // attacker uses COA_B's path
        signedData: challengeData,  // data Alice signed for COA_A
        keyIndices: keyIndices,
        signatures: aliceSignatures,
        evmAddress: coaBAddress     // COA_B's EVM address
    )
    // Returns isValid: true — cross-address replay succeeds
}
```

The same replay is achievable from EVM by calling `COA_B.isValidSignature(challengeData_hash, encoded_proof_with_coa_b_path)`, which internally calls `verifyCOAOwnershipProof(address(this)=COA_B, challengeData_hash, proof)`. [8](#0-7)

func (f *proofVerifier) ComputeGas(input []byte) uint64 {
	// we compute the gas using a fixed base fee and extra fees
	// per signatures. Note that the input data is already trimmed from the function selector
	// and the remaining is ABI encoded of the inputs

	// skip to the encoded signature part of args (skip address and bytes32 data part)
	index := EncodedAddressSize + Bytes32DataReadSize
	// Reading the encoded signature bytes
	encodedSignature, err := ReadBytes(input, index)
	if err != nil {
		// if any error run would anyway fail, so returning any non-zero value here is fine
		return ProofVerifierBaseGas
	}
	// this method would return the number of signatures from the encoded signature data
	// this saves the extra time needed for full decoding
	// given ComputeGas function is called before charging the gas, we need to keep
	// this function as light as possible
	count, err := types.COAOwnershipProofSignatureCountFromEncoded(encodedSignature)
	if err != nil {
		// if any error run would anyway fail, so returning any non-zero value here is fine
		return ProofVerifierBaseGas
	}
	return ProofVerifierBaseGas + uint64(count)*ProofVerifierGasMultiplerPerSignature
}

func (f *proofVerifier) Run(input []byte) ([]byte, error) {
	proof, err := DecodeABIEncodedProof(input)
	if err != nil {
		return nil, err
	}
	verified, err := f.proofVerifier(proof)
	if err != nil {
		return nil, err
	}

	buffer := make([]byte, EncodedBoolSize)
	return buffer, EncodeBool(verified, buffer, 0)
}

					types.FlowAddressSemaType,
					types.PublicPathSemaType,
					types.SignedDataSemaType,

					types.AddressBytesSemaType,

			},

			if types.IsAFatalError(err) {
