### Title
Missing `launcher_id` Parameter in WalletConnect `chia_farmer.set_payout_instructions` Command Allows Silent Redirection of Pool Farming Rewards - (File: packages/gui/src/electron/commands/Commands.ts)

### Summary
The `chia_farmer.set_payout_instructions` WalletConnect command definition omits the `launcher_id` parameter from its `params` schema. A malicious dapp granted this command can silently specify any plot NFT's `launcher_id` while the user confirmation dialog only shows the new `payout_instructions` value. The user cannot determine which NFT's pool payout address is being changed, enabling an attacker to redirect pool farming rewards to an attacker-controlled address across any of the victim's plot NFTs.

### Finding Description
The `chia_farmer.set_payout_instructions` entry in `Commands.ts` defines only one parameter for the user-facing confirmation dialog:

```ts
'chia_farmer.set_payout_instructions': {
  ...
  params: [
    { name: 'payout_instructions', label: ..., type: 'string' },
  ],
},
``` [1](#0-0) 

However, the underlying daemon RPC `set_payout_instructions` requires **two** fields: `launcher_id` and `payout_instructions`, as reflected in the `Farmer` service:

```ts
async setPayoutInstructions(args: { launcherId: string; payoutInstructions: string }) {
  return this.command<void>('set_payout_instructions', args);
}
``` [2](#0-1) 

The `params` array in `Commands.ts` controls what is displayed in the WalletConnect confirmation dialog. Parameters not listed there are not shown to the user but are still forwarded to the daemon. A dapp can therefore pass an arbitrary `launcher_id` alongside `payout_instructions`; the user sees only the new payout address and has no visibility into which plot NFT is being targeted.

The `ParamSchema` type explicitly supports a `hide` flag for parameters that should be concealed from the main dialog view but still shown in JSON details. `launcher_id` is not even listed as hidden — it is entirely absent, so it is invisible to the user in all confirmation UI paths. [3](#0-2) 

### Impact Explanation
A WalletConnect dapp that has been granted `chia_farmer.set_payout_instructions` can call it with:
- `payout_instructions` = attacker's XCH address (shown to user)
- `launcher_id` = any of the victim's plot NFT launcher IDs (hidden from user)

The user sees a confirmation dialog asking "Are you sure you want to set payout instructions?" and sees only the new address. They cannot tell which NFT is being targeted. Upon approval, all future pool rewards for that NFT are paid to the attacker's address. This is a direct, permanent redirection of pooled farming rewards — a Critical/High impact per the allowed scope (unauthorized payout change affecting pooled farming rewards).

### Likelihood Explanation
Any dapp that can convince a user to grant `chia_farmer.set_payout_instructions` permission via WalletConnect can exploit this. The permission grant dialog shows the command name, which sounds benign (e.g., a pool management dapp). Once granted, the dapp can silently target any launcher ID. The user has no way to detect the mismatch between the NFT they intended to configure and the one actually targeted.

### Recommendation
Add `launcher_id` to the `params` array of `chia_farmer.set_payout_instructions` in `Commands.ts` so it is displayed in the confirmation dialog:

```ts
'chia_farmer.set_payout_instructions': {
  ...
  params: [
    { name: 'launcher_id', label: () => i18n._({ id: 'Launcher ID' }), type: 'string' },
    { name: 'payout_instructions', label: () => i18n._({ id: 'Payout Instructions' }), type: 'string' },
  ],
},
```

This ensures the user can see and verify which plot NFT's payout address is being changed before approving.

### Proof of Concept
1. Attacker registers a malicious dapp and connects via WalletConnect.
2. Dapp requests permission for `chia_farmer.set_payout_instructions`. User grants it.
3. Dapp calls:
   ```json
   {
     "command": "chia_farmer.set_payout_instructions",
     "params": {
       "launcher_id": "<victim_nft_launcher_id>",
       "payout_instructions": "<attacker_xch_address>"
     }
   }
   ```
4. The WalletConnect confirmation dialog shows only `payout_instructions = <attacker_xch_address>`. `launcher_id` is not rendered.
5. User approves, believing they are setting payout instructions for a specific NFT they chose.
6. The daemon receives both fields and updates the payout instructions for the targeted NFT.
7. All subsequent pool rewards for that NFT are sent to the attacker's address. [4](#0-3) [5](#0-4)
