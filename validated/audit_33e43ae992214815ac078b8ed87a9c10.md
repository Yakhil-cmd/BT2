Audit Report

## Title
Unprivileged Caller Can Force-Register Any Principal as SNS Swap Buyer via `refresh_buyer_tokens` — (`rs/sns/swap/canister/canister.rs`)

## Summary
The `refresh_buyer_tokens` canister endpoint accepts an arbitrary `buyer` field without verifying it matches the caller. Any unprivileged principal can supply a victim's principal ID, causing the victim to be registered as a swap buyer if their subaccount holds sufficient ICP. Once registered, the victim is blocked from using `error_refund_icp` to self-rescue their ICP until `sweep_icp` completes finalization, and if the swap commits, their ICP is permanently converted into SNS neurons they never consented to acquire.

## Finding Description
In `rs/sns/swap/canister/canister.rs` at lines 130–133, when `arg.buyer` is non-empty, the endpoint resolves the buyer principal directly from the caller-supplied string with no assertion that it equals `caller_principal_id()`:

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    PrincipalId::from_str(&arg.buyer).unwrap()  // no caller == buyer check
};
```

The resolved principal is passed directly to `refresh_buyer_token_e8s` (`swap.rs` line 1137). That function queries the ICP ledger balance of `principal_to_subaccount(&buyer)` on the swap canister (`swap.rs` lines 1153–1163). If the balance meets `min_participant_icp_e8s`, it inserts a `BuyerState` entry for the victim (`swap.rs` lines 1285–1288):

```rust
self.buyers
    .entry(buyer.to_string())
    .or_insert_with(|| BuyerState::new(0))
    .set_amount_icp_e8s(new_balance_e8s);
```

`BuyerState::new(0)` initializes `transfer_success_timestamp_seconds: 0` (`types.rs` lines 540–551). After the swap closes, `error_refund_icp` checks for a `BuyerState` with `transfer_success_timestamp_seconds == 0` and returns a precondition error (`swap.rs` lines 1950–1960), blocking the victim from recovering their ICP until `sweep_icp` completes.

The `confirmation_text` check (`swap.rs` lines 363–384) provides no real barrier: the text is stored in `Init`, which is publicly readable via the `get_init` query endpoint. An attacker trivially reads and supplies it.

## Impact Explanation
If the swap **commits**: after `sweep_icp` runs, the victim's ICP is transferred to SNS governance and the victim receives unwanted, potentially illiquid SNS neurons — a permanent, unconsented conversion of ICP. If the swap **aborts**: the victim's ICP is eventually returned, but they are denied the ability to self-rescue via `error_refund_icp` during the window between swap close and `sweep_icp` completion. This constitutes unauthorized disposition of user funds and forced participation in SNS governance, matching the **High** impact category: unauthorized access to/disposition of ledger-controlled funds with concrete user harm.

## Likelihood Explanation
The attack requires no special privileges — only the ability to send an ingress message to the swap canister. The attacker needs to know the victim's principal ID (publicly observable on-chain) and that the victim's subaccount holds at least `min_participant_icp_e8s` ICP (observable by querying the ICP ledger). The realistic trigger is a user who sent ICP to their swap subaccount but has not yet called `refresh_buyer_tokens` (the normal two-step participation flow), or a user who accidentally sent ICP and intended to recover it after the swap closed. The attack window is the entire OPEN lifecycle of the swap. The `confirmation_text` mitigation is ineffective as the value is publicly readable.

## Recommendation
In the canister endpoint, enforce that the `buyer` field, if provided, must equal the caller:

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    let specified = PrincipalId::from_str(&arg.buyer).unwrap();
    assert_eq!(specified, caller_principal_id(),
        "buyer field must match the caller");
    specified
};
```

Alternatively, remove the `buyer` field from `RefreshBuyerTokensRequest` entirely and always use `caller_principal_id()`.

## Proof of Concept
1. Victim accidentally sends `min_participant_icp_e8s` ICP to `subaccount(swap_canister, victim_principal)` on the ICP ledger.
2. Attacker (any principal) reads `confirmation_text` via `get_init`, then calls:
   ```
   refresh_buyer_tokens(RefreshBuyerTokensRequest {
       buyer: victim_principal.to_string(),
       confirmation_text: <value from get_init>,
   })
   ```
3. The swap canister queries the victim's subaccount balance, finds sufficient ICP, and inserts `BuyerState { icp: TransferableAmount { amount_e8s: X, transfer_success_timestamp_seconds: 0, ... } }` for the victim.
4. Swap closes (COMMITTED). Victim calls `error_refund_icp` — blocked by the `transfer_success_timestamp_seconds == 0` guard at `swap.rs` lines 1951–1959.
5. `sweep_icp` runs, transferring victim's ICP to SNS governance. Victim receives unwanted SNS neurons.

A deterministic integration test using PocketIC or `StateMachine` can reproduce this by: (a) funding a victim subaccount, (b) calling `refresh_buyer_tokens` from a different principal with the victim's ID, (c) advancing the swap to COMMITTED, (d) asserting `error_refund_icp` returns a precondition error, and (e) asserting `sweep_icp` transfers the victim's ICP.
