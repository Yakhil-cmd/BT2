### Title
WalletConnect Pair Approval Persists After Wallet Key Deletion, Enabling Ghost Authorization on Key Re-import - (File: `packages/core/src/screens/SelectKey/WalletDeleteDialog.tsx`)

### Summary

When a user deletes a wallet key via `WalletDeleteDialog`, the WalletConnect pair records bound to that key's fingerprint are never revoked. If the user later re-imports the same key (same mnemonic → same fingerprint), every previously-approved dapp immediately regains full command authority — including `chia_sendTransaction`, `chia_transferNFT`, and other asset-moving operations — without any new user approval dialog.

### Finding Description

The `WalletDeleteDialog.handleSubmit` function performs two cleanup steps on key deletion:

1. `deleteKey({ fingerprint })` — removes the key from the daemon keyring
2. `removeFingerprintPrefs()` — removes `fingerprintSettings` and `sortedWallets` from local storage [1](#0-0) 

Neither step touches the WalletConnect pair store. Every `PermissionsPairRecord` that was registered against the deleted fingerprint — including its granted `commands` list and any bypass permissions — remains intact in the main-process pair store and the WalletConnect SDK's IndexedDB. [2](#0-1) 

The `deleteKey` RTK mutation only invalidates `Keys` and `DaemonKey` cache tags; it triggers no WalletConnect side-effect: [3](#0-2) 

The startup cleanup function `cleanupPairingsAndSessions` only removes WalletConnect SDK sessions/pairings that lack a corresponding pair record. It does **not** remove pair records whose fingerprint no longer exists in the keyring: [4](#0-3) 

When the user re-imports the same mnemonic (producing the same fingerprint) and logs in, `useWalletConnectCommand` performs the fingerprint guard:

```
requestedFingerprint !== pair.fingerprint || currentFingerprint !== pair.fingerprint
```

Because `currentFingerprint` now equals the re-imported fingerprint, and `pair.fingerprint` was never cleared, the guard passes and the dapp's commands execute without any new approval prompt. [5](#0-4) 

### Impact Explanation

A previously-approved WalletConnect dapp retains full command authority over the re-imported wallet. Commands such as `chia_sendTransaction`, `chia_transferNFT`, `chia_createOfferForIds`, and `chia_cancelOffer` can be dispatched by the dapp without any new user confirmation dialog. This constitutes unauthorized signing and transfer of XCH, CATs, and NFTs — a **Critical** impact under the allowed scope.

### Likelihood Explanation

Key re-import after deletion is a routine user action: migrating to a new device, restoring from a seed backup, or recovering after a hardware failure. The dapp (attacker) needs only to have been approved once before the deletion. No cryptographic break, phishing, or host compromise is required — the dapp simply waits for the user to re-import and then resumes sending `session_request` events over the still-live WalletConnect relay.

### Recommendation

In `WalletDeleteDialog.handleSubmit`, before calling `deleteKey`, enumerate all pairs whose `fingerprint` matches the deleted key and call `disconnectPair` (which calls `permissionsAPI.revokePair` and disconnects the WalletConnect session) for each:

```typescript
async function handleSubmit(values: FormData) {
  if (values.fingerprint !== fingerprint.toString()) {
    throw new Error(t`Fingerprint does not match`);
  }
  // Revoke all WalletConnect pairs bound to this fingerprint
  const allPairs = await window.permissionsAPI.getPairs();
  const boundPairs = allPairs.filter((p) => p.fingerprint === fingerprint);
  await Promise.all(boundPairs.map((p) => disconnectPair(p.topic)));
  // Then delete the key
  await deleteKey({ fingerprint }).unwrap();
  removeFingerprintPrefs();
  onClose?.();
}
```

Alternatively, the main-process `deleteKey` IPC handler in `packages/gui/src/electron/main.tsx` should automatically revoke all pairs for the deleted fingerprint before forwarding the delete to the daemon. [6](#0-5) 

### Proof of Concept

1. User opens the Chia GUI and logs in with wallet key fingerprint `F`.
2. User pairs a WalletConnect dapp (e.g., a DEX) and grants it `chia_sendTransaction`.
3. User navigates to **Select Key → Delete** and deletes fingerprint `F`. `WalletDeleteDialog.handleSubmit` calls `deleteKey` and `removeFingerprintPrefs` — no WalletConnect cleanup occurs.
4. The pair record for fingerprint `F` remains in the pair store; the WalletConnect SDK session remains in IndexedDB.
5. User re-imports the same mnemonic (fingerprint `F`) and logs in.
6. The dapp sends a `session_request` for `chia_sendTransaction` over the existing WalletConnect relay.
7. `processSessionRequest` finds the pair (still present), `useWalletConnectCommand` passes the fingerprint guard (`currentFingerprint === F === pair.fingerprint`), and the transaction is dispatched — **no approval dialog is shown to the user**. [7](#0-6)

async function cleanupPairingsAndSessions(client: Client) {
  try {
    const pairs = await window.permissionsAPI.getPairs();

    // disconnect all sessions that are not in the pairs list
    const clientSessions = client.session.getAll();

    const sessionsToDisconnect = clientSessions.filter((s) => !pairs.some((p) => p.topic === s.pairingTopic));
    await Promise.all(
      sessionsToDisconnect.map((session) =>
        client.disconnect({
          topic: session.topic,
          reason: getSdkError('USER_DISCONNECTED'),
        }),
      ),
    );

    // disconnect all pairings that are not in the pairs list
    const clientPairings = client.pairing.getAll();

    const pairingsToDisconnect = clientPairings.filter(
      (clientPair) => !pairs.some((p) => clientPair.topic === p.topic),
    );

    await Promise.all(pairingsToDisconnect.map((pairing) => client.core.pairing.disconnect({ topic: pairing.topic })));
  } catch (e) {
    log('Cleanup pairings error', e);
    processError(e as Error);
  }
}
