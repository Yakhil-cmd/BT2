### Title
`poolDeployer` Cannot Be Updated After Initial Set — (`MetricOmmPoolFactory.sol`)

### Summary

`MetricOmmPoolFactory.setPoolDeployer` enforces a write-once guard that permanently locks the deployer address. If the deployer ever needs to be replaced, the factory itself must be redeployed. Because every pool stores `FACTORY` as an immutable, redeploying the factory orphans all existing pools, permanently blocking fee collection, emergency pause, and price-provider updates on those pools.

### Finding Description

`setPoolDeployer` contains a one-time-set guard:

```solidity
function setPoolDeployer(address _poolDeployer) external override onlyOwner {
    if (poolDeployer != address(0)) revert PoolDeployerAlreadySet();
    poolDeployer = _poolDeployer;
    emit PoolDeployerSet(_poolDeployer);
}
``` [1](#0-0) 

Once set, there is no owner function to update `poolDeployer`. If the deployer contract must be replaced (bug fix, upgrade, or compromise), the only recourse is to redeploy `MetricOmmPoolFactory`.

Every pool bakes the factory address in as an immutable at construction time:

```solidity
address internal immutable FACTORY;
``` [2](#0-1) 

All privileged pool operations — `collectFees`, `setPoolFees`, `setPause`, `setBinAdditionalFees`, and `setPriceProvider` — are gated by `onlyFactory`, which checks `msg.sender != FACTORY`: [3](#0-2) 

A redeployed factory has a different address, so it can never call any of those functions on pools that were created under the original factory.

### Impact Explanation

When the factory is redeployed to work around the frozen `poolDeployer`:

1. **Protocol fee loss** — `collectFees` is `onlyFactory`. Spread and notional fees that have accrued inside every existing pool can never be extracted; they are permanently stranded. [4](#0-3) 

2. **Emergency pause broken** — `protocolPausePool` and `pausePool` both route through `setPause(onlyFactory)`. If a critical vulnerability is discovered in a live pool, the protocol cannot halt it. [5](#0-4) 

3. **Price-provider updates blocked** — `setPriceProvider` is `onlyFactory`. Mutable-provider pools cannot have their oracle updated, leaving them exposed to a stale or compromised feed indefinitely. [6](#0-5) 

### Likelihood Explanation

Deployer contracts are a common upgrade target (gas optimisations, CREATE2 salt changes, new pool types). The probability that `poolDeployer` will need to change over the protocol's lifetime is non-trivial. The trigger requires no attacker — it is a normal operational event.

### Recommendation

Remove the `PoolDeployerAlreadySet` guard and allow the owner to update `poolDeployer` at any time (optionally behind a timelock):

```solidity
function setPoolDeployer(address _poolDeployer) external override onlyOwner {
    address old = poolDeployer;
    poolDeployer = _poolDeployer;
    emit PoolDeployerSet(old, _poolDeployer);
}
```

Alternatively, if the one-time-set behaviour is intentional, make `FACTORY` mutable in `MetricOmmPool` (behind an `onlyFactory` setter) so that pools can be migrated to a new factory without being orphaned.

### Proof of Concept

1. Owner deploys `MetricOmmPoolFactory` and calls `setPoolDeployer(deployerV1)`.
2. Pools A, B, C are created; each stores `FACTORY = factoryV1` as an immutable.
3. A bug is found in `deployerV1`; owner must replace it.
4. `setPoolDeployer(deployerV2)` reverts with `PoolDeployerAlreadySet`.
5. Owner redeploys `MetricOmmPoolFactory` → `factoryV2`.
6. `factoryV2.collectPoolFees(poolA)` calls `poolA.collectFees(...)` which checks `msg.sender != FACTORY` → `FACTORY` is `factoryV1` → reverts with `OnlyFactory`.
7. Spread and notional fees inside pools A, B, C are permanently unrecoverable. `protocolPausePool` and `setPriceProvider` are equally blocked. [7](#0-6) [8](#0-7)
