### Title
Unrestricted `disable_rewards` Parameter in `update_rewards` Allows Any Caller to Permanently Suppress Per-Block Consensus Rewards — (File: src/staking/staking.cairo)

---

### Summary

The `update_rewards` function in the `Staking` contract has no caller restriction and exposes a `disable_rewards: bool` parameter to any account. An unprivileged attacker can call `update_rewards(valid_staker, disable_rewards: true)` once per block to consume the single per-block reward slot — updating `last_reward_block` — while suppressing all reward distribution. Because the slot is consumed, no legitimate caller can recover rewards for that block.

---

### Finding Description

`IStakingRewardsManager::update_rewards` is a public, permissionless entry point: [1](#0-0) 

Its implementation first writes `last_reward_block` unconditionally, then branches on `disable_rewards`: [2](#0-1) 

The guard that prevents double-execution per block is: [3](#0-2) 

Because `last_reward_block` is written **before** the `disable_rewards` branch is evaluated, any caller who passes `disable_rewards: true` permanently exhausts the reward slot for the current block. No subsequent call can pass the `current_block_number > last_reward_block` check for that block.

There is no `assert_caller_is_*` guard, no role check, and no restriction on who may supply `disable_rewards: true`. The full reward-distribution path is only reached when `disable_rewards` is `false` **and** consensus rewards are active: [4](#0-3) 

---

### Impact Explanation

An attacker who calls `update_rewards(any_active_staker, disable_rewards: true)` once per block:

1. Permanently prevents consensus block rewards from being credited to any staker for that block — the `_update_rewards` path that credits `unclaimed_rewards_own` and forwards pool rewards is never reached.
2. Because `last_reward_block` is already set to the current block, no honest caller can retroactively fix the missed distribution.

Sustained over many blocks, this permanently freezes all unclaimed consensus-era yield for every staker and every delegation pool. This matches the allowed impact: **Permanent freezing of unclaimed yield or unclaimed royalties (High)**.

---

### Likelihood Explanation

- The entry point is fully public; no token, stake, or role is required.
- The cost is one cheap transaction per block.
- The attacker needs only to know any valid, active staker address (trivially obtained from on-chain events such as `NewStaker`).
- There is no economic barrier and no on-chain defence.

Likelihood: **High**.

---

### Recommendation

Restrict `update_rewards` to a trusted caller (e.g., the consensus layer contract or a dedicated role), or remove the `disable_rewards` parameter from the public interface entirely and handle the "no-reward" case through a separate, access-controlled path. At minimum, the `last_reward_block` write must not occur when `disable_rewards` is `true` unless the caller is authorised.

---

### Proof of Concept

```
// Attacker script — one call per block, zero stake required

for each block B:
    staking_contract.update_rewards(
        staker_address = <any valid active staker>,
        disable_rewards = true          // ← attacker-controlled
    )
    // Effect:
    //   last_reward_block := B         (slot consumed)
    //   _update_rewards never called   (no yield credited)
    //
    // Any honest call to update_rewards for block B now reverts:
    //   assert!(current_block_number > last_reward_block)  → FAIL
```

All stakers and pool members accumulate zero consensus rewards for every block the attacker targets. The lost yield is irrecoverable because the per-block slot is a write-once value.
