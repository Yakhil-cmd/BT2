### Title
Unvalidated Zero Address for `refund_recipient` in L1→L2 Transactions Causes Permanent Loss of Refunded Base Tokens - (`File: basic_bootloader/src/bootloader/transaction_flow/zk/process_l1_transaction.rs`)

---

### Summary

L1→L2 priority transactions carry a `refund_recipient` field (`reserved[1]`) that designates the L2 address to receive unused gas refunds (and the full deposit minus operator fee on revert). The bootloader never validates this field against `address(0)`. When a sender sets `refund_recipient = address(0)` — which is the default value — the refund amount is minted from the treasury to `B160::ZERO`, permanently destroying those base tokens.

---

### Finding Description

The `refund_recipient` for an L1→L2 transaction is stored in `reserved[1]` of the ABI-encoded transaction. During `validate_structure()`, the code explicitly skips validation of this field with a `// TODO: validate address?` comment:

```rust
// reserved[1] = refund recipient for l1 to l2 and upgrade txs
match tx_type {
    Self::L1_L2_TX_TYPE | Self::UPGRADE_TX_TYPE => {
        // TODO: validate address?
    }
    _ => unreachable!(),
}
```

Later, in `process_l1_transaction`, the refund recipient is read unconditionally and passed directly to `mint_base_token()`:

```rust
if to_refund_recipient > U256::ZERO {
    let refund_recipient = u256_to_b160_checked(transaction.reserved[1].read());
    mint_base_token::<S, Config>(
        system,
        system_functions,
        memories.reborrow(),
        &to_refund_recipient,
        &refund_recipient,   // ← can be B160::ZERO
        ...
    )?;
}
```

`mint_base_token()` calls `transfer_from_treasury()`, which subtracts `to_refund_recipient` from the treasury balance and adds it to the `to` address. When `to` is `B160::ZERO`, the treasury is debited but the tokens are credited to the zero address — they are permanently inaccessible.

The `ZKsyncL1Tx` struct's `refund_recipient` field defaults to `Address::default()` (i.e., `address(0)`), meaning any L1 transaction that omits an explicit refund recipient will silently route all refunds to the zero address. The existing regression test at `tests/instances/transactions/src/lib.rs:1843` even explicitly uses `address(0)` as the refund recipient and asserts that the refund is credited there, confirming the behavior is currently accepted.

---

### Impact Explanation

**Impact: Medium**

For every L1→L2 transaction where `refund_recipient = address(0)`:
- The unused gas refund (`(gas_limit - gas_used) * gas_price`) is minted to `address(0)` and permanently lost.
- On a failed/reverted transaction, the entire deposit minus the operator fee (`total_deposited - pay_to_operator`) is minted to `address(0)` and permanently lost.

These are real base tokens drawn from the treasury (`BASE_TOKEN_HOLDER_ADDRESS`) and credited to an uncontrolled address. The treasury balance is correctly reduced, but the tokens become permanently inaccessible. This constitutes a direct, irreversible loss of user funds.

---

### Likelihood Explanation

**Likelihood: Medium**

The `refund_recipient` field defaults to `Address::default()` = `address(0)` in the `ZKsyncL1Tx` struct. Any L1 transaction submitted without explicitly setting a refund recipient will trigger this path. Users unfamiliar with the field, or tooling that omits it, will silently lose their gas refunds. The scenario is realistic and requires no special attacker capability — it is triggered by ordinary L1→L2 transaction submission.

---

### Recommendation

1. In `validate_structure()`, reject L1/upgrade transactions where `reserved[1]` decodes to `B160::ZERO`:
   ```rust
   Self::L1_L2_TX_TYPE | Self::UPGRADE_TX_TYPE => {
       let refund_recipient = u256_try_to_b160(self.reserved[1].read());
       if refund_recipient.map_or(true, |a| a == B160::ZERO) {
           return Err(());
       }
   }
   ```
2. Alternatively, if `reserved[1]` is zero, fall back to sending the refund to `transaction.from` (the sender).

---

### Proof of Concept

The existing test `test_treasury_based_token_distribution_regression` in `tests/instances/transactions/src/lib.rs` already demonstrates the issue: it sets `refund_recipient = address(0)` and asserts that the refund amount is credited to `address(0)`. The treasury is debited, the operator is paid, and the remainder goes to the zero address — confirming the funds-loss path is live.
