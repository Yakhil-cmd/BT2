### Title
Unbounded `pool_member_epoch_balance` Trace Causes Unbounded Gas in `calculate_rewards`, Permanently Freezing Delegator Yield - (File: `src/pool/pool.cairo`)

### Summary

The `calculate_rewards` internal function in the Pool contract iterates over every entry in a delegator's `pool_member_epoch_balance` trace without any upper bound. The trace grows by one entry each time a delegator calls `add_to_delegation_pool` or `exit_delegation_pool_intent` in a new epoch. A delegator who accumulates enough balance-change entries without claiming rewards will eventually be unable to call `claim_rewards` because the transaction exceeds Starknet's per-transaction gas limit, permanently freezing their unclaimed yield. The codebase itself acknowledges the risk with the comment: *"This loop is unbounded but unlikely to exceed gas limits."*

### Finding Description

**Root cause — unbounded loop in `calculate_rewards`:** [1](#0-0) 

```cairo
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
    if pool_member_checkpoint.epoch() >= until_epoch {
        break;
    }
    let to_sigma = self.find_sigma(pool_member_checkpoint, curr_epoch: until_epoch);
    rewards += compute_rewards_rounded_down(...);
    entry_to_claim_from += 1;
}
```

The loop runs from `entry_to_claim_from` (the index saved at the last successful `claim_rewards`) up to `pool_member_trace_length`. Each iteration performs at least one storage read via `pool_member_trace.at(...)` and a call to `find_sigma`, which itself reads from `cumulative_rewards_trace` storage.

**How the trace grows:**

`set_member_balance` is called by both `add_to_delegation_pool` and `exit_delegation_pool_intent`. It calls `trace.insert(key: self.get_epoch_plus_k(), ...)`. [2](#0-1) 

The `insert` implementation only appends a **new** checkpoint when the epoch key differs from the last entry's key; otherwise it overwrites the last entry. [3](#0-2) 

Therefore, each call to `add_to_delegation_pool` or `exit_delegation_pool_intent` in a **different epoch** appends one new entry to the trace. There is no cap on the trace length.

**How `claim_rewards` and `pool_member_info_v1` consume the trace:**

`claim_rewards` calls `calculate_rewards` starting from `pool_member_info.entry_to_claim_from` and updates that index only on success. [4](#0-3) 

`pool_member_info_v1` (a view function) also calls `calculate_rewards` from the same starting index but never updates it. [5](#0-4) 

If `claim_rewards` reverts due to gas exhaustion, `entry_to_claim_from` is never advanced, so every subsequent attempt re-iterates the same ever-growing set of entries.

**Attack / accumulation path:**

1. Delegator calls `enter_delegation_pool` (first trace entry).
2. Each epoch, delegator calls `add_to_delegation_pool` with the minimum non-zero amount — appending one new trace entry per epoch.
3. Delegator never calls `claim_rewards`, so `entry_to_claim_from` stays at 0.
4. After N epochs of balance changes, `calculate_rewards` must iterate N entries, each requiring multiple storage reads.
5. Once N is large enough to exceed the Starknet per-transaction gas limit, `claim_rewards` reverts unconditionally, and the delegator's accumulated yield is permanently inaccessible.

No privileged role is required. The delegator only needs a small token balance (minimum 1 unit per epoch) and the ability to call `add_to_delegation_pool` repeatedly.

### Impact Explanation

**Permanent freezing of unclaimed yield.** Once the trace is large enough that `calculate_rewards` exceeds the gas limit, the delegator can never successfully call `claim_rewards`. Because `entry_to_claim_from` is only advanced inside a successful `claim_rewards` execution, there is no recovery path: every future attempt starts from the same index and hits the same gas wall. All accumulated STRK rewards are permanently locked in the pool contract with no mechanism to retrieve them.

This matches the allowed impact: *"Permanent freezing of unclaimed yield or unclaimed royalties."*

### Likelihood Explanation

The trace grows at most one entry per epoch, so the attack requires sustained activity over many epochs. However:

- The minimum cost per epoch is 1 token unit plus gas for `add_to_delegation_pool`.
- A delegator who legitimately changes their delegation frequently (e.g., dollar-cost averaging) can reach a large trace organically without malicious intent.
- The developers themselves flagged the loop as unbounded, indicating awareness that the bound is not enforced.
- Once the threshold is crossed, the freeze is irreversible.

Likelihood is **medium**: not trivially instant, but reachable by any delegator over a sufficiently long horizon without any privileged access.

### Recommendation

Enforce an upper bound on the number of unclaimed balance-change entries before allowing further balance modifications, analogous to how `maxTopics` is enforced in the referenced report. Concretely:

1. Add a constant `MAX_UNCLAIMED_BALANCE_CHANGES` (e.g., 500).
2. In `set_member_balance`, assert that `pool_member_trace_length - entry_to_claim_from < MAX_UNCLAIMED_BALANCE_CHANGES` before appending a new entry, or require the delegator to call `claim_rewards` first to advance `entry_to_claim_from`.

Alternatively, require `claim_rewards` to be called (or auto-claim) whenever a balance change is recorded, keeping the unclaimed window bounded to O(1).

### Proof of Concept

```
1. Deploy staking + pool contracts.
2. Delegator calls enter_delegation_pool(reward_address, amount=MIN_STAKE).
3. For i in 1..N:
     a. advance_epoch()
     b. delegator calls add_to_delegation_pool(pool_member, amount=1)
        → appends one new entry to pool_member_epoch_balance trace
4. After N epochs, call claim_rewards(pool_member).
   → calculate_rewards iterates N entries, each doing storage reads via
     pool_member_trace.at() and find_sigma().
   → At sufficiently large N, the transaction exceeds the Starknet gas limit
     and reverts.
5. All subsequent claim_rewards calls also revert (entry_to_claim_from unchanged).
   → Delegator's yield is permanently frozen.
```

The exact value of N at which gas exhaustion occurs depends on Starknet's current gas schedule, but the loop is explicitly acknowledged as unbounded in the source code, confirming the absence of any protective cap.
