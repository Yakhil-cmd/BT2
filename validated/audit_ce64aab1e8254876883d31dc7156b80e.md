### Title
Cross-Function Reentrancy via Malicious `reward_address` in `claim_rewards` Allows Theft of Unclaimed Yield - (File: src/staking/staking.cairo)

---

### Summary

The `claim_rewards` function in `src/staking/staking.cairo` violates the Checks-Effects-Interactions (CEI) pattern. It makes two external calls — to the reward supplier and to the STRK token contract — before committing the zeroed `unclaimed_rewards_own` to storage. A staker who sets their `reward_address` to a malicious contract can reenter `claim_rewards` during the STRK transfer, reading the still-non-zero `unclaimed_rewards_own` from storage and draining the reward supplier's `unclaimed_rewards` balance across multiple recursive calls.

---

### Finding Description

`claim_rewards` calls the internal helper `send_rewards_to_staker`, which performs two external calls before the storage write:

1. `claim_from_reward_supplier(...)` — calls `reward_supplier.claim_rewards(amount)`, which decrements `unclaimed_rewards` in the reward supplier and transfers STRK to the staking contract. [1](#0-0) 

2. `token_dispatcher.checked_transfer(recipient: reward_address, amount)` — transfers STRK to the staker's `reward_address`. [2](#0-1) 

Only after both external calls does the local variable get zeroed: [3](#0-2) 

And only after `send_rewards_to_staker` returns does `write_staker_info` commit the zeroed value to storage: [4](#0-3) 

Because `write_staker_info` has not yet been called when the STRK transfer to `reward_address` executes, the staking contract's storage still holds the original non-zero `unclaimed_rewards_own`. A malicious `reward_address` contract can call `claim_rewards` again at this point. The reentrant call reads the stale non-zero value from storage, passes the caller check (since `reward_address` is an authorized caller), and triggers another full reward payout cycle. [5](#0-4) 

The reward supplier's `claim_rewards` function checks `amount <= unclaimed_rewards` and decrements `unclaimed_rewards` before transferring, but it does not prevent the staking contract from calling it multiple times for the same staker reward: [6](#0-5) 

The `claim_from_reward_supplier` utility only verifies that the staking contract's balance increased by the expected amount — it does not guard against reentrancy: [7](#0-6) 

Notably, the developers recognized this risk in `unstake_action` and added a comment "This is done here to avoid re-entrancy," calling `write_staker_info` before subsequent external calls. The same protection was not applied to `claim_rewards`. [8](#0-7) 

---

### Impact Explanation

An attacker can recursively drain the reward supplier's `unclaimed_rewards` balance. This balance represents the total STRK owed to all stakers across the protocol. By repeatedly claiming their own (small) reward before the storage is updated, the attacker can exhaust the reward supplier's accounting balance, preventing all other stakers and pool members from ever claiming their earned yield. This constitutes **theft of unclaimed yield** — a High severity impact under the allowed scope.

---

### Likelihood Explanation

Any staker can set an arbitrary `reward_address` via `change_reward_address`. No privileged role is required. The attacker only needs to:
1. Stake the minimum amount and accumulate any non-zero rewards.
2. Set `reward_address` to a malicious contract they control.
3. Call `claim_rewards`.

The reentrancy window exists on every call to `claim_rewards` as long as the reward supplier holds sufficient STRK balance, which by design it always does (it proactively requests funds from L1 to maintain a buffer above `unclaimed_rewards`). [9](#0-8) 

---

### Recommendation

Apply the CEI pattern in `send_rewards_to_staker` (or in `claim_rewards` before calling it): zero out `staker_info.unclaimed_rewards_own` in storage **before** making any external call. Concretely, call `write_staker_info` with the zeroed `staker_info` before calling `claim_from_reward_supplier` and the STRK transfer. Alternatively, add a reentrancy guard (mutex flag in storage) to `claim_rewards` and `unstake_action`.

---

### Proof of Concept

```
1. Attacker deploys malicious contract M with a `fallback` / `receive` that calls
   staking.claim_rewards(attacker_address).

2. Attacker calls staking.stake(..., reward_address=M, ...) with minimum stake.

3. Attacker waits for rewards to accrue (unclaimed_rewards_own = R > 0).

4. M calls staking.claim_rewards(attacker_address):
   a. staking reads unclaimed_rewards_own = R from storage.
   b. staking calls reward_supplier.claim_rewards(R):
      - reward_supplier.unclaimed_rewards -= R  (e.g., 10000 → 9900)
      - reward_supplier transfers R STRK to staking contract.
   c. staking calls STRK.transfer(M, R).
   d. M receives R STRK and immediately calls staking.claim_rewards(attacker_address) again.
      - staking reads unclaimed_rewards_own = R from storage (UNCHANGED — write_staker_info
        not yet called).
      - staking calls reward_supplier.claim_rewards(R):
        - reward_supplier.unclaimed_rewards -= R  (9900 → 9800)
        - reward_supplier transfers R STRK to staking contract.
      - staking calls STRK.transfer(M, R).
      - M reenters again... (repeat until gas limit or reward_supplier balance exhausted)
   e. Eventually the stack unwinds; the original call's write_staker_info(unclaimed=0)
      executes, but M has already received N*R STRK.

5. Other stakers call claim_rewards and fail because reward_supplier.unclaimed_rewards
   has been drained below their owed amounts.
```

        fn claim_rewards(ref self: ContractState, amount: Amount) {
            // Asserts.
            let staking_contract = self.staking_contract.read();
            assert!(
                get_caller_address() == staking_contract,
                "{}",
                GenericError::CALLER_IS_NOT_STAKING_CONTRACT,
            );
            let unclaimed_rewards = self.unclaimed_rewards.read();
            assert!(amount <= unclaimed_rewards, "{}", GenericError::AMOUNT_TOO_HIGH);

            // Update unclaimed_rewards and transfer the requested rewards to the staking contract.
            self.unclaimed_rewards.write(unclaimed_rewards - amount);
            let token_dispatcher = self.token_dispatcher.read();
            token_dispatcher.checked_transfer(recipient: staking_contract, amount: amount.into());
        }

        fn request_funds(ref self: ContractState, unclaimed_rewards: Amount) {
            // Read current balance.
            let token_dispatcher = self.token_dispatcher.read();
            let balance: Amount = token_dispatcher
                .balance_of(account: get_contract_address())
                .try_into()
                .expect_with_err(InternalError::BALANCE_ISNT_AMOUNT_TYPE);

            // Calculate credit, which is the contract's balance plus the amount already requested
            // from L1, and the debit, which is the unclaimed rewards.
            let mut l1_pending_requested_amount = self.l1_pending_requested_amount.read();
            let credit = balance + l1_pending_requested_amount;
            let debit = unclaimed_rewards;

            // If there isn't enough credit to cover the debit + threshold, request funds.
            let base_mint_amount = self.base_mint_amount.read();
            let threshold = compute_threshold(base_mint_amount);
            if credit < debit + threshold {
                let diff = debit + threshold - credit;
                let num_msgs = ceil_of_division(dividend: diff, divisor: base_mint_amount);
                let total_amount = num_msgs * base_mint_amount;
                for _ in 0..num_msgs {
                    self.send_mint_request_to_l1_reward_supplier();
                }
                self.emit(Events::MintRequest { total_amount, num_msgs });
                l1_pending_requested_amount += total_amount;
            }

            // Commit to storage the requested amount, which is now part of the credit.
            self.l1_pending_requested_amount.write(l1_pending_requested_amount);
        }

pub(crate) fn claim_from_reward_supplier(
    reward_supplier_dispatcher: IRewardSupplierDispatcher,
    amount: Amount,
    token_dispatcher: IERC20Dispatcher,
) {
    let staking_contract = get_contract_address();
    let balance_before = token_dispatcher.balance_of(account: staking_contract);
    reward_supplier_dispatcher.claim_rewards(:amount);
    let balance_after = token_dispatcher.balance_of(account: staking_contract);
    assert!(balance_after - balance_before == amount.into(), "{}", Error::UNEXPECTED_BALANCE);
}
