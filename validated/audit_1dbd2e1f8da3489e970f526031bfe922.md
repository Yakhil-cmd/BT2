### Title
Gateway `skip_stateful_validations` admits invoke transactions with unverified signatures into the mempool — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function bypasses the `__validate__` entry point (signature verification) for any invoke transaction with `nonce=1` when the sender account has **any** transaction present in the mempool or a recent block. An unprivileged attacker who observes a victim's pending `deploy_account` transaction can inject an invoke with an arbitrary/invalid signature for that victim's account address, and the gateway will admit it to the mempool without ever verifying the signature. This satisfies the **High** impact criterion: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

---

### Finding Description

`extract_state_nonce_and_run_validations` in `stateful_transaction_validator.rs` orchestrates gateway-level stateful validation in three steps:

```
account_nonce = get_nonce_from_state(address)
skip_validate = run_pre_validation_checks(tx, account_nonce, mempool)
run_validate_entry_point(tx, skip_validate)
```

`run_pre_validation_checks` calls `skip_stateful_validations`, which returns `true` (skip `__validate__`) when all three conditions hold:

1. The transaction is an `Invoke` with `nonce == 1`
2. The on-chain account nonce is `0` (account not yet deployed)
3. `account_tx_in_pool_or_recent_block(sender_address)` returns `true`

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs:437
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
    return mempool_client
        .account_tx_in_pool_or_recent_block(tx.sender_address())
        .await ...;
}
```

`account_tx_in_pool_or_recent_block` is implemented as:

```rust
// crates/apollo_mempool/src/mempool.rs:697-700
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
```

This returns `true` for **any** account with a transaction in the pool or recent block — not specifically a `deploy_account`. The code comment acknowledges this: *"either it has a deploy_account transaction or transactions with future nonces that passed validations."*

When `skip_validate = true`, `run_validate_entry_point` constructs `ExecutionFlags` with `validate: false`:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs:310-312
let strict_nonce_check = false;
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
```

With `validate: false`, `StatefulValidator::perform_validations` for an Invoke transaction calls `perform_pre_validation_stage` (nonce/fee checks only) and then **returns `Ok(())` immediately** without ever calling `validate_tx` (the `__validate__` entry point):

```rust
// crates/blockifier/src/blockifier/stateful_validator.rs:78-81
tx.perform_pre_validation_stage(self.state(), &tx_context)?;
if !tx.execution_flags.validate {
    return Ok(());
}
// `__validate__` call — never reached when validate=false
```

The mempool's `validate_tx` (called via `validate_by_mempool` before the skip check) only inspects `ValidationArgs`, which contains `address`, `account_nonce`, `tx_hash`, `tx_nonce`, `tip`, and `max_l2_gas_price` — **no signature field**:

```rust
// crates/apollo_mempool_types/src/mempool_types.rs:50-57
pub struct ValidationArgs {
    pub address: ContractAddress,
    pub account_nonce: Nonce,
    pub tx_hash: TransactionHash,
    pub tx_nonce: Nonce,
    pub tip: Tip,
    pub max_l2_gas_price: GasPrice,
}
```

The mempool's `validate_tx` checks only for duplicate tx_hash and fee escalation rules — never the signature. Therefore, an invoke with a completely invalid signature passes all gateway checks and enters the mempool.

---

### Impact Explanation

**Broken invariant:** Every transaction admitted to the mempool must have passed `__validate__` (signature verification).

**Corrupted value:** The mempool contains invoke transactions whose signatures have never been verified.

**Concrete consequences:**

1. **Invalid transaction injection:** An attacker can inject unsigned/invalid-signature invokes for any account that has a pending `deploy_account`. The batcher will later execute these, fail `__validate__`, and reject them — wasting sequencer execution resources.

2. **Griefing via fee escalation:** If the victim also submitted a legitimate invoke with `nonce=1` (the common UX pattern: `deploy_account + invoke` simultaneously), the attacker can replace it with an invalid invoke by paying a higher fee. The mempool's fee escalation logic (`validate_fee_escalation`) accepts the replacement based solely on fee, not signature validity. The victim's legitimate invoke is evicted from the mempool and lost.

3. **Admission of unauthorized transactions:** The gateway's admission decision is wrong — it accepts a transaction that would be rejected by the protocol's authorization check (`__validate__`).

---

### Likelihood Explanation

The attack requires:
- Monitoring the mempool (or P2P gossip) for `deploy_account` transactions — trivially achievable by any network participant.
- Crafting an invoke with `nonce=1` for the victim's address — requires only knowing the address, which is public.
- Paying a fee higher than the victim's invoke to trigger fee escalation — requires spending some STRK/ETH, but the cost is bounded by the victim's fee.

The UX pattern of submitting `deploy_account + invoke` simultaneously is explicitly encouraged by the codebase (it is the stated purpose of this feature), making the precondition common in production.

---

### Recommendation

1. **Verify the signature even when skipping `__validate__`:** Perform a lightweight stateless signature check (ECDSA/Stark curve) at the gateway before admitting the transaction, independent of the blockifier's `__validate__` execution.

2. **Restrict the skip condition to confirmed `deploy_account` presence:** Instead of checking `account_tx_in_pool_or_recent_block` (which matches any transaction type), check specifically that a `deploy_account` transaction for the sender address is present in the pool. This closes the window where an attacker exploits the generic account-presence check.

3. **Add signature to `ValidationArgs`:** Include the transaction signature in `ValidationArgs` so the mempool can perform a basic signature-format check before admission.

---

### Proof of Concept

```
1. Victim submits deploy_account for address A (nonce=0, valid signature).
   → Gateway admits it; mempool now has A in pool.
   → account_tx_in_pool_or_recent_block(A) == true.

2. Victim also submits invoke for A (nonce=1, valid signature, fee=F).
   → skip_stateful_validations: nonce=1, account_nonce=0, A in pool → skip=true.
   → __validate__ skipped; invoke admitted to mempool.

3. Attacker observes both transactions via P2P gossip.

4. Attacker crafts invoke' for A (nonce=1, INVALID signature, fee=F+1).
   → Gateway stateless check: passes (signature format may be valid length).
   → validate_by_mempool: checks tx_hash uniqueness (different hash) and fee
     escalation (F+1 > F) → passes.
   → skip_stateful_validations: nonce=1, account_nonce=0, A in pool → skip=true.
   → run_validate_entry_point with validate=false → __validate__ NOT run.
   → Invalid invoke' admitted to mempool, replacing victim's invoke via fee escalation.

5. Batcher executes invoke':
   → perform_pre_validation_stage: nonce check passes (strict_nonce_check=true
     in batcher, nonce=1 matches incremented state after deploy_account).
   → __validate__ runs → FAILS (invalid signature) → transaction rejected.

6. Victim's legitimate invoke is gone from the mempool. Victim must resubmit.
```

**Exact corrupted value:** The mempool's transaction pool contains `invoke'` with an unverified (invalid) signature at `(address=A, nonce=1)`, displacing the victim's valid transaction. The admission decision at the gateway is wrong: a transaction that will fail protocol authorization is accepted as valid. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
