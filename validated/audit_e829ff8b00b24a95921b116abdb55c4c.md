### Title
Malicious Participant Can Corrupt CKD Output Without Detection — (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

The CKD coordinator unconditionally sums participant share contributions without any proof-of-correctness verification. A single malicious participant can send arbitrary group elements in place of their correctly computed shares, immediately corrupting the finalized `CKDOutput` that the TEE uses to derive a confidential key. There is no challenge or validity-proof mechanism before the output is accepted.

---

### Finding Description

In `do_ckd_coordinator` (`src/confidential_key_derivation/protocol.rs`, lines 35–57), the coordinator collects `(norm_big_y, norm_big_c)` from every participant and adds them together unconditionally:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
Ok(Some(ckd_output))
``` [1](#0-0) 

Each participant is supposed to compute and send:

- `norm_big_y = λᵢ · yᵢ · G₁`
- `norm_big_c = λᵢ · (xᵢ · H(pk ‖ app_id) + yᵢ · app_pk)`

as implemented in `compute_signature_share`: [2](#0-1) 

However, the coordinator performs **no verification** that the received values satisfy these relations. There is no zero-knowledge proof, no commitment-then-reveal, and no consistency check against the participant's public key share. The output is finalized immediately after the single receive round.

The correct aggregate satisfies:

```
C - app_sk · Y  =  msk · H(pk ‖ app_id)
```

If participant `i` instead sends `(norm_big_y_i + Δ_Y, norm_big_c_i + Δ_C)` for arbitrary non-zero G₁ elements `Δ_Y`, `Δ_C`, the coordinator computes:

```
Y'  = Y_correct + Δ_Y
C'  = C_correct + Δ_C
```

The TEE then unmasks:

```
C' - app_sk · Y'  =  msk · H(pk ‖ app_id)  +  Δ_C  -  app_sk · Δ_Y
```

Because the malicious participant does not know `app_sk` (the TEE's secret), they cannot satisfy `Δ_C = app_sk · Δ_Y`, so the unmasked value is wrong. The TEE silently accepts an incorrect confidential key with no error signal.

---

### Impact Explanation

**High — Corruption of CKD output causing honest parties to accept an unusable cryptographic output.**

The TEE derives a confidential key that does not equal `msk · H(pk ‖ app_id)`. Any downstream use of that key (e.g., encrypting secrets for delivery to an application) produces silently wrong results. The corruption is undetectable by the coordinator, the TEE, or any honest participant, because no party holds `msk` in the clear to cross-check the output.

---

### Likelihood Explanation

**High.** Any single participant in the CKD session can mount this attack. It requires no cryptographic capability beyond the ability to send an arbitrary G₁ element over the protocol channel — a trivial capability for any participant who controls their own node. The attack succeeds in a single protocol execution with no prior setup.

---

### Recommendation

Introduce a proof-of-correctness for each participant's contribution before the coordinator finalizes the output. Concretely, each participant should accompany `(norm_big_y, norm_big_c)` with a Sigma/Schnorr proof demonstrating:

1. Knowledge of `yᵢ` such that `norm_big_y = λᵢ · yᵢ · G₁`.
2. That `norm_big_c` is consistent with the participant's committed public key share `Xᵢ = xᵢ · G₂` and the same `yᵢ`, i.e., `norm_big_c = λᵢ · (Xᵢ_G1 · H(pk ‖ app_id) + yᵢ · app_pk)` where `Xᵢ_G1` is derived from the DKG output.

The coordinator must verify all proofs before summing contributions. This mirrors the proof-of-knowledge checks already present in the DKG protocol (`src/dkg.rs`, `verify_proof_of_knowledge`). [3](#0-2) 

---

### Proof of Concept

```
Setup: n honest participants + 1 malicious participant P_m.
       All hold valid DKG key shares. Coordinator is honest.

Step 1: P_m runs compute_signature_share() locally to obtain
        (norm_big_y_m, norm_big_c_m).

Step 2: P_m picks arbitrary non-zero Δ_Y, Δ_C ∈ G₁.

Step 3: P_m sends (norm_big_y_m + Δ_Y, norm_big_c_m + Δ_C)
        to the coordinator via chan.send_private().

Step 4: do_ckd_coordinator() receives the tampered values,
        adds them to the running sum without verification,
        and returns CKDOutput::new(Y_correct + Δ_Y, C_correct + Δ_C).

Step 5: TEE calls ckd_output.unmask(app_sk):
        result = (C_correct + Δ_C) - app_sk * (Y_correct + Δ_Y)
               = msk·H(pk‖app_id) + Δ_C - app_sk·Δ_Y
               ≠ msk·H(pk‖app_id)

Step 6: TEE silently uses the wrong confidential key.
        No error is raised anywhere in the protocol.
``` [4](#0-3) [5](#0-4)

    mut chan: SharedChannel,
    participants: ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,

    app_pk: PublicKey,

    // Receive everyone's inputs and add them together
    let waitpoint = chan.next_waitpoint();

        norm_big_c += participant_output.big_c();

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}

fn verify_proof_of_knowledge<C: Ciphersuite>(
    session_id: &HashOutput,
    domain_separator: &mut DomainSeparator,
    threshold: ReconstructionLowerBound,
    participant: Participant,
    old_participants: Option<ParticipantList>,
    commitment: &VerifiableSecretSharingCommitment<C>,
    proof_of_knowledge: Option<&Signature<C>>,
) -> Result<(), ProtocolError> {
    let threshold = threshold.value();
    match proof_of_knowledge {
        // if participant did not send anything but he is actually an old participant
        None => {
            // if basic dkg or participant is old
            if old_participants.is_none_or(|p| p.contains(participant)) {
                return Err(ProtocolError::MaliciousParticipant(participant));
            }
            // since previous line did not abort, then we know participant is new indeed
            // check the commitment length is threshold - 1
            if commitment.coefficients().len() != threshold - 1 {
                return Err(ProtocolError::IncorrectNumberOfCommitments);
            }
            // nothing to verify
            Ok(())
        }
        // now we know the proof is not none
        Some(proof_of_knowledge) => {
            // if participant sent something but he is actually a new participant
            if old_participants.is_some_and(|p| !p.contains(participant)) {
                return Err(ProtocolError::MaliciousParticipant(participant));
            }
            // since the previous did not abort, we know the participant is old or we are dealing with a dkg
            if commitment.coefficients().len() != threshold {
                return Err(ProtocolError::IncorrectNumberOfCommitments);
            }

            // creating an identifier as required by the syntax of verify_proof_of_knowledge of frost_core
            internal_verify_proof_of_knowledge(
                session_id,
                domain_separator,
                participant,
                commitment,
                proof_of_knowledge,
            )
        }
    }
}
