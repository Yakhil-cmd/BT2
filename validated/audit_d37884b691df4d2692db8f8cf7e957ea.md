### Title
Unbounded Loop in `calculate_rewards` Allows Permanent Freezing of Delegator Unclaimed Yield - (File: src/pool/pool.cairo)

### Summary
The `calculate_rewards` function in the Pool contract contains an explicitly acknowledged unbounded loop that iterates over a delegator's entire `pool_member_epoch_balance` trace. Because each call to `add_to_delegation_pool` or `exit_delegation_pool_intent` in a new epoch appends a new checkpoint to this trace, a delegator who makes repeated balance changes across many epochs without claiming rewards will accumulate an unbounded number of trace entries. When `claim_rewards` is eventually called, the loop must process all accumulated entries, and if the trace is large enough, the transaction will exceed Starknet's gas limit and revert — permanently freezing the delegator's unclaimed yield.

### Finding Description
In `src/pool/pool.cairo`, the internal `calculate_rewards` function iterates over every entry in the `pool_member_epoch_balance` trace between the stored `reward_checkpoint` and the current epoch:

```
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    ...
    entry_to_claim_from += 1;
}
```

The trace grows via `set_member_balance`, which calls `trace.insert(key: self.get_epoch_plus_k(), ...)`. The `insert` function in `src/pool/pool_member_balance_trace/trace.cairo` only merges entries sharing the same key; if the key differs (i.e., the call is made in a different epoch), a new checkpoint is **appended**:

```
} else {
    assert!(last.key < key, "{}", TraceErrors::UNORDERED_INSERTION);
    checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
}
```

`set_member_balance` is called by both `add_to_delegation_pool` (line 242) and `exit_delegation_pool_intent` (line 278). Each call in a distinct epoch appends one new entry. The `entry_to_claim_from` cursor is only advanced when `claim_rewards` is successfully executed. If a delegator makes N balance changes across N epochs without ever claiming rewards, the loop in `calculate_rewards` must process all N entries on the next `claim_rewards` call.

### Impact Explanation
If the trace grows large enough, the `claim_rewards` call will exceed Starknet's per-transaction gas limit and revert. Because `entry_to_claim_from` is only updated inside a successful `claim_rewards` execution, there is no way to partially process the trace across multiple transactions. The delegator's accumulated unclaimed yield becomes permanently inaccessible — a **permanent freezing of unclaimed yield** (High impact).

### Likelihood Explanation
The scenario is reachable by any unprivileged delegator. A delegator who:
1. Delegates a minimum amount to a pool,
2. Calls `add_to_delegation_pool` with a tiny amount (e.g., 1 unit) once per epoch,
3. Never calls `claim_rewards`,

will grow their trace by one entry per epoch. Over a period of hundreds or thousands of epochs (which is realistic given the protocol's intended long-term operation), the trace can grow to a size that causes gas exhaustion. Additionally, since `add_to_delegation_pool` can be called by the pool member **or** the reward address, a malicious reward address can deliberately inflate the trace of a victim pool member at the cost of small token transfers.

### Recommendation
1. Enforce a maximum number of unclaimed balance-change entries per delegator (e.g., reject `add_to_delegation_pool` / `exit_delegation_pool_intent` if the trace length since `entry_to_claim_from` exceeds a safe cap).
2. Alternatively, require `claim_rewards` to be called (or auto-claim) before a new balance change is recorded, keeping the pending trace length bounded.
3. Remove or replace the developer comment "unlikely to exceed gas limits" with an enforced invariant.

### Proof of Concept
1. Staker stakes and opens a pool.
2. Delegator calls `enter_delegation_pool` with minimum amount.
3. For each of N epochs, delegator calls `add_to_delegation_pool` with amount = 1 (without calling `claim_rewards`). Each call appends one entry to `pool_member_epoch_balance` via `set_member_balance` → `trace.insert`.
4. After N epochs, `pool_member_epoch_balance.length()` = N.
5. Delegator calls `claim_rewards`. `calculate_rewards` enters the `while entry_to_claim_from < pool_member_trace_length` loop and iterates N times, each iteration performing storage reads (`pool_member_trace.at(...)`) and `find_sigma` lookups.
6. For sufficiently large N, the transaction reverts due to gas exhaustion. The delegator's unclaimed rewards are permanently frozen.

            // Asserts.
            let mut pool_member_info = self.internal_pool_member_info(:pool_member);
            let caller_address = get_caller_address();

                "{}",
                Error::POOL_CLAIM_REWARDS_FROM_UNAUTHORIZED_ADDRESS,

            // Calculate rewards and update entry_to_claim_from.

            // Emit event.

                        pool_member, reward_address, amount: rewards,
