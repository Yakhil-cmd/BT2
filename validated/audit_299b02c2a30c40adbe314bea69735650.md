### Title
Missing Validity Check in `DACommitmentScheme::try_from_oracle` Accepts Uninitialized/Unsupported Scheme IDs — (`zk_ee/src/common_structs/da_commitment_scheme.rs`)

---

### Summary

`DACommitmentScheme::try_from_oracle` accepts `DACommitmentScheme::None` (value `0`, explicitly documented as **"Invalid option / Internal system state only"**) and `DACommitmentScheme::PubdataKeccak256` (value `2`, **"Currently not supported"**) without rejecting them. This is the direct analog of the original bug: just as `setCollectionSecurityPolicy` failed to check whether a policy was initialized before using its ID, `try_from_oracle` fails to check whether the returned scheme is actually valid before accepting it. The invalid scheme propagates through the system and causes a panic when `da_commitment_generator_from_scheme` is called.

---

### Finding Description

**Root cause — `zk_ee/src/common_structs/da_commitment_scheme.rs`**

`DACommitmentScheme::try_from_oracle` queries the oracle for the DA commitment scheme ID and converts it via `TryFrom<u8>`:

```
0 → DACommitmentScheme::None          // "Invalid option" / "Internal system state only"
2 → DACommitmentScheme::PubdataKeccak256  // "Currently not supported"
``` [1](#0-0) 

The `try_from_oracle` function performs no additional check to reject these invalid/unsupported variants after the conversion. If the oracle returns `0` or `2`, the function returns `Ok(...)` without error, and the invalid scheme is stored in `io.da_commitment_scheme`. [2](#0-1) 

**Propagation — `basic_bootloader/src/bootloader/block_flow/zk/post_tx_op/da_commitment_generator/mod.rs`**

`da_commitment_generator_from_scheme` only handles `BlobsAndPubdataKeccak256`, `EmptyNoDA`, and `BlobsZKsyncOS`. All other variants — including `None` and `PubdataKeccak256` — fall through to the catch-all arm:

```rust
_ => Err(internal_error!("Unsupported DA commitment scheme")),
``` [3](#0-2) 

**Panic site — `basic_bootloader/src/bootloader/block_flow/zk/post_tx_op/post_tx_op_proving_singleblock_batch.rs`**

The result is immediately `.unwrap()`'d:

```rust
let mut da_commitment_generator =
    da_commitment_generator_from_scheme(io.da_commitment_scheme.unwrap(), A::default())
        .unwrap();
``` [4](#0-3) 

This causes an unconditional panic in the proving system whenever the oracle supplies `0` or `2` for `DA_COMMITMENT_SCHEME_QUERY_ID`.

---

### Impact Explanation

A malicious prover or misconfigured sequencer can supply `0` (or `2`) for `DA_COMMITMENT_SCHEME_QUERY_ID` in the oracle input. Because `try_from_oracle` accepts these values without rejection, the invalid scheme is silently stored and used. The proving system then panics at the `.unwrap()` call, making the block **unprovable** — a valid-execution unprovability / denial-of-service on the proving pipeline. The `da_commitment_scheme` field is part of `BatchOutput` which feeds into the public inputs hash; a panic before that point means no proof is ever generated. [5](#0-4) 

---

### Likelihood Explanation

In the proving system the oracle is the CSR-based non-determinism source whose data is supplied as proof input — an "oracle-data-influencing caller" per the scope definition. A malicious prover or a misconfigured sequencer (who sets `DACommitmentSchemeResponder.da_commitment_scheme = Some(DACommitmentScheme::None)`) can trivially trigger this path. The missing guard makes accidental misconfiguration equally easy to hit. [6](#0-5) 

---

### Recommendation

Add an explicit validity guard inside `try_from_oracle` immediately after the `TryFrom` conversion, rejecting the "uninitialized" and "unsupported" variants before they can propagate:

```rust
pub fn try_from_oracle<O: IOOracle>(oracle: &mut O) -> Result<Self, InternalError> {
    let raw: u8 = oracle.query_with_empty_input(DA_COMMITMENT_SCHEME_QUERY_ID)?;
    let scheme = DACommitmentScheme::try_from(raw)
        .map_err(|_| internal_error!("Invalid DA commitment scheme ID"))?;
    match scheme {
        DACommitmentScheme::None =>
            Err(internal_error!("DA commitment scheme is None (uninitialized)")),
        DACommitmentScheme::PubdataKeccak256 =>
            Err(internal_error!("PubdataKeccak256 DA commitment scheme is not supported")),
        valid => Ok(valid),
    }
}
```

This mirrors the recommendation in the original report: add an `isCreated`-equivalent check at the point where the ID is consumed, not only at the point where it is used. [1](#0-0) 

---

### Proof of Concept

1. Configure the oracle (or proof input) to return `0` for `DA_COMMITMENT_SCHEME_QUERY_ID`.
2. `try_from_oracle` returns `Ok(DACommitmentScheme::None)` — no error.
3. The scheme is stored as `Some(DACommitmentScheme::None)` in `io.da_commitment_scheme`.
4. At block finalisation, `da_commitment_generator_from_scheme(DACommitmentScheme::None, alloc)` returns `Err(internal_error!("Unsupported DA commitment scheme"))`.
5. `.unwrap()` panics — the proving system halts and no proof is produced for the block. [7](#0-6) [8](#0-7)
