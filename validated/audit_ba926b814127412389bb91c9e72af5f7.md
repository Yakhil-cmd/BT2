Audit Report

## Title
Unbounded `data.reserve(size)` in `FrameDecoder::decode` Allows Compromised Sandbox to Abort Replica Controller — (`rs/canister_sandbox/src/frame_decoder.rs`)

## Summary

`FrameDecoder::decode` reads an attacker-controlled `u64` length prefix from the IPC socket and calls `data.reserve(size)` with no upper-bound guard. A compromised sandbox process can write 8 bytes (`0xFF` × 8) to its socket fd, causing the controller's `CanisterSandboxIPC` thread to call `data.reserve(usize::MAX)`, which invokes `handle_alloc_error` → `abort()`, killing the replica process. The canister sandbox is an explicit security boundary whose design assumes the sandbox may be compromised while the controller must remain safe; this assumption is violated.

## Finding Description

In `rs/canister_sandbox/src/frame_decoder.rs`, `decode` reads a `u64` from the wire and stores it as `FrameDecoderState::Length(size)`. On the next loop iteration it casts to `usize` and calls `data.reserve(size)` with no maximum check:

```
// frame_decoder.rs L40-48
let size = data.get_u64();                    // attacker-controlled
self.state = FrameDecoderState::Length(size);
// ...
FrameDecoderState::Length(size) => {
    let size: usize = *size as usize;         // no bounds check
    if data.len() < size {
        data.reserve(size);                   // ← no upper-bound guard
        return None;
    }
```

`SocketReaderConfig` (transport.rs L289–323) contains only `idle_malloc_trim` and `idle_timeout`; there is no `max_frame_size` field and no check is performed anywhere in `socket_read_messages` before `decode` is called.

The exploit path is fully confirmed in the code:

1. `spawn_canister_sandbox_process` creates a `UnixStream::pair()` and passes `sock_sandbox` as fd 3 to the child process.
2. The controller spawns a `CanisterSandboxIPC` thread that calls `transport::socket_read_messages` → `FrameDecoder::decode` in a tight loop.
3. A compromised sandbox writes `\xFF\xFF\xFF\xFF\xFF\xFF\xFF\xFF` (8 bytes) to fd 3.
4. `decode` reads the `u64` value `0xFFFFFFFFFFFFFFFF`, casts to `usize::MAX`, and calls `data.reserve(usize::MAX)`.
5. Rust's global allocator cannot satisfy the request; `handle_alloc_error` calls `abort()`, terminating the replica process.

The same unguarded path exists for the `SandboxLauncherIPC` thread via `spawn_launcher_process`, which also uses `SocketReaderConfig::default()` with no frame size limit.

## Impact Explanation

This is a **High** severity finding matching: *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."*

A compromised sandbox process can unconditionally abort the replica controller with 8 bytes. If the same canister is scheduled across multiple nodes simultaneously (normal subnet operation), all affected replicas crash, halting consensus on that subnet. The canister sandbox is an explicit security boundary; the controller is supposed to remain safe even when the sandbox is fully compromised.

## Likelihood Explanation

Triggering this requires first achieving code execution inside a sandbox process (e.g., via a Wasm executor or embedder bug). Once inside the sandbox, the attacker legitimately holds fd 3 and can write arbitrary bytes to it — no further privilege is needed. The write is 8 bytes, requires no timing, no race condition, and is immediately effective. The controller performs zero validation on the length prefix before reserving memory.

## Recommendation

Add a maximum frame size guard in `FrameDecoder::decode` at `rs/canister_sandbox/src/frame_decoder.rs` before the `data.reserve(size)` call (line 47):

```rust
FrameDecoderState::Length(size) => {
    let size: usize = *size as usize;
    const MAX_FRAME_SIZE: usize = 256 * 1024 * 1024; // 256 MiB
    if size > MAX_FRAME_SIZE {
        panic!("IPC frame size {} exceeds maximum allowed size", size);
    }
    if data.len() < size {
        data.reserve(size);
        return None;
    }
```

The constant should be chosen to exceed the largest legitimate IPC message (compiled Wasm modules are the largest payload). Optionally, `SocketReaderConfig` could carry a `max_frame_size` field so different IPC channels can enforce different limits.

## Proof of Concept

```rust
// Unit test / fuzz target — safe to run locally, does NOT abort if guard is present
#[test]
fn test_oversized_frame_aborts_without_guard() {
    let mut decoder = FrameDecoder::<SomeMessage>::new();
    let mut buf = BytesMut::new();
    // Craft a u64 length of usize::MAX in big-endian
    buf.extend_from_slice(&0xFFFFFFFFFFFFFFFFu64.to_be_bytes());
    // Without the guard, this call invokes handle_alloc_error → abort():
    decoder.decode(&mut buf);
}
```
