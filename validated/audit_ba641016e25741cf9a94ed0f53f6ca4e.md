### Title
Unchecked Precompile Call Return Value After Token Burn Causes Permanent Fund Loss - (File: `etc/eth-contracts/contracts/EvmErc20.sol`, `etc/eth-contracts/contracts/EvmErc20V2.sol`)

---

### Summary

Both `EvmErc20` and `EvmErc20V2` implement `withdrawToNear()` and `withdrawToEthereum()` by first burning the caller's tokens and then invoking a hardcoded precompile address via inline assembly. The return value of the assembly `call()` is captured in a local variable `res` but is **never checked**. If the precompile call fails for any reason, the tokens are permanently destroyed with no withdrawal ever occurring — a silent, unrecoverable fund loss for the user.

---

### Finding Description

In both `EvmErc20.sol` and `EvmErc20V2.sol`, the withdrawal functions follow this pattern:

1. `_burn(_msgSender(), amount)` — tokens are irreversibly destroyed.
2. An assembly `call()` is made to a hardcoded precompile address.
3. The result `res` is stored but **never inspected**; no `require` or revert follows.

**`EvmErc20.sol` — `withdrawToNear()`:** [1](#0-0) 

**`EvmErc20.sol` — `withdrawToEthereum()`:** [2](#0-1) 

**`EvmErc20V2.sol` — `withdrawToNear()`:** [3](#0-2) 

**`EvmErc20V2.sol` — `withdrawToEthereum()`:** [4](#0-3) 

The precompile addresses are:
- `0xe9217bc70b7ed1f598ddd3199e80b093fa71124f` (NEAR withdrawal)
- `0xb0bd02f6a392af548bdf1cfaee5dfa0eefcc8eab` (Ethereum withdrawal)

If either precompile call returns `0` (failure), execution continues normally — the function returns without reverting, and the burned tokens are gone forever.

---

### Impact Explanation

**Critical — Permanent freezing/destruction of user funds.**

The burn is unconditional and precedes the precompile call. A failed precompile call (returning `0`) leaves the state with tokens destroyed and no corresponding withdrawal credited on NEAR or Ethereum. The user's funds are permanently lost with no recovery path, since the burn cannot be undone and no event or revert signals the failure.

---

### Likelihood Explanation

Any token holder can call `withdrawToNear()` or `withdrawToEthereum()` directly. The `recipient` bytes parameter in `withdrawToNear()` is fully user-controlled and variable-length, creating a realistic surface for malformed input to cause the precompile to reject the call silently. Additionally, any transient precompile-level failure (e.g., out-of-gas within the precompile, unexpected input encoding) would trigger the same silent loss. The functions are public and require no special privileges. [5](#0-4) 

---

### Recommendation

Check the return value of the assembly `call()` and revert if it indicates failure. The burn should either be moved **after** a successful precompile call, or the call result must be asserted:

```solidity
// Option A: check res and revert on failure
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}

// Option B: burn only after confirmed success
// (move _burn after the assembly block and gate it on res == 1)
```

Apply the same fix to all four withdrawal functions across both `EvmErc20.sol` and `EvmErc20V2.sol`.

---

### Proof of Concept

1. User holds 1000 `EvmErc20` tokens.
2. User calls `withdrawToNear(malformed_recipient_bytes, 1000)`.
3. `_burn(user, 1000)` executes — user balance drops to 0, total supply decreases.
4. Assembly `call()` to the NEAR withdrawal precompile fails (returns `0`) due to malformed recipient encoding.
5. `res` is silently discarded; function returns normally.
6. User has 0 tokens on Aurora and receives nothing on NEAR — 1000 tokens are permanently destroyed. [6](#0-5) [7](#0-6)
