### Title
Malicious Participant Can Corrupt CKD Output by Sending Cross-Session Contributions — (`src/confidential_key_derivation/protocol.rs`)

### Summary

`do_ckd_coordinator` aggregates participant contributions with no verification that each `(big_y, big_c)` pair was computed for the current session's `(app_id, app_pk)`. A malicious participant can send a contribution computed for any prior session — or any arbitrary group elements — and the coordinator will silently aggregate it, producing a `CKDOutput` that fails `unmask`/`verify_signature`.

### Finding Description

The CKD protocol is a single-round aggregation. Each participant computes:

```
big_c_i = x_i * H(pk || app_id) + y_i * app_pk
```

and sends `(λ_i * big_y_i, λ_i * big_c_i)` privately to the coordinator. [1](#0-0) 

The coordinator collects these via `recv_from_others` and sums them unconditionally: [2](#0-1) 

`recv_from_others` enforces only two things: the sender is in the participant list, and each participant sends exactly once. It performs no content validation: [3](#0-2) 

The channel tag used for all CKD sessions is a **fixed constant** derived from `NEAR_CHANNEL_TAGS_DOMAIN || "root shared"` — it does not incorporate `app_id`, `app_pk`, or any session nonce: [4](#0-3) 

There is no zero-knowledge proof, commitment, or any other mechanism binding a participant's contribution to the current session's `(app_id, app_pk)`.

### Impact Explanation

A malicious participant P_j who participated in session 1 (`app_id_1`, `app_pk_1`) can replay their session-1 contribution `(λ_j * y_j * G, λ_j * (x_j * H(pk || app_id_1) + y_j * app_pk_1))` into session 2 (`app_id_2`, `app_pk_2`). The coordinator aggregates:

```
big_C = Σ_{honest i} λ_i*(x_i*H(pk||app_id_2) + y_i*app_pk_2)
      + λ_j*(x_j*H(pk||app_id_1) + y_j*app_pk_1)
```

This is not equal to `msk * H(pk || app_id_2) + Y * app_pk_2`. The resulting `CKDOutput` is cryptographically invalid: `unmask(app_sk_2)` yields garbage, and `verify_signature` fails. [5](#0-4) 

The coordinator returns this corrupted output with no error, since there is no final-output verification step: [6](#0-5) 

Impact: **High** — corruption of CKD output so honest parties accept an unusable cryptographic output.

### Likelihood Explanation

Any single malicious MPC node participating in two CKD sessions can execute this attack. The attacker only needs to record their own outgoing message from session 1 (which they generated themselves) and re-deliver it during session 2. No cryptographic assumptions need to be broken. The attack is trivially local-testable.

### Recommendation

Add a binding between participant contributions and the current session's `(app_id, app_pk)`. Options include:

1. **ZK proof of correct computation**: Each participant proves in zero-knowledge that their `(big_y, big_c)` satisfies `big_c = x_i * H(pk || app_id) + y_i * app_pk` for the coordinator-specified `(app_id, app_pk)`.
2. **Session-tagged channel**: Incorporate `app_id` and `app_pk` into the `ChannelTag` derivation so that messages from a different session are structurally incompatible.
3. **Coordinator echo**: The coordinator broadcasts `(app_id, app_pk)` before collecting contributions, and participants include a MAC/signature over `(app_id, app_pk, big_y, big_c)` in their message.

### Proof of Concept

```rust
//   msg_j_session1 = (norm_big_y_j, norm_big_c_j)  [computed for app_id_1, app_pk_1]

assert!(verify_signature(&public_key, &app_id_2, &confidential_key).is_err());
// ^ This assertion passes — the output is corrupted/unusable.
```

```rust
pub async fn recv_from_others<T>(
    chan: &SharedChannel,
    waitpoint: u64,
    participants: &ParticipantList,
    me: Participant,
) -> Result<Vec<(Participant, T)>, ProtocolError>
where
    T: serde::de::DeserializeOwned,
{
    let mut seen = ParticipantCounter::new(participants);
    seen.put(me);
    let mut messages = Vec::with_capacity(participants.others(me).count());

    while !seen.full() {
        let (from, msg) = chan.recv(waitpoint).await?;
        if seen.put(from) {
            messages.push((from, msg));
        }
    }

    Ok(messages)
```
