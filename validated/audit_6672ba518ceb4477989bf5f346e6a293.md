### Title
Silent Failure in Promise Result Handling Causes Permanent Fund Loss on wNEAR Exit — (`engine/src/contract_methods/connector.rs`)

### Summary

`exit_to_near_precompile_callback` checks only for `PromiseResult::Successful` and falls through to a bare `else { None }` branch when the upstream `ft_transfer_call` promise fails and the `error_refund` compile-time feature is disabled. This is the direct Rust/NEAR analog of the Solidity `catch Error(string memory)` pattern: only one specific outcome is handled; all other failure modes are silently swallowed. The result is that a user's ERC-20 wNEAR tokens are permanently burned in the EVM with no on-chain refund.

---

### Finding Description

In `engine/src/contract_methods/connector.rs`, `exit_to_near_precompile_callback` is the NEAR callback that fires after the `ft_transfer_call` (or `ft_transfer`) promise that moves wNEAR out of Aurora:

```rust
let maybe_result = if let Some(PromiseResult::Successful(_)) = handler.promise_result(0) {
    // success path – optionally schedule NEAR transfer
    ...
    None
} else if let Some(args) = args.refund {          // ← only reached when error_refund is ON
    let refund_result = engine::refund_on_error(...)?;
    ...
    Some(refund_result)
} else {
    None   // ← silent swallow of PromiseResult::Failed
};
``` [1](#0-0) 

The `args.refund` field is populated at the call-site only when the `error_refund` Cargo feature is compiled in:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,
    transfer_near: transfer_near_args,
};
``` [2](#0-1) 

`ExitToNearPrecompileCallbackArgs` is defined as:

```rust
pub struct ExitToNearPrecompileCallbackArgs {
    pub refund: Option<RefundCallArgs>,
    pub transfer_near: Option<TransferNearArgs>,
}
``` [3](#0-2) 

When `error_refund` is absent, `refund` is always `None`. The callback is still scheduled whenever `transfer_near` is `Some` (the wNEAR-unwrap path). If the `ft_transfer_call` promise then returns `PromiseResult::Failed`, the `else if let Some(args) = args.refund` guard is false, and execution falls into the bare `else { None }` arm — returning `Ok(None)` with no refund, no error, and no state correction.

The three-variant `PromiseResult` enum is:

```rust
pub enum PromiseResult {
    NotReady,
    Successful(Vec<u8>),
    Failed,
}
``` [4](#0-3) 

The engine's own `promise_result_check` helper correctly guards against both `Failed` **and** `NotReady`:

```rust
if let Some(PromiseResult::Failed | PromiseResult::NotReady) =
    self.promise_result(index)
{
    return Some(false);
}
``` [5](#0-4) 

`exit_to_near_precompile_callback` uses neither of these guards; it only pattern-matches `Successful`.

---

### Impact Explanation

When `error_refund` is not compiled in:

1. A user calls the `ExitToNear` precompile to unwrap wNEAR (flag byte `0x00` or `0x01` with `:unwrap` suffix).
2. The EVM burns the user's ERC-20 wNEAR balance.
3. Aurora schedules `ft_transfer_call` on the wNEAR NEP-141 contract with a callback to `exit_to_near_precompile_callback`.
4. The `ft_transfer_call` fails for any reason (wNEAR contract paused, insufficient storage deposit, gas exhaustion in the callee, etc.).
5. The callback fires with `PromiseResult::Failed`.
6. The `else { None }` branch executes; the function returns `Ok(None)`.
7. The burned ERC-20 tokens are gone; no NEAR is transferred; no ERC-20 is re-minted.

The user suffers a **permanent, irrecoverable loss** of their wNEAR. This matches the "Permanent freezing of funds" / "Direct theft of user funds" impact categories.

The test suite explicitly confirms this behavior:

```rust
// If the refund feature is not enabled, then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let expected_balance = Wei::new_u64(INITIAL_ETH_BALANCE - ETH_EXIT_AMOUNT);
``` [6](#0-5) 

---

### Likelihood Explanation

Any EVM user holding wNEAR on Aurora can trigger this path by calling the `ExitToNear` precompile with the `:unwrap` suffix. The `ft_transfer_call` to the wNEAR contract can fail due to:

- The wNEAR contract being paused or upgraded mid-flight.
- The Aurora engine account not being registered for storage on the wNEAR contract.
- Gas exhaustion in the wNEAR `ft_on_transfer` callback.
- Any future change to the wNEAR contract that causes it to revert.

The vulnerability is present in every build where `error_refund` is not a compiled feature. The `ExitToNear` precompile is a core, publicly accessible interface.

---

### Recommendation

Remove the compile-time gating on the refund logic. The failure branch should always attempt a refund regardless of the `error_refund` feature flag. At minimum, a failed promise with no refund args should return an explicit error rather than `Ok(None)`:

```diff
-        } else {
-            None
+        } else {
+            // Promise failed but no refund args available; surface the error
+            return Err(errors::ERR_REFUND_FAILURE.into());
         };
```

Ideally, the `refund` field should always be populated (unconditionally) so that any `ft_transfer_call` failure triggers a re-mint or ETH credit, mirroring the behavior the `error_refund` feature already implements.

---

### Proof of Concept

1. Deploy Aurora without the `error_refund` feature.
2. Bridge wNEAR to Aurora (user holds ERC-20 wNEAR).
3. Drain the Aurora engine's NEP-141 storage balance so the next `ft_transfer_call` will fail (as done in `test_exit_to_near_eth_refund`).
4. Call the `ExitToNear` precompile with the `:unwrap` suffix.
5. Observe: ERC-20 wNEAR is burned; `ft_transfer_call` fails; callback returns `Ok(None)`; user balance is permanently reduced by the exit amount with no refund. [7](#0-6)
