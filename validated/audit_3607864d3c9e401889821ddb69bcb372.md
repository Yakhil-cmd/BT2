Audit Report

## Title
Hot Keys Not Cleared When NNS Neuron Is Spawned to a Different Controller - (File: `rs/nns/governance/src/governance.rs`)

## Summary
`spawn_neuron` unconditionally copies the parent neuron's `hot_keys` list to the child neuron even when the child is assigned a completely different controller via `spawn.new_controller`. Because hot keys are authorized to vote and change followees on any neuron they are attached to, the original owner's hot keys retain unauthorized governance control over a neuron now owned by a different principal.

## Finding Description
In `spawn_neuron`, the `child_controller` is correctly set to the caller-supplied `new_controller` when present: [1](#0-0) 

However, the child neuron is then built with an unconditional copy of the parent's hot keys: [2](#0-1) 

`is_authorized_to_vote` grants full voting and followee-management rights to any principal in `hot_keys`, regardless of whether that principal is the controller: [3](#0-2) 

There is no conditional guard anywhere in `spawn_neuron` that clears or filters `hot_keys` when `child_controller != parent_neuron.controller()`. The new controller (Bob) receives a neuron with hot keys entirely controlled by the spawning party (Alice), effective immediately upon neuron creation.

## Impact Explanation
This is **High** severity: unauthorized access to a governance neuron. Alice's retained hot keys can vote on NNS proposals using Bob's neuron's voting power and redirect Bob's liquid-democracy delegation by changing followees — both without Bob's knowledge or consent. Bob must proactively discover and remove the hot keys; until he does, Alice exercises governance rights over a neuron Bob controls. This matches the allowed impact: *"Unauthorized access to neurons, governance assets… where exploitation requires meaningful per-target work or other constraints"* — here the constraint is that the attacker must hold a neuron with maturity and at least one registered hot key.

## Likelihood Explanation
The attack requires no special privileges. Any principal holding a neuron with accumulated maturity and at least one hot key registered can call `manage_neuron → Spawn` with `new_controller` set to any victim principal. The endpoint is publicly documented and accessible via standard ingress. The attack is repeatable and costs only the maturity consumed by the spawn operation.

## Recommendation
When `spawn.new_controller` is set to a principal different from `parent_neuron.controller()`, initialize the child neuron with an empty hot keys list:

```rust
let inherited_hot_keys = if child_controller == parent_neuron.controller() {
    parent_neuron.hot_keys.clone()
} else {
    vec![]
};
// ...
.with_hot_keys(inherited_hot_keys)
```

The new controller can add their own hot keys after taking ownership. The same pattern should be reviewed for `split_neuron`, though impact there is lower since the controller is always inherited unchanged.

## Proof of Concept
1. Alice registers hot key `H` on her neuron `N` and accumulates maturity.
2. Alice calls `manage_neuron` with `Command::Spawn(Spawn { new_controller: Some(Bob), nonce: Some(42), percentage_to_spawn: Some(100) })`.
3. Governance creates child neuron `C` with `controller = Bob` and `hot_keys = [H]` (confirmed at line 2714).
4. `H` immediately calls `manage_neuron` on `C` with `Command::Follow(...)` or `Command::RegisterVote(...)`, exercising Bob's voting power on any open proposal.
5. Bob must query `get_full_neuron` and issue `RemoveHotKey` before `H` acts — a race condition with no notification mechanism.

A deterministic integration test can confirm this by: spawning a neuron to a different controller, asserting `hot_keys` on the child equals the parent's list, then verifying `is_authorized_to_vote` returns `true` for the original hot key principal against the child neuron.
