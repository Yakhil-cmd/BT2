### Title
Missing Caller Restriction on `update_rewards` Allows Any Actor to Permanently Suppress Block Reward Distribution - (File: src/staking/staking.cairo)

### Summary
The `update_rewards` function in the `Staking` contract is specified to be callable only by the Starkware sequencer, but the implementation contains no caller check. Any unprivileged address can call `update_rewards` with `disable_rewards: true` on any valid staker address, consuming the single global `last_reward_block` slot for that block. This permanently prevents the sequencer from distributing block rewards to any staker for that block, causing irreversible loss of unclaimed yield.

### Finding Description
The spec for `update_rewards` explicitly states:

> **Access control:** Only starkware sequencer.

However, the implementation at `src/staking/staking.cairo` lines 1449–1488 contains no such check. The only guards are:

1. `general_prerequisites()` — checks the pause flag only.
2. `current_block_number > self.last_reward_block.read()` — a global, single-slot anti-replay guard.
3. Staker existence and balance checks.

```rust
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
    let current_block_number = starknet::get_block_number();
    assert!(
        current_block_number > self.last_reward_block.read(),
        "{}",
        Error::REWARDS_ALREADY_UPDATED,
    );
    ...
    // Update last block rewards.
    self.last_reward_block.write(current_block_number);

    if disable_rewards || self.is_pre_consensus() {
        return;   // ← exits with no rewards distributed
    }
    ...
```

The `last_reward_block` field is a **single global** storage slot:

```rust
/// Last block number for which rewards were distributed.
last_reward_block: BlockNumber,
```

Because it is global (not per-staker), a single call to `update_rewards` in block N — regardless of which staker is named — exhausts the slot for the entire block. Any subsequent call in the same block reverts with `REWARDS_ALREADY_UPDATED`.

An attacker can therefore:
1. Call `update_rewards(any_valid_staker, disable_rewards: true)` at the first transaction of every block.
2. `last_reward_block` is written to the current block number.
3. The function returns immediately because `disable_rewards == true`, distributing **zero** rewards.
4. Every subsequent sequencer call for every staker in that block fails with `REWARDS_ALREADY_UPDATED`.
5. Block rewards for that block are **permanently lost** — they are never credited to `unclaimed_rewards_own` and never minted.

### Impact Explanation
Block rewards that are not distributed in a given block are permanently destroyed. The `RewardSupplier` only increments `unclaimed_rewards` when `_update_rewards` actually runs; a suppressed block leaves no residual credit. All stakers and their delegators lose the yield they would have earned for every suppressed block. This matches the allowed impact: **Permanent freezing of unclaimed yield**.

### Likelihood Explanation
The attack requires only a valid staker address (trivially obtained from on-chain `NewStaker` events) and enough gas to submit one transaction per block. There is no economic barrier. A griefing actor with no profit motive can sustain this indefinitely. The attack is also front-runnable: the attacker simply ensures their transaction lands before the sequencer's `update_rewards` call in each block.

### Recommendation
Add an explicit caller check matching the specification. The simplest fix is to restrict `update_rewards` to the sequencer address stored in contract configuration, analogous to how `update_rewards_from_attestation_contract` restricts to the attestation contract:

```rust
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
    assert!(
        get_caller_address() == self.sequencer_address.read(),
        "{}",
        Error::CALLER_IS_NOT_SEQUENCER,
    );
    ...
```

Alternatively, if the sequencer address is not stored, restrict to the attestation contract or introduce a dedicated sequencer role consistent with the existing `RolesComponent`.

### Proof of Concept

```
// Attacker script (pseudocode, executed once per block):
//
// 1. Read any active staker address from NewStaker events.
//    staker_address = <any valid staker>
//
// 2. Call update_rewards with disable_rewards = true.
//    staking.update_rewards(staker_address, disable_rewards: true)
//    → last_reward_block is set to current block
//    → function returns immediately, zero rewards distributed
//
// 3. Sequencer attempts its own update_rewards call for any staker:
//    staking.update_rewards(staker_B, disable_rewards: false)
//    → PANICS: "Rewards already updated for this block"
//
// Result: No staker receives block rewards for this block.
//         Repeating every block permanently suppresses all consensus reward issuance.
```
