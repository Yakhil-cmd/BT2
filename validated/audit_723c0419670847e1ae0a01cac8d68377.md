### Title
Missing `threshold < 2` Guard in FROST `assert_sign_inputs` — Inconsistent Duplicate Validation Relative to DKG (`assert_key_invariants`) - (File: src/frost/mod.rs)

---

### Summary

The DKG initialization path (`assert_key_invariants`) and triple-generation path (`validate_triple_inputs`) both enforce a minimum threshold of 2. The FROST signing entry-point (`assert_sign_inputs`) is a copy of the same validation pattern but omits the `threshold < 2` guard. A caller who passes `threshold = 1` to the FROST signing protocol bypasses the minimum-threshold invariant, causing Lagrange interpolation to treat a single participant's share as the full secret and producing a cryptographically invalid, unusable signature output.

---

### Finding Description

`assert_key_invariants` in `src/dkg.rs` enforces three threshold constraints:

```
threshold > participants.len()  → ThresholdTooLarge
threshold < 2                   → ThresholdTooSmall   ← present here
``` [1](#0-0) 

`validate_triple_inputs` in the OT-based triple generation path mirrors this exactly, including the `threshold_value < 2` guard: [2](#0-1) 

`assert_sign_inputs` in `src/frost/mod.rs` is the analogous validation function for FROST signing. It checks `threshold.value() > participants.len()` but **omits** the `threshold < 2` lower-bound check entirely: [3](#0-2) 

Similarly, `presign` in `src/ecdsa/ot_based_ecdsa/presign.rs` checks the upper bound but not the lower bound: [4](#0-3) 

(Note: the OT-based presign case is partially mitigated because the threshold is cross-checked against the pre-generated triple thresholds, which were themselves validated with `validate_triple_inputs`. The FROST `assert_sign_inputs` path has no such secondary guard.)

---

### Impact Explanation

When `threshold = 1` is accepted by `assert_sign_inputs`:

1. The participant-count check `threshold.value() > participants.len()` passes for any non-empty list.
2. The signing session proceeds with a single-participant Lagrange basis. For a set of size 1, the Lagrange coefficient is identically **1**, so the participant's signing share `x_i` is used directly as if it were the reconstructed secret `x`.
3. The resulting FROST signature is computed over `x_i` (a share) rather than `x` (the secret). It is cryptographically invalid and fails verification against the public key derived from `x`.
4. Honest parties relying on the output receive an unusable signature — a corrupted signing output.

**Allowed impact matched:** *High — Corruption of sign outputs so honest parties accept unusable cryptographic outputs.*

---

### Likelihood Explanation

`assert_sign_inputs` is a public library function. Any unprivileged library caller (application integrator, malicious coordinator, or participant constructing their own signing session) can supply `threshold = 1` directly. The `ReconstructionLowerBound` type does not enforce a minimum of 2 at the type level — the `ThresholdTooSmall` error variant exists precisely because the type admits values below 2, and the only enforcement is the explicit guard that is present in DKG but absent in signing. [5](#0-4) 

---

### Recommendation

Add the same lower-bound guard to `assert_sign_inputs` (and to the OT-based `presign` as a defence-in-depth measure):

```rust
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

This mirrors the check already present in `assert_key_invariants` and `validate_triple_inputs`, making the validation consistent across all protocol entry points.

---

### Proof of Concept

1. Call `assert_sign_inputs` with `threshold = 1` and a valid two-participant list. The call succeeds (no error).
2. Proceed with FROST signing using the returned `ParticipantList` containing only one participant.
3. The Lagrange coefficient for that sole participant evaluates to scalar `1`.
4. The signature share sent to the coordinator equals `x_i · nonce_i`, not the correct Lagrange-weighted combination.
5. The coordinator aggregates a single share and outputs a signature that fails `verify()` against the group public key — a corrupted, unusable signing output.

The root cause is the missing guard at: [6](#0-5) 
compared to the equivalent guard that is present at: [7](#0-6)

```rust
pub fn assert_sign_inputs(
    participants: &[Participant],
    threshold: impl Into<ReconstructionLowerBound>,
    me: Participant,
    coordinator: Participant,
) -> Result<ParticipantList, InitializationError> {
    let threshold = threshold.into();
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // validate threshold
    if threshold.value() > participants.len() {
        return Err(InitializationError::ThresholdTooLarge {
            threshold: threshold.value(),
            max: participants.len(),
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }
    Ok(participants)
}
