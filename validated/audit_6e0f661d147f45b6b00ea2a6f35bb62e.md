### Title
ETH Permanently Locked in `RewardSupplier` When `tick()` Called With No Pending Mint Requests - (File: `L1/starkware/solidity/stake/RewardSupplier.sol`)

---

### Summary

`RewardSupplier.sol`'s `tick()` function is `payable` and accepts ETH as L1→L2 message fees. When there are no pending L2 mint requests (`amountToMint == 0`), the entire `msg.value` is silently discarded into the contract with no refund path, no `receive` function, and no ETH withdrawal mechanism anywhere in the contract or its inheritance chain.

---

### Finding Description

`tick()` is declared `external payable` and splits `msg.value` between two downstream calls only inside the `if (amountToMint > 0)` branch:

```solidity
function tick() external payable {
    (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

    if (amountToMint > 0) {
        // ...
        uint256 msgFee = msg.value / 2;
        bridge().depositWithMessage{value: msgFee}(...);   // consumes half
        msgFee = msg.value - msgFee;
        messagingContract().sendMessageToL2{value: msgFee}(...); // consumes rest
    }
    // amountToMint == 0: msg.value is never used, never refunded
}
``` [1](#0-0) 

When `amountToMint == 0`, execution exits the function without touching `msg.value`. The ETH accumulates in the contract. Neither `RewardSupplierStorage` nor `ProxySupportImpl` define a `receive` function, a `fallback` function, or any ETH withdrawal/sweep mechanism. [2](#0-1) [3](#0-2) 

---

### Impact Explanation

Any ETH sent to `tick()` during a no-op call (when `amountToMint == 0`) is permanently unrecoverable. There is no admin sweep, no refund, and no `receive`/`fallback` to route it. This constitutes **permanent freezing of the caller's funds** (High impact per the allowed scope).

---

### Likelihood Explanation

`tick()` is `external` with no access control — any public caller can invoke it. A realistic race condition: a caller reads `requiredMinting()` off-chain, observes pending messages, prepares a transaction with ETH, but a competing caller processes those messages first. The original transaction lands with `amountToMint == 0` and the ETH is locked. Additionally, any caller who simply calls `tick()` with ETH when the queue is empty (e.g., as a keep-alive ping) loses their ETH permanently.

---

### Recommendation

Add an ETH refund when `amountToMint == 0`, or require `msg.value == 0` in that branch:

```solidity
function tick() external payable {
    (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

    if (amountToMint > 0) {
        // ... existing logic ...
    } else {
        // Refund any ETH sent when there is nothing to process.
        if (msg.value > 0) {
            (bool ok, ) = msg.sender.call{value: msg.value}("");
            require(ok, "ETH_REFUND_FAILED");
        }
    }
}
```

Alternatively, enforce `require(msg.value == 0 || amountToMint > 0, "NO_MINT_NEEDED")` at the top of `tick()` to prevent callers from accidentally sending ETH on a no-op call.

---

### Proof of Concept

1. Deploy `RewardSupplier` with no pending L2→L1 mint-request messages (i.e., `messagingContract().l2ToL1Messages(msgHash) == 0`).
2. Call `tick{value: 1 ether}()` from any EOA.
3. `requiredMinting()` returns `(0, 0)`.
4. The `if (amountToMint > 0)` branch is skipped entirely.
5. The function returns. The contract's ETH balance increases by `1 ether`.
6. No function in `RewardSupplier`, `RewardSupplierStorage`, or `ProxySupportImpl` allows recovering that ETH.
7. The 1 ETH is permanently locked. [1](#0-0)

    function tick() external payable {
        // Check if minting is required, and how much.
        (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

        if (amountToMint > 0) {
            // Prepare the L2->L1 mintRequest message for consumption.
            uint256[] memory messagePayload = new uint256[](1);
            messagePayload[0] = TOKENS_PER_MINT_REQUEST;

            // Consume the mintRequest messages.
            for (uint256 i = 0; i < numMsgsToConsume; i++) {
                messagingContract().consumeMessageFromL2(mintRequestSource(), messagePayload);
            }

            // Reuest minting of the requested amount from the mint manager.
            mintManager().mintRequest(token(), amountToMint);

            // Deposit the minted amount onto the bridge to the credit of `mintDestination`.
            uint256 msgFee = msg.value / 2;
            bridge().depositWithMessage{value: msgFee}(
                token(),
                amountToMint,
                mintDestination(),
                new uint256[](0)
            );
            emit ConsumedL2MintRequests(numMsgsToConsume, amountToMint);

            // Send a totalSupply update to L2MintCurve.
            msgFee = msg.value - msgFee;
            messagePayload[0] = IERC20(token()).totalSupply();
            messagingContract().sendMessageToL2{value: msgFee}(
                mintingCurve(),
                UPDATE_TOTAL_SUPPLY_SELECTOR,
                messagePayload
            );
        }
    }
