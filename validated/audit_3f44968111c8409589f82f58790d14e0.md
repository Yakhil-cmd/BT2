### Title
Partial NEP-141 `ft_transfer_call` Refund Silently Discarded in Omni ERC-20 Exit — Permanent Loss of User Funds - (`engine/src/contract_methods/connector.rs`)

### Summary

When an EVM user calls `withdrawToNear` with an Omni message on an ERC-20 mirror token, the engine burns the full ERC-20 amount and schedules `ft_transfer_call` on the NEP-141 contract. If the NEP-141 receiver's `ft_on_transfer` returns a non-zero partial refund (i.e., it did not consume all tokens), the NEP-141 standard credits the unused tokens back to Aurora's NEP-141 balance and returns `Successful(<refund_amount>)`. The `exit_to_near_precompile_callback` matches on `Successful(_)` — discarding the return value entirely — and performs no ERC-20 re-mint. The partial refund amount is permanently stranded in Aurora's NEP-141 balance with no corresponding ERC-20 tokens, and the user suffers an irrecoverable loss.

---

### Finding Description

**Step 1 — ERC-20 burn and precompile dispatch**

`EvmErc20.withdrawToNear(recipient:msg, amount)` calls `_burn(sender, amount)` then invokes the `ExitToNear` precompile. Inside `exit_erc20_token_to_near`, the `Message::Omni(msg)` branch is taken: [1](#0-0) 

This schedules `ft_transfer_call(receiver_id, amount, msg)` on the NEP-141 contract. The event records the **full original `amount`**.

**Step 2 — `refund_call_args` encodes the full amount for the error path** [2](#0-1) 

`refund_call_args` stores the full event `amount` in `RefundCallArgs`. This is intended for the **failure** path only.

**Step 3 — Callback construction** [3](#0-2) 

The callback `exit_to_near_precompile_callback` is always scheduled when `error_refund` is enabled (or when `transfer_near` is set). The `callback_args.refund` holds the full original amount.

**Step 4 — The callback ignores the `ft_transfer_call` return value on success** [4](#0-3) 

The critical flaw is at line 214. The pattern `Successful(_)` discards the return value of `ft_transfer_call`. Per the NEP-141 standard, `ft_transfer_call` returns the amount that was **not consumed** by the receiver (the partial refund). When the receiver returns `30` out of `100`, the NEP-141 contract:
1. Credits `30` back to Aurora's NEP-141 balance.
2. Returns `Successful("30")` as the promise result.

The callback sees `Successful(_)`, enters the first branch, does nothing (no `transfer_near` in the Omni path), and returns `None`. The `else if let Some(args) = args.refund` branch — which would re-mint ERC-20 tokens — is **never reached** because the promise succeeded.

**Step 5 — Accounting imbalance**

| | Before | After |
|---|---|---|
| User ERC-20 balance | N | N − 100 |
| Aurora NEP-141 balance | M | M − 70 (net: sent 100, got 30 back) |
| ERC-20 total supply | N | N − 100 |

Aurora holds 30 more NEP-141 tokens than there are ERC-20 tokens to back them. The user permanently loses 30 ERC-20 tokens. Those 30 NEP-141 tokens are stranded in Aurora's account with no redemption path.

---

### Impact Explanation

**Critical — Direct theft of user funds.** Any user who exits ERC-20 tokens via the Omni path to a NEP-141 receiver that partially consumes the transfer loses the unconsumed portion permanently. The tokens cannot be recovered: there is no ERC-20 to burn for a second exit, and no admin path to re-mint them.

---

### Likelihood Explanation

The Omni path is a production feature. The NEP-141 standard explicitly supports partial refunds from `ft_on_transfer`. Any legitimate Omni bridge contract (e.g., the NEAR Omni-bridge) that routes tokens and returns unused amounts will trigger this. No privileged access, no key compromise, and no external oracle is required — only a standard NEP-141 `ft_transfer_call` with a receiver that returns a non-zero refund.

### Recommendation

In `exit_to_near_precompile_callback`, read the return value of the `ft_transfer_call` promise result and, if it is non-zero, re-mint that amount of ERC-20 tokens to the original sender:

```rust
if let Some(PromiseResult::Successful(bytes)) = handler.promise_result(0) {
    // Parse the refund amount returned by ft_transfer_call
    let refund_amount: u128 = std::str::from_utf8(&bytes)
        .ok()
        .and_then(|s| s.trim_matches('"').parse().ok())
        .unwrap_or(0);

    if refund_amount > 0 {
        if let Some(refund_args) = args.refund {
            // Re-mint the partial refund back to the original sender
            let partial_refund = RefundCallArgs {
                amount: types::u256_to_arr(&U256::from(refund_amount)),
                ..refund_args
            };
            engine::refund_on_error(io, env, state, &partial_refund, handler)?;
        }
    }
    // ... existing transfer_near logic
}
```

### Proof of Concept

1. Deploy a NEP-141 token `token.near`; bridge it to Aurora as ERC-20 `ERC20_TOKEN`.
2. Deploy a malicious (or standard) NEP-141 receiver `receiver.near` whose `ft_on_transfer` always returns `amount / 2` (uses half, refunds half).
3. On Aurora, call `ERC20_TOKEN.withdrawToNear("receiver.near:<omni_msg>", 100)`.
4. Observe: `_burn(user, 100)` executes; `ft_transfer_call(receiver.near, 100, omni_msg)` is called; `receiver.near` returns `"50"`; NEP-141 credits 50 back to Aurora's `token.near` balance.
5. `exit_to_near_precompile_callback` fires with `Successful("50")` → enters the `if` branch → returns `None` → no re-mint.
6. Final state: user's ERC-20 balance decreased by 100; Aurora's `token.near` balance increased by 50 (stranded); user lost 50 ERC-20 tokens with no recovery path.

The root cause is the wildcard discard at: [5](#0-4) 

and the absence of any partial-refund handling in the success branch of `exit_to_near_precompile_callback`.
