### Title
Missing Lower-Bound Validation on `threshold` in `assert_sign_inputs` Enables Denial of EdDSA Signing — (File: `src/frost/mod.rs`)

---

### Summary

`assert_sign_inputs` in `src/frost/mod.rs` checks that `threshold ≤ participants.len()` but never checks that `threshold ≥ 2`. The sibling function `assert_key_invariants` in `src/dkg.rs` enforces both bounds. A caller or malicious coordinator can supply `threshold = 0` or `threshold = 1` to the EdDSA/RedJubjub signing entry-point; the initialization succeeds silently, and the downstream FROST signing protocol produces an unverifiable signature, permanently denying signing for honest parties.

---

### Finding Description

`assert_key_invariants` in `src/dkg.rs` enforces a strict lower bound:

```rust
// src/dkg.rs  lines 580-582
if threshold < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
}
``` [1](#0-0) 

`assert_sign_inputs` in `src/frost/mod.rs` only checks the upper bound:

```rust
// src/frost/mod.rs  lines 144-150
// validate threshold
if threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge {
        threshold: threshold.value(),
        max: participants.len(),
    });
}
``` [2](#0-1) 

There is no corresponding `threshold < 2` guard. The same omission exists in the `presign` entry-point in the same file:

```rust
// src/frost/mod.rs  lines 71-77
// validate threshold
if args.threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
// ← no lower-bound check
``` [3](#0-2) 

`ReconstructionLowerBound` is a plain `usize` newtype with no invariant enforcement of its own:

```rust
// src/thresholds.rs  lines 9-12
pub struct ReconstructionLowerBound(usize);
impl ReconstructionLowerBound {
    pub fn value(self) -> usize { self.0 }
}
``` [4](#0-3) 

Any `usize` value, including `0` and `1`, converts into `ReconstructionLowerBound` via the derived `From` impl and passes both entry-points without error.

---

### Impact Explanation

In FROST, the threshold controls how many Lagrange-weighted shares are combined to reconstruct the group nonce and signing scalar. When the signing threshold is set to `1` but the key was generated with threshold `≥ 2`, the interpolation uses only one share, producing a scalar that does not correspond to the actual secret. The resulting signature fails the final `verify` step, and the coordinator returns an error. Honest participants have completed their round-trip communication but receive no valid signature. A malicious coordinator who repeatedly initiates sessions with `threshold = 1` causes **permanent denial of signing** for the honest participant set.

With `threshold = 0`, Lagrange interpolation over an empty set is undefined; the library may panic or return a protocol error, again denying signing.

This matches the allowed impact: **High — Permanent denial of signing for honest parties under valid protocol inputs and documented trust assumptions.**

---

### Likelihood Explanation

The coordinator role is explicitly modelled as potentially adversarial in the library's threat model. The coordinator supplies `threshold` directly to `assert_sign_inputs`. No cryptographic material is required to exploit this — passing an integer `1` is sufficient. The missing check is a single-line omission that is easy to overlook because the upper-bound check is present and gives a false sense of completeness.

---

### Recommendation

Add the same lower-bound guard that `assert_key_invariants` already enforces:

```rust
// src/frost/mod.rs — assert_sign_inputs and presign
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
``` [5](#0-4) 

Consider also encoding the invariant directly in `ReconstructionLowerBound::new` so that no construction site can bypass it.

---

### Proof of Concept

```
1. Run DKG with participants = [P1, P2, P3], threshold = 2.
   → assert_key_invariants enforces threshold ≥ 2 ✓

2. Call presign(participants, me, PresignArguments { threshold: 1.into(), ... }, rng)
   → assert_sign_inputs / presign accept threshold = 1 (no lower-bound check).
   → Protocol initialises successfully.

3. Call sign(participants, coordinator, threshold=1, ...)
   → assert_sign_inputs accepts threshold = 1.
   → FROST signing proceeds; Lagrange interpolation uses a single share.
   → Reconstructed scalar ≠ actual signing scalar.
   → sig.verify() fails → ProtocolError returned.
   → Honest participants receive no valid signature.

4. Coordinator repeats step 3 indefinitely → permanent denial of signing.
``` [6](#0-5) [7](#0-6)

pub fn assert_sign_inputs(
    participants: &[Participant],
    threshold: impl Into<ReconstructionLowerBound>,
    me: Participant,
    coordinator: Participant,
) -> Result<ParticipantList, InitializationError> {
    let threshold = threshold.into();
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // validate threshold
    if threshold.value() > participants.len() {
        return Err(InitializationError::ThresholdTooLarge {
            threshold: threshold.value(),
            max: participants.len(),
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }
    Ok(participants)
}
