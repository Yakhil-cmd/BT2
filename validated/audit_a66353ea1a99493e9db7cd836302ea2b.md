### Title
Reward Supplier `claim_rewards` Checks Accounting Variable Instead of Actual Token Balance, Causing Revert That Temporarily Freezes Staker Funds - (File: src/reward_supplier/reward_supplier.cairo)

### Summary
`RewardSupplier::claim_rewards` validates `amount <= unclaimed_rewards` (an accounting variable) but never checks whether the contract's actual ERC-20 token balance is sufficient. During the L1→L2 funding gap — a normal operational state where `unclaimed_rewards` has grown but bridged tokens have not yet arrived — the downstream `checked_transfer` reverts. Because `unstake_action` and `claim_rewards` in the staking contract call through this path atomically, the entire transaction reverts, temporarily preventing stakers from withdrawing their principal even after the exit-wait window has elapsed.

### Finding Description

`RewardSupplier::claim_rewards` (the function called by the staking contract to pull reward tokens) performs only an accounting-level guard:

```cairo
let unclaimed_rewards = self.unclaimed_rewards.read();
assert!(amount <= unclaimed_rewards, "{}", GenericError::AMOUNT_TOO_HIGH);
// ...
token_dispatcher.checked_transfer(recipient: staking_contract, amount: amount.into());
``` [1](#0-0) 

`unclaimed_rewards` is an accounting counter incremented by `update_unclaimed_rewards_from_staking_contract` every time the staking contract records new rewards. The actual ERC-20 balance of the reward supplier contract is replenished asynchronously from L1 via `on_receive` (StarkGate callback). There is always a window — potentially hours to a day — between when `unclaimed_rewards` is incremented and when the bridged tokens actually land. [2](#0-1) 

During that window, `actual_balance < unclaimed_rewards`. The `assert!(amount <= unclaimed_rewards)` guard passes, but `checked_transfer` reverts because the ERC-20 balance is insufficient.

The staking contract calls this path through `claim_from_reward_supplier`:

```cairo
pub(crate) fn claim_from_reward_supplier(...) {
    let balance_before = token_dispatcher.balance_of(account: staking_contract);
    reward_supplier_dispatcher.claim_rewards(:amount);   // ← reverts here
    let balance_after = token_dispatcher.balance_of(account: staking_contract);
    assert!(balance_after - balance_before == amount.into(), "{}", Error::UNEXPECTED_BALANCE);
}
``` [3](#0-2) 

`claim_from_reward_supplier` is called inside `send_rewards_to_staker`: [4](#0-3) 

`send_rewards_to_staker` is called unconditionally at the start of both `unstake_action` and `claim_rewards` in the staking contract: [5](#0-4) [6](#0-5) 

Because the reward-claiming step is not separated from the principal-return step, a revert in `send_rewards_to_staker` rolls back the entire `unstake_action`, including the transfer of the staker's principal. The staker is left unable to recover their tokens until the reward supplier is replenished.

### Impact Explanation

Any staker who calls `unstake_action` (or `claim_rewards`) while the reward supplier's actual ERC-20 balance is below `unclaimed_rewards` will receive a revert. Their staked principal — already removed from the total-stake accounting during `unstake_intent` — remains locked in the staking contract. This constitutes a **temporary freezing of funds** (principal, not just yield) for every affected staker. The freeze lasts until the L1→L2 bridge message is processed and `on_receive` credits the reward supplier.

### Likelihood Explanation

The L1 funding gap is a routine, expected operational state. `unclaimed_rewards` grows with every attestation epoch; the actual balance only grows when a StarkGate deposit is processed. Any staker whose exit-wait window expires during this gap — which can span hours to a day under normal L1 conditions — will be unable to execute `unstake_action`. With many stakers active simultaneously, the probability that at least one staker is affected during any given funding gap is high.

### Recommendation

Decouple the reward-claiming step from the principal-return step in `unstake_action`, or add a real-balance guard in `RewardSupplier::claim_rewards` analogous to the external report's fix:

```cairo
let actual_balance = token_dispatcher.balance_of(account: get_contract_address());
let transfer_amount = if amount > actual_balance { actual_balance } else { amount };
// transfer transfer_amount; defer the remainder
```

Alternatively, restructure `unstake_action` so that a reward-supplier shortfall does not block the return of the staker's principal (e.g., record deferred rewards and allow principal withdrawal independently).

### Proof of Concept

1. Staker calls `unstake_intent()` — exit window starts.
2. Multiple attestation epochs pass; `update_unclaimed_rewards_from_staking_contract` is called, growing `unclaimed_rewards` beyond the reward supplier's current ERC-20 balance. A mint request is sent to L1 but has not yet been bridged.
3. Staker's exit window elapses; staker calls `unstake_action(staker_address)`.
4. Execution path: `unstake_action` → `send_rewards_to_staker` → `claim_from_reward_supplier` → `RewardSupplier::claim_rewards(amount)`.
5. Inside `claim_rewards`: `assert!(amount <= unclaimed_rewards)` **passes** (accounting is correct).
6. `token_dispatcher.checked_transfer(recipient: staking_contract, amount: amount.into())` **reverts** — actual ERC-20 balance of reward supplier < `amount`.
7. Entire `unstake_action` transaction reverts. Staker's principal remains locked in the staking contract despite the exit window having fully elapsed.

        fn send_rewards_to_staker(
            ref self: ContractState,
            staker_address: ContractAddress,
            ref staker_info: InternalStakerInfoLatest,
            token_dispatcher: IERC20Dispatcher,
        ) {
            let reward_address = staker_info.reward_address;
            let amount = staker_info.unclaimed_rewards_own;
