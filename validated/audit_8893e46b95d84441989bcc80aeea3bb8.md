### Title
Counter-Offer Destination Hijack via Crafted Blockchain Notification with `ph` Field — (`packages/gui/src/hooks/useBlockchainNotifications.tsx`)

### Summary

A commented-out line in `prepareNotifications` causes any blockchain notification with `t: 1` and a `ph` (puzzle hash) field to be stored as `type: NotificationType.OFFER` with a `puzzleHash` property attached. Because `canCounterOffer` in `NotificationOffer` accepts both `OFFER` and `COUNTER_OFFER` types, the Counter button is shown and the attacker-controlled puzzle hash is forwarded as the counter-offer destination address.

### Finding Description

**Step 1 — Parsing in `prepareNotifications`** [1](#0-0) 

When `ph` is present in the payload the code was clearly intended to emit `type: NotificationType.COUNTER_OFFER` (the original line is still there, commented out). Instead it emits `type: NotificationType.OFFER` while still attaching `puzzleHash` from the attacker-controlled field:

```ts
// type: NotificationType.COUNTER_OFFER,   ← commented out
type: NotificationType.OFFER,              ← actually emitted
...
puzzleHash,                                ← attacker value, still present
```

**Step 2 — `canCounterOffer` evaluates to `true`** [2](#0-1) 

```ts
const canCounterOffer =
  [NotificationType.OFFER, NotificationType.COUNTER_OFFER].includes(type) &&
  'puzzleHash' in notification &&
  !!notification.puzzleHash;
```

Because the check explicitly includes `NotificationType.OFFER`, and the runtime object carries `puzzleHash` (TypeScript types are erased at runtime), `canCounterOffer` is `true` for the crafted notification.

**Step 3 — Attacker puzzle hash forwarded as counter-offer address** [3](#0-2) 

```ts
navigate('/dashboard/offers/view', {
  state: {
    ...
    canCounterOffer,
    address: 'puzzleHash' in notification ? notification.puzzleHash : undefined,
  },
});
```

The attacker's puzzle hash is passed as `address` to the offer builder, routing the victim's counter-offer to the attacker.

**Type system note**

The TypeScript type `NotificationOffer` does not include `puzzleHash`; only `NotificationCounterOffer` does. [4](#0-3) 

However, because the returned object is cast at runtime and TypeScript types are erased, the `puzzleHash` field is present in the live JavaScript object and the `'puzzleHash' in notification` check succeeds.

### Impact Explanation

When the victim clicks the notification and then the Counter button, the offer builder receives `address: attackerPuzzleHash`. The counter-offer is constructed with the attacker's address as the recipient, meaning any assets the victim commits to the counter-offer are directed to the attacker. This is a direct asset-loss path triggered by a single on-chain message the attacker can send to any wallet address.

### Likelihood Explanation

Sending an on-chain notification requires only a small XCH fee and knowledge of the victim's receive address (publicly derivable from any prior transaction). No local access, leaked keys, or social engineering beyond the notification itself is required. The victim only needs to click Counter on what appears to be a normal offer notification.

### Recommendation

1. Restore the commented-out line so that a payload with `ph` produces `type: NotificationType.COUNTER_OFFER`, not `OFFER`: [5](#0-4) 

2. Tighten `canCounterOffer` to require `NotificationType.COUNTER_OFFER` exclusively:
   ```ts
   const canCounterOffer =
     type === NotificationType.COUNTER_OFFER &&
     'puzzleHash' in notification &&
     !!notification.puzzleHash;
   ``` [2](#0-1) 

3. Validate that `puzzleHash` is a well-formed Chia puzzle hash before accepting it.
