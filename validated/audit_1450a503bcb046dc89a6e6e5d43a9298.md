### Title
Gateway `skip_stateful_validations` Hardcodes `Nonce(Felt::ONE)` and Ignores `max_nonce_for_validation_skip` Config — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The gateway's `skip_stateful_validations` function hardcodes the nonce threshold `Nonce(Felt::ONE)` for bypassing `__validate__` entry-point execution, while `StatefulTransactionValidatorConfig` exposes a `max_nonce_for_validation_skip` field that is serialised, documented, and deployed in `config_schema.json` — but is never read by the gateway path. The config field is dead code in the gateway. This creates two admission-correctness failures depending on how an operator sets the field.

### Finding Description

`StatefulTransactionValidatorConfig` declares:

```rust
pub max_nonce_for_validation_skip: Nonce,
```

with default `Nonce(Felt::ONE)` and the description *"Maximum nonce for which the validation is skipped."* [1](#0-0) 

The production `config_schema.json` exposes it as `"0x1"`: [2](#0-1) 

The gateway's `skip_stateful_validations` free function, however, never receives the config and hardcodes the threshold directly:

```rust
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
``` [3](#0-2) 

The caller `run_pre_validation_checks` holds `self.config` but does not forward `self.config.max_nonce_for_validation_skip` to the function: [4](#0-3) 

By contrast, the `PyValidator` path correctly uses the configurable field:

```rust
let nonce_small_enough_to_qualify_for_validation_skip =
    tx_nonce <= self.max_nonce_for_validation_skip;
``` [5](#0-4) 

### Impact Explanation

Two distinct admission failures arise from the mismatch:

**Case A — operator sets `max_nonce_for_validation_skip = 0` to disable the skip:**  
The gateway ignores the setting and still skips `__validate__` for any invoke transaction with `nonce == 1` whose sender has a deploy-account in the mempool. An attacker who controls such an address can submit an invoke transaction carrying an arbitrary (invalid) signature and have it admitted to the mempool without signature verification. The transaction will fail at execution time, but it is accepted by the gateway contrary to the operator's explicit configuration — matching the impact category *"Mempool/gateway admission accepts invalid transactions."*

**Case B — operator sets `max_nonce_for_validation_skip > 1` to extend the UX window:**  
The gateway still only skips for `nonce == 1`. Invoke transactions with `nonce ∈ [2, max_nonce_for_validation_skip]` that the operator intended to admit without validation are instead subjected to `__validate__` against a not-yet-deployed account, causing them to be rejected — matching *"rejects valid transactions before sequencing."*

### Likelihood Explanation

The default value of `max_nonce_for_validation_skip` is `Nonce(Felt::ONE)`, which coincidentally matches the hardcoded literal, so the bug is dormant with the shipped default. Exploitation requires an operator to change the config field (Case A: set to 0; Case B: set to > 1). Because the field is publicly documented and serialised, an operator who reads the description *"Maximum nonce for which the validation is skipped"* has a reasonable expectation that changing it will take effect. Likelihood is therefore **Low** in practice but the broken invariant is reachable without any privileged on-chain action.

### Recommendation

Pass `self.config.max_nonce_for_validation_skip` into `skip_stateful_validations` and replace the hardcoded literal:

```rust
// in run_pre_validation_checks
let skip_validate = skip_stateful_validations(
    executable_tx,
    account_nonce,
    mempool_client.clone(),
    self.config.max_nonce_for_validation_skip,   // ← add
).await?;

// in skip_stateful_validations signature
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
    max_nonce_for_validation_skip: Nonce,        // ← add
) -> ... {
    if let ExecutableTransaction::Invoke(...) = tx {
        if tx.nonce() <= max_nonce_for_validation_skip   // ← was == Nonce(Felt::ONE)
            && account_nonce == Nonce(Felt::ZERO)
        { ... }
    }
}
```

This aligns the gateway with the `PyValidator` logic and makes the config field effective.

### Proof of Concept

1. Deploy a sequencer node with `max_nonce_for_validation_skip = 0x0` in `gateway_config.static_config.stateful_tx_validator_config.max_nonce_for_validation_skip`.
2. Submit a `DeployAccount` transaction for address `A` to the gateway; confirm it enters the mempool (`account_tx_in_pool_or_recent_block(A) == true`).
3. Submit an `Invoke` transaction for address `A` with `nonce = 1` and a zeroed/garbage `signature`.
4. Observe: the gateway calls `skip_stateful_validations`, which evaluates `tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO)` → `true`, ignores `max_nonce_for_validation_skip = 0`, skips `__validate__`, and returns `Ok(())`.
5. The invoke transaction is admitted to the mempool despite carrying an invalid signature and despite the operator having set the skip threshold to zero.

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
