### Title
`skip_stateful_validations` Bypasses Invoke Signature Validation via Attacker-Controlled Deploy Account - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function is designed to skip `__validate__` for an invoke transaction when a deploy_account for the same address is pending in the mempool (UX feature). The guard it uses — `account_tx_in_pool_or_recent_block` — returns `true` for **any** transaction from the address, not specifically a deploy_account. An attacker who controls a class with a trivially-passing `__validate_deploy__` can place a deploy_account for an arbitrary address into the mempool, then submit an invoke transaction with nonce=1 and a forged/empty signature for that same address. The gateway skips `__validate__` and admits the invalid invoke transaction to the mempool.

### Finding Description

`skip_stateful_validations` triggers the bypass when three conditions hold simultaneously:

```
tx.nonce() == Nonce(Felt::ONE)
account_nonce == Nonce(Felt::ZERO)
account_tx_in_pool_or_recent_block(sender_address) == true
``` [1](#0-0) 

The code comment asserts: *"It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."* This reasoning is flawed. The check `account_tx_in_pool_or_recent_block` does not distinguish transaction types: [2](#0-1) 

When `skip_validate=true` is returned, `run_validate_entry_point` sets `execution_flags.validate = false`, and the blockifier's `StatefulValidator::perform_validations` returns `Ok(())` immediately after `perform_pre_validation_stage` without ever calling `__validate__`: [3](#0-2) [4](#0-3) 

`perform_pre_validation_stage` checks nonce, fee bounds, and proof facts — but **not** the signature: [5](#0-4) 

The mempool's `validate_tx` (called before `skip_stateful_validations`) also does not check the signature — it only checks for duplicate hashes and fee escalation: [6](#0-5) 

**Attack path:**

1. Attacker declares a class `C` whose `__validate_deploy__` always returns `VALIDATED` (attacker declares from their own funded account — a normal declare transaction).
2. Attacker picks salt `S` and constructor calldata `D`, computes `X = calculate_contract_address(C, S, D, 0)`.
3. Attacker funds `X` with enough STRK to satisfy `verify_can_pay_committed_bounds`.
4. Attacker submits `DeployAccount` for `X`. The gateway fully executes it (including `__validate_deploy__`), which passes trivially. `X` is now in the mempool.
5. Attacker submits `Invoke` with `nonce=1`, `sender_address=X`, and an **arbitrary/forged signature**.
6. Gateway evaluates: `nonce==1`, `account_nonce==0`, `account_tx_in_pool_or_recent_block(X)==true` → `skip_validate=true`.
7. `__validate__` is never called. The invalid invoke transaction is admitted to the mempool.

The nonce check for invoke allows `account_nonce(0) ≤ tx_nonce(1) ≤ max_allowed_nonce_gap`, so it passes: [7](#0-6) 

The existing test `should_skip_validation` confirms this exact path (nonce=1, account_nonce=0, `contains_tx=true`) produces `skip_validate=true`: [8](#0-7) 

### Impact Explanation

The gateway/mempool admission layer accepts an invoke transaction whose signature has never been verified. This matches: **High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.** The batcher will eventually reject the transaction during execution (because `__validate__` runs there with the account's real logic), but the invalid transaction occupies mempool capacity and wastes batcher resources. An attacker can repeat the pattern at scale — each iteration costs one deploy_account fee — to flood the mempool with signature-less invoke transactions, degrading throughput and causing effective DoS.

### Likelihood Explanation

The attacker needs only: (a) a declared class with trivially-passing `__validate_deploy__` (achievable by declaring a new class from any funded account), and (b) enough STRK to fund each target address for the deploy_account fee. No privileged access, no cryptographic break, and no dependency on external systems is required. The UX skip feature is enabled by default and the condition is easy to satisfy.

### Recommendation

Replace the type-agnostic `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a **deploy_account** transaction for the sender address is pending. The mempool should expose a dedicated query such as `deploy_account_in_pool(address) -> bool` that inspects the transaction type, so `skip_stateful_validations` can only fire when a genuine deploy_account is present — not when an attacker-controlled deploy_account for an arbitrary address happens to be in the pool.

### Proof of Concept

```
// Step 1 – attacker declares a trivial-validate class from their own account (normal declare tx).
// Step 2 – compute target address.
let class_hash = trivial_class_hash; // declared in step 1
let salt = ContractAddressSalt(Felt::from(42u64));
let ctor = calldata![];
let X = calculate_contract_address(salt, class_hash, &ctor, ContractAddress::default()).unwrap();

// Step 3 – fund X (transfer STRK to X from attacker's account).

// Step 4 – submit DeployAccount for X; gateway fully executes __validate_deploy__ → VALIDATED.
let deploy_tx = rpc_deploy_account_tx(deploy_account_tx_args!(
    class_hash: trivial_class_hash,
    contract_address_salt: salt,
    constructor_calldata: ctor,
    resource_bounds: valid_resource_bounds(),
    // signature: anything — __validate_deploy__ ignores it
));
gateway.add_tx(deploy_tx, None).await.unwrap();
// Now account_tx_in_pool_or_recent_block(X) == true.

// Step 5 – submit Invoke with nonce=1 and a forged signature.
let forged_invoke = rpc_invoke_tx(invoke_tx_args!(
    sender_address: X,
    nonce: nonce!(1),
    resource_bounds: valid_resource_bounds(),
    signature: TransactionSignature(vec![Felt::from(0xdeadbeefu64)].into()), // invalid
    calldata: calldata![felt!("0x1337")],
));
// Gateway: nonce=1, account_nonce=0, account_in_pool=true → skip_validate=true → __validate__ skipped.
let result = gateway.add_tx(forged_invoke, None).await;
assert!(result.is_ok()); // admitted without signature check
```

    pub fn perform_pre_validation_stage<S: State + StateReader>(
        &self,
        state: &mut S,
        tx_context: &TransactionContext,

        let tx_info = &tx_context.tx_info;
        Self::handle_nonce(state, tx_info, self.execution_flags.strict_nonce_check)?;
