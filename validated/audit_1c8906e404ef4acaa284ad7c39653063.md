### Title
Missing `IbcTimeout > 0` Validation Allows Governance to Permanently Halt All Outbound IBC Transfers — (`x/cronos/types/params.go`, `x/cronos/keeper/ibc.go`)

### Summary

`Params.Validate()` and `validateIsUint64` only check that `IbcTimeout` is of type `uint64`; they do not reject the value `0`. A governance proposal (`MsgUpdateParams`) with `IbcTimeout: 0` passes all validation and is stored. Every subsequent call to `ibcSendTransfer` then constructs a `MsgTransfer` with `TimeoutTimestamp = uint64(ctx.BlockTime().UnixNano()) + 0`, which equals the current block time — an already-expired timestamp that the IBC channel keeper rejects. This permanently halts all outbound IBC transfers from Cronos, including the `SendCroToIbc` EVM hook path, with no on-chain recovery path short of a second governance proposal.

### Finding Description

**Root cause — `validateIsUint64` is a type-only check:** [1](#0-0) 

The function returns `nil` for any `uint64`, including `0`. `Params.Validate()` delegates to this function for `IbcTimeout`: [2](#0-1) 

`MsgUpdateParams.ValidateBasic()` calls `msg.Params.Validate()`, so the proposal passes mempool and ante-handler checks: [3](#0-2) 

The governance handler calls `SetParams` with the validated (but semantically broken) params: [4](#0-3) 

**Trigger — every IBC send uses `params.IbcTimeout` directly:** [5](#0-4) 

With `IbcTimeout = 0`, `timeoutTimestamp = uint64(ctx.BlockTime().UnixNano())`. The ibc-go channel keeper rejects any packet whose `TimeoutTimestamp ≤ ctx.BlockTime().UnixNano()` at send time, so every call to `ibcSendTransfer` returns an error.

**All outbound IBC paths are affected:**
- `MsgTransferTokens` → `IbcTransferCoins` → `ibcSendTransfer` [6](#0-5) 
- EVM `SendToIbcHandler` → `IbcTransferCoins` → `ibcSendTransfer` [7](#0-6) 

**The integration test configs confirm `ibc_timeout: 0` is a reachable genesis/runtime state:** [8](#0-7) 

### Impact Explanation

After the proposal executes, every outbound IBC transfer attempt — whether via `MsgTransferTokens` or the EVM `SendCroToIbc` hook — fails with a timeout-in-past error. Users holding IBC vouchers or EVM-denom tokens cannot bridge them back. The state persists until a second governance proposal restores a non-zero `IbcTimeout`. This is a permanent, long-lived inability for honest users to process IBC transfers, matching the High impact criterion.

### Likelihood Explanation

The governance path is explicitly in scope per the audit rules. The proposal requires passing a governance vote, which is a meaningful barrier. However, the missing validation is a clear code defect: the comment in `ibcSendTransfer` states "it means it can never fail by timeout," which is only true for `IbcTimeout > 0`. The integration test configs already exercise `ibc_timeout: 0`, demonstrating the state is reachable and not guarded against.

### Recommendation

Add a positive lower-bound check in `Params.Validate()` and in `validateIsUint64` (or a dedicated `validateIbcTimeout` helper):

```go
// In Params.Validate(), replace the IbcTimeout check with:
if p.IbcTimeout == 0 {
    return fmt.Errorf("ibc_timeout must be greater than zero")
}
```

This ensures `MsgUpdateParams.ValidateBasic()` rejects the proposal before it reaches `SetParams`.

### Proof of Concept

1. Submit a governance proposal: `MsgUpdateParams{Authority: govModuleAddr, Params: Params{IbcTimeout: 0, IbcCroDenom: <valid>, ...}}`
2. Proposal passes `ValidateBasic` (type check only) and governance vote.
3. `SetParams` stores `IbcTimeout = 0`.
4. Call `MsgTransferTokens` with any IBC voucher coin.
5. `ibcSendTransfer` computes `timeoutTimestamp = blockTime + 0 = blockTime`.
6. `transferKeeper.Transfer` returns error: timeout timestamp is not in the future.
7. All subsequent `MsgTransferTokens` and EVM `SendToIbc` hook calls fail identically.

Unit test plan: in `keeper_test`, call `SetParams` with `IbcTimeout: 0`, then call `IbcTransferCoins` and assert the IBC transfer error is returned; separately assert that `Params{IbcTimeout: 0}.Validate()` returns a non-nil error (currently it does not).
