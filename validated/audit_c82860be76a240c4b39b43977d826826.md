### Title
Pool Admin Can Set Uncapped Per-Bin Additional Fees, Bypassing the Fee Cap System and Causing Swappers to Overpay - (File: metric-core/contracts/MetricOmmPoolFactory.sol)

### Summary

`setPoolBinAdditionalFees` imposes no upper-bound validation on `addFeeBuyE6` / `addFeeSellE6`, while every other admin fee path (`setPoolAdminFees`, `setPoolProtocolFee`) is gated by `maxAdminSpreadFeeE6` / `maxAdminNotionalFeeE8`. A pool admin can therefore set per-bin additional fees to `type(uint16).max` (65 535 E6 ≈ 6.55 %) on any bin, silently increasing the cost of every swap that crosses that bin beyond what the documented fee-cap system permits.

### Finding Description

The factory enforces explicit caps on the base admin spread and notional fees:

```solidity
// MetricOmmPoolFactory.sol – setPoolAdminFees
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (newAdminNotionalFeeE8 > maxAdminNotionalFeeE8) revert AdminFeeTooHigh();
``` [1](#0-0) 

The per-bin path has no equivalent guard:

```solidity
// MetricOmmPoolFactory.sol – setPoolBinAdditionalFees
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
``` [2](#0-1) 

The pool's `setBinAdditionalFees` only validates the bin index, not the fee magnitudes:

```solidity
// MetricOmmPool.sol – setBinAdditionalFees
if (bin < LOWEST_BIN || bin > HIGHEST_BIN) revert InvalidBinIndex(bin);
BinState storage s = _binStates[bin];
s.addFeeBuyE6 = addFeeBuyE6;
s.addFeeSellE6 = addFeeSellE6;
``` [3](#0-2) 

During every swap the per-bin additional fee is added directly on top of the base spread fee:

```solidity
params.baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6)
``` [4](#0-3) 

A pool admin can therefore call `setPoolBinAdditionalFees` with `addFeeBuyE6 = type(uint16).max` (65 535) immediately before a victim's swap lands, adding ≈ 6.55 % extra cost to that swap on top of whatever the base spread already is — with no factory-level cap to prevent it.

### Impact Explanation

- **Direct loss of user funds**: the swapper pays more input tokens (exact-output path) or receives fewer output tokens (exact-input path) than the fee-cap system is supposed to allow.
- The hard owner cap `HARD_MAX_SPREAD_FEE_E6 = 200 000` (20 %) bounds the base spread, but per-bin fees are additive and uncapped, so the effective total fee can silently exceed the documented maximum. [5](#0-4) 
- The router's `amountOutMinimum` / `amountInMaximum` slippage guards only protect users who set tight bounds; users relying on the fee-cap system for safety receive no protection. [6](#0-5) 

### Likelihood Explanation

- Trigger requires only the pool admin role — a semi-trusted actor the protocol explicitly constrains via caps.
- The attack is a single transaction (`setPoolBinAdditionalFees`) that can be timed to front-run any pending swap.
- No timelock, no two-step confirmation, and no on-chain cap check stands in the way. [2](#0-1) 

### Recommendation

Add a cap check in `setPoolBinAdditionalFees` analogous to the one in `setPoolAdminFees`. A reasonable bound is `maxAdminSpreadFeeE6` (already stored in factory state), or a dedicated `maxBinAdditionalFeeE6` constant:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
+   if (addFeeBuyE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
+   if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

### Proof of Concept

```
1. Factory is deployed with maxAdminSpreadFeeE6 = 50_000 (5 %).
2. Pool is created; pool admin is attacker.
3. Victim submits exactInputSingle with amountOutMinimum = 0 (or any loose bound).
4. Attacker front-runs: calls setPoolBinAdditionalFees(pool, activeBin, 65535, 65535).
   → No revert; 65535 > 50_000 but no check exists.
5. Victim's swap executes; SwapMath applies baseFeeX64 + mulDiv(65535, ONE_X64, 1e6)
   → effective additional fee ≈ 6.55 % on top of base spread.
6. Victim receives far fewer output tokens than expected; excess is captured as LP fee
   credited to the bin, benefiting the attacker's LP position or simply lost to the victim.
7. Attacker resets: setPoolBinAdditionalFees(pool, activeBin, 0, 0) to avoid detection.
``` [7](#0-6) [2](#0-1)
