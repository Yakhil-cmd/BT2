### Title
Unsigned Invoke Transaction Admitted for Undeployed Account via Overly Broad `skip_stateful_validations` Check — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

`skip_stateful_validations` skips the `__validate__` entry point (signature verification) for an invoke with `nonce=1` from an undeployed account whenever `account_tx_in_pool_or_recent_block(sender_address)` returns `true`. That helper returns `true` for **any** transaction from the address in the mempool — not specifically a `DeployAccount`. An attacker who observes a victim's pending `DeployAccount` can immediately submit an unsigned invoke with the victim's address as sender, bypass signature verification at admission, and have the transaction accepted into the mempool.

---

### Finding Description

The gateway stateful validation path in `extract_state_nonce_and_run_validations` calls `run_pre_validation_checks`, which calls `skip_stateful_validations`: [1](#0-0) 

The skip condition fires when:
1. Transaction type is `Invoke`
2. `tx.nonce() == Nonce(Felt::ONE)`
3. `account_nonce == Nonce(Felt::ZERO)` (account not yet deployed on-chain)
4. `account_tx_in_pool_or_recent_block(sender_address)` returns `true`

When the skip fires, `run_validate_entry_point` is called with `skip_validate = true`, which sets `validate = false` in `ExecutionFlags`, causing `__validate__` to be entirely bypassed: [2](#0-1) 

The `account_tx_in_pool_or_recent_block` implementation checks for **any** transaction from the address, regardless of type: [3](#0-2) 

It delegates to `MempoolState.contains_account` (checks `staged` or `committed` maps) and `TransactionPool.contains_account` (checks any pooled tx by address): [4](#0-3) [5](#0-4) 

Neither the stateless validator nor the mempool's `validate_tx` checks the signature: [6](#0-5) [7](#0-6) 

The code comment acknowledges the weaker-than-necessary check: *"It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."* This reasoning is circular for the `nonce=1` case and does not hold when the attacker is the one placing the nonce-1 invoke.

---

### Impact Explanation

An attacker can submit an invoke transaction with:
- `sender_address = victim_address` (any undeployed account with a pending `DeployAccount`)
- `nonce = 1`
- Arbitrary `calldata`
- Empty or garbage `signature`

The transaction is admitted to the mempool without signature verification. When the batcher later executes it (after the `DeployAccount` deploys the account), `__validate__` is called with the invalid signature and the transaction reverts — but in Starknet v3, reverted transactions still pay fees charged to the sender (the victim). The attacker can repeat this to drain the victim's fee balance. Additionally, the mempool slot is consumed, potentially displacing legitimate transactions.

**Impact category**: High — Mempool/gateway admission accepts an invalid (unsigned) transaction before sequencing.

---

### Likelihood Explanation

The attack window is the period between a victim broadcasting a `DeployAccount` and it being committed on-chain. Any observer of the public mempool or gateway can trigger this. No special privileges are required; the attacker only needs the victim's address, which is revealed by the `DeployAccount` transaction itself.

---

### Recommendation

Replace the type-agnostic `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a `DeployAccount` transaction exists for the sender address. The mempool should expose a method such as `has_pending_deploy_account(address: ContractAddress) -> bool` that inspects the transaction type stored in the pool, and `skip_stateful_validations` should call that instead:

```rust
// Instead of:
mempool_client.account_tx_in_pool_or_recent_block(tx.sender_address())

// Use:
mempool_client.has_pending_deploy_account(tx.sender_address())
```

This mirrors the mitigation suggested in the original flashloan report: verify that the operation was initiated by the trusted contract (here: that the account's first transaction is specifically a `DeployAccount`), not merely that some transaction from the address exists.

---

### Proof of Concept

```
1. Victim broadcasts: DeployAccount { sender: V, nonce: 0, signature: <valid> }
   → Admitted to mempool; account_tx_in_pool_or_recent_block(V) now returns true.

2. Attacker broadcasts: Invoke { sender_address: V, nonce: 1, calldata: [drain_funds], signature: [] }

3. Gateway stateless validation: passes (no signature check).

4. extract_state_nonce_and_run_validations:
   - account_nonce = get_nonce_from_state(V) = 0  (V not deployed on-chain)
   - validate_nonce: 1 ∈ [0, 0+max_gap] → OK
   - validate_by_mempool: no duplicate tx_hash → OK
   - skip_stateful_validations:
       nonce == 1 ✓
       account_nonce == 0 ✓
       account_tx_in_pool_or_recent_block(V) = true ✓  ← victim's DeployAccount triggers this
       → returns true (skip validation)

5. run_validate_entry_point(skip_validate=true):
   ExecutionFlags { validate: false, ... }
   __validate__ is NOT called → signature never checked.

6. Attacker's unsigned invoke is admitted to the mempool.

7. Batcher executes block:
   - DeployAccount(V, nonce=0) executes → V is deployed.
   - Invoke(V, nonce=1) executes → __validate__ called → invalid signature → REVERT.
   - Fee is charged to V for the reverted transaction.

8. Attacker repeats steps 1–7 for every new DeployAccount they observe.
```

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
