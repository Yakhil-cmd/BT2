### Title
`calculateNFTRoyalties` Does Not Deduct Royalty from `nftSellerNetAmount`, Causing Offer Confirmation to Display Inflated "Net Proceeds" and Suppressing Negative-Amount Guard - (File: packages/gui/src/components/offers/utils.ts)

### Summary
`calculateNFTRoyalties` computes a `royaltyAmount` but never subtracts it from `nftSellerNetAmount`, which is left equal to the full pre-royalty `amount`. This incorrect value propagates into both the NFT Offer Editor ("You will receive") and the NFT Offer Viewer ("Net Proceeds"), causing the GUI to display the full token amount as what the NFT seller will receive, when the protocol will actually deduct the royalty. A secondary consequence is that the negative-amount guard in the editor (`showNegativeAmountWarning`) can never trigger, because `nftSellerNetAmount` is always positive.

### Finding Description
In `packages/gui/src/components/offers/utils.ts`, `calculateNFTRoyalties` computes:

```ts
const royaltyAmount: number = royaltyPercentage ? (royaltyPercentage / 100) * amount : 0;
const royaltyAmountString: string = formatAmount(royaltyAmount);
const nftSellerNetAmount: number = amount;   // ← BUG: should be amount - royaltyAmount
// : parseFloat(
//     (amount - parseFloat(royaltyAmountString) - makerFee).toFixed(12),
//   );
```

The commented-out block shows the correct deduction was intentionally designed but then disabled. [1](#0-0) 

`nftSellerNetAmount` is returned and consumed in two places:

**1. NFT Offer Editor (`NFTOfferEditor.tsx` lines 271, 364–372)**

```ts
const showNegativeAmountWarning = (nftSellerNetAmount ?? 0) < 0;
```

Because `nftSellerNetAmount === amount` (always positive), this guard never fires. The "You will receive" label renders `nftSellerNetAmount` — the full pre-royalty amount — as what the seller will pocket. [2](#0-1) [3](#0-2) 

**2. NFT Offer Viewer (`NFTOfferViewer.tsx` lines 397–402, 595–597)**

For `TokenForNFT` offers, `overrideNFTSellerAmount` is derived from `nftSellerNetAmount` and passed into `NFTOfferSummary` → `NFTOfferSummaryRow` → `OfferSummaryTokenRow`, where it replaces the real on-chain amount in the "You will receive" row: [4](#0-3) 

```ts
const amount = overrideNFTSellerAmount ?? originalAmount;
``` [5](#0-4) 

The "Net Proceeds" section also renders `nftSaleInfo?.nftSellerNetAmount` directly: [6](#0-5) 

### Impact Explanation
An NFT seller creating or reviewing an offer sees "You will receive X tokens" / "Net Proceeds: X tokens", where X is the full asking price. The Chia protocol enforces royalties at the coin-spend level, so the seller actually receives `X − royaltyAmount`. The GUI systematically overstates what the seller will net. A seller who sets an asking price of 10 XCH on an NFT with a 10% royalty will see "You will receive: 10 XCH" but will actually receive 9 XCH. This constitutes display of the wrong amount in the offer confirmation/creation flow, directly matching the High impact criterion: *"spoofing… of offer… state that causes a user to… display the wrong… amount."* The seller may repeatedly create under-priced offers, losing royalty-equivalent value on every sale.

### Likelihood Explanation
Any NFT with a non-zero `royaltyPercentage` triggers the bug. No attacker is required; the miscalculation is self-inflicted by the GUI on every NFT offer creation or viewing session. The affected code paths (`NFTOfferEditor`, `NFTOfferViewer`) are the primary UI surfaces for NFT trading.

### Recommendation
Restore the deduction inside `calculateNFTRoyalties`:

```diff
- const nftSellerNetAmount: number = amount;
- // : parseFloat(
- //     (amount - parseFloat(royaltyAmountString) - makerFee).toFixed(12),
- //   );
+ const nftSellerNetAmount: number = parseFloat(
+   (amount - parseFloat(royaltyAmountString) - makerFee).toFixed(12),
+ );
```

This makes `nftSellerNetAmount` reflect the actual post-royalty proceeds, restores the `showNegativeAmountWarning` guard in `NFTOfferEditor`, and corrects the "Net Proceeds" / "You will receive" displays in `NFTOfferViewer`.

### Proof of Concept
1. Open the Chia GUI and navigate to the NFT Offer Editor.
2. Select "Sell an NFT" for an NFT with `royaltyPercentage = 1000` (10%).
3. Enter a token amount of 10 XCH.
4. Observe: "Creator Fee: 1 XCH", but "You will receive: **10 XCH**" (should be 9 XCH).
5. Create the offer and accept it from another wallet.
6. Observe on-chain: the seller receives 9 XCH; the royalty address receives 1 XCH.
7. The GUI never showed the seller the correct net amount, and the negative-amount guard at `showNegativeAmountWarning` never triggered even when `makerFee + royaltyAmount > amount`.
