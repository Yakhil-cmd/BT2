### Title
`validate_resource_bounds()` skips `max_l2_gas_amount` cap for Declare transactions, enabling gateway admission bypass — (`File: crates/apollo_gateway/src/stateless_transaction_validator.rs`)

### Summary

`StatelessTransactionValidator::validate_resource_bounds()` enforces a `max_l2_gas_amount` upper bound on `l2_gas.max_amount` for Invoke and DeployAccount transactions, but explicitly exempts Declare transactions from this check. Any funded account can submit a Declare transaction with an arbitrarily large `l2_gas.max_amount`, bypassing the gateway's stateless admission control. No downstream compensating control closes this gap for values between the configured limit (1,210,000,000) and the block's sierra gas ceiling (5,000,000,000).

### Finding Description

In `validate_resource_bounds()`, the `max_l2_gas_amount` check is guarded by a type branch that silently skips Declare transactions:

```rust
// TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
if let RpcTransaction::Declare(_) = tx {
} else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
        gas_amount: resource_bounds.l2_gas.max_amount,
        max_gas_amount: self.config.max_l2_gas_amount,
    });
}
``` [1](#0-0) 

The production deployment sets `max_l2_gas_amount = 1,210,000,000`: [2](#0-1) 

For Invoke and DeployAccount, any `l2_gas.max_amount > 1,210,000,000` is rejected at the stateless stage. For Declare, no such check exists. The test `valid_l2_gas_amount_on_declare` explicitly confirms this as the current behavior — a Declare with `max_amount = 200` passes when `max_l2_gas_amount = 100`: [3](#0-2) 

**Downstream controls do not compensate:**

1. The stateful validator's `validate_resource_bounds()` only checks the L2 gas *price* against the previous block's price — it never checks the L2 gas *amount*: [4](#0-3) 

2. `verify_can_pay_committed_bounds()` uses `max_possible_fee()`, which applies **saturating arithmetic**: [5](#0-4) 

For a Declare with `l2_gas.max_amount = 1,210,000,001` and `max_price_per_unit = 8,000,000,000` (the production minimum), the required balance is `≈ 9.68 × 10¹⁸` STRK — large but achievable for a whale account. For values between the gateway limit (1.21B) and the block's `sierra_gas` ceiling (5B), a sufficiently funded sender passes all gateway checks and is admitted to the mempool. [6](#0-5) 

3. `check_fee_bounds()` in `perform_pre_validation_stage()` only checks that `max_amount >= minimal_gas_amount` (a lower bound), not an upper bound: [7](#0-6) 

4. The bouncer uses *actual* gas consumed, not declared `max_amount`, so it provides no admission-stage guard: [8](#0-7) 

### Impact Explanation

A Declare transaction with `l2_gas.max_amount` between `max_l2_gas_amount + 1` (1,210,000,001) and the block's `sierra_gas` limit (5,000,000,000) is admitted by the gateway when it should be rejected. This is a direct gateway admission bypass matching the impact category: **High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

The asymmetry means Declare transactions can claim up to ~4× more L2 gas per transaction than Invoke or DeployAccount, potentially monopolizing block gas capacity and displacing legitimate transactions.

### Likelihood Explanation

Medium. The attacker needs a funded account with balance ≥ `max_amount × min_gas_price`. For values just above the limit (e.g., `max_amount = 1,210,000,001`), the required balance is ~9.68 × 10¹⁸ STRK. The bypass requires no special privileges — any user can submit a Declare transaction. The TODO comment in the source confirms the gap is known but unresolved.

### Recommendation

Remove the Declare-type exemption in `validate_resource_bounds()` and apply the same `max_l2_gas_amount` check uniformly to all transaction types:

```rust
if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
        gas_amount: resource_bounds.l2_gas.max_amount,
        max_gas_amount: self.config.max_l2_gas_amount,
    });
}
```

If Declare transactions legitimately require a higher L2 gas ceiling (e.g., due to compilation overhead), introduce a separate `max_l2_gas_amount_declare` config field rather than skipping the check entirely.

### Proof of Concept

1. Construct a `RpcDeclareTransactionV3` with:
   - `resource_bounds.l2_gas.max_amount = GasAmount(1_210_000_001)` (one above the production limit)
   - `resource_bounds.l2_gas.max_price_per_unit = GasPrice(8_000_000_000)` (production minimum)
   - Valid Sierra class, correct nonce, valid signature
2. Fund the sender account with ≥ `1_210_000_001 × 8_000_000_000 = 9_680_000_008_000_000_000` STRK
3. Submit to the gateway's `add_tx` endpoint
4. **Stateless validation passes**: the `if let RpcTransaction::Declare(_) = tx {}` branch skips the `max_l2_gas_amount` check
5. **Stateful validation passes**: `validate_resource_bounds` only checks price; `verify_can_pay_committed_bounds` passes because balance ≥ `max_possible_fee`
6. Transaction is admitted to the mempool with `l2_gas.max_amount` 4× above the limit enforced for all other transaction types

The test `test_invalid_max_l2_gas_amount` confirms the check fires for `TransactionType::DeployAccount` and `TransactionType::Invoke` but is absent for `TransactionType::Declare`: [9](#0-8)

pub fn verify_can_pay_committed_bounds(
    state: &mut dyn StateReader,
    tx_context: &TransactionContext,
) -> TransactionFeeResult<()> {
    let tx_info = &tx_context.tx_info;
    let committed_fee = tx_context.max_possible_fee();
    let (balance_low, balance_high, can_pay) =
        get_balance_and_if_covers_fee(state, tx_context, committed_fee)?;
    if can_pay {
        Ok(())
    } else {
        Err(match tx_info {
            TransactionInfo::Current(context) => match &context.resource_bounds {
                L1Gas(l1_gas) => TransactionFeeError::GasBoundsExceedBalance {
                    resource: Resource::L1Gas,
                    max_amount: l1_gas.max_amount,
                    max_price: l1_gas.max_price_per_unit,
                    balance: balance_to_big_uint(&balance_low, &balance_high),
                },
                AllResources(bounds) => TransactionFeeError::ResourcesBoundsExceedBalance {
                    bounds: *bounds,
                    balance: balance_to_big_uint(&balance_low, &balance_high),
                },
            },
            TransactionInfo::Deprecated(context) => TransactionFeeError::MaxFeeExceedsBalance {
                max_fee: context.max_fee,
                balance: balance_to_big_uint(&balance_low, &balance_high),
            },
        })
    }
}

    pub fn try_update<S: StateReader>(
        &mut self,
        state_reader: &S,
        tx_state_changes_keys: &StateChangesKeys,
        tx_execution_summary: &ExecutionSummary,
        tx_builtin_counters: &CairoPrimitiveCounterMap,
        tx_resources: &TransactionResources,
        versioned_constants: &VersionedConstants,
        receipt_l2_gas: GasAmount,
    ) -> TransactionExecutorResult<()> {
        // The countings here should be linear in the transactional state changes and execution info
        // rather than the cumulative state attributes.
        let marginal_state_changes_keys =
            tx_state_changes_keys.difference(&self.state_changes_keys);
        let marginal_executed_class_hashes = tx_execution_summary
            .executed_class_hashes
            .difference(&self.get_executed_class_hashes())
            .cloned()
            .collect();
        let n_marginal_visited_storage_entries = tx_execution_summary
            .visited_storage_entries
            .difference(&self.visited_storage_entries)
            .count();
        let tx_weights = get_tx_weights(
            state_reader,
            &marginal_executed_class_hashes,
            n_marginal_visited_storage_entries,
            tx_resources,
            &marginal_state_changes_keys,
            versioned_constants,
            tx_builtin_counters,
            &self.bouncer_config,
            receipt_l2_gas,
        )?;

        let tx_bouncer_weights = tx_weights.bouncer_weights;

        // Check if the transaction can fit the current block available capacity.
        let err_msg = format!(
            "Addition overflow. Transaction weights: {tx_bouncer_weights:?}, block weights: {:?}.",
            self.get_bouncer_weights()
        );
        let next_accumulated_weights =
            self.get_bouncer_weights().checked_add(tx_bouncer_weights).expect(&err_msg);
        if !self.bouncer_config.has_room(next_accumulated_weights) {
            let exceeded_weights =
                self.bouncer_config.get_exceeded_weights(next_accumulated_weights);
            log::debug!(
                "Transaction cannot be added to the current block, block capacity reached; \
                 transaction weights: {:?}, block weights: {:?}. Block max capacity reached on \
                 fields: {}",
                tx_weights.bouncer_weights,
                self.get_bouncer_weights(),
                exceeded_weights
            );
            record_exceeded_bouncer_resources(&exceeded_weights);
            Err(TransactionExecutorError::BlockFull)?
        }

        self.update(tx_weights, tx_execution_summary, &marginal_state_changes_keys);

        Ok(())
