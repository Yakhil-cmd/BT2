### Title
`RandomSeed` Precompile Exposes Zero-Cost, Fully Predictable Per-Transaction Entropy, Enabling Selective Outcome Manipulation on Any Aurora Contract Using Randomness — (`engine-precompiles/src/random.rs`)

---

### Summary

The Aurora Engine exposes a custom `RandomSeed` precompile at address `0xc104f4840573bed437190daf5d2898c2bdf928ac` that returns the NEAR-derived per-transaction random seed. This precompile charges **zero EVM gas** (explicitly marked `TODO`), and the seed it returns is **fully deterministic and computable off-chain before the transaction is submitted**. Any EVM contract on Aurora that uses this precompile — or the `PREVRANDAO` opcode, which returns the same value via `block_randomness()` — for randomness-based fee-bearing or outcome-determining logic is vulnerable to the same attack class as the NFTX report: an attacker can always obtain the favorable outcome at zero marginal cost.

---

### Finding Description

**Root cause 1 — Zero gas cost:** [1](#0-0) 

`RANDOM_BYTES_GAS` is set to `EthGas::new(0)`. The `TODO(#483)` comment acknowledges this is unresolved. Calling the precompile from any EVM contract costs nothing, making it free to call in a loop or as a conditional check before reverting.

**Root cause 2 — Fully deterministic, pre-computable seed:** [2](#0-1) 

The random seed is `sha256(action_hash || block_random_value)`. The `action_hash` is derived from the receipt ID, block hash, and action index: [3](#0-2) 

All inputs are public and known before the transaction executes. An attacker can compute the exact seed they will receive for any future transaction they plan to submit.

**Root cause 3 — Same seed exposed via `PREVRANDAO` opcode:** [4](#0-3) 

`block_randomness()` returns `Some(self.env.random_seed())`, meaning the standard EVM `PREVRANDAO` opcode also returns this same predictable value. Contracts using `PREVRANDAO` instead of the custom precompile are equally vulnerable.

**Attack flow:**

1. Attacker observes the current block's `block_random_value` (public on NEAR).
2. Attacker computes their `action_hash` from their known receipt ID, block hash, and action index.
3. Attacker computes `sha256(action_hash || block_random_value)` — the exact seed their transaction will receive.
4. If the outcome is unfavorable, the attacker simply does not submit. No gas is spent.
5. If the attacker wants to verify on-chain (e.g., inside a contract that checks the seed and reverts), they call the `RandomSeed` precompile for **0 gas** and revert — also at zero EVM gas cost for the precompile call itself.
6. The attacker repeats across blocks until the desired outcome is obtained.

This is strictly stronger than the NFTX attack: no Flashbots-equivalent is needed because the outcome is computable entirely off-chain before submission.

---

### Impact Explanation

Any EVM contract deployed on Aurora that uses `RandomSeed` or `PREVRANDAO` for:
- Randomized NFT/token selection with differential fees (direct analog to NFTX)
- Lottery or prize distribution
- Random reward allocation

…is fully gameable. An attacker can always obtain the favorable outcome, draining funds from other participants or bypassing premium fees. This constitutes **direct theft of user funds** held in or flowing through such contracts.

**Impact: Critical** — Direct theft of any user funds at-rest or in-motion in contracts relying on Aurora's randomness primitives.

---

### Likelihood Explanation

**High.** The attack requires:
- Knowledge of the NEAR block random value (public)
- Knowledge of one's own receipt ID / action hash (self-controlled)
- A single `sha256` computation off-chain

No privileged access, no special infrastructure, and no gas cost for the precompile call are required. The `RANDOM_BYTES_GAS = 0` constant means even on-chain conditional reversion is free at the precompile level. Any developer or user interacting with a randomness-dependent Aurora contract can execute this.

---

### Recommendation

1. **Assign a non-zero gas cost** to the `RandomSeed` precompile. Resolve `TODO(#483)`. A cost comparable to `SHA256` (60 gas + 12/word) is a reasonable baseline. [1](#0-0) 

2. **Document clearly** that `RandomSeed` and `PREVRANDAO` on Aurora are **not suitable for use as a source of unpredictable randomness** in any contract where the outcome affects fund distribution or fee selection. The seed is deterministic and pre-computable by the transaction sender.

3. **Consider a commit-reveal pattern** at the application layer (as recommended in the NFTX report) for any Aurora contract requiring unpredictable randomness. Alternatively, integrate a VRF (Verifiable Random Function) oracle.

    pub(super) const RANDOM_BYTES_GAS: EthGas = EthGas::new(0);

pub struct RandomSeed {
    random_seed: H256,
}

impl RandomSeed {
    /// Random bytes precompile address
    /// This is a per-block entropy source which could then be used to create a random sequence.
    /// It will return the same seed if called multiple time in the same block.
    ///
    /// Address: `0xc104f4840573bed437190daf5d2898c2bdf928ac`
    /// This address is computed as: `&keccak("randomSeed")[12..]`
    pub const ADDRESS: Address = make_address(0xc104f484, 0x0573bed437190daf5d2898c2bdf928ac);

    #[must_use]
    pub const fn new(random_seed: H256) -> Self {
        Self { random_seed }
    }
}

impl Precompile for RandomSeed {
    fn required_gas(_input: &[u8]) -> Result<EthGas, ExitError> {
        Ok(costs::RANDOM_BYTES_GAS)
    }

    fn run(
        &self,
        input: &[u8],
        target_gas: Option<EthGas>,
        context: &Context,
        _is_static: bool,
    ) -> EvmPrecompileResult {
        utils::validate_no_value_attached_to_precompile(context.apparent_value)?;
        let cost = Self::required_gas(input)?;
        if let Some(target_gas) = target_gas
            && cost > target_gas
        {
            return Err(ExitError::OutOfGas);
        }

        Ok(PrecompileOutput::without_logs(
            cost,
            self.random_seed.as_bytes().to_vec(),
        ))
    }
