### Title
Stale Connector Reference After `set_eth_connector_contract_account` Permanently Freezes All User NEP-141 ETH Balances — (`engine/src/contract_methods/connector.rs`)

---

### Summary

The Aurora Engine stores the address of the external ETH connector contract in mutable NEAR storage under the key `EthConnectorStorageId::EthConnectorAccount`. Every user-facing NEP-141 operation (`withdraw`, `ft_transfer`, `ft_transfer_call`, `storage_withdraw`, `storage_unregister`, `ft_balance_of`, `storage_balance_of`) is forwarded to whichever account ID is currently stored in that key via the shared `return_promise` helper. The owner (or a private/self-call) can atomically overwrite this stored account ID at any time via `set_eth_connector_contract_account`. After the update, all engine-level forwarding targets the new connector contract, while every user's NEP-141 ETH balance remains recorded in the old connector contract's storage — permanently inaccessible through the engine's standard interface.

---

### Finding Description

The `return_promise` function is the single dispatch point for all connector-proxied calls:

```rust
// engine/src/contract_methods/connector.rs  lines 598-616
fn return_promise<I: IO + PromiseHandler, E: Env>(
    mut io: I,
    env: &E,
    method: &str,
    args: Vec<u8>,
    deposit: Yocto,
) -> Result<(), ContractError> {
    let promise_args = PromiseCreateArgs {
        target_account_id: get_connector_account_id(&io)?,   // ← always reads current stored ID
        ...
    };
    ...
}
```

`get_connector_account_id` reads the value stored at `EthConnectorStorageId::EthConnectorAccount` at the moment of each call. [1](#0-0) 

`set_eth_connector_contract_account` overwrites that stored value unconditionally:

```rust
// engine/src/contract_methods/connector.rs  lines 418-438
pub fn set_eth_connector_contract_account<I: IO + Copy, E: Env>(io: I, env: &E) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        ...
        set_connector_account_id(io, &args.account);          // ← overwrites stored ID
        set_connector_withdraw_serialization_type(io, &args.withdraw_serialize_type);
        Ok(())
    })
}
``` [2](#0-1) 

`set_connector_account_id` writes the new account ID directly to NEAR storage with no migration, no snapshot of the old ID, and no mechanism to drain balances from the old connector first:

```rust
pub fn set_connector_account_id<I: IO + Copy>(mut io: I, account_id: &AccountId) {
    io.write_borsh(
        &construct_contract_key(EthConnectorStorageId::EthConnectorAccount),
        account_id,
    );
}
``` [3](#0-2) 

Every user-facing function that moves or queries NEP-141 ETH balances goes through `return_promise`:

| Engine entrypoint | Forwarded method |
|---|---|
| `withdraw` | `engine_withdraw` |
| `ft_transfer` | `engine_ft_transfer` |
| `ft_transfer_call` | `engine_ft_transfer_call` |
| `storage_withdraw` | `engine_storage_withdraw` |
| `storage_unregister` | `engine_storage_unregister` |
| `ft_balance_of` | `ft_balance_of` |
| `storage_balance_of` | `storage_balance_of` | [4](#0-3) [5](#0-4) [6](#0-5) 

After the connector account ID is updated, all of these calls are dispatched to the new connector contract. The old connector contract still holds every user's NEP-141 ETH balance in its own NEAR storage, but the engine no longer routes any call to it. There is no engine-level escape hatch that lets a user call the old connector directly to recover their funds.

A secondary breakage occurs in `ft_on_transfer`: it uses `get_connector_account_id` to decide whether an incoming `ft_transfer_call` is a base-token deposit from the connector or an ERC-20 deposit. After the update, a transfer from the old connector is no longer recognized as a base-token deposit and is misrouted as an ERC-20 transfer, causing the deposit to fail or be misaccounted. [7](#0-6) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

All NEP-141 ETH balances held in the old connector contract become permanently inaccessible through the Aurora Engine's standard interface the moment `set_eth_connector_contract_account` is executed. Users cannot `withdraw`, `ft_transfer`, or otherwise move their bridged ETH. The old connector contract's `engine_withdraw` and related methods are gated to be called only by the engine contract (`predecessor == engine`), so users cannot bypass the engine and call the old connector directly either. The freeze is irreversible without a further owner action to repoint the connector back, and even then any state changes made in the new connector during the interim are lost.

---

### Likelihood Explanation

The owner of the Aurora Engine contract is a known, privileged NEAR account. `set_eth_connector_contract_account` is an intentional administrative function — it exists precisely to allow connector upgrades. Any legitimate connector upgrade (e.g., deploying a new version of the ETH connector) that does not first drain all user balances from the old connector will trigger this freeze. The function requires no timelock, no multi-sig, and no on-chain migration step. A single owner transaction is sufficient. Given that connector upgrades are a planned operational activity, the likelihood of this being triggered — even accidentally — is realistic.

---

### Recommendation

Before overwriting the stored connector account ID, the engine should verify that the old connector's total NEP-141 supply is zero (i.e., all balances have been migrated), or implement a two-phase migration:

1. **Pause** all connector-proxied operations.
2. **Migrate** all user balances from the old connector to the new one (or require the caller to prove migration is complete via a cross-contract view call).
3. **Update** the stored connector account ID only after migration is confirmed.

Alternatively, store the old connector account ID alongside the new one and allow users to call a dedicated `withdraw_from_legacy_connector` function that routes to the old account ID for a bounded migration window.

---

### Proof of Concept

1. User Alice calls `deposit` on the ETH connector (`connector_A`), bridging 10 ETH. Her NEP-141 balance of `10 ETH` is recorded in `connector_A`'s storage.

2. The Aurora Engine owner calls:
   ```
   set_eth_connector_contract_account({ account: "connector_B", withdraw_serialize_type: Borsh })
   ```
   This executes `set_connector_account_id(io, "connector_B")`, overwriting the stored key. [8](#0-7) 

3. Alice calls `withdraw` on the Aurora Engine contract. The engine executes:
   ```rust
   return_promise(io, env, "engine_withdraw", args, ONE_YOCTO)
   ```
   Inside `return_promise`, `get_connector_account_id(&io)` now returns `"connector_B"`. The cross-contract call is dispatched to `connector_B.engine_withdraw(...)`. [9](#0-8) 

4. `connector_B` has no record of Alice's balance. The call fails or returns zero. Alice's 10 ETH remains in `connector_A`'s storage with no engine-level path to retrieve it.

5. Alice cannot call `connector_A.engine_withdraw` directly because `connector_A` enforces `predecessor == aurora` (the engine contract), and the engine no longer routes to `connector_A`.
