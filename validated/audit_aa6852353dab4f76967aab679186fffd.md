### Title
Unauthorized Transfer and Burn of Native Bank Tokens via Missing Caller Authorization in Bank Precompile `transfer` Method - (File: x/cronos/keeper/precompiles/bank.go)

### Summary
The `BankContract.Run()` precompile handler for the `transfer` method accepts an arbitrary `sender` address from calldata and executes `bankKeeper.SendCoins(ctx, from, to, ...)` without verifying that `from == contract.Caller()`. Any EVM contract can therefore transfer `evm/<callerContract>` native bank tokens out of any victim address that holds such tokens. The same flaw exists in the `burn` branch, where the target address to burn from is also taken from calldata without an authorization check.

### Finding Description

The bank precompile at `x/cronos/keeper/precompiles/bank.go` exposes three mutating methods: `mint`, `burn`, and `transfer`. The denom for all operations is derived from the EVM caller: `EVMDenom(contract.Caller())` = `evm/<callerContractAddress>`.

In the `transfer` case (lines 167–200), the `sender` field is unpacked directly from calldata:

```go
sender := args[0].(common.Address)   // arbitrary, from calldata
recipient := args[1].(common.Address)
amount := args[2].(*big.Int)
...
from := sdk.AccAddress(sender.Bytes())
to := sdk.AccAddress(recipient.Bytes())
...
denom := EVMDenom(contract.Caller())  // evm/<callerContract>
...
bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt))
```

There is no check that `sender == contract.Caller()`. Any contract can call `bank.transfer(victimAddress, attackerAddress, amount)` and the precompile will execute `SendCoins` from the victim's account.

The same pattern applies to `burn` (lines 113–156): `recipient` (the address to burn from) is taken from calldata, and `SendCoinsFromAccountToModule(ctx, addr, ...)` is called without verifying `addr == contract.Caller()`.

By contrast, the other precompiles in the codebase use the `exec` helper in `utils.go` which explicitly enforces `caller == signer`:

```go
caller := common.BytesToAddress(signers[0])
if caller != e.caller {
    return nil, fmt.Errorf("caller is not authenticated: ...")
}
```

The bank precompile bypasses this pattern entirely.

The intended usage, as shown in `TestBank.sol`, is for a contract to pass `msg.sender` as the `sender`:

```solidity
function nativeTransfer(address recipient, uint256 amount) public returns (bool) {
    return bank.transfer(msg.sender, recipient, amount);
}
```

But the precompile does not enforce this.

### Impact Explanation

**Critical** — unauthorized transfer and burn of precompile-controlled assets (`evm/<callerContract>` native bank tokens).

Any contract can call `bank.transfer(victimAddress, attackerAddress, amount)` to move `evm/<callerContract>` tokens from the victim to the attacker, or `bank.burn(victimAddress, amount)` to destroy them, with no consent from the victim. This is a direct unauthorized balance change for precompile-controlled assets, matching the Critical impact tier.

The practical constraint is that the victim must hold `evm/<callerContract>` tokens. This is satisfied in two realistic scenarios:
1. A legitimate, widely-deployed contract (e.g., a DEX pair or lending pool) has users holding its native bank tokens. If that contract exposes any function that calls `bank.transfer` with attacker-controlled parameters, all users' balances are drainable.
2. An attacker first calls `bank.mint(victimAddress, amount)` (which any contract can do for its own denom) to seed victims with tokens, then calls `bank.transfer(victimAddress, attackerAddress, amount)` to drain them — useful for griefing or accounting manipulation.

### Likelihood Explanation

Any unprivileged EVM account can deploy a contract and call the bank precompile. The entry path is fully reachable with no special permissions. The only precondition is that the victim holds tokens of the calling contract's denom, which is a real but not insurmountable constraint.

### Recommendation

In the `transfer` case, enforce that the `sender` argument equals `contract.Caller()`:

```go
case TransferMethodName:
    ...
    sender := args[0].(common.Address)
    if sender != contract.Caller() {
        return nil, errors.New("sender must be the caller")
    }
```

Similarly, in the `burn` case, enforce that the address to burn from equals `contract.Caller()`:

```go
case BurnMethodName:
    recipient := args[0].(common.Address)
    if recipient != contract.Caller() {
        return nil, errors.New("burn target must be the caller")
    }
```

This mirrors the pattern already used in `utils.go`'s `exec` helper for other precompiles.

### Proof of Concept

1. Alice holds 1000 `evm/LegitToken` native bank tokens (obtained via `LegitToken.moveToNative()`).
2. Attacker deploys `MaliciousContract` at the same address as `LegitToken` (or, more practically, calls into `LegitToken` via a reentrancy or callback that passes attacker-controlled `sender`).
3. `MaliciousContract` calls `bank.transfer(alice, attacker, 1000)` on the bank precompile at `0x0000000000000000000000000000000000000064`.
4. The precompile executes `bankKeeper.SendCoins(ctx, alice, attacker, 1000 evm/MaliciousContract)` with no authorization check.
5. Alice's native bank balance is drained to the attacker.

The missing guard is at: [1](#0-0) 

The `sender` is taken from calldata at line 175 and used directly as `from` in `SendCoins` at line 192, with no check against `contract.Caller()`.

Compare with the authorized pattern in: [2](#0-1) 

The `exec` helper verifies `caller == signer` before executing any state change, a guard entirely absent from the bank precompile's `transfer` and `burn` branches.
