### Title
WalletConnect `chia_createOfferForIds` Silently Misclassifies NFT Launcher IDs as CATs in Confirmation Dialog — (File: `packages/gui/src/electron/commands/parseCommandDisplay.ts`)

---

### Summary
When a WalletConnect dapp sends a `chia_createOfferForIds` request with a bytes32 NFT launcher ID on the **receiving (positive-amount) side** and omits the `driver_dict`, the confirmation dialog silently displays the asset as a CAT token instead of an NFT. A malicious dapp can exploit this to trick the user into creating an offer that spends real XCH in exchange for a worthless NFT, then immediately accept that offer from the dapp's own wallet — resulting in direct XCH loss.

---

### Finding Description

In `parseCommandDisplay.ts`, the function `createOfferAssetKinds` handles the WalletConnect `chia_wallet.create_offer_for_ids` command. It classifies each asset in the offer by type for display in the confirmation dialog.

For the **spending side** (negative amounts — assets the user is offering), bytes32 asset IDs without a `driver_dict` entry correctly cause the confirmation to fail with `'Asset type is not valid'`, blocking the request: [1](#0-0) 

For the **receiving side** (positive amounts — assets the user is requesting), bytes32 asset IDs without a `driver_dict` entry silently fall back to `'cat'`: [2](#0-1) 

The `driver_dict` parameter is optional and defaults to `{}` when absent: [3](#0-2) 

The `chia_createOfferForIds` dapp command is fully exposed via WalletConnect: [4](#0-3) 

The existing test suite explicitly documents the asymmetry — spending-side bytes32 assets without drivers fail closed, but there is no equivalent test asserting that receiving-side bytes32 assets without drivers are rejected: [5](#0-4) 

The confirmation dialog is rendered from `parseCommandDisplay`'s output before the command is dispatched to the wallet: [6](#0-5) 

---

### Impact Explanation

A malicious dapp sends:
```json
{
  "offer": {
    "1": "-1000000000000",
    "<worthless_nft_launcher_id>": "1"
  }
}
```
with no `driver_dict`. The confirmation dialog renders: **"Spending 1 XCH — Receiving 1 of Unknown CAT"**. The user approves, believing they are creating an offer to buy a CAT. The actual on-chain offer locks 1 XCH and requests the NFT. The dapp then accepts the offer from its own wallet (no WalletConnect interaction required), delivering the worthless NFT and claiming the user's XCH. The user suffers a direct, irreversible XCH loss.

---

### Likelihood Explanation

The user must have an active WalletConnect session with the malicious dapp and must approve the spoofed offer creation dialog. Both conditions are realistic for a user who has connected to a dapp they believe is legitimate. The dapp can mint a worthless NFT cheaply before executing the attack. No leaked keys, host compromise, or cryptographic break is required.

---

### Recommendation

Remove the `?? 'cat'` fallback in `createOfferAssetKinds` for the receiving side. Receiving-side bytes32 assets without a `driver_dict` entry should be treated the same as spending-side bytes32 assets: the confirmation should fail closed (throw `'Asset type is not valid'`), forcing any dapp that includes an NFT on the receiving side to supply a valid `driver_dict` entry that identifies it as a `singleton`. [7](#0-6) 

---

### Proof of Concept

1. Establish a WalletConnect session between a malicious dapp and the user's Chia GUI wallet.
2. Dapp calls `chia_createOfferForIds` with the payload:
   ```json
   {
     "offer": { "1": "-1000000000000", "<attacker_nft_launcher_id>": "1" }
   }
   ```
   (no `driver_dict` field)
3. The GUI confirmation dialog displays: **"Spending 1 XCH / Receiving 1 of Unknown CAT"** — the NFT launcher ID is classified as `'cat'` via the `?? 'cat'` fallback.
4. User clicks **Create**, believing they are making a CAT-buy offer.
5. The wallet backend creates the offer: user offers 1 XCH, requests the attacker's NFT.
6. Attacker's own wallet (outside WalletConnect) calls `take_offer` with the returned offer blob, accepting it immediately.
7. User's 1 XCH is transferred to the attacker; user receives the worthless NFT.
