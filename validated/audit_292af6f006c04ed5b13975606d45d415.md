### Title
Invoke `__validate__` Bypass via Non-Specific `account_tx_in_pool_or_recent_block` Check Allows Signature-Free Mempool Admission — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

`skip_stateful_validations` in the gateway's stateful validator skips the `__validate__` entry point (account signature verification) for any invoke transaction with `nonce == 1` targeting an account whose on-chain nonce is `0`, provided `account_tx_in_pool_or_recent_block` returns `true`. That helper returns `true` for **any** transaction in the pool for the target address — not exclusively a `deploy_account`. Once a legitimate user's `deploy_account` enters the mempool, an attacker can immediately submit an invoke with `nonce = 1` and an arbitrary/invalid signature for the same address, bypassing `__validate__` entirely and gaining mempool admission without a valid signature.

---

### Finding Description

`skip_stateful_validations` is the sequencer analog of the `if (cooldownStart == 0)` guard in the external report: a one-time state flag (`account_tx_in_pool_or_recent_block`) is checked to decide whether to skip an expensive enforcement step (`__validate__`), but the flag is set by a weaker condition than the one the skip is meant to represent.

**The guard (lines 437–456):**

```rust
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
    return mempool_client
        .account_tx_in_pool_or_recent_block(tx.sender_address())
        .await
        ...;
}
``` [1](#0-0) 

The comment states the check is sufficient because the account "either has a deploy_account transaction **or** transactions with future nonces that passed validations." The second branch is the flaw: the check does not distinguish between a `deploy_account` and any other transaction type.

**`account_tx_in_pool_or_recent_block` implementation:**

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [2](#0-1) 

It returns `true` for **any** transaction in the pool or recent-block history for the address.

**Effect when `skip_validate = true`:**

```rust
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [3](#0-2) 

With `validate = false`, `perform_validations` returns `Ok(())` immediately without calling `__validate__`:

```rust
if !tx.execution_flags.validate {
    return Ok(());
}
``` [4](#0-3) 

No other check in `run_pre_validation_checks` verifies the signature:

```rust
async fn run_pre_validation_checks(...) -> ... {
    self.validate_state_preconditions(executable_tx, account_nonce).await?;
    validate_by_mempool(executable_tx, account_nonce, mempool_client.clone()).await?;
    let skip_validate =
        skip_stateful_validations(executable_tx, account_nonce, mempool_client.clone()).await?;
    Ok(skip_validate)
}
``` [5](#0-4) 

`validate_by_mempool` only checks nonce validity and duplicates — not signatures.

---

### Impact Explanation

An attacker can inject invoke transactions carrying an arbitrary/invalid signature into the mempool for any account that has a pending `deploy_account`. The gateway admits these transactions without running `__validate__`. They propagate to peer nodes via P2P (every peer runs the same gateway logic and will also skip validation for the same reason). The transactions will ultimately fail during batcher execution, but they are admitted as valid by the gateway/mempool layer.

**Impact category:** High — Mempool/gateway admission accepts invalid transactions before sequencing.

---

### Likelihood Explanation

- Requires no privileged access.
- The trigger condition (`deploy_account` in the mempool for a target address) is a normal, observable network event.
- The attacker only needs to know the target address and submit a single transaction.
- The window is open for the entire duration the `deploy_account` remains in the mempool (potentially minutes).

---

### Recommendation

Replace the non-specific `account_tx_in_pool_or_recent_block` check with one that verifies a `deploy_account` transaction specifically exists in the pool for the target address. Alternatively, add a dedicated `deploy_account_in_pool(address)` query to the mempool that only returns `true` when the pending transaction for that address is of type `DeployAccount`. This mirrors the fix recommended in the external report: enforce the guard condition precisely rather than relying on a weaker proxy.

---

### Proof of Concept

1. Alice submits `deploy_account` for address `X` (class exists, valid constructor). It passes full execution in `perform_validations` and enters the mempool. `account_tx_in_pool_or_recent_block(X)` now returns `true`.

2. Attacker submits `Invoke { sender_address: X, nonce: 1, signature: [0xdead, 0xbeef] }` (arbitrary invalid signature).

3. Gateway stateful validation:
   - `get_nonce_from_state(X)` → `0` (account not yet on-chain).
   - `validate_nonce`: `0 <= 1 <= 200` ✓.
   - `validate_by_mempool`: nonce gap check passes ✓.
   - `skip_stateful_validations`: `nonce == 1 && account_nonce == 0 && account_tx_in_pool_or_recent_block(X) == true` → returns `true`.

4. `run_validate_entry_point` is called with `skip_validate = true` → `execution_flags.validate = false` → `perform_validations` returns `Ok(())` without calling `__validate__`.

5. The attacker's invoke with invalid signature is forwarded to the mempool via `add_tx` and propagated to all peers. The broken invariant: every invoke admitted to the mempool must have passed signature verification or have a legitimate deploy-account-pending reason — here neither is true for the attacker's transaction.
