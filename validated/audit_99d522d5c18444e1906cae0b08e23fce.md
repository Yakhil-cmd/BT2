### Title
Missing Caller Authorization on `update_rewards` Allows Any Address to Permanently Freeze Staker Block Rewards - (File: src/staking/staking.cairo)

### Summary
The `update_rewards` function in `IStakingRewardsManager` is specified to be callable only by the Starkware sequencer, but the implementation contains no caller check whatsoever. Any unprivileged address can call it with `disable_rewards=true`, which advances the global `last_reward_block` without distributing rewards. This permanently destroys the block rewards for every staker for that block, since the sequencer's subsequent call for the same block will be rejected with `REWARDS_ALREADY_UPDATED`.

### Finding Description

The specification at `docs/spec.md` line 1645 explicitly states:

> **access control**: Only starkware sequencer.

The implementation at `src/staking/staking.cairo` lines 1449–1507 contains no such check:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();                          // only checks pause
    let current_block_number = starknet::get_block_number();
    assert!(
        current_block_number > self.last_reward_block.read(),
        "{}",
        Error::REWARDS_ALREADY_UPDATED,
    );
    // ... NO assert_caller_is_sequencer() ...
    let staker_info = self.internal_staker_info(:staker_address);
    // ...
    self.last_reward_block.write(current_block_number);   // ← written unconditionally

    if disable_rewards || self.is_pre_consensus() {
        return;                                           // ← exits without distributing
    }
    // ... distribute rewards ...
}
```

The critical sequence is:
1. `last_reward_block` is written to `current_block_number` **before** the `disable_rewards` branch.
2. If `disable_rewards=true`, the function returns immediately without distributing any rewards.
3. The guard `current_block_number > self.last_reward_block.read()` is now satisfied only for a strictly later block.

`last_reward_block` is a **single global storage slot** shared across all stakers. The test `test_update_rewards_miss_blocks` confirms this: calling `update_rewards` twice with a 2-block gap yields `strk_block_rewards * 2`, meaning the reward calculation uses `last_reward_block` to count missed blocks.

**Attack path:**
- Attacker calls `update_rewards(any_valid_staker, disable_rewards=true)` in block N.
- `last_reward_block` is set to N; no rewards are distributed.
- The legitimate sequencer calls `update_rewards(staker, disable_rewards=false)` in block N+1.
- The sequencer's call succeeds, but rewards are calculated only for the interval `(N, N+1]` — block N's rewards are permanently lost for every staker.
- Repeated every block, this permanently eliminates all consensus-phase block rewards.

### Impact Explanation

**Permanent freezing of unclaimed yield.** Every block where the attacker front-runs the sequencer, all stakers permanently lose their `unclaimed_rewards_own` increment for that block. Pool members also lose their share. Because `last_reward_block` is global, a single attacker call per block is sufficient to zero out rewards for the entire protocol indefinitely. The lost rewards are never recoverable — there is no mechanism to retroactively credit a block whose `last_reward_block` slot has already been consumed.

### Likelihood Explanation

The function is publicly callable with no access restriction. Any EOA or contract can invoke it. The attacker needs only a valid `staker_address` (any active staker suffices) and the ability to submit a transaction. On Starknet L2 the sequencer controls ordering, but the sequencer does not call `update_rewards` in every block for every staker; gaps are normal and expected (the missed-block accumulation logic exists precisely for this reason). An attacker can exploit any such gap. Sustained griefing requires one cheap transaction per block.

### Recommendation

Add a sequencer-only guard at the top of `update_rewards`, analogous to `assert_caller_is_attestation_contract` used in `update_rewards_from_attestation_contract`:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
    self.assert_caller_is_sequencer();   // ← add this
    // ...
}
```

The roles component already provides the infrastructure for such checks (see `only_app_governor`, `only_token_admin`, etc.). A dedicated sequencer role or a hardcoded sequencer address check should be introduced and enforced here.

### Proof of Concept

```
1. Deploy the system (consensus rewards active).
2. Attacker (any address) calls:
       staking.update_rewards(staker_address=<any_active_staker>, disable_rewards=true)
   in block N.
   → last_reward_block = N; no rewards distributed.
3. Sequencer calls:
       staking.update_rewards(staker_address=<staker>, disable_rewards=false)
   in block N+1.
   → Succeeds, but rewards cover only 1 block (N+1), not 2 (N and N+1).
   → Block N rewards are permanently lost.
4. Repeat step 2 every block.
   → All stakers receive zero consensus rewards indefinitely.
```

The existing test `test_update_rewards_miss_blocks` at `src/staking/tests/test.cairo` lines 3530–3561 already demonstrates the missed-block accumulation mechanic; the same setup with an attacker-controlled first call (no `cheat_caller_address_once`) reproduces the freeze. fn test_update_rewards_miss_blocks() {
    let mut cfg: StakingInitConfig = Default::default();
    general_contract_system_deployment(ref :cfg);
    let staking_contract = cfg.test_info.staking_contract;
    let staking_dispatcher = IStakingDispatcher { contract_address: staking_contract };
    let staking_rewards_dispatcher = IStakingRewardsManagerDispatcher {
        contract_address: staking_contract,
    };
    let staking_config_dispatcher = IStakingConfigDispatcher { contract_address: staking_contract };
    let minting_curve_contract = cfg.reward_supplier.minting_curve_contract;
    advance_epoch_global();
    let current_epoch = staking_dispatcher.get_current_epoch();
    cheat_caller_address_once(
        contract_address: staking_contract, caller_address: cfg.test_info.app_governor,
    );
    staking_config_dispatcher.set_consensus_rewards_first_epoch(epoch_id: current_epoch + 2);
    // Advance `K` epochs to start consensus rewards.
    advance_k_epochs_global();
    stake_for_testing_using_dispatcher(:cfg);
    advance_k_epochs_global();
    let staker_address = cfg.test_info.staker_address;
    let staker_info_before = staking_dispatcher.staker_info_v1(:staker_address);
    let (strk_block_rewards, _) = calculate_current_block_rewards_v3(:minting_curve_contract);
    let staker_info_expected = StakerInfoV1 {
        unclaimed_rewards_own: strk_block_rewards * 2, ..staker_info_before,
    };
    staking_rewards_dispatcher.update_rewards(:staker_address, disable_rewards: false);
    advance_block_number_global(blocks: 2);
    staking_rewards_dispatcher.update_rewards(:staker_address, disable_rewards: false);
    let staker_info_after = staking_dispatcher.staker_info_v1(:staker_address);
    assert!(staker_info_after == staker_info_expected);
}
