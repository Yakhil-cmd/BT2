### Title
Logout-as-Authorization-Boundary Bypass: WalletConnect Commands Execute After User Clicks Logout — (`packages/gui/src/components/app/LogoutButton.tsx`, `packages/core/src/components/Auth/AuthProvider.tsx`, `packages/gui/src/components/walletConnect/WalletConnectProvider.tsx`)

---

### Summary

The "Log Out" button does not log out from the daemon. `handleLogOut` is an explicit no-op. `WalletConnectProvider` is mounted above the router and keeps its `session_request` listener active regardless of route. All fingerprint guards query the still-active daemon and pass. A dApp that keeps a WalletConnect session alive can continue executing commands — including bypassed signing commands — after the user clicks logout.

---

### Finding Description

**Root cause — `handleLogOut` is a no-op:**

```typescript
// AuthProvider.tsx lines 52-55
const handleLogOut = useCallback(async () => {
  // do nothing until backend change API,
  // user is still logged in and syncing with previously selected fingerprint
}, []);
```

`LogoutButton` calls `await logOut()` (this no-op) then `navigate('/')`. The daemon fingerprint remains active. [1](#0-0) [2](#0-1) 

**Correction to the question's proposed mechanism:** The question claims `AuthProvider.useEffect` detects `currentFingerprint !== fingerprint (undefined)` and calls `processNewFingerprint`. This is wrong. `handleLogOut` never calls `setFingerprint(undefined)`, so `fingerprint` state stays set and the condition `currentFingerprint !== fingerprint` is **false**. The re-establishment path does not fire. The actual problem is simpler: the daemon is never logged out at all. [3](#0-2) 

**`WalletConnectProvider` is mounted above the router and stays alive:**

```tsx
// AppProviders.tsx lines 105-108
<WalletConnectProvider projectId={WalletConnectChiaProjectId}>
  <AppState>{outlet ? <Outlet /> : children}</AppState>
  <ModalDialogs />
</WalletConnectProvider>
```

The `session_request` listener is registered in a `useEffect` that depends only on `client`. It is never torn down by a route change. [4](#0-3) [5](#0-4) 

**Fingerprint guards pass because the daemon is still logged in:**

In `useWalletConnectCommand`, `currentFingerprint` comes from `useGetLoggedInFingerprintQuery()` (daemon query). Since the daemon is still logged in, `currentFingerprint === pair.fingerprint` and the guard passes:

```typescript
// useWalletConnectCommand.tsx lines 38-46
const requestedFingerprint = fingerprint ?? currentFingerprint;
if (
  typeof requestedFingerprint !== 'number' ||
  !requestedFingerprint ||
  requestedFingerprint !== pair.fingerprint ||
  currentFingerprint !== pair.fingerprint
) {
  throw new WcError(`Fingerprint not allowed for this command`, WcErrorCode.UNAUTHORIZED_METHOD);
}
``` [6](#0-5) 

The electron-side `dispatchPairRequest` calls `getLoggedInFingerprint()` directly against the daemon — same result, guard passes. [7](#0-6) 

**Bypassed commands execute without any confirmation:**

```typescript
// dispatchPairRequest.ts lines 57-59
if (pair.bypass.includes(command)) {
  return process(context);
}
``` [8](#0-7) 

**What can be bypassed:** The pairing UI allows bypass for `innocuous`, `balance`, `sign`, and `notifications` command groups. Spending commands (`chia_sendTransaction`, etc.) explicitly **cannot** be bypassed — `addDappBypassPermissions` rejects them with "Command cannot bypass confirmation." However, **signing commands** (`chia_signMessageById`, `chia_signMessageByAddress`, etc.) are in the bypassable `sign` group. [9](#0-8) [10](#0-9) 

**Secondary issue — `AppAutoLogin` re-redirects to dashboard:** `isReady` is a module-level variable (never reset). If `enableAutoLogin` is true, `AppAutoLogin` immediately calls `navigate('/dashboard/wallets')` because the daemon fingerprint is still active, undoing the logout navigation entirely. [11](#0-10) 

---

### Impact Explanation

A dApp with an active WalletConnect session and bypass permission for signing commands can silently sign messages with the user's key after the user clicks logout. The user believes they have revoked access; the dApp has not lost it. For non-bypassed commands (spending), a confirmation dialog still appears — but it appears on a screen the user has "left," creating a confusing and exploitable UX where the user may dismiss or accept it without understanding the context.

This is a **High** finding: bypass of WalletConnect signing approval with direct security impact (unauthorized signing after user-initiated logout).

---

### Likelihood Explanation

- Requires an established WalletConnect session with bypass granted for signing commands (user must have explicitly enabled this during pairing — not uncommon for dApps that request it).
- The dApp simply needs to send a `session_request` after the user clicks logout; no race condition or timing precision is required since the daemon stays logged in indefinitely.
- The `AppAutoLogin` re-redirect makes the window of vulnerability unbounded when `enableAutoLogin` is on.

---

### Recommendation

1. `handleLogOut` must actually log out from the daemon (call the `log_out` RPC) and clear `fingerprint` state via `setFingerprint(undefined)`.
2. `WalletConnectProvider` should check an "is logged in" signal (e.g., `AuthContext.fingerprint`) before processing `session_request` events, and reject with `USER_REJECTED` when no fingerprint is set.
3. On logout, all active WalletConnect sessions should be disconnected (call `disconnectPair` for all pairs or at minimum suspend command processing).
4. `AppAutoLogin.isReady` being a module-level variable that is never reset is a separate bug that prevents the logout navigation from having any effect when `enableAutoLogin` is true.

---

### Proof of Concept

1. Establish a WalletConnect session; grant bypass for `chia_signMessageByAddress`.
2. Click the LogoutButton in the dashboard header.
3. Immediately (or at any time after) send a `session_request` for `chia_signMessageByAddress` from the dApp.
4. Observe: the request is processed and a signature is returned — no confirmation dialog, no rejection.
5. Expected: the request should be rejected with `USER_REJECTED` because the user logged out.

Integration test assertion: after `LogoutButton` click, any `session_request` (bypassed or not) must return an error with code `USER_REJECTED`.
