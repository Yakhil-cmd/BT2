### Title
Missing `require_owner_only` in `deploy_upgrade` Allows Any Caller to Force Deployment of Staged Contract Code — (File: engine/src/lib.rs)

---

### Summary

The `deploy_upgrade` entrypoint in the Aurora Engine NEAR contract is callable by **any** NEAR account. It deploys whatever bytecode the owner previously staged via `stage_upgrade`. Because no owner-authorization check is present, an unprivileged caller can force the deployment of staged code at any point after the upgrade delay expires — including in a window where the owner has decided to abort the upgrade.

---

### Finding Description

The two-step upgrade flow works as follows:

1. **`stage_upgrade`** (admin.rs:154–167) — stores new bytecode and records a future block height as the earliest deployment time. Correctly gated by `require_owner_only`.
2. **`deploy_upgrade`** (lib.rs:171–185) — reads the stored bytecode and calls `Runtime::self_deploy`. Only checks `require_running`; **no `require_owner_only` check**. [1](#0-0) 

Compare with `stage_upgrade`, which correctly enforces owner authorization: [2](#0-1) 

And compare with the direct `upgrade` path, which also enforces owner authorization: [3](#0-2) 

The `require_owner_only` helper enforces that `predecessor_account_id == state.owner_id`: [4](#0-3) 

`deploy_upgrade` calls none of these guards — only `require_running`: [5](#0-4) 

---

### Impact Explanation

Once the upgrade delay block height is reached, any NEAR account can call `deploy_upgrade` and atomically replace the entire Aurora Engine WASM bytecode with whatever the owner staged. If the owner staged code that was later found to contain a critical bug (e.g., an unauthorized withdrawal path, a broken accounting invariant, or a broken bridge accounting function) and attempted to abort the upgrade by staging replacement code, an attacker racing to call `deploy_upgrade` before the owner's cancellation transaction lands can force the vulnerable bytecode onto the live contract. A successful deployment of vulnerable code can result in:

- **Critical — Direct theft of user funds**: if the deployed bytecode contains an exploitable withdrawal or ERC-20 accounting bug.
- **Critical — Permanent freezing of funds**: if the deployed bytecode breaks the ETH connector or bridge accounting irreversibly.

---

### Likelihood Explanation

- The upgrade delay (`upgrade_delay_blocks`) is a publicly readable value; an attacker can compute exactly when `deploy_upgrade` becomes callable.
- NEAR block times are ~1 second, giving an attacker a precise, automatable trigger window.
- The owner must race to call `stage_upgrade` again (overwriting the staged code) before the attacker calls `deploy_upgrade`. On a live network with mempool competition, this race is not guaranteed to be won by the owner.
- The precondition (owner having staged code with a latent bug) is realistic in any active development cycle.

---

### Recommendation

Add `require_owner_only` to `deploy_upgrade`, mirroring the pattern used in `stage_upgrade` and `upgrade`:

```rust
pub extern "C" fn deploy_upgrade() {
    let mut io = Runtime;
    let env = Runtime;                          // add env
    let state = state::get_state(&io).sdk_unwrap();
    require_running(&state)
        .map_err(ContractError::msg)
        .sdk_unwrap();
    require_owner_only(&state, &env.predecessor_account_id())  // add this
        .map_err(ContractError::msg)
        .sdk_unwrap();
    let index = internal_get_upgrade_index();
    if io.block_height() <= index {
        sdk::panic_utf8(errors::ERR_NOT_ALLOWED_TOO_EARLY);
    }
    Runtime::self_deploy(&bytes_to_key(KeyPrefix::Config, CODE_KEY));
    io.remove_storage(&bytes_to_key(KeyPrefix::Config, CODE_STAGE_KEY));
}
```

---

### Proof of Concept

1. Owner calls `stage_upgrade` with bytecode `B` (contains a latent accounting bug). Delay of `D` blocks is recorded.
2. Owner discovers the bug before deployment and prepares a corrected bytecode `B'`.
3. At block `N = staged_block + D`, the delay expires. The attacker monitors the chain for this block.
4. At block `N`, the attacker submits a call to `deploy_upgrade`. Because no owner check exists, the call succeeds and `B` (the buggy bytecode) is deployed as the live Aurora Engine contract.
5. The owner's corrective `stage_upgrade(B')` transaction, even if submitted in the same block, arrives after the attacker's `deploy_upgrade` (NEAR transaction ordering within a block is not guaranteed to favor the owner).
6. The Aurora Engine now runs `B`. The attacker exploits the accounting bug in `B` to drain bridged ETH or freeze user balances.

pub fn stage_upgrade<I: IO + Copy, E: Env>(io: I, env: &E) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        let delay_block_height = env.block_height() + state.upgrade_delay_blocks;
        require_owner_only(&state, &env.predecessor_account_id())?;
        io.read_input_and_store(&storage::bytes_to_key(KeyPrefix::Config, CODE_KEY));
        io.write_storage(
            &storage::bytes_to_key(KeyPrefix::Config, CODE_STAGE_KEY),
            &delay_block_height.to_le_bytes(),
        );
        Ok(())
    })
}

pub fn require_owner_only(
    state: &state::EngineState,
    predecessor_account_id: &AccountId,
) -> Result<(), ContractError> {
    if &state.owner_id != predecessor_account_id {
        return Err(errors::ERR_NOT_ALLOWED.into());
    }
    Ok(())
}
