### Title
Malicious Participant Can Corrupt CKD Output Without Detection Due to Missing Contribution Validation — (`File: src/confidential_key_derivation/protocol.rs`)

### Summary

The `do_ckd_coordinator` function in the CKD protocol aggregates each participant's `(big_y, big_c)` contribution by simple addition, with no cryptographic proof that the participant used their actual key share `x_i`. A single malicious participant can send arbitrary group elements, causing the coordinator to produce a silently corrupted `CKDOutput`. The honest coordinator has no mechanism to detect or reject the invalid contribution.

### Finding Description

In `do_ckd_coordinator`, the coordinator collects one `CKDOutput` from every other participant and unconditionally adds the received group elements into the running aggregate: [1](#0-0) 

Each honest participant is supposed to compute:

```
big_s_i  = x_i · H(pk ‖ app_id)
big_c_i  = big_s_i + y_i · app_pk
big_y_i  = y_i · G
```

and then Lagrange-weight both values before sending them. The coordinator's correctness depends entirely on every participant having used their real signing share `x_i`. There is no zero-knowledge proof attached to the message, and the coordinator performs no pairing-based or algebraic check against the participant's public verification-key share `vk_share_i` (which is available in the public key package from DKG).

A verifiable check would be, for example:

```
e(big_c_i − big_y_i · app_pk, G2) == e(H(pk ‖ app_id), vk_share_i)
```

This check is absent. The `compute_signature_share` function that produces the honest values is: [2](#0-1) 

The `recv_from_others` helper waits for exactly one message from every participant and returns it unconditionally: [3](#0-2) 

No validation is applied to the deserialized `CKDOutput` before it is added to the aggregate.

**Contrast with DKG:** The DKG protocol (`src/dkg.rs`) requires every participant to attach a Schnorr proof-of-knowledge over their secret coefficient and verifies it before accepting the commitment: [4](#0-3) 

No equivalent proof exists in the CKD round.

### Impact Explanation

A malicious participant sends a crafted `(big_y′, big_c′)` pair instead of the honest values. The coordinator aggregates these without complaint. The resulting `CKDOutput` satisfies:

```
C − a · Y  ≠  msk · H(pk ‖ app_id)
```

so `unmask(app_sk)` returns a point that is not the correct BLS signature / confidential derived key. The coordinator and any downstream consumer silently accept this corrupted output. This maps directly to the allowed High impact: **Corruption of CKD outputs so honest parties accept unusable cryptographic outputs**. [5](#0-4) 

### Likelihood Explanation

Any participant in a CKD session is a reachable, unprivileged attacker. The CKD protocol is single-round and requires all `n` participants to contribute; a single malicious participant is sufficient to corrupt the output. No special keys, network position, or timing is required. The attack is deterministic and undetectable by the coordinator under the current code.

### Recommendation

Add a pairing-based consistency check in `do_ckd_coordinator` before accumulating each participant's contribution. For participant `p` with public verification-key share `vk_share_p` (available from the `KeygenOutput.public_key` package):

```rust
// Verify: e(big_c_i - big_y_i * app_pk, G2_generator) == e(H(pk || app_id), vk_share_p)
let lhs = pairing(participant_output.big_c() - participant_output.big_y() * app_pk, G2::generator());
let rhs = pairing(hash_point, vk_share_p.to_element());
if lhs != rhs {
    return Err(ProtocolError::MaliciousParticipant(from));
}
```

Alternatively, require each participant to attach a zero-knowledge proof of correct formation (analogous to the Schnorr proof-of-knowledge used in DKG), binding the contribution to the participant's known public key share.

### Proof of Concept

1. Honest setup: run DKG for participants `[P1, P2, P3]` with threshold 2, obtaining `(msk, vk_shares)`.
2. Malicious participant `P2` participates in CKD but instead of computing the honest `(big_y_2, big_c_2)`, sends `(G, G)` (the generator point for both fields).
3. The coordinator (`P1`) calls `recv_from_others` and receives `P2`'s crafted `CKDOutput { big_y: G, big_c: G }`.
4. At line 53–54, the coordinator adds these values into `norm_big_y` and `norm_big_c` without any check.
5. The final `CKDOutput` is returned to the caller. `ckd_output.unmask(app_sk)` returns a point that is not `msk · H(pk ‖ app_id)`.
6. Any downstream verification (e.g., `verify_signature` as called in `tests/ckd.rs` line 74) fails, or worse, the corrupted key is silently used. [6](#0-5) [7](#0-6)

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}

pub async fn recv_from_others<T>(
    chan: &SharedChannel,
    waitpoint: u64,
    participants: &ParticipantList,
    me: Participant,
) -> Result<Vec<(Participant, T)>, ProtocolError>
where
    T: serde::de::DeserializeOwned,
{
    let mut seen = ParticipantCounter::new(participants);
    seen.put(me);
    let mut messages = Vec::with_capacity(participants.others(me).count());

    while !seen.full() {
        let (from, msg) = chan.recv(waitpoint).await?;
        if seen.put(from) {
            messages.push((from, msg));
        }
    }

    Ok(messages)

/// Verifies the proof of knowledge of the secret coefficients used to generate the
/// public secret sharing commitment.
fn internal_verify_proof_of_knowledge<C: Ciphersuite>(
    session_id: &HashOutput,
    domain_separator: &mut DomainSeparator,
    participant: Participant,
    commitment: &VerifiableSecretSharingCommitment<C>,
    proof_of_knowledge: &Signature<C>,
) -> Result<(), ProtocolError> {
    // creates an identifier for the participant
    let id = participant.scalar::<C>();
    let vk_share = commitment
        .coefficients()
        .first()
        .ok_or_else(|| ProtocolError::AssertionFailed("Empty coefficient list".to_string()))?;

    let big_r = proof_of_knowledge.R();
    let z = proof_of_knowledge.z();
    let c = challenge::<C>(domain_separator, session_id, id, vk_share, big_r)?;
    if *big_r != <C::Group>::generator() * *z - vk_share.value() * c.to_scalar() {
        return Err(ProtocolError::InvalidProofOfKnowledge(participant));
    }
    Ok(())
}
