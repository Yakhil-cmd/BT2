### Title
Counter-Offer Destination Hijack via Attacker-Controlled `ph` in Blockchain Notification — (`packages/gui/src/hooks/useBlockchainNotifications.tsx`, `packages/gui/src/components/notification/NotificationOffer.tsx`)

### Summary

An unprivileged attacker who can send an on-chain notification (a standard Chia blockchain feature) can craft a payload with `t: 1` and `d.ph: <attackerPuzzleHash>`. The `prepareNotifications` function emits a notification object typed as `NotificationType.OFFER` but with `puzzleHash` attached. Because `canCounterOffer` in `NotificationOffer` accepts `NotificationType.OFFER` as a valid type for counter-offering, the UI presents a Counter button. When the user clicks it, the attacker's puzzle hash is passed as the counter-offer destination address, routing the user's counter-offer to the attacker.

### Finding Description

**Step 1 — `prepareNotifications` creates a hybrid object**

In `useBlockchainNotifications.tsx`, when the decoded payload has `type === 1` and a non-empty `ph` field, the code returns:

```typescript
// type: NotificationType.COUNTER_OFFER,   ← commented out
type: NotificationType.OFFER,
...
puzzleHash,                                 ← attacker-controlled value still attached
``` [1](#0-0) 

The commented-out line is a direct indicator that the type was intentionally changed from `COUNTER_OFFER` to `OFFER`, but `puzzleHash` was left in the returned object. The TypeScript type for `NotificationOffer` does not include `puzzleHash`, but the runtime object carries it anyway.

**Step 2 — `canCounterOffer` does not restrict to `COUNTER_OFFER` type**

In `NotificationOffer.tsx`:

```typescript
const canCounterOffer =
  [NotificationType.OFFER, NotificationType.COUNTER_OFFER].includes(type) &&
  'puzzleHash' in notification &&
  !!notification.puzzleHash;
``` [2](#0-1) 

Because `NotificationType.OFFER` is included in the type check, any OFFER-typed notification that carries a `puzzleHash` at runtime will set `canCounterOffer = true`.

**Step 3 — Attacker's puzzle hash is forwarded as the counter-offer destination**

```typescript
navigate('/dashboard/offers/view', {
  state: {
    ...
    canCounterOffer,
    address: 'puzzleHash' in notification ? notification.puzzleHash : undefined,
  },
});
``` [3](#0-2) 

The offer view receives `canCounterOffer: true` and `address: attackerPuzzleHash`. The same pattern is confirmed in `OfferIncomingTable.tsx`, where `puzzleHash` is converted to a bech32m address and passed to the offer builder as the counter-offer destination: [4](#0-3) 

### Impact Explanation

The user's counter-offer is sent to the attacker's address. The attacker, who controls that puzzle hash, receives the counter-offer notification and can accept it on their own terms. Depending on the offer contents, this can result in direct loss of XCH, CATs, or NFTs. This falls under **High** impact: spoofed notification state causes the user to send an offer to the wrong destination.

### Likelihood Explanation

Sending an on-chain notification is a standard, permissionless Chia operation requiring only a small XCH fee. The attacker needs no special access. The user only needs to click the notification and then click Counter — a natural action when receiving what appears to be a legitimate offer.

### Recommendation

1. **Fix `canCounterOffer`** to only allow counter-offering when the notification type is explicitly `COUNTER_OFFER`:
   ```typescript
   const canCounterOffer =
     type === NotificationType.COUNTER_OFFER &&
     'puzzleHash' in notification &&
     !!notification.puzzleHash;
   ``` [2](#0-1) 

2. **Restore the commented-out type** in `prepareNotifications` so that a payload with `ph` present produces a `COUNTER_OFFER`-typed notification, not an `OFFER`-typed one with a dangling `puzzleHash`:
   ```typescript
   type: NotificationType.COUNTER_OFFER,  // restore this
   ``` [5](#0-4) 

Both fixes together close the gap: the type system and the runtime check will agree, and `canCounterOffer` will only be true for genuine counter-offer notifications.

### Proof of Concept

1. Attacker broadcasts an on-chain notification with hex-encoded JSON:
   ```json
   {"v":1,"t":1,"d":{"u":"https://legitimate-offer-host/offer123","ph":"<attacker_puzzle_hash_hex>"}}
   ```
2. Victim's wallet receives the notification. `prepareNotifications` decodes it and returns `{ type: NotificationType.OFFER, offerURL: "...", puzzleHash: "<attacker_puzzle_hash_hex>" }`.
3. `NotificationOffer` renders with `canCounterOffer = true` (OFFER type passes the check, `puzzleHash` is present).
4. Victim clicks the notification → navigated to `/dashboard/offers/view` with `canCounterOffer: true` and `address: <attacker_puzzle_hash_hex>`.
5. Victim clicks Counter → offer builder opens with `address` set to the attacker's bech32m address.
6. Victim completes and submits the counter-offer. Attacker receives it and accepts, draining the victim's offered assets.
