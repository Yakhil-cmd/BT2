### Title
NFT Seller Net Proceeds Displayed as Gross Amount Due to Commented-Out Royalty Deduction - (File: `packages/gui/src/components/offers/utils.ts`)

### Summary
In `calculateNFTRoyalties`, the `nftSellerNetAmount` field is unconditionally set to the full gross `amount` because the correct deduction logic was commented out. Every UI surface that shows "You will receive" or "Net Proceeds" to an NFT seller therefore displays the pre-royalty amount, not the actual post-royalty proceeds. The negative-amount safety guard is also permanently disabled as a consequence.

### Finding Description
In `packages/gui/src/components/offers/utils.ts`, `calculateNFTRoyalties` computes `nftSellerNetAmount` as:

```typescript
const nftSellerNetAmount: number = amount;
// : parseFloat(
//     (amount - parseFloat(royaltyAmountString) - makerFee).toFixed(12),
//   );
``` [1](#0-0) 

The correct formula — `amount - royaltyAmount - makerFee` — is commented out, so `nftSellerNetAmount` always equals the gross `amount` regardless of royalty percentage.

This value propagates to two distinct UI flows:

**1. Offer creation (`NFTOfferEditor.tsx`):** When a user selects "Sell an NFT" (`NFTForToken` tab), the label "You will receive" renders `nftSellerNetAmount`, which is the full gross amount. [2](#0-1) 

The negative-amount guard `showNegativeAmountWarning` is derived from `nftSellerNetAmount < 0`. Because `nftSellerNetAmount === amount` (always positive), this guard can never fire even when royalties exceed the asking price. [3](#0-2) 

**2. Offer viewing/acceptance (`NFTOfferViewer.tsx`):** `overrideNFTSellerAmount` is computed from `nftSaleInfo?.nftSellerNetAmount` and injected into `NFTOfferSummary` as the displayed amount for the NFT seller side. It is also shown as "Net Proceeds" in the purchase summary panel. [4](#0-3) [5](#0-4) 

### Impact Explanation
An NFT seller is shown that they will receive the full gross amount (e.g., 10 XCH) when they will actually receive the gross amount minus creator royalties (e.g., 9 XCH for a 10% royalty NFT). The user approves and signs the offer based on a materially incorrect financial figure. The negative-amount guard being permanently disabled means a seller can create an offer where royalties exceed the asking price with no warning — they receive zero net proceeds while the offer is still accepted on-chain.

### Likelihood Explanation
Any NFT with a non-zero `royaltyPercentage` triggers this path. The NFT offer editor and viewer are standard, unprivileged UI flows reachable by any wallet user. No special access or attacker capability is required — the seller themselves is the victim of the miscalculation.

### Recommendation
Restore the commented-out net-amount formula in `calculateNFTRoyalties`:

```typescript
const nftSellerNetAmount: number = parseFloat(
  (amount - parseFloat(royaltyAmountString) - makerFee).toFixed(12),
);
```

This will also restore the `showNegativeAmountWarning` guard to functional status.

### Proof of Concept
1. Open the Chia GUI and navigate to the NFT offer editor.
2. Select "Sell an NFT" tab and choose an NFT with a 10% royalty.
3. Enter an asking price of 10 XCH.
4. Observe: "You will receive: 10 XCH" — the royalty deduction (1 XCH) is absent.
5. Create the offer and view it in the viewer; "Net Proceeds" still shows 10 XCH.
6. On-chain, the seller receives only 9 XCH; the creator receives 1 XCH.
7. Repeat with an asking price of 0.5 XCH on a 10% royalty NFT: no negative-amount warning appears even though the royalty (0.05 XCH) is deducted from the seller's proceeds, leaving them with 0.45 XCH — and if royalty exceeded the price, no warning would fire at all.
