### Title
Missing `msg_hash != 0` Validation in OT-Based ECDSA `sign()` Allows Unauthorized Threshold Signature for Attacker-Chosen Zero Hash - (File: src/ecdsa/ot_based_ecdsa/sign.rs)

### Summary

The OT-based ECDSA `sign()` function accepts `msg_hash = 0` without rejection, while the robust ECDSA `sign()` explicitly rejects it. A malicious coordinator can broadcast `h = 0` to all participants, causing them to produce valid signature shares for the zero message hash. The resulting signature is a cryptographically valid threshold ECDSA signature for an attacker-chosen input. If the same presignature is ever reused across two sessions (one with `h = 0` and one with `h ≠ 0`), the nonce `k` and then the aggregate secret key `x` can be extracted algebraically.

### Finding Description

**Root cause:** `src/ecdsa/ot_based_ecdsa/sign.rs` `sign()` performs no check on `msg_hash` being zero, while `src/ecdsa/robust_ecdsa/sign.rs` `sign()` explicitly guards against it.

Robust ECDSA `sign()` (lines 91–95):
```rust
if bool::from(msg_hash.is_zero()) {
    return Err(InitializationError::BadParameters(
        "msg_hash cannot be 0 to avoid potential split view attacks".to_string(),
    ));
}
```

OT-based ECDSA `sign()` (lines 22–76): no such check exists. The function accepts any `msg_hash: Scalar`, including zero, and proceeds directly to protocol execution.

**Signing formula in OT-based ECDSA** (`compute_signature_share`, lines 139–159):
```rust
let r = x_coordinate(&presignature.big_r);
Ok(msg_hash * k_i + r * sigma_i)
```

With `msg_hash = 0`, this reduces to `s_i = R_x * sigma_i`, where `sigma_i` is a share of `sigma = k * x` (nonce times secret key). The coordinator sums shares to get `s = R_x * k * x`. The coordinator's final verification (`sig.verify(&public_key, &msg_hash)` at line 129) passes because `(R, s)` is a valid ECDSA signature for `h = 0`.

**Key extraction path (if presignature is reused):** If a coordinator obtains two signing sessions over the same presignature — one with `h = 0` yielding `s₀ = R_x * k * x`, and one with `h ≠ 0` yielding `s₁ = h * k + R_x * k * x` — then:
- `s₁ - s₀ = h * k` → `k = (s₁ - s₀) / h`
- `x = s₀ / (R_x * k)`

The library does not enforce presignature non-reuse at the API level; that is left entirely to the caller.

**Attacker-controlled entry path:** In a real deployment, the coordinator broadcasts the message hash to participants, who each independently call `sign()`. A malicious coordinator broadcasts `h = 0`; all honest participants call `ot_based_ecdsa::sign()` with `msg_hash = Scalar::ZERO`; the library does not reject this; valid signature shares are produced and aggregated.

### Impact Explanation

**Primary impact (always reachable):** A malicious coordinator can produce a valid threshold ECDSA signature for `h = 0` — an attacker-chosen input — without any honest participant being able to detect or prevent it at the library level. This is an unauthorized creation of a valid threshold signature for attacker-chosen inputs (Critical).

**Secondary impact (if presignature reuse occurs):** If the coordinator aborts and retries with the same presignature using a non-zero hash, the nonce `k` and aggregate secret key `x` are fully recoverable from the two public signatures. This is extraction of aggregate secret material (Critical).

### Likelihood Explanation

The primary impact requires only a malicious coordinator — a role explicitly within the threat model (the library's own warning in `src/ecdsa/robust_ecdsa/sign.rs` lines 25–32 acknowledges coordinator malice). The coordinator controls what message hash is broadcast to participants. No cryptographic break, leaked key, or privileged access is required. The secondary impact additionally requires presignature reuse, which the library does not prevent at the API level and which can occur through coordinator-induced abort-and-retry.

### Recommendation

Add the same `msg_hash.is_zero()` guard to `src/ecdsa/ot_based_ecdsa/sign.rs` that already exists in `src/ecdsa/robust_ecdsa/sign.rs`:

```rust
if bool::from(msg_hash.is_zero()) {
    return Err(InitializationError::BadParameters(
        "msg_hash cannot be 0".to_string(),
    ));
}
```

This should be placed in the `sign()` initializer before `make_protocol` is called, consistent with the robust ECDSA pattern.

### Proof of Concept

1. Run OT-based ECDSA keygen + triple generation + presigning for `N` participants.
2. Call `ot_based_ecdsa::sign()` on all participants with `msg_hash = Scalar::ZERO`.
3. Observe that the call succeeds (no `InitializationError`), all participants produce shares, and the coordinator outputs a valid `Signature` that passes `sig.verify(&public_key, &Scalar::ZERO)`.
4. Contrast with `robust_ecdsa::sign()` called with `msg_hash = Scalar::ZERO`, which immediately returns `Err(InitializationError::BadParameters(...))`.

```rust
pub fn sign(
    participants: &[Participant],
    coordinator: Participant,
    threshold: impl Into<ReconstructionLowerBound>,
    me: Participant,
    public_key: AffinePoint,
    presignature: RerandomizedPresignOutput,
    msg_hash: Scalar,
) -> Result<impl Protocol<Output = SignatureOption>, InitializationError> {
    let threshold = usize::from(threshold.into());
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }

    let participants =
        ParticipantList::new(participants).ok_or(InitializationError::DuplicateParticipants)?;

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }

    // ensure number of participants during the signing phase is >= threshold
    if participants.len() < threshold {
        return Err(InitializationError::NotEnoughParticipantsForThreshold {
            threshold,
            participants: participants.len(),
        });
    }

    let ctx = Comms::new();
    let fut = fut_wrapper(
        ctx.shared_channel(),
        participants,
        coordinator,
        me,
        public_key,
        presignature,
        msg_hash,
    );
    Ok(make_protocol(ctx, fut))
}
```
