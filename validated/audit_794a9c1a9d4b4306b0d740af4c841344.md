### Title
Staker Can Instantly Increase Commission to 100% via `set_commission_commitment` + `set_commission`, Stealing All Future Delegator Yield — (File: `src/staking/staking.cairo`)

---

### Summary

A staker can atomically set a commission commitment with `max_commission = 10000` (100%) and `expiration_epoch = current_epoch + 1`, then immediately call `set_commission(10000)` in the same epoch. Because `is_commission_commitment_active` becomes true the instant the commitment is written, there is no enforced delay before the increase takes effect. All subsequent pool reward distributions apply 100% commission, routing every delegator's yield to the staker. The codebase itself acknowledges this gap with a developer note directly above `set_commission_commitment`.

---

### Finding Description

**Vulnerability class:** Accounting / state-transition bug — a semi-privileged actor (staker) can set a critical parameter (commission) to its maximum value without any time-lock protecting existing delegators.

**Root cause — `set_commission_commitment` (staking.cairo:748–785):**

```cairo
/// **Note**: Current commission increase safeguards still allow for sudden commission
/// changes.
fn set_commission_commitment(
    ref self: ContractState, max_commission: Commission, expiration_epoch: Epoch,
) {
    ...
    assert!(expiration_epoch > current_epoch, "{}", Error::EXPIRATION_EPOCH_TOO_EARLY);
    ...
    let commission_commitment = CommissionCommitment { max_commission, expiration_epoch };
    staker_pool_info.commission_commitment.write(Option::Some(commission_commitment));
    ...
}
``` [1](#0-0) 

The only temporal guard is `expiration_epoch > current_epoch`, so the minimum valid commitment is `expiration_epoch = current_epoch + 1`. The commitment is written to storage immediately.

**`is_commission_commitment_active` (staking.cairo:2178–2182):**

```cairo
fn is_commission_commitment_active(
    self: @ContractState, commission_commitment: CommissionCommitment,
) -> bool {
    self.get_current_epoch() < commission_commitment.expiration_epoch
}
``` [2](#0-1) 

Because `current_epoch < current_epoch + 1` is always true, the commitment is **active the moment it is set**. There is no waiting period.

**`update_commission` (staking.cairo:1573–1609):**

```cairo
if self.is_commission_commitment_active(:commission_commitment) {
    assert!(
        commission <= commission_commitment.max_commission,
        "{}",
        Error::INVALID_COMMISSION_WITH_COMMITMENT,
    );
    assert!(commission != old_commission, "{}", Error::INVALID_SAME_COMMISSION);
}
``` [3](#0-2) 

With an active commitment where `max_commission = 10000`, the staker can call `set_commission(10000)` immediately, jumping from any prior commission (e.g. 2%) to 100% in the same epoch.

**Effect on delegator rewards — `calculate_staker_pools_rewards` (staking.cairo:1989–1991):**

```cairo
let (commission_rewards, pool_rewards) = split_rewards_with_commission(
    rewards_including_commission: pool_rewards_including_commission, :commission,
);
``` [4](#0-3) 

With `commission = 10000`, `commission_rewards = pool_rewards_including_commission` and `pool_rewards = 0`. Every token of delegator yield is redirected to the staker.

---

### Impact Explanation

All future STRK (and BTC) pool rewards are routed to the staker as commission. Delegators who remain in the pool after the commission jump earn zero yield. Their principal is recoverable only after the exit wait window, during which they continue to earn nothing. This constitutes **theft of unclaimed/future yield** from delegators — a **High** impact per the allowed scope.

---

### Likelihood Explanation

Any staker who has opened a delegation pool can execute this with two sequential transactions in the same epoch. No special access, leaked key, or external dependency is required. The attacker is the staker themselves — a role explicitly listed as in-scope. The developer note at line 745–746 (`"Current commission increase safeguards still allow for sudden commission changes"`) confirms the team is aware the safeguard is insufficient. [5](#0-4) 

---

### Recommendation

Enforce a minimum delay between the epoch in which a commission commitment is **set** and the epoch in which it can be **used** to increase commission. For example, require `expiration_epoch >= current_epoch + MIN_COMMITMENT_NOTICE_EPOCHS` where `MIN_COMMITMENT_NOTICE_EPOCHS` is large enough for delegators to observe the on-chain event and call `exit_intent` before the higher commission takes effect. Alternatively, make commission increases take effect only at `expiration_epoch`, not immediately upon calling `set_commission`.

---

### Proof of Concept

```
Epoch N:
  1. Staker (commission = 200, i.e. 2%) calls:
       set_commission_commitment(max_commission: 10000, expiration_epoch: N+1)
     → commitment is immediately active (N < N+1)

  2. Staker immediately calls:
       set_commission(commission: 10000)
     → passes: 10000 <= 10000 (max_commission), 10000 != 200 (old_commission)
     → commission is now 10000 (100%)

Epoch N+1 (next reward distribution):
  3. update_rewards / attest is called for the staker.
     calculate_staker_pools_rewards reads commission = 10000.
     split_rewards_with_commission → pool_rewards = 0, commission_rewards = total.
     All delegator yield is sent to the staker's unclaimed_rewards_own.
     Delegators receive 0 from update_rewards_from_staking_contract.

  4. Delegators who have not yet called exit_intent continue to earn 0 yield
     for every subsequent epoch until they exit.
```

The two-step attack (set commitment → set commission) can be executed in consecutive transactions within the same block, giving delegators zero on-chain reaction time before the commission change is final.

        /// **Note**: Current commission increase safeguards still allow for sudden commission
        /// changes.
        /// **Note**: Updating epoch info can impact the commission commitment expiration date.
        fn set_commission_commitment(
            ref self: ContractState, max_commission: Commission, expiration_epoch: Epoch,
        ) {
            self.general_prerequisites();
            assert!(max_commission <= COMMISSION_DENOMINATOR, "{}", Error::COMMISSION_OUT_OF_RANGE);
            let staker_address = get_caller_address();
            let staker_info = self.internal_staker_info(:staker_address);
            assert!(staker_info.unstake_time.is_none(), "{}", Error::UNSTAKE_IN_PROGRESS);
            let staker_pool_info = self.staker_pool_info.entry(staker_address);
            assert!(staker_pool_info.has_pool(), "{}", Error::MISSING_POOL_CONTRACT);
            let current_epoch = self.get_current_epoch();
            if let Option::Some(commission_commitment) = staker_pool_info
                .commission_commitment
                .read() {
                assert!(
                    !self.is_commission_commitment_active(:commission_commitment),
                    "{}",
                    Error::COMMISSION_COMMITMENT_EXISTS,
                );
            }
            // Staker must have a commission since it has a pool.
            let current_commission = staker_pool_info.commission();
            assert!(current_commission <= max_commission, "{}", Error::MAX_COMMISSION_TOO_LOW);
            assert!(expiration_epoch > current_epoch, "{}", Error::EXPIRATION_EPOCH_TOO_EARLY);
            assert!(
                expiration_epoch - current_epoch <= self.get_epoch_info().epochs_in_year(),
                "{}",
                Error::EXPIRATION_EPOCH_TOO_FAR,
            );
            let commission_commitment = CommissionCommitment { max_commission, expiration_epoch };
            staker_pool_info.commission_commitment.write(Option::Some(commission_commitment));
