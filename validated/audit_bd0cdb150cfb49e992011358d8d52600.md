### Title
Zero Native Resource Charge on All EIP-2537 BLS12-381 Precompiles Allows Unbounded Prover Work Without Resource Accounting — (`basic_bootloader/src/bootloader/block_flow/ethereum/hooks/eip_2537/`)

---

### Summary

Every EIP-2537 BLS12-381 precompile (G1/G2 addition, G1/G2 MSM, pairing check, field-to-curve mappings) charges `cost_native = 0` regardless of input size or pair count. EVM gas is charged correctly, but the native resource — which models RISC-V proving cycles and is the mechanism that prevents a block from containing more proving work than the prover can handle — is never incremented by any BLS12-381 call. An unprivileged transaction sender can fill a block with BLS12-381 MSM or pairing calls that consume zero native resource while imposing arbitrarily large RISC-V proving work, bypassing the block native limit entirely.

---

### Finding Description

ZKsync OS implements double resource accounting: EVM gas (ergs) and a native resource that models RISC-V proving cycles. The block-level native limit (`MAX_NATIVE_COMPUTATIONAL`) is enforced in `check_for_block_limits` and is the primary guard against blocks that are too expensive to prove.

All seven EIP-2537 precompiles hard-code `cost_native = 0`:

**G1/G2 Addition** (`addition.rs` lines 22, 63):
```rust
let cost_native = 0;
resources.charge(&R::from_ergs_and_native(cost_ergs, ...from_computational(cost_native)))?;
```

**G1/G2 MSM** (`msm.rs` lines 198, 266):
```rust
let cost_native = 0;
resources.charge(&R::from_ergs_and_native(cost_ergs, ...from_computational(cost_native)))?;
```

**Pairing Check** (`pairing.rs` line 36):
```rust
let cost_native = 0;
resources.charge(&R::from_ergs_and_native(cost_ergs, ...from_computational(cost_native)))?;
```

**Field-to-Curve Mappings** (`mappings.rs` line 26):
```rust
let cost_native = 0;
```

All seven are unconditionally registered in `initialize_eip_2537` with no feature gate, making them live in production.

Contrast with operations that do charge native resources correctly:
- `ModExpImpl` charges `MODEXP_WORST_CASE_NATIVE_PER_GAS = 300` native per gas unit
- `IdentityPrecompile` charges `ID_BASE_NATIVE_COST + ID_BYTE_NATIVE_COST * len`
- `ECRECOVER_NATIVE_COST`, `BN254_PAIRING_BASE_NATIVE_COST`, etc. are all non-zero

The MSM and pairing precompiles are the most severe cases because their EVM gas cost scales with the number of input pairs, meaning a single transaction can legitimately spend its entire gas budget on BLS12-381 work while contributing exactly zero to the block's native resource counter.

**Attack path:**

1. Attacker submits a transaction with a high gas limit (e.g., 30M gas).
2. The transaction calls the G2 MSM precompile (`BLS12_G2MSM`) with maximum pairs: `floor(30_000_000 / 22_500) ≈ 1333` pairs (each pair is 288 bytes of input).
3. EVM gas is exhausted correctly; the transaction is valid.
4. Native resource consumed by this transaction: ~0 (only intrinsic overhead).
5. `check_for_block_limits` sees `computational_native_used ≈ 0`, well below `MAX_NATIVE_COMPUTATIONAL`.
6. The block accepts more such transactions until the gas limit is reached.
7. The prover must execute 1333 × (G2 MSM scalar multiplications) per transaction, multiplied by however many such transactions fit in the block — all unaccounted for in native resource tracking.

---

### Impact Explanation

**Resource accounting bug / forward-proving divergence.** The sequencer's forward execution accepts a block as valid (native limit not exceeded) because BLS12-381 operations contribute zero to the native counter. The prover then attempts to prove the block and must execute the actual RISC-V computation for all BLS12-381 operations, which was never bounded by the native limit. This can cause:

- **Prover timeout or failure**: The prover runs out of its actual cycle budget proving a block the sequencer considered cheap.
- **Block proving stall**: The proving pipeline stalls on a block it cannot prove within its time/resource budget, halting L2 finality.
- **Sustained DoS**: An attacker can continuously submit BLS12-381-heavy transactions at normal gas prices, each costing the attacker only EVM gas but imposing disproportionate proving cost on the network.

The G2 MSM and pairing precompiles are the highest-severity instances because their per-call RISC-V cost is large and scales with input size, while their native charge is always zero.

---

### Likelihood Explanation

**Medium-High.** EIP-2537 precompiles are unconditionally registered in production. Any unprivileged user can call them with crafted inputs. The only cost to the attacker is EVM gas at the standard rate. No privileged access, leaked keys, or external oracle manipulation is required. The attack is deterministic and reproducible.

---

### Recommendation

Apply the same pattern used by `ModExpImpl` and `IdentityPrecompile`: charge a non-zero native cost proportional to the actual RISC-V work performed.

For MSM precompiles, native cost should scale with `num_pairs`:
```rust
let cost_native = num_pairs as u64 * BLS12_G1_MSM_NATIVE_PER_PAIR;
```

For pairing:
```rust
let cost_native = BLS12_PAIRING_BASE_NATIVE + num_pairs as u64 * BLS12_PAIRING_NATIVE_PER_PAIR;
```

Benchmark each BLS12-381 operation on the RISC-V target to derive accurate constants (analogous to `MODEXP_WORST_CASE_NATIVE_PER_GAS`). Until benchmarks are complete, use a conservative upper-bound estimate. Add a CI check that rejects any `cost_native = 0` in precompile implementations that perform variable-cost computation.

---

### Proof of Concept

**G2 MSM zero-native exhaustion:**

```
Transaction gas limit: 30,000,000
G2 MSM gas per pair:   22,500
Max pairs per call:    floor(30,000,000 / 22,500) = 1,333

Native resource consumed by transaction: 0
Block native limit (MAX_NATIVE_COMPUTATIONAL): not triggered

Prover actual work: 1,333 × G2 scalar multiplications on BLS12-381
(each G2 scalar mul ≈ hundreds of thousands of RISC-V cycles)
```
