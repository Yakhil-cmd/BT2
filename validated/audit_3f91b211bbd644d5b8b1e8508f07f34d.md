### Title
Signature Bypass via `skip_stateful_validations` Allows Unsigned Invoke Transactions into Mempool — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator unconditionally skips the `__validate__` entry point (account signature verification) for any invoke transaction with nonce=1 when the account nonce is 0 and **any** transaction exists in the mempool for that account address. A third party who observes a pending `deploy_account` for address X can inject an invoke with an arbitrary invalid signature into the mempool by pre-funding X with STRK tokens, bypassing the only signature check at the gateway admission layer.

### Finding Description

`skip_stateful_validations` at lines 429–461 of `crates/apollo_gateway/src/stateful_transaction_validator.rs` returns `true` (skip) when all of the following hold:

1. The transaction is an `Invoke`.
2. `tx.nonce() == Nonce(ONE)`.
3. `account_nonce == Nonce(ZERO)` (account not yet deployed on-chain).
4. `mempool_client.account_tx_in_pool_or_recent_block(sender_address)` returns `true`. [1](#0-0) 

When `skip_validate = true`, `run_validate_entry_point` constructs an `AccountTransaction` with `execution_flags.validate = false`: [2](#0-1) 

`StatefulValidator::perform_validations` then returns `Ok(())` immediately after `perform_pre_validation_stage`, **without** calling the `__validate__` entry point: [3](#0-2) 

`perform_pre_validation_stage` only checks nonce ordering, fee bounds, and balance — it does **not** verify the account signature: [4](#0-3) 

The critical flaw is that `account_tx_in_pool_or_recent_block` is not specific to `deploy_account` transactions. It returns `true` for **any** transaction in the pool for the account: [5](#0-4) 

The code comment acknowledges this: *"It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."* However, this reasoning does not account for a third party who observes the pending `deploy_account` and exploits the skip window. [6](#0-5) 

### Impact Explanation

The gateway accepts an invoke transaction carrying an **invalid signature** into the mempool. This matches the allowed impact: *"High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

Concrete consequences:
- The attacker's invalid invoke occupies the nonce=1 slot for address X in the mempool.
- The legitimate user's invoke (submitted as part of the deploy_account + invoke UX flow) is either rejected as a duplicate nonce or must pay a higher fee to displace the attacker's transaction via fee escalation.
- The invalid invoke will fail during block execution (the batcher calls `__validate__` with `validate=true`), wasting sequencer execution resources.
- The legitimate user's deploy_account + invoke atomic flow is disrupted.

### Likelihood Explanation

The attack requires:
1. **Observing the mempool**: Possible via the gateway's RPC endpoint (e.g., `starknet_pendingTransactions`) or by operating a node.
2. **Pre-funding address X with STRK tokens**: Required because `verify_can_pay_committed_bounds` checks the sender's balance. This is a trivial, low-cost on-chain operation.
3. **Submitting before the legitimate user's invoke**: The window is the time between the `deploy_account` being submitted and the legitimate user's invoke being submitted — typically seconds in a UX flow where both are sent together.

The attack is low-cost and repeatable. Any account undergoing first-time deployment is vulnerable during the submission window.

### Recommendation

1. **Narrow the skip condition**: Instead of checking `account_tx_in_pool_or_recent_block` (any transaction), specifically verify that a `deploy_account` transaction exists in the mempool for the sender address. Add a dedicated `deploy_account_in_pool(address)` query to the mempool API.

2. **Bind the skip to the deploy_account hash**: Require the invoke transaction to include the hash of the corresponding `deploy_account` transaction (as `native_blockifier`'s `PyValidator::should_run_stateful_validations` does via `deploy_account_tx_hash`), and verify that this hash is present in the mempool before skipping validation. [7](#0-6) 

3. **Restrict the skip to the submitter**: Only skip validation when the invoke is submitted in the same gateway request as the `deploy_account` (i.e., as a batch), preventing third-party exploitation.

### Proof of Concept

```

   → deploy_account enters mempool; account_tx_in_pool_or_recent_block(X) = true.

3. Attacker sends STRK tokens to address X (pre-fund to pass balance check).

4. Attacker submits invoke:
     sender_address = X
     nonce          = 1
     signature      = [0xdead, 0xbeef]   ← invalid
     resource_bounds = { l2_gas: { max_amount: 1000000, max_price_per_unit: 1 }, ... }

   a. validate_state_preconditions:
        validate_nonce: account_nonce=0, tx_nonce=1, 0 ≤ 1 ≤ max_gap → OK

        no duplicate nonce=1 for X → OK

        tx is Invoke ✓, nonce=1 ✓, account_nonce=0 ✓,
        account_tx_in_pool_or_recent_block(X) = true ✓

6. Legitimate user's invoke (nonce=1) is rejected as DuplicateNonce
   or must pay escalated fee to displace the attacker's invalid invoke.

7. During block execution: attacker's invoke reaches __validate__,
   fails with invalid signature, sequencer wastes execution resources.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}

    pub fn should_run_stateful_validations(
        &mut self,
        account_tx: &AccountTransaction,
        deploy_account_tx_hash: Option<TransactionHash>,
    ) -> StatefulValidatorResult<bool> {
        if account_tx.tx_type() != TransactionType::InvokeFunction {
            return Ok(true);
        }
        let tx_info = account_tx.create_tx_info();
        let nonce = self.stateful_validator.get_nonce(tx_info.sender_address())?;

        let deploy_account_not_processed =
            deploy_account_tx_hash.is_some() && nonce == Nonce(Felt::ZERO);
        let tx_nonce = tx_info.nonce();
        let is_post_deploy_nonce = Nonce(Felt::ONE) <= tx_nonce;
        let nonce_small_enough_to_qualify_for_validation_skip =
            tx_nonce <= self.max_nonce_for_validation_skip;

        let skip_validate = deploy_account_not_processed
            && is_post_deploy_nonce
            && nonce_small_enough_to_qualify_for_validation_skip;

        Ok(!skip_validate)
    }
