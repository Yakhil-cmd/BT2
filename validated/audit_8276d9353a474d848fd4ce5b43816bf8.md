### Title
Missing Proof of Correctness for Participant Contributions in CKD Protocol Allows Malicious Participant to Corrupt CKD Output - (File: src/confidential_key_derivation/protocol.rs)

### Summary
The Confidential Key Derivation (CKD) coordinator aggregates participant contributions `(norm_big_y, norm_big_c)` without any zero-knowledge proof or consistency check that each contribution was correctly computed relative to the participant's actual signing share. A malicious participant can send arbitrary group elements, corrupting the final CKD output accepted by the coordinator.

### Finding Description
In `do_ckd_coordinator` (lines 50–55 of `src/confidential_key_derivation/protocol.rs`), the coordinator receives `(norm_big_y, norm_big_c)` from every other participant and blindly sums them:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
```

Each honest participant is supposed to compute, in `compute_signature_share` (lines 148–182):

```
norm_big_y = λi · y · G₁
norm_big_c = λi · (x_i · H(pk ‖ app_id) + y · app_pk)
```

where `x_i` is the participant's private signing share and `y` is a fresh random scalar. The coordinator has no way to verify that the received `norm_big_c` was formed using the participant's actual `x_i` (whose public counterpart `X_i = x_i · G₂` is known from DKG output), nor that `norm_big_y` and `norm_big_c` are consistent with the same ephemeral `y`. No discrete-log equality proof (DLogEq) or similar NIZK is produced by `do_ckd_participant` and no such proof is checked by `do_ckd_coordinator`.

This is directly analogous to the external report's pattern: a protocol step accounts for only part of the required computation (the coordinator's own share) while silently omitting the verification of the other participants' contributions.

### Impact Explanation
A single malicious participant can replace their legitimate `(norm_big_y, norm_big_c)` with arbitrary group elements `(norm_big_y', norm_big_c')`. The coordinator sums all contributions and outputs a `CKDOutput` that does not equal `msk · H(pk ‖ app_id)` under any honest unmasking. Every honest party that relies on this output receives a corrupted confidential derived key. This maps to:

> **High: Corruption of CKD outputs so honest parties accept inconsistent or unusable cryptographic outputs.**

### Likelihood Explanation
Any single participant in the CKD session is an attacker-controlled entry point. The protocol is one-round: the participant sends `(norm_big_y, norm_big_c)` privately to the coordinator with no subsequent challenge or verification step. No out-of-band mechanism prevents a participant from substituting arbitrary values. The attack requires no cryptographic break, no leaked key, and no external dependency — only the ability to send a malformed message, which is within the documented threat model of a malicious participant.

### Recommendation
Add a non-interactive zero-knowledge proof of correct formation alongside each participant's contribution. Concretely, each participant should prove:

1. **DLog consistency**: `norm_big_y = λi · y · G₁` for the same `y` used in `norm_big_c`.
2. **Correct share use**: `norm_big_c = λi · (x_i · H(pk ‖ app_id) + y · app_pk)`, where `x_i` is consistent with the participant's public key share `X_i` from DKG.

A DLogEq proof (already implemented in `src/crypto/proofs/dlogeq.rs`) can bind `y` across `norm_big_y` and the `y · app_pk` term in `norm_big_c`. A separate proof of correct share application (proving knowledge of `x_i` such that `X_i = x_i · G₂` and `x_i · H(pk ‖ app_id)` is the claimed contribution) should also be included. The coordinator must verify all proofs before aggregating.

### Proof of Concept

1. Honest participants P1, P2, P3 run CKD with coordinator P1.
2. Malicious P2 intercepts the send in `do_ckd_participant` and replaces `(norm_big_y, norm_big_c)` with `(G₁, G₁)` (arbitrary group elements).
3. P1's coordinator loop at lines 50–55 adds P2's bogus values without any check.
4. The resulting `CKDOutput` satisfies `big_c - app_sk · big_y ≠ msk · H(pk ‖ app_id)`.
5. Any consumer calling `ckd_output.unmask(app_sk)` receives a garbage confidential key, permanently denying the application access to the correct derived secret. [1](#0-0) [2](#0-1) [3](#0-2)

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}
