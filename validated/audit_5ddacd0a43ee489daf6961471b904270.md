### Title
Signature Replay in `allowPushers` Breaks Revocation Finality — (`smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`allowPushers` contains no nonce and no used-signature registry. A creator holding a pusher's delegation signature with a far-future deadline can call `allowPushers` again with the identical signature after the pusher has called `revokePusher()`, unconditionally restoring `namespaceRemapping[pusher] == creator` for as long as `block.timestamp <= deadline`.

---

### Finding Description

The signature committed in `allowPushers` is:

```
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
``` [1](#0-0) 

The only replay barrier is `_ensureDeadline`, which checks `block.timestamp <= deadline`. [2](#0-1) 

`revokePusher` clears the mapping to `address(0)`: [3](#0-2) 

Because there is no nonce and no `usedSignatures` set, the creator can immediately call `allowPushers` again with the same `(deadline, pusher, sig)` tuple. The signature still passes ECDSA recovery and `_ensureDeadline`, so `namespaceRemapping[pusher]` is written back to `creator`. The code comment on `allowPushers` explicitly acknowledges the concern ("an undated signature could re-establish a delegation AFTER the pusher revoked it") but the deadline-only fix only prevents replay after the deadline expires — not within the deadline window. [4](#0-3) 

---

### Impact Explanation

The revocation-finality invariant is broken: a pusher who calls `revokePusher()` cannot permanently exit a creator's namespace within the deadline window. The creator can replay the old signature an unlimited number of times, each time restoring the delegation.

Regarding the specific "extreme prices / fee miscalculation" claim in the question: this path is **overstated**. The replay does not by itself inject bad prices. For bad prices to reach pools, the pusher would also need to push manipulated data. A pusher who wants to push bad data would not have revoked in the first place; a pusher who revoked to stop contributing can simply stop pushing entirely, which is a complete mitigation regardless of the delegation state. The creator already controls their own namespace directly and does not need the replay to push arbitrary data.

The concrete, scoped impact is: **broken core oracle functionality** — a pusher's self-revocation is not final, violating the documented invariant. This is a Medium-severity finding under the contest rules (broken core functionality), but it does not directly cause pool insolvency, swap conservation failure, or fee miscalculation on its own.

---

### Likelihood Explanation

Any creator who retains a pusher's delegation signature (which is off-chain and freely storable) can execute this replay at any time before the deadline. No privileged role is required — `allowPushers` is a public function callable by any `msg.sender`.

---

### Recommendation

Add a per-pusher nonce or a `usedSignatures` mapping to invalidate a signature after first use:

```solidity
mapping(bytes32 => bool) private _usedDelegationSigs;

// inside allowPushers loop:
bytes32 sigHash = keccak256(signatures[i]);
require(!_usedDelegationSigs[sigHash], "signature already used");
_usedDelegationSigs[sigHash] = true;
```

Alternatively, include a per-pusher nonce in the signed payload and increment it on each successful delegation or revocation.

---

### Proof of Concept

```solidity
// 1. Pusher signs with deadline = block.timestamp + 365 days
uint256 deadline = block.timestamp + 365 days;
bytes memory sig = pusherSign(deadline, pusher, creator);

// 2. Creator establishes delegation
vm.prank(creator);
oracle.allowPushers(deadline, toArray(pusher), toArray(sig));
assertEq(oracle.namespaceRemapping(pusher), creator); // delegated

// 3. Pusher revokes
vm.prank(pusher);
oracle.revokePusher();
assertEq(oracle.namespaceRemapping(pusher), address(0)); // revoked

// 4. Creator replays the SAME signature — succeeds because deadline is still valid
vm.prank(creator);
oracle.allowPushers(deadline, toArray(pusher), toArray(sig));
assertEq(oracle.namespaceRemapping(pusher), creator); // delegation restored — invariant violated
```

    /// @notice Delegates pusher wallets into the caller's namespace. The pusher's EIP-191
    ///         signature is REQUIRED — without it anyone could remap a foreign pusher
    ///         wallet into their own namespace and silently swallow its pushes. The
    ///         deadline is likewise required: the signed consent carries no timestamp of
    ///         its own, so an undated signature could re-establish a delegation AFTER the
    ///         pusher revoked it.
    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
    }
