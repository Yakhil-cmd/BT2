### Title
Staker Can Increase Commission via `set_commission_commitment` to Steal Delegator Yield - (File: `src/staking/staking.cairo`)

### Summary

A staker can unilaterally increase their pool commission to up to 100% at any time by first calling `set_commission_commitment` with a high `max_commission`, then immediately calling `set_commission` to raise the commission to that ceiling. Delegators who joined at a low commission rate have no protection and will have all future rewards calculated at the inflated rate until they exit.

### Finding Description

The protocol enforces that commission can only be **decreased** without a commitment:

```cairo
// src/staking/staking.cairo, update_commission, line 1596
assert!(commission < old_commission, "{}", Error::INVALID_COMMISSION);
```

However, `set_commission_commitment` allows a staker to register a `max_commission` ceiling that is **greater than or equal to** the current commission:

```cairo
// src/staking/staking.cairo, set_commission_commitment, line 770
assert!(current_commission <= max_commission, "{}", Error::MAX_COMMISSION_TOO_LOW);
```

Once an active commitment exists, `update_commission` only enforces two constraints — the new commission must not exceed `max_commission` and must differ from the old value:

```cairo
// src/staking/staking.cairo, update_commission, lines 1583–1589
if self.is_commission_commitment_active(:commission_commitment) {
    assert!(commission <= commission_commitment.max_commission, ...);
    assert!(commission != old_commission, ...);
```

`is_commission_commitment_active` simply checks `current_epoch < expiration_epoch`, so a commitment with `expiration_epoch = current_epoch + 1` is immediately active and valid for one epoch — long enough to execute the increase.

The new commission is written to storage immediately and takes effect for the very next reward calculation:

```cairo
// src/staking/staking.cairo, calculate_staker_pools_rewards, line 1964
let commission = staker_pool_info.commission();
```

The codebase itself acknowledges the gap in a developer note directly above `set_commission_commitment`:

```cairo
// src/staking/staking.cairo, lines 745–746
/// **Note**: Current commission increase safeguards still allow for sudden commission
/// changes.
```

### Impact Explanation

**High — Theft of unclaimed yield.**

A delegator who joined a pool at 1% commission can have all subsequent epoch rewards redirected to the staker at 100% commission. Because rewards are computed using the live commission value at attestation time (not the commission at delegation time), every reward epoch after the increase is fully captured by the staker. The delegator's principal is returned intact, but all accrued yield from the moment of the increase onward is stolen.

### Likelihood Explanation

Any staker who operates a pool can execute this attack with two sequential transactions in the same epoch. No special access, leaked key, or external dependency is required. The minimum commitment window is one epoch (`expiration_epoch > current_epoch`), which is sufficient to complete the attack. The only on-chain signal is a `CommissionCommitmentSet` event, which delegators are unlikely to monitor in real time. Likelihood is **High**.

### Recommendation

1. **Enforce a time-lock on commission increases.** When `set_commission` would raise the commission above its current value, require that the new value has been publicly committed for at least `K` epochs before it takes effect, giving delegators time to exit.
2. **Snapshot commission at delegation time.** Store the commission rate in each pool member's record and apply it to that member's rewards, analogous to the PuttyV2 fix of storing fees in the order struct.
3. **Cap `max_commission` relative to current commission.** Restrict `max_commission` in `set_commission_commitment` to at most `current_commission + delta` for some protocol-defined `delta`, preventing a single commitment from enabling a jump to 100%.

### Proof of Concept

```
Epoch N:
  Staker stakes with commission = 100 (1%).
  Delegator enters pool with 10,000 STRK.

Epoch N (same block):
  Staker calls set_commission_commitment(max_commission: 10000, expiration_epoch: N+1).
    → Passes: current_commission (100) <= max_commission (10000). ✓

  Staker calls set_commission(commission: 10000).
    → Commitment is active (N < N+1). ✓
    → 10000 <= 10000. ✓
    → 10000 != 100. ✓
    → Commission written to storage as 10000 (100%).

Epoch N+1 (attestation):
  calculate_staker_pools_rewards reads commission = 10000.
  split_rewards_with_commission allocates 100% of pool rewards to staker as commission.
  Delegator receives 0 yield.

Epoch N+1 (commitment expires):
  Staker can lower commission back to 100 to avoid detection.
  Delegator's unclaimed yield for epoch N+1 is permanently stolen.
```
