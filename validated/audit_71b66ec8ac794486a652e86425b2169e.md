### Title
Cross-Chain Replay of COA Ownership Proof Signatures Due to Chain-Agnostic Domain Separation Tag - (File: fvm/evm/stdlib/contract.cdc)

### Summary
`EVM.validateCOAOwnershipProof` verifies Cadence account signatures over arbitrary `signedData` using the hardcoded domain separation tag `"FLOW-V0.0-user"`, which contains no chain ID or network identifier. A valid COA ownership proof signature produced on one Flow network (e.g., Testnet) is cryptographically indistinguishable from one produced on another (e.g., Mainnet), enabling cross-chain replay attacks.

### Finding Description
The `validateCOAOwnershipProof` function in `fvm/evm/stdlib/contract.cdc` performs signature verification at line 1082–1086:

```cadence
let isValid = keyList.verify(
    signatureSet: signatureSet,
    signedData: signedData,
    domainSeparationTag: "FLOW-V0.0-user"
)
```

The domain separation tag `"FLOW-V0.0-user"` is a hardcoded, network-agnostic string. It does not incorporate the Flow `ChainID` (e.g., `"flow-mainnet"`, `"flow-testnet"`). Flow defines distinct chain IDs precisely to prevent replay attacks across networks, as documented in `model/flow/chain.go`:

```go
// A ChainID is a unique identifier for a specific Flow network instance.
// Chain IDs are used to prevent replay attacks and to support network-specific address generation.
type ChainID string
```

Because the tag is identical on every Flow network, a signature `Sign(signedData, "FLOW-V0.0-user")` produced on Testnet is valid on Mainnet for the same `signedData`, the same account keys, and the same COA path — all of which are realistic since Flow account addresses are deterministically assigned and users commonly hold accounts on multiple networks.

The code comment at line 1001–1009 acknowledges a related but narrower concern (cross-address replay within the same chain), but does not address cross-chain replay:

```cadence
/// Note: this function does not enforce that `signedData` includes `evmAddress`.
```

The `signedData` parameter is fully attacker-supplied (passed in from the EVM precompile caller), so there is no implicit binding to a specific chain.

### Impact Explanation
The `validateCOAOwnershipProof` function is the backend for the `verifyCOAOwnershipProof` Arch precompile, which is used by the COA contract's ERC-1271 `isValidSignature` implementation:

```solidity
// coa.sol line 118
(bool ok, bytes memory data) = cadenceArch.staticcall(
    abi.encodeWithSignature("verifyCOAOwnershipProof(address,bytes32,bytes)", address(this), _hash, _sig)
);
```

Any EVM contract on Flow Mainnet that uses ERC-1271 with a COA address as the signer can be deceived: an attacker who obtains a valid COA ownership proof signature from Testnet (e.g., by tricking a user into signing a message on Testnet) can replay it on Mainnet to pass `isValidSignature`. This constitutes unauthorized transaction authorization — the EVM contract accepts a signature as valid when the signer never authorized the action on Mainnet.

### Likelihood Explanation
- Flow account addresses are deterministic and identical across networks for the same account creation order, so the same address commonly exists on both Testnet and Mainnet.
- The same account keys are frequently reused across networks.
- The attacker entry path requires no privileged access: any unprivileged user can submit an EVM transaction calling a contract that invokes `isValidSignature`, or directly call `EVM.validateCOAOwnershipProof` via a Cadence script.
- Obtaining a Testnet signature is trivial — Testnet is public and permissionless.

### Recommendation
Include the Flow `ChainID` in the domain separation tag used by `validateCOAOwnershipProof`. Instead of the hardcoded `"FLOW-V0.0-user"`, derive the tag from the chain ID at contract deployment or pass it as a context parameter, e.g., `"FLOW-V0.0-user-flow-mainnet"`. This mirrors how `TransactionDomainTag` is intended to be chain-scoped and how the EVM layer already uses distinct numeric chain IDs (`FlowEVMMainNetChainID = 747`, `FlowEVMTestNetChainID = 545`) to prevent EVM-level replay.

### Proof of Concept

**Setup:**
1. Alice holds a Flow account at address `0xABCD` on both Testnet and Mainnet, with the same key pair, and a COA stored at `/public/coa` on both networks.

**Step 1 — Obtain Testnet signature:**
Alice (or an attacker who tricks Alice) signs `signedData = keccak256("approve transfer X")` on Testnet using `"FLOW-V0.0-user"` as the domain tag. This produces a valid `COAOwnershipProof`.

**Step 2 — Replay on Mainnet:**
The attacker submits an EVM transaction on Mainnet to a contract `VaultContract` that calls:
```solidity
require(IERC1271(coaAddress).isValidSignature(hash, encodedProof) == 0x1626ba7e);
```
The COA's `isValidSignature` calls `verifyCOAOwnershipProof` → `validateCOAOwnershipProof` on Mainnet. Since the domain tag `"FLOW-V0.0-user"` is identical on both networks, the Testnet signature verifies successfully on Mainnet. `VaultContract` authorizes the action as if Alice signed it on Mainnet.
