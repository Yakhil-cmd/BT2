### Title
Pool Admin Can Set Per-Bin Additional Fees Without Any Cap, Bypassing the `maxAdminSpreadFeeE6` Hard Limit - (File: metric-core/contracts/MetricOmmPoolFactory.sol)

### Summary

`setPoolBinAdditionalFees` in `MetricOmmPoolFactory` forwards `addFeeBuyE6` / `addFeeSellE6` directly to the pool with no cap validation, while the parallel `setPoolAdminFees` path enforces `maxAdminSpreadFeeE6`. A pool admin can therefore set per-bin additional fees to `type(uint16).max` (65 535 in E6 ≈ 6.55 %), stacking on top of the already-capped global spread fee and pushing the effective per-bin fee above the protocol's hard ceiling of 20 %.

### Finding Description

The factory enforces a hard cap hierarchy for admin spread fees:

- `HARD_MAX_SPREAD_FEE_E6 = 200_000` (20 %) — absolute ceiling the owner cannot exceed.
- `maxAdminSpreadFeeE6` — owner-settable cap (≤ 20 %) that bounds `setPoolAdminFees`.

`setPoolAdminFees` correctly enforces this:

```solidity
// MetricOmmPoolFactory.sol:414-415
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (newAdminNotionalFeeE8 > maxAdminNotionalFeeE8) revert AdminFeeTooHigh();
``` [1](#0-0) 

But `setPoolBinAdditionalFees` performs **no cap check** at all:

```solidity
// MetricOmmPoolFactory.sol:450-457
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
``` [2](#0-1) 

The pool's `setBinAdditionalFees` also performs no cap check — only a bin-index range check:

```solidity
// MetricOmmPool.sol:464-474
function setBinAdditionalFees(int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6) external onlyFactory ... {
    if (bin < LOWEST_BIN || bin > HIGHEST_BIN) revert InvalidBinIndex(bin);
    BinState storage s = _binStates[bin];
    s.addFeeBuyE6 = addFeeBuyE6;
    s.addFeeSellE6 = addFeeSellE6;
    ...
}
``` [3](#0-2) 

During a swap the effective fee applied to a bin is:

```solidity
// MetricOmmPool.sol:541
uint256 buyFeeX64  = baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6,  ONE_X64, 1e6);
uint256 sellFeeX64 = baseFeeX64 + Math.mulDiv(binState.addFeeSellE6, ONE_X64, 1e6);
``` [4](#0-3) 

`baseFeeX64` is derived from `spreadFeeE6` (the global spread, capped at 20 %). The bin additional fee is added on top with no ceiling. At `addFeeBuyE6 = type(uint16).max = 65 535` the additional component is `65 535 / 1 000 000 ≈ 6.55 %`, yielding a total effective per-bin fee of up to **≈ 26.55 %** — well above the 20 % hard cap. [5](#0-4) 

### Impact Explanation

Traders swapping through a bin whose `addFeeBuyE6` / `addFeeSellE6` has been set to the uint16 maximum pay an effective spread fee that exceeds the protocol's hard ceiling. The excess fee accrues to LP positions in that bin (not to the protocol or admin directly), but the admin can concentrate their own liquidity in the targeted bin and drain the excess from traders. This constitutes direct loss of user principal above the fee level users can reasonably expect from the published cap system, and is an admin-boundary break: the pool admin exceeds the cap the factory is supposed to enforce.

### Likelihood Explanation

The pool admin is a semi-trusted role. The call path is a single transaction with no timelock. Any pool admin who wishes to extract additional value from traders can do so immediately by calling `setPoolBinAdditionalFees` with `addFeeBuyE6 = 65535` on the active bin. The asymmetry with `setPoolAdminFees` (which does enforce a cap) makes this an unintentional omission rather than a deliberate design choice.

### Recommendation

Add a cap check in `setPoolBinAdditionalFees` (and optionally in `setBinAdditionalFees` on the pool as a defence-in-depth layer). The cap should be consistent with `maxAdminSpreadFeeE6` so that the sum of the global spread fee and any per-bin additional fee cannot exceed the hard ceiling:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (addFeeBuyE6  > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

### Proof of Concept

1. Factory is deployed with `maxAdminSpreadFeeE6 = 200_000` (20 %) and `spreadFeeE6 = 200_000`.
2. Pool admin calls `factory.setPoolBinAdditionalFees(pool, 0, 65535, 65535)` — no revert.
3. A trader calls `pool.swap(...)` routing through bin 0.
4. Inside `_getBidAndAskPriceX64`:
   - `baseFeeX64 = f(200_000 / 1e6)` → 20 % in Q64.64
   - `buyFeeX64 = baseFeeX64 + mulDiv(65535, ONE_X64, 1e6)` → ≈ 26.55 % in Q64.64
5. The trader pays ≈ 26.55 % spread instead of the capped 20 %, with the excess retained in the bin as LP fee — extractable by the admin's own LP position. [2](#0-1) [3](#0-2) [4](#0-3)
