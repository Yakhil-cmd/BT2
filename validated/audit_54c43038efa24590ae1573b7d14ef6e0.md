### Title
Gateway Admits Signature-Invalid Invoke Transactions for Undeployed Accounts via `skip_stateful_validations` Fee-Escalation Front-Run — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` UX feature, which allows an invoke transaction with nonce=1 to bypass the `__validate__` entry-point when a matching deploy-account is pending in the mempool, can be weaponised by an unprivileged attacker. Because the gateway skips signature validation for such transactions, and because the mempool supports fee-escalation replacement, an attacker can submit an invoke transaction carrying an **arbitrary (invalid) signature** for any address whose deploy-account is visible in the mempool, pay a marginally higher tip, and atomically evict the legitimate user's invoke from the pool. The attacker's transaction is admitted by the gateway, replaces the user's transaction in the mempool, and only fails at batcher-execution time — after the user's transaction is permanently gone.

---

### Finding Description

**Relevant code path**

`extract_state_nonce_and_run_validations` calls `run_pre_validation_checks`, which in turn calls `skip_stateful_validations`:

```
extract_state_nonce_and_run_validations
  └─ run_pre_validation_checks
       ├─ validate_state_preconditions   (nonce / resource-bounds)
       ├─ validate_by_mempool            (duplicate / fee-escalation pre-check)
       └─ skip_stateful_validations      ← returns true when deploy-account is in pool
  └─ run_validate_entry_point(skip_validate=true)
       └─ execution_flags.validate = false  ← __validate__ is NOT called
``` [1](#0-0) 

The condition that triggers the skip is:

```rust
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
    return mempool_client
        .account_tx_in_pool_or_recent_block(tx.sender_address())
        .await ...
}
```

`account_tx_in_pool_or_recent_block` returns `true` whenever **any** transaction for the address is in the pool — it does not verify that the pooled transaction is specifically a deploy-account: [2](#0-1) 

When `skip_validate = true`, `run_validate_entry_point` sets `execution_flags.validate = false` and calls the blockifier's `StatefulValidator::perform_validations`. For an invoke transaction, the validator then returns immediately without calling `__validate__`: [3](#0-2) [4](#0-3) 

The mempool's fee-escalation logic allows any transaction at `(address, nonce)` to be replaced by a new one whose `tip` and `max_l2_gas_price` are each at least `fee_escalation_percentage` (default 10 %) higher: [5](#0-4) 

The production default confirms fee escalation is enabled: [6](#0-5) 

**Attack chain**

1. Victim submits `deploy_account` for address X (valid; passes `__validate_deploy__`).
2. Victim submits `invoke` (nonce=1) for address X (valid; accepted via `skip_stateful_validations` because the deploy-account is now in the pool).
3. Attacker observes the mempool and sees the deploy-account for address X.
4. Attacker crafts an `invoke` (nonce=1) for address X with **any arbitrary signature** and a tip ≥ 110 % of the victim's invoke tip.
5. Gateway evaluation of the attacker's invoke:
   - `validate_state_preconditions`: passes (nonce=1 ≤ account_nonce=0 + gap, resource bounds OK).
   - `validate_by_mempool` → `validate_fee_escalation`: passes (attacker's tip is higher).
   - `skip_stateful_validations`: `account_tx_in_pool_or_recent_block(X)` = `true` → returns `true`.
   - `run_validate_entry_point(skip_validate=true)`: `__validate__` is **not called**; returns `Ok(())`.
6. Gateway calls `add_tx` on the mempool; fee-escalation removes the victim's invoke and inserts the attacker's.
7. Batcher executes the block: deploy-account succeeds; attacker's invoke reaches `__validate__`, fails (invalid signature), and is rejected.
8. The victim's invoke is **permanently gone** from the mempool.

The attack is repeatable: every time the victim resubmits their invoke, the attacker can front-run it again with a higher tip.

---

### Impact Explanation

The gateway permanently admits a transaction whose signature has never been verified. This satisfies the **High** impact criterion: *"Mempool/gateway/RPC admission accepts invalid transactions … before sequencing."* The victim's legitimate post-deploy invoke is evicted from the mempool and must be resubmitted indefinitely, making the deploy-account + invoke UX flow unreliable under adversarial conditions. The attacker's only cost is the gas for the failed invoke (which reverts), while the victim bears repeated resubmission overhead and potential loss of time-sensitive operations.

---

### Likelihood Explanation

The mempool is observable (transactions are broadcast over P2P and visible to any node). Any unprivileged party can monitor for deploy-account transactions, compute the deterministic contract address, and submit a competing invoke with a marginally higher tip. No privileged access, special key material, or off-chain coordination is required. The attack is cheap: the attacker's invoke fails at execution and the fee is bounded by the resource bounds they set.

---

### Recommendation

1. **Verify the pooled transaction type.** `skip_stateful_validations` should confirm that the account's pooled transaction is specifically a `DeployAccount`, not merely that the account appears in the pool. The mempool should expose a dedicated `has_pending_deploy_account(address)` query.

2. **Do not skip validation for fee-escalation replacements.** If an invoke with nonce=1 is replacing an existing invoke at the same nonce (i.e., `validate_fee_escalation` returns `Some(_)`), the gateway must still run `__validate__` regardless of the deploy-account UX flag.

3. **Alternatively**, bind the skip-validation grant to the specific transaction hash of the paired deploy-account (as the `native_blockifier` path does via `deploy_account_tx_hash`), so that only the original submitter's invoke benefits from the skip. [7](#0-6)

### Proof of Concept

```
# State: address X not yet deployed (on-chain nonce = 0)

# Step 1 – victim submits deploy_account (valid signature, tip=100)
POST /gateway/add_transaction  { type: DEPLOY_ACCOUNT, salt: S, class_hash: C, sig: valid, tip: 100 }
# → accepted, X enters mempool

# Step 2 – victim submits invoke nonce=1 (valid signature, tip=100)
POST /gateway/add_transaction  { type: INVOKE, sender: X, nonce: 1, sig: valid, tip: 100 }
# → skip_stateful_validations returns true (deploy_account in pool)
# → __validate__ skipped, accepted into mempool

# Step 3 – attacker submits invoke nonce=1 (INVALID signature, tip=111 ≥ 110% of 100)
POST /gateway/add_transaction  { type: INVOKE, sender: X, nonce: 1, sig: [0xdeadbeef], tip: 111, max_l2_gas_price: 111% of victim's }
# → validate_by_mempool: fee escalation passes (111 ≥ 110)
# → skip_stateful_validations: account_tx_in_pool_or_recent_block(X) = true → skip=true
# → run_validate_entry_point: __validate__ NOT called → Ok(())
# → mempool: victim's invoke evicted, attacker's invoke inserted

# Step 4 – batcher executes block
# deploy_account(X) → success
# attacker's invoke → __validate__ called → FAILS (invalid sig) → rejected
# victim's invoke → gone from mempool
```

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}

    fn validate_fee_escalation(
        &self,
        incoming_tx_reference: TransactionReference,
    ) -> MempoolResult<Option<TransactionReference>> {
        let TransactionReference { address, nonce, .. } = incoming_tx_reference;

        self.validate_no_delayed_declare_front_run(incoming_tx_reference)?;

        if !self.config.static_config.enable_fee_escalation {
            if self.tx_pool.get_by_address_and_nonce(address, nonce).is_some() {
                return Err(MempoolError::DuplicateNonce { address, nonce });
            };

            return Ok(None);
        }

        let Some(existing_tx_reference) = self.tx_pool.get_by_address_and_nonce(address, nonce)
        else {
            // Replacement irrelevant: no existing transaction with the same nonce for address.
            return Ok(None);
        };

        if !self.should_replace_tx(&existing_tx_reference, &incoming_tx_reference) {
            info!(
                "{existing_tx_reference} was not replaced by {incoming_tx_reference} due to \
                 insufficient fee escalation."
            );
            // TODO(Elin): consider adding a more specific error type / message.
            return Err(MempoolError::DuplicateNonce { address, nonce });
        }

        Ok(Some(existing_tx_reference))
    }

    pub fn should_run_stateful_validations(
        &mut self,
        account_tx: &AccountTransaction,
        deploy_account_tx_hash: Option<TransactionHash>,
    ) -> StatefulValidatorResult<bool> {
        if account_tx.tx_type() != TransactionType::InvokeFunction {
            return Ok(true);
        }
        let tx_info = account_tx.create_tx_info();
        let nonce = self.stateful_validator.get_nonce(tx_info.sender_address())?;

        let deploy_account_not_processed =
            deploy_account_tx_hash.is_some() && nonce == Nonce(Felt::ZERO);
        let tx_nonce = tx_info.nonce();
        let is_post_deploy_nonce = Nonce(Felt::ONE) <= tx_nonce;
        let nonce_small_enough_to_qualify_for_validation_skip =
            tx_nonce <= self.max_nonce_for_validation_skip;

        let skip_validate = deploy_account_not_processed
            && is_post_deploy_nonce
            && nonce_small_enough_to_qualify_for_validation_skip;

        Ok(!skip_validate)
    }
