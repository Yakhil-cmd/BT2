### Title
Gateway Admits Invoke Transactions with Invalid Signatures via `skip_stateful_validations` UX Bypass — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the gateway's stateful validation path bypasses the `__validate__` entry point (account signature verification) for invoke transactions with nonce=1 when the account has **any** transaction in the mempool pool. An attacker who observes a victim's `deploy_account` transaction in the mempool can submit an invoke with an invalid signature for that account, and the gateway will admit it without signature verification. The victim's legitimate invoke is then blocked by a `DuplicateNonce` error.

---

### Finding Description

**The validation skip path:**

`extract_state_nonce_and_run_validations` calls `run_pre_validation_checks`, which calls `skip_stateful_validations`: [1](#0-0) 

`skip_stateful_validations` returns `true` when all four conditions hold: [2](#0-1) 

When `skip_validate=true`, `run_validate_entry_point` sets `validate: false` in `ExecutionFlags`: [3](#0-2) 

Inside `StatefulValidator::perform_validations`, when `validate=false`, `perform_pre_validation_stage` runs (nonce, fee bounds, balance checks) but the function returns early **before** calling `__validate__`: [4](#0-3) 

The `StatefulValidatorTrait::validate` method delegates to `perform_validations`: [5](#0-4) 

**The broken invariant in `account_tx_in_pool_or_recent_block`:**

The comment in `skip_stateful_validations` claims: *"It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."*

This is false. `account_tx_in_pool_or_recent_block` returns `true` if the account has **any** transaction in the pool: [6](#0-5) 

The mempool's `validate_tx` (called before `skip_stateful_validations` in `run_pre_validation_checks`) only checks for duplicate transaction hashes and nonce ordering — it does **not** verify signatures: [7](#0-6) 

`MempoolState::validate_incoming_tx` only rejects if `tx_nonce < account_nonce`: [8](#0-7) 

**Attack chain:**

1. Victim pre-funds address X and submits `deploy_account` (nonce=0) → enters mempool pool.
2. Attacker submits `invoke` for address X with nonce=1 and an **invalid signature**.
3. `validate_by_mempool`: nonce=1 ≥ account_nonce=0, no duplicate hash → passes.
4. `skip_stateful_validations`: `account_tx_in_pool_or_recent_block` returns `true` (victim's `deploy_account` is in pool) → returns `true`.
5. `run_validate_entry_point` with `validate=false` → `__validate__` **not called**.
6. Gateway accepts the attacker's invalid invoke and adds it to the mempool.
7. Victim submits `invoke` for address X with nonce=1 and valid signature.
8. Mempool `validate_fee_escalation` returns `DuplicateNonce` → victim's invoke rejected (unless victim pays a strictly higher fee for fee escalation).
9. Attacker's invalid invoke fails during execution (revert) because `__validate__` is called during block execution and the signature is invalid.

The `run_pre_validation_checks` call order confirms `validate_by_mempool` runs before `skip_stateful_validations`: [9](#0-8) 

---

### Impact Explanation

The gateway admits an invoke transaction with an invalid (attacker-controlled) signature for an undeployed account. This violates the core admission invariant that all accepted transactions must carry a valid account signature. Concrete consequences:

- The victim's first invoke after deployment is blocked or requires fee escalation.
- The attacker's reverted transaction wastes block space.
- If the account is funded, the reverted transaction charges fee to the victim.

This matches: **High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

---

### Likelihood Explanation

Prerequisites are low-friction:
- The victim's `deploy_account` is observable in the public mempool.
- The attacker only needs to submit before the victim's invoke (a race that is trivially won by monitoring the mempool).
- The victim's account must be pre-funded, which is a standard requirement for deployment.

No privileged access is required. Any unprivileged actor can execute this attack against any newly deploying account.

---

### Recommendation

`skip_stateful_validations` must verify that the account has a **`deploy_account` transaction specifically** in the mempool, not just any transaction. The current `account_tx_in_pool_or_recent_block` check is too broad — it returns `true` for any transaction type from that address.

A targeted fix: expose a `deploy_account_in_pool(address)` query on the mempool that returns `true` only when a `DeployAccount` transaction for that address is present. Replace the `account_tx_in_pool_or_recent_block` call in `skip_stateful_validations` with this narrower check.

---

### Proof of Concept

```
1. Alice pre-funds address X with STRK.
2. Alice submits deploy_account(class_hash, salt, ...) for address X → mempool pool.
3. Bob submits invoke(sender=X, nonce=1, signature=[0xdead, 0xbeef]).
   - validate_by_mempool: nonce 1 >= account_nonce 0, no dup hash → OK.
   - skip_stateful_validations:
       tx.nonce() == 1 ✓
       account_nonce == 0 ✓
       account_tx_in_pool_or_recent_block(X) == true (Alice's deploy_account) ✓
     → returns true (skip).
   - run_validate_entry_point: ExecutionFlags { validate: false, ... }
     → perform_pre_validation_stage passes (balance OK, fee bounds OK).
     → early return, __validate__ never called.
   - Gateway: Ok(account_nonce) → Bob's invalid invoke admitted to mempool.
4. Alice submits invoke(sender=X, nonce=1, signature=[valid]).
   - validate_fee_escalation: nonce=1 already in pool for address X
     → DuplicateNonce error → Alice's invoke rejected.
5. Block execution:
   - deploy_account(X) executes → X deployed.
   - Bob's invoke executes → __validate__ called → signature invalid → revert.
   - Alice's invoke never executes (blocked at gateway).
```

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
