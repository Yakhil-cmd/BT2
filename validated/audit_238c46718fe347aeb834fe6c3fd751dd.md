### Title
Unbounded Loop in `calculate_rewards` Enables Permanent Freezing of Pool Member Unclaimed Yield - (File: `src/pool/pool.cairo`)

---

### Summary

The `calculate_rewards` internal function in `Pool` iterates over every entry in a pool member's `pool_member_epoch_balance` trace without any cap. The codebase itself acknowledges this: *"This loop is unbounded but unlikely to exceed gas limits."* A delegator can deliberately grow their own trace to an arbitrary length by repeatedly changing their delegation balance across many epochs. Once the trace is large enough, any call to `claim_rewards` for that pool member will exceed the Starknet block gas limit, permanently bricking their ability to claim accumulated yield.

---

### Finding Description

`calculate_rewards` in `src/pool/pool.cairo` (lines 837–888) iterates over the entire `pool_member_epoch_balance` trace for a given pool member:

```cairo
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
    if pool_member_checkpoint.epoch() >= until_epoch {
        break;
    }
    ...
    entry_to_claim_from += 1;
}
``` [1](#0-0) 

The trace grows by one entry every time the pool member changes their balance in a new epoch (delegate more, partially exit, switch pool, etc.). There is no cap on how many entries can accumulate.

`calculate_rewards` is called from the pool member's `claim_rewards` flow. The `entry_to_claim_from` cursor is stored in the pool member's checkpoint and advances across claims, but it only advances up to `until_epoch` (the current epoch). If the delegator has accumulated N balance-change entries across N past epochs, the loop must traverse all N entries in a single transaction. [2](#0-1) 

---

### Impact Explanation

**High — Permanent freezing of unclaimed yield.**

Once the trace length is large enough that iterating it in a single transaction exceeds the Starknet execution gas limit, every future `claim_rewards` call for that pool member will revert. The pool member's accumulated unclaimed yield becomes permanently inaccessible. There is no alternative code path to claim rewards in smaller batches, and no administrative escape hatch to clear or truncate the trace.

---

### Likelihood Explanation

**Medium.** The attack requires the delegator to perform many balance-change transactions across many epochs, each costing gas. However:

- The cost is paid by the attacker on themselves (self-griefing to lock their own funds, or a griefing attack on a victim by front-running their delegation entries — though the latter is harder).
- More practically, a legitimate long-term delegator who changes their balance frequently (e.g., a bot that rebalances every epoch) can organically reach a trace length that causes their own `claim_rewards` to run out of gas, with no malicious intent required.
- Starknet's per-transaction gas limit is finite and well-defined; the threshold is reachable.

---

### Recommendation

Introduce a maximum number of trace entries processed per `claim_rewards` call (a "pagination" or "batch size" cap), and persist the `entry_to_claim_from` cursor so that subsequent calls resume where the previous one left off. This is the standard pattern for unbounded-loop mitigation in on-chain reward accounting. Alternatively, enforce a protocol-level cap on how many balance-change entries a single pool member may accumulate (e.g., by requiring a minimum epoch gap between balance changes).

---

### Proof of Concept

1. Delegator `A` joins a pool and delegates the minimum amount.
2. Each epoch, `A` calls `add_to_delegation_pool` with 1 wei, then immediately calls `exit_delegation_pool_intent` for 1 wei. Each such pair in a distinct epoch appends one entry to `pool_member_epoch_balance`.
3. After `N` epochs of this, `pool_member_trace_length` for `A` equals `N`.
4. `A` (or anyone) calls `claim_rewards` for `A`. The transaction must execute the `while` loop `N` times, each iteration reading storage and calling `find_sigma`.
5. At a sufficiently large `N` (determined by Starknet's gas limit per transaction), the call reverts with an out-of-gas error every time.
6. `A`'s unclaimed yield — accumulated over all those epochs — is permanently frozen.

The loop bound is `pool_member_trace_length` with no guard: [3](#0-2)
