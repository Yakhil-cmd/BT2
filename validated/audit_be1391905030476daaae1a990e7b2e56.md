### Title
Silo Whitelist Access Control Bypassed via `call` and `deploy_code` — (`engine/src/contract_methods/evm_transactions.rs`)

### Summary

Aurora Engine's Silo mode enforces per-account and per-address whitelists to restrict who may submit EVM transactions or deploy EVM contracts. These checks are applied exclusively inside `submit_with_alt_modexp` via `assert_access`. The sibling NEAR contract methods `call` and `deploy_code` perform no equivalent whitelist check, allowing any NEAR account — including those explicitly excluded from the silo whitelists — to execute EVM calls and deploy contracts, bypassing the intended access control entirely.

### Finding Description

**Root cause — missing whitelist check in `call` and `deploy_code`**

The silo whitelist enforcement lives in a single function:

```rust
// engine/src/engine.rs
fn assert_access<I: IO + Copy, E: Env>(
    io: &I,
    env: &E,
    transaction: &NormalizedEthTransaction,
) -> Result<(), EngineError> {
    let allowed = if transaction.to.is_some() {
        silo::is_allow_submit(io, &env.predecessor_account_id(), &transaction.address)
    } else {
        silo::is_allow_deploy(io, &env.predecessor_account_id(), &transaction.address)
    };
    if !allowed {
        return Err(EngineError { kind: EngineErrorKind::NotAllowed, gas_used: 0 });
    }
    Ok(())
}
``` [1](#0-0) 

This is called only from `submit_with_alt_modexp`: [2](#0-1) 

The `call` method in `evm_transactions.rs` skips this check entirely — it only calls `require_running` and then directly invokes `engine.call_with_args`: [3](#0-2) 

Likewise, `deploy_code` only calls `require_running` before invoking `engine.deploy_code_with_input`: [4](#0-3) 

**What the whitelist is supposed to enforce**

`is_allow_submit` requires both the NEAR predecessor account to be in the `Account` whitelist and the EVM sender address to be in the `Address` whitelist. `is_allow_deploy` requires the `Admin` and `EvmAdmin` whitelists: [5](#0-4) 

Whitelists are disabled by default and must be explicitly enabled. When enabled, `submit` correctly rejects non-whitelisted callers: [6](#0-5) 

**The bypass**

Because `call` and `deploy_code` are public NEAR contract methods with no whitelist guard, any NEAR account — regardless of whether it appears in the `Account`, `Address`, `Admin`, or `EvmAdmin` whitelists — can:

1. Call `call` to execute arbitrary EVM transactions as their NEAR-account-derived EVM address.
2. Call `deploy_code` to deploy arbitrary EVM bytecode, bypassing the `Admin`/`EvmAdmin` deployment restriction.

The EVM sender used in both cases is `predecessor_address(&env.predecessor_account_id())`, which deterministically maps the NEAR account to an EVM address — so the attacker controls a real EVM identity with any funds it holds.

### Impact Explanation

**Critical — Direct theft of user funds / whitelist bypass**

In a Silo deployment where the operator has enabled whitelists to create a permissioned EVM environment:

- A non-whitelisted NEAR account can call `call` to invoke any EVM contract (e.g., an ERC-20 `transfer`) as their mapped EVM address, draining any token balance held at that address.
- A non-whitelisted NEAR account can call `deploy_code` to deploy arbitrary contracts, circumventing the `Admin`/`EvmAdmin` deployment gate entirely.

The whitelist system's security guarantee — that only approved accounts and addresses may interact with the silo EVM — is completely voided.

### Likelihood Explanation

Silo mode with whitelists enabled is an explicitly supported production configuration. The `call` and `deploy_code` methods are standard public NEAR contract entry points reachable by any NEAR account with no special privilege. Any actor who knows their NEAR account maps to an EVM address with a balance, or who simply wants to deploy a contract in a restricted silo, can exploit this with a single NEAR transaction. No leaked keys, governance capture, or social engineering is required.

### Recommendation

Add the silo whitelist check to both `call` and `deploy_code` before executing EVM logic. For `call`, apply `is_allow_submit` using the predecessor account and the target/sender address. For `deploy_code`, apply `is_allow_deploy`. The check should mirror the logic already present in `assert_access`:

```rust
// In deploy_code and call, after require_running:
if !silo::is_allow_deploy(&io, &env.predecessor_account_id(), &sender_address) {
    return Err(errors::ERR_NOT_ALLOWED.into());
}
```

### Proof of Concept

1. Operator deploys Aurora Engine in Silo mode and enables all four whitelists (`Admin`, `EvmAdmin`, `Account`, `Address`).
2. Attacker's NEAR account `attacker.near` is **not** added to any whitelist.
3. Attacker calls `submit` with a signed EVM transaction → rejected with `NotAllowed` (whitelist enforced).
4. Attacker calls `call` with `CallArgs { contract_address: erc20_token, input: transfer(victim, amount) }` → **succeeds**, transferring tokens from the attacker's EVM address.
5. Attacker calls `deploy_code` with arbitrary bytecode → **succeeds**, deploying a contract despite not being in the `Admin`/`EvmAdmin` whitelist.

Steps 4 and 5 succeed because neither `call` nor `deploy_code` invokes `assert_access`. [3](#0-2) [4](#0-3) [1](#0-0) [7](#0-6)

pub fn call<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<SubmitResult, ContractError> {
    with_logs_hashchain(io, env, function_name!(), |mut io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        let bytes = io.read_input().to_vec();
        let args = CallArgs::deserialize(&bytes).ok_or(errors::ERR_BORSH_DESERIALIZE)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();

        let mut engine: Engine<_, E, AuroraModExp> = Engine::new_with_state(
            state,
            predecessor_address(&predecessor_account_id),
            current_account_id,
            io,
            env,
        );
        let result = engine.call_with_args(args, handler)?;
        let result_bytes = borsh::to_vec(&result).map_err(|_| errors::ERR_SERIALIZE)?;
        io.return_output(&result_bytes);
        Ok(result)
    })
}

pub fn is_allow_submit<I: IO + Copy>(io: &I, account: &AccountId, address: &Address) -> bool {
    is_address_allowed(io, address) && is_account_allowed(io, account)
}

/// Check if a user has the right to receive erc20 tokens.
pub fn is_allow_receive_erc20_tokens<I: IO + Copy>(io: &I, address: &Address) -> bool {
    is_address_allowed(io, address)
}

fn is_account_allowed_deploy<I: IO + Copy>(io: &I, account_id: &AccountId) -> bool {
    let list = Whitelist::init(io, WhitelistKind::Admin);
    !list.is_enabled() || list.is_exist(account_id)
}

fn is_address_allowed_deploy<I: IO + Copy>(io: &I, address: &Address) -> bool {
    let list = Whitelist::init(io, WhitelistKind::EvmAdmin);
    !list.is_enabled() || list.is_exist(address)
}

fn is_address_allowed<I: IO + Copy>(io: &I, address: &Address) -> bool {
