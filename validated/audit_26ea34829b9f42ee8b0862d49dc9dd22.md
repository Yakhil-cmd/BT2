### Title
Invoke Transaction with Invalid Signature Admitted to Mempool via `skip_stateful_validations` Bypass — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` UX feature in the gateway's stateful validator allows an invoke transaction with an **invalid or arbitrary signature** to be admitted to the mempool without the account's `__validate__` entry point being executed. An unprivileged attacker can trigger this by first placing any transaction from the target account address into the mempool (e.g., a deploy_account), then submitting an invoke with nonce=1 and a forged signature. The gateway accepts the invoke without signature verification, violating the invariant that all mempool-admitted transactions have passed account validation.

### Finding Description

`skip_stateful_validations` in `crates/apollo_gateway/src/stateful_transaction_validator.rs` returns `true` (skip) when three conditions hold simultaneously:

1. The transaction is an `Invoke` with `nonce == Nonce(Felt::ONE)`.
2. The on-chain account nonce is `Nonce(Felt::ZERO)` (account not yet deployed).
3. `mempool_client.account_tx_in_pool_or_recent_block(sender_address)` returns `true`. [1](#0-0) 

When `skip_validate = true`, `run_validate_entry_point` sets `execution_flags.validate = false` and calls the blockifier's `StatefulValidator::perform_validations`, which returns `Ok(())` immediately for Invoke transactions without calling `__validate__`: [2](#0-1) [3](#0-2) 

The `account_tx_in_pool_or_recent_block` check is satisfied by **any** transaction from the sender address in the pool — not specifically a deploy_account: [4](#0-3) 

The `validate_by_mempool` call that precedes `skip_stateful_validations` only checks for duplicate transactions and nonce ordering; it does not verify the account's cryptographic signature: [5](#0-4) 

### Impact Explanation

An invoke transaction carrying an invalid or attacker-controlled signature is admitted to the mempool without the account's `__validate__` entry point being executed. This breaks the admission invariant: every transaction in the mempool should have passed account-level signature verification. The mempool is polluted with transactions that will fail (or revert) at batcher execution time. If the account has no balance after deployment, the batcher cannot recover fees for the failed execution, enabling cost-free mempool spam. This matches the **High** impact category: *Mempool/gateway/RPC admission accepts invalid transactions before sequencing*.

### Likelihood Explanation

The attack is fully unprivileged and requires only two sequential gateway submissions:

1. A `deploy_account` for address A (with a valid `__validate_deploy__` signature — the attacker controls the deployer key and can choose any class with a permissive validator, or use an already-declared class).
2. An `invoke` with `nonce=1` from A carrying an arbitrary/invalid signature.

The default `max_allowed_nonce_gap` configuration permits `nonce=1` when `account_nonce=0`, as confirmed by the existing test suite: [6](#0-5) 

No privileged access is required. The feature is reachable from the public RPC endpoint.

### Recommendation

Replace the coarse `account_tx_in_pool_or_recent_block` check with a check that specifically confirms a **deploy_account** transaction (not just any transaction) is present in the mempool for the sender address. Alternatively, add a separate mempool query `deploy_account_tx_in_pool(sender_address)` that returns `true` only when the pending transaction is of type `DeployAccount`. This preserves the UX intent (deploy + invoke in one batch) while preventing an attacker from triggering the skip with an unrelated transaction type.

Additionally, consider adding a comment or assertion that the `skip_validate` path is only reachable when a deploy_account is the pending transaction, and add a test case that verifies an invoke with an invalid signature is **rejected** when the only mempool entry for the account is a non-deploy-account transaction.

### Proof of Concept

```
1. Attacker declares (or reuses) a contract class C whose __validate_deploy__ always returns "VALID".

2. Attacker computes address A = deploy_account_address(class_hash=C, salt=S, constructor_calldata=[]).

3. Attacker submits deploy_account(class_hash=C, salt=S, nonce=0) signed with attacker key.
   → Gateway runs __validate_deploy__ via full execution; passes. A is now in the mempool.

4. Attacker submits invoke(sender=A, nonce=1, calldata=<arbitrary>, signature=<garbage>).
   → Gateway calls skip_stateful_validations:
       tx.nonce() == 1  ✓
       account_nonce == 0  ✓
       account_tx_in_pool_or_recent_block(A) == true  ✓  (deploy_account is in pool)
   → skip_validate = true → __validate__ is NOT called.
   → validate_by_mempool passes (no signature check).
   → Invoke is admitted to the mempool with an invalid signature.

5. Batcher retrieves both transactions. Executes deploy_account (A deployed, no balance).
   Executes invoke: __validate__ is called, fails (invalid signature).
   Transaction reverts; no fee charged (zero balance).

Result: Invalid invoke admitted to mempool and executed by batcher at zero cost to attacker.
``` [7](#0-6) [1](#0-0)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
