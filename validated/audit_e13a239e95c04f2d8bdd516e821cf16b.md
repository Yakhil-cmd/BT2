Audit Report

## Title
Byzantine Peer Permanent 429 Loop Causes Unbounded State Sync Stall — (`rs/p2p/state_sync_manager/src/ongoing.rs`)

## Summary
A Byzantine replica peer that always responds with HTTP 429 (`TOO_MANY_REQUESTS`) to chunk requests can hold a victim node's state sync in an infinite retry loop. `DownloadChunkError::Overloaded` does not evict the peer from `active_downloads`, and `ChunksToDownload::download_failed` unconditionally re-queues the chunk with no retry limit, backoff, or counter. The sync loops indefinitely at full concurrency (10 slots, 10 s per-chunk timeout), never making progress and never self-terminating without external cancellation.

## Finding Description
The root cause is the asymmetric treatment of transient vs. permanent errors in `handle_downloaded_chunk_result` (`ongoing.rs` L185–223). `NoContent` and `RequestError` both call `active_downloads.remove(&peer_id)` and decrement `allowed_downloads`, eventually draining `active_downloads` and breaking the loop. `Overloaded`, `Timeout`, and `Cancelled` do neither — they only call `self.chunks_to_download.download_failed(chunk_id)`, which is a pure unconditional push with no counter or limit (`chunks_to_download.rs` L26–28).

`parse_chunk_handler_response` maps HTTP 429 → `DownloadChunkError::Overloaded` (`chunk.rs` L129). `spawn_chunk_downloads` builds a `WeightedIndex` over `active_downloads` and will always select the sole remaining Byzantine peer (`ongoing.rs` L249–263). The loop in `OngoingStateSync::run` exits only on `cancellation.cancelled()` or `active_downloads.is_empty()` (`ongoing.rs` L116–176); neither condition is ever reached when the Byzantine peer is the only entry and always returns 429.

The external escape via `cancel_if_running` in `StateSyncManager::handle_advert` (`lib.rs` L159–163) fires only when a new advert arrives and the `StateSyncClient` decides the sync should be cancelled. If the node is still behind, `cancel_if_running` returns `false` and the stalled sync continues. There is no global wall-clock timeout for the entire `OngoingStateSync`.

## Impact Explanation
The victim replica node cannot complete state sync, cannot advance its certified state height, and therefore cannot participate in consensus. The node is effectively partitioned from the subnet for as long as the Byzantine peer remains its sole `active_downloads` entry. This is a targeted, persistent liveness denial-of-service against a specific replica node — matching the allowed impact class: **Application/platform-level DoS, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS** — **High ($2,000–$10,000)**.

## Likelihood Explanation
The scenario requires a single Byzantine replica node (well below the f < n/3 fault threshold) to be the sole entry in the victim's `active_downloads`. This is reachable via two realistic paths: (1) the Byzantine peer is the first or only peer to advertise the state to the victim, and honest peers' adverts are delayed or dropped (unreliable broadcast is explicitly noted in `lib.rs` L229–230); (2) honest peers are present initially but get evicted because they return `NoContent` (state already pruned) or `RequestError`, leaving only the Byzantine peer. Both paths require only a single Byzantine node and no victim mistakes.

## Recommendation
- **Evict after N consecutive `Overloaded`/`Timeout` responses**: maintain a per-peer failure counter in `active_downloads` (or a parallel map); remove the peer after a configurable threshold (e.g., 20 consecutive transient failures).
- **Global state sync deadline**: add a wall-clock timeout for the entire `OngoingStateSync` (e.g., tied to the CUP interval), after which the sync is cancelled and restarted with fresh peer selection.
- **Exponential backoff on `Overloaded`**: delay re-scheduling chunks from an overloaded peer instead of immediately re-queuing, which also limits the tight busy-loop and reduces wasted bandwidth.

## Proof of Concept
The existing test `test_cancel_if_running` (`ongoing.rs` L429–461) already demonstrates the exact scenario: a `MockTransport` that always returns `StatusCode::TOO_MANY_REQUESTS`, a single chunk to download, and a single peer (`NODE_1`). The test shows the sync only terminates via external `ongoing.shutdown.shutdown()`. A modified version of this test that asserts termination *without* calling `shutdown()` within a bounded wall-clock window (e.g., `tokio::time::timeout(Duration::from_secs(60), ...)`) would hang and fail, confirming the bug. No mainnet interaction is required.
