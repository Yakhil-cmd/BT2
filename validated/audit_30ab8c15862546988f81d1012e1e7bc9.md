### Title
Watermark decay rounds to zero in `_decayed` when swaps are frequent, permanently suppressing stop-loss recovery — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

---

### Summary

`OracleValueStopLossExtension._decayed` computes `hwm - (hwm * factor) / E8` where `factor = ratePerSecondE8 * dt`. When `hwm * factor < E8`, integer division truncates the decay to zero and the watermark is returned unchanged. In actively traded pools where swaps occur every 1–2 seconds, this means the high watermark **never decays**, keeping the stop-loss floor permanently elevated and causing false-positive stop-loss triggers that block legitimate swaps indefinitely.

---

### Finding Description

The decay function is:

```solidity
// metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol L319-324
function _decayed(uint256 hwm, uint256 ratePerSecondE8, uint256 dt) private pure returns (uint256) {
    if (ratePerSecondE8 == 0 || dt == 0 || hwm == 0) return hwm;
    uint256 factor = ratePerSecondE8 * dt;
    if (factor >= E8) return 0;
    return hwm - (hwm * factor) / E8;
}
```

The decay amount `(hwm * factor) / E8` truncates to zero whenever `hwm * ratePerSecondE8 * dt < E8 (1e8)`.

The code comment documents `ratePerSecondE8 = 58` as the canonical "~5%/day" setting. With `METRIC_SCALE = 1e6`, a typical per-share watermark is on the order of `1e6`. Plugging in:

```
factor = 58 * 1 = 58          (dt = 1 second)
(1e6 * 58) / 1e8 = 0.58 → 0  (truncated)
```

No decay occurs. The minimum `dt` for any decay to register is `ceil(E8 / (hwm * ratePerSecondE8)) = ceil(1e8 / 5.8e7) = 2` seconds.

`_checkAndUpdateWatermarks` is called on every **successful** swap via `afterSwap`, and it writes `hwmS.lastDecayTs = uint32(block.timestamp)` on each non-reverting call. In an actively traded pool, `dt` is always 1–2 seconds, so the decay is always 0 and the watermark is never reduced. [1](#0-0) 

The watermark update path: [2](#0-1) 

`lastDecayTs` is reset to `block.timestamp` on every successful swap, so `dt` is always the inter-swap interval — exactly the small value that causes the rounding to zero.

---

### Impact Explanation

The stop-loss breach check is:

```solidity
breached = metric < (hwm * floorMultiplier) / E6;
``` [3](#0-2) 

If `hwm` never decays, the floor `(hwm * floorMultiplier) / E6` never lowers. The intended design (documented in the contract NatSpec) is that the watermark decays over time so the pool can recover from temporary drawdowns without permanently blocking swaps. With the rounding bug, any pool value drop that exceeds `drawdownE6` against the undecayed peak watermark will trigger the stop-loss and block swaps in the affected direction — indefinitely, as long as the pool remains actively traded. Swaps are an unusable flow for the affected direction. [4](#0-3) 

---

### Likelihood Explanation

No privileged access is required. Normal swap activity in any pool with `OracleValueStopLossExtension` and swap frequency above one swap per `ceil(E8 / (hwm * ratePerSecondE8))` seconds triggers the bug. For the documented canonical rate of 58 E8/s and typical per-share metrics near `METRIC_SCALE = 1e6`, the threshold is ~2 seconds — well within normal DeFi trading cadence. Any high-frequency pool using this extension is affected by default. [5](#0-4) 

---

### Recommendation

Reorder the multiplication before division (the same class of fix applied in the referenced external bug) to avoid premature truncation:

```diff
- return hwm - (hwm * factor) / E8;
+ uint256 decay = Math.mulDiv(hwm, factor, E8);
+ return decay >= hwm ? 0 : hwm - decay;
```

`Math.mulDiv` still truncates for very small products, so a fully correct fix should either enforce a minimum decay of 1 unit when `factor > 0 && hwm > 0`, or track fractional decay in a higher-precision accumulator across calls.

### Proof of Concept

```
Setup:
  ratePerSecondE8 = 58  (~5%/day, the documented canonical value)
  hwm             = 1e6 (METRIC_SCALE, typical per-share metric)
  dt              = 1 second (swap every block)

Decay calculation:
  factor = 58 * 1 = 58
  (1e6 * 58) / 1e8 = 0  ← truncated to zero

After 86,400 seconds (1 day) of 1-second swaps:
  Expected watermark decay: 5% of 1e6 = 50,000 units
  Actual watermark decay:   0 units

Result:
  Stop-loss floor stays at 1e6 * (1e6 - drawdownE6) / 1e6 = 950,000
  instead of decaying to ~902,500 after one day.
  Any pool value drop to 940,000 (6% from peak) triggers the stop-loss
  and permanently blocks swaps in the affected direction,
  even though the intended 5%/day decay should have re-armed the pool.
```
