Audit Report

## Title
`get_allowances` Break Condition Ignores Subaccount, Leaking Sibling Subaccount Allowances — (`rs/ledger_suite/icrc1/ledger/src/lib.rs`)

## Summary
The `get_allowances` function in the ICRC1 ledger iterates a `StableBTreeMap` keyed by `(account, spender)` and terminates only when `account_spender.account.owner != from.owner`. Because the map is ordered by the full key `(owner, subaccount, spender)`, entries for sibling subaccounts of the same owner sort contiguously and are never stopped by this guard. A caller requesting allowances for `(owner=X, subaccount=S1)` receives all allowances for every subaccount of `X`, disclosing subaccount existence, approved spenders, amounts, and expiry times.

## Finding Description
In `rs/ledger_suite/icrc1/ledger/src/lib.rs`, `get_allowances` (lines 1174–1223) opens a range scan on `ALLOWANCES_MEMORY` starting from `start_account_spender` which is constructed with the full `from` account (owner + subaccount). The only "wrong account" termination guard is at line 1204:

```rust
if account_spender.account.owner != from.owner {
    break;
}
```

`Account` is `{ owner: Principal, subaccount: Option<[u8;32]> }`. The `StableBTreeMap` orders keys by their `Storable` byte encoding, which encodes the full account including subaccount. Entries for `(owner=X, subaccount=[2u8;32])` sort immediately after `(owner=X, subaccount=[1u8;32])` — the owner field is identical, so the break never fires. All sibling-subaccount entries are appended to `result` and returned to the caller.

The `GetAllowancesArgs` type confirms `from_account` is a full `Account` (owner + optional subaccount), making the intent clear that filtering should be by the complete account identity, not just the owner principal.

## Impact Explanation
Any unprivileged caller can enumerate the complete allowance map for every subaccount of any target principal `X` via a single `icrc103_get_allowances` query, paginating through results. This discloses: which subaccounts of `X` hold active approvals, which spenders are approved, approved token amounts, and expiry timestamps — information that is not otherwise enumerable without knowing both the subaccount and spender in advance. This constitutes a significant privacy and information-disclosure impact on the ICRC ledger, a core in-scope financial integration component, with concrete user harm (targeted exploitation of high-value approvals that would otherwise be unknown). This fits the High impact category: significant ledger security impact with concrete user harm.

## Likelihood Explanation
The `icrc103_get_allowances` endpoint is a public query callable by any unauthenticated principal with no rate limiting beyond `MAX_TAKE_ALLOWANCES = 500` per call. The exploit requires a single canister query call with a known target principal. No special privileges, victim interaction, or social engineering is required. It is trivially and repeatably exploitable.

## Recommendation
Change the break condition at line 1204 from:
```rust
if account_spender.account.owner != from.owner {
    break;
}
```
to:
```rust
if account_spender.account != from {
    break;
}
```
This ensures iteration stops as soon as the full account (owner + subaccount) no longer matches the requested `from` account.

## Proof of Concept
State-machine test:
1. Fund subaccounts `(X, [1u8;32])` and `(X, [2u8;32])`.
2. `icrc2_approve(from={owner:X, subaccount:Some([1u8;32])}, spender:A, amount:100)`.
3. `icrc2_approve(from={owner:X, subaccount:Some([2u8;32])}, spender:B, amount:200)`.
4. Call `icrc103_get_allowances({from_account:{owner:X, subaccount:Some([1u8;32])}, prev_spender:None, take:None})`.
5. Assert result contains **only** the allowance `(X,[1u8;32]) → A:100`.
6. Observe: result **also** contains `(X,[2u8;32]) → B:200`, violating the subaccount boundary invariant.
