### Title
Permanently Disabled EOA Signature Validation in Forward Run Creates Forward/Proving Divergence - (File: `forward_system/src/run/mod.rs`)

### Summary
The production sequencer block-execution entry point `run_block` is hardwired to `BasicBootloaderForwardSimulationConfig`, which permanently sets `VALIDATE_EOA_SIGNATURE = false`. Because the proving run uses `BasicBootloaderProvingExecutionConfig` with `VALIDATE_EOA_SIGNATURE = true`, a transaction carrying an invalid ECDSA signature is silently accepted by the forward run but rejected by the prover, creating an unprovable block and a liveness-breaking forward/proving divergence.

### Finding Description

`BasicBootloaderForwardSimulationConfig` is defined with both security-relevant flags set to values that disable signature checking: [1](#0-0) 

`VALIDATE_EOA_SIGNATURE = false` and `SIMULATION = false`. The guard in both the Ethereum and ZK validation paths is:

```rust
if !Config::VALIDATE_EOA_SIGNATURE | Config::SIMULATION {
    // skip – no ecrecover performed
} else {
    // validate signature
}
``` [2](#0-1) [3](#0-2) 

For `BasicBootloaderForwardSimulationConfig`, `!false | false = true`, so the entire ecrecover block is skipped. The `from` address supplied by the sequencer is accepted without cryptographic verification.

The public production entry point `run_block` — exposed via `RunBlockForward` and called by the sequencer — is hardcoded to this config: [4](#0-3) 

There is no runtime mechanism to flip `VALIDATE_EOA_SIGNATURE` to `true` for the forward run; it is a compile-time constant permanently fixed at `false`.

The proving run uses the opposite config: [5](#0-4) 

`VALIDATE_EOA_SIGNATURE = true`, so the prover **does** perform ecrecover and will reject any transaction whose recovered address does not match `from`.

### Impact Explanation

An attacker submits a transaction whose `from` field is set to an arbitrary victim address but whose `(v, r, s)` signature does not recover to that address. The forward run accepts the transaction (no ecrecover), executes it, and commits state changes (nonce increment, balance debit, arbitrary call). When the prover attempts to prove the same block with `BasicBootloaderProvingExecutionConfig`, it runs ecrecover, finds a mismatch, and returns `InvalidTransaction::IncorrectFrom`, producing a different state transition. The block is unprovable. Repeated submission of such transactions causes persistent proof-generation failure — a liveness-breaking denial of service against the ZK rollup's settlement layer.

### Likelihood Explanation

Any unprivileged user can craft a transaction with a valid structure but an invalid signature (e.g., set `r = s = 1`, `v = 27`, `from = victim`). The sequencer's `run_block` path performs no signature check at the bootloader level. Unless the sequencer independently validates signatures before calling `run_block` (which is not enforced by the `RunBlockForward` API), such transactions reach block inclusion. The `RunBlockForward` interface exposes no config parameter to callers: [6](#0-5) 

### Recommendation

Replace `BasicBootloaderForwardSimulationConfig` with `BasicBootloaderProvingExecutionConfig` (or a new config with `VALIDATE_EOA_SIGNATURE = true, SIMULATION = false`) in the production `run_block` path, so that the forward run and proving run apply identical signature-validation logic. The "optimization" comment acknowledges the flag can be disabled, but the production sequencer path should not rely on an out-of-band guarantee that callers pre-validate signatures.

### Proof of Concept

1. Craft a ZKsync/Ethereum transaction: set `from = 0xVICTIM`, set `(v, r, s)` to values that do **not** recover to `0xVICTIM` (e.g., `r = s = 1`).
2. Submit to the sequencer. The sequencer calls `run_block` → `run_forward::<BasicBootloaderForwardSimulationConfig>`.
3. In `validation_impl.rs`, `!Config::VALIDATE_EOA_SIGNATURE | Config::SIMULATION` = `true`; the ecrecover block is skipped entirely. The transaction passes validation, nonce is incremented for `0xVICTIM`, and execution proceeds.
4. The prover receives the same block and runs `BasicBootloaderProvingExecutionConfig`. The ecrecover branch executes, recovers a different address, and returns `InvalidTransaction::IncorrectFrom { recovered: 0xOTHER, tx: 0xVICTIM }`.
5. The forward and proving runs produce divergent state roots. The block is unprovable; proof generation halts. [7](#0-6) [8](#0-7)

pub fn run_block<T: ReadStorageTree, PS: PreimageSource, TS: TxSource, TR: TxResultCallback>(
    block_context: BlockContext,
    tree: T,
    preimage_source: PS,
    tx_source: TS,
    tx_result_callback: TR,
    tracer: &mut impl Tracer<ForwardRunningSystem>,
    validator: &mut impl TxValidator<ForwardRunningSystem>,
) -> Result<BlockOutput, ForwardSubsystemError> {
    let block_metadata_responder = BlockMetadataResponder {
        block_metadata: block_context,
    };
    let tx_data_responder = TxDataResponder {
        tx_source,
        next_tx: None,
        next_tx_format: None,
        next_tx_from: None,
    };
    let preimage_responder = GenericPreimageResponder { preimage_source };
    let tree_responder = ReadTreeResponder { tree };

    let mut oracle = ZkEENonDeterminismSource::default();
    oracle.add_external_processor(block_metadata_responder);
    oracle.add_external_processor(tx_data_responder);
    oracle.add_external_processor(preimage_responder);
    oracle.add_external_processor(tree_responder);

    let mut result_keeper = ForwardRunningResultKeeper::new(tx_result_callback);

    run_forward::<BasicBootloaderForwardSimulationConfig>(
        oracle,
        &mut result_keeper,
        tracer,
        validator,
    );
    Ok(result_keeper.into())
}
