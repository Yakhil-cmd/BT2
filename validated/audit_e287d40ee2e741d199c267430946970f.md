### Title
Gateway Validates L2 Gas Price Against Stale `l2_gas_price` Instead of `next_l2_gas_price`, Causing Admission/Execution Mismatch - (File: crates/apollo_gateway/src/stateful_transaction_validator.rs)

### Summary

The gateway's stateful validator validates a transaction's `max_price_per_unit` against the **current block's** `l2_gas_price`, but the batcher executes that same transaction against the **next block's** `l2_gas_price` (`next_l2_gas_price` stored in the block header). Because EIP-1559 adjusts the price every block, these two values routinely differ. The gateway therefore admits transactions that will fail `check_fee_bounds` in the batcher (price rising) and rejects transactions that would succeed (price falling).

### Finding Description

`GatewayFixedBlockSyncStateClient::get_block_info_from_sync_client` constructs a `BlockInfo` from the latest committed block header. It populates `gas_prices.strk_gas_prices.l2_gas_price` from `block_header.l2_gas_price` — the price **of** block N — but silently drops `block_header.next_l2_gas_price`, which is the price **for** block N+1 (the block being built). [1](#0-0) 

`validate_resource_bounds` then reads this stale price and uses it as the admission threshold:

```rust
// TODO(Arni): getnext_l2_gas_price from the block header.
let previous_block_l2_gas_price = self
    .gateway_fixed_block_state_reader
    .get_block_info()
    .await?
    .gas_prices
    .strk_gas_prices
    .l2_gas_price;
``` [2](#0-1) 

`run_validate_entry_point` then builds the blockifier `BlockContext` from the same stale `block_info`, only incrementing the block number while leaving all gas prices at block N's values:

```rust
let mut block_info = self.gateway_fixed_block_state_reader.get_block_info().await?;
block_info.block_number = block_info.block_number.unchecked_next();
let block_context = BlockContext::new(block_info, ...);
``` [3](#0-2) 

The batcher, by contrast, builds block N+1's `BlockContext` with `next_l2_gas_price` from block N's header as the L2 gas price. `perform_pre_validation_stage` → `check_fee_bounds` then enforces:

```
resource_bounds.max_price_per_unit >= actual_gas_price   // actual = next_l2_gas_price
``` [4](#0-3) 

The two sides therefore check against different prices for the same block.

The `next_l2_gas_price` is computed by `calculate_next_l2_gas_price_for_fin` / `calculate_next_base_gas_price` via EIP-1559 and stored in the block header: [5](#0-4) [6](#0-5) 

It is never surfaced through `GatewayFixedBlockStateReader::get_block_info`, so the gateway has no path to the correct value today.

### Impact Explanation

**High — Mempool/gateway admission accepts invalid transactions or rejects valid transactions before sequencing.**

*Price-rising case (common under congestion):* A user submits a V3 transaction with `max_price_per_unit = l2_gas_price_N`. The gateway's `check_fee_bounds` passes (`l2_gas_price_N >= l2_gas_price_N`). The batcher's `check_fee_bounds` fails (`l2_gas_price_N < next_l2_gas_price_N`). The transaction is admitted to the mempool and occupies a slot but can never be included in block N+1. Repeated submissions fill the mempool with permanently-unexecutable transactions.

*Price-falling case:* A user submits a transaction with `max_price_per_unit = next_l2_gas_price_N`. The gateway rejects it (`next_l2_gas_price_N < l2_gas_price_N * threshold%` when `threshold < 100`). The batcher would have accepted it. Valid transactions are denied admission.

### Likelihood Explanation

EIP-1559 adjusts the L2 gas price every block. Any block with gas usage above or below the target produces a non-zero delta between `l2_gas_price` and `next_l2_gas_price`. The discrepancy is bounded per block by `gas_price_max_change_denominator`, but it is present on virtually every block. No special privilege is required; any user submitting a V3 (`AllResources`) transaction can land in the discrepancy window.

### Recommendation

`GatewayFixedBlockSyncStateClient::get_block_info_from_sync_client` should populate the returned `BlockInfo.gas_prices` L2 fields from `block_header.next_l2_gas_price` rather than `block_header.l2_gas_price`. This is exactly what the existing TODO comment requests:

```
// TODO(Arni): getnext_l2_gas_price from the block header.
``` [7](#0-6) 

Alternatively, expose `next_l2_gas_price` as a dedicated field on `GatewayFixedBlockStateReader` and use it in both `validate_resource_bounds` and `run_validate_entry_point`.

### Proof of Concept

1. Observe block N with `l2_gas_price = P` and `next_l2_gas_price = P'` where `P' > P` (any block with gas usage above target).
2. Submit a V3 invoke transaction with `l2_gas.max_price_per_unit = P`.
3. Gateway `validate_resource_bounds`: `P >= P * min_gas_price_percentage / 100` → passes (assuming `min_gas_price_percentage <= 100`).
4. Gateway `run_validate_entry_point` → `check_fee_bounds`: `block_context.l2_gas_price = P`, so `P >= P` → passes.
5. Transaction is admitted to the mempool.
6. Batcher builds block N+1 with `l2_gas_price = P'`.
7. Batcher `check_fee_bounds`: `P < P'` → `MaxGasPriceTooLow` → transaction fails pre-validation and is never included.
8. The mempool slot is consumed by a permanently-unexecutable transaction; repeat to exhaust mempool capacity.
