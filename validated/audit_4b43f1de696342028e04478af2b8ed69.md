### Title
Permanent Sub-Precision attoFlow Dust Lock in COA EVM Balance — (`fvm/evm/stdlib/contract.cdc`, `fvm/evm/impl/impl.go`, `fvm/evm/types/balance.go`)

---

### Summary

The `CadenceOwnedAccount.withdraw()` function in the Flow EVM contract enforces a minimum withdrawal floor of `1e10` attoFlow (the smallest UFix64-representable unit). Any attoFlow balance below this threshold — or any sub-precision remainder left after a truncating withdrawal — is permanently locked inside the COA's EVM address with no recovery mechanism. This is the direct Flow analog of the `BridgeExecutorBase.sol` issue: assets accumulate in a contract/account with no way to retrieve them.

---

### Finding Description

Flow EVM stores balances at 18-decimal (attoFlow) precision, while Cadence `FlowToken.Vault` uses `UFix64` at 8-decimal precision. The conversion multiplier is `1e10` attoFlow per UFix64 unit.

**Root cause — three cooperating code points:**

**1. Minimum-amount guard** (`fvm/evm/types/balance.go`, `AttoFlowBalanceIsValidForFlowVault`): [1](#0-0) 

Any balance strictly less than `1e10` attoFlow is rejected as un-withdrawable.

**2. Silent truncation in `InternalEVM.withdraw`** (`fvm/evm/impl/impl.go`): [2](#0-1) 

The requested attoFlow amount is integer-divided by `1e10`, discarding the remainder. The EVM account is debited only the truncated amount; the remainder stays in the EVM balance.

**3. No `withdrawAll` / `withdrawMax` on `CadenceOwnedAccount`** (`fvm/evm/stdlib/contract.cdc`): [3](#0-2) 

The only exit path for FLOW from a COA to Cadence is `withdraw(balance:)`. There is no function that drains the maximum representable amount or burns the sub-precision remainder.

**How dust accumulates:**

- A COA receives EVM-level transfers (from DeFi contracts, ERC-20 bridge payouts, gas refunds, or direct EVM-to-EVM transfers) that produce a balance with a sub-precision tail, e.g. `1_000_000_000_009_999_999_999` attoFlow (1 FLOW + 9,999,999,999 attoFlow dust).
- The owner calls `withdraw(balance: EVM.Balance(attoflow: 1_000_000_000_009_999_999_999))`.
- `InternalEVM.withdraw` truncates to `1_000_000_000_000_000_000` attoFlow (1 FLOW) and returns that vault.
- The remaining `9_999_999_999` attoFlow stays in the EVM account.
- Any subsequent attempt to withdraw this remainder fails with `ErrWithdrawBalanceRounding` because `9_999_999_999 < 1e10`. [4](#0-3) 

The test suite explicitly confirms this truncation behavior and the resulting locked remainder: [5](#0-4) 

---

### Impact Explanation

Sub-precision attoFlow dust (up to `9,999,999,999` attoFlow ≈ `0.000000001 FLOW` per COA) is permanently locked in the COA's EVM address. The COA owner cannot recover it: `withdraw()` rejects amounts below `1e10` attoFlow, and no alternative drain function exists. This is a systematic, irreversible cross-VM asset loss for every COA that ever holds a non-UFix64-aligned EVM balance.

---

### Likelihood Explanation

Any COA that interacts with EVM-side DeFi protocols, receives ERC-20 bridge payouts, or is the target of arbitrary EVM-level transfers can accumulate sub-precision dust. The truncation path is exercised by the protocol's own test suite, confirming it is a reachable, non-exceptional code path. Any unprivileged Cadence transaction sender who owns a COA and calls `withdraw()` with a non-aligned balance triggers this silently.

---

### Recommendation

Add a `withdrawMax()` (or `withdrawAll()`) function to `CadenceOwnedAccount` that:
1. Reads the current EVM balance in attoFlow.
2. Computes the largest UFix64-representable amount (`balance - (balance mod 1e10)`).
3. Withdraws that amount, leaving the sub-precision dust in the EVM account (or, optionally, burning it by transferring to the zero address).

This mirrors the external report's recommendation: provide a mechanism to retrieve the maximum recoverable asset from the contract/account balance.

---

### Proof of Concept

```cadence
import EVM from <EVMAddress>
import FlowToken from <FlowTokenAddress>

transaction() {
    prepare(account: auth(BorrowValue) &Account) {
        // Suppose COA has 1_000_000_000_009_999_999_999 attoFlow
        // (1 FLOW + 9_999_999_999 attoFlow dust from an EVM-level transfer)

        let coa = account.storage
            .borrow<auth(EVM.Withdraw) &EVM.CadenceOwnedAccount>(from: /storage/coa)!

        // Step 1: withdraw the UFix64-representable portion — succeeds, returns 1 FLOW vault
        let bal = EVM.Balance(attoflow: 1_000_000_000_009_999_999_999)
        let vault <- coa.withdraw(balance: bal)
        // vault.balance == 1.00000000 FLOW (truncated)
        // COA EVM balance is now 9_999_999_999 attoFlow (dust)
        destroy vault

        // Step 2: attempt to withdraw the remaining dust — PANICS with ErrWithdrawBalanceRounding
        // because 9_999_999_999 < 1e10 (minimum withdrawable unit)
        let dustBal = EVM.Balance(attoflow: 9_999_999_999)
        let dustVault <- coa.withdraw(balance: dustBal)  // <-- panics, dust is permanently locked
        destroy dustVault
    }
}
```

The `9_999_999_999` attoFlow remainder is permanently unrecoverable. The `AttoFlowBalanceIsValidForFlowVault` guard at `fvm/evm/types/balance.go:105` and the `ErrWithdrawBalanceRounding` panic at `fvm/evm/impl/impl.go:780` together ensure no withdrawal path exists for this dust. // AttoFlowBalanceIsValidForFlowVault returns true if the given balance,
// represented as atto-flow, can be stored in a Flow Vault, without loss
// in precision.
//
// Warning! The smallest unit of Flow token that a Flow Vault (Cadence)
// can store, is 1e-8 .
// This means the minimum balance, in atto-flow, that can be stored in a
// Flow Vault, is 1e10 .
func AttoFlowBalanceIsValidForFlowVault(bal *big.Int) bool {
	return bal.Cmp(UFixToAttoConversionMultiplier) >= 0
}

        /// Withdraws the balance from the cadence owned account's balance.
        /// Note that amounts smaller than 1e10 attoFlow can't be withdrawn,
        /// given that Flow Token Vaults use UFix64 to store balances.
        /// In other words, the smallest withdrawable amount is 1e10 attoFlow.
        /// Amounts smaller than 1e10 attoFlow, will cause the function to panic
        /// with: "withdraw failed! smallest unit allowed to transfer is 1e10 attoFlow".
        /// If the given balance conversion to UFix64 results in rounding loss,
        /// the withdrawal amount will be truncated to the maximum precision for UFix64.
        ///
        /// @param balance: The EVM balance to withdraw
        ///
        /// @return A FlowToken Vault with the requested balance
        access(Owner | Withdraw)
        fun withdraw(balance: Balance): @FlowToken.Vault {
            pre {
                !EVM.isPaused(): "EVM operations are temporarily paused"
            }

            if balance.isZero() {
                return <-FlowToken.createEmptyVault(vaultType: Type<@FlowToken.Vault>())
            }
            let vault <- InternalEVM.withdraw(
                from: self.addressBytes,
                amount: balance.attoflow
            ) as! @FlowToken.Vault
            emit FLOWTokensWithdrawn(
                address: self.address().toString(),
                amount: balance.inFLOW(),
                withdrawnUUID: vault.uuid,
                balanceAfterInAttoFlow: self.balance().attoflow
            )
            return <-vault
        }
