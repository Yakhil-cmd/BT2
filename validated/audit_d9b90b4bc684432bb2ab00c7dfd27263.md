### Title
Post-Logout WalletConnect Commands Execute Against Active Fingerprint Due to No-Op `handleLogOut` — (`packages/core/src/components/Auth/AuthProvider.tsx`)

### Summary

`AuthProvider.handleLogOut` is explicitly a no-op by design. After the user clicks the logout button, the daemon fingerprint remains active, `useGetLoggedInFingerprintQuery` continues to return the old fingerprint, the `WalletConnectProvider` `session_request` listener stays mounted, and the fingerprint guard in `useWalletConnectCommand.handleProcess` passes — allowing a paired dApp to dispatch `chia_sendTransaction` or `chia_signCoinSpends` after the user has explicitly ended their session.

### Finding Description

**1. `handleLogOut` is a documented no-op.**

`AuthProvider.handleLogOut` contains an explicit comment and does nothing: [1](#0-0) 

No backend RPC is called. The daemon fingerprint stays active.

**2. `currentFingerprint` still resolves to the old fingerprint.**

`useWalletConnectCommand` reads the logged-in fingerprint via: [2](#0-1) 

Because `handleLogOut` never calls a log-out RPC, the daemon still reports the old fingerprint. `useGetLoggedInFingerprintQuery` returns it unchanged.

**3. The fingerprint guard in `handleProcess` passes.**

The guard is: [3](#0-2) 

The condition `currentFingerprint !== pair.fingerprint` is the only runtime check that could block a post-logout request. Since the daemon is still logged in with the same fingerprint, this check passes and the command proceeds to `window.permissionsAPI.dispatchAsPair`.

**4. The `session_request` listener stays mounted.**

`WalletConnectProvider` registers the listener in a `useEffect` keyed on `client`: [4](#0-3) 

`WalletConnectProvider` is mounted at the application root, above the router outlet: [5](#0-4) 

Navigating to `/` after logout changes the route content but does **not** unmount `WalletConnectProvider`. The listener remains active.

**5. `processRef` stays valid.**

`WalletConnectProvider` keeps `processRef.current` up-to-date: [6](#0-5) 

Because the provider is never unmounted, `processRef.current` continues to point to the live `handleProcess` function.

### Impact Explanation

A malicious WalletConnect dApp that is already paired can, immediately after the user clicks logout, fire a `session_request` for `chia_sendTransaction` or `chia_signCoinSpends`. The request will be processed, the fingerprint guard will pass (daemon is still logged in), and the transaction will be dispatched — causing direct, unauthorized asset loss from the wallet the user believed they had logged out of.

### Likelihood Explanation

Any paired dApp can trigger this. The window of exposure is unbounded: the daemon fingerprint stays active indefinitely after logout (until the user selects a different key or restarts the daemon). No user interaction beyond the initial pairing approval is required from the attacker's side.

### Recommendation

`handleLogOut` must call the daemon's log-out RPC (e.g., `log_out` / `logout_all_fingerprints`) before navigating away. Until the backend API supports this, the minimum mitigation is to disconnect all WalletConnect sessions and revoke all pairs on logout, so that no `session_request` can be matched to a valid pair after the user has ended their session.

### Proof of Concept

1. Pair a WalletConnect dApp with a wallet fingerprint `F`.
2. Click the logout button in the GUI.
3. Immediately (from the dApp side) send a `session_request` for `chia_sendTransaction` with `fingerprint: F`.
4. Observe: `processSessionRequest` is invoked → `handleProcess` is called → `currentFingerprint === F === pair.fingerprint` → guard passes → `dispatchAsPair` executes the transaction.
5. Expected (secure) behavior: the request is rejected with an `UNAUTHORIZED_METHOD` error.
