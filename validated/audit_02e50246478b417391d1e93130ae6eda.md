### Title
Permanent Fund Freeze When `ft_transfer` Promise Fails in `ExitToNear` Precompile Without `error_refund` Feature - (`engine-precompiles/src/native.rs`)

---

### Summary

When the `error_refund` compile-time feature is not enabled, the `ExitToNear` precompile burns a user's ETH or ERC-20 tokens in the EVM state and then schedules an asynchronous `ft_transfer` promise on the NEP-141 connector — with **no callback attached**. If the `ft_transfer` promise fails for any reason (e.g., recipient not registered with the NEP-141 contract, connector paused, out of gas), the tokens are permanently destroyed in the EVM with no recovery path. This is the direct Aurora Engine analog of the Allora "silent error handling" class: a two-step operation (burn → transfer) where the second step can fail silently, leaving state permanently inconsistent.

---

### Finding Description

In `ExitToNear::run()`, the precompile constructs a `callback_args` struct:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,          // ← hardcoded None when feature is off
    transfer_near: transfer_near_args,
};
``` [1](#0-0) 

For the common ETH and ERC-20 exit paths (no wNEAR unwrap), `transfer_near_args` is also `None`, making `callback_args == ExitToNearPrecompileCallbackArgs::default()`. The code then takes the no-callback branch:

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // ← no callback attached
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs { ... })
};
``` [2](#0-1) 

The `ExitToEthereum` precompile has the same structural gap — it always creates a bare `PromiseArgs::Create(withdraw_promise)` with no callback: [3](#0-2) 

The two-step sequence is:
1. **Step 1 (EVM, atomic, committed):** ETH is transferred from the user to the precompile address, or ERC-20 tokens are burned by the ERC-20 contract calling the precompile.
2. **Step 2 (NEAR promise, async, unguarded):** `ft_transfer` / `withdraw` is called on the external NEP-141 connector. If this fails, there is no callback to detect the failure and re-mint/refund the tokens.

The `exit_to_near_precompile_callback` function that *would* handle the refund is only scheduled when `args.refund` is `Some(...)`, which only happens when `error_refund` is compiled in: [4](#0-3) 

The test suite explicitly acknowledges the fund-loss behavior when the feature is absent:

```rust
// If the refund feature is not enabled, then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let expected_balance = Wei::new_u64(INITIAL_ETH_BALANCE - ETH_EXIT_AMOUNT);
``` [5](#0-4) 

The same pattern is confirmed for ERC-20 exits: [6](#0-5) 

---

### Impact Explanation

When `error_refund` is not compiled in and the `ft_transfer` promise fails:

- The user's ETH or ERC-20 tokens are **permanently destroyed** in the EVM (Step 1 is committed and irreversible).
- The NEP-141 tokens are never received by the recipient.
- There is no recovery mechanism — no callback, no re-mint, no refund.

This constitutes **Critical — Permanent freezing of funds**.

---

### Likelihood Explanation

The `ft_transfer` promise can fail under realistic, user-reachable conditions:

1. **Recipient not registered with the NEP-141 contract** — the most common case. Any user who exits to a NEAR account that has not called `storage_deposit` on the connector will trigger this failure.
2. **ETH connector paused** — the connector exposes a pause mechanism; if paused at the moment the promise executes, the transfer fails.
3. **Insufficient gas forwarded** — `FT_TRANSFER_GAS` is set to 10 TGas; if the connector's `ft_transfer` requires more, the promise fails.

Case 1 is trivially reachable by any unprivileged EVM user with no special setup.

---

### Recommendation

When `error_refund` is disabled, the precompile should either:

1. **Always attach a callback** that detects promise failure and re-mints/refunds the burned tokens (i.e., make `error_refund` behavior unconditional), or
2. **Validate at precompile execution time** that the recipient is registered and the connector is live before burning tokens, reverting the EVM call if the preconditions are not met.

The existing `refund_on_error` / `exit_to_near_precompile_callback` machinery already implements the correct fix — it should not be gated behind an optional feature flag for production deployments. [7](#0-6) 

---

### Proof of Concept

1. Deploy Aurora Engine **without** the `error_refund` feature.
2. Fund an EVM address with ETH on Aurora.
3. Call the `exitToNear` precompile targeting a NEAR account that has **not** registered storage with the ETH connector (e.g., a freshly created account).
4. The EVM transaction succeeds: ETH is deducted from the user's EVM balance and transferred to the precompile address.
5. The `ft_transfer` promise to the ETH connector fails (storage not registered).
6. No callback fires. No refund occurs.
7. Observe: the user's EVM ETH balance is reduced by `ETH_EXIT_AMOUNT`, the NEAR recipient receives nothing, and the ETH is permanently locked at the precompile address with no recovery path.

This is confirmed by the existing test `test_exit_to_near_eth_refund` which explicitly asserts the reduced balance when `error_refund` is absent. [8](#0-7)

pub fn refund_on_error<I: IO + Copy, E: Env, P: PromiseHandler>(
    io: I,
    env: &E,
    state: EngineState,
    args: &RefundCallArgs,
    handler: &mut P,
) -> EngineResult<SubmitResult> {
    let current_account_id = env.current_account_id();
    if let Some(erc20_address) = args.erc20_address {
        // ERC-20 exit; re-mint burned tokens
        let erc20_admin_address = current_address(&current_account_id);
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, erc20_admin_address, current_account_id, io, env);

        let refund_address = args.recipient_address;
        let amount = U256::from_big_endian(&args.amount);
        let input = setup_refund_on_error_input(amount, refund_address);

        engine.call(
            &erc20_admin_address,
            &erc20_address,
            Wei::zero(),
            input,
            u64::MAX,
            Vec::new(),
            Vec::new(),
            handler,
        )
    } else {
        // ETH exit; transfer ETH back from precompile address
        let exit_address = exit_to_near::ADDRESS;
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, exit_address, current_account_id, io, env);
        let refund_address = args.recipient_address;
        let amount = Wei::new(U256::from_big_endian(&args.amount));
        engine.call(
            &exit_address,
            &refund_address,
            amount,
            Vec::new(),
            u64::MAX,
            vec![
                (exit_address.raw(), Vec::new()),
                (refund_address.raw(), Vec::new()),
            ],
            Vec::new(),
            handler,
        )
    }
