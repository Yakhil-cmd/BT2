Audit Report

## Title
Unused `delivered_ids` in `validate_payload` Allows Byzantine Block Proposer to Inject Duplicate Bitcoin Adapter Responses — (`rs/bitcoin/consensus/src/payload_builder.rs`)

## Summary

`validate_payload` in `BitcoinPayloadBuilder` computes `delivered_ids` from `past_payloads` but never uses the set to reject duplicate `callback_id` values in the proposed payload. A single Byzantine block proposer can craft a `SelfValidatingPayload` containing a `callback_id` already present in `past_payloads`; honest validators notarize and finalize the block, and the execution layer receives a duplicate `ConsensusResponse`, corrupting replicated state in the Bitcoin integration.

## Finding Description

In `validate_payload` (`BatchPayloadBuilder` impl), `delivered_ids` is computed at line 369 from `past_payloads` via `parse::parse_past_payload_ids`, but the variable is never consulted again: [1](#0-0) 

The call to `validate_self_validating_payload_impl` at line 379 receives only the decoded payload and `validation_context` — `delivered_ids` is silently dropped. `validate_self_validating_payload_impl` itself performs no duplicate check; it only short-circuits on an empty payload and returns the byte count: [2](#0-1) 

The `SelfValidatingPayloadBuilder` impl has the same gap — `_past_payloads` is explicitly ignored: [3](#0-2) 

By contrast, `build_payload` correctly passes `delivered_ids` to `get_self_validating_payload_impl` and skips already-delivered IDs: [4](#0-3) 

After a duplicate-callback block is finalized, `demux.process_payload` calls `push_response_bitcoin` for every response in the payload: [5](#0-4) 

For `GetSuccessorsResponse`, `push_response` uses `get_mut` (not `remove`) on `bitcoin_get_successors_contexts`, so the context survives the first call and the second call also succeeds: [6](#0-5) 

The second call overwrites `bitcoin_get_successors_follow_up_responses` with the duplicate's follow-up blocks, corrupting the paginated block data the bitcoin canister serves to callers. For `SendTransactionResponse`, `GetSuccessorsReject`, and `SendTransactionReject`, there is no context lookup at all — both the original and duplicate `ConsensusResponse` are unconditionally pushed to `consensus_queue`: [7](#0-6) 

The file-level `#![allow(dead_code, unused_variables)]` suppresses the Rust compiler warning that would otherwise flag the unused `delivered_ids` binding: [8](#0-7) 

## Impact Explanation

This is a **High** severity finding. A Byzantine block proposer can corrupt replicated state in the Bitcoin/Chain Fusion integration by injecting a duplicate `GetSuccessorsResponse` callback. The overwrite of `bitcoin_get_successors_follow_up_responses` causes the bitcoin canister to serve incorrect paginated block data, directly disrupting ckBTC block-ingestion logic. For other response types, duplicate entries in `consensus_queue` cause the scheduler to attempt delivery to an already-consumed callback, producing execution errors. This constitutes a concrete, non-hypothetical disruption of Chain Fusion / ckBTC infrastructure with direct protocol harm, matching the allowed impact: *Significant Chain Fusion, ck-token, ledger, Rosetta, boundary/API, XRC, Internet Identity, NNS, SNS, or infrastructure security impact with concrete user or protocol harm.*

## Likelihood Explanation

Exploitation requires exactly one Byzantine block proposer — a single malicious replica acting within its normal consensus turn. No threshold corruption, no key compromise, and no external infrastructure attack is needed. The proposer crafts a `SelfValidatingPayload` whose `callback_id` list overlaps with an entry in `past_payloads`. All honest validators call `validate_payload`, compute `delivered_ids`, and then ignore it, so they notarize and finalize the block normally. The attack is repeatable every time the Byzantine node holds a proposer slot.

## Recommendation

In `validate_payload`, after computing `delivered_ids` and decoding the payload, iterate over the responses and reject any whose `callback_id` is already in `delivered_ids`:

```rust
let delivered_ids = parse::parse_past_payload_ids(past_payloads, &self.log);
let payload = parse::bytes_to_payload(payload).map_err(|e| { ... })?;

for response in &payload {
    if delivered_ids.contains(&response.callback_id) {
        return Err(ValidationError::InvalidArtifact(
            consensus::InvalidPayloadReason::InvalidSelfValidatingPayload(
                InvalidSelfValidatingPayloadReason::DuplicateResponse,
            ),
        ));
    }
}
```

Apply the same fix to `validate_self_validating_payload` — stop ignoring `_past_payloads` and perform the same check. Additionally, `push_response` for `GetSuccessorsResponse` should use `remove` rather than `get_mut` so that any duplicate response reaching the state layer is caught by `BitcoinNonMatchingResponse` rather than silently overwriting follow-up data.

## Proof of Concept

```rust
// In validate_payload unit tests:
// 1. Encode a payload containing callback_id = 0.
// 2. Pass that encoding as a PastPayload entry.
// 3. Call validate_payload with the same payload bytes AND past_payloads = [that entry].
// 4. Assert Ok(()) is returned — demonstrating no duplicate check occurs.
let payload_bytes = encode_response(callback_id: 0);
let past = PastPayload { height: Height::new(1), payload: &payload_bytes, ... };
let result = builder.validate_payload(
    Height::new(2),
    &proposal_context,
    &payload_bytes,
    &[past],
);
assert_eq!(result, Ok(())); // passes today; should be Err after fix
```
