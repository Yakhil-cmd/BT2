### Title
Dual Hidden Governance: Contract Updates Bypass Authorized-Deployment List While Contract-Removal Restriction Is Governed by a Hidden Off-Chain Context Flag — (`fvm/environment/contract_updater.go`)

---

### Summary

The FVM contract-governance layer in `fvm/environment/contract_updater.go` operates with two separate, inconsistent authorization models that are not visible to on-chain observers:

1. **New contract deployments** are gated by an on-chain authorized-addresses list (`ContractDeploymentAuthorizedAddressesPath`) when `RestrictedDeploymentEnabled()` is true.
2. **Contract updates** (to already-deployed contracts) **unconditionally bypass** that same list — the comment in `SetContract()` explicitly states *"Contract updates are always allowed."*
3. **Contract removal restriction** is governed exclusively by a hidden off-chain context flag (`RestrictContractRemoval`) rather than on-chain state, while the analogous deployment restriction is read from on-chain state. A `TODO` comment in the code acknowledges this asymmetry has never been resolved.

This mirrors the original report's pattern exactly: one governance path is visible and on-chain; a second, hidden path bypasses it.

---

### Finding Description

In `fvm/environment/contract_updater.go`, `SetContract()` enforces the authorized-addresses check only for *new* deployments:

```go
// Initial contract deployments must be authorized by signing accounts.
//
// Contract updates are always allowed.
exists, err := updater.accounts.ContractExists(location.Name, flow.Address(location.Address))
...
if !exists && !updater.isAuthorizedForDeployment(signingAccounts) {
    return fmt.Errorf("deploying contract failed: ...")
}
// No authorization check when exists == true
updater.draftUpdates[location] = ContractUpdate{...}
``` [1](#0-0) 

Once a contract exists at an address, any transaction signed by that account (holding `auth(UpdateContract)`) can overwrite the contract code with no check against the authorized-deployment list. The FVM-level governance (the on-chain list) is silently bypassed.

Separately, `RestrictedRemovalEnabled()` reads only from the off-chain context flag:

```go
func (impl *contractUpdaterStubsImpl) RestrictedRemovalEnabled() bool {
    // TODO read this from the chain similar to the contract deployment
    // but for now we would honor the fallback context flag
    return impl.RestrictContractRemoval
}
``` [2](#0-1) 

While `RestrictedDeploymentEnabled()` reads from on-chain state with a context-flag fallback: [3](#0-2) 

The on-chain paths for deployment restriction are defined in `fvm/blueprints/contracts.go`: [4](#0-3) 

No equivalent on-chain path exists for removal restriction. The `ContractUpdaterParams` struct documents the deployment flag as deprecated but still carries both flags: [5](#0-4) 

---

### Impact Explanation

**Hidden update bypass:** An account that was previously authorized to deploy a contract, but has since been removed from the authorized-deployment list via on-chain governance, can still overwrite its existing contract code at will. The on-chain governance action (removing the account from the list) is silently ineffective for existing contracts. Any user or auditor inspecting the on-chain authorized-addresses list would incorrectly conclude that the removed account can no longer modify contracts.

**Hidden removal governance:** The removal restriction is controlled by a node-operator context flag invisible to on-chain observers. On-chain governance can set the deployment restriction; it cannot set the removal restriction. The two operations appear symmetric to users but are governed by entirely different (and inconsistent) mechanisms.

---

### Likelihood Explanation

The update bypass is reachable by any account that has ever deployed a contract. No privileged access is required beyond owning the account's signing key. The attacker-controlled entry path is a standard `UpdateContract` Cadence transaction signed by the contract-owning account. The `fvm_blockcontext_test.go` integration test explicitly confirms this behavior is exercised in production: [6](#0-5) 

The removal-restriction asymmetry is a structural property of every deployed network; it does not require any special triggering condition.

---

### Recommendation

1. **Unify update authorization with deployment authorization.** `SetContract()` should apply `isAuthorizedForDeployment()` for both new deployments and updates when `RestrictedDeploymentEnabled()` is true, or introduce a separate `isAuthorizedForUpdate()` check backed by the same on-chain list.

2. **Resolve the `TODO` in `RestrictedRemovalEnabled()`.** Read the removal restriction from on-chain state (analogous to `getIsContractDeploymentRestricted()`) so that both deployment and removal governance are transparent and auditable on-chain. The `ContractRemovalAuthorizedAddressesPath` storage path already exists in `blueprints/contracts.go` and can be extended to carry the restriction flag.

---

### Proof of Concept

**Update bypass:**

```
// Step 1: Service account adds AccountA to the authorized deployment list.
// Step 2: AccountA deploys ContractX (authorized).
// Step 3: Service account removes AccountA from the authorized deployment list (on-chain governance).
// Step 4: AccountA submits:
transaction {
    prepare(signer: auth(UpdateContract) &Account) {
        signer.contracts.update(name: "ContractX", code: "<malicious_code>".decodeHex())
    }
}
// signed by AccountA only — no service account co-signature required.
// SetContract() sees exists==true, skips isAuthorizedForDeployment(), succeeds.
```

The `UpdateContractUnathorizedDeploymentTransaction` fixture in the test suite confirms this transaction shape is valid: [7](#0-6) 

**Removal restriction hidden governance:**

The `SystemChunkContext` in the execution computer sets both flags to `false` at the Go level, bypassing any on-chain state for removal: [8](#0-7) 

No on-chain transaction can override `WithContractRemovalRestricted(false)` set here, because `RestrictedRemovalEnabled()` never reads on-chain state.

func (updater *ContractUpdaterImpl) SetContract(
	location common.AddressLocation,
	code []byte,
	signingAccounts []flow.Address,
) error {
	// Initial contract deployments must be authorized by signing accounts.
	//
	// Contract updates are always allowed.
	exists, err := updater.accounts.ContractExists(location.Name, flow.Address(location.Address))
	if err != nil {
		return err
	}

	if !exists && !updater.isAuthorizedForDeployment(signingAccounts) {
		return fmt.Errorf(
			"deploying contract failed: %w",
			errors.NewOperationAuthorizationErrorf(
				"SetContract",
				"deploying contracts requires authorization from specific "+
					"accounts"))

	}

	updater.draftUpdates[location] = ContractUpdate{
		Location: location,
		Code:     code,
	}

	return nil
}
