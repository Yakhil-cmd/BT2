### Title
Missing Native Resource Charge for Initcode Word Cost in CREATE/CREATE2 — (`evm_interpreter/src/instructions/host.rs`)

### Summary

In `evm_interpreter/src/instructions/host.rs`, the `create` function charges the per-word initcode cost using `self.gas.spend_gas(initcode_cost)` instead of `self.gas.spend_gas_and_native(initcode_cost, native_cost)`. This is the direct analog of the external report: a simpler variant of a function is called where the full variant (which also updates a critical accounting state) should be used. The result is that the native (proving) resource is never charged for the initcode word cost in CREATE and CREATE2 operations, breaking the double-resource accounting invariant that every EVM gas charge must be accompanied by a corresponding native resource charge.

### Finding Description

ZKsync OS implements a **double resource accounting** model: every EVM operation charges both *ergs* (EVM gas × `ERGS_PER_GAS`) and a *native* resource that models the RISC-V proving cost. Two helper functions exist on `Gas<S>`:

- `spend_gas(gas)` — charges only ergs, **no native resource**.
- `spend_gas_and_native(gas, native)` — charges both ergs and native resource.

Every other opcode that has a per-byte/per-word dynamic cost (e.g., `calldatacopy`, `codecopy`, `extcodecopy`, `mcopy`) calls `spend_gas_and_native` for the dynamic portion. The `create` function is the sole exception:

```rust
// evm_interpreter/src/instructions/host.rs  lines 320-327
let cost_per_word = if IS_CREATE2 {
    INITCODE_WORD_COST + SHA3WORD
} else {
    INITCODE_WORD_COST
};
let initcode_cost = cost_per_word * (len as u64).div_ceil(32);
self.gas.spend_gas(initcode_cost)?;   // ← only ergs, native resource skipped
```

The base CREATE/CREATE2 native cost is charged correctly at the top of the function via `spend_gas_and_native`, but the per-word initcode cost — which can be up to `MAX_INITCODE_SIZE / 32` words — is charged only in EVM gas. No native resource constant for initcode word cost exists in `native_resource_constants.rs`, confirming the omission is not intentional.

### Impact Explanation

The native resource is the mechanism by which ZKsync OS ensures that users pay for the actual RISC-V proving cost of their transactions. When native resource is under-charged:

1. **Incorrect fee accounting**: The `deltaGas` adjustment at the end of a transaction (`deltaGas = native_used / native_per_gas − gas_used`) is computed from the actual native resource consumed by the prover, not from what was charged. Because the initcode word cost was not deducted from the native resource counter, `native_used` is higher than what was charged, causing `deltaGas > 0` and forcing extra EVM gas to be consumed — but only *after* execution, not during. A transaction with large initcode can therefore pass the native resource limit check during execution while the prover actually consumes more cycles than the user paid for.

2. **Proving cost subsidy / resource drain**: An attacker can repeatedly deploy contracts with near-maximum initcode size (`MAX_INITCODE_SIZE = 2 × 24576 = 49152 bytes`) and pay only the EVM gas cost for the initcode words, not the native proving cost. This allows the attacker to consume proving resources at a discount, potentially draining operator revenue or causing proving queues to back up.

### Likelihood Explanation

The entry path is fully unprivileged: any EOA can submit a CREATE or CREATE2 transaction. The initcode size is bounded by `MAX_INITCODE_SIZE` but that bound is large enough to make the discrepancy meaningful. The bug is triggered on every CREATE/CREATE2 with non-zero initcode length, which is the common case for contract deployment.

### Recommendation

Replace the `spend_gas` call for the initcode word cost with `spend_gas_and_native`, adding a per-word native cost constant analogous to `COPY_BYTE_NATIVE_COST` used by other copy operations:

```rust
// evm_interpreter/src/native_resource_constants.rs
pub const INITCODE_WORD_NATIVE_COST: u64 = /* calibrated value, e.g. COPY_BYTE_NATIVE_COST * 32 */;

// evm_interpreter/src/instructions/host.rs
let initcode_native_cost = INITCODE_WORD_NATIVE_COST
    .saturating_mul((len as u64).div_ceil(32));
self.gas.spend_gas_and_native(initcode_cost, initcode_native_cost)?;
```

### Proof of Concept

1. Deploy a contract using CREATE2 with initcode of length `MAX_INITCODE_SIZE` (49152 bytes).
2. Observe that `spend_gas_and_native` is called once at the top of `create` with `CREATE2_NATIVE_COST`, but the per-word cost `(INITCODE_WORD_COST + SHA3WORD) * (49152 / 32) = 4704 gas` is charged via `spend_gas` only.
3. The native resource counter is not decremented for those 1536 words of initcode.
4. At transaction end, `compute_gas_refund` computes `native_used` from the actual prover consumption (which includes the initcode hashing cycles), producing a `deltaGas > 0` that forces extra EVM gas to be charged retroactively — but the user's native resource budget was never reduced during execution, so the native limit check during execution was effectively bypassed for this cost component.
