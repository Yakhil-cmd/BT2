### Title
Sliding-Digest Window Eviction Allows Duplicate Key Re-Registration, Enabling Key-Revocation Bypass — (`fvm/environment/account-key-metadata/digest.go`, `fvm/environment/accounts_status.go`)

### Summary
With `MaxStoredDigests=2`, an account holder can evict an existing key's digest from the two-slot window by adding two distinct keys, then re-add the original key. `FindDuplicateKey` searches only the live window and returns `found=false`, causing `AppendUniqueKeyMetadata` to allocate a second independent stored-key register entry for the same public-key material. Each entry carries its own `revoked` flag. Revoking the original key index leaves the second entry unrevoked, so transactions signed with the same private key at the new index continue to pass authorization.

### Finding Description
`MaxStoredDigests = 2` is the production constant. [1](#0-0) 

`appendDigest` evicts the oldest digest when the window is full: [7](#0-6) 

`findDuplicateDigest` searches only the live `digestBytes` slice: [8](#0-7) 

`appendAccountPublicKeyMetadata` calls `FindDuplicateKey` once and, on `false`, unconditionally calls `AppendUniqueKeyMetadata`: [4](#0-3) 

There is no secondary scan of all stored keys, no cross-index revocation propagation, and no guard that prevents the same key bytes from being written to two different `apk_N` registers.

### Impact Explanation
Two independent stored-key entries exist for the same public-key material. Each has its own `revoked` flag in `weightAndRevokedStatusBytes`. Revoking key index 0 sets `revoked=true` only for that index. Key index 3 (the re-added duplicate) retains `revoked=false`. A transaction signed with the corresponding private key, referencing key index 3, passes signature verification and authorization checks normally. The operator's revocation action is silently ineffective.

### Likelihood Explanation
The precondition is that the attacker holds a key with weight ≥ 1000 (or sufficient weight in a multisig) on the target account. The exploit requires exactly three `AddAccountKey` calls in a specific order — trivially scriptable in a single Cadence transaction. `MaxStoredDigests=2` means only two interleaving keys are needed to evict any target digest. No brute force, no privileged node access, and no external dependency is required.

### Recommendation
1. **Increase `MaxStoredDigests`** to a value large enough that eviction within a realistic key-addition sequence is impractical (e.g., 64 or unbounded for accounts with few keys).
2. **Alternatively, enforce a full-scan deduplication path** when the requested key's digest falls below `startIndexForDigests` — i.e., when the digest may have been evicted — by iterating all stored keys via `getStoredKey` before accepting the addition.
3. **Document the security boundary explicitly**: if the design intentionally allows duplicate key material at multiple indices, the revocation API must be documented to revoke by key material (scanning all indices), not by index alone.

func (m *KeyMetadataAppender) appendDigest(digest uint64) {
	digestCount := 1 + len(m.digestBytes)/digestSize

	if digestCount > int(m.maxStoredDigests) {
		// Remove digest from front
		removeCount := digestCount - int(m.maxStoredDigests)
		m.digestBytes = slices.Delete(m.digestBytes, 0, removeCount*digestSize)

		// Adjust digest start index
		m.startIndexForDigests += uint32(removeCount)
	}

	var digestBytes [digestSize]byte
	binary.BigEndian.PutUint64(digestBytes[:], digest)

	m.digestBytes = append(m.digestBytes, digestBytes[:]...)
}

func (m *KeyMetadataAppender) findDuplicateDigest(digest uint64) (found bool, duplicateStoredKeyIndex uint32) {
	if len(m.digestBytes) == 0 {
		return false, 0
	}

	var digestBytes [digestSize]byte
	binary.BigEndian.PutUint64(digestBytes[:], digest)

	for off, i := 0, uint32(0); off < len(m.digestBytes); off, i = off+digestSize, i+1 {
		if bytes.Equal(digestBytes[:], m.digestBytes[off:off+digestSize]) {
			return true, m.startIndexForDigests + i
		}
	}

	return false, 0
}
