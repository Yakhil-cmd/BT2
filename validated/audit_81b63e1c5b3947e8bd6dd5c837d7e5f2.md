### Title
Gateway Admission Uses Stale `l2_gas_price` Instead of `next_l2_gas_price` for Fee Threshold Validation ŌĆö (`File: crates/apollo_gateway/src/gateway_fixed_block_state_reader.rs`)

---

### Summary

`GatewayFixedBlockSyncStateClient::get_block_info_from_sync_client` populates `BlockInfo.gas_prices.strk_gas_prices.l2_gas_price` from `block_header.l2_gas_price` (the **current** block's price). The gateway's `validate_resource_bounds` and `run_validate_entry_point` both consume this value. However, the block being built by the batcher/orchestrator uses `block_header.next_l2_gas_price` (the fee-market-computed price for the **next** block) as its actual L2 gas price. The mismatch means the gateway validates transactions against the wrong price, causing valid transactions to be rejected and invalid transactions to be admitted.

---

### Finding Description

`GatewayFixedBlockSyncStateClient::get_block_info_from_sync_client` builds a `BlockInfo` from the latest committed block header:

```rust
// crates/apollo_gateway/src/gateway_fixed_block_state_reader.rs  lines 36-57
let block_info = BlockInfo {
    gas_prices: GasPrices {
        strk_gas_prices: GasPriceVector {
            l2_gas_price: block_header.l2_gas_price.price_in_fri.try_into()?,
            // ^^^ reads l2_gas_price (current block), NOT next_l2_gas_price
            ...
        },
    },
    ...
};
```

`BlockHeaderWithoutHash` carries **both** fields:

```rust
// crates/starknet_api/src/block.rs  lines 237-239
pub l2_gas_price: GasPricePerToken,      // price used in block N
pub next_l2_gas_price: GasPrice,         // fee-market price for block N+1
```

The gateway's `validate_resource_bounds` then reads this stale price and computes the admission threshold:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs  lines 229-240
// TODO(Arni): getnext_l2_gas_price from the block header.
let previous_block_l2_gas_price = self
    .gateway_fixed_block_state_reader
    .get_block_info()
    .await?
    .gas_prices
    .strk_gas_prices
    .l2_gas_price;                       // ŌåÉ stale: block N price, not block N+1 price
self.validate_tx_l2_gas_price_within_threshold(
    executable_tx.resource_bounds(),
    previous_block_l2_gas_price,
)?;
```

The same stale `block_info` is forwarded into `run_validate_entry_point` to build the `BlockContext` used for blockifier pre-validation (`check_fee_bounds`):

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs  lines 323-330
let mut block_info = self.gateway_fixed_block_state_reader.get_block_info().await?;
block_info.block_number = block_info.block_number.unchecked_next();
let block_context = BlockContext::new(block_info, ...);
// block_context.block_info.gas_prices.strk_gas_prices.l2_gas_price == P_N (stale)
```

Meanwhile, the orchestrator maintains `self.l2_gas_price` which is updated to `next_l2_gas_price` after every committed block via `update_l2_gas_price`, and passes this value to the batcher when building block N+1. The batcher therefore executes transactions under `P_{N+1}`, while the gateway validated them under `P_N`.

---

### Impact Explanation

**Impact: High ŌĆö Mempool/gateway admission accepts invalid transactions or rejects valid transactions before sequencing.**

Two concrete failure modes arise whenever `P_{N+1} ŌēĀ P_N`:

1. **Gas price rising (`P_{N+1} > P_N`)**: A transaction with `tx_l2_gas_price` satisfying `threshold(P_N) Ōēż tx_l2_gas_price < P_{N+1}` passes both the gateway threshold check and the gateway's blockifier `check_fee_bounds` (which also uses `P_N`), is admitted to the mempool, and then fails the batcher's `check_fee_bounds` with `ResourceBoundsError::MaxGasPriceTooLow{resource: L2Gas}` because the batcher uses `P_{N+1}`. The transaction is sequenced but reverts, and the user is charged a fee for a transaction that the gateway should have rejected.

2. **Gas price falling (`P_{N+1} < P_N`)**: A transaction with `tx_l2_gas_price` satisfying `threshold(P_{N+1}) Ōēż tx_l2_gas_price < threshold(P_N)` is valid for the next block but is rejected by the gateway because the threshold is computed from the higher stale price `P_N`. Valid transactions are denied admission.

---

### Likelihood Explanation

The fee market (`calculate_next_base_gas_price`) adjusts `next_l2_gas_price` every block based on `l2_gas_consumed` vs. `gas_target`. During any period of sustained high or low demand the two prices diverge by more than the `min_gas_price_percentage` margin, making both failure modes reachable without any privileged action. An attacker can deliberately craft transactions that sit in the gap to force repeated failed sequencing.

---

### Recommendation

In `GatewayFixedBlockSyncStateClient::get_block_info_from_sync_client`, replace the `l2_gas_price` field with `next_l2_gas_price` from the block header:

```rust
// crates/apollo_gateway/src/gateway_fixed_block_state_reader.rs
strk_gas_prices: GasPriceVector {
    l1_gas_price: block_header.l1_gas_price.price_in_fri.try_into()?,
    l1_data_gas_price: block_header.l1_data_gas_price.price_in_fri.try_into()?,
-   l2_gas_price: block_header.l2_gas_price.price_in_fri.try_into()?,
+   l2_gas_price: block_header.next_l2_gas_price.try_into()?,
},
```

Apply the same fix to the ETH side. This resolves the TODO already noted in the code and aligns the gateway's validation price with the price the batcher will actually enforce.

---

### Proof of Concept

1. Block N is committed: `l2_gas_price = 100 fri`, `next_l2_gas_price = 150 fri` (demand spike).
2. `GatewayFixedBlockSyncStateClient` caches `BlockInfo` with `strk_gas_prices.l2_gas_price = 100`.
3. User submits an invoke V3 tx with `l2_gas.max_price_per_unit = 60 fri`, `min_gas_price_percentage = 50`.
4. Gateway threshold = `50% ├Ś 100 = 50`; tx passes (`60 Ōēź 50`). Gateway `check_fee_bounds` also passes because it uses the same stale `block_context` with `l2_gas_price = 100` (`60 < 100` would fail ŌĆö but `check_fee_bounds` checks `max_price Ōēź block_price`, so `60 < 100` actually fails here too).

Let me correct the PoC: the more precise failure mode is the **rejection** direction:

1. Block N: `l2_gas_price = 150 fri`, `next_l2_gas_price = 100 fri` (demand drop).
2. Gateway caches `l2_gas_price = 150`. Threshold = `50% ├Ś 150 = 75`.
3. User submits tx with `l2_gas.max_price_per_unit = 80 fri`.
4. Gateway rejects: `80 < 75`? No ŌĆö `80 Ōēź 75`, passes. But a tx with `max_price_per_unit = 60` would be rejected (`60 < 75`) even though `60 Ōēź threshold(100) = 50` and the next block only requires `100 fri`.
