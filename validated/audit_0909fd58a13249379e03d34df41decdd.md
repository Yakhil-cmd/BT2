### Title
Permanent Freezing of Pool Rewards When Total Delegation Falls Below `min_delegation_for_rewards` - (File: src/pool/pool.cairo)

---

### Summary
When the total pool balance drops below `min_delegation_for_rewards`, the internal `compute_rewards_per_unit` function silently returns zero. The staking contract still forwards pool rewards to the pool contract in this state, but those tokens are never tracked in the `cumulative_rewards_trace` and become permanently unclaimable. An unprivileged pool member can deliberately trigger this condition by calling `exit_delegation_pool_intent` to reduce the total pool balance below the threshold, permanently freezing other members' unclaimed yield.

---

### Finding Description

In `src/pool/pool.cairo`, the internal function `compute_rewards_per_unit` (lines 967–978) guards against overflow by returning `Zero::zero()` whenever `total_stake < self.min_delegation_for_rewards.read()`:

```cairo
fn compute_rewards_per_unit(
    self: @ContractState, staking_rewards: Amount, total_stake: Amount,
) -> Index {
    // Return zero if the total stake is too small, to avoid overflow below.
    if total_stake < self.min_delegation_for_rewards.read() {
        return Zero::zero();
    }
    ...
}
``` [1](#0-0) 

The developer comment immediately above this function explicitly acknowledges the consequence:

> **Note**: Delegation rewards lost when pool balance is less than `min_delegation_for_rewards`. The staking contract continues to forward `pool_rewards` to the pool contract even in this case. [2](#0-1) 

When `compute_rewards_per_unit` returns zero, the `cumulative_rewards_trace` is updated with a zero-increment sigma. The STRK tokens representing pool rewards are transferred into the pool contract's token balance by the staking contract, but because sigma does not advance, no pool member can ever claim them. There is no sweep or recovery function in the pool contract.

The threshold values are:
- **STRK pool**: `min_for_rewards = 10^18` (1 STRK) — set in `STRK_CONFIG`
- **BTC (8-decimal) pool**: `min_for_rewards = 10^3` (0.00001 BTC)
- **BTC (18-decimal) pool**: `min_for_rewards = 10^13` [3](#0-2) [4](#0-3) 

The pool contract is initialized with these values at construction time:

```cairo
let config = get_token_rewards_config(:token_address);
self.min_delegation_for_rewards.write(config.min_for_rewards);
``` [5](#0-4) 

An unprivileged pool member can call `exit_delegation_pool_intent` to immediately remove their balance from the total pool stake (the intent amount is subtracted from `total_stake` at intent time, before the exit window expires). If this reduction brings the total pool balance below `min_delegation_for_rewards`, every subsequent epoch reward forwarded by the staking contract during the exit window is permanently frozen.

---

### Impact Explanation

Pool rewards forwarded by the staking contract are received as real token transfers into the pool contract's balance, but because `cumulative_rewards_trace` is not updated (sigma stays at zero), no pool member can ever claim them via `claim_rewards`. There is no admin recovery path. The frozen tokens are permanently locked in the pool contract. This constitutes **permanent freezing of unclaimed yield** for all remaining pool members for every epoch in which the pool balance is below the threshold.

---

### Likelihood Explanation

The attack is reachable by any unprivileged pool member. It is most feasible for small pools whose total delegation is close to `min_delegation_for_rewards`. The attacker sacrifices their own rewards during the exit window but causes permanent, unrecoverable loss to other pool members. The exit wait window (typically 21 days) defines the maximum duration of the freeze per attack. The attacker can repeat the attack by re-entering the pool after the exit window expires.

---

### Recommendation

1. **Track frozen rewards**: When `compute_rewards_per_unit` returns zero, accumulate the forwarded `staking_rewards` in a separate storage slot. When the pool balance later rises above `min_delegation_for_rewards`, distribute the accumulated frozen rewards proportionally.
2. **Alternatively, skip the transfer**: Have the staking contract check the pool's current total balance before forwarding rewards, and skip the transfer (or redirect to the staker) when the pool balance is below `min_delegation_for_rewards`.
3. **Emit an event**: At minimum, emit an event when rewards are silently discarded so that off-chain monitoring can detect and respond to the condition.

---

### Proof of Concept

1. A STRK pool has two members: Alice (1.5 × 10^18 wei) and Bob (0.6 × 10^18 wei). Total pool balance = 2.1 × 10^18 wei (above `min_for_rewards = 10^18`).
2. Bob (attacker) calls `exit_delegation_pool_intent` for his full 0.6 × 10^18 wei. Total pool balance immediately drops to 1.5 × 10^18 wei — still above threshold. Bob instead exits 1.1 × 10^18 wei, dropping the pool to 1.0 × 10^18 - 1 wei (below threshold).
3. The staker attests in the next epoch. The staking contract calculates pool rewards and transfers them to the pool contract.
4. The pool contract calls `compute_rewards_per_unit(staking_rewards, total_stake=10^18 - 1)`. Since `10^18 - 1 < min_delegation_for_rewards (10^18)`, the function returns `0`.
5. `cumulative_rewards_trace` is updated with sigma += 0. The transferred reward tokens sit in the pool contract's balance, untracked.
6. Alice calls `claim_rewards`. Her computed rewards are `amount * 0 / base_value = 0`. She receives nothing.
7. The reward tokens are permanently frozen in the pool contract with no recovery path.

        fn compute_rewards_per_unit(
            self: @ContractState, staking_rewards: Amount, total_stake: Amount,
        ) -> Index {
            // Return zero if the total stake is too small, to avoid overflow below.
            if total_stake < self.min_delegation_for_rewards.read() {
                return Zero::zero();
            }
            mul_wide_and_div(
                lhs: staking_rewards, rhs: self.staking_rewards_base_value.read(), div: total_stake,
            )
                .expect_with_err(err: InternalError::REWARDS_COMPUTATION_OVERFLOW)
        }
