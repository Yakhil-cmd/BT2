### Title
Inconsistent Rounding Direction in `native_per_pubdata` Calculation Causes Systematic Undercharging for Pubdata Costs — (`File: basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

`native_per_gas` is computed with ceiling division (`div_ceil`) to favor the protocol, but `native_per_pubdata` is computed with floor division (`wrapping_div`) in the same validation path. This inconsistency means every transaction that generates pubdata when `pubdata_price % native_price != 0` is systematically undercharged for pubdata costs, leaking value from the system in favor of transaction senders.

---

### Finding Description

In `validate_and_compute_fee_for_transaction`, two related ratios are computed from the same `native_price` denominator:

**`native_per_gas` — uses ceiling division (rounds up, protocol-favorable):** [1](#0-0) 

```rust
u256_try_to_u64(&gas_price.div_ceil(native_price)).ok_or(TxError::Validation(
    InvalidTransaction::NativeResourcesAreTooExpensive,
))?
```

**`native_per_pubdata` — uses floor division (rounds down, user-favorable):** [2](#0-1) 

```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

The same inconsistency is present in the API helper that mirrors bootloader behavior: [3](#0-2) 

```rust
// native_per_pubdata = pubdata_price / native_price
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
```

`native_per_pubdata` flows directly into `get_resources_to_charge_for_pubdata`, which computes the native resource cost of pubdata as `pubdata_bytes * native_per_pubdata`: [4](#0-3) 

This value is then used in `compute_gas_refund` to determine `delta_gas`: [5](#0-4) 

```rust
let delta_gas = if native_per_gas == 0 {
    0
} else {
    (native_used / native_per_gas) as i64 - (gas_used as i64)
};
```

The documentation confirms the intended formula: `deltaGas := (nativeUsed / nativePerGas) - gasUsed`. [6](#0-5) 

When `native_per_pubdata` is truncated (floor), `native_used` is smaller than the true cost, `delta_gas` is smaller, `gas_used` is smaller, and the final fee `gas_price * gas_used` is smaller than it should be.

A secondary rounding issue exists in `compute_gas_refund` itself: `native_used / native_per_gas` on line 72 also uses floor division, further reducing `delta_gas` below its true value. [7](#0-6) 

---

### Impact Explanation

**Vulnerability class:** Resource accounting bug — systematic undercharging for pubdata costs.

The effect chain is:
1. `native_per_pubdata = floor(pubdata_price / native_price)` — truncated by up to 1 unit
2. `native_cost_for_pubdata = pubdata_bytes × native_per_pubdata` — undercharged by up to `pubdata_bytes` native units
3. `native_used` is smaller → `delta_gas` is smaller → `gas_used` is smaller
4. `fee = gas_price × gas_used` is smaller → operator receives less

Per transaction the loss is small (at most `pubdata_bytes / native_per_gas` gas units), but it is systematic and affects every transaction that writes pubdata when `pubdata_price % native_price ≠ 0`. Over the lifetime of the chain, the cumulative undercharging to the operator is unbounded. The user/attacker benefits by paying slightly less than the true cost of their pubdata footprint.

---

### Likelihood Explanation

**High.** The operator sets `native_price` and `pubdata_price` independently as block-level oracle values. There is no mechanism that enforces `pubdata_price % native_price == 0`. In practice, these values are set based on market conditions and will almost never be exactly divisible. Every L2 transaction that touches storage (the common case) generates pubdata and is affected. No special attacker capability is required — any unprivileged transaction sender benefits automatically.

---

### Recommendation

Replace `wrapping_div` with `div_ceil` for `native_per_pubdata` to match the rounding direction used for `native_per_gas`:

```rust
// Before (floor division — user-favorable):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;

// After (ceiling division — protocol-favorable, consistent with native_per_gas):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

Apply the same fix in `api/src/helpers.rs` line 427. [3](#0-2) 

Additionally, consider replacing the floor division in `compute_gas_refund` line 72 with ceiling division for `native_used / native_per_gas` to ensure `delta_gas` is never rounded in the user's favor: [7](#0-6)

### Proof of Concept

**Setup:**
- `native_price = 3`
- `pubdata_price = 10`
- `native_per_pubdata_actual = ceil(10/3) = 4` (correct, protocol-favorable)
- `native_per_pubdata_actual = floor(10/3) = 3` (current code, user-favorable)

**Transaction generates 300 pubdata bytes:**
- Correct native charge: `300 × 4 = 1200` native units
- Actual native charge: `300 × 3 = 900` native units
- Undercharge: `300` native units

**With `native_per_gas = 2` and `gas_price = 1000`:**
- Correct `delta_gas` contribution from pubdata: `1200 / 2 = 600` extra gas
- Actual `delta_gas` contribution: `900 / 2 = 450` extra gas
- Undercharged gas: `150` gas units
- Undercharged fee: `150 × 1000 = 150,000` wei per transaction

Any unprivileged sender submitting storage-writing transactions triggers this path. The entry point is any standard L2 transaction (`ZkTransactionFlowOnlyEOA`) that writes to storage, routed through `validate_and_compute_fee_for_transaction` → `native_per_pubdata` computation → `get_resources_to_charge_for_pubdata` → `compute_gas_refund`.
