### Title
Pool Admin Can Front-Run Swaps by Instantly Raising Fees Without Any Timelock — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`setPoolAdminFees` and `setPoolBinAdditionalFees` allow the pool admin to change swap fees immediately and without any delay. Unlike price-provider rotations (which are protected by `priceProviderTimelock`), fee changes take effect in the same block they are submitted. A pool admin can front-run pending swaps by spiking fees to the maximum cap, collect the inflated fee revenue, then reset fees to attract the next batch of users.

---

### Finding Description

`MetricOmmPoolFactory.setPoolAdminFees` lets the pool admin update `adminSpreadFeeE6` (up to `maxAdminSpreadFeeE6`, hard-capped at 20%) and `adminNotionalFeeE8` (up to `maxAdminNotionalFeeE8`, hard-capped at 1%) with no timelock: [1](#0-0) 

`setPoolBinAdditionalFees` passes `addFeeBuyE6` / `addFeeSellE6` (both `uint16`, max 65,535 E6 ≈ 6.55%) directly to the pool with **no cap check at all**: [2](#0-1) 

By contrast, price-provider changes are explicitly protected by a per-pool `priceProviderTimelock`: [3](#0-2) 

The updated fees are written to the pool's `spreadFeeE6` / `notionalFeeE8` / per-bin `addFeeBuyE6` / `addFeeSellE6` storage slots and are consumed immediately on the very next swap: [4](#0-3) 

The spread fee is deducted from LP fee revenue during each bin iteration: [5](#0-4) 

The notional fee is charged on top of the swap output (exact-in) or input (exact-out): [6](#0-5) 

---

### Impact Explanation

A malicious pool admin can extract value from swappers by:

1. Advertising low fees to attract volume.
2. Observing a pending swap in the mempool and front-running it with `setPoolAdminFees(pool, maxAdminSpreadFeeE6, maxAdminNotionalFeeE8)` — up to **20% spread + 1% notional** — or `setPoolBinAdditionalFees` up to **6.55% per bin**.
3. The swap executes at the inflated fee rate; the admin fee share is credited to `adminFeeDestination`.
4. The admin resets fees to low values to attract the next wave of users.

Users who interact directly with the pool (bypassing the router) have zero slippage protection. Even users going through `MetricOmmSimpleRouter` are only protected if they set a tight `amountOutMinimum` / `amountInMaximum`; a fee spike within that tolerance still causes unannounced loss. [7](#0-6) 

The hard caps (`HARD_MAX_SPREAD_FEE_E6 = 200_000`, `HARD_MAX_NOTIONAL_FEE_E8 = 1_000_000`) bound the worst case but do not prevent the attack: [8](#0-7) 

---

### Likelihood Explanation

The pool admin is explicitly classified as **semi-trusted only inside caps and timelocks**. The design already enforces a timelock for oracle changes, demonstrating that the protocol recognises the front-running risk for sensitive admin actions. The absence of an equivalent timelock on fee changes is an inconsistency that any pool admin (EOA or compromised multisig) can exploit on any EVM chain where mempool visibility is available. No special setup is required beyond being the registered pool admin.

---

### Recommendation

Apply the same propose-then-execute timelock pattern used for price-provider rotations to `setPoolAdminFees` and `setPoolBinAdditionalFees`. Specifically:

- Add a `pendingAdminFees` / `pendingAdminFeesExecuteAfter` mapping pair analogous to `pendingPriceProvider` / `pendingPriceProviderExecuteAfter`.
- Require a minimum delay (e.g., the same `priceProviderTimelock` value, or a separate configurable `feeUpdateTimelock`) before the new rates take effect.
- Additionally, add an explicit cap check in `setPoolBinAdditionalFees` (currently absent) so `addFeeBuyE6` and `addFeeSellE6` cannot exceed a factory-governed maximum.

---

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.35;

// Demonstrates pool admin front-running a swap with inflated fees.
// Run against MetricOmmPoolFactory + MetricOmmPool test harness.

function test_poolAdmin_frontRuns_swap_with_maxFees() public {
    address pool = _createPool(); // admin fee starts at 0

    // 1. Admin spikes fees to the maximum cap in the same block as the swap.
    vm.prank(admin);
    factory.setPoolAdminFees(
        pool,
        factory.maxAdminSpreadFeeE6(),   // e.g. 200_000 = 20%
        factory.maxAdminNotionalFeeE8()  // e.g. 1_000_000 = 1%
    );

    // 2. Victim's swap executes at the inflated rate.
    vm.prank(victim);
    (int256 amount0, int256 amount1) = IMetricOmmPoolActions(pool).swap(
        victim,
        true,          // zeroForOne
        int256(1_000), // exact-in
        0,             // no price limit
        "",
        ""
    );

    // 3. Admin resets fees to attract the next user.
    vm.prank(admin);
    factory.setPoolAdminFees(pool, 0, 0);

    // Victim received significantly less token1 than at the advertised 0% admin fee.
    // The difference is captured in adminFeeDestination.
}
```
