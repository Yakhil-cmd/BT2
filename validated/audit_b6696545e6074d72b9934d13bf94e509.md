### Title
Missing Zero-Address Validation in `change_reward_address` Allows Permanent Freezing of Staker Rewards - (File: src/staking/staking.cairo)

### Summary
The `change_reward_address` function in the `Staking` contract does not validate that the new `reward_address` is non-zero. A staker who passes the zero address will have all future reward claims permanently transferred to the zero address, irreversibly destroying their unclaimed yield.

### Finding Description
In `src/staking/staking.cairo`, the `change_reward_address` function performs only one address-quality check — that the new address is not a registered token (`REWARD_ADDRESS_IS_TOKEN`) — but omits a zero-address guard:

```cairo
fn change_reward_address(ref self: ContractState, reward_address: ContractAddress) {
    self.general_prerequisites();
    assert!(
        !self.does_token_exist(token_address: reward_address),
        "{}",
        GenericError::REWARD_ADDRESS_IS_TOKEN,
    );
    let staker_address = get_caller_address();
    let mut staker_info = self.internal_staker_info(:staker_address);
    let old_address = staker_info.reward_address;
    staker_info.reward_address = reward_address;   // zero accepted here
    self.write_staker_info(:staker_address, :staker_info);
    ...
}
``` [1](#0-0) 

After the call, every subsequent `claim_rewards` execution reads `staker_info.reward_address` and transfers STRK to it:

```cairo
// claim_rewards transfers to reward_address
token_dispatcher.checked_transfer(recipient: reward_address, amount: rewards.into());
``` [1](#0-0) 

The same zero-address gap exists in the initial `stake` call, which also accepts `reward_address` without a zero check: [2](#0-1) 

The error catalogue already defines `ZERO_ADDRESS` and `CALLER_IS_ZERO_ADDRESS` for analogous guards elsewhere in the codebase, confirming the pattern is intentional in other paths but was omitted here: [3](#0-2) 

### Impact Explanation
**High — Permanent freezing of unclaimed yield.**

Once `reward_address` is set to `0x0`, every `claim_rewards` call silently transfers STRK to the zero address. The tokens are unrecoverable on Starknet. The staker has no way to redirect future rewards because `change_reward_address` will again accept zero without error, and the staker cannot call `claim_rewards` themselves to intercept the transfer (it always goes to the stored `reward_address`). All accumulated and future rewards are permanently destroyed.

### Likelihood Explanation
**Medium.** The function is callable by any active staker with no privileged role required. Realistic trigger paths include:

- A front-end or script that passes an uninitialized `ContractAddress` (defaults to `0` in Cairo/Starknet tooling).
- A staker intentionally "clearing" their reward address under the mistaken belief that it will be ignored or reset later.
- An integration contract that constructs the address from user-supplied calldata without sanitising it.

No leaked key, bridge compromise, or governance action is required.

### Recommendation
Add a non-zero assertion at the top of `change_reward_address` (and symmetrically in `stake`):

```cairo
fn change_reward_address(ref self: ContractState, reward_address: ContractAddress) {
    self.general_prerequisites();
    assert!(reward_address.is_non_zero(), "{}", GenericError::ZERO_ADDRESS);   // add this
    assert!(
        !self.does_token_exist(token_address: reward_address),
        "{}",
        GenericError::REWARD_ADDRESS_IS_TOKEN,
    );
    ...
}
```

The same guard should be applied to the `reward_address` parameter in `stake`.

### Proof of Concept

1. Staker calls `stake(reward_address: VALID, ...)` and accumulates rewards over several epochs.
2. Staker calls `change_reward_address(reward_address: 0x0)` — succeeds with no error.
3. Staker (or anyone) calls `claim_rewards(staker_address)`.
4. Inside `claim_rewards`, the contract reads `staker_info.reward_address == 0x0` and executes `checked_transfer(recipient: 0x0, amount: rewards)`.
5. STRK tokens are transferred to the zero address and are permanently unrecoverable.
6. The staker's `unclaimed_rewards_own` is reset to zero; the funds are gone. [1](#0-0) [4](#0-3)
