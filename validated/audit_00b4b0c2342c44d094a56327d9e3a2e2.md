### Title
ETH Permanently Locked in `RewardSupplier` When `tick()` Called With No Pending L2 Mint Requests — (`L1/starkware/solidity/stake/RewardSupplier.sol`)

---

### Summary

`tick()` is declared `external payable` but only forwards `msg.value` inside the `if (amountToMint > 0)` branch. When no L2 mint-request messages are pending, the branch is skipped entirely, `msg.value` is silently accepted and never spent, and no ETH withdrawal path exists anywhere in the contract or its inheritance chain.

---

### Finding Description

`tick()` at line 107 is marked `payable`, allowing any caller to attach ETH: [1](#0-0) 

When `amountToMint == 0` the entire body is skipped: [2](#0-1) 

`msg.value` is only consumed inside that branch (split between `depositWithMessage` and `sendMessageToL2`): [3](#0-2) 

A grep across every L1 Solidity file for `withdraw`, `receive()`, and `fallback()` returns zero matches — confirming there is no ETH recovery path anywhere in `RewardSupplier`, `RewardSupplierStorage`, `RewardSupplierExternalInterfaces`, or `ProxySupportImpl`. [4](#0-3) [5](#0-4) 

---

### Impact Explanation

Any ETH attached to a `tick()` call that returns early (zero pending messages) is permanently locked in the contract. There is no owner-rescue, no `withdraw`, no `receive`/`fallback` that could drain it. The ETH balance of the contract increases monotonically with no mechanism to decrease it.

**Impact: High — permanent freezing of funds (caller's ETH irrecoverably locked).**

---

### Likelihood Explanation

- `tick()` is a public, permissionless function — no role check, no access control.
- The L2 mint-request queue is empty between minting epochs (the normal steady state).
- A caller who follows the spec diagram (which shows `anyAccount` calling `tick()` with a fee) and sends ETH during a quiet period loses those funds permanently.
- Requires no privileged access, no key compromise, no external dependency.

---

### Recommendation

Add a guard at the top of `tick()` that reverts if ETH is sent when it will not be used:

```solidity
function tick() external payable {
    (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();
    if (amountToMint == 0) {
        require(msg.value == 0, "NO_MINT_REQUIRED_NO_FEE_NEEDED");
        return;
    }
    // ... existing logic ...
}
```

Alternatively, remove `payable` and require callers to pre-fund the contract, or add an ETH rescue function restricted to governance.

---

### Proof of Concept

```solidity
// 1. Deploy RewardSupplier with no pending L2 mint messages.
// 2. Call tick{value: 1 ether}() from any EOA.
// 3. requiredMinting() returns (0, 0) → the if-branch is skipped.
// 4. assert(address(rewardSupplier).balance == 1 ether);
// 5. Confirm no withdraw/rescue function exists in the ABI.
// → 1 ETH is permanently locked.
```

        // Check if minting is required, and how much.
        (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

        if (amountToMint > 0) {
            // Prepare the L2->L1 mintRequest message for consumption.
            uint256[] memory messagePayload = new uint256[](1);
            messagePayload[0] = TOKENS_PER_MINT_REQUEST;

            // Consume the mintRequest messages.
            for (uint256 i = 0; i < numMsgsToConsume; i++) {
                messagingContract().consumeMessageFromL2(mintRequestSource(), messagePayload);
            }

            // Reuest minting of the requested amount from the mint manager.
            mintManager().mintRequest(token(), amountToMint);

            // Deposit the minted amount onto the bridge to the credit of `mintDestination`.
