### Title
STRK Rewards Permanently Frozen in Pool Contract When Pool Balance Falls Below `min_delegation_for_rewards` — (File: `src/pool/pool.cairo`)

---

### Summary

When a delegation pool's native-decimal balance is below `min_delegation_for_rewards`, the pool's `compute_rewards_per_unit` returns zero, but the staking contract has already transferred STRK rewards to the pool contract. Because the cumulative sigma is not updated, delegators can never claim those rewards, and no withdrawal function exists to recover them. The STRK tokens are permanently frozen.

---

### Finding Description

The staking contract's `_update_rewards` flow (staking.cairo lines 2313–2376) calculates pool rewards using **normalized 18-decimal amounts** and unconditionally transfers them to the pool contract before calling `update_rewards_from_staking_contract`. The pool contract then calls `compute_rewards_per_unit` with the **native-decimal** pool balance.

**Step 1 — Staking contract calculates non-zero pool rewards using normalized amounts:**

In `calculate_staker_pools_rewards` (staking.cairo lines 1979–1998), `pool_balance_curr_epoch.to_amount_18_decimals()` is used. For a BTC pool with 8 decimals and a balance of 999 satoshis, the normalized value is `999 × 10^10`. This can yield a non-zero `pool_rewards` even though the native balance is below threshold. [1](#0-0) 

**Step 2 — STRK rewards are transferred to the pool contract unconditionally:**

In `update_pool_rewards` (staking.cairo lines 1865–1891), for every pool with `pool_rewards > 0`, STRK is transferred to the pool contract via `send_rewards_to_delegation_pool` before `update_rewards_from_staking_contract` is called. [2](#0-1) 

**Step 3 — Pool's `compute_rewards_per_unit` returns zero for sub-threshold balance:**

Inside `update_rewards_from_staking_contract` (pool.cairo lines 571–587), `compute_rewards_per_unit` is called with the **native-decimal** `pool_balance`. If `total_stake < min_delegation_for_rewards`, it returns 0, so the cumulative sigma is not updated. [3](#0-2) 

The code comment at pool.cairo lines 964–966 explicitly acknowledges this mismatch: *"Delegation rewards lost when pool balance is less than `min_delegation_for_rewards`. The staking contract continues to forward `pool_rewards` to the pool contract even in this case."* [4](#0-3) 

**Step 4 — No recovery mechanism exists:**

Delegator rewards are computed as `compute_rewards_rounded_down(amount, to_sigma - from_sigma, base_value)` (pool.cairo lines 870–873). If sigma never increased, the difference is zero and delegators receive nothing. The pool contract has no admin withdrawal function to recover the stuck STRK. [5](#0-4) 

The `min_delegation_for_rewards` thresholds are:
- STRK (18 decimals): `10^(18−5) = 10^13` FRI (0.00001 STRK)
- BTC (8 decimals): `10^(8−5) = 1000` satoshis (0.00001 BTC) [6](#0-5) 

---

### Impact Explanation

STRK rewards that are minted, claimed from the reward supplier, and transferred to the pool contract are permanently unclaimable when the pool's native-decimal balance is below `min_delegation_for_rewards`. These tokens accumulate in the pool contract with every attestation epoch and can never be recovered. This constitutes **permanent freezing of unclaimed yield** (HIGH).

---

### Likelihood Explanation

Any unprivileged delegator can enter a pool with an amount just below `min_delegation_for_rewards` (e.g., 999 satoshis for an 8-decimal BTC pool). The threshold is very small (0.00001 of the token), making it easy to trigger either deliberately or accidentally. Once triggered, every subsequent attestation by the staker causes more STRK to be frozen. The entry path requires no privileged access.

---

### Recommendation

1. **In `calculate_staker_pools_rewards`**: Before calculating and forwarding pool rewards, check whether the pool's native-decimal balance is below `min_delegation_for_rewards`. If so, skip reward calculation and transfer for that pool entirely, consistent with what `compute_rewards_per_unit` would do.

2. **Alternatively**: Add a privileged `withdraw_excess_strk` function to the pool contract that allows recovery of STRK tokens not tracked by the cumulative sigma (i.e., `strk_balance - tracked_rewards`), analogous to the `withdraw()` function recommended in the external report.

---

### Proof of Concept

1. Staker stakes STRK and opens a BTC pool (8-decimal token, `min_delegation_for_rewards = 1000 satoshis`).
2. Delegator delegates **999 satoshis** to the pool.
3. After K epochs, staker attests; `update_rewards` is called.
4. `calculate_staker_pools_rewards` computes `pool_rewards > 0` using normalized 18-decimal amounts (`999 × 10^10`).
5. `update_pool_rewards` transfers the STRK rewards to the pool contract.
6. `update_rewards_from_staking_contract` calls `compute_rewards_per_unit(staking_rewards, total_stake=999)`.
7. `999 < 1000 = min_delegation_for_rewards` → returns 0; cumulative sigma unchanged.
8. Delegator calls `claim_rewards` → receives 0 STRK (sigma difference = 0).
9. STRK rewards are permanently frozen in the pool contract with no recovery path. [7](#0-6) [8](#0-7)

        /// Compute the rewards for the pool trace.
        ///
        /// `staking_rewards` is in `STRK_DECIMALS` decimals.
        /// `total_stake` is in the contract's token decimals.
        /// **Note**: Delegation rewards lost when pool balance is less than
        /// `min_delegation_for_rewards`. The staking contract continues to forward
        /// `pool_rewards` to the pool contract even in this case.
        fn compute_rewards_per_unit(
            self: @ContractState, staking_rewards: Amount, total_stake: Amount,
        ) -> Index {
            // Return zero if the total stake is too small, to avoid overflow below.
            if total_stake < self.min_delegation_for_rewards.read() {
                return Zero::zero();
            }
            mul_wide_and_div(
                lhs: staking_rewards, rhs: self.staking_rewards_base_value.read(), div: total_stake,
            )
                .expect_with_err(err: InternalError::REWARDS_COMPUTATION_OVERFLOW)
        }
