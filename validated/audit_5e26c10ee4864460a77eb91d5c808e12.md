### Title
Signature Validation Bypass via `skip_stateful_validations` Allows Unauthorized Invoke Admission — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator skips the account's `__validate__` entry point for any invoke transaction with `nonce == 1` when the account's on-chain nonce is `0` and `account_tx_in_pool_or_recent_block` returns `true`. Because `account_tx_in_pool_or_recent_block` returns `true` for **any** account that has a transaction in the mempool pool or in a recent committed block — not specifically a deploy-account transaction — an unprivileged attacker can submit an invoke with an arbitrary/invalid signature from any address that has a pending deploy-account in the mempool, and the gateway will admit it without ever calling `__validate__`.

---

### Finding Description

**Relevant code path:**

`extract_state_nonce_and_run_validations` → `run_pre_validation_checks` → `skip_stateful_validations` → `run_validate_entry_point`

**Step 1 — nonce check passes.**
`validate_nonce` for a non-deploy-account, non-declare transaction uses the "allowed nonce gap" branch:

```rust
// Other transactions must be within the allowed nonce range.
_ => {
    let max_allowed_nonce =
        Nonce(account_nonce.0 + Felt::from(self.config.max_allowed_nonce_gap));
    if !(account_nonce <= incoming_tx_nonce && incoming_tx_nonce <= max_allowed_nonce) {
        return Err(...)
    }
}
```

With `account_nonce = 0` and `tx_nonce = 1`, this passes as long as `max_allowed_nonce_gap >= 1`, which is required for the UX feature to function at all. [1](#0-0) 

**Step 2 — mempool validation passes.**
`validate_by_mempool` calls `mempool.validate_tx`, which only checks for duplicate tx_hash and fee-escalation rules. It does **not** verify signatures. [2](#0-1) 

**Step 3 — validation skip is triggered.**
`skip_stateful_validations` returns `true` (skip) when:
- The transaction is an `Invoke`
- `tx.nonce() == Nonce(Felt::ONE)`
- `account_nonce == Nonce(Felt::ZERO)`
- `account_tx_in_pool_or_recent_block(sender_address)` returns `true`

```rust
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
    return mempool_client
        .account_tx_in_pool_or_recent_block(tx.sender_address())
        .await
        ...
}
``` [3](#0-2) 

The comment claims this is safe because "it means that either it has a deploy_account transaction or transactions with future nonces that passed validations." This reasoning is **incorrect**: `account_tx_in_pool_or_recent_block` returns `true` if the address appears in the mempool pool **or** in the mempool's committed-block state, regardless of transaction type.

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [4](#0-3) 

**Step 4 — `__validate__` is not called.**
`run_validate_entry_point` sets `execution_flags.validate = !skip_validate`. When `skip_validate = true`, `execution_flags.validate = false`. `StatefulValidator::perform_validations` then returns early without calling `__validate__`:

```rust
if !tx.execution_flags.validate {
    return Ok(());
}
``` [5](#0-4) 

The transaction is admitted to the mempool with its signature completely unchecked. [6](#0-5) 

---

### Impact Explanation

An attacker who observes a pending deploy-account for address `A` in the mempool can submit an invoke transaction with `nonce = 1`, `sender_address = A`, and an **arbitrary or invalid signature**. The gateway admits it without signature verification. Concrete consequences:

1. **Unauthorized admission**: An invoke with a forged/invalid signature from an undeployed account is accepted into the mempool, violating the invariant that every admitted transaction must pass account-level authorization.
2. **Griefing via fee escalation**: If the attacker's fake invoke offers a higher tip than the legitimate user's invoke (also at nonce 1), the mempool's fee-escalation logic will replace the legitimate transaction with the attacker's invalid one, causing the legitimate invoke to be dropped.
3. **Block-space waste / DoS**: The admitted invalid invoke will be included in a block, fail during blockifier execution (when `__validate__` is called for real), and waste block space.

Matching impact: **High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

---

### Likelihood Explanation

- The attack requires no privileged access; any unprivileged user can submit an `RpcTransaction` to the gateway.
- The attacker only needs to observe the mempool for pending deploy-account transactions (public information).
- The attacker-controlled fields (`signature`, `calldata`) are unconstrained by any gateway check once the skip is triggered.
- The UX feature (`skip_stateful_validations`) is enabled by default and is exercised in integration tests. [7](#0-6) 

---

### Recommendation

Replace the coarse `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a **deploy-account** transaction exists for the address in the mempool. The mempool should expose a dedicated `has_pending_deploy_account(address)` query that inspects transaction type, rather than merely checking address presence. Alternatively, require the gateway to pass the deploy-account transaction hash (as the native blockifier's `PyValidator` does via `deploy_account_tx_hash`) and verify it refers to a deploy-account transaction type before skipping validation. [8](#0-7) 

---

### Proof of Concept

1. Legitimate user submits `deploy_account` for address `A` (class_hash, salt, constructor_calldata chosen by user). The gateway validates it fully (including `__validate_deploy__`) and it enters the mempool. `account_tx_in_pool_or_recent_block(A)` now returns `true`.

2. Attacker submits:
   ```
   RpcTransaction::Invoke(V3) {
       sender_address: A,
       nonce: 1,
       signature: [0xdeadbeef],   // arbitrary invalid signature
       calldata: [...],            // arbitrary calldata
       resource_bounds: { l2_gas: { max_amount: X, max_price_per_unit: Y } },
       tip: <higher than user's invoke tip>,
       ...
   }
   ```

3. Gateway stateless validation passes (no signature length limit exceeded, nonce DA mode L1, etc.).

4. `validate_nonce`: `account_nonce=0`, `tx_nonce=1`, `max_allowed_nonce_gap >= 1` → passes. [1](#0-0) 

5. `validate_by_mempool`: no duplicate hash, fee escalation check passes → passes.

6. `skip_stateful_validations`: `tx_nonce==1`, `account_nonce==0`, `account_tx_in_pool_or_recent_block(A)==true` → returns `true`. [9](#0-8) 

7. `run_validate_entry_point(skip_validate=true)`: `execution_flags.validate = false` → `__validate__` is never called → transaction admitted. [10](#0-9) 

8. Because the attacker's tip is higher, the mempool's fee-escalation logic replaces the legitimate user's invoke (nonce 1) with the attacker's invalid invoke. The legitimate invoke is dropped. The attacker's invalid invoke is sequenced, fails `__validate__` during blockifier execution, and the user's intended action never executes.

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}

    pub fn should_run_stateful_validations(
        &mut self,
        account_tx: &AccountTransaction,
        deploy_account_tx_hash: Option<TransactionHash>,
    ) -> StatefulValidatorResult<bool> {
        if account_tx.tx_type() != TransactionType::InvokeFunction {
            return Ok(true);
        }
        let tx_info = account_tx.create_tx_info();
        let nonce = self.stateful_validator.get_nonce(tx_info.sender_address())?;

        let deploy_account_not_processed =
            deploy_account_tx_hash.is_some() && nonce == Nonce(Felt::ZERO);
        let tx_nonce = tx_info.nonce();
        let is_post_deploy_nonce = Nonce(Felt::ONE) <= tx_nonce;
        let nonce_small_enough_to_qualify_for_validation_skip =
            tx_nonce <= self.max_nonce_for_validation_skip;

        let skip_validate = deploy_account_not_processed
            && is_post_deploy_nonce
            && nonce_small_enough_to_qualify_for_validation_skip;

        Ok(!skip_validate)
    }
