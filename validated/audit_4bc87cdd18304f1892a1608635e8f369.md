The core question is whether `setPoolBinAdditionalFees` enforces the `maxAdminSpreadFeeE6` cap. Let me verify the exact code path and the fee arithmetic.

### Title
Pool Admin Bypasses `maxAdminSpreadFeeE6` Cap via Uncapped `setPoolBinAdditionalFees` — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`setPoolBinAdditionalFees` forwards per-bin fee values directly to the pool with no validation against `maxAdminSpreadFeeE6`. The parallel function `setPoolAdminFees` enforces the cap; `setPoolBinAdditionalFees` does not. A pool admin can therefore set `addFeeBuyE6` / `addFeeSellE6` to `uint16` max (65 535 = 6.5535 %) on any bin, regardless of what `maxAdminSpreadFeeE6` is configured to, and those values are applied verbatim to every swap that crosses the bin.

---

### Finding Description

`setPoolAdminFees` enforces the cap:

```solidity
// MetricOmmPoolFactory.sol:414-415
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (newAdminNotionalFeeE8 > maxAdminNotionalFeeE8) revert AdminFeeTooHigh();
``` [1](#0-0) 

`setPoolBinAdditionalFees` has no such check — it is a one-line passthrough:

```solidity
// MetricOmmPoolFactory.sol:450-457
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
  external override nonReentrant onlyPoolAdmin(pool)
{
  IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
``` [2](#0-1) 

`setBinAdditionalFees` on the pool stores the values without any cap check:

```solidity
// MetricOmmPool.sol:464-474
function setBinAdditionalFees(int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
  external onlyFactory nonReentrant(PoolActions.SET_BIN_ADDITIONAL_FEES)
{
  if (bin < LOWEST_BIN || bin > HIGHEST_BIN) revert InvalidBinIndex(bin);
  BinState storage s = _binStates[bin];
  s.addFeeBuyE6 = addFeeBuyE6;
  s.addFeeSellE6 = addFeeSellE6;
  ...
}
``` [3](#0-2) 

Every swap through that bin then applies the uncapped value directly:

```solidity
// MetricOmmPool.sol:999 (buy token0, specified-in path)
params.baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6)
``` [4](#0-3) 

The same pattern appears in all four swap variants (specified-in / specified-out × buy token0 / buy token1). [5](#0-4) [6](#0-5) 

---

### Impact Explanation

`maxAdminSpreadFeeE6` is the on-chain guarantee that a pool admin cannot charge traders more than a declared ceiling. The factory owner can lower this cap (via `setFeeCaps`) to any value down to 0. [7](#0-6) 

Because `setPoolBinAdditionalFees` ignores the cap entirely, the pool admin can set per-bin fees to `uint16` max (65 535 = 6.5535 %) on every bin. If `maxAdminSpreadFeeE6` has been lowered to, say, 1 000 (0.1 %), the admin can still impose 65× the declared ceiling on every swap through any bin. Even at the default cap of 200 000 (20 %), the per-bin fee is additive on top of the base spread fee, pushing the total effective fee above the cap.

The concrete harm is that traders executing swaps through the affected bin pay a higher fee than the protocol's declared maximum, constituting a direct financial loss to traders and an admin-boundary break.

---

### Likelihood Explanation

The pool admin role is semi-trusted and the call is permissionless for that role — no timelock, no multisig requirement, no additional precondition. A single transaction from the pool admin address is sufficient to set any bin's fees to `uint16` max. The bypass is permanent until the admin resets the value or the factory owner lowers the cap further (which has no effect on already-stored bin fees).

---

### Recommendation

Add the same cap check in `setPoolBinAdditionalFees` that exists in `setPoolAdminFees`:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
  external override nonReentrant onlyPoolAdmin(pool)
{
  if (addFeeBuyE6  > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
  if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
  IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

`maxAdminSpreadFeeE6` is `uint24`; the comparison with `uint16` values is safe and always meaningful. [8](#0-7) 

---

### Proof of Concept

1. Factory owner calls `setFeeCaps(200_000, 1_000, 1_000_000, 1_000_000)` — admin spread cap lowered to 0.1 %.
2. Pool admin calls `setPoolAdminFees(pool, 1_001, 0)` → reverts `AdminFeeTooHigh`. Cap is enforced.
3. Pool admin calls `setPoolBinAdditionalFees(pool, 0, 65_535, 65_535)` → **succeeds**. No revert.
4. `_binStates[0].addFeeBuyE6 == 65_535` (6.5535 %) is now stored.
5. Any swap routed through bin 0 pays `baseFeeX64 + 65535/1e6` as its spread fee — 65× the declared admin cap — with the excess going to the LP fee pool rather than being blocked.
