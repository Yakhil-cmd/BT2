### Title
Unchecked Pre/Post NEP-141 Balance in `receive_erc20_tokens` Allows ERC-20 Over-Minting — (`engine/src/engine.rs`)

---

### Summary

`receive_erc20_tokens` in `engine/src/engine.rs` mints ERC-20 tokens on Aurora equal to the `amount` field supplied in the `FtOnTransferArgs` struct, without verifying that Aurora's actual NEP-141 balance increased by that amount. For fee-on-transfer NEP-141 tokens, the actual balance received is less than the claimed amount, causing Aurora to mint more ERC-20 tokens than it holds NEP-141 backing, resulting in insolvency for the bridged token pool.

---

### Finding Description

The `ft_on_transfer` entrypoint in `engine/src/contract_methods/connector.rs` is called by a NEP-141 token contract after a `ft_transfer_call`. It reads the transfer amount directly from the JSON arguments and passes it to `receive_erc20_tokens`: [1](#0-0) 

Inside `receive_erc20_tokens`, the amount used for minting is taken verbatim from `args.amount`: [2](#0-1) 

This amount is then passed directly to the ERC-20 `mint` call: [3](#0-2) 

At no point does the function query Aurora's NEP-141 balance before and after the transfer to confirm the actual amount received. The `FtOnTransferArgs.amount` field is the amount the NEP-141 contract *claims* was transferred — it is the original parameter from `ft_transfer_call`, not the actual balance delta at Aurora's account. [4](#0-3) 

For a fee-on-transfer NEP-141 token, the NEP-141 contract deducts a fee during the internal transfer, so Aurora's balance increases by `amount - fee`, but `ft_on_transfer` is still called with the original `amount`. Aurora then mints `amount` ERC-20 tokens while holding only `amount - fee` NEP-141 tokens.

---

### Impact Explanation

**Insolvency / Permanent fund freeze (Critical).**

Every deposit via a fee-on-transfer NEP-141 token inflates the ERC-20 supply beyond the NEP-141 backing held by Aurora. When users attempt to exit (burn ERC-20 → receive NEP-141 via `withdrawToNear`), the last users to exit will find the NEP-141 pool exhausted. Their ERC-20 tokens are permanently stranded with no NEP-141 to redeem, constituting both insolvency of the bridged token pool and permanent freezing of those funds.

---

### Likelihood Explanation

Any user can call `deploy_erc20_token` to register an arbitrary NEP-141 token on Aurora, and any user can then call `ft_transfer_call` on that NEP-141 contract to deposit into Aurora. There is no allowlist or fee-on-transfer guard in the bridge path. A fee-on-transfer NEP-141 token is a realistic and deployable contract on NEAR. The attacker does not need any privileged access.

---

### Recommendation

Before minting ERC-20 tokens, record Aurora's NEP-141 balance of the incoming token, then after the transfer completes, compare the post-balance to the pre-balance. Mint only the actual delta, not the claimed `args.amount`. Because NEAR is asynchronous, this requires a cross-contract view call to the NEP-141 contract before processing, or alternatively, the engine should explicitly document and enforce that fee-on-transfer NEP-141 tokens are not supported (e.g., by rejecting `deploy_erc20_token` calls for such tokens, or by adding a post-transfer callback that reconciles the minted amount against the actual balance change).

---

### Proof of Concept

1. Deploy a NEP-141 token on NEAR that charges a 10% fee on every transfer (i.e., transferring 100 tokens results in the recipient receiving 90 tokens).
2. Call `deploy_erc20_token` on Aurora Engine to register this NEP-141 token and create its ERC-20 mirror.
3. Call `ft_transfer_call` on the NEP-141 contract with `receiver_id = aurora`, `amount = 1000`, `msg = <evm_recipient_address>`.
4. The NEP-141 contract transfers 900 tokens to Aurora (after 10% fee) and calls `ft_on_transfer` on Aurora with `amount = 1000`.
5. Aurora's `ft_on_transfer` calls `receive_erc20_tokens` with `amount = 1000`.
6. `receive_erc20_tokens` mints 1000 ERC-20 tokens to the EVM recipient.
7. Aurora now holds 900 NEP-141 tokens but has issued 1000 ERC-20 tokens — a 100-token deficit.
8. Repeat deposits to widen the deficit. Eventually, legitimate depositors who deposited standard (non-fee) NEP-141 tokens cannot fully exit because the pool is insolvent. [5](#0-4) [6](#0-5)

#[named]
pub fn ft_on_transfer<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();
        let mut engine: Engine<_, _> = Engine::new(
            predecessor_address(&predecessor_account_id),
            current_account_id.clone(),
            io,
            env,
        )?;

        sdk::log!("Call ft_on_transfer");

        let args: FtOnTransferArgs = read_json_args(&io)?;
        let result = if predecessor_account_id == get_connector_account_id(&io)? {
            engine.receive_base_tokens(&args)
        } else {
            engine.receive_erc20_tokens(
                &predecessor_account_id,
                &args,
                &current_account_id,
                handler,
            )
        };

        #[allow(clippy::used_underscore_binding)]
        let amount_to_return = if let Err(_err) = &result {
            sdk::log!("Error in ft_on_transfer: {_err:?}");
            // An error occurred, so we need to return the amount of tokens to the sender.
            args.amount.as_u128()
        } else {
            // Everything is ok, so return 0.
            0
        };

        let output = crate::prelude::format!("\"{amount_to_return}\"");
        io.return_output(output.as_bytes());

        // In case of an error, we just return Ok(None) to avoid a panic in the contract. It's ok
        // because in case of an error, we already returned the amount of tokens to the sender.
        Ok(result.unwrap_or(None))
    })
}

    pub fn receive_erc20_tokens<P: PromiseHandler>(
        &mut self,
        token: &AccountId,
        args: &FtOnTransferArgs,
        current_account_id: &AccountId,
        handler: &mut P,
    ) -> Result<Option<SubmitResult>, ContractError> {
        let amount = args.amount.as_u128();
        // Parse message to determine recipient
        let mut recipient = {
            // The message should contain the recipient EOA address.
            let message = args.msg.strip_prefix("0x").unwrap_or(&args.msg);
            // Recipient - 40 characters (Address in hex without '0x' prefix)
            if message.len() < 40 {
                return Err(ParseOnTransferMessageError::WrongMessageFormat.into());
            }
            let mut address_bytes = [0; 20];
            hex::decode_to_slice(&message[..40], &mut address_bytes)
                .map_err(|_| ParseOnTransferMessageError::WrongMessageFormat)?;
            Address::from_array(address_bytes)
        };

        if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
            && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
        {
            recipient = fallback_address;
        }

        let erc20_token = get_erc20_from_nep141(&self.io, token)?;
        let erc20_admin_address = current_address(current_account_id);
        let result = self
            .call(
                &erc20_admin_address,
                &erc20_token,
                Wei::zero(),
                setup_receive_erc20_tokens_input(&recipient, amount),
                u64::MAX,
                Vec::new(), // TODO: are there values we should put here?
                Vec::new(),
                handler,
            )
            .and_then(submit_result_or_err)?;

        sdk::log!("Mint {amount} ERC-20 tokens for: {}", recipient.encode());

        // Return SubmitResult so that it can be accessed in standalone engine.
        // This is used to help with the indexing of bridge transactions.
        Ok(Some(result))
    }
