The vulnerability is real. Here is the analysis:

**Key facts from the code:**

1. `push_message` at [1](#0-0)  uses a strict `< MessageHeader::LEN` guard — a message of **exactly** 40 bytes passes this check and is accepted into the buffer.

2. `MessageHeader::LEN` is 40 bytes (32-byte `ChannelTag` + 8-byte `u64` waitpoint): [2](#0-1) 

3. In `Comms::recv`, `data.get(MessageHeader::LEN..)` on a 40-byte slice returns `Some(&[])` — an empty slice, **not** `None` — so the `ok_or_else` guard at [3](#0-2)  does **not** fire. Instead, `rmp_serde::decode::from_slice(&[])` is called on an empty slice, which fails, and `decoded?` at [4](#0-3)  propagates the error.

4. In `recv_from_others`, the error is propagated immediately via `?` at [5](#0-4)  — before any participant-list filtering (`seen.put(from)`) can occur.

5. Channel tags are deterministic (SHA256-based), so a malicious participant who is part of the protocol can compute the correct 40-byte header for any waitpoint and inject it.

### Title
Malicious participant aborts honest party's protocol via 40-byte header-only message — (`src/protocol/internal.rs`)

### Summary
A malicious protocol participant can send a message of exactly `MessageHeader::LEN` (40) bytes — a valid header with no payload — which passes `push_message`'s length guard, is routed into the correct message slot, and causes `Comms::recv` to fail deserialization on an empty slice. This error propagates through `recv_from_others` via `?`, aborting the honest participant's protocol execution.

### Finding Description
`push_message` silently drops messages shorter than `MessageHeader::LEN` bytes but **accepts** messages of exactly that length:

```rust
// src/protocol/internal.rs:287-289
if message.len() < MessageHeader::LEN {
    return;
}
```

A 40-byte message passes this check. The header is parsed successfully and the message is pushed to the incoming buffer. When `Comms::recv` later pops it:

```rust
// src/protocol/internal.rs:335-340
let message_data = data.get(MessageHeader::LEN..).ok_or_else(|| { ... })?;
// data.get(40..) on a 40-byte slice = Some(&[]) — no error here
let decoded: Result<T, _> = rmp_serde::decode::from_slice(message_data)...;
Ok((from, decoded?))  // decoded fails on empty slice → error propagates
```

`recv_from_others` propagates this immediately:

```rust
// src/protocol/helpers.rs:20
let (from, msg) = chan.recv(waitpoint).await?;  // ? aborts on error
```

### Impact Explanation
**High — Permanent denial of signing, key generation, reshare, refresh, or CKD for honest parties.** Any single malicious participant in the protocol can abort every honest participant's execution at any `recv_from_others` round by injecting one 40-byte message per round. Since the channel tags are deterministic and known to all participants, the attacker can target any specific waitpoint. Retrying the protocol does not help as the attacker can repeat the attack indefinitely.

### Likelihood Explanation
The attacker is a legitimate protocol participant (no external assumptions needed). Channel tags are computed deterministically from public constants, so the attacker can craft the correct 40-byte header for any round without any secret knowledge. The exploit requires sending a single small message.

### Recommendation
Change the guard in `push_message` from strict-less-than to less-than-or-equal, so header-only messages are also silently dropped:

```rust
// src/protocol/internal.rs:287
if message.len() <= MessageHeader::LEN {
    return;
}
```

Alternatively, add the same guard inside `Comms::recv` to return a non-fatal error that `recv_from_others` can skip rather than propagate.

### Proof of Concept
1. Participant A is malicious; participants B and C are honest in a 3-party protocol.
2. During any `recv_from_others` round, A sends a 40-byte message: the 32-byte SHA256 channel tag for the shared root channel concatenated with the 8-byte little-endian waitpoint value (both computable from public constants).
3. `push_message` accepts it (`40 < 40` is false).
4. B's `recv_from_others` pops the message, `rmp_serde::decode::from_slice(&[])` fails, `?` propagates the error, and B aborts.
5. C similarly aborts when it pops A's message.
6. The protocol never completes despite B and C being fully honest.
