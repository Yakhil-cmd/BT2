### Title
Unguarded `new` Initializer Allows Front-Running to Seize Engine Ownership - (File: `engine/src/contract_methods/admin.rs`)

### Summary
The Aurora Engine's `new` initialization function imposes no caller restriction. Any NEAR account can call it before the legitimate deployer does. Because deployment and initialization are two separate transactions, an attacker who observes the deploy transaction can race to call `new` first, supplying their own `owner_id`. Once owner, the attacker can invoke `attach_full_access_key` to add a full-access key to the Aurora Engine NEAR account, gaining complete control over all bridged assets.

### Finding Description
The `new` function is the sole initialization path for the Aurora Engine contract state. Its only guard is a check that state does not already exist:

```rust
pub fn new<I: IO + Copy, E: Env>(mut io: I, env: &E) -> Result<(), ContractError> {
    if state::get_state(&io).is_ok() {
        return Err(b"ERR_ALREADY_INITIALIZED".into());
    }
    // ...
    let state: EngineState = args.into();
    state::set_state(&mut io, &state)?;
    Ok(())
}
``` [1](#0-0) 

There is no check on `env.predecessor_account_id()`. The `owner_id` field written into `EngineState` is taken entirely from the caller-supplied `NewCallArgs`: [2](#0-1) 

This function is exposed as a public, permissionless NEAR entrypoint: [3](#0-2) 

The standard deployment helper (`deploy_and_init`) issues the `DeployContract` action and the `new` call as **two separate transactions**, creating a window between them: [4](#0-3) 

### Impact Explanation
`owner_id` is the root of all administrative authority in the engine. The most severe downstream capability is `attach_full_access_key`, which adds a full-access key to the Aurora Engine NEAR account itself: [5](#0-4) 

The code comment at line 505 explicitly acknowledges this is dangerous and relies entirely on `require_owner_only` for safety. An attacker who controls `owner_id` bypasses that check and can add their own key, then use it to drain all ETH and ERC-20 tokens held by the engine. Additional owner-gated paths (`upgrade`, `stage_upgrade`, `set_owner`) provide further escalation routes.

**Impact class:** Critical — direct theft of all user funds at rest in the Aurora Engine.

### Likelihood Explanation
On NEAR Protocol, `DeployContract` and a subsequent `FunctionCall` to `new` are separate transactions. A monitoring bot can observe the deploy transaction in a block, then submit a `new` call in the very next block before the deployer's own `new` transaction is included. No special privilege is required — any NEAR account can call `new`. The attack is deterministic and requires no brute force.

### Recommendation
Add a predecessor check at the top of `new` that requires the caller to be the contract account itself (i.e., `env.predecessor_account_id() == env.current_account_id()`). This enforces that only a self-call — which can only originate from a batch transaction that also deploys the contract — can initialize the engine. Alternatively, combine the `DeployContract` and `FunctionCall` to `new` into a single NEAR batch action so they execute atomically, eliminating the race window entirely.

### Proof of Concept

1. Deployer broadcasts `DeployContract` for the Aurora Engine WASM to account `aurora`.
2. Attacker observes this transaction included in block N.
3. In block N+1, attacker submits:
   ```
   aurora.new({
     chain_id: <legitimate chain id>,
     owner_id: "attacker.near",
     upgrade_delay_blocks: 0,
     key_manager: "attacker.near"
   })
   ```
   This succeeds because `state::get_state` returns `Err` (state not yet set), so the `ERR_ALREADY_INITIALIZED` guard does not fire. [6](#0-5) 
4. Deployer's own `new` call arrives in block N+1 or later and reverts with `ERR_ALREADY_INITIALIZED`.
5. Attacker, now `owner_id`, calls `attach_full_access_key` with their own public key: [7](#0-6) 
6. With a full-access key on the `aurora` NEAR account, the attacker can transfer all NEAR, ETH, and NEP-141 tokens held by the engine to themselves.

pub fn new<I: IO + Copy, E: Env>(mut io: I, env: &E) -> Result<(), ContractError> {
    if state::get_state(&io).is_ok() {
        return Err(b"ERR_ALREADY_INITIALIZED".into());
    }

    let input = io.read_input().to_vec();
    let args = NewCallArgs::deserialize(&input).map_err(|_| errors::ERR_BORSH_DESERIALIZE)?;

    let initial_hashchain = args.initial_hashchain();
    let state: EngineState = args.into();

    if let Some(block_hashchain) = initial_hashchain {
        let block_height = env.block_height();
        let mut hashchain = Hashchain::new(
            state.chain_id,
            env.current_account_id(),
            block_height,
            block_hashchain,
        );

        hashchain.add_block_tx(
            block_height,
            function_name!(),
            &input,
            &[],
            &Bloom::default(),
        )?;
        crate::hashchain::save_hashchain(&mut io, &hashchain)?;
    }

    state::set_state(&mut io, &state)?;
    Ok(())
}

pub fn attach_full_access_key<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<(), ContractError> {
    let state = state::get_state(&io)?;

    require_running(&state)?;
    require_owner_only(&state, &env.predecessor_account_id())?;

    let public_key = serde_json::from_slice::<FullAccessKeyArgs>(&io.read_input().to_vec())
        .map(|args| args.public_key)
        .map_err(|_| errors::ERR_JSON_DESERIALIZE)?;
    let current_account_id = env.current_account_id();
    let action = PromiseAction::AddFullAccessKey {
        public_key,
        nonce: 0, // not actually used - depends on block height
    };
    let promise = PromiseBatchAction {
        target_account_id: current_account_id,
        actions: vec![action],
    };
    // SAFETY: This action is dangerous because it adds a new full access key (FAK) to the Engine account.
    // However, it is safe to do so here because of the `require_owner_only` check above; only the
    // (trusted) owner account can add a new FAK.
    let promise_id = handler.promise_create_batch(&promise);

    handler.promise_return(promise_id);

    Ok(())
