### Title
Gateway `skip_stateful_validations` Bypasses `__validate__` for Nonce-1 Invoke Transactions via Overly Broad Pool-Membership Check — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

`skip_stateful_validations` is intended to skip the `__validate__` entry-point only when a `deploy_account` transaction is pending in the mempool (UX feature: deploy + invoke submitted together). The guard it uses — `account_tx_in_pool_or_recent_block` — returns `true` for **any** transaction type in the pool, not exclusively `deploy_account`. An attacker who controls an already-deployed account with on-chain nonce 0 can first submit a valid nonce-0 invoke, then submit a nonce-1 invoke with an **invalid signature or rejected calldata**, and the gateway will admit the second transaction to the mempool without ever calling `__validate__`.

---

### Finding Description

`skip_stateful_validations` in `crates/apollo_gateway/src/stateful_transaction_validator.rs` fires when three conditions hold simultaneously:

1. The incoming transaction is an `Invoke`.
2. `tx.nonce() == Nonce(Felt::ONE)`.
3. `account_nonce == Nonce(Felt::ZERO)` (on-chain state nonce).
4. `mempool_client.account_tx_in_pool_or_recent_block(sender)` returns `true`. [1](#0-0) 

When it returns `true`, `run_validate_entry_point` is called with `validate: false`, meaning the blockifier's `__validate__` entry-point is **never executed** at the gateway level: [2](#0-1) 

The guard `account_tx_in_pool_or_recent_block` is implemented as:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [3](#0-2) 

`tx_pool.contains_account` returns `true` for **any** transaction type stored for that address — `Invoke`, `Declare`, or `DeployAccount`: [4](#0-3) 

The inline comment in `skip_stateful_validations` claims:

> "It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations." [5](#0-4) 

This reasoning is **incorrect** for an already-deployed account whose on-chain nonce is still 0. Such an account can have an `Invoke(nonce=0)` in the pool (which passed its own `__validate__` check), satisfying `tx_pool.contains_account` without any `deploy_account` ever being submitted. The subsequent nonce-1 invoke then enters the mempool with `validate: false`.

---

### Impact Explanation

An invoke transaction carrying an **invalid signature** or **calldata that the account's custom `__validate__` would normally reject** is admitted to the mempool without gateway-level account validation. This matches:

> **High — Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.**

The transaction will fail during batcher execution (the batcher calls `__validate__` in strict mode), but it has already been admitted, consuming mempool capacity and batcher execution resources. For accounts with custom `__validate__` logic (e.g., multi-sig, session-key, or allowlist contracts), this bypasses the intended admission invariant: every transaction entering the mempool must have passed the account's own validation logic.

---

### Likelihood Explanation

The trigger requires only that the attacker controls an account whose on-chain nonce is 0 (a freshly deployed but never-used account). The attacker:

1. Submits `Invoke(nonce=0, valid_signature)` — passes `__validate__`, enters pool.
2. Submits `Invoke(nonce=1, invalid_signature_or_rejected_calldata)` — `skip_stateful_validations` returns `true`, `__validate__` skipped, enters pool.

No privileged access is required beyond owning the account. The attack is repeatable for any account that has not yet consumed its nonce-0 slot.

---

### Recommendation

Replace the overly broad `account_tx_in_pool_or_recent_block` check with a check that specifically confirms a `DeployAccount` transaction is pending for the sender address. The mempool should expose a dedicated query such as `deploy_account_in_pool(address) -> bool` that inspects only `DeployAccount` entries. Alternatively, tag the pool entry type and filter accordingly inside `skip_stateful_validations`.

---

### Proof of Concept

```
1. Deploy contract C at address A (on-chain nonce of A becomes 0 after deployment).
   (A is now a live account with nonce=0 in state.)

2. Submit Invoke(sender=A, nonce=0, calldata=X, signature=VALID_SIG).
   → Gateway: account_nonce=0, tx_nonce=0 → skip_stateful_validations returns false
     → __validate__ runs, passes → tx enters mempool.
   → tx_pool.contains_account(A) is now true.

3. Submit Invoke(sender=A, nonce=1, calldata=FORBIDDEN_CALLDATA, signature=GARBAGE).
   → Gateway: account_nonce=0, tx_nonce=1
     → skip_stateful_validations: nonce==1 ✓, account_nonce==0 ✓,
       account_tx_in_pool_or_recent_block(A) == true (due to step 2) ✓
     → validate flag set to false → __validate__ NOT called
   → tx admitted to mempool with invalid/rejected payload.

4. Batcher picks up both transactions:
   - Invoke(nonce=0) executes successfully.
   - Invoke(nonce=1) hits __validate__ in strict execution mode → reverts.
   
Result: A transaction that should have been rejected at the gateway
        (invalid signature / custom __validate__ rejection) was admitted
        to the mempool, bypassing the account's admission invariant.
``` [6](#0-5) [3](#0-2)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
