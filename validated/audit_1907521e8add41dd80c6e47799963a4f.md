### Title
Receipt Size Limit Bypassed via `promise_return` Post-Validation `output_data_receivers` Mutation - (File: `runtime/runtime/src/lib.rs`)

### Summary

A receipt created by a contract via `promise_return` (the `ReturnData::ReceiptIndex` path) is validated against `max_receipt_size` before the runtime appends the parent receipt's `output_data_receivers` to it. This post-validation mutation can push the receipt's serialized size above `max_receipt_size` (4 MiB), violating the hard invariant that all receipts in the system must be ≤ `max_receipt_size`. The nearcore team acknowledges this as an open bug (issue #12606) and has added workarounds in production code, but the root cause remains unfixed.

### Finding Description

The receipt size check in `validate_receipt` is gated on `ValidateReceiptMode::NewReceipt`:

```rust
// runtime/runtime/src/verifier.rs:533-541
if mode == ValidateReceiptMode::NewReceipt {
    let receipt_size: u64 =
        borsh::object_length(receipt).unwrap().try_into()...;
    if receipt_size > limit_config.max_receipt_size {
        return Err(ReceiptValidationError::ReceiptSizeExceeded { ... });
    }
}
``` [1](#0-0) 

After this check passes, the runtime processes the `ReturnData::ReceiptIndex` result path and **mutates** the already-validated receipt by appending the parent receipt's `output_data_receivers` to it without re-checking the size:

```rust
// runtime/runtime/src/lib.rs:1020-1037
if let Ok(ReturnData::ReceiptIndex(receipt_index)) = result.result {
    match result.new_receipts.get_mut(receipt_index as usize)...receipt_mut() {
        ReceiptEnum::Action(new_action_receipt) | ... => new_action_receipt
            .output_data_receivers
            .extend_from_slice(&action_receipt.output_data_receivers()),
        ...
    }
}
``` [2](#0-1) 

A contract can craft a promise DAG `[A -then-> B]` where promise A creates receipt C at exactly `max_receipt_size` bytes (passing validation), then calls `promise_return(C)`. The runtime then appends B's `output_data_receivers` to C's receipt, pushing it above `max_receipt_size`. The resulting oversized receipt enters the outgoing buffer and is forwarded to the receiving shard.

The `ExistingReceipt` mode was introduced specifically to tolerate these oversized receipts, confirming the invariant is broken in production:

The congestion control layer also has a workaround that clamps the oversized receipt's size to `max_receipt_size` to prevent it from getting permanently stuck in the outgoing buffer:

### Impact Explanation

The `max_receipt_size` limit (4 MiB) is a hard limit designed to keep `ChunkStateWitness` under the 17 MiB target. Oversized receipts that bypass this limit inflate the state witness beyond its design budget. The `outgoing_receipts_usual_size_limit` (100 KiB) and `outgoing_receipts_big_size_limit` (4.5 MiB) are also undermined because the clamping workaround causes the congestion control layer to account for less bandwidth than the receipt actually consumes. This can cause state witnesses to exceed the 17 MiB target, potentially causing chunk validation failures or degraded network performance.

**Scope:** Receipt causality — the receipt size invariant is broken through the `promise_return` mechanism.

### Likelihood Explanation

Any unprivileged user can trigger this by deploying a contract that:
1. Creates a promise DAG `[A -then-> B]`
2. When A executes, creates receipt C with args sized to fill `max_receipt_size` minus the base receipt overhead
3. Calls `promise_return(C)`

The test `test_max_receipt_size_promise_return` in `test-loop-tests/src/tests/max_receipt_size.rs` demonstrates this is reproducible and confirms the oversized receipt reaches the incoming receipts of the target shard. [6](#0-5) 

### Recommendation

Re-validate the receipt size **after** appending `output_data_receivers` in the `ReturnData::ReceiptIndex` branch of `runtime/runtime/src/lib.rs`. If the post-mutation size exceeds `max_receipt_size`, the `promise_return` should fail with `ReceiptSizeExceeded` rather than silently producing an oversized receipt. This is the same check already applied to directly-created receipts in `validate_receipt` with `ValidateReceiptMode::NewReceipt`.

### Proof of Concept

1. Deploy a contract with a method that:
   - Creates promise DAG `[A -then-> B]` via `promise_create` + `promise_then`
   - When A executes: creates receipt C with `args` sized to `max_receipt_size - base_receipt_overhead` bytes, then calls `promise_return(C)`
2. The runtime validates C at creation (passes, size == `max_receipt_size`)
3. The runtime then appends B's `output_data_receivers` to C (size > `max_receipt_size`)
4. C is forwarded to the receiving shard with size > 4 MiB
5. `assert_oversized_receipt_occurred` confirms the oversized receipt is present in the chain

The exact args size calculation is demonstrated in the test at `test-loop-tests/src/tests/max_receipt_size.rs:175-177`:

```rust
