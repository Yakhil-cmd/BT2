Audit Report

## Title
SNS Governance `cast_vote_and_cascade_follow` Performs Unbounded Synchronous BFS Without Instruction-Limit Checks - (`rs/sns/governance/src/governance.rs`)

## Summary
The SNS governance canister's `cast_vote_and_cascade_follow` function executes a synchronous BFS over the entire neuron follow graph with no per-iteration instruction-limit checks or mechanism to defer work across messages. Unlike the NNS counterpart, which uses `noop_self_call_if_over_instructions` and a `ProposalVotingStateMachine` to break work across timer invocations, the SNS version runs to completion or traps the entire `register_vote` message. With up to 200,000 neurons and an unbounded fan-in, a sufficiently large SNS community (achievable through normal swap participation) can cause `register_vote` to consistently trap, leaving the root neuron's ballot unrecorded and blocking proposal execution.

## Finding Description
**Root cause:** `cast_vote_and_cascade_follow` in `rs/sns/governance/src/governance.rs` (L3749–3836) is a plain synchronous function containing a `while !induction_votes.is_empty()` BFS loop with no instruction-limit check inside: [1](#0-0) 

Each BFS tier performs `BTreeMap`/`BTreeSet` lookups and insertions for every follower neuron. There is no `async`, no `await`, no soft/hard limit check, and no timer-based deferral path.

**Contrast with NNS:** The NNS `cast_vote_and_cascade_follow` in `rs/nns/governance/src/voting.rs` (L150–176) explicitly calls `noop_self_call_if_over_instructions` with `SOFT_VOTING_INSTRUCTIONS_LIMIT` and `HARD_VOTING_INSTRUCTIONS_LIMIT` inside its processing loop, breaking work across messages and deferring remaining ballot recording to timer jobs: [2](#0-1) 

**Call path:** `register_vote` (L3854–3947) is a synchronous `fn` (not `async fn`) that calls `cast_vote_and_cascade_follow` directly and unconditionally: [3](#0-2) 

**Scale bounds:** `MAX_NUMBER_OF_NEURONS_CEILING = 200_000` sets the upper bound on BFS nodes: [4](#0-3) 

The `follow` function enforces a per-neuron followee limit (fan-out ≤ `max_followees_per_function`) but places **no limit on how many neurons can follow a given neuron** (fan-in is unbounded): [5](#0-4) 

**Existing guards are insufficient:** The only guard in `follow` is a fan-out check. There is no fan-in cap, no BFS depth limit, and no instruction budget check anywhere in the SNS vote cascade path.

## Impact Explanation
When the BFS exhausts the per-message instruction limit, the `register_vote` update call traps. The neuron's ballot is not recorded. If the trapping neuron is a root neuron followed by thousands of swap-participant neurons, its vote cannot be cast, and any proposal requiring that neuron's vote to reach quorum cannot be decided. This constitutes a governance-level DoS: a critical SNS governance operation becomes impossible to execute, matching the **High** impact class — "Application/platform-level DoS... or significant SNS security impact with concrete user or protocol harm."

## Likelihood Explanation
No malicious action is required. As of Proposal 136582, SNS swap neuron baskets are configured so all non-root neurons in each basket follow the root neuron on **all topics, including critical ones**: [6](#0-5) 

A swap with tens of thousands of participants naturally creates a fan-in of that magnitude on the root neuron. The NNS integration test `test_voting_can_span_multiple_rounds` demonstrates that even 999 neurons in a follow chain already exceeds the per-message instruction budget in the NNS (which has the mitigation), confirming the SNS synchronous path would trap under similar conditions: [7](#0-6) 

The condition is therefore reachable by any SNS with a large enough swap, triggered by any neuron owner calling `register_vote` on the root neuron.

## Recommendation
1. Port the NNS instruction-limit pattern to SNS: make `cast_vote_and_cascade_follow` (or its caller `register_vote`) `async`, add per-iteration calls to `noop_self_call_if_over_instructions` inside the BFS loop in `rs/sns/governance/src/governance.rs`, and introduce a `ProposalVotingStateMachine` equivalent for SNS to persist and resume BFS state across messages/timer invocations.
2. Alternatively, enforce a hard cap on the number of neurons that can follow a single neuron (fan-in limit), bounding the BFS work per vote to a safe per-message budget.
3. Add an SNS integration test analogous to `test_voting_can_span_multiple_rounds` to catch regressions.

## Proof of Concept
1. Deploy an SNS with `max_number_of_neurons` set to a large value (e.g., 50,000+).
2. Conduct a swap with many participants; each participant's neuron basket follows the root neuron on all topics (default post-Proposal 136582).
3. Have the root neuron call `register_vote` on an open proposal.
4. Observe that `cast_vote_and_cascade_follow` enters the BFS loop at `rs/sns/governance/src/governance.rs:3749` and iterates over all follower neurons synchronously with no instruction-limit escape hatch.
5. With a sufficiently large swap, the message exhausts the per-message instruction limit and traps, leaving the root neuron's ballot unrecorded and the proposal undecidable.
6. A deterministic reproduction can be constructed as an SNS integration test mirroring `test_voting_can_span_multiple_rounds` in `rs/nns/integration_tests/src/neuron_voting.rs`, substituting SNS governance and a large swap-participant neuron set.
