### Title
`FunctionInvocation.execution_resources.gas_consumed` Hardcoded to Zero in All RPC Simulation/Trace Responses — (`crates/apollo_rpc_execution/src/objects.rs`)

---

### Summary

`vm_resources_to_execution_resources` unconditionally sets `gas_consumed` to `StarknetApiGasVector::default()` (all zeros) for every `FunctionInvocation` produced during RPC simulation and tracing. Any Cairo 1 contract that consumes nonzero L2 gas will have its per-call `gas_consumed` silently zeroed in the trace, making `starknet_simulateTransactions` and `starknet_traceTransaction` return authoritative-looking but incorrect values.

---

### Finding Description

In `vm_resources_to_execution_resources`, the `gas_consumed` field of the returned `ExecutionResources` is hardcoded:

```rust
Ok(ExecutionResources {
    steps: u64_from_usize(vm_resources.n_steps),
    builtin_instance_counter,
    memory_holes: u64_from_usize(vm_resources.n_memory_holes),
    da_gas_consumed: StarknetApiGasVector { l1_gas, l2_gas, l1_data_gas },
    gas_consumed: StarknetApiGasVector::default(),   // always zero
})
``` [1](#0-0) 

The function receives a `GasVector { l1_gas, l1_data_gas, l2_gas }` parameter and correctly maps it into `da_gas_consumed`, but `gas_consumed` is never populated from any source — it is always the zero default.

This function is called from `FunctionInvocation::try_from`, which is the conversion path for every call frame in a simulation or trace: [2](#0-1) 

Additionally, the same outer `gas_vector` is propagated unchanged to all inner calls: [3](#0-2) 

This means even `da_gas_consumed` is wrong for inner calls (they all receive the top-level gas vector), but `gas_consumed` is wrong for every call frame unconditionally.

---

### Impact Explanation

Every response from `starknet_simulateTransactions` and `starknet_traceTransaction` will report `gas_consumed = { l1_gas: 0, l2_gas: 0, l1_data_gas: 0 }` for every `FunctionInvocation.execution_resources`, regardless of actual L2 gas consumed. This is an authoritative-looking wrong value in the RPC trace/simulation surface.

Developers and tooling that rely on per-call `gas_consumed` in traces to profile gas usage, detect gas-heavy calls, or build gas estimation tooling will receive systematically incorrect data. The impact is confined to the RPC layer — actual fee charging and blockifier-internal accounting are unaffected, so there is no economic or state-mutation consequence.

---

### Likelihood Explanation

The bug is unconditional and triggered by any call to the simulation or tracing RPC endpoints for any Cairo 1 contract that consumes L2 gas. No special attacker action is required; any unprivileged user invoking `starknet_simulateTransactions` or `starknet_traceTransaction` will observe the wrong value.

---

### Recommendation

Populate `gas_consumed` from `call_info.resources` rather than hardcoding `default()`. The blockifier's `CallInfo` already tracks per-call gas consumption; the conversion should extract the actual L2 gas consumed from `call_info.execution.gas_consumed` (or the equivalent field in the blockifier's resource tracking) and assign it to `gas_consumed` in the returned `ExecutionResources`. Inner calls should similarly use their own `call_info.resources` rather than inheriting the outer `gas_vector`.

---

### Proof of Concept

1. Deploy a Cairo 1 contract with an entry point that performs a storage write (consuming nonzero L2 gas under a block context with nonzero `l2_gas_price`).
2. Call `starknet_simulateTransactions` targeting that entry point.
3. Inspect the returned `transaction_trace` → `execute_invocation` → `execution_resources.gas_consumed`.
4. Observe `{ l1_gas: 0, l2_gas: 0, l1_data_gas: 0 }` regardless of actual L2 gas consumed, confirming the hardcoded zero at line 399. [4](#0-3)

fn vm_resources_to_execution_resources(
    vm_resources: VmExecutionResources,
    GasVector { l1_gas, l1_data_gas, l2_gas }: GasVector,
) -> ExecutionResult<ExecutionResources> {
    let mut builtin_instance_counter = HashMap::new();
    for (builtin_name, count) in vm_resources.builtin_instance_counter {
        if count == 0 {
            continue;
        }
        let count = u64_from_usize(count);
        match builtin_name {
            BuiltinName::output => continue,
            BuiltinName::pedersen => builtin_instance_counter.insert(Builtin::Pedersen, count),
            BuiltinName::range_check => builtin_instance_counter.insert(Builtin::RangeCheck, count),
            BuiltinName::ecdsa => builtin_instance_counter.insert(Builtin::Ecdsa, count),
            BuiltinName::bitwise => builtin_instance_counter.insert(Builtin::Bitwise, count),
            BuiltinName::ec_op => builtin_instance_counter.insert(Builtin::EcOp, count),
            BuiltinName::keccak => builtin_instance_counter.insert(Builtin::Keccak, count),
            BuiltinName::poseidon => builtin_instance_counter.insert(Builtin::Poseidon, count),
            BuiltinName::segment_arena => {
                builtin_instance_counter.insert(Builtin::SegmentArena, count)
            }
            // TODO(DanB): what about the following?
            // BuiltinName::range_check96 => todo!(),
            // BuiltinName::add_mod => todo!(),
            // BuiltinName::mul_mod => todo!(),
            _ => {
                return Err(ExecutionError::UnknownBuiltin { builtin_name });
            }
        };
    }
    Ok(ExecutionResources {
        steps: u64_from_usize(vm_resources.n_steps),
        builtin_instance_counter,
        memory_holes: u64_from_usize(vm_resources.n_memory_holes),
        da_gas_consumed: StarknetApiGasVector { l1_gas, l2_gas, l1_data_gas },
        gas_consumed: StarknetApiGasVector::default(),
    })
}
