### Title
Gateway `__validate__` Bypass via `skip_stateful_validations` Allows Unsigned Invoke Admission for Newly-Deployed Accounts - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator unconditionally skips the `__validate__` entry-point (account signature verification) for any Invoke V3 transaction with `nonce == 1` when the account's on-chain nonce is `0` and `account_tx_in_pool_or_recent_block` returns `true`. Because `account_tx_in_pool_or_recent_block` is satisfied by **any** transaction from that address in the mempool — not exclusively a `deploy_account` — an unprivileged attacker who observes a legitimate `deploy_account` in the mempool can submit a forged Invoke with nonce=1 and an arbitrary (invalid) signature that passes all gateway checks and is admitted to the mempool without signature verification.

### Finding Description

The gateway stateful validation path is:

```
extract_state_nonce_and_run_validations
  → run_pre_validation_checks
      → validate_state_preconditions   (nonce range, resource bounds)
      → validate_by_mempool            (nonce ordering, fee escalation — no sig check)
      → skip_stateful_validations      ← returns true when conditions met
  → run_validate_entry_point(skip_validate=true)
      → execution_flags.validate = !skip_validate = false
      → StatefulValidator::perform_validations
          → perform_pre_validation_stage  (nonce increment, fee bounds)
          → if !execution_flags.validate: return Ok(())  ← __validate__ SKIPPED
```

The bypass condition in `skip_stateful_validations`:

```rust
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
    return mempool_client
        .account_tx_in_pool_or_recent_block(tx.sender_address())
        .await ...;
}
```

`account_tx_in_pool_or_recent_block` is implemented as:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
```

This returns `true` if **any** transaction from the address is in the pool — including the legitimate `deploy_account` submitted by the victim. The attacker does not need to control the `deploy_account`; they only need to observe it in the public mempool.

When `skip_validate = true`, `run_validate_entry_point` sets `execution_flags.validate = false`, and `StatefulValidator::perform_validations` returns `Ok(())` immediately after `perform_pre_validation_stage` without ever calling `validate_tx` / `__validate__`.

The `validate_by_mempool` call that precedes `skip_stateful_validations` only checks nonce ordering and fee escalation — it does not verify signatures. A forged invoke with nonce=1 and account_nonce=0 passes this check as a valid "future nonce" transaction.

### Impact Explanation

An attacker can submit an Invoke transaction with nonce=1, arbitrary calldata, and an invalid/forged signature for any account whose `deploy_account` is currently in the mempool. The transaction bypasses the gateway's `__validate__` check and is admitted to the mempool. This satisfies the **High** impact criterion: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

At block execution time, `__validate__` is called and fails (invalid signature), causing the transaction to revert. In Starknet, reverted transactions still consume the nonce. This means the legitimate user's Invoke with nonce=1 is permanently invalidated — the attacker can grief any newly-deployed account by front-running its first post-deploy invoke.

### Likelihood Explanation

- The `deploy_account` transaction is publicly visible in the mempool.
- The attacker needs only to submit a single Invoke with nonce=1 and any signature bytes.
- No privileged access, no special contract, no cryptographic capability is required.
- The window is the time between the `deploy_account` entering the mempool and being committed to a block — typically several seconds to minutes.
- Fee escalation (if enabled) allows the attacker to replace the legitimate invoke even after it is submitted.

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a **`deploy_account` transaction** (not just any transaction) is pending for the address. Alternatively, restrict the skip to cases where the mempool contains a `deploy_account` with nonce=0 for the same sender address, and verify the `deploy_account` itself passed signature verification before granting the skip.

### Proof of Concept

1. Victim submits `deploy_account` for address `A` (nonce=0, valid signature). It enters the mempool; on-chain nonce for `A` remains `0`.

2. Attacker observes `A` in the mempool and submits:
   ```
   Invoke V3 {
     sender_address: A,
     nonce: 1,
     calldata: [drain_funds_to_attacker],
     signature: [0xDEAD, 0xBEEF],   // invalid
     resource_bounds: { l2_gas: { max_amount: X, max_price: Y } }
   }
   ```

3. Gateway stateful validation:
   - `account_nonce = 0` (from state)
   - `tx_nonce = 1` → nonce range check passes
   - `validate_by_mempool` → nonce=1 ≥ account_nonce=0, no duplicate → passes
   - `skip_stateful_validations`: `tx.nonce()==1 && account_nonce==0 && account_tx_in_pool_or_recent_block(A)==true` → returns `true`
   - `run_validate_entry_point(skip_validate=true)` → `execution_flags.validate=false` → `__validate__` skipped → `Ok(())`

4. Attacker's forged invoke is admitted to the mempool.

5. If fee escalation is enabled, attacker can replace the victim's legitimate invoke with nonce=1 by offering higher fees.

6. At block execution: `deploy_account` executes (nonce→1), then attacker's invoke executes, `__validate__` fails (bad signature), transaction reverts, nonce→2. Victim's nonce=1 invoke is permanently invalid.

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
```
