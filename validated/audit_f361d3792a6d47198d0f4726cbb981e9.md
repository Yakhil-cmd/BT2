### Title
TOCTOU: Second Independent `getPoolInfo` Fetch in `prepareSubmitData` Allows Malicious Pool Server to Substitute `targetPuzzleHash` After User Verification — (`PlotNFTSelectPool.tsx`)

---

### Summary

There are two completely independent HTTP fetches to the pool URL. The first populates the "Verify Pool Details" UI card; the second happens at submit time inside `prepareSubmitData` and its result is what actually gets committed on-chain. A malicious pool server can return a legitimate `targetPuzzleHash` on the first request and an attacker-controlled one on the second, causing all pooled farming rewards to be redirected to the attacker while the user sees nothing wrong.

---

### Finding Description

**Fetch 1 — display only:**
`PlotNFTSelectBase` calls `usePoolInfo(poolUrl)`, which calls `getPoolInfo(normalizedUrl)` reactively whenever `poolUrl` changes. The result is rendered in the "Verify Pool Details" card. [1](#0-0) [2](#0-1) 

**Fetch 2 — on-chain commitment:**
When the user clicks "Create", `handleSubmit` calls `prepareSubmitData`, which issues a **new, independent** call to `getPoolInfo`. The `targetPuzzleHash` returned by *this* second fetch is what gets written into `initialTargetState` and passed to `createNewPoolWallet`. [3](#0-2) [4](#0-3) 

**Wallet creation uses the second fetch's value:** [5](#0-4) 

There is no caching, no comparison between the two fetched values, and no re-use of the already-fetched `poolInfo` from `usePoolInfo`. The two fetches are entirely decoupled.

---

### Impact Explanation

The `targetPuzzleHash` embedded in the singleton's `initialTargetState` is the puzzle hash to which the pool pays out farming rewards. If the attacker substitutes their own puzzle hash on the second fetch, **all pooled farming rewards are permanently redirected to the attacker** for as long as the user farms to that pool. This is an unauthorized payout change affecting pooled farming rewards — Critical scope.

---

### Likelihood Explanation

The precondition is that the user connects to a pool URL controlled by the attacker. This is a realistic scenario: a user could be directed to a malicious pool via a link, a community post, or a typosquatted domain. The attacker only needs to serve different responses on the first vs. second `GET /pool_info` request (trivially implemented with a request counter). No local access, no key leakage, and no cryptographic break is required.

---

### Recommendation

Remove the second fetch entirely. The `poolInfo` already fetched and displayed by `usePoolInfo` should be passed down into `prepareSubmitData` (or returned via the `ref`/`getSubmitData` path) so that the `targetPuzzleHash` committed on-chain is **the same object** the user visually verified — not a fresh network round-trip. Concretely:

- Expose the cached `poolInfo` value from `PlotNFTSelectBase`/`usePoolInfo` up to `PlotNFTSelectPool`.
- In `prepareSubmitData`, accept the already-fetched `poolInfo` as a parameter instead of calling `getPoolInfo` again.
- Optionally, re-validate that the URL in the cached result still matches the form field before submitting.

---

### Proof of Concept

1. Stand up an HTTP server at `http://attacker-pool.example/`:
   - Request 1 (`GET /pool_info`): return legitimate pool JSON with `target_puzzle_hash = "0xLEGIT..."`.
   - Request 2+ (`GET /pool_info`): return `target_puzzle_hash = "0xATTACKER..."`.
2. Open the GUI, navigate to "Add a Plot NFT", select "Connect to pool", enter `http://attacker-pool.example/`.
3. The "Verify Pool Details" card renders with `0xLEGIT...` — the user sees nothing suspicious.
4. User clicks "Create".
5. `prepareSubmitData` fires a second `getPoolInfo` call, receives `0xATTACKER...`.
6. `createNewPoolWallet` is called with `targetPuzzleHash = "0xATTACKER..."`.
7. Assert in a test harness: the argument passed to `createNewPoolWallet` contains the attacker hash while the UI displayed the legitimate one.

```typescript
async function prepareSubmitData(data: FormData): SubmitData {
  const { self, fee, poolUrl } = data;
  const initialTargetState = {
    state: self ? 'SELF_POOLING' : 'FARMING_TO_POOL',
  };

  if (!self && poolUrl) {
    const normalizedPoolUrl = normalizeUrl(poolUrl);
    const { targetPuzzleHash, relativeLockHeight } = await getPoolInfo(normalizedPoolUrl);
    if (!targetPuzzleHash) {
      throw new Error(t`Pool does not provide targetPuzzleHash.`);
    }
    if (relativeLockHeight === undefined) {
      throw new Error(t`Pool does not provide relativeLockHeight.`);
    }

    initialTargetState.poolUrl = normalizedPoolUrl;
    initialTargetState.targetPuzzleHash = targetPuzzleHash;
    initialTargetState.relativeLockHeight = relativeLockHeight;
  }

  const feeMojos = chiaToMojo(fee || '0');

  return {
    fee: feeMojos,
    initialTargetState,
  };
}
```
