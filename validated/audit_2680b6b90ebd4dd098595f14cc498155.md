### Title
Cross-Network Replay of COA Ownership Proof Signatures Due to Missing Chain-ID Binding in `signedData` - (File: `fvm/evm/stdlib/contract.cdc`)

### Summary
`EVM.validateCOAOwnershipProof` verifies Cadence key signatures over an arbitrary `signedData` blob using the fixed domain-separation tag `"FLOW-V0.0-user"`. Neither the tag nor the protocol enforces that `signedData` encodes the Flow network chain ID or the target `evmAddress`. A signature produced on one Flow network (e.g., testnet) is therefore cryptographically valid on every other Flow network (e.g., mainnet) that shares the same Cadence account keys and COA address, enabling cross-network replay of COA ownership proofs.

### Finding Description
`EVM.validateCOAOwnershipProof` in `fvm/evm/stdlib/contract.cdc` performs two independent checks:

1. It verifies that the supplied `signatures` are valid over `signedData` using `Crypto.KeyList.verify` with `domainSeparationTag: "FLOW-V0.0-user"`.
2. It checks that the COA stored at `path` in the given Cadence `address` has an EVM address equal to the supplied `evmAddress`. [1](#0-0) 

The domain-separation tag `"FLOW-V0.0-user"` is a protocol-wide constant; it is identical on mainnet, testnet, previewnet, and every other Flow network. The function accepts any caller-supplied `signedData` without requiring it to commit to the network chain ID or to the `evmAddress` being proven.

The code comment at lines 1001–1009 explicitly acknowledges the cross-address replay risk ("a signature produced for one purpose could be replayed here against a different COA owned by the same Cadence account") but dismisses it as low-risk and does **not** mention cross-network replay. [2](#0-1) 

The `COAOwnershipProof` struct that is RLP-encoded and passed as `signatures` contains no chain ID field: [3](#0-2) 

The EVM-side precompile `verifyCOAOwnershipProof` (selector `ProofVerifierFuncSig`) calls `validateCOAOwnershipProof` via the Cadence backend, forwarding the caller-supplied `signedData` verbatim: [4](#0-3) [5](#0-4) 

The COA Solidity contract exposes ERC-1271 `isValidSignature` by delegating to this precompile: [6](#0-5) 

### Impact Explanation
Any ERC-1271–compliant protocol on mainnet that relies on `isValidSignature` to authorize asset transfers (token approvals, NFT transfers, multisig operations, etc.) can be tricked into accepting a signature that was legitimately produced on testnet. Because the COA contract is the ERC-1271 signer, a successful replay grants the attacker the ability to authorize any on-chain action the victim's COA is permitted to perform on mainnet — including draining ERC-20/ERC-721 assets held by or approved to the COA.

### Likelihood Explanation
The attack requires:
1. The victim's Cadence account to exist on both networks with the same key material (common for developers and early adopters).
2. The victim's COA to be stored at the same public path on both networks.
3. The COA to have the same EVM address on both networks — possible when the same ordinal COA is created on both networks (COA addresses are assigned sequentially by the address allocator, so the Nth COA on each network shares the same address).
4. The attacker to have obtained a valid `COAOwnershipProof` signature from the victim for any `signedData` on the source network.

Conditions 1–3 are realistic for users who mirror their setup across networks. Condition 4 is achievable by any application that legitimately requests a COA ownership proof (e.g., a testnet dApp). Likelihood is **medium**.

### Recommendation
Bind `signedData` to both the network and the target COA address at the protocol level. Two complementary mitigations:

1. **Enforce chain-ID inclusion**: Require `signedData` to be structured as `hash(chainID || evmAddress || applicationData)` and verify this inside `validateCOAOwnershipProof`, analogous to EIP-712 domain separation.
2. **Include chain ID in the domain-separation tag**: Replace the hardcoded `"FLOW-V0.0-user"` tag with a network-specific tag such as `"FLOW-V0.0-user-<chainID>"`, derived from the runtime chain ID, so signatures are cryptographically bound to a single network. [7](#0-6) 

### Proof of Concept
1. Alice has a Cadence account `0xALICE` with key `K` on both testnet and mainnet. She creates a COA on each network; by coincidence (or by being the Nth user on both), both COAs receive EVM address `0xCOA`.
2. A testnet dApp asks Alice to sign arbitrary bytes `D` to prove COA ownership. Alice signs `D` with key `K` under tag `"FLOW-V0.0-user"`, producing signature `σ`.
3. The attacker constructs a `COAOwnershipProof{Address: 0xALICE, CapabilityPath: "coa", KeyIndices: [0], Signatures: [σ]}` and calls `verifyCOAOwnershipProof(0xCOA, D, encodedProof)` on the Cadence Arch precompile **on mainnet**.
4. `validateCOAOwnershipProof` on mainnet verifies `σ` over `D` with tag `"FLOW-V0.0-user"` — valid, because Alice's key `K` is identical on mainnet. It then checks that the COA at `/public/coa` in `0xALICE` has address `0xCOA` — also true. The function returns `isValid: true`.
5. The attacker submits this forged ERC-1271 proof to a mainnet lending protocol that uses `isValidSignature` to authorize a withdrawal from Alice's COA. The protocol accepts the proof and transfers Alice's mainnet assets to the attacker. [1](#0-0) [6](#0-5)
