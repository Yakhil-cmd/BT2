Audit Report

## Title
Stale ICP/XDR Rate Used Without Staleness Guard When Minting Cycles — (`rs/nns/cmc/src/main.rs`)

## Summary
The Cycles Minting Canister stores an ICP/XDR conversion rate that is refreshed via heartbeat every five minutes. When the Exchange Rate Canister (XRC) is unavailable, the heartbeat silently fails and the stored rate grows stale. Neither `process_top_up` nor `process_mint_cycles` checks the age of `icp_xdr_conversion_rate` before converting ICP to cycles, allowing any unprivileged caller to mint cycles at an arbitrarily old rate during an XRC outage window.

## Finding Description
**Rate update path:** `update_exchange_rate` in `rs/nns/cmc/src/exchange_rate_canister.rs` is invoked from the heartbeat and gated by `UpdateExchangeRateGuard`. On any retrieval or validation failure, the guard reschedules for the next minute and returns an error — the stored `state.icp_xdr_conversion_rate` is left unchanged. [1](#0-0) 

**Minting path:** Both `process_top_up` and `process_mint_cycles` call `tokens_to_cycles(amount)` as their first step. This function reads `state.icp_xdr_conversion_rate` directly with no check on `timestamp_seconds` relative to the current time. [2](#0-1) [3](#0-2) 

**Write-time guard is insufficient:** `do_set_icp_xdr_conversion_rate` only enforces that a new rate's timestamp is strictly greater than the current one — it places no upper bound on how old the stored rate may be when it is *read* for minting. [4](#0-3) 

The refresh interval constant confirms the intended freshness window is five minutes, but nothing enforces this at read time. [5](#0-4) 

## Impact Explanation
During an XRC outage, if the ICP market price drops, the CMC continues minting cycles at the pre-drop (inflated) XDR/ICP rate. Any caller who sends ICP during this window receives proportionally more cycles than the current market rate warrants — the protocol effectively sells cycles below their current ICP cost. The per-hour rate limiter (`base_cycles_limit`, 150 P cycles/hour) bounds total damage per window but does not prevent the mispricing itself. [6](#0-5) 

At 150 P cycles/hour and a 20% ICP price drop, an attacker extracts approximately 25% excess cycles per ICP sent, representing tens of thousands of dollars of protocol value per hour of outage. This constitutes illegal minting / protocol value extraction and maps to the **High** impact tier: significant NNS/XRC/CMC infrastructure security impact with concrete protocol harm.

## Likelihood Explanation
The XRC is a system canister on the NNS subnet. The codebase itself documents that CMC upgrades can leave the XRC in a stopping state for hours, during which the CMC heartbeat fails to refresh the rate. This is a known, recurring operational condition, not a theoretical one. [7](#0-6) 

An attacker who monitors XRC canister status (publicly observable on-chain) and ICP market price can deterministically time a `notify_top_up` or `notify_mint_cycles` call to exploit the gap. No special privileges are required — any principal with ICP can call these endpoints.

## Recommendation
Add a maximum-staleness guard inside `tokens_to_cycles` or at the entry points `process_top_up` / `process_mint_cycles`. Before performing the conversion, compute `now() - icp_xdr_conversion_rate.timestamp_seconds`; if it exceeds a configurable threshold (e.g., 30 minutes), return a retriable `NotifyError` so callers retry once the rate is refreshed. The threshold should be configurable via a state field to allow governance adjustment without a canister upgrade.

## Proof of Concept
1. Observe that the XRC canister enters the stopping state (e.g., during a CMC or XRC upgrade). The CMC heartbeat begins returning `FailedToRetrieveRate` and rescheduling; `state.icp_xdr_conversion_rate` is frozen at its last value.
2. Wait for the ICP market price to drop materially (e.g., 20%) while the XRC remains unavailable.
3. Transfer ICP to the CMC's top-up subaccount for a target canister and call `notify_top_up` with the resulting block index. The CMC reads the stale `icp_xdr_conversion_rate`, computes cycles at the inflated pre-drop rate, and deposits ~25% more cycles than the current market rate warrants.
4. Repeat up to the `base_cycles_limit` per hour.
5. A deterministic integration test can reproduce this by: (a) initializing the CMC with a known rate, (b) disabling the XRC mock to simulate unavailability, (c) advancing the mock clock by >30 minutes, (d) calling `notify_top_up`, and (e) asserting that cycles minted exceed what the current (lower) market rate would produce — demonstrating the absence of any staleness rejection.
