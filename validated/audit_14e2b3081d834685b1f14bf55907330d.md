### Title
WebAuthn Transaction Authentication Accepts Arbitrary `origin` and `rpIdHash` Without Validation Б─■ (`model/flow/webauthn.go`)

### Summary

The `validateWebAuthNExtensionData` function in `model/flow/webauthn.go` implements the WebAuthn authentication scheme for Flow transactions (FLIP 264). It validates the challenge, type, and authenticator flags, but **never validates the `origin` field in `clientDataJSON` and never validates the `rpIdHash` against any expected value**. The only `rpIdHash` check is a negative one: it rejects the hash only if it equals `TransactionDomainTag`. Any attacker who possesses a valid account key can sign a WebAuthn transaction with a completely arbitrary `origin` (e.g., a phishing site) and an arbitrary `rpIdHash`, and the FVM will accept it as a valid WebAuthn-authenticated transaction. This is the direct Flow analog of the reported "unverified identity header" vulnerability: a security-critical field that is parsed but never verified against an expected value.

### Finding Description

In `model/flow/webauthn.go`, the function `validateWebAuthNExtensionData` performs the following checks:

1. Decodes the RLP-encoded `WebAuthnExtensionData`.
2. Parses `clientDataJSON` and extracts `type`, `challenge`, and `origin`.
3. Verifies `type == "webauthn.get"` Б°⌠
4. Verifies `challenge == SHA2_256(TransactionDomainTag || payload)` Б°⌠
5. Verifies `len(authenticatorData) >= 37` Б°⌠
6. Extracts `rpIdHash` from `authenticatorData[0:32]`.
7. **Checks only that `rpIdHash != TransactionDomainTag`** Б─■ any other 32-byte value is accepted Б°≈
8. **Never checks `origin`** Б─■ any string (including `""`) is accepted Б°≈

The `origin` field in WebAuthn is the security boundary that binds a credential to a specific Relying Party origin (e.g., `https://wallet.flow.com`). The `rpIdHash` is the SHA-256 hash of the Relying Party ID (domain), which the authenticator embeds to prove the credential was used on the correct site. Both are left completely unvalidated.

The test suite explicitly confirms this is intentional/accepted behavior:

```go
{
    description: "empty origin (valid)",
    // ...
    extensionOk: true,
    signatureOk: true,
},
```

The `origin` field is parsed into `CollectedClientData.Origin` but is never read after parsing.

The attack path through the FVM is:
1. Attacker submits a transaction via the Access API (`SendTransaction` / `SendAndSubscribeTransactionStatuses`).
2. `access/validator/validator.go` Б├▓ `checkSignatureFormat` calls `ValidateExtensionDataAndReconstructMessage` which calls `validateWebAuthNExtensionData`. This passes because `origin` and `rpIdHash` are not checked.
3. The transaction is forwarded to Collection nodes, then Execution nodes.
4. `fvm/transactionVerifier.go` Б├▓ `verifyTransaction` Б├▓ `verifySignatures` Б├▓ `signatureContinuation.verify()` calls `ValidateExtensionDataAndReconstructMessage` again to reconstruct the message, then calls `crypto.VerifySignatureFromTransaction`. The reconstructed message is `authenticatorData || SHA2_256(clientDataJSON)`.
5. Since the attacker controls both `authenticatorData` (including the arbitrary `rpIdHash`) and `clientDataJSON` (including the arbitrary `origin`), and the signature is over this attacker-controlled message, the cryptographic check passes as long as the attacker signs with the correct account key.

### Impact Explanation

The direct impact is **authentication context bypass in the WebAuthn scheme**: the `origin` and `rpIdHash` fields Б─■ which in standard WebAuthn are the primary phishing-resistance mechanisms Б─■ are completely ignored by the FVM. A malicious dApp or phishing site can construct a valid WebAuthn-signed Flow transaction with `origin: "https://evil.com"` and an arbitrary `rpIdHash`, and the FVM will execute it identically to a transaction signed on the legitimate origin. Any wallet that relies on the FVM to enforce origin binding for WebAuthn credentials provides no phishing protection at the protocol level. A user tricked into signing a WebAuthn assertion on a phishing site will have their transaction executed on-chain with no protocol-level rejection.

The impact class is **broken authentication** (the analog of the reported H-03): the authentication scheme accepts attacker-controlled identity/context fields without verification, allowing unauthorized execution of transactions under a false authentication context.

### Likelihood Explanation

The entry path is fully reachable by any unprivileged transaction sender via the public Access API. No staked node access, no admin keys, and no privileged position is required. The attacker only needs to hold a valid account key (or trick a user into signing via WebAuthn on a phishing site). The `origin` field is explicitly tested as valid when empty, confirming the missing check is a deliberate design gap, not an oversight in a single code path.

### Recommendation

1. **Validate `origin`**: The FVM should enforce that `clientData.Origin` is non-empty and matches an expected set of trusted origins (or at minimum is non-empty, as an empty origin is meaningless for phishing resistance).
2. **Validate `rpIdHash`**: The FVM should verify that `rpIdHash` equals `SHA2_256(expectedRpID)` where `expectedRpID` is a protocol-defined constant (e.g., `"flow-wallet.fcl.dev"` or a chain-specific value), not merely that it differs from `TransactionDomainTag`.
3. These checks should be added to `validateWebAuthNExtensionData` in `model/flow/webauthn.go` before the function returns `true`.

### Proof of Concept

An attacker constructs a WebAuthn-scheme transaction with:

```go
// Arbitrary rpIdHash Б─■ any 32 bytes except TransactionDomainTag
rpIdHash := []byte{0xAA, 0xBB, ..., 0xFF} // 32 bytes, attacker-chosen

// Arbitrary origin Б─■ including phishing site or empty string
clientDataJSON := map[string]string{
    "type":      "webauthn.get",
    "challenge": base64url(SHA2_256(TransactionDomainTag || envelopePayload)),
    "origin":    "https://evil-phishing-site.com",  // NEVER CHECKED
}

authenticatorData := concat(rpIdHash, []byte{0x01}, sigCounter) // UP flag set
```

The `validateWebAuthNExtensionData` function at line 125 only checks:

```go
if bytes.Equal(TransactionDomainTag[:], rpIdHash) {
    return false, nil
}
```

Since `rpIdHash` is `0xAA...FF` (not `TransactionDomainTag`), this check passes. The `origin` field is never read after line 85 where `clientData` is parsed. The function returns `true` and the reconstructed message `authenticatorData || SHA2_256(clientDataJSON)`. func (entry *signatureContinuation) verify() errors.CodedError {
	if entry.invokedVerify {
		return entry.verifyErr
	}

	entry.invokedVerify = true

	valid, message := entry.ValidateExtensionDataAndReconstructMessage(entry.payload)
	if !valid {
		entry.verifyErr = entry.newError(fmt.Errorf("signature extension data is not valid"))
		return entry.verifyErr
	}

	valid, err := crypto.VerifySignatureFromTransaction(
		entry.Signature,
		message,
		entry.accountKey.PublicKey,
		entry.accountKey.HashAlgo,
	)
	if err != nil {
		entry.verifyErr = entry.newError(err)
	} else if !valid {
		entry.verifyErr = entry.newError(fmt.Errorf("signature is not valid"))
	}

	return entry.verifyErr
