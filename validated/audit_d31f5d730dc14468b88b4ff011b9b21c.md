### Title
All EIP-2537 BLS12-381 Precompiles Charge Zero Native Resources, Allowing Unbounded Proving-Cost Extraction - (`basic_bootloader/src/bootloader/block_flow/ethereum/hooks/eip_2537/`)

---

### Summary

Every BLS12-381 precompile introduced by EIP-2537 (G1/G2 addition, G1/G2 MSM, pairing check, MAP_FP_TO_G1, MAP_FP2_TO_G2) hardcodes `cost_native = 0`. The native resource is ZKsync OS's second accounting dimension that models RISC-V proving cycles. Charging zero native cost for these cryptographically expensive operations means an unprivileged caller can exhaust the EVM gas budget entirely on BLS12-381 work and force the prover to execute orders-of-magnitude more RISC-V cycles than the transaction's native resource limit would otherwise permit.

---

### Finding Description

ZKsync OS implements a double resource accounting model: EVM gas (Ergs) and a native resource that tracks proving cost. The native resource limit for a transaction is derived as `gasLimit × nativePerGas`, and if a transaction exhausts it the entire transaction is reverted. This is the mechanism that prevents a user from paying for cheap EVM gas while forcing the prover to do expensive work.

All BLS12-381 precompiles hardcode `cost_native = 0`:

**G1/G2 Addition** (`addition.rs`): [1](#0-0) 

**G1/G2 MSM** (`msm.rs`): [2](#0-1) 

**Pairing Check** (`pairing.rs`): [3](#0-2) 

**MAP_FP_TO_G1 / MAP_FP2_TO_G2** (`mappings.rs`): [4](#0-3) 

Contrast this with the BN254 precompiles, which correctly charge substantial native costs: [5](#0-4) 

The native resource accounting model is documented explicitly: [6](#0-5) 

The `create_resources_for_tx` function caps the native limit at `MAX_NATIVE_COMPUTATIONAL` and derives it from `gasLimit × nativePerGas`: [7](#0-6) 

Because BLS12-381 precompiles never decrement the native counter, the cap is irrelevant — the native resource budget is never consumed regardless of how many pairing or MSM calls are made.

---

### Impact Explanation

The native resource is the only mechanism that bounds the RISC-V proving cost of a transaction. With `cost_native = 0` for all BLS12-381 operations:

1. An attacker fills a transaction's entire EVM gas budget with BLS12-381 pairing or G2 MSM calls.
2. EVM gas is consumed correctly, so the transaction is valid from the sequencer's perspective.
3. The native resource counter is never decremented, so the transaction never hits the native resource limit.
4. The prover must execute the full BLS12-381 computation (Miller loop, final exponentiation, cofactor clearing, etc.) for every call, consuming RISC-V cycles far beyond what the transaction's native budget would normally allow.
5. This is a resource accounting bug that allows more proving computation to be extracted than intended — the direct analog of the external report's "larger rebalances than should be possible."

The BLS12-381 pairing check is the most severe case: it involves a full Miller loop and final exponentiation per pair, which is among the most expensive operations in the codebase. The G2 MSM is similarly expensive.

---

### Likelihood Explanation

- EIP-2537 precompiles are callable by any unprivileged EVM transaction targeting addresses `0x0b`–`0x11`.
- No special role, governance, or privileged access is required.
- The attacker only needs to submit a standard EVM transaction with sufficient gas.
- The bug is deterministic and reproducible on every call.

---

### Recommendation

Assign realistic native costs to all BLS12-381 precompiles, following the same pattern used for BN254:

```rust
// Example for pairing check — benchmark actual RISC-V cycle counts
pub const BLS12_381_PAIRING_BASE_NATIVE_COST: u64 = native_with_delegations!(...);
pub const BLS12_381_PAIRING_PER_PAIR_NATIVE_COST: u64 = native_with_delegations!(...);

let cost_native = BLS12_381_PAIRING_BASE_NATIVE_COST
    + (num_pairs as u64) * BLS12_381_PAIRING_PER_PAIR_NATIVE_COST;
```

Each precompile should have its native cost benchmarked on the RISC-V target and defined in `basic_system/src/cost_constants.rs` alongside the existing BN254 constants. [8](#0-7) 

---

### Proof of Concept

1. Deploy or call any contract that invokes the BLS12-381 pairing precompile at address `0x0f` in a loop.
2. Submit a transaction with a large gas limit (e.g., 30,000,000 gas).
3. Each pairing call costs `37700 + 32600 × k` EVM gas but **zero** native resources.
4. With 30M gas, an attacker can execute ~900 single-pair pairing calls (`30,000,000 / 32,600 ≈ 920`).
5. Observe that the native resource counter remains at its initial value throughout, confirming no native cost is charged.
6. The prover must execute ~920 full BLS12-381 pairings, each involving a Miller loop and final exponentiation — work that would normally be bounded by the native resource limit but is entirely unbounded here.

## Native resource

The native resource models the offchain cost of processing a transaction. Currently, this is dominated by proving and publishing data. A good intuition for it is "how many RISC-V cycles it takes to prove a given computation".

If a transaction execution runs out of native resources, the entire transaction is reverted. If the same happens during transaction validation, the transaction is considered invalid.
