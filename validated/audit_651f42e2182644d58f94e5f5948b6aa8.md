### Title
Silent Precompile Call Failure Burns ERC-20 Mirror Tokens Without Releasing NEP-141 — (`etc/eth-contracts/contracts/EvmErc20.sol`, `etc/eth-contracts/contracts/EvmErc20V2.sol`)

---

### Summary

In `EvmErc20.sol` and `EvmErc20V2.sol`, both `withdrawToNear()` and `withdrawToEthereum()` call `_burn()` on the caller's balance **before** invoking the exit precompile via inline assembly. The assembly `call` return value is captured in a local variable `res` but is **never checked**. If the precompile call fails and returns `0`, the ERC-20 tokens are permanently destroyed while the corresponding NEP-141 tokens are never released from Aurora's connector account — causing a permanent, irrecoverable fund freeze for the user.

---

### Finding Description

Both `EvmErc20` and `EvmErc20V2` implement the same withdrawal pattern:

```solidity
// EvmErc20.sol, lines 53–63
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // ← tokens destroyed here

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        // res is NEVER checked — no revert if call returns 0
    }
}
``` [1](#0-0) 

The same unchecked pattern appears in `withdrawToEthereum()` in both contracts: [2](#0-1) [3](#0-2) [4](#0-3) 

The exit precompile (`ExitToNear` at `0xe9217bc70b7ed1f598ddd3199e80b093fa71124f`, `ExitToEthereum` at `0xb0bd02f6a392af548bdf1cfaee5dfa0eefcc8eab`) can fail for multiple reasons reachable by an unprivileged user:

1. **Invalid recipient account ID**: The precompile's `ExitToNearParams::try_from(input)` validates the NEAR account ID. If the recipient bytes do not form a valid NEAR account ID, the precompile returns `ExitError`, causing the EVM `call` to return `0`. [5](#0-4) 

2. **Precompile paused**: The engine supports pausing `EXIT_TO_NEAR` and `EXIT_TO_ETHEREUM` precompiles independently via `pause_precompiles`. When paused, the precompile returns `ERR_PAUSED`, causing the `call` to return `0`. [6](#0-5) 

3. **NEP-141 not registered**: If `get_nep141_from_erc20` cannot find the mapping for the calling ERC-20 address, the precompile returns an error. [7](#0-6) 

In all cases, because `res` is never checked and no `revert` is issued, the Solidity function returns successfully. The ERC-20 tokens are gone, but the NEP-141 tokens remain locked in Aurora's connector account with no recovery path.

---

### Impact Explanation

**Permanent freezing of funds.** The ERC-20 mirror token supply decreases (tokens burned) while the NEP-141 backing supply held by Aurora's connector does not decrease (no `ft_transfer` or `ft_transfer_call` is issued). The user loses their ERC-20 tokens and receives nothing. The NEP-141 tokens are permanently stranded in Aurora's connector account with no mechanism to reclaim them, since the bridge accounting is now permanently out of sync.

---

### Likelihood Explanation

**Medium.** The trigger is reachable by any unprivileged ERC-20 token holder. A user who provides a malformed NEAR account ID (e.g., an Ethereum hex address, an empty string, or a string exceeding NEAR's account ID length limit) as the `recipient` argument to `withdrawToNear()` will silently lose their tokens. This is a realistic user error. Additionally, if the precompile is paused at the time of the call (an operational state that can occur during incident response), any user who calls `withdrawToNear()` or `withdrawToEthereum()` during that window will permanently lose their tokens with no revert to protect them.

---

### Recommendation

Check the return value of the assembly `call` and revert on failure in both `withdrawToNear()` and `withdrawToEthereum()` in both `EvmErc20.sol` and `EvmErc20V2.sol`:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

This ensures that if the precompile call fails for any reason, the entire transaction reverts — including the `_burn()` — so the user's tokens are never destroyed without a corresponding NEP-141 release.

---

### Proof of Concept

1. User holds 1000 ERC-20 mirror tokens (backed 1:1 by 1000 NEP-141 tokens held by Aurora's connector).
2. User calls `withdrawToNear("0xdeadbeef...invalid_near_id", 1000)` — the recipient is an Ethereum hex address, not a valid NEAR account ID.
3. `_burn(user, 1000)` executes: user's ERC-20 balance drops to 0, total ERC-20 supply decreases by 1000.
4. The assembly `call` to `ExitToNear` precompile executes. Inside the precompile, `ExitToNearParams::try_from(input)` fails to parse the recipient as a valid NEAR `AccountId` and returns `ExitError`. The EVM `call` opcode returns `0`.
5. `res == 0` is never checked. The Solidity function returns without reverting.
6. No `ft_transfer` or `ft_transfer_call` is issued to the NEP-141 contract. The 1000 NEP-141 tokens remain locked in Aurora's connector account.
7. User has 0 ERC-20 tokens and 0 NEP-141 tokens. Funds are permanently frozen with no recovery path.
