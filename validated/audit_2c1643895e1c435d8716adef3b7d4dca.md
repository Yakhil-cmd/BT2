### Title
Pool Admin Bypasses `maxAdminSpreadFeeE6` Cap via Uncapped `setPoolBinAdditionalFees` ‚ÄĒ (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

The factory enforces `maxAdminSpreadFeeE6` and `maxAdminNotionalFeeE8` caps on the pool admin's spread and notional fees via `setPoolAdminFees`. However, a second, parallel fee vector ‚ÄĒ per-bin additional fees set through `setPoolBinAdditionalFees` ‚ÄĒ carries **no cap check at all**. A pool admin can set `addFeeBuyE6`/`addFeeSellE6` to `type(uint16).max` (65 535 = 6.5535% in E6 units) on every bin, stacking uncapped extra fees on top of the already-maxed admin spread fee and causing traders to pay more than the protocol's admin-fee boundary permits.

---

### Finding Description

`setPoolAdminFees` correctly enforces the cap:

```solidity
// MetricOmmPoolFactory.sol L414-415
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (newAdminNotionalFeeE8 > maxAdminNotionalFeeE8) revert AdminFeeTooHigh();
``` [1](#0-0) 

`setPoolBinAdditionalFees`, callable by the same pool admin, passes the values straight through with **no cap check**:

```solidity
// MetricOmmPoolFactory.sol L450-457
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
``` [2](#0-1) 

Inside the pool, `setBinAdditionalFees` writes the values directly to `BinState` storage with only a bin-index range check:

```solidity
// MetricOmmPool.sol L464-474
function setBinAdditionalFees(int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external onlyFactory nonReentrant(PoolActions.SET_BIN_ADDITIONAL_FEES)
{
    if (bin < LOWEST_BIN || bin > HIGHEST_BIN) revert InvalidBinIndex(bin);
    BinState storage s = _binStates[bin];
    s.addFeeBuyE6 = addFeeBuyE6;
    s.addFeeSellE6 = addFeeSellE6;
    emit BinAdditionalFeesUpdated(bin, addFeeBuyE6, addFeeSellE6);
}
``` [3](#0-2) 

During every swap these per-bin fees are added directly to the base spread fee before computing the ask/bid price:

```solidity
// MetricOmmPool.sol L540-541
uint256 buyFeeX64 = baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6);
uint256 sellFeeX64 = baseFeeX64 + Math.mulDiv(binState.addFeeSellE6, ONE_X64, 1e6);
``` [4](#0-3) 

The ask price paid by the buyer and the bid price received by the seller are then derived from these inflated fee values:

```solidity
// MetricOmmPool.sol L543-548
uint256 askBeforeNotional = Math.mulDiv(marginalPriceX64, ONE_X64 + buyFeeX64, ONE_X64, Math.Rounding.Ceil);
uint256 bidAfterSpread   = Math.mulDiv(marginalPriceX64, ONE_X64, ONE_X64 + sellFeeX64, Math.Rounding.Floor);
...
buyPriceX64  = Math.mulDiv(askBeforeNotional, 1e8, 1e8 - nf, Math.Rounding.Ceil).toUint128();
sellPriceX64 = Math.mulDiv(bidAfterSpread,    1e8 - nf, 1e8, Math.Rounding.Floor).toUint128();
``` [5](#0-4) 

The same inflated `addFeeBuyE6`/`addFeeSellE6` values flow into every `SwapMath.buyToken0InBinSpecifiedOut` / `buyToken0InBinSpecifiedIn` call during actual swaps (e.g. line 910 in `MetricOmmPool.sol`), so the extra fee is charged on real token transfers, not just on price quotes. [6](#0-5) 

---

### Impact Explanation

A pool admin who has already set `adminSpreadFeeE6 = maxAdminSpreadFeeE6` (up to 200 000 = 20 %) can additionally set `addFeeBuyE6 = addFeeSellE6 = 65 535` (‚Čą 6.55 %) on every bin. The combined effective fee charged to traders exceeds the cap the factory is supposed to enforce. Traders receive less output token than the protocol's admin-fee boundary permits, constituting a direct loss of user principal on every swap routed through those bins. Because the per-bin fees persist in storage and apply to all future swaps until changed, the impact is continuous rather than one-off.

---

### Likelihood Explanation

Any address that holds the `poolAdmin` role for a pool can trigger this with a single transaction. The pool admin role is semi-trusted and is explicitly expected to be constrained by the factory's fee caps. No special preconditions (flash loans, oracle manipulation, etc.) are required ‚ÄĒ only the normal `setPoolBinAdditionalFees` call path.

---

### Recommendation

Add a cap check inside `setPoolBinAdditionalFees` before forwarding to the pool, analogous to the check in `setPoolAdminFees`. For example:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (addFeeBuyE6  > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

Alternatively, introduce a dedicated `maxAdminBinAdditionalFeeE6` cap and enforce it here and in `setFeeCaps`.

---

### Proof of Concept

1. Factory owner sets `maxAdminSpreadFeeE6 = 200_000` (20 %).
2. Pool admin calls `setPoolAdminFees(pool, 200_000, 0)` ‚ÄĒ accepted, at the cap.
3. Pool admin calls `setPoolBinAdditionalFees(pool, 0, 65_535, 65_535)` ‚ÄĒ **no revert**; bin 0 now carries an extra ‚Čą 6.55 % buy and sell fee.
4. A trader swaps token1 ‚Üí token0 through bin 0. The effective fee applied is ‚Čą 26.55 %, well above the 20 % cap the factory is supposed to enforce.
5. The trader's output is reduced by the uncapped extra fee; the surplus accrues as spread fee revenue collectible by the pool admin via `collectPoolFees`.
