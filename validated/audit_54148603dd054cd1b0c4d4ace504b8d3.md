### Title
Missing Zero-Address Validation for `reward_address` Enables Permanent Yield Freeze - (File: `src/staking/staking.cairo`, `src/pool/pool.cairo`)

---

### Summary
Both `change_reward_address` in `src/staking/staking.cairo` and `change_reward_address` in `src/pool/pool.cairo` accept `0x0` as a valid reward address. When rewards are subsequently claimed, the STRK transfer is sent to the zero address, permanently destroying the unclaimed yield.

---

### Finding Description

`change_reward_address` in the staking contract validates only that the new address is not a registered token address, but performs no zero-address check: [1](#0-0) 

The only guard is:
```cairo
assert!(
    !self.does_token_exist(token_address: reward_address),
    "{}",
    GenericError::REWARD_ADDRESS_IS_TOKEN,
);
```

`0x0` is not a registered token, so this assertion passes. The zero address is then written directly into `staker_info.reward_address`. [2](#0-1) 

When `claim_rewards` or `unstake_action` is later called, `send_rewards_to_staker` reads `reward_address` from storage and transfers tokens to it without any zero-address guard: [3](#0-2) 

The same pattern exists in the pool contract. `change_reward_address` in `src/pool/pool.cairo` only checks that the address is not the pool's token: [4](#0-3) 

And `claim_rewards` in the pool transfers directly to the stored `reward_address`: [5](#0-4) 

The initial `stake` call has the same gap — `reward_address` is accepted without a zero check: [6](#0-5) 

---

### Impact Explanation

Any STRK transferred to `0x0` on Starknet is permanently unrecoverable. A staker or pool member who sets `reward_address` to `0x0` (intentionally or by mistake) will have all accumulated and future unclaimed yield burned on every subsequent `claim_rewards` or `unstake_action` call. The principal stake is returned to `staker_address` / `pool_member` directly, so only the yield is affected — matching **High: Permanent freezing of unclaimed yield**. [7](#0-6) 

---

### Likelihood Explanation

The entry path requires only an unprivileged staker or pool member to call `change_reward_address(0x0)`. No special role or privileged access is needed. Accidental triggering is realistic: a front-end bug, a scripting error, or a user passing an uninitialized variable could silently set the reward address to zero. The protocol provides no safety net — the transaction succeeds and emits a `StakerRewardAddressChanged` event with `new_address: 0x0` with no revert. [8](#0-7) 

---

### Recommendation

Add an explicit zero-address assertion in both `change_reward_address` implementations and in `stake`/`enter_delegation_pool`:

```cairo
assert!(reward_address.is_non_zero(), "{}", GenericError::ZERO_ADDRESS);
```

The error variant `GenericError::ZERO_ADDRESS` already exists in the codebase: [9](#0-8) 

---

### Proof of Concept

1. Staker calls `stake(reward_address: 0x0, operational_address: X, amount: MIN_STAKE)` — succeeds, no revert.
2. Alternatively, an existing staker calls `change_reward_address(reward_address: 0x0)` — succeeds.
3. Protocol accumulates unclaimed rewards for the staker over epochs.
4. Anyone calls `claim_rewards(staker_address)`.
5. `send_rewards_to_staker` executes `token_dispatcher.checked_transfer(recipient: 0x0, amount: rewards)`.
6. STRK is transferred to the zero address and is permanently lost.

The same four-step flow applies to a pool member via `Pool::change_reward_address(0x0)` followed by `Pool::claim_rewards`. [10](#0-9) [5](#0-4)
