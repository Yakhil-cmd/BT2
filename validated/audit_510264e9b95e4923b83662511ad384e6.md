### Title
Missing Caller Validation in `update_rewards` Allows Any Caller to Permanently Freeze Consensus Block Rewards - (File: src/staking/staking.cairo)

### Summary

The `update_rewards` function in the Staking contract carries no access control. Any unprivileged caller can invoke it with `disable_rewards: true`, which advances the global `last_reward_block` cursor without distributing any rewards. Because a per-block guard (`REWARDS_ALREADY_UPDATED`) prevents a second call in the same block, the legitimate block proposer is permanently locked out of claiming their block reward for that block. Repeating this once per block permanently freezes all consensus-phase staker rewards.

### Finding Description

`update_rewards` is exposed through the `IStakingRewardsManager` public interface and is implemented with no role check beyond `general_prerequisites()` (which only asserts the contract is unpaused and the caller is non-zero). [1](#0-0) 

The function accepts a caller-supplied `disable_rewards: bool` flag. When `true`, the function writes the current block number into `last_reward_block` and returns immediately without calling `_update_rewards`. [2](#0-1) 

The per-block guard that follows is unconditional — it fires regardless of whether rewards were actually distributed: [3](#0-2) 

Because `last_reward_block` is a single global slot (not per-staker), one successful call with `disable_rewards: true` exhausts the reward slot for the entire block for every staker in the system.

The only input validation performed is that `staker_address` must be an active staker with non-zero STRK balance — a condition any attacker can satisfy by supplying any legitimate staker's address. [4](#0-3) 

### Impact Explanation

An attacker who calls `update_rewards(<any_valid_staker>, disable_rewards: true)` once per block:

1. Advances `last_reward_block` to the current block.
2. Distributes zero rewards.
3. Causes every subsequent call in the same block to revert with `REWARDS_ALREADY_UPDATED`.

The legitimate block proposer — who would call `update_rewards(<their_address>, disable_rewards: false)` — is permanently blocked from claiming their block reward. Sustained across all blocks, this constitutes **permanent freezing of unclaimed consensus-phase yield** for all stakers and their delegators.

This maps directly to the allowed High impact: *Permanent freezing of unclaimed yield or unclaimed royalties*.

### Likelihood Explanation

The attack requires no privileged role, no leaked key, and no external dependency. Any EOA or contract can call `update_rewards` with an arbitrary valid staker address. Gas costs on Starknet are low enough to make per-block griefing economically viable. The attacker gains no direct profit but inflicts protocol-wide reward denial.

### Recommendation

Restrict `update_rewards` so that only the designated block proposer (or an authorized sequencer/operator role) may call it. One approach is to require `get_caller_address() == staker_address` (i.e., the staker must self-report their own block production). Alternatively, gate the function behind an existing privileged role (e.g., `only_app_governor`) and remove the public `disable_rewards` escape hatch, since the pre-consensus path is already handled by `update_rewards_from_attestation_contract`.

### Proof of Concept

```
// Attacker script — run once per block
loop {
    staking_contract.update_rewards(
        staker_address: any_active_staker,   // any valid staker satisfies the guard
        disable_rewards: true,               // no rewards distributed
    );
    // last_reward_block is now == current_block
    // All subsequent calls this block revert with REWARDS_ALREADY_UPDATED
    // The legitimate block proposer cannot claim their reward
    wait_for_next_block();
}
```

Step-by-step:
1. Attacker calls `update_rewards(valid_staker, true)` at block N.
2. `last_reward_block` is written to N; function returns with no reward distribution.
3. Block proposer calls `update_rewards(their_address, false)` — reverts: `REWARDS_ALREADY_UPDATED`.
4. Block proposer's `unclaimed_rewards_own` is never incremented for block N.
5. Repeated every block → all stakers and delegators receive zero consensus rewards indefinitely.
