### Title
Stop-Loss Watermark Decay Continues During Pool Pause, Bypassing LP Protection After Unpause — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

---

### Summary

`OracleValueStopLossExtension` uses a lazy, time-based linear decay on per-bin high watermarks. The decay clock (`lastDecayTs`) is only advanced when a swap executes. When the pool is paused — blocking all swaps — `lastDecayTs` freezes while `block.timestamp` keeps advancing. On the first swap after unpause, the accumulated `dt` is applied in full, potentially decaying the watermark to zero and silently resetting the stop-loss baseline to the current (potentially much lower) metric value, without ever triggering a breach.

---

### Finding Description

`_checkAndUpdateWatermarks` computes elapsed time as:

```solidity
uint256 dt = block.timestamp - hwmS.lastDecayTs;
``` [1](#0-0) 

This `dt` is fed directly into `_decayed`:

```solidity
function _decayed(uint256 hwm, uint256 ratePerSecondE8, uint256 dt) private pure returns (uint256) {
    if (ratePerSecondE8 == 0 || dt == 0 || hwm == 0) return hwm;
    uint256 factor = ratePerSecondE8 * dt;
    if (factor >= E8) return 0;
    return hwm - (hwm * factor) / E8;
}
``` [2](#0-1) 

The pool's `setPause` / `pausePool` / `unpausePool` path blocks all swaps when `pauseLevel != 0`, but does not interact with the extension in any way: [3](#0-2) 

Because `afterSwap` is the only hook that calls `_checkAndUpdateWatermarks`, and swaps are blocked during a pause, `lastDecayTs` is never updated during the pause window. When the pool is unpaused, the first swap computes `dt` = (pause duration + time since last pre-pause swap), applies the full accumulated decay, and then writes the new (possibly zero) watermark back to storage.

`_applyWatermark` then sees `hwm == 0`, so `metric >= hwm` is trivially true, the ratchet fires, and the new watermark is set to the current metric — no breach is reported regardless of how far the per-share value has fallen:

```solidity
function _applyWatermark(uint256 metric, uint256 hwm, uint256 floorMultiplier)
    private pure returns (uint256 newHwm, bool breached)
{
    if (metric >= hwm) return (metric, false);   // ← always taken when hwm == 0
    ...
}
``` [4](#0-3) 

---

### Impact Explanation

The `OracleValueStopLossExtension` is the sole on-chain guard preventing per-share LP value from being extracted below the configured drawdown floor. After a sufficiently long pause, that guard is silently voided: the watermark resets to whatever the current (potentially much lower) metric is, and the attacker can drain LP value in the first post-unpause swap without triggering `OracleStopLossTriggered`. LP principal is directly at risk.

At the default illustrative decay rate of 58 E8/s (~5 %/day), `factor >= E8` (watermark → 0) after ≈ 20 days of pause. Even a shorter pause proportionally weakens the protection: a 10-day pause halves the effective watermark, doubling the allowed drawdown.

---

### Likelihood Explanation

Pool pauses are a normal, semi-trusted admin operation (`pausePool`) and a protocol-level emergency tool (`protocolPausePool`). Neither path is exotic or requires an attacker to control privileged keys — a legitimate pause for oracle issues, an L2 sequencer outage, or a security incident is sufficient. An attacker who can observe or anticipate a pause can position a swap immediately after unpause to exploit the decayed watermark. The trigger is therefore reachable through ordinary protocol operation, not a malicious setup assumption.

---

### Recommendation

Exclude pause time from the decay window. Two equivalent approaches:

1. **Reset `lastDecayTs` on unpause.** Add an extension hook (e.g., `onUnpause`) called by the factory's `unpausePool` / `protocolUnpausePool` paths. The extension sets `hwmS.lastDecayTs = block.timestamp` for all active bins, so the next swap's `dt` counts only post-unpause time.

2. **Record pause start/end in the extension.** Store `pausedAt` when the pool is paused and subtract `(unpausedAt - pausedAt)` from `dt` in `_checkAndUpdateWatermarks`.

Either fix ensures the decay clock only advances while swaps are actually possible, mirroring the intent of the Arcadia fix (exclude sequencer downtime from the auction price curve).

### Proof of Concept

```
Setup:
  drawdownE6  = 50_000  (5 % max drawdown)
  decayPerSecondE8 = 58  (~5 %/day)
  Pool has bin 0 with watermark hwm0 = 1_000_000 (METRIC_SCALE units)

Step 1 – Normal operation:
  A swap sets hwm0 = 1_000_000, lastDecayTs = T0.

Step 2 – Admin pauses pool at T0 + 1 day.
  No swaps possible; lastDecayTs stays at T0.

Step 3 – Pool remains paused for 20 days (T0 + 21 days).
  block.timestamp = T0 + 21 days
  dt = 21 * 86_400 = 1_814_400 s
  factor = 58 * 1_814_400 = 105_235_200 >= 1e8
  → _decayed returns 0

Step 4 – Admin unpauses pool. Attacker immediately swaps.
  _checkAndUpdateWatermarks:
    decayed_hwm0 = 0
    metric0 (current, post-extraction) = 600_000  (40 % loss)
    _applyWatermark(600_000, 0, 950_000):
      metric >= hwm → newHwm = 600_000, breached = false  ← NO REVERT
    hwmS.token0 = 600_000, lastDecayTs = now

Result:
  40 % of LP value extracted with no OracleStopLossTriggered revert.
  Stop-loss baseline silently reset to the post-extraction level.
``` [5](#0-4)
