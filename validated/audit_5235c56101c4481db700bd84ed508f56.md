### Title
Missing Minimum Threshold Check in FROST `assert_sign_inputs` and `presign` Allows Corrupted Signing Outputs - (File: src/frost/mod.rs)

### Summary
The `assert_sign_inputs` and `presign` functions in `src/frost/mod.rs` validate that `threshold <= participants.len()` but omit the lower-bound check `threshold >= 2`. This check is present in `assert_key_invariants` in `src/dkg.rs` but is absent from the FROST signing path. A malicious coordinator or caller can supply `threshold = 1`, causing the FROST signing protocol to proceed with incorrect Lagrange coefficients relative to the key generation threshold, producing an invalid and unusable aggregate signature that honest parties cannot detect during the protocol.

### Finding Description

`assert_key_invariants` in `src/dkg.rs` enforces both bounds on `threshold`:

```rust
// src/dkg.rs lines 572-582
if threshold > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
if threshold < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
}
``` [1](#0-0) 

However, `assert_sign_inputs` in `src/frost/mod.rs` only checks the upper bound:

```rust
// src/frost/mod.rs lines 144-150
if threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
// No lower-bound check: threshold >= 2 is MISSING
``` [2](#0-1) 

The same omission exists in `presign`:

```rust
// src/frost/mod.rs lines 71-77
if args.threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
// No lower-bound check here either
``` [3](#0-2) 

`ReconstructionLowerBound` is a plain `usize` wrapper with no built-in minimum, so `threshold = 1` (or even `0`) passes through both functions without error: [4](#0-3) 

### Impact Explanation

Keys are always generated with `threshold >= 2` (enforced by `assert_key_invariants`). When `assert_sign_inputs` or `presign` is called with `threshold = 1`, the FROST signing protocol proceeds using Lagrange coefficients computed for a 1-of-N scheme, while the key shares were produced for a t-of-N scheme (t ≥ 2). The resulting signature shares are computed with wrong coefficients; the aggregate signature is cryptographically invalid. Because the library performs no internal signature verification (by design), honest participants accept and return the corrupted output without detecting the failure.

This matches: **High — Corruption of sign outputs so honest parties accept unusable cryptographic outputs.**

### Likelihood Explanation

The entry point is any caller of `assert_sign_inputs` or `presign` in the FROST path (EdDSA or RedJubjub). A malicious coordinator controlling the signing session parameters can trivially supply `threshold = 1`. The `ReconstructionLowerBound` type imposes no compile-time or runtime minimum, so no additional precondition is needed to trigger the bug.

### Recommendation

Add the same lower-bound check that `assert_key_invariants` already enforces, in both `assert_sign_inputs` and `presign` in `src/frost/mod.rs`:

```rust
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold: threshold.value(), min: 2 });
}
```

This mirrors the existing guard in `src/dkg.rs` at lines 580–582. [5](#0-4) 

### Proof of Concept

1. Run DKG with `participants = [P1, P2, P3]`, `threshold = 2`. Key shares are generated for a 2-of-3 scheme. `assert_key_invariants` enforces `threshold >= 2`. ✓
2. Call `assert_sign_inputs([P1, P2, P3], threshold=1, me=P1, coordinator=P1)`. Checks pass: `participants.len()=3 >= 2` ✓, `threshold=1 <= 3` ✓. No `threshold >= 2` check. Returns `Ok(participants)`.
3. Call `presign([P1, P2, P3], me=P1, args={threshold=1, keygen_out=...}, rng)`. Same missing check — protocol proceeds.
4. FROST signing aggregates shares using Lagrange coefficients for threshold=1 (i.e., coefficient = 1 for the single participant), but the key shares encode a degree-1 polynomial (threshold=2). The aggregate signature is invalid.
5. Honest parties receive `Ok(signature)` with no indication of failure. [6](#0-5) [7](#0-6)

/// Verifies that the sign inputs are valid
pub fn assert_sign_inputs(
    participants: &[Participant],
    threshold: impl Into<ReconstructionLowerBound>,
    me: Participant,
    coordinator: Participant,
) -> Result<ParticipantList, InitializationError> {
    let threshold = threshold.into();
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // validate threshold
    if threshold.value() > participants.len() {
        return Err(InitializationError::ThresholdTooLarge {
            threshold: threshold.value(),
            max: participants.len(),
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }
    Ok(participants)
}
