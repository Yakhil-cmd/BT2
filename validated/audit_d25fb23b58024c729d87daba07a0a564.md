### Title
Same Beaver Triple Accepted for Both `triple0` and `triple1` in OT-Based ECDSA Presigning, Enabling Private Key Extraction — (File: `src/ecdsa/ot_based_ecdsa/presign.rs`)

---

### Summary

The `presign()` function in the OT-based ECDSA pipeline accepts a `PresignArguments` struct containing two Beaver triples (`triple0`, `triple1`). The function validates threshold consistency and participant uniqueness but **never checks that `triple0` and `triple1` are distinct triples**. A malicious coordinator who distributes the same triple for both roles causes the presigning protocol to complete without error while leaking every honest participant's private key share through the broadcast values.

---

### Finding Description

The OT-based ECDSA presigning protocol requires two **independently generated** Beaver triples `(k, d, e)` and `(a, b, c)` with `e = kd` and `c = ab`. The protocol uses shares from `triple0` as the `(k, _, e)` role and shares from `triple1` as the `(a, b, c)` role.

In `src/ecdsa/ot_based_ecdsa/presign.rs`, the public entry point `presign()` performs the following checks: [1](#0-0) 

```rust
if args.threshold != args.triple0.1.threshold || args.threshold != args.triple1.1.threshold {
    return Err(InitializationError::BadParameters(
        "New threshold must match the threshold of both triples".to_string(),
    ));
}
```

There is **no check** that `args.triple0.1 != args.triple1.1` (i.e., that the two triples are cryptographically distinct). The `TriplePub` type derives `PartialEq` and `Eq`, making such a comparison trivially possible: [2](#0-1) 

When the same triple is supplied for both roles, the internal `do_presign` function extracts: [3](#0-2) 

- `a_i = triple1.0.a`, `k_i = triple0.0.a` → **both equal `a_i`**
- `e_i = triple0.0.c = a_i * b_i` (since `c = ab`)
- `big_k = big_a`, `big_d = big_b`, `big_e = big_c`

All three protocol consistency checks still pass:

1. **`e * G == big_e`**: `e = Σ λ_j · a_j · b_j = c`, and `big_e = C = c·G`. ✓
2. **`α·G == big_k + big_a`**: `α = Σ 2λ_j·a_j = 2a`, and `big_k + big_a = A + A = 2a·G`. ✓
3. **`β·G == big_x + big_b`**: `β = Σ λ_j·(x_j + b_j) = x + b`, and `big_x + big_b = (x+b)·G`. ✓

The protocol completes successfully and produces a valid presignature. No error is raised.

---

### Impact Explanation

Because `k_i = a_i`, the broadcast values sent by each honest participant `j` are:

| Broadcast value | Formula (same-triple case) |
|---|---|
| `e_j` | `λ_j · a_j · b_j` |
| `α_j` | `2 · λ_j · a_j` |
| `β_j` | `λ_j · (x_j + b_j)` |

An adversary who observes all three values for participant `j` can compute:

1. `λ_j · a_j = α_j / 2`
2. `b_j = e_j / (α_j / 2) = 2·e_j / α_j`
3. `x_j = β_j / λ_j − b_j`

Since `λ_j` (the Lagrange coefficient) is public, the adversary recovers the **private key share `x_j`** for every honest participant. With `t` or more shares, the full private signing key `x` is reconstructed.

This satisfies the Critical impact: **extraction of private signing shares and aggregate secret material**.

---

### Likelihood Explanation

**Low.** The attack requires a malicious coordinator or orchestrator who controls triple distribution to deliberately supply the same `TriplePub` for both `triple0` and `triple1` to all participants. In a correctly operated deployment this would not occur. However, the library provides no defense against this class of misuse, and a compromised or malicious orchestration layer (which is explicitly in scope as "malicious coordinator") can trigger it silently.

---

### Recommendation

Add a check in `presign()` that the two triples are distinct before the protocol begins:

```rust
// In src/ecdsa/ot_based_ecdsa/presign.rs, inside presign()
if args.triple0.1 == args.triple1.1 {
    return Err(InitializationError::BadParameters(
        "triple0 and triple1 must be distinct triples".to_string(),
    ));
}
```

Because `TriplePub` already derives `PartialEq` + `Eq`, this check is zero-cost and requires no additional infrastructure.

pub fn presign(
    participants: &[Participant],
    me: Participant,
    args: PresignArguments,
) -> Result<impl Protocol<Output = PresignOutput>, InitializationError> {
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }
    // Spec 1.1
    if args.threshold.value() > participants.len() {
        return Err(InitializationError::ThresholdTooLarge {
            threshold: args.threshold.value(),
            max: participants.len(),
        });
    }

    // NOTE: We omit the check that the new participant set was present for
    // the triple generation, because presumably they need to have been present
    // in order to have shares.

    // Also check that we have enough participants to reconstruct shares.
    if args.threshold != args.triple0.1.threshold || args.threshold != args.triple1.1.threshold {
        return Err(InitializationError::BadParameters(
            "New threshold must match the threshold of both triples".to_string(),
        ));
    }

    let participants =
        ParticipantList::new(participants).ok_or(InitializationError::DuplicateParticipants)?;

    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    let ctx = Comms::new();
    let fut = do_presign(ctx.shared_channel(), participants, me, args);
    Ok(make_protocol(ctx, fut))
}
