### Title
Missing Participant-Set Validation Against Triple's Recorded Participants in OT-Based ECDSA Presign - (File: src/ecdsa/ot_based_ecdsa/presign.rs)

### Summary
The OT-based ECDSA `presign` function records the participant set inside `TriplePub.participants` during triple generation but never validates that the signing participant set matches it. A malicious coordinator can invoke `presign` with a participant set that differs from the one used to generate the triples, causing Lagrange coefficients to be computed over the wrong set. Honest parties accept the resulting corrupted `PresignOutput` without any error, and the subsequent signing protocol produces an invalid signature, permanently consuming the one-time-use triple.

### Finding Description

**Root cause — the omitted check is explicitly acknowledged but not mitigated.**

`TriplePub` records the participant set that took part in triple generation: [1](#0-0) 

The `presign` entry-point validates that the threshold matches the triple's threshold, but explicitly skips the participant-set check with a comment that treats the invariant as an assumption rather than an enforced constraint: [2](#0-1) 

Inside `do_presign`, the Lagrange coefficient `lambda_me` is computed exclusively over the caller-supplied `participants` list, not over `triple.participants`: [3](#0-2) 

**Exploit path.**

1. Honest parties run triple generation with participant set `{A, B, C}` at threshold `t = 2`, producing `TriplePub { participants: [A,B,C], threshold: 2, … }` and per-party `TripleShare` values that are Shamir shares over `{A, B, C}`.
2. A malicious coordinator calls `presign` with `participants = [A, B]` (only two parties) while supplying the same `TripleShare` values and the same `TriplePub` (threshold still `2`).
3. The threshold equality check at line 43 passes (`2 == 2`). The participant-set check is absent.
4. `do_presign` computes `lambda_A` and `lambda_B` as Lagrange coefficients over `{A, B}`, not `{A, B, C}`. Because the shares `k_A`, `k_B` are Shamir shares over `{A, B, C}`, the interpolation `λ_A({A,B})·k_A + λ_B({A,B})·k_B ≠ k`. The resulting `PresignOutput.big_r`, `k`, and `sigma` are all cryptographically inconsistent.
5. No error is returned; honest parties store and forward the corrupted `PresignOutput`.
6. The signing protocol consumes the triple (it is one-time-use) and produces a signature that fails the final `sig.verify` check inside `do_sign_coordinator`, permanently denying that signing session. [4](#0-3) 

### Impact Explanation

Honest parties accept a `PresignOutput` whose internal scalars (`k`, `sigma`, `big_r`) are computed over the wrong participant set. This matches the allowed High impact: **Corruption of presign outputs so honest parties accept inconsistent participant sets and unusable cryptographic outputs.** Additionally, because triples are one-time-use, the consumed triple cannot be reused, causing **permanent denial of that signing session** for honest parties. A malicious coordinator can repeat this for every presign invocation, permanently blocking signing under valid protocol inputs.

### Likelihood Explanation

The `presign` function is a public API entry point. Any caller who controls the `participants` slice — including a malicious coordinator orchestrating the signing session — can supply a participant set that differs from `TriplePub.participants`. No special privilege beyond being a protocol participant is required. The threshold equality check at line 43 passes whenever the threshold value is the same, which is the common case (e.g., threshold `t` used for both triple generation and presign). The attack is therefore reachable by any malicious coordinator under normal operating conditions.

### Recommendation

Add an explicit check in `presign` that the caller-supplied `participants` set matches `args.triple0.1.participants` and `args.triple1.1.participants`. For example:

```rust
// Enforce that the signing participants match the triple's recorded participants.
let triple0_participants = ParticipantList::new(&args.triple0.1.participants)
    .ok_or(InitializationError::DuplicateParticipants)?;
let triple1_participants = ParticipantList::new(&args.triple1.1.participants)
    .ok_or(InitializationError::DuplicateParticipants)?;

if participants != triple0_participants || participants != triple1_participants {
    return Err(InitializationError::BadParameters(
        "Signing participants must match the participants recorded in both triples".to_string(),
    ));
}
```

This mirrors the existing threshold check and closes the gap between what `TriplePub.participants` records and what `presign` actually enforces.

pub fn presign(
    participants: &[Participant],
    me: Participant,
    args: PresignArguments,
) -> Result<impl Protocol<Output = PresignOutput>, InitializationError> {
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }
    // Spec 1.1
    if args.threshold.value() > participants.len() {
        return Err(InitializationError::ThresholdTooLarge {
            threshold: args.threshold.value(),
            max: participants.len(),
        });
    }

    // NOTE: We omit the check that the new participant set was present for
    // the triple generation, because presumably they need to have been present
    // in order to have shares.

    // Also check that we have enough participants to reconstruct shares.
    if args.threshold != args.triple0.1.threshold || args.threshold != args.triple1.1.threshold {
        return Err(InitializationError::BadParameters(
            "New threshold must match the threshold of both triples".to_string(),
        ));
    }

    let participants =
        ParticipantList::new(participants).ok_or(InitializationError::DuplicateParticipants)?;

    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    let ctx = Comms::new();
    let fut = do_presign(ctx.shared_channel(), participants, me, args);
    Ok(make_protocol(ctx, fut))
}
