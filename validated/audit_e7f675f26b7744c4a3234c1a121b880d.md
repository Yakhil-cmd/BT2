### Title
Missing Lower-Bound Threshold Validation in FROST Presign and Sign Allows Sub-Threshold Protocol Execution — (`src/frost/mod.rs`)

### Summary

`assert_sign_inputs` and `presign` in `src/frost/mod.rs` validate that `threshold <= participants.len()` but omit the symmetric lower-bound check (`threshold >= 2`) that is explicitly enforced in `assert_key_invariants` in `src/dkg.rs`. Because `ReconstructionLowerBound` is an unconstrained `usize` wrapper, a caller can supply `threshold = 1` (or `0`) to the FROST signing and presigning entry points. The protocol then executes with a sub-threshold value, producing a signature whose Lagrange reconstruction uses fewer shares than the key was generated with, yielding an unusable (cryptographically invalid) output that honest participants cannot distinguish from a legitimate failure.

### Finding Description

**Root cause — asymmetric validation between DKG and FROST signing.**

`assert_key_invariants` (`src/dkg.rs`, lines 558–596) enforces both bounds on the threshold:

```rust
if threshold > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { … });
}
if threshold < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
}
``` [1](#0-0) 

`assert_sign_inputs` (`src/frost/mod.rs`, lines 120–159) enforces only the upper bound:

```rust
if threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { … });
}
// ← no lower-bound check; threshold = 1 or 0 passes silently
``` [2](#0-1) 

The same omission exists in `presign` (`src/frost/mod.rs`, lines 44–88):

```rust
if args.threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { … });
}
// ← no lower-bound check
``` [3](#0-2) 

`ReconstructionLowerBound` is a plain `usize` newtype with no invariant enforcement at the type level:

```rust
pub struct ReconstructionLowerBound(usize);
``` [4](#0-3) 

**Exploit path.**

A malicious coordinator (or any library caller) constructs a `ReconstructionLowerBound(1)` and passes it to `assert_sign_inputs` or `presign`. Both functions accept it. The FROST signing protocol then proceeds: the coordinator collects only one signature share, computes Lagrange coefficients for a single-point set (coefficient = 1), and combines it into a final signature. Because the key was generated with threshold ≥ 2, the single share is a point-evaluation of the secret polynomial, not the secret itself. The resulting `s` value is wrong; the signature is invalid against the master public key.

### Impact Explanation

Honest participants complete their side of the protocol and receive what appears to be a successful signing output, but the produced signature is cryptographically invalid and unusable. This constitutes **corruption of sign outputs so honest parties accept unusable cryptographic outputs** — a **High** severity impact under the allowed scope. The coordinator can trigger this silently on any FROST EdDSA or RedJubjub signing session without any privileged key material. [5](#0-4) 

### Likelihood Explanation

The entry point is the public `assert_sign_inputs` and `presign` functions, callable by any library user acting as coordinator. No cryptographic secrets are required. The only precondition is the ability to initiate a signing session, which is the normal role of a coordinator. The `ReconstructionLowerBound` type imposes no compile-time or runtime lower-bound constraint, so the invalid value passes silently through all existing checks.

### Recommendation

Add the same lower-bound guard to both `assert_sign_inputs` and `presign` in `src/frost/mod.rs` that already exists in `assert_key_invariants`:

```rust
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

This mirrors the check at `src/dkg.rs` lines 580–582 and closes the asymmetry. Alternatively, encode the invariant directly in `ReconstructionLowerBound::new` so it is impossible to construct a value below 2. [6](#0-5) 

### Proof of Concept

```rust
// Attacker-controlled coordinator calls assert_sign_inputs with threshold = 1
let participants = vec![p1, p2, p3]; // 3 participants, key generated with threshold = 2

// Passes all current checks: 1 <= 3, coordinator present, self present
