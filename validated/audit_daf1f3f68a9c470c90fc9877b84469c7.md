### Title
Cross-COA Signature Replay via Missing `evmAddress` Binding in `signedData` — (`fvm/evm/stdlib/contract.cdc`, `fvm/evm/precompiles/arch.go`)

---

### Summary

`validateCOAOwnershipProof` verifies that signatures are valid over `signedData`, but `signedData` is never required to commit to `evmAddress` or `path`. Because both `evmAddress` and `path` are attacker-supplied ABI/RLP fields that are independently checked against each other (not against the signed payload), a valid signature obtained for COA_A can be replayed verbatim as a proof of ownership for COA_B, provided both COAs are owned by the same Cadence account. The code explicitly documents this gap.

---

### Finding Description

**Call chain:**

`proofVerifier.Run` → `DecodeABIEncodedProof` → `coaOwnershipProofValidator` → `validateCOAOwnershipProof`

**Step 1 — `proofVerifier.Run` / `DecodeABIEncodedProof` (`arch.go`)**

`DecodeABIEncodedProof` reads three independent attacker-controlled fields from the ABI-encoded input:
- `caller` → becomes `proof.EVMAddress` (the EVM address being claimed)
- `hash` → becomes `proof.SignedData`
- `encodedProof` → RLP-decoded into `COAOwnershipProof{Address, CapabilityPath, KeyIndices, Signatures}` [1](#0-0) 

There is **no check** that `caller` (EVMAddress) equals `msg.sender` or any other execution-context value. Any EVM transaction can supply any EVMAddress.

**Step 2 — Signature verification (`contract.cdc`)**

`validateCOAOwnershipProof` verifies the signatures over `signedData` using the Cadence account's keys: [2](#0-1) 

`signedData` is a raw byte array. Neither `evmAddress` nor `path` (CapabilityPath) is included in or committed to by `signedData`. The code itself documents this: [3](#0-2) 

**Step 3 — Address match check (`contract.cdc`)**

After signature verification, the code borrows the COA at the attacker-supplied `path` and checks that its address equals the attacker-supplied `evmAddress`: [4](#0-3) 

The attacker controls **both** sides of this comparison: they set `path = 'coa_b'` (in the RLP proof) and `evmAddress = COA_B.address` (in the ABI input). The check `coaRef.address().bytes == evmAddress` trivially passes because both values point to COA_B.

**The missing invariant:** The proof never binds the signed payload to the specific COA being claimed. The `COAOwnershipProof` struct carries `CapabilityPath` but it is not signed over: [5](#0-4) 

---

### Impact Explanation

An attacker who has obtained a valid `COAOwnershipProof` signed over any `signedData` for COA_A can replay it as a proof of ownership for COA_B (owned by the same Cadence account) by:
- Keeping the same `Address`, `KeyIndices`, and `Signatures`
- Changing `CapabilityPath` to `'coa_b'` in the RLP proof
- Setting `EVMAddress = COA_B.address` in the ABI input

The precompile returns `true`, granting the attacker ERC-1271 `isValidSignature` approval as COA_B. Any EVM smart contract that uses `verifyCOAOwnershipProof` for access control (e.g., token approvals, NFT marketplace orders, bridge authorization flows built on top of this primitive) will treat the attacker as the authorized owner of COA_B.

---

### Likelihood Explanation

**Preconditions required:**
1. The victim Cadence account must own two or more COAs (uncommon today but not prevented by the protocol).
2. The attacker must have obtained a valid `COAOwnershipProof` signed by the victim for COA_A — achievable by presenting a legitimate-looking authentication or sign-in request (e.g., "prove you own COA_A to use this dApp"), which is a standard wallet interaction.
3. The attacker must know COA_B's EVM address and its public capability path — both are publicly readable on-chain.

The comment's claim that "Flow wallets historically create at most one COA per account" is a social/historical observation, not a protocol constraint. As multi-COA patterns become more common (e.g., for account abstraction or bridge isolation), the precondition becomes easier to satisfy.

---

### Recommendation

Require that `signedData` commits to both `evmAddress` and `path` before signature verification. Concretely, `validateCOAOwnershipProof` should enforce:

```cadence
// Reconstruct the expected signed payload and verify it matches signedData
let expectedPayload = evmAddress.concat(path.toString().utf8)
// ... or use a canonical encoding that includes both fields
```

Alternatively, enforce at the precompile level (`DecodeABIEncodedProof`) that the `EVMAddress` ABI argument matches the `msg.sender` of the calling EVM context, so the proof cannot be submitted on behalf of an arbitrary address.

### Proof of Concept

On the Flow emulator:

```cadence
// Setup: one Cadence account with two COAs
transaction {
    prepare(account: auth(Capabilities, SaveValue) &Account) {
        let coaA <- EVM.createCadenceOwnedAccount()
        account.storage.save(<-coaA, to: /storage/coa_a)
        let capA = account.capabilities.storage.issue<&EVM.CadenceOwnedAccount>(/storage/coa_a)
        account.capabilities.publish(capA, at: /public/coa_a)

        let coaB <- EVM.createCadenceOwnedAccount()
        account.storage.save(<-coaB, to: /storage/coa_b)
        let capB = account.capabilities.storage.issue<&EVM.CadenceOwnedAccount>(/storage/coa_b)
        account.capabilities.publish(capB, at: /public/coa_b)
    }
}
```

```go
// Off-chain: sign signedData with the account's key (as if proving ownership of COA_A)
signedData := []byte("arbitrary data")
sig, _ := privateKey.Sign(signedData, hasher) // valid sig over signedData

// Craft proof pointing to COA_B's path, with COA_A's signatures
craftedProof := types.COAOwnershipProof{
    KeyIndices:     []uint64{0},
    Address:        flowAccount,
    CapabilityPath: "coa_b",   // <-- redirected to COA_B
    Signatures:     []types.Signature{sig},
}
encodedProof, _ := craftedProof.Encode()

// Call precompile with EVMAddress = COA_B.address
proofInCtx, _ := types.NewCOAOwnershipProofInContext(signedData, coaBAddress, encodedProof)
result, _ := proofVerifier(proofInCtx)
// result == true  ← COA_B ownership "proven" using COA_A's signature
```

The `validateCOAOwnershipProof` call passes because:
1. `keyList.verify(signedData, "FLOW-V0.0-user")` → valid (same key, same data)
2. `acc.capabilities.borrow(path='coa_b')` → returns COA_B reference
3. `coaRef.address().bytes == evmAddress` → `COB_B.address == COA_B.address` → `true` [6](#0-5) [7](#0-6)

func DecodeABIEncodedProof(input []byte) (*types.COAOwnershipProofInContext, error) {
	index := 0
	caller, err := ReadAddress(input, index)
	index += FixedSizeUnitDataReadSize
	if err != nil {
		return nil, err
	}

	hash, err := ReadBytes32(input, index)
	index += Bytes32DataReadSize
	if err != nil {
		return nil, err
	}

	encodedProof, err := ReadBytes(input, index)
	if err != nil {
		return nil, err
	}

	return types.NewCOAOwnershipProofInContext(
		hash,
		types.Address(caller),
		encodedProof,
	)
}
