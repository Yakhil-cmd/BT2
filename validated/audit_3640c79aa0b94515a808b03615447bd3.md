### Title
Malicious Participant Can Corrupt CKD Output by Using a Different `app_id` — (`src/confidential_key_derivation/protocol.rs`)

### Summary

The CKD protocol has no cross-participant consistency check on `app_id`. Each participant computes their share using their locally-supplied `app_id`, and the coordinator blindly aggregates the resulting EC points. A single malicious participant who passes a different `app_id` to their `ckd()` call silently corrupts the aggregated output, producing an EC point that is not a valid BLS signature for any `app_id`, making the derived key permanently unusable for the requesting app.

---

### Finding Description

In `compute_signature_share`, each participant independently computes:

```
hash_point = H(pk || app_id)   // local app_id, never verified
S_i = x_i * hash_point
C_i = S_i + y_i * A
``` [1](#0-0) 

The resulting `(λ_i·Y_i, λ_i·C_i)` is sent privately to the coordinator. The coordinator aggregates all received contributions:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
``` [2](#0-1) 

There is **no commitment to `app_id`** in the protocol messages, and **no proof of correct computation** (no ZK proof, no Pedersen commitment, no consistency check). The coordinator receives raw EC points and adds them — it cannot detect that a participant used a different `app_id`.

If participant `j` uses `app_id_j ≠ app_id`, the aggregated `C` becomes:

```
C = Σ_{i≠j} λ_i·x_i·H(pk||app_id) + λ_j·x_j·H(pk||app_id_j) + (Σ λ_i·y_i)·A
```

After the app unmasks with `C − a·Y`, the result is:

```
Σ_{i≠j} λ_i·x_i·H(pk||app_id) + λ_j·x_j·H(pk||app_id_j)
```

This is not `msk·H(pk||app_id)` for any `app_id`, so `verify_signature` will return `Err(InvalidSignature)`. [3](#0-2) 

---

### Impact Explanation

The app receives a `CKDOutput` whose `unmask()` produces an EC point that fails BLS pairing verification. The app cannot derive its deterministic key. Because the corruption is silent (the coordinator returns `Ok(Some(ckd_output))` with no error), the honest coordinator and honest participants have no indication anything went wrong.

Impact: **High — Corruption of CKD output so honest parties accept an unusable cryptographic output**, matching the stated scope.

---

### Likelihood Explanation

The `ckd()` function is a public library API. Any participant node that is under attacker control simply passes a different byte string as `app_id`. No cryptographic capability is required — it is a trivial one-line change at the call site. A single malicious participant out of `n` is sufficient to corrupt every CKD request routed through that participant. [4](#0-3) 

---

### Recommendation

Bind `app_id` into the protocol messages so the coordinator can verify consistency. Two standard approaches:

1. **Commitment in the message**: Each participant includes a hash commitment `com_i = H("ckd-app-id" || app_id || nonce_i)` alongside `(Y_i, C_i)`. The coordinator checks all commitments match before aggregating.
2. **Include `app_id` in the sent tuple**: Participants send `(app_id, norm_big_y, norm_big_c)` and the coordinator rejects any message whose `app_id` differs from its own.

Option 2 is simpler and sufficient since the coordinator is trusted to enforce consistency before aggregation.

---

### Proof of Concept

```rust
// Participant 1 (honest): app_id = b"real_app"
let protocol_p1 = ckd(&participants, coordinator, p1, key_pair_1,
    AppId::try_from(b"real_app").unwrap(), app_pk, rng1).unwrap();

// Participant 2 (malicious): app_id = b"evil_app"  <-- different!
let protocol_p2 = ckd(&participants, coordinator, p2, key_pair_2,
    AppId::try_from(b"evil_app").unwrap(), app_pk, rng2).unwrap();

// Run protocol — coordinator returns Ok(Some(ckd_output)) with no error
let result = run_protocol(vec![(p1, Box::new(protocol_p1)),
                               (p2, Box::new(protocol_p2))]).unwrap();

let ckd_output = /* extract coordinator output */;
let derived = ckd_output.unmask(app_sk);

// verify_signature fails — output is permanently unusable
assert!(verify_signature(&public_key, b"real_app", &derived).is_err());
```

The coordinator produces `Ok(Some(...))` with no error, yet the output is cryptographically invalid. The honest app has no recourse.

pub fn ckd(
    participants: &[Participant],
    coordinator: Participant,
    me: Participant,
    key_pair: KeygenOutput,
    app_id: impl Into<AppId>,
    app_pk: PublicKey,
    rng: impl CryptoRngCore + Send + 'static,
) -> Result<impl Protocol<Output = CKDOutputOption>, InitializationError> {
    // not enough participants
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }

    // kick out duplicates
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }

    let comms = Comms::new();
    let chan = comms.shared_channel();

    let fut = run_ckd_protocol(
        chan,
        coordinator,
        me,
        participants,
        key_pair,
        app_id.into(),
        app_pk,
        rng,
    );
    Ok(make_protocol(comms, fut))
}
