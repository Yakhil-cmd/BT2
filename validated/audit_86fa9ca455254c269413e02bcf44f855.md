### Title
Malicious Participant Can Corrupt CKD Output Without Detection — (File: `src/confidential_key_derivation/protocol.rs`)

### Summary
In the Confidential Key Derivation (CKD) protocol, the coordinator accumulates `CKDOutput` contributions from every participant by simple addition, with no cryptographic proof of correctness attached to any received value. A single malicious participant can send arbitrary `(big_y, big_c)` group elements; the coordinator blindly adds them into the running sum, silently producing a corrupted confidential key that honest parties will accept as valid.

### Finding Description
`do_ckd_coordinator` collects each participant's share and sums them unconditionally:

```rust
// src/confidential_key_derivation/protocol.rs  lines 50-55
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
``` [1](#0-0) 

The honest computation performed by every participant in `compute_signature_share` is:

```
norm_big_y = λᵢ · yᵢ · G
norm_big_c = λᵢ · (xᵢ · H(pk ‖ app_id) + yᵢ · app_pk)
``` [2](#0-1) 

No zero-knowledge proof, commitment, or consistency check is sent alongside these values. The coordinator has no means to distinguish a correctly computed share from an arbitrary group element. Contrast this with the DKG protocol, which attaches a proof-of-knowledge and a commitment hash to every polynomial coefficient before accepting any share: [3](#0-2) 

The CKD protocol has no equivalent guard.

### Impact Explanation
Let the honest aggregate be `(Y_honest, C_honest)`. A malicious participant substitutes `(ΔY, ΔC)` for their correct contribution. The coordinator outputs:

```
big_y_final = Y_honest + ΔY
big_c_final = C_honest + ΔC
```

The `unmask` step computes `big_c_final − app_sk · big_y_final`. For the result to equal the true confidential key `msk · H(pk ‖ app_id)`, the malicious participant would need `ΔC = app_sk · ΔY`, i.e., knowledge of the application secret key. Without that knowledge, any non-zero `(ΔY, ΔC)` pair that does not satisfy this relation silently corrupts the derived key. Honest parties receive and store the wrong key with no indication of failure.

**Matched allowed impact:** *High — Corruption of CKD outputs so honest parties accept unusable or incorrect cryptographic outputs.*

### Likelihood Explanation
The attack requires only that one participant in the CKD session be malicious. No special cryptographic capability is needed: the attacker simply replaces their `send_private` payload with arbitrary group elements. The coordinator performs no validation before incorporating the value. Because participants send their shares privately to the coordinator, other honest participants cannot observe or flag the deviation. [4](#0-3) 

### Recommendation
Require each participant to accompany their `(norm_big_y, norm_big_c)` with a Sigma-protocol (or equivalent NIZK) proving:

1. Knowledge of `yᵢ` such that `norm_big_y = λᵢ · yᵢ · G`.
2. Consistency: `norm_big_c = λᵢ · (xᵢ · H(pk ‖ app_id) + yᵢ · app_pk)`, binding `xᵢ` to the participant's public verification share from the DKG output.

The coordinator must verify all proofs before summing contributions and abort if any proof fails, identifying the malicious participant.

### Proof of Concept

1. All `n` participants run `ckd(...)`. Honest participants compute correct `(norm_big_y, norm_big_c)` and call `chan.send_private(waitpoint, coordinator, &(norm_big_y, norm_big_c))`.
2. Malicious participant `m` instead calls `chan.send_private(waitpoint, coordinator, &(ΔY, ΔC))` for arbitrary non-zero `ΔY, ΔC` with `ΔC ≠ app_sk · ΔY`.
3. The coordinator's loop at lines 50–55 adds `ΔY` and `ΔC` into the running sums without any check.
4. `CKDOutput::new(norm_big_y, norm_big_c)` is returned with corrupted values.
5. Any caller of `ckd_output.unmask(app_sk)` recovers `msk · H(pk ‖ app_id) + (ΔC − app_sk · ΔY)` — a wrong key — with no error or warning. [5](#0-4)

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
```
