### Title
Malicious CKD Participant Corrupts Derived Key Output via Unverified Share Aggregation — (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary
The CKD coordinator blindly aggregates `(norm_big_y, norm_big_c)` tuples from every participant with no cryptographic proof that each tuple was computed honestly. Any single malicious participant can inject arbitrary group elements, causing the coordinator to output a permanently unusable derived key. This is the direct analog of the ETH2.0 deposit bug: just as the deposit contract accepted a top-up without validating the expected deposit root, the CKD coordinator accepts participant shares without validating the expected computation.

---

### Finding Description

`do_ckd_coordinator` (lines 35–58) drives the aggregation phase:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
Ok(Some(ckd_output))
```

Each participant is supposed to compute, inside `compute_signature_share` (lines 148–182):

```
norm_big_y = λ_i · y_i · G
norm_big_c = λ_i · (x_i · H(pk ‖ app_id) + app_pk · y_i)
```

so that the coordinator's sum satisfies:

```
C − a · Y  =  msk · H(pk ‖ app_id)
```

where `a` is the application's secret key.

**No proof of correct computation is attached to the participant's message.** The coordinator cannot distinguish a legitimately computed `(norm_big_y, norm_big_c)` from an arbitrary pair of group elements. The participant's `do_ckd_participant` function (lines 17–33) simply sends whatever `compute_signature_share` returns—and a malicious caller can substitute any values before or instead of calling that function, because the protocol message is just a serialized `CKDOutput` struct with two `ElementG1` fields.

The root cause is the complete absence of a DLEQ / Schnorr proof binding `norm_big_y` and `norm_big_c` to the participant's committed signing share and the agreed `app_pk`.

---

### Impact Explanation

**High — Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.**

When a malicious participant injects `(norm_big_y', norm_big_c')` ≠ correct values, the coordinator computes:

```
Y_bad = Y_honest + norm_big_y'
C_bad = C_honest + norm_big_c'
```

Unmasking yields:

```
C_bad − a · Y_bad  =  msk · H(pk ‖ app_id)  +  (norm_big_c' − a · norm_big_y')
```

The extra term `(norm_big_c' − a · norm_big_y')` is non-zero for any adversarially chosen pair, so the derived confidential key is permanently wrong. The coordinator has no way to detect the corruption; it returns `Some(ckd_output)` as if the protocol succeeded. Every downstream consumer of the derived key (e.g., a TEE that uses it to decrypt secrets) will silently fail.

---

### Likelihood Explanation

**High.** The attacker only needs to be one of the `n` participants listed in the `participants` slice passed to `ckd()`. No privileged role is required—the coordinator is explicitly excluded from the attack path (the coordinator computes its own share honestly and is the victim of the corrupted sum). The malicious participant simply sends two arbitrary `G1` points instead of the correct share; no cryptographic capability is needed.

---

### Recommendation

Attach a DLEQ (Discrete-Log Equality) proof to each participant's message proving that `norm_big_c` was formed with the same `y_i` used in `norm_big_y` and with the agreed `app_pk`. Concretely, each participant should prove:

```
DLEQ( norm_big_y / λ_i  =  y_i · G ,
      (norm_big_c / λ_i − big_s)  =  y_i · app_pk )
```

where `big_s = x_i · H(pk ‖ app_id)` is committed via the participant's public signing share. The coordinator must verify this proof before adding the contribution to the running sum, analogous to the ETH2.0 fix of checking `get_deposit_root()` against an expected value before accepting a deposit.

### Proof of Concept

```
Setup:
  participants = [P1 (honest), P2 (honest), M (malicious)]
  app_pk = a · G  (legitimate application key)
  app_id = "myapp"

Step 1 – M sends garbage:
  M sends  norm_big_y' = G,  norm_big_c' = G
  (two arbitrary non-identity points, unrelated to any secret share)

Step 2 – Coordinator aggregates:
  Y_bad = (λ_1·y_1 + λ_2·y_2)·G  +  G
  C_bad = msk·H(pk‖app_id) + app_pk·(λ_1·y_1 + λ_2·y_2)·G  +  G

Step 3 – Application unmasks:
  C_bad − a·Y_bad
    = msk·H(pk‖app_id) + G − a·G
    = msk·H(pk‖app_id) + (1 − a)·G   ← wrong

Step 4 – Impact:
  The derived confidential key is permanently incorrect.
  The coordinator returns Ok(Some(ckd_output)) with no error.
  No honest party can detect the corruption without an external oracle.
```
