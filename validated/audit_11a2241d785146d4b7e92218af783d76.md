### Title
`app_pk` Not Bound to BLS Signature Share Enables Coordinator Split-View Corruption of CKD Output — (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

In the CKD protocol, each participant computes a BLS signature share `big_s_i = H(pk || app_id) * x_i` and an ElGamal ciphertext `big_c_i = big_s_i + app_pk * y_i`. The ElGamal public key `app_pk` is used in the ciphertext but is **not included in the hash** `H(pk || app_id)`. Because `big_s_i` does not commit to `app_pk`, participants have no cryptographic means to verify that they are all operating on the same `app_pk`. A malicious coordinator can present different `app_pk` values to different participants within the same session, causing the aggregated CKD output to be irreversibly corrupted and undecryptable.

---

### Finding Description

**Root cause — `app_pk` absent from the hash:**

In `src/confidential_key_derivation/mod.rs` lines 67–70, `hash_app_id_with_pk` hashes only the master public key `pk` and the `app_id`:

```rust
pub fn hash_app_id_with_pk(pk: &VerifyingKey, app_id: &[u8]) -> ElementG1 {
    let compressed_pk = pk.to_element().to_compressed();
    let input = [compressed_pk.as_slice(), app_id].concat();
    ciphersuite::hash_to_curve(&input)
}
```

`app_pk` (the ElGamal public key) is entirely absent from this hash. [1](#0-0) 

**How `app_pk` is used without being bound:**

In `compute_signature_share` (`src/confidential_key_derivation/protocol.rs`, lines 148–182):

```rust
let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);
let big_s = hash_point * private_share.to_scalar();   // no app_pk here
let big_c = big_s + app_pk * y.0;                     // app_pk used here only
```

`big_s` is the BLS signature share. It is identical for any value of `app_pk`. The `app_pk` only enters through the ElGamal blinding term `app_pk * y`, which is private to each participant. The coordinator never sees `y`, so it cannot recompute or substitute `app_pk` in a received share — but it **can** supply a different `app_pk` to each participant before the session begins. [2](#0-1) 

**No cross-participant consistency check:**

In `do_ckd_coordinator` (lines 35–57), the coordinator simply sums received shares with no verification:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
```

There is no check that all participants used the same `app_pk`, no ZK proof of correct share formation, and no session binding that would tie `app_pk` to the aggregated output. [3](#0-2) 

---

### Impact Explanation

**Category:** Corruption of CKD outputs — **High**.

A malicious coordinator supplies `app_pk = A` to participant P₁ and `app_pk = A′ ≠ A` to participant P₂. The aggregated ciphertext becomes:

```
big_c = λ₁·(S₁ + A·y₁) + λ₂·(S₂ + A′·y₂)
      = msk·H(pk‖app_id) + λ₁·A·y₁ + λ₂·A′·y₂
```

When the application decrypts with secret `a` (where `A = a·G₁`):

```
big_c − a·big_y = msk·H(pk‖app_id) + λ₁·(A − A)·y₁ + λ₂·(A′ − A)·y₂
                = msk·H(pk‖app_id) + λ₂·(A′ − A)·y₂
                ≠ msk·H(pk‖app_id)
```

The output is permanently corrupted. The application cannot recover the correct confidential key. Honest participants complete the protocol without error and have no way to detect the corruption.

---

### Likelihood Explanation

In any deployment where the coordinator is the entity that distributes the CKD request (including `app_pk`) to other participants — the common MPC orchestration model — a single malicious coordinator node can trivially execute this attack with no special capabilities beyond its normal protocol role. The attack requires zero cryptographic breaks, no leaked keys, and no external assumptions. It is a one-round, single-session attack.

---

### Recommendation

**Short term:** Include `app_pk` in the hash so that the BLS signature share commits to it:

```rust
pub fn hash_app_id_with_pk(pk: &VerifyingKey, app_id: &[u8], app_pk: &ElementG1) -> ElementG1 {
    let compressed_pk = pk.to_element().to_compressed();
    let compressed_app_pk = app_pk.to_compressed();
    let input = [compressed_pk.as_slice(), app_id, compressed_app_pk.as_slice()].concat();
    ciphersuite::hash_to_curve(&input)
}
```

This ensures that a share computed for `app_pk = A` is cryptographically invalid for `app_pk = A′`, directly preventing the split-view attack.

**Long term:** Add a ZK proof of correct share formation (a Schnorr-style proof that `big_s_i` is a valid BLS signature share and `big_c_i` is a correctly formed ElGamal ciphertext under the agreed `app_pk`). This prevents any malicious participant — not just the coordinator — from submitting arbitrary shares.

---

### Proof of Concept

**Setup:** 2-of-2 CKD session. Participants P₁ and P₂. Master secret key `msk = x₁ + x₂`. Application ElGamal key pair `(a, A)` where `A = a·G₁`.

**Attack steps:**

1. Malicious coordinator calls `ckd(..., app_pk = A, ...)` for itself (P₁).
2. Malicious coordinator calls `ckd(..., app_pk = A′, ...)` for P₂, where `A′ ≠ A`.
3. P₁ computes `norm_big_c₁ = λ₁·(H(pk‖app_id)·x₁ + A·y₁)` and sends to coordinator.
4. P₂ computes `norm_big_c₂ = λ₂·(H(pk‖app_id)·x₂ + A′·y₂)` and sends to coordinator.
5. Coordinator aggregates: `big_c = norm_big_c₁ + norm_big_c₂`.
6. Application decrypts: `big_c − a·big_y = msk·H(pk‖app_id) + λ₂·(A′ − A)·y₂ ≠ msk·H(pk‖app_id)`.

**Expected outcome:** Protocol completes without error on all honest nodes. The returned `CKDOutput` is silently corrupted. The application's `unmask(a)` call yields a wrong group element, permanently denying the correct confidential key. [4](#0-3) [1](#0-0)

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}
