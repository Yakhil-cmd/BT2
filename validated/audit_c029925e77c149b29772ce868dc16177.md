### Title
Unbound `app_pk` in CKD Hash Allows Malicious Participant to Corrupt CKD Output — (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

In `compute_signature_share`, the hash `H(pk || app_id)` used to compute each participant's BLS signature share does not include `app_pk` (the app's ElGamal public key `A`). Because `app_pk` is the field that determines who can decrypt the CKD output, and because no proof of correct encryption is required, a malicious participant can substitute their own `app_pk` when computing their share. The coordinator aggregates all contributions without verification, producing a `CKDOutput` that is permanently undecryptable by the legitimate app.

---

### Finding Description

In `compute_signature_share` (`src/confidential_key_derivation/protocol.rs`, lines 148–182), each participant computes:

```
hash_point = H(pk || app_id)          // app_pk is NOT included
S_i        = x_i · hash_point
C_i        = S_i + y_i · app_pk       // app_pk used here but not bound above
``` [1](#0-0) 

The function `hash_app_id_with_pk` hashes only the MPC network public key `pk` and `app_id`, explicitly omitting `app_pk`:

```rust
pub fn hash_app_id_with_pk(pk: &VerifyingKey, app_id: &[u8]) -> ElementG1 {
    let compressed_pk = pk.to_element().to_compressed();
    let input = [compressed_pk.as_slice(), app_id].concat();
    ciphersuite::hash_to_curve(&input)
}
``` [2](#0-1) 

Each participant sends `(norm_big_y, norm_big_c)` privately to the coordinator. The coordinator in `do_ckd_coordinator` aggregates all received values with no verification that participants used the correct `app_pk`:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
``` [3](#0-2) 

Because `app_pk` is not bound in the hash and no proof of correct encryption is attached to each share, a malicious participant can call `ckd()` with `app_pk = A_attacker` (their own ElGamal key). Their contribution becomes `C_malicious = S_malicious + y_malicious · A_attacker`. The coordinator aggregates this with honest participants' shares (which used `A_legitimate`), producing:

```
C_final = msk_partial · H(pk, app_id)
        + (Σ λ_j · y_j) · A_legitimate
        + λ_malicious · y_malicious · A_attacker
```

This `C_final` is not a valid ElGamal ciphertext under any single key. The legitimate app cannot recover `msk · H(pk, app_id)` from it, and neither can the attacker.

---

### Impact Explanation

**High — Corruption of CKD output so honest parties receive an unusable cryptographic output.**

The final `CKDOutput` `(Y, C)` delivered to the legitimate app is permanently corrupted. The app's `unmask(a)` call — computing `C - a · Y` — will not yield `msk · H(pk, app_id)` because `C` contains a term encrypted under `A_attacker`. The derived confidential key is irrecoverably wrong. The CKD protocol must be re-run (if the on-chain contract permits it), and the malicious participant can repeat the attack indefinitely, constituting permanent denial of CKD for honest parties. [4](#0-3) 

---

### Likelihood Explanation

**Medium.** The attack requires one malicious MPC participant — an explicitly in-scope attacker type per the audit method. The participant only needs to pass a different `app_pk` value when invoking `ckd()`. No cryptographic break, no key leakage, and no external dependency is required. The attack is trivially repeatable on every CKD request. [5](#0-4) 

---

### Recommendation

Add a **proof of correct ElGamal encryption** to each participant's share. Each participant should attach a NIZK (e.g., a Chaum-Pedersen proof) demonstrating that `C_i - S_i = y_i · A` for the agreed-upon `A`, where `Y_i = y_i · G`. The coordinator must verify this proof before aggregating any share.

Alternatively, bind `app_pk` into the hash: `H(pk || app_id || app_pk)`. This makes the BLS signature share dependent on `app_pk`, so a participant using a different `app_pk` produces a share that is inconsistent with the others and can be detected. Note that this changes the derived key for different `app_pk` values, which is acceptable since `app_pk` is a fresh per-request key. [6](#0-5) 

---

### Proof of Concept

1. Honest participants `P_1, …, P_{n-1}` call `ckd(participants, coordinator, p_i, key_pair, app_id, A_legitimate, rng)`.
2. Malicious participant `P_n` calls `ckd(participants, coordinator, p_n, key_pair, app_id, A_attacker, rng)` where `A_attacker = a_attacker · G_1` is their own ElGamal key.
3. `P_n`'s `compute_signature_share` computes `hash_point = H(pk || app_id)` (identical to honest participants, since `app_pk` is absent from the hash), then `C_n = S_n + y_n · A_attacker`.
4. `P_n` sends `(λ_n · Y_n, λ_n · C_n)` to the coordinator.
5. The coordinator aggregates without verification:
   `C_final = Σ_{j≠n} λ_j · C_j + λ_n · C_n`
   `= msk · H(pk, app_id) + (Σ_{j≠n} λ_j · y_j) · A_legitimate + λ_n · y_n · A_attacker`
6. The legitimate app receives `(Y_final, C_final)` and computes `C_final - a_legitimate · Y_final`, which does not equal `msk · H(pk, app_id)` because of the `A_attacker` term.
7. `verify_signature` fails; the CKD output is unusable. [7](#0-6) [8](#0-7)

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}

pub fn verify_signature(
    verifying_key: &VerifyingKey,
    msg: &[u8],
    signature: &Signature,
) -> Result<(), frost_core::Error<BLS12381SHA256>> {
    let element1: G1Affine = signature.into();
    if (!element1.is_on_curve() | !element1.is_torsion_free() | element1.is_identity()).into() {
        return Err(frost_core::Error::InvalidSignature);
    }
    let element2: G2Affine = verifying_key.to_element().into();
    if (!element2.is_on_curve() | !element2.is_torsion_free() | element2.is_identity()).into() {
        return Err(frost_core::Error::MalformedVerifyingKey);
    }

    // Concatenate the master public key (96 bytes) in the hash computation
    // H(pk || app_id) when H is a random oracle
    let base1 = hash_app_id_with_pk(verifying_key, msg).into();
    let base2 =
        <<BLS12381SHA256 as frost_core::Ciphersuite>::Group as frost_core::Group>::generator()
            .into();

    if blstrs::pairing(&base1, &element2).eq(&blstrs::pairing(&element1, &base2)) {
        Ok(())
    } else {
        Err(frost_core::Error::InvalidSignature)
    }
