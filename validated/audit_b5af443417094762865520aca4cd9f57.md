### Title
Pool Admin Bypasses `maxAdminSpreadFeeE6` Cap via Uncapped `setPoolBinAdditionalFees` — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary
`MetricOmmPoolFactory.setPoolBinAdditionalFees` enforces no upper-bound check on `addFeeBuyE6` / `addFeeSellE6` against `maxAdminSpreadFeeE6`, while the parallel path `setPoolAdminFees` does. A pool admin can therefore set per-bin additional fees up to `uint16.max = 65535` (6.5535%) on any bin, regardless of the configured admin fee cap, and traders executing swaps through that bin pay excess fees beyond what the protocol's cap system permits.

### Finding Description
The factory maintains `maxAdminSpreadFeeE6` as the ceiling for admin-controlled spread fees. `setPoolAdminFees` enforces this ceiling:

```solidity
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
``` [1](#0-0) 

`setPoolBinAdditionalFees` performs no equivalent check:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
``` [5](#0-4) 

`setBinAdditionalFees` on the pool only validates the bin index: [3](#0-2) 

The stored `addFeeBuyE6` / `addFeeSellE6` are then added directly to `baseFeeX64` in every swap through that bin, with no clamping: [4](#0-3) 

The same uncapped values appear in `getSellAndBuyPrices`: [6](#0-5) 

### Impact Explanation
Traders swapping through a bin with inflated additional fees pay more than the protocol's configured maximum allows. The excess fee is extracted from trader token balances on every swap through that bin. This is a direct loss of user principal and an admin-boundary break: the pool admin exceeds the cap the factory owner set to protect traders.

### Likelihood Explanation
Requires a malicious or compromised pool admin. The cap system exists precisely to bound what a pool admin can do to traders; the bin-level path is a second, unchecked route to the same effect. Any pool whose admin is not fully trusted is exposed.

### Recommendation
Add a cap check in `setPoolBinAdditionalFees` before forwarding to the pool:

```solidity
if (addFeeBuyE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
```

### Proof of Concept
1. Factory owner sets `maxAdminSpreadFeeE6 = 1_000` (0.1%).
2. Pool admin calls `factory.setPoolBinAdditionalFees(pool, 0, 65535, 65535)` — no revert.
3. `_binStates[0].addFeeBuyE6 = 65535`, `addFeeSellE6 = 65535` are stored.
4. Trader executes a swap through bin 0.
5. Swap math applies `baseFeeX64 + mulDiv(65535, ONE_X64, 1e6)` — an additional 6.5535% spread fee — far exceeding the 0.1% cap.
6. Assert: fee charged > `maxAdminSpreadFeeE6`.
