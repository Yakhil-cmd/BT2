### Title
Unbounded `stakers` Vec Growth Enables Permanent DoS on `get_stakers()` via Stake-Unstake Cycling - (File: src/staking/staking.cairo)

### Summary

The `stakers` storage `Vec` in the `Staking` contract grows permanently with every unique staker address that has ever called `stake()`. Staker addresses are **never removed** from this Vec, even after a full `unstake_intent()` + `unstake_action()` cycle. The `get_stakers()` function iterates over the **entire** Vec on every call. An attacker can cycle through many addresses — staking the minimum, then unstaking to recover capital — to permanently inflate the Vec and cause unbounded gas consumption in `get_stakers()`.

### Finding Description

In `stake()`, every new staker address is appended to the `stakers` Vec: [1](#0-0) 

The storage declaration explicitly documents that entries are never removed: [2](#0-1) 

`remove_staker()`, called during `unstake_action()`, clears `staker_info`, the operational address mapping, and pool info — but never touches `self.stakers`: [3](#0-2) 

`get_stakers()` iterates over the **full range** of the Vec on every invocation: [4](#0-3) 

Although inactive stakers are skipped via `is_staker_active`, every entry in the Vec is still read from storage. As the Vec grows, the gas cost of `get_stakers()` grows proportionally and without bound.

### Impact Explanation

`get_stakers()` is the consensus-layer entry point for determining the active staker set for a given epoch. If this function exceeds the Starknet gas limit due to Vec bloat, the consensus layer cannot retrieve the staker list, causing a **permanent denial of service** for the consensus reward distribution mechanism. This matches the **Medium: Unbounded gas consumption / Griefing** impact category.

### Likelihood Explanation

The attack requires:
1. Many unique staker addresses (trivially generated on Starknet).
2. Each address stakes `min_stake` STRK, then calls `unstake_intent()` and waits for the exit window, then calls `unstake_action()` to **fully recover the staked capital**.

The only permanent cost to the attacker is gas fees for the stake/unstake transactions. The staked capital is fully returned. This makes the attack economically viable at scale. The `min_stake` check at line 317 raises the per-address cost but does not prevent the attack since funds are recoverable: [5](#0-4) 

### Recommendation

One of the following mitigations should be applied:

1. **Remove stakers from the Vec on `unstake_action`**: Implement a swap-and-pop or similar pattern to remove the staker's address from `self.stakers` when `remove_staker()` is called.
2. **Paginate `get_stakers()`**: Accept an offset/limit parameter so the iteration is bounded per call.
3. **Maintain a separate active-staker count**: Track only active stakers in a separate bounded structure used by `get_stakers()`, keeping the full Vec only for historical/migration purposes.

### Proof of Concept

```
// Attacker flow (pseudocode):
for i in 0..N:
    addr_i = new_address()
    // Fund addr_i with min_stake STRK
    staking.stake(addr_i, reward_addr, op_addr, min_stake)
    // stakers.len() is now i+1, entry is permanent
    staking.unstake_intent()          // called from addr_i
    advance_time(exit_wait_window)
    staking.unstake_action(addr_i)    // capital fully recovered
    // addr_i entry remains in self.stakers forever

// After N iterations:
// self.stakers.len() == N (all inactive, but all iterated)
// get_stakers(epoch_id) must read all N entries
// Gas cost = O(N) — unbounded
```

The `DiverseStakerVecFlow` test in the codebase already demonstrates that unstaked stakers remain in the Vec: [6](#0-5) 

The test confirms that after `staker_in_action` completes a full exit, `get_stakers()` still returns 4 entries (including the exited staker's address in the Vec), confirming the permanent growth behavior.
