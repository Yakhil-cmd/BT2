### Title
Unauthorized Contract Update Bypasses Restricted Deployment Authorization - (File: `fvm/environment/contract_updater.go`)

---

### Summary

When restricted contract deployment is enabled on the Flow network, the FVM's `SetContract` function enforces the deployment authorization check **only for new contract deployments**. Updates to already-deployed contracts bypass this check entirely. Any account that has a contract deployed â€” even one that is no longer in the authorized deployment list â€” can update that contract to arbitrary code, defeating the purpose of the restriction.

---

### Finding Description

`SetContract` in `fvm/environment/contract_updater.go` is the FVM-level gate for all contract write operations. It is invoked by `UpdateAccountContractCode` (the Cadence runtime callback for `account.contracts.update()`): [1](#0-0) 

Inside `SetContract`, the authorization check is conditioned on `!exists`: [2](#0-1) 

The comment at line 381 explicitly states: **"Contract updates are always allowed."** This means when a contract already exists at the target address, `isAuthorizedForDeployment` is never called, and the update proceeds unconditionally.

The authorization function that is bypassed: [3](#0-2) 

When `RestrictedDeploymentEnabled()` is true, `isAuthorizedForDeployment` checks the signer against the `ContractDeploymentAuthorizedAddressesPath` list stored in the service account. The default configuration enables this restriction: [4](#0-3) 

Critically, the docstring for `SetContractDeploymentAuthorizersTransaction` in `fvm/blueprints/contracts.go` describes the list as controlling accounts "allowed to **deploy/update** contracts" â€” confirming the intent is to restrict both operations: [5](#0-4) 

Yet the implementation only enforces this for initial deployments.

---

### Impact Explanation

An account that is **not** in the `ContractDeploymentAuthorizedAddressesPath` list but has a contract already deployed (e.g., deployed before restrictions were enabled, or after being removed from the authorized list) can submit a transaction calling `account.contracts.update()` on their own account. The FVM will accept the update without any authorization check. The attacker can replace the contract with malicious code â€” for example, a token contract that redirects withdrawals, a DeFi contract that drains a vault, or any contract that users trust and interact with. This is a direct unauthorized mutation of on-chain state that bypasses the network's contract governance controls.

---

### Likelihood Explanation

The attack is reachable by any ordinary transaction sender who owns an account with a pre-existing contract and is not currently in the authorized deployment list. Two realistic scenarios trigger this:

1. **Restriction enabled after deployment**: The network starts unrestricted, accounts deploy contracts freely, then the service account enables restricted deployment. All previously deployed accounts that are not added to the authorized list can still update their contracts.
2. **Account removed from authorized list**: An account is removed from the authorized list (e.g., due to a governance decision) but its deployed contracts remain live and updatable.

Neither scenario requires compromised keys, staked node control, or social engineering. The attacker only needs to own the account and submit a standard transaction.

---

### Recommendation

Remove the `!exists` guard from the authorization check in `SetContract`. The check should apply to both new deployments and updates when restricted deployment is enabled:

```go
// Before (vulnerable):
if !exists && !updater.isAuthorizedForDeployment(signingAccounts) {
    return fmt.Errorf("deploying contract failed: ...")
}

// After (fixed):
if !updater.isAuthorizedForDeployment(signingAccounts) {
    return fmt.Errorf("deploying or updating contract failed: ...")
}
```

This aligns the implementation with the stated intent in `SetContractDeploymentAuthorizersTransaction`'s docstring ("deploy/update contracts") and closes the gap between new deployments and updates.

---

### Proof of Concept

1. Bootstrap a Flow network with `RestrictContractDeployment: true` (the default).
2. Account `0xALICE` is in the authorized list and deploys contract `Vault`.
3. The service account removes `0xALICE` from `ContractDeploymentAuthorizedAddressesPath`.
4. `0xALICE` submits the following transaction:
   ```cadence
   transaction {
     prepare(signer: auth(UpdateContract) &Account) {
       signer.contracts.update(name: "Vault", code: "<malicious code>".decodeHex())
     }
   }
   ```
5. The Cadence runtime calls `UpdateAccountContractCode` â†’ `SetContract` with `signingAccounts = [0xALICE]`.
6. `SetContract` finds the contract exists (`exists = true`), skips the `isAuthorizedForDeployment` check, and queues the update.
7. At transaction commit, `Commit()` calls `accounts.SetContract(...)`, persisting the malicious code.
8. All users who subsequently interact with `Vault` are exposed to the attacker-controlled logic. [6](#0-5) [7](#0-6)

func (updater *ContractUpdaterImpl) SetContract(
	location common.AddressLocation,
	code []byte,
	signingAccounts []flow.Address,
) error {
	// Initial contract deployments must be authorized by signing accounts.
	//
	// Contract updates are always allowed.
	exists, err := updater.accounts.ContractExists(location.Name, flow.Address(location.Address))
	if err != nil {
		return err
	}

	if !exists && !updater.isAuthorizedForDeployment(signingAccounts) {
		return fmt.Errorf(
			"deploying contract failed: %w",
			errors.NewOperationAuthorizationErrorf(
				"SetContract",
				"deploying contracts requires authorization from specific "+
					"accounts"))

	}

	updater.draftUpdates[location] = ContractUpdate{
		Location: location,
		Code:     code,
	}

	return nil
}

func (updater *ContractUpdaterImpl) Commit() (ContractUpdates, error) {
	updateList := updater.updates()
	updater.Reset()

	contractUpdates := ContractUpdates{
		Updates:   make([]common.AddressLocation, 0, len(updateList)),
		Deploys:   make([]common.AddressLocation, 0, len(updateList)),
		Deletions: make([]common.AddressLocation, 0, len(updateList)),
	}

	var err error
	for _, v := range updateList {
		var currentlyExists bool
		currentlyExists, err = updater.accounts.ContractExists(v.Location.Name, flow.Address(v.Location.Address))
		if err != nil {
			return ContractUpdates{}, err
		}
		shouldDelete := len(v.Code) == 0

		if shouldDelete {
			// this is a removal
			contractUpdates.Deletions = append(contractUpdates.Deletions, v.Location)
			err = updater.accounts.DeleteContract(v.Location.Name, flow.Address(v.Location.Address))
			if err != nil {
				return ContractUpdates{}, err
			}
		} else {
			if !currentlyExists {
				// this is a deployment
				contractUpdates.Deploys = append(contractUpdates.Deploys, v.Location)
			} else {
				// this is an update
				contractUpdates.Updates = append(contractUpdates.Updates, v.Location)
			}

			err = updater.accounts.SetContract(
				v.Location.Name,
				flow.Address(v.Location.Address),
				v.Code,
			)
			if err != nil {
				return ContractUpdates{}, err
			}
		}
	}

	return contractUpdates, nil
