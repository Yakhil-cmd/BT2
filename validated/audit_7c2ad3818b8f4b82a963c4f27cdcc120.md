### Title
`native_per_pubdata` Integer-Division Truncation Silently Zeros Pubdata Fee When `pubdata_price < native_price` - (File: `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

The `native_per_pubdata` ratio used to charge native resources for L2 pubdata is computed with a plain integer floor-division (`wrapping_div`). When the operator-supplied `pubdata_price` is smaller than `native_price`, the result truncates to zero. Every downstream charging function then multiplies by zero, making all pubdata completely free regardless of how much state-diff data the transaction produces. An unprivileged sender can exploit this to generate maximum pubdata at zero native cost, forcing the operator to absorb L1 publication costs without compensation.

---

### Finding Description

In `validate_and_compute_fee_for_transaction` (ZK transaction path), the per-pubdata-byte native cost is derived as:

```rust
// basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs  line 142
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

`wrapping_div` is Rust's standard integer floor-division. When `pubdata_price < native_price` (e.g. `pubdata_price = 999`, `native_price = 1000`), the result is `0`.

The same truncation is replicated in the off-chain helper:

```rust
// api/src/helpers.rs  line 427
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
```

The truncated `native_per_pubdata = 0` propagates through every pubdata-charging site:

**Intrinsic pubdata overhead** (`gas_helpers.rs` line 352):
```rust
let intrinsic_pubdata_overhead = native_per_pubdata_byte.saturating_mul(intrinsic_pubdata);
// → 0 * intrinsic_pubdata = 0
```

**Execution pubdata charge** (`gas_helpers.rs` line 430–432):
```rust
let native = current_pubdata_spent
    .checked_mul(native_per_pubdata)   // → 0
    .ok_or(out_of_native_resources!())?;
```

**Sufficiency check** (`gas_helpers.rs` line 454):
```rust
let enough = resources.has_enough(&resources_for_pubdata);
// resources_for_pubdata = 0 → always enough
```

Contrast with `native_per_gas`, which uses ceiling division (`div_ceil`) to avoid under-charging:

```rust
// validation_impl.rs  line 135
u256_try_to_u64(&gas_price.div_ceil(native_price))
```

No equivalent ceiling (or even rounding) is applied to `native_per_pubdata`.

---

### Impact Explanation

**Resource accounting bug / operator-fund-loss path.**

When `pubdata_price` is set to any value strictly less than `native_price`, the entire pubdata fee is silently dropped to zero. A concrete example:

| Parameter | Value |
|---|---|
| `native_price` | 1 000 |
| `pubdata_price` | 999 |
| Intended `native_per_pubdata` | ≈ 1 (0.999 rounded) |
| Actual `native_per_pubdata` | **0** |
| Pubdata generated by attacker | 100 000 bytes |
| Expected native charge for pubdata | ~99 900 |
| Actual native charge for pubdata | **0** |

The operator bears the full L1 data-publication cost (ETH spent on calldata / blob fees) with zero on-chain compensation. Because `check_enough_resources_for_pubdata` always returns `enough = true` when `native_per_pubdata = 0`, the transaction is never reverted for insufficient pubdata payment, and the state diff is committed to L1 regardless.

The deviation is not a few basis points (as in the USDC example of the external report) — it is a **100% undercharge** for any `pubdata_price` in the range `[1, native_price - 1]`.

---

### Likelihood Explanation

The condition `pubdata_price < native_price` is a realistic operational configuration. The default test fixture already demonstrates it:

```rust
// zk_ee/src/system/metadata/zk_metadata.rs  line 209
pubdata_price: U256::from(0u64),
native_price: U256::from(10),
```

And the integration test `test_check_pubdata_has_timestamp` uses:

```rust
let native_price = U256::from(100);
let pubdata_price = U256::from(2);
```

Here `native_per_pubdata = 2 / 100 = 0`, so pubdata is already free in that test. Any block epoch where the operator legitimately sets a low pubdata price (e.g. during periods of cheap L1 calldata) exposes every transaction in that block to free pubdata. An attacker monitoring the mempool for such blocks can submit storage-heavy transactions to maximise free pubdata generation.

---

### Recommendation

1. **Use ceiling division for `native_per_pubdata`**, consistent with `native_per_gas`:
   ```rust
   let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
       .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
   ```
2. **Apply the same fix in `api/src/helpers.rs`** (`validate_l2_tx_intrinsic_native_resources`) so the off-chain pre-validation mirrors the on-chain behaviour.
3. Consider storing the numerator and denominator separately (as the external report recommends) to avoid precision loss entirely when the ratio is sub-unit.

### Proof of Concept

**Step 1** – `native_per_pubdata` is computed:
```
999.wrapping_div(1000) = 0
``` [1](#0-0) 

**Step 2** – Resources are created with `native_per_pubdata_byte = 0`; intrinsic pubdata overhead is zero:
```
0 * intrinsic_pubdata = 0  →  native_limit unchanged
``` [2](#0-1) 

**Step 3** – Attacker's transaction writes 20 storage slots (generating ~640 bytes of pubdata). Post-execution pubdata charge:
```
640 * 0 = 0 native  →  check_enough_resources_for_pubdata returns true
``` [3](#0-2) 

**Step 4** – `compute_gas_refund` sees `native_used ≈ 0` for pubdata, `delta_gas = 0`; the attacker receives a full gas refund for the pubdata portion and pays only EVM gas for computation. [4](#0-3)
