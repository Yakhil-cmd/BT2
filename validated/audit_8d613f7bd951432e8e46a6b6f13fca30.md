### Title
Key Revocation Bypass via Duplicate Key Deduplication — (`fvm/environment/account-key-metadata/key_index_mapping_group.go`, `fvm/environment/accounts_status.go`, `fvm/environment/account_public_key_util.go`)

### Summary
When a duplicate key is added to an account, the deduplication system stores an independent revoked status per key index in `weightAndRevokedStatusBytes`, while sharing the underlying stored key bytes. Revoking keyIndex=0 only marks the stored key bytes as revoked; it does not propagate to the independent revoked-status entry for any other key index that maps to the same `storedKeyIndex`. An attacker who has previously added a duplicate key retains the ability to sign transactions using the revoked private key via the surviving duplicate key index.

### Finding Description
The account key metadata system stores revoked status per key index independently in `weightAndRevokedStatusBytes`. When `RevokeAccountPublicKey(addr, 0)` is called, it modifies only the `AccountPublicKey0RegisterID` register (the stored key bytes). The key metadata's `weightAndRevokedStatusBytes` for any other key index (e.g., keyIndex=1) that maps to `storedKeyIndex=0` via the deduplication mapping is never updated.

`GetRuntimeAccountPublicKey(addr, 1)` returns `Revoked=false` (from kB's independent metadata) and `PublicKey=kA's public key` (from `storedKeyIndex=0`). The transaction verifier's revocation check at line 305 of `transactionVerifier.go` passes, and the cryptographic signature verification against kA's public key succeeds for any signature produced with kA's private key. [12](#0-11) [13](#0-12) 

### Impact Explanation
After `RevokeKey(kA_index=0)`, an attacker holding kA's private key can submit transactions signed with keyIndex=1 (kB) and kA's private key. Both the sequence number checker and the transaction verifier will accept these transactions. The account owner's revocation of kA does not prevent the attacker from continuing to authorize transactions, drain assets, or modify account state.

### Likelihood Explanation
The attacker must have had prior access to the account (to submit the `AddKey(kB_duplicate)` transaction). This is realistic in:
- Compromised-key scenarios: attacker learns kA's private key, adds kB, account owner revokes kA.
- Multi-sig setups where a co-signer adds a duplicate before being removed.

The attacker needs no special node access, no admin privileges, and no brute force. The exploit is submittable as a standard transaction.

### Recommendation
When revoking a key at any index, the runtime must also mark as revoked all other key indices that share the same `storedKeyIndex` via the deduplication mapping. Alternatively, `GetRuntimeAccountPublicKey` for a deduplicated key should check the revoked status of the underlying `storedKeyIndex=0` key (from `AccountPublicKey0RegisterID`) in addition to the key index's own metadata entry, and return `Revoked=true` if either is revoked.

// GetKeyMetadata returns weight, revoked status, and stored key index for given account public key index.
func GetKeyMetadata(b []byte, keyIndex uint32, deduplicated bool) (
	weight uint16,
	revoked bool,
	storedKeyIndex uint32,
	err error,
) {
	// Key metadata only stores weight and revoked status for keys at index > 0.

	if keyIndex == 0 {
		err = errors.NewKeyMetadataUnexpectedKeyIndexError("failed to query key metadata", 0)
		return
	}

	// Get raw weight and revoked status bytes
	weightAndRevokedStatusBytes, rest, err := parseWeightAndRevokedStatusFromKeyMetadataBytes(b)
	if err != nil {
		return 0, false, 0, err
	}

	// Get weight and revoked status for given account key index
	revoked, weight, err = getWeightAndRevokedStatus(weightAndRevokedStatusBytes, keyIndex-1)
	if err != nil {
		return 0, false, 0, err
	}

	// If keys are not deduplicated, storedKeyIndex is the same as the given keyIndex.
	if !deduplicated {
		return weight, revoked, keyIndex, nil
	}

	// Get raw key index mapping bytes
	startIndexForMapping, mappingBytes, _, err := parseStoredKeyMappingFromKeyMetadataBytes(rest)
	if err != nil {
		return 0, false, 0, err
	}

	// StoredKeyIndex is the same as the given keyIndex if deduplication happens afterwards.
	if keyIndex < startIndexForMapping {
		return weight, revoked, keyIndex, nil
	}

	// Get stored key index from mapping
	storedKeyIndex, err = getStoredKeyIndexFromMappings(mappingBytes, keyIndex-startIndexForMapping)
	if err != nil {
		return 0, false, 0, err
	}

	return weight, revoked, storedKeyIndex, nil
}
