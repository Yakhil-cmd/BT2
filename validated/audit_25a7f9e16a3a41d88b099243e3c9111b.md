Audit Report

## Title
Silent Integer Truncation of `length` Parameter in ICRC-1 Ledger `get_blocks` / `get_transactions` on wasm32 - (File: `rs/ledger_suite/icrc1/ledger/src/main.rs`)

## Summary
Both `get_blocks` and `get_transactions` query endpoints in the ICRC-1 ledger cast the caller-supplied `length` value from `u64` to `usize` with an unchecked `as` cast. On wasm32 (the only IC canister target), `usize` is 32 bits, so any `length` value that is a nonzero multiple of `2^32` — most critically `(u32::MAX as u64) + 1 = 4_294_967_296` — silently truncates to `0`. The downstream `block_locations` call then returns an empty local range and empty archive pointers, causing the endpoint to return zero blocks with no error, silently hiding the entire ledger history from the caller.

## Finding Description
In `rs/ledger_suite/icrc1/ledger/src/main.rs` lines 793–807, both endpoints call `req.as_start_and_length()` and immediately cast the returned `u64` length to `usize` without any guard:

```rust
// line 797
Access::with_ledger(|ledger| ledger.get_transactions(start, length as usize))
// line 806
Access::with_ledger(|ledger| ledger.get_blocks(start, length as usize))
```

`as_start_and_length()` in `packages/icrc-ledger-types/src/icrc3/blocks.rs` lines 64–83 only caps `length` to `u64::MAX` via `to_u64()`, performing no `usize::MAX` guard. On wasm32, `u64 as usize` is equivalent to `u64 as u32`; the upper 32 bits are discarded. For `length = 4_294_967_296`, the result is `0`, and `block_locations(self, start, 0)` returns an empty range and empty archive callbacks — a structurally valid but semantically empty response.

By contrast, the ICP ledger at `rs/ledger_suite/icp/ledger/src/main.rs` line 1065 explicitly guards:
```rust
let locations = block_locations(&*ledger, start, length.min(usize::MAX as u64) as usize);
```
and has a regression test at `rs/ledger_suite/icp/ledger/tests/tests.rs` lines 1884–1886 with the comment: *"If this is cast (in a wasm32 ledger) using `as usize`, it will overflow to 0u32."*

The `icrc3_get_blocks` path in `rs/ledger_suite/icrc1/ledger/src/lib.rs` line 1120 is also correctly guarded with `max_length.min(length).min(usize::MAX as u64) as usize`, confirming the `get_blocks`/`get_transactions` paths were not updated when the fix was applied elsewhere.

## Impact Explanation
This is a High-severity finding. The ICRC-1 ledger is deployed on mainnet for ckBTC, ckETH, ckERC20, and SNS tokens — all in-scope financial integrations. Any caller (including anonymous query callers) can cause the ledger to return a structurally valid but empty block response, silently hiding all ledger history. Downstream indexers, chain-fusion bridges, and wallets that use large batch `length` values will observe a complete absence of blocks, leading to missed ICRC-1 transfers, incorrect balance reconstruction, and denial of historical data availability. No trap or error is raised, making the bug hard to detect. This matches the allowed High impact: *"Significant Chain Fusion, ck-token, ledger... security impact with concrete user or protocol harm."*

## Likelihood Explanation
The `length` field is a `Nat` (arbitrary-precision integer) in the Candid interface, so any unprivileged caller — including anonymous query callers — can supply `length = (u32::MAX as u64) + 1` with zero cost and no special privilege. The attack is trivially repeatable and requires no victim mistake. The ICP ledger's own regression test demonstrates the exact truncation scenario, confirming the platform team is aware of the wasm32 hazard. The ICRC-1 ledger simply lacks the equivalent fix.

## Recommendation
Apply the same guard used in the ICP ledger and in `icrc3_get_blocks` to both affected functions in `rs/ledger_suite/icrc1/ledger/src/main.rs`:

```rust
fn get_transactions(req: GetTransactionsRequest) -> GetTransactionsResponse {
    let (start, length) = req
        .as_start_and_length()
        .unwrap_or_else(|msg| ic_cdk::api::trap(&msg));
    Access::with_ledger(|ledger| {
        ledger.get_transactions(start, length.min(usize::MAX as u64) as usize)
    })
}

fn get_blocks(req: GetBlocksRequest) -> GetBlocksResponse {
    let (start, length) = req
        .as_start_and_length()
        .unwrap_or_else(|msg| ic_cdk::api::trap(&msg));
    Access::with_ledger(|ledger| {
        ledger.get_blocks(start, length.min(usize::MAX as u64) as usize)
    })
}
```

Alternatively, move the `usize::MAX` cap into `as_start_and_length()` itself so all callers benefit automatically.

## Proof of Concept
Mirror the ICP ledger's existing regression test (`test_query_blocks_large_length` in `rs/ledger_suite/icp/ledger/tests/tests.rs`):

1. Install the ICRC-1 ledger canister in a `StateMachine` or PocketIC environment.
2. Seed at least one transaction so the ledger contains blocks.
3. Issue a `get_blocks` query with `start = Nat::from(0u64)` and `length = Nat::from((u32::MAX as u64) + 1)`.
4. On the unfixed wasm32 canister, `length as usize` evaluates to `0`, `block_locations` returns an empty range, and `res.blocks` is empty despite the ledger containing blocks.
5. Assert `!res.blocks.is_empty()` — this assertion fails on the unfixed canister, confirming the truncation.
