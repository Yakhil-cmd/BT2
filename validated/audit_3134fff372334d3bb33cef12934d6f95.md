### Title
Malicious Participant Can Send Fake Presign Commitment to Permanently Deny Signing With No Culprit Identification — (`src/frost/mod.rs`, `src/frost/redjubjub/sign.rs`, `src/frost/eddsa/sign.rs`)

---

### Summary

A malicious participant can broadcast a `SigningCommitment` during `do_presign` that does not correspond to their actual nonces. Because no proof-of-knowledge of the nonces is required, and because cheater detection is explicitly disabled at aggregation time, the resulting `aggregate()` call will always fail with `ErrorFrostAggregation` and there is no mechanism to identify the offending participant.

---

### Finding Description

**Step 1 — Presign: no commitment binding to nonces**

In `do_presign`, each participant generates their own nonces and commitment, then broadcasts the commitment to all others:

```rust
let (nonces, commitments) = commit(&signing_share, &mut rng);
commitments_map.insert(me.to_identifier()?, commitments);
chan.send_many(commit_waitpoint, &commitments)?;
```

Received commitments from other participants are inserted into the map with zero validation:

```rust
for (from, commitment) in recv_from_others(&chan, commit_waitpoint, &participants, me).await? {
    commitments_map.insert(from.to_identifier()?, commitment);
}
``` [1](#0-0) 

There is no zero-knowledge proof of knowledge of the nonces, no binding to the participant's signing share, and no cross-check against any public commitment. A malicious participant can generate `(nonces_real, commitment_real)` for themselves but broadcast `commitment_fake` (from arbitrary nonces) to all other participants.

**Step 2 — Divergent signing packages**

After presign, each participant holds a `PresignOutput` with their own `nonces` and a `commitments_map`. The malicious participant's own map contains `commitment_real` for themselves (inserted before broadcasting), while every other participant's map contains `commitment_fake` for the malicious participant.

During sign, each participant constructs a `SigningPackage` from their own `presignature.commitments_map`:

- Coordinator's `signing_package` → contains `commitment_fake` for the malicious participant
- Malicious participant's `signing_package` → contains `commitment_real` for themselves

The malicious participant computes `round2::sign` against a signing package with `commitment_real`, producing a share that is cryptographically inconsistent with the coordinator's signing package (which has `commitment_fake`). [2](#0-1) [3](#0-2) 

**Step 3 — Cheater detection explicitly disabled**

The coordinator calls `aggregate()` with an empty `BTreeMap` for verifying shares:

```rust
// We use empty BTreeMap because "cheater-detection" feature is disabled
let pk_package = PublicKeyPackage::new(BTreeMap::new(), keygen_output.public_key);
let signature = aggregate(&signing_package, &signature_shares, &pk_package, &randomized_params)
    .map_err(|_| ProtocolError::ErrorFrostAggregation)?;
``` [4](#0-3) 

The same pattern is present in the EdDSA path: [5](#0-4) [6](#0-5) 

With an empty verifying-shares map, `aggregate()` cannot verify individual shares against per-participant public keys. The aggregated signature will fail the final group-key verification check, returning an error. There is no way to determine which participant's share was invalid.

---

### Impact Explanation

- `aggregate()` returns `Err(ErrorFrostAggregation)` every time the malicious participant is included in the signing set.
- Because cheater detection is disabled, the coordinator receives only a generic aggregation error with no indication of which participant is responsible.
- Honest parties cannot exclude the malicious participant without external out-of-band attribution.
- The malicious participant can repeat this attack on every signing attempt, permanently blocking signing for any session that includes them.
- This matches **High: Permanent denial of signing for honest parties under valid protocol inputs and documented trust assumptions**.

---

### Likelihood Explanation

Any participant with a valid keygen output can execute this attack by simply sending a `SigningCommitment` derived from different nonces during presign. No cryptographic capability beyond participation in the protocol is required. The attack is trivially repeatable and requires no coordination.

---

### Recommendation

1. **Require a proof of knowledge of nonces**: During presign, each participant should accompany their `SigningCommitment` with a Schnorr proof-of-knowledge (or equivalent) of the discrete logarithms of the hiding and binding nonce commitments. This prevents sending commitments for nonces the participant does not hold.

2. **Enable cheater detection**: Populate the `PublicKeyPackage` with the actual per-participant verifying shares (derived from `keygen_output`). This allows `aggregate()` to identify and report which participant's share is invalid, enabling honest parties to exclude the culprit and retry.

3. **Alternatively, use a consistent signing package**: In the redjubjub presign+sign split, the coordinator should broadcast the canonical `commitments_map` to all participants before `round2::sign`, so all parties compute their share against the same signing package. This does not prevent a malicious commitment but ensures the signing package is consistent, shifting the failure to a detectable mismatch.

---

### Proof of Concept

```rust
// Setup: 3-of-3 keygen
let (p1, keygen1) = ...; // honest
let (p2, keygen2) = ...; // honest
let (p3, keygen3) = ...; // MALICIOUS

// Presign: p3 generates real nonces but broadcasts fake commitment
let (nonces_real, _commitment_real) = commit(&keygen3.private_share, &mut rng);
let (_nonces_fake, commitment_fake) = commit(&different_signing_share, &mut rng);
// p3 sends commitment_fake to p1 and p2 during do_presign

// After presign:
// p1.presign.commitments_map[p3] = commitment_fake
// p3.presign.commitments_map[p3] = commitment_real  (inserted before broadcast)
// p3.presign.nonces = nonces_real

// Sign: coordinator (p1) builds signing_package with commitment_fake for p3
// p3 computes round2::sign with commitment_real in its own signing_package
// p3's share is inconsistent with coordinator's signing_package

// aggregate() fails:
// assert!(result == Err(ProtocolError::ErrorFrostAggregation));
// No participant is identified as the culprit.
```
