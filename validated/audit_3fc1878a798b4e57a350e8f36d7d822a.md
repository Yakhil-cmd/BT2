### Title
Permanent ERC-20 Token Loss When `ft_transfer` Fails in `ExitToNear` Precompile Without `error_refund` Feature — (File: `engine-precompiles/src/native.rs`)

---

### Summary

The `ExitToNear` precompile burns ERC-20 tokens in the Aurora EVM and then issues a NEAR promise to call `ft_transfer` on the corresponding NEP-141 contract. When the `error_refund` compile-time feature is not enabled, no callback is attached to handle `ft_transfer` failures. If `ft_transfer` fails (e.g., the recipient NEAR account is not registered with the NEP-141 token), the burned ERC-20 tokens are permanently unrecoverable — an exact structural analog to the XVS bridge locked-funds vulnerability.

---

### Finding Description

In `engine-precompiles/src/native.rs`, the `ExitToNear::run()` function constructs a NEAR promise after burning ERC-20 tokens. The critical branching logic is:

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // NO callback attached
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs { ... })  // callback attached
};
``` [1](#0-0) 

`callback_args` equals `ExitToNearPrecompileCallbackArgs::default()` (i.e., `{refund: None, transfer_near: None}`) whenever:

1. The `error_refund` feature is **not** compiled in — making `refund` unconditionally `None`:

```rust
#[cfg(feature = "error_refund")]
refund: refund_call_args(&exit_to_near_params, &exit_event),
#[cfg(not(feature = "error_refund"))]
refund: None,
``` [2](#0-1) 

2. The exit is a regular ERC-20 `ft_transfer` (not wNEAR unwrap, not Omni), making `transfer_near` also `None`:

```rust
_ => {
    (nep141_account_id,
     format!(r#"{{"receiver_id":"{}","amount":"{}"}}"#, ...),
     "ft_transfer",
     None,   // <-- transfer_near_args is None
     ...)
}
``` [3](#0-2) 

When both fields are `None`, the promise is created **without any callback**. The ERC-20 tokens are already burned at the EVM level before the NEAR promise executes. If `ft_transfer` then fails on NEAR (e.g., recipient not registered with the NEP-141 token), there is no `exit_to_near_precompile_callback` invocation to re-mint the burned tokens.

The `exit_to_near_precompile_callback` function — which is the only place where `refund_on_error` is called to re-mint burned ERC-20 tokens — is never scheduled in this code path:

```rust
pub fn exit_to_near_precompile_callback<...>(...) {
    ...
    } else if let Some(args) = args.refund {
        // Exit call failed; need to refund tokens
        let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;
        ...
    }
``` [4](#0-3) 

The `refund_on_error` function re-mints ERC-20 tokens to the user's address, but it is only reachable through the callback: [5](#0-4) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

When `ft_transfer` fails:
- The user's ERC-20 tokens are **permanently burned** in the Aurora EVM (gone from the user's balance).
- The corresponding NEP-141 tokens remain in Aurora's NEP-141 account balance, inaccessible to the user and creating a bridge accounting discrepancy (Aurora holds more NEP-141 than there are ERC-20 tokens in circulation).
- There is no retry mechanism, no admin recovery function, and no way for the user to reclaim their funds.

This is confirmed explicitly by the production test suite, which documents the permanent loss:

```rust
#[cfg(feature = "error_refund")]
let balance = FT_TRANSFER_AMOUNT.into();
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
``` [6](#0-5) 

---

### Likelihood Explanation

**Medium.**

`ft_transfer` on a NEP-141 contract fails when the recipient account is not registered (has not called `storage_deposit`). This is a common real-world condition:
- New NEAR accounts that have never interacted with a specific NEP-141 token are unregistered by default.
- The Aurora EVM has no mechanism to validate NEAR-side registration before burning ERC-20 tokens.
- A user who specifies an unregistered recipient (including a typo or a fresh account) will permanently lose their tokens.
- The `error_refund` feature is a compile-time flag, not a runtime default, meaning deployments compiled without it are fully exposed.

The attacker-controlled entry path is: any EVM user calls the ERC-20 contract's burn/withdraw function, which internally calls the `ExitToNear` precompile at address `0xe9217bc70b7ed1f598ddd3199e80b093fa71124f`, specifying a NEAR recipient that is not registered with the NEP-141 token. [7](#0-6) 

---

### Recommendation

1. **Enable `error_refund` unconditionally** in production builds, or promote it from a feature flag to always-on behavior. The refund path is the correct behavior and should not be optional.
2. Alternatively, **always attach the callback** regardless of whether `refund` args are present, so that a failed `ft_transfer` can at minimum be detected and logged, with a recovery path available.
3. Consider adding a pre-flight check (e.g., a view call to `storage_balance_of` on the NEP-141 contract) before burning ERC-20 tokens, though this adds complexity to the promise chain.

---

### Proof of Concept

1. User holds ERC-20 tokens on Aurora corresponding to NEP-141 token `token.near`.
2. User calls the ERC-20 contract's `withdrawToNear` (or equivalent burn function), specifying recipient `fresh-account.near` which has never called `storage_deposit` on `token.near`.
3. The ERC-20 contract calls the `ExitToNear` precompile. The precompile burns the ERC-20 tokens from the user's EVM balance.
4. The precompile constructs `PromiseArgs::Create(transfer_promise)` — **no callback** — because `error_refund` is not enabled and the exit is a plain `ft_transfer`. [8](#0-7) 

5. The NEAR runtime executes `ft_transfer` on `token.near`. It fails with `"The account fresh-account.near is not registered"`.
6. Because no callback was attached, `exit_to_near_precompile_callback` is never called, `refund_on_error` is never invoked, and the burned ERC-20 tokens are never re-minted.
7. The user's ERC-20 tokens are permanently gone. The NEP-141 tokens remain in Aurora's account with no user-accessible recovery path.
