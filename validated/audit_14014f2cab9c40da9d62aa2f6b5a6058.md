### Title
Missing Failure-Callback in `ExitToEthereum` Precompile Causes Permanent ERC-20 Token Destruction on NEAR-Side Withdraw Failure — (`engine-precompiles/src/native.rs`)

---

### Summary

`ExitToEthereum` schedules the NEAR-side `withdraw` promise as a bare `PromiseArgs::Create` with no callback. If the eth-connector's `withdraw` call fails, the ERC-20 tokens that were already burned on the EVM side are permanently destroyed with no refund path. The `ExitToNear` precompile has an explicit `error_refund` feature that attaches a `PromiseArgs::Callback` to re-mint tokens on failure; `ExitToEthereum` has no equivalent mechanism.

---

### Finding Description

**Step 1 — EVM burn is unconditional and committed before the promise.**

`EvmErc20.withdrawToEthereum` burns the caller's tokens first, then calls the precompile:

```solidity
// etc/eth-contracts/contracts/EvmErc20.sol lines 65-76
function withdrawToEthereum(address recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // ← tokens destroyed here
    ...
    call(gas(), 0xb0bd02f6..., ...)       // ← precompile invoked after burn
}
``` [1](#0-0) 

The EVM state (burn) is committed as part of the Aurora transaction. The NEAR promise is only *scheduled* after EVM execution completes; a subsequent promise failure cannot revert the EVM state.

**Step 2 — `ExitToEthereum` always emits `PromiseArgs::Create` — no callback, ever.**

```rust
// engine-precompiles/src/native.rs line 985
let promise = borsh::to_vec(&PromiseArgs::Create(withdraw_promise)).unwrap();
``` [2](#0-1) 

There is no conditional or feature-gated path that attaches a callback. The entire `ExitToEthereum` implementation contains zero references to `error_refund`, `RefundCallArgs`, or `PromiseArgs::Callback`.

**Step 3 — `ExitToNear` has the exact refund mechanism that `ExitToEthereum` lacks.**

`ExitToNear` conditionally wraps the promise in a `PromiseArgs::Callback` pointing to `exit_to_near_precompile_callback` on the engine, which re-mints tokens if the NEAR-side call fails:

```rust
// engine-precompiles/src/native.rs lines 470-483
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs {
        base: transfer_promise,
        callback: PromiseCreateArgs {
            target_account_id: self.current_account_id.clone(),
            method: "exit_to_near_precompile_callback".to_string(),
            ...
        },
    })
};
``` [3](#0-2) 

The `error_refund` feature populates `RefundCallArgs` with the ERC-20 address and amount so the callback can re-mint: [4](#0-3) 

`ExitToEthereum` has no analogous structure.

**Step 4 — `filter_promises_from_logs` faithfully dispatches whatever promise type the precompile emits.**

```rust
// engine/src/engine.rs lines 1653-1664
PromiseArgs::Create(promise) => {
    let id = match previous_promise {
        Some(base_id) => schedule_promise_callback(handler, base_id, &promise),
        None => schedule_promise(handler, &promise),
    };
    previous_promise = Some(id);
}
``` [5](#0-4) 

Because `ExitToEthereum` always emits `PromiseArgs::Create`, no callback is ever attached at the dispatch layer either.

---

### Impact Explanation

If the eth-connector's `withdraw` call fails (panic, paused state, insufficient NEP-141 balance, gas exhaustion, or any other NEAR-level error):

- The ERC-20 tokens are already burned — supply is permanently reduced.
- The eth-connector never issues a burn receipt to the Ethereum light client, so the ETH backing on Ethereum is never released.
- The user loses both their ERC-20 tokens and the corresponding ETH — net asset destruction.
- This is a direct insolvency condition: ERC-20 supply decreases without a corresponding decrease in ETH locked on Ethereum.

---

### Likelihood Explanation

Any unprivileged EVM user can call `withdrawToEthereum`. The trigger condition — the eth-connector's `withdraw` panicking — can arise from:

- The eth-connector contract being paused (a supported operational state).
- A bug in the eth-connector's `withdraw` logic.
- The NEP-141 account being deleted or misconfigured.
- Insufficient NEAR gas propagated to the promise (though 100 TGas is allocated, edge cases exist).

The developers explicitly recognized this failure mode for `ExitToNear` and added the `error_refund` feature. The omission for `ExitToEthereum` is a confirmed asymmetry in the codebase.

---

### Recommendation

Mirror the `error_refund` pattern from `ExitToNear` in `ExitToEthereum`:

1. Accept a `refund_address` (the EVM caller) in the precompile input.
2. Build a `PromiseArgs::Callback` where the callback calls a new engine method (e.g., `exit_to_ethereum_precompile_callback`) on the engine account.
3. In that callback, check `promise_result_check()`; if the `withdraw` failed, re-mint the ERC-20 tokens to `refund_address`.
