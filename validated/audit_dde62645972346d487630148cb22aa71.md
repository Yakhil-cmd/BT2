### Title
Presignature Replay Enables Secret Key Extraction via Nonce Reuse — (`src/ecdsa/robust_ecdsa/mod.rs`, `src/ecdsa/ot_based_ecdsa/mod.rs`)

---

### Summary

Both `PresignOutput` types (robust ECDSA and OT-based ECDSA) derive `Clone` and `Serialize`/`Deserialize`, and `RerandomizedPresignOutput::rerandomize_presign` accepts the original presignature by shared reference (`&PresignOutput`). The library provides no mechanism to enforce single-use semantics on a presignature. A malicious coordinator can therefore orchestrate two signing sessions over different messages using the same presignature material, obtaining two signature shares with multiplicatively related nonces. This is sufficient to recover the aggregate secret key via standard ECDSA nonce-reuse algebra — an impact the library's own security documentation explicitly identifies as catastrophic.

---

### Finding Description

`PresignOutput` in both ECDSA variants is declared with `#[derive(Clone, Serialize, Deserialize, ...)]`:

- Robust ECDSA: [1](#0-0) 
- OT-based ECDSA: [2](#0-1) 

`RerandomizedPresignOutput::rerandomize_presign` takes the original presignature by shared reference in both variants:

- Robust ECDSA: [3](#0-2) 
- OT-based ECDSA: [4](#0-3) 

While `sign()` consumes the `RerandomizedPresignOutput` by value (preventing direct reuse of the rerandomized form), the underlying `PresignOutput` is never consumed. A caller can call `rerandomize_presign(&presign_output, args1)` and `rerandomize_presign(&presign_output, args2)` with two different `(msg_hash, tweak, entropy)` tuples, producing two distinct `RerandomizedPresignOutput` values from the same nonce material, and pass each to `sign()`.

The presign comment acknowledges the requirement but does not enforce it: [5](#0-4) 

The sign function carries the same advisory-only warning: [6](#0-5) 

No runtime guard, used-presignature registry, or type-system enforcement (e.g., consuming `PresignOutput` by value in `rerandomize_presign`) exists anywhere in the production code.

---

### Impact Explanation

The library's own security documentation states:

> "If different subsets of size at least 2t+1 sign different (h, ε) values using shares derived from the same presignature, the resulting signatures use multiplicatively related nonces and the secret key can be recovered using standard ECDSA nonce-reuse attacks."

> "a novel split-view attack exists that can extract the secret key using as few as 2t+2 presigning participants, with as few as two signing sessions." [7](#0-6) 

Two signatures `(R₁, s₁)` and `(R₂, s₂)` produced from the same presignature with different messages satisfy a linear relation in the secret key shares. Standard ECDSA nonce-reuse recovery directly yields the aggregate secret key. This maps to the **Critical** impact class: *extraction, reconstruction, or disclosure of private signing shares or aggregate secret material*.

---

### Likelihood Explanation

The attack requires only that a coordinator (a role explicitly modeled in the library) initiates two signing sessions referencing the same presignature. Because `PresignOutput` is `Clone` and `Serialize`, it is trivially storable and reusable. Honest participants have no way to detect that the presignature they are being asked to sign with has already been used. The coordinator is a semi-trusted role in the documented threat model, and the library provides no countermeasure at the API level.

---

### Recommendation

1. **Consume `PresignOutput` by value in `rerandomize_presign`** — change the signature from `rerandomize_presign(presignature: &PresignOutput, ...)` to `rerandomize_presign(presignature: PresignOutput, ...)`. This makes the type system enforce single-use at the call site.
2. **Remove `Clone` from `PresignOutput`** — deriving `Clone` on a one-time-use secret structure directly contradicts the single-use requirement and enables silent reuse.
3. **Consider removing `Serialize`/`Deserialize` from `PresignOutput`**, or at minimum document that deserialization reintroduces a previously-consumed presignature and must be guarded by the caller with a persistent used-presignature registry.

---

### Proof of Concept

```rust
// Each participant runs presign once and obtains their PresignOutput.
let presign_out: PresignOutput = /* result of presign protocol */;

// Session 1: sign message hash h1
let args1 = RerandomizationArguments::new(pk, tweak1, h1, big_r, participants.clone(), entropy1);
let rerand1 = RerandomizedPresignOutput::rerandomize_presign(&presign_out, &args1).unwrap();
// rerand1 is consumed by sign(), but presign_out is still alive

// Session 2: sign a DIFFERENT message hash h2 — same presignature, no error
let args2 = RerandomizationArguments::new(pk, tweak2, h2, big_r, participants.clone(), entropy2);
let rerand2 = RerandomizedPresignOutput::rerandomize_presign(&presign_out, &args2).unwrap();
// presign_out reused; nonces are multiplicatively related across sessions
// A coordinator collecting (sig1, sig2) can now recover the secret key via nonce-reuse algebra
```

The `rerandomize_presign` call on line 2 succeeds without any error because `presign_out` was never consumed. The library's `N1 = N2 = 2t+1` enforcement and `msg_hash != 0` check do not prevent this scenario. [8](#0-7) [9](#0-8)

impl RerandomizedPresignOutput {
    pub fn rerandomize_presign(
        presignature: &PresignOutput,
        args: &RerandomizationArguments,
    ) -> Result<Self, ProtocolError> {
        if presignature.big_r != args.big_r {
            return Err(ProtocolError::IncompatibleRerandomizationInputs);
        }
        let delta = args.derive_randomness()?;
        if delta.is_zero().into() {
            return Err(ProtocolError::ZeroScalar);
        }

        // cannot be zero due to the previous check
        let inv_delta = delta.invert().unwrap();

        // delta * R
        let rerandomized_big_r = presignature.big_r * delta;

        // alpha * delta^{-1}
        let rerandomized_alpha = presignature.alpha * inv_delta;

        // (beta + c*tweak) * delta^{-1}
        let rerandomized_beta =
            (presignature.beta + presignature.c * args.tweak.value()) * inv_delta;

        Ok(Self {
            big_r: rerandomized_big_r.into(),
            alpha: rerandomized_alpha,
            beta: rerandomized_beta,
            e: presignature.e,
        })
    }
