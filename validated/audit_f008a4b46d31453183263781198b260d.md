### Title
Gateway Admits Unsigned Invoke Transactions by Exploiting `skip_stateful_validations` Deploy-Account UX Bypass — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator unconditionally skips the `__validate__` entry-point call (which performs signature verification) for any Invoke transaction with `nonce == 1` when `account_tx_in_pool_or_recent_block` returns `true`. Because that helper checks for **any** transaction from the sender address in the pool — not specifically a `deploy_account` — an unprivileged attacker who observes a legitimate pending `deploy_account` for address A can immediately submit a nonce-1 Invoke for address A carrying an arbitrary/fake signature. The gateway admits it to the mempool without ever verifying the signature. The batcher later executes it with `validate: true`, the `__validate__` call fails, and the transaction is included in the block as a reverted transaction, wasting block space.

---

### Finding Description

**Root cause — `skip_stateful_validations`:** [1](#0-0) 

The function returns `true` (skip validation) when all three conditions hold:

1. The transaction is an Invoke with `nonce == Nonce(Felt::ONE)`
2. The on-chain account nonce is `Nonce(Felt::ZERO)` (account not yet deployed)
3. `mempool_client.account_tx_in_pool_or_recent_block(sender_address)` returns `true`

The comment claims condition 3 proves a `deploy_account` is pending. But the underlying implementation checks for **any** transaction from the address: [2](#0-1) 

It checks `self.state.contains_account(address) || self.tx_pool.contains_account(address)` — no transaction-type filter.

**How the skip propagates to the entry-point call:** [3](#0-2) 

`validate: !skip_validate` — when `skip_validate` is `true`, the `__validate__` entry point is never called, so the signature is never verified at the gateway.

**Batcher re-validates with `validate: true`:** [4](#0-3) 

`AccountTransaction::new_for_sequencing` always sets `validate: true`. The fake invoke therefore fails during batcher execution and is included in the block as a reverted transaction.

---

### Impact Explanation

An unprivileged attacker can inject one invalid (fake-signature) Invoke transaction into the mempool for every address that has a pending `deploy_account`. The gateway's admission invariant — "only transactions with valid signatures are accepted" — is broken for this specific nonce-1 window. The reverted transaction consumes block space and batcher execution resources. Because the mempool is observable (P2P propagation, gateway monitoring), the attack requires no privileged access.

Matching impact category: **High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

---

### Likelihood Explanation

- The mempool is publicly observable via P2P propagation.
- Every new wallet deployment creates a `deploy_account` transaction that opens the window.
- The attacker only needs to craft a well-formed Invoke with `nonce=1` and an arbitrary signature field; no cryptographic work is required.
- The attack is repeatable for every new `deploy_account` observed.

---

### Recommendation

Replace the generic `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a **`deploy_account` transaction** is pending for the sender address. Alternatively, add a dedicated `deploy_account_in_pool(address)` query to the mempool that only returns `true` when the pending transaction is of type `DeployAccount`. This preserves the UX feature while closing the signature-bypass window.

---

### Proof of Concept

```
1. Legitimate user submits:
     deploy_account { sender: A, nonce: 0, valid_signature: S_owner }
   → Accepted by gateway (full validation), placed in mempool.

2. Attacker observes A in the mempool (P2P or gateway monitoring).

3. Attacker submits:
     invoke { sender: A, nonce: 1, signature: [0xdeadbeef] }  ← fake signature

4. Gateway stateful validator calls skip_stateful_validations:
     - tx.nonce() == 1                                  ✓
     - account_nonce (from state) == 0                  ✓
     - account_tx_in_pool_or_recent_block(A) == true    ✓  (deploy_account is in pool)
   → Returns true (skip validation)

5. run_validate_entry_point sets validate: false → __validate__ never called.
   Fake invoke is admitted to the mempool.

6. Batcher retrieves fake invoke, calls AccountTransaction::new_for_sequencing
   (validate: true), executes __validate__ → FAILS (invalid signature).
   Transaction included in block as REVERTED, consuming block space.
``` [5](#0-4) [2](#0-1) [4](#0-3)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
