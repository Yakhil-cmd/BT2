### Title
Unchecked EVM Call Result in `Account.Call()` and `Account.Deploy()` Allows Silent Cross-VM Asset Transfer Failure — (File: `fvm/evm/handler/handler.go`)

---

### Summary

`Account.Call()` and `Account.Deploy()` in the Flow EVM handler use `panicOnError(err)` to handle the result of an EVM execution, while the analogous fund-moving operations `Account.Deposit()`, `Account.Withdraw()`, and `Account.Transfer()` all use the stricter `panicOnErrorOrInvalidOrFailedState(res, err)`. When an EVM call or deployment fails (EVM reverts), `err` is `nil` and `panicOnError` does not panic. The Cadence transaction continues silently, the caller receives a `ResultSummary` with `Status.failed`, and any FLOW balance that was supposed to be transferred to the target address stays locked in the COA — without the Cadence layer reverting or signalling failure.

---

### Finding Description

`panicOnErrorOrInvalidOrFailedState` checks three conditions:

```go
func panicOnErrorOrInvalidOrFailedState(res *types.Result, err error) {
    if res != nil && res.Invalid() {
        panic(fvmErrors.NewEVMError(res.ValidationError))
    }
    if res != nil && res.Failed() {
        panic(fvmErrors.NewEVMError(res.VMError))
    }
    ...
    panicOnError(err)
}
``` [1](#0-0) 

`panicOnError` only checks the Go-level error:

```go
func panicOnError(err error) {
    if err == nil { return }
    ...
}
``` [2](#0-1) 

`Deposit`, `Withdraw`, and `Transfer` all use the strict form: [3](#0-2) [4](#0-3) [5](#0-4) 

But `Deploy` and `Call` use only `panicOnError`:

```go
func (a *Account) Deploy(...) *types.ResultSummary {
    res, err := a.executeAndHandleAuthorizedCall(...)
    panicOnError(err)          // ← does NOT check res.Failed() or res.Invalid()
    return res.ResultSummary()
}

func (a *Account) Call(...) *types.ResultSummary {
    res, err := a.executeAndHandleAuthorizedCall(...)
    panicOnError(err)          // ← does NOT check res.Failed() or res.Invalid()
    return res.ResultSummary()
}
``` [6](#0-5) [7](#0-6) 

When `Call` is invoked with a non-zero `balance`, the comment explicitly states the balance is deducted from the COA and transferred to the target: [8](#0-7) 

If the target EVM contract reverts, `executeAndHandleCall` returns `(res, nil)` where `res.Failed() == true`. The EVM state is rolled back (balance stays in COA), but `panicOnError(nil)` is a no-op. The Cadence transaction does **not** revert. The caller receives a `ResultSummary` with `Status.failed` and may not inspect it, silently believing the transfer succeeded.

---

### Impact Explanation

Any Cadence transaction or contract that calls `coa.call(to: ..., data: ..., gasLimit: ..., value: someBalance)` and does not explicitly assert `result.status == EVM.Status.successful` will silently proceed after a failed EVM execution. The FLOW tokens intended for the target address remain in the COA undistributed, with no Cadence-level revert or error propagation. This is a cross-VM asset accounting failure: the Cadence layer believes the transfer occurred while the EVM layer reverted it.

**Impact: Medium** — FLOW tokens are not lost (they remain in the COA), but the intended recipient does not receive them and the caller has no automatic signal of failure.

---

### Likelihood Explanation

**Likelihood: High** — The `call()` and `deploy()` functions are the primary cross-VM interaction surface for Cadence Owned Accounts. Any caller that follows the pattern of calling `coa.call()` with a value and not checking the returned `Result.status` is silently affected. The inconsistency with `Transfer`/`Deposit`/`Withdraw` (which panic on failure) makes it easy for developers to assume `Call` also panics on failure, leading to unchecked results.

---

### Recommendation

Apply `panicOnErrorOrInvalidOrFailedState(res, err)` in `Account.Deploy()` and `Account.Call()` consistently with `Deposit`, `Withdraw`, and `Transfer`. If the design intent is to allow callers to inspect failed results, the Cadence-facing `call` and `deploy` functions in `contract.cdc` should enforce a result-check pre/post-condition, or the Go-layer should at minimum document that callers must check `ResultSummary.Status`.

---

### Proof of Concept

1. A Cadence transaction creates a COA, deposits 1.0 FLOW, then calls a target EVM contract that always reverts:
   ```cadence
   let res = coa.call(to: revertingContract, data: [], gasLimit: 100_000, value: EVM.Balance(attoflow: 1_000_000_000_000_000_000))
   // res.status == EVM.Status.failed — but no panic, no revert
   // 1.0 FLOW stays in COA, target receives nothing
   ```
2. `Account.Call()` calls `executeAndHandleAuthorizedCall`, which calls `executeAndHandleCall`.
3. `executeAndHandleCall` returns `(res, nil)` with `res.Failed() == true` because the EVM reverted.
4. `panicOnError(nil)` is a no-op.
5. `res.ResultSummary()` is returned with `Status: failed`.
6. The Cadence transaction does not revert; the 1.0 FLOW is not transferred to the target. [9](#0-8) [10](#0-9)

func (h *ContractHandler) executeAndHandleCall(
	call *types.DirectCall,
	totalSupplyDiff *big.Int,
	deductSupplyDiff bool,
) (_ *types.Result, err error) {
	defer func() {
		if err == nil {
			// Invalidate drycall cache if EVM state is changed (executeAndHandleCall is successful).
			h.invalidateDryCallCache()
		}
	}()

	// step 1 - check enough computation is available
	if err := h.checkGasLimit(types.GasLimit(call.GasLimit)); err != nil {
		return nil, err
	}

	// step 2 - prepare the block context
	// load the block proposal
	bp, err := h.getBlockProposal()
	if err != nil {
		return nil, err
	}
	ctx, err := h.getBlockContext(bp)
	if err != nil {
		return nil, err
	}

	// step 3 - create block view
	blk, err := h.emulator.NewBlockView(ctx)
	if err != nil {
		return nil, err
	}

	// step 4 - run direct call
	var res *types.Result
	h.backend.RunWithMeteringDisabled(
		func() {
			res, err = blk.DirectCall(call)
		})
	// check backend errors first
	if err != nil {
		return nil, err
	}
	if res == nil { // safety check for result
		return nil, types.ErrUnexpectedEmptyResult
	}

	// step 5 - gas meter even invalid or failed status
	err = h.meterGasUsage(res)
	if err != nil {
		return nil, err
	}

	// step 6 - if is invalid skip the rest of states
	if res.Invalid() {
		return res, nil
	}

	// step 7 - update block proposal
	// append transaction to the block proposal
	bp.AppendTransaction(res)

	// update total supply if applicable
	if res.Successful() && totalSupplyDiff != nil {
		if deductSupplyDiff {
			bp.TotalSupply = new(big.Int).Sub(bp.TotalSupply, totalSupplyDiff)
			if bp.TotalSupply.Sign() < 0 {
				return nil, types.ErrInsufficientTotalSupply
			}
		} else {
			bp.TotalSupply = new(big.Int).Add(bp.TotalSupply, totalSupplyDiff)
		}
	}

	// update the block proposal
	h.backend.StageBlockProposal(bp)

	// step 8 - emit transaction event
	encoded, err := call.Encode()
	if err != nil {
		return nil, err
	}
	err = h.emitEvent(
		events.NewTransactionEvent(res, encoded, bp.Height),
	)
	if err != nil {
		return nil, err
	}

	// step 9 - report metrics
	h.backend.EVMTransactionExecuted(
		res.GasConsumed,
		true,
		res.Failed(),
	)

	return res, nil
}

func panicOnErrorOrInvalidOrFailedState(res *types.Result, err error) {

	if res != nil && res.Invalid() {
		panic(fvmErrors.NewEVMError(res.ValidationError))
	}

	if res != nil && res.Failed() {
		panic(fvmErrors.NewEVMError(res.VMError))
	}

	// this should never happen
	if err == nil && res == nil {
		panic(fvmErrors.NewEVMError(types.ErrUnexpectedEmptyResult))
	}

	panicOnError(err)
}

func panicOnError(err error) {
	if err == nil {
		return
	}

	if types.IsAFatalError(err) {
		panic(fvmErrors.NewEVMFailure(err))
	}

	if types.IsABackendError(err) {
		// backend errors doesn't need wrapping
		panic(err)
	}

	// any other returned errors are non-fatal errors
	panic(fvmErrors.NewEVMError(err))
}
