### Title
`StatelessTransactionValidator::validate_resource_bounds` unconditionally checks `l2_gas.max_price_per_unit` against `min_gas_price` even when `l2_gas.max_amount = 0`, incorrectly rejecting valid V3 transactions that pay only via L1 gas — (`crates/apollo_gateway/src/stateless_transaction_validator.rs`)

---

### Summary

The stateless validator's `validate_resource_bounds` applies the `min_gas_price` floor unconditionally to `l2_gas.max_price_per_unit`, regardless of whether `l2_gas.max_amount` is zero. A V3 transaction with `AllResourceBounds{l1_gas: non-zero, l2_gas: {amount:0, price:0}, l1_data_gas: zero}` passes the non-zero-fee guard (because `max_possible_fee > 0` from L1 gas) but is then rejected with `MaxGasPriceTooLow` because `0 < min_gas_price`. The codebase's own test suite explicitly marks such a transaction as valid (`valid_l1_gas` positive-flow case), confirming the intent to accept it.

---

### Finding Description

In `stateless_transaction_validator.rs`, `validate_resource_bounds` performs two sequential checks:

**Check 1 — non-zero fee guard** (line 66):
```rust
if ValidResourceBounds::AllResources(resource_bounds).max_possible_fee(Tip::ZERO) == Fee(0) {
    return Err(ZeroResourceBounds { ... });
}
```
`max_possible_fee` sums all three resources. A transaction with `l1_gas = {amount:1000, price:1_000_000}` produces fee = 1,000,000,000 ≠ 0, so this check passes.

**Check 2 — min_gas_price guard** (line 71):
```rust
if resource_bounds.l2_gas.max_price_per_unit.0 < self.config.min_gas_price {
    return Err(MaxGasPriceTooLow { ... });
}
```
This check is **unconditional**: it fires even when `l2_gas.max_amount = 0`, making the price economically irrelevant (0 × price = 0 fee contribution). With the production default `min_gas_price = 8_000_000_000`, any transaction where `l2_gas.max_price_per_unit < 8_000_000_000` — including the zero case — is rejected.

The test suite at lines 70–81 of `stateless_transaction_validator_test.rs` explicitly marks `AllResourceBounds{l1_gas: NON_EMPTY_RESOURCE_BOUNDS, ..Default::default()}` (l2_gas = zero) as a **positive flow** case. That test only passes because `DEFAULT_VALIDATOR_CONFIG_FOR_TESTING` sets `min_gas_price: 0`. With the production config (`min_gas_price: 8_000_000_000`), the identical transaction is rejected. ### Impact Explanation

**Impact: High — Mempool/gateway admission rejects valid transactions before sequencing.**

Any V3 transaction that legitimately sets `l2_gas.max_amount = 0` (paying only via L1 gas or L1 data gas) is permanently rejected at the stateless gateway layer with `MaxGasPriceTooLow`. The rejection is deterministic and not recoverable without changing the transaction structure. The corrupted admission value is concrete: `MaxGasPriceTooLow { gas_price: GasPrice(0), min_gas_price: 8_000_000_000 }` for a transaction that has a valid, non-zero max possible fee. [4](#0-3) 

---

### Likelihood Explanation

The `valid_l1_gas` and `valid_l1_data_gas` test cases confirm the protocol intends to accept such transactions. Any user or SDK that constructs a V3 transaction with `l2_gas = {amount:0, price:0}` (a natural default when not using L2 gas) will be rejected by any sequencer running the production config. The production `min_gas_price` default of `8_000_000_000` makes this trigger on every such transaction. [5](#0-4) [6](#0-5) 

---

### Recommendation

Gate the `min_gas_price` check on `l2_gas.max_amount > 0`:

```rust
if resource_bounds.l2_gas.max_amount.0 > 0
    && resource_bounds.l2_gas.max_price_per_unit.0 < self.config.min_gas_price
{
    return Err(StatelessTransactionValidatorError::MaxGasPriceTooLow { ... });
}
```

This mirrors the logic already used for the `max_l2_gas_amount` check at line 80, which is also conditioned on the transaction type (and implicitly on l2_gas being relevant). [7](#0-6) 

---

### Proof of Concept

The existing test infrastructure already demonstrates the bug. The `valid_l1_gas` case at line 70 passes only because `min_gas_price = 0` in the test config. Switching to the production config exposes the rejection:

```rust
#[test]
fn test_l1_only_tx_rejected_with_production_min_gas_price() {
    // Production config: min_gas_price = 8_000_000_000
    let config = StatelessTransactionValidatorConfig::default();
    assert_eq!(config.min_gas_price, 8_000_000_000);

    let validator = StatelessTransactionValidator { config };

    // Valid V3 transaction: pays only via L1 gas, l2_gas zeroed
    let resource_bounds = AllResourceBounds {
        l1_gas: ResourceBounds {
            max_amount: GasAmount(1000),
            max_price_per_unit: GasPrice(1_000_000),
        },
        l2_gas: ResourceBounds {
            max_amount: GasAmount(0),
            max_price_per_unit: GasPrice(0),
        },
        l1_data_gas: ResourceBounds::default(),
    };

    let tx = rpc_tx_for_testing(
        TransactionType::Invoke,
        RpcTransactionArgs { resource_bounds, ..Default::default() },
    );

    // max_possible_fee = 1000 * 1_000_000 = 1_000_000_000 > 0 → passes ZeroResourceBounds check
    // l2_gas.max_price_per_unit (0) < min_gas_price (8_000_000_000) → INCORRECTLY rejected
    assert_eq!(
        validator.validate(&tx).unwrap_err(),
        StatelessTransactionValidatorError::MaxGasPriceTooLow {
            gas_price: GasPrice(0),
            min_gas_price: 8_000_000_000,
        }
    );
}

    pub fn max_possible_fee(&self, tip: Tip) -> Fee {
        match self {
            ValidResourceBounds::L1Gas(l1_bounds) => {
                l1_bounds.max_amount.saturating_mul(l1_bounds.max_price_per_unit)
            }
            ValidResourceBounds::AllResources(AllResourceBounds {
                l1_gas,
                l2_gas,
                l1_data_gas,
