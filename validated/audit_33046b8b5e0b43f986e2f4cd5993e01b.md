### Title
Permanent Fund Loss When `ft_transfer` Promise Fails in `ExitToNear` Precompile Without `error_refund` Feature — (`engine-precompiles/src/native.rs`)

---

### Summary

The `ExitToNear` precompile burns ETH (or ERC-20 tokens) from the EVM side and schedules a NEAR `ft_transfer` promise to deliver NEP-141 tokens to the recipient. When the `error_refund` feature is **not** compiled in, no refund callback is registered. If the `ft_transfer` promise fails for any reason, the EVM-side burn is irreversible and the user's funds are permanently destroyed.

---

### Finding Description

In `engine-precompiles/src/native.rs`, the `ExitToNear::run()` method constructs a `callback_args` struct that conditionally includes a refund address only when the `error_refund` Cargo feature is enabled:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,
    transfer_near: transfer_near_args,
};
``` [1](#0-0) 

For the standard ETH base-token exit and for non-wNEAR ERC-20 exits, `transfer_near_args` is also `None`, so `callback_args` equals the default value. The branch that decides whether to attach a callback then selects the no-callback path:

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // ← no callback registered
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs { ... })
};
``` [2](#0-1) 

The `transfer_promise` calls `ft_transfer` on the ETH connector contract:

```rust
let transfer_promise = PromiseCreateArgs {
    target_account_id: nep141_address,
    method,                          // "ft_transfer"
    args: args.into_bytes(),
    attached_balance: Yocto::new(1),
    attached_gas,
};
``` [3](#0-2) 

Because the EVM state change (ETH deduction / ERC-20 burn) is committed **before** the async NEAR promise executes, a failure of `ft_transfer` leaves the EVM balance permanently reduced with no corresponding NEP-141 credit on the NEAR side and no mechanism to restore the EVM balance.

The project's own integration test explicitly documents this behavior:

```rust
#[cfg(feature = "error_refund")]
let expected_balance = Wei::new_u64(INITIAL_ETH_BALANCE);
// If the refund feature is not enabled, then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let expected_balance = Wei::new_u64(INITIAL_ETH_BALANCE - ETH_EXIT_AMOUNT);
``` [4](#0-3) 

---

### Impact Explanation

**Critical — Permanent freezing / destruction of user funds.**

When `ft_transfer` fails, the ETH (or ERC-20 tokens) are burned from the EVM state with no recovery path. The funds do not arrive on the NEAR side and cannot be reclaimed from the EVM side. This is a one-way, irreversible loss.

---

### Likelihood Explanation

`ft_transfer` on a NEP-141 contract fails in several realistic, user-triggerable scenarios:

1. **Recipient has no storage deposit** on the ETH connector NEP-141 contract. A user exiting ETH to a freshly created NEAR account that has never registered storage will cause `ft_transfer` to panic, permanently destroying the ETH.
2. **Aurora contract NEP-141 balance is insufficient** due to any accounting discrepancy — the `ft_transfer` will fail.
3. **Gas exhaustion** at the NEAR level during the promise execution.

Scenario 1 is trivially reachable by any unprivileged user who simply provides a NEAR account ID that has not called `storage_deposit` on the connector. No admin compromise or special privilege is required.

---

### Recommendation

Always register a refund callback regardless of the `error_refund` feature flag, or make `error_refund` a mandatory (non-optional) feature for production builds. The callback should restore the EVM balance (ETH credit or ERC-20 re-mint) whenever the `ft_transfer` promise returns a failure result, mirroring the existing `refund_on_error` logic already present in `engine/src/engine.rs`. [5](#0-4) 

---

### Proof of Concept

1. Deploy Aurora Engine **without** the `error_refund` feature.
2. Create a NEAR account `victim.near` that has **not** called `storage_deposit` on the ETH connector NEP-141 contract.
3. From an EVM address with balance, call the `ExitToNear` precompile (`0xe9217bc70b7ed1f598ddd3199e80b093fa71124f`) with flag `0x00` and recipient `victim.near`.
4. The EVM deducts the ETH from the caller's balance immediately.
5. The scheduled `ft_transfer` promise to the ETH connector fails because `victim.near` has no storage registration.
6. No callback fires; the EVM balance is not restored.
7. Observe: caller's EVM ETH balance is reduced, `victim.near` NEP-141 balance is zero, and there is no way to recover the funds.

This matches the test scenario in `test_exit_to_near_eth_refund` which explicitly confirms the `INITIAL_ETH_BALANCE - ETH_EXIT_AMOUNT` outcome (funds lost) when `error_refund` is absent. [6](#0-5)

pub fn refund_on_error<I: IO + Copy, E: Env, P: PromiseHandler>(
    io: I,
    env: &E,
    state: EngineState,
    args: &RefundCallArgs,
    handler: &mut P,
) -> EngineResult<SubmitResult> {
    let current_account_id = env.current_account_id();
    if let Some(erc20_address) = args.erc20_address {
        // ERC-20 exit; re-mint burned tokens
        let erc20_admin_address = current_address(&current_account_id);
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, erc20_admin_address, current_account_id, io, env);

        let refund_address = args.recipient_address;
        let amount = U256::from_big_endian(&args.amount);
        let input = setup_refund_on_error_input(amount, refund_address);

        engine.call(
            &erc20_admin_address,
            &erc20_address,
            Wei::zero(),
            input,
            u64::MAX,
            Vec::new(),
            Vec::new(),
            handler,
        )
    } else {
        // ETH exit; transfer ETH back from precompile address
        let exit_address = exit_to_near::ADDRESS;
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, exit_address, current_account_id, io, env);
        let refund_address = args.recipient_address;

        engine.call(

            &refund_address,
            amount,
            Vec::new(),
            u64::MAX,
            vec![
                (exit_address.raw(), Vec::new()),
                (refund_address.raw(), Vec::new()),
            ],
            Vec::new(),
            handler,
        )
