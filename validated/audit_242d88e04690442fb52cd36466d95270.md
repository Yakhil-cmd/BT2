### Title
ERC-20 Mirror Token Balances Don't Account for NEP-141 Token Rebases — (File: `engine/src/engine.rs`)

---

### Summary

When a NEP-141 token is bridged into Aurora via `ft_on_transfer`, the engine mints a fixed ERC-20 balance equal to the transferred amount and stores it in Aurora's EVM key-value state. This stored balance is never updated to reflect external changes to the NEP-141 token's supply (rebases). If the underlying NEP-141 token undergoes a rebase-down (e.g., slashing), the total ERC-20 supply inside Aurora exceeds the actual NEP-141 balance held by the Aurora contract, causing bridge insolvency for that token. If the token rebases up, the surplus NEP-141 tokens are permanently locked with no corresponding ERC-20 representation.

---

### Finding Description

**Mint path (bridge-in):**

`ft_on_transfer` in `engine/src/contract_methods/connector.rs` dispatches to `engine.receive_erc20_tokens` when the predecessor is a registered NEP-141 token (not the ETH connector). [1](#0-0) 

`receive_erc20_tokens` calls the ERC-20 contract's `mint` function for exactly `args.amount` tokens and stores the resulting balance in Aurora's EVM state: [2](#0-1) 

The minted amount is a fixed snapshot of the NEP-141 tokens transferred at bridge-in time. It is written to NEAR key-value storage via `set_balance`: [3](#0-2) 

There is no mechanism anywhere in the engine to re-sync this stored ERC-20 balance against the actual NEP-141 balance held by the Aurora contract.

**Exit path (bridge-out):**

`exit_erc20_token_to_near` in `engine-precompiles/src/native.rs` burns the ERC-20 tokens inside the EVM and then issues an `ft_transfer` (or `ft_transfer_call`) on the NEP-141 contract for the exact burned amount: [4](#0-3) 

**The divergence:**

If the NEP-141 token rebases while held by Aurora:

- **Rebase-down (slashing):** Aurora's actual NEP-141 balance decreases, but the aggregate ERC-20 supply inside the EVM is unchanged. The first users to exit succeed; later users' `ft_transfer` calls fail because Aurora no longer holds enough NEP-141 tokens. The remaining ERC-20 holders cannot exit — their tokens are permanently frozen.

- **Rebase-up:** Aurora's NEP-141 balance grows, but no additional ERC-20 tokens are minted. The surplus NEP-141 tokens are locked inside the Aurora contract with no corresponding ERC-20 representation and no mechanism to claim them — constituting theft of yield.

---

### Impact Explanation

**Rebase-down (primary impact — Critical):** Bridge insolvency for the affected NEP-141 token. The total ERC-20 supply exceeds the NEP-141 balance Aurora can actually transfer out. Users who attempt to exit after the rebase event find their `ft_transfer` calls failing. Their ERC-20 tokens are permanently frozen inside Aurora with no redemption path.

**Rebase-up (secondary impact — High):** The accrued yield (extra NEP-141 tokens held by Aurora) is permanently inaccessible to any user, constituting theft of unclaimed yield.

---

### Likelihood Explanation

Liquid staking tokens on NEAR (e.g., stNEAR, LINEAR) are rebasing NEP-141 tokens that are realistic candidates for bridging to Aurora. Any user can bridge any registered NEP-141 token to Aurora via the standard `ft_transfer_call` → `ft_on_transfer` flow. No privileged access is required. A rebase event is a normal, scheduled operation of such tokens, not an attack. The combination of a rebasing token being bridged and a rebase occurring while tokens are held by Aurora is a realistic scenario.

---

### Recommendation

Add an explicit comment in `receive_erc20_tokens` and in the `_balances`-equivalent storage functions (`set_balance`, `get_balance`) documenting that ERC-20 mirror balances do not account for NEP-141 token rebases. Consider either:

1. Rejecting known rebasing NEP-141 tokens at `deploy_erc20_token` time (requires an on-chain registry or flag), or
2. Implementing a `sync_erc20_balance` mechanism that queries the actual NEP-141 balance held by Aurora and adjusts the ERC-20 supply accordingly.

---

### Proof of Concept

1. Deploy a rebasing NEP-141 token on NEAR (e.g., a liquid staking token that increases/decreases balances proportionally).
2. Call `deploy_erc20_token` on Aurora to register the NEP-141 → ERC-20 mapping.
3. Two users (`Alice` and `Bob`) each call `ft_transfer_call` on the NEP-141, transferring 100 tokens each to Aurora. Aurora mints 100 ERC-20 tokens for each.
   - Aurora's NEP-141 balance: 200. Total ERC-20 supply: 200.
4. The NEP-141 token undergoes a 20% slash. Aurora's NEP-141 balance drops to 160. Total ERC-20 supply remains 200.
5. Alice calls the `ExitToNear` precompile, burning 100 ERC-20 tokens. Aurora calls `ft_transfer(alice, 100)` — succeeds. Alice receives 100 NEP-141 tokens.
   - Aurora's NEP-141 balance: 60. Bob's ERC-20 balance: 100.
6. Bob calls the `ExitToNear` precompile, burning 100 ERC-20 tokens. Aurora calls `ft_transfer(bob, 100)` — **fails** (Aurora only holds 60).
7. Bob's 100 ERC-20 tokens are permanently frozen inside Aurora with no exit path.

The root cause is in `receive_erc20_tokens` [2](#0-1)  and `set_balance` [3](#0-2) , which together store a fixed ERC-20 balance that is never reconciled against the actual NEP-141 balance held by the Aurora contract.

    pub fn receive_erc20_tokens<P: PromiseHandler>(
        &mut self,
        token: &AccountId,
        args: &FtOnTransferArgs,
        current_account_id: &AccountId,
        handler: &mut P,
    ) -> Result<Option<SubmitResult>, ContractError> {
        let amount = args.amount.as_u128();
        // Parse message to determine recipient
        let mut recipient = {
            // The message should contain the recipient EOA address.
            let message = args.msg.strip_prefix("0x").unwrap_or(&args.msg);
            // Recipient - 40 characters (Address in hex without '0x' prefix)
            if message.len() < 40 {
                return Err(ParseOnTransferMessageError::WrongMessageFormat.into());
            }
            let mut address_bytes = [0; 20];
            hex::decode_to_slice(&message[..40], &mut address_bytes)
                .map_err(|_| ParseOnTransferMessageError::WrongMessageFormat)?;
            Address::from_array(address_bytes)
        };

        if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
            && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
        {
            recipient = fallback_address;
        }

        let erc20_token = get_erc20_from_nep141(&self.io, token)?;
        let erc20_admin_address = current_address(current_account_id);
        let result = self
            .call(
                &erc20_admin_address,
                &erc20_token,
                Wei::zero(),
                setup_receive_erc20_tokens_input(&recipient, amount),
                u64::MAX,
                Vec::new(), // TODO: are there values we should put here?
                Vec::new(),
                handler,
            )
            .and_then(submit_result_or_err)?;

        sdk::log!("Mint {amount} ERC-20 tokens for: {}", recipient.encode());

        // Return SubmitResult so that it can be accessed in standalone engine.
        // This is used to help with the indexing of bridge transactions.
        Ok(Some(result))
    }
