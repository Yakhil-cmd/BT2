### Title
`protocolUnpausePool` Sets Pause Level to 1 Instead of 0, Leaving Pool Permanently Paused - (`metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary

`MetricOmmPoolFactory.protocolUnpausePool` calls `setPause(1)` (admin-paused) instead of `setPause(0)` (active). After the protocol owner "unpauses" a pool that was active before being protocol-paused, the pool remains paused at level 1, blocking all swaps indefinitely.

### Finding Description

The pause-level semantics are documented in `MetricOmmPool.sol`: [1](#0-0) 

```
/// @dev 0 = active, 1 = paused by admin, 2 = paused by protocol.
```

`protocolPausePool` accepts a pool at level 0 **or** level 1 and elevates it to level 2: [2](#0-1) 

`protocolUnpausePool` is supposed to reverse that, but it calls `setPause(1)` ‚Äî the admin-paused level ‚Äî instead of `setPause(0)`: [3](#0-2) 

```solidity
function protocolUnpausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 2) revert InvalidPauseTransition(cur, 1);
    IMetricOmmPoolFactoryActions(pool).setPause(1);   // ‚Üê should be setPause(0)
}
```

`MetricOmmPool.swap` is gated by `whenNotPaused`, which reverts on **any** non-zero pause level: [4](#0-3) 

So after `protocolUnpausePool` executes, `pauseLevel == 1` and every swap call still reverts with `PoolPaused`.

The admin-side `unpausePool` only accepts a transition from level 1 ‚Üí 0: [5](#0-4) 

This means the pool admin must separately call `unpausePool` to reach level 0 ‚Äî a step the protocol owner's action was supposed to accomplish. If the pool was originally at level 0 before the protocol pause, the protocol owner has no path to restore it to active without the pool admin's cooperation.

### Impact Explanation

Every swap on the affected pool reverts with `PoolPaused` even after the protocol owner has called `protocolUnpausePool`. Core pool functionality (swaps) is broken. The pool is effectively frozen for trading until the pool admin independently calls `unpausePool`. If the admin key is unavailable or the admin is a contract without that call path, the pool is permanently bricked for swaps.

### Likelihood Explanation

The trigger is a normal, expected operational sequence: protocol owner pauses an active pool (level 0 ‚Üí 2) for an emergency, then calls `protocolUnpausePool` to restore service. This is a routine admin action with no special preconditions. The wrong constant is always used ‚Äî there is no code path inside `protocolUnpausePool` that reaches `setPause(0)`.

### Recommendation

Change `setPause(1)` to `setPause(0)` in `protocolUnpausePool`:

```solidity
function protocolUnpausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 2) revert InvalidPauseTransition(cur, 0);
    IMetricOmmPoolFactoryActions(pool).setPause(0);   // restore to active
}
```

If the design intent is to restore the pool to admin-paused (level 1) when it was at level 1 before the protocol pause, the factory must store the pre-protocol-pause level and restore it. The current code does neither correctly.

### Proof of Concept

1. Pool is active: `pauseLevel == 0`.
2. Protocol owner calls `protocolPausePool(pool)` ‚Üí `pauseLevel == 2`. All swaps revert.
3. Emergency resolved; protocol owner calls `protocolUnpausePool(pool)`.
4. `setPause(1)` executes ‚Üí `pauseLevel == 1`.
5. Any user calls `pool.swap(...)` ‚Üí `_checkNotPaused()` sees `pauseLevel != 0` ‚Üí reverts `PoolPaused`.
6. Protocol owner has no further recourse; only the pool admin calling `unpausePool` can reach level 0.
