### Title
Single malicious participant in sorted positions 0..=threshold can permanently abort presign for all honest parties by broadcasting an off-polynomial `R_me` — (`src/ecdsa/robust_ecdsa/presign.rs`)

### Summary
The robust ECDSA presign protocol claims to tolerate up to `t` malicious participants with `N = 2t+1`. However, a single malicious participant whose sorted index falls in `0..=threshold` can broadcast an arbitrary `R_me = g^{k'_me}` (not on the honest degree-t polynomial), causing the exponent interpolation check at step 3.2 to fail for every honest participant, permanently aborting presign.

### Finding Description
In `do_presign`, each participant broadcasts `big_r_me = g^{k_me}` (step 2.3) with no proof of knowledge or commitment binding it to the actual `k_me` used in the secret polynomial shares. After collecting all `R` values, the code splits the sorted list at `threshold+1` and uses the first `t+1` entries as the Lagrange interpolation basis. It then checks that every remaining participant's received `R_i` equals the value interpolated from that basis.

Because the basis is taken blindly from the first `t+1` sorted positions, a single malicious participant in those positions can inject an arbitrary group element. The resulting interpolation polynomial differs from the honest one, so every honest participant in positions `threshold+1..2t+1` fails the consistency check, and all honest parties return `Err(ProtocolError::AssertionFailed)`.

### Impact Explanation
Permanent denial of presign (and therefore signing) for all honest parties. The protocol is documented as robust against up to `t` malicious participants, but a single adversary in the first `t+1` sorted slots can break this guarantee. Since participant IDs are `u32` values chosen at setup, an attacker can trivially select a small ID to guarantee placement in the vulnerable range.

### Likelihood Explanation
High. The attacker only needs to be a registered participant with a sufficiently small ID (e.g., `Participant::from(0u32)`). No cryptographic assumption needs to be broken. The attack requires sending one malformed message per presign round.

### Recommendation
Bind each participant's `R_me` to their actual nonce `k_me` before the interpolation check is applied. Concretely:
- Require each participant to provide a Schnorr proof of knowledge of `k_me` alongside `R_me = g^{k_me}`, or
- Use a commit-then-reveal scheme so that `R_me` is committed before any values are revealed, and the commitment is verified before the interpolation check.

Additionally, the interpolation check should not treat the first `t+1` values as unconditionally trusted ground truth. A robust scheme should use all `2t+1` values and apply an error-correction or majority-vote approach to identify and exclude inconsistent points rather than aborting on the first mismatch.
