### Title
Unvalidated On-Chain `ph` Puzzle Hash Spoofs Counter-Offer Notification Destination, Causing Victim to Send XCH and Counter-Offer to Attacker — (`packages/gui/src/hooks/useBlockchainNotifications.tsx`, `packages/gui/src/components/offers2/OfferIncomingTable.tsx`)

---

### Summary

An unprivileged on-chain attacker can embed an arbitrary `ph` (puzzle hash) field in a blockchain notification message. The GUI parses this value without any validation and propagates it through the counter-offer flow, ultimately pre-populating a **disabled, uneditable** `NotificationSendDialog` address field with the attacker's address. When the victim clicks "Send Message," they send 0.0001 XCH directly to the attacker's address and deliver their counter-offer to the attacker instead of the original offer sender.

---

### Finding Description

**Step 1 — Attacker-controlled `puzzleHash` stored without validation**

In `prepareNotifications`, the `ph` field from the on-chain JSON message is extracted and stored directly into the notification object with no format, length, or ownership check: [1](#0-0) 

Any hex string the attacker places in `ph` becomes `notification.puzzleHash`.

**Step 2 — `handleCounterOffer` converts it to a bech32m address with no validation**

`toBech32m` accepts any valid hex string of any length and encodes it. There is no check that the puzzle hash is 32 bytes or belongs to any known wallet: [2](#0-1) 

The resulting `address` (attacker's bech32m address) is passed as navigation state to `/dashboard/offers/builder`.

**Step 3 — `CreateOfferBuilder` forwards the address to `onOfferCreated`**

After the counter-offer is created, the attacker's address is forwarded verbatim: [3](#0-2) 

**Step 4 — `handleOfferCreated` in `OfferManager` passes it to `OfferShareDialog`** [4](#0-3) 

**Step 5 — `OfferShareDialog` opens `NotificationSendDialog` with the attacker's address** [5](#0-4) 

**Step 6 — `NotificationSendDialog` uses the attacker's address as the disabled, uneditable destination**

The address field is `disabled`; the victim cannot change it. On submit, `sendNotification` sends 0.0001 XCH to `fromBech32m(address)` — the attacker's puzzle hash: [6](#0-5) 

`toBech32m` itself only rejects non-hex input; it imposes no 32-byte or wallet-ownership constraint: [7](#0-6) 

---

### Impact Explanation

- **Direct XCH transfer**: The victim sends 0.0001 XCH to the attacker's address as the notification fee. The address field is `disabled` — the victim has no opportunity to correct it.
- **Counter-offer misdirection**: The attacker receives the counter-offer notification and can accept the counter-offer, executing the trade on terms the victim set while believing they were responding to the original offer sender.
- **Spoofing**: The victim is shown the attacker's address labeled as "Notification recipient's address" with no indication it is attacker-controlled.

This satisfies the High scope criterion: *"unsafe trust of… notification… state that causes a user to… send… to the wrong… destination."*

---

### Likelihood Explanation

- Sending a blockchain notification costs a small amount of XCH and requires no special privilege.
- The victim only needs to click "Counter" on the notification and then "Send Message" in the share dialog — both are normal, expected user actions.
- The address field being `disabled` removes any chance of the victim correcting the destination.

---

### Recommendation

1. **Validate `puzzleHash` in `prepareNotifications`**: Reject any `ph` value that is not a valid 64-character lowercase hex string (32 bytes). Discard the notification if validation fails.
2. **Do not trust on-chain `ph` as a transaction destination**: Treat the `ph` field as informational only; require the user to explicitly confirm or enter the counter-offer notification destination.
3. **Make the destination field editable** in `NotificationSendDialog` so users can verify and correct the pre-populated address before sending.

```typescript
export default function toBech32m(value: string, prefix: string): string {
  if (value.startsWith(prefix)) {
    return value;
  }

  const pureHash = removePrefix(value, '0x');
  const hexBytes = Buffer.from(pureHash, 'hex');

  if (pureHash.length / 2 !== hexBytes.length) {
    throw new Error('Invalid hex string');
  }

  const words = bech32m.toWords(hexBytes);

  return bech32m.encode(prefix, words);
}
```
