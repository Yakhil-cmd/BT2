### Title
Hardcoded `L1_TX_NATIVE_PRICE = 10` Causes Incorrect Native Resource Accounting for L1→L2 Transactions — (`basic_bootloader/src/bootloader/constants.rs`)

---

### Summary

`L1_TX_NATIVE_PRICE` is hardcoded to `10` and is used as the sole divisor when computing the native resource budget for every L1→L2 priority transaction. The developers themselves flag this as a placeholder with a `TODO` comment. Because the operator-set `native_price` for L2 transactions reflects actual proving costs and can be orders of magnitude larger than `10`, L1→L2 transactions receive a disproportionately large native resource budget relative to what they paid. Any unprivileged actor who can submit an L1→L2 transaction can exploit this to force the prover to perform far more work than the fee covers.

---

### Finding Description

In `basic_bootloader/src/bootloader/constants.rs`, the constant is defined as:

```rust
// Default native price for L1->L2 transactions.
// TODO (EVM-1157): find a reasonable value for it.
pub const L1_TX_NATIVE_PRICE: U256 = U256::from_limbs([10, 0, 0, 0]);
``` [1](#0-0) 

This constant is consumed in `prepare_and_check_resources` inside `process_l1_transaction.rs`:

```rust
// For L1->L2 txs, we use a constant native price to avoid censorship.
let native_price = L1_TX_NATIVE_PRICE;
...
u256_try_to_u64(&gas_price.div_ceil(native_price))
``` [2](#0-1) 

The resulting `native_per_gas` drives the native resource budget:

```
native_limit = gas_limit × gas_price / L1_TX_NATIVE_PRICE
             = gas_limit × gas_price / 10
``` [3](#0-2) 

For L2 transactions, the equivalent formula uses the operator-supplied `native_price` from block metadata:

```rust
let native_price = system.get_native_price();
...
u256_try_to_u64(&gas_price.div_ceil(native_price))
``` [4](#0-3) 

The operator-set `native_price` reflects actual proving costs. Realistic test values show `native_price = 911_615`, while the hardcoded L1 value is `10` — a ratio of ~91,000×. [5](#0-4) 

The same `native_per_gas` feeds the delta-gas calculation in `compute_gas_refund`:

```rust
let delta_gas = (native_used / native_per_gas) as i64 - (gas_used as i64);
if delta_gas > 0 { gas_used += delta_gas as u64; }
``` [6](#0-5) 

Because `native_per_gas` is inflated (due to the tiny divisor `10`), `native_used / native_per_gas` is deflated, so `delta_gas` is near zero or negative. The attacker is never charged the extra gas that would normally compensate for the excess native resource consumption.

---

### Impact Explanation

When the operator-set `native_price` is significantly larger than `10` (the realistic case in production), every L1→L2 transaction receives a native resource budget that is `native_price / 10` times larger than an equivalent L2 transaction. An attacker who submits an L1→L2 transaction with maximum `gas_limit` and `gas_price` can force the prover to execute up to `gas_limit × gas_price / 10` native units of work, while only paying fees proportional to `gas_used × gas_price` in base tokens. The proving cost for the excess native consumption is borne by the protocol/operator, constituting a direct public funds-loss path. Additionally, the delta-gas correction mechanism fails to recoup this cost because `native_per_gas` is inflated.

---

### Likelihood Explanation

The likelihood is high:
- The `TODO (EVM-1157)` comment in the source code explicitly acknowledges the value is a placeholder and not yet calibrated.
- Any user with ETH on L1 can submit an L1→L2 priority transaction — no privileged access is required.
- The discrepancy between `L1_TX_NATIVE_PRICE = 10` and realistic operator `native_price` values (e.g., `911_615`) is already visible in the existing test suite, confirming the gap exists in realistic deployments.
- L1→L2 transactions cannot be invalidated by the sequencer (by design, to prevent censorship), so the bootloader must process every such transaction regardless of the resource imbalance.

---

### Recommendation

1. Replace the placeholder `L1_TX_NATIVE_PRICE = 10` with a value calibrated to actual RISC-V proving costs, or derive it from a governance-controlled parameter rather than a compile-time constant.
2. If the anti-censorship invariant must be preserved (i.e., the operator cannot raise the price to block L1→L2 transactions), introduce a floor/ceiling mechanism: use `max(L1_TX_NATIVE_PRICE, operator_native_price)` so the budget is never more generous than what the operator charges L2 transactions.
3. Resolve the tracked issue `EVM-1157` before mainnet deployment.

---

### Proof of Concept

**Setup:** Operator sets `native_price = 911_615` for L2 transactions (matching the realistic test value).

**Attack:**
1. Attacker submits an L1→L2 priority transaction with `gas_limit = 10_000_000` and `gas_price = 1_000_000`.
2. ZKsync OS computes `native_per_gas = 1_000_000 / 10 = 100_000`.
3. Native resource budget: `10_000_000 × 100_000 = 10^12` native units.
4. An equivalent L2 transaction would receive: `10_000_000 × (1_000_000 / 911_615) ≈ 10_970` native units.
5. The L1→L2 transaction receives ~91,000× more native resource budget than an L2 transaction paying the same fee.
6. The attacker crafts calldata that drives maximum native consumption (e.g., repeated storage writes, hash operations). The prover must prove all consumed native units; the attacker's fee covers only a tiny fraction of the actual proving cost.
7. Because `native_per_gas` is inflated, `delta_gas ≈ 0`, so no corrective gas charge is applied, and the attacker receives a near-full gas refund on unused EVM gas.
