### Title
Signature Bypass via Unverified `account_tx_in_pool_or_recent_block` in `skip_stateful_validations` — (File: crates/apollo_gateway/src/stateful_transaction_validator.rs)

---

### Summary

The gateway's `skip_stateful_validations` function omits the `__validate__` entry-point call (i.e., signature verification) for any Invoke transaction with nonce=1 when the on-chain account nonce is 0 and `account_tx_in_pool_or_recent_block` returns `true`. That helper returns `true` for **any** transaction in the pool for the target address — not exclusively a `deploy_account`. An unprivileged attacker who observes a victim's pending `deploy_account` in the public mempool can therefore submit an Invoke with an arbitrary invalid signature for the victim's address, and the gateway will accept it without running `__validate__`, admitting an unauthenticated transaction into the mempool.

---

### Finding Description

`skip_stateful_validations` (lines 429–461 of `crates/apollo_gateway/src/stateful_transaction_validator.rs`) is the UX shortcut that lets a user broadcast `deploy_account + invoke(nonce=1)` atomically before the account is on-chain. Its guard is:

```
tx is Invoke
  AND tx.nonce() == Nonce(Felt::ONE)
  AND account_nonce_in_state == Nonce(Felt::ZERO)
  AND mempool.account_tx_in_pool_or_recent_block(sender) == true
``` [1](#0-0) 

When all four conditions hold, `run_pre_validation_checks` returns `skip_validate = true`, which propagates into `run_validate_entry_point` as `ExecutionFlags { validate: false, … }`. [2](#0-1) 

Inside `StatefulValidator::perform_validations`, `validate: false` causes an early return before the `__validate__` call: [3](#0-2) 

The fourth guard delegates to:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [4](#0-3) 

This returns `true` for **any** transaction type in the pool for that address. The code comment claims this is safe because "it means that either it has a deploy_account transaction or transactions with future nonces that passed validations," but neither branch guarantees the *incoming* nonce-1 Invoke is authentic:

- **Branch 1 (deploy_account in pool):** The deploy_account belongs to the victim. The attacker has no private key for the address but can still submit a nonce-1 Invoke with a fabricated signature.
- **Branch 2 (future-nonce Invoke in pool):** Those transactions passed their own signature checks, but that says nothing about the new nonce-1 Invoke being submitted now. [5](#0-4) 

The mempool's `validate_tx` (called by `validate_by_mempool` before `skip_stateful_validations`) only checks nonce ordering, duplicate hashes, and fee-escalation rules — it never inspects the signature. [6](#0-5) 

Fee escalation is **enabled by default** (`enable_fee_escalation: true`, `fee_escalation_percentage: 10`). [7](#0-6) 

This means an attacker who bids a tip ≥ 110 % of the victim's pending nonce-1 Invoke will atomically **replace** the victim's legitimate transaction with the attacker's invalid one. [8](#0-7) 

---

### Impact Explanation

**Matching scope item:** *High — Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.*

Two concrete harms:

1. **Unauthenticated admission.** Any Invoke with a forged signature for a not-yet-deployed account is accepted by the gateway and stored in the mempool, violating the invariant that every admitted transaction has passed account-level authentication.

2. **Victim transaction displacement via fee escalation.** If the victim has already submitted a legitimate nonce-1 Invoke, the attacker can replace it with an invalid one by paying a marginally higher tip. The invalid transaction will be reverted during block execution, but the victim's original transaction is permanently gone from the mempool and must be resubmitted — a targeted, low-cost denial-of-service against the deploy-account UX flow.

---

### Likelihood Explanation

- Mempool contents are broadcast over P2P gossip, so pending `deploy_account` transactions are publicly observable.
- The attacker needs only to craft an Invoke with `sender_address = victim_address`, `nonce = 1`, and any signature bytes; no cryptographic material is required.
- Fee escalation requires only a 10 % tip increase over the victim's transaction, which is economically trivial.
- The attack is repeatable: after the invalid transaction is rejected at execution time and removed from the mempool, the attacker can immediately resubmit.

---

### Recommendation

Replace the coarse `account_tx_in_pool_or_recent_block` check with a type-specific query that returns `true` only when a `deploy_account` transaction for the address is present in the pool or was recently committed. Concretely:

1. Add a `deploy_account_in_pool_or_recent_block(address)` API to the mempool that inspects the transaction type stored in `tx_pool`.
2. In `skip_stateful_validations`, call this narrower API instead of the existing one.

This preserves the intended UX (user sends `deploy_account + invoke` atomically) while closing the path by which a third party can trigger the skip for an address they do not control.

---

### Proof of Concept

```
// State: Alice's deploy_account for address A is in the mempool.
// account_nonce_in_state(A) == 0 (not yet committed).

// Attacker submits:
RpcInvokeTransactionV3 {
    sender_address: A,          // victim's address
    nonce: 1,                   // triggers the skip condition
    signature: [0xdead, 0xbeef], // arbitrary invalid bytes
    calldata: [...],            // arbitrary
    resource_bounds: { l2_gas: { max_amount: X, max_price_per_unit: Y } },
    // Y must be >= 110% of Alice's pending invoke's max_price_per_unit
    // to trigger fee-escalation replacement
    ...
}

// Gateway flow:
// 1. StatelessTransactionValidator::validate  → passes (address format OK, bounds OK)
// 2. convert_rpc_tx_to_internal               → passes (Invoke, no class hash check)
// 3. StatefulTransactionValidator:
//    a. validate_nonce: 0 <= 1 <= 0+max_gap   → passes
//    b. validate_by_mempool (nonce/dup check)  → passes
//    c. skip_stateful_validations:
//         tx.nonce() == 1                      ✓
//         account_nonce == 0                   ✓
//         account_tx_in_pool_or_recent_block(A)
//           → tx_pool.contains_account(A)      ✓  (Alice's deploy_account)
//         → returns true (skip __validate__)
//    d. run_validate_entry_point(skip=true)    → returns Ok(()) immediately
// 4. mempool.add_tx(attacker_invoke)           → accepted, replaces Alice's invoke
//    if attacker tip >= 110% of Alice's tip
```

The attacker's Invoke is now in the mempool. When the batcher executes it, `__validate__` runs, the signature check fails, and the transaction is reverted — but Alice's legitimate nonce-1 Invoke has already been evicted and is lost. [9](#0-8) [4](#0-3) [10](#0-9) [11](#0-10)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
