Audit Report

## Title
RestrictedRead Callers Leak Rule Lifecycle Metadata for Undisclosed Incidents via `get_rules_by_incident_id` — (`rs/boundary_node/rate_limits/canister/confidentiality_formatting.rs`, `getter.rs`, `add_config.rs`)

## Summary

`RuleConfidentialityFormatter` redacts only `rule_raw` and `description` for non-disclosed rules, leaving `added_in_version`, `removed_in_version`, and `rule_id` fully exposed to any `RestrictedRead` (anonymous) caller. Combined with the fact that `StorableIncident.rule_ids` permanently accumulates all rule IDs (including removed ones) and that incident IDs are not redacted in `get_config` responses, any unauthenticated caller can reconstruct the complete addition/removal timeline for every rule ever associated with a confidential, undisclosed incident.

## Finding Description

**Incident ID discovery via `get_config`:**
`get_config` is a `#[query]` method. Non-replicated query calls bypass `inspect_message` entirely. `ConfigConfidentialityFormatter::format` (`confidentiality_formatting.rs:17-28`) redacts only `rule_raw` and `description`; `incident_id` is left visible for every rule including those linked to undisclosed incidents. Any anonymous caller can enumerate all incident IDs in the current config.

**Historical rule enumeration via `get_rules_by_incident_id`:**
`get_rules_by_incident_id` (`canister.rs:136-146`) is a `#[query]` method not listed in `UPDATE_METHODS` or `REPLICATED_QUERY_METHOD`. As a non-replicated query it bypasses `inspect_message`. `IncidentGetter::get` (`getter.rs:161-202`) iterates over `stored_incident.rule_ids` and for `RestrictedRead` callers applies `RuleConfidentialityFormatter::format` (`confidentiality_formatting.rs:34-42`), which only nulls `description` and `rule_raw`. The fields `added_in_version`, `removed_in_version`, and `rule_id` are returned in full.

**Permanent accumulation of removed rule IDs:**
In `commit_changes` (`add_config.rs:219-232`), when a rule is removed from a config, only `removed_in_version` is set on the `StorableRule`. The rule ID is never removed from `StorableIncident.rule_ids`; the `extend` call only appends. Removed rules remain permanently queryable via `get_rules_by_incident_id`.

**Version-to-timestamp correlation:**
`get_config(version)` returns `active_since` for any version, allowing the attacker to map every `added_in_version`/`removed_in_version` integer to a wall-clock timestamp.

The existing test at `getter.rs:500-508` explicitly asserts that `added_in_version` and `removed_in_version` are returned to `RestrictedRead` callers for non-disclosed rules, confirming this is the actual runtime behavior.

## Impact Explanation

This is an information disclosure vulnerability in a production boundary node security component. The confidentiality system is designed to prevent adversaries from learning about ongoing rate-limit incidents. The leak exposes: the number of rules per incident, the exact config versions when each rule was introduced and retired, and (via `active_since` correlation) the wall-clock timestamps of those events — for every incident the operator intended to keep confidential. This matches **Medium ($200–$2,000)**: a real information disclosure in a production security system, requiring no privileges, but limited to operational metadata rather than the actual rule content.

## Likelihood Explanation

The exploit requires no privileges, no keys, and no special conditions. Both `get_config` and `get_rules_by_incident_id` are non-replicated query methods callable by any Internet Computer principal including anonymous callers. The attack is fully deterministic and repeatable. The only precondition is that at least one rule has been added to a confidential incident, which is the normal operating state of the system.

## Recommendation

1. **`RuleConfidentialityFormatter::format` (`confidentiality_formatting.rs:34-42`):** For non-disclosed rules, also set `added_in_version` to a sentinel (e.g., `0`) and `removed_in_version` to `None`.

2. **`IncidentGetter::get` (`getter.rs:161-172`):** Before iterating `stored_incident.rule_ids`, check `stored_incident.is_disclosed`. If the incident is not disclosed and the caller is `RestrictedRead`, return `NotFound` or return only the subset of rules that are individually disclosed.

3. **`ConfigConfidentialityFormatter::format` (`confidentiality_formatting.rs:17-28`):** Optionally redact `incident_id` for non-disclosed rules to prevent incident ID harvesting at the `get_config` level.

## Proof of Concept

```rust
// State setup (privileged add_config calls)
add_config([rule R linked to incident I])   // → version V, active_since T_V
add_config([])                              // → version V+1, active_since T_V1 (rule R removed)

// Attacker: anonymous / RestrictedRead, non-replicated query calls

// Step 1: discover incident ID I from get_config
let cfg = get_config(Some(V));
// cfg.config.rules[0].incident_id == I  (not redacted)
// cfg.config.rules[0].rule_raw == None  (redacted)

// Step 2: retrieve full lifecycle of incident I
let rules = get_rules_by_incident_id(I);
// rules[0] == OutputRuleMetadata {
//   rule_id: <uuid>,          // not redacted
//   incident_id: I,           // not redacted
//   rule_raw: None,           // redacted
//   description: None,        // redacted
//   added_in_version: V,      // NOT redacted ← leak
//   removed_in_version: V+1,  // NOT redacted ← leak
// }

// Step 3: correlate versions to wall-clock time
let t_v  = get_config(Some(V)).active_since;    // = T_V
let t_v1 = get_config(Some(V+1)).active_since;  // = T_V1
// Attacker knows: rule R was active from T_V to T_V1
```

This can be reproduced as a deterministic PocketIC integration test by initializing the canister, calling `add_config` twice as the authorized principal, then calling `get_config` and `get_rules_by_incident_id` as `Principal::anonymous()` and asserting that `added_in_version` and `removed_in_version` are non-null in the response.
