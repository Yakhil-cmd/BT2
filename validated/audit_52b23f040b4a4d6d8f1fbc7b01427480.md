### Title
Stray CAT On-Chain Name Trusted Without Validation to Create Spoofed Wallet Identity — (`packages/wallets/src/hooks/useWalletsList.ts`)

---

### Summary

The attack path is real and fully traceable through production code. An unprivileged attacker who sends any CAT token with an attacker-chosen `name` field (e.g., `"USDS"`, `"Chia"`) to a victim's address causes that name to flow, unsanitized, from the `get_stray_cats` RPC response all the way into `createWalletForExisting` + `cat_set_name`, creating a persistent wallet entry with the attacker-controlled identity.

---

### Finding Description

**Data source — `getStrayCats` RPC transform:**

The `getStrayCats` query in `wallet.ts` passes the stray cat's `name` field through verbatim from the daemon RPC response. Only `assetId` and `senderPuzzleHash` are normalized (hex); the `name` is untouched. [1](#0-0) 

**List construction — `useWalletsList.ts`:**

`nonAddedStrayCats` are built directly from the RPC response. The `name` field used in the list item comes from `getCATName()`, which falls through to `strayCAT?.name` when no wallet exists yet for that assetId. [2](#0-1) [3](#0-2) 

**Wallet creation — `handleShow`:**

When the user toggles the visibility switch for a stray cat that has no existing wallet, `handleShow` is called with the `assetId` string. It first checks `catList` (by assetId — no match since the attacker's assetId differs from any legitimate token's assetId), then falls through to the stray cat branch and calls `addCATToken` with `strayCat.name` and `strayCat.assetId` — both attacker-controlled. [4](#0-3) 

**`addCATToken` mutation:**

This calls `createWalletForExisting` (creates a CAT wallet for the attacker's assetId) and then `setName` (persists the attacker-controlled name as the wallet's display name). [5](#0-4) 

There is **no guard** anywhere in this chain that:
- Validates `strayCat.name` against a blocklist of reserved names (`"Chia"`, known CAT names, etc.)
- Warns the user that the name is attacker-supplied on-chain data
- Checks whether the name collides with an existing wallet's identity

---

### Impact Explanation

After the user toggles the stray cat visible, a persistent CAT wallet entry named e.g. `"USDS"` backed by the attacker's assetId exists in the wallet list. Concrete downstream harms:

1. **Wrong-asset offer creation**: The user opens the offer builder, sees `"USDS"` in their CAT wallet list, selects it, and creates an offer to sell `"USDS"` — actually offering the attacker's worthless token. A counterparty who does not verify the assetId on-chain could accept it.
2. **Wrong-asset receive**: The user shares the receive address of the fake `"USDS"` wallet expecting to receive real USDS, directing a sender to fund the wrong asset.
3. **Balance confusion**: The user believes they hold a legitimate token and makes financial decisions based on a fabricated identity.

This fits the defined High scope: *"unsafe trust of RPC … state that causes a user to … import, sign, send … or display the wrong asset, identity … or status."*

---

### Likelihood Explanation

- Sending a CAT with an arbitrary name to any address is a standard, zero-cost on-chain operation requiring no special access.
- The stray cat appears automatically in the "Manage token list" UI without any user opt-in to receive it.
- The only required user action is toggling the visibility switch — a natural action when a user sees an unexpected token they believe they received legitimately.
- No existing UI warning distinguishes attacker-named stray cats from legitimate ones.

---

### Recommendation

1. **Blocklist reserved names**: In `handleShow`, before calling `addCATToken`, reject or warn if `strayCat.name` matches any name in `catList` (the known-good list) or equals `"Chia"` / `"XCH"`.
2. **Display provenance warning**: In `WalletTokenCard`, visually distinguish `STRAY_CAT` entries with a warning that the name is unverified on-chain data, not from the official CAT list.
3. **Use assetId as default name for stray cats**: Do not use `strayCat.name` as the wallet name at all; default to the assetId and let the user rename it manually after verifying the token's identity.

---

### Proof of Concept

1. Attacker mints a CAT with `name = "USDS"` and sends 1 mojo to the victim's address.
2. Victim opens "Manage token list" — sees `"USDS"` listed as a stray cat with the attacker's assetId shown as subtitle.
3. Victim toggles the switch to enable it.
4. `handleShow(attacker_assetId)` → `strayCats.find()` returns `{ name: "USDS", assetId: attacker_assetId }` → `addCATToken({ name: "USDS", assetId: attacker_assetId })` → `createWalletForExisting` + `cat_set_name`.
5. A wallet named `"USDS"` backed by `attacker_assetId` now appears in the victim's wallet list.
6. **Assert**: `wallet.meta.assetId !== real_USDS_assetId` — the wallet identity is spoofed.
