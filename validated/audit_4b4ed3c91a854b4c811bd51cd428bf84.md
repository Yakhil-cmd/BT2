### Title
Missing Minimum Threshold Enforcement in FROST Signing Allows Malicious Coordinator to Corrupt Sign Output — (File: `src/frost/mod.rs`)

---

### Summary

The `assert_sign_inputs` function in `src/frost/mod.rs` validates signing parameters for FROST EdDSA and RedJubjub but does not enforce a minimum threshold of 2. A malicious coordinator can invoke the signing protocol with `threshold = 1`, which is accepted without error. Because the DKG phase enforces `threshold >= 2`, the Lagrange interpolation during signing uses an incorrect reconstruction degree, producing an invalid, unusable signature. Honest participants cannot detect the mismatch at the validation boundary and complete the protocol with a corrupted output.

---

### Finding Description

The `assert_sign_inputs` function performs the following checks: [1](#0-0) 

It rejects `participants.len() < 2` and `threshold > participants.len()`, but **never rejects `threshold < 2`**. The `ReconstructionLowerBound` type is a plain `usize` wrapper with no enforced lower bound: [2](#0-1) 

By contrast, the DKG path (`assert_key_invariants`, called from `assert_reshare_keys_invariants`) enforces `threshold >= 2` and returns `InitializationError::ThresholdTooSmall { threshold, min: 2 }` for any smaller value: [3](#0-2) 

Similarly, OT-based triple generation enforces the same lower bound: [4](#0-3) 

The FROST signing path has no equivalent guard. A coordinator who calls the signing entry point with `threshold = 1` passes all validation in `assert_sign_inputs` and proceeds into the protocol. Each participant then computes Lagrange coefficients for a degree-0 (constant) polynomial reconstruction, which is mathematically inconsistent with the degree-(t−1) polynomial used during DKG. The combined signature share is therefore computed with wrong coefficients, and the resulting `(R, z)` pair fails standard ECDSA/EdDSA verification. The output is an unusable cryptographic object.

The `presign` function for FROST has the same gap: [5](#0-4) 

No minimum threshold check appears there either.

---

### Impact Explanation

A malicious coordinator supplies `threshold = 1` to the FROST signing entry point. All honest participants complete their rounds and produce signature shares. The coordinator aggregates them under the wrong Lagrange basis. The resulting signature is invalid and cannot be used by any downstream consumer. Honest parties have expended nonces and communication rounds and received a corrupted, unusable signing output. This matches **High: Corruption of sign outputs so honest parties accept unusable cryptographic outputs**.

---

### Likelihood Explanation

The coordinator role is reachable by any participant designated to orchestrate a signing session. No key material, cryptographic break, or external compromise is required — only the ability to call the public signing API with a caller-controlled `threshold` argument. The `ReconstructionLowerBound` type imposes no compile-time or runtime lower bound, so passing `1` is trivially possible.

---

### Recommendation

Add a minimum threshold check in `assert_sign_inputs` (and in the `presign` function) mirroring the guard already present in DKG and triple generation:

```rust
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

This is the direct analog of the VDF fix: just as the VDF verifier was patched to require `t >= 2`, the FROST signing validator should require `threshold >= 2`.

---

### Proof of Concept

1. Run DKG with `n = 3` participants and `threshold = 2` (accepted, produces valid key shares).
2. Call the FROST signing entry point with the same participants but `threshold = 1`.
3. `assert_sign_inputs` accepts the call (no `ThresholdTooSmall` error).
4. Each participant computes their signature share using Lagrange coefficient `λ_i = 1` (trivial, degree-0 basis) instead of the correct degree-1 Lagrange coefficient derived from the DKG polynomial.
5. The coordinator aggregates: `z = Σ z_i` under the wrong basis.
6. Signature verification of `(R, z)` against the public key produced by DKG fails — the output is an unusable cryptographic object. [6](#0-5)

    participants: &[Participant],
    threshold: impl Into<ReconstructionLowerBound>,
    me: Participant,
    coordinator: Participant,
) -> Result<ParticipantList, InitializationError> {
    let threshold = threshold.into();
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    if threshold.value() > participants.len() {
        return Err(InitializationError::ThresholdTooLarge {
            threshold: threshold.value(),
            max: participants.len(),
        });

            role: "coordinator",
            participant: coordinator,
        });
