### Title
`receive_erc20_tokens` Mints ERC-20 Tokens Based on Sender-Specified Amount, Not Actual Received Amount for Fee-on-Transfer NEP-141 Tokens, Causing Insolvency - (`engine/src/engine.rs`)

---

### Summary

When a fee-on-transfer NEP-141 token is bridged into Aurora via `ft_on_transfer`, the engine mints ERC-20 mirror tokens equal to `args.amount` — the amount the sender *specified* — rather than the amount Aurora *actually received* after the transfer fee is deducted. This creates a permanent accounting surplus of ERC-20 tokens relative to the NEP-141 tokens held by Aurora, leading to insolvency when users attempt to exit.

---

### Finding Description

The NEAR NEP-141 standard allows tokens to implement transfer fees. When a user calls `ft_transfer_call` on a fee-on-transfer NEP-141 token with Aurora as the receiver, the NEP-141 contract:

1. Deducts a fee from the transferred amount.
2. Credits Aurora with `amount - fee` tokens.
3. Calls `ft_on_transfer` on Aurora with the **original** `amount` (not the post-fee amount).

Aurora's `ft_on_transfer` handler dispatches to `receive_erc20_tokens`:

```rust
// engine/src/contract_methods/connector.rs, line 84
engine.receive_erc20_tokens(
    &predecessor_account_id,
    &args,
    &current_account_id,
    handler,
)
```

Inside `receive_erc20_tokens`, the mint amount is taken directly from `args.amount`:

```rust
// engine/src/engine.rs, line 803
let amount = args.amount.as_u128();
// ...
setup_receive_erc20_tokens_input(&recipient, amount),
```

`setup_receive_erc20_tokens_input` encodes a call to the ERC-20 `mint(address, uint256)` selector with the full `amount`:

```rust
// engine/src/engine.rs, lines 1306-1314
pub fn setup_receive_erc20_tokens_input(recipient: &Address, amount: u128) -> Vec<u8> {
    let selector = ERC20_MINT_SELECTOR;
    let tail = ethabi::encode(&[
        ethabi::Token::Address(recipient.raw().0.into()),
        ethabi::Token::Uint(amount.into()),
    ]);
    [selector, tail.as_slice()].concat()
}
```

The `EvmErc20.mint` function mints exactly `amount` tokens:

```solidity
// etc/eth-contracts/contracts/EvmErc20.sol, line 49-51
function mint(address account, uint256 amount) public onlyAdmin {
    _mint(account, amount);
}
```

There is no mechanism to reconcile the minted ERC-20 amount against the actual NEP-141 balance change Aurora experienced. The discrepancy accumulates with every bridging operation involving a fee-on-transfer NEP-141 token.

---

### Impact Explanation

**Insolvency (Critical).** Aurora's ERC-20 mirror token supply will exceed the NEP-141 tokens Aurora actually holds. When users call `withdrawToNear` (which burns ERC-20 tokens and triggers `ft_transfer` of NEP-141 tokens back to NEAR), the last users to exit will find Aurora unable to fulfill their withdrawal — the NEP-141 balance is exhausted before all ERC-20 tokens are redeemed. This is a direct protocol insolvency.

---

### Likelihood Explanation

Any NEP-141 token that implements a transfer fee and is registered with Aurora via `deploy_erc20_token` triggers this path. The entry point (`ft_on_transfer`) is a standard NEAR callback invoked automatically by the NEP-141 contract — no special attacker action is required beyond initiating a normal `ft_transfer_call`. The discrepancy is permanent and cumulative.

---

### Recommendation

In `receive_erc20_tokens`, instead of using `args.amount` directly, Aurora should measure its actual NEP-141 balance change before and after the transfer callback resolves, and mint only the net received amount. Since `ft_on_transfer` is a callback (the transfer has already occurred by the time it is called), the correct approach is to query the NEP-141 balance of Aurora before and after, or to accept the amount reported by the NEP-141 contract only if the token is known to be non-fee-bearing. At minimum, the protocol should reject registration of fee-on-transfer NEP-141 tokens, or document and enforce this restriction.

---

### Proof of Concept

1. Deploy a NEP-141 token with a 1% transfer fee on NEAR.
2. Register it with Aurora via `deploy_erc20_token` — this creates an ERC-20 mirror.
3. Call `ft_transfer_call` on the NEP-141 contract with `amount = 1000`, receiver = Aurora, msg = `<evm_address>`.
4. The NEP-141 contract deducts 10 tokens as fee, credits Aurora with 990 NEP-141 tokens, and calls `ft_on_transfer` on Aurora with `args.amount = 1000`.
5. Aurora's `receive_erc20_tokens` mints **1000** ERC-20 tokens to `<evm_address>`.
6. Aurora holds only **990** NEP-141 tokens but has issued **1000** ERC-20 tokens.
7. Repeat: after N users bridge in, the ERC-20 supply exceeds NEP-141 holdings by `N * fee`. The last users to `withdrawToNear` will find Aurora's NEP-141 balance insufficient, permanently freezing their funds.
