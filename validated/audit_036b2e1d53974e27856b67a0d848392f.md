### Title
`native_per_pubdata` Truncates to Zero When `pubdata_price < native_price`, Allowing Pubdata Generation Without Payment - (File: `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

When the operator-set `pubdata_price` is smaller than `native_price`, the floor division used to compute `native_per_pubdata` silently truncates to zero. This makes all pubdata (L1 state-diff data) effectively free for any transaction sender, while the operator still bears the actual L1 data-availability cost. The root cause is a direct analog of the Teleporter bridge's `scaleTokens` truncation-to-zero: a denomination-ratio division that silently floors to zero when the numerator is smaller than the denominator.

---

### Finding Description

In `validate_and_compute_fee_for_transaction` (L2 transactions) the ratio of pubdata cost to native cost is computed as:

```rust
// basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs, line 142
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

`wrapping_div` on unsigned integers is plain floor (truncating) division. When `pubdata_price < native_price` — a realistic condition whenever L1 calldata/blob fees are low relative to the proving cost — the result is **0**. [1](#0-0) 

The same pattern is replicated in the public API helper:

```rust
// api/src/helpers.rs, line 427
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
``` [2](#0-1) 

Once `native_per_pubdata = 0`, the pubdata charging function `get_resources_to_charge_for_pubdata` returns a zero-cost resource object for any amount of pubdata:

```rust
// basic_bootloader/src/bootloader/transaction_flow/gas_helpers.rs
let native = current_pubdata_spent
    .checked_mul(native_per_pubdata)   // 0 * anything = 0
    .ok_or(out_of_native_resources!())?;
``` [3](#0-2) 

Consequently `check_enough_resources_for_pubdata` always returns `enough = true`, and the transaction is never charged for pubdata regardless of how many storage slots it writes. [4](#0-3) 

The documentation explicitly states that `native_per_pubdata` is the per-byte pubdata cost and that pubdata is charged from native resources after execution: [5](#0-4) 

---

### Impact Explanation

When `native_per_pubdata = 0`:

1. A transaction can write to an arbitrary number of storage slots (generating maximum pubdata) and pay only for EVM gas and computational native resources.
2. The operator must still publish all state diffs to L1 (calldata or blobs), paying the actual L1 data-availability cost.
3. The operator collects no pubdata fee from the user to offset this cost, resulting in a direct financial loss proportional to the pubdata generated.

The imbalance is structurally identical to the Teleporter bridge bug: the "source" (user) pays nothing for a resource that the "destination" (operator/L1) must still provide, because the denomination-ratio division silently truncated to zero.

---

### Likelihood Explanation

The condition `pubdata_price < native_price` is realistic and not pathological:

- `native_price` reflects the cost of a single RISC-V proving cycle (set by the operator, typically in the range of hundreds to thousands of wei).
- `pubdata_price` reflects the current L1 calldata/blob cost per byte, which fluctuates with L1 gas prices and can be lower than `native_price` during low-activity L1 periods.
- The operator may intentionally set `pubdata_price = 1` (non-zero, intending to charge something) while `native_price = 10`, expecting a ratio of 0.1 — but the floor division silently produces 0.
- There is no minimum-ratio guard or warning in the code.

An attacker monitoring block metadata can detect when `pubdata_price < native_price` and immediately submit storage-heavy transactions to exploit the free-pubdata window.

---

### Recommendation

Replace the floor division with ceiling division (matching the pattern already used for `native_per_gas`):

```rust
// Current (floor, can produce 0):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))...;

// Fix (ceiling, consistent with native_per_gas):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))...;
``` [6](#0-5) 

Alternatively, add an explicit guard that rejects (or warns on) a zero `native_per_pubdata` when `pubdata_price > 0`, mirroring the existing `native_price != 0` guard. The same fix must be applied to `api/src/helpers.rs` line 427.

---

### Proof of Concept

**Setup**: operator sets `native_price = 10`, `pubdata_price = 9` (both non-zero; operator intends to charge for pubdata).

**Computation**:
```
native_per_pubdata = 9 / 10 = 0  (floor division)
```

**Attacker transaction**: writes to 1000 distinct storage slots (generating ~32 000 bytes of pubdata).

**Charge collected from attacker**: `1000 * 32 * 0 = 0` native units for pubdata.

**Operator cost**: must publish ~32 000 bytes to L1 at `pubdata_price = 9` per byte = 288 000 units of L1 cost, with zero reimbursement from the transaction fee.

The condition is reachable via any standard L2 transaction when the block context satisfies `pubdata_price < native_price`, which requires no privileged access — only observation of the current block metadata. [7](#0-6) [8](#0-7)

pub fn get_resources_to_charge_for_pubdata<S: EthereumLikeTypes>(
    system: &mut System<S>,
    native_per_pubdata: u64,
    base_pubdata: Option<u64>,
) -> Result<(u64, S::Resources), SystemError> {
    let current_pubdata_spent = system
        .net_pubdata_used()?
        .saturating_sub(base_pubdata.unwrap_or(0));
    let native = current_pubdata_spent
        .checked_mul(native_per_pubdata)
        .ok_or(out_of_native_resources!())?;
    let native = <S::Resources as zk_ee::system::Resources>::Native::from_computational(native);
    Ok((current_pubdata_spent, S::Resources::from_native(native)))
}

    base_pubdata: Option<u64>,
