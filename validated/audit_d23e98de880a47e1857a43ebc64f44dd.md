### Title
`DelegateAction` Signed Payload Lacks Chain-ID Domain Separation, Enabling Cross-Network Replay — (`core/primitives/src/action/delegate.rs`)

---

### Summary

The `DelegateAction` (NEP-366 meta-transaction) signed payload commits only to a NEP-number discriminant plus the action body fields (`sender_id`, `receiver_id`, `actions`, `nonce`, `max_block_height`, `public_key`). It does **not** commit to any network identifier (`chain_id`, genesis hash, or genesis block hash). A `SignedDelegateAction` produced for testnet is therefore cryptographically valid on mainnet — and vice versa — whenever the same account and key exist on both networks and the nonce is acceptable on the target chain.

---

### Finding Description

`DelegateAction::get_nep461_hash()` constructs the signed digest as:

```
SHA-256( borsh(discriminant) || borsh(DelegateAction) )
```

where `discriminant` is the constant `1<<30 + 366` (the NEP-366 on-chain tag) and `DelegateAction` is:

```rust
pub struct DelegateAction {
    pub sender_id: AccountId,
    pub receiver_id: AccountId,
    pub actions: Vec<NonDelegateAction>,
    pub nonce: Nonce,
    pub max_block_height: BlockHeight,
    pub public_key: PublicKey,
}
``` [1](#0-0) 

No `chain_id`, genesis hash, or any other network-scoped field is included in the preimage. [2](#0-1) 

The `SignableMessage` wrapper adds only the NEP discriminant — a protocol-type tag, not a network identifier:

```rust
pub struct SignableMessage<'a, T> {
    pub discriminant: MessageDiscriminant,   // constant: NEP-366 number
    pub msg: &'a T,
}
``` [3](#0-2) 

`apply_delegate_action` verifies the signature and checks `max_block_height`, `sender_id`, nonce, and key existence — but performs no chain-identity check:

```rust
if !signed_delegate_action.verify() { … }
if apply_state.block_height > delegate_action.max_block_height() { … }
if delegate_action.sender_id().as_str() != sender_id.as_str() { … }
validate_delegate_action_key(…);
``` [4](#0-3) 

Regular `SignedTransaction` avoids this problem because it commits to a `block_hash` that must be a recent block on the **current** chain, making cross-chain replay structurally impossible. `DelegateAction` has no equivalent anchor.

---

### Impact Explanation

An attacker who obtains a `SignedDelegateAction` produced for one NEAR network (e.g., testnet) can wrap it in a fresh outer transaction and submit it on a different network (e.g., mainnet). If Alice's account exists on mainnet with the same key and the DelegateAction's nonce is greater than Alice's current mainnet key nonce, `apply_delegate_action` accepts the signature and emits a new action receipt executing the inner actions with `predecessor_id = alice`. Those inner actions may include:

- `Transfer` — drains Alice's mainnet NEAR balance
- `FunctionCall` — calls arbitrary contract methods (e.g., `ft_transfer`) as Alice
- `AddKey` / `DeleteKey` — installs attacker-controlled keys or removes Alice's keys

The corrupted values are Alice's on-chain account state (balance, storage, access keys) on the target network. The attacker pays only the relayer gas cost on the target network; Alice pays nothing and has no recourse.

---

### Likelihood Explanation

The preconditions are realistic:

1. **Same account on both networks** — NEAR account IDs are human-readable strings; users routinely register the same name on mainnet and testnet.
2. **Same key on both networks** — wallets (NEAR Wallet, Ledger, etc.) commonly derive or reuse the same key pair across networks.
3. **Valid nonce on target network** — if Alice has used her testnet key more than her mainnet key (common during development), the testnet DelegateAction nonce will exceed the mainnet key nonce, satisfying the `> current_nonce` check.
4. **Attacker can observe the signed payload** — once Alice submits the DelegateAction to a testnet relayer, the `SignedDelegateAction` is visible in the testnet transaction. Any observer can extract and replay it.
5. **Attacker can act as their own relayer** — submitting a transaction on mainnet requires no special role; any funded mainnet account suffices.

---

### Recommendation

Include the network's `chain_id` (available from `GenesisConfig::chain_id` / `ApplyState`) in the signed preimage of `DelegateAction` and `DelegateActionV2`. The minimal fix is to add `chain_id` to both structs and to `get_nep461_hash()`:

```rust
pub fn get_nep461_hash(&self, chain_id: &str) -> CryptoHash {
    // chain_id binds the signature to a specific network
    let signable = SignableMessage::new(
        &(chain_id, &self),
        SignableMessageType::DelegateAction,
    );
    let bytes = borsh::to_vec(&signable).expect("serialization failed");
    hash(&bytes)
}
```

Alternatively, follow the EIP-712 pattern and include the genesis block hash (which is unique per network) as a domain separator. Either approach must be gated behind a new `ProtocolFeature` to avoid breaking existing signed delegate actions in flight.

---

### Proof of Concept

1. Alice holds account `alice.near` on both testnet and mainnet, using the same ED25519 key. Her mainnet key nonce is 10; her testnet key nonce is 15.
2. Alice signs a testnet `DelegateAction` with `nonce = 16`, `max_block_height = 200_000_000`, `actions = [FunctionCall { method: "ft_transfer", args: {receiver: attacker, amount: 1000}, deposit: 1 }]` targeting a testnet FT contract.
3. Alice sends the `SignedDelegateAction` to a testnet relayer. The relayer broadcasts it; it appears in a testnet transaction.
4. The attacker extracts the identical `SignedDelegateAction` bytes from the testnet transaction.
5. The attacker constructs a mainnet `SignedTransaction` from their own mainnet account to `alice.near`, containing `Action::Delegate(signed_delegate_action)`.
6. `apply_delegate_action` on mainnet:
   - `verify()` → passes (same key, same payload hash, no chain_id in preimage). [5](#0-4) 
   - `block_height ≤ max_block_height` → passes (mainnet height < 200_000_000).
   - `sender_id == alice.near` → passes.
   - `validate_delegate_action_key`: nonce 16 > current mainnet nonce 10 → passes; key exists → passes.
7. A new action receipt is emitted on mainnet with `predecessor_id = alice.near`, executing `ft_transfer` on the mainnet FT contract, transferring Alice's mainnet FT balance to the attacker.
