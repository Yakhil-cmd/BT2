### Title
Broken Underflow Guard in `DeltaView.SubBalance` Silently Corrupts EVM Account Balance — (File: `fvm/evm/emulator/state/delta.go`)

---

### Summary

`DeltaView.SubBalance` uses `newBalance.Sign() < 0` to detect arithmetic underflow after calling `uint256.Int.Sub`. Because `uint256.Int` is an **unsigned** 256-bit integer type, its `Sign()` method never returns a negative value — it returns only `0` or `1`. The guard is therefore permanently dead code. When `amount > orgBalance`, the subtraction wraps modulo 2^256 to an astronomically large value, which is silently committed to the EVM account state instead of returning an error.

---

### Finding Description

In `fvm/evm/emulator/state/delta.go`, `DeltaView.SubBalance` performs:

```go
newBalance := new(uint256.Int).Sub(orgBalance, amount)

// if new balance is negative error
if newBalance.Sign() < 0 {
    return fmt.Errorf("account balance is negative %d", newBalance)
}
``` [1](#0-0) 

The import confirms the type is `github.com/holiman/uint256`: [2](#0-1) 

`holiman/uint256.Int` is a `[4]uint64` — always non-negative. Its `Sign()` method returns `0` for zero and `1` for any non-zero value; it **never** returns `-1`. Therefore `newBalance.Sign() < 0` is always `false`, and the underflow check is unreachable.

When `orgBalance < amount`, `uint256.Int.Sub` wraps around modulo 2^256, producing a value near `2^256`. This value is stored directly into `d.balances[addr]` without error.

The project's own test at `delta_test.go` line 167 documents the intended behavior — it expects an error when subtracting more than the balance — but the check as written cannot produce that error: [3](#0-2) 

The `StateDB.SubBalance` wrapper calls `DeltaView.SubBalance` and propagates its error via `db.handleError(err)`. Since no error is ever returned on underflow, `handleError` is never invoked for this case: [4](#0-3) 

---

### Impact Explanation

The direct call path in `emulator.go` `withdrawFrom` calls `proc.state.SubBalance(bridge, value, ...)` on the bridge account **after** the EVM transaction runs: [5](#0-4) 

If the bridge account's balance is less than `value` at that point (e.g., due to a Solidity contract in the EVM transaction draining the bridge mid-execution, or any future code path that bypasses `CanTransfer`), the balance wraps to `2^256 - delta` and is committed to the EVM state. An account holding this corrupted balance could subsequently withdraw far more FLOW tokens than were ever deposited — a cross-VM bridge escrow mis-accounting leading to unauthorized asset extraction from the Cadence side.

The correct underflow guard used elsewhere in the codebase (e.g., `types.SubBalance`) uses `big.Int.Cmp` before subtracting: [6](#0-5) 

---

### Likelihood Explanation

The EVM's own `CanTransfer` check provides a primary guard for standard EVM transactions. However, `DeltaView.SubBalance` is also called directly by Flow-specific bridge logic (`withdrawFrom`) outside the normal EVM transfer path, where `CanTransfer` is not invoked. Any scenario where the bridge account's balance is less than the requested withdrawal amount — reachable by a Cadence transaction author submitting a `DirectCall` — would silently corrupt state. The broken guard also means any future code path added to the EVM subsystem that calls `SubBalance` directly has no safety net.

---

### Recommendation

Replace the dead `Sign() < 0` check with a proper pre-subtraction comparison, matching the pattern used in `types.SubBalance`:

```go
func (d *DeltaView) SubBalance(addr gethCommon.Address, amount *uint256.Int) error {
    if amount.Sign() == 0 {
        return nil
    }
    orgBalance, err := d.GetBalance(addr)
    if err != nil {
        return err
    }
    // Correct underflow guard: compare before subtracting
    if orgBalance.Cmp(amount) < 0 {
        return fmt.Errorf("account balance is negative")
    }
    newBalance := new(uint256.Int).Sub(orgBalance, amount)
    d.balances[addr] = newBalance
    d.dirtyAddresses[addr] = struct{}{}
    return nil
}
```

Alternatively, use `uint256.Int.SubOverflow` which returns a boolean carry flag indicating underflow.

---

### Proof of Concept

1. The existing unit test at `delta_test.go:167` already encodes the proof: it calls `view.SubBalance(addr2, uint256.NewInt(100))` when `addr2` has balance `3`, and asserts `require.Error(t, err)`. With the current implementation, `newBalance.Sign() < 0` is always false, so no error is returned and the test assertion fails — the balance is silently set to `2^256 - 97`.

2. On-chain: a Cadence transaction author submits a `DirectCall` of type `WithdrawCall` targeting an EVM address whose balance is less than the requested amount. The EVM transaction itself may fail (caught by `res.Failed()`), but if the bridge account's balance is manipulated to be less than `value` before the `SubBalance` call at line 471 of `emulator.go`, the bridge account receives a wrapped-around balance of `2^256 - delta`, which can then be used to drain the Flow-side vault. // SubBalance subtracts the amount from the current balance of the given address
func (d *DeltaView) SubBalance(addr gethCommon.Address, amount *uint256.Int) error {
	// if amount is 0 skip
	if amount.Sign() == 0 {
		return nil
	}

	// get the latest balance
	orgBalance, err := d.GetBalance(addr)
	if err != nil {
		return err
	}

	// update the new balance
	newBalance := new(uint256.Int).Sub(orgBalance, amount)

	// if new balance is negative error
	if newBalance.Sign() < 0 {
		return fmt.Errorf("account balance is negative %d", newBalance)
	}

	// update the balance
	d.balances[addr] = newBalance

	// flag the address as dirty
	d.dirtyAddresses[addr] = struct{}{}
	return nil
}

func (db *StateDB) SubBalance(
	addr gethCommon.Address,
	amount *uint256.Int,
	reason gethTracing.BalanceChangeReason,
) uint256.Int {
	// negative amounts are not accepted.
	if amount.Sign() < 0 {
		db.handleError(types.ErrInvalidBalance)
		return uint256.Int{}
	}
	prevBalance, err := db.latestView().GetBalance(addr)
	db.handleError(err)

	err = db.latestView().SubBalance(addr, amount)
	db.handleError(err)

	return *prevBalance
}

func (proc *procedure) withdrawFrom(
	call *types.DirectCall,
) (*types.Result, error) {
	// check and convert value
	value, isValid := checkAndConvertValue(call.Value)
	if !isValid {
		return types.NewInvalidResult(
			call.Type,
			call.Hash(),
			types.ErrInvalidBalance,
		), nil
	}

	// check balance is not prone to rounding error
	if !types.AttoFlowBalanceIsValidForFlowVault(call.Value) {
		return types.NewInvalidResult(
			call.Type,
			call.Hash(),
			types.ErrWithdrawBalanceRounding,
		), nil
	}

	// create bridge account if not exist
	bridge := call.To.ToCommon()
	if !proc.state.Exist(bridge) {
		proc.state.CreateAccount(bridge)
	}

	// withdraw the amount and move it to the bridge account
	res, err := proc.run(call.Message(), call.Hash(), types.DirectCallTxType)
	if err != nil {
		return res, err
	}

	// if any error (invalid or vm) on the internal call, revert and don't commit any change
	// this prevents having cases that we deduct the balance from the account
	// but doesn't return it as a vault.
	if res.Invalid() || res.Failed() {
		// reset the state to revert the add balances
		proc.state.Reset()
		return res, nil
	}

	// now deduct the balance from the bridge
	proc.state.SubBalance(bridge, value, gethTracing.BalanceIncreaseWithdrawal)

	// commit and finalize the state and return any stateDB error
	res.StateChangeCommitment, err = proc.commit(true)
	return res, err
}
