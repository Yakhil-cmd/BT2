### Title
Missing Storage Collision Check in CREATE/CREATE2 Deployment Allows Deployment to Address with Pre-existing Storage — (`evm_interpreter/src/ee_trait_impl.rs`)

### Summary

ZKsync OS's EVM interpreter omits the EIP-684 storage-non-empty collision check during CREATE/CREATE2 deployment. The check in `before_executing_frame` only guards against non-zero `code_len` or `nonce`, but explicitly skips the storage check. This is a self-acknowledged deviation from EVM semantics, confirmed by disabled test cases across all four test index files. An attacker can exploit this within a single transaction to deploy a contract at a CREATE2 address that already has non-empty storage (e.g., after a same-transaction SELFDESTRUCT), causing the new contract to inherit unexpected storage values that Ethereum would have prevented.

---

### Finding Description

In `evm_interpreter/src/ee_trait_impl.rs`, the `before_executing_frame` function performs the CREATE/CREATE2 collision check:

```rust
// Check there's no contract already deployed at this address.
// NB: EVM also specifies that the address should have empty storage,
// but we cannot perform such a check for now.
if deployee_code_len != 0 || deployee_nonce != 0 {
    // ... burn gas, emit CreateCollision, return false
}
```

The comment explicitly acknowledges the missing storage check. EIP-684 (enforced since Spurious Dragon, reaffirmed in Paris) requires that CREATE/CREATE2 also fails if the target address has non-empty storage. ZKsync OS skips this third condition entirely.

This is confirmed by four test index files, all of which disable the relevant Ethereum test cases with explicit comments:

- `create2collisionStorageParis.json` → `enabled: false`, `comment: We do not check storage for collision`
- `RevertInCreateInInitCreate2Paris.json` → `enabled: false`, `comment: We do not check for storage collisions`

The concrete exploit path within a single transaction:

1. A contract calls `CREATE2` with a known salt, deploying a child contract at address `A`.
2. The child contract's constructor writes to one or more storage slots.
3. The child contract calls `SELFDESTRUCT` within the same constructor (or the parent calls it). Pre-Cancun semantics: code and nonce are cleared at end of transaction, but storage clearing is deferred. Within the same transaction, the address has `nonce=0`, `code_len=0`, but non-empty storage.
4. The parent contract calls `CREATE2` again with the same salt, targeting address `A`.
5. **On Ethereum**: the second CREATE2 fails (returns 0) because storage is non-empty — EIP-684.
6. **On ZKsync OS**: the second CREATE2 succeeds because only `code_len` and `nonce` are checked. The new contract is deployed at `A` and inherits the storage written in step 2.

---

### Impact Explanation

**EVM semantic mismatch / state-transition divergence**: ZKsync OS produces a different post-state than Ethereum for the same transaction. On Ethereum the second CREATE2 returns `0` (failure) and the caller's logic branches accordingly. On ZKsync OS the deployment succeeds and the new contract starts with unexpected, attacker-influenced storage values.

Concrete consequences:
- A contract relying on the CREATE2 + SELFDESTRUCT + re-deploy upgrade pattern (a known pattern for upgradeable contracts) will have its re-deployed instance inherit stale or attacker-written storage, corrupting its invariants.
- Any contract that checks `CREATE2` return value to detect collision and takes a security-critical branch will behave incorrectly on ZKsync OS.
- Funds held or managed by the re-deployed contract can be stolen or locked if the inherited storage corrupts access-control mappings, balance accounting, or ownership variables.

---

### Likelihood Explanation

The attack requires crafting a single transaction that: deploys via CREATE2, writes storage, self-destructs, and re-deploys at the same address. This is a well-known EVM pattern (used in upgradeable proxy designs and factory contracts). Any contract that uses this pattern and is ported to ZKsync OS is affected. The attacker does not need any privileged role — a standard EOA transaction is sufficient.

---

### Recommendation

Implement the storage-non-empty collision check in `before_executing_frame`. Before allowing a constructor frame to execute, read the storage root (or check whether any storage slot is non-zero) for the target address. If storage is non-empty, treat it as a collision: burn all gas and return `false`, consistent with the existing `code_len`/`nonce` collision path.

If a full storage-root check is too expensive at the current architecture level, the minimum fix is to track whether any storage write has ever been made to the address (e.g., via a dirty-bit in the account cache) and use that as the collision signal.
