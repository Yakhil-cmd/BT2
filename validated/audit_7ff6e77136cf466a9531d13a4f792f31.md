### Title
Authz-Enabled Overwrite of Victim's E2EE Encryption Key via Missing Signer Check — (`x/e2ee/keeper/keeper.go`)

### Summary

`Keeper.RegisterEncryptionKey` performs no check that the transaction signer matches `req.Address`. Because `MsgRegisterEncryptionKey` is absent from `DisabledAuthzMsgs`, an attacker holding an authz grant from a victim can execute `MsgExec([MsgRegisterEncryptionKey{address=victim, key=attacker_key}])` and silently replace the victim's on-chain encryption key with an attacker-controlled value.

---

### Finding Description

**Root cause — no signer guard in the keeper:**

`RegisterEncryptionKey` blindly writes whatever `req.Address` and `req.Key` contain:

```go
func (k Keeper) RegisterEncryptionKey(
    ctx context.Context,
    req *types.MsgRegisterEncryptionKey,
) (*types.MsgRegisterEncryptionKeyResponse, error) {
    if err := k.registerEncryptionKey(ctx, req.Address, []byte(req.Key)); err != nil {
        return nil, err
    }
    return &types.MsgRegisterEncryptionKeyResponse{}, nil
}
``` [1](#0-0) 

There is no `sdk.UnwrapSDKContext(ctx).EventManager()` signer retrieval, no comparison of the signing address to `req.Address`, and no authority check of any kind.

**Proto declares `address` as the signer field:**

```proto
message MsgRegisterEncryptionKey {
  option (cosmos.msg.v1.signer) = "address";
  string address = 1;
  string key     = 2;
}
``` [2](#0-1) 

This annotation means the Cosmos SDK derives `GetSigners()` from the `address` field. Under a normal (non-authz) transaction the SDK enforces that the tx signer equals `address`, so a direct attack is blocked. However, authz replaces this enforcement: the authz module verifies only that the granter matches the inner message's signer field, then dispatches the message directly to the keeper — bypassing the normal per-tx signer check.

**`MsgRegisterEncryptionKey` is not in `DisabledAuthzMsgs`:**

```go
DisabledAuthzMsgs: []string{
    sdk.MsgTypeURL(&evmtypes.MsgEthereumTx{}),
    sdk.MsgTypeURL(&vestingtypes.MsgCreateVestingAccount{}),
    sdk.MsgTypeURL(&vestingtypes.MsgCreatePermanentLockedAccount{}),
    sdk.MsgTypeURL(&vestingtypes.MsgCreatePeriodicVestingAccount{}),
},
``` [3](#0-2) 

`/e2ee.Msg/RegisterEncryptionKey` is not listed, so the ante-handler does not reject it when wrapped in `MsgExec`.

**`ValidateBasic` adds no signer constraint:**

```go
func (m *MsgRegisterEncryptionKey) ValidateBasic() error {
    if _, err := sdk.AccAddressFromBech32(m.Address); err != nil {
        return fmt.Errorf("invalid address: %w", err)
    }
    return ValidateRecipientKey(m.Key)
}
``` [4](#0-3) 

Only bech32 validity and key format are checked — no ownership assertion.

---

### Impact Explanation

An attacker who holds (or socially engineers) an authz grant for `MsgRegisterEncryptionKey` from a victim can replace the victim's canonical on-chain encryption key with an attacker-controlled Age X25519 public key. All future senders who query the chain for the victim's key will encrypt to the attacker's key, giving the attacker plaintext access to every subsequent encrypted message intended for the victim. This is **corruption of e2ee key/message state with direct security impact** — a High-severity finding per the scope rules.

---

### Likelihood Explanation

The precondition is that the victim has previously issued a `MsgGrant` for `MsgRegisterEncryptionKey` to the attacker. This is a realistic scenario in delegated-key-management setups (e.g., a key-management service, a multisig coordinator, or a wallet that auto-grants broad authz permissions). Once the grant exists, exploitation is a single `MsgExec` transaction requiring no further victim interaction.

---

### Recommendation

Add an explicit signer check at the top of `RegisterEncryptionKey`:

```go
func (k Keeper) RegisterEncryptionKey(
    ctx context.Context,
    req *types.MsgRegisterEncryptionKey,
) (*types.MsgRegisterEncryptionKeyResponse, error) {
    sdkCtx := sdk.UnwrapSDKContext(ctx)
    signers := sdkCtx.TxSigner() // or retrieve from msg server context
    if signers.String() != req.Address {
        return nil, sdkerrors.ErrUnauthorized.Wrapf(
            "signer %s cannot register key for address %s", signers, req.Address)
    }
    ...
}
```

Alternatively (defense-in-depth), add `sdk.MsgTypeURL(&types.MsgRegisterEncryptionKey{})` to `DisabledAuthzMsgs` in `app/app.go`. [3](#0-2) 

---

### Proof of Concept

```
1. victim  → MsgGrant(granter=victim, grantee=attacker,
               authorization=GenericAuthorization{msg="/e2ee.Msg/RegisterEncryptionKey"})

2. attacker → MsgExec(grantee=attacker, msgs=[
               MsgRegisterEncryptionKey{address=victim, key=<attacker_age_pubkey>}
             ])

3. Query: KeyRequest{address=victim}
   → KeyResponse.key == <attacker_age_pubkey>   // victim's original key is gone
```

Unit test: construct the grant + exec tx pair in `x/e2ee/keeper/keeper_test.go`, assert that after step 2 the stored key for `victim` equals `attacker_age_pubkey` and not `victim_original_key`. The keeper will succeed with no error because it never inspects the signer. [1](#0-0)
