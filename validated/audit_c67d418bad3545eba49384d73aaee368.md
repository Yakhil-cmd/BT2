### Title
Silent Truncation of Pending Clawback Transactions Beyond 100 Hides Claimable Coins — (`packages/wallets/src/components/WalletHistoryPending.tsx`)

### Summary
`WalletHistoryPending` fetches at most 100 pending clawback transactions with no pagination and no total-count indicator. An attacker who sends 101+ `INCOMING_CLAWBACK_RECEIVE` transactions to a victim causes all transactions beyond index 100 to be silently invisible in the GUI. Because the main transaction history explicitly excludes clawback types, the victim has no GUI path to discover or claim the hidden coins before the clawback window expires, allowing the attacker to recover those funds.

### Finding Description
`WalletHistoryPending` invokes `useWalletTransactions` with `defaultRowsPerPage: 100` and `defaultPage: 0`. The hook computes `start=0, end=100` and issues a single `getTransactions` RPC call for that slice. The component renders only the returned array and exposes no pagination controls. The count badge displays `transactions?.length` (the slice length, ≤100), not the true backend count. The main `WalletHistory` table passes a `typeFilter` that **excludes** `INCOMING_CLAWBACK_RECEIVE` and `INCOMING_CLAWBACK_SEND`, making `WalletHistoryPending` the sole GUI surface for these transaction types. Any pending clawback beyond index 100 is permanently invisible to the victim through the GUI. [6](#0-5) [7](#0-6) 

### Impact Explanation
A victim who does not claim a clawback transaction before its `timeLock` expires loses those funds — the attacker recovers them via the clawback mechanism. This is a direct balance/accounting change affecting XCH. The victim has no GUI indication that additional claimable transactions exist beyond the 100 shown.

### Likelihood Explanation
The attacker must send 101+ on-chain clawback transactions, each incurring a fee. This is a non-trivial cost but is entirely within the capability of an unprivileged on-chain actor. The attack is more economical when the hidden transactions carry large amounts relative to the fee cost of 101 transactions.

### Recommendation
- Replace the hard-coded `defaultRowsPerPage: 100` with `rowsPerPage: -1` (the "fetch all" sentinel already handled by `useWalletTransactions`: `const all = rowsPerPage === -1`), so all pending clawback transactions are always fetched. [8](#0-7) 
- Alternatively, add pagination to `WalletHistoryPending` and display the true `count` from the hook so the victim is aware of hidden transactions.
- Display the true backend count (from `useGetTransactionsCountQuery`) in the chip rather than `transactions?.length`.

### Proof of Concept
Mock `useGetTransactionsQuery` to return 101 `INCOMING_CLAWBACK_RECEIVE` transactions and `useGetTransactionsCountQuery` to return `{ data: 101 }`. Mount `WalletHistoryPending`. Assert that only 100 claim buttons are rendered (the 101st is absent). Confirm the chip reads "100" with no indication of the missing transaction.
