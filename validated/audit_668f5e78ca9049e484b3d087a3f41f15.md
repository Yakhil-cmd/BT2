### Title
EVM Pause Mechanism Is Permanently Inactivatable тАФ No Function Exists to Write `true` to `/storage/evmOperationsPaused` тАФ (File: `fvm/evm/stdlib/contract.cdc`)

---

### Summary

The `EVM` contract defines an `isPaused()` guard that protects every state-mutating EVM operation. However, no function exists anywhere in the contract тАФ or in any Cadence transaction in the repository тАФ to write `true` to the `/storage/evmOperationsPaused` storage path. The pause mechanism is structurally present but permanently inactivatable, leaving all EVM operations (deposits, withdrawals, contract calls, bridge transfers) running without any emergency stop capability.

---

### Finding Description

`EVM.isPaused()` reads a `Bool` from `/storage/evmOperationsPaused` with a `?? false` default: [1](#0-0) 

This guard is applied to every state-mutating entry point in the contract:

- `EVMAddress.deposit` [2](#0-1) 
- `CadenceOwnedAccount.withdraw` [3](#0-2) 
- `CadenceOwnedAccount.deploy` [4](#0-3) 
- `CadenceOwnedAccount.call` / `callWithSigAndArgs` [5](#0-4) 
- `CadenceOwnedAccount.depositNFT`, `withdrawNFT`, `depositTokens`, `withdrawTokens` [6](#0-5) 
- `EVM.run`, `EVM.batchRun`, `EVM.createCadenceOwnedAccount` [7](#0-6) 

The contract `init()` only calls `setupHeartbeat()` and never writes to `/storage/evmOperationsPaused`: [8](#0-7) 

The contract comment acknowledges that "Only the Governance Committee can pause the EVM transactions, with a multi-sig Cadence transaction," but no such transaction exists anywhere in the repository. A search for `evmOperationsPaused` returns exactly two hits: the `isPaused()` read in `contract.cdc` and a single test reference in `evm_test.go`. No write path exists.

The `PauseBridgeTransaction` helper in `fvm/blueprints/bridge.go` targets a separate bridge-level pause (`update_bridge_pause_status.cdc`) and does not write to `/storage/evmOperationsPaused`: [9](#0-8) 

---

### Impact Explanation

In the event of an active exploit against EVM contracts, a required emergency upgrade, or any situation requiring an immediate halt of EVM activity, the Governance Committee has no on-chain mechanism to activate the pause. All EVM deposits, withdrawals, contract calls, and bridge transfers continue to execute unimpeded. User funds bridged into or held in EVM accounts remain exposed for the full duration of the incident, with no protocol-level circuit breaker available.

---

### Likelihood Explanation

The likelihood is medium. The missing write path is a design omission, not a race condition or edge case. Any emergency scenario тАФ exploit, critical bug, required upgrade тАФ that would motivate use of the pause will find it non-functional. The comment in the contract explicitly anticipates governance use of this mechanism, confirming the intent was real and the omission is not deliberate.

---

### Recommendation

Add a dedicated administrative function to the `EVM` contract that writes the pause state to `/storage/evmOperationsPaused`, accessible only to the service account (e.g., `access(account)`):

```cadence
access(account)
fun setOperationsPaused(_ paused: Bool) {
    if self.account.storage.type(at: /storage/evmOperationsPaused) != nil {
        self.account.storage.load<Bool>(from: /storage/evmOperationsPaused)
    }
    self.account.storage.save(paused, to: /storage/evmOperationsPaused)
}
```

A corresponding multi-sig Cadence transaction should be added to the repository so the Governance Committee can invoke this function in an emergency.

---

### Proof of Concept

1. Deploy the EVM contract as-is on any network.
2. Attempt to call `EVM.isPaused()` тАФ it returns `false` (the `?? false` default, since nothing was ever written to `/storage/evmOperationsPaused`).
3. Search the entire contract for any `storage.save` targeting `/storage/evmOperationsPaused` тАФ none exists.
4. Search the repository for any Cadence transaction that writes to this path тАФ none exists.
5. Conclusion: the pause flag can never be set to `true` through any on-chain mechanism, making every `!EVM.isPaused()` guard permanently a no-op, and the emergency stop permanently unavailable.
