### Title
Unvalidated Participant Shares in CKD Coordinator Enable Malicious Participant to Corrupt Derived Key Output — (File: src/confidential_key_derivation/protocol.rs)

### Summary
The CKD coordinator in `do_ckd_coordinator` blindly accumulates `(big_y, big_c)` values received from every participant with no zero-knowledge proof, commitment check, or any other cryptographic validation. A single malicious participant can substitute arbitrary group elements, causing the coordinator to output a structurally valid but cryptographically wrong `CKDOutput`. The TEE application that consumes this output will silently derive an incorrect confidential key.

### Finding Description
The external report's root cause is that a privileged back-end component (the coordinator) accepts and acts on inputs from a less-trusted front-end component (participants) without validating those inputs. The exact same structural flaw exists in the CKD protocol.

In `do_ckd_coordinator`, after computing its own share, the coordinator loops over all other participants and unconditionally adds their contributions:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
Ok(Some(ckd_output))
``` [1](#0-0) 

The honest computation each participant is supposed to perform is:

```
y_i  ← random scalar
big_y_i = λ_i · y_i · G
big_c_i = λ_i · (x_i · H(pk ‖ app_id) + y_i · app_pk)
``` [2](#0-1) 

Nothing in the protocol forces a participant to follow this computation. The participant function `do_ckd_participant` simply calls `compute_signature_share` and sends the result; a malicious implementation can skip that call entirely and send any two group elements. [3](#0-2) 

Unlike the signing protocols (OT-based and Robust ECDSA), which protect against this by verifying the assembled signature against the public key before returning it, the CKD coordinator performs **no post-assembly verification**. It returns the raw sum directly. [4](#0-3) 

### Impact Explanation
A malicious participant sends `(big_y′, big_c′)` of its choice instead of the correctly derived values. The coordinator outputs:

```
big_Y_total = Σ_{j≠i} λ_j·y_j·G  +  big_y′
big_C_total = Σ_{j≠i} λ_j·(x_j·H + y_j·A)  +  big_c′
```

The TEE application then computes `big_C_total − big_Y_total · app_sk`, which no longer equals `msk · H(pk ‖ app_id)`. The derived confidential key is silently wrong. All honest parties accept this corrupted output because the coordinator is the sole aggregator and returns it without any integrity check.

This matches the allowed impact: **High — Corruption of CKD outputs so honest parties accept unusable or inconsistent cryptographic outputs.**

### Likelihood Explanation
Any participant in the protocol can mount this attack with zero cryptographic knowledge — it requires only sending two arbitrary group elements instead of the correct ones. No special access, leaked key, or external assumption is needed. The attacker is a documented threat model participant (malicious participant), and the attack is a single-round, single-message substitution.

### Recommendation
Require each participant to accompany `(big_y, big_c)` with a zero-knowledge proof of correct formation. Concretely, a Schnorr-style proof of knowledge of `(y_i, x_i)` such that `big_y_i = λ_i · y_i · G` and `big_c_i = λ_i · x_i · H + λ_i · y_i · app_pk` would bind the contribution to the participant's committed key share. The coordinator must verify all proofs before summing. Alternatively, a commitment-then-reveal round can be added so that participants cannot adaptively choose their contribution after seeing others'.

### Proof of Concept
1. Honest participants run `compute_signature_share` and send correct `(norm_big_y, norm_big_c)` to the coordinator.
2. Malicious participant `P_m` instead sends `(ElementG1::identity(), ElementG1::identity())` — two identity elements.
3. The coordinator sums all contributions; `P_m`'s Lagrange-weighted share is replaced by the identity, so the aggregate `big_C_total` is missing `λ_m · x_m · H(pk ‖ app_id)`.
4. The coordinator returns `CKDOutput::new(norm_big_y, norm_big_c)` with no error.
5. The TEE application calls `ckd_output.unmask(app_sk)` and obtains a key that differs from `msk · H(pk ‖ app_id)` by exactly `−λ_m · x_m · H(pk ‖ app_id)`.
6. Any operation the TEE performs with this key silently produces wrong results; no party detects the corruption. [5](#0-4)

async fn do_ckd_coordinator(
    mut chan: SharedChannel,
    participants: ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<CKDOutputOption, ProtocolError> {
    let (mut norm_big_y, mut norm_big_c) =
        compute_signature_share(&participants, me, key_pair, app_id, app_pk, rng)?;

    // Receive everyone's inputs and add them together
    let waitpoint = chan.next_waitpoint();

    for (_, participant_output) in
        recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
    {
        norm_big_y += participant_output.big_y();
        norm_big_c += participant_output.big_c();
    }
    let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
    Ok(Some(ckd_output))
}
