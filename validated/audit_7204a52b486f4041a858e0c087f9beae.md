### Title
Unprivileged Caller Can Permanently Skip Block Rewards for All Stakers via `update_rewards(disable_rewards: true)` - (File: src/staking/staking.cairo)

---

### Summary

The `update_rewards` function in the Staking contract is publicly callable by any non-zero address and accepts a caller-controlled `disable_rewards: bool` parameter. Critically, the global `last_reward_block` storage slot is written **before** the `disable_rewards` guard is evaluated. An attacker can call `update_rewards(any_active_staker, disable_rewards: true)` once per block to consume the block's single reward-update slot without distributing any rewards, permanently denying all stakers their consensus-phase block rewards for that block.

---

### Finding Description

`IStakingRewardsManager::update_rewards` is exposed as a public ABI entry point with no role or access-control guard beyond `general_prerequisites()`, which only checks that the contract is unpaused and the caller is non-zero. [1](#0-0) 

The function enforces a per-block exclusivity invariant: only one call per block is allowed, enforced by comparing `current_block_number` against the stored `last_reward_block`. [2](#0-1) 

`last_reward_block` is then unconditionally written to the current block number **before** the `disable_rewards` flag is checked: [3](#0-2) 

Because `last_reward_block` is a single global storage variable shared across all stakers: [4](#0-3) 

any caller who invokes `update_rewards(any_active_staker, disable_rewards: true)` in block N will:
1. Advance `last_reward_block` to N.
2. Return early at the `disable_rewards` guard without distributing any rewards.
3. Cause every subsequent `update_rewards` call in block N to revert with `REWARDS_ALREADY_UPDATED`.

The `disable_rewards` parameter has no access control and is fully attacker-controlled: [5](#0-4) 

`general_prerequisites` imposes no role check: [6](#0-5) 

---

### Impact Explanation

In the consensus-rewards phase, block rewards are the sole mechanism by which stakers accumulate `unclaimed_rewards_own`. Each block where the attacker fires the griefing transaction, **all stakers** permanently lose the block reward they would have earned. Because `last_reward_block` is never decremented and the missed block cannot be retroactively replayed, the yield loss is irreversible.

This constitutes **permanent freezing of unclaimed yield** — a High-severity impact under the allowed scope.

---

### Likelihood Explanation

- The entry point is fully public; no privileged role, leaked key, or external dependency is required.
- The attacker only needs to submit one transaction per block, costing only gas.
- The attack is sustainable indefinitely and can be automated.
- There is no on-chain defense: the staking contract cannot distinguish a legitimate `disable_rewards: true` call (e.g., during migration) from a malicious one.

Likelihood: **High**.

---

### Recommendation

1. **Remove `disable_rewards` from the public interface entirely**, or gate it behind a privileged role (e.g., `only_app_governor`).
2. Alternatively, write `last_reward_block` only **after** the `disable_rewards` guard, so a skipped-reward call does not consume the block slot:

```cairo
if disable_rewards || self.is_pre_consensus() {
    return;
}
// Only advance the block pointer when rewards are actually distributed.
self.last_reward_block.write(current_block_number);
// ... reward calculation ...
```

3. Consider making `update_rewards` callable only by the staker themselves or a designated keeper role, consistent with how `update_rewards_from_attestation_contract` is restricted to the attestation contract. [7](#0-6) 

---

### Proof of Concept

**Setup**: Consensus rewards are active (`is_pre_consensus()` returns `false`). Staker Alice is active with non-zero balance.

**Attack** (repeated every block):

```
// Attacker (any EOA) calls in block N:
staking.update_rewards(alice_address, disable_rewards: true)
```

**Execution trace**:
1. `general_prerequisites()` passes (contract unpaused, caller non-zero). ✓
2. `current_block_number (N) > last_reward_block (N-1)` — assertion passes. ✓
3. Alice is active with non-zero balance — assertions pass. ✓
4. `last_reward_block.write(N)` — block slot consumed. ✓
5. `disable_rewards == true` → early return, **zero rewards distributed**. ✓
6. Any legitimate call to `update_rewards` in block N reverts: `N > N` is false → `REWARDS_ALREADY_UPDATED`. ✓

**Result**: Alice (and every other staker) earns zero rewards for block N. The attacker repeats this each block to permanently deny all consensus-phase block rewards across the entire protocol.
