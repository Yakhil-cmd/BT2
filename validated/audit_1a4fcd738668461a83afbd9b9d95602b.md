### Title
Malicious Participant Can Corrupt CKD Output via Unverified Contribution — (File: `src/confidential_key_derivation/protocol.rs`)

### Summary

The CKD coordinator aggregates participant contributions `(norm_big_y, norm_big_c)` by simple addition with no proof of correctness. Any single malicious participant can send arbitrary BLS12-381 G1 points in place of their honest share, causing the coordinator to produce a corrupted `CKDOutput`. Honest parties then accept an unusable derived key with no indication of failure.

### Finding Description

**Vulnerability class (analog):** Unvalidated external input processed without sanitization. The XSS report's root cause is that `getEthAssets`/`getPolygonAssets`/`getBSCAssets` accept token data from untrusted on-chain sources and render it without filtering. The analog here is that `do_ckd_coordinator` accepts cryptographic contributions from untrusted participants and aggregates them without any proof of correctness.

**Root cause — `do_ckd_coordinator`:** [1](#0-0) 

The coordinator loops over every other participant's message and unconditionally adds their `big_y` and `big_c` fields:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
```

There is no zero-knowledge proof, commitment binding, or any other check that participant `i` actually computed:

```
norm_big_y_i = λ_i · y_i · G₁
norm_big_c_i = λ_i · (x_i · H(pk, app_id) + y_i · A)
```

for their actual key share `x_i` and a fresh random `y_i`. The protocol specification confirms this is the intended computation: [2](#0-1) 

but no verification of received values is performed anywhere in `do_ckd_coordinator`.

**Attacker-controlled entry path:**

A malicious participant `j` is a legitimate member of the `participants` list (they hold a valid DKG share). They call `ckd(...)` but, instead of running `compute_signature_share` honestly, they send an arbitrary pair `(P, Q)` of G1 points to the coordinator via `chan.send_private`. The coordinator's `recv_from_others` call accepts this message because it comes from a known participant: [3](#0-2) 

The malicious `(P, Q)` is added directly into `norm_big_y` and `norm_big_c`, producing a final `CKDOutput` that does not equal `(Y, C) = (Σ λ_i Y_i, msk·H(pk,app_id) + a·Y)`.

**Why no existing check catches this:**

The only validation at initialization time is participant-list and coordinator membership: [4](#0-3) 

No runtime check on the received G1 points exists. The `CKDOutput::unmask` function simply computes `C − a·Y` on whatever `(Y, C)` the coordinator assembled: [5](#0-4) 

There is no final verification that the result equals `msk · H(pk, app_id)`.

### Impact Explanation

**High — Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.**

A single malicious participant causes the coordinator to output a `CKDOutput` whose `unmask(app_sk)` value is not `msk · H(pk, app_id)`. The application silently derives a wrong key. Because the protocol has no integrity check on the final result, honest parties cannot distinguish a corrupted output from a correct one. The derived key is permanently wrong for that session, and any data encrypted or authenticated under it is unrecoverable.

### Likelihood Explanation

Any participant who is legitimately enrolled in the CKD session can trigger this. No privileged access, leaked key, or cryptographic break is required — only the ability to send an arbitrary serialized G1 point pair to the coordinator, which is exactly what the protocol's messaging layer permits. The CKD protocol requires all `n` participants to contribute (it uses `recv_from_others` which waits for every other participant), so even one malicious participant out of `n` is sufficient.

### Recommendation

Require each participant to accompany their `(norm_big_y, norm_big_c)` with a non-interactive zero-knowledge proof of correct computation — specifically a proof of knowledge of `(y_i, x_i)` such that:

- `norm_big_y = λ_i · y_i · G₁`
- `norm_big_c = λ_i · (x_i · H(pk, app_id) + y_i · A)`
- `x_i · G₂ = verifying_share_i` (binding to the DKG public share)

The coordinator must verify each proof before adding the contribution. Alternatively, add a final consistency check: after aggregation, verify the output satisfies the BLS pairing equation `e(C, G₂) = e(H(pk,app_id), pk_G2) · e(Y, A_G2)` (where `A_G2` is the app public key lifted to G2), which would detect any corrupted contribution.

### Proof of Concept

1. Run DKG with participants `[P1, P2, P3]`, threshold 2, producing honest key shares `x_1, x_2, x_3` and master public key `pk`.
2. Participant `P3` is malicious. Instead of calling `compute_signature_share` honestly, it sends `(G₁, G₁)` (the generator point) to the coordinator for both `norm_big_y` and `norm_big_c`.
3. The coordinator (say `P1`) runs `do_ckd_coordinator`:
   - Computes its own honest `(norm_big_y_1, norm_big_c_1)`.
   - Receives honest `(norm_big_y_2, norm_big_c_2)` from `P2`.
   - Receives `(G₁, G₁)` from `P3` and adds it unconditionally.
4. The resulting `CKDOutput` has `big_c = norm_big_c_1 + norm_big_c_2 + G₁` and `big_y = norm_big_y_1 + norm_big_y_2 + G₁`.
5. The app calls `ckd_output.unmask(app_sk)` and obtains `big_c − app_sk · big_y`, which is not `msk · H(pk, app_id)`.
6. The app silently uses the wrong derived key. No error is returned by any party.

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
