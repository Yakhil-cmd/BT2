### Title
Unbounded Loop in `calculate_rewards` Allows Permanent Freezing of Delegator's Unclaimed Yield - (File: `src/pool/pool.cairo`)

---

### Summary
The `calculate_rewards` internal function in `DelegationPool` contains a self-admitted unbounded loop that iterates over a delegator's entire balance-change trace since their last claim. A delegator who repeatedly calls `add_to_delegation_pool` (or `exit_delegation_pool_intent`) across many different epochs without ever claiming rewards will grow their `pool_member_epoch_balance` trace without bound. Once the trace is large enough, any subsequent `claim_rewards` call will exceed the Starknet block gas limit and revert, permanently freezing the delegator's accumulated unclaimed yield.

---

### Finding Description

`calculate_rewards` in `src/pool/pool.cairo` iterates over every entry in the caller's `pool_member_epoch_balance` trace that falls between the last-claimed checkpoint and the current epoch:

```
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
    ...
    let to_sigma = self.find_sigma(pool_member_checkpoint, curr_epoch: until_epoch);
    ...
    entry_to_claim_from += 1;
}
``` [1](#0-0) 

Each iteration performs at least two storage reads (`pool_member_trace.at(...)` and `find_sigma` which reads `cumulative_rewards_trace`), making it expensive per iteration.

The trace grows by one new checkpoint every time `set_member_balance` is called with a key (epoch) that differs from the last stored key:

```
fn set_member_balance(ref self: ContractState, pool_member: ContractAddress, amount: Amount) {
    ...
    trace.insert(key: self.get_epoch_plus_k(), value: pool_member_balance);
}
``` [2](#0-1) 

`set_member_balance` is called by both `increase_member_balance` (used in `add_to_delegation_pool`) and directly in `exit_delegation_pool_intent`: [3](#0-2) [4](#0-3) 

The `trace.insert` logic only merges into the last checkpoint when the epoch key is identical; otherwise it appends a new entry:

```
if last.key == key {
    last.value = value;
    checkpoints[len - 1].write(last);
} else {
    assert!(last.key < key, "{}", TraceErrors::UNORDERED_INSERTION);
    checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
}
``` [5](#0-4) 

Because `get_epoch_plus_k()` returns `current_epoch + K` (K = 2), a call in epoch E writes to epoch E+2. A call in epoch E+1 writes to epoch E+3. These are distinct keys, so each call in a new epoch appends a fresh checkpoint. There is no cap on how many checkpoints a single delegator can accumulate.

Critically, `entry_to_claim_from` is only advanced on a **successful** `claim_rewards` execution. If the loop OOGs, the pointer is never updated and the delegator can never recover their rewards. [6](#0-5) 

---

### Impact Explanation

Once the trace is large enough to cause an OOG revert in `claim_rewards`, the delegator's accumulated STRK rewards are permanently inaccessible. The `entry_to_claim_from` pointer is never advanced (the transaction reverts), so every future `claim_rewards` attempt will also OOG. The STRK tokens allocated as rewards remain locked in the pool contract with no recovery path.

**Impact: High — Permanent freezing of unclaimed yield.**

---

### Likelihood Explanation

The attack is self-inflicted but realistic over a long protocol lifetime:

1. A delegator enters the pool with the minimum amount.
2. Each epoch, the delegator calls `add_to_delegation_pool` with 1 token (the minimum non-zero amount), never calling `claim_rewards`.
3. After N epochs, the trace has N entries.
4. The single `claim_rewards` call must iterate all N entries, each requiring multiple storage reads.

On Starknet, storage reads are among the most gas-expensive operations. With a sufficiently large N (reachable over months of protocol operation), the transaction will exceed the block gas limit.

The delegator need not be malicious — a long-lived delegator who frequently adjusts their stake and rarely claims rewards will naturally accumulate a large trace. The protocol's own documentation acknowledges the risk: *"This loop is unbounded but unlikely to exceed gas limits."*

**Likelihood: Low-to-Medium** (requires sustained inaction on claiming over many epochs, but is a realistic long-term scenario).

---

### Recommendation

1. **Enforce a claim before balance changes**: Require `claim_rewards` to be called (or auto-claim) before any `add_to_delegation_pool` or `exit_delegation_pool_intent` that would append a new trace entry. This bounds the loop to at most 1–2 iterations per claim.
2. **Alternatively, cap the trace length**: Reject balance-change operations if the unclaimed trace window exceeds a safe maximum (e.g., 500 entries), forcing the delegator to claim first.
3. **Paginated claiming**: Introduce a `claim_rewards_partial(max_entries)` entry point that advances `entry_to_claim_from` by at most `max_entries` per call, allowing recovery even from a bloated trace.
