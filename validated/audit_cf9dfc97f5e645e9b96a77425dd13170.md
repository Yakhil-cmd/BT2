### Title
User-Controlled `executionEffort = 0` Bypasses Computation Limit in Scheduled Transaction Executor — (`File: fvm/blueprints/scheduled_callback.go`, `model/access/systemcollection/system_collection_v0.go`)

---

### Summary

When a user schedules a transaction via `FlowTransactionScheduler`, they supply a user-controlled `executionEffort` value. This value is read from the emitted `PendingExecution` event and used verbatim as the `GasLimit` of the system executor transaction. Only an upper-bound cap is applied; there is no lower-bound check. Setting `executionEffort = 0` causes the FVM's `ComputationLimit()` fallback to grant the executor transaction the full default computation limit, while the user may have deposited only minimal fees — directly analogous to the reported "zero gas price avoids fees" pattern.

---

### Finding Description

When a user calls `FlowTransactionScheduler.schedule()`, they specify `executionEffort` (a `UInt64`) and deposit a `fees` vault. At execution time, the scheduler contract emits a `PendingExecution` event containing the user-supplied `executionEffort`. The FVM reads this event in `callbackArgsFromEvent`:

```go
// fvm/blueprints/scheduled_callback.go
func callbackArgsFromEvent(event flow.Event) (uint64, uint64, error) {
    cadenceId, cadenceEffort, err := ParsePendingExecutionEvent(event)
    ...
    effort := uint64(cadenceEffort)
    if effort > flow.DefaultMaxTransactionGasLimit {
        effort = flow.DefaultMaxTransactionGasLimit
    }
    return uint64(cadenceId), uint64(effort), nil
}
``` [1](#0-0) 

Only an upper-bound cap is applied. `effort = 0` passes through unchanged. This value is then used directly as the `GasLimit` of the executor transaction:

```go
tx, err := flow.NewTransactionBodyBuilder().
    ...
    SetComputeLimit(effort).  // effort = 0 passes through
    Build()
``` [2](#0-1) 

The same pattern exists in the v0 system collection builder: [3](#0-2) 

When `GasLimit = 0` is set, `TransactionProcedure.ComputationLimit()` falls back to the context's computation limit or `DefaultComputationLimit`:

```go
computationLimit := proc.Transaction.GasLimit  // = 0
if computationLimit == 0 {
    computationLimit = ctx.ComputationLimit
    if computationLimit == 0 {
        computationLimit = DefaultComputationLimit
    }
}
``` [4](#0-3) 

The `checkGasLimit` validator that would normally reject `GasLimit == 0` explicitly skips service-account-payer transactions — which executor transactions are: [5](#0-4) 

The result: a user who deposits minimal fees and sets `executionEffort = 0` receives the full default computation budget for their scheduled transaction execution.

---

### Impact Explanation

**Impact: Medium**

An unprivileged user can schedule a transaction with `executionEffort = 0` and a minimal fees vault. The executor transaction will run with the full default computation limit (up to `flow.DefaultMaxTransactionGasLimit`). If the scheduled transaction handler consumes significant computation, the deposited fees may be insufficient to cover the actual execution cost, causing the system executor account to bear the shortfall. This is a fee underpayment / computation budget bypass affecting the integrity of the scheduled transaction fee model.

---

### Likelihood Explanation

**Likelihood: Medium**

Any unprivileged account can call `FlowTransactionScheduler.schedule()` with `executionEffort = 0`. The path is directly reachable from a Cadence transaction with no special privileges. The only prerequisite is depositing a (potentially minimal) fees vault. The behavior is deterministic and reproducible.

---

### Recommendation

Add a lower-bound check in `callbackArgsFromEvent` (both in `fvm/blueprints/scheduled_callback.go` and `model/access/systemcollection/system_collection_v0.go`) to enforce a minimum non-zero `effort`. If `effort == 0`, either reject the scheduled transaction or substitute a protocol-defined minimum computation limit, rather than silently falling back to the full default limit. Additionally, the `FlowTransactionScheduler` Cadence contract should enforce a minimum `executionEffort` at scheduling time to prevent zero-effort registrations from reaching the FVM layer.

---

### Proof of Concept

1. Attacker calls `FlowTransactionScheduler.schedule()` with `executionEffort: UInt64(0)` and a minimal fees vault (e.g., 0.0001 FLOW).
2. The scheduler contract emits `PendingExecution(id: X, executionEffort: 0, fees: 0.0001, ...)`.
3. FVM calls `callbackArgsFromEvent` → `effort = 0` passes the only check (`0 > DefaultMaxTransactionGasLimit` is false).
4. `generateExecuteCallbacksTransaction` sets `GasLimit = 0` on the executor transaction.
5. `TransactionProcedure.ComputationLimit()` sees `GasLimit == 0` and returns `ctx.ComputationLimit` or `DefaultComputationLimit` (the full default limit).
6. The attacker's scheduled transaction handler executes with the maximum computation budget while having deposited only minimal fees. [6](#0-5) [7](#0-6)

func callbackArgsFromEvent(event flow.Event) (uint64, uint64, error) {
	cadenceId, cadenceEffort, err := ParsePendingExecutionEvent(event)
	if err != nil {
		return 0, 0, err
	}

	effort := uint64(cadenceEffort)

	if effort > flow.DefaultMaxTransactionGasLimit {
		log.Warn().Uint64("effort", effort).Msg("effort is greater than max transaction gas limit, setting to max")
		effort = flow.DefaultMaxTransactionGasLimit
	}

	return uint64(cadenceId), uint64(effort), nil
}

func (proc *TransactionProcedure) ComputationLimit(ctx Context) uint64 {
	// TODO for BFT (enforce max computation limit, already checked by collection nodes)
	// TODO replace tx.Gas with individual limits for computation and memory

	// decide computation limit
	computationLimit := proc.Transaction.GasLimit
	// if the computation limit is set to zero by user, fallback to the gas limit set by the context
	if computationLimit == 0 {
		computationLimit = ctx.ComputationLimit
		// if the context computation limit is also zero, fallback to the default computation limit
		if computationLimit == 0 {
			computationLimit = DefaultComputationLimit
		}
	}

func (v *TransactionValidator) checkGasLimit(tx *flow.TransactionBody) error {
	// if service account is the payer of the transaction accepts any gas limit
	// note that even though we don't enforce any limit here, exec node later
	// enforce a max value for any transaction
	if tx.Payer == v.serviceAccountAddress {
		return nil
	}
	if tx.GasLimit > v.options.MaxGasLimit || tx.GasLimit == 0 {
		return InvalidGasLimitError{
			Actual:  tx.GasLimit,
			Maximum: v.options.MaxGasLimit,
		}
	}
