Audit Report

## Title
Finalization Instruction Overhead Charged Against All Canisters (Including Cold/Idle) Per Inner-Loop Iteration, Enabling Subnet Throughput Degradation - (File: rs/execution_environment/src/scheduler.rs)

## Summary

At the end of every inner-loop iteration in `inner_round`, the scheduler deducts `instruction_overhead_per_canister_for_finalization * state.num_canisters()` from the round instruction budget. `state.num_canisters()` counts all canisters on the subnet — including cold/idle canisters that are explicitly excluded from per-round operations by the hot/cold partitioning in `CanisterStates`. The actual per-iteration finalization work in `finish_round` only iterates over `scheduled_canisters` and `fully_executed_canisters` (small subsets). An unprivileged canister developer who creates many idle canisters can inflate this overhead, causing the round instruction budget to be consumed faster than actual work warrants, degrading subnet throughput for all users.

## Finding Description

**Root cause:** In `inner_round` at `rs/execution_environment/src/scheduler.rs` lines 559–563:

```rust
round_limits.instructions -= as_round_instructions(
    self.config
        .instruction_overhead_per_canister_for_finalization
        * state.num_canisters() as u64,
);
```

`state.num_canisters()` returns `hot.len() + cold.len()` — the total count of every canister on the subnet, including cold/idle ones.

The `CanisterStates` module (`rs/replicated_state/src/canister_states.rs`, lines 1–23) explicitly documents that cold canisters are "definitely idle" and that per-round operations should skip them. The hot/cold partitioning was introduced precisely to make per-round operations `O(|hot|)` instead of `O(|all canisters|)`.

The actual `finish_round` function (`rs/execution_environment/src/scheduler/round_schedule.rs`, lines 525–637) only iterates over:
- `self.fully_executed_canisters` (lines 534–543) — canisters that ran this iteration
- `self.scheduled_canisters` (lines 552–590) — canisters that were scheduled this iteration
- `subnet_schedule` (lines 600–637) — the priority queue of historically scheduled canisters (not cold canisters that have never been scheduled)

Cold canisters that have never been scheduled are never visited in `finish_round`, yet they are counted in the overhead deduction every iteration.

The constant `INSTRUCTION_OVERHEAD_PER_CANISTER_FOR_FINALIZATION = 12_000` (`rs/config/src/subnet_config.rs`, lines 60–65) was calibrated from a 13ms finalization time with 5,000 canisters — before the hot/cold partitioning was introduced. The doc comment at lines 244–249 still says "active canisters" but the code uses `state.num_canisters()` (all canisters).

**Exploit flow:**
1. Attacker deploys N idle canisters on an application subnet (no heartbeat, no pending messages). After a few rounds they are demoted to the cold pool.
2. Cold canisters are never executed, never charged per-round, and never appear in `scheduled_canisters` — but they permanently inflate `state.num_canisters()`.
3. Every inner-loop iteration deducts `12,000 × N` instructions from the round budget, regardless of how many canisters actually had work done.
4. With sufficient N, the round budget is exhausted after fewer inner-loop iterations, starving cross-canister call chains (callbacks cannot run).

**Existing guards are insufficient:** The `max_number_of_canisters` registry parameter bounds the attack but does not eliminate it. At a typical limit of 100,000 canisters, the overhead per iteration is `12,000 × 100,000 = 1.2B` instructions — 30% of the 4B round budget — significantly degrading throughput.

## Impact Explanation

This is a **High** severity finding matching: *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."*

With `INSTRUCTION_OVERHEAD_PER_CANISTER_FOR_FINALIZATION = 12_000` and `MAX_INSTRUCTIONS_PER_ROUND = 4,000,000,000`:
- 50,000 idle canisters → 600M instructions overhead per inner-loop iteration (15% of round budget)
- 100,000 idle canisters → 1.2B instructions overhead per inner-loop iteration (30% of round budget)

The test `finalization_overhead_per_canister_reduces_instruction_budget` (`rs/execution_environment/src/scheduler/tests/limits.rs`, lines 567–646) directly confirms the starvation: with only 3 canisters and proportionally large overhead, the callback iteration (iteration 3 of a call→response→callback chain) cannot run. The impact is subnet-wide — all users on the subnet experience reduced message processing capacity.

## Likelihood Explanation

Any unprivileged canister developer can create canisters on an application subnet. The attack requires:
- A one-time cycle expenditure to create N idle canisters (no ongoing cost)
- No special privileges, no social engineering, no node compromise

Once created, idle canisters are demoted to the cold pool and impose no per-round execution cost on the attacker — but they permanently inflate the overhead charged against every inner-loop iteration on the subnet. The asymmetry (one-time cost vs. perpetual overhead inflation) makes this economically viable. The attack is bounded by `max_number_of_canisters` and the attacker's cycle budget, but significant throughput degradation is achievable well below the subnet canister limit.

## Recommendation

Replace `state.num_canisters()` with the count of hot (active) canisters in the overhead calculation, since cold canisters are not visited in per-round finalization:

```rust
round_limits.instructions -= as_round_instructions(
    self.config
        .instruction_overhead_per_canister_for_finalization
        * state.num_hot_canisters() as u64,  // or scheduled_canisters.len()
);
```

Alternatively, re-calibrate `INSTRUCTION_OVERHEAD_PER_CANISTER_FOR_FINALIZATION` to reflect the actual post-partitioning finalization cost, which is now `O(|hot|)` for most operations. The overhead for `charge_canisters_for_resource_allocation_and_usage` (which does visit all canisters) should be amortized across the 50-round `CHARGE_INTERVAL_ROUNDS` interval rather than charged every iteration.

## Proof of Concept

The existing test `finalization_overhead_per_canister_reduces_instruction_budget` in `rs/execution_environment/src/scheduler/tests/limits.rs` (lines 567–646) directly demonstrates the starvation mechanism with 3 canisters and proportionally large overhead. To reproduce the attack at scale:

1. In the test framework, create N idle canisters (no messages, no heartbeat) using `test.create_canister()` — these will be demoted to cold after a round.
2. Set `instruction_overhead_per_canister_for_finalization` to `NumInstructions::new(12_000)` (production value).
3. Set `max_instructions_per_round` to `NumInstructions::new(4_000_000_000)` (production value).
4. Send a cross-canister call chain (A → B → callback to A) that normally produces 3 inner-loop iterations.
5. Call `test.execute_round(ExecutionRoundType::OrdinaryRound)`.
6. Assert that `metrics.inner_loop_processed_non_zero_inputs_count` is less than 3 — the callback iteration is starved.
7. Scale N to demonstrate proportional throughput degradation up to the `max_number_of_canisters` limit.
