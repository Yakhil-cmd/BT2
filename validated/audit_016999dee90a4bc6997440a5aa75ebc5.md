### Title
Missing Participant-Set Cross-Validation Between Triple Generation and Presigning Corrupts Presignature Output - (`File: src/ecdsa/ot_based_ecdsa/presign.rs`)

### Summary

The OT-based ECDSA presigning protocol explicitly omits the check that the presigning participant set matches the participant set that generated the Beaver triples. Because Lagrange linearization is applied using the presigning participant set's coefficients to shares that were polynomial-evaluated for the triple-generation participant set, any mismatch between the two sets produces a cryptographically invalid presignature that all honest parties silently accept.

### Finding Description

The OT-based ECDSA signing pipeline requires three phases: triple generation (`P_0`), presigning (`P_1 ⊆ P_0`), and signing (`P_2 ⊆ P_1`). The protocol specification in `docs/ecdsa/ot_based_ecdsa/signing.md` explicitly states that the two Beaver triples consumed by presigning must have been **generated by the same set of parties P_0**:

> "Two multiplicative (Beaver) triples private shares (a_i, b_i, c_i) and (k_i, d_i, e_i) **generated by the same set of parties P_0**"

The `TriplePub` struct records the exact participant set and threshold used during triple generation: [1](#0-0) 

The `presign()` initialization function checks that the threshold values match, but contains an explicit comment acknowledging that the participant-set cross-check is **omitted**: [2](#0-1) 

Inside `do_presign`, the Lagrange coefficient `lambda_me` is computed from the **presigning** participant set: [3](#0-2) 

This `lambda_me` is then applied to the triple shares `(k_i, e_i, a_i, b_i)` and the key share `x_i`. However, those triple shares are polynomial evaluations at participant IDs from the **triple-generation** participant set `P_0`. If `P_1 ≠ P_0` (even with the same threshold), the Lagrange coefficients are computed for the wrong participant set, and the linearized values `k'_i`, `e_i`, `a'_i`, `b'_i`, `x'_i` are all wrong. The resulting presignature `(R, k_i, sigma_i)` will not satisfy the required algebraic relationship `sigma = k * x`, making it cryptographically unusable.

The `TriplePub.participants` field is available at presign time but is never compared against the presigning `participants` argument: [4](#0-3) 

### Impact Explanation

A malicious coordinator or library caller can invoke `presign()` with a participant set `P_1` whose IDs differ from those in `triple0.1.participants` / `triple1.1.participants`, while keeping the threshold equal (satisfying the only cross-check that exists). All honest participants will execute the protocol to completion and each will output a `PresignOutput` that appears structurally valid but is cryptographically corrupt — the presignature cannot produce a valid ECDSA signature. Since the library explicitly performs no internal presignature verification, honest parties have no way to detect the corruption until the signing phase fails. This maps to: **High — Corruption of presign outputs so honest parties accept inconsistent/unusable cryptographic outputs.**

### Likelihood Explanation

The entry path is the public `presign()` API. Any caller who controls the `participants` argument (e.g., a malicious coordinator assembling the presigning session) can trigger this by supplying a participant list that differs from the one stored in `TriplePub.participants`. The omission is explicitly acknowledged in a code comment, confirming it is a known gap rather than an oversight in documentation. The threshold-equality check at line 43 provides a false sense of security, as it does not prevent participant-set mismatches.

### Recommendation

Add an explicit check in `presign()` that the presigning participant set is a subset of (or exactly equal to) `args.triple0.1.participants` and `args.triple1.1.participants`. For example:

```rust
// Verify presigning participants are consistent with triple generation participants
let triple0_participants = ParticipantList::new(&args.triple0.1.participants)
    .ok_or(InitializationError::DuplicateParticipants)?;
let triple1_participants = ParticipantList::new(&args.triple1.1.participants)
    .ok_or(InitializationError::DuplicateParticipants)?;

for p in participants.iter() {
    if !triple0_participants.contains(p) || !triple1_participants.contains(p) {
        return Err(InitializationError::BadParameters(
            "Presigning participant set must be a subset of triple generation participants".to_string(),
        ));
    }
}
```

This mirrors the analogous constraint enforced in the reshare protocol, where the intersection of old and new participant sets is validated before proceeding. [5](#0-4) 

### Proof of Concept

1. Run triple generation with participant set `P_0 = {P1, P2, P3}` at threshold 2. Each party receives a `TripleShare` and a `TriplePub` with `participants = [P1, P2, P3]`.
2. A malicious coordinator calls `presign()` with participant set `P_1 = {P1, P2, P4}` (same size, same threshold=2, but `P4` replaces `P3`), passing the triples generated by `P_0`.
3. The threshold check at line 43 passes (both thresholds are 2). The participant-set check is omitted per the comment at line 38–40.
4. `do_presign` computes `lambda_me = participants.lagrange(me)` using `{P1, P2, P4}` Lagrange coefficients, but applies them to shares that are polynomial evaluations at `{P1, P2, P3}` IDs.
5. All parties complete the protocol and output a `PresignOutput`. The output is structurally valid but the algebraic invariant `sigma = k * x` does not hold.
6. Any subsequent signing attempt using this presignature will produce an invalid ECDSA signature, permanently denying signing for this presignature. [2](#0-1) [1](#0-0)

pub fn presign(
    participants: &[Participant],
    me: Participant,
    args: PresignArguments,
) -> Result<impl Protocol<Output = PresignOutput>, InitializationError> {
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }
    // Spec 1.1
    if args.threshold.value() > participants.len() {
        return Err(InitializationError::ThresholdTooLarge {
            threshold: args.threshold.value(),
            max: participants.len(),
        });
    }

    // NOTE: We omit the check that the new participant set was present for
    // the triple generation, because presumably they need to have been present
    // in order to have shares.

    // Also check that we have enough participants to reconstruct shares.
    if args.threshold != args.triple0.1.threshold || args.threshold != args.triple1.1.threshold {
        return Err(InitializationError::BadParameters(
            "New threshold must match the threshold of both triples".to_string(),
        ));
    }

    let participants =
        ParticipantList::new(participants).ok_or(InitializationError::DuplicateParticipants)?;

    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    let ctx = Comms::new();
    let fut = do_presign(ctx.shared_channel(), participants, me, args);
    Ok(make_protocol(ctx, fut))
}
