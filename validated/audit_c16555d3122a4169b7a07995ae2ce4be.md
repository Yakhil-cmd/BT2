### Title
Commission Change Applied Retroactively to Full Epoch Rewards Enables Staker to Steal Delegator Yield - (File: `src/staking/staking.cairo`)

### Summary
A staker can use `set_commission_commitment` to set a `max_commission` higher than their current rate, then increase commission to that maximum near the end of an epoch. Because commission is read from storage at reward-calculation time rather than snapshotted at epoch start, the new higher commission is applied retroactively to the entire epoch's rewards — including the period when the lower commission was in effect — stealing delegators' unclaimed yield.

### Finding Description
The root cause is in `calculate_staker_pools_rewards` in `src/staking/staking.cairo`:

```cairo
let commission = staker_pool_info.commission();
```

The commission is read live from storage at the moment `update_rewards_from_attestation_contract` (V2) or `update_rewards` (V3) is called. There is no per-epoch commission snapshot. Any commission change made between the start of an epoch and the reward-calculation call is therefore applied to the **entire** epoch's rewards. [1](#0-0) 

The `set_commission_commitment` function allows a staker to set a `max_commission` that is **higher** than their current commission:

```cairo
let current_commission = staker_pool_info.commission();
assert!(current_commission <= max_commission, "{}", Error::MAX_COMMISSION_TOO_LOW);
``` [2](#0-1) 

With an active commitment, `update_commission` permits increasing commission up to `max_commission` (the only constraints are `commission <= max_commission` and `commission != old_commission`):

```cairo
if self.is_commission_commitment_active(:commission_commitment) {
    assert!(commission <= commission_commitment.max_commission, ...);
    assert!(commission != old_commission, ...);
}
``` [3](#0-2) 

Without a commitment, commission can only decrease. The commitment mechanism is the only path to an increase, and the code itself acknowledges this gap:

> **Note**: Current commission increase safeguards still allow for sudden commission changes. [4](#0-3) 

The commission change takes effect immediately in storage (`staker_pool_info.commission.write(Option::Some(commission))`), with no delay or epoch-boundary activation. [5](#0-4) 

### Impact Explanation
When the staker increases commission to 100% just before attestation, `split_rewards_with_commission` assigns 100% of the pool's epoch rewards to the staker and 0% to the pool. The pool's cumulative sigma (`update_rewards_from_staking_contract`) is updated with zero rewards, so every delegator's `calculate_rewards` call returns zero for that epoch. Delegators lose their entire epoch's yield; the staker captures it. This is **theft of unclaimed yield** (High impact). [6](#0-5) [7](#0-6) 

### Likelihood Explanation
The attack requires only that the staker:
1. Have an active delegation pool with at least one delegator.
2. Call `set_commission_commitment(max_commission: COMMISSION_DENOMINATOR, expiration_epoch: current_epoch + 1)` — a public, permissionless function.
3. Call `set_commission(commission: COMMISSION_DENOMINATOR)` at any point before the epoch's attestation.

No privileged access, leaked keys, or external dependencies are needed. The staker is an unprivileged actor. The test suite confirms the commitment-then-increase flow works as coded. [8](#0-7) 

### Recommendation
Snapshot the commission at the start of each epoch (e.g., store `commission_at_epoch_start` when the first reward event of an epoch fires) and use that snapshot in `calculate_staker_pools_rewards` instead of the live storage value. Alternatively, require that any commission change first triggers a reward settlement for the current epoch before the new rate takes effect — analogous to the H-06 mitigation of calculating interest before updating the APR.

### Proof of Concept
1. Staker stakes with `commission = 1000` (10%), opens STRK pool.
2. Delegator delegates 1,000,000 STRK.
3. Epoch N begins; both parties expect 10% commission for the epoch.
4. Staker calls `set_commission_commitment(max_commission: 10000, expiration_epoch: current_epoch + 1)`.
5. One block before attestation, staker calls `set_commission(commission: 10000)` (100%).
6. Attestation contract calls `update_rewards_from_attestation_contract(staker_address)`.
7. `calculate_staker_pools_rewards` reads `commission = 10000`; `split_rewards_with_commission` gives 100% to the staker and 0% to the pool.
8. Pool's cumulative sigma is incremented by 0; delegator's `claim_rewards` returns 0 for epoch N.
9. Staker claims the full epoch reward that should have been split 90/10.
