### Title
Malicious FROST Coordinator Can Permanently Deny Signing and CKD for All Honest Participants — (`File: src/frost/eddsa/sign.rs`, `src/frost/redjubjub/sign.rs`, `src/confidential_key_derivation/protocol.rs`)

---

### Summary

The FROST EdDSA, RedJubjub, and CKD protocols each designate a single `coordinator` participant who is the exclusive aggregator and sole producer of the final protocol output. A malicious coordinator can permanently deny signing or CKD completion for all honest participants by simply withholding required messages or refusing to aggregate, with no bypass mechanism available to honest parties.

---

### Finding Description

All three affected protocols share the same structural pattern: a single `coordinator` participant is the only party that can produce the final output (signature or CKD result). Honest non-coordinator participants have no alternative path to complete the protocol.

**EdDSA `sign_v1` (`src/frost/eddsa/sign.rs`)**

In `do_sign_participant_v1`, after sending their commitment privately to the coordinator, participants enter an unbounded loop waiting exclusively for the signing package from the coordinator:

```rust
// Step 2.1
let r2_wait_point = chan.next_waitpoint();
let signing_package = loop {
    let (from, signing_package): (_, frost_ed25519::SigningPackage) =
        chan.recv(r2_wait_point).await?;
    if from != coordinator {
        continue;   // messages from non-coordinators are silently discarded
    }
    break signing_package;
};
```

If the coordinator never sends the signing package, this loop never terminates. There is no timeout, no fallback, and no way for participants to proceed without the coordinator's message. After receiving the signing package, participants send their signature share exclusively back to the coordinator (`chan.send_private(r2_wait_point, coordinator, &signature_share)`), and only the coordinator calls `aggregate()` and returns `Ok(Some(signature))`. Non-coordinator participants always return `Ok(None)`.

**RedJubjub `sign` (`src/frost/redjubjub/sign.rs`)**

In `do_sign_participant`, participants enter an identical unbounded loop waiting for the randomizer from the coordinator:

```rust
let randomizer = loop {
    let (from, randomizer): (_, Randomizer) = chan.recv(wait_round_1).await?;
    if from != coordinator {
        continue;
    }
    break randomizer;
};
```

Without the randomizer, participants cannot compute their signature share. They then send their share exclusively to the coordinator (`chan.send_private(sign_waitpoint, coordinator, &signature_share)`), and again only the coordinator aggregates and returns the signature.

**CKD (`src/confidential_key_derivation/protocol.rs`)**

In `do_ckd_participant`, participants compute their CKD share and send it exclusively to the coordinator, then immediately return `Ok(None)`. Only `do_ckd_coordinator` accumulates all shares and returns `Ok(Some(ckd_output))`. A coordinator that refuses to aggregate simply never produces the CKD output.

**Robust ECDSA `sign` (`src/ecdsa/robust_ecdsa/sign.rs`)**

`do_sign_participant` sends the signature share exclusively to the coordinator and returns `Ok(None)`. Only `do_sign_coordinator` aggregates and returns the signature.

The `recv_from_others` helper used by the coordinator also has no timeout — it loops until all participants have responded — but the inverse direction (participants waiting for the coordinator) is the critical blocking path.

---

### Impact Explanation

**High — Permanent denial of signing and CKD for honest parties.**

A single malicious coordinator can:
1. Collect all participant commitments (EdDSA v1) or presignatures (v2, RedJubjub, ECDSA), then simply never send the signing package or randomizer, leaving all honest participants blocked indefinitely in their receive loops.
2. Collect all signature shares or CKD shares, then refuse to call `aggregate()`, so no output is ever produced.
3. Repeat this for every signing session, permanently preventing the threshold group from ever producing a valid signature or CKD output.

Because participants return `Ok(None)` and only the coordinator returns `Ok(Some(...))`, there is no way for honest participants to reconstruct the output themselves — the aggregation step is structurally gated behind the coordinator. This maps directly to the allowed impact: **permanent denial of signing and CKD for honest parties**.

---

### Likelihood Explanation

The coordinator is a single externally-designated participant passed as a plain `Participant` argument to `sign_v1`, `sign_v2`, `sign` (RedJubjub), and `ckd`. Any party assigned the coordinator role — whether malicious by intent, compromised, or coerced — can trigger this denial. There is no coordinator rotation, no quorum-based coordinator selection, and no mechanism for participants to detect and bypass a non-responsive coordinator. The attack requires no cryptographic capability: the coordinator simply stops sending messages or stops calling `aggregate`.

---

### Recommendation

1. **Document the liveness trust assumption explicitly**: State clearly in the API documentation that the coordinator is trusted for liveness (not just correctness), and that a malicious coordinator can permanently deny signing.
2. **Coordinator rotation / fallback**: Allow participants to re-run the signing protocol with a different coordinator if the current one is unresponsive, without requiring a new DKG.
3. **Coordinator-free aggregation**: Restructure the signing protocol so that participants broadcast their signature shares to all other participants (not just the coordinator), enabling any participant to aggregate the final signature once threshold shares are collected. This eliminates the single point of failure.
4. **Timeout with re-election**: Implement a timeout on the coordinator receive loops so that participants can detect a non-responsive coordinator and trigger a coordinator re-election.

---

### Proof of Concept

**Scenario**: 5-of-7 EdDSA threshold signing. Participant 0 is designated coordinator.

1. All 7 participants call `sign_v1(participants, threshold=5, me, coordinator=P0, keygen_output, message, rng)`.
2. Non-coordinator participants (P1–P6) each execute `do_sign_participant_v1`, generate nonces, and send their commitment privately to P0 via `chan.send_private(commit_waitpoint, coordinator, &commitments)`.
3. P0 (malicious coordinator) receives all 6 commitments but **never calls `chan.send_many(r2_wait_point, &signing_package)`**.
4. P1–P6 each block indefinitely at:
   ```rust
   let signing_package = loop {
       let (from, signing_package): (_, frost_ed25519::SigningPackage) =
           chan.recv(r2_wait_point).await?;   // ← blocks forever
       if from != coordinator { continue; }
       break signing_package;
   };
   ```
5. No signing session ever completes. The threshold group is permanently unable to produce a signature, despite having 6 of 7 honest participants with valid key shares.

The same attack applies to RedJubjub (coordinator withholds the randomizer at `wait_round_1`) and CKD (coordinator receives all shares but never calls `aggregate`). [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5)

pub async fn recv_from_others<T>(
    chan: &SharedChannel,
    waitpoint: u64,
    participants: &ParticipantList,
    me: Participant,
) -> Result<Vec<(Participant, T)>, ProtocolError>
where
    T: serde::de::DeserializeOwned,
{
    let mut seen = ParticipantCounter::new(participants);
    seen.put(me);
    let mut messages = Vec::with_capacity(participants.others(me).count());

    while !seen.full() {
        let (from, msg) = chan.recv(waitpoint).await?;
        if seen.put(from) {
            messages.push((from, msg));
        }
    }

    Ok(messages)
