### Title
Presignature Reuse by Malicious Coordinator Enables Secret Key Extraction — (`src/ecdsa/robust_ecdsa/sign.rs`, `src/ecdsa/robust_ecdsa/mod.rs`)

---

### Summary

The robust ECDSA signing protocol enforces some split-view attack mitigations in code (`participants.len() == 2*max_malicious+1`, `msg_hash != 0`) but does not enforce the equally critical requirement that a presignature is never reused across sessions. A malicious coordinator — a reachable, unprivileged attacker role — can rerandomize the same `PresignOutput` twice with different `entropy` values, run two honest signing sessions, and recover the aggregate secret key from the two resulting signatures via a standard ECDSA nonce-reuse attack.

---

### Finding Description

`RerandomizedPresignOutput::rerandomize_presign()` accepts `presignature: &PresignOutput` — a shared reference — and can be called arbitrarily many times on the same `PresignOutput` with different `RerandomizationArguments`:

```rust
// src/ecdsa/robust_ecdsa/mod.rs
pub fn rerandomize_presign(
    presignature: &PresignOutput,   // <-- reference, not consumed
    args: &RerandomizationArguments,
) -> Result<Self, ProtocolError> {
    ...
    let delta = args.derive_randomness()?;
    let rerandomized_big_r = presignature.big_r * delta;
    let rerandomized_alpha = presignature.alpha * inv_delta;
    let rerandomized_beta =
        (presignature.beta + presignature.c * args.tweak.value()) * inv_delta;
    ...
}

The `sign()` entry point enforces `participants.len() == 2*max_malicious+1` and `msg_hash != 0`, but has no mechanism to detect or prevent reuse of the same underlying presignature material:

The security specification explicitly lists presignature non-reuse as a mandatory constraint:

> **Never reuse a presignature**, even across failed, aborted, or partially completed signing sessions. [4](#0-3) 

The code enforces requirements 1 and 4 from that list but leaves requirement 3 entirely unenforced.

---

### Impact Explanation

The rerandomization scalar is `delta = HKDF(entropy, pk, tweak, msg_hash, big_r, participants)`. Two sessions using the same `PresignOutput` but different `entropy` values produce:

- Session 1: nonce `k'₁ = k · delta₁⁻¹`, signature `(R'₁, s₁)` for `h₁`
- Session 2: nonce `k'₂ = k · delta₂⁻¹`, signature `(R'₂, s₂)` for `h₂`

The coordinator knows `delta₁`, `delta₂`, `h₁`, `h₂`, `R'₁`, `R'₂`, `s₁`, `s₂`. The nonces satisfy `k'₁ / k'₂ = delta₂ / delta₁` — a known ratio. This is a textbook ECDSA nonce-reuse scenario: two equations in two unknowns (`k` and the aggregate secret key `y`), solvable in closed form. The full aggregate private key is extracted.

This matches the allowed Critical impact: **extraction of aggregate secret material / private signing shares**.

---

### Likelihood Explanation

The coordinator is a participant — reachable without any privileged assumptions. The coordinator controls which `RerandomizationArguments` (including `entropy`) are distributed to participants before each session. Honest participants have no visibility into whether the coordinator is reusing a presignature; they each receive their own `PresignOutput` share and a set of rerandomization arguments, compute their share, and send it privately to the coordinator. The attack requires only two completed signing sessions and standard linear algebra over the scalar field.

---

### Recommendation

1. **Consume the presignature on use**: Change `rerandomize_presign` to take `presignature: PresignOutput` (by value), so the `PresignOutput` is moved and cannot be used again in the same process.
2. **Bind entropy to a multi-party commitment**: Derive `entropy` from a coin-flip sub-protocol (e.g., commit-reveal) so no single coordinator can choose it unilaterally.
3. **Add a session-unique nonce to `RerandomizationArguments`** that is agreed upon by all participants via the echo-broadcast layer before rerandomization, preventing the coordinator from silently running two sessions with the same presignature.

// Attacker is the coordinator (participant C).
// All parties ran presign() and hold PresignOutput presig.

// Session 1: coordinator uses entropy_1
let args1 = RerandomizationArguments::new(pk, tweak, h1_bytes, presig.big_r, participants, entropy_1);

let rerand2_i = RerandomizedPresignOutput::rerandomize_presign(&presig_i, &args2)?;
// run sign() → coordinator collects s_i shares → gets valid sig (R'_2, s_2) for h2

// Coordinator now holds (R'_1, s_1, delta_1, h1) and (R'_2, s_2, delta_2, h2).
// k'_1 / k'_2 = delta_2 / delta_1  (known ratio)
// s_1 = (h1 + Rx'_1 * y) / k'_1
// s_2 = (h2 + Rx'_2 * y) / k'_2
// → solve two linear equations for y (aggregate secret key) and k.

impl RerandomizedPresignOutput {
    pub fn rerandomize_presign(
        presignature: &PresignOutput,
        args: &RerandomizationArguments,
    ) -> Result<Self, ProtocolError> {
        if presignature.big_r != args.big_r {
            return Err(ProtocolError::IncompatibleRerandomizationInputs);
        }
        let delta = args.derive_randomness()?;
        if delta.is_zero().into() {
            return Err(ProtocolError::ZeroScalar);
        }

        // cannot be zero due to the previous check
        let inv_delta = delta.invert().unwrap();

        // delta * R
        let rerandomized_big_r = presignature.big_r * delta;

        // alpha * delta^{-1}
        let rerandomized_alpha = presignature.alpha * inv_delta;

        // (beta + c*tweak) * delta^{-1}
        let rerandomized_beta =
            (presignature.beta + presignature.c * args.tweak.value()) * inv_delta;

        Ok(Self {
            big_r: rerandomized_big_r.into(),
            alpha: rerandomized_alpha,
            beta: rerandomized_beta,
            e: presignature.e,
        })
    }

# Security considerations

Before implementing or using the robust ECDSA scheme implemented here,
be aware that it is vulnerable to **split-view attacks** in the robust setting when the
signing parameters are not globally consistent. If different subsets of size at least
$2t + 1$ sign different $(h, \epsilon)$ values using shares derived from the same
presignature, the resulting signatures use multiplicatively related nonces and the
secret key can be recovered using standard ECDSA nonce-reuse attacks.

Moreover, due to protocol modifications relative to [[DJNPO20](https://eprint.iacr.org/2020/501)] (notably signature-share
linearization), **a novel split-view attack exists that can extract the secret key using as
few as $2t + 2$ presigning participants**, with as few as two signing sessions.

To reduce the risk of accidental misuse, enforce the following constraints:

1. **Use exactly $N_1 = N_2 = 2t + 1$ participants for both presigning and signing.**
   Do **not** allow any deviation from this value. In particular:

   * Do **not** allow $N_1 > 2t + 1$, and
   * Do **not** allow $N_2 < N_1$.

   Allowing larger presigning sets or smaller signing sets enables split-view and
   presignature-reuse attacks when a coordinator can run parallel or partially overlapping
   signing sessions.

2. **Ensure all participants agree on $(h, \epsilon)$ and the signing set.**
   The coordinator must not be able to present different message hashes, tweaks, or
   participant lists to different signers.

3. **Never reuse a presignature**, even across failed, aborted, or partially completed
   signing sessions.
