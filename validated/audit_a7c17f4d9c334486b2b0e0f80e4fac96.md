### Title
Dynamic Modification of `wnear_address` Breaks In-Flight XCC Withdrawals, Permanently Freezing User wNEAR - (File: engine/src/contract_methods/xcc.rs)

---

### Summary

The `factory_set_wnear_address` function allows the owner to update the global `wnear_address` at any time. The `withdraw_wnear_to_router` callback reads this address at execution time rather than at the time the XCC operation was initiated. If the owner updates `wnear_address` between the XCC precompile execution (which already transferred the user's wNEAR ERC-20 tokens to the engine's implicit address on the **old** wNEAR contract) and the `withdraw_wnear_to_router` callback, the callback will attempt to call `withdrawToNear` on the **new** wNEAR contract where the engine's implicit address has no balance. The call fails, and the user's wNEAR tokens are permanently frozen at the engine's implicit address with no recovery path.

---

### Finding Description

The XCC flow involves two separate NEAR receipts:

**Receipt 1 — XCC precompile execution:**
In `CrossContractCall::run_with_handle`, the current `wnear_address` is read and `transferFrom(user, engine_implicit_address, amount)` is called on the **old** wNEAR ERC-20 contract, moving the user's wNEAR into the engine's implicit EVM address. A `withdraw_wnear_to_router` callback is then scheduled as a future NEAR receipt. [1](#0-0) 

**Receipt 2 — `withdraw_wnear_to_router` callback (separate NEAR receipt, executes later):**
The callback re-reads `wnear_address` from storage at callback time: [2](#0-1) 

If the owner calls `factory_set_wnear_address` between these two receipts: [3](#0-2) 

…the callback will call `withdrawToNear` on the **new** wNEAR contract. The engine's implicit address has zero balance on the new contract, so the EVM call fails. The function then returns `ERR_WITHDRAW_FAILED`: [4](#0-3) 

The user's wNEAR ERC-20 tokens remain at the engine's implicit address on the **old** wNEAR contract. There is no recovery mechanism: the engine's implicit address is not directly controllable by users, and no contract method exists to sweep stuck wNEAR from it.

The root cause is that `wnear_address` is a single mutable global with no checkpoint or snapshot mechanism, analogous to `maxPrizeCount` in M-03. [5](#0-4) 

---

### Impact Explanation

**High — Temporary or permanent freezing of user funds.**

User wNEAR ERC-20 tokens transferred to the engine's implicit address during the XCC precompile step are irrecoverable if the wNEAR address changes before the callback executes. The engine's implicit address is not user-controlled, and no sweep/recovery function exists in the contract.

---

### Likelihood Explanation

**Low-Medium.** The owner must call `factory_set_wnear_address` during the window between the XCC precompile receipt and the `withdraw_wnear_to_router` callback receipt. This is a realistic scenario during a wNEAR contract migration or upgrade. NEAR receipts execute across multiple blocks, so the window is non-zero. The owner may not be aware of in-flight XCC operations at the time of the update.

---

### Recommendation

Capture the `wnear_address` at the time the XCC operation is initiated and pass it explicitly through the callback arguments (`WithdrawWnearToRouterArgs`), rather than re-reading it from global storage at callback time. This ensures the callback always uses the same wNEAR contract that received the `transferFrom` in the first receipt.

Alternatively, add a guard in `factory_set_wnear_address` that prevents the address from being changed while any XCC operations may be in-flight, or emit a warning/delay mechanism similar to the `upgrade_delay_blocks` pattern already present in the engine. [6](#0-5) 

---

### Proof of Concept

1. User calls the XCC precompile with `required_near > 0`. The precompile calls `transferFrom(user_address, engine_implicit_address, amount)` on `wnear_address_v1`, moving the user's wNEAR to the engine's implicit address. A `withdraw_wnear_to_router` callback is scheduled.
2. Before the callback executes (in a subsequent NEAR block), the owner calls `factory_set_wnear_address(wnear_address_v2)`.
3. The `withdraw_wnear_to_router` callback executes. It reads `wnear_address_v2` from storage and calls `withdrawToNear` on `wnear_address_v2`. The engine's implicit address has zero balance on `wnear_address_v2`.
4. The EVM call fails. `result.status.is_ok()` is false. The function returns `ERR_WITHDRAW_FAILED`.
5. The user's wNEAR tokens remain at the engine's implicit address on `wnear_address_v1` with no recovery path. [7](#0-6) [8](#0-7)

#[named]
pub fn factory_set_wnear_address<I: IO + Copy, E: Env>(
    io: I,
    env: &E,
) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        require_owner_only(&state, &env.predecessor_account_id())?;
        let address = io.read_input_arr20()?;
        xcc::set_wnear_address(&mut io, &Address::from_array(address));
        Ok(())
    })
}
