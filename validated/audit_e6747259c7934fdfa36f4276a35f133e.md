### Title
`vm_resources_to_execution_resources` Returns `UnknownBuiltin` Error for Valid Contracts Using `range_check96`/`add_mod`/`mul_mod` Builtins â€” (`crates/apollo_rpc_execution/src/objects.rs`)

---

### Summary

`vm_resources_to_execution_resources` in `objects.rs` does not handle the `range_check96`, `add_mod`, or `mul_mod` `BuiltinName` variants. Any contract that uses these builtins with a non-zero count causes the function to return `ExecutionError::UnknownBuiltin`, which propagates as an error out of `simulate_transactions` and `starknet_traceTransaction` â€” even though the underlying blockifier execution succeeded.

---

### Finding Description

The match arm in `vm_resources_to_execution_resources` covers eight legacy builtins and silently skips `output` and zero-count entries, but falls through to the catch-all error arm for any other `BuiltinName`:

```rust
// objects.rs:385-391
// TODO(DanB): what about the following?
// BuiltinName::range_check96 => todo!(),
// BuiltinName::add_mod => todo!(),
// BuiltinName::mul_mod => todo!(),
_ => {
    return Err(ExecutionError::UnknownBuiltin { builtin_name });
}
``` [1](#0-0) 

These three builtins are not dead code â€” they are fully tracked by the blockifier (18 references in `bouncer.rs`) and are emitted into `VmExecutionResources.builtin_instance_counter` whenever a Cairo contract uses modular arithmetic opcodes (`u256_mul`, `u512_safe_divmod_by_u256`, etc.).

The function is called during trace construction for every `CallInfo` node:

```rust
// objects.rs:352-355
execution_resources: vm_resources_to_execution_resources(
    call_info.resources.vm_resources,
    gas_vector,
)?,
``` [2](#0-1) 

This `TryFrom<(CallInfo, GasVector)>` impl is invoked from every trace constructor (`InvokeTransactionTrace`, `DeclareTransactionTrace`, `DeployAccountTransactionTrace`, `L1HandlerTransactionTrace`) via `try_into()` calls: [3](#0-2) 

The error propagates through `simulate_transactions`:

```rust
// lib.rs:1016-1023
match trace_constructor(tx_execution_output.execution_info) {
    Ok(transaction_trace) => Ok(TransactionSimulationOutput { ... }),
    Err(e) => Err(e),
}
``` [4](#0-3) 

---

### Impact Explanation

Any unprivileged user who deploys or calls a contract that uses `range_check96`, `add_mod`, or `mul_mod` builtins (standard Cairo 1.x modular arithmetic) will receive an `UnknownBuiltin` error from `starknet_simulateTransactions` or `starknet_traceTransaction`, even though the transaction would execute (or did execute) successfully on-chain. The simulation output is corrupted â€” an error is returned instead of a valid `TransactionSimulationOutput`. This matches the allowed impact: **"RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value."**

---

### Likelihood Explanation

These builtins are used by any contract performing 256-bit or 384-bit modular arithmetic â€” a common pattern in cryptographic contracts (e.g., ECDSA verifiers, hash functions, EVM-compatibility libraries). The blockifier already accounts for them in the bouncer. Any user can trigger this by submitting a simulate or trace RPC call for such a transaction.

---

### Recommendation

Add the three missing arms to the match in `vm_resources_to_execution_resources`. The `starknet_api::execution_resources::Builtin` enum should already contain or need to be extended with corresponding variants. At minimum:

```rust
BuiltinName::range_check96 => builtin_instance_counter.insert(Builtin::RangeCheck96, count),
BuiltinName::add_mod => builtin_instance_counter.insert(Builtin::AddMod, count),
BuiltinName::mul_mod => builtin_instance_counter.insert(Builtin::MulMod, count),
```

If `Builtin` does not yet have these variants, they should be added. The existing TODO comment at lines 385â€“388 already identifies this gap. [1](#0-0)

fn vm_resources_to_execution_resources(
    vm_resources: VmExecutionResources,
    GasVector { l1_gas, l1_data_gas, l2_gas }: GasVector,
) -> ExecutionResult<ExecutionResources> {
    let mut builtin_instance_counter = HashMap::new();
    for (builtin_name, count) in vm_resources.builtin_instance_counter {
        if count == 0 {
            continue;
        }
        let count = u64_from_usize(count);
        match builtin_name {
            BuiltinName::output => continue,
            BuiltinName::pedersen => builtin_instance_counter.insert(Builtin::Pedersen, count),
            BuiltinName::range_check => builtin_instance_counter.insert(Builtin::RangeCheck, count),
            BuiltinName::ecdsa => builtin_instance_counter.insert(Builtin::Ecdsa, count),
            BuiltinName::bitwise => builtin_instance_counter.insert(Builtin::Bitwise, count),
            BuiltinName::ec_op => builtin_instance_counter.insert(Builtin::EcOp, count),
            BuiltinName::keccak => builtin_instance_counter.insert(Builtin::Keccak, count),
            BuiltinName::poseidon => builtin_instance_counter.insert(Builtin::Poseidon, count),
            BuiltinName::segment_arena => {
                builtin_instance_counter.insert(Builtin::SegmentArena, count)
            }
            // TODO(DanB): what about the following?
            // BuiltinName::range_check96 => todo!(),
            // BuiltinName::add_mod => todo!(),
            // BuiltinName::mul_mod => todo!(),
            _ => {
                return Err(ExecutionError::UnknownBuiltin { builtin_name });
            }
        };
    }
    Ok(ExecutionResources {
        steps: u64_from_usize(vm_resources.n_steps),
        builtin_instance_counter,
        memory_holes: u64_from_usize(vm_resources.n_memory_holes),
        da_gas_consumed: StarknetApiGasVector { l1_gas, l2_gas, l1_data_gas },
        gas_consumed: StarknetApiGasVector::default(),
    })
}
