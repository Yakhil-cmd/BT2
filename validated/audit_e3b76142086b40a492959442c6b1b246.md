### Title
Stakers and Pool Members Cannot Execute Exit Actions When Contract Is Paused, Causing Temporary Freezing of Funds - (File: `src/staking/staking.cairo`, `src/pool/pool.cairo`)

### Summary
When the Starknet Staking contract is paused, both `unstake_action` and `exit_delegation_pool_action` are blocked by the `general_prerequisites()` pause check. This means stakers and pool members who have already submitted their exit intent, waited the full `exit_wait_window`, and whose funds have already been removed from the total stake, cannot retrieve their funds until the contract is unpaused by a privileged role (`security_admin`). This creates a state where user funds are temporarily frozen even though the protocol's intent-action design is meant to guarantee eventual fund retrieval after the exit window elapses.

### Finding Description

The Starknet Staking protocol uses an **intent-action** two-step withdrawal pattern:

1. **Intent phase** (`unstake_intent` / `exit_delegation_pool_intent`): The staker/pool member signals intent to exit. The funds are immediately removed from the total stake, and an `exit_wait_window` timer starts.
2. **Action phase** (`unstake_action` / `exit_delegation_pool_action`): After the exit window elapses, anyone can call the action to transfer the funds back to the staker/pool member.

The critical issue is that `unstake_action` calls `general_prerequisites()` as its very first step: [1](#0-0) 

The `general_prerequisites()` function enforces the pause check and reverts with `CONTRACT_IS_PAUSED` if the contract is paused. This is confirmed by the spec: [2](#0-1) 

Similarly, `remove_from_delegation_pool_action` (called internally by `exit_delegation_pool_action` in the Pool contract) also calls `general_prerequisites()`: [3](#0-2) 

And the spec confirms `exit_delegation_pool_action` also requires the contract to be unpaused: [4](#0-3) 

The pause and unpause functions are restricted to privileged roles: [5](#0-4) 

This means that once the contract is paused by the `security_agent`, only the `security_admin` can unpause it. Users who have already completed the intent phase and waited the full exit window have no way to retrieve their funds independently.

The test suite explicitly confirms this behavior: [6](#0-5) [7](#0-6) 

### Impact Explanation

**Temporary freezing of funds (High).**

A staker who has:
1. Called `unstake_intent()` — their own stake and delegated stake are already removed from `total_stake`
2. Waited the full `exit_wait_window` (default: 1 week, max: 12 weeks)
3. Is ready to call `unstake_action()`

...will have their STRK tokens locked inside the staking contract with no ability to retrieve them if the contract is paused. The funds are not earning rewards (they were removed from total stake at intent time), and the user cannot exit. The freeze lasts until the `security_admin` unpauses the contract. If governance is compromised, slow to act, or unavailable, the freeze is indefinite.

The same applies to pool members via `exit_delegation_pool_action`. [8](#0-7) 

The funds are already removed from total stake at intent time, so blocking the action phase serves no security purpose — it only harms users.

### Likelihood Explanation

**Realistic.** The `security_agent` can pause the contract at any time for any reason (e.g., suspected exploit, upgrade preparation). There is no coordination mechanism to ensure that stakers in the action-ready phase are protected. Given that the `exit_wait_window` is up to 12 weeks, there is a significant window during which a pause could occur after a staker's exit window has elapsed but before they call `unstake_action`.

### Recommendation

Allow `unstake_action` and `exit_delegation_pool_action` (and its underlying `remove_from_delegation_pool_action`) to bypass the pause check when the exit window has already elapsed. Since the funds were removed from total stake at intent time, allowing the action phase when paused does not compromise protocol security.

Concretely, `unstake_action` and `remove_from_delegation_pool_action` should skip the `general_prerequisites()` pause assertion (or have a dedicated check that only enforces the pause for non-exit-action calls). For example:

```cairo
fn unstake_action(ref self: ContractState, staker_address: ContractAddress) -> Amount {
    // Do NOT call general_prerequisites() here — allow execution even when paused
    // since funds were already removed from total stake at intent time.
    let mut staker_info = self.internal_staker_info(:staker_address);
    let unstake_time = staker_info
        .unstake_time
        .expect_with_err(Error::MISSING_UNSTAKE_INTENT);
    assert!(Time::now() >= unstake_time, "{}", GenericError::INTENT_WINDOW_NOT_FINISHED);
    // ... rest of logic
}
```

### Proof of Concept

1. Staker calls `unstake_intent()` — funds removed from total stake, exit timer starts.
2. `exit_wait_window` (e.g., 1 week) elapses — staker is now eligible to call `unstake_action`.
3. `security_agent` calls `pause()` on the staking contract.
4. Staker calls `unstake_action(staker_address)` — **reverts with "Contract is paused"**.
5. Staker's STRK tokens remain locked in the staking contract.
6. Staker has no recourse until `security_admin` calls `unpause()`.

This is directly confirmed by the existing test: [6](#0-5)

    impl StakingPauseImpl of IStakingPause<ContractState> {
        fn pause(ref self: ContractState) {
            self.roles.only_security_agent();
            if self.is_paused() {
                return;
            }
            self.is_paused.write(true);
            self.emit(PauseEvents::Paused { account: get_caller_address() });
        }

        fn unpause(ref self: ContractState) {
            self.roles.only_security_admin();
            if !self.is_paused() {
                return;
            }
            self.is_paused.write(false);
            self.emit(PauseEvents::Unpaused { account: get_caller_address() });
        }
