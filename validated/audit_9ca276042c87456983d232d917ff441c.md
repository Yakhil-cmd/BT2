Audit Report

## Title
Permanent DoS of index-ng Canister via `ic_cdk::trap()` in `credit()`/`debit()` Leaving `is_build_index_running` Permanently Set - (File: `rs/ledger_suite/icrc1/index-ng/src/main.rs`)

## Summary

The `credit()` and `debit()` functions in the ICRC-1 index-ng canister call `ic_cdk::trap()` on token amount overflow or underflow. Because `build_index()` commits `is_build_index_running = true` to stable state before its first `await` point, and because `ic_cdk::trap()` terminates Wasm execution without running Rust destructors (including the `scopeguard` that resets the flag), a malicious ledger canister can craft a single block that permanently halts the index-ng canister's indexing loop with no recovery path short of a canister upgrade.

## Finding Description

**`credit()` and `debit()` trap on attacker-controlled input:**

Both functions call `ic_cdk::trap()` when block amounts cause arithmetic overflow or underflow:

```rust
// line 1141-1143
balance.checked_sub(&amount).unwrap_or_else(|| {
    ic_cdk::trap(format!("Block {block_index} caused an underflow ..."));
})
// line 1149-1151
balance.checked_add(&amount).unwrap_or_else(|| {
    ic_cdk::trap(format!("Block {block_index} caused an overflow ..."))
})
```

The `amount` values come directly from blocks returned by the configured ledger canister — an externally controlled source.

**State committed before the first `await`:**

`build_index()` sets `is_build_index_running = true` and installs a `scopeguard` to reset it, then immediately suspends at `find_get_blocks_method().await`:

```rust
// lines 710-718
mutate_state(|state| { state.is_build_index_running = true; });
let _reset_is_build_index_running_flag_guard = guard((), |_| {
    mutate_state(|state| { state.is_build_index_running = false; });
});
let num_indexed = match find_get_blocks_method().await { ... };
```

In IC's execution model, state mutations before an `await` are committed when the outgoing inter-canister call is dispatched. The `is_build_index_running = true` write is committed at that point and is not subject to rollback by any later callback trap.

**Scopeguard destructor never runs on trap:**

`ic_cdk::trap()` invokes the `ic0.trap` Wasm host function, which terminates Wasm execution immediately without stack unwinding. Rust destructors — including the `scopeguard` guard — are never invoked. When a callback traps, the IC rolls back heap changes made during that callback to the pre-callback state, which already contains `is_build_index_running = true` (committed from the initial message). The flag remains `true` permanently.

**Timer is never rescheduled:**

The `set_build_index_timer` call only executes after `num_indexed` is resolved (lines 722-742). A trap inside the callback chain (`fetch_blocks_via_get_blocks` → `append_blocks` → `append_block` → `process_balance_changes` → `credit`/`debit`) abandons the future entirely. The timer reschedule code never runs.

**Every future invocation is a no-op:**

```rust
// lines 707-709
if with_state(|state| state.is_build_index_running) {
    return None;
}
```

With `is_build_index_running = true` permanently committed and no timer rescheduled, the canister stops indexing forever.

**The only reset path is the scopeguard** (confirmed by code search: one occurrence of `is_build_index_running = false` in the file, inside the scopeguard at line 715). No `post_upgrade` hook resets this flag.

**Full call chain:**
`build_index()` → `fetch_blocks_via_get_blocks().await` → callback → `append_blocks()` → `append_block()` → `process_balance_changes()` → `credit()`/`debit()` → `ic_cdk::trap()`

## Impact Explanation

This is a **High** severity application-level DoS. The index-ng canister permanently stops indexing blocks. All user-facing queries (`get_account_transactions`, `icrc1_balance_of`, `list_subaccounts`) return stale or empty data indefinitely. Any SNS project or token ecosystem relying on the index-ng canister for account history loses that service permanently until an operator performs a canister upgrade. This matches the allowed impact: *"Application/platform-level DoS... or subnet availability impact not based on raw volumetric DDoS"* and *"Significant... ledger... security impact with concrete user or protocol harm."*

## Likelihood Explanation

Any canister developer can deploy a malicious ICRC-1-compatible ledger canister on the IC. The attack requires no privileged access, no consensus corruption, and no key compromise — only that the target index-ng canister is configured to poll the attacker's ledger. The crafted block is trivial to construct (e.g., a `Mint` of `Tokens::MAX` to an account with any nonzero balance, or a `Burn` exceeding any account's balance). The attack is one-shot and irreversible without an operator upgrade. The constraint that the attacker must control the polled ledger canister is the primary limiting factor, placing this at the lower end of High.

## Recommendation

1. **Replace `ic_cdk::trap()` with `Result` propagation** in `credit()` and `debit()`. Return `Result<Tokens, SyncError>` and propagate errors up through `process_balance_changes()` → `append_block()` → `append_blocks()` as a non-retriable `SyncError`, allowing the existing error-handling path in `build_index()` to stop the timer gracefully without corrupting `is_build_index_running`.

2. **Reset `is_build_index_running` unconditionally in the timer wrapper**, not via a scopeguard. Schedule a follow-up one-shot timer or use a cleanup callback that resets the flag in a separate message after the async body completes, so a trap in any callback cannot leave the flag permanently set.

3. **Validate block amounts before processing**: before calling `credit()`, check `balance.checked_add(&amount).is_some()`; before calling `debit()`, check `balance >= amount`. Return a non-retriable `SyncError` on violation rather than trapping.

## Proof of Concept

1. Deploy a malicious ICRC-1 ledger canister `M` on the IC (or a local PocketIC instance).
2. Deploy an index-ng canister `I` with `ledger_id = M`.
3. `M` initially returns one valid `Mint` block crediting `account_A` with `1` token, so `I` indexes it normally.
4. `M` then returns a second `Mint` block: `{ to: account_A, amount: Tokens::MAX }`.
5. `I`'s timer fires → `build_index()` sets `is_build_index_running = true` → `find_get_blocks_method().await` commits state.
6. Callback fetches blocks from `M` → `append_block()` → `process_balance_changes()` → `credit(account_A, Tokens::MAX)` → `1 + Tokens::MAX` overflows → `ic_cdk::trap()`.
7. Callback state is rolled back; `is_build_index_running = true` remains committed; scopeguard never runs; timer never rescheduled.
8. Assert: all subsequent calls to `build_index()` return `None` immediately; `get_account_transactions` returns stale data permanently.

This is reproducible as a deterministic PocketIC integration test by controlling the mock ledger's `get_blocks` response.
