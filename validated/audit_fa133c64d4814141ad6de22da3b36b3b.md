Audit Report

## Title
Third-Party Caller Can Bypass `confirmation_text` Consent Gate and Force Participation Recording for Any Buyer — (File: `rs/sns/swap/canister/canister.rs`)

## Summary

The `refresh_buyer_tokens` endpoint in the SNS Swap canister accepts an arbitrary `buyer` principal without verifying that the caller equals the specified buyer. When an SNS is initialized with a required `confirmation_text`, any unprivileged third party can read that text from the public `get_init` endpoint and submit it on behalf of any buyer who has already deposited ICP, recording their participation and bypassing the explicit-consent gate the SNS deployer intended. Once participation is recorded, the buyer's ICP is locked in escrow and cannot be recovered via `error_refund_icp` until after the swap closes.

## Finding Description

**Root cause — no caller-equals-buyer check:**

In `rs/sns/swap/canister/canister.rs` at lines 128–143, when `arg.buyer` is non-empty the handler resolves the target principal directly from the request argument with no authorization check:

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    PrincipalId::from_str(&arg.buyer).unwrap()  // no assert caller == p
};
``` [1](#0-0) 

**Confirmation text is public and trivially obtainable:**

`validate_confirmation_text` in `rs/sns/swap/src/swap.rs` only compares the supplied string against the stored value; it does not check who is calling:

```rust
(Some(expected_text), Some(text)) => {
    if &text != expected_text { Err(...) } else { Ok(()) }
}
``` [2](#0-1) 

The `confirmation_text` is stored in the public `Init` struct and is returned by the `get_init` query endpoint, making it trivially readable by any observer. [3](#0-2) 

**Participation recording locks ICP in escrow:**

Once `refresh_buyer_token_e8s` records a buyer in `self.buyers`, `error_refund_icp` explicitly blocks refunds for that buyer while ICP is in escrow:

```rust
if let Some(buyer_state) = self.buyers.get(&source_principal_id.to_string()) {
    if let Some(transfer) = &buyer_state.icp
        && transfer.transfer_success_timestamp_seconds == 0
    {
        return ErrorRefundIcpResponse::new_precondition_error(format!(
            "ICP cannot be refunded as principal {} has {} ICP (e8s) in escrow", ...
        ));
    }
}
``` [4](#0-3) 

This is confirmed by the test comment: "If user2 has sent ICP in Error but never committed their tokens, i.e. never called `refresh_buyer_tokens` they should be able to get their funds back even after the swap is committed." [5](#0-4) 

**Design intent confirms third-party calling is intentional, but confirmation_text bypass is not:**

The system test at `rs/tests/nns/sns/lib/src/sns_deployment.rs` explicitly calls `refresh_buyer_tokens` for `wealthy_user_identity` from a `default_sns_agent`, confirming the third-party notification path is intentional for the no-confirmation-text case. [6](#0-5) 

However, the proto documentation states "a **participant** should send the confirmation text via `refresh_buyer_tokens`," implying the buyer themselves must supply consent — not an arbitrary third party. [7](#0-6) 

## Impact Explanation

When an SNS is deployed with `confirmation_text` set, the deployer's intent is that each buyer must explicitly confirm agreement before their ICP is committed. This mechanism is nullified: any unprivileged caller can supply the correct (public) confirmation text on behalf of any buyer who has deposited ICP, recording their participation without their knowledge or consent. If the swap subsequently commits, the buyer's ICP is swept to the SNS governance canister and they receive SNS tokens they never agreed to acquire. The `error_refund_icp` path is blocked for recorded buyers until after `sweep_icp` runs, so the buyer has no recourse during the open phase.

This constitutes **High** impact: significant SNS security impact with concrete user financial harm — a buyer's ICP is committed to a swap and their explicit-consent gate is bypassed, directly matching the allowed impact class "Significant... SNS... security impact with concrete user or protocol harm."

## Likelihood Explanation

**Medium.** The attacker requires only: (1) the target buyer's principal, derivable from their subaccount in the swap canister's public state; (2) the confirmation text, readable from the public `get_init` query; (3) the buyer to have already deposited ICP into their swap subaccount. No privileged access, key material, or subnet-majority corruption is needed. The attack is a single ingress update call, repeatable for any buyer in any open SNS swap that uses `confirmation_text`.

## Recommendation

Add a caller-equals-buyer assertion in the canister endpoint when `arg.buyer` is explicitly specified:

```rust
async fn refresh_buyer_tokens(arg: RefreshBuyerTokensRequest) -> RefreshBuyerTokensResponse {
    let caller = caller_principal_id();
    let p: PrincipalId = if arg.buyer.is_empty() {
        caller
    } else {
        let buyer = PrincipalId::from_str(&arg.buyer).unwrap();
        assert_eq!(caller, buyer,
            "Caller {} is not authorized to refresh tokens for buyer {}", caller, buyer);
        buyer
    };
    // ... rest unchanged
}
```

If third-party notification (without confirmation text) is a desired composability feature, split the endpoint: allow unauthenticated third-party calls only when no `confirmation_text` is required, and require `caller == buyer` whenever `confirmation_text` is present.

## Proof of Concept

1. An SNS is initialized with `confirmation_text = "I agree to the SNS swap terms"`.
2. Alice transfers 100 ICP to the swap canister's subaccount derived from her principal (a deliberate deposit, but she has not yet confirmed).
3. Bob (attacker) calls `swap.get_init()` (public query) to retrieve the confirmation text.
4. Bob submits an ingress update:
   ```
   refresh_buyer_tokens({
     buyer: "<Alice's principal>",
     confirmation_text: "I agree to the SNS swap terms"
   })
   ```
5. `validate_confirmation_text` passes (text matches). The swap reads Alice's subaccount balance (100 ICP) and records her in `self.buyers` with `amount_icp_e8s = 100 * E8`.
6. Alice's ICP is now in escrow. `error_refund_icp` returns a precondition error blocking any refund.
7. When the swap commits, `sweep_icp` transfers Alice's ICP to the SNS governance canister; Alice receives SNS tokens she never explicitly agreed to acquire.

A deterministic unit test can reproduce this by constructing a `SwapBuilder` with `with_confirmation_text(...)`, calling `refresh_buyer_token_e8s` with a different `buyer` principal than the simulated caller, and asserting the buyer appears in `self.buyers` and that `error_refund_icp` returns a precondition error.
