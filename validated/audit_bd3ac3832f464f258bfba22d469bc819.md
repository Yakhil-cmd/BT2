### Title
Pool Admin Can Instantly Raise Per-Bin Fees Without Timelock or Cap, Front-Running Traders - (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary

`setPoolBinAdditionalFees` imposes no upper-bound cap and no timelock on `addFeeBuyE6` / `addFeeSellE6`. A pool admin can raise per-bin fees to `type(uint16).max` (65 535 E6 ≈ 6.55 %) in the same block as a pending swap, causing traders to pay materially more than the fee rate they observed. This is the direct Metric OMM analog of the Audius `deployerCut` bug: a privileged party changes a fee parameter immediately before a critical operation with no delay for counterparties to react.

### Finding Description

`setPoolAdminFees` enforces two explicit caps and collects accrued fees before updating rates:

```solidity
// MetricOmmPoolFactory.sol:414-415
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (newAdminNotionalFeeE8 > maxAdminNotionalFeeE8) revert AdminFeeTooHigh();
``` [1](#0-0) 

`setPoolBinAdditionalFees`, by contrast, passes the caller-supplied values straight through to the pool with no cap check and no pre-collection step:

```solidity
// MetricOmmPoolFactory.sol:450-457
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
``` [2](#0-1) 

The pool-level `setBinAdditionalFees` only validates the bin index, not the fee magnitude:

```solidity
// MetricOmmPool.sol:464-474
function setBinAdditionalFees(int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external onlyFactory nonReentrant(PoolActions.SET_BIN_ADDITIONAL_FEES)
{
    if (bin < LOWEST_BIN || bin > HIGHEST_BIN) revert InvalidBinIndex(bin);
    BinState storage s = _binStates[bin];
    s.addFeeBuyE6 = addFeeBuyE6;
    s.addFeeSellE6 = addFeeSellE6;
    ...
}
``` [3](#0-2) 

During every swap the bin additional fee is added directly to the effective fee used in `SwapMath`:

```solidity
// MetricOmmPool.sol:910
params.baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6)
``` [4](#0-3) 

The same pattern applies to the sell-side path. [5](#0-4) 

The price-provider rotation path, by contrast, enforces a timelock before any change takes effect:

```solidity
// MetricOmmPoolFactory.sol:487
uint256 executeAfter = block.timestamp + timelock;
``` [6](#0-5) 

No equivalent delay exists for bin additional fees.

### Impact Explanation

`uint16` max = 65 535; in E6 units that is ≈ 6.55 % additional spread per bin. A pool admin who also controls the base spread fee at `maxAdminSpreadFeeE6` can therefore impose a total effective spread that exceeds the factory-enforced admin cap on a per-bin basis. More critically, because the change is instant, the admin can:

1. Observe a large pending swap in the mempool (or submit one themselves via a router).
2. Front-run it with `setPoolBinAdditionalFees(pool, activeBin, 65535, 65535)`.
3. The swap executes at the inflated fee; the surplus is split between admin and protocol according to the stored `adminSpreadFeeE6 / protocolSpreadFeeE6` ratio, so the admin captures their proportional share of the inflated amount.
4. Reset fees to normal in the same or next block.

Traders receive fewer output tokens than the fee rate they observed on-chain, constituting a direct loss of user funds on every affected swap.

### Likelihood Explanation

The pool admin is a semi-trusted role that is expected to operate within caps and timelocks. The price-provider rotation path already demonstrates the protocol's intent to impose timelocks on admin-controlled parameters that affect swap economics. The absence of any equivalent guard on `setPoolBinAdditionalFees` is an oversight rather than a design choice, making exploitation straightforward for any pool admin willing to act adversarially. No special conditions (oracle staleness, unusual tokens, etc.) are required.

### Recommendation

Apply the same two guards that `setPoolAdminFees` already uses:

1. **Cap**: Add a `maxAdminBinAdditionalFeeE6` factory constant (or reuse `maxAdminSpreadFeeE6`) and revert if either `addFeeBuyE6` or `addFeeSellE6` exceeds it.
2. **Timelock**: Require a propose-then-execute pattern (mirroring `proposePoolPriceProvider` / `executePoolPriceProviderUpdate`) so traders have time to exit before a fee increase takes effect.

Alternatively, enforce that `addFeeBuyE6 + spreadFeeE6 ≤ maxAdminSpreadFeeE6 + protocolSpreadFeeE6` so the combined effective fee per bin cannot exceed the factory-level cap.

### Proof of Concept

```
// 1. Pool deployed with spreadFeeE6 = 5_000 (0.5%), addFeeBuyE6 = 0 on all bins.
// 2. Trader signs a large exactInput swap; tx sits in mempool.
// 3. Pool admin calls:
factory.setPoolBinAdditionalFees(pool, activeBin, 65535, 65535);
// Effective buy fee for that bin: 5_000 + 65_535 = 70_535 E6 ≈ 7.05%
// 4. Trader's swap executes; they pay ~7.05% instead of 0.5%.
// 5. Admin calls setPoolBinAdditionalFees(pool, activeBin, 0, 0) to reset.
// Net: admin extracted ~6.55% of the swap notional from the trader with no on-chain warning.
``` [2](#0-1) [3](#0-2)
