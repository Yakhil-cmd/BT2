### Title
EIP-7702 Authorization List Intrinsic Pubdata Overcharge for Failed Entries — (`basic_bootloader/src/bootloader/transaction_flow/gas_helpers.rs`)

---

### Summary

When a user submits an EIP-7702 (type-4) transaction, ZKsync OS pre-charges `L2_TX_INTRINSIC_PUBDATA_PER_AUTHORIZATION = 80 bytes` of pubdata cost for **every** authorization entry, including entries that fail validation and produce **zero** actual pubdata. Because this intrinsic pubdata deduction is never refunded, users pay a pubdata fee proportional to the total number of authorization entries regardless of how many succeed. This is a resource-accounting bug directly analogous to the NFTX pattern: users pay for a per-item protocol operation whose actual output is near-zero.

---

### Finding Description

`calculate_l2_tx_intrinsic_pubdata` unconditionally multiplies the authorization list length by `L2_TX_INTRINSIC_PUBDATA_PER_AUTHORIZATION`:

```rust
// basic_bootloader/src/bootloader/transaction_flow/gas_helpers.rs
pub fn calculate_l2_tx_intrinsic_pubdata(authorization_list_num: u64, is_service: bool) -> u64 {
    ...
    intrinsic_pubdata = intrinsic_pubdata.saturating_add(
        authorization_list_num.saturating_mul(L2_TX_INTRINSIC_PUBDATA_PER_AUTHORIZATION),
    );
    intrinsic_pubdata
}
``` [1](#0-0) 

`L2_TX_INTRINSIC_PUBDATA_PER_AUTHORIZATION` is defined as 80 bytes (key + metadata + versioning + nonce + balance + bytecode fields): [2](#0-1) 

This value is passed into `create_resources_for_tx`, which deducts it from the user's native resource budget **upfront** before execution:

```rust
let intrinsic_pubdata_overhead = native_per_pubdata_byte.saturating_mul(intrinsic_pubdata);
let native_limit = match native_limit.checked_sub(intrinsic_pubdata_overhead) { ... };
``` [3](#0-2) 

The actual authorization processing in `validate_and_apply_delegation` returns `false` (no state change, no pubdata) for entries that fail any of: wrong chain ID, nonce overflow, bad `s` value, failed ecrecover, authority is a contract, or nonce mismatch: [4](#0-3) 

A failed authorization writes nothing to storage and produces **zero** pubdata. Yet the 80-byte intrinsic pubdata deduction for that entry is never returned. The post-execution refund path in `before_refund` adds `intrinsic_pubdata` to `total_pubdata_used` but only charges for *additional* execution pubdata beyond `validation_pubdata`; the upfront intrinsic deduction is already gone: [5](#0-4) 

The gas-cost side is handled correctly (a refund of `NEWACCOUNT − PER_AUTH_BASE_COST` ergs is issued for non-empty authorities), but **no analogous pubdata refund exists for failed entries**.

---

### Impact Explanation

For each failing authorization entry, the user is charged:

```
extra_native = 80 bytes × native_per_pubdata
             = 80 × (pubdata_price / native_price)
```

This native overconsumption propagates through the `deltaGas` mechanism into extra EVM gas charged:

```
extra_gas = extra_native / native_per_gas
          = 80 × pubdata_price / gas_price
```

At elevated L1 pubdata prices (e.g., `pubdata_price = 100 gwei/byte`, `gas_price = 1 gwei/gas`), each failed entry costs the user **8,000 extra gas**. With N failing entries, the total overcharge is `N × 8,000` gas — a direct, measurable loss of user funds with no corresponding protocol benefit. There is no cap on the authorization list length in the EIP-7702 transaction parser.

---

### Likelihood Explanation

Any unprivileged user can craft an EIP-7702 (type-4) transaction with an arbitrarily long authorization list where every entry is designed to fail (e.g., all entries carry a mismatched `chain_id`, or reference an authority whose nonce does not match). The transaction passes validation (the authorization failures are soft — they return `false`, not an error), so the block includes it and the overcharge is realized. No privileged access, governance action, or external oracle is required. The entry path is a standard signed transaction submitted to the sequencer.

---

### Recommendation

- **Short term**: After processing the authorization list, count the number of entries that actually succeeded (returned `true` from `validate_and_apply_delegation`) and refund `(authorization_list_num − successful_count) × L2_TX_INTRINSIC_PUBDATA_PER_AUTHORIZATION × native_per_pubdata` native resources to the transaction's resource pool.
- **Long term**: Split the intrinsic pubdata estimate into a per-entry component that is only committed once the entry is confirmed to have succeeded, mirroring the existing gas-refund mechanism for non-empty authorities.

---

### Proof of Concept

1. Alice constructs an EIP-7702 transaction with `N = 100` authorization entries. Each entry specifies `chain_id = 0xDEAD` (not the current chain). All entries will fail at step 1 of `validate_and_apply_delegation`.
2. The bootloader calls `calculate_l2_tx_intrinsic_pubdata(100, false)`, returning `L2_TX_INTRINSIC_PUBDATA + 100 × 80 = base + 8000` bytes.
3. `create_resources_for_tx` deducts `8000 × native_per_pubdata` from Alice's native budget upfront.
4. During validation, all 100 entries return `false`; no storage writes occur; actual pubdata from authorizations = 0 bytes.
5. Post-execution, `get_resources_to_charge_for_pubdata` measures only the real pubdata delta; the 8000-byte intrinsic deduction is not recovered.
6. Alice's `gas_used` is inflated by `8000 × pubdata_price / gas_price` gas units compared to a transaction with zero authorization entries performing identical EVM work — a direct, quantifiable overcharge with no corresponding state change.

pub fn calculate_l2_tx_intrinsic_pubdata(authorization_list_num: u64, is_service: bool) -> u64 {
