### Title
Permanent Loss of ERC-20 Tokens When `ExitToEthereum` Precompile's `withdraw` Promise Fails — (`engine-precompiles/src/native.rs`)

---

### Summary

The `ExitToEthereum` precompile burns a user's ERC-20 tokens on Aurora and then schedules a `withdraw` promise on the eth-connector NEP-141 contract. Unlike the `ExitToNear` precompile, `ExitToEthereum` attaches **no callback** to handle promise failure. If the eth-connector's `withdraw` is paused or fails for any reason after the burn is committed, the burned ERC-20 tokens are permanently unrecoverable.

---

### Finding Description

In `engine-precompiles/src/native.rs`, the `ExitToEthereum::run()` function handles both ETH and ERC-20 exits to Ethereum. For ERC-20 tokens, the flow is:

1. The ERC-20 contract (`EvmErc20.sol` / `EvmErc20V2.sol`) calls `_burn(_msgSender(), amount)` to destroy the user's tokens on Aurora.
2. The ERC-20 contract then calls the `ExitToEthereum` precompile at `0xb0bd02f6a392af548bdf1cfaee5dfa0eefcc8eab`.
3. The precompile schedules a `withdraw` promise on the NEP-141 eth-connector account.

The critical code path in `ExitToEthereum::run()`:

```rust
let withdraw_promise = PromiseCreateArgs {
    target_account_id: nep141_address,
    method: "withdraw".to_string(),
    args: serialized_args,
    attached_balance: Yocto::new(1),
    attached_gas: costs::WITHDRAWAL_GAS,
};

let promise = borsh::to_vec(&PromiseArgs::Create(withdraw_promise)).unwrap();
```

This uses `PromiseArgs::Create` — a fire-and-forget promise with **no callback**. If the `withdraw` call on the eth-connector fails, there is no mechanism to re-mint the already-burned ERC-20 tokens.

Contrast this with `ExitToNear`, which (when the `error_refund` feature is enabled) attaches a `PromiseArgs::Callback` pointing to `exit_to_near_precompile_callback`. That callback calls `engine::refund_on_error`, which re-mints the burned ERC-20 tokens if the `ft_transfer` promise fails. `ExitToEthereum` has no equivalent path, regardless of any feature flag.

The eth-connector's `withdraw` method is pausable. The `PAUSE_WITHDRAW` bitmask (`1 << 1`) exists precisely for this purpose, and the test `test_withdraw_from_near_pausability` confirms that when `engine_withdraw` is paused, the withdrawal fails with `"Pausable: Method is paused"`. If a user calls `withdrawToEthereum` while the eth-connector's `withdraw` is paused:

- The EVM transaction succeeds (the precompile is not paused, so the burn executes and the promise is scheduled).
- The NEAR promise executes asynchronously and fails.
- There is no callback to re-mint the burned tokens.
- The user's ERC-20 tokens are permanently destroyed with no corresponding release of NEP-141 tokens.

---

### Impact Explanation

**Critical — Permanent freezing/loss of user funds.**

The user's ERC-20 tokens (representing bridged NEP-141 assets) are burned on Aurora. The corresponding NEP-141 tokens remain locked in the eth-connector. There is no recovery path: the user cannot retry the exit (tokens are already burned), and there is no admin function to re-mint ERC-20 tokens for affected users. The funds are permanently lost.

---

### Likelihood Explanation

**High.** The eth-connector's `withdraw` is explicitly designed to be pausable (the `PAUSE_WITHDRAW` flag and `pa_pause_feature` admin method exist for this purpose). Any user who calls `withdrawToEthereum` during a pause window — even a brief one — permanently loses their tokens. Additionally, any transient failure in the `withdraw` promise (e.g., insufficient attached gas, contract-level bugs in the eth-connector) triggers the same permanent loss. The `ExitToEthereum` precompile is a standard user-facing bridge exit path, making this reachable by any unprivileged EVM user.

---

### Recommendation

Add a failure-handling callback to `ExitToEthereum` analogous to `exit_to_near_precompile_callback`. The callback should:

1. Check the promise result.
2. If the `withdraw` promise failed, call `engine::refund_on_error` to re-mint the burned ERC-20 tokens to the original sender.

This requires:
- Changing `PromiseArgs::Create` to `PromiseArgs::Callback` in `ExitToEthereum::run()`.
- Adding a new NEAR callback method (e.g., `exit_to_ethereum_precompile_callback`) that mirrors the logic in `exit_to_near_precompile_callback`.
- Passing the refund address (sender) and amount through the callback args, similar to how `EvmErc20V2.sol`'s `withdrawToNear` includes the sender address for `error_refund`.

---

### Proof of Concept

1. The eth-connector admin calls `pa_pause_feature` with `{"key": "engine_withdraw"}` (or the equivalent `withdraw` pause flag), pausing the eth-connector's `withdraw` method.
2. A user calls `withdrawToEthereum(recipient, amount)` on an `EvmErc20` contract on Aurora.
3. `EvmErc20.withdrawToEthereum` calls `_burn(_msgSender(), amount)` — the user's ERC-20 tokens are destroyed. [1](#0-0) 
4. `EvmErc20.withdrawToEthereum` calls the `ExitToEthereum` precompile. The precompile constructs a `PromiseArgs::Create` (no callback) targeting the eth-connector's `withdraw` method. [2](#0-1) 
5. The NEAR runtime executes the `withdraw` promise; it fails because `withdraw` is paused.
6. Because there is no callback, no re-mint occurs. The `ExitToNear` precompile's `PromiseArgs::Callback` path (which calls `refund_on_error` to re-mint) does not exist in `ExitToEthereum`. [3](#0-2) 
7. The user's ERC-20 tokens are permanently gone. The NEP-141 tokens remain locked in the eth-connector. [4](#0-3)
