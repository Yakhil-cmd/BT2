### Title
Unprivileged Caller Can Permanently Suppress Per-Block Reward Distribution via `update_rewards(disable_rewards: true)` — (File: `src/staking/staking.cairo`)

---

### Summary

The `update_rewards` function in the Staking contract writes the global `last_reward_block` state variable **before** checking the `disable_rewards` flag. Because `update_rewards` is callable by any non-zero address, an attacker can call it with `disable_rewards: true` to consume the per-block reward slot without distributing any rewards, permanently destroying that block's yield for all stakers.

---

### Finding Description

`update_rewards` is a public function gated only by `general_prerequisites()`, which checks that the contract is unpaused and the caller is non-zero — no role or identity check is performed. [1](#0-0) 

Inside the function, the global `last_reward_block` is written unconditionally **before** the `disable_rewards` branch: [2](#0-1) 

`general_prerequisites` only enforces: [3](#0-2) 

Because `last_reward_block` is a **global** (not per-staker) gate: [4](#0-3) 

…only **one** call per block can pass the `current_block_number > last_reward_block` assertion. Once that slot is consumed by an attacker with `disable_rewards: true`, every subsequent call in the same block reverts with `REWARDS_ALREADY_UPDATED`, and the block's rewards are permanently unrecoverable.

The analog to the external report is exact: just as `depositTokens()` performed an accounting state change on an unvalidated `treeIndex` (operating on zero-initialized storage), `update_rewards` performs a critical global state change (`last_reward_block.write`) without validating that the caller is authorized to suppress reward distribution. In both cases, the missing input validation allows an unprivileged caller to corrupt protocol accounting.

---

### Impact Explanation

An attacker calling `update_rewards(any_valid_staker, disable_rewards: true)` once per block permanently destroys that block's consensus-phase rewards for every staker and pool in the protocol. Repeated across blocks, this constitutes **permanent freezing of unclaimed yield** — the rewards are never minted into `unclaimed_rewards` and can never be claimed.

Matches allowed impact: **High — Permanent freezing of unclaimed yield.**

---

### Likelihood Explanation

- Requires no capital, no special role, no leaked key.
- Any EOA can execute the attack.
- The attacker only needs to submit a transaction before the legitimate block-reward caller in each block.
- On Starknet, transaction ordering within a block is deterministic and front-running is feasible via mempool observation or simply racing.

Likelihood: **High.**

---

### Recommendation

Move `last_reward_block.write(current_block_number)` to **after** the `disable_rewards` guard, so the global slot is only consumed when rewards are actually distributed:

```cairo
if disable_rewards || self.is_pre_consensus() {
    return;
}
// Update last block rewards only when rewards are actually distributed.
self.last_reward_block.write(current_block_number);
// ... rest of reward calculation
```

Alternatively, restrict who may pass `disable_rewards: true` to a privileged role (e.g., the block proposer or attestation contract).

---

### Proof of Concept

1. Consensus rewards are active (`is_pre_consensus()` returns `false`).
2. Attacker calls `update_rewards(valid_staker_address, disable_rewards: true)` in block N.
3. Execution path: `general_prerequisites()` passes → staker existence and activity checks pass → `last_reward_block.write(N)` executes → `disable_rewards` branch returns early — **no rewards distributed**.
4. Legitimate block-reward caller attempts `update_rewards(valid_staker_address, disable_rewards: false)` in block N → assertion `current_block_number > last_reward_block` fails → reverts with `REWARDS_ALREADY_UPDATED`. [4](#0-3) 

5. Block N's rewards are permanently lost. Attacker repeats every block to drain all future yield.
