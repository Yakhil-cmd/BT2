### Title
WebAuthn Transaction Authentication Bypass via Missing Origin and rpIdHash Validation - (File: `model/flow/webauthn.go`)

### Summary

The `validateWebAuthNExtensionData` function in `model/flow/webauthn.go` validates the WebAuthn challenge binding to the transaction payload but omits two mandatory WebAuthn spec checks: it never validates the `origin` field from `CollectedClientData` against any expected relying party origin, and it only rejects `rpIdHash` if it equals `TransactionDomainTag` rather than verifying it against an expected relying party ID hash. An unprivileged attacker who can present a crafted WebAuthn challenge to a victim can obtain a valid WebAuthn assertion from any origin and use it to authorize a malicious Flow transaction signed by the victim's registered key, bypassing the intent-binding that origin and rpId enforcement are designed to provide.

### Finding Description

`validateWebAuthNExtensionData` is the sole validation path for the `WebAuthnScheme` authentication scheme introduced in FLIP 264. It is called from `ValidateExtensionDataAndReconstructMessage`, which is invoked both at the Access/Collection node layer (`checkSignatureFormat`) and at FVM execution time inside `signatureContinuation.verify()`.

The function performs the following checks:

- Decodes the RLP-encoded `WebAuthnExtensionData` ✓
- Verifies `clientData.Type == "webauthn.get"` ✓
- Verifies `clientData.Challenge == SHA256(TransactionDomainTag || payload)` ✓
- Verifies `len(authenticatorData) >= 37` ✓
- Verifies `rpIdHash != TransactionDomainTag` ✓ (negative check only)
- Validates user-presence and backup flags ✓

What it **does not** do:

1. **No origin validation.** `clientData.Origin` is parsed into `CollectedClientData.Origin` but is never read or compared against any expected value. The W3C WebAuthn Level 3 spec (§7.2 step 7) and FLIP 264 both require the relying party to assert that `C.origin` matches the expected application origin.

2. **No rpIdHash validation.** `rpIdHash` (bytes 0–31 of `authenticatorData`) is only checked to be non-equal to `TransactionDomainTag`. Any other 32-byte value is accepted unconditionally. The spec requires `rpIdHash == SHA-256(expectedRpId)`. [1](#0-0) [2](#0-1) 

The reconstructed message returned to the caller is `authenticatorData || SHA256(clientDataJSON)`, which is then verified against the account's registered public key in `VerifySignatureFromTransaction`. Because the signature check passes (the user's hardware key did sign), the transaction is accepted as fully authorized. [3](#0-2) 

### Impact Explanation

An attacker who can present a WebAuthn challenge to a victim can obtain a valid assertion from **any** origin and use it to authorize a malicious Flow transaction:

1. Victim has a WebAuthn hardware key (e.g., YubiKey) registered on their Flow account.
2. Attacker controls `https://evil.com` and has the same public key registered there (or tricks the victim into registering it).
3. Attacker computes `challenge = SHA256(TransactionDomainTag || maliciousTransactionPayload)` and presents it as a WebAuthn authentication request on `evil.com`.
4. Victim authenticates (user-presence gesture on hardware key), producing a valid assertion with `origin: "https://evil.com"` and `rpIdHash = SHA256("evil.com")`.
5. Attacker submits the malicious transaction with the WebAuthn extension data containing this assertion.
6. `validateWebAuthNExtensionData` accepts it: challenge matches, flags are valid, rpIdHash ≠ TransactionDomainTag.
7. `VerifySignatureFromTransaction` confirms the cryptographic signature is valid.
8. The transaction executes with full authorization of the victim's account — unauthorized token transfers, contract deployments, or capability grants become possible.

The missing origin check is the direct analog to the Android CryptoObject omission: the challenge binds the assertion to the transaction (like the CryptoObject binds to the key), but without origin enforcement the binding to the *intended application* is absent, enabling cross-origin credential reuse.

**Impact:** Transaction authorization bypass → unauthorized account mutation (token transfer, contract deployment, capability grant). Impact: **High**. Likelihood: **Low-Medium** (requires victim to have a WebAuthn key registered on Flow and to be phished into authenticating with a crafted challenge). [4](#0-3) 

### Likelihood Explanation

WebAuthn hardware keys (YubiKey, passkeys synced via iCloud/Google) are increasingly used across multiple services. A user who registers the same hardware key on both a Flow wallet and an attacker-controlled site is a realistic target. The attacker only needs to present a 32-byte challenge — indistinguishable from a normal login prompt — to obtain the assertion. No privileged access, no staked node, and no key compromise is required. The attack is fully reachable via the public Access API gRPC/REST endpoint (`SendTransaction`).

### Recommendation

1. **Validate `origin`:** Define an allowlist of expected origins for Flow WebAuthn credentials (e.g., the canonical wallet domain). In `validateWebAuthNExtensionData`, after unmarshalling `clientData`, assert `clientData.Origin` is in the allowlist. If Flow intentionally supports multiple origins, require the origin to be explicitly registered alongside the account key.

2. **Validate `rpIdHash`:** Define the expected relying party ID(s) for Flow. Assert `rpIdHash == SHA256(expectedRpId)` rather than only rejecting the `TransactionDomainTag` value.

3. **Bind rpId to account key registration:** When a WebAuthn public key is added to a Flow account, record the expected `rpIdHash` alongside it and verify it matches on every transaction. [5](#0-4)

// Attacker-controlled software authenticator signs with arbitrary origin and rpIdHash.
// All values below are attacker-chosen; only the challenge is constrained.

maliciousTx := buildTransferTransaction(victim, attacker, 1000_FLOW)
payload    := maliciousTx.EnvelopeMessage()

// Challenge must match — attacker computes it honestly
challenge := SHA256(TransactionDomainTag || payload)   // 32 bytes

clientDataJSON := `{"type":"webauthn.get","challenge":"<base64url(challenge)>","origin":"https://evil.com"}`

// rpIdHash = SHA256("evil.com") — any value except TransactionDomainTag is accepted
rpIdHash   := SHA256([]byte("evil.com"))               // 32 bytes
userFlags  := byte(0x01)                               // UP set, no extensions
sigCounter := [4]byte{}
authenticatorData := concat(rpIdHash, userFlags, sigCounter)

// Victim's hardware key signs authenticatorData || SHA256(clientDataJSON)
// (victim sees only a generic WebAuthn prompt on evil.com)
sig := victimHardwareKey.Sign(concat(authenticatorData, SHA256(clientDataJSON)))

extensionData := WebAuthnExtensionData{AuthenticatorData: authenticatorData, ClientDataJson: clientDataJSON}
txSig := TransactionSignature{
    Address:       victim,
    KeyIndex:      0,
    Signature:     sig,
    ExtensionData: concat([]byte{WebAuthnScheme}, RLPEncode(extensionData)),
}
maliciousTx.EnvelopeSignatures = []TransactionSignature{txSig}

// validateWebAuthNExtensionData returns (true, message) — origin never checked
// VerifySignatureFromTransaction returns true — signature is cryptographically valid
// Transaction executes with victim's authorization
accessNode.SendTransaction(maliciousTx)  // succeeds

func validateWebAuthNExtensionData(extensionData []byte, payload []byte) (bool, []byte) {
	// See FLIP 264 for more details
	if len(extensionData) == 0 {
		return false, nil
	}
	rlpEncodedWebAuthnData := extensionData[1:]
	decodedWebAuthnData := &WebAuthnExtensionData{}
	if err := rlp.DecodeBytes(rlpEncodedWebAuthnData, decodedWebAuthnData); err != nil {
		return false, nil
	}

	clientData, err := decodedWebAuthnData.GetUnmarshalledCollectedClientData()
	if err != nil {
		return false, nil
	}

	// base64url decode the challenge, as that's the encoding used client side according to https://www.w3.org/TR/webauthn-3/#dictionary-client-data
	clientDataChallenge, err := base64.RawURLEncoding.DecodeString(clientData.Challenge)
	if err != nil {
		return false, nil
	}

	if strings.Compare(clientData.Type, WebAuthnTypeGet) != 0 || len(clientDataChallenge) != webAuthnChallengeLength {
		// invalid client data
		return false, nil
	}

	// make sure the challenge is the hash of the transaction payload
	hasher := hash.NewSHA2_256()
	_, err = hasher.Write(TransactionDomainTag[:])
	if err != nil {
		return false, nil
	}
	_, err = hasher.Write(payload)
	if err != nil {
		return false, nil
	}
	computedChallenge := hasher.SumHash()
	if !computedChallenge.Equal(clientDataChallenge) {
		return false, nil
	}

	// Validate authenticatorData
	if len(decodedWebAuthnData.AuthenticatorData) < webAuthnExtensionDataMinimumLength {
		return false, nil
	}

	// extract rpIdHash, userFlags, sigCounter, extensions
	rpIdHash := decodedWebAuthnData.AuthenticatorData[:webAuthnChallengeLength]
	userFlags := decodedWebAuthnData.AuthenticatorData[webAuthnChallengeLength]
	extensions := decodedWebAuthnData.AuthenticatorData[webAuthnExtensionDataMinimumLength:]
	if bytes.Equal(TransactionDomainTag[:], rpIdHash) {
		return false, nil
	}

	// validate user flags according to FLIP 264
	if err := validateFlags(userFlags, extensions); err != nil {
		return false, nil
	}

	clientDataHash := hash.NewSHA2_256().ComputeHash(decodedWebAuthnData.ClientDataJson)

	return true, slices.Concat(decodedWebAuthnData.AuthenticatorData, clientDataHash)
}

func (entry *signatureContinuation) verify() errors.CodedError {
	if entry.invokedVerify {
		return entry.verifyErr
	}

	entry.invokedVerify = true

	valid, message := entry.ValidateExtensionDataAndReconstructMessage(entry.payload)
	if !valid {
		entry.verifyErr = entry.newError(fmt.Errorf("signature extension data is not valid"))
		return entry.verifyErr
	}

	valid, err := crypto.VerifySignatureFromTransaction(
		entry.Signature,
		message,
		entry.accountKey.PublicKey,
		entry.accountKey.HashAlgo,
	)
	if err != nil {
		entry.verifyErr = entry.newError(err)
	} else if !valid {
		entry.verifyErr = entry.newError(fmt.Errorf("signature is not valid"))
	}

	return entry.verifyErr
}
