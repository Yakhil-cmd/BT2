### Title
Missing Lower-Bound Threshold Validation in `assert_sign_inputs` Allows `threshold = 1` in FROST Signing — (File: `src/frost/mod.rs`)

---

### Summary

`assert_key_invariants` (used by `keygen` and `reshare`) enforces **both** an upper bound (`threshold > participants.len()`) and a lower bound (`threshold < 2`) on the threshold parameter. `assert_sign_inputs` (used by every FROST signing entry point) enforces **only** the upper bound. A caller can pass `threshold = 1` to `sign_v1`, `sign_v2`, or `redjubjub::sign` and the library will accept it, silently violating the minimum-threshold invariant that keygen enforces.

---

### Finding Description

`assert_key_invariants` in `src/dkg.rs` validates both directions:

```rust
// src/dkg.rs  lines 573-582
if threshold > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
if threshold < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
}
```

`assert_sign_inputs` in `src/frost/mod.rs` validates only the upper bound:

```rust
// src/frost/mod.rs  lines 144-150
if threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
// ← no lower-bound check; threshold = 1 is silently accepted
```

`sign_v1`, `sign_v2` (EdDSA), and `redjubjub::sign` all delegate to `assert_sign_inputs` and then forward the caller-supplied `threshold` directly into the FROST aggregation machinery:

```rust
// src/frost/eddsa/sign.rs  lines 46-60
let threshold = threshold.into();
let participants = assert_sign_inputs(participants, threshold, me, coordinator)?;
// threshold is then passed to fut_wrapper_v1 → KeyPackage (min_signers = threshold)
```

```rust
// src/frost/redjubjub/sign.rs  lines 49-64
let threshold = threshold.into();
let participants = assert_sign_inputs(participants, threshold, me, coordinator)?;
// threshold forwarded to fut_wrapper → KeyPackage (min_signers = threshold)
```

The FROST `KeyPackage` built inside the signing future uses `threshold` as `min_signers`. With `threshold = 1`, the FROST aggregation layer requires only **one** signature share to produce an output, regardless of how many participants are present or what threshold was used during keygen.

The `frost::presign` entry point has the same gap:

```rust
// src/frost/mod.rs  lines 71-77
if args.threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
// no lower-bound check
```

---

### Impact Explanation

When a malicious coordinator calls `sign_v1` / `sign_v2` / `redjubjub::sign` with `threshold = 1`:

1. `assert_sign_inputs` accepts the call.
2. The FROST `KeyPackage` is constructed with `min_signers = 1`.
3. The coordinator can aggregate using only its own signature share (the FROST library's aggregation check passes because `1 >= min_signers`).
4. The resulting signature is cryptographically invalid: a single share `x_i` does not equal the full secret `x`, so `s_i · G ≠ R + c · X`.
5. All honest participants have already committed their nonces and sent their signature shares; they receive back a signature that fails external verification.

This corrupts the signing output — honest parties complete the protocol and accept a transcript that produces an unusable signature — matching the **High** impact category: *Corruption of sign outputs so honest parties accept inconsistent or unusable cryptographic outputs*.

Additionally, because the threshold is a free caller-supplied parameter (not bound to the keygen output), a malicious participant who is not the coordinator can also trigger this by calling their local `sign_v1` with `threshold = 1`, causing their local signing instance to misbehave and potentially stall the protocol.

---

### Likelihood Explanation

The threshold parameter is passed directly by the library caller at every signing invocation. There is no binding between the threshold stored in `KeygenOutput` and the threshold supplied to signing — the library trusts the caller to pass the correct value. Any participant acting as coordinator, or any library integrator who makes an off-by-one error (e.g., passing `max_malicious` instead of `max_malicious + 1`), can reach `threshold = 1`. The missing check is a single-line omission in a shared validation helper that is called by three separate signing entry points.

---

### Recommendation

Add the same lower-bound guard that `assert_key_invariants` already enforces:

```rust
// src/frost/mod.rs — assert_sign_inputs
pub fn assert_sign_inputs(
    participants: &[Participant],
    threshold: impl Into<ReconstructionLowerBound>,
    me: Participant,
    coordinator: Participant,
) -> Result<ParticipantList, InitializationError> {
    let threshold = threshold.into();
    // ... existing checks ...

    // validate threshold upper bound
    if threshold.value() > participants.len() {
        return Err(InitializationError::ThresholdTooLarge {
            threshold: threshold.value(),
            max: participants.len(),
        });
    }

+   // validate threshold lower bound — mirrors assert_key_invariants
+   if threshold.value() < 2 {
+       return Err(InitializationError::ThresholdTooSmall {
+           threshold: threshold.value(),
+           min: 2,
+       });
+   }

    // ...
}
```

Apply the same fix to `frost::presign` in `src/frost/mod.rs` (line 72 area).

### Proof of Concept

```
1. Run keygen with threshold = 2, participants = [A, B, C].
   → assert_key_invariants enforces threshold >= 2. Succeeds.

2. Malicious coordinator calls sign_v1(
       participants = [A, B, C],
       threshold    = 1,          // ← below the keygen minimum
       me           = A,
       coordinator  = A,
       keygen_output, message, rng
   )
   → assert_sign_inputs: threshold(1) <= participants.len()(3) → no error.
   → KeyPackage built with min_signers = 1.

3. Coordinator A collects nonce commitments from B and C (presign round).

4. Coordinator A sends signing package to B and C.

5. B and C compute and return their signature shares.

6. Coordinator A calls FROST aggregate() with only its own share (1 >= min_signers=1).
   → aggregate() returns a "signature".

7. The signature fails external Ed25519 verification because
   s_A · G ≠ R + c · X  (x_A ≠ x).

8. Honest parties B and C have completed the protocol and believe signing succeeded,
   but the output is unusable.
```

**Root cause**: `assert_sign_inputs` (`src/frost/mod.rs`, line 144) checks only `threshold > participants.len()` and omits the `threshold < 2` guard present in `assert_key_invariants` (`src/dkg.rs`, lines 580–582). All three FROST signing entry points (`sign_v1`, `sign_v2`, `redjubjub::sign`) delegate to `assert_sign_inputs` without any additional threshold floor check.
