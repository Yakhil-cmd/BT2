Audit Report

## Title
Stale ICP/XDR Exchange Rate Used in Cycles Minting Without Freshness Validation - (File: `rs/nns/cmc/src/main.rs`)

## Summary
The `tokens_to_cycles` function in `rs/nns/cmc/src/main.rs` reads `state.icp_xdr_conversion_rate` and uses only `rate.xdr_permyriad_per_icp`, never checking `rate.timestamp_seconds`. If the Exchange Rate Canister (XRC) becomes unavailable for any period, the CMC silently continues minting cycles at the last cached rate indefinitely. An unprivileged user who observes that the CMC's published rate timestamp is stale and that the ICP/XDR market price has diverged can call `notify_top_up`, `notify_create_canister`, or `notify_mint_cycles` to receive cycles at the wrong rate, constituting illegal minting at the protocol level.

## Finding Description

**Root cause:** `tokens_to_cycles` (L1900–1922, `rs/nns/cmc/src/main.rs`) extracts only `rate.xdr_permyriad_per_icp` from the stored rate and never inspects `rate.timestamp_seconds`:

```rust
fn tokens_to_cycles(amount: Tokens) -> Result<Cycles, NotifyError> {
    with_state(|state| {
        let xdr_permyriad_per_icp = state
            .icp_xdr_conversion_rate
            .as_ref()
            .map(|rate| rate.xdr_permyriad_per_icp);  // timestamp never checked
        match xdr_permyriad_per_icp {
            Some(xdr_permyriad_per_icp) => Ok(TokensToCycles { ... }.to_cycles(amount)),
            None => Err(...),
        }
    })
}
```

**Refresh scheduling vs. use-time gating:** `REFRESH_RATE_INTERVAL_SECONDS` (5 minutes, `rs/nns/cmc/src/exchange_rate_canister.rs` L16) controls only when the heartbeat schedules the next XRC call. It is never consulted at the point of conversion. When the XRC returns errors, `schedule_next_attempt` reschedules at the next minute (`exchange_rate_canister.rs` L153–160) but leaves `state.icp_xdr_conversion_rate` unchanged. The stored rate can therefore be arbitrarily old.

**`do_set_icp_xdr_conversion_rate` does not enforce maximum age:** It only rejects a new rate whose timestamp is not strictly greater than the current one (`main.rs` L1022–1030). It imposes no upper bound on how old the stored rate may be.

**All three public conversion paths call `tokens_to_cycles` unconditionally:**
- `process_top_up` → L1991
- `process_create_canister` → L1932
- `process_mint_cycles` → L1965

**Governance fallback weakened:** The `IcpXdrConversionRate` NNS proposal type has been marked obsolete and removed from submission (`rs/nns/governance/CHANGELOG.md`, Proposal 136693: "The `IcpXdrConversionRate` proposal is now obsolete and cannot be submitted"). While the Governance canister can still call `set_icp_xdr_conversion_rate` via a generic canister-call proposal, this requires a full NNS governance cycle (days), during which the stale rate remains in use.

**Integration test confirms the behavior:** `rs/nns/integration_tests/src/cycles_minting_canister_with_exchange_rate_canister.rs` L179–185 explicitly asserts that after an XRC error, the CMC's rate timestamp stays at the previous value and the rate value is unchanged — the CMC continues operating with the stale rate.

## Impact Explanation

**Impact class:** High ($2,000–$10,000) — Significant XRC/NNS/CMC financial integrity impact with concrete user and protocol harm.

During any XRC outage (canister upgrade, subnet disruption, or sustained error responses), the CMC continues minting cycles at the last cached ICP/XDR rate. ICP/XDR is volatile. A user who observes that the CMC's `get_icp_xdr_conversion_rate` timestamp is stale and that the current ICP/XDR market rate has dropped below the cached value can send ICP and receive more cycles than the current rate warrants. Cycles are the fundamental compute resource on the IC and are minted by burning ICP; receiving excess cycles at a stale high rate constitutes illegal minting — ICP is burned at the wrong rate relative to cycles issued, directly harming protocol economic integrity. The inverse (users receiving fewer cycles during a price spike) also constitutes incorrect accounting. The magnitude scales with outage duration and price volatility.

## Likelihood Explanation

Any XRC outage triggers the condition. XRC canister upgrades are routine governance operations and cause temporary unavailability. The CMC's heartbeat will fail to update the rate during this window and retry every minute, but the stored rate remains stale. The staleness is publicly observable via the `get_icp_xdr_conversion_rate` query (no authentication required). Any unprivileged principal can call `notify_top_up`, `notify_create_canister`, or `notify_mint_cycles` at any time. No special role, leaked key, or social engineering is required. The attacker only needs to observe the stale timestamp and a favorable price divergence, then submit an ICP transfer and notification. The attack is repeatable for the entire duration of the outage.

## Recommendation

In `tokens_to_cycles`, compare `rate.timestamp_seconds` against the current canister time (`ic_cdk::api::time() / 1_000_000_000`). If the age exceeds a defined maximum staleness threshold (e.g., a small multiple of `REFRESH_RATE_INTERVAL_SECONDS`, or a fixed bound such as 1 hour), return a `NotifyError` rather than proceeding with the conversion. This enforces the freshness intent of `REFRESH_RATE_INTERVAL_SECONDS` at the point of use, not just at the point of refresh scheduling. The error message should indicate that the rate is temporarily unavailable due to staleness, allowing callers to retry once the rate is refreshed.

## Proof of Concept

1. Deploy a local replica with the CMC and a mock XRC (as done in `rs/nns/integration_tests/src/cycles_minting_canister_with_exchange_rate_canister.rs`).
2. Allow the CMC to receive an initial rate from the XRC (e.g., `xdr_permyriad_per_icp = 200_000`, `timestamp_seconds = T`).
3. Reinstall the mock XRC to return `ExchangeRateError::StablecoinRateTooFewRates` (as in the existing integration test at L165–171).
4. Advance time by several hours. Confirm via `get_icp_xdr_conversion_rate` that `timestamp_seconds` remains at `T` while wall-clock time has advanced significantly.
5. Simulate a price drop by noting that the "true" rate should now be lower (e.g., `100_000`).
6. Call `notify_top_up` or `notify_mint_cycles` with an ICP amount. Observe that the CMC mints cycles using `200_000` (the stale rate) rather than `100_000`, yielding 2× the correct cycle amount.
7. The existing integration test at L179–185 already asserts this behavior (stale timestamp, unchanged rate value after XRC error), confirming the exploit precondition is reproducible with the existing test infrastructure.
