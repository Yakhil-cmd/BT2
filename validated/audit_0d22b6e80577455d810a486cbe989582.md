I found a direct analog. Let me confirm the exact field access discrepancy between the two offer viewer components.### Title
NFT Offer Validity Check Always Silently Returns False Due to Wrong Response Field Access — (File: `packages/gui/src/components/offers/NFTOfferViewer.tsx`)

---

### Summary

In `NFTOfferDetails`, the `checkOfferValidity` mutation result is read via `response.data?.valid` instead of the correct `response.valid`. Because the unwrapped RTK Query mutation response has no `data` property, this expression is always `undefined`, which evaluates to `false`. Every imported NFT offer is therefore permanently marked invalid, and the Accept button is unconditionally disabled.

---

### Finding Description

The `checkOfferValidity` RTK Query mutation is defined in `packages/api-react/src/services/wallet.ts`:

```ts
checkOfferValidity: mutation(build, WalletService, 'checkOfferValidity', {
  transformResponse: (response) => ({
    ...response,
    id: normalizeHex(response.id),
  }),
}),
```

`transformResponse` spreads the raw RPC response (`{ success, valid, id }`) and normalises `id`. After `.unwrap()`, the caller receives a flat object `{ success, valid, id }` — there is no nested `data` key.

In `NFTOfferViewer.tsx` (`NFTOfferDetails`), the validity check reads:

```ts
valid = response.data?.valid === true;   // response.data is always undefined
```

`response.data` is always `undefined`, so `valid` stays `false` and `setIsValid(false)` is always called via the `finally` block.

The parallel component `OfferBuilderViewer.tsx` performs the identical call correctly:

```ts
setIsValid(response.valid === true);     // correct field path
```

The initial state is `useState<boolean>(tradeRecord !== undefined)`. For imported offers `tradeRecord` is `undefined`, so `isValid` starts `false`. After the validity check runs and writes `false` again, the state is permanently `false` regardless of the actual on-chain validity of the offer.

The Accept button is gated on this state:

```ts
disabled={!isValid || isMissingRequestedAsset || isLoading}
```

And the offer header shows the invalid banner:

```ts
isInvalid={!isCheckOfferValidityLoading && !isValid}
```

Both are permanently triggered for every imported NFT offer.

---

### Impact Explanation

Every valid NFT offer imported through `NFTOfferViewer` is displayed with the wrong status ("invalid") and the Accept button is permanently disabled. A user who receives a valid NFT offer — for example, an NFT-for-XCH or NFT-for-CAT trade — cannot accept it through the GUI. The offer state presented to the user is corrupted: the RPC correctly reports `valid: true`, but the GUI discards that result and shows `invalid`. This fits the High impact category: corruption of offer state that causes the user to display the wrong status for an NFT offer.

---

### Likelihood Explanation

This code path is exercised every time a user opens an imported NFT offer (i.e., any offer file or URL passed to `NFTOfferViewer` with `imported=true` and no pre-existing `tradeRecord`). The bug is deterministic and 100% reproducible — no special conditions are required.

---

### Recommendation

In `packages/gui/src/components/offers/NFTOfferViewer.tsx`, line 414, change:

```ts
valid = response.data?.valid === true;
```

to:

```ts
valid = response.valid === true;
```

This matches the correct field access already used in `OfferBuilderViewer.tsx` line 137. Additionally, the two offer-viewer components should share a single validity-check utility to prevent this class of near-duplicate divergence from recurring.

---

### Proof of Concept

1. Obtain any valid NFT offer string (e.g., from a counterparty or a marketplace URL).
2. Open the Chia GUI and navigate to **Offers → View Offer File**, importing the offer string.
3. The offer is routed to `NFTOfferViewer` → `NFTOfferDetails`.
4. `checkOfferValidity` is called; the RPC returns `{ valid: true, id: "...", success: true }`.
5. The GUI reads `response.data?.valid` → `undefined` → `false`; `setIsValid(false)` is called.
6. The offer header displays the "invalid" banner; the **Accept Offer** button is `disabled`.
7. The user cannot accept the offer regardless of its actual on-chain validity.
