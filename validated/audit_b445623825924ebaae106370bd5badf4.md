### Title
Stale-Closure Race Condition in NFT Ownership Resolution Causes Misdirected On-Chain Notification Spend - (File: `packages/gui/src/hooks/useResolveNFTOffer.ts`)

### Summary

`useResolveNFTOffer` fires all async NFT ownership checks concurrently inside a `forEach` loop without awaiting them, and each callback closes over stale React state. When an offer contains multiple NFTs on the same side (e.g., two requested NFTs), concurrent callbacks overwrite each other's state updates. The result is that the wrong NFT ID ends up in `unownedNFTIds`, which is then used as the `notificationDestination` in the offer-sharing dialogs. The user is presented with a "Notify Current Owner" button that, when clicked, sends an on-chain XCH payment to the wrong NFT holder's address.

### Finding Description

In `useResolveNFTOffer`, the `useMemo` callback iterates over all NFTs in the offer using `forEach(async ...)`. Because `forEach` does not await async callbacks, all `checkNFTOwnership` calls are dispatched concurrently. Each callback closes over the React state values (`ownedNFTIds`, `unownedNFTIds`, `ownedNFTOfferSides`) at the time the `useMemo` ran — not at the time the callback resolves. [1](#0-0) 

When two NFTs on the same offer side (e.g., both requested, both not owned by the user) resolve concurrently:

- Callback A resolves first: `setUnownedNFTIds([...[], 'NFT_A'])` → state becomes `['NFT_A']`
- Callback B resolves second: `setUnownedNFTIds([...[], 'NFT_B'])` — it captured the same stale `[]` — → state becomes `['NFT_B']`, **overwriting NFT_A**

Only the last callback to resolve survives. Additionally, `setResolved(true)` is called synchronously after the `forEach` (line 64), before any async operation completes, so `isResolving` flips to `false` prematurely. [2](#0-1) 

The corrupted `unownedNFTIds` is consumed directly in `OfferShareDexieDialog`, `OfferShareSpacescanDialog`, and `OfferShareOfferpoolDialog`: [3](#0-2) [4](#0-3) [5](#0-4) 

`notificationDestination` is then passed to `NotificationSendDialog`, which resolves the NFT holder's p2 address and sends an on-chain XCH notification to it: [6](#0-5) 

### Impact Explanation

When the race condition fires, the wrong NFT ID is in `unownedNFTIds`. The user sees "Notify Current Owner" pointing at the wrong NFT. Clicking it triggers `sendNotification` — an on-chain XCH spend — to the wrong recipient's address. This is a direct, user-confirmed asset transfer to the wrong destination caused by corrupted notification state.

This matches the allowed High impact: *"Corruption … of … notification … state that causes a user to … send … [to the] wrong … destination."*

### Likelihood Explanation

The race condition is triggered whenever an offer contains two or more NFTs on the same side (both requested, or both offered). An attacker can craft such an offer (e.g., requesting two NFTs from the user) and send it via the incoming-offer notification channel. When the user opens the share dialog for their counter-offer or their own multi-NFT offer, the race fires deterministically — `forEach(async)` always dispatches all callbacks concurrently with the same stale closure. No special timing control is needed; the last IPC response to return wins.

### Recommendation

Replace the `forEach(async ...)` pattern with `Promise.all` and collect results before calling any `setState`. Use functional state updater form (`setUnownedNFTIds(prev => [...prev, nftId])`) to avoid stale closures. Move `setResolved(true)` to after all ownership checks have settled.

### Proof of Concept

1. Attacker creates an offer requesting two NFTs (NFT_X and NFT_Y) from the user and delivers it via the notification channel.
2. User creates a counter-offer that also requests both NFTs, then opens the share dialog (Dexie/Spacescan/Offerpool).
3. `useResolveNFTOffer` fires. Both NFTs are not owned by the user; both callbacks call `setUnownedNFTIds([...[], nftId])` with the same stale `[]`.
4. The last callback to resolve (say NFT_Y) overwrites the first. `unownedNFTIds = ['NFT_Y']`; NFT_X is silently dropped.
5. `notificationDestination = 'NFT_Y'`. The "Notify Current Owner" button resolves to NFT_Y's holder address.
6. User clicks the button. `sendNotification` sends 0.0001 XCH on-chain to NFT_Y's holder — not NFT_X's holder, which was the intended recipient.
7. The XCH is spent to the wrong address; the intended NFT_X holder never receives the offer notification. [7](#0-6) [8](#0-7) [9](#0-8)

  useMemo(async () => {
    const sides = Object.keys(nftsBySide) as ('requested' | 'offered')[];

    sides.forEach(async (side) => {
      nftsBySide[side]!.forEach(async (nftId) => {
        if (ownedNFTIds.includes(nftId) || unownedNFTIds.includes(nftId)) {
          return;
        }

        const owned = await checkNFTOwnership(nftId);

        if (owned) {
          setOwnedNFTIds([...ownedNFTIds, nftId]);
          setOwnedNFTOfferSides([...ownedNFTOfferSides, side as any]);
        } else {
          setUnownedNFTIds([...unownedNFTIds, nftId]);
        }
      });
    });

    setResolved(true);
  }, [
    nftsBySide,

    setUnownedNFTIds,
