### Title
Signature Validation Bypass for Invoke Transactions via Overly Broad `skip_stateful_validations` Admission Check — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator is designed to skip `__validate__` entry-point execution for an invoke with `nonce=1` when a `deploy_account` is pending in the mempool (UX improvement). However, the proxy check it uses — `account_tx_in_pool_or_recent_block` — returns `true` for **any** transaction from the address, not exclusively a `deploy_account`. An unprivileged attacker who observes a `deploy_account` in the mempool can submit an invoke with `nonce=1` carrying an invalid/arbitrary signature for the victim's address, bypass the gateway's `__validate__` call entirely, and have the transaction admitted to the mempool. The victim's account then pays the fee for the attacker's reverted transaction during block execution.

---

### Finding Description

**Root cause — overly broad proxy check:**

`skip_stateful_validations` fires when `tx.nonce() == 1 && account_nonce == 0` and `account_tx_in_pool_or_recent_block` returns `true`: [1](#0-0) 

`account_tx_in_pool_or_recent_block` is implemented as: [2](#0-1) 

This returns `true` for **any** transaction from the address in the pool or in the committed-nonce history — not only for a `deploy_account`. The comment in the code acknowledges this:

> "It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."

That reasoning is incorrect for the admission decision: the existence of *other* transactions does not mean the *current* transaction's signature is valid.

**Consequence — `run_validate_entry_point` is skipped:**

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` is called with `validate = false`: [3](#0-2) 

The blockifier's `StatefulValidator::validate` is therefore invoked with `ExecutionFlags { validate: false, … }`, meaning the account's `__validate__` entry point (signature verification) is never executed at the gateway level.

**The nonce and mempool checks do not compensate:**

`validate_state_preconditions` only checks nonce range: [4](#0-3) 

`validate_by_mempool` only checks nonce ordering and fee escalation in the mempool: [5](#0-4) 

Neither verifies the transaction signature.

**Attack path (step-by-step):**

1. Legitimate user submits `deploy_account` for address X (nonce=0, valid signature). It enters the mempool pool.
2. Attacker observes the pending `deploy_account` and submits an `Invoke` for address X with `nonce=1` and an **invalid/arbitrary signature**.
3. Gateway stateless validation passes (signature size, resource bounds, etc.).
4. Gateway stateful validation:
   - `get_nonce_from_state` → `0` (account not yet deployed).
   - `validate_nonce`: `0 ≤ 1 ≤ max_allowed_nonce_gap` → passes.
   - `validate_by_mempool`: `nonce=1 ≥ account_nonce=0`, no duplicate → passes.
   - `skip_stateful_validations`: `nonce=1`, `account_nonce=0`, `account_tx_in_pool_or_recent_block(X)=true` (deploy_account is in pool) → **returns `true`**.
   - `run_validate_entry_point`: **skipped** (`validate=false`).
5. Attacker's invoke is admitted to the mempool without any signature check.
6. Batcher executes the block: `deploy_account` runs (nonce 0→1), then attacker's invoke runs with `validate=true` → `__validate__` fails (invalid signature) → transaction reverts.
7. Fee is charged from account X (victim pays for attacker's reverted transaction).

Additionally, the attacker's `nonce=1` invoke occupies the nonce slot, forcing the victim's own `nonce=1` invoke to either pay a higher fee (fee escalation) or be rejected as a duplicate nonce.

---

### Impact Explanation

This is a **High** impact finding matching: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

- An unauthorized transaction with an invalid signature is accepted through the gateway's account-validation path.
- The victim's account is charged fees for the attacker's reverted transaction.
- The attacker can front-run or displace the victim's own first invoke after deployment.
- No privileged access is required; any observer of the mempool can execute this attack.

---

### Likelihood Explanation

- Mempool contents are observable (P2P gossip, monitoring endpoint).
- The attack requires only submitting two transactions: one `deploy_account` (by the victim, already in flight) and one malicious invoke (by the attacker).
- No special cryptographic capability is needed — the attacker only needs to craft a transaction with an arbitrary signature.
- The window of opportunity is the time between the `deploy_account` entering the mempool and being committed to a block.

---

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` proxy with a check that specifically verifies a **`deploy_account` transaction** is pending for the address. Add a dedicated mempool API, e.g. `deploy_account_in_pool(address: ContractAddress) -> bool`, that inspects the transaction type in the pool rather than merely checking address presence. The current check:

```rust
// VULNERABLE: returns true for any tx type
mempool_client.account_tx_in_pool_or_recent_block(tx.sender_address()).await
```

should become something like:

```rust
// CORRECT: only true when a deploy_account is pending
mempool_client.deploy_account_in_pool(tx.sender_address()).await
```

        versioned_constants.disable_casm_hash_migration();

        block_info.block_number = block_info.block_number.unchecked_next();

            versioned_constants,
            BouncerConfig::max(),

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
