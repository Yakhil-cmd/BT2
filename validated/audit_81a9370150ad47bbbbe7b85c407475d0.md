### Title
Unchecked Vec length in `PolynomialCommitmentsMessageMany` causes panic-based DoS in `do_generation_many` — (`src/ecdsa/ot_based_ecdsa/triples/generation.rs`)

### Summary
A malicious participant can send a `PolynomialCommitmentsMessageMany` message with any of its six `Vec` fields shorter than `N` in round 2 of `do_generation_many`. Because the code indexes all fields with `their.field_v[i]` for `i in 0..N` without bounds-checking, an out-of-bounds panic occurs in the honest party's async task, permanently aborting triple generation for that party.

### Finding Description
`PolynomialCommitmentsMessageMany` is a serde-derived struct with six `Vec` fields. `rmp_serde` deserializes each Vec to whatever length is present in the wire message — no minimum-length constraint is applied. After deserialization, `do_generation_many` iterates `for i in 0..N` and accesses each field with direct Rust indexing (`their.randomizer_v[i]`, etc.). If any field has fewer than `N` elements, Rust panics. The protocol framework (`make_protocol` / `Comms`) does not use `catch_unwind`, so the panic propagates to the caller, crashing the honest party's protocol instance. The malicious participant can repeat this on every retry, making the denial permanent for any session that includes them.

### Impact Explanation
Triple generation is permanently blocked for all honest parties in any session containing the malicious participant. Since triples are the prerequisite for every OT-based ECDSA presignature and signature, this is a complete denial of the signing pipeline. Impact: **High — Permanent denial of signing for honest parties under valid protocol inputs.**

### Likelihood Explanation
Any participant in the protocol can craft and send a malformed `PolynomialCommitmentsMessageMany` message. No special privilege, leaked key, or cryptographic break is required. The attacker only needs to be a registered participant and to serialize a struct with one Vec field one element short.

### Recommendation
After deserializing `PolynomialCommitmentsMessageMany`, validate that every Vec field has exactly `N` elements before indexing. Return a `ProtocolError::AssertionFailed` (not a panic) if any field length mismatches. Use `.get(i).ok_or(...)` instead of `[i]` for all six fields, or add an upfront length check:

```rust
if their.randomizer_v.len() != N || their.big_e_v.len() != N /* ... */ {
    return Err(ProtocolError::AssertionFailed(
        format!("message from {from:?} has wrong vector lengths")
    ));
}
```

### Proof of Concept
In a 3-party session with `N=2`, have the malicious participant serialize a `PolynomialCommitmentsMessageMany` where `randomizer_v` has length 1 instead of 2 (all other fields correct). Deliver this to an honest party at waitpoint `wait2`. The honest party's loop reaches `i=1`, panics at `their.randomizer_v[1]`, and the protocol instance crashes. Repeat on every retry to permanently deny triple generation.
