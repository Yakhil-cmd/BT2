Audit Report

## Title
Unbounded `ingress_expiries` Vec Allocation via Attacker-Controlled `ingress_start`/`ingress_end` — (`rs/rosetta-api/icp/src/request_handler/construction_payloads.rs`)

## Summary

The ICP Rosetta `construction_payloads` handler accepts caller-supplied `ingress_start` and `ingress_end` nanosecond timestamps with no server-side range validation. These values directly control the iteration count of an unbounded `while` loop that pushes `u64` entries into a `Vec`. An unauthenticated attacker can supply `ingress_start=0` and `ingress_end=u64::MAX` to force ~153 billion loop iterations, exhausting process memory and crashing the Rosetta node.

## Finding Description

`ConstructionPayloadsRequestMetadata` declares both fields as raw `Option<u64>` with no constraints: [1](#0-0) 

In `construction_payloads`, the interval is computed as:

```
MAX_INGRESS_TTL(300s) - PERMITTED_DRIFT(60s) - 120s = 120s = 120_000_000_000 ns
``` [2](#0-1) [3](#0-2) 

The caller-supplied values are consumed directly without any bounds check, and the loop runs unconstrained: [4](#0-3) 

With `ingress_start=0` and `ingress_end=u64::MAX (18_446_744_073_709_551_615)`:

```
iterations ≈ 18_446_744_073_709_551_615 / 120_000_000_000 ≈ 153,722,867,281
memory     ≈ 153 billion × 8 bytes ≈ 1.23 TB
```

The process OOMs before the loop completes. Additionally, `ic_types::time::Time::add_assign` internally computes `Duration::from_nanos(self.0) + other` (a `std::time::Duration` which can exceed `u64::MAX` nanoseconds), then casts back via `t.as_nanos() as u64`, which truncates silently in release mode: [5](#0-4) [6](#0-5) 

When `now` nears `u64::MAX`, the `as u64` truncation wraps `now` back to a small value, making `now < ingress_end` permanently true — the loop becomes **infinite** after the wrap-around point. In practice, OOM occurs first, but the infinite-loop path is also present.

The ICRC1 Rosetta implementation has a partial guard (`ingress_end >= now + ingress_interval` and `ingress_start < ingress_end`), but it does **not** bound the range between `ingress_start` and `ingress_end` and is also susceptible to the same attack with `ingress_start=0, ingress_end=u64::MAX`. The ICP Rosetta implementation lacks even those partial checks. [7](#0-6) 

## Impact Explanation

The Rosetta node is a standalone HTTP service used for ICP ledger operations (transfers, neuron management, etc.). A single malformed POST to `/construction/payloads` causes the process to exhaust all available memory and be OOM-killed. This constitutes an **application/platform-level DoS** matching the High ($2,000–$10,000) bounty impact tier: the node becomes permanently unavailable (the attack is repeatable immediately after restart), blocking all ICP ledger operations routed through Rosetta.

## Likelihood Explanation

The endpoint is publicly reachable with no authentication. The attack payload is a standard small JSON POST body requiring no cryptographic material, tokens, or prior state. Any party aware of the Rosetta API spec can trigger it with a single HTTP request. The attack is trivially repeatable.

## Recommendation

Add an explicit upper bound on `ingress_end - ingress_start` before the loop:

```rust
const MAX_INGRESS_WINDOW: u64 = 24 * 60 * 60 * 1_000_000_000; // 24h in ns
let ingress_start_nanos = ingress_start.as_nanos_since_unix_epoch();
let ingress_end_nanos = ingress_end.as_nanos_since_unix_epoch();
if ingress_end_nanos.saturating_sub(ingress_start_nanos) > MAX_INGRESS_WINDOW {
    return Err(ApiError::invalid_request(
        "ingress_end - ingress_start exceeds maximum allowed window",
    ));
}
```

Alternatively, cap `ingress_expiries` with a hard `Vec` size limit (e.g., 720 entries for a 24-hour window at 120s intervals) and return an error if the computed count would exceed it. Also add a check that `ingress_start < ingress_end` to mirror the ICRC1 guard.

```rust

    ingress_start: Some(0),
    ingress_end: Some(u64::MAX),

    created_at_time: None,

// iterations ≈ 153_722_867_281 → OOM before completion

**HTTP PoC:**
```json
POST /construction/payloads
{
  "network_identifier": { "blockchain": "Internet Computer", "network": "<ledger_canister_id>" },
  "operations": [ { "operation_identifier": {"index": 0}, "type": "TRANSACTION", ... } ],
  "public_keys": [ { "hex_bytes": "<valid_pubkey>", "curve_type": "edwards25519" } ],
  "metadata": {
    "ingress_start": 0,
    "ingress_end": 18446744073709551615
  }
}
```

The Rosetta process will exhaust memory and be killed by the OS before returning a response. The attack is repeatable immediately after restart.
