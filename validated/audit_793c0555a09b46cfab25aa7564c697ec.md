### Title
`WalletDeleteDialog` Key Deletion Does Not Revoke WalletConnect Pairs ‚Äî (`File: packages/core/src/screens/SelectKey/WalletDeleteDialog.tsx`)

---

### Summary

`WalletDeleteDialog.handleSubmit` deletes a wallet key and removes UI preferences but never revokes the WalletConnect pairs that were granted to that fingerprint. The pairs persist in the `pairStore` indefinitely. If the user later re-imports the same mnemonic (same fingerprint), every previously-paired dApp regains full command authority ‚Äî including any commands it was granted `bypass` (no-confirmation) permission for ‚Äî without any new user approval.

---

### Finding Description

`handleSubmit` in `WalletDeleteDialog.tsx` performs exactly two cleanup steps on key deletion:

1. `deleteKey({ fingerprint })` ‚Äî removes the key from the daemon keyring.
2. `removeFingerprintPrefs()` ‚Äî removes `fingerprintSettings` and `sortedWallets` UI prefs. [1](#0-0) 

`removeFingerprintPrefs` only touches two preference keys: [2](#0-1) 

Neither step touches the WalletConnect `pairStore`. Every `PairRecord` that was registered for the deleted fingerprint ‚Äî including its `commands` list and its `bypass` list ‚Äî remains on disk, fully intact.

The `pairStore` is a separate, file-backed store: [3](#0-2) 

The `disconnectPair` helper that *does* call `window.permissionsAPI.revokePair(topic)` is only invoked from the WalletConnect UI disconnect flow and from error-cleanup inside `handlePair`. It is never called from the key-deletion path: [4](#0-3) 

The `dispatchPairRequest` gate in the main process enforces that the currently logged-in fingerprint matches `pair.fingerprint`: [5](#0-4) 

This means the stale pair is dormant while the key is absent. However, the moment the user re-imports the same mnemonic (restoring from backup, migrating devices, or undoing an accidental deletion), the fingerprint is identical, the pair record is found, and `dispatchPairRequest` passes all checks ‚Äî including the `bypass` fast-path that skips the user-confirmation dialog entirely: [6](#0-5) 

The renderer-side `useWalletConnectCommand` applies the same fingerprint check and also has no awareness that the pair should have been revoked: [7](#0-6) 

---

### Impact Explanation

**Critical** ‚Äî if the dApp was granted `bypass` permission for any signing or spending command (e.g. `chia_sendTransaction`, `chia_takeOffer`), it can execute those commands silently ‚Äî no confirmation dialog ‚Äî the instant the user re-logs in with the same key. The user's explicit intent in deleting the key was to revoke all third-party access; that intent is not honored.

Even without `bypass`, the stale pair allows the dApp to trigger confirmation dialogs for every command it was originally granted, which constitutes a bypass of the WalletConnect approval lifecycle (the user never re-approved the dApp after deletion).

---

### Likelihood Explanation

**Low-to-Medium.** Re-importing the same mnemonic is a routine operation: backup restoration, device migration, or accidental-deletion recovery. A malicious dApp that was paired before the deletion needs only to wait. The user has no indication that the old pair is still active, because the WalletConnect UI shows pairs based on the live `pairStore`, which still contains the record.

---

### Recommendation

In `WalletDeleteDialog.handleSubmit`, after `deleteKey` succeeds, enumerate all pairs whose `fingerprint` matches the deleted fingerprint and call `window.permissionsAPI.revokePair(topic)` for each. The same cleanup should be applied in `SelectKey.handleDeleteAllKeys`.

```ts
// pseudocode addition to handleSubmit
const allPairs = await window.permissionsAPI.getPairs();
const stale = allPairs.filter((p) => p.fingerprint === fingerprint);
await Promise.all(stale.map((p) => window.permissionsAPI.revokePair(p.topic)));
```

---

### Proof of Concept

1. User pairs wallet (fingerprint `F`) with a dApp and grants `bypass` for `chia_sendTransaction`.
2. User opens `WalletDeleteDialog`, types fingerprint `F`, clicks **Delete**. Key is removed from daemon; `pairStore` entry for `F` is untouched.
3. User later re-imports the same 24-word mnemonic (same fingerprint `F`) ‚Äî e.g., after realising the deletion was a mistake.
4. User logs in with fingerprint `F`.
5. The dApp sends a `chia_sendTransaction` WalletConnect request. `dispatchPairRequest` finds the stale pair, fingerprint matches, command is in `bypass` ‚Üí transaction is signed and broadcast **with no confirmation dialog shown to the user**. [1](#0-0) [6](#0-5)

async function disconnectPair(client: Client, topic: string) {
  try {
    // revoke pair from main process
    await window.permissionsAPI.revokePair(topic);

    // get all sessions for the pairing
    const sessions = client.session.getAll();
    const relatedSessions = sessions.filter((s) => s.pairingTopic === topic);

    // disconnect all related sessions
    await Promise.all(
      relatedSessions.map((session) =>
        client.disconnect({
          topic: session.topic,
          reason: getSdkError('USER_DISCONNECTED'),
        }),
      ),
    );

    // disconnect the pairing
    await client.core.pairing.disconnect({ topic });
  } catch (e) {
    processError(e as Error);
  }
}
