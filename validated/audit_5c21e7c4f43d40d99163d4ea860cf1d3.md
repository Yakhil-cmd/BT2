### Title
Pool Admin Can Set Uncapped Per-Bin Additional Fees, Bypassing Factory-Enforced Fee Caps — (File: metric-core/contracts/MetricOmmPoolFactory.sol)

---

### Summary

`setPoolBinAdditionalFees` in `MetricOmmPoolFactory.sol` forwards `addFeeBuyE6` and `addFeeSellE6` directly to the pool with **no cap validation**, while the parallel `setPoolAdminFees` path enforces `maxAdminSpreadFeeE6` / `maxAdminNotionalFeeE8` caps. A pool admin can therefore set per-bin additional fees up to `uint16` max (65 535 in E6 = 6.5535 %) on any bin, silently exceeding the factory-enforced fee boundary and extracting more from traders than the protocol's cap system permits.

---

### Finding Description

The factory maintains explicit fee caps and enforces them in `setPoolAdminFees`:

```solidity
// MetricOmmPoolFactory.sol lines 414-415
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (newAdminNotionalFeeE8 > maxAdminNotionalFeeE8) revert AdminFeeTooHigh();
``` [1](#0-0) 

The sibling function `setPoolBinAdditionalFees` performs **no equivalent check**:

```solidity
// MetricOmmPoolFactory.sol lines 450-457
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
``` [2](#0-1) 

The pool-level `setBinAdditionalFees` also applies no cap:

```solidity
// MetricOmmPool.sol lines 464-474
function setBinAdditionalFees(int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external onlyFactory nonReentrant(PoolActions.SET_BIN_ADDITIONAL_FEES)
{
    if (bin < LOWEST_BIN || bin > HIGHEST_BIN) revert InvalidBinIndex(bin);
    BinState storage s = _binStates[bin];
    s.addFeeBuyE6 = addFeeBuyE6;
    s.addFeeSellE6 = addFeeSellE6;
    emit BinAdditionalFeesUpdated(bin, addFeeBuyE6, addFeeSellE6);
}
``` [3](#0-2) 

The only validation is a bin-index range check. The fee values themselves are written verbatim into `BinState.addFeeBuyE6` / `addFeeSellE6`.

The factory hard-caps are:

| Cap constant | Value | Meaning |
|---|---|---|
| `HARD_MAX_SPREAD_FEE_E6` | 200 000 | 20 % total spread |
| `HARD_MAX_NOTIONAL_FEE_E8` | 1 000 000 | 1 % notional | [4](#0-3) 

`uint16` max is **65 535**, which in E6 units equals **6.5535 %**. Because bin additional fees are additive on top of the base spread fee, a pool admin can push the effective per-bin spread to `20 % + 6.5535 % = 26.5535 %`, silently exceeding the hard cap the factory is designed to enforce.

---

### Impact Explanation

Every swap routed through a bin whose `addFeeBuyE6` or `addFeeSellE6` has been set to `uint16` max pays up to **6.5535 % more** than the factory-capped maximum. This is a direct, per-swap extraction of trader principal beyond the protocol's stated fee ceiling. Because the pool admin controls which bins are active and can target the bin the current price sits in, the excess fee applies to the majority of swap volume, not just edge bins.

**Severity: Medium** — bounded by `uint16` max but meaningful (6.5535 % excess per swap), requires a semi-trusted pool admin acting outside their intended cap.

---

### Likelihood Explanation

The pool admin is explicitly classified as semi-trusted and is expected to operate only within factory-enforced caps. The asymmetry between `setPoolAdminFees` (capped) and `setPoolBinAdditionalFees` (uncapped) is an oversight rather than a design choice. Any pool admin — whether malicious or compromised — can exploit this without any timelock or additional privilege escalation.

**Likelihood: Low-Medium** — requires a malicious or compromised pool admin, but no further preconditions.

---

### Recommendation

Add cap validation in `setPoolBinAdditionalFees` before forwarding to the pool, mirroring the pattern in `setPoolAdminFees`:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (addFeeBuyE6 > maxAdminBinAdditionalFeeE6) revert AdminFeeTooHigh();
    if (addFeeSellE6 > maxAdminBinAdditionalFeeE6) revert AdminFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

Alternatively, enforce that `base spread fee + max bin additional fee ≤ HARD_MAX_SPREAD_FEE_E6` at the factory level.

---

### Proof of Concept

1. Factory is deployed with `maxAdminSpreadFeeE6 = 200_000` (20 %).
2. Pool admin calls:
   ```solidity
   factory.setPoolBinAdditionalFees(pool, 0, 65535, 65535);
   ```
   No revert occurs — `addFeeBuyE6 = 65535` and `addFeeSellE6 = 65535` are written directly to `_binStates[0]`.
3. A trader swaps through bin 0. The effective spread applied is `spreadFeeE6 (≤ 200 000) + 65 535`, i.e., up to **26.5535 %** — 6.5535 % above the factory hard cap.
4. The excess fee accrues to the pool and is later collected by the admin via `collectPoolFees`, extracting value from traders beyond the protocol's stated maximum. [2](#0-1) [3](#0-2)
