### Title
WalletConnect `take_offer` Confirmation Dialog Displays Understated XCH Cost When NFT Royalty Lookup Fails Due to Attacker-Controlled Offer `infos` Field - (File: `packages/gui/src/electron/commands/parseCommandDisplay.ts`)

---

### Summary

When a WalletConnect dapp sends `chia_wallet.take_offer`, the GUI computes `amountWithRoyalties` (the total XCH/CAT cost shown in the approval dialog) using the royalty percentage from the offer's attacker-controlled `infos` field as a fallback when the on-chain `nftGetInfo` lookup fails. An attacker can craft an offer embedding `royalty_percentage: 0` in the `infos` field and time the attack before the user's wallet has indexed the newly minted NFT, causing the confirmation dialog to display a lower spend amount than what the wallet daemon actually enforces on-chain.

---

### Finding Description

In `parseCommandDisplay.ts`, the `take_offer` path calls `offerSummaryRoyaltyPercentages()` to extract royalty percentages from the offer summary's `infos` field:

```typescript
// lines 184-206
function offerSummaryRoyaltyPercentages(offerSummary: OfferSummaryForDisplay): AssetRoyaltyPercentages {
  const { infos } = offerSummary;
  // ...
  for (const assetId of Object.keys(offerSummary.offered)) {
    royaltyPercentages.receiving[parsedAssetId] = royaltyPercentageForDriverInfo(infos[assetId]);
  }
}
```

The `infos` field is parsed directly from the attacker-supplied offer string by the wallet daemon's `getOfferSummary`. The attacker fully controls its content. [1](#0-0) 

These attacker-supplied royalty values are passed as the initial `royaltyPercentage` to `parseWalletDeltaItem`. The code then attempts to override them with on-chain data via `nftGetInfo`:

```typescript
// lines 302-328
if (assetKind === 'nft') {
  const result: DisplayWalletDeltaItem = {
    kind: 'nft',
    nftId,
    royaltyPercentage,   // ← attacker-controlled initial value
  };
  try {
    const nftInfo = await nftGetInfo(key);
    if (nftInfo && nftInfo.success && nftInfo.nft_info && nftInfo.nft_info.data_uris) {
      if ('royalty_percentage' in nftInfo.nft_info) {
        result.royaltyPercentage = parseRoyaltyPercentage(nftInfo.nft_info.royalty_percentage);
      }
    }
  } catch {
    // NFT type has already been resolved from offer data; metadata is best effort.
  }
  return result;
}
``` [2](#0-1) 

When `nftGetInfo` returns `success: false` or throws, the attacker-controlled value is silently retained. The comment "metadata is best effort" documents this intentional fallback, but it creates a security gap: the royalty used to compute `amountWithRoyalties` in the confirmation dialog is then derived from the attacker's data.

The `amountWithRoyalties` field is computed in `withRoyaltyTotals` → `formatAmountWithRoyalties`, which reads `royaltyPercentage` from the NFT line items:

```typescript
// lines 377-394
function withRoyaltyTotals(items, amounts, oppositeSideLines) {
  const royaltyPercentages = royaltyPercentagesForSide(oppositeSideLines);
  return items.map(({ key, line }) => {
    const amount = amounts[key];
    const amountWithRoyalties = amount ? formatAmountWithRoyalties(line, amount, royaltyPercentages) : undefined;
    return amountWithRoyalties ? { ...line, amountWithRoyalties } : line;
  });
}
``` [3](#0-2) 

This `amountWithRoyalties` value is passed directly to the `Confirm` dialog shown to the user in `main.tsx`:

### Impact Explanation

The wallet daemon enforces the correct on-chain royalty when executing the offer (the royalty is baked into the NFT's puzzle hash). If the confirmation dialog shows `amountWithRoyalties: 1 XCH` (0% royalty from attacker's `infos`) but the actual on-chain royalty is 50%, the wallet daemon spends 1.5 XCH. The user approved a transaction believing they were spending 1 XCH. This is a direct, concrete financial loss: the user is caused to approve the wrong amount.

This matches the allowed High impact: *"Corruption, spoofing, or unsafe trust of RPC, event, offer, NFT metadata, DataLayer, notification, or WalletConnect state that causes a user to approve, import, sign, send, revoke, burn, join, or display the wrong asset, identity, amount, destination, or status."*

---

### Likelihood Explanation

The attack requires `nftGetInfo` to fail for the offered NFT. This is achievable by:

1. Attacker mints a new NFT with a high royalty percentage (e.g., 5000 = 50%).
2. Immediately creates an offer for this NFT with `royalty_percentage: 0` (or omitted) in the offer's `infos` field.
3. Sends the offer to the victim via WalletConnect (`chia_wallet.take_offer`) before the victim's wallet has indexed the newly minted NFT.
4. The victim's wallet daemon returns `success: false` from `nftGetInfo` because the NFT is not yet in its local state.
5. The confirmation dialog shows 0% royalty / base amount only.
6. Victim approves; wallet daemon enforces the 50% royalty on-chain.

The timing window is realistic: NFT minting and offer creation can happen in the same block, and wallet sync lag is common. The attacker controls the timing entirely.

---

### Recommendation

Replace the "best effort" fallback with a hard failure when on-chain royalty data cannot be confirmed for an NFT in a `take_offer` command. Specifically, in `parseWalletDeltaItem`, if `nftGetInfo` fails or does not return a `royalty_percentage` for an NFT that is on the receiving side of a `take_offer`, the confirmation dialog should either:

- Refuse to display `amountWithRoyalties` and instead show a warning that royalty information could not be verified, blocking approval until the wallet is synced; or
- Treat the royalty as unknown/unverifiable and display a conservative worst-case estimate or an explicit "royalty unknown" warning.

The comment `// NFT type has already been resolved from offer data; metadata is best effort` should not apply to royalty percentage, which has direct financial impact on the user's spend amount. [6](#0-5) 

---

### Proof of Concept

1. Attacker mints NFT `nft1abc...` with `royalty_percentage = 5000` (50%) on-chain.
2. Attacker creates offer: NFT offered, 1 XCH requested. Offer `infos` field contains `{ type: 'singleton' }` (no royalty info).
3. Attacker sends `chia_wallet.take_offer` via WalletConnect to victim immediately after minting (before victim's wallet syncs the NFT).
4. `parseCommandDisplay` calls `offerSummaryRoyaltyPercentages` → royalty = `undefined` (no royalty in `infos`).
5. `parseWalletDeltaItem` calls `nftGetInfo(nftLauncherId)` → returns `{ success: false }` (NFT not yet indexed).
6. `result.royaltyPercentage` remains `undefined`. `royaltyPercentagesForSide` returns `[]`. `amountWithRoyalties` is not set.
7. Confirmation dialog shows: **Spending: 1 XCH** (no royalty line).
8. Victim approves.
9. Wallet daemon builds spend bundle enforcing the 50% on-chain royalty → **1.5 XCH deducted** from victim's wallet.

function offerSummaryRoyaltyPercentages(offerSummary: OfferSummaryForDisplay): AssetRoyaltyPercentages {
  const royaltyPercentages: AssetRoyaltyPercentages = {
    spending: {},

  };

  const { infos } = offerSummary;
  if (!isPlainObject(infos)) {
    return royaltyPercentages;
  }

  for (const assetId of Object.keys(offerSummary.requested)) {
    const parsedAssetId = assetId === 'xch' ? '1' : assetId;
    royaltyPercentages.spending[parsedAssetId] = royaltyPercentageForDriverInfo(infos[assetId]);

    const parsedAssetId = assetId === 'xch' ? '1' : assetId;
    royaltyPercentages.receiving[parsedAssetId] = royaltyPercentageForDriverInfo(infos[assetId]);
  }

  return royaltyPercentages;
}

function withRoyaltyTotals(
  items: DisplayWalletDeltaItemWithKey[],
  amounts: Record<string, bigint>,
  oppositeSideLines: DisplayWalletDeltaItem[],

  const royaltyPercentages = royaltyPercentagesForSide(oppositeSideLines);

    if (line.kind === 'nft') {

    const amountWithRoyalties = amount ? formatAmountWithRoyalties(line, amount, royaltyPercentages) : undefined;

    return amountWithRoyalties ? { ...line, amountWithRoyalties } : line;
  });
}
