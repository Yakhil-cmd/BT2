Audit Report

## Title
Stale ICP/XDR Rate Used Unconditionally for Cycle Minting Without Freshness Check - (File: rs/nns/cmc/src/main.rs)

## Summary
The `tokens_to_cycles()` function in the Cycles Minting Canister reads only `xdr_permyriad_per_icp` from the cached `icp_xdr_conversion_rate` state and never inspects `timestamp_seconds`. When the Exchange Rate Canister (XRC) fails, the heartbeat logs the error but leaves the cached rate unchanged with no maximum-age enforcement, allowing all three minting entry points to continue converting ICP to cycles at an arbitrarily stale rate. An unprivileged caller can exploit a stale high rate during an XRC outage coinciding with an ICP price drop to receive excess cycles per ICP burned.

## Finding Description
`tokens_to_cycles()` at `rs/nns/cmc/src/main.rs:1900–1922` extracts only `rate.xdr_permyriad_per_icp` and discards `rate.timestamp_seconds` entirely:

```rust
let xdr_permyriad_per_icp = state
    .icp_xdr_conversion_rate
    .as_ref()
    .map(|rate| rate.xdr_permyriad_per_icp);
```

This function is called unconditionally by all three minting paths: `process_create_canister` (L1932), `process_mint_cycles` (L1965), and `process_top_up` (L1991).

In `rs/nns/cmc/src/exchange_rate_canister.rs:259–275`, when `xrc_client.get_icp_to_xdr_exchange_rate()` returns an error, the function returns `Err(UpdateExchangeRateError::FailedToRetrieveRate(...))` without touching `state.icp_xdr_conversion_rate`. The heartbeat at `main.rs:2397–2428` logs the error and exits, leaving the cached rate intact.

`do_set_icp_xdr_conversion_rate` at `main.rs:1009–1040` does enforce that a new rate must have a strictly greater `timestamp_seconds` than the current one (L1024–1030), but this guard only applies when *setting* a new rate — it provides no protection against *reading* a stale rate at minting time.

On failure, `schedule_next_attempt` in `exchange_rate_canister.rs:149–161` reschedules the next XRC call at the next minute (not the 5-minute interval), which shortens the typical window. However, if the XRC remains unavailable for an extended period, the stale rate is used indefinitely with no upper bound.

The `IcpXdrConversionRate` struct at `rs/nns/cmc/src/lib.rs:488–497` carries `timestamp_seconds` explicitly for this purpose, but it is never consulted in the minting path.

## Impact Explanation
This matches the **High** impact category: "Significant XRC, NNS, or infrastructure security impact with concrete user or protocol harm." An unprivileged caller who sends ICP to the CMC during an XRC outage while the ICP market price has dropped receives more cycles than the burned ICP is worth at current prices. Cycles are the IC's computational resource currency; excess minting at an inflated rate constitutes illegal minting that underprices network computation. The `base_cycles_limit` caps the per-hour damage globally but does not prevent repeated exploitation across multiple hours during a prolonged XRC outage, and does not prevent exploitation within each hourly window.

## Likelihood Explanation
The XRC is an external canister dependency whose failure is explicitly anticipated and handled in the CMC codebase. Transient inter-canister call failures, XRC downtime, or insufficient data sources are all realistic failure modes. A significant ICP price drop (e.g., 20–30%) combined with even a brief XRC outage creates the exploitable window. The attack requires no special privileges — any principal can call `notify_top_up`, `notify_create_canister`, or `notify_mint_cycles`. The window is bounded in the common case (retry at next minute) but unbounded if the XRC remains unavailable, and the NNS governance fallback (`set_icp_xdr_conversion_rate`) requires a governance proposal cycle to correct.

## Recommendation
Add a maximum-age check inside `tokens_to_cycles()`. Define a `MAX_RATE_AGE_SECONDS` constant (e.g., a small multiple of `REFRESH_RATE_INTERVAL_SECONDS`) and reject minting if the stored rate's `timestamp_seconds` is older than this threshold:

```rust
fn tokens_to_cycles(amount: Tokens) -> Result<Cycles, NotifyError> {
    with_state(|state| {
        match state.icp_xdr_conversion_rate.as_ref() {
            Some(rate) => {
                let now = ic_cdk::api::time() / 1_000_000_000;
                let age = now.saturating_sub(rate.timestamp_seconds);
                if age > MAX_RATE_AGE_SECONDS {
                    return Err(NotifyError::Other {
                        error_code: NotifyErrorCode::Internal as u64,
                        error_message: format!(
                            "ICP/XDR conversion rate is stale ({} seconds old)", age
                        ),
                    });
                }
                Ok(TokensToCycles {
                    xdr_permyriad_per_icp: rate.xdr_permyriad_per_icp,
                    cycles_per_xdr: state.cycles_per_xdr,
                }.to_cycles(amount))
            }
            None => Err(/* existing None error */),
        }
    })
}
```

This ensures minting is rejected rather than silently proceeding with a stale rate, and callers receive a clear error they can retry once the XRC recovers.

## Proof of Concept
1. Configure a test CMC with an XRC client that returns an error on all calls.
2. Set an initial `icp_xdr_conversion_rate` with a high `xdr_permyriad_per_icp` (e.g., representing ICP at $10) and a `timestamp_seconds` of `T`.
3. Advance the canister clock past `T + REFRESH_RATE_INTERVAL_SECONDS` (5 minutes). Confirm the heartbeat fires, `update_exchange_rate` returns `FailedToRetrieveRate`, and `state.icp_xdr_conversion_rate` is unchanged.
4. As an unprivileged principal, send ICP to the CMC subaccount and call `notify_top_up`.
5. Observe that `process_top_up` → `tokens_to_cycles` returns cycles computed from the stale high rate, not from the current (lower) market rate.
6. Verify the returned `Cycles` value exceeds what the current market rate would justify, confirming excess minting without any staleness rejection.

A deterministic PocketIC or state-machine integration test can implement this scenario by injecting a mock XRC client that always errors and asserting the cycle count returned by `notify_top_up` against the expected value at the stale vs. current rate.
