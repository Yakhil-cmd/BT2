### Title
`DelegateAction` (NEP-366 Meta Transaction) Signature Lacks Chain Binding — Cross-Network Replay Possible - (`core/primitives/src/action/delegate.rs`, `core/primitives/src/signable_message.rs`)

---

### Summary

The `DelegateAction` / `DelegateActionV2` signed payload does not include any chain-specific identifier (genesis hash, chain ID, or network name). The NEP-461 `SignableMessage` wrapper adds only a fixed 4-byte discriminant (`(1<<30) + 366` for NEP-366, `(1<<30) + 611` for NEP-611). Because the outer `SignedTransaction` binds to a chain via its `block_hash` field but the inner `SignedDelegateAction` does not, an attacker who observes a `SignedDelegateAction` on one NEAR network (e.g., testnet) can extract it and re-wrap it in a fresh `SignedTransaction` on a different NEAR network (e.g., mainnet), causing Alice's inner actions to execute on a network she never authorized.

---

### Finding Description

**Signed payload construction — no chain binding**

`DelegateAction.get_nep461_hash()` serializes `discriminant || borsh(DelegateAction)`:

```rust
// core/primitives/src/action/delegate.rs
pub fn get_nep461_hash(&self) -> CryptoHash {
    let signable = SignableMessage::new(&self, SignableMessageType::DelegateAction);
    let bytes = borsh::to_vec(&signable).expect("Failed to deserialize");
    hash(&bytes)
}
```

`SignableMessage` contains only a fixed discriminant and the message body:

```rust
// core/primitives/src/signable_message.rs
pub struct SignableMessage<'a, T> {
    pub discriminant: MessageDiscriminant,  // constant: (1<<30)+366
    pub msg: &'a T,
}
```

`DelegateAction` itself contains `sender_id`, `receiver_id`, `actions`, `nonce`, `max_block_height`, `public_key` — **no genesis hash, no chain ID, no network identifier**:

```rust
// core/primitives/src/action/delegate.rs
pub struct DelegateAction {
    pub sender_id: AccountId,
    pub receiver_id: AccountId,
    pub actions: Vec<NonDelegateAction>,
    pub nonce: Nonce,
    pub max_block_height: BlockHeight,
    pub public_key: PublicKey,
}
```

**Contrast with the outer transaction**

`TransactionV1` includes `block_hash`, which is chain-specific and enforced by `check_transaction_validity_period`. This prevents outer-transaction replay across networks. But the `SignedDelegateAction` is extracted from the outer transaction and re-verified independently in `apply_delegate_action`:

```rust
// runtime/runtime/src/actions.rs
pub(crate) fn apply_delegate_action(...) -> Result<(), RuntimeError> {
    if !signed_delegate_action.verify() {          // only checks sig vs public_key
        result.result = Err(ActionErrorKind::DelegateActionInvalidSignature.into());
        return Ok(());
    }
    if apply_state.block_height > delegate_action.max_block_height() {  // height, not hash
        result.result = Err(ActionErrorKind::DelegateActionExpired.into());
        return Ok(());
    }
    ...
    validate_delegate_action_key(state_update, apply_state, delegate_action, result)?;
```

`max_block_height` is a bare integer. Block heights on NEAR mainnet and testnet are not synchronized, so a `max_block_height` that has already passed on mainnet may still be in the future on testnet, and vice versa.

**Attack path (unprivileged user)**

1. Alice signs a `DelegateAction` on testnet (e.g., `ft_transfer` of 1000 tokens to Bob, `nonce=5`, `max_block_height=10_000_000`). She sends the `SignedDelegateAction` off-chain to a testnet relayer.
2. The relayer submits it on testnet. The `SignedDelegateAction` is now visible on-chain in the testnet transaction body.
3. An attacker (any unprivileged observer of testnet) extracts the `SignedDelegateAction` bytes from the testnet transaction.
4. The attacker wraps the same `SignedDelegateAction` in a new `SignedTransaction` pointing to mainnet (using a recent mainnet `block_hash`), and submits it to mainnet.
5. On mainnet, `apply_delegate_action` verifies the signature (valid — same key, same payload), checks `max_block_height` (still in the future on mainnet), and checks the nonce against Alice's mainnet access key nonce. If Alice's mainnet nonce is < 5, the action executes — transferring 1000 mainnet tokens Alice never intended to move.

The nonce guard prevents replay on the **same** chain after the nonce advances, but it does not prevent replay on a **different** chain where the nonce state is independent.

---

### Impact Explanation

A successfully replayed `DelegateAction` executes the full inner action list (`actions`) with `sender_id` as `predecessor_id`. This can include:
- `Transfer` actions draining Alice's mainnet token balance
- `AddKey` / `DeleteKey` actions modifying Alice's mainnet access keys
- `FunctionCall` actions invoking contracts on Alice's behalf (e.g., `ft_transfer`, `stake`)

The corrupted state is Alice's mainnet account balance, access key set, or any contract state that depends on `predecessor_id == alice.near`. The relayer on the target chain pays gas, so the attacker's only cost is a single transaction fee.

---

### Likelihood Explanation

Conditions required:
1. Alice has the same account name on both networks (common — NEAR account names are human-readable and reused across mainnet/testnet).
2. Alice's access key nonce on the target network is lower than the `nonce` in the `DelegateAction` (likely when testnet is used for testing before mainnet deployment, or when mainnet account is newer).
3. The target network's block height is below `max_block_height` (easily satisfied since users typically set generous deadlines).
4. The attacker can observe the `SignedDelegateAction` — either from the source chain's transaction body (public) or by intercepting the off-chain relay channel.

All four conditions are routinely satisfied in the testnet→mainnet direction for any user who tests meta transactions on testnet before using them on mainnet.

---

### Recommendation

Include a chain-specific binding in the `DelegateAction` signed payload. The cleanest approach is to add a `chain_id` or `genesis_hash` field to `DelegateAction` / `DelegateActionV2` and incorporate it into the `get_nep461_hash()` computation. Alternatively, include the genesis block hash inside `SignableMessage` so all NEP-461 messages are automatically chain-bound. This mirrors the fix recommended in the external report (include `chainid` in the signed message).

---

### Proof of Concept

```
1. On testnet, Alice (alice.near, nonce=4) signs:
   DelegateAction {
     sender_id: "alice.near",
     receiver_id: "ft.near",
     actions: [FunctionCall { method: "ft_transfer", args: {receiver:"bob",amount:"1000"}, deposit:1 }],
     nonce: 5,
     max_block_height: 200_000_000,
     public_key: alice_ed25519_key,
   }
   → get_nep461_hash() = SHA256( LE32(1<<30 + 366) || borsh(DelegateAction) )
   → signature = alice_sk.sign(hash)

2. Relayer submits SignedTransaction{ ..., block_hash: testnet_block, actions:[Delegate(signed)] }
   on testnet. Transaction is included; SignedDelegateAction is visible in testnet block.

3. Attacker extracts SignedDelegateAction from testnet block.

4. Attacker submits SignedTransaction{ ..., block_hash: mainnet_recent_block,
   actions:[Delegate(same_signed_delegate_action)] } to mainnet.

5. apply_delegate_action on mainnet:
   - verify(): SHA256(LE32(1<<30+366) || borsh(same_DelegateAction)) → same hash → sig valid ✓
   - block_height(mainnet) < 200_000_000 ✓
   - sender_id("alice.near") == receipt.receiver("alice.near") ✓
   - validate_delegate_action_key: alice.near mainnet nonce=4 < 5 ✓
   → ft_transfer executes on mainnet, draining Alice's mainnet FT balance.
```
