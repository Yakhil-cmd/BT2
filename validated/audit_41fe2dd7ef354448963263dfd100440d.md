### Title
Gateway `skip_stateful_validations` Admits Unsigned Invoke Transactions for Any Undeployed Account — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator unconditionally skips the `__validate__` entry point (account signature verification) for any invoke transaction with `nonce == 1` sent to an address whose on-chain nonce is `0`, provided that address has *any* transaction in the mempool or a recent committed block. Because the check is on the *sender address* rather than on the *type* of the existing pool transaction, an unprivileged attacker who observes a victim's `deploy_account` in the mempool can immediately submit a competing invoke with an arbitrary/invalid signature for the victim's address, and the gateway will admit it without ever calling `__validate__`. When fee escalation is enabled, the attacker can also replace the victim's legitimate first invoke with the invalid one.

**The broken invariant**

The comment on `skip_stateful_validations` says:
> "It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."

This reasoning is correct for the *legitimate* UX case (Alice sends `deploy_account + invoke` together). However, the check `account_tx_in_pool_or_recent_block` (mempool, line 697-700) returns `true` for *any* address that has *any* transaction in the pool:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
```

This means a third party (the attacker) who does not own the account can also trigger the skip, because Alice's `deploy_account` already satisfies the pool-presence condition.

**Attack path**

1. Alice submits `deploy_account(nonce=0, valid_sig)` → gateway accepts, enters mempool pool.
2. Attacker observes Alice's address in the mempool (public information).
3. Attacker submits `invoke(nonce=1, sender=Alice, calldata=arbitrary, sig=garbage)`.
   - Gateway `validate_nonce`: `account_nonce=0 ≤ tx_nonce=1 ≤ max_allowed` → passes.
   - Gateway `validate_by_mempool` → mempool `validate_tx` checks duplicate hash/nonce → passes (no nonce-1 tx yet, or fee escalation allows replacement).
   - Gateway `skip_stateful_validations`: `tx.nonce()==1 && account_nonce==0 && account_tx_in_pool_or_recent_block==true` → returns `true`.
   - `run_validate_entry_point` sets `validate=false` → `__validate__` **never called**.
   - Attacker's invoke with invalid signature is **admitted to the mempool**.
4. If fee escalation is enabled (`enable_fee_escalation: true`) and attacker's tip/gas-price exceeds Alice's invoke (nonce=1), `validate_fee_escalation` (lines 760-792) allows the replacement, and Alice's legitimate invoke is **evicted** from the pool.
5. Block is built: Alice's `deploy_account` executes (account deployed), then attacker's invoke executes — blockifier calls `__validate__` with `validate=true` → fails (invalid signature) → reverted. Alice's invoke is gone; she must resubmit.

**Why the blockifier does not save the invariant at admission time**

The gateway's `run_validate_entry_point` is the *only* place where `__validate__` is called during admission. Once the transaction is in the mempool as an `InternalRpcTransaction`, the signature is never re-checked before execution. The blockifier will catch the invalid signature at execution time, but by then the legitimate transaction may already have been displaced.

---

### Impact Explanation

Matches **High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

- Any unprivileged actor can inject a signature-less invoke for any account whose `deploy_account` is pending in the mempool.
- With fee escalation enabled, the attacker can displace the victim's legitimate first invoke, forcing a resubmission and potentially delaying the account's first on-chain action.
- The attacker pays fees for the reverted transaction, but the cost is bounded and predictable, making repeated DoS economically feasible against high-value targets.

---

### Likelihood Explanation

Medium. The attacker must:
- Monitor the mempool for `deploy_account` transactions (trivially possible via the public gateway).
- Submit the competing invoke before the block containing the `deploy_account` is committed.
- Pay fees for the reverted transaction.

The window is one block time, which is short but sufficient for an automated attacker. No privileged access is required.

---

### Recommendation

Replace the address-only pool check with a type-specific check. The mempool should expose a `deploy_account_tx_in_pool_or_recent_block(address)` query that returns `true` only when the existing pool entry for that address is a `DeployAccount` transaction. `skip_stateful_validations` should call this narrower predicate instead of `account_tx_in_pool_or_recent_block`.

Alternatively, the gateway can inspect the type of the existing pool transaction before deciding to skip validation:

```rust
// Pseudocode
let existing_tx = mempool_client.get_tx_by_address_and_nonce(address, Nonce::ZERO).await?;
let is_deploy_account = matches!(existing_tx.tx, InternalRpcTransactionWithoutTxHash::DeployAccount(_));
return Ok(is_deploy_account);
```

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
