### Title
Integer Underflow in `ComputationMeter.ComputationRemaining` Silently Reports Unlimited Capacity After Limit Exceeded via Merge - (File: `fvm/meter/computation_meter.go`)

---

### Summary

`ComputationMeter.ComputationRemaining` performs an unchecked `uint64` subtraction `m.params.computationLimit - m.computationUsed`. When `computationUsed` exceeds `computationLimit` — a state that is explicitly permitted and expected after `Merge` — the subtraction silently wraps to a near-`MaxUint64` value. The intended guard `if remainingComputationUsage <= 0` is dead code for `uint64` (a `uint64` can never be negative or zero after underflow), so the function returns a huge number instead of `0`, falsely reporting that nearly unlimited computation remains.

---

### Finding Description

In `fvm/meter/computation_meter.go`, the `ComputationRemaining` function computes remaining capacity as:

```go
remainingComputationUsage := m.params.computationLimit - m.computationUsed
if remainingComputationUsage <= 0 {   // ← dead code: uint64 is never < 0
    return 0
}
return remainingComputationUsage / w
``` [1](#0-0) 

The `Merge` function unconditionally adds child usage to the parent without any limit check or error:

```go
func (m *ComputationMeter) Merge(child ComputationMeter) {
    m.computationUsed = m.computationUsed + child.computationUsed
    ...
}
``` [2](#0-1) 

The test suite explicitly documents and accepts this behavior — "merge hits limit, but is accepted" — confirming that `computationUsed > computationLimit` is a reachable state after `Merge`: [3](#0-2) 

Once `computationUsed > computationLimit`, calling `ComputationRemaining` produces:

- `computationLimit - computationUsed` wraps to approximately `MaxUint64 - (computationUsed - computationLimit)`
- The `<= 0` guard never fires (uint64 arithmetic)
- The function returns `~MaxUint64 / w`, a near-maximum value

This is the same class of defect as the Stars.sol report: an arithmetic subtraction that can underflow, with a guard that does not actually protect against the underflow case.

---

### Impact Explanation

`ComputationRemaining` is the primary interface used by the FVM and EVM integration layer to query how much computation of a given kind is still available before committing to an operation. When it returns a near-`MaxUint64` value after the limit has been exceeded via `Merge`, any caller that relies on this value to gate further execution — including EVM gas allocation derived from remaining Flow computation — will believe the transaction has nearly unlimited remaining capacity. This can allow a transaction to consume more computation than its declared gas limit permits, undermining the computation metering that protects execution nodes from resource exhaustion and ensures fee correctness.

---

### Likelihood Explanation

`Merge` is called on every transaction that uses nested execution contexts (inner function calls, EVM calls, etc.), which is the normal case for any non-trivial transaction. A transaction that uses computation close to its limit in the outer context and then executes a child context will trigger the post-merge overflow condition. This is reachable by any unprivileged transaction sender submitting a standard Cadence or EVM transaction.

---

### Recommendation

1. Add an explicit underflow guard before the subtraction in `ComputationRemaining`:

```go
if m.computationUsed >= m.params.computationLimit {
    return 0
}
remainingComputationUsage := m.params.computationLimit - m.computationUsed
```

2. Add an overflow check in `Merge` and return or propagate an error if the merged total exceeds the limit, consistent with how `MeterComputation` handles the same condition.

3. Add an overflow check in `MeterComputation` for the `w * intensity` multiplication at line 100, analogous to the SafeMath recommendation in the original report:

```go
// Before: m.computationUsed += w * intensity
product := w * intensity
if w != 0 && product/w != intensity {
    return errors.NewComputationLimitExceededError(...)
}
m.computationUsed += product
``` [4](#0-3) 

---

### Proof of Concept

The existing test at `fvm/meter/meter_test.go` lines 297–311 already demonstrates the reachable state: [3](#0-2) 

After `m.MergeMeter(child3)` causes `computationUsed` to exceed `computationLimit`, calling `m.ComputationRemaining(compKind)` returns `(computationLimit - computationUsed) / w` where the subtraction has wrapped to a near-`MaxUint64` value, instead of returning `0`. Any subsequent code path that uses this return value to decide whether to proceed with computation will incorrectly conclude that nearly unlimited computation remains.
