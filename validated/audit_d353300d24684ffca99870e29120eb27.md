### Title
Unimplemented `add_batch_actions` in XCC Router Panics and Freezes User wNEAR Funds - (File: `etc/xcc-router/src/lib.rs`)

---

### Summary

The XCC router contract contains a stub implementation of `add_batch_actions` that unconditionally panics via `unimplemented!()` when the `all-promise-actions` Cargo feature is absent. Because the XCC precompile transfers wNEAR from the caller to the engine's implicit EVM address *before* the router promise is dispatched, a user who submits a `PromiseArgs::Recursive` payload containing any `SimpleNearPromise::Batch` action will have their wNEAR committed to the engine's implicit address while the router promise fails, leaving the tokens permanently inaccessible.

---

### Finding Description

**Root cause — `etc/xcc-router/src/lib.rs` lines 284–286:**

```rust
#[cfg(not(feature = "all-promise-actions"))]
fn add_batch_actions(_id: PromiseIndex, _actions: &[PromiseAction]) {
    unimplemented!("NEAR batch transactions are not supported. …")
}
``` [1](#0-0) 

This stub is reached from `recursive_promise_create` whenever the deserialized `PromiseArgs` contains a `SimpleNearPromise::Batch` variant:

```rust
SimpleNearPromise::Batch(batch) => {
    let target = batch.target_account_id.as_ref().parse().unwrap();
    let id = env::promise_batch_create(&target);
    Self::add_batch_actions(id, &batch.actions);   // ← panics
    id
}
``` [2](#0-1) 

The same panic path exists for the `NearPromise::Then { callback: SimpleNearPromise::Batch(…) }` arm: [3](#0-2) 

`recursive_promise_create` is called from `promise_create`, which is called from the public `execute` entry-point: [4](#0-3) 

**Fund-freeze mechanism — `engine-precompiles/src/xcc.rs`:**

Before the router promise is ever dispatched, the XCC precompile performs an EVM-level `transferFrom` that moves wNEAR from the caller to the engine's implicit EVM address:

```rust
let (exit_reason, return_value) =
    handle.call(wnear_address.raw(), None, tx_data, None, false, &context);
``` [5](#0-4) 

This EVM state change is committed when the outer EVM transaction succeeds. The NEAR promise to the router is dispatched asynchronously afterward. When the router panics, NEAR rolls back only the router's own state changes — the already-committed EVM wNEAR transfer is unaffected. The wNEAR now sits at the engine's implicit EVM address, which is not controlled by any private key and has no built-in recovery path.

---

### Impact Explanation

**High — Temporary (potentially permanent) freezing of user funds.**

Any wNEAR charged for storage staking (`STORAGE_AMOUNT = 2 NEAR`) or for attached NEAR in the cross-contract call is transferred to the engine's implicit EVM address before the router executes. When the router panics, those tokens are stranded. Recovery requires out-of-band admin action (contract upgrade or a bespoke admin EVM call), which is not guaranteed and may not be possible for the implicit address. [6](#0-5) 

---

### Likelihood Explanation

The XCC precompile accepts arbitrary Borsh-encoded `CrossContractCallArgs` from any EVM caller. A `PromiseArgs::Recursive` payload containing a `SimpleNearPromise::Batch` variant is valid Borsh and requires no special privilege to construct. Any unprivileged EVM user who attempts to use NEAR batch-transaction semantics via the XCC precompile (a documented feature of the `PromiseArgs` type) will trigger the panic and lose their wNEAR. [7](#0-6) 

---

### Recommendation

Two complementary fixes:

1. **Fail-safe in the precompile**: Before dispatching the router promise, validate that the `PromiseArgs` payload contains no `SimpleNearPromise::Batch` variants when the `all-promise-actions` feature is absent. Return a `PrecompileFailure::Revert` (before the wNEAR transfer) so the entire EVM transaction reverts atomically and no funds are lost.

2. **Implement `add_batch_actions` unconditionally**: Remove the feature gate and implement the function for all builds, or at minimum replace `unimplemented!()` with a graceful `env::panic_str("ERR_BATCH_NOT_SUPPORTED")` *before* the wNEAR transfer occurs. [1](#0-0) 

---

### Proof of Concept

1. Deploy Aurora Engine without the `all-promise-actions` feature (default build).
2. As an unprivileged EVM user, call the XCC precompile (`0x516cded1d16af10cad47d6d49128e2eb7d27b372`) with a Borsh-encoded `CrossContractCallArgs::Eager(PromiseArgs::Recursive(NearPromise::Simple(SimpleNearPromise::Batch { target_account_id: <any>, actions: [PromiseAction::Transfer { amount: 1 }] })))`.
3. Observe: the EVM transaction succeeds and the wNEAR `transferFrom` (user → engine implicit address) is committed to EVM state.
4. Observe: the subsequent NEAR promise to `router.execute(…)` panics at `add_batch_actions` with `not implemented`.
5. Confirm: the user's wNEAR is now held at the engine's implicit EVM address with no recovery path. [4](#0-3) [8](#0-7)
