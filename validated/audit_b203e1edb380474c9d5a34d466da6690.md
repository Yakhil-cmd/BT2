### Title
Zero-Gas Function Call via `VMRunnerError::LoadingError` on Oversized WASM Instance (pre-`FixContractLoadingError`) — (`runtime/near-vm-runner/src/wasmtime_runner/mod.rs`)

### Summary

Before protocol version 86 (`FixContractLoadingError`), an unprivileged user can deploy a valid WASM binary containing 100,000 globals, then call it repeatedly. Each call causes `Module::deserialize` to fail inside `WasmtimeVM::with_compiled_and_loaded` because the Wasmtime pooling allocator's `max_core_instance_size` (1 MiB) is exceeded. The resulting `VMRunnerError::LoadingError` is unconditionally mapped to `VMOutcome::nop_outcome` in `runtime/runtime/src/function_call.rs`, which sets `burnt_gas = 0` and `used_gas = 0`. The attacker pays zero execution gas while validators perform real deserialization work on every call.

### Finding Description

**Step 1 — Deploy.** An unprivileged user submits a `DeployContract` action with a WASM binary containing 100,000 mutable globals. Each global contributes 8 bytes of instance data (800 kB total), which is valid WASM and passes all preparation/validation checks. The deployment costs gas proportional to code size, but that is a one-time cost.

**Step 2 — Call.** The user submits a `FunctionCall` action targeting the deployed contract. Inside `WasmtimeVM::with_compiled_and_loaded`, the compiled artifact is read from the disk cache and `Module::deserialize` is called: [1](#0-0) 

When `fix_contract_loading_error` is `false` (protocol version < 86), the `Err` branch at line 762 returns `Err(VMRunnerError::LoadingError(...))` directly — bypassing the memory cache (the closure returns `Err`, so `try_lookup` propagates it via `?` without caching).

**Step 3 — Zero-gas mapping.** Back in `function_call.rs`, the `LoadingError` arm unconditionally returns a nop outcome: [2](#0-1) 

`VMOutcome::nop_outcome` explicitly sets both `burnt_gas` and `used_gas` to zero: [3](#0-2) 

**Step 4 — Repeat.** Because the deserialization failure is not cached in the memory cache, every subsequent call re-reads the compiled artifact from disk and re-attempts `Module::deserialize`, failing again with zero gas charged each time.

The test in the repository explicitly documents this behavior: [4](#0-3) 

### Impact Explanation

The exact corrupted ledger value is `burnt_gas = 0` and `used_gas = 0` on a function-call receipt that caused real validator compute work (compiled-artifact I/O + Wasmtime deserialization of an ~800 kB instance). The gas-charging invariant — that all execution work is charged to the caller — is violated. An attacker can spam calls to the oversized contract, consuming validator CPU and I/O for free beyond the base transaction fee.

### Likelihood Explanation

The attack requires only standard unprivileged actions: `DeployContract` (one-time, pays code-size gas) followed by repeated `FunctionCall` actions. No validator, node-admin, or trusted-service access is needed. The WASM binary is trivially constructable (100k globals is valid WASM). The vulnerable path is active on any node running protocol version < 86.

### Recommendation

The fix is already implemented and gated behind `FixContractLoadingError` (protocol version 86): [5](#0-4) 

Ensure protocol version 86 is activated on mainnet. No additional code change is needed once the feature is live.
