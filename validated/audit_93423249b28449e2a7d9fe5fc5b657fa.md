### Title
`native_per_pubdata` Rounds Down (Floor Division) While `native_per_gas` Rounds Up (Ceiling Division), Causing Users to Underpay Native Resources for Pubdata - (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

In ZKsync OS's double-resource accounting model, the conversion from `pubdata_price` to `native_per_pubdata` uses floor division (`wrapping_div`), while the analogous conversion from `gas_price` to `native_per_gas` uses ceiling division (`div_ceil`). This inconsistency means users are charged slightly less native resource per pubdata byte than they should be, allowing them to publish more pubdata than their gas fee strictly covers.

---

### Finding Description

In `validation_impl.rs`, `native_per_gas` is computed with ceiling division (protocol-favorable rounding):

```rust
let native_per_gas = u256_try_to_u64(&gas_price.div_ceil(native_price)).ok_or(TxError::Validation(
    InvalidTransaction::NativeResourcesAreTooExpensive,
))?
``` [1](#0-0) 

But `native_per_pubdata` is computed with floor division (user-favorable rounding):

```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
``` [2](#0-1) 

The same floor-division pattern is repeated in `api/src/helpers.rs`:

```rust
// native_per_pubdata = pubdata_price / native_price
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
``` [3](#0-2) 

`native_per_pubdata` is subsequently used in two places that charge the user:

1. **Intrinsic pubdata overhead** deducted from the user's native limit upfront:
   `intrinsic_pubdata_overhead = native_per_pubdata * intrinsic_pubdata` [4](#0-3) 

2. **Per-transaction pubdata charge** at execution end:
   `native = current_pubdata_spent * native_per_pubdata` [5](#0-4) 

Additionally, the `delta_gas` conversion in `refund_calculation.rs` also uses floor division:

```rust
(native_used / native_per_gas) as i64 - (gas_used as i64)
``` [6](#0-5) 

This means that even after `native_used` is already understated (due to the `native_per_pubdata` floor rounding), the conversion back to `delta_gas` also rounds down, compounding the underpayment.

The documentation confirms the intended formula is `nativePerGas := gasPrice/nativePrice` and `deltaGas := (nativeUsed / nativePerGas) - gasUsed`, but does not specify rounding direction. The code's own choice to use `div_ceil` for `native_per_gas` establishes the protocol-favorable convention that `native_per_pubdata` violates. [7](#0-6) 

---

### Impact Explanation

The combined effect of both floor-division sites:

1. `native_per_pubdata` is at most `(native_price − 1)` units lower than the ceiling value. For `pubdata_used` bytes, the user's `native_used` is understated by up to `pubdata_used × (native_price − 1)` native units.
2. `delta_gas = native_used / native_per_gas` is then also rounded down, understating `gas_used` by up to 1 additional gas unit.

The maximum financial underpayment per transaction is:

```
pubdata_used × (native_price − 1) / native_per_gas  +  1  gas units
```

multiplied by `gas_price` in ETH. For example, with `native_price = 10`, `pubdata_used = 1000 bytes`, `native_per_gas = 1000`, `gas_price = 1 gwei`, the protocol loses up to ~9 gwei per transaction. Across high-throughput blocks, this accumulates. When `pubdata_price < native_price`, `native_per_pubdata` collapses to 0, meaning the user pays **zero** native resource for all pubdata — a complete bypass of pubdata native charging.

---

### Likelihood Explanation

Every L2 transaction that generates pubdata is affected. The rounding error is present unconditionally in the production validation path. An attacker can craft transactions that maximize `pubdata_used` (e.g., many SSTORE operations) to maximize the underpayment per transaction. No privileged access is required — any unprivileged transaction sender triggers this path.

---

### Recommendation

Replace `wrapping_div` with `div_ceil` for `native_per_pubdata` in both locations, consistent with how `native_per_gas` is computed:

```rust
// In validation_impl.rs
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;

// In api/src/helpers.rs
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price)).ok_or(())?;
```

For `delta_gas`, consider using ceiling division:
```rust
let delta_gas = native_used.div_ceil(native_per_gas) as i64 - gas_used as i64;
```

### Proof of Concept

**Setup:** `pubdata_price = 101`, `native_price = 10`, `gas_price = 1000`, `native_per_gas = ceil(1000/10) = 100`.

**Current behavior (floor):**
- `native_per_pubdata = floor(101/10) = 10`
- For 1000 bytes of pubdata: `native_charged_for_pubdata = 10 × 1000 = 10_000`
- `native_used = 10_000`
- `delta_gas = floor(10_000 / 100) = 100` gas units charged

**Correct behavior (ceil):**
- `native_per_pubdata = ceil(101/10) = 11`
- For 1000 bytes of pubdata: `native_charged_for_pubdata = 11 × 1000 = 11_000`
- `native_used = 11_000`
- `delta_gas = ceil(11_000 / 100) = 110` gas units charged

**Underpayment:** 10 gas units × `gas_price` per transaction. A user submitting 10,000 such transactions underpays by 100,000 gas units × `gas_price` in total. At `gas_price = 1 gwei`, this is 0.0001 ETH — small per batch but unbounded over time.

First we define the ratio between EVM gas and native resource as:
  `nativePerGas := gasPrice/nativePrice`
Note: for call simulation we use a constant for it, as gasPrice might be set to 0.

Next we define the limit for the native resource as:
  `nativeLimit := gasLimit * nativePerGas`

Then we process the transaction, charging both Ergs for EE execution and native resource for any kind of computation (EE, bootloader or system work).

If execution doesn't run out of native resources, we first charge for pubdata from native resource.
Then we compute the difference between the implicit gas used derived from native resource consumption and the gas used by EEs from the ergs used. We call this value `deltaGas`.
  `deltaGas := (nativeUsed / nativePerGas) - gasUsed`

If `deltaGas > 0`, we add it to `gasUsed` and charge it from ergs. This ensures that gas estimation will include additional gas to cover for native resources using just base fee. We expect the base fee to be enough to cover most transactions without the need of additional gas.
