### Title
Notional Fee Dust Permanently Lost on `collectFees` Due to Scaled-to-External Truncation — (`File: metric-core/contracts/MetricOmmPool.sol`)

---

### Summary

`collectFees` unconditionally resets `notionalFeeToken0Scaled` and `notionalFeeToken1Scaled` to zero after converting them to external token amounts via `deltasScaledToExternal(..., Math.Rounding.Floor)`. For tokens with fewer than 18 decimals (e.g., USDC, 6 decimals → `TOKEN_0_SCALE_MULTIPLIER = 10^12`), any accumulated notional fee below the scale multiplier truncates to zero external tokens — no transfer occurs — yet the scaled state is cleared. The dust is permanently lost to protocol and admin fee recipients. Because `collectPoolFees` is a public, permissionless function, an adversary can call it repeatedly after each small swap to prevent notional fees from ever accumulating past the truncation threshold.

---

### Finding Description

**Bug class (from external report):** Integer division truncation causes a financial operation to silently produce zero output while consuming the input state.

**Metric OMM analog:**

In `MetricOmmPool.collectFees`, notional fees accumulated in scaled units are converted to external token amounts:

```solidity
(uint256 totalFee0ToAdmin, uint256 totalFee1ToAdmin) =
    deltasScaledToExternal(totalFee0ToAdminScaled, totalFee1ToAdminScaled, Math.Rounding.Floor);
(uint256 totalFee0ToProtocol, uint256 totalFee1ToProtocol) =
    deltasScaledToExternal(totalFee0ToProtocolScaled, totalFee1ToProtocolScaled, Math.Rounding.Floor);
```

`deltasScaledToExternal` divides by `TOKEN_0_SCALE_MULTIPLIER` / `TOKEN_1_SCALE_MULTIPLIER` with floor rounding. For a 6-decimal token, the multiplier is `10^12`. Any scaled notional fee amount below `10^12` produces an external amount of **zero** — no transfer is issued.

Immediately after, regardless of whether any transfer occurred:

```solidity
notionalFeeToken0Scaled = 0;
notionalFeeToken1Scaled = 0;
```

The scaled fee state is cleared unconditionally. The tokens that were tracked as owed notional fees remain in the pool balance but are no longer attributed to fee recipients — they silently become pool surplus, benefiting LPs rather than the protocol/admin.

The trigger is `collectPoolFees`, which has no access control:

```solidity
function collectPoolFees(address pool) external override nonReentrant {
```

Any caller can invoke it at any time.

---

### Impact Explanation

- **Protocol and admin notional fees are permanently lost** whenever `collectFees` is called while accumulated notional fees are below the scale-multiplier threshold.
- For USDC/USDT pools (6 decimals, multiplier `10^12`), any notional fee accumulation below 1 raw USDC unit is silently zeroed and cleared.
- The lost fees are re-classified as pool surplus, which benefits LPs — a redistribution from fee recipients to LPs that is not intended by the protocol design.
- The loss is bounded per call but is **unbounded in aggregate** because the attack can be repeated indefinitely after every small swap.

---

### Likelihood Explanation

- `collectPoolFees` is **permissionless** — any EOA or contract can call it.
- An adversary monitoring the mempool can call `collectPoolFees` immediately after each swap that generates a small notional fee, preventing accumulation past the truncation threshold.
- Pools with low-decimal tokens (USDC, USDT) are the primary target; 18-decimal token pools have a multiplier of 1 and are unaffected.
- No special role, no malicious setup, no non-standard token behavior required.

---

### Recommendation

Before clearing `notionalFeeToken0Scaled` and `notionalFeeToken1Scaled`, verify that the full scaled amount was actually transferred. If the external conversion truncated to zero, retain the residual scaled amount rather than discarding it:

```solidity
// Instead of unconditionally zeroing:
notionalFeeToken0Scaled = 0;
notionalFeeToken1Scaled = 0;

// Compute residuals and retain them:
uint256 transferred0Scaled = uint256(totalFee0ToAdmin + totalFee0ToProtocol) * TOKEN_0_SCALE_MULTIPLIER;
uint256 transferred1Scaled = uint256(totalFee1ToAdmin + totalFee1ToProtocol) * TOKEN_1_SCALE_MULTIPLIER;
notionalFeeToken0Scaled = uint128(notionalFee0AmountScaled - transferred0Scaled);
notionalFeeToken1Scaled = uint128(notionalFee1AmountScaled - transferred1Scaled);
```

This ensures dust accumulates across multiple `collectFees` calls until it reaches a full external unit.

### Proof of Concept

1. Deploy a pool with USDC (6 decimals) as token0. `TOKEN_0_SCALE_MULTIPLIER = 10^12`. Set `adminNotionalFeeE8 = 1e6` (1%).
2. Execute a swap that generates `notionalFeeToken0Scaled = 5 * 10^11` (0.5 raw USDC in scaled units, below the `10^12` threshold).
3. Call `collectPoolFees(pool)` (permissionless).
4. Inside `collectFees`:
   - `notionalFee0AmountScaled = 5 * 10^11`
   - `totalFee0ToAdminScaled + totalFee0ToProtocolScaled = 5 * 10^11`
   - `deltasScaledToExternal(5e11, ..., Floor)` → `5e11 / 1e12 = 0` (floor division)
   - No token0 transfer to admin or protocol
   - `notionalFeeToken0Scaled = 0` ← cleared unconditionally
5. The 0.5 scaled USDC of notional fees is permanently lost to fee recipients.
6. Repeat after every small swap to prevent any notional fee from ever reaching 1 USDC. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
