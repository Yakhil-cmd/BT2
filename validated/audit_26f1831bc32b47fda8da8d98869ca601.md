### Title
`CadenceOwnedAccount.deposit()` Missing `isPaused()` Guard While All Other State-Mutating COA Functions Are Protected - (File: `fvm/evm/stdlib/contract.cdc`)

### Summary
`EVM.CadenceOwnedAccount.deposit()` is the only state-mutating function on the `CadenceOwnedAccount` resource that does not check `!EVM.isPaused()` before executing. Every other state-mutating COA function — `withdraw`, `deploy`, `call`, `callWithSigAndArgs`, `depositNFT`, `withdrawNFT`, `depositTokens`, `withdrawTokens` — and every top-level EVM entry point — `run`, `batchRun`, `createCadenceOwnedAccount` — all carry the guard. `deposit()` does not.

### Finding Description
`EVM.isPaused()` reads a governance-controlled boolean from `/storage/evmOperationsPaused` and is intended to put the EVM into a read-only mode where no state updates are executed. [1](#0-0) 

Every state-mutating function on `CadenceOwnedAccount` enforces this guard in a `pre` condition, for example `withdraw`: [2](#0-1) 

`deploy`, `call`, `callWithSigAndArgs`, `depositNFT`, `withdrawNFT`, `depositTokens`, and `withdrawTokens` all follow the same pattern: [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5) 

`CadenceOwnedAccount.deposit()`, however, has no such guard: [7](#0-6) 

It calls `self.address().deposit(from: <-from)` directly, which transfers FLOW tokens from Cadence into the COA's EVM balance — a concrete EVM state mutation — without checking the pause flag.

The test suite confirms this is unintended: `TestEVMPauseFunctionality` includes a sub-test that calls `coa.deposit()` while EVM is paused and asserts the transaction must fail with `"EVM operations are temporarily paused"`: [8](#0-7) 

The absence of the guard in the contract means this assertion fails at runtime — the deposit succeeds when it should be rejected.

### Impact Explanation
When the Governance Committee activates the pause to halt all EVM state changes, any Cadence transaction can still call `coa.deposit(from: <-vault)` to move FLOW tokens from Cadence into the EVM environment, mutating EVM account balances. This breaks the invariant that EVM is in a read-only state during the pause, and is directly inconsistent with the behavior of every other COA state-mutating function.

### Likelihood Explanation
The entry path is a standard, unprivileged Cadence transaction. Any account that holds a `CadenceOwnedAccount` resource (or a public reference to one) can call `deposit()`. No special entitlement is required — the function is `access(all)`. The pause is a governance-level emergency measure, so the window during which this matters is exactly when correctness of the restriction is most critical.

### Recommendation
Add the same `pre` condition guard used by every other state-mutating COA function:

```cadence
access(all)
fun deposit(from: @FlowToken.Vault) {
    pre {
        !EVM.isPaused(): "EVM operations are temporarily paused"
    }
    self.address().deposit(from: <-from)
}
```

### Proof of Concept
1. Governance activates the EVM pause by storing `true` at `/storage/evmOperationsPaused` on the EVM contract account.
2. `EVM.isPaused()` now returns `true`.
3. Any call to `coa.withdraw()`, `coa.call()`, `EVM.run()`, etc. reverts with `"EVM operations are temporarily paused"`.
4. A transaction calling `coa.deposit(from: <-vault)` succeeds — no revert — and the COA's EVM balance increases, mutating EVM state in violation of the pause invariant. [7](#0-6) [9](#0-8)
