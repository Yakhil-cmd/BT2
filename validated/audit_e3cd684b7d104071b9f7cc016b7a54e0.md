### Title
`protocolUnpausePool` Always Transitions to Admin-Paused State, Leaving Previously-Active Pools Permanently Stuck — (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`protocolUnpausePool` hardcodes the target pause level to `1` (admin-paused) regardless of the pool's pre-pause state. When the protocol pauses an active pool (`0 → 2`) and then unpauses it (`2 → 1`), the pool is left in admin-paused state even though it was never admin-paused. Core functionality — swaps, add/remove liquidity — remains broken until the pool admin manually calls `unpausePool`. This is the direct analog of the external bug: a recovery/protective operation (unpause) fails to restore the system to its operational state due to an incorrect fixed value applied during the transition.

---

### Finding Description

In `MetricOmmPoolFactory.sol`, `protocolPausePool` can transition a pool from level `0` (active) **or** level `1` (admin-paused) to level `2` (protocol-paused):

```solidity
// MetricOmmPoolFactory.sol L392-L396
function protocolPausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 0 && cur != 1) revert InvalidPauseTransition(cur, 2);
    IMetricOmmPoolFactoryActions(pool).setPause(2);
}
```

`protocolUnpausePool` then **always** transitions to level `1`, regardless of what the level was before the protocol pause:

```solidity
// MetricOmmPoolFactory.sol L398-L403
function protocolUnpausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 2) revert InvalidPauseTransition(cur, 1);
    IMetricOmmPoolFactoryActions(pool).setPause(1);
}
```

The factory stores no pre-pause level. When the pool was at level `0` (active) before the protocol pause, after `protocolUnpausePool` it is at level `1` (admin-paused) — a state it was never in. All swap and liquidity operations then revert because `_checkNotPaused` requires `pauseLevel == 0`:

```solidity
// MetricOmmPool.sol L643-L645
function _checkNotPaused() internal view {
    if (pauseLevel != 0) revert PoolPaused();
}
```

The pool is stuck until the pool admin manually calls `unpausePool`, which enforces `cur == 1 → 0`. The admin may be a slow multisig, unaware of the requirement, or unavailable.

---

### Impact Explanation

After a protocol pause-unpause cycle on a previously-active pool, all core operations — `swap`, `addLiquidity`, `removeLiquidity` — revert with `PoolPaused`. LPs cannot exit positions. Traders cannot execute. The pool is effectively bricked until the pool admin takes a separate, unannounced action. This matches the allowed impact gate: **broken core pool functionality causing unusable withdraw/swap/liquidity flows**.

---

### Likelihood Explanation

Emergency protocol pauses are a standard operational scenario (e.g., oracle failure, market stress). The protocol owner is a valid semi-trusted actor performing a legitimate operation. The bug triggers on every protocol pause-unpause cycle where the pool was active (`level == 0`) at the time of the pause — which is the common case. No malicious setup is required.

---

### Recommendation

Store the pre-pause level in the factory and restore it on unpause:

```solidity
mapping(address => uint8) public prePauseLevel;

function protocolPausePool(address pool) external onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 0 && cur != 1) revert InvalidPauseTransition(cur, 2);
    prePauseLevel[pool] = cur;  // store pre-pause state
    IMetricOmmPoolFactoryActions(pool).setPause(2);
}

function protocolUnpausePool(address pool) external onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 2) revert InvalidPauseTransition(cur, prePauseLevel[pool]);
    IMetricOmmPoolFactoryActions(pool).setPause(prePauseLevel[pool]);  // restore
    delete prePauseLevel[pool];
}
```

---

### Proof of Concept

1. Pool is deployed and active: `pauseLevel == 0`.
2. Protocol owner calls `protocolPausePool(pool)` — pool transitions `0 → 2`.
3. Emergency resolved; protocol owner calls `protocolUnpausePool(pool)` — pool transitions `2 → 1` (hardcoded).
4. Pool is now at `pauseLevel == 1` (admin-paused) even though the admin never paused it.
5. Any call to `swap`, `addLiquidity`, or `removeLiquidity` reverts with `PoolPaused`.
6. Only the pool admin calling `unpausePool` (which enforces `cur == 1 → 0`) can restore the pool — an action the admin was never notified to take and may not perform promptly. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
