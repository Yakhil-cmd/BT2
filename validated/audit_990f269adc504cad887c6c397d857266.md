### Title
Missing Length Validation of Deserialized Commitment Vector Causes Panic in Triple Generation — (`File: src/ecdsa/ot_based_ecdsa/triples/generation.rs`)

### Summary

In `do_generation_many`, a peer-supplied `Vec<Commitment>` is deserialized from a network message and immediately indexed with `commitments[i]` for `i in 0..N` without first verifying that the vector contains exactly `N` elements. A single malicious participant can send a shorter-than-expected vector, triggering an out-of-bounds index panic that crashes the triple generation task for all honest participants, permanently denying signing capability until the session is manually restarted.

### Finding Description

The function `do_generation_many<const N: usize>` in `src/ecdsa/ot_based_ecdsa/triples/generation.rs` receives polynomial commitment messages from peers via `chan.recv(wait0)`. The deserialized type is `Vec<Commitment>` — a heap-allocated vector whose length is entirely controlled by the sender. After deserialization, the code immediately iterates `for i in 0..N` and accesses `commitments[i]`:

```rust
// lines 176-184
while all_commitments_vec
    .iter()
    .any(|all_commitments| !all_commitments.full())
{
    let (from, commitments): (_, Vec<_>) = chan.recv(wait0).await?;
    for i in 0..N {
        all_commitments_vec[i].put(from, commitments[i]);  // ← panics if commitments.len() < N
    }
}
```

There is no guard of the form `if commitments.len() != N { return Err(...); }` before the indexed access. In Rust, `Vec::index` panics (does not return `Err`) when the index is out of bounds. Because the protocol runs inside an async task, this panic propagates as an unrecoverable task failure, aborting the entire triple generation session for every honest participant waiting on that session.

The deserialization layer (`rmp_serde::decode::from_slice` in `src/protocol/internal.rs:339`) faithfully deserializes whatever byte sequence the peer sends. A `Vec` with zero elements is a perfectly valid MessagePack encoding and will deserialize without error, passing the type check. The length mismatch is only discovered — fatally — at the first `commitments[0]` access.

### Impact Explanation

Triple generation is a prerequisite for OT-based ECDSA presigning and signing. A panic in `do_generation_many` terminates the session for all honest participants simultaneously. Because the malicious participant can repeat this attack on every new session attempt, honest parties are permanently denied the ability to generate triples and therefore permanently denied signing capability. This matches the **High** allowed impact: *Permanent denial of signing for honest parties under valid protocol inputs and documented trust assumptions.*

### Likelihood Explanation

Any participant admitted to the triple generation protocol — including a threshold-1 coalition of colluding participants — can trigger this with a single crafted message. No cryptographic material needs to be compromised. The attacker only needs to be a valid protocol participant (the documented malicious-participant threat model). The attack is deterministic and requires no timing or side-channel capability.

### Recommendation

Add an explicit length check immediately after deserialization, before the indexed loop:

```rust
let (from, commitments): (_, Vec<_>) = chan.recv(wait0).await?;
if commitments.len() != N {
    return Err(ProtocolError::AssertionFailed(format!(
        "expected {N} commitments from {:?}, got {}",
        from,
        commitments.len()
    )));
}
for i in 0..N {
    all_commitments_vec[i].put(from, commitments[i]);
}
```

Alternatively, deserialize into a fixed-size array `[Commitment; N]` so the type system enforces the length at deserialization time, making the mismatch a recoverable `Err` rather than a panic.

### Proof of Concept

1. Honest parties `A`, `B`, `C` initiate `do_generation_many::<4>(...)` (N=4 triples).
2. Malicious participant `M` intercepts the `wait0` waitpoint and sends a `PolynomialCommitmentsMessageMany`-shaped MessagePack payload where `big_e_v`, `big_f_v`, etc. are empty vectors (length 0).
3. `chan.recv(wait0)` on any honest party deserializes this as `Vec<Commitment>` with length 0 — no error.
4. The loop `for i in 0..4` executes `commitments[0]`, which panics with `index out of bounds: the len is 0 but the index is 0`.
5. The async task for triple generation terminates with a panic for all honest participants.
6. No triples are produced; presigning and signing are impossible until a new session is started, which `M` can immediately attack again. [1](#0-0) [2](#0-1) [3](#0-2)
