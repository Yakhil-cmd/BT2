### Title
Malicious CKD Participant Can Corrupt Derived Key Without Proof Verification - (File: src/confidential_key_derivation/protocol.rs)

### Summary
The `do_ckd_coordinator` function in the CKD protocol aggregates participant shares `(big_y, big_c)` without any proof of correctness. A single malicious participant can send arbitrary group elements, causing the coordinator to output a silently corrupted derived key that honest parties will accept as valid.

### Finding Description

In `do_ckd_coordinator`, the coordinator receives each participant's `(norm_big_y, norm_big_c)` contribution and blindly adds them together: [1](#0-0) 

Each participant is supposed to compute and send:
- `norm_big_y = λi · yi · G`
- `norm_big_c = λi · (xi · H(pk, app_id) + yi · app_pk)`

as computed in `compute_signature_share`: [2](#0-1) 

There is **no zero-knowledge proof, no pairing check, and no consistency verification** that the received `(big_y, big_c)` pair was computed using the participant's actual secret key share `xi` or that `yi` is consistent between the two components. The coordinator simply sums whatever it receives: [3](#0-2) 

Contrast this with the robust ECDSA signing coordinator (`do_sign_coordinator`), which has a final `sig.verify(&public_key, &msg_hash)` check that catches corrupted shares before returning: [4](#0-3) 

The CKD coordinator has no equivalent safeguard.

### Impact Explanation

The final CKD output is `(Y, C)` where the application recovers the confidential key as `C − app_sk · Y = msk · H(pk, app_id)`. If participant `i` sends `(big_y', big_c')` instead of their legitimate share, the coordinator computes:

```
Y_final  = Y_honest + (big_y' − λi·yi·G)
C_final  = C_honest + (big_c' − λi·(xi·H + yi·app_pk))
```

The application then decrypts:
```
C_final − app_sk·Y_final = msk·H + big_c' − λi·xi·H − app_sk·big_y'
```

For example, a malicious participant sending `(0, 0)` removes their legitimate contribution, yielding `(msk − λi·xi)·H` — a silently wrong key. The coordinator outputs this corrupted `CKDOutput` with no error, and honest parties accept it as the correct derived key.

This matches the allowed impact: **High — Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.**

### Likelihood Explanation

Any single participant in the CKD protocol can trigger this. The `ckd()` entry point accepts any `Participant` in the list as a valid caller: [5](#0-4) 

No special privilege is required beyond being a listed participant. The attack requires only sending a malformed message in round 1, which is a trivially reachable code path for any participant.

### Recommendation

Add a pairing-based proof of correctness for each participant's contribution. Since the coordinator knows each participant's public key share `vk_share_i = xi · G2` (from the DKG output), it can verify:

```
e(big_c/λi − app_pk · (big_y/λi), G2) == e(H(pk, app_id), vk_share_i)
```

This pairing check confirms that `big_c − app_pk · big_y = λi · xi · H(pk, app_id)` without revealing `xi`. Alternatively, require participants to attach a Schnorr-style DLEQ proof showing that `big_c/λi − app_pk · (big_y/λi)` and `vk_share_i` share the same discrete log with respect to `H(pk, app_id)` and `G2` respectively.

### Proof of Concept

1. Honest participants 1…N−1 run `compute_signature_share` correctly and send their `(norm_big_y, norm_big_c)` to the coordinator.
2. Malicious participant N sends `(ElementG1::identity(), ElementG1::identity())` — i.e., the additive identity — instead of their legitimate share.
3. The coordinator's loop at lines 50–55 adds these zero elements, silently dropping participant N's contribution.
4. The coordinator returns `CKDOutput::new(norm_big_y, norm_big_c)` where both fields are missing participant N's Lagrange-weighted secret key contribution.
5. The application calls `ckd_output.unmask(app_sk)` and obtains `(msk − λN·xN) · H(pk, app_id)` — a wrong key — with no error or warning. [6](#0-5) [7](#0-6)

async fn do_ckd_coordinator(
    mut chan: SharedChannel,
    participants: ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<CKDOutputOption, ProtocolError> {
    let (mut norm_big_y, mut norm_big_c) =
        compute_signature_share(&participants, me, key_pair, app_id, app_pk, rng)?;

    // Receive everyone's inputs and add them together
    let waitpoint = chan.next_waitpoint();

    for (_, participant_output) in
        recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
    {
        norm_big_y += participant_output.big_y();
        norm_big_c += participant_output.big_c();
    }
    let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
    Ok(Some(ckd_output))

pub fn ckd(
    participants: &[Participant],
    coordinator: Participant,
    me: Participant,
    key_pair: KeygenOutput,
    app_id: impl Into<AppId>,
    app_pk: PublicKey,
    rng: impl CryptoRngCore + Send + 'static,
) -> Result<impl Protocol<Output = CKDOutputOption>, InitializationError> {
    // not enough participants
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }

    // kick out duplicates
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }
