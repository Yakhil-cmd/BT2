### Title
Stale Presign Output Usage After Key Refresh Causes Denial of Signing — (`src/ecdsa/ot_based_ecdsa/presign.rs`, `src/ecdsa/robust_ecdsa/presign.rs`)

---

### Summary

Both the OT-based and Robust ECDSA presign protocols embed the participant's current private key share into the `PresignOutput` at presign time. After a key refresh (which rotates shares while preserving the public key), those stored values become cryptographically stale. The signing functions consume them without any binding check against the current key material, causing every signing attempt that uses a pre-refresh presign output to fail. The library provides no mechanism to detect or signal this staleness.

---

### Finding Description

**OT-based ECDSA — `src/ecdsa/ot_based_ecdsa/presign.rs`**

During `do_presign`, the participant's current private key share is extracted and baked into `sigma_i`:

```rust
let private_share = args.keygen_out.private_share.to_scalar();   // line 102
...
let sigma_i = alpha * private_share - (beta * a_i - c_i);        // line 179
```

`sigma_i` is stored verbatim in `PresignOutput.sigma`. [1](#0-0) [2](#0-1) 

During signing, `compute_signature_share` reads `presignature.sigma` directly — there is no check that it was produced with the same key share that is currently active:

```rust
let sigma_i = lambda * presignature.sigma;   // line 153
let r = x_coordinate(&presignature.big_r);
Ok(msg_hash * k_i + r * sigma_i)             // line 158
``` [3](#0-2) 

**Robust ECDSA — `src/ecdsa/robust_ecdsa/presign.rs`**

The same pattern appears: the key share is multiplied into `beta_me` and stored in `PresignOutput.beta`:

```rust
let x_me = args.keygen_out.private_share.to_scalar();   // line 322
let beta_me = c_me * x_me;                              // line 323
``` [4](#0-3) 

During signing, `compute_signature_share` uses `presignature.beta` without any freshness check:

```rust
let beta = presignature.beta * big_r_x_coordinate + presignature.e;   // line 179
let s_me = msg_hash * presignature.alpha + beta;                       // line 181
``` [5](#0-4) 

**The missing binding**

`PresignOutput` in neither variant carries any reference to the `KeygenOutput` (no public-key hash, no epoch/version tag). After `do_refresh` rotates every participant's share, the stored `sigma` / `beta` values are inconsistent with the new shares, yet the signing path has no way to detect this. The coordinator's final `sig.verify` call will fail, returning `ProtocolError::AssertionFailed("signature failed to verify")`, silently discarding all pre-computed presign material. [6](#0-5) [7](#0-6) 

---

### Impact Explanation

Any batch of presign outputs computed before a key refresh becomes entirely unusable after the refresh completes. Because the library emits no warning and the `PresignOutput` type carries no key-epoch identifier, honest parties operating under the documented trust model have no programmatic way to know their cached presign material is stale. Every signing attempt using that material will fail at the coordinator's verification step, producing a denial of signing for all honest parties until they discover the root cause and regenerate presign outputs from scratch.

This maps to: **High — Permanent denial of signing for honest parties under valid protocol inputs and documented trust assumptions.**

---

### Likelihood Explanation

**Medium.** Key refresh (`do_refresh` / `do_reshare`) is a first-class, documented feature of the library. The offline presign phase is explicitly designed to be run ahead of time and cached for later use — that is its entire purpose. Any deployment that (a) pre-computes presign batches for efficiency and (b) performs a key refresh between presign and sign will silently invalidate the entire presign cache. No adversarial action is required; a routine, legitimate key-hygiene operation is sufficient to trigger the failure.

---

### Recommendation

1. **Bind presign outputs to key material.** Add a `key_id: HashOutput` field to `PresignOutput` (e.g., `H(public_key || participant_set)`). Populate it during `do_presign` from `args.keygen_out.public_key`. In the signing entry-points (`sign` in both variants), verify that `presignature.key_id` matches a freshly computed hash of the supplied `public_key` before proceeding.

2. **Document the lifecycle invariant.** Explicitly state in the API documentation for `presign` that all `PresignOutput` values are invalidated by any subsequent `do_refresh` or `do_reshare` and must be discarded and regenerated.

---

### Proof of Concept

```
1. Run do_keygen  → each party holds (private_share_v1, public_key).
2. Run presign    → each party holds PresignOutput { sigma = f(private_share_v1), ... }.
3. Run do_refresh → each party now holds (private_share_v2, public_key)
                    (public key unchanged; shares rotated).
4. Run sign using the PresignOutput from step 2 and public_key from step 3.
   - compute_signature_share uses sigma = f(private_share_v1).
   - The reconstructed s is computed against private_share_v1 shares.
   - sig.verify(public_key, msg_hash) fails because the aggregate s
     is inconsistent with the current (v2) key material.
   - ProtocolError::AssertionFailed("signature failed to verify") is returned.
   - All pre-computed presign outputs are now permanently unusable.
```
