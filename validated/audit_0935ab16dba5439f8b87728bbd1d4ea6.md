### Title
Unvalidated Owner Transfer Allows Permanent Loss of Protocol Control - (File: `engine/src/contract_methods/admin.rs`)

### Summary
The `set_owner` function accepts any syntactically valid NEAR `AccountId` as the new owner without verifying the account actually exists on-chain. A single mistaken call by the current owner permanently locks all owner-gated protocol functions, with no recovery path.

### Finding Description
`set_owner` in `engine/src/contract_methods/admin.rs` performs only two checks before committing the new owner to state: (1) the engine is not paused, and (2) the new owner differs from the current owner. [1](#0-0) 

The `AccountId` type used for `args.new_owner` only validates string format (length 2–64, lowercase alphanumeric plus `-`, `_`, `.`). It performs no on-chain existence check. [2](#0-1) 

If the owner supplies a syntactically valid but non-existent NEAR account (e.g., a typo like `"aurora-near"` instead of `"aurora.near"`), the call succeeds and `state.owner_id` is permanently set to an uncontrollable account. [3](#0-2) 

The `require_owner_only` guard used by every privileged function then permanently rejects all callers. [4](#0-3) 

### Impact Explanation
Every critical owner-gated function becomes permanently inaccessible:

- `upgrade` / `stage_upgrade` — contract can never be patched
- `pause_contract` / `resume_contract` — a live exploit cannot be halted
- `resume_precompiles` — bridge precompiles (`EXIT_TO_NEAR`, `EXIT_TO_ETHEREUM`) paused by the authorizer can never be unpaused
- `set_key_manager` / `attach_full_access_key` — key management is frozen
- `factory_update` (XCC router) — cross-contract call infrastructure cannot be updated [5](#0-4) [6](#0-5) [7](#0-6) 

If a critical bug is discovered after ownership is lost, the engine cannot be paused or upgraded, resulting in **permanent freezing of all user funds** held in or bridged through the engine. This maps to the Critical impact tier: permanent freezing of funds.

### Likelihood Explanation
Likelihood is low (mirrors the reference report's score of 1). The current owner must make a mistake — a typo, a copy-paste error, or a misconfigured deployment script — when calling `set_owner`. However, the consequence is irreversible and catastrophic, and NEAR account IDs are short, human-readable strings where typos are plausible. There is no two-step confirmation or on-chain existence check to catch the error before it is committed.

### Recommendation
Implement a two-step ownership transfer pattern:
1. The current owner proposes a `pending_owner` in state.
2. The `pending_owner` account must call an `accept_ownership` function to confirm it controls the account.

This ensures the new owner account is live and controllable before the transfer is finalized, eliminating the risk of transferring to a non-existent or uncontrollable account.

Alternatively, at minimum, add a NEAR host-function call to verify the proposed account exists on-chain before committing the state change.

### Proof of Concept

**Setup**: Aurora Engine is deployed on NEAR. The current `owner_id` is `aurora.near`.

**Attack / Mistake Sequence**:

1. The owner intends to transfer control to `new-team.near` but mistypes the account as `new-teem.near` (a syntactically valid, non-existent NEAR account).

2. Owner calls `set_owner` with:
   ```
   SetOwnerArgs { new_owner: "new-teem.near" }
   ```

3. `set_owner` executes:
   - `require_running` passes (engine is running).
   - `require_owner_only` passes (caller is `aurora.near`, the current owner).
   - `AccountId::validate("new-teem.near")` passes — the string is syntactically valid.
   - `state.owner_id != args.new_owner` passes (different from current owner).
   - `state.owner_id` is set to `"new-teem.near"` and persisted. [8](#0-7) 

4. From this point, every call to `upgrade`, `pause_contract`, `resume_contract`, `resume_precompiles`, `stage_upgrade`, `set_key_manager`, `attach_full_access_key`, or `factory_update` fails with `ERR_NOT_ALLOWED` because `predecessor_account_id != "new-teem.near"` for any real caller. [9](#0-8) 

5. There is no recovery function callable by any other party. The engine is permanently unmanageable.

#[named]
pub fn set_owner<I: IO + Copy, E: Env>(io: I, env: &E) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        let mut state = state::get_state(&io)?;

        require_running(&state)?;
        require_owner_only(&state, &env.predecessor_account_id())?;

        let args: SetOwnerArgs = io.read_input_borsh()?;
        if state.owner_id == args.new_owner {
            return Err(errors::ERR_SAME_OWNER.into());
        }

        state.owner_id = args.new_owner;
        state::set_state(&mut io, &state)?;

        Ok(())
    })
}

    pub fn validate(account_id: &str) -> Result<(), ParseAccountError> {
        if account_id.len() < MIN_ACCOUNT_ID_LEN {
            Err(ParseAccountError::TooShort)
        } else if account_id.len() > MAX_ACCOUNT_ID_LEN {
            Err(ParseAccountError::TooLong)
        } else {
            // Adapted from https://github.com/near/near-sdk-rs/blob/fd7d4f82d0dfd15f824a1cf110e552e940ea9073/near-sdk/src/environment/env.rs#L819

            // NOTE: We don't want to use Regex here, because it requires extra time to compile it.
            // The valid account ID regex is /^(([a-z\d]+[-_])*[a-z\d]+\.)*([a-z\d]+[-_])*[a-z\d]+$/
            // Instead the implementation is based on the previous character checks.

            // We can safely assume that last char was a separator.
            let mut last_char_is_separator = true;

            for c in account_id.bytes() {
                let current_char_is_separator = match c {
                    b'a'..=b'z' | b'0'..=b'9' => false,
                    b'-' | b'_' | b'.' => true,
                    _ => {
                        return Err(ParseAccountError::Invalid);
                    }
                };
                if current_char_is_separator && last_char_is_separator {
                    return Err(ParseAccountError::Invalid);
                }
                last_char_is_separator = current_char_is_separator;
            }

            (!last_char_is_separator)
                .then_some(())
                .ok_or(ParseAccountError::Invalid)
        }
    }
