### Title
Pre-EIP-155 Legacy Transaction Cross-Chain Replay Bypasses Chain ID Guard — (`engine/src/engine.rs`, `engine-transactions/src/legacy.rs`)

### Summary

The Aurora Engine intentionally re-enabled acceptance of pre-EIP-155 legacy transactions (v=27 or v=28, `chain_id = None`) for EIP-1820 compatibility. The chain ID validation gate in `submit_with_alt_modexp` is guarded by `if let Some(chain_id) = transaction.chain_id`, which is never entered when `chain_id` is `None`. This means any pre-EIP-155 transaction signed on any EVM network (Ethereum mainnet, BSC, etc.) can be submitted to Aurora and will pass chain ID validation unconditionally, with only a nonce check remaining as a barrier.

### Finding Description

**Step 1 — Signature recovery (`engine-transactions/src/legacy.rs`)**

`LegacyEthSignedTransaction::sender()` handles v=27/28 by setting `chain_id = None` and computing `rec_id = v - 27`. It then calls `rlp_append_unsigned` with `chain_id = None`, which produces a 6-field RLP (no chain ID appended), and recovers the signer via `ecrecover`. This is correct per pre-EIP-155 semantics and succeeds for any valid pre-EIP-155 signature from any network. [1](#0-0) 

**Step 2 — Normalization (`engine-transactions/src/lib.rs`)**

`NormalizedEthTransaction::try_from(Legacy(tx))` sets `chain_id: tx.chain_id()`. For v=27/28, `chain_id()` returns `None`. [2](#0-1) 

**Step 3 — Chain ID guard is skipped (`engine/src/engine.rs`)**

The only chain ID enforcement in `submit_with_alt_modexp` is:

```rust
if let Some(chain_id) = transaction.chain_id
    && U256::from(chain_id) != U256::from_big_endian(&state.chain_id)
{
    return Err(EngineErrorKind::InvalidChainId.into());
}
```

When `transaction.chain_id` is `None`, the entire block is skipped. No other chain ID check exists. Execution proceeds to `check_nonce` and then full EVM execution. [3](#0-2) 

**Step 4 — Only nonce remains as a barrier**

After the chain ID check, `check_nonce` verifies the transaction nonce matches the sender's stored nonce on Aurora. This is the sole remaining guard. [4](#0-3) 

### Impact Explanation

If an attacker possesses a pre-EIP-155 signed transaction (v=27/28) from a victim that:
- transfers ETH to an attacker-controlled address, and
- has a nonce matching the victim's current Aurora nonce,

the attacker can submit it to Aurora's `submit()` endpoint and the engine will execute it, debiting the victim's Aurora ETH balance. The victim's funds are transferred without their consent on Aurora, constituting direct theft of user funds.

The EIP-1820 compatibility rationale (CHANGES.md line 586) confirms this is intentional behavior, not a parsing bug — making it a permanent, unfixed acceptance path. [5](#0-4) 

### Likelihood Explanation

**Constrained but non-zero.** The nonce must match the victim's Aurora nonce. This is the primary practical barrier. However:

- Fresh Aurora accounts (nonce=0) are common. Any victim who signed a pre-EIP-155 nonce=0 transaction on any other EVM network is vulnerable if they later fund an Aurora account.
- Pre-EIP-155 transactions were common before EIP-155 was widely adopted (pre-2016 Ethereum, some BSC/legacy tooling).
- The attacker only needs to observe the victim's historical on-chain transactions on other networks to find a matching nonce.

The attack is passive (no interaction with the victim required beyond observing their historical transactions) and requires no privileged access.

### Recommendation

Replace the conditional chain ID check with an explicit rejection of `chain_id = None` transactions that carry nonzero value or state-mutating data, unless the destination is specifically the EIP-1820 registry address. Alternatively, enforce that all accepted transactions must carry Aurora's chain ID in the signature (v ≥ 35), and handle EIP-1820 deployment as a special-cased privileged operation rather than a general pre-EIP-155 acceptance path.

### Proof of Concept

```rust
// 1. Take any real pre-EIP-155 Ethereum mainnet tx (v=27 or v=28) where:
//    - sender has ETH on Aurora
//    - sender's Aurora nonce == tx.nonce
//    - tx.to == attacker_address
//    - tx.value > 0
// 2. Decode it:
let tx = LegacyEthSignedTransaction::decode(&Rlp::new(&mainnet_tx_bytes)).unwrap();
assert_eq!(tx.chain_id(), None); // pre-EIP-155
assert!(tx.sender().is_ok());    // valid signature

// 3. Submit to Aurora runner (no chain_id in SubmitArgs):
let result = runner.call(SUBMIT, CALLER_ACCOUNT_ID, rlp::encode(&tx).to_vec());
// chain_id check is skipped (None), nonce check passes if nonces align
assert!(result.is_ok());

// 4. Verify victim balance decreased, attacker balance increased on Aurora
assert!(runner.get_balance(victim_address) < initial_victim_balance);
assert!(runner.get_balance(attacker_address) > initial_attacker_balance);
```

The `test_eth_signed_no_chain_sender` test in `legacy.rs` already confirms that `sender()` succeeds for a real v=28 transaction, validating step 2. [6](#0-5)
