### Title
Blocklist Enforcement Checks EVM Tx Destination But Not Event-Level Recipient in `__CronosSendToAccount` Hook — (`app/block_address.go`, `app/proposal.go`)

### Summary
The Cronos blocklist enforcement checks the EVM transaction's direct `To()` address at the mempool and proposal level, but does not inspect the `recipient` field embedded in EVM log events (specifically `__CronosSendToAccount`) that are processed by the EVM hook at end-of-block. An unprivileged, non-blocked address can route native CRO tokens to a blocklisted address by calling a contract that emits `__CronosSendToAccount(blockedAddress, amount)`, bypassing the blocklist entirely.

### Finding Description
Cronos enforces its blocklist at two points:

1. **`BlockAddressesDecorator.AnteHandle`** (`app/block_address.go:32-83`) — runs at `CheckTx` time and checks (a) tx signers, (b) the EVM tx `To()` address, and (c) EIP-7702 authorization addresses.
2. **`ProposalHandler.ValidateTransaction`** (`app/proposal.go:262-336`) — runs at `ProcessProposal` time and performs the same three checks. [1](#0-0) [2](#0-1) 

Neither enforcement point inspects the `recipient` argument carried inside EVM log data. The `__CronosSendToAccount` event — defined in ADR-004 and emitted by any contract — instructs the EVM hook to transfer a native token amount from the contract account to an arbitrary `recipient` address at end-of-block. Because the EVM hook runs during `FinalizeBlock` (after all mempool/proposal checks have already passed), the blocklist is never consulted for the event-level recipient. [3](#0-2) 

The structural gap is identical to the external NFT report: the checked entity (EVM tx `To()` = the contract) is not the actual beneficiary (`recipient` in the event). The CHANGELOG entry for v1.6.0 confirms the destination-address check was a deliberate addition to close a prior gap for direct EVM transfers, but the event-recipient path was not covered: [4](#0-3) 

### Impact Explanation
A blocked address can receive native CRO (or any token handled by `__CronosSendToAccount`) from an intermediary contract, defeating the blocklist's purpose of preventing sanctioned/blocked addresses from accumulating assets. This is a **High** impact: bypass of the Cronos block-list authorization check, allowing unauthorized balance changes for CRO to a blocklisted address.

### Likelihood Explanation
Any unprivileged, non-blocked address can deploy or call a contract that emits `__CronosSendToAccount`. No special permissions, leaked keys, or admin access are required. The attack is fully on-chain and repeatable.

### Recommendation
Extend the EVM hook handler for `__CronosSendToAccount` to check the `recipient` address against the active blocklist before executing the native token transfer. The same check should be applied to any other event-driven transfer path (e.g., `__CronosSendToIbc` if the recipient maps to a Cronos address). The blocklist lookup should mirror the pattern already used in the bank precompile: [5](#0-4) 

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

contract BlocklistBypass {
    // Emitting this event causes the Cronos EVM hook to transfer
    // `amount` of native CRO from this contract to `recipient`
    // at end-of-block, with no blocklist check on `recipient`.
    event __CronosSendToAccount(address recipient, uint256 amount);

    function sendToBlocked(address blockedAddr) external payable {
        // EVM tx To() = this contract address (not blocked) → passes
        // BlockAddressesDecorator and ProposalHandler checks.
        // The actual beneficiary `blockedAddr` is never checked.
        emit __CronosSendToAccount(blockedAddr, msg.value);
    }
}
```

**Attack steps:**
1. Attacker (non-blocked) deploys `BlocklistBypass`.
2. Attacker calls `sendToBlocked{value: X}(blockedAddress)`.
3. `BlockAddressesDecorator` and `ProposalHandler.ValidateTransaction` check `ethTx.To()` = contract address → not blocked → transaction is accepted into a block.
4. At end-of-block, the EVM hook processes the `__CronosSendToAccount` log and transfers `X` CRO to `blockedAddress`.
5. `blockedAddress` receives CRO despite being on the blocklist. [6](#0-5) [7](#0-6)

func (h *ProposalHandler) ValidateTransaction(tx sdk.Tx, txBz []byte) error {
	if len(h.blocklist) == 0 {
		// fast path, accept all txs
		return nil
	}

	var err error
	if tx == nil {
		tx, err = h.TxDecoder(txBz)
		if err != nil {
			return err
		}
	}

	sigTx, ok := tx.(signing.SigVerifiableTx)
	if !ok {
		return fmt.Errorf("tx of type %T does not implement SigVerifiableTx", tx)
	}

	signers, err := sigTx.GetSigners()
	if err != nil {
		return err
	}
	for _, signer := range signers {
		encoded, err := h.addressCodec.BytesToString(signer)
		if err != nil {
			return fmt.Errorf("invalid bech32 address: %s, err: %w", signer, err)
		}
		if _, ok := h.blocklist[encoded]; ok {
			return fmt.Errorf("signer is blocked: %s", encoded)
		}
	}
