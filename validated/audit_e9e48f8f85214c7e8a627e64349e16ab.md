### Title
NFT Seller Shown Incorrect "Net Proceeds" and "You Will Receive" Amount Due to Commented-Out Royalty Deduction — (`packages/gui/src/components/offers/utils.ts`)

---

### Summary

In `calculateNFTRoyalties`, the deduction of royalties from `nftSellerNetAmount` is commented out, leaving it equal to the full offer amount. This incorrect value propagates to both the "You will receive" summary row and the explicit "Net Proceeds" label in `NFTOfferDetails`, causing an NFT seller reviewing an imported `TokenForNFT` offer to be shown they will receive the full offer price when they will actually receive less after royalty deduction.

---

### Finding Description

In `packages/gui/src/components/offers/utils.ts`, `calculateNFTRoyalties` computes `nftSellerNetAmount` as:

```typescript
const nftSellerNetAmount: number = amount;
// : parseFloat(
//     (amount - parseFloat(royaltyAmountString) - makerFee).toFixed(12),
//   );
``` [1](#0-0) 

The correct deduction (`amount - royalty - makerFee`) is commented out. `nftSellerNetAmount` always equals `amount` regardless of royalty percentage.

This value flows into `NFTOfferDetails` in `NFTOfferViewer.tsx`:

```typescript
const overrideNFTSellerAmount =
    exchangeType === NFTOfferExchangeType.TokenForNFT
      ? assetType === OfferAsset.CHIA
        ? chiaToMojo(nftSaleInfo?.nftSellerNetAmount ?? 0)  // = chiaToMojo(amount), not net
        : catToMojo(nftSaleInfo?.nftSellerNetAmount ?? 0)
      : undefined;
``` [2](#0-1) 

This `overrideNFTSellerAmount` (full amount in mojo) is passed to `NFTOfferSummary` → `NFTOfferSummaryRow` → `OfferSummaryTokenRow`, which uses it directly as the displayed amount:

```typescript
const amount = overrideNFTSellerAmount ?? originalAmount;
``` [3](#0-2) 

Additionally, the dedicated "Net Proceeds" section — whose tooltip explicitly states it shows "the asking price, minus any associated creator fees" — also renders `nftSaleInfo?.nftSellerNetAmount`, which equals the full amount: [4](#0-3) 

---

### Impact Explanation

An NFT seller (taker) who imports a `TokenForNFT` offer for an NFT with a non-zero `royaltyPercentage` will see:

- **"You will receive: 100 XCH"** — should be 90 XCH for a 10% royalty
- **"Net Proceeds: 100 XCH"** — label promises post-royalty amount, shows pre-royalty amount

The "Creator Fee" line is correctly computed and displayed separately, but the two most prominent amount fields ("You will receive" and "Net Proceeds") both show the wrong value. The "Net Proceeds" label is specifically designed to communicate the post-royalty amount; showing the full amount there directly contradicts its stated semantics and is the most likely figure a seller relies on before clicking Accept.

This causes the NFT seller to accept an offer believing they will receive more XCH/CAT than they actually will, resulting in direct financial loss proportional to the royalty percentage and offer size.

---

### Likelihood Explanation

- Triggered by any imported `TokenForNFT` offer for any NFT with `royaltyPercentage > 0` — a common, standard scenario.
- The NFT seller owns the NFT, so `useNFT(launcherId)` resolves it from their wallet, satisfying the `nft !== undefined` guard that enables `nftSaleInfo` computation.
- No special attacker action is required beyond crafting a standard offer file; the bug fires unconditionally for all such offers.
- The attacker entry point is the standard offer import flow (`importOffer` → `NFTOfferViewer` → `NFTOfferDetails`). [5](#0-4) 

---

### Recommendation

Uncomment and restore the royalty deduction in `calculateNFTRoyalties`:

```typescript
const nftSellerNetAmount: number = parseFloat(
  (amount - parseFloat(royaltyAmountString) - makerFee).toFixed(12),
);
``` [1](#0-0) 

Also verify the `totalAmount` formula for `TokenForNFT` — currently it adds royalty on top of `amount` (`amount + makerFee + royaltyAmount`), but per the UI tooltip, royalties are paid from the offered amount, not added on top. [6](#0-5) 

---

### Proof of Concept

Unit test for `calculateNFTRoyalties` with a 10% royalty and 100 XCH offer:

```typescript
const result = calculateNFTRoyalties(
  100,       // amount: 100 XCH
  0,         // makerFee
  10,        // royaltyPercentage: 10%
  NFTOfferExchangeType.TokenForNFT,
);

// Currently fails — nftSellerNetAmount is 100, not 90
expect(result.nftSellerNetAmount).toBe(90);
```

In the UI: import any offer file for a `TokenForNFT` trade on an NFT with `royaltyPercentage = 1000` (10%). The "Net Proceeds" field and "You will receive" row will both display the full offer amount instead of 90% of it.
