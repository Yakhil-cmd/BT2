### Title
Missing wNEAR Storage Registration for Existing Router Sub-Accounts After `factory_set_wnear_address` - (File: `engine/src/contract_methods/xcc.rs`)

---

### Summary

`factory_set_wnear_address` updates the engine's stored wNEAR EVM address but performs no corresponding re-registration of existing XCC router sub-accounts with the new wNEAR NEP-141 contract. Because each router's storage deposit was made against the **old** wNEAR account at initialization time, any subsequent XCC call that requires NEAR payment will attempt an `ft_transfer` to an unregistered recipient on the new wNEAR contract, causing the NEAR-level callback to fail and leaving the user's wNEAR locked in the engine's implicit EVM address.

---

### Finding Description

When a new XCC router sub-account is first deployed, `handle_precompile_promise` (or `fund_xcc_sub_account`) calls the router's `initialize` with `must_register: true`, which issues a `storage_deposit` promise to the wNEAR NEP-141 contract so the router can receive wNEAR tokens: [1](#0-0) 

The wNEAR NEP-141 account used for that deposit is resolved at deploy time from the engine's current wNEAR EVM address: [2](#0-1) 

The owner can later change the wNEAR EVM address via `factory_set_wnear_address`: [3](#0-2) 

The underlying setter only writes the new address to storage and does nothing else: [4](#0-3) 

After the change, every XCC call that requires NEAR payment reaches `withdraw_wnear_to_router`, which fetches the **new** wNEAR EVM address and calls `withdrawToNear` on it: [5](#0-4) 

`withdrawToNear` on the new wNEAR ERC-20 contract issues an `ft_transfer` on the **new** NEP-141 wNEAR contract targeting the router sub-account. Because the router was never registered (via `storage_deposit`) with the new NEP-141 contract, the `ft_transfer` fails.

Critically, the EVM-level `transferFrom` that moved the user's wNEAR to the engine's implicit address already executed and committed **before** this NEAR-level callback runs: [6](#0-5) 

The NEAR callback failure does not revert the EVM state, so the user's wNEAR remains locked in the engine's implicit EVM address with no direct recovery path for the user.

Additionally, if the new wNEAR EVM address has not yet been registered in the NEP-141↔ERC-20 map, `handle_precompile_promise` will unconditionally panic when trying to deploy any new router: [7](#0-6) 

---

### Impact Explanation

- **Temporary/permanent fund freeze**: User wNEAR transferred to the engine's implicit address during the XCC precompile execution is stranded when the NEAR-level `withdraw_wnear_to_router` callback fails because the router is unregistered with the new wNEAR contract.
- **DoS of all NEAR-attached XCC calls**: Every existing router sub-account becomes unable to receive wNEAR from the new contract, blocking all cross-contract calls that require NEAR payment.

Impact class: **High — Temporary (potentially permanent) freezing of user funds.**

---

### Likelihood Explanation

`factory_set_wnear_address` is an owner-only function intended for legitimate operational use (e.g., upgrading or replacing the wNEAR contract). The bug is triggered by the normal, intended use of this setter — no malicious intent is required. Any wNEAR contract upgrade or migration on Aurora mainnet would trigger this path. Likelihood is **medium**: the operation is infrequent but is a documented, expected admin action.

---

### Recommendation

Mirror the pattern used during initial router deployment: when `factory_set_wnear_address` is called, emit a NEAR promise (or batch of promises) that calls `storage_deposit` on the new wNEAR NEP-141 contract for every known router sub-account, or at minimum re-initialize each router via `deploy_upgrade` / `initialize` with `must_register: true` and the new `wnear_account`. At a minimum, document that callers must manually re-register all existing routers before the new wNEAR address becomes active, and consider adding a two-phase commit (set-pending / activate) to enforce this ordering.

---

### Proof of Concept

1. Deploy Aurora Engine; set wNEAR address to `wnear_v1` via `factory_set_wnear_address`.
2. User A calls the XCC precompile with 1 NEAR attached. A router sub-account is created and registered with `wnear_v1`.
3. Owner calls `factory_set_wnear_address` with `wnear_v2` (a new wNEAR ERC-20/NEP-141 pair).
4. User A calls the XCC precompile again with 1 NEAR attached.
   - EVM executes: `transferFrom(userA, engineImplicit, amount)` on `wnear_v2` — succeeds; user's wNEAR is now in the engine's implicit address.
   - NEAR callback `withdraw_wnear_to_router` fires: calls `withdrawToNear` on `wnear_v2`, which calls `ft_transfer` to the router on the `wnear_v2` NEP-141 contract.
   - The router has no `storage_deposit` on `wnear_v2` → `ft_transfer` panics/fails.
   - NEAR callback fails; EVM state is **not** rolled back.
5. User A's wNEAR is permanently locked in the engine's implicit EVM address.

#[named]
pub fn factory_set_wnear_address<I: IO + Copy, E: Env>(
    io: I,
    env: &E,
) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        require_owner_only(&state, &env.predecessor_account_id())?;
        let address = io.read_input_arr20()?;
        xcc::set_wnear_address(&mut io, &Address::from_array(address));
        Ok(())
    })
}
