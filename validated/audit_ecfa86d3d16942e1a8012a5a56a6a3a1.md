### Title
ERC-20 Tokens Permanently Burned on Failed `ExitToNear` Promise Without Error Refund — (`engine-precompiles/src/native.rs`, `etc/eth-contracts/contracts/EvmErc20.sol`)

### Summary

The `ExitToNear` precompile burns ERC-20 tokens before scheduling the outbound `ft_transfer` promise. When the `error_refund` feature is not compiled in (the non-default build), no callback is registered and no refund is issued if the promise fails. The burned ERC-20 tokens are permanently unrecoverable — a direct analog to the original report's missing ERC-721 recovery path.

### Finding Description

`EvmErc20.withdrawToNear` calls `_burn` on the caller's balance **before** invoking the `ExitToNear` precompile: [1](#0-0) 

The precompile then constructs callback args. When the `error_refund` feature is absent, `refund` is hardcoded to `None`: [2](#0-1) 

Because `ExitToNearPrecompileCallbackArgs { refund: None, transfer_near: None }` equals `default()`, the promise is scheduled as a bare `PromiseArgs::Create` with **no callback at all** for the standard ERC-20 (non-wNEAR) exit path: [3](#0-2) 

If the `ft_transfer` promise to the NEP-141 contract fails (e.g., the recipient account is not registered with the NEP-141 contract), `exit_to_near_precompile_callback` is never invoked. Even when it is invoked, without `error_refund` the refund branch is unreachable: [4](#0-3) 

The code itself documents this risk explicitly: [5](#0-4) 

The same structural gap exists in `ExitToEthereum`, which schedules only a bare `PromiseArgs::Create` with no callback under any build configuration: [6](#0-5) 

### Impact Explanation

**Permanent freezing of funds.** The ERC-20 burn is irreversible within the EVM. If the outbound NEP-141 `ft_transfer` fails, the user's ERC-20 tokens are destroyed and the corresponding NEP-141 tokens remain locked inside the Aurora contract with no mechanism to re-mint or recover the ERC-20 side. The test suite confirms this behavior explicitly: [7](#0-6) 

### Likelihood Explanation

Any EVM user can trigger this by calling `withdrawToNear` with a NEAR recipient account that is not registered with the target NEP-141 contract. NEAR's NEP-141 standard requires storage registration before receiving tokens; `ft_transfer` to an unregistered account panics and the promise fails. This is a realistic user-reachable condition requiring no admin involvement. The `error_refund` feature is opt-in (non-default in Rust), so production builds that omit it are fully exposed.

### Recommendation

1. Make the `error_refund` feature the **default** for production builds, or unconditionally include the refund callback for ERC-20 exits.
2. Apply the same callback pattern to `ExitToEthereum` — it currently has no error recovery path under any build configuration.
3. Consider reversing the operation order: attempt the NEP-141 transfer first and only burn ERC-20 tokens in the success callback, eliminating the burn-before-transfer race entirely.

### Proof of Concept

1. Deploy a NEP-141 token and bridge it to Aurora (creating an `EvmErc20` mirror).
2. Fund an EVM address with the bridged ERC-20 tokens.
3. Call `EvmErc20.withdrawToNear("unregistered.near", amount)` from that address.
4. `_burn` executes immediately, reducing the ERC-20 balance.
5. The `ft_transfer` promise to the NEP-141 contract fails because `unregistered.near` has no storage deposit.
6. Without `error_refund`, no callback fires and no ERC-20 tokens are re-minted.
7. The user's ERC-20 balance is zero; the NEP-141 tokens remain in the Aurora contract; the funds are permanently frozen.
