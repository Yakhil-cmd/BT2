### Title
`protocolUnpausePool` Passes Wrong Pause Level to `setPause()`, Leaving Pool Permanently Swap-Blocked After Protocol Unpause — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary
`protocolUnpausePool` calls `setPause(1)` (admin-paused) instead of `setPause(0)` (active). After the protocol owner invokes this function in good faith to restore a previously-active pool, the pool remains paused and all swaps continue to revert.

### Finding Description
The pause-level semantics are:

| Level | Meaning |
|---|---|
| 0 | Active |
| 1 | Paused by admin |
| 2 | Paused by protocol |

`protocolPausePool` can freeze a pool that is at level 0 **or** level 1. When the protocol later calls `protocolUnpausePool`, the intent is to restore the pool to a usable state. However, the implementation hard-codes the target level as **1** (admin-paused) rather than **0** (active):

```solidity
// MetricOmmPoolFactory.sol  lines 399-403
function protocolUnpausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 2) revert InvalidPauseTransition(cur, 1);
    IMetricOmmPoolFactoryActions(pool).setPause(1);   // ← wrong value: should be 0
}
``` [1](#0-0) 

The `whenNotPaused` guard in the pool rejects any non-zero pause level:

```solidity
// MetricOmmPool.sol  line 644
function _checkNotPaused() internal view {
    if (pauseLevel != 0) revert PoolPaused();
}
``` [2](#0-1) 

`swap` is gated by `whenNotPaused`:

```solidity
// MetricOmmPool.sol  line 224
function swap(...) external whenNotPaused nonReentrant(PoolActions.SWAP) ...
``` [3](#0-2) 

So after `protocolUnpausePool` executes, `pauseLevel == 1 != 0`, and every swap call still reverts with `PoolPaused`.

The only recovery path is for the pool admin to separately call `unpausePool`, which checks `cur == 1` and sets to 0:

```solidity
// MetricOmmPoolFactory.sol  lines 467-471
function unpausePool(address pool) external override nonReentrant onlyPoolAdmin(pool) {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 1) revert InvalidPauseTransition(cur, 0);
    IMetricOmmPoolFactoryActions(pool).setPause(0);
}
``` [4](#0-3) 

This is the exact wrong-value pattern from the external report: a value of one unit (admin-pause level `1`) is passed where a different value (active level `0`) is required, silently corrupting the pool's operational state.

### Impact Explanation
Every swap on the pool continues to revert after the protocol owner believes they have restored it. LPs cannot trade, arbitrageurs cannot rebalance, and the pool is effectively insolvent from a utility standpoint until the admin independently discovers the issue and calls `unpausePool`. If the admin is unresponsive or unaware, the pool remains permanently swap-blocked. This satisfies the "unusable swap flow" criterion.

### Likelihood Explanation
The scenario is straightforward: protocol owner pauses an active pool (level 0 → 2) during an emergency, then calls `protocolUnpausePool` once the emergency is resolved. The function silently leaves the pool at level 1. No adversarial action is required; the bug fires on every good-faith invocation of `protocolUnpausePool` when the pool was active before the protocol pause.

### Recommendation
```diff
function protocolUnpausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
-   if (cur != 2) revert InvalidPauseTransition(cur, 1);
-   IMetricOmmPoolFactoryActions(pool).setPause(1);
+   if (cur != 2) revert InvalidPauseTransition(cur, 0);
+   IMetricOmmPoolFactoryActions(pool).setPause(0);
}
```

### Proof of Concept
1. Pool is active (`pauseLevel == 0`).
2. Protocol owner calls `protocolPausePool(pool)` → `pauseLevel == 2`.
3. Protocol owner calls `protocolUnpausePool(pool)` → `setPause(1)` → `pauseLevel == 1`.
4. Any user calls `pool.swap(...)` → `_checkNotPaused()` → `pauseLevel != 0` → `revert PoolPaused()`.
5. Pool is permanently swap-blocked until the admin separately calls `unpausePool`.
