### Title
Unprivileged Caller Can Suppress All Consensus Block Rewards via `update_rewards` — (File: `src/staking/staking.cairo`)

---

### Summary

`StakingRewardsManagerImpl::update_rewards` has no caller access control and accepts a caller-supplied `disable_rewards: bool` flag. Because `last_reward_block` is a **single global** storage slot, any unprivileged address can call `update_rewards(any_active_staker, disable_rewards: true)` once per block to consume the block's sole reward slot without distributing rewards, permanently preventing all stakers from receiving consensus block rewards.

---

### Finding Description

`update_rewards` is the entry point through which a block proposer claims per-block consensus rewards in V3: [1](#0-0) 

The function's only gate is `general_prerequisites()`, which checks only that the contract is unpaused and the caller is non-zero: [2](#0-1) 

After validating that the staker exists and is active, the function unconditionally writes the current block number to the **global** `last_reward_block`: [3](#0-2) 

If `disable_rewards` is `true`, the function returns immediately without distributing any rewards. Because `last_reward_block` is now equal to the current block, every subsequent call in the same block fails with `REWARDS_ALREADY_UPDATED`: [4](#0-3) 

The storage declaration confirms `last_reward_block` is a single global slot, not per-staker: [5](#0-4) 

Contrast this with the pre-consensus path `update_rewards_from_attestation_contract`, which correctly enforces `assert_caller_is_attestation_contract()`: [6](#0-5) 

No equivalent guard exists on `update_rewards`.

---

### Impact Explanation

An attacker who calls `update_rewards(any_active_staker, disable_rewards: true)` at the start of every block:

1. Marks the block as "already rewarded" (`last_reward_block = current_block`).
2. Distributes zero rewards (early return due to `disable_rewards: true`).
3. Causes every legitimate block-proposer call in that block to revert.

All stakers receive **zero** consensus block rewards for every block the attack is sustained. This constitutes **permanent freezing of unclaimed yield** (High impact) as long as the attacker continues, with no mechanism for stakers to recover the lost rewards.

---

### Likelihood Explanation

- Requires no special role, leaked key, or privileged access.
- Any address with enough gas can execute it.
- The attacker needs only one cheap transaction per block.
- Active staker addresses are publicly visible on-chain, so a valid `staker_address` argument is trivially obtained.
- The attack is economically rational for a competitor wishing to suppress staking yields.

---

### Recommendation

Add an access-control check to `update_rewards` so that only the staker themselves (or a designated consensus-layer contract) can invoke it, mirroring the pattern already used in `update_rewards_from_attestation_contract`:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
+   let caller = get_caller_address();
+   assert!(
+       caller == staker_address || caller == self.internal_staker_info(:staker_address).operational_address,
+       "{}",
+       Error::UNAUTHORIZED_CALLER,
+   );
    ...
}
```

Alternatively, remove `disable_rewards` from the public interface entirely and handle reward suppression through a separate privileged function callable only by the consensus layer.

---

### Proof of Concept

```
Block N:
  1. Attacker calls: staking.update_rewards(alice_staker, disable_rewards=true)
     → last_reward_block := N
     → no rewards distributed (early return)

  2. Alice (block proposer) calls: staking.update_rewards(alice_staker, disable_rewards=false)
     → assert!(N > N) FAILS → REWARDS_ALREADY_UPDATED

Block N+1:
  3. Attacker repeats step 1 with any active staker.

Result: All stakers receive 0 consensus block rewards indefinitely.
```
