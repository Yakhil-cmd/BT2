### Title
`#[payable]` `execute_scheduled` Permanently Locks Any Attached NEAR in the XCC Router Contract - (File: `etc/xcc-router/src/lib.rs`)

---

### Summary

The `execute_scheduled` function in the XCC Router contract is marked `#[payable]`, allowing any caller to attach NEAR tokens to the call. The function never reads, forwards, or refunds the attached deposit, and the Router contract has no general NEAR withdrawal mechanism. Any NEAR attached to a call to `execute_scheduled` is permanently locked in the Router contract.

---

### Finding Description

In `etc/xcc-router/src/lib.rs`, the `execute_scheduled` function is annotated with `#[payable]`: [1](#0-0) 

In NEAR SDK, `#[payable]` means the function will not panic when NEAR tokens are attached to the call. Without this attribute, attaching NEAR would cause an automatic panic. By marking the function `#[payable]`, the contract explicitly opts in to receiving NEAR — but the function body never calls `env::attached_deposit()`, never forwards the deposit to a downstream promise, and never refunds it to the caller.

The contract's only outbound NEAR transfer is `send_refund`, which transfers a hardcoded constant `REFUND_AMOUNT` of 2 NEAR to the parent account: [2](#0-1) 

This is a fixed-amount operational refund, not a general withdrawal mechanism. There is no function in the Router contract that can recover arbitrary NEAR balances.

The `deploy_upgrade` function is also marked `#[payable]` with the same defect — it passes `NearToken::default()` (zero) to the downstream `initialize` call, silently discarding any attached deposit: [3](#0-2) 

However, `deploy_upgrade` is gated by `assert_preconditions` (caller must be the parent Aurora Engine), so it is not an unprivileged entry point. `execute_scheduled` is the critical path.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Any NEAR tokens attached to a call to `execute_scheduled` are irrecoverably locked inside the Router contract. The Router has no sweep, rescue, or general withdrawal function. The only exit for NEAR is the hardcoded `send_refund` (2 NEAR to parent), which does not account for arbitrary locked balances. Locked NEAR cannot be recovered by the user, the parent engine, or any admin.

---

### Likelihood Explanation

`execute_scheduled` is explicitly documented as callable by anyone: [4](#0-3) 

Because the function is public and permissionless, any EVM user or NEAR account interacting with the XCC Router can trigger this path. A user who attaches NEAR (e.g., expecting the function to forward it as part of a scheduled promise, or by mistake) will permanently lose those funds. The `#[payable]` annotation provides no warning or refund guard, and the NEAR SDK will silently accept the deposit.

---

### Recommendation

Remove the `#[payable]` attribute from `execute_scheduled` (and from `deploy_upgrade` if the parent never intentionally attaches NEAR to that call). Removing `#[payable]` causes the NEAR runtime to automatically reject and refund any attached deposit, preventing accidental or intentional locking. If attached NEAR is ever legitimately needed by either function in the future, a corresponding forwarding or refund path must be added.

---

### Proof of Concept

1. A NEAR account calls `execute_scheduled` on a deployed XCC Router contract, attaching any nonzero amount of NEAR (e.g., 1 NEAR).
2. The NEAR runtime accepts the deposit because the function is `#[payable]`.
3. The function body executes: it removes the scheduled promise and creates it via `promise_create`. No reference to `env::attached_deposit()` appears anywhere in the function.
4. The 1 NEAR is now held by the Router contract account.
5. No function in the Router contract can transfer this 1 NEAR out. `send_refund` transfers exactly `REFUND_AMOUNT` (2 NEAR) to the parent — it does not drain the full balance and is restricted to the parent caller.
6. The attached NEAR is permanently frozen. [1](#0-0) [5](#0-4)

    #[payable]
    pub fn execute_scheduled(&mut self, nonce: U64) {
        let Some(promise) = self.scheduled_promises.remove(&nonce.0) else {
            env::panic_str("ERR_PROMISE_NOT_FOUND")
        };
        let promise_id = Self::promise_create(promise);
        env::promise_return(promise_id);
    }
