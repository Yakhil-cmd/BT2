### Title
WalletConnect Dapp-Controlled `royalty_percentage` in `driver_dict` Spoofs "Total Amount with Royalties" in Offer Confirmation Dialog — (`packages/gui/src/electron/commands/parseCommandDisplay.ts`)

---

### Summary

When a WalletConnect dapp calls `chia_createOfferForIds` or `chia_wallet.take_offer`, the `royalty_percentage` used to compute the "Total Amount with Royalties" shown in the user-facing confirmation dialog is taken from the dapp-supplied `driver_dict` (or the offer file's embedded `infos`) without cross-validation against on-chain NFT data whenever the `nftGetInfo` call fails. A malicious dapp can set `royalty_percentage: '0'` to suppress the royalty line entirely, causing the user to approve an offer while unaware of the additional royalty cost that the Chia blockchain will enforce on execution.

---

### Finding Description

**Root cause — `createOfferRoyaltyPercentages` trusts dapp input directly:**

In `parseCommandDisplay.ts`, the `create_offer_for_ids` branch reads royalty percentages exclusively from the dapp-provided `driver_dict`:

```typescript
function createOfferRoyaltyPercentages(
  walletDelta: WalletDelta,
  driverDict: Record<string, unknown>,
): AssetRoyaltyPercentages {
  for (const assetId of Object.keys(walletDelta.receiving)) {
    royaltyPercentages.receiving[assetId] = royaltyPercentageForDriverInfo(
      driverDict[assetId] ?? driverDict[`0x${assetId}`],
    );
  }
  ...
}
``` [1](#0-0) 

**Partial on-chain override — only when `nftGetInfo` succeeds AND returns `data_uris`:**

`parseWalletDeltaItem` does attempt to override the dapp-supplied value by calling `nftGetInfo`, but the override is conditional:

```typescript
if (nftInfo && nftInfo.success && nftInfo.nft_info && nftInfo.nft_info.data_uris) {
  if ('royalty_percentage' in nftInfo.nft_info) {
    result.royaltyPercentage = parseRoyaltyPercentage(nftInfo.nft_info.royalty_percentage);
  }
}
``` [2](#0-1) 

If `nftGetInfo` returns `success: false` (e.g., the NFT is not yet tracked by the user's wallet — a common case when the user is *buying* an NFT they do not yet own), the dapp-supplied `royalty_percentage` is used unchanged. The project's own test explicitly documents this fallback:

```typescript
mockNftGetInfo.mockResolvedValue({ success: false });
// driver_dict royalty_percentage: '250' is used as-is
``` [3](#0-2) 

**`amountWithRoyalties` calculation uses the (possibly spoofed) value:**

`royaltyPercentagesForSide` collects royalty percentages from the NFT line items and `formatAmountWithRoyalties` adds them to the fungible spending amount:

```typescript
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
const totalAmount = amount + royaltyAmount;
``` [4](#0-3) 

If the dapp supplies `royalty_percentage: '0'`, `royaltyPercentagesForSide` filters it out (the filter requires `> 0`), `amountWithRoyalties` is never set, and the confirmation dialog shows only the base amount with no royalty disclosure. [5](#0-4) 

**Confirmation dialog renders the spoofed total:**

The `Confirm.tsx` dialog renders `line.amountWithRoyalties` as "Total Amount with Royalties" directly from the computed display data:

```tsx
{line.amountWithRoyalties && (
  <div className="text-xs text-chia-text-secondary">
    {i18n._(/* i18n */ { id: 'Total Amount with Royalties' })}: {line.amountWithRoyalties} ...
  </div>
)}
``` [6](#0-5) 

---

### Impact Explanation

For `chia_wallet.take_offer`: the buyer pays the offer's base XCH amount **plus** the on-chain royalty enforced by the NFT's puzzle hash. If the dapp embeds `royalty_percentage: 0` in the offer file's driver dict and `nftGetInfo` fails (NFT not yet in the user's wallet), the confirmation dialog shows no royalty line. The user approves believing they pay 1 XCH; the wallet daemon debits 1 XCH + royalty (e.g., 1.025 XCH). The user is charged more than they consented to.

For `chia_createOfferForIds`: the dapp directly controls `driver_dict`. With `royalty_percentage: 0` and a failing `nftGetInfo`, the user creates a buy offer without seeing the royalty they will owe when the offer is taken.

This matches the allowed High impact: **"Corruption, spoofing, or unsafe trust of … WalletConnect state that causes a user to approve … the wrong … amount."**

---

### Likelihood Explanation

- Requires a malicious WalletConnect dapp the user has already paired with.
- `nftGetInfo` failing for an NFT the user does not yet own is the **normal case** when buying an NFT via `create_offer_for_ids` — the test suite itself mocks `success: false` for this scenario.
- No special privileges, leaked keys, or cryptographic breaks are needed.

---

### Recommendation

In `parseWalletDeltaItem`, do not fall back to the dapp-supplied `royaltyPercentage` when `nftGetInfo` fails. Instead, either:

1. **Treat missing on-chain data as unknown royalty** — omit `amountWithRoyalties` and display a warning that royalty information could not be verified, or
2. **Always call `nftGetInfo` and block confirmation** if the call fails for an NFT asset, preventing approval without verified royalty data.

Additionally, for `take_offer`, `offerSummaryRoyaltyPercentages` should not be used as the initial value passed to `parseWalletDeltaItem`; the offer file's embedded `royalty_percentage` is attacker-controlled and should only serve as a display hint after on-chain verification succeeds.

---

### Proof of Concept

1. Attacker operates a WalletConnect dapp paired with the victim's Chia GUI.
2. Attacker constructs an offer file for an NFT with a real on-chain royalty of 5% (500 basis points), but embeds `royalty_percentage: '0'` in the offer's driver dict `infos`.
3. Attacker calls `chia_wallet.take_offer` via WalletConnect with this offer string.
4. GUI calls `getOfferSummary` → returns `infos[nftId].transfer_program.royalty_percentage = '0'`.
5. `offerSummaryRoyaltyPercentages` sets `royaltyPercentages.receiving[nftId] = 0`.
6. `parseWalletDeltaItem` calls `nftGetInfo(nftId)` — returns `success: false` (NFT not in user's wallet).
7. NFT line item retains `royaltyPercentage: 0`; `royaltyPercentagesForSide` filters it out.
8. `amountWithRoyalties` is never set; confirmation dialog shows only "You Spend: 1 XCH" with no royalty disclosure.
9. User approves.
10. Wallet daemon executes `take_offer`; on-chain royalty puzzle enforces 5% → user's wallet is debited 1.05 XCH instead of the 1 XCH shown.

```typescript
function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],
): string | undefined {
  if (royaltyPercentages.length === 0 || line.kind === 'nft') {
    return undefined;
  }

  const splitAmount = amount / BigInt(royaltyPercentages.length);
  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;

  if (line.kind === 'xch') {
    return mojoToChiaLocaleString(totalAmount);
  }

  return mojoToCATLocaleString(totalAmount);
}
