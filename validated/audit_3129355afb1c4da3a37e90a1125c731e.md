### Title
`upgrade_delay_blocks` Security Guarantee Bypassed via Direct `upgrade()` Entrypoint — (`engine/src/contract_methods/admin.rs`)

---

### Summary

The Aurora Engine exposes two distinct upgrade paths: a staged path (`stage_upgrade` → `deploy_upgrade`) that enforces `upgrade_delay_blocks`, and a direct path (`upgrade()`) that deploys new WASM code immediately with no delay check whatsoever. Because both paths are gated only by `require_owner_only`, the `upgrade_delay_blocks` delay — the sole mechanism protecting users from an immediate malicious contract replacement — is completely bypassable by the owner in a single transaction.

---

### Finding Description

`EngineState` stores `upgrade_delay_blocks` as a security parameter intended to give users advance warning before a new contract binary is deployed:

```rust
/// How many blocks after staging upgrade can deploy it.
pub upgrade_delay_blocks: u64,
```

The intended safe upgrade flow enforces this delay:

1. `stage_upgrade` stores the code and records `delay_block_height = env.block_height() + state.upgrade_delay_blocks`.
2. `deploy_upgrade` checks `if io.block_height() <= index { panic }` before deploying.

However, a parallel entrypoint `upgrade()` exists that directly schedules `PromiseAction::DeployContract` with no reference to `upgrade_delay_blocks` at all:

```rust
pub fn upgrade<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I, env: &E, handler: &mut H,
) -> Result<(), ContractError> {
    let state = state::get_state(&io)?;
    require_running(&state)?;
    require_owner_only(&state, &env.predecessor_account_id())?;
    // ... reads code from input ...
    let batch = PromiseBatchAction {
        target_account_id,
        actions: vec![
            PromiseAction::DeployContract { code },   // ← no delay check
            PromiseAction::FunctionCall { name: "state_migration".to_string(), ... },
        ],
    };
    handler.promise_return(handler.promise_create_batch(&batch));
    Ok(())
}
```

This is exposed as a public NEAR entrypoint:

```rust
#[unsafe(no_mangle)]
pub extern "C" fn upgrade() {
    contract_methods::admin::upgrade(io, &env, &mut handler)...
}
```

Additionally, `set_upgrade_delay_blocks` itself has no minimum-value guard and no delay on the change, so the owner can also reduce `upgrade_delay_blocks` to `0` immediately and then use the staged path — but the `upgrade()` bypass makes even that step unnecessary.

---

### Impact Explanation

**Critical — Direct theft of all user funds / permanent freezing of funds.**

The Aurora Engine contract holds bridged ETH (via the ETH connector) and is the authoritative executor of all EVM state. Replacing the WASM binary with malicious code in a single block allows the owner to:

- Drain all bridged ETH balances by overriding withdrawal logic.
- Corrupt or freeze all EVM account state, permanently freezing user funds.
- Redirect any future deposits to an attacker-controlled address.

The `upgrade_delay_blocks` parameter exists precisely to give users a window to observe a staged upgrade and exit before it takes effect. The `upgrade()` entrypoint eliminates that window entirely.

---

### Likelihood Explanation

The attack requires the owner account to be the caller. The `upgrade_delay_blocks` mechanism is the protocol's own stated defense against a malicious or compromised owner. Its purpose is to make the threat of owner key compromise survivable for users. Because `upgrade()` silently bypasses this defense, the protection is illusory: any owner-key compromise or malicious owner action can be executed in one block with no observable warning, making the risk materially higher than if the delay were properly enforced.

---

### Recommendation

1. **Remove or gate the direct `upgrade()` entrypoint.** If it must exist for emergency use, require it to go through the same `stage_upgrade` + `deploy_upgrade` delay path, or require a multi-sig / DAO approval that is itself time-locked.
2. **Enforce a minimum value in `set_upgrade_delay_blocks`.** Add a lower-bound check (e.g., `require!(args.upgrade_delay_blocks >= MIN_DELAY)`) so the delay cannot be reduced to zero without governance.
3. **Apply a delay to `set_upgrade_delay_blocks` itself.** Changes to the delay parameter should only take effect after the current delay has elapsed, preventing an owner from atomically zeroing the delay and then upgrading.

---

### Proof of Concept

**Step 1 — Owner calls `upgrade()` directly (no staging, no delay):**

```
near call aurora upgrade '{"code": "<malicious_wasm_bytes>"}' \
  --accountId owner.near \
  --gas 300000000000000
```

This invokes `contract_methods::admin::upgrade`, which immediately schedules `PromiseAction::DeployContract` with the malicious binary. [1](#0-0) 

**Step 2 — The malicious binary is live in the same block (next receipt).**

No `upgrade_delay_blocks` check is performed anywhere in this path. [2](#0-1) 

**Contrast with the intended safe path** (`deploy_upgrade` enforces the delay):

```rust
if io.block_height() <= index {
    sdk::panic_utf8(errors::ERR_NOT_ALLOWED_TOO_EARLY);
}
``` [3](#0-2) 

The `upgrade_delay_blocks` field in `EngineState` is never read inside `upgrade()`: [4](#0-3)

pub fn upgrade<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<(), ContractError> {
    let state = state::get_state(&io)?;
    require_running(&state)?;
    require_owner_only(&state, &env.predecessor_account_id())?;

    let input = io.read_input().to_vec();
    let (code, state_migration_gas) = match UpgradeParams::try_from_slice(&input) {
        Ok(args) => (
            args.code,
            args.state_migration_gas
                .map_or(GAS_FOR_STATE_MIGRATION, NearGas::new),
        ),
        Err(_) => (input, GAS_FOR_STATE_MIGRATION), // Backward compatibility
    };

    let target_account_id = env.current_account_id();
    let batch = PromiseBatchAction {
        target_account_id,
        actions: vec![
            PromiseAction::DeployContract { code },
            PromiseAction::FunctionCall {
                name: "state_migration".to_string(),
                args: vec![],
                attached_yocto: ZERO_YOCTO,
                gas: state_migration_gas,
            },
        ],
    };
    let promise_id = handler.promise_create_batch(&batch);

    handler.promise_return(promise_id);

    Ok(())
