### Title
`skip_stateful_validations` Silently Bypasses `__validate__` Signature Check for Recently-Deployed Accounts — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The gateway's `skip_stateful_validations` function silently skips the `__validate__` entry-point call (the account's signature-verification step) for any invoke transaction whose nonce is `1` when `account_tx_in_pool_or_recent_block` returns `true`. Because that helper returns `true` not only when a `deploy_account` is still pending in the mempool but also when the account has **any** transaction in a recent block, an attacker can submit an invoke with nonce `1` and an invalid signature for any recently-deployed account (deploy_account already committed, on-chain nonce still `0`). The gateway accepts the transaction without verifying the signature, and it enters the mempool.

---

When `skip_stateful_validations` returns `true`, the caller sets `execution_flags.validate = false`: [2](#0-1)

```

Inside `StatefulValidator::perform_validations`, a `false` `validate` flag causes the `__validate__` entry point to be skipped entirely: [3](#0-2) 

```rust

    return Ok(());
}

```

**The over-broad `account_tx_in_pool_or_recent_block` condition:**

The inline comment in `skip_stateful_validations` states:

> "It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."

However, the function is named `account_tx_in_pool_or_recent_block` — it also returns `true` when the account has a transaction in a **recent block**. For a freshly-deployed account (deploy_account committed to a recent block, on-chain nonce still `0`), this condition is satisfied without any pending deploy_account in the mempool. An attacker who observes the deployment can immediately submit an invoke with nonce `1` and an arbitrary (invalid) signature; the gateway will skip `__validate__` and forward the transaction to the mempool.

**Propagation chain (analog to the external report):**

| Step | External report | Sequencer analog |
|------|----------------|-----------------|
| Trigger | Chainlink oracle returns stale data | Account has deploy_account in recent block |
| Silent failure | `getPriceAt()` returns `0` silently | `account_tx_in_pool_or_recent_block` returns `true`, `skip_stateful_validations` returns `true` silently |

**Gateway admission flow:** [4](#0-3) 

```rust
async fn extract_state_nonce_and_run_validations(...) {
    let account_nonce = self.get_nonce_from_state(...).await?;
    let skip_validate =
        self.run_pre_validation_checks(executable_tx, account_nonce, mempool_client).await?;
    self.run_validate_entry_point(executable_tx, skip_validate).await?;
    Ok(account_nonce)
}
```

`run_pre_validation_checks` calls `skip_stateful_validations`, which returns `true` for the attack scenario. `run_validate_entry_point` then runs the blockifier validator with `validate = false`, so `__validate__` is never called.

---

### Impact Explanation

An unprivileged attacker can submit an invoke transaction with a forged or empty signature for any account that:
1. Was deployed in a recent block (deploy_account committed on-chain), and
2. Has on-chain nonce `0` (has not yet sent its first transaction).

The gateway accepts the transaction without signature verification and forwards it to the mempool. When the batcher later executes the transaction, the blockifier **does** run `__validate__` (the batcher sets its own `execution_flags` independently of the gateway). If the signature is invalid, `__validate__` fails, the transaction reverts, and the victim account is charged a fee for the failed validation step. This is a griefing/fee-drain attack on newly-deployed accounts.

Impact category: **High — Mempool/gateway admission accepts invalid transactions before sequencing.**

---

### Likelihood Explanation

- Newly-deployed accounts are publicly visible on-chain; an attacker can monitor for deploy_account transactions and immediately target the victim.
- The attack window is the period between the deploy_account being committed and the account sending its first transaction (nonce `0 → 1`).
- No special privileges are required; any user can submit transactions to the gateway.
- The only constraint is that the victim account must have a non-zero STRK balance (otherwise `verify_can_pay_committed_bounds` in the batcher rejects the transaction before `__validate__` runs).

---

### Recommendation

Tighten the skip condition to only match when a **pending deploy_account is in the mempool**, not when any transaction exists in a recent block:

```rust
async fn skip_stateful_validations(...) -> ... {
    if let ExecutableTransaction::Invoke(...) = tx {
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            // Only skip if a deploy_account is still pending in the mempool,
            // NOT if the account merely appeared in a recent block.
            return mempool_client
                .has_pending_deploy_account(tx.sender_address())  // new, narrower check
                .await
                ...
        }
    }
    Ok(false)
}
```

Alternatively, add a separate mempool API that distinguishes between "deploy_account pending" and "account seen in recent block," and only skip `__validate__` in the former case.

### Proof of Concept

```
1. Deploy account A on-chain (deploy_account committed in block N).
   Account A now has on-chain nonce = 0.

2. Observe block N; extract account A's address.

3. Construct an invoke transaction:
   - sender_address = A
   - nonce = 1
   - signature = [0x0, 0x0]  // invalid / empty signature
   - resource_bounds = valid (passes stateless check)

4. Submit to the gateway via POST /add_transaction.

5. Gateway flow:
   a. StatelessTransactionValidator::validate → passes (no signature check here).
   b. convert_rpc_tx_to_internal_and_executable_txs → passes.
   c. extract_state_nonce_and_run_validations:
      - get_nonce_from_state(A) → 0
      - validate_state_preconditions: nonce 1 ∈ [0, 200] → passes
      - validate_by_mempool → passes (no signature check)
      - skip_stateful_validations:
          tx.nonce() == 1 ✓, account_nonce == 0 ✓
          account_tx_in_pool_or_recent_block(A) → true (deploy_account in block N) ✓
          returns true (skip __validate__)
      - run_validate_entry_point with validate=false → __validate__ NOT called
   d. Transaction forwarded to mempool. ✓

6. Batcher picks up the transaction and executes it:
   - perform_pre_validation_stage: checks fee bounds and balance → passes if A has balance.
   - __validate__ IS called by the batcher (validate=true in batcher's execution_flags).
   - __validate__ fails (invalid signature) → transaction reverts.
   - Account A is charged a fee for the failed validation.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)

    pub fn perform_validations(&mut self, tx: AccountTransaction) -> StatefulValidatorResult<()> {
        // Deploy account transaction should be fully executed, since the constructor must run
        // before `__validate_deploy__`. The execution already includes all necessary validations,
        // so they are skipped here.
        // Declare transaction should also be fully executed - otherwise, if we only go through
        // the validate phase, we would miss the check that the class was not declared before.
        match tx.tx {
            ApiTransaction::DeployAccount(_) | ApiTransaction::Declare(_) => self.execute(tx),
            ApiTransaction::Invoke(_) => {
                let tx_context = Arc::new(self.tx_executor.block_context.to_tx_context(&tx));
                tx.perform_pre_validation_stage(self.state(), &tx_context)?;
                if !tx.execution_flags.validate {
                    return Ok(());
                }

                // `__validate__` call.
                let (_optional_call_info, actual_cost) = self.validate(&tx, tx_context.clone())?;

                // Post validations.
                PostValidationReport::verify(
                    &tx_context,
                    &actual_cost,
                    tx.execution_flags.charge_fee,
                )?;

                Ok(())
            }
        }
