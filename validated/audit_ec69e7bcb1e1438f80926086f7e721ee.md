### Title
Unauthorized Transfer and Burn of Native EVM-Denom Tokens via Missing Caller Authorization in Bank Precompile - (File: `x/cronos/keeper/precompiles/bank.go`)

---

### Summary

The `BankContract.Run()` function in the bank precompile handles `transfer` and `burn` operations without verifying that the `sender`/`addr` argument matches the calling contract (`contract.Caller()`). Any deployed EVM contract can invoke the bank precompile to transfer or burn native `evm/<ContractAddress>` tokens from any holder's account without their consent.

---

### Finding Description

In `x/cronos/keeper/precompiles/bank.go`, the `TransferMethodName` case unpacks a `sender` address from the ABI-encoded call arguments and passes it directly to `bc.bankKeeper.SendCoins(ctx, from, to, ...)`:

```go
// bank.go lines 175–192
sender := args[0].(common.Address)
recipient := args[1].(common.Address)
amount := args[2].(*big.Int)
// ...
from := sdk.AccAddress(sender.Bytes())
to := sdk.AccAddress(recipient.Bytes())
// ...
denom := EVMDenom(contract.Caller())   // evm/<CallerAddress>
amt := sdk.NewCoin(denom, sdkmath.NewIntFromBigInt(amount))
// ...
bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt))
```

There is **no check** that `sender == contract.Caller()`. The calling contract supplies an arbitrary `sender` address, and the bank module transfers that address's `evm/<CallerAddress>` tokens to any `recipient` without the token holder's authorization.

The same flaw exists in the `BurnMethodName` case:

```go
// bank.go lines 121–150
recipient := args[0].(common.Address)   // misleadingly named; this is the address burned FROM
amount := args[1].(*big.Int)
addr := sdk.AccAddress(recipient.Bytes())
// ...
denom := EVMDenom(contract.Caller())
// ...
bc.bankKeeper.SendCoinsFromAccountToModule(ctx, addr, types.ModuleName, sdk.NewCoins(amt))
bc.bankKeeper.BurnCoins(ctx, types.ModuleName, sdk.NewCoins(amt))
```

Again, `addr` is attacker-controlled and there is no check that `addr == contract.Caller()`.

By contrast, the relayer precompile's `exec()` utility in `x/cronos/keeper/precompiles/utils.go` (lines 38–49) correctly enforces `caller == signers[0]` before executing any native action. The bank precompile has no equivalent guard.

---

### Impact Explanation

**Critical — Unauthorized transfer/burn of native EVM-denom tokens.**

Any EVM contract deployed at address `A` can:
1. Call `transfer(victimAddress, attackerAddress, amount)` on the bank precompile (address `0x0000000000000000000000000000000000000064`).
2. The precompile executes `SendCoins(victimAddress, attackerAddress, amount evm/A)` with no authorization check.
3. All holders of `evm/A` tokens can be drained to any attacker-controlled address.

Alternatively, calling `burn(victimAddress, amount)` destroys the victim's `evm/A` tokens without consent.

The scope is limited to tokens of the calling contract's own denom (`evm/<CallerAddress>`), but this is precisely the denom that the contract itself has minted to users — making every token holder of that contract a potential victim.

---

### Likelihood Explanation

**High.** The bank precompile is callable by any EVM contract with no access restriction. A malicious contract author deploys a contract, mints `evm/<ContractAddress>` tokens to users (e.g., as part of a DeFi protocol), and then calls `transfer` to drain all holders. No privileged keys, governance access, or cryptographic break is required — only the ability to deploy and call an EVM contract.

---

### Recommendation

Add an authorization check in both the `transfer` and `burn` cases to ensure the `sender`/`addr` argument equals `contract.Caller()`:

```go
// transfer case
if sender != contract.Caller() {
    return nil, errors.New("sender must be the calling contract")
}

// burn case
if recipient != contract.Caller() {
    return nil, errors.New("burn target must be the calling contract")
}
```

This mirrors the pattern already used in the relayer precompile's `exec()` function (`utils.go` lines 46–49), which rejects any message whose signer does not match the EVM caller.

---

### Proof of Concept

1. Attacker deploys `MaliciousToken` at EVM address `0xABC`.
2. `MaliciousToken` calls bank precompile `mint(victimAddress, 1000)` — victim now holds `1000 evm/0xABC`.
3. Attacker calls bank precompile `transfer(victimAddress, attackerAddress, 1000)` from `MaliciousToken`.
4. Bank precompile executes `SendCoins(victimAddress, attackerAddress, 1000 evm/0xABC)` — **no authorization check performed**.
5. Victim loses all `evm/0xABC` tokens; attacker receives them.
