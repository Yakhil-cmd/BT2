### Title
Recreated EVM Account Silently Deleted at Commit Due to Missing `recreated` Guard in `HasSelfDestructed` ‚Äî (`fvm/evm/emulator/state/stateDB.go`, `delta.go`)

---

### Summary

`Commit()` in `StateDB` calls `db.HasSelfDestructed(addr)`, which walks the `DeltaView` parent chain and returns `true` if any ancestor view has `addr` in its `toBeDestructed` map. It does **not** check whether a child view has since set `recreated[addr]`. When a contract is self-destructed in view[0] and then recreated via CREATE2 in view[1] (a child snapshot), `Commit()` unconditionally deletes the account and short-circuits with `continue`, never reaching the `IsCreated` branch. The recreated account's balance, code, and storage are silently dropped.

```go

    bal, found := d.toBeDestructed[addr]
```

The `recreated` map is populated by `CreateAccount` when it detects that the address was previously self-destructed: [3](#0-2) 

```go
d.recreated[addr] = struct{}{}   // set in child view[1]
...
d.created[addr] = struct{}{}
```

But `HasSelfDestructed` never consults `d.recreated`. A child view that has `recreated[A]` set is semantically saying "A was destroyed in a parent, but I have resurrected it." The parent-chain walk in `HasSelfDestructed` ignores this entirely.

| Step | View | Operation | Effect |
|------|------|-----------|--------|
| 1 | view[0] | `SelfDestruct6780(A)` (A is a new contract) | `view[0].toBeDestructed[A] = bal`, `view[0].balances[A] = 0` |

| 4 | view[1] | `AddBalance(A, X)` | `view[1].balances[A] = X` |
| 5 | ‚Äî | `Commit(true)` | `db.HasSelfDestructed(A)` ‚Üí view[1] has no `toBeDestructed[A]` ‚Üí walks to view[0] ‚Üí **returns true** ‚Üí `DeleteAccount(A)` ‚Üí `continue` ‚Üí `IsCreated` never checked |

### Impact Explanation

The recreated contract account is permanently deleted from the ledger. Any ETH balance held by the recreated contract is irrecoverably lost. If the contract was used as an escrow or vault (e.g., a CREATE2-deterministic address used to receive funds before deployment), those funds are silently burned. This constitutes unauthorized destruction of an EVM resource and permanent loss of escrowed assets ‚Äî matching the Critical scope target.

---

### Likelihood Explanation

The attack is reachable through a standard EVM transaction submitted to Flow's EVM gateway. No privileged access is required. The pattern (CREATE2 deploy ‚Üí SELFDESTRUCT ‚Üí CREATE2 resurrect in same tx) is a well-known EVM pattern and is valid under EIP-6780 (Cancun), which Flow EVM supports. An attacker-controlled contract can trigger this sequence entirely within a single transaction.

---

### Recommendation

`DeltaView.HasSelfDestructed` must return `(false, nil)` when `d.recreated[addr]` is set, because recreation in a child view supersedes destruction in a parent view:

```go
func (d *DeltaView) HasSelfDestructed(addr gethCommon.Address) (bool, *uint256.Int) {
    // If this view recreated the account, the prior self-destruct is superseded.
    if _, recreated := d.recreated[addr]; recreated {
        return false, nil
    }
    bal, found := d.toBeDestructed[addr]
    if found {
        return true, bal
    }
    return d.parent.HasSelfDestructed(addr)
}
```

Alternatively, `Commit()` should check `db.IsCreated(addr)` **before** `db.HasSelfDestructed(addr)` and treat a created+self-destructed address as a net creation (matching Geth's own `stateObject` logic where `newborn` overrides `selfdestructed`).

### Proof of Concept

```go
// Construct StateDB
db, _ := NewStateDB(ledger, root)

addrA := gethCommon.HexToAddress("0xdeadbeef...")

// Step 1: Simulate contract creation and self-destruct in view[0]
db.CreateAccount(addrA)
db.CreateContract(addrA)                          // marks as newContract
db.AddBalance(addrA, uint256.NewInt(1e18), 0)
db.SelfDestruct6780(addrA)                        // sets toBeDestructed[A] in view[0]

// Step 2: Snapshot ‚Üí view[1]
db.Snapshot()

// Step 3: Recreate A in view[1] and fund it
db.CreateAccount(addrA)                           // sets recreated[A], created[A] in view[1]
db.AddBalance(addrA, uint256.NewInt(42), 0)       // view[1].balances[A] = 42

// Step 4: Commit
db.Commit(true)

// Step 5: Read back from a fresh StateDB over the same ledger
db2, _ := NewStateDB(ledger, root)
assert(db2.Exist(addrA) == true)                  // FAILS: account was deleted
assert(db2.GetBalance(addrA).Eq(uint256.NewInt(42))) // FAILS: balance is 0
```

The assertions fail because `Commit()` calls `DeleteAccount(A)` (driven by the parent's `toBeDestructed` entry) and never calls `CreateAccount(A)` for the resurrected version.
