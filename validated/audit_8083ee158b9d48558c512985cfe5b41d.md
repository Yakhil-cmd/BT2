### Title
Gateway L2 Gas Price Admission Check Uses Previous Block's `l2_gas_price` Instead of `next_l2_gas_price`, Causing False Accepts and False Rejects - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

`StatefulTransactionValidator::validate_resource_bounds` reads the L2 gas price from the **current block's** `gas_prices.strk_gas_prices.l2_gas_price` field to threshold-check incoming transactions. The price that will actually govern execution of those transactions is `next_l2_gas_price` (the EIP-1559-derived price stored in the same block header for use in the *next* block). Because the two values diverge every block, the gateway admission check is always comparing against the wrong reference price. The code itself carries an explicit acknowledgement of this: `// TODO(Arni): getnext_l2_gas_price from the block header.`

### Finding Description

**Root cause — wrong field read in `GatewayFixedBlockSyncStateClient::get_block_info_from_sync_client`**

`GatewayFixedBlockSyncStateClient` constructs a `BlockInfo` from the latest committed block header. It populates `gas_prices.strk_gas_prices.l2_gas_price` from `block_header.l2_gas_price.price_in_fri` — the price that was used for transactions *inside* that block. [1](#0-0) 

The block header also carries a separate `next_l2_gas_price` field (the EIP-1559 output for the *following* block), but `get_block_info_from_sync_client` never reads it.

**Wrong price propagated into `validate_resource_bounds`**

`validate_resource_bounds` calls `gateway_fixed_block_state_reader.get_block_info()` and names the result `previous_block_l2_gas_price`, then uses it as the threshold reference: [2](#0-1) 

The TODO comment on line 229 explicitly flags that `next_l2_gas_price` should be used instead.

**What the batcher actually uses**

`calculate_next_l2_gas_price_for_fin` / `update_l2_gas_price` compute the EIP-1559 price from block N's gas usage and store it as `next_l2_gas_price` in block N's header. When the batcher builds block N+1, it uses this value as the block's `l2_gas_price`. The blockifier's `check_fee_bounds` then enforces `tx.max_price_per_unit >= next_l2_gas_price`. [3](#0-2) 

**The divergence**

| Price | Source | Used by |
|---|---|---|
| `block_N.l2_gas_price` | Price for txs in block N | Gateway admission check (wrong) |
| `block_N.next_l2_gas_price` | EIP-1559 output → price for block N+1 | Batcher execution (correct) |

The EIP-1559 algorithm adjusts the price by up to ~0.3 % per block (1/333 of current price), so after a sustained period of high or low utilisation the two values can differ by tens of percent.

**`run_validate_entry_point` has the same stale-price problem**

The same `gateway_fixed_block_state_reader.get_block_info()` result (with the wrong `l2_gas_price`) is also fed into the blockifier `BlockContext` used for the `__validate__` entry-point call inside the gateway: [4](#0-3) 

So both the admission threshold check and the blockifier pre-validation inside the gateway use the stale price.

### Impact Explanation

**Scenario A — False acceptance (price rising, `next_l2_gas_price > l2_gas_price`)**

A user submits a transaction with `max_price_per_unit = l2_gas_price` (exactly the previous block's price). The gateway threshold check passes (`l2_gas_price >= 100 % × l2_gas_price`). The gateway blockifier validation also passes (same stale price in `BlockContext`). The transaction enters the mempool. When the batcher executes it in block N+1, `check_fee_bounds` compares against `next_l2_gas_price > l2_gas_price` and rejects with `MaxGasPriceTooLow`. The transaction is admitted to the mempool but can never be included in a block. An attacker can flood the mempool with such transactions at zero cost (they are never executed, so no fee is charged).

**Scenario B — False rejection (price falling, `next_l2_gas_price < l2_gas_price`)**

A user submits a transaction with `max_price_per_unit = next_l2_gas_price` (the correct price for the next block). The gateway threshold check fails (`next_l2_gas_price < 100 % × l2_gas_price`) and the transaction is rejected with `GAS_PRICE_TOO_LOW`. The transaction is valid and would succeed in block N+1, but the user is denied service.

Both scenarios are reachable by any unprivileged user submitting a standard V3 (`AllResources`) transaction. The EIP-1559 price moves every block, so the window of divergence is always present.

### Likelihood Explanation

The EIP-1559 fee market (`calculate_next_l2_gas_price_for_fin`) adjusts the L2 gas price every block based on actual gas usage relative to `gas_target`. Under normal network load the price drifts continuously. The divergence between `l2_gas_price` and `next_l2_gas_price` is therefore present in every block, not just under adversarial conditions. The TODO comment in the production code confirms the team is aware the wrong field is being read.

### Recommendation

In `GatewayFixedBlockSyncStateClient::get_block_info_from_sync_client`, replace the `l2_gas_price` field population with `block_header.next_l2_gas_price` (the EIP-1559 output for the next block):

```rust
// Before (wrong):
l2_gas_price: block_header.l2_gas_price.price_in_fri.try_into()?,

// After (correct):
l2_gas_price: block_header.next_l2_gas_price.try_into()?,
```

This single change propagates the correct reference price into both `validate_resource_bounds` and the `BlockContext` used in `run_validate_entry_point`, aligning the gateway's admission check with the price the batcher will actually enforce during execution.

### Proof of Concept

1. Observe block N with `l2_gas_price = P` and `next_l2_gas_price = P'` where `P' > P` (sustained high utilisation).
2. Craft a V3 `InvokeTransaction` with `resource_bounds.l2_gas.max_price_per_unit = P`.
3. Submit to the gateway. `validate_resource_bounds` computes threshold = `100% × P = P`; the transaction's price equals the threshold → **admitted**.
4. The transaction enters the mempool.
5. The batcher picks it up for block N+1. `check_fee_bounds` compares `P < P'` → `MaxGasPriceTooLow` → transaction is **reverted/dropped** and no fee is charged.
6. Repeat from step 2 to continuously fill the mempool with permanently-unexecutable transactions. [5](#0-4) [6](#0-5) [7](#0-6)

/// Compute the next L2 gas price (for the fin or for updating state). Respects override when set.
pub fn calculate_next_l2_gas_price_for_fin(
    current_l2_gas_price: GasPrice,
    height: BlockNumber,
    l2_gas_used: GasAmount,
    override_l2_gas_price_fri: Option<u128>,
    min_l2_gas_price_per_height: &[PricePerHeight],
    fee_actual: Option<GasPrice>,
) -> GasPrice {
    if let Some(override_value) = override_l2_gas_price_fri {
        info!(
            "L2 gas price ({}) is not updated, remains on override value of {override_value} fri",
            current_l2_gas_price.0
        );
        return GasPrice(override_value);
    }
    let gas_target = VersionedConstants::latest_constants().gas_target;
    let config_min = get_min_gas_price_for_height(height, min_l2_gas_price_per_height);
    let effective_min = match fee_actual {
        Some(fa) => GasPrice(max(config_min.0, fa.0)),
        None => config_min,
    };
    calculate_next_base_gas_price(current_l2_gas_price, l2_gas_used, gas_target, effective_min)
}
