### Title
`gas_per_pubdata_limit` Can Be Zero for L1→L2 Transactions, Bypassing Pubdata Cost Enforcement - (File: `basic_bootloader/src/bootloader/transaction_flow/zk/process_l1_transaction.rs`)

---

### Summary

The `gas_per_pubdata_limit` field of L1→L2 (priority) transactions is read directly from the transaction without any zero-value guard. When a user submits an L1→L2 transaction with `gas_per_pubdata_limit = 0`, the derived `native_per_pubdata` collapses to zero, completely disabling pubdata cost enforcement for that transaction. The transaction can then generate significant pubdata (storage writes) without paying for it, while the protocol still bears the L1 publication cost.

---

### Finding Description

In `process_l1_transaction.rs`, the pubdata limit is read verbatim from the transaction:

```rust
let gas_per_pubdata = transaction.gas_per_pubdata_limit.read();
``` [1](#0-0) 

It is then used to derive `native_per_pubdata` in `prepare_and_check_resources`:

```rust
let native_per_pubdata = (gas_per_pubdata as u64)
    .checked_mul(native_per_gas)
    .unwrap_or_else(|| { ... u64::MAX });
``` [2](#0-1) 

When `gas_per_pubdata = 0`, `native_per_pubdata = 0 × native_per_gas = 0`. This zero propagates through the entire pubdata accounting chain:

**Step 1 — Intrinsic pubdata overhead zeroed out** in `create_resources_for_tx`:

```rust
let intrinsic_pubdata_overhead = native_per_pubdata_byte.saturating_mul(intrinsic_pubdata);
// = 0 × intrinsic_pubdata = 0  → no charge for intrinsic pubdata
``` [3](#0-2) 

**Step 2 — Post-execution pubdata cost zeroed out** in `get_resources_to_charge_for_pubdata`:

```rust
let native = current_pubdata_spent
    .checked_mul(native_per_pubdata)
    .ok_or(out_of_native_resources!())?;
// = pubdata_used × 0 = 0  → zero cost for any amount of pubdata
``` [4](#0-3) 

**Step 3 — Pubdata sufficiency check always passes** in `check_enough_resources_for_pubdata`:

```rust
let enough = resources.has_enough(&resources_for_pubdata);
// resources_for_pubdata = 0  → always enough
``` [5](#0-4) 

The `validate_structure` function in `abi_encoded/mod.rs` only enforces that `gas_per_pubdata_limit` is zero for *non*-L1 transaction types; it imposes no lower bound for L1 transactions:

```rust
// gas_per_pubdata_limit should be zero for non L1 transactions
match tx_type {
    Self::UPGRADE_TX_TYPE | Self::L1_L2_TX_TYPE => {}
    _ => {
        if self.gas_per_pubdata_limit.read() != 0 {
            return Err(());
        }
    }
}
``` [6](#0-5) 

The code comment in `prepare_and_check_resources` acknowledges the design intent but does not add a guard:

> "Note that the 'validation errors' are practically unreachable, as gas_limit, gas_price and gas_per_pubdata are either checked or set by the L1 contracts." [7](#0-6) 

This is the exact analog of the reported Gearbox bug: a limit variable (`gas_per_pubdata_limit` / `expectedLiquidityLimit`) that defaults to or can be set to zero, bypassing the enforcement it is meant to provide.

---

### Impact Explanation

An L1→L2 transaction with `gas_per_pubdata_limit = 0`:

1. Executes normally and can write to many storage slots, generating large pubdata.
2. The post-execution pubdata check (`check_enough_resources_for_pubdata`) always returns `enough = true` because `native_per_pubdata = 0`.
3. The transaction succeeds and its pubdata is committed to the block.
4. The pubdata is published to L1 (Ethereum), incurring real ETH cost borne by the protocol/operator.
5. The attacker paid only for EVM gas (deposited on L1), not for pubdata.

The block-level pubdata limit (`check_for_block_limits`) still caps the total pubdata per block, so the attack is bounded per block. However, within that cap, an attacker can consume the entire block's pubdata budget for free, repeatedly across blocks, draining the operator's L1 publication budget. [8](#0-7) 

---

### Likelihood Explanation

Any user who can submit an L1→L2 priority transaction can set `gas_per_pubdata_limit = 0` in the ABI-encoded transaction fields. The ZKsync OS code itself provides no defense-in-depth validation against this. The only protection is the L1 contract layer, which is external to ZKsync OS. If the L1 contracts have any gap (e.g., after an upgrade that changes the minimum `gas_per_pubdata_limit` requirement, or if the L1 contract validation is ever bypassed), this path becomes directly exploitable by any unprivileged caller.

The comment in the code explicitly acknowledges that the L1 contracts are the sole enforcement point, making ZKsync OS entirely dependent on external validation for this invariant.

---

### Recommendation

Add a zero-value guard for `gas_per_pubdata_limit` in L1 transaction processing. The most appropriate location is `validate_structure` in `abi_encoded/mod.rs`, mirroring the existing check for non-L1 types:

```rust
Self::L1_L2_TX_TYPE => {
    if self.gas_per_pubdata_limit.read() == 0 {
        return Err(());
    }
}
```

Alternatively, add the check in `prepare_and_check_resources` and handle it with the same saturating/logging approach used for other L1 transaction resilience cases, emitting a system log and using a safe fallback value rather than silently proceeding with zero.

---

### Proof of Concept

1. Construct an ABI-encoded L1→L2 transaction (`tx_type = 0x7F`) with `gas_per_pubdata_limit = 0`, a non-zero `gas_limit`, and calldata that writes to 10+ storage slots (generating pubdata).
2. Submit the transaction through the L1 priority queue (or directly in a test using `L1TxBuilder::new().gas_per_pubdata_byte_limit(0)`).
3. Observe that `native_per_pubdata = 0` is computed in `prepare_and_check_resources`.
4. Observe that `check_enough_resources_for_pubdata` returns `enough = true` regardless of pubdata generated.
5. The transaction succeeds; pubdata is committed to the block and published to L1 at zero cost to the sender. [2](#0-1) [9](#0-8)
