### Title
Unbounded `pool_member_epoch_balance` Trace Growth Causes Permanent Freezing of Unclaimed Yield in `claim_rewards()` - (File: `src/pool/pool.cairo`)

---

### Summary

The `calculate_rewards()` function in the delegation pool contract contains an explicitly acknowledged unbounded loop that iterates over every entry in a pool member's `pool_member_epoch_balance` trace. Because any delegator can grow this trace by one entry per epoch through repeated calls to `add_to_delegation_pool()` or `exit_delegation_pool_intent()`, a delegator who accumulates enough trace entries without claiming rewards will eventually find their `claim_rewards()` call permanently exceeding Starknet's gas limit, freezing their unclaimed yield forever.

---

### Finding Description

In `src/pool/pool.cairo`, the internal `calculate_rewards()` function iterates over the pool member's balance trace:

```
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
    if pool_member_checkpoint.epoch() >= until_epoch { break; }
    let to_sigma = self.find_sigma(pool_member_checkpoint, curr_epoch: until_epoch);
    ...
    entry_to_claim_from += 1;
}
```

The developers themselves flag this as unbounded. The trace (`pool_member_epoch_balance`) is a `Vec`-backed append-only structure defined in `src/pool/pool_member_balance_trace/trace.cairo`. Its `insert()` method appends a new checkpoint whenever the epoch key differs from the last entry:

```cairo
} else {
    assert!(last.key < key, "{}", TraceErrors::UNORDERED_INSERTION);
    checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
}
```

Every call to `add_to_delegation_pool()` or `exit_delegation_pool_intent()` invokes `set_member_balance()` / `increase_member_balance()`, which calls `trace.insert(key: self.get_epoch_plus_k(), ...)`. Because the key is `current_epoch + K`, each call in a distinct epoch appends a new entry. There is no cap on how many entries can accumulate.

The `entry_to_claim_from` pointer stored in `pool_member_info` is only advanced on a successful `claim_rewards()` call. If a delegator makes one balance change per epoch across N epochs without claiming, the loop must iterate N times on the next `claim_rewards()` call. Each iteration performs multiple storage reads (`pool_member_trace.at()` and `find_sigma()` reading from `cumulative_rewards_trace`), which are expensive. Once N is large enough to exhaust the Starknet per-transaction gas budget, the call reverts permanently.

There is no partial-claim mechanism and no trace-pruning path. Once the threshold is crossed, the delegator's accumulated rewards are irrecoverable.

---

### Impact Explanation

**Permanent freezing of unclaimed yield.** Once the trace length exceeds the gas-limit threshold, every future `claim_rewards()` call for that pool member will revert. The rewards are locked in the pool contract with no recovery path. This matches the allowed impact: *"Permanent freezing of unclaimed yield or unclaimed royalties."*

---

### Likelihood Explanation

**Medium.** The condition requires a delegator to make balance changes across many epochs without claiming rewards. This can happen:

1. **Inadvertently** — a delegator who actively adjusts their stake each epoch (e.g., dollar-cost averaging in or out) and neglects to claim rewards for months or years.
2. **Deliberately** — a malicious delegator who intentionally inflates their own trace to permanently forfeit rewards, causing accounting noise or demonstrating a protocol weakness.

The minimum cost per epoch is one `add_to_delegation_pool()` call with 1 token unit. No privileged access is required; any delegator can trigger this against their own account.

---

### Recommendation

1. **Bound the loop**: Process at most `MAX_ENTRIES_PER_CLAIM` trace entries per `claim_rewards()` call and persist the updated `entry_to_claim_from`, allowing incremental reward collection across multiple transactions.
2. **Prune claimed entries**: After advancing `entry_to_claim_from`, mark or compact old entries so they are not re-iterated.
3. **Rate-limit balance changes**: Enforce a maximum number of pending unclaimed trace entries before allowing further `add_to_delegation_pool()` or `exit_delegation_pool_intent()` calls.

---

### Proof of Concept

1. Delegator calls `enter_delegation_pool()` with the minimum non-zero amount.
2. Each epoch, delegator calls `add_to_delegation_pool()` with 1 token unit. Each call in a new epoch appends one entry to `pool_member_epoch_balance` via `set_member_balance()` → `trace.insert(key: get_epoch_plus_k(), ...)`.
3. Delegator never calls `claim_rewards()`, so `entry_to_claim_from` stays at 0 and the trace grows unboundedly.
4. After sufficiently many epochs (trace length = N), delegator calls `claim_rewards()`.
5. `calculate_rewards()` enters the loop with `entry_to_claim_from = 0` and `pool_member_trace_length = N`. Each iteration reads from storage twice (trace entry + sigma lookup). The transaction runs out of gas and reverts.
6. All subsequent `claim_rewards()` calls also revert. The delegator's accumulated yield is permanently frozen.
