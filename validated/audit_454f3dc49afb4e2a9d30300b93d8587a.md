### Title
Missing Caller Authentication on `update_rewards` Allows Any Address to Permanently Freeze Staker Yield - (File: src/staking/staking.cairo)

### Summary
The `update_rewards` function in `StakingRewardsManagerImpl` is specified to be callable only by the Starkware sequencer, but the implementation contains no caller check. Any unprivileged address can call it with `disable_rewards: true` every block, consuming the single per-block reward slot and permanently preventing the legitimate sequencer from distributing rewards to a staker.

### Finding Description
The protocol specification explicitly states that `update_rewards` has access control restricted to "Only starkware sequencer": [1](#0-0) 

However, the implementation at `src/staking/staking.cairo` contains no such check. The function only validates that the contract is unpaused, that the block number has advanced past `last_reward_block`, and that the staker is active: [2](#0-1) 

The critical state mutation is at line 1485: `self.last_reward_block.write(current_block_number)`. This write occurs unconditionally before the `disable_rewards` check, meaning any caller — including an attacker — can consume the per-block reward slot with `disable_rewards: true` and no rewards will be distributed. [3](#0-2) 

When the legitimate sequencer subsequently calls `update_rewards` with `disable_rewards: false` in the same block, it reverts with `REWARDS_ALREADY_UPDATED`: [4](#0-3) 

The `IStakingRewardsManager` interface exposes this function publicly with no access-control annotation: [5](#0-4) 

### Impact Explanation
An attacker who calls `update_rewards(victim_staker, disable_rewards: true)` at the start of every block permanently prevents the sequencer from crediting that staker with consensus rewards. Because `last_reward_block` is a single global slot (not per-staker), a single front-run per block is sufficient to zero out all reward accrual for the targeted staker indefinitely. This constitutes **permanent freezing of unclaimed yield** for the targeted staker and their delegators.

### Likelihood Explanation
The entry point is a public, permissionless function callable by any Starknet address with no token approval or stake required. The only cost is the gas for the call. A motivated attacker targeting a high-value staker can automate this at negligible cost. Likelihood is **High**.

### Recommendation
Add a caller check at the top of `update_rewards` that asserts `get_caller_address()` equals the registered Starkware sequencer address, consistent with the specification's stated access control. Store the sequencer address in contract storage during initialization and expose a governance-controlled setter for it.

### Proof of Concept
1. Staker Alice stakes STRK and waits K epochs for her balance to become effective.
2. Consensus rewards are active (`is_pre_consensus()` returns false).
3. Attacker Eve calls `update_rewards(alice_staker_address, disable_rewards: true)` at block N. The call succeeds: `last_reward_block` is set to N, no rewards are distributed.
4. The Starkware sequencer calls `update_rewards(alice_staker_address, disable_rewards: false)` at block N. The call reverts with `REWARDS_ALREADY_UPDATED` because `current_block_number (N) > last_reward_block (N)` is false.
5. Eve repeats step 3 at every subsequent block.
6. Alice's `unclaimed_rewards_own` never increases. Her delegators' pool reward trace is never updated. All yield is permanently frozen. [6](#0-5)

        fn update_rewards(
            ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
        ) {
            self.general_prerequisites();
            let current_block_number = starknet::get_block_number();
            assert!(
                current_block_number > self.last_reward_block.read(),
                "{}",
                Error::REWARDS_ALREADY_UPDATED,
            );

            // Assert staker exists and active.
            // Staker is considered to exist from the moment of `stake` (when `InternalStakerInfo`
            // struct is created) until the calling to `unstake_action` (when `InternalStakerInfo`
            // struct is deleted).
            // Staker remains active until the intent period begins, i.e. K epochs after
            // `unstake_intent` is called.
            let staker_info = self.internal_staker_info(:staker_address);
            let curr_epoch = self.get_current_epoch();
            assert!(
                self.is_staker_active(:staker_address, epoch_id: curr_epoch),
                "{}",
                Error::INVALID_STAKER,
            );

            let staker_pool_info = self.staker_pool_info.entry(staker_address).as_non_mut();
            let (staker_total_strk_balance, staker_total_btc_balance) = self
                .get_staker_total_strk_btc_balance_at_epoch(
                    :staker_address, :staker_pool_info, epoch_id: curr_epoch,
                );
            // Assert staker has non-zero balance.
            // Staker exists with zero balance for the first K epochs after `stake`, then the stake
            // becomes effective.
            assert!(staker_total_strk_balance.is_non_zero(), "{}", Error::INVALID_STAKER);

            // Update last block rewards.
            self.last_reward_block.write(current_block_number);

            if disable_rewards || self.is_pre_consensus() {
                return;
            }

            // Get current block data and update rewards.
            let reward_supplier_dispatcher = self.reward_supplier_dispatcher.read();
            let (strk_block_rewards, btc_block_rewards) = self
                .calculate_block_rewards(:reward_supplier_dispatcher, :curr_epoch);
            self
                ._update_rewards(
                    :staker_address,
                    strk_total_rewards: strk_block_rewards,
                    btc_total_rewards: btc_block_rewards,
                    strk_total_stake: staker_total_strk_balance,
                    btc_total_stake: staker_total_btc_balance,
                    :staker_info,
                    :staker_pool_info,
                    :reward_supplier_dispatcher,
                    :curr_epoch,
                );
        }
