### Title
Unbounded Iteration Over Append-Only `stakers` Vec in `get_stakers` Causes Permanent DoS of Consensus Interface - (File: src/staking/staking.cairo)

### Summary
The `stakers` storage vector in `staking.cairo` is append-only: every new staker is pushed to it on `stake()`, but stakers are **never removed** from it even after fully unstaking. The `get_stakers` function iterates over the **full range** of this vector on every call. As the protocol accumulates stakers over time (including all historical, fully-exited stakers), this unbounded iteration will eventually exceed Starknet's per-call step/gas limits, permanently bricking the consensus interface.

### Finding Description

Every call to `stake()` appends the new staker address to `self.stakers`: [1](#0-0) 

The `get_stakers` consensus function then iterates over the **entire** vector unconditionally via `into_iter_full_range()`: [2](#0-1) 

Stakers that have completed `unstake_action` are filtered out by `is_staker_active`, but the loop still **visits every historical entry** before skipping them. The `stakers` vec is never pruned. This is confirmed by the `DiverseStakerVecFlow` test, which shows that a staker who has completed `staker_exit_action` remains in the vec: [3](#0-2) 

A secondary unbounded loop also exists in `calculate_rewards` in `pool.cairo`, where the code itself acknowledges the risk: [4](#0-3) 

### Impact Explanation

`get_stakers` is the primary interface used by the consensus layer to retrieve the active validator set for a given epoch. If this call exceeds Starknet's per-call resource limits (steps/gas), the consensus layer cannot determine the validator set. This constitutes a permanent, irrecoverable DoS of the consensus interface — no upgrade path exists to shrink the `stakers` vec without a contract migration.

**Allowed impact match:** Medium — Griefing with no profit motive but damage to users or protocol; Unbounded gas consumption.

### Likelihood Explanation

The Starknet staking protocol is designed for long-term operation. As the network grows and stakers rotate in and out over months/years, the `stakers` vec grows monotonically. No attacker action is required — organic protocol usage alone will eventually trigger this. An adversary could accelerate it by registering many stakers (each paying `min_stake`) and then unstaking them, permanently bloating the vec at the cost of only the exit wait window.

### Recommendation

Two complementary fixes:

1. **Remove exited stakers from the vec**: When `unstake_action` completes, remove the staker's address from `self.stakers`. This requires a swap-and-pop pattern on the storage vec.

2. **Alternatively, use a paginated interface**: Replace `get_stakers(epoch_id)` with a paginated version `get_stakers(epoch_id, offset, limit)` so callers can retrieve the validator set in bounded chunks.

### Proof of Concept

1. Register N stakers (e.g., N = 10,000) each with `min_stake` STRK.
2. Have all N stakers call `unstake_intent()`, wait for `exit_wait_window`, then call `unstake_action()`.
3. All N stakers are now fully exited, but `self.stakers` still contains all N addresses.
4. Call `get_stakers(current_epoch)`. The function executes `into_iter_full_range()` over all N entries, calling `is_staker_active` and `get_staker_staking_power_at_epoch` for each — both of which involve storage reads and trace lookups.
5. At sufficient N, the call exceeds Starknet's step limit and reverts, permanently preventing consensus from reading the validator set.
