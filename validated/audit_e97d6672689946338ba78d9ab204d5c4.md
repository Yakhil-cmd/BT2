### Title
Attached NEAR Deposit Permanently Locked in `rlp_execute` on Early-Return Paths — (File: `runtime/near-wallet-contract/implementation/wallet-contract/src/lib.rs`)

---

### Summary

The `rlp_execute` function in the `WalletContract` is marked `#[payable]` and therefore accepts an attached NEAR deposit from any caller. However, two code paths inside the function return `PromiseOrValue::Value(...)` — a successful execution outcome — without refunding the deposit. Because the NEAR runtime only auto-refunds deposits when a receipt **fails** (panics), any deposit attached to these paths is permanently locked inside the wallet contract's account balance with no recovery mechanism.

---

### Finding Description

`rlp_execute` is the sole public entry point of the wallet contract, deployed as a global contract for every ETH-implicit account on NEAR.

```rust
#[payable]
pub fn rlp_execute(
    &mut self,
    target: AccountId,
    tx_bytes_b64: String,
) -> PromiseOrValue<ExecuteResponse> {
    if self.has_in_flight_tx {                          // ← Path 1
        return PromiseOrValue::Value(ExecuteResponse {
            success: false, ...
        });
    }
    ...
    match result {
        Ok(promise) => { ... PromiseOrValue::Promise(promise) }
        Err(Error::Relayer(_)) if signer == self => { ... PromiseOrValue::Promise(...) }
        Err(e) => PromiseOrValue::Value(e.into()),      // ← Path 2
    }
}
```

**Path 1 — `has_in_flight_tx` guard** (lines 97–105): When a prior transaction is still in flight, the function returns a successful `Value` immediately. No refund of `env::attached_deposit()` is issued.

**Path 2 — `inner_rlp_execute` error** (line 126): When `inner_rlp_execute` returns any error that is not `Error::Relayer` (e.g., `Error::User`, `Error::AccountNonceExhausted`, `Error::AccountId`), the function again returns a successful `Value`. The `CallerDeposit` object is constructed inside `inner_rlp_execute` but is only threaded through the promise chain to `rlp_execute_callback`; it is never reached in this error branch.

The NEAR runtime's deposit-refund rule, confirmed in `runtime/runtime/src/lib.rs`:

```rust
let deposit_refund = if result.result.is_err() { total_deposit } else { Balance::ZERO };
```

means a successful return value retains the deposit in the contract. The `CallerDeposit` mechanism only issues a refund inside `rlp_execute_callback` when the downstream cross-contract call fails — it provides no coverage for the two early-return paths above.

Three additional deposit-loss paths exist in the callback functions where `caller_deposit` is silently dropped on early returns: `address_check_callback` on `PromiseResult::Failed` (line 143) and on the non-self-signer registrar hit (line 168), and `nep_141_storage_balance_callback` on `PromiseResult::Failed` (line 205) and on a non-`FunctionCall` action (line 246).

---

### Impact Explanation

The wallet contract is deployed as a **global contract** for every ETH-implicit account on NEAR mainnet. Any relayer that attaches a non-zero NEAR deposit to `rlp_execute` while `has_in_flight_tx` is `true`, or while submitting a transaction that triggers a user/nonce/account-ID error, will have that deposit permanently locked inside the wallet contract. There is no withdrawal function, no admin escape hatch, and no protocol-level recovery path. The corrupted value is the exact yoctoNEAR amount of `env::attached_deposit()` at the time of the call.

---

### Likelihood Explanation

The `has_in_flight_tx` path is reachable by any unprivileged external caller: a relayer racing with another relayer, or a batch transaction where the second action arrives while the first is still resolving (as demonstrated by the existing `test_simultaneous_transactions` test). The `Err(e)` path is reachable whenever a relayer submits a transaction with an invalid nonce or malformed payload — both common operational conditions. No special privilege is required; the attacker-controlled input is the `attached_deposit` field of the NEAR function-call action.

---

### Recommendation

In both early-return paths of `rlp_execute`, explicitly refund `env::attached_deposit()` to `env::predecessor_account_id()` before returning, mirroring the pattern already used in `rlp_execute_callback`. Alternatively, remove the `#[payable]` attribute and reject calls that carry a non-zero deposit, since the function's purpose is to relay user-signed Ethereum transactions, not to hold NEAR.

Apply the same fix to the three callback early-return paths where `caller_deposit` is silently dropped.

---

### Proof of Concept

1. Deploy the wallet contract (already done globally for ETH-implicit accounts).
2. Submit a first `rlp_execute` call that succeeds and sets `has_in_flight_tx = true`.
3. Before the first transaction's receipts resolve, submit a second `rlp_execute` call **with a non-zero attached deposit** (e.g., 1 NEAR).
4. The second call hits the `has_in_flight_tx` guard at line 97, returns `PromiseOrValue::Value(ExecuteResponse { success: false, ... })` — a successful receipt execution.
5. The runtime sees `result.result.is_ok()`, sets `deposit_refund = Balance::ZERO`, and the 1 NEAR deposit is credited permanently to the wallet contract's balance.
6. No withdrawal function exists; the funds are irrecoverable.

    pub fn rlp_execute_callback(
        &mut self,
        caller_deposit: Option<CallerDeposit>,
    ) -> ExecuteResponse {
        self.has_in_flight_tx = false;
        let n = env::promise_results_count();

        if n == 0 {
            // `rlp_execute_callback` is called directly in the case of an emulated self-transfer.
            return ExecuteResponse { success: true, success_value: None, error: None };
        } else if n > 1 {
            return ExecuteResponse {
                success: false,
                success_value: None,
                error: Some(format!(
                    "Invariant violation: this callback comes after a single promise. n={n}"
                )),
            };
        }

        match env::promise_result(0) {
            PromiseResult::Failed => {
                // The cross-contract call failed, refund the caller if needed
                if let Some(CallerDeposit { account_id, yocto_near }) = caller_deposit {
                    let refund_promise = env::promise_batch_create(&account_id);
                    env::promise_batch_action_transfer(
                        refund_promise,
                        NearToken::from_yoctonear(yocto_near.into()),
                    );
                }

                ExecuteResponse {
                    success: false,
                    success_value: None,
                    error: Some("Failed Near promise".into()),
                }
            }
            PromiseResult::Successful(value) => {
                ExecuteResponse { success: true, success_value: Some(value), error: None }
            }
        }

/// A data type to keep track of the deposit given by an external caller.
/// This allows us to refund the caller's deposit if the cross-contract call fails.
#[derive(Debug, PartialEq, Eq, Clone, serde::Serialize, serde::Deserialize)]
pub struct CallerDeposit {
    pub account_id: AccountId,
    pub yocto_near: NonZeroU128,

impl CallerDeposit {
    pub fn new(context: &ExecutionContext) -> Option<Self> {
        // Only track for external (non-self) callers
        if context.current_account_id == context.predecessor_account_id {
            return None;
        }

        NonZeroU128::new(context.attached_deposit.as_yoctonear()).map(|yocto_near| Self {
            account_id: context.predecessor_account_id.clone(),
            yocto_near,
        })
    }
}
