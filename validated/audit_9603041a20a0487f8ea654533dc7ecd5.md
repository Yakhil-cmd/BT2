### Title
Gateway L2 Gas Price Admission Check Uses Stale `l2_gas_price` Instead of `next_l2_gas_price`, Causing Incorrect Mempool Admission Decisions - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The gateway's stateful admission check for L2 gas price compares a transaction's `max_price_per_unit` against the **current committed block's** `l2_gas_price`. However, the next block that will actually execute the transaction uses `next_l2_gas_price` (the EIP-1559-adjusted price stored in the same block header) as its actual L2 gas price. These two values diverge whenever blocks are not exactly at the gas target. The code itself acknowledges the bug with a TODO comment: `// TODO(Arni): getnext_l2_gas_price from the block header.`

### Finding Description

`validate_resource_bounds` in `StatefulTransactionValidator` reads the reference price for the admission threshold as follows:

```rust
// TODO(Arni): getnext_l2_gas_price from the block header.
let previous_block_l2_gas_price = self
    .gateway_fixed_block_state_reader
    .get_block_info()
    .await?
    .gas_prices
    .strk_gas_prices
    .l2_gas_price;   // ← reads l2_gas_price of block N
``` [1](#0-0) 

`GatewayFixedBlockSyncStateClient::get_block_info_from_sync_client` populates `BlockInfo.gas_prices.strk_gas_prices.l2_gas_price` from `block_header.l2_gas_price.price_in_fri` — the price **of** block N, not the price **for** block N+1:

```rust
strk_gas_prices: GasPriceVector {
    ...
    l2_gas_price: block_header.l2_gas_price.price_in_fri.try_into()?,
    // next_l2_gas_price is never read here
},
``` [2](#0-1) 

The block header also carries a distinct field `next_l2_gas_price`, computed by the EIP-1559 formula from block N's gas consumption:

```rust
pub next_l2_gas_price: GasPrice,   // in StorageBlockHeader
``` [3](#0-2) 

Block N+1 is built with `l2_gas_price = block_N.next_l2_gas_price`. The blockifier's `check_fee_bounds` then rejects any transaction whose `max_price_per_unit < block_N+1.l2_gas_price`:

```rust
if resource_bounds.max_price_per_unit < actual_gas_price.get() {
    insufficiencies_resource.push(ResourceBoundsError::MaxGasPriceTooLow { ... });
}
``` [4](#0-3) 

The EIP-1559 adjustment formula is:

```
next_price = price ± (price × |gas_used − gas_target|) / (gas_target × denominator)
``` [5](#0-4) 

When blocks are consistently full (`gas_used > gas_target`), `next_l2_gas_price > l2_gas_price`. The gateway threshold is computed from the lower stale value, so transactions with `max_price_per_unit` in the range `[threshold_stale, next_l2_gas_price)` pass gateway admission but are rejected by `check_fee_bounds` at execution time. When blocks are consistently under-full, the opposite holds: the gateway threshold is too high and valid transactions are rejected at the gateway.

### Impact Explanation

**Incorrect admission (blocks full):** The gateway admits transactions whose `max_price_per_unit` is below the actual next-block L2 gas price. These transactions enter the mempool, consume a slot, and are then rejected by the blockifier's `check_fee_bounds` during execution. The mempool fills with unexecutable transactions, degrading throughput and wasting sequencer resources.

**Incorrect rejection (blocks under-full):** The gateway rejects transactions whose `max_price_per_unit` is above `next_l2_gas_price` but below the stale `l2_gas_price`-derived threshold. These transactions would have been valid at execution time but are denied entry. This is a liveness failure for users.

Both outcomes match the allowed impact: **High — Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.**

### Likelihood Explanation

The condition is triggered whenever the L2 gas price is not exactly at the EIP-1559 equilibrium, which is the normal operating state. During sustained high load (blocks consistently full), `next_l2_gas_price` grows above `l2_gas_price` every block. Any user who sets `max_price_per_unit` to exactly the current block's price (a natural choice) will be admitted but then rejected. The `min_gas_price_percentage` config is non-zero in production deployments, making the check active. [6](#0-5) 

### Recommendation

Replace the read of `block_info.gas_prices.strk_gas_prices.l2_gas_price` with `block_header.next_l2_gas_price` in `validate_resource_bounds`. This requires either:

1. Extending the `GatewayFixedBlockStateReader` trait to expose `next_l2_gas_price` directly, or
2. Returning the raw `BlockHeaderWithoutHash` (which already carries `next_l2_gas_price`) alongside `BlockInfo`.

The fix resolves the TODO comment and aligns the gateway admission threshold with the price the blockifier will actually enforce.

### Proof of Concept

**Setup:** Block N has `l2_gas_price = 10_000` fri and was completely full, so `next_l2_gas_price = 10_033` fri (≈ 0.33% increase). `min_gas_price_percentage = 100`.

**Step 1:** User submits an invoke transaction with `l2_gas.max_price_per_unit = 10_000`.

**Step 2:** Gateway calls `validate_resource_bounds`. It reads `previous_block_l2_gas_price = 10_000` (stale `l2_gas_price`). Threshold = `100% × 10_000 = 10_000`. Check: `10_000 >= 10_000` → **admitted**.

**Step 3:** Transaction enters the mempool.

**Step 4:** Batcher builds block N+1 with `l2_gas_price = 10_033` (= block N's `next_l2_gas_price`).

**Step 5:** Blockifier calls `check_fee_bounds`. Check: `max_price_per_unit (10_000) < actual_gas_price (10_033)` → `MaxGasPriceTooLow` error → **rejected at execution**.

**Result:** Transaction was admitted to the mempool but cannot be executed. Under sustained load, every transaction priced at exactly the current block's L2 gas price exhibits this behavior, filling the mempool with unexecutable transactions. [7](#0-6) [8](#0-7) [9](#0-8)

    let gas_delta = U256::from(gas_used.0.abs_diff(gas_target.0));

    // Calculate price change by multiplying first, then dividing. This avoids the precision loss

    let denominator =
        gas_target_u256 * U256::from(versioned_constants.gas_price_max_change_denominator);
    let price_change = (price_u256 * gas_delta) / denominator;

        if gas_used > gas_target { price_u256 + price_change } else { price_u256 - price_change };

        gas_used > gas_target && adjusted_price_u256 >= price_u256
