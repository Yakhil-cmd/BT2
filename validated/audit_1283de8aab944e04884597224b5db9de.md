### Title
Missing Output Validation in CKD Coordinator Allows Malicious Participant to Corrupt Encrypted BLS Output - (File: `src/confidential_key_derivation/protocol.rs`)

### Summary
The `do_ckd_coordinator` function blindly aggregates signature shares from all participants without any verification that each share is correctly formed. A single malicious participant can inject an arbitrary G1 element as their share, causing the coordinator to produce and return a corrupted `CKDOutput` that honest parties accept as valid — analogous to a deposit function that mints tokens without checking the output meets a minimum, allowing silent manipulation of the result.

### Finding Description
In `src/confidential_key_derivation/protocol.rs`, `do_ckd_coordinator` (lines 35–58) receives `(norm_big_y, norm_big_c)` from every participant via `recv_from_others` and unconditionally accumulates them:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
Ok(Some(ckd_output))
``` [1](#0-0) 

There is no check that each received `(Y_j, C_j)` is correctly derived from the participant's private share `x_j` and a fresh random scalar `y_j`. The `compute_signature_share` function computes the share but produces no zero-knowledge proof of correctness: [2](#0-1) 

The coordinator has no mechanism to distinguish a legitimate share from an arbitrary G1 point. The `verify_signature` function in `ciphersuite.rs` exists for BLS signature verification but is never invoked in the CKD protocol flow: [3](#0-2) 

No pairing-based or algebraic check is applied to the aggregated output before it is returned. By contrast, the ECDSA signing coordinators (both OT-based and robust) explicitly assert that the produced signature is valid before returning it — a protection entirely absent in the CKD path. [4](#0-3) 

### Impact Explanation
A malicious participant can replace their legitimate share with any arbitrary G1 element. The coordinator sums all shares and returns the corrupted `CKDOutput` as the protocol result. Honest parties receive and accept this output, which encodes an incorrect ElGamal encryption of the BLS signature. The decrypted confidential key will not equal `msk · H(pk ‖ app_id)`, making the CKD output cryptographically wrong or unusable. This matches the allowed **High** impact: *Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.*

### Likelihood Explanation
Any single participant in the CKD protocol can trigger this. No privileged access, leaked keys, or cryptographic breaks are required. The attacker simply sends a malformed `CKDOutput` message to the coordinator during the single protocol round. The attack is deterministic and requires no timing or network manipulation.

### Recommendation
Each participant should accompany their share `(Y_i, C_i)` with a zero-knowledge proof of correct formation — e.g., a Chaum-Pedersen proof demonstrating that `Y_i = y_i · G1` and `C_i = x_i · H(pk ‖ app_id) + y_i · app_pk` for the same `y_i`, without revealing `x_i` or `y_i`. The coordinator should verify all proofs before aggregating shares. This mirrors the approach used in the ECDSA signing phase where the coordinator asserts signature validity before returning the output.

### Proof of Concept
1. Malicious participant P_j computes their legitimate share but instead sends `(Y_j, C_j) = (G1::generator(), G1::identity())` to the coordinator.
2. The coordinator executes `do_ckd_coordinator`, receives the malformed share, and adds it to the running sum at lines 50–55 without rejection.
3. The final `CKDOutput` is `(Y_honest_sum + G1::generator(), C_honest_sum + G1::identity())` — shifted by an attacker-controlled offset.
4. The coordinator returns this corrupted output as `Some(ckd_output)` at line 57.
5. The caller invokes `ckd_output.unmask(app_sk)` and obtains a value that differs from `msk · H(pk ‖ app_id)` by an attacker-controlled term, producing a silently wrong confidential key with no error returned to the caller. [5](#0-4)

async fn do_ckd_coordinator(
    mut chan: SharedChannel,
    participants: ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<CKDOutputOption, ProtocolError> {
    let (mut norm_big_y, mut norm_big_c) =
        compute_signature_share(&participants, me, key_pair, app_id, app_pk, rng)?;

    // Receive everyone's inputs and add them together
    let waitpoint = chan.next_waitpoint();

    for (_, participant_output) in
        recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
    {
        norm_big_y += participant_output.big_y();
        norm_big_c += participant_output.big_c();
    }
    let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
    Ok(Some(ckd_output))
}

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}

pub fn verify_signature(
    verifying_key: &VerifyingKey,
    msg: &[u8],
    signature: &Signature,
) -> Result<(), frost_core::Error<BLS12381SHA256>> {
    let element1: G1Affine = signature.into();
    if (!element1.is_on_curve() | !element1.is_torsion_free() | element1.is_identity()).into() {
        return Err(frost_core::Error::InvalidSignature);
    }
    let element2: G2Affine = verifying_key.to_element().into();
    if (!element2.is_on_curve() | !element2.is_torsion_free() | element2.is_identity()).into() {
        return Err(frost_core::Error::MalformedVerifyingKey);
    }

    // Concatenate the master public key (96 bytes) in the hash computation
    // H(pk || app_id) when H is a random oracle
    let base1 = hash_app_id_with_pk(verifying_key, msg).into();
    let base2 =
        <<BLS12381SHA256 as frost_core::Ciphersuite>::Group as frost_core::Group>::generator()
            .into();

    if blstrs::pairing(&base1, &element2).eq(&blstrs::pairing(&element1, &base2)) {
        Ok(())
    } else {
        Err(frost_core::Error::InvalidSignature)
    }
}
