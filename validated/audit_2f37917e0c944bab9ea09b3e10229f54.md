Audit Report

## Title
Single Principal Can Exhaust Global `MAX_CONCURRENT_PENDING_REQUESTS` Queue, DoSing All Withdrawals — (`rs/bitcoin/ckbtc/minter/src/updates/retrieve_btc.rs`)

## Summary

`retrieve_btc_with_approval` enforces a single global cap of 5,000 pending withdrawal requests with no per-principal sub-limit. The `retrieve_btc_guard` only prevents the same `Account` from having two **concurrent in-flight** calls simultaneously; it is dropped as soon as each call returns. A single funded attacker can sequentially (or in batches across subaccounts) fill the entire queue, causing every subsequent legitimate caller to receive `TemporarilyUnavailable` until the minter drains all 5,000 requests.

## Finding Description

`retrieve_doge_with_approval` in the ckDOGE minter delegates directly to `retrieve_btc_with_approval`: [1](#0-0) 

Inside `retrieve_btc_with_approval`, the guard is keyed on `Account { owner: caller, subaccount: args.from_subaccount }`: [2](#0-1) 

The guard implementation (`Guard<RetrieveBtcUpdates>`) inserts the account into `state.retrieve_btc_accounts` on creation and removes it on `Drop`: [3](#0-2) 

`MAX_CONCURRENT = 100` in the guard limits only the number of **simultaneously executing** async calls, not the number of requests that have already been enqueued and returned `Ok`. Once a call returns successfully, the guard is dropped and the same account (or any other account/subaccount) can immediately submit another request.

The only global throttle on the pending queue is: [4](#0-3) 

`count_incomplete_retrieve_btc_requests` aggregates across all principals with no per-principal breakdown: [5](#0-4) 

Each successful call appends to `pending_retrieve_btc_requests` via `accept_retrieve_btc_request`: [6](#0-5) 

`retrieve_btc_accounts` is a `BTreeSet<Account>` used only as an in-flight concurrency lock, not as a per-principal pending-queue counter: [7](#0-6) 

**Attack sequence:**
1. Attacker acquires `5000 × retrieve_doge_min_amount` ckDOGE.
2. Attacker calls `icrc2_approve` + `retrieve_doge_with_approval` 5,000 times sequentially (or in batches of up to 100 using different subaccounts to stay under the guard's `MAX_CONCURRENT` limit).
3. Each call burns ckDOGE, passes the global check (counter < 5,000), and enqueues a `RetrieveBtcRequest`.
4. Once the counter reaches 5,000, every subsequent call from any principal returns `TemporarilyUnavailable("too many pending retrieve_btc requests")`.
5. The attacker's ckDOGE is burned but they receive DOGE back once the minter processes the queue. Net cost: 5,000 × `LEDGER_TRANSFER_FEE`.

## Impact Explanation

This is a complete withdrawal DoS on the ckDOGE chain-fusion bridge. All users are blocked from converting ckDOGE back to DOGE for the entire duration of queue drainage. On the Dogecoin network, with its characteristically slow confirmation times, this window can be extended significantly. This matches the allowed High impact: **"Significant Chain Fusion, ck-token, ledger … security impact with concrete user or protocol harm."**

## Likelihood Explanation

No privileged access is required. The attacker only needs a sufficient ckDOGE balance (obtainable by depositing DOGE) and the ability to call a public update endpoint. The economic cost is only ledger transfer fees — the attacker recovers their principal. The attack is repeatable: once the queue drains, it can be refilled immediately.

## Recommendation

Introduce a per-principal pending request limit (e.g., 10–50 requests per `owner` principal) enforced inside `retrieve_btc_with_approval` before the global check. Track per-principal pending counts in `CkBtcMinterState` alongside the global counter, incrementing on `accept_retrieve_btc_request` and decrementing on finalization/cancellation. This prevents any single actor from monopolising the global queue while preserving the global cap as a secondary safety valve.

## Proof of Concept

State-machine test sketch:

```rust
// Fill the queue from a single attacker principal using sequential calls
for i in 0..MAX_CONCURRENT_PENDING_REQUESTS {
    icrc2_approve(attacker, min_amount, minter_id);
    let result = minter.retrieve_doge_with_approval(
        attacker,
        RetrieveDogeWithApprovalArgs {
            amount: min_amount,
            address: valid_doge_addr.clone(),
            from_subaccount: None,
        },
    );
    assert!(result.is_ok(), "request {} failed: {:?}", i, result);
}

// Legitimate user is now blocked
let victim_result = minter.retrieve_doge_with_approval(
    victim,
    RetrieveDogeWithApprovalArgs {
        amount: min_amount,
        address: valid_doge_addr.clone(),
        from_subaccount: None,
    },
);
assert_eq!(
    victim_result,
    Err(RetrieveDogeWithApprovalError::TemporarilyUnavailable(
        "too many pending retrieve_btc requests".to_string()
    ))
);
```

```rust
impl<PR: PendingRequests> Guard<PR> {
    /// Attempts to create a new guard for the current block. Fails if there is
    /// already a pending request for the specified [principal] or if there
    /// are at least [MAX_CONCURRENT] pending requests.
    pub fn new(account: Account) -> Result<Self, GuardError> {
        mutate_state(|s| {
            let accounts = PR::pending_requests(s);
            if accounts.contains(&account) {
                return Err(GuardError::AlreadyProcessing);
            }
            if accounts.len() >= MAX_CONCURRENT {
                return Err(GuardError::TooManyConcurrentRequests);
            }
            accounts.insert(account);
            Ok(Self {
                account,
                _marker: PhantomData,
            })
        })
    }
}

impl<PR: PendingRequests> Drop for Guard<PR> {
    fn drop(&mut self) {
        mutate_state(|s| PR::pending_requests(s).remove(&self.account));
    }
}
```

```rust
pub fn accept_retrieve_btc_request<R: CanisterRuntime>(
    state: &mut CkBtcMinterState,
    request: RetrieveBtcRequest,
    runtime: &R,
) {
    record_event(
        EventType::AcceptedRetrieveBtcRequest(request.clone()),
        runtime,
    );
    state.pending_retrieve_btc_requests.push(request.clone());
    if let Some(account) = request.reimbursement_account {
        state
            .retrieve_btc_account_to_block_indices
            .entry(account)
            .and_modify(|entry| entry.push(request.block_index))
            .or_insert(vec![request.block_index]);
    }
```
