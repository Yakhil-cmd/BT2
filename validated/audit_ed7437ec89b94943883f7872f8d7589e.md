Audit Report

## Title
Information Disclosure of Non-Disclosed Rule Identifiers via Unguarded Query Calls — (`rs/boundary_node/rate_limits/canister/getter.rs`, `canister.rs`, `confidentiality_formatting.rs`)

## Summary
The `inspect_message` hook in `canister.rs` is the sole access-control gate for `get_config`, but it is only invoked for ingress (update) messages on the IC — never for regular query calls. All three read methods (`get_config`, `get_rule_by_id`, `get_rules_by_incident_id`) are annotated `#[query]`, so any anonymous caller can invoke them as non-replicated queries, bypassing `inspect_message` entirely. The confidentiality formatters applied on the query path only null out `rule_raw` and `description`; `rule_id`, `incident_id`, `added_in_version`, and `removed_in_version` are always returned, even for non-disclosed rules.

## Finding Description
`canister.rs` defines `REPLICATED_QUERY_METHOD = "get_config"` and guards it in `inspect_message` so that only `FullAccess` or `FullRead` callers may invoke it as a replicated query. However, `get_config` (line 110), `get_rule_by_id` (line 123), and `get_rules_by_incident_id` (line 136) are all `#[query]` methods. On the IC, `inspect_message` is never called for non-replicated query calls; the guard is therefore completely ineffective for the query path.

When an unprivileged caller reaches `ConfigGetter::get`, the code checks `is_authorized_viewer` and, for `RestrictedRead`, calls `self.formatter.format(config)`. `ConfigConfidentialityFormatter::format` (lines 17–28 of `confidentiality_formatting.rs`) only sets `rule.description = None` and `rule.rule_raw = None` for non-disclosed rules; `rule.id` and `rule.incident_id` are untouched. `RuleConfidentialityFormatter::format` (lines 34–42) does the same, additionally leaving `added_in_version` and `removed_in_version` intact.

The unit test at `getter.rs` lines 358–384 explicitly asserts that a `RestrictedRead` response for a non-disclosed rule contains a populated `rule_id` and `incident_id`. The test at lines 500–520 confirms that `added_in_version` and `removed_in_version` are also returned for non-disclosed rules via `IncidentGetter`.

Exploit chain:
1. Call `get_config(None)` as a query → obtain `current_version` and all `(rule_id, incident_id)` pairs, including non-disclosed rules.
2. Iterate `get_config(Some(v))` for `v = 1..current_version` → collect the full historical set of non-disclosed rule UUIDs and their incident groupings.
3. Call `get_rules_by_incident_id(incident_id)` for each collected `incident_id` → obtain `added_in_version` and `removed_in_version` for every rule in that incident.

No authentication, no keys, no special tooling required.

## Impact Explanation
An anonymous caller learns the complete structural topology of all rate-limit rules, including non-disclosed (active security) ones: every rule UUID, incident grouping, and the exact version range during which each rule was active. This constitutes a meaningful information disclosure against the boundary-node rate-limit infrastructure, enabling an attacker to correlate observed enforcement-behavior changes at boundary nodes with known rule lifetimes, narrowing the search space for probing the actual rule content. This matches the Medium allowed impact class: a moderate security impact against boundary/API infrastructure, achievable without any privileges.

## Likelihood Explanation
The attack requires only the ability to send IC query calls, which is available to any internet user via the public IC API. It is fully deterministic, requires at most `current_version + N` query calls, and is trivially repeatable. No victim interaction, no special conditions, and no cost beyond network access.

## Recommendation
1. In `ConfigConfidentialityFormatter::format` and `RuleConfidentialityFormatter::format`, also null out `rule_id` (replace with a placeholder/empty value) and `incident_id` for non-disclosed rules when the caller is `RestrictedRead`, or filter non-disclosed rules out of the response entirely.
2. Add an explicit access-level check inside `get_config`, `get_rule_by_id`, and `get_rules_by_incident_id` (not only in `inspect_message`) so that the guard is effective for both query and update call paths.
3. Consider whether `get_config` should remain a `#[query]` method at all, or be changed to a `#[update]` or replicated-query-only method, given the intended access restrictions.

## Proof of Concept
The existing unit test at `getter.rs` lines 358–384 already proves the precondition: a `RestrictedRead` caller receives a response where `rule_id` and `incident_id` are non-null for a non-disclosed rule (`disclosed_at: None`). The test at lines 498–520 proves that `get_rules_by_incident_id` also returns `added_in_version` and `removed_in_version` for non-disclosed rules to a `RestrictedRead` caller. No additional test infrastructure is needed; the existing tests are sufficient to demonstrate the full information disclosure.
