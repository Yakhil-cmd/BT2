Looking at the key files to confirm the complete attack path:

### Title
Unauthenticated Farming Reward Payout Address Change via No-Op `PrivateRoute` and Unguarded `FarmManageFarmingRewards` — (`packages/core/src/components/PrivateRoute/PrivateRoute.tsx`, `packages/gui/src/components/farm/FarmManageFarmingRewards.tsx`)

### Summary

`PrivateRoute` is a complete no-op that provides zero authentication enforcement. The `dashboard/farm/*` route in `AppRouter` has no auth guard. `FarmManageFarmingRewards` calls `useSetRewardTargetsMutation` with attacker-controlled addresses and no fingerprint/session check. Because `set_reward_targets` is a farmer-daemon RPC that does not require a wallet fingerprint, the mutation succeeds even when no key is logged in, redirecting all future XCH farming rewards to attacker-controlled addresses.

### Finding Description

**`PrivateRoute` is a stub with no logic:** [1](#0-0) 

It unconditionally renders `<Route {...props} />` — no fingerprint check, no redirect, no guard of any kind.

**`AppRouter` places `dashboard/farm/*` behind no authentication gate:** [2](#0-1) 

The only wrapper is `LayoutDashboard` (a visual shell). `AppState` only blocks on keyring-locked, needs-migration, or daemon-disconnected states — it never checks whether a fingerprint is selected. [3](#0-2) 

`AppAutoLogin` navigates to `/dashboard/wallets` on auto-login but does **not** block direct navigation to other routes when no fingerprint is active.

**`FarmManageFarmingRewards` submits `setRewardTargets` with no auth check:** [4](#0-3) 

`handleSubmit` calls `setRewardTargets({ farmerTarget, poolTarget })` directly. There is no check against `AuthContext.fingerprint`, no confirmation dialog, and no passphrase re-prompt.

**The farmer RPC itself does not require a wallet fingerprint:** [5](#0-4) 

`set_reward_targets` is a farmer-daemon command independent of the wallet daemon. It succeeds regardless of whether any key is logged in.

### Impact Explanation

An attacker who can interact with the running app (physical access while the screen is unattended, or a local process that can drive the Electron renderer) while no fingerprint is selected can:

1. Navigate directly to `/#/dashboard/farm`
2. Click "Manage Farming Rewards" to open `FarmManageFarmingRewards`
3. Enter attacker-controlled XCH addresses for `farmerTarget` and `poolTarget`
4. Submit — `setRewardTargets` fires against the live farmer daemon with no auth check

All subsequent block rewards (farmer reward 0.25 XCH + pool reward 1.75 XCH for solo, or 0.125 XCH for pooled plots) are permanently redirected until the legitimate user notices and reverts. This is a **Critical** impact: unauthorized payout address change affecting XCH farming rewards.

### Likelihood Explanation

The precondition (app open, keyring unlocked, no fingerprint selected) is the normal state between daemon startup and key selection, and also occurs any time the user is on the `SelectKey` screen. The attack requires only local UI interaction — no credentials, no cryptographic material, no special tooling.

### Recommendation

1. **Fix `PrivateRoute`**: Implement a real auth guard that reads `AuthContext.fingerprint` and redirects to `/` when undefined.
2. **Guard `FarmManageFarmingRewards`**: Check `fingerprint` from `useAuth()` before enabling the form or calling `setRewardTargets`.
3. **Add a confirmation dialog** with passphrase re-prompt before any reward-target mutation, consistent with how other sensitive wallet operations are protected.

### Proof of Concept

```
1. Start the Chia node + farmer daemon (keyring unlocked, no key selected in GUI).
2. In the Electron app, the user is on the SelectKey screen (fingerprint = undefined).
3. In DevTools console or via direct URL navigation: window.location.hash = '#/dashboard/farm'
4. The Farm page renders without any auth challenge.
5. Click the "Manage Farming Rewards" button to open FarmManageFarmingRewards.
6. Enter attacker-controlled addresses in both fields (valid bech32m XCH addresses).
7. Click Save.
8. Assert: useSetRewardTargetsMutation fires with attacker addresses; farmer daemon
   confirms success; get_reward_targets now returns the attacker addresses.
9. All future block rewards are paid to the attacker until manually reverted.
```
