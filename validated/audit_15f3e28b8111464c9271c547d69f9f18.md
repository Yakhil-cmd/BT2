### Title
Unauthorized Invoke Transaction Admitted to Mempool via `skip_stateful_validations` Signature Bypass — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator unconditionally skips the `__validate__` entry-point (signature verification) for any invoke transaction with `nonce == 1` whose `sender_address` has any transaction in the mempool or a recent block. Because `sender_address` is a freely-settable field supplied by the submitter, an unprivileged attacker can craft an invoke transaction targeting a victim's address with a garbage signature and have it admitted to the mempool without any signature check.

### Finding Description

`skip_stateful_validations` in `crates/apollo_gateway/src/stateful_transaction_validator.rs` returns `true` (skip validation) when all three conditions hold:

1. The transaction is an `Invoke` with `nonce == Nonce(Felt::ONE)`.
2. The on-chain account nonce is `Nonce(Felt::ZERO)` (account not yet deployed).
3. `mempool_client.account_tx_in_pool_or_recent_block(tx.sender_address())` returns `true`. [1](#0-0) 

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` sets `execution_flags.validate = false`, which causes `validate_tx` in the blockifier to return `Ok(None)` immediately without calling the account's `__validate__` entry point: [2](#0-1) [3](#0-2) 

The only guard that fires before this skip is `validate_by_mempool`, which checks nonce gaps and fee escalation — it does **not** verify the signature: [4](#0-3) 

The stateless validator checks signature **length** but not cryptographic validity: [5](#0-4) 

The `account_tx_in_pool_or_recent_block` check only confirms that *some* transaction from that address exists in the mempool — it does not verify that the submitter of the new invoke is the account owner: [6](#0-5) 

**Attack scenario:**

1. Victim submits a `deploy_account` transaction (nonce=0) to the mempool. `account_tx_in_pool_or_recent_block(victim_address)` now returns `true`.
2. Attacker crafts an invoke transaction with `sender_address = victim_address`, `nonce = 1`, arbitrary `calldata`, and a garbage `signature`.
3. Gateway stateless validation passes (signature length is within bounds).
4. `validate_state_preconditions` passes (nonce=1 is within the allowed gap from account nonce=0).
5. `validate_by_mempool` passes (no duplicate nonce yet, fee escalation not triggered).
6. `skip_stateful_validations` returns `true` because all three conditions are met.
7. `run_validate_entry_point` is called with `skip_validate=true` → `__validate__` is **never called** → the garbage-signature invoke is admitted to the mempool. [7](#0-6) 

### Impact Explanation

Once the attacker's unsigned invoke occupies the victim's nonce=1 slot in the mempool:

- The victim's own legitimate invoke with nonce=1 is rejected by the mempool as `DuplicateNonce` (unless the victim pays a higher tip for fee escalation, which may not be possible if the attacker keeps escalating).
- The deploy_account + invoke UX flow — the explicit purpose of the `skip_stateful_validations` feature — is broken for the victim.
- The attacker can repeat this for every new account deployment observed in the mempool, at negligible cost (no valid signature required, only a valid-length signature array).

This matches the impact category: **High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

### Likelihood Explanation

The attack window is the period between a victim's `deploy_account` entering the mempool and being committed to a block. During this window, any observer of the mempool (or the public gateway) can execute the attack. No privileged access is required; the only prerequisite is that the victim has submitted a `deploy_account` transaction.

### Recommendation

In `skip_stateful_validations`, replace the `account_tx_in_pool_or_recent_block` check with a check that verifies a **deploy_account** transaction specifically for the sender address exists in the mempool (not just any transaction). Additionally, consider requiring that the invoke transaction's signature is non-empty and structurally valid before skipping the `__validate__` entry point, so that at minimum a plausible signature is present.

Alternatively, the skip should only apply when the gateway itself is processing a batch of `[deploy_account, invoke]` submitted together in the same request, rather than trusting the mempool state as a proxy for account ownership.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
