### Title
Scheduled Callback Execution Transaction Created with Zero Compute Limit Due to Missing Minimum Floor on `executionEffort` - (File: `fvm/blueprints/scheduled_callback.go`)

---

### Summary

The `callbackArgsFromEvent` function in `fvm/blueprints/scheduled_callback.go` reads the `executionEffort` field from the `PendingExecution` event and uses it directly as the `ComputeLimit` for the callback execution transaction. Only an upper-bound cap is enforced (capped at `DefaultMaxTransactionGasLimit`); there is no lower-bound (minimum floor) check. If a malicious callback owner schedules a transaction with `executionEffort = 0`, the execution transaction is constructed with `SetComputeLimit(0)`. The FVM treats a zero compute limit as unlimited computation. This allows a malicious callback owner to obtain unbounded execution for their callback without paying proportional fees, directly analogous to the Chainlink oracle claiming payment without performing the callback work.

---

### Finding Description

When a scheduled transaction's timestamp is reached, the `FlowTransactionScheduler` contract emits a `PendingExecution` event containing the `executionEffort` value originally specified by the callback owner at scheduling time. The Go layer reads this value in `callbackArgsFromEvent`:

```go
effort := uint64(cadenceEffort)

if effort > flow.DefaultMaxTransactionGasLimit {
    log.Warn().Uint64("effort", effort).Msg("effort is greater than max transaction gas limit, setting to max")
    effort = flow.DefaultMaxTransactionGasLimit
}

return uint64(cadenceId), uint64(effort), nil
``` [1](#0-0) 

Only an upper-bound is enforced. The resulting `effort` value (which can be `0`) is then passed directly to `SetComputeLimit`:

```go
tx, err := flow.NewTransactionBodyBuilder().
    AddAuthorizer(sc.ScheduledTransactionExecutor.Address).
    SetScript(script).
    AddArgument(encID).
    SetComputeLimit(effort).
    Build()
``` [2](#0-1) 

The identical pattern exists in the v0 system collection builder: [3](#0-2) 

The FVM explicitly treats a zero compute limit as unlimited computation, confirmed by the runtime test:

```go
{
    label:    "Zero",
    script:   gasLimitScript(100),
    gasLimit: 0,
    check: func(t *testing.T, output fvm.ProcedureOutput) {
        // gas limit of zero is ignored by runtime
        require.NoError(t, output.Err)
    },
},
``` [4](#0-3) 

The `fees` field in the `PendingExecution` event is separate from `executionEffort` and is paid upfront at scheduling time. If the `FlowTransactionScheduler` Cadence contract charges fees proportional to `executionEffort`, scheduling with `executionEffort = 0` means paying zero (or minimal) fees while obtaining unlimited computation for the callback body. Even if fees are fixed, the attacker still obtains unbounded execution resources beyond what they paid for.

---

### Impact Explanation

A malicious callback owner (any unprivileged user who can submit a scheduling transaction) can:

1. Schedule a transaction with `executionEffort = 0`, paying zero or minimal fees.
2. The `PendingExecution` event emits `executionEffort = 0`.
3. `callbackArgsFromEvent` passes `effort = 0` through with no lower-bound check.
4. The execution transaction is built with `ComputeLimit = 0`.
5. The FVM executes the callback with unlimited computation.
6. The callback body can perform arbitrarily expensive operations (e.g., creating many accounts, complex Cadence computations, EVM interactions) without paying proportional fees.

This constitutes unauthorized consumption of on-chain computation resources — a fee underpayment / resource mis-accounting impact. The callback owner receives more execution work than they paid for, directly draining execution node resources at the protocol level.

---

### Likelihood Explanation

Any unprivileged user who can call the `FlowTransactionScheduler` contract to schedule a transaction can trigger this. The entry path requires only a standard user transaction. The `executionEffort` field is user-controlled at scheduling time and flows through the Go layer without a minimum floor check. The likelihood is high once the scheduled transactions feature is live, as the attack requires no special privileges, no leaked keys, and no compromised nodes.

---

### Recommendation

Enforce a minimum floor on `effort` in `callbackArgsFromEvent` (and its v0 counterpart) before using it as `ComputeLimit`. Define a protocol constant `MinCallbackExecutionEffort` and apply it:

```go
const MinCallbackExecutionEffort = uint64(400_000) // analogous to Chainlink's resolution

effort := uint64(cadenceEffort)

if effort < MinCallbackExecutionEffort {
    effort = MinCallbackExecutionEffort
}
if effort > flow.DefaultMaxTransactionGasLimit {
    effort = flow.DefaultMaxTransactionGasLimit
}
```

Apply the same fix in `model/access/systemcollection/system_collection_v0.go` `callbackArgsFromEvent`. [5](#0-4) [6](#0-5) 

---

### Proof of Concept

1. Deploy a Cadence contract implementing the `FlowTransactionScheduler` handler interface with a callback body that performs expensive computation (e.g., a tight loop creating accounts).
2. Submit a scheduling transaction calling `FlowTransactionScheduler.schedule(...)` with `executionEffort: UInt64 = 0` and `fees` set to the minimum accepted value.
3. Wait for the scheduled timestamp to be reached.
4. Observe that the `PendingExecution` event is emitted with `executionEffort = 0`.
5. Observe that `callbackArgsFromEvent` returns `effort = 0` (no lower-bound rejection).
6. Observe that `generateExecuteCallbacksTransaction` builds a transaction with `GasLimit = 0`.
7. The FVM executes the callback with unlimited computation ("gas limit of zero is ignored by runtime"), consuming arbitrary execution resources while the callback owner paid fees corresponding to `executionEffort = 0`. [7](#0-6) [5](#0-4) [8](#0-7)

func callbackArgsFromEvent(event flow.Event) (uint64, uint64, error) {
	cadenceId, cadenceEffort, err := ParsePendingExecutionEvent(event)
	if err != nil {
		return 0, 0, err
	}

func (b *builderV0) ExecuteCallbacksTransactions(chain flow.Chain, processEvents flow.EventsList) ([]*flow.TransactionBody, error) {
	txs := make([]*flow.TransactionBody, 0, len(processEvents))
	sc := systemcontracts.SystemContractsForChain(chain.ChainID())
	env := sc.AsTemplateEnv()
	script := b.prepareExecuteScheduledTransactionTemplate(env)

	slices.SortFunc(processEvents, func(a, b flow.Event) int {
		return int(a.EventIndex - b.EventIndex)
	})

	for _, event := range processEvents {
		// skip any fee events or other events that are not pending execution events
		if !blueprints.IsPendingExecutionEvent(env, event) {
			continue
		}

		id, effort, err := b.callbackArgsFromEvent(event)
		if err != nil {
			return nil, fmt.Errorf("failed to get callback args from event: %w", err)
		}

		tx, err := flow.NewTransactionBodyBuilder().
			AddAuthorizer(sc.FlowServiceAccount.Address).
			SetScript(script).
			AddArgument(id).
			SetComputeLimit(effort).
			Build()
		if err != nil {
			return nil, fmt.Errorf("failed to construct execute callback transactions: %w", err)
		}
		txs = append(txs, tx)
	}

	return txs, nil
}
