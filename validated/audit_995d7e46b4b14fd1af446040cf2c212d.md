### Title
Pool Rewards Permanently Trapped When Delegated Balance Falls Below `min_delegation_for_rewards` — (`src/pool/pool.cairo`, `src/staking/staking.cairo`)

---

### Summary

The staking contract unconditionally transfers STRK rewards to a delegation pool contract whenever `pool_rewards > 0`, regardless of whether the pool's delegated balance meets the `min_delegation_for_rewards` threshold. Inside the pool contract, `compute_rewards_per_unit` silently returns zero when the balance is below that threshold, so the cumulative rewards index is never updated and no pool member can ever claim the transferred tokens. Those tokens are permanently frozen in the pool contract with no recovery path.

---

### Finding Description

**Step 1 — Staking contract calculates and forwards rewards unconditionally.**

In `calculate_staker_pools_rewards`, the staking contract computes `pool_rewards` proportionally to `pool_balance_curr_epoch` (a `NormalizedAmount`). The only gate before forwarding is `pool_rewards.is_non_zero()`: [1](#0-0) 

If `pool_balance_curr_epoch` is, say, `5 × 10^17` (0.5 STRK), the staking contract computes a non-zero `pool_rewards` and appends the pool to `pool_rewards_array`.

**Step 2 — `update_pool_rewards` transfers the tokens to the pool contract.** [2](#0-1) 

`send_rewards_to_delegation_pool` performs an ERC-20 transfer of `pool_rewards` STRK to the pool contract address, then calls `update_rewards_from_staking_contract` with `pool_balance = pool_balance_curr_epoch.to_native_amount(:decimals)`.

**Step 3 — Pool contract silently drops the rewards when balance is below threshold.**

Inside `compute_rewards_per_unit`, if `total_stake < min_delegation_for_rewards` (e.g., `< 10^18` for STRK), the function returns zero: [3](#0-2) 

The code comment itself acknowledges the discrepancy: *"Delegation rewards lost when pool balance is less than `min_delegation_for_rewards`. The staking contract continues to forward `pool_rewards` to the pool contract even in this case."*

Because `compute_rewards_per_unit` returns zero, the `cumulative_rewards_trace` is not advanced. Every subsequent call to `calculate_rewards` / `find_sigma` in `claim_rewards` will compute zero rewards for the affected epochs.

**Step 4 — No recovery path exists.**

The pool contract exposes no admin or sweep function. `claim_rewards` distributes only what the cumulative index accounts for. `exit_delegation_pool_action` returns only the delegated principal. The trapped STRK sits in the pool contract's ERC-20 balance forever. [4](#0-3) 

---

### Impact Explanation

STRK reward tokens are transferred into the pool contract and permanently frozen there. They cannot be claimed by any pool member, returned to the reward supplier, or recovered by any governance action. This constitutes **permanent freezing of unclaimed yield** — a High-severity impact under the allowed scope.

---

### Likelihood Explanation

The trigger condition is straightforward and reachable by any unprivileged delegator:

- For STRK pools, `min_delegation_for_rewards = 10^18` (1 STRK). Any delegator who enters with less than 1 STRK satisfies the condition.
- For BTC pools, the threshold is the token-specific `min_for_rewards` value set at pool construction. [5](#0-4) 

No special privilege is required. The delegator simply calls `enter_delegation_pool` with a sub-threshold amount, and the trap is set for every subsequent attestation epoch.

---

### Recommendation

**Short term:** In `update_pool_rewards` (or in `calculate_staker_pools_rewards`), skip forwarding rewards to a pool when `pool_balance.to_native_amount(:decimals) < min_delegation_for_rewards`. The commission portion already accrues to the staker; only the pool-member share needs to be withheld.

**Long term:** Add an invariant test asserting that the sum of all pool contract ERC-20 balances equals the sum of all claimable pool-member rewards after every attestation cycle, analogous to the existing flow-test accounting checks. [6](#0-5) 

---

### Proof of Concept

1. Alice stakes the minimum stake and opens a STRK delegation pool.
2. Bob calls `enter_delegation_pool` with `amount = 5 × 10^17` (0.5 STRK, below `min_delegation_for_rewards = 10^18`).
3. Alice attests. The staking contract computes `pool_rewards ≈ 0.5 × epoch_rewards / total_stake`, which is non-zero.
4. `update_pool_rewards` transfers those STRK tokens to the pool contract.
5. `update_rewards_from_staking_contract` calls `compute_rewards_per_unit(staking_rewards: pool_rewards, total_stake: 5×10^17)`. Because `5×10^17 < 10^18`, it returns zero. The `cumulative_rewards_trace` is not updated.
6. Bob calls `claim_rewards`. The calculation yields zero. The transferred STRK remains in the pool contract's balance indefinitely with no mechanism to retrieve it. [7](#0-6) [8](#0-7)

            // Asserts.

                        pool_member, reward_address, amount: rewards,
                    },

        }

        /// Compute the rewards for the pool trace.
        ///
        /// `staking_rewards` is in `STRK_DECIMALS` decimals.
        /// `total_stake` is in the contract's token decimals.
        /// **Note**: Delegation rewards lost when pool balance is less than
        /// `min_delegation_for_rewards`. The staking contract continues to forward
        /// `pool_rewards` to the pool contract even in this case.
        fn compute_rewards_per_unit(
            self: @ContractState, staking_rewards: Amount, total_stake: Amount,
        ) -> Index {
            // Return zero if the total stake is too small, to avoid overflow below.
            if total_stake < self.min_delegation_for_rewards.read() {
                return Zero::zero();
            }
            mul_wide_and_div(
                lhs: staking_rewards, rhs: self.staking_rewards_base_value.read(), div: total_stake,
            )
                .expect_with_err(err: InternalError::REWARDS_COMPUTATION_OVERFLOW)
        }
