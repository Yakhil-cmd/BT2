### Title
IBC Refund Permanently Locked When Smart Contract Initiates `send_to_ibc` — (File: x/cronos/keeper/evmhandlers/send_to_ibc.go)

### Summary
The EVM hook handlers for `__CronosSendToIbc` and `__CronosSendCroToIbc` derive the IBC `MsgTransfer.Sender` directly from `msg.sender` in the emitted EVM event. When the caller is a smart contract, the IBC refund address becomes the contract's cosmos-side account. If the IBC transfer times out, the refunded IBC vouchers or source tokens land at that cosmos address, which the contract has no standard mechanism to recover, permanently locking user funds.

### Finding Description

In `SendToIbcHandler.handle()`, the sender is taken verbatim from the EVM log:

```go
sender := sdk.AccAddress(senderAddress.Bytes())   // senderAddress == msg.sender in Solidity
// ...
// First, transfer IBC coin to user so that he will be the refunded address if transfer fails
if err = h.bankKeeper.SendCoins(ctx, contractAddr, sender, coins); err != nil { ... }
// Initiate IBC transfer from sender account
if err = h.cronosKeeper.IbcTransferCoins(ctx, sender.String(), recipient, coins, channelId); err != nil { ... }
``` [1](#0-0) 

`IbcTransferCoins` → `ibcSendTransfer` then sets `MsgTransfer.Sender = sender.String()`:

```go
msg := ibctransfertypes.MsgTransfer{
    Sender:   sender.String(),
    Receiver: destination,
    TimeoutTimestamp: timeoutTimestamp,   // non-zero; transfer CAN time out
    ...
}
``` [2](#0-1) 

The identical pattern exists in `SendCroToIbcHandler.Handle()`: [3](#0-2) 

When `msg.sender` in the Solidity contract is an EOA, this is correct: the user's cosmos address is the IBC sender and receives any timeout refund. But when `msg.sender` is a smart contract (e.g., a DeFi aggregator or vault calling `send_to_ibc` on behalf of depositors), the IBC `Sender` becomes the contract's cosmos address — a 20-byte address derived from the contract's EVM address. The IBC transfer module will refund to that address on timeout. The contract has no built-in path to retrieve cosmos-side bank balances; unless the developer explicitly coded a bank-precompile withdrawal, the tokens are stranded.

The `IBCConversionModule` middleware's `OnTimeoutPacket` and `OnAcknowledgementPacket` only attempt to re-convert vouchers back to EVM tokens for the `data.Sender` address: [4](#0-3) 

If `data.Sender` is a contract address, `ConvertVouchersToEvmCoins` / `OnRecvVouchers` will attempt to mint CRC21 tokens to that contract address. For source-coin (`cronos0x…`) denoms this calls `transfer_from_cronos_module` on the CRC21 contract, which transfers tokens to the contract's own EVM balance — still inaccessible to the original depositors unless the contract has explicit withdrawal logic.

### Impact Explanation

Any IBC vouchers or source-denom tokens refunded on timeout are deposited into the calling smart contract's cosmos address. Without explicit recovery logic in the contract, these assets are permanently locked. This constitutes a permanent inability for honest users to recover funds from a valid bridge/conversion flow — matching the **High** impact tier: *"Permanent or long-lived inability for honest users … to process valid … bridge/conversion flows … under normal network assumptions."*

### Likelihood Explanation

- DeFi aggregators, vaults, and router contracts routinely call `send_to_ibc` on CRC20/CRC21 tokens on behalf of users.
- IBC transfers time out under normal conditions (relayer downtime, congested destination chain). The `timeoutTimestamp` is always set to a finite value (`blockTime + IbcTimeout`).
- Contract developers following the CRC20/CRC21 standard have no documentation warning them that the IBC refund address will be their contract's cosmos address rather than the original depositor's address.

### Recommendation

1. Allow callers to pass an explicit `refundAddress` argument to `send_to_ibc` (analogous to the LiFi fix: take the recovery parameter as an argument).
2. In the EVM hook handler, validate that `senderAddress` is not a contract (check `HasContractCode`); if it is, reject the operation or require an explicit refund address.
3. Document clearly that `msg.sender` must be an EOA when calling `send_to_ibc` directly.

### Proof of Concept

1. Deploy a vault contract `Vault` that holds user CRC21 tokens and exposes `bridgeToIBC(string recipient, uint amount)` which internally calls `crc21.send_to_ibc(recipient, amount)`.
2. A user deposits tokens into `Vault` and calls `bridgeToIBC`.
3. The CRC21 contract emits `__CronosSendToIbc(msg.sender=Vault, recipient, amount)`.
4. The EVM hook sets `sender = Vault's cosmos address`, transfers coins there, and initiates `MsgTransfer{Sender: vault_cosmos_addr, Receiver: recipient, ...}`.
5. The relayer goes offline; the packet times out.
6. IBC refunds `amount` to `vault_cosmos_addr` (cosmos bank balance of `Vault`).
7. `IBCConversionModule.OnTimeoutPacket` attempts `OnRecvVouchers(ctx, coins, vault_cosmos_addr)` — for IBC vouchers this calls `ConvertVouchersToEvmCoins` which mints CRC21 tokens to `Vault`'s EVM address.
8. `Vault` has no `withdrawStrandedTokens()` function. The user's funds are permanently locked inside `Vault`'s EVM balance with no recovery path.

// OnTimeoutPacket implements the IBCModule interface
func (im IBCConversionModule) OnTimeoutPacket(
	ctx sdk.Context,
	channelVersion string,
	packet channeltypes.Packet,
	relayer sdk.AccAddress,
) error {
	err := im.app.OnTimeoutPacket(ctx, channelVersion, packet, relayer)
	// If no error on the refund
	if err == nil {
		data, err := transferTypes.UnmarshalPacketData(packet.GetData(), channelVersion, "")
		if err != nil {
			return err
		}
		denom := im.getIbcDenomFromDataForRefund(data.Token)
		if im.canBeConverted(ctx, denom) {
			if err := im.convertVouchers(
				ctx,
				data.Token.Amount,
				data.Sender,
				data.Receiver,
				denom,
				true,
			); err != nil {
				// Intentional: log and continue so the IBC refund is not blocked.
				// Sender keeps the refunded IBC vouchers and can retry conversion manually.
				im.cronoskeeper.Logger(ctx).Error(
					"failed to convert refund vouchers on timeout",
					"denom", denom,
					"sender", data.Sender,
					"error", err,
				)
			}
		}

	}
	return err
}
