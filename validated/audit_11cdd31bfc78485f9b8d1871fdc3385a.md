### Title
Missing Zero-Address Validation in `change_reward_address` / `stake()` Enables Permanent Reward Burning via Permissionless `unstake_action` — (File: `src/staking/staking.cairo`)

---

### Summary

`stake()` and `change_reward_address()` in `staking.cairo` only guard against `reward_address` being a token contract address, but impose no check that `reward_address != 0`. Because `unstake_action()` is callable by **any** address with no access restriction, a third party can trigger `send_rewards_to_staker()` after a staker has set their `reward_address` to zero, permanently burning all accumulated unclaimed rewards by transferring them to address(0). The same root cause exists in `pool.cairo`'s `enter_delegation_pool()` and `change_reward_address()`.

---

### Finding Description

**Root cause — `stake()` in `staking.cairo`:** [1](#0-0) 

Only `REWARD_ADDRESS_IS_TOKEN` is asserted. There is no `assert!(reward_address.is_non_zero(), ...)` guard. A staker can call `stake(reward_address: 0, ...)` and register with a zero reward address from the start.

**Root cause — `change_reward_address()` in `staking.cairo`:** [2](#0-1) 

Again, only the token-address check is present. A staker can call `change_reward_address(0)` at any time, silently overwriting a valid address with zero.

**Root cause — `enter_delegation_pool()` and `change_reward_address()` in `pool.cairo`:** [3](#0-2) [4](#0-3) 

Same pattern: only `REWARD_ADDRESS_IS_TOKEN` is checked; zero is accepted.

**Trigger — `unstake_action()` has no caller restriction:** [5](#0-4) 

The spec explicitly states "Any address can execute." After the exit window, any third party can call `unstake_action(staker_address)`, which internally calls `send_rewards_to_staker`.

**Transfer to address(0) — `send_rewards_to_staker()`:** [6](#0-5) 

`reward_address` is read directly from storage and passed to `checked_transfer` with no zero guard. If `reward_address` is zero, the transfer targets address(0).

---

### Impact Explanation

- If the underlying ERC20 `checked_transfer` allows transfers to address(0) (no zero-recipient guard), all accumulated `unclaimed_rewards_own` are permanently burned — **permanent loss of unclaimed yield** (High).
- If `checked_transfer` reverts on a zero recipient, `unstake_action` reverts entirely, locking the staker's principal in the contract until `change_reward_address` is called — **temporary freezing of funds** (Medium).

Both outcomes fall within the allowed impact scope.

---

### Likelihood Explanation

A staker can set `reward_address = 0` via a frontend bug, a scripting error, or a deliberate misconfiguration. Once set, the window is open: any third party can call the permissionless `unstake_action` after the exit window expires, triggering the burn or freeze without any further cooperation from the staker. The combination of an easily reachable setter (no zero guard) and a permissionless executor (`unstake_action`) makes this a realistic griefing path.

---

### Recommendation

Add a non-zero assertion in every function that accepts or stores a `reward_address`:

1. `stake()` in `staking.cairo` — assert `reward_address.is_non_zero()` before writing.
2. `change_reward_address()` in `staking.cairo` — same guard.
3. `enter_delegation_pool()` in `pool.cairo` — same guard.
4. `change_reward_address()` in `pool.cairo` — same guard.

Optionally, add a defensive zero-check inside `send_rewards_to_staker()` and `claim_rewards()` before calling `checked_transfer`, mirroring the LootBox recommendation to guard the transfer site as well as the setter.

---

### Proof of Concept

1. Staker calls `staking.change_reward_address(reward_address: 0)`.
   - Passes the only guard (`REWARD_ADDRESS_IS_TOKEN`); zero is not a token address.
   - `staker_info.reward_address` is now `0`.
2. Staker calls `staking.unstake_intent()`.
   - Sets `unstake_time = now + exit_wait_window`.
3. After the exit window, **any unprivileged address** calls `staking.unstake_action(staker_address)`.
   - No caller check; proceeds unconditionally.
   - Calls `send_rewards_to_staker` → `claim_from_reward_supplier` pulls rewards → `checked_transfer(recipient: 0, amount: rewards)`.
   - Rewards are sent to address(0) (burned) or the call reverts, freezing the principal.
