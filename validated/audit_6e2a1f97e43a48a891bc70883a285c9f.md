### Title
Pool Admin Bypasses `maxAdminSpreadFeeE6` Cap via Unchecked `setPoolBinAdditionalFees` — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary
`MetricOmmPoolFactory.setPoolBinAdditionalFees` forwards `addFeeBuyE6`/`addFeeSellE6` directly to the pool without checking them against `maxAdminSpreadFeeE6`, allowing a pool admin to set per-bin fees up to `uint16.max` (65,535 E6 = 6.5535%) regardless of the factory's configured admin fee cap.

### Finding Description
The factory enforces `maxAdminSpreadFeeE6` in `setPoolAdminFees`:

```solidity
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
```

But `setPoolBinAdditionalFees` has no equivalent guard:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

The pool's `setBinAdditionalFees` also has no cap check — it only validates the bin index range. The stored `addFeeBuyE6`/`addFeeSellE6` are then added directly to `baseFeeX64` during every swap through that bin:

```solidity
params.baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6)
```

### Impact Explanation
Traders swapping through the affected bin pay an effective fee of `baseFee + addFeeBuyE6/1e6`, where `addFeeBuyE6` can be up to 65,535 E6 (6.5535%) regardless of `maxAdminSpreadFeeE6`. If `maxAdminSpreadFeeE6 = 1,000` (0.1%), the admin can impose a 65× higher fee on a specific bin, causing direct loss of swap output to traders on every swap through that bin. The excess fee accrues to LPs in that bin (not to the admin directly), but traders receive materially less output than the documented cap permits.

### Likelihood Explanation
Any pool admin can call this immediately with no timelock, no co-signer, and no protocol approval. The only prerequisite is holding the `poolAdmin[pool]` role, which is assigned at pool creation and transferable. The attack is a single transaction.

### Recommendation
Add a cap check in `setPoolBinAdditionalFees` before forwarding to the pool:

```solidity
if (addFeeBuyE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
```

Note: `maxAdminSpreadFeeE6` is `uint24` and `addFeeBuyE6`/`addFeeSellE6` are `uint16`, so the comparison is safe without casting.

### Proof of Concept
1. Deploy factory with `maxAdminSpreadFeeE6 = 1_000` (0.1%).
2. Create a pool; pool admin calls `factory.setPoolBinAdditionalFees(pool, 0, 65535, 65535)`.
3. No revert occurs; `BinState.addFeeBuyE6 = 65535` is stored.
4. Execute a swap through bin 0; the effective fee applied is `baseFee + 65535/1e6 ≈ baseFee + 6.55%`, far exceeding the 0.1% cap.
5. Assert `amountOut` is materially less than it would be with `addFeeBuyE6 = 1_000`.
