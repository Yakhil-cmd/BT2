### Title
Protocol Unpause Conflates Admin-Paused and Active States, Leaving Pools Swap-Blocked Without Admin Cooperation - (File: metric-core/contracts/MetricOmmPoolFactory.sol)

### Summary
`protocolUnpausePool` always transitions a pool to pause level 1 (admin-paused) regardless of whether the pool was at level 0 (active) or level 1 (admin-paused) before the protocol pause. This conflates two distinct pause contexts: the protocol cannot independently restore a previously-active pool to active state after a protocol pause — it always requires an additional admin action. The pool admin thereby holds an effective veto over the protocol's unpause.

### Finding Description
The pause system encodes three states in `pauseLevel`:

| Level | Meaning | Swaps |
|-------|---------|-------|
| 0 | Active | Allowed |
| 1 | Admin-paused | Blocked |
| 2 | Protocol-paused | Blocked |

The enforced transitions are: [1](#0-0) 

- `pausePool` (admin): 0 → 1
- `unpausePool` (admin): 1 → 0
- `protocolPausePool` (owner): 0 **or** 1 → 2
- `protocolUnpausePool` (owner): 2 → **1** (always, never 0) [2](#0-1) 

The protocol can pause a pool from either level 0 or level 1, but its unpause path is hardcoded to level 1. The factory stores no record of the pre-pause state. Consequently, a pool that was fully active (level 0) before a protocol pause is left at level 1 (admin-paused) after the protocol unpauses it. The only path back to level 0 is `unpausePool`, which is gated exclusively to the pool admin: [3](#0-2) 

The `swap` function enforces `whenNotPaused`, which reverts for any `pauseLevel != 0`: [4](#0-3) [5](#0-4) 

So after a protocol unpause, swaps remain blocked at level 1 until the admin separately calls `unpausePool`. The protocol has no function that can set `pauseLevel` to 0 directly.

This is the direct analog of the external bug: two distinct operational contexts (protocol-pause recovery and admin-pause) share the same state variable and transition target. Enabling one (protocol unpause) inadvertently requires the other (admin unpause) to complete the operation, and neither can be independently resolved.

### Impact Explanation
After a valid protocol emergency pause-and-unpause cycle on a previously-active pool, all swaps remain blocked (`whenNotPaused` reverts at level 1). If the pool admin is unavailable, unresponsive, or adversarial, the pool's core swap functionality is permanently broken without any recourse from the protocol. LPs cannot trade through the pool; only liquidity withdrawal remains available (no `whenNotPaused` on `removeLiquidity`). This constitutes broken core pool functionality causing unusable swap flows.

### Likelihood Explanation
The scenario requires: (1) the protocol pausing a pool that was at level 0, (2) the protocol subsequently unpausing it, and (3) the admin not cooperating. Steps 1 and 2 are routine emergency-response operations. Step 3 is realistic if the admin key is lost, the admin is a multisig with lost signers, or the admin is adversarial. No special permissions or malicious setup are required beyond the protocol exercising its documented pause/unpause authority.

### Recommendation
Record the pool's `pauseLevel` at the time of `protocolPausePool` and restore it on `protocolUnpausePool`. Concretely, store a `mapping(address => uint8) prePauseLevel` in the factory, set it in `protocolPausePool`, and use it as the target in `protocolUnpausePool` instead of the hardcoded `1`. Alternatively, allow `protocolUnpausePool` to accept a target level (0 or 1) so the protocol can independently restore a pool to active state when appropriate.

### Proof of Concept
```
Pool state: pauseLevel = 0 (active, swaps working)

1. Owner calls protocolPausePool(pool)
   → requires cur == 0 or cur == 1 ✓
   → setPause(2) called → pauseLevel = 2

2. Owner calls protocolUnpausePool(pool)
   → requires cur == 2 ✓
   → setPause(1) called → pauseLevel = 1

3. User calls pool.swap(...)
   → _checkNotPaused(): pauseLevel (1) != 0 → revert PoolPaused()

4. Owner has no function to set pauseLevel = 0 directly.
   Only pool admin can call unpausePool (1 → 0).
   If admin is unavailable → swaps blocked indefinitely.
```
