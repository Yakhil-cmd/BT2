### Title
Malicious Participant Can Corrupt CKD Output by Sending Arbitrary Unverified Shares to Coordinator - (File: src/confidential_key_derivation/protocol.rs)

### Summary
In `do_ckd_coordinator`, the coordinator aggregates each participant's `(norm_big_y, norm_big_c)` contribution by simple addition with no cryptographic verification that the values are consistent with the sender's public key share. A single malicious participant can substitute arbitrary group elements, causing the coordinator to output a corrupted CKD ciphertext. The application then unmasks an incorrect confidential key, permanently corrupting the CKD output for that session.

### Finding Description
The CKD protocol computes a threshold ElGamal encryption of `msk · H(pk ‖ app_id)`. Each honest participant runs `compute_signature_share` to produce:

```
big_y  = y · G                          (random blinding)
big_s  = x_i · H(pk ‖ app_id)          (private-share contribution)
big_c  = big_s + y · app_pk
norm_big_y = λ_i · big_y
norm_big_c = λ_i · big_c
```

These are sent privately to the coordinator. The coordinator aggregates them in `do_ckd_coordinator`: [1](#0-0) 

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
```

There is **no check** that the received `(norm_big_y, norm_big_c)` pair is consistent with the sender's known public key share. The coordinator simply adds whatever values arrive. [2](#0-1) 

A malicious participant deviates from the protocol and sends `(M_y, M_c)` of their choosing. The coordinator computes:

```
total_big_y = Σ_{i≠adv} λ_i·y_i·G  +  M_y
total_big_c = Σ_{i≠adv} λ_i·big_c_i  +  M_c
```

The application unmasks via `confidential_key = total_big_c − app_sk · total_big_y`, yielding:

```
msk · H(pk ‖ app_id)
  + (M_c − λ_adv · big_c_adv)
  − app_sk · (M_y − λ_adv · big_y_adv)
```

Because the adversary does not know `app_sk`, they cannot cancel the error term, so the output is a uniformly wrong group element rather than the correct confidential key.

The root cause is the complete absence of any proof-of-correct-computation on participant messages, directly analogous to the paymaster's missing check on the `postOpReverted` path: in both cases a critical operation (fee deduction / share verification) is simply not performed for a reachable attacker-controlled input. [3](#0-2) 

### Impact Explanation
A single malicious participant corrupts the CKD output accepted by the coordinator and, transitively, by the TEE application. The application derives and uses a wrong confidential key with no indication of failure. This matches:

> **High: Corruption of DKG, reshare, refresh, presign, sign, or CKD outputs so honest parties accept inconsistent public keys, participant sets, transcripts, or unusable cryptographic outputs.**

### Likelihood Explanation
Any participant who holds a valid key share and is admitted to a CKD session can exploit this. No privileged access, leaked secrets, or cryptographic breaks are required. The attacker simply sends two arbitrary `ElementG1` values instead of their honest contribution. The network layer (`recv_from_others`) authenticates the sender's identity but not the content of the message. [4](#0-3) 

### Recommendation
Each participant must accompany their `(norm_big_y, norm_big_c)` with a proof of correct computation. Because the coordinator knows each participant's public key share `V_i = x_i · G2` (from DKG), it can verify via a BLS pairing:

```
e( (norm_big_c − app_pk · norm_big_y) / λ_i , G2 )
  == e( H(pk ‖ app_id) , V_i )
```

Alternatively, require a Sigma/Schnorr proof that `norm_big_c − app_pk · norm_big_y` is a scalar multiple of `H(pk ‖ app_id)` with the same discrete log as `V_i` relative to `G2`. Either check must be performed by the coordinator before adding any participant's contribution to the running sum.

### Proof of Concept
1. Honest participants run `ckd(...)` normally; the malicious participant intercepts the call to `do_ckd_participant` and instead sends `(M_y, M_c) = (ElementG1::generator(), ElementG1::generator())` (or any fixed non-zero pair) to the coordinator.
2. The coordinator's loop at lines 50–55 adds `M_y` and `M_c` to the aggregate without complaint.
3. The coordinator returns `CKDOutput::new(total_big_y, total_big_c)` containing the corrupted values.
4. The TEE application calls `ckd_output.unmask(app_sk)` and receives a wrong group element instead of `msk · H(pk ‖ app_id)`.
5. Any downstream operation that depends on the correct confidential key (e.g., symmetric key derivation inside the TEE) silently uses the wrong value.

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
