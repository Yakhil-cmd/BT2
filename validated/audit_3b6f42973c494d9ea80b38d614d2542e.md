### Title
Copy-Paste Bug: `diff_compression_length` Includes `artifacts_len` for `not_publish_bytecode=true` Branch That `diff_compression` Never Writes — (`File: basic_system/src/system_implementation/flat_storage_model/account_cache_entry.rs`)

---

### Summary

`AccountProperties::diff_compression_length` and `AccountProperties::diff_compression` implement two parallel code paths for the same encoding format. In the `full_diff && not_publish_bytecode == true` branch (encoding format 4), `diff_compression_length` counts `+ 4 // artifacts len` at line 247, but `diff_compression` never writes `artifacts_len` in that branch (lines 364–366). The format specification comment at line 307 explicitly states `artifacts_len (4 BE bytes)` belongs in format 4, confirming the omission in `diff_compression` is the bug. This is a direct copy-paste discrepancy between the length-estimation function and the actual serialization function.

---

### Finding Description

`diff_compression_length` (lines 227–290) and `diff_compression` (lines 311–439) are supposed to be consistent: the former estimates how many bytes the latter will write. In the `full_diff` branch, both functions branch on `not_publish_bytecode`:

**`diff_compression_length`, `not_publish_bytecode = true` (lines 241–248):**
```
1 (metadata) + 8 (versioning) + nonce_diff + balance_diff
+ 32 (bytecode_hash) + 4 (artifacts_len) + 4 (observable_bytecode_len)
```

**`diff_compression`, `not_publish_bytecode = true` (lines 364–398):**
```
1 (metadata) + 8 (versioning) + nonce_diff + balance_diff
+ 32 (bytecode_hash)                       + 4 (observable_bytecode_len)
```

`diff_compression` writes `bytecode_hash` (32 bytes) then immediately writes `observable_bytecode_len` (4 bytes). It never writes `artifacts_len`. The `not_publish_bytecode = false` branch correctly writes `artifacts_len` (line 370–371). The `not_publish_bytecode = true` branch was copy-pasted from the length estimator without removing `artifacts_len` from the write path (or vice-versa: the length estimator was copied from the `false` branch without removing `artifacts_len`).

The format spec comment (lines 307–308) confirms `artifacts_len` must be present in format 4:
> `4. versioning_data(8 BE bytes) & nonce_diff & balance_diff & bytecode_hash (32 bytes) & artifacts_len (4 BE bytes) & observable_len (4 BE bytes)`

---

### Impact Explanation

Two concrete effects:

1. **Incorrect pubdata commitment.** `diff_compression` is called by `apply_storage_diffs_pubdata` to write the actual state-diff pubdata committed to L1. For every account that undergoes a full-diff with `not_publish_bytecode = true` (force-deployed contracts), the emitted pubdata is missing 4 bytes (`artifacts_len`). Any L1 verifier or prover that parses format-4 account diffs per the documented spec will misparse the stream from that point onward, producing an incorrect or unprovable state root.

2. **Resource accounting overcharge.** `diff_compression_length` is called by `calculate_pubdata_used_by_tx` to charge native resources for pubdata. It charges 4 extra bytes that are never written. Any transaction that touches a force-deployed account is overcharged by `4 × native_per_pubdata` native units. If the user's native budget is tight, the transaction reverts with out-of-native when it should succeed.

---

### Likelihood Explanation

`not_publish_bytecode = true` is set on accounts that undergo force deployments (protocol upgrades). Once such an account exists on-chain, **any unprivileged transaction** that modifies it (e.g., calling a system contract that was force-deployed) will trigger both effects. The attacker does not need any special role; they only need to send a transaction that touches a force-deployed account during a block that includes a full-diff for that account.

---

### Recommendation

In `diff_compression`, add the missing `artifacts_len` write in the `not_publish_bytecode = true` branch, immediately after writing `bytecode_hash`, to match the format spec and `diff_compression_length`:

```rust
if not_publish_bytecode {
    dst.write(r#final.bytecode_hash.as_u8_ref());
    result_keeper.pubdata(r#final.bytecode_hash.as_u8_ref());
    // ADD: write artifacts_len to match format 4 spec and diff_compression_length
    dst.write(&r#final.artifacts_len.to_be_bytes());
    result_keeper.pubdata(&r#final.artifacts_len.to_be_bytes());
} else { ... }
```

Alternatively, if `artifacts_len` is intentionally omitted from format 4, remove `+ 4 // artifacts len` from `diff_compression_length` at line 247 and update the format spec comment at line 307.

Add a unit test that calls both functions with `not_publish_bytecode = true` and asserts `diff_compression_length == actual bytes written by diff_compression`.

---

The 4-byte gap between what is charged (`diff_compression_length`) and what is written (`diff_compression`) is the root cause. Every transaction touching a force-deployed account is overcharged, and the pubdata stream emitted to L1 is malformed relative to the documented format 4 encoding.
