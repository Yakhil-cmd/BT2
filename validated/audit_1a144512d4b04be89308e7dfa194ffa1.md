### Title
Missing Caller Authorization in `BankContract` `transfer` Method Allows Unauthorized Native Token Transfers - (`x/cronos/keeper/precompiles/bank.go`)

### Summary

The `transfer` method of the bank precompile accepts an arbitrary `sender` address as a function argument but never verifies that `sender == contract.Caller()`. Any EVM contract can invoke `bank.transfer(victimAddr, attackerAddr, amount)` to move `evm/<callerContract>` native bank tokens out of any address without that address's consent.

### Finding Description

In `BankContract.Run`, the `TransferMethodName` case unpacks `sender` from `args[0]` and passes it directly to `bankKeeper.SendCoins` as the `from` address:

```go
sender := args[0].(common.Address)
recipient := args[1].(common.Address)
...
from := sdk.AccAddress(sender.Bytes())
to := sdk.AccAddress(recipient.Bytes())
...
denom := EVMDenom(contract.Caller())          // denom is tied to caller ✓
...
bc.bankKeeper.SendCoins(ctx, from, to, ...)   // but `from` is caller-supplied ✗
```

There is no check that `sender == contract.Caller()`. [1](#0-0) 

The same pattern exists in the `burn` case: `args[0]` (the address whose tokens are burned) is never verified to equal `contract.Caller()`. [2](#0-1) 

By contrast, the relayer precompile's `exec` utility explicitly enforces `caller == message signer` before executing any IBC message:

```go
caller := common.BytesToAddress(signers[0])
if caller != e.caller {
    return nil, fmt.Errorf("caller is not authenticated: ...")
}
``` [3](#0-2) 

The bank precompile has no equivalent guard.

`bankKeeper.SendCoins` performs no authorization check of its own — it simply moves coins between accounts. Authorization is entirely the precompile's responsibility. [4](#0-3) 

### Impact Explanation

**Critical — Unauthorized transfer of precompile-controlled native bank assets.**

The denom is `evm/<contract.Caller()>`, so a malicious contract M can only manipulate tokens of denom `evm/<M>`. However, users legitimately hold `evm/<M>` tokens whenever:

- Contract M calls `bank.mint(user, amount)` to issue native bank representations (the standard CRC20/CRC21 pattern).
- IBC tokens are auto-converted via `ConvertVouchersToEvmCoins` into `evm/<M>` tokens.

Once users hold `evm/<M>` tokens, any function on M that passes user-controlled addresses to `bank.transfer` — or M itself calling `bank.transfer(victim, attacker, amount)` — drains those balances without the victim's consent. The `burn` variant allows the same contract to destroy any holder's balance.

This maps directly to the allowed Critical impact: *"Unauthorized transfer … for … CRC20, CRC21, ERC20, or precompile-controlled assets."*

### Likelihood Explanation

Medium. The attack requires victims to hold native bank tokens of the calling contract's denom. This is the normal operating state for any CRC20/CRC21 contract that uses the bank precompile's `mint` function, or for any denom that flows through the IBC conversion middleware. [5](#0-4) 

### Recommendation

Add a caller-equality check at the top of the `TransferMethodName` and `BurnMethodName` cases:

```go
case TransferMethodName:
    sender := args[0].(common.Address)
    if sender != contract.Caller() {
        return nil, errors.New("sender must equal the calling contract")
    }
    ...

case BurnMethodName:
    target := args[0].(common.Address)
    if target != contract.Caller() {
        return nil, errors.New("burn target must equal the calling contract")
    }
    ...
```

This mirrors the pattern already used in the relayer precompile's `exec` helper. [3](#0-2) 

### Proof of Concept

1. Attacker deploys contract M.
2. M calls `bank.mint(alice, 1_000)` — Alice now holds 1 000 `evm/<M>` tokens (e.g., as part of a legitimate deposit flow).
3. Attacker calls M's function that invokes `bank.transfer(alice, attacker, 1_000)`.
4. The bank precompile executes `bankKeeper.SendCoins(ctx, alice, attacker, 1_000 evm/<M>)` with no authorization check.
5. Alice's entire `evm/<M>` balance is transferred to the attacker without Alice's signature or approval.

The same call with `bank.burn(alice, 1_000)` permanently destroys Alice's balance instead. [6](#0-5)
