Audit Report

## Title
VIP Slot Bypass Broken: Exhausted `AVAILABLE_MANAGEMENT_CANISTER_CALL_SLOT_COUNT` Blocks Governance Snapshot Operations — (File: `rs/nervous_system/clients/src/management_canister_client.rs`)

## Summary
`LimitedOutstandingCallsManagementCanisterClient::try_borrow_slot` evaluates the exhaustion guard `if *available_slot_count == 0` unconditionally before the VIP branch has any effect on slot consumption. When all 167 slots are held by concurrent unprivileged `canister_status` callers, the counter reaches 0 and every subsequent call — including from NNS Governance — is rejected with `SysTransient`. This defeats the intended starvation protection and allows an unprivileged attacker to block Governance's `take_canister_snapshot` and `load_canister_snapshot` disaster-recovery operations.

## Finding Description
`AVAILABLE_MANAGEMENT_CANISTER_CALL_SLOT_COUNT` is initialized to 167 in `canister.rs` (line 50). `new_management_canister_client()` sets `is_caller_vip = true` only when the caller is in `ALL_NNS_CANISTER_IDS` (lines 57–66); external callers receive `false`.

In `try_borrow_slot` (lines 264–287 of `management_canister_client.rs`):

```rust
fn try_borrow_slot(&self) -> Result<SlotLoan, (i32, String)> {
    let used_slot_count = if self.is_caller_vip { 0 } else { 1 };

    self.available_slot_count
        .with_borrow_mut(|available_slot_count| {
            if *available_slot_count == 0 {   // ← fires for ALL callers
                return Err((SysTransient, "Unavailable. Maybe, try again later?"));
            }
            *available_slot_count = available_slot_count.saturating_sub(used_slot_count);
            Ok(())
        })?;
    ...
}
```

The `used_slot_count = 0` branch for VIPs is only reached after the exhaustion guard passes. When the counter is already 0, the guard fires first and returns `Err` regardless of VIP status. The `SlotLoan` drop implementation (lines 365–372) correctly performs a no-op `saturating_add(0)` for VIPs, confirming the only defect is the premature exhaustion check.

`canister_status` (lines 88–98 of `canister.rs`) is a public `#[update]` endpoint with no access control. Each call holds a slot for the full duration of the management canister round-trip. `take_canister_snapshot` (lines 244–254) and `load_canister_snapshot` (lines 258–268) both call `new_management_canister_client()` and then invoke `try_borrow_slot` via the wrapper — these are the only paths through which Governance can snapshot or restore NNS canisters.

## Impact Explanation
An unprivileged attacker who saturates all 167 slots via concurrent `canister_status` calls causes every Governance-initiated `take_canister_snapshot` and `load_canister_snapshot` proposal to fail with `SysTransient`. These operations are the designated disaster-recovery mechanism for NNS canisters. Blocking them during a critical recovery window (e.g., after a governance canister bug) prevents timely restoration of NNS state. This matches the **High** bounty impact: *Application/platform-level DoS... or NNS security impact with concrete user or protocol harm.*

## Likelihood Explanation
The `canister_status` endpoint requires no credentials. Sending 167 concurrent update calls is within normal IC message queue capacity (the code comment itself notes queues fill at 500). Each call naturally holds its slot for the management canister round-trip latency, so the attacker need only sustain a steady stream of ingress messages — no special privilege, no key material, no coordination. The attack is cheap to maintain and indistinguishable from legitimate load.

## Recommendation
Short-circuit `try_borrow_slot` before the exhaustion check when `is_caller_vip` is true:

```rust
fn try_borrow_slot(&self) -> Result<SlotLoan, (i32, String)> {
    if self.is_caller_vip {
        return Ok(SlotLoan {
            available_slot_count: self.available_slot_count,
            used_slot_count: 0,
        });
    }
    self.available_slot_count.with_borrow_mut(|available_slot_count| {
        if *available_slot_count == 0 {
            return Err((RejectCode::SysTransient as i32, "Unavailable. Maybe, try again later?".to_string()));
        }
        *available_slot_count -= 1;
        Ok(())
    })?;
    let available_slot_count = self.available_slot_count;
    Ok(SlotLoan { available_slot_count, used_slot_count: 1 })
}
```

This ensures VIP callers bypass the exhaustion guard entirely while non-VIP callers remain subject to it.

## Proof of Concept
A state-machine or PocketIC test would:
1. Deploy the NNS root canister with `AVAILABLE_MANAGEMENT_CANISTER_CALL_SLOT_COUNT = 167`.
2. Send 167 concurrent `canister_status` update calls from an unprivileged principal, each targeting a canister configured to delay its management canister response, keeping all slots occupied.
3. While those calls are in-flight, send a `take_canister_snapshot` call from the NNS Governance canister principal.
4. Assert the snapshot call returns `Err` with code `SysTransient` — demonstrating that the VIP bypass fails when the pool is exhausted.
5. Apply the fix (early return for VIPs) and assert the same snapshot call now succeeds.
