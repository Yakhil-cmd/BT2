### Title
Caller-Supplied `RerandomizationArguments.pk` Is Explicitly "Non-Binding" With No Cross-Participant Consistency Enforcement, Enabling Signing Output Corruption ŌĆö (`src/ecdsa/mod.rs`)

---

### Summary

The `RerandomizationArguments` struct accepts several caller-supplied fields ŌĆö most critically `pk` (explicitly annotated as "non-binding"), `tweak`, `entropy`, `big_r`, and `participants` ŌĆö that are used to derive the rerandomization scalar `delta` via HKDF. There is no protocol-level mechanism (e.g., echo broadcast) to enforce that all participants use identical values. A malicious participant can supply a divergent `pk` or `entropy`, producing a different `delta`, which causes their rerandomized presignature shares to be inconsistent with honest parties. The coordinator then receives incoherent shares and cannot assemble a valid signature.

---

### Finding Description

In `src/ecdsa/mod.rs`, the `RerandomizationArguments` struct is defined as:

```rust

    pub participants: ParticipantList,
    /// Fresh, Unpredictable, and Public source of entropy

The `derive_randomness()` method hashes all of these fields together ŌĆö including `pk` ŌĆö through HKDF to produce `delta`:

```rust

`delta` is then used to rerandomize each participant's presignature shares:

- `R' = R^delta`
- `alpha'_i = alpha_i * delta^{-1}`
- `beta'_i = (beta_i + c_i * epsilon) * delta^{-1}`

Each participant constructs their own `RerandomizationArguments` independently and calls `RerandomizedPresignOutput::rerandomize_presign()`. There is no protocol round that broadcasts or echo-broadcasts these arguments to achieve consensus. The `pk` field is explicitly documented as "non-binding", meaning the library imposes no check that it matches the actual keygen output. Similarly, `entropy` is required to be "public, freshly generated, and unpredictable" by documentation alone, with no enforcement. [3](#0-2) 

In the test harness, a single shared `rerand_args` object is constructed and reused for all participants ŌĆö masking the real-world risk where each participant constructs their own independently: [4](#0-3) 

---

### Impact Explanation

**High ŌĆö Corruption of sign outputs so honest parties accept unusable cryptographic outputs.**

If a malicious participant P_m supplies a different `pk` (or `entropy`) in their `RerandomizationArguments`, they compute a different `delta_m ŌēĀ delta`. Their rerandomized shares `(alpha'_m, beta'_m)` are therefore inconsistent with every honest participant's shares. When the coordinator sums all signature shares, the result is an invalid ECDSA signature that fails verification. Honest parties have completed valid protocol steps under valid inputs and yet receive an unusable output. The malicious participant can repeat this across every signing session, causing persistent denial of signing.

---

### Likelihood Explanation

**Medium.** Any participant in the signing set is an unprivileged library caller who independently constructs `RerandomizationArguments`. No out-of-band coordination or allowlist prevents a participant from supplying a divergent `pk` or `entropy`. The attack requires only that one participant deviate ŌĆö a realistic threat in any adversarial MPC deployment.

---

### Recommendation

1. **Add a consensus round for rerandomization parameters.** Before rerandomizing, participants should echo-broadcast their `RerandomizationArguments` (or a hash thereof) and abort if any disagreement is detected. This mirrors the existing echo-broadcast pattern already used in `do_broadcast` elsewhere in the protocol.
2. **Validate `pk` against the keygen output.** Remove the "non-binding" designation and require that `pk` matches the `public_key` from `KeygenOutput`. This eliminates the entire class of divergence via `pk`.
3. **Distribute `entropy` through the protocol.** Rather than having each participant supply their own entropy, derive a shared entropy via a coin-toss or commit-reveal sub-protocol, ensuring all participants use the same value.

---

### Proof of Concept

1. Honest parties P_1 ŌĆ” P_n run DKG ŌåÆ keygen output with master public key `X`.
2. Honest parties run presigning ŌåÆ presignature `(R, alpha_i, beta_i, c_i, e_i)` for each P_i.
3. Malicious participant P_m constructs `RerandomizationArguments` with `pk = X'` where `X' ŌēĀ X` (any arbitrary point).
4. P_m calls `RerandomizedPresignOutput::rerandomize_presign(&presign_m, &malicious_args)`, obtaining `delta_m` derived from `X'`.
5. Honest participants P_1 ŌĆ” P_{n-1} call `rerandomize_presign` with correct `pk = X`, obtaining `delta` derived from `X`.
6. Since `delta_m ŌēĀ delta`, P_m's shares `(alpha'_m, beta'_m)` are inconsistent with honest shares.
7. The coordinator sums all shares: `s = sum(s_i)` is incorrect; the resulting `(R', s)` fails ECDSA verification against `X + epsilon*G`.
8. The signing session produces an unusable output. P_m can repeat this for every signing session. [2](#0-1) [5](#0-4)

```rust
    pub fn derive_randomness(&self) -> Result<Scalar, ProtocolError> {
        // create a string containing (pk, msg_hash, big_r, sorted(participants))
        let pk_encoded_point = self.pk.to_encoded_point(true);
        let encoded_pk: &[u8] = pk_encoded_point.as_bytes();
        let encoded_tweak: &[u8] = &<Secp256K1ScalarField as Field>::serialize(&self.tweak.value());
        let encoded_msg_hash: &[u8] = &self.msg_hash;
        let big_r_encoded_point = self.big_r.to_encoded_point(true);
        let encoded_big_r: &[u8] = big_r_encoded_point.as_bytes();

        // concatenate all the bytes
        let mut concatenation = Vec::new();
        // 1 byte counter, used in the unlikely case that the hash result is 0
        concatenation.extend_from_slice(&[0u8, 1]);
        concatenation.extend_from_slice(encoded_pk);
        concatenation.extend_from_slice(encoded_tweak);
        concatenation.extend_from_slice(encoded_msg_hash);
        concatenation.extend_from_slice(encoded_big_r);
        // Append each ParticipantId's
        for participant in self.participants.participants() {
            concatenation.extend_from_slice(&participant.bytes());
        }

        // initiate hkdf with the salt and with some `good' entropy
        let hk = Hkdf::<sha3::Sha3_256>::new(Some(&Self::SALT), &self.entropy);

        let mut delta = Scalar::ZERO;
        // If the randomness created is 0 then we want to generate a new randomness
        while bool::from(delta.is_zero()) {
            // Generate randomization out of HKDF(counter, entropy, pk, msg_hash, big_r, participants, )
            // where entropy is a public but unpredictable random string.
            // The counter depends on the number of times we enter into this loop
            let mut okm = [0u8; 32];

            hk.expand(&concatenation, &mut okm)
                .map_err(|_| ProtocolError::HashingError)?;

            // derive the randomness delta
            delta = Scalar::from_repr(okm.into()).unwrap_or(
                // if delta falls outside the field
                // probability is negligible: in the order of 1/2^224
                Scalar::ZERO,
            );
            // Increment the counter, the probability that this overflows is astronomically low
            let concatenation_0 = concatenation
                .first_mut()
                .ok_or(ProtocolError::InvalidIndex)?;
            *concatenation_0 += 1;
        }
        Ok(delta)
    }
