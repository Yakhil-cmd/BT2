### Title
Uncapped `addFeeBuyE6`/`addFeeSellE6` in `setPoolBinAdditionalFees` Allows Pool Admin to Impose Unbounded Per-Bin Fees, Causing Bad-Price Execution — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary

`MetricOmmPoolFactory.setPoolBinAdditionalFees` forwards `addFeeBuyE6` and `addFeeSellE6` directly to the pool with no upper-bound validation. The base spread fee is hard-capped at `HARD_MAX_SPREAD_FEE_E6 = 200_000` (20%), but the per-bin additional fees are `uint16` with no enforced ceiling. A pool admin can set either value to `type(uint16).max = 65535` (6.5535% in E6 units), inflating the effective ask/bid price beyond what the oracle curve permits for any targeted bin.

### Finding Description

`setPoolBinAdditionalFees` in the factory passes the caller-supplied values straight through to `setBinAdditionalFees` on the pool:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
``` [1](#0-0) 

The pool's `setBinAdditionalFees` only validates the bin index, not the fee magnitudes:

```solidity
function setBinAdditionalFees(int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external onlyFactory nonReentrant(PoolActions.SET_BIN_ADDITIONAL_FEES)
{
    if (bin < LOWEST_BIN || bin > HIGHEST_BIN) revert InvalidBinIndex(bin);
    BinState storage s = _binStates[bin];
    s.addFeeBuyE6 = addFeeBuyE6;
    s.addFeeSellE6 = addFeeSellE6;
``` [2](#0-1) 

These values feed directly into the swap price computation:

```solidity
uint256 buyFeeX64 = baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6);
uint256 sellFeeX64 = baseFeeX64 + Math.mulDiv(binState.addFeeSellE6, ONE_X64, 1e6);

uint256 askBeforeNotional = Math.mulDiv(marginalPriceX64, ONE_X64 + buyFeeX64, ONE_X64, Math.Rounding.Ceil);
uint256 bidAfterSpread = Math.mulDiv(marginalPriceX64, ONE_X64, ONE_X64 + sellFeeX64, Math.Rounding.Floor);
``` [3](#0-2) 

The same uncapped path exists at pool creation time: `_unpackAndValidateBinStates` only checks that `length != 0` and never validates `addFeeBuyE6`/`addFeeSellE6` packed into the bin data array. [4](#0-3) 

By contrast, the base spread fee has a hard factory-level ceiling:

```solidity
uint24 internal constant HARD_MAX_SPREAD_FEE_E6 = 200_000;   // 20%
``` [5](#0-4) 

No equivalent ceiling exists for `addFeeBuyE6`/`addFeeSellE6`.

### Impact Explanation

Setting `addFeeBuyE6 = type(uint16).max = 65535` (6.5535% in E6) on the current active bin inflates the effective ask price by 6.5535% above the oracle-derived marginal price. Traders executing swaps in that bin pay materially more than the oracle/bin curve permits — a direct loss of user principal on every swap routed through the affected bin. The pool admin can target any bin, including the current active bin, making the impact immediate and continuous until corrected.

### Likelihood Explanation

The pool admin is a semi-trusted role that can call `setPoolBinAdditionalFees` at any time without a timelock. Any pool whose admin key is compromised, or any pool whose admin acts adversarially within the bounds of their role, can trigger this. The call requires no special preconditions beyond holding the `poolAdmin` role.

### Recommendation

Add an upper-bound check in `setPoolBinAdditionalFees` (and symmetrically in `_unpackAndValidateBinStates` at creation time), analogous to the existing `HARD_MAX_SPREAD_FEE_E6` guard:

```solidity
uint16 internal constant MAX_BIN_ADDITIONAL_FEE_E6 = <chosen_limit>;

function setPoolBinAdditionalFees(...) external ... {
    if (addFeeBuyE6 > MAX_BIN_ADDITIONAL_FEE_E6) revert BinAdditionalFeeTooHigh();
    if (addFeeSellE6 > MAX_BIN_ADDITIONAL_FEE_E6) revert BinAdditionalFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

Apply the same guard inside `_unpackAndValidateBinStates` when unpacking initial bin data.

### Proof of Concept

1. Pool admin calls `factory.setPoolBinAdditionalFees(pool, curBinIdx, 65535, 65535)`.
2. `setBinAdditionalFees` stores `addFeeBuyE6 = 65535`, `addFeeSellE6 = 65535` with no revert.
3. Next swap in that bin computes `buyFeeX64 = baseFeeX64 + mulDiv(65535, ONE_X64, 1e6)`, adding ≈6.5535% on top of the oracle spread.
4. `askBeforeNotional = mulDiv(marginalPriceX64, ONE_X64 + buyFeeX64, ONE_X64)` — the ask price delivered to the trader is 6.5535% above the oracle-permitted price.
5. The trader's `amountIn` is inflated by 6.5535% relative to a fair oracle execution; the excess accrues as LP spread fees, not to the trader. [3](#0-2) [1](#0-0)
