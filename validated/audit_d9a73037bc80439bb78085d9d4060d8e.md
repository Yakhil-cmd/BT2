### Title
Nonce Reuse via Cloneable `PresignOutput` Enables Signing Share Recovery in FROST EdDSA `sign_v2` — (`src/frost/eddsa/sign.rs`, `src/frost/mod.rs`)

---

### Summary

`PresignOutput<C>` derives `Clone` with no one-time-use enforcement. A malicious coordinator can clone a `PresignOutput` and drive **three** `sign_v2` sessions against different messages using the same nonces, collecting all participants' signature shares each time. Three sessions yield a solvable linear system that recovers every participant's signing share `s_i`.

---

### Impact Explanation

In FROST EdDSA, each participant's signature share is:

```
z_i = (d_i + e_i * rho_i) + lambda_i * c * s_i
```

where `d_i`, `e_i` are the hiding/binding nonces from `SigningNonces`, `rho_i = H(i, commitments_map, message)` is the binding factor (message-dependent), `c = H(R, pk, message)` is the challenge, and `s_i` is the secret signing share.

Because `rho_i` depends on the message, **two** sessions with the same `(d_i, e_i)` but different messages yield:

```
z_i1 = d_i + e_i*rho_i1 + lambda_i*c1*s_i
z_i2 = d_i + e_i*rho_i2 + lambda_i*c2*s_i
```

That is 2 equations with 3 unknowns (`d_i`, `e_i`, `s_i`) — underdetermined. However, **three** sessions yield:

```
z_i1 - z_i2 = e_i*(rho_i1 - rho_i2) + lambda_i*(c1 - c2)*s_i
z_i1 - z_i3 = e_i*(rho_i1 - rho_i3) + lambda_i*(c1 - c3)*s_i
```

Two equations, two unknowns (`e_i`, `s_i`) — directly solvable for `s_i` (the signing share) for every participant whose shares the coordinator observed.

The question's claim that 2 sessions suffice is incorrect for FROST (unlike plain Schnorr/EdDSA with a single nonce), but **3 sessions are sufficient and trivially achievable** since `PresignOutput` is `Clone`.

---

### Likelihood Explanation

A malicious coordinator:
1. Obtains a `PresignOutput` from the presign phase.
2. Calls `.clone()` twice to produce three identical copies.
3. Runs `sign_v2` three times with messages `m1`, `m2`, `m3`.
4. Collects `z_i1`, `z_i2`, `z_i3` for each participant from the protocol's message channel.
5. Solves the 2×2 linear system per participant to recover `s_i`.

No cryptographic assumptions are broken. The library provides zero enforcement against this: no consumption token, no `ZeroizeOnDrop` on `PresignOutput`, no session counter, no audit log.

---

### Recommendation

1. **Remove `Clone` from `PresignOutput`** or wrap nonces in a type that cannot be cloned.
2. **Consume `PresignOutput` on first use** — the current by-value API already moves it, but callers can clone before passing. Consider wrapping in a newtype with no `Clone` impl and a `take()`-style API that zeroes the nonces.
3. **Zeroize nonces on drop** — add `ZeroizeOnDrop` to `PresignOutput` so that even if a clone is made, the nonce material is cleared after use.
4. **Document the one-time-use invariant** prominently on `sign_v2`, as is done for the ECDSA variant.

### Proof of Concept

```rust
// Attacker is the coordinator. Clone the same PresignOutput three times.
let presig_clone1 = presig.clone();
let presig_clone2 = presig.clone();
let presig_clone3 = presig.clone();

let (z1, c1, rho1) = run_sign_v2_collect_shares(&keys, presig_clone1, b"msg1");
let (z2, c2, rho2) = run_sign_v2_collect_shares(&keys, presig_clone2, b"msg2");
let (z3, c3, rho3) = run_sign_v2_collect_shares(&keys, presig_clone3, b"msg3");

// For each participant i:
// z_i1 - z_i2 = e_i*(rho_i1 - rho_i2) + lambda_i*(c1 - c2)*s_i
// z_i1 - z_i3 = e_i*(rho_i1 - rho_i3) + lambda_i*(c1 - c3)*s_i
// Solve 2x2 system -> recover s_i
```
