### Title
Gateway Skips All Signature Verification for Invoke Transactions Targeting Undeployed Accounts, Allowing Unauthorized Mempool Admission - (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the Apollo gateway skips the `__validate__` entry point — the sole signature-verification mechanism — for any invoke transaction with `nonce == 1` when the `sender_address` has a deploy-account transaction in the mempool. Because the check is keyed only on the *sender address* (not on who submitted the deploy-account), any third party can craft an invoke transaction for a victim's address, satisfy the skip condition, and have it admitted to the mempool with zero signature verification.

### Finding Description

In `crates/apollo_gateway/src/stateful_transaction_validator.rs`, `skip_stateful_validations` returns `true` (skip) when all three conditions hold:

1. The transaction is an `Invoke` with `nonce == 1`
2. The on-chain account nonce is `0` (account not yet deployed)
3. `account_tx_in_pool_or_recent_block(sender_address)` returns `true` [1](#0-0) 

When `skip_validate == true`, `run_validate_entry_point` sets `execution_flags.validate = false`, so the `__validate__` entry point is never called: [2](#0-1) 

In Starknet, `__validate__` is the *only* place where the account contract verifies the transaction signature. No other code path in the gateway independently checks the signature. `validate_by_mempool` checks nonce/fee rules only; `validate_state_preconditions` checks nonce and resource bounds only. Therefore, when `skip_validate == true`, the transaction is admitted with **no signature check whatsoever**.

`account_tx_in_pool_or_recent_block` returns `true` if the address has *any* transaction in the pool — not specifically a deploy-account transaction submitted by the same user: [3](#0-2) 

**Attack path:**

1. Victim broadcasts a `DeployAccount` transaction for address `V`. It enters the mempool; on-chain nonce for `V` is still `0`.
2. Attacker observes `V` in the mempool. Attacker crafts an `InvokeTransaction` with `sender_address = V`, `nonce = 1`, and arbitrary (malicious) calldata. The attacker provides any signature (or none).
3. Gateway evaluates `skip_stateful_validations`: `nonce == 1` ✓, on-chain nonce `== 0` ✓, `account_tx_in_pool_or_recent_block(V) == true` ✓ → returns `true`.
4. `run_validate_entry_point` is called with `skip_validate = true`; `__validate__` is never invoked.
5. The attacker's transaction is admitted to the mempool without any signature verification.

The comment in the code acknowledges the intent but does not guard against third-party abuse: [4](#0-3) 

### Impact Explanation

This matches the **High** impact category: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

The gateway's invariant — that every transaction in the mempool has passed signature verification — is broken. Concretely:

- An attacker can inject unsigned (or arbitrarily signed) invoke transactions for any victim address that has a pending deploy-account in the mempool.
- These transactions occupy the `nonce = 1` slot. If the attacker submits with a higher tip/fee than the victim's legitimate invoke, the attacker's transaction is sequenced first. At execution time the batcher calls `__validate__`, which fails (attacker has no valid key), the transaction is rejected, and the nonce is *not* incremented — but the victim's transaction has been delayed and the mempool slot was occupied.
- An attacker can repeat this continuously during the deploy-account's mempool window, causing sustained DoS against the victim's first post-deploy invoke.
- For account contracts whose `__validate__` is permissive (e.g., accepts any signature), the attacker's transaction would execute successfully, causing arbitrary state changes on behalf of the victim.

### Likelihood Explanation

**Medium.** The attack window is the time between a victim's deploy-account transaction entering the mempool and being committed to a block. On a live network this window is short but observable. The attacker needs only to monitor the mempool for deploy-account transactions and submit a competing invoke transaction — no privileged access is required.

### Recommendation

The `skip_stateful_validations` path must not bypass all signature verification. Options:

1. **Perform a stateless signature check** independently of `__validate__` before admitting the transaction when the skip condition is triggered. This requires the gateway to know the account's public key, which is available from the deploy-account transaction's constructor calldata.
2. **Bind the skip to the deploy-account submitter**: record the address of the user who submitted the deploy-account transaction and only grant the skip to invoke transactions whose signature can be verified against the same key material.
3. **Require the invoke transaction to be submitted in the same RPC call / batch as the deploy-account transaction**, so the gateway can cryptographically link them.

### Proof of Concept

```
1. Victim submits DeployAccount for address V (nonce=0, valid signature).
   → Mempool: {V: DeployAccount(nonce=0)}
   → On-chain nonce for V: 0

2. Attacker submits Invoke(sender=V, nonce=1, calldata=<malicious>, signature=<garbage>).

3. Gateway stateful validation:
   account_nonce = get_nonce_from_state(V) = 0
   skip_stateful_validations:
     - tx is Invoke ✓
     - tx.nonce() == 1 ✓
     - account_nonce == 0 ✓
     - account_tx_in_pool_or_recent_block(V) == true ✓  (DeployAccount is in pool)
   → returns true (skip __validate__)

4. run_validate_entry_point called with skip_validate=true:
   execution_flags.validate = false
   __validate__ is NOT called → no signature check

5. Attacker's Invoke(V, nonce=1, malicious) is admitted to mempool.
   Mempool: {V: DeployAccount(nonce=0), Invoke(nonce=1, malicious)}

6. Batcher sequences DeployAccount(V), then Invoke(V, nonce=1, malicious).
   At execution: __validate__ is called → fails (wrong key) → tx rejected.
   But victim's legitimate Invoke(V, nonce=1) was displaced or delayed.
```

        let charge_fee = enforce_fee(executable_tx, only_query);

            ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };

        let account_tx = AccountTransaction { tx: executable_tx.clone(), execution_flags };

        // Build block context.
        let mut versioned_constants = VersionedConstants::get_versioned_constants(
            self.config.versioned_constants_overrides.clone(),
        );
        // The validation of a transaction is not affected by the casm hash migration.
        versioned_constants.disable_casm_hash_migration();

        let mut block_info = self.gateway_fixed_block_state_reader.get_block_info().await?;
        block_info.block_number = block_info.block_number.unchecked_next();
        let block_context = BlockContext::new(
            block_info,
            self.chain_info.clone(),
            versioned_constants,
            BouncerConfig::max(),
        );

        // Move state into the blocking task and run CPU-heavy validation.
        let state_reader_and_contract_manager = self.take_state_reader_and_contract_manager();

        let cur_span = Span::current();
        #[allow(clippy::result_large_err)]
        tokio::task::spawn_blocking(move || {
            cur_span.in_scope(|| {
                let state = CachedState::new(state_reader_and_contract_manager);
                let mut blockifier_validator = StatefulValidator::create(state, block_context);
                blockifier_validator.validate(account_tx)
            })
        })

            code: StarknetErrorCode::KnownErrorCode(KnownStarknetErrorCode::ValidateFailure),
            message: e.to_string(),
        })?;

/// Check if validation of an invoke transaction should be skipped due to deploy_account not being
/// processed yet. This feature is used to improve UX for users sending deploy_account + invoke at
/// once.
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
