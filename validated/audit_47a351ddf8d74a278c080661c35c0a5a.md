### Title
Echo-Broadcast Threshold Degenerates to Zero for n ≤ 3, Enabling Split-View Attack in DKG — (File: `src/protocol/echo_broadcast.rs`)

---

### Summary

The `echo_ready_thresholds` function returns `(0, 0)` for any participant count `n ≤ 3`, collapsing the echo-broadcast protocol into a trivial single-message delivery with no Byzantine fault tolerance. Because `assert_key_invariants` permits DKG with `n = 3` and `threshold = 2` (implying one tolerated malicious party), a malicious participant can equivocate during the commitment broadcast phase of DKG, causing the two honest parties to derive inconsistent public keys and accept a corrupted `KeygenOutput`.

---

### Finding Description

**Root cause — threshold reset to zero:** [1](#0-0) 

```rust
fn echo_ready_thresholds(n: usize) -> (usize, usize) {
    if n <= 3 {
        return (0, 0);   // ← both echo_t and ready_t become 0
    }
    let broadcast_threshold = (n - 1) / 3;
    let echo_threshold = usize::midpoint(n, broadcast_threshold);
    (echo_threshold, broadcast_threshold)
}
```

With `echo_t = 0` and `ready_t = 0`, every threshold comparison in `reliable_broadcast_receive_all` reduces to `count > 0`, meaning a single received message immediately advances the state machine through all phases: [2](#0-1) [3](#0-2) 

A party transitions to Ready after its own simulated Echo, then delivers after its own simulated Ready — without waiting for any other honest party's confirmation. A malicious sender that delivers `Send(A)` to P1 and `Send(B)` to P2 causes P1 to deliver A and P2 to deliver B: a classic split-view.

**DKG does not enforce n > 3:** [4](#0-3) 

`assert_key_invariants` only requires `participants.len() >= 2` and `threshold >= 2`. A caller with `n = 3, threshold = 2` passes all checks, yet the underlying echo-broadcast provides zero Byzantine protection.

**DKG uses `do_broadcast` for the commitment round:** [5](#0-4) 

```rust
let commitments_and_proofs_map = do_broadcast(
    &mut chan,
    &participants,
    me,
    (commitment, proof_of_knowledge),
).await?;
```

**The commitment hash is sent point-to-point, not via broadcast:** [6](#0-5) 

```rust
chan.send_many(wait_round_1, &commitment_hash)?;
// ...
for (from, their_commitment_hash) in
    recv_from_others(&chan, wait_round_1, &participants, me).await?
{
    all_hash_commitments.put(from, their_commitment_hash);
}
```

Because the hash is delivered point-to-point, a malicious party M can send `hash_A` to P1 and `hash_B` to P2. It then sends `commitment_A` to P1 and `commitment_B` to P2 via the degenerate echo-broadcast. Each honest party independently verifies its commitment against the hash it received — both checks pass — and each computes a different group public key. [7](#0-6) 

The final `broadcast_success` call does not detect the inconsistency because M can send `(true, session_id)` to both honest parties, and the same degenerate echo-broadcast delivers it without cross-checking. [8](#0-7) 

---

### Impact Explanation

Both honest parties complete `do_keyshare` successfully but hold different `KeygenOutput.public_key` values. Any subsequent signing session will fail or produce signatures that verify under only one party's view of the public key. This matches the allowed High impact: **Corruption of DKG outputs so honest parties accept inconsistent public keys**.

---

### Likelihood Explanation

`n = 3, threshold = 2` is the smallest non-trivial threshold configuration and is a natural first choice for a 2-of-3 setup. The API imposes no restriction against it. Any single malicious participant in such a session can mount the attack without any special capability beyond controlling their own network messages.

---

### Recommendation

1. **Enforce `n > 3` in `assert_key_invariants`** (or equivalently `n >= 4`) when the caller expects Byzantine fault tolerance, matching the `n >= 3f+1` requirement of the echo-broadcast protocol.
2. Alternatively, **broadcast the commitment hash via `do_broadcast`** instead of `send_many`, so equivocation on the hash is detectable regardless of `n`.
3. Document clearly that for `n ≤ 3` the echo-broadcast provides no Byzantine guarantees and DKG is only secure when all parties are honest.

---

### Proof of Concept

```
Setup: participants = [P1, P2, M], threshold = 2

Round 1 (session-id broadcast — M is honest here):
  M → P1: session_id_M
  M → P2: session_id_M          (same, so both compute identical session_id)

Round 1 (commitment-hash, point-to-point):
  M → P1: hash_A = H(commitment_A)
  M → P2: hash_B = H(commitment_B)   ← equivocation, undetected

Commitment broadcast (do_broadcast, n=3, echo_t=ready_t=0):
  M sends Send(commitment_A) to P1
  M sends Send(commitment_B) to P2

  P1: receives Send(A) → echoes A → receives own Echo(A) (count=1>0) →
      sends Ready(A) → receives own Ready(A) (count=1>0) → delivers A
  P2: receives Send(B) → echoes B → receives own Echo(B) (count=1>0) →
      sends Ready(B) → receives own Ready(B) (count=1>0) → delivers B

verify_commitment_hash:
  P1: H(commitment_A) == hash_A  ✓
  P2: H(commitment_B) == hash_B  ✓

public_key_from_commitments:
  P1 computes public_key_1  (using commitment_A from M)
  P2 computes public_key_2  (using commitment_B from M)
  public_key_1 ≠ public_key_2

broadcast_success:
  M sends (true, session_id) to both → both pass

Result: P1.KeygenOutput.public_key ≠ P2.KeygenOutput.public_key
```

```rust
fn echo_ready_thresholds(n: usize) -> (usize, usize) {
    // case where no malicious parties are assumed: when n <= 3/
    // In this case the echo and ready thresholds are both 0
    // later we compare if we have collected more votes than these thresholds
    if n <= 3 {
        return (0, 0);
    }
    // we should always have n >= 3*threshold + 1
    let broadcast_threshold = (n - 1) / 3;
    let echo_threshold = usize::midpoint(n, broadcast_threshold);
    (echo_threshold, broadcast_threshold)
}
```

```rust
pub fn assert_key_invariants(
    participants: &[Participant],
    me: Participant,
    threshold: impl Into<ReconstructionLowerBound>,
) -> Result<ParticipantList, InitializationError> {
    let threshold = usize::from(threshold.into());
    // need enough participants
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }

    // Step 1.1
    // validate threshold
    if threshold > participants.len() {
        return Err(InitializationError::ThresholdTooLarge {
            threshold,
            max: participants.len(),
        });
    }
    // Step 1.1
    if threshold < 2 {
        return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
    }

    // ensure uniqueness of participants in the participant list
    let participants =
        ParticipantList::new(participants).ok_or(InitializationError::DuplicateParticipants)?;

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }
    Ok(participants)
}
```
