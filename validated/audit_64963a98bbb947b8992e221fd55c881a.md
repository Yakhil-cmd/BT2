### Title
Missing Access Control on `update_rewards` Allows Permanent Griefing of All Consensus Rewards - (File: `src/staking/staking.cairo`)

---

### Summary
The `update_rewards` function in the Staking contract has no caller access control. Any unprivileged address can invoke it with `disable_rewards: true`, consuming the global per-block `last_reward_block` slot without distributing any rewards. Repeated every block, this permanently freezes unclaimed yield for every staker in the consensus rewards phase.

---

### Finding Description

`update_rewards` is exposed as a public entrypoint under `IStakingRewardsManager`. Its only gate is `general_prerequisites()`, which checks the pause flag and that the caller is non-zero — no role check of any kind. [1](#0-0) 

The function enforces a global, single-use-per-block invariant via `last_reward_block`: [2](#0-1) 

Critically, `last_reward_block` is written to storage **before** the `disable_rewards` branch is evaluated: [3](#0-2) 

When `disable_rewards` is `true`, the function returns immediately after writing `last_reward_block`, distributing nothing: [4](#0-3) 

Because `last_reward_block` is now equal to the current block number, every subsequent call to `update_rewards` in the same block — including the legitimate consensus-layer call — will revert with `REWARDS_ALREADY_UPDATED`. The attacker repeats this in every block.

The `general_prerequisites` guard that is the only protection: [5](#0-4) 

---

### Impact Explanation

**HIGH — Permanent freezing of unclaimed yield for all stakers.**

Once the consensus rewards phase is active (`is_pre_consensus()` returns `false`), all staker rewards flow exclusively through `update_rewards`. An attacker who calls `update_rewards(any_valid_staker, disable_rewards: true)` once per block prevents every staker in the protocol from ever accumulating consensus rewards. The yield is not stolen — it is simply never credited — matching the "Permanent freezing of unclaimed yield" impact category.

---

### Likelihood Explanation

**HIGH.** The function is permissionlessly callable by any non-zero address. The attacker needs only one cheap transaction per block. There is no economic barrier, no special privilege required, and no way for the protocol to distinguish the malicious call from a legitimate one at the contract level.

---

### Recommendation

Restrict `update_rewards` to a trusted caller — for example, the consensus-layer contract address stored in a dedicated storage slot, enforced with an `assert_caller_is_consensus_contract()` guard analogous to the existing `assert_caller_is_attestation_contract()`: [6](#0-5) 

Alternatively, move the `self.last_reward_block.write(current_block_number)` call to **after** the `disable_rewards` guard so that a no-op call does not consume the per-block slot.

---

### Proof of Concept

1. Consensus rewards phase is active (`consensus_rewards_first_epoch` has been set and passed).
2. Attacker identifies any staker `S` that is active and has non-zero STRK balance at the current epoch.
3. In block `N`, attacker sends: `staking.update_rewards(S, disable_rewards: true)`.
4. Contract writes `last_reward_block = N` and returns — no rewards distributed.
5. The legitimate consensus-layer call `update_rewards(any_staker, false)` in block `N` reverts: `REWARDS_ALREADY_UPDATED`. [2](#0-1) 

6. Attacker repeats step 3 in every subsequent block.
7. Result: `unclaimed_rewards_own` for every staker remains at zero indefinitely; no pool rewards are ever forwarded; the reward supplier's unclaimed balance is never incremented.
