### Title
Unauthorized Invoke Transaction Accepted via `skip_stateful_validations` Signature Bypass — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator skips the `__validate__` entry-point call (i.e., signature verification) for any invoke transaction whose nonce is 1 and whose sender address appears in the mempool or a recent block. The check does not verify that the submitter of the invoke is the same party who submitted the `deploy_account`. Any attacker who observes a `deploy_account` transaction in the mempool can immediately submit a second invoke transaction for that address — with arbitrary calldata and a forged/empty signature — and the gateway will admit it without any signature check.

### Finding Description

`skip_stateful_validations` is called from `run_pre_validation_checks` inside `extract_state_nonce_and_run_validations`:

```
// crates/apollo_gateway/src/stateful_transaction_validator.rs
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                ...
        }
    }
    Ok(false)
}
``` [1](#0-0) 

When this returns `true`, `run_validate_entry_point` sets `execution_flags.validate = false`:

```rust
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [2](#0-1) 

With `validate: false`, `validate_tx` in the blockifier returns `Ok(None)` immediately without calling the account's `__validate__` entry point:

```rust
pub fn validate_tx(...) -> TransactionExecutionResult<Option<CallInfo>> {
    if !self.execution_flags.validate {
        return Ok(None);
    }
    ...
}
``` [3](#0-2) 

The condition that gates the skip is:

1. Transaction is `Invoke`
2. `tx.nonce() == 1`
3. `account_nonce == 0` (account not yet deployed in committed state)
4. `account_tx_in_pool_or_recent_block(sender_address)` returns `true` [4](#0-3) 

The fourth condition is satisfied by the presence of **any** transaction from that address in the mempool — not specifically a `deploy_account` transaction. The comment acknowledges this:

> "It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations." [5](#0-4) 

The mempool's `account_tx_in_pool_or_recent_block` checks both the pool and the committed-block state:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [6](#0-5) 

There is no check that the party submitting the invoke is the same party who submitted the `deploy_account`. This is the exact structural analog to the WPunkGateway bug: User A "approves" the gateway by submitting `deploy_account`; anyone can then submit an invoke on behalf of User A's address without a valid signature.

### Impact Explanation

The gateway admits an invoke transaction that carries an arbitrary (attacker-controlled) signature and calldata for a victim account. The admitted transaction:

1. **Occupies the nonce=1 slot** in the mempool for the victim's address. If the victim also submitted a legitimate invoke with nonce=1, the mempool may reject it as a duplicate nonce (or require fee escalation to displace the attacker's transaction).
2. **Delays or blocks the victim's first post-deploy invoke** — the core UX flow the `skip_stateful_validations` feature was designed to enable.
3. **Forces the victim to race or escalate fees** to displace the attacker's slot.

When the batcher later executes the block, it uses `AccountTransaction::new_for_sequencing` which sets `validate: true`:

```rust
pub fn new_for_sequencing(tx: Transaction) -> Self {
    let execution_flags = ExecutionFlags {
        validate: true,
        ...
    };
    ...
}
``` [7](#0-6) 

So the attacker's transaction will fail `__validate__` during execution and be rejected (no state change, no fee charge). The harm is therefore admission-level: the gateway accepted an unauthorized transaction, disrupting the victim's mempool slot.

### Likelihood Explanation

The attack requires only:
- Monitoring the public mempool (or the gateway's response) for `deploy_account` transactions.
- Submitting an invoke with `sender_address` = victim's address, `nonce = 1`, any calldata, any signature, before the victim's own invoke arrives.

No privileged access, no special funds, and no cryptographic capability are required. The window is the time between the victim's `deploy_account` being admitted and the victim's own invoke being admitted — typically seconds in a live system.

### Recommendation

The `skip_stateful_validations` path should not bypass signature verification entirely. Two options:

1. **Verify the signature off-chain at the gateway**: For the skip-validate case, derive the expected public key from the `deploy_account`'s constructor calldata (which is known from the mempool entry) and verify the invoke's ECDSA signature against the computed transaction hash before admitting the transaction.

2. **Restrict the skip to the same submitter**: Track the source (e.g., IP or authenticated session) of the `deploy_account` and only allow the skip for invokes arriving from the same source. This is weaker but simpler.

3. **Remove the skip entirely**: Require users to wait for the `deploy_account` to be committed before submitting the invoke. This eliminates the UX optimization but closes the admission bypass.

### Proof of Concept

```
1. Victim submits deploy_account for address X (class_hash C, salt S, constructor_calldata D).
   Gateway accepts it; mempool now contains (X, nonce=0).

2. Attacker observes the deploy_account in the mempool (public).

3. Attacker constructs:
     invoke_tx {
       sender_address: X,
       nonce: 1,
       calldata: [<drain_victim_funds_calldata>],
       signature: [0x0, 0x0],   // arbitrary — never checked
       resource_bounds: <valid>,
     }

4. Attacker submits invoke_tx to the gateway.

5. Gateway stateful validator:
   - account_nonce = get_nonce(X) = 0   (X not yet deployed)
   - tx.nonce() = 1
   - account_tx_in_pool_or_recent_block(X) = true  (deploy_account is in pool)
   → skip_validate = true
   → execution_flags.validate = false
   → __validate__ is NOT called
   → invoke_tx admitted to mempool

6. Mempool now contains both (X, nonce=0, deploy_account) and (X, nonce=1, attacker_invoke).

7. Victim submits their own legitimate invoke (nonce=1).
   Mempool rejects it: DuplicateNonce for (X, 1), or requires fee escalation.

8. Victim's first post-deploy action is blocked or delayed.
```

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
