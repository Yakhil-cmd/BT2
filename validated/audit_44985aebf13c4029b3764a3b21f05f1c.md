### Title
Gateway Admits Invoke(nonce=1) With Invalid Signature by Checking Any Pool Presence Instead of Deploy-Account Presence — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

`skip_stateful_validations` skips the blockifier `__validate__` entry-point check for an Invoke transaction with `nonce=1` when `account_nonce=0` in state, provided `account_tx_in_pool_or_recent_block` returns `true`. That helper returns `true` for **any** transaction from the sender address in the pool — not specifically a `DeployAccount`. An attacker who observes a victim's pending `DeployAccount` in the mempool can immediately submit an Invoke(nonce=1) for the same address with an arbitrary/invalid signature; the gateway skips `__validate__`, and the transaction is admitted to the mempool.

### Finding Description

`skip_stateful_validations` is the sequencer's analog of the external report's "override pending operation" pattern: a guard that was set for one pending operation (the `DeployAccount`) is consumed by a second, unrelated operation (the attacker's Invoke), because the guard only checks for *any* pool presence rather than the specific operation that justified skipping validation.

**Exact code path:**

`extract_state_nonce_and_run_validations` calls `run_pre_validation_checks`, which calls `skip_stateful_validations`: [1](#0-0) 

Inside `run_pre_validation_checks`, `skip_stateful_validations` is called after `validate_by_mempool` (which only checks nonce ordering and fee escalation, not signatures): [2](#0-1) 

`skip_stateful_validations` returns `true` (skip `__validate__`) when:
- tx is Invoke
- `tx.nonce() == 1`
- `account_nonce == 0` (account not yet on-chain)
- `account_tx_in_pool_or_recent_block(sender)` returns `true` [3](#0-2) 

The comment claims this is sufficient because the account "either has a deploy_account transaction or transactions with future nonces that passed validations." But the actual check is: [4](#0-3) 

`tx_pool.contains_account` returns `true` for **any** transaction from that address — including a previously admitted Invoke(nonce=1) from the same attacker, or the victim's own `DeployAccount`. There is no type-specific check.

When `skip_validate=true`, `run_validate_entry_point` is called with `skip_validate=true`: [5](#0-4) 

The blockifier `StatefulValidator::validate` is called with `execution_flags.validate = false`, meaning the account's `__validate__` Cairo entry point is never executed at the gateway stage.

### Impact Explanation

An attacker can submit an Invoke(nonce=1) transaction for any undeployed account address that has a pending `DeployAccount` in the mempool, with an **arbitrary signature** (or no valid signature at all). The gateway admits this transaction to the mempool without running the account's `__validate__` entry point. The transaction carries the wrong signer's payload and is sequenced into a block. This matches:

> **High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.**

Secondary consequence: the admitted invalid Invoke occupies a mempool slot at the victim's nonce=1, potentially blocking or delaying the victim's own legitimate Invoke(nonce=1) (which would be rejected as `DuplicateNonce` unless it pays sufficient fee escalation to replace the attacker's tx).

### Likelihood Explanation

- Requires the victim to have a pending `DeployAccount` in the mempool (observable on-chain/mempool).
- The attacker only needs to know the victim's sender address (public information from the `DeployAccount` transaction).
- No privileged access, no special contract, no cryptographic material needed.
- The attack window is the time between the victim's `DeployAccount` entering the mempool and being committed to a block.
- Exploitable by any unprivileged network participant.

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a `DeployAccount` transaction for the sender address is present in the pool. Add a dedicated mempool query such as `deploy_account_in_pool(address) -> bool` that inspects the transaction type, rather than relying on any-address presence. Alternatively, restrict the skip logic to only apply when the mempool confirms the pending transaction at nonce=0 is of type `DeployAccount`.

### Proof of Concept

1. Victim submits `DeployAccount` for address `A` (class_hash `C`, salt `S`). The tx enters the mempool. `account_tx_in_pool_or_recent_block(A)` now returns `true`.

2. Attacker observes the pending `DeployAccount` for `A` in the mempool.

3. Attacker constructs `Invoke(sender=A, nonce=1, calldata=<arbitrary>, signature=<garbage>)` and submits it to the gateway.

4. Gateway flow:
   - `get_nonce_from_state(A)` → `0` (A not yet on-chain). [6](#0-5) 
   - `validate_nonce`: `0 <= 1 <= max_allowed_nonce_gap` → passes. [7](#0-6) 
   - `validate_by_mempool`: nonce=1 >= account_nonce=0, no existing tx at nonce=1 → passes.
   - `skip_stateful_validations`: nonce==1 && account_nonce==0 → checks `account_tx_in_pool_or_recent_block(A)` → `true` (victim's `DeployAccount` is in pool) → returns `true`. [8](#0-7) 
   - `run_validate_entry_point(skip_validate=true)` → blockifier called with `execution_flags.validate = false` → `__validate__` **not executed**. [9](#0-8) 

5. Attacker's Invoke(nonce=1) with garbage signature is admitted to the mempool. The victim's legitimate Invoke(nonce=1) is now blocked by `DuplicateNonce` unless it pays fee escalation above the attacker's tx.

    async fn run_validate_entry_point(
        &mut self,
        executable_tx: &ExecutableTransaction,
        skip_validate: bool,
    ) -> StatefulTransactionValidatorResult<()> {
        let only_query = false;
        let charge_fee = enforce_fee(executable_tx, only_query);
        let strict_nonce_check = false;
        let execution_flags =
            ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };

        let account_tx = AccountTransaction { tx: executable_tx.clone(), execution_flags };

        // Build block context.
        let mut versioned_constants = VersionedConstants::get_versioned_constants(
            self.config.versioned_constants_overrides.clone(),
        );
        // The validation of a transaction is not affected by the casm hash migration.
        versioned_constants.disable_casm_hash_migration();

        let mut block_info = self.gateway_fixed_block_state_reader.get_block_info().await?;
        block_info.block_number = block_info.block_number.unchecked_next();
        let block_context = BlockContext::new(
            block_info,
            self.chain_info.clone(),
            versioned_constants,
            BouncerConfig::max(),
        );

        // Move state into the blocking task and run CPU-heavy validation.
        let state_reader_and_contract_manager = self.take_state_reader_and_contract_manager();

        let cur_span = Span::current();
        #[allow(clippy::result_large_err)]
        tokio::task::spawn_blocking(move || {
            cur_span.in_scope(|| {
                let state = CachedState::new(state_reader_and_contract_manager);
                let mut blockifier_validator = StatefulValidator::create(state, block_context);
                blockifier_validator.validate(account_tx)
            })
        })
        .await
        .map_err(|e| StarknetError {
            code: StarknetErrorCode::UnknownErrorCode(
                "StarknetErrorCode.InternalError".to_string(),
            ),
            message: format!("Blocking task join error when running the validate entry point: {e}"),
        })?
        .map_err(|e| StarknetError {
            code: StarknetErrorCode::KnownErrorCode(KnownStarknetErrorCode::ValidateFailure),
            message: e.to_string(),
        })?;
        Ok(())
    }

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
