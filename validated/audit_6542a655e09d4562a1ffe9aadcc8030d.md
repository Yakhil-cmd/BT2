### Title
WalletConnect `chia_sendTransaction` Accepts Dapp-Controlled `wallet_id` Hidden from Confirmation Dialog, Enabling Silent Wallet Substitution - (File: packages/gui/src/electron/commands/Commands.ts)

### Summary
The `chia_sendTransaction` WalletConnect command accepts a `wallet_id` parameter from the dapp but marks it `hide: true` in the confirmation schema, so it is never shown to the user in the approval dialog. A connected dapp can silently substitute any wallet ID (e.g., a high-balance savings wallet instead of the default spending wallet), and the user will approve the transaction without knowing which wallet is being drained.

### Finding Description
In `Commands.ts`, the `chia_wallet.send_transaction` command schema declares `wallet_id` with `hide: true`:

```ts
{ name: 'wallet_id', label: () => i18n._(/* i18n */ { id: 'Wallet Id' }), type: 'number', hide: true },
```

The dapp-facing `chia_sendTransaction` command inherits this params list and sets a default of `wallet_id: 1`:

```ts
dapp: [{ command: 'chia_sendTransaction', requiresSync: true, defaults: { wallet_id: 1 } }],
```

`parseDappParams` validates that `wallet_id` is in the allowlist (it is, inherited from the parent schema) and accepts any dapp-supplied value. The test suite explicitly confirms this behavior:

```ts
parseDappParams('chia_sendTransaction', serialize({ amount: '1', fee: '0', address: 'txch1address', walletId: 2 }))
// → { wallet_id: 2, ... }
```

Because `wallet_id` has `hide: true`, the confirmation dialog rendered by `Confirm.tsx` (which iterates `rows` derived from the schema) omits it entirely. The user sees only `amount`, `fee`, and `address`. The actual RPC call to `chia_wallet.send_transaction` uses whatever `wallet_id` the dapp supplied.

### Impact Explanation
A malicious dapp paired via WalletConnect can set `walletId` to any wallet ID belonging to the logged-in fingerprint (e.g., a high-balance XCH wallet, wallet ID 3 instead of the default 1). The user's confirmation dialog shows the correct destination address and amount but gives no indication of the source wallet. The user approves believing funds leave their default wallet; instead, a different wallet is drained. This satisfies the High criterion: "causes a user to approve… the wrong asset, identity, amount, destination, or status" — specifically, the wrong source wallet/asset.

### Likelihood Explanation
Any dapp that has been granted `chia_sendTransaction` permission can exploit this on every call. No additional privileges, leaked keys, or social engineering beyond the initial WalletConnect pairing are required. The dapp simply includes `walletId: N` in its request payload.

### Recommendation
Either:
1. Remove `wallet_id` from the dapp-accessible params list for `chia_sendTransaction` and always use the hardcoded default (`wallet_id: 1`), so the dapp cannot influence which wallet is spent; or
2. Remove `hide: true` from the `wallet_id` param so the source wallet is always visible in the confirmation dialog, giving the user full awareness of which wallet will be debited before they approve.

### Proof of Concept
1. Pair a dapp with the Chia GUI via WalletConnect and obtain `chia_sendTransaction` permission.
2. Send the following WalletConnect session request:
   ```json
   {
     "method": "chia_sendTransaction",
     "params": {
       "amount": "1000000000",
       "fee": "0",
       "address": "xch1attacker...",
       "walletId": 3
     }
   }
   ```
3. The confirmation dialog displays amount `0.001 XCH`, fee `0 XCH`, address `xch1attacker...` — no wallet ID is shown.
4. The user approves. The RPC call `chia_wallet.send_transaction` is dispatched with `wallet_id: 3`, draining wallet 3 (e.g., the user's savings wallet) rather than the expected default wallet 1. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
