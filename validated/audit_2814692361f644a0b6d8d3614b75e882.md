Audit Report

## Title
Permanent Stuck Entry in `ongoing_xnet_reshares` via Empty-Receiver `reshare_chain_key` Call — (`rs/consensus/idkg/src/payload_builder/resharing.rs`)

## Summary
Any canister can call `reshare_chain_key` on the management canister with an empty `nodes` list. Because `get_set_of_nodes()` only checks for duplicates and not for emptiness, a `ReshareChainKeyContext` with an empty node set is stored in replicated state. `initiate_reshare_requests` then inserts a `ReshareOfUnmaskedParams` with empty receivers into `ongoing_xnet_reshares`. Every subsequent call to `update_completed_reshare_requests` fails at `translate()` (because `IDkgReceivers::new` returns `ReceiversEmpty`), logs a warning, and `continue`s — leaving the entry permanently stuck with no removal path, causing unbounded state growth and per-block overhead.

## Finding Description

**Step 1 — No empty-node guard in `get_set_of_nodes()`.**
`ReshareChainKeyArgs::get_set_of_nodes()` iterates the node list checking only for duplicates. An empty `nodes` field returns `Ok(BTreeSet::new())` without error. [1](#0-0) 

**Step 2 — Empty context stored in replicated state.**
`reshare_chain_key` calls `args.get_set_of_nodes()?` and stores the result (which may be an empty set) directly into `subnet_call_context_manager` as a `ReshareChainKeyContext`. No further validation of `nodes` occurs. [2](#0-1) 

**Step 3 — `initiate_reshare_requests` inserts the stuck entry unconditionally.**
`receivers` is built from `request.receiving_node_ids` (empty), and `ReshareOfUnmaskedParams::new` is called with that empty set. The entry is inserted into `ongoing_xnet_reshares` without any receiver-count guard. [3](#0-2) 

**Step 4 — `translate()` always fails for the stuck entry.**
`IDkgTranscriptParamsRef::translate` calls `IDkgTranscriptParams::new` → `IDkgReceivers::new(empty_set)` → `Err(IDkgParamsValidationError::ReceiversEmpty)`. The error is mapped to `TranscriptParamsError::ParamsValidation`. [4](#0-3) 

**Step 5 — No removal path for the stuck entry.**
`update_completed_reshare_requests` catches the `translate()` error, logs it with `every_n_seconds => 10` rate limiting, and `continue`s. The only call to `ongoing_xnet_reshares.remove` is inside the `completed_reshares` loop, which is never reached for this entry because `translate()` fails before `InitialIDkgDealings::new` is called. [5](#0-4) [6](#0-5) 

**Deduplication constraint.** `IDkgReshareRequest` keys are deduplicated by `(master_key_id, receiving_node_ids, registry_version)`. Multiple calls with the same registry version map to the same key and are deduped. However, as the registry version advances (which happens regularly on mainnet), each new registry version allows a fresh stuck entry to be inserted, making accumulation unbounded over time.

## Impact Explanation

Each stuck entry is iterated every block in `update_completed_reshare_requests`. As entries accumulate across registry versions, per-block processing overhead grows linearly. The `ongoing_xnet_reshares` map is part of the IDKG payload included in consensus blocks; unbounded growth increases block processing time and state size. The calling canister's inter-canister callback is permanently unresolved. This constitutes an **application/platform-level DoS and certified-state disruption** impact — **High ($2,000–$10,000)**.

## Likelihood Explanation

Any canister on any subnet hosting a chain key can trigger this. The attacker only needs a valid key ID (public information from the registry) and cycles to pay for the call. There is no rate limit or authorization gate beyond key existence. As the registry version advances over time, the attacker can insert additional stuck entries. The attack is cheap and repeatable.

## Recommendation

1. **Validate non-empty receivers in `get_set_of_nodes()`** in `rs/types/management_canister_types/src/lib.rs`:
   ```rust
   if set.is_empty() {
       return Err(UserError::new(
           ErrorCode::InvalidManagementPayload,
           "nodes list must not be empty",
       ));
   }
   ```
2. **Add a guard in `initiate_reshare_requests`** before inserting into `ongoing_xnet_reshares` in `rs/consensus/idkg/src/payload_builder/resharing.rs`:
   ```rust
   if receivers.is_empty() {
       warn!(...); continue;
   }
   ```
3. **Add a removal path** in `update_completed_reshare_requests` for entries whose `translate()` fails with a non-transient error (e.g., `ReceiversEmpty`), so they do not accumulate indefinitely.

## Proof of Concept

```rust
// Unit test sketch in rs/consensus/idkg/src/payload_builder/resharing.rs
let mut payload = /* payload with valid current key transcript */;
let empty_request = IDkgReshareRequest {
    master_key_id: valid_key_id,
    receiving_node_ids: vec![],   // empty
    registry_version: RegistryVersion::from(1),
};

initiate_reshare_requests(
    &mut payload,
    BTreeSet::from([empty_request.clone()]),
    None,
    &no_op_logger(),
);
assert_eq!(payload.ongoing_xnet_reshares.len(), 1); // entry inserted

let transcript_builder = TestIDkgTranscriptBuilder::new();
for _ in 0..100 {
    update_completed_reshare_requests(
        &mut payload,
        &BTreeMap::new(),
        &block_reader,
        &transcript_builder,
        None,
        &no_op_logger(),
    );
    // Entry is never removed; translate() always returns ReceiversEmpty
    assert_eq!(payload.ongoing_xnet_reshares.len(), 1);
    assert!(payload.xnet_reshare_agreements.is_empty());
}
```
