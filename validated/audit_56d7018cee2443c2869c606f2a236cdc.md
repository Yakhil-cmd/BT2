Audit Report

## Title
SNS Treasury Transfer Cap Bypassed via Stale Genesis-Price Oracle for SNS Token Valuation — (`rs/sns/governance/token_valuation/src/lib.rs`)

## Summary
The SNS governance treasury protection system computes SNS token value using only the genesis swap price adjusted for supply inflation, with no mechanism to track current market price. When an SNS token appreciates significantly post-genesis, the stale price undervalues tokens, causing the 7-day 300,000 XDR hard cap to be bypassed. A governance-adopted `TransferSnsTreasuryFunds` or `MintSnsTokens` proposal can drain treasury tokens whose actual market value is orders of magnitude above the intended cap.

## Finding Description
`IcpsPerSnsTokenClient::fetch_icps_per_sns_token()` in `rs/sns/governance/token_valuation/src/lib.rs` (lines 314–415) computes the SNS token price as:

```
current_icps_per_token = (1 / genesis_sns_tokens_per_icp) / (current_supply / initial_supply)
```

The `genesis_sns_tokens_per_icp` is fetched from the swap canister's `get_derived_state` endpoint. After swap finalization, `derived_state()` in `rs/sns/swap/src/swap.rs` (lines 2992–2995) computes `sns_tokens_per_icp` purely from the frozen post-swap values of `tokens_available_for_swap` and `participant_total_icp_e8s`. These values never change after the swap completes, so `genesis_sns_tokens_per_icp` is permanently fixed at the swap finalization price.

The valuation model adjusts only for token supply inflation — it has no secondary price source, no deviation threshold, and no staleness check.

This stale valuation feeds directly into `ProposalsAmountTotalUpperBound::in_tokens()` in `rs/sns/governance/proposals_amount_total_limit/src/lib.rs` (lines 88–110), which computes:

```
allowance_tokens = MAX_XDR (300,000) * (1 / xdrs_per_token_stale)
```

The existing `clamp_xdrs_per_icp` guard (lines 137–140) only clamps the ICP/XDR rate; there is no analogous clamp on `icps_per_token`. The code comment at lines 59–63 explicitly acknowledges the absence of a `MAX_XDRS_PER_ICP` clamp but does not address the analogous unbounded `icps_per_token` problem.

The valuation is snapshotted at proposal submission time (stored as `ActionAuxiliary::TransferSnsTreasuryFunds(valuation)` in `rs/sns/governance/src/proposal.rs` lines 570–593) and reused at execution time (`rs/sns/governance/src/governance.rs` lines 3000–3005). Both the submission-time and execution-time checks use the same stale genesis-anchored price.

## Impact Explanation
The 7-day treasury transfer cap is the primary on-chain safeguard against large SNS treasury drains — it is explicitly designed to block even governance-approved proposals from exceeding 300,000 XDR per 7-day window. When an SNS token has appreciated 100× since genesis with 2× supply inflation, the stale price is 200× lower than market, causing `allowance_tokens` to be 200× larger than intended. A single governance-adopted proposal can drain tokens worth 60,000,000 XDR when the cap should be 300,000 XDR. This matches the High impact category: unauthorized access to canister-controlled funds (SNS treasury) with significant per-target constraints (governance majority required), with potential losses well above $1M for successful SNS projects.

## Likelihood Explanation
Any SNS token that has appreciated since its genesis swap is affected — a normal and expected outcome for successful projects. The attacker requires sufficient voting power to pass a `TransferSnsTreasuryFunds` proposal through normal governance. No privileged access, key compromise, or threshold attack is required. The treasury cap is specifically designed to be a hard limit that applies even to governance-approved proposals; the vulnerability defeats this last-line-of-defense safeguard. The bypass magnitude scales with token appreciation, making it most severe for the most successful SNS DAOs.

## Recommendation
1. Replace the single genesis-price oracle with a dual-source approach: retain the inflation-adjusted genesis price as a reference, and integrate a current on-chain price source (e.g., a DEX TWAP or exchange-rate canister query).
2. Use the **higher** of the two prices when computing the treasury cap (conservative valuation), so the XDR cap is not bypassed when the token appreciates.
3. At minimum, add a `MAX_ICPS_PER_TOKEN` clamp in `clamp_xdrs_per_icp` (or a parallel `clamp_icps_per_token` function) analogous to the existing `MIN_XDRS_PER_ICP`, to bound how far the stale genesis price can diverge from a reasonable current estimate.
4. Add a staleness guard that rejects the genesis price if it deviates more than a configurable threshold from the secondary source.

## Proof of Concept
**Setup**: Genesis swap at 10 SNS tokens/ICP. Supply doubled since genesis (2× inflation). Current market: 10 ICP/SNS token (100× appreciation). ICP/XDR rate: 10 XDR/ICP. Treasury: 1,000,000 SNS tokens.

**Stale valuation computed by `fetch_icps_per_sns_token()`**:
- `initial_icps_per_sns_token` = 1/10 = 0.1
- `total_inflation` = 2
- `icps_per_token` = 0.1 / 2 = **0.05 ICP/token**
- `xdrs_per_token` = 0.05 × 10 = **0.5 XDR/token**
- Treasury XDR value = 1,000,000 × 0.5 = 500,000 XDR → "large" regime → `MAX_XDR` = 300,000 applies
- `allowance_tokens` = 300,000 / 0.5 = **600,000 tokens**

**Actual market value of allowance**:
- 600,000 × 10 ICP/token × 10 XDR/ICP = **60,000,000 XDR**

**Intended cap**: 300,000 XDR. **Enforced cap**: 60,000,000 XDR — **200× bypass**.

The attacker submits a `TransferSnsTreasuryFunds` proposal for 600,000 SNS tokens. The check in `treasury_valuation_if_proposal_amount_is_small_enough_or_err` passes (600,000 ≤ 600,000). The proposal is adopted and executed via `perform_transfer_sns_treasury_funds`, draining 60,000,000 XDR worth of tokens.

A deterministic unit test can reproduce this by constructing a mock `IcpsPerSnsTokenClient` returning the stale price, a mock `XdrsPerIcpClient` returning 10, and verifying that `transfer_sns_treasury_funds_7_day_total_upper_bound_tokens` returns 600,000 tokens against a treasury valued at 500,000 XDR.
