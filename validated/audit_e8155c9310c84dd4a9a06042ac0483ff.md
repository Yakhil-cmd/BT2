### Title
Missing WebAuthn `origin` Validation Allows Phishing-Site Assertions to Authorize Transactions — (`model/flow/webauthn.go`)

### Summary

`validateWebAuthNExtensionData` parses `clientDataJSON.origin` into `CollectedClientData` but never checks it against any expected relying-party origin. A WebAuthn assertion obtained from a phishing page — carrying the correct challenge but an arbitrary `origin` — passes all FVM signature checks and authorizes the victim's account.

### Finding Description

`CollectedClientData` captures the `origin` field: [1](#0-0) 

`validateWebAuthNExtensionData` validates `type` and `challenge` but performs **no check on `clientData.Origin`**: [2](#0-1) 

The only `rpIdHash` check is a negative guard (must not equal `TransactionDomainTag`), not a positive check against the expected RP ID hash: [3](#0-2) 

The existing unit test explicitly labels an empty `origin` as **valid** and asserts `extensionOk: true`: [4](#0-3) 

The integration test helper hard-codes `"https://testing.com"` as the origin with no expectation that any other value would be rejected: [5](#0-4) 

This function is the sole WebAuthn validation gate called at both the Access-node level: [6](#0-5) 

and at FVM execution time: [7](#0-6) 

### Impact Explanation

The W3C WebAuthn Level 3 spec (§7.2 steps 11 and 13) requires the relying party to:
- Verify `C.origin` matches the expected origin.
- Verify `rpIdHash` is the SHA-256 of the expected RP ID.

Both checks are absent. An attacker who tricks a victim into authenticating on a phishing page (where the challenge is set to `SHA2_256(TransactionDomainTag || maliciousPayload)`) obtains a fully valid WebAuthn assertion — correct cryptographic signature, correct challenge, but `origin='https://evil.com'`. Flow accepts this assertion without complaint, executing the malicious transaction under the victim's account authority.

### Likelihood Explanation

The attack requires:
1. The victim has a WebAuthn key registered on their Flow account.
2. The attacker constructs a malicious transaction, computes its challenge, and presents it via a phishing page.
3. The victim authenticates on the phishing page (e.g., Touch ID / Face ID prompt).

If the Flow wallet uses a broad RP ID (e.g., `flow.com`), any attacker-controlled subdomain (`evil.flow.com`) can request assertions for that RP ID; the browser permits it, and the resulting `origin` differs from the legitimate wallet's origin — a difference Flow never checks. Even without subdomain access, a native-app or custom WebAuthn client can set `origin` to any string.

The test suite already proves the empty-origin path is accepted end-to-end, confirming the missing check is reachable on an unmodified emulator.

### Recommendation

Inside `validateWebAuthNExtensionData`, after unmarshalling `clientData`, add:

1. **Origin check**: compare `clientData.Origin` against a protocol-defined allowlist of legitimate Flow wallet origins (or a configurable set of trusted origins). Reject if not in the list.
2. **rpIdHash check**: compute `SHA2_256(expectedRPID)` and compare it to `rpIdHash` from `authenticatorData`. Reject if they differ.

These are mandatory steps per the WebAuthn spec and are the primary defenses against phishing-based credential forwarding.

### Proof of Concept

On a localnet emulator:

1. Register a WebAuthn key on account A.
2. Construct a malicious transaction (e.g., transfer tokens from A).
3. Compute `challenge = SHA2_256(TransactionDomainTag || envelopeMessage)`.
4. Build `clientDataJSON = {"type":"webauthn.get","challenge":"<base64url(challenge)>","origin":"https://evil.com"}`.
5. Build `authenticatorData` with any random `rpIdHash` (≠ `TransactionDomainTag`) and `UP` flag set.
6. Sign `authenticatorData || SHA256(clientDataJSON)` with account A's private key.
7. RLP-encode `WebAuthnExtensionData{AuthenticatorData, ClientDataJson}`, prepend `0x01`.
8. Submit via Access API.

Expected (broken) result: transaction sealed successfully — `origin` is never checked.

Repeat with `origin=""` — also accepted, as confirmed by the existing test case at `model/flow/transaction_test.go:510–518`. [8](#0-7)

```go
func validateWebAuthNExtensionData(extensionData []byte, payload []byte) (bool, []byte) {
	// See FLIP 264 for more details
	if len(extensionData) == 0 {
		return false, nil
	}
	rlpEncodedWebAuthnData := extensionData[1:]
	decodedWebAuthnData := &WebAuthnExtensionData{}
	if err := rlp.DecodeBytes(rlpEncodedWebAuthnData, decodedWebAuthnData); err != nil {
		return false, nil
	}

	clientData, err := decodedWebAuthnData.GetUnmarshalledCollectedClientData()
	if err != nil {
		return false, nil
	}

	// base64url decode the challenge, as that's the encoding used client side according to https://www.w3.org/TR/webauthn-3/#dictionary-client-data
	clientDataChallenge, err := base64.RawURLEncoding.DecodeString(clientData.Challenge)
	if err != nil {
		return false, nil
	}

	if strings.Compare(clientData.Type, WebAuthnTypeGet) != 0 || len(clientDataChallenge) != webAuthnChallengeLength {
		// invalid client data
		return false, nil
	}

	// make sure the challenge is the hash of the transaction payload
	hasher := hash.NewSHA2_256()
	_, err = hasher.Write(TransactionDomainTag[:])
	if err != nil {
		return false, nil
	}
	_, err = hasher.Write(payload)
	if err != nil {
		return false, nil
	}
	computedChallenge := hasher.SumHash()
	if !computedChallenge.Equal(clientDataChallenge) {
		return false, nil
	}

	// Validate authenticatorData
	if len(decodedWebAuthnData.AuthenticatorData) < webAuthnExtensionDataMinimumLength {
		return false, nil
	}

	// extract rpIdHash, userFlags, sigCounter, extensions
	rpIdHash := decodedWebAuthnData.AuthenticatorData[:webAuthnChallengeLength]
	userFlags := decodedWebAuthnData.AuthenticatorData[webAuthnChallengeLength]
	extensions := decodedWebAuthnData.AuthenticatorData[webAuthnExtensionDataMinimumLength:]
	if bytes.Equal(TransactionDomainTag[:], rpIdHash) {
		return false, nil
	}

	// validate user flags according to FLIP 264
	if err := validateFlags(userFlags, extensions); err != nil {
		return false, nil
	}

	clientDataHash := hash.NewSHA2_256().ComputeHash(decodedWebAuthnData.ClientDataJson)

	return true, slices.Concat(decodedWebAuthnData.AuthenticatorData, clientDataHash)
}
```
