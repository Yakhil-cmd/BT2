### Title
Unchecked Exit-Precompile Return Value in `withdrawToNear`/`withdrawToEthereum` Causes Permanent Token Loss - (File: etc/eth-contracts/contracts/EvmErc20.sol)

### Summary
`EvmErc20` and `EvmErc20V2` irreversibly burn a caller's ERC-20 tokens before invoking the Aurora exit precompile via inline assembly. The assembly `call` return value (`res`) is captured but never checked. If the precompile call fails for any reason, the tokens are permanently destroyed with no corresponding withdrawal initiated and no revert to protect the user.

### Finding Description
Both `withdrawToNear()` and `withdrawToEthereum()` follow the same unsafe pattern:

1. `_burn(_msgSender(), amount)` — tokens are irreversibly destroyed.
2. An inline-assembly `call` is made to the hardcoded exit precompile address.
3. The return value `res` is stored in a local variable but is **never inspected**; there is no `require(res != 0)` or equivalent guard.

In `EvmErc20.sol` `withdrawToEthereum`: [2](#0-1)

If the precompile `call` returns `0` (failure), the EVM transaction still succeeds. The caller's tokens are already burned, but no exit event is recorded and no NEAR/Ethereum withdrawal is triggered. There is no recovery path.

### Impact Explanation
**Critical — Permanent destruction of user funds.** Any token holder who calls `withdrawToNear()` or `withdrawToEthereum()` at a moment when the precompile call fails will permanently lose their tokens. The burn is irreversible, and because the transaction does not revert, the user receives no indication that the withdrawal was not processed. Funds are permanently lost with no on-chain recovery mechanism. [5](#0-4) 

### Likelihood Explanation
**Low.** The precompile addresses (`0xe9217bc70b7ed1f598ddd3199e80b093fa71124f` for `exitToNear`, `0xb0bd02f6a392af548bdf1cfaee5dfa0eefcc8eab` for `exitToEthereum`) are hardcoded and are registered in a correctly deployed Aurora Engine mainnet instance. However, the silent failure path is reachable when:
- The engine is paused and the precompile rejects the call.
- The contract is deployed in a Silo or non-standard environment where the precompile is not registered (calling an address with no code returns EVM success with empty data, so `res` may be `1` but nothing happens).
- A future upgrade changes the precompile address without updating the contract bytecode. [6](#0-5) 

### Recommendation
Check the return value of the precompile call and revert if it indicates failure:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

This ensures tokens are only burned if the withdrawal precompile call succeeds. Apply the same fix to both `withdrawToNear` and `withdrawToEthereum` in both `EvmErc20.sol` and `EvmErc20V2.sol`.

### Proof of Concept
1. User holds `amount` tokens in `EvmErc20` and calls `withdrawToNear(recipient, amount)`.
2. `_burn(_msgSender(), amount)` executes — tokens are irreversibly destroyed.
3. The assembly `call` to `0xe9217bc70b7ed1f598ddd3199e80b093fa71124f` fails (e.g., engine is paused, or precompile not registered in a Silo deployment) — returns `0`.
4. No `require` or `if iszero(res) { revert(...) }` guard exists, so the transaction succeeds.
5. User's tokens are permanently gone; no withdrawal is recorded on NEAR. The user has no recourse. [1](#0-0) [7](#0-6)
