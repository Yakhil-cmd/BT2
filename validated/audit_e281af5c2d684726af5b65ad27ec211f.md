### Title
Unprivileged Caller Can Invoke `update_rewards` with `disable_rewards: true` to Permanently Block Consensus Reward Distribution for All Stakers - (File: `src/staking/staking.cairo`)

---

### Summary

The `update_rewards` function in the Staking contract is callable by **any address** and accepts a `disable_rewards` boolean parameter. When called with `disable_rewards: true`, the function advances the global `last_reward_block` checkpoint to the current block without distributing any rewards. Because `last_reward_block` is a single global storage slot, any subsequent legitimate call to `update_rewards` for the same block reverts with `REWARDS_ALREADY_UPDATED`. A malicious actor can exploit this to permanently suppress all consensus-era staking rewards by front-running every block.

---

### Finding Description

In `src/staking/staking.cairo`, the `IStakingRewardsManager::update_rewards` implementation (around line 1449) contains the following logic:

```cairo
fn update_rewards(ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool) {
    self.general_prerequisites();                                          // only checks pause
    let current_block_number = starknet::get_block_number();
    assert!(
        current_block_number > self.last_reward_block.read(),
        "{}", Error::REWARDS_ALREADY_UPDATED,
    );
    // ... staker active + non-zero balance checks ...
    self.last_reward_block.write(current_block_number);   // ← global write, no caller check
    if disable_rewards || self.is_pre_consensus() {
        return;                                            // ← exits without distributing rewards
    }
    // ... actual reward calculation and distribution ...
}
```

Key observations:

1. **No access control on the caller.** The spec explicitly states "Any address can execute." `general_prerequisites()` only checks the pause flag and zero-address caller; it does not restrict who may call with `disable_rewards: true`.

2. **`last_reward_block` is a single global slot.** It is not keyed by `staker_address`. One call for *any* active staker advances the global checkpoint for *all* stakers.

3. **`disable_rewards: true` advances the checkpoint without distributing rewards.** The rewards that would have been earned for that block are permanently lost — they are never queued, accumulated, or redistributable.

**Attack path:**

```
For every new Starknet block N:
  attacker calls update_rewards(any_active_staker_address, disable_rewards: true)
  → last_reward_block := N
  → no rewards distributed

Any legitimate call update_rewards(staker_X, disable_rewards: false) for block N:
  → assert(N > last_reward_block) FAILS → REWARDS_ALREADY_UPDATED
```

The attacker needs only to know one active staker address (public on-chain) and pay gas per block. No tokens, no privileged role, no bridge access required.

This is directly analogous to the CultureIndex M-03 pattern: a global ordering/gating variable (`last_reward_block` ↔ `topVotedPiece`) can be manipulated by an unprivileged actor to permanently block a critical protocol operation (reward distribution ↔ `dropTopVotedPiece`), with no easy on-chain recovery path.

---

### Impact Explanation

- Every block for which the attacker fires the call, the consensus-era staking rewards for **all** stakers are permanently destroyed — they cannot be retroactively recalculated or redistributed.
- As long as the attack is sustained, **no staker or pool member earns any consensus rewards**, freezing all unclaimed yield accrual.
- This matches **High: Permanent freezing of unclaimed yield or unclaimed royalties** from the allowed impact scope.

---

### Likelihood Explanation

- The function is publicly callable with no privilege requirement.
- The attacker needs only one valid active-staker address (trivially obtained from on-chain events).
- Starknet transaction fees are low; automating one call per block is economically feasible for a motivated attacker.
- There is no on-chain circuit-breaker that detects or prevents this pattern.

Likelihood: **Medium** (requires sustained gas spend, but zero privilege barrier and straightforward automation).

---

### Recommendation

Restrict who may call `update_rewards` with `disable_rewards: true`. Options:

1. **Caller must be the staker or their reward address** when `disable_rewards: true` is passed.
2. **Separate the two concerns**: expose a permissionless `update_rewards_for_block(staker_address)` (always distributes) and a restricted `skip_block_rewards(staker_address)` callable only by the staker or a governance role.
3. **Remove `disable_rewards` from the public interface** entirely and handle the "no-reward" case internally (e.g., detect it from staker state rather than an external flag).

---

### Proof of Concept

```
1. Deploy staking contract; set consensus_rewards_first_epoch to current epoch.
2. Staker A stakes and waits K epochs (balance becomes effective).
3. Attacker (any EOA) monitors the chain.
4. For each new block B:
     call staking.update_rewards(staker_A_address, disable_rewards=true)
     → last_reward_block = B, no rewards distributed
5. Staker A calls update_rewards(staker_A_address, disable_rewards=false) for block B:
     → panics: REWARDS_ALREADY_UPDATED
6. Staker A claims rewards → zero.
7. Repeat indefinitely: staker A (and every other staker) earns zero consensus rewards.
