### Title
Pool Admin Can Set Uncapped Per-Bin Additional Fees, Bypassing the Hard Spread Fee Cap - (File: metric-core/contracts/MetricOmmPoolFactory.sol)

### Summary

The factory enforces hard caps on spread fees (`HARD_MAX_SPREAD_FEE_E6 = 200_000`, i.e. 20%) for all main fee setters, but `setPoolBinAdditionalFees` passes `addFeeBuyE6`/`addFeeSellE6` directly to the pool with **no cap validation**. Because bin additional fees are added on top of the base spread fee in every swap, the pool admin can push the total effective per-bin fee well above the hard cap.

### Finding Description

`MetricOmmPoolFactory` defines two hard constants that bound every fee path:

```
uint24 internal constant HARD_MAX_SPREAD_FEE_E6   = 200_000;   // 20 %
uint24 internal constant HARD_MAX_NOTIONAL_FEE_E8 = 1_000_000; // 100 %
```

Every main fee setter enforces these bounds:

- `setFeeCaps` — checks against `HARD_MAX_*` [1](#0-0) 
- `setPoolAdminFees` — checks `newAdminSpreadFeeE6 > maxAdminSpreadFeeE6` [2](#0-1) 
- `setPoolProtocolFee` — checks `newProtocolSpreadFeeE6 > maxProtocolSpreadFeeE6` [3](#0-2) 

However, `setPoolBinAdditionalFees` performs **no cap check at all** — it forwards the caller-supplied values straight to the pool:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
``` [4](#0-3) 

The pool's `setBinAdditionalFees` likewise applies no cap — it only validates the bin index:

```solidity
function setBinAdditionalFees(int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external onlyFactory nonReentrant(PoolActions.SET_BIN_ADDITIONAL_FEES)
{
    if (bin < LOWEST_BIN || bin > HIGHEST_BIN) revert InvalidBinIndex(bin);
    BinState storage s = _binStates[bin];
    s.addFeeBuyE6 = addFeeBuyE6;
    s.addFeeSellE6 = addFeeSellE6;
    ...
}
``` [5](#0-4) 

In the swap path, the bin additional fee is added directly on top of the base spread fee:

```solidity
uint256 buyFeeX64 = baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6);
uint256 sellFeeX64 = baseFeeX64 + Math.mulDiv(binState.addFeeSellE6, ONE_X64, 1e6);
``` [6](#0-5) 

`uint16` allows values up to 65 535, i.e. **6.5535 %** of additional fee per bin. With `spreadFeeE6` already at the hard cap of 20 %, the total effective fee for a swap in that bin reaches **≈ 26.5 %**, far above the 20 % hard cap the factory is supposed to enforce.

The same absence of a cap check exists at pool-creation time: `_unpackAndValidateBinStates` unpacks `buyFee`/`sellFee` from the caller-supplied packed data and stores them without any bound check. [7](#0-6) 

### Impact Explanation

A pool admin (semi-trusted, not fully trusted) can set `addFeeBuyE6` or `addFeeSellE6` to `type(uint16).max = 65 535` on any bin. Every swap routed through that bin pays the uncapped additional fee on top of the already-capped spread fee. The excess fee accrues to LP providers in that bin, effectively extracting value from traders beyond the protocol-guaranteed hard cap. This breaks the admin-boundary invariant: the pool admin is supposed to be constrained by caps, but this path has no cap at all.

### Likelihood Explanation

Any pool admin can call `setPoolBinAdditionalFees` at any time with no preconditions beyond holding the `poolAdmin` role. The call requires no timelock and no owner approval. The trigger is a single transaction by a semi-trusted actor.

### Recommendation

Add a cap check in `setPoolBinAdditionalFees` (and symmetrically in `_unpackAndValidateBinStates` during pool creation) to ensure that `addFeeBuyE6` and `addFeeSellE6` do not exceed a defined maximum (e.g. `HARD_MAX_SPREAD_FEE_E6` or a dedicated `HARD_MAX_BIN_ADDITIONAL_FEE_E6` constant). The pool-level `setBinAdditionalFees` should also enforce this bound as a defence-in-depth check.

### Proof of Concept

1. Deploy a pool with `spreadFeeE6 = 200_000` (the hard cap, 20 %).
2. Pool admin calls `setPoolBinAdditionalFees(pool, 0, 65535, 65535)`.
3. No revert — `addFeeBuyE6 = 65535` is stored in `_binStates[0]`.
4. A trader swaps through bin 0. The effective buy fee is `baseFeeX64 + 65535/1e6`, i.e. ≈ 26.5 % — 6.5 percentage points above the hard cap.
5. The excess fee is retained in the bin and distributed to LP providers, bypassing the 20 % hard limit the factory is supposed to enforce.
