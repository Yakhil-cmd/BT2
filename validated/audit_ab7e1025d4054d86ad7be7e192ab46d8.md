### Title
`induced_state_diff` Unconditionally Includes Reverted DeclareV0/V1 Class Hash in `deprecated_declared_classes` — (`crates/apollo_rpc_execution/src/lib.rs`)

### Summary

In `execute_transactions`, the `deprecated_declared_class_hash` is extracted from a `DeclareV0`/`DeclareV1` transaction **before execution** and passed unconditionally to `induced_state_diff`. If the transaction reverts (returns `Ok(TransactionExecutionInfo { revert_error: Some(...), ... })`), the returned `ThinStateDiff` will still list the class hash in `deprecated_declared_classes`, even though the class was never actually declared.

### Finding Description

In `crates/apollo_rpc_execution/src/lib.rs` lines 733–753: [1](#0-0) 

The `deprecated_declared_class_hash` is set to `Some(class_hash)` for `DeclareV0`/`DeclareV1` unconditionally, before `blockifier_tx.execute(...)` is called. The result of execution is stored in `tx_execution_info_result` but is **not consulted** before `induced_state_diff` is called on line 752–753.

In `crates/apollo_rpc_execution/src/execution_utils.rs` lines 141–143: [2](#0-1) 

`deprecated_declared_classes` is populated purely from the caller-supplied `deprecated_declared_class_hash` argument — it is **not derived from the actual state maps** produced by `transactional_state.to_state_diff()`. For non-deprecated declares (V2+), the class hash would appear in `class_hash_to_compiled_class_hash` from the state maps, which blockifier correctly empties on revert. For deprecated declares, there is no such rollback path because the field bypasses the state maps entirely.

The contrast is explicit: `deployed_contracts`, `storage_diffs`, `nonces`, and `class_hash_to_compiled_class_hash` all come from `blockifier_state_diff` (which reflects the rolled-back state on revert), while `deprecated_declared_classes` comes from the pre-execution extraction. [3](#0-2) 

### Impact Explanation

When `simulate_transactions` (or any RPC path calling `execute_transactions`) processes a `DeclareV1` transaction whose `__validate_declare__` entry point reverts (e.g., account has insufficient fee balance when `charge_fee = true`, or the validation logic explicitly fails), the returned `TransactionExecutionOutput::induced_state_diff` will contain the class hash in `deprecated_declared_classes` even though the class was never committed. Any RPC consumer (block explorer, SDK, dApp) that trusts the simulation state diff to determine whether a class is declared will receive a wrong answer.

This matches the allowed impact: **"RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value."**

### Likelihood Explanation

- `DeclareV1` transactions run `__validate_declare__`, which is attacker-controlled code. An attacker can craft an account whose `__validate_declare__` always reverts.
- The simulation RPC endpoint is publicly accessible with no privilege requirement.
- The trigger requires only submitting a `simulate_transactions` call with a `DeclareV1` transaction from such an account.
- `DeclareV0` is less likely to revert in practice (no validation entry point in modern blockifier), but `DeclareV1` is sufficient.

### Recommendation

After executing the transaction, check whether `tx_execution_info_result` indicates a revert before passing `deprecated_declared_class_hash` to `induced_state_diff`. Specifically:

```rust
let reverted = tx_execution_info_result
    .as_ref()
    .map(|info| info.is_reverted())
    .unwrap_or(false); // Err case: function will propagate error anyway

let effective_deprecated_class_hash = if reverted {
    None
} else {
    deprecated_declared_class_hash
};

let state_diff = induced_state_diff(
    &mut transactional_state,
    effective_deprecated_class_hash,
)?;
```

Alternatively, derive `deprecated_declared_classes` from the actual state maps (as is done for non-deprecated classes via `class_hash_to_compiled_class_hash`), so revert rollback is automatically reflected.

### Proof of Concept

1. Deploy an account contract whose `__validate_declare__` always reverts.
2. Construct a `DeclareV1` transaction signed by that account, declaring any deprecated class.
3. Call `starknet_simulateTransactions` with `charge_fee = true` (so the revert path is taken rather than a hard error).
4. Observe that the returned `state_diff.deprecated_declared_classes` contains the class hash.
5. Verify the transaction's `execution_status` is `REVERTED`.
6. The state diff is wrong: it claims the class was declared when it was not.

A Rust unit test can replicate this by calling `execute_transactions` with a `DeclareV1` whose account reverts in `__validate_declare__`, then asserting `result[0].induced_state_diff.deprecated_declared_classes.is_empty()` — this assertion will **fail** with the current code. [4](#0-3)
