### Title
No Zero-Address Validation on `reward_address` in `change_reward_address` Allows Permanent Freezing of Unclaimed Yield - (File: `src/staking/staking.cairo`, `src/pool/pool.cairo`)

---

### Summary
Both `change_reward_address` in the staking contract and in the pool contract accept a zero address as a valid `reward_address` without any non-zero check. A staker or pool member who sets their reward address to `0x0` will have all future (and pending) reward claims permanently transferred to the zero address, making them irretrievable.

---

### Finding Description
`change_reward_address` in `src/staking/staking.cairo` performs only one validation: that the new address is not a registered token address. [1](#0-0) 

There is no `reward_address.is_non_zero()` guard. The zero address passes this check because it is not a registered token.

The same pattern exists in the pool contract: [2](#0-1) 

The initial `stake()` entry point has the same gap — it checks `!does_token_exist(reward_address)` but never asserts `reward_address.is_non_zero()`: [3](#0-2) 

And `enter_delegation_pool` in the pool contract mirrors this: [4](#0-3) 

---

### Impact Explanation
When `claim_rewards` is called on the staking contract, it reads `staker_info.reward_address` and calls `checked_transfer` to that address: [5](#0-4) 

If `reward_address` is `0x0`, the STRK tokens are transferred to the zero address and are permanently unrecoverable. The same applies in the pool contract's `claim_rewards`: [6](#0-5) 

**Impact: High — Permanent freezing of unclaimed yield.** All accumulated and future staker/delegator rewards are burned to the zero address with no recovery path.

---

### Likelihood Explanation
Any staker or pool member can call `change_reward_address` at any time. A front-end bug, a scripting error, or a deliberate self-harm action can pass `0x0`. The function is permissionlessly callable by the staker/pool member themselves — no privileged role is required. Once set to zero and `claim_rewards` is called, the yield is gone.

---

### Recommendation
Add a non-zero check in both `change_reward_address` implementations and in `stake` / `enter_delegation_pool`:

```cairo
assert!(reward_address.is_non_zero(), "{}", GenericError::ZERO_ADDRESS);
```

This mirrors the existing pattern already used elsewhere in the codebase (e.g., `add_token`): [7](#0-6) 

---

### Proof of Concept

1. Staker calls `stake(reward_address: 0x0, operational_address: <valid>, amount: <min_stake>)` — succeeds because zero address is not a token address.
2. Alternatively, an existing staker calls `change_reward_address(reward_address: 0x0)` — succeeds for the same reason.
3. Rewards accumulate over epochs via attestation.
4. Anyone calls `claim_rewards(staker_address)` — the staking contract executes `checked_transfer(recipient: 0x0, amount: rewards)`.
5. STRK tokens are sent to address zero and are permanently lost.

The same four-step flow applies to pool members via `enter_delegation_pool(reward_address: 0x0, ...)` or `pool.change_reward_address(0x0)` followed by `pool.claim_rewards(pool_member)`. [8](#0-7) [9](#0-8)
