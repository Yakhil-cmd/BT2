### Title
Unchecked Exit-Precompile Return Value in `withdrawToNear`/`withdrawToEthereum` Causes Permanent Fund Freeze — (File: `etc/eth-contracts/contracts/EvmErc20.sol`, `etc/eth-contracts/contracts/EvmErc20V2.sol`)

---

### Summary

`EvmErc20.sol` and `EvmErc20V2.sol` call `_burn` before invoking the exit precompile, and never check the precompile's return value. If the precompile call fails with `ExitError` (e.g., invalid NEAR recipient account ID), the ERC-20 tokens are permanently destroyed on Aurora while the corresponding NEP-141 tokens remain locked in the Aurora contract on NEAR forever — a permanent fund freeze with no recovery path.

---

### Finding Description

Both `EvmErc20.sol` and `EvmErc20V2.sol` implement `withdrawToNear` and `withdrawToEthereum` using the same pattern:

```solidity
// EvmErc20.sol lines 53-63
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // ← tokens destroyed here

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        // res is NEVER checked — silent failure
    }
}
```

The `_burn` is unconditional and executes before the precompile call. The assembly `call` captures the return value in `res` but never inspects it. If the precompile returns `ExitError`, the EVM `call` opcode returns `0` and execution continues normally — the Solidity function returns successfully with the tokens already gone.

The `ExitToNear` precompile (`engine-precompiles/src/native.rs`) returns `ExitError` (not `ExitFatal`) in multiple reachable paths:

- **Invalid NEAR account ID in `recipient`**: `parse_recipient` validates the account ID and returns `ExitError` for malformed input. The unit test at line 1136 confirms: `assert!(parse_recipient(b"test@.near").is_err())` and `assert!(parse_recipient(&[0xc2]).is_err())`.
- **ERC-20 not registered**: `get_nep141_from_erc20` returns `ExitError::Other(ERR_TARGET_TOKEN_NOT_FOUND)` if the ERC-20 address has no NEP-141 mapping.
- **Invalid message format**: `ERR_INVALID_MESSAGE` from `exit_base_token_to_near`.
- **Out of gas**: `ExitError::OutOfGas` if gas budget is exhausted.

The same pattern exists in `withdrawToEthereum` (lines 65-76 in `EvmErc20.sol`, lines 66-77 in `EvmErc20V2.sol`) calling the `ExitToEthereum` precompile, which also returns `ExitError` for `ERR_INVALID_RECIPIENT_ADDRESS`, `ERR_TARGET_TOKEN_NOT_FOUND`, etc.

Note: the **paused precompile** case returns `ExitFatal` (confirmed by `engine-tests/src/tests/pausable_precompiles.rs` line 51-54), which does revert the entire transaction including `_burn` — that case is safe. The vulnerability is specific to `ExitError` failure modes.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

When the precompile fails silently:
1. The user's ERC-20 tokens are burned and gone from Aurora's EVM state.
2. No NEAR cross-contract call (`ft_transfer` / `withdraw`) is ever scheduled.
3. The NEP-141 tokens remain locked in the Aurora contract on NEAR with no mechanism to release them.
4. Re-minting the ERC-20 tokens is impossible — `mint` is `onlyAdmin`.
5. The bridge accounting is permanently broken: the NEP-141 supply held by Aurora overestimates the real circulating ERC-20 supply by the burned amount.

This is the direct Aurora analog of the external report: burning on the EVM side without updating the NEAR side's accounting, causing the NEAR-side locked balance to permanently overestimate the real supply.

---

### Likelihood Explanation

**Medium.**

The `withdrawToNear` function accepts arbitrary `bytes memory recipient`. Any user who passes bytes that do not form a valid NEAR account ID (e.g., bytes containing `@`, `.` at the start, non-UTF-8 sequences, or an account ID exceeding 64 characters) will trigger the silent failure. This is a realistic user error given that the function accepts raw `bytes` with no Solidity-level validation. The `withdrawToEthereum` path is less likely to fail silently because the input format is more constrained (fixed 53 bytes), but `ERR_TARGET_TOKEN_NOT_FOUND` remains a reachable path.

---

### Recommendation

Reorder the operations so that `_burn` only executes after confirming the precompile call succeeded, or check `res` and revert on failure:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
_burn(_msgSender(), amount);
```

Alternatively, move `_burn` after the assembly block and add the `iszero` revert check. Apply the same fix to `withdrawToEthereum` in both `EvmErc20.sol` and `EvmErc20V2.sol`.

**Attack path:**
1. User holds ERC-20 tokens on Aurora (bridged from NEP-141 on NEAR).
2. User calls `withdrawToNear(bytes("\xc2"), amount)` — a non-UTF-8 byte sequence that is an invalid NEAR account ID.
3. `_burn` executes: ERC-20 balance decremented, tokens gone.
4. Precompile call returns `0` (`ExitError` from `parse_recipient`).
5. `res` is never checked; `withdrawToNear` returns successfully.
6. No `ft_transfer` promise is ever created on NEAR.
7. NEP-141 tokens remain locked in Aurora's NEAR account permanently. No admin function exists to release them.
