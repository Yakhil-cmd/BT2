### Title
EVM Bridge Deposit Credits EVM Balance from Unverified Resource `balance` Field ŌĆö (`fvm/evm/impl/impl.go`)

---

### Summary

The `InternalEVM.deposit` built-in function accepts `@AnyResource` as its `from` parameter and credits the target EVM address based solely on the `balance` field read from the passed resource, without verifying that the resource is a genuine `FlowToken.Vault`. An attacker can deploy a custom Cadence contract defining a resource with an inflated `balance: UFix64` field, call `InternalEVM.deposit` directly (bypassing the `@FlowToken.Vault`-typed public wrapper), and receive unearned EVM balance. They can then withdraw real FLOW tokens from the bridge escrow vault.

---

### Finding Description

**Root cause ŌĆö `fvm/evm/stdlib/contract.go` (type declaration)**

`InternalEVMTypeDepositFunctionType` declares the `from` parameter as `sema.AnyResourceType`, not `@FlowToken.Vault`: [1](#0-0) 

This means the Cadence type-checker permits any resource to be passed to `InternalEVM.deposit`.

**Root cause ŌĆö `fvm/evm/impl/impl.go` (runtime implementation)**

`newInternalEVMTypeDepositFunction` reads the `balance` field from whatever `CompositeValue` is passed and uses it to credit the EVM account, with no type identity check: [2](#0-1) 

Critically, the vault is **intentionally not destroyed** (the comment at lines 670ŌĆō673 explains this is by design to avoid a burn), so the attacker retains their fake resource after the deposit call.

**The public wrapper enforces `@FlowToken.Vault` ŌĆö but `InternalEVM` is a built-in accessible to all Cadence code**

The public-facing `EVM.EVMAddress.deposit(from: @FlowToken.Vault)` in `contract.cdc` does enforce the correct type at the Cadence level: [3](#0-2) 

However, `InternalEVM` is a built-in contract registered in the Cadence environment ŌĆö not a restricted internal module. Any user transaction can import and call `InternalEVM.deposit` directly, bypassing the `@FlowToken.Vault` type guard entirely.

The `CadenceOwnedAccount.deposit` path also delegates through the same unguarded `InternalEVM.deposit`: [4](#0-3) 

---

### Impact Explanation

The EVM bridge maintains a main FLOW token escrow vault. EVM account balances are tracked internally in the EVM state. If an attacker inflates their EVM account balance by depositing a fake resource, they can subsequently call `CadenceOwnedAccount.withdraw` to drain real FLOW tokens from the bridge escrow. This is a direct cross-VM asset loss: real FLOW tokens leave the bridge in exchange for a worthless custom resource. [5](#0-4) 

---

### Likelihood Explanation

The attack requires:
1. Deploying a Cadence contract that defines a resource with a `balance: UFix64` field ŌĆö trivially achievable by any account.
2. Calling `InternalEVM.deposit` directly from a user transaction ŌĆö possible because `InternalEVM` is a built-in accessible to all Cadence code.

No privileged access, leaked keys, or staked node control is required. Any unprivileged Flow account can execute this.

---

### Recommendation

1. **Change the `from` parameter type** in `InternalEVMTypeDepositFunctionType` from `sema.AnyResourceType` to the concrete `FlowToken.Vault` sema type, so the Cadence type-checker rejects non-`FlowToken.Vault` resources at the `InternalEVM.deposit` call site. [1](#0-0) 

2. **Add a runtime type identity check** in `newInternalEVMTypeDepositFunction` to assert that the passed `CompositeValue`'s qualified type ID matches `FlowToken.Vault` before reading the `balance` field. [6](#0-5) 

---

### Proof of Concept

```cadence
// Step 1: Attacker deploys a contract with a fake vault resource
contract FakeToken {
    resource FakeVault {
        pub var balance: UFix64
        init(balance: UFix64) { self.balance = balance }
    }
    pub fun createFakeVault(balance: UFix64): @FakeVault {
        return <- create FakeVault(balance: balance)
    }
}

// Step 2: Attacker transaction
import InternalEVM from <builtin>
import EVM from <evm_address>
import FakeToken from <attacker_address>

transaction {
    prepare(attacker: auth(BorrowValue) &Account) {
        // Create a fake vault with 1,000,000 FLOW balance ŌĆö no real tokens deposited
        let fakeVault <- FakeToken.createFakeVault(balance: 1_000_000.0)

        // Call InternalEVM.deposit directly, bypassing the @FlowToken.Vault type check
        let attackerEVMAddr: [UInt8; 20] = [/* attacker's EVM address bytes */]
        InternalEVM.deposit(from: <-fakeVault, to: attackerEVMAddr)
        // fakeVault is NOT destroyed ŌĆö attacker retains it

        // Step 3: Withdraw real FLOW tokens from the bridge escrow
        let coa <- EVM.createCadenceOwnedAccount()
        // Transfer EVM balance to COA, then withdraw
        let realVault <- coa.withdraw(balance: EVM.Balance(attoflow: 1_000_000_000_000_000_000_000_000))
        // realVault now contains real FLOW tokens drained from the bridge escrow
        destroy coa
        destroy realVault
    }
}
```

The Go implementation credits the EVM account with `1_000_000.0` FLOW based on the fake resource's `balance` field: [7](#0-6)

        fun deposit(from: @FlowToken.Vault) {

            let amount = from.balance

// Deposit deposits the token from the given vault into the flow evm main vault
// and update the account balance with the new amount
func (a *Account) Deposit(v *types.FLOWTokenVault) {
	defer a.fch.backend.StartChildSpan(trace.FVMEVMDeposit).End()

	bridge := a.fch.addressAllocator.NativeTokenBridgeAddress()
	bridgeAccount := a.fch.AccountByAddress(bridge, false)
	// Note: its not an authorized call
	res, err := a.fch.executeAndHandleCall(
		types.NewDepositCall(
			bridge,
			a.address,
			v.Balance(),
			bridgeAccount.Nonce(),
		),
		v.Balance(),
		false,
	)
	panicOnErrorOrInvalidOrFailedState(res, err)
