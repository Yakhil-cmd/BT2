### Title
Malicious Participant Can Inject Arbitrary Contribution Into CKD Aggregation Without Verification — (`src/confidential_key_derivation/protocol.rs`)

---

### Summary

In `do_ckd_coordinator`, the coordinator blindly aggregates `(big_y, big_c)` contributions from every participant with no proof of correctness. Any single malicious participant can send arbitrary group elements, causing the coordinator to produce a corrupted `CKDOutput`. The TEE that calls `unmask` will then derive a wrong confidential key, permanently breaking the CKD output for that invocation.

---

### Finding Description

The CKD protocol is designed so that the coordinator aggregates each participant's Lagrange-weighted ElGamal ciphertext share `(λ_i · Y_i, λ_i · C_i)` and the TEE recovers `msk · H(pk ‖ app_id)` by subtracting `app_sk · big_y` from `big_c`.

The coordinator's aggregation loop in `do_ckd_coordinator` is:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
``` [1](#0-0) 

There is no zero-knowledge proof, commitment-then-reveal, or any other check that the received `(big_y, big_c)` pair was honestly computed from the participant's actual signing share and the agreed `app_id`. The coordinator accepts and sums whatever bytes arrive over the channel.

The honest computation each participant is supposed to perform is:

```rust
let big_s = hash_point * private_share.to_scalar();   // x_i · H(pk‖app_id)
let big_c = big_s + app_pk * y.0;                     // x_i·H + y_i·A
let norm_big_y = big_y * lambda_i;
let norm_big_c = big_c * lambda_i;
``` [2](#0-1) 

A malicious participant ignores this computation entirely and sends attacker-chosen group elements `(P, Q)` instead. Because the coordinator sums all contributions without verification, the final `CKDOutput` is:

```
big_y_final  = Σ_{honest} λ_i·Y_i  + P
big_c_final  = Σ_{honest} λ_i·C_i  + Q
```

The TEE then computes `big_c_final − app_sk · big_y_final`, which equals `msk·H(pk‖app_id) + (Q − app_sk·P)`. The attacker controls `(P, Q)` freely, so the derived key is shifted by an arbitrary attacker-chosen offset `Q − app_sk·P`, making the output wrong and unusable.

The participant path sends its share directly with no commitment:

```rust
chan.send_private(waitpoint, coordinator, &(norm_big_y, norm_big_c))?;
``` [3](#0-2) 

The analog to the original report is exact: just as `adminWithdrawBacking` accepts any withdrawal amount with no tracking of what was legitimately injected, `do_ckd_coordinator` accepts any `(big_y, big_c)` pair with no tracking of what was legitimately contributed.

---

### Impact Explanation

**High — Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.**

The TEE receives a derived key that is not `msk · H(pk ‖ app_id)`. Every downstream operation that depends on that key (e.g., decrypting application secrets, authenticating to a service) will silently fail or produce wrong results. Because the corruption is undetectable by the TEE (it has no way to verify the expected output), honest parties cannot distinguish a corrupted run from a correct one.

---

### Likelihood Explanation

**High.** Any single participant in the CKD protocol can trigger this. The protocol requires all `n` participants to contribute (there is no threshold-of-`n` aggregation that could tolerate one bad actor). No special privilege, leaked key, or external assumption is required — the attacker only needs to be one of the listed participants and to send a malformed message over the existing channel.

---

### Recommendation

Add a zero-knowledge proof of correct contribution. Each participant should prove, alongside `(norm_big_y, norm_big_c)`, that:

1. `norm_big_y = λ_i · y_i · G` for some scalar `y_i` they know, and
2. `norm_big_c = λ_i · (x_i · H(pk‖app_id) + y_i · app_pk)` consistent with their committed public key share `x_i · G2`.

A Chaum–Pedersen DLEQ proof over G1 can prove (1) and (2) simultaneously without revealing `x_i` or `y_i`. The coordinator must verify every proof before including the contribution in the sum.

---

### Proof of Concept

```
Setup: 3 participants [A, B, C], coordinator = A, threshold = 3-of-3.
       Honest public key msk·G2 is known. app_pk = app_sk·G1.

Step 1. B and C compute their correct (norm_big_y, norm_big_c) and send to A.

Step 2. Malicious participant C instead sends:
          big_y_malicious = G1   (generator, arbitrary choice)
          big_c_malicious = G1

Step 3. Coordinator A aggregates:
          big_y_final = λ_A·Y_A + λ_B·Y_B + G1
          big_c_final = λ_A·C_A + λ_B·C_B + G1

Step 4. TEE calls unmask(app_sk):
          result = big_c_final − app_sk · big_y_final
                 = msk·H(pk‖app_id) + (G1 − app_sk·G1)
                 = msk·H(pk‖app_id) + (1 − app_sk)·G1

        The result is not msk·H(pk‖app_id). The TEE silently derives
        a wrong key. No error is raised anywhere in the protocol.
```
