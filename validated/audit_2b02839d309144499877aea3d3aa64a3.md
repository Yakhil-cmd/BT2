### Title
Truncating Floor Division in `native_per_pubdata` Calculation Causes Systematic Undercharging for Pubdata Native Resources - (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

The `native_per_pubdata` ratio — which controls how many native resource units are charged per byte of pubdata written — is computed using floor (truncating) integer division (`wrapping_div`), while the analogous `native_per_gas` ratio is computed using ceiling division (`div_ceil`). This asymmetry causes a systematic undercharge of native resources for every transaction that writes pubdata when `pubdata_price % native_price != 0`, allowing users to write pubdata at a lower cost than intended and leaving the operator undercompensated.

---

### Finding Description

In `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`, the two key resource ratios are computed as:

```rust
// native_per_gas uses CEILING division
let native_per_gas = u256_try_to_u64(&gas_price.div_ceil(native_price))...

// native_per_pubdata uses FLOOR division
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))...
``` [1](#0-0) 

The same floor division is replicated in the public API helper `validate_l2_tx_intrinsic_native_resources`:

```rust
// native_per_pubdata = pubdata_price / native_price  (floor)
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
``` [2](#0-1) 

The `native_per_pubdata` value is then used in two places:

1. **Intrinsic pubdata overhead** deducted from the native limit before execution:
   `intrinsic_pubdata_overhead = native_per_pubdata_byte.saturating_mul(intrinsic_pubdata)` [3](#0-2) 

2. **Post-execution pubdata charge** via `get_resources_to_charge_for_pubdata`:
   `native = current_pubdata_spent.checked_mul(native_per_pubdata)` [4](#0-3) 

The double resource accounting model is documented to use `nativePerGas := gasPrice/nativePrice` as a ratio, but the design intent for pubdata is the same: the operator must be compensated for the full proving cost of each pubdata byte. [5](#0-4) 

---

### Impact Explanation

When `pubdata_price % native_price != 0`, the floor division produces a `native_per_pubdata` that is 1 unit less than the true cost per byte. For a transaction writing `N` bytes of pubdata, the total undercharge is `N` native units. Since the pubdata limit per block can be large (defaulting to `u64::MAX` in the test rig), the cumulative undercharge across many transactions is unbounded in principle. [6](#0-5) 

Concretely:
- If `pubdata_price = 101` and `native_price = 100`, then `native_per_pubdata = 1` (floor) instead of `2` (ceil).
- A transaction writing 1,000 bytes of pubdata is charged 1,000 native units instead of 2,000.
- The operator loses 1,000 native units of compensation per such transaction.
- Users can write pubdata at half the intended cost in this example.

This is a **resource accounting bug** causing operator funds loss — the direct analog of the ERC4626 decimal mismatch where the conversion ratio is systematically wrong.

---

### Likelihood Explanation

The operator sets `pubdata_price` and `native_price` as arbitrary `U256` values in the block metadata oracle. In production, these values are derived from L1 gas prices and proving cost estimates, which are unlikely to always be exact multiples of each other. The condition `pubdata_price % native_price != 0` will occur regularly in any real deployment, making this a persistent, not edge-case, undercharge. [7](#0-6) 

---

### Recommendation

Replace `wrapping_div` with `div_ceil` for `native_per_pubdata` in both locations, consistent with how `native_per_gas` is computed:

```rust
// In validation_impl.rs line 142
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;

// In api/src/helpers.rs line 427
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price)).ok_or(())?;
```

This ensures the operator is always compensated for at least the full cost of each pubdata byte, matching the ceiling-division semantics already applied to `native_per_gas`.

---

### Proof of Concept

**Setup:** Operator sets `pubdata_price = 101`, `native_price = 100`.

**Current behavior:**
- `native_per_pubdata = floor(101 / 100) = 1`
- Transaction writes 1,000 bytes of pubdata → charged `1,000 × 1 = 1,000` native units

**Correct behavior:**
- `native_per_pubdata = ceil(101 / 100) = 2`
- Transaction writes 1,000 bytes of pubdata → charged `1,000 × 2 = 2,000` native units

**Attacker path:** An unprivileged user submits any transaction that writes storage (e.g., a simple ERC-20 transfer). The bootloader computes `native_per_pubdata` via `wrapping_div` at validation time. The user's transaction is processed with a native budget that is effectively 50% larger than intended for pubdata costs. The operator receives only half the expected native compensation for the pubdata written. This is reachable by any unprivileged transaction sender on any block where `pubdata_price % native_price != 0`. [8](#0-7) [9](#0-8)
