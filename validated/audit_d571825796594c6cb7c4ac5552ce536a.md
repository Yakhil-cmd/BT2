Audit Report

## Title
Missing Received-Amount Verification in ERC20 Deposit Helper Contracts Enables ckERC20 Over-Minting — (File: `rs/ethereum/cketh/minter/DepositHelperWithSubaccount.sol`, `rs/ethereum/cketh/minter/ERC20DepositHelper.sol`)

## Summary

Both ERC20 deposit helper contracts emit a deposit event carrying the caller-supplied `amount` parameter without verifying the minter's actual before/after token balance. The IC ckETH minter canister scrapes these events and mints ckERC20 tokens equal to the logged `amount`. For any fee-on-transfer or rebasing ERC20 token where the actual received amount is less than `amount`, the minter mints more ckERC20 than the ERC20 tokens it actually holds, breaking the 1:1 backing invariant and eventually rendering the system insolvent for the last redeemers.

## Finding Description

In `DepositHelperWithSubaccount.sol` L511–532, `CkDeposit.depositErc20()` calls `safeTransferFrom` and immediately emits `ReceivedEthOrErc20` with the caller-supplied `amount`, with no balance snapshot before or after the transfer:

```solidity
erc20Token.safeTransferFrom(msg.sender, minterAddress, amount);
emit ReceivedEthOrErc20(erc20Address, msg.sender, amount, principal, subaccount);
```

The same pattern exists in `ERC20DepositHelper.sol` L498–503 (`CkErc20Deposit.deposit()`):

```solidity
erc20Token.safeTransferFrom(msg.sender, cketh_minter_main_address, amount);
emit ReceivedErc20(erc20_address, msg.sender, amount, principal);
```

The IC minter's `mint()` function in `rs/ethereum/cketh/minter/src/deposit.rs` L73–81 reads the scraped event and mints exactly `event.value()` — the `amount` field from the event log — with no cross-check against the actual ERC20 balance increase at the minter's Ethereum address:

```rust
amount: event.value(),
```

The internal `Erc20Balances::erc20_add()` in `rs/ethereum/cketh/minter/src/state.rs` L742–755 also records the inflated `amount`, so the minter's own accounting diverges from the real on-chain balance, compounding the error across repeated deposits.

There are no existing guards in either the Solidity contracts or the Rust minter that compare the pre- and post-transfer balance of the minter address. The only existing guard (the `QuarantinedDeposit` double-minting guard in `deposit.rs` L43–52) prevents re-processing the same event, but does nothing to correct the over-minted amount.

## Impact Explanation

This is a chain-fusion ledger conservation violation. For any supported ERC20 token with fee-on-transfer semantics, the minter's Ethereum address receives `amount - fee` tokens while the IC ckERC20 ledger mints `amount` tokens. Over repeated deposits, total ckERC20 supply exceeds actual ERC20 collateral held by the minter. When users later withdraw ckERC20 back to ERC20, the minter cannot fulfill all redemptions — the last withdrawers receive nothing, constituting direct, permanent loss of funds and protocol insolvency. This matches the allowed Critical impact: **illegal minting and protocol insolvency involving in-scope chain-key/ledger assets**.

USDT (`0xdAC17F958D2ee523a2206206994597C13D831ec7`) is already a supported ckERC20 token (added via NNS proposal 131374). The USDT contract contains a fee mechanism that is currently set to zero but is unilaterally activatable by Tether at any time without any IC governance action. Upon activation, every subsequent `depositErc20` call with USDT would silently over-mint ckUSDT. Given the total value locked in ckUSDT, the insolvency exposure would readily exceed $1M.

## Likelihood Explanation

**Medium-to-High.** The exploit path itself is fully unprivileged — any Ethereum user can call `depositErc20` on the helper contract. The constraint is that a supported token must have active fee-on-transfer behavior. Two realistic paths exist: (1) Tether enables the USDT fee mechanism (an external action requiring no IC governance), or (2) a future NNS proposal adds a fee-on-transfer token. Neither path requires attacker privilege. The attack is repeatable and cumulative, with no per-deposit cost beyond normal Ethereum gas.

## Recommendation

In both helper contracts, capture the minter's actual balance before and after `safeTransferFrom` and emit the difference as the deposit amount:

```solidity
uint256 balanceBefore = erc20Token.balanceOf(minterAddress);
erc20Token.safeTransferFrom(msg.sender, minterAddress, amount);
uint256 actualReceived = erc20Token.balanceOf(minterAddress) - balanceBefore;
require(actualReceived > 0, "ERC20: no tokens received");
emit ReceivedEthOrErc20(erc20Address, msg.sender, actualReceived, principal, subaccount);
```

Apply the same fix to `CkErc20Deposit.deposit()` in `ERC20DepositHelper.sol`. No changes to the Rust minter are required if the Solidity fix is applied, since the event will then carry the correct received amount.

## Proof of Concept

1. USDT fee mechanism is enabled by Tether (or any other supported fee-on-transfer token is added via NNS proposal).
2. Attacker calls `depositErc20(usdtAddress, 1_000_000, principal, subaccount)` on `CkDeposit` after approving the helper contract for `1_000_000` USDT.
3. `safeTransferFrom` transfers only `990_000` USDT to the minter (1% fee deducted by the USDT contract).
4. The contract emits `ReceivedEthOrErc20(..., amount=1_000_000, ...)`.
5. The IC minter scrapes the event and calls `mint()` with `event.value() = 1_000_000`.
6. The attacker receives `1_000_000` ckUSDT on the IC ledger while the minter holds only `990_000` USDT.
7. Repeating inflates ckUSDT supply beyond actual collateral; the last redeemers cannot withdraw.

A deterministic integration test can be written using the existing `CkErc20Setup` test harness in `rs/ethereum/cketh/test_utils/src/ckerc20.rs` by deploying a mock ERC20 token with a configurable transfer fee, calling `depositErc20`, and asserting that the minted ckERC20 amount equals the actual received amount rather than the input `amount`.
