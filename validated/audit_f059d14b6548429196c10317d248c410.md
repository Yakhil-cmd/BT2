### Title
Missing Caller Restriction on `update_rewards` Allows Any Address to Suppress Block Rewards for All Stakers — (`File: src/staking/staking.cairo`)

---

### Summary

The spec mandates that `update_rewards` is callable by "Only starkware sequencer," but the production implementation enforces **no caller check**. Because `last_reward_block` is a **global** (not per-staker) storage variable, any unprivileged address can call `update_rewards(any_valid_staker, disable_rewards=true)` first in every block, consuming the one-call-per-block slot and permanently skipping reward distribution for all stakers.

---

### Finding Description

The `IStakingRewardsManager::update_rewards` implementation at `src/staking/staking.cairo:1449` performs only two guards before writing state:

1. `self.general_prerequisites()` — checks for contract pause only.
2. `current_block_number > self.last_reward_block.read()` — the global per-block gate. [1](#0-0) 

There is no `assert_caller_is_sequencer` or equivalent. The spec at `docs/spec.md:1645` explicitly states access control is "Only starkware sequencer," but this is not implemented in code. [2](#0-1) 

`last_reward_block` is a **single global storage slot** — not keyed by `staker_address`. Once it is written for any staker in a block, every subsequent call for any staker in that same block reverts with `REWARDS_ALREADY_UPDATED`. [3](#0-2) 

The early-return path at line 1487 (`if disable_rewards || self.is_pre_consensus() { return; }`) skips all reward accounting but still writes `last_reward_block`, permanently consuming the block's reward slot. [4](#0-3) 

---

### Impact Explanation

An attacker who calls `update_rewards(valid_staker, disable_rewards=true)` as the first transaction in every block:

- Writes `last_reward_block = current_block_number` with zero rewards distributed.
- Causes every legitimate sequencer call for that block to revert with `REWARDS_ALREADY_UPDATED`.
- Repeating this every block permanently freezes all unclaimed yield for all stakers.

This matches **High: Permanent freezing of unclaimed yield**.

---

### Likelihood Explanation

The function is publicly callable with no role gate. Any address can submit this transaction. On Starknet, transaction ordering within a block is controlled by the sequencer, but the attacker only needs to be included before the legitimate sequencer call — achievable by paying higher fees or by the attacker being the sequencer themselves. The attack is cheap (one call per block, no stake required) and fully repeatable.

---

### Recommendation

Add a caller check matching the spec's stated access control. For example:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
    assert!(
        starknet::get_caller_address() == self.sequencer_address.read(),
        "{}",
        Error::CALLER_IS_NOT_SEQUENCER,
    );
    // ... rest of function
}
```

Alternatively, restrict the function to an internal call path only reachable from the sequencer's designated contract.

---

### Proof of Concept

1. Deploy two active stakers (`staker_A`, `staker_B`), both past the K-epoch activation window, with consensus rewards active.
2. At the start of each new block, an unprivileged EOA calls:
   ```
   update_rewards(staker_A, disable_rewards=true)
   ```
3. `last_reward_block` is set to the current block; no rewards are distributed.
4. The legitimate sequencer's call `update_rewards(staker_A, disable_rewards=false)` reverts with `REWARDS_ALREADY_UPDATED`.
5. Any call for `staker_B` in the same block also reverts.
6. After N blocks, both stakers have accumulated zero rewards despite being fully eligible.

The existing test at `src/staking/tests/test.cairo:3887` already confirms that calling with `disable_rewards=true` followed by any second call in the same block produces `REWARDS_ALREADY_UPDATED` — the only missing piece is that the first caller is not restricted. [5](#0-4)
