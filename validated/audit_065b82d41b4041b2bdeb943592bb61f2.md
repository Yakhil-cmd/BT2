### Title
Hidden `wallet_id` Override in `chia_sendTransaction` Allows dApp to Drain Wrong Wallet Without User Awareness — (`packages/gui/src/electron/commands/Commands.ts`, `parseDappParams.ts`, `humanizeParams.ts`)

---

### Summary

A paired dApp can supply `walletId` (camelCase) in a `chia_sendTransaction` WalletConnect call. `parseDappParams` normalizes it to `wallet_id` via `toSnakeCase`, the value passes the allowlist check, and the default of `1` is never applied because the field is already defined. The confirmation dialog never shows `wallet_id` because the schema marks it `hide: true`. The user approves what appears to be a normal XCH send, but the daemon executes the spend from whichever wallet the dApp chose — including a CAT wallet.

---

### Finding Description

**Step 1 — camelCase normalization.**
`parseDappParams` calls `toSnakeCase` on the raw JSON before any other processing: [1](#0-0) 

`walletId: 5` becomes `wallet_id: 5`.

**Step 2 — allowlist passes.**
The `chia_wallet.send_transaction` schema explicitly lists `wallet_id` as an allowed param: [2](#0-1) 

So the allowlist check at `parseDappParams.ts` line 34–38 does not throw.

**Step 3 — default is skipped.**
The default `wallet_id: 1` is only applied when the field is `undefined`: [3](#0-2) 

Because `wallet_id` is already `5`, the default is never applied. The test suite explicitly documents this behavior: [4](#0-3) 

**Step 4 — confirmation dialog hides `wallet_id`.**
`humanizeParams` unconditionally excludes any param with `hide: true`: [5](#0-4) 

The `wallet_id` field is marked `hide: true` in the schema: [2](#0-1) 

The `humanizeParams` test confirms `wallet_id` is silently dropped even when it holds a non-default value: [6](#0-5) 

**Step 5 — execution.**
`parsedParams` (containing `wallet_id: 5`) is passed directly to `sendCommand` / `dispatchPairRequest` after the user approves: [7](#0-6) 

The daemon receives `send_transaction` with `wallet_id: 5` and spends from that wallet.

---

### Impact Explanation

The user sees a confirmation dialog showing the correct destination address and amount, but the wallet from which funds are spent is silently controlled by the dApp. If wallet 5 is a CAT wallet, CAT tokens are sent instead of XCH. If wallet 5 is a second XCH wallet belonging to a different identity, funds leave the wrong identity. In both cases the user has approved a materially different transaction than what was displayed. This maps directly to the allowed High impact: *"WalletConnect state that causes a user to approve the wrong asset, identity, amount, destination, or status."*

---

### Likelihood Explanation

Any dApp that has completed the WalletConnect pairing flow and holds `chia_sendTransaction` permission can trigger this with a single crafted call. No additional privileges, leaked keys, or host compromise are required. The camelCase-to-snake_case normalization is documented and tested, so the path is deterministic and reproducible.

---

### Recommendation

Either:
- Remove `wallet_id` from the `chia_sendTransaction` dapp params list entirely, so any dApp-supplied `wallet_id` is rejected by the allowlist check (the default of `1` would then always be applied via the `defaults` mechanism); or
- Keep `wallet_id` in the params list but remove `hide: true`, so the confirmation dialog always shows the wallet being spent from, giving the user the information needed to detect a mismatch.

The first option is safer: `chia_sendTransaction` is documented as an XCH-only send from the primary wallet, and there is no legitimate reason for a dApp to redirect it to an arbitrary wallet ID.

---

### Proof of Concept

```ts
// Reproducible with the existing test harness:
import { parseDappParams } from './parseDappParams';
import JSONbig from 'json-bigint';

const result = parseDappParams(
  'chia_sendTransaction',
  JSONbig.stringify({ amount: '1000000', fee: '0', address: 'txch1victim', walletId: 5 }),
);

console.log(result.wallet_id); // 5  — default of 1 was NOT applied

// humanizeDappCommand will return rows for amount, fee, address only.
// wallet_id: 5 is never shown to the user.
// The daemon call goes to wallet 5 (e.g., a CAT wallet).
```
