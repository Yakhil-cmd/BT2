### Title
Attacker-Supplied `receiver_id` in `GetExecutionOutcome` Produces Invalid `outcome_root_proof` for Cross-Shard Receipts — (`chain/client/src/view_client_actor.rs`)

### Summary

The `GetExecutionOutcome` handler in `ViewClientActor` derives `target_shard_index` entirely from the caller-supplied `receiver_id`, without ever validating it against the actual receipt stored on-chain. When an attacker supplies a `receiver_id` that hashes to a different shard than the receipt's true receiver, the handler returns an `outcome_root_proof` that is the Merkle path for the **wrong shard's** outcome root. The two-step light-client verification invariant — `compute_root(sha256(borsh(shard_outcome_root)), outcome_root_proof) == block_header_lite.inner_lite.outcome_root` — is broken for every such request.

### Finding Description

In `chain/client/src/view_client_actor.rs`, the handler for `GetExecutionOutcome`:

**Step 1 — `account_id` is taken verbatim from the caller:** [1](#0-0) 

**Step 2 — Outcome lookup is keyed only on `receipt_id`; `receiver_id` is never cross-checked:** [2](#0-1) 

**Step 3 — `target_shard_id` and `target_shard_index` are derived from the attacker-controlled `account_id`:** [3](#0-2) 

**Step 4 — `get_next_block_hash_with_new_chunk` is called with the wrong shard, potentially selecting a different block than the one that committed the receipt's outcome:** [4](#0-3) 

**Step 5 — `outcome_root_proof` is the Merkle path at `target_shard_index` in the wrong block's `outcome_roots` array:** [5](#0-4) 

**Concrete 2-shard scenario:**

| Variable | Value |
|---|---|
| Receipt actual receiver | `zoe.near` → shard 1 |
| Attacker-supplied `receiver_id` | `alice.near` → shard 0 |
| `target_shard_index` | 0 (wrong) |
| `outcome_roots` | `[shard_0_root, shard_1_root]` |
| `outcome_root_proof` returned | `merklize(&outcome_roots).1[0]` = path for `shard_0_root` |

The `outcome_proof.proof` (the within-shard Merkle path) is retrieved correctly by `receipt_id` and proves membership in `shard_1_root`. But the `outcome_root_proof` is the path for `shard_0_root`. The two halves are mismatched.

### Impact Explanation

Light-client verification (per the protocol spec): [6](#0-5) 

Step 1 correctly reconstructs `shard_1_outcome_root` from `outcome_proof.proof`. Step 2 then applies `path_for_shard_0` to `shard_1_outcome_root`, producing a `block_outcome_root` that does not match `block_header_lite.inner_lite.outcome_root`. The proof is cryptographically invalid.

Additionally, because `get_next_block_hash_with_new_chunk` is called with the wrong `target_shard_id`, the `block_hash` embedded in `outcome_proof` may point to a different block than the one that actually committed the receipt's outcome — compounding the mismatch.

**Exact corrupted values:**
- `outcome_root_proof`: MerklePath for `shard_0_outcome_root` instead of `shard_1_outcome_root`.
- `outcome_proof.block_hash`: hash of the next block with a new chunk for shard 0, not shard 1.
- `block_outcome_root` computed by light client: wrong value, does not equal `block_header_lite.inner_lite.outcome_root`.

### Likelihood Explanation

Any unprivileged user can trigger this via the public `EXPERIMENTAL_light_client_proof` RPC endpoint: [7](#0-6) 

The attacker only needs a valid `receipt_id` for any cross-shard receipt (trivially obtained from any transaction receipt) and supplies a `receiver_id` that maps to a different shard. No special privileges are required. The `TransactionOrReceiptId` type accepts arbitrary `receiver_id`: [8](#0-7) 

### Recommendation

After retrieving the outcome, look up the actual receipt from the store and derive `target_shard_id` from the receipt's true `receiver_id`, not from the caller-supplied field. Alternatively, validate that the caller-supplied `receiver_id` matches the actual receipt's `receiver_id` before proceeding, returning an error if they differ.

### Proof of Concept

1. Build a 2-shard test-loop environment (e.g., using `TwoShardHarness`).
2. Execute a cross-shard transfer: `alice.near` (shard 0) → `zoe.near` (shard 1).
3. Capture the `receipt_id` of the cross-shard action receipt.
4. Call `GetExecutionOutcome` with `TransactionOrReceiptId::Receipt { receipt_id, receiver_id: alice.near }` (wrong shard).
5. Receive `GetExecutionOutcomeResponse { outcome_proof, outcome_root_proof }`.
6. Reconstruct `shard_outcome_root = compute_root(sha256(borsh(execution_outcome)), outcome_proof.proof)` — this correctly yields shard 1's outcome root.
7. Assert `verify_path(block_header_lite.inner_lite.outcome_root, outcome_root_proof, shard_outcome_root)` returns **false** — the proof is for the wrong shard's root.

#[test]
#[cfg_attr(feature = "protocol_feature_spice", ignore)]
fn test_rpc_light_client_proof_unknown_outcome() {
    init_test_logger();
    let mut h = TwoShardHarness::new();

    let validator = h.validator.clone();
    let alice = h.alice.clone();
    let zoe = h.zoe.clone();
    let alice_node = h.alice_node.clone();
    let zoe_node = h.zoe_node.clone();

    // Run a few blocks so we have a final head to use as `light_client_head`.
    let target_height = h.env.node_for_account(&validator).head().height + 5;
    h.env.runner_for_account(&validator).run_until_executed_height(target_height);
    h.env.runner_for_account(&alice_node).run_until_executed_height(target_height);
    h.env.runner_for_account(&zoe_node).run_until_executed_height(target_height);

    let light_client_head =
        h.env.node_for_account(&alice_node).client().chain.final_head().unwrap().last_block_hash;

    let run_proof_query = |h: &mut TwoShardHarness,
                           node_id: &AccountId,
                           id: TransactionOrReceiptId|
     -> Result<RpcLightClientExecutionProofResponse, RpcError> {
        h.env.runner_for_account(node_id).run_with_jsonrpc_client(
            |client| {
                client.light_client_proof(RpcLightClientExecutionProofRequest {
                    id: id.clone(),
                    light_client_head,
                })
            },
            Duration::seconds(5),
        )
    };

    // Bogus transaction hash — routes by `sender_id` to alice's shard.
    let bogus_tx = TransactionOrReceiptId::Transaction {
        transaction_hash: CryptoHash::hash_bytes(b"bogus-tx"),
        sender_id: alice,
    };
    for node_id in [&alice_node, &zoe_node] {
        let err = run_proof_query(&mut h, node_id, bogus_tx.clone())
            .expect_err("expected UNKNOWN_TRANSACTION_OR_RECEIPT");
        assert_rpc_error(&err, "UNKNOWN_TRANSACTION_OR_RECEIPT");
    }

    // Bogus receipt id — routes by `receiver_id` to zoe's shard.
    let bogus_receipt = TransactionOrReceiptId::Receipt {
        receipt_id: CryptoHash::hash_bytes(b"bogus-receipt"),
        receiver_id: zoe,
    };
    for node_id in [&alice_node, &zoe_node] {
        let err = run_proof_query(&mut h, node_id, bogus_receipt.clone())
            .expect_err("expected UNKNOWN_TRANSACTION_OR_RECEIPT");
        assert_rpc_error(&err, "UNKNOWN_TRANSACTION_OR_RECEIPT");
