Audit Report

## Title
SNS Governance `compute_ballots_for_new_proposal` Performs Unbounded Linear Neuron Scan Without Instruction-Limit Guard, Enabling Governance DoS - (File: rs/sns/governance/src/governance.rs)

## Summary

`compute_ballots_for_new_proposal` iterates synchronously over every neuron in `self.proto.neurons` on every `make_proposal` call with no instruction-limit check or multi-message splitting. NNS Governance has been explicitly patched with hard/soft instruction limits and `noop_self_call_if_over_instructions` for the equivalent operation; SNS Governance has received none of these mitigations. If the neuron count grows large enough, every `make_proposal` call traps at the IC instruction limit, permanently freezing proposal submission for the affected SNS.

## Finding Description

`compute_ballots_for_new_proposal` at line 5255 iterates unconditionally over the entire `self.proto.neurons` `BTreeMap`:

```rust
for (k, v) in self.proto.neurons.iter() {
    if v.dissolve_delay_seconds(now_seconds) < min_dissolve_delay_for_vote {
        continue;
    }
    let voting_power = v.voting_power(...);
    total_power += voting_power as u128;
    electoral_roll.insert(k.clone(), Ballot { ... });
}
``` [1](#0-0) 

This function is called unconditionally from `make_proposal`:

```rust
let (_, electoral_roll) = self
    .compute_ballots_for_new_proposal()
    .map_err(...)?;
``` [2](#0-1) 

There is no `ic_cdk::api::instruction_counter()` check, no soft/hard limit guard, and no mechanism to split work across messages. The only bound on neuron count is `max_number_of_neurons`, a governance-configurable parameter enforced by `check_neuron_population_can_grow`: [3](#0-2) 

By contrast, NNS Governance defines explicit instruction limits and uses `noop_self_call_if_over_instructions` to split ballot computation across messages: [4](#0-3) [5](#0-4) 

The constraints integration test confirms `MAX_NUMBER_OF_NEURONS_CEILING` is sized to accommodate swap participants, developer neurons, and Neurons' Fund neurons — potentially hundreds of thousands: [6](#0-5) 

At a conservative ~60,000 instructions per neuron (heap `BTreeMap<String, Neuron>`, string key cloning, four-parameter voting-power formula, `BTreeMap<String, Ballot>` insertion), the 40-billion IC instruction limit is reached at roughly 667,000 neurons — within the range the ceiling is designed to accommodate.

## Impact Explanation

When the neuron count reaches the instruction-exhaustion threshold, every `make_proposal` call traps and rolls back. No new proposals can be submitted to the affected SNS: no upgrades, no parameter changes, no treasury transfers, no dapp management. This is a concrete application/platform-level DoS of SNS governance, matching the **High** impact class ($2,000–$10,000): "Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS." The impact is scoped to individual SNS instances rather than the entire IC, which places it at High rather than Critical.

## Likelihood Explanation

Two realistic paths exist that do not require governance majority:

1. **Organic growth**: In a large SNS deployment, token holders naturally stake and claim neurons. If the SNS was launched with a high `max_number_of_neurons` (set at genesis or raised by an earlier governance proposal), the neuron count can reach the dangerous threshold through normal participation without any attacker action.

2. **Funded attacker**: A single principal acquires enough SNS tokens to register neurons up to the current `max_number_of_neurons` limit. No governance proposal is required — only capital. The cost scales with the SNS token price and the minimum stake per neuron.

A third path (raising `max_number_of_neurons` via governance proposal) does require governance majority, but the first two paths do not. The NNS Governance team has already recognized and fixed this exact class of bug in their own canister, confirming it is a realistic concern.

## Recommendation

1. Add an instruction-limit check inside the loop in `compute_ballots_for_new_proposal` mirroring NNS Governance: check `ic_cdk::api::instruction_counter()` against a hard limit and return an error, or use `noop_self_call_if_over_instructions` to split across messages.
2. Pre-compute and cache a voting-power snapshot (as NNS Governance now does via `compute_ballots_for_standard_proposal`) so `make_proposal` reads from a snapshot rather than iterating all neurons inline.
3. Enforce a tighter `MAX_NUMBER_OF_NEURONS_CEILING` calibrated against the per-neuron instruction cost to guarantee the loop always completes within the IC instruction limit.

## Proof of Concept

```
1. Deploy an SNS with max_number_of_neurons set to a value N large enough
   that N * ~60,000 instructions > 40 billion (i.e., N > ~667,000),
   or wait for organic growth to reach that threshold.
2. Distribute SNS tokens to N principals.
3. Each principal calls manage_neuron { ClaimOrRefresh } with a dissolve
   delay >= neuron_minimum_dissolve_delay_to_vote_seconds.
4. Once N neurons are registered, call make_proposal from any eligible neuron.
5. Execution enters compute_ballots_for_new_proposal at line 5255,
   iterates over all N neurons, exhausts the 40B instruction limit, and traps.
6. The call is rolled back; the proposal is never created.
7. All subsequent make_proposal calls trap identically — governance is frozen.

Reproducible as a PocketIC integration test: instantiate SNS governance with
a pre-populated neuron store of sufficient size, call make_proposal, and assert
the call traps with an instruction-limit error.
```
