### Title
Gateway Skips `__validate__` for Nonce-1 Invoke When Any Prior Tx Exists in Mempool, Not Just `deploy_account` - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator is intended to skip `__validate__` only when a `deploy_account` transaction is pending in the mempool (UX feature for deploy+invoke pairs). However, the check uses `account_tx_in_pool_or_recent_block`, which returns `true` for **any** pending transaction from that account — not exclusively a `deploy_account`. An attacker controlling an account deployed via a factory contract (nonce=0 in state) can first submit a valid nonce-0 invoke (which passes `__validate__`), then submit a nonce-1 invoke with an **invalid signature**. The gateway skips `__validate__` for the nonce-1 invoke because the nonce-0 invoke is already in the pool, admitting an unauthorized transaction to the mempool.

### Finding Description

In `crates/apollo_gateway/src/stateful_transaction_validator.rs`, the function `skip_stateful_validations` decides whether to skip the `__validate__` entry point:

```rust
async fn skip_stateful_validations(...) -> ... {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                ...
        }
    }
    Ok(false)
}
``` [1](#0-0) 

The condition `tx.nonce() == 1 && account_nonce == 0` is intended to detect the "deploy_account not yet processed" scenario. The check `account_tx_in_pool_or_recent_block` is supposed to confirm a `deploy_account` is pending. However, `account_tx_in_pool_or_recent_block` returns `true` for **any** transaction from that address in the pool or recent block:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [2](#0-1) 

When `skip_validate=true`, `run_validate_entry_point` sets `validate: false` in `ExecutionFlags`, and `StatefulValidator::perform_validations` returns early before calling `__validate__`:

```rust
if !tx.execution_flags.validate {
    return Ok(());
}
``` [3](#0-2) 

The gateway's `run_validate_entry_point` constructs the flags directly from `skip_validate`:

```rust
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [4](#0-3) 

### Impact Explanation

An attacker with an account deployed via a factory contract (nonce=0 in on-chain state, account IS deployed) can:

1. Submit a nonce-0 invoke with a valid signature → passes `__validate__`, enters the mempool.
2. Submit a nonce-1 invoke with an **invalid signature** (e.g., bypassing a multisig threshold).
   - `validate_state_preconditions`: nonce 1 is within `[0, max_gap]` → passes.
   - `validate_by_mempool`: staged nonce after nonce-0 is 1, so nonce-1 is not too old → passes.
   - `skip_stateful_validations`: `tx.nonce()==1`, `account_nonce==0`, `account_tx_in_pool_or_recent_block==true` (nonce-0 invoke is in pool) → returns `true`.
   - `run_validate_entry_point`: `validate=false` → `__validate__` is **skipped entirely**.
3. The nonce-1 invoke with invalid signature is admitted to the mempool without signature verification.

The gateway admits an invalid transaction — one that would fail `__validate__` at execution time — bypassing the admission invariant that only valid, authorized transactions should enter the mempool.

### Likelihood Explanation

Accounts deployed via factory contracts (using the `deploy` syscall from another contract) start with nonce=0 and are fully deployed. This is a standard Starknet pattern (e.g., account factories, proxy deployments). Any such account is vulnerable. The attacker only needs to submit two transactions: one valid nonce-0 invoke and one nonce-1 invoke with an invalid/unauthorized signature.

### Recommendation

In `skip_stateful_validations`, replace the broad `account_tx_in_pool_or_recent_block` check with a specific check that verifies a `deploy_account` transaction is pending for the sender address. The mempool should expose a dedicated `has_pending_deploy_account(address)` query, or the gateway should inspect the transaction type of the pending transaction before deciding to skip `__validate__`.

Alternatively, restrict the skip condition to only apply when the account has no deployed class hash in state (i.e., `get_class_hash_at(sender_address) == ClassHash::default()`), which is the true indicator that the account is not yet deployed.

### Proof of Concept

1. Deploy an account contract via a factory (using `deploy` syscall). The account has `nonce=0` in on-chain state and a valid `__validate__` that enforces a multisig requiring 2 signatures.
2. Attacker holds only 1 of 2 required keys.
3. Submit `invoke_tx(nonce=0, signature=[valid_sig_key1, valid_sig_key2])` → passes `__validate__` (2 sigs present), enters mempool. `account_tx_in_pool_or_recent_block` now returns `true`.
4. Submit `invoke_tx(nonce=1, signature=[valid_sig_key1])` (only 1 sig, would fail multisig `__validate__`).
   - Gateway: `tx.nonce()==1`, `account_nonce==0`, `account_tx_in_pool_or_recent_block==true` → `skip_validate=true` → `__validate__` is **not called**.
   - The nonce-1 invoke with insufficient signatures is admitted to the mempool.
5. The transaction fails at blockifier execution time (blockifier always runs `__validate__`), but the gateway's admission invariant is broken: an unauthorized transaction entered the mempool. [5](#0-4) [6](#0-5)

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
