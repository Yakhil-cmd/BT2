### Title
Pre-EIP-155 Legacy Transactions Bypass Chain-ID Validation, Enabling Cross-Deployment Replay — (`engine/src/engine.rs`)

### Summary

Aurora Engine's `submit` entrypoint accepts pre-EIP-155 legacy transactions (those with `v = 27` or `v = 28`). For these transactions, `chain_id` is `None`, and the engine's chain-ID guard is never evaluated. A signed legacy transaction broadcast on one Aurora deployment (e.g., mainnet, chain ID `1313161554`) can be replayed verbatim on any other Aurora deployment (e.g., testnet, chain ID `1313161555`) by any unprivileged observer, draining the signer's funds on the target chain.

---

### Finding Description

**Root cause — `engine-transactions/src/legacy.rs`, lines 88–93**

`LegacyEthSignedTransaction::chain_id()` returns `None` whenever `v ∈ {27, 28}`:

```rust
pub const fn chain_id(&self) -> Option<u64> {
    match self.v {
        0..=34 => None,
        _ => Some((self.v - 35) / 2),
    }
}
``` [1](#0-0) 

This `None` propagates into `NormalizedEthTransaction::chain_id`, which is the value read by the engine.

**Guard — `engine/src/engine.rs`, lines 1054–1059**

```rust
// Validate the chain ID, if provided inside the signature:
if let Some(chain_id) = transaction.chain_id
    && U256::from(chain_id) != U256::from_big_endian(&state.chain_id)
{
    return Err(EngineErrorKind::InvalidChainId.into());
}
``` [2](#0-1) 

The guard is conditional on `chain_id` being `Some`. When it is `None` (all pre-EIP-155 legacy transactions), the block is skipped entirely and the transaction is accepted regardless of which Aurora deployment is processing it.

**Entry path**

Any caller can invoke the `submit` NEAR entrypoint: [3](#0-2) 

No access control prevents an unprivileged account from submitting a raw legacy transaction. The engine parses the bytes, recovers the sender via `ecrecover`, and executes — all without ever checking that the signature was intended for this chain.

---

### Impact Explanation

**Critical — Direct theft of user funds.**

A user who signs a legacy transaction (e.g., an ETH transfer or contract call) on Aurora mainnet produces a byte string that is equally valid on Aurora testnet or any other Aurora deployment. An attacker who observes the raw transaction (e.g., from a public mempool or block explorer) can submit it to a different Aurora deployment. If the victim's nonce on the target chain matches (e.g., both accounts are fresh, nonce = 0), the transaction executes and the victim's funds are transferred to the attacker-controlled destination.

---

### Likelihood Explanation

**Medium.** Pre-EIP-155 transactions (`v = 27/28`) are still produced by older Ethereum tooling, hardware wallets in legacy mode, and some SDKs. Aurora explicitly accepts them (the `v = 27..=28` branch in `legacy.rs` is a supported code path, not an error). A victim need only have a matching nonce on the target deployment — trivially true for any account that has not yet transacted on that chain.

---

### Recommendation

Reject pre-EIP-155 legacy transactions at the `submit` boundary, or treat a missing chain-ID as implicitly matching the engine's configured chain-ID and enforce the check unconditionally:

```rust
// Enforce chain-ID for all transaction types, including legacy (v=27/28).
let tx_chain_id = transaction.chain_id
    .ok_or(EngineErrorKind::InvalidChainId)?;  // reject if absent
if U256::from(tx_chain_id) != U256::from_big_endian(&state.chain_id) {
    return Err(EngineErrorKind::InvalidChainId.into());
}
```

Alternatively, document explicitly that pre-EIP-155 transactions are intentionally accepted and warn users that such transactions are replayable across all Aurora deployments.

---

### Proof of Concept

1. Alice holds 1 ETH on both Aurora mainnet (`chain_id = 1313161554`) and Aurora testnet (`chain_id = 1313161555`). Both accounts are at nonce 0.
2. Alice signs a legacy transfer (v = 28) on mainnet: `send 1 ETH to Bob`.
3. The raw RLP-encoded transaction is publicly visible on the mainnet block explorer.
4. Eve copies the raw bytes and calls `submit` on Aurora testnet with the identical payload.
5. `LegacyEthSignedTransaction::chain_id()` returns `None` for `v = 28`.
6. The `if let Some(chain_id) = transaction.chain_id` guard is not entered; no chain-ID mismatch error is raised.
7. `ecrecover` recovers Alice's address; nonce 0 matches; the transaction executes on testnet.
8. Alice loses 1 ETH on testnet to Eve with no action required beyond observing the mainnet transaction.

        require_running(&state)?;

            state,
            current_account_id,
            relayer_address,
            handler,
        )?;
        let result_bytes = borsh::to_vec(&result).map_err(|_| errors::ERR_SERIALIZE)?;
        io.return_output(&result_bytes);

    })
