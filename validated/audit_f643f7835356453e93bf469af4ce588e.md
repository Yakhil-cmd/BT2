### Title
`vm_resources_to_execution_resources` Hardcodes `gas_consumed` to Zero in All Simulation Traces - (`crates/apollo_rpc_execution/src/objects.rs`)

### Summary

`vm_resources_to_execution_resources` unconditionally sets `gas_consumed` to `StarknetApiGasVector::default()` (all zeros) in every `FunctionInvocation::execution_resources` returned by `starknet_simulateTransactions` and `starknet_traceTransaction`. Any Cairo 1 contract that consumes nonzero L2 gas will appear to have consumed zero L2 gas in the simulation trace, producing an authoritative-looking wrong value.

### Finding Description

The function `vm_resources_to_execution_resources` at line 363 of `crates/apollo_rpc_execution/src/objects.rs` constructs an `ExecutionResources` struct. It correctly populates `da_gas_consumed` from the `GasVector` argument, but hardcodes `gas_consumed` to `StarknetApiGasVector::default()`:

```rust
Ok(ExecutionResources {
    steps: u64_from_usize(vm_resources.n_steps),
    builtin_instance_counter,
    memory_holes: u64_from_usize(vm_resources.n_memory_holes),
    da_gas_consumed: StarknetApiGasVector { l1_gas, l2_gas, l1_data_gas },
    gas_consumed: StarknetApiGasVector::default(),   // always zero
})
``` [1](#0-0) 

The `ExecutionResources` type in `crates/starknet_api/src/execution_resources.rs` defines `gas_consumed` and `da_gas_consumed` as two distinct `GasVector` fields: [2](#0-1) 

`gas_consumed` is the per-call execution gas (including `l2_gas` for Cairo 1 contracts). `da_gas_consumed` is the data-availability gas. They are semantically different. The function receives the transaction-level `receipt.da_gas` as its `GasVector` argument and places it into `da_gas_consumed`, but never populates `gas_consumed` with any per-call execution gas.

Every call site passes `transaction_execution_info.receipt.da_gas` as the gas vector: [3](#0-2) [4](#0-3) 

### Impact Explanation

Every `FunctionInvocation` returned by `starknet_simulateTransactions` or `starknet_traceTransaction` will report `execution_resources.gas_consumed = { l1_gas: 0, l1_data_gas: 0, l2_gas: 0 }` regardless of actual L2 gas consumed. Callers relying on this field to estimate per-call L2 gas usage (e.g., for gas profiling, fee estimation tooling, or wallet UX) will receive a systematically wrong value. This matches the allowed impact: **"High. RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value."**

Actual fee charging is unaffected — it reads `receipt.gas` directly: [5](#0-4) 

### Likelihood Explanation

This triggers for every Cairo 1 contract call that consumes nonzero L2 gas, which is the normal case for any non-trivial contract. No special attacker input is required; any unprivileged user submitting a simulate or trace RPC call will observe the wrong value.

### Recommendation

Populate `gas_consumed` from the per-call `CallInfo` resources rather than hardcoding it to zero. The blockifier's `CallInfo` already tracks per-call gas consumption; `call_info.resources` should contain the relevant L2 gas figure. The existing TODO at line 104 (`// TODO(Tzahi): Add l2_gas_consumed`) confirms this is a known gap. [6](#0-5) 

### Proof of Concept

1. Deploy a Cairo 1 contract with a function that performs a storage write (consumes nonzero L2 gas).
2. Call `starknet_simulateTransactions` with a transaction invoking that function.
3. Inspect the returned `FunctionInvocation::execution_resources.gas_consumed`.
4. Assert `gas_consumed.l2_gas == 0` — this will always hold regardless of actual L2 gas consumed, because `vm_resources_to_execution_resources` hardcodes it to `StarknetApiGasVector::default()` at line 399. [7](#0-6)

```rust
fn vm_resources_to_execution_resources(
    vm_resources: VmExecutionResources,
    GasVector { l1_gas, l1_data_gas, l2_gas }: GasVector,
) -> ExecutionResult<ExecutionResources> {
    let mut builtin_instance_counter = HashMap::new();
    for (builtin_name, count) in vm_resources.builtin_instance_counter {
        if count == 0 {
            continue;
        }
        let count = u64_from_usize(count);
        match builtin_name {
            BuiltinName::output => continue,
            BuiltinName::pedersen => builtin_instance_counter.insert(Builtin::Pedersen, count),
            BuiltinName::range_check => builtin_instance_counter.insert(Builtin::RangeCheck, count),
            BuiltinName::ecdsa => builtin_instance_counter.insert(Builtin::Ecdsa, count),
            BuiltinName::bitwise => builtin_instance_counter.insert(Builtin::Bitwise, count),
            BuiltinName::ec_op => builtin_instance_counter.insert(Builtin::EcOp, count),
            BuiltinName::keccak => builtin_instance_counter.insert(Builtin::Keccak, count),
            BuiltinName::poseidon => builtin_instance_counter.insert(Builtin::Poseidon, count),
            BuiltinName::segment_arena => {
                builtin_instance_counter.insert(Builtin::SegmentArena, count)
            }
            // TODO(DanB): what about the following?
            // BuiltinName::range_check96 => todo!(),
            // BuiltinName::add_mod => todo!(),
            // BuiltinName::mul_mod => todo!(),
            _ => {
                return Err(ExecutionError::UnknownBuiltin { builtin_name });
            }
        };
    }
    Ok(ExecutionResources {
        steps: u64_from_usize(vm_resources.n_steps),
        builtin_instance_counter,
        memory_holes: u64_from_usize(vm_resources.n_memory_holes),
        da_gas_consumed: StarknetApiGasVector { l1_gas, l2_gas, l1_data_gas },
        gas_consumed: StarknetApiGasVector::default(),
    })
}
