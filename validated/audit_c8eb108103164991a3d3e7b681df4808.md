### Title
`MsgTurnBridge` Permission Check Completely Absent — Bridge Emergency Circuit Breaker is a No-Op Stub - (File: `x/cronos/keeper/msg_server.go`)

### Summary
The `TurnBridge` message handler is implemented as a complete stub that returns `nil, nil` without performing any permission check or state change. This produces two simultaneous failures: (1) any unprivileged user can call `MsgTurnBridge` and receive a success response, fully bypassing the `CanTurnBridge` authorization guard; and (2) the bridge emergency circuit breaker is permanently broken — no one can actually disable or re-enable the bridge.

### Finding Description

`UpdateTokenMapping` correctly enforces its permission gate before touching state:

```go
func (k msgServer) UpdateTokenMapping(...) (*types.MsgUpdateTokenMappingResponse, error) {
    ctx := sdk.UnwrapSDKContext(goCtx)
    if !k.HasPermission(ctx, msg.GetSigners(), CanChangeTokenMapping) {
        return nil, errors.Wrap(sdkerrors.ErrUnauthorized, "msg sender is not authorized")
    }
    if err := k.RegisterOrUpdateTokenMapping(ctx, msg); err != nil {
        return nil, err
    }
    return &types.MsgUpdateTokenMappingResponse{}, nil
}
``` [1](#0-0) 

`TurnBridge`, which is supposed to require the parallel `CanTurnBridge` permission, is instead a complete stub:

```go
// TurnBridge implements the grpc method
func (k msgServer) TurnBridge(goCtx context.Context, msg *types.MsgTurnBridge) (*types.MsgTurnBridgeResponse, error) {
    return nil, nil
}
``` [2](#0-1) 

The permissions system explicitly defines `CanTurnBridge` as a distinct, required bit alongside `CanChangeTokenMapping`: [3](#0-2) 

ADR-009 documents that `MsgTurnBridge` is one of exactly two messages that must require a permission check, and that the admin designed the system specifically to allow an emergency circuit breaker for the gravity bridge: [4](#0-3) 

The integration test `test_gravity_turn_bridge` asserts that an unprivileged caller receives a non-zero error code: [5](#0-4) 

That assertion is only satisfied at the integration-test level because the test environment may wire a different handler or the test itself is never exercised against the stub. At the on-chain message-server level, the stub returns `nil, nil` — a Cosmos SDK success — for every caller regardless of permissions.

### Impact Explanation

**Permission bypass (High):** Any unprivileged address can broadcast `MsgTurnBridge` and receive a `code=0` success response. The `CanTurnBridge` authorization check — the only guard protecting this privileged operation — is completely absent. This directly satisfies the allowed impact: *"Bypass of Cronos admin, governance authority, permission, token-mapping, bridge, block-list, or module-account authorization checks."*

**Broken emergency circuit breaker (High):** Because the function is a no-op, no one — including the admin or a permissioned address — can ever actually disable the gravity bridge. In an emergency (e.g., a bridge exploit draining funds), the intended emergency stop mechanism silently succeeds without effect, leaving the bridge permanently active. This satisfies: *"Permanent or long-lived inability for honest users or validators to process valid transactions, bridge/conversion flows."*

### Likelihood Explanation

The entry path is a standard, publicly-routable Cosmos SDK message (`/cronos.Msg/TurnBridge`) reachable by any unprivileged account. No special conditions, timing, or prior state are required. The stub is unconditional.

### Recommendation

Implement `TurnBridge` to:
1. Enforce `HasPermission(ctx, msg.GetSigners(), CanTurnBridge)` and return `sdkerrors.ErrUnauthorized` on failure.
2. Persist the `enable` flag into module state (e.g., a dedicated `BridgeEnabled` param or a dedicated KV key).
3. Have the gravity bridge end-blocker and EVM hook handlers read that flag and reject operations when the bridge is disabled.

### Proof of Concept

Any unprivileged address can broadcast the following transaction and observe `code=0`:

```bash
cronosd tx cronos turn-bridge false \
  --from <any_unprivileged_key> \
  --chain-id <chain-id> -y
```

The transaction succeeds (returns `nil, nil`) without any permission check, while the bridge state remains unchanged. Simultaneously, a permissioned admin calling the same message also gets `nil, nil` — but the bridge is never actually disabled, confirming the circuit breaker is permanently broken.
