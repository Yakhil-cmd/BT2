### Title
Self-Referential `beneficiary_id` in `DeleteAccountAction` Causes Permanent Balance Loss - (File: `runtime/runtime/src/actions.rs`)

### Summary

`DeleteAccountAction` accepts a `beneficiary_id` equal to the account being deleted. The runtime deletes the account first, then emits a `Receipt::new_balance_refund` to the now-deleted account. Because system refund receipts that fail do not generate further refunds, the entire account balance is permanently burned.

### Finding Description

In `action_delete_account`, the execution sequence is:

1. Read `account_balance` from the live account.
2. Push `Receipt::new_balance_refund(&delete_account.beneficiary_id, account_balance)` onto `result.new_receipts`.
3. Call `remove_account(state_update, account_id)` — the account ceases to exist in the trie. [1](#0-0) 

The balance-refund receipt is a system receipt (`predecessor_id = "system"`). When it is processed in the next block and the receiver account does not exist, the `Transfer` action fails with `AccountDoesNotExist`. Because the predecessor is `"system"`, no further refund receipt is generated — the balance is burned.

The only validation performed on `beneficiary_id` before execution is:

```rust
fn validate_delete_action(action: &DeleteAccountAction) -> Result<(), ActionsValidationError> {
    validate_action_account_id(&action.beneficiary_id)?;
    Ok(())
}
``` [2](#0-1) 

This checks only that `beneficiary_id` is a syntactically valid account ID. There is no guard that `beneficiary_id != account_id` (the receipt's `receiver_id`, which is the account being deleted).

The `new_balance_refund` helper constructs a system receipt: [3](#0-2) 

When this receipt arrives at the deleted account, `check_transfer_to_nonexisting_account` returns `AccountDoesNotExist` (the account is not implicit), the action fails, and — because the predecessor is `"system"` — no refund chain is started. The tokens are burned.

### Impact Explanation

Any user who submits `DeleteAccount { beneficiary_id: <own_account_id> }` permanently loses their entire account balance. The token-conservation invariant is broken: tokens that should be transferred to a beneficiary are instead burned. The corrupted value is `account_ref.amount()` — the full liquid balance of the deleted account, which can be arbitrarily large.

Scope: **accounting** (token conservation).

### Likelihood Explanation

The input is fully unprivileged-user-controlled: `beneficiary_id` is a field in a user-submitted transaction. No special role is required. A user can trigger this accidentally (e.g., copy-pasting their own account ID as the beneficiary) or a malicious contract can call `promise_batch_action_delete_account` with the current account as beneficiary via the host function: [4](#0-3) 

No existing guard prevents `beneficiary_id == account_id` at either the transaction-validation layer or the receipt-execution layer.

### Recommendation

Add a self-reference check in `validate_delete_action` (or in `action_delete_account` before emitting the refund receipt):

```rust
fn validate_delete_action(
    action: &DeleteAccountAction,
    receiver: &AccountId,          // the account being deleted
) -> Result<(), ActionsValidationError> {
    validate_action_account_id(&action.beneficiary_id)?;
    if &action.beneficiary_id == receiver {
        return Err(ActionsValidationError::DeleteAccountBeneficiaryIsSelf);
    }
    Ok(())
}
```

This mirrors the fix applied in the Chiliz bridge (adding identity validation to prevent source == destination).

### Proof of Concept

```
// Alice (account_id = "alice.near", balance = 1000 NEAR) submits:
Transaction {
    signer_id:   "alice.near",
    receiver_id: "alice.near",
    actions: [DeleteAccount { beneficiary_id: "alice.near" }],
}

// Execution in action_delete_account:
//   account_balance = 1000 NEAR
//   new_receipts.push(Receipt::new_balance_refund("alice.near", 1000 NEAR))
//   remove_account("alice.near")   ← account deleted
//
// Next block: balance-refund receipt arrives at "alice.near"
//   → AccountDoesNotExist
//   → predecessor == "system" → no further refund
//   → 1000 NEAR permanently burned
```
