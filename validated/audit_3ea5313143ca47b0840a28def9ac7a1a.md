### Title
Missing Callback on wNEAR Unwrap Transfer Causes Permanent NEAR Lock at Aurora Engine - (`engine/src/contract_methods/connector.rs`)

---

### Summary

When a user exits wNEAR ERC-20 via the `ExitToNear` precompile with the `:unwrap` suffix (`Message::UnwrapWnear`), the engine executes a two-step promise chain: (1) `near_withdraw` on the wNEAR NEP-141 contract, then (2) `exit_to_near_precompile_callback` which issues a `PromiseAction::Transfer` to the user-specified NEAR account. The Transfer promise has **no callback**. If the Transfer fails — e.g., because the target account does not exist — the NEAR stays at the Aurora engine account permanently, while the wNEAR ERC-20 tokens were already burned with no refund path for this specific failure mode.

---

### Finding Description

**Entrypoint:** Any EVM transaction that calls the wNEAR ERC-20 contract's burn function, which invokes the `ExitToNear` precompile with flag `0x1` and a `receiver_account_id:unwrap` suffix.

**Step 1 — Precompile dispatch (`exit_erc20_token_to_near`):**

When `Message::UnwrapWnear` is matched and the caller is the registered wNEAR ERC-20 address, the function returns `method = "near_withdraw"` and populates `transfer_near_args` with the user-supplied `receiver_account_id`: [1](#0-0) 

The `receiver_account_id` is only validated as a syntactically correct NEAR account ID (format check). There is no on-chain existence check.

**Step 2 — Promise chain construction:**

The precompile's `run` method builds a `PromiseWithCallbackArgs`: `near_withdraw` as the base, `exit_to_near_precompile_callback` as the callback. The `refund` field is either `None` (without `error_refund` feature) or set to a `RefundCallArgs` (with the feature): [2](#0-1) 

**Step 3 — Callback execution (`exit_to_near_precompile_callback`):**

When `near_withdraw` succeeds, the callback creates a `PromiseBatchAction` with a single `PromiseAction::Transfer` targeting the user-supplied account. **No further callback is attached to this Transfer promise:** [3](#0-2) 

The `else if let Some(args) = args.refund` branch only fires when `near_withdraw` itself fails — it does not cover the case where `near_withdraw` succeeds but the subsequent Transfer fails: [4](#0-3) 

**Step 4 — Transfer failure:**

In NEAR Protocol, a `Transfer` action in a batch receipt to a non-existent account fails. When the receipt fails, the NEAR being transferred is not deducted from Aurora's balance — it remains at the Aurora engine account. There is no second-level callback to detect this failure and re-mint the burned ERC-20 tokens.

**The `error_refund` feature does not close this gap.** `refund_call_args` populates the `refund` field for the `near_withdraw` failure case only. The `ExitToNearPrecompileCallbackArgs` struct has no field for a "transfer failed" refund: [5](#0-4) 

---

### Impact Explanation

The user's wNEAR ERC-20 tokens are burned (ERC-20 balance decreases), `near_withdraw` succeeds (wNEAR NEP-141 is burned, NEAR is credited to Aurora engine), but the NEAR Transfer to the non-existent account fails silently. The NEAR is permanently stranded at the Aurora engine account with no recovery path. The invariant that "burned wNEAR ERC-20 must result in NEAR delivered to the recipient or a full ERC-20 refund" is violated.

---

### Likelihood Explanation

The path is reachable by any user who:
- Holds wNEAR ERC-20 on Aurora
- Calls the exit precompile with a syntactically valid but non-existent NEAR account ID and the `:unwrap` suffix

This can happen through a typo, a deleted account, or a freshly generated account ID that has never been initialized. No admin compromise or special privilege is required.

---

### Recommendation

Attach a second-level callback to the `PromiseAction::Transfer` promise created inside `exit_to_near_precompile_callback`. If the Transfer fails, the callback should re-mint the burned wNEAR ERC-20 tokens to the original sender (analogous to how `refund_on_error` handles `near_withdraw` failure). Concretely:

1. Add a `transfer_near_refund: Option<RefundCallArgs>` field to `ExitToNearPrecompileCallbackArgs` (or reuse the existing `refund` field with a new variant).
2. In `exit_to_near_precompile_callback`, instead of `handler.promise_return(promise_id)`, attach a new callback to `promise_id` that calls `exit_to_near_precompile_callback` again (or a dedicated `exit_to_near_transfer_callback`) with the refund args.
3. In that second callback, check `promise_result(0)`: if the Transfer failed, call `engine::refund_on_error` to re-mint the ERC-20 tokens.

---

### Proof of Concept

```
1. Deploy Aurora with wNEAR ERC-20 registered via `factory_set_wnear_address`.
2. Bridge wNEAR NEP-141 tokens into Aurora → user holds wNEAR ERC-20.
3. User calls the wNEAR ERC-20 burn function with ExitToNear input:
     flag=0x1 | amount (32 bytes) | "nonexistent-account-xyz.near:unwrap"
   where "nonexistent-account-xyz.near" does not exist on NEAR.
4. Observe:
   - wNEAR ERC-20 balance of user: decreased by `amount`
   - `near_withdraw` receipt: succeeds
   - `exit_to_near_precompile_callback` receipt: succeeds (creates Transfer promise)
   - Transfer receipt: fails (account does not exist)
   - NEAR balance of "nonexistent-account-xyz.near": 0
   - NEAR balance of Aurora engine: increased by `amount` (NEAR stranded)
   - wNEAR ERC-20 balance of user: NOT restored
5. Assert: user lost `amount` yoctoNEAR with no recourse.
```
