### Title
Missing Validation of `app_pk` Allows Confidential Derived Key Disclosure - (File: src/confidential_key_derivation/protocol.rs)

### Summary
The `ckd()` function in the CKD protocol accepts an arbitrary `app_pk: PublicKey` (a BLS12-381 G1 point) without validating that it is a non-identity, legitimate application public key. If a caller supplies the identity element as `app_pk`, the ElGamal encryption degenerates and the CKD output directly exposes the confidential derived key `S = msk · H(pk, app_id)` in plaintext.

### Finding Description
The CKD protocol is designed to deliver a deterministic secret `S = msk · H(pk, app_id)` to a TEE application, encrypted under the application's public key `app_pk` using ElGamal encryption. Each participant computes:

```
Y  = y · G
C  = S + y · app_pk
```

and the coordinator aggregates the Lagrange-weighted shares to produce the final `(Y, C)` output. The TEE application recovers `S` as `C − app_sk · Y`.

The `ckd()` entry point at line 66 of `src/confidential_key_derivation/protocol.rs` performs several structural checks (duplicate participants, self-membership, coordinator membership) but performs **no validation on `app_pk`**. In particular, it does not check that `app_pk` is not the group identity element.

If `app_pk = identity`:
- `app_pk * y = identity * y = identity` (the additive identity of G1)
- `C = S + identity = S`
- The coordinator's aggregated output becomes `C = Σ λ_i · S_i = S = msk · H(pk, app_id)`

The confidential derived key is emitted in the clear inside the `CKDOutput`.

The missing check is in `compute_signature_share` at line 174:

```rust
let big_c = big_s + app_pk * y.0;  // no guard: app_pk may be identity
``` [1](#0-0)

### Impact Explanation
**Critical — Extraction or disclosure of confidential derived secrets.**

When `app_pk` is the identity element, the ElGamal masking term vanishes entirely. The coordinator's final `CKDOutput.big_c` field equals `msk · H(pk, app_id)` — the exact confidential derived key the protocol is designed to protect. Any party that can invoke `ckd()` with a chosen `app_pk` (including the coordinator itself, or any participant node whose software is under attacker control) can recover the full confidential derived key without needing the TEE application's secret key.

### Likelihood Explanation
The `app_pk` parameter is fully caller-controlled with no library-level guard. A malicious coordinator, a compromised participant node, or any unprivileged library caller that initiates a CKD session can supply `ElementG1::identity()` (a valid, serializable value in all standard BLS12-381 Rust crates). No cryptographic break, key leakage, or external dependency compromise is required — only the ability to call `ckd()` with a chosen argument.

### Recommendation
Add an explicit check in `ckd()` (or in `compute_signature_share`) that rejects the identity element and any other structurally invalid `app_pk` before the protocol begins:

```rust
// Reject the identity element — it collapses ElGamal encryption
if app_pk == ElementG1::identity() {
    return Err(InitializationError::BadParameters(
        "app_pk must not be the identity element".to_string(),
    ));
}
```

Additionally, consider documenting that callers are responsible for verifying `app_pk` belongs to a trusted TEE application (analogous to whitelisting vault addresses in the ThorSwap report), since the library cannot enforce application-level trust policies.

### Proof of Concept
1. Caller invokes `ckd(participants, coordinator, me, key_pair, app_id, ElementG1::identity(), rng)`.
2. Each participant's `compute_signature_share` computes `big_c = big_s + identity * y = big_s`.
3. The coordinator aggregates: `C = Σ λ_i · big_s_i = msk · H(pk, app_id)`.
4. The coordinator's `CKDOutput` contains `big_c = S` — the confidential derived key — in plaintext.
5. The attacker reads `ckd_output.big_c` directly; no `app_sk` is needed to "decrypt." [3](#0-2)

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}
