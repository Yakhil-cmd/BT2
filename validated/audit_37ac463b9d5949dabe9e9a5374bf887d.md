### Title
Pool Admin Can Exceed Protocol Hard Fee Cap via Uncapped Per-Bin Additional Fees in `setPoolBinAdditionalFees` - (File: metric-core/contracts/MetricOmmPoolFactory.sol)

### Summary
The factory enforces a hard cap of `HARD_MAX_SPREAD_FEE_E6 = 200_000` (20%) on each spread fee component, but the pool admin can set per-bin additional fees (`addFeeBuyE6`, `addFeeSellE6`) through `setPoolBinAdditionalFees` with no cap validation whatsoever. These fees are added directly on top of the base spread fee in every swap, allowing the pool admin to silently charge traders more than the protocol's hard ceiling permits.

### Finding Description

The factory hard cap is enforced for the base spread fee components: [1](#0-0) 

```solidity
uint24 internal constant HARD_MAX_SPREAD_FEE_E6 = 200_000;
uint24 internal constant HARD_MAX_NOTIONAL_FEE_E8 = 1_000_000;
```

`setFeeCaps` enforces this ceiling on `maxAdminSpreadFeeE6` and `maxProtocolSpreadFeeE6`, and `setPoolAdminFees` validates against those caps: [2](#0-1) 

However, the pool admin path for per-bin fees performs **no cap check at all**: [3](#0-2) 

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

The pool itself also performs no cap check — only a bin-index range check: [4](#0-3) 

```solidity
function setBinAdditionalFees(int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external onlyFactory nonReentrant(PoolActions.SET_BIN_ADDITIONAL_FEES)
{
    if (bin < LOWEST_BIN || bin > HIGHEST_BIN) revert InvalidBinIndex(bin);
    BinState storage s = _binStates[bin];
    s.addFeeBuyE6 = addFeeBuyE6;
    s.addFeeSellE6 = addFeeSellE6;
    ...
}
```

`BinState.addFeeBuyE6` and `BinState.addFeeSellE6` are `uint16` fields (max value 65 535, i.e. **6.5535 % in E6 units**): [5](#0-4) 

In every swap these per-bin fees are added **on top of** the base spread fee with no ceiling: [6](#0-5) 

```solidity
uint256 buyFeeX64 = baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6);
uint256 sellFeeX64 = baseFeeX64 + Math.mulDiv(binState.addFeeSellE6, ONE_X64, 1e6);
```

The same uncapped path is used inside `_swapToken0ForToken1SpecifiedInput` / `_swapToken1ForToken0SpecifiedInput` and their `SpecifiedOut` counterparts: [7](#0-6) 

### Impact Explanation

A pool admin can call `setPoolBinAdditionalFees(pool, bin, 65535, 65535)` on any bin. With a base `spreadFeeE6` already at the allowed maximum (e.g. 200 000, i.e. 20 %), the effective per-bin fee becomes **20 % + 6.55 % = 26.55 %**, silently exceeding the protocol hard cap. Every trader whose swap routes through that bin pays the excess fee directly from their output amount — a direct, quantifiable loss of principal. The fee is charged inside the swap loop before the output token is transferred to the recipient, so there is no way for the trader to detect or avoid it at execution time.

### Likelihood Explanation

The pool admin role is semi-trusted and is explicitly in scope per the audit pivot ("Admin-boundary break: pool admin exceeds caps … look for bypasses in … `setBinAdditionalFees`"). The call requires only `onlyPoolAdmin(pool)` — no timelock, no owner approval, no two-step process. A malicious or compromised pool admin can execute this in a single transaction at any time after pool deployment.

### Recommendation

Add a cap check in `setPoolBinAdditionalFees` (factory) before forwarding to the pool, mirroring the existing check in `setPoolAdminFees`:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (addFeeBuyE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

Alternatively, enforce the cap at the pool level inside `setBinAdditionalFees` by reading `spreadFeeE6` and rejecting values that would push the total above `HARD_MAX_SPREAD_FEE_E6`.

### Proof of Concept

```solidity
// 1. Deploy pool with max admin spread fee
params.adminSpreadFeeE6 = 200_000; // 20% — at the hard cap
address pool = factory.createPool(params);

// 2. Pool admin sets per-bin additional fee to uint16 max (no revert)
vm.prank(admin);
factory.setPoolBinAdditionalFees(pool, 0, 65535, 65535);

// 3. Trader swaps through bin 0
// Effective buy fee = baseFeeX64 (≈20%) + 65535/1e6 (≈6.55%) = ≈26.55%
// Trader receives ~6.55% less output than the hard cap permits
// The excess fee accrues as pool surplus, collectible by the admin
```

The `setBinAdditionalFees` call on the pool succeeds because the only guard is the bin-index range check: [8](#0-7) 

and the factory forwards the call without any fee-magnitude validation: [3](#0-2)
