### Title
Malicious Participant Corrupts CKD Output via Unconstrained `public_key` in `KeygenOutput` — (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

The `ckd` public API accepts a caller-supplied `KeygenOutput` whose `public_key` field is used verbatim to compute the per-participant hash `H(pk ‖ app_id)`. No check is performed—either at initialization or during the protocol—that this `public_key` matches the actual master public key established during DKG. A malicious participant can substitute an arbitrary `public_key` in their `KeygenOutput`, causing their contribution to be computed over a different hash point. The coordinator blindly sums all contributions, accepting a corrupted CKD output.

---

### Finding Description

`compute_signature_share` in `src/confidential_key_derivation/protocol.rs` computes the hash point as:

```rust
let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);
let big_s = hash_point * private_share.to_scalar();
``` [1](#0-0) 

`key_pair.public_key` is taken directly from the caller-provided `KeygenOutput` with no constraint that it equals the real master public key `pk` established during DKG. The `ckd` entry point validates only participant-list membership: [2](#0-1) 

There is no `address = real_pk` style check analogous to the remediation in the external report. The coordinator then aggregates all participants' `(Y_i, C_i)` pairs without verifying that each participant hashed with the same `pk`:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
``` [3](#0-2) 

The `hash_app_id_with_pk` helper that embeds `pk` into the hash domain: [4](#0-3) 

`KeygenOutput` is a plain struct with two public fields and no internal consistency invariant between `private_share` and `public_key`: [5](#0-4) 

---

### Impact Explanation

Let honest participants use the real master public key `pk` and let the malicious participant `P_m` use `pk′ ≠ pk`. The coordinator computes:

```
C = (msk − λ_m · x_m) · H(pk ‖ app_id)
  + λ_m · x_m · H(pk′ ‖ app_id)
  + y · A
```

The app decrypts `C − a·Y`, obtaining a value that is **not** a valid BLS signature over `H(pk ‖ app_id)`. The app receives an unusable derived key. The coordinator and all honest parties accept this corrupted output without any error, because no cross-participant consistency check on `pk` exists.

This matches the allowed High impact: **Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.**

---

### Likelihood Explanation

Any participant who calls `ckd(...)` controls the `key_pair: KeygenOutput` argument entirely. Constructing a `KeygenOutput` with a real `private_share` and an arbitrary `public_key` requires no special capability beyond being a registered participant. The attack is a single-line substitution at call time and leaves no detectable trace in the protocol transcript.

---

### Recommendation

Enforce that `key_pair.public_key` equals the agreed master public key before the protocol proceeds. The simplest fix mirrors the external report's remediation: add an explicit address/equality check in `ckd` or `compute_signature_share`:

```rust
// In ckd() or compute_signature_share():
if key_pair.public_key != expected_master_pk {
    return Err(InitializationError::BadParameters(
        "key_pair.public_key does not match the master public key".to_string(),
    ));
}
```

Alternatively, broadcast `key_pair.public_key` from every participant in Round 1 and abort if any participant's value differs from the coordinator's, analogous to how `do_keyshare` checks `old_vk != verifying_key` during reshare: [6](#0-5) 

---

### Proof of Concept

1. Honest participants complete DKG and each hold a `KeygenOutput { private_share: x_i, public_key: pk }`.
2. Malicious participant `P_m` constructs `KeygenOutput { private_share: x_m, public_key: pk′ }` where `pk′` is any point in G2 different from `pk`.
3. All participants call `ckd(&participants, coordinator, me, key_pair, app_id, app_pk, rng)`. `P_m` passes the tampered `key_pair`.
4. Inside `compute_signature_share`, `P_m` computes `hash_point = H(pk′ ‖ app_id)` and `big_s = hash_point * x_m`, contributing a value inconsistent with the honest participants' hash domain.
5. The coordinator sums all `(Y_i, C_i)` pairs without detecting the inconsistency and returns a `CKDOutput`.
6. The app decrypts the output and obtains a value that fails BLS signature verification against `pk`, rendering the derived key unusable.
7. The honest coordinator and app have no mechanism to detect that `P_m` used a different `pk′`.

/// Hashes the app id and the public key as of
/// H(pk || `app_id`) where H is a random oracle
pub fn hash_app_id_with_pk(pk: &VerifyingKey, app_id: &[u8]) -> ElementG1 {
    let compressed_pk = pk.to_element().to_compressed();

    ciphersuite::hash_to_curve(&input)
}
