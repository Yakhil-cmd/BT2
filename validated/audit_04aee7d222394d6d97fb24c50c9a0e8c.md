### Title
Pool Rewards Permanently Frozen When `pool_balance < min_delegation_for_rewards` — (`src/pool/pool.cairo`)

### Summary

In the Pool contract, `update_rewards_from_staking_contract` calls `compute_rewards_per_unit`, which returns `0` when `total_stake < min_delegation_for_rewards`. However, the staking contract has **already transferred** the STRK reward tokens to the pool contract before this call. Because the `cumulative_rewards_trace` (sigma) does not advance, no pool member can ever claim those tokens — they are permanently frozen in the pool contract.

This is the direct analog of the LoopFi H-01 bug: in LoopFi, `lastBalance` advances while `index` stays flat; here, the pool's STRK balance advances while `cumulative_rewards_trace` stays flat.

---

### Finding Description

**Step 1 — Staking contract calculates and transfers pool rewards unconditionally.**

In `_update_rewards` → `calculate_staker_pools_rewards`, pool rewards are computed in 18-decimal normalized units:

```cairo
let pool_rewards_including_commission = if total_stake.is_non_zero() {
    mul_wide_and_div(
        lhs: total_rewards,
        rhs: pool_balance_curr_epoch.to_amount_18_decimals(),
        div: total_stake.to_amount_18_decimals(),
    )
    ...
``` [1](#0-0) 

If `pool_rewards > 0`, the entry is appended to `pool_rewards_array` and `update_pool_rewards` is called:

```cairo
if pool_rewards.is_non_zero() {
    pool_rewards_array.append(
        (pool_contract, token_address, pool_balance_curr_epoch, pool_rewards),
    );
}
``` [2](#0-1) 

Inside `update_pool_rewards`, STRK tokens are **transferred first**, then the pool is notified:

```cairo
self.send_rewards_to_delegation_pool(
    :staker_address,
    pool_address: pool_contract,
    amount: pool_rewards,
    token_dispatcher: strk_token_dispatcher,
);
...
pool_dispatcher.update_rewards_from_staking_contract(
    rewards: pool_rewards,
    pool_balance: pool_balance.to_native_amount(:decimals),
);
``` [3](#0-2) 

**Step 2 — Pool contract silently drops the rewards when balance is below the minimum.**

`update_rewards_from_staking_contract` inserts `last + compute_rewards_per_unit(...)` into the cumulative trace:

```cairo
fn update_rewards_from_staking_contract(
    ref self: ContractState, rewards: Amount, pool_balance: Amount,
) {
    self.assert_caller_is_staking_contract();
    let (_, last) = self.cumulative_rewards_trace.last().unwrap();
    self.cumulative_rewards_trace.insert(
        key: self.get_current_epoch(),
        value: last + self.compute_rewards_per_unit(
            staking_rewards: rewards, total_stake: pool_balance,
        ),
    );
}
``` [4](#0-3) 

`compute_rewards_per_unit` returns **zero** when `total_stake < min_delegation_for_rewards`:

```cairo
fn compute_rewards_per_unit(
    self: @ContractState, staking_rewards: Amount, total_stake: Amount,
) -> Index {
    // Return zero if the total stake is too small, to avoid overflow below.
    if total_stake < self.min_delegation_for_rewards.read() {
        return Zero::zero();
    }
    ...
}
``` [5](#0-4) 

The code itself acknowledges the consequence:

> **Note**: Delegation rewards lost when pool balance is less than `min_delegation_for_rewards`. The staking contract continues to forward `pool_rewards` to the pool contract even in this case. [6](#0-5) 

**Step 3 — Sigma never advances; pool members can never claim the stuck tokens.**

All pool member reward claims go through `calculate_rewards`, which computes `amount * (sigma_to - sigma_from) / base_value`. If sigma never advanced, the difference is zero and no rewards are distributed regardless of how much STRK sits in the pool contract. [7](#0-6) 

---

### Impact Explanation

STRK tokens are irrecoverably locked in the pool contract. There is no administrative withdrawal function, no recovery path, and no mechanism to redistribute the stuck tokens. Every reward cycle where `pool_balance < min_delegation_for_rewards` silently burns that epoch's pool rewards.

This matches the allowed impact: **Permanent freezing of unclaimed yield (High)**.

---

### Likelihood Explanation

**BTC pools (realistic):** For a BTC token with 8 decimals, `min_delegation_for_rewards = 10^(8−5) = 1000 satoshis` and `base_value = 10^13`. [8](#0-7) 

The staking contract normalizes the pool balance to 18 decimals for reward calculation, so a pool with 999 satoshis (native) = `9.99 × 10^12` (normalized) still generates non-zero `pool_rewards` in the staking contract's arithmetic. Those rewards are transferred to the pool contract, but `compute_rewards_per_unit` returns 0 because `999 < 1000`.

**STRK pools (lower likelihood):** `min_delegation_for_rewards = 10^18` (1 STRK). [9](#0-8) 

A pool with any balance below 1 STRK triggers the same path. New pools or pools where delegators have partially exited can transiently fall below this threshold.

---

### Recommendation

Before transferring rewards to the pool contract, the staking contract should check whether the pool balance (in native decimals) meets `min_delegation_for_rewards`. If it does not, skip the transfer and the `update_rewards_from_staking_contract` call entirely, so no tokens are sent that cannot be credited.

Alternatively, the pool contract's `update_rewards_from_staking_contract` should revert (or the staking contract should gate the call) when `pool_balance < min_delegation_for_rewards`, preventing the token transfer from occurring in the first place.

---

### Proof of Concept

1. Deploy a BTC pool (8 decimals). A single delegator enters with 500 satoshis (`pool_balance = 500`, native).
2. The sequencer calls `update_rewards`. The staking contract computes `pool_rewards` using the 18-decimal normalized balance (`5 × 10^12`), yielding a non-zero STRK reward.
3. `update_pool_rewards` calls `send_rewards_to_delegation_pool`, transferring, say, `R` STRK to the pool contract.
4. `pool.update_rewards_from_staking_contract(rewards: R, pool_balance: 500)` is called.
5. `compute_rewards_per_unit(staking_rewards: R, total_stake: 500)` returns `0` because `500 < 1000`.
6. `cumulative_rewards_trace` is updated with `last + 0 = last` — sigma is unchanged.
7. The delegator calls `claim_rewards`.

```text
        fn calculate_rewards(
            self: @ContractState,
            pool_member: ContractAddress,
            from_checkpoint: PoolMemberCheckpoint,
            until_checkpoint: PoolMemberCheckpoint,
            mut entry_to_claim_from: VecIndex,
        ) -> (Amount, VecIndex) {
            let pool_member_trace = self.pool_member_epoch_balance.entry(pool_member);
            // Note: `until_epoch` is the current epoch.
            let until_epoch = until_checkpoint.epoch();

            let mut rewards = 0;

            let pool_member_trace_length = pool_member_trace.length();

            let mut from_sigma = self.find_sigma(from_checkpoint, curr_epoch: until_epoch);
            let mut from_balance = from_checkpoint.balance();

            let base_value = self.staking_rewards_base_value.read();

            // **Note**: The loop iterates over the balance changes in the pool member's balance
            // trace. This loop is unbounded but unlikely to exceed gas limits.
            while entry_to_claim_from < pool_member_trace_length {
                let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
                // If the balance change is after `until_epoch` (and therefore does not affect
                // the current reward computation), exit the loop.
                if pool_member_checkpoint.epoch() >= until_epoch {
                    break;
                }

                // Compute rewards from (inclusive) the previous balance change (or from
                // `from_checkpoint`) to (exclusive) the current entry.
                let to_sigma = self.find_sigma(pool_member_checkpoint, curr_epoch: until_epoch);
                rewards +=
                    compute_rewards_rounded_down(
                        amount: from_balance, interest: to_sigma - from_sigma, :base_value,
                    );
                from_sigma = to_sigma;
                from_balance = pool_member_checkpoint.balance();
                entry_to_claim_from += 1;
            }

            // Compute the remaining rewards from (inclusive) the last visited balance change in
            // `pool_member_trace` (or from `from_checkpoint`) to (exclusive) `until_checkpoint`.
            let to_sigma = self.find_sigma(until_checkpoint, curr_epoch: until_epoch);
            rewards +=
                compute_rewards_rounded_down(
                    amount: from_balance, interest: to_sigma - from_sigma, :base_value,
                );

            (rewards, entry_to_claim_from)
        }
