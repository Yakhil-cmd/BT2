### Title
Malicious Coordinator Can Permanently Block FROST EdDSA Signing by Withholding the Signing Package — (File: src/frost/eddsa/sign.rs)

---

### Summary

In the FROST EdDSA `sign_v1` protocol, non-coordinator participants send their round-1 commitments to the coordinator and then enter an **unbounded blocking loop** waiting for the coordinator to return the signing package. A malicious coordinator can collect all commitments and then simply never send the signing package, permanently halting the signing protocol for every honest participant with no bypass, timeout, or fallback mechanism in the library.

---

### Finding Description

In `do_sign_participant_v1` (`src/frost/eddsa/sign.rs`), after a participant sends its round-1 commitment to the coordinator at line 265, it enters the following loop at lines 272–280:

```rust
// Step 2.1
let r2_wait_point = chan.next_waitpoint();
let signing_package = loop {
    let (from, signing_package): (_, frost_ed25519::SigningPackage) =
        chan.recv(r2_wait_point).await?;
    if from != coordinator {
        continue;
    }
    break signing_package;
};
``` [1](#0-0) 

This loop has **no timeout, no abort condition, and no bypass path**. The only exit is receiving a message from the coordinator. If the coordinator never sends the signing package, every non-coordinator participant is permanently suspended at this `await` point.

The identical structural flaw exists in the RedJubjub signing participant path, where participants wait indefinitely for the coordinator's randomizer:

```rust
let randomizer = loop {
    let (from, randomizer): (_, Randomizer) = chan.recv(wait_round_1).await?;
    if from != coordinator {
        continue;
    }
    break randomizer;
};
``` [2](#0-1) 

The coordinator role is not restricted to a trusted external party — it is simply one of the signing participants, chosen by the caller: [3](#0-2) 

The protocol specification (`docs/eddsa/signing.md`) confirms that step 1.5 is exclusively the coordinator's responsibility — sending `(commits, m)` to every participant — and that step 2.1 is a mandatory blocking wait on that message: [4](#0-3) 

There is no mechanism in the library for honest participants to detect coordinator non-responsiveness and proceed or abort with a meaningful error.

---

### Impact Explanation

**High — Permanent denial of signing for honest parties.**

A malicious coordinator can:
1. Receive all round-1 commitments from every honest participant (gaining knowledge of their binding and hiding nonce commitments).
2. Refuse to send the signing package.
3. All honest participants remain permanently blocked in the `loop { chan.recv(...).await? }` call with no library-level escape.

The signing protocol never completes. The presignature material (nonces) consumed by participants in round 1 is now spent and cannot be safely reused, so honest parties must restart from scratch with fresh nonces — but they cannot do so without first aborting the stalled session, which the library provides no mechanism for.

This matches the allowed impact: **"High: Permanent denial of signing … for honest parties under valid protocol inputs and documented trust assumptions."**

---

### Likelihood Explanation

The coordinator is an ordinary participant selected by the caller at protocol initialization. Any participant who is designated coordinator can exploit this. In a setting where participants are mutually distrusting (the standard threshold-signature threat model), a single malicious participant who is ever selected as coordinator can permanently block any signing session they are assigned to coordinate. No special privilege beyond being named coordinator is required.

---

### Recommendation

1. **Introduce a coordinator-wait timeout**: The `chan.recv(r2_wait_point).await?` call should be wrapped with a configurable deadline. If the deadline expires without receiving the coordinator's message, the participant should return a `ProtocolError` (e.g., `ProtocolError::CoordinatorTimeout`) rather than blocking indefinitely.

2. **Document the coordinator trust assumption explicitly**: If the coordinator is assumed honest by design, this must be clearly stated in the public API documentation for `sign_v1` and `sign` (RedJubjub), so callers can enforce it at the application layer.

3. **Consider coordinator-rotation**: Allow honest participants to detect an unresponsive coordinator and re-run the protocol with a different coordinator, analogous to the recommendation in the external report to allow users to call `resolveAuction` themselves after a timeout.

---

### Proof of Concept

```
Setup: 3 participants P1 (coordinator), P2, P3; threshold = 2.

1. All three call sign_v1(..., coordinator = P1, ...).

2. P2 and P3 execute do_sign_participant_v1:
   - Each generates nonces and commitments.
   - Each sends commitments to P1 via chan.send_private(commit_waitpoint, coordinator, &commitments).

3. P1 (malicious) executes do_sign_coordinator_v1:
   - Receives commitments from P2 and P3.
   - Constructs the signing_package internally.
   - Does NOT call chan.send_many(r2_wait_point, &signing_package).
   - Simply drops the channel / returns early.

4. P2 and P3 are now permanently suspended at:
     chan.recv(r2_wait_point).await?
   inside the `loop { ... }` at sign.rs:273–279.

5. No library-level error is ever returned to P2 or P3.
   The signing session is permanently stalled.
   The nonces consumed in step 2 cannot be safely reused.
```
