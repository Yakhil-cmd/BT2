### Title
Malicious Participant Can Corrupt CKD Output by Substituting `app_pk` Without Detection — (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

In the Confidential Key Derivation (CKD) protocol, each participant independently uses the `app_pk` (application ElGamal public key) supplied to their own `ckd()` call. There is no cross-participant commitment or verification that all participants used the same `app_pk`. A single malicious participant can silently substitute an arbitrary `app_pk` in their share computation, corrupting the final aggregated ciphertext so that the legitimate application cannot recover the derived secret. The coordinator has no mechanism to detect the substitution.

---

### Finding Description

The CKD protocol is a threshold ElGamal encryption of `msk · H(pk ‖ app_id)` under the application's public key `app_pk`. Each participant computes a share of the ciphertext in `compute_signature_share`:

```rust
// C <- S + y . A
let big_c = big_s + app_pk * y.0;
``` [1](#0-0) 

The coordinator then aggregates every participant's `(norm_big_y, norm_big_c)` by simple addition:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
``` [2](#0-1) 

No proof-of-knowledge, commitment, or consistency check is attached to the shares. The coordinator cannot distinguish a share computed with the legitimate `app_pk` from one computed with an attacker-chosen `malicious_pk`.

**Attack path:**

A malicious participant `j` calls `ckd()` with `malicious_pk` instead of the agreed `app_pk`. Their contribution becomes:

```
norm_big_c_j = λ_j · (big_s_j + malicious_pk · y_j)
```

After aggregation the coordinator holds:

```
big_c = msk · H(pk, app_id)
      + app_pk · Σ_{i≠j} λ_i · y_i
      + malicious_pk · λ_j · y_j
```

The legitimate application attempts decryption:

```
big_c − app_sk · big_y  ≠  msk · H(pk, app_id)
```

because the `malicious_pk` term does not cancel. The derived secret is irrecoverably corrupted for that CKD invocation.

The root cause is that `app_pk` is accepted as an unauthenticated caller-supplied parameter with no cross-participant binding, exactly analogous to the oracle parameter in the NFTPairWithOracle report: a critical cryptographic parameter can be silently swapped by one party after the protocol context has been established, with no detection by honest parties. [3](#0-2) 

---

### Impact Explanation

**High — Corruption of CKD outputs so honest parties accept inconsistent or unusable cryptographic outputs.**

The coordinator and all honest participants complete the protocol without error, yet the resulting `CKDOutput` is an invalid ciphertext. The application (e.g., a TEE) that calls `ckd_output.unmask(app_sk)` receives a garbage group element instead of `msk · H(pk, app_id)`. Every downstream operation that depends on the derived secret (key derivation, decryption, authentication) silently fails or produces wrong results. Because the protocol produces no error, honest parties have no indication that the output is unusable.

---

### Likelihood Explanation

Any single participant in the CKD protocol can mount this attack unilaterally. The attacker only needs to pass a different `PublicKey` value to their own `ckd()` call — a trivial one-line deviation from the honest protocol. No cryptographic capability, no colluding majority, and no privileged position (coordinator or otherwise) is required. The attack is deterministic and always succeeds.

---

### Recommendation

Bind `app_pk` into the protocol transcript so that all participants commit to the same value before shares are exchanged:

1. **Commitment round**: Before computing shares, each participant broadcasts `H(app_pk)`. The coordinator (and each participant) aborts if any received hash differs from their own.
2. **Alternatively**, include `app_pk` in the session-ID / domain-separator hash (analogous to how `session_id` is derived from all participants' random contributions in `do_keyshare`), so that a participant using a different `app_pk` operates in a different session and their shares are cryptographically incompatible.
3. **Alternatively**, require participants to attach a Sigma-protocol proof that `norm_big_c_i − λ_i · big_s_i = app_pk · norm_big_y_i` (a proof of correct ElGamal encryption under the agreed key), though this requires exposing `big_s_i` commitments.

The minimal fix analogous to the NFTPairWithOracle mitigation is option 1: add a broadcast-and-check round for `H(app_pk)` before any secret material is exchanged.

### Proof of Concept

```
Setup:
  participants = [P1 (honest), P2 (honest), P3 (malicious)]
  app_sk, app_pk = app_sk · G   (legitimate application key pair)
  malicious_pk   = random_scalar · G  (attacker-chosen, ≠ app_pk)

P1 calls: ckd(..., app_pk, ...)   → norm_big_c_1 = λ1·(S1 + app_pk·y1)
P2 calls: ckd(..., app_pk, ...)   → norm_big_c_2 = λ2·(S2 + app_pk·y2)
P3 calls: ckd(..., malicious_pk, ...) → norm_big_c_3 = λ3·(S3 + malicious_pk·y3)

Coordinator aggregates (no error raised):
  big_c = msk·H(pk,app_id) + app_pk·(λ1·y1 + λ2·y2) + malicious_pk·λ3·y3
  big_y = λ1·y1·G + λ2·y2·G + λ3·y3·G

Application decrypts:
  big_c − app_sk·big_y
  = msk·H(pk,app_id) + malicious_pk·λ3·y3 − app_sk·λ3·y3·G
  = msk·H(pk,app_id) + λ3·y3·(malicious_pk − app_pk)
  ≠ msk·H(pk,app_id)   (since malicious_pk ≠ app_pk)

Result: CKD output is silently corrupted; no error is returned by the protocol.

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
