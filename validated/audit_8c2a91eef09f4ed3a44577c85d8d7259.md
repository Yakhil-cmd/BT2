### Title
COA EVM Address Derived from Global UUID Counter Instead of Owner Identity, Enabling Relayer Address Manipulation - (File: fvm/evm/stdlib/contract.cdc, fvm/evm/handler/addressAllocator.go)

---

### Summary

When a Cadence-Owned Account (COA) is created via `EVM.createCadenceOwnedAccount()`, its EVM address is derived deterministically from the Cadence resource's `uuid`, which is a **global, block-and-transaction-ordering-dependent counter**. The address does not incorporate the owner's identity in any way. This mirrors the PRBProxy vulnerability: just as `tx.origin` (the gas payer) was used instead of the owner, Flow uses a global UUID counter instead of the Cadence account that will own the COA. In relayer/gasless scenarios, a malicious or incorrectly-ordered relayer can cause a user to receive a different COA EVM address than predicted, with no on-chain mechanism for the user to validate the expected address before creation.

---

### Finding Description

**Address derivation chain:**

`EVM.createCadenceOwnedAccount()` in `contract.cdc` creates a `CadenceOwnedAccount` resource, reads its `uuid`, and passes it to `InternalEVM.createCadenceOwnedAccount(uuid: acc.uuid)`:

```cadence
let acc <-create CadenceOwnedAccount()
let addr = InternalEVM.createCadenceOwnedAccount(uuid: acc.uuid)
acc.initAddress(addressBytes: addr)
``` [1](#0-0) 

This calls `handler.DeployCOA(uint64(uuid))` in Go, which calls `AllocateCOAAddress(uuid)`:

```go
target := h.addressAllocator.AllocateCOAAddress(uuid)
``` [2](#0-1) 

`AllocateCOAAddress` applies a deterministic bijective mapping over the uuid:

```go
func (aa *AddressAllocator) AllocateCOAAddress(uuid uint64) types.Address {
    return MakeCOAAddress(uuid)
}
func MakeCOAAddress(index uint64) types.Address {
    return makePrefixedAddress(mapAddressIndex(index), types.FlowEVMCOAAddressPrefix)
}
func mapAddressIndex(index uint64) uint64 {
    return uint64(index * addressIndexMultiplierConstant)
}
``` [3](#0-2) 

**The uuid is a global counter, not owner-bound:**

The Cadence `uuid` is generated by `GenerateUUID()` in `uuids.go`, which reads and increments a per-partition counter. The partition is determined by `(blockHeader.ID(), txnIndex)`:

```go
generator.partition = uuidPartition(generator.blockHeader.ID(), generator.txnIndex)
// ...
return ((counter & 0xFF_FF00_0000_0000) << 8) | (uint64(generator.partition) << 40) | (counter & 0xFF_FFFF_FFFF), nil
``` [4](#0-3) 

The uuid — and therefore the COA EVM address — depends entirely on:
1. Which block the transaction is included in
2. The transaction's index within that block
3. How many Cadence resources were created before this COA in the same transaction

**No owner identity is incorporated.** There is no view function to predict the COA address before creation.

---

### Impact Explanation

In a relayer/gasless scenario:

1. A relayer predicts Alice's COA EVM address based on the current UUID counter state.
2. Alice pre-funds the predicted EVM address (a standard EVM pattern for pre-funding CREATE2 addresses), or signs off-chain EVM messages/approvals referencing that address.
3. The relayer includes another resource-creating transaction before Alice's in the same block, incrementing the UUID counter.
4. Alice's COA is deployed at a different EVM address than predicted.
5. Funds sent to the predicted address are now at an EVM address with no COA controlling it, effectively lost or inaccessible to Alice.

Additionally, any pre-signed EVM transactions (e.g., ERC-20 approvals, bridge interactions) that referenced the predicted COA address are now invalid or point to the wrong address.

There is no on-chain mechanism for Alice to validate the expected COA address before the transaction executes, mirroring the exact design flaw described in the PRBProxy report.

---

### Likelihood Explanation

This requires a relayer or gasless transaction scenario, which is a common and growing use case in EVM ecosystems (EIP-2771 meta-transactions, EIP-4337 UserOperations, and similar patterns are explicitly mentioned in the PRBProxy report as motivating contexts). Any Flow application that offers gasless COA creation — where the transaction fee payer differs from the COA owner — is exposed. The attacker-controlled entry path is an unprivileged transaction sender acting as a relayer who controls transaction ordering within a block.

---

### Recommendation

1. **Incorporate owner identity into the COA address derivation.** Instead of deriving the EVM address solely from the global UUID, include the Cadence owner account address in the salt/seed. For example: `MakeCOAAddress(hash(ownerAddress, uuid))`. This ensures the address is bound to the owner, not just the global counter.

2. **Provide a view function** that allows a user to predict their COA EVM address given their Cadence account address and current state, before the transaction executes. This is the direct analog to the recommendation in the PRBProxy report.

3. **Alternatively**, use a per-account nonce (tracked per Cadence account address) instead of the global UUID counter, so the address depends on the owner's identity and their personal deployment sequence.

---

### Proof of Concept

**Step 1**: Relayer observes current UUID counter state and predicts Alice's COA address will be `MakeCOAAddress(N)`.

**Step 2**: Alice pre-funds EVM address `MakeCOAAddress(N)` with FLOW tokens, expecting her COA to be deployed there.

**Step 3**: Relayer submits a transaction that creates any Cadence resource (e.g., a `FlowToken.Vault`) before Alice's COA creation transaction in the same block. This increments the UUID counter to `N+1`.

**Step 4**: Alice's `EVM.createCadenceOwnedAccount()` executes, gets UUID `N+1`, and her COA is deployed at `MakeCOAAddress(N+1)`.

**Step 5**: The FLOW tokens Alice pre-funded at `MakeCOAAddress(N)` are now at an EVM address with no COA controlling it. Alice has no recourse.

The deterministic address mapping confirms this:
```go
// uuid=N   → address MakeCOAAddress(N)   = prefix + mapAddressIndex(N)
// uuid=N+1 → address MakeCOAAddress(N+1) = prefix + mapAddressIndex(N+1)
// These are always distinct (bijective mapping), so Alice's funds are stranded.
``` [5](#0-4) [6](#0-5) [7](#0-6)

func MakePrecompileAddress(index uint64) types.Address {
	return makePrefixedAddress(index, types.FlowEVMExtendedPrecompileAddressPrefix)
}

func makePrefixedAddress(
	index uint64,
	prefix [types.FlowEVMSpecialAddressPrefixLen]byte,
) types.Address {
	var addr types.Address
	copy(addr[:], prefix[:])
	// only works if `len(addr) - len(prefix)` is exactly 8 bytes
	binary.BigEndian.PutUint64(addr[len(prefix):], index)
	return addr
}

// `mapAddressIndex` maps an index of 64 bits to a deterministic random-looking 64 bits.
//
// The mapping function must be an injective mapping (in this case bijective)
// where every two indices always map to two different results. Multiple injective
// mappings are possible.
//
// The current implementation uses a simple modular multiplication by a constant modulo 2^64.
// The multiplier constant can be any odd number. Since odd numbers are co-prime with 2^64, they
// have a multiplicative inverse modulo 2^64.
// This makes multiplying by an odd number an injective function (and therefore bijective).
//
// Multiplying modulo 2^64 is implicitly implemented as a uint64 multiplication with a uin64 result.
func mapAddressIndex(index uint64) uint64 {
	return uint64(index * addressIndexMultiplierConstant)

// GenerateUUID generates a new uuid and persist the data changes into state
func (generator *uUIDGenerator) GenerateUUID() (uint64, error) {
	defer generator.tracer.StartExtensiveTracingChildSpan(
		trace.FVMEnvGenerateUUID).End()

	err := generator.meter.MeterComputation(
		common.ComputationUsage{
			Kind:      ComputationKindGenerateUUID,
			Intensity: 1,
		},
	)
	if err != nil {
		return 0, fmt.Errorf("generate uuid failed: %w", err)
	}

	generator.maybeInitializePartition()

	counter, err := generator.getCounter()
	if err != nil {
		return 0, fmt.Errorf("cannot generate UUID: %w", err)
	}

	err = generator.setCounter(counter + 1)
	if err != nil {
		return 0, fmt.Errorf("cannot generate UUID: %w", err)
	}

	// Since the partition counter only goes up to MaxUint56, we can use the
	// assemble a UUID value with the partition (P) and the counter (C).
	// Note: partition (P) is represented by the 6th byte
	// (C7 C6) | P | (C5 C4 C3 C2 C1)
	return ((counter & 0xFF_FF00_0000_0000) << 8) | (uint64(generator.partition) << 40) | (counter & 0xFF_FFFF_FFFF), nil
