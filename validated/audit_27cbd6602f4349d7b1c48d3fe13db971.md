### Title
Integer Division Truncation in XCC Precompile Undercharges EVM Gas for Attached NEAR Gas - (File: engine-precompiles/src/xcc.rs)

### Summary
The XCC (Cross-Contract Call) precompile converts user-specified NEAR gas into an EVM gas cost using integer division that truncates (rounds down). This means every XCC call with a non-zero `attached_gas` remainder systematically undercharges the caller, allowing up to `CROSS_CONTRACT_CALL_NEAR_GAS - 1 = 174,999,999` NEAR gas units to be consumed per call without any corresponding EVM gas payment.

### Finding Description
In `engine-precompiles/src/xcc.rs`, line 174, the EVM gas cost for the NEAR gas attached to a cross-contract call promise is computed as:

```rust
cost += EthGas::new(promise.attached_gas.as_u64() / costs::CROSS_CONTRACT_CALL_NEAR_GAS);
``` [1](#0-0) 

The constant `CROSS_CONTRACT_CALL_NEAR_GAS = 175_000_000` has units of NEAR Gas per EVM Gas. [2](#0-1) 

For any `attached_gas = k * 175_000_000 + r` (where `0 ≤ r < 175_000_000`):
- **EVM gas charged** = `k`
- **NEAR gas actually consumed** = `k * 175_000_000 + r`
- **NEAR gas not paid for** = `r` (up to 174,999,999 units per call)

The `attached_gas` value originates from the user-supplied `CrossContractCallArgs` decoded from calldata: [3](#0-2) 

The `total_gas()` of the promise is directly derived from user-controlled promise arguments, making the remainder `r` fully attacker-controlled.

### Impact Explanation
The Aurora Engine contract prepays NEAR gas for XCC calls from its own NEAR balance. When the EVM gas charged to the caller is less than the actual NEAR gas consumed, the protocol subsidizes the difference. A caller can craft XCC calls with `attached_gas` values that maximize the remainder (e.g., `attached_gas = N * 175_000_000 + 174_999_999`), extracting up to 174,999,999 NEAR gas units per call without paying the corresponding EVM gas cost. Repeated exploitation across many calls constitutes a systematic gas/fee-accounting deficit — an insolvency vector against the Aurora contract's NEAR gas budget.

### Likelihood Explanation
Any unprivileged EVM user who calls the XCC precompile with a non-zero `attached_gas` value that is not an exact multiple of `CROSS_CONTRACT_CALL_NEAR_GAS` triggers this rounding error. The entry path is fully permissionless: a user submits an Ethereum transaction calling the XCC precompile address (`0x516cded1d16af10cad47d6d49128e2eb7d27b372`) with crafted `CrossContractCallArgs`. No special privileges are required. [4](#0-3) 

### Recommendation
Replace the truncating division with ceiling division to ensure the EVM gas charged always covers the full NEAR gas cost:

```rust
// Before (truncates):
cost += EthGas::new(promise.attached_gas.as_u64() / costs::CROSS_CONTRACT_CALL_NEAR_GAS);

// After (ceiling division):
cost += EthGas::new(
    promise.attached_gas.as_u64()
        .div_ceil(costs::CROSS_CONTRACT_CALL_NEAR_GAS)
);
```

This mirrors the fix recommended in the GroupBuy report: round up rather than down when computing a per-unit price from a total, so that the collected amount always meets or exceeds the actual cost.

### Proof of Concept
1. Construct a `CrossContractCallArgs::Eager` promise with `attached_gas = 174_999_999` (one unit below the conversion ratio).
2. The computed EVM gas addition is `174_999_999 / 175_000_000 = 0` — zero EVM gas charged for the NEAR gas.
3. The Aurora contract still forwards 174,999,999 NEAR gas units to the router, consuming real NEAR gas budget.
4. Repeat across many calls; each call extracts up to 174,999,999 NEAR gas units at zero EVM gas cost, systematically draining the protocol's NEAR gas budget. [5](#0-4)
