### Title
Unbounded `stakers` Vec Growth Causes Permanent DoS of `get_stakers()` - (File: `src/staking/staking.cairo`)

### Summary

The `stakers: Vec<ContractAddress>` storage field in the Staking contract grows without bound because staker addresses are appended on every `stake()` call but are **never removed** when a staker unstakes. The `get_stakers()` function iterates the entire vector on every call, so as the vector grows, gas consumption grows linearly until the function becomes permanently uncallable.

### Finding Description

The storage field is declared with an explicit acknowledgment of the problem:

```
/// Vector of staker addresses.
/// **Note**: Stakers are not removed from this vector when they unstake.
stakers: Vec<ContractAddress>,
``` [1](#0-0) 

On every `stake()` call, the caller's address is appended unconditionally:

```cairo
// Add staker address to the stakers vector.
self.stakers.push(staker_address);
``` [2](#0-1) 

On `unstake_action()`, the staker's `staker_info` record is erased and their balance zeroed, but their address is **never popped or swapped out** of `stakers`. The `remove_staker` internal function does not touch the vector at all: [3](#0-2) 

`get_stakers()` then iterates the **full** vector on every invocation, reading each slot from storage and calling `is_staker_active` (additional storage reads) per entry:

```cairo
for staker_address_ptr in self.stakers.into_iter_full_range() {
    let staker_address = staker_address_ptr.read();
    if !self.is_staker_active(:staker_address, :epoch_id) {
        continue;
    }
    ...
}
``` [4](#0-3) 

`assert_staker_address_not_reused` prevents the same address from re-staking, so each unique address occupies exactly one slot. However, an unbounded number of distinct addresses can stake and unstake over the protocol's lifetime, causing the vector to grow without limit.

### Impact Explanation

`get_stakers()` is the function the consensus layer calls to enumerate the active validator set for a given epoch. As the `stakers` vector grows (each entry requiring at least one storage read plus additional reads inside `is_staker_active`), the gas cost of `get_stakers()` grows linearly with the total number of addresses that have **ever** staked, not just those currently active. Once the vector is large enough, the function will exceed Starknet's per-transaction resource limits and revert on every call, permanently breaking the consensus layer's ability to determine the validator set. This maps to **permanent freezing** of the protocol's consensus and reward distribution mechanism.

### Likelihood Explanation

Any unprivileged address can call `stake()` with the minimum stake amount, then call `unstake_intent()` and wait for the exit window before calling `unstake_action()`. By cycling many distinct addresses (e.g., via lightweight proxy contracts or a script), an attacker can bloat the vector cheaply relative to the damage caused. The protocol is intended to run indefinitely, so even organic usage will eventually cause the vector to grow large enough to be problematic.

### Recommendation

Remove the staker's address from the `stakers` vector during `unstake_action()` / `remove_staker()`. Since `Vec` in Cairo does not support O(1) removal by value, the standard approach is a **swap-and-pop**: store a reverse mapping from `staker_address → Vec index`, swap the target entry with the last entry, update the reverse mapping for the swapped entry, then pop the last element. Alternatively, replace the `Vec` with an `IterableMap` (already used elsewhere in the contract for `btc_tokens`) which supports deletion.

### Proof of Concept

1. Deploy the staking contract.
2. In a loop, for each of N distinct addresses:
   - Fund the address with `min_stake` STRK.
   - Call `stake(...)` → address is appended to `stakers`.
   - Call `unstake_intent()`.
   - Advance time past `exit_wait_window`.
   - Call `unstake_action(staker_address)` → address remains in `stakers`.
3. After N iterations, call `get_stakers(epoch_id)`.
4. Observe that gas consumption scales as O(N) and eventually exceeds the block/transaction gas limit, causing the call to revert.

The existing test `get_stakers_staker_exit_action_flow_test` already confirms that after `unstake_action`, the staker's address **remains** in the `stakers` vector (the test only checks the returned span is empty, not the underlying vector length). [5](#0-4) [1](#0-0)

        fn get_stakers(
            self: @ContractState, epoch_id: Epoch,
        ) -> Span<(ContractAddress, StakingPower, Option<PublicKey>, Option<PeerId>)> {
            let curr_epoch = self.get_current_epoch();
            assert!(
                curr_epoch <= epoch_id && epoch_id < curr_epoch + K.into(),
                "{}",
                Error::INVALID_EPOCH,
            );

            let (strk_total_stake, btc_total_stake) = self
                .get_total_staking_power_at_epoch(:epoch_id);

            let mut stakers: Array<
                (ContractAddress, StakingPower, Option<PublicKey>, Option<PeerId>),
            > =
                array![];
            for staker_address_ptr in self.stakers.into_iter_full_range() {
                let staker_address = staker_address_ptr.read();
                if !self.is_staker_active(:staker_address, :epoch_id) {
                    continue;
                }

                let staking_power = self
                    .get_staker_staking_power_at_epoch(
                        :staker_address, :epoch_id, :strk_total_stake, :btc_total_stake,
                    );
                if staking_power.is_zero() {
                    continue;
                }

                let public_key = self.get_public_key_at_epoch(:staker_address, :epoch_id);
                let peer_id = self.get_peer_id_at_epoch(:staker_address, :epoch_id);
                stakers.append((staker_address, staking_power, public_key, peer_id));
            }
            stakers.span()
        }
