### Title
Cross-Address COA Ownership Proof Replay via Missing EVM Address Binding in `validateCOAOwnershipProof` - (File: `fvm/evm/stdlib/contract.cdc`)

### Summary

`EVM.validateCOAOwnershipProof` does not enforce that `signedData` encodes the target `evmAddress`. A signature produced for one COA (Cadence-Owned Account) can be replayed against a different COA owned by the same Cadence account, because the proof's `CapabilityPath` field is attacker-controlled and the signature is only verified over the raw `signedData` bytes — not over any COA-specific context. This is the direct analog of the external report's pattern: a handler that delegates all replay protection to an external component (the settlement contract / the COA path check) without binding the proof to the specific execution context.

### Finding Description

`EVM.validateCOAOwnershipProof` in `fvm/evm/stdlib/contract.cdc` verifies that:
1. The provided signatures are valid over `signedData` using the Cadence account's keys.
2. The COA resource stored at the given `path` has an EVM address matching the supplied `evmAddress`.

The code itself documents the gap:

> *"Note: this function does not enforce that `signedData` includes `evmAddress`. In principle, a signature produced for one purpose could be replayed here against a different COA owned by the same Cadence account."* [1](#0-0) 

The `COAOwnershipProof` struct that is RLP-encoded and passed as `_sig` to `isValidSignature` contains a `CapabilityPath` field that is fully attacker-controlled: [2](#0-1) 

The COA's ERC-1271 `isValidSignature` implementation passes `address(this)` as the `evmAddress` argument to the precompile, then delegates all validation to `validateCOAOwnershipProof`: [3](#0-2) 

The precompile handler invokes `validateCOAOwnershipProof` with the proof's fields verbatim, including the attacker-supplied `path`: [4](#0-3) 

### Impact Explanation

**Attack scenario** (requires same Cadence account to own two COAs at different paths):

1. Victim account `A` owns `COA_1` at `/public/coa` and `COA_2` at `/public/coa2`.
2. Victim legitimately signs data `D` to authorize an ERC-1271 check on `COA_1`. The resulting `COAOwnershipProof` contains `path=/public/coa`, `signedData=D`, and valid signatures.
3. Attacker observes this proof (e.g., from a public NFT marketplace listing or on-chain calldata).
4. Attacker constructs a new proof with `path=/public/coa2` but the **same** `signedData=D` and **same** signatures.
5. Attacker calls `COA_2.isValidSignature(D, modifiedProof)`.
6. `validateCOAOwnershipProof` succeeds:
   - Signature over `D` is valid (same key, same data).
   - COA at `/public/coa2` is `COA_2`, whose address matches `address(this)` = `COA_2`'s EVM address.
7. `isValidSignature` returns `ValidERC1271Signature` for `COA_2` — a forgery.

Any EVM contract using ERC-1271 to gate asset transfers, token approvals, or account-abstraction operations on `COA_2` can be deceived into accepting this forged authorization, leading to unauthorized cross-VM asset access. [5](#0-4) 

### Likelihood Explanation

- **Entry path**: Fully reachable by any unprivileged EVM transaction sender. No staked node, admin key, or privileged role is required.
- **Precondition**: The victim Cadence account must own at least two COAs at different public paths. While uncommon today (the comment notes "Flow wallets historically create at most one COA per account"), the protocol imposes no such limit, and multi-COA accounts are a valid and growing use case (e.g., protocol-managed vaults, multi-purpose dApps).
- **Signature acquisition**: The attacker only needs to observe a legitimate ERC-1271 call to `COA_1` — no social engineering is required. Signatures are routinely public in NFT marketplace listings, permit flows, and on-chain calldata.

### Recommendation

Enforce that `signedData` commits to the target `evmAddress`. The simplest fix is to require callers to include the EVM address in the signed payload and verify this inside `validateCOAOwnershipProof`:

```cadence
// Inside validateCOAOwnershipProof, after signature verification:
// Require that signedData encodes evmAddress as its first 20 bytes (or via a hash commitment).
let encodedAddr = evmAddress.toConstantSized<[UInt8; 20]>()!
// verify signedData contains evmAddress...
```

Alternatively, the EVM-side `isValidSignature` implementation in `coa.sol` should hash `address(this)` into `_hash` before passing it to the precompile, so the signed data is always COA-address-specific:

```solidity
bytes32 boundHash = keccak256(abi.encodePacked(address(this), _hash));
cadenceArch.staticcall(abi.encodeWithSignature(
    "verifyCOAOwnershipProof(address,bytes32,bytes)", address(this), boundHash, _sig
));
```

### Proof of Concept

1. Deploy two COAs from the same Cadence account: `COA_1` at `/public/coa`, `COA_2` at `/public/coa2`.
2. Sign arbitrary data `D` with the account's key, producing `proof_1 = {path: /public/coa, signedData: D, signatures: [sig]}`.
3. Construct `proof_2 = {path: /public/coa2, signedData: D, signatures: [sig]}` (only `path` changed).
4. Call `COA_2.isValidSignature(D, rlp_encode(proof_2))` from any EVM address.
5. Observe `ValidERC1271Signature` (0x1626ba7e) returned — `COA_2` has accepted a signature it never authorized. [6](#0-5) [7](#0-6) [8](#0-7)
