Audit Report

## Title
Silent Burn Failure in Cycles Minting Canister Breaks ICP Supply Conservation - (File: `rs/nns/cmc/src/main.rs`)

## Summary
The `burn_and_log` function in the Cycles Minting Canister deliberately discards ICP ledger errors when burning ICP after successfully minting cycles or creating canisters. If the ICP ledger is temporarily unavailable (e.g., during a routine NNS upgrade), cycles are minted or canisters are created without the corresponding ICP being burned. Because the notification block index is finalized as processed regardless of burn outcome, there is no retry path and the unburned ICP is permanently stranded in CMC's subaccount, breaking the ICP/cycles supply conservation invariant.

## Finding Description
In `rs/nns/cmc/src/main.rs`, `burn_and_log` is called after every successful operation in `process_create_canister` (L1945), `process_top_up` (L2001), and `process_mint_cycles` (L1968). The function issues a `send_pb` call to the ICP ledger. On failure, it only logs the error and returns `()` — no error is propagated, no retry is scheduled, and no corrective state is recorded (L2042–2048). The callers do not check the return value because it is `()`. The design comment at L2014–2016 explicitly acknowledges this trade-off: *"Burning doesn't return errors - we don't want to reject the transaction notification because then it could be retried."* There is no `PendingBurn` state, no timer-based retry, and no recovery mechanism anywhere in the CMC codebase. Once the notification is finalized as `NotifiedTopUp`/`NotifiedCreateCanister`/`NotifiedMint`, the block index is permanently marked processed and cannot be re-submitted by the user.

## Impact Explanation
Each failed burn permanently inflates the effective ICP supply relative to cycles minted, violating the core ICP/cycles economic invariant. The unburned ICP accumulates in CMC's subaccount with no protocol-level mechanism to recover or re-burn it. This constitutes a significant NNS/ledger protocol harm: the ICP ledger's total supply is higher than the protocol requires, and the discrepancy is irreversible per incident. This matches the allowed High impact: *"Significant NNS, SNS, or infrastructure security impact with concrete user or protocol harm."*

## Likelihood Explanation
The ICP ledger undergoes routine upgrades on the NNS subnet, during which it is temporarily unavailable for seconds to minutes. NNS upgrade proposals are publicly visible, making the upgrade window predictable. Any unprivileged user who sends ICP to CMC's subaccount and calls `notify_top_up`, `notify_create_canister`, or `notify_mint_cycles` while the ledger is upgrading will trigger this path. No special privileges, keys, or coordination are required. The condition is repeatable across every ledger upgrade cycle.

## Recommendation
Track failed burns in persistent CMC state. On a `burn_and_log` failure, insert a `PendingBurn { subaccount, amount }` entry into a dedicated map in CMC state. Process pending burns on a periodic timer, retrying the `send_pb` call until it succeeds. This is analogous to how ckBTC/ckETH minters handle failed mint retries. Alternatively, record the failed burn in `blocks_notified` state and process it on subsequent timer ticks, ensuring every successful cycles-minting operation is eventually matched by a confirmed ICP burn.

## Proof of Concept
1. Monitor NNS governance for an upcoming ICP ledger upgrade proposal.
2. Send N ICP to `CMC_CANISTER_ID` subaccount with `MEMO_TOP_UP_CANISTER`.
3. Wait for the ledger upgrade to begin (canister is stopped/upgrading).
4. Call `notify_top_up` on CMC during the upgrade window.
5. CMC calls `deposit_cycles` → succeeds, cycles are deposited to the target canister.
6. CMC calls `burn_and_log(sub, amount)` → ledger returns a reject (unavailable).
7. `burn_and_log` logs the error and returns `()` (L2044–2047).
8. `process_top_up` returns `Ok(cycles)` to the caller (L2002).
9. The block index is finalized as `NotifiedTopUp` in `blocks_notified`.
10. Result: N ICP worth of cycles minted; N ICP remain in CMC's subaccount unburned; notification cannot be retried; supply conservation invariant permanently broken for this transaction.
