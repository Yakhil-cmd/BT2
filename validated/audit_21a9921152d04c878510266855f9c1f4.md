### Title
Malicious Participant Can Corrupt CKD Output by Sending Arbitrary Share Contributions Without Verification - (File: src/confidential_key_derivation/protocol.rs)

### Summary
The `do_ckd_coordinator` function in `src/confidential_key_derivation/protocol.rs` aggregates participant-contributed `(big_y, big_c)` values without any cryptographic validation. A single malicious participant can send arbitrary or zero values, causing the coordinator to output a corrupted `CKDOutput`. The derived confidential key will be permanently wrong and unusable, with no way for honest parties to detect or recover from the corruption.

### Finding Description

The CKD protocol is a threshold BLS-based protocol where each participant computes a signature share `(norm_big_y, norm_big_c)` using their private signing share `x_i`, a random blinding scalar `y`, and the application public key `app_pk`. The coordinator aggregates all contributions and outputs the final `CKDOutput`.

In `do_ckd_coordinator`:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
``` [1](#0-0) 

The coordinator blindly sums every received `(big_y, big_c)` pair with no proof of correctness. There is no zero-knowledge proof, no commitment binding, and no consistency check that the received `big_c` was computed as `x_i * H(pk || app_id) + y * app_pk` using the participant's actual secret share.

Contrast this with the DKG protocol in `src/dkg.rs`, which enforces:
1. A proof of knowledge of the secret coefficient (`proof_of_knowledge` / `verify_proof_of_knowledge`)
2. A commitment hash binding (`verify_commitment_hash`)
3. Per-share verification against the public commitment (`validate_received_share`) [2](#0-1) 

The CKD protocol has none of these safeguards. The participant-side function `do_ckd_participant` is not even `async` ‚Äî it fires and forgets, with no round-trip that would allow the coordinator to challenge or verify the contribution:

```rust
fn do_ckd_participant(...) -> Result<CKDOutputOption, ProtocolError> {
    let (norm_big_y, norm_big_c) =
        compute_signature_share(participants, me, key_pair, app_id, app_pk, rng)?;
    let waitpoint = chan.next_waitpoint();
    chan.send_private(waitpoint, coordinator, &(norm_big_y, norm_big_c))?;
    Ok(None)
}
``` [3](#0-2) 

### Impact Explanation

The `CKDOutput::unmask(app_sk)` operation recovers the confidential key as:

```
total_big_c - app_sk * total_big_y
= Œ£ Œª_i * (x_i * H(pk||app_id) + y_i * app_pk) - app_sk * Œ£ Œª_i * y_i * G
= H(pk||app_id) * Œ£ Œª_i * x_i   (= H(pk||app_id) * msk)
```

If a malicious participant `m` sends `(big_y', big_c')` instead of their correct share, the unmask yields:

```
H(pk||app_id) * Œ£_{i‚âÝm} Œª_i * x_i  +  Œª_m * (big_c' - app_sk * big_y')
```

This is not equal to `H(pk||app_id) * msk` unless the malicious contribution happens to cancel perfectly. The coordinator outputs a permanently wrong confidential key with no error signal. Honest parties have no mechanism to detect the corruption, and the protocol cannot be re-run to fix it without re-keying.

This matches the allowed impact: **High ‚Äî Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.**

### Likelihood Explanation

Any single participant in the CKD protocol can trigger this. The attacker needs only to be a registered participant (holding a valid key share from DKG) and deviate from the protocol by sending `(0, 0)` or any arbitrary group elements. No special privileges, no coordinator access, and no cryptographic break are required. The attack is trivially executable by any malicious participant.

### Recommendation

Add a zero-knowledge proof of correct share computation to each participant's contribution. Specifically, each participant should prove in zero-knowledge that their submitted `big_c` satisfies:

```
big_c - y * app_pk = x_i * H(pk || app_id)
```

where `x_i` is consistent with the participant's public commitment from DKG (i.e., the verification share `g^{x_i}` is known). A DLEQ (Discrete Log Equality) proof can demonstrate that the discrete log of `big_c - y * app_pk` with respect to `H(pk || app_id)` equals the discrete log of the participant's public verification share with respect to the generator `G`. The coordinator must verify this proof before accepting any contribution.

This mirrors the approach already used in `src/dkg.rs` where `validate_received_share` verifies each secret share against the public polynomial commitment before accepting it. [4](#0-3)

fn validate_received_share<C: Ciphersuite>(
    me: Participant,
    from: Participant,
    signing_share_from: &SigningShare<C>,
    commitment: &VerifiableSecretSharingCommitment<C>,
) -> Result<(), ProtocolError> {
    let id = me.to_identifier::<C>()?;

    // The verification is exactly the same as the regular SecretShare verification;
    // however the required components are in different places.
    // Build a temporary SecretShare so what we can call verify().
    let secret_share = SecretShare::new(id, *signing_share_from, commitment.clone());

    // Verify the share. We don't need the result.
    // Identify the culprit if an InvalidSecretShare error is returned.

        if let Error::InvalidSecretShare { .. } = e {
            ProtocolError::InvalidSecretShare(from)
        } else {
            ProtocolError::AssertionFailed(format!(
                "could not
            extract the verification key matching the secret
            share sent by {from:?}"
            ))
        }
    })?;
    Ok(())
}
