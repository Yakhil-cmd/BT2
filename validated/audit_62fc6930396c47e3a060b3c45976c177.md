### Title
Unauthorized `evm/<contract>` Token Drain via Bank Precompile `transfer` — (`x/cronos/keeper/precompiles/bank.go`)

---

### Summary

The `TransferMethodName` case in `BankContract.Run` accepts an arbitrary `sender` address from ABI-decoded calldata and passes it directly to `bankKeeper.SendCoins` as the `from` account, with **no check** that `sender == contract.Caller()`. Any EVM contract can therefore drain `evm/<itself>` tokens from any holder without that holder's consent.

---

### Finding Description

In `BankContract.Run`, the `transfer` branch: [1](#0-0) 

`sender` is taken directly from ABI argument `args[0]` — fully attacker-controlled. The denom is correctly restricted to `EVMDenom(contract.Caller())`: [2](#0-1) 

But then `SendCoins` is called with `from = sdk.AccAddress(sender.Bytes())`, where `sender` is the attacker-supplied address, not `contract.Caller()`: [3](#0-2) 

There is no guard of the form `if sender != contract.Caller() { return error }` anywhere in this path. The only checks present are:
- `readonly` flag check [4](#0-3) 
- `amount.Sign() <= 0` check [5](#0-4) 
- `checkBlockedAddr` on the **recipient** only [6](#0-5) 

None of these prevent an arbitrary `from` address.

---

### Impact Explanation

A deployed contract A can:
1. Mint `evm/A` tokens to victim V (via the `mint` method, which is an intended capability).
2. Call `bank.transfer(V, A, balance)` — passing victim's address as `sender`.
3. `bankKeeper.SendCoins(ctx, from=V, to=A, evm/A coins)` executes unconditionally.
4. V's entire `evm/A` balance is transferred to A without V's knowledge or consent.

This is an **unauthorized transfer** of native Cosmos bank tokens, matching the Critical impact category: *Unauthorized transfer of precompile-controlled assets*.

---

### Likelihood Explanation

The attack requires only:
- Deploying a contract (permissionless on Cronos).
- Distributing `evm/<contract>` tokens to victims (the contract itself controls minting).
- Calling the precompile's `transfer` method with victim's address as `sender`.

No privileged access, leaked keys, or external assumptions are needed. The path is fully on-chain and locally reproducible.

---

### Recommendation

Add an authorization check at the start of the `TransferMethodName` branch, enforcing that the `sender` argument must equal `contract.Caller()`:

```go
case TransferMethodName:
    // ...
    sender := args[0].(common.Address)
    if sender != contract.Caller() {
        return nil, errors.New("transfer sender must be the calling contract")
    }
    // ... rest of logic
```

Alternatively, remove the `sender` argument entirely and always use `contract.Caller()` as `from`, consistent with how `mint`/`burn` use `contract.Caller()` for the denom authority.

---

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

interface IBank {
    function mint(address recipient, uint256 amount) external returns (bool);
    function transfer(address sender, address recipient, uint256 amount) external returns (bool);
    function balanceOf(address token, address account) external view returns (uint256);
}

contract AttackerContract {
    IBank constant bank = IBank(address(100)); // bankContractAddress = bytes{100}

    // Step 1: mint evm/<this> tokens to victim
    function setupVictim(address victim, uint256 amount) external {
        bank.mint(victim, amount);
    }

    // Step 2: drain victim's evm/<this> tokens without victim's consent
    function drain(address victim, address attacker, uint256 amount) external {
        bank.transfer(victim, attacker, amount);
        // victim's balance drops; attacker receives tokens
    }
}
```

**Test plan:**
1. Deploy `AttackerContract` at address `A`.
2. Call `setupVictim(victim, 1000)` — victim now holds 1000 `evm/A` tokens.
3. Call `drain(victim, attacker, 1000)` — asserts victim balance drops to 0, attacker balance increases by 1000, with no transaction signed by victim.
