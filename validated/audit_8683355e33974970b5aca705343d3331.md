### Title
Missing On-Chain Access Control on `update_rewards` Allows Any Caller to Block Per-Block Reward Distribution - (File: `src/staking/staking.cairo`)

### Summary
The `update_rewards` function in the staking contract enforces a global per-block guard (`last_reward_block`) to prevent double-distribution in the same block. However, the function has **no on-chain access control** — the spec mandates "Only starkware sequencer" but the code only calls `general_prerequisites()`, which checks `assert_is_unpaused()` and `assert_caller_is_not_zero()`. Any unprivileged actor can call `update_rewards(any_valid_staker, disable_rewards: true)` before the legitimate sequencer call, consuming the block's reward slot and causing the legitimate call to revert with `REWARDS_ALREADY_UPDATED`. Stakers lose all block rewards for every griefed block.

### Finding Description

`update_rewards` in `src/staking/staking.cairo` is the consensus-phase reward distribution entry point. Its only gate is:

```cairo
fn general_prerequisites(ref self: ContractState) {
    self.assert_is_unpaused();
    assert_caller_is_not_zero();
}
``` [1](#0-0) 

No role check, no sequencer check, no allowlist. The spec explicitly states access is restricted to "Only starkware sequencer": [2](#0-1) 

Inside `update_rewards`, the global `last_reward_block` is written **before** the `disable_rewards` branch:

```cairo
// Update last block rewards.
self.last_reward_block.write(current_block_number);

if disable_rewards || self.is_pre_consensus() {
    return;
}
``` [3](#0-2) 

The guard that prevents a second call in the same block is:

```cairo
assert!(
    current_block_number > self.last_reward_block.read(),
    "{}",
    Error::REWARDS_ALREADY_UPDATED,
);
``` [4](#0-3) 

`last_reward_block` is a **single global slot** — not per-staker: [5](#0-4) 

An attacker who calls `update_rewards(any_active_staker, disable_rewards: true)` in block N will:
1. Pass the `REWARDS_ALREADY_UPDATED` guard (new block).
2. Write `last_reward_block = N`.
3. Return early — no rewards distributed.
4. Cause every subsequent `update_rewards` call in block N to revert with `REWARDS_ALREADY_UPDATED`.

The test suite itself demonstrates that a second call in the same block always panics: [6](#0-5) 

### Impact Explanation

Every block in which the attacker fires first, **all stakers lose their consensus block rewards** for that block. Because `last_reward_block` is global, a single attacker transaction targeting any valid staker address is sufficient to suppress rewards for the entire protocol for that block. Repeated across many blocks this constitutes sustained theft of unclaimed yield with zero cost to the attacker beyond gas.

This matches the allowed impact: **High — Theft of unclaimed yield**.

### Likelihood Explanation

- `update_rewards` is a public, permissionless entry point (no role check in code).
- The attacker only needs one valid staker address (publicly readable from events/storage) and enough gas.
- The call with `disable_rewards: true` costs minimal gas and has no token transfer.
- On Starknet the sequencer orders transactions, but the sequencer is not the same entity as the keeper/bot expected to call `update_rewards`; if the sequencer includes attacker transactions before the keeper's, the attack succeeds. Even if the sequencer is the caller, the missing on-chain guard means any future sequencer decentralization or keeper rotation immediately opens the window.

### Recommendation

Add an explicit on-chain access control check at the top of `update_rewards`, analogous to the check already present in `update_current_epoch_block_rewards` in the reward supplier:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
    self.roles.only_rewards_manager(); // or assert_caller_is_sequencer()
    ...
}
```

Alternatively, introduce a dedicated `REWARDS_MANAGER` role and grant it only to the authorized sequencer address, mirroring the pattern used for `CALLER_IS_NOT_STAKING_CONTRACT` in the reward supplier. [7](#0-6) 

### Proof of Concept

```
Block N:
  Tx 1 (attacker): update_rewards(alice_staker, disable_rewards=true)
    → passes REWARDS_ALREADY_UPDATED guard (new block)
    → last_reward_block := N
    → returns early, no rewards distributed

  Tx 2 (sequencer/keeper): update_rewards(alice_staker, disable_rewards=false)
    → assert!(N > N)  ← FALSE
    → reverts: REWARDS_ALREADY_UPDATED

Result: alice_staker (and all other stakers, since last_reward_block is global)
        receives zero block rewards for block N.
```

The existing test already confirms the second-call revert behavior: [8](#0-7)
