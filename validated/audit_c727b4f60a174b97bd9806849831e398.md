### Title
Pool Admin Can Set Uncapped Per-Bin Additional Fees, Bypassing the Protocol Hard Spread Fee Cap — (File: metric-core/contracts/MetricOmmPoolFactory.sol)

---

### Summary

`setPoolBinAdditionalFees` in `MetricOmmPoolFactory` forwards `addFeeBuyE6` and `addFeeSellE6` directly to the pool with no cap validation. The protocol enforces a hard cap of `HARD_MAX_SPREAD_FEE_E6 = 200_000` (20%) on base spread fees, but the per-bin additional fees are `uint16` values (max 65 535, i.e. 6.5535 % in E6 scale) with zero enforcement. A pool admin can therefore push the effective per-bin swap fee above the protocol hard cap.

---

### Finding Description

The factory defines hard limits for all base spread and notional fees: [1](#0-0) 

`setPoolAdminFees` correctly enforces those caps before writing to the pool: [2](#0-1) 

`setPoolBinAdditionalFees`, however, performs **no cap check** and passes the caller-supplied values straight through: [3](#0-2) 

The pool-side `setBinAdditionalFees` only validates the bin index, never the fee magnitudes: [4](#0-3) 

During every swap, `addFeeBuyE6` / `addFeeSellE6` are added directly to `baseFeeX64` (the oracle-derived spread) before the price is quoted to the trader: [5](#0-4) 

The same unchecked addition appears in all four swap paths: [6](#0-5) 

---

### Impact Explanation

`uint16` max is 65 535, which in E6 scale equals **6.5535 %**. With the base spread fee already at the hard cap of 20 %, a pool admin can push the effective per-bin fee to **≈ 26.55 %** — 33 % above the protocol hard cap — on any individual bin. Every trader whose swap routes through that bin receives a worse execution price than the protocol's own hard cap is supposed to guarantee. This is an unclamped bid/ask quote reaching a pool swap and an admin-boundary break (pool admin exceeds caps).

---

### Likelihood Explanation

The pool admin is a semi-trusted role. The admin must be malicious or compromised to exploit this, but the path requires only a single permissioned call with no timelock. Any pool whose admin key is weak or transferable is immediately at risk.

---

### Recommendation

Add a hard cap check inside `setPoolBinAdditionalFees` (and mirror it in `setBinAdditionalFees` as a defence-in-depth guard):

```solidity
// MetricOmmPoolFactory.sol
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (addFeeBuyE6  > MAX_BIN_ADDITIONAL_FEE_E6) revert BinAdditionalFeeTooHigh();
    if (addFeeSellE6 > MAX_BIN_ADDITIONAL_FEE_E6) revert BinAdditionalFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

`MAX_BIN_ADDITIONAL_FEE_E6` should be set to a value consistent with `HARD_MAX_SPREAD_FEE_E6` (e.g. the same 200 000, or a separate configurable cap governed by `setFeeCaps`).

---

### Proof of Concept

1. Pool admin calls `factory.setPoolBinAdditionalFees(pool, 0, 65535, 65535)`.
2. The factory forwards the values to `pool.setBinAdditionalFees(0, 65535, 65535)` with no validation.
3. `_binStates[0].addFeeBuyE6 = 65535` and `addFeeSellE6 = 65535` are stored.
4. A trader calls `swap(...)` routing through bin 0. The effective buy fee becomes:

   ```
   buyFeeX64 = baseFeeX64 + mulDiv(65535, ONE_X64, 1e6)
             = oracle_spread_X64 + 6.5535%_X64
   ```

   On top of the pool's `spreadFeeE6` (up to 20 %), the trader faces a total effective spread of **≈ 26.55 %**, exceeding the protocol hard cap of 20 % with no revert or guard.
