### Title
Silo `Address` Whitelist Bypass for ERC-20 Token Receipt When No Fallback Address Is Configured - (File: `engine/src/engine.rs`)

---

### Summary

In Aurora's Silo mode, the `Address` whitelist is intended to restrict which EVM addresses may receive bridged ERC-20 tokens. However, the whitelist check inside `receive_erc20_tokens` is gated behind a `if let Some(fallback_address) = ...` guard. When no `erc20_fallback_address` is configured, the entire whitelist check is silently skipped, and any EVM address — including non-whitelisted ones — can receive ERC-20 tokens via the `ft_on_transfer` bridge path.

---

### Finding Description

The Silo subsystem exposes `is_allow_receive_erc20_tokens`, which checks the `Address` whitelist to decide whether an EVM address is permitted to receive bridged ERC-20 tokens. This check is called in `Engine::receive_erc20_tokens`:

```rust
// engine/src/engine.rs lines 818-822
if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
    && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
{
    recipient = fallback_address;
}
```

This is a Rust `if let … &&` chain. The entire block — including the whitelist check — is only evaluated when `get_erc20_fallback_address` returns `Some`. When no fallback address is stored (the default state, since `get_erc20_fallback_address` returns `None` if the key is absent), the condition short-circuits and `is_allow_receive_erc20_tokens` is **never called**. The caller-supplied `recipient` address is used unconditionally. [1](#0-0) 

`get_erc20_fallback_address` returns `None` whenever the storage key is absent:

```rust
// engine/src/contract_methods/silo/mod.rs lines 59-62
pub fn get_erc20_fallback_address<I: IO>(io: &I) -> Option<Address> {
    let key = erc20_fallback_address_key();
    io.read_storage(&key)?.to_value().ok()
}
``` [2](#0-1) 

The fallback address can also be explicitly cleared to `None` via `set_erc20_fallback_address(None)` or `set_silo_params(None)`:

```rust
// engine/src/contract_methods/silo/mod.rs lines 65-73
pub fn set_erc20_fallback_address<I: IO>(io: &mut I, address: Option<Address>) {
    let key = erc20_fallback_address_key();
    if let Some(address) = address {
        io.write_storage(&key, address.as_bytes());
    } else {
        io.remove_storage(&key);  // clears the key → get_erc20_fallback_address returns None
    }
}
``` [3](#0-2) 

By contrast, the `submit` path enforces the whitelist unconditionally via `assert_access`, which calls `silo::is_allow_submit` regardless of any fallback address:

```rust
// engine/src/engine.rs lines 1756-1775
fn assert_access<I: IO + Copy, E: Env>(
    io: &I,
    env: &E,
    transaction: &NormalizedEthTransaction,
) -> Result<(), EngineError> {
    let allowed = if transaction.to.is_some() {
        silo::is_allow_submit(io, &env.predecessor_account_id(), &transaction.address)
    } else {
        silo::is_allow_deploy(io, &env.predecessor_account_id(), &transaction.address)
    };
    if !allowed {
        return Err(EngineError { kind: EngineErrorKind::NotAllowed, gas_used: 0 });
    }
    Ok(())
}
``` [4](#0-3) 

The `ft_on_transfer` entry point calls `receive_erc20_tokens` directly, with no independent whitelist gate:

```rust
// engine/src/contract_methods/connector.rs lines 81-90
let result = if predecessor_account_id == get_connector_account_id(&io)? {
    engine.receive_base_tokens(&args)
} else {
    engine.receive_erc20_tokens(
        &predecessor_account_id,
        &args,
        &current_account_id,
        handler,
    )
};
``` [5](#0-4) 

The `SiloParamsArgs` documentation itself acknowledges this conditional behavior:

> "Note: the logic described above works only if the fallback address is set by `set_silo_params` function. In other words, in Silo mode." [6](#0-5) 

This means the `Address` whitelist for ERC-20 receipt is silently inoperative whenever the fallback address is absent — a configuration that is both the default state and reachable at any time by the owner calling `set_erc20_fallback_address(None)`.

---

### Impact Explanation

**Impact: High — Temporary Freezing of Funds**

When a Silo operator enables the `Address` whitelist (to restrict which EVM addresses may participate) but has not configured an `erc20_fallback_address`, any NEAR account can call `ft_transfer_call` on a registered NEP-141 token, specifying Aurora as the receiver and any arbitrary non-whitelisted EVM address as the recipient. Aurora's `ft_on_transfer` → `receive_erc20_tokens` will mint the ERC-20 tokens to that address without any whitelist check.

Because the same `Address` whitelist also gates `submit` (via `assert_access`), the non-whitelisted recipient address cannot submit EVM transactions to move or recover those tokens. The tokens are effectively frozen at an address that has no ability to act within the Silo, until the operator either adds the address to the whitelist or configures a fallback address and re-routes them.

---

### Likelihood Explanation

**Likelihood: High**

- The vulnerable state (whitelist enabled, no fallback address) is the **default** for any Silo that enables the `Address` whitelist without also calling `set_erc20_fallback_address` or `set_silo_params`.
- The fallback address can be cleared at any time by the owner, re-opening the window.
- The attack requires only a standard `ft_transfer_call` on any registered NEP-141 token — a normal, permissionless NEAR operation available to any token holder.
- No special privileges, leaked keys, or governance capture are required.

---

### Recommendation

Remove the dependency on the fallback address from the whitelist enforcement logic. The whitelist check should be unconditional:

```rust
// Proposed fix in engine/src/engine.rs
if !silo::is_allow_receive_erc20_tokens(&self.io, &recipient) {
    if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io) {
        recipient = fallback_address;
    } else {
        return Err(/* appropriate error */);
    }
}
```

This ensures that when the `Address` whitelist is enabled, a non-whitelisted recipient is always rejected or redirected — regardless of whether a fallback address is configured.

---

### Proof of Concept

1. Silo operator enables the `Address` whitelist: `set_whitelist_status({ kind: Address, active: true })`.
2. Silo operator does **not** call `set_erc20_fallback_address` (or calls it with `None`). `get_erc20_fallback_address` now returns `None`.
3. A NEP-141 token `token.near` is registered with Aurora via `deploy_erc20_token`.
4. Attacker holds `token.near` tokens and calls `ft_transfer_call` on `token.near`, with `receiver_id = aurora`, `amount = X`, `msg = <hex-encoded non-whitelisted EVM address>`.
5. NEAR runtime calls `aurora.ft_on_transfer(sender_id=attacker, amount=X, msg=<address>)`.
6. `ft_on_transfer` → `receive_erc20_tokens` is invoked. At line 818, `get_erc20_fallback_address` returns `None`, so the `if let` block is skipped entirely.
7. `is_allow_receive_erc20_tokens` is **never called**. The non-whitelisted address is used as `recipient`.
8. The ERC-20 mint call succeeds; the non-whitelisted address now holds ERC-20 tokens.
9. Because the `Address` whitelist also blocks `submit`, the non-whitelisted address cannot submit any EVM transaction to move those tokens — they are frozen. [1](#0-0) [7](#0-6) [8](#0-7)

pub fn ft_on_transfer<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();
        let mut engine: Engine<_, _> = Engine::new(
            predecessor_address(&predecessor_account_id),
            current_account_id.clone(),
            io,
            env,
        )?;

        sdk::log!("Call ft_on_transfer");

        let args: FtOnTransferArgs = read_json_args(&io)?;
        let result = if predecessor_account_id == get_connector_account_id(&io)? {
            engine.receive_base_tokens(&args)
        } else {
            engine.receive_erc20_tokens(
                &predecessor_account_id,
                &args,
                &current_account_id,
                handler,
            )
        };

        #[allow(clippy::used_underscore_binding)]
        let amount_to_return = if let Err(_err) = &result {
            sdk::log!("Error in ft_on_transfer: {_err:?}");
            // An error occurred, so we need to return the amount of tokens to the sender.
            args.amount.as_u128()
        } else {
            // Everything is ok, so return 0.
            0
        };

        let output = crate::prelude::format!("\"{amount_to_return}\"");
        io.return_output(output.as_bytes());

        // In case of an error, we just return Ok(None) to avoid a panic in the contract. It's ok
        // because in case of an error, we already returned the amount of tokens to the sender.
        Ok(result.unwrap_or(None))
    })
}
