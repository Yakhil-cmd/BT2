### Title
Pool admin can set uncapped per-bin additional fees via `setBinAdditionalFees`, bypassing `maxAdminSpreadFeeE6` — (`metric-core/contracts/MetricOmmPool.sol` / `MetricOmmPoolFactory.sol`)

### Summary
`MetricOmmPoolFactory.setPoolBinAdditionalFees` forwards `addFeeBuyE6` / `addFeeSellE6` to the pool without any cap check, while the analogous global-fee setter `setPoolAdminFees` enforces `maxAdminSpreadFeeE6`. A pool admin can therefore set `addFeeBuyE6 = 65535` (6.5535 %) on any bin, causing traders who swap through that bin to pay up to 6.5535 % more than the factory-capped fee, with no timelock and no factory guard.

### Finding Description
`MetricOmmPoolFactory.setPoolBinAdditionalFees` (lines 450–457) passes `addFeeBuyE6` and `addFeeSellE6` directly to `IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees` with only an `onlyPoolAdmin` role check. No comparison against `maxAdminSpreadFeeE6` or any other factory cap is performed. The pool-side `setBinAdditionalFees` (lines 464–474) likewise performs no cap validation. By contrast, `setPoolAdminFees` (lines 414–415) explicitly reverts with `AdminFeeTooHigh` when the requested fee exceeds `maxAdminSpreadFeeE6`.

In `_swapToken1ForToken0SpecifiedOutput` (line 910), the per-bin fee is composed as:
```
currBinBuyFeeX64 = params.baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6)
```
and then applied in `buyToken0InBinSpecifiedOut` (line 409):
```
feeAmountScaled = Math.ceilDiv(amountInScaled * currBinBuyFeeX64, ONE_X64)
```
With `addFeeBuyE6 = 65535`, the per-bin component alone is 6.5535 %, stacked on top of the global spread fee.

### Impact Explanation
Every trader whose swap routes through the configured bin pays up to 6.5535 % more token-1 input than the oracle-fair value for the specified token-0 output. This is a direct, quantifiable loss of user principal above Sherlock Medium thresholds. The excess fee accrues to LPs in the bin's token-1 balance. The pool admin can apply this immediately, to any bin, without a timelock.

### Likelihood Explanation
The pool admin is a semi-trusted role that is expected to operate only within factory-configured caps. The missing cap check is a single-transaction action requiring only the pool admin key. Any pool whose admin is compromised, incentivised, or simply misconfigured can trigger this path.

### Recommendation
Add a cap check in `setPoolBinAdditionalFees` analogous to the one in `setPoolAdminFees`:
```solidity
if (addFeeBuyE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
```
Alternatively, introduce a dedicated `maxBinAdditionalFeeE6` factory parameter and enforce it in both the factory gateway and the pool-side `setBinAdditionalFees`.

### Proof of Concept
1. Pool admin calls `factory.setPoolBinAdditionalFees(pool, 0, 65535, 0)` — succeeds with no revert.
2. Trader calls `swap` with `amountSpecified < 0, zeroForOne = false` (exact-output, buying token0).
3. The swap routes through bin 0; `currBinBuyFeeX64 = baseFeeX64 + Math.mulDiv(65535, ONE_X64, 1e6)`.
4. `feeAmountScaled = Math.ceilDiv(amountInScaled * currBinBuyFeeX64, ONE_X64)` charges 6.5535 % on top of the base fee.
5. Assert that `amountIn` exceeds the oracle-fair value by ≈ 6.5535 % — the uncapped per-bin fee.
