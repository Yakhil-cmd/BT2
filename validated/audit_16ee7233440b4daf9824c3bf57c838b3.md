### Title
Gateway Stateful Admission Uses Stale Previous-Block Gas Prices, Admitting Transactions That Fail at Execution - (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The gateway's stateful validator checks a transaction's resource bounds against the **previous block's** gas prices rather than the next block's gas prices that will govern actual execution. Both the explicit `validate_resource_bounds` admission check and the blockifier's `check_fee_bounds` called inside `run_validate_entry_point` use the same stale `BlockInfo` from the previous block. When the next block's gas prices are higher, transactions that passed gateway admission will fail at execution — the gateway admits transactions that are invalid at execution time.

### Finding Description

In `StatefulTransactionValidator::validate_resource_bounds`, the L2 gas price threshold is computed from the previous block:

```rust
// TODO(Arni): getnext_l2_gas_price from the block header.
let previous_block_l2_gas_price = self
    .gateway_fixed_block_state_reader
    .get_block_info()
    .await?
    .gas_prices
    .strk_gas_prices
    .l2_gas_price;
self.validate_tx_l2_gas_price_within_threshold(
    executable_tx.resource_bounds(),
    previous_block_l2_gas_price,
)?;
``` [1](#0-0) 

The embedded TODO explicitly acknowledges the wrong value is being used.

In `run_validate_entry_point`, the `BlockInfo` used to build the `BlockContext` for blockifier validation is also the previous block's info — only the block number is incremented, while all gas prices remain from block N:

```rust
let mut block_info = self.gateway_fixed_block_state_reader.get_block_info().await?;
block_info.block_number = block_info.block_number.unchecked_next();
let block_context = BlockContext::new(block_info, ...);
``` [2](#0-1) 

This `block_context` is passed to `StatefulValidator::validate`, which calls `perform_pre_validation_stage` → `check_fee_bounds`. Inside `check_fee_bounds`, the actual block gas prices from `block_context.block_info.gas_prices` are compared against the transaction's `max_price_per_unit` for L1, L2, and L1 data gas: [3](#0-2) 

All three price comparisons use the stale previous-block prices. When the batcher later executes the transaction, it uses the **actual** next block's gas prices (set by the consensus/batcher pipeline), which can be higher. The `check_fee_bounds` call during actual execution will then reject the transaction that the gateway admitted.

The `GatewayFixedBlockSyncStateClient` caches the block info in a `OnceCell`, so the same stale value is reused for both the admission check and the blockifier validation within a single validator instance: [4](#0-3) 

### Impact Explanation

Any transaction admitted to the mempool with `max_price_per_unit` values equal to the previous block's gas prices will fail at execution if the next block's gas prices are higher. This means:

1. **Invalid transactions are admitted**: The gateway's admission decision is wrong — it accepts transactions that will be rejected by the blockifier at execution time.
2. **Mempool pollution**: Such transactions occupy mempool slots and consume sequencer resources (batcher execution attempts) before being discarded.
3. **Systematic failure during gas price increases**: During any sustained gas price increase (e.g., network congestion), a large fraction of admitted transactions will fail at execution, degrading throughput.

The `min_gas_price_percentage` config provides a partial buffer (e.g., if set to 90%, the transaction's price must be ≥ 90% of the previous block's price), but it does not prevent the mismatch — it only reduces the magnitude of the discrepancy that causes failures.

### Likelihood Explanation

L2 gas prices on Starknet are dynamic and change block-to-block via the `compute_fee_proposal` mechanism. Any block-to-block price increase beyond the `min_gas_price_percentage` buffer causes this failure mode. This is a normal operating condition, not an edge case. An attacker can also deliberately submit transactions priced at exactly the previous block's price to fill the mempool with transactions that will fail at execution.

### Recommendation

Replace the previous-block gas price with the **next block's** projected gas price in both `validate_resource_bounds` and the `BlockContext` constructed in `run_validate_entry_point`. The TODO comment at line 229 already identifies this gap. The next block's L2 gas price should be obtained from the block header being built (e.g., from the batcher's proposed `block_info`), or at minimum the `BlockContext` passed to blockifier validation should use the same gas prices that the batcher will use for execution.

### Proof of Concept

1. Previous block N has `strk_gas_prices.l2_gas_price = 100`.
2. User submits an invoke transaction with `l2_gas.max_price_per_unit = 100`.
3. Gateway `validate_resource_bounds`: `100 >= min_gas_price_percentage% × 100` → **passes** (for any `min_gas_price_percentage ≤ 100`).
4. Gateway `run_validate_entry_point` builds `BlockContext` with block N's gas prices (block number set to N+1). Blockifier `check_fee_bounds`: `100 >= 100` → **passes**.
5. Transaction is admitted to the mempool.
6. Batcher builds block N+1 with `l2_gas_price = 110` (gas price increased by 10%).
7. Batcher calls `perform_pre_validation_stage` → `check_fee_bounds` with the real block N+1 context: `100 >= 110` → **fails** with `ResourceBoundsError::MaxGasPriceTooLow`.
8. Transaction is rejected at execution despite having passed gateway admission — the gateway admitted an invalid transaction. [5](#0-4) [6](#0-5) [7](#0-6)

    // TODO(Arni): Consider running this validation for all gas prices.
    fn validate_tx_l2_gas_price_within_threshold(
        &self,
        tx_resource_bounds: ValidResourceBounds,
        previous_block_l2_gas_price: NonzeroGasPrice,
    ) -> StatefulTransactionValidatorResult<()> {
        match tx_resource_bounds {
            ValidResourceBounds::AllResources(tx_resource_bounds) => {
                let tx_l2_gas_price = tx_resource_bounds.l2_gas.max_price_per_unit;
                let gas_price_threshold_multiplier =
                    Ratio::new(self.config.min_gas_price_percentage.into(), 100_u128);
                let threshold = (gas_price_threshold_multiplier
                    * previous_block_l2_gas_price.get().0)
                    .to_integer();
                if tx_l2_gas_price.0 < threshold {
                    return Err(StarknetError {
                        // We didn't have this kind of an error.
                        code: StarknetErrorCode::UnknownErrorCode(
                            "StarknetErrorCode.GAS_PRICE_TOO_LOW".to_string(),
                        ),
                        message: format!(
                            "Transaction L2 gas price {tx_l2_gas_price} is below the required \
                             threshold {threshold}.",
                        ),
                    });
                }
            }
            ValidResourceBounds::L1Gas(_) => {
                // No validation required for legacy transactions.
            }
        }
        Ok(())
    }

    // Performs static checks before executing validation entry point.

    pub fn perform_pre_validation_stage<S: State + StateReader>(
        &self,
        state: &mut S,
        tx_context: &TransactionContext,
    ) -> TransactionPreValidationResult<()> {
        let tx_info = &tx_context.tx_info;
        Self::handle_nonce(state, tx_info, self.execution_flags.strict_nonce_check)?;

        if self.execution_flags.charge_fee {
            self.check_fee_bounds(tx_context)?;

            verify_can_pay_committed_bounds(state, tx_context).map_err(Box::new)?;
        }

        self.validate_proof_facts(&tx_context.block_context, state)?;

        Ok(())
    }
