### Title
Missing Lower-Bound Threshold Check in `assert_sign_inputs` and `presign` — (File: `src/frost/mod.rs`)

---

### Summary

`assert_key_invariants` in `src/dkg.rs` enforces both an upper bound (`threshold <= participants.len()`) and a lower bound (`threshold >= 2`) on the threshold parameter. The analogous validation functions `assert_sign_inputs` and `presign` in `src/frost/mod.rs` enforce only the upper bound, omitting the lower-bound check entirely. A caller can pass `threshold = 1` (or `0`) to these functions, which will pass all validation and allow the signing protocol to proceed with a threshold that is inconsistent with the one used during key generation, corrupting the signing output.

---

### Finding Description

`assert_key_invariants` in `src/dkg.rs` enforces both bounds:

```rust
// src/dkg.rs lines 573–582
if threshold > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
if threshold < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
}
``` [1](#0-0) 

`assert_sign_inputs` in `src/frost/mod.rs` only checks the upper bound:

```rust
// src/frost/mod.rs lines 144–150
// validate threshold
if threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
// ← no check for threshold < 2
``` [2](#0-1) 

`presign` in the same file has the identical gap:

```rust
// src/frost/mod.rs lines 71–77
if args.threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
// ← no check for args.threshold < 2
``` [3](#0-2) 

`ReconstructionLowerBound` is a plain `usize` newtype with no internal invariant, so `ReconstructionLowerBound::from(1usize)` or `from(0usize)` are both constructible by any caller. [4](#0-3) 

The OT-based ECDSA `sign` function in `src/ecdsa/ot_based_ecdsa/sign.rs` has the same omission — it checks `participants.len() < threshold` but never `threshold < 2`: [5](#0-4) 

---

### Impact Explanation

**High — Corruption of signing outputs so honest parties accept unusable cryptographic outputs.**

DKG always produces key shares under `threshold >= 2` (enforced by `assert_key_invariants`). If a caller invokes `assert_sign_inputs` or `presign` with `threshold = 1`, validation succeeds and the signing session proceeds. The FROST signing protocol then performs Lagrange interpolation using a threshold of 1 over a participant set whose shares were split under threshold 2. The resulting signature share combination is cryptographically incorrect relative to the public key, producing a signature that fails external verification. Honest participants who complete the protocol receive an unusable output with no protocol-level error, because the inconsistency is not detected inside the signing round itself.

---

### Likelihood Explanation

**Medium.** This is a library with a public API. Any application-layer caller — including a malicious coordinator who controls the session parameters — can supply `threshold = 1` to `assert_sign_inputs` or `presign`. No special privilege is required beyond being a valid library caller. The `ReconstructionLowerBound` type imposes no compile-time or runtime minimum, so the invalid value passes silently through the entire validation path.

---

### Recommendation

Add the same lower-bound guard present in `assert_key_invariants` to both `assert_sign_inputs` and `presign` in `src/frost/mod.rs`, and to `sign` in `src/ecdsa/ot_based_ecdsa/sign.rs`:

```rust
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

This mirrors the existing check at `src/dkg.rs` lines 580–582 and closes the inconsistency between the keygen and signing validation paths.

---

### Proof of Concept

1. Generate a key with `threshold = 2` and `participants = [A, B]` via `keygen` — this succeeds because `assert_key_invariants` enforces `threshold >= 2`.
2. Call `assert_sign_inputs(&[A, B], 1usize, A, B)` — this returns `Ok(participants)` because the lower-bound check is absent.
3. Call `presign(&[A, B], A, PresignArguments { threshold: 1.into(), keygen_out })` — this also returns `Ok(protocol)` for the same reason.
4. Run the signing protocol to completion. The Lagrange interpolation inside the signing round uses `threshold = 1` against shares that were split under `threshold = 2`, producing a signature whose `s` component does not satisfy the ECDSA/EdDSA verification equation for the public key.
5. The protocol completes without a protocol-level error; honest parties receive an unusable signature output.
