### Title
WalletConnect `chia_createOfferForIds` Trusts Attacker-Controlled `driver_dict` Royalty Percentage for Confirmation Display — (`packages/gui/src/electron/commands/parseCommandDisplay.ts`)

---

### Summary

When a WalletConnect dapp sends `chia_createOfferForIds`, the GUI's confirmation dialog computes and displays the `amountWithRoyalties` (total XCH the user will spend including creator royalties) using the `royalty_percentage` value read directly from the attacker-controlled `driver_dict` parameter. A fallback `nftGetInfo` call is made to override this value with on-chain data, but it is explicitly treated as "best effort" and silently swallowed on failure. When it fails, the attacker's royalty value drives the displayed total, causing the user to approve an offer under a false accounting of what the seller will actually receive.

---

### Finding Description

In `parseCommandDisplay.ts`, the `chia_wallet.create_offer_for_ids` branch calls `createOfferRoyaltyPercentages()`, which reads `royalty_percentage` directly from the caller-supplied `driver_dict`: [1](#0-0) 

This value is then passed as the initial `royaltyPercentage` for each NFT line item in `parseWalletDeltaItem()`: [2](#0-1) 

The code attempts to override this with the on-chain value from `nftGetInfo`, but the entire block is wrapped in a `try/catch` that silently discards all errors, and the override is additionally gated on `nftInfo.nft_info.data_uris` being present. The comment explicitly acknowledges this is "best effort":

```typescript
} catch {
  // NFT type has already been resolved from offer data; metadata is best effort.
}
```

The resulting `royaltyPercentage` on each NFT line is then consumed by `royaltyPercentagesForSide()` and `formatAmountWithRoyalties()` to compute the `amountWithRoyalties` field shown to the user: [3](#0-2) 

The `chia_createOfferForIds` dapp command is registered and exposed to WalletConnect dapps: [4](#0-3) 

The `driver_dict` is passed through without any pre-display validation against on-chain NFT state: [5](#0-4) 

The project's own test suite explicitly exercises the failure path — when `nftGetInfo` returns `success: false`, the `driver_dict` value is used verbatim for the royalty display: [6](#0-5) 

---

### Impact Explanation

A malicious WalletConnect dapp sends `chia_createOfferForIds` with a `driver_dict` containing `royalty_percentage: '0'` for an NFT whose on-chain royalty is, say, 10%. If `nftGetInfo` fails (wallet not fully synced, NFT newly minted, transient RPC error), the confirmation dialog shows `amountWithRoyalties` equal to the base XCH amount with no royalty added. The user approves believing the seller will receive the full offered amount. When the offer is accepted on-chain, the NFT puzzle enforces the real 10% royalty, so the seller receives only 90% of the locked XCH. The user has approved an offer whose actual economic outcome (seller net proceeds) differs materially from what was displayed at approval time. This satisfies the High impact criterion: **unsafe trust of WalletConnect state causes a user to approve the wrong amount**.

---

### Likelihood Explanation

Any WalletConnect-connected dapp can send `chia_createOfferForIds` with an arbitrary `driver_dict`. The `nftGetInfo` failure condition is reachable without attacker control (wallet sync lag, newly minted NFT, RPC timeout), and the code is explicitly designed to proceed without it. No privilege escalation or key compromise is required.

---

### Recommendation

Before rendering the confirmation dialog for `create_offer_for_ids`, cross-validate the `royalty_percentage` from `driver_dict` against the value returned by `nftGetInfo`. If `nftGetInfo` succeeds and the values differ, either reject the request with an error or display a prominent warning. Do not silently fall back to the caller-supplied value when on-chain data is available. The "best effort" comment should not apply to security-critical accounting fields shown in approval dialogs.

---

### Proof of Concept

1. Connect a malicious dapp via WalletConnect to a Chia GUI wallet.
2. Send `chia_createOfferForIds` with:
   ```json
   {
     "offer": { "1": "-1000000000000", "<nft_launcher_id>": "1" },
     "driver_dict": {
       "<nft_launcher_id>": {
         "type": "singleton",
         "launcher_id": "0x<nft_launcher_id>",
         "also": {
           "type": "metadata",
           "also": {
             "type": "ownership",
             "transfer_program": {
               "type": "royalty transfer program",
               "royalty_percentage": "0"
             }
           }
         }
       }
     }
   }
   ```
   where the real on-chain NFT has `royalty_percentage = 1000` (10%).
3. Trigger a condition where `nftGetInfo` fails (e.g., wallet mid-sync).
4. Observe the confirmation dialog shows `amountWithRoyalties = 1 XCH` (no royalty) instead of `1.1 XCH`.
5. User approves; offer is created. When accepted, the seller receives only 0.9 XCH; 0.1 XCH goes to the creator — the seller net proceeds shown at approval time were wrong.

function createOfferRoyaltyPercentages(
  walletDelta: WalletDelta,
  driverDict: Record<string, unknown>,
): AssetRoyaltyPercentages {
  const royaltyPercentages: AssetRoyaltyPercentages = {
    spending: {},
    receiving: {},
  };

  for (const assetId of Object.keys(walletDelta.spending)) {
    royaltyPercentages.spending[assetId] = royaltyPercentageForDriverInfo(
      driverDict[assetId] ?? driverDict[`0x${assetId}`],
    );
  }

  for (const assetId of Object.keys(walletDelta.receiving)) {
    royaltyPercentages.receiving[assetId] = royaltyPercentageForDriverInfo(
      driverDict[assetId] ?? driverDict[`0x${assetId}`],
    );
  }

  return royaltyPercentages;
}

function royaltyPercentagesForSide(lines: DisplayWalletDeltaItem[]): number[] {
  return lines
    .filter((line): line is Extract<DisplayWalletDeltaItem, { kind: 'nft' }> => line.kind === 'nft')
    .map((line) => line.royaltyPercentage)
    .filter(
      (royaltyPercentage): royaltyPercentage is number => royaltyPercentage !== undefined && royaltyPercentage > 0,
    );
}

function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],
): string | undefined {
  if (royaltyPercentages.length === 0 || line.kind === 'nft') {
    return undefined;
  }

  const splitAmount = amount / BigInt(royaltyPercentages.length);
  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;

  if (line.kind === 'xch') {
    return mojoToChiaLocaleString(totalAmount);
  }

  return mojoToCATLocaleString(totalAmount);
}
