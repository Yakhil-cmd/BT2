### Title
Commission Increase via Active Commitment Applies Retroactively to Unsettled Epoch Rewards — (File: src/staking/staking.cairo)

---

### Summary

`set_commission` in `src/staking/staking.cairo` updates the commission rate immediately in storage without first settling rewards at the old rate. When a staker holds an active `commission_commitment`, they can **increase** their commission. Because reward calculation reads the commission from storage at attestation time and applies it to the entire unsettled epoch, the new higher commission is applied retroactively to rewards that accrued under the old lower rate, stealing unclaimed yield from delegators.

---

### Finding Description

`set_commission` (line 725) delegates to `update_commission` (line 1573). When an active `commission_commitment` exists, `update_commission` permits setting commission to any value up to `max_commission` — including values **higher** than the current commission:

```cairo
if self.is_commission_commitment_active(:commission_commitment) {
    assert!(commission <= commission_commitment.max_commission, ...);
    assert!(commission != old_commission, ...);
``` [1](#0-0) 

Neither `set_commission` nor `update_commission` triggers any reward settlement before writing the new commission to storage:

```cairo
// Update commission in storage.
staker_pool_info.commission.write(Option::Some(commission));
``` [2](#0-1) 

Rewards are calculated lazily — only when `update_rewards_from_attestation_contract` is called during attestation (once per epoch in V2). At that point, `calculate_staker_pools_rewards` reads the **current** commission from storage and applies it to the entire epoch's pool rewards:

```cairo
let commission = staker_pool_info.commission();
...
let (commission_rewards, pool_rewards) = split_rewards_with_commission(
    rewards_including_commission: pool_rewards_including_commission, :commission,
);
``` [3](#0-2) 

The attestation flow confirms that `update_rewards_from_attestation_contract` is called once per epoch, triggered by the staker's operational address: [4](#0-3) 

The codebase itself acknowledges this gap with a comment directly above `set_commission_commitment`:

```
/// **Note**: Current commission increase safeguards still allow for sudden commission changes.
``` [5](#0-4) 

---

### Impact Explanation

A staker can retroactively extract a higher commission from rewards that accrued during the epoch at the old (lower) rate. In the extreme case, a staker with `max_commission = 10000` (100%) can set commission to 100% immediately before attesting, causing delegators to receive **zero** pool rewards for the entire epoch. This is a direct theft of unclaimed yield from delegators.

Matches allowed impact: **High — Theft of unclaimed yield**.

---

### Likelihood Explanation

The attack requires the staker to have previously called `set_commission_commitment` with a high `max_commission`. This is a normal, permissionless operation available to any staker with a pool. The staker controls both the commission change (via their staker address) and the attestation timing (via their operational address). There is no cooldown, no delay, and no on-chain enforcement preventing the staker from increasing commission in the same transaction block immediately before attesting. The only prerequisite is having an active commitment, which the staker can set up at any time.

---

### Recommendation

Before updating the commission in `update_commission`, first settle the current epoch's rewards at the old commission rate. This mirrors the fix described in the external report: checkpoint the current state before changing the rate. Concretely, `set_commission` should invoke the reward update logic (equivalent to `update_rewards_from_attestation_contract`) for the staker before writing the new commission to storage, so that any rewards accrued under the old rate are locked in at that rate.

---

### Proof of Concept

1. Staker stakes and opens a STRK delegation pool with `commission = 200` (2%).
2. Delegator delegates a large amount to the pool.
3. K epochs pass; rewards accrue for the current epoch at 2% commission.
4. Staker calls `set_commission_commitment(max_commission=10000, expiration_epoch=current+1)`. [6](#0-5) 
5. Staker calls `set_commission(commission=10000)` — succeeds because commitment is active and `10000 <= max_commission`. [7](#0-6) 
6. Staker's operational address calls `attest(...)` → `update_rewards_from_attestation_contract` is triggered → `calculate_staker_pools_rewards` reads `commission = 10000` and applies it to the full epoch's pool rewards. [8](#0-7) 
7. Delegators receive 0 pool rewards for the epoch; the staker captures 100% as commission — despite the epoch having accrued under 2% commission.

        fn set_commission(ref self: ContractState, commission: Commission) {
            // Prerequisites and asserts.
            self.general_prerequisites();
            assert!(commission <= COMMISSION_DENOMINATOR, "{}", Error::COMMISSION_OUT_OF_RANGE);
            let staker_address = get_caller_address();
            let staker_info = self.internal_staker_info(:staker_address);
            assert!(staker_info.unstake_time.is_none(), "{}", Error::UNSTAKE_IN_PROGRESS);

            let staker_pool_info = self.staker_pool_info.entry(staker_address);
            if let Option::Some(old_commission) = staker_pool_info.commission.read() {
                self
                    .update_commission(
                        :staker_address, :staker_pool_info, :old_commission, :commission,
                    );
            } else {
                staker_pool_info.commission.write(Option::Some(commission));
                self.emit(Events::CommissionInitialized { staker_address, commission });
            }
        }

        fn set_commission_commitment(
            ref self: ContractState, max_commission: Commission, expiration_epoch: Epoch,
        ) {
            self.general_prerequisites();
            assert!(max_commission <= COMMISSION_DENOMINATOR, "{}", Error::COMMISSION_OUT_OF_RANGE);
            let staker_address = get_caller_address();
            let staker_info = self.internal_staker_info(:staker_address);
            assert!(staker_info.unstake_time.is_none(), "{}", Error::UNSTAKE_IN_PROGRESS);
            let staker_pool_info = self.staker_pool_info.entry(staker_address);
            assert!(staker_pool_info.has_pool(), "{}", Error::MISSING_POOL_CONTRACT);
            let current_epoch = self.get_current_epoch();
            if let Option::Some(commission_commitment) = staker_pool_info
                .commission_commitment
                .read() {
                assert!(
                    !self.is_commission_commitment_active(:commission_commitment),
                    "{}",
                    Error::COMMISSION_COMMITMENT_EXISTS,
                );
            }
            // Staker must have a commission since it has a pool.
            let current_commission = staker_pool_info.commission();
            assert!(current_commission <= max_commission, "{}", Error::MAX_COMMISSION_TOO_LOW);
            assert!(expiration_epoch > current_epoch, "{}", Error::EXPIRATION_EPOCH_TOO_EARLY);
            assert!(
                expiration_epoch - current_epoch <= self.get_epoch_info().epochs_in_year(),
                "{}",
                Error::EXPIRATION_EPOCH_TOO_FAR,
            );
            let commission_commitment = CommissionCommitment { max_commission, expiration_epoch };
            staker_pool_info.commission_commitment.write(Option::Some(commission_commitment));
