### Title
Unbounded Loop in `calculate_rewards()` Enables Permanent Freezing of Unclaimed Yield - (File: src/pool/pool.cairo)

### Summary
The `calculate_rewards()` function in `pool.cairo` contains an explicitly acknowledged unbounded loop that iterates over a pool member's entire balance-change trace since their last reward claim. A pool member who makes many balance changes across many epochs without claiming rewards will accumulate a trace large enough to cause the `claim_rewards()` transaction to exceed Starknet's gas limit, permanently freezing their unclaimed yield with no recovery path.

### Finding Description

**Root cause**: `calculate_rewards()` at `src/pool/pool.cairo:859` iterates over every entry in `pool_member_epoch_balance` from `entry_to_claim_from` to `pool_member_trace_length`. The code itself acknowledges this:

```
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
``` [1](#0-0) 

**How the trace grows**: Every call to `set_member_balance()` inserts a new checkpoint at `current_epoch + K` into `pool_member_epoch_balance`. The trace deduplicates only if the same epoch key is reused (i.e., two changes in the same epoch window), but each distinct epoch produces a new entry. The following public pool functions all call `set_member_balance()` or `increase_member_balance()`:

- `enter_delegation_pool` → `set_member_balance`
- `add_to_delegation_pool` → `increase_member_balance` → `set_member_balance`
- `exit_delegation_pool_intent` → `set_member_balance`
- `enter_delegation_pool_from_staking_contract` (pool switch) → `set_member_balance` [2](#0-1) 

**Why the pointer does not advance**: `entry_to_claim_from` is stored in `pool_member_info` and is only updated upon a *successful* `claim_rewards()` call:

```cairo
pool_member_info.entry_to_claim_from = updated_entry_to_claim_from;
``` [3](#0-2) 

If `claim_rewards()` reverts due to gas exhaustion, `entry_to_claim_from` is never advanced. There is no partial-claim mechanism. The pool member is permanently unable to claim rewards.

**Exploit path**:
1. Pool member calls `enter_delegation_pool` to join.
2. Pool member repeatedly calls `add_to_delegation_pool` and/or `exit_delegation_pool_intent` across many different epochs, each time producing a new trace entry.
3. Pool member never calls `claim_rewards` during this period.
4. After enough epochs, `claim_rewards()` exceeds the Starknet gas limit inside the `while` loop and reverts.
5. Since `entry_to_claim_from` is never advanced, every subsequent `claim_rewards()` call also reverts. Unclaimed yield is permanently frozen.

### Impact Explanation

**High — Permanent freezing of unclaimed yield.**

The pool member's principal is recoverable via `exit_delegation_pool_action` (which does not call `calculate_rewards`), but all accumulated rewards computed from the balance trace are permanently inaccessible. There is no administrative override or partial-claim path to rescue the frozen rewards. [4](#0-3) 

### Likelihood Explanation

**Medium-High.** Any unprivileged pool member can trigger this on themselves through normal protocol usage: delegating, adjusting amounts, and switching pools across many epochs without claiming. Long-term active delegators who frequently adjust their delegation (e.g., to optimize staking power) are the most realistic victims. The protocol has no minimum claim frequency requirement, so this is a reachable steady-state for legitimate users.

### Recommendation

1. **Bound the loop**: Introduce a `MAX_BALANCE_TRACE_ITERATIONS` constant and process at most that many entries per `claim_rewards()` call, persisting `entry_to_claim_from` after each partial run. This mirrors the `MAX_ROYALTY_RECIPIENTS_INDEX` cap recommended in the external report.
2. **Alternatively, enforce periodic claiming**: Require `claim_rewards()` to be called before any balance-changing operation, ensuring `entry_to_claim_from` stays close to the trace head and the loop is always short.
3. **Emit a warning or enforce a cap** on the number of unclaimed trace entries before allowing further balance changes.

### Proof of Concept

1. Deploy the pool contract and have a pool member enter with a small amount.
2. In a loop over N epochs (e.g., N = 10,000), call `add_to_delegation_pool(1)` then `exit_delegation_pool_intent(1)` each epoch, producing 2 new trace entries per epoch.
3. After N epochs, call `claim_rewards()`. The `calculate_rewards` loop must iterate over ~2N entries, each requiring a storage read via `pool_member_trace.at(entry_to_claim_from)` and a `find_sigma` call.
4. Observe that the transaction reverts with an out-of-gas error.
5. Observe that all subsequent `claim_rewards()` calls also revert, permanently freezing the accumulated yield. [5](#0-4)

        fn calculate_rewards(
            self: @ContractState,
            pool_member: ContractAddress,
            from_checkpoint: PoolMemberCheckpoint,
            until_checkpoint: PoolMemberCheckpoint,
            mut entry_to_claim_from: VecIndex,
        ) -> (Amount, VecIndex) {
            let pool_member_trace = self.pool_member_epoch_balance.entry(pool_member);
            // Note: `until_epoch` is the current epoch.
            let until_epoch = until_checkpoint.epoch();

            let mut rewards = 0;

            let pool_member_trace_length = pool_member_trace.length();

            let mut from_sigma = self.find_sigma(from_checkpoint, curr_epoch: until_epoch);
            let mut from_balance = from_checkpoint.balance();

            let base_value = self.staking_rewards_base_value.read();

            // **Note**: The loop iterates over the balance changes in the pool member's balance
            // trace. This loop is unbounded but unlikely to exceed gas limits.
            while entry_to_claim_from < pool_member_trace_length {
                let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
                // If the balance change is after `until_epoch` (and therefore does not affect
                // the current reward computation), exit the loop.
                if pool_member_checkpoint.epoch() >= until_epoch {
                    break;
                }

                // Compute rewards from (inclusive) the previous balance change (or from
                // `from_checkpoint`) to (exclusive) the current entry.
                let to_sigma = self.find_sigma(pool_member_checkpoint, curr_epoch: until_epoch);
                rewards +=
                    compute_rewards_rounded_down(
                        amount: from_balance, interest: to_sigma - from_sigma, :base_value,
                    );
                from_sigma = to_sigma;
                from_balance = pool_member_checkpoint.balance();
                entry_to_claim_from += 1;
            }

            // Compute the remaining rewards from (inclusive) the last visited balance change in
            // `pool_member_trace` (or from `from_checkpoint`) to (exclusive) `until_checkpoint`.
            let to_sigma = self.find_sigma(until_checkpoint, curr_epoch: until_epoch);
            rewards +=
                compute_rewards_rounded_down(
                    amount: from_balance, interest: to_sigma - from_sigma, :base_value,
                );

            (rewards, entry_to_claim_from)
        }
