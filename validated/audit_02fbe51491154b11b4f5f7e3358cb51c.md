### Title
Blacklisted `adminFeeDestination` permanently blocks `collectFees`, `setPoolAdminFees`, and `setPoolProtocolFee` for USDC/USDT pools — (`metric-core/contracts/MetricOmmPool.sol`)

---

### Summary

`MetricOmmPool.collectFees` transfers tokens directly to `adminFeeDestination_` via `safeTransfer`. If the pool's TOKEN0 or TOKEN1 is USDC or USDT and `adminFeeDestination_` is blacklisted by that token, every call to `collectFees` reverts. Because `setPoolAdminFees` and `setPoolProtocolFee` in the factory both call `collectFees` as a mandatory first step, those fee-management operations are also blocked for the duration of the blacklist.

---

### Finding Description

`MetricOmmPool.collectFees` performs unconditional `safeTransfer` calls to two hard-coded recipients: [1](#0-0) 

```solidity
if (totalFee0ToAdmin > 0) {
    transferToken0(adminFeeDestination_, totalFee0ToAdmin);   // ← reverts if blacklisted
}
if (totalFee1ToAdmin > 0) {
    transferToken1(adminFeeDestination_, totalFee1ToAdmin);   // ← reverts if blacklisted
}
if (totalFee0ToProtocol > 0) {
    transferToken0(FACTORY, totalFee0ToProtocol);
}
if (totalFee1ToProtocol > 0) {
    transferToken1(FACTORY, totalFee1ToProtocol);
}
```

`transferToken0` / `transferToken1` are thin wrappers around `IERC20.safeTransfer`: [2](#0-1) 

USDC and USDT maintain on-chain blacklists; a `safeTransfer` to a blacklisted address reverts the entire transaction.

`collectFees` is invoked in three factory paths, all of which become permanently blocked:

| Factory function | Caller | Effect when blocked |
|---|---|---|
| `collectPoolFees` | Anyone (permissionless) | Fee collection DoS |
| `setPoolAdminFees` | Pool admin | Cannot change admin fee rates |
| `setPoolProtocolFee` | Factory owner | Cannot change protocol fee rates | [3](#0-2) [4](#0-3) [5](#0-4) 

The `notionalFeeToken0Scaled` / `notionalFeeToken1Scaled` accumulators are only zeroed inside `collectFees`: [6](#0-5) 

While the destination is blacklisted, these accumulators keep growing but can never be flushed, so all accrued protocol and admin fees remain locked in the pool.

---

### Impact Explanation

- **Protocol fees stuck**: All spread-surplus and notional fees accrued since the last successful collection are locked in the pool contract. Neither the protocol owner nor the pool admin can extract them.
- **Fee-rate management blocked**: `setPoolAdminFees` and `setPoolProtocolFee` both require a successful `collectFees` call before they update rates. If the pool needs an emergency fee reduction (e.g., during a market event), neither the admin nor the protocol owner can execute it.
- **Permissionless collection DoS**: `collectPoolFees` is callable by anyone and is the expected routine collection path; it is fully blocked.

Swaps and liquidity operations are unaffected. The DoS is recoverable: the pool admin can call `setPoolAdminFeeDestination` (which does **not** invoke `collectFees`) to rotate to a non-blacklisted address, after which collection resumes. However, until that remediation is executed, all accrued fees are inaccessible and fee governance is frozen.

---

### Likelihood Explanation

- USDC and USDT are the most common stablecoins used in AMM pools; pools pairing either token against ETH/WBTC/etc. are the primary deployment target.
- `adminFeeDestination` is a mutable address set by the pool admin. It can be a multisig, DAO treasury, or any EOA — all of which are realistic USDC/USDT blacklist targets (e.g., OFAC-sanctioned addresses).
- No attacker action is required; the blacklisting is performed by the token issuer and is outside the protocol's control.

---

### Recommendation

Wrap the admin-destination transfer in a try/catch (or use a low-level call) and, on failure, credit the owed amount to an internal escrow mapping keyed by `adminFeeDestination_`. Add a separate `claimAdminFees(address pool)` function that lets the destination (or the admin) pull the escrowed amount once the blacklist is lifted or after the destination is rotated. The protocol-side transfer to `FACTORY` is a contract address under the protocol's control and is lower risk, but the same pattern can be applied for defense-in-depth.

---

### Proof of Concept

1. Deploy a pool with TOKEN0 = USDC, TOKEN1 = WETH.
2. Set `adminFeeDestination` to address `A` (e.g., a multisig).
3. USDC issuer blacklists address `A`.
4. Swaps occur; `notionalFeeToken0Scaled` accumulates USDC-denominated fees.
5. Call `collectPoolFees(pool)` → `collectFees` is called → `transferToken0(A, totalFee0ToAdmin)` → `USDC.safeTransfer(A, ...)` → **reverts** (blacklisted).
6. Call `setPoolAdminFees(pool, newFee, newFee)` → same revert path.
7. Call `setPoolProtocolFee(pool, newFee, newFee)` → same revert path.
8. All accrued USDC fees remain locked; fee rates cannot be updated until the admin separately calls `setPoolAdminFeeDestination(pool, nonBlacklistedAddress)` and then retries collection.

    override
    onlyOwner
    nonReentrant
  {
    if (newProtocolSpreadFeeE6 > maxProtocolSpreadFeeE6) revert ProtocolFeeTooHigh();
    if (newProtocolNotionalFeeE8 > maxProtocolNotionalFeeE8) revert ProtocolFeeTooHigh();

    PoolFeeConfig memory c = poolFeeConfig[pool];
    IMetricOmmPoolCollectFees(pool)
      .collectFees(
        c.protocolSpreadFeeE6,
        c.adminSpreadFeeE6,
        c.protocolNotionalFeeE8,
        c.adminNotionalFeeE8,
        poolAdminFeeDestination[pool]

    uint24 aSpread = c.adminSpreadFeeE6;
    uint24 aNotional = c.adminNotionalFeeE8;
    if (aSpread > maxAdminSpreadFeeE6) {
      aSpread = maxAdminSpreadFeeE6;
      emit PoolAdminSpreadFeeUpdated(pool, aSpread);

      aNotional = maxAdminNotionalFeeE8;
      emit PoolAdminNotionalFeeUpdated(pool, aNotional);

    c = PoolFeeConfig({

      adminSpreadFeeE6: aSpread,

      adminNotionalFeeE8: aNotional

    IMetricOmmPoolFactoryActions(pool)
