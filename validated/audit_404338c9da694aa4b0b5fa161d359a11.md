### Title
`skip_stateful_validations` Admits Forged-Signature Invoke Transactions to the Mempool by Checking Any Mempool Presence Instead of a Deploy-Account Transaction - (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The gateway's `skip_stateful_validations` function is designed to skip the `__validate__` entry-point call for an Invoke(nonce=1) transaction when the sender's account is not yet deployed, provided a deploy-account transaction is pending in the mempool. However, the guard it uses — `account_tx_in_pool_or_recent_block` — returns `true` for **any** transaction from that address in the pool, not exclusively a deploy-account transaction. An unprivileged attacker who observes that a victim's deploy-account transaction is in the mempool can submit a forged-signature Invoke(nonce=1) for the victim's address. The gateway skips `__validate__`, and the forged transaction is admitted to the mempool without any cryptographic signature verification.

### Finding Description

**Root cause — wrong predicate in `skip_stateful_validations`:** [1](#0-0) 

The function returns `true` (skip validation) when:
1. The transaction is `Invoke` with `nonce == 1`,
2. The on-chain account nonce is `0` (account not yet deployed), and
3. `account_tx_in_pool_or_recent_block(sender)` returns `true`.

The comment claims this is sufficient because it "means that either it has a deploy_account transaction or transactions with future nonces that passed validations." However, `account_tx_in_pool_or_recent_block` is implemented as: [2](#0-1) 

It returns `true` if **any** transaction from that address is in `tx_pool` — not specifically a deploy-account transaction. A victim's deploy-account transaction in the pool satisfies this predicate for an attacker's forged Invoke targeting the same address.

**Exploit path:**

1. Victim submits a valid `DeployAccount` for address `A`; it passes full gateway validation and enters the mempool.
2. Attacker submits `Invoke(sender=A, nonce=1, signature=<forged>)`.
3. Gateway stateless validation passes (signature *size* is checked, not validity).
4. `validate_nonce` passes: `0 ≤ 1 ≤ max_allowed_nonce_gap`. [3](#0-2) 

5. `validate_by_mempool` passes: the mempool sees nonce=1 as a valid future nonce for account_nonce=0.
6. `skip_stateful_validations` returns `true` because `account_tx_in_pool_or_recent_block(A)` is `true` (victim's deploy-account is in the pool). [4](#0-3) 

7. `run_validate_entry_point` is called with `skip_validate=true`, setting `execution_flags.validate = false`. [5](#0-4) 

8. Inside `StatefulValidator::perform_validations`, the `validate=false` flag causes an early return before `__validate__` is ever invoked: [6](#0-5) 

9. `perform_pre_validation_stage` checks nonce and fee bounds but **never checks the signature**: [7](#0-6) 

10. The forged Invoke is admitted to the mempool via `mempool_client.add_tx`. [8](#0-7) 

### Impact Explanation

An unprivileged attacker can inject Invoke transactions with arbitrary (forged) signatures into the mempool for any address that has a pending deploy-account transaction. The forged transaction bypasses the only cryptographic check (`__validate__`) that the gateway performs. The transaction will revert during batcher execution (because the batcher calls `__validate__` with `validate=true`), but it is **admitted to the mempool as a valid transaction**, consuming mempool slots, batcher execution resources, and potentially displacing or delaying legitimate transactions from the victim's account. This matches the impact: **High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

### Likelihood Explanation

Any deploy-account transaction in the mempool is publicly observable (the mempool propagates transactions via P2P). An attacker needs only to observe a pending deploy-account for address `A` and submit a crafted Invoke(nonce=1, sender=A) with a zero or random signature. No privileged access, no keys, and no special infrastructure are required. The attack is repeatable for every new deploy-account that enters the mempool.

### Recommendation

Replace the `account_tx_in_pool_or_recent_block` predicate with a check that specifically verifies a **deploy-account** transaction exists for the sender address. The mempool should expose a dedicated query such as `deploy_account_in_pool(address) -> bool` that only returns `true` when a `DeployAccount` transaction (not any transaction) is present for that address. Alternatively, require that the deploy-account transaction hash be provided by the submitter (as `PyValidator.should_run_stateful_validations` does via `deploy_account_tx_hash`) and verify it against the mempool before skipping `__validate__`. [9](#0-8) 

### Proof of Concept

```
1. Victim: submit DeployAccount(address=A, class_hash=H, salt=S, ctor_calldata=[...])
   → Gateway validates fully, tx enters mempool.
   → mempool.account_tx_in_pool_or_recent_block(A) == true

2. Attacker: submit Invoke(sender=A, nonce=1, calldata=[...], signature=[0x0, 0x0])
   → stateless_tx_validator.validate() passes (signature length 2 ≤ max)
   → validate_nonce() passes (0 ≤ 1 ≤ 200)
   → validate_by_mempool() passes (nonce gap allowed)
   → skip_stateful_validations() returns true (A is in pool)
   → run_validate_entry_point(skip_validate=true) → __validate__ NOT called
   → mempool.add_tx() succeeds

3. Batcher executes block:
   → DeployAccount(A) executes → A is deployed
   → Invoke(A, nonce=1, sig=[0,0]) executes → __validate__ called → REVERT
   → Forged tx consumed sequencer resources; legitimate nonce=1 tx from victim is displaced
```

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}

    pub fn should_run_stateful_validations(
        &mut self,
        account_tx: &AccountTransaction,
        deploy_account_tx_hash: Option<TransactionHash>,
    ) -> StatefulValidatorResult<bool> {
        if account_tx.tx_type() != TransactionType::InvokeFunction {
            return Ok(true);
        }
        let tx_info = account_tx.create_tx_info();
        let nonce = self.stateful_validator.get_nonce(tx_info.sender_address())?;

        let deploy_account_not_processed =
            deploy_account_tx_hash.is_some() && nonce == Nonce(Felt::ZERO);
        let tx_nonce = tx_info.nonce();
        let is_post_deploy_nonce = Nonce(Felt::ONE) <= tx_nonce;
        let nonce_small_enough_to_qualify_for_validation_skip =
            tx_nonce <= self.max_nonce_for_validation_skip;

        let skip_validate = deploy_account_not_processed
            && is_post_deploy_nonce
            && nonce_small_enough_to_qualify_for_validation_skip;

        Ok(!skip_validate)
    }
