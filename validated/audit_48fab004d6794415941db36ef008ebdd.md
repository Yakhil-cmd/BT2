### Title
Unbounded Loop Over `pool_member_epoch_balance` Trace in `calculate_rewards` Permanently Freezes Pool Member Yield - (File: `src/pool/pool.cairo`)

---

### Summary

The `calculate_rewards` internal function in `src/pool/pool.cairo` iterates over every entry in a pool member's `pool_member_epoch_balance` trace since their last claim. Because there is no cap on how many entries can accumulate in this trace, a pool member who makes balance changes across many epochs without claiming rewards will eventually be unable to call `claim_rewards` — the transaction will exceed Starknet's gas limit, permanently freezing their unclaimed yield.

---

### Finding Description

`calculate_rewards` (lines 837–888 of `src/pool/pool.cairo`) contains an explicitly unbounded `while` loop:

```cairo
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
    ...
    let to_sigma = self.find_sigma(pool_member_checkpoint, curr_epoch: until_epoch);
    ...
    entry_to_claim_from += 1;
}
```

The loop variable `pool_member_trace_length` is the length of the `pool_member_epoch_balance` storage `Vec` for the given pool member. Each iteration performs at least two storage reads (`pool_member_trace.at(...)` and `find_sigma(...)`), making each step non-trivial in gas cost.

**How the trace grows:**

`set_member_balance` (lines 718–729) calls `trace.insert(key: self.get_epoch_plus_k(), value: ...)`. The `insert` implementation in `src/pool/pool_member_balance_trace/trace.cairo` (lines 152–175) appends a **new** `PoolMemberBalanceCheckpoint` to the `Vec` whenever the key (epoch) differs from the last entry's key. A new entry is therefore appended on every call to:

- `enter_delegation_pool` (line 201) — first entry
- `add_to_delegation_pool` (line 242, via `increase_member_balance`) — each call in a new epoch
- `exit_delegation_pool_intent` (line 278) — each call in a new epoch
- `enter_delegation_pool_from_staking_contract` (lines 456, 464) — on pool switch

There is **no upper bound** on the number of entries that can accumulate. The `entry_to_claim_from` cursor stored in `pool_member_info` advances only when `claim_rewards` is successfully called. If a pool member makes balance changes across N epochs without claiming, the next `claim_rewards` call must iterate over all N entries in a single transaction.

---

### Impact Explanation

**Permanent freezing of unclaimed yield (High).**

Once the `pool_member_epoch_balance` trace grows large enough that iterating over it in a single transaction exceeds Starknet's gas limit, every subsequent call to `claim_rewards` for that pool member will revert. The pool member's accumulated rewards are permanently inaccessible — they cannot be claimed in batches because `entry_to_claim_from` is only updated on a successful completion of `calculate_rewards`, and there is no partial-claim mechanism.

---

### Likelihood Explanation

**Medium-to-High likelihood.**

The minimum amount for `add_to_delegation_pool` is `amount.is_non_zero()` (i.e., 1 token unit). A pool member who is active over many epochs and makes even infrequent balance changes (e.g., topping up once per epoch) will grow their trace linearly with time. Additionally, since `add_to_delegation_pool` can be called by either the `pool_member` address **or** the `pool_member_info.reward_address` (line 229), a malicious or compromised reward address can deliberately inflate the trace on behalf of the pool member, griefing them. No privileged role is required — any registered pool member or their reward address can trigger this.

---

### Recommendation

1. **Paginate reward claims**: Allow `claim_rewards` to accept a `max_entries` parameter and process only that many trace entries per call, advancing `entry_to_claim_from` incrementally.
2. **Bound trace growth**: Enforce a maximum number of balance-change entries per pool member per epoch (the `insert` function already deduplicates within the same epoch, but does not cap across epochs).
3. **Acknowledge and document the risk**: At minimum, document the maximum safe trace length and add a protocol-level guard that prevents balance changes once the trace exceeds a safe threshold.

---

### Proof of Concept

**Setup:**
1. Staker stakes and opens a STRK delegation pool.
2. Pool member enters the pool with 1 STRK.
3. For each of N epochs (e.g., N = 10,000), the pool member calls `add_to_delegation_pool` with 1 token unit, advancing the epoch between calls. Each call appends a new entry to `pool_member_epoch_balance` because `get_epoch_plus_k()` returns a different key each time.
4. The pool member never calls `claim_rewards` during this period.

**Trigger:**
After N epochs, the pool member calls `claim_rewards`. The call enters `calculate_rewards`, which iterates from `entry_to_claim_from = 0` to `pool_member_trace_length = N`. Each iteration reads storage twice. For large N, the transaction exceeds Starknet's gas limit and reverts.

**Result:**
All subsequent `claim_rewards` calls revert. The pool member's unclaimed yield is permanently frozen.

            // Calculate rewards and update entry_to_claim_from.

            // Emit event.

                        pool_member, reward_address, amount: rewards,

                );

    fn insert(
        self: StoragePath<Mutable<PoolMemberBalanceTrace>>, key: Epoch, value: PoolMemberBalance,
    ) -> (PoolMemberBalance, PoolMemberBalance) {
        let checkpoints = self.checkpoints;

        let len = checkpoints.len();
        if len == Zero::zero() {
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
            return (Zero::zero(), value);
        }

        // Update or append new checkpoint.
        let mut last = checkpoints[len - 1].read();
        let prev = last.value;
        if last.key == key {
            last.value = value;
            checkpoints[len - 1].write(last);
        } else {
            // Checkpoint keys must be non-decreasing.
            assert!(last.key < key, "{}", TraceErrors::UNORDERED_INSERTION);
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
        }
        (prev, value)
    }
