### Title
Pool Admin Bypasses `maxAdminSpreadFeeE6` Cap via Unchecked `setPoolBinAdditionalFees` — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`MetricOmmPoolFactory.setPoolBinAdditionalFees` forwards arbitrary `addFeeBuyE6`/`addFeeSellE6` values to the pool with no validation against `maxAdminSpreadFeeE6`, while the parallel admin fee setter `setPoolAdminFees` does enforce that cap. A pool admin can therefore set per-bin additional fees to `uint16.max` (65535 = 6.5535%) on any bin, causing every swap through that bin to pay fees far above the protocol-configured cap.

---

### Finding Description

`MetricOmmPoolFactory.setPoolAdminFees` correctly guards the admin spread fee:

```solidity
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
``` [1](#0-0) 

But `setPoolBinAdditionalFees` performs no such check before forwarding to the pool:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
``` [2](#0-1) 

`MetricOmmPool.setBinAdditionalFees` writes the value directly to `BinState` storage with no cap:

```solidity
s.addFeeBuyE6 = addFeeBuyE6;
s.addFeeSellE6 = addFeeSellE6;
``` [3](#0-2) 

During every swap, the per-bin additional fee is added directly to `baseFeeX64` (the oracle-derived spread) before computing the effective execution price:

```solidity
params.baseFeeX64 + Math.mulDiv(binState.addFeeSellE6, ONE_X64, 1e6),
``` [4](#0-3) 

The same pattern applies to the buy direction (`addFeeBuyE6`) in `_swapToken1ForToken0SpecifiedInput` and `_swapToken1ForToken0SpecifiedOutput`.

---

### Impact Explanation

A pool admin calling `setPoolBinAdditionalFees(pool, 0, 65535, 65535)` sets `addFeeBuyE6 = addFeeSellE6 = 65535` (6.5535%) on bin 0. Every swap routed through that bin pays an effective fee of `baseFeeX64 + 6.5535%`, regardless of the factory's `maxAdminSpreadFeeE6` cap (e.g., 1000 = 0.1%). The excess fee is extracted from traders on every swap through the affected bin, constituting direct loss of user principal above Sherlock Medium thresholds. The `BinState` struct stores `addFeeBuyE6`/`addFeeSellE6` as `uint16`, so the maximum exploitable value is 65535. [5](#0-4) 

---

### Likelihood Explanation

The pool admin is a semi-trusted role explicitly scoped by `maxAdminSpreadFeeE6`. The factory owner sets that cap to constrain pool admin behavior. Any pool admin — including a compromised multisig or a malicious actor who acquired the role via `acceptPoolAdmin` — can exploit this path immediately with a single transaction, with no timelock and no protocol override possible after the fact. The factory owner cannot retroactively remove the per-bin fee; only the pool admin can reset it.

---

### Recommendation

Add a cap check in `setPoolBinAdditionalFees` consistent with the guard already present in `setPoolAdminFees`:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (addFeeBuyE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
``` [2](#0-1)
