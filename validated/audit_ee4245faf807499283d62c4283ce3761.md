### Title
Zero `priceProviderTimelock` bypasses oracle-rotation delay, enabling atomic pool-admin oracle swap — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`_validatePoolParameters` accepts any value for `priceProviderTimelock` that is not `type(uint256).max`, including zero. A zero timelock collapses the propose-then-execute oracle-rotation delay to nothing: `proposePoolPriceProvider` sets `pendingPriceProviderExecuteAfter[pool] = block.timestamp + 0 = block.timestamp`, and `executePoolPriceProviderUpdate` immediately passes the guard `if (block.timestamp < execAfter)` (false when equal). The pool admin can therefore swap the live price provider atomically within a single transaction, with no window for LPs to observe and exit.

---

### Finding Description

`createPool` is permissionless. Any caller may supply `params.priceProviderTimelock = 0`. The factory stores it verbatim:

```solidity
// MetricOmmPoolFactory.sol line 213
priceProviderTimelock[pool] = params.priceProviderTimelock;
```

`_validatePoolParameters` checks tokens, admin, price-provider token match, fee caps, initial amounts, and minimal liquidity — but contains **no check** that `priceProviderTimelock > 0` when the pool is in mutable mode:

```solidity
// MetricOmmPoolFactory.sol lines 548-563
function _validatePoolParameters(PoolParameters calldata params) internal view {
    if (params.token0 == address(0) || ...) revert InvalidTokenConfig();
    if (params.admin == address(0)) revert InvalidAdmin();
    _validatePriceProvider(params.token0, params.token1, params.priceProvider);
    if (params.adminFeeDestination == address(0)) revert InvalidAdminFeeDestination();
    // ... fee cap checks ...
    if (params.initialAmount0PerShareE18 == 0 || ...) revert InvalidInitialAmount();
    if (params.minimalMintableLiquidity == 0) revert InvalidMinimalMintableLiquidity();
    // ← NO check: priceProviderTimelock > 0
}
```

With `timelock = 0`, the propose step computes:

```solidity
// MetricOmmPoolFactory.sol line 487
uint256 executeAfter = block.timestamp + timelock; // = block.timestamp
```

And the execute step's guard:

```solidity
// MetricOmmPoolFactory.sol line 499
if (block.timestamp < execAfter) revert PriceProviderTimelockNotElapsed(...);
// block.timestamp < block.timestamp → false → passes immediately
```

There is no `cancelPoolPriceProvider` function in the factory, so once proposed the admin can execute in the same transaction with no opportunity for LPs to react.

---

### Impact Explanation

A pool admin controlling a pool deployed with `priceProviderTimelock = 0` can atomically (within one transaction) replace the live oracle with a malicious `IPriceProvider` that returns arbitrary `bidPriceX64` / `askPriceX64`. Every subsequent swap in that transaction executes against the manipulated bid/ask, allowing the admin to extract LP principal. Because pool creation is permissionless, an attacker can deploy such a pool, attract LP deposits, and then execute the atomic oracle swap + drain in a single block. This is a direct loss of LP principal above contest thresholds and an admin-boundary break: the timelock is the only cap constraining the pool admin's oracle authority, and a zero value eliminates it entirely.

---

### Likelihood Explanation

Pool creation is fully permissionless once `poolDeployer` is set. Any address can supply `priceProviderTimelock = 0` without any on-chain barrier. The pool admin role is assigned at creation and can be the same address as the creator. The only off-chain barrier is LP due-diligence on the `PoolCreated` event's `priceProviderTimelock` field. Likelihood is **Medium**: the attacker must attract meaningful LP deposits before executing, but the on-chain path is entirely open.

---

### Recommendation

Add a minimum-timelock guard in `_validatePoolParameters`:

```solidity
// in MetricOmmPoolFactory._validatePoolParameters
if (
    params.priceProviderTimelock != type(uint256).max &&
    params.priceProviderTimelock == 0
) revert InvalidPriceProviderTimelock();
```

Or enforce a protocol-defined minimum (e.g., 1 hour) for mutable-mode pools:

```solidity
uint256 internal constant MIN_PRICE_PROVIDER_TIMELOCK = 1 hours;

if (
    params.priceProviderTimelock != type(uint256).max &&
    params.priceProviderTimelock < MIN_PRICE_PROVIDER_TIMELOCK
) revert PriceProviderTimelockTooShort();
```

### Proof of Concept

```
1. Attacker calls factory.createPool({
       priceProviderTimelock: 0,   // mutable mode, zero delay
       admin: attackerContract,
       priceProvider: legitimateOracle,
       ...
   })
   → Pool deployed; priceProviderTimelock[pool] = 0.

2. LPs add liquidity to the pool (attracted by legitimate oracle).

3. Attacker deploys MaliciousOracle that returns
   bidPriceX64 = 0, askPriceX64 = type(uint128).max
   (or any price that benefits the attacker's swap direction).

4. In a single transaction, attackerContract calls:
   a. factory.proposePoolPriceProvider(pool, maliciousOracle)
      → pendingPriceProviderExecuteAfter[pool] = block.timestamp + 0 = block.timestamp
   b. factory.executePoolPriceProviderUpdate(pool)
      → block.timestamp < block.timestamp → false → passes
      → pool.setPriceProvider(maliciousOracle)
   c. pool.swap(...) using malicious bid/ask
      → attacker receives more token than oracle-fair value
      → LP principal is drained as the difference.

5. Attacker repays any flash loan and keeps the profit.
```

**Root cause:** `_validatePoolParameters` at [1](#0-0)  contains no check that `priceProviderTimelock > 0` for mutable-mode pools.

**Zero-timelock execution path:** `proposePoolPriceProvider` computes `executeAfter = block.timestamp + 0` at [2](#0-1) , and `executePoolPriceProviderUpdate` passes the guard `block.timestamp < execAfter` immediately at [3](#0-2) .

**Factory stores timelock verbatim without validation:** [4](#0-3) 

**`priceProviderTimelock` field in `PoolParameters` accepts any `uint256`:** [5](#0-4)
