### Title
Commission Increase via `set_commission_commitment` Takes Immediate Effect, Retroactively Reducing Rewards for Active Delegators - (File: src/staking/staking.cairo)

---

### Summary

A staker can use `set_commission_commitment` to establish a high `max_commission` ceiling, then immediately call `set_commission` to raise their commission to that ceiling. Unlike balance changes (which use a K-epoch delay via `get_epoch_plus_k()`), commission changes are written to storage with **no epoch delay**. The new commission is applied at the very next reward calculation, reducing rewards for all existing delegators who entered the pool under the old (lower) commission rate. Delegators cannot exit immediately — they must go through `exit_delegation_pool_intent` plus the full `exit_wait_window` — so they are forced to absorb the higher commission during the exit period.

---

### Finding Description

**Root cause — no epoch delay on commission writes:**

In `update_commission()`, the new commission is written directly to storage with no delay:

```rust
// Update commission in storage.
staker_pool_info.commission.write(Option::Some(commission));
``` [1](#0-0) 

Compare this to every other state change that affects reward calculations (balance updates, token enable/disable), which all use `get_epoch_plus_k()` to defer the effect by K epochs:

```rust
fn update_total_stake(...) {
    self.tokens_total_stake_trace.entry(token_address)
        .insert(key: self.get_epoch_plus_k(), ...);
}
``` [2](#0-1) 

**Commission increase is allowed via `set_commission_commitment`:**

Without a commitment, `set_commission` enforces `commission < old_commission` (decrease only). However, with an active `CommissionCommitment`, the staker may set commission to any value up to `max_commission`, including a large increase:

```rust
if self.is_commission_commitment_active(:commission_commitment) {
    assert!(commission <= commission_commitment.max_commission, ...);
    assert!(commission != old_commission, ...);
} else {
    assert!(commission < old_commission, ...);
}
``` [3](#0-2) 

`max_commission` can be set up to `COMMISSION_DENOMINATOR` (10000 = 100%):

```rust
assert!(max_commission <= COMMISSION_DENOMINATOR, "{}", Error::COMMISSION_OUT_OF_RANGE);
``` [4](#0-3) 

**Commission is read at reward calculation time:**

In `calculate_staker_pools_rewards()`, the commission used to split rewards is the **current** commission at the moment rewards are distributed — not the commission that was in effect when delegators entered the pool:

```rust
let commission = staker_pool_info.commission();
...
let (commission_rewards, pool_rewards) = split_rewards_with_commission(
    rewards_including_commission: pool_rewards_including_commission, :commission,
);
``` [5](#0-4) 

**Delegators cannot exit immediately:**

`exit_delegation_pool_intent` initiates an exit but requires waiting for `exit_wait_window` before `exit_delegation_pool_action` can be called. During this window, rewards continue to be calculated with the new higher commission:

```rust
assert!(Time::now() >= unpool_time, "{}", GenericError::INTENT_WINDOW_NOT_FINISHED);
``` [6](#0-5) 

---

### Impact Explanation

**High — Theft of unclaimed yield.**

A staker who sets `max_commission = 10000` (100%) and then calls `set_commission(10000)` will cause all subsequent reward distributions to send 100% of pool rewards to the staker as commission, leaving delegators with 0 rewards. Delegators who entered the pool under a 0% or low commission rate are forced to absorb this change for the entire duration of the exit wait window (up to `MAX_EXIT_WAIT_WINDOW = 12 weeks`). The rewards that would have accrued to delegators during this period are instead captured by the staker.

---

### Likelihood Explanation

**Medium.** The attack requires the staker to first call `set_commission_commitment`, which is an on-chain event. Vigilant delegators monitoring the `CommissionCommitmentSet` event could exit preemptively. However:
1. Most delegators do not actively monitor on-chain events.
2. Even if they notice, the exit window means they cannot avoid the commission increase for the full wait period.
3. The commitment can be set and the commission raised in consecutive transactions within the same block, giving delegators no practical reaction time.

The protocol's own code comment acknowledges this gap: *"Current commission increase safeguards still allow for sudden commission changes."* [7](#0-6) 

---

### Recommendation

Apply the same K-epoch delay to commission increases that is applied to all other state changes affecting reward calculations. Specifically, when `set_commission` results in a commission **increase** (only possible under an active commitment), the new commission should be stored with an activation epoch of `current_epoch + K`, so that existing delegators have time to exit before the higher commission takes effect. Commission decreases can remain immediate (they benefit delegators).

---

### Proof of Concept

1. Staker stakes and opens a delegation pool with `commission = 0`.
2. Delegators enter the pool expecting 0% commission.
3. Staker calls `set_commission_commitment(max_commission: 10000, expiration_epoch: current_epoch + 1)`.
4. In the same block, staker calls `set_commission(commission: 10000)`.
5. Commission is immediately written to storage with no delay.
6. The next call to `update_rewards_from_attestation_contract` or `update_rewards` invokes `calculate_staker_pools_rewards`, which reads `commission = 10000` and calls `split_rewards_with_commission`, returning `pool_rewards = 0` for all delegators.
7. Delegators call `exit_delegation_pool_intent` but must wait the full `exit_wait_window` (up to 12 weeks).
8. During the entire exit window, every reward distribution gives delegators 0 rewards. All pool rewards flow to the staker as commission.
