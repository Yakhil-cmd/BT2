Audit Report

## Title
Missing Timestamp Freshness Validation in CMC Exchange Rate Acceptance Allows Stale-Rate Cycles Over-Minting - (File: rs/nervous_system/clients/src/exchange_rate_canister_client.rs)

## Summary
`validate_exchange_rate` in `rs/nervous_system/clients/src/exchange_rate_canister_client.rs` checks only data-source counts on the `ExchangeRate` returned by the XRC, never comparing `exchange_rate.timestamp` against the canister's current time. `do_set_icp_xdr_conversion_rate` in `rs/nns/cmc/src/main.rs` enforces only monotonicity of the timestamp, not proximity to `now`. As a result, when the XRC returns a rate whose timestamp lags the current time by minutes to hours (a realistic outcome during exchange-source degradation), the CMC silently stores and indefinitely uses that stale rate for all cycles-minting operations, enabling any unprivileged caller to mint cycles at an incorrect price.

## Finding Description

**Root cause — `validate_exchange_rate`** (`rs/nervous_system/clients/src/exchange_rate_canister_client.rs`, L110–129):

```rust
pub fn validate_exchange_rate(
    exchange_rate: &ExchangeRate,
) -> Result<(), ValidateExchangeRateError> {
    if exchange_rate.metadata.base_asset_num_received_rates < MINIMUM_ICP_SOURCES { … }
    if exchange_rate.metadata.quote_asset_num_received_rates < MINIMUM_CXDR_SOURCES { … }
    Ok(())
}
```

The function performs exactly two checks. There is no comparison of `exchange_rate.timestamp` against the canister's current time.

**Rate acceptance — `do_set_icp_xdr_conversion_rate`** (`rs/nns/cmc/src/main.rs`, L1009–1040):

```rust
if let Some(current_conversion_rate) = state.icp_xdr_conversion_rate.as_ref()
    && proposed_conversion_rate.timestamp_seconds
        <= current_conversion_rate.timestamp_seconds
{
    return Err("Proposed conversion rate must have greater timestamp …");
}
state.icp_xdr_conversion_rate = Some(proposed_conversion_rate.clone());
```

The only temporal guard is monotonicity (`new > old`). `env.now_timestamp_seconds()` is available in scope but is never consulted.

**Point-of-use — `tokens_to_cycles`** (`rs/nns/cmc/src/main.rs`, L1900–1923):

```rust
fn tokens_to_cycles(amount: Tokens) -> Result<Cycles, NotifyError> {
    with_state(|state| {
        let xdr_permyriad_per_icp = state
            .icp_xdr_conversion_rate
            .as_ref()
            .map(|rate| rate.xdr_permyriad_per_icp);
        …
    })
}
```

`tokens_to_cycles` reads `xdr_permyriad_per_icp` directly from state with no check on `rate.timestamp_seconds` vs. current time. This function is called unconditionally by `process_top_up`, `process_create_canister`, and `process_mint_cycles`.

**Exploit flow:**
1. The XRC's HTTPS-outcall data sources experience transient unavailability (normal exchange downtime).
2. The XRC returns a rate whose `timestamp` is, say, 30–60 minutes old but still newer than the CMC's currently stored rate, and with ≥4 ICP sources and ≥4 CXDR sources.
3. `validate_exchange_rate` passes (source counts are sufficient).
4. `do_set_icp_xdr_conversion_rate` passes (new timestamp > old timestamp).
5. The stale rate is stored and used for the next refresh window (5 minutes, or longer if subsequent XRC calls also fail — retries fall back to once-per-minute).
6. An attacker who observes via the public `get_icp_xdr_conversion_rate` query that the stored rate diverges from the real market price calls `notify_top_up` / `notify_mint_cycles` with ICP purchased at the current market price, receiving cycles priced at the stale rate.

## Impact Explanation

The ICP/XDR rate stored in the CMC directly determines how many cycles are minted per ICP burned. A stale high rate causes over-minting (attacker receives more cycles than the current market warrants, extracting value from the protocol). A stale low rate causes under-minting (users are harmed). Both outcomes are reachable by any unprivileged caller of `notify_top_up`, `notify_create_canister`, or `notify_mint_cycles` — no special role is required. This constitutes a significant XRC/NNS financial infrastructure security impact with concrete protocol economic harm, matching the High ($2,000–$10,000) impact class: "Significant XRC, NNS, or infrastructure security impact with concrete user or protocol harm."

The per-hour cycles-minting rate limit (`CYCLES_MINTING_LIMIT = 150e15` cycles) constrains the maximum damage per exploitation window but does not eliminate it; the limit resets hourly and the stale-rate window can persist across multiple reset periods if XRC degradation continues.

## Likelihood Explanation

The XRC aggregates ICP/USD prices via HTTPS outcalls to external exchanges. Transient exchange outages or XRC-internal rate-limiting are documented operational realities, not theoretical edge cases. No attacker capability is required to cause the degraded state — the attacker only needs to observe it via the public `get_icp_xdr_conversion_rate` query (which exposes `timestamp_seconds`). Once a divergence is observed, exploitation requires only a standard ICP transfer and a call to `notify_top_up` — no privileged access, no governance majority, no network-level attack. The condition is repeatable whenever XRC data sources degrade.

## Recommendation

1. **Add a maximum-age check in `validate_exchange_rate`** (or as a separate step in `update_exchange_rate` after the XRC call) that rejects any rate whose `timestamp` is older than `now - (REFRESH_RATE_INTERVAL_SECONDS * 2)` (e.g., 10 minutes). The `env.now_timestamp_seconds()` value is already available at the call site in `update_exchange_rate`.

2. **Add a point-of-use staleness guard in `tokens_to_cycles`** that returns a retriable `NotifyError` if `now - icp_xdr_conversion_rate.timestamp_seconds` exceeds a safe bound (e.g., 15–30 minutes), rather than minting at a potentially stale price.

## Proof of Concept

**Deterministic integration test plan (using `StateMachine` / PocketIC):**

1. Initialize the NNS with a mock XRC canister.
2. Advance time so the CMC fetches an initial rate at timestamp `T` with a known value (e.g., 10 XDR/ICP).
3. Reconfigure the mock XRC to return a rate with timestamp `T + 60` (one minute newer than stored, so monotonicity passes) but with a value of 14 XDR/ICP, simulating a stale high rate while real market is at 10 XDR/ICP.
4. Advance time by 5 minutes and tick to trigger the CMC heartbeat.
5. Assert that `get_icp_xdr_conversion_rate` returns the stale 14 XDR/ICP rate.
6. Call `notify_top_up` with 1 ICP; assert that the minted cycles correspond to 14 XDR/ICP rather than the current market rate of 10 XDR/ICP — a 40% over-mint.
7. Confirm that `validate_exchange_rate` accepted the rate solely because source counts were ≥ 4, with no timestamp proximity check.
