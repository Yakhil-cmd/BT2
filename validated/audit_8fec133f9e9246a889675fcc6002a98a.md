Audit Report

## Title
Single Buyer Can Abort SNS Swap Immediately When `max_participant_icp_e8s == max_direct_participation_icp_e8s` and `min_participants > 1` - (File: rs/sns/init/src/lib.rs)

## Summary
The `validate_participation_constraints` function in `rs/sns/init/src/lib.rs` permits configurations where `max_participant_icp_e8s == max_direct_participation_icp_e8s` alongside `min_participants > 1`. This combination, explicitly endorsed by the proto documentation as a way to "disable" the per-participant cap, allows a single unprivileged buyer to fill the entire ICP target, triggering `icp_target_reached` without `sufficient_participation`, and causing the swap to abort on the next heartbeat — before the due date and before other participants can join.

## Finding Description
The validation at `rs/sns/init/src/lib.rs` line 1603–1608 only rejects `max_participant_icp_e8s > max_direct_participation_icp_e8s`; equality is explicitly permitted:

```rust
// (4)
if max_participant_icp_e8s > max_direct_participation_icp_e8s {
    return Err(...);
}
```

There is no guard checking `max_participant_icp_e8s >= max_direct_participation_icp_e8s` when `min_participants > 1`.

The proto field comment at `rs/sns/swap/proto/ic_sns_swap/pb/v1/swap.proto` line 600–604 actively encourages this configuration:
> "Can effectively be disabled by setting it to `max_icp_e8s`."

Once the swap is open, `can_abort` at `rs/sns/swap/src/swap.rs` line 2906–2914 returns `true` when:
```rust
(self.swap_due(now_seconds) || self.icp_target_progress().is_reached_or_exceeded())
    && !self.sufficient_participation()
```

`sufficient_participation` (`swap.rs` line 2796–2797) requires both `min_participation_reached()` (buyers count ≥ `min_participants`) and `min_direct_participation_icp_e8s_reached()`. With a single buyer filling `max_direct_participation_icp_e8s`, `icp_target_progress().is_reached_or_exceeded()` becomes `true` while `min_participation_reached()` remains `false` (1 buyer < `min_participants`), so `can_abort` fires immediately.

The existing test `test_try_commit_or_abort_insufficient_participation_with_max_icp` at `swap.rs` line 4672–4709 uses exactly this configuration (`max_participant_icp_e8s = max_direct_participation_icp_e8s = 20`, `min_participants = 2`) and asserts the swap aborts — confirming the behavior is reachable and exercised.

## Impact Explanation
An unprivileged buyer can abort any SNS swap configured with `max_participant_icp_e8s == max_direct_participation_icp_e8s` and `min_participants > 1` by contributing the full target amount in a single `refresh_buyer_token_e8s` call. The SNS fails to decentralize; the project loses its decentralization window and must restart the entire NNS proposal process. SNS governance tokens remain locked in the swap canister until `finalize` is called. This constitutes a significant SNS security impact with concrete protocol harm, qualifying as **High ($2,000–$10,000)** under "Significant NNS, SNS, or infrastructure security impact with concrete user or protocol harm."

## Likelihood Explanation
The proto documentation explicitly presents `max_participant_icp_e8s = max_direct_participation_icp_e8s` as a valid, recommended way to disable the per-participant cap. An SNS creator who wants no per-participant limit but still requires multiple participants (`min_participants > 1`) would naturally arrive at this exact dangerous combination by following the documentation. NNS governance validation passes it without error. Once the swap is open, any buyer who can supply `max_direct_participation_icp_e8s` ICP can trigger the abort; their ICP is fully refunded after finalization, making the attack cost-free beyond opportunity cost.

## Recommendation
Add a guard in `validate_participation_constraints` (`rs/sns/init/src/lib.rs`) immediately after the existing check (4):

```rust
if min_participants > 1 && max_participant_icp_e8s >= max_direct_participation_icp_e8s {
    return Err(format!(
        "Error: when min_participants ({min_participants}) > 1, \
         max_participant_icp_e8s ({max_participant_icp_e8s}) must be strictly less than \
         max_direct_participation_icp_e8s ({max_direct_participation_icp_e8s}), \
         otherwise a single buyer can abort the swap immediately by filling the ICP target."
    ));
}
```

Additionally, update the proto comment at `swap.proto` line 602–603 to warn that setting `max_participant_icp_e8s = max_icp_e8s` is unsafe when `min_participants > 1`.

## Proof of Concept
The existing test harness directly supports reproduction. Using `SwapBuilder`:

```rust
let mut swap = SwapBuilder::new()
    .with_lifecycle(Lifecycle::Open)
    .with_swap_start_due(None, Some(1000))
    .with_min_participants(2)
    .with_min_max_participant_icp(10, 100)
    .with_min_max_direct_participation(50, 100) // max_direct == max_participant
    .build();

let buyers = btreemap! {
    PrincipalId::new_user_test_id(0).to_string() => BuyerState::new(100),
};
swap.buyers = buyers;
swap.update_derived_fields();

// now=50, due=1000 — swap is not due yet
assert!(swap.try_abort(50));
assert_eq!(swap.lifecycle, Lifecycle::Aborted as i32);
```

This mirrors the pattern already confirmed by `test_try_commit_or_abort_insufficient_participation_with_max_icp` (`swap.rs` line 4672–4709), which uses `max_participant_icp_e8s = max_direct_participation_icp_e8s = 20` with `min_participants = 2` and asserts `try_abort` succeeds mid-swap.
