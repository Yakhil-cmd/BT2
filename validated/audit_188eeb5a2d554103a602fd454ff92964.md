### Title
Signature Validation Unconditionally Skipped for Invoke Transactions When Any Account Transaction Exists in Mempool - (File: crates/apollo_gateway/src/stateful_transaction_validator.rs)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator skips the `__validate__` entry point (signature verification) for any invoke transaction with `nonce == 1` when the sender account has **any** transaction in the mempool or recent block history. The check does not verify that the existing mempool transaction is specifically a `deploy_account` transaction. An attacker who observes a victim's pending `deploy_account` transaction can immediately submit a malicious invoke transaction with an arbitrary/invalid signature targeting the same account address, and it will be admitted to the mempool without any signature check.

### Finding Description

In `skip_stateful_validations` (lines 429â€“461 of `crates/apollo_gateway/src/stateful_transaction_validator.rs`), the gateway skips the blockifier `__validate__` call when three conditions hold simultaneously:

1. The incoming transaction is an `Invoke` with `nonce == Nonce(Felt::ONE)`
2. The on-chain account nonce is `Nonce(Felt::ZERO)` (account not yet deployed)
3. `mempool_client.account_tx_in_pool_or_recent_block(sender_address)` returns `true` [1](#0-0) 

The comment at line 441â€“443 states the assumption:

> "It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."

This assumption is **incorrect**. `account_tx_in_pool_or_recent_block` is implemented as:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [2](#0-1) 

`tx_pool.contains_account` returns `true` if the account has **any** transaction in the pool â€” not specifically a `deploy_account`. `state.contains_account` returns `true` if the account appears in the mempool's committed or staged nonce maps: [3](#0-2) 

When `skip_validate` is `true`, `run_validate_entry_point` sets `execution_flags.validate = false`: [4](#0-3) 

Inside `StatefulValidator::perform_validations`, when `validate == false`, execution returns immediately without calling `__validate__`: [5](#0-4) 

The `validate_by_mempool` call that precedes `skip_stateful_validations` only checks nonce ordering and fee escalation â€” it contains no signature field and performs no cryptographic check: [6](#0-5) 

### Impact Explanation

An invoke transaction with an **invalid or completely absent signature** is admitted to the mempool without any signature verification. This breaks the core admission invariant that every mempool transaction must have passed account-level validation. The corrupted admission decision is:

- **Wrong value**: `skip_validate = true` (should be `false` unless a `deploy_account` specifically exists)
- **Broken invariant**: Mempool contains a transaction whose signer identity has never been verified
- **Scope match**: "High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."

During actual sequencing, `BlockifierTransaction::new_for_sequencing` always sets `validate: true`: [7](#0-6) 

So the malicious tx will fail at execution time (after the deploy_account is processed and `__validate__` is called with the bad signature). However, the broken invariant at admission time means:
- The mempool is polluted with cryptographically unverified transactions
- Sequencer CPU is wasted attempting to execute them
- An attacker can repeat this for every observed pending `deploy_account` transaction, enabling sustained mempool DoS

### Likelihood Explanation

The trigger requires no privileged access. The attacker only needs to:
1. Observe a pending `deploy_account` transaction in the mempool (publicly visible via RPC)
2. Submit an invoke transaction with the same sender address, `nonce = 1`, and any signature

The `deploy_account` + `invoke` UX pattern is an explicitly supported and documented flow (see `create_deploy_account_tx_and_invoke_tx` in integration tests), so the attack window is a normal, expected operational state â€” not a rare edge case. [8](#0-7) 

### Recommendation

Replace the generic `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a `deploy_account` transaction exists in the mempool for the sender address. The mempool should expose a dedicated `has_pending_deploy_account(address)` query that inspects the transaction type, not just account presence. Alternatively, the gateway should verify the transaction type of the existing mempool entry before granting the validation skip.

### Proof of Concept

**Setup**: Account `A` does not exist on-chain (nonce = 0).

**Step 1**: Legitimate user submits a `deploy_account` transaction for account `A`. The gateway accepts it and it enters the mempool pool. Now `tx_pool.contains_account(A) == true`.

**Step 2**: Attacker submits an invoke transaction:
- `sender_address = A`
- `nonce = 1`
- `signature = [0xdeadbeef, 0xdeadbeef]` (arbitrary garbage)
- Valid resource bounds and calldata

**Step 3**: Gateway stateful validation runs `skip_stateful_validations`:
- `tx.nonce() == Nonce(Felt::ONE)` âś“
- `account_nonce == Nonce(Felt::ZERO)` âś“ (A not deployed)
- `account_tx_in_pool_or_recent_block(A)` â†’ `true` âś“ (deploy_account is in pool)
- Returns `true` â†’ `skip_validate = true`

**Step 4**: `run_validate_entry_point` is called with `skip_validate = true`, setting `execution_flags.validate = false`. The blockifier's `StatefulValidator::perform_validations` returns `Ok(())` at line 80 without calling `__validate__`. [9](#0-8) 

**Result**: The malicious invoke transaction with an invalid signature is admitted to the mempool. The gateway returns a success response to the attacker. The mempool now contains a transaction that has never had its signer identity verified.

/// Generates a deploy account transaction followed by an invoke transaction from the same account.
/// The first invoke_tx can be inserted to the first block right after the deploy_tx due to
/// the skip_validate feature. This feature allows the gateway to accept this transaction although
/// the account does not exist yet.
pub fn create_deploy_account_tx_and_invoke_tx(
    tx_generator: &mut MultiAccountTransactionGenerator,
    account_id: AccountId,
) -> Vec<RpcTransaction> {
    let undeployed_account_tx_generator = tx_generator.account_with_id_mut(account_id);
    assert!(!undeployed_account_tx_generator.is_deployed());
    let deploy_tx = undeployed_account_tx_generator.generate_deploy_account();
    let invoke_tx = undeployed_account_tx_generator.generate_trivial_rpc_invoke_tx(1);
    vec![deploy_tx, invoke_tx]
}
