### Title
Malicious Participant Aborts DKG/Presign Protocol via Deserialization Error Propagation in `recv_from_others` — (`src/protocol/helpers.rs`)

### Summary

The `recv_from_others` helper function, used in every multi-party round of DKG, reshare, refresh, and OT-based presign, propagates deserialization errors directly via `?`. A single malicious participant can craft a message with a syntactically valid channel header but an invalid body, causing the receiving honest party's protocol instance to abort immediately. Because the message is consumed from the buffer on failure, the honest party cannot recover and must restart the entire protocol — which the attacker can abort again indefinitely.

### Finding Description

`recv_from_others` in `src/protocol/helpers.rs` collects one message from every other participant before proceeding: [1](#0-0) 

The inner call `chan.recv(waitpoint).await?` delegates to `Comms::recv`, which pops the message from the buffer and attempts MessagePack deserialization: [2](#0-1) 

If `rmp_serde::decode::from_slice` fails, the error is returned via `?` and the message is permanently consumed from the buffer. The `?` in `recv_from_others` then propagates this `ProtocolError::DeserializationError` all the way up to the caller, aborting the protocol for that honest participant.

This function is called at multiple critical points in DKG: [3](#0-2) [4](#0-3) 

And in OT-based presign: [5](#0-4) [6](#0-5) 

**Attack path:**
1. Attacker is a legitimate participant in a DKG, reshare, refresh, or presign session.
2. At any round that calls `recv_from_others`, the attacker sends a message whose header bytes match the expected `MessageHeader` (channel tag + waitpoint — both deterministic and derivable from the open-source protocol) but whose body is arbitrary garbage.
3. `push_message` accepts the message because the header is valid.
4. When an honest party's `recv_from_others` loop dequeues this message, deserialization fails, the error propagates via `?`, and the honest party's protocol instance terminates with `ProtocolError::DeserializationError`.
5. The attacker repeats this on every restart, permanently blocking completion.

Contrast this with `reliable_broadcast_receive_all` in `src/protocol/echo_broadcast.rs`, which explicitly handles receive errors gracefully by continuing the loop: [7](#0-6) 

`recv_from_others` has no equivalent guard.

### Impact Explanation

**High — Permanent denial of key generation, reshare, refresh, and presign for honest parties.**

Any single participant in a DKG or presign session can abort the protocol for every honest party at any round. Because the attacker can repeat this on every restart, honest parties can never complete key generation or presigning as long as the attacker remains in the participant set. Removing the attacker requires an out-of-band reshare with a different participant set, which itself uses the same vulnerable `recv_from_others` path and can be aborted again.

### Likelihood Explanation

**High.** The attack requires only that the attacker be a legitimate protocol participant — no privileged keys, no cryptographic breaks, no external assumptions. The header format is fully deterministic and derivable from the open-source code. Sending a single malformed message per round is trivially cheap. The attacker does not need to know any secret material.

### Recommendation

Replace the bare `?` propagation in `recv_from_others` with per-message error handling that skips malformed messages and continues waiting. To avoid waiting forever for a participant who sent garbage, the lower-level `recv` should be refactored to return the sender identity even on deserialization failure, so the caller can mark that participant as seen and move on:

```rust
// In recv_from_others:
while !seen.full() {
    // recv_raw returns (Participant, Result<T, ProtocolError>)
    let (from, result) = chan.recv_raw(waitpoint).await;
    if let Ok(msg) = result {
        if seen.put(from) {
            messages.push((from, msg));
        }
    } else {
        // Mark sender as seen so we don't wait for them again,
        // and optionally record them as malicious.
        seen.put(from);
    }
}
```

This mirrors the pattern already used in `reliable_broadcast_receive_all`, which continues on receive errors rather than aborting.

### Proof of Concept

```
Setup: 3-party DKG with threshold=2. Participants: P0 (honest), P1 (honest), P2 (attacker).

Round 1 (wait_round_1 in do_keyshare):
  - P0 and P1 send valid commitment_hash messages.
  - P2 constructs a message whose first MessageHeader::LEN bytes are a valid
    header for wait_round_1 (channel tag = root_shared, waitpoint = 0),
    followed by random bytes as the body.
  - P2 sends this to P0.

Result at P0:
  recv_from_others(&chan, wait_round_1, &participants, me).await
    → chan.recv(wait_round_1).await  [dequeues P2's message]
    → rmp_serde::decode::from_slice(body) fails
    → Err(ProtocolError::DeserializationError(...))
    → ? propagates → do_keyshare returns Err
    → DKG aborted for P0.

P2 repeats on every restart. DKG never completes.
```

        &self,
        header: MessageHeader,
    ) -> Result<(Participant, T), ProtocolError> {

        let message_data = data.get(MessageHeader::LEN..).ok_or_else(|| {

        Ok((from, decoded?))
    }
