### Title
Malicious Participant Can Permanently Abort Honest Party's DKG via Type-Mismatched Private Share at `wait_round_3` — (`src/protocol/helpers.rs`, `src/dkg.rs`, `src/protocol/internal.rs`)

---

### Summary

A malicious participant in a DKG session can permanently abort any honest party's key generation by sending a private message at `wait_round_3` whose body fails to deserialize as `SigningShare<C>`. The error propagates unconditionally through `recv_from_others` → `do_keyshare` with no recovery path, permanently denying key generation to the targeted honest party.

---

### Finding Description

**Step 1 — Attacker-controlled entrypoint.**

In Round 5 of `do_keyshare`, each participant sends a private `SigningShare` to every other participant at `wait_round_3`, then collects all shares via `recv_from_others`: [1](#0-0) 

The `wait_round_3` waitpoint value is deterministic: it is the 4th sequential waitpoint allocated by `chan.next_waitpoint()` in `do_keyshare`. The channel tag (`root_shared`) is also deterministic. A malicious participant who has participated honestly through rounds 1–4 knows exactly which `MessageHeader` to use.

**Step 2 — Deserialization failure in `Comms::recv`.**

When `recv_from_others` calls `chan.recv(waitpoint)`, the call chain reaches `Comms::recv`: [2](#0-1) 

At line 340, `decoded?` converts a `rmp_serde` deserialization failure into `ProtocolError::Other(...)` via the blanket `From<Box<dyn error::Error + Send + Sync>>` impl: [3](#0-2) 

**Step 3 — Unconditional propagation with no recovery.**

In `recv_from_others`, the `?` at line 20 propagates the error immediately — before `seen.put(from)` is ever reached: [4](#0-3) 

The error then propagates via `?` at the `do_keyshare` call site: [5](#0-4) 

There is no `match`, no retry, no skip-bad-sender logic anywhere in this path. The honest party's DKG future terminates with an error.

---

### Impact Explanation

Permanent denial of key generation for the targeted honest party. The honest party cannot complete `do_keyshare` and will never obtain a `KeygenOutput`. This matches the **High** impact: *Permanent denial of key generation for honest parties under valid protocol inputs and documented trust assumptions.*

The attack requires only 1 malicious participant — well within the documented `MaxFaulty = floor((N-1)/3)` bound under which honest parties are supposed to complete the protocol.

---

### Likelihood Explanation

High. The attack is trivial to execute:
- No cryptographic knowledge is required.
- The malicious participant only needs to participate honestly through rounds 1–4, then send a single malformed private message (e.g., MessagePack-encoded `true`) at `wait_round_3`.
- The `MessageHeader` (channel tag + waitpoint) is fully deterministic and computable by any participant.
- The attack is targeted: the malicious participant sends the bad message only to the victim party, leaving other honest parties unaffected.

---

### Recommendation

`recv_from_others` should not propagate deserialization errors as fatal. Options:

1. **Skip and attribute**: On deserialization failure, record the sender as malicious and skip their message (treating it as if they sent nothing, or as an explicit abort contribution). This requires the caller to handle the missing share.
2. **Return attributed errors**: Change `recv_from_others` to return `Vec<(Participant, Result<T, ProtocolError>)>` so callers can decide per-sender whether to abort or continue.
3. **Identify and abort with attribution**: Return `ProtocolError::MaliciousParticipant(from)` so the caller knows *who* misbehaved, enabling the session to be retried without that participant.

The root cause is that `recv_from_others` conflates "network/deserialization error" with "protocol abort" — there is no distinction between an honest party failing to send and a malicious party sending garbage.

```rust
pub async fn recv_from_others<T>(
    chan: &SharedChannel,
    waitpoint: u64,
    participants: &ParticipantList,
    me: Participant,
) -> Result<Vec<(Participant, T)>, ProtocolError>
where
    T: serde::de::DeserializeOwned,
{
    let mut seen = ParticipantCounter::new(participants);
    seen.put(me);
    let mut messages = Vec::with_capacity(participants.others(me).count());

    while !seen.full() {
        let (from, msg) = chan.recv(waitpoint).await?;
        if seen.put(from) {
            messages.push((from, msg));
        }
    }

    Ok(messages)
}
```
