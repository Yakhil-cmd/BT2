### Title
`set_upgrade_delay_blocks` Takes Effect Immediately, Allowing Owner to Bypass the Upgrade Delay Protection — (File: `engine/src/contract_methods/admin.rs`)

---

### Summary

The `upgrade_delay_blocks` parameter is the only on-chain mechanism giving users advance notice before a contract upgrade. However, `set_upgrade_delay_blocks` takes effect immediately with no timelock of its own. An owner can atomically reduce the delay to zero and then stage and deploy a new contract in the same block, giving users zero time to react and exit.

---

### Finding Description

`stage_upgrade` computes the earliest block at which an upgrade may be deployed as:

```
delay_block_height = env.block_height() + state.upgrade_delay_blocks
``` [1](#0-0) 

`state.upgrade_delay_blocks` is read directly from the current state at the moment `stage_upgrade` is called. There is no snapshot of the delay value at the time it was originally configured.

`set_upgrade_delay_blocks` writes the new value into state immediately, with no waiting period:

```rust
pub fn set_upgrade_delay_blocks<I: IO + Copy, E: Env>(io: I, env: &E) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        ...
        state.upgrade_delay_blocks = args.upgrade_delay_blocks;
        state::set_state(&mut io, &state)?;
        Ok(())
    })
}
``` [2](#0-1) 

The `upgrade` function itself performs no independent check against the staged block height; it only requires `require_owner_only` and `require_running`, then immediately schedules a `DeployContract` promise: [3](#0-2) 

The `EngineState` struct confirms `upgrade_delay_blocks` is a plain mutable field with no shadow or pending-value concept: [4](#0-3) 

Both `set_upgrade_delay_blocks` and `upgrade` are exposed as top-level NEAR contract entry points: [5](#0-4) 

---

### Impact Explanation

All user ETH and ERC-20 balances held inside the Aurora EVM are at risk. A malicious or compromised owner can deploy an arbitrary new contract implementation with zero user-visible delay. The new implementation could:

- Redirect `withdraw` / `engine_withdraw` to drain balances — **Critical: direct theft of user funds**.
- Permanently disable withdrawals — **Critical: permanent fund freeze**.

The `upgrade_delay_blocks` field is the sole on-chain protection users have against surprise upgrades. Bypassing it eliminates that protection entirely.

---

### Likelihood Explanation

The attack requires the owner account to act maliciously or be compromised. On Aurora mainnet the owner is the Aurora Labs multisig. The likelihood of a fully malicious owner is low, but the design flaw means that even an accidental reduction of the delay (e.g., set to 0 for an emergency patch) permanently removes the user-exit window for any upgrade staged in the same block. The absence of any enforcement inside `upgrade` means the delay is advisory only, not a hard guarantee.

---

### Recommendation

Apply a two-step commit-reveal pattern to `set_upgrade_delay_blocks` itself:

1. A first call records the *pending* new delay and the block at which it becomes active (using the *current* `upgrade_delay_blocks` as the waiting period).
2. A second call, only callable after that block, commits the new value.

This mirrors the recommendation in the referenced report: every system-parameter change should require a mandatory time window so users can exit if they do not accept the change.

---

### Proof of Concept

```
Block N:   owner → set_upgrade_delay_blocks(0)
           // state.upgrade_delay_blocks = 0 immediately

Block N:   owner → stage_upgrade(malicious_code)
           // delay_block_height = N + 0 = N  (already reached)

Block N:   owner → upgrade(malicious_code)
           // require_owner_only passes, no delay check, DeployContract fires
           // malicious contract is live; user funds can be stolen or frozen
``` [2](#0-1) [1](#0-0) [3](#0-2)

pub fn stage_upgrade<I: IO + Copy, E: Env>(io: I, env: &E) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        let delay_block_height = env.block_height() + state.upgrade_delay_blocks;
        require_owner_only(&state, &env.predecessor_account_id())?;
        io.read_input_and_store(&storage::bytes_to_key(KeyPrefix::Config, CODE_KEY));
        io.write_storage(
            &storage::bytes_to_key(KeyPrefix::Config, CODE_STAGE_KEY),
            &delay_block_height.to_le_bytes(),
        );
        Ok(())
    })

pub fn upgrade<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<(), ContractError> {
    let state = state::get_state(&io)?;
    require_running(&state)?;
    require_owner_only(&state, &env.predecessor_account_id())?;

    let input = io.read_input().to_vec();
    let (code, state_migration_gas) = match UpgradeParams::try_from_slice(&input) {
        Ok(args) => (
            args.code,
            args.state_migration_gas
                .map_or(GAS_FOR_STATE_MIGRATION, NearGas::new),
        ),
        Err(_) => (input, GAS_FOR_STATE_MIGRATION), // Backward compatibility
    };

    let target_account_id = env.current_account_id();
    let batch = PromiseBatchAction {
        target_account_id,
        actions: vec![
            PromiseAction::DeployContract { code },
            PromiseAction::FunctionCall {
                name: "state_migration".to_string(),
                args: vec![],
                attached_yocto: ZERO_YOCTO,
                gas: state_migration_gas,
            },
        ],
    };
    let promise_id = handler.promise_create_batch(&batch);

    handler.promise_return(promise_id);

    Ok(())
}
