### Title
Stale `target_shard_index` after shard split corrupts `outcome_root_proof` MerklePath in `GetExecutionOutcome` — (`chain/client/src/view_client_actor.rs`)

### Summary

`GetExecutionOutcome` computes `target_shard_index` from the **outcome block's epoch shard layout** (the parent layout), then calls `get_next_block_hash_with_new_chunk` which may cross an epoch boundary and return a **child shard ID** from the new layout. The index is never recomputed against the new layout. The subsequent `merklize(&outcome_roots).1[target_shard_index]` therefore selects the wrong shard's Merkle path, producing an `outcome_root_proof` that fails `verify_path` for any valid receipt whose shard was split between the outcome block and the next block with a new chunk.

### Finding Description

The handler in `view_client_actor.rs` executes the following sequence:

**Step 1 — index computed from parent epoch layout:** [1](#0-0) 

`shard_layout` is the layout of the epoch that contains the outcome block. `target_shard_index` is the parent shard's position in that layout (e.g., index `I`).

**Step 2 — `get_next_block_hash_with_new_chunk` may cross an epoch boundary:** [2](#0-1) 

When the next epoch has a different shard layout, `shard_ids` is expanded to the child shards and `shard_layout` is updated to the new layout internally. The function returns `(h, child_shard_id)` — a child shard ID from the **new** layout.

**Step 3 — `target_shard_index` is never updated; wrong path is selected:** [3](#0-2) 

`target_shard_id` is shadowed by the child shard ID, but `target_shard_index` still holds the parent's index `I`. `outcome_roots` is collected from block `h` in the **new epoch**, which has `N+1` shards (one more than the parent epoch). `merklize(&outcome_roots).1[I]` selects the Merkle path for whatever shard occupies index `I` in the new layout — not the child shard that actually contains the receipt's outcome root.

In `ShardLayoutV2`/`V3`, shard IDs and shard indices are explicitly decoupled: [4](#0-3) 

After a split, the two child shards get new indices `J0` and `J1` in the new layout. Index `I` (the parent's old position) maps to a completely different shard in the new layout.

### Impact Explanation

The `outcome_root_proof` field of `GetExecutionOutcomeResponse` is a `MerklePath` that is supposed to prove `chunk_outcome_root` is a leaf of `block_header_lite.inner_lite.outcome_root`. With the wrong index, the path proves a different shard's outcome root, so:

```
verify_path(block_header_lite.inner_lite.outcome_root, outcome_root_proof, chunk_outcome_root)
```

returns `false` for a valid receipt. Any light client or bridge relying on this RPC to verify receipt execution will reject the proof as invalid. This breaks the client-proof invariant for all receipts whose shard is split between the outcome block and the next block with a new chunk.

### Likelihood Explanation

The condition requires a shard split to occur between the outcome block's epoch and the epoch of the next block that contains a new chunk for the (now-split) shard. Dynamic resharding (`ShardLayoutConfig::Dynamic`) is an active production feature. Once enabled, any receipt executed in the last block of an epoch before a split will trigger this path. No special attacker privilege is required — any user submitting a receipt to the affected shard and then calling `EXPERIMENTAL_light_client_proof` or `GetExecutionOutcome` will receive a broken proof.

### Recommendation

After `get_next_block_hash_with_new_chunk` returns `(h, target_shard_id)`, recompute `target_shard_index` from the shard layout of block `h`'s epoch rather than reusing the stale index from the outcome block's epoch:

```rust
if let Some((h, target_shard_id)) = res {
    outcome_proof.block_hash = h;
    let new_epoch_id = *self.chain.get_block(&h)?.header().epoch_id();
    let new_shard_layout = self.epoch_manager.get_shard_layout(&new_epoch_id).into_chain_error()?;
    let target_shard_index = new_shard_layout
        .get_shard_index(target_shard_id)
        .map_err(Into::into)
        .into_chain_error()?;
    // ... rest unchanged
```

### Proof of Concept

1. Configure a test-loop with dynamic resharding enabled and a low memory threshold so a shard split is scheduled at epoch boundary N→N+1.
2. Submit a receipt to an account in the parent shard; wait for it to be executed in the last block of epoch N (the outcome block).
3. Let the chain advance past the epoch boundary so the split takes effect in epoch N+1.
4. Call `GetExecutionOutcome` with `TransactionOrReceiptId::Receipt { receipt_id, receiver_id }`.
5. Retrieve `block_header_lite` for the returned `outcome_proof.block_hash` via `GetBlockProof`.
6. Assert `verify_path(block_header_lite.inner_lite.outcome_root, outcome_root_proof, chunk_outcome_root)` — this assertion **fails** because `outcome_root_proof` was built with the parent epoch's shard index against the child epoch's `outcome_roots` array.

The existing test in `test-loop-tests/src/tests/light_client.rs` (`check_outcome_proofs`) exercises the non-resharding path and passes; extending it with a dynamic shard split between outcome execution and proof retrieval would reproduce the failure. [5](#0-4)

    // Advance so the outcome's block is final and included in later blocks' merkle roots.

        transaction_hash: outcome.transaction_outcome.id,
        sender_id: account_id.clone(),

        ids.push(TransactionOrReceiptId::Receipt {
            receipt_id: receipt_outcome.id,
            receiver_id: account_id.clone(),
        });

    // The light client head and the block merkle root all proofs are checked against.
