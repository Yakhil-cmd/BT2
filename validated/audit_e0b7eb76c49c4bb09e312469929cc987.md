### Title
Malicious Participant Can Corrupt CKD Output Without Validation - (`src/confidential_key_derivation/protocol.rs`)

### Summary
In the CKD protocol, the coordinator blindly accumulates `CKDOutput` contributions from all participants with no zero-knowledge proof or consistency check. A malicious participant can send arbitrary elliptic curve points for `big_y` and `big_c`, silently corrupting the final derived key. Unlike the DKG protocol — which enforces proofs of knowledge and commitment hashes — the CKD coordinator has no mechanism to detect or reject invalid contributions.

### Finding Description
In `do_ckd_coordinator` (`src/confidential_key_derivation/protocol.rs` lines 50–55), the coordinator receives each participant's `(big_y, big_c)` pair and unconditionally adds it to the running sum:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
```

The correct contribution from participant `i` must satisfy:
- `big_y_i = y_i · G` (random blinding)
- `big_c_i = x_i · H(pk ‖ app_id) + y_i · app_pk` (ElGamal encryption of the secret share contribution)

There is no proof that `big_c_i` is correctly formed relative to the participant's actual secret share `x_i` and their claimed `big_y_i`. Any participant can substitute arbitrary group elements.

Contrast this with `do_keyshare` in `src/dkg.rs`, which enforces:
- A proof of knowledge of the secret coefficient (`verify_proof_of_knowledge`, lines 452–460)
- A commitment hash binding (`verify_commitment_hash`, lines 463–469)
- Per-share Feldman VSS verification (`validate_received_share`, line 522)

None of these safeguards exist in the CKD path.

The `compute_signature_share` function (`src/confidential_key_derivation/protocol.rs` lines 148–182) correctly computes the honest contribution, but nothing prevents a malicious caller from bypassing this function and sending hand-crafted values through the channel.

### Impact Explanation
The final CKD output `(Y, C)` is the sum of all participant contributions. A single malicious participant who sends `(big_y = G, big_c = G)` (or any arbitrary points) shifts both `Y` and `C` by a fixed attacker-controlled offset. The coordinator and all honest parties accept this corrupted output. When the application attempts to unmask the confidential key via `C - Y · app_sk`, the result is `msk · H(pk ‖ app_id) + attacker_offset`, which is an incorrect and unusable derived key. This is a direct corruption of CKD outputs causing honest parties to accept an unusable cryptographic output — **High** impact under the allowed scope.

### Likelihood Explanation
The entry path requires only that the attacker be a legitimate participant in the CKD session (an unprivileged library caller). No special privileges, leaked keys, or external assumptions are needed. The attack is a single-round substitution of two group elements, trivially reachable by any participant who controls their own protocol instance.

### Recommendation
Each participant should accompany their `(big_y, big_c)` contribution with a zero-knowledge proof of correct formation — specifically a proof that `big_c - big_y · app_pk = x_i · H(pk ‖ app_id)` for the same `x_i` committed to during DKG (i.e., a proof of discrete-log equality between `big_c - big_y · app_pk` and the participant's public key share). The coordinator should verify all such proofs before accumulating contributions, mirroring the validation discipline already present in `do_keyshare`.

### Proof of Concept
1. Malicious participant `P_m` participates in a CKD session.
2. Instead of calling `compute_signature_share`, `P_m` constructs `(norm_big_y, norm_big_c) = (G, G)` (generator points).
3. `P_m` sends these via `chan.send_private(waitpoint, coordinator, &(norm_big_y, norm_big_c))`.
4. In `do_ckd_coordinator`, the coordinator executes `norm_big_y += G; norm_big_c += G` with no error.
5. The returned `CKDOutput` is `(Y + G, C + G)` instead of the correct `(Y, C)`.
6. The application calls `ckd_output.unmask(app_sk)`, computing `(C + G) - (Y + G) · app_sk = msk · H(pk ‖ app_id) + G - G · app_sk`, which is not equal to the intended `msk · H(pk ‖ app_id)`.
7. All honest parties accept this corrupted output; the CKD session completes with no error. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

    mut chan: SharedChannel,
    participants: ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,

    let (mut norm_big_y, mut norm_big_c) =
        compute_signature_share(&participants, me, key_pair, app_id, app_pk, rng)?;

    // Receive everyone's inputs and add them together
    let waitpoint = chan.next_waitpoint();

    for (_, participant_output) in
        recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?

        norm_big_c += participant_output.big_c();
    }
    let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
    Ok(Some(ckd_output))

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}
