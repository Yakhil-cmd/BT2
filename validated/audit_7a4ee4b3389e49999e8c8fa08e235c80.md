### Title
Randomness-State Exposure via `revertibleRandom()` Enables Transaction-Abort Manipulation of On-Chain Outcomes - (File: fvm/environment/random_generator.go)

### Summary
The Flow FVM exposes `revertibleRandom()` (Cadence built-in) and the `revertibleRandom` EVM precompile, both of which deliver the current block's random value within the same transaction that consumes it. Because the random value is fully determined before execution begins—seeded from the block's beacon signature (QC) and the transaction ID—an unprivileged transaction sender can observe the output and deliberately abort/panic the transaction if the outcome is unfavorable, then resubmit until a favorable result is obtained. This is the direct Flow analog of the Chainlink VRF misuse described in the external report: randomness is consumed in the same atomic unit that determines the outcome, giving the requester selective control over the result.

### Finding Description
`ReadRandom()` in `fvm/environment/random_generator.go` (lines 80–124) seeds a ChaCha20 CSPRG from two inputs:

1. The current block's beacon signature, obtained via `gen.entropySource.RandomSource()` (line 83), which resolves to `model.BeaconSignature(qc)` in `state/protocol/badger/snapshot.go` (lines 354–364). This QC is produced by consensus nodes and stored before execution begins—it is public and fixed.
2. The transaction ID (`gen.salt`, line 96), which is the hash of the transaction body and is also fixed before execution.

Both inputs are therefore known before the transaction runs. The resulting random stream is deterministic and observable by anyone who knows the block's QC and the transaction ID.

This value is surfaced to callers through two paths:

- **Cadence**: `revertibleRandom<T>()` calls `ReadRandom()` directly. A Cadence transaction can call this, inspect the result, and `panic()` to abort if the value is unfavorable. The abort rolls back all state changes; the sender pays only the gas fee.
- **EVM**: The `revertibleRandom` precompile in `fvm/evm/precompiles/arch.go` (lines 204–217) calls `revertibleRandomGenerator(backend)` in `fvm/evm/handler/precompiles.go` (lines 101–111), which calls `backend.ReadRandom()`. An EVM contract can call this precompile, inspect the result, and `revert()` to abort the EVM sub-transaction. The outer Cadence transaction can also be aborted.

The function is explicitly named "revertible" to acknowledge this property, and the `RandomSourceHistoryProvider` interface in `fvm/environment/history_random_source_provider.go` (lines 15–24) documents that `RandomSourceHistory()` exists specifically "to implement commit-reveal schemes"—confirming that `revertibleRandom()` is not safe for manipulation-resistant selection.

### Impact Explanation
Any on-chain lottery, random winner selection, NFT mint, or asset distribution mechanism that uses `revertibleRandom()` (Cadence or EVM) can be gamed by the transaction sender. The attacker submits a transaction, observes the random output, aborts if unfavorable, and retries. Over enough attempts the attacker can guarantee a favorable outcome. This constitutes unauthorized access to on-chain assets (e.g., winning a lottery, obtaining a rare NFT) without any privileged access. The impact is identical to the external report: the random value is consumed in the same atomic unit that determines the outcome, before any commitment is made.

### Likelihood Explanation
High. The attack requires no special privileges, no staked nodes, and no key compromise. Any account holder can submit Cadence transactions or EVM transactions. The only cost is the transaction fee per aborted attempt. The EVM path is particularly accessible because Solidity contracts can call the `revertibleRandom()` precompile via `staticcall` and revert on an unfavorable result, as shown in `fvm/evm/testutils/contracts/test.sol` (lines 91–96). The block's QC (and thus the random seed) is publicly observable on-chain before execution, making the outcome fully predictable for a determined attacker.

### Recommendation
For any randomness-dependent asset distribution, replace `revertibleRandom()` with the `RandomBeaconHistory` commit-reveal scheme. The correct pattern is:
1. **Commit phase**: The user commits to their choice in block N, recording it on-chain.
2. **Reveal phase**: In a later block N+K (K ≥ 1), the contract reads `RandomBeaconHistory.sourceOfRandomness(N)` to obtain the random source for block N—a value that was unknown at commit time and cannot be influenced by the user.

This is exactly what `RandomSourceHistoryProvider.RandomSourceHistory()` and the `randomSourceProvider` in `fvm/evm/handler/precompiles.go` (lines 62–98) are designed to support. The `getRandomSource(uint64)` EVM precompile exposes this committed historical randomness and should be used instead of `revertibleRandom()` for manipulation-resistant selection.

### Proof of Concept

**Cadence path (unprivileged transaction sender):**
```cadence
transaction {
    execute {
        let r = revertibleRandom<UInt64>()
        // Attacker checks if r maps to a winning outcome
        if r % 100 != 0 {
            panic("not a winner, abort and retry")
        }
        // Only reaches here when r % 100 == 0
        // Attacker claims prize
    }
}
```
The transaction is submitted repeatedly. Each abort costs only a fee. After ~100 attempts on average the attacker obtains the winning outcome.

**EVM path (via `revertibleRandom()` precompile):**
```solidity
function claimPrize() external {
    (bool ok, bytes memory data) = cadenceArch.staticcall(
        abi.encodeWithSignature("revertibleRandom()")
    );
    require(ok, "arch call failed");
    uint64 r = abi.decode(data, (uint64));
    require(r % 100 == 0, "not a winner, revert");
    // transfer prize
}
```
The EVM transaction reverts on unfavorable outcomes. The outer Cadence transaction is retried until the condition is met.

```go
func (gen *randomGenerator) createPRG() (random.Rand, error) {
	// Use the protocol state source of randomness [SoR] for the current block's
	// execution
	source, err := gen.entropySource.RandomSource()
	// `RandomSource` does not error in normal operations.
	// Any error should be treated as an exception.
	if err != nil {
		return nil, fmt.Errorf("reading random source from state failed: %w", err)
	}

	// Use the state/protocol PRG derivation from the source of randomness:
	//  - for the transaction execution case, the PRG used must be a CSPRG
	//  - use the state/protocol/prg customizer defined for the execution environment
	//  - use the salt as an extra diversifier of the CSPRG. Although this
	//    does not add any extra entropy to the output, it allows creating an independent
	//    PRG for each transaction or script.
	csprg, err := prg.New(source, prg.ExecutionEnvironment, gen.salt)
	if err != nil {
		return nil, fmt.Errorf("failed to create a CSPRG from source: %w", err)
	}

	return csprg, nil
}
```

```go
func (gen *randomGenerator) ReadRandom(buf []byte) error {
	defer gen.tracer.StartExtensiveTracingChildSpan(
		trace.FVMEnvRandom).End()

	// PRG creation is only done once.
	if !gen.isPRGCreated {
		newPRG, err := gen.createPRG()
		if err != nil {
			return err
		}
		gen.prg = newPRG
		gen.isPRGCreated = true
	}

	gen.prg.Read(buf)
	return nil
```

```go
func revertibleRandomGenerator(backend backends.Backend) func() (uint64, error) {
	return func() (uint64, error) {
		rand := make([]byte, 8)
		err := backend.ReadRandom(rand)
		if err != nil {
			return 0, err
		}

		return binary.BigEndian.Uint64(rand), nil
	}
}
```
