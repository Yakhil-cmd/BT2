Audit Report

## Title
Unprivileged Caller Can Abort In-Progress SNS Upgrade via `fail_stuck_upgrade_in_progress` - (File: `rs/sns/governance/canister/canister.rs`)

## Summary
The `fail_stuck_upgrade_in_progress` update endpoint in the SNS Governance canister is registered as a plain `#[update]` with no caller authentication. Any unprivileged principal can invoke it once `now > pending_version.mark_failed_at_seconds` (5 minutes after upgrade kick-off) to unconditionally clear `pending_version` and mark the associated upgrade proposal as `ExternalFailure`, even when the actual canister upgrade succeeded. This leaves `deployed_version` permanently stale and disrupts the SNS upgrade pipeline.

## Finding Description
The canister endpoint at `rs/sns/governance/canister/canister.rs` L526–535 is a bare `#[update]` with no `caller()` check:

```rust
#[update]
fn fail_stuck_upgrade_in_progress(
    request: FailStuckUpgradeInProgressRequest,
) -> FailStuckUpgradeInProgressResponse {
    log!(INFO, "fail_stuck_upgrade_in_progress");
    FailStuckUpgradeInProgressResponse::from(governance_mut().fail_stuck_upgrade_in_progress(
        sns_gov_pb::FailStuckUpgradeInProgressRequest::from(request),
    ))
}
``` [1](#0-0) 

It delegates directly to `Governance::fail_stuck_upgrade_in_progress` in `rs/sns/governance/src/governance.rs` L6327–6361, which also performs no caller check. The sole guard is a time check:

```rust
if now > pending_version.mark_failed_at_seconds {
    ...
    self.complete_sns_upgrade_to_next_version(
        pending_version.proposal_id, status, message, None,
    );
}
``` [2](#0-1) 

`complete_sns_upgrade_to_next_version` unconditionally sets `self.proto.pending_version = None` and calls `set_proposal_execution_status` with an `ExternalFailure` error, without updating `deployed_version` (the `deployed_version` argument is `None` in this call path): [3](#0-2) 

The deadline is set to only 5 minutes after upgrade kick-off: [4](#0-3) 

The intended success path is `check_upgrade_status`, an async periodic task that queries root for the running version and calls `complete_sns_upgrade_to_next_version` with `Status::Success` and the new `deployed_version`. This task runs inside `run_periodic_tasks` gated by `acquire_upgrade_periodic_task_lock`: [5](#0-4) 

The comment on `acquire_upgrade_periodic_task_lock` explicitly states the lock blocks re-entry for **10 minutes** per invocation. Since the deadline is only 5 minutes, if the first `check_upgrade_status` run (at upgrade kick-off) does not yet observe the completed upgrade (the actual canister upgrade is asynchronous and may not be reflected immediately), the lock prevents a second check for 10 minutes — well past the 5-minute deadline. During this window, any unprivileged caller can invoke `fail_stuck_upgrade_in_progress` and win the race against the periodic task. [6](#0-5) 

Additionally, `check_upgrade_status` increments `checking_upgrade_lock` before its async inter-canister call and resets it to 0 after. If the call is slow or the canister is under load, subsequent heartbeat invocations see `lock > 1` and return early, further widening the window: [7](#0-6) 

## Impact Explanation
When the attack succeeds:
1. `pending_version` is cleared without updating `deployed_version`, leaving governance with a permanently stale view of the SNS version.
2. The upgrade proposal is marked `ExternalFailure`, so `deployed_version` is never advanced to the new version.
3. Future `UpgradeSnsToNextVersion` proposals call `check_no_upgrades_in_progress`, which passes (since `pending_version` is now `None`), but subsequent upgrade path computation uses the stale `deployed_version`, causing incorrect upgrade steps or re-upgrade attempts of already-upgraded canisters. [8](#0-7) 

This constitutes a **High** severity SNS infrastructure impact: an unprivileged principal can repeatedly disrupt the SNS upgrade pipeline at the cost of a single ingress message per upgrade attempt, causing concrete harm to SNS governance operations.

## Likelihood Explanation
The attack requires no special privileges, tokens, or neurons. The attacker monitors `get_running_sns_version` (a public query) to observe `mark_failed_at_seconds`, then submits a single ingress call immediately after the deadline. The 10-minute periodic task lock creates a reliable window: if the first `check_upgrade_status` run does not confirm success (likely, since the actual canister upgrade is asynchronous), the next run is blocked for 10 minutes while the deadline is only 5 minutes. The attack is cheap, deterministic, and repeatable on every SNS upgrade proposal. [9](#0-8) 

## Recommendation
Add a caller check to `fail_stuck_upgrade_in_progress` in `rs/sns/governance/canister/canister.rs`. The function should only be callable by the SNS governance canister itself (`canister_self()`), the SNS root canister, or via a governance proposal. This is consistent with how other privileged recovery operations in the same file are gated (e.g., `set_mode` checks the swap canister caller at L545). At minimum, reject calls from anonymous or non-governance principals. [10](#0-9) 

## Proof of Concept
1. An SNS upgrade proposal is adopted; governance sets `pending_version.mark_failed_at_seconds = now + 300`.
2. The first `check_upgrade_status` heartbeat runs immediately but the actual canister upgrade is not yet reflected; it returns without confirming success and acquires the 10-minute periodic task lock.
3. The actual canister upgrade completes at T≈60s.
4. At T=301s, `mark_failed_at_seconds` has passed. The periodic task lock is still held (won't expire until T=600s).
5. Attacker queries `get_running_sns_version`, observes `mark_failed_at_seconds` has passed, and sends ingress: `fail_stuck_upgrade_in_progress({})`.
6. `now > mark_failed_at_seconds` → `complete_sns_upgrade_to_next_version(proposal_id, ExternalFailure, ..., None)` → `pending_version = None`, proposal marked failed, `deployed_version` unchanged.
7. At T=600s, `check_upgrade_status` runs again but `pending_version` is `None`, so `should_check_upgrade_status()` returns false and no recovery occurs.
8. All subsequent `UpgradeSnsToNextVersion` proposals compute upgrade paths from the stale `deployed_version`, causing incorrect behavior.

A deterministic integration test using `NativeEnvironment` can reproduce this by: setting `env.now = mark_failed_at_seconds - 1`, calling `run_periodic_tasks` (first check, upgrade not yet visible), advancing time to `mark_failed_at_seconds + 1`, calling `fail_stuck_upgrade_in_progress`, and asserting `pending_version` is `None` while `deployed_version` remains at the old version — matching the existing test pattern in `rs/sns/governance/src/governance/fail_stuck_upgrade_in_progress_tests.rs`. [11](#0-10)

            ));

                    self.proto
                        .pending_version
                        .as_ref()
                        .map(|p| p.mark_failed_at_seconds)
                ),
            ));
        }

        Ok(())
    }

    /// Attempts to acquire the lock over SNS upgrade-related periodic tasks.
    ///
    /// Succeeds if the lock is currently released or was last acquired
    /// over `UPGRADE_PERIODIC_TASK_LOCK_TIMEOUT_SECONDS` ago.
    ///
    /// Returns whether the lock was acquired.
    ///
    /// This function is made public so that it can be called from
    /// rs/sns/governance/tests/governance.rs where we need to disable upgrade-related periodic
    /// tasks while testing a orthogonal SNS features (e.g., disburse maturity).
    pub fn acquire_upgrade_periodic_task_lock(&mut self) -> bool {
        let now = self.env.now();
        match self.upgrade_periodic_task_lock {
            Some(time_acquired)
                if now
                    >= time_acquired
                        .checked_add(UPGRADE_PERIODIC_TASK_LOCK_TIMEOUT_SECONDS)
                        // In case of overflow, we'll unwrap to 0, which should always cause this to evaluate to true
                        .unwrap_or(0) =>
            {
                self.upgrade_periodic_task_lock = Some(now);
                true
            }
            Some(_) => false,
            None => {
                self.upgrade_periodic_task_lock = Some(now);
                true
            }
        }
    }

#[test]
fn test_fails_proposal_and_removes_upgrade_if_upgrade_attempt_is_expired() {
    // Step 1: Prepare the world

    let env = {
        let mut env = NativeEnvironment::new(Some(*TEST_GOVERNANCE_CANISTER_ID));

        // Note that NativeEnvironment only advances time when you tell it
        // to. Therefore, this is the time that Governance will see
        // throughout this test.
        env.now = UPGRADE_DEADLINE_TIMESTAMP_SECONDS + 1;

        env
    };

    let mut governance = Governance::new(
        ValidGovernanceProto::try_from(GOVERNANCE_PROTO.clone()).unwrap(),
        Box::new(env),
        Box::new(DoNothingLedger {}),
        Box::new(DoNothingLedger {}),
        Box::new(FakeCmc::new()),
    );

    // The code being tested is supposed to affect these fields. We
    // inspect them here to make sure that any expected changes are
    // real, not just because the world was (accidentally) already the
    // way we expected them afterwards.
    assert_eq!(
        governance.proto.pending_version.clone().unwrap(),
        PendingVersion {
            target_version: Some(SNS_VERSION_2.clone()),
            mark_failed_at_seconds: UPGRADE_DEADLINE_TIMESTAMP_SECONDS,
            checking_upgrade_lock: 10,
            proposal_id: Some(UPGRADE_PROPOSAL_ID),
        }
    );
    assert_eq!(
        governance.proto.deployed_version.clone().unwrap(),
        SNS_VERSION_1.clone()
    );

    // Step 2: Run the code being tested.
    assert_eq!(
        governance.fail_stuck_upgrade_in_progress(FailStuckUpgradeInProgressRequest {}),
        FailStuckUpgradeInProgressResponse {},
    );

    // Step 3: Inspect results.

    // Assert pending version has been cleared.
    let pending_version = &governance.proto.pending_version;
    assert!(pending_version.is_none(), "{pending_version:#?}");
    // Assert deployed_version unchanged from before.
    assert_eq!(
        governance.proto.deployed_version.clone().unwrap(),
        SNS_VERSION_1.clone()
    );

    // Assert proposal failed
    let proposal = governance.get_proposal(&GetProposal {
        proposal_id: Some(ProposalId {
            id: UPGRADE_PROPOSAL_ID,
        }),
    });
    let proposal_data = match proposal.result.unwrap() {
        get_proposal_response::Result::Error(e) => {
            panic!("Error: {e:?}")
        }
        get_proposal_response::Result::Proposal(proposal) => proposal,
    };
    assert_ne!(proposal_data.failed_timestamp_seconds, 0);

    // Inspect the proposal's failure_reason.
    let governance_error = proposal_data.failure_reason.unwrap();
    assert_eq!(
        ErrorType::try_from(governance_error.error_type),
        Ok(ErrorType::External),
        "{governance_error:#?}",
    );
    assert!(
        governance_error.error_message.contains("manually aborted"),
        "{governance_error:#?}",
    );
}
