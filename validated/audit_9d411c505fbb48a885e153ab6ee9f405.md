Audit Report

## Title
Byzantine Peer Triggers Spurious Peer Fetches via `StrippedBlockProposal` with `value=None` and Non-Empty `ingress_messages` — (`rs/p2p/artifact_downloader/src/fetch_stripped_artifact/types/stripped.rs`)

## Summary

The `TryFrom<pb::StrippedBlockProposal>` implementation uses `is_some_and` to guard against a non-empty ingress payload, but `is_some_and` returns `false` when the outer `Option` is `None`, so a proto with `pruned_block_proposal.value = None` passes the guard unconditionally. The top-level `ingress_messages` field is then populated independently into `stripped_ingress_payload`, producing a structurally accepted `StrippedBlockProposal` with arbitrarily many fake ingress IDs. `assemble_message` spawns one `get_or_fetch` Tokio task per entry — each contacting peers over the transport layer — before ever calling `try_assemble()`, which ultimately fails with `DeserializationFailed` after all fetches complete.

## Finding Description

**Root cause — guard is vacuously true when `value` is `None`** (`stripped.rs` lines 75–83):

```rust
if pruned_block_proposal_proto
    .value
    .as_ref()
    .is_some_and(|block| block.ingress_payload.is_some())
{
    return Err(...);
}
```

`Option::is_some_and` evaluates to `false` when the outer `Option` is `None`, so a proto carrying `pruned_block_proposal.value = None` passes this check unconditionally.

**`ingress_messages` is populated from the proto's top-level field, not from `value`** (`stripped.rs` lines 102–108):

```rust
stripped_ingress_payload: StrippedIngressPayload {
    ingress_messages: value
        .ingress_messages
        .into_iter()
        .map(SignedIngressId::try_from)
        .collect::<Result<Vec<_>, _>>()?,
},
```

A proto with `pruned_block_proposal.value = None` and N entries in `ingress_messages` therefore deserializes successfully into a `StrippedBlockProposal` with N entries in `stripped_ingress_payload.ingress_messages`.

**Fetches are spawned before structural validity is checked** (`assembler.rs` lines 209–226):

`BlockProposalAssembler::new` seeds `self.ingress_messages` from `stripped_ingress_payload.ingress_messages`; `missing_stripped_messages()` returns all N entries; `assemble_message` then spawns N `get_or_fetch` tasks, each of which first checks the local ingress pool and then contacts random peers over the transport layer.

**`try_assemble()` fails after all fetches complete** (`assembler.rs` lines 630–639):

```rust
if let Some(block) = reconstructed_block_proposal_proto.value.as_mut() {
    Self::try_reconstruct_payload(ingress_messages, &mut block.ingress_payload)?;
    ...
}
// value is None → branch not taken
let reconstructed_block_proposal: BlockProposal = reconstructed_block_proposal_proto
    .try_into()
    .map_err(AssemblyError::DeserializationFailed)?;
```

Because `value` is `None`, the reconstruction branch is skipped entirely, and `try_into()` fails. The assembly returns `AssembleResult::Unwanted` — but only after all N peer-fetch tasks have already been awaited.

The bouncer (`assembler.rs` lines 233–242) can abort the loop early if the block becomes unwanted, but it does not prevent the initial burst of N task spawns, and the attacker can rotate across multiple currently-wanted block IDs to sustain the effect.

## Impact Explanation

A Byzantine subnet peer (below the BFT fault threshold) can force the victim replica to spawn up to ~1000 `get_or_fetch` Tokio tasks per wanted block ID (the ingress payload cap noted in the code), each making outbound transport-layer RPCs to random peers. By rotating across multiple concurrently-wanted block IDs and repeating across block heights, the attacker can sustain elevated Tokio task and thread consumption and unnecessary outbound RPC traffic on the victim node. This constitutes an application/platform-level DoS with concrete subnet availability impact, matching the **High** bounty tier.

## Likelihood Explanation

The attacker must be a Byzantine subnet node (below the consensus fault threshold). Such a node has full knowledge of the current consensus state and can trivially identify wanted block IDs. The crafted proto requires no cryptographic material and passes all existing deserialization guards. The exploit is repeatable across block heights and rounds for as long as the attacker remains a subnet member.

## Recommendation

In `TryFrom<pb::StrippedBlockProposal>` (`stripped.rs`), add an explicit consistency check immediately after the existing `is_some_and` guard: if `pruned_block_proposal_proto.value` is `None`, both `ingress_messages` and `stripped_idkg_dealings` must be empty.

```rust
if pruned_block_proposal_proto.value.is_none()
    && (!value.ingress_messages.is_empty() || !value.stripped_idkg_dealings.is_empty())
{
    return Err(ProxyDecodeError::Other(String::from(
        "pruned_block_proposal has no value but stripped lists are non-empty",
    )));
}
```

This enforces the invariant that structural validity of the pruned proto is verified before any peer-fetch work is initiated.

## Proof of Concept

State-machine unit test (no network required):

```rust
// Craft a pb::StrippedBlockProposal with value=None but non-empty ingress_messages
let proto = pb::StrippedBlockProposal {
    pruned_block_proposal: Some(pb::BlockProposal { value: None, ..Default::default() }),
    ingress_messages: vec![/* any valid pb::StrippedIngressMessage */; 100],
    unstripped_consensus_message_id: Some(/* real wanted block id */),
    stripped_idkg_dealings: vec![],
};

// TryFrom succeeds today — this is the bug
let stripped = StrippedBlockProposal::try_from(proto).unwrap();

// missing_stripped_messages returns 100 entries
let assembler = BlockProposalAssembler::new(stripped);
assert_eq!(assembler.missing_stripped_messages().len(), 100);
// → 100 get_or_fetch tasks would be spawned before try_assemble() is ever called
// → try_assemble() then fails with AssemblyError::DeserializationFailed
```
