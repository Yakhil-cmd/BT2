### Title
Threshold-1 Input Causes Permanent Denial of Triple Generation and Signing — (`src/ecdsa/ot_based_ecdsa/triples/generation.rs`)

### Summary

In `do_generation_many`, the `l` polynomial degree is computed as `threshold - 2`. When `threshold = 1` (a valid 1-of-N reconstruction bound accepted by the `ReconstructionLowerBound` type), `checked_sub(2)` returns `None`, propagating `ProtocolError::IntegerOverflow` through every call path that depends on triple generation, permanently blocking signing for all honest parties.

### Finding Description

`ReconstructionLowerBound` is a plain `usize` newtype with no enforced minimum: [1](#0-0) 

Inside `do_generation_many`, two polynomial degrees are derived: [2](#0-1) 

`degree1 = threshold - 1` succeeds for `threshold = 1`, but `degree2 = threshold - 2` underflows and returns `Err(ProtocolError::IntegerOverflow)` immediately, before any network communication occurs. `do_generation` delegates entirely to `do_generation_many`: [3](#0-2) 

No guard enforcing `threshold >= 2` exists anywhere in the triple-generation call chain. The `ReconstructionLowerBound` type itself imposes no lower bound: [4](#0-3) 

The DKG path in `src/dkg.rs` only subtracts 1, so it succeeds for `threshold = 1` and does not catch this gap: [5](#0-4) 

### Impact Explanation

Every OT-based ECDSA signing operation requires a pre-generated Beaver triple. If triple generation always returns `IntegerOverflow` for a given threshold, no triple can ever be produced, and signing is permanently unavailable for all honest participants in that session. This matches:

> **High: Permanent denial of signing for honest parties under valid protocol inputs and documented trust assumptions.**

### Likelihood Explanation

`threshold = 1` is a semantically valid reconstruction bound (any single participant can reconstruct). The wiki states the OT-based scheme "supports N ≥ t participants," which includes t = 1. A library caller who configures a 1-of-N OT-based ECDSA deployment will pass `ReconstructionLowerBound(1)` to triple generation, hit the underflow on every invocation, and be permanently unable to sign. No malicious actor is required; an honest misconfiguration is sufficient.

### Recommendation

Add an explicit lower-bound check at the entry point of `do_generation_many` (or in the public-facing wrapper) before the polynomial degree computations:

```rust
if threshold.value() < 2 {
    return Err(ProtocolError::InvalidInput(
        "OT-based triple generation requires threshold >= 2".to_string(),
    ));
}
```

Alternatively, encode the constraint in a dedicated `TripleGenerationThreshold` newtype that enforces `value >= 2` at construction time, mirroring the recommendation in the external report to return a safe default (0%) rather than reverting.

### Proof of Concept

1. Caller invokes the public triple-generation entry point with `threshold = ReconstructionLowerBound(1)` and any valid participant list.
2. `do_generation_many` reaches line 134–137; `1usize.checked_sub(2)` returns `None`.
3. `ok_or(ProtocolError::IntegerOverflow)?` propagates the error immediately.
4. `do_generation` returns `Err(ProtocolError::IntegerOverflow)`.
5. No triple is ever produced; all subsequent signing attempts fail identically.
6. Honest parties with a legitimately configured 1-of-N OT-based ECDSA setup can never sign. [6](#0-5)
