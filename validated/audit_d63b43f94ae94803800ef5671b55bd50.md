### Title
Malformed Message Body Causes Unrecoverable Protocol Abort via Unauthenticated `recv` Deserialization — (`src/protocol/internal.rs`)

### Summary

`Comms::recv` deserializes the body of the first queued message for a given `MessageHeader` with no retry on failure and no prior sender validation. A single message with a valid 40-byte header prefix but a garbage body causes `rmp_serde` to return an error that propagates unconditionally through the async future and out of `poke()`, permanently terminating the honest party's protocol session. Because `MessageHeader` values are fully deterministic and public, any network-level attacker can target any round of any active session.

---

### Finding Description

**Entry point — `ProtocolExecutor::message` / `Comms::push_message`**

`ProtocolExecutor::message` accepts messages from any `from: Participant` without validating membership in the protocol's participant list: [1](#0-0) 

`push_message` stores the message in the buffer as long as the first 40 bytes parse as a `MessageHeader`; the body is never inspected: [2](#0-1) 

**Deserialization with no retry — `Comms::recv`**

When the protocol future awaits a message, `recv` pops the **first** item from the per-header FIFO queue and immediately attempts `rmp_serde` deserialization. On failure the error is returned with no retry and no skip: [3](#0-2) 

The `decoded?` on line 340 converts the `rmp_serde` error via `From<Box<dyn error::Error + Send + Sync>> for ProtocolError` (→ `ProtocolError::Other`), which is then returned as `Err(...)` from the async future.

**Error propagation through `poke()`**

`ProtocolExecutor::poke` stores the resolved future result and re-raises it on the next call: [4](#0-3) 

Once `poke()` returns `Err(...)`, the protocol is permanently dead — there is no resume path.

**`recv_from_others` propagates before sender validation**

The widely-used helper deserializes first, then checks the sender. The `?` on line 20 aborts the entire collection loop before `seen.put(from)` is ever reached: [5](#0-4) 

**`MessageHeader` is fully predictable**

The shared-channel tag is `SHA256(NEAR_CHANNEL_TAGS_DOMAIN || "root shared")` — a public constant. The waitpoint is a sequential counter starting at 0. Both are deterministic and known to any observer of the protocol specification: [6](#0-5) [7](#0-6) 

Private-channel tags depend only on the two participant IDs (also public): [8](#0-7) 

**Contrast: echo broadcast already handles this correctly**

`reliable_broadcast_receive_all` explicitly swallows `recv` errors and continues the loop (`_ => continue`), showing the developers recognized the need for resilience — but this pattern was not applied to `recv_from_others` or any direct `chan.recv(...).await?` call in DKG, reshare, refresh, presign, sign, or CKD: [9](#0-8) 

---

### Impact Explanation

Any active DKG, reshare, refresh, presign, sign, or CKD session can be permanently aborted for any targeted honest party. The attacker injects one crafted message per target round; the honest party's `poke()` returns `Err(...)` and the session cannot be resumed. Repeated injection across restarts constitutes indefinite denial of all threshold operations for the targeted party.

---

### Likelihood Explanation

The attack requires only the ability to deliver one network message to the target before the legitimate sender does — a race that is trivially winnable by a co-located or network-adjacent malicious participant. The header bytes are fully computable from public information. No cryptographic material is needed.

---

### Recommendation

Apply the same error-resilience pattern already used in `reliable_broadcast_receive_all` to `Comms::recv` and `recv_from_others`:

1. On deserialization failure, **log and skip** the offending message rather than returning `Err`.
2. Validate `from` against the expected participant set **before** attempting deserialization, so messages from unknown senders are dropped at `push_message` time (requires passing the participant list into `Comms`).
3. Alternatively, wrap `chan.recv(...).await` call-sites in protocol loops with `match ... { Ok(v) => v, Err(_) => continue }` consistently, as is already done in echo broadcast.

---

### Proof of Concept

```rust
// Attacker knows: NEAR_CHANNEL_TAGS_DOMAIN, "root shared", waitpoint=0 for round 1
let channel_tag = Sha256::digest([NEAR_CHANNEL_TAGS_DOMAIN, b"root shared"].concat());
let waitpoint: u64 = 0;

let mut header_bytes = [0u8; 40];
header_bytes[..32].copy_from_slice(&channel_tag);
header_bytes[32..].copy_from_slice(&waitpoint.to_le_bytes());

// Garbage body — any bytes that are not valid rmp_serde for the expected type
let mut crafted = header_bytes.to_vec();
crafted.extend_from_slice(&[0xFF, 0xFE, 0xFD]); // invalid msgpack

// Inject before the honest participant's message arrives
honest_party_protocol.message(attacker_participant, crafted);

// Next poke() returns Err(ProtocolError::Other("...rmp_serde error..."))
// Protocol is permanently aborted
assert!(matches!(honest_party_protocol.poke(), Err(_)));
```

        &self,
        header: MessageHeader,
    ) -> Result<(Participant, T), ProtocolError> {

        let message_data = data.get(MessageHeader::LEN..).ok_or_else(|| {

        Ok((from, decoded?))
    }
