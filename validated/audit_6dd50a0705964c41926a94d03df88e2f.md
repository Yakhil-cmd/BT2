### Title
Nonce-less `allowPushers` Signature Enables Creator to Replay Pusher Delegation After Revocation ŌĆö (File: `smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`CompressedOracleV1.allowPushers` verifies an EIP-191 pusher-consent signature that commits to `(chainid, address(this), deadline, pusher, creator)` but contains **no nonce**. The deadline limits the replay window but does not prevent the creator from re-submitting the same signature multiple times within that window. After a pusher calls `revokePusher()`, the creator can immediately replay the original signature to re-establish the delegation, rendering the revocation ineffective until the deadline expires.

---

### Finding Description

`allowPushers` constructs and verifies the signed digest as:

```solidity
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
);
require(pusher == ECDSA.recover(hash, signatures[i]));
namespaceRemapping[pusher] = msg.sender;
``` [1](#0-0) 

The signed tuple `(chainid, oracle, deadline, pusher, creator)` is **identical** on every call that uses the same deadline. There is no per-pusher nonce, no used-signature bitmap, and no state change that invalidates the signature after first use.

`revokePusher` clears the mapping:

```solidity
namespaceRemapping[msg.sender] = address(0);
``` [2](#0-1) 

But because the signature is still valid (deadline not yet elapsed), the creator can immediately call `allowPushers` again with the same `(deadline, [pusher], [sig])` arguments, writing `namespaceRemapping[pusher] = creator` back into storage. The revocation is silently undone.

The code comment itself acknowledges the concern but treats the deadline as sufficient mitigation:

> *"The deadline is likewise required: the signed consent carries no timestamp of its own, so an undated signature could re-establish a delegation AFTER the pusher revoked it."* [3](#0-2) 

The deadline prevents replay *after* expiry but does nothing to prevent replay *within* the deadline window ŌĆö which is exactly the window in which a revoked pusher expects their revocation to be honoured.

The `fallback` push path resolves the namespace at call time:

```solidity
address creator = namespaceRemapping[msg.sender];
if (creator == address(0)) creator = msg.sender;
``` [4](#0-3) 

So every push the revoked-but-replayed pusher makes continues to land in the **creator's namespace** rather than the pusher's own namespace.

---

### Impact Explanation

After the creator replays the delegation:

1. **Pusher's own namespace receives no updates.** All of the pusher's price pushes are silently redirected to the creator's namespace. The pusher's own feeds remain at their last pre-revocation values.
2. **Stale oracle data on pusher-backed feeds.** Any `IPriceProvider` / pool that reads from the pusher's namespace will see a timestamp that no longer advances. The pool's staleness guard (`maxTimeDelta` / `maxRefStaleness`) will eventually reject the price, causing every swap on that pool to revert.
3. **Creator's namespace receives unsolicited price data.** If the creator's namespace is used by a different pool, that pool now receives the pusher's price stream ŌĆö potentially for a different asset pair ŌĆö corrupting its oracle quotes and enabling bad-price execution.

Both outcomes ŌĆö stale oracle causing swap DoS and wrong-asset price injected into a live pool ŌĆö fall within the allowed impact gate: *broken core pool functionality / bad-price execution reaching a pool swap*.

---

### Likelihood Explanation

- The pusher must have previously signed a delegation consent (a normal operational step).
- The creator retains the signature (they submitted it in the first `allowPushers` call).
- The creator needs only to call `allowPushers` again with the same arguments ŌĆö a single, zero-cost transaction ŌĆö any time before the deadline.
- No special privileges, no front-running, no external conditions beyond the deadline window.

Likelihood is **medium**: it requires a prior delegation and a creator willing to act adversarially, but the mechanical execution is trivial.

---

### Recommendation

Add a per-pusher revocation nonce to the signed message and increment it on every successful revocation:

```solidity
mapping(address => uint256) public pusherNonce;

// In allowPushers:
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(
        block.chainid, address(this), deadline,
        pusher, msg.sender,
        pusherNonce[pusher]   // <-- add nonce
    ))
);

// In revokePusher / removePushers:
pusherNonce[msg.sender]++;   // invalidate all outstanding signatures
```

This ensures that once a pusher revokes, every previously issued consent signature becomes cryptographically invalid regardless of its deadline.
