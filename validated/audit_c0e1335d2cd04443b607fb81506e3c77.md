### Title
Gateway `skip_stateful_validations` Admits Unsigned Invoke Transactions for Any Account with a Pending Deploy-Account in the Mempool - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function bypasses the `__validate__` entry-point check for invoke transactions with `tx_nonce == 1` when `account_nonce == 0`, relying on `account_tx_in_pool_or_recent_block` to confirm a deploy-account exists. That helper returns `true` for **any** transaction from the account address, not specifically a deploy-account transaction. An unprivileged attacker who observes a victim's deploy-account transaction in the mempool can submit a malicious invoke transaction (nonce=1, arbitrary calldata, invalid signature) for the victim's account, have it admitted to the mempool without signature validation, and—if the attacker's tip exceeds the victim's—replace the victim's legitimate invoke transaction via fee escalation.

### Finding Description

`skip_stateful_validations` (lines 429–461) applies the following logic:

```rust
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
    return mempool_client
        .account_tx_in_pool_or_recent_block(tx.sender_address())
        .await ...;
}
``` [1](#0-0) 

When this returns `true`, `run_validate_entry_point` is called with `skip_validate = true`, which sets `validate: false` in `ExecutionFlags`, causing `perform_validations` to return `Ok(())` without ever calling `__validate__`: [2](#0-1) 

The mempool helper `account_tx_in_pool_or_recent_block` checks:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [3](#0-2) 

This returns `true` for **any** transaction from the address—including the victim's own deploy-account transaction. The code comment acknowledges the intent ("either it has a deploy_account transaction or transactions with future nonces that passed validations"), but the second branch is circular: those future-nonce transactions may themselves have been admitted via the same skip path.

The gateway's `run_pre_validation_checks` calls `validate_by_mempool` (nonce + fee-escalation check only, no signature) **before** `skip_stateful_validations`: [4](#0-3) 

So the attacker's transaction passes mempool nonce/fee validation, then skips `__validate__` entirely at the gateway.

**Attack path:**

1. Victim submits `deploy_account` (nonce=0) + `invoke` (nonce=1, tip=T).
2. `deploy_account` is admitted; `account_tx_in_pool_or_recent_block` now returns `true` for the victim's address.
3. Attacker submits a malicious `invoke` (nonce=1, arbitrary calldata, invalid signature, tip=T+1) for the victim's address.
4. Gateway: `account_tx_in_pool_or_recent_block` → `true` → `skip_validate = true` → `__validate__` is **not called** → transaction admitted.
5. Mempool fee-escalation: attacker's tip > victim's tip → victim's invoke is **replaced and dropped**.
6. Batcher executes `deploy_account` (succeeds), then attacker's `invoke` (fails `__validate__` during block execution with wrong signature → rejected).
7. Victim's invoke is permanently gone from the mempool; victim must resubmit.

### Impact Explanation

The gateway admits an invalid transaction—one with an arbitrary/invalid signature—to the mempool. This directly matches the **High** impact criterion: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."* The victim's legitimate invoke transaction can be displaced from the mempool, forcing a resubmission and potentially causing the victim's nonce-1 slot to be consumed by a failing transaction in the block.

### Likelihood Explanation

The attack is unprivileged: any observer of the public mempool can execute it. The only cost is a marginally higher tip than the victim's. The window is narrow (nonce=1, account_nonce=0) but is precisely the UX flow the feature is designed to support, making it a predictable target. Automated bots monitoring for deploy-account transactions could exploit this systematically.

### Recommendation

Replace the coarse `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a **deploy-account** transaction exists for the address. This could be a new mempool API `deploy_account_tx_in_pool(address)` that inspects only `InternalRpcTransactionWithoutTxHash::DeployAccount` entries. Alternatively, store the deploy-account transaction hash alongside the account state in the mempool and expose it for this query.

### Proof of Concept

```
1. Victim:   submit deploy_account(nonce=0, tip=10) → admitted, account enters mempool pool
2. Victim:   submit invoke(nonce=1, calldata=X, sig=valid, tip=10) → skip_validate=true, admitted
3. Attacker: submit invoke(nonce=1, calldata=drain_funds, sig=GARBAGE, tip=11)
             → validate_by_mempool: nonce OK, tip 11 > 10 → fee escalation replaces victim's tx
             → skip_stateful_validations: account_tx_in_pool_or_recent_block=true → skip_validate=true
             → __validate__ NOT called at gateway → admitted
4. Mempool:  victim's invoke(nonce=1) is evicted; attacker's invoke(nonce=1) is queued
5. Batcher:  executes deploy_account → success (account deployed)
             executes attacker's invoke → __validate__ called → FAILS (wrong sig) → rejected
6. Result:   victim's invoke is gone; victim must resubmit nonce=1
```
