### Title
Missing Refund Callback in `ExitToEthereum` Causes Permanent Loss of Burned ERC-20 Tokens on NEP-141 Withdrawal Failure — (`engine-precompiles/src/native.rs`)

---

### Summary

The `ExitToEthereum` precompile unconditionally emits only `PromiseArgs::Create` for the downstream NEP-141 `withdraw` call, with no error-handling callback. If that promise fails, the ERC-20 tokens already burned in the same EVM transaction are permanently unrecoverable. The `ExitToNear` precompile has an explicit refund callback (`exit_to_near_precompile_callback`) for exactly this scenario; `ExitToEthereum` has no equivalent.

---

### Finding Description

**Step 1 — ERC-20 burn is unconditional.**

`EvmErc20V2.withdrawToEthereum` burns the caller's tokens before invoking the precompile:

```solidity
function withdrawToEthereum(address recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);   // ← tokens destroyed here
    ...
    assembly { let res := call(..., 0xb0bd02f6..., ...) }
}
``` [1](#0-0) 

**Step 2 — Precompile schedules only a bare `Create` promise, no callback.**

`ExitToEthereum::run` builds a single `PromiseArgs::Create(withdraw_promise)` targeting the NEP-141 `withdraw` method and returns it as a log. There is no `PromiseArgs::Callback` wrapping it, and no `#[cfg(feature = "error_refund")]` branch anywhere in the `ExitToEthereum` implementation:

```rust
let promise = borsh::to_vec(&PromiseArgs::Create(withdraw_promise)).unwrap();
``` [2](#0-1) 

**Step 3 — Contrast with `ExitToNear`, which has a full refund path.**

`ExitToNear::run` conditionally wraps its promise in a `PromiseArgs::Callback` pointing to `exit_to_near_precompile_callback` on the engine itself:

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs {
        base: transfer_promise,
        callback: PromiseCreateArgs {
            target_account_id: self.current_account_id.clone(),
            method: "exit_to_near_precompile_callback".to_string(),
            ...
        },
    })
};
``` [3](#0-2) 

That callback calls `engine::refund_on_error` to re-mint burned tokens when the downstream promise fails:

```rust
} else if let Some(args) = args.refund {
    let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;
``` [4](#0-3) 

**Step 4 — `filter_promises_from_logs` would handle a `Callback` variant correctly** if one were present, but `ExitToEthereum` never produces one: [5](#0-4) 

**Step 5 — The NEP-141 `withdraw` can realistically fail.**

The eth-connector exposes a pausability gate (`pa_pause_feature` / `engine_withdraw`). A paused connector, a connector with insufficient internal balance, or any transient NEAR-side error causes the `withdraw` promise to fail. The existing test suite explicitly covers the pause scenario: [6](#0-5) 

When that happens under `ExitToEthereum`, the ERC-20 tokens are already gone with no recovery path.

---

### Impact Explanation

The burned ERC-20 tokens represent real value bridged from Ethereum. When the NEP-141 `withdraw` fails, those tokens are destroyed in the EVM with no corresponding unlock on Ethereum and no re-mint in the EVM. The user's funds are permanently frozen/destroyed. This meets the **Critical — Permanent freezing of funds** criterion.

---

### Likelihood Explanation

The eth-connector's `withdraw` method is pausable by the connector owner. Any pause window during which a user calls `withdrawToEthereum` triggers the loss. Additionally, any NEAR-side gas exhaustion, connector upgrade, or balance inconsistency can cause the same failure. The path is reachable by any user holding bridged ERC-20 tokens.

---

### Recommendation

Mirror the `ExitToNear` pattern for `ExitToEthereum`:

1. Add a `#[cfg(feature = "error_refund")]` branch in `ExitToEthereum::run` that wraps the `withdraw_promise` in a `PromiseArgs::Callback` targeting a new `exit_to_ethereum_precompile_callback` method on the engine.
2. Implement `exit_to_ethereum_precompile_callback` to call `engine::refund_on_error` (re-minting the burned ERC-20 amount to the original sender) when `handler.promise_result(0)` is not `Successful`.
3. Pass `erc20_address` and `recipient_address` through `RefundCallArgs` so the callback knows which token to re-mint and to whom.

---

### Proof of Concept

```rust
// Integration test sketch (local sandbox, no mainnet)
// 1. Deploy Aurora + NEP-141 + bridge ERC-20
// 2. Pause the eth-connector's withdraw feature
// 3. Call EvmErc20V2.withdrawToEthereum(recipient, amount) — tokens are burned
// 4. The NEP-141 withdraw promise fails (connector paused)
// 5. Assert: erc20.balanceOf(user) == 0  (tokens not restored)
// 6. Assert: no refund mint event emitted
// 7. Assert: eth-connector total supply unchanged (no unlock on Ethereum side)
// → user has lost `amount` tokens permanently
```

The existing `test_exit_to_near_refund` test in `engine-tests/src/tests/erc20_connector.rs` (lines 623–665) demonstrates the analogous refund working for `ExitToNear`; the same test adapted for `withdrawToEthereum` would show the refund does **not** occur. [7](#0-6)
