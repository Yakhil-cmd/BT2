### Title
Malicious CKD Participant Sends Unverified Contribution to Corrupt Coordinator Output - (File: `src/confidential_key_derivation/protocol.rs`)

### Summary
The CKD coordinator aggregates `(norm_big_y, norm_big_c)` pairs from every participant by simple addition, with no zero-knowledge proof or consistency check tying each pair to the sender's public key share. Any single malicious participant can substitute arbitrary group elements, silently biasing the final `CKDOutput` so that every honest party who relies on it receives a cryptographically wrong confidential key — an exact structural parallel to the ERC-4626 "donation inflates the ratio" class, where unvalidated external input poisons a shared accumulator that all subsequent consumers trust.

### Finding Description

**Root cause — no verification of participant contributions**

In `do_ckd_coordinator` the coordinator loops over every peer's message and unconditionally adds the received elements:

```rust
// src/confidential_key_derivation/protocol.rs  lines 50-55
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
```

The honest computation each participant is supposed to perform is (lines 159–180):

```
y  ← random scalar
Y  = y · G₁
S  = x_i · H(pk ∥ app_id)          // x_i = private share
C  = S + y · A                      // A = app public key
(norm_Y, norm_C) = (λ_i · Y, λ_i · C)
```

The coordinator has no way to verify that the received `(norm_Y, norm_C)` satisfies this relation, because:
- It does not hold the individual public key shares `X_i = x_i · G₂`.
- No Sigma/Chaum-Pedersen proof is attached to the message.
- No echo-broadcast or commitment-then-reveal round is used for these values.

A malicious participant simply replaces their honest output with any two group elements `(Y′, C′)` of their choice before calling `chan.send_private`.

**Exploit flow**

1. Malicious participant `M` holds a legitimate key share and participates in a CKD session.
2. Instead of computing the correct `(λ_M · y · G₁, λ_M · (x_M · H + y · A))`, `M` sends `(Y′, C′)` — e.g., the generator point for both, or a point that cancels the honest aggregate.
3. The coordinator accumulates the tampered values alongside all honest contributions and emits `CKDOutput { big_y: Σ, big_c: Σ }` (lines 56-57).
4. The TEE application unmasks with `ckd_output.unmask(app_sk)` (test line 273), computing `C_total − a · Y_total`. Because `M`'s contribution is wrong, the result is `msk · H(pk ∥ app_id) + (C′ − a · Y′)` — a value that is neither the correct confidential key nor detectable as wrong by any honest party.

### Impact Explanation

The corrupted `CKDOutput` is accepted by the coordinator and returned as the protocol result. Every downstream consumer (TEE application) derives a wrong confidential key. Depending on `M`'s choice of `(Y′, C′)`:

- **Denial of correct key**: the TEE cannot decrypt data encrypted under the real key.
- **Targeted bias**: `M` can choose `(Y′, C′)` such that the resulting "key" is a value `M` already knows (e.g., `C′ = a · Y′` makes the unmasked result equal to `msk · H` plus a known offset), potentially allowing `M` to predict or control the derived secret.

This maps directly to the allowed High impact: **Corruption of CKD outputs so honest parties accept unusable or inconsistent cryptographic outputs**.

### Likelihood Explanation

Any single participant in a CKD session can trigger this. The attacker needs only a valid key share (obtained through normal DKG) and the ability to deviate from the protocol when sending their private message to the coordinator. No privileged access, no leaked secrets, and no external assumptions are required. The attack is silent — the coordinator emits `Ok(Some(ckd_output))` with no error.

### Recommendation

Require each participant to attach a non-interactive zero-knowledge proof (e.g., a Chaum-Pedersen / Sigma proof) demonstrating that the submitted `(norm_Y, norm_C)` is consistent with their public key share `X_i`:

- Prove knowledge of `y` such that `Y = y · G₁` **and** `C − X_i · H(pk ∥ app_id) = y · A` (a standard DLEQ proof over two bases `G₁` and `A`).
- The coordinator verifies each proof before adding the contribution; any failing proof aborts the session and identifies the malicious participant.

Alternatively, use a commit-then-reveal round (analogous to the DKG's `commitment_hash` round) so that participants cannot adaptively choose `(Y′, C′)` after seeing others' values.

### Proof of Concept

```
Setup:
  participants = [P1 (honest), P2 (honest), M (malicious)]
  coordinator  = P1
  app_id, app_pk = normal application parameters
  All three hold valid key shares from a prior DKG.

Attack:
  M intercepts its own send_private call and substitutes:
    norm_big_y = ElementG1::generator()   // arbitrary, not λ_M · y · G
    norm_big_c = ElementG1::generator()   // arbitrary, not λ_M · C

Coordinator accumulates:
  total_Y = (λ_P1·y_P1·G + λ_P2·y_P2·G) + G          // G injected by M
  total_C = (msk·H + (λ_P1·y_P1+λ_P2·y_P2)·A) + G    // G injected by M

TEE unmasks:
  total_C − app_sk · total_Y
  = msk·H + (λ_P1·y_P1+λ_P2·y_P2)·A + G
    − app_sk·(λ_P1·y_P1·G + λ_P2·y_P2·G + G)
  = msk·H + G − app_sk·G          // ≠ msk·H(pk‖app_id)

Expected: msk · H(pk ‖ app_id)
Actual:   msk · H(pk ‖ app_id) + (1 − app_sk) · G   ← wrong, undetected
```

The coordinator returns `Ok(Some(corrupted_output))` with no error; the honest parties have no mechanism to detect the deviation.
