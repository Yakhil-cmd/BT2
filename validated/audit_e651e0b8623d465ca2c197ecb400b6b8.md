### Title
Missing Zero-Address Validation for `reward_address` Allows Permanent Freezing of Unclaimed Yield - (File: src/staking/staking.cairo, src/pool/pool.cairo)

### Summary
The `stake`, `change_reward_address` (staking contract), `enter_delegation_pool`, and `change_reward_address` (pool contract) functions accept a `reward_address` parameter without validating it is non-zero. If a staker or pool member supplies or later changes their `reward_address` to the zero address, all accrued yield is routed to address `0x0` — either burned permanently (if the ERC20 allows it) or causing every `claim_rewards` / `unstake_action` call to revert, permanently freezing unclaimed yield.

### Finding Description
The staking contract validates that `reward_address` is not a registered token address (`REWARD_ADDRESS_IS_TOKEN`), but performs no check that it is non-zero:

**`stake` in `staking.cairo`** (lines 307–311):
```cairo
assert!(
    !self.does_token_exist(token_address: reward_address),
    "{}",
    GenericError::REWARD_ADDRESS_IS_TOKEN,
);
```
No `reward_address.is_non_zero()` guard exists.

**`change_reward_address` in `staking.cairo`** (lines 520–524):
```cairo
assert!(
    !self.does_token_exist(token_address: reward_address),
    "{}",
    GenericError::REWARD_ADDRESS_IS_TOKEN,
);
```
Same omission.

**`enter_delegation_pool` in `pool.cairo`** (lines 194–195):
```cairo
assert!(token_address != pool_member, "{}", Error::POOL_MEMBER_IS_TOKEN);
assert!(token_address != reward_address, "{}", GenericError::REWARD_ADDRESS_IS_TOKEN);
```
No zero-address guard.

**`change_reward_address` in `pool.cairo`** (lines 506–510):
```cairo
assert!(
    self.token_dispatcher.contract_address.read() != reward_address,
    "{}",
    GenericError::REWARD_ADDRESS_IS_TOKEN,
);
```
No zero-address guard.

Once `reward_address = 0x0` is stored, every reward-disbursement path routes funds there:

- `send_rewards_to_staker` in `staking.cairo` (line 1625): `token_dispatcher.checked_transfer(recipient: reward_address, amount: amount.into())`
- `claim_rewards` in `pool.cairo` (line 366): `reward_token.checked_transfer(recipient: reward_address, amount: rewards.into())`
- `unstake_action` in `staking.cairo` (line 495): calls `send_rewards_to_staker` before returning principal

The protocol already defines `ZERO_ADDRESS` as a named error (spec line 2822) and enforces `assert_caller_is_not_zero` for callers, but never applies the equivalent guard to the `reward_address` input.

### Impact Explanation
A staker or pool member who supplies `reward_address = 0x0` (intentionally or due to a frontend/wallet bug) will have all accrued STRK yield permanently lost:

- If the underlying ERC20 allows transfer to `0x0`: rewards are burned — **permanent freezing of unclaimed yield** (High).
- If the ERC20 reverts on transfer to `0x0`: `claim_rewards` and `unstake_action` both revert, blocking principal withdrawal until `change_reward_address` is called — **temporary freezing of funds** (High).

The `unstake_action` path is particularly severe: it calls `send_rewards_to_staker` before returning the staker's principal, so a zero `reward_address` blocks the entire exit flow.

### Likelihood Explanation
Likelihood is **Low**. The staker or pool member must supply `0x0` themselves — either through a deliberate act or a frontend/wallet defect that zero-initialises the field. No third party can force this on another user. However, the absence of a guard that the protocol already applies to callers (and to token addresses) makes accidental misconfiguration plausible.

### Recommendation
Add a non-zero assertion for `reward_address` in all four entry points, mirroring the existing `assert_caller_is_not_zero` pattern:

```cairo
assert!(reward_address.is_non_zero(), "{}", GenericError::ZERO_ADDRESS);
```

Apply this in:
- `stake` — `src/staking/staking.cairo` (after line 311)
- `change_reward_address` — `src/staking/staking.cairo` (after line 524)
- `enter_delegation_pool` — `src/pool/pool.cairo` (after line 195)
- `change_reward_address` — `src/pool/pool.cairo` (after line 510)

### Proof of Concept

1. Deploy the staking system normally.
2. Call `stake(reward_address: 0x0, operational_address: <valid>, amount: <min_stake>)` — succeeds because only the token-address check is performed.
3. Advance epochs so rewards accrue.
4. Call `claim_rewards(staker_address)` — `send_rewards_to_staker` attempts `checked_transfer(recipient: 0x0, ...)`, either burning rewards or reverting.
5. Call `unstake_intent()`, wait for exit window, then call `unstake_action(staker_address)` — same `send_rewards_to_staker` call blocks the entire unstake, temporarily freezing the principal.

The same sequence applies to a pool member via `enter_delegation_pool(reward_address: 0x0, amount: ...)` followed by `claim_rewards`. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5) [7](#0-6) [8](#0-7)
