### Title
Pool Admin Bypasses Fee Cap via Unchecked `setPoolBinAdditionalFees` — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary

`MetricOmmPoolFactory.setPoolBinAdditionalFees` forwards `addFeeBuyE6` and `addFeeSellE6` directly to the pool with no upper-bound validation. The pool's `setBinAdditionalFees` also applies no cap. A pool admin can therefore set per-bin additional fees up to `uint16` max (65 535 in E6 = 6.5535%) on any bin, stacking them on top of the already-capped global admin spread fee and pushing the effective per-bin swap fee above the hard cap the factory is supposed to enforce.

### Finding Description

`MetricOmmPoolFactory` enforces fee caps in two places:

- `setPoolAdminFees` checks `newAdminSpreadFeeE6 > maxAdminSpreadFeeE6` and reverts.
- `setFeeCaps` / `createPool` validate against `HARD_MAX_SPREAD_FEE_E6 = 200_000` (20%).

However, `setPoolBinAdditionalFees` contains no such check:

```solidity
// MetricOmmPoolFactory.sol L450-456
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
``` [1](#0-0) 

The pool's `setBinAdditionalFees` also applies no cap — it only validates the bin index:

```solidity
// MetricOmmPool.sol L464-474
function setBinAdditionalFees(int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external onlyFactory nonReentrant(PoolActions.SET_BIN_ADDITIONAL_FEES)
{
    if (bin < LOWEST_BIN || bin > HIGHEST_BIN) revert InvalidBinIndex(bin);
    BinState storage s = _binStates[bin];
    s.addFeeBuyE6 = addFeeBuyE6;
    s.addFeeSellE6 = addFeeSellE6;
    emit BinAdditionalFeesUpdated(bin, addFeeBuyE6, addFeeSellE6);
}
``` [2](#0-1) 

During a swap the per-bin additional fee is added directly on top of the base spread fee:

```solidity
uint256 buyFeeX64 = baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6);
``` [3](#0-2) 

The global admin spread fee is capped at `maxAdminSpreadFeeE6` (≤ `HARD_MAX_SPREAD_FEE_E6 = 200_000`): [4](#0-3) 

But `addFeeBuyE6` / `addFeeSellE6` are `uint16` (max 65 535 in E6 = 6.5535%) and are never checked against any cap. A pool admin who has already set `adminSpreadFeeE6 = maxAdminSpreadFeeE6` can additionally set `addFeeBuyE6 = 65535` on every bin, pushing the effective per-bin fee to `maxAdminSpreadFeeE6 + 6.5535%` — well above the hard cap.

### Impact Explanation

Traders executing swaps in affected bins pay fees exceeding the protocol's hard cap. The excess fee is collected into the pool's accrued spread balance and distributed to the admin at the next `collectFees` call. This is a direct, quantifiable loss of user principal (the excess fee amount) that the fee-cap system was designed to prevent.

### Likelihood Explanation

The trigger is a single call by the pool admin — a role that is semi-trusted only within the cap system. No timelock, no co-signer, and no off-chain precondition is required. The admin can call `setPoolBinAdditionalFees` at any time on any bin they control.

### Recommendation

Add an upper-bound check in `MetricOmmPoolFactory.setPoolBinAdditionalFees` before forwarding to the pool:

```solidity
if (uint256(addFeeBuyE6) + uint256(poolFeeConfig[pool].adminSpreadFeeE6) > maxAdminSpreadFeeE6)
    revert AdminFeeTooHigh();
if (uint256(addFeeSellE6) + uint256(poolFeeConfig[pool].adminSpreadFeeE6) > maxAdminSpreadFeeE6)
    revert AdminFeeTooHigh();
```

Alternatively, introduce a dedicated `maxBinAdditionalFeeE6` cap and enforce it here. The pool-level `setBinAdditionalFees` should remain a pass-through (factory is the authority), so the check belongs in the factory.

### Proof of Concept

1. Factory owner sets `maxAdminSpreadFeeE6 = 200_000` (20%).
2. Pool admin calls `setPoolAdminFees(pool, 200_000, 0)` — accepted, at the cap.
3. Pool admin calls `setPoolBinAdditionalFees(pool, 0, 65535, 65535)` — accepted with no revert.
4. A trader swaps through bin 0. Effective buy fee = `200_000/1e6 + 65535/1e6 = 26.5535%` — 6.5535 percentage points above the hard cap.
5. The excess fee accrues in the pool and is swept to the admin's fee destination on the next `collectPoolFees` call. [5](#0-4) [6](#0-5)
