### Title
Hardcoded 2,300-Gas Stipend in `NewDepositCall` Blocks Native FLOW Delivery to Smart Contracts - (`File: fvm/evm/types/call.go`)

### Summary

`NewDepositCall` constructs every Cadence-to-EVM native FLOW deposit with a fixed `GasLimit` of `21,000 + 2,300 = 23,300`. After the 21,000-unit intrinsic cost is consumed, only 2,300 gas remains for the recipient's `receive()`/`fallback()` function — the same stipend that Solidity's deprecated `transfer` primitive forwards. Any EVM smart contract whose `receive()`/`fallback()` requires more than 2,300 gas will have the deposit reverted, and the emulator's `mintTo` path resets state, so the tokens are never credited. The Cadence transaction then panics via `panicOnErrorOrInvalidOrFailedState`, making the deposit permanently undeliverable through the standard bridge path.

### Finding Description

`DefaultGasLimitForTokenTransfer` is defined as:

```go
IntrinsicFeeForTokenTransfer = gethParams.TxGas          // 21_000
DefaultGasLimitForTokenTransfer = IntrinsicFeeForTokenTransfer + 2_300  // 23_300
DepositCallGasLimit = DefaultGasLimitForTokenTransfer
``` [1](#0-0) 

`NewDepositCall` hard-wires this limit into every bridge deposit:

```go
return &DirectCall{
    ...
    GasLimit: DepositCallGasLimit,   // always 23_300
}
``` [2](#0-1) 

The emulator's `mintTo` procedure runs the call and, if the result is `Failed()` (i.e., the recipient's `receive()`/`fallback()` ran out of gas), it resets all state changes and returns the failed result without crediting any balance:

```go
if res.Invalid() || res.Failed() {
    proc.state.Reset()
    return res, nil
}
``` [3](#0-2) 

Back in the handler, `panicOnErrorOrInvalidOrFailedState` converts the failed result into a Cadence-level panic:

```go
if res != nil && res.Failed() {
    panic(fvmErrors.NewEVMError(res.VMError))
}
``` [4](#0-3) 

The full call chain from Cadence is:

`EVM.EVMAddress.deposit(from: <-vault)` → `InternalEVM.deposit()` → `impl.newInternalEVMTypeDepositFunction` → `account.Deposit(vault)` → `types.NewDepositCall(..., GasLimit: DepositCallGasLimit)` → `mintTo` → revert. [5](#0-4) [6](#0-5) 

The codebase itself documents and tests this failure mode:

```go
t.Run("tokens deposit to an smart contract that doesn't accept native token", func(t *testing.T) {
    ...
    require.Equal(t, res.VMError, gethVM.ErrExecutionReverted)
``` [7](#0-6) 

The comment in the constant block even acknowledges the 2,300 figure is the "max gas allowed for receive/fallback methods", confirming the design mirrors Solidity's `transfer` semantics: [8](#0-7) 

### Impact Explanation

Any EVM smart contract whose `receive()` or `fallback()` function consumes more than 2,300 gas — a category that includes proxy contracts, multisig wallets, contracts that emit events on receipt, or any contract that writes to storage on receipt — **cannot receive native FLOW tokens via the Cadence bridge deposit path**. The deposit call reverts, state is reset, and the Cadence transaction aborts. The tokens are returned to the caller's vault (no permanent loss in the simple case), but the contract address is structurally unreachable via this bridge mechanism. If the deposit is part of a larger atomic Cadence transaction (e.g., a bridge protocol that locks Cadence-side tokens before crediting EVM-side), the revert can cause cross-VM accounting inconsistencies or permanent denial of service for that contract as a bridge recipient.

**Impact: 4** — Cross-VM asset delivery failure; smart contract addresses with non-trivial receive logic are permanently excluded from the native FLOW deposit bridge path.

### Likelihood Explanation

The entry point is fully unprivileged: any Cadence transaction can call `EVM.EVMAddress.deposit(from: <-vault)` targeting any EVM address. The triggering condition — a `receive()`/`fallback()` that uses more than 2,300 gas — is extremely common in production EVM contracts (proxy patterns, gnosis-safe style multisigs, ERC-4337 account abstraction contracts, any contract emitting an event on receive). No special keys, staked nodes, or admin access are required.

**Likelihood: 3** — Moderate-to-high; the pattern is common and the entry path is open to any transaction sender.

### Recommendation

Replace the hardcoded `DepositCallGasLimit` with a configurable or sufficiently large gas limit (analogous to using `call{value: amount}("")` instead of `transfer` in Solidity). The `NewDepositCall` constructor should accept a caller-supplied or protocol-defined gas limit that is not artificially capped at `21,000 + 2,300`. A reasonable default would be a value large enough to accommodate typical `receive()`/`fallback()` implementations (e.g., 100,000 gas), while still being bounded by the Flow computation limit to prevent abuse.

### Proof of Concept

1. Deploy an EVM contract with a `receive()` that emits an event (costs ~3,000+ gas):
   ```solidity
   event Received(address sender, uint256 amount);
   receive() external payable { emit Received(msg.sender, msg.value); }
   ```
2. From a Cadence transaction, call:
   ```cadence
   let vault <- flowTokenVault.withdraw(amount: 1.0) as! @FlowToken.Vault
   let contractAddr = EVM.EVMAddress(bytes: <deployed contract bytes>)
   contractAddr.deposit(from: <-vault)
   ```
3. Observe: `NewDepositCall` is constructed with `GasLimit = 23,300`. After the 21,000 intrinsic cost, only 2,300 gas is available. The `emit` opcode plus event data exceeds this. The EVM returns `ErrExecutionReverted`. `mintTo` resets state. `panicOnErrorOrInvalidOrFailedState` panics. The Cadence transaction aborts.

The existing test at `fvm/evm/emulator/emulator_test.go:83–110` already demonstrates the revert behavior for any contract that does not trivially accept native tokens under the 2,300-gas constraint. [1](#0-0) [2](#0-1) [6](#0-5) [9](#0-8) [10](#0-9)

func (proc *procedure) mintTo(
	call *types.DirectCall,
) (*types.Result, error) {
	// check and convert value
	value, isValid := checkAndConvertValue(call.Value)
	if !isValid {
		return types.NewInvalidResult(
			call.Type,
			call.Hash(),
			types.ErrInvalidBalance,
		), nil
	}

	// create bridge account if not exist
	bridge := call.From.ToCommon()
	if !proc.state.Exist(bridge) {
		proc.state.CreateAccount(bridge)
	}

	// add balance to the bridge account before transfer
	proc.state.AddBalance(bridge, value, gethTracing.BalanceIncreaseWithdrawal)
	// check state errors
	if err := proc.state.Error(); err != nil {
		return nil, err
	}

	// withdraw the amount and move it to the bridge account
	res, err := proc.run(call.Message(), call.Hash(), types.DirectCallTxType)
	if err != nil {
		return res, err
	}

	// if any error (invalid or vm) on the internal call, revert and don't commit any change
	// this prevents having cases that we add balance to the bridge but the transfer
	// fails due to gas, etc.
	if res.Invalid() || res.Failed() {
		// reset the state to revert the add balances
		proc.state.Reset()
		return res, nil
	}

	// commit and finalize the state and return any stateDB error
	res.StateChangeCommitment, err = proc.commit(true)
	return res, err
}

func panicOnErrorOrInvalidOrFailedState(res *types.Result, err error) {

	if res != nil && res.Invalid() {
		panic(fvmErrors.NewEVMError(res.ValidationError))
	}

	if res != nil && res.Failed() {
		panic(fvmErrors.NewEVMError(res.VMError))
	}

	// this should never happen
	if err == nil && res == nil {
		panic(fvmErrors.NewEVMError(types.ErrUnexpectedEmptyResult))
	}

	panicOnError(err)
