### Title
Gateway Admits Unsigned Invoke Transactions via Overly Broad `skip_stateful_validations` Condition — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator skips the `__validate__` entry-point (signature verification) for any Invoke transaction with `nonce == 1` when `account_nonce == 0` and `account_tx_in_pool_or_recent_block` returns `true`. The check `account_tx_in_pool_or_recent_block` is not restricted to deploy-account transactions — it returns `true` for **any** transaction from that address in the pool or a recent block. An unprivileged attacker who observes a victim's pending `deploy_account` in the mempool can submit a forged Invoke (nonce=1, arbitrary calldata, invalid signature) targeting the victim's address and have it admitted to the mempool without any signature check.

### Finding Description

The UX feature is documented in the code comment:

> "We verify that a deploy_account transaction exists for this account. It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."

The check used is:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs
async fn skip_stateful_validations(...) -> ... {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                ...;
        }
    }
    Ok(false)
}

The `validate_by_mempool` call that precedes `skip_stateful_validations` only checks for duplicate tx-hash and nonce ordering — it does **not** verify signatures: [5](#0-4) 

### Impact Explanation

An attacker who observes a victim's `deploy_account` (nonce=0) in the mempool can submit a forged Invoke with `nonce=1`, arbitrary calldata, and an invalid/empty signature targeting the victim's address. The gateway will:

1. Pass the nonce check (`0 ≤ 1 ≤ 0 + max_allowed_nonce_gap`).
2. Pass `validate_by_mempool` (no duplicate tx-hash, nonce ordering valid).
3. Enter `skip_stateful_validations`: `tx.nonce() == 1`, `account_nonce == 0`, `account_tx_in_pool_or_recent_block == true` (victim's deploy_account is in pool) → returns `true`.
4. Skip `run_validate_entry_point` entirely.
5. Admit the forged Invoke to the mempool.

The forged transaction carries an invalid signature and will fail `__validate__` when the batcher executes it (after the deploy_account deploys the account). However, the admission invariant is broken: **a transaction with an invalid signature is accepted by the gateway and enters the mempool**. This matches the "High" impact: *Mempool/gateway/RPC admission accepts invalid transactions before sequencing*.

Secondary consequences:
- The attacker can flood the mempool with forged nonce=1 invokes for every visible `deploy_account`, consuming mempool capacity at zero cost (failed transactions are aborted, not charged).
- If the attacker's forged invoke has a higher tip than the victim's legitimate invoke, it may be dequeued first by the batcher, consuming batcher execution resources before failing.

### Likelihood Explanation

The mempool is observable (transactions are propagated via P2P). Any `deploy_account` transaction is visible to network participants. The attack requires no privileged access, no special contract, and no on-chain funds — only the ability to submit a well-formed RPC transaction with `nonce=1` and any signature to the gateway. The default `max_allowed_nonce_gap` is 200, so the nonce check is trivially satisfied. [6](#0-5) 

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a `deploy_account` transaction exists in the pool for the target address. The mempool should expose a dedicated `has_pending_deploy_account(address)` query, or the gateway should inspect the transaction type of the pending transaction before skipping validation. Alternatively, restrict the skip to cases where the gateway itself received the `deploy_account` in the same request batch (as the Python gateway does via the `deploy_account_tx_hash` parameter in `PyValidator::should_run_stateful_validations`). [7](#0-6) 

### Proof of Concept

1. Victim submits `deploy_account` for address `0xVICTIM` (nonce=0). It enters the mempool.
2. Attacker calls the gateway's `add_transaction` RPC with:
   - type: `INVOKE`
   - `sender_address`: `0xVICTIM`
   - `nonce`: `0x1`
   - `calldata`: `[<drain_funds_selector>, <attacker_address>]`
   - `signature`: `[]` (empty / invalid)
   - `resource_bounds`: valid non-zero bounds
3. Gateway flow:
   - `StatelessTransactionValidator::validate` passes (valid structure, non-zero bounds, nonce DA mode L1).
   - `extract_state_nonce_and_run_validations`: `account_nonce = 0`.
   - `validate_state_preconditions`: nonce `1 ≤ 0 + 200` → passes.
   - `validate_by_mempool`: no duplicate hash, nonce ordering valid → passes.
   - `skip_stateful_validations`: `nonce==1`, `account_nonce==0`, `account_tx_in_pool_or_recent_block(0xVICTIM)==true` → returns `true`.
   - `run_validate_entry_point` called with `validate=false` → `__validate__` is **never called**.
4. Forged Invoke is admitted to the mempool. Gateway returns a transaction hash to the attacker.
5. When the batcher later executes the forged Invoke (after the deploy_account), `__validate__` is called with `validate=true` (default `ExecutionFlags`), the invalid signature causes it to fail, and the transaction is aborted — but the mempool admission invariant was already violated at step 4. async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}

    pub fn should_run_stateful_validations(
        &mut self,
        account_tx: &AccountTransaction,
        deploy_account_tx_hash: Option<TransactionHash>,
    ) -> StatefulValidatorResult<bool> {
        if account_tx.tx_type() != TransactionType::InvokeFunction {
            return Ok(true);
        }
        let tx_info = account_tx.create_tx_info();
        let nonce = self.stateful_validator.get_nonce(tx_info.sender_address())?;

        let deploy_account_not_processed =
            deploy_account_tx_hash.is_some() && nonce == Nonce(Felt::ZERO);
        let tx_nonce = tx_info.nonce();
        let is_post_deploy_nonce = Nonce(Felt::ONE) <= tx_nonce;
        let nonce_small_enough_to_qualify_for_validation_skip =
            tx_nonce <= self.max_nonce_for_validation_skip;

        let skip_validate = deploy_account_not_processed
            && is_post_deploy_nonce
            && nonce_small_enough_to_qualify_for_validation_skip;

        Ok(!skip_validate)
    }
