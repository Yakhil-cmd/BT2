### Title
`calculateNFTRoyalties()` Uses Pre-Royalty Amount as Net Proceeds, Bypassing Negative-Amount Guard and Misleading Offer Acceptance - (File: packages/gui/src/components/offers/utils.ts)

### Summary

`calculateNFTRoyalties()` in `packages/gui/src/components/offers/utils.ts` hardcodes `nftSellerNetAmount = amount` (the full pre-royalty amount) instead of the correct post-royalty value `amount - royaltyAmount - makerFee`. The correct formula is present but commented out. This propagates into two critical display paths: the "Net Proceeds" label shown to a user before they accept a `TokenForNFT` offer, and the `overrideNFTSellerAmount` value injected into the offer summary. It also permanently disables the `showNegativeAmountWarning` guard that is supposed to block offer creation when royalties would exceed the offered amount.

### Finding Description

In `packages/gui/src/components/offers/utils.ts`, `calculateNFTRoyalties()` computes the royalty amount correctly but then assigns the wrong value to `nftSellerNetAmount`:

```typescript
// utils.ts line 314-317
const nftSellerNetAmount: number = amount;
// : parseFloat(
//     (amount - parseFloat(royaltyAmountString) - makerFee).toFixed(12),
//   );
```

The commented-out block is the correct formula. The live code sets `nftSellerNetAmount` to the full offered `amount`, ignoring royalties and fees entirely.

This value flows into three consumer sites:

**1. `NFTOfferEditor.tsx` — safety guard bypass**

```typescript
// NFTOfferEditor.tsx line 271
const showNegativeAmountWarning = (nftSellerNetAmount ?? 0) < 0;
```

Because `nftSellerNetAmount` is always equal to the user-entered `amount` (a positive number), `showNegativeAmountWarning` is always `false`. The guard that is supposed to display "Unable to create an offer where the net amount is negative" and block submission never fires, even when royalties exceed the offered amount.

**2. `NFTOfferEditor.tsx` — "You will receive" / "They will receive" display**

```typescript
// NFTOfferEditor.tsx line 364-366
<FormatLargeNumber value={new BigNumber(nftSellerNetAmount ?? 0)} />
```

The label shown to the user during offer creation displays the full pre-royalty amount as what the seller will receive.

**3. `NFTOfferViewer.tsx` — `overrideNFTSellerAmount` and "Net Proceeds"**

```typescript
// NFTOfferViewer.tsx line 397-402
const overrideNFTSellerAmount =
  exchangeType === NFTOfferExchangeType.TokenForNFT
    ? assetType === OfferAsset.CHIA
      ? chiaToMojo(nftSaleInfo?.nftSellerNetAmount ?? 0)
      : catToMojo(nftSaleInfo?.nftSellerNetAmount ?? 0)
    : undefined;
```

`overrideNFTSellerAmount` is derived from the wrong `nftSellerNetAmount` and is passed into `NFTOfferSummary` to override the displayed amount in the "Purchase Summary" panel. Additionally, the "Net Proceeds" label at line 596 renders the same wrong value:

```typescript
// NFTOfferViewer.tsx line 595-597
<FormatLargeNumber value={new BigNumber(nftSaleInfo?.nftSellerNetAmount ?? 0)} /> {displayName}
```

### Impact Explanation

When a user views an imported `TokenForNFT` offer (someone offering XCH/CAT to buy their NFT), the GUI displays "Net Proceeds = full offered amount" instead of "Net Proceeds = offered amount − royalties". The user sees an inflated figure and may accept the offer believing they will receive more than they actually will. The blockchain executes the royalty deduction correctly, so the actual settlement differs from what the GUI showed at the moment of approval. This is a direct case of the GUI causing a user to approve the wrong amount/status for an NFT transaction.

Additionally, the `showNegativeAmountWarning` guard is permanently suppressed. An NFT with a royalty percentage ≥ 100% (technically encodable in the NFT metadata as a uint16 basis-point value) would produce a negative real net proceeds, but the GUI shows a positive number and allows offer creation and acceptance without any warning.

### Likelihood Explanation

Any NFT with a non-zero royalty percentage triggers the wrong display. This is the normal case for royalty-bearing NFTs. No special attacker capability is required beyond creating an NFT with a royalty set — a standard, unprivileged action. The user path (view imported offer → see "Net Proceeds" → click "Accept Offer") is the primary offer-acceptance flow.

### Recommendation

Restore the commented-out formula in `calculateNFTRoyalties()`:

```typescript
// packages/gui/src/components/offers/utils.ts
const nftSellerNetAmount: number = parseFloat(
  (amount - parseFloat(royaltyAmountString) - makerFee).toFixed(12),
);
```

Apply this only for the `TokenForNFT` exchange type (where royalties are deducted from the offered amount). For `NFTForToken`, the seller specifies the net amount they want and the buyer pays royalties on top, so `nftSellerNetAmount = amount` is correct for that branch. The fix should be conditional on `exchangeType`.

### Proof of Concept

1. Create an NFT with a 10% royalty.
2. A buyer creates a `TokenForNFT` offer: offering 1 XCH for the NFT.
3. The NFT seller opens the offer in the GUI.
4. **Expected display:** "Net Proceeds: 0.9 XCH" (1 XCH − 10% royalty).
5. **Actual display:** "Net Proceeds: 1 XCH" (full amount, royalty not subtracted).
6. The seller accepts, believing they will receive 1 XCH; the blockchain settles 0.9 XCH to the seller and 0.1 XCH to the royalty recipient.

For the guard bypass: set royalty to 110% (basis-point value 11000). Enter any positive amount in the offer editor. `showNegativeAmountWarning` remains `false` and the error message "Unable to create an offer where the net amount is negative" never appears, allowing the offer to be submitted.

export function calculateNFTRoyalties(
  amount: number,
  makerFee: number,
  royaltyPercentage: number,
  exchangeType: NFTOfferExchangeType,
): CalculateNFTRoyaltiesResult {
  const royaltyAmount: number = royaltyPercentage ? (royaltyPercentage / 100) * amount : 0;
  const royaltyAmountString: string = formatAmount(royaltyAmount);
  const nftSellerNetAmount: number = amount;
  // : parseFloat(
  //     (amount - parseFloat(royaltyAmountString) - makerFee).toFixed(12),
  //   );
  const totalAmount: number =
    exchangeType === NFTOfferExchangeType.NFTForToken ? amount + royaltyAmount : amount + makerFee + royaltyAmount;
  const totalAmountString: string = formatAmount(totalAmount);

  return {
    royaltyAmount,
    royaltyAmountString,
    nftSellerNetAmount,
    totalAmount,
    totalAmountString,
  };
}
