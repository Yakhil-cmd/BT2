### Title
Missing Lower-Bound Threshold Check in `assert_sign_inputs` Allows Signing Session Corruption - (`File: src/frost/mod.rs`)

### Summary

`assert_sign_inputs` in `src/frost/mod.rs` validates the threshold parameter with only an upper-bound check (`threshold > participants.len()`), but omits the lower-bound check (`threshold >= 2`) that is present in the DKG entry-point `assert_key_invariants`. A malicious participant or coordinator can supply `threshold = 1` (or `0`) to any FROST signing call, causing every participant to compute Lagrange-weighted signature shares under the wrong polynomial degree. The aggregated signature is cryptographically invalid, permanently aborting the signing session for all honest parties.

---

### Finding Description

`assert_key_invariants` in `src/dkg.rs` enforces both bounds on the threshold:

```rust
if threshold > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { … });
}
if threshold < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
}
``` [1](#0-0) 

`assert_sign_inputs` in `src/frost/mod.rs`, which gates every FROST signing call, enforces only the upper bound:

```rust
// validate threshold
if threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { … });
}
``` [2](#0-1) 

There is no corresponding `threshold < 2` guard. `ReconstructionLowerBound` is a plain `usize` newtype with no invariant enforcement: [3](#0-2) 

The same omission appears in the FROST presign entry-point: [4](#0-3) 

and in the OT-based ECDSA presign entry-point: [5](#0-4) 

`assert_sign_inputs` is the sole validation gate for all three FROST signing functions: [6](#0-5) [7](#0-6) [8](#0-7) 

After validation passes, the threshold is forwarded directly into `construct_key_package`, which builds the FROST `KeyPackage` with `min_signers = threshold`. Each participant then computes their signature share using Lagrange coefficients derived from that threshold value: [9](#0-8) 

---

### Impact Explanation

In FROST, the Lagrange coefficient for participant *i* is computed over the polynomial degree implied by the threshold. When a malicious caller supplies `threshold = 1` while the DKG was conducted at `threshold = 3`, every participant computes their linearized share `s_i = λ_i(threshold=1) · (nonce_i + share_i · challenge)` using the wrong Lagrange weight. The coordinator sums these shares; the result does not equal the correct signature scalar for the actual degree-2 secret polynomial. The `aggregate` call verifies the signature internally and returns an error, making the signing session permanently unusable. Honest parties cannot distinguish this failure from a network fault, and the presignature nonces consumed in a V2 flow cannot be reused. A malicious coordinator who consistently supplies `threshold = 1` prevents any signing session from completing.

This maps to: **High — Corruption of sign outputs so honest parties accept unusable cryptographic outputs / Permanent denial of signing for honest parties.**

---

### Likelihood Explanation

Any participant who calls `sign_v1`, `sign_v2`, or `redjubjub::sign` controls the `threshold` argument locally. No cryptographic capability is required; the attacker simply passes `1` instead of the agreed threshold. The malicious participant is indistinguishable from an honest one until aggregation fails, and the failure is attributable to any participant or to the network. In the V2 presign flow, the attacker can also pass `threshold = 1` to `frost::presign`, wasting the nonce material of all honest participants.

---

### Recommendation

Add the same lower-bound guard that `assert_key_invariants` already enforces:

```rust
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

Apply this check in:
- `assert_sign_inputs` (`src/frost/mod.rs`, after line 150)
- `presign` in `src/frost/mod.rs` (after line 77)
- `presign` in `src/ecdsa/ot_based_ecdsa/presign.rs` (after line 36)

Alternatively, encode the invariant in `ReconstructionLowerBound` itself so it cannot be constructed with a value below 2.

---

### Proof of Concept

```
Setup:
  - Run DKG with participants = [P1, P2, P3], threshold = 3.
  - All parties obtain valid KeygenOutput with shares of a degree-2 polynomial.

Attack (malicious coordinator P1):
  - P1 calls sign_v1(participants, threshold=1, me=P1, coordinator=P1, …)
  - assert_sign_inputs passes: 1 <= 3 (upper bound only).
  - P1's construct_key_package is built with min_signers=1.
  - P1 computes its signature share with Lagrange coefficient λ_P1(threshold=1).
  - P2 and P3 call sign_v1 with threshold=3 (correct); their shares use λ(threshold=3).
  - Coordinator aggregates: sum of shares with mixed Lagrange weights ≠ valid signature scalar.
  - aggregate() returns Err; signing session fails for all honest parties.
  - In V2 flow, presignature nonces for P2 and P3 are permanently consumed.
``` [10](#0-9) [11](#0-10)

pub fn assert_sign_inputs(
    participants: &[Participant],
    threshold: impl Into<ReconstructionLowerBound>,
    me: Participant,
    coordinator: Participant,
) -> Result<ParticipantList, InitializationError> {
    let threshold = threshold.into();
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // validate threshold
    if threshold.value() > participants.len() {
        return Err(InitializationError::ThresholdTooLarge {
            threshold: threshold.value(),
            max: participants.len(),
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }
    Ok(participants)
}
