### Title
Router ETH Balance Consumed by Any Caller's WETH Swap, Stealing Residual ETH Left by Prior Users — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

---

### Summary

`PeripheryPayments.pay()` checks `address(this).balance` before pulling WETH from the payer. Any ETH sitting on the router — left there by a prior user who sent `msg.value` with a non-WETH swap and omitted `refundETH()` — is silently consumed to cover the next caller's WETH obligation. The prior user loses their ETH; the attacker's wallet is never debited.

---

### Finding Description

`pay()` implements a three-branch WETH path: [1](#0-0) 

```
nativeBalance >= value  → wrap router ETH, send WETH to pool, pull nothing from payer
nativeBalance > 0       → wrap router ETH, send partial WETH, pull remainder from payer
nativeBalance == 0      → pull full amount from payer via transferFrom
```

ETH accumulates on the router because every swap entry-point is `payable` and there is no guard that rejects `msg.value > 0` when `tokenIn != WETH`: [2](#0-1) [3](#0-2) 

The `receive()` guard only blocks plain ETH transfers; it does **not** fire when ETH arrives as `msg.value` in a function call: [4](#0-3) 

So a user who calls `exactInputSingle(tokenIn=USDC, msg.value=1 ETH)` and omits `refundETH()` leaves 1 ETH stranded on the router across transaction boundaries.

---

### Impact Explanation

An attacker observing the router's ETH balance calls `exactInputSingle(tokenIn=WETH, amountIn=R)` where `R ≤ address(router).balance`. The pool callback fires `pay(WETH, attacker, pool, R)`. Because `nativeBalance >= R`, the router wraps its own ETH and forwards WETH to the pool — the attacker's wallet is never debited. The attacker receives the full swap output for free; the prior user's stranded ETH is permanently lost.

Partial-subsidy case (`nativeBalance < R`): attacker pays only `R − nativeBalance`, stealing the difference.

---

### Likelihood Explanation

- All four swap entry-points are `payable` with no `msg.value == 0` assertion for non-WETH tokens.
- Multicall users routinely batch swaps and may omit `refundETH()`.
- The router's ETH balance is publicly readable; a bot can monitor and front-run any non-zero balance.
- No special role or privileged access is required — any EOA can trigger the exploit.

---

### Recommendation

1. **Reject stray ETH at entry:** add `if (msg.value > 0 && params.tokenIn != WETH) revert UnexpectedETH();` in each swap function.
2. **Alternatively**, remove the native-ETH hybrid branch from `pay()` entirely and require callers to wrap ETH themselves before calling the router, eliminating the shared-balance risk.
3. At minimum, document that callers **must** append `refundETH()` in every multicall that sends ETH, and enforce it with a modifier or wrapper.

---

### Proof of Concept

```
// Block N
User A: exactInputSingle{value: 1 ether}(tokenIn=USDC, amountIn=1000e6, ...)
// User A forgets refundETH(). Router now holds 1 ETH.

// Block N+1
Attacker: exactInputSingle{value: 0}(tokenIn=WETH, amountIn=1e18, ...)
// Pool callback fires → pay(WETH, attacker, pool, 1e18)
// address(router).balance == 1e18 >= 1e18
// Router wraps its 1 ETH → sends 1 WETH to pool
// Attacker's WETH.transferFrom is never called
// Attacker receives swap output; User A's 1 ETH is gone.
```

The "FACTORY_STATE" framing in the question is a red herring — no factory state is involved. The mechanism is purely the router's shared ETH balance being consumed by any subsequent WETH swap caller.
