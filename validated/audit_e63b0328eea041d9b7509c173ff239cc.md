### Title
Missing Lower-Bound Validation on `threshold` in `assert_sign_inputs` Enables Permanent Denial of FROST Signing — (File: `src/frost/mod.rs`)

### Summary
`assert_sign_inputs`, the shared validation gate for all FROST EdDSA and RedJubjub signing entry points, enforces an upper bound on `threshold` but omits the lower-bound check (`threshold >= 2`) that `assert_key_invariants` enforces for DKG. A caller supplying `threshold = 0` or `threshold = 1` passes all initialization guards and enters the live protocol, where frost_core's `KeyPackage` construction will reject the value and abort signing for every honest participant in the session.

### Finding Description

`assert_key_invariants` (used by `keygen` and `reshare`) explicitly rejects any threshold below 2:

```rust
// src/dkg.rs:580-582
if threshold < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
}
```

`assert_sign_inputs` (used by every FROST signing entry point) only checks the upper bound:

```rust
// src/frost/mod.rs:144-150
if threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge {
        threshold: threshold.value(),
        max: participants.len(),
    });
}
// ← no lower-bound check; threshold = 0 or 1 passes here
```

All three FROST signing functions delegate to this validator:

- `sign_v1` / `sign_v2` in `src/frost/eddsa/sign.rs` — both call `assert_sign_inputs` before launching the protocol.
- `sign` in `src/frost/redjubjub/sign.rs` — same.

After `assert_sign_inputs` returns `Ok`, the coordinator path calls `construct_key_package(threshold, me, signing_share, &vk_package)`. frost_core's `KeyPackage` requires `min_signers >= 2`; passing 0 or 1 causes it to return an error that propagates as a `ProtocolError`, aborting the signing future for every participant waiting on that session.

### Impact Explanation

**High — Permanent denial of signing for honest parties.**

When any participant (including a misconfigured but otherwise honest one) invokes `sign_v1`, `sign_v2`, or `redjubjub::sign` with `threshold = 0` or `threshold = 1`, the protocol future aborts after the network round-trip has already begun. All other honest participants are left waiting indefinitely (the library documents that it "waits indefinitely for expected messages") or receive a protocol error, making the signing session unrecoverable without a full restart. Because presignatures are one-time-use, an aborted session also consumes the presignature material, compounding the denial.

### Likelihood Explanation

**Medium.** `ReconstructionLowerBound` is a plain `usize` newtype with no enforced minimum at the type level. A library integrator who misreads the threshold semantics (e.g., interpreting it as "number of additional signers beyond one" rather than "minimum quorum") will naturally pass `1`. The DKG path would have caught this, but the signing path does not, so the error surfaces only at runtime inside the multi-party protocol rather than at the initialization boundary.

### Recommendation

Add the same lower-bound guard to `assert_sign_inputs` that already exists in `assert_key_invariants`:

```rust
// src/frost/mod.rs — inside assert_sign_inputs, after the upper-bound check
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

This mirrors the DKG invariant documented in `docs/dkg.md` ("Each $P_i$ asserts that $1 < \mathsf{threshold} < N$") and makes the signing API self-consistent with the keygen API.

### Proof of Concept

1. Run DKG normally with `threshold = 3`, `N = 5` participants — succeeds because `assert_key_invariants` enforces `threshold >= 2`.
2. Call `sign_v1(participants, threshold = 1, me, coordinator, keygen_output, message, rng)`.
3. `assert_sign_inputs` checks `1 > 5` → false, so no error is returned.
4. The protocol future starts; the coordinator calls `construct_key_package(ReconstructionLowerBound(1), ...)`.
5. frost_core rejects `min_signers = 1` (below its internal minimum), returning an error.
6. The signing future resolves to `Err(ProtocolError)` for the coordinator; all other participants are left blocked waiting for the signing package that never arrives, permanently denying the signing session. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

pub fn assert_sign_inputs(
    participants: &[Participant],
    threshold: impl Into<ReconstructionLowerBound>,
    me: Participant,
    coordinator: Participant,
) -> Result<ParticipantList, InitializationError> {
    let threshold = threshold.into();
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // validate threshold
    if threshold.value() > participants.len() {
        return Err(InitializationError::ThresholdTooLarge {
            threshold: threshold.value(),
            max: participants.len(),
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }
    Ok(participants)
}
