### Title
`skip_stateful_validations` Admits Invoke Transactions Without Signature Verification via Stale Pool Membership Check ‚ÄĒ (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function is designed to skip `__validate__` for an invoke transaction with nonce=1 when a deploy-account for the same sender is pending in the mempool. However, the membership check it relies on ‚ÄĒ `account_tx_in_pool_or_recent_block` ‚ÄĒ returns `true` for **any** transaction in the pool for that address, not specifically a deploy-account. Once the deploy-account is evicted from the pool (TTL expiry, capacity eviction, or fee-escalation replacement), a subsequent fee-escalation invoke with nonce=1 and an **invalid signature** can be admitted to the mempool without ever calling `__validate__`, because the original invoke's presence in the pool satisfies the check.

---

### Finding Description

**Root cause ‚ÄĒ `skip_stateful_validations`:** [1](#0-0) 

The function checks `account_tx_in_pool_or_recent_block` to decide whether to skip `__validate__`:

```rust
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
    return mempool_client
        .account_tx_in_pool_or_recent_block(tx.sender_address())
        .await
        ...
}
```

The comment at line 441 states: *"It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."* This reasoning is circular: a future-nonce invoke that itself passed via `skip_validate` satisfies the check, even after the deploy-account that originally justified the skip has been evicted.

**`account_tx_in_pool_or_recent_block` does not distinguish transaction types:** [2](#0-1) 

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
```

`tx_pool.contains_account` returns `true` for any transaction type (invoke, declare, deploy-account) in the pool.

**Fee escalation path in `validate_tx` allows a replacement before `skip_stateful_validations` is evaluated:** [3](#0-2) 

`validate_tx` calls `validate_fee_escalation`, which returns `Ok(Some(existing_tx_reference))` when the incoming transaction has a sufficiently higher fee. This pre-check passes without mutating the pool, so when `skip_stateful_validations` is subsequently called, the old invoke is **still in the pool**, making `account_tx_in_pool_or_recent_block` return `true`.

**`run_validate_entry_point` with `skip_validate=true` sets `validate=false`:** [4](#0-3) 

```rust
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
```

When `skip_validate=true`, `validate=false` and the blockifier's `StatefulValidator::perform_validations` short-circuits before calling `__validate__`: [5](#0-4) 

```rust
ApiTransaction::Invoke(_) => {
    ...
    if !tx.execution_flags.validate {
        return Ok(());
    }
    // `__validate__` call.
```

**Gateway flow that sequences these calls:** [6](#0-5) 

```rust
async fn run_pre_validation_checks(...) -> ... {
    self.validate_state_preconditions(executable_tx, account_nonce).await?;
    validate_by_mempool(executable_tx, account_nonce, mempool_client.clone()).await?;
    let skip_validate =
        skip_stateful_validations(executable_tx, account_nonce, mempool_client.clone()).await?;
    Ok(skip_validate)
}
```

`validate_by_mempool` (which calls `mempool.validate_tx`) runs first and approves the fee-escalation replacement **without removing the old invoke from the pool**. Then `skip_stateful_validations` sees the old invoke still present and returns `true`.

---

### Attack Sequence (Proof of Concept)

1. **Attacker observes** that account `A` (nonce=0 on-chain) has a pending invoke `I‚āĀ` with nonce=1 in the mempool. This invoke was admitted via `skip_validate` when a deploy-account `D` was in the pool. `D` has since been evicted (TTL, capacity, or replaced).

2. **Attacker crafts** a new invoke `I‚āā` for address `A`, nonce=1, with an **invalid signature** (e.g., all-zero), and a fee higher than `I‚āĀ` by the escalation threshold (e.g., 10%+).

3. **Gateway receives `I‚āā`:**
   - `StatelessTransactionValidator::validate` passes (signature length check only, not cryptographic verification).
   - `extract_state_nonce_and_run_validations` is called.
   - `get_nonce_from_state` returns `Nonce(0)` (account not deployed).
   - `validate_state_preconditions` passes: nonce=1 is within `[0, 0+200]`.
   - `validate_by_mempool` ‚Üí `mempool.validate_tx` ‚Üí `validate_fee_escalation` returns `Ok(Some(I‚āĀ_ref))` (fee escalation approved, pool not yet mutated).
   - `skip_stateful_validations`: nonce=1, account_nonce=0, `account_tx_in_pool_or_recent_block(A)` = **true** (I‚āĀ still in pool) ‚Üí returns **true**.
   - `run_validate_entry_point(skip_validate=true)` ‚Üí `validate=false` ‚Üí `__validate__` **never called**.

4. **`mempool.add_tx`** is called: `I‚āĀ` is replaced by `I‚āā`. `I‚āā` is now in the mempool with an invalid signature, having bypassed all cryptographic account validation.

5. **At batcher execution time**, the batcher attempts to execute `I‚āā`. Since account `A` is not deployed, execution fails and the transaction reverts. The sequencer has wasted execution resources on a provably invalid transaction.

---

### Impact Explanation

**Impact: High** ‚ÄĒ Mempool/gateway/RPC admission accepts invalid transactions before sequencing.

The gateway's stateful validation invariant is: *"an invoke transaction is admitted only if its `__validate__` entry point succeeds or a deploy-account for the sender is pending."* This invariant is broken. An attacker can inject an invoke with an arbitrary (invalid) signature into the mempool for any account that has a pending invoke with nonce=1, without needing to control the deploy-account. The mempool is polluted with cryptographically unverified transactions. At scale, this enables a low-cost DoS: the attacker repeatedly fee-escalates to keep invalid transactions in the pool, consuming mempool capacity and batcher execution resources.

---

### Likelihood Explanation

**Likelihood: Medium.** The precondition ‚ÄĒ an account with a pending invoke at nonce=1 and no deploy-account in the pool ‚ÄĒ arises naturally in production whenever a deploy-account is evicted due to TTL expiry or capacity pressure. The `transaction_ttl` is configurable and finite. An attacker monitoring the mempool can detect this window and exploit it without any privileged access.

---

### Recommendation

Change `skip_stateful_validations` to verify that the account has specifically a **deploy-account** transaction pending, not just any transaction. One approach: expose a `has_pending_deploy_account(address)` query on the mempool that checks `tx_pool` entries by transaction type. Alternatively, track deploy-account addresses in a dedicated set in the mempool state and query that set instead of the generic `account_tx_in_pool_or_recent_block`. [1](#0-0) 

The fix should replace:
```rust
return mempool_client
    .account_tx_in_pool_or_recent_block(tx.sender_address())
    .await
    ...
```
with a call that specifically checks for a pending deploy-account transaction for `tx.sender_address()`, so that the skip is only granted when the deploy-account is actually present in the pool, not merely because a previously-skipped invoke is present.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
