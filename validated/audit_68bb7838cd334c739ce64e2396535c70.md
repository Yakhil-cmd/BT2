### Title
Unvalidated `puzzleHash` from Blockchain Notification Used as Counter-Offer Destination — (File: `packages/gui/src/hooks/useBlockchainNotifications.tsx`)

### Summary
The `ph` (puzzle hash) field extracted from blockchain notification messages is accepted without any validation and propagated directly as the counter-offer reply-to address. An unprivileged attacker who sends a crafted on-chain notification to a victim's wallet can cause the victim's counter-offer notification (and its XCH fee) to be delivered to the attacker's address instead of the legitimate sender's address.

### Finding Description
In `prepareNotifications` inside `useBlockchainNotifications.tsx`, the notification payload is decoded and the `puzzleHash` is extracted with no validation:

```js
const { u: url, ph: puzzleHash } = data;
```

The raw value is stored directly in the notification object:

```js
return {
  type: NotificationType.OFFER,
  id,
  source: 'BLOCKCHAIN',
  timestamp: timestampData.timestamp,
  offerURL: url,
  puzzleHash,   // ← attacker-supplied, never validated
};
``` [1](#0-0) 

This `puzzleHash` then flows through the UI without any content check:

1. **`NotificationOffer.tsx`** passes it verbatim as the `address` prop to the offer-view route state:
   ```js
   address: 'puzzleHash' in notification ? notification.puzzleHash : undefined,
   ``` [2](#0-1) 

2. **`OfferIncomingTable.tsx`** converts it to a bech32m address and passes it as the counter-offer destination:
   ```js
   const address = currencyCode && puzzleHash
     ? toBech32m(puzzleHash, currencyCode.toLowerCase()) : '';
   navigate('/dashboard/offers/builder', {
     state: { isCounterOffer: true, address, offer },
   });
   ``` [3](#0-2) 

3. **`NotificationSendDialog.tsx`** uses that address as the `target` of `sendNotification`, which transfers 0.0001 XCH on-chain to the destination:
   ```js
   const targetPuzzleHash = fromBech32m(address);
   await sendNotification({ target: targetPuzzleHash, amount: amountMojos, ... });
   ``` [4](#0-3) 

The `offerURL` field (`u`) is validated against a domain allowlist in `OffersProvider.prepareOfferData`, but the `puzzleHash` field (`ph`) has **no equivalent validation** anywhere in the chain. [5](#0-4) 

### Impact Explanation
An attacker who sends a blockchain notification containing their own wallet's puzzle hash in the `ph` field causes the victim to:

- Send a counter-offer notification (carrying 0.0001 XCH on-chain) to the **attacker's address** rather than the legitimate sender's address.
- Expose the counter-offer terms (offer URL / offer data) to the attacker, who can then choose to accept the counter-offer.
- Never deliver the counter-offer to the legitimate sender, breaking the intended negotiation flow.

The destination address field in `NotificationSendDialog` is rendered as a **disabled (read-only)** text field, so the victim cannot correct it and may not notice it is wrong. [6](#0-5) 

This satisfies the **High** impact criterion: *unsafe trust of notification state that causes a user to send to the wrong destination*.

### Likelihood Explanation
Sending a blockchain notification to any wallet address is an unprivileged, low-cost operation (≈0.0001 XCH fee). The attacker only needs to know the victim's wallet address (publicly observable on-chain) and craft a notification with their own puzzle hash in the `ph` field. No key material, host access, or social engineering beyond the notification itself is required. The victim must click "Counter" on the notification, which is the expected user action when they want to negotiate.

### Recommendation
1. **Validate `puzzleHash` format** in `prepareNotifications`: reject values that are not a 32-byte lowercase hex string before storing them in the notification object.
2. **Display the resolved bech32m address prominently** in the counter-offer flow and require explicit user confirmation that the destination is correct, rather than pre-populating a disabled field.
3. Consider **warning the user** that the reply-to address originates from an untrusted on-chain message and has not been verified.

### Proof of Concept
1. Attacker derives their own wallet's puzzle hash (e.g., `aabbcc...` — 32 bytes, valid hex).
2. Attacker encodes a notification payload: `{"v":1,"t":1,"d":{"u":"https://dexie.space/...","ph":"aabbcc..."}}` and sends it on-chain to the victim's wallet address.
3. Victim's GUI receives the notification via `useGetNotificationsQuery` → `prepareNotifications` stores `puzzleHash = "aabbcc..."` without validation.
4. Victim sees "You have a new offer" and clicks **Counter**.
5. `OfferIncomingTable.handleCounterOffer` converts `"aabbcc..."` → attacker's bech32m address and navigates to the offer builder with `address = <attacker address>`.
6. Victim fills in counter-offer terms and clicks **Send Message** in `NotificationSendDialog`.
7. `sendNotification` transfers 0.0001 XCH on-chain to the attacker's address and delivers the counter-offer notification to the attacker.
8. Attacker receives the counter-offer, inspects its terms, and may accept it; the legitimate original sender receives nothing.

  async function handleCounterOffer(notification: Notification) {
    try {
      const puzzleHash = 'puzzleHash' in notification ? notification.puzzleHash : undefined;
      const offerId =
        'offerURL' in notification
          ? notification.offerURL
          : 'offerData' in notification
            ? notification.offerData
            : undefined;
      const offerState = getOffer(offerId);

      if (!offerState || !puzzleHash || !currencyCode) {
        return;
      }

      const address = currencyCode && puzzleHash ? toBech32m(puzzleHash, currencyCode.toLowerCase()) : '';
      const offerSummary = offerState.offer?.summary;

      if (!offerSummary || isDataLayerOfferSummary(offerSummary)) {
        return;
      }

      const offer = offerToOfferBuilderData(offerSummary);

      navigate('/dashboard/offers/builder', {
        state: {
          referrerPath: location.pathname,
          isCounterOffer: true,
          address,
          offer,
        },
      });
    } catch (e) {
      showError(e);
    }
  }
