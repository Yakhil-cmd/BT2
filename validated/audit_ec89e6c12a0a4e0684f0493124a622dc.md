### Title
Gateway Skips `__validate__` (Signature + Chain-ID) for Invoke nonce=1 When Deploy-Account Is Pending — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator unconditionally bypasses the `__validate__` entry-point call — the only place where the account contract verifies the transaction signature and, implicitly, the chain ID — for any invoke transaction with `nonce = 1` whose sender account has a pending deploy-account transaction in the mempool or a recent block. An unprivileged attacker who observes a pending deploy-account transaction can inject an invoke transaction with an arbitrary or cross-chain signature for that account, and the gateway will admit it to the mempool without any signature check.

### Finding Description

In `run_pre_validation_checks`, three checks are executed in order:

1. `validate_state_preconditions` — nonce range and resource bounds (passes for nonce=1, account_nonce=0)
2. `validate_by_mempool` — mempool nonce-gap check (passes)
3. `skip_stateful_validations` — returns `true` (skip) when the invoke tx has `nonce == 1`, the on-chain account nonce is `0`, and `account_tx_in_pool_or_recent_block` returns `true` [1](#0-0) 

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` sets `execution_flags.validate = false`: [2](#0-1) 

With `validate = false`, `AccountTransaction::validate_tx` returns `Ok(None)` immediately without calling the account contract's `__validate__` entry point: [3](#0-2) 

The `__validate__` entry point is the sole mechanism that verifies the ECDSA signature over the transaction hash. The transaction hash is computed with the sequencer's `chain_id`: [4](#0-3) 

Because `__validate__` is skipped at the gateway, neither the signature nor the chain ID embedded in the hash is verified before the transaction is admitted to the mempool.

The `account_tx_in_pool_or_recent_block` check is not a substitute for signature verification: it only confirms that *some* transaction from that address exists in the mempool or a recent block, not that the current invoke transaction was signed by the account's private key. [5](#0-4) 

### Impact Explanation

**High — Mempool/gateway admission accepts invalid transactions before sequencing.**

An attacker can inject an invoke transaction carrying an invalid signature (or a signature valid only on a different chain) into the mempool for any account that has a pending deploy-account transaction. The transaction will eventually revert during blockifier execution (because `__validate__` runs with `validate = true` by default when the batcher converts the stored `InternalRpcTransaction` to an `ExecutableTransaction`), but it is admitted to the mempool without any signature check. Consequences include:

- **Mempool slot exhaustion / griefing**: The attacker can flood the mempool with unsigned nonce-1 transactions targeting every visible pending deploy-account.
- **Displacement of legitimate transactions**: If the attacker's nonce-1 tx is admitted first, the legitimate user's nonce-1 tx may be rejected or require a higher tip to replace it, breaking the intended deploy-account + invoke UX flow.
- **Cross-chain replay admission**: A transaction signed for a different chain ID is admitted without rejection at the gateway layer.

### Likelihood Explanation

**Medium.** The precondition — a pending deploy-account transaction for the target account — is publicly observable from the mempool. No privileged access is required. The attacker only needs to submit an invoke transaction with `nonce = 1` and any signature bytes before the legitimate user's invoke transaction arrives.

### Recommendation

Do not skip `__validate__` entirely. Instead:

1. **Verify the signature against the expected class hash**: The deploy-account transaction carries the `class_hash` of the account contract to be deployed. The gateway can instantiate the contract class and call `__validate__` against it even before the account is on-chain, using the class already stored in the class manager.
2. **Alternatively, restrict the skip to same-batch submissions**: Only skip `__validate__` when the deploy-account and invoke transactions arrive in the same gateway call (atomically), so the attacker cannot inject a standalone nonce-1 invoke for a third-party account.
3. **At minimum, verify that the invoke transaction's signature is non-empty** before skipping `__validate__`, to raise the cost of the attack.

### Proof of Concept

```
1. User A submits RpcDeployAccountTransaction for account X (nonce=0, valid signature).
   → Gateway admits it; mempool now contains a deploy-account for X.

2. Attacker observes the pending deploy-account for X via the public mempool.

3. Attacker crafts RpcInvokeTransactionV3:
     sender_address = X
     nonce          = 1
     signature      = [0x41, 0x41, ...]   // arbitrary / cross-chain bytes

4. Attacker submits the invoke tx to the gateway.

5. Gateway flow:
   a. StatelessTransactionValidator::validate — passes (no signature check here).
   b. convert_rpc_tx_to_internal — computes tx_hash with self.chain_id; no mismatch error.
   c. extract_state_nonce_and_run_validations:
      - get_nonce_from_state(X) → 0
      - validate_state_preconditions: nonce 1 within [0, 0+max_gap] → OK
      - validate_by_mempool: nonce gap OK → OK
      - skip_stateful_validations:
            tx.nonce() == 1  ✓
            account_nonce == 0  ✓
            account_tx_in_pool_or_recent_block(X) → true  ✓
          → returns true (SKIP __validate__)
      - run_validate_entry_point: validate=false → __validate__ NOT called

6. Attacker's invalid/cross-chain invoke tx is added to the mempool.

7. Batcher later executes the tx with validate=true → __validate__ runs → tx reverts.
   But the invalid tx was admitted and occupied a mempool slot.
``` [6](#0-5)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
