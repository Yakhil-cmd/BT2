### Title
Placeholder `L1_TX_NATIVE_PRICE` Constant Causes Incorrect Native Resource Accounting for L1→L2 Priority Transactions — (`File: basic_bootloader/src/bootloader/constants.rs`)

---

### Summary

The constant `L1_TX_NATIVE_PRICE` used to compute the native resource budget for all L1→L2 priority transactions is explicitly marked as an unresolved placeholder with a `TODO` comment. Its current value of `10` is arbitrary and acknowledged as incorrect. Because this constant directly controls how many native (proving) resources an L1 transaction is allocated per unit of gas price, an incorrect value causes systematic miscalculation of the native resource budget and the post-execution `deltaGas` refund adjustment for every L1→L2 priority transaction.

---

### Finding Description

In `basic_bootloader/src/bootloader/constants.rs`, the constant is defined as:

```rust
// Default native price for L1->L2 transactions.
// TODO (EVM-1157): find a reasonable value for it.
pub const L1_TX_NATIVE_PRICE: U256 = U256::from_limbs([10, 0, 0, 0]);
``` [1](#0-0) 

This constant is consumed in `prepare_and_check_resources` inside `process_l1_transaction.rs` to derive `native_per_gas` for every L1→L2 priority transaction:

```rust
let native_price = L1_TX_NATIVE_PRICE;
let native_per_gas = ...
    u256_try_to_u64(&gas_price.div_ceil(native_price)).unwrap_or_else(|| u64::MAX)
``` [2](#0-1) 

`native_per_gas` then drives two downstream calculations:

1. **Native resource budget allocation** — `native_prepaid_from_gas = native_per_gas * gas_limit`, which is passed to `create_resources_for_tx` and capped at `MAX_NATIVE_COMPUTATIONAL` (2^35) for computational work, with the excess held as withheld pubdata resources. [3](#0-2) 

2. **Post-execution `deltaGas` adjustment** — in `compute_gas_refund`, `delta_gas = (native_used / native_per_gas) - gas_used` is added to `gas_used` to ensure the user is charged for native resource consumption. [4](#0-3) 

If `L1_TX_NATIVE_PRICE = 10` is too low relative to the true proving cost:
- `native_per_gas` is inflated, so `native_prepaid_from_gas` is inflated.
- The transaction is allocated far more native resources than it economically paid for.
- `native_used / native_per_gas` is deflated, so `delta_gas` is smaller (or zero), meaning the gas-refund mechanism under-charges the user for actual native resource consumption.

The contrast with the intentional `FREE_L1_TX_NATIVE_PER_GAS = 10000` (used for upgrade/service transactions, deliberately set to exceed `MAX_NATIVE_COMPUTATIONAL` at 72M gas) makes clear that the `L1_TX_NATIVE_PRICE = 10` value has no analogous design rationale and is purely a placeholder. [5](#0-4) 

---

### Impact Explanation

**Resource accounting bug / incorrect constant value.**

- An attacker submitting an L1→L2 priority transaction with a moderate `gas_price` (e.g., 100 wei) receives `native_per_gas = ceil(100/10) = 10`, yielding a native budget of `10 * gas_limit`. With `gas_limit = 72,000,000`, this is 720 million native units — well below `MAX_NATIVE_COMPUTATIONAL` (34 billion), so the full budget is available for computation.
- If the correct `L1_TX_NATIVE_PRICE` should be, say, 1,000, the same transaction would receive `native_per_gas = 1`, yielding only 72 million native units — a 10× reduction.
- The inflated budget allows computationally expensive L1 transactions to succeed that should exhaust native resources and revert.
- The deflated `deltaGas` means the user is refunded more gas than they should be, underpaying for the actual proving cost of their transaction.
- Because L1 transactions cannot be invalidated (they must be processed to avoid halting the priority queue), the system has no fallback to reject them after the fact.

---

### Likelihood Explanation

**High.** Every L1→L2 priority transaction processed by the ZKsync OS bootloader uses `L1_TX_NATIVE_PRICE` unconditionally. Any user who submits an L1→L2 transaction triggers this miscalculation. No special privileges, oracle manipulation, or governance access are required — the entry path is a standard L1 deposit/priority transaction.

---

### Recommendation

- **Short term**: Determine the correct value for `L1_TX_NATIVE_PRICE` based on measured RISC-V proving cycles per unit of gas, and replace the placeholder. The value should satisfy: `gas_price / L1_TX_NATIVE_PRICE ≈ actual_native_cycles_per_gas_unit`.
- **Long term**: Add a compile-time or runtime assertion that `L1_TX_NATIVE_PRICE` is not the placeholder value (e.g., assert it is above a minimum threshold derived from benchmarks). Resolve the tracked issue `EVM-1157`.

---

### Proof of Concept

1. Submit an L1→L2 priority transaction with `gas_price = 100`, `gas_limit = 72_000_000`.
2. In `prepare_and_check_resources`, `native_price = L1_TX_NATIVE_PRICE = 10`, so `native_per_gas = ceil(100/10) = 10`.
3. `native_prepaid_from_gas = 10 * 72_000_000 = 720_000_000`.
4. This is below `MAX_NATIVE_COMPUTATIONAL` (2^35 ≈ 34B), so the full 720M native units are available for computation.
5. If the correct `L1_TX_NATIVE_PRICE` were 1,000, `native_per_gas = 1` and the budget would be 72M — 10× less.
6. A computationally expensive transaction body that consumes between 72M and 720M native units succeeds under the current placeholder but would fail under the correct constant.
7. In `compute_gas_refund`, `delta_gas = (native_used / 10) - gas_used` is smaller than it should be, so the user is charged less gas than the actual proving cost warrants. [1](#0-0) [2](#0-1) [6](#0-5) [7](#0-6)
