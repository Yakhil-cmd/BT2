### Title
Incorrect Multi-NFT Royalty Amount Calculation in WalletConnect Signing Approval Dialog Understates Total Cost - (File: packages/gui/src/electron/commands/parseCommandDisplay.ts)

### Summary
The `formatAmountWithRoyalties` function in `parseCommandDisplay.ts` incorrectly divides the fungible amount by the number of NFTs before applying each royalty percentage. When a WalletConnect `chia_wallet.take_offer` or `chia_wallet.create_offer_for_ids` command involves multiple NFTs with royalties, the "Total Amount with Royalties" shown in the signing approval dialog is systematically understated. A user approves the transaction believing they will spend less than the wallet actually deducts.

### Finding Description
In `formatAmountWithRoyalties` (lines 363–367), the code computes a `splitAmount` by dividing the total fungible amount by the count of NFTs, then applies each NFT's royalty percentage to that split amount:

```typescript
const splitAmount = amount / BigInt(royaltyPercentages.length);
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
``` [1](#0-0) 

The correct formula applies each royalty percentage to the **full** amount, not a fractional share:

```
royaltyAmount = Σ (amount × royaltyPercentage_i / 10_000)
```

The current code computes:

```
royaltyAmount = Σ ((amount / N) × royaltyPercentage_i / 10_000)
```

which equals the correct result only when `N = 1`. For `N > 1`, the displayed total is understated by a factor of `N`.

This display value is rendered in the WalletConnect signing approval dialog (`Confirm.tsx`) as "Total Amount with Royalties": [2](#0-1) 

The function is invoked via `withRoyaltyTotals` → `walletDeltaToDisplay` for both `chia_wallet.take_offer` and `chia_wallet.create_offer_for_ids` WalletConnect commands: [3](#0-2) 

### Impact Explanation
The wallet backend computes and deducts the correct royalty amounts on-chain; the GUI display is the only source of truth the user sees before approving. When the displayed total is lower than what will actually be spent, the user is misled into approving a transaction that costs more than shown. An attacker who controls the offer (e.g., via a malicious dApp using WalletConnect) can craft an offer with multiple high-royalty NFTs to maximize the gap between the displayed and actual cost, causing the victim to unknowingly overspend XCH or CAT tokens.

**Concrete example** — two NFTs each with `royalty_percentage = 1000` (10%), amount = 1 XCH (1 000 000 000 000 mojos):
- **Displayed** (current): `splitAmount = 500 000 000 000`; royalty = 2 × (500 000 000 000 × 1000 / 10 000) = 100 000 000 000 mojos → total shown: **1.1 XCH**
- **Actual deducted**: royalty = 2 × (1 000 000 000 000 × 1000 / 10 000) = 200 000 000 000 mojos → total paid: **1.2 XCH**

The discrepancy scales linearly with the number of NFTs and their royalty percentages.

### Likelihood Explanation
Any WalletConnect-connected dApp can issue a `chia_wallet.take_offer` request containing an offer with multiple NFTs. No special privileges are required. The victim only needs to have the WalletConnect session active and interact with the approval dialog. The existing test at line 251–334 of `parseCommandDisplay.test.ts` encodes and asserts the wrong value (`0.00010255` instead of the correct `0.0001051`), confirming the bug is present and undetected. [4](#0-3) 

### Recommendation
Remove the `splitAmount` division. Each royalty percentage must be applied to the full `amount`:

```typescript
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (amount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
```

Update the corresponding test assertions to reflect the corrected values.

### Proof of Concept
Using the existing multi-NFT test scenario (line 251–334 of `parseCommandDisplay.test.ts`):

- Amount: `100 000 000` mojos (0.0001 XCH)
- NFT 1: `royalty_percentage = 500` (5%)
- NFT 2: `royalty_percentage = 10` (0.1%)

**Current (buggy) output** asserted by the test: `amountWithRoyalties: '0.00010255'`

Calculation:
- `splitAmount = 100 000 000 / 2 = 50 000 000`
- royalty = `(50 000 000 × 500)/10 000 + (50 000 000 × 10)/10 000` = `2 500 000 + 50 000` = `2 550 000`
- total = `102 550 000` mojos = `0.00010255` XCH ← **what the user sees**

**Correct calculation**:
- royalty = `(100 000 000 × 500)/10 000 + (100 000 000 × 10)/10 000` = `5 000 000 + 100 000` = `5 100 000`
- total = `105 100 000` mojos = `0.0001051` XCH ← **what the wallet actually deducts**

The user is shown a total that is `2 550 000` mojos (~0.0000026 XCH) less than what is actually spent. With higher royalties or more NFTs, the gap grows proportionally. [5](#0-4)

function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],
): string | undefined {
  if (royaltyPercentages.length === 0 || line.kind === 'nft') {
    return undefined;
  }

  const splitAmount = amount / BigInt(royaltyPercentages.length);
  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;

  if (line.kind === 'xch') {
    return mojoToChiaLocaleString(totalAmount);
  }

  return mojoToCATLocaleString(totalAmount);
}
