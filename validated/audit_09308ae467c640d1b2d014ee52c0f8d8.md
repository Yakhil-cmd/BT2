### Title
Pool Admin Bypasses `maxAdminSpreadFeeE6` Cap via Uncapped `setPoolBinAdditionalFees` — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`setPoolAdminFees` enforces `maxAdminSpreadFeeE6` on the base spread fee, but `setPoolBinAdditionalFees` passes `addFeeBuyE6`/`addFeeSellE6` directly to the pool with **no cap check**, allowing a pool admin to impose up to `uint16` max (65535 ≈ 6.55% in E6 units) of additional per-bin fee regardless of what `maxAdminSpreadFeeE6` is set to.

---

### Finding Description

`setPoolAdminFees` correctly enforces the admin fee cap:

```solidity
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
``` [1](#0-0) 

But `setPoolBinAdditionalFees` performs no such check — it forwards the caller-supplied values directly to the pool:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
``` [2](#0-1) 

The pool's `setBinAdditionalFees` also has no cap validation — it stores whatever is passed:

```solidity
s.addFeeBuyE6 = addFeeBuyE6;
s.addFeeSellE6 = addFeeSellE6;
``` [3](#0-2) 

During a swap, the bin additional fee is applied **additively** on top of the base spread fee:

```solidity
uint256 buyFeeX64 = baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6);
uint256 sellFeeX64 = baseFeeX64 + Math.mulDiv(binState.addFeeSellE6, ONE_X64, 1e6);
``` [4](#0-3) 

The same additive pattern applies inside the swap loop: [5](#0-4) 

`HARD_MAX_SPREAD_FEE_E6 = 200_000` (20%), and `maxAdminSpreadFeeE6` is initialized to that value but can be lowered by the factory owner via `setFeeCaps`. [6](#0-5) 

When the factory owner lowers `maxAdminSpreadFeeE6` below 65535 (e.g., to 1% = 10,000 E6), the pool admin can still set `addFeeBuyE6 = 65535` via `setPoolBinAdditionalFees`, imposing a 6.55% additional fee on swappers through that bin — far exceeding the intended cap.

---

### Impact Explanation

Traders swapping through the affected bin pay up to 6.55% additional fee on top of the base spread fee, directly extracting principal from swappers beyond the admin fee boundary the factory owner intended to enforce. This is a direct, quantifiable loss of user funds via an admin-boundary break.

---

### Likelihood Explanation

Any pool admin can call `setPoolBinAdditionalFees` at any time with no preconditions beyond holding the `poolAdmin[pool]` role. The bypass is unconditional — it requires no special state, no timelock, and no cooperation from the factory owner. The only prerequisite is that `maxAdminSpreadFeeE6` has been set below 65535 by the factory owner, which is the normal operational intent of the cap mechanism.

---

### Recommendation

Add the same cap check in `setPoolBinAdditionalFees` that exists in `setPoolAdminFees`:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (addFeeBuyE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

---

### Proof of Concept

1. Factory owner calls `setFeeCaps` to lower `maxAdminSpreadFeeE6` to 10,000 (1%).
2. Pool admin calls `setPoolBinAdditionalFees(pool, 0, 65535, 65535)` — no revert.
3. A trader executes a swap through bin 0.
4. The effective buy/sell fee = `baseFeeX64 + mulDiv(65535, ONE_X64, 1e6)` ≈ base + 6.55%, far exceeding the 1% admin cap.
5. Assert `effectiveFee > maxAdminSpreadFeeE6` in a Foundry integration test.
