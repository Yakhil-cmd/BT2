### Title
`EVM.mustRun()` Silently Passes Failed EVM Execution Status to Cadence Caller — (`File: fvm/evm/stdlib/contract.cdc`)

---

### Summary
`EVM.mustRun()` is a public Cadence function in the Flow EVM system contract that is named and documented to imply guaranteed EVM execution. However, it explicitly allows `Status.failed` EVM results to pass through to the Cadence caller without reverting the outer Cadence transaction. Any Cadence transaction or contract that uses `mustRun()` for EVM token transfers without explicitly checking `result.status` will proceed as if the transfer succeeded even when the EVM call failed — directly analogous to the unsafe `transfer()` return-value pattern in TRST-M-4.

---

### Finding Description

`EVM.mustRun()` is defined in the production EVM system contract:

```cadence
/// mustRun runs the transaction using EVM.run
/// It will rollback if the tx execution status is unknown or invalid.
/// Note that this method does not rollback if transaction
/// is executed but an vm error is reported as the outcome
/// of the execution (status: failed).
access(all)
fun mustRun(tx: [UInt8], coinbase: EVMAddress): Result {
    let runResult = self.run(tx: tx, coinbase: coinbase)
    assert(
        runResult.status == Status.failed || runResult.status == Status.successful,
        message: "EVM.mustRun(): The provided transaction is not valid for execution"
    )
    return runResult
}
``` [1](#0-0) 

The `assert` only rejects `Status.invalid` (pre-execution rejection) and `Status.unknown`. It explicitly passes `Status.failed` — meaning an EVM transaction that was executed but reverted (e.g., an ERC20 `transfer()` that returned `false`, ran out of gas, or hit a `revert`) — back to the Cadence caller **without reverting the Cadence transaction**.

The EVM `Status` enum makes this distinction explicit:

```cadence
/// Returned when execution of an evm transaction/call
/// has been successful but the vm has reported an error in
/// the outcome of execution (e.g. running out of gas).
/// A failed tx/call is included in a block.
access(all) case failed
``` [2](#0-1) 

Contrast this with the internal Go-layer functions `Deposit`, `Withdraw`, and `Transfer` on `Account`, which all call `panicOnErrorOrInvalidOrFailedState` — explicitly panicking (reverting) on `Status.failed`:

```go
func panicOnErrorOrInvalidOrFailedState(res *types.Result, err error) {
    if res != nil && res.Invalid() {
        panic(fvmErrors.NewEVMError(res.ValidationError))
    }
    if res != nil && res.Failed() {
        panic(fvmErrors.NewEVMError(res.VMError))
    }
    ...
}
``` [3](#0-2) 

The Cadence-layer `mustRun()` provides no equivalent safety guarantee, despite its name implying one.

---

### Impact Explanation

A Cadence transaction or contract that calls `EVM.mustRun()` to execute an EVM token transfer (e.g., an ERC20 `transfer()`, a bridge withdrawal, or a DEX swap) and does not explicitly check `result.status == EVM.Status.successful` will:

1. Proceed with subsequent Cadence logic (e.g., releasing escrowed Cadence-side assets, updating accounting state, minting tokens) as if the EVM transfer succeeded.
2. The EVM-side transfer did not happen — the EVM nonce was consumed and gas was charged, but the intended asset movement was reverted within the EVM.
3. This creates a **cross-VM asset accounting discrepancy**: Cadence state reflects a completed transfer; EVM state does not.

This is the direct analog to TRST-M-4: just as `sendAllFundsToLP()` proceeds after a silent `transfer()` failure, any `mustRun()` caller proceeds after a silent EVM execution failure.

---

### Likelihood Explanation

- `EVM.mustRun()` is `access(all)` — callable by any unprivileged Cadence transaction or deployed contract.
- The function name `mustRun` strongly implies "must succeed," matching the naming convention of safe-transfer wrappers (e.g., `safeTransfer`). Developers will naturally use it expecting a revert guarantee.
- The warning in the NatSpec comment is easy to overlook, especially since the internal Go-layer equivalents (`Deposit`, `Withdraw`, `Transfer`) all do panic on failure.
- Any Cadence contract bridging EVM tokens or executing EVM-side swaps via `mustRun()` without a post-call status assertion is exploitable by an attacker who can cause the EVM call to revert (e.g., by draining the EVM account's gas balance, or by deploying a contract that conditionally reverts).

---

### Recommendation

Either:

1. **Rename** `mustRun` to `run` (or `tryRun`) to remove the false guarantee implied by the name, and update documentation to clearly state callers must check `result.status`.
2. **Or**, add a second variant `mustRunSuccessful` that asserts `runResult.status == Status.successful` and reverts the Cadence transaction on any non-successful EVM outcome, matching the behavior of the internal `panicOnErrorOrInvalidOrFailedState` guard used by `Deposit`, `Withdraw`, and `Transfer`.

The internal Go layer already demonstrates the correct pattern:

```go
// Deposit, Withdraw, Transfer all use:
panicOnErrorOrInvalidOrFailedState(res, err)
``` [4](#0-3) [5](#0-4) [6](#0-5) 

The Cadence-layer `mustRun()` should provide the same guarantee.

---

### Proof of Concept

**Attacker-controlled entry path:**

1. A Cadence contract is deployed that uses `EVM.mustRun()` to execute an EVM ERC20 `transfer()` and then releases Cadence-side escrowed tokens:

```cadence
import EVM from <EVMAddress>
import FungibleToken from <FTAddress>

transaction(evmTx: [UInt8], coinbase: EVM.EVMAddress, escrowVault: @FungibleToken.Vault) {
    execute {
        // Attacker ensures the EVM tx reverts (e.g., ERC20 transfer fails)
        let result = EVM.mustRun(tx: evmTx, coinbase: coinbase)
        // result.status == EVM.Status.failed — but NO REVERT here
        
        // Cadence continues: releases escrowed tokens even though EVM transfer failed
        let receiver = getAccount(attackerAddress)
            .capabilities.borrow<&{FungibleToken.Receiver}>(/public/receiver)!
        receiver.deposit(from: <-escrowVault)  // attacker receives Cadence tokens
    }
}
```

2. The attacker submits an EVM transaction that is guaranteed to revert (e.g., calls an ERC20 `transfer()` with insufficient balance, or a contract that always reverts).
3. `mustRun()` returns `Status.failed` without reverting the Cadence transaction.
4. The Cadence transaction releases the escrowed vault to the attacker.
5. **Result:** Attacker receives Cadence-side tokens; EVM-side transfer never occurred. Cross-VM asset loss.

func panicOnErrorOrInvalidOrFailedState(res *types.Result, err error) {

	if res != nil && res.Invalid() {
		panic(fvmErrors.NewEVMError(res.ValidationError))
	}

	if res != nil && res.Failed() {
		panic(fvmErrors.NewEVMError(res.VMError))
	}

	// this should never happen
	if err == nil && res == nil {
		panic(fvmErrors.NewEVMError(types.ErrUnexpectedEmptyResult))
	}

	panicOnError(err)
}
