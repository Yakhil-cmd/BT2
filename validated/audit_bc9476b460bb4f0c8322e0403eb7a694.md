### Title
Malicious Participant Can Corrupt CKD Output by Sending Unvalidated Shares Without Detection — (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

In `do_ckd_coordinator`, the coordinator receives each participant's ElGamal share pair `(norm_big_y, norm_big_c)` and blindly accumulates them with no cryptographic verification. A single malicious participant can substitute arbitrary group elements for their correct contribution. The coordinator silently accepts the tampered values, the summation proceeds, and every honest party accepts a corrupted `CKDOutput` as legitimate — yielding a wrong confidential derived key with no error signal.

---

### Finding Description

The CKD coordinator loop in `do_ckd_coordinator` is:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
``` [1](#0-0) 

Each honest participant is supposed to compute:

```rust
let norm_big_y = big_y * lambda_i;          // λ_i · y_i · G
let norm_big_c = big_s + app_pk * y.0;      // λ_i · (x_i · H(pk,app_id) + y_i · A)
``` [2](#0-1) 

The coordinator sums all contributions to produce `(Y, C)` where the correct decryption is `C − a·Y = msk · H(pk, app_id)`. There is **no zero-knowledge proof, no commitment binding, and no consistency check** on any received `(norm_big_y, norm_big_c)` pair before it is added to the running total.

This is structurally identical to the SIZE auction bug: just as `pubKey = (0,0)` or a zero `commitment` was silently accepted and caused a bid to be skipped via `continue`, here any arbitrary `G1` element pair is silently accepted and folded into the aggregate — with no abort, no attribution, and no way for honest parties to distinguish a valid run from a corrupted one.

---

### Impact Explanation

A single malicious participant substitutes `(P, Q)` — any pair of valid `G1` points of their choosing — for their correct `(λ_i · y_i · G, λ_i · (x_i · H(pk,app_id) + y_i · A))`. The coordinator computes:

```
Y'  = Σ_{j≠i} norm_big_y_j  +  P
C'  = Σ_{j≠i} norm_big_c_j  +  Q
```

The decrypted key becomes `C' − a·Y' = (msk − λ_i·x_i)·H(pk,app_id) + Q − a·P`, which is not equal to `msk · H(pk, app_id)` for arbitrary `P, Q`. The `CKDOutput` returned to every honest party is cryptographically wrong. Because the app's secret key `a` is never available to the participants or coordinator, no party can verify the output before accepting it.

**Matched allowed impact:** *High — Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.*

---

### Likelihood Explanation

- Any participant in the CKD session is a reachable attacker; no privileged key material is required.
- The attack requires sending one message with two arbitrary `G1` points — a trivial deviation from the honest protocol.
- The coordinator performs zero cryptographic checks on received shares; the corruption is guaranteed to succeed.
- The corrupted output is indistinguishable from a correct one without the app secret key `a`, so detection is impossible at the protocol layer.

---

### Recommendation

Require each participant to accompany their `(norm_big_y, norm_big_c)` with a non-interactive zero-knowledge proof of correct formation — specifically, a proof of knowledge of `(y_i, x_i)` such that `norm_big_y = λ_i · y_i · G` and `norm_big_c = λ_i · (x_i · H(pk,app_id) + y_i · A)`, binding to the participant's public verification share from the DKG output. The coordinator must verify every proof before accumulating the share. Alternatively, use a verifiable secret sharing approach where each participant's contribution can be checked against their public polynomial commitment.

---

### Proof of Concept

1. Honest participants P_1 … P_{n-1} run `compute_signature_share` correctly and send their `(norm_big_y_j, norm_big_c_j)` to the coordinator.
2. Malicious participant P_n sends `(G1::identity(), G1::identity())` — the identity element — instead of their correct share.
3. The coordinator loop at lines 50–55 adds `(0, 0)` to the running sum without error.
4. The final `CKDOutput` is `(Σ_{j<n} norm_big_y_j, Σ_{j<n} norm_big_c_j)`, missing P_n's Lagrange-weighted secret contribution `λ_n · x_n · H(pk, app_id)`.
5. The app calls `ckd_output.unmask(app_sk)` and obtains `(msk − λ_n·x_n) · H(pk, app_id)` — a wrong key — with no error returned anywhere in the protocol.
