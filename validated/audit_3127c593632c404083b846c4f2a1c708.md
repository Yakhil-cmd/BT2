### Title
Malicious CKD Participant Sends Arbitrary Shares to Corrupt Coordinator's Derived Key Output — (`File: src/confidential_key_derivation/protocol.rs`)

### Summary

The `do_ckd_coordinator` function in `src/confidential_key_derivation/protocol.rs` blindly aggregates `(big_y, big_c)` share contributions received from every participant with no proof-of-correct-computation check. Any single malicious participant can send arbitrary group elements, silently biasing the coordinator's final `CKDOutput` to an attacker-chosen wrong value. The coordinator has no way to detect the manipulation and will return a corrupted derived key to the caller.

### Finding Description

**Root cause — `do_ckd_coordinator`, lines 50–55:**

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
```

Each participant is supposed to send `(λ_i · y_i · G, λ_i · (x_i · H(pk, app_id) + y_i · app_pk))`. The coordinator sums all contributions and returns the aggregate as the CKD output. There is no zero-knowledge proof, commitment scheme, or any other check that the received elements were honestly computed. [1](#0-0) 

**Contrast with DKG**, which enforces proof-of-knowledge verification, commitment-hash binding, and per-share Feldman verification before accepting any participant contribution: [2](#0-1) 

The CKD participant path (`do_ckd_participant`) simply computes and sends its share with no binding commitment sent in advance: [3](#0-2) 

**Attack path:**

1. Honest parties initiate a CKD session. The malicious participant is a legitimate member of `participants`.
2. Instead of computing `(λ_i · y_i · G, λ_i · (x_i · H(pk, app_id) + y_i · app_pk))`, the malicious participant sends `(Δ_Y, Δ_C)` — arbitrary group elements of its choice.
3. The coordinator adds `Δ_Y` and `Δ_C` to the running sums without any check.
4. The final output is `(Y + Δ_Y, C + Δ_C)` instead of the correct `(Y, C)`.
5. The caller unmasks the output as `(C + Δ_C) − app_sk · (Y + Δ_Y)`, obtaining `msk · H(pk, app_id) + Δ_C − app_sk · Δ_Y` — a wrong key — with no error or warning. [4](#0-3) 

### Impact Explanation

The coordinator accepts and returns a silently corrupted `CKDOutput`. Any downstream consumer that calls `CKDOutput::unmask(app_sk)` will obtain a wrong confidential derived key. Because the corruption is undetectable at the protocol level, honest parties have no signal that the output is invalid. This maps directly to the allowed High impact: **Corruption of CKD outputs so honest parties accept unusable or inconsistent cryptographic outputs**.

### Likelihood Explanation

Any participant in the CKD session — a role reachable without privileged assumptions — can execute this attack in a single protocol round. No special knowledge (e.g., `app_sk`, `msk`) is required. The attack requires only the ability to send a crafted message, which every participant already does as part of the normal protocol flow.

### Recommendation

Before the coordinator aggregates contributions, each participant should attach a non-interactive zero-knowledge proof (e.g., a Schnorr proof of discrete-log equality) demonstrating that `big_y` and `big_c` were computed from the same blinding scalar `y_i` and from the participant's committed key share `x_i`. Alternatively, adopt a commit-then-reveal pattern analogous to the DKG's commitment-hash round so that each participant's contribution is bound before it is opened, and verify each opened value against its commitment before summing.

async fn do_ckd_coordinator(
    mut chan: SharedChannel,
    participants: ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<CKDOutputOption, ProtocolError> {
    let (mut norm_big_y, mut norm_big_c) =
        compute_signature_share(&participants, me, key_pair, app_id, app_pk, rng)?;

    // Receive everyone's inputs and add them together
    let waitpoint = chan.next_waitpoint();

    for (_, participant_output) in
        recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
    {
        norm_big_y += participant_output.big_y();
        norm_big_c += participant_output.big_c();
    }
    let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
    Ok(Some(ckd_output))
}

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}
