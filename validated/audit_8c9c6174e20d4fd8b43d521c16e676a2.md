### Title
Unimplemented `TurnBridge` Handler Allows Any Unprivileged User to Submit the Message While Bridge State Is Never Updated — (File: `x/cronos/keeper/msg_server.go`)

### Summary
The `TurnBridge` gRPC handler at line 84 of `x/cronos/keeper/msg_server.go` is completely unimplemented. It returns `nil, nil` with no permission check and no state mutation. Any unprivileged user can submit `MsgTurnBridge` and receive a success response, while the bridge enable/disable state is never actually changed. The admin's ability to disable the bridge is permanently broken.

### Finding Description
`MsgTurnBridge` is the on-chain message intended to allow the Cronos admin to enable or disable the IBC/CRC20 bridge. Its handler in `msg_server.go` is a stub:

```go
// TurnBridge implements the grpc method
func (k msgServer) TurnBridge(goCtx context.Context, msg *types.MsgTurnBridge) (*types.MsgTurnBridgeResponse, error) {
    return nil, nil
}
``` [1](#0-0) 

Two distinct problems exist simultaneously:

1. **No authorization check**: Every other privileged handler in the same file checks `HasPermission`, `CronosAdmin`, or `authority` before proceeding. `TurnBridge` performs none of these checks, so any account can submit the message and receive a success response. [2](#0-1) 

2. **No state mutation**: The bridge enable/disable flag is never written. The `MsgTurnBridge` message carries an `Enable bool` field, but it is never read or persisted, so the bridge state cannot be changed by any caller. [3](#0-2) 

### Impact Explanation
- **High — Bypass of Cronos admin/bridge authorization**: The permission gate that should restrict `TurnBridge` to the admin is entirely absent. Any unprivileged account can call this message and receive a success response, bypassing the intended access control.
- **High — Permanent inability to disable the bridge**: Because no state is written, the bridge can never be turned off via the intended governance path. If the bridge needs to be halted in an emergency (e.g., exploit in progress), the admin has no functional mechanism to do so through `MsgTurnBridge`.

### Likelihood Explanation
The message is proto-defined and exposed as a live gRPC endpoint. Any user with a funded account can submit `MsgTurnBridge` with `Enable: false` or `Enable: true` and receive a success response. No special privileges, leaked keys, or cryptographic breaks are required.

### Recommendation
Implement `TurnBridge` with:
1. An authorization check matching the pattern used by `UpdatePermissions` and `StoreBlockList` (compare `msg.Sender` against `params.CronosAdmin` or use `HasPermission`).
2. A state write that persists the `Enable` flag to the KV store and is read by the bridge transfer path to gate `ConvertVouchers`/`IbcTransferCoins`.

### Proof of Concept
1. Any account submits `MsgTurnBridge{Sender: <any_address>, Enable: false}`.
2. The handler returns `(nil, nil)` — SDK interprets this as success; the tx is included in a block.
3. The bridge enable flag in state is unchanged; bridge transfers continue to process normally.
4. The admin has no working mechanism to halt the bridge. [1](#0-0) [4](#0-3)
