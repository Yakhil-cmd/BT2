### Title
Malicious Participant Can Corrupt CKD Output Without Detection — (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

In the CKD protocol, the coordinator aggregates each participant's contribution `(norm_big_y, norm_big_c)` with no cryptographic proof that the participant used their correct private share. This is structurally identical to the external report's root cause: an off-chain computation is trusted without any on-chain (or in-protocol) verification of the underlying condition. A malicious participant can submit arbitrary group elements, silently corrupting the derived confidential key accepted by an honest coordinator.

---

### Finding Description

In `do_ckd_coordinator`, the coordinator collects outputs from all other participants and sums them unconditionally:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
``` [1](#0-0) 

Each honest participant is supposed to compute their share as:

```
big_s  = H(pk || app_id) * x_i          // uses actual private share
big_c  = big_s + app_pk * y
norm_big_y = lambda_i * (y * G)
norm_big_c = lambda_i * big_c
``` [2](#0-1) 

There is **no zero-knowledge proof, no commitment, and no consistency check** that `big_s` was computed from the participant's actual private share `x_i` (the one consistent with the shared public key). The coordinator has no way to distinguish a correctly computed `norm_big_c` from an arbitrary group element.

Compare this to the DKG protocol, which enforces correctness of participant contributions via proof-of-knowledge verification and commitment hashing: [3](#0-2) 

No equivalent verification exists in the CKD protocol.

---

### Impact Explanation

**High — Corruption of CKD outputs so honest parties accept unusable or incorrect cryptographic outputs.**

The coordinator is the sole recipient of the final `CKDOutput`. If a malicious participant sends a fabricated `(norm_big_y_fake, norm_big_c_fake)`, the coordinator computes:

```
final_norm_big_c = sum of all norm_big_c_i  (includes the fake one)
```

When the coordinator calls `ckd_output.unmask(app_sk)`, the result is:

```
final_norm_big_c - app_sk * final_norm_big_y  ≠  H(pk || app_id) * msk
```

The coordinator silently accepts a wrong confidential derived key. Downstream consumers of this key (e.g., decryption, re-encryption, or key agreement) will produce incorrect results with no indication of failure.

---

### Likelihood Explanation

**Medium.** Any single non-coordinator participant in the CKD session can trigger this. No special privilege is required beyond being a listed participant. The attacker does not need to break any cryptographic primitive — they simply send a malformed group element. The protocol provides no detection mechanism.

---

### Recommendation

Add a DLEQ (Discrete Log Equality) proof alongside each participant's `(norm_big_y, norm_big_c)` submission, proving that the same private share `x_i` was used to compute both `big_s = H(pk || app_id) * x_i` and the participant's public verification share (already known from DKG). The coordinator must verify this proof before accepting the contribution. The existing `dlogeq` proof infrastructure in the codebase can be reused: [4](#0-3) 

---

### Proof of Concept

1. Run a CKD session with participants `{P1, P2, P3}` and coordinator `P1`.
2. Malicious participant `P2` intercepts the protocol at `do_ckd_participant` and instead of computing `(norm_big_y, norm_big_c)` from their real private share, sends `(G, G)` (generator points).
3. `P1` (honest coordinator) receives `P2`'s fake output and adds it to the running sum at lines 53–54.
4. The final `CKDOutput` returned by `do_ckd_coordinator` is corrupted.
5. `ckd_output.unmask(app_sk)` returns a value that does not equal `H(pk || app_id) * msk`, silently breaking the confidential key derivation for the honest coordinator with no error raised. [5](#0-4)

/// Verify that a proof attesting to the validity of some statement.
///
/// We use a transcript in order to verify the Fiat-Shamir transformation.
pub fn verify<C: Ciphersuite>(
    transcript: &mut Transcript,
    statement: Statement<'_, C>,
    proof: &Proof<C>,
) -> Result<bool, ProtocolError>
where
    Element<C>: ConstantTimeEq,
{
    if statement.generator1.ct_eq(&C::Group::identity()).into() {
        return Err(ProtocolError::IdentityElement);
    }

    transcript.message(NEAR_DLOGEQ_STATEMENT_LABEL, &statement.encode()?);

    let (phi0, phi1) = statement.phi(&proof.s.0);
    let big_k0 = phi0 - *statement.public0 * proof.e.0;
    let big_k1 = phi1 - *statement.public1 * proof.e.0;

    let enc = encode_two_points::<C>(&big_k0, &big_k1)?;

    transcript.message(NEAR_DLOGEQ_COMMITMENT_LABEL, &enc);
    let mut rng = transcript.challenge_then_build_rng(NEAR_DLOGEQ_CHALLENGE_LABEL);
    let e = frost_core::random_nonzero::<C, _>(&mut rng);

    Ok(e == proof.e.0)
