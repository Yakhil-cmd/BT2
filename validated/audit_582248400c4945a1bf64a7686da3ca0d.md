### Title
Missing Zero-Address Validation in `change_reward_address` Allows Permanent Loss of Unclaimed Yield - (File: `src/staking/staking.cairo`, `src/pool/pool.cairo`)

### Summary
Both the `Staking` and `Pool` contracts allow setting a reward address to the zero address (`0`) without any validation. A staker or pool member who calls `change_reward_address(reward_address: 0)` by mistake will have all future reward claims permanently transferred to the zero address, resulting in irreversible loss of unclaimed yield.

### Finding Description
`Staking::change_reward_address` accepts a `reward_address: ContractAddress` parameter and only validates that the address is not a registered token address. There is no check that `reward_address != 0`. [1](#0-0) 

The only guard present is:

```cairo
assert!(
    !self.does_token_exist(token_address: reward_address),
    "{}",
    GenericError::REWARD_ADDRESS_IS_TOKEN,
);
```

The zero address is not a registered token, so this check passes. The `reward_address` is then written directly to storage.

When `claim_rewards` is subsequently called, `send_rewards_to_staker` reads the stored `reward_address` and unconditionally transfers STRK to it: [2](#0-1) 

The same pattern exists in `Pool::change_reward_address`, which only checks that the reward address is not the pool's token address: [3](#0-2) 

And `Pool::claim_rewards` transfers directly to the stored `reward_address`: [4](#0-3) 

The same missing check also exists in the initial `stake()` and `enter_delegation_pool()` entry points, which accept `reward_address` at registration time without a zero-address guard. [5](#0-4) [6](#0-5) 

### Impact Explanation
Any STRK rewards transferred to address `0` are permanently unrecoverable. Once `reward_address` is set to zero and rewards accumulate, every subsequent `claim_rewards` call silently burns the rewards. This constitutes **permanent freezing of unclaimed yield**, which falls under the High impact category. The `unstake_action` path also calls `send_rewards_to_staker` before returning principal, meaning the final reward payout is also lost. [7](#0-6) 

### Likelihood Explanation
The call is a single, permissionless transaction by the staker or pool member themselves. A user who accidentally passes `0` (e.g., a scripting error, a UI bug, or a copy-paste mistake) has no recovery path — the address change is immediate and irreversible. The `change_reward_address` function is a routine maintenance operation, making accidental misuse realistic.

### Recommendation
Add a non-zero check for `reward_address` in both `Staking::change_reward_address`, `Pool::change_reward_address`, `stake()`, and `enter_delegation_pool()`:

```cairo
assert!(reward_address.is_non_zero(), "Reward address cannot be zero");
```

This mirrors the existing `assert!(amount.is_non_zero(), ...)` pattern already used throughout the codebase. [8](#0-7) 

### Proof of Concept

**Staker path:**
1. Staker is registered with a valid `reward_address` and accumulates rewards via attestation.
2. Staker calls `Staking::change_reward_address(reward_address: 0)`.
3. The only check — `!self.does_token_exist(token_address: 0)` — passes because `0` is not a registered token.
4. `staker_info.reward_address` is written as `0` to storage.
5. Any caller invokes `Staking::claim_rewards(staker_address)`.
6. `send_rewards_to_staker` executes `token_dispatcher.checked_transfer(recipient: 0, amount: rewards)`.
7. STRK rewards are sent to address `0` and are permanently lost.

**Pool member path:**
1. Pool member calls `Pool::change_reward_address(reward_address: 0)`.
2. The only check — `token_dispatcher.contract_address != 0` — passes.
3. `pool_member_info.reward_address` is written as `0`.
4. `Pool::claim_rewards(pool_member)` transfers STRK rewards to address `0`. [9](#0-8) [10](#0-9)

            let reward_address = staker_info.reward_address;
