### Title
Hardcoded Fixed NEAR Gas on `ExitToNear` Bridge Promises Causes Permanent Fund Loss When `error_refund` Is Disabled — (File: `engine-precompiles/src/native.rs`)

---

### Summary

The `ExitToNear` precompile attaches a hardcoded `FT_TRANSFER_GAS = 10 TGas` to NEAR `ft_transfer` promises. Because the `error_refund` feature is **not enabled by default** in the production build, if the target NEP-141 contract's `ft_transfer` requires more than 10 TGas, the NEAR promise fails silently and the user's bridged ETH or ERC-20 tokens are permanently lost with no recovery path. The same structural defect exists in `ExitToEthereum`, which uses an explicitly unresolved `WITHDRAWAL_GAS = 100 TGas` (marked `TODO`) and has **no error callback at all**.

---

### Finding Description

**Root cause — fixed gas constants:**

In `engine-precompiles/src/native.rs`, the `costs` module defines hardcoded NEAR gas amounts for all bridge-exit promises: [1](#0-0) 

`FT_TRANSFER_GAS = 10 TGas` is attached to every `ft_transfer` NEAR promise created by `ExitToNear`. `WITHDRAWAL_GAS = 100 TGas` is attached to every `withdraw` promise created by `ExitToEthereum`, and its own comment acknowledges the value is unresolved.

**Root cause — no default refund path:**

The `error_refund` feature that enables the refund callback is absent from the `default` feature set in both crates: [2](#0-1) [3](#0-2) 

Without `error_refund`, the `ExitToNear` precompile sets `refund: None` in the callback args: [4](#0-3) 

When `callback_args` equals the default (both `refund` and `transfer_near` are `None`), no callback is scheduled at all — only a bare `PromiseArgs::Create` is emitted: [5](#0-4) 

**`ExitToEthereum` — zero error handling:**

`ExitToEthereum` always emits `PromiseArgs::Create` with no callback, regardless of any feature flag: [6](#0-5) 

If the `withdraw` promise fails for any reason (gas exhaustion, contract panic, etc.), there is no recovery path.

**Token deduction precedes promise execution:**

EVM state changes (ETH deduction from caller, ERC-20 burn) are committed during EVM execution, before NEAR promises are dispatched. A failed promise therefore leaves the EVM state permanently modified with no corresponding NEAR-side credit.

---

### Impact Explanation

**Permanent freezing of funds (Critical).**

- For ETH exits via `ExitToNear`: ETH is transferred from the caller to the precompile address in the EVM. If the NEAR `ft_transfer` promise fails, the ETH is stranded in the precompile address with no recovery.
- For ERC-20 exits via `ExitToNear`: ERC-20 tokens are burned by the ERC-20 contract before the precompile fires the promise. A failed promise means the tokens are destroyed on the EVM side and never credited on NEAR.
- For all exits via `ExitToEthereum`: same permanent loss, with zero callback protection.

---

### Likelihood Explanation

Any EVM user bridging to a NEP-141 contract whose `ft_transfer` implementation requires more than 10 TGas triggers the loss. This is realistic because:

1. 10 TGas is at the lower bound of NEP-141 transfer costs; contracts with storage writes, hooks, or proxy indirection routinely exceed it.
2. The `error_refund` feature is opt-in and absent from the default production build, so the refund path is inactive for all standard deployments.
3. The attacker-controlled entry path is fully unprivileged: any EVM account can call the `ExitToNear` precompile at address `0xe9217bc70b7ed1f598ddd3199e80b093fa71124f` with a value transfer. [7](#0-6) 

---

### Recommendation

1. **Dynamic gas forwarding**: Replace the hardcoded `FT_TRANSFER_GAS` constant with a dynamic calculation based on remaining prepaid gas, analogous to `calculate_attached_gas` already used in `engine/src/contract_methods/connector.rs`: [8](#0-7) 

2. **Enable `error_refund` by default**: Add `error_refund` to the `default` feature list in `engine/Cargo.toml` so the refund callback is always active.

3. **Add an error callback to `ExitToEthereum`**: Mirror the `exit_to_near_precompile_callback` pattern so that a failed `withdraw` promise triggers a refund to the EVM caller.

---

### Proof of Concept

1. Deploy a NEP-141 contract on NEAR whose `ft_transfer` method performs storage writes or cross-contract calls requiring >10 TGas.
2. From an Aurora EVM account, send ETH to the `ExitToNear` precompile (`0xe9217bc70b7ed1f598ddd3199e80b093fa71124f`) targeting that NEP-141 contract.
3. The EVM deducts ETH from the caller and commits the state change.
4. The NEAR `ft_transfer` promise is dispatched with exactly 10 TGas attached.
5. The promise fails due to gas exhaustion on the NEAR side.
6. Because `error_refund` is not enabled, `refund: None` is set and no callback fires.
7. The ETH is permanently stranded — neither returned to the EVM caller nor credited on NEAR.

The same outcome is reproducible via `ExitToEthereum` if the eth connector's `withdraw` function exceeds `WITHDRAWAL_GAS = 100 TGas`, with the additional aggravation that no callback exists under any build configuration. [9](#0-8)

fn calculate_attached_gas<E: Env>(env: &E) -> NearGas {
    let required_gas = env.used_gas().saturating_add(GAS_FOR_PROMISE_CREATION);

    if required_gas >= env.prepaid_gas() {
        NearGas::new(0)
    } else {
        env.prepaid_gas() - required_gas
    }
}
