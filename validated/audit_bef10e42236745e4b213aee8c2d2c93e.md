### Title
Append-Only `stakers` Vector Never Prunes Exited Stakers, Enabling Unbounded Gas Consumption in `get_stakers` - (File: src/staking/staking.cairo)

### Summary
The `stakers: Vec<ContractAddress>` storage field in the `Staking` contract is append-only. Every call to `stake()` pushes a new address into the vector, but no code path ever removes an address — not even after a staker completes `unstake_action()`. The `get_stakers` / `get_stakers(epoch_id)` functions iterate over the entire vector. An unprivileged actor can repeatedly stake with fresh addresses, recover their funds after the exit window, and grow the vector without bound, eventually making `get_stakers` too expensive to execute.

### Finding Description

`stake()` unconditionally appends the caller to `stakers`: [1](#0-0) 

The storage declaration carries an explicit design note confirming removal never happens: [2](#0-1) 

The flow test confirms that a staker who has already completed `unstake_action()` (`staker_in_action`) remains in the vector at index 3 alongside active stakers: [3](#0-2) 

`get_stakers(epoch_id)` (the consensus-facing variant) iterates over every entry in this vector to compute per-staker staking power for the requested epoch. Because exited stakers are never pruned, the iteration cost grows monotonically with the total number of addresses that have ever staked, not just the currently active set.

### Impact Explanation

`get_stakers(epoch_id)` is consumed by consensus nodes to determine the validator set for a given epoch. If the vector is large enough, the function will exceed the Starknet transaction gas limit and revert on every call, permanently preventing consensus nodes from reading the validator set on-chain. This matches the **Unbounded gas consumption** medium impact in the allowed scope.

### Likelihood Explanation

Any address holding the minimum stake can participate. The attacker stakes, waits for the exit window (`DEFAULT_EXIT_WAIT_WINDOW = 1 week`), calls `unstake_action()` to recover funds, then repeats with a fresh address. Because `assert_staker_address_not_reused` blocks address reuse, each iteration requires a new address but the same capital can be recycled across iterations. The economic cost is gas fees plus one week of opportunity cost per iteration — non-trivial but achievable by a motivated griever over time. [4](#0-3) [5](#0-4) 

### Recommendation

Maintain a separate active-staker count or a compacted active-staker set. When a staker completes `unstake_action()`, either remove the address from `stakers` (requires a swap-and-pop pattern on the `Vec`) or mark it as inactive in a parallel bitmap/map so that `get_stakers` can skip inactive entries without iterating the full historical vector. Alternatively, cap the maximum number of registered stakers to bound the iteration cost.

### Proof of Concept

1. Deploy the staking contract with `min_stake = M`.
2. For `i = 1 … N` (using `N` fresh addresses each funded with `M` STRK):
   - Call `stake(reward_address, operational_address, M)` from address `addr_i` → `stakers.push(addr_i)`.
   - Advance time past `DEFAULT_EXIT_WAIT_WINDOW`.
   - Call `unstake_intent()` then `unstake_action()` → funds returned, but `addr_i` remains in `stakers`.
3. Call `get_stakers(epoch_id)`. The function must iterate all `N` entries. For sufficiently large `N`, the call reverts with out-of-gas, permanently blocking consensus nodes from reading the validator set on-chain. [2](#0-1) [1](#0-0) [6](#0-5)
