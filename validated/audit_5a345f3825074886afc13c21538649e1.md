Based on my investigation of the production code, this is a **real vulnerability**.

## Key Findings

**1. No access control on `update_rewards`**

The spec at line 1645 states access control is "Only starkware sequencer," but the implementation has zero caller check: [1](#0-0) 

The function begins with `self.general_prerequisites()` (pause check only) and then immediately checks `last_reward_block` — no `assert_caller_is_sequencer` or equivalent guard exists.

**2. `last_reward_block` is a global singleton, not per-staker**

The gate is a single contract-wide storage slot. Writing it for any staker blocks all other stakers for that block: [2](#0-1) 

`last_reward_block` is written **before** the `disable_rewards` branch. So calling `update_rewards(any_valid_staker, disable_rewards: true)` will:
1. Pass the `current_block_number > last_reward_block` gate (first call this block)
2. Write `current_block_number` into `last_reward_block`
3. Return early — zero rewards distributed

Any subsequent call in the same block (by the legitimate sequencer or anyone else) hits `REWARDS_ALREADY_UPDATED` and reverts.

**3. The spec confirms the discrepancy** [3](#0-2) 

The spec explicitly says access is "Only starkware sequencer" and that `last_reward_block` is updated as the final step (step 6), implying rewards are distributed first. The implementation inverts this: it writes `last_reward_block` at step 1 of the reward path, before the `disable_rewards` guard.

### Title
Missing Caller Guard on `update_rewards` Allows Any Address to Suppress All Block Rewards — (`src/staking/staking.cairo`)

### Summary
`update_rewards` is callable by any unprivileged address. Because `last_reward_block` is a global per-block gate written unconditionally before the `disable_rewards` check, an attacker can call `update_rewards(any_active_staker, disable_rewards: true)` every block to permanently zero out all consensus block rewards for all stakers.

### Finding Description
`StakingRewardsManagerImpl::update_rewards` has no caller restriction. The global `last_reward_block` is written at line 1485 before the `if disable_rewards || self.is_pre_consensus() { return; }` branch at line 1487. An attacker who calls this function first in any block with `disable_rewards: true` consumes the block's reward slot without distributing anything. All subsequent calls in that block revert with `REWARDS_ALREADY_UPDATED`. Repeating this every block permanently starves all stakers of consensus rewards.

### Impact Explanation
Every block's rewards are permanently lost. Stakers and delegators accumulate zero `unclaimed_rewards_own` indefinitely. This matches **High: Permanent freezing of unclaimed yield**.

### Likelihood Explanation
The call is cheap (no token transfer, no privileged setup), requires only knowledge of one active staker address (publicly readable from `get_stakers`), and can be automated. The attacker has no cost beyond gas.

### Recommendation
Add a sequencer-only guard at the top of `update_rewards`, consistent with the spec's stated access control. For example, assert `get_caller_address() == sequencer_address` before any state mutation.

### Proof of Concept
1. Deploy with two active stakers A and B, consensus rewards active.
2. Each block, call `update_rewards(staker_A, disable_rewards: true)` from an arbitrary EOA before the sequencer's call.
3. Observe `last_reward_block` is set; the sequencer's call reverts with `REWARDS_ALREADY_UPDATED`.
4. After N blocks, both stakers' `unclaimed_rewards_own` remain zero.
5. Contrast with a baseline run where the attacker does not front-run: both stakers accumulate expected rewards.
