### Title
Missing ETH Refund in `tick()` When `amountToMint == 0` Causes Permanent Caller ETH Lock — (`L1/starkware/solidity/stake/RewardSupplier.sol`)

---

### Summary

`tick()` is declared `external payable` and uses `msg.value` as bridge/messaging fees, but only inside the `if (amountToMint > 0)` branch. When no L2 mint-request messages are pending, the branch is skipped entirely and any ETH sent by the caller is silently retained by the contract with no refund path and no recovery mechanism.

---

### Finding Description

`tick()` splits `msg.value` between two fee-bearing calls: [1](#0-0) 

Both uses of `msg.value` are gated inside `if (amountToMint > 0)`: [2](#0-1) 

When `requiredMinting()` returns `(0, 0)` — i.e., `messagingContract().l2ToL1Messages(msgHash) == 0` — the branch is never entered, `msg.value` is never forwarded, and no refund is issued: [3](#0-2) 

A complete search of all L1 Solidity files confirms:
- No `receive()` or `fallback()` function exists anywhere in the inheritance chain (`RewardSupplier`, `RewardSupplierStorage`, `ProxySupportImpl`, `GovernanceStub`, `ProxySupport`).
- No ETH withdrawal, rescue, or sweep function exists. [4](#0-3) 

Any ETH sent to `tick()` when no minting is pending is permanently locked in the contract.

---

### Impact Explanation

**Medium — Griefing / damage to users.**

The caller's ETH is permanently frozen in the contract. Because `tick()` is a permissionless keeper function that is expected to be called with ETH to cover bridge and L2-messaging fees, any caller (including automated keeper bots) that sends ETH during a period of zero pending mint requests suffers an irrecoverable loss. The contract has no mechanism to return or sweep the stranded ETH.

This does not fit Critical (no theft of third-party funds) or High (no yield/royalty freeze, no temporary freeze of others' funds). It fits **Medium** as griefing with direct financial damage to the calling user.

---

### Likelihood Explanation

**Low-to-Medium.** The scenario requires a caller to supply `msg.value > 0` when `requiredMinting()` returns zero. This is realistic for:
- Keeper bots that call `tick()` on a fixed schedule with a standard ETH fee regardless of state.
- Users who do not pre-check `requiredMinting()` before calling.

The condition (`numMsgsToConsume == 0`) is the normal steady-state between L2 mint-request epochs, so the window is wide.

---

### Recommendation

Add an explicit refund when `amountToMint == 0`:

```solidity
function tick() external payable {
    (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

    if (amountToMint > 0) {
        // ... existing fee-forwarding logic ...
    } else if (msg.value > 0) {
        // Refund unused ETH
        (bool ok, ) = msg.sender.call{value: msg.value}("");
        require(ok, "ETH_REFUND_FAILED");
    }
}
```

Alternatively, remove `payable` from the signature and require callers to supply ETH only via a separate fee-deposit path, or validate `msg.value == 0` when `amountToMint == 0`.

    function tick() external payable {
        // Check if minting is required, and how much.
        (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

        if (amountToMint > 0) {
            // Prepare the L2->L1 mintRequest message for consumption.
            uint256[] memory messagePayload = new uint256[](1);
            messagePayload[0] = TOKENS_PER_MINT_REQUEST;

            // Consume the mintRequest messages.
            for (uint256 i = 0; i < numMsgsToConsume; i++) {
                messagingContract().consumeMessageFromL2(mintRequestSource(), messagePayload);
            }

            // Reuest minting of the requested amount from the mint manager.
            mintManager().mintRequest(token(), amountToMint);

            // Deposit the minted amount onto the bridge to the credit of `mintDestination`.
            uint256 msgFee = msg.value / 2;
            bridge().depositWithMessage{value: msgFee}(
                token(),
                amountToMint,
                mintDestination(),
                new uint256[](0)
            );
            emit ConsumedL2MintRequests(numMsgsToConsume, amountToMint);

            // Send a totalSupply update to L2MintCurve.
            msgFee = msg.value - msgFee;
            messagePayload[0] = IERC20(token()).totalSupply();
            messagingContract().sendMessageToL2{value: msgFee}(
                mintingCurve(),
                UPDATE_TOTAL_SUPPLY_SELECTOR,
                messagePayload
            );
        }
    }
