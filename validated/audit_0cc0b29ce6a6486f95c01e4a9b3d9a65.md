### Title
Stale `target_shard_index` After Shard Split Corrupts `outcome_root_proof` in `GetExecutionOutcome` ā€” (`chain/client/src/view_client_actor.rs`)

---

### Summary

`GetExecutionOutcome` computes `target_shard_index` from the **old** (pre-split) shard layout, then calls `get_next_block_hash_with_new_chunk` which may return a **child** shard in the **new** (post-split) layout. The index is never recomputed for the new layout, so `merklize(&outcome_roots).1[target_shard_index]` selects the wrong entry from the post-split block's chunk list, producing an `outcome_root_proof` that fails light-client verification for any receipt executed before the epoch boundary.

---

### Finding Description

The handler in `chain/client/src/view_client_actor.rs` computes `target_shard_index` once, before calling `get_next_block_hash_with_new_chunk`:

```rust
// epoch_id and shard_layout are from the OUTCOME block (old epoch)
let shard_layout = self.epoch_manager.get_shard_layout(&epoch_id)...;
let target_shard_id = account_id_to_shard_id(..., &epoch_id)...;
let target_shard_index = shard_layout.get_shard_index(target_shard_id)...;  // OLD index

let res = self.chain.get_next_block_hash_with_new_chunk(
    &outcome_proof.block_hash,
    target_shard_id,   // parent shard
)?;
if let Some((h, target_shard_id)) = res {   // target_shard_id now = CHILD shard
    ...
    // outcome_roots is indexed by the NEW shard layout (post-split block h)
    outcome_root_proof: merklize(&outcome_roots).1[target_shard_index].clone(),
    //                                                ^^^^^^^^^^^^^^^^
    //                                   still the PARENT's index in the OLD layout
}
``` [1](#0-0) [2](#0-1) 

Inside `get_next_block_hash_with_new_chunk`, when the epoch boundary is crossed and the shard layout changes, `shard_ids` is expanded to the children of the parent shard, `shard_layout` is updated to the new layout, and the function returns the first child shard that has a new chunk: [3](#0-2) 

The returned `target_shard_id` is a **child** shard whose index in the new layout is different from the parent's index in the old layout. The `outcome_roots` vector is built from block `h` (in the new epoch, with `new_num_shards` entries), so it is indexed by the new layout. Using the stale `target_shard_index` (parent's position in the old layout) selects the wrong `prev_outcome_root` entry.

---

### Impact Explanation

The exact corrupted value is `outcome_root_proof` = `merklize(&outcome_roots).1[old_parent_index]`, which is the Merkle path for a **different** shard's outcome root, not the child shard that actually contains the receipt's execution outcome.

Light-client verification reconstructs:
```python
block_outcome_root = compute_root(sha256(borsh(shard_outcome_root)), outcome_root_proof)
```
and checks `block_outcome_root == block_header_lite.inner_lite.outcome_root`. Because `outcome_root_proof` proves the wrong leaf, this check fails for every receipt executed in the parent shard before the split. The client-proof invariant ā€” that `EXPERIMENTAL_light_client_proof` returns a verifiable proof for any confirmed receipt ā€” is broken. [4](#0-3) 

---

### Impact Explanation

- **Scope**: read-only RPC proof generation; no ledger state, balances, or consensus are affected.
- **Broken invariant**: `verify_path(block_header_lite.inner_lite.outcome_root, outcome_root_proof, chunk_outcome_root)` returns `false` for a valid, confirmed receipt executed before a dynamic shard split.
- **Who is affected**: any caller of `EXPERIMENTAL_light_client_proof` (or `light_client_proof`) for a receipt whose parent shard was split between the outcome block and the next block with a new chunk. No special privileges required ā€” a normal user submitting a cross-contract call to any account on the parent shard is sufficient.

---

### Likelihood Explanation

Dynamic resharding (`ShardLayoutV3`, `ProtocolFeature::DynamicResharding`) is the trigger. Once enabled on mainnet and a shard split occurs, every historical receipt in the split shard becomes unprovable via the light-client RPC. The condition (receipt executed before split, proof requested after split) is routine and requires no adversarial action.

---

### Recommendation

After `get_next_block_hash_with_new_chunk` returns, recompute `target_shard_index` using the new block's epoch and the returned child `target_shard_id`:

```rust
if let Some((h, target_shard_id)) = res {
    outcome_proof.block_hash = h;
    let new_epoch_id = *self.chain.get_block(&h)?.header().epoch_id();
    let new_shard_layout = self.epoch_manager.get_shard_layout(&new_epoch_id).into_chain_error()?;
    let target_shard_index = new_shard_layout
        .get_shard_index(target_shard_id)
        .map_err(Into::into)
        .into_chain_error()?;
    let outcome_roots = ...;
    ...
    outcome_root_proof: merklize(&outcome_roots).1[target_shard_index].clone(),
}
```

### Proof of Concept

1. Configure a test-loop with dynamic resharding enabled and a 2-shard genesis layout.
2. Submit a cross-contract receipt to an account on shard 0 (parent). Wait for it to be executed and finalized.
3. Trigger a shard split (shard 0 ā†’ children C0, C1) at the next epoch boundary.
4. Call `GetExecutionOutcome` (or `EXPERIMENTAL_light_client_proof`) for the receipt's ID with the correct `receiver_id`.
5. Retrieve `outcome_root_proof` and `block_header_lite`.
6. Assert `verify_path(block_header_lite.inner_lite.outcome_root, outcome_root_proof, chunk_outcome_root)` ā€” it returns `false`, confirming the proof is invalid.

The bug is deterministic: `target_shard_index` is the parent's index in the old layout (e.g., `0`), but after the split the child shards occupy different indices in the new layout (e.g., `0` and `1` for C0 and C1, or higher indices if other shards precede them). The mismatch is guaranteed whenever the child's new-layout index differs from the parent's old-layout index.
