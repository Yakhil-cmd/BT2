### Title
`collectFees` Double-Counts Notional Fees Against Spread-Fee Surplus, Draining LP Bin Reserves — (File: `metric-core/contracts/MetricOmmPool.sol`)

---

### Summary

When both spread fees and notional fees are simultaneously active, `collectFees` computes spread-fee payouts using the **entire pool surplus** as the base — a surplus that already includes the accumulated notional fees — and then **also** pays out the notional fees as a separate line item. The combined transfer exceeds the actual fee surplus, and the shortfall is silently taken from LP bin reserves, causing pool insolvency over time.

---

### Finding Description

The `collectFees` function computes two independent payouts and sums them:

**Spread-fee leg** (lines 391–395):
```solidity
uint256 spreadFee0ToAdminScaled    = (surplus0Scaled * adminSpreadFeeE6_)    / spreadSumE6;
uint256 spreadFee0ToProtocolScaled = (surplus0Scaled * protocolSpreadFeeE6_) / spreadSumE6;
```
Because `spreadSumE6 = adminSpreadFeeE6_ + protocolSpreadFeeE6_`, the two fractions sum to exactly `surplus0Scaled` — **the entire surplus is allocated to spread fees**. [1](#0-0) 

**Notional-fee leg** (lines 397–403):
```solidity
uint256 notionalFee0ToAdminScaled    = (notionalFee0AmountScaled * adminNotionalFeeE8_) / notionalSumE8;
uint256 notionalFee0ToProtocolScaled = notionalFee0AmountScaled - notionalFee0ToAdminScaled;
```
The full `notionalFee0AmountScaled` is paid out on top. [2](#0-1) 

**Combined transfer** (lines 405–427):
```solidity
uint256 totalFee0ToAdminScaled    = spreadFee0ToAdminScaled    + notionalFee0ToAdminScaled;
uint256 totalFee0ToProtocolScaled = spreadFee0ToProtocolScaled + notionalFee0ToProtocolScaled;
// → transfers totalFee0ToAdmin + totalFee0ToProtocol = surplus0Scaled + notionalFee0AmountScaled
``` [3](#0-2) 

The root cause is that `surplus0Scaled` is computed as:

```
surplus0Scaled = balance0 × scaleMultiplier − totalScaledToken0InBins
```

Notional fees are held in the pool's ERC-20 balance but are **not** credited to any bin, so they are fully included in `surplus0Scaled`. Paying out `surplus0Scaled` (spread leg) **and** `notionalFee0AmountScaled` (notional leg) double-counts the notional portion. The excess must be sourced from LP bin reserves.

The test suite explicitly documents this invariant break:

```solidity
assertGt(total0Attempted, surplus0Scaled,
    "token0 attempted payout exceeds computed surplus");
assertGt(total1Attempted, surplus1Scaled,
    "token1 attempted payout exceeds computed surplus");
``` [4](#0-3) 

The ERC-20 `safeTransfer` succeeds because the pool's total balance (`bin_reserves + surplus`) covers the inflated payout as long as `notionalFee0AmountScaled ≤ bin_reserves`. The transfer does not revert; it silently drains LP principal.

---

### Impact Explanation

Every call to the **permissionless** `collectPoolFees` (factory, line 379) drains LP bin reserves by exactly the accumulated notional fee amount. Repeated calls — by keepers, bots, or a griefing attacker — progressively reduce the reserves backing LP shares. LPs who remove liquidity after sufficient fee collections receive less than their entitled principal. At the extreme, the pool becomes insolvent: `totalScaledToken0InBins` exceeds the actual token balance. [5](#0-4) 

---

### Likelihood Explanation

- `collectPoolFees` is **permissionless** — any address may call it at any time.
- The bug fires whenever **both** `spreadFeeE6 > 0` and `notionalFeeE8 > 0`, which is the standard production configuration (protocol spread fee + notional fee are both set at pool creation).
- No privileged role, no special token, no malicious setup is required. A keeper or MEV bot calling `collectPoolFees` on a normal pool is sufficient.

---

### Recommendation

Before computing the spread-fee payout, subtract the notional fee accumulator from the surplus so the two legs operate on disjoint portions of the pool balance:

```solidity
// Correct: exclude notional fees from the spread-fee base
uint256 spreadSurplus0Scaled = surplus0Scaled - notionalFee0AmountScaled;
uint256 spreadSurplus1Scaled = surplus1Scaled - notionalFee1AmountScaled;

uint256 spreadFee0ToAdminScaled    = spreadSumE6 == 0 ? 0
    : (spreadSurplus0Scaled * adminSpreadFeeE6_) / spreadSumE6;
uint256 spreadFee0ToProtocolScaled = spreadSumE6 == 0 ? 0
    : (spreadSurplus0Scaled * protocolSpreadFeeE6_) / spreadSumE6;
// notional leg unchanged — uses notionalFee0AmountScaled directly
```

This ensures `spreadLeg + notionalLeg = spreadSurplus + notionalFees = surplus`, matching the actual available balance.

---

### Proof of Concept

1. Pool is deployed with `adminSpreadFeeE6 = 300 000`, `protocolSpreadFeeE6 = 200 000`, `notionalFeeE8 = 1 000 000` (1 %).
2. LPs deposit; bin reserves = 1 000 000 tokens. Pool balance = 1 000 000.
3. Swaps occur; spread fees accumulate 1 000 tokens, notional fees accumulate 500 tokens. Pool balance = 1 001 500; surplus = 1 500; `notionalFeeToken0Scaled` = 500.
4. Anyone calls `factory.collectPoolFees(pool)`.
5. Inside `collectFees`:
   - Spread leg: `surplus0Scaled = 1 500` → total spread payout = 1 500.
   - Notional leg: `notionalFee0AmountScaled = 500` → total notional payout = 500.
   - Combined transfer = **2 000 tokens**.
6. Pool balance = 1 001 500 ≥ 2 000 → `safeTransfer` succeeds.
7. After collection: pool balance = 999 500; bin reserves still recorded as 1 000 000 in storage.
8. LP attempts `removeLiquidity` for their full share → pool cannot cover the 500-token shortfall → **LP principal loss / revert**. [6](#0-5)
