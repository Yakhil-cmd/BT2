### Title
Gateway Skips Signature Verification for Invoke Transactions Based on Sender Mempool Presence Alone — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

`skip_stateful_validations` in the gateway's stateful validator skips the `__validate__` entry-point call (the only place the account contract verifies the transaction signature) for any invoke transaction with `nonce == 1` whenever the sender's account has *any* transaction present in the mempool. An attacker who observes a victim's `deploy_account` transaction in the mempool can immediately submit an invoke transaction for the victim's account carrying an arbitrary/wrong signature, and the gateway will admit it to the mempool without ever checking the signature. This is the direct sequencer analog of the ZeroLend `unstakeLP` bug: both check a property of the *caller* (ZEROvp balance / "account exists in mempool") instead of verifying authorization over the *specific resource* (tokenId ownership / transaction signature).

**The authorization gap**

`account_tx_in_pool_or_recent_block` returns `true` if the account has *any* transaction in the pool or was seen in a recent block: [4](#0-3) 

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
```

The check answers "does this sender have *something* in the mempool?" — a property of the sender — but never asks "is the signature on *this specific transaction* valid?" — authorization over the specific resource. This is structurally identical to the ZeroLend bug: `unstakeLP` checked `lpPower[tokenId]` balance of `msg.sender` (a property of the caller) instead of checking ownership of `tokenId`.

**Attack path**

1. Victim broadcasts `deploy_account` (nonce 0) for address `V`.
2. Attacker observes `V`'s `deploy_account` in the mempool; `account_tx_in_pool_or_recent_block(V)` now returns `true`.
3. Attacker submits `Invoke(sender=V, nonce=1, signature=<garbage>)` with a tip/gas-price high enough to win fee escalation.
4. Gateway stateful validator:
   - `validate_state_preconditions`: nonce 1 ≥ account nonce 0 → passes.
   - `validate_by_mempool`: no duplicate at (V, nonce=1) yet → passes (no signature check here).
   - `skip_stateful_validations`: `nonce==1 && account_nonce==0 && account_tx_in_pool_or_recent_block(V)==true` → returns `true`.
   - `run_validate_entry_point`: `validate=false` → `__validate__` is **never called**.
5. Attacker's transaction is admitted to the mempool.
6. If fee escalation is enabled and attacker's tip/gas-price exceeds victim's invoke by the configured percentage, the victim's legitimate invoke at nonce 1 is **replaced and dropped**.
7. During block execution the batcher calls `__validate__` with the default `ExecutionFlags` (`validate: true`); the attacker's transaction fails signature verification and is rejected. The victim's invoke is gone.

The mempool explicitly supports fee escalation: [5](#0-4)

### Impact Explanation

- **Invalid transaction admitted**: A transaction with an arbitrary/wrong signature bypasses the gateway's only signature-verification step and enters the mempool. This directly satisfies *"Mempool/gateway/RPC admission accepts invalid transactions … before sequencing."*
- **Valid transaction rejected**: Via fee escalation the attacker's invalid transaction replaces the victim's valid invoke at nonce 1. The victim's transaction is silently dropped from the mempool. This satisfies *"… rejects valid transactions before sequencing."*
- **Free, repeatable DoS**: Because the attacker's transaction fails `__validate__` during block execution it is rejected (not reverted), so no fee is charged to the attacker. The victim must resubmit; the attacker can repeat indefinitely at zero cost, permanently preventing the victim from executing their first post-deployment invoke.

---

### Likelihood Explanation

- The trigger window is the period between a victim broadcasting `deploy_account` and the block being committed — a window of seconds to minutes in normal operation.
- No privileged access is required; any unprivileged party can submit an `RpcTransaction` to the gateway.
- The `deploy_account + invoke` UX pattern is explicitly documented and tested as a supported flow, so victims will regularly create this window.
- Fee escalation is a configurable but production-present feature; even without it the attacker can occupy the nonce-1 slot if the victim has not yet submitted their own invoke.

---

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a check that specifically confirms a `deploy_account` transaction for the sender is present in the mempool. Add a dedicated mempool query such as `deploy_account_tx_in_pool(address) -> bool` that inspects the transaction type, not merely account presence. This preserves the intended UX while closing the authorization gap:

```rust
// Instead of:
mempool_client.account_tx_in_pool_or_recent_block(tx.sender_address()).await

// Use:
mempool_client.deploy_account_tx_in_pool(tx.sender_address()).await
```

Additionally, document that `skip_stateful_validations` is a security-sensitive path and add an integration test that verifies a third-party invoke with a wrong signature for an undeployed account is **rejected** even when a `deploy_account` for that account is in the mempool.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
