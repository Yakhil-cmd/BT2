### Title
Operational Address Front-Running Enables DoS Against Staker Registration — (`src/staking/staking.cairo`)

---

### Summary

The `stake()` function accepts a caller-chosen `operational_address` parameter and enforces global uniqueness via a storage check that reverts with `OPERATIONAL_EXISTS`. An attacker who observes a pending `stake()` transaction can front-run it by submitting their own `stake()` with the same `operational_address`, causing the legitimate staker's transaction to fail permanently for that address.

---

### Finding Description

In `src/staking/staking.cairo`, the `stake()` function performs the following uniqueness check on the caller-supplied `operational_address`:

```cairo
// Lines 298-302
assert!(
    self.operational_address_to_staker_address.read(operational_address).is_zero(),
    "{}",
    Error::OPERATIONAL_EXISTS,
);
``` [1](#0-0) 

The `operational_address` is a free parameter chosen by the staker — it is not derived from `get_caller_address()` or any other caller-bound value. The mapping `operational_address_to_staker_address` enforces a strict 1-to-1 relationship:

```cairo
// Line 342
self.operational_address_to_staker_address.write(operational_address, staker_address);
``` [2](#0-1) 

**Attack path:**

1. Alice submits `stake(reward_address=R, operational_address=OP, amount=MIN)` to the sequencer.
2. Eve observes the pending transaction (e.g., via sequencer mempool API or a monitoring node) and extracts `OP`.
3. Eve submits her own `stake(reward_address=R', operational_address=OP, amount=MIN)` with higher priority/fee.
4. Eve's transaction is sequenced first; `OP` is now registered under Eve's staker address.
5. Alice's transaction is sequenced next and reverts with `"Operational address already exists"`. [3](#0-2) [4](#0-3) 

Eve gains an empty stake position (she can unstake after the exit window) at the cost of capital lockup and gas. Alice's entire `stake()` call fails — no funds are transferred because the check precedes the `transfer_from` call. [5](#0-4) 

---

### Impact Explanation

**Medium — Griefing with no profit motive but damage to users.**

The legitimate staker's registration transaction fails entirely. While Alice can retry with a different `operational_address`, her intended validator node address is now permanently occupied by Eve (until Eve unstakes, which requires waiting out the exit window). If Alice's operational address is tied to a specific validator node key (as is typical in PoS systems), she cannot simply pick an arbitrary replacement — she must either reconfigure her node or wait for Eve to release the address. This constitutes a targeted, repeatable DoS against staker onboarding.

---

### Likelihood Explanation

**Low.**

Starknet currently uses a centralized sequencer operated by StarkWare, which reduces the practical window for front-running compared to Ethereum's public mempool. However:
- The sequencer's pending transaction queue may be observable via public RPC endpoints.
- As Starknet moves toward decentralized sequencing, this attack surface grows significantly.
- The attack requires only the minimum stake amount and is otherwise costless to repeat.

---

### Recommendation

Adopt a two-step commit-reveal or pre-declaration pattern for `operational_address` registration, analogous to the `OPERATIONAL_NOT_ELIGIBLE` / `OPERATIONAL_IN_USE` pattern already present in the error set for address changes: [6](#0-5) 

Specifically:
1. Require the operational address to first call a `declare_for_staker(staker_address)` function, binding it to a specific staker before `stake()` is called.
2. In `stake()`, verify that `operational_address` has been pre-declared for `get_caller_address()` (the `OPERATIONAL_NOT_ELIGIBLE` check), rather than only checking global uniqueness.

This ensures that even if an attacker observes the pending `stake()` call, they cannot register the same `operational_address` under their own staker identity, because the operational address is already committed to Alice's staker address.

---

### Proof of Concept

```
1. Alice's node has operational key OP.
2. Alice approves MIN_STAKE STRK to the staking contract.
3. Alice submits: stake(reward_address=Alice_R, operational_address=OP, amount=MIN_STAKE)
4. Eve observes the pending tx, extracts OP.
5. Eve approves MIN_STAKE STRK to the staking contract.
6. Eve submits (higher fee): stake(reward_address=Eve_R, operational_address=OP, amount=MIN_STAKE)
7. Eve's tx executes first → operational_address_to_staker_address[OP] = Eve_address
8. Alice's tx executes → assert!(self.operational_address_to_staker_address.read(OP).is_zero()) FAILS
   → Reverts with "Operational address already exists"
9. Alice cannot stake with her validator node's operational address.
   Eve holds OP hostage until she completes the exit wait window.
```
