### Title
Coordinator Accepts Unverified Participant Shares in CKD Protocol, Enabling Output Corruption — (File: `src/confidential_key_derivation/protocol.rs`)

### Summary
The CKD coordinator in `do_ckd_coordinator` aggregates cryptographic share contributions from all participants without any cryptographic verification of the received values. A single malicious participant can send arbitrary elliptic-curve points as their `(norm_big_y, norm_big_c)` contribution, and the coordinator will silently incorporate them into the final `CKDOutput`. This is the direct analog of the Frax `transferFrom` unchecked-return-value bug: a critical operation's result is accepted without validation, allowing a malicious actor to corrupt the protocol output while appearing to participate honestly.

### Finding Description

In `do_ckd_coordinator` (`src/confidential_key_derivation/protocol.rs`, lines 35–58), the coordinator:

1. Computes its own share via `compute_signature_share` (line 44–45).
2. Receives a `CKDOutput` — a pair `(norm_big_y, norm_big_c)` — from every other participant (lines 50–55).
3. Adds each received pair directly into the running totals with no verification whatsoever (lines 53–54).
4. Wraps the result in `CKDOutput::new` and returns it (lines 56–57).


The honest computation (lines 148–181) produces:

- `big_y = y · G` (random blinding)
- `big_s = x_i · H(pk ‖ app_id)` (secret share contribution)
- `big_c = big_s + y · app_pk`
- Normalized: `(λ_i · big_y, λ_i · big_c)`

A malicious participant is free to substitute any pair of curve points — including the identity element for `norm_big_y` and an attacker-chosen point for `norm_big_c` — and the coordinator has no mechanism to detect the substitution. There is no zero-knowledge proof of correct share formation, no consistency check against the public key, and no threshold-reconstruction step that would average out the bad contribution.

This is structurally identical to the Frax vulnerability: just as `collateral_token.transferFrom` can return `false` (signaling failure) and the pool proceeds as if collateral was received, here a participant can return a malformed share (signaling nothing) and the coordinator proceeds as if it was correctly formed.

### Impact Explanation

**High — Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.**

The `CKDOutput` produced by the coordinator is the sole output of the protocol. It is used by the application to derive a confidential key via `ckd_output.unmask(app_sk)`. If even one participant sends wrong values, the aggregated `(norm_big_y, norm_big_c)` will not correspond to the correct threshold BLS + ElGamal ciphertext of `msk · H(pk ‖ app_id)`. The derived key will be wrong, and the TEE will silently receive an incorrect secret — with no error, no abort, and no indication that anything went wrong.

### Likelihood Explanation

Any participant in a CKD session can trigger this with zero special privileges. The attacker only needs to be a valid member of the participant list (the same precondition as any honest participant). The attack requires sending a single malformed message in a one-round protocol — it is trivially reachable by any unprivileged library caller who controls one participant's execution.

### Recommendation

Each participant must accompany their `(norm_big_y, norm_big_c)` contribution with a zero-knowledge proof of correct formation — specifically, a proof that `norm_big_c` is a valid ElGamal encryption of `λ_i · x_i · H(pk ‖ app_id)` under `app_pk`, consistent with the participant's public key share. The coordinator must verify all such proofs before aggregating. This is the same fix as using `SafeERC20.safeTransferFrom` in the Frax context: replace silent acceptance with enforced validation.

### Proof of Concept

1. Initialize a CKD session with participants `[P1, P2, P3]` and coordinator `P1`.
2. `P2` (malicious) overrides its protocol execution to send `(identity_G1, arbitrary_point)` instead of its correctly computed share.
3. `P3` sends its honest share.
4. The coordinator (`P1`) receives both, adds them without verification (lines 53–54), and produces `CKDOutput::new(norm_big_y, norm_big_c)` where `norm_big_c` is shifted by `arbitrary_point - honest_P2_contribution`.
5. The caller invokes `ckd_output.unmask(app_sk)` and obtains a value that does not equal `msk · H(pk ‖ app_id)`.
6. No error is raised at any point; the protocol returns `Ok(Some(corrupted_output))`. [1](#0-0) [2](#0-1) [3](#0-2)

```rust
fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
```
