### Title
Unchecked Precompile Call Return Value After ERC-20 Burn Causes Permanent Fund Freeze — (`etc/eth-contracts/contracts/EvmErc20.sol`, `etc/eth-contracts/contracts/EvmErc20V2.sol`)

---

### Summary

Both `EvmErc20` and `EvmErc20V2` mirror token contracts burn the caller's ERC-20 balance **before** invoking the exit precompile, but never check whether the precompile call succeeded. If the precompile call fails for any reason, the ERC-20 tokens are permanently destroyed while the corresponding NEP-141 tokens on the NEAR side are never released, causing an irreversible loss of user funds.

---

### Finding Description

In both `withdrawToNear` and `withdrawToEthereum`, the sequence is:

1. `_burn(_msgSender(), amount)` — destroys the caller's ERC-20 tokens (state change is final).
2. An inline `assembly` block issues a low-level `call` to the exit precompile address.
3. The return value `res` of the `call` opcode is captured into a local variable but **is never inspected**. No `require(res == 1)` or equivalent check exists.

```solidity
// EvmErc20.sol lines 53-63
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // tokens destroyed here

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        // res is never checked — silent failure
    }
}
```

The same unchecked pattern appears in `withdrawToEthereum` (lines 65–76 of `EvmErc20.sol` and lines 66–77 of `EvmErc20V2.sol`).

The `ExitToNear` precompile (`engine-precompiles/src/native.rs`) can return `Err(ExitError)` under several reachable conditions:

- The `recipient` account ID string exceeds `MAX_INPUT_SIZE` (1 024 bytes), causing input validation to fail.
- The NEP-141 ↔ ERC-20 mapping lookup (`get_nep141_from_erc20`) fails because the token is not registered.
- The precompile is called in static context (`ERR_INVALID_IN_STATIC`).
- The precompile is called via `delegatecall` (`ERR_INVALID_IN_DELEGATE`).

When SputnikVM executes a precompile that returns `Err`, the EVM `call` instruction sets its success flag to 0 but does **not** revert the outer execution frame. Because the Solidity code never checks `res`, the outer transaction completes successfully, the `_burn` is committed, and no NEAR-side promise is ever scheduled.

This is structurally identical to the reported `_burnGroupBalances` bug: an operation (burn) completes on one tracking layer while the corresponding operation on the other layer (NEP-141 release) is silently skipped, with no post-condition check to detect the inconsistency.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

A user who triggers a failed precompile call loses their ERC-20 tokens with no recovery path:

- The ERC-20 burn is committed and irreversible within the EVM state.
- No NEAR promise is created, so no NEP-141 tokens are transferred to the recipient.
- The total supply of the ERC-20 decreases while the NEP-141 supply held by the Aurora contract remains unchanged, creating a permanent accounting divergence.
- There is no refund mechanism in the non-`error_refund` build path (the `#[cfg(not(feature = "error_refund"))]` branch sets `refund: None`).

---

### Likelihood Explanation

**Medium.**

The most straightforward trigger requires no special privilege:

- Any token holder can call `withdrawToNear` with a `recipient` byte string longer than 1 024 bytes. The precompile enforces `MAX_INPUT_SIZE = 1_024` and will reject the input, but the burn has already executed.
- A user who mistypes or programmatically constructs an invalid NEAR account ID (e.g., containing forbidden characters) will hit the same failure.
- The `ExitToEthereum` path is similarly reachable: if the ERC-20 contract is deployed but its NEP-141 mapping has not yet been registered, `get_nep141_from_erc20` returns an error, the precompile fails, and the burn is still committed.

No admin access, key compromise, or governance capture is required.

---

### Recommendation

After the assembly `call`, require that `res` equals 1; otherwise revert the entire transaction so the burn is rolled back:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

Apply the same fix to `withdrawToEthereum` in both `EvmErc20.sol` and `EvmErc20V2.sol`. Additionally, consider moving the `_burn` call to **after** the precompile call succeeds (checks-effects-interactions order), or using a two-phase commit pattern where the burn is only finalised once the NEAR-side promise resolves.

---

### Proof of Concept

1. Deploy `EvmErc20` on Aurora (or use an existing mirror token).
2. Alice holds 1 000 tokens.
3. Alice calls `withdrawToNear` with `recipient = bytes(1025)` (a 1 025-byte string, exceeding `MAX_INPUT_SIZE`).
4. `_burn(Alice, 1000)` executes — Alice's ERC-20 balance drops to 0.
5. The precompile rejects the oversized input and returns `ExitError`; the EVM `call` returns `res = 0`.
6. The assembly block does not check `res`; the Solidity function returns normally.
7. The transaction is mined successfully. Alice's ERC-20 balance is 0. No NEP-141 tokens are transferred. Alice's funds are permanently frozen. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5)
