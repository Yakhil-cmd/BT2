### Title
Precision Loss in `native_per_pubdata` Calculation Causes Operator to Receive Less Fee Than Expected - (File: `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

### Summary

For every L2 transaction that generates pubdata, the operator (coinbase) receives less fee than expected due to compounding integer truncation in the native resource accounting pipeline. The root cause is `wrapping_div` (floor division) used to compute `native_per_pubdata`, which then propagates through a second truncating division in `delta_gas`, producing a "division before multiplication" precision loss directly analogous to the external report.

### Finding Description

The native resource accounting for L2 transactions follows this pipeline:

**Step 1 — `native_per_pubdata` is computed with floor division:**

In `validation_impl.rs` line 142:
```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
``` [1](#0-0) 

`wrapping_div` is floor division. The correct value is `pubdata_price / native_price` (exact). The truncation error is up to `(native_price − 1)` native units per pubdata byte.

**Step 2 — `native_used` is smaller than it should be:**

In `gas_helpers.rs`, the native charged for pubdata is:
```rust
let native = current_pubdata_spent
    .checked_mul(native_per_pubdata)
    .ok_or(out_of_native_resources!())?;
``` [2](#0-1) 

Because `native_per_pubdata` is underestimated, less native is consumed, so `native_used` is smaller than the true cost.

**Step 3 — `delta_gas` is computed with a second truncating division:**

In `refund_calculation.rs` line 72:
```rust
let delta_gas = if native_per_gas == 0 {
    0
} else {
    (native_used / native_per_gas) as i64 - (gas_used as i64)
};
``` [3](#0-2) 

`native_used / native_per_gas` is again floor division. The already-underestimated `native_used` is divided by `native_per_gas`, truncating again. The resulting `delta_gas` is smaller than it should be.

**Step 4 — Operator receives less fee:**

`gas_used` is incremented by `delta_gas` only if positive. A smaller `delta_gas` means a smaller `gas_used`, which means a larger refund to the user and less fee paid to the operator. [4](#0-3) 

The same `wrapping_div` pattern also appears in the API helper: [5](#0-4) 

Note that `native_per_gas` itself uses `div_ceil` (rounds up), which is correct and consistent with the user getting a fair native limit. The inconsistency is that `native_per_pubdata` uses floor division, causing systematic undercharging for pubdata. [6](#0-5) 

### Impact Explanation

The operator (coinbase) receives less fee than expected on every L2 transaction that generates pubdata and where native resource consumption is the bottleneck (i.e., `delta_gas > 0`). The undercharge compounds across two truncations:

- Per pubdata byte: up to `(native_price − 1)` native units undercharged
- Over `N` pubdata bytes: up to `N × (native_price − 1)` native units undercharged
- Translated to gas: `floor(N × (native_price − 1) / native_per_gas)` gas units undercharged

**Concrete example** (`pubdata_price = 1000`, `native_price = 3`, `gas_price = 9`, `N = 1000` pubdata bytes):
- `native_per_gas = ceil(9/3) = 3`
- `native_per_pubdata` (floor) = `333` vs correct `334`
- `native_used` undercharge: `1000 × 1 = 1000` native units
- `delta_gas` undercharge: `floor(1000 / 3) = 333` gas units
- Fee undercharge: `333 × 9 = 2997` tokens per transaction

This loss is non-trivial and accumulates across all pubdata-generating transactions.

### Likelihood Explanation

**High.** Every L2 transaction that writes to storage or emits logs generates pubdata. The precision loss is deterministic and systematic — it occurs on every such transaction without any special conditions. Any unprivileged user submitting a normal EVM transaction triggers this code path.

### Recommendation

Replace `wrapping_div` with `div_ceil` for `native_per_pubdata` in `validation_impl.rs` to be consistent with how `native_per_gas` is computed:

```diff
- let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
+ let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
      .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

Apply the same fix in `api/src/helpers.rs` line 427.

Additionally, consider using `div_ceil` in the `delta_gas` computation in `refund_calculation.rs`:
```diff
- (native_used / native_per_gas) as i64 - (gas_used as i64)
+ (native_used.div_ceil(native_per_gas)) as i64 - (gas_used as i64)
```

### Proof of Concept

The following scenario demonstrates the undercharge:

1. Deploy a contract that writes to 100 storage slots (generating ~3200 bytes of pubdata).
2. Set block context: `pubdata_price = 1000`, `native_price = 3`, `gas_price = 9`.
3. Execute the transaction.
4. Observe that `gas_used` is lower than expected because `native_per_pubdata = 333` instead of `334`, causing `native_used` to be underestimated by `3200` native units, and `delta_gas` to be underestimated by `floor(3200/3) = 1066` gas units.
5. The operator receives `1066 × 9 = 9594` fewer tokens than the correct fee.

The existing test `test_pubdata_native_calculation_overflow` in `tests/instances/transactions/src/lib.rs` already exercises the pubdata native charging path and can be extended to assert exact fee amounts. [7](#0-6)
