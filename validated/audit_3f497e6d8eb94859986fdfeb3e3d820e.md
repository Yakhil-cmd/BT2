### Title
Unverified Participant Shares in CKD Protocol Allow Malicious Participant to Corrupt CKD Output — (File: `src/confidential_key_derivation/protocol.rs`)

### Summary
The CKD coordinator blindly accumulates `(norm_big_y, norm_big_c)` contributions from every participant with no cryptographic verification that each share is consistent with the sender's committed key share. A single malicious participant can substitute arbitrary group elements, causing the coordinator to output a structurally valid but cryptographically wrong `CKDOutput`, permanently destroying the confidential key for that invocation.

### Finding Description

In `do_ckd_coordinator`, the coordinator collects each participant's `(norm_big_y, norm_big_c)` pair and adds them unconditionally: [1](#0-0) 

The correct contribution from participant `i` is:

```
norm_big_y_i = λ_i · y_i · G₁
norm_big_c_i = λ_i · (H(pk ‖ app_id) · x_i  +  app_pk · y_i)
```

computed inside `compute_signature_share`: [2](#0-1) 

Nothing in the protocol binds `norm_big_c_i` to the participant's actual signing share `x_i`. The coordinator holds only the aggregate public key `pk = x · G₂` (from `KeygenOutput`): [3](#0-2) 

Individual public-key shares `X_i = x_i · G₂` are not stored in `KeygenOutput` and are not transmitted during the CKD round, so the coordinator has no material with which to perform a pairing-based consistency check. The malicious participant's message is accepted as-is: [4](#0-3) 

This is the direct analog of the sandwich-attack root cause: a security-critical aggregate value (`big_c = H(pk,app_id)·x + app_pk·Y`) is computed dynamically from per-participant inputs that any participant can freely manipulate, with no commitment or proof binding those inputs to previously agreed-upon state.

### Impact Explanation

The final CKD output satisfies:

```
big_c − app_sk · big_y  =  H(pk ‖ app_id) · x        (correct)
```

If malicious participant `m` sends `(0, 0)` (identity elements) instead of their correct share, the coordinator computes:

```
big_y'  = big_y  − λ_m · y_m · G₁
big_c'  = big_c  − λ_m · (H(pk,app_id)·x_m + app_pk·y_m)
```

After the app unmasks:

```
big_c' − app_sk · big_y'  =  H(pk ‖ app_id) · (x − λ_m · x_m)
```

This is a wrong point in G₁ — not the intended confidential key. Any data encrypted or derived under the correct key is permanently inaccessible for this CKD invocation. The attack is undetectable: the coordinator cannot distinguish a correct share from a crafted one, and the output is a well-formed `CKDOutput` that passes all structural checks.

**Impact:** High — Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.

### Likelihood Explanation

- Requires only one malicious participant out of `n`.
- No cryptographic capability is needed; sending identity elements or random group elements suffices.
- The attack is silent: no error is raised, and the coordinator returns `Some(ckd_output)` as if the protocol succeeded.
- The malicious participant is not identified or excluded.

**Likelihood:** High.

### Recommendation

1. **Include per-participant public key shares** (`X_i = x_i · G₂`) in `KeygenOutput` or distribute them during the CKD round so the coordinator can perform a pairing-based consistency check:
   ```
   e(norm_big_c_i − app_pk · norm_big_y_i / λ_i,  G₂)
       ==  e(λ_i · H(pk ‖ app_id),  X_i)
   ```
2. **Alternatively**, require each participant to attach a zero-knowledge proof of correct share formation (e.g., a Schnorr-style proof of knowledge of `y_i` and consistency of `norm_big_c_i` with `X_i`).
3. If robustness against malicious participants is out of scope, document this explicitly and enforce that the CKD protocol is only invoked in a setting where all participants are trusted.

### Proof of Concept

```
Setup:
  n = 3 participants; participant P₂ is malicious.
  Honest participants P₁, P₃ compute correct (norm_big_y_i, norm_big_c_i).
  P₂ sends (G₁::identity(), G₁::identity()) to the coordinator.

Coordinator accumulates:
  big_y  = norm_big_y₁ + 0 + norm_big_y₃   (missing P₂'s contribution)
  big_c  = norm_big_c₁ + 0 + norm_big_c₃   (missing P₂'s contribution)

App unmasks:
  big_c − app_sk · big_y
    = H(pk,app_id)·(x − λ₂·x₂)   ≠   H(pk,app_id)·x

Result: coordinator returns Ok(Some(ckd_output)) with no error;
        the confidential key is permanently wrong for this invocation.
``` [5](#0-4)

```rust
async fn do_ckd_coordinator(
    mut chan: SharedChannel,
    participants: ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<CKDOutputOption, ProtocolError> {
    let (mut norm_big_y, mut norm_big_c) =
        compute_signature_share(&participants, me, key_pair, app_id, app_pk, rng)?;

    // Receive everyone's inputs and add them together
    let waitpoint = chan.next_waitpoint();

    for (_, participant_output) in
        recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
    {
        norm_big_y += participant_output.big_y();
        norm_big_c += participant_output.big_c();
    }
    let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
    Ok(Some(ckd_output))
}
