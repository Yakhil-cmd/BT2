### Title
Coordinator-Controlled Randomizer Accepted Without Consensus Validation in RedJubjub Signing - (File: `src/frost/redjubjub/sign.rs`)

---

### Summary

In the RedJubjub FROST signing protocol, each non-coordinator participant blindly accepts a `Randomizer` value sent by the coordinator with no consensus check and no validation that all participants received the same value. A malicious coordinator — a role reachable by any participant in the signing set — can send distinct randomizer values to distinct participants, causing their signature shares to be computed over inconsistent effective keys. The aggregated signature is then cryptographically invalid, permanently denying signing for honest parties in that session. This is the direct analog of the Pickle Finance `execute()` pattern: a privileged role passes arbitrary unvalidated data into a cryptographic operation on behalf of honest parties.

---

### Finding Description

In `do_sign_participant` the participant waits for a `Randomizer` from the coordinator and uses it without any further check:

The coordinator generates the randomizer and distributes it to every participant via individual private channels. Because the distribution is point-to-point and there is no Byzantine Reliable Broadcast (echo-broadcast) step, no participant can verify that every other participant received the same value. The coordinator can trivially send `r₁` to participant A and `r₂ ≠ r₁` to participant B.

The RedJubjub `round2::sign` computes the signature share as:

```
s_i = nonce_i + (x_i + randomizer) · challenge
```

With different randomizers the effective signing keys differ across participants, so the shares do not sum to a valid signature. The coordinator — who collects all shares — observes the failure but honest participants have already consumed their presignature nonces and cannot retry without a fresh presigning phase.

The coordinator function sends the randomizer to all participants before collecting shares: [2](#0-1) 

No echo-broadcast or commitment-then-reveal is used for the randomizer distribution, unlike the DKG phase which does use `verify_commitment_hash` for its distributed values: [3](#0-2) 

The communication layer explicitly supports private point-to-point messages (`send_private`) that are invisible to other participants: [4](#0-3) 

This makes split-view delivery of the randomizer trivially achievable by any coordinator.

---

### Impact Explanation

**High — Corruption of signing outputs / Permanent denial of signing.**

A malicious coordinator sends randomizer `r₁` to a subset of participants and `r₂` to the rest. Every participant behaves honestly and produces a signature share, but the shares are computed over different effective keys (`x_i + r₁` vs `x_i + r₂`). The aggregated value fails the final ECDSA/EdDSA verification check inside `aggregate()`. The presignature nonces are consumed and cannot be reused (reuse would expose the secret key). Honest parties must abort and restart from a fresh presigning phase, which the same malicious coordinator can repeat indefinitely as long as it remains in the signing set.

A secondary impact is key-isolation bypass: sending a zero randomizer causes the signature to be produced for the base (non-rerandomized) key, breaking the unlinkability guarantee that rerandomization is intended to provide.

Both outcomes fall within the allowed High impact: *"Corruption of… sign… outputs so honest parties accept inconsistent… transcripts, or unusable cryptographic outputs"* and *"Permanent denial of signing… for honest parties under valid protocol inputs and documented trust assumptions."*

---

### Likelihood Explanation

The coordinator role is not a fixed trusted party — it is selected from the signing participant set (e.g., at random, as shown in the test utilities). Any participant who is chosen as coordinator can execute this attack with zero cryptographic capability: it only requires sending two different serialized `Randomizer` bytes over the existing private channels. The attack is deterministic, requires no leaked keys, and is repeatable across sessions.

---

### Recommendation

Distribute the randomizer via the existing Byzantine Reliable Broadcast (`echo_broadcast`) primitive already present in `src/protocol/echo_broadcast.rs`, so all honest participants are guaranteed to receive the same value or all abort. Alternatively, apply the commit-then-reveal pattern already used in DKG (`src/crypto/commitment.rs`) — the coordinator commits to the randomizer, broadcasts the commitment to all participants, then reveals the randomizer; each participant verifies the opening before proceeding. [5](#0-4)

### Proof of Concept

1. Instantiate a RedJubjub signing session with participants `{A, B, C}` and malicious coordinator `M`.
2. `M` generates two distinct randomizers `r₁` and `r₂`.
3. `M` sends `r₁` to participant `A` and `r₂` to participants `B` and `C` via `send_private`.
4. Each participant computes `round2::sign(…, randomizer)` with their respective randomizer and sends the share back to `M`.
5. `M` calls `aggregate()` on the collected shares. The aggregated `s = Σ sⱼ` is computed over inconsistent effective keys and fails the internal signature verification.
6. `M` reports failure; honest participants `A`, `B`, `C` have consumed their presignature nonces and cannot retry without a new presigning round.
7. `M` repeats this for every subsequent signing session in which it is selected as coordinator, achieving indefinite denial of signing.

/// Returns a future that executes signature protocol for *a Participant*.
///
/// WARNING: Extracted from FROST documentation:
/// In all of the main FROST ciphersuites, the entire message must be sent
/// to participants. In some cases, where the message is too big, it may be
/// necessary to send a hash of the message instead. We strongly suggest
/// creating a specific ciphersuite for this, and not just sending the hash
/// as if it were the message.
/// For reference, see how RFC 8032 handles "pre-hashing".
async fn do_sign_participant(
    mut chan: SharedChannel,
    threshold: ReconstructionLowerBound,
    me: Participant,
    coordinator: Participant,
    keygen_output: KeygenOutput,
    presignature: PresignOutput,
    message: Vec<u8>,
) -> Result<SignatureOption, ProtocolError> {
    // --- Round 1.
    if coordinator == me {
        return Err(ProtocolError::InvalidInput(
            "the do_sign_participant function cannot be called
            for a coordinator"
                .to_string(),
        ));
    }

    // Receive the Randomizer from the coordinator
    let wait_round_1 = chan.next_waitpoint();
    let randomizer = loop {
        let (from, randomizer): (_, Randomizer) = chan.recv(wait_round_1).await?;
        if from != coordinator {
            continue;
        }
        break randomizer;
    };

    let key_package = construct_key_package(threshold, me, &keygen_output)?;
    let key_package = Zeroizing::new(key_package);
    let nonces = Zeroizing::new(presignature.nonces);
    let signing_package = SigningPackage::new(presignature.commitments_map, &message);
    let signature_share = round2::sign(&signing_package, &nonces, &key_package, randomizer)
        .map_err(|_| ProtocolError::ErrorFrostSigningFailed)?;

    let sign_waitpoint = chan.next_waitpoint();
    chan.send_private(sign_waitpoint, coordinator, &signature_share)?;

    Ok(None)
}
