### Title
ETH Permanently Frozen in `RewardSupplier` When `tick()` Called With `msg.value` and No Pending Mint Requests - (File: `L1/starkware/solidity/stake/RewardSupplier.sol`)

### Summary
The `tick()` function in the L1 `RewardSupplier` contract is `payable` but contains no guard against receiving ETH when there is no minting work to perform. When `amountToMint == 0`, the entire `msg.value` is silently retained by the contract with no refund path and no ETH withdrawal mechanism anywhere in the contract, permanently freezing the caller's ETH.

### Finding Description
`tick()` is a public, permissionless, `payable` function. Its sole purpose is to consume pending L2→L1 mint-request messages, mint STRK, and bridge the tokens back to L2 — paying two cross-chain message fees with `msg.value`.

```solidity
// L1/starkware/solidity/stake/RewardSupplier.sol  lines 107-143
function tick() external payable {
    (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

    if (amountToMint > 0) {
        // ...
        uint256 msgFee = msg.value / 2;
        bridge().depositWithMessage{value: msgFee}(...);   // fee 1
        msgFee = msg.value - msgFee;
        messagingContract().sendMessageToL2{value: msgFee}(...); // fee 2
    }
    // ← no else-branch, no revert, no refund
}
```

When `requiredMinting()` returns `(0, 0)` — the normal steady-state between reward epochs — the `if` block is skipped entirely. Any ETH attached to the call accumulates in the contract's balance. The contract has:

- No `receive()` / `fallback()` function.
- No ETH withdrawal or sweep function.
- No `require(msg.value == 0)` guard on the no-op path.

The only ETH-consuming operations are the two forwarding calls inside the `if` block, which are never reached.

### Impact Explanation
Any ETH sent by a caller to `tick()` when `amountToMint == 0` is permanently frozen inside `RewardSupplier`. There is no on-chain recovery path. This constitutes a **permanent freezing of funds** for every caller who attaches ETH to a no-op `tick()` invocation.

### Likelihood Explanation
`tick()` carries no access control — any Ethereum address may call it. The function is designed to be called by off-chain keepers or bots that monitor the L2→L1 message queue. Because `requiredMinting()` returns zero whenever there are no pending L2 mint-request messages (the common state between epochs), a keeper that naively attaches ETH on every call will lose funds on the majority of invocations. The pattern of "send ETH with every `tick()` call to ensure the fee is covered" is the natural defensive coding style for such keepers, making this a realistic and recurring loss scenario.

### Recommendation
Add a guard at the top of `tick()` that reverts if ETH is sent when there is no minting work:

```solidity
function tick() external payable {
    (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();
    if (amountToMint == 0) {
        require(msg.value == 0, "NO_MINT_REQUIRED_NO_FEE_NEEDED");
        return;
    }
    // ... existing logic
}
```

Alternatively, refund any unused ETH at the end of the function.

### Proof of Concept

1. The L2 `RewardSupplier` has not yet emitted any mint-request messages (e.g., early in an epoch). `requiredMinting()` therefore returns `(0, 0)`.
2. A keeper calls `tick{value: 0.01 ether}()` to ensure the bridge fee is covered.
3. `amountToMint == 0` → the `if` block is skipped.
4. The function returns normally. The contract's ETH balance increases by `0.01 ether`.
5. There is no function in `RewardSupplier` that can move this ETH out.
6. The keeper's `0.01 ETH` is permanently frozen.

        (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

                messagingContract().consumeMessageFromL2(mintRequestSource(), messagePayload);

            // Reuest minting of the requested amount from the mint manager.
            mintManager().mintRequest(token(), amountToMint);

            // Deposit the minted amount onto the bridge to the credit of `mintDestination`.
            uint256 msgFee = msg.value / 2;
