### Title
`PriceVelocityGuardExtension` Uses `block.number` Exclusively, Causing Legitimate Swaps to Revert When Oracle Price Moves More Than `maxChange` Within a Single Block — (`File: metric-periphery/contracts/extensions/PriceVelocityGuardExtension.sol`)

---

### Summary

`PriceVelocityGuardExtension.beforeSwap` computes `blockDiff = block.number - prevBlock` and gates every swap on `changeE18² ≤ maxChange² × (1 + blockDiff)`. When two swaps land in the same block (`blockDiff = 0`), the allowed budget collapses to exactly `maxChange²`. On any chain where `block.number` advances slowly relative to transaction throughput — Arbitrum (L1 block number), zkSync, or simply high-throughput Ethereum/Base blocks with an intra-block oracle update — the guard permanently under-counts elapsed time, causing legitimate swaps to revert with `PriceVelocityExceeded`.

---

### Finding Description

`PriceVelocityGuardExtension` stores `lastUpdateBlock` as a `uint64` holding `block.number` and computes the velocity budget using:

```solidity
uint256 blockDiff = block.number - prevBlock;                          // line 63
uint256 allowedSq = uint256(maxChange) * uint256(maxChange) * (1 + blockDiff); // line 70
if (actualSq > allowedSq) revert PriceVelocityExceeded(...);          // line 72-73
``` [1](#0-0) 

The formula is designed to relax the cap as more blocks pass (`sqrt(1 + blockDiff)` growth). However, `block.number` is the only time dimension used. There is no `block.timestamp` component.

**Intra-block oracle update scenario (Ethereum / Base):**

1. Block N, tx 1: Swap A executes. `lastMidPriceX64 = P1`, `lastUpdateBlock = N`.
2. Block N, tx 2: The oracle pushes a new price `P2` where `|P2 − P1| / P1 > maxChange` (legitimate large market move).
3. Block N, tx 3: Swap B executes. `blockDiff = N − N = 0`, so `allowedSq = maxChange²`. The check fails and the swap reverts even though the oracle price is valid and fresh.

**L2 chains with slow `block.number` (Arbitrum, zkSync):**

On Arbitrum, `block.number` returns the L1 block number (~12 s per increment). Dozens of L2 transactions execute within a single L1 block. Every swap in that window sees `blockDiff = 0`, so the velocity budget never relaxes beyond `maxChange²` regardless of how much real time has elapsed. Any oracle price movement exceeding `maxChange` between two L2 swaps in the same L1 block causes a revert.

The `setLastMidPrice` admin reset function has the same flaw — it also writes `block.number`:

```solidity
s.lastUpdateBlock = uint64(block.number);   // line 32
``` [2](#0-1) 

If the admin calls `setLastMidPrice` in the same block as a subsequent swap, `blockDiff` is still 0 and the guard is still at maximum restriction.

---

### Impact Explanation

All swaps routed through a pool that has `PriceVelocityGuardExtension` registered in `beforeSwap` will revert whenever the oracle mid-price moves by more than `maxChangePerBlockE18` within a single block. This renders the pool's swap functionality completely unusable during any legitimate high-volatility period or intra-block oracle update. LPs cannot rebalance, arbitrageurs cannot correct the pool price, and traders cannot execute — matching the "unusable swap flows" criterion in the allowed impact gate. [3](#0-2) 

---

### Likelihood Explanation

- **Ethereum / Base**: Any block that contains both an oracle price update and a subsequent swap where the price moved more than `maxChange` triggers the bug. During volatile markets this is routine.
- **Arbitrum / zkSync**: Every pair of swaps within the same L1 block sees `blockDiff = 0`. Given Arbitrum's ~250 ms L2 block time vs. ~12 s L1 block time, roughly 48 L2 blocks share the same `block.number`. Any oracle update during that window that moves price by more than `maxChange` permanently blocks swaps until the next L1 block.
- The extension is opt-in but is a first-party periphery contract intended for production use by pool admins who want price-manipulation protection. [4](#0-3) 

---

### Recommendation

Replace `block.number` with `block.timestamp` (seconds) as the time dimension, or use a combined measure. `block.timestamp` advances with every block on all EVM-compatible chains including Arbitrum and zkSync, and is the correct unit for "how much real time has elapsed since the last swap."

```solidity
// In PriceVelocityState: replace lastUpdateBlock with lastUpdateTimestamp
uint64 lastUpdateTimestamp;

// In beforeSwap / setLastMidPrice:
s.lastUpdateTimestamp = uint64(block.timestamp);

// Velocity check (e.g., per-second budget):
uint256 timeDiff = block.timestamp - prevTimestamp;
uint256 allowedSq = uint256(maxChange) * uint256(maxChange) * (1 + timeDiff);
```

This directly mirrors the fix applied in the referenced upstream PR (adding `block.timestamp` to the hash) and eliminates the false-positive revert on all target chains.

    // casting to `uint128` is safe: geometric mid of two uint128 bid/ask quotes fits uint128 (same bound as pool)

    uint128 midPrice = uint128(midPriceX64);

    PriceVelocityState storage s = priceVelocityState[pool_];
    uint128 prevMid = s.lastMidPriceX64;
    uint64 prevBlock = s.lastUpdateBlock;

    s.lastMidPriceX64 = midPrice;
    s.lastUpdateBlock = uint64(block.number);

    if (prevMid != 0) {
      uint64 maxChange = s.maxChangePerBlockE18;
      if (maxChange != 0) {
        uint256 blockDiff = block.number - prevBlock;

        uint256 delta = midPrice > prevMid ? uint256(midPrice - prevMid) : uint256(prevMid - midPrice);

        uint256 changeE18 = (delta * 1e18) / uint256(prevMid);

        uint256 actualSq = changeE18 * changeE18;
        uint256 allowedSq = uint256(maxChange) * uint256(maxChange) * (1 + blockDiff);

        if (actualSq > allowedSq) {
          revert PriceVelocityExceeded(actualSq, allowedSq);
        }
      }
    }

  }
