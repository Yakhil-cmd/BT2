Audit Report

## Title
Cycles Minting Canister Uses Stale ICP/XDR Rate Without Staleness Enforcement in `tokens_to_cycles` - (File: rs/nns/cmc/src/main.rs)

## Summary
The CMC's `tokens_to_cycles` function reads a cached `icp_xdr_conversion_rate` with no check on the age of the stored rate. The rate is updated only via a periodic heartbeat that calls the Exchange Rate Canister (XRC); when the XRC returns errors, the stored rate is silently retained indefinitely. Any unprivileged user can call `notify_top_up`, `notify_create_canister`, or `notify_mint_cycles` during a stale-rate window and receive more cycles per ICP than the current market rate warrants, constituting illegal minting at the protocol's expense.

## Finding Description

`tokens_to_cycles` at `rs/nns/cmc/src/main.rs` L1900–1922 reads `state.icp_xdr_conversion_rate` and uses it directly:

```rust
fn tokens_to_cycles(amount: Tokens) -> Result<Cycles, NotifyError> {
    with_state(|state| {
        let xdr_permyriad_per_icp = state
            .icp_xdr_conversion_rate
            .as_ref()
            .map(|rate| rate.xdr_permyriad_per_icp);
        ...
    })
}
``` [1](#0-0) 

There is no comparison of `rate.timestamp_seconds` against the current canister time. The function only returns an error when the rate is `None`, which cannot happen in practice because `State::default()` initialises `icp_xdr_conversion_rate` with a hardcoded value: [2](#0-1) 

All three public minting entry points call `tokens_to_cycles` unconditionally:
- `process_create_canister` L1932 [3](#0-2) 
- `process_mint_cycles` L1965 [4](#0-3) 
- `process_top_up` L1991 [5](#0-4) 

The rate is refreshed only by the heartbeat via `update_exchange_rate`. On XRC failure (`FailedToRetrieveRate`, `FailedToSetRate`, or `InvalidRate`), `UpdateExchangeRateGuard::schedule_next_attempt` schedules a retry at the next minute but leaves the stored rate unchanged: [6](#0-5) 

`do_set_icp_xdr_conversion_rate` only enforces that a new rate has a strictly greater timestamp than the current one; it imposes no maximum age on the stored rate: [7](#0-6) 

The XRC refresh interval is 5 minutes under normal conditions: [8](#0-7) 

By contrast, NNS Governance already implements a staleness guard (`should_refresh_xdr_rate`) that rejects use of a rate older than one day: [9](#0-8) 

No equivalent guard exists anywhere in the CMC minting path.

## Impact Explanation

When the XRC is unavailable and ICP market price falls, the CMC continues minting cycles at the old (higher) ICP price. Every call to `notify_top_up` or `notify_create_canister` during the stale window mints more cycles than the burned ICP is worth at current market rates. This is illegal minting: cycles are created without corresponding ICP backing, diluting the value of cycles across the entire network. The `base_limiter` caps minting at 150 × 10¹⁵ cycles per hour (`DEFAULT_CYCLES_LIMIT`), which bounds but does not prevent per-transaction mispricing. This matches the **High** bounty impact: "Significant XRC/NNS security impact with concrete user or protocol harm." [10](#0-9) 

## Likelihood Explanation

XRC failure is a documented, integration-tested failure mode (`StablecoinRateTooFewRates`, `CryptoBaseAssetNotFound`, insufficient source counts). The integration test suite explicitly verifies that the CMC retains the old rate when the XRC returns `StablecoinRateTooFewRates`. No special privilege is required: `notify_top_up` and `notify_create_canister` are callable by any principal. The attacker does not need to cause the XRC failure; they only need to observe that `get_icp_xdr_conversion_rate` (a public query) returns a `timestamp_seconds` significantly older than the current time, then submit ICP for conversion during that window.

## Recommendation

Add a maximum staleness check inside `tokens_to_cycles`. Compare `rate.timestamp_seconds` against `ic_cdk::api::time() / 1_000_000_000` (or the `Environment::now_timestamp_seconds()` abstraction already used elsewhere in the CMC). If the stored rate is older than a defined threshold (e.g., 30 minutes to 1 hour), return a retriable `NotifyError::Other` so callers can retry once the rate is refreshed. This mirrors the pattern already used in `should_refresh_xdr_rate` in governance. Optionally, expose the staleness threshold as a governance-configurable parameter.

## Proof of Concept

1. Deploy a mock XRC that returns `ExchangeRateError::StablecoinRateTooFewRates` for all requests.
2. Advance canister time by 60+ minutes (e.g., using `StateMachine::advance_time`).
3. Trigger heartbeats; confirm via `get_icp_xdr_conversion_rate` that `timestamp_seconds` is 60+ minutes old while `xdr_permyriad_per_icp` still reflects the pre-failure rate.
4. Call `notify_top_up` with a valid ICP transfer. Observe that `tokens_to_cycles` succeeds and mints cycles at the stale (higher) rate.
5. Reduce the mock XDR rate to simulate a 40% ICP price drop; confirm the minted cycles still reflect the old rate, yielding ~1.67× the correct cycle count.
6. Repeat until `base_limiter` is exhausted, confirming the per-hour cap is the only bound on total excess minting.

This is directly reproducible using the existing `cycles_minting_canister_with_exchange_rate_canister` integration test harness, which already sets up the mock XRC and advances time.

    fn should_refresh_xdr_rate(&self) -> bool {
        let xdr_conversion_rate = &self.heap_data.xdr_conversion_rate;

        let now_seconds = self.env.now();

        let seconds_since_last_conversion_rate_refresh =
            now_seconds.saturating_sub(xdr_conversion_rate.timestamp_seconds);

        // Return `true` if more than 1 day has passed since the last `xdr_conversion_rate` was
        // updated. This assumes that `xdr_conversion_rate.timestamp_seconds` is rounded down to
        // the nearest day's beginning.
        seconds_since_last_conversion_rate_refresh > ONE_DAY_SECONDS
    }
