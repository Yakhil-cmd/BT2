### Title
`tick()` Is `payable` But Silently Discards `msg.value` When No Mint Requests Are Pending — (`File: L1/starkware/solidity/stake/RewardSupplier.sol`)

---

### Summary

The `tick()` function in `RewardSupplier.sol` is declared `external payable` and uses `msg.value` to pay L1→L2 bridge and messaging fees. However, `msg.value` is only consumed inside the `if (amountToMint > 0)` branch. When no L2 mint requests are pending (`amountToMint == 0`), the function returns silently without spending or refunding the ETH, permanently locking any ETH sent by the caller in the contract. There is no `receive()`, `fallback()`, or ETH-withdrawal function anywhere in the contract or its storage layer.

---

### Finding Description

`tick()` is an unrestricted public function: [1](#0-0) 

It uses `msg.value` only inside the conditional block that executes when `amountToMint > 0`: [2](#0-1) 

Specifically, `msg.value` is split between the bridge deposit call and the `sendMessageToL2` call: [3](#0-2) 

When `amountToMint == 0`, the entire `if` block is skipped. Any ETH attached to the call is accepted by the `payable` function but never forwarded, refunded, or tracked. The `RewardSupplierStorage` contract defines no ETH-recovery mechanism: [4](#0-3) 

---

### Impact Explanation

Any ETH sent to `tick()` when no mint requests are pending is permanently frozen in the `RewardSupplier` contract. Because there is no withdrawal function, the ETH cannot be recovered by the caller or by any admin. This matches the allowed impact: **Permanent freezing of funds**.

---

### Likelihood Explanation

`tick()` has no access control — any public caller can invoke it. A caller who wants to trigger reward minting must supply ETH to cover L1→L2 message fees. A natural race condition exists: a caller reads `requiredMinting()` off-chain, sees pending requests, attaches ETH, and submits the transaction — but another caller's `tick()` transaction is mined first, consuming all pending messages. The second caller's transaction then executes with `amountToMint == 0`, and their ETH is silently lost. This is a realistic, low-effort scenario requiring no privileged access.

---

### Recommendation

Add an explicit guard or refund path for the case where no minting is required:

```solidity
function tick() external payable {
    (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

    if (amountToMint == 0) {
        // Refund any ETH sent when there is nothing to process.
        if (msg.value > 0) {
            (bool ok, ) = msg.sender.call{value: msg.value}("");
            require(ok, "ETH_REFUND_FAILED");
        }
        return;
    }
    // ... rest of existing logic
}
```

Alternatively, remove the `payable` modifier and require callers to pre-fund the contract separately, or add a require at the top: `require(amountToMint > 0, "NOTHING_TO_MINT")` before accepting any ETH.

---

### Proof of Concept

1. L2 reward supplier has zero pending mint-request messages (either initially, or after another caller already processed them).
2. Attacker (or innocent caller) calls `tick{value: 1 ether}()`.
3. `requiredMinting()` returns `(0, 0)`.
4. The `if (amountToMint > 0)` block is skipped entirely.
5. Function returns. The 1 ETH is now held by `RewardSupplier` with no withdrawal path.
6. Caller's ETH is permanently frozen. [5](#0-4)

    function tick() external payable {
        // Check if minting is required, and how much.
        (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

        if (amountToMint > 0) {
            // Prepare the L2->L1 mintRequest message for consumption.
            uint256[] memory messagePayload = new uint256[](1);
            messagePayload[0] = TOKENS_PER_MINT_REQUEST;

            // Consume the mintRequest messages.
            for (uint256 i = 0; i < numMsgsToConsume; i++) {
                messagingContract().consumeMessageFromL2(mintRequestSource(), messagePayload);
            }

            // Reuest minting of the requested amount from the mint manager.
            mintManager().mintRequest(token(), amountToMint);

            // Deposit the minted amount onto the bridge to the credit of `mintDestination`.
            uint256 msgFee = msg.value / 2;
            bridge().depositWithMessage{value: msgFee}(
                token(),
                amountToMint,
                mintDestination(),
                new uint256[](0)
            );
            emit ConsumedL2MintRequests(numMsgsToConsume, amountToMint);

            // Send a totalSupply update to L2MintCurve.
            msgFee = msg.value - msgFee;
            messagePayload[0] = IERC20(token()).totalSupply();
            messagingContract().sendMessageToL2{value: msgFee}(
                mintingCurve(),
                UPDATE_TOTAL_SUPPLY_SELECTOR,
                messagePayload
            );
        }
    }
