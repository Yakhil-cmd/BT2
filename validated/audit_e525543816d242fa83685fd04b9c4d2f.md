The direct analog exists. `setPoolAdminFeeDestination` changes the fee recipient without first flushing accrued fees, exactly mirroring the Cooler pattern where state is mutated before pending amounts are processed.

Let me verify the full picture of the `setPoolProtocolFee` function to confirm the collect-first pattern is consistently applied elsewhere:

### Title
`setPoolAdminFeeDestination` Redirects Accrued Admin Fees to New Destination Without Prior Collection — (`File: metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`setPoolAdminFeeDestination` updates `poolAdminFeeDestination[pool]` immediately without first flushing accrued admin fees. Every other fee-config mutator in the factory (`setPoolAdminFees`, `setPoolProtocolFee`) calls `collectFees` before changing state. The missing flush causes all fees earned under the old destination to be paid to the new destination on the next `collectPoolFees` call, permanently depriving the original recipient of their owed tokens.

---

### Finding Description

`MetricOmmPoolFactory` maintains two types of accrued admin fees on each pool:

1. **Spread surplus** — the token balance above `binTotals.scaledToken0/1` that accumulates from the bid/ask spread on every swap.
2. **Notional fees** — tracked explicitly in `notionalFeeToken0Scaled` / `notionalFeeToken1Scaled` on the pool.

Both are distributed to `poolAdminFeeDestination[pool]` when `collectFees` is called.

`setPoolAdminFees` and `setPoolProtocolFee` both follow the collect-first pattern:

```solidity
// setPoolAdminFees — lines 418-425

    c.protocolSpreadFeeE6, c.adminSpreadFeeE6,
    c.protocolNotionalFeeE8, c.adminNotionalFeeE8,
    poolAdminFeeDestination[pool]   // ← old destination used here
);
// ... then update config
``` [1](#0-0) 

`setPoolAdminFeeDestination` does not:

```solidity
function setPoolAdminFeeDestination(address pool, address newAdminFeeDestination)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (newAdminFeeDestination == address(0)) revert InvalidAdminFeeDestination();
    poolAdminFeeDestination[pool] = newAdminFeeDestination;   // ← no collect first
    emit PoolAdminFeeDestinationUpdated(pool, newAdminFeeDestination);
}
``` [2](#0-1) 

After the destination is overwritten, the next call to `collectPoolFees` (which is permissionless) reads the new destination and sends all accumulated admin fees — including those earned before the change — to the new address:

```solidity
function collectPoolFees(address pool) external override nonReentrant {
    PoolFeeConfig memory c = poolFeeConfig[pool];
    IMetricOmmPoolCollectFees(pool).collectFees(
        c.protocolSpreadFeeE6, c.adminSpreadFeeE6,
        c.protocolNotionalFeeE8, c.adminNotionalFeeE8,
        poolAdminFeeDestination[pool]   // ← now the new address
    );
}
``` [3](#0-2) 

Inside `collectFees` on the pool, the admin share of both spread surplus and notional fees is transferred to whatever `adminFeeDestination_` is passed in at call time: [4](#0-3) 

There is no snapshot, checkpoint, or per-epoch accounting. The entire accumulated balance is paid to the current destination.

---

### Impact Explanation

The old admin fee destination (e.g., a treasury contract or multisig) permanently loses all admin fees that accrued before the destination change. The tokens are not burned or locked — they are transferred to the new destination — but the original recipient has no recourse to recover them. This is a direct, irreversible loss of owed protocol revenue for the old fee destination.

---

### Likelihood Explanation

Any routine treasury rotation, multisig migration, or fee-destination update by a pool admin triggers the bug. The pool admin is a semi-trusted role that is expected to perform these operations. `collectPoolFees` is permissionless, so a keeper or MEV bot can front-run the destination change and trigger collection after the overwrite, but even without front-running the next organic collection call causes the loss. No special conditions or malicious intent are required.

---

### Recommendation

Collect accrued fees to the old destination before updating `poolAdminFeeDestination`, mirroring the pattern used in `setPoolAdminFees` and `setPoolProtocolFee`:

```diff
function setPoolAdminFeeDestination(address pool, address newAdminFeeDestination)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (newAdminFeeDestination == address(0)) revert InvalidAdminFeeDestination();
+   PoolFeeConfig memory c = poolFeeConfig[pool];
+   IMetricOmmPoolCollectFees(pool).collectFees(
+       c.protocolSpreadFeeE6,
+       c.adminSpreadFeeE6,
+       c.protocolNotionalFeeE8,
+       c.adminNotionalFeeE8,
+       poolAdminFeeDestination[pool]   // flush to old destination first
+   );
    poolAdminFeeDestination[pool] = newAdminFeeDestination;
    emit PoolAdminFeeDestinationUpdated(pool, newAdminFeeDestination);
}
```

### Proof of Concept

1. Pool is deployed with `adminFeeDestination = treasuryA`. Swaps accumulate spread surplus and notional fees over time.
2. Pool admin calls `setPoolAdminFeeDestination(pool, treasuryB)`. `poolAdminFeeDestination[pool]` is overwritten; no fees are collected.
3. Anyone calls `collectPoolFees(pool)`. The factory reads `poolAdminFeeDestination[pool] == treasuryB` and passes it to `collectFees`.
4. `collectFees` computes the full admin share of both spread surplus and `notionalFeeToken0/1Scaled` and transfers it to `treasuryB`.
5. `treasuryA` receives nothing despite having earned those fees during all swaps in step 1. The loss equals the entire admin fee share accumulated between the last collection and the destination change. [2](#0-1) [5](#0-4) [6](#0-5)
