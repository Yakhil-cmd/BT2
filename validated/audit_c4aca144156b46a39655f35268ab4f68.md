### Title
L1→L2 Deposit Minted to `from` Address Without L2 Recipient Override — (`basic_bootloader/src/bootloader/transaction_flow/zk/process_l1_transaction.rs`)

---

### Summary

In `process_l1_transaction`, the deposited base-token value (`to_transfer = total_deposited − max_fee_commitment`) is unconditionally minted to the transaction's `from` address on L2. No mechanism exists for the submitter to specify a different L2 recipient for this value. For account-abstraction wallets (e.g., Safe multisig) whose L1 address is not controlled by the same parties on L2, the deposited funds are permanently lost.

---

### Finding Description

When an L1→L2 priority transaction is processed, `execute_l1_transaction_and_notify_result` computes the value to transfer as:

```
to_transfer = total_deposited − (gas_price × gas_limit)
```

and immediately mints it to `from` (the L1 sender address) on L2:

```rust
// process_l1_transaction.rs line 631-644
if to_transfer > U256::ZERO || Config::SIMULATION {
    resources.with_infinite_ergs(|inf_resources| {
        mint_base_token::<S, Config>(
            system, system_functions, memories.reborrow(),
            &to_transfer,
            &from,          // ← always the L1 sender address
            l1_chain_id, inf_resources, tracer, validator,
        )
    })...
}
``` [1](#0-0) 

The `from` value is taken directly from the ABI-encoded transaction field with no override path:

```rust
let from = transaction.from.read();
``` [2](#0-1) 

The protocol does provide a separate `refund_recipient` field (`reserved[1]`) that lets the submitter direct the gas-refund portion to a different L2 address:

```rust
let refund_recipient = u256_to_b160_checked(transaction.reserved[1].read());
``` [3](#0-2) 

This field is exposed in the `ZKsyncL1Tx` struct:

```rust
pub refund_recipient: Address,
``` [4](#0-3) 

However, **no equivalent override exists for the initial deposit value**. `reserved[2]` and `reserved[3]` are both hardcoded to `U256::ZERO` and are ignored by the bootloader: [5](#0-4) 

The `mint_base_token` helper calls `transfer_from_treasury`, which credits the L2 balance of the supplied address: [6](#0-5) 

If the main transaction body succeeds, the frame is committed and the minted balance at `from` is permanent:

```rust
system.finish_global_frame(reverted.then_some(&rollback_handle))?;
``` [7](#0-6) 

---

### Impact Explanation

An AA wallet (Safe multisig, ERC-4337 account, etc.) deployed at address `X` on L1 is typically **not** deployed at the same address on L2, or is deployed with different owners/bytecode. When such a wallet submits an L1→L2 priority transaction with `to_mint > 0` and the transaction succeeds, the deposited base tokens are credited to address `X` on L2 — an address the original L1 owner does not control. The funds are permanently inaccessible. There is no recovery path because the bootloader commits the balance unconditionally on success.

---

### Likelihood Explanation

Safe multisig and ERC-4337 wallets are widely used on Ethereum L1. Any such user who bridges base tokens via an L1→L2 priority transaction is affected. The condition is: (1) the user submits from an AA wallet, (2) `to_mint > 0`, and (3) the L2 transaction succeeds. All three conditions are routine in normal bridge usage. No privileged access or attacker cooperation is required — the loss is self-inflicted by the protocol's address assumption.

---

### Recommendation

Introduce a `value_recipient` field (e.g., using `reserved[2]`) in the L1 transaction encoding, analogous to the existing `refund_recipient` (`reserved[1]`). When non-zero, use it as the L2 recipient for the initial deposit mint instead of `from`:

```rust
// In execute_l1_transaction_and_notify_result:
let value_recipient_raw = transaction.reserved[2].read();
let value_recipient = if value_recipient_raw.is_zero() {
    from
} else {
    u256_to_b160_checked(value_recipient_raw)
};

mint_base_token::<S, Config>(
    system, system_functions, memories.reborrow(),
    &to_transfer,
    &value_recipient,   // ← user-specified L2 recipient
    ...
)
```

This mirrors the existing `refund_recipient` pattern and gives AA wallet users a safe L2 address to receive their deposited value.

---

### Proof of Concept

1. Alice controls a Safe multisig at address `0xSafe` on L1. The same address on ZKsync OS L2 is either undeployed or owned by a different party.
2. Alice submits an L1→L2 priority transaction: `from = 0xSafe`, `to_mint = 10 ETH`, `gas_limit = 100_000`, `gas_price = 1 gwei`.
3. `total_deposited = 10 ETH`, `max_fee_commitment = 0.0001 ETH`, `to_transfer = 9.9999 ETH`.
4. The bootloader calls `mint_base_token(..., &to_transfer, &from, ...)` at line 634–644, crediting `0xSafe` on L2 with 9.9999 ETH.
5. The main tx body succeeds; the frame is committed.
6. Alice has no way to access `0xSafe` on L2. The 9.9999 ETH is permanently lost.
7. The `refund_recipient` field only covers the unused-gas refund (0.0001 ETH or less), not the deposited value. [8](#0-7)
