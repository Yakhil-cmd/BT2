### Title
Unprotected `new` Initialization Allows Front-Running to Seize Full Engine Ownership - (File: `engine/src/contract_methods/admin.rs`)

---

### Summary

The `new` function that initializes the Aurora Engine's core state performs no caller validation. Any NEAR account can race to call it before the legitimate deployer, setting an arbitrary `owner_id` and `key_manager`. Because the owner controls contract upgrades, a successful front-run yields complete, irrecoverable control of the engine and all funds it holds.

---

### Finding Description

The public NEAR entrypoint `new()` is exported from `engine/src/lib.rs` with no access restriction: [1](#0-0) 

It delegates directly to `contract_methods::admin::new`, which only checks whether state already exists, then unconditionally writes whatever `owner_id`, `key_manager`, and `chain_id` the caller supplies: [2](#0-1) 

There is no check of `env.predecessor_account_id()` anywhere in this function. The `EngineState` written to storage is fully attacker-controlled: [3](#0-2) 

`NewCallArgs` (V2–V4) all carry `owner_id` and `key_manager` fields that flow directly into the stored state: [4](#0-3) 

Every subsequent privileged operation — `set_owner`, `stage_upgrade`, `upgrade`, `set_upgrade_delay_blocks`, `pause_precompiles`, `resume_precompiles` — gates on `require_owner_only`, which compares `state.owner_id` against `predecessor_account_id`: [5](#0-4) 

Once the attacker owns `state.owner_id`, they pass every one of those gates.

---

### Impact Explanation

**Critical — Direct theft of all user funds / permanent freeze.**

The `upgrade` function, gated only on `require_owner_only`, deploys an arbitrary WASM binary to the engine account: [6](#0-5) 

An attacker who controls `owner_id` can replace the entire engine contract with code that drains every ETH balance, ERC-20 mirror balance, and NEP-141 token held by the engine, or simply brick the contract permanently. All user funds at rest in the Aurora Engine are at risk.

---

### Likelihood Explanation

**High.** The window of vulnerability is the gap between contract deployment and the first successful call to `new`. On NEAR, transactions are publicly visible before finality. A monitoring bot needs only to detect the deployment transaction and submit a `new` call with attacker-controlled args in the same or next block. No special privilege, leaked key, or social engineering is required — only the ability to submit a NEAR transaction.

---

### Recommendation

Add a caller check at the top of `new` before any state is written. The standard pattern for NEAR upgradeable contracts is to require that the predecessor equals the contract's own account ID (enforcing that initialization is done atomically in the same batch as deployment):

```rust
pub fn new<I: IO + Copy, E: Env>(mut io: I, env: &E) -> Result<(), ContractError> {
    if state::get_state(&io).is_ok() {
        return Err(b"ERR_ALREADY_INITIALIZED".into());
    }
    // Require initialization to be performed by the contract account itself
    // (i.e., via a batch transaction: DeployContract + FunctionCall("new")).
    if env.predecessor_account_id() != env.current_account_id() {
        return Err(b"ERR_NOT_ALLOWED".into());
    }
    // ... rest of initialization
}
```

This mirrors the protection already present in the XCC router's `initialize`, which validates the caller before accepting state: [7](#0-6) 

---

### Proof of Concept

1. Attacker monitors NEAR for a transaction deploying the Aurora Engine WASM to account `aurora`.
2. In the next block, attacker submits a call to `aurora` → `new` with:
   ```json
   {
     "chain_id": 1313161554,
     "owner_id": "attacker.near",
     "upgrade_delay_blocks": 0,
     "key_manager": "attacker.near",
     "initial_hashchain": null
   }
   ```
3. `state::get_state` returns `Err` (no state yet) → the guard at line 57 passes.
4. No caller check exists → `state::set_state` writes `owner_id = "attacker.near"`.
5. The legitimate deployer's `new` call now returns `ERR_ALREADY_INITIALIZED`.
6. Attacker calls `stage_upgrade` with malicious WASM, then `upgrade` — both pass `require_owner_only` because `state.owner_id == "attacker.near"`.
7. Malicious contract drains all ETH and ERC-20 balances from the engine. [2](#0-1) [8](#0-7) [6](#0-5)

pub fn new<I: IO + Copy, E: Env>(mut io: I, env: &E) -> Result<(), ContractError> {
    if state::get_state(&io).is_ok() {
        return Err(b"ERR_ALREADY_INITIALIZED".into());
    }

    let input = io.read_input().to_vec();
    let args = NewCallArgs::deserialize(&input).map_err(|_| errors::ERR_BORSH_DESERIALIZE)?;

    let initial_hashchain = args.initial_hashchain();
    let state: EngineState = args.into();

    if let Some(block_hashchain) = initial_hashchain {
        let block_height = env.block_height();
        let mut hashchain = Hashchain::new(
            state.chain_id,
            env.current_account_id(),
            block_height,
            block_hashchain,
        );

        hashchain.add_block_tx(
            block_height,
            function_name!(),
            &input,
            &[],
            &Bloom::default(),
        )?;
        crate::hashchain::save_hashchain(&mut io, &hashchain)?;
    }

    state::set_state(&mut io, &state)?;
    Ok(())
}

                name: "state_migration".to_string(),
                args: vec![],
