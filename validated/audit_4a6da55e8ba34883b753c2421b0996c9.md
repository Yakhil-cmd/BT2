Audit Report

## Title
Precision Loss in `convert_tokens_to_string_representation` Causes Incorrect Amount Display in ICRC-21 Consent Messages - (`File: packages/icrc-ledger-types/src/icrc21/responses.rs`)

## Summary
`convert_tokens_to_string_representation` converts an arbitrary-precision `Nat` token amount to `f64` before formatting it into the ICRC-21 `GenericDisplayMessage` consent string. Because `f64` has only 53 bits of mantissa, any raw token amount exceeding 2^53 (~9×10^15) is silently rounded to the nearest representable floating-point value. The consent message shown to the user therefore displays a different number than the one actually being transferred or approved, breaking the core security guarantee of the ICRC-21 consent flow.

## Finding Description
The vulnerable function at `packages/icrc-ledger-types/src/icrc21/responses.rs` lines 318–327:

```rust
fn convert_tokens_to_string_representation(
    tokens: Nat,
    decimals: u8,
) -> Result<String, Icrc21Error> {
    let tokens = tokens.0.to_f64().ok_or(Icrc21Error::GenericError { ... })?;
    Ok(format!("{}", tokens / 10_f64.pow(decimals)))
}
```

`BigUint::to_f64()` returns `Some` for any value up to ~1.8×10^308, so the error path is never triggered for realistic inputs. However, `f64` has only 53 bits of mantissa (~15–16 significant decimal digits). Any `Nat` larger than 2^53 is rounded to the nearest representable `f64` before the division and formatting occur.

This function is called from four public methods on `ConsentMessage` for the `GenericDisplayMessage` branch:
- `add_amount` (L113) — transfer/transfer_from amount
- `add_fee` (L141) — transaction fee
- `add_allowance` (L186) — requested allowance in approve
- `add_existing_allowance` (L212) — existing allowance in approve

All four paths are reachable by any unprivileged caller submitting a `ConsentMessageRequest` with `device_spec = GenericDisplay`. The `FieldsDisplayMessage` branch is unaffected because it passes the raw `Nat` through `nat_to_u64`, which correctly errors on overflow rather than silently rounding.

Concrete precision loss for 8-decimal tokens (ICP, ckBTC): amount `10_000_000_000_000_001` (10^16 + 1 e8s) → `to_f64()` rounds to `10000000000000000.0` (the final unit is dropped) → divided by 10^8 → displayed as `"100000000"` instead of `"100000000.00000001"`. For 18-decimal tokens (ckETH-style), any amount above ~9×10^15 raw units (~0.009 ETH) is affected; consecutive f64 values near 10^18 differ by 256 raw units, so amounts within a 256-wei window display identically.

Note: the submitted claim that Rust's `{}` format emits scientific notation (e.g. `"1e18"`) for large `f64` values is incorrect — Rust's `Display` for `f64` always uses decimal notation. The precision-loss claim is nonetheless accurate and independently sufficient.

The correct exact-integer approach is already used in `rs/bitcoin/ckbtc/minter/src/updates/icrc21.rs` lines 211–222 (`format_amount`) and the analogous ckDOGE minter, both of which perform integer division and modulo on the raw `u64` amount without any floating-point conversion.

## Impact Explanation
The ICRC-21 consent message is the sole security mechanism by which a user verifies what they are signing before authorizing a transfer or approval. Displaying a rounded amount breaks this guarantee: a user relying on the `GenericDisplayMessage` to confirm the exact amount will be shown a value that differs from the one encoded in the actual transaction argument. This constitutes a concrete integrity failure in the ck-token/ICRC ledger consent flow, matching the allowed High impact: "Significant Chain Fusion, ck-token, ledger, Rosetta, boundary/API, XRC, Internet Identity, NNS, SNS, or infrastructure security impact with concrete user or protocol harm." The absolute financial discrepancy per transaction is small (sub-cent for typical amounts), which constrains severity to the lower end of the High range or upper Medium.

## Likelihood Explanation
No special privilege is required. Any caller can invoke `icrc21_canister_call_consent_message` on any deployed ICRC-1/ICRC-2 ledger canister (ICP ledger, ckBTC, ckETH, SNS ledgers) with a crafted `amount` field exceeding 2^53 raw units. For 18-decimal tokens this threshold is ~0.009 ETH — well within normal user balances. The attack is trivially repeatable and requires no on-chain state or prior setup.

## Recommendation
Replace the lossy `f64` conversion with exact integer arithmetic on the underlying `BigUint`, mirroring the pattern already used in the ckBTC and ckDOGE minters:

```rust
use num_bigint::BigUint;
use num_traits::Zero;

fn convert_tokens_to_string_representation(
    tokens: Nat,
    decimals: u8,
) -> Result<String, Icrc21Error> {
    let divisor = BigUint::from(10u64).pow(decimals as u32);
    let whole = &tokens.0 / &divisor;
    let frac  = &tokens.0 % &divisor;
    if frac.is_zero() {
        Ok(whole.to_string())
    } else {
        let frac_str = format!("{:0>width$}", frac, width = decimals as usize);
        let trimmed = frac_str.trim_end_matches('0');
        Ok(format!("{}.{}", whole, trimmed))
    }
}
```

This requires no new dependencies (`num_bigint` and `num_traits` are already imported in the file).

## Proof of Concept
Add the following unit test to `packages/icrc-ledger-types/src/icrc21/responses.rs`:

```rust
#[test]
fn test_precision_loss_above_2_pow_53() {
    use candid::Nat;
    // 10^16 + 1 raw units, 8 decimals
    let amount = Nat::from(10_000_000_000_000_001_u64);
    let result = convert_tokens_to_string_representation(amount, 8).unwrap();
    // With f64: rounds to "100000000" (drops the 0.00000001 fractional part)
    // Correct:  "100000000.00000001"
    assert_eq!(result, "100000000.00000001",
        "got: {result}");
}
```

Running this test against the current implementation will fail, demonstrating the precision loss. Alternatively, call `icrc21_canister_call_consent_message` on any deployed ICRC-1 ledger with `method = "icrc1_transfer"`, `amount = 10_000_000_000_000_001`, and `device_spec = GenericDisplay`; the returned message will contain `**Amount:** \`100000000 <SYMBOL>\`` instead of `**Amount:** \`100000000.00000001 <SYMBOL>\``.
