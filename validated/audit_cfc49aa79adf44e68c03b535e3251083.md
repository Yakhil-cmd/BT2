### Title
CKD Coordinator Accepts Participant Contributions Without Any Proof of Correctness, Allowing Malicious Participant to Corrupt the Derived Confidential Key - (File: src/confidential_key_derivation/protocol.rs)

### Summary

The Confidential Key Derivation (CKD) coordinator blindly sums `(big_y, big_c)` contributions received from each participant with no zero-knowledge proof or consistency check. Any single malicious participant can send an arbitrarily crafted pair, causing the coordinator to produce a corrupted `CKDOutput`. Every honest party that calls `unmask` on this output will silently derive an incorrect confidential key. Every other protocol in the library (DKG, triple generation, presigning) requires participants to attach ZK proofs to their contributions; CKD is the sole exception.

### Finding Description

In `do_ckd_coordinator` (`src/confidential_key_derivation/protocol.rs`, lines 35–57), the coordinator receives one `CKDOutput` per participant and unconditionally adds the two group elements together:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
```

No proof is required that `big_c_i = λ_i · (x_i · H(pk ‖ app_id) + y_i · A)` for the participant's actual secret share `x_i` and the discrete log `y_i` of `big_y_i`. The coordinator performs no check whatsoever before incorporating each contribution into the aggregate.

The correct output satisfies:

```
C - app_sk · Y  =  Σ λ_i · x_i · H(pk ‖ app_id)  =  msk · H(pk ‖ app_id)
```

A malicious participant `P_m` can instead send `big_c'_m = big_c_m + Δ` (for any group element `Δ` of their choice) while keeping `big_y'_m = big_y_m`. The coordinator accumulates this without complaint. The resulting `CKDOutput` satisfies:

```
C' - app_sk · Y  =  msk · H(pk ‖ app_id) + Δ
```

`unmask` in `CKDOutput` simply computes `C - app_sk · Y` and returns the result; it has no way to detect the corruption. The caller receives a silently wrong confidential key.

**Contrast with every other protocol in the library:**

- DKG (`src/dkg.rs`, lines 452–469): every commitment is accompanied by a proof of knowledge (`verify_proof_of_knowledge`) and a hash-commitment binding (`verify_commitment_hash`).
- Triple generation (`src/ecdsa/ot_based_ecdsa/triples/generation.rs`, lines 368–389, 483–491, 596–604): every round's contribution is guarded by `dlog::verify` or `dlogeq::verify` before the value is accumulated.

CKD is the only protocol that accumulates participant-supplied group elements with zero verification.

### Impact Explanation

A single malicious participant (one of the `n` signers who holds a key share) can corrupt the CKD output accepted by the coordinator and all downstream consumers. The `unmask` call silently returns an incorrect confidential key. Any TEE application or downstream system that relies on this key will operate on wrong secret material. Because the corruption is undetectable at the protocol level, honest parties have no way to identify the attack or recover the correct key without re-running the protocol with the malicious participant excluded.

This matches the allowed High impact: **Corruption of CKD outputs so honest parties accept unusable cryptographic outputs**.

### Likelihood Explanation

Any participant in a CKD session can mount this attack. No special privilege beyond holding a valid key share is required. The attacker simply sends a crafted `(big_y, big_c)` pair in the single protocol round. The attack is deterministic, requires no timing window, and succeeds with probability 1 against the current code.

### Recommendation

Require each participant to attach a zero-knowledge proof of correct formation alongside their `(big_y, big_c)` contribution. Concretely, the participant must prove in zero knowledge that:

1. `big_y = y_i · G` (discrete log of `big_y` is known), and
2. `big_c = x_i · H(pk ‖ app_id) + y_i · A` (the ciphertext is correctly formed with the participant's actual secret share `x_i` and the same `y_i`).

This is a standard ElGamal encryption proof of correct formation (a `dlogeq`-style proof over two generators `G` and `A`). The coordinator must verify this proof for every received contribution before adding it to the aggregate, mirroring the pattern already used in triple generation and DKG.

### Proof of Concept

**Setup:** 3 participants, threshold 2. Participant `P_2` is malicious.

**Honest execution** (`compute_signature_share` in `src/confidential_key_derivation/protocol.rs`, lines 148–181):
- `P_2` computes `big_y_2 = y_2 · G`, `big_c_2 = λ_2 · (x_2 · H(pk ‖ app_id) + y_2 · A)`.

**Attack:** `P_2` instead sends `big_y'_2 = big_y_2` and `big_c'_2 = big_c_2 + Δ` where `Δ` is any non-identity point (e.g., the generator `G`).

**Coordinator path** (`do_ckd_coordinator`, lines 50–55):
```rust
// No proof check — Δ is silently absorbed
norm_big_y += participant_output.big_y();   // unchanged
norm_big_c += participant_output.big_c();   // now includes Δ
```

**Result:** `ckd_output.unmask(app_sk)` returns `msk · H(pk ‖ app_id) + Δ` instead of `msk · H(pk ‖ app_id)`. The honest coordinator and any consumer of the output have no indication that the key is wrong. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
