### Title
Unspent Native ETH Stranded in Router After Partial Fill via `priceLimitX64` — (`metric-periphery/contracts/MetricOmmSimpleRouter.sol`)

---

### Summary

`exactInputSingle` accepts native ETH (`msg.value`) when `tokenIn` is WETH, but never refunds the portion not consumed when a `priceLimitX64` causes a partial fill. The leftover ETH sits in the router and is immediately extractable by any caller via the public `refundETH()`.

---

### Finding Description

`exactInputSingle` is `payable` and forwards the full `params.amountIn` as the `amountSpecified` to the pool's `swap()`. When `priceLimitX64` is a non-open sentinel the pool may consume only a fraction of that amount before halting at the price boundary.

The pool's swap callback fires with `amount0Delta`/`amount1Delta` reflecting only the **actual** tokens consumed. `_justPayCallback` calls `pay(token, payer, pool, actualConsumed)`:

```solidity
// PeripheryPayments.sol L74-77
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();   // wraps only `actualConsumed`
        IERC20(WETH).safeTransfer(recipient, value);
    }
```

`value` here is the actual amount the pool requested — not `params.amountIn`. The difference (`params.amountIn − actualConsumed`) remains as raw ETH in the router.

`exactInputSingle` then returns with no refund:

```solidity
// MetricOmmSimpleRouter.sol L81-86
int128 out = MetricOmmSwapResults.extractAmountOut(...);
amountOut = MetricOmmSwapInputs.int128ToUint128(out);
if (amountOut < params.amountOutMinimum) revert InsufficientOutput(...);
_clearExpectedCallbackPool();
// ← no refundETH() call
```

The public `refundETH()` in `PeripheryPayments` sends the entire ETH balance to `msg.sender`:

```solidity
// PeripheryPayments.sol L58-63
function refundETH() external payable override {
    uint256 balance = address(this).balance;
    if (balance > 0) {
        _transferETH(msg.sender, balance);
    }
}
```

Any address — including a MEV bot watching the mempool — can call `refundETH()` in the same block and drain the stranded ETH.

Note: `exactInput` (multi-hop) is **not** affected because it uses `MetricOmmSwapPath.openLimit(zeroForOne)` (no price limit) and reverts on any partial fill via `InvalidInputAmountAtHop`. `exactOutputSingle` is also unaffected because the pool always delivers exactly the requested output or reverts.

---

### Impact Explanation

A user selling native ETH with a price limit set loses the unspent portion of their ETH to the first caller of `refundETH()`. The loss is proportional to how far the price limit is from the current market price. In the worst case (price limit immediately hit), nearly the entire `msg.value` is lost. This is a direct loss of user principal with no recovery path.

---

### Likelihood Explanation

- `exactInputSingle` is the primary single-hop entry point for retail users.
- Setting a `priceLimitX64` is a documented, supported feature of the interface.
- Native ETH swaps are the most common user pattern (no pre-approval required).
- MEV bots routinely monitor routers for stranded ETH; `refundETH()` is a well-known extraction vector.
- No special privilege or setup is required; any user who sets a price limit and sends ETH triggers the condition.

---

### Recommendation

Add an automatic ETH refund at the end of `exactInputSingle` (and defensively in `exactOutputSingle`) when the input token is WETH:

```solidity
function exactInputSingle(ExactInputSingleParams calldata params) external payable returns (uint256 amountOut) {
    _checkDeadline(params.deadline);
    uint128 priceLimitX64 = MetricOmmSwapPath.normalizePriceLimit(params.zeroForOne, params.priceLimitX64);

    _setNextCallbackContext(params.pool, CALLBACK_MODE_JUST_PAY, msg.sender, params.tokenIn);
    (int128 amount0Delta, int128 amount1Delta) = IMetricOmmPoolActions(params.pool)
      .swap(params.recipient, params.zeroForOne,
            MetricOmmSwapInputs.asAmountSpecifiedIn(params.amountIn),
            priceLimitX64, "", params.extensionData);
    int128 out = MetricOmmSwapResults.extractAmountOut(params.zeroForOne, amount0Delta, amount1Delta);
    amountOut = MetricOmmSwapInputs.int128ToUint128(out);
    if (amountOut < params.amountOutMinimum) revert InsufficientOutput(amountOut, params.amountOutMinimum);

    _clearExpectedCallbackPool();

+   // Refund any unspent native ETH (occurs when priceLimitX64 causes a partial fill)
+   if (address(this).balance > 0) {
+       _transferETH(msg.sender, address(this).balance);
+   }
}
```

Alternatively, document that callers **must** append a `refundETH()` call via `multicall` when using native ETH with a price limit — but the automatic refund is strongly preferred to prevent accidental loss.

---

### Proof of Concept

1. Alice calls `exactInputSingle{value: 1 ether}(ExactInputSingleParams({ tokenIn: WETH, amountIn: 1e18, priceLimitX64: <mid-market price>, ... }))`.
2. The pool executes a partial swap, consuming 0.6 ETH before the price limit is reached.
3. The swap callback fires; `pay(WETH, Alice, pool, 0.6e18)` wraps and transfers 0.6 ETH to the pool.
4. `exactInputSingle` returns. The router holds 0.4 ETH.
5. A MEV bot calls `router.refundETH()` in the same block and receives 0.4 ETH.
6. Alice has lost 0.4 ETH with no recourse.

`exactInputSingle` — no post-swap ETH refund: [1](#0-0)

`refundETH` — public, sends entire ETH balance to caller: [3](#0-2) 

`_justPayCallback` — passes only the pool-requested amount to `pay`: [4](#0-3)
