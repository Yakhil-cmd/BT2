### Title
Malicious Participant Can Corrupt CKD Output by Sending Arbitrary Unverified Shares to Coordinator — (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

The CKD coordinator aggregates participant-provided elliptic curve points `(big_y, big_c)` directly into the final derived-key output with no zero-knowledge proof or cryptographic consistency check. Any single malicious participant can inject arbitrary group elements, causing every honest party to accept a silently corrupted CKD result.

---

### Finding Description

In `do_ckd_coordinator`, the coordinator collects one `CKDOutput` per participant and unconditionally accumulates the values:

```rust
// src/confidential_key_derivation/protocol.rs  lines 50-55
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
``` [1](#0-0) 

The honest computation that each participant is *supposed* to perform is defined in `compute_signature_share`:

```
norm_big_y = λ_i · y_i · G
norm_big_c = λ_i · (x_i · H(pk ‖ app_id) + y_i · app_pk)
``` [2](#0-1) 

There is **no proof** that a participant used their actual secret share `x_i`, chose a valid `y_i`, or formed the ElGamal ciphertext correctly. The coordinator has no way to distinguish a correctly formed `(norm_big_y, norm_big_c)` from an arbitrary pair of group elements.

The final output is assembled and returned as authoritative:

```rust
let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
Ok(Some(ckd_output))
``` [3](#0-2) 

The `unmask` function then computes `big_C − app_sk · big_Y` and returns the result as the confidential derived key, with no further integrity check: [4](#0-3)

### Impact Explanation

A malicious participant sends `(big_y_m, big_c_m)` — arbitrary G1 points — instead of their correct share. The coordinator computes:

```
big_Y_out = Σ_{honest} λ_i·y_i·G  +  big_y_m
big_C_out = Σ_{honest} λ_i·(x_i·H + y_i·A)  +  big_c_m
```

After `unmask(app_sk)`:

```
result = big_C_out − app_sk · big_Y_out
       = msk · H(pk ‖ app_id)  +  (big_c_m − app_sk · big_y_m)
```

The additive offset `(big_c_m − app_sk · big_y_m)` is attacker-controlled and non-zero in general, so the derived key is wrong. Honest parties receive and use this corrupted output with no indication of failure. This matches the allowed impact:

> **High: Corruption of CKD outputs so honest parties accept inconsistent or unusable cryptographic outputs.**

---

### Likelihood Explanation

- Any single participant in the protocol can mount this attack; no special privilege is required beyond being a listed participant.
- The attack requires only sending two arbitrary G1 points instead of the correct computation — trivially achievable by any participant who deviates from the protocol.
- There is no retry or detection mechanism; the coordinator returns the corrupted value immediately.

---

### Recommendation

Add a zero-knowledge proof of correct share computation to each participant's message. Concretely, each participant should prove in zero knowledge:

1. Knowledge of `y_i` such that `norm_big_y = λ_i · y_i · G`.
2. That `norm_big_c = λ_i · x_i · H(pk ‖ app_id) + λ_i · y_i · app_pk`, consistent with their committed public verification share `x_i · G2` from the DKG output.

A standard Chaum–Pedersen / Schnorr-style proof over G1 suffices for (1); (2) requires additionally binding the proof to the participant's known public share. The coordinator must verify all proofs before accumulating any contribution.

---

### Proof of Concept

1. Honest participants P1, P2 and malicious participant P3 run `ckd(...)`.
2. P3 calls `chan.send_private(waitpoint, coordinator, &(arbitrary_g1_point_y, arbitrary_g1_point_c))` instead of the correct `compute_signature_share` result.
3. The coordinator's loop at lines 50–55 adds `arbitrary_g1_point_y` and `arbitrary_g1_point_c` to the running sums without any check.
4. `CKDOutput::new(norm_big_y, norm_big_c)` is returned to the caller.
5. The application calls `ckd_output.unmask(app_sk)` and obtains `msk · H(pk ‖ app_id) + (arbitrary_g1_point_c − app_sk · arbitrary_g1_point_y)` — a wrong key — with no error or warning.
6. Any downstream use of this key (e.g., decryption, authentication) silently fails or produces attacker-influenced output.

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
