### Title
WalletConnect Commands Execute Against Logged-Out Wallet Due to No-Op `handleLogOut` — (`packages/core/src/components/Auth/AuthProvider.tsx`)

### Summary

`handleLogOut` is an intentional no-op. Because it issues no RPC call to the daemon, the backend wallet service remains logged in to the previous fingerprint after the user clicks "Log Out." `WalletConnectProvider` stays mounted at the app root and its `session_request` listener remains active. Both the renderer-side fingerprint guard in `useWalletConnectCommand` and the main-process guard in `dispatchPairRequest` derive the "currently logged-in fingerprint" by querying the daemon live — which still returns the old fingerprint. All guards therefore pass, and a paired dApp can execute commands (including `chia_sendTransaction`) against the wallet the user believes they have logged out of.

### Finding Description

**Step 1 — `handleLogOut` is a confirmed no-op.** [1](#0-0) 

The comment explicitly acknowledges the backend remains logged in. No RPC call is made; `setFingerprint` is not cleared.

**Step 2 — `WalletConnectProvider` is mounted at the application root and never unmounts on navigation.** [2](#0-1) 

The `session_request` event listener registered inside `WalletConnectProvider` remains active regardless of which route the user navigates to after logout. [3](#0-2) 

**Step 3 — The renderer-side fingerprint guard reads the daemon's live fingerprint.** [4](#0-3) 

`currentFingerprint` comes from `useGetLoggedInFingerprintQuery()`, which calls `get_logged_in_fingerprint` on the daemon. Because `handleLogOut` issued no logout RPC, the daemon still reports the old fingerprint X. `pair.fingerprint` is also X (set at pairing time). The condition `currentFingerprint !== pair.fingerprint` evaluates to `false`, so no error is thrown.

**Step 4 — The main-process guard in `dispatchPairRequest` also queries the daemon live and reaches the same conclusion.** [5](#0-4) 

`getLoggedInFingerprint()` calls `get_logged_in_fingerprint` on the daemon directly. [6](#0-5) 

Since the daemon is still logged in to fingerprint X, `loggedInFingerprint === pair.fingerprint` and the guard passes.

**Step 5 — For pairs with bypass enabled, `dispatchPairRequest` skips the confirmation dialog entirely.** [7](#0-6) 

The user set "always allow" for `chia_sendTransaction` at pairing time. After logout, the dApp sends the command, all guards pass, and the transaction is broadcast with zero user interaction.

Even without bypass, the confirmation dialog appears for a wallet the user believes is logged out — the logout security boundary is violated in either case.

### Impact Explanation

**Critical (bypass-enabled pairs):** A paired WalletConnect dApp can silently broadcast a `chia_sendTransaction` (or any other bypassed command) against the wallet the user has "logged out" of, with no confirmation dialog and no indication to the user. This constitutes unauthorized transaction signing and broadcast.

**High (non-bypassed pairs):** The dApp can still trigger a confirmation dialog for the logged-out wallet. The user, believing they have logged out, may be confused into approving an operation against the old wallet, or the dApp can probe wallet state (e.g., `chia_getSyncStatus`, `chia_getPublicKey`) without any logout enforcement.

### Likelihood Explanation

- The attacker only needs a previously approved WalletConnect pairing (obtained legitimately during normal use).
- The window is indefinite: the backend stays logged in until the user explicitly logs in to a different wallet or closes the app. There is no timeout.
- For bypass-enabled pairs, no user interaction is required after the logout click.
- The `LogoutButton` in the GUI navigates to `/` immediately after calling the no-op `logOut()`, giving the user a false sense of security. [8](#0-7) 

### Recommendation

`handleLogOut` must issue a daemon RPC call (e.g., `log_out` or equivalent) to actually deauthenticate the backend wallet service, and must clear the Redux fingerprint cache. Until the backend exposes a proper logout endpoint, the GUI should at minimum disconnect all active WalletConnect sessions and revoke all pairs on logout, so the `session_request` listener has no valid pairs to dispatch against.

### Proof of Concept

1. Open the Chia GUI and log in to wallet with fingerprint X.
2. Pair a WalletConnect dApp and grant it `chia_sendTransaction` with "always allow" (bypass).
3. Click "Log Out" in the GUI — the app navigates to `/`.
4. From the dApp, immediately send a `session_request` for `chia_sendTransaction` with `fingerprint: X`.
5. Observe: `processSessionRequest` → `handleProcess` → `currentFingerprint === X === pair.fingerprint` → guard passes → `dispatchAsPair` → `dispatchPairRequest` → `loggedInFingerprint === X === pair.fingerprint` → bypass skips confirmation → `sendCommand('send_transaction', ...)` reaches the daemon → transaction broadcast.
6. Assert: the daemon's transaction pool contains the submitted spend bundle, and no confirmation dialog was shown to the user.
