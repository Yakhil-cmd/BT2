### Title
Pool Admin Bypasses `maxAdminSpreadFeeE6` Cap via Unchecked `setPoolBinAdditionalFees` — (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

The factory enforces `maxAdminSpreadFeeE6` and `maxAdminNotionalFeeE8` caps in `setPoolAdminFees`, but applies **no cap check** in `setPoolBinAdditionalFees`. A pool admin can therefore bypass the owner-imposed fee ceiling by routing excess fees through per-bin additional fees, charging traders amounts that exceed the protocol-intended maximum.

---

### Finding Description

`setPoolAdminFees` correctly validates both fee dimensions against the owner-set caps: [1](#0-0) 

```solidity
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (newAdminNotionalFeeE8 > maxAdminNotionalFeeE8) revert AdminFeeTooHigh();
```

`setPoolBinAdditionalFees`, callable by the same pool admin, passes values directly to the pool with **no cap validation**: [2](#0-1) 

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
  external override nonReentrant onlyPoolAdmin(pool)
{
  IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

The pool's `setBinAdditionalFees` only validates the bin index, not the fee magnitudes: [3](#0-2) 

During every swap, the per-bin additional fee is added directly on top of the oracle-derived base fee: [4](#0-3) 

```solidity
params.baseFeeX64 + Math.mulDiv(binState.addFeeSellE6, ONE_X64, 1e6)
```

The `uint16` type allows `addFeeBuyE6` / `addFeeSellE6` up to **65 535** (≈ 6.55 % in E6 units). The owner can set `maxAdminSpreadFeeE6` to any value from 0 up to `HARD_MAX_SPREAD_FEE_E6 = 200 000`: [5](#0-4) 

Whenever the owner sets `maxAdminSpreadFeeE6` below 65 535 (e.g., 1 000 = 0.1 %), the pool admin can still push per-bin fees to the full `uint16` ceiling, completely circumventing the cap.

---

### Impact Explanation

Every swap routed through a bin with inflated `addFeeBuyE6`/`addFeeSellE6` charges traders a fee that exceeds the owner-sanctioned maximum. The excess fee accrues as spread surplus inside the pool and is collected by the admin via `collectFees`. This is a direct, quantifiable loss of trader principal — the same class of harm as the M-22 analog (a check enforced on one path, absent on another, enabling a position that should be forbidden).

The `maxAdminSpreadFeeE6` cap is the only mechanism preventing a semi-trusted pool admin from extracting value beyond protocol limits. Its bypass via bin fees renders the cap meaningless.

---

### Likelihood Explanation

- The pool admin is a **valid, non-privileged** actor relative to the fee-cap invariant; the caps exist precisely to bound what the admin can do.
- No timelock, no multi-sig, no secondary approval is required to call `setPoolBinAdditionalFees`.
- The bypass is available **immediately** after pool creation and can be applied to every bin independently.
- The owner lowering `maxAdminSpreadFeeE6` (a routine governance action) is the trigger that makes the gap exploitable; the pool admin need only wait for that event.

---

### Recommendation

Add the same cap guard in `setPoolBinAdditionalFees` that exists in `setPoolAdminFees`. Because bin additional fees are additive with the global admin spread fee, the combined total should be bounded:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
  external override nonReentrant onlyPoolAdmin(pool)
{
  if (addFeeBuyE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
  if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
  IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

Alternatively, expose a dedicated `maxAdminBinFeeE6` cap and validate against it, keeping global and per-bin limits independently configurable but always enforced.

---

### Proof of Concept

1. Factory owner calls `setFeeCaps(…, newMaxAdminSpreadFeeE6 = 1_000, …)` — capping admin spread fees at 0.1 %.
2. Pool admin calls `setPoolAdminFees(pool, 1_000, 0)` — complies with the cap.
3. Pool admin calls `setPoolBinAdditionalFees(pool, 0, 65_535, 65_535)` — **no revert**; bin 0 now carries an additional 6.55 % buy/sell fee.
4. Any trader swapping through bin 0 pays ≈ 6.55 % extra on top of the oracle spread, far above the 0.1 % ceiling the owner intended.
5. The surplus is collected by the pool admin via `collectPoolFees`, extracting value directly from traders. [6](#0-5) [2](#0-1) [3](#0-2)
