### Title
Hardcoded `L1_TX_NATIVE_PRICE` Causes Native Resource Over-Allocation for L1→L2 Priority Transactions — (`basic_bootloader/src/bootloader/constants.rs`)

---

### Summary

`L1_TX_NATIVE_PRICE` is hardcoded at `10` in production bootloader code and carries an explicit `TODO` acknowledging the value is not yet calibrated. For every L1→L2 priority transaction, the native resource budget is computed as `gas_price / 10` instead of `gas_price / native_price` (the operator-set, dynamically calibrated value used for all L2 transactions). When the operator's `native_price` exceeds `10` — the normal production case — L1→L2 transactions receive a native resource budget that is `native_price / 10` times larger than what they economically justify, allowing an unprivileged L1 sender to force the prover to perform far more work than was paid for.

---

### Finding Description

For L2 transactions, `native_per_gas` is computed dynamically:

```rust
// validation_impl.rs line 107
let native_price = system.get_native_price();   // operator-set per block
let native_per_gas = gas_price.div_ceil(native_price);
``` [1](#0-0) 

For L1→L2 priority transactions, a hardcoded constant is used instead:

```rust
// process_l1_transaction.rs line 453-454
// For L1->L2 txs, we use a constant native price to avoid censorship.
let native_price = L1_TX_NATIVE_PRICE;   // = 10, hardcoded
let native_per_gas = gas_price.div_ceil(native_price);
``` [2](#0-1) 

The constant itself carries an explicit unresolved TODO:

```rust
// Default native price for L1->L2 transactions.
// TODO (EVM-1157): find a reasonable value for it.
pub const L1_TX_NATIVE_PRICE: U256 = U256::from_limbs([10, 0, 0, 0]);
``` [3](#0-2) 

From `native_per_gas`, two downstream quantities are derived:

1. `native_prepaid_from_gas = native_per_gas * gas_limit` — the total native resource budget for the transaction.
2. `native_per_pubdata = gas_per_pubdata * native_per_gas` — the pubdata budget. [4](#0-3) 

Both are inflated by the factor `native_price / 10` relative to what the same gas price would yield for an L2 transaction.

The computational native is capped at `MAX_NATIVE_COMPUTATIONAL = 2^35`: [5](#0-4) 

Excess above the cap is placed in `withheld` resources, which can still be spent on pubdata: [6](#0-5) 

So both the computational proving budget and the pubdata budget are over-allocated relative to the fee paid.

---

### Impact Explanation

**Resource accounting bug / Economic DoS of the prover.**

If the operator sets `native_price = 1000` (a plausible production value reflecting actual RISC-V proving cycles), an L1→L2 transaction with `gas_price = 1000` receives:

- L2 path: `native_per_gas = 1000 / 1000 = 1`
- L1→L2 path: `native_per_gas = 1000 / 10 = 100` → **100× over-allocation**

The prover must execute up to `MAX_NATIVE_COMPUTATIONAL = 2^35 ≈ 34 billion` native cycles per transaction regardless of the fee paid. An attacker who submits many L1→L2 transactions with moderate gas prices forces the prover to perform orders-of-magnitude more work than the collected fees justify, causing:

- **Economic loss for the operator**: proving costs exceed fees collected per transaction.
- **Block native resource starvation**: L1→L2 transactions consume the block's native budget at a fraction of the cost compared to L2 transactions, crowding out legitimate L2 traffic.
- **Pubdata budget inflation**: `native_per_pubdata` is also inflated, allowing L1→L2 transactions to consume more pubdata budget than their fee warrants.

---

### Likelihood Explanation

An unprivileged user can submit L1→L2 priority transactions on L1 with any `gas_price` and `gas_limit`. No privileged access, oracle manipulation, or key compromise is required. The attacker pays L1 gas to submit the transaction; the economic viability of the attack scales with the ratio `native_price / 10`. In production, where `native_price` is expected to be significantly above `10`, the attacker receives a large multiple of native resources per unit of fee paid, making the attack economically rational.

---

### Recommendation

Replace the hardcoded `L1_TX_NATIVE_PRICE = 10` with a value that is either:

1. **Dynamically derived** from the operator-provided `native_price` at block execution time (with a floor to prevent censorship), or
2. **Calibrated** to a value that reflects actual proving costs and is updated via governance when proving costs change.

At minimum, resolve `TODO (EVM-1157)` before production deployment. A reasonable interim approach is to use `max(system.get_native_price(), CENSORSHIP_FLOOR_NATIVE_PRICE)` so that L1→L2 transactions cannot be censored by an operator setting an arbitrarily high `native_price`, while still preventing the over-allocation described above.
