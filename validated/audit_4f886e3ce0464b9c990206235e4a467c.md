### Title
Signature-Invalid Invoke Admitted to Mempool via `skip_stateful_validations` Triggered by Any Pooled Transaction â€” (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

`skip_stateful_validations` skips `__validate__` (the account signature check) for any invoke with `nonce=1` and `account_nonce=0` whenever `account_tx_in_pool_or_recent_block` returns `true`. That function returns `true` for **any** transaction from the sender address in the pool â€” not only a `deploy_account`. An attacker who observes that address A has any pending transaction in the mempool (a `deploy_account`, or a future-nonce invoke that already passed its own `__validate__`) can submit a `nonce=1` invoke with a garbage signature for address A and have it admitted without signature verification.

---

### Finding Description

The gateway stateful validation path in `extract_state_nonce_and_run_validations` is:

1. Read `account_nonce` from state.
2. Call `run_pre_validation_checks` â†’ `validate_state_preconditions` (nonce range, resource bounds) â†’ `validate_by_mempool` (nonce-too-old / duplicate-nonce / fee-escalation only, **no signature**) â†’ `skip_stateful_validations`.
3. Call `run_validate_entry_point` with `skip_validate` from step 2. [1](#0-0) 

`skip_stateful_validations` returns `true` when all three conditions hold:

```
tx is Invoke  AND  tx.nonce() == 1  AND  account_nonce == 0
AND  account_tx_in_pool_or_recent_block(sender) == true
``` [2](#0-1) 

`account_tx_in_pool_or_recent_block` returns `true` if the address appears in **either** `tx_pool` (any pooled tx) **or** `state` (any recently committed tx): [3](#0-2) 

The code comment itself acknowledges the broader trigger: *"It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction **or transactions with future nonces that passed validations**."* [4](#0-3) 

The flaw: a future-nonce invoke (e.g., `nonce=2`) that passed its own `__validate__` for address A being in the pool does **not** imply that a `nonce=1` invoke from address A carries a valid signature. The two transactions can have entirely different signers/keys.

`validate_by_mempool` (called before `skip_stateful_validations`) only checks nonce ordering and fee escalation â€” it never inspects the signature: [5](#0-4) [6](#0-5) 

When `skip_validate=true`, `run_validate_entry_point` sets `execution_flags.validate = false`, so `validate_tx` returns `Ok(None)` immediately without calling the account's `__validate__` entry point: [7](#0-6) [8](#0-7) 

---

### Impact Explanation

A signature-invalid invoke transaction is admitted to the mempool. This satisfies the **High** impact criterion: *"Mempool/gateway/RPC admission accepts invalid transactions."*

Concrete consequences:
- The attacker's garbage-signature `nonce=1` invoke occupies the `(address, nonce=1)` slot in the pool.
- If fee escalation is enabled, the attacker can outbid the legitimate `nonce=1` invoke and evict it.
- If fee escalation is disabled, the legitimate `nonce=1` invoke is rejected with `DuplicateNonce`.
- The admitted invalid tx will fail during sequencing (signature check at execution time), but the damage to the legitimate user's nonce=1 slot has already occurred.

---

### Likelihood Explanation

The mempool is observable (transactions are public). Any attacker can:
1. Watch for accounts with a pending `deploy_account` or any future-nonce invoke in the pool.
2. Craft a `nonce=1` invoke with a garbage signature for that address.
3. Submit it to the gateway.

No privileged access is required. The only precondition is that the target address has `account_nonce=0` on-chain and at least one transaction in the pool â€” a condition that is routinely true for every new account deployment.

---

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a check that is specific to `deploy_account` transactions. The mempool should expose a dedicated query such as `deploy_account_tx_in_pool(address) -> bool` that only returns `true` when the pooled transaction for that address is a `DeployAccount` type. This preserves the intended UX (deploy_account + invoke submitted together) while closing the bypass for arbitrary pooled transactions.

---

### Proof of Concept

**Scenario A â€” deploy_account in pool (most common):**

1. Legitimate user submits `deploy_account` for address A â†’ admitted to mempool (`tx_pool.contains_account(A) = true`).
2. Attacker submits `Invoke { sender: A, nonce: 1, signature: [0xdeadbeef] }`.
3. Gateway reads `account_nonce = 0` from state.
4. `validate_by_mempool`: nonce=1 â‰Ą 0, no duplicate â†’ passes.
5. `skip_stateful_validations`: nonce=1 âś“, account_nonce=0 âś“, `account_tx_in_pool_or_recent_block(A)` = true âś“ â†’ returns `true`.
6. `run_validate_entry_point(skip_validate=true)` â†’ `execution_flags.validate = false` â†’ `__validate__` not called.
7. Attacker's garbage-signature invoke is admitted to the mempool.

**Scenario B â€” future-nonce invoke in pool:**

1. Account A is deployed on-chain with `nonce=0` (contract exists, no txs executed yet).
2. Legitimate user submits `Invoke { sender: A, nonce: 2, signature: valid }` â†’ `skip_stateful_validations` returns `false` (nonceâ‰ 1) â†’ `__validate__` runs â†’ passes â†’ admitted to pool.
3. Attacker submits `Invoke { sender: A, nonce: 1, signature: [0xdeadbeef] }`.
4. `skip_stateful_validations`: nonce=1 âś“, account_nonce=0 âś“, `account_tx_in_pool_or_recent_block(A)` = true (nonce=2 tx is in pool) âś“ â†’ returns `true`.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
