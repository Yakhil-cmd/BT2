### Title
Deployment Authorization Bypass via Unrestricted Contract Updates — (`fvm/environment/contract_updater.go`)

### Summary
Flow's restricted contract deployment feature enforces an authorization whitelist for *initial* contract deployments but explicitly skips that check for *updates* to already-deployed contracts. Any account that previously deployed a contract — whether before restrictions were enabled or while it was on the authorized list — can later replace that contract with arbitrary malicious code without any authorization check.

### Finding Description
`ContractUpdaterImpl.SetContract` in `fvm/environment/contract_updater.go` is the single FVM entry point for both deploying and updating Cadence contracts. The function checks `ContractExists` to distinguish the two cases and only enforces the deployment authorization check when the contract does not yet exist:

```go
// Initial contract deployments must be authorized by signing accounts.
//
// Contract updates are always allowed.
exists, err := updater.accounts.ContractExists(location.Name, flow.Address(location.Address))
...
if !exists && !updater.isAuthorizedForDeployment(signingAccounts) {
    return fmt.Errorf("deploying contract failed: ...")
}
// No authorization check for updates — falls through unconditionally
updater.draftUpdates[location] = ContractUpdate{Location: location, Code: code}
``` [1](#0-0) 

`isAuthorizedForDeployment` consults the on-chain `ContractDeploymentAuthorizedAddressesPath` whitelist when `RestrictedDeploymentEnabled()` is true: [2](#0-1) 

The whitelist is populated via `SetContractDeploymentAuthorizersTransaction` and stored at `authorizedAddressesToDeployContracts` in the service account: [3](#0-2) 

Because the `exists` branch is never re-authorized, the whitelist is permanently bypassed for any contract that has already been deployed.

### Impact Explanation
The deployment restriction is the protocol's primary mechanism for keeping malicious Cadence contracts off the network. Once bypassed, an attacker who controls any account with a previously-deployed contract can:

1. Replace the contract body with arbitrary Cadence code (e.g., code that drains resources from callers, emits fraudulent events, or corrupts shared state).
2. All existing importers of that contract will silently execute the new malicious code on their next transaction, because Cadence resolves contracts by address+name at execution time.

This is a direct unauthorized account mutation / transaction authorization bypass with concrete on-chain asset impact.

### Likelihood Explanation
The scenario is realistic and requires no privileged access beyond owning an account with a previously-deployed contract:

- **Scenario A (restriction tightened after deployment):** Deployment was unrestricted at network genesis or during an open period. Any account that deployed a contract during that window retains permanent update rights. The service account later enables restriction via `SetIsContractDeploymentRestrictedTransaction`, but all pre-existing contracts remain freely upgradeable.
- **Scenario B (account removed from whitelist):** An account is added to the authorized list, deploys a contract, and is later removed from the list (e.g., after a security review). The account can still update its contract to malicious code.

The attacker entry path is a standard Cadence transaction signed only by the contract-owning account:
```cadence
transaction {
  prepare(signer: auth(UpdateContract) &Account) {
    signer.contracts.update(name: "MyContract", code: <malicious_code>.utf8)
  }
}
```

No admin keys, no staked nodes, no quorum compromise required.

### Recommendation
Apply the same authorization check to updates as to initial deployments. In `SetContract`, remove the `!exists` guard so that `isAuthorizedForDeployment` is called unconditionally when `RestrictedDeploymentEnabled()` is true:

```go
if !updater.isAuthorizedForDeployment(signingAccounts) {
    return fmt.Errorf("updating/deploying contract failed: %w",
        errors.NewOperationAuthorizationErrorf(...))
}
```

Alternatively, introduce a separate `ContractUpdateAuthorizedAddressesPath` whitelist (mirroring the existing removal whitelist pattern) so that update authorization can be managed independently of initial deployment authorization. [4](#0-3) 

### Proof of Concept

**Step 1 — Deploy a contract while restriction is off (or while authorized):**
```cadence
transaction {
  prepare(signer: auth(AddContract) &Account) {
    signer.contracts.add(name: "Vault", code: "<benign_code>".utf8)
  }
}
```
Signed by `accounts[0]`. Succeeds because `!exists` and `isAuthorizedForDeployment` passes.

**Step 2 — Service account enables deployment restriction:**
```go
blueprints.SetIsContractDeploymentRestrictedTransaction(serviceAddress, true)
``` [5](#0-4) 

`accounts[0]` is NOT on the authorized list.

**Step 3 — Attacker updates the contract to malicious code:**
```cadence
transaction {
  prepare(signer: auth(UpdateContract) &Account) {
    signer.contracts.update(name: "Vault", code: "<malicious_code>".utf8)
  }
}
```
Signed only by `accounts[0]`. In `SetContract`, `exists == true`, so the `!exists && !isAuthorizedForDeployment(...)` guard is skipped entirely. The malicious code is committed to chain. [6](#0-5) 

The update succeeds. All subsequent transactions that import `A.<accounts[0]>.Vault` now execute the malicious implementation.

func (updater *ContractUpdaterImpl) isAuthorizedForDeployment(
	signingAccounts []flow.Address,
) bool {
	if updater.RestrictedDeploymentEnabled() {
		return updater.isAuthorized(
			signingAccounts,
			blueprints.ContractDeploymentAuthorizedAddressesPath)
	}
	return true
}

func SetIsContractDeploymentRestrictedTransaction(serviceAccount flow.Address, restricted bool) (*flow.TransactionBody, error) {
	argRestricted, err := jsoncdc.Encode(cadence.Bool(restricted))
	if err != nil {
		return nil, err
	}

	argPath, err := jsoncdc.Encode(IsContractDeploymentRestrictedPath)
	if err != nil {
		return nil, err
	}

	return flow.NewTransactionBodyBuilder().
		SetScript([]byte(setIsContractDeploymentRestrictedTransactionTemplate)).
		SetPayer(serviceAccount).
		AddAuthorizer(serviceAccount).
		AddArgument(argRestricted).
		AddArgument(argPath).
		Build()
}
