### Title
`new_settlement_layer_chain_id_storage` Not Rolled Back on Frame Revert, Permanently Blocking Legitimate Settlement Layer Chain ID Updates Within a Block - (`basic_system/src/system_implementation/system/io_subsystem.rs`)

---

### Summary

`FullIO::finish_io_frame()` rolls back all sub-storages when a frame is reverted, but omits `new_settlement_layer_chain_id_storage`. Because `NewSettlementLayerChainIdStorage` enforces a strict one-update-per-block invariant, a reverted frame that triggered an update permanently exhausts the update quota for the rest of the block, either DoS-ing a legitimate subsequent update or causing the block to commit a chain-ID change that was supposed to be reverted.

---

### Finding Description

`FullIO` holds six sub-storages. Five of them are snapshotted and rolled back in `start_io_frame` / `finish_io_frame`:

```
storage, transient_storage, logs_storage, events_storage, interop_root_storage
```

`new_settlement_layer_chain_id_storage` is the sixth field but is **absent** from both `FullIOStateSnapshot` and the rollback path:

```rust
// basic_system/src/system_implementation/system/io_subsystem.rs

pub struct FullIOStateSnapshot<M: StorageModel> {
    io: M::StateSnapshot,
    transient: CacheSnapshotId,
    messages: usize,
    events: usize,
    interop_roots: usize,
    // ← new_settlement_layer_chain_id is missing
}

fn start_io_frame(&mut self) -> Result<Self::StateSnapshot, InternalError> {
    let io        = self.storage.start_frame();
    let transient = self.transient_storage.start_frame();
    let messages  = self.logs_storage.start_frame();
    let events    = self.events_storage.start_frame();
    let interop_roots = self.interop_root_storage.start_frame();
    // ← self.new_settlement_layer_chain_id_storage.start_frame() never called
    Ok(FullIOStateSnapshot { io, transient, messages, events, interop_roots })
}

fn finish_io_frame(&mut self, rollback_handle: Option<&Self::StateSnapshot>) -> Result<(), InternalError> {
    self.storage.finish_frame(rollback_handle.map(|x| &x.io))?;
    self.transient_storage.finish_frame(rollback_handle.map(|x| &x.transient))?;
    self.logs_storage.finish_frame(rollback_handle.map(|x| x.messages));
    self.events_storage.finish_frame(rollback_handle.map(|x| x.events));
    self.interop_root_storage.finish_frame(rollback_handle.map(|x| x.interop_roots));
    // ← self.new_settlement_layer_chain_id_storage.finish_frame(...) never called
    Ok(())
}

```rust

---

### Impact Explanation

**Scenario A – DoS on legitimate update (high impact):**

1. Transaction T1 emits `SettlementLayerChainIdUpdated`; the hook calls `update()`, which succeeds and sets `history.value = Some(X)`.
2. T1 reverts (out-of-gas, explicit revert, pubdata overflow, etc.). `finish_io_frame(Some(&snapshot))` is called for all storages **except** `new_settlement_layer_chain_id_storage`, so `history.value` remains `Some(X)`.
3. Transaction T2 (the legitimate service transaction) emits the same event. The hook calls `update()`, which now returns `Err(internal_error!("Tried to update settlement layer chain id more than once in a block"))`.
4. This internal error propagates up and halts block processing, permanently preventing the settlement-layer chain-ID migration for the entire block.

**Scenario B – Phantom chain-ID committed to block output:**

If block finalization reads `new_settlement_layer_chain_id_storage.value()` to include the new chain ID in the block commitment, a reverted T1 causes the block to commit a chain-ID change that was never actually applied to persistent storage, creating a forward/proving divergence.

---

### Likelihood Explanation

The `system_context_event_hook` does not gate on the emitter address. Any EVM contract that emits a log with topic `0x208daf0b...` can trigger `update_settlement_layer_chain_id`. An unprivileged user can:

1. Deploy a contract that emits `SettlementLayerChainIdUpdated(X)` and then reverts.
2. Submit a transaction calling that contract.
3. The hook fires, `new_settlement_layer_chain_id_storage` is updated, the frame rolls back, but the storage is not rolled back.
4. The block's settlement-layer chain-ID update quota is exhausted for the rest of the block.

Even without an unprivileged path, any service transaction that legitimately calls SystemContext and then reverts (e.g., due to out-of-gas) triggers the same stale-state condition.

---

### Recommendation

Add `new_settlement_layer_chain_id_storage` to `FullIOStateSnapshot` and include it in both `start_io_frame` and `finish_io_frame`, mirroring the pattern used for `interop_root_storage`:

```rust
pub struct FullIOStateSnapshot<M: StorageModel> {
    io: M::StateSnapshot,
    transient: CacheSnapshotId,
    messages: usize,
    events: usize,
    interop_roots: usize,
+   new_sl_chain_id: NewSettlementLayerChainIdSnapshotId,
}

fn start_io_frame(&mut self) -> Result<Self::StateSnapshot, InternalError> {
    // ... existing snapshots ...
+   let new_sl_chain_id = self.new_settlement_layer_chain_id_storage.start_frame();
    Ok(FullIOStateSnapshot { io, transient, messages, events, interop_roots, new_sl_chain_id })
}

fn finish_io_frame(&mut self, rollback_handle: Option<&Self::StateSnapshot>) -> Result<(), InternalError> {
    // ... existing rollbacks ...
+   self.new_settlement_layer_chain_id_storage
+       .finish_frame(rollback_handle.map(|x| x.new_sl_chain_id));
    Ok(())
}
```

### Proof of Concept

```
Block execution:
  TX1: deploy contract C { emit SettlementLayerChainIdUpdated(42); revert(); }
  TX2: call C
       → hook fires → new_settlement_layer_chain_id_storage.update(42) succeeds
       → TX2 reverts → finish_io_frame(Some(snap)) called
       → new_settlement_layer_chain_id_storage NOT rolled back → value = Some(42)

  TX3: legitimate service tx calling SystemContext.setSettlementLayerChainId(99)
       → hook fires → new_settlement_layer_chain_id_storage.update(99)
       → guard: self.value().is_some() == true → internal_error! → block halts
```
