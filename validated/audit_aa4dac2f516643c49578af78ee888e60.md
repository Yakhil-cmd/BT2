### Title
Invoke Transaction with Invalid Signature Bypasses Gateway `__validate__` via Overly-Broad `skip_stateful_validations` Check — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function skips the account's `__validate__` entry point (signature verification) for any Invoke transaction with nonce=1 submitted against an undeployed account, whenever `account_tx_in_pool_or_recent_block` returns `true`. The guard is satisfied by the presence of **any** transaction from that account in the mempool — not specifically a `DeployAccount`. An attacker who first submits a valid `DeployAccount` can immediately follow with an Invoke carrying an arbitrary (invalid) signature, and the gateway will admit it into the mempool without ever verifying the signature.

---

### Finding Description

**Relevant code path:**

`run_pre_validation_checks` calls `skip_stateful_validations`, whose return value is forwarded directly to `run_validate_entry_point`:

```
run_pre_validation_checks
  └─ skip_stateful_validations  →  bool
       └─ account_tx_in_pool_or_recent_block(sender)
run_validate_entry_point(skip_validate = <result above>)
  └─ ExecutionFlags { validate: !skip_validate, … }
``` [1](#0-0) 

When `skip_validate = true`, `run_validate_entry_point` constructs `ExecutionFlags` with `validate: false`, so the blockifier's `StatefulValidator::validate` call never invokes the account's `__validate__` entry point — the signature is never checked at the gateway layer: [2](#0-1) 

The condition that sets `skip_validate = true` is:

```rust
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
    return mempool_client
        .account_tx_in_pool_or_recent_block(tx.sender_address())
        .await …;
}
``` [3](#0-2) 

The code comment explicitly acknowledges the breadth of the check:

> "It is sufficient to check if the account exists in the mempool since it means that **either it has a deploy_account transaction or transactions with future nonces that passed validations**." [4](#0-3) 

The second branch of that disjunction — "transactions with future nonces that passed validations" — is circular: those future-nonce transactions themselves passed validation only because `skip_stateful_validations` returned `true` for them. More concretely, the check does **not** assert that the matching mempool entry is a `DeployAccount`; it asserts only that the account has *some* entry. An attacker can satisfy this condition with a legitimate `DeployAccount` and then inject an Invoke with a forged/zeroed signature.

**Attack steps:**

1. Generate a fresh account address `A` (attacker controls the private key).
2. Submit `DeployAccount(A, valid_sig)` → passes full stateful validation → enters mempool.
3. Submit `Invoke(sender=A, nonce=1, sig=0x0…0)`:
   - Stateless validator: no signature check — passes.
   - `validate_nonce`: nonce=1, account_nonce=0, gap=1 ≤ `max_allowed_nonce_gap` — passes.
   - `validate_by_mempool`: checks nonce ordering and fee bounds only — passes.
   - `skip_stateful_validations`: `nonce==1 && account_nonce==0 && account_tx_in_pool_or_recent_block(A)==true` → returns `true`.
   - `run_validate_entry_point`: `validate=false` → `__validate__` is **never called**.
   - Transaction admitted into mempool with an invalid signature.

The `validate_by_mempool` call that precedes `skip_stateful_validations` only performs mempool-level checks (nonce gap, fee ordering); it does not execute the account's `__validate__` entry point: [5](#0-4) 

---

### Impact Explanation

An unprivileged attacker can inject Invoke transactions carrying arbitrary (invalid) signatures into the mempool, bypassing the gateway's only signature-verification gate. This matches the allowed impact:

> **High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.**

Downstream consequences:
- The batcher will dequeue and attempt to execute these transactions. During block execution the blockifier will call `__validate__` with `validate=true`; the transaction will fail, consuming sequencer CPU and block capacity without fee recovery (since `__validate__` failure prevents fee transfer).
- An attacker operating many fresh accounts (each seeded with one valid `DeployAccount`) can sustain a low-cost, high-volume DoS against the mempool and block-production pipeline.

---

### Likelihood Explanation

The precondition — having a valid `DeployAccount` in the mempool — is trivially achievable by any user. No privileged access, no special network position, and no cryptographic break is required. The attack is fully unprivileged and repeatable at scale.

---

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a **`DeployAccount` transaction** for the sender address is present in the mempool or was included in a recent block. The mempool client interface should expose a dedicated predicate such as `deploy_account_in_pool_or_recent_block(address)` that inspects the transaction type, not merely the account's presence.

Alternatively, if the UX goal is to allow simultaneous `DeployAccount + Invoke` submission, the gateway should accept the Invoke into a *pending* state and only admit it to the mempool after the corresponding `DeployAccount` has been confirmed present and is of the correct type.

---

### Proof of Concept

```
# 1. Generate fresh account address A (attacker holds private key)
A = compute_contract_address(class_hash, salt, constructor_calldata)

# 2. Submit valid DeployAccount — passes full validation, enters mempool
gateway.add_tx(DeployAccount {
    sender: A,
    nonce: 0,
    signature: sign(deploy_account_hash, private_key),   # valid
    ...
})

# 3. Submit Invoke with zeroed signature — skip_stateful_validations returns true
gateway.add_tx(Invoke {
    sender: A,
    nonce: 1,
    signature: [0x0, 0x0],   # invalid — never checked
    calldata: <arbitrary>,
    ...
})
# → Gateway returns success; Invoke is now in the mempool.
# → When the batcher executes it, __validate__ fails; sequencer bears the cost.
```

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
