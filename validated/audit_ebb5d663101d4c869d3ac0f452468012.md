### Title
Reentrancy in `claim_rewards` Allows Double-Claiming of Staker Rewards — (File: src/staking/staking.cairo)

---

### Summary

The `claim_rewards` function in the Staking contract transfers rewards to `reward_address` via an external ERC-20 call **before** writing the zeroed `unclaimed_rewards_own` back to storage. A staker who controls `reward_address` (a malicious contract) can reenter `claim_rewards` during the transfer and claim the same accumulated rewards a second time, draining the reward supplier.

---

### Finding Description

In `claim_rewards` (lines 411–431), the execution order is:

1. Read `staker_info` from storage — `unclaimed_rewards_own = X` (non-zero).
2. Call `send_rewards_to_staker` (lines 1614–1629), which:
   - Calls `claim_from_reward_supplier` (external call to reward supplier).
   - Calls `token_dispatcher.checked_transfer(recipient: reward_address, ...)` — **external call to attacker-controlled address**.
   - Sets **local variable** `staker_info.unclaimed_rewards_own = Zero::zero()` (storage is NOT yet updated).
3. Only after `send_rewards_to_staker` returns: `write_staker_info` persists the zeroed value to storage (line 429). [1](#0-0) [2](#0-1) 

During step 2's `checked_transfer`, if `reward_address` is a malicious contract, it can call `claim_rewards(staker_address)` again. At that moment, storage still holds `unclaimed_rewards_own = X`. The reentrant call passes the authorization check (`caller_address == reward_address`), reads the stale non-zero rewards, calls `claim_from_reward_supplier` a second time, and transfers rewards again. The reentrant call then writes `unclaimed_rewards_own = 0` to storage. When control returns to the original call, it also writes `unclaimed_rewards_own = 0` — but the damage is done: rewards were paid out twice.

There is no reentrancy guard in `general_prerequisites`: [3](#0-2) 

Notably, the developers were aware of this class of issue in `unstake_action`, where they explicitly write `staker_info` to storage **before** any external transfer and comment *"This is done here to avoid re-entrancy"*: [4](#0-3) 

The same protection was not applied to `claim_rewards`.

---

### Impact Explanation

**High — Theft of unclaimed yield.**

A staker who sets `reward_address` to a malicious contract can claim their accumulated `unclaimed_rewards_own` multiple times per transaction. Each reentrant call draws fresh funds from the reward supplier. The attacker can drain the reward supplier of STRK tokens proportional to their accumulated rewards, directly stealing yield that belongs to the protocol and other stakers.

---

### Likelihood Explanation

**High.** Any staker can freely set their `reward_address` to an arbitrary contract address via `change_reward_address`. No privileged role or external dependency is required. The malicious contract only needs to implement a callback that calls `claim_rewards` during the ERC-20 `transfer` hook. The entry path is fully attacker-controlled and requires no special conditions beyond having accumulated non-zero rewards.

---

### Recommendation

Zero out `unclaimed_rewards_own` in storage **before** performing any external transfer. The simplest fix is to write `staker_info` to storage before calling `send_rewards_to_staker`, mirroring the pattern already used in `unstake_action`:

```cairo
fn claim_rewards(ref self: ContractState, staker_address: ContractAddress) -> Amount {
    self.general_prerequisites();
    let mut staker_info = self.internal_staker_info(:staker_address);
    let caller_address = get_caller_address();
    let reward_address = staker_info.reward_address;
    assert!(
        caller_address == staker_address || caller_address == reward_address,
        "{}",
        Error::CLAIM_REWARDS_FROM_UNAUTHORIZED_ADDRESS,
    );

    let amount = staker_info.unclaimed_rewards_own;
    // Zero out rewards in storage BEFORE the external transfer.
    staker_info.unclaimed_rewards_own = Zero::zero();
    self.write_staker_info(:staker_address, :staker_info);

    let token_dispatcher = strk_token_dispatcher();
    self.send_rewards_to_staker(:staker_address, ref :staker_info, :token_dispatcher);
    amount
}
```

---

### Proof of Concept

1. Staker A calls `change_reward_address` to set `reward_address` to malicious contract `M`.
2. Rewards accumulate: `staker_info.unclaimed_rewards_own = R` (written to storage by `_update_rewards`).
3. `M` calls `claim_rewards(staker_A)`.
4. `send_rewards_to_staker` calls `claim_from_reward_supplier` (pulls `R` STRK into staking contract), then calls `token.transfer(M, R)`.
5. During `transfer`, `M`'s receive hook calls `claim_rewards(staker_A)` again.
6. Reentrant call reads storage: `unclaimed_rewards_own = R` (not yet zeroed). Calls `claim_from_reward_supplier` again (pulls another `R` STRK), calls `token.transfer(M, R)` again. Writes `unclaimed_rewards_own = 0` to storage.
7. Original call resumes, writes `unclaimed_rewards_own = 0` to storage (already zero).
8. `M` has received `2R` STRK; reward supplier has been debited `2R`. [5](#0-4)
