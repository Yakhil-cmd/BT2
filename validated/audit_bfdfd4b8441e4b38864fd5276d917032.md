### Title
Unprivileged Caller Can Permanently Suppress Consensus Block Rewards via `update_rewards(disable_rewards: true)` - (File: src/staking/staking.cairo)

---

### Summary

The `update_rewards` function in the `Staking` contract accepts a `disable_rewards` flag from any unprivileged caller. Because the global `last_reward_block` is written **before** the `disable_rewards` early-return, an attacker can call `update_rewards(any_valid_staker, disable_rewards: true)` once per block to permanently consume the block's reward slot without distributing any rewards, rendering the entire consensus reward incentivization system ineffective.

---

### Finding Description

`update_rewards` is a public function with no role-based access control — only `general_prerequisites()` is checked, which only asserts the contract is unpaused and the caller is non-zero. [1](#0-0) 

The function writes `last_reward_block` to the current block number unconditionally, **before** checking `disable_rewards`: [2](#0-1) 

`last_reward_block` is a single global storage slot (not per-staker): [3](#0-2) 

The guard at the top of `update_rewards` enforces that only one call per block can succeed: [4](#0-3) 

When `disable_rewards: true` is passed, the function returns immediately after writing `last_reward_block`, distributing nothing: [5](#0-4) 

**Attack path:**
1. Attacker identifies any valid, active staker with non-zero STRK balance (publicly readable from chain state).
2. Each block, attacker calls `update_rewards(valid_staker_address, disable_rewards: true)`.
3. `last_reward_block` is set to the current block; the function returns early with no rewards distributed.
4. Any subsequent legitimate call to `update_rewards` in the same block reverts with `REWARDS_ALREADY_UPDATED`.
5. The block's reward allocation is permanently lost — it cannot be reclaimed retroactively.

---

### Impact Explanation

Each block's consensus rewards that are suppressed are **permanently unrecoverable** — the protocol has no mechanism to retroactively distribute rewards for a past block. An attacker running this every block destroys all consensus-phase staker and delegator yield indefinitely. This matches **High: Permanent freezing of unclaimed yield**. [6](#0-5) 

---

### Likelihood Explanation

The attack requires only a single transaction per block from any EOA or contract. No privileged access, no leaked keys, no special setup — just a valid staker address (trivially obtained from on-chain events) and gas. The cost to the attacker is minimal; the damage to all stakers and delegators is total suppression of consensus rewards.

---

### Recommendation

1. **Remove the `disable_rewards` parameter from the public interface**, or restrict it to a privileged role (e.g., `only_security_agent`).
2. Alternatively, move the `self.last_reward_block.write(current_block_number)` call to **after** the `disable_rewards` guard, so that a no-op call does not consume the block's reward slot.
3. Consider making `update_rewards` callable only by the staker themselves or a designated operator, analogous to how `claim_rewards` is restricted to the staker or their reward address. [7](#0-6) 

---

### Proof of Concept

```
// Pseudocode — executable by any unprivileged address

// Step 1: Obtain any valid staker address from on-chain events (NewStaker event)
let victim_staker = <any_active_staker_address>;

// Step 2: Each block, before any legitimate update_rewards call:
staking_contract.update_rewards(
    staker_address: victim_staker,
    disable_rewards: true   // no access control prevents this
);
// Result:
//   - last_reward_block = current_block  (slot consumed)
//   - No STRK/BTC rewards distributed to any staker or pool
//   - All subsequent update_rewards calls this block revert with REWARDS_ALREADY_UPDATED
//   - Block rewards permanently lost
```

The attacker repeats this every block. All stakers and delegators receive zero consensus rewards for the duration of the attack, with no recovery path.
