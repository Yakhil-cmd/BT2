Gateway Stateless Validator Unconditionally Applies L2 Gas Price Floor to All Resource Bounds, Rejecting Valid L1-Gas-Only Transactions — (`File: crates/apollo_gateway/src/stateless_transaction_validator.rs`)

---

### Summary

`StatelessTransactionValidator::validate_resource_bounds` applies the `min_gas_price` floor check unconditionally against `l2_gas.max_price_per_unit`, even when a transaction carries only L1 gas bounds (with `l2_gas` fields at their zero default). In production, `min_gas_price` is `8_000_000_000` fri. Any V3 transaction that legitimately sets only `l1_gas` (leaving `l2_gas.max_price_per_unit = 0`) will always fail the check `0 < 8_000_000_000` and be rejected with `MaxGasPriceTooLow` before it ever reaches the mempool.

---

### Finding Description

`validate_resource_bounds` performs two sequential checks:

1. **Zero-fee guard** — rejects transactions whose `max_possible_fee(Tip::ZERO) == 0`. A transaction with a non-zero `l1_gas` bound passes this guard.
2. **L2 gas price floor** — unconditionally asserts `l2_gas.max_price_per_unit >= min_gas_price`. [1](#0-0) 

The production default for `min_gas_price` is `8_000_000_000`: [2](#0-1) 

This value is also baked into the shipped `config_schema.json`: [3](#0-2) 

A Starknet V3 transaction may legitimately carry only L1 gas bounds — the `AllResourceBounds` struct defaults `l2_gas` to `ResourceBounds { max_amount: 0, max_price_per_unit: 0 }`. Such a transaction has a positive `max_possible_fee` (from `l1_gas`) and is therefore protocol-valid, but the second check evaluates `0 < 8_000_000_000 → true` and returns `MaxGasPriceTooLow`, permanently blocking the transaction at the gateway.

The test suite confirms the intent: the test case `valid_l1_gas` explicitly marks an L1-gas-only transaction as a positive (should-pass) case: [4](#0-3) 

That test only passes because the testing config overrides `min_gas_price` to `0`: [5](#0-4) 

No equivalent positive test exists with the production `min_gas_price = 8_000_000_000`, so the regression is invisible in CI.

---

### Impact Explanation

Every V3 transaction that specifies only `l1_gas` bounds (a valid and documented resource-bounds variant) is unconditionally rejected by the Apollo gateway before it reaches the mempool or blockifier. The gateway returns a `MaxGasPriceTooLow` error, which is an authoritative-looking admission decision. Affected users cannot work around this without also setting a non-zero `l2_gas.max_price_per_unit ≥ 8 Gwei`, which changes the transaction's hash and fee semantics.

**Impact category**: High — Mempool/gateway admission rejects valid transactions before sequencing.

---

### Likelihood Explanation

Any wallet, SDK, or dApp that constructs a V3 transaction with only `l1_gas` bounds set (a pattern supported by the Starknet spec and the `ValidResourceBounds::L1Gas` variant) will be silently blocked in production. The bug is masked in all existing tests by the `min_gas_price: 0` test override, so it has not been caught by CI. Likelihood is high for any user or tooling that relies on L1-gas-only transactions.

---

### Recommendation

Guard the `min_gas_price` check so it is only applied when the transaction actually carries a non-zero L2 gas bound:

```rust
// Only enforce the L2 gas price floor when L2 gas is actually being used.
if resource_bounds.l2_gas.max_price_per_unit.0 > 0
    && resource_bounds.l2_gas.max_price_per_unit.0 < self.config.min_gas_price
{
    return Err(StatelessTransactionValidatorError::MaxGasPriceTooLow { ... });
}
```

Alternatively, add a dedicated test that exercises `valid_l1_gas` with the production `min_gas_price = 8_000_000_000` to surface the failure immediately.

---

### Proof of Concept

1. Construct a V3 invoke transaction with:
   ```
   l1_gas  = { max_amount: 1, max_price_per_unit: 1 }
   l2_gas  = { max_amount: 0, max_price_per_unit: 0 }   // default
   l1_data_gas = { max_amount: 0, max_price_per_unit: 0 } // default
   ```
2. Submit to the Apollo gateway running with the default production config (`min_gas_price = 8_000_000_000`, `validate_resource_bounds = true`).
3. `max_possible_fee(Tip::ZERO)` is non-zero (passes the zero-fee guard).
4. `l2_gas.max_price_per_unit.0 = 0 < 8_000_000_000` → `MaxGasPriceTooLow` is returned.
5. The transaction is rejected despite being protocol-valid.

The existing unit test `valid_l1_gas` in `stateless_transaction_validator_test.rs` already encodes this exact transaction shape and asserts `Ok(())`; running it with `min_gas_price: 8_000_000_000` instead of `0` reproduces the failure. [6](#0-5) [2](#0-1) [7](#0-6)

    fn validate_resource_bounds(
        &self,
        tx: &RpcTransaction,
    ) -> StatelessTransactionValidatorResult<()> {
        if !self.config.validate_resource_bounds {
            return Ok(());
        }

        let resource_bounds = *tx.resource_bounds();
        // The resource bounds should be positive even without the tip.
        if ValidResourceBounds::AllResources(resource_bounds).max_possible_fee(Tip::ZERO) == Fee(0)
        {
            return Err(StatelessTransactionValidatorError::ZeroResourceBounds { resource_bounds });
        }

        if resource_bounds.l2_gas.max_price_per_unit.0 < self.config.min_gas_price {
            return Err(StatelessTransactionValidatorError::MaxGasPriceTooLow {
                gas_price: resource_bounds.l2_gas.max_price_per_unit,
                min_gas_price: self.config.min_gas_price,
            });
        }

        // TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
        if let RpcTransaction::Declare(_) = tx {
        } else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
            return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
                gas_amount: resource_bounds.l2_gas.max_amount,
                max_gas_amount: self.config.max_l2_gas_amount,
            });
        }

        Ok(())
    }
