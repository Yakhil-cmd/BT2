### Title
Pool Admin Can Silently Overwrite a Pending Price Provider Proposal, Defeating the Timelock Notice Guarantee — (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`proposePoolPriceProvider` unconditionally overwrites `pendingPriceProvider[pool]` and `pendingPriceProviderExecuteAfter[pool]` with no check for an already-pending proposal. A pool admin can exploit this to perform a bait-and-switch: announce a benign provider so LPs decide not to exit, then silently replace it with a malicious provider just before the original timelock expires, resetting the countdown. LPs who made their exit decision based on the original proposal are now exposed to a provider they never evaluated, with the full timelock period restarted from the moment of the silent switch.

---

### Finding Description

The timelock mechanism was introduced specifically to fix audit finding M01 ("Admin can arbitrarily change the pool priceProvider"). Its invariant is: LPs must have at least `priceProviderTimelock[pool]` seconds of advance notice of the **specific** provider that will govern their swaps before it takes effect.

`proposePoolPriceProvider` contains no guard against overwriting an existing pending proposal:

```solidity
// metric-core/contracts/MetricOmmPoolFactory.sol lines 487-490
uint256 executeAfter = block.timestamp + timelock;
pendingPriceProvider[pool] = newPriceProvider;          // unconditional overwrite
pendingPriceProviderExecuteAfter[pool] = executeAfter;  // resets the countdown
emit PoolPriceProviderChangeProposed(pool, current, newPriceProvider, executeAfter);
```

There is no `if (pendingPriceProvider[pool] != address(0)) revert ...` guard anywhere in the function.

The analogous `proposePoolAdminTransfer` also overwrites silently, but admin transfers require the incoming party to `acceptPoolAdmin`, so the worst case is a confused nominee — not a live oracle switch. The price provider path has no such acceptance step: once `executePoolPriceProviderUpdate` is called after the timelock, the new provider is applied immediately.

**Attack sequence (timelock = T seconds):**

1. At time 0, admin calls `proposePoolPriceProvider(pool, providerA)`. `executeAfter = T`.
2. LPs observe the `PoolPriceProviderChangeProposed` event for `providerA`, evaluate it as acceptable, and decide not to withdraw.
3. At time `T − ε` (just before the original deadline), admin calls `proposePoolPriceProvider(pool, providerB)` where `providerB` is a manipulated oracle. `executeAfter` is reset to `T − ε + T`.
4. LPs who were watching for `providerA` to execute at time `T` see nothing happen and may not notice the new event for `providerB`.
5. At time `2T − ε`, admin calls `executePoolPriceProviderUpdate`. `providerB` is now the live oracle.
6. Swaps now execute against `providerB`'s manipulated bid/ask, causing LP fund loss.

The effective notice period for `providerB` is `T` seconds from step 3, but LPs who made their exit decision at step 2 based on `providerA` receive zero meaningful notice of the actual provider that will govern their funds.

---

### Impact Explanation

Once `providerB` is live, every swap reads its bid/ask via `getBidAndAskPrice()`. A manipulated provider can return prices that are stale, inverted, or outside any safe band, causing traders to receive more than the fair oracle price (swap conservation failure) or LPs to absorb losses on every trade. This is a direct, quantifiable loss of LP principal — the exact impact class listed as "bad-price execution" in the allowed impact gate.

The admin-boundary break is precise: the timelock was the only constraint preventing the admin from arbitrarily rotating the oracle (M01 fix). The overwrite allows the admin to change the specific provider that will be applied without LPs having the full timelock period to evaluate and react to **that specific provider**.

---

### Likelihood Explanation

Requires a malicious pool admin (semi-trusted actor). The pool admin is explicitly scoped as "semi-trusted only inside caps and timelocks" — the timelock is the cap being bypassed here. The attack requires two `proposePoolPriceProvider` calls timed around the original deadline, which is straightforward to automate. Likelihood is Low, but the admin is the exact actor the timelock was designed to constrain.

---

### Recommendation

Add a check in `proposePoolPriceProvider` that reverts if a proposal is already pending:

```solidity
function proposePoolPriceProvider(address pool, address newPriceProvider)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (pendingPriceProvider[pool] != address(0)) revert PriceProviderProposalAlreadyPending();
    // ... rest of function
}
```

If cancellation is needed, add a separate `cancelPoolPriceProviderProposal` function (mirroring `cancelPoolAdminTransfer`) that explicitly clears the pending state and emits a cancellation event, giving LPs a clear on-chain signal that the previous proposal was withdrawn.

### Proof of Concept

```solidity
// 1. Deploy pool with priceProviderTimelock = 7 days
// 2. Admin proposes benign providerA — LPs see event, decide not to exit
vm.prank(admin);
factory.proposePoolPriceProvider(pool, address(providerA));

// 3. Warp to just before the deadline
vm.warp(block.timestamp + 7 days - 1);

// 4. Admin silently overwrites with malicious providerB — resets countdown
vm.prank(admin);
factory.proposePoolPriceProvider(pool, address(providerB));
// pendingPriceProvider[pool] == providerB, executeAfter == now + 7 days

// 5. Warp past new deadline
vm.warp(block.timestamp + 7 days);

// 6. Execute — providerB is now live
vm.prank(admin);
factory.executePoolPriceProviderUpdate(pool);

// providerB.getBidAndAskPrice() now returns manipulated prices
// All subsequent swaps execute against the manipulated oracle
```

  function executePoolPriceProviderUpdate(address pool) external override nonReentrant onlyPoolAdmin(pool) {
    address pending = pendingPriceProvider[pool];
    if (pending == address(0)) revert NoPriceProviderChangeProposed();
    uint256 execAfter = pendingPriceProviderExecuteAfter[pool];
    // forge-lint: disable-next-line(block-timestamp) -- timelock enforcement legitimately relies on `block.timestamp`.
    if (block.timestamp < execAfter) revert PriceProviderTimelockNotElapsed(execAfter, block.timestamp);
    PoolImmutables memory p = IMetricOmmPool(pool).getImmutables();
    if (p.immutablePriceProvider != address(0)) revert PriceProviderImmutable();
    _validatePriceProvider(p.token0, p.token1, pending);
    IMetricOmmPoolFactoryActions(pool).setPriceProvider(pending);
    delete pendingPriceProvider[pool];
    delete pendingPriceProviderExecuteAfter[pool];
