### Title
NEAR Deposit Permanently Stuck in XCC Router When `fund_xcc_sub_account` Triggers Upgrade Path — (`engine/src/xcc.rs` + `etc/xcc-router/src/lib.rs`)

---

### Summary

When a user calls `fund_xcc_sub_account` on the Aurora Engine with an attached NEAR deposit and their XCC router already exists but requires an upgrade (not initial creation), the engine forwards the full user deposit as `attached_yocto` to the router's `deploy_upgrade` function. However, `deploy_upgrade` is `#[payable]` and never uses or forwards the attached deposit — it calls `initialize` with `NearToken::default()` (zero). The user's NEAR is permanently trapped in the router sub-account with no automatic recovery path.

---

### Finding Description

**Root cause — engine side (`engine/src/xcc.rs`, `fund_xcc_sub_account`):**

When the router exists but is outdated (`create_needed = false`), the engine constructs a `FunctionCall` action that attaches the full `fund_amount` (the user's `env.attached_deposit()`) to the `deploy_upgrade` call: [1](#0-0) 

```rust
} else {
    let deploy_args = DeployUpgradeParams { ... };
    promise_actions.push(PromiseAction::FunctionCall {
        name: "deploy_upgrade".into(),
        args: borsh::to_vec(&deploy_args).expect(ERR_UPGRADE_ARG_SERIALIZATION),
        attached_yocto: fund_amount,   // ← user's NEAR forwarded here
        gas: UPGRADE_GAS + INITIALIZE_GAS,
    });
}
```

The `fund_amount` is derived directly from the caller's attached deposit: [2](#0-1) 

**Root cause — router side (`etc/xcc-router/src/lib.rs`, `deploy_upgrade`):**

The router's `deploy_upgrade` is `#[payable]` and accepts the deposit, but calls `initialize` with `NearToken::default()` (zero NEAR), never touching `env::attached_deposit()`: [3](#0-2) 

```rust
#[payable]
pub fn deploy_upgrade(&mut self, #[serializer(borsh)] args: DeployUpgradeParams) {
    self.assert_preconditions();
    let promise_id = env::promise_batch_create(&env::current_account_id());
    env::promise_batch_action_deploy_contract(promise_id, &args.code);
    env::promise_batch_action_function_call(
        promise_id,
        INITIALIZE,
        &args.initialize_args,
        NearToken::default(),   // ← zero, deposit is silently discarded
        INITIALIZE_GAS,
    );
    env::promise_return(promise_id);
}
```

In NEAR Protocol, a successful (non-panicking) function call does **not** automatically refund the attached deposit. Because `deploy_upgrade` returns normally, the `fund_amount` NEAR is absorbed into the router sub-account's balance with no mechanism to recover it.

**Contrast with `handle_precompile_promise`:** The same upgrade branch in `handle_precompile_promise` correctly uses `ZERO_YOCTO` when calling `deploy_upgrade`, so that code path is not affected. [4](#0-3) 

The discrepancy between the two callers confirms this is an unintentional omission in `fund_xcc_sub_account`.

---

### Impact Explanation

Any NEAR attached by the user to `fund_xcc_sub_account` is permanently frozen in the router sub-account when the upgrade path is taken. The router has no withdrawal function accessible to the user directly. `send_refund` only sends a fixed `REFUND_AMOUNT` (2 NEAR) back to the parent engine and is only callable as a callback by the engine itself — it does not recover the user's arbitrary deposit. [5](#0-4) [6](#0-5) 

**Impact class: Permanent freezing of funds (Critical).**

---

### Likelihood Explanation

The trigger conditions are:
1. A user has an existing XCC router sub-account (i.e., they have previously used the XCC precompile).
2. The engine admin has deployed a new version of the router bytecode (a routine maintenance operation).
3. The user calls `fund_xcc_sub_account` with any non-zero NEAR attached.

All three conditions are realistic and expected during normal protocol operation after any router upgrade. Any user who calls `fund_xcc_sub_account` to top up their router balance after an engine upgrade will silently lose their entire deposit. **Likelihood: Medium.**

---

### Recommendation

In `fund_xcc_sub_account` in `engine/src/xcc.rs`, when the upgrade path is taken (`create_needed = false`), do not attach `fund_amount` to the `deploy_upgrade` call. Instead, issue a separate `Transfer` action after the upgrade to deliver the user's NEAR to the router, mirroring the `UpToDate` branch:

```rust
// upgrade branch — do NOT attach fund_amount to deploy_upgrade
promise_actions.push(PromiseAction::FunctionCall {
    name: "deploy_upgrade".into(),
    args: borsh::to_vec(&deploy_args).expect(ERR_UPGRADE_ARG_SERIALIZATION),
    attached_yocto: ZERO_YOCTO,   // ← fix: use zero, same as handle_precompile_promise
    gas: UPGRADE_GAS + INITIALIZE_GAS,
});
// then transfer the user's deposit separately
promise_actions.push(PromiseAction::Transfer { amount: fund_amount });
```

Alternatively, add a check in `deploy_upgrade` in the router to assert that no deposit is attached (analogous to the report's recommendation of `require(msg.value == 0)`):

```rust
#[payable]
pub fn deploy_upgrade(&mut self, args: DeployUpgradeParams) {
    assert_eq!(
        env::attached_deposit(),
        NearToken::from_yoctonear(0),
        "ERR_UNEXPECTED_DEPOSIT"
    );
    ...
}
```

---

### Proof of Concept

1. User's XCC router exists at version `N`. Engine admin upgrades router bytecode to version `N+1`.
2. User calls `fund_xcc_sub_account` on the Aurora Engine with `10 NEAR` attached.
3. Engine detects `AddressVersionStatus::DeployNeeded { create_needed: false }`.
4. Engine issues a batch with `deploy_upgrade` call carrying `attached_yocto = 10 NEAR`.
5. Router's `deploy_upgrade` executes successfully, deploys new code, calls `initialize` with `0 NEAR`.
6. The `10 NEAR` remains in the router's account balance.
7. User's `fund_xcc_sub_account` call returns success with no error.
8. User has lost `10 NEAR` permanently — the router balance increased by `10 NEAR` but there is no user-accessible path to recover it. [7](#0-6) [8](#0-7)
