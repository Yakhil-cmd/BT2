### Title
Missing Upper Bound on NFT Royalty Percentage in WalletConnect Offer Confirmation Causes Spoofed Amount Display - (File: packages/gui/src/electron/commands/parseCommandDisplay.ts)

### Summary
`parseRoyaltyPercentage` in `parseCommandDisplay.ts` accepts any positive integer as a royalty percentage with no upper bound check. A malicious dapp connected via WalletConnect can supply an arbitrarily large `royalty_percentage` in the `driver_dict` of a `chia_wallet.create_offer_for_ids` command, causing `formatAmountWithRoyalties` to compute and display a grossly inflated "Total Amount with Royalties" in the signing confirmation dialog. The user is shown the wrong spend amount before approving or rejecting the transaction.

### Finding Description
`parseRoyaltyPercentage` only validates that the value is a positive integer:

```typescript
// packages/gui/src/electron/commands/parseCommandDisplay.ts
function parseRoyaltyPercentage(value: unknown): number | undefined {
  if (typeof value === 'number' && Number.isInteger(value) && value > 0) {
    return value;  // no upper bound — accepts 20000, 999999, etc.
  }
  if (typeof value === 'string' && /^\d+$/.test(value)) {
    return Number(value);  // same: no upper bound
  }
  return undefined;
}
``` [1](#0-0) 

This value flows directly into `formatAmountWithRoyalties`, which treats it as a basis-point figure divided by `10_000n`:

```typescript
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
const totalAmount = amount + royaltyAmount;
``` [2](#0-1) 

The Chia NFT protocol encodes royalty in basis points where `10000 = 100%`. If a dapp injects `royalty_percentage: 20000` (200%) into the `driver_dict`, the computed `royaltyAmount` equals `2 × amount`, and `totalAmount` becomes `3 × amount`. The confirmation dialog then shows "Total Amount with Royalties: 3 XCH" for a 1 XCH offer.

The `chia_wallet.create_offer_for_ids` command is reachable via WalletConnect (`chia_createOfferForIds`), and the `driver_dict` is supplied entirely by the dapp: [3](#0-2) 

The `driver_dict` royalty percentages are extracted via `createOfferRoyaltyPercentages` → `royaltyPercentageForDriverInfo` → `parseRoyaltyPercentage`, with no cap applied at any stage: [4](#0-3) 

By contrast, the GUI's own NFT offer creation path (`NFTOfferEditor`) does enforce a `> 100%` guard before proceeding: [5](#0-4) 

That guard is absent in the WalletConnect code path.

### Impact Explanation
A malicious dapp with an active WalletConnect session can craft a `chia_wallet.create_offer_for_ids` payload with `royalty_percentage` set to any value above `10000` in the `driver_dict`. The WalletConnect confirmation dialog will display a fabricated, inflated "Total Amount with Royalties," causing the user to see the wrong spend amount before approving or rejecting the transaction. This constitutes spoofing of WalletConnect state that causes the user to display the wrong amount — a High-impact condition per the allowed scope.

### Likelihood Explanation
Any dapp that has obtained WalletConnect approval from the user can issue `chia_createOfferForIds` with an arbitrary `driver_dict`. No additional privileges, leaked keys, or social engineering beyond the initial WalletConnect pairing are required. The attack is fully programmatic.

### Recommendation
Add an upper bound check in `parseRoyaltyPercentage` rejecting values above `10000` (the maximum valid basis-point royalty representing 100%):

```typescript
function parseRoyaltyPercentage(value: unknown): number | undefined {
  if (typeof value === 'number' && Number.isInteger(value) && value > 0 && value <= 10000) {
    return value;
  }
  if (typeof value === 'string' && /^\d+$/.test(value)) {
    const n = Number(value);
    if (n > 0 && n <= 10000) return n;
  }
  return undefined;
}
```

This mirrors the existing guard in `NFTOfferEditor` and aligns with the Chia NFT protocol's defined maximum of `10000` basis points.

### Proof of Concept
A dapp sends the following WalletConnect command:

```json
{
  "command": "chia_createOfferForIds",
  "params": {
    "offer": { "1": "-1000000000000", "<nft_launcher_id>": "1" },
    "driver_dict": {
      "<nft_launcher_id>": {
        "type": "singleton",
        "launcher_id": "0x<nft_launcher_id>",
        "also": {
          "type": "metadata",
          "also": {
            "type": "ownership",
            "transfer_program": {
              "type": "royalty transfer program",
              "royalty_percentage": "20000"
            }
          }
        }
      }
    }
  }
}
```

`parseRoyaltyPercentage("20000")` returns `20000`. `formatAmountWithRoyalties` computes `royaltyAmount = 1,000,000,000,000 × 20000 / 10000 = 2,000,000,000,000 mojos`. `totalAmount = 3,000,000,000,000 mojos = 3 XCH`. The confirmation dialog displays "Total Amount with Royalties: 3 XCH" for a 1 XCH offer, spoofing the user's view of the actual cost. [6](#0-5)

```typescript
function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],
): string | undefined {
  if (royaltyPercentages.length === 0 || line.kind === 'nft') {
    return undefined;
  }

  const splitAmount = amount / BigInt(royaltyPercentages.length);
  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;

  if (line.kind === 'xch') {
    return mojoToChiaLocaleString(totalAmount);
  }

  return mojoToCATLocaleString(totalAmount);
}
