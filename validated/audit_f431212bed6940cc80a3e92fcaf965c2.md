### Title
Unvalidated `app_pk` Identity Element in CKD Protocol Exposes Confidential Derived Key - (File: src/confidential_key_derivation/protocol.rs)

### Summary
The `ckd` function in `src/confidential_key_derivation/protocol.rs` accepts a caller-supplied `app_pk: PublicKey` (an `ElementG1`) without validating that it is not the group identity element. If `app_pk` is the G1 identity, the ElGamal masking term `y * app_pk` collapses to zero, causing each participant's `big_c` share to equal the raw secret BLS signature share `x_i * H(pk || app_id)` in plaintext. The coordinator then sums these unmasked shares and outputs the full confidential derived key `msk * H(pk || app_id)` directly in the `CKDOutput`, with no encryption.

### Finding Description
The CKD protocol is designed to produce an ElGamal-encrypted output `(Y, C)` where:
- `Y = y * G`
- `C = S + y * A`, with `S = x_i * H(pk || app_id)` and `A = app_pk`

The masking term `y * A` is what keeps `S` confidential; only the holder of `app_sk` (where `A = app_sk * G`) can unmask it.

In `compute_signature_share` at line 174:

```rust
let big_c = big_s + app_pk * y.0;
```

If `app_pk` is the G1 identity element, `app_pk * y.0 = identity`, so `big_c = big_s = hash_point * private_share.to_scalar()` — the raw secret share, unmasked.

The `ckd` entry point performs no validation on `app_pk`:

```rust
pub fn ckd(
    participants: &[Participant],
    coordinator: Participant,
    me: Participant,
    key_pair: KeygenOutput,
    app_id: impl Into<AppId>,
    app_pk: PublicKey,   // ← never checked for identity
    rng: impl CryptoRngCore + Send + 'static,
) -> Result<impl Protocol<Output = CKDOutputOption>, InitializationError>
```

After all participants compute with `app_pk = identity`, the coordinator sums their Lagrange-weighted shares:

```
big_c = Σ λ_i * x_i * H(pk || app_id) = msk * H(pk || app_id)
```

This is the confidential derived key, now stored in plaintext in `CKDOutput::big_c`.

### Impact Explanation
**Critical — Extraction/disclosure of confidential derived secrets.**

The entire security guarantee of the CKD protocol is that `msk * H(pk || app_id)` is never revealed to any party other than the holder of `app_sk`. With `app_pk = identity`, this guarantee is completely broken: the coordinator's output `CKDOutput { big_y, big_c }` contains the full unmasked confidential derived key in `big_c`. Any party that observes the coordinator's output (including the coordinator itself) learns the secret.

### Likelihood Explanation
The `app_pk` is a library-caller-supplied parameter. In the documented use case, it is provided by the application client requesting key derivation. A malicious application client, or any caller who controls the `app_pk` argument, can trivially pass `blstrs::G1Projective::identity()` — a valid, serializable group element — without triggering any error. No cryptographic capability is required; it is a single-field substitution in the function call.

### Recommendation
Validate that `app_pk` is not the group identity element at the start of `ckd` before any protocol execution begins:

```rust
use elliptic_curve::Group;
if app_pk == ElementG1::identity() {
    return Err(InitializationError::BadParameters(
        "app_pk must not be the identity element".to_string(),
    ));
}
```

This mirrors the existing pattern used elsewhere in the codebase (e.g., the `IdentityElement` error in `src/ecdsa/robust_ecdsa/presign.rs` line 231).

### Proof of Concept

1. Caller invokes `ckd(participants, coordinator, me, key_pair, app_id, ElementG1::identity(), rng)` — all participants use `app_pk = G1::identity()`.
2. Each participant's `compute_signature_share` computes:
   - `big_c = big_s + identity * y = big_s = x_i * H(pk || app_id)` [1](#0-0) 
3. Each participant sends `(λ_i * Y, λ_i * big_c)` to the coordinator. [2](#0-1) 
4. The coordinator sums all received shares: [3](#0-2) 
   - `big_c = Σ λ_i * x_i * H(pk || app_id) = msk * H(pk || app_id)`
5. `CKDOutput { big_y, big_c }` is returned with `big_c` equal to the full unmasked confidential derived key. [4](#0-3) 
6. The `ckd` entry point performs no check on `app_pk` before proceeding. [5](#0-4)
