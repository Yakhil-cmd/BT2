### Title
Missing Caller Validation on `update_rewards` Allows Any Address to Permanently Freeze Staker Yield - (File: `src/staking/staking.cairo`)

---

### Summary

`IStakingRewardsManager::update_rewards` is documented as callable only by the Starkware sequencer, but the implementation contains no caller check. Any unprivileged address can call it with `disable_rewards: true` every block, front-running the sequencer and permanently preventing reward distribution for all stakers and delegators.

---

### Finding Description

The specification at `docs/spec.md` line 1645 states:

> **access control**: Only starkware sequencer.

The implementation in `src/staking/staking.cairo` at `StakingRewardsManagerImpl::update_rewards` (lines 1449–1507) performs no such check:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();                          // only checks: not paused
    let current_block_number = starknet::get_block_number();
    assert!(
        current_block_number > self.last_reward_block.read(),
        "{}",
        Error::REWARDS_ALREADY_UPDATED,
    );
    // ... staker validity checks ...
    self.last_reward_block.write(current_block_number);   // global write
    if disable_rewards || self.is_pre_consensus() {
        return;                                            // exits without distributing
    }
    // ... reward distribution ...
}
```

`last_reward_block` is a **global** (contract-wide, not per-staker) `BlockNumber` storage variable declared at line 187 of the `Storage` struct. Writing it to the current block number blocks any further call to `update_rewards` for the remainder of that block.

An attacker can:
1. Monitor the mempool for the sequencer's `update_rewards` call.
2. Front-run it every block by calling `update_rewards(any_valid_staker, disable_rewards: true)`.
3. This sets `last_reward_block` to the current block and returns early — no rewards are distributed.
4. The sequencer's subsequent call in the same block reverts with `REWARDS_ALREADY_UPDATED`.
5. Repeating this every block permanently freezes all yield.

The `IStakingRewardsManager` interface (lines 303–311 of `src/staking/interface.cairo`) exposes this function publicly with no restriction.

---

### Impact Explanation

**High — Permanent freezing of unclaimed yield.**

All stakers and delegators earn zero rewards as long as the attacker front-runs the sequencer each block. Because `last_reward_block` is global, a single call per block is sufficient to suppress rewards for every staker in the protocol. Delegators' `unclaimed_rewards` in pool contracts never increase. Stakers' `unclaimed_rewards_own` never increase. The attack requires no capital and no privileged access.

---

### Likelihood Explanation

**High.** The function is publicly callable on-chain with no access control. Starknet supports fee-based transaction ordering; an attacker can reliably front-run the sequencer's `update_rewards` call by submitting a transaction with a higher fee in the same block. The attack is cheap (one transaction per block, no stake required) and fully automated.

---

### Recommendation

Add a caller check at the top of `update_rewards` to enforce that only the designated sequencer address (or an authorized operator role) may call it, matching the specification:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
    assert!(
        get_caller_address() == self.sequencer_address.read(),
        "{}",
        Error::CALLER_IS_NOT_SEQUENCER,
    );
    // ... rest of function
}
```

Alternatively, restrict via the existing roles component (e.g., `OPERATOR` role) consistent with how other privileged functions are guarded.

---

### Proof of Concept

1. Deploy the protocol and advance to consensus rewards epoch.
2. A valid staker exists with non-zero balance.
3. Attacker (any EOA) calls:
   ```
   IStakingRewardsManager(staking_contract).update_rewards(staker_address, disable_rewards: true)
   ```
   in block N. This succeeds — no caller check exists.
4. `last_reward_block` is now set to N.
5. Sequencer attempts `update_rewards(staker_address, disable_rewards: false)` in block N → reverts with `REWARDS_ALREADY_UPDATED`.
6. Attacker repeats step 3 in every subsequent block.
7. After K epochs, staker's `unclaimed_rewards_own` remains zero; delegators' pool balances are never topped up.
