### Title
Wrong Value Passed as `strk_total_stake` in Consensus Rewards Path — Staker's Own Balance Used Instead of Total Network Stake (`File: src/staking/staking.cairo`)

---

### Summary

In `update_rewards`, the staker's own balance (`staker_total_strk_balance`) is passed as the `strk_total_stake` argument to `_update_rewards`, instead of the total network staking power. The sibling code path (`update_rewards_from_attestation_contract`) correctly passes the total network stake. This discrepancy causes each staker to receive a grossly inflated share of block rewards, leading to protocol insolvency.

---

### Finding Description

There are two code paths that call `_update_rewards`:

**Path 1 — `update_rewards_from_attestation_contract` (pre-consensus, correct):**

```cairo
let (strk_total_stake, btc_total_stake) = self.get_current_total_staking_power();
self._update_rewards(
    :staker_address,
    strk_total_rewards: strk_epoch_rewards,
    :strk_total_stake,   // ← total network stake
    :btc_total_stake,
    ...
);
``` [1](#0-0) 

**Path 2 — `update_rewards` (consensus rewards, buggy):**

```cairo
let (staker_total_strk_balance, staker_total_btc_balance) = self
    .get_staker_total_strk_btc_balance_at_epoch(
        :staker_address, :staker_pool_info, epoch_id: curr_epoch,
    );
...
self._update_rewards(
    :staker_address,
    strk_total_rewards: strk_block_rewards,
    strk_total_stake: staker_total_strk_balance,  // ← staker's OWN balance, not total network stake
    btc_total_stake: staker_total_btc_balance,
    ...
);
``` [2](#0-1) 

`get_staker_total_strk_btc_balance_at_epoch` returns the individual staker's combined balance (own + delegated), not the total network staking power. The parameter name `strk_total_stake` in `_update_rewards` is consistent with the attestation path, where it receives the full network total from `get_current_total_staking_power()`. [3](#0-2) 

`_update_rewards` uses `strk_total_stake` as the denominator when computing each staker's proportional share of `strk_total_rewards`. With the staker's own balance in the denominator:

```
staker_rewards = strk_block_rewards × staker_balance / staker_balance
               = strk_block_rewards   (100% of all block rewards per staker)
```

Every staker receives the full block reward budget, regardless of their actual proportional stake.

---

### Impact Explanation

**Severity: Critical — Protocol Insolvency**

Each call to `update_rewards` for any staker causes that staker to accrue the entire network block reward (`strk_block_rewards`) rather than their proportional share. With N active stakers, the total rewards credited per block is `N × strk_block_rewards`. The `RewardSupplier` tracks `unclaimed_rewards` and requests minting from L1 to cover them. [4](#0-3) 

This causes `unclaimed_rewards` to grow at `N×` the intended rate, triggering unbounded L1 mint requests and draining the reward supply. Stakers can claim these inflated rewards via `claim_rewards`, constituting direct theft of protocol funds and eventual insolvency.

---

### Likelihood Explanation

`update_rewards` has no access control beyond the general pause check — any public caller can invoke it for any staker address on every block. [5](#0-4) 

The consensus rewards path activates after `consensus_rewards_first_epoch` is set. Once active, every block processed triggers the inflated accounting. No special privilege, leaked key, or external dependency is required.

---

### Recommendation

Replace `staker_total_strk_balance` with the total network staking power in the `update_rewards` call to `_update_rewards`, mirroring the attestation path:

```cairo
// Before (buggy):
strk_total_stake: staker_total_strk_balance,
btc_total_stake: staker_total_btc_balance,

// After (correct):
let (strk_total_stake, btc_total_stake) = self.get_current_total_staking_power();
// ... pass strk_total_stake, btc_total_stake to _update_rewards
``` [2](#0-1) 

---

### Proof of Concept

1. Deploy the staking system and register two stakers, each with 10% of total network stake.
2. Advance past `consensus_rewards_first_epoch`.
3. Call `update_rewards(staker_A, false)` on block B. `_update_rewards` receives `strk_total_stake = staker_A_balance` (10% of network). Computed rewards = `block_rewards × staker_A_balance / staker_A_balance = block_rewards` (100%).
4. Call `update_rewards(staker_B, false)` on the same block (different staker, same block number check passes per staker). Staker B also accrues 100% of block rewards.
5. Both stakers call `claim_rewards`. Each receives the full block reward. Total paid out = 2× block rewards for a single block.
6. `RewardSupplier.unclaimed_rewards` grows at 2× rate; L1 mint requests are issued for 2× the intended amount, draining the reward supply and causing protocol insolvency. [3](#0-2) [6](#0-5)

        fn claim_rewards(ref self: ContractState, amount: Amount) {
            // Asserts.
            let staking_contract = self.staking_contract.read();
            assert!(
                get_caller_address() == staking_contract,
                "{}",
                GenericError::CALLER_IS_NOT_STAKING_CONTRACT,
            );
            let unclaimed_rewards = self.unclaimed_rewards.read();
            assert!(amount <= unclaimed_rewards, "{}", GenericError::AMOUNT_TOO_HIGH);

            // Update unclaimed_rewards and transfer the requested rewards to the staking contract.
            self.unclaimed_rewards.write(unclaimed_rewards - amount);
            let token_dispatcher = self.token_dispatcher.read();
            token_dispatcher.checked_transfer(recipient: staking_contract, amount: amount.into());
        }
