### Title
Panic on Malformed EIP-6110 Deposit Event in For-Loop Causes Block Finalization Halt - (`File: basic_bootloader/src/bootloader/block_flow/ethereum/eip_6110_deposit_events_parser/mod.rs`)

---

### Summary

`eip6110_events_parser` iterates over all block events in a for-loop. When it finds an event from the deposit contract address with the correct `DepositEvent` signature topic, it calls `validate_and_write_event_data`. If that validation fails (e.g., wrong data length), the code unconditionally `panic!`s instead of returning an error. An attacker who can cause the deposit contract address to emit an event with the correct signature but malformed data can halt block finalization entirely.

---

### Finding Description

In `eip6110_events_parser`, the for-loop at line 36 iterates over all events. When an event matches the deposit contract address and the `DepositEvent` signature hash, `validate_and_write_event_data` is called. If validation fails, the code panics:

```rust
let Ok(_) = validate_and_write_event_data(event, requests_hasher, &mut logger) else {
    panic!("invalid deposit event structure");
};
``` [1](#0-0) 

`validate_and_write_event_data` returns `Err(())` for any of the following:
- Event data length is not exactly 576 bytes
- Any of the 10 ABI offset/length fields do not match their expected exact values [2](#0-1) 

This function is called during block finalization in both proving mode and sequencing mode:

- **Proving mode** (`post_tx_op_proving.rs` line 207): called with `.expect(...)` — but the `panic!` fires before the `expect` is reached.
- **Sequencing mode** (`post_tx_op_sequencing.rs` line 90): called with `let _ = ...` discarding the `Result` — but the `panic!` inside the function still aborts execution unconditionally. [3](#0-2) [4](#0-3) 

The deposit contract address is `0x00000000219ab540356cBB839Cbe05303d7705Fa`: [5](#0-4) 

On ZKsync OS (an L2), this address is not guaranteed to have the canonical Ethereum deposit contract deployed. If it is empty, an attacker can deploy a contract there via CREATE2 (by computing the appropriate salt) that emits an event with the correct `DepositEvent` signature hash (`0x649bbc62...`) but with data of any length other than 576 bytes, or with incorrect ABI offset fields.

---

### Impact Explanation

The `panic!` in `eip6110_events_parser` aborts the RISC-V execution environment entirely (see `zksync_os/src/main.rs` panic handler calling `rust_abort()`). This means:

- **Block finalization is halted**: `post_op` in both proving and sequencing mode cannot complete.
- **Block cannot be sealed**: The `result_keeper.record_sealed_block(...)` call in sequencing mode is never reached.
- **Proof cannot be generated**: In proving mode, the requests hash computation and all subsequent `assert_eq!` checks are never reached.

This is a **block-halting DoS** vulnerability. A single attacker transaction in a block can prevent that entire block from being finalized, causing a liveness failure. [6](#0-5) 

---

### Likelihood Explanation

The deposit contract address `0x00000000219ab540356cBB839Cbe05303d7705Fa` is a well-known Ethereum mainnet address. On ZKsync OS as an L2, this address is not a system-reserved address and may be empty. An attacker can:

1. Compute a CREATE2 salt such that `keccak256(0xff ++ deployer ++ salt ++ keccak256(init_code))[12:]` equals the deposit contract address.
2. Deploy a contract there that, when called, emits `LOG1(data, DEPOSIT_EVENT_SIGNATURE_HASH)` with `data.length != 576`.
3. Include a transaction calling that contract in any block.
4. Block finalization panics.

This requires only an unprivileged transaction sender and standard EVM CREATE2 mechanics. No privileged access, oracle manipulation, or external dependency is required.

---

### Recommendation

Replace the `panic!` with a graceful error return. The function signature already returns `Result<bool, SystemError>`, so the error can be propagated:

```rust
// Instead of:
let Ok(_) = validate_and_write_event_data(event, requests_hasher, &mut logger) else {
    panic!("invalid deposit event structure");
};

// Use:
validate_and_write_event_data(event, requests_hasher, &mut logger)
    .map_err(|_| SystemError::LeafDefect(internal_error!("invalid deposit event structure")))?;
```

In proving mode, the caller already uses `.expect(...)` which will convert the `Err` into a controlled failure rather than an uncontrolled panic. In sequencing mode, the result is currently discarded — the caller should also be updated to propagate or log the error rather than silently ignoring it. [7](#0-6) 

---

### Proof of Concept

1. Compute a CREATE2 salt `s` such that `CREATE2(deployer=D, salt=s, initcode=C)` produces address `0x00000000219ab540356cBB839Cbe05303d7705Fa`.
2. Deploy contract `C` at that address. `C` implements a fallback that executes:
   ```solidity
   // Emit DepositEvent signature with 32 bytes of data (not 576)
   assembly {
       mstore(0, 0)
       log1(0, 32, 0x649bbc62d0e31342afea4e5cd82d4049e7e1ee912fc0889aa790803be39038c5)
   }
   ```
3. Submit a transaction calling `C` (any calldata).
4. The block containing this transaction reaches `eip6110_events_parser` during finalization.
5. The for-loop finds the event: `event.address == DEPOSIT_CONTRACT_ADDRESS` and `event.topics[0] == DEPOSIT_EVENT_SIGNATURE_HASH`.
6. `validate_and_write_event_data` returns `Err(())` because `data.len() == 32 != 576`.
7. `panic!("invalid deposit event structure")` fires, aborting the RISC-V process.
8. Block finalization halts; the block cannot be sealed or proved.

fn validate_and_write_event_data(
    event: GenericEventContentWithTxRef<'_, MAX_EVENT_TOPICS, EthereumIOTypesConfig>,
    requests_hasher: &mut impl crypto::sha256::Digest,
    logger: &mut impl Logger,
) -> Result<(), ()> {
    let data = event.data;
    if data.len() != 576 {
        return Err(());
    }
    let mut chunks = data.as_chunks::<32>().0.iter();
    validate_u16_at_most(chunks.next().unwrap(), 160)?;
    validate_u16_at_most(chunks.next().unwrap(), 256)?;
    validate_u16_at_most(chunks.next().unwrap(), 320)?;
    validate_u16_at_most(chunks.next().unwrap(), 384)?;
    validate_u16_at_most(chunks.next().unwrap(), 512)?;
    drop(chunks);

    validate_u16_at_most(data[160..192].try_into().unwrap(), 48)?;
    validate_u16_at_most(data[256..288].try_into().unwrap(), 32)?;
    validate_u16_at_most(data[320..352].try_into().unwrap(), 8)?;
    validate_u16_at_most(data[384..416].try_into().unwrap(), 96)?;
    validate_u16_at_most(data[512..544].try_into().unwrap(), 8)?;
```
