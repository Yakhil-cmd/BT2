### Title
Missing Zero-Address Validation on `reward_address` Allows Permanent Loss of Unclaimed Yield — (`src/staking/staking.cairo`, `src/pool/pool.cairo`)

---

### Summary

Both `change_reward_address` in `staking.cairo` and `change_reward_address` in `pool.cairo` accept a zero address as the new `reward_address` without any validation. When rewards are later claimed or the staker exits, `send_rewards_to_staker` transfers accumulated yield to `0x0`, permanently destroying it. This is the direct analog of the reported pattern: a critical address field lacks a non-zero guard, and the downstream transfer silently routes funds to an unrecoverable destination.

---

### Finding Description

**`staking.cairo` — `change_reward_address` (line 517–540)**

The only guard applied is a token-address collision check:

```cairo
assert!(
    !self.does_token_exist(token_address: reward_address),
    "{}",
    GenericError::REWARD_ADDRESS_IS_TOKEN,
);
```

There is no `reward_address.is_non_zero()` assertion. A staker can therefore call `change_reward_address(0x0)` and the call succeeds, writing zero into `staker_info.reward_address`.

**`pool.cairo` — `change_reward_address` (line 505–526)**

Identical gap: only the token-address check is present, no zero-address guard.

**Downstream sink — `send_rewards_to_staker` (staking.cairo line 1614–1629)**

```cairo
fn send_rewards_to_staker(...) {
    let reward_address = staker_info.reward_address;   // may be 0x0
    let amount = staker_info.unclaimed_rewards_own;
    claim_from_reward_supplier(...);
    token_dispatcher.checked_transfer(recipient: reward_address, amount: amount.into());
    staker_info.unclaimed_rewards_own = Zero::zero();
    ...
}
```

This function is called unconditionally from both `claim_rewards` (line 428) and `unstake_action` (line 495). If `reward_address` is zero, the transfer targets `0x0`.

**`stake` (staking.cairo line 288–317)** also accepts a zero `reward_address` at registration time — the same token-address check is present but no zero check.

---

### Impact Explanation

Two concrete outcomes depending on ERC-20 behaviour for zero-recipient transfers:

1. **`checked_transfer` to `0x0` succeeds** — all accumulated `unclaimed_rewards_own` are minted from the reward supplier and transferred to the zero address. The yield is permanently destroyed. This satisfies: *Permanent freezing of unclaimed yield* (High).

2. **`checked_transfer` to `0x0` reverts** — `send_rewards_to_staker` reverts, causing `unstake_action` to revert. The staker's principal stake is locked in the contract until they call `change_reward_address` again with a valid address. This satisfies: *Temporary freezing of funds* (High).

In either case the staker's unclaimed yield is at risk.

---

### Likelihood Explanation

- Any active staker or pool member can call `change_reward_address` at any time; no privileged role is required.
- The call succeeds silently with no warning, making accidental zero-address entry (e.g., a scripting error, a UI bug, or a migration script that omits the address) a realistic scenario.
- The `stake` entry point also accepts zero at registration, so the issue is present from the very first transaction.

---

### Recommendation

Add a non-zero guard in both `change_reward_address` implementations and in `stake`:

```cairo
assert!(reward_address.is_non_zero(), "{}", Error::ZERO_ADDRESS);
```

The error constant `ZERO_ADDRESS` ("Zero address") already exists in the codebase's error catalogue.

---

### Proof of Concept

1. Staker calls `stake(reward_address: 0x0, operational_address: X, amount: MIN_STAKE)` — passes all existing checks.
2. Epochs advance; attestation occurs; `_update_rewards` credits `unclaimed_rewards_own`.
3. Staker calls `claim_rewards(staker_address)` (caller == staker_address satisfies the auth check at line 418).
4. `send_rewards_to_staker` is invoked; `claim_from_reward_supplier` pulls tokens into the staking contract; `checked_transfer(recipient: 0x0, amount: rewards)` executes — yield is sent to the zero address and lost.

Alternatively, via `change_reward_address`:

1. Staker stakes normally with a valid reward address.
2. Staker calls `change_reward_address(0x0)` — no revert.
3. On `unstake_action`, `send_rewards_to_staker` fires with `reward_address = 0x0`, destroying all accumulated yield before the principal is returned.
