### Title
Unauthorized Token Transfer via Missing Caller Validation in Bank Precompile `transfer` Method - (File: x/cronos/keeper/precompiles/bank.go)

### Summary
The `transfer` method of the bank precompile accepts the `sender` address as a caller-supplied argument but never validates that `contract.Caller()` matches the supplied `sender`. Any EVM contract can therefore drain `evm/<contract_address>`-denominated native tokens from any holder without authorization.

### Finding Description
In `x/cronos/keeper/precompiles/bank.go`, the `TransferMethodName` case unpacks `sender` directly from ABI-decoded call arguments and uses it as the debit account in a `bankKeeper.SendCoins` call:

```go
sender := args[0].(common.Address)   // caller-supplied, never validated
...
from := sdk.AccAddress(sender.Bytes())
denom := EVMDenom(contract.Caller())  // "evm/<caller_hex>"
amt  := sdk.NewCoin(denom, ...)
bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt))
```

There is no check that `sender == contract.Caller()`. The calling contract therefore controls both the denom (implicitly, via its own address) and the source account (explicitly, via the argument). This is the exact same root cause as the reported Solana finding: a privileged/special parameter is accepted from the caller without being validated against the expected authoritative value.

Compare this with the relayer precompile's `exec` helper in `utils.go`, which explicitly enforces caller authentication:

```go
caller := common.BytesToAddress(signers[0])
if caller != e.caller {
    return nil, fmt.Errorf("caller is not authenticated: ...")
}
```

No equivalent guard exists in the bank precompile's `transfer` path.

### Impact Explanation
**Critical – Unauthorized transfer of bank-precompile-managed native assets.**

A malicious contract `M` can:
1. Mint `evm/M` tokens to a victim address (via the `mint` method, which is unrestricted as to recipient).
2. Later call `transfer(victim, attacker, balance)` on the bank precompile.
3. Because `denom = evm/M` and `from = victim`, `bankKeeper.SendCoins` moves the victim's `evm/M` balance to the attacker with no consent from the victim.

This constitutes an unauthorized transfer of precompile-controlled assets, matching the Critical impact tier ("unauthorized transfer … for … precompile-controlled assets").

### Likelihood Explanation
The entry path is fully unprivileged: any deployed EVM contract can call the bank precompile at address `0x0000…0064`. No admin key, governance vote, or special permission is required. The only precondition is that the victim holds a non-zero balance of the calling contract's `evm/<addr>` denom, which the attacker can arrange by minting to the victim first.

### Recommendation
Validate that the `sender` argument equals `contract.Caller()` before executing the transfer, mirroring the pattern already used in the relayer precompile:

```go
case TransferMethodName:
    ...
    sender := args[0].(common.Address)
    if sender != contract.Caller() {
        return nil, errors.New("sender must be the caller")
    }
    ...
```

Alternatively, remove the `sender` argument entirely and derive the debit account from `contract.Caller()`, consistent with how `mint` and `burn` derive the denom.

### Proof of Concept

1. Attacker deploys contract `M` on Cronos EVM.
2. `M` calls bank precompile `mint(victim_addr, 1000)` → victim now holds 1000 `evm/M`.
3. `M` calls bank precompile `transfer(victim_addr, attacker_addr, 1000)`.
4. Because `sender = victim_addr` is accepted without checking `contract.Caller() == victim_addr`, `bankKeeper.SendCoins` debits the victim and credits the attacker.
5. Victim loses 1000 `evm/M` with no approval or signature. [1](#0-0) [2](#0-1)
