### Title
EVM Chain ID Collision Enables Cross-Network Transaction Replay — (`fvm/evm/types/chainIDs.go`)

### Summary

`EVMChainIDFromFlowChainID` maps every Flow chain ID that is not `flow.Mainnet` or `flow.Testnet` to the same EVM chain ID (646, the PreviewNet value). Because EIP-155 replay protection is keyed on the EVM chain ID embedded in the transaction signature, a signed EVM transaction is cryptographically valid on every Flow network that resolves to chain ID 646. An unprivileged EVM transaction sender can therefore replay any EVM transaction from one such network on any other.

### Finding Description

`EVMChainIDFromFlowChainID` in `fvm/evm/types/chainIDs.go` has a `default` branch that returns `FlowEVMPreviewNetChainID` (646) for all Flow chain IDs other than `flow.Mainnet` and `flow.Testnet`:

```go
// fvm/evm/types/chainIDs.go
func EVMChainIDFromFlowChainID(flowChainID flow.ChainID) *big.Int {
    switch flowChainID {
    case flow.Mainnet:
        return FlowEVMMainNetChainID   // 747
    case flow.Testnet:
        return FlowEVMTestNetChainID   // 545
    default:
        return FlowEVMPreviewNetChainID // 646  ← Sandboxnet, Previewnet, Localnet, Emulator, Benchnet, BftTestnet all land here
    }
}
``` [1](#0-0) 

The affected Flow chain IDs that all collapse to EVM chain ID 646 are: `flow.Previewnet`, `flow.Sandboxnet`, `flow.Localnet`, `flow.Emulator`, `flow.Benchnet`, `flow.BftTestnet`, and `flow.MonotonicEmulator`. [2](#0-1) 

This function is the authoritative source for the EVM chain ID used in block context construction. It is called directly in `fvm/evm/offchain/blocks/block_context.go`:

```go
// fvm/evm/offchain/blocks/block_context.go
return types.BlockContext{
    ChainID: types.EVMChainIDFromFlowChainID(chainID),
    ...
}
``` [3](#0-2) 

The EVM chain ID in the block context is what the go-ethereum signer uses to validate EIP-155 signatures. The `GetSigner` / `GetDefaultSigner` functions in `fvm/evm/emulator/signer.go` derive the signer from `cfg.ChainConfig`, and `DefaultChainConfig` is explicitly set to `PreviewNetChainConfig` (chain ID 646):

```go
// fvm/evm/emulator/config.go
var DefaultChainConfig = PreviewNetChainConfig  // chain ID 646
``` [4](#0-3) 

Additionally, `WithChainID` silently ignores any chain ID that is not one of the three known values, leaving the config at the PreviewNet default:

```go
// fvm/evm/emulator/config.go
func WithChainID(chainID *big.Int) Option {
    return func(c *Config) *Config {
        switch chainID.Uint64() {
        case types.FlowEVMPreviewNetChainIDInUInt64:
            c.ChainConfig = PreviewNetChainConfig
        case types.FlowEVMTestNetChainIDInUInt64:
            c.ChainConfig = TestNetChainConfig
        case types.FlowEVMMainNetChainIDInUInt64:
            c.ChainConfig = MainNetChainConfig
        }
        // unknown chain IDs: silently no-op, config stays at PreviewNet default
        return c
    }
}
``` [5](#0-4) 

### Impact Explanation

Any EVM transaction signed with chain ID 646 (targeting previewnet, sandboxnet, the emulator, localnet, benchnet, or bfttestnet) is cryptographically indistinguishable from a transaction targeting any other network in that set. An attacker who observes a signed EVM transaction on one of these networks can submit it verbatim to any other network sharing chain ID 646, and the EVM signature verifier will accept it. This enables:

- **Cross-network asset replay**: A user's EVM token transfer or contract call signed on the emulator can be replayed on previewnet (a public network with real test tokens), draining the user's EVM account balance on previewnet without their consent.
- **Unauthorized account mutation**: The replayed transaction executes under the original signer's EVM address, mutating on-chain state (balances, contract storage) on the target network without the account owner's authorization.

### Likelihood Explanation

The entry path requires only that an attacker observe a signed EVM transaction from one of the affected networks (e.g., from a shared test environment, a public RPC endpoint, or a broadcast mempool) and submit it to another network sharing chain ID 646. No privileged access, staked node control, or key compromise is required. The `EVM.run` and `EVM.batchRun` Cadence entry points are callable by any unprivileged transaction sender, making this reachable from the Access API. [6](#0-5) 

### Recommendation

Assign a distinct EVM chain ID to every Flow network. Extend `EVMChainIDFromFlowChainID` to return unique values for `flow.Sandboxnet`, `flow.Localnet`, `flow.Emulator`, `flow.Benchnet`, `flow.BftTestnet`, and `flow.MonotonicEmulator` instead of collapsing them all to 646. Update `WithChainID` to handle (or explicitly reject) all known chain IDs rather than silently no-oping on unrecognized values. Update `DefaultChainConfig` to use a chain ID that is not shared with any live network.

### Proof of Concept

1. On the Flow emulator (chain ID 646), sign an EVM transfer transaction using `GetDefaultSigner()` (which uses chain ID 646 via `DefaultChainConfig = PreviewNetChainConfig`).
2. RLP-encode the signed transaction.
3. Submit the identical encoded bytes to Flow previewnet via a Cadence transaction calling `EVM.run(tx: encodedTx, coinbase: ...)`.
4. The go-ethereum signer on previewnet calls `EVMChainIDFromFlowChainID(flow.Previewnet)` → 646, constructs a signer for chain ID 646, and validates the signature — which passes because the transaction was signed for chain ID 646.
5. The transaction executes on previewnet under the original signer's EVM address, transferring assets without the account owner's authorization on that network. [1](#0-0) [7](#0-6)
