### Title
Unbounded `stakers` Vec Growth Enables Permanent Gas Exhaustion DoS on `get_stakers()` - (File: `src/staking/staking.cairo`)

### Summary
The `stakers` storage Vec is append-only: every new staker is pushed in but entries are never removed on unstake. The `get_stakers()` consensus function iterates over the **entire** Vec on every call. An attacker can permanently inflate this Vec by cycling stake/unstake across many addresses, recovering their principal each time, leaving permanent ghost entries that make every future `get_stakers()` call progressively more expensive.

### Finding Description
In `src/staking/staking.cairo`, the `stakers` field is declared as an append-only `Vec<ContractAddress>`:

```
/// Vector of staker addresses.
/// **Note**: Stakers are not removed from this vector when they unstake.
stakers: Vec<ContractAddress>,
``` [1](#0-0) 

Every call to `stake()` pushes the caller's address unconditionally:

```cairo
// Add staker address to the stakers vector.
self.stakers.push(staker_address);
``` [2](#0-1) 

The `get_stakers()` function, which is the consensus-layer entry point for determining the validator set each epoch, iterates over the **full range** of this Vec:

```cairo
for staker_address_ptr in self.stakers.into_iter_full_range() {
    let staker_address = staker_address_ptr.read();
    if !self.is_staker_active(:staker_address, :epoch_id) {
        continue;
    }
    ...
}
``` [3](#0-2) 

Even for inactive (fully unstaked) stakers, the loop must read each address from storage and evaluate `is_staker_active`. There is no early-exit or pruning mechanism.

**Attack path:**
1. Attacker creates N addresses, each funded with `min_stake` STRK.
2. Calls `stake()` with each address → N entries appended to `stakers` Vec.
3. Calls `unstake_intent()` on each, waits for the exit window, then calls `unstake_action()` → full principal recovered.
4. The `stakers` Vec now permanently contains N ghost entries that can never be removed.
5. Every future call to `get_stakers()` must iterate over all N ghost entries plus legitimate stakers.

The `assert_staker_address_not_reused` guard prevents address reuse, so each cycle requires a fresh address, but the attacker's net cost is only gas fees — the staked principal is fully returned. [4](#0-3) 

### Impact Explanation
`get_stakers()` is the function the consensus layer calls to obtain the validator set for a given epoch. As the `stakers` Vec grows without bound, each call to `get_stakers()` consumes more gas (at minimum one storage read per entry). With a sufficiently large Vec, the function will exceed Starknet's per-transaction gas ceiling and become permanently uncallable. This constitutes **unbounded gas consumption** and griefing of the protocol with no profit motive — the consensus layer loses the ability to enumerate validators, degrading or halting the pre-consensus attestation mechanism.

### Likelihood Explanation
Medium. The attacker's only recurring cost is gas fees; the staked principal is fully recovered after each unstake cycle. The `min_stake` requirement gates the rate of inflation but not the ultimate impact, since funds are returned. The damage is permanent: once ghost entries are written to the Vec, they cannot be removed by any existing function. The attack scales linearly with attacker gas budget.

### Recommendation
Remove the staker's address from the `stakers` Vec (or mark it as inactive with a tombstone flag) when `unstake_action()` completes. Alternatively, replace the Vec with a data structure that supports O(1) deletion (e.g., a swap-and-pop pattern or a separate active-staker count), so that `get_stakers()` only iterates over currently active stakers.

### Proof of Concept
1. Attacker deploys a script that creates 50,000 fresh addresses, each pre-funded with `min_stake` STRK.
2. Each address calls `stake()` → `stakers` Vec length grows to 50,000+.
3. Each address calls `unstake_intent()`, waits for `exit_wait_window`, then calls `unstake_action()` → all principal recovered.
4. `stakers` Vec still contains 50,000 entries (confirmed by the explicit comment: "Stakers are not removed from this vector when they unstake").
5. Any subsequent call to `get_stakers()` must execute 50,000+ storage reads and `is_staker_active` evaluations before returning results.
6. Repeating the cycle further inflates the Vec; at sufficient scale the function reverts with an out-of-gas error, permanently breaking the consensus validator-set query.
