### Title
Gateway Signature Verification Bypass via Overly Broad `skip_stateful_validations` Mempool Check — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway stateful path uses `account_tx_in_pool_or_recent_block` as a proxy for "a deploy_account transaction exists for this account." However, that check returns `true` for **any** transaction from the account in the pool — including future-nonce invoke transactions. An unprivileged attacker can exploit this to admit an Invoke transaction with an **invalid signature** to the mempool, bypassing the `__validate__` entry point entirely.

### Finding Description

The gateway stateful validation path in `extract_state_nonce_and_run_validations` calls `run_pre_validation_checks`, which calls `skip_stateful_validations`: [1](#0-0) 

The skip condition triggers when:
1. The transaction is `Invoke`
2. `tx.nonce() == Nonce(Felt::ONE)`
3. `account_nonce == Nonce(Felt::ZERO)` (account not yet deployed on-chain)
4. `mempool_client.account_tx_in_pool_or_recent_block(sender)` returns `true`

When `skip_validate = true`, `run_validate_entry_point` sets `execution_flags.validate = false`: [2](#0-1) 

Inside `StatefulValidator::perform_validations`, when `validate = false` for an Invoke, execution returns `Ok(())` immediately after `perform_pre_validation_stage` without ever calling `validate_tx` (the `__validate__` entry point): [3](#0-2) 

The `__validate__` entry point is the **only** place where the cryptographic signature is verified: [4](#0-3) 

The comment in `skip_stateful_validations` acknowledges the flaw in its own reasoning:

> "It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction **or transactions with future nonces that passed validations**."

Future-nonce invoke transactions that "passed validations" are not deploy_account transactions. An account with only a future-nonce invoke in the mempool is still undeployed, and a nonce=1 invoke for it with an invalid signature should be rejected — but is not.

The nonce validation in `validate_nonce` allows nonces in `[account_nonce, account_nonce + max_allowed_nonce_gap]` (default gap = 200) for invoke transactions: [5](#0-4) 

This means nonce=200 is accepted for an account with on-chain nonce=0, which is sufficient to trigger the bypass for a subsequent nonce=1 transaction.

### Impact Explanation

An attacker can admit Invoke transactions with **arbitrary/invalid signatures** to the mempool:

1. Attacker (or any account with on-chain nonce=0) submits `invoke(nonce=200, valid_sig)` → admitted to mempool.
2. `account_tx_in_pool_or_recent_block(sender)` now returns `true`.
3. Attacker submits `invoke(nonce=1, invalid_sig)` for the same sender.
4. Nonce check passes: `0 ≤ 1 ≤ 200`.
5. `validate_by_mempool` passes: no duplicate hash, nonce in range.
6. `skip_stateful_validations` returns `true`: nonce=1, account_nonce=0, account in pool.
7. `run_validate_entry_point` runs with `validate=false` → `__validate__` never called.
8. Transaction with invalid signature is admitted to the mempool.

Additionally, an attacker monitoring the mempool can submit nonce=1 invokes with invalid signatures **for victim accounts** that have any future-nonce transaction in the pool, without needing the victim's private key.

**Matching impact**: *High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.*

### Likelihood Explanation

The conditions are easily met: any account with on-chain nonce=0 and any transaction in the mempool (including a self-submitted future-nonce invoke) satisfies the bypass. No privileged access is required. The attacker can create fresh account addresses at will.

### Recommendation

Replace the overly broad `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a **deploy_account** transaction exists for the sender in the mempool. The mempool client should expose a dedicated `has_deploy_account_in_pool(address)` query, or the existing check should be restricted to only return `true` when a deploy_account transaction is present.

Alternatively, restrict the skip to only apply when the mempool confirms a deploy_account transaction is pending for the sender address, not any arbitrary transaction.

### Proof of Concept

```
// Precondition: account A has on-chain nonce = 0 (never deployed).

// Step 1: Submit a valid future-nonce invoke to populate the mempool.
invoke_tx_1 = InvokeV3 {
    sender_address: A,
    nonce: 200,
    signature: valid_signature,  // attacker knows private key for A
    ...
}
gateway.add_tx(invoke_tx_1)  // Admitted: nonce 200 ≤ 0 + 200

// Now: account_tx_in_pool_or_recent_block(A) == true

// Step 2: Submit a nonce=1 invoke with an INVALID signature.
invoke_tx_2 = InvokeV3 {
    sender_address: A,
    nonce: 1,
    signature: [0x0, 0x0],  // invalid/arbitrary signature
    ...
}
gateway.add_tx(invoke_tx_2)
// Nonce check: 0 <= 1 <= 200 ✓
// validate_by_mempool: no duplicate ✓
// skip_stateful_validations: nonce==1, account_nonce==0, in_pool==true → skip=true
// run_validate_entry_point: validate=false → __validate__ NOT called
// Result: invoke_tx_2 with invalid signature ADMITTED to mempool

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
