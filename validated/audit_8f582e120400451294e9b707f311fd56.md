### Title
WalletConnect Offer Confirmation Displays Understated Total Amount with Royalties for Multi-NFT Offers - (File: packages/gui/src/electron/commands/parseCommandDisplay.ts)

### Summary
The `formatAmountWithRoyalties` function in `parseCommandDisplay.ts` incorrectly calculates the "Total Amount with Royalties" shown in the WalletConnect spend-confirmation dialog when an offer involves multiple NFTs. The displayed total is systematically lower than the amount actually deducted from the user's wallet, causing the user to approve a transaction under a false understanding of its cost.

### Finding Description
When a WalletConnect-connected dApp calls `chia_wallet.take_offer` (or `chia_wallet.create_offer_for_ids`) with an offer that has multiple NFTs on one side, `parseCommandDisplay` builds a `walletDelta` and calls `withRoyaltyTotals` → `formatAmountWithRoyalties` to compute the "Total Amount with Royalties" label shown in the `Confirm.tsx` dialog.

The faulty logic in `formatAmountWithRoyalties`:

```typescript
// packages/gui/src/electron/commands/parseCommandDisplay.ts  L363-L368
const splitAmount = amount / BigInt(royaltyPercentages.length);   // ← divides by N
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
const totalAmount = amount + royaltyAmount;
```

This computes:

```
royaltyAmount = (amount / N) × Σ(royalty_i / 10000)
              = amount × avg(royalty_i) / 10000
```

But the Chia protocol applies **each** NFT's royalty to the **full** fungible amount independently:

```
actual_royaltyAmount = amount × Σ(royalty_i / 10000)
```

The display underestimates the royalty by a factor of **N** (the number of NFTs). The resulting `amountWithRoyalties` string is then rendered directly in the WalletConnect confirmation dialog:

```tsx
// packages/gui/src/electron/dialogs/Confirm/Confirm.tsx  L109-L113
{line.amountWithRoyalties && (
  <div className="text-xs text-chia-text-secondary">
    {i18n._(/* i18n */ { id: 'Total Amount with Royalties' })}: {line.amountWithRoyalties}{' '}
    {networkPrefix ? networkPrefix.toUpperCase() : 'XCH'}
  </div>
)}
```

The three code paths that compute the "same" royalty-inclusive amount are inconsistent:

| Code path | Formula |
|---|---|
| `formatAmountWithRoyalties` (display) | `amount × avg(royalties)` — **wrong** |
| `offerBuilderDataToOffer` (conflict check) | compounds multiplicatively — different |
| Chia protocol (actual spend) | `amount × Σ(royalties)` — correct | [1](#0-0) [2](#0-1) [3](#0-2) 

### Impact Explanation
The WalletConnect confirmation dialog is the sole security gate between a dApp request and a wallet spend. A user who sees "Total Amount with Royalties: 1.05 XCH" and approves will actually have 1.10 XCH (or more) deducted. The discrepancy scales with the number of NFTs and their royalty percentages. This causes a user to approve a transaction for the wrong amount — a direct match for the High impact criterion: *"WalletConnect state that causes a user to approve … the wrong … amount."* [4](#0-3) [5](#0-4) 

### Likelihood Explanation
Any WalletConnect-paired dApp can craft a `chia_wallet.take_offer` request containing an offer with two or more royalty-bearing NFTs. No special privileges are required. The victim only needs to have a WalletConnect session open and to click "Approve" on the confirmation dialog, which is the normal user flow. [6](#0-5) 

### Recommendation
Replace the split-then-sum formula with the correct additive formula: apply each royalty percentage to the **full** amount and sum the results.

```typescript
// Correct implementation
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (amount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
const totalAmount = amount + royaltyAmount;
```

Additionally, align the royalty estimation in `offerBuilderDataToOffer.ts` (which currently compounds multiplicatively) with the same additive formula so that all three code paths — display, conflict-detection, and protocol — are consistent. [7](#0-6) 

### Proof of Concept
The existing test in `parseCommandDisplay.test.ts` (lines 251–335) already demonstrates the bug: with two NFTs having royalties of 500 (5 %) and 10 (0.1 %), the test asserts `amountWithRoyalties: '0.00010255'` (2.55 % total royalty). The correct value is `0.0001051` (5.1 % total royalty — each royalty applied to the full 100 000 000 mojos). The test passes only because it was written to match the buggy output.

A dApp attacker constructs an offer with two NFTs each carrying a 10 % royalty and sends it via WalletConnect `take_offer`. The dialog shows "Total Amount with Royalties: 1.10 XCH" (10 % royalty displayed), but the actual spend is 1.20 XCH (20 % royalty charged). The user approves 1.10 XCH and loses 1.20 XCH. function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],
): string | undefined {
  if (royaltyPercentages.length === 0 || line.kind === 'nft') {
    return undefined;
  }

  const splitAmount = amount / BigInt(royaltyPercentages.length);
  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;

  if (line.kind === 'xch') {
    return mojoToChiaLocaleString(totalAmount);
  }

  return mojoToCATLocaleString(totalAmount);
}

function withRoyaltyTotals(
  items: DisplayWalletDeltaItemWithKey[],
  amounts: Record<string, bigint>,
  oppositeSideLines: DisplayWalletDeltaItem[],
): DisplayWalletDeltaItem[] {
  const royaltyPercentages = royaltyPercentagesForSide(oppositeSideLines);

  return items.map(({ key, line }) => {
    if (line.kind === 'nft') {
      return line;
    }

    const amount = amounts[key];
    const amountWithRoyalties = amount ? formatAmountWithRoyalties(line, amount, royaltyPercentages) : undefined;

    return amountWithRoyalties ? { ...line, amountWithRoyalties } : line;
  });
}

export async function parseCommandDisplay(command: string, params: Record<string, unknown>) {
  if (command === 'chia_wallet.take_offer') {
    if (!params.offer || typeof params.offer !== 'string') {
      throw new Error('Offer is not valid');
    }

    const offerSummary = await getOfferSummary(params.offer);
    if (!offerSummary || !offerSummary.summary || !offerSummary.success) {
      throw new Error('Offer is not valid');
    }

    const { summary } = offerSummary;

    const walletDelta = offerSummaryToWalletDelta(summary);
    const walletInfos = await getWalletInfos();
    const assetKinds = offerSummaryAssetKinds(summary);
    const royaltyPercentages = offerSummaryRoyaltyPercentages(summary);
    const fees = parseMojos(summary.fees);

    return {
      walletDelta: await walletDeltaToDisplay(walletDelta, walletInfos, assetKinds, royaltyPercentages, fees),
    };
  }
