### Title
Cross-COA Signature Replay: `signedData` Not Bound to `evmAddress` Enables Proof Reuse Across COAs Owned by the Same Cadence Account — (`fvm/evm/stdlib/contract.cdc`)

---

### Summary

`validateCOAOwnershipProof` verifies that a Cadence account's keys signed `signedData`, then checks that the capability at the supplied `path` resolves to a COA whose EVM address matches the supplied `evmAddress`. Because `signedData` is never required to commit to `evmAddress` or `CapabilityPath`, a valid proof obtained for COA_A can be replayed verbatim — with only `CapabilityPath` and `evmAddress` swapped to COA_B — and will pass all three checks, returning `isValid: true` for COA_B.

---

### Finding Description

`validateCOAOwnershipProof` performs three independent checks in sequence:

**1. Signature check** — verifies that the Cadence account's keys produced valid signatures over `signedData`: [1](#0-0) 

**2. Capability borrow** — borrows the COA resource at the caller-supplied `path`: [2](#0-1) 

**3. EVM address match** — checks that the borrowed COA's address equals the caller-supplied `evmAddress`: [3](#0-2) 

None of these checks bind `signedData` to `evmAddress` or `CapabilityPath`. The code itself documents this gap: [4](#0-3) 

The `COAOwnershipProof` struct that is RLP-encoded and passed as the `_sig` bytes contains `CapabilityPath` as an attacker-controlled field: [5](#0-4) 

`DecodeABIEncodedProof` in the Go precompile reads `evmAddress` from the ABI-encoded call arguments (set by the EVM caller) and `CapabilityPath` from the RLP proof bytes (also set by the EVM caller): [6](#0-5) 

The ERC-1271 entry point in `coa.sol` fixes `evmAddress = address(this)` (the COA being queried) but leaves `_sig` entirely attacker-supplied: [7](#0-6) 

**Attack construction:**

| Field | Legitimate proof for COA_A | Crafted proof for COA_B |
|---|---|---|
| `evmAddress` (ABI arg) | `COA_A.address` | `COA_B.address` |
| `Address` (RLP) | `victim_cadence` | `victim_cadence` (unchanged) |
| `CapabilityPath` (RLP) | `coa_a` | `coa_b` (swapped) |
| `KeyIndices` (RLP) | `[0]` | `[0]` (unchanged) |
| `Signatures` (RLP) | `sig(signedData)` | `sig(signedData)` (unchanged) |
| `signedData` (ABI arg) | `H` | `H` (unchanged) |

Trace through `validateCOAOwnershipProof` with the crafted proof:

- **Signature check**: `keyList.verify(signedData=H, "FLOW-V0.0-user")` — passes, because the signatures are valid for `victim_cadence`'s keys over `H`. [1](#0-0) 
- **Capability borrow**: `acc.capabilities.borrow<&EVM.CadenceOwnedAccount>(/public/coa_b)` — succeeds, because COA_B is published at that path. [2](#0-1) 
- **Address match**: `coaRef.address() == COA_B.address` — passes, because the borrowed resource is COA_B and `evmAddress` was set to `COA_B.address`. [3](#0-2) 
- **Result**: `ValidationResult(isValid: true)` — the proof is accepted for COA_B. [8](#0-7) 

---

### Impact Explanation

An attacker who has observed a valid `COAOwnershipProof` for COA_A (e.g., from an on-chain transaction where the victim called `isValidSignature` on COA_A for a DeFi protocol) can replay it against COA_B by changing only `CapabilityPath` in the RLP bytes. The result is that `isValidSignature` on COA_B returns `ValidERC1271Signature` (0x1626ba7e) for a hash the victim never intended to authorize for COA_B. [9](#0-8) 

Concrete downstream consequences:
- Any ERC-1271-gated DeFi protocol (token approvals, permit-style flows, order books) will accept the forged signature as coming from COA_B's owner, enabling unauthorized asset transfers from COA_B.
- Any bridge or withdrawal flow that uses `verifyCOAOwnershipProof` to authorize a withdrawal from COA_B will be bypassed.

---

### Likelihood Explanation

Preconditions that must hold simultaneously:

1. The victim Cadence account owns two or more COAs (nothing in the protocol prevents this; the `createCadenceOwnedAccount` function imposes no per-account limit).
2. The attacker has obtained a valid proof for COA_A over the target hash `H`. This is realistic: proofs are passed as calldata in EVM transactions and are therefore publicly observable on-chain.
3. The target DeFi protocol uses ERC-1271 (`isValidSignature`) for authorization and the attacker can trigger a call to COA_B's `isValidSignature` with the crafted `_sig`.

The comment in the contract acknowledges the gap and dismisses it as "low-risk" based on a historical wallet convention (one COA per account), not a protocol invariant. As multi-COA usage grows (e.g., treasury management, protocol-owned liquidity), the precondition becomes easier to satisfy. [4](#0-3) 

---

### Recommendation

Bind `signedData` to both `evmAddress` and `CapabilityPath` inside `validateCOAOwnershipProof` before performing the signature check. Concretely, replace the raw `signedData` passed to `keyList.verify` with a commitment that encodes all three: the caller-supplied `signedData`, the `evmAddress`, and the `CapabilityPath`. For example:

```cadence
let commitment = HashAlgorithm.SHA3_256.hash(
    signedData
        .concat(evmAddress.toConstantSized()!)
        .concat(path.toString().utf8)
)
let isValid = keyList.verify(
    signatureSet: signatureSet,
    signedData: commitment,
    domainSeparationTag: "FLOW-V0.0-user"
)
```

This ensures a signature produced for COA_A at `coa_a` cannot be replayed for COA_B at `coa_b`, even if both are owned by the same Cadence account. Wallets and off-chain signers must be updated to sign the commitment rather than raw `signedData`.

### Proof of Concept

```cadence
// On the Flow emulator:

// 1. Create one Cadence account with two COAs
transaction {
    prepare(acct: auth(Capabilities, SaveValue) &Account) {
        let coaA <- EVM.createCadenceOwnedAccount()
        acct.storage.save(<-coaA, to: /storage/coa_a)
        let capA = acct.capabilities.storage.issue<&EVM.CadenceOwnedAccount>(/storage/coa_a)
        acct.capabilities.publish(capA, at: /public/coa_a)

        let coaB <- EVM.createCadenceOwnedAccount()
        acct.storage.save(<-coaB, to: /storage/coa_b)
        let capB = acct.capabilities.storage.issue<&EVM.CadenceOwnedAccount>(/storage/coa_b)
        acct.capabilities.publish(capB, at: /public/coa_b)
    }
}

// 2. Obtain a real signature over signedData=H for COA_A
//    (sign H with the account's key using tag "FLOW-V0.0-user")
//    Build proof_A: {KeyIndices:[0], Address:victim, CapabilityPath:"coa_a", Signatures:[sig]}

// 3. Craft proof_B by RLP-re-encoding with CapabilityPath="coa_b", all else identical

// 4. Call validateCOAOwnershipProof directly:
script {
    import EVM from 0xEVM
    access(all) fun main(): Bool {
        let result = EVM.validateCOAOwnershipProof(
            address: victim_cadence_address,
            path: /public/coa_b,          // swapped
            signedData: H,
            keyIndices: [0],
            signatures: [sig],             // original sig for COA_A
            evmAddress: COA_B_evm_address  // COA_B's address
        )
        assert(result.isValid)             // passes — exploit confirmed
        return result.isValid
    }
}
```

The assertion passes, confirming that a proof signed for COA_A validates as ownership proof for COA_B, violating the invariant that each COA's proof must be independently unforgeable.

func DecodeABIEncodedProof(input []byte) (*types.COAOwnershipProofInContext, error) {
	index := 0
	caller, err := ReadAddress(input, index)
	index += FixedSizeUnitDataReadSize
	if err != nil {
		return nil, err
	}

	hash, err := ReadBytes32(input, index)
	index += Bytes32DataReadSize
	if err != nil {
		return nil, err
	}

	encodedProof, err := ReadBytes(input, index)
	if err != nil {
		return nil, err
	}

	return types.NewCOAOwnershipProofInContext(
		hash,
		types.Address(caller),
		encodedProof,
	)
}
