### Title
Unverified NFT Metadata Accepted When `metadataHash` Is Empty, Enabling Spoofed Name/Collection in Offer Confirmation - (File: `packages/gui/src/hooks/useFetchAndProcessMetadata.ts`)

### Summary

When an NFT is minted with an empty or absent `metadataHash` (on-chain value `0x`), the Chia GUI fetches and unconditionally trusts the JSON content served at `metadataUri`. Because the metadata's `name`, `collection.name`, `description`, and `attributes` fields are rendered directly in offer confirmation dialogs and NFT detail views without any integrity guard or unverified-metadata warning, an attacker can mint a worthless NFT whose metadata server returns content that impersonates a high-value NFT. A victim who opens the attacker's offer file sees the spoofed name and collection and may accept the offer, losing XCH or CATs in exchange for a worthless coin.

### Finding Description

`useFetchAndProcessMetadata` performs a hash check only when the `hash` argument is truthy:

```typescript
if (hash && !compareChecksums(checksum, hash)) {
  throw new Error('Checksum mismatch');
}
``` [1](#0-0) 

The caller, `useMetadataData`, passes `metadataHash` directly from the on-chain NFT record, which can be `undefined` or the sentinel value `"0x"`:

```typescript
const { metadataUris = [], metadataHash } = nft;
...
const metadata = await fetchAndProcessMetadata(firstUri, metadataHash);
``` [2](#0-1) 

When `metadataHash` is falsy the `if (hash && ...)` guard is skipped entirely, and the raw JSON from the attacker-controlled URI is parsed and stored as the authoritative `Metadata` object. [3](#0-2) 

`NFTTitle` renders `metadata.name` directly: [4](#0-3) 

`NFTMetadata` renders arbitrary metadata paths (name, description, collection.name) without any verification-status guard: [5](#0-4) 

These components are used in the offer confirmation flow (`NFTSummary`, `OfferSummaryRow`, `NFTDetailV2`): [6](#0-5) [7](#0-6) 

The `NFTDetails` panel suppresses the metadata hash row entirely when the hash is absent, so the user receives no indication that metadata is unverified: [8](#0-7) 

`NFTHashStatus` only reflects the integrity of the main data (image) URI, not the metadata URI, so its green checkmark does not imply that the displayed name or collection is trustworthy. [9](#0-8) 

### Impact Explanation

A victim who opens a crafted offer file sees a spoofed NFT name (e.g., "Chia Friends #1") and collection (e.g., "Chia Friends") in the offer confirmation dialog and NFT summary card. Believing they are acquiring a legitimate high-value NFT, they accept the offer and transfer XCH or CATs to the attacker in exchange for a worthless coin. This is a direct asset loss caused by unsafe trust of NFT metadata state in the offer confirmation flow, matching the **High** impact category: *"Corruption, spoofing, or unsafe trust of… NFT metadata… that causes a user to approve… the wrong asset."*

### Likelihood Explanation

Any unprivileged actor can mint a Chia NFT with `metadata_hash = 0x` and `metadata_uris` pointing to an attacker-controlled server. The attacker then creates a standard offer file and distributes it. No leaked keys, host compromise, or cryptographic break is required. The attack is fully self-contained and exploitable by anyone who can mint an NFT and create an offer.

### Recommendation

1. **Treat missing `metadataHash` as unverified**: When `metadataHash` is absent or `"0x"`, display a prominent "Unverified Metadata" warning badge alongside the NFT name and collection in all offer confirmation and NFT detail views — analogous to the existing `NFTHashStatus` chip for data URIs.
2. **Extend `NFTHashStatus` to cover metadata**: Add a separate metadata-hash verification state so users can see at a glance whether the displayed name/collection is hash-bound.
3. **Suppress or caveat unverified metadata fields in offer dialogs**: In `NFTSummary` and `OfferSummaryRow`, replace the raw `metadata.name` / `metadata.collection.name` with a placeholder (e.g., "Name unverified") when `metadataHash` is absent, to prevent impersonation in the highest-risk user flow.

### Proof of Concept

1. Attacker mints an NFT on Chia mainnet with:
   - `metadata_uris = ["https://attacker.example/meta.json"]`
   - `metadata_hash = 0x` (empty)
   - `data_uris` / `data_hash` pointing to any image (the image hash check passes normally)
2. Attacker's server at `/meta.json` returns:
   ```json
   {"name":"Chia Friends #1","collection":{"name":"Chia Friends","id":"..."},"description":"Original Chia Friends NFT"}
   ```
3. Attacker creates an offer: "Give me 50 XCH, receive this NFT."
4. Victim opens the `.offer` file in Chia GUI. `useMetadataData` fetches the metadata URI; `useFetchAndProcessMetadata` skips the hash check because `metadataHash` is falsy; the parsed object `{name: "Chia Friends #1", collection: {name: "Chia Friends"}}` is stored.
5. The offer confirmation dialog renders `<NFTTitle>` → "Chia Friends #1" and `<NFTSummary>` → collection "Chia Friends". No warning is shown.
6. Victim accepts the offer, losing 50 XCH and receiving the attacker's worthless NFT.
