### Title
`ExitToNear` Precompile Burns ERC-20 Tokens Without Refund When NEAR-Side Transfer Fails (Without `error_refund` Feature) - (File: `engine-precompiles/src/native.rs`)

### Summary

When the `error_refund` compile-time feature is disabled, the `ExitToNear` precompile permanently burns a user's ERC-20 tokens on Aurora but provides no recovery path if the subsequent NEAR-side `ft_transfer` promise fails. This creates a permanent accounting inconsistency: ERC-20 supply decreases while the NEP-141 balance held by Aurora does not, resulting in a permanent freeze of the user's bridged funds.

### Finding Description

The `ExitToNear` precompile's `run` function constructs `ExitToNearPrecompileCallbackArgs` with a `refund` field that is unconditionally set to `None` when the `error_refund` feature is not compiled in:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,   // <-- no refund args ever set
    transfer_near: transfer_near_args,
};
``` [1](#0-0) 

The callback `exit_to_near_precompile_callback` then checks the promise result. If the NEAR-side `ft_transfer` fails and `args.refund` is `None`, the else-branch silently returns `None` — no re-mint, no ETH credit, no recovery:

```rust
} else if let Some(args) = args.refund {
    // Exit call failed; need to refund tokens
    let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;
    ...
} else {
    None   // <-- silent no-op: tokens are gone
};
``` [2](#0-1) 

The V1 ERC-20 contract (`EvmErc20.sol`) encodes the precompile input **without** a refund address field, matching the non-`error_refund` wire format:

```solidity
bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
``` [3](#0-2) 

The burn happens unconditionally before the precompile call:

```solidity
_burn(_msgSender(), amount);
``` [4](#0-3) 

The test suite explicitly acknowledges this behavior as a known outcome when `error_refund` is absent:

```rust
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
``` [5](#0-4) 

The same pattern applies to ETH exits:

```rust
// If the refund feature is not enabled, then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let expected_balance = Wei::new_u64(INITIAL_ETH_BALANCE - ETH_EXIT_AMOUNT);
``` [6](#0-5) 

### Impact Explanation

When the NEAR-side `ft_transfer` (or `ft_transfer_call`) promise fails after a successful ERC-20 burn:

- The ERC-20 token supply on Aurora is permanently reduced (tokens burned).
- The NEP-141 balance held by the Aurora contract on NEAR is **not** reduced.
- The user receives neither their ERC-20 tokens back nor the NEP-141 tokens on NEAR.
- There is no on-chain recovery path: the burned ERC-20 tokens cannot be re-minted and the NEP-141 tokens remain locked in Aurora's account indefinitely.

This is a **permanent freeze of user funds** and an **ERC-20 mirror accounting inconsistency** (ERC-20 total supply diverges from the NEP-141 balance held by Aurora). [7](#0-6) 

### Likelihood Explanation

The NEAR-side `ft_transfer` promise can fail for several reasons reachable by any unprivileged user:

1. The recipient NEAR account is not registered with the NEP-141 contract (most common user error).
2. The NEP-141 contract is paused (e.g., via `pa_pause_feature`).
3. The Aurora contract's NEP-141 balance is insufficient due to a prior accounting bug.

A user calling `withdrawToNear` with an unregistered recipient account triggers this path deterministically. The V1 ERC-20 contract is deployed on mainnet for all bridged NEP-141 tokens that predate V2. [8](#0-7) 

### Recommendation

1. **Enable `error_refund` by default** in all production builds. The V2 ERC-20 contract (`EvmErc20V2.sol`) already encodes the sender as the refund address, matching the `error_refund` wire format.
2. **Migrate V1 ERC-20 contracts** to V2 or add a guard in the precompile that rejects calls from V1-format inputs when `error_refund` is enabled.
3. Alternatively, validate the NEAR recipient registration **before** burning ERC-20 tokens (though this is not possible atomically across the EVM/NEAR boundary). [9](#0-8) 

### Proof of Concept

1. Deploy a NEP-141 token and bridge it to Aurora (creating a V1 ERC-20 via `deploy_erc20_token`).
2. Bridge some NEP-141 tokens into Aurora via `ft_on_transfer` → user holds ERC-20 balance.
3. User calls `withdrawToNear("unregistered.near", amount)` on the V1 ERC-20 contract.
4. V1 ERC-20 burns `amount` tokens from the user and calls the `ExitToNear` precompile.
5. Aurora schedules a NEAR promise: `ft_transfer(receiver_id="unregistered.near", amount=...)`.
6. The promise fails because `unregistered.near` has no storage deposit with the NEP-141 contract.
7. `exit_to_near_precompile_callback` is invoked; `args.refund == None`; the else-branch returns `None`.
8. **Result**: User's ERC-20 balance is zero; NEP-141 balance of `unregistered.near` is zero; Aurora's NEP-141 balance is unchanged. Funds are permanently frozen. [10](#0-9) [11](#0-10)

```rust
pub fn exit_to_near_precompile_callback<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        env.assert_private_call()?;

        // This function should only be called as the callback of
        // exactly one promise.
        if handler.promise_results_count() != 1 {
            return Err(errors::ERR_PROMISE_COUNT.into());
        }

        let args: ExitToNearPrecompileCallbackArgs = io.read_input_borsh()?;

        let maybe_result = if let Some(PromiseResult::Successful(_)) = handler.promise_result(0) {
            if let Some(args) = args.transfer_near {
                let action = PromiseAction::Transfer {
                    amount: Yocto::new(args.amount),
                };
                let promise = PromiseBatchAction {
                    target_account_id: args.target_account_id,
                    actions: vec![action],
                };

                // Safety: this call is safe because it comes from the exit to near precompile, not users.
                // The call is to transfer the unwrapped wNEAR tokens.
                let promise_id = handler.promise_create_batch(&promise);
                handler.promise_return(promise_id);
            }

            None
        } else if let Some(args) = args.refund {
            // Exit call failed; need to refund tokens
            let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;

            if !refund_result.status.is_ok() {
                return Err(errors::ERR_REFUND_FAILURE.into());
            }

            Some(refund_result)
        } else {
            None
        };

        Ok(maybe_result)
    })
