### Title
Gateway Skips Signature Validation for Invoke nonce=1 Transactions When Any Sender Transaction Exists in Mempool, Enabling Invalid-Signature Admission and Fee-Escalation Displacement - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

`skip_stateful_validations` in the gateway stateful validator returns `true` (skip `__validate__`) for any Invoke transaction with `nonce=1` when `account_nonce=0` and `account_tx_in_pool_or_recent_block` returns `true`. The mempool check is used as a proxy for "a valid deploy_account exists," but it returns `true` for **any** transaction from that address in the pool. An attacker who observes a victim's pending `deploy_account` can submit an Invoke with `nonce=1`, an **invalid signature**, and a higher tip. The gateway admits it without signature verification. The mempool's fee-escalation logic then replaces the victim's legitimate Invoke with the attacker's invalid one, permanently losing the victim's transaction.

### Finding Description

**Root cause — `skip_stateful_validations`:** [1](#0-0) 

The function returns `true` when all three conditions hold:
1. The transaction is `ExecutableTransaction::Invoke`
2. `tx.nonce() == Nonce(Felt::ONE)`
3. `account_nonce == Nonce(Felt::ZERO)`
4. `mempool_client.account_tx_in_pool_or_recent_block(sender_address)` returns `true`

The comment states: *"It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."*

This reasoning is incorrect. `account_tx_in_pool_or_recent_block` returns `true` if **any** transaction from that address is in the pool or was seen in a recent committed block: [2](#0-1) 

It does not verify that the pooled transaction is specifically a `deploy_account`, nor that it carries a valid signature.

**Effect on validation:**

When `skip_validate=true`, `run_validate_entry_point` sets `execution_flags.validate = false`: [3](#0-2) 

This causes `AccountTransaction::validate_tx` to return `Ok(None)` immediately without calling the account's `__validate__` entry point: [4](#0-3) 

**Fee-escalation displacement:**

The mempool supports fee escalation: a new transaction at the same `(address, nonce)` replaces an existing one if its tip and `max_l2_gas_price` are sufficiently higher: [5](#0-4) 

The `validate_by_mempool` call that precedes `skip_stateful_validations` only checks nonce ranges and duplicate hashes — it does not verify signatures: [6](#0-5) 

### Impact Explanation

An attacker can:

1. Observe a victim's pending `deploy_account` for address A via P2P propagation (the mempool propagates transactions to peers).
2. Craft an Invoke transaction: `sender_address=A`, `nonce=1`, arbitrary calldata, **invalid signature**, tip and `max_l2_gas_price` exceeding the victim's legitimate Invoke by the fee-escalation threshold.
3. Submit it to the gateway. The gateway admits it because `skip_validate=true` (account_nonce=0, tx_nonce=1, A is in pool).
4. The mempool's fee-escalation logic replaces the victim's legitimate Invoke with the attacker's invalid one.
5. When the batcher executes the block: `deploy_account` succeeds, then the invalid Invoke fails in `__validate__` (signature check). The victim's legitimate Invoke is permanently gone from the mempool.

This matches the **High** impact scope: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

### Likelihood Explanation

- The attack requires only that the victim's `deploy_account` be visible in the mempool, which is the normal P2P propagation path.
- No privileged access is required; any unprivileged network participant can submit transactions to the gateway.
- The attacker does not need the victim's private key.
- The fee-escalation threshold is a configurable percentage (default 10%), so the attacker's cost is marginal.

### Recommendation

The `skip_stateful_validations` check should be tightened in one or both of the following ways:

1. **Verify the pooled transaction is specifically a `deploy_account`**: The mempool should expose a dedicated query (e.g., `deploy_account_in_pool(address)`) that returns `true` only when a `DeployAccount` transaction for that address is pending, not any arbitrary transaction.

2. **Perform a lightweight signature check even when skipping the full `__validate__` entry point**: The gateway could verify the ECDSA/Stark signature against the transaction hash before admitting the transaction, independent of the account contract's `__validate__` logic.

The analog to the GMX fix is exact: just as the GMX team replaced the unverified GMX price with a Chainlink cross-check, the sequencer should replace the unverified mempool-presence proxy with an authoritative check (either a deploy_account-specific query or a direct signature verification).

### Proof of Concept

```
1. Alice submits DeployAccount for address A (nonce=0, valid signature).
   → Gateway: full validation passes, admitted to mempool.
   → Mempool: account_tx_in_pool_or_recent_block(A) = true.

2. Alice submits Invoke for address A (nonce=1, valid signature, tip=100).
   → Gateway: skip_stateful_validations returns true (nonce=1, account_nonce=0, A in pool).
   → __validate__ NOT called. Admitted to mempool.

3. Attacker submits Invoke for address A (nonce=1, INVALID signature, tip=111).
   → Gateway stateless: passes (signature length check only, not cryptographic validity).
   → Gateway stateful: skip_stateful_validations returns true (same conditions).
   → __validate__ NOT called. Admitted to mempool.
   → Mempool fee-escalation: tip 111 > 100 * 1.10 = 110 → replaces Alice's Invoke.

4. Batcher executes block:
   → DeployAccount(A) succeeds. Account deployed.
   → Attacker's Invoke(A, nonce=1, invalid sig) → __validate__ fails → reverted.
   → Alice's Invoke is gone from the mempool. Alice's transaction is permanently lost.
```

        versioned_constants.disable_casm_hash_migration();

        let mut block_info = self.gateway_fixed_block_state_reader.get_block_info().await?;
        block_info.block_number = block_info.block_number.unchecked_next();
        let block_context = BlockContext::new(
            block_info,
            self.chain_info.clone(),
            versioned_constants,
            BouncerConfig::max(),
        );

        // Move state into the blocking task and run CPU-heavy validation.

        let cur_span = Span::current();

        tokio::task::spawn_blocking(move || {
            cur_span.in_scope(|| {

                blockifier_validator.validate(account_tx)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
