### Title
`broadcast_success` Always Broadcasts Hardcoded `true`, Making Failure-Detection Check Dead Code and Enabling Malicious Abort of DKG/Reshare/Refresh — (`src/dkg.rs`)

---

### Summary

The `broadcast_success` function in `src/dkg.rs` is documented to accept an error parameter and broadcast `false` (failure) when an error occurred. However, the implementation always broadcasts the hardcoded value `(true, session_id)` regardless of any error state. This makes the boolean failure-detection check dead code for honest participants, while simultaneously allowing any malicious participant to broadcast `(false, session_id)` and force all honest parties to abort DKG, reshare, or refresh — even when all honest parties have successfully completed every prior validation step.

---

### Finding Description

The `broadcast_success` function carries the following documentation comment:

```
/// This function takes err as input.
/// If err is None then broadcast success
/// otherwise, broadcast failure
/// If during broadcast it receives an error then propagates it
/// This function is used in the final round of DKG
``` [1](#0-0) 

However, the actual function signature has **no `err` parameter**, and the broadcast payload is hardcoded to `true`:

```rust
async fn broadcast_success(
    chan: &mut SharedChannel,
    participants: &ParticipantList,
    me: Participant,
    session_id: HashOutput,
) -> Result<(), ProtocolError> {
    // broadcast node me succeded
    let vote_list = do_broadcast(chan, participants, me, (true, session_id)).await?;
``` [2](#0-1) 

The function then checks whether any participant broadcast `false`:

```rust
if !vote_list.iter().all(|&(boolean, _)| boolean) {
    return Err(ProtocolError::AssertionFailed(
        "A participant
            seems to have failed its checks. Aborting Protocol!"
            .to_string(),
    ));
}
``` [3](#0-2) 

Because every honest participant always broadcasts `true`, this check is **dead code** for honest participants — it can never trigger from an honest party's broadcast. The only way it can trigger is if a malicious participant deviates and broadcasts `(false, session_id)`.

This function is called unconditionally at the end of `do_keyshare`, which is the shared implementation for DKG, reshare, and refresh:

```rust
// Step 5.4 and Step 5.5
broadcast_success(&mut chan, &participants, me, session_id).await?;
``` [4](#0-3) 

The protocol specification (Step 5.4 in `docs/dkg.md`) states: *"Each $P_i$ reliably broadcasts $\mathsf{success_i}$"* — implying participants should be able to broadcast failure. The implementation breaks this by hardcoding success. [5](#0-4) 

---

### Impact Explanation

**High: Permanent denial of key generation, reshare, and refresh for honest parties.**

A single malicious participant who has behaved honestly throughout all prior rounds of DKG (sent valid commitments, valid shares, passed all checks) can, at the final `broadcast_success` step, deviate and broadcast `(false, session_id)` instead of `(true, session_id)`. The echo broadcast protocol (`do_broadcast`) guarantees that all honest parties receive and agree on this `false` value. The check at line 329 then causes **all** honest participants to abort with `ProtocolError::AssertionFailed`, even though every honest party has a valid key share and DKG would otherwise have completed successfully.

This violates the protocol's security guarantee: DKG should complete successfully when the number of malicious parties is within the documented fault tolerance. Instead, a **single** malicious participant can abort DKG for all honest parties at the last step, at zero cost (they need not reveal any secret material or deviate earlier).

The same vulnerability applies to reshare and refresh, since both call `do_keyshare` and thus `broadcast_success`. [6](#0-5) 

---

### Likelihood Explanation

**High.** Any participant in the DKG/reshare/refresh protocol can execute this attack. The attacker only needs to deviate at the very last protocol step, after behaving honestly throughout all prior rounds. No cryptographic material needs to be compromised. The attack is deterministic and requires no special knowledge beyond participation in the protocol.

---

### Recommendation

Add an `err: Option<ProtocolError>` parameter to `broadcast_success` (matching the documented intent). When `err` is `Some`, broadcast `(false, session_id)` to signal failure to all other participants. When `err` is `None`, broadcast `(true, session_id)`. Call sites should pass the result of prior validation steps:

```rust
async fn broadcast_success(
    chan: &mut SharedChannel,
    participants: &ParticipantList,
    me: Participant,
    session_id: HashOutput,
    err: Option<&ProtocolError>,
) -> Result<(), ProtocolError> {
    let success = err.is_none();
    let vote_list = do_broadcast(chan, participants, me, (success, session_id)).await?;
    // ... existing checks ...
}
```

This restores the intended semantics: honest participants who fail can signal failure to others, and the boolean check becomes meaningful.

---

### Proof of Concept

1. N participants begin DKG. Malicious participant P_m behaves honestly through all rounds 1–5 (sends valid commitments, valid shares, passes all checks).
2. All honest participants reach `broadcast_success` (line 531 of `src/dkg.rs`).
3. P_m deviates: instead of calling `broadcast_success` with `(true, session_id)`, it directly calls `do_broadcast` with `(false, session_id)`.
4. The echo broadcast protocol (`do_broadcast` → `reliable_broadcast_receive_all`) guarantees all honest parties agree on P_m's broadcast value of `false`.
5. The check at line 329 (`if !vote_list.iter().all(|&(boolean, _)| boolean)`) evaluates to `true` for all honest participants.
6. All honest participants return `Err(ProtocolError::AssertionFailed("A participant seems to have failed its checks. Aborting Protocol!"))`.
7. DKG is permanently aborted for all honest parties, even though every honest party holds a valid key share. [5](#0-4) [7](#0-6)

    chan: SharedChannel,
    participants: ParticipantList,
    me: Participant,

    old_public_key: VerifyingKey<C>,
    old_participants: ParticipantList,
    mut rng: impl CryptoRngCore,
) -> Result<KeygenOutput<C>, ProtocolError> {
    let threshold = threshold.into();
    let intersection = old_participants.intersection(&participants);
