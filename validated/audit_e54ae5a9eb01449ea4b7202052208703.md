### Title
Gateway Stateful Validator Checks L2 Gas Price Against Stale Previous-Block Price While Blockifier Enforces Current-Block Price, Admitting Transactions That Fail Execution - (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The gateway's `StatefulTransactionValidator::validate_resource_bounds` checks a transaction's `l2_gas.max_price_per_unit` against the **previous committed block's** L2 gas price. The blockifier's `AccountTransaction::check_fee_bounds`, called during actual sequencing, checks the same field against the **block being built's** L2 gas price, which is computed by the EIP-1559-style fee market and rises whenever blocks are full. When gas prices increase between blocks, transactions admitted by the gateway will fail at execution time with `MaxGasPriceTooLow`, meaning the gateway/mempool admission accepts transactions that are invalid for sequencing.

### Finding Description

In `StatefulTransactionValidator::validate_resource_bounds` (lines 223–243 of `crates/apollo_gateway/src/stateful_transaction_validator.rs`), the gateway reads `previous_block_l2_gas_price` from `gateway_fixed_block_state_reader.get_block_info()` and enforces:

```
tx.l2_gas.max_price_per_unit >= (min_gas_price_percentage / 100) * previous_block_l2_gas_price
```

The code carries an explicit acknowledgement of the problem:

```rust
// TODO(Arni): getnext_l2_gas_price from the block header.
let previous_block_l2_gas_price = self
    .gateway_fixed_block_state_reader
    .get_block_info()
    .await?
    .gas_prices
    .strk_gas_prices
    .l2_gas_price;
```

The `run_validate_entry_point` path (lines 302–356) also reuses the same previous-block `BlockInfo` (only incrementing `block_number`, not the gas prices), so the gateway's blockifier-level validation is equally stale.

During actual sequencing, the consensus orchestrator computes the next block's L2 gas price via `calculate_next_l2_gas_price_for_fin` (`crates/apollo_consensus_orchestrator/src/fee_market/mod.rs`, lines 55–77) using the EIP-1559 formula:

```rust
// gas_used > gas_target → price rises
let price_change = (price_u256 * gas_delta) / denominator;
let adjusted_price_u256 = if gas_used > gas_target { price_u256 + price_change } ...
```

The batcher builds blocks with this higher price. When the blockifier executes a transaction inside that block, `AccountTransaction::check_fee_bounds` (lines 374–475 of `crates/blockifier/src/transaction/account_transaction.rs`) enforces:

```rust
if resource_bounds.max_price_per_unit < actual_gas_price.get() {
    insufficiencies_resource.push(ResourceBoundsError::MaxGasPriceTooLow { ... });
}
```

where `actual_gas_price` is the **current block's** L2 gas price — the one computed by the fee market, not the previous block's price used at admission.

The default `min_gas_price_percentage = 100` (from `StatefulTransactionValidatorConfig::default()` in `crates/apollo_gateway_config/src/config.rs`, line 296) means the gateway threshold equals exactly the previous block's price. Any transaction with `max_price_per_unit = previous_block_price` passes the gateway but fails the blockifier whenever the fee market raises the price for the next block.

### Impact Explanation

The gateway is the admission gatekeeper. Transactions it admits are expected to be executable. When the L2 gas price rises between blocks — the normal outcome whenever a block is full — any transaction with `max_price_per_unit` in the range `[previous_block_price, next_block_price)` will:

1. Pass `validate_resource_bounds` at the gateway (admitted to mempool).
2. Fail `check_fee_bounds` inside `perform_pre_validation_stage` at execution time with `TransactionPreValidationError::TransactionFeeError(InsufficientResourceBounds { MaxGasPriceTooLow })`.

This matches the **High** impact: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

An attacker can deliberately flood the mempool with such transactions during any period of high network utilization, polluting the mempool with entries that consume sequencer resources (mempool storage, execution slots, fee-check overhead) but never land in a block.

### Likelihood Explanation

The condition is triggered automatically whenever consecutive blocks are above the gas target — a routine occurrence during normal network load. No privileged access is required. Any unprivileged user can submit a transaction with `max_price_per_unit` set to the previous block's L2 gas price (publicly observable from the block header), which is exactly the value that passes the gateway threshold but may fall below the next block's price.

### Recommendation

Replace the stale `previous_block_l2_gas_price` in `validate_resource_bounds` with the predicted next-block L2 gas price. The fee-market formula already exists in `calculate_next_l2_gas_price_for_fin`; the gateway should call it (or read the `next_l2_gas_price` field already stored in block headers by the consensus orchestrator) to compute the price the batcher will actually use, and validate against that value. The existing TODO comment (`// TODO(Arni): getnext_l2_gas_price from the block header.`) identifies exactly this fix.

### Proof of Concept

1. Block N is full (`l2_gas_used > gas_target`). Its L2 gas price (STRK) is **P**.
2. The fee market computes the next block's price as **P' > P** via `calculate_next_base_gas_price`.
3. An attacker submits an `AllResources` invoke transaction with `l2_gas.max_price_per_unit = P`.
4. **Gateway `validate_resource_bounds`**: `P >= 100% × P` → **passes**. Transaction enters the mempool.
5. The batcher builds block N+1 with gas price **P'**.
6. **Blockifier `check_fee_bounds`** inside `perform_pre_validation_stage`: `P < P'` → `MaxGasPriceTooLow` → **fails**.
7. The transaction is rejected at execution time despite having been admitted by the gateway.

Repeating step 3 at scale during any busy period fills the mempool with transactions that will never be sequenced, wasting mempool capacity and execution resources. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5)

```rust
pub fn calculate_next_base_gas_price(
    price: GasPrice,
    gas_used: GasAmount,
    gas_target: GasAmount,
    min_gas_price: GasPrice,
) -> GasPrice {
    let versioned_constants = VersionedConstants::latest_constants();
    assert!(
        gas_target < versioned_constants.max_block_size,
        "Gas target must be lower than max block size."
    );
    assert!(gas_target.0 > 0, "Gas target must be greater than zero.");
    assert!(
        versioned_constants.gas_price_max_change_denominator > 0,
        "Denominator constant must be greater than zero."
    );

    // If the current price is below the minimum, apply a gradual adjustment and return early.
    // This allows the price to increase by at most 1/MIN_GAS_PRICE_INCREASE_DENOMINATOR per block.
    if price < min_gas_price {
        let max_increase = price.0 / MIN_GAS_PRICE_INCREASE_DENOMINATOR;
        let adjusted = price.0 + max_increase;
        // Cap at min_gas_price to avoid overshooting
        let adjusted_price = adjusted.min(min_gas_price.0);
        info!(
            "Fee Market: Price {} below minimum gas price {}, adjusted price: {} )",
            price.0, min_gas_price.0, adjusted_price
        );
        return GasPrice(adjusted_price);
    }

    // Use U256 to avoid overflow, as multiplying a u128 by a u64 remains within U256 bounds.
    let gas_delta = U256::from(gas_used.0.abs_diff(gas_target.0));
    let gas_target_u256 = U256::from(gas_target.0);
    let price_u256 = U256::from(price.0);

    // Calculate price change by multiplying first, then dividing. This avoids the precision loss
    // that occurs when dividing before multiplying.
    let denominator =
        gas_target_u256 * U256::from(versioned_constants.gas_price_max_change_denominator);
    let price_change = (price_u256 * gas_delta) / denominator;

    let adjusted_price_u256 =
        if gas_used > gas_target { price_u256 + price_change } else { price_u256 - price_change };

    // Sanity check: ensure direction of change is correct
    assert!(
        gas_used > gas_target && adjusted_price_u256 >= price_u256
            || gas_used <= gas_target && adjusted_price_u256 <= price_u256
    );

    // Price should not realistically exceed u128::MAX, bound to avoid theoretical overflow.
    let adjusted_price = u128::try_from(adjusted_price_u256).unwrap_or(u128::MAX);
    GasPrice(max(adjusted_price, min_gas_price.0))
```
