### Title
Missing Validation of Participant Contributions in CKD Protocol Allows Malicious Participant to Corrupt Derived Confidential Key - (File: src/confidential_key_derivation/protocol.rs)

### Summary
The CKD coordinator in `do_ckd_coordinator` aggregates `(norm_big_y, norm_big_c)` contributions from each participant and produces a `CKDOutput` without performing any validation on the received values. A single malicious participant can send arbitrary crafted group elements, causing the coordinator to output a corrupted `(Y, C)` pair. The TEE then unmasks an incorrect confidential key with no ability to detect the manipulation.

### Finding Description

In `do_ckd_coordinator`, the coordinator collects each participant's `CKDOutput` and blindly adds the components together: [1](#0-0) 

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
```

No check is performed to verify that:
- The received `big_y` or `big_c` values are not the identity element.
- The received values are consistent with the participant's public key share (i.e., no zero-knowledge proof of correct computation is required or verified).
- The final aggregated `(Y, C)` satisfies any expected algebraic relationship.

The correct per-participant contribution is `(lambda_i * y_i * G, lambda_i * (x_i * H(pk, app_id) + y_i * app_pk))`, computed in `compute_signature_share`: [2](#0-1) 

A malicious participant can substitute any `(Y_i', C_i')` for their correct share. The coordinator has no mechanism to reject it. The final output becomes `(Y_correct + delta_Y, C_correct + delta_C)` where `delta_Y = Y_i' - lambda_i * y_i * G` and `delta_C = C_i' - lambda_i * (x_i * H(pk, app_id) + y_i * app_pk)`. The TEE unmasks this as:

```
C' - app_sk * Y' = x * H(pk, app_id) + delta_C - app_sk * delta_Y
```

which is an incorrect confidential key. The TEE cannot detect the corruption because it does not know the expected derived key in advance (it would require knowing the aggregate secret `x`).

**Contrast with other protocols in the same codebase.** The OT-based ECDSA presign protocol performs explicit consistency checks on aggregated participant contributions against public commitments before proceeding: [3](#0-2) [4](#0-3) 

The robust ECDSA presign protocol similarly checks that interpolated values match received ones and that intermediate scalars are non-zero: [5](#0-4) 

The CKD protocol has no equivalent safeguard.

### Impact Explanation

A malicious participant causes the coordinator to output a corrupted `CKDOutput`. The TEE unmasks an incorrect confidential key and uses it for downstream operations (e.g., signing, encryption) without any indication of failure. Honest parties accept the corrupted output as valid. This directly matches the allowed High impact: **Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.**

### Likelihood Explanation

The attacker must be one of the participants in the CKD protocol session. The CKD ciphersuite is documented under the `MaxMalicious` threshold model, implying the design intent is to tolerate up to `t` malicious participants. A single malicious participant is sufficient to corrupt the output. No special privilege beyond participation is required; the attack is a straightforward substitution of the outgoing message.

### Recommendation

1. **Require a zero-knowledge proof of correct computation** alongside each `(norm_big_y, norm_big_c)` contribution. Each participant should prove in zero-knowledge that their `C_i` was formed as `x_i * H(pk, app_id) + y_i * app_pk` using their committed public key share.
2. **Add a pairing-based consistency check at the coordinator** after aggregation. The coordinator can verify `e(C, G2) == e(H(pk, app_id), pk) * e(Y, app_pk)` using the aggregate public key `pk` and the known `app_pk`. This check holds for an honest output and fails for any corrupted one.
3. **Reject identity-element contributions** as an immediate sanity check before aggregation.

### Proof of Concept

1. Honest participants `P_1, ..., P_{n-1}` compute and send correct `(norm_big_y_i, norm_big_c_i)`.
2. Malicious participant `P_n` sends `(G, G)` (the generator point) instead of their correct share.
3. The coordinator at `do_ckd_coordinator` adds all values including `P_n`'s crafted contribution with no error.
4. The resulting `CKDOutput` encodes `(Y + G - lambda_n * y_n * G, C + G - lambda_n * (x_n * H(pk, app_id) + y_n * app_pk))`.
5. The TEE calls `unmask(app_sk)` and derives `x * H(pk, app_id) + (G - lambda_n * (x_n * H(pk, app_id) + y_n * app_pk)) - app_sk * (G - lambda_n * y_n * G)`, which is an incorrect key.
6. No error is raised anywhere in the protocol; the corrupted key is silently accepted.
