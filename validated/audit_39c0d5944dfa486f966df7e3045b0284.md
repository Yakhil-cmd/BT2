### Title
Unauthenticated Participant Contribution Allows CKD Output Corruption — (`src/confidential_key_derivation/protocol.rs`)

### Summary

`do_ckd_coordinator` blindly aggregates `CKDOutput` values received from participants via `recv_from_others` with zero cryptographic validation. Any valid participant can send arbitrary G1 group elements (including the identity) as their `(big_y, big_c)` contribution, causing the coordinator to produce a `CKDOutput` that does not equal `msk · H(pk, app_id)`.

### Finding Description

**Serialization layer — no type barrier:**

Messages are serialized with `rmp_serde` (MessagePack) in compact/array mode. [1](#0-0) 

In this mode, a named struct like `CKDOutput { big_y, big_c }` serializes identically to a 2-tuple `(ElementG1, ElementG1)`. A malicious participant can craft any raw bytes that decode as a valid `CKDOutput` and inject them via the `Protocol::message()` entry point. [2](#0-1) 

**`recv_from_others` — no validation:**

The helper simply deserializes whatever bytes arrive from any listed participant and returns them. [3](#0-2) 

There is no subgroup check, no identity check, and no proof-of-knowledge.

**`do_ckd_coordinator` — blind aggregation:**

The coordinator iterates over the received outputs and adds `big_y` and `big_c` directly into its running sums: [4](#0-3) 

No commitment, no Schnorr proof, no consistency check against the participant's public key share is performed.

**`CKDOutput::new` — no invariant enforcement:** [5](#0-4) 

The constructor accepts any pair of G1 elements unconditionally.

### Impact Explanation

The correct aggregation requires every participant `i` to contribute `(λ_i · y_i · G, λ_i · (x_i · H(pk, app_id) + y_i · A))`. The final unmask step recovers `C − a·Y = msk · H(pk, app_id)`. [6](#0-5) 

If a malicious participant substitutes `big_y = identity` and `big_c = identity`, their Lagrange-weighted secret share `λ_i · x_i · H(pk, app_id)` is silently dropped from the sum. The coordinator returns a `CKDOutput` whose `unmask(app_sk)` equals `sum_{j≠i} λ_j · x_j · H(pk, app_id)`, which is not `msk · H(pk, app_id)`. The app receives an incorrect BLS signature and derives a wrong key. More generally, the attacker can inject any valid G1 point to shift the output to an arbitrary value.

### Likelihood Explanation

The attacker only needs to be a valid member of the `ParticipantList` passed to `ckd()`. No key material needs to be leaked; the attack is a single-round message substitution. The `Protocol::message()` API is the public entry point and accepts raw bytes from any caller. [7](#0-6) 

### Recommendation

Add a zero-knowledge proof of correct contribution. Each participant should send a Schnorr proof alongside `(big_y, big_c)` proving knowledge of `y_i` such that `big_y = y_i · G` and `big_c = x_i · H(pk, app_id) + y_i · A` (using the publicly known `H(pk, app_id)` and `A`). The coordinator must verify each proof before aggregating. Alternatively, use a verifiable ElGamal encryption scheme with a NIZK. At minimum, reject identity contributions, though that alone does not prevent arbitrary-point substitution.

### Proof of Concept

```rust
// Malicious participant: instead of calling do_ckd_participant honestly,
// craft a message with identity elements and inject it via Protocol::message().

use blstrs::G1Projective;
use group::Group;

// Serialize CKDOutput { big_y: identity, big_c: identity }
let identity = G1Projective::identity();
let crafted = CKDOutput::new(identity, identity);
let mut bytes = header_bytes.to_vec();
rmp_serde::encode::write(&mut bytes, &crafted).unwrap();

// Deliver as the malicious participant's message to the coordinator's protocol
coordinator_protocol.message(malicious_participant, bytes);

// The coordinator aggregates: norm_big_y += identity (no-op), norm_big_c += identity (no-op)
// Final CKDOutput.unmask(app_sk) == sum_{j≠malicious} λ_j·x_j·H(pk,app_id)
//                                 ≠ msk · H(pk, app_id)
assert_ne!(ckd_output.unmask(app_sk), hash_app_id_with_pk(&pk, &app_id) * msk);
```

The coordinator's aggregation loop at lines 50–55 of `src/confidential_key_derivation/protocol.rs` will silently incorporate the crafted identity values, producing a corrupted `CKDOutput`. [4](#0-3)

pub async fn recv_from_others<T>(
    chan: &SharedChannel,
    waitpoint: u64,
    participants: &ParticipantList,
    me: Participant,
) -> Result<Vec<(Participant, T)>, ProtocolError>
where
    T: serde::de::DeserializeOwned,
{
    let mut seen = ParticipantCounter::new(participants);
    seen.put(me);
    let mut messages = Vec::with_capacity(participants.others(me).count());

    while !seen.full() {
        let (from, msg) = chan.recv(waitpoint).await?;
        if seen.put(from) {
            messages.push((from, msg));
        }
    }

    Ok(messages)
}
