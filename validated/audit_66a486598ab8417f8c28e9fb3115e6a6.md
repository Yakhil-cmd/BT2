### Title
Missing Signer-Address Ownership Check in `RegisterEncryptionKey` Allows Attacker to Overwrite Any Victim's E2EE Key — (`x/e2ee/keeper/keeper.go`, `x/e2ee/types/msg.go`)

### Summary

`MsgRegisterEncryptionKey` contains two compounding defects: (1) the keeper never verifies that the transaction signer matches `req.Address`, so any account can register a key for any other address; (2) `ValidateRecipientKey` only calls `age.ParseX25519Recipient` and does not reject degenerate curve points. Together, an attacker can silently replace a victim's on-chain encryption key with an attacker-controlled or degenerate key, causing all subsequent messages encrypted to the victim's on-chain key to be decryptable by the attacker.

### Finding Description

**Issue 1 — Missing ownership check (primary)**

`Keeper.RegisterEncryptionKey` writes the supplied key directly to state without any signer verification: [1](#0-0) 

The `MsgRegisterEncryptionKey` proto struct carries a free-form `Address` field: [2](#0-1) 

`ValidateBasic` only checks bech32 format and key syntax — it never asserts `signer == m.Address`: [3](#0-2) 

The CLI sets `Address` from the local signer as a convention, but a raw transaction can supply any address: [4](#0-3) 

**Issue 2 — No degenerate-point rejection (secondary)**

`ValidateRecipientKey` delegates entirely to `age.ParseX25519Recipient`, which accepts any syntactically valid bech32-encoded 32-byte value, including the all-zeros point and known low-order points: [5](#0-4) 

**Attack path**

The `encrypt` CLI queries the on-chain key for a recipient address and passes it directly to `age.Encrypt`: [6](#0-5) 

An attacker submits a raw `MsgRegisterEncryptionKey{Address: victim, Key: attacker_pubkey}`. From that point, any sender who runs `encrypt --recipient victim` fetches the attacker's key from chain and encrypts to it. The victim's legitimate key is silently gone.

### Impact Explanation

Corruption of e2ee key/message state with direct security impact (allowed High scope). All messages encrypted to the victim's on-chain key after the overwrite are readable by the attacker. The victim may not detect the substitution until they attempt to decrypt and fail.

### Likelihood Explanation

Fully unprivileged — requires only the ability to submit a standard Cosmos SDK transaction. No special role, leaked key, or external assumption is needed. The attack is silent and the victim has no on-chain notification mechanism.

### Recommendation

1. In `Keeper.RegisterEncryptionKey`, extract the signer from the SDK context and assert it equals `req.Address` before writing to state.
2. In `ValidateRecipientKey`, after parsing, check the decoded 32-byte public key against the known list of low-order/degenerate X25519 points (all-zeros, the eight low-order points of Curve25519) and return an error if matched.

### Proof of Concept

```
// Attacker controls account B, victim is account A.
// 1. Attacker generates their own age keypair: age_priv_B, age_pub_B
// 2. Attacker broadcasts:
MsgRegisterEncryptionKey{
    Address: "cosmos1<victim_A>",
    Key:     "age1<attacker_pub_B_bech32>",
}
// No signer check → accepted, victim's key overwritten.
// 3. Any sender runs: cronosd tx e2ee encrypt --recipient cosmos1<victim_A> secret.txt
//    → encrypt.go queries chain, gets age_pub_B, encrypts to attacker.
// 4. Attacker decrypts with age_priv_B.
//
// Degenerate variant: replace step 1 with the all-zeros X25519 pubkey
// (age1<bech32(0x00...00)>). age.ParseX25519Recipient accepts it.
// ECDH with all-zeros yields all-zeros shared secret → anyone can decrypt.
```
