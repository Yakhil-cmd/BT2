### Title
ETH Permanently Frozen at `exit_to_near` Precompile Address on Failed NEAR Promise When `error_refund` Feature Is Disabled - (File: engine-precompiles/src/native.rs)

### Summary

When the `error_refund` compile-time feature is not enabled, ETH (base tokens) transferred to the `ExitToNear` precompile address during a bridge-out operation is permanently frozen at that address if the corresponding NEAR-side `ft_transfer` promise fails. There is no recovery path for these funds.

### Finding Description

The `ExitToNear` precompile (`exit_to_near::ADDRESS`) handles bridging ETH out of Aurora to NEAR. When a user calls this precompile with an ETH value, the EVM deducts the ETH from the caller and credits it to the precompile address. A NEAR promise (`ft_transfer` or `ft_transfer_call`) is then scheduled to complete the bridge.

The refund callback is conditionally compiled:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,
    transfer_near: transfer_near_args,
};
``` [1](#0-0) 

When `error_refund` is not enabled, `refund` is always `None`. The promise selection logic then skips attaching any callback:

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // no callback attached
} else {
    PromiseArgs::Callback(...)              // callback with refund logic
};
``` [2](#0-1) 

If the NEAR-side `ft_transfer` fails (e.g., the recipient account is not registered with the eth-connector), no `exit_to_near_precompile_callback` is invoked, so `refund_on_error` is never called. The ETH remains at the `exit_to_near` precompile address with no mechanism to recover it.

The only recovery path is `refund_on_error`, which transfers ETH back from the precompile address to the user:

```rust
// ETH exit; transfer ETH back from precompile address
let exit_address = exit_to_near::ADDRESS;
...
engine.call(&exit_address, &refund_address, amount, ...)
``` [3](#0-2) 

This function is only reachable through `exit_to_near_precompile_callback`: [4](#0-3) 

Which is only scheduled when `error_refund` is enabled. Without it, the precompile address silently accumulates ETH with no withdrawal function.

### Impact Explanation

**Critical — Permanent freezing of funds.**

ETH sent to the `ExitToNear` precompile is irrecoverably frozen at the precompile address when the NEAR-side promise fails and `error_refund` is not compiled in. The precompile address is not a regular EVM contract; no user or admin can call it to withdraw the balance. The only mechanism that moves ETH out of it (`refund_on_error`) is gated behind the disabled feature flag.

The test suite explicitly confirms this behavior:

```rust
// If the refund feature is not enabled, then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let expected_balance = Wei::new_u64(INITIAL_ETH_BALANCE - ETH_EXIT_AMOUNT);
``` [5](#0-4) 

And for ERC-20 exits:

```rust
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
``` [6](#0-5) 

### Likelihood Explanation

**High.** Any EVM user who calls `ExitToNear` with ETH value targeting a NEAR account that is not registered with the eth-connector will trigger this. NEAR FT storage registration is a prerequisite that many users are unaware of. The failure is silent from the EVM perspective (the `submit` call succeeds), making it easy to lose funds without realizing it.

### Recommendation

Ensure the `error_refund` feature is always enabled in production builds, or unconditionally populate the `refund` field in `ExitToNearPrecompileCallbackArgs` so that a recovery callback is always attached when ETH value is non-zero. The conditional compilation should be removed in favor of always-on refund logic:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    transfer_near: transfer_near_args,
};
``` [1](#0-0) 

### Proof of Concept

1. Deploy Aurora Engine **without** the `error_refund` feature.
2. Fund an EVM address with ETH on Aurora.
3. Call the `ExitToNear` precompile (`0xe9217bc70b7ed1f598ddd3199e80b093fa71124f`) with non-zero ETH value, targeting a NEAR account that has **not** registered storage with the eth-connector.
4. The EVM transaction succeeds; ETH is deducted from the caller and credited to the precompile address.
5. The NEAR `ft_transfer` promise fails (unregistered account).
6. No callback fires; no refund occurs.
7. Observe that the caller's ETH balance is permanently reduced and the ETH is now at the precompile address with no recovery path.

The test `test_exit_to_near_eth_refund` in `engine-tests/src/tests/erc20_connector.rs` demonstrates exactly this scenario under the `#[cfg(not(feature = "error_refund"))]` branch, confirming the permanent loss. [7](#0-6)
