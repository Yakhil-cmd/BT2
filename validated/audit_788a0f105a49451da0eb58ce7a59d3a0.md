### Title
Integer Division Truncation in `native_per_pubdata` Silently Zeros Pubdata Native Charge for L2 Transactions — (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary
When `pubdata_price < native_price`, the `native_per_pubdata` calculation in L2 transaction validation truncates to zero via integer floor division (`wrapping_div`). This is the direct structural analog of the VotingEscrow rounding bug: a numerator smaller than the denominator yields zero, silently disabling the native resource charge for all pubdata generated by the transaction.

---

### Finding Description

In `validation_impl.rs` at line 142, `native_per_pubdata` is computed using `wrapping_div` (floor division):

```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
``` [1](#0-0) 

When `pubdata_price < native_price` (e.g., `pubdata_price = 5`, `native_price = 10`), the result is `5 / 10 = 0`.

This is structurally inconsistent with the `native_per_gas` calculation on lines 131–138, which uses `div_ceil` to guarantee at least 1 native unit per gas when `gas_price > 0`:

```rust
u256_try_to_u64(&gas_price.div_ceil(native_price)).ok_or(TxError::Validation(
    InvalidTransaction::NativeResourcesAreTooExpensive,
))?
``` [2](#0-1) 

Both `pubdata_price` and `native_price` are operator-set block-context values, independent of each other:

```rust
let pubdata_price = system.get_pubdata_price();
let native_price = system.get_native_price();
``` [3](#0-2) 

When `native_per_pubdata = 0`, the downstream effects are:

**1. Intrinsic pubdata overhead zeroed in `create_resources_for_tx`:**

```rust
let intrinsic_pubdata_overhead = native_per_pubdata_byte.saturating_mul(intrinsic_pubdata);
// → 0 * intrinsic_pubdata = 0; no native deducted for intrinsic pubdata
let native_limit = match native_limit.checked_sub(intrinsic_pubdata_overhead) { ... };
``` [4](#0-3) 

**2. The zero value propagates into `TxContextForPreAndPostProcessing.native_per_pubdata`**, which is used throughout the transaction lifecycle to charge native resources for execution-time pubdata: [5](#0-4) 

**3. In `compute_gas_refund`, the `native_used` calculation uses `native_per_gas` (not `native_per_pubdata`) for the delta-gas adjustment**, meaning the pubdata native cost is never recovered:

```rust
let delta_gas = if native_per_gas == 0 {
    0
} else {
    (native_used / native_per_gas) as i64 - (gas_used as i64)
};
``` [6](#0-5) 

The double resource accounting model is documented to charge native resources for pubdata:

> "Then we process the transaction, charging both Ergs for EE execution and native resource for any kind of computation (EE, bootloader or system work)." [7](#0-6) 

When `native_per_pubdata = 0`, this invariant is silently broken for all pubdata.

---

### Impact Explanation

When `pubdata_price < native_price`, any L2 transaction that generates pubdata (e.g., cold storage writes) does so without consuming native resources. Since native resources model RISC-V proving cycles, the prover must prove pubdata that was never budgeted for. Concretely:

- A transaction can write to many storage slots, generating significant pubdata, while its native resource consumption reflects only computation — not pubdata proving cost.
- The native resource limit (`gas_limit * native_per_gas`) is not enforced for pubdata, allowing pubdata generation to exceed what the native budget should permit.
- This breaks the double resource accounting invariant and can cause block-level pubdata to exceed the proving budget, leading to increased proving costs not covered by user fees.

The impact class matches the external report: **resource accounting fails to charge for a critical quantity, but no direct fund loss occurs at the transaction level**.

---

### Likelihood Explanation

`pubdata_price` and `native_price` are independent operator-set block-context parameters. There is no enforced constraint that `pubdata_price >= native_price`. An operator subsidizing pubdata (setting `pubdata_price` low relative to `native_price`) triggers this silently. Once triggered, **any unprivileged L2 transaction sender** exploits it simply by submitting a transaction that writes to storage. No special access, leaked keys, or governance control is required.

---

### Recommendation

Replace `wrapping_div` with `div_ceil` for `native_per_pubdata`, consistent with `native_per_gas`:

```rust
// Before (floor division — truncates to 0 when pubdata_price < native_price):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;

// After (ceiling division — guarantees ≥1 native per pubdata byte when pubdata_price > 0):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

This mirrors the fix analogous to the VotingEscrow report's recommendation of increasing the MULTIPLIER: ensure the numerator is never silently swallowed by the denominator.

---

### Proof of Concept

1. Set block context: `native_price = 10`, `pubdata_price = 5`.
2. Observe: `native_per_pubdata = U256::from(5).wrapping_div(U256::from(10)) = 0`.
3. Submit an L2 transaction that writes to 10 new storage slots (generating ~320 bytes of pubdata).
4. Observe that `intrinsic_pubdata_overhead = 0 * intrinsic_pubdata = 0` — no native deducted for intrinsic pubdata.
5. Observe that all execution-time pubdata charges use `native_per_pubdata = 0` — no native consumed for pubdata.
6. The transaction completes without native resource exhaustion despite generating significant pubdata, violating the double resource accounting invariant.
