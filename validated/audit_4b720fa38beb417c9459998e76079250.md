### Title
Unguarded `new()` Initialization Allows Frontrunning to Seize Engine Ownership - (File: engine/src/contract_methods/admin.rs)

---

### Summary

The Aurora Engine's `new()` initialization function performs no caller authentication. Any NEAR account can call it before the legitimate deployer does, supplying an arbitrary `owner_id`. A successful frontrun grants the attacker full owner privileges over the engine, including the ability to pause the contract (temporarily freezing all user funds) or stage and deploy a malicious upgrade (enabling direct fund theft).

---

### Finding Description

`engine/src/contract_methods/admin.rs` exposes the `new()` function as the sole initialization path for the engine's `EngineState`:

```rust
// engine/src/contract_methods/admin.rs  lines 56-88
pub fn new<I: IO + Copy, E: Env>(mut io: I, env: &E) -> Result<(), ContractError> {
    if state::get_state(&io).is_ok() {
        return Err(b"ERR_ALREADY_INITIALIZED".into());
    }

    let input = io.read_input().to_vec();
    let args = NewCallArgs::deserialize(&input)...;
    ...
    let state: EngineState = args.into();   // owner_id comes from attacker-supplied args
    ...
    state::set_state(&mut io, &state)?;
    Ok(())
}
```

The only guard is the "already initialized" check on line 57. There is **no check on `env.predecessor_account_id()`** — the function is callable by any NEAR account. The `owner_id` written into `EngineState` is taken directly from the caller-supplied input bytes, not from the deployer's account.

In NEAR, contract deployment and the subsequent `new()` call are two separate transactions unless explicitly batched. The workspace deployment helper (`engine-workspace/src/lib.rs` lines 107–125) shows the deploy and `new()` call issued as separate steps, confirming the gap exists in the intended deployment flow. An attacker who observes the deploy transaction on-chain can submit a `new()` call in the next block, before the legitimate deployer does, and write their own account as `owner_id`.

The `EngineState.owner_id` field is the root of all privileged access:

```rust
// engine/src/contract_methods/mod.rs  lines 79-86
pub fn require_owner_only(
    state: &state::EngineState,
    predecessor_account_id: &AccountId,
) -> Result<(), ContractError> {
    if &state.owner_id != predecessor_account_id {
        return Err(errors::ERR_NOT_ALLOWED.into());
    }
    Ok(())
}
```

Every privileged function — `set_owner`, `pause_contract`, `resume_contract`, `stage_upgrade`, `deploy_upgrade`, `set_key_manager`, and others — gates on `require_owner_only`. Ownership of `owner_id` is therefore equivalent to full administrative control of the engine.

---

### Impact Explanation

**Impact: Critical — Permanent or temporary freezing of funds; direct theft of user funds.**

An attacker who seizes `owner_id` can:

1. **Pause the contract** via `pause_contract` (guarded by `require_owner_only`). All EVM execution and withdrawals halt. User ETH and ERC-20 balances locked in the engine become inaccessible — a temporary (or indefinite) fund freeze.
2. **Stage and deploy a malicious upgrade** via `stage_upgrade` + `deploy_upgrade`. The attacker replaces the engine's WASM with code that drains all ETH and NEP-141 token balances held by the contract — direct theft of all user funds at rest.

Both paths are reachable immediately after a successful frontrun, with no further preconditions.

---

### Likelihood Explanation

**Likelihood: Medium.**

NEAR transactions are publicly visible on-chain. A watcher monitoring for `DeployContract` actions targeting the Aurora engine account can respond in the very next block with a `new()` call. The attacker needs no special access, no tokens, and no prior relationship with the protocol — only the ability to submit a NEAR transaction. The window is narrow (one block) but the action is trivially scripted. The deployer would need to batch deploy + `new()` atomically to close the window, which the current deployment flow does not do.

---

### Recommendation

Restrict `new()` so it can only be called by the contract account itself (i.e., `env.current_account_id() == env.predecessor_account_id()`). In NEAR, only the account that owns the contract can include a `DeployContract` action in a batch, so tying initialization to a self-call enforces that deploy and init are atomic:

```rust
pub fn new<I: IO + Copy, E: Env>(mut io: I, env: &E) -> Result<(), ContractError> {
    if state::get_state(&io).is_ok() {
        return Err(b"ERR_ALREADY_INITIALIZED".into());
    }
    // Ensure only the contract account itself can initialize (deploy + init must be batched)
    if env.predecessor_account_id() != env.current_account_id() {
        return Err(b"ERR_NOT_ALLOWED".into());
    }
    ...
}
```

Alternatively, require the deployer to always batch `DeployContract` and the `new()` function call in a single NEAR batch transaction, and document this as a hard deployment requirement.

---

### Proof of Concept

1. Attacker monitors NEAR for a `DeployContract` action targeting the Aurora engine account ID.
2. In the next block, attacker submits a transaction calling `new()` on the engine account with `owner_id` set to their own account.
3. Because no state exists yet, the `state::get_state(&io).is_ok()` check returns `Err`, the guard passes, and `state::set_state` writes the attacker's account as `owner_id`.
4. The legitimate deployer's subsequent `new()` call fails with `ERR_ALREADY_INITIALIZED`.
5. Attacker calls `pause_contract()` — all user funds are frozen.
6. Attacker calls `stage_upgrade()` with malicious WASM, waits `upgrade_delay_blocks`, then calls `deploy_upgrade()` — engine code is replaced and all funds are drained.

```rust
pub fn new<I: IO + Copy, E: Env>(mut io: I, env: &E) -> Result<(), ContractError> {
    if state::get_state(&io).is_ok() {
        return Err(b"ERR_ALREADY_INITIALIZED".into());
    }

    let input = io.read_input().to_vec();
    let args = NewCallArgs::deserialize(&input).map_err(|_| errors::ERR_BORSH_DESERIALIZE)?;

    let initial_hashchain = args.initial_hashchain();
    let state: EngineState = args.into();

    if let Some(block_hashchain) = initial_hashchain {
        let block_height = env.block_height();
        let mut hashchain = Hashchain::new(
            state.chain_id,
            env.current_account_id(),
            block_height,
            block_hashchain,
        );

        hashchain.add_block_tx(
            block_height,
            function_name!(),
            &input,
            &[],
            &Bloom::default(),
        )?;
        crate::hashchain::save_hashchain(&mut io, &hashchain)?;
    }

    state::set_state(&mut io, &state)?;
    Ok(())
}
```
