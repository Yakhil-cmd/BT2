### Title
Inconsistent Threshold Lower-Bound Validation Across Protocol Entry Points Allows Signing with `threshold = 1` — (File: `src/frost/mod.rs`)

---

### Summary

The `keygen`, `reshare`, and `refresh` entry points enforce a minimum threshold of 2 via `assert_key_invariants`. The FROST `presign` function and `assert_sign_inputs` helper — which gate the EdDSA and RedJubjub signing paths — omit this lower-bound check entirely. A caller or malicious coordinator can supply `threshold = 1` to the presign/sign pipeline, causing the FROST protocol to execute with a cryptographically invalid threshold and producing corrupted, unusable signature outputs that honest parties cannot distinguish from a legitimate protocol run until external verification fails.

---

### Finding Description

`assert_key_invariants` in `src/dkg.rs`, called by every DKG entry point (`keygen`, `reshare`, `refresh`), explicitly enforces a minimum threshold of 2: [1](#0-0) 

```rust
if threshold < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
}
```

The shared FROST `presign` function in `src/frost/mod.rs` validates only the upper bound: [2](#0-1) 

```rust
if args.threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
```

`assert_sign_inputs`, the shared validation helper for EdDSA and RedJubjub signing, has the identical gap — only an upper-bound check, no lower-bound check: [3](#0-2) 

```rust
if threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
```

The `ReconstructionLowerBound` type does not enforce a minimum of 2 at construction time; the DKG test suite confirms that the raw integer `1` converts into a valid `ReconstructionLowerBound` and is only rejected by the explicit check inside `assert_key_invariants`: [4](#0-3) 

The `PresignArguments` struct carries `threshold` as a caller-supplied field with no internal invariant: [5](#0-4) 

This means a caller who successfully completes keygen (threshold ≥ 2 enforced) can then invoke `presign` with `PresignArguments { keygen_out, threshold: 1.into() }` and the library will proceed without error.

---

### Impact Explanation

**High — Corruption of sign outputs so honest parties accept unusable cryptographic outputs.**

In FROST, the threshold governs how many Lagrange coefficients are combined during signing. With `threshold = 1`, the protocol runs with a single participant's share treated as if it were the full secret, producing a signature whose `z` value is computed from incorrect Lagrange interpolation. The protocol completes without returning an error; honest participants receive an `Action::Return` with a signature value that silently fails external verification. Parties have no in-protocol signal that the output is invalid.

---

### Likelihood Explanation

**Medium.** The attacker must be either (a) a malicious coordinator who constructs `PresignArguments` with `threshold = 1` and distributes the resulting presign session to honest signers, or (b) a buggy library consumer who passes a threshold inconsistent with the one used at keygen time. No cryptographic capability is required. The inconsistency is invisible at the type level because `ReconstructionLowerBound` accepts 1, so a developer relying on the type system for safety will not be warned.

---

### Recommendation

Add the same lower-bound guard present in `assert_key_invariants` to both `presign` and `assert_sign_inputs`:

```rust
if args.threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: args.threshold.into(),
        min: 2,
    });
}
```

Alternatively, enforce the invariant inside `ReconstructionLowerBound`'s constructor so that no entry point — DKG or signing — can ever receive a threshold below 2, eliminating the need for repeated ad-hoc checks across the codebase. This mirrors the Fairswap recommendation: define the constraint once in a shared location and inherit it everywhere.

---

### Proof of Concept

```rust
// Step 1: keygen succeeds with threshold = 2 (lower-bound enforced)
let keygen_out = run_keygen(&participants, 2, &mut rng);

// Step 2: presign accepts threshold = 1 — no error returned
let args = PresignArguments {
    keygen_out: keygen_out[0].1.clone(),
    threshold: ReconstructionLowerBound::from(1usize),  // accepted silently
};
let presign_protocol = presign(&participants, me, &args, rng).unwrap();
// protocol runs to completion

// Step 3: sign proceeds with threshold = 1
// assert_sign_inputs only checks threshold <= participants.len(), passes for threshold=1
// FROST signing executes with a single Lagrange coefficient = 1
// Output signature is cryptographically invalid but no ProtocolError is raised

// Step 4: external verification fails — honest parties discover the corruption
//         only after the fact, with no in-protocol indication of the root cause
```

The root cause is the absence of a unified threshold validation interface shared across all protocol entry points — an exact structural analog to the Fairswap inconsistency, where the same conceptual constraint (`threshold ≥ 2`) is enforced in DKG entry points but silently absent from the signing entry points.
