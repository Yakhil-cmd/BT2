### Title
Unbounded Loop in `calculate_rewards` Allows Permanent Freezing of Pool Member Unclaimed Yield — (`src/pool/pool.cairo`)

---

### Summary

The `calculate_rewards` function in `pool.cairo` contains an explicitly unbounded loop over a pool member's balance trace (`pool_member_epoch_balance`). A pool member can inflate this trace by repeatedly calling `add_to_delegation_pool` or `exit_delegation_pool_intent` across epochs without claiming rewards. When `claim_rewards` is eventually called, the loop must iterate over every accumulated entry, potentially exceeding the Starknet transaction gas limit and permanently freezing the pool member's unclaimed yield.

---

### Finding Description

In `src/pool/pool.cairo`, the internal `calculate_rewards` function iterates over all entries in `pool_member_epoch_balance` from `entry_to_claim_from` up to `pool_member_trace_length`: [1](#0-0) 

The developers themselves acknowledge the risk with the comment:

> `// **Note**: The loop iterates over the balance changes in the pool member's balance trace. This loop is unbounded but unlikely to exceed gas limits.` [2](#0-1) 

Each call to `set_member_balance` (invoked by `enter_delegation_pool`, `exit_delegation_pool_intent`, and `enter_delegation_pool_from_staking_contract`) or `increase_member_balance` (invoked by `add_to_delegation_pool`) appends a new checkpoint to the trace: [3](#0-2) 

The `entry_to_claim_from` cursor stored in `pool_member_info` advances only when `claim_rewards` is successfully called: [4](#0-3) 

If a pool member accumulates many balance-change entries across epochs without claiming rewards, a single `claim_rewards` call must process all of them in one transaction, which can exceed the gas limit.

`add_to_delegation_pool` is callable by both the pool member and their `reward_address`: [5](#0-4) 

This means either the pool member or their designated reward address can inflate the trace.

---

### Impact Explanation

If the trace grows large enough that a single `claim_rewards` call exceeds the Starknet transaction gas limit, the pool member's accumulated rewards become permanently unclaimable. The `entry_to_claim_from` cursor can never advance past the bloated range, so no future call can succeed either. This constitutes **permanent freezing of unclaimed yield**, which is within the allowed impact scope.

---

### Likelihood Explanation

A pool member who actively manages their delegation (calling `add_to_delegation_pool` or `exit_delegation_pool_intent` frequently across many epochs) without regularly calling `claim_rewards` will organically accumulate trace entries. A deliberate attacker can accelerate this by calling `add_to_delegation_pool` with the minimum non-zero amount many times per epoch. Since there is no per-epoch cap on balance-change entries and no minimum amount enforced beyond `amount.is_non_zero()`, the trace can be inflated at low cost. The attack requires no privileged access — any pool member or their reward address can trigger it.

---

### Recommendation

1. **Enforce a maximum number of unclaimed balance-change entries** before allowing further `add_to_delegation_pool` or `exit_delegation_pool_intent` calls, requiring the pool member to call `claim_rewards` first to advance `entry_to_claim_from`.
2. **Alternatively**, process only a bounded number of entries per `claim_rewards` call and persist the updated `entry_to_claim_from` cursor, allowing incremental claiming across multiple transactions.
3. **At minimum**, add a hard cap on the number of entries that can be inserted into `pool_member_epoch_balance` per pool member per epoch.

---

### Proof of Concept

1. Pool member Alice calls `enter_delegation_pool` with a small amount, establishing her trace.
2. Alice calls `add_to_delegation_pool(pool_member: alice, amount: 1)` N times across N epochs, never calling `claim_rewards`. Each call appends a new entry to `pool_member_epoch_balance[alice]` via `increase_member_balance` → `set_member_balance`.
3. After N epochs, `pool_member_trace_length` for Alice equals N.
4. Alice (or her reward address) calls `claim_rewards(pool_member: alice)`.
5. `calculate_rewards` enters the `while entry_to_claim_from < pool_member_trace_length` loop and must execute N iterations, each performing storage reads (`pool_member_trace.at(entry_to_claim_from)`) and arithmetic.
6. For sufficiently large N, the transaction reverts with an out-of-gas error.
7. Because `entry_to_claim_from` is only written on success, it remains at 0. Every subsequent `claim_rewards` call also reverts. Alice's accumulated rewards are permanently frozen. [6](#0-5)

```text
        fn calculate_rewards(
            self: @ContractState,
            pool_member: ContractAddress,
            from_checkpoint: PoolMemberCheckpoint,
            until_checkpoint: PoolMemberCheckpoint,
            mut entry_to_claim_from: VecIndex,
        ) -> (Amount, VecIndex) {
            let pool_member_trace = self.pool_member_epoch_balance.entry(pool_member);
            // Note: `until_epoch` is the current epoch.
            let until_epoch = until_checkpoint.epoch();

            let mut rewards = 0;

            let pool_member_trace_length = pool_member_trace.length();

            let mut from_sigma = self.find_sigma(from_checkpoint, curr_epoch: until_epoch);
            let mut from_balance = from_checkpoint.balance();

            let base_value = self.staking_rewards_base_value.read();

            // **Note**: The loop iterates over the balance changes in the pool member's balance
            // trace. This loop is unbounded but unlikely to exceed gas limits.
            while entry_to_claim_from < pool_member_trace_length {
                let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
                // If the balance change is after `until_epoch` (and therefore does not affect
                // the current reward computation), exit the loop.
                if pool_member_checkpoint.epoch() >= until_epoch {
                    break;
                }

                // Compute rewards from (inclusive) the previous balance change (or from
                // `from_checkpoint`) to (exclusive) the current entry.
                let to_sigma = self.find_sigma(pool_member_checkpoint, curr_epoch: until_epoch);
                rewards +=
                    compute_rewards_rounded_down(
                        amount: from_balance, interest: to_sigma - from_sigma, :base_value,
                    );
                from_sigma = to_sigma;
                from_balance = pool_member_checkpoint.balance();
                entry_to_claim_from += 1;
            }

            // Compute the remaining rewards from (inclusive) the last visited balance change in
            // `pool_member_trace` (or from `from_checkpoint`) to (exclusive) `until_checkpoint`.
            let to_sigma = self.find_sigma(until_checkpoint, curr_epoch: until_epoch);
            rewards +=
                compute_rewards_rounded_down(
                    amount: from_balance, interest: to_sigma - from_sigma, :base_value,
                );

            (rewards, entry_to_claim_from)
        }
```
