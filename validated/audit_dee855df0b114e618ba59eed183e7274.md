### Title
COA Ownership Proof Cross-Address Replay via Unbound `signedData` - (File: `fvm/evm/stdlib/contract.cdc`)

### Summary
`EVM.validateCOAOwnershipProof` verifies signatures over `signedData` but does not enforce that `signedData` encodes the target `evmAddress` or `path`. A proof produced by a Cadence account owner for one COA can be replayed against a different COA owned by the same Cadence account, because the signature commitment does not cover the COA-identifying fields.

### Finding Description
`validateCOAOwnershipProof` accepts six parameters: `address`, `path`, `signedData`, `keyIndices`, `signatures`, and `evmAddress`. The cryptographic verification step verifies the `signatures` only over `signedData`:

```cadence
let isValid = keyList.verify(
    signatureSet: signatureSet,
    signedData: signedData,
    domainSeparationTag: "FLOW-V0.0-user"
)
```

Neither `evmAddress` nor `path` is included in the signed message. After signature verification passes, the function borrows the COA capability at the caller-supplied `path` and checks that its EVM address matches the caller-supplied `evmAddress`. Both `path` and `evmAddress` are fully attacker-controlled inputs that are never committed to by the signer.

The code itself acknowledges this:

> "this function does not enforce that `signedData` includes `evmAddress`. In principle, a signature produced for one purpose could be replayed here against a different COA owned by the same Cadence account." [1](#0-0) [2](#0-1) [3](#0-2) 

The `COAOwnershipProof` struct encodes `CapabilityPath` as a field, but the `Signatures` field contains signatures over `signedData` only — not over the proof struct itself. [4](#0-3) 

### Impact Explanation
An attacker who observes a valid `COAOwnershipProof` that Alice produced for COA_A can replay it against COA_B (a different COA stored at a different path on the same Cadence account) by substituting `path=/public/coaB` and `evmAddress=COA_B_address`. The signature check passes because the signatures are over `signedData` which is unchanged. The COA address check passes because the attacker pointed to the correct COA for the substituted path.

Any Cadence contract or off-chain authentication flow that calls `EVM.validateCOAOwnershipProof` directly and grants access to COA-controlled EVM assets based on the result is vulnerable to this replay. The attacker gains unauthorized proof of ownership of COA_B, potentially enabling unauthorized withdrawal of EVM assets held by COA_B.

The EVM precompile path (`verifyCOAOwnershipProof`) is **not** vulnerable because it always passes `address(this)` as the EVM address:

```solidity
cadenceArch.staticcall(abi.encodeWithSignature(
    "verifyCOAOwnershipProof(address,bytes32,bytes)",
    address(this), _hash, _sig));
``` [5](#0-4) 

The vulnerable surface is the direct Cadence call path to `EVM.validateCOAOwnershipProof`, which is a public `access(all)` function callable by any unprivileged transaction or script.

### Likelihood Explanation
**Medium.** Requires the victim to have created two or more COAs on the same Cadence account (currently uncommon but not impossible, and increasingly likely as the ecosystem grows), and requires the attacker to have observed a valid proof the victim produced for one of those COAs. No privileged access is required. The attacker entry path is a standard Cadence transaction or script.

### Recommendation
Enforce that `signedData` commits to the target `evmAddress` (and optionally `path`) inside `validateCOAOwnershipProof`, or document and enforce this as a hard precondition by reverting (rather than returning `isValid: false`) when `signedData` does not encode `evmAddress`. Alternatively, include `evmAddress` in the signed message at the protocol level rather than relying on callers to do so.

### Proof of Concept
1. Alice owns a Cadence account with two COAs: COA_A at `/public/coaA` (EVM address `0xAAAA`) and COA_B at `/public/coaB` (EVM address `0xBBBB`).
2. Alice signs `signedData = "authenticate"` with her account key to prove ownership of COA_A. She produces `proof_A = {address: Alice, path: "coaA", keyIndices: [0], signatures: [sig]}`.
3. Bob (attacker) observes `proof_A` on-chain or off-chain.
4. Bob calls `EVM.validateCOAOwnershipProof(address: Alice, path: /public/coaB, signedData: "authenticate", keyIndices: [0], signatures: [sig], evmAddress: 0xBBBB)`.
5. Step 1 (signature check): `keyList.verify(signatureSet, signedData: "authenticate", ...)` passes — the signature is valid over `"authenticate"` regardless of which COA is targeted.
6. Step 2 (COA address check): `acc.capabilities.borrow<&EVM.CadenceOwnedAccount>(/public/coaB)` returns COA_B whose address is `0xBBBB`, matching the attacker-supplied `evmAddress`. Check passes.
7. `validateCOAOwnershipProof` returns `isValid: true` for COA_B using Alice's signature that was only intended for COA_A.
8. Any Cadence contract that grants access to COA_B's EVM assets based on this result is now exploited. [6](#0-5) [7](#0-6)
