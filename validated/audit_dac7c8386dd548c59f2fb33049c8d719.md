### Title
Bank Precompile `transfer` Allows Any Calling Contract to Drain Arbitrary Holders' Native Token Balances Without Authorization - (File: x/cronos/keeper/precompiles/bank.go)

### Summary
The bank precompile's `transfer` method accepts the `sender` address as a caller-supplied ABI argument and performs no authorization check that the calling contract is the `sender` or that the `sender` has approved the transfer. Any deployed contract can invoke `transfer(victim, attacker, amount)` to drain `victim`'s entire balance of the `evm/<callerContract>` native token without the victim's consent.

### Finding Description
The `BankContract.Run` function in `x/cronos/keeper/precompiles/bank.go` handles the `transfer(address sender, address recipient, uint256 amount)` method as follows:

```go
sender := args[0].(common.Address)   // fully attacker-controlled
recipient := args[1].(common.Address)
amount := args[2].(*big.Int)
...
from := sdk.AccAddress(sender.Bytes())
to := sdk.AccAddress(recipient.Bytes())
...
denom := EVMDenom(contract.Caller())  // evm/<callerContractAddress>
amt := sdk.NewCoin(denom, sdkmath.NewIntFromBigInt(amount))
...
bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt))
```

The denom is correctly scoped to `evm/<contract.Caller()>`, but the `from` address for the bank `SendCoins` call is taken verbatim from the ABI-decoded `sender` argument. There is no check that:
- `sender == contract.Caller()`, or
- `sender` has granted an allowance to `contract.Caller()`, or
- any other form of authorization from the `sender`.

The only guards present are a `readonly` check, an `amount > 0` check, a blocked-address check on the recipient, and an `IsSendEnabledCoins` check — none of which restrict who the `sender` can be.

An attacker deploys a contract (permissionless on Cronos), uses `mint(user, amount)` via the same bank precompile to distribute `evm/<attackerContract>` tokens to users (e.g., as a yield token, wrapped asset, or any incentive), and then at any time calls `transfer(victim, attacker, victimBalance)` to drain every holder's balance with no approval required.

### Impact Explanation
This is a **Critical** unauthorized transfer of precompile-controlled native assets. The `evm/<contract>` denom is a first-class Cosmos SDK bank module coin. `SendCoins` moves real balances tracked in the bank module state. Any holder of `evm/<attackerContract>` tokens can be fully drained at will by the contract deployer or by any contract that calls the bank precompile's `transfer` method with an arbitrary `sender`. This matches the criterion: *Unauthorized transfer of precompile-controlled assets*.

### Likelihood Explanation
The attack path requires only deploying a contract — a permissionless operation available to any Cronos user. No admin key, governance vote, or validator compromise is needed. The only prerequisite is that users hold the attacker's `evm/<contract>` token, which is achievable by any contract that calls `mint` on the bank precompile. The vulnerability is reachable through a standard EVM transaction.

### Recommendation
Add an authorization check in the `TransferMethodName` case that enforces `sender == contract.Caller()`:

```go
if sender != contract.Caller() {
    return nil, errors.New("transfer: sender must be the calling contract")
}
```

This mirrors the correct design of the `mint` and `burn` cases, where the denom is derived from `contract.Caller()` and the operation is implicitly authorized by the calling contract acting on its own token. The `transfer` method should only allow the calling contract to move tokens from its own address, not from arbitrary third-party addresses.

### Proof of Concept

1. Attacker deploys `MaliciousToken` contract at address `0xATK`.
2. `MaliciousToken` calls `bankPrecompile.mint(alice, 1_000_000)` — Alice now holds `1_000_000 evm/0xATK` in the Cosmos bank module.
3. Alice uses the token normally (e.g., in DeFi protocols that accept `evm/0xATK`).
4. Attacker calls `bankPrecompile.transfer(alice, attacker, 1_000_000)` from `MaliciousToken`.
5. The bank precompile executes `bankKeeper.SendCoins(ctx, alice, attacker, 1_000_000 evm/0xATK)` with no approval from Alice.
6. Alice's entire balance is transferred to the attacker.
