### Title
Gateway `skip_stateful_validations` Admits Unsigned Invoke Transactions for Undeployed Accounts Without Signature Verification - (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

`skip_stateful_validations` in the Apollo Gateway stateful validator unconditionally skips the `__validate__` entry-point call for any invoke transaction with `nonce == 1` when the on-chain account nonce is `0` and *any* transaction for that sender exists in the mempool. Because the check is purely on the sender address and nonce value — with no cryptographic binding to the account owner — an unprivileged attacker can inject an unsigned invoke transaction on behalf of a victim who has a pending `deploy_account` transaction, bypassing the only signature-verification gate at the gateway layer.

### Finding Description

`extract_state_nonce_and_run_validations` fetches the on-chain nonce for `executable_tx.contract_address()` and then calls `run_pre_validation_checks`, which in turn calls `skip_stateful_validations`: [1](#0-0) 

`skip_stateful_validations` returns `true` (skip `__validate__`) when: [2](#0-1) 

The three conditions are:
1. The transaction is an `Invoke`.
2. `tx.nonce() == Nonce(Felt::ONE)`.
3. `account_nonce == Nonce(Felt::ZERO)`.
4. `account_tx_in_pool_or_recent_block(tx.sender_address())` returns `true`.

None of these conditions verify that the submitter of the invoke transaction is the same entity that submitted the `deploy_account` transaction, nor that the invoke transaction carries a signature that will pass the account's `__validate__` entry point. The check is purely positional (nonce value) and observational (mempool membership).

The `sender_address` field of an `RpcInvokeTransactionV3` is attacker-controlled: [3](#0-2) 

The stateless validator only checks that `sender_address` is in the valid address range; it does not verify the signature against the account.

### Impact Explanation

An attacker who observes a victim's `deploy_account` transaction in the mempool can submit an invoke transaction with:
- `sender_address = victim_address`
- `nonce = 1`
- Arbitrary calldata
- An invalid or empty signature

The gateway will:
1. Fetch `account_nonce = 0` for `victim_address` (account not yet deployed).
2. Find the victim's `deploy_account` tx via `account_tx_in_pool_or_recent_block`.
3. Return `skip_validate = true`, bypassing the `__validate__` entry-point call entirely.
4. Admit the unsigned transaction to the mempool. [4](#0-3) 

Once in the mempool, the attacker's transaction occupies the victim's `nonce = 1` slot. If the mempool replaces or rejects the victim's legitimate `nonce = 1` invoke (submitted later or with a lower fee), the victim's transaction is displaced. When the batcher executes the attacker's transaction after the `deploy_account` is processed, `__validate__` is called with `validate = true` (sequencing default), fails on the invalid signature, and the transaction is dropped — but the victim's mempool slot has already been consumed and their legitimate transaction may have been evicted.

The broken invariant is: **every transaction admitted to the mempool must have passed signature verification for the account it claims to originate from.** The `skip_stateful_validations` path breaks this invariant for the specific `(nonce=1, account_nonce=0, mempool_hit)` case, allowing any observer to inject unsigned transactions on behalf of any account in the process of being deployed.

### Likelihood Explanation

The attack requires only:
1. Observing the public mempool for `deploy_account` transactions (trivially feasible).
2. Submitting an invoke transaction with `sender_address = victim`, `nonce = 1`, and any resource bounds that pass the stateless and stateful resource-bound checks.

No privileged access, no special contract, and no on-chain funds are required. The window is the time between the victim's `deploy_account` entering the mempool and the victim's legitimate `nonce = 1` invoke being submitted.

### Recommendation

The `skip_stateful_validations` check must be restricted to transactions that were submitted by the same entity as the corresponding `deploy_account`. The minimal fix is to require that the mempool entry for the sender is specifically a `deploy_account` transaction (not just any transaction), and additionally to verify that the invoke transaction's signature is structurally consistent with the class hash declared in that `deploy_account` transaction. Alternatively, the skip should only apply when the invoke transaction was submitted in the same atomic request as the `deploy_account` transaction (bundled submission), so the gateway can cryptographically bind the two. [5](#0-4) 

### Proof of Concept

1. Victim submits `RpcDeployAccountTransaction::V3` with `class_hash=C`, `salt=S`, `constructor_calldata=D`. The computed `contract_address = A`. The gateway admits it; mempool now contains a `deploy_account` for `A`.

2. Attacker calls the gateway's `add_tx` endpoint with:
   ```
   RpcInvokeTransaction::V3 {
       sender_address: A,   // victim's not-yet-deployed address
       nonce: 1,
       calldata: [<arbitrary>],
       signature: [],       // empty / invalid
       resource_bounds: <valid non-zero bounds>,
       ...
   }
   ```

3. Gateway stateless validation passes (`sender_address` is in range, signature length ≤ max).

4. Gateway stateful validation: `get_nonce(A) = 0`, `validate_nonce` passes (`0 ≤ 1 ≤ max_gap`), `skip_stateful_validations` returns `true` because `nonce==1 && account_nonce==0 && account_tx_in_pool_or_recent_block(A)==true`.

5. `run_validate_entry_point` is skipped. The attacker's transaction is admitted to the mempool at `(A, nonce=1)`.

6. Victim later submits their legitimate `nonce=1` invoke. The mempool either rejects it (slot taken) or replaces the attacker's tx only if the victim pays a higher fee — giving the attacker a fee-based griefing lever.

7. When the batcher processes the block: `deploy_account` for `A` executes first (nonce 0→1). The attacker's invoke is then executed; `__validate__` is called, fails on the empty signature, and the transaction is dropped. The victim's legitimate transaction has been evicted from the mempool.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
