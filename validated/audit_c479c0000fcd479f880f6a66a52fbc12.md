Audit Report

## Title
Unprivileged Caller Can Force Any Principal's SNS Swap Participation via `refresh_buyer_tokens` - (File: rs/sns/swap/canister/canister.rs)

## Summary
The `refresh_buyer_tokens` update method accepts a caller-supplied `buyer` principal without verifying that the caller is authorized to act on behalf of that principal. Any unprivileged ingress sender can register another user's ICP into the swap, creating a `BuyerState` entry that blocks the victim from reclaiming their ICP via `error_refund_icp` until after the swap finalizes and sweeps. If the swap commits, the victim is forced to receive SNS tokens in exchange for ICP they did not choose to commit.

## Finding Description
In `rs/sns/swap/canister/canister.rs` (L128–143), when `arg.buyer` is non-empty the caller-supplied string is parsed and used verbatim as the effective buyer principal with no check that `caller_principal_id() == p`:

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    PrincipalId::from_str(&arg.buyer).unwrap()   // ← attacker-controlled
};
``` [1](#0-0) 

This principal is passed directly to `refresh_buyer_token_e8s`, which reads the ICP ledger balance of `swap_subaccount(buyer)` and writes a `BuyerState` entry keyed by that principal: [2](#0-1) 

Once the `BuyerState` entry exists with `transfer_success_timestamp_seconds == 0`, `error_refund_icp` returns a precondition error and refuses to transfer funds back:

```rust
if let Some(buyer_state) = self.buyers.get(&source_principal_id.to_string()) {
    if let Some(transfer) = &buyer_state.icp
        && transfer.transfer_success_timestamp_seconds == 0
    {
        return ErrorRefundIcpResponse::new_precondition_error(format!(
            "ICP cannot be refunded as principal {} has {} ICP (e8s) in escrow", ...
        ));
    }
``` [3](#0-2) 

The `confirmation_text` guard does not mitigate this: it is stored in `Init` and returned verbatim by the public `get_init` query, so any attacker can read it before crafting the call. [4](#0-3) [5](#0-4) 

## Impact Explanation
This matches the **High** bounty impact: *Unauthorized access to ledger/canister-controlled funds where exploitation requires meaningful per-target work or other constraints.* A victim who transferred ICP to `swap_subaccount(victim)` but had not yet confirmed participation loses the ability to exit via `error_refund_icp`. If the swap commits, the victim is forced to receive SNS tokens in exchange for ICP they did not choose to commit — a concrete, irreversible loss of user choice over their funds. The attacker spends only the cycles cost of one update call per victim.

## Likelihood Explanation
The subaccount scheme is deterministic and public (`principal_to_subaccount(buyer)`). An attacker can monitor the ICP ledger for transfers whose destination matches any `swap_subaccount(P)` pattern, extract `P`, read `confirmation_text` from `get_init`, and immediately call `refresh_buyer_tokens(buyer = P)`. No privileged access, key material, or social engineering is required. The attack is fully automatable from an ordinary ingress sender and can be applied to every address observed sending ICP to the swap canister's subaccounts. [6](#0-5) 

## Recommendation
In `refresh_buyer_tokens`, when `arg.buyer` is non-empty, assert `caller_principal_id() == p` before proceeding. If third-party helpers (e.g., NNS dapp) legitimately need to call on behalf of others, maintain an explicit allowlist of trusted callers, analogous to `authorize_caller_to_call_notify_create_canister_on_behalf_of_creator` in the CMC. This pattern already exists in the codebase and should be applied here. [7](#0-6) 

## Proof of Concept
1. Alice transfers 100 ICP to `AccountIdentifier(swap_canister, subaccount(Alice))` on the ICP ledger, intending to participate but waiting to confirm swap terms.
2. Eve monitors the ICP ledger, observes the transfer, extracts Alice's principal, and reads `confirmation_text` via `get_init`.
3. Eve calls `refresh_buyer_tokens({ buyer: "<Alice's principal>", confirmation_text: <text from get_init> })` from any identity.
4. `refresh_buyer_token_e8s` reads Alice's 100 ICP balance and writes `buyers["Alice"] = BuyerState { icp: 100 ICP, transfer_success_timestamp_seconds: 0 }`.
5. Alice calls `error_refund_icp` — it returns `"ICP cannot be refunded as principal Alice has 100 ICP (e8s) in escrow"`.
6. If the swap commits, Alice's ICP is swept to SNS governance and she receives SNS tokens she did not choose to acquire; she cannot recover her ICP.

A deterministic unit test can reproduce this by calling `refresh_buyer_token_e8s` with a different `buyer` than the mock caller, then asserting that `error_refund_icp` returns a `Precondition` error for that buyer while the subaccount still holds funds. [8](#0-7) [3](#0-2)

fn authorize_caller_to_call_notify_create_canister_on_behalf_of_creator(
    caller: PrincipalId,
    creator: PrincipalId,
) -> Result<(), NotifyError> {
    if caller == creator {
        return Ok(());
    }

    // This is a hack to enable testing (related features) of nns-dapp. In
    // tests, the nns-dapp backend canister happens to use ID of the production
    // ICP ledger archive 1 canister. Ideally, the test nns-dapp backend
    // canister would have the same ID as the production nns-dapp backend
    // canister. This difference should probably be considered a bug. This hack
    // can be removed after that bug is fixed.
    const TEST_NNS_DAPP_BACKEND_CANISTER_ID: CanisterId = ICP_LEDGER_ARCHIVE_1_CANISTER_ID;
    lazy_static! {
        static ref ALLOWED_CALLERS: [PrincipalId; 2] = [
            PrincipalId::from(*NNS_DAPP_BACKEND_CANISTER_ID),
            PrincipalId::from(TEST_NNS_DAPP_BACKEND_CANISTER_ID),
        ];
    }

    if ALLOWED_CALLERS.contains(&caller) {
        return Ok(());
    }

    // Other is used, because adding a Unauthorized variant to NotifyError would
    // confuse old clients.
    let err = NotifyError::Other {
        error_code: NotifyErrorCode::Unauthorized as u64,
        error_message: format!(
            "{caller} is not authorized to call notify_create_canister on behalf \
             of {creator}. (Do not retry, because the same result will occur.)",
        ),
    };

    Err(err)
}
