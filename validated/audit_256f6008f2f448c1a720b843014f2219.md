### Title
Blocklist Destination Check Bypassed via Internal EVM Contract Calls — (File: `app/proposal.go`, `app/block_address.go`)

### Summary
Cronos enforces a blocklist that prevents blocked addresses from being the direct destination of EVM transactions. However, both the mempool ante handler and the proposal-level validator only inspect the outer `ethTx.To()` field. Internal EVM calls made during contract execution are never checked, allowing an unprivileged attacker to route token transfers to a blocked address through an intermediary contract, fully bypassing the destination restriction.

### Finding Description
The destination check is applied in two places:

**`app/block_address.go` — `BlockAddressesDecorator.AnteHandle`** (mempool, CheckTx only):
```go
if ethTx.To() != nil {
    if _, ok := bad.blockedMap[sdk.AccAddress(ethTx.To().Bytes()).String()]; ok {
        return ctx, fmt.Errorf("destination address is blocked: %s", ...)
    }
}
``` [1](#0-0) 

**`app/proposal.go` — `ProposalHandler.ValidateTransaction`** (PrepareProposal / ProcessProposal):
```go
if ethTx.To() != nil {
    encoded, err := h.addressCodec.BytesToString(ethTx.To().Bytes())
    if _, ok := h.blocklist[encoded]; ok {
        return fmt.Errorf("destination address is blocked: %s", encoded)
    }
}
``` [2](#0-1) 

Both checks are limited to the **top-level transaction destination**. Neither the ante handler nor the proposal validator inspects internal EVM calls (CALL/DELEGATECALL opcodes) that occur during contract execution. The `ProcessProposalHandler` that enforces this at block-inclusion time has the same gap: [3](#0-2) 

The destination check was deliberately added as a security feature (changelog entry `#1922`: *"Feat: check destination address in the blocklist"*), confirming the intent is to prevent tokens from reaching blocked addresses. The `ModuleCRC21.send_to_ibc` and `send_to_evm_chain` functions use `transferFrom` internally, and any ERC-20/CRC-20/CRC-21 `transfer` call made from within a contract during execution is invisible to the blocklist enforcement layer. [4](#0-3) 

### Impact Explanation
**High — Bypass of block-list authorization check.**

An unprivileged attacker can transfer CRO, CRC-20, or CRC-21 tokens to a blocked address by routing the transfer through an intermediary contract. The outer transaction's `To` field points to the attacker-controlled contract (not blocked), so both the ante handler and `ProcessProposalHandler` accept the transaction. The internal `transfer(blockedAddr, amount)` call executes at the EVM level without any blocklist check, crediting the blocked address with tokens. This directly defeats the purpose of the destination restriction and constitutes a bypass of the block-list authorization check, which is explicitly listed as a High impact in the allowed scope.

### Likelihood Explanation
Any EVM user can deploy a contract and call it — no special role, key, or privilege is required. The attacker only needs to know the blocked address and hold (or acquire) the token to transfer. The attack is trivially repeatable and requires no coordination with validators.

### Recommendation
Enforce the blocklist destination check at EVM execution time, not only at the transaction-envelope level. Options include:

1. **EVM state-DB hook**: Override the EVM's `Transfer` hook (or equivalent in the Ethermint state DB) to reject any value transfer or token-transfer call whose recipient is in the blocklist.
2. **Precompile-level guard**: In the bank precompile (`x/cronos/keeper/precompiles/bank.go`), the `checkBlockedAddr` function already exists for SDK-level blocked addresses; extend it to also consult the validator blocklist.
3. **EVM log / post-execution hook**: After each EVM transaction, scan emitted `Transfer` events for blocked recipients and revert the transaction if any are found.

### Proof of Concept

```solidity
// Attacker deploys this contract
interface IERC20 { function transfer(address to, uint256 amount) external returns (bool); }

contract BlocklistBypass {
    function bypass(address token, address blocked, uint256 amount) external {
        // outer tx.To = address(this) — NOT in the blocklist
        // internal call to blocked address — never checked
        IERC20(token).transfer(blocked, amount);
    }
}
```

1. Validator blocklist contains `0xBLOCKED`.
2. Attacker deploys `BlocklistBypass` at `0xCONTRACT` (not blocked).
3. Attacker sends EVM tx: `To = 0xCONTRACT`, calls `bypass(crc20Token, 0xBLOCKED, amount)`.
4. `ValidateTransaction` checks `ethTx.To() == 0xCONTRACT` → not blocked → transaction included in block. [5](#0-4) 
5. EVM executes the internal `transfer(0xBLOCKED, amount)` call — no blocklist check at this layer.
6. `0xBLOCKED` receives tokens despite being on the blocklist, bypassing the destination restriction added in `#1922`. [6](#0-5)

func (h *ProposalHandler) ProcessProposalHandler() sdk.ProcessProposalHandler {
	return func(ctx sdk.Context, req *abci.RequestProcessProposal) (*abci.ResponseProcessProposal, error) {
		if len(h.blocklist) == 0 {
			// fast path, accept all txs
			return &abci.ResponseProcessProposal{Status: abci.ResponseProcessProposal_ACCEPT}, nil
		}

		for _, txBz := range req.Txs {
			if err := h.ValidateTransaction(nil, txBz); err != nil {
				return &abci.ResponseProcessProposal{Status: abci.ResponseProcessProposal_REJECT}, nil
			}
		}

		return &abci.ResponseProcessProposal{Status: abci.ResponseProcessProposal_ACCEPT}, nil
	}
}
