### Title
Pool admin can immediately raise fees without timelock, enabling frontrunning of user swaps — (File: metric-core/contracts/MetricOmmPoolFactory.sol)

### Summary
`setPoolAdminFees` and `setPoolBinAdditionalFees` allow the pool admin to change fees with immediate effect and no timelock. A malicious pool admin can frontrun a pending user swap by raising fees to the protocol-allowed maximum, causing the user to pay significantly more than expected. The protocol explicitly timelocks price-provider changes but provides no equivalent protection for fee changes.

### Finding Description

`MetricOmmPoolFactory.setPoolAdminFees` takes effect in the same transaction with no delay: [1](#0-0) 

It immediately calls `setPoolFees` on the pool: [2](#0-1) 

`setPoolBinAdditionalFees` is equally immediate — no cap check, no delay: [3](#0-2) 

Both propagate directly to the pool's live state: [4](#0-3) [5](#0-4) 

By contrast, price-provider changes are protected by a two-step propose/execute flow with a configurable `priceProviderTimelock[pool]`: [6](#0-5) [7](#0-6) 

No equivalent protection exists for fee changes.

The pool's `swap` function accepts only a `priceLimitX64` (a marginal price limit), which does not protect against fee increases — there is no `minAmountOut` or `maxAmountIn` guard at the pool level: [8](#0-7) 

The hard caps allow the pool admin to raise `adminSpreadFeeE6` up to 200,000 (20%) and `adminNotionalFeeE8` up to 1,000,000 (1%): [9](#0-8) 

### Impact Explanation

A malicious pool admin can:
1. Observe a pending swap in the mempool.
2. Frontrun it with `setPoolAdminFees(pool, 200_000, 1_000_000)` — raising spread fee to 20% and notional fee to 1% in the same block.
3. The user's swap executes with inflated fees; the user loses up to ~21% of swap value relative to the advertised fee.
4. The admin lowers fees back and calls `collectPoolFees(pool)` to extract the excess.

The `collectFees` call inside `setPoolAdminFees` settles accrued fees at the *old* rate before the raise, so the excess from the frontrunned swap is cleanly captured in the subsequent collection. [10](#0-9) 

### Likelihood Explanation

Requires a malicious pool admin. Pool admins are semi-trusted actors expected to be constrained by both caps *and* timelocks. The protocol's own design acknowledges this risk for price-provider changes (explicit timelock) but omits equivalent protection for fee changes, creating an exploitable inconsistency. On-chain mempool visibility makes frontrunning straightforward on non-private mempools.

### Recommendation

Introduce a two-step propose/execute flow for `setPoolAdminFees` and `setPoolBinAdditionalFees`, mirroring the existing `proposePoolPriceProvider` / `executePoolPriceProviderUpdate` pattern. A configurable `feeChangeTimelock` per pool (defaulting to a minimum of 1–2 blocks or a fixed duration) would prevent same-block frontrunning while preserving the admin's ability to respond to market conditions.

### Proof of Concept

```
// Setup: pool deployed with adminSpreadFeeE6 = 0
// User submits: pool.swap(recipient, true, 1000e18, priceLimitX64, ...)

// Pool admin frontruns (higher gas):
factory.setPoolAdminFees(pool, 200_000, 0);  // raise spread fee to 20%

// User's swap executes with 20% spread fee instead of 0%
// User receives ~20% less token1 than expected

// Pool admin restores and collects:
factory.setPoolAdminFees(pool, 0, 0);        // lower fees back
factory.collectPoolFees(pool);               // extract excess fees
```
