### Title
`EVM.validateCOAOwnershipProof` Does Not Bind `signedData` to `evmAddress`, Enabling Cross-COA Signature Replay - (File: `fvm/evm/stdlib/contract.cdc`)

### Summary

`EVM.validateCOAOwnershipProof` validates that a set of signatures over `signedData` are valid for a given Cadence account, then separately checks that the COA at the provided `path` matches `evmAddress`. Because `signedData` is never required to encode `evmAddress`, a signature produced to prove ownership of one COA can be replayed to prove ownership of a different COA owned by the same Cadence account.

### Finding Description

`EVM.validateCOAOwnershipProof` in `fvm/evm/stdlib/contract.cdc` accepts six parameters: `address`, `path`, `signedData`, `keyIndices`, `signatures`, and `evmAddress`. The function performs two independent checks:

1. Verifies that `signatures` are valid over `signedData` using the account's keys at `keyIndices`.
2. Borrows the COA capability at `path` and checks that its EVM address matches `evmAddress`.

The code itself documents the flaw at lines 1001–1009:

> "Note: this function does not enforce that `signedData` includes `evmAddress`. In principle, a signature produced for one purpose could be replayed here against a different COA owned by the same Cadence account." [1](#0-0) 

The signature check and the EVM address check are entirely decoupled. The `keyList.verify` call at line 1082 only verifies that the signatures are valid over `signedData`; it has no knowledge of `evmAddress`. [2](#0-1) 

The EVM address check at lines 1095–1105 only verifies that the COA at `path` matches `evmAddress`; it has no knowledge of what was signed. [3](#0-2) 

Because `EVM.validateCOAOwnershipProof` is `access(all)`, any Cadence script or transaction can call it with arbitrary parameters, including a `path` and `evmAddress` that differ from what the signer intended. [4](#0-3) 

The Go-side precompile wires this function into the EVM environment via `coaOwnershipProofValidator`, which is reachable from any EVM transaction through the Cadence Arch precompile's `verifyCOAOwnershipProof` selector. [5](#0-4) 

The `COAOwnershipProofInContext` struct shows that `SignedData` and `EVMAddress` are separate fields with no enforced relationship between them. [6](#0-5) 

### Impact Explanation

If a Cadence account owns two COAs (e.g., COA_A at `/public/coa_a` and COA_B at `/public/coa_b`), an attacker who has observed a valid signature from that account over any `signedData` (e.g., from a prior on-chain EVM transaction) can call `EVM.validateCOAOwnershipProof` with:
- `address` = the victim account
- `path` = `/public/coa_b`
- `signedData` = the previously observed signed bytes
- `signatures` = the previously observed signatures
- `evmAddress` = COA_B's address

The function returns `isValid: true` for COA_B even though the signature was never intended to prove ownership of COA_B. Any EVM contract or Cadence contract that relies on `validateCOAOwnershipProof` / `verifyCOAOwnershipProof` for authorization (e.g., ERC-1271 `isValidSignature` as implemented in `coa.sol`) can be bypassed, potentially enabling unauthorized asset transfers or contract interactions on behalf of the victim's COA_B. [7](#0-6) 

### Likelihood Explanation

The code comment explicitly acknowledges the issue and notes it is "low-risk" because Flow wallets historically create at most one COA per account. However:
- The protocol places no on-chain restriction on creating multiple COAs per account.
- As the EVM ecosystem on Flow matures, multi-COA accounts will become more common.
- The attacker-controlled entry path (a Cadence script calling `EVM.validateCOAOwnershipProof` directly, or an EVM transaction calling the Cadence Arch precompile) requires no special privileges.
- The attacker only needs a previously observed valid signature from the victim account, which is available on-chain from any prior EVM interaction.

### Recommendation

Enforce that `signedData` encodes `evmAddress` before verifying signatures. One approach is to require callers to sign `keccak256(abi.encode(evmAddress, applicationData))` and verify this inside `validateCOAOwnershipProof` by reconstructing the expected signed payload from `evmAddress` and rejecting proofs where the signed data does not commit to the claimed EVM address.

### Proof of Concept

1. Account A creates two COAs: COA_A at `/public/coa_a` and COA_B at `/public/coa_b`.
2. Account A signs `signedData = someHash` to prove ownership of COA_A for EVM contract X. This signature appears on-chain.
3. An attacker observes the signature and constructs a `COAOwnershipProof` with `CapabilityPath = "coa_b"` and the same `signedData` and `signatures`.
4. The attacker calls (from a Cadence script or EVM transaction via the Cadence Arch precompile):
   ```
   EVM.validateCOAOwnershipProof(
     address: A,
     path: /public/coa_b,
     signedData: someHash,
     keyIndices: [0],
     signatures: [observedSig],
     evmAddress: COA_B
   )
   ```
5. The function verifies `observedSig` over `someHash` — valid. It then borrows the COA at `/public/coa_b` and confirms its address is COA_B — valid. It returns `isValid: true`.
6. Any EVM contract using `verifyCOAOwnershipProof` for authorization now accepts the attacker's forged proof as a valid ownership claim for COA_B, enabling unauthorized actions on behalf of Account A's COA_B.
