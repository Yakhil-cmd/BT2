### Title
Wrong Variable Passed as Total Stake in `update_rewards` Consensus Path — (File: `src/staking/staking.cairo`)

---

### Summary

In the `update_rewards` function (consensus rewards path), the **staker's own balance** (`staker_total_strk_balance`) is incorrectly passed as the **system-wide total stake** (`strk_total_stake`) argument to `_update_rewards`. The pre-consensus attestation path correctly calls `get_current_total_staking_power()` for this argument. The mismatch causes each staker to receive a disproportionately large share of block rewards.

---

### Finding Description

`update_rewards` is the public entry point for consensus-phase per-block reward distribution. It is defined in `StakingRewardsManagerImpl` and callable by any non-zero address.

At lines 1474–1478, the function computes the **individual staker's** STRK and BTC balances:

```cairo
let (staker_total_strk_balance, staker_total_btc_balance) = self
    .get_staker_total_strk_btc_balance_at_epoch(
        :staker_address, :staker_pool_info, epoch_id: curr_epoch,
    );
```

Then at lines 1493–1506, these per-staker values are forwarded to `_update_rewards` under the parameter names `strk_total_stake` and `btc_total_stake`:

```cairo
self
    ._update_rewards(
        :staker_address,
        strk_total_rewards: strk_block_rewards,
        btc_total_rewards: btc_block_rewards,
        strk_total_stake: staker_total_strk_balance,   // ← BUG: staker balance, not total stake
        btc_total_stake: staker_total_btc_balance,     // ← BUG: staker balance, not total stake
        ...
    );
``` [1](#0-0) 

Contrast this with the **pre-consensus attestation path** in `update_rewards_from_attestation_contract` (lines 1406–1422), which correctly fetches the system-wide total staking power:

```cairo
let (strk_total_stake, btc_total_stake) = self.get_current_total_staking_power();
self
    ._update_rewards(
        :staker_address,
        strk_total_rewards: strk_epoch_rewards,
        btc_total_rewards: btc_epoch_rewards,
        :strk_total_stake,
        :btc_total_stake,
        ...
    );
``` [2](#0-1) 

The parameter name `strk_total_stake` in `_update_rewards` is unambiguous: it is the denominator used to compute each staker's proportional share of rewards:

```
staker_rewards = total_block_rewards × staker_balance / strk_total_stake
```

When `strk_total_stake` is set to `staker_balance` (the bug), the division cancels:

```
staker_rewards = total_block_rewards × staker_balance / staker_balance
               = total_block_rewards   (100% of block rewards)
```

Every staker who triggers `update_rewards` for themselves receives the **entire block reward** rather than their proportional share.

---

### Impact Explanation

`update_rewards` is gated by a single global `last_reward_block` check (line 1454–1458), meaning only one staker can be updated per block. However, any staker can call the function for themselves each block. The staker who calls it receives `strk_block_rewards` in full — the entire system-wide block reward — instead of `strk_block_rewards × (staker_balance / total_stake)`. The excess is drawn from the `RewardSupplier`, which mints new STRK. This constitutes **theft of unclaimed yield** from all other stakers and inflates the token supply beyond the intended minting curve. [3](#0-2) 

Impact classification: **High — Theft of unclaimed yield**.

---

### Likelihood Explanation

- `update_rewards` is a public `#[abi(embed_v0)]` function with no role restriction; any non-zero address can call it.
- The consensus rewards phase is the production steady-state after `consensus_rewards_first_epoch` is set.
- A rational staker calls `update_rewards` for themselves every block to maximize rewards.
- No special conditions, flash loans, or governance compromise are required.

Likelihood: **High**.

---

### Recommendation

Replace the staker-scoped balance variables with the system-wide total staking power, mirroring the attestation path:

```cairo
// Consensus path fix — fetch total staking power, not staker's own balance
let (strk_total_stake, btc_total_stake) = self.get_current_total_staking_power();
self
    ._update_rewards(
        :staker_address,
        strk_total_rewards: strk_block_rewards,
        btc_total_rewards: btc_block_rewards,
        :strk_total_stake,
        :btc_total_stake,
        :staker_info,
        :staker_pool_info,
        :reward_supplier_dispatcher,
        :curr_epoch,
    );
```

The staker's own balance (`staker_total_strk_balance`, `staker_total_btc_balance`) is still needed for the `is_non_zero` guard and can be retained for that purpose only.

---

### Proof of Concept

1. Consensus rewards are active (`consensus_rewards_first_epoch` has been set and passed).
2. Staker A holds 1% of total STRK stake. Total system stake = 1,000,000 STRK. Staker A stake = 10,000 STRK.
3. Block rewards for the epoch = 1,000 STRK.
4. Staker A calls `update_rewards(staker_address: A, disable_rewards: false)`.
5. **Expected**: `_update_rewards` receives `strk_total_stake = 1,000,000`. Staker A earns `1,000 × 10,000 / 1,000,000 = 10 STRK`.
6. **Actual (bug)**: `_update_rewards` receives `strk_total_stake = 10,000` (staker A's own balance). Staker A earns `1,000 × 10,000 / 10,000 = 1,000 STRK` — the full block reward.
7. Staker A repeats every block, draining the `RewardSupplier` and stealing yield owed to all other stakers. [4](#0-3)

        fn update_rewards(
            ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
        ) {
            self.general_prerequisites();
            let current_block_number = starknet::get_block_number();
            assert!(
                current_block_number > self.last_reward_block.read(),
                "{}",
                Error::REWARDS_ALREADY_UPDATED,
            );

            // Assert staker exists and active.
            // Staker is considered to exist from the moment of `stake` (when `InternalStakerInfo`
            // struct is created) until the calling to `unstake_action` (when `InternalStakerInfo`
            // struct is deleted).
            // Staker remains active until the intent period begins, i.e. K epochs after
            // `unstake_intent` is called.
            let staker_info = self.internal_staker_info(:staker_address);
            let curr_epoch = self.get_current_epoch();
            assert!(
                self.is_staker_active(:staker_address, epoch_id: curr_epoch),
                "{}",
                Error::INVALID_STAKER,
            );

            let staker_pool_info = self.staker_pool_info.entry(staker_address).as_non_mut();
            let (staker_total_strk_balance, staker_total_btc_balance) = self
                .get_staker_total_strk_btc_balance_at_epoch(
                    :staker_address, :staker_pool_info, epoch_id: curr_epoch,
                );
            // Assert staker has non-zero balance.
            // Staker exists with zero balance for the first K epochs after `stake`, then the stake
            // becomes effective.
            assert!(staker_total_strk_balance.is_non_zero(), "{}", Error::INVALID_STAKER);

            // Update last block rewards.
            self.last_reward_block.write(current_block_number);

            if disable_rewards || self.is_pre_consensus() {
                return;
            }

            // Get current block data and update rewards.
            let reward_supplier_dispatcher = self.reward_supplier_dispatcher.read();
            let (strk_block_rewards, btc_block_rewards) = self
                .calculate_block_rewards(:reward_supplier_dispatcher, :curr_epoch);
            self
                ._update_rewards(
                    :staker_address,
                    strk_total_rewards: strk_block_rewards,
                    btc_total_rewards: btc_block_rewards,
                    strk_total_stake: staker_total_strk_balance,
                    btc_total_stake: staker_total_btc_balance,
                    :staker_info,
                    :staker_pool_info,
                    :reward_supplier_dispatcher,
                    :curr_epoch,
                );
        }
