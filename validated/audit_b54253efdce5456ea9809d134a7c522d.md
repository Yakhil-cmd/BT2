### Title
Gateway Skips Signature Validation for Invoke Transactions When Any Account Transaction Exists in Mempool — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator skips the `__validate__` entry-point call (i.e., account signature verification) for any invoke transaction with `tx_nonce == 1` when the on-chain `account_nonce == 0` and `account_tx_in_pool_or_recent_block` returns `true`. The membership check returns `true` for **any** transaction in the mempool for that account — not exclusively a `deploy_account` transaction. An unprivileged attacker who observes a victim's pending `deploy_account` transaction in the mempool can submit an invoke with nonce=1 and an arbitrary/invalid signature for the victim's account. The gateway admits this transaction without signature verification; the batcher later executes it, `__validate__` fails, the transaction reverts, and fees are charged to the victim's account.

---

### Finding Description

`skip_stateful_validations` is designed to improve UX for the simultaneous `deploy_account + invoke` pattern: because the account does not yet exist on-chain, `__validate__` cannot run. The guard is:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs  lines 437-456
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
    return mempool_client
        .account_tx_in_pool_or_recent_block(tx.sender_address())
        .await
        ...
}
``` [1](#0-0) 

`account_tx_in_pool_or_recent_block` is implemented as:

```rust
// crates/apollo_mempool/src/mempool.rs  lines 697-700
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [2](#0-1) 

It returns `true` for **any** transaction type in the mempool for that address. The code comment acknowledges this: *"it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."* The second case is safe (the account exists and the caller controls it). The first case is the vulnerability: a victim's `deploy_account` tx in the mempool satisfies the check, allowing an attacker who does **not** control the account to submit an invoke with nonce=1 and skip `__validate__`.

When `skip_validate == true`, `run_validate_entry_point` sets `execution_flags.validate = false`:

```rust
// lines 310-312
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [3](#0-2) 

`StatefulValidator::perform_validations` then returns early without calling `__validate__`:

```rust
// crates/blockifier/src/blockifier/stateful_validator.rs  lines 79-81
if !tx.execution_flags.validate {
    return Ok(());
}
``` [4](#0-3) 

The fee check (`verify_can_pay_committed_bounds`) still runs inside `perform_pre_validation_stage` and passes because the victim's account has balance (required for deployment). The invalid invoke is therefore admitted to the mempool. [5](#0-4) 

During batcher execution, `__validate__` is called with `validate = true` (sequencing flags), fails on the invalid signature, the transaction reverts, and fees are charged to the victim's account.

---

### Impact Explanation

Matches: **High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

An attacker with no privileges can inject a transaction bearing an invalid (attacker-chosen) signature into the mempool for any account whose `deploy_account` is pending. The transaction is included in a block, reverts, and the resulting fee is debited from the victim's account — a direct, unprivileged financial loss to the victim.

---

### Likelihood Explanation

`deploy_account` transactions are publicly visible in the mempool. Any account being deployed for the first time is vulnerable during the window between mempool admission of `deploy_account` and its on-chain execution. The attack requires only a single gateway call with a crafted invoke transaction; no special access or key material is needed.

---

### Recommendation

Replace the coarse `account_tx_in_pool_or_recent_block` check with a type-specific query that confirms a `deploy_account` transaction (not just any transaction) is pending for the sender address. Alternatively, expose a `deploy_account_tx_in_pool` predicate from the mempool and use it in `skip_stateful_validations`. This preserves the UX intent while closing the signature-bypass window.

---

### Proof of Concept

1. Victim funds account `X` and submits `deploy_account` (nonce=0) to the gateway. The transaction enters the mempool; on-chain `account_nonce(X) == 0`.
2. Attacker calls the gateway `add_tx` with an `InvokeV3` for account `X`, `nonce=1`, and an all-zero (invalid) signature.
3. Gateway stateless check: signature length ≤ 4000 — passes.
4. Gateway stateful check:
   - `account_nonce = 0` (from state).
   - `validate_nonce`: `0 ≤ 1 ≤ 200` — passes.
   - `validate_resource_bounds`: account has balance — passes.
   - `skip_stateful_validations`: `tx_nonce==1 && account_nonce==0 && account_tx_in_pool_or_recent_block(X)==true` → returns `true`.
   - `run_validate_entry_point` with `validate=false`: `perform_pre_validation_stage` (fee check passes), `__validate__` **skipped**.
5. Invalid invoke is admitted to the mempool.
6. Batcher executes `deploy_account` (nonce=0), deploying account `X`.
7. Batcher executes invoke (nonce=1) with `validate=true`; `__validate__` runs, fails on invalid signature, transaction reverts.
8. Fee is charged to account `X` (victim).
