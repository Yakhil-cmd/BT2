### Title
Missing Identity-Point Validation for `app_pk` in `ckd()` Initialization Allows Confidential Derived Key Disclosure — (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

The `ckd()` entry-point function accepts an `app_pk: PublicKey` (a `blstrs::G1Projective` point) without checking whether it is the group identity (point at infinity). If a caller supplies the identity point, the ElGamal blinding term vanishes entirely, and the coordinator receives the raw, unblinded BLS signature shares. Aggregating them yields the confidential derived key `s = msk · H(pk ∥ app_id)` in plaintext — the exact secret the protocol is designed to keep hidden.

---

### Finding Description

The CKD protocol is an ElGamal-based scheme. Each participant computes:

```
C_i = S_i + y_i · A      where A = app_pk, S_i = x_i · H(pk ∥ app_id)
Y_i = y_i · G
```

The coordinator aggregates the Lagrange-normalized shares to obtain `(Y, C)` such that:

```
C - a · Y  =  msk · H(pk ∥ app_id)  =  s
```

The blinding term `y_i · A` is what prevents any single party (or the coordinator) from learning `s` during the protocol. If `A = identity`, then `y_i · A = identity` for every participant, so:

```
C_i = S_i + identity = S_i = x_i · H(pk ∥ app_id)
```

The coordinator aggregates these to get `C = msk · H(pk ∥ app_id)` — the confidential derived key — directly, with no blinding at all.

The `ckd()` function performs four checks at initialization: [1](#0-0) 

None of these checks validate `app_pk`. The parameter flows unchecked into `compute_signature_share`: [2](#0-1) 

Specifically, line 174 computes `big_c = big_s + app_pk * y.0` — if `app_pk` is the identity, this reduces to `big_c = big_s`, stripping all blinding. [3](#0-2) 

The `verify_signature` helper in `ciphersuite.rs` does reject identity elements, but it is only called by external consumers after the fact — it is not invoked inside `ckd()`: [4](#0-3) 

The `PublicKey` type alias is a raw `ElementG1 = blstrs::G1Projective`, which has no built-in non-identity invariant: [5](#0-4) 

---

### Impact Explanation

**Critical — Disclosure of confidential derived secrets.**

The entire confidentiality guarantee of CKD is that no MPC node (including the coordinator) ever sees `s = msk · H(pk ∥ app_id)`. With `app_pk = identity`, the coordinator's aggregated `C` value *is* `s` in plaintext. The `unmask(a)` call by the TEE app becomes `C - 0·Y = C = s`, confirming the leak. This directly matches the allowed critical impact: *"Extraction, reconstruction, or disclosure of… confidential derived secrets."*

---

### Likelihood Explanation

The `app_pk` is supplied by the caller of `ckd()` — in the documented deployment model, this originates from an on-chain CKD request carrying the TEE app's freshly generated public key `A`. A malicious TEE app, a compromised developer contract, or any entity that can submit a CKD request on-chain can trivially set `A` to the identity point. No cryptographic capability is required; it is a single-field substitution in the transaction payload. The MPC nodes accept it without complaint.

---

### Recommendation

Add an explicit identity-point check for `app_pk` inside `ckd()` before the protocol is launched, analogous to the check already present in `verify_signature`:

```rust
// In ckd(), after the participant-list checks:
if app_pk.is_identity().into() {
    return Err(InitializationError::BadParameters(
        "app_pk must not be the identity point".to_string(),
    ));
}
```

This mirrors the pattern already used in `BLS12381G1Group::serialize` and `verify_signature` to reject identity elements. [6](#0-5)
