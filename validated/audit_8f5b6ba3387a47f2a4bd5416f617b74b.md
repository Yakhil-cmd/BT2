### Title
Transaction `Authorizers` Array Not Validated for Duplicate Addresses — (`File: fvm/transactionVerifier.go`)

---

### Summary

The `tx.Authorizers` slice in a Flow transaction is never checked to be a unique set. An unprivileged sender can craft a transaction that lists the same authorizer address multiple times. The FVM's `verifyTransaction` function iterates over the raw `tx.Authorizers` slice when checking key-weight sufficiency, and because `payloadWeights` is a map keyed by address, a duplicate authorizer entry causes the weight check to pass for the same address twice — once legitimately and once as a free repeat — while the Cadence runtime receives the full (duplicated) authorizer list and binds a separate `&Account` reference for each occurrence.

---

### Finding Description

In `fvm/transactionVerifier.go`, `verifyTransaction` iterates `tx.Authorizers` to enforce that each authorizer has sufficient signing weight:

```go
for _, addr := range tx.Authorizers {
    if addr == tx.Payer {
        continue
    }
    if !v.hasSufficientKeyWeight(payloadWeights, addr, keyWeightThreshold) {
        return errors.NewAccountAuthorizationErrorf(...)
    }
}
```

`payloadWeights` is a `map[flow.Address]int`. If `tx.Authorizers` contains `[addrA, addrA]`, the map lookup `payloadWeights[addrA]` returns the same accumulated weight both times, so the check passes for both iterations with only one real signature. No deduplication of the `Authorizers` slice itself is performed anywhere in the validation pipeline.

The upstream Access/Collection node validator (`access/validator/validator.go`) also does not check for duplicate authorizer addresses — `checkAddresses` only validates each address against the chain's address generator, and `checkAccounts` only deduplicates *signatures*, not the authorizer list itself.

The Cadence runtime receives `tx.Authorizers` verbatim and binds one `&Account` signer reference per entry in the `prepare` block. A transaction with `prepare(a: &Account, b: &Account)` and `Authorizers: [addrX, addrX]` will execute with both `a` and `b` bound to the same account, bypassing the intent that two *distinct* accounts must authorize the transaction.

---

### Impact Explanation

An attacker who controls a single account `addrX` can submit a transaction that declares `Authorizers: [addrX, addrX]` and a script requiring two distinct authorizers (`prepare(a: &Account, b: &Account)`). The FVM weight check passes because `payloadWeights[addrX]` is checked twice against the same accumulated weight. The Cadence runtime then provides two `&Account` references both pointing to `addrX`. Any on-chain logic that assumes the two `prepare` parameters are distinct accounts — e.g., a contract that transfers assets from `a` to `b` — will operate on the same account twice, potentially causing double-spend of resources, incorrect state mutations, or bypassing multi-party authorization requirements enforced purely by the number of `prepare` parameters.

---

### Likelihood Explanation

The entry path is fully unprivileged: any user can submit a transaction via the Access API (gRPC `SendTransaction` or REST `POST /transactions`). The `Authorizers` field is a free-form array in the transaction body with no deduplication enforced at any layer before execution. The only limit is `maxAuthorizers = 100` (REST parser). The attack requires only a single valid account and a single valid signature.

---

### Recommendation

In `access/validator/validator.go`, add a uniqueness check for `tx.Authorizers` inside `checkAddresses` (or a dedicated step):

```go
seen := make(map[flow.Address]struct{}, len(tx.Authorizers))
for _, addr := range tx.Authorizers {
    if _, dup := seen[addr]; dup {
        return DuplicateAuthorizerError{Address: addr}
    }
    seen[addr] = struct{}{}
}
```

Additionally, `verifyTransaction` in `fvm/transactionVerifier.go` should deduplicate `tx.Authorizers` before the weight-check loop as a defense-in-depth measure, mirroring how `newSignatureEntries` already deduplicates signatures.

---

### Proof of Concept

1. Craft a transaction with:
   - `Script`: `transaction { prepare(a: &Account, b: &Account) { /* assert a.address != b.address */ } }`
   - `Authorizers`: `[addrX, addrX]`
   - `Payer`: `addrX`
   - One valid envelope signature from `addrX`

2. Submit via `SendTransaction` to any Access node.

3. The `checkAddresses` loop validates each address individually — both pass since `addrX` is a valid chain address. [1](#0-0) 

4. The `checkAccounts` loop checks for duplicate *signatures* (by address+keyIndex pair), not duplicate *authorizer addresses* — passes. [2](#0-1) 

5. In `verifyTransaction`, the weight loop iterates `[addrX, addrX]`; `payloadWeights[addrX]` satisfies the threshold on both iterations. [3](#0-2) 

6. The Cadence runtime receives the raw `Authorizers` slice with two entries of `addrX` and binds two `&Account` references to the same account, satisfying a `prepare(a: &Account, b: &Account)` block with only one real authorizer.

```go
func (v *TransactionValidator) checkAccounts(tx *flow.TransactionBody) error {
	// check for duplicate account key
	type uniqueKey struct {
		address flow.Address
		index   uint32
	}
	observedSigs := make(map[uniqueKey]bool)
	for _, sig := range append(tx.PayloadSignatures, tx.EnvelopeSignatures...) {
		if observedSigs[uniqueKey{sig.Address, sig.KeyIndex}] {
			return DuplicatedSignatureError{Address: sig.Address, KeyIndex: sig.KeyIndex}
		}
		observedSigs[uniqueKey{sig.Address, sig.KeyIndex}] = true
	}
	// check for minimum account signatures
	observedEnvelopeSig := make(map[flow.Address]bool)
	observedPayloadSig := make(map[flow.Address]bool)
	for _, sig := range tx.EnvelopeSignatures {
		observedEnvelopeSig[sig.Address] = true
	}
	for _, sig := range tx.PayloadSignatures {
		observedPayloadSig[sig.Address] = true
	}

	if !observedEnvelopeSig[tx.Payer] {
		return MissingSignatureError{Address: tx.Payer, Message: "payer envelope signature is missing"}
	}

	if !observedEnvelopeSig[tx.ProposalKey.Address] && !observedPayloadSig[tx.ProposalKey.Address] {
		return MissingSignatureError{Address: tx.ProposalKey.Address, Message: "proposer signature on either payload or envelope is missing"}
	}

	for _, authorizer := range tx.Authorizers {
		if authorizer == tx.Payer || authorizer == tx.ProposalKey.Address {
			// at this point, payer and proposer are guaranteed to have signatures
			continue
		}
		if !observedEnvelopeSig[authorizer] && !observedPayloadSig[authorizer] {
			return MissingSignatureError{Address: authorizer, Message: "authorizer signature on either payload or envelope is missing"}
		}
	}
```
