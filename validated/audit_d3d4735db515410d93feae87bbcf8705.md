### Title
Permanent Loss of Bridged Funds When `ExitToEthereum` Precompile's Downstream `withdraw` Promise Fails — (File: `engine-precompiles/src/native.rs`)

---

### Summary

The `ExitToEthereum` precompile burns ERC-20 tokens (or deducts ETH) in the EVM before dispatching a NEAR promise to call `withdraw` on the eth-connector. Unlike `ExitToNear`, which attaches a `exit_to_near_precompile_callback` that can re-mint/refund tokens on failure, `ExitToEthereum` creates only a bare `PromiseArgs::Create` with **no callback and no refund path**. If the downstream `withdraw` promise fails for any reason (eth-connector paused, `withdraw` feature paused, contract upgrade, etc.), the burned tokens are permanently unrecoverable.

---

### Finding Description

**Step 1 — Burn happens before the cross-chain promise.**

In `EvmErc20.sol`, `withdrawToEthereum` first burns the caller's tokens, then calls the `ExitToEthereum` precompile:

```solidity
function withdrawToEthereum(address recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // tokens destroyed here
    ...
    assembly {
        let res := call(gas(), 0xb0bd02f6..., 0, ...)  // ExitToEthereum precompile
    }
}
``` [1](#0-0) 

**Step 2 — `ExitToEthereum::run` creates a fire-and-forget promise with no callback.**

The precompile constructs a `PromiseArgs::Create` targeting the eth-connector's `withdraw` method and returns. There is no `PromiseWithCallbackArgs`, no `exit_to_ethereum_precompile_callback`, and no refund logic:

```rust
let withdraw_promise = PromiseCreateArgs {
    target_account_id: nep141_address,
    method: "withdraw".to_string(),
    ...
};
let promise = borsh::to_vec(&PromiseArgs::Create(withdraw_promise)).unwrap();
``` [2](#0-1) 

**Step 3 — Contrast: `ExitToNear` has a refund callback; `ExitToEthereum` does not.**

`ExitToNear::run` conditionally wraps the promise in a `PromiseArgs::Callback` that calls `exit_to_near_precompile_callback`, which invokes `engine::refund_on_error` to re-mint burned ERC-20 tokens or return ETH if the promise fails: [3](#0-2) [4](#0-3) 

`ExitToEthereum` has no equivalent. The asymmetry is structural and intentional in the current code, but leaves the Ethereum-exit path unprotected.

**Step 4 — The eth-connector's `withdraw` can be paused independently.**

The eth-connector exposes `pa_pause_feature` with keys like `"engine_withdraw"` and `"withdraw"`, and the engine itself has `pause_contract`. Any of these can cause the `withdraw` promise to fail after the EVM burn has already committed: [5](#0-4) [6](#0-5) 

The `ExitToEthereum` precompile pause (`PrecompileFlags::EXIT_TO_ETHEREUM`) only prevents the EVM call from reaching the precompile — it causes the EVM transaction to revert, which is safe. The dangerous window is when the precompile itself is **not** paused but the eth-connector's `withdraw` is paused or fails: [7](#0-6) [8](#0-7) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

When the eth-connector's `withdraw` promise fails after the EVM burn has committed, the burned ERC-20 tokens (or ETH) are gone from Aurora with no on-chain recovery path. There is no retry mechanism, no claimable mapping, and no callback to re-mint. The user's funds are permanently destroyed.

---

### Likelihood Explanation

**Medium.** The scenario requires the eth-connector's `withdraw` to fail while the `ExitToEthereum` precompile is not paused. This is a realistic operational condition:

- Admins may pause the eth-connector for maintenance, upgrades, or security incidents without first pausing the `ExitToEthereum` precompile.
- Any in-flight `withdrawToEthereum` transaction submitted before the precompile pause takes effect will have its tokens burned with no refund.
- The `ExitToNear` path already demonstrates the team is aware of this failure mode (hence the `error_refund` feature), making the omission on the Ethereum-exit path a known gap.

---

### Recommendation

1. Add a `exit_to_ethereum_precompile_callback` analogous to `exit_to_near_precompile_callback` that re-mints ERC-20 tokens (or returns ETH) if the `withdraw` promise fails.
2. Change `ExitToEthereum::run` to emit `PromiseArgs::Callback(PromiseWithCallbackArgs { base: withdraw_promise, callback: refund_callback })` instead of the bare `PromiseArgs::Create`.
3. Operationally: always pause the `ExitToEthereum` precompile (via `pause_precompiles`) before pausing or upgrading the eth-connector, and ensure all in-flight promises have settled before doing so.

---

### Proof of Concept

1. Deploy Aurora with an ERC-20 token bridged from NEAR (NEP-141 ↔ ERC-20).
2. User calls `EvmErc20.withdrawToEthereum(recipient, amount)` — tokens are burned in the EVM, `ExitToEthereum` precompile fires a `withdraw` promise to the eth-connector.
3. Before the promise executes, admin calls `pa_pause_feature({"key": "withdraw"})` on the eth-connector (or pauses the eth-connector contract entirely).
4. The `withdraw` NEAR receipt executes and fails (paused).
5. No callback fires. The burned ERC-20 tokens are gone. The user's balance on Aurora is zero, and no ETH is released on Ethereum.
6. Even after unpausing, there is no on-chain mechanism to recover the lost tokens — confirmed by the absence of any claimable mapping or retry entry point in `ExitToEthereum`. [1](#0-0) [9](#0-8)
