### Title
Permanent Fund Loss in `ExitToNear` Precompile When `ft_transfer` Promise Fails Without `error_refund` Feature - (File: `engine-precompiles/src/native.rs`)

---

### Summary

The `ExitToNear` precompile burns ERC-20 tokens (or deducts ETH) from a user's EVM balance and then schedules a NEAR `ft_transfer` promise to deliver the corresponding NEP-141 tokens to the recipient. When the `error_refund` feature is not compiled in, no callback is attached to the `ft_transfer` promise for the standard ERC-20 and ETH exit paths. If the `ft_transfer` promise fails (e.g., the recipient is not registered with the NEP-141 contract), the EVM-side tokens are permanently destroyed with no recovery path.

---

### Finding Description

In `engine-precompiles/src/native.rs`, the `ExitToNear::run` method constructs `callback_args` as follows:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,
    transfer_near: transfer_near_args,
};
``` [1](#0-0) 

When `error_refund` is not enabled, `refund` is always `None`. For standard ERC-20 exits and ETH exits (non-wNEAR unwrap), `transfer_near_args` is also `None`. This means `callback_args` equals `ExitToNearPrecompileCallbackArgs::default()`, triggering the branch:

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // <-- fire-and-forget, no callback
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs { ... })
};
``` [2](#0-1) 

A fire-and-forget `PromiseArgs::Create` is scheduled — no `exit_to_near_precompile_callback` is ever invoked. In `exit_to_near_precompile_callback`, the refund path is:

```rust
} else if let Some(args) = args.refund {
    // Exit call failed; need to refund tokens
    let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;
    ...
} else {
    None  // <-- no refund, no error
};
``` [3](#0-2) 

Since `refund` is always `None` without the feature, even if the callback were invoked, no refund would occur. But more critically, the callback is never scheduled at all for the standard exit paths.

The `ExitToNearPrecompileCallbackArgs` struct confirms `refund` is optional and defaults to `None`: [4](#0-3) 

The test suite explicitly acknowledges this behavior:

```rust
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
``` [5](#0-4) 

And for ETH:

```rust
// If the refund feature is not enabled, then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let expected_balance = Wei::new_u64(INITIAL_ETH_BALANCE - ETH_EXIT_AMOUNT);
``` [6](#0-5) 

---

### Impact Explanation

**Critical — Permanent freezing/destruction of funds.**

When a user calls the `ExitToNear` precompile (directly for ETH, or via an ERC-20 contract's burn-and-exit function for NEP-141-backed tokens):

1. ERC-20 tokens are burned from the user's EVM balance (irreversible at EVM level).
2. A `ft_transfer` NEAR promise is scheduled to deliver NEP-141 tokens to the recipient.
3. If `ft_transfer` fails (e.g., recipient not registered with the NEP-141 contract), the NEP-141 tokens remain in Aurora's account — but the EVM-side tokens are already gone.
4. No callback is scheduled to detect the failure and re-mint/refund the EVM tokens.

The user loses their funds permanently. The NEP-141 tokens are effectively locked in Aurora's account with no on-chain mechanism to return them to the original EVM address.

---

### Likelihood Explanation

**High.** The failure condition — an unregistered recipient account — is a routine, user-triggerable scenario. Any EVM user who calls `ExitToNear` targeting a NEAR account that has not called `storage_deposit` on the NEP-141 contract will trigger this loss. This requires no special privileges, no admin compromise, and no external oracle. The attacker-controlled entry path is:

1. User holds ERC-20 tokens (bridged NEP-141) on Aurora.
2. User calls the ERC-20 contract's withdraw/exit function, specifying any unregistered NEAR account as recipient.
3. The ERC-20 contract burns the tokens and calls the `ExitToNear` precompile.
4. The precompile schedules `ft_transfer` with no callback.
5. `ft_transfer` fails silently; tokens are gone.

The same applies to ETH exits via `withdrawEthToNear`. [7](#0-6) 

---

### Recommendation

Enable the `error_refund` feature in all production builds, or unconditionally attach the `exit_to_near_precompile_callback` as a callback to every `ft_transfer` promise in the `ExitToNear` precompile. The callback must check the promise result and, on failure, re-mint the burned ERC-20 tokens (or restore the ETH balance) to the original sender's EVM address. The `error_refund` feature and `refund_call_args` logic already implement this correctly — the issue is that it is gated behind a compile-time feature flag that may not be enabled in all deployments. [8](#0-7) 

---

### Proof of Concept

1. Deploy Aurora without the `error_refund` feature.
2. Bridge a NEP-141 token to Aurora (creating an ERC-20 mirror).
3. From an EVM account holding the ERC-20, call the exit function targeting `"unregistered.near"` (any account that has not called `storage_deposit` on the NEP-141).
4. Observe: the ERC-20 balance decreases (tokens burned), the `ft_transfer` promise fails, no callback fires, and the ERC-20 balance is not restored.
5. The NEP-141 tokens remain in Aurora's account; the user's ERC-20 tokens are permanently destroyed.

This is directly confirmed by the existing test `test_exit_to_near_refund` in `engine-tests/src/tests/erc20_connector.rs`, which asserts the reduced balance `(FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT)` when `error_refund` is not enabled — proving the loss is real and acknowledged. [9](#0-8)

#[cfg(feature = "error_refund")]
#[allow(clippy::unnecessary_wraps)]
fn refund_call_args(
    params: &ExitToNearParams,
    event: &events::ExitToNear,
) -> Option<RefundCallArgs> {
    Some(RefundCallArgs {
        recipient_address: match params {
            ExitToNearParams::BaseToken(params) => params.refund_address,
            ExitToNearParams::Erc20TokenParams(params) => params.refund_address,
        },
        erc20_address: match params {
            ExitToNearParams::BaseToken(_) => None,
            ExitToNearParams::Erc20TokenParams(_) => {
                let erc20_address = match event {
                    events::ExitToNear::Legacy(legacy) => legacy.erc20_address,
                    events::ExitToNear::Omni(omni) => omni.erc20_address,
                };
                Some(erc20_address)
            }
        },
        amount: types::u256_to_arr(&match event {
            events::ExitToNear::Legacy(legacy) => legacy.amount,
            events::ExitToNear::Omni(omni) => omni.amount,
        }),
    })
}
