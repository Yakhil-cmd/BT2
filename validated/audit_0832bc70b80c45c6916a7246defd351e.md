### Title
Missing Expiration Check on Scheduled XCC Promises Enables Stale Execution at Attacker-Chosen Time - (File: etc/xcc-router/src/lib.rs)

---

### Summary

The XCC router's `execute_scheduled` function has no expiration/deadline check and the router provides no cancellation mechanism. Once a user schedules a cross-contract call via `CrossContractCallArgs::Delayed`, the associated NEAR is irrevocably locked in the router sub-account and the promise can be triggered by any external account at any future block. This is a direct analog to the missing deadline check described in the reference report.

---

### Finding Description

When a user invokes the XCC precompile with `CrossContractCallArgs::Delayed`, the following happens:

1. The precompile computes `attached_near = call.total_near()` and initiates a `withdraw_wnear_to_router` call that moves the user's wNEAR out of the EVM and into the router sub-account as real NEAR. [1](#0-0) 

2. The router's `schedule` function (callable only by the parent Aurora Engine) stores the `PromiseArgs` keyed by an auto-incrementing nonce. No timestamp, block height, or deadline is stored alongside the promise. [2](#0-1) 

3. `execute_scheduled` is intentionally open to **any caller** with no expiration guard:

```rust
#[payable]
pub fn execute_scheduled(&mut self, nonce: U64) {
    let Some(promise) = self.scheduled_promises.remove(&nonce.0) else {
        env::panic_str("ERR_PROMISE_NOT_FOUND")
    };
    let promise_id = Self::promise_create(promise);
    env::promise_return(promise_id);
}
``` [3](#0-2) 

4. The `PromiseCreateArgs` type carries no deadline field — only `target_account_id`, `method`, `args`, `attached_balance`, and `attached_gas`. [4](#0-3) 

5. There is **no `cancel_scheduled` or equivalent function** in the router. The full public interface is `initialize`, `get_version`, `execute`, `schedule`, `execute_scheduled`, `deploy_upgrade`, and `send_refund`. [5](#0-4) 

Consequence: once a promise is scheduled and NEAR is deposited into the router, the user has no recourse. The NEAR is locked and the promise will execute whenever any external account decides to call `execute_scheduled`, which can be arbitrarily far in the future.

---

### Impact Explanation

**Temporary freezing of funds (High):** The user's NEAR (unwrapped from wNEAR by `withdraw_wnear_to_router`) sits in the router sub-account with no cancellation path. The user cannot recover it. [6](#0-5) 

**Financial loss from stale execution (High):** A scheduled promise that targets a NEAR DeFi contract (e.g., a DEX swap, liquidity deposit, or lending action) encodes its parameters at scheduling time. If an adversary — or even a neutral third party — calls `execute_scheduled` after market conditions have shifted significantly, the operation executes with stale parameters. Because the `PromiseCreateArgs.args` bytes are fixed and there is no minimum-output guard at the router layer, the user receives whatever the target contract returns at that moment, which may be far below their expectation. [7](#0-6) 

---

### Likelihood Explanation

- `CrossContractCallArgs::Delayed` is a documented, production-facing feature of the XCC precompile. [1](#0-0) 
- Any EVM contract or EOA on Aurora can invoke the XCC precompile and choose the `Delayed` path.
- `execute_scheduled` requires no special permission and no attached deposit; any NEAR account can call it at zero cost beyond gas. [3](#0-2) 
- MEV searchers on NEAR already monitor public contract state; a scheduled promise with a large `attached_balance` is visible on-chain and can be timed for maximum harm.

---

### Recommendation

1. **Add an optional `deadline` field to `PromiseArgs` / `PromiseCreateArgs`** (block height or timestamp). `execute_scheduled` should panic if `env::block_timestamp() > deadline`.
2. **Add a `cancel_scheduled(nonce)` function** restricted to the parent (Aurora Engine), callable via a new EVM precompile entry point, so users can reclaim locked NEAR.
3. **Alternatively**, enforce that `execute_scheduled` can only be called by the parent, removing the open-caller design for the `Delayed` path.

---

### Proof of Concept

1. Alice calls the XCC precompile from her EVM address with `CrossContractCallArgs::Delayed(swap_100_NEAR_for_TOKEN_at_market)`. Her wNEAR is unwrapped and 100 NEAR lands in her router sub-account. The promise is stored at nonce `0`. [8](#0-7) 

2. Alice intends to call `execute_scheduled(0)` herself within the next few blocks. However, she is front-run or simply delayed.

3. Three days later, the TOKEN price has crashed 80%. Bob (any NEAR account) calls:
   ```
   alice_router.execute_scheduled({"nonce": "0"})
   ```
   with no restrictions. [9](#0-8) 

4. The swap executes at the current (crashed) price. Alice receives 20% of the TOKEN she expected. Her 100 NEAR is gone with no recourse, because there is no cancellation path and no expiration guard. [2](#0-1)

#[near]
impl Router {
    #[init(ignore_state)]
    #[must_use]
    pub fn initialize(wnear_account: AccountId, must_register: bool) -> Self {
        // The first time this function is called there is no state and the parent is set to be
        // the predecessor account id. In subsequent calls, only the original parent is allowed to
        // call this function. The idea is that the Create, Deploy and Initialize actions are done in a single
        // NEAR batch when a new router is deployed by the engine, so the caller will be the Aurora
        // engine instance that the user's address belongs to. If we update this contract and deploy
        // a new version of it, again the Deploy and Initialize actions will be done in a single batch
        // by the engine.
        let caller = env::predecessor_account_id();
        let mut parent = LazyOption::new(StorageKey::Parent, None);
        match parent.get() {
            None => {
                parent.set(&caller);
            }
            Some(parent) => {
                // Allow self-calls to `initialize` also.
                // This happens during the upgrade flow.
                if (caller != parent) && (caller != env::current_account_id()) {
                    env::panic_str(ERR_ILLEGAL_CALLER);
                }
            }
        }

        if must_register {
            env::promise_create(
                wnear_account.clone(),
                "storage_deposit",
                b"{}",
                WNEAR_REGISTER_AMOUNT,
                WNEAR_REGISTER_GAS,
            );
        }

        let mut version = LazyOption::new(StorageKey::Version, None);
        if version.get().unwrap_or_default() != CURRENT_VERSION {
            // Future migrations would go here

            version.set(&CURRENT_VERSION);
        }

        let nonce = LazyOption::new(StorageKey::Nonce, None);
        let scheduled_promises = LookupMap::new(StorageKey::Map);
        Self {
            parent,
            version,
            nonce,
            scheduled_promises,
            wnear_account,
        }
    }

    pub fn get_version(&self) -> u32 {
        self.version.get().unwrap_or_default()
    }

    /// This function can only be called by the parent account (i.e. Aurora engine) to ensure that
    /// no one can create calls on behalf of the user this router contract is deployed for.
    /// The engine only calls this function when the special precompile in the EVM for NEAR cross
    /// contract calls is used by the address associated with the sub-account this router contract
    /// is deployed at.
    pub fn execute(&self, #[serializer(borsh)] promise: PromiseArgs) {
        self.assert_preconditions();

        let promise_id = Self::promise_create(promise);
        env::promise_return(promise_id);
    }

    /// Similar security considerations here as for `execute`.
    pub fn schedule(&mut self, #[serializer(borsh)] promise: PromiseArgs) {
        self.assert_preconditions();

        let nonce = self.nonce.get().unwrap_or_default();
        self.scheduled_promises.insert(nonce, promise);
        self.nonce.set(&(nonce + 1));

        near_sdk::log!("Promise scheduled at nonce {}", nonce);
    }

    /// It is intentional that this function can be called by anyone (not just the parent).
    /// There is no security risk to allowing this function to be open because it can only
    /// act on promises that were created via `schedule`.
    #[payable]
    pub fn execute_scheduled(&mut self, nonce: U64) {
        let Some(promise) = self.scheduled_promises.remove(&nonce.0) else {
            env::panic_str("ERR_PROMISE_NOT_FOUND")
        };
        let promise_id = Self::promise_create(promise);
        env::promise_return(promise_id);
    }

    /// Allows the parent contract to trigger an update to the logic of this contract
    /// (by deploying a new contract to this account);
    #[payable]
    pub fn deploy_upgrade(&mut self, #[serializer(borsh)] args: DeployUpgradeParams) {
        self.assert_preconditions();

        let promise_id = env::promise_batch_create(&env::current_account_id());
        env::promise_batch_action_deploy_contract(promise_id, &args.code);
        env::promise_batch_action_function_call(
            promise_id,
            INITIALIZE,
            &args.initialize_args,
            NearToken::default(),
            INITIALIZE_GAS,
        );
        env::promise_return(promise_id);
    }

    pub fn send_refund(&self) -> Promise {
        let parent = self.get_parent().unwrap_or_else(env_panic);

        require_caller(&parent)
            .and_then(|_| require_no_failed_promises())
            .unwrap_or_else(env_panic);

        Promise::new(parent).transfer(REFUND_AMOUNT)
    }
}

pub fn withdraw_wnear_to_router<I: IO + Copy, E: Env, M: ModExpAlgorithm, H: PromiseHandler>(
    recipient: &AccountId,
    amount: Yocto,
    wnear_address: Address,
    engine: &mut Engine<I, E, M>,
    handler: &mut H,
) -> EngineResult<(SubmitResult, Vec<PromiseId>)> {
    let mut interceptor = PromiseInterceptor::new(handler);
    let withdraw_call_args = withdraw_wnear_call_args(recipient, amount, wnear_address);
    let result = engine.call_with_args(withdraw_call_args, &mut interceptor)?;
    Ok((result, interceptor.promises))
}
