### Title
Unchecked Precompile Return Value in `withdrawToNear`/`withdrawToEthereum` Causes Permanent ERC-20 Token Loss - (File: `etc/eth-contracts/contracts/EvmErc20.sol`)

---

### Summary

`EvmErc20.sol` implements `withdrawToNear()` and `withdrawToEthereum()` by first burning the caller's EVM-side tokens via `_burn()`, then invoking the exit precompile via a raw assembly `call`. The return value of the assembly `call` is captured in a local variable `res` but is **never checked**. If the precompile call fails for any reason, the EVM-side burn is committed permanently while the NEAR-side (or Ethereum-side) credit never occurs, creating an irrecoverable state discrepancy that destroys user funds.

---

### Finding Description

In `EvmErc20.sol`, both withdrawal functions follow the same pattern:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // EVM state updated: tokens destroyed

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        // res is NEVER checked — execution continues regardless of success or failure
    }
}
``` [1](#0-0) 

```solidity
function withdrawToEthereum(address recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // EVM state updated: tokens destroyed

    bytes32 amount_b = bytes32(amount);
    bytes20 recipient_b = bytes20(recipient);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient_b);
    uint input_size = 1 + 32 + 20;

    assembly {
        let res := call(gas(), 0xb0bd02f6a392af548bdf1cfaee5dfa0eefcc8eab, 0, add(input, 32), input_size, 0, 32)
        // res is NEVER checked
    }
}
``` [2](#0-1) 

The EVM `call` opcode returns `0` on failure and `1` on success. When the precompile at `0xe921...` (exit-to-near) or `0xb0bd...` (exit-to-ethereum) fails, the assembly block silently captures `res = 0` and the outer Solidity function returns normally — **no revert is triggered**. The `_burn()` that already executed is not rolled back. The two states — EVM token supply and NEAR/Ethereum-side token credit — are now permanently out of sync.

This is structurally identical to the ERC2981 analog: one authoritative state (EVM balance) is updated while the paired state (NEAR-side NEP-141 balance or Ethereum bridge credit) is silently skipped.

---

### Impact Explanation

A user calling `withdrawToNear()` or `withdrawToEthereum()` with a recipient that causes the precompile to fail (e.g., a malformed NEAR account ID, a paused precompile, or an unregistered token) will have their EVM tokens permanently burned with zero credit on the destination chain. The tokens are irrecoverable: the EVM supply is reduced, the NEAR-side supply is unchanged, and there is no refund path in the contract. This constitutes **permanent freezing/destruction of user funds**.

**Impact class: Critical — Permanent freezing of funds.**

---

### Likelihood Explanation

The precompile call can fail under several realistic conditions reachable by any unprivileged user:

1. **Invalid NEAR account ID**: `withdrawToNear` accepts a raw `bytes memory recipient`. A user supplying a syntactically invalid NEAR account ID will cause the exit-to-near precompile to reject the call, returning failure. The burn is already committed.
2. **Precompile paused**: Aurora Engine supports pausing precompiles via `EnginePrecompilesPauser`. If the exit-to-near or exit-to-ethereum precompile is paused at the time of the call, the assembly `call` returns `0`, the burn is permanent, and the user loses funds.
3. **Token not registered in connector**: If the NEP-141 backing token is not registered in the ETH connector, the precompile will fail.

All three scenarios are reachable by an ordinary token holder interacting with the intended `EvmErc20` interface.

---

### Recommendation

Check the return value of the assembly `call` and revert if it is zero:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

This ensures the `_burn()` is atomically rolled back if the precompile call fails, keeping EVM-side and NEAR-side token accounting in sync. The same fix must be applied to `withdrawToEthereum`. [3](#0-2) 

---

### Proof of Concept

1. Deploy `EvmErc20` on Aurora with a valid NEP-141 backing token.
2. Mint tokens to `alice` (EVM address).
3. `alice` calls `withdrawToNear(invalidNearAccountBytes, amount)` where `invalidNearAccountBytes` is a byte sequence that is not a valid NEAR account ID (e.g., contains uppercase letters or exceeds 64 characters).
4. The exit-to-near precompile at `0xe921...` rejects the call and returns failure.
5. The assembly block captures `res = 0` and the function returns normally.
6. `alice`'s EVM balance is now `0` (tokens burned). The NEAR-side NEP-141 balance is unchanged.
7. `alice` has permanently lost her tokens with no recourse. [1](#0-0)
