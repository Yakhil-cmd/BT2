### Title
Unrestricted `update_rewards` Allows Any Caller to Consume the Per-Block Reward Slot with `disable_rewards: true`, Blocking Legitimate Consensus Reward Distribution - (File: src/staking/staking.cairo)

---

### Summary

The `update_rewards` function in the Staking contract carries no access-control check on the caller. Any address can invoke it with `disable_rewards: true` and a valid `staker_address`. Because `last_reward_block` is written to storage **before** the `disable_rewards` branch is evaluated, the call consumes the single per-block reward slot without distributing any rewards. Every subsequent legitimate call in the same block reverts with `REWARDS_ALREADY_UPDATED`. Repeated across blocks, this permanently freezes consensus-phase reward accrual for all stakers.

---

### Finding Description

`IStakingRewardsManager::update_rewards` is a public entry-point with no role or identity guard:

```cairo
// src/staking/staking.cairo  lines 1448-1500
#[abi(embed_v0)]
impl StakingRewardsManagerImpl of IStakingRewardsManager<ContractState> {
    fn update_rewards(
        ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
    ) {
        self.general_prerequisites();                          // only checks pause flag
        let current_block_number = starknet::get_block_number();
        assert!(
            current_block_number > self.last_reward_block.read(),
            "{}",
            Error::REWARDS_ALREADY_UPDATED,          // ← gate that blocks all other callers
        );
        // ... staker validity checks ...

        // *** last_reward_block written HERE, before the disable_rewards branch ***
        self.last_reward_block.write(current_block_number);  // line 1485

        if disable_rewards || self.is_pre_consensus() {
            return;                                           // line 1487 – exits with NO rewards
        }
        // ... actual reward distribution ...
    }
}
```

The execution order is:

1. Check `current_block_number > last_reward_block` (gate).
2. Validate staker exists and has non-zero balance.
3. **Write `last_reward_block = current_block_number`.**
4. If `disable_rewards == true`, return immediately — no rewards distributed.

Because step 3 precedes step 4, a caller who passes `disable_rewards: true` permanently closes the reward window for the current block. Any subsequent call — including the legitimate one from the consensus mechanism — hits the gate at step 1 and reverts.

The root cause is the absence of any `assert!(get_caller_address() == <authorized_address>)` guard, analogous to how `addSigningKeysOperatorBH()` in the Lido report lacked a restriction that prevented inactive operators from incrementing `KEYS_OP_INDEX_POSITION` and blocking `depositBufferedEther()`. ### Impact Explanation

`update_rewards` is the sole mechanism for distributing consensus-phase block rewards to stakers. If an attacker front-runs every block with `update_rewards(any_valid_staker, true)`:

- `last_reward_block` is set to the current block number.
- The legitimate call (with `disable_rewards: false`) reverts with `REWARDS_ALREADY_UPDATED`.
- No staker receives block rewards for that block.

Sustained across blocks, this constitutes **permanent freezing of unclaimed yield** for all stakers in the consensus phase — matching the "High: Permanent freezing of unclaimed yield" impact category. Even a sporadic attack causes **temporary freezing of unclaimed yield** (also High).

---

### Likelihood Explanation

- **No privilege required.** Any externally-owned address can call `update_rewards`.
- **Staker address is public.** Valid staker addresses are emitted in `NewStaker` events and readable from `stakers` storage.
- **Cost is bounded.** On Starknet, transaction fees are low; calling once per block is economically feasible for a motivated griefer (e.g., a competing validator set).
- **Front-running is straightforward.** The attacker simply submits the griefing transaction with a higher fee in the same block.

---

### Recommendation

Restrict `update_rewards` to a single authorized caller — for example, the attestation contract address stored in `self.attestation_contract`, or a dedicated `REWARDS_MANAGER` role:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
    // Add: assert caller is the authorized rewards manager / attestation contract
    assert!(
        get_caller_address() == self.attestation_contract.read(),
        "{}",
        Error::CALLER_IS_NOT_ATTESTATION_CONTRACT,
    );
    ...
}
```

Alternatively, move the `last_reward_block.write` to after the `disable_rewards` branch so that a no-op call does not consume the per-block slot.

---

### Proof of Concept

```
Block N:
  Attacker tx:  update_rewards(staker=<any_valid_staker>, disable_rewards=true)
    → last_reward_block written to N
    → returns immediately, zero rewards distributed

  Legitimate tx (same block N):  update_rewards(staker=<target_staker>, disable_rewards=false)
    → assert!(N > N)  ← FALSE
    → REVERTS with REWARDS_ALREADY_UPDATED

Block N+1:
  Attacker repeats → same outcome

Result: All stakers are permanently denied consensus block rewards.
```
