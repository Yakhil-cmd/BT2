### Title
Single Malicious Participant Causes Unattributable Permanent Signing Denial via Disabled Cheater-Detection in FROST Aggregation - (File: `src/frost/eddsa/sign.rs`, `src/frost/redjubjub/sign.rs`)

---

### Summary

The FROST EdDSA (`sign_v1`, `sign_v2`) and RedJubjub (`sign`) signing coordinators deliberately pass an empty `verifying_shares` map to `PublicKeyPackage::new()`, disabling per-share verification during `aggregate()`. A single malicious participant can submit an invalid signature share; the coordinator's `aggregate()` call fails, but the coordinator cannot identify which participant submitted the bad share. Because the malicious participant is unidentifiable, honest parties cannot exclude them and retry, resulting in permanent denial of signing.

---

### Finding Description

In both FROST signing implementations, the coordinator collects one signature share from every participant and then calls the upstream `aggregate()` function. The FROST `aggregate()` function supports per-share verification (cheater detection) when a non-empty `verifying_shares` map is supplied inside the `PublicKeyPackage`. This codebase explicitly constructs the package with an empty map:

**`src/frost/eddsa/sign.rs`, `do_sign_coordinator_v1` (lines 159–165):**
```rust
// We supply empty map as `verifying_shares` because we have disabled "cheater-detection" feature flag.
// Feature "cheater-detection" only points to a malicious participant, if there's such.
// It doesn't bring any additional guarantees.
let public_key_package = PublicKeyPackage::new(BTreeMap::new(), vk_package);
let signature = aggregate(&signing_package, &signature_shares, &public_key_package)
    .map_err(|e| ProtocolError::AssertionFailed(e.to_string()))?;
```

The identical pattern appears in `do_sign_coordinator_v2` (lines 215–220) and in `src/frost/redjubjub/sign.rs` `do_sign_coordinator` (lines 174–184).

When `aggregate()` is called with an empty `verifying_shares` map, the upstream FROST library skips the per-share validity check and only verifies the final aggregated signature against the group public key. If any single share is invalid, the final signature fails to verify and `aggregate()` returns an error — but the error carries no information about which participant's share was the cause.

The comment's claim that cheater detection "doesn't bring any additional guarantees" is incorrect in the context of recovery: per-share verification is the only mechanism by which the coordinator can attribute the failure to a specific participant, exclude them, and retry with the remaining honest quorum.

---

### Impact Explanation

**High — Permanent denial of signing for honest parties.**

A single malicious participant (one of the `threshold`-many tolerated by the protocol's security model) submits a syntactically valid but cryptographically incorrect signature share. The coordinator's `aggregate()` call returns `ProtocolError::AssertionFailed` (EdDSA) or `ProtocolError::ErrorFrostAggregation` (RedJubjub). The coordinator and all honest participants receive only a generic failure with no attribution. Because the malicious participant cannot be identified, honest parties cannot form a new signing quorum that excludes the attacker. The attacker can repeat this in every subsequent signing attempt, permanently blocking all signing operations for the honest set.

---

### Likelihood Explanation

**High.** Any participant in a signing session is a reachable, unprivileged attacker entry point. The attack requires no cryptographic break, no key compromise, and no external dependency — only the ability to send an arbitrary scalar as a signature share to the coordinator. The coordinator's `recv_from_others` loop at lines 150–153 (`sign.rs` EdDSA v1) and lines 206–209 (EdDSA v2) accepts any deserialized `SignatureShare` from any participant without pre-validation. The malicious share is inserted directly into `signature_shares` and forwarded to `aggregate()`.

---

### Recommendation

Enable per-share verification by supplying the actual `verifying_shares` map (the per-participant public verification shares derived from the DKG output) when constructing `PublicKeyPackage`. This allows `aggregate()` to identify and report the specific participant whose share is invalid. The coordinator can then exclude that participant and retry signing with the remaining honest quorum, restoring liveness.

Concretely, `KeygenOutput` already carries `public_key` (the group verifying key). The per-participant verifying shares are derivable from the DKG output and should be stored alongside `private_share` so the coordinator can populate `verifying_shares` at signing time.

---

### Proof of Concept

1. Run a FROST EdDSA signing session with participants `{P1, P2, P3}` and threshold 2.
2. `P2` is malicious. In `do_sign_participant_v1`, instead of computing the correct `signature_share` via `round2::sign(...)`, `P2` sends an arbitrary random scalar as its `SignatureShare` to the coordinator.
3. The coordinator (`P1`) collects all shares at lines 150–153 of `src/frost/eddsa/sign.rs` and calls `aggregate()` at line 164 with `PublicKeyPackage::new(BTreeMap::new(), vk_package)`.
4. `aggregate()` computes `s = s1 + s2_bad + s3`, verifies `(R, s)` against the group public key — verification fails — and returns an error.
5. The coordinator receives `ProtocolError::AssertionFailed(...)` with no participant attribution. It cannot distinguish whether `P2` or `P3` submitted the bad share.
