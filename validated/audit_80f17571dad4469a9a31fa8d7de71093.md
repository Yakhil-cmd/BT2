### Title
Gateway Proof-Facts Block-Number/State Mismatch Causes Valid Client-Side-Proving Transactions to Be Rejected - (File: crates/apollo_gateway/src/stateful_transaction_validator.rs)

### Summary

The gateway's stateful validator constructs a `BlockContext` with `block_number = latest_block + 1` (via `unchecked_next()`) but uses the state snapshot from `latest_block`. The `validate_proof_facts` path inside `perform_pre_validation_stage` uses the inflated block number to compute `max_allowed_proof_block_number = (N+1) - STORED_BLOCK_HASH_BUFFER`, which is one higher than what the state actually contains. The subsequent `validate_proof_block_hash` call reads from the same state-N snapshot, where the hash for block `N - STORED_BLOCK_HASH_BUFFER + 1` has not yet been written. The result is that a transaction whose `proof_block_number` equals exactly `N - STORED_BLOCK_HASH_BUFFER + 1` passes the number-range check but fails the hash-lookup check with a spurious "Block hash mismatch" error, even though the same transaction would pass execution in block N+1 (after `pre_process_block` writes that hash).

### Finding Description

**Two-source inconsistency in `run_validate_entry_point`:**

```
// stateful_transaction_validator.rs L323-324
let mut block_info = self.gateway_fixed_block_state_reader.get_block_info().await?;
block_info.block_number = block_info.block_number.unchecked_next();   // → N+1
``` [1](#0-0) 

The state reader, however, is created from the latest committed block N:

```
// stateful_transaction_validator.rs L92-94
let (blockifier_state_reader, ...) = self
    .state_reader_factory
    .get_blockifier_state_reader_and_gateway_fixed_block_from_latest_block()
    ...
``` [2](#0-1) 

`validate_proof_facts` (called from `perform_pre_validation_stage`) uses the inflated block number for the range check:

```rust
// account_transaction.rs L244-250
let max_allowed =
    current_block_number.0.checked_sub(STORED_BLOCK_HASH_BUFFER)...;
// With current_block_number = N+1: max_allowed = N+1 - 10 = N-9
``` [3](#0-2) 

Then immediately reads the hash from state-N:

```rust
// account_transaction.rs L278-279
let stored_block_hash = state
    .get_storage_at(block_hash_contract_address, StorageKey::from(proof_block_number))?;
``` [4](#0-3) 

State-N only contains hashes for blocks `0 … N - STORED_BLOCK_HASH_BUFFER`. The hash for block `N - STORED_BLOCK_HASH_BUFFER + 1` is written by `pre_process_block` at the start of block N+1 execution — which the gateway never runs. So `stored_block_hash` is `Felt::ZERO`, and the check `stored_block_hash != proof_block_hash` fires.

**Contrast with actual execution:** The OS Cairo code and the blockifier's `pre_process_block` write the hash for `current_block - STORED_BLOCK_HASH_BUFFER` before any transaction is executed, so the hash is present when `validate_proof_facts` runs during real block execution. [5](#0-4) 

### Impact Explanation

Any user who generates a client-side SNOS proof anchored to block `N - STORED_BLOCK_HASH_BUFFER + 1` (the newest block whose hash will be available in the next execution block) and submits the resulting Invoke V3 transaction to the gateway will receive a spurious `InvalidProofFacts / Block hash mismatch` rejection. The transaction is structurally valid and would succeed if executed in block N+1, but the gateway permanently rejects it until the user resubmits after block N+1 is committed. This matches the **High** impact category: "Mempool/gateway/RPC admission rejects valid transactions before sequencing."

### Likelihood Explanation

The boundary condition is hit whenever a prover targets the most recent eligible block. Provers naturally want to use the freshest possible block hash (to minimize staleness of their proof). The one-block window is narrow but deterministic and reproducible: any prover that generates a proof for block `N - STORED_BLOCK_HASH_BUFFER + 1` while block N is the chain tip will hit this rejection every time until the next block is committed.

### Recommendation

Align the block number used for `validate_proof_block_number` with the block hashes actually present in the gateway's state snapshot. The simplest fix is to use the un-incremented block number (N) when computing `max_allowed` inside `validate_proof_facts` during gateway validation, or equivalently, to call `pre_process_block` on the cached state before running `validate_proof_facts` so the hash for `N - STORED_BLOCK_HASH_BUFFER + 1` is available. A comment should document the deliberate off-by-one so future refactors do not reintroduce it.

### Proof of Concept

Let `N = 100`, `STORED_BLOCK_HASH_BUFFER = 10`.

1. Chain tip is block 100. State-100 contains block-hash entries for blocks 0–90.
2. User generates a SNOS proof anchored to block 91 (`100 - 10 + 1`).
3. User submits `RpcInvokeTransaction::V3` with `proof_facts.block_number = 91`, `proof_facts.block_hash = H(91)`.
4. Gateway calls `run_validate_entry_point`:
   - `block_info.block_number = 100.unchecked_next() = 101`
   - `max_allowed = 101 - 10 = 91` → number check **passes**
   - `state.get_storage_at(block_hash_contract, key=91)` → returns `Felt::ZERO` (not yet written)
   - `Felt::ZERO != H(91)` → **"Block hash mismatch"** error returned to user
5. If the same transaction were included in block 101 by the batcher, `pre_process_block` would write `H(91)` at key 91 before execution, and `validate_proof_facts` would **pass**. [6](#0-5) [7](#0-6)

    fn validate_proof_facts(
        &self,
        block_context: &BlockContext,
        state: &mut dyn State,
    ) -> TransactionPreValidationResult<()> {
        // Only Invoke V3 transactions can carry proof facts.
        let Transaction::Invoke(invoke_tx) = &self.tx else {
            return Ok(());
        };
        if invoke_tx.version() < TransactionVersion::THREE {
            return Ok(());
        }

        // Parse proof facts.
        let proof_facts = invoke_tx.proof_facts();
        let snos_proof_facts = match ProofFactsVariant::try_from(&proof_facts)
            .map_err(|e| TransactionPreValidationError::InvalidProofFacts(e.to_string()))?
        {
            ProofFactsVariant::Empty => return Ok(()),
            ProofFactsVariant::Snos(snos_proof_facts) => snos_proof_facts,
        };
        let os_constants = &block_context.versioned_constants.os_constants;

        if !os_constants.allowed_proof_versions.contains(&snos_proof_facts.proof_version.as_felt())
        {
            return Err(TransactionPreValidationError::InvalidProofFacts(format!(
                "Proof version {} is not allowed under this protocol version.",
                snos_proof_facts.proof_version
            )));
        }

        // Validate the program hash.
        let allowed = &os_constants.allowed_virtual_os_program_hashes;
        if !allowed.contains(&snos_proof_facts.program_hash) {
            return Err(TransactionPreValidationError::InvalidProofFacts(format!(
                "Virtual OS program hash {} is not allowed",
                snos_proof_facts.program_hash
            )));
        }

        // Validate the block hash and block number.
        let proof_block_hash = snos_proof_facts.block_hash.0;
        let proof_block_number = snos_proof_facts.block_number.0;
        Self::validate_proof_block_number(
            proof_block_number,
            block_context.block_info.block_number,
        )?;
        Self::validate_proof_block_hash(proof_block_hash, proof_block_number, os_constants, state)?;

        // Validate the config hash.
        let virtual_os_config_hash = block_context.virtual_os_config_hash();
        let proof_config_hash = snos_proof_facts.config_hash;
        if virtual_os_config_hash != proof_config_hash {
            return Err(TransactionPreValidationError::InvalidProofFacts(format!(
                "Virtual OS config hash mismatch. Computed virtual OS config hash: \
                 {virtual_os_config_hash}, expected virtual OS config hash: {proof_config_hash}."
            )));
        }

        Ok(())
    }
