### Title
`setPoolAdminFeeDestination` Changes Fee Destination Without Collecting Accrued Fees, Redirecting Owed Admin Fees to New Recipient — (`File: metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`setPoolAdminFeeDestination` updates `poolAdminFeeDestination[pool]` without first calling `collectFees` at the old destination. Every other fee-state transition in the factory (`setPoolAdminFees`, `setPoolProtocolFee`) collects accrued fees before mutating fee configuration. The missing collection step means all fees accrued since the last sweep are silently redirected to the new destination when `collectPoolFees` is next called, causing a direct loss of owed tokens for the old admin fee destination.

---

### Finding Description

`setPoolAdminFees` and `setPoolProtocolFee` both follow the same safe pattern: collect accrued fees at the current rates/destination **before** updating any fee-related state. [1](#0-0) 

`setPoolAdminFeeDestination` does not follow this pattern. It directly overwrites `poolAdminFeeDestination[pool]` with no prior collection: [2](#0-1) 

After the update, the next call to `collectPoolFees` (permissionless — any keeper or bot may call it) passes the **new** destination to `pool.collectFees`: [3](#0-2) 

Inside `collectFees`, the admin share of both spread-surplus and notional-fee tokens is transferred directly to the `adminFeeDestination_` argument supplied by the factory: [4](#0-3) 

All fees that accrued while the old destination was set are therefore swept to the new destination, with no on-chain mechanism to recover them for the original recipient.

---

### Impact Explanation

Any fees accrued between the last `collectPoolFees` call and the `setPoolAdminFeeDestination` call are permanently redirected. If the old admin fee destination is a separate entity from the pool admin (e.g., a DAO treasury, a revenue-sharing contract, or a co-admin), that entity suffers a direct, unrecoverable loss of owed tokens. The magnitude equals the full admin share of all spread surplus and notional fees accumulated in the pool at the time of the destination change.

---

### Likelihood Explanation

The pool admin is semi-trusted and explicitly in scope for fee-destination analysis per the audit pivots. The trigger is a single permissioned call (`setPoolAdminFeeDestination`) that any pool admin can make at any time. Fee accrual is continuous during normal pool operation, so there will almost always be uncollected fees when the destination is changed. No special market conditions or attacker coordination are required.

---

### Recommendation

Add a `collectFees` call before updating the destination, mirroring the pattern used in `setPoolAdminFees`:

```diff
function setPoolAdminFeeDestination(address pool, address newAdminFeeDestination)
    external
    override
    nonReentrant
    onlyPoolAdmin(pool)
  {
    if (newAdminFeeDestination == address(0)) revert InvalidAdminFeeDestination();
+   PoolFeeConfig memory c = poolFeeConfig[pool];
+   IMetricOmmPoolCollectFees(pool).collectFees(
+     c.protocolSpreadFeeE6,
+     c.adminSpreadFeeE6,
+     c.protocolNotionalFeeE8,
+     c.adminNotionalFeeE8,
+     poolAdminFeeDestination[pool]   // old destination
+   );
    poolAdminFeeDestination[pool] = newAdminFeeDestination;
    emit PoolAdminFeeDestinationUpdated(pool, newAdminFeeDestination);
  }
```

---

### Proof of Concept

1. Pool is created; `poolAdminFeeDestination[pool] = treasuryA`.
2. Swaps execute, accruing spread surplus and notional fees inside the pool.
3. Pool admin calls `setPoolAdminFeeDestination(pool, treasuryB)` — no collection occurs; `poolAdminFeeDestination[pool]` is now `treasuryB`.
4. Any address calls `collectPoolFees(pool)`. The factory reads `poolAdminFeeDestination[pool]` → `treasuryB` and passes it to `pool.collectFees`.
5. All admin-share tokens (both spread and notional) accumulated since step 2 are transferred to `treasuryB`.
6. `treasuryA` receives nothing despite being the designated recipient when those fees were earned.
