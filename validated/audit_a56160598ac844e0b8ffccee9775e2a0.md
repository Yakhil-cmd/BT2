### Title
Plaintext `dapp-pairs.yaml` Allows Local Attacker to Inject Bypass Permissions for Signing Commands - (File: packages/gui/src/electron/utils/pairStore.ts)

### Summary
WalletConnect pair records — including the wallet fingerprint, granted commands, and the `bypass` list that controls which commands execute **without user confirmation** — are persisted to a plaintext YAML file (`dapp-pairs.yaml`) with no encryption, MAC, or integrity protection. A local attacker with access to an unlocked workstation can directly edit this file to add high-impact commands such as `chia_sendTransaction` or `chia_takeOffer` to the `bypass` array of an existing pair. Because `dispatchPairRequest` trusts the `bypass` list from the file without re-validating that each entry is actually allowed to skip confirmation, the attacker's dapp can subsequently execute those commands silently, without any approval dialog shown to the wallet owner.

### Finding Description

**Unprotected persistence layer**

`pairStore.ts` writes all pair records — including `fingerprint`, `commands`, and `bypass` — to `dapp-pairs.yaml` via `yamlUtils.writeData`, which calls `fs.writeFileSync(path, yamlData, { encoding: 'utf-8' })` with no encryption or integrity seal: [1](#0-0) [2](#0-1) 

**Bypass list is trusted verbatim**

`dispatchPairRequest` reads the pair from the file and, at line 57, executes the command immediately if it appears in `pair.bypass` — with no check that the command is actually in the application's `allowConfirmationBypass` whitelist: [3](#0-2) 

**Application-level bypass guard is only enforced on the write path**

`addDappBypassPermissions` calls `canBypassCommand` to reject commands that lack `allowConfirmationBypass: true` (e.g., `chia_sendTransaction`): [4](#0-3) 

But this guard is only invoked during the `chia_requestPermissions` flow. It is never re-applied when `dispatchPairRequest` reads the bypass list from disk. A file-level edit completely bypasses this guard.

**`chia_sendTransaction` is not allowed to bypass confirmation by design**

`Commands.ts` defines `chia_sendTransaction` without `allowConfirmationBypass`: [5](#0-4) 

Compare with read-only commands that do carry `allowConfirmationBypass: true`: [6](#0-5) 

### Impact Explanation

An attacker who edits `dapp-pairs.yaml` to add `chia_sendTransaction` (or `chia_takeOffer`, `chia_createOfferForIds`, etc.) to the `bypass` array of a pair they control gains the ability to execute those commands against the victim's wallet **without any confirmation dialog**. This constitutes a bypass of WalletConnect signing approval with direct asset-loss impact (unauthorized XCH/CAT/NFT transfer or offer acceptance).

### Likelihood Explanation

The attack requires:
1. A local attacker with access to an unlocked workstation (the classic "Eve at Alice's desk" scenario from the original report).
2. An existing WalletConnect pairing between Alice's wallet and a dapp the attacker controls (e.g., a previously connected dapp, or one Alice was socially engineered into connecting).

Both conditions are realistic for a targeted local attack. The file is a human-readable YAML file in a well-known user-data directory, trivially editable with any text editor or a one-line shell command.

### Recommendation

1. **Short term**: In `dispatchPairRequest`, re-validate that every command in `pair.bypass` is actually `allowConfirmationBypass`-eligible before skipping confirmation. This closes the integrity gap without requiring filesystem changes:
   ```typescript
   if (pair.bypass.includes(command) && canBypassCommand(command)) {
     return process(context);
   }
   ```
2. **Long term**: Protect `dapp-pairs.yaml` with an HMAC or digital signature keyed to a secret stored in the platform's secure credential store (macOS Keychain, Windows Credential Manager, Linux Secret Service). Reject the file on load if the MAC does not verify. This is the direct analog of the original report's recommendation to use platform encryption for data at rest.

### Proof of Concept

```bash
# 1. Locate the file (path varies by OS/app name)
PAIRS_FILE="$HOME/Library/Application Support/Chia Blockchain/dapp-pairs.yaml"

# 2. Inspect existing pairs
cat "$PAIRS_FILE"
# Output (example):
# pairs:
#   - topic: abc123...
#     fingerprint: 1234567890
#     commands: [chia_getWallets, chia_getWalletBalance]
#     bypass: [chia_getWallets]
#     ...

# 3. Inject chia_sendTransaction into bypass and commands
python3 - <<'EOF'
import yaml, sys
with open(sys.argv[1]) as f:
    data = yaml.safe_load(f)
pair = data['pairs'][0]
if 'chia_sendTransaction' not in pair['commands']:
    pair['commands'].append('chia_sendTransaction')
if 'chia_sendTransaction' not in pair['bypass']:
    pair['bypass'].append('chia_sendTransaction')
with open(sys.argv[1], 'w') as f:
    yaml.dump(data, f)
print("Injected.")
EOF "$PAIRS_FILE"

# 4. From the attacker's dapp (already paired), send:
#    chia_sendTransaction { address: <attacker_address>, amount: <all_funds>, fee: 0 }
#    → dispatchPairRequest sees bypass includes 'chia_sendTransaction',
#      skips confirm(), calls process() directly — no dialog shown to Alice.
```

The `pairRecordSchema` validation in `pairStore.load()` only checks field types, not semantic validity of bypass entries, so the modified file loads without error. [7](#0-6) [8](#0-7)

function load(): PairRecord[] {
  if (cache) {
    return cache;
  }

  const data = readData(getPath());

  const pairRecords: PairRecord[] = [];

  if (data?.pairs && Array.isArray(data.pairs)) {
    for (const pair of data.pairs) {
      try {
        const record = pairRecordSchema.parse(pair);
        pairRecords.push(record);
      } catch (error) {
        console.error(`Invalid pair record: ${pair}`, error);
      }
    }
  }

  cache = pairRecords;
  return pairRecords;
}
