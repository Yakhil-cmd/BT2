### Title
Silo Deploy Whitelist Bypassed via `if-else` in `assert_access` When `transaction.to` Is Set - (File: `engine/src/engine.rs`)

### Summary

The `assert_access` function in `engine/src/engine.rs` uses a mutually exclusive `if-else` branch to decide whether to check submit permissions or deploy permissions. Because the two checks are exclusive, a user who is only on the submit whitelist (`Account` + `Address`) but **not** on the deploy whitelist (`Admin` + `EvmAdmin`) can bypass the deploy restriction entirely: they submit a call transaction (`to = Some(factory_contract)`) to any already-deployed contract that internally executes `CREATE` or `CREATE2`, causing unauthorized contract deployment inside the Silo EVM without ever triggering the deploy whitelist check.

### Finding Description

**Root cause — `engine/src/engine.rs`, `assert_access`, lines 1761–1765:**

```rust
fn assert_access<I: IO + Copy, E: Env>(
    io: &I,
    env: &E,
    transaction: &NormalizedEthTransaction,
) -> Result<(), EngineError> {
    let allowed = if transaction.to.is_some() {
        silo::is_allow_submit(io, &env.predecessor_account_id(), &transaction.address)
    } else {
        silo::is_allow_deploy(io, &env.predecessor_account_id(), &transaction.address)
    };
    ...
}
``` [1](#0-0) 

The two whitelist families are entirely separate:

- **Submit path** (`to` is `Some`): checks `WhitelistKind::Account` (NEAR account) and `WhitelistKind::Address` (EVM address).
- **Deploy path** (`to` is `None`): checks `WhitelistKind::Admin` (NEAR account) and `WhitelistKind::EvmAdmin` (EVM address). [2](#0-1) 

`assert_access` is the **only** access-control gate called before EVM execution: [3](#0-2) 

After the gate, execution branches again on `transaction.to`: [4](#0-3) 

When `to` is `Some`, `engine.call(...)` is invoked. SputnikVM then executes the callee's bytecode, which may contain `CREATE`/`CREATE2` opcodes. Those opcodes deploy new contracts **inside the EVM** with no further whitelist check. The deploy whitelist (`Admin`/`EvmAdmin`) is never consulted for this path.

**Structural analog to the reported bug:** The original bug used `if deposit > 0 { check_level(1) } else { check_level(2) }`, so including a deposit skipped the borrow/withdraw check. Here, setting `transaction.to = Some(...)` skips the deploy check entirely — the same `if-else` anti-pattern applied to a two-tier whitelist.

### Impact Explanation

In Silo Mode, the `Admin`/`EvmAdmin` whitelists exist precisely to restrict which parties may deploy EVM contracts. An attacker who is only on the submit whitelist can:

1. Call any pre-existing factory contract (e.g., a Uniswap-style pair factory, or any contract the Silo operator already deployed) that internally calls `CREATE`/`CREATE2`.
2. Cause arbitrary new contracts to be deployed inside the Silo EVM without being on the deploy whitelist.

The deployed contract can be attacker-controlled bytecode (if the factory accepts arbitrary init-code), enabling the attacker to deploy a malicious contract that steals funds from users who interact with it, or manipulates ERC-20 token accounting within the Silo. This constitutes a **Silo deploy whitelist bypass** with downstream potential for direct theft of user funds.

**Impact: High — Silo whitelist/access-control bypass enabling unauthorized contract deployment, with realistic downstream path to fund theft.**

### Likelihood Explanation

- The attacker only needs to be on the submit whitelist (`Account` + `Address`), which is the lower-privilege tier intended for ordinary Silo users.
- Any Silo deployment that has at least one factory-style contract already deployed (extremely common in DeFi Silos) provides the necessary vehicle.
- No privileged keys, no governance capture, and no external oracle is required.
- The attacker submits a standard EVM call transaction — a normal, unprivileged operation.

**Likelihood: High.**

### Recommendation

Replace the `if-else` with independent checks so that both submit and deploy permissions are always evaluated when applicable:

```rust
fn assert_access<I: IO + Copy, E: Env>(
    io: &I,
    env: &E,
    transaction: &NormalizedEthTransaction,
) -> Result<(), EngineError> {
    // Always check submit permissions for any transaction.
    if !silo::is_allow_submit(io, &env.predecessor_account_id(), &transaction.address) {
        return Err(EngineError { kind: EngineErrorKind::NotAllowed, gas_used: 0 });
    }
    // Additionally check deploy permissions when the transaction is a deployment.
    if transaction.to.is_none()
        && !silo::is_allow_deploy(io, &env.predecessor_account_id(), &transaction.address)
    {
        return Err(EngineError { kind: EngineErrorKind::NotAllowed, gas_used: 0 });
    }
    Ok(())
}
```

For complete protection against `CREATE`/`CREATE2` bypass via call transactions, the Silo should also intercept contract-creation opcodes at the EVM level (e.g., via a custom SputnikVM handler) and apply the deploy whitelist check there as well.

### Proof of Concept

**Setup (Silo Mode, all whitelists enabled):**

1. Silo operator adds attacker's NEAR account to `WhitelistKind::Account` and attacker's EVM address to `WhitelistKind::Address` (submit whitelist only).
2. Attacker's NEAR account and EVM address are **not** in `WhitelistKind::Admin` / `WhitelistKind::EvmAdmin`.
3. A factory contract `F` (e.g., a minimal contract whose bytecode is `CREATE`-based) was previously deployed by an authorized deployer.

**Attack:**

```
attacker submits EVM transaction:
  to   = address(F)          // Some(...) → triggers is_allow_submit check only
  data = abi.encode(malicious_init_code)
```

**Execution trace:**

1. `assert_access` sees `transaction.to.is_some()` → calls `is_allow_submit` → **passes** (attacker is on submit whitelist).
2. `is_allow_deploy` is **never called**.
3. `engine.call(sender, F, ...)` executes `F`'s bytecode.
4. `F` executes `CREATE(malicious_init_code)` → new malicious contract deployed at a deterministic address.
5. Attacker's malicious contract now lives in the Silo EVM, bypassing the `Admin`/`EvmAdmin` deploy whitelist entirely. [5](#0-4) [6](#0-5) [7](#0-6)

}

fn is_account_allowed_deploy<I: IO + Copy>(io: &I, account_id: &AccountId) -> bool {
    let list = Whitelist::init(io, WhitelistKind::Admin);
    !list.is_enabled() || list.is_exist(account_id)
}

fn is_address_allowed_deploy<I: IO + Copy>(io: &I, address: &Address) -> bool {

}

fn is_address_allowed<I: IO + Copy>(io: &I, address: &Address) -> bool {

}

fn is_account_allowed<I: IO + Copy>(io: &I, account: &AccountId) -> bool {
