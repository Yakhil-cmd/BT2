Audit Report

## Title
SNS Swap `refresh_buyer_token_e8s` TOCTOU Race Condition Locks Excess ICP Until Swap Closes - (File: `rs/sns/swap/src/swap.rs`)

## Summary
The `refresh_buyer_token_e8s` function in the SNS Swap canister makes an inter-canister `await` call to the ICP ledger to read a buyer's subaccount balance. During this yield point, concurrent participants can consume the remaining swap capacity, causing the resuming caller's accepted ICP amount to be far less than what was transferred. The excess ICP is locked in the buyer's subaccount of the swap canister with no immediate refund path — recovery via `error_refund_icp` is gated behind the swap reaching COMMITTED or ABORTED state. The code explicitly acknowledges this with `TODO(NNS1-1682): attempt to refund ICP that cannot be accepted.`

## Finding Description
The two-step participation flow requires a buyer to transfer ICP to their personal subaccount of the swap canister, then call `refresh_buyer_token_e8s` to register the participation.

Inside `refresh_buyer_token_e8s`, the function performs a pre-await lifecycle check at L1144–1147, then suspends at the inter-canister `await` to the ICP ledger at L1158–1162:

```rust
icp_ledger
    .account_balance(account)
    .await
    .map_err(|x| x.to_string())?
    .get_e8s()
```

This is a mandatory yield point on the Internet Computer. While Alice's call is suspended, Bob can submit his own `refresh_buyer_token_e8s` call, which runs to completion and updates `direct_participation_icp_e8s` in the swap's state. When Alice's call resumes, `available_direct_participation_e8s()` is recomputed from the now-updated state at L1177:

```rust
let max_increment_e8s = self.available_direct_participation_e8s();
```

The accepted increment is then capped at L1223–1225:

```rust
let requested_increment_e8s = e8s - old_amount_icp_e8s;
let actual_increment_e8s = std::cmp::min(max_increment_e8s, requested_increment_e8s);
let new_balance_e8s = old_amount_icp_e8s.saturating_add(actual_increment_e8s);
```

If `new_balance_e8s` falls below `min_participant_icp_e8s`, the call returns an `Err` at L1241–1245, and Alice's full transferred balance remains locked in her subaccount. Even in the variant where a tiny amount is accepted, the excess ICP is locked with no immediate refund.

The only recovery path, `error_refund_icp`, is explicitly gated at L1932–1935:

```rust
if !(self.lifecycle() == Lifecycle::Aborted || self.lifecycle() == Lifecycle::Committed) {
    return ErrorRefundIcpResponse::new_precondition_error(
        "Error refunds can only be performed when the swap is ABORTED or COMMITTED",
    );
}
```

The post-await lifecycle rechecks at L1168–1171 guard only against the swap closing, not against concurrent capacity consumption — they do not prevent the race condition.

## Impact Explanation
A buyer who transfers a large amount of ICP (e.g., 500,000 ICP) may have only a tiny fraction accepted or be rejected entirely due to concurrent participation. The excess ICP is locked in the swap canister's subaccount for the entire duration of the swap — potentially days or weeks — with no immediate remedy. The buyer cannot recover the locked ICP until the swap finalizes. This constitutes a significant SNS security impact with concrete user harm: large capital is rendered inaccessible, the buyer receives far fewer SNS tokens than intended, and no staking rewards accrue on the locked ICP during the lockup period. This maps to **High ($2,000–$10,000)**: significant SNS security impact with concrete user or protocol harm. It does not reach Critical because the ICP is not permanently lost — it is recoverable after swap finalization.

## Likelihood Explanation
SNS decentralization swaps are high-demand, time-limited events with many concurrent participants. The inter-canister `await` to the ICP ledger is mandatory and unavoidable in the current design. No special privileges are required — any unprivileged ingress sender can call `refresh_buyer_token_e8s`. The scenario is most likely near the end of a swap when remaining capacity is small and competition is highest. The race can occur naturally without any malicious actor; it is an inherent consequence of the async execution model.

## Recommendation
Implement the already-acknowledged fix referenced by `TODO(NNS1-1682)`: within `refresh_buyer_token_e8s`, after computing `new_balance_e8s` and `actual_increment_e8s`, immediately refund the excess ICP (`e8s - new_balance_e8s`) back to the buyer's principal account on the ICP ledger before returning. This ensures the amount of ICP locked in escrow always equals the accepted participation amount, eliminating the conservation mismatch. Alternatively, reserve the buyer's capacity before the `await` and release it if the balance check fails.

## Proof of Concept
**Setup**: SNS swap is OPEN. `max_direct_participation_icp_e8s = 500_000 * E8`. `min_participant_icp_e8s = 1 * E8`. Current participation = `499_999 * E8` (1 ICP remaining capacity).

1. Alice transfers `500_000 * E8` ICP to `subaccount(swap_canister, Alice)` on the ICP ledger.
2. Alice calls `refresh_buyer_token_e8s`. Pre-await lifecycle checks pass. The function calls `icp_ledger.account_balance(Alice's subaccount)` — **yield point**.
3. During the yield, Bob calls `refresh_buyer_token_e8s` with a balance of `1 * E8` ICP. Bob's call completes atomically, consuming the last `1 * E8` of capacity. `available_direct_participation_e8s()` is now `0`.
4. Alice's call resumes. `max_increment_e8s = available_direct_participation_e8s() = 0`.
5. `actual_increment_e8s = min(0, 500_000 * E8) = 0`. `new_balance_e8s = 0`.
6. The check `new_balance_e8s < min_participant_icp_e8s` triggers at L1241; Alice's call returns `Err`.
7. Alice's `500_000 * E8` ICP remains locked in `subaccount(swap_canister, Alice)`.
8. Alice calls `error_refund_icp` — rejected with "Error refunds can only be performed when the swap is ABORTED or COMMITTED."

A deterministic integration test using PocketIC can reproduce this by interleaving two `refresh_buyer_token_e8s` calls with a mock ledger that introduces a delay on Alice's balance response, allowing Bob's call to complete first and exhaust remaining capacity before Alice's call resumes.
