Audit Report

## Title
Unbounded Synchronous Loop in `create_sns_neuron_recipes` Can Permanently Strand Finalization Lock After Await Boundary - (`File: rs/sns/swap/src/swap.rs`)

## Summary
`finalize_inner` acquires a `finalize_swap_in_progress` lock and crosses at least two `await` boundaries (at `sweep_icp` and `settle_neurons_fund_participation`) before calling `create_sns_neuron_recipes` synchronously. Each `await` is an IC message boundary that commits the lock to stable state. If `create_sns_neuron_recipes` exhausts the IC instruction budget and traps, the IC rolls back only the current message's state changes — the lock, committed in a prior message, remains set. All subsequent calls to `finalize_swap` return immediately with an error, permanently blocking finalization without a canister upgrade.

## Finding Description

`finalize` acquires the lock at line 1506 and delegates to `finalize_inner`: [1](#0-0) 

The code itself acknowledges the lock-stuck risk for panics at line 1528–1530: [2](#0-1) 

Inside `finalize_inner`, two `await` points precede the synchronous call to `create_sns_neuron_recipes`: [3](#0-2) 

After the first `await` at line 1558, the lock is committed to stable state and cannot be rolled back by a later trap. `create_sns_neuron_recipes` is then called synchronously at line 1588 and contains two unbounded loops — one over all direct buyers and one over all Neurons' Fund participants × their `cf_neurons`: [4](#0-3) [5](#0-4) [6](#0-5) 

There are zero `is_over_instructions_limit()` calls anywhere in `rs/sns/swap/src/` — confirmed by grep. The pattern exists in NNS governance (`rs/nns/governance/src/reward/distribution.rs`, `rs/nns/governance/src/voting.rs`) but was never applied to the swap canister.

The participant cap enforced at participation time is a **memory bound**, not an instruction-count bound: [7](#0-6) 

The integration test confirms the ceiling is sized for memory, not instruction cost: [8](#0-7) 

A secondary unbounded synchronous scan exists in `claim_swap_neurons` (lines 1675–1697), which builds a full `claimable_neurons_index` over all `neuron_recipes` before its first `await`, presenting the same lock-stuck risk after three prior `await` boundaries: [9](#0-8) 

## Impact Explanation

If `create_sns_neuron_recipes` traps due to instruction exhaustion: (1) the IC rolls back only the current message's state changes; (2) the `finalize_swap_in_progress = true` lock, committed in a prior message, is not rolled back; (3) every subsequent call to `finalize_swap` hits the lock check at line 1506 and returns immediately with an error; (4) the swap is permanently stuck in `Lifecycle::Committed`; (5) direct participants cannot receive their SNS neurons, and ICP already swept to SNS governance cannot be reclaimed. Recovery requires a canister upgrade to clear the lock. This is a significant SNS platform-level DoS with concrete, irreversible user harm — matching the **High** impact tier for application/platform-level DoS and significant SNS security impact with concrete user or protocol harm.

## Likelihood Explanation

`finalize_swap` is a public update call with no access control — any principal can invoke it. The trigger condition is a swap near `MAX_NEURONS_FOR_DIRECT_PARTICIPANTS` direct participants combined with large Neurons' Fund participation (up to `MAX_NEURONS_FUND_PARTICIPANTS = 5,000`). High-profile SNS launches routinely attract large participant counts. The participant cap was designed to bound memory, not instruction cost; no benchmarks or instruction-limit guards protect the synchronous path. The `MAX_NEURONS_FUND_PARTICIPANTS` constant's own documentation notes instruction-cost concerns for NNS governance operations but no analogous analysis was done for the swap canister: [10](#0-9) 

## Recommendation

1. Add `is_over_instructions_limit()` checks inside both loops in `create_sns_neuron_recipes`, persisting partial progress via the existing `has_created_neuron_recipes` idempotency flag and returning a partial `SweepResult` so the caller can retry.
2. Apply the same chunked-processing pattern to the `claim_swap_neurons` pre-loop (lines 1675–1697), splitting index construction across multiple messages if needed.
3. Implement the post-upgrade hook that unconditionally clears `finalize_swap_in_progress` — the comment at line 1528 acknowledges this need but it is not implemented.
4. Add an instruction-count benchmark for `create_sns_neuron_recipes` at maximum participant counts, analogous to the existing memory-bound test in `constraints_dependencies.rs`.

## Proof of Concept

1. Launch an SNS swap with `min_participant_icp_e8s` at minimum and `neuron_basket_construction_parameters.count` set to a large value (e.g., 10).
2. Fill the swap to near `MAX_NEURONS_FOR_DIRECT_PARTICIPANTS` direct participants and ensure maximum NF participation (`MAX_NEURONS_FUND_PARTICIPANTS = 5,000`).
3. Allow the swap to reach `Lifecycle::Committed`.
4. Call `finalize_swap` from any principal.
5. `sweep_icp` completes → lock is committed to stable state across the `await` boundary at line 1558.
6. `create_sns_neuron_recipes` is called synchronously over the full participant set → instruction limit trap.
7. Observe: `finalize_swap_in_progress = true` persists; all subsequent `finalize_swap` calls return `"Finalize swap is already in progress"`.
8. Swap is permanently stuck; participant SNS neurons cannot be distributed without a canister upgrade.

A deterministic integration test using PocketIC can reproduce this by configuring a swap at maximum participant counts and verifying that after a simulated instruction-limit trap in `create_sns_neuron_recipes`, the lock field remains set and `finalize_swap` returns the lock-held error on all subsequent calls.
