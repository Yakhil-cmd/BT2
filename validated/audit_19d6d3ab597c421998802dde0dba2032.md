### Title
Unchecked `sender` Argument in Bank Precompile `transfer` (and `burn`) Enables Unauthorized Token Drain — (File: `x/cronos/keeper/precompiles/bank.go`)

---

### Summary

The bank precompile's `transfer` and `burn` methods accept a caller-supplied `sender`/`address` argument without verifying it equals `contract.Caller()`. Any EVM contract can therefore move or destroy `evm/<callerContract>` native tokens held by any victim address, with no authorization from the token holder.

---

### Finding Description

In `x/cronos/keeper/precompiles/bank.go`, the `TransferMethodName` case unpacks `sender` directly from ABI-encoded call data and passes it as the `from` address to `bankKeeper.SendCoins`:

```go
sender := args[0].(common.Address)   // fully attacker-controlled
recipient := args[1].(common.Address)
// ...
from := sdk.AccAddress(sender.Bytes())
denom := EVMDenom(contract.Caller())  // denom is tied to calling contract
// ...
bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt))
```

There is **no check** that `sender == contract.Caller()`. [1](#0-0) 

The identical pattern exists in the `BurnMethodName` branch, where `recipient` (the address to burn *from*) is also caller-supplied and never validated:

```go
recipient := args[0].(common.Address)   // attacker-controlled burn target
// ...
bc.bankKeeper.SendCoinsFromAccountToModule(ctx, addr, types.ModuleName, ...)
bc.bankKeeper.BurnCoins(ctx, types.ModuleName, ...)
``` [2](#0-1) 

The denom is always `EVMDenomPrefix + contract.Caller().Hex()` (`"evm/<callerAddress>"`), so the scope of the drain is limited to tokens whose denom corresponds to the calling contract. [3](#0-2) 

---

### Impact Explanation

**Critical** — unauthorized transfer/burn of precompile-controlled assets.

Any EVM contract can call `bankPrecompile.transfer(victimAddress, attackerAddress, amount)` and the Cosmos bank module will execute `SendCoins(victim → attacker)` for denom `evm/<attackerContract>` with no signature or approval from the victim. The same contract can call `burn(victimAddress, amount)` to destroy the victim's balance outright. Both operations are irreversible on-chain.

This maps directly to the allowed Critical impact: *"Unauthorized … transfer … or balance/accounting change for … precompile-controlled assets."*

---

### Likelihood Explanation

**Medium.** The prerequisite is that the victim holds `evm/<attackerContract>` tokens. This requires the attacker's contract to have previously minted or transferred those tokens to the victim (e.g., via a fake airdrop, a DeFi interaction, or a legitimate-looking token contract). Once any victim holds such a balance, the drain is trivially executable by the same contract with a single call to the precompile. No privileged access, leaked keys, or governance action is required.

---

### Recommendation

Enforce that the `sender` argument in `transfer` equals `contract.Caller()`, and that the `recipient` argument in `burn` equals `contract.Caller()`:

```go
// transfer
if sender != contract.Caller() {
    return nil, errors.New("sender must be the calling contract")
}

// burn
if recipient != contract.Caller() {
    return nil, errors.New("burn target must be the calling contract")
}
```

This mirrors the correct pattern already used for `mint`, where the denom is derived from `contract.Caller()` and the operation is implicitly scoped to the calling contract's own token supply. [4](#0-3) 

---

### Proof of Concept

1. Attacker deploys `MaliciousToken` contract on Cronos EVM.
2. `MaliciousToken` calls `bankPrecompile.mint(victim, 1_000_000)` — victim now holds `1_000_000 evm/<MaliciousToken>` tokens (e.g., distributed as a "free airdrop").
3. Attacker calls `MaliciousToken.drain(victim)`, which internally calls:
   ```solidity
   IBankModule(0x0000...0064).transfer(victim, attacker, 1_000_000);
   ```
4. The bank precompile executes `SendCoins(victim → attacker, 1_000_000 evm/<MaliciousToken>)` with no check that `victim` authorized the transfer.
5. Victim's entire balance is drained. The same call with `burn(victim, amount)` destroys the balance instead.

The attacker is fully unprivileged; the only requirement is that the victim holds a balance of the attacker's contract denom, which the attacker can engineer through any token distribution mechanism.
