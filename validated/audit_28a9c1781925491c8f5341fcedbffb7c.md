### Title
Commission Front-Running via `set_commission_commitment` Allows Staker to Steal Delegator Yield — (File: `src/staking/staking.cairo`)

---

### Summary

A staker who has an active `CommissionCommitment` can atomically raise their commission to `max_commission` (up to 100 %) in the same block that a delegator submits `enter_delegation_pool`. Because `enter_delegation_pool` accepts no `max_commission` guard parameter, the delegator has no slippage protection and can be forced into a pool that immediately takes all of their yield.

---

### Finding Description

The protocol normally prevents commission increases: without a commitment, `set_commission` enforces `commission < old_commission`. [1](#0-0) 

However, `set_commission_commitment` lets a staker pre-register a `max_commission` that is **greater than or equal to** the current commission, explicitly unlocking upward movement. [2](#0-1) 

While the commitment is active, `update_commission` (called by `set_commission`) only requires `commission <= commitment.max_commission`, so the staker can jump to any value up to 100 % in a single transaction. [3](#0-2) 

The codebase itself acknowledges this gap with an explicit developer note: [4](#0-3) 

On the delegator side, `enter_delegation_pool` takes only `reward_address` and `amount`; there is no `max_commission` parameter that would let the caller bound the commission they are willing to accept. [5](#0-4) [6](#0-5) 

The same absence applies to `add_to_delegation_pool`, which existing members use to increase their stake. [7](#0-6) 

---

### Impact Explanation

**High — Theft of unclaimed yield.**

A staker who raises commission to 10 000 (100 %) before a delegator's entry transaction is included captures every unit of yield that would otherwise belong to the delegator. The delegator's principal is safe (they can exit), but all accrued rewards flow to the staker's reward address for as long as the delegator remains in the pool. This is a direct, quantifiable theft of unclaimed yield from an unprivileged pool member.

---

### Likelihood Explanation

**Medium.**

The attack requires two preconditions that are both publicly observable on-chain:

1. The staker must have called `set_commission_commitment` with a high `max_commission` (e.g., 10 000). This event is emitted and visible. [8](#0-7) 

2. A delegator must submit `enter_delegation_pool` or `add_to_delegation_pool` while that commitment is active.

Starknet's sequencer currently processes transactions in a deterministic order that is observable before finality, making mempool-level front-running feasible for a staker who monitors pending transactions. The staker is the only party who benefits and has the necessary key, so the attack is targeted and rational.

---

### Recommendation

1. **Add a `max_commission` guard to `enter_delegation_pool` and `add_to_delegation_pool`.** The pool contract should revert if the current commission exceeds the caller-supplied bound at the time of execution, analogous to a DEX slippage limit.

2. **Enforce a time-delay on commission increases.** Rather than allowing an immediate jump to `max_commission`, require that any increase takes effect only at the start of the next epoch, giving delegators time to exit.

3. **Alternatively, prohibit commission increases entirely** (remove the `set_commission_commitment` mechanism or cap `max_commission` at the current commission), which is the simplest fix and eliminates the attack surface.

---

### Proof of Concept

```
Epoch N:
  Staker calls set_commission_commitment(max_commission=10000, expiration_epoch=N+1)
  // CommissionCommitmentSet event is emitted; current commission = 500 (5%)

Epoch N (same block, staker monitors mempool):
  Delegator submits: enter_delegation_pool(reward_address=D, amount=1_000_000)

  Staker front-runs with: set_commission(commission=10000)
  // update_commission passes: 10000 <= commitment.max_commission (10000) ✓
  // Commission is now 100%

  Delegator's transaction executes:
  // assert_staker_is_active() passes
  // No commission check — delegator joins at 100% commission

Epoch N+1 onward:
  Staker attests; rewards are distributed.
  Pool receives 0 STRK (100% taken as commission by staker).
  Delegator's reward_address receives nothing.
  Delegator must call exit_delegation_pool_intent + wait exit_wait_window to recover principal.
```

The root cause is the combination of:
- `set_commission_commitment` permitting `max_commission > current_commission` [2](#0-1) 
- `update_commission` allowing immediate jumps to `max_commission` within the same block [3](#0-2) 
- `enter_delegation_pool` providing no commission bound for the caller [9](#0-8)

        fn enter_delegation_pool(
            ref self: ContractState, reward_address: ContractAddress, amount: Amount,
        ) {
            // Asserts.
            self.assert_staker_is_active();
            let pool_member = get_caller_address();
            assert!(
                self.pool_member_info.read(pool_member).is_none(), "{}", Error::POOL_MEMBER_EXISTS,
            );
            assert!(amount.is_non_zero(), "{}", GenericError::AMOUNT_IS_ZERO);
            let token_dispatcher = self.token_dispatcher.read();
            let token_address = token_dispatcher.contract_address;
            assert!(token_address != pool_member, "{}", Error::POOL_MEMBER_IS_TOKEN);
            assert!(token_address != reward_address, "{}", GenericError::REWARD_ADDRESS_IS_TOKEN);
            // Transfer funds from the delegator to the staking contract.
            let staker_address = self.staker_address.read();
            transfer_from_delegator(:pool_member, :amount, :token_dispatcher);
            self.transfer_to_staking_contract(:amount, :token_dispatcher, :staker_address);

            self.set_member_balance(:pool_member, :amount);

            // Create the pool member record.
            self
                .pool_member_info
                .write(pool_member, VInternalPoolMemberInfoTrait::new_latest(:reward_address));

            // Emit events.
            self
                .emit(
                    Events::NewPoolMember { pool_member, staker_address, reward_address, amount },
                );
            self
                .emit(
                    Events::PoolMemberBalanceChanged {
                        pool_member, old_delegated_stake: Zero::zero(), new_delegated_stake: amount,
                    },
                );
        }
