### Title
Gateway Skips `__validate__` Signature Check for Invoke Transactions When Any Pending Transaction Exists for the Target Account — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

`skip_stateful_validations` bypasses the `__validate__` entry-point (the only on-chain signature check at the gateway) for an invoke transaction with nonce=1 whenever `account_tx_in_pool_or_recent_block` returns `true` for the sender address. That helper returns `true` as soon as **any** transaction from that address is in the pool — including the legitimate user's own pending `deploy_account`. An unprivileged attacker who observes a pending `deploy_account` for address `A` can immediately submit an invoke with nonce=1 from `A` carrying an arbitrary/empty signature, and the gateway will admit it to the mempool without ever verifying the signature.

---

### Finding Description

**Broken invariant:** Every invoke transaction admitted by the gateway must have a valid signature (or be a `deploy_account` whose constructor runs `__validate_deploy__`).

**Root cause — `skip_stateful_validations`:** [1](#0-0) 

The function returns `true` (skip) when all three conditions hold:
1. The transaction is an `Invoke` with `nonce == 1`.
2. The on-chain account nonce is `0` (account not yet deployed).
3. `account_tx_in_pool_or_recent_block(sender)` returns `true`.

When `skip_validate = true`, `run_validate_entry_point` sets `execution_flags.validate = false`: [2](#0-1) 

Inside `StatefulValidator::perform_validations`, the `__validate__` call is guarded by that flag and is entirely skipped: [3](#0-2) 

**The over-broad membership check:**

`account_tx_in_pool_or_recent_block` returns `true` if the address has **any** transaction in the pool — not specifically a `deploy_account`: [4](#0-3) 

`tx_pool.contains_account` is address-only with no transaction-type filter: [5](#0-4) 

The comment in `skip_stateful_validations` acknowledges the reasoning: "it means that either it has a deploy_account transaction or transactions with future nonces that passed validations." The second branch of that disjunction is sound (future-nonce txs must have passed `__validate__`), but the first branch is not: the deploy_account itself is in the pool, so `contains_account` is `true` before the account exists on-chain, and the attacker's nonce=1 invoke is the very first transaction that exploits the skip.

**No other guard catches the missing signature:**

`validate_by_mempool` only checks for duplicate tx hash and nonce ordering — it never touches the signature: [6](#0-5) 

The stateless validator checks only signature **length**, not cryptographic validity: [7](#0-6) 

---

### Impact Explanation

An attacker can inject an invoke transaction with an arbitrary/empty signature into the mempool for any address that has a pending `deploy_account`. The malicious nonce=1 invoke occupies the nonce=1 slot for that address. The legitimate user's own nonce=1 invoke is then rejected by the mempool as a duplicate nonce (unless the user pays a higher fee to escalate). The attacker can repeat this continuously, permanently blocking the first post-deploy invoke for any victim account they choose to target. This matches the **High** impact category: "Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."

---

### Likelihood Explanation

- Pending `deploy_account` transactions are publicly observable in the mempool.
- The attack requires no special privileges, no stake, and no cryptographic material.
- The attacker only needs to submit a well-formed invoke (valid resource bounds, nonce=1, any signature bytes that pass the length check) targeting the victim's address.
- The attack can be automated and applied to every new `deploy_account` seen in the mempool.

---

### Recommendation

Replace the coarse `account_tx_in_pool_or_recent_block` check with a type-specific query that returns `true` only when a `deploy_account` transaction is pending for the address. Concretely:

1. Add a `deploy_account_in_pool(address) -> bool` method to the mempool that inspects the transaction type stored in `tx_pool` for the given address at nonce=0.
2. In `skip_stateful_validations`, replace the call to `account_tx_in_pool_or_recent_block` with this new method.

This preserves the UX intent (allow the paired invoke to be submitted before the deploy_account is mined) while closing the window for unsigned invokes from arbitrary attackers.

---

### Proof of Concept

```
1. Legitimate user submits deploy_account for addr_A (nonce=0, valid signature).
   → deploy_account enters the mempool pool.
   → tx_pool.contains_account(addr_A) == true.

2. Attacker observes the pending deploy_account for addr_A.

3. Attacker submits:
     Invoke { sender: addr_A, nonce: 1, calldata: <anything>, signature: [] }
   with valid resource bounds (passes stateless length checks).

4. Gateway stateful validation path:
   a. get_nonce_from_state(addr_A) → Nonce(0)   [account not deployed yet]
   b. validate_state_preconditions:
        validate_nonce: 0 ≤ 1 ≤ max_allowed_nonce_gap  → OK
        validate_resource_bounds                         → OK
   c. validate_by_mempool:
        validate_incoming_tx: no duplicate hash, nonce ≥ 0 → OK
   d. skip_stateful_validations:
        nonce == 1 && account_nonce == 0 → true
        account_tx_in_pool_or_recent_block(addr_A) → true  (deploy_account is in pool)
        → returns true  (SKIP)
   e. run_validate_entry_point(skip_validate=true):
        execution_flags.validate = false
        __validate__ is NOT called
        → Ok(())

5. Malicious invoke (empty signature) is added to the mempool at nonce=1 for addr_A.

6. Legitimate user's own nonce=1 invoke is rejected: DuplicateNonce.
```

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
