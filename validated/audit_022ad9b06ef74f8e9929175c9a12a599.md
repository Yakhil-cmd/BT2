### Title
Partial Eviction Without Rollback in `try_make_space` Permanently Removes Valid Third-Party Transactions When a Fee-Escalation Replacement Is Ultimately Rejected — (`crates/apollo_mempool/src/mempool.rs`)

### Summary
When a fee-escalation replacement transaction is larger than the existing one, `add_tx_validations` calls `handle_capacity_overflow` → `try_make_space` to evict gap-account transactions for the size delta. If the total evictable space is insufficient, `try_make_space` returns `false` and the replacement is rejected with `MempoolFull`. However, the evictions that already occurred inside `try_make_space` are **never rolled back**. The original transaction remains in the pool, the replacement is rejected, and the evicted third-party transactions are permanently gone. An unprivileged attacker can exploit this to selectively purge competitor transactions from the mempool at zero cost.

### Finding Description

The call chain is:

```
add_tx
  └─ add_tx_validations
       ├─ validate_fee_escalation  → returns Some(existing_tx_reference)
       ├─ freed_bytes = existing_tx.total_bytes()
       ├─ exceeds_capacity(replacement, freed_bytes) == true
       └─ handle_capacity_overflow(replacement, account_nonce, freed_bytes)
            ├─ required_space = replacement.total_bytes() - freed_bytes  (delta only)
            ├─ creating_gap == false  (attacker's nonce == account_nonce)
            └─ try_make_space(required_space)
                 ├─ evicts third-party gap-account txs from tx_pool  ← PERMANENT
                 └─ returns false  (not enough evictable space)
            └─ returns Err(MempoolFull)   ← replacement rejected
       └─ ? propagates error; remove_replaced_tx is never called
```

Inside `try_make_space`, each eviction is an immediate, irreversible `tx_pool.remove()`: [1](#0-0) 

When the loop exhausts all evictable accounts before reaching `required_space`, it returns `false`: [2](#0-1) 

`handle_capacity_overflow` then returns `Err(MempoolFull)`: [3](#0-2) 

`add_tx_validations` propagates the error; `remove_replaced_tx` is never reached, so the original transaction stays in the pool: [4](#0-3) 

The existing test suite **explicitly asserts and accepts** this behavior: [5](#0-4) 

### Impact Explanation

An unprivileged attacker who already has a transaction at `(address, nonce)` in the mempool can:

1. Submit a replacement with a large signature (e.g., padded to `existing_bytes + evictable_pool_bytes + 1`) and fees just above the escalation threshold.
2. `try_make_space` evicts all available gap-account transactions (freeing `evictable_pool_bytes`), then returns `false` because the delta still exceeds available space.
3. The replacement is rejected; the attacker's original transaction is untouched.
4. All evicted third-party transactions are permanently removed from the mempool.

The attacker pays nothing (their original transaction remains) and can repeat the attack to continuously drain competitor transactions from the mempool. This directly corrupts the admission decision: valid transactions that were accepted into the mempool are permanently expelled without being sequenced.

This falls under: **High — Mempool/gateway/RPC admission rejects valid transactions before sequencing.**

### Likelihood Explanation

- Requires only a standard fee-escalation submission (unprivileged public API).
- The attacker needs one existing transaction in the pool and knowledge of the approximate total evictable space (observable from mempool metrics or by probing).
- The attack is repeatable at negligible cost (the original transaction is never lost).
- The `fee_escalation_larger_replacement_evicts_to_make_room` test confirms the eviction path is reachable in normal operation. [6](#0-5) 

### Recommendation

Before mutating the pool, compute whether sufficient evictable space exists. Options:

1. **Dry-run check**: Sum the bytes of all gap-account transactions before calling `try_make_space`. If the sum is less than `required_space`, return `Err(MempoolFull)` immediately without evicting anything.
2. **Transactional eviction**: Collect candidates, verify sufficiency, then commit removals only if the total meets `required_space`.

Either approach ensures evictions are atomic with respect to the admission decision: either the incoming transaction is admitted and evictions are committed, or the incoming transaction is rejected and no evictions occur.

fn fee_escalation_replacement_rejected_but_third_party_evicted() {
    // existing_tx: 0 extra sig felts (small)
    let existing_tx = invoke_tx_with_signature_size(1, "0x0", 0, 90, 90, 0);
    // evictable_tx: 2 extra sig felts (small gap-account tx)
    let evictable_tx = invoke_tx_with_signature_size(2, "0x1", 5, 0, 100, 2);
    let capacity = existing_tx.total_bytes() + evictable_tx.total_bytes();
    let mut mempool = full_mempool_with_fee_escalation(capacity);
    add_tx(&mut mempool, &add_tx_input_for(existing_tx.clone(), "0x0", 0));
    add_tx(&mut mempool, &add_tx_input_for(evictable_tx.clone(), "0x1", 0));

    // Replacement delta (100 sig felts) >> evictable_tx size → try_make_space returns false.
    let oversized_replacement =
        add_tx_input_for(invoke_tx_with_signature_size(3, "0x0", 0, 100, 100, 100), "0x0", 0);
    add_tx_expect_error(&mut mempool, &oversized_replacement, MempoolError::MempoolFull);

    // Original tx must still be present.
    assert!(mempool.tx_pool.get_by_tx_hash(existing_tx.tx_hash()).is_ok(),
        "original tx must survive rejection");
    // Third-party tx must NOT have been evicted (this assertion FAILS before the fix).
    assert!(mempool.tx_pool.get_by_tx_hash(evictable_tx.tx_hash()).is_ok(),
        "third-party tx must not be evicted when replacement is rejected");
}

    // Delta (4 signature felts) is far smaller than the evictable tx, so eviction frees enough.
