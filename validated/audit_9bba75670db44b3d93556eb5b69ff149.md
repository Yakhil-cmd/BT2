### Title
COA Ownership Proof Signature Lacks Chain-ID and Address Binding, Enabling Cross-Chain and Cross-COA Replay - (File: `fvm/evm/stdlib/contract.cdc`)

---

### Summary

`EVM.validateCOAOwnershipProof` verifies Cadence account signatures over arbitrary `signedData` using the hardcoded domain tag `"FLOW-V0.0-user"`. This tag does not encode the chain ID, the target COA EVM address, or the Flow contract address. A valid signature produced in one context (e.g., testnet, or for a different COA on the same account) is cryptographically indistinguishable from a valid signature in another context, enabling cross-chain and cross-address replay of COA ownership proofs.

---

### Finding Description

`validateCOAOwnershipProof` is an `access(all)` function in the deployed EVM system contract. It is also exposed as the EVM precompile `verifyCOAOwnershipProof(address,bytes32,bytes)`, callable from any EVM transaction or contract. The signature verification step is:

```cadence
let isValid = keyList.verify(
    signatureSet: signatureSet,
    signedData: signedData,
    domainSeparationTag: "FLOW-V0.0-user"
)
``` [1](#0-0) 

The tag `"FLOW-V0.0-user"` is a generic Flow user-signing tag. It contains no chain-specific or address-specific information. The function accepts `signedData` as a caller-supplied byte array and `evmAddress` as a separate argument — but the signature is only verified over `signedData`; `evmAddress` is checked only after the signature passes, and only against the COA resource stored at the given `path`. The code itself acknowledges the cross-address replay risk in its own comment:

> "Note: this function does not enforce that `signedData` includes `evmAddress`. In principle, a signature produced for one purpose could be replayed here against a different COA owned by the same Cadence account." [2](#0-1) 

Two concrete replay surfaces exist:

**1. Cross-chain replay:** Because the domain tag contains no chain ID, a COA ownership proof signature produced on Flow Testnet is byte-for-byte valid on Flow Mainnet (and vice versa), provided the same `signedData` bytes are used. An attacker who observes a victim's testnet proof can replay it on mainnet.

**2. Cross-COA replay within the same account:** If a Flow account holds two COAs at different `PublicPath` locations, a proof signed for COA-A is valid as a proof for COA-B, because the signature does not commit to the COA address. The attacker supplies the victim's existing signature alongside a different `path` and `evmAddress` pointing to COA-B.

The EVM-side precompile entry point is:

```go
ProofVerifierFuncSig = ComputeFunctionSelector(
    "verifyCOAOwnershipProof",
    []string{"address", "bytes32", "bytes"},
)
``` [3](#0-2) 

The precompile's `Run` method decodes the proof and delegates to `coaOwnershipProofValidator`, which calls `validateCOAOwnershipProof` in the EVM contract: [4](#0-3) 

The COA contract's `isValidSignature` (ERC-1271) calls this precompile directly:

```solidity
function isValidSignature(bytes32 _hash, bytes memory _sig) external view virtual returns (bytes4){
    (bool ok, bytes memory data) = cadenceArch.staticcall(
        abi.encodeWithSignature("verifyCOAOwnershipProof(address,bytes32,bytes)", address(this), _hash, _sig));
``` [5](#0-4)

### Impact Explanation

The COA ownership proof is the mechanism by which a Flow account asserts control over a COA EVM address. It is used for ERC-1271 signature validation (`isValidSignature`) in the deployed COA contract. EVM contracts that rely on ERC-1271 for authorization (e.g., token approvals, multisig wallets, NFT marketplaces) will accept a replayed proof as a valid signature.

Concretely:
- An attacker who obtains a victim's COA ownership proof signature from testnet (or from a different COA on the same account) can replay it on mainnet to pass ERC-1271 checks on behalf of the victim's COA, enabling unauthorized EVM-side actions (token transfers, approvals, contract interactions) that require the victim's COA signature.

---

### Likelihood Explanation

The `validateCOAOwnershipProof` function is `access(all)` and the precompile is callable from any EVM transaction. No privileged access is required. The cross-chain replay requires only that the attacker observe a valid proof signature from testnet (e.g., from a public testnet dApp interaction). The cross-COA replay requires only that the target account has more than one COA, which is a supported and documented configuration. The `signedData` is fully attacker-controlled when calling the function directly.

---

### Recommendation

Bind the signed message to the chain ID and the target COA EVM address by constructing a structured message before verification, analogous to EIP-712's `domainSeparator`. At minimum, the data that is actually signed should commit to:
- The Flow chain ID (e.g., `flow.Mainnet`, `flow.Testnet`)
- The target COA EVM address (`evmAddress`)
- A purpose-specific context string

For example, the verified payload should be `hash(chainID || evmAddress || signedData)` rather than raw `signedData`, and the domain tag should be specific to this proof type rather than the generic `"FLOW-V0.0-user"`.

---

### Proof of Concept

**Cross-chain replay:**

1. On Flow Testnet, a dApp asks a user to prove COA ownership by signing `data = keccak256("login:dapp.example.com")` with their Flow account key using tag `"FLOW-V0.0-user"`. The resulting `COAOwnershipProof` is published.
2. An attacker takes the identical `(signedData, keyIndices, signatures, address, path)` tuple.
3. On Flow Mainnet, the attacker calls the EVM precompile `verifyCOAOwnershipProof(coaEVMAddress, data, encodedProof)` from within an EVM transaction targeting a contract that uses ERC-1271 (`isValidSignature`) for authorization.
4. `validateCOAOwnershipProof` verifies the signature against `"FLOW-V0.0-user"` — which passes, because the tag is identical on both chains.
5. The ERC-1271 check returns `0x1626ba7e` (valid), and the attacker gains unauthorized access on mainnet using the victim's testnet signature.
