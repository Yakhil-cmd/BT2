### Title
Pool Admin Can Silently Overwrite a Pending Oracle Proposal and Reset the Timelock Clock — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary
`proposePoolPriceProvider` contains no guard preventing a second call while a proposal is already pending. A pool admin can propose oracle A, wait until one block before the timelock elapses, then propose oracle B, resetting `pendingPriceProviderExecuteAfter` to `now + timelock`. Users who were monitoring the original proposal and planning to exit before the switch have their exit window silently stolen.

### Finding Description
`proposePoolPriceProvider` (lines 474–491) reads the timelock, validates the new provider, then unconditionally writes:

```solidity
pendingPriceProvider[pool] = newPriceProvider;
pendingPriceProviderExecuteAfter[pool] = executeAfter;
``` [2](#0-1) 

There is no `if (pendingPriceProvider[pool] != address(0)) revert(...)` or similar guard. The function is callable by the pool admin at any time, any number of times. Each call silently discards the previous pending entry and restarts the clock.

`_validatePriceProvider` only checks that the new oracle's `token0()`/`token1()` match the pool's token pair: [3](#0-2) 

It does not validate price quality, so any oracle that returns the correct token pair — including one that returns manipulated prices — passes validation.

### Impact Explanation
The timelock exists specifically to give LPs and traders advance notice before an oracle switch, allowing them to exit positions they would not hold under a different oracle. By re-proposing a different oracle (oracle B) one block before the original timelock elapses, the admin:

1. Discards the publicly-announced oracle A proposal.
2. Resets the clock to `now + timelock`, giving users zero effective notice of the switch to oracle B.
3. After the new timelock elapses, executes oracle B — which may return prices that cause bad-price execution on swaps (e.g., inverted or manipulated bid/ask), leading to direct loss of LP principal or swap settlement failure.

This is an admin-boundary break: the pool admin exceeds the constraint imposed by the timelock, which is the only mechanism protecting users from sudden oracle changes.

### Likelihood Explanation
The pool admin is a semi-trusted role. The attack requires only two calls to `proposePoolPriceProvider` and a `vm.warp` (or real-time wait). No external conditions, no special tokens, no protocol owner involvement. Any pool admin who controls a mutable-oracle pool can execute this.

### Recommendation
Add a guard at the top of `proposePoolPriceProvider` that reverts if a pending proposal already exists:

```solidity
if (pendingPriceProvider[pool] != address(0)) revert PriceProviderChangePending();
```

Alternatively, add a separate `cancelPoolPriceProviderProposal` function (callable only by the pool admin) so that cancellation is explicit and observable on-chain, rather than silent overwrite.

### Proof of Concept

```solidity
// 1. Admin proposes oracle A; timelock = T seconds
vm.prank(admin);
factory.proposePoolPriceProvider(pool, address(oracleA));
uint256 originalExecuteAfter = factory.pendingPriceProviderExecuteAfter(pool);

// 2. Warp to one second before the timelock elapses
vm.warp(originalExecuteAfter - 1);

// 3. Admin proposes oracle B — silently overwrites oracle A and resets the clock
vm.prank(admin);
factory.proposePoolPriceProvider(pool, address(oracleB));

// 4. Assert: pending is now B, and executeAfter is in the future (timelock reset)
assertEq(factory.pendingPriceProvider(pool), address(oracleB));
assertGt(factory.pendingPriceProviderExecuteAfter(pool), block.timestamp);
// Users who planned to exit before originalExecuteAfter now have no recourse
```
