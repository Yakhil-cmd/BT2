### Title
Unbounded Memory Growth via Malicious Message Injection into `MessageBuffer` - (File: `src/protocol/internal.rs`)

### Summary
The `Protocol::message()` entry point feeds all incoming messages into a `MessageBuffer` backed by an unbounded `HashMap<MessageHeader, SubMessageQueue>`. A malicious participant can craft messages with arbitrary `MessageHeader` values (unique channel tags or waitpoints) that the honest protocol code never polls, causing the map to grow without bound. The codebase itself contains a test that explicitly proves this attack path.

### Finding Description
`ProtocolExecutor::message()` calls `Comms::push_message()`, which parses the first `MessageHeader::LEN` (40) bytes of any incoming message as a routing header and inserts it into the `MessageBuffer`: [1](#0-0) 

`push_message` performs no size check on the map before inserting: [2](#0-1) 

`MessageBuffer::push` calls `entry(...).or_default()`, which allocates a new `SubMessageQueue` (an `UnboundedSender`/`UnboundedReceiver` pair) for every previously-unseen header: [3](#0-2) 

Each `SubMessageQueue` is itself an `futures::channel::mpsc::unbounded()` channel with no capacity limit: [4](#0-3) 

A `MessageHeader` is a 32-byte `ChannelTag` plus an 8-byte `Waitpoint`, giving an attacker 2^320 distinct keys to inject. The attacker can also flood a single valid header's queue with many messages, since `unbounded_send` never blocks or fails: [5](#0-4) 

The codebase itself contains a test that explicitly asserts this behavior — 512 injected messages with distinct waitpoints all land in the map: [6](#0-5) 

### Impact Explanation
A malicious participant in any threshold protocol (DKG, reshare, refresh, presign, sign, CKD) can call `Protocol::message()` with a stream of crafted messages whose headers are never consumed by the honest state machine. Each message allocates a new `HashMap` entry plus an `UnboundedReceiver` heap object. Sustained flooding causes the honest node's process to exhaust available memory, crashing or severely degrading the node. This permanently denies signing and key-generation service to honest parties.

**Impact: Medium** — Griefing/resource-exhaustion by a malicious participant causing unbounded memory growth beyond documented behavior.

### Likelihood Explanation
Any registered participant in a running protocol can call `Protocol::message()` — this is the standard API for delivering network messages. No special privilege is required beyond being a participant. The attack requires only sending byte strings with crafted 40-byte headers, which is trivially constructable. The `MessageHeader` format is fully deterministic and documented in the source.

### Recommendation
1. **Cap the `MessageBuffer` map size**: Reject or drop messages when the number of distinct headers in `MessageBuffer.messages` exceeds a protocol-specific bound (e.g., `n_participants × max_waitpoints`).
2. **Cap per-header queue depth**: Replace `futures::channel::mpsc::unbounded()` with a bounded channel (e.g., `futures::channel::mpsc::channel(MAX_QUEUE_DEPTH)`) and drop or error on overflow.
3. **Authenticate before buffering**: Validate that the incoming `ChannelTag` and `Waitpoint` fall within the expected set for the current protocol phase before inserting into the buffer.

### Proof of Concept
The existing test in `src/protocol/internal.rs` already demonstrates the attack:

```rust
// src/protocol/internal.rs lines 530-554
#[test]
fn attacker_can_fill_message_buffer_with_unused_waitpoints() {
    let comms = Comms::new();
    let attacker = Participant::from(99_u32);
    let attack_count = 512_u64;

    for i in 0..attack_count {
        let header =
            MessageHeader::new(ChannelTag::root_shared()).with_waitpoint(1_000_000 + i);
        let mut message = header.to_bytes().to_vec();
        message.extend_from_slice(&i.to_le_bytes());

        // Attacker injects messages for waitpoints the honest code never polls.
        comms.push_message(attacker, message);
    }

    let messages = comms.incoming.messages.lock().expect("lock should not fail");
    // The map now holds 512 entries — one per injected waitpoint.
    assert!(messages.len() == usize::try_from(attack_count).unwrap());
}
```

To escalate to OOM: replace `attack_count = 512` with a loop that runs until the process is killed, or attach large payloads to each message to exhaust heap faster. In a real protocol, the attacker simply sends crafted `Message::Many` or `Message::Private` bytes through the normal network layer; the honest node's `Protocol::message()` call routes them all into the unbounded map.
