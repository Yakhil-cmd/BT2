### Title
Malicious Participant Can Corrupt CKD Output Without Detection — (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

The CKD coordinator aggregates participant shares by simple addition with no cryptographic verification. A single malicious participant can send arbitrary `(big_y, big_c)` values, silently corrupting the final `CKDOutput` that honest parties accept.

---

### Finding Description

In `do_ckd_coordinator`, the coordinator collects each participant's contribution and sums them unconditionally:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
``` [1](#0-0) 

Each honest participant i is supposed to send:

```
norm_big_y_i = λ_i · y_i · G
norm_big_c_i = λ_i · (x_i · H(pk ‖ app_id) + y_i · app_pk)
```

computed in `compute_signature_share`: [2](#0-1) 

There is **no zero-knowledge proof, no consistency check, and no range check** on the values received from other participants. The coordinator has no mechanism to distinguish a correctly-formed share from an arbitrary group element pair.

The analog to the external report is direct:

| External report | This codebase |
|---|---|
| Fee applied to Principal amount | Coordinator's own share is computed correctly |
| Fee **not** applied to OFT amount | Other participants' shares are **not verified** |
| User exploits the inconsistency to bypass fees | Malicious participant exploits the missing check to corrupt the output |

---

### Impact Explanation

A malicious participant sends `(big_y', big_c')` of their choosing. The coordinator computes:

```
Y'  = Y_honest  + (big_y'  − λ_i · y_i · G)
C'  = C_honest  + (big_c'  − λ_i · (x_i · H + y_i · app_pk))
```

The app then unmasks with `C' − app_sk · Y'`, which no longer equals `msk · H(pk ‖ app_id)`. The derived confidential key is silently wrong. Honest parties receive and accept this corrupted `CKDOutput` with no indication of failure.

This matches the allowed impact: **High — Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.** [3](#0-2) 

---

### Likelihood Explanation

Any single participant in the CKD protocol can trigger this. The attacker needs only to be a registered participant (i.e., appear in the `participants` list passed to `ckd()`). No special privilege, no key material beyond their own share, and no coordination with other parties is required. The attack is a single-round deviation: send `(0, 0)` or any chosen group elements instead of the correct share. [4](#0-3) 

---

### Recommendation

Each participant should accompany their `(norm_big_y, norm_big_c)` with a zero-knowledge proof of correct formation — specifically, a proof that:

1. `norm_big_y = λ_i · y_i · G` for a known `λ_i` and a committed `y_i`.
2. `norm_big_c = λ_i · (x_i · H + y_i · app_pk)` for the same `y_i` and the participant's public key share `x_i · G2`.

A Schnorr-style proof (consistent with the Maurer/Fiat-Shamir infrastructure already present in `src/crypto/proofs/`) would suffice. The coordinator must verify all proofs before summing.



---

### Proof of Concept

1. Honest participants P₁, P₂ and malicious participant P_m run `ckd()`.
2. P_m's `do_ckd_participant` sends `(ElementG1::identity(), ElementG1::identity())` to the coordinator instead of the correct share.
3. The coordinator's loop at line 50–55 adds these zero elements, effectively subtracting P_m's legitimate contribution from the aggregate.
4. The coordinator returns `CKDOutput { big_y: Y_wrong, big_c: C_wrong }`.
5. The application calls `ckd_output.unmask(app_sk)` and obtains `C_wrong − app_sk · Y_wrong ≠ msk · H(pk ‖ app_id)`.
6. No error is raised anywhere in the protocol; honest parties accept the corrupted output. [5](#0-4)

    coordinator: Participant,

    key_pair: KeygenOutput,
    app_id: impl Into<AppId>,
    app_pk: PublicKey,
    rng: impl CryptoRngCore + Send + 'static,

    // not enough participants
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }

    // kick out duplicates
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }
