### Title
`skip_stateful_validations` Bypasses `__validate__` Signature Check via Non-Deploy-Account Mempool Presence — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function is intended to skip the `__validate__` entry point only when a `deploy_account` transaction is pending for an account (UX improvement for the deploy+invoke pattern). However, the mempool check it relies on — `account_tx_in_pool_or_recent_block` — returns `true` for **any** transaction type from the account, not just `deploy_account`. An attacker can prime the mempool with a valid invoke (nonce=0) for a freshly deployed account, then submit a second invoke (nonce=1) with an arbitrary/invalid signature that bypasses `__validate__` entirely and is admitted to the mempool.

### Finding Description

In `crates/apollo_gateway/src/stateful_transaction_validator.rs`, the function `skip_stateful_validations` decides whether to skip the `__validate__` entry point for an incoming invoke transaction:

```rust
async fn skip_stateful_validations(...) -> ... {
    if let ExecutableTransaction::Invoke(...) = tx {
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await ...;
        }
    }
    Ok(false)
}
``` [1](#0-0) 

The mempool implementation of `account_tx_in_pool_or_recent_block` checks for **any** transaction from the account, regardless of type:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [2](#0-1) 

The test `tx_from_address_exists` confirms this returns `true` for any transaction type (including invoke with nonce=0): [3](#0-2) 

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` sets `execution_flags.validate = false`:

```rust
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [4](#0-3) 

This causes `perform_validations` in the blockifier to return early without calling `__validate__`:

```rust
if !tx.execution_flags.validate {
    return Ok(());
}
// `__validate__` call.
let (_optional_call_info, actual_cost) = self.validate(&tx, tx_context.clone())?;
``` [5](#0-4) 

And `validate_tx` itself also short-circuits:

```rust
if !self.execution_flags.validate {
    return Ok(None);
}
``` [6](#0-5) 

### Impact Explanation

An attacker controlling a freshly deployed account (on-chain nonce = 0) can:

1. Submit `invoke_tx1` with nonce=0 and a **valid** signature → passes all checks including `__validate__`, enters the mempool.
2. Submit `invoke_tx2` with nonce=1 and an **invalid/arbitrary** signature:
   - `validate_nonce`: `0 <= 1 <= max_allowed` → passes.
   - `skip_stateful_validations`: `nonce==1 && account_nonce==0` → true; `account_tx_in_pool_or_recent_block` → true (tx1 is in pool) → returns `true`.
   - `run_validate_entry_point` is called with `skip_validate=true` → `execution_flags.validate=false` → `__validate__` is **not called**.
   - `invoke_tx2` is admitted to the mempool **without signature verification**.

The `InternalRpcTransaction` stored in the mempool carries no `execution_flags`. When the batcher retrieves and executes it, `ExecutionFlags::default()` sets `validate=true`, so `__validate__` runs and the transaction reverts. However, the transaction has already been **admitted** past the gateway/mempool admission gate with an invalid signature, satisfying the High impact criterion: *"Mempool/gateway/RPC admission accepts invalid transactions."* [7](#0-6) 

### Likelihood Explanation

The precondition — a deployed account with on-chain nonce=0 and at least one pending invoke in the mempool — is trivially achievable by the attacker themselves. No privileged access is required. The attacker only needs to submit two transactions via the public RPC endpoint.

### Recommendation

Replace the `account_tx_in_pool_or_recent_block` check with a type-specific check that only returns `true` when a **`deploy_account`** transaction is present in the mempool for the given address. The mempool should expose a dedicated `deploy_account_tx_in_pool(address)` query, or the existing check should be narrowed to inspect the transaction type stored for the account.

Alternatively, restrict the skip condition to only apply when the account does **not** exist on-chain (i.e., `get_class_hash_at(address) == ClassHash::default()`), which is the true precondition for the deploy+invoke UX pattern.

### Proof of Concept

```
1. Deploy account A on-chain (nonce becomes 0).

2. Submit via RPC:
   invoke_tx1 = {
     sender_address: A,
     nonce: 0,
     signature: <valid ECDSA signature>,
     calldata: [...],
     resource_bounds: {...}
   }
   → Gateway: validate_nonce(0 <= 0) ✓, __validate__ runs ✓, admitted to mempool.

3. Submit via RPC:
   invoke_tx2 = {
     sender_address: A,
     nonce: 1,
     signature: [0xDEAD, 0xBEEF],   // arbitrary invalid signature
     calldata: [...],
     resource_bounds: {...}
   }
   → Gateway stateful path:
       account_nonce = 0 (from state)
       validate_nonce(0 <= 1 <= 200) ✓
       skip_stateful_validations:
           nonce==1 && account_nonce==0 → true
           account_tx_in_pool_or_recent_block(A) → true (tx1 is in pool)
           returns true (skip __validate__)
       run_validate_entry_point(skip_validate=true):
           execution_flags.validate = false
           __validate__ NOT called
   → invoke_tx2 admitted to mempool with invalid signature.

4. Batcher executes invoke_tx2:
   execution_flags = ExecutionFlags::default() (validate=true)
   __validate__ runs → fails (invalid signature) → tx reverts.
   But the invalid tx was already admitted past the gateway.
``` [8](#0-7) [9](#0-8)

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}

fn tx_from_address_exists(mut mempool: Mempool) {
    const ACCOUNT_ADDRESS: &str = "0x1";
    let account_address = contract_address!(ACCOUNT_ADDRESS);

    // Account is not known to the mempool.
    assert_eq!(mempool.account_tx_in_pool_or_recent_block(account_address), false);

    // The account has a tx in the mempool.
    add_tx(
        &mut mempool,
        &add_tx_input!(tx_hash: 1, address: ACCOUNT_ADDRESS, tx_nonce: 0, account_nonce: 0),
    );
    assert_eq!(mempool.account_tx_in_pool_or_recent_block(account_address), true);

    // The account has a staged tx in the mempool.
    let get_tx_response = mempool.get_txs(1).unwrap();
    assert_eq!(get_tx_response.first().unwrap().contract_address(), account_address);
    assert_eq!(mempool.account_tx_in_pool_or_recent_block(account_address), true);

    // The account has no txs in the pool, but is known through a committed block.
    commit_block(&mut mempool, [(ACCOUNT_ADDRESS, 1)], []);
    MempoolTestContentBuilder::new().with_pool([]).build().assert_eq(&mempool.content());
    assert_eq!(mempool.account_tx_in_pool_or_recent_block(account_address), true);
}
