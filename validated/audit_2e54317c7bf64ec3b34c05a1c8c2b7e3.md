### Title
Presign Accepts Triples Generated by a Different Participant Set Without Validation - (`src/ecdsa/ot_based_ecdsa/presign.rs`)

### Summary

The `presign` function in the OT-based ECDSA pipeline validates only that the threshold stored in `TriplePub` matches the requested presign threshold, but never checks that the participant set recorded in `TriplePub.participants` matches the participant set passed to `presign`. Because `TriplePub` explicitly stores the generating participant set and Lagrange coefficients are participant-set-dependent, using a triple generated for one participant set in a presign session with a different participant set silently corrupts the presignature output.

### Finding Description

`TriplePub` records both the threshold and the exact participant set that generated the triple:

```rust
// src/ecdsa/ot_based_ecdsa/triples/mod.rs
pub struct TriplePub {
    pub big_a: AffinePoint,
    pub big_b: AffinePoint,
    pub big_c: AffinePoint,
    /// The participants in generating this triple.
    pub participants: Vec<Participant>,
    /// The threshold which will be able to reconstruct it.
    pub threshold: ReconstructionLowerBound,
}
``` [1](#0-0) 

The `presign` entry-point checks only the threshold field and explicitly skips the participant-set check with a comment acknowledging the omission:

```rust
// NOTE: We omit the check that the new participant set was present for
// the triple generation, because presumably they need to have been present
// in order to have shares.

if args.threshold != args.triple0.1.threshold || args.threshold != args.triple1.1.threshold {
    return Err(InitializationError::BadParameters(...));
}
``` [2](#0-1) 

Inside `do_presign`, Lagrange coefficients are computed from the **current presign participant set**:

```rust
let lambda_me = participants.lagrange::<Secp256>(me)?;
let k_prime_i = lambda_me * k_i;
let e_i: Scalar = lambda_me * e_i;
let a_prime_i = lambda_me * a_i;
let b_prime_i = lambda_me * b_i;
let x_prime_i = lambda_me * private_share;
``` [3](#0-2) 

But the shares `k_i`, `a_i`, `b_i`, `c_i`, `e_i` are polynomial evaluations produced during triple generation for a **different participant set**. Lagrange interpolation is participant-set-specific: the coefficient `λ_i(S)` for participant `i` over set `S` is defined as `∏_{j∈S\{i}} (-j)/(i-j)`. If the presign participant set `S'` differs from the triple generation set `S`, then `λ_i(S') ≠ λ_i(S)`, and the linearization step produces wrong values. The subsequent consistency checks (`alpha*G =?= K + A`, `beta*G =?= X + B`) will fail or, if the attacker controls the triple shares, can be made to pass with attacker-chosen values.

### Impact Explanation

When the presign participant set differs from the triple generation participant set, the linearized shares `k'_i`, `a'_i`, `b'_i`, `e_i`, `x'_i` are computed with wrong Lagrange coefficients. The aggregated `alpha`, `beta`, and `sigma_i` values are therefore cryptographically inconsistent. The resulting `PresignOutput` (`big_r`, `k`, `sigma`) is corrupted and cannot produce a valid signature. This matches the **High** impact: corruption of presign outputs so honest parties accept unusable cryptographic outputs.

In a stronger scenario where a malicious participant controls which triple shares they contribute, the mismatch in Lagrange coefficients can be exploited to bias the presignature toward attacker-chosen values, potentially enabling signature forgery.

### Likelihood Explanation

The `TriplePub.participants` field is populated and stored during every triple generation run. A caller (coordinator or any participant who manages triple distribution) can trivially supply a `PresignArguments` struct pairing a `TriplePub` from one session with a different `participants` slice. This requires no cryptographic capability — only the ability to call the public `presign` API with mismatched inputs, which is the normal caller role. Participant set changes (reshare, rotation) make this scenario realistic in production deployments.

### Recommendation

Add an explicit check in `presign` that the participant set stored in each `TriplePub` matches the participant set passed to the function:

```rust
// After constructing `participants` (ParticipantList):
let triple0_participants: ParticipantList =
    ParticipantList::new(&args.triple0.1.participants)
        .ok_or(InitializationError::DuplicateParticipants)?;
let triple1_participants: ParticipantList =
    ParticipantList::new(&args.triple1.1.participants)
        .ok_or(InitializationError::DuplicateParticipants)?;

if triple0_participants != participants || triple1_participants != participants {
    return Err(InitializationError::BadParameters(
        "Triple participant set must match presign participant set".to_string(),
    ));
}
```

This mirrors the fix recommended in the Aave Lens report: instead of checking only an indirect property (threshold / handle existence), check that the specific stored identity (participant set / profile ID) matches the one being used.

### Proof of Concept

1. Participants `{0, 1, 2}` run `generate_triple` with threshold 2, producing `(TripleShare_i, TriplePub{participants:[0,1,2], threshold:2})` for each `i`.
2. Participant 2 is rotated out; participant 3 joins. New set is `{0, 1, 3}`.
3. A coordinator calls `presign(&[0,1,3], me=0, PresignArguments { triple0: (share_for_0, triple_pub_from_step1), triple1: ..., threshold: 2 })`.
4. The threshold check at line 43 passes (`2 == 2`). The participant-set check is absent (lines 38–40).
5. `do_presign` computes `lambda_0` using Lagrange over `{0,1,3}`, but `k_0` is a polynomial evaluation for `{0,1,2}`. The linearization `k'_0 = λ_0({0,1,3}) * k_0` is wrong.
6. The aggregated presignature `(big_r, k, sigma)` is corrupted; any subsequent `sign` call using it will produce an invalid signature, permanently denying signing for the honest parties in this session. [4](#0-3) [1](#0-0)

pub fn presign(
    participants: &[Participant],
    me: Participant,
    args: PresignArguments,
) -> Result<impl Protocol<Output = PresignOutput>, InitializationError> {
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }
    // Spec 1.1
    if args.threshold.value() > participants.len() {
        return Err(InitializationError::ThresholdTooLarge {
            threshold: args.threshold.value(),
            max: participants.len(),
        });
    }

    // NOTE: We omit the check that the new participant set was present for
    // the triple generation, because presumably they need to have been present
    // in order to have shares.

    // Also check that we have enough participants to reconstruct shares.
    if args.threshold != args.triple0.1.threshold || args.threshold != args.triple1.1.threshold {
        return Err(InitializationError::BadParameters(
            "New threshold must match the threshold of both triples".to_string(),
        ));
    }

    let participants =
        ParticipantList::new(participants).ok_or(InitializationError::DuplicateParticipants)?;

    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    let ctx = Comms::new();
    let fut = do_presign(ctx.shared_channel(), participants, me, args);
    Ok(make_protocol(ctx, fut))
