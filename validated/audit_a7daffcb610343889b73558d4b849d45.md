### Title
Understated "Total Amount with Royalties" in WalletConnect Offer Confirmation Due to Incorrect Multi-NFT Royalty Calculation - (File: `packages/gui/src/electron/commands/parseCommandDisplay.ts`)

---

### Summary

`formatAmountWithRoyalties` in `parseCommandDisplay.ts` incorrectly divides the fungible amount by the number of NFTs before computing each royalty, producing a royalty total that is `1/N` of the correct value when `N > 1` NFTs are involved. The understated `amountWithRoyalties` is rendered in the WalletConnect confirmation dialog under "You Spend → Total Amount with Royalties," causing the user to approve an offer believing they will pay less than the blockchain will actually deduct.

---

### Finding Description

`formatAmountWithRoyalties` collects all royalty percentages from the receiving side of an offer and then computes:

```typescript
const splitAmount = amount / BigInt(royaltyPercentages.length);   // line 363
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
const totalAmount = amount + royaltyAmount;
``` [1](#0-0) 

The correct formula is: each NFT's royalty is `amount × royalty_i / 10_000`, summed over all NFTs. The code instead computes `(amount / N) × royalty_i / 10_000` per NFT, which is `1/N` of the correct per-NFT royalty. The total royalty shown is therefore `1/N` of what the blockchain will actually charge.

The test fixture for the two-NFT case (royalties 500 and 10, amount 100,000,000 mojos) expects `amountWithRoyalties: '0.00010255'`, which encodes the buggy result:

- **Buggy**: splitAmount = 50,000,000; royalty = 2,500,000 + 50,000 = 2,550,000 mojos → total 0.00010255 XCH
- **Correct**: royalty = 5,000,000 + 100,000 = 5,100,000 mojos → total 0.0001051 XCH [2](#0-1) 

The computed `amountWithRoyalties` is attached to the spending line and rendered in `Confirm.tsx` as "Total Amount with Royalties" inside the WalletConnect approval dialog: [3](#0-2) 

`withRoyaltyTotals` feeds this value into the `DisplayWalletDelta` returned by `walletDeltaToDisplay`, which is the data structure the confirmation dialog renders: [4](#0-3) 

---

### Impact Explanation

A user reviewing a WalletConnect `chia_wallet.take_offer` request that involves multiple NFTs with royalties sees a "Total Amount with Royalties" figure that is materially lower than what the blockchain will deduct. The user approves the offer based on the understated cost. The actual spend (royalties are enforced on-chain by the NFT puzzle) is higher than what was displayed. This constitutes displaying the wrong amount in a signing/approval context, matching the **High** impact tier: *"state that causes a user to approve … the wrong … amount."*

---

### Likelihood Explanation

Any WalletConnect-connected dApp can issue a `chia_wallet.take_offer` command with an offer blob containing two or more NFTs that carry royalties. No special privileges are required. The trigger condition (multiple NFTs with royalties in a single offer) is a normal, supported Chia offer type. The discrepancy grows with the number of NFTs: for 3 NFTs the displayed royalty is 1/3 of the true value.

---

### Recommendation

Remove the `splitAmount` division. Each royalty should be calculated on the full `amount`:

```typescript
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (amount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
```

Update the corresponding test expectation for the two-NFT case from `'0.00010255'` to `'0.0001051'`.

---

### Proof of Concept

Given a WalletConnect `chia_wallet.take_offer` with an offer containing:
- Two NFTs: NFT-A (royalty 5 % = 500 basis points), NFT-B (royalty 0.1 % = 10 basis points)
- Requested: 100,000,000 mojos (0.0001 XCH)

`formatAmountWithRoyalties` is called with `amount = 100_000_000n`, `royaltyPercentages = [500, 10]`:

1. `splitAmount = 100_000_000n / 2n = 50_000_000n`
2. royalty for NFT-A: `50_000_000n * 500n / 10_000n = 2_500_000n`
3. royalty for NFT-B: `50_000_000n * 10n / 10_000n = 50_000n`
4. `royaltyAmount = 2_550_000n`; `totalAmount = 102_550_000n` → displayed as **0.00010255 XCH**

Correct value: `(100_000_000 × 500 + 100_000_000 × 10) / 10_000 = 5_100_000`; total = **0.0001051 XCH**

The user sees they will spend 0.00010255 XCH and approves. The blockchain charges 0.0001051 XCH — approximately 2.4 % more than displayed. The gap scales linearly with the number of NFTs and their royalty rates. [5](#0-4)

function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],
): string | undefined {
  if (royaltyPercentages.length === 0 || line.kind === 'nft') {
    return undefined;
  }

  const splitAmount = amount / BigInt(royaltyPercentages.length);
  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;

  if (line.kind === 'xch') {
    return mojoToChiaLocaleString(totalAmount);
  }

  return mojoToCATLocaleString(totalAmount);
}
