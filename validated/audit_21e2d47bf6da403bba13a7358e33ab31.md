### Title
Silo Mode Whitelist Bypass: `assert_access` Checks Sender Address But Not EVM Transaction Recipient - (File: `engine/src/engine.rs`)

---

### Summary

In Silo mode, the `assert_access` function enforces whitelist checks on the NEAR predecessor account and the EVM transaction **sender** address, but never validates the EVM transaction **recipient** (`transaction.to`). A whitelisted EVM address can therefore transfer ETH directly to any non-whitelisted EVM address, bypassing the `WhitelistKind::Address` restriction that is explicitly documented to govern both submission rights and token receipt.

---

### Finding Description

The `WhitelistKind::Address` tier is documented as controlling which EVM addresses "can submit transactions and receive tokens." [1](#0-0) 

The enforcement point for this whitelist during transaction submission is `assert_access`: [2](#0-1) 

When `transaction.to.is_some()` (a regular call, not a deployment), `assert_access` calls `silo::is_allow_submit` with:
- `env.predecessor_account_id()` — the NEAR caller
- `transaction.address` — the **sender's** EVM address (recovered from the transaction signature)

`transaction.to` — the **recipient** EVM address — is never passed to any whitelist check. `is_allow_submit` itself only checks the sender and the NEAR account: [3](#0-2) 

A separate function `is_allow_receive_erc20_tokens` exists for ERC-20 token receipt checks, and the `erc20_fallback_address` mechanism redirects ERC-20 tokens to a fallback address when the recipient is not whitelisted: [4](#0-3) 

However, **no equivalent check or fallback exists for native ETH transfers via EVM transactions**. The `assert_access` path for `transaction.to.is_some()` never inspects the recipient address at all.

---

### Impact Explanation

A whitelisted EVM sender can craft a standard ETH-transfer transaction targeting any non-whitelisted EVM address. The transaction passes `assert_access` (sender is whitelisted), executes successfully, and credits ETH to the non-whitelisted recipient. Because the recipient is not in `WhitelistKind::Address`, it cannot submit transactions to move those funds. The ETH is effectively frozen at the non-whitelisted address until an admin intervenes.

**Impact: High — Temporary freezing of funds.** The Silo operator's intent to restrict fund receipt to whitelisted addresses is defeated; ETH accumulates at addresses that cannot transact, freezing those funds.

---

### Likelihood Explanation

Any whitelisted EVM address can trigger this with a single standard ETH transfer transaction. No special privileges beyond being on the `WhitelistKind::Address` whitelist are required. The attacker-controlled entry path is the standard `submit` / `submit_with_args` NEAR contract method, which routes through `assert_access`. [5](#0-4) 

---

### Recommendation

In the `transaction.to.is_some()` branch of `assert_access`, additionally validate the recipient address against the `WhitelistKind::Address` whitelist (mirroring the `is_allow_receive_erc20_tokens` pattern already used for ERC-20 tokens). If the recipient is not whitelisted, either reject the transaction or redirect the value to the configured `erc20_fallback_address`, consistent with the ERC-20 fallback behavior already implemented in `silo::mod.rs`.

---

### Proof of Concept

1. Silo mode is active; `WhitelistKind::Address` and `WhitelistKind::Account` whitelists are enabled.
2. Alice's EVM address `A` and NEAR account `alice.near` are whitelisted.
3. Bob's EVM address `B` is **not** whitelisted.
4. Alice submits a standard ETH transfer transaction: `from=A, to=B, value=1 ETH`.
5. `assert_access` is called: `transaction.to = Some(B)`, so it calls `is_allow_submit(io, "alice.near", A)` — both pass.
6. The EVM executes the transfer; Bob's address `B` now holds 1 ETH.
7. Bob attempts to submit any transaction from `B`; `assert_access` calls `is_allow_submit(io, "bob.near", B)` — fails with `NotAllowed`.
8. The 1 ETH at address `B` is frozen: Bob cannot move it, and no fallback mechanism redirects it. [2](#0-1) [6](#0-5) [7](#0-6)

```rust
fn assert_access<I: IO + Copy, E: Env>(
    io: &I,
    env: &E,
    transaction: &NormalizedEthTransaction,
) -> Result<(), EngineError> {
    let allowed = if transaction.to.is_some() {
        silo::is_allow_submit(io, &env.predecessor_account_id(), &transaction.address)
    } else {
        silo::is_allow_deploy(io, &env.predecessor_account_id(), &transaction.address)
    };

    if !allowed {
        return Err(EngineError {
            kind: EngineErrorKind::NotAllowed,
            gas_used: 0,
        });
    }

    Ok(())
}
```
