### Title
Missing Proof-of-Correctness for Participant Contributions in CKD Protocol Allows Malicious Participant to Corrupt CKD Output - (File: src/confidential_key_derivation/protocol.rs)

### Summary
The `do_ckd_coordinator` function in `src/confidential_key_derivation/protocol.rs` accepts and blindly aggregates `(norm_big_y, norm_big_c)` contributions from all participants without any cryptographic proof that each participant computed their share correctly using their actual private signing share. A single malicious participant can send arbitrary group elements, causing the coordinator to produce and accept a corrupted CKD output, making the derived confidential key incorrect and unusable.

### Finding Description

The CKD protocol is a threshold BLS-based scheme where each participant `i` computes:
- `big_y = y * G` (random nonce commitment)
- `big_s = x_i * H(pk || app_id)` (BLS signature share using private share `x_i`)
- `big_c = big_s + y * app_pk` (ElGamal-style encryption)
- `norm_big_y = lambda_i * big_y`, `norm_big_c = lambda_i * big_c`

and sends `(norm_big_y, norm_big_c)` to the coordinator.

The coordinator in `do_ckd_coordinator` (lines 50–55) simply receives and sums all contributions:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
```

There is **no NIZK proof or any other verification** that `(norm_big_y, norm_big_c)` was computed using the participant's actual private share `x_i` and a consistent nonce `y`. The coordinator has no mechanism to detect a malicious or incorrect contribution.

The root cause is the complete absence of a proof-of-correctness check — analogous to the external report's missing fee barrier — that would force participants to demonstrate their contribution is well-formed before it is accepted and aggregated.

The correct CKD output satisfies:
```
unmask(app_sk) = sum(norm_big_c) - app_sk * sum(norm_big_y)
               = sum(lambda_i * x_i) * H(pk || app_id)
               = msk * H(pk || app_id)
```

If any participant sends arbitrary `(big_y', big_c')`, the sum is poisoned and `unmask` produces an incorrect value that is indistinguishable from a valid output to the coordinator. [1](#0-0) [2](#0-1) 

### Impact Explanation

A malicious participant causes the coordinator to accept a corrupted `CKDOutput`. The `unmask` function will then produce an incorrect confidential key — one that does not equal `msk * H(pk || app_id)`. The coordinator has no way to detect this corruption. This maps directly to the allowed impact: **High — Corruption of CKD outputs so honest parties accept unusable cryptographic outputs**. The derived confidential key is silently wrong, breaking the security guarantee of the CKD scheme. [3](#0-2) 

### Likelihood Explanation

Any single participant in the CKD protocol can trigger this. The protocol uses `recv_from_others`, which waits for all `n-1` other participants to respond before the coordinator aggregates. Since there is no threshold subset selection — all participants must contribute — a single malicious participant is sufficient. The attacker needs no privileged access, only participation in the protocol, which is the baseline attacker model. [4](#0-3) 

### Recommendation

Each participant must accompany their `(norm_big_y, norm_big_c)` with a NIZK proof of correct computation — specifically a proof of knowledge that demonstrates:
1. `norm_big_c - norm_big_y_scaled = lambda_i * x_i * H(pk || app_id)` for the participant's committed public share `X_i = x_i * G2`.
2. The nonce `y` used in `big_y` and `big_c` is consistent.

A standard approach is a Chaum-Pedersen DLEQ proof showing that the discrete log of `norm_big_c - lambda_i * x_i * H(pk || app_id)` with respect to `app_pk` equals the discrete log of `norm_big_y` with respect to `G`. The coordinator must verify this proof before including any participant's contribution in the aggregation.

### Proof of Concept

**Setup**: 3 participants run CKD. Participant 2 is malicious.

**Honest behavior**: Participant 2 should send `(lambda_2 * y_2 * G, lambda_2 * (x_2 * H(pk||app_id) + y_2 * app_pk))`.

**Attack**: Participant 2 instead sends `(G, G)` — arbitrary group elements.

**Result**: The coordinator computes:
```
sum_norm_big_y = (lambda_1 * y_1 * G) + G + (lambda_3 * y_3 * G)
sum_norm_big_c = (lambda_1 * big_c_1) + G + (lambda_3 * big_c_3)
```

`unmask(app_sk)` = `sum_norm_big_c - app_sk * sum_norm_big_y` ≠ `msk * H(pk || app_id)`.

The coordinator accepts and returns this corrupted `CKDOutput` with no error. The derived confidential key is incorrect and the CKD protocol's security guarantee is violated. No existing check in `do_ckd_coordinator` catches this. [3](#0-2) [5](#0-4)

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
