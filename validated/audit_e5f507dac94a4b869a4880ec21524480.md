### Title
Pool Admin Bypasses `maxAdminSpreadFeeE6` Cap via Uncapped `setPoolBinAdditionalFees` — (File: metric-core/contracts/MetricOmmPoolFactory.sol)

---

### Summary

`setPoolBinAdditionalFees` forwards `addFeeBuyE6` and `addFeeSellE6` directly to the pool with no validation against `maxAdminSpreadFeeE6`. Because these per-bin fees are added to the base spread fee at swap time, a pool admin can charge traders fees that exceed the factory owner's intended cap, breaking the semi-trusted admin boundary.

---

### Finding Description

The factory enforces `maxAdminSpreadFeeE6` in `setPoolAdminFees`: [1](#0-0) 

```solidity
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (newAdminNotionalFeeE8 > maxAdminNotionalFeeE8) revert AdminFeeTooHigh();
```

However, the sibling function `setPoolBinAdditionalFees` passes the caller-supplied values straight through with no cap check: [2](#0-1) 

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
  external override nonReentrant onlyPoolAdmin(pool)
{
  IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

Inside the pool, `setBinAdditionalFees` also performs no cap check — it writes the values directly to storage: [3](#0-2) 

At swap time, the per-bin fee is added on top of the base spread fee: [4](#0-3) 

```solidity
uint256 buyFeeX64  = baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6,  ONE_X64, 1e6);
uint256 sellFeeX64 = baseFeeX64 + Math.mulDiv(binState.addFeeSellE6, ONE_X64, 1e6);
```

`addFeeBuyE6` and `addFeeSellE6` are `uint16`, so their maximum value is 65 535, which equals **6.5535 %** in E6 units. This additional fee is charged to every trader who swaps through the affected bin, on top of whatever `spreadFeeE6` is already set.

---

### Impact Explanation

The factory owner uses `setFeeCaps` / `maxAdminSpreadFeeE6` to bound how much fee a pool admin can impose on traders. If the owner sets `maxAdminSpreadFeeE6 = 0` (or any value below 65 535), the pool admin can still call `setPoolBinAdditionalFees(pool, bin, 65535, 65535)` and impose a 6.5535 % per-bin surcharge on every swap through that bin. Traders pay more than the protocol's stated cap allows; the factory owner's fee-governance invariant is broken. This is a direct, quantifiable extraction from traders that bypasses the admin-boundary cap — matching the "admin-boundary break: pool admin exceeds caps" criterion in the allowed impact gate.

---

### Likelihood Explanation

Any pool admin (a semi-trusted role) can trigger this unilaterally with a single transaction at any time after pool creation. No special preconditions, no timelock, and no factory-owner co-signature are required. The only prerequisite is that the factory owner has set `maxAdminSpreadFeeE6` to a value below 65 535 (i.e., below 6.5535 %), which is the normal operating assumption for a meaningful cap.

---

### Recommendation

Add a cap check in `setPoolBinAdditionalFees` before forwarding to the pool:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
  external override nonReentrant onlyPoolAdmin(pool)
{
  if (addFeeBuyE6  > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
  if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
  IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

Alternatively, enforce the cap inside `MetricOmmPool.setBinAdditionalFees` by reading the factory's `maxAdminSpreadFeeE6` via a factory call, though the factory-side check is simpler and cheaper.

---

### Proof of Concept

1. Factory owner deploys factory and calls `setFeeCaps(1000, 1000, ...)` — capping admin spread fee at 0.1 % (1 000 in E6).
2. Pool is created with `adminSpreadFeeE6 = 0`; `setPoolAdminFees` correctly enforces the 0.1 % cap.
3. Pool admin calls `setPoolBinAdditionalFees(pool, 0, 65535, 65535)` — no revert, values written to bin 0.
4. A trader swaps through bin 0. The effective fee is `baseFeeX64 + 6.5535 %`, a **65×** bypass of the 0.1 % cap.
5. The excess fee accrues as surplus in the pool and is later collected by the factory owner and pool admin via `collectPoolFees`, extracting value from traders beyond what the governance cap permits.
