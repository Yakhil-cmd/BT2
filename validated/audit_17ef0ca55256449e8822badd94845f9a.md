### Title
Asymmetric Price Velocity Calculation Blocks Valid Upward-Price Swaps While Under-Protecting Against Downward Manipulation — (`metric-periphery/contracts/extensions/PriceVelocityGuardExtension.sol`)

---

### Summary

`PriceVelocityGuardExtension.beforeSwap()` computes the oracle mid-price percentage change using `prevMid` as the fixed denominator. Because `prevMid` is always the *previous* price — not the larger of the two prices — the guard is asymmetric: the same absolute price movement produces a larger `changeE18` when the price rises and a smaller one when it falls. This causes valid upward-price swaps to be incorrectly blocked (DoS) while downward price movements of identical magnitude pass the guard unchallenged.

---

### Finding Description

In `PriceVelocityGuardExtension.sol`:

```solidity
uint256 delta = midPrice > prevMid
    ? uint256(midPrice - prevMid)
    : uint256(prevMid - midPrice);

uint256 changeE18 = (delta * 1e18) / uint256(prevMid);   // ← always divides by prevMid

uint256 actualSq  = changeE18 * changeE18;
uint256 allowedSq = uint256(maxChange) * uint256(maxChange) * (1 + blockDiff);

if (actualSq > allowedSq) {
    revert PriceVelocityExceeded(actualSq, allowedSq);
}
``` [1](#0-0) 

`prevMid` is always the denominator regardless of direction. For the same absolute price movement `Δ`:

| Direction | prevMid | midPrice | delta | changeE18 |
|---|---|---|---|---|
| Up 11 units | 100 | 111 | 11 | 11/100 = **11%** |
| Down 11 units | 111 | 100 | 11 | 11/111 ≈ **9.9%** |

The two cases represent the same absolute price movement but yield different `changeE18` values. With `maxChangePerBlockE18 = 10%`, the upward move is **blocked** while the downward move of identical magnitude **passes**.

---

### Impact Explanation

1. **DoS on valid swaps**: When the oracle price rises legitimately (e.g., after a market event), `changeE18` is inflated relative to the symmetric true percentage, causing `actualSq > allowedSq` and reverting the `beforeSwap` hook. Swaps that should succeed are blocked.

2. **Weakened downward-manipulation protection**: A price drop of the same absolute magnitude produces a smaller `changeE18` (because `prevMid` is now the larger value), making it easier for a downward price manipulation to slip under the `maxChangePerBlockE18` cap.

This directly breaks the core swap flow (the `beforeSwap` hook reverts) and undermines the security invariant the extension is designed to enforce.

---

### Likelihood Explanation

The asymmetry is triggered by any non-trivial oracle price movement. In normal market conditions, prices regularly move in both directions. The guard is most likely to fire incorrectly during upward price rallies — exactly the scenario where the pool admin expects it to behave symmetrically. No special attacker action is required; the flaw is structural.

---

### Recommendation

Use the **larger** of `midPrice` and `prevMid` as the denominator so the percentage is always computed relative to the higher price, making the check symmetric in both directions:

```solidity
uint256 denominator = midPrice > prevMid ? uint256(midPrice) : uint256(prevMid);
uint256 changeE18 = (delta * 1e18) / denominator;
```

This mirrors the fix recommended in the external report: always divide by the larger value so that `|P₁ − P₀| / max(P₀, P₁)` is invariant to direction.

---

### Proof of Concept

**Setup**: `maxChangePerBlockE18 = 0.10e18` (10% cap), single block elapsed (`blockDiff = 0`), so `allowedSq = (0.10e18)² = 1e34`.

**Case A — price rises 11%** (`prevMid = 100e18`, `midPrice = 111e18`):
- `delta = 11e18`
- `changeE18 = 11e18 * 1e18 / 100e18 = 0.11e18`
- `actualSq = (0.11e18)² = 1.21e34 > allowedSq` → **`PriceVelocityExceeded` revert** ✗

**Case B — price falls by the same 11 units** (`prevMid = 111e18`, `midPrice = 100e18`):
- `delta = 11e18`
- `changeE18 = 11e18 * 1e18 / 111e18 ≈ 0.0991e18`
- `actualSq ≈ (0.0991e18)² ≈ 0.982e34 < allowedSq` → **swap proceeds** ✓

The same absolute oracle price movement (11 units) is blocked in one direction and permitted in the other. A pool configured with a 10% velocity cap silently provides ~9.9% protection downward and 10% protection upward — and any upward move just above the cap is incorrectly rejected as manipulation.
