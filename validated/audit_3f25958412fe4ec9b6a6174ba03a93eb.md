### Title
Fixed 2,300-Gas Stipend on Bridge Deposit and COA Transfer Direct Calls Causes Revert for EVM Contracts with Non-Trivial `receive`/`fallback` Logic - (File: `fvm/evm/types/call.go`)

---

### Summary

Flow EVM's native token bridge uses a hardcoded gas limit of `21,000 + 2,300 = 23,300` for all `DepositCall`, `WithdrawCall`, and `TransferCall` direct calls. After the 21,000-unit intrinsic cost is consumed, only 2,300 gas remains for the recipient contract's `receive()` or `fallback()` function — the same fixed stipend as Solidity's deprecated `transfer()`. Any EVM contract whose `receive`/`fallback` requires more than 2,300 gas (e.g., writes to storage, emits events, or calls another contract) will always revert when FLOW tokens are deposited to it via the bridge or COA transfer, permanently blocking cross-VM asset delivery to that address.

---

### Finding Description

In `fvm/evm/types/call.go`, three constants govern the gas limit for all protocol-level token movement direct calls:

```go
IntrinsicFeeForTokenTransfer    = gethParams.TxGas          // 21,000
DefaultGasLimitForTokenTransfer = IntrinsicFeeForTokenTransfer + 2_300  // 23,300
DepositCallGasLimit             = DefaultGasLimitForTokenTransfer
WithdrawCallGasLimit            = DefaultGasLimitForTokenTransfer
``` [1](#0-0) 

`NewDepositCall`, `NewWithdrawCall`, and `NewTransferCall` all hard-wire this limit: [2](#0-1) 

These calls are issued by the handler's `Deposit` and `Transfer` methods, which are the only protocol paths for moving FLOW tokens from Cadence into EVM contract addresses: [3](#0-2) [4](#0-3) 

The comment in the code explicitly acknowledges the 2,300-gas ceiling mirrors the Solidity `transfer()` stipend:

> `// 21_000 is the minimum for a transaction + max gas allowed for receive/fallback methods` [5](#0-4) 

The existing test suite already demonstrates the revert outcome for a contract that cannot process the deposit within this budget: [6](#0-5) 

---

### Impact Explanation

Any EVM smart contract deployed on Flow EVM whose `receive()` or `fallback()` function consumes more than 2,300 gas — a common pattern in multisigs, vaults, ERC-777 hooks, and DeFi protocols that write to storage or emit events on receipt — **can never receive FLOW tokens via the bridge deposit or COA transfer path**. The direct call reverts with `ErrExecutionReverted`, and `panicOnErrorOrInvalidOrFailedState` propagates the failure up to the Cadence transaction, rolling it back. The intended cross-VM asset delivery is permanently blocked for the lifetime of that contract. This constitutes a cross-VM asset delivery failure: tokens cannot reach their intended EVM destination through any protocol-provided mechanism.

---

### Likelihood Explanation

Any unprivileged user can deploy an EVM contract on Flow EVM. Modern Solidity patterns routinely exceed 2,300 gas in `receive`/`fallback` (a single `SSTORE` costs 20,000 gas; a single `LOG1` costs ~375 gas plus data). The condition is trivially triggered by any contract that does anything non-trivial on receipt of ether. The entry path requires only a Cadence transaction calling `cadenceOwnedAccount.deposit(from: <-vault)` or `account.Transfer(...)` targeting such a contract — both are accessible to any unprivileged Flow account holder.

---

### Recommendation

Replace the fixed `21,000 + 2,300` gas ceiling for `DepositCallGasLimit` and `DefaultGasLimitForTokenTransfer` with a configurable or sufficiently large gas limit (e.g., matching the block gas limit or a protocol-defined maximum), consistent with how `NewContractCall` and `NewDeployCall` already accept caller-supplied `gasLimit` values. This mirrors the recommendation in the original report: use a low-level call pattern that forwards available gas rather than capping at the legacy 2,300-unit stipend.

---

### Proof of Concept

1. Deploy an EVM contract on Flow EVM with a `receive()` function that writes to storage (costs >2,300 gas):
   ```solidity
   contract GasHeavyReceiver {
       uint256 public count;
       receive() external payable { count += 1; }  // SSTORE costs ~20,000 gas
   }
   ```
2. From a Cadence transaction, create a COA, fund it, and call `cadenceOwnedAccount.deposit(from: <-vault)` targeting the deployed contract address.
3. The `NewDepositCall` is constructed with `GasLimit = 23,300`. After the 21,000 intrinsic cost, only 2,300 gas remains for `receive()`. The `SSTORE` exhausts gas; the EVM reverts with `ErrExecutionReverted`.
4. `panicOnErrorOrInvalidOrFailedState` panics, the Cadence transaction aborts, and the FLOW tokens are never credited to the contract.

The existing test at `fvm/evm/emulator/emulator_test.go:83–110` already confirms `res.VMError == gethVM.ErrExecutionReverted` for this scenario. [7](#0-6) [6](#0-5)

// Deposit deposits the token from the given vault into the flow evm main vault
// and update the account balance with the new amount
func (a *Account) Deposit(v *types.FLOWTokenVault) {
	defer a.fch.backend.StartChildSpan(trace.FVMEVMDeposit).End()

	bridge := a.fch.addressAllocator.NativeTokenBridgeAddress()
	bridgeAccount := a.fch.AccountByAddress(bridge, false)
	// Note: its not an authorized call
	res, err := a.fch.executeAndHandleCall(
		types.NewDepositCall(
			bridge,
			a.address,
			v.Balance(),
			bridgeAccount.Nonce(),
		),
		v.Balance(),
		false,
	)
	panicOnErrorOrInvalidOrFailedState(res, err)
}
