### Title
WalletConnect Multi-NFT Offer Confirmation Understates Total Royalty Cost, Causing Users to Approve Wrong Amount - (File: packages/gui/src/electron/commands/parseCommandDisplay.ts)

### Summary
The `formatAmountWithRoyalties` function in `parseCommandDisplay.ts` contains an accounting error: when multiple NFTs are on the receiving side of an offer, it divides the payment amount by the number of NFTs before computing each royalty, producing a "Total Amount with Royalties" that is systematically understated. A malicious dApp connected via WalletConnect can exploit this to trick a user into approving a `chia_takeOffer` or `chia_createOfferForIds` command where the actual XCH/CAT deduction is substantially larger than what the confirmation dialog shows.

### Finding Description

`formatAmountWithRoyalties` is the sole function that computes the "Total Amount with Royalties" label shown in the WalletConnect `Confirm` dialog:

```js
// packages/gui/src/electron/commands/parseCommandDisplay.ts  lines 363-368
const splitAmount = amount / BigInt(royaltyPercentages.length);   // ← divides first
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
const totalAmount = amount + royaltyAmount;
``` [1](#0-0) 

The correct formula for N NFTs with royalty percentages R₁…Rₙ is:

```
royaltyAmount = amount × (R₁ + R₂ + … + Rₙ) / 10_000
```

The code instead computes:

```
royaltyAmount = (amount / N) × (R₁ + R₂ + … + Rₙ) / 10_000
```

This is exactly **N times smaller** than the correct value. The existing test suite confirms the wrong result is the intended output:

```
// 2 NFTs, royalties 500 (5%) and 10 (0.1%), amount 0.0001 XCH
// Test expects amountWithRoyalties: '0.00010255'   ← displayed
// Correct value would be:           '0.0001051'    ← actual deduction
``` [2](#0-1) 

The wrong value is propagated through `withRoyaltyTotals` → `walletDeltaToDisplay` → `parseCommandDisplay` and rendered in `Confirm.tsx` as the "Total Amount with Royalties" subtitle under the "You Spend" section: [3](#0-2) 

`parseCommandDisplay` is invoked for both `chia_wallet.take_offer` and `chia_wallet.create_offer_for_ids` WalletConnect commands: [4](#0-3) 

### Impact Explanation

The "Total Amount with Royalties" figure is the primary signal a user relies on to decide whether to approve a WalletConnect offer request. When it is understated, the user approves believing they will spend X but the wallet actually deducts Y > X. The excess flows to the royalty puzzle-hash addresses embedded in the NFTs — which an attacker who minted those NFTs controls.

Concretely, with 2 attacker-minted NFTs each carrying a 50% royalty (royalty_percentage = 5000):

| | Displayed | Actual |
|---|---|---|
| Base amount | 1 XCH | 1 XCH |
| Royalty | 0.5 XCH | 1 XCH |
| **Total** | **1.5 XCH** | **2 XCH** |

The user approves 1.5 XCH but 2 XCH is deducted — a 33% overcharge that goes directly to the attacker's royalty address. The discrepancy scales linearly with the number of NFTs and their royalty percentages.

This matches the allowed High impact: *"Corruption, spoofing, or unsafe trust of … WalletConnect state that causes a user to approve … the wrong … amount."*

### Likelihood Explanation

The attacker path requires only:
1. Minting one or more NFTs with a royalty puzzle-hash they control (a normal, permissionless on-chain action).
2. Constructing a valid offer string that bundles those NFTs against a fungible payment.
3. Sending `chia_takeOffer` (or `chia_createOfferForIds`) via any WalletConnect-paired dApp.

No leaked keys, host compromise, or social engineering beyond the normal WalletConnect pairing flow is required. The victim only needs to trust the displayed total and click "Accept."

### Recommendation

Remove the erroneous division by `royaltyPercentages.length`. Each royalty is applied to the full payment amount:

```js
// packages/gui/src/electron/commands/parseCommandDisplay.ts
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (amount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
const totalAmount = amount + royaltyAmount;
```

Additionally, add an upper-bound guard on `royaltyPercentage` values accepted from offer data (e.g., reject values > 10 000) to prevent a crafted offer from producing an astronomically large displayed total that could overflow or confuse users in the opposite direction.

### Proof of Concept

1. Attacker mints NFT-A and NFT-B on-chain, each with `royalty_percentage = 5000` (50%) pointing to attacker's address.
2. Attacker creates an offer: NFT-A + NFT-B for 1 XCH (1 000 000 000 000 mojos).
3. Attacker's dApp (WalletConnect-paired) sends `chia_takeOffer` with the offer string to the victim's GUI.
4. `parseCommandDisplay` calls `formatAmountWithRoyalties(line, 1_000_000_000_000n, [5000, 5000])`:
   - `splitAmount = 1_000_000_000_000n / 2n = 500_000_000_000n`
   - `royaltyAmount = 250_000_000_000n + 250_000_000_000n = 500_000_000_000n`
   - Displayed total: **1.5 XCH**
5. Victim sees "Total Amount with Royalties: 1.5 XCH" and clicks Accept.
6. Wallet deducts the correct royalties: 500 000 000 000 + 500 000 000 000 = 1 XCH in royalties → **2 XCH total** leaves the victim's wallet.
7. Attacker receives 1 XCH in royalties; victim is 0.5 XCH worse off than the confirmation dialog indicated. [5](#0-4)

function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],
): string | undefined {
  if (royaltyPercentages.length === 0 || line.kind === 'nft') {
    return undefined;
  }

  const splitAmount = amount / BigInt(royaltyPercentages.length);
  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;

  if (line.kind === 'xch') {
    return mojoToChiaLocaleString(totalAmount);
  }

  return mojoToCATLocaleString(totalAmount);
}

export async function parseCommandDisplay(command: string, params: Record<string, unknown>) {
  if (command === 'chia_wallet.take_offer') {
    if (!params.offer || typeof params.offer !== 'string') {
      throw new Error('Offer is not valid');
    }

    const offerSummary = await getOfferSummary(params.offer);
    if (!offerSummary || !offerSummary.summary || !offerSummary.success) {
      throw new Error('Offer is not valid');
    }

    const { summary } = offerSummary;

    const walletDelta = offerSummaryToWalletDelta(summary);
    const walletInfos = await getWalletInfos();
    const assetKinds = offerSummaryAssetKinds(summary);
    const royaltyPercentages = offerSummaryRoyaltyPercentages(summary);
    const fees = parseMojos(summary.fees);

    return {
      walletDelta: await walletDeltaToDisplay(walletDelta, walletInfos, assetKinds, royaltyPercentages, fees),
    };
  }

  if (command === 'chia_wallet.create_offer_for_ids') {
    if (!params.offer || !isPlainObject(params.offer)) {
      throw new Error('Offer is not valid');
    }

    if (params.driver_dict !== undefined && !isPlainObject(params.driver_dict)) {
      throw new Error('Driver Dict is not valid');
    }

    const walletDelta = createOfferToWalletDelta(params.offer);
    const walletInfos = await getWalletInfos();
    const driverDict = params.driver_dict ?? {};
    const assetKinds = createOfferAssetKinds(walletDelta, walletInfos, driverDict);
    const royaltyPercentages = createOfferRoyaltyPercentages(walletDelta, driverDict);

    return {
      walletDelta: await walletDeltaToDisplay(walletDelta, walletInfos, assetKinds, royaltyPercentages, undefined),
    };
  }
