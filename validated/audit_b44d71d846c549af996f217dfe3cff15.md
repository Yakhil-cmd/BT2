### Title
Replay of `allowPushers` Consent Signature Re-establishes Revoked Pusher Delegation, Corrupting Oracle Namespace Data - (File: `smart-contracts-poc/contracts/oracles/compressed/CompressedOracle.sol`)

---

### Summary

`CompressedOracle.allowPushers` verifies a pusher's EIP-191 consent signature but tracks no nonce and records no used-signature state. Within the deadline window the identical signature can be submitted an unlimited number of times. A creator who holds a pusher's old consent signature can replay it after the pusher has called `revokePusher()`, silently re-establishing the delegation the pusher intended to terminate. Subsequent pushes from the revoked pusher are then written into the creator's namespace instead of the pusher's own, corrupting the oracle data that downstream `PriceProvider` contracts and live pools consume.

---

### Finding Description

`allowPushers` signs consent as:

```
keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
``` [1](#0-0) 

The signed tuple binds chain, contract, deadline, pusher address, and creator address. It does **not** include a nonce, a per-pusher sequence counter, or any flag that marks the signature as consumed after first use. The function performs no used-signature bookkeeping: [2](#0-1) 

`revokePusher` clears `namespaceRemapping[msg.sender]` to `address(0)`: [3](#0-2) 

Because the signature is not invalidated on first use, the creator can call `allowPushers` again with the same `(deadline, pusher, signature)` tuple at any point before `deadline`, overwriting the cleared mapping back to `creator`. The pusher's revocation is silently undone.

The `fallback` push path resolves the namespace by reading `namespaceRemapping[msg.sender]`: [4](#0-3) 

After the replay, every subsequent push from the pusher writes into the creator's namespace, not the pusher's own.

---

### Impact Explanation

The corrupted namespace propagates to live pool pricing through the oracle stack. A `PriceProvider` reads from the creator's namespace to produce bid/ask quotes consumed by `MetricOmmPool.swap`. If the pusher is now pushing data intended for a different feed (their own namespace or another creator's), the creator's namespace receives wrong prices. Pools that rely on that namespace execute swaps at incorrect bid/ask values, causing traders to receive more tokens than the oracle permits or the pool to receive fewer input tokens than owed — a direct swap conservation failure and potential fund loss for LPs.

---

### Likelihood Explanation

The attack requires the creator to retain the pusher's original consent signature and to replay it before the deadline expires. Deadlines are caller-chosen and can be set days or weeks in the future. The creator has a clear economic motive (keeping a data source active) and the replay costs only gas. The pusher must continue pushing after revoking for the namespace corruption to produce wrong oracle data, which is the normal operational case when a pusher switches to serving their own namespace or a different creator.

---

### Recommendation

Record each consumed signature hash in a `mapping(bytes32 => bool) usedSignatures` and revert if the hash has already been seen:

```solidity
bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
    keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
);
require(!usedSignatures[hash], "signature already used");
usedSignatures[hash] = true;
require(pusher == ECDSA.recover(hash, signatures[i]));
```

Alternatively, include a per-pusher nonce in the signed payload and increment it on each successful `allowPushers` call, so that any previously issued signature is automatically invalidated.

### Proof of Concept

```solidity
// 1. Pusher signs consent for creator with a far-future deadline.
uint256 deadline = block.timestamp + 365 days;
bytes memory sig = sign(PUSHER_KEY, chainid, oracle, deadline, pusher, creator);

// 2. Creator establishes delegation.
vm.prank(creator);
oracle.allowPushers(deadline, [pusher], [sig]);
assert(oracle.namespaceRemapping(pusher) == creator);

// 3. Pusher revokes — intends to push into own namespace from now on.
vm.prank(pusher);
oracle.revokePusher();
assert(oracle.namespaceRemapping(pusher) == address(0));

// 4. Creator replays the SAME signature before deadline.
vm.prank(creator);
oracle.allowPushers(deadline, [pusher], [sig]); // succeeds — no nonce check
assert(oracle.namespaceRemapping(pusher) == creator); // revocation undone

// 5. Pusher's subsequent pushes (intended for own namespace) land in creator's namespace.
vm.prank(pusher);
oracle.call(encodedSlotWord); // writes to creator's namespace, not pusher's
// Creator's PriceProvider now returns wrong bid/ask → pool swap at bad price.
```

    function allowPushers(uint256 deadline, address[] calldata pushers, bytes[] memory signatures) external {
        _ensureDeadline(deadline);

        uint256 l = pushers.length;
        require(l == signatures.length);
        for (uint256 i; i < l; i++) {
            address pusher = pushers[i];

            if (pusher == msg.sender) {
                revert NoSelfRemapping();
            }

            bytes32 hash = MessageHashUtils.toEthSignedMessageHash(
                keccak256(abi.encode(block.chainid, address(this), deadline, pusher, msg.sender))
            );
            require(pusher == ECDSA.recover(hash, signatures[i]));

            namespaceRemapping[pusher] = msg.sender;
            emit PusherAuthorized(pusher, msg.sender);
        }
