### Title
Missing Validation of `TriplePub.participants` Against Presign Participant Set Enables Denial of Signing - (File: `src/ecdsa/ot_based_ecdsa/presign.rs`)

### Summary
The OT-based ECDSA `presign()` function explicitly omits the check that the presign participant set matches the triple generation participant set recorded in `TriplePub.participants`. A malicious coordinator can provide participants with triple shares generated by a different participant set, causing the presigning protocol to abort for all honest parties and permanently preventing signing.

### Finding Description
The `TriplePub` struct records the participants who generated the triple in its `participants` field. [1](#0-0) 

The presigning specification explicitly requires that triples be "generated by the same set of parties P_0" as the presigning set: [2](#0-1) 

However, `presign()` only validates the threshold value against `TriplePub.threshold`, never the participant set against `TriplePub.participants`. The code explicitly acknowledges this omission with a comment: [3](#0-2) 

This is the direct analog of the SafeTransferLib issue: the library trusts the caller to provide consistent inputs without actually verifying it. The `TriplePub.participants` field is populated during triple generation: [4](#0-3) 

But it is never read or validated inside `presign()` or `do_presign()`. When a participant set mismatch occurs, the Lagrange coefficients computed in `do_presign()` are wrong — they use the presign participant set instead of the triple generation participant set: [5](#0-4) 

This causes the `E =?= e*G` check (Spec 1.5) to fail, aborting the protocol: [6](#0-5) 

### Impact Explanation
A malicious coordinator who controls the distribution of triple shares can provide participants with shares from a different triple generation session (with a different participant set). Since `TriplePub.participants` is never validated at initialization, participants cannot detect this mismatch when `presign()` is called — it returns `Ok(...)` silently. The presigning protocol only aborts later during execution when the `E =?= e*G` check fails. The attacker can repeat this indefinitely, permanently preventing any signing session from completing.

This maps to: **High: Permanent denial of signing for honest parties under valid protocol inputs and documented trust assumptions.**

### Likelihood Explanation
In a real deployment, a coordinator orchestrates signing sessions and is responsible for telling participants which triple shares to use for a given session. A malicious coordinator can exploit this gap by directing participants to use triple shares from a session with a different participant set. The attack requires no cryptographic capabilities — only the ability to provide wrong `TriplePub` values, which is within the coordinator's role. The `TriplePub` struct is fully public and serializable, making it trivial to substitute.

### Recommendation
Add a validation check in `presign()` that verifies `args.triple0.1.participants` and `args.triple1.1.participants` match the presign `participants` argument before proceeding:

```rust
// Validate that the triple participants match the presign participants
let participants_vec: Vec<Participant> = participants.iter().collect();
if args.triple0.1.participants != participants_vec
    || args.triple1.1.participants != participants_vec
{
    return Err(InitializationError::BadParameters(
        "Triple participants must match presign participants".to_string(),
    ));
}
```

This mirrors the existing threshold check and closes the gap that the comment at lines 38–40 explicitly leaves open.
