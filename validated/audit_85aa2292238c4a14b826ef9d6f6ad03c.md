### Title
Pool Admin Bypasses Hard Fee Cap via Uncapped `setPoolBinAdditionalFees` — (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary
`setPoolBinAdditionalFees` passes `addFeeBuyE6`/`addFeeSellE6` directly to the pool with no cap validation. A pool admin can set per-bin additional fees up to `uint16.max` (65 535 ≈ 6.55 % in E6 terms). Because the swap math adds these values on top of the base `spreadFeeE6`, the effective per-bin fee can exceed the protocol's hard ceiling (`HARD_MAX_SPREAD_FEE_E6` = 20 %), causing traders to lose more principal than the protocol's own cap permits.

### Finding Description
The factory enforces fee caps on the base spread fee in `setPoolAdminFees`:

```solidity
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
``` [1](#0-0) 

`maxAdminSpreadFeeE6` is itself bounded by `HARD_MAX_SPREAD_FEE_E6` (20 %). [2](#0-1) 

However, `setPoolBinAdditionalFees` applies **no analogous cap**:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
``` [3](#0-2) 

The pool's `setBinAdditionalFees` only validates the bin index, not the fee magnitude:

```solidity
function setBinAdditionalFees(int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6) external onlyFactory ... {
    if (bin < LOWEST_BIN || bin > HIGHEST_BIN) revert InvalidBinIndex(bin);
    BinState storage s = _binStates[bin];
    s.addFeeBuyE6 = addFeeBuyE6;
    s.addFeeSellE6 = addFeeSellE6;
``` [4](#0-3) 

In the swap loop these per-bin fees are **added** to the base fee:

```solidity
params.baseFeeX64 + Math.mulDiv(binState.addFeeSellE6, ONE_X64, 1e6)
``` [5](#0-4) 

`baseFeeX64` is derived from `spreadFeeE6` (the total base spread, already at or below 20 %). With `addFeeSellE6 = 65535` the effective per-bin fee becomes `spreadFeeE6 + 65535`, i.e. up to **26.55 %**, silently exceeding the hard cap.

This is the direct analog of the external bug: just as CryptoJS defaulted to an unspecified, unauthenticated cipher mode because no mode was explicitly required, `setPoolBinAdditionalFees` omits the required cap specification, allowing the per-bin fee parameter to reach an unconstrained value that breaks the protocol's own invariant.

### Impact Explanation
Traders swapping through a bin with elevated per-bin fees receive less output than the protocol's hard cap permits. The excess fee (above 20 %) is extracted from the trader's input. For a swap of size `S` through a single bin, the over-charge is `S × addFeeSellE6 / 1e6` (up to ≈ 6.55 % of `S`). This is a direct, quantifiable loss of user principal on every affected swap.

### Likelihood Explanation
Pool creation is **permissionless** — any actor can deploy a pool naming themselves as admin. The pool admin is semi-trusted only within caps; this path lets them exceed those caps without any privileged factory-owner involvement. A malicious pool creator can deploy a legitimate-looking pool, attract liquidity and swap volume, then call `setPoolBinAdditionalFees` with `addFeeBuyE6 = 65535` on the active bin to silently drain excess fees from every subsequent swap.

### Recommendation
Add a cap check in `setPoolBinAdditionalFees` (and mirror it in `setBinAdditionalFees`) so that the per-bin additional fee cannot cause the total effective fee to exceed `HARD_MAX_SPREAD_FEE_E6`. For example:

```solidity
// In MetricOmmPoolFactory.setPoolBinAdditionalFees:
uint24 currentSpread = poolFeeConfig[pool].protocolSpreadFeeE6 + poolFeeConfig[pool].adminSpreadFeeE6;
if (uint256(currentSpread) + addFeeBuyE6  > HARD_MAX_SPREAD_FEE_E6) revert AdminFeeTooHigh();
if (uint256(currentSpread) + addFeeSellE6 > HARD_MAX_SPREAD_FEE_E6) revert AdminFeeTooHigh();
```

Alternatively, introduce a dedicated `MAX_BIN_ADDITIONAL_FEE_E6` constant and enforce it at both the factory and pool layers.

### Proof of Concept
1. Deploy a pool with `adminSpreadFeeE6 = maxAdminSpreadFeeE6` and `spreadProtocolFeeE6` at its current default, so `spreadFeeE6` is at or near 20 %.
2. Pool admin calls `factory.setPoolBinAdditionalFees(pool, 0, 65535, 65535)` — succeeds with no revert.
3. A trader calls `swap(…, zeroForOne=true, …)` routing through bin 0.
4. The swap math applies `baseFeeX64 + mulDiv(65535, ONE_X64, 1e6)`, charging ≈ 26.55 % total instead of the capped 20 %.
5. The trader receives ≈ 6.55 % less token0 than the protocol's hard cap would allow, with the surplus retained in the pool as LP/admin fee. [3](#0-2) [4](#0-3) [5](#0-4)
