### Title
Echo Broadcast Consistency Failure for n≤3 Enables DKG Output Corruption — (`src/protocol/echo_broadcast.rs`)

### Summary

The `echo_ready_thresholds` function silently returns `(0, 0)` for participant counts `n ≤ 3`. With both thresholds at zero, the delivery condition `> 2 * ready_t` is satisfied by a single ready vote — the participant's own simulated vote — before any network message is received. A malicious participant can exploit this by sending different values to different honest parties (equivocation), causing each honest party to immediately deliver a different value. In the DKG context this means honest parties accept different polynomial commitments for the malicious participant, compute different public keys, and produce inconsistent DKG outputs.

---

### Finding Description

**Root cause — `echo_ready_thresholds`:** [1](#0-0) 

```rust
fn echo_ready_thresholds(n: usize) -> (usize, usize) {
    if n <= 3 {
        return (0, 0);          // ← both thresholds collapse to zero
    }
    let broadcast_threshold = (n - 1) / 3;
    let echo_threshold = usize::midpoint(n, broadcast_threshold);
    (echo_threshold, broadcast_threshold)
}
```

With `echo_t = 0` and `ready_t = 0`, the three delivery guards in `reliable_broadcast_receive_all` become:

| Guard | Condition | Meaning |
|---|---|---|
| Echo → Ready | `count > 0` | 1 echo suffices |
| Ready → Amplify | `count > 0` | 1 ready suffices |
| Ready → Deliver | `count > 2*0 = 0` | 1 ready suffices | [2](#0-1) [3](#0-2) 

The critical observation is that after processing a `Send(V)` message, the code immediately simulates an `Echo(V)` from self, which immediately triggers `Ready(V)` from self, which immediately satisfies the delivery condition `> 0` — all before any network message from another party arrives.

**Equivocation attack for n=3:**

Suppose participants are `{A, B, C}` and A is malicious.

1. A sends `Send(V1)` to B and `Send(V2)` to C (equivocation).
2. B processes `Send(V1)` → simulates `Echo(V1)` → `data_echo[V1]=1 > 0` → sends `Ready(V1)`, sets `finish_echo=true` → simulates `Ready(V1)` → `data_ready[V1]=1 > 0` → amplifies → simulates `Ready(V1)` again (skipped, already seen) → `data_ready[V1]=1 > 0 = 2*ready_t` → **delivers V1**, sets `finish_ready=true`.
3. C performs the symmetric sequence and **delivers V2**.
4. When B later receives `Ready(V2)` from C, it is skipped because `finish_ready=true`. Same for C receiving `Ready(V1)` from B.

B and C have permanently diverged on A's broadcast value.

**Propagation into DKG (`do_keyshare`):**

The DKG uses `do_broadcast` in two critical places: [4](#0-3) 

```rust
let session_ids = do_broadcast(&mut chan, &participants, me, my_session_id).await?;
``` [5](#0-4) 

```rust
let commitments_and_proofs_map = do_broadcast(
    &mut chan, &participants, me,
    (commitment, proof_of_knowledge),
).await?;
```

A malicious participant A can:
- In round 1, send different `commitment_hash` values to B and C via `chan.send_many` (not echo-broadcast, so no consistency protection).
- In round 3, exploit the echo broadcast failure to deliver different `(commitment, proof)` pairs to B and C.

The `verify_commitment_hash` check at line 463–469 passes for both B and C because each received a matching hash/commitment pair from A. [6](#0-5) 

B and C then call `public_key_from_commitments` with different commitment sets, producing different `verifying_key` values. The `broadcast_success` round does not detect this because each honest party independently believes its own view is consistent. [7](#0-6) 

---

### Impact Explanation

**Impact: High — Corruption of DKG outputs so honest parties accept inconsistent public keys.**

Honest parties B and C complete the DKG protocol without error but hold different public keys. Any subsequent signing session will fail because the parties disagree on the group public key. This constitutes a permanent denial of signing for honest parties under valid protocol inputs (3-party DKG with threshold 2 is explicitly supported by `assert_key_invariants`). [8](#0-7) 

---

### Likelihood Explanation

- The minimum participant count enforced by `assert_key_invariants` is 2; n=3 is the smallest practically useful configuration and is explicitly tested.
- A single malicious participant is sufficient — no collusion required.
- The attack requires only that the malicious party send different byte strings to different peers, which is trivially achievable by any participant controlling their own network stack.
- The failure is deterministic (not probabilistic): the simulated-vote fast-path always fires before any network message, so B always delivers V1 and C always delivers V2.

---

### Recommendation

1. **Remove the n≤3 special case** or replace it with thresholds that still enforce consistency. For n=3 with f=1 malicious party, the standard BRB requirement is n ≥ 3f+1 = 4, so n=3 cannot tolerate even one malicious sender. Either enforce `n ≥ 4` at the API level, or document and enforce that n=3 assumes all-honest participants.
2. **Add a runtime guard** in `echo_ready_thresholds` that returns an error (or panics) when `n < 4` and the caller expects Byzantine fault tolerance.
3. **Alternatively**, for n=3, use a simpler all-to-all broadcast that requires receiving the value from all n−1 peers before delivering, which provides consistency under the all-honest assumption.

---

### Proof of Concept

```
Participants: A (malicious), B (honest), C (honest). n=3, echo_t=0, ready_t=0.

A → B: Send(V1)   [session sid_A]
A → C: Send(V2)   [session sid_A]

B: recv Send(V1) → simulate Echo(V1) → data_echo[V1]=1>0 → send Ready(V1)
   → simulate Ready(V1) → data_ready[V1]=1>0 → amplify Ready(V1)
   → simulate Ready(V1) [skipped, seen] → data_ready[V1]=1>0 → DELIVER V1, finish_ready=true

C: recv Send(V2) → simulate Echo(V2) → data_echo[V2]=1>0 → send Ready(V2)
   → simulate Ready(V2) → data_ready[V2]=1>0 → amplify Ready(V2)
   → simulate Ready(V2) [skipped, seen] → data_ready[V2]=1>0 → DELIVER V2, finish_ready=true

B later receives Ready(V2) from C → skipped (finish_ready=true)
C later receives Ready(V1) from B → skipped (finish_ready=true)

        B computes public_key = f(commitment_1, ...) ≠ g(commitment_2, ...) = public_key computed by C.
        DKG completes without error on both sides but with inconsistent public keys.

fn echo_ready_thresholds(n: usize) -> (usize, usize) {
    // case where no malicious parties are assumed: when n <= 3/
    // In this case the echo and ready thresholds are both 0
    // later we compare if we have collected more votes than these thresholds

    let broadcast_threshold = (n - 1) / 3;
    let echo_threshold = usize::midpoint(n, broadcast_threshold);
    (echo_threshold, broadcast_threshold)
}

```rust
pub fn assert_key_invariants(
    participants: &[Participant],
    me: Participant,
    threshold: impl Into<ReconstructionLowerBound>,
) -> Result<ParticipantList, InitializationError> {
    let threshold = usize::from(threshold.into());
    // need enough participants
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }

    // Step 1.1
    // validate threshold
    if threshold > participants.len() {
        return Err(InitializationError::ThresholdTooLarge {
            threshold,
            max: participants.len(),
        });
    }
    // Step 1.1
    if threshold < 2 {
        return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
    }

    // ensure uniqueness of participants in the participant list
    let participants =
        ParticipantList::new(participants).ok_or(InitializationError::DuplicateParticipants)?;

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }
    Ok(participants)
}
