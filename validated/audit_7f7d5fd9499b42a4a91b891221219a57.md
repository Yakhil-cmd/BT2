### Title
COA Contract Implements ERC-1271 `isValidSignature` But Omits ERC-1271 Interface ID from `supportsInterface` - (File: `fvm/evm/handler/coa/coa.sol`)

### Summary

The `COA` contract — the Flow EVM system's Cadence-Owned Account smart contract wallet — implements `isValidSignature` (ERC-1271) but does not declare the ERC-1271 interface ID (`0x1626ba7e`) in its `supportsInterface` function. This is a direct structural analog to the reported `TGKNFTContract` bug: a contract implements functions from a standard it does not formally declare in its interface registry. Any EVM contract on Flow EVM that guards `isValidSignature` calls behind an ERC-165 `supportsInterface` check will receive `false` for ERC-1271, causing COA-based signature validation to silently fail. Because COAs have no private key by design, there is no fallback signing path, and assets locked behind ERC-1271-gated authorization become inaccessible to the COA owner.

### Finding Description

`COA` inherits from `ERC1155TokenReceiver`, `ERC777TokensRecipient`, `ERC721TokenReceiver`, and `IERC165`, and its `supportsInterface` function checks exactly those four interface IDs:

```solidity
// fvm/evm/handler/coa/coa.sol, lines 67-73
function supportsInterface(bytes4 id) external view virtual override returns (bool) {
    return
        id == type(ERC1155TokenReceiver).interfaceId ||
        id == type(ERC721TokenReceiver).interfaceId ||
        id == type(ERC777TokensRecipient).interfaceId ||
        id == type(IERC165).interfaceId;
}
```

The contract also implements `isValidSignature` (ERC-1271) and even defines the magic value constant:

```solidity
// fvm/evm/handler/coa/coa.sol, lines 61-63
bytes4 constant internal ValidERC1271Signature = 0x1626ba7e;
```

But `0x1626ba7e` — which is simultaneously the ERC-1271 function selector and the ERC-1271 interface ID — is never returned by `supportsInterface`. The contract comment in `contract.cdc` explicitly lists ERC-1271 as a supported standard:

> "COAs are smart contract wallets that accept native token transfers and support several ERCs including ERC-165, ERC-721, ERC-777, ERC-1155, **ERC-1271**."

The omission is therefore inconsistent with the contract's own specification.

### Impact Explanation

**Vulnerability class:** Incorrect interface conformance declaration → cross-VM asset loss.

ERC-1271 is the standard mechanism by which smart contract wallets (including COAs) authorize transactions in EVM DeFi protocols, multisigs, DAOs, and NFT marketplaces. The ERC-1271 specification and ERC-165 best practice both recommend that callers check `supportsInterface(0x1626ba7e)` before invoking `isValidSignature`. Any such protocol deployed on Flow EVM will receive `false` from a COA and will not call `isValidSignature`, causing the COA's authorization to be silently rejected.

Because COAs have no private key by design — the Cadence side owns the account and there is no EOA fallback — a COA owner whose assets are locked behind an ERC-1271-gated authorization cannot recover them through any alternative signing path. This constitutes cross-VM asset loss: FLOW or ERC-20/ERC-721 tokens deposited into an EVM protocol that uses `supportsInterface`-guarded ERC-1271 checks become permanently inaccessible to the COA owner.

### Likelihood Explanation

ERC-1271 is the dominant smart-contract-wallet signing standard in the EVM ecosystem. Protocols such as Gnosis Safe, OpenSea (Seaport), Unipermit, and many lending protocols guard `isValidSignature` calls behind `supportsInterface` checks. Any such protocol deployed on Flow EVM — by any unprivileged user — will silently reject COA signatures. The COA is Flow's primary cross-VM wallet primitive and is the intended signer for EVM-side operations. The likelihood of a COA owner interacting with an ERC-1271-aware protocol is high, and the failure mode is silent (no revert, just a `false` return from `supportsInterface`), making it difficult to diagnose.

### Recommendation

1. Add an ERC-1271 interface declaration to the `COA` inheritance list (or define a local `IERC1271` interface) and include the ERC-1271 interface ID in `supportsInterface`:

```solidity
id == 0x1626ba7e ||   // ERC-1271: isValidSignature(bytes32,bytes)
```

2. Regenerate `coa_abi.json` and `coa_bytes.hex` from the corrected source.

### Proof of Concept

1. Deploy the following contract on Flow EVM (no privileges required):

```solidity
interface IERC165 { function supportsInterface(bytes4) external view returns (bool); }
interface IERC1271 { function isValidSignature(bytes32, bytes memory) external view returns (bytes4); }

contract ERC1271Guard {
    function canSign(address wallet, bytes32 hash, bytes memory sig) external view returns (bool) {
        // ERC-165 guard — standard best practice
        if (!IERC165(wallet).supportsInterface(0x1626ba7e)) {
            return false;  // COA always reaches here
        }
        return IERC1271(wallet).isValidSignature(hash, sig) == 0x1626ba7e;
    }
}
```

2. Call `canSign(coaAddress, hash, sig)` where `coaAddress` is any COA.
3. `supportsInterface(0x1626ba7e)` returns `false` on the COA (line 67-73 of `coa.sol`), so `canSign` returns `false` without ever calling `isValidSignature`, even though the COA's `isValidSignature` would return `0x1626ba7e` for a valid proof.
4. Any assets deposited into `ERC1271Guard`-gated logic by a COA owner are now inaccessible.
