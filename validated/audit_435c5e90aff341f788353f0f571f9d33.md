### Title
ERC-20 Tokens Permanently Burned Without Refund When `ft_transfer` Fails and `error_refund` Feature Is Disabled ŌĆö (`engine-precompiles/src/native.rs`)

---

### Summary

When the `error_refund` compile-time feature is absent, the `ExitToNear` precompile constructs a `callback_args` that equals `ExitToNearPrecompileCallbackArgs::default()` for every standard ERC-20 exit. This causes the engine to schedule a bare `PromiseArgs::Create` (no callback) for the NEP-141 `ft_transfer`. If that promise fails, no `exit_to_near_precompile_callback` fires, no ERC-20 re-mint occurs, and the user's tokens are permanently lost.

---

### Finding Description

**Step 1 ŌĆö `callback_args` construction without `error_refund`**

In `ExitToNear::run`:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,          // ŌåÉ always None when feature is absent
    transfer_near: transfer_near_args,
};
``` [1](#0-0) 

For the standard ERC-20 legacy path (flag `0x1`, no `:unwrap` suffix), `exit_erc20_token_to_near` returns `transfer_near_args = None`: [2](#0-1) 

So `callback_args = { refund: None, transfer_near: None }` which equals `ExitToNearPrecompileCallbackArgs::default()`. [3](#0-2) 

**Step 2 ŌĆö Branch selection: no callback attached**

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // ŌåÉ bare promise, no callback
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs { ... })
};
``` [4](#0-3) 

**Step 3 ŌĆö No recovery path when `ft_transfer` fails**

`exit_to_near_precompile_callback` only re-mints ERC-20 tokens when `args.refund` is `Some(...)`:

```rust
} else if let Some(args) = args.refund {
    // Exit call failed; need to refund tokens
    let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;
``` [5](#0-4) 

Because no callback is scheduled at all (step 2), this function is never called. Even if it were, `args.refund` would be `None`, so the `else` branch at line 240 (`None`) would execute ŌĆö no refund.

**Step 4 ŌĆö Test explicitly confirms the loss**

The existing integration test `test_exit_to_near_refund` directly encodes this behavior:

```rust
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
``` [6](#0-5) 

The test confirms: ERC-20 balance decreases by `FT_EXIT_AMOUNT`, NEP-141 balance of the recipient stays at zero, and no refund is issued.

---

### Impact Explanation

- The user's ERC-20 tokens are burned inside the EVM (irreversible state change).
- The corresponding NEP-141 tokens remain locked in Aurora's NEP-141 account balance ŌĆö inaccessible to the user with no on-chain recovery path.
- Total supply invariant across EVM and NEAR is broken: ERC-20 supply decreases, NEP-141 supply held by Aurora does not decrease, recipient receives nothing.
- Impact: **High ŌĆö Temporary freezing of funds** (NEP-141 tokens stranded in Aurora's account; ERC-20 tokens burned from user's balance).

---

### Likelihood Explanation

- Triggerable by any user holding a bridged ERC-20 token who exits to an unregistered NEAR account, a paused NEP-141 contract, or an account with insufficient storage deposit ŌĆö all ordinary, non-adversarial conditions.
- No admin compromise or special privilege required.
- The only precondition is that the engine is compiled without the `error_refund` feature.

---

### Recommendation

1. **Enable `error_refund` in production builds.** The feature already implements the correct refund path via `refund_call_args` and `refund_on_error`.
2. If `error_refund` must remain optional, add a compile-time or runtime guard that prevents the ERC-20 burn from succeeding unless a callback with refund data is attached.
3. Alternatively, always attach a callback (even when `transfer_near` is `None`) so that a failed `ft_transfer` can trigger a re-mint regardless of the feature flag.

---

### Proof of Concept

The existing test `test_exit_to_near_refund` in `engine-tests/src/tests/erc20_connector.rs` is a direct proof of concept. Compiled without `error_refund`:

1. User holds bridged ERC-20 tokens (`FT_TRANSFER_AMOUNT`).
2. User calls `exit_to_near` targeting `"unregistered.near"` (not registered with the NEP-141 contract).
3. ERC-20 tokens are burned: balance drops to `FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT`.
4. `ft_transfer` promise fails (unregistered receiver).
5. No callback fires; no re-mint occurs.
6. NEP-141 balance of `"unregistered.near"` = 0; NEP-141 balance of Aurora = `FT_TRANSFER_AMOUNT` (unchanged).
7. User has lost `FT_EXIT_AMOUNT` tokens with no recovery. [7](#0-6)
