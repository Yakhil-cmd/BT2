### Title
Missing Access Control on `update_rewards` Allows Any Caller to Permanently Suppress Block Rewards - (File: src/staking/staking.cairo)

### Summary
The `update_rewards` function in `src/staking/staking.cairo` is documented in the spec as callable by "Only starkware sequencer," but no caller check is implemented in the code. Any unprivileged address can call it with `disable_rewards: true`, causing `last_reward_block` to be written to the current block number while skipping reward distribution entirely. Because the per-block guard prevents a second call in the same block, the sequencer cannot recover the missed rewards for that block. An attacker can repeat this every block, permanently suppressing all consensus-phase reward accrual for stakers and delegators.

### Finding Description
The spec at `docs/spec.md` line 1645 states:

```
#### access control
Only starkware sequencer.
```

The implementation in `src/staking/staking.cairo` at `StakingRewardsManagerImpl::update_rewards` (lines 1449–1510) contains no such check:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();                          // only checks pause flag
    let current_block_number = starknet::get_block_number();
    assert!(
        current_block_number > self.last_reward_block.read(),
        "{}",
        Error::REWARDS_ALREADY_UPDATED,
    );
    ...
    self.last_reward_block.write(current_block_number);   // guard consumed here

    if disable_rewards || self.is_pre_consensus() {
        return;                                            // rewards skipped, guard already consumed
    }
    ...
```

`general_prerequisites()` only asserts the contract is not paused; it performs no caller identity check. The `last_reward_block` storage slot is updated unconditionally before the `disable_rewards` branch, so once any caller invokes the function for a given block, no further call is possible for that block.

This is structurally identical to the referenced report: a security mechanism (access restriction) is explicitly specified in the documentation but is never initialized/implemented in the code, leaving the protection absent.

### Impact Explanation
When `disable_rewards = true` is passed, the staker's `unclaimed_rewards_own` is not incremented and no pool rewards are transferred. The block's reward allocation is permanently lost — there is no catch-up mechanism. An attacker who front-runs the sequencer's call every block causes all stakers and delegators to accrue zero rewards indefinitely during the consensus-rewards phase, constituting permanent freezing of unclaimed yield.

### Likelihood Explanation
The entry path requires no privilege: any externally-owned address can call `update_rewards` on the deployed staking contract. Staker addresses are publicly discoverable from on-chain `NewStaker` events. The attacker only needs to submit one transaction per block with a valid staker address and `disable_rewards: true`. The cost is low and the attack is repeatable every block.

### Recommendation
Add a sequencer-address access control check at the top of `update_rewards`, analogous to how `update_rewards_from_attestation_contract` enforces `assert_caller_is_attestation_contract()`. Store the authorized sequencer address in contract storage (set during construction or via a privileged setter) and assert it at the start of `update_rewards`:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
+   assert!(
+       get_caller_address() == self.sequencer_address.read(),
+       "{}",
+       Error::CALLER_IS_NOT_SEQUENCER,
+   );
    ...
```

### Proof of Concept

1. Consensus rewards are active (`consensus_rewards_first_epoch` has been reached).
2. Attacker observes a valid `staker_address` from on-chain events.
3. At the start of block N, attacker calls:
   ```
   staking.update_rewards(staker_address, disable_rewards: true)
   ```
4. `last_reward_block` is written to block N; the function returns early — no rewards distributed.
5. The sequencer attempts `update_rewards(staker_address, disable_rewards: false)` in the same block → reverts with `REWARDS_ALREADY_UPDATED`.
6. Block N's rewards are permanently lost for the staker and all its pool delegators.
7. Attacker repeats step 3 every block; stakers and delegators accrue zero rewards.

            ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,

            self.general_prerequisites();
            let current_block_number = starknet::get_block_number();
            assert!(
                current_block_number > self.last_reward_block.read(),
                "{}",

            // Assert staker exists and active.

            // Staker remains active until the intent period begins, i.e. K epochs after

            let staker_info = self.internal_staker_info(:staker_address);

            assert!(

                "{}",
                Error::INVALID_STAKER,

            let staker_pool_info = self.staker_pool_info.entry(staker_address).as_non_mut();
            let (staker_total_strk_balance, staker_total_btc_balance) = self
