### Title
Stale `target_shard_index` from old epoch's shard layout used to index new epoch's `outcome_roots` in `GetExecutionOutcome`, corrupting `outcome_root_proof` MerklePath for receipts executed at the resharding boundary — (`chain/client/src/view_client_actor.rs`)

### Summary

`GetExecutionOutcome` computes `target_shard_index` using the **old epoch's** shard layout before calling `get_next_block_hash_with_new_chunk`. When that function crosses the epoch boundary and returns a block in the **new epoch** (after resharding), the code collects `outcome_roots` ordered by the **new epoch's** shard layout but indexes into them with the **old** `target_shard_index`. In V2/V3 shard layouts, shard IDs and shard indices are explicitly decoupled, so the old index does not correspond to the correct position in the new layout. The resulting `outcome_root_proof` MerklePath is wrong, and any light client verifying it against the block's `outcome_root` gets a false negative.

### Finding Description

In `view_client_actor.rs`, the `GetExecutionOutcome` handler:

1. Reads `epoch_id` from the **outcome block** (old epoch, before resharding). [1](#0-0) 

2. Computes `target_shard_index` via `shard_layout.get_shard_index(target_shard_id)` using the **old epoch's** `shard_layout`. This index is never updated afterward. [2](#0-1) 

3. Calls `get_next_block_hash_with_new_chunk`, which explicitly handles epoch transitions: when the shard layout changes, it maps `shard_ids` to children shards and updates its internal `shard_layout` to the new epoch's layout, then returns `(new_epoch_block_hash, child_shard_id)`. [3](#0-2) 

4. Collects `outcome_roots` from the **new epoch block's** chunks — ordered by the **new** shard layout. [4](#0-3) 

5. Indexes into `outcome_roots` with the **stale** `target_shard_index` from the old layout. [5](#0-4) 

The shard layout documentation explicitly states: **"In V2 & V3 the shard id and shard index are not the same."** [6](#0-5) 

**Concrete example using V2 `derive` logic:**

- Old layout: `boundary_accounts = ["bbb"]`, `shard_ids = [0, 1]` → shard 0 at index 0, shard 1 at index 1.
- New boundary account `"aaa"` is inserted at position 0. Shard 0 (at index 0) is spliced out and replaced by children `[2, 3]`.
- New layout: `shard_ids = [2, 3, 1]` → shard 2 at index 0, shard 3 at index 1, shard 1 at index **2**. [7](#0-6) 

A receipt executed in old shard 1 (index 1) has `target_shard_index = 1`. After resharding, `get_next_block_hash_with_new_chunk` returns child shard 1 (which is now at index **2** in the new layout). The code uses index 1 to pick `outcome_roots[1]`, which is the outcome root of shard 3 — not shard 1. The returned `outcome_root_proof` is the Merkle path for the wrong leaf.

The bounds check at line 1174 only guards against out-of-bounds access; it does not detect or correct the index mismatch when the new layout has more shards than the old one. [8](#0-7) 

### Impact Explanation

The exact corrupted value is `outcome_root_proof: merklize(&outcome_roots).1[target_shard_index]` — a `MerklePath` that is the Merkle proof for the wrong shard's outcome root. A light client running:

```
verify_path(block_header.outcome_root, &outcome_root_proof, &chunk_outcome_root)
```

will get `false` for a legitimately executed receipt, because the proof corresponds to a different shard's leaf in the outcome root Merkle tree. This breaks the core light client proof invariant: a valid receipt execution cannot be proven to a light client for any receipt executed in a shard whose index shifts between the old and new epoch layouts.

### Likelihood Explanation

This requires:
1. A resharding event (V2 or V3 layout, where shard ID ≠ shard index).
2. A receipt executed in a shard whose index changes in the new layout (i.e., a shard that is not the split shard itself but is shifted by the insertion of new children).
3. The receipt's outcome block is close enough to the epoch boundary that `get_next_block_hash_with_new_chunk` crosses into the new epoch.

All three conditions are met for any shard that appears after the split point in the `shard_ids` array. On mainnet, resharding events have occurred at protocol versions 75, 76, and 78. Any user who submitted a cross-shard receipt in the last few blocks of those epochs and then requested a `light_client_proof` would have received a broken proof.

### Recommendation

After `get_next_block_hash_with_new_chunk` returns `(h, child_shard_id)`, recompute `target_shard_index` using the **new epoch's** shard layout:

```rust
if let Some((h, target_shard_id)) = res {
    outcome_proof.block_hash = h;
    // Recompute index using the new epoch's shard layout
    let new_epoch_id = *self.chain.get_block(&h)?.header().epoch_id();
    let new_shard_layout = self.epoch_manager.get_shard_layout(&new_epoch_id).into_chain_error()?;
    let target_shard_index = new_shard_layout
        .get_shard_index(target_shard_id)
        .map_err(Into::into)
        .into_chain_error()?;
    // ... rest unchanged
```

### Proof of Concept

A test-loop test can demonstrate this:

1. Configure a two-epoch test with a V2 shard layout that reshards at epoch boundary (e.g., 2 shards → 3 shards, where shard 1 splits into shards 2 and 3, shifting old shard 1's index from 1 to 2).
2. Submit a cross-shard receipt in the last block of the old epoch, targeting an account in old shard 1.
3. Advance past the epoch boundary so the receipt's outcome block is in the old epoch and `get_next_block_hash_with_new_chunk` returns a block in the new epoch.
4. Call `GetExecutionOutcome` for the receipt ID.
5. Verify the returned `outcome_root_proof` against the new epoch block's `outcome_root` using `verify_path` — it will return `false`, confirming the corrupted proof.

The existing test in `test-loop-tests/src/tests/light_client.rs` (`check_outcome_proofs`) provides the verification harness; it only needs to be extended to cover a resharding epoch boundary. [9](#0-8)

fn check_outcome_proofs(
    env: &mut TestLoopEnv,
    account_id: &AccountId,
    seed_hash: CryptoHash,
    tx: SignedTransaction,
) {
    let outcome = env.rpc_runner().execute_tx(tx, Duration::seconds(5)).unwrap();
    // Advance so the outcome's block is final and included in later blocks' merkle roots.
    env.rpc_runner().run_for_number_of_blocks(4);

    let mut ids = vec![TransactionOrReceiptId::Transaction {
        transaction_hash: outcome.transaction_outcome.id,
        sender_id: account_id.clone(),
    }];
    for receipt_outcome in &outcome.receipts_outcome {
        ids.push(TransactionOrReceiptId::Receipt {
            receipt_id: receipt_outcome.id,
            receiver_id: account_id.clone(),
        });
    }

    let mut rpc = env.rpc_node_mut();
    let view_client = rpc.view_client_actor();

    // The light client head and the block merkle root all proofs are checked against.
    let light_client_block = view_client
        .handle(GetNextLightClientBlock { last_block_hash: seed_hash })
        .unwrap()
        .unwrap();
    let light_client_head = light_client_block_hash(&light_client_block);
    let block_merkle_root = light_client_block.inner_lite.block_merkle_root;

    for id in ids {
        let execution_outcome = view_client.handle(GetExecutionOutcome { id }).unwrap();
        let outcome_proof = execution_outcome.outcome_proof;
        let block_proof = view_client
            .handle(GetBlockProof {
                block_hash: outcome_proof.block_hash,
                head_block_hash: light_client_head,
            })
            .unwrap();

        // outcome -> chunk outcome root -> block outcome root.
        let chunk_outcome_root =
            compute_root_from_path_and_item(&outcome_proof.proof, &outcome_proof.to_hashes());
        assert!(verify_path(
            block_proof.block_header_lite.inner_lite.outcome_root,
            &execution_outcome.outcome_root_proof,
            &chunk_outcome_root,
        ));

        // The light block header recomputes to the proof's block hash.
        assert_eq!(block_proof.block_header_lite.hash(), outcome_proof.block_hash);

        // block hash -> light client head's block merkle root.
        assert!(verify_hash(block_merkle_root, &block_proof.proof, outcome_proof.block_hash));
    }
