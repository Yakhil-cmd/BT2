### Title
Missing Error-Recovery Callback in `ExitToNear` Precompile Causes Permanent Fund Loss When NEAR-Side Promise Fails - (File: `engine-precompiles/src/native.rs`)

---

### Summary

The `ExitToNear` precompile, when compiled without the `error_refund` feature, creates a NEAR `ft_transfer` promise with **no callback**. If that promise fails (e.g., the recipient NEAR account is not registered with the NEP-141 contract), the EVM-side funds — ERC-20 tokens already burned or ETH already deducted — are permanently unrecoverable. This is the Aurora Engine analog of Solidity's `transfer()` gas-stipend trap: a one-way fund movement with no fallback path on failure.

---

### Finding Description

In `ExitToNear::run()`, the callback arguments are constructed as: [1](#0-0) 

When the `error_refund` feature is **not** enabled, `refund` is hardcoded to `None`. For all standard ERC-20 and ETH exits (non-wNEAR), `transfer_near` is also `None`. This makes `callback_args` equal to the default value: [2](#0-1) 

Because `callback_args == ExitToNearPrecompileCallbackArgs::default()` evaluates to `true`, the code takes the `PromiseArgs::Create(transfer_promise)` branch — a bare promise with **no callback**. If the downstream `ft_transfer` call on the NEP-141 contract fails, there is no `exit_to_near_precompile_callback` invoked, and therefore no refund path executes.

The `exit_to_near_precompile_callback` function, which is the only place `engine::refund_on_error` is called, is never scheduled: [3](#0-2) 

The `ExitToEthereum` precompile has the same structural problem — it unconditionally emits a bare `PromiseArgs::Create` with no callback, regardless of any feature flag: [4](#0-3) 

---

### Impact Explanation

When a user (or a smart contract on Aurora) calls the `ExitToNear` precompile:

1. ERC-20 tokens are **burned** from the caller's EVM balance (or ETH is deducted from `context.apparent_value`).
2. A NEAR `ft_transfer` promise is scheduled.
3. If the promise fails — e.g., the recipient NEAR account has no storage deposit with the NEP-141 contract — the NEP-141 tokens remain locked in Aurora's account.
4. The EVM-side tokens are already gone. No refund is issued.

The project's own integration tests confirm this outcome explicitly: [5](#0-4) [6](#0-5) 

**Impact: Permanent freezing of funds (Critical/High).** The burned ERC-20 tokens or deducted ETH cannot be recovered by any user-accessible mechanism.

---

### Likelihood Explanation

The trigger condition is straightforward and reachable by any unprivileged EVM user:

- Call `ExitToNear` with a recipient NEAR account that has not registered storage with the target NEP-141 contract.
- This is a common operational mistake (e.g., sending to a freshly created NEAR account, a DAO contract, or any account that never called `storage_deposit` on the NEP-141).
- A malicious EVM contract could also deliberately route victim funds through `ExitToNear` to an unregistered account, causing permanent loss.

The `ft_transfer` NEP-141 standard requires the recipient to have a storage deposit; failure to meet this requirement is a well-known and frequent cause of failed transfers on NEAR.

---

### Recommendation

1. **Always register a callback** on the `ft_transfer` promise in `ExitToNear::run()`, unconditionally — not gated behind `error_refund`. The callback should refund the caller's EVM balance (ETH or re-mint ERC-20 tokens) when the promise result is not `Successful`.
2. Apply the same fix to `ExitToEthereum::run()`: attach a callback to the `withdraw` promise so that if the eth-connector call fails, the ETH is restored to the caller's EVM balance.
3. Make `error_refund` behavior the default (or remove the feature flag entirely), as the no-refund path is demonstrably unsafe.

---

### Proof of Concept

**Entry path (ERC-20 exit):**

1. Deploy a NEP-141 token; bridge it to Aurora as an ERC-20.
2. Call the ERC-20's `withdrawToNear` function targeting a NEAR account that has **not** called `storage_deposit` on the NEP-141 contract.
3. The ERC-20 tokens are burned in the EVM.
4. The `ft_transfer` NEAR promise fails (recipient not registered).
5. No callback fires; no refund is issued.
6. The user's ERC-20 balance is zero; the NEP-141 tokens remain in Aurora's account; the recipient receives nothing.
