### Title
`skip_stateful_validations` Bypasses `__validate__` for Nonce-1 Invoke Transactions When Any Nonce-0 Tx Is in the Mempool Pool - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator skips the account's `__validate__` entry point for invoke transactions with nonce=1 whenever the account has **any** transaction in the mempool pool. The comment claims this is safe because it implies a deploy_account tx exists, but the check `account_tx_in_pool_or_recent_block` returns `true` for any transaction type — including a nonce-0 invoke tx submitted by the legitimate account owner. An unprivileged attacker can exploit this to inject a nonce-1 invoke transaction with an invalid (or absent) signature for any account that currently has a pending nonce-0 invoke tx in the mempool, bypassing gateway signature validation entirely.

### Finding Description

`skip_stateful_validations` in `crates/apollo_gateway/src/stateful_transaction_validator.rs` (lines 429–461) decides whether to skip the blockifier's `__validate__` call:

``` [1](#0-0) 

The comment states: *"It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."*

`account_tx_in_pool_or_recent_block` is implemented as:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [2](#0-1) 

`tx_pool.contains_account` returns `true` for **any** transaction type in the pool — not only deploy_account transactions. If account X (deployed, on-chain nonce = 0) has a legitimate nonce-0 invoke tx in the pool, `tx_pool.contains_account(X)` is `true`. An attacker who submits a nonce-1 invoke tx for X with an invalid signature will:

1. Pass `validate_nonce` — nonce 1 is within `[0, 0 + max_allowed_nonce_gap]` where `max_allowed_nonce_gap = 200` by default. [3](#0-2) 
2. Pass `validate_by_mempool` — the mempool's `validate_tx` only checks for duplicate hashes and nonce-too-old; it does not verify signatures. [4](#0-3) 
3. Trigger `skip_stateful_validations` returning `true` — nonce=1, account_nonce=0, account in pool. [5](#0-4) 
4. Have `run_validate_entry_point` called with `skip_validate=true`, setting `execution_flags.validate = false`, so the blockifier's `StatefulValidator::perform_validations` returns `Ok(())` without calling `__validate__`. [6](#0-5) 

The invalid transaction is then forwarded to the mempool via `add_tx`. The batcher will later attempt to execute it, at which point `__validate__` runs and the tx is rejected — but the gateway has already accepted and admitted it.

The `max_nonce_for_validation_skip` config field (`Nonce(Felt::ONE)` by default) is **not** used in the gateway's `skip_stateful_validations`; it is only used in the legacy `PyValidator::should_run_stateful_validations` path. The gateway hardcodes the nonce-1 check. [3](#0-2) 

### Impact Explanation

The gateway's stateful admission invariant is broken: a transaction with an invalid (or forged) signature for an account the attacker does not control is accepted into the mempool. This matches the **High** impact scope: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

Practical consequences:
- Attacker can pollute the mempool with signature-invalid transactions for any account that has a pending nonce-0 tx, consuming mempool capacity and wasting batcher execution resources.
- The invalid tx occupies the nonce-1 slot, potentially blocking the legitimate owner's nonce-1 tx (duplicate-nonce rejection) until the invalid tx is evicted.

### Likelihood Explanation

**Medium-High.** The precondition — an account with on-chain nonce=0 and a pending nonce-0 invoke tx in the mempool — is the normal state for any newly deployed account that has just sent its first transaction. An attacker can observe the mempool (it is public) and immediately submit a nonce-1 tx with a garbage signature. No privileged access is required; the only cost is meeting the stateless resource-bound checks (valid gas price, non-zero bounds).

### Recommendation

`skip_stateful_validations` must verify that the account has a **deploy_account** transaction in the mempool, not merely any transaction. The mempool should expose a dedicated query such as `has_deploy_account_tx(address)`, or the gateway should check the transaction type of the existing pool entry before granting the skip. Alternatively, restrict the skip to the case where `tx_pool.contains_deploy_account(account_address)` is true.

### Proof of Concept

```
1. Deploy account X (class_hash C, salt S) → on-chain nonce = 0.
2. Owner of X submits InvokeV3(sender=X, nonce=0, calldata=..., sig=VALID_SIG)
   → gateway runs __validate__ → passes → tx enters mempool pool.
   Now: tx_pool.contains_account(X) == true.

3. Attacker submits InvokeV3(sender=X, nonce=1, calldata=ARBITRARY, sig=GARBAGE)
   via the HTTP gateway endpoint.

   Gateway stateless checks:
     - contract_address valid ✓
     - resource_bounds non-zero ✓
     - nonce_da_mode == L1 ✓

   Gateway stateful checks (extract_state_nonce_and_run_validations):
     - get_nonce_from_state(X) → 0
     - validate_nonce: 0 <= 1 <= 0+200 ✓
     - validate_by_mempool: nonce 1 >= 0, no duplicate hash ✓
     - skip_stateful_validations:
         tx.nonce()==1 && account_nonce==0 → check mempool
         account_tx_in_pool_or_recent_block(X) → true (step 2 tx still in pool)
         → returns true
     - run_validate_entry_point(skip_validate=true):
         execution_flags.validate = false
         StatefulValidator::perform_validations → returns Ok(()) without __validate__

   → Gateway returns HTTP 200, tx_hash for the invalid tx.
   → Mempool now contains the signature-invalid nonce-1 tx for account X.

4. Batcher picks up nonce-0 tx, executes it (nonce incremented to 1 on-chain).
5. Batcher picks up nonce-1 tx, runs __validate__ → FAIL (invalid signature).
   Tx rejected; nonce-1 slot freed.
   But: legitimate owner's nonce-1 tx was blocked from the mempool during step 3–5
   (DuplicateNonce error if owner tried to submit it).
```

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
