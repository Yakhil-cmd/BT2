### Title
Pool Admin Bypasses Fee Caps via Uncapped `setPoolBinAdditionalFees` — (`File: metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary

`setPoolAdminFees` enforces `maxAdminSpreadFeeE6` / `maxAdminNotionalFeeE8` caps on the pool admin's fee authority. `setPoolBinAdditionalFees`, a parallel admin path that sets per-bin additional buy/sell fees charged directly to traders, performs **no cap check at all**. A pool admin can set `addFeeBuyE6` or `addFeeSellE6` to the full `uint16` maximum (65 535 E6 ≈ 6.55 %) on any bin, silently exceeding the protocol's intended fee ceiling and extracting value from every trader who crosses that bin.

### Finding Description

The factory exposes two pool-admin fee paths:

**Path A — capped** (`setPoolAdminFees`):
```solidity
// MetricOmmPoolFactory.sol:414-415
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (newAdminNotionalFeeE8 > maxAdminNotionalFeeE8) revert AdminFeeTooHigh();
```

**Path B — uncapped** (`setPoolBinAdditionalFees`):
```solidity
// MetricOmmPoolFactory.sol:450-457
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

No cap, no timelock, no validation beyond the `uint16` type bound. The pool-level `setBinAdditionalFees` likewise performs only a bin-index range check:

```solidity
// MetricOmmPool.sol:464-474
function setBinAdditionalFees(int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external onlyFactory nonReentrant(PoolActions.SET_BIN_ADDITIONAL_FEES)
{
    if (bin < LOWEST_BIN || bin > HIGHEST_BIN) revert InvalidBinIndex(bin);
    BinState storage s = _binStates[bin];
    s.addFeeBuyE6 = addFeeBuyE6;
    s.addFeeSellE6 = addFeeSellE6;
    ...
}
```

During every swap these per-bin fees are added **on top of** the oracle-derived base spread fee before the bin math runs:

```solidity
// MetricOmmPool.sol:999 (buyToken0 specified-in path)
params.baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6)

// MetricOmmPool.sol:1088 (buyToken1 specified-out path)
params.baseFeeX64 + Math.mulDiv(binState.addFeeSellE6, ONE_X64, 1e6)
```

The same additive pattern appears in `getSellAndBuyPrices` and the periphery `MetricOmmPoolDataProvider._marginalBestBidAsk`. The bin additional fee is therefore a **direct trader cost**, not a protocol-split parameter, and it is entirely uncapped.

The same gap exists at pool creation: `_unpackAndValidateBinStates` validates bin lengths and cumulative distances but never checks the `addFeeBuyE6` / `addFeeSellE6` fields packed into each bin word, so a pool creator can embed maximum-value additional fees from genesis.

### Impact Explanation

A pool admin calls `setPoolBinAdditionalFees(pool, bin, 65535, 65535)`. Every trader whose swap crosses that bin pays ≈ 6.55 % extra on top of the oracle spread. The excess fee accrues to LPs (the admin may be the dominant LP). Traders receive fewer output tokens than the oracle price and any disclosed fee schedule imply — a direct, per-swap loss of principal. Because the call requires no timelock and emits only a `BinAdditionalFeesUpdated` event (easily missed), the overcharge can persist undetected across many swaps.

### Likelihood Explanation

The pool admin is a valid, reachable actor who can call `setPoolBinAdditionalFees` at any time with no delay. The Smart Audit Pivots for this contest explicitly flag `setBinAdditionalFees` as a bypass surface and state the pool admin is "semi-trusted only inside caps and timelocks." The absence of any cap on this path is therefore a reachable, in-scope trigger.

### Recommendation

Add a cap check in `setPoolBinAdditionalFees` analogous to the one in `setPoolAdminFees`. Define a `maxAdminBinAdditionalFeeE6` constant (or a factory-owner-settable cap) and enforce it:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (addFeeBuyE6 > maxAdminBinAdditionalFeeE6) revert AdminFeeTooHigh();
    if (addFeeSellE6 > maxAdminBinAdditionalFeeE6) revert AdminFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

Apply the same validation inside `_unpackAndValidateBinStates` for the initial bin data supplied at pool creation.

### Proof of Concept

1. Pool is created with `maxAdminSpreadFeeE6 = 10_000` (1 %) and `maxAdminNotionalFeeE8 = 0`.
2. Pool admin calls `factory.setPoolAdminFees(pool, 20_000, 0)` → reverts `AdminFeeTooHigh`. Cap is enforced.
3. Pool admin calls `factory.setPoolBinAdditionalFees(pool, 0, 65535, 65535)` → **succeeds**. No cap check.
4. A trader swaps token1 → token0 through bin 0. The effective buy fee passed to `SwapMath.buyToken0InBinSpecifiedIn` is `baseFeeX64 + mulDiv(65535, ONE_X64, 1e6)` — approximately 6.55 % above the oracle mid-price, far exceeding the 1 % cap the protocol owner intended.
