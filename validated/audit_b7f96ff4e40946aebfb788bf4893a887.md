### Title
Missing Zero-Address Validation for L1→L2 Refund Recipient Burns User Funds - (File: `basic_bootloader/src/bootloader/transaction_flow/zk/process_l1_transaction.rs`)

### Summary
The ZKsync OS bootloader processes L1→L2 transactions and mints a gas refund to a caller-supplied `refund_recipient` address (`transaction.reserved[1]`). There is no zero-address guard on this field. If the refund recipient is the zero address — whether by user error or a bug in the L1 bridge — the refund amount is permanently burned to `address(0)` with no revert or warning.

### Finding Description
In `process_l1_transaction`, after computing the gas refund amount, the bootloader reads the refund recipient directly from the transaction's `reserved[1]` field and passes it to `mint_base_token` without any zero-address check:

```rust
// process_l1_transaction.rs ~line 336-359
if to_refund_recipient > U256::ZERO {
    let refund_recipient = u256_to_b160_checked(transaction.reserved[1].read());
    mint_base_token::<S, Config>(
        system,
        system_functions,
        memories.reborrow(),
        &to_refund_recipient,
        &refund_recipient,   // ← no check: could be B160::ZERO
        l1_chain_id,
        &mut inf_resources,
        tracer,
        validator,
    )
``` [1](#0-0) 

`u256_to_b160_checked` simply extracts the lower 20 bytes of the `U256`; a zero input produces `B160::ZERO` with no error. The `mint_base_token` call then credits the refund to `address(0)`, permanently destroying those tokens.

The existing integration test `test_treasury_based_token_distribution_regression` explicitly sets `refund_recipient = address(0)` and the block executes successfully, confirming the code path is live and unguarded: [2](#0-1) 

The analogous value-mint path (minting the deposit to `from`) does not have this problem because `from` is validated elsewhere. Only the refund-recipient path is unguarded.

### Impact Explanation
Any L1→L2 transaction whose `reserved[1]` field is zero (zero address) will have its entire unused-gas refund permanently burned to `address(0)`. The treasury is debited, the tokens leave the treasury, and no user account is credited. The funds are irrecoverably lost. This is a direct, permanent loss of user funds with no recovery path.

### Likelihood Explanation
The `reserved[1]` field is set by the L1 bridge contract or directly by the transaction submitter. A user who omits or zero-initialises this field (a common mistake when constructing raw transactions or interacting with a front-end that leaves the field blank) will silently lose their refund. The ZKsync OS bootloader performs no defensive check, and the existing test suite demonstrates the zero-address path succeeds without error.

### Recommendation
Add a zero-address guard before calling `mint_base_token` for the refund recipient. If the refund recipient is `B160::ZERO`, either revert the refund step, default to the transaction `from` address, or skip the mint (effectively burning intentionally only when explicitly desired):

```rust
if to_refund_recipient > U256::ZERO {
    let refund_recipient = u256_to_b160_checked(transaction.reserved[1].read());
    // Guard: treat zero address as "refund to sender"
    let refund_recipient = if refund_recipient == B160::ZERO {
        from
    } else {
        refund_recipient
    };
    mint_base_token::<S, Config>(..., &refund_recipient, ...)?;
}
```

### Proof of Concept
1. Submit an L1→L2 priority transaction with `reserved[1] = U256::ZERO` (zero refund recipient), a non-zero `gas_limit`, and a `gas_price > 0`.
2. The transaction executes (succeeds or reverts — both paths reach the refund step).
3. `to_refund_recipient = gas_price * (gas_limit - gas_used) > 0`.
4. `refund_recipient = u256_to_b160_checked(0) = B160::ZERO`.
5. `mint_base_token(..., &to_refund_recipient, &B160::ZERO, ...)` is called.
6. The treasury balance decreases by `to_refund_recipient`; `address(0)` balance increases by the same amount.
7. The user receives no refund; the funds are permanently burned.

The existing test `test_treasury_based_token_distribution_regression` already demonstrates this exact scenario passing without error, with `refund_recipient_final_balance` increasing at `address(0)`. [3](#0-2) [4](#0-3)
