### Title
ETH Permanently Blackholed in `tick()` When No Pending Mint Requests Exist - (File: `L1/starkware/solidity/stake/RewardSupplier.sol`)

---

### Summary

The `tick()` function in `RewardSupplier.sol` is declared `external payable` and accepts ETH intended as L1→L2 message fees. However, the ETH is only consumed inside a conditional block guarded by `amountToMint > 0`. If no pending L2 mint requests exist at execution time, `msg.value` is silently retained by the contract with no refund path and no ETH recovery function, permanently blackholing the caller's ETH.

---

### Finding Description

`tick()` splits `msg.value` across two L1→L2 calls: [1](#0-0) 

The entire fee-spending logic is wrapped in `if (amountToMint > 0)`: [2](#0-1) 

When `amountToMint == 0`, the function returns without touching `msg.value`. The contract has no `receive()`, `fallback()`, or ETH-sweep function. The ETH is irrecoverably locked.

This mirrors the original report's root cause class: a cross-chain messaging entry point that accepts ETH for fees but fails to return it under a reachable condition, permanently blackholing the caller's funds.

---

### Impact Explanation

Any ETH sent to `tick()` when `requiredMinting()` returns zero is permanently lost. Because `tick()` has no access control, a front-runner can race ahead of a legitimate caller:

1. Caller submits `tick{value: X ETH}()` after observing pending L2 mint messages.
2. Attacker front-runs with their own `tick{value: minimal ETH}()`, consuming all pending messages.
3. Caller's transaction executes; `amountToMint == 0`; `X ETH` is trapped in the contract forever.

This constitutes **griefing with no profit motive but direct, irreversible damage to the caller** — matching the Medium impact tier.

---

### Likelihood Explanation

- `tick()` is permissionless (`external payable`, no role check). [3](#0-2) 
- Ethereum's public mempool makes front-running straightforward.
- Legitimate callers (bots, keepers, or any address) are expected to send ETH to cover the two L1→L2 fees; the protocol design invites this usage.
- The race window exists every time pending messages are near-zero or are consumed between a caller's off-chain check and on-chain execution.

---

### Recommendation

Add an early revert when there is nothing to process but ETH was sent:

```solidity
function tick() external payable {
    (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();
    if (amountToMint == 0) {
        require(msg.value == 0, "NO_WORK_ETH_REFUND_UNSUPPORTED");
        return;
    }
    // ... existing logic
}
```

Alternatively, refund `msg.value` to `msg.sender` when `amountToMint == 0`.

---

### Proof of Concept

1. L2 `RewardSupplier` emits one pending mint-request message to L1.
2. Alice's keeper bot detects the message and submits `tick{value: 0.01 ETH}()`.
3. Bob (attacker) sees Alice's pending transaction in the mempool and front-runs with `tick{value: 1 wei}()`, consuming the single pending message.
4. Alice's transaction is mined next: `requiredMinting()` returns `(0, 0)`; the `if` block is skipped; `0.01 ETH` is permanently locked in `RewardSupplier`.
5. No function exists to recover the ETH. Alice's `0.01 ETH` is blackholed.

        // Check if minting is required, and how much.
        (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

        if (amountToMint > 0) {
            // Prepare the L2->L1 mintRequest message for consumption.
            uint256[] memory messagePayload = new uint256[](1);
            messagePayload[0] = TOKENS_PER_MINT_REQUEST;

            // Consume the mintRequest messages.
            for (uint256 i = 0; i < numMsgsToConsume; i++) {
                messagingContract().consumeMessageFromL2(mintRequestSource(), messagePayload);
            }

            // Reuest minting of the requested amount from the mint manager.
            mintManager().mintRequest(token(), amountToMint);

            // Deposit the minted amount onto the bridge to the credit of `mintDestination`.
            uint256 msgFee = msg.value / 2;
            bridge().depositWithMessage{value: msgFee}(
                token(),
                amountToMint,
                mintDestination(),
                new uint256[](0)
            );
            emit ConsumedL2MintRequests(numMsgsToConsume, amountToMint);

            // Send a totalSupply update to L2MintCurve.
            msgFee = msg.value - msgFee;
            messagePayload[0] = IERC20(token()).totalSupply();
            messagingContract().sendMessageToL2{value: msgFee}(
                mintingCurve(),
                UPDATE_TOTAL_SUPPLY_SELECTOR,
                messagePayload
            );
