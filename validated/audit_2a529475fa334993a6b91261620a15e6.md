### Title
Unimplemented `TurnBridge` stub silently succeeds for all callers, breaking the bridge circuit breaker — (`x/cronos/keeper/msg_server.go`)

### Summary

`msgServer.TurnBridge` is registered in the live `MsgServer` but its body is a bare `return nil, nil`. It performs no permission check and no state mutation. Every caller — privileged or not — receives a success response while the bridge state is left unchanged.

### Finding Description

`UpdateTokenMapping` (the sibling restricted message) correctly gates execution behind `HasPermission`: [1](#0-0) 

`TurnBridge`, which is supposed to require the same `CanTurnBridge` permission, is instead a stub: [2](#0-1) 

The permission constant and the `HasPermission` helper exist and are tested: [3](#0-2) [4](#0-3) 

They are simply never called from `TurnBridge`. The message is fully wired into the gRPC service descriptor and reachable by any on-chain transaction: [5](#0-4) 

`DummyTracer` in `tracer.go` is an unrelated EVM-logger no-op and plays no role in this call path.

### Impact Explanation

Two distinct harms follow from the stub:

1. **Authorization bypass** — any unprivileged address can submit `MsgTurnBridge` and receive a `code 0` success response. The permission system designed for this message is entirely inert.

2. **Broken circuit breaker** — even a legitimately permissioned admin or the `CronosAdmin` address cannot disable the bridge. The transaction succeeds, the caller believes the bridge is off, but the gravity bridge continues processing batches. In an emergency (e.g., an active bridge exploit), the intended kill-switch produces a false success and leaves the bridge running.

The ADR explicitly describes `MsgTurnBridge` as the emergency stop for the gravity bridge: [6](#0-5) 

### Likelihood Explanation

The message is live and reachable by any transaction sender. No special setup is required. The integration test confirms the intent — `turn_bridge("false", from_="community")` is expected to fail with a non-zero code — but the production handler makes it succeed for everyone: [7](#0-6) 

### Recommendation

Replace the stub with the full implementation: check `HasPermission(ctx, msg.GetSigners(), CanTurnBridge)`, return `ErrUnauthorized` on failure, and on success call the gravity keeper to set the bridge-enabled flag and emit the appropriate event — mirroring the pattern used in `UpdateTokenMapping`.

### Proof of Concept

```go
// unit test — no privileged setup needed
func (suite *KeeperTestSuite) TestTurnBridgeStub() {
    suite.SetupTest()
    srv := keeper.NewMsgServerImpl(suite.app.CronosKeeper)

    // random unprivileged sender
    priv, _ := ethsecp256k1.GenerateKey()
    sender := sdk.AccAddress(common.BytesToAddress(priv.PubKey().Address().Bytes()).Bytes())

    msg := types.NewMsgTurnBridge(sender.String(), false)
    resp, err := srv.TurnBridge(suite.ctx, msg)

    // Both assertions pass against the current stub:
    suite.Require().NoError(err)   // succeeds — no permission check
    suite.Require().Nil(resp)      // nil response, no state written
    // Bridge is still enabled; no gravity keeper call was made.
}
```

```go
func (k Keeper) HasPermission(ctx sdk.Context, accounts []sdk.AccAddress, permissionsToCheck uint64) bool {
	// case when no permission is needed
	if permissionsToCheck == 0 {
		return true
	}
	admin := k.GetParams(ctx).CronosAdmin
	for _, account := range accounts {
		if admin == account.String() {
			return true
		}
		permission := k.GetPermissions(ctx, account)
		if permission&permissionsToCheck == permissionsToCheck {
			return true
		}
	}

	return false
```
