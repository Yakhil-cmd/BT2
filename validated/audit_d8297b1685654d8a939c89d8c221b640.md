### Title
Hardcoded 2,300-Gas Stipend in `NewDepositCall`/`NewTransferCall` Prevents Smart Contract Wallets from Receiving FLOW Tokens — (File: `fvm/evm/types/call.go`)

---

### Summary

The Flow EVM bridge hardcodes a gas limit of `21,000 + 2,300 = 23,300` for all deposit and transfer direct calls. After the 21,000 intrinsic gas is consumed, only 2,300 gas remains for the recipient's `receive`/`fallback` function — the same stipend as Solidity's deprecated `transfer` opcode. Any EVM smart contract (including COAs, which are explicitly described as smart contract wallets) with a `receive` function requiring more than 2,300 gas will fail to receive FLOW tokens via the protocol-level bridge deposit and transfer paths.

---

### Finding Description

In `fvm/evm/types/call.go`, the gas limits for all token-transfer direct calls are hardcoded:

```go
IntrinsicFeeForTokenTransfer    = gethParams.TxGas              // 21,000
DefaultGasLimitForTokenTransfer = IntrinsicFeeForTokenTransfer + 2_300  // 23,300
DepositCallGasLimit             = DefaultGasLimitForTokenTransfer
WithdrawCallGasLimit            = DefaultGasLimitForTokenTransfer
```

The code comment explicitly acknowledges the design intent:
> `// 21_000 is the minimum for a transaction + max gas allowed for receive/fallback methods`

These constants are consumed by three protocol-level call constructors:

- `NewDepositCall` — used in `handler.go` `Account.Deposit()` to credit FLOW to an EVM address
- `NewWithdrawCall` — used in `handler.go` `Account.Withdraw()` to move FLOW from a COA to the bridge
- `NewTransferCall` — used in `handler.go` `Account.Transfer()` to move FLOW between EVM accounts [1](#0-0) 

After the 21,000 intrinsic gas is consumed, only 2,300 gas remains for the recipient's `receive` or `fallback` function. COAs are explicitly described as smart contract wallets implementing ERC-165, ERC-721, ERC-777, ERC-1155, and ERC-1271: [2](#0-1) 

Their `receive` function may require more than 2,300 gas. When `Deposit()` or `Transfer()` targets a COA or any other smart contract with non-trivial receive logic, the EVM call fails with `ErrExecutionReverted`.

The `Account.Deposit()` implementation in `handler.go` constructs the deposit call with the hardcoded `DepositCallGasLimit`: [3](#0-2) 

The `Account.Transfer()` implementation uses `NewTransferCall` with `DefaultGasLimitForTokenTransfer`: [4](#0-3) 

The existing test at `fvm/evm/emulator/emulator_test.go:83–111` confirms this behavior: depositing to a contract whose `receive` reverts yields `VMError = gethVM.ErrExecutionReverted` with no validation error: [5](#0-4) 

---

### Impact Explanation

Any EVM smart contract — including COAs, which are the primary smart contract wallet type on Flow EVM — whose `receive` function requires more than 2,300 gas cannot receive FLOW tokens via the bridge `Deposit()` or `Transfer()` paths. If `panicOnErrorOrInvalidOrFailedState` does not revert the Cadence transaction on EVM VM errors (as distinct from validation errors), FLOW tokens already consumed from the Cadence vault will not be credited to the EVM address, resulting in **cross-VM asset loss**. Even if the Cadence transaction does revert, users are permanently denied the ability to deposit FLOW to their smart contract wallet via the standard bridge path, which is the only protocol-provided mechanism.

---

### Likelihood Explanation

COAs implement multiple ERCs and their `receive` function may perform non-trivial work (event emission, storage updates, ERC-777 hooks). Any user who deploys a smart contract with a `receive` function exceeding 2,300 gas and attempts to receive FLOW via `Deposit()` or `Transfer()` will trigger this. The entry path is fully unprivileged: any Cadence transaction calling `coa.deposit(from: <-vault)` or `coa.call(...)` with value targeting such a contract is sufficient. Likelihood is **medium** given COAs are the primary smart contract wallet type and the 2,300 gas stipend is well-known to be insufficient for modern smart contract wallets.

---

### Recommendation

Increase the gas limit for `NewDepositCall` and `NewTransferCall` to a value sufficient for smart contract wallet receive logic (e.g., 100,000 gas, consistent with OpenZeppelin's `Address.sendValue`). The gas limit should not be hardcoded to `21,000 + 2,300`; instead, forward all available gas or use a configurable higher default. Ensure reentrancy protection is in place if the gas limit is increased.

---

### Proof of Concept

1. Deploy an EVM smart contract whose `receive()` function requires more than 2,300 gas (e.g., emits an event or writes to storage).
2. From a Cadence transaction, call `coa.deposit(from: <-vault)` targeting that contract's EVM address, or call `coa.call(to: contractAddress, data: [], gasLimit: ..., value: bal)` with a non-zero value.
3. The `NewDepositCall` / `NewTransferCall` is constructed with `GasLimit = 23,300`.
4. After 21,000 intrinsic gas, only 2,300 gas reaches the `receive` function; it runs out of gas and reverts.
5. The EVM result carries `VMError = ErrExecutionReverted`; `panicOnErrorOrInvalidOrFailedState` is called.
6. If the Cadence transaction does not fully revert, the FLOW tokens are consumed from the vault but never credited to the EVM address — cross-VM asset loss. [6](#0-5) [7](#0-6)
