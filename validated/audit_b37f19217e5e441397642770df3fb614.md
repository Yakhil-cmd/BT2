Looking at the critical indexing logic in `do_generation_many`:

The vulnerability claim is concrete and traceable. Here is the analysis:

**The exact vulnerable code** is at lines 180–183 of `generation.rs`:

```rust
let (from, commitments): (_, Vec<_>) = chan.recv(wait0).await?;
for i in 0..N {
    all_commitments_vec[i].put(from, commitments[i]); // ← panics if commitments.len() < N
}
``` [1](#0-0) 

There is **no length check** on `commitments` before indexing. If a malicious participant sends a `Vec<Commitment>` with fewer than `N` elements (e.g., an empty vec), `commitments[i]` is a Rust slice index operation that **panics** — it does not return a `Result`. This is not a graceful error path.

The `ParticipantMap::put` method is safe (it silently ignores unknown participants), but the panic happens before `put` is even reached. [2](#0-1) 

The `do_generation` entry point calls `do_generation_many::<1>`, so N=1 in the standard single-triple path. A malicious participant sending an empty `Vec<Commitment>` at `wait0` triggers `commitments[0]` on an empty vec — an immediate panic. [3](#0-2) 

The protocol is run cooperatively without a tokio runtime (per the README: "polling it cooperatively without requiring a tokio or other async runtime"), so the panic propagates directly to the caller's poll site — it is not caught and converted to an error.

### Title
Unchecked `Vec<Commitment>` length in round 1 causes panic and permanent DoS of triple generation — (`src/ecdsa/ot_based_ecdsa/triples/generation.rs`)

### Summary
In `do_generation_many`, a malicious participant can send a `Vec<Commitment>` of length ≠ N during round 1 (`wait0`). The receiving honest party panics at `commitments[i]` (Rust slice index, not a `Result`), permanently aborting triple generation for all honest parties in that session.

### Finding Description
At lines 180–183 of `generation.rs`, after receiving a message deserialized as `Vec<Commitment>`, the code immediately indexes into it with `commitments[i]` for `i in 0..N` without first asserting `commitments.len() == N`. A malicious participant can send a zero-length (or otherwise short) vec. Rust's `[]` operator panics on out-of-bounds access. Since the protocol future is polled cooperatively (not inside a `tokio::spawn` that would catch panics), the panic unwinds to the caller, terminating the honest party's protocol execution. [4](#0-3) 

### Impact Explanation
Triple generation is permanently aborted for every honest party that receives the malicious message. The session cannot be recovered; a new session must be initiated. This matches the **High** impact: "Permanent denial of signing, key generation, reshare, refresh, or CKD for honest parties under valid protocol inputs and documented trust assumptions."

### Likelihood Explanation
Any participant in the triple generation protocol can trigger this. No special privilege is required — only the ability to send a protocol message at `wait0`. The attack is trivially reproducible.

### Recommendation
Add an explicit length check immediately after receiving the commitment vector:

```rust
let (from, commitments): (_, Vec<_>) = chan.recv(wait0).await?;
if commitments.len() != N {
    return Err(ProtocolError::AssertionFailed(format!(
        "expected {N} commitments from {from:?}, got {}", commitments.len()
    )));
}
for i in 0..N {
    all_commitments_vec[i].put(from, commitments[i]);
}
```

The same pattern should be applied to all other rounds that receive `Vec`s indexed by `i in 0..N` (e.g., `a_j_i_v[i]`, `b_j_i_v[i]`, `big_c_j_v[i]`, `their_phi_proofs[i]`, `c_j_i_v[i]`). [5](#0-4) [6](#0-5) [7](#0-6) 

### Proof of Concept
A unit test where one participant's `wait0` message is replaced with an empty `Vec<Commitment>` (or a vec of length 0) would demonstrate that all other honest parties panic rather than returning a `ProtocolError`.
