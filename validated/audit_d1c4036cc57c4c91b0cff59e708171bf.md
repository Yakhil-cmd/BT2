### Title
Pool Member's Remaining Funds Temporarily Frozen When Staker Is Removed While a Partial Exit Intent Is In Progress - (`src/pool/pool.cairo`)

---

### Summary

When a pool member has submitted a **partial** `exit_delegation_pool_intent` (i.e., `unpool_amount < total balance`, so `unpool_time` is set), and the staker is subsequently fully removed via `unstake_action`, the pool member's **remaining balance** (the portion not covered by the original intent) is transferred into the pool contract by `transfer_to_pools_when_unstake` but becomes **inaccessible** until the original intent window expires. The guard in `undelegate_from_staking_contract_intent` that prevents a new intent while `unpool_time.is_some()` is correct for the active-staker case but over-broad for the removed-staker case, where the funds are already sitting in the pool contract and no staking-contract-side accounting update is needed.

---

### Finding Description

**Root cause — `src/pool/pool.cairo`, `undelegate_from_staking_contract_intent`:**

```
fn undelegate_from_staking_contract_intent(
    self: @ContractState, pool_member: ContractAddress, amount: Amount,
) -> Timestamp {
    if !self.is_staker_active() {
        assert!(
            self.internal_pool_member_info(:pool_member).unpool_time.is_none(),
            "{}",
            Error::UNDELEGATE_IN_PROGRESS,
        );
        return Time::now();
    }
    ...
``` [1](#0-0) 

When `staker_removed == true`, the function allows a new intent **only if** `unpool_time.is_none()`. If the pool member already has an in-progress intent (`unpool_time.is_some()`), the call reverts with `UNDELEGATE_IN_PROGRESS` regardless of the requested amount — including `amount = 0` (cancellation) and amounts targeting the *remaining* balance that was never part of the original intent.

**Why the remaining balance is already in the pool contract:**

`unstake_action` calls `transfer_to_pools_when_unstake`, which iterates over all pools and transfers the staker's recorded delegated balance (the portion *not* in any exit intent) directly to the pool contract:

```
token_dispatcher.checked_transfer(
    recipient: pool_contract,
    amount: pool_balance.to_native_amount(:decimals).into(),
);
``` [2](#0-1) 

After this transfer, the pool contract holds the remaining funds, but the pool member cannot withdraw them because `exit_delegation_pool_intent` always routes through `undelegate_from_staking_contract_intent`, which blocks the call. [3](#0-2) 

---

### Impact Explanation

A pool member's funds that are **already inside the pool contract** are inaccessible for up to `exit_wait_window` (default 1 week; maximum `MAX_EXIT_WAIT_WINDOW = 12 weeks`). [4](#0-3) 

The member must wait for the original intent's `unpool_time` to expire, call `exit_delegation_pool_action` to collect the original intent amount, and only then can they call `exit_delegation_pool_intent` again for the remaining balance. During this entire window the remaining principal is frozen.

This matches the allowed impact: **Temporary freezing of funds**.

---

### Likelihood Explanation

The scenario requires two ordinary, unprivileged actions in sequence:

1. A pool member calls `exit_delegation_pool_intent` with an amount **less than** their total balance (a partial exit — e.g., to partially rebalance while keeping some stake active).
2. The staker subsequently calls `unstake_intent` + `unstake_action` (a normal, permissionless staker lifecycle action).

Neither step requires a privileged role, a leaked key, or any external dependency. Partial exits are a documented and expected use-case. Staker removal is a routine event. The combination is realistic and likely to occur in production.

---

### Recommendation

Inside `undelegate_from_staking_contract_intent`, when the staker is already removed, the guard should be relaxed to allow a new intent even when `unpool_time.is_some()`, because the funds are already in the pool contract and no staking-contract-side accounting is required. The fix mirrors the Compound M-13 recommendation: detect the "deprecated/removed" state and bypass the normal validation that is only meaningful while the staker is active.

```diff
fn undelegate_from_staking_contract_intent(
    self: @ContractState, pool_member: ContractAddress, amount: Amount,
) -> Timestamp {
    if !self.is_staker_active() {
-       assert!(
-           self.internal_pool_member_info(:pool_member).unpool_time.is_none(),
-           "{}",
-           Error::UNDELEGATE_IN_PROGRESS,
-       );
+       // Staker is removed; funds are already in this pool contract.
+       // Allow the member to set (or reset) their intent freely.
        return Time::now();
    }

``` [5](#0-4)

### Proof of Concept

**State before attack:**
- Pool member holds 100 STRK in the pool (`get_last_member_balance = 100`).
- `unpool_amount = 0`, `unpool_time = None`.

**Step 1 — Partial exit intent (pool member, unprivileged):**
```
pool.exit_delegation_pool_intent(amount: 30)
```
Result: `unpool_amount = 30`, `unpool_time = now + exit_wait_window`, member balance = 70.
The staking contract records an `UndelegateIntentKey` for 30 STRK.

**Step 2 — Staker removes itself (staker, unprivileged):**
```
staking.unstake_intent()
staking.unstake_action(staker_address)
```
`transfer_to_pools_when_unstake` transfers the remaining 70 STRK to the pool contract and calls `pool.set_staker_removed()`. [6](#0-5) 

**Step 3 — Pool member tries to exit remaining 70 STRK (pool member, unprivileged):**
```
pool.exit_delegation_pool_intent(amount: 70)
```
Execution path:
- `exit_delegation_pool_intent` → `undelegate_from_staking_contract_intent`
- `!is_staker_active()` → `true`
- `unpool_time.is_none()` → `false` (set in Step 1)
- **PANICS: `UNDELEGATE_IN_PROGRESS`** [7](#0-6) 

**Result:** 70 STRK sits in the pool contract, owned by the pool member, but unreachable until `unpool_time` (from Step 1) expires — up to `exit_wait_window` (max 12 weeks). Only after calling `exit_delegation_pool_action` to collect the 30 STRK can the member finally create a new intent for the remaining 70 STRK.

        fn undelegate_from_staking_contract_intent(
            self: @ContractState, pool_member: ContractAddress, amount: Amount,
        ) -> Timestamp {
            if !self.is_staker_active() {
                // Don't allow intent if an intent is already in progress and the staker is erased.
                // Avoid the following flow:
                // 1. Member intent - moves member balance from the staker's `pool_amount` to
                // `UndelegateIntentKey`.
                // 2. Staker intent.
                // 3. Staker action - transfer the `pool_amount` from the staker to the pool
                // contract.
                // 4. Member intent - here it is no longer possible to move balance between the
                // `pool_amount` and the `UndelegateIntentKey`.
                assert!(
                    self.internal_pool_member_info(:pool_member).unpool_time.is_none(),
                    "{}",
                    Error::UNDELEGATE_IN_PROGRESS,
                );
                return Time::now();
            }
            let staking_pool_dispatcher = self.staking_pool_dispatcher.read();
            let staker_address = self.staker_address.read();
            staking_pool_dispatcher
                .remove_from_delegation_pool_intent(
                    :staker_address, identifier: pool_member.into(), :amount,
                )
        }
