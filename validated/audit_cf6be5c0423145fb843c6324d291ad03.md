### Title
Token Balance Manipulation Suppresses L1 Mint Requests, Temporarily Freezing Reward Claims - (File: src/reward_supplier/reward_supplier.cairo)

### Summary

The `request_funds` internal function in `RewardSupplier` uses `balance_of(get_contract_address())` — the live ERC-20 balance — to decide whether to send a mint request to L1. Because any address can transfer STRK tokens directly to the `RewardSupplier` contract, an attacker can inflate the observed balance, suppress L1 mint requests, and eventually cause `claim_rewards` to fail when the donated tokens are exhausted.

### Finding Description

`request_funds` is called every time `update_unclaimed_rewards_from_staking_contract` is invoked (once per epoch by the staking contract). It computes:

```
credit = balance_of(self) + l1_pending_requested_amount
debit  = unclaimed_rewards
```

and only sends a mint request to L1 when `credit < debit + threshold`. [1](#0-0) 

The `balance` term is read directly from the ERC-20 contract: [2](#0-1) 

Because STRK is a standard ERC-20 token, anyone can call `transfer` to send tokens to the `RewardSupplier` address. This increases `balance_of(self)` without triggering `on_receive`, so `l1_pending_requested_amount` is **not** reduced. The net effect is that `credit` is artificially inflated, the `credit < debit + threshold` condition evaluates to `false`, and no mint request is emitted. [3](#0-2) 

The donated tokens are real and will be consumed by subsequent `claim_rewards` calls. Once they are exhausted, the contract's balance drops below what is needed to cover `unclaimed_rewards`, but no L1 mint request was ever sent during the donation window. The next epoch's `update_unclaimed_rewards_from_staking_contract` will finally trigger a request, but there is a mandatory L1→L2 message-processing delay before new tokens arrive. During that gap, `claim_rewards` attempts a `checked_transfer` that fails because the contract balance is insufficient, reverting the entire reward-distribution transaction. [4](#0-3) 

### Impact Explanation

Reward distribution in the staking contract calls `claim_rewards` on the `RewardSupplier`. If that call reverts (due to insufficient balance), the entire attestation/reward-update transaction reverts. Stakers and delegators receive no rewards for the affected epoch(s). This constitutes **temporary freezing of unclaimed yield** — a Medium-severity impact under the allowed scope.

### Likelihood Explanation

The attack requires the attacker to hold and donate STRK tokens. The donation amount needed equals roughly `unclaimed_rewards + threshold − balance − l1_pending_requested_amount`. For a protocol with significant staking activity this can be large, but a well-capitalised adversary (e.g., a competing validator or a short-seller) has a clear economic motive to disrupt reward distribution. The attack is repeatable every epoch and requires no privileged access.

### Recommendation

Replace the live `balance_of` read with a dedicated storage variable (e.g., `deposited_balance`) that is incremented only inside `on_receive` (the authorised bridge callback) and decremented when tokens are transferred out via `claim_rewards`. Use this internal accounting variable instead of `balance_of` in `request_funds`:

```cairo
// Instead of:
let balance = token_dispatcher.balance_of(account: get_contract_address());

// Use:
let balance = self.deposited_balance.read();
```

This mirrors the short-term recommendation from the original report: track balances with a storage variable that cannot be manipulated by arbitrary token transfers.

### Proof of Concept

1. Observe the current `unclaimed_rewards` (`U`), `balance` (`B`), `l1_pending_requested_amount` (`P`), and `threshold` (`T`) of the `RewardSupplier`.
2. Compute the donation amount `D = U + T − B − P + 1`.
3. Call `STRK.transfer(reward_supplier_address, D)` from any account holding at least `D` STRK.
4. The next call to `update_unclaimed_rewards_from_staking_contract` (triggered by the staking contract during attestation) reads `balance_of(self) = B + D`, computes `credit = B + D + P ≥ U + T`, and skips the L1 mint request.
5. Over subsequent epochs, the `D` donated tokens are consumed by `claim_rewards` calls.
6. Once the balance drops below the amount needed for the next `claim_rewards` call, the `checked_transfer` inside `claim_rewards` reverts, causing the staking contract's reward-distribution transaction to revert.
7. No L1 mint request was sent during the donation window, so there are no incoming tokens. Rewards are frozen until the next epoch triggers a fresh mint request and L1 fulfils it. [5](#0-4)

        fn request_funds(ref self: ContractState, unclaimed_rewards: Amount) {
            // Read current balance.
            let token_dispatcher = self.token_dispatcher.read();
            let balance: Amount = token_dispatcher
                .balance_of(account: get_contract_address())
                .try_into()
                .expect_with_err(InternalError::BALANCE_ISNT_AMOUNT_TYPE);

            // Calculate credit, which is the contract's balance plus the amount already requested
            // from L1, and the debit, which is the unclaimed rewards.
            let mut l1_pending_requested_amount = self.l1_pending_requested_amount.read();
            let credit = balance + l1_pending_requested_amount;
            let debit = unclaimed_rewards;

            // If there isn't enough credit to cover the debit + threshold, request funds.
            let base_mint_amount = self.base_mint_amount.read();
            let threshold = compute_threshold(base_mint_amount);
            if credit < debit + threshold {
                let diff = debit + threshold - credit;
                let num_msgs = ceil_of_division(dividend: diff, divisor: base_mint_amount);
                let total_amount = num_msgs * base_mint_amount;
                for _ in 0..num_msgs {
                    self.send_mint_request_to_l1_reward_supplier();
                }
                self.emit(Events::MintRequest { total_amount, num_msgs });
                l1_pending_requested_amount += total_amount;
            }

            // Commit to storage the requested amount, which is now part of the credit.
            self.l1_pending_requested_amount.write(l1_pending_requested_amount);
        }
