### Title
Missing Proof-of-Correctness for Participant Contributions in CKD Protocol Allows Malicious Participant to Corrupt Derived Key Output - (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

The Confidential Key Derivation (CKD) coordinator blindly aggregates `(norm_big_y, norm_big_c)` shares received from participants with no cryptographic proof that each contribution was computed honestly. A single malicious participant can send arbitrary group elements, causing the coordinator to output a structurally valid but cryptographically incorrect `CKDOutput`. Any honest party that unmasks this output receives a wrong derived key.

---

### Finding Description

In `do_ckd_coordinator` (`src/confidential_key_derivation/protocol.rs`, lines 50–55), the coordinator receives one `CKDOutput` per participant and unconditionally adds each received `big_y` and `big_c` into the running aggregate:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
``` [1](#0-0) 

Each participant is supposed to compute their share as:

```
norm_big_y = lambda_i * (y_i * G1)
norm_big_c = lambda_i * (x_i * H(pk || app_id) + y_i * app_pk)
```

where `x_i` is their DKG private share and `y_i` is a fresh random scalar. [2](#0-1) 

There is no zero-knowledge proof, commitment, or any other mechanism that binds the received `(norm_big_y, norm_big_c)` to the participant's actual private share `x_i` committed during DKG, nor to a consistent randomness `y_i`. The coordinator has no way to distinguish a correctly computed share from an arbitrary pair of group elements.

This is structurally identical to the external report's Issue 1 (no enforcement that the liquidator provides the required repayment) and Issue 3 (no health/validity check before accepting the operation): the protocol proceeds and produces output regardless of whether participant contributions are valid.

---

### Impact Explanation

The coordinator outputs a `CKDOutput { big_y, big_c }` that is the sum of all contributions. If any participant substitutes arbitrary values, the aggregate no longer satisfies the invariant:

```
big_c - app_sk * big_y = msk * H(pk || app_id)
```

When the client calls `ckd_output.unmask(app_sk)` it receives a wrong group element — not the intended confidential derived key. [3](#0-2) 

The coordinator and client (honest parties) accept and act on a corrupted CKD output. This matches the allowed impact: **High — Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.**

---

### Likelihood Explanation

Any single participant in the CKD session can trigger this. No special privilege is required beyond being included in the `participants` list passed to `ckd()`. The attack is a single-message substitution with no detectable side-effect at the protocol layer, since the coordinator performs no verification before finalizing the output. [4](#0-3) 

---

### Recommendation

Each participant should accompany their `(norm_big_y, norm_big_c)` with a zero-knowledge proof of correct computation — specifically a proof of knowledge of `(x_i, y_i)` such that:

- `norm_big_y = lambda_i * y_i * G1`
- `norm_big_c = lambda_i * (x_i * H(pk || app_id) + y_i * app_pk)`
- `x_i * G2 = vk_share_i` (binding to the DKG-committed verification share)

The coordinator must verify all proofs before aggregating. Alternatively, a commitment-then-reveal scheme can be used to detect equivocation, though a full NIZK is preferable for binding to the DKG share.

---

### Proof of Concept

1. Honest participants P_1 … P_{n-1} compute and send correct `(norm_big_y_i, norm_big_c_i)`.
2. Malicious participant P_m sends `(G1::identity(), G1::identity())` (or any arbitrary pair).
3. Coordinator aggregates: `big_y = sum_{i≠m}(norm_big_y_i)`, `big_c = sum_{i≠m}(norm_big_c_i)` — both missing P_m's Lagrange-weighted contribution.
4. Client calls `unmask(app_sk)`: receives `big_c - app_sk * big_y`, which equals `(msk - lambda_m * x_m) * H(pk || app_id)` — a wrong key with no error or indication of failure.
5. Any system relying on the derived key (e.g., TEE secret provisioning) silently operates on an incorrect value. [5](#0-4)

pub fn ckd(
    participants: &[Participant],
    coordinator: Participant,
    me: Participant,
    key_pair: KeygenOutput,
    app_id: impl Into<AppId>,
    app_pk: PublicKey,
    rng: impl CryptoRngCore + Send + 'static,
) -> Result<impl Protocol<Output = CKDOutputOption>, InitializationError> {
    // not enough participants
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }

    // kick out duplicates
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }

    let comms = Comms::new();
    let chan = comms.shared_channel();

    let fut = run_ckd_protocol(
        chan,
        coordinator,
        me,
        participants,
        key_pair,
        app_id.into(),
        app_pk,
        rng,
    );
    Ok(make_protocol(comms, fut))
}
