### Title
WalletConnect `create_offer_for_ids` Trusts Attacker-Controlled `driver_dict` Royalty to Compute Displayed Spend Total - (File: packages/gui/src/electron/commands/parseCommandDisplay.ts)

### Summary
When a dApp sends `chia_wallet.create_offer_for_ids` via WalletConnect, the `royalty_percentage` used to compute the "Total Amount with Royalties" shown in the approval dialog is read directly from the attacker-controlled `driver_dict` parameter. The on-chain override via `nftGetInfo` is silently skipped whenever the NFT is not already in the user's wallet (i.e., the normal case when the user is buying an NFT), leaving the displayed spend total entirely under dApp control.

### Finding Description

`parseCommandDisplay` handles `chia_wallet.create_offer_for_ids` by calling `createOfferRoyaltyPercentages`, which reads `royalty_percentage` from the dApp-supplied `driver_dict` without any on-chain verification: [1](#0-0) 

The extracted royalty is then passed to `parseWalletDeltaItem` for each NFT asset. Inside that function, the code attempts to override the royalty with the on-chain value from `nftGetInfo`: [2](#0-1) 

The override only fires when `nftInfo.success` is `true` **and** `nft_info.data_uris` is present. When the NFT is not in the user's wallet — the normal state when a user is *buying* an NFT — `nft_get_info` returns `success: false`, so the condition is never satisfied and the dApp-supplied royalty is used unchanged. The project's own test suite explicitly models this scenario: [3](#0-2) 

The unchecked royalty then flows into `formatAmountWithRoyalties` / `withRoyaltyTotals`, which computes the `amountWithRoyalties` field shown to the user in the WalletConnect confirmation dialog: [4](#0-3) 

The confirmation dialog renders this as "Total Amount with Royalties": [5](#0-4) 

### Impact Explanation

A malicious dApp sets `royalty_percentage: 0` (or any arbitrary value) in the `driver_dict` it sends via WalletConnect. Because `nftGetInfo` returns `success: false` for NFTs not yet in the user's wallet, the approval dialog shows `amountWithRoyalties` equal to the base offer amount with no royalties. The user approves believing their total XCH spend is X. When the offer is later accepted on-chain, the real NFT royalty (e.g. 10%) is enforced by the puzzle hash, so the user actually spends X + royalties — more than the amount they approved. This is a direct, concrete asset loss caused by the GUI displaying the wrong spend total at the moment of user approval.

Conversely, a dApp can inflate `royalty_percentage` to an extreme value to make a legitimate offer appear to cost far more than it does, causing the user to reject it (denial-of-service on offer acceptance).

### Likelihood Explanation

Any dApp connected via WalletConnect can send `chia_wallet.create_offer_for_ids` with an arbitrary `driver_dict`. No special privilege is required beyond an established WalletConnect session. The vulnerable code path is exercised every time a user buys an NFT through a WalletConnect-connected dApp, which is the primary intended use case. The fallback to driver_dict royalty when `nftGetInfo` fails is not a transient error condition — it is the guaranteed outcome for NFTs not in the user's wallet.

### Recommendation

Replace the best-effort `nftGetInfo` override with a mandatory on-chain lookup. For `create_offer_for_ids`, the `royalty_percentage` used to compute `amountWithRoyalties` must come exclusively from `nftGetInfo` (on-chain data). If `nftGetInfo` returns `success: false` or throws, the dialog should either:
- Display the amount **without** an `amountWithRoyalties` field and show a warning that royalty information could not be verified, or
- Block approval until on-chain royalty data is available.

The dApp-supplied `driver_dict` royalty should never be used as the authoritative source for the displayed spend total.

### Proof of Concept

1. Attacker dApp establishes a WalletConnect session with the victim's Chia GUI.
2. Attacker sends:
   ```json
   {
     "command": "chia_wallet.create_offer_for_ids",
     "params": {
       "offer": { "1": "-1000000000000", "<nft_launcher_id>": "1" },
       "driver_dict": {
         "<nft_launcher_id>": {
           "type": "singleton",
           "launcher_id": "0x<nft_launcher_id>",
           "also": {
             "type": "metadata",
             "also": {
               "type": "ownership",
               "transfer_program": {
                 "type": "royalty transfer program",
                 "royalty_percentage": "0"
               }
             }
           }
         }
       }
     }
   }
   ```
3. `createOfferRoyaltyPercentages` reads `royalty_percentage: 0` from `driver_dict`. [6](#0-5) 
4. `parseWalletDeltaItem` calls `nftGetInfo("<nft_launcher_id>")`. Since the NFT is not in the user's wallet, it returns `{ success: false }`. The override block is skipped. [7](#0-6) 
5. `royaltyPercentagesForSide` collects `royaltyPercentage: 0` from the NFT item. `formatAmountWithRoyalties` computes `amountWithRoyalties = 1 XCH` (no royalties added). [8](#0-7) 
6. The approval dialog shows **"Total Amount with Royalties: 1 XCH"**. The real NFT has a 10% royalty. User approves.
7. When the offer is accepted on-chain, the user's wallet pays **1.1 XCH** — 0.1 XCH more than displayed at approval time.

function createOfferRoyaltyPercentages(
  walletDelta: WalletDelta,
  driverDict: Record<string, unknown>,
): AssetRoyaltyPercentages {
  const royaltyPercentages: AssetRoyaltyPercentages = {
    spending: {},
    receiving: {},
  };

  for (const assetId of Object.keys(walletDelta.spending)) {
    royaltyPercentages.spending[assetId] = royaltyPercentageForDriverInfo(
      driverDict[assetId] ?? driverDict[`0x${assetId}`],
    );
  }

  for (const assetId of Object.keys(walletDelta.receiving)) {
    royaltyPercentages.receiving[assetId] = royaltyPercentageForDriverInfo(
      driverDict[assetId] ?? driverDict[`0x${assetId}`],
    );
  }

  return royaltyPercentages;
}

function royaltyPercentagesForSide(lines: DisplayWalletDeltaItem[]): number[] {
  return lines
    .filter((line): line is Extract<DisplayWalletDeltaItem, { kind: 'nft' }> => line.kind === 'nft')
    .map((line) => line.royaltyPercentage)
    .filter(
      (royaltyPercentage): royaltyPercentage is number => royaltyPercentage !== undefined && royaltyPercentage > 0,
    );
}

function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],
): string | undefined {
  if (royaltyPercentages.length === 0 || line.kind === 'nft') {
    return undefined;
  }

  const splitAmount = amount / BigInt(royaltyPercentages.length);
  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;

  if (line.kind === 'xch') {
    return mojoToChiaLocaleString(totalAmount);
  }

  return mojoToCATLocaleString(totalAmount);
}

function withRoyaltyTotals(
  items: DisplayWalletDeltaItemWithKey[],
  amounts: Record<string, bigint>,
  oppositeSideLines: DisplayWalletDeltaItem[],
): DisplayWalletDeltaItem[] {
  const royaltyPercentages = royaltyPercentagesForSide(oppositeSideLines);

  return items.map(({ key, line }) => {
    if (line.kind === 'nft') {
      return line;
    }

    const amount = amounts[key];
    const amountWithRoyalties = amount ? formatAmountWithRoyalties(line, amount, royaltyPercentages) : undefined;

    return amountWithRoyalties ? { ...line, amountWithRoyalties } : line;
  });
}
