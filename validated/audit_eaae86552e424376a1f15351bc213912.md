Audit Report

## Title
SNS Governance `distribute_rewards` Settles Proposals Belonging to the Next Reward Round — (`File: rs/sns/governance/src/governance.rs`)

## Summary
`ready_to_be_settled_proposal_ids` in SNS governance captures `now` internally and returns all proposals whose voting deadline has passed as of the current wall-clock time. `distribute_rewards` calls this function and then computes `reward_event_end_timestamp_seconds` as the end of the most recently completed round, a value strictly ≤ `now`. Any proposal whose deadline falls in the gap `(reward_event_end_timestamp_seconds, now]` is swept into the current reward event one full round early, with its ballots permanently cleared and maturity incorrectly credited to voters in the current round.

## Finding Description
`ready_to_be_settled_proposal_ids` captures `now = self.env.now()` and filters proposals where `reward_status(now) == ReadyToSettle`: [1](#0-0) 

`distribute_rewards` calls this function unconditionally: [2](#0-1) 

It then computes the round boundary as: [3](#0-2) 

This value equals `reward_start + floor((now - reward_start) / round_duration) * round_duration`, which is always ≤ `now`. The gap `now − reward_event_end_timestamp_seconds` can be anywhere from 0 to `round_duration_seconds − 1` seconds. Any proposal whose `get_deadline_timestamp_seconds()` falls inside this gap satisfies `accepts_vote(now) == false`: [4](#0-3) 

and therefore returns `ReadyToSettle`: [5](#0-4) 

Such proposals are included in `considered_proposals`, their ballots are permanently cleared via `p.ballots.clear()`: [6](#0-5) 

and their `reward_event_end_timestamp_seconds` is set to the current round's boundary even though the proposal's deadline postdates that boundary.

NNS governance avoids this by passing a truncated round-boundary timestamp to `ready_to_be_settled_proposal_ids`: [7](#0-6) 

computed by `most_recent_fully_elapsed_reward_round_end_timestamp_seconds`, which floors `now` to the nearest completed-round boundary: [8](#0-7) 

NNS `ready_to_be_settled_proposal_ids` accepts this explicit parameter: [9](#0-8) 

SNS has no equivalent guard.

## Impact Explanation
This is a significant SNS governance security impact with concrete financial harm. Proposals settled one round early cause: (1) permanent ballot data loss — `p.ballots.clear()` is irreversible; (2) reward dilution for legitimate current-round voters, since the fixed rewards purse is spread across more voting-power shares than intended; (3) premature maturity accrual for neurons that voted on proposals whose deadlines fall in the gap window, giving those neurons a timing advantage over all other participants. This matches the allowed impact class: **High ($2,000–$10,000) — Significant SNS security impact with concrete user or protocol harm**, as it directly distorts the SNS reward accounting mechanism in a deterministic, repeatable manner affecting all SNS instances.

## Likelihood Explanation
The bug fires on every invocation of `distribute_rewards` where at least one proposal's deadline falls in the window `(reward_event_end_timestamp_seconds, now]`. Because `should_distribute_rewards` fires as soon as `seconds_since_last_reward_event > round_duration_seconds`: [10](#0-9) 

and proposals can close at any second, this window is non-empty in virtually every real reward distribution cycle. No special privilege is required beyond the ability to submit a proposal (standard SNS governance participation). The window size is bounded only by the timer granularity, not by any protocol guard.

## Recommendation
Mirror the NNS approach in SNS governance:
1. Add an `as_of_timestamp_seconds: u64` parameter to `ready_to_be_settled_proposal_ids` in `rs/sns/governance/src/governance.rs`.
2. Compute the round-boundary timestamp analogous to NNS `most_recent_fully_elapsed_reward_round_end_timestamp_seconds` — i.e., `reward_event_end_timestamp_seconds` (already computed at line 5837) — and pass it to `ready_to_be_settled_proposal_ids` instead of using `now` internally.
3. This ensures only proposals whose voting period closed on or before the end of the most recently completed reward round are settled in the current event.

## Proof of Concept
**Setup:**
- SNS with `round_duration_seconds = 604800` (7 days).
- Last reward event `end_timestamp_seconds = T₀`.
- Neuron A creates Proposal P with `initial_voting_period_seconds = 604801` at time `T₀ − 1`.
- Proposal P's deadline = `T₀ − 1 + 604801 = T₀ + 604800`.

**Execution:**
1. At `T₀ + 604800 + 2`, the governance timer fires `should_distribute_rewards` → `true`.
2. `distribute_rewards` runs: `new_rounds_count = floor(604802 / 604800) = 1`, `reward_event_end_timestamp_seconds = T₀ + 604800`.
3. `ready_to_be_settled_proposal_ids()` evaluates `reward_status(now = T₀ + 604800 + 2)`: Proposal P's deadline = `T₀ + 604800 < now` → `accepts_vote = false` → `ReadyToSettle`.
4. Proposal P is included in `considered_proposals` and settled under the current reward event with `end_timestamp_seconds = T₀ + 604800`, even though P closed 1 second into round 2.
5. Neuron A receives full proportional maturity for Proposal P in round 1. Ballots are permanently cleared.
6. In round 2, Proposal P is already `Settled`; no other neuron can earn rewards for it.

A deterministic unit test using `GovernanceCanisterFixtureBuilder` with controlled time (`set_start_time`) and a proposal deadline set 1 second past the round boundary would reproduce this exactly, verifiable against the existing SNS governance test infrastructure in `rs/sns/governance/tests/governance.rs`.

    pub fn ready_to_be_settled_proposal_ids(
        &self,
        as_of_timestamp_seconds: u64,
    ) -> impl Iterator<Item = ProposalId> + '_ {
        self.heap_data
            .proposals
            .iter()
            .filter(move |(_, proposal)| {
                let topic = proposal.topic();
                let voting_period_seconds = self.voting_period_seconds()(topic);
                let reward_status =
                    proposal.reward_status(as_of_timestamp_seconds, voting_period_seconds);

                reward_status == ProposalRewardStatus::ReadyToSettle
            })
            .map(|(k, _)| ProposalId { id: *k })
    }

    fn most_recent_fully_elapsed_reward_round_end_timestamp_seconds(&self) -> u64 {
        let now = self.env.now();
        let genesis_timestamp_seconds = self.heap_data.genesis_timestamp_seconds;

        if genesis_timestamp_seconds > now {
            println!(
                "{}WARNING: genesis is in the future: {} vs. now = {})",
                LOG_PREFIX, genesis_timestamp_seconds, now,
            );
            return 0;
        }

        (now - genesis_timestamp_seconds) // Duration since genesis (in seconds).
            / REWARD_DISTRIBUTION_PERIOD_SECONDS // This is where the truncation happens. Whole number of rounds.
            * REWARD_DISTRIBUTION_PERIOD_SECONDS // Convert back into seconds.
            + self.heap_data.genesis_timestamp_seconds // Convert from duration to back to instant.
    }
