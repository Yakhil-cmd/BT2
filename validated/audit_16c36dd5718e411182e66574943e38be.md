### Title
Unvalidated, Unconfirmed `targetPuzzleHash` from Malicious Pool Server Redirects Farming Rewards â€” (`packages/gui/src/components/plotNFT/select/PlotNFTSelectPool.tsx`)

### Summary
`prepareSubmitData` fetches `/pool_info` from the attacker-controlled pool URL, extracts `targetPuzzleHash` with only a truthiness check, and passes it verbatim to `pwJoinPool`. No format validation and no user-facing confirmation of the hash value occur anywhere in the GUI join flow.

### Finding Description

`prepareSubmitData` in `PlotNFTSelectPool.tsx` calls `getPoolInfo(normalizedPoolUrl)` and extracts `targetPuzzleHash` from the server response: [1](#0-0) 

The only guard is a truthiness check (`if (!targetPuzzleHash)`). There is no hex-format validation, no length check, and no confirmation dialog presented to the user showing which puzzle hash they are about to commit to.

`handleSubmit` in `PlotNFTChangePool.tsx` then passes the server-supplied value directly to `pwJoinPool`: [2](#0-1) 

`pwJoinPool` in `WalletService.ts` forwards it as-is to the wallet daemon RPC `pw_join_pool`: [3](#0-2) 

`PoolInfo.tsx` does render `targetPuzzleHash` in a display card, but that component is a read-only info panel; it is not wired into the submit confirmation path and the hash is only fetched at submit time (inside `prepareSubmitData`), so the user never sees it before the RPC fires. [4](#0-3) 

### Impact Explanation
In Chia's pooling protocol the `target_puzzle_hash` embedded in the pool-join singleton spend determines the address to which the pool sends farming rewards. An attacker who operates a pool server can return their own puzzle hash, causing every block reward earned by the victim's plots to be paid to the attacker's wallet. The user has no GUI mechanism to detect or prevent this.

### Likelihood Explanation
Any operator of a publicly reachable HTTPS endpoint can serve a crafted `/pool_info` JSON. The user only needs to enter the URL and click "Change" / "Create". No special privileges, no local access, and no cryptographic weakness are required.

### Recommendation
1. Validate `targetPuzzleHash` format (64-character lowercase hex string) before accepting it.
2. Display the fetched `targetPuzzleHash` in a `ConfirmDialog` before calling `pwJoinPool`, so the user can compare it against the pool's published value.
3. Fetch pool info eagerly (when the URL is entered) so the hash is visible before the submit button is active.

### Proof of Concept
1. Run a local HTTPS server at `https://attacker.pool/` returning:
   ```json
   {
     "target_puzzle_hash": "0xdeadbeef...attacker_hash...",
     "relative_lock_height": 32,
     "name": "Legit Pool"
   }
   ```
2. In the Chia GUI, navigate to a Plot NFT â†’ Change Pool, enter `https://attacker.pool/`.
3. Click "Change" â€” no confirmation of the hash is shown.
4. Observe `pw_join_pool` is called with `targetPuzzlehash = "0xdeadbeef...attacker_hash..."`.
5. All subsequent farming rewards for that NFT are paid to the attacker's address. [5](#0-4)

async function prepareSubmitData(data: FormData): SubmitData {
  const { self, fee, poolUrl } = data;
  const initialTargetState = {
    state: self ? 'SELF_POOLING' : 'FARMING_TO_POOL',
  };

  if (!self && poolUrl) {
    const normalizedPoolUrl = normalizeUrl(poolUrl);
    const { targetPuzzleHash, relativeLockHeight } = await getPoolInfo(normalizedPoolUrl);
    if (!targetPuzzleHash) {
      throw new Error(t`Pool does not provide targetPuzzleHash.`);
    }
    if (relativeLockHeight === undefined) {
      throw new Error(t`Pool does not provide relativeLockHeight.`);
    }

    initialTargetState.poolUrl = normalizedPoolUrl;
    initialTargetState.targetPuzzleHash = targetPuzzleHash;
    initialTargetState.relativeLockHeight = relativeLockHeight;
  }

  const feeMojos = chiaToMojo(fee || '0');

  return {
    fee: feeMojos,
    initialTargetState,
  };
}
