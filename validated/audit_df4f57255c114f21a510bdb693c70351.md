### Title
`ExitToEthereum` Precompile Burns EVM Tokens Without Failure Callback, Creating Permanent Fund Freeze and Bridge Accounting Discrepancy — (File: `engine-precompiles/src/native.rs`)

---

### Summary

The `ExitToEthereum` precompile burns ERC-20 or ETH tokens from the EVM state before dispatching a NEAR-side `withdraw` promise, but — unlike `ExitToNear` — attaches **no callback** to handle promise failure. If the `withdraw` call to the ETH connector fails for any reason, the user's EVM tokens are permanently destroyed with no refund path, creating a lasting discrepancy between the EVM token supply and the NEAR-side locked token balance. This is the direct Aurora analog to the Lyra `lockedCollateral.base` / sETH balance divergence.

---

### Finding Description

**Burn-before-promise with no failure handler in `ExitToEthereum`**

The `ExitToEthereum` precompile (`engine-precompiles/src/native.rs`, lines 844–1003) follows a strict burn-then-promise pattern:

1. **ERC-20 path**: `EvmErc20.sol`'s `withdrawToEthereum` calls `_burn(sender, amount)` *before* calling the precompile. [1](#0-0) 

2. **ETH base-token path**: the ETH value is consumed from `context.apparent_value` during EVM execution before the precompile returns.

3. The precompile then creates a `withdraw` promise to the ETH connector — always as a bare `PromiseArgs::Create` with **no callback**:

```rust
let promise = borsh::to_vec(&PromiseArgs::Create(withdraw_promise)).unwrap();
``` [2](#0-1) 

**Contrast with `ExitToNear`**, which conditionally attaches `exit_to_near_precompile_callback` to re-mint tokens if the NEAR-side transfer fails:

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs {
        base: transfer_promise,
        callback: PromiseCreateArgs {
            method: "exit_to_near_precompile_callback".to_string(),
            ...
        },
    })
};
``` [3](#0-2) 

The `exit_to_near_precompile_callback` handler in `connector.rs` explicitly re-mints burned tokens on failure: [4](#0-3) 

`ExitToEthereum` has no equivalent. The asymmetry is structural and unconditional — no feature flag can enable a refund path for the Ethereum exit.

**Accounting discrepancy created**

When the `withdraw` promise fails:
- EVM token supply: **decreased** (tokens burned, irreversible)
- NEAR-side locked tokens in the ETH connector: **unchanged** (withdraw rejected)

This is the direct analog to the Lyra bug:

| | Lyra | Aurora |
|---|---|---|
| State updated | `lockedCollateral.base += amount` | ERC-20 `_burn` / ETH consumed |
| Action skipped | swap not executed | `withdraw` promise fails |
| Discrepancy | `lockedCollateral.base` > sETH balance | EVM supply < NEAR locked tokens |
| Protocol effect | distorted liquidity values | bridge accounting divergence + fund freeze |

---

### Impact Explanation

**Permanent fund freeze (Critical)**: Any user who calls `withdrawToEthereum` on an `EvmErc20` contract, or sends ETH to the `ExitToEthereum` precompile, will have their EVM tokens permanently destroyed if the NEAR-side `withdraw` call fails. The user receives nothing on Ethereum and has no recovery path. The bridge accounting becomes permanently distorted: the EVM supply is lower than the NEAR-side locked balance, which can affect future bridge operations and total-supply invariants.

The `Accounting` struct in `engine/src/accounting.rs` tracks ETH supply changes per-transaction but has no cross-transaction reconciliation against the NEAR-side connector balance, so the divergence accumulates silently. [5](#0-4) 

---

### Likelihood Explanation

The `withdraw` call to the ETH connector can fail under realistic conditions:

1. **ETH connector paused**: The connector is a separate NEAR contract with its own pause mechanism. If it is paused for any security reason while a user's `ExitToEthereum` transaction is in flight, the `withdraw` promise fails and the user's tokens are burned with no refund.
2. **Gas exhaustion**: `WITHDRAWAL_GAS` is fixed at `100_000_000_000_000` TGas. If the connector's `withdraw` logic exceeds this budget, the promise fails silently. [6](#0-5) 
3. **Connector state inconsistency**: Any unexpected failure in the connector (e.g., storage issues, balance underflow) causes the promise to fail.

The `ExitToNear` callback was specifically added to handle exactly this class of failure (see `test_exit_to_near_refund` and `test_exit_to_near_eth_refund`), confirming the developers recognize the risk. The absence of an equivalent for `ExitToEthereum` is an oversight, not a design choice. [7](#0-6) 

---

### Recommendation

Add a failure callback to `ExitToEthereum` mirroring the `exit_to_near_precompile_callback` pattern:

1. Wrap the `withdraw_promise` in a `PromiseWithCallbackArgs` that calls a new `exit_to_ethereum_precompile_callback` method on the engine contract.
2. In the callback, check `handler.promise_result(0)`. If the promise failed, call `refund_on_error` to re-mint the burned ERC-20 tokens (or credit ETH back) to the user's address.
3. The `refund_on_error` function already supports both ETH and ERC-20 refund paths and can be reused directly. [8](#0-7) 

---

### Proof of Concept

**ERC-20 exit path:**

1. User holds 100 units of a bridged ERC-20 token (`EvmErc20`) on Aurora.
2. User calls `withdrawToEthereum(recipient_eth_address, 100)` on the `EvmErc20` contract.
3. `EvmErc20.sol` executes `_burn(msg.sender, 100)` — EVM balance decremented, irreversible within the transaction. [1](#0-0) 
4. `EvmErc20.sol` calls the `ExitToEthereum` precompile with the burn amount and Ethereum recipient.
5. The precompile constructs a `withdraw` promise targeting the NEP-141 contract with no callback attached. [2](#0-1) 
6. The NEAR runtime executes the `withdraw` promise. The ETH connector is paused (or fails for any other reason). The promise fails.
7. Because there is no callback, no `exit_to_ethereum_precompile_callback` is invoked. No re-mint occurs.
8. **Result**: User's 100 ERC-20 tokens are permanently burned. User receives 0 ETH on Ethereum. The NEAR-side NEP-141 balance held by Aurora is unchanged, creating a permanent discrepancy between the EVM token supply and the NEAR-side locked balance — the direct Aurora analog to the Lyra `lockedCollateral.base` / sETH divergence.

#[named]
pub fn exit_to_near_precompile_callback<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        env.assert_private_call()?;

        // This function should only be called as the callback of
        // exactly one promise.
        if handler.promise_results_count() != 1 {
            return Err(errors::ERR_PROMISE_COUNT.into());
        }

        let args: ExitToNearPrecompileCallbackArgs = io.read_input_borsh()?;

        let maybe_result = if let Some(PromiseResult::Successful(_)) = handler.promise_result(0) {
            if let Some(args) = args.transfer_near {
                let action = PromiseAction::Transfer {
                    amount: Yocto::new(args.amount),
                };
                let promise = PromiseBatchAction {
                    target_account_id: args.target_account_id,
                    actions: vec![action],
                };

                // Safety: this call is safe because it comes from the exit to near precompile, not users.
                // The call is to transfer the unwrapped wNEAR tokens.
                let promise_id = handler.promise_create_batch(&promise);
                handler.promise_return(promise_id);
            }

            None
        } else if let Some(args) = args.refund {
            // Exit call failed; need to refund tokens
            let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;

            if !refund_result.status.is_ok() {
                return Err(errors::ERR_REFUND_FAILURE.into());
            }

            Some(refund_result)
        } else {
            None
        };

        Ok(maybe_result)
    })

use aurora_engine_types::U256;
use core::cmp::Ordering;

/// This struct tracks changes to the supply of a U256 quantity.
/// It is used in our code to keep track of the total supply of ETH on Aurora.
/// This struct is intentionally designed to avoid doing subtraction as much as possible
/// to avoid complexities of signed values and over/underflow.
#[derive(Debug, Clone, Copy, PartialEq, Eq, Default)]
pub struct Accounting {
    gained: U256,
    lost: U256,
}

impl Accounting {
    pub fn change(&mut self, amount: Change) {
        match amount.new_value.cmp(&amount.old_value) {
            Ordering::Greater => {
                let net_gained = amount.new_value - amount.old_value;
                self.gained = self.gained.saturating_add(net_gained);
            }
            Ordering::Less => {
                let net_lost = amount.old_value - amount.new_value;
                self.lost = self.lost.saturating_add(net_lost);
            }
            Ordering::Equal => (),
        }
    }

    pub fn remove(&mut self, amount: U256) {
        self.lost = self.lost.saturating_add(amount);
    }

    #[must_use]
    pub fn net(&self) -> Net {
        match self.gained.cmp(&self.lost) {
            Ordering::Equal => Net::Zero,
            Ordering::Greater => Net::Gained(self.gained - self.lost),
            Ordering::Less => Net::Lost(self.lost - self.gained),
        }
    }
}

pub fn refund_on_error<I: IO + Copy, E: Env, P: PromiseHandler>(
    io: I,
    env: &E,
    state: EngineState,
    args: &RefundCallArgs,
    handler: &mut P,
) -> EngineResult<SubmitResult> {
    let current_account_id = env.current_account_id();
    if let Some(erc20_address) = args.erc20_address {
        // ERC-20 exit; re-mint burned tokens
        let erc20_admin_address = current_address(&current_account_id);
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, erc20_admin_address, current_account_id, io, env);

        let refund_address = args.recipient_address;
        let amount = U256::from_big_endian(&args.amount);
        let input = setup_refund_on_error_input(amount, refund_address);

        engine.call(
            &erc20_admin_address,
            &erc20_address,
            Wei::zero(),
            input,
            u64::MAX,
            Vec::new(),
            Vec::new(),
            handler,
        )
    } else {
        // ETH exit; transfer ETH back from precompile address
        let exit_address = exit_to_near::ADDRESS;
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, exit_address, current_account_id, io, env);
        let refund_address = args.recipient_address;
        let amount = Wei::new(U256::from_big_endian(&args.amount));
        engine.call(
            &exit_address,
            &refund_address,
            amount,
            Vec::new(),
            u64::MAX,
            vec![
                (exit_address.raw(), Vec::new()),
                (refund_address.raw(), Vec::new()),
            ],
            Vec::new(),
            handler,
        )
    }
