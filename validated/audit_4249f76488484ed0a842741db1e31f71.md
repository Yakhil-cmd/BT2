### Title
Missing EVM Gas Charge for LOG Opcode Allows Under-Priced Event Emission - (`File: evm_interpreter/src/instructions/host.rs`)

### Summary

The `log` function in the EVM interpreter charges **0 EVM gas** for the LOG opcode, despite the EVM specification requiring `375 + 375*N + 8*data_len` gas. The `LOG`, `LOGDATA`, and `LOGTOPIC` constants are defined in `gas_constants.rs` but are never applied in the `log` implementation. An attacker can deploy a contract that emits many large LOG events, consuming significant node resources (event storage, pubdata) while paying only memory expansion gas.

### Finding Description

In `evm_interpreter/src/instructions/host.rs`, the `log<const N: usize>` function handles all LOG0–LOG4 opcodes:

```rust
pub fn log<const N: usize>(...) -> InstructionResult {
    self.gas.spend_gas_and_native(0, LOG_NATIVE_COST)?;  // ← 0 EVM gas
    ...
    let (mem_offset, len) = self.stack.pop_2()?;
    ...
    self.resize_heap(mem_offset, len)?;   // only memory expansion gas charged
    system.emit_event(..., self.gas.resources_mut(), ...)?;
    Ok(())
}
``` [1](#0-0) 

The EVM gas argument is hardcoded to `0`. The EVM specification mandates:
- **Base cost**: `LOG = 375` gas
- **Per topic**: `LOGTOPIC = 375` gas × N
- **Per data byte**: `LOGDATA = 8` gas × `len`

All three constants exist in `gas_constants.rs` but are never referenced in the `log` function: [2](#0-1) 

The consistent pattern throughout the interpreter is to charge EVM gas explicitly via `spend_gas_and_native` before any system call. For example, `sload`, `sstore`, `create`, and `call` all charge their EVM gas explicitly. The `emit_event` system call receives `self.gas.resources_mut()` and charges native resources for pubdata, but does not apply EVM-layer LOG gas semantics. [3](#0-2) 

### Impact Explanation

An attacker deploys a contract that loops emitting LOG4 events with large data payloads. Under correct EVM pricing, a single `LOG4` with 10 KB of data costs `375 + 4*375 + 8*10240 = 83,375` gas. Under the current implementation, the same operation costs only the memory expansion gas (a few hundred gas for the first expansion, then zero for subsequent calls to the same memory region). This breaks the economic security model: node operators must process, store, and publish event data to L1 as pubdata, but the EVM gas paid does not reflect this cost. While the native resource (proving cost) is partially covered by `emit_event`'s internal native charge, the EVM gas layer — which determines what users pay — is systematically under-charged.

### Likelihood Explanation

Exploitation requires only deploying a standard EVM contract. No privileged access, oracle manipulation, or governance attack is needed. The attacker controls the number of topics (`N`) and data length (`len`) directly from the stack, making the under-pricing proportional to attacker choice. Any unprivileged transaction sender can trigger this path.

### Recommendation

Apply the standard EVM LOG gas formula in the `log` function before the system call:

```rust
pub fn log<const N: usize>(...) -> InstructionResult {
    let log_gas = gas_constants::LOG
        + gas_constants::LOGTOPIC * N as u64
        + gas_constants::LOGDATA * len as u64;
    self.gas.spend_gas_and_native(log_gas, LOG_NATIVE_COST)?;
    ...
}
```

Note that `len` must be resolved from the stack before the gas charge, consistent with how `calldatacopy` and `codecopy` handle dynamic-length gas.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

contract LogSpam {
    event BigEvent(
        bytes32 indexed t1, bytes32 indexed t2,
        bytes32 indexed t3, bytes32 indexed t4,
        bytes data
    );

    // Emits LOG4 with 10 KB of data.
    // Correct EVM cost: 375 + 4*375 + 8*10240 = 83,375 gas + memory expansion
    // Actual cost charged: ~96 gas (memory expansion only, first call)
    // Subsequent calls to same memory region: ~0 gas for memory expansion
    function exploit() external {
        bytes memory data = new bytes(10240);
        for (uint i = 0; i < 100; i++) {
            emit BigEvent(
                bytes32(uint256(i)), bytes32(uint256(i+1)),
                bytes32(uint256(i+2)), bytes32(uint256(i+3)),
                data
            );
        }
    }
}
```

Each iteration emits a LOG4 with 10 KB of data. The correct EVM gas per iteration is 83,375 gas; the actual gas charged is only the memory expansion cost (near zero after the first iteration since the heap is already sized). Over 100 iterations, the attacker pays ~8,000 gas instead of ~8,337,500 gas — a ~1000× discount — while forcing nodes to process and publish 1 MB of event data.

        &mut self,
        system: &mut System<S>,
        hooks: &mut HooksStorage<S, S::Allocator>,
        tracer: &mut impl Tracer<S>,
    ) -> InstructionResult {
        assert!(N <= MAX_EVENT_TOPICS);
        self.gas.spend_gas_and_native(0, LOG_NATIVE_COST)?;

        if self.is_static_frame() {
            return Err(EvmError::StateChangeDuringStaticCall.into());
        }

        let (mem_offset, len) = self.stack.pop_2()?;
        let (mem_offset, len) =
            Self::cast_offset_and_len(&mem_offset, &len, EvmError::InvalidOperandOOG.into())?;
        let mut topics: arrayvec::ArrayVec<Bytes32, 4> = arrayvec::ArrayVec::new();
        for _ in 0..N {
            topics.push(Bytes32::from_u256_be(self.stack.pop_1()?));
        }

        // resize memory

        let data = &self.heap[mem_offset..mem_offset + len];

        tracer.on_event(THIS_EE_TYPE, &self.address, &topics, data);

        system.emit_event(
            hooks,
            ExecutionEnvironmentType::EVM,
            self.gas.resources_mut(),
            &self.address,
            &topics,
            data,
        )?;

        Ok(())
    }
