### Title
`TurnBridge` Emergency Stop is a No-Op — Admin Bridge Authorization Permanently Bypassed (`x/cronos/keeper/msg_server.go`)

### Summary
The `TurnBridge` message handler returns `(nil, nil)` without performing any state change. The admin's ability to halt the Gravity Bridge in an emergency is permanently non-functional. Any unprivileged user can continue to bridge tokens even after the admin has successfully submitted a "disable bridge" transaction, because the transaction silently succeeds while doing nothing.

### Finding Description
`TurnBridge` in `x/cronos/keeper/msg_server.go` is a stub that returns success unconditionally:

```go
// TurnBridge implements the grpc method
func (k msgServer) TurnBridge(goCtx context.Context, msg *types.MsgTurnBridge) (*MsgTurnBridgeResponse, error) {
    return nil, nil
}
```

The proto definition at `proto/cronos/tx.proto` declares `MsgTurnBridge` with a `sender` and `enable` field, and the message is registered in the live gRPC service. The `CanTurnBridge` permission bit exists in `x/cronos/keeper/permissions.go` and is granted via `MsgUpdatePermissions`. The integration test `test_gravity_turn_bridge` explicitly asserts that after `turn_bridge("false")` succeeds, subsequent bridge transactions revert — an assertion that can never pass because no bridge-enabled state is ever written.

Contrast with `UpdateTokenMapping`, which performs a real permission check before acting:

```go
if !k.HasPermission(ctx, msg.GetSigners(), CanChangeTokenMapping) {
    return nil, errors.Wrap(sdkerrors.ErrUnauthorized, "msg sender is not authorized")
}
```

`TurnBridge` performs no permission check and no state write. The `MsgTurnBridge` transaction is accepted by the chain, emits a success code, and leaves the bridge fully operational.

The analog to the external report is exact: just as linked tophats have `eligibility` permanently frozen at `address(0)` so admins can never revoke wearer status, the Cronos bridge-disable path is permanently frozen as a no-op so the admin can never halt bridge flows.

### Impact Explanation
**High — Bypass of Cronos admin bridge authorization check.**

The `CanTurnBridge` permission and the `CronosAdmin` parameter exist specifically to give the admin an emergency circuit-breaker over the Gravity Bridge. Because `TurnBridge` is a no-op, that circuit-breaker is permanently inoperative. An unprivileged user exploiting a bridge vulnerability (e.g., unauthorized minting via a Gravity Bridge bug) can continue draining assets even after the admin has submitted and confirmed a disable transaction. The admin receives a false success signal and has no alternative on-chain mechanism to stop bridge flows.

### Likelihood Explanation
The bug is present in every execution of `MsgTurnBridge`. No special precondition is required. The admin will discover the failure only after observing that bridge transactions continue to succeed post-disable, at which point the window for emergency response has already closed.

### Recommendation
Implement `TurnBridge` with:
1. A `BridgeEnabled` boolean stored in the keeper's KV store (or as a module parameter).
2. A permission check (`HasPermission(ctx, msg.GetSigners(), CanTurnBridge)`) before writing state.
3. Guards in the Gravity Bridge EVM hook and IBC bridge handler that reject processing when `BridgeEnabled == false`.
4. Clearing the enabled flag on `enable=false` and restoring it on `enable=true`.

### Proof of Concept
1. Admin holds `CanTurnBridge` permission (granted via `MsgUpdatePermissions`).
2. Admin broadcasts `MsgTurnBridge{sender: admin, enable: false}` — chain returns `code: 0`.
3. No state is written; bridge remains fully operational.
4. Unprivileged user calls `send_to_evm_chain` on a CRC21 contract — transaction succeeds.
