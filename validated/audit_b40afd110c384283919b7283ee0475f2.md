### Title
Missing Participant-Set Consistency Validation Between Keygen and CKD Protocol — (File: `src/confidential_key_derivation/protocol.rs`)

### Summary
The `ckd()` function accepts a `participants` list and a `key_pair: KeygenOutput` as fully independent inputs with no validation that the two are consistent. Because `KeygenOutput` stores neither the original participant set nor the threshold used during keygen, the library cannot enforce the invariant that the CKD signing set must be a valid quorum of the keygen participants. A malicious participant or misconfigured caller can supply a participant list that differs from the keygen set, causing every Lagrange coefficient to be computed against the wrong domain, corrupting the CKD output accepted by the coordinator.

### Finding Description

`ckd()` performs the following checks before starting the protocol: [1](#0-0) 

- `participants.len() >= 2`
- no duplicate participants
- `me` is in the list
- `coordinator` is in the list

It does **not** check:
1. That `participants` is a subset of the participants who ran the keygen that produced `key_pair`.
2. That `participants.len() >= threshold` (the reconstruction threshold from keygen).

The `KeygenOutput` struct used by CKD stores only `private_share` and `public_key`: [2](#0-1) 

There is no field for the original participant set or threshold, so the library has no mechanism to enforce consistency between the two inputs.

Inside `compute_signature_share`, the Lagrange coefficient is computed against whatever `participants` list was supplied: [3](#0-2) 

If `participants` differs from the keygen set, `lambda_i` is wrong. The coordinator blindly sums every participant's `(norm_big_y, norm_big_c)` pair: [4](#0-3) 

Because the Lagrange coefficients no longer satisfy the interpolation identity for the secret `msk`, the aggregated result `sum(lambda_i * x_i) * H(pk, app_id)` diverges from `msk * H(pk, app_id)`, producing a silently corrupted CKD output.

An identical structural gap exists in the OT-based ECDSA presign path, where the code explicitly acknowledges the missing check: [5](#0-4) 

### Impact Explanation
A corrupted CKD output is accepted by the coordinator as a valid protocol result. Honest parties have no way to detect the corruption because there is no consistency proof binding the CKD output to the master public key and the correct participant set. This maps directly to the allowed High impact: **Corruption of CKD outputs so honest parties accept inconsistent or unusable cryptographic outputs.**

### Likelihood Explanation
The entry path requires only that a participant (or the application layer) call `ckd()` with a participant list that does not match the keygen set. This can happen:
- **Accidentally**: after a reshare changes the participant set, the caller forgets to update the CKD participant list.
- **Maliciously**: a single participant deliberately passes a different `participants` slice, causing their Lagrange-weighted share to be wrong; the coordinator sums it in without verification.

No privileged access, leaked keys, or cryptographic breaks are required.

### Recommendation
1. Extend `KeygenOutput` (or a CKD-specific wrapper) to record the participant set and threshold used during keygen.
2. In `ckd()`, validate that the supplied `participants` is a subset of the recorded keygen participants and that `participants.len() >= threshold`.
3. Alternatively, bind the CKD participant set into the protocol transcript so that any mismatch between participants causes an explicit protocol abort rather than a silent wrong result.
4. Apply the same fix to the OT-based ECDSA presign path, removing the acknowledged omitted check.

### Proof of Concept

```
Keygen: participants = {P1, P2, P3}, threshold = 2
  → KeygenOutput { private_share: x_i, public_key: X }
  → msk = λ1·x1 + λ2·x2 + λ3·x3  (Lagrange over {P1,P2,P3})

Attacker (P2) calls ckd() with participants = {P1, P2}:
  → lambda_2 is now computed over {P1, P2}, not {P1, P2, P3}
  → norm_big_c_2 = lambda_2' * (x2 * H(pk,app_id) + y2 * app_pk)
     where lambda_2' ≠ lambda_2

Coordinator sums:
  norm_big_c = norm_big_c_1 + norm_big_c_2 + norm_big_c_3
             ≠ msk * H(pk, app_id) + y * app_pk

CKDOutput is silently wrong; no error is raised.
```

The root cause is that `ckd()` at `src/confidential_key_derivation/protocol.rs:66–117` accepts `participants` and `key_pair` as independent, unchecked inputs, while `KeygenOutput` at `src/lib.rs:51–55` carries no participant-set or threshold metadata to enforce the invariant.
