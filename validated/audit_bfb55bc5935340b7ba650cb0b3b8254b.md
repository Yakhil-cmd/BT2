### Title
Signature Validation Bypass via `skip_stateful_validations` Overly-Broad Mempool Presence Check — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

`skip_stateful_validations` skips the `__validate__` entry-point (signature check) for any invoke transaction with `nonce == 1` targeting an account whose on-chain nonce is `0`, provided `account_tx_in_pool_or_recent_block` returns `true`. That helper returns `true` for **any** transaction type in the pool for that address — not specifically a `deploy_account`. An unprivileged attacker who observes a victim's pending `deploy_account` in the mempool can immediately submit a crafted invoke with `nonce = 1`, an arbitrary payload, and an invalid signature, and the gateway will admit it without ever calling `__validate__`.

### Finding Description

In `crates/apollo_gateway/src/stateful_transaction_validator.rs`, the function `skip_stateful_validations` (lines 429–461) decides whether to skip the blockifier's `__validate__` call:

```rust
async fn skip_stateful_validations(...) -> ... {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await ...;
        }
    }
    Ok(false)
}

When this returns `true`, `run_validate_entry_point` sets `validate: false`:

Inside `StatefulValidator::perform_validations`, when `validate == false` for an invoke, execution returns immediately after `perform_pre_validation_stage` — the `__validate__` call is never reached:

```rust

    return Ok(());

// `__validate__` call.
let (_optional_call_info, actual_cost) = self.validate(&tx, tx_context.clone())?;
``` [3](#0-2) 

`perform_pre_validation_stage` checks nonce, fee bounds, and proof facts — but **not** the transaction signature. [4](#0-3) 

The mempool helper `account_tx_in_pool_or_recent_block` returns `true` if **any** transaction for the address is in the pool or committed state — it does not filter for `deploy_account` specifically:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [5](#0-4) 

The code comment claims this is sufficient because the pool entry "means that either it has a deploy_account transaction or transactions with future nonces that passed validations." However, the check does not enforce this invariant — it only tests address presence.

### Impact Explanation

An attacker who observes a victim's `deploy_account` transaction in the mempool (e.g., via P2P propagation or the RPC) can submit an invoke transaction for the victim's address with `nonce = 1`, arbitrary calldata, and a completely invalid signature. The gateway's stateful validator will:

1. Pass the nonce check (`0 ≤ 1 ≤ max_allowed_nonce_gap`).
2. Pass `validate_by_mempool` (nonce/fee checks only, no signature).
3. Detect the victim's `deploy_account` in the pool via `account_tx_in_pool_or_recent_block`.
4. Set `skip_validate = true`, bypassing `__validate__` entirely.
5. Admit the unsigned/malformed invoke into the mempool.

The admitted transaction will be included in a block. When the blockifier executes it (after the `deploy_account` deploys the account), `__validate__` will fail and the transaction will revert — but it has already consumed a mempool slot and block space. This matches the **High** impact category: *Mempool/gateway admission accepts invalid transactions before sequencing*.

### Likelihood Explanation

The attack requires only that a victim's `deploy_account` be visible in the mempool, which is a normal and observable network event. No privileged access, special keys, or unusual conditions are needed. The attacker simply races to submit the crafted invoke before the `deploy_account` is committed.

### Recommendation

Replace the overly-broad `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a `deploy_account` transaction exists for the address. One approach: add a dedicated `deploy_account_in_pool(address)` query to the mempool that inspects transaction type, and only skip `__validate__` when that returns `true`. Alternatively, require that the transaction type stored in the pool for the address is `DeployAccount` before granting the skip.

### Proof of Concept

1. Victim precomputes address `X` from `(class_hash, salt, constructor_calldata)`.
2. Victim submits a valid `deploy_account` for `X`; it enters the mempool.
3. Attacker observes `X` in the mempool (RPC or P2P).
4. Attacker constructs an invoke:
   - `sender_address = X`, `nonce = 1`, `calldata = [arbitrary]`, `signature = [0x0, 0x0]`
5. Attacker submits to the gateway.
6. Gateway stateful path:
   - `get_nonce_from_state(X)` → `0`
   - `validate_nonce`: `0 ≤ 1 ≤ max_allowed_nonce_gap` → passes
   - `validate_by_mempool`: nonce/fee valid → passes
   - `skip_stateful_validations`: `nonce==1 && account_nonce==0 && account_tx_in_pool_or_recent_block(X)==true` → returns `true`
   - `run_validate_entry_point(skip_validate=true)`: `__validate__` skipped → `Ok(())`
7. Invalid invoke admitted to mempool without signature verification.
8. At execution time: `deploy_account` runs first, then the attacker's invoke runs `__validate__` → fails → reverts. Block space consumed; victim's legitimate nonce-1 invoke may be displaced or delayed. [6](#0-5) [7](#0-6)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
