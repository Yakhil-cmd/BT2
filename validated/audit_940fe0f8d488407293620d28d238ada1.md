### Title
ETH Permanently Locked in `RewardSupplier` When `tick()` Called With No Pending Mint Requests - (File: L1/starkware/solidity/stake/RewardSupplier.sol)

### Summary
`RewardSupplier.tick()` is `payable` and requires ETH to cover L1→L2 messaging fees. However, when there are no pending mint requests (`amountToMint == 0`), the entire fee-spending logic is skipped while `msg.value` is silently accepted. The contract has no `receive()` function, no `fallback()`, and no ETH withdrawal mechanism, so any ETH sent in this case is permanently locked.

### Finding Description
The `tick()` function in `RewardSupplier.sol` is declared `external payable` and internally splits `msg.value` between two fee-bearing calls:

```solidity
uint256 msgFee = msg.value / 2;
bridge().depositWithMessage{value: msgFee}(...);
// ...
msgFee = msg.value - msgFee;
messagingContract().sendMessageToL2{value: msgFee}(...);
```

Both calls are inside the `if (amountToMint > 0)` branch. When `amountToMint == 0` (i.e., `requiredMinting()` returns zero pending messages), the function body is a no-op, yet the function remains `payable` and accepts any ETH sent by the caller. That ETH is credited to the contract's balance with no path to spend or recover it:

- No `receive()` or `fallback()` function exists in `RewardSupplier` or its visible parent `RewardSupplierStorage`.
- No `withdraw()` or ETH-rescue function is present anywhere in the contract.
- The only other ETH-spending site is inside the same `if (amountToMint > 0)` block, which is unreachable when the balance was deposited during a zero-mint call. [1](#0-0) 

### Impact Explanation
Any ETH sent to `tick()` when `amountToMint == 0` is permanently frozen inside the contract. Because `tick()` carries no access control, any public caller can trigger this path. The locked ETH cannot be recovered by any party — there is no admin rescue, no `selfdestruct`, and no withdrawal function. This constitutes permanent freezing of the caller's funds.

### Likelihood Explanation
`tick()` is designed to be called by an external keeper/operator on a regular cadence. The caller is expected to supply ETH for fees. A caller who does not first query `requiredMinting()` — or who races with another caller that already consumed the pending messages — will send ETH into a zero-mint `tick()` call and lose it. This is a realistic operational scenario, especially under concurrent or automated calling patterns.

### Recommendation
Add a guard that reverts when `msg.value > 0` but `amountToMint == 0`, or refund `msg.value` in the zero-mint branch:

```solidity
function tick() external payable {
    (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();
    if (amountToMint == 0) {
        require(msg.value == 0, "NO_MINT_REQUIRED_ETH_REJECTED");
        return;
    }
    // ... existing fee-spending logic ...
}
```

Alternatively, add a `receive()` function paired with an owner-only ETH withdrawal function so that any accidentally locked ETH can be recovered.

### Proof of Concept
1. Observe that `requiredMinting()` returns `(0, 0)` when no L2→L1 mint-request messages are pending.
2. Call `tick{value: 1 ether}()` in this state.
3. The `if (amountToMint > 0)` branch is skipped; `msg.value` (1 ETH) is credited to the contract.
4. Confirm there is no `receive()`, `fallback()`, or `withdraw()` function in `RewardSupplier` or `RewardSupplierStorage`.
5. The 1 ETH is permanently unrecoverable. [2](#0-1) [3](#0-2)

    function tick() external payable {
        // Check if minting is required, and how much.
        (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

        if (amountToMint > 0) {
            // Prepare the L2->L1 mintRequest message for consumption.
            uint256[] memory messagePayload = new uint256[](1);
            messagePayload[0] = TOKENS_PER_MINT_REQUEST;

            // Consume the mintRequest messages.
            for (uint256 i = 0; i < numMsgsToConsume; i++) {
                messagingContract().consumeMessageFromL2(mintRequestSource(), messagePayload);
            }

            // Reuest minting of the requested amount from the mint manager.
            mintManager().mintRequest(token(), amountToMint);

            // Deposit the minted amount onto the bridge to the credit of `mintDestination`.
            uint256 msgFee = msg.value / 2;
            bridge().depositWithMessage{value: msgFee}(
                token(),
                amountToMint,
                mintDestination(),
                new uint256[](0)
            );
            emit ConsumedL2MintRequests(numMsgsToConsume, amountToMint);

            // Send a totalSupply update to L2MintCurve.
            msgFee = msg.value - msgFee;
            messagePayload[0] = IERC20(token()).totalSupply();
            messagingContract().sendMessageToL2{value: msgFee}(
                mintingCurve(),
                UPDATE_TOTAL_SUPPLY_SELECTOR,
                messagePayload
            );
        }
    }
