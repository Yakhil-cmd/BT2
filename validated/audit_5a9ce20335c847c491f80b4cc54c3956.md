### Title
Unbounded Loop in `calculate_rewards` Enables Permanent Freezing of Pool Member Unclaimed Yield - (File: src/pool/pool.cairo)

### Summary

The `calculate_rewards` function in `Pool.cairo` contains an explicitly acknowledged unbounded loop that iterates over a pool member's entire balance-change trace. A pool member who makes balance changes (deposits or partial exit intents) across many epochs without claiming rewards will grow this trace without bound. Once the trace is large enough, every call to `claim_rewards` (and `pool_member_info_v1`) will exceed the Starknet transaction gas limit, permanently freezing the pool member's unclaimed yield.

### Finding Description

`calculate_rewards` in `src/pool/pool.cairo` iterates over every entry in `pool_member_epoch_balance` from `entry_to_claim_from` up to the current trace length:

```
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
    if pool_member_checkpoint.epoch() >= until_epoch { break; }
    let to_sigma = self.find_sigma(pool_member_checkpoint, curr_epoch: until_epoch);
    ...
    entry_to_claim_from += 1;
}
``` [1](#0-0) 

The trace grows via `set_member_balance`, which calls `trace.insert(key: self.get_epoch_plus_k(), ...)`. The underlying `insert` implementation appends a **new** checkpoint whenever the key differs from the last entry's key — i.e., whenever the call happens in a different epoch: [2](#0-1) [3](#0-2) 

`set_member_balance` is called by both `add_to_delegation_pool` and `exit_delegation_pool_intent`: [4](#0-3) [5](#0-4) 

Therefore, every epoch in which a pool member deposits or partially exits adds one new entry to the trace. The `entry_to_claim_from` cursor (stored in `pool_member_info`) advances only after a successful `claim_rewards`. If the pool member never claims between balance changes, the loop must traverse all accumulated entries in a single transaction.

`claim_rewards` and `pool_member_info_v1` both call `calculate_rewards`: [6](#0-5) [7](#0-6) 

Each loop iteration performs multiple storage reads (`pool_member_trace.at(...)` and `find_sigma` which reads `cumulative_rewards_trace`). On Starknet, storage reads are the dominant gas cost. With a sufficiently large trace, the transaction will exceed the block gas limit.

### Impact Explanation

Once the trace is large enough that `claim_rewards` always runs out of gas, the pool member's accumulated unclaimed yield is permanently frozen — it can never be transferred to the reward address. The funds remain locked in the pool contract with no recovery path, because every attempt to claim will revert. This matches the allowed impact: **Permanent freezing of unclaimed yield**.

### Likelihood Explanation

**Low.** The attack requires the pool member to make balance changes across a large number of epochs without ever claiming rewards. This is a realistic long-term scenario for a passive delegator who frequently tops up their delegation but never claims. The protocol itself acknowledges the risk in a comment ("This loop is unbounded but unlikely to exceed gas limits"), confirming the developers are aware but consider it improbable rather than impossible.

### Recommendation

1. **Enforce a claim before balance changes**: Require `claim_rewards` to be called (or auto-claim) before any `add_to_delegation_pool` or `exit_delegation_pool_intent` that would append a new trace entry. This resets `entry_to_claim_from` to the current trace length, bounding the loop to at most a small constant number of entries per claim.

2. **Cap the loop with a maximum iteration count**: Introduce a configurable `MAX_ITERATIONS` constant and process only up to that many entries per call, storing the updated `entry_to_claim_from` so subsequent calls continue from where the previous one stopped. This turns a potentially unbounded single transaction into a series of bounded ones.

3. **Periodic auto-claim on balance change**: Automatically settle pending rewards and advance `entry_to_claim_from` whenever a new balance-trace entry is written, keeping the pending window to at most one entry.

### Proof of Concept

1. A pool member calls `enter_delegation_pool` with a small amount in epoch E.
2. In each subsequent epoch E+1, E+2, …, E+N, the pool member calls `add_to_delegation_pool` with a minimal amount (1 token). Each call appends one new entry to `pool_member_epoch_balance` because `get_epoch_plus_k()` returns a different key each epoch.
3. The pool member never calls `claim_rewards`, so `entry_to_claim_from` remains at 0.
4. After N epochs, the pool member (or their reward address) calls `claim_rewards`. The loop in `calculate_rewards` must iterate over all N entries, each requiring multiple storage reads via `pool_member_trace.at(...)` and `find_sigma(...)`.
5. For sufficiently large N (determined by Starknet's per-transaction gas/step limit), the transaction reverts with an out-of-gas error.
6. All subsequent `claim_rewards` calls also revert. The pool member's accumulated yield is permanently frozen. [1](#0-0) [2](#0-1) [8](#0-7)

        fn claim_rewards(ref self: ContractState, pool_member: ContractAddress) -> Amount {
            // Asserts.
            let mut pool_member_info = self.internal_pool_member_info(:pool_member);
            let caller_address = get_caller_address();
            let reward_address = pool_member_info.reward_address;
            assert!(
                caller_address == pool_member || caller_address == reward_address,
                "{}",
                Error::POOL_CLAIM_REWARDS_FROM_UNAUTHORIZED_ADDRESS,
            );

            let until_checkpoint = self.get_current_checkpoint(:pool_member);

            // Calculate rewards and update entry_to_claim_from.
            let (mut rewards, updated_entry_to_claim_from) = self
                .calculate_rewards(
                    :pool_member,
                    from_checkpoint: pool_member_info.reward_checkpoint,
                    :until_checkpoint,
                    entry_to_claim_from: pool_member_info.entry_to_claim_from,
                );
            rewards += pool_member_info._unclaimed_rewards_from_v0;
            pool_member_info._unclaimed_rewards_from_v0 = Zero::zero();
            pool_member_info.entry_to_claim_from = updated_entry_to_claim_from;
            pool_member_info.reward_checkpoint = until_checkpoint;

            // Write the updated pool member info to storage.
            self.write_pool_member_info(:pool_member, :pool_member_info);

            // Transfer rewards to the pool member.
            let reward_token = IERC20Dispatcher { contract_address: STRK_TOKEN_ADDRESS };
            reward_token.checked_transfer(recipient: reward_address, amount: rewards.into());

            // Emit event.
            self
                .emit(
                    Events::PoolMemberRewardClaimed {
                        pool_member, reward_address, amount: rewards,
                    },
                );

            rewards
        }

    fn insert(
        self: StoragePath<Mutable<PoolMemberBalanceTrace>>, key: Epoch, value: PoolMemberBalance,
    ) -> (PoolMemberBalance, PoolMemberBalance) {
        let checkpoints = self.checkpoints;

        let len = checkpoints.len();
        if len == Zero::zero() {
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
            return (Zero::zero(), value);
        }

        // Update or append new checkpoint.
        let mut last = checkpoints[len - 1].read();
        let prev = last.value;
        if last.key == key {
            last.value = value;
            checkpoints[len - 1].write(last);
        } else {
            // Checkpoint keys must be non-decreasing.
            assert!(last.key < key, "{}", TraceErrors::UNORDERED_INSERTION);
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
        }
        (prev, value)
    }
