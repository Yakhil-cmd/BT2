### Title
Unprivileged Key Overwrite in `MsgRegisterEncryptionKey` Permanently Freezes Validator Blocklist via `lastBlockList` Cache Poisoning — (`x/e2ee/keeper/keeper.go`, `app/proposal.go`)

---

### Summary

`MsgRegisterEncryptionKey` does not verify that the transaction signer owns the `Address` field. Any account can overwrite any validator's registered age X25519 public key. Combined with a pre-decryption update of `lastBlockList` in `SetBlockList`, a single transaction permanently prevents the victim validator's blocklist from ever being updated to the new blob, silently leaving it stale or empty.

---

### Finding Description

**Root Cause 1 — No ownership check in `MsgRegisterEncryptionKey`**

`ValidateBasic` only checks bech32 format and age key format: [1](#0-0) 

The keeper stores the key for the supplied `Address` without checking whether the signer equals `Address`: [2](#0-1) 

There is no ante handler, middleware, or keeper guard that enforces `signer == req.Address`. Any funded account can submit `MsgRegisterEncryptionKey{Address: victim_validator_addr, Key: attacker_pubkey}` and overwrite the victim's registered key.

**Root Cause 2 — `lastBlockList` is updated before successful decryption**

In `SetBlockList`, the cache field `lastBlockList` is written at lines 218–219 **before** `age.Decrypt` is attempted at line 226: [3](#0-2) 

If `age.Decrypt` returns `ErrIncorrectIdentity` (because the blob was encrypted to the attacker's key, not the validator's real identity), the function returns the error — but `lastBlockList` already holds the new blob. On every subsequent call with the same blob, the early-exit guard at line 215 fires and returns `nil` without touching `h.blocklist`. The blocklist map is permanently frozen at its previous state.

---

### Impact Explanation

The blocklist is the mechanism by which the Cronos admin can prevent specific addresses from having transactions included in proposed blocks. After the attack:

- The victim validator's `h.blocklist` remains empty or stale.
- `ValidateTransaction` fast-paths through `len(h.blocklist) == 0` and accepts all transactions.
- Addresses that the admin intended to block can have their transactions included in every block proposed by the victim validator.

This is a direct bypass of the block-list authorization check, matching the High impact category: *"Bypass of Cronos admin, governance authority, permission, token-mapping, bridge, block-list, or module-account authorization checks."* [4](#0-3) 

---

### Likelihood Explanation

- The attacker needs only a funded account and the victim validator's account address (publicly derivable from their consensus key or on-chain validator record).
- No privileged access, leaked keys, or social engineering is required.
- The attack is a single on-chain transaction costing only gas fees.
- The damage is permanent for the current blob: the admin must re-encrypt and resubmit `MsgStoreBlockList` with a fresh blob to recover, but the attacker can immediately repeat the attack.

---

### Recommendation

1. **Enforce signer == address in `RegisterEncryptionKey`**: In the keeper's `RegisterEncryptionKey`, compare `sdk.UnwrapSDKContext(ctx)` signer (extracted from the message server context) against `req.Address` and reject mismatches.

2. **Update `lastBlockList` only after successful decryption**: Move the `lastBlockList` assignment to after `h.blocklist = m` at line 258, so a failed decryption does not poison the cache and the function will retry on the next call with the same blob. [5](#0-4) 

---

### Proof of Concept

```
1. Attacker generates age X25519 key pair: (attacker_priv, attacker_pub)
2. Attacker submits:
     MsgRegisterEncryptionKey{
       Address: victim_validator_account_addr,
       Key:     attacker_pub,
     }
   signed by any funded account.
3. On-chain state: victim_validator_account_addr → attacker_pub

4. CronosAdmin calls MsgStoreBlockList with blob encrypted to all
   validators' registered keys (now including attacker_pub for victim).

5. Victim validator node calls SetBlockList(blob):
   - lastBlockList != blob → proceeds
   - lastBlockList = blob  ← cache poisoned here
   - age.Decrypt(blob, victim_identity) → ErrIncorrectIdentity
   - returns error; h.blocklist unchanged (empty or stale)

6. Any subsequent SetBlockList(blob) call:
   - bytes.Equal(lastBlockList, blob) == true → returns nil immediately
   - h.blocklist is never updated

7. Victim validator proposes blocks with no blocklist enforcement.
   Blocked addresses' transactions are included in every block it proposes.

8. Attacker can repeat step 2 after any admin re-encryption to maintain
   the freeze indefinitely.
```
