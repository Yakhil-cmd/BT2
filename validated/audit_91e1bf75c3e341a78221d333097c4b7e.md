### Title
Threshold Narrowed to `u16` in EdDSA `construct_key_package` Causes Permanent Denial of Signing - (File: src/frost/eddsa/sign.rs)

### Summary

The `construct_key_package` function in the EdDSA signing path converts the `ReconstructionLowerBound` threshold (backed by `usize`) to `u16` when building the FROST `KeyPackage`. Any threshold value exceeding `u16::MAX` (65 535) causes every participant and the coordinator to fail at signing, permanently denying the signing operation for all honest parties.

### Finding Description

`ReconstructionLowerBound` is defined as a newtype over `usize`: [1](#0-0) 

The public API accepts any `usize` value as threshold. The invariant check in `assert_key_invariants` only requires `2 ≤ threshold ≤ participants.len()`: [2](#0-1) 

No upper bound of 65 535 is enforced there. However, inside the EdDSA signing path, `construct_key_package` performs a narrowing conversion: [3](#0-2) 

Specifically at line 365:
```rust
u16::try_from(threshold.value()).map_err(|_| {
    ProtocolError::Other("threshold cannot be converted to u16".to_string())
})?,
```

`construct_key_package` is called by all four signing paths — `do_sign_coordinator_v1`, `do_sign_coordinator_v2`, `do_sign_participant_v1`, and `do_sign_participant_v2`: [4](#0-3) [5](#0-4) [6](#0-5) [7](#0-6) 

If `threshold > 65 535`, every call to `construct_key_package` returns `Err(ProtocolError::Other(...))`, aborting the signing round for every participant simultaneously.

### Impact Explanation

Matches **High: Permanent denial of signing for honest parties under valid protocol inputs and documented trust assumptions.**

A caller who supplies a threshold in the range `(65 535, usize::MAX]` — which passes all documented invariant checks — will find that the EdDSA signing protocol is permanently broken: no participant can produce a signature share, and the coordinator cannot aggregate. The DKG output (keys) remains intact but is rendered permanently unusable for signing.

### Likelihood Explanation

Low. The threshold is bounded by `participants.len()`, so triggering the bug requires more than 65 535 participants. No real deployment is expected to reach that scale. The bug is nonetheless a latent correctness defect: the type accepted by the public API (`usize`) is strictly wider than the type consumed internally (`u16`), with no guard at the API boundary, mirroring the exact pattern flagged in the reference report.

### Recommendation

Add an explicit upper-bound check at the API boundary (e.g., in `assert_sign_inputs` or `assert_key_invariants`) that rejects any threshold exceeding `u16::MAX`, and document this constraint. Alternatively, if the FROST library's `KeyPackage::new` signature is under the project's control, widen the threshold parameter to `usize` or `u32` to match the rest of the codebase.

```rust
// In assert_sign_inputs or assert_key_invariants:
if threshold.value() > u16::MAX as usize {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
```

### Proof of Concept

1. Create a participant set of size 65 536 (or mock `ReconstructionLowerBound(65_536)`).
2. Run DKG successfully — `assert_key_invariants` passes because `threshold ≤ participants.len()`.
3. Call `sign_v1` or `sign_v2` with the same threshold.
4. Every participant hits `construct_key_package` → `u16::try_from(65_536)` → `Err` → `ProtocolError::Other("threshold cannot be converted to u16")`.
5. No signature share is produced; the coordinator cannot aggregate; signing is permanently denied. [8](#0-7)

```rust
fn construct_key_package(
    threshold: ReconstructionLowerBound,
    me: Participant,
    signing_share: SigningShare,
    verifying_key: &VerifyingKey,
) -> Result<KeyPackage, ProtocolError> {
    let identifier = me.to_identifier()?;
    let verifying_share = signing_share.into();

    Ok(KeyPackage::new(
        identifier,
        signing_share,
        verifying_share,
        *verifying_key,
        u16::try_from(threshold.value()).map_err(|_| {
            ProtocolError::Other("threshold cannot be converted to u16".to_string())
        })?,
    ))
}
```
