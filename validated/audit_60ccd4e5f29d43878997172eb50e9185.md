### Title
BLAKE2F Precompile (EIP-152, `0x09`) Absent in Production Builds Causes Silent Wrong-Result State Transitions — (`system_hooks/src/lib.rs`, `evm_interpreter/src/precompile_addresses.rs`, `basic_bootloader/src/bootloader/block_flow/zk/post_init_op.rs`)

---

### Summary

The BLAKE2F precompile at EVM address `0x09` is never registered in production ZKsync OS builds. Any EVM contract that calls `0x09` receives a **successful call with empty return data** instead of the correct BLAKE2F hash. Because the call does not revert, the caller cannot detect the failure, and any downstream state that depends on the hash value is silently corrupted. This is a direct EVM-semantic-mismatch / state-transition bug reachable by any unprivileged EVM transaction.

---

### Finding Description

**Step 1 — Production feature sets omit `eip-152`.**

`forward_system/Cargo.toml` line 49 defines the production feature set:

```
production = ["basic_bootloader/eip-7702", "system_hooks/p256_precompile"]
```

`proof_running_system/Cargo.toml` line 46 mirrors this:

```
production = ["basic_bootloader/eip-7702", "system_hooks/p256_precompile"]
```

Neither includes `eip-152` or `mock-unsupported-precompiles`. [1](#0-0) [2](#0-1) 

**Step 2 — `BLAKE2F_HOOK_ADDRESS_LOW` is conditionally compiled out.**

In `evm_interpreter/src/precompile_addresses.rs`, the constant and its entry in `PRECOMPILE_ADDRESSES_LOWS` are both gated:

```rust
#[cfg(any(feature = "eip-152", feature = "mock-unsupported-precompiles"))]
pub const BLAKE2F_HOOK_ADDRESS_LOW: u16 = 0x0009;
...
#[cfg(any(feature = "eip-152", feature = "mock-unsupported-precompiles"))]
BLAKE2F_HOOK_ADDRESS_LOW,
```

Without either feature, `0x0009` is absent from `PRECOMPILE_ADDRESSES_LOWS`. [3](#0-2) [4](#0-3) 

**Step 3 — The production ZK post-init path never registers BLAKE2F.**

The production block-execution path uses `ZKHeaderPostInitOp`. Its `post_init_op` calls `system_hooks::add_precompiles` (which is also feature-gated for BLAKE2F) but never calls `initialize_eip_152`:

```rust
fn post_init_op<Config: BasicBootloaderExecutionConfig>(
    _system: &mut System<S>,
    system_functions: &mut HooksStorage<S, <S as SystemTypes>::Allocator>,
) -> Result<(), InternalError> {
    system_hooks::add_precompiles(system_functions)?;
    // initialize_eip_152 is NOT called here
    ...
}
``` [5](#0-4) 

The Ethereum-flow counterpart (`EthereumPostInitOp`) does call `initialize_eip_152`, but that path is only activated by the `eth_stf` feature, which is not production. [6](#0-5) 

**Step 4 — Unregistered hook address is silently treated as an empty account.**

`HooksStorage::try_intercept` returns `Ok(None)` when no hook is registered for an address:

```rust
let Some(hook) = self.call_hooks.get(&address_low) else {
    return Ok((None, return_memory));
};
``` [7](#0-6) 

The bootloader then treats the missing hook as an empty account and returns a **successful** call with empty return data:

```rust
} else {
    // it's an empty account for all the purposes
    ...
    Ok((
        resources_returned,
        CallResult::Successful {
            return_values: ReturnValues::empty(),
        },
    ))
}
``` [8](#0-7) 

**Step 5 — The test suite confirms this is the production behavior.**

The test comment at line 1343 explicitly states:

> "blake2f (0x09) is not enabled in test/production builds, so it must be cold" [9](#0-8) 

---

### Impact Explanation

On Ethereum mainnet, calling `0x09` with valid 213-byte BLAKE2F input returns a 64-byte hash. On production ZKsync OS, the same call returns `success = true` with **zero bytes of return data**. Any contract that:

- uses BLAKE2F for hash-based access control, signature verification, or commitment schemes, or
- checks `returndatasize` / reads the 64-byte output after the call

will silently receive wrong data and proceed with a corrupted state. Because the call does not revert, the caller has no error signal. This is a state-transition bug: the ZKsync OS state root diverges from what Ethereum-equivalent execution would produce for the same transaction.

---

### Likelihood Explanation

The entry path is fully unprivileged: any EVM contract can issue a `CALL` to address `0x09`. No special role, key, or oracle manipulation is required. Any protocol deployed on ZKsync OS that relies on BLAKE2F (e.g., Zcash-bridge contracts, BLAKE2-based Merkle trees, or any contract ported from a chain where BLAKE2F is available) is affected on every invocation.

---

### Recommendation

Register the BLAKE2F precompile unconditionally in the production ZK post-init path, or add `eip-152` to the `production` feature set in both `forward_system/Cargo.toml` and `proof_running_system/Cargo.toml`. The full implementation (`Blake2FPrecompile`) already exists in `basic_bootloader/src/bootloader/block_flow/ethereum/hooks/eip_152/` and is correct; it only needs to be wired into `ZKHeaderPostInitOp::post_init_op` the same way `EthereumPostInitOp` does it.

---

### Proof of Concept

1. Deploy the following EVM contract on production ZKsync OS (no `eip-152` feature):

```solidity
contract Blake2fCheck {
    function check() external returns (bool ok, uint256 rdsz) {
        bytes memory input = new bytes(213); // valid-length BLAKE2F input
        (ok,) = address(0x09).call(input);
        rdsz = returndatasize();
        // On Ethereum: ok=true, rdsz=64
        // On ZKsync OS production: ok=true, rdsz=0  ← silent wrong result
    }
}
```

2. Call `check()`. The call to `0x09` returns `success = true` and `returndatasize() == 0` instead of 64.
3. Any downstream logic that reads the 64-byte hash (e.g., `returndatacopy`) reads zeroes, silently corrupting state.

The root cause is confirmed at:
- `system_hooks/src/lib.rs` lines 169–184 (BLAKE2F only registered under `mock-unsupported-precompiles`) [10](#0-9) 
- `basic_bootloader/src/bootloader/block_flow/zk/post_init_op.rs` lines 8–27 (production path never calls `initialize_eip_152`) [5](#0-4) 
- `basic_bootloader/src/bootloader/runner.rs` lines 611–625 (missing hook → silent empty-account success) [8](#0-7)
