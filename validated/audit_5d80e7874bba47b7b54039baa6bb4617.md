### Title
Silent Swallowing of Process Callback Output Error Causes Scheduled Transaction Loss and Execution/Verification Divergence - (`engine/execution/computation/computer/computer.go`)

---

### Summary

In `blockComputer.executeProcessCallback()`, when the `ProcessCallbacksTransaction` (which drives the `FlowTransactionScheduler.process()` system call) returns a Cadence-level output error (`txn.Output().Err != nil`), the execution node logs a "critical_error" but **returns `nil` instead of an error**, allowing block execution to continue without any scheduled callback transactions. This silently drops all scheduled transactions for that block. Simultaneously, the chunk verifier (`ChunkVerifier.createSystemChunk`) treats the same condition as a **fatal error**, creating an irreconcilable divergence between what the execution node produces and what verification nodes can verify.

---

### Finding Description

In `engine/execution/computation/computer/computer.go`, `executeProcessCallback` executes the `processCallbackTxn` (the `ProcessCallbacksTransaction`) to discover which scheduled transactions should run in the current block. After execution, the output error is checked:

```go
if txn.Output().Err != nil {
    // if the process transaction fails we log the critical error but don't return an error
    // so that block execution continues and only the scheduled transactions halt
    callbackLogger.Error().
        Err(txn.Output().Err).
        Bool("critical_error", true).
        Uint64("height", request.ctx.BlockHeader.Height).
        Msg("system process transaction output error")

    return nil, txnIndex, nil   // ← error swallowed; no callbacks returned
}
``` [1](#0-0) 

The function returns `nil` for both the error and the callback list. The caller `executeSystemTransactions` then builds the system chunk with **zero callback transactions** and proceeds normally. [2](#0-1) 

The chunk verifier (`module/chunks/chunkVerifier.go`) handles the identical condition in `createSystemChunk` with the **opposite behavior** — it treats a non-nil `processOutput.Err` as a fatal, propagated error:

```go
if processOutput.Err != nil {
    return nil, nil, fmt.Errorf("process callback transaction failed: %w", processOutput.Err)
}
``` [3](#0-2) 

This divergence is confirmed by the block computer's own test suite, which explicitly asserts that when `processOutputError != nil`, `ExecuteBlock` returns **no error** and execution continues:

```go
if processOutputError != nil {
    require.NoError(t, err)
    // verify system chunk transaction succeeded
    systemChunkResult := result.AllTransactionResults()[1]
    require.Empty(t, systemChunkResult.ErrorMessage, ...)
    return
}
``` [4](#0-3) 

---

### Impact Explanation

**Scenario 1 — Loss of scheduled transaction execution (direct fund loss):**
Users who paid fees to schedule transactions via `FlowTransactionScheduler` have their callbacks silently dropped when the `process()` call fails. The fees paid for scheduling are consumed but the scheduled execution never occurs. This is the direct analog to the `harvest()` cooldown/revert scenario in the original report: a critical sub-call fails silently during a state transition, and the transition proceeds without the expected side-effect (reward/callback execution).

**Scenario 2 — Execution/verification divergence (chain halt):**
The execution node produces an execution result where the process callback transaction has `Failed: true` in its `LightTransactionResult` and zero callback transactions in the system chunk. The chunk verifier, when re-executing the same system chunk, calls `createSystemChunk`, which returns a fatal error on `processOutput.Err != nil`. Verification nodes therefore cannot produce a valid `ResultApproval` for this execution result. The result can never accumulate sufficient approvals to be sealed, halting the sealing pipeline. [5](#0-4) [6](#0-5) 

---

### Likelihood Explanation

The `processCallbackTxn` is a system transaction that calls `FlowTransactionScheduler.process()`. This Cadence call can produce a non-nil `output.Err` (a Cadence-level transaction error, distinct from a Go-level execution error) under several conditions:

- A bug or panic in the `FlowTransactionScheduler` contract itself.
- Computation limit exhaustion: if many transactions are scheduled, the `process()` call may exceed its gas budget, producing a `ComputationLimitExceededError` in `output.Err`. An unprivileged user can schedule transactions (paying the scheduling fee), and if enough accumulate, the `process()` call overflows its limit — an attacker-reachable path.
- Any Cadence runtime error within the scheduler contract triggered by malformed scheduled transaction metadata.

The condition is reachable without privileged access. The silent swallowing means the failure is invisible to on-chain observers and produces no revert or alert beyond a log line.

---

### Recommendation

The `executeProcessCallback` function must not silently swallow `txn.Output().Err`. The behavior should be made consistent with the chunk verifier. Two options:

1. **Propagate the error** (consistent with `chunkVerifier.createSystemChunk`): return the output error as a fatal execution error, halting block execution. This prevents the divergence and forces the issue to be addressed at the contract level.

2. **Treat as a no-op with explicit empty result** (if liveness is prioritized): if the design intent is to continue execution when the scheduler fails, the chunk verifier must be updated to match — it must also tolerate a failed process callback and produce a consistent (empty-callback) system chunk rather than returning a fatal error.

The current state — where the execution node silently continues but the chunk verifier fatally rejects — is the worst of both worlds: funds are lost **and** the chain halts. [1](#0-0) [7](#0-6) 

---

### Proof of Concept

1. A user schedules enough transactions via `FlowTransactionScheduler` that the `process()` call in the next block exceeds its computation limit, causing `txn.Output().Err != nil` (a `ComputationLimitExceededError`).
2. `executeProcessCallback` logs `"system process transaction output error"` with `critical_error: true` and returns `nil, txnIndex, nil`.
3. `executeSystemTransactions` builds the system chunk with only the system transaction (no callbacks). All scheduled transactions are dropped.
4. The execution result is committed with the process callback transaction marked `Failed: true` and zero callback transactions.
5. A verification node receives the execution result and calls `ChunkVerifier.Verify` → `verifyTransactionsInContext` → `createSystemChunk`.
6. `createSystemChunk` re-executes the process callback, gets `processOutput.Err != nil`, and returns `fmt.Errorf("process callback transaction failed: %w", processOutput.Err)`.
7. Verification fails; no `ResultApproval` is produced; the execution result is never sealed; sealing halts. [8](#0-7) [6](#0-5) [9](#0-8)

// executeProcessCallback submits a transaction that invokes the `process` method
// on the transaction scheduler contract.
//
// The `process` method scans for scheduled transactions and emits an event for each that should
// be executed. These emitted events are used to construct scheduled transaction execution transactions,
// which are then added to the system transaction collection.
//
// If the `process` transaction fails, a fatal error is returned.
//
// Note: this transaction is executed serially and not concurrently with the system transaction.
// This is because it's unclear whether the scheduled transaction executions triggered by `process`
// will result in additional system transactions.
// In theory, if no additional transactions are emitted, concurrent execution could be optimized.
// However, due to the added complexity, this optimization was deferred.
func (e *blockComputer) executeProcessCallback(
	systemCtx fvm.Context,
	systemCollectionInfo collectionInfo,
	database *transactionCoordinator,
	blockSpan otelTrace.Span,
	txnIndex uint32,
	systemLogger zerolog.Logger,
) ([]*flow.TransactionBody, uint32, error) {
	callbackLogger := systemLogger.With().
		Bool("system_transaction", true).
		Logger()

	request := newTransactionRequest(
		systemCollectionInfo,
		systemCtx,
		callbackLogger,
		txnIndex,
		e.processCallbackTxn,
		ComputerTransactionTypeSystem,
		false)

	txnIndex++

	txn, err := e.executeTransactionInternal(blockSpan, database, request, 1)
	if err != nil {
		snapshotTime := logical.Time(0)
		if txn != nil {
			snapshotTime = txn.SnapshotTime()
		}

		return nil, 0, fmt.Errorf(
			"failed to execute system process transaction %v (%d@%d) for block %s at height %v: %w",
			request.txnIdStr,
			request.txnIndex,
			snapshotTime,
			request.blockIdStr,
			request.ctx.BlockHeader.Height,
			err)
	}

	if txn.Output().Err != nil {
		// if the process transaction fails we log the critical error but don't return an error
		// so that block execution continues and only the scheduled transactions halt
		callbackLogger.Error().
			Err(txn.Output().Err).
			Bool("critical_error", true).
			Uint64("height", request.ctx.BlockHeader.Height).
			Msg("system process transaction output error")

		return nil, txnIndex, nil
	}

	callbackTxs, err := blueprints.ExecuteCallbacksTransactions(e.vmCtx.Chain, txn.Output().Events)
	if err != nil {
		return nil, 0, err
	}

	if len(callbackTxs) > 0 {
		// calculate total computation limits for execute scheduled transaction transactions
		var totalExecuteComputationLimits uint64
		for _, tx := range callbackTxs {
			totalExecuteComputationLimits += tx.GasLimit
		}

		// report metrics for scheduled transactions executed
		e.metrics.ExecutionScheduledTransactionsExecuted(
			len(callbackTxs),
			txn.Output().ComputationUsed,
			totalExecuteComputationLimits,
		)
	}

	return callbackTxs, txnIndex, nil
}

func (fcv *ChunkVerifier) createSystemChunk(
	callbackCtx fvm.Context,
	snapshotTree *snapshot.SnapshotTree,
	chunkState *fvmState.ExecutionState,
	transactionOffset uint32,
	events *flow.EventsList,
	serviceEvents *flow.ServiceEventList,
	unknownRegTouch map[flow.RegisterID]struct{},
	execResID flow.Identifier,
	chIndex uint64,
) ([]*fvm.TransactionProcedure, *flow.LightTransactionResult, error) {
	txIndex := transactionOffset

	// If scheduled transactions are disabled we only have the system transaction in the chunk
	if !fcv.vmCtx.ScheduledTransactionsEnabled {
		txBody, err := blueprints.SystemChunkTransaction(fcv.vmCtx.Chain)
		if err != nil {
			return nil, nil, fmt.Errorf("could not get system chunk transaction: %w", err)
		}

		// Need to return a placeholder result that will be filled by the caller
		// when the transaction is actually executed
		return []*fvm.TransactionProcedure{
			fvm.Transaction(txBody, txIndex),
		}, nil, nil
	}

	processBody, err := blueprints.ProcessCallbacksTransaction(fcv.vmCtx.Chain)
	if err != nil {
		return nil, nil, fmt.Errorf("could not get process callback transaction: %w", err)
	}
	processTx := fvm.Transaction(processBody, txIndex)

	// Execute process callback transaction
	executionSnapshot, processOutput, err := fcv.vm.Run(callbackCtx, processTx, *snapshotTree)
	if err != nil {
		return nil, nil, fmt.Errorf("failed to execute process callback transaction: %w", err)
	}
	if processOutput.Err != nil {
		return nil, nil, fmt.Errorf("process callback transaction failed: %w", processOutput.Err)
	}

	processResult := &flow.LightTransactionResult{
		TransactionID:   processTx.ID,
		ComputationUsed: processOutput.ComputationUsed,
		Failed:          false,
	}

	if len(unknownRegTouch) > 0 {
		var missingRegs []string
		for id := range unknownRegTouch {
			missingRegs = append(missingRegs, id.String())
		}
		return nil, nil, chmodels.NewCFMissingRegisterTouch(missingRegs, chIndex, execResID, processTx.ID)
	}

	// Generate callback execution transactions from the events
	callbackTxs, err := blueprints.ExecuteCallbacksTransactions(fcv.vmCtx.Chain, processOutput.Events)
	if err != nil {
		return nil, nil, fmt.Errorf("failed to generate callback execution transactions: %w", err)
	}

	// Build the final transaction list: [processCallback, ...callbackExecutions, systemTx]
	transactions := make([]*fvm.TransactionProcedure, 0, len(callbackTxs)+2)
	transactions = append(transactions, processTx)

	// Add callback execution transactions
	for _, c := range callbackTxs {
		txIndex++
		transactions = append(transactions, fvm.Transaction(c, txIndex))
	}

	// Add the system transaction as last transaction in collection
	systemTx, err := blueprints.SystemChunkTransaction(fcv.vmCtx.Chain)
	if err != nil {
		return nil, nil, fmt.Errorf("could not get system chunk transaction: %w", err)
	}

	txIndex++
	transactions = append(transactions, fvm.Transaction(systemTx, txIndex))

	// Add events with pointers to reflect the change to the caller
	*events = append(*events, processOutput.Events...)
	*serviceEvents = append(*serviceEvents, processOutput.ConvertedServiceEvents...)

	*snapshotTree = snapshotTree.Append(executionSnapshot)
	err = chunkState.Merge(executionSnapshot)
	if err != nil {
		return nil, nil, fmt.Errorf("failed to merge process callback: %w", err)
	}

	return transactions, processResult, nil
```
