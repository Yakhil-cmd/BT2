### Title
Pool Contract Allows Setting Reward Address to STRK Token Contract in Non-STRK Pools, Permanently Freezing Unclaimed Yield - (File: src/pool/pool.cairo)

---

### Summary

`pool.cairo`'s `enter_delegation_pool` and `change_reward_address` only validate that `reward_address` differs from the pool's own staking token. For non-STRK pools (e.g., BTC pools), this check does **not** cover `STRK_TOKEN_ADDRESS` — the hardcoded reward token. A pool member can set their `reward_address` to `STRK_TOKEN_ADDRESS`, causing all future STRK reward claims to be transferred into the STRK token contract itself, permanently locking them.

---

### Finding Description

Pool rewards are always paid in STRK, hardcoded in `claim_rewards`:

```cairo
// pool.cairo line 365-366
let reward_token = IERC20Dispatcher { contract_address: STRK_TOKEN_ADDRESS };
reward_token.checked_transfer(recipient: reward_address, amount: rewards.into());
``` [1](#0-0) 

The guard in `enter_delegation_pool` only rejects `reward_address` equal to the pool's own staking token:

```cairo
// pool.cairo line 193-195
let token_address = token_dispatcher.contract_address;
assert!(token_address != pool_member, "{}", Error::POOL_MEMBER_IS_TOKEN);
assert!(token_address != reward_address, "{}", GenericError::REWARD_ADDRESS_IS_TOKEN);
``` [2](#0-1) 

The same incomplete guard appears in `change_reward_address`:

```cairo
// pool.cairo line 506-510
assert!(
    self.token_dispatcher.contract_address.read() != reward_address,
    "{}",
    GenericError::REWARD_ADDRESS_IS_TOKEN,
);
``` [3](#0-2) 

For a **BTC pool**, `token_dispatcher.contract_address` is the BTC token address — not `STRK_TOKEN_ADDRESS`. Therefore, passing `reward_address = STRK_TOKEN_ADDRESS` passes both guards.

By contrast, the staking contract's `change_reward_address` correctly calls `does_token_exist`, which covers all managed tokens (STRK + all BTC tokens):

```cairo
// staking.cairo line 520-524
assert!(
    !self.does_token_exist(token_address: reward_address),
    "{}",
    GenericError::REWARD_ADDRESS_IS_TOKEN,
);
``` [4](#0-3) 

The same comprehensive check is applied in `stake`: [5](#0-4) 

The pool contract's validation is inconsistent with the staking contract's validation — the exact analog of the external report's root cause.

---

### Impact Explanation

When a pool member in a BTC pool has `reward_address = STRK_TOKEN_ADDRESS`, every call to `claim_rewards` executes:

```cairo
reward_token.checked_transfer(recipient: STRK_TOKEN_ADDRESS, amount: rewards.into());
```

STRK tokens are transferred into the STRK ERC-20 contract itself. Standard ERC-20 contracts have no withdrawal mechanism for tokens sent to their own address; those tokens are permanently inaccessible. This constitutes **permanent freezing of unclaimed yield** — a High-severity impact in the allowed scope.

---

### Likelihood Explanation

- BTC pools are a live, supported feature of the protocol (`add_token`, `enable_token`, `set_open_for_delegation` with a BTC token address).
- `enter_delegation_pool` and `change_reward_address` are unprivileged, callable by any pool member.
- A pool member only needs to supply `STRK_TOKEN_ADDRESS` as `reward_address` — either by mistake (confusing the reward token with a valid recipient) or deliberately (self-griefing or testing).
- No privileged role, leaked key, or external dependency is required.

Likelihood: **Medium** (requires a BTC pool to be active and a pool member to supply the STRK token address as their reward address).

---

### Recommendation

In `pool.cairo`, replace the single-token check in both `enter_delegation_pool` and `change_reward_address` with a check that also rejects `STRK_TOKEN_ADDRESS`:

```cairo
// In enter_delegation_pool and change_reward_address:
assert!(token_address != reward_address, "{}", GenericError::REWARD_ADDRESS_IS_TOKEN);
assert!(STRK_TOKEN_ADDRESS != reward_address, "{}", GenericError::REWARD_ADDRESS_IS_TOKEN);
```

Alternatively, mirror the staking contract's approach by querying the staking contract for all managed tokens and rejecting any of them as a reward address. This ensures the pool contract's validation stays consistent with the staking contract's `does_token_exist` check regardless of future token additions.

---

### Proof of Concept

1. Token admin calls `staking.add_token(btc_token)` and `staking.enable_token(btc_token)`.
2. A staker calls `staking.set_open_for_delegation(btc_token)` → a BTC `pool_contract` is deployed.
3. Attacker (pool member) calls `pool_contract.enter_delegation_pool(reward_address: STRK_TOKEN_ADDRESS, amount: X)`.
   - Check at line 195: `btc_token != STRK_TOKEN_ADDRESS` → **passes**.
   - Pool member record is created with `reward_address = STRK_TOKEN_ADDRESS`.
4. Rewards accrue to the pool member over time via `update_rewards_from_staking_contract`.
5. Anyone calls `pool_contract.claim_rewards(pool_member: attacker)`.
   - Line 366: `reward_token.checked_transfer(recipient: STRK_TOKEN_ADDRESS, amount: rewards)`.
   - STRK rewards are sent to the STRK token contract and permanently locked.

The same flow applies if the pool member first joins normally and later calls `change_reward_address(STRK_TOKEN_ADDRESS)` (line 506–510 check passes for BTC pools).
