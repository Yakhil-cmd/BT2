### Title
Gateway Stateful Validator Uses Stale Previous-Block L2 Gas Price for Resource Bounds Admission, Causing False Rejections and False Admissions — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The gateway's stateful resource-bounds check compares a transaction's `max_price_per_unit` against the **previous block's** L2 gas price. Because the L2 gas price is recalculated every block via EIP-1559, the admission check is always one block behind the price that will actually govern execution. This is the direct sequencer analog of M-44: a stale "exchange rate" is used at admission time, while a different (updated) rate is applied at execution time.

---

### Finding Description

In `validate_resource_bounds`, the gateway fetches `previous_block_l2_gas_price` from `gateway_fixed_block_state_reader.get_block_info()` and checks that the transaction's `l2_gas.max_price_per_unit` is at least `min_gas_price_percentage%` of that stale price:

```rust
// TODO(Arni): getnext_l2_gas_price from the block header.
let previous_block_l2_gas_price = self
    .gateway_fixed_block_state_reader
    .get_block_info()
    .await?
    .gas_prices
    .strk_gas_prices
    .l2_gas_price;
self.validate_tx_l2_gas_price_within_threshold(
    executable_tx.resource_bounds(),
    previous_block_l2_gas_price,
)?;
``` [1](#0-0) 

The TODO comment is the team's own acknowledgement that the **next** block's L2 gas price should be used here instead.

The actual execution price is computed fresh by the batcher via EIP-1559 (`calculate_next_base_gas_price`) and pushed to the mempool before block building begins:

```rust
mempool_client
    .update_gas_price(
        propose_block_input.block_info.gas_prices.strk_gas_prices.l2_gas_price.get(),
    )
    .await
``` [2](#0-1) 

The mempool's `update_gas_price` call moves any transaction whose `max_l2_gas_price < current_block_l2_gas_price` into the **pending queue**, where it is invisible to the batcher. The blockifier's `check_fee_bounds` independently enforces the same condition at execution time:

```rust
if resource_bounds.max_price_per_unit < actual_gas_price.get() {
    insufficiencies_resource.push(
        ResourceBoundsError::MaxGasPriceTooLow { ... }
    );
}
``` [3](#0-2) 

This creates two distinct failure modes driven by the EIP-1559 price delta between consecutive blocks:

**Mode 1 — False admission (price rising, blocks full):**
- `prev_block_price = 100`, `min_gas_price_percentage = 90%` → gateway threshold = 90
- EIP-1559 raises `next_block_price` to 112
- Transaction with `max_l2_gas_price = 95` passes the gateway check (95 ≥ 90)
- Batcher calls `update_gas_price(112)` → transaction demoted to pending queue, never executed
- User receives a gateway success response for a transaction that will never be sequenced at the current price

**Mode 2 — False rejection (price falling, blocks empty):**
- `prev_block_price = 100`, `min_gas_price_percentage = 90%` → gateway threshold = 90
- EIP-1559 lowers `next_block_price` to 88
- Transaction with `max_l2_gas_price = 89` is **rejected** by the gateway (89 < 90)
- The transaction would have passed `check_fee_bounds` at execution (89 ≥ 88)
- A valid transaction is incorrectly refused admission

The EIP-1559 formula in `calculate_next_base_gas_price` guarantees the price changes every block whenever `gas_used ≠ gas_target`:

```rust
let adjusted_price_u256 =
    if gas_used > gas_target { price_u256 + price_change } else { price_u256 - price_change };
``` [4](#0-3) 

The `next_l2_gas_price` is already stored in the block header and propagated through the sync path, so the correct value is available: [5](#0-4) 

Additionally, `validate_tx_l2_gas_price_within_threshold` silently skips the check for `ValidResourceBounds::L1Gas` (legacy V1/V2 transactions), meaning those transactions bypass the price floor entirely:

```rust
ValidResourceBounds::L1Gas(_) => {
    // No validation required for legacy transactions.
}
``` [6](#0-5) 

---

### Impact Explanation

**High — Mempool/gateway admission accepts invalid transactions or rejects valid transactions before sequencing.**

- **False rejections** directly harm users: a correctly-priced transaction is refused at the gateway with `GAS_PRICE_TOO_LOW` even though it would pass `check_fee_bounds` in the blockifier. The user must resubmit with a higher price, paying more than necessary.
- **False admissions** mislead users: the gateway returns success, but the transaction is silently parked in the mempool's pending queue and will not be sequenced until the gas price falls back below the transaction's `max_price_per_unit`. The user has no indication their transaction is stuck.

---

### Likelihood Explanation

The EIP-1559 mechanism guarantees the L2 gas price changes every block whenever actual gas usage deviates from `gas_target`. Under any sustained load (either above or below target), the price drifts monotonically, making the stale-price window continuously non-zero. The magnitude of the per-block change is bounded by `gas_price_max_change_denominator`, but even a 0.3% change per block is sufficient to create the mismatch for transactions priced near the threshold. No special attacker capability is required — any ordinary user submitting a transaction during a period of changing gas prices will be affected.

---

### Recommendation

Replace the `previous_block_l2_gas_price` fetch with the `next_l2_gas_price` value already stored in the latest block header (the field the TODO comment refers to). The `next_l2_gas_price` is the price that will govern the next block's execution and is the correct reference for admission-time resource bounds validation.

```rust
// Before (wrong):
let previous_block_l2_gas_price = self
    .gateway_fixed_block_state_reader
    .get_block_info()
    .await?
    .gas_prices
    .strk_gas_prices
    .l2_gas_price;

// After (correct):
let next_block_l2_gas_price = self
    .gateway_fixed_block_state_reader
    .get_block_info()
    .await?
    .next_l2_gas_price; // use the EIP-1559-projected price for the next block
```

### Proof of Concept

**Scenario: valid transaction incorrectly rejected (Mode 2)**

1. Latest committed block has `l2_gas_price = 100 FRI` and was under-full (`gas_used < gas_target`).
2. EIP-1559 computes `next_l2_gas_price = 88 FRI`.
3. Gateway config: `min_gas_price_percentage = 90` → admission threshold = `floor(0.90 × 100) = 90`.
4. User submits an invoke V3 transaction with `l2_gas.max_price_per_unit = 89 FRI`.
5. `validate_tx_l2_gas_price_within_threshold` evaluates `89 < 90` → returns `GAS_PRICE_TOO_LOW`.
6. Transaction is rejected. However, `check_fee_bounds` in the blockifier would evaluate `89 ≥ 88` → the transaction would have been accepted and executed successfully.

**Scenario: invalid transaction incorrectly admitted (Mode 1)**

1. Latest committed block has `l2_gas_price = 100 FRI` and was over-full (`gas_used > gas_target`).
2. EIP-1559 computes `next_l2_gas_price = 112 FRI`.
3. Gateway config: `min_gas_price_percentage = 90` → admission threshold = 90.
4. User submits an invoke V3 transaction with `l2_gas.max_price_per_unit = 95 FRI`.
5. `validate_tx_l2_gas_price_within_threshold` evaluates `95 ≥ 90` → transaction admitted, gateway returns success.
6. Batcher calls `mempool_client.update_gas_price(112)` → mempool demotes the transaction (95 < 112) to the pending queue.
7. Transaction is never served to the batcher; user's transaction is silently stuck.
