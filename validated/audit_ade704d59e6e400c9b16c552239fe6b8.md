### Title
Accrued Admin Fees Silently Redirected to New Destination When `setPoolAdminFeeDestination` Is Called Without Prior Settlement — (File: metric-core/contracts/MetricOmmPoolFactory.sol)

---

### Summary

`setPoolAdminFeeDestination` updates `poolAdminFeeDestination[pool]` without first flushing accrued fees to the old destination. Every other fee-configuration mutator (`setPoolAdminFees`, `setPoolProtocolFee`) calls `collectFees` before writing new state. The omission here means all fees that accrued under the old destination are silently redirected to the new one on the next `collectPoolFees` call.

---

### Finding Description

`setPoolAdminFees` and `setPoolProtocolFee` both follow a collect-then-update pattern: they call `collectFees` with the current `poolAdminFeeDestination[pool]` before writing any new configuration, ensuring fees earned at old rates/destinations are settled first. [1](#0-0) [2](#0-1) 

`setPoolAdminFeeDestination` performs no such settlement:

```solidity
function setPoolAdminFeeDestination(address pool, address newAdminFeeDestination)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (newAdminFeeDestination == address(0)) revert InvalidAdminFeeDestination();
    poolAdminFeeDestination[pool] = newAdminFeeDestination;   // ← no collectFees first
    emit PoolAdminFeeDestinationUpdated(pool, newAdminFeeDestination);
}
``` [3](#0-2) 

After the call, `poolAdminFeeDestination[pool]` points to the new address. The pool still holds two categories of unsettled admin fees:

1. **Notional fees** — stored in `notionalFeeToken0Scaled` / `notionalFeeToken1Scaled` on the pool, accumulated since the last `collectFees`.
2. **Spread surplus** — computed live as `balance - binTotals - notionalFees`; this surplus also contains the admin's share of spread fees earned since the last collection. [4](#0-3) 

When `collectPoolFees` (permissionless) is next called, `collectFees` on the pool receives `adminFeeDestination_` = the **new** address and transfers all of the above to it: [5](#0-4) 

The old destination receives nothing for the period it was the configured recipient.

---

### Impact Explanation

The old `poolAdminFeeDestination` loses all admin fees that accrued during its tenure as the configured recipient. Those tokens are transferred to the new destination instead. This is a direct, quantifiable loss of protocol fees owed to the old destination. The magnitude equals the full admin share of both notional and spread fees accumulated since the last `collectFees` call — potentially significant for high-volume pools with long collection intervals.

---

### Likelihood Explanation

The pool admin is a semi-trusted role that legitimately calls `setPoolAdminFeeDestination` (e.g., treasury rotation, multisig upgrade). The trigger requires only a single valid admin transaction while fees are accrued, which is the normal operating state of any active pool. No special market conditions or external actors are needed.

---

### Recommendation

Mirror the collect-then-update pattern used by `setPoolAdminFees` and `setPoolProtocolFee`: call `collectFees` with the **current** `poolAdminFeeDestination[pool]` before overwriting it.

```solidity
function setPoolAdminFeeDestination(address pool, address newAdminFeeDestination)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (newAdminFeeDestination == address(0)) revert InvalidAdminFeeDestination();

    // Settle accrued fees to the OLD destination before rotating
    PoolFeeConfig memory c = poolFeeConfig[pool];
    IMetricOmmPoolCollectFees(pool).collectFees(
        c.protocolSpreadFeeE6,
        c.adminSpreadFeeE6,
        c.protocolNotionalFeeE8,
        c.adminNotionalFeeE8,
        poolAdminFeeDestination[pool]   // old destination
    );

    poolAdminFeeDestination[pool] = newAdminFeeDestination;
    emit PoolAdminFeeDestinationUpdated(pool, newAdminFeeDestination);
}
```

---

### Proof of Concept

1. Pool is active; `poolAdminFeeDestination[pool]` = `OldTreasury`. Swaps accumulate `notionalFeeToken0Scaled = X` and spread surplus `S` on the pool.
2. Pool admin calls `setPoolAdminFeeDestination(pool, NewTreasury)`. No `collectFees` is triggered; `poolAdminFeeDestination[pool]` is now `NewTreasury`.
3. Anyone calls `collectPoolFees(pool)`. The factory reads `poolAdminFeeDestination[pool]` = `NewTreasury` and passes it to `collectFees`.
4. Inside `collectFees`, the admin share of `X` (notional) and `S` (spread) is transferred to `NewTreasury`.
5. `OldTreasury` receives `0` tokens for the entire period it was the configured recipient — a direct loss equal to its accrued admin fee share. [6](#0-5) [7](#0-6)
