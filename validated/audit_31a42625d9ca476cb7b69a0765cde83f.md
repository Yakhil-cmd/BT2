### Title
Unbounded Signature Replay in COA Ownership Proof Validation — (`File: fvm/evm/stdlib/contract.cdc`)

---

### Summary

`EVM.validateCOAOwnershipProof` in `fvm/evm/stdlib/contract.cdc` accepts signatures over an arbitrary caller-supplied `signedData` blob with no nonce and no expiry enforced inside the proof structure. Any valid signature observed on-chain can be replayed indefinitely by an unprivileged attacker to impersonate a COA owner in any EVM-side protocol that relies on ERC-1271 (`isValidSignature`).

---

### Finding Description

`validateCOAOwnershipProof` verifies that a set of Flow account keys have signed `signedData` and that the account holds a `CadenceOwnedAccount` resource at the given path whose EVM address matches `evmAddress`. [1](#0-0) 

The `signedData` parameter is a raw `[UInt8]` blob (a `bytes32` hash at the EVM layer). The `COAOwnershipProof` struct that carries the signatures contains no nonce field and no expiry field: [2](#0-1) 

The Cadence contract itself acknowledges a related replay risk in a comment but dismisses it as low-risk: [3](#0-2) 

The EVM-side entry point is the `verifyCOAOwnershipProof` precompile, which is wired through `coaOwnershipProofValidator` and ultimately calls `validateCOAOwnershipProof`: [4](#0-3) 

The COA's ERC-1271 `isValidSignature` implementation calls this precompile with `address(this)` (the COA's own EVM address) and the caller-supplied `_hash`: [5](#0-4) 

Because neither the proof encoding nor the Cadence validation logic enforces a nonce or an expiry timestamp, a signature over hash `H` that was legitimately produced at time T₁ remains cryptographically valid and accepted by `validateCOAOwnershipProof` at any future time T₂, for any EVM contract that presents the same `H`.

---

### Impact Explanation

**High.** An attacker who observes a valid COA ownership proof signature (e.g., from a public Flow transaction or an off-chain authentication flow) can replay it indefinitely to pass ERC-1271 `isValidSignature` checks on behalf of the victim COA. EVM protocols that gate asset transfers, governance votes, or privileged operations behind ERC-1271 verification are directly affected. The victim account owner has no mechanism to invalidate the captured signature short of revoking or removing the signing key entirely, which is a destructive operation.

---

### Likelihood Explanation

**Low.** The attacker must:
1. Observe a valid, complete `COAOwnershipProof` signature over a specific `bytes32` hash.
2. Find or construct an EVM-side context where the same hash `H` is accepted as authorization for a different action.

For dApps that use predictable or fixed message hashes (e.g., a static "prove ownership" string), or for off-chain authentication flows that reuse the same hash across sessions, the replay is straightforward. The absence of any expiry means the attack window is unbounded.

---

### Recommendation

1. **Add a nonce field to `COAOwnershipProof`** (or require it to be encoded inside `signedData`) that is checked against a per-COA on-chain counter, incremented on each successful proof verification. This prevents replay of any previously used signature.

2. **Add an `expiresAt` field** (block height or timestamp) inside the signed payload. `validateCOAOwnershipProof` should reject proofs whose `expiresAt` is in the past relative to the current block height/timestamp.

3. **Enforce that `signedData` encodes `evmAddress`** at the protocol level rather than leaving it as a caller responsibility (the current comment at line 1003–1009 acknowledges this gap but does not enforce it).

---

### Proof of Concept

1. Alice calls an EVM dApp that requires ERC-1271 proof of COA ownership. She signs hash `H = keccak256("authorize_action_A")` and submits a `COAOwnershipProof` containing her Flow account address, key index, and signature. The transaction is public on-chain.

2. An attacker extracts the encoded `COAOwnershipProof` bytes from the transaction.

3. Six months later, a second EVM protocol uses the same hash `H` (same message format, no domain separator) for a different privileged action B.

4. The attacker calls the second protocol's contract, which internally calls:
   ```solidity
   cadenceArch.staticcall(
       abi.encodeWithSignature(
           "verifyCOAOwnershipProof(address,bytes32,bytes)",
           alice_coa_address, H, captured_proof_bytes
       )
   );
   ```

5. `validateCOAOwnershipProof` verifies the signature (Alice's key is still active, not revoked), confirms the COA address matches, and returns `isValid: true`. [6](#0-5) 

6. The second protocol accepts this as Alice's authorization for action B, which Alice never intended to authorize. There is no nonce check and no expiry check anywhere in the validation path. [7](#0-6)

func coaOwnershipProofValidator(contractAddress flow.Address, backend backends.Backend) func(proof *types.COAOwnershipProofInContext) (bool, error) {
	return func(proof *types.COAOwnershipProofInContext) (bool, error) {
		value, err := backend.Invoke(
			environment.ContractFunctionSpec{
				AddressFromChain: func(_ flow.Chain) flow.Address {
					return contractAddress
				},
				LocationName: "EVM",
				FunctionName: "validateCOAOwnershipProof",
				ArgumentTypes: []sema.Type{
					types.FlowAddressSemaType,
					types.PublicPathSemaType,
					types.SignedDataSemaType,
					types.KeyIndicesSemaType,
					types.SignaturesSemaType,
					types.AddressBytesSemaType,
				},
			},
			proof.ToCadenceValues(),
		)
		if err != nil {
			if types.IsAFatalError(err) {
				panic(err)
			}
			return false, err
		}
		data, ok := value.(cadence.Struct)
		if !ok {
			return false, fmt.Errorf("invalid output data received from validateCOAOwnershipProof")
		}

		isValidValue := cadence.SearchFieldByName(data, ValidationResultTypeIsValidFieldName)
		if isValidValue == nil {
			return false, fmt.Errorf("invalid output data received from validateCOAOwnershipProof")
		}

		return bool(isValidValue.(cadence.Bool)), nil
	}
}
