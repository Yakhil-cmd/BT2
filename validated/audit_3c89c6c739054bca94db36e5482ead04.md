### Title
Gateway blocks mempool admission on non-essential GCS proof archive write, causing valid proof-bearing transactions to be rejected - (File: crates/apollo_gateway/src/gateway.rs)

### Summary
In `GenericGateway::add_tx_inner`, a fully-validated transaction carrying proof facts is blocked from mempool admission by `await_proof_archiving`, which waits up to a hard 5-second timeout for a GCS (Google Cloud Storage) archive write to complete. If the GCS write fails or times out, the transaction is rejected even though it has already passed all stateless and stateful validations and its proof has been durably stored in the proof manager. The GCS archive is a secondary backup and is not required for transaction correctness or execution.

### Finding Description

`add_tx_inner` processes a transaction in this order:

1. Stateless validation (`stateless_tx_validator.validate`)
2. Conversion to internal/executable transaction (`convert_rpc_tx_to_internal_and_executable_txs`)
3. Stateful validation (`extract_state_nonce_and_run_validations`)
4. `store_proof_and_spawn_archiving` — stores the proof durably in the proof manager (filesystem) **and** spawns a GCS archive write task in parallel
5. `await_proof_archiving` — waits up to `PROOF_ARCHIVE_WRITE_TIMEOUT` (5 s) for the GCS task
6. `mempool_client.add_tx` — admits the transaction to the mempool [1](#0-0) 

`await_proof_archiving` converts three distinct failure modes into a hard rejection: [2](#0-1) 

In all three cases (`GCS write error`, `task panic`, `timeout elapsed`) the function returns a `StarknetError::internal` and the transaction is dropped. At that point:

- The proof is already durably stored in the proof manager (step 4 succeeded).
- The transaction passed every stateless and stateful check.
- The GCS archive is explicitly described as a secondary backup ("avoid double writes" for P2P, `NoOpProofArchiveWriter` when bucket is empty). [3](#0-2) 

The `PROOF_ARCHIVE_WRITE_TIMEOUT` constant is 5 seconds: [4](#0-3) 

This is the direct sequencer analog of M-13: a critical operation (mempool admission) is atomically coupled with a non-essential secondary operation (GCS archiving) that can fail independently, causing the critical operation to fail.

### Impact Explanation

Any user submitting a non-P2P invoke transaction with non-empty `proof_facts` during a GCS outage, network degradation, or when the GCS write exceeds 5 seconds receives an internal error and is denied mempool admission. The transaction is valid — it passed all stateless and stateful checks and its proof is durably stored — but it is rejected before sequencing. This matches the allowed impact: **High — Mempool/gateway/RPC admission rejects valid transactions before sequencing.**

### Likelihood Explanation

GCS outages, network timeouts, and slow uploads are realistic operational events. The 5-second hard timeout is short for a cloud storage write. The trigger is unprivileged: any user can submit a transaction with `proof_facts` (the client-side proving path). The issue is active whenever `proof_archive_writer_config.bucket_name` is non-empty (production configuration). [5](#0-4) 

### Recommendation

Decouple GCS archiving from mempool admission. After the proof is stored in the proof manager (step 4), admit the transaction to the mempool immediately. Let the GCS archive task run to completion in the background; log failures as warnings but do not propagate them as transaction errors. This mirrors the M-13 fix: separate the essential operation (mempool admission) from the non-essential operation (GCS archiving).

### Proof of Concept

1. Deploy the gateway with a non-empty `proof_archive_writer_config.bucket_name` (production configuration).
2. Submit an invoke V3 transaction with non-empty `proof_facts` and a valid `proof` via the HTTP server (non-P2P path).
3. Arrange for the GCS write to fail (e.g., revoke bucket permissions, inject a network partition, or delay the response beyond 5 seconds).
4. Observe that `await_proof_archiving` returns `Err(StarknetError::internal(...))` and `add_tx_inner` returns the error to the caller — the transaction is never forwarded to `mempool_client.add_tx`.
5. Confirm that the proof is present in the proof manager (filesystem) while the transaction is absent from the mempool.
6. The user receives an internal error and must retry; retries also fail for the duration of the GCS unavailability, permanently blocking client-side proving transactions from being sequenced.

    async fn await_proof_archiving(proof_archive_handle: ProofArchiveHandle) -> GatewayResult<()> {
        let Some((handle, tx_hash)) = proof_archive_handle else {
            return Ok(());
        };

        let abort_handle = handle.abort_handle();
        match timeout(PROOF_ARCHIVE_WRITE_TIMEOUT, handle).await {
            Ok(Ok((proof_facts_hash, Ok(())))) => {
                info!(
                    "Proof archived successfully. proof_facts_hash: {proof_facts_hash:?}, \
                     tx_hash: {tx_hash:?}"
                );
                Ok(())
            }
            Ok(Ok((proof_facts_hash, Err(e)))) => {
                GATEWAY_PROOF_ARCHIVE_WRITE_FAILURE.increment(1);
                Err(StarknetError::internal_with_logging(
                    &format!(
                        "Failed to archive proof to GCS. proof_facts_hash: {proof_facts_hash:?}, \
                         tx_hash: {tx_hash:?}"
                    ),
                    e,
                ))
            }
            Ok(Err(e)) => {
                GATEWAY_PROOF_ARCHIVE_WRITE_FAILURE.increment(1);
                Err(StarknetError::internal_with_logging(
                    &format!("Proof archive writer task panicked. tx_hash: {tx_hash:?}"),
                    e,
                ))
            }
            Err(elapsed) => {
                abort_handle.abort();
                GATEWAY_PROOF_ARCHIVE_WRITE_FAILURE.increment(1);
                Err(StarknetError::internal_with_logging(
                    &format!(
                        "Proof archive writer timed out after {PROOF_ARCHIVE_WRITE_TIMEOUT:?}. \
                         tx_hash: {tx_hash:?}"
                    ),
                    elapsed,
                ))
            }
        }
    }
