### Title
Staking contract pause blocks `unstake_action` and `remove_from_delegation_pool_action`, temporarily freezing principal funds for users who have completed their exit window — (File: `src/staking/staking.cairo`)

### Summary
The `Staking` contract exposes a `pause` mechanism that blocks all state-changing functions via `general_prerequisites()`. Critically, `unstake_action` and `remove_from_delegation_pool_action` are both gated behind this check. A staker or delegator who has already called `unstake_intent` / `remove_from_delegation_pool_intent`, waited out the full exit window, and is now entitled to withdraw their principal, is completely blocked from doing so while the contract is paused. Their funds are temporarily frozen with no recourse until the security admin unpauses the contract.

### Finding Description

The `Staking` contract stores a boolean `is_paused` flag: [1](#0-0) 

The `pause` function (callable by the security agent) sets this flag. Every state-changing function begins with `general_prerequisites()`, which reverts with `"Contract is paused"` when the flag is set. This is confirmed by the test suite: [2](#0-1) [3](#0-2) 

The two withdrawal-finalisation functions are:

**`unstake_action`** — called by a staker after their exit window has elapsed: [4](#0-3) 

**`remove_from_delegation_pool_action`** — called by a pool contract after a delegator's exit window has elapsed: [5](#0-4) 

Both call `general_prerequisites()` unconditionally as their very first step, with no exception for users who have already satisfied the exit-window condition.

The exit window is a fixed timestamp computed at intent time: [6](#0-5) 

The default window is one week: [7](#0-6) 

Once `unstake_intent` is called, the staker's own stake and all delegated stake are removed from `total_stake` immediately: [8](#0-7) 

The staker is no longer earning rewards and is no longer counted in consensus, yet their principal remains locked in the contract until `unstake_action` succeeds.

### Impact Explanation

**High — Temporary freezing of funds.**

A staker or delegator who has completed the mandatory exit window is entitled to their principal. If the contract is paused at any point after the exit window expires, `unstake_action` and `remove_from_delegation_pool_action` both revert. The principal (STRK or BTC tokens) remains locked in the staking contract for the entire duration of the pause. The user has no alternative withdrawal path. The pause has no maximum duration enforced on-chain; only the security admin can lift it.

### Likelihood Explanation

The security agent can pause the contract at any time for any reason (e.g., a suspected exploit elsewhere in the system). The pause is a legitimate operational tool, so it will be used. Any staker or delegator who happens to have a matured exit intent during a pause period is affected. Given that exit windows are one week and pauses can last days, the overlap is realistic.

### Recommendation

Exempt `unstake_action` and `remove_from_delegation_pool_action` from the pause check when the caller's exit window has already elapsed. Alternatively, introduce a separate "withdrawal-only" mode that keeps these two functions callable even while the contract is otherwise paused, so that users who have already completed the exit process can always retrieve their principal.

### Proof of Concept

1. Staker calls `unstake_intent()`. `unstake_time` is set to `now + 1 week`. Staker's stake is removed from `total_stake`.
2. One week passes. Staker's exit window is now satisfied (`Time::now() >= unstake_time`).
3. Security agent calls `pause()` on the staking contract.
4. Staker calls `unstake_action(staker_address)`. The call hits `general_prerequisites()` first, which reads `is_paused == true` and panics with `"Contract is paused"`.
5. Staker's principal remains locked in the contract. They cannot stake elsewhere, cannot earn rewards, and cannot withdraw — until the security admin decides to unpause.

The same sequence applies to a delegator via `remove_from_delegation_pool_action` called through the pool contract, which in turn calls `general_prerequisites()` on the staking contract: [9](#0-8)
