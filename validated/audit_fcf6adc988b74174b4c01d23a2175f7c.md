### Title
Missing Permission Check and No-Op Implementation in `MsgTurnBridge` Handler Bypasses Bridge Authorization - (File: `x/cronos/keeper/msg_server.go`)

### Summary

The `TurnBridge` message handler in the Cronos keeper is a stub that performs no permission check and executes no state change. Any unprivileged user can submit a `MsgTurnBridge` transaction and receive a success response, completely bypassing the `CanTurnBridge` permission system that was explicitly designed to restrict this operation.

### Finding Description

The Cronos module defines a permission system (ADR-009) with two restricted operations: `CanChangeTokenMapping` and `CanTurnBridge`. The `UpdateTokenMapping` handler correctly enforces its permission:

```go
// UpdateTokenMapping implements the grpc method
func (k msgServer) UpdateTokenMapping(...) (...) {
    if !k.HasPermission(ctx, msg.GetSigners(), CanChangeTokenMapping) {
        return nil, errors.Wrap(sdkerrors.ErrUnauthorized, "msg sender is not authorized")
    }
    ...
}
```

However, the `TurnBridge` handler — the other restricted operation — is implemented as a bare stub:

```go
// TurnBridge implements the grpc method
func (k msgServer) TurnBridge(goCtx context.Context, msg *types.MsgTurnBridge) (*types.MsgTurnBridgeResponse, error) {
    return nil, nil
}
```

There is no call to `HasPermission(ctx, ..., CanTurnBridge)`, and no state is written. The function unconditionally returns success.

The permission constant `CanTurnBridge` is defined alongside `CanChangeTokenMapping`, the `MsgTurnBridge` proto message is fully defined with a `sender` signer field, and the integration test `test_gravity_turn_bridge` explicitly asserts that an unprivileged `community` address must be rejected. All of this confirms the permission check is a required, intentional guard that is simply absent.

### Impact Explanation

Two distinct impacts arise from this single stub:

1. **Auth bypass (High)**: Any unprivileged address can submit `MsgTurnBridge{enable: false}` and receive `MsgTurnBridgeResponse{}` with code 0. The `CanTurnBridge` permission check — the sole authorization gate for this operation — is never evaluated. This directly satisfies: *"High: Bypass of Cronos admin, governance authority, permission, token-mapping, bridge, block-list, or module-account authorization checks."*

2. **Permanent inability to disable the bridge (High)**: Because the handler writes no state, even a legitimately authorized admin or permissioned address cannot actually disable the Gravity bridge. In an active bridge exploit scenario, the emergency circuit breaker is permanently non-functional. This satisfies: *"High: Permanent or long-lived inability for honest users or validators to process valid transactions, bridge/conversion flows … under normal network assumptions."*

### Likelihood Explanation

The entry path is a standard Cosmos SDK transaction (`/cronos.Msg/TurnBridge`) reachable by any address on the network. No special privileges, leaked keys, or cryptographic breaks are required. The transaction fee is the only barrier.

### Recommendation

Implement `TurnBridge` with the same pattern used by `UpdateTokenMapping`:

```go
func (k msgServer) TurnBridge(goCtx context.Context, msg *types.MsgTurnBridge) (*types.MsgTurnBridgeResponse, error) {
    ctx := sdk.UnwrapSDKContext(goCtx)

    senderAddr, err := sdk.AccAddressFromBech32(msg.Sender)
    if err != nil {
        return nil, err
    }
    if !k.HasPermission(ctx, []sdk.AccAddress{senderAddr}, CanTurnBridge) {
        return nil, errors.Wrap(sdkerrors.ErrUnauthorized, "msg sender is not authorized")
    }

    // update the EnableAutoDeployment or bridge-enabled param as appropriate
    params := k.GetParams(ctx)
    params.EnableAutoDeployment = msg.Enable
    if err := k.SetParams(ctx, params); err != nil {
        return nil, err
    }
    return &types.MsgTurnBridgeResponse{}, nil
}
```

### Proof of Concept

The existing integration test `test_gravity_turn_bridge` already encodes the expected behavior and demonstrates the bug:

```python
# turn off bridge
rsp = cli.turn_bridge("false", from_="community")
assert rsp["code"] != 0, "should not have the permission"  # FAILS: code is 0 (success)

rsp = cli.turn_bridge("false", from_="validator")
assert rsp["code"] == 0, rsp["raw_log"]  # passes, but bridge is NOT actually disabled
```

With the current stub, the first assertion fails because the unprivileged `community` address receives `code == 0`. The second call also succeeds but writes no state, so the bridge remains enabled and subsequent assertions that bridge operations are blocked will also fail.

```go
// HasPermission check if an account has a specific permission. by default cronos admin has all permissions
func (k Keeper) HasPermission(ctx sdk.Context, accounts []sdk.AccAddress, permissionsToCheck uint64) bool {
	// case when no permission is needed
	if permissionsToCheck == 0 {
		return true
	}
	admin := k.GetParams(ctx).CronosAdmin
	for _, account := range accounts {
		if admin == account.String() {
			return true
		}
		permission := k.GetPermissions(ctx, account)
		if permission&permissionsToCheck == permissionsToCheck {
			return true
		}
	}

	return false
```
