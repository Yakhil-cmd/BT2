### Title
Unchecked Precompile `call` Return Value After `_burn` in Exit Functions — (`File: etc/eth-contracts/contracts/EvmErc20.sol`, `etc/eth-contracts/contracts/EvmErc20V2.sol`)

---

### Summary

Both `EvmErc20.sol` and `EvmErc20V2.sol` implement `withdrawToNear` and `withdrawToEthereum` functions that first burn the caller's ERC-20 tokens and then invoke an Aurora exit precompile via a low-level `call`. The return value of that `call` (stored in `res`) is **never checked**. If the precompile call fails, the burn is irreversible but no bridge transfer occurs, permanently destroying the user's funds.

---

### Finding Description

In both `EvmErc20.sol` and `EvmErc20V2.sol`, the exit functions follow this pattern:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // tokens destroyed here

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        // res is NEVER checked — no revert if call fails
    }
}

The same pattern appears in `withdrawToEthereum` (precompile `0xb0bd02f6a392af548bdf1cfaee5dfa0eefcc8eab`): [2](#0-1) 

And identically in `EvmErc20V2.sol`: [3](#0-2) [4](#0-3) 

The `_burn` is executed unconditionally before the precompile call. If the low-level `call` returns `0` (failure), the EVM does not revert — the transaction succeeds, the tokens are gone, and no NEAR or Ethereum-side transfer is initiated.

---

### Impact Explanation

**Critical — Permanent freezing / direct theft of user funds.**

A user who calls `withdrawToNear` or `withdrawToEthereum` has their ERC-20 balance permanently destroyed. If the precompile call fails silently, the corresponding NEP-141 or Ethereum-side tokens are never released. The user suffers a total, unrecoverable loss of the bridged asset with no on-chain indication of failure.

---

### Likelihood Explanation

**Medium.** The exit precompiles can fail under realistic conditions:

- The Aurora engine contract is paused (admin-controlled pause flag is set).
- The precompile receives malformed or out-of-bounds input (e.g., a very long `recipient` field causing gas exhaustion inside the precompile).
- Insufficient gas is forwarded to the precompile (the `gas()` opcode forwards all remaining gas, but if the caller's transaction was submitted with a tight gas limit, the inner call can run out).

Any token holder can trigger this path without any special privilege.

---

### Recommendation

Check the return value of the low-level `call` and revert if it indicates failure. The fix mirrors the `safeTransfer` pattern from the original report — wrap the critical operation so that failure propagates:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

Apply this fix to both `withdrawToNear` and `withdrawToEthereum` in both `EvmErc20.sol` and `EvmErc20V2.sol`.

---

### Proof of Concept

1. Deploy `EvmErc20` (or `EvmErc20V2`) on Aurora.
2. Mint tokens to `attacker` address.
3. Pause the Aurora engine (or simulate a precompile failure by calling with insufficient gas).
4. Call `withdrawToNear(recipient, amount)` from `attacker`.
5. Observe: `_burn` succeeds → `attacker`'s ERC-20 balance is zero. The low-level `call` to the exit precompile returns `0` (failure). No revert occurs. The transaction is marked successful.
6. Result: `attacker` has lost `amount` tokens permanently with no NEAR-side credit.
