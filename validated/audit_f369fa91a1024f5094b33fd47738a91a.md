### Title
Unbounded Loop in `calculate_rewards` Over Unclaimed Balance Trace Entries Can Permanently Freeze Pool Member Yield - (File: `src/pool/pool.cairo`)

### Summary
`Pool::calculate_rewards` iterates over every balance-change checkpoint in a pool member's `pool_member_epoch_balance` trace since their last claim. Because the trace grows by one entry per epoch in which the member changes their balance, a member who makes frequent balance changes without claiming rewards accumulates an unbounded trace. When the trace is large enough, the `claim_rewards` transaction exceeds the Starknet gas limit and reverts, permanently freezing the member's unclaimed yield.

### Finding Description

`calculate_rewards` in `src/pool/pool.cairo` contains an explicitly acknowledged unbounded loop:

```
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    ...
    entry_to_claim_from += 1;
}
``` [1](#0-0) 

The trace (`pool_member_epoch_balance`) is a `Vec`-backed append-only structure. A new checkpoint is pushed whenever `set_member_balance` is called with a key (`current_epoch + K`) that differs from the last stored key:

```cairo
fn set_member_balance(...) {
    trace.insert(key: self.get_epoch_plus_k(), value: pool_member_balance);
}
``` [2](#0-1) 

The `insert` implementation appends a new checkpoint when the epoch key is strictly greater than the last stored key:

```cairo
checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
``` [3](#0-2) 

`set_member_balance` is called from three unprivileged entry points:
- `enter_delegation_pool` (line 201)
- `add_to_delegation_pool` → `increase_member_balance` (line 242)
- `exit_delegation_pool_intent` (line 278) [4](#0-3) [5](#0-4) [6](#0-5) 

Each call made in a different epoch appends a distinct entry. A pool member who performs one balance-change operation per epoch accumulates one new trace entry per epoch. The `entry_to_claim_from` cursor is only advanced inside `claim_rewards` after the loop completes successfully:

```cairo
pool_member_info.entry_to_claim_from = updated_entry_to_claim_from;
``` [7](#0-6) 

If the loop runs out of gas, the transaction reverts, `entry_to_claim_from` is never updated, and every subsequent `claim_rewards` attempt faces the same (or larger) loop — permanently blocking reward withdrawal.

### Impact Explanation

A pool member who makes balance changes across many epochs without claiming rewards will eventually be unable to call `claim_rewards` successfully. Because `entry_to_claim_from` is only persisted on a successful transaction, no partial progress is possible. The member's entire accumulated unclaimed yield is permanently frozen.

**Impact: High — Permanent freezing of unclaimed yield.**

### Likelihood Explanation

Any pool member who actively manages their delegation (adding or partially exiting across many epochs) and defers reward claims for a long period will naturally accumulate a large trace. The protocol imposes no upper bound on trace length and no mechanism to prune old entries. On Starknet, where per-transaction gas limits are enforced, a trace of sufficient size (reachable over months of normal use) will cause every `claim_rewards` call to revert. No adversarial action is required — normal delegator behavior is sufficient.

### Recommendation

After a successful `claim_rewards`, prune all trace entries with index `< updated_entry_to_claim_from` from `pool_member_epoch_balance`. Alternatively, cap the number of iterations per `claim_rewards` call and allow partial claiming so that `entry_to_claim_from` advances incrementally across multiple transactions, preventing permanent lock-up.

### Proof of Concept

1. Pool member calls `enter_delegation_pool` in epoch `E`.
2. In each subsequent epoch `E+1, E+2, ..., E+N`, the member calls `add_to_delegation_pool` with a minimal amount. Each call inserts a new checkpoint at `current_epoch + K` into `pool_member_epoch_balance`.
3. The member never calls `claim_rewards`, so `entry_to_claim_from` remains at `0`.
4. After `N` epochs, `pool_member_trace_length == N`.
5. The member calls `claim_rewards`. `calculate_rewards` enters the `while` loop and iterates `N` times, each iteration reading a storage slot and calling `find_sigma`.
6. For sufficiently large `N` (bounded only by Starknet's per-transaction gas limit), the transaction runs out of gas and reverts.
7. `entry_to_claim_from` is never updated. Every future `claim_rewards` call faces the same or larger loop and also reverts. The member's yield is permanently frozen. [8](#0-7) [9](#0-8)

            let mut pool_member_info = self.internal_pool_member_info(:pool_member);

        fn calculate_rewards(
            self: @ContractState,
            pool_member: ContractAddress,
            from_checkpoint: PoolMemberCheckpoint,
            until_checkpoint: PoolMemberCheckpoint,
            mut entry_to_claim_from: VecIndex,
        ) -> (Amount, VecIndex) {
            let pool_member_trace = self.pool_member_epoch_balance.entry(pool_member);
            // Note: `until_epoch` is the current epoch.
            let until_epoch = until_checkpoint.epoch();

            let mut rewards = 0;

            let pool_member_trace_length = pool_member_trace.length();

            let mut from_sigma = self.find_sigma(from_checkpoint, curr_epoch: until_epoch);
            let mut from_balance = from_checkpoint.balance();

            let base_value = self.staking_rewards_base_value.read();

            // **Note**: The loop iterates over the balance changes in the pool member's balance
            // trace. This loop is unbounded but unlikely to exceed gas limits.
            while entry_to_claim_from < pool_member_trace_length {
                let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
                // If the balance change is after `until_epoch` (and therefore does not affect
                // the current reward computation), exit the loop.
                if pool_member_checkpoint.epoch() >= until_epoch {
                    break;
                }

                // Compute rewards from (inclusive) the previous balance change (or from
                // `from_checkpoint`) to (exclusive) the current entry.
                let to_sigma = self.find_sigma(pool_member_checkpoint, curr_epoch: until_epoch);
                rewards +=
                    compute_rewards_rounded_down(
                        amount: from_balance, interest: to_sigma - from_sigma, :base_value,
                    );
                from_sigma = to_sigma;
                from_balance = pool_member_checkpoint.balance();
                entry_to_claim_from += 1;
            }

            // Compute the remaining rewards from (inclusive) the last visited balance change in
            // `pool_member_trace` (or from `from_checkpoint`) to (exclusive) `until_checkpoint`.
            let to_sigma = self.find_sigma(until_checkpoint, curr_epoch: until_epoch);
            rewards +=
                compute_rewards_rounded_down(
                    amount: from_balance, interest: to_sigma - from_sigma, :base_value,
                );

            (rewards, entry_to_claim_from)
        }
