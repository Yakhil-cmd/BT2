### Title
Cross-COA Signature Replay in `validateCOAOwnershipProof` Allows Forged ERC-1271 Ownership Proofs - (File: `fvm/evm/stdlib/contract.cdc`)

### Summary

`EVM.validateCOAOwnershipProof` verifies that a Cadence account's keys signed `signedData`, then separately checks that the `evmAddress` argument matches the COA resource at the given `path`. Because `evmAddress` is never bound into `signedData`, a valid proof produced for COA_A can be replayed against COA_B owned by the same Cadence account simply by substituting the `CapabilityPath` field in the proof. The COA's ERC-1271 `isValidSignature` entry point is directly reachable by any unprivileged EVM transaction sender, making this exploitable without any privileged access.

### Finding Description

`validateCOAOwnershipProof` in `fvm/evm/stdlib/contract.cdc` (lines 1001ŌĆō1116) accepts six parameters: `address`, `path`, `signedData`, `keyIndices`, `signatures`, and `evmAddress`. The cryptographic check at lines 1082ŌĆō1086 verifies the signatures only over `signedData` with domain tag `"FLOW-V0.0-user"`:

```cadence
let isValid = keyList.verify(
    signatureSet: signatureSet,
    signedData: signedData,
    domainSeparationTag: "FLOW-V0.0-user"
)
```

The `evmAddress` parameter is checked in a completely separate step (lines 1095ŌĆō1110) by borrowing the COA resource at `path` and comparing its `.address()` to `evmAddress`. These two checks are independent: the signature does not commit to `evmAddress`, and `evmAddress` is not verified against `signedData`.

The code itself acknowledges this in the comment at lines 1003ŌĆō1009:

> "this function does not enforce that `signedData` includes `evmAddress`. In principle, a signature produced for one purpose could be replayed here against a different COA owned by the same Cadence account."

The `COAOwnershipProof` struct (encoded in the `_sig` bytes passed to `isValidSignature`) contains an attacker-controlled `CapabilityPath` field. The ERC-1271 entry point in `coa.sol` (lines 114ŌĆō125) passes `address(this)` as the fixed `evmAddress` but accepts the entire `_sig` blob from the caller:

```solidity
cadenceArch.staticcall(abi.encodeWithSignature(
    "verifyCOAOwnershipProof(address,bytes32,bytes)",
    address(this), _hash, _sig   // _sig is fully attacker-controlled
));
```

**Exploit path:**

1. Cadence account `A` owns two COAs: COA_X at `/public/coaX` and COA_Y at `/public/coaY`.
2. Account `A` legitimately signs `hash_H` to prove ownership of COA_X. The resulting `COAOwnershipProof` contains `CapabilityPath = /public/coaX` and valid signatures over `hash_H`.
3. An attacker observes this proof (it is passed as calldata in an EVM transaction, which is public).
4. The attacker constructs a new `COAOwnershipProof` with identical `Address`, `KeyIndices`, and `Signatures`, but sets `CapabilityPath = /public/coaY`.
5. The attacker calls `COA_Y.isValidSignature(hash_H, new_proof)`.
6. `verifyCOAOwnershipProof(COA_Y_address, hash_H, new_proof)` is invoked via the Cadence Arch precompile.
7. Signature verification passes: same keys, same `hash_H`.
8. COA address check passes: the resource at `/public/coaY` has address `COA_Y_address` Ō£ō.
9. `isValidSignature` returns `0x1626ba7e` (valid) ŌĆö the attacker has forged a valid ERC-1271 signature for COA_Y.

### Impact Explanation

Any EVM smart contract that uses ERC-1271 to gate asset transfers, access control, or approvals on a COA can be bypassed. An attacker who observes a single valid ownership proof for any COA of a multi-COA Cadence account can replay it against every other COA owned by the same account. This enables unauthorized authorization of EVM-side actions (token approvals, DAO votes, NFT marketplace listings, etc.) on behalf of COAs the account owner never intended to authorize. The impact is unauthorized state change and potential asset loss on the EVM side.

### Likelihood Explanation

The `validateCOAOwnershipProof` function is `access(all)` and callable from any Cadence script or transaction. The ERC-1271 `isValidSignature` path is reachable by any unprivileged EVM transaction sender. The only precondition is that the target Cadence account owns more than one COA ŌĆö a condition that is already possible today (any Cadence transaction can call `EVM.createCadenceOwnedAccount()` multiple times) and will become increasingly common as the ecosystem matures. The attacker needs only to observe a single valid proof in on-chain calldata, which is always public.

### Recommendation

Bind `evmAddress` into the signed message. The `signedData` that the Cadence account signs should commit to the specific COA EVM address being proven. Concretely, `validateCOAOwnershipProof` should enforce that `signedData` encodes `evmAddress` (e.g., by prepending or hashing it in), or the function should construct the message to be verified as `hash(evmAddress || signedData)` before calling `keyList.verify`. Wallets and off-chain tooling that produce `COAOwnershipProof` values must be updated to include the target `evmAddress` in the signed payload.

// COAOwnershipProof is a proof that a flow account
// controls a COA resource. To do so, the flow
// account (Address is address of this account)

	Signatures     Signatures
