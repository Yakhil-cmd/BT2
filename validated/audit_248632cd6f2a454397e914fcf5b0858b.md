### Title
Permanent Freeze of Unwrapped NEAR When Transfer to Non-Existent Target Account Fails — (`engine/src/contract_methods/connector.rs`)

---

### Summary

When a user calls the `ExitToNear` precompile to unwrap wNEAR ERC-20 tokens into native NEAR (using the `:unwrap` suffix), the flow executes two sequential NEAR promises: first `near_withdraw` on the wNEAR NEP-141 contract, then a `PromiseAction::Transfer` to the user-specified target NEAR account. If `near_withdraw` succeeds but the subsequent `Transfer` fails (e.g., the target named NEAR account does not exist), the unwrapped NEAR is permanently stranded in the Aurora engine contract's balance. The user's wNEAR ERC-20 tokens were already burned in the EVM (irreversible), and no recovery or refund mechanism exists for this specific failure path.

---

### Finding Description

The wNEAR unwrap path is set up in `exit_erc20_token_to_near` when `Message::UnwrapWnear` is matched: [1](#0-0) 

This produces `transfer_near_args = Some(TransferNearArgs { target_account_id, amount })`, which is embedded into `ExitToNearPrecompileCallbackArgs` and attached as a callback to the `near_withdraw` base promise: [2](#0-1) 

When `near_withdraw` succeeds, `exit_to_near_precompile_callback` fires and creates a bare `PromiseAction::Transfer` to `args.target_account_id`: [3](#0-2) 

The critical flaw is on line 226–227: `promise_create_batch` is called and `promise_return` is invoked, but **no further callback is attached** to handle the case where this Transfer promise fails. In NEAR Protocol, a `Transfer` action to a named account that does not exist will fail. When it does, the NEAR remains in the Aurora engine contract's balance with no mechanism to return it to the user.

The existing `error_refund` feature only covers the case where `near_withdraw` itself fails (the `else if let Some(args) = args.refund` branch at line 231). It does not cover the case where `near_withdraw` succeeds but the downstream Transfer fails: [4](#0-3) 

There is no second-level callback, no re-mint of wNEAR ERC-20, and no admin recovery function scoped to this scenario.

---

### Impact Explanation

**Permanent freezing of funds (Critical).**

The user's wNEAR ERC-20 tokens are burned in the EVM state before any NEAR-side promise executes. Once `near_withdraw` succeeds, the NEAR is held by the Aurora engine contract. If the Transfer to the target account then fails, the NEAR is stranded in the engine's balance indefinitely. The user has no on-chain path to recover it: they cannot re-trigger the transfer, they cannot reclaim the burned ERC-20 tokens, and there is no admin escape hatch scoped to this specific stuck balance.

---

### Likelihood Explanation

Any EVM user can trigger this path by calling the `ExitToNear` precompile at `0xe9217bc70b7ed1f598ddd3199e80b093fa71124f` with a wNEAR ERC-20 burn and a `:unwrap` suffix pointing to a named NEAR account that does not exist. NEAR account existence is not validated at EVM call time — the precompile only parses the account ID string for syntactic validity. A user who mistypes a NEAR account name, uses a deleted account, or is unaware that the target account must be pre-created will silently lose their funds. This is a realistic, unprivileged, user-reachable scenario.

---

### Recommendation

Attach a second-level callback to the `Transfer` promise created inside `exit_to_near_precompile_callback`. If the Transfer fails, the callback should re-mint the equivalent amount of wNEAR ERC-20 tokens to the original sender's EVM address (analogous to the existing `refund_on_error` path used when `near_withdraw` itself fails). This mirrors the pattern already established by the `error_refund` feature for the first promise hop and closes the gap for the second hop.

---

### Proof of Concept

1. User holds wNEAR ERC-20 tokens on Aurora and calls the `ExitToNear` precompile (`0xe9217bc70b7ed1f598ddd3199e80b093fa71124f`) with:
   - Flag `0x1` (ERC-20 exit)
   - Amount: e.g. `1000000000000000000` (1 wNEAR)
   - Recipient: `doesnotexist.near:unwrap`
2. The EVM burns 1 wNEAR ERC-20 from the user's balance (irreversible).
3. A `near_withdraw` promise is scheduled on the wNEAR NEP-141 contract; it succeeds and transfers 1 NEAR to the Aurora engine account.
4. `exit_to_near_precompile_callback` fires. Since `promise_result(0)` is `Successful`, it creates a `PromiseAction::Transfer { amount: 1 NEAR }` targeting `doesnotexist.near`.
5. The Transfer fails because `doesnotexist.near` does not exist on NEAR.
6. No callback handles this failure. The 1 NEAR remains in the Aurora engine contract's balance. The user's wNEAR ERC-20 tokens are gone. There is no recovery path. [5](#0-4) [3](#0-2)
