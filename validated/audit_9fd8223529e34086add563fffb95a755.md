### Title
Unauthorized Invoke Admission via `skip_stateful_validations` Bypasses Signature Verification for Any Account with a Pending Deploy-Account — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator skips the `__validate__` entry-point call (which performs signature verification) for any Invoke transaction whose sender address has *any* transaction in the mempool and whose on-chain nonce is still zero. Because the check `account_tx_in_pool_or_recent_block` does not verify that the caller is the account owner, any third party can submit an unsigned (or maliciously-signed) Invoke with `nonce = 1` for a victim account that has a pending `DeployAccount`, and the gateway will admit it to the mempool without ever verifying the signature.

---

### Finding Description

**Invariant broken:** Only the account owner (or an authorized party whose signature passes `__validate__`) should be able to submit an Invoke transaction on behalf of an account.

**Root cause — `skip_stateful_validations`:** [1](#0-0) 

The function returns `true` (skip validation) when all three conditions hold:

1. The transaction is an `Invoke`.
2. `tx.nonce() == Nonce(Felt::ONE)` — the transaction claims to be the first post-deploy invoke.
3. `account_nonce == Nonce(Felt::ZERO)` — the account is not yet deployed on-chain.
4. `account_tx_in_pool_or_recent_block(sender_address)` returns `true`.

Condition 4 only checks whether *any* transaction for that address exists in the mempool or was recently committed: [2](#0-1) 

It does **not** verify that the submitter of the current Invoke is the same party who submitted the `DeployAccount`. Any observer who sees a `DeployAccount` in the mempool can immediately satisfy condition 4 for the victim's address.

**Effect of the skip:**

When `skip_validate = true`, `run_validate_entry_point` sets `validate: false`: [3](#0-2) 

Inside `StatefulValidator::perform_validations`, the `__validate__` call is short-circuited: [4](#0-3) 

`validate_tx` itself also returns `Ok(None)` immediately when `validate = false`: [5](#0-4) 

The signature is therefore **never checked** at gateway admission time.

**Mempool nonce-slot occupation:**

The mempool's `validate_tx` checks only for duplicate tx-hash and nonce ordering — not signatures: [6](#0-5) 

Once the attacker's unsigned Invoke occupies `(address, nonce=1)`, a subsequent legitimate Invoke from Alice at the same nonce is either rejected as `DuplicateNonce` or must pay a fee-escalation premium to replace it: [7](#0-6) 

**What happens at execution time:**

The batcher creates transactions with `validate: true` via `new_for_sequencing`: [8](#0-7) 

So when the batcher eventually executes the attacker's Invoke, `__validate__` runs and fails (invalid signature → `PanicInValidate` or `ValidateTransactionError`). The transaction is rejected and no fee is charged to Alice. However, the damage has already occurred at the mempool level.

---

### Impact Explanation

**High — Mempool/gateway admission accepts invalid transactions before sequencing.**

1. An attacker can submit an unsigned Invoke for any account that has a pending `DeployAccount` in the mempool.
2. The gateway admits it without signature verification.
3. The attacker's transaction occupies the victim's `nonce = 1` slot in the mempool.
4. Alice's legitimate Invoke is either rejected (`DuplicateNonce`) or must pay a fee-escalation premium (tip + max-l2-gas-price both increased by `fee_escalation_percentage`, default 10%) to displace the attacker's transaction.
5. If Alice does not resubmit in time, her `DeployAccount` is processed but her intended first Invoke is absent from the block, breaking the UX guarantee the feature was designed to provide.

---

### Likelihood Explanation

- The mempool is observable (transactions are broadcast).
- The attack window is the time between a `DeployAccount` entering the mempool and being committed on-chain — typically one block.
- No special privilege is required; any network participant can submit the malicious Invoke.
- The attacker only needs to know the victim's address (derivable from the `DeployAccount` fields).

---

### Recommendation

Replace the ownership-blind `account_tx_in_pool_or_recent_block` check with a check that verifies the mempool contains a **`DeployAccount`** transaction specifically for the sender address, not just any transaction. Alternatively, record the `tx_hash` of the `DeployAccount` at submission time and require the paired Invoke to reference it (as the `native_blockifier` Python path already does via `deploy_account_tx_hash`): [9](#0-8) 

The Rust gateway path should adopt the same pattern: require the caller to supply the `deploy_account_tx_hash` and verify it exists in the mempool as a `DeployAccount` for the same sender address before skipping `__validate__`.

---

### Proof of Concept

**Setup:**
- Alice pre-funds address `A` and submits `DeployAccount(class_hash=C, salt=S, nonce=0)`. The mempool now contains Alice's `DeployAccount`; `account_tx_in_pool_or_recent_block(A)` returns `true`.

**Attack:**
1. Bob observes Alice's `DeployAccount` in the mempool.
2. Bob submits `Invoke(sender_address=A, nonce=1, calldata=<arbitrary>, signature=[])` to the gateway.
3. Gateway calls `skip_stateful_validations`:
   - `tx.nonce() == 1` ✓
   - `account_nonce == 0` ✓ (A not yet deployed)
   - `account_tx_in_pool_or_recent_block(A)` ✓ (Alice's DeployAccount is there)
   - Returns `true` → `validate: false` → `__validate__` is never called.
4. Bob's Invoke is admitted to the mempool at `(A, nonce=1)`.
5. Alice submits her legitimate `Invoke(sender_address=A, nonce=1, calldata=<real>, signature=<valid>)`.
6. The mempool rejects Alice's Invoke with `DuplicateNonce` (or requires fee escalation ≥ 10% higher tip and gas price to replace Bob's transaction).
7. The batcher processes Alice's `DeployAccount` (nonce 0), then Bob's Invoke (nonce 1). `__validate__` fails on Bob's Invoke → rejected, no fee charged. But Alice's legitimate Invoke was never included.

**Corrupted value:** The mempool admission decision — an unsigned Invoke is accepted as valid, and the victim's nonce slot is occupied by an unauthorized transaction.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}

    fn validate_fee_escalation(
        &self,
        incoming_tx_reference: TransactionReference,
    ) -> MempoolResult<Option<TransactionReference>> {
        let TransactionReference { address, nonce, .. } = incoming_tx_reference;

        self.validate_no_delayed_declare_front_run(incoming_tx_reference)?;

        if !self.config.static_config.enable_fee_escalation {
            if self.tx_pool.get_by_address_and_nonce(address, nonce).is_some() {
                return Err(MempoolError::DuplicateNonce { address, nonce });
            };

            return Ok(None);
        }

        let Some(existing_tx_reference) = self.tx_pool.get_by_address_and_nonce(address, nonce)
        else {
            // Replacement irrelevant: no existing transaction with the same nonce for address.
            return Ok(None);
        };

        if !self.should_replace_tx(&existing_tx_reference, &incoming_tx_reference) {
            info!(
                "{existing_tx_reference} was not replaced by {incoming_tx_reference} due to \
                 insufficient fee escalation."
            );
            // TODO(Elin): consider adding a more specific error type / message.
            return Err(MempoolError::DuplicateNonce { address, nonce });
        }

        Ok(Some(existing_tx_reference))
    }

    pub fn should_run_stateful_validations(
        &mut self,
        account_tx: &AccountTransaction,
        deploy_account_tx_hash: Option<TransactionHash>,
    ) -> StatefulValidatorResult<bool> {
        if account_tx.tx_type() != TransactionType::InvokeFunction {
            return Ok(true);
        }
        let tx_info = account_tx.create_tx_info();
        let nonce = self.stateful_validator.get_nonce(tx_info.sender_address())?;

        let deploy_account_not_processed =
            deploy_account_tx_hash.is_some() && nonce == Nonce(Felt::ZERO);
        let tx_nonce = tx_info.nonce();
        let is_post_deploy_nonce = Nonce(Felt::ONE) <= tx_nonce;
        let nonce_small_enough_to_qualify_for_validation_skip =
            tx_nonce <= self.max_nonce_for_validation_skip;

        let skip_validate = deploy_account_not_processed
            && is_post_deploy_nonce
            && nonce_small_enough_to_qualify_for_validation_skip;

        Ok(!skip_validate)
    }
