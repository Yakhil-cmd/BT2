### Title
Changing ETH Connector Account Without Outstanding Balance Check Can Permanently Freeze User ETH — (File: `engine/src/contract_methods/connector.rs`)

---

### Summary

`set_eth_connector_contract_account` allows the Aurora Engine owner to replace the NEP-141 connector account that backs all EVM ETH balances. No check is performed to verify that the old connector holds zero outstanding NEP-141 token balance before the swap. If the new connector does not hold the corresponding NEP-141 tokens, every subsequent user withdrawal (exit to NEAR) is routed to the new connector, which cannot fulfill the transfer, permanently freezing all ETH held inside Aurora.

---

### Finding Description

`set_eth_connector_contract_account` writes a new connector `AccountId` directly to storage: [1](#0-0) 

It performs no check on the outstanding NEP-141 balance held by the old connector. The stored connector account ID is then used by every outbound promise — including withdrawals — via `get_connector_account_id`: [2](#0-1) 

All withdrawal paths call `return_promise`, which resolves the target account through `get_connector_account_id` at call time: [3](#0-2) 

This means the moment the connector account is swapped, every pending and future withdrawal is directed to the new connector. If the new connector does not hold the NEP-141 tokens that back existing EVM ETH balances, those withdrawals fail. The EVM-side balances remain in Aurora's internal storage: [4](#0-3) 

but there is no mechanism to recover them through normal contract interfaces.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

All ETH deposited into Aurora is backed 1:1 by NEP-141 tokens held by the connector contract. Replacing the connector with an account that does not hold those tokens severs the backing relationship. Users' EVM balances are still recorded in Aurora's storage but cannot be withdrawn because every exit call is routed to the new, unfunded connector. There is no fallback path or recovery function in the contract to redirect withdrawals back to the old connector's token pool.

---

### Likelihood Explanation

**Low-to-Medium.**

The function is owner-only, so it requires a legitimate administrative action. However, the scenario is realistic: an owner upgrading or migrating the connector contract (a normal operational event) could swap the account before ensuring the new connector is fully funded, or before draining the old one. The absence of any safety guard means a single administrative transaction with no balance pre-check causes irreversible fund loss for all current depositors. The MarinateV2 report demonstrates this exact class of mistake occurs in practice.

---

### Recommendation

Before overwriting the connector account ID, assert that the old connector holds zero outstanding NEP-141 balance. Concretely, issue a cross-contract view call to the old connector's `ft_total_supply` (or equivalent) and require it returns zero before committing the new account ID. Alternatively, enforce a two-phase migration: pause withdrawals, drain the old connector to the new one, then update the account ID.

---

### Proof of Concept

1. Users deposit ETH into Aurora; the connector mints NEP-141 tokens and Aurora records EVM balances via `set_balance`.
2. Owner calls `set_eth_connector_contract_account` with a freshly deployed, unfunded connector account. No balance check occurs; the new account ID is written immediately. [5](#0-4) 
3. Any user calls `ft_withdraw` (or any exit precompile). `return_promise` resolves the target to the new connector via `get_connector_account_id`. [6](#0-5) 
4. The new connector has no NEP-141 tokens; the cross-contract transfer fails. The user's EVM balance remains in Aurora's storage with no withdrawal path.
5. The old connector still holds the NEP-141 tokens but is no longer reachable through any Aurora contract interface, making the funds permanently unrecoverable through normal means.

pub fn set_eth_connector_contract_account<I: IO + Copy, E: Env>(
    io: I,
    env: &E,
) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        let is_private = env.assert_private_call();

        if is_private.is_err() {
            require_owner_only(&state, &env.predecessor_account_id())?;
        }

        let args: SetEthConnectorContractAccountArgs = io.read_input_borsh()?;

        set_connector_account_id(io, &args.account);
        set_connector_withdraw_serialization_type(io, &args.withdraw_serialize_type);

        Ok(())
    })
}

fn get_connector_account_id<I: IO>(io: &I) -> Result<AccountId, ContractError> {
    io.read_storage(&construct_contract_key(
        EthConnectorStorageId::EthConnectorAccount,
    ))
    .ok_or(errors::ERR_CONNECTOR_STORAGE_KEY_NOT_FOUND)
    .and_then(|x| {
        x.to_value()
            .map_err(|_| errors::ERR_BORSH_DESERIALIZE.as_bytes())
    })
    .map_err(Into::into)
}
