### Title
Permanent Fund Freeze When `ExitToNear` `ft_transfer` Promise Fails Without Error-Refund Callback - (File: engine-precompiles/src/native.rs)

---

### Summary

The `ExitToNear` precompile burns ERC-20 tokens on Aurora **before** the NEAR-side `ft_transfer` promise is confirmed. When the `error_refund` compile-time feature is absent, no callback is registered to handle promise failures. If the `ft_transfer` promise fails for any reason, the burned ERC-20 tokens are permanently unrecoverable — a direct connector/bridge accounting bug with permanent fund-freeze impact.

---

### Finding Description

**Deposit path (NEAR → Aurora)** — `ft_on_transfer` in `engine/src/contract_methods/connector.rs`:

When NEP-141 tokens arrive at Aurora, `receive_erc20_tokens` mints the corresponding ERC-20 balance. If the EVM call fails, the function returns the full `args.amount` to the NEP-141 contract's `ft_resolve_transfer`, which refunds the sender. The deposit path is safe and symmetric. [1](#0-0) 

**Exit path (Aurora → NEAR)** — `ExitToNear::run()` in `engine-precompiles/src/native.rs`:

The ERC-20 contract calls the precompile as part of its `burn` function. At this point the tokens are already destroyed. The precompile then constructs a NEAR promise to call `ft_transfer` on the NEP-141 contract. A callback (`exit_to_near_precompile_callback`) is only attached when `callback_args != ExitToNearPrecompileCallbackArgs::default()`: [2](#0-1) 

The `refund` field of `callback_args` is populated **only** when the `error_refund` feature is compiled in: [3](#0-2) 

For a regular ERC-20 exit (non-wNEAR) without `error_refund`:
- `callback_args.refund = None`
- `callback_args.transfer_near = None`
- `callback_args == ExitToNearPrecompileCallbackArgs::default()` → **true**
- Result: `PromiseArgs::Create(transfer_promise)` — **no callback at all**

If the `ft_transfer` promise fails, there is no `exit_to_near_precompile_callback` invocation, no call to `engine::refund_on_error`, and no re-minting of the burned ERC-20 tokens. [4](#0-3) 

The refund logic itself (re-minting via `ERC20_MINT_SELECTOR`) is correct when reached, but it is **never reached** without the feature: [5](#0-4) 

The test suite explicitly documents this asymmetry: [6](#0-5) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

When `ft_transfer` fails after `ExitToNear` is called without the `error_refund` feature:
- ERC-20 tokens are permanently burned on Aurora.
- NEP-141 tokens remain locked in the Aurora engine account on NEAR (never transferred to the recipient).
- The bridge accounting is permanently inconsistent: the NEP-141 supply held by Aurora exceeds the ERC-20 supply on Aurora.
- There is no admin recovery path — the tokens cannot be re-minted without a contract upgrade.

---

### Likelihood Explanation

**Medium.** The `error_refund` feature is an opt-in compile-time flag. If the production WASM binary is built without it (which the code structure implies is the default, given `#[cfg(not(feature = "error_refund"))]` is the fallback branch), any of the following user-reachable conditions triggers permanent loss:

1. Recipient NEAR account is not registered with the NEP-141 contract (most common — registration is a separate step in NEP-141).
2. The NEP-141 contract has its `ft_transfer` method paused.
3. Insufficient gas attached to the `ft_transfer` promise (the constant `FT_TRANSFER_GAS = 10 TGas` may be insufficient for some NEP-141 implementations). [7](#0-6) 

A user calling `ExitToNear` with an unregistered recipient — a realistic mistake — permanently loses their tokens.

---

### Recommendation

1. **Make `error_refund` unconditional**: Remove the `#[cfg(feature = "error_refund")]` gate and always populate `callback_args.refund` for every `ExitToNear` call. The refund address can be derived from `context.caller` (the ERC-20 contract address for ERC-20 exits, or the transaction sender for ETH exits).

2. **Alternatively, always attach a callback**: Even when `callback_args == default()`, attach the `exit_to_near_precompile_callback` so that a failed `ft_transfer` is always detected and handled.

3. **Validate before burning**: Before burning ERC-20 tokens, verify (via a view call or storage check) that the recipient is registered on the NEP-141 contract, analogous to the report's recommendation to validate deposit amounts against the current quota before accepting them.

---

### Proof of Concept

1. User holds 1000 units of ERC-20 token `T` on Aurora (bridged from NEP-141 `t.near`).
2. User calls the ERC-20 contract's `withdraw` function, which burns 1000 tokens and calls the `ExitToNear` precompile with recipient `unregistered.near`.
3. The precompile constructs `PromiseArgs::Create(ft_transfer(..., "unregistered.near", 1000))` — **no callback**.
4. NEAR executes the promise; `ft_transfer` panics because `unregistered.near` has no storage deposit on `t.near`.
5. The promise fails. No callback fires. `engine::refund_on_error` is never called.
6. Result: 1000 ERC-20 tokens are gone from Aurora; 1000 NEP-141 tokens remain locked in the Aurora engine account forever. [8](#0-7) [9](#0-8)
