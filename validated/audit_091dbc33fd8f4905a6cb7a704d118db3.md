### Title
Unauthorized Overwrite of Any Account's E2EE Encryption Key Enables Attacker-Controlled Decryption — (`x/e2ee/keeper/keeper.go`)

### Summary

`MsgRegisterEncryptionKey` accepts an arbitrary `Address` field and writes the supplied age public key to that address's slot in the KV store. Neither `ValidateBasic` nor the keeper's `RegisterEncryptionKey` handler verifies that the transaction signer matches `req.Address`. Any unprivileged account can therefore overwrite any other account's registered encryption key with an attacker-controlled age public key, causing all subsequent senders who query `Query/Key` for the victim to encrypt messages to the attacker's key.

---

### Finding Description

`RegisterEncryptionKey` in the keeper writes directly to the store using the address supplied in the message: [1](#0-0) 

The only validation performed before this write is in `ValidateBasic`: [2](#0-1) 

`ValidateBasic` checks only that `Address` is a valid bech32 string and that `Key` is a parseable age X25519 recipient key. It does **not** check that the transaction signer equals `req.Address`. The keeper handler likewise performs no such check. There is no `GetSigners()` method generated in `tx.pb.go` that would bind the required signer to the `Address` field, and no ante-handler or middleware in the module enforces this invariant. [3](#0-2) 

The `Query/Key` and `Query/Keys` handlers return whatever is stored for the requested address without any attestation that the stored key was registered by the address owner: [4](#0-3) 

---

### Impact Explanation

An attacker who controls any funded account can:

1. Generate an age keypair locally.
2. Submit `MsgRegisterEncryptionKey{Address: victim, Key: attacker_age_pubkey}` signed by the attacker's own key.
3. The keeper overwrites the victim's KV entry with the attacker's public key.
4. Any sender who calls `Query/Key(victim)` receives the attacker's public key and encrypts their message to it.
5. The attacker decrypts every such message with their age private key; the victim never receives the plaintext.

This directly corrupts e2ee key/message state with concrete security impact (confidentiality break for all messages directed at the victim after the overwrite), matching the allowed High impact category: *"Corruption of … e2ee key/message state with direct security impact."*

---

### Likelihood Explanation

The attack requires only a valid funded account and a single on-chain transaction. No special privileges, leaked keys, or external assumptions are needed. The attacker can target any address that has (or will) register an encryption key.

---

### Recommendation

In `RegisterEncryptionKey`, extract the signer from the SDK context and assert it equals `req.Address` before writing to the store:

```go
func (k Keeper) RegisterEncryptionKey(
    ctx context.Context,
    req *types.MsgRegisterEncryptionKey,
) (*types.MsgRegisterEncryptionKeyResponse, error) {
    signerAddr := sdk.UnwrapSDKContext(ctx).EventManager()... // extract signer
    if signerAddr.String() != req.Address {
        return nil, sdkerrors.ErrUnauthorized.Wrap("signer must match address")
    }
    ...
}
```

Alternatively, add the `cosmos.msg.v1.signer` proto annotation on the `address` field so the SDK auto-generates and enforces `GetSigners()`.

---

### Proof of Concept

```
# Step 1 – attacker generates age keypair
age-keygen -o attacker.key
ATTACKER_PUBKEY=$(age-keygen -y attacker.key)

# Step 2 – attacker overwrites victim's key (signed by attacker, not victim)
cronosd tx e2ee register-encryption-key \
    --address <victim_bech32> \
    --key "$ATTACKER_PUBKEY" \
    --from attacker --chain-id testchain -y

# Step 3 – sender queries victim's key and encrypts
RETURNED_KEY=$(cronosd query e2ee key <victim_bech32> --output json | jq -r .key)
echo "secret message" | age -r "$RETURNED_KEY" > ciphertext.age

# Step 4 – attacker decrypts (victim cannot)
age --decrypt -i attacker.key ciphertext.age
# Output: secret message
```

The returned key in Step 3 is the attacker's pubkey; the victim's age private key cannot decrypt the ciphertext.
