### Title
Division Before Multiplication in `delta_gas` Computation Causes Users to Underpay for Native Resource Consumption — (`File: basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`)

---

### Summary

In `compute_gas_refund`, the `delta_gas` adjustment that charges users for native resource consumption uses a pre-truncated intermediate value (`native_per_gas`) as the divisor. Because `native_per_gas` is computed via `ceil(gas_price / native_price)` before being stored as a `u64`, the subsequent division `native_used / native_per_gas` compounds the rounding error. The correct formula `(native_used * native_price) / gas_price` is never computed, causing users to be systematically undercharged for native resource usage and receive a larger gas refund than they are entitled to.

---

### Finding Description

**Step 1 — `native_per_gas` is computed with integer division (rounding up):**

In `validation_impl.rs` (L2 transactions): [1](#0-0) 

In `process_l1_transaction.rs` (L1 transactions): [2](#0-1) 

Both compute `native_per_gas = ceil(gas_price / native_price)` and store it as a `u64`. This truncates the true rational ratio `gas_price / native_price` to the nearest integer ceiling.

**Step 2 — `delta_gas` divides by the already-truncated `native_per_gas`:**

In `refund_calculation.rs`: [3](#0-2) 

The expression `native_used / native_per_gas` is the division-before-multiplication bug. The full compound expression is:

```
delta_gas = native_used / ceil(gas_price / native_price)
```

The mathematically correct expression is:

```
delta_gas = (native_used * native_price) / gas_price
```

Because `ceil(gas_price / native_price) >= gas_price / native_price`, the denominator is inflated, making `delta_gas` smaller than the true value.

---

### Impact Explanation

`delta_gas` is the extra gas charged to a user when native resource consumption exceeds what EVM gas alone accounts for. It is added to `gas_used`, which directly determines the fee paid: `fee = gas_used * gas_price`. [4](#0-3) 

When `delta_gas` is underestimated, `gas_used` is smaller than it should be, so the user pays less than the native resources they consumed cost the protocol. The operator/protocol absorbs the shortfall.

**Worst-case quantification:**

Let `gas_price = native_price + 1` (e.g., `gas_price = 11`, `native_price = 10`):
- `native_per_gas = ceil(11 / 10) = 2`
- True ratio: `11 / 10 = 1.1`
- Computed `delta_gas = native_used / 2`
- True `delta_gas = native_used * 10 / 11 ≈ 0.909 * native_used`
- Underestimation: `≈ 45%` of the native-driven gas surcharge is lost

This is a **resource accounting bug** causing underpayment of fees proportional to native resource consumption.

---

### Likelihood Explanation

This is triggered by any transaction where:
1. `gas_price` is not an exact multiple of `native_price` (extremely common in practice), AND
2. `delta_gas > 0`, i.e., native resource consumption is the bottleneck (common for compute-heavy or pubdata-heavy transactions on ZKsync OS)

No privileged access is required. Any unprivileged transaction sender can trigger this path simply by submitting a transaction. The condition `gas_price % native_price != 0` is the normal case.

---

### Recommendation

Pass `native_price` (as `U256`) and `gas_price` (as `U256`) into `compute_gas_refund` instead of the pre-divided `native_per_gas`, and compute `delta_gas` directly:

```rust
// In refund_calculation.rs
// Before (division before multiplication):
let delta_gas = (native_used / native_per_gas) as i64 - (gas_used as i64);

// After (multiplication before division):
let delta_gas = if gas_price.is_zero() {
    0i64
} else {
    // native_used * native_price / gas_price, then subtract gas_used
    let numerator = (native_used as u128) * native_price_u128;
    let true_delta = (numerator / gas_price_u128) as i64;
    true_delta - (gas_used as i64)
};
```

The function signature of `compute_gas_refund` should be updated to accept `native_price: U256` and `gas_price: U256` instead of the pre-computed `native_per_gas: u64`. [5](#0-4)

### Proof of Concept

**Setup:** `native_price = 10`, `gas_price = 11`, `gas_limit = 100_000`, `native_used = 100_000`

**Current behavior:**
```
native_per_gas = ceil(11 / 10) = 2
delta_gas      = 100_000 / 2 = 50_000
gas_used       += 50_000
fee_paid       = (gas_used_base + 50_000) * 11
```

**Correct behavior:**
```
delta_gas_true = (100_000 * 10) / 11 = 90_909
gas_used       += 90_909
fee_paid       = (gas_used_base + 90_909) * 11
```

**Shortfall per transaction:** `(90_909 - 50_000) * 11 = 450_000` wei (≈ 45% of the native-driven surcharge is not collected). Any user submitting a transaction where `gas_price` is just above a multiple of `native_price` benefits from this truncation at the protocol's expense.
