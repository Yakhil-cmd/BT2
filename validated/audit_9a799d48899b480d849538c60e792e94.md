### Title
Unvalidated Pool Response Trusted for `targetPuzzleHash` in Plot NFT Join/Change Pool Flow (TOCTOU) - (File: `packages/gui/src/components/plotNFT/select/PlotNFTSelectPool.tsx`)

### Summary
The Chia GUI pool-joining flow makes two separate, uncoordinated HTTP calls to the pool's own endpoint: one to display pool details to the user (check), and a second at submit time to extract `targetPuzzleHash` and `relativeLockHeight` for the on-chain transaction (use). A malicious pool operator can serve different responses to these two calls, causing the user to approve a transaction that redirects farming-reward payouts to an attacker-controlled puzzle hash — while the UI showed a legitimate-looking one.

### Finding Description

**Check (display) path:**
`PlotNFTSelectBase` calls `usePoolInfo(poolUrl)` which internally calls `getPoolInfo` via `window.appAPI.fetchPoolInfo`. The result is rendered in `<PoolInfo poolInfo={poolInfo.poolInfo} />`, showing the user the pool's `targetPuzzleHash` and `relativeLockHeight`. [1](#0-0) [2](#0-1) 

**Use (submit) path:**
When the user clicks "Create" or "Change", `prepareSubmitData` makes a completely independent second call to `getPoolInfo(normalizedPoolUrl)` and blindly uses the returned `targetPuzzleHash` and `relativeLockHeight` to build the `pw_join_pool` RPC call — with no cross-check against what was displayed. [3](#0-2) [4](#0-3) 

The only validation performed on the pool response is:
- `targetPuzzleHash` must be truthy (non-empty)
- `relativeLockHeight` must not be `undefined`

No format validation, no range check on `relativeLockHeight`, and no cross-check between the displayed value and the submitted value. [5](#0-4) 

The `targetPuzzleHash` is then forwarded verbatim to `pw_join_pool`: [6](#0-5) [7](#0-6) 

### Impact Explanation

`targetPuzzleHash` is the on-chain puzzle hash to which all pooled farming rewards are paid. By returning a legitimate-looking hash in the display call and an attacker-controlled hash in the submit call, a malicious pool operator causes the user to sign and broadcast a `pw_join_pool` transaction that permanently redirects all future farming rewards to the attacker's address. The user sees no indication that the submitted hash differs from the displayed one. This is a **payout change affecting pooled farming rewards** — explicitly within the Critical impact scope.

Additionally, `relativeLockHeight` is accepted without an upper-bound check. A malicious pool could return an arbitrarily large value (e.g., `2^32 - 1`), locking the user's Plot NFT in the pool for an impractical number of blocks, effectively preventing them from ever leaving. [8](#0-7) 

### Likelihood Explanation

The attacker must operate a pool server reachable via HTTPS (enforced on mainnet by `usePoolInfo`). The server needs only to serve two different JSON responses to the same endpoint — a trivial implementation. No cryptographic material, no host compromise, and no dependency compromise is required. The user must enter the attacker's pool URL, which is a realistic scenario (shared pool lists, QR codes, phishing links to a convincing pool domain). Once the user clicks "Create" or "Change", the attack executes silently. [9](#0-8) 

### Recommendation

1. **Eliminate the TOCTOU**: Cache the pool info fetched for display and reuse the same object in `prepareSubmitData` rather than making a second network call. Pass the already-fetched `poolInfo` down from `PlotNFTSelectBase` to the submit handler.
2. **Validate `relativeLockHeight`**: Enforce an upper bound (the Chia pool protocol specifies a maximum of 4608 blocks) before accepting the value.
3. **Validate `targetPuzzleHash` format**: Confirm it is a valid 32-byte hex string before use.

### Proof of Concept

1. Attacker operates `https://evil-pool.example` with a `/pool_info` endpoint.
2. On the **first** request (display), the endpoint returns a plausible `targetPuzzleHash` (e.g., copied from a legitimate pool).
3. On the **second** request (submit), the endpoint returns `targetPuzzleHash` set to the attacker's puzzle hash.
4. User navigates to "Join a Pool" or "Change Pool", enters `https://evil-pool.example`.
5. `PlotNFTSelectBase` renders the legitimate-looking puzzle hash via `usePoolInfo` → `PoolInfo`.
6. User clicks "Create"/"Change". `prepareSubmitData` fires a new `getPoolInfo` call, receives the attacker's puzzle hash, and passes it to `pw_join_pool`.
7. The on-chain Plot NFT singleton is updated with the attacker's `targetPuzzleHash`. All subsequent farming rewards flow to the attacker. [10](#0-9) [11](#0-10)

async function prepareSubmitData(data: FormData): SubmitData {
  const { self, fee, poolUrl } = data;
  const initialTargetState = {
    state: self ? 'SELF_POOLING' : 'FARMING_TO_POOL',
  };

  if (!self && poolUrl) {
    const normalizedPoolUrl = normalizeUrl(poolUrl);
    const { targetPuzzleHash, relativeLockHeight } = await getPoolInfo(normalizedPoolUrl);
    if (!targetPuzzleHash) {
      throw new Error(t`Pool does not provide targetPuzzleHash.`);
    }
    if (relativeLockHeight === undefined) {
      throw new Error(t`Pool does not provide relativeLockHeight.`);
    }

    initialTargetState.poolUrl = normalizedPoolUrl;
    initialTargetState.targetPuzzleHash = targetPuzzleHash;
    initialTargetState.relativeLockHeight = relativeLockHeight;
  }

  const feeMojos = chiaToMojo(fee || '0');

  return {
    fee: feeMojos,
    initialTargetState,
  };
}
