### Title
Pool Admin Bypasses Spread-Fee Cap via Uncapped Per-Bin Additional Fees in `setPoolBinAdditionalFees()` — (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`setPoolBinAdditionalFees()` passes `addFeeBuyE6` / `addFeeSellE6` directly to the pool with no cap validation, while every other admin fee path is bounded by `maxAdminSpreadFeeE6`. A pool admin can set per-bin additional fees up to `uint16` max (65 535 in E6 units ≈ 6.55 %), stacking on top of the already-capped base spread fee and silently exceeding the protocol's hard fee ceiling for any bin they choose.

---

### Finding Description

The factory enforces `maxAdminSpreadFeeE6` (≤ `HARD_MAX_SPREAD_FEE_E6 = 200 000`, i.e. 20 %) on the base spread fee via `setPoolAdminFees`:

```solidity
// MetricOmmPoolFactory.sol:414-415
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (newAdminNotionalFeeE8 > maxAdminNotionalFeeE8) revert AdminFeeTooHigh();
```

But the parallel per-bin path has no such guard:

```solidity
// MetricOmmPoolFactory.sol:450-457
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

The pool's `setBinAdditionalFees` also performs no cap check — only a bin-index range check:

```solidity
// MetricOmmPool.sol:464-474
function setBinAdditionalFees(int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external onlyFactory nonReentrant(PoolActions.SET_BIN_ADDITIONAL_FEES)
{
    if (bin < LOWEST_BIN || bin > HIGHEST_BIN) revert InvalidBinIndex(bin);
    BinState storage s = _binStates[bin];
    s.addFeeBuyE6 = addFeeBuyE6;
    s.addFeeSellE6 = addFeeSellE6;
    emit BinAdditionalFeesUpdated(bin, addFeeBuyE6, addFeeSellE6);
}
```

The swap loop then adds the per-bin fee directly on top of the base fee:

```solidity
// MetricOmmPool.sol:910, 999, 1088
params.baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6)
params.baseFeeX64 + Math.mulDiv(binState.addFeeSellE6, ONE_X64, 1e6)
```

`baseFeeX64` is derived from `spreadFeeE6` (capped at 20 %). `addFeeBuyE6` / `addFeeSellE6` are `uint16` (max 65 535 ≈ 6.55 % in E6 units) and are added with no ceiling. The effective per-bin fee therefore reaches up to **26.55 %**, silently exceeding `HARD_MAX_SPREAD_FEE_E6`.

---

### Impact Explanation

Traders who swap through a bin where the pool admin has set `addFeeBuyE6 = 65535` or `addFeeSellE6 = 65535` pay a fee of up to 26.55 % instead of the documented hard maximum of 20 %. The excess fee accrues as surplus inside the pool and is later collected by the admin via `collectPoolFees`, constituting a direct loss of trader principal beyond the protocol's stated cap. Because the per-bin fee is invisible to traders who only inspect the pool's `spreadFeeE6` immutable, the overcharge is silent.

---

### Likelihood Explanation

The pool admin is a semi-trusted role that is explicitly scoped to operate "only inside caps and timelocks." The `setPoolBinAdditionalFees` path is a normal, post-deployment admin operation with no timelock. Any pool admin who wishes to extract additional value from traders on a specific bin can do so immediately and unilaterally. The trigger requires no special conditions beyond holding the `poolAdmin` role.

---

### Recommendation

Add a cap check in `setPoolBinAdditionalFees` before forwarding to the pool, mirroring the guard already present in `setPoolAdminFees`:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (addFeeBuyE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

Alternatively, enforce the cap inside `setBinAdditionalFees` on the pool, or introduce a dedicated `maxAdminBinAdditionalFeeE6` cap that the factory owner can configure.

---

### Proof of Concept

1. Factory is deployed with `HARD_MAX_SPREAD_FEE_E6 = 200_000` (20 %) and `maxAdminSpreadFeeE6 = 200_000`.
2. Pool admin calls `setPoolAdminFees(pool, 200_000, 0)` — accepted, base spread fee is at the hard cap.
3. Pool admin calls `setPoolBinAdditionalFees(pool, 0, 65535, 65535)` — **no revert**; bin 0 now carries an additional 6.55 % fee.
4. A trader swaps through bin 0. The swap loop computes:
   - `currBinBuyFeeX64 = baseFeeX64 + mulDiv(65535, ONE_X64, 1e6)`
   - Effective fee ≈ 20 % + 6.55 % = **26.55 %**, exceeding `HARD_MAX_SPREAD_FEE_E6`.
5. The excess fee accrues as pool surplus and is swept to the admin fee destination on the next `collectPoolFees` call.
