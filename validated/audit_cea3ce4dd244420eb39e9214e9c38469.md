### Title
Pool Admin Bypasses `maxAdminSpreadFeeE6` Cap via Uncapped `setPoolBinAdditionalFees` — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary
`MetricOmmPoolFactory.setPoolBinAdditionalFees` allows the pool admin to set per-bin `addFeeBuyE6`/`addFeeSellE6` to any `uint16` value (up to 65535) without checking against `maxAdminSpreadFeeE6`. This bypasses the fee-cap invariant that bounds all other admin fee-setting paths.

### Finding Description
The factory enforces `maxAdminSpreadFeeE6` on `setPoolAdminFees` via `revert AdminFeeTooHigh()`. No equivalent check exists in `setPoolBinAdditionalFees`. The pool's `setBinAdditionalFees` only validates the bin index. The per-bin fee is added to `baseFeeX64` in all four swap paths (`buyToken0InBinSpecifiedIn/Out`, `buyToken1InBinSpecifiedIn/Out`), directly widening the effective spread paid by traders in that bin.

### Impact Explanation
A pool admin can set `addFeeBuyE6 = 65535` on any bin, imposing a 6.5535% additional spread on traders buying token0 in that bin, regardless of the factory's configured `maxAdminSpreadFeeE6`. The extra spread is captured as LP fees. If the admin is also an LP (or controls LP positions), they extract value from traders beyond their authorized cap. Even without LP overlap, traders suffer bad-price execution — they pay materially more than the protocol's fee-cap model permits. This is a direct admin-boundary break.

### Likelihood Explanation
The pool admin is a semi-trusted role that can call `setPoolBinAdditionalFees` at any time with no timelock. The path requires only `onlyPoolAdmin(pool)`. Any pool admin who wishes to extract excess value or harm traders can do so immediately.

### Recommendation
Add a cap check in `MetricOmmPoolFactory.setPoolBinAdditionalFees`:

```solidity
if (addFeeBuyE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
```

Alternatively, enforce the cap inside `MetricOmmPool.setBinAdditionalFees` by reading the factory's `maxAdminSpreadFeeE6` via a factory call, though the factory-side check is simpler and consistent with the existing pattern.

### Proof of Concept
1. Deploy factory with `maxAdminSpreadFeeE6 = 1000` (0.1%).
2. Create a pool with `adminSpreadFeeE6 = 0`.
3. As pool admin, call `factory.setPoolBinAdditionalFees(pool, 0, 65535, 0)`.
4. Observe `_binStates[0].addFeeBuyE6 == 65535` — 6.5535%, far above the 0.1% cap.
5. Execute a swap buying token0 in bin 0; the effective fee applied is `baseFeeX64 + mulDiv(65535, ONE_X64, 1e6)`, confirming the cap is bypassed.
