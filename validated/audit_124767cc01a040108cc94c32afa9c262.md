The `handle_capacity_overflow` logic at lines 1039–1042 has a concrete boolean error. Let me trace it precisely.

### Title
Gap-Creating Transaction with Existing Non-Gap Pool Txs Bypasses `creating_gap` Guard, Evicting Legitimate Accounts' Transactions — (`crates/apollo_mempool/src/mempool.rs`)

---

### Summary

The boolean formula for `creating_gap` in `handle_capacity_overflow` is incorrect. When an account already has transactions in the pool but no recorded gap, and an attacker submits a new transaction with `tx_nonce > account_nonce`, the formula evaluates `creating_gap = false`. This causes `try_make_space` to be called, which evicts transactions belonging to other gap-accounts (e.g., address B) to make room for the attacker's gap-creating transaction. The invariant — that gap-creating transactions must not trigger eviction of other accounts' transactions — is violated.

---

### Finding Description

The guard is at: [1](#0-0) 

```rust
let account_has_gap = self.accounts_with_gap.contains(&address);
let account_has_txs = self.tx_pool.contains_account(address);
let closing_gap = tx.nonce() == account_nonce;
let creating_gap = (account_has_gap || !account_has_txs) && !closing_gap;
```

The formula `(account_has_gap || !account_has_txs) && !closing_gap` is intended to be `true` when the incoming transaction would create a nonce gap. However, it misses the case where:

- `account_has_gap = false` — the account has no existing gap
- `account_has_txs = true` — the account already has transactions in the pool
- `tx.nonce() > account_nonce` — the incoming tx creates a new gap

Substituting: `(false || !true) && !false` = `(false || false) && true` = `false`.

So `creating_gap = false` even though the incoming transaction unambiguously creates a nonce gap (`tx_nonce > account_nonce`).

The guard at line 1047 then proceeds to call `try_make_space`: [2](#0-1) 

`try_make_space` evicts transactions exclusively from `accounts_with_gap` (via `get_evictable_account`): [3](#0-2) 

So legitimate gap-account transactions (e.g., address B's) are permanently removed from the pool to make room for the attacker's gap-creating transaction.

After admission, `update_accounts_with_gap` is called (line 511), which correctly adds address A to `accounts_with_gap` — but the damage is already done: B's transactions have been evicted. [4](#0-3) 

The correct formula is simply:

```rust
let creating_gap = !closing_gap;
// i.e., tx.nonce() > account_nonce
// (tx.nonce() < account_nonce is already rejected by validate_incoming_tx as NonceTooOld)
```

---

### Impact Explanation

An unprivileged attacker who controls address A (with at least one non-gap transaction already in the pool) can submit a high-nonce transaction for A to force eviction of any gap-account's transactions. This is a targeted, repeatable denial-of-service against specific accounts: the attacker identifies a victim gap-account B, fills the pool to capacity with B's transactions, then submits a gap-creating tx for A to evict B. The attacker's own tx is then admitted and A is added to `accounts_with_gap`, making A itself evictable — but B's transactions are already gone.

Impact category: **High — Mempool admission evicts valid transactions and admits a gap-creating transaction that should have been rejected or self-evicted.**

---

### Likelihood Explanation

The attacker needs only:
1. One existing non-gap transaction for address A in the pool (trivially achievable).
2. The pool to be at or near capacity (achievable by flooding with gap-account transactions from a throwaway address B, or waiting for organic fullness).
3. Submit a single gap-creating transaction for A.

No privileged access, no special permissions. The attack is fully unprivileged and deterministic.

---

### Recommendation

Replace the `creating_gap` formula with a direct nonce comparison. Since `validate_incoming_tx` already rejects `tx_nonce < account_nonce`, at this point `tx_nonce >= account_nonce` is guaranteed:

```rust
// Before (incorrect):
let creating_gap = (account_has_gap || !account_has_txs) && !closing_gap;

// After (correct):
let creating_gap = !closing_gap; // equivalent to tx.nonce() > account_nonce
```

This ensures any transaction that would introduce a nonce gap — regardless of whether the account already has pool transactions — is treated as gap-creating and denied the ability to evict other accounts' transactions.

    fn handle_capacity_overflow(
        &mut self,
        tx: &InternalRpcTransaction,
        account_nonce: Nonce,
        freed_bytes: u64,
    ) -> Result<(), MempoolError> {
        let address = tx.contract_address();

        let account_has_gap = self.accounts_with_gap.contains(&address);
        let account_has_txs = self.tx_pool.contains_account(address);
        let closing_gap = tx.nonce() == account_nonce;
        let creating_gap = (account_has_gap || !account_has_txs) && !closing_gap;

        // Only the net growth must be evicted: an accompanying replacement removal frees
        // `freed_bytes` (0 when there is no replacement).
        let required_space = tx.total_bytes().saturating_sub(freed_bytes);
        if !creating_gap && self.try_make_space(required_space) {
            return Ok(());
        }

        Err(MempoolError::MempoolFull)
    }
