### Title
Staker Can Raise Commission to 100% via `set_commission_commitment`, Stealing All Delegator Rewards - (File: src/staking/staking.cairo)

### Summary
A staker with a delegation pool can call `set_commission_commitment` with `max_commission = COMMISSION_DENOMINATOR` (10000 = 100%), then immediately call `set_commission(10000)` to raise their commission to 100%. This routes all pool rewards to the staker as commission, leaving delegators with zero yield. Delegators cannot exit instantly — they must wait through the `DEFAULT_EXIT_WAIT_WINDOW` (one week), during which all accruing rewards are stolen.

### Finding Description
`COMMISSION_DENOMINATOR` is defined as `10000` (representing 100%). [1](#0-0) 

`set_commission_commitment` places only two constraints on `max_commission`: it must be `<= COMMISSION_DENOMINATOR` and `>= current_commission`. There is no protocol-level cap below 100%. [2](#0-1) 

Once a commitment is active, `update_commission` (called inside `set_commission`) only checks `commission <= commitment.max_commission` and `commission != old_commission`. The normal "commission can only decrease" guard is bypassed entirely. [3](#0-2) 

With `max_commission = 10000`, the staker can immediately call `set_commission(10000)`. The reward split in `split_rewards_with_commission` then computes `commission_rewards = pool_rewards_including_commission * 10000 / 10000 = pool_rewards_including_commission`, leaving `pool_rewards = 0`. [4](#0-3) 

This zero `pool_rewards` value is forwarded to the pool contract, so every delegator's `claim_rewards` returns 0 for the affected epochs. [5](#0-4) 

### Impact Explanation
All unclaimed and future yield belonging to pool members (delegators) is redirected to the staker as commission. This matches the allowed impact: **Theft of unclaimed yield**. Delegators cannot recover rewards already accrued under the 100% commission rate, and they cannot exit immediately — the two-step exit requires waiting through `DEFAULT_EXIT_WAIT_WINDOW` (one week). [6](#0-5) 

### Likelihood Explanation
Any registered staker who has a pool can execute this in two transactions with no special privileges beyond owning the staker address. The commitment can be set to expire in as little as one epoch (`expiration_epoch = current_epoch + 1`), so the attack window opens immediately. Delegators have no on-chain mechanism to detect and exit before the commission change takes effect within the same epoch.

### Recommendation
Introduce a protocol-enforced maximum commission cap (e.g., 5000 = 50%) stored as a governance-controlled parameter, and enforce it in both `set_commission` and `set_commission_commitment`:

```cairo
assert!(commission <= max_protocol_commission, Error::COMMISSION_OUT_OF_RANGE);
assert!(max_commission <= max_protocol_commission, Error::COMMISSION_OUT_OF_RANGE);
```

Alternatively, require that any commission increase via a commitment takes effect only after a mandatory delay (e.g., one epoch), giving delegators time to exit before the new rate applies.

### Proof of Concept

1. Staker registers with `commission = 500` (5%) and opens a delegation pool. Delegators deposit funds.
2. Staker calls:
   ```
   set_commission_commitment(max_commission: 10000, expiration_epoch: current_epoch + 1)
   ```
   This passes because `current_commission (500) <= max_commission (10000)` and `10000 <= COMMISSION_DENOMINATOR (10000)`. [7](#0-6) [8](#0-7) 

3. In the same block, staker calls:
   ```
   set_commission(commission: 10000)
   ```
   Inside `update_commission`, the active-commitment branch fires: `10000 <= 10000` ✓ and `10000 != 500` ✓. Commission is written as 10000. [9](#0-8) 

4. From this point, `split_rewards_with_commission` computes:
   - `commission_rewards = pool_rewards * 10000 / 10000 = pool_rewards` (all rewards)
   - `pool_rewards = 0` [10](#0-9) 

5. Delegators call `claim_rewards` and receive 0. They must wait one week (`DEFAULT_EXIT_WAIT_WINDOW`) to exit, during which all further rewards are also stolen.

        fn set_commission_commitment(
            ref self: ContractState, max_commission: Commission, expiration_epoch: Epoch,
        ) {
            self.general_prerequisites();
            assert!(max_commission <= COMMISSION_DENOMINATOR, "{}", Error::COMMISSION_OUT_OF_RANGE);
            let staker_address = get_caller_address();
            let staker_info = self.internal_staker_info(:staker_address);
            assert!(staker_info.unstake_time.is_none(), "{}", Error::UNSTAKE_IN_PROGRESS);

            let current_epoch = self.get_current_epoch();
            if let Option::Some(commission_commitment) = staker_pool_info
                .commission_commitment
                .read() {
                assert!(
                    !self.is_commission_commitment_active(:commission_commitment),
                    "{}",
                    Error::COMMISSION_COMMITMENT_EXISTS,
                );
            }
            // Staker must have a commission since it has a pool.
            let current_commission = staker_pool_info.commission();

            assert!(expiration_epoch > current_epoch, "{}", Error::EXPIRATION_EPOCH_TOO_EARLY);

                "{}",
                Error::EXPIRATION_EPOCH_TOO_FAR,
            );
            let commission_commitment = CommissionCommitment { max_commission, expiration_epoch };

pub(crate) fn split_rewards_with_commission(
    rewards_including_commission: Amount, commission: Commission,
) -> (Amount, Amount) {
    let commission_rewards = compute_commission_amount_rounded_down(
        :rewards_including_commission, :commission,
    );
    let pool_rewards = rewards_including_commission - commission_rewards;
    (commission_rewards, pool_rewards)
}

pub(crate) fn compute_commission_amount_rounded_down(
    rewards_including_commission: Amount, commission: Commission,
) -> Amount {
    mul_wide_and_div(
        lhs: rewards_including_commission,
        rhs: commission.into(),
        div: COMMISSION_DENOMINATOR.into(),
    )
        .expect_with_err(err: InternalError::COMMISSION_ISNT_AMOUNT_TYPE)
}
