### Title
Unrestricted Direct Call to `ft_on_transfer` Allows Minting ERC-20 Tokens Without NEP-141 Backing - (File: engine/src/contract_methods/connector.rs)

### Summary

`ft_on_transfer` in the Aurora Engine is a NEP-141 callback that is supposed to be invoked only by a legitimate token contract as part of the `ft_transfer_call` flow. However, it has no caller restriction and can be called directly by any NEAR account. Combined with the fact that `deploy_erc20_token` is also unrestricted, an attacker can register their own NEAR account as a NEP-141 token and then call `ft_on_transfer` directly to mint arbitrary amounts of ERC-20 tokens without any actual NEP-141 token transfer occurring.

### Finding Description

`ft_on_transfer` is exposed as a public NEAR contract entrypoint with no `assert_private_call()` guard and no check that the caller is a legitimate NEP-141 token contract:

### Impact Explanation

An attacker can mint unbacked ERC-20 tokens on Aurora:

1. Attacker calls `deploy_erc20_token` with `nep141 = attacker.near` (their own NEAR account ID). This registers the mapping `attacker.near → erc20_address` in Aurora's storage.
2. Attacker calls `ft_on_transfer` directly on the Aurora Engine contract with `sender_id = attacker.near`, `amount = <arbitrary large value>`, `msg = <attacker EVM address>`.
3. Since `predecessor_account_id (attacker.near) ≠ connector_account_id`, the engine routes to `receive_erc20_tokens(attacker.near, ...)`.
4. `get_erc20_from_nep141(attacker.near)` resolves to the ERC-20 deployed in step 1.
5. The engine mints `amount` ERC-20 tokens to the attacker's EVM address with no actual NEP-141 transfer having occurred.

The ERC-20 token is now insolvent: more tokens exist than are backed by real NEP-141 deposits. If the attacker sells these tokens on a DEX or uses them in DeFi, legitimate buyers suffer direct fund loss when they attempt to exit back to NEAR and find no NEP-141 reserves.

**Impact class**: Insolvency / ERC-20 mirror accounting bug. Matches the allowed scope: direct theft of user funds and permanent insolvency of the bridged token.

### Likelihood Explanation

The attack requires no special privileges. Both `deploy_erc20_token` and `ft_on_transfer` are public NEAR entrypoints callable by any NEAR account. The two-step attack is straightforward and requires only standard NEAR transaction submission. There is no economic barrier beyond NEAR gas fees.

### Recommendation

Add a caller restriction to `ft_on_transfer` to ensure it can only be invoked as a legitimate NEP-141 callback. The standard approach in NEAR is to verify the call originates from a known token contract. At minimum, add a check that the `predecessor_account_id` is a registered NEP-141 token in Aurora's mapping before proceeding with `receive_erc20_tokens`. Alternatively, use `assert_private_call()` if the function is only ever meant to be called as a cross-contract callback, or maintain an explicit allowlist of valid NEP-141 token callers.

### Proof of Concept

```
# Step 1: Register attacker's NEAR account as a NEP-141 token in Aurora
near call aurora.near deploy_erc20_token \
  '{"nep141": "attacker.near"}' \
  --accountId attacker.near

# Step 2: Call ft_on_transfer directly, minting 1,000,000 ERC-20 tokens
# with no actual NEP-141 transfer
near call aurora.near ft_on_transfer \
  '{"sender_id": "attacker.near", "amount": "1000000", "msg": "<attacker_evm_address>"}' \
  --accountId attacker.near
```

After step 2, `attacker_evm_address` holds 1,000,000 ERC-20 tokens on Aurora with zero NEP-141 backing. The attacker can sell these on any Aurora DEX. When legitimate buyers attempt to exit via `exit_to_near`, the NEP-141 reserves are insufficient, causing permanent loss of funds for those users.
