### Title
Missing Ownership Check on `GlobalContractDeployMode::AccountId` Allows Any Contract to Overwrite Another Account's Global Contract Namespace Entry ŌĆö (`runtime/runtime/src/global_contracts.rs`)

### Summary
When a contract emits a `DeployGlobalContract` receipt with `GlobalContractDeployMode::AccountId` targeting an arbitrary `receiver_id`, the runtime stores the attacker-supplied code under `TrieKey::GlobalContractCode { identifier: AccountId(receiver_id) }` without verifying that the receipt's `predecessor_id` equals `receiver_id`. Any account that subsequently calls `UseGlobalContract(AccountId(victim))` will execute the attacker's code.

### Finding Description
The `GlobalContractDeployMode::AccountId` mode is documented as "deployed under the **owner** account id ŌĆ” allows the **owner** to update the contract for all its users." [6](#0-5) 

However, the runtime enforces no such ownership. The execution path is:

1. Attacker contract A calls `promise_batch_action_deploy_global_contract_by_account_id` with a promise whose `receiver_id` is victim account B. [7](#0-6) 

2. The receipt arrives at account B. `action_deploy_global_contract` is invoked with `account_id = "account_b"`. The only check performed is a balance check on account B. [3](#0-2) 

3. `initiate_distribution` constructs `GlobalContractIdentifier::AccountId(account_id.clone())` ŌĆö using the receiver's ID, not the predecessor's ŌĆö and emits a `GlobalContractDistributionReceipt` carrying the attacker's code. [2](#0-1) 

4. `apply_distribution_current_shard` writes `TrieKey::GlobalContractCode { identifier: AccountId("account_b") }` with the attacker's code on every shard. [5](#0-4) 

5. `validate_deploy_global_contract_action` ŌĆö the only validation gate ŌĆö checks only code size. [1](#0-0) 

There is no `predecessor_id == receiver_id` guard anywhere in the `DeployGlobalContract` path, unlike `DeployContract`, `Stake`, `AddKey`, `DeleteKey`, and `DeleteAccount` which all require self-send.

### Impact Explanation
- **Global contract namespace corruption**: `TrieKey::GlobalContractCode { identifier: AccountId("account_b") }` is permanently overwritten with attacker-controlled bytecode.
- **Arbitrary code execution on third parties**: Any account C that calls `UseGlobalContract(AccountId("account_b"))` will have its contract pointer set to the attacker's code and will execute it on every subsequent function call.
- **Balance drain on victim**: Account B's balance is charged the storage cost for the attacker's code.
- The corrupted entry propagates to all shards via the distribution receipt mechanism. [8](#0-7) 

### Likelihood Explanation
Exploitable by any account that can deploy a contract. The attacker only needs to pay gas for the cross-account promise; storage cost is deducted from the victim's balance. No special privileges are required.

### Recommendation
Add a guard in `action_deploy_global_contract` (or in `validate_actions`/`validate_deploy_global_contract_action`) that, when `deploy_mode == GlobalContractDeployMode::AccountId`, requires `predecessor_id == receiver_id` (i.e., the action must be a self-send). This mirrors the existing restriction on `DeployContract`. [9](#0-8) 

### Proof of Concept
1. Deploy contract A (attacker) with code that, on any function call, does:
   ```rust
   let promise = env::promise_batch_create("account_b");
   env::promise_batch_action_deploy_global_contract_by_account_id(
       promise, MALICIOUS_WASM_CODE
   );
   ```
2. Call any function on contract A. This emits a `DeployGlobalContract(AccountId)` receipt targeting `account_b`.
3. The receipt executes on `account_b`. `TrieKey::GlobalContractCode { identifier: AccountId("account_b") }` is written with the malicious WASM.
4. Account C submits `UseGlobalContract(AccountId("account_b"))`. Its contract pointer is now set to the attacker's code.
5. Any subsequent function call on account C executes the attacker's WASM.

The existing test infrastructure in `test-loop-tests/src/tests/global_contracts.rs` can be extended to confirm this: deploy a global contract from account A targeting account B's ID, then have account C use it and assert it executes A's code. [10](#0-9)
