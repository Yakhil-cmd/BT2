### Title
Malicious Participant Can Silently Corrupt CKD Output Without Detection — (File: `src/confidential_key_derivation/protocol.rs`)

### Summary
The CKD coordinator aggregates each participant's contribution `(norm_big_y, norm_big_c)` with no cryptographic proof of correct computation. A single malicious participant can substitute arbitrary group elements, causing the coordinator to silently output a corrupted derived key that is indistinguishable from a valid one. Unlike every other multi-party protocol in this codebase (DKG, triple generation), the CKD aggregation step performs zero verification of received shares.

### Finding Description
In `do_ckd_coordinator` (`src/confidential_key_derivation/protocol.rs`, lines 50–55), the coordinator unconditionally sums every received `CKDOutput`:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
``` [1](#0-0) 

The correct contribution from participant `P_i` is:
- `norm_big_y_i = λ_i · y_i · G`
- `norm_big_c_i = λ_i · (x_i · H(pk ‖ app_id) + y_i · app_pk)`

where `x_i` is the private share and `y_i` is a fresh random scalar chosen in `compute_signature_share`. [2](#0-1) 

No proof of knowledge is required or verified. The coordinator has no mechanism to check that the received group elements were honestly computed. This is a sharp contrast with:

- **DKG** (`src/dkg.rs`): verifies Schnorr proofs of knowledge and commitment hashes for every participant before accepting their polynomial commitment. [3](#0-2) 

- **Triple generation** (`src/ecdsa/ot_based_ecdsa/triples/generation.rs`): verifies `dlog` proofs for `E_i(0)` and `F_i(0)`, commitment opening checks, and `dlogeq` proofs for the cross-term `C_i`. [4](#0-3) 

The CKD protocol has none of these safeguards. A malicious participant sends arbitrary `(P, Q) ∈ G₁ × G₁` in place of their correct contribution. The coordinator adds these values without any check and returns the corrupted aggregate as the protocol output.

### Impact Explanation
The coordinator outputs `(Y_corrupt, C_corrupt)` where:

```
Y_corrupt = Σ λ_i · y_i · G  +  P_attacker
C_corrupt = Σ λ_i · (x_i · H(pk‖app_id) + y_i · app_pk)  +  Q_attacker
```

When the application unmasks the output via `ckd_output.unmask(app_sk)`:

```
C_corrupt − app_sk · Y_corrupt
  = msk · H(pk‖app_id) + Q_attacker − app_sk · P_attacker
  ≠ msk · H(pk‖app_id)
```

The derived confidential key is silently wrong. The coordinator has no way to detect the corruption — the protocol returns `Ok(Some(ckd_output))` regardless. This matches the allowed **High** impact: *Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.*

### Likelihood Explanation
Any single participant in a CKD session can execute this attack. No special privileges are required beyond being a legitimate protocol participant. The attacker only needs to send a crafted `CKDOutput` message (two arbitrary `G₁` points) instead of their honest contribution. The attack is trivially reproducible and requires no cryptographic capability.

### Recommendation
Add a zero-knowledge proof of correct computation for each participant's contribution, consistent with the approach used in DKG and triple generation. Each participant should prove knowledge of `y_i` such that `norm_big_y = λ_i · y_i · G` (a standard dlog proof) and that `norm_big_c` is correctly formed relative to the public key and `app_pk` (a dlogeq-style proof). The coordinator must verify these proofs before accepting any contribution.

### Proof of Concept
1. Malicious participant `P_m` is part of a CKD session with honest coordinator `C` and `n−1` honest participants.
2. `P_m` computes the correct `(norm_big_y_m, norm_big_c_m)` but instead sends `(G₁_generator, G₁_generator)` to `C` via the private channel.
3. `C` executes `do_ckd_coordinator`, receives the forged values, and adds them without verification.
4. `C` returns `CKDOutput::new(Y + G₁_generator, C_agg + G₁_generator)`.
5. The application calls `ckd_output.unmask(app_sk)` and obtains `msk · H(pk‖app_id) + G₁_generator − app_sk · G₁_generator`, which is not the intended confidential key.
6. The coordinator has no indication that the output is wrong; the protocol completed successfully from its perspective.

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))

/// Verifies the proof of knowledge of the secret coefficients used to generate the
/// public secret sharing commitment.
fn internal_verify_proof_of_knowledge<C: Ciphersuite>(
    session_id: &HashOutput,
    domain_separator: &mut DomainSeparator,
    participant: Participant,
    commitment: &VerifiableSecretSharingCommitment<C>,
    proof_of_knowledge: &Signature<C>,
) -> Result<(), ProtocolError> {
    // creates an identifier for the participant
    let id = participant.scalar::<C>();
    let vk_share = commitment
        .coefficients()
        .first()
        .ok_or_else(|| ProtocolError::AssertionFailed("Empty coefficient list".to_string()))?;

    let big_r = proof_of_knowledge.R();
    let z = proof_of_knowledge.z();
    let c = challenge::<C>(domain_separator, session_id, id, vk_share, big_r)?;
    if *big_r != <C::Group>::generator() * *z - vk_share.value() * c.to_scalar() {
        return Err(ProtocolError::InvalidProofOfKnowledge(participant));
    }
    Ok(())
}
