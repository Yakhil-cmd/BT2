### Title
Unprivileged Caller Can Permanently Suppress Per-Block Reward Distribution via `update_rewards(disable_rewards=true)` - (File: src/staking/staking.cairo)

### Summary
The `update_rewards` function in the Staking contract is publicly callable with no access control beyond the general pause check. Any caller can invoke it with `disable_rewards=true`, which writes `last_reward_block` to the current block number **before** the `disable_rewards` guard is evaluated. Because `last_reward_block` is a global, single-slot value, this permanently consumes the reward slot for that block: no subsequent caller can distribute rewards for it. Repeated across every block, this constitutes a persistent griefing attack that permanently destroys all staker and delegator yield.

---

### Finding Description

`update_rewards` is exposed through the `IStakingRewardsManager` ABI with no role check:

```cairo
#[abi(embed_v0)]
impl StakingRewardsManagerImpl of IStakingRewardsManager<ContractState> {
    fn update_rewards(
        ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
    ) {
        self.general_prerequisites();          // only checks pause flag
        let current_block_number = starknet::get_block_number();
        assert!(
            current_block_number > self.last_reward_block.read(),
            "{}",
            Error::REWARDS_ALREADY_UPDATED,
        );
        ...
        // ← last_reward_block is committed HERE, before the guard below
        self.last_reward_block.write(current_block_number);

        if disable_rewards || self.is_pre_consensus() {
            return;                            // ← exits without distributing rewards
        }
        ...
    }
}
``` [1](#0-0) 

The ordering is the root cause:

1. `last_reward_block` is written to `current_block_number`.
2. The function then checks `disable_rewards` and returns early if `true`.

Because `last_reward_block` is already committed, the `REWARDS_ALREADY_UPDATED` assertion will reject every subsequent call to `update_rewards` for the same block number. The reward slot for that block is permanently consumed with zero distribution.

The `last_reward_block` field is a single global slot shared across all stakers:

```cairo
/// Last block number for which rewards were distributed.
last_reward_block: BlockNumber,
``` [2](#0-1) 

A single call with `disable_rewards=true` therefore silences rewards for **all** stakers and delegators in that block, not just the `staker_address` argument.

The contrast with `update_rewards_from_attestation_contract` is instructive — that function correctly gates itself:

```cairo
fn update_rewards_from_attestation_contract(...) {
    self.general_prerequisites();
    assert!(self.is_pre_consensus(), ...);
    self.assert_caller_is_attestation_contract();   // ← access control present
``` [3](#0-2) 

`update_rewards` has no equivalent guard.

---

### Impact Explanation

An attacker who front-runs the legitimate `update_rewards` call in every block with `disable_rewards=true` causes:

- **All per-block rewards** for every staker and delegator to be permanently destroyed (not deferred — the reward calculation is per-block and there is no catch-up mechanism).
- **Staker `unclaimed_rewards_own`** and **pool `cumulative_rewards_trace`** are never incremented, so `claim_rewards` and `pool.claim_rewards` return zero indefinitely.

This maps to the allowed impact: **Griefing with no profit motive but damage to users or protocol (Medium)**, and at sustained execution escalates to **Permanent freezing of unclaimed yield (High)**.

---

### Likelihood Explanation

- The function is part of the public ABI; no special role or key is required.
- The attacker only needs to submit a transaction per block with a valid (active) `staker_address` and `disable_rewards=true`.
- On Starknet, where block times are short and gas costs are low, this is economically feasible for a motivated griever.
- The attacker gains nothing financially, but the damage to stakers and delegators is total if sustained.

---

### Recommendation

1. **Add access control**: Restrict `update_rewards` to a designated caller (e.g., the attestation contract or a governance-controlled rewards manager), mirroring the pattern in `update_rewards_from_attestation_contract`.

2. **Alternatively, reorder the write**: Move `self.last_reward_block.write(current_block_number)` to **after** the `disable_rewards` branch, so that a call with `disable_rewards=true` does not consume the block's reward slot:

```cairo
if disable_rewards || self.is_pre_consensus() {
    return;
}
self.last_reward_block.write(current_block_number);  // only commit if distributing
// ... distribute rewards
```

---

### Proof of Concept

1. Consensus rewards are active (`is_pre_consensus()` returns `false`).
2. Attacker observes block N is about to be processed.
3. Attacker submits `update_rewards(any_active_staker, disable_rewards=true)` in block N, front-running the legitimate call.
4. `last_reward_block` is set to N; function returns without distributing rewards.
5. Any legitimate call to `update_rewards` in block N now panics with `REWARDS_ALREADY_UPDATED`.
6. All stakers and delegators receive zero rewards for block N.
7. Repeating steps 2–6 every block permanently zeroes all yield.
