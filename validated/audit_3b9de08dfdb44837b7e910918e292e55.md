### Title
Missing Access Control on `update_rewards` Allows Any Caller to Permanently Freeze All Consensus Rewards — (`src/staking/staking.cairo`)

---

### Summary

`IStakingRewardsManager::update_rewards` is specified as "Only starkware sequencer" but the implementation enforces **no caller check**. Any EOA can call it with `disable_rewards=true` once per block, consuming the global `last_reward_block` slot without distributing rewards, and permanently denying all stakers their consensus-mode yield.

---

### Finding Description

The spec explicitly states the access control for `update_rewards`:

> **access control**: Only starkware sequencer. [1](#0-0) 

The implementation, however, only calls `general_prerequisites()`: [2](#0-1) 

`general_prerequisites()` performs exactly two checks — unpaused and non-zero caller — with **no role or identity check**: [3](#0-2) 

The function then writes `last_reward_block` to the current block **unconditionally**, before the `disable_rewards` branch: [4](#0-3) 

`last_reward_block` is a **global singleton** (not per-staker). Once written, any subsequent call in the same block reverts with `REWARDS_ALREADY_UPDATED`: [5](#0-4) 

The interface definition confirms there is no access-control annotation on the trait itself: [6](#0-5) 

---

### Impact Explanation

An attacker calls `update_rewards(any_valid_active_staker, disable_rewards=true)` on every new block. Each call:

1. Passes all precondition checks (staker exists, is active, has non-zero balance).
2. Writes `last_reward_block = current_block` — consuming the one allowed slot per block.
3. Returns early at the `disable_rewards` branch — **zero rewards distributed**.
4. Blocks every legitimate sequencer call for that block with `REWARDS_ALREADY_UPDATED`.

Because `last_reward_block` is global, **all stakers across the entire protocol** lose rewards for every block the attacker frontruns. Sustained over time this constitutes **permanent freezing of all consensus-mode unclaimed yield** for all stakers and delegators.

Impact category: **High — Permanent freezing of unclaimed yield**.

---

### Likelihood Explanation

- Requires no privilege, no stake, no special setup — any EOA suffices.
- One transaction per block; on Starknet this is cheap and automatable.
- The attacker has no profit motive but causes maximum damage to all stakers.
- The only liveness requirement is that at least one valid, active staker with non-zero balance exists (trivially true in any live deployment).

---

### Recommendation

Add a sequencer-only guard at the top of `update_rewards`, consistent with the spec. For example, assert that `get_caller_address() == sequencer_address` where `sequencer_address` is a stored, governance-controlled value, or reuse whatever role-check mechanism is used elsewhere in the protocol for sequencer-gated calls.

---

### Proof of Concept

```
// Preconditions:
//   - consensus rewards active (is_pre_consensus() == false)
//   - staker_victim is a valid, active staker with non-zero balance
//   - attacker is any EOA

loop every new block:
    staking_contract.update_rewards(
        staker_address: staker_victim,
        disable_rewards: true      // <-- no rewards distributed
    )
    // last_reward_block is now == current_block
    // Any legitimate sequencer call this block reverts: REWARDS_ALREADY_UPDATED
    // All stakers earn 0 rewards for this block

// After N blocks:
// assert staker_victim.unclaimed_rewards_own == 0  (despite being active and attested)
// assert all other stakers' unclaimed_rewards_own == 0
```

The existing test `update_rewards_disable_rewards_consensus_rewards_flow_test` already demonstrates that calling `update_rewards` with `disable_rewards=true` in consensus mode produces zero rewards and blocks any same-block retry — the only missing piece is that the test calls it from a trusted context, while the production code imposes no such restriction.
