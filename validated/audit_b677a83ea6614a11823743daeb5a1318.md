### Title
Gateway Admits Unsigned Invoke Transactions via `skip_stateful_validations` Bypass — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator unconditionally skips the `__validate__` entry-point call (the account's signature-verification step) for any invoke transaction whose nonce is `1` and whose sender address appears in the mempool or a recent committed block. Because the mempool check is not restricted to deploy-account transactions submitted by the same sender, any unprivileged attacker can inject an invoke transaction with an arbitrary (or empty) signature for any account that has a pending deploy-account in the mempool, and the gateway will admit it without verifying the signature.

### Finding Description

**Root cause — `skip_stateful_validations`** [1](#0-0) 

The function returns `true` (skip `__validate__`) when all three conditions hold:

1. The transaction is an `Invoke` with `nonce == 1`.
2. The on-chain account nonce is `0` (account not yet deployed).
3. `account_tx_in_pool_or_recent_block(sender_address)` returns `true`.

The third condition is satisfied whenever **any** transaction for that address exists in the pool or was seen in a recent committed block: [2](#0-1) 

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` is called with `skip_validate = true`: [3](#0-2) 

Inside `run_validate_entry_point`, the `ExecutionFlags` are built with `validate: !skip_validate`, so `validate = false`: [4](#0-3) 

This is passed to `StatefulValidator::perform_validations`, which short-circuits before calling `__validate__`: [5](#0-4) 

The `validate_tx` implementation confirms: when `execution_flags.validate == false`, the entry point is never called: [6](#0-5) 

**Exploit path (no privileges required)**

1. Alice submits a `deploy_account` transaction for address `A`. It enters the mempool.
2. Bob observes Alice's pending deploy-account (the mempool is observable via the public RPC).
3. Bob crafts an `Invoke V3` transaction: `sender_address = A`, `nonce = 1`, `signature = []` (empty or garbage).
4. Bob submits it to the gateway's `add_tx` endpoint.
5. The gateway checks: nonce=1 ✓, on-chain nonce=0 ✓, `account_tx_in_pool_or_recent_block(A)` = `true` ✓ → skips `__validate__`.
6. Bob's unsigned invoke passes all remaining checks (nonce range, resource bounds) and is admitted to the mempool.

The `validate_by_mempool` call that precedes `skip_stateful_validations` only checks nonce validity and duplicate detection — it does not verify the signature: [7](#0-6) 

**What happens at execution time**

When the batcher executes the block, `__validate__` is called with `validate = true`. The invalid signature causes the transaction to fail. In Starknet, a `__validate__` failure means the transaction is not included in the block and no fee is charged. The nonce increment is rolled back via the transactional state. There is no permanent state corruption.

### Impact Explanation

The gateway's admission invariant — "every transaction in the mempool has passed signature verification" — is broken for invoke transactions with nonce=1 targeting any account with a pending deploy-account. An attacker can flood the mempool with unsigned invoke transactions for any such account, consuming mempool capacity and forcing the batcher to attempt (and fail) execution of invalid transactions. This matches the defined High impact: **"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."**

Financial impact is zero because `__validate__` failure at execution time prevents inclusion and fee charging. The harm is mempool pollution and batcher resource waste.

### Likelihood Explanation

- The mempool is publicly observable via the RPC, so identifying accounts with pending deploy-accounts is trivial.
- Crafting an invoke transaction with nonce=1 and an empty signature requires no privileged access.
- The attack is repeatable at low cost (no fee is charged for rejected transactions).
- The only limiting factor is the mempool's capacity eviction policy.

### Recommendation

Restrict the validation skip to invoke transactions that were submitted **together with** (i.e., in the same API call as) a deploy-account transaction for the same address, rather than relying on a mempool membership check that any third party can satisfy. Alternatively, require that the invoke transaction carry a syntactically valid signature (even if it cannot be verified against a not-yet-deployed account class) before admitting it to the mempool, and defer the cryptographic check to block execution.

### Proof of Concept

```
# 1. Alice submits a deploy_account for address A (valid, signed).
POST /add_transaction  { type: "DEPLOY_ACCOUNT", sender_address: A, nonce: 0, signature: [valid] }

# 2. Bob submits an unsigned invoke for address A with nonce=1.
POST /add_transaction  { type: "INVOKE", sender_address: A, nonce: 1, signature: [] }

# Gateway evaluation:
#   skip_stateful_validations:
#     tx.nonce() == 1                              → true
#     account_nonce == 0                           → true
#     account_tx_in_pool_or_recent_block(A)        → true  (Alice's deploy_account is in pool)
#   → skip_validate = true  →  __validate__ NOT called
#   → Bob's unsigned invoke is admitted to the mempool.

# 3. Batcher builds a block:
#   - Executes Alice's deploy_account (succeeds, account A is deployed).
#   - Executes Bob's invoke: __validate__ called → fails (empty signature).
#   - Transaction dropped; no fee charged; no state change.
```

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
