### Title
EOA Signature Validation Disabled in Production Forward Run Causes Forward/Proving Divergence - (`forward_system/src/run/mod.rs`, `basic_bootloader/src/bootloader/config.rs`)

---

### Summary

The production sequencer forward execution path (`run_block`) uses `BasicBootloaderForwardSimulationConfig`, which sets `VALIDATE_EOA_SIGNATURE = false`. This causes the bootloader to skip ECDSA signature verification for all EOA transactions during forward (sequencer) execution. The proving run uses `BasicBootloaderProvingExecutionConfig` with `VALIDATE_EOA_SIGNATURE = true`, enforcing full signature checks. An attacker can submit a transaction with a forged `from` address and any invalid signature; the sequencer accepts and executes it (mutating local state), while the prover rejects it — producing an unprovable block and halting the chain.

---

### Finding Description

`BasicBootloaderForwardSimulationConfig` is defined in `basic_bootloader/src/bootloader/config.rs` with:

```rust
impl BasicBootloaderExecutionConfig for BasicBootloaderForwardSimulationConfig {
    const VALIDATE_EOA_SIGNATURE: bool = false;
    const SIMULATION: bool = false;
}
``` [1](#0-0) 

This config is passed directly to the production `run_block` function — the sequencer's main block execution entry point — which is not gated by any `#[cfg(feature = "testing")]` guard:

```rust
run_forward::<BasicBootloaderForwardSimulationConfig>(oracle, &mut result_keeper, tracer, validator);
``` [2](#0-1) 

In both the Ethereum-STF and ZK-STF validation paths, the signature check guard evaluates as:

```rust
if !Config::VALIDATE_EOA_SIGNATURE | Config::SIMULATION {
    // skip: no ecrecover, no from-address check
} else {
    // actual signature validation
}
```

With `VALIDATE_EOA_SIGNATURE = false` and `SIMULATION = false`, the condition `!false | false = true`, so the entire ecrecover + `recovered_from != from` check is bypassed. [3](#0-2) [4](#0-3) 

Crucially, nonce validation is **not** skipped (it is only skipped when `SIMULATION = true`), so the attacker must supply the correct nonce — which is publicly readable from chain state:

```rust
if !Config::SIMULATION {
    // nonce check runs
    require!(old_nonce == originator_expected_nonce, err, system)?;
}
``` [5](#0-4) 

The proving run uses `BasicBootloaderProvingExecutionConfig` with `VALIDATE_EOA_SIGNATURE = true`, enforcing full signature validation:

```rust
impl BasicBootloaderExecutionConfig for BasicBootloaderProvingExecutionConfig {
    const SIMULATION: bool = false;
    const VALIDATE_EOA_SIGNATURE: bool = true;
}
``` [6](#0-5) 

The prover calls `run_prepared::<BasicBootloaderProvingExecutionConfig>`: [7](#0-6) 

This creates a structural forward/proving divergence: any transaction accepted by the sequencer with an invalid signature will be rejected by the prover.

---

### Impact Explanation

**Forward/proving divergence leading to chain halt and sequencer state corruption.**

1. The sequencer accepts and executes a transaction with a forged `from` address and invalid signature, mutating its local state (e.g., draining a victim's balance, deploying contracts on behalf of any address).
2. The prover, running with `VALIDATE_EOA_SIGNATURE = true`, rejects the same transaction with `InvalidTransaction::IncorrectFrom`.
3. The sequencer has committed to a state root that the prover cannot reproduce — the block is unprovable.
4. The chain cannot finalize on L1 until the sequencer is patched and the block is re-executed.
5. In the sequencer's soft-confirmed view, victim funds are already drained, causing user-visible loss even before L1 finality.

---

### Likelihood Explanation

**High.** The attacker entry point is a standard transaction submission (RPC `eth_sendRawTransaction`). No privileged access is required. The only constraint is knowing the victim's current nonce, which is publicly available via `eth_getTransactionCount`. The attacker constructs a transaction with `from = victim`, correct nonce, `to = attacker`, `value = victim_balance`, and any 65-byte byte string as the signature. The sequencer accepts it unconditionally.

---

### Recommendation

Replace `BasicBootloaderForwardSimulationConfig` (which has `VALIDATE_EOA_SIGNATURE = false`) with `BasicBootloaderForwardETHLikeConfig` (which has `VALIDATE_EOA_SIGNATURE = true`) in the production `run_block` function:

```rust
// forward_system/src/run/mod.rs, line 96
// Before:
run_forward::<BasicBootloaderForwardSimulationConfig>(...)
// After:
run_forward::<BasicBootloaderForwardETHLikeConfig>(...)
``` [8](#0-7) 

The `VALIDATE_EOA_SIGNATURE = false` optimization is only safe for `eth_call`/`eth_estimateGas` simulation paths (which already use `BasicBootloaderCallSimulationConfig` with `SIMULATION = true`). It must never be used for block production. The comment "It can be used to optimize forward run" in `config.rs` should be removed or replaced with an explicit warning that this config is unsafe for production use. [9](#0-8) 

---

### Proof of Concept

1. Observe victim address `V` with balance `B` and current nonce `N` (via `eth_getTransactionCount`).
2. Construct a raw EIP-1559 transaction:
   - `from = V`, `nonce = N`, `to = attacker_address`, `value = B`, `gas_limit = 21000`, valid gas price fields, `chain_id = <chain_id>`
   - Signature: any 65 bytes (e.g., all zeros: `v=0, r=0, s=0`)
3. Submit via `eth_sendRawTransaction`.
4. The sequencer's `run_block` executes with `BasicBootloaderForwardSimulationConfig`. The guard `!false | false = true` skips ecrecover. Nonce `N == N` passes. Balance transfer executes. Sequencer state shows `V`'s balance drained to attacker.
5. When the prover attempts to prove this block using `BasicBootloaderProvingExecutionConfig`, ecrecover runs, `recovered_from != V`, and the transaction is rejected with `IncorrectFrom`. The block is unprovable.
6. The sequencer has produced a state root it cannot prove, halting L1 finalization. [10](#0-9) [11](#0-10)
