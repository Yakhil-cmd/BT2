### Title
Unauthorized Encryption Key Overwrite via Missing Signer Authorization in `RegisterEncryptionKey` — (`x/e2ee/keeper/keeper.go`)

### Summary

`RegisterEncryptionKey` writes an arbitrary key to any address without verifying that the transaction signer owns that address. Any unprivileged account can overwrite another account's registered encryption key.

### Finding Description

The keeper's `RegisterEncryptionKey` handler accepts `req.Address` and `req.Key` and unconditionally writes to the KV store: [1](#0-0) 

There is no check that the transaction signer equals `req.Address`. The `registerEncryptionKey` helper also performs no such check: [2](#0-1) 

`ValidateBasic()` only validates bech32 format and key format — it does not enforce ownership: [3](#0-2) 

The generated `tx.pb.go` contains no `GetSigners()` method, and a grep across the entire `x/e2ee` package returns zero matches for `GetSigners` or `signer`. In Cosmos SDK v0.50+, signer enforcement is driven by the `option (cosmos.msg.v1.signer)` proto annotation; without it, no signer-to-address binding is enforced by the framework. [4](#0-3) 

**Critical unknown:** The proto source file `proto/e2ee/tx.proto` could not be read due to index limits. If it contains `option (cosmos.msg.v1.signer) = "address"`, the SDK's reflection-based signer extractor would enforce signer = address and block this attack. The absence of any `GetSigners()` in the generated pb.go is a strong indicator that this annotation is missing, but this must be confirmed by reading `proto/e2ee/tx.proto` directly.

### Impact Explanation

If the signer annotation is absent, any attacker can submit:

```
MsgRegisterEncryptionKey{Address: victim_address, Key: attacker_pubkey}
```

signed by their own key. The keeper writes `attacker_pubkey` under `victim_address`. All future senders encrypting to `victim_address` will use the attacker's key; the attacker can decrypt those messages and the victim cannot. This is corruption of e2ee key state with direct security impact (High).

### Likelihood Explanation

The attack requires only a valid transaction with a valid age X25519 public key. No special privileges, leaked keys, or external assumptions are needed. The mempool is public, so front-running a victim's pending registration is straightforward.

### Recommendation

1. **Confirm the proto annotation:** Read `proto/e2ee/tx.proto` and verify `option (cosmos.msg.v1.signer) = "address"` is present on `MsgRegisterEncryptionKey`.
2. **Add an explicit signer check in the keeper** regardless of proto annotation:
   ```go
   signerAddr := sdk.UnwrapSDKContext(ctx).EventManager()... // or extract from ctx
   if signerAddr != req.Address {
       return nil, sdkerrors.ErrUnauthorized
   }
   ```
3. **Add a `GetSigners()` method** to `MsgRegisterEncryptionKey` returning `[]sdk.AccAddress{sdk.MustAccAddressFromBech32(m.Address)}` as a defense-in-depth measure.

### Proof of Concept

```
1. Alice submits MsgRegisterEncryptionKey{Address: alice_addr, Key: alice_pubkey} (pending in mempool)
2. Attacker observes mempool, submits MsgRegisterEncryptionKey{Address: alice_addr, Key: attacker_pubkey} with higher gas
3. Attacker's tx executes first (or in same block, last-writer wins)
4. Query: Key(alice_addr) → attacker_pubkey
5. Bob encrypts a message to alice_addr using attacker_pubkey; attacker decrypts it; Alice cannot
```

**Verdict:** Likely **High** vulnerability pending confirmation that `proto/e2ee/tx.proto` lacks `option (cosmos.msg.v1.signer) = "address"`. The Go-layer evidence (no `GetSigners()`, no keeper signer check) strongly supports the finding. A Devin session should read `proto/e2ee/tx.proto` to confirm.

func (k Keeper) registerEncryptionKey(
	ctx context.Context,

	bz, err := k.addressCodec.StringToBytes(address)

		return err
	}
	sdk.UnwrapSDKContext(ctx).KVStore(k.storeKey).Set(types.KeyPrefix(bz), key)
	return nil
}
