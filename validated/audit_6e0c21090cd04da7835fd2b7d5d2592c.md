Audit Report

## Title
SNS Governance Voting Rewards Permanently Lost When All Voters Have Zero Voting Power During a Reward Round - (File: `rs/sns/governance/src/governance.rs`)

## Summary
In `distribute_rewards`, when `total_reward_shares == 0` (all ballots on `ReadyToSettle` proposals recorded zero voting power), the rewards purse is computed but never distributed and never rolled over. The proposals are still settled and ballots cleared, causing `rewards_rolled_over()` to return `false`, which silently drops the entire purse with no recovery path.

## Finding Description
The `distribute_rewards` function computes `rewards_purse_e8s` from token supply and reward rate. It then sums `total_reward_shares` from ballot `voting_power` fields across all `considered_proposals`. When `total_reward_shares == dec!(0)`, the maturity distribution loop is skipped entirely and `distributed_e8s_equivalent` remains `0`.

Despite this, the function unconditionally iterates over `considered_proposals`, sets `reward_event_end_timestamp_seconds` and `reward_event_round` on each proposal, and clears all ballots — permanently marking them as `Settled`. [1](#0-0) 

The final `RewardEvent` is written with `settled_proposals: considered_proposals` (non-empty) and `distributed_e8s_equivalent: 0`. [2](#0-1) 

In the next round, `e8s_equivalent_to_be_rolled_over()` calls `rewards_rolled_over()`, which returns `self.settled_proposals.is_empty()`. [3](#0-2) 

Because `settled_proposals` is non-empty, `rewards_rolled_over()` returns `false`, `e8s_equivalent_to_be_rolled_over()` returns `0`, and the entire `rewards_purse_e8s` is permanently discarded. The same structural flaw exists in NNS governance's `rewards_rolled_over()`. [4](#0-3) 

The existing test `zero_total_reward_shares` uses an `EmptyLedger` with `total_supply = 0`, so `rewards_purse_e8s` is also `0` in that test — it only verifies no divide-by-zero occurs, not that a non-zero purse is properly rolled over. [5](#0-4) 

## Impact Explanation
The entire `rewards_purse_e8s` for the affected round — computed as `supply × reward_rate × round_duration` plus any previously rolled-over amount — is permanently unrecoverable. It is never minted as maturity for any neuron and is not carried forward to the next round. For an SNS with a meaningful token supply and active reward rate, this constitutes a significant and irreversible loss of governance token maturity per affected round. This matches the allowed impact: **High ($2,000–$10,000) — Significant SNS security impact with concrete user or protocol harm**.

## Likelihood Explanation
A neuron's `voting_power` is zero when `cached_neuron_stake_e8s <= neuron_fees_e8s`. The `neuron_fees_e8s` field accumulates from rejected governance proposals. In a small SNS with few neurons, an unprivileged participant can engineer this condition by: (1) creating a neuron and submitting proposals that get rejected until `neuron_fees_e8s >= cached_neuron_stake_e8s`; (2) ensuring this depleted neuron is the only one that voted on a proposal that reaches `ReadyToSettle`; (3) triggering `distribute_rewards`, at which point `total_reward_shares == 0` with non-empty `considered_proposals`, and the purse is dropped. No special privileges are required; the attack is repeatable each reward round.

## Recommendation
When `total_reward_shares == 0` but `considered_proposals` is non-empty, the rewards purse must be rolled over rather than silently discarded. The most targeted fix is to extend `rewards_rolled_over()` (or `e8s_equivalent_to_be_rolled_over()`) to also return the purse when `distributed_e8s_equivalent == 0` and `total_available_e8s_equivalent > 0`, regardless of whether proposals were settled. Alternatively, skip settling proposals when `total_reward_shares == 0` so the rollover path is taken naturally — though this defers proposal settlement and may have other implications.

## Proof of Concept
1. Deploy an SNS with one neuron `N`: `cached_neuron_stake_e8s = 1_000_000`, `neuron_fees_e8s = 1_000_000` (voting power = 0). Token supply is non-zero (e.g., `100_000_000` e8s).
2. Create proposal `P` while `N` has zero voting power; `N`'s ballot records `voting_power: 0`. Proposal `P` is decided and enters `ReadyToSettle`.
3. Advance time past the reward round boundary and call `run_periodic_tasks()` / `distribute_rewards`.
4. Observe: `rewards_purse_e8s > 0`, `total_reward_shares == 0`, maturity distribution skipped, proposal settled, `RewardEvent` written with `settled_proposals = [P]` and `distributed_e8s_equivalent = 0`.
5. Advance to the next reward round and call `distribute_rewards` again.
6. Observe: `e8s_equivalent_to_be_rolled_over()` returns `0` (because `settled_proposals` was non-empty in the prior event), confirming the purse from step 4 is permanently lost.

This can be demonstrated by modifying the existing `zero_total_reward_shares` test to use a non-zero `total_supply` and asserting that `total_available_e8s_equivalent > 0` in the reward event while also asserting that the next round's rolled-over amount equals that value — the assertion will fail, proving the loss.
