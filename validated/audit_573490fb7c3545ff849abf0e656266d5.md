### Title
`skip_stateful_validations` Ignores `max_nonce_for_validation_skip` Config, Hardcodes `Nonce(Felt::ONE)` — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

`StatefulTransactionValidatorConfig` exposes a configurable field `max_nonce_for_validation_skip` that is documented as "Maximum nonce for which the validation is skipped." The Rust gateway's `skip_stateful_validations` free function never reads this field; it hardcodes `Nonce(Felt::ONE)` unconditionally. The legacy Python path (`PyValidator::should_run_stateful_validations`) correctly consults `self.max_nonce_for_validation_skip`. The discrepancy means the config knob is silently dead in the production Rust gateway, producing two distinct admission errors depending on how an operator tunes the parameter.

The threshold `Nonce(Felt::ONE)` is a literal constant. A `grep` across the entire `crates/apollo_gateway/` tree finds **zero** references to `max_nonce_for_validation_skip`, confirming the field is never consumed in the Rust gateway path.

### Impact Explanation

**Scenario A — operator sets `max_nonce_for_validation_skip = Nonce(Felt::ZERO)` to disable the UX skip entirely:**

The code still evaluates `tx.nonce() == Nonce(Felt::ONE)` as `true` for any invoke with nonce 1 from an undeployed account that has a `deploy_account` in the mempool. The gateway skips `__validate__` and admits the transaction, directly contradicting the operator's intent. An attacker who controls the account contract can craft a `__validate__` that would reject the invoke under normal circumstances (e.g., a multisig that requires co-signers), yet the transaction is admitted without that check running.

**Scenario B — operator sets `max_nonce_for_validation_skip > Nonce(Felt::ONE)` (e.g., `5`) to extend the UX window:**

Invoke transactions with nonces 2–5 from accounts with a pending `deploy_account` are not skipped by the hardcoded `== Nonce(Felt::ONE)` check. They proceed to `run_validate_entry_point` against a not-yet-deployed account, fail with a `ValidateFailure`, and are rejected — even though the config says they should be admitted. This is a gateway-level rejection of valid transactions before sequencing.

Both scenarios fall within the allowed impact scope:
> *High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.*

---

### Likelihood Explanation

- The default value of `max_nonce_for_validation_skip` is `Nonce(Felt::ONE)`, which coincidentally matches the hardcoded literal, so the bug is dormant under default configuration.
- The field is serialised in `config_schema.json` and the deployment app-config JSONs, making it a visible, operator-tunable knob.
- Any operator who reads the field description ("Maximum nonce for which the validation is skipped") and adjusts it — either to `0` to harden the gateway or to `>1` to widen the UX window — will silently get the wrong behaviour.
- No runtime error or warning is emitted; the misconfiguration is invisible.

---

### Recommendation

Pass `self.config.max_nonce_for_validation_skip` into `skip_stateful_validations` (or make it a method on `StatefulTransactionValidator`) and replace the hardcoded literal with a range check mirroring the Python path:

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    max_nonce_for_validation_skip: Nonce,   // <-- add parameter
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        let tx_nonce = tx.nonce();
        if tx_nonce >= Nonce(Felt::ONE)
            && tx_nonce <= max_nonce_for_validation_skip   // use config
            && account_nonce == Nonce(Felt::ZERO)
        {
            // existing mempool check …
        }
    }
    Ok(false)
}
```

Update the call-site in `run_pre_validation_checks` to supply `self.config.max_nonce_for_validation_skip`.

### Proof of Concept

**Trigger for Scenario A (gateway admits transaction that should be rejected):**

1. Operator deploys the Rust gateway with `max_nonce_for_validation_skip = Nonce(Felt::ZERO)` (intending to disable the skip feature).
2. Attacker deploys a Cairo account whose `__validate__` rejects all callers except a specific co-signer.
3. Attacker submits `deploy_account` (nonce 0) + `invoke` (nonce 1, no co-signer signature) to the gateway simultaneously.
4. `skip_stateful_validations` evaluates `tx.nonce() == Nonce(Felt::ONE)` → `true`; `account_nonce == Nonce(Felt::ZERO)` → `true`; `account_tx_in_pool_or_recent_block` → `true` (deploy_account is in mempool).
5. `skip_validate = true` is returned regardless of the config value.
6. `run_validate_entry_point` sets `execution_flags.validate = false`; `__validate__` is never called.
7. The invoke is admitted to the mempool and eventually executed without the account's signature check.
