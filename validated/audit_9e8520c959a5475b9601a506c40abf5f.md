### Title
Self-Referential Vault UUID in `addWithdrawal` Silently Drops FT Transfers from the Extended Index — (`File: module/state_synchronization/indexer/extended/transfers/ft_group.go`)

---

### Summary

`ftTxEventGroup.addWithdrawal` in `ft_group.go` registers a vault's `WithdrawnUUID` into `withdrawalByUUID` **before** it looks up `FromUUID` to find the parent vault. There is no guard that `FromUUID != WithdrawnUUID`. When a `FungibleToken.Withdrawn` event carries equal values for both fields, the vault resolves itself as its own parent, its tracked amount is subtracted from itself (zeroed), and `ResolvePairs` silently discards the transfer — producing no burn, mint, or paired record in the extended FT transfer index.

---

### Finding Description

`addWithdrawal` processes each `FungibleToken.Withdrawn` event in two sequential steps:

**Step 1 — register the new vault:**
```go
g.withdrawals = append(g.withdrawals, w)
g.withdrawalByUUID[decoded.WithdrawnUUID] = len(g.withdrawals) - 1
``` [1](#0-0) 

**Step 2 — look up the parent vault:**
```go
parentIdx, ok := g.withdrawalByUUID[w.decoded.FromUUID]
``` [2](#0-1) 

When `FromUUID == WithdrawnUUID`, Step 2 finds the entry that was just inserted in Step 1. The vault is treated as its own parent. The code then executes:

```go
parent.decoded.Amount -= w.decoded.Amount
``` [3](#0-2) 

Because `parent` and `w` point to the same struct, this is `w.Amount -= w.Amount`, zeroing the tracked amount. In `ResolvePairs`, any withdrawal with `Amount == 0` is explicitly skipped:

```go
if g.withdrawals[wIdx].decoded.Amount == 0 {
    // ignore fully consumed withdrawals
    continue
}
``` [4](#0-3) 

The transfer produces no `ftPairedResult` — it is permanently absent from the extended index. The existing duplicate-UUID guard does **not** protect against this case because it fires only when `WithdrawnUUID` is already present before the current event is inserted:

```go
if _, exists := g.withdrawalByUUID[decoded.WithdrawnUUID]; exists {
    return fmt.Errorf("duplicate withdrawal resource UUID ...")
}
``` [5](#0-4) 

The self-referential event passes this guard cleanly.

---

### Impact Explanation

The `FTParser.Parse` pipeline feeds `ProcessBlockData` in `FungibleTokenTransfers`, which writes the indexed transfer records used by the Access node's extended backend:

```go
ftEntries, err := a.ftParser.Parse(data.Events, data.Header.Height)
``` [6](#0-5) 

A transfer silently omitted from this index is **permanently missing** — there is no retry or reconciliation path. Wallets, DeFi applications, and audit tools that rely on the extended transfer index will show an incorrect history for the affected accounts. A malicious actor can exploit this to make a token transfer undetectable by any consumer of the extended index, while the on-chain state is updated normally.

---

### Likelihood Explanation

The `FTParser` processes all events whose type matches `A.{fungibleTokenAddress}.FungibleToken.Withdrawn`. In Cadence, vault implementations import the `FungibleToken` contract and emit its `Withdrawn` event, supplying the `fromUUID` and `withdrawnUUID` field values themselves. A malicious contract author can deploy a vault that emits this event with `fromUUID == withdrawnUUID`. The event type will be `A.{fungibleTokenAddress}.FungibleToken.Withdrawn` — exactly what the parser accepts — because the event is defined in the imported `FungibleToken` contract. No privileged access, staked node control, or key compromise is required; deploying a contract is an unprivileged operation on Flow.

---

### Recommendation

Add an explicit guard at the top of `addWithdrawal` rejecting any event where `FromUUID == WithdrawnUUID`:

```go
if decoded.FromUUID == decoded.WithdrawnUUID {
    return fmt.Errorf("self-referential withdrawal: FromUUID == WithdrawnUUID == %d in transaction %d",
        decoded.WithdrawnUUID, event.TransactionIndex)
}
```

This mirrors the recommendation in the original report: validate that the distinguishing field (reward amount / vault UUID pair) is non-degenerate before mutating shared state.

---
