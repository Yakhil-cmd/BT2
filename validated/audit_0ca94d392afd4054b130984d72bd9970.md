### Title
`calculateNFTRoyalties` hardcodes `nftSellerNetAmount = amount`, causing "Net Proceeds" to display the full sale amount instead of `amount - royaltyAmount`, misleading NFT sellers into accepting offers where 100% of proceeds drain to the attacker's royalty address — (`packages/gui/src/components/offers/utils.ts`)

### Summary

`calculateNFTRoyalties` in `utils.ts` unconditionally sets `nftSellerNetAmount = amount` (the correct `amount - royaltyAmount` subtraction is commented out). Every UI surface that displays "Net Proceeds", "You will receive", or "They will receive" for an NFT offer therefore shows the full pre-royalty amount. An attacker who mints an NFT with `royaltyPercentage=10000` (100%) and creates or facilitates a `TokenForNFT` offer can cause a victim NFT seller to accept the offer believing they will receive the full token amount, while on-chain the entire amount is routed to the attacker's royalty puzzle hash.

### Finding Description

In `packages/gui/src/components/offers/utils.ts`:

```typescript
// line 312
const royaltyAmount: number = royaltyPercentage ? (royaltyPercentage / 100) * amount : 0;
const royaltyAmountString: string = formatAmount(royaltyAmount);
const nftSellerNetAmount: number = amount;   // ← BUG: should be amount - royaltyAmount
// : parseFloat(
//     (amount - parseFloat(royaltyAmountString) - makerFee).toFixed(12),
//   );
```

With `royaltyPercentage=10000`, `convertRoyaltyToPercentage` returns `100`, so `royaltyAmount = amount`. The correct `nftSellerNetAmount` is `0`, but the function returns `amount`.

This propagates to two surfaces:

1. **`NFTOfferViewer.tsx` — offer acceptance screen (taker/victim path)**
   - `overrideNFTSellerAmount = chiaToMojo(nftSaleInfo?.nftSellerNetAmount)` = full amount → passed to `OfferSummaryTokenRow` → displayed as "You will receive: X XCH"
   - The explicit **"Net Proceeds"** `<h5>` label renders `nftSaleInfo?.nftSellerNetAmount` = full amount, with a tooltip that promises it is "asking price minus creator fees"

2. **`NFTOfferEditor.tsx` — offer creation screen (maker path)**
   - "You will receive" / "They will receive" renders `nftSellerNetAmount` = full amount
   - `showNegativeAmountWarning = (nftSellerNetAmount ?? 0) < 0` is never triggered because `nftSellerNetAmount` is always `amount` (positive)

### Impact Explanation

A victim NFT seller viewing a `TokenForNFT` offer for an NFT with 100% royalty sees:

| Label | Displayed | Actual |
|---|---|---|
| You will receive | X XCH | X XCH |
| Creator Fee (100%) | X XCH | X XCH |
| **Net Proceeds** | **X XCH** | **0 XCH** |
| Total Amount Offered | 2X XCH | 2X XCH |

The "Net Proceeds" label is the most prominent summary figure (rendered as `<h5 fontWeight="bold">`). It is explicitly described as "asking price minus creator fees" but shows the full amount. The victim accepts the offer, gives up the NFT, and receives 0 XCH. The attacker's royalty address receives the full X XCH.

### Likelihood Explanation

The attack requires the attacker to have previously minted and transferred an NFT with 100% royalty to the victim. The attacker then creates a `TokenForNFT` offer. The victim, seeing a prominent "Net Proceeds: X XCH" label, accepts. The royalty warning (`>= 20%`) is shown, but the "Net Proceeds" figure directly contradicts what a careful reading of the royalty section implies, creating genuine confusion. The `royaltyPercentage > 100` guard in the editor does not block this because `convertRoyaltyToPercentage(10000) = 100` exactly.

### Recommendation

Restore the commented-out subtraction in `calculateNFTRoyalties`:

```typescript
const nftSellerNetAmount: number = parseFloat(
  (amount - parseFloat(royaltyAmountString) - makerFee).toFixed(12),
);
```

And add a guard to prevent displaying or accepting offers where `nftSellerNetAmount <= 0`.
