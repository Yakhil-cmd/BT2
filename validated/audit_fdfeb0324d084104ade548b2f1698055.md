### Title
Pool Admin Bypasses `maxAdminSpreadFeeE6` Cap via Uncapped `setPoolBinAdditionalFees` — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary

`MetricOmmPoolFactory.setPoolBinAdditionalFees` passes `addFeeBuyE6` / `addFeeSellE6` directly to the pool with no cap check against `maxAdminSpreadFeeE6`, while the parallel function `setPoolAdminFees` does enforce that cap. A pool admin can set per-bin additional fees to any value up to `uint16.max` (65535 / 1e6 = 6.5535%), bypassing the factory-configured admin spread cap entirely.

### Finding Description

`setPoolAdminFees` correctly enforces the cap:

```solidity
// MetricOmmPoolFactory.sol:414-415
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (newAdminNotionalFeeE8 > maxAdminNotionalFeeE8) revert AdminFeeTooHigh();
``` [1](#0-0) 

`setPoolBinAdditionalFees` has no such check — it forwards the caller-supplied values directly:

```solidity
// MetricOmmPoolFactory.sol:450-457
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
``` [2](#0-1) 

`setBinAdditionalFees` on the pool only validates the bin index, not the fee magnitude:

```solidity
// MetricOmmPool.sol:464-474
function setBinAdditionalFees(int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external onlyFactory nonReentrant(PoolActions.SET_BIN_ADDITIONAL_FEES)
{
    if (bin < LOWEST_BIN || bin > HIGHEST_BIN) revert InvalidBinIndex(bin);
    BinState storage s = _binStates[bin];
    s.addFeeBuyE6 = addFeeBuyE6;
    s.addFeeSellE6 = addFeeSellE6;
    ...
}
``` [3](#0-2) 

The persisted `addFeeBuyE6` / `addFeeSellE6` are then used directly in swap price computation:

```solidity
// MetricOmmPool.sol:540-541
uint256 buyFeeX64 = baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6);
uint256 sellFeeX64 = baseFeeX64 + Math.mulDiv(binState.addFeeSellE6, ONE_X64, 1e6);
``` [4](#0-3) 

These inflate the ask price (buy) or deflate the bid price (sell) paid by every trader routing through that bin, with no upper bound enforced by the factory.

### Impact Explanation

With `maxAdminSpreadFeeE6 = 1000` (0.1%) and `addFeeBuyE6 = 65535` (6.5535%), every trader buying token0 through the affected bin pays ~65× the permitted admin spread cap. The excess is embedded in the execution price — the trader receives less output than the oracle-derived price permits, constituting direct loss of user principal above Sherlock thresholds. This satisfies the "Admin-boundary break: pool admin exceeds caps" impact gate. [5](#0-4) 

### Likelihood Explanation

The pool admin is explicitly modeled as semi-trusted only within caps. The factory enforces caps on the global admin fee path but omits them on the per-bin path. Any pool admin (a role that can be transferred via a two-step process) can exploit this immediately after pool creation with a single transaction, with no timelock or protocol approval required. [6](#0-5) 

### Recommendation

Add a cap check in `setPoolBinAdditionalFees` before forwarding to the pool:

```solidity
if (addFeeBuyE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
```

Alternatively, enforce the check inside `setBinAdditionalFees` on the pool by reading the factory's cap, though the factory-side check is simpler and consistent with the existing pattern in `setPoolAdminFees`. [7](#0-6) 

### Proof of Concept

```solidity
// Foundry unit test sketch
function test_binFeeExceedsCap() public {
    // Factory configured with maxAdminSpreadFeeE6 = 1000 (0.1%)

    // No revert — cap is not enforced
    factory.setPoolBinAdditionalFees(pool, 0, 65535, 65535);

    (,,, uint16 buyFee,) = PoolStateLibrary._binState(pool, 0);
    assertEq(buyFee, 65535); // 6.5535% >> 0.1% cap

}
