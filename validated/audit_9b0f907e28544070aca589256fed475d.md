Looking at the `skip_stateful_validations` function and the surrounding gateway stateful validation path, I need to trace how the mempool presence check interacts with signature verification bypass.

### Title
Forged Invoke Transaction Bypasses Signature Verification via Unprivileged Mempool Presence Check in `skip_stateful_validations` - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the gateway's stateful validation path skips the `__validate__` entry point (signature verification) for any invoke transaction with `nonce=1` targeting an undeployed account, provided that account's address already appears in the mempool. Because the mempool presence check (`account_tx_in_pool_or_recent_block`) is keyed only on the **sender address** — not on whether a deploy-account transaction for that address was submitted by the same party — any unprivileged actor can front-run a victim's deploy-account submission and inject a forged invoke for the victim's address into the mempool without a valid signature.

---

### Finding Description

**The skip-validate path**

`extract_state_nonce_and_run_validations` calls `run_pre_validation_checks`, which in turn calls `skip_stateful_validations`:

```
extract_state_nonce_and_run_validations
  └─ run_pre_validation_checks
       ├─ validate_state_preconditions   (nonce range, resource bounds)
       ├─ validate_by_mempool            (duplicate-nonce / fee-escalation)
       └─ skip_stateful_validations      ← returns bool
  └─ run_validate_entry_point(skip_validate)
       └─ ExecutionFlags { validate: !skip_validate, … }
```

`skip_stateful_validations` returns `true` — causing `validate: false` and therefore **no `__validate__` call** — when all three conditions hold simultaneously:

1. The transaction is an `Invoke`.
2. `tx.nonce() == Nonce(Felt::ONE)` (hardcoded).
3. `account_nonce == Nonce(Felt::ZERO)` (account not yet on-chain).
4. `mempool_client.account_tx_in_pool_or_recent_block(sender_address)` returns `true`.

The fourth condition is satisfied by **any** transaction for the target address that is already in the pool or in a recently committed block:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)   // staged or committed
        || self.tx_pool.contains_account(account_address)  // pending pool
}
```

`MempoolState::contains_account` checks only the address key in `staged` or `committed` maps — it does not distinguish deploy-account transactions from invoke transactions, nor does it verify who submitted the transaction.

**The broken invariant**

The design intent (documented in the code comment) is:

> *"It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."*

This reasoning is circular: the very skip-validate path being guarded is what allows a third party to insert a transaction for the victim's address that "passed validations" — because it passed validations only because the skip-validate path was triggered.

**Attack flow**

1. **Alice** submits `deploy_account` for address `X` (nonce 0). The gateway fully validates it (deploy-account always runs `execute`, not just `validate`). Alice's address `X` is now in `tx_pool`.

2. **Mallory** observes `X` in the public mempool. Mallory constructs an `Invoke` transaction with:
   - `sender_address = X`
   - `nonce = 1`
   - arbitrary `calldata`
   - arbitrary (invalid) `signature`
   - resource bounds ≥ stateless minimum

3. Mallory submits this forged invoke to the gateway. The stateful validator runs:
   - `validate_nonce`: `0 ≤ 1 ≤ max_allowed_nonce_gap (200)` → **passes**
   - `validate_by_mempool`: no nonce-1 tx for `X` yet → **passes**
   - `skip_stateful_validations`: `nonce==1`, `account_nonce==0`, `account_tx_in_pool_or_recent_block(X)==true` → **returns `true`**
   - `run_validate_entry_point`: `ExecutionFlags { validate: false, … }` → **`__validate__` is never called**

4. Mallory's forged invoke is **admitted to the mempool without any signature check**.

5. If Alice has already submitted her legitimate invoke (nonce 1), Mallory can replace it by offering a higher tip (standard fee-escalation path in `validate_fee_escalation`). Alice's real invoke is evicted.

6. When the batcher later executes the block:
   - Alice's `deploy_account` (nonce 0) executes → account `X` is deployed.
   - Mallory's forged invoke (nonce 1) executes → `__validate__` is called by the blockifier with `validate: true` → signature check fails → transaction is **rejected** (not included in the block).
   - Alice's nonce remains at 1, but her real invoke is no longer in the mempool.

7. Alice must resubmit. Mallory repeats step 2–6 indefinitely, achieving a **persistent DoS** against Alice's first post-deployment invoke.

---

### Impact Explanation

**Impact: High — Mempool/gateway admission accepts an invalid transaction before sequencing.**

The gateway's stateful validation path admits an invoke transaction carrying an arbitrary, unverified signature into the production mempool. The invariant that every admitted invoke must have passed `__validate__` (or be a deploy/declare that was fully executed) is broken. The concrete consequences are:

- A forged invoke for any undeployed account whose deploy-account is visible in the mempool is unconditionally admitted without signature verification.
- The victim's legitimate nonce-1 invoke can be displaced by fee escalation, forcing repeated resubmission.
- The attack is repeatable: each time the victim resubmits, Mallory can front-run again.

---

### Likelihood Explanation

**Likelihood: Medium.**

- The mempool is public; deploy-account transactions are visible to all observers.
- The attacker needs only to craft a well-formed invoke (valid address, non-zero resource bounds, nonce=1) — no cryptographic material belonging to the victim is required.
- The attack window is the time between Alice's deploy-account entering the mempool and the batcher committing the block containing it. On a live sequencer this window spans multiple seconds to minutes.
- The only cost to the attacker is the gas for submitting the forged invoke; no fee is charged to the attacker's account because the `sender_address` is the victim's address.

---

### Recommendation

The root cause is that `skip_stateful_validations` uses a flat address-keyed presence check rather than verifying that the specific transaction enabling the skip is a deploy-account submitted by the same party.

**Option 1 (minimal fix):** Change `account_tx_in_pool_or_recent_block` to return `true` only when a **deploy-account** transaction for the address is present in the pool, not any transaction type. This requires the mempool to track transaction types per address.

**Option 2 (structural fix, analogous to the EthenaMinting nested-mapping fix):** Replace the flat address check with a nested structure: `deploy_account_in_pool[sender_address] → bool`, populated only when a deploy-account transaction is added and cleared when it is committed or evicted. This prevents a third-party invoke from satisfying the condition.

**Option 3 (conservative fix):** Remove the skip-validate UX feature entirely for the gateway path, or require that even when `skip_validate=true` the gateway still runs a lightweight signature-format check (e.g., verifying the signature length and that it is non-empty) before admission.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
