### Title
Ungated `MessageType::Send` Processing Propagates Fatal Error for Non-Participant Senders, Enabling Permanent DOS of DKG/Reshare/Refresh — (File: `src/protocol/echo_broadcast.rs`)

---

### Summary

In `reliable_broadcast_receive_all`, the `MessageType::Send` branch calls `participants.index(from)?`, which propagates a fatal `ProtocolError::InvalidIndex` if `from` is not in the participant list. The `MessageType::Echo` and `MessageType::Ready` branches correctly use `seen_echo.put(from)` / `seen_ready.put(from)`, which silently return `false` for unknown senders. This inconsistency means any party that can inject a single `MessageType::Send` message attributed to a non-participant identity can abort every honest node's `do_broadcast` call — and therefore every DKG, reshare, or refresh run — permanently.

---

### Finding Description

`reliable_broadcast_receive_all` is the core receive loop used by `do_broadcast`, which is called throughout `do_keyshare` (DKG, reshare, refresh) and `broadcast_success`. [1](#0-0) 

Inside the loop, the three message-type arms handle unknown senders differently:

**`MessageType::Echo` and `MessageType::Ready`** — safe:

```rust
// Echo branch (line 215)
if !state_sid.seen_echo.put(from) || state_sid.finish_echo {
    continue;
}
// Ready branch (line 268)
if !state_sid.seen_ready.put(from) || state_sid.finish_ready {
    continue;
}
```

`ParticipantCounter::put` returns `false` for any `from` not in the participant list, so the message is silently skipped. [2](#0-1) 

**`MessageType::Send`** — fatal:

```rust
// Send branch (line 199)
if state_sid.finish_send || sid != participants.index(from)? {
    continue;
}
```

`participants.index(from)` returns `Err(ProtocolError::InvalidIndex)` when `from` is absent from the list. [3](#0-2) 

The `?` operator propagates this error immediately out of `reliable_broadcast_receive_all`, aborting the entire `do_broadcast` call and every protocol that depends on it. [4](#0-3) 

The `Protocol::message` entry point — the only way external messages enter the system — accepts any `Participant` as `from` with no validation: [5](#0-4) 

`Comms::push_message` likewise performs no participant-list check before buffering the message: [6](#0-5) 

The library's own test explicitly acknowledges that an attacker can inject messages attributed to arbitrary participant IDs: [7](#0-6) 

---

### Impact Explanation

Any party that can deliver one network message to an honest participant — attributed to a `Participant` ID not in the session's participant list — can cause that participant's `reliable_broadcast_receive_all` to return `Err(ProtocolError::InvalidIndex)`. Because `do_broadcast` is called multiple times inside `do_keyshare`: [8](#0-7) [9](#0-8) 

…a single injected `Send` message aborts the entire DKG, reshare, or refresh run for the targeted honest node. By repeating the injection at the start of each retry, the attacker achieves permanent denial of key generation and resharing for all honest parties — matching the **High: Permanent denial of key generation / reshare / refresh** impact class.

---

### Likelihood Explanation

The `Protocol::message(from, data)` interface is the documented integration point for library consumers. No documentation or enforcement requires callers to pre-validate that `from` belongs to the session's participant list before routing a received network packet. In any deployment where the transport layer does not authenticate sender identity (e.g., unauthenticated UDP, or a relay that forwards messages from unknown peers), an off-path or on-path attacker can craft a single `MessageType::Send` packet with a spoofed or unknown `from` ID and deliver it to any honest participant. The attack requires no cryptographic material and no participation in the protocol.

---

### Recommendation

Replace the error-propagating `participants.index(from)?` with a safe check that mirrors the `Echo`/`Ready` branches — silently skipping messages from unknown senders:

```rust
MessageType::Send(data) => {
    // Safely ignore senders not in the participant list,
    // consistent with Echo/Ready handling.
    let Ok(from_index) = participants.index(from) else {
        continue;
    };
    if state_sid.finish_send || sid != from_index {
        continue;
    }
    // ... rest of Send handling unchanged
}
``` [10](#0-9) 

Additionally, consider adding sender-list validation inside `Protocol::message` or `Comms::push_message` so that messages from unknown participants are dropped before they reach any protocol logic.

### Proof of Concept

1. Honest participants `P1`, `P2`, `P3` initiate DKG (threshold = 2). `do_keyshare` calls `do_broadcast` at line 435.
2. Attacker `A` (ID not in `{P1, P2, P3}`) sends a raw message to `P1` over the network. The message body deserializes as `(sid=0, MessageType::Send(arbitrary_data))`.
3. `P1`'s host calls `protocol.message(A, raw_bytes)` — permitted by the `Protocol` trait with no validation.
4. The message is buffered and delivered to `reliable_broadcast_receive_all` via `chan.recv(wait)`.
5. The `MessageType::Send` arm executes `participants.index(A)?`. Since `A ∉ {P1,P2,P3}`, this returns `Err(ProtocolError::InvalidIndex)`.
6. The `?` propagates the error; `reliable_broadcast_receive_all` returns `Err`, `do_broadcast` returns `Err`, `do_keyshare` returns `Err`.
7. `P1`'s DKG session is permanently aborted. Repeating for `P2` and `P3` prevents any quorum from completing DKG. [11](#0-10) [12](#0-11)
