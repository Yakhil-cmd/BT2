### Title
Unchecked Return Value of Low-Level Precompile `call()` After Token Burn Causes Permanent Fund Loss - (File: `etc/eth-contracts/contracts/EvmErc20.sol`, `etc/eth-contracts/contracts/EvmErc20V2.sol`)

---

### Summary

Both `EvmErc20.withdrawToNear` and `EvmErc20.withdrawToEthereum` (and their V2 counterparts) burn the caller's ERC-20 tokens first, then invoke the exit precompile via a low-level assembly `call()`. The return value of that `call()` is captured in a local variable `res` but is **never checked**. If the precompile call fails for any reason (out of gas, paused precompile, invalid state), the burn is permanent and no withdrawal promise is ever scheduled — the user's funds are silently destroyed.

---

### Finding Description

In `EvmErc20.sol`, both exit functions follow this pattern:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // tokens destroyed here

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        // res is never checked — if call fails, execution continues silently
    }
}
``` [1](#0-0) 

The same unchecked pattern appears in `withdrawToEthereum`: [2](#0-1) 

And identically in `EvmErc20V2.sol`: [3](#0-2) 

The precompile addresses called are:
- `0xe9217bc70b7ed1f598ddd3199e80b093fa71124f` — `ExitToNear` precompile
- `0xb0bd02f6a392af548bdf1cfaee5dfa0eefcc8eab` — `ExitToEthereum` precompile [4](#0-3) 

In the EVM, a failed sub-`call()` does **not** revert the caller's state changes. The `_burn()` modifies the ERC-20 balance mapping before the `call()` is made. If the precompile call returns `0` (failure), the assembly block exits silently, the function returns normally, and the burn is committed — but no NEAR-side withdrawal promise is ever scheduled.

---

### Impact Explanation

**Critical — Permanent loss of user funds.**

When the precompile call fails silently:
1. The user's ERC-20 mirror tokens are permanently burned (balance reduced to zero for the withdrawn amount).
2. No `ExitToNear` or `ExitToEthereum` promise is scheduled on the NEAR side.
3. The corresponding NEP-141 tokens remain locked in the Aurora engine contract with no mechanism for the user to recover them.

The user loses both their ERC-20 representation and their ability to reclaim the underlying NEP-141 asset. This constitutes a permanent, irreversible fund loss.

---

### Likelihood Explanation

**Medium.** Multiple realistic failure conditions exist for the precompile call:

1. **Insufficient gas forwarded**: A user or contract calling `withdrawToNear` with a gas limit tight enough to execute `_burn` but insufficient for the precompile's internal logic will trigger a silent failure. The `call(gas(), ...)` forwards all remaining gas, but if the caller's transaction was submitted with a low gas limit, the precompile may run out mid-execution and return `0`.

2. **Paused precompile**: The codebase explicitly supports pausing exit precompiles via `pause_precompiles` / `resume_precompiles`. [5](#0-4)  When the `ExitToNear` or `ExitToEthereum` precompile is paused, any call to it returns an error. Any user who calls `withdrawToNear` or `withdrawToEthereum` while the precompile is paused will have their tokens burned with no recourse.

3. **Invalid input edge cases**: The precompile validates input size and other parameters. [6](#0-5)  A malformed `recipient` argument that passes Solidity-level encoding but fails precompile-level validation would cause a silent failure after the burn.

---

### Recommendation

**Short term:** Check the return value of the assembly `call()` and revert if it is zero:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

This ensures that if the precompile call fails for any reason, the entire transaction (including the `_burn`) is reverted, preserving the user's balance.

**Long term:** Restructure the withdrawal flow to schedule the precompile call before burning, or use a two-phase commit pattern (lock → confirm burn only on callback success) to eliminate the burn-before-confirm race condition entirely.

---

### Proof of Concept

1. Admin calls `pause_precompiles` to pause the `ExitToNear` precompile (a legitimate operational action, e.g., during an upgrade).
2. User Alice holds 1000 units of a NEP-141-backed ERC-20 mirror token on Aurora.
3. Alice calls `EvmErc20.withdrawToNear("alice.near", 1000)`.
4. `_burn(alice, 1000)` executes — Alice's ERC-20 balance drops to 0.
5. The assembly `call()` to the paused `ExitToNear` precompile returns `0`.
6. `res` is never checked; the function returns normally with no revert.
7. Alice's 1000 ERC-20 tokens are permanently destroyed. No NEP-141 transfer is scheduled. Alice's underlying NEP-141 tokens remain locked in the Aurora engine contract with no recovery path. [1](#0-0) [7](#0-6)
