### Title
Missing Lower-Bound Validation on `threshold` in FROST `assert_sign_inputs` Allows Signing with Threshold = 1 — (`File: src/frost/mod.rs`)

### Summary
`assert_sign_inputs` in `src/frost/mod.rs` validates the signing threshold only from above (`threshold > participants.len()`), but never from below. A caller — including a malicious coordinator — can pass `threshold = 1` (or `threshold = 0`) to the FROST signing entry point. Every other validation function in the codebase enforces `threshold >= 2`, but this check is absent in the signing path, creating an inconsistency that corrupts signing outputs.

### Finding Description

`assert_key_invariants` in `src/dkg.rs` enforces a strict lower bound:

```rust
// src/dkg.rs lines 580-582
if threshold < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
}
``` [1](#0-0) 

`validate_triple_inputs` in `src/ecdsa/ot_based_ecdsa/triples/generation.rs` also enforces it:

```rust
// lines 699-703
if threshold_value < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold_value,
        min: 2,
    });
}
``` [2](#0-1) 

But `assert_sign_inputs` in `src/frost/mod.rs` only checks the upper bound:

```rust
// src/frost/mod.rs lines 144-150
// validate threshold
if threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge {
        threshold: threshold.value(),
        max: participants.len(),
    });
}
``` [3](#0-2) 

There is **no** corresponding `ThresholdTooSmall` guard. A caller supplying `threshold = 1` with `participants.len() = 2` passes every check in `assert_sign_inputs` and proceeds into the FROST signing protocol. [4](#0-3) 

### Impact Explanation

When `threshold = 1` is accepted by the signing path, the Lagrange interpolation inside the FROST signing protocol uses basis polynomials of degree 0 (a single-point reconstruction), which is inconsistent with the keygen threshold of `>= 2`. The aggregated signature scalar is computed from the wrong Lagrange coefficients, producing a signature that fails external verification against the public key that was generated with the correct threshold. Honest participants complete the protocol without error and receive an unusable, cryptographically invalid signature — matching **High: Corruption of sign outputs so honest parties accept unusable cryptographic outputs**.

If `threshold = 0` is passed, `threshold.value() > participants.len()` evaluates to `0 > N` which is false, so the protocol proceeds into Lagrange interpolation with a zero-degree basis, likely causing a panic or undefined scalar arithmetic — a **High: Permanent denial of signing**.

### Likelihood Explanation

The entry point is the public FROST signing API. Any library caller — including a malicious or misconfigured coordinator — controls the `threshold` argument directly. No special privilege, leaked key, or cryptographic break is required. The inconsistency between the keygen validation (which rejects `threshold < 2`) and the signing validation (which does not) makes accidental misuse highly plausible and deliberate exploitation trivial.

### Recommendation

Add the same lower-bound guard present in `assert_key_invariants` and `validate_triple_inputs` to `assert_sign_inputs`:

```rust
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

This should be inserted immediately after `let threshold = threshold.into();` in `assert_sign_inputs`, mirroring the pattern at `src/dkg.rs` lines 580–582. [5](#0-4) 

### Proof of Concept

1. Run keygen with `participants = [P1, P2, P3]`, `threshold = 2` → produces `KeygenOutput` with a public key derived from a degree-1 polynomial.
2. Call the FROST `sign` entry point (which calls `assert_sign_inputs`) with the same participants but `threshold = 1`.
3. `assert_sign_inputs` passes all checks: `participants.len() = 3 >= 2`, `threshold.value() = 1 <= 3`.
4. The signing protocol executes, computing Lagrange coefficients for a degree-0 (threshold-1) interpolation instead of degree-1.
5. The aggregated signature scalar is wrong; the returned signature fails `verify(public_key, msg_hash)` with any standard ECDSA/EdDSA verifier.
6. Honest participants observe a completed protocol with no `ProtocolError`, but hold an invalid signature — the signing output is silently corrupted. [3](#0-2) [6](#0-5)

pub fn assert_sign_inputs(
    participants: &[Participant],
    threshold: impl Into<ReconstructionLowerBound>,
    me: Participant,
    coordinator: Participant,
) -> Result<ParticipantList, InitializationError> {
    let threshold = threshold.into();
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // validate threshold
    if threshold.value() > participants.len() {
        return Err(InitializationError::ThresholdTooLarge {
            threshold: threshold.value(),
            max: participants.len(),
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }
    Ok(participants)
}
