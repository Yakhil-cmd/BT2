### Title
Silo Mode Whitelist Removal Permanently Freezes User Funds - (`engine/src/engine.rs`, `engine/src/contract_methods/silo/mod.rs`)

### Summary
In Silo Mode, the `assert_access` guard applies the `Address`/`Account` whitelist check to every EVM transaction without exception. If a user's address is removed from the whitelist after they have accumulated ERC-20 or ETH balances, they can no longer submit any transaction — including withdrawals — permanently freezing their funds.

### Finding Description

`receive_erc20_tokens` in `engine/src/engine.rs` credits ERC-20 tokens to a user's EVM address when that address is whitelisted (or redirects to a fallback address if not): [1](#0-0) 

Once tokens are credited, the only way to move them is via an EVM transaction through `submit()`. Every call to `submit()` passes through `submit_with_alt_modexp`, which calls `assert_access`: [2](#0-1) 

`assert_access` unconditionally checks the Silo whitelist for all transactions — both calls (`transaction.to.is_some()`) and deploys: [3](#0-2) 

`is_allow_submit` requires the sender's EVM address to be present in the enabled `Address` whitelist: [4](#0-3) [5](#0-4) 

There is no exception for "exit" transactions (e.g., `exit_to_near`, ERC-20 transfers). If the Silo admin removes a user's address from the whitelist via `remove_entry_from_whitelist`, the user is permanently blocked from submitting any EVM transaction: [6](#0-5) 

The existing test `test_submit_with_removing_entries` confirms this behavior — after removal, all submissions revert with `EngineErrorKind::NotAllowed`: [7](#0-6) 

### Impact Explanation

**Critical — Permanent freezing of funds.**

Any ERC-20 tokens or ETH balance held by a de-whitelisted address become permanently inaccessible. The user has no unilateral mechanism to recover their funds: all EVM exit paths (`exit_to_near`, ERC-20 `transfer`, `exit_to_ethereum`) require submitting an EVM transaction, which is blocked by `assert_access`. The funds remain locked in the Aurora EVM state indefinitely unless the admin re-adds the user to the whitelist.

### Likelihood Explanation

**Medium.** Silo Mode is an explicitly supported production configuration. Any Silo operator who removes a user from the `Address` whitelist (for compliance, security, or operational reasons) while that user holds a non-zero balance triggers this freeze. The operator may not realize the consequence. The scenario is realistic and requires no attacker — only a routine administrative action.

### Recommendation

Introduce a whitelist exemption for "exit" transactions. Specifically, transactions whose `to` field targets the `exit_to_near` or `exit_to_ethereum` precompile addresses should bypass the `Address`/`Account` whitelist check in `assert_access`, analogous to the M-16 recommendation of allowing module disabling regardless of blacklist status. Alternatively, provide a dedicated `withdraw` entrypoint that is not gated by the Silo whitelist.

### Proof of Concept

```
1. Deploy Aurora Engine in Silo Mode with Address whitelist enabled.
2. Add user address U to the Address whitelist.
3. Bridge NEP-141 tokens to U via ft_transfer_call → receive_erc20_tokens.
   → U now holds ERC-20 tokens inside the Aurora EVM.
4. Admin calls remove_entry_from_whitelist({kind: Address, address: U}).
5. U attempts to submit any EVM transaction (transfer, exit_to_near, etc.).
   → assert_access → is_allow_submit → is_address_allowed returns false
   → EngineErrorKind::NotAllowed
6. U's tokens are permanently frozen with no recovery path.
``` [3](#0-2) [6](#0-5)

fn assert_access<I: IO + Copy, E: Env>(
    io: &I,
    env: &E,
    transaction: &NormalizedEthTransaction,
) -> Result<(), EngineError> {
    let allowed = if transaction.to.is_some() {
        silo::is_allow_submit(io, &env.predecessor_account_id(), &transaction.address)
    } else {
        silo::is_allow_deploy(io, &env.predecessor_account_id(), &transaction.address)
    };

    if !allowed {
        return Err(EngineError {
            kind: EngineErrorKind::NotAllowed,
            gas_used: 0,
        });
    }

    Ok(())
}
