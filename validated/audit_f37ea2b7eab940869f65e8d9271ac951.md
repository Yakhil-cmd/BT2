### Title
EVM Gas Fee Collector Mis-Accounting via Coinbase Balance Injection — (File: `fvm/evm/handler/handler.go`)

---

### Summary

`runWithGasFeeRefund` in `fvm/evm/handler/handler.go` determines gas fees by measuring the balance change of the fixed `CoinbaseAddress` before and after executing EVM transactions. Because any EVM-level token transfer to `CoinbaseAddress` — not only protocol-generated gas fees — inflates the measured delta, an attacker who can cause tokens from a contract they do not own to reach `CoinbaseAddress` during execution will have those tokens mis-credited as gas fees and forwarded to their `gasFeeCollector`. This is the direct structural analog of the Olympus treasury balance-injection vulnerability.

---

### Finding Description

`runWithGasFeeRefund` (lines 252–266 of `fvm/evm/handler/handler.go`) implements the following pattern:

```
initCoinbaseBalance = balance(CoinbaseAddress)
f()                                          // executes one or more EVM txs
afterBalance       = balance(CoinbaseAddress)
diff               = afterBalance - initCoinbaseBalance
cb.Transfer(gasFeeCollector, diff)           // entire delta forwarded
``` [1](#0-0) 

This function is invoked by every public EVM execution entry-point — `RunOrPanic`, `Run`, and `BatchRun`: [2](#0-1) 

`types.CoinbaseAddress` is a fixed, well-known EVM address. Any EVM contract can transfer tokens to it with a plain `CALL` or via `SELFDESTRUCT`. The implementation makes no distinction between tokens that arrived as EVM-protocol gas fees and tokens that arrived via an arbitrary EVM transfer. The entire balance increase is unconditionally forwarded to `gasFeeCollector`.

The `gasFeeCollector` is the COA (Cadence-Owned Account) EVM address of the transaction sender, so the attacker controls it. [3](#0-2) 

---

### Impact Explanation

If a victim EVM contract holds FLOW tokens and exposes any callable path that transfers tokens to `CoinbaseAddress` (e.g., a public `tip()` function, a `SELFDESTRUCT` with coinbase as beneficiary, or a flash-loan callback that routes funds through coinbase), an attacker can:

1. Submit a transaction (via `Run` / `BatchRun`) that triggers that path.
2. `runWithGasFeeRefund` measures the coinbase balance increase, which now includes the victim's tokens on top of legitimate gas fees.
3. The full delta — victim tokens + gas fees — is transferred to the attacker's `gasFeeCollector` COA address.
4. The attacker withdraws the surplus from their COA to Cadence via the normal `withdraw` path.

The victim contract loses tokens it did not intend to send to the attacker. The attacker gains FLOW tokens they do not own. This is a cross-VM asset loss: tokens that were legitimately held in the EVM environment are drained to an unauthorized Cadence account. [4](#0-3) 

---

### Likelihood Explanation

**Medium-low.** The precondition is that at least one deployed EVM contract exposes a callable path that sends tokens to `CoinbaseAddress`. In standard EVM ecosystems, "coinbase bribery" contracts (MEV tip contracts, validator-incentive contracts) are common. Flow EVM is EVM-compatible and will attract the same contract patterns. A `SELFDESTRUCT`-based variant requires only that the attacker deploys their own contract and self-destructs it with coinbase as beneficiary — but in that case the attacker is moving their own tokens, so the gain is zero. The high-impact case (stealing from a third-party contract) requires a victim contract with a reachable coinbase-transfer path, which is realistic as the ecosystem grows. The entry path is fully unprivileged: any Flow account can submit an EVM transaction.

---

### Recommendation

Replace the before/after balance-difference heuristic with explicit, per-transaction gas fee accounting. The EVM execution result already contains the gas consumed and the effective gas price; the exact fee owed to the collector can be computed directly from `result.GasConsumed * effectiveGasPrice` without ever reading `CoinbaseAddress`'s balance. This eliminates the injection surface entirely.

Alternatively, restrict `CoinbaseAddress` so that it can only receive tokens via the EVM protocol's own gas-fee mechanism (i.e., reject or revert any direct `CALL` or `SELFDESTRUCT` that targets it), though this requires emulator-level enforcement.

---

### Proof of Concept

1. Attacker deploys EVM contract `VictimTip` that holds 100 FLOW and exposes `function tipCoinbase() external { payable(block.coinbase).transfer(address(this).balance); }`.
2. Attacker calls `EVM.run(encodedTx, attackerCOAAddress)` where `encodedTx` invokes `VictimTip.tipCoinbase()`.
3. Inside `runWithGasFeeRefund`:
   - `initCoinbaseBalance` = 0 (coinbase starts empty each block).
   - `f()` executes: EVM gas fees (e.g., 0.001 FLOW) + 100 FLOW from `VictimTip` both land at `CoinbaseAddress`.
   - `afterBalance` = 100.001 FLOW.
   - `diff` = 100.001 FLOW.
4. `cb.Transfer(attackerCOAAddress, 100.001 FLOW)` is executed.
5. Attacker calls `cadenceOwnedAccount.withdraw(balance: ...)` to move 100.001 FLOW to Cadence.

`VictimTip` loses 100 FLOW; attacker gains 100 FLOW. The protocol's EVM vault remains balanced (tokens moved within EVM then out via COA withdrawal), but the victim contract's funds are stolen without authorization. [1](#0-0) [5](#0-4)

func (h *ContractHandler) runWithGasFeeRefund(gasFeeCollector types.Address, f func()) {
	// capture coinbase init balance
	cb := h.AccountByAddress(types.CoinbaseAddress, true)
	initCoinbaseBalance := cb.Balance()
	f()
	// transfer the gas fees collected to the gas fee collector address
	afterBalance := cb.Balance()
	diff := new(big.Int).Sub(afterBalance, initCoinbaseBalance)
	if diff.Sign() > 0 {
		cb.Transfer(gasFeeCollector, diff)
	}
	if diff.Sign() < 0 { // this should never happen but in case
		panic(fvmErrors.NewEVMError(fmt.Errorf("negative balance change on coinbase")))
	}
}
