### Title
`protocolUnpausePool` Always Sets Pause Level to 1 (Admin-Paused) Instead of Restoring Active State, Leaving Swaps Permanently Blocked Until Admin Intervenes - (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`protocolUnpausePool` unconditionally calls `setPause(1)` (admin-paused) rather than `setPause(0)` (active). When the protocol pauses a pool that was previously active (level 0) and then unpauses it, the pool is left in admin-paused state. Because `_checkNotPaused()` reverts on any non-zero pause level, swaps remain blocked indefinitely until the pool admin separately calls `unpausePool`. The pool admin never paused the pool, yet must now explicitly unpause it to restore normal operation.

---

### Finding Description

The pause level semantics are:

```
0 = active
1 = paused by admin
2 = paused by protocol
```

`protocolPausePool` accepts pools at level 0 or 1 and moves them to level 2:

```solidity
// MetricOmmPoolFactory.sol L392-L396
function protocolPausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 0 && cur != 1) revert InvalidPauseTransition(cur, 2);
    IMetricOmmPoolFactoryActions(pool).setPause(2);
}
```

`protocolUnpausePool` always moves from level 2 to level 1, never to level 0:

```solidity
// MetricOmmPoolFactory.sol L398-L403
function protocolUnpausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 2) revert InvalidPauseTransition(cur, 1);
    IMetricOmmPoolFactoryActions(pool).setPause(1);   // ← always 1, never 0
}
```

`swap` is gated by `whenNotPaused`, which reverts on any non-zero level:

```solidity
// MetricOmmPool.sol L643-L645
function _checkNotPaused() internal view {
    if (pauseLevel != 0) revert PoolPaused();
}
```

Scenario:
1. Pool is active (level 0).
2. Protocol owner calls `protocolPausePool` → level becomes 2.
3. Protocol owner calls `protocolUnpausePool` → level becomes 1 (admin-paused).
4. Every `swap` call reverts with `PoolPaused()`.
5. The pool admin — who never paused the pool — must now call `unpausePool` to restore level 0.

The factory enforces that `unpausePool` (admin path) only works from level 1:

```solidity
// MetricOmmPoolFactory.sol L467-L471
function unpausePool(address pool) external override nonReentrant onlyPoolAdmin(pool) {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 1) revert InvalidPauseTransition(cur, 0);
    IMetricOmmPoolFactoryActions(pool).setPause(0);
}
```

So the admin *can* fix it, but only if they are aware of the issue and take action. There is no mechanism for the protocol owner to directly restore a pool to active state.

---

### Impact Explanation

Every swap on the pool reverts with `PoolPaused()` after `protocolUnpausePool` is called on a previously-active pool. This is broken core swap functionality. LPs cannot trade; integrators and users receive unexpected reverts from a pool that the protocol owner believed they had unpaused. The pool is effectively unusable for its primary purpose until the admin takes a separate, unrelated action.

---

### Likelihood Explanation

This triggers in the normal operational flow: protocol pauses a pool for an emergency or upgrade, then unpauses it. Any pool that was at level 0 before the protocol pause is affected. The protocol owner has no way to know from the function signature or NatSpec that calling `protocolUnpausePool` will leave the pool paused. The bug fires on every such cycle.

---

### Recommendation

Store the pre-pause level before moving to level 2, or unconditionally restore to level 0 on protocol unpause:

```solidity
function protocolUnpausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 2) revert InvalidPauseTransition(cur, 0);
    IMetricOmmPoolFactoryActions(pool).setPause(0); // restore to active
}
```

If the intent is to allow the admin to review before going live, document this explicitly and add a separate `protocolUnpauseToActive` path, or emit an event warning that the pool remains admin-paused.

---

### Proof of Concept

1. Deploy factory and pool; pool starts at pause level 0 (active).
2. Owner calls `protocolPausePool(pool)` → `pauseLevel == 2`.
3. Owner calls `protocolUnpausePool(pool)` → `pauseLevel == 1`.
4. Any user calls `pool.swap(...)` → reverts with `PoolPaused()`.
5. Pool admin calls `unpausePool(pool)` → `pauseLevel == 0`; swaps now succeed.
