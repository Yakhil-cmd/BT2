### Title
EVM Hook Handlers Blindly Trust Event-Reported `amount` Without Balance Verification, Enabling Unauthorized Native Token Minting via Deflationary/Upgraded External Contracts - (File: x/cronos/keeper/evmhandlers/send_to_ibc.go)

---

### Summary

`SendToIbcHandler.handle()` and `SendToIbcV2Handler.Handle()` read the `amount` field directly from the `__CronosSendToIbc` EVM event log and use it to mint or transfer native Cosmos tokens without ever verifying that the emitting contract actually locked or burned that exact amount of EVM tokens. When an externally-registered contract (e.g., `ModuleCRC20Proxy`) wraps an upgradeable CRC20 that is later upgraded to deflationary behavior, the event reports the requested `amount` while the proxy only locks `amount − fee`. The handler mints the full `amount` of native tokens, creating unbacked supply that can be IBC-transferred out of Cronos.

---

### Finding Description

**Vulnerability class**: Accounting bug / bridge flaw — exact analog of the Polkaswap "blindly trusts upgradeable ERC20 proxy tokens" report.

**Root cause in `send_to_ibc.go`**:

```go
// x/cronos/keeper/evmhandlers/send_to_ibc.go  lines 105-117
amount := sdkmath.NewIntFromBigInt(amountInt)          // taken verbatim from event log
coins := sdk.NewCoins(sdk.NewCoin(denom, amount))

if types.IsSourceCoin(denom) {
    // mints amount of native tokens — no balance-change verification
    if err = h.bankKeeper.MintCoins(ctx, types.ModuleName, coins); err != nil {
        return err
    }
    if err = h.bankKeeper.SendCoinsFromModuleToAccount(ctx, types.ModuleName, sender, coins); err != nil {
        return err
    }
}
``` [1](#0-0) 

The handler never snapshots the contract's EVM balance before and after the transfer to confirm the actual amount locked. It trusts whatever integer the contract placed in the event.

**The emitting contract — `ModuleCRC20Proxy.send_to_ibc` (isSource = true)**:

```solidity
// contracts/src/ModuleCRC20Proxy.sol  lines 78-85
function send_to_ibc(string memory recipient, uint amount, uint channel_id, bytes memory extraData) public {
    if (isSource) {
        crc20Contract.move(msg.sender, address(this), amount);  // ← actual transfer
    } else {
        crc20_burn(msg.sender, amount);
    }
    emit __CronosSendToIbc(msg.sender, channel_id, recipient, amount, extraData);  // ← reports amount
}
``` [2](#0-1) 

`crc20Contract.move()` calls the underlying CRC20's transfer. If that CRC20 is deflationary (burns a fee on every transfer), the proxy receives `amount − fee` but the event still reports `amount`. The Cosmos-side handler sees only the event.

The same gap exists in `SendToIbcV2Handler`, which delegates to the same `handle()` method: [3](#0-2) 

**Contrast with the Gravity bridge contract**, which explicitly guards against this:

```solidity
// integration_tests/contracts/contracts/Gravity.sol  lines 656-682
uint256 ourStartingBalance = IERC20(_tokenContract).balanceOf(address(this));
IERC20(_tokenContract).safeTransferFrom(msg.sender, address(this), _amount);
uint256 ourEndingBalance = IERC20(_tokenContract).balanceOf(address(this));
// emit to Cosmos the actual amount our balance has changed
emit SendToCosmosEvent(..., ourEndingBalance - ourStartingBalance, ...);
``` [4](#0-3) 

The Gravity bridge snapshots balances and emits the **actual** delta. The Cronos EVM hook handlers have no equivalent guard.

---

### Impact Explanation

**Critical — Unauthorized minting of CRC20/native tokens and incorrect bridge accounting.**

For the `isSource = true` path (denom = `cronos0x…`):

- The handler calls `bankKeeper.MintCoins(ctx, types.ModuleName, coins)` for the full event-reported `amount`.
- If the underlying CRC20 burned `fee` on transfer, the proxy holds `amount − fee` but `amount` native tokens were minted.
- The attacker IBC-transfers `amount` native tokens to a destination chain, receiving more than deposited.
- On the return path, `amount` EVM tokens are minted back to the user, but only `amount − fee` were originally locked — net gain of `fee` EVM tokens per round trip.
- Repeated calls inflate the native token supply without backing, constituting unauthorized minting. [5](#0-4) 

---

### Likelihood Explanation

**High.** The `ModuleCRC20Proxy` contract is a production artifact explicitly designed to wrap an external CRC20 and is registered via `UpdateTokenMapping` by the Cronos admin. The underlying CRC20 is an independent contract whose proxy admin can upgrade it at any time without Cronos admin knowledge or consent. Cronos has no upgrade-tracking mechanism and no ability to pause interactions with a token whose semantics have changed post-registration. Once the underlying token is upgraded to deflationary, any unprivileged user can trigger the exploit by calling `send_to_ibc` on the proxy. [6](#0-5) [7](#0-6) 

---

### Recommendation

1. **Short term**: In `SendToIbcHandler.handle()` and `SendToIbcV2Handler.Handle()`, for the `isSource` path, snapshot the contract's EVM token balance before and after the transfer (via a `balanceOf` call) and use the **actual delta** as the authoritative amount, rejecting or capping if the delta is less than the event-reported amount.

2. **Medium term**: Implement an on-chain upgrade-detection mechanism (e.g., track the implementation slot of registered proxy contracts) and automatically pause bridge interactions with any contract whose implementation address has changed since registration, pending re-approval by the Cronos admin.

3. **Immediate mitigation**: Audit all currently registered external contracts (especially `ModuleCRC20Proxy` instances) to verify their underlying CRC20 is not upgradeable or deflationary.

---

### Proof of Concept

1. Admin registers `ModuleCRC20Proxy(crc20=DeflationaryToken, isSource=true)` at address `P` with denom `cronos0xP` via `MsgUpdateTokenMapping`. At registration time, `DeflationaryToken` is a standard ERC20.

2. `DeflationaryToken`'s proxy admin upgrades the implementation to burn 1% on every transfer.

3. Attacker approves `P` to spend 1000 tokens, then calls:
   ```solidity
   P.send_to_ibc("cosmos1attacker...", 1000, 0, "")
   ```

4. Inside `send_to_ibc`:
   - `crc20Contract.move(attacker, P, 1000)` executes; due to 1% burn, `P` receives 990 tokens.
   - `emit __CronosSendToIbc(attacker, 0, "cosmos1attacker...", 1000, "")` — reports 1000.

5. `SendToIbcV2Handler.Handle()` unpacks `amount = 1000`, calls `bankKeeper.MintCoins(cronos, 1000 cronos0xP)`, sends 1000 to attacker's Cosmos address, initiates IBC transfer of 1000.

6. Attacker receives 1000 tokens on the destination chain but only locked 990 EVM tokens on Cronos.

7. Attacker IBC-transfers 1000 tokens back to Cronos; `ConvertVouchersToEvmCoins` mints 1000 EVM tokens back. Net gain: 10 EVM tokens per round trip, with the deficit accumulating as unbacked supply. [8](#0-7) [2](#0-1) [9](#0-8)

func (h SendToIbcHandler) handle(
	ctx sdk.Context,
	contract common.Address,
	senderAddress common.Address,
	recipient string,
	amountInt *big.Int,
	id *big.Int,
) error {
	denom, found := h.cronosKeeper.GetDenomByContract(ctx, contract)
	if !found {
		return fmt.Errorf("contract %s is not connected to native token", contract)
	}

	if !types.IsValidIBCDenom(denom) && !types.IsValidCronosDenom(denom) {
		return fmt.Errorf("the native token associated with the contract %s is neither an ibc voucher or a cronos token", contract)
	}

	contractAddr := sdk.AccAddress(contract.Bytes())
	sender := sdk.AccAddress(senderAddress.Bytes())
	amount := sdkmath.NewIntFromBigInt(amountInt)
	coins := sdk.NewCoins(sdk.NewCoin(denom, amount))

	var err error
	if types.IsSourceCoin(denom) {
		// it is a source token, we need to mint coins
		if err = h.bankKeeper.MintCoins(ctx, types.ModuleName, coins); err != nil {
			return err
		}
		// send the coin to the user
		if err = h.bankKeeper.SendCoinsFromModuleToAccount(ctx, types.ModuleName, sender, coins); err != nil {
			return err
		}
	} else {
		// First, transfer IBC coin to user so that he will be the refunded address if transfer fails
		if err = h.bankKeeper.SendCoins(ctx, contractAddr, sender, coins); err != nil {
			return err
		}
	}

	channelId := ""
	if id != nil {
		channelId = "channel-" + id.String()
	}
	// Initiate IBC transfer from sender account
	if err = h.cronosKeeper.IbcTransferCoins(ctx, sender.String(), recipient, coins, channelId); err != nil {
		return err
	}
	return nil

func (k Keeper) ConvertVouchersToEvmCoins(ctx sdk.Context, from string, coins sdk.Coins) error {
	acc, err := sdk.AccAddressFromBech32(from)
	if err != nil {
		return err
	}

	params := k.GetParams(ctx)
	evmParams := k.GetEvmParams(ctx)
	for _, c := range coins {
		switch c.Denom {
		case params.IbcCroDenom:
			if params.IbcCroDenom == "" {
				return errorsmod.Wrap(types.ErrIbcCroDenomEmpty, "ibc is disabled")
			}

			// Send ibc tokens to escrow address
			err := k.bankKeeper.SendCoinsFromAccountToModule(ctx, acc, types.ModuleName, sdk.NewCoins(c))
			if err != nil {
				return err
			}
			// Compute new amount, because basecro is a 8 decimals token, we need to multiply by 10^10 to make it
			// a 18 decimals token
			amount18dec := sdk.NewCoin(evmParams.EvmDenom, c.Amount.Mul(sdkmath.NewIntFromBigInt(types.TenPowTen)))

			// Mint new evm tokens
			if err := k.bankKeeper.MintCoins(
				ctx, types.ModuleName, sdk.NewCoins(amount18dec),
			); err != nil {
				return err
			}

			// Send evm tokens to receiver
			if err := k.bankKeeper.SendCoinsFromModuleToAccount(
				ctx, types.ModuleName, acc, sdk.NewCoins(amount18dec),
			); err != nil {
				return err
			}

		default:
			err := k.ConvertCoinFromNativeToCRC21(ctx, common.BytesToAddress(acc.Bytes()), c, params.EnableAutoDeployment)
			if err != nil {
				return err
			}
		}
