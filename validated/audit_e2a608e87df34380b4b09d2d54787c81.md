### Title
Inconsistent `max_malicious` Enforcement Between Presign and Sign Phases Enables Split-View Secret Key Extraction — (`src/ecdsa/robust_ecdsa/sign.rs`)

---

### Summary

The Robust ECDSA `presign()` and `sign()` functions each independently enforce the invariant `participants.len() == 2 * max_malicious + 1`, but neither function cross-validates that the `max_malicious` supplied to `sign()` matches the one used during `presign()`. A malicious coordinator can supply a smaller `max_malicious` to `sign()`, reducing the signing set below the presigning set size, creating non-overlapping signing quorums from the same presignature and enabling the documented split-view secret-key-extraction attack.

---

### Finding Description

The security documentation for Robust ECDSA explicitly requires:

> **Use exactly N₁ = N₂ = 2t + 1 participants for both presigning and signing. Do not allow N₂ < N₁.**

`presign()` enforces this for its own phase: [1](#0-0) 

```rust
let robust_ecdsa_threshold = args
    .max_malicious
    .value()
    .checked_mul(2)
    .and_then(|v| v.checked_add(1)) ...;
// enforces N1 == 2*max_malicious+1
if participants.len() != robust_ecdsa_threshold {
    return Err(InitializationError::BadParameters(...));
}
```

`sign()` independently enforces the same shape of invariant, but using a **separate, caller-supplied** `max_malicious` parameter that is never compared against the presignature's stored threshold: [2](#0-1) 

```rust
let robust_ecdsa_threshold = max_malicious   // ← caller-supplied, not from presignature
    .into()
    .value()
    .checked_mul(2)
    .and_then(|v| v.checked_add(1)) ...;
if participants.len() != robust_ecdsa_threshold {
    return Err(...);
}
```

The `presignature: RerandomizedPresignOutput` argument is passed directly to `fut_wrapper` without any check that its embedded threshold matches the `max_malicious` provided to `sign()`: [3](#0-2) 

The two phases therefore enforce the invariant independently and inconsistently — exactly the same structural defect as the external report, where `closeTradeMarketCallback` and `executeAutomationCloseOrderCallback` each compute `isLiquidated` with different inputs and no cross-validation.

---

### Impact Explanation

**Critical — Extraction of the aggregate secret key.**

The security documentation states:

> *"A novel split-view attack exists that can extract the secret key using as few as 2t+2 presigning participants, with as few as two signing sessions."* [4](#0-3) 

Attack scenario:

1. Malicious coordinator runs `presign()` with `max_malicious = t_large` → N₁ = 2·t_large + 1 participants.
2. Coordinator calls `sign()` twice with `max_malicious = t_small` (where t_small < t_large) → N₂ = 2·t_small + 1 < N₁.
3. Because N₂ < N₁, two **non-overlapping** signing subsets of size N₂ can be drawn from the N₁ presigning participants (possible when N₁ ≥ 2·N₂, e.g., N₁ = 7, N₂ = 3).
4. Coordinator presents different `(msg_hash, tweak)` pairs to each non-overlapping subset, reusing the same presignature.
5. The resulting signatures have multiplicatively related nonces; standard ECDSA nonce-reuse analysis recovers the secret key.

---

### Likelihood Explanation

**Medium.** The coordinator is a named, reachable participant in the protocol API. No external assumptions, leaked keys, or cryptographic breaks are required. The coordinator only needs to call `sign()` with a deliberately reduced `max_malicious` value — a single-parameter change that passes all current validation in `sign()`. The attack requires two signing sessions, which is operationally trivial.

---

### Recommendation

Inside `sign()`, after constructing `participants`, verify that the signing `max_malicious` (and thus N₂) matches the threshold embedded in `presignature` (i.e., the `max_malicious` used during `presign()`). Concretely:

- Expose the presigning `max_malicious` (or N₁) from `RerandomizedPresignOutput`.
- In `sign()`, assert `participants.len() == presignature.participant_count()` (or equivalently, that the caller-supplied `max_malicious` equals the presignature's stored value) before proceeding.
- Alternatively, remove the separate `max_malicious` parameter from `sign()` entirely and derive it exclusively from the presignature, eliminating the possibility of caller-supplied inconsistency.

---

### Proof of Concept

```
Presign phase:
  participants = [P1, P2, P3, P4, P5, P6, P7]  (N1 = 7)
  max_malicious = 3  →  robust_ecdsa_threshold = 7  ✓ (passes presign check)

Sign phase — Session A (malicious coordinator):
  participants = [P1, P2, P3]  (N2 = 3)
  max_malicious = 1  →  robust_ecdsa_threshold = 3  ✓ (passes sign check)
  msg_hash = h1, tweak = ε1
  same presignature reused

Sign phase — Session B (malicious coordinator):
  participants = [P4, P5, P6]  (N2 = 3, non-overlapping with Session A)
  max_malicious = 1  →  robust_ecdsa_threshold = 3  ✓ (passes sign check)
  msg_hash = h2, tweak = ε2
  same presignature reused

Result:
  Two valid signature shares produced from the same presignature nonce
  with different messages → nonce reuse → secret key recovered.
```

The `sign()` initialization at `src/ecdsa/robust_ecdsa/sign.rs` lines 67–90 passes in both sessions because it only checks internal consistency of the caller-supplied `max_malicious` against `participants.len()`, never against the presignature's embedded threshold. [2](#0-1) [5](#0-4) [6](#0-5)
