### Title
Missing Proof-of-Correct-Computation for Participant CKD Share Submissions Allows Malicious Participant to Corrupt Derived Key Output - (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

In the Confidential Key Derivation (CKD) protocol, the coordinator blindly aggregates `(Y_i, C_i)` values received from participants without any cryptographic verification that each `C_i` was computed using the participant's legitimate key share. This is the direct analog of the Serum v4 missing vault ownership check: just as that program accepted vault accounts without verifying they were owned by `spl_token::ID`, this protocol accepts participant CKD shares without verifying they are "owned by" (i.e., bound to) the participant's legitimate secret share.

---

### Finding Description

In `do_ckd_coordinator` (`src/confidential_key_derivation/protocol.rs`, lines 35–57), the coordinator:

1. Computes its own share `(norm_big_y, norm_big_c)` correctly using `compute_signature_share`.
2. Calls `recv_from_others` to collect `CKDOutput` values from all other participants.
3. **Unconditionally adds** each received `(big_y, big_c)` into the running sum with no validation. [1](#0-0) 

The protocol specification requires each participant `i` to send:
- `Y_i = y_i * G1` (fresh randomness)
- `C_i = x_i * H(pk, app_id) + y_i * A` (key share contribution masked by app public key)

The coordinator then sums these to reconstruct `C = msk * H(pk, app_id) + a * Y`, which the app unmasks to recover the BLS signature `msk * H(pk, app_id)`. [2](#0-1) 

**The missing check:** There is no DLEQ proof or any other binding that forces `C_i` to use the participant's legitimate key share `x_i`. The coordinator has no way to verify this because `KeygenOutput` stores only the aggregate master public key, not individual verification shares `vk_i = x_i * G2`. [3](#0-2) 

A malicious participant can send an arbitrary `C_i' = r * G1` for any scalar `r`, substituting a value unrelated to their key share. The coordinator sums it in without complaint. [4](#0-3) 

---

### Impact Explanation

The final `CKDOutput` produced by the coordinator is:

```
C = Σ λ_i * C_i
Y = Σ λ_i * Y_i
```

If one participant substitutes `C_i'` for the correct `C_i`, the aggregate `C` deviates from `msk * H(pk, app_id) + a * Y`. The app computes `s = C - a * Y`, which will not equal `msk * H(pk, app_id)`. The derived confidential key `s` is therefore wrong and attacker-influenced.

This maps directly to the allowed impact: **High — Corruption of CKD outputs so honest parties accept inconsistent or unusable cryptographic outputs.** The TEE application receives a silently corrupted key derivation result. There is no error returned; the protocol completes successfully with a wrong answer.

---

### Likelihood Explanation

Any single participant in the CKD protocol can trigger this. The attacker only needs to be one of the `n` MPC nodes. The attack requires no leaked keys, no external assumptions, and no cryptographic breaks — only the ability to send an arbitrary `G1` point instead of the correct `C_i`. The entry path is the standard `ckd()` public API call with a malicious `key_pair` or a malicious implementation of the participant role. [5](#0-4) 

---

### Recommendation

Each participant should attach a DLEQ (Discrete Log Equality) proof demonstrating that the same secret `x_i` used in `S_i = x_i * H(pk, app_id)` corresponds to the participant's known verification share `vk_i = x_i * G2`. This requires:

1. Storing individual verification shares `vk_i` in `KeygenOutput` (or making them derivable from the DKG commitment output).
2. Each participant sending `(Y_i, C_i, π_i)` where `π_i` is a DLEQ proof over `(G2, vk_i, H(pk, app_id), S_i)`.
3. The coordinator verifying each `π_i` before adding `(Y_i, C_i)` to the aggregate.

---

### Proof of Concept

```
Setup: 3 participants P1, P2 (honest), P3 (malicious). Coordinator = P1.

1. P1 and P2 compute correct (Y_i, C_i) using compute_signature_share().
2. P3 instead sends (Y_3' = r*G1, C_3' = s*G1) for arbitrary scalars r, s.
3. do_ckd_coordinator receives P2's and P3's outputs via recv_from_others.
4. Coordinator sums: norm_big_y += Y_2 + Y_3', norm_big_c += C_2 + C_3'.
5. No error is raised. CKDOutput is returned with corrupted (Y, C).
6. App computes confidential_key = C - a*Y ≠ msk * H(pk, app_id).
7. The derived key is wrong and attacker-influenced; the app silently uses it.
```

The `do_ckd_coordinator` function has no call to any proof verification between `recv_from_others` and the summation loop, confirming the check is entirely absent. [6](#0-5)

    mut chan: SharedChannel,
    participants: ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<CKDOutputOption, ProtocolError> {
    let (mut norm_big_y, mut norm_big_c) =
        compute_signature_share(&participants, me, key_pair, app_id, app_pk, rng)?;

    // Receive everyone's inputs and add them together
    let waitpoint = chan.next_waitpoint();

    for (_, participant_output) in
        recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
    {
        norm_big_y += participant_output.big_y();
        norm_big_c += participant_output.big_c();
    }
    let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
    Ok(Some(ckd_output))

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
