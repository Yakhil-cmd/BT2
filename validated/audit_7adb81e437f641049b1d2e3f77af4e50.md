### Title
`MsgTurnBridge` Handler Has No Permission Check and Is a No-Op, Permanently Disabling the Bridge Circuit Breaker - (File: `x/cronos/keeper/msg_server.go`)

### Summary
The `TurnBridge` message handler in `x/cronos/keeper/msg_server.go` is implemented as a bare `return nil, nil`. It performs no permission check and makes no state change. Any unprivileged user can submit `MsgTurnBridge` and receive a success response, while the bridge state is never actually modified. The emergency circuit breaker for the gravity bridge is permanently non-functional.

### Finding Description
The Cronos permission system defines `CanTurnBridge` as a dedicated permission bit and the ADR-009 design explicitly states that `MsgTurnBridge` must be restricted to addresses holding that permission (or the admin). The `HasPermission` helper exists precisely for this purpose and is correctly used in `UpdateTokenMapping`.

However, the actual `TurnBridge` handler is:

```go
// TurnBridge implements the grpc method
func (k msgServer) TurnBridge(goCtx context.Context, msg *types.MsgTurnBridge) (*types.MsgTurnBridgeResponse, error) {
    return nil, nil
}
```

Two distinct failures are present simultaneously:

1. **No authorization check**: The handler never calls `HasPermission(ctx, msg.GetSigners(), CanTurnBridge)`. Any address can submit `MsgTurnBridge` and receive a success response.
2. **No state mutation**: The handler never reads or writes the bridge-enabled state. The `enable` field of `MsgTurnBridge` is silently discarded. The bridge can never be turned off (or on) through this message.

Compare with `UpdateTokenMapping`, which correctly gates on `HasPermission`:

```go
if !k.HasPermission(ctx, msg.GetSigners(), CanChangeTokenMapping) {
    return nil, errors.Wrap(sdkerrors.ErrUnauthorized, "msg sender is not authorized")
}
```

`TurnBridge` has no equivalent guard.

### Impact Explanation
**High — Bypass of Cronos bridge authorization check + permanent inability to disable bridge flows.**

- Any unprivileged user can broadcast `MsgTurnBridge` with `enable: false` and receive a success receipt, creating a false impression that the bridge has been disabled.
- The admin and governance can never actually disable the gravity bridge through the intended `MsgTurnBridge` path. In an emergency (e.g., a bridge exploit draining funds), the circuit breaker is inoperative, leaving bridge conversion flows permanently active regardless of admin intent.
- This maps directly to the allowed High impacts: "Bypass of Cronos admin, governance authority, permission, token-mapping, bridge, block-list, or module-account authorization checks" and "Permanent or long-lived inability for honest users or validators to process valid transactions, bridge/conversion flows."

### Likelihood Explanation
The message is registered in the live proto service, has a CLI command (`turn-bridge`), and is reachable by any unprivileged user submitting a standard Cosmos SDK transaction. No special access is required. The broken state is permanent until a code fix is deployed.

### Recommendation
Implement `TurnBridge` with the same pattern used by `UpdateTokenMapping`:

```go
func (k msgServer) TurnBridge(goCtx context.Context, msg *types.MsgTurnBridge) (*types.MsgTurnBridgeResponse, error) {
    ctx := sdk.UnwrapSDKContext(goCtx)

    senderAddr, err := sdk.AccAddressFromBech32(msg.Sender)
    if err != nil {
        return nil, err
    }
    if !k.HasPermission(ctx, []sdk.AccAddress{senderAddr}, CanTurnBridge) {
        return nil, errors.Wrap(sdkerrors.ErrUnauthorized, "msg sender is not authorized")
    }

    params := k.GetParams(ctx)
    params.EnableAutoDeployment = msg.Enable  // or a dedicated BridgeActive field
    if err := k.SetParams(ctx, params); err != nil {
        return nil, err
    }
    return &types.MsgTurnBridgeResponse{}, nil
}
```

The exact state field to toggle should match whatever field the bridge-active check reads in the end-blocker / IBC hooks.

### Proof of Concept

1. Any unprivileged address `alice` broadcasts:
   ```
   cronosd tx cronos turn-bridge false --from alice
   ```
2. The transaction is accepted on-chain and returns code `0` (success).
3. Query the bridge-active parameter — it is unchanged; the bridge continues processing.
4. The admin broadcasts the same message — also returns code `0` but the bridge is still not disabled.
5. Confirm: `HasPermission` is never called inside `TurnBridge` (line 85–87 of `x/cronos/keeper/msg_server.go`), and no keeper state is written.
