Audit Report

## Title
Unbounded `active_assembles` / `artifact_processor_tasks` Growth via Malicious Peer Slot Updates — (`rs/p2p/consensus_manager/src/receiver.rs`)

## Summary
`handle_slot_update_receive` in `ConsensusManagerReceiver` inserts a new entry into `active_assembles` and spawns a new Tokio task for every unique artifact ID received from a peer, gated only by a per-peer slot-table limit. For consensus, certification, DKG, iDKG, and HTTPS-outcalls artifact types, that limit is `usize::MAX` (`SLOT_TABLE_NO_LIMIT`), making the guard effectively absent. A single authenticated Byzantine subnet peer can therefore drive unbounded memory and task growth on every receiving node until it is OOM-killed or becomes unresponsive.

## Finding Description
**Root cause — slot limit configuration** (`rs/replica/setup_ic_network/src/lib.rs`, lines 72–75):

```rust
/// This limit is used to protect against a malicious peer advertising many ingress messages.
const SLOT_TABLE_LIMIT_INGRESS: usize = 50_000;
const SLOT_TABLE_NO_LIMIT: usize = usize::MAX;
``` [1](#0-0) 

Only ingress uses the bounded constant (line 257); consensus (lines 237, 246), certifier (line 268), DKG (line 279), iDKG (line 291), and HTTPS-outcalls (line 303) all pass `SLOT_TABLE_NO_LIMIT`. [2](#0-1) [3](#0-2) 

**Slot-limit guard** (`rs/p2p/consensus_manager/src/receiver.rs`, line 373):

```rust
Entry::Vacant(empty_slot) if peer_slot_table_len < self.slot_limit => {
``` [4](#0-3) 

With `self.slot_limit = usize::MAX`, `peer_slot_table_len < usize::MAX` is always true, so every new slot number from a peer is accepted unconditionally.

**Unbounded insertion and task spawn** (lines 399–420): for each accepted slot whose artifact ID is not already in `active_assembles`, a new `watch` channel is created, the ID is inserted into the global map, and a new async task is spawned into `artifact_processor_tasks`. [5](#0-4) 

**Tasks never terminate for malicious peers** (lines 518–521): the `AssembleResult::Unwanted` branch blocks on `peer_rx.wait_for(|p| p.is_empty())`, which resolves only when the advertising peer removes the ID from its slot table. A malicious peer simply never does so. [6](#0-5) 

**Exploit flow:**
1. Attacker (authenticated subnet peer, below fault threshold) opens a QUIC connection to a victim node.
2. Attacker sends `SlotUpdate { slot_id: N, commit_id: N, update: Update::Id(<unique_id_N>) }` for monotonically increasing `N`.
3. Each message passes the slot-limit check (always true), misses `active_assembles`, inserts a new entry, and spawns a new task.
4. Tasks accumulate indefinitely; the malicious peer never removes IDs, so no task ever exits via the `all_peers_deleted_artifact` path either.
5. Victim node exhausts heap memory and is OOM-killed or becomes unresponsive.

## Impact Explanation
This is a **High** severity finding matching the allowed impact: *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."* Crashing one or more replica nodes on a small subnet (e.g., 13 nodes, f=4) can drop the subnet below the honest-majority threshold required for continued consensus, causing a subnet availability outage. The attack is not volumetric — it exploits a logical unboundedness in the data structure, not raw bandwidth.

## Likelihood Explanation
The attacker must be an authenticated subnet member (TLS/QUIC peer identity is verified by the transport layer). Joining a subnet requires an NNS governance proposal, but the protocol explicitly tolerates up to f Byzantine nodes. The attack requires no cryptographic material beyond the node's own TLS key, is trivially scriptable with a loop over monotonically increasing slot IDs, and is repeatable indefinitely. The ingress artifact type already has an explicit 50,000-slot cap with the exact same rationale stated in the comment at line 72, confirming the protection mechanism is understood and intentional — its absence for other artifact types is an oversight. [1](#0-0) 

## Recommendation
Pass a finite `slot_limit` instead of `SLOT_TABLE_NO_LIMIT` when registering consensus, certification, DKG, iDKG, and HTTPS-outcalls channels in `AbortableBroadcastChannels::new`. The bound should reflect the maximum number of in-flight artifacts a legitimate node would ever advertise simultaneously for each type (e.g., a few hundred for consensus, a few thousand for iDKG). The existing `SLOT_TABLE_LIMIT_INGRESS` mechanism already works correctly — the `slot_table_limit_exceeded` unit test at lines 846–887 confirms it. [7](#0-6) 

As defense-in-depth, consider adding a global cap on `active_assembles.len()` since even a per-peer slot limit still allows `N_peers × slot_limit` total entries.

## Proof of Concept
The existing `slot_table_limit_exceeded` unit test (lines 846–887) already demonstrates the guard works when `slot_limit = 2`. The same test run with `slot_limit = usize::MAX` (i.e., `SLOT_TABLE_NO_LIMIT`) would show that the third `handle_slot_update_receive` call is never rejected and `active_assembles.len()` grows to 3, confirming the unbounded growth. A fuzz/stress test can be written as:

```rust
#[tokio::test]
async fn unbounded_growth_no_limit() {
    let (mut mgr, _channels) = ReceiverManagerBuilder::new()
        .with_slot_limit(usize::MAX)
        .build();
    let cancellation = CancellationToken::new();
    for i in 0..10_000u64 {
        mgr.handle_slot_update_receive(
            SlotUpdate {
                slot_number: SlotNumber::from(i),
                commit_id: CommitId::from(i),
                update: Update::Id(i),
            },
            NODE_1,
            ConnId::from(1),
            cancellation.clone(),
        );
    }
    // With SLOT_TABLE_NO_LIMIT, all 10_000 entries are accepted:
    assert_eq!(mgr.active_assembles.len(), 10_000);
    assert_eq!(mgr.artifact_processor_tasks.len(), 10_000);
}
``` [7](#0-6)
