### Title
Silent `u32`→`u16` Truncation of `tx_number_in_block` in L2→L1 Log Encoding — (`zk_ee/src/common_structs/logs_storage.rs`)

---

### Summary

The `From<&LogContent<A>> for L2ToL1Log` implementation in `logs_storage.rs` silently truncates the internally-stored `tx_number: u32` to `u16` when constructing the on-chain `L2ToL1Log` structure. If a block contains more than 65,535 transactions, every log emitted by transactions numbered ≥ 65,536 will carry a wrong `tx_number_in_block`, corrupting the L2→L1 log Merkle root committed to L1.

---

### Finding Description

`GenericLogContent` stores the transaction number as `u32`: [1](#0-0) 

```rust
pub struct GenericLogContent<IOTypes: SystemIOTypesConfig, A: Allocator = Global> {
    pub tx_number: u32,
    ...
}
```

`L2ToL1Log`, the structure that is serialised and hashed into the Merkle tree committed to L1, stores the same field as `u16`: [2](#0-1) 

```rust
pub struct L2ToL1Log {
    ...
    pub tx_number_in_block: u16,
    ...
}
```

The conversion between the two types performs an unchecked narrowing cast: [3](#0-2) 

```rust
Self {
    l2_shard_id: 0,
    is_service: true,
    tx_number_in_block: m.tx_number as u16,   // ← silent truncation
    sender,
    key,
    value,
}
```

There is no bounds check, no `u16::try_from`, and no assertion that `m.tx_number ≤ u16::MAX` anywhere in the call chain. The `push_message` and `push_l1_l2_tx_log` functions accept `tx_number: u32` without restriction: [4](#0-3) 

The serialised encoding of `L2ToL1Log` writes `tx_number_in_block` directly into the 88-byte log buffer that feeds the Merkle tree: [5](#0-4) 

```rust
buffer[2..4].copy_from_slice(&self.tx_number_in_block.to_be_bytes());
```

---

### Impact Explanation

`tx_number_in_block` is a field in every L2→L1 log leaf. These leaves are hashed into the L2→L1 log Merkle tree whose root is committed to L1 as part of the block's public input. If `tx_number` overflows `u16`, the leaf hash is wrong, the Merkle root is wrong, and any L1 contract or off-chain verifier that reads `tx_number_in_block` from a log (e.g., to prove which transaction triggered a withdrawal or an L1 message) receives a silently incorrect value. Specifically, transaction 65,536 would appear as transaction 0, transaction 65,537 as transaction 1, etc., making it impossible to distinguish logs from different transactions that alias to the same `tx_number_in_block`.

---

### Likelihood Explanation

The `tx_number` counter is incremented by the bootloader for every transaction processed in a block. The internal type is `u32`, which imposes no practical ceiling below ~4 billion. The only soft limit visible in the file is `MAX_NUMBER_OF_LOGS: u64 = 16_384`, which caps *logs*, not transactions: [6](#0-5) 

A block with more than 65,535 transactions — achievable if block gas limits or sequencer batch sizes are raised — would silently produce incorrect log metadata. Because the sequencer controls block composition, an unprivileged user cannot force this condition alone, but the absence of any guard means the bug activates automatically once the threshold is crossed, with no error or panic.

---

### Recommendation

Replace the silent cast with a checked conversion and propagate the error:

```rust
// In From<&LogContent<A>> for L2ToL1Log
tx_number_in_block: u16::try_from(m.tx_number)
    .expect("tx_number exceeds u16::MAX; block contains too many transactions"),
```

Or, if the protocol guarantees `tx_number ≤ u16::MAX`, change the field type in `GenericLogContent` from `u32` to `u16` to make the invariant explicit and eliminate the widening/narrowing round-trip.

---

### Proof of Concept

1. Construct a block containing 65,537 transactions, each emitting one L2→L1 log.
2. Transaction 0 emits log with `tx_number = 0` → `tx_number_in_block = 0` ✓
3. Transaction 65,536 emits log with `tx_number = 65536` → `65536u32 as u16 = 0` → `tx_number_in_block = 0` ✗
4. Both logs hash to the same leaf value for the `tx_number_in_block` field, making them indistinguishable on L1 and producing a Merkle root that does not faithfully represent the block's transaction ordering.
