After thorough investigation of the codebase, I found a valid analog to the Curve DOS bug. The Curve bug's core invariant is: a partial/incomplete state (one token non-zero in an empty pool) causes a critical function to behave incorrectly. The sequencer analog is in the gateway's `skip_stateful_validations` logic: a partial deployment state (deploy_account in mempool, account not yet on-chain) causes the gateway to skip signature verification for a subsequent invoke, allowing any attacker to inject signature-unverified transactions into the mempool.

---

### Title
Gateway Admits Signature-Unverified Invoke Transactions via `skip_stateful_validations` When a Deploy-Account Is In-Flight — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator skips the blockifier `__validate__` entry-point call (i.e., skips signature verification) for any Invoke transaction with `nonce=1` whenever `account_tx_in_pool_or_recent_block` returns `true` for the sender address. Because `account_tx_in_pool_or_recent_block` returns `true` for **any** account that has **any** transaction in the pool — not specifically a deploy-account — an unprivileged attacker who observes a victim's deploy-account transaction in the mempool can immediately inject an Invoke with `nonce=1` carrying an arbitrary (invalid) signature. The gateway admits it without calling `__validate__`, so the invalid transaction enters the mempool and occupies the `nonce=1` slot for that account.

### Finding Description

`skip_stateful_validations` is the gateway-side UX shortcut that allows a user to submit `deploy_account + invoke(nonce=1)` atomically without waiting for the deploy to be mined:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs  lines 429-460
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            // "It is sufficient to check if the account exists in the mempool since it means
            //  that either it has a deploy_account transaction or transactions with future
            //  nonces that passed validations."
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                ...;
        }
    }
    Ok(false)
}

The guard `account_tx_in_pool_or_recent_block` is implemented as:

```rust
// crates/apollo_mempool/src/mempool.rs  lines 697-700
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [4](#0-3) 

`state.contains_account` returns `true` if the address appears in either the `staged` or `committed` nonce maps — i.e., for **any** account that has ever had a transaction in the pool, not exclusively one that currently holds a deploy-account. The comment's assumption ("either it has a deploy_account transaction or transactions with future nonces that passed validations") is therefore incorrect: the check cannot distinguish a legitimate deploy-account from any other transaction type.

**Attack path (no privilege required):**

1. Victim broadcasts `deploy_account` for address `A`. It passes gateway validation (including `__validate_deploy__`) and enters the mempool.
2. Attacker observes the mempool (public), sees `deploy_account` for `A`.
3. Attacker crafts `invoke(sender=A, nonce=1, signature=<garbage>)` and submits it to the gateway.
4. Gateway stateless checks pass (valid resource bounds, correct DA mode, etc.).
5. `extract_state_nonce_and_run_validations` reads `account_nonce = 0` from state.
6. `validate_by_mempool` passes: nonce `1 >= 0`, no duplicate hash.
7. `skip_stateful_validations`: `nonce==1 && account_nonce==0 && account_tx_in_pool_or_recent_block(A)==true` → returns `true`.
8. `run_validate_entry_point` sets `validate=false` → `__validate__` is **never called**; the garbage-signature invoke is admitted to the mempool.
9. Victim submits their legitimate `invoke(nonce=1)`. Mempool rejects it: `DuplicateNonce`.
10. Batcher eventually executes the attacker's invoke; `__validate__` fails; the slot is freed.
11. Attacker immediately repeats from step 3, continuously occupying the `nonce=1` slot.

### Impact Explanation

The gateway is the admission gate for the mempool. Admitting a transaction with an unverified signature violates the invariant that every transaction in the mempool has passed signature/account validation. Concretely:

- **Mempool slot squatting**: The attacker occupies the `nonce=1` slot for any account that is mid-deployment, preventing the legitimate post-deploy invoke from entering the mempool until the batcher rejects the attacker's transaction.
- **Repeated DOS**: Because the batcher's rejection frees the slot and the gateway will re-admit the next garbage invoke immediately, the attacker can sustain the block indefinitely at negligible cost (only gas for the deploy-account observation; the attacker's invokes are never executed on-chain and pay no fee since `__validate__` fails before fee deduction).
- **Matches impact scope**: "Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing." (High)

### Likelihood Explanation

- The mempool is public; any observer can detect a pending deploy-account.
- The attacker needs only to submit a well-formed RPC invoke (valid resource bounds, correct DA mode) with any signature value — no cryptographic capability required.
- The race window is the entire time the deploy-account sits unconfirmed in the mempool (potentially many blocks).
- No privileged access, no special network position, no dependency on external contracts.

### Recommendation

Replace the overly broad `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a **deploy-account** transaction is pending for the sender address. The mempool should expose a dedicated `has_pending_deploy_account(address)` query, or the gateway should inspect the transaction type of the pooled transaction before granting the validation skip.

Alternatively, always run `__validate__` at the gateway for invoke transactions, accepting the UX trade-off that the first post-deploy invoke may fail validation until the deploy-account is confirmed.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
