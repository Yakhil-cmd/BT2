### Title
Unchecked Precompile Call Return Value After `_burn` Enables Permanent ERC-20 Token Loss Without NEP-141 Transfer - (`etc/eth-contracts/contracts/EvmErc20.sol` / `EvmErc20V2.sol`)

---

### Summary

In `EvmErc20.sol` and `EvmErc20V2.sol`, both `withdrawToNear()` and `withdrawToEthereum()` first call `_burn()` to destroy the caller's ERC-20 tokens, then invoke the Aurora exit precompile via inline assembly. The return value of the assembly `call` is captured in a local variable `res` but is **never checked**. If the precompile call fails for any reason (e.g., invalid NEAR recipient account ID, precompile input parsing error), the ERC-20 tokens are permanently destroyed while no corresponding NEP-141 transfer is scheduled. The function returns successfully, leaving the bridge accounting in an irrecoverably inconsistent state.

---

### Finding Description

In `EvmErc20.withdrawToNear()`:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);                          // ERC-20 tokens destroyed

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        // res is NEVER checked — silent failure is possible
    }
}
```

The same pattern appears in `withdrawToEthereum()` (both `EvmErc20` and `EvmErc20V2`).

The `ExitToNear` precompile (`engine-precompiles/src/native.rs`) can return `ExitError` — causing the EVM `call` opcode to return `0` — in several cases reachable without admin privileges:

- `parse_recipient()` fails when `recipient` bytes are not valid UTF-8 or not a valid NEAR account ID (`ERR_INVALID_RECEIVER_ACCOUNT_ID`).
- `get_nep141_from_erc20()` fails if the ERC-20 → NEP-141 mapping is absent (`ERR_TARGET_TOKEN_NOT_FOUND`).
- Input size validation fails (`validate_input_size`).

When any of these occur, `res = 0` is silently discarded. The `_burn` is not reverted. No NEAR promise is scheduled. The `error_refund` callback mechanism (`exit_to_near_precompile_callback`) is never reached because it only handles failures of the downstream NEAR promise, not failures of the precompile call itself.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

A user's ERC-20 tokens (representing bridged NEP-141 assets) are irreversibly burned. The corresponding NEP-141 tokens remain locked in the Aurora engine contract with no mechanism to recover them. The bridge accounting is permanently inconsistent: the ERC-20 total supply decreases but the NEP-141 balance held by Aurora does not decrease. The user receives nothing on either side.

---

### Likelihood Explanation

**Medium.** Any user who calls `withdrawToNear()` with a `recipient` value that fails NEAR account ID validation (e.e., non-UTF-8 bytes, empty string, account ID exceeding 64 characters, or containing invalid characters) will trigger this silently. This is reachable by any EVM user interacting with the bridge ERC-20 contract directly (e.g., via Etherscan, a custom script, or a buggy frontend). No admin compromise is required.

---

### Recommendation

Check the return value of the assembly `call` and revert if it is zero:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

This must be applied to all four affected functions: `withdrawToNear` and `withdrawToEthereum` in both `EvmErc20.sol` and `EvmErc20V2.sol`. The `_burn` call should either be moved to after a successful precompile call, or the revert-on-failure pattern above must be used so that the EVM atomically rolls back the burn if the precompile fails.

### Proof of Concept

**Root cause — unchecked return value in `EvmErc20.sol`:** [1](#0-0) 

**Same pattern in `EvmErc20V2.sol`:** [2](#0-1) 

**`ExitToNear` precompile returns `ExitError` on invalid recipient — causes `call` to return 0:** [3](#0-2) 

**`error_refund` callback only handles downstream NEAR promise failure, not precompile call failure:** [4](#0-3) 

**Attack path:**
1. Attacker (or any user) holds bridged ERC-20 tokens on Aurora.
2. Calls `EvmErc20.withdrawToNear(invalid_near_bytes, amount)` where `invalid_near_bytes` fails `parse_recipient` in the precompile.
3. `_burn(msg.sender, amount)` executes — ERC-20 balance decremented permanently.
4. Assembly `call` to precompile returns `0` (failure); `res` is never checked.
5. Function returns `success`. No NEAR promise is created. No `exit_to_near_precompile_callback` is scheduled.
6. ERC-20 tokens are gone; NEP-141 tokens remain locked in Aurora engine. Funds are permanently frozen.
