### Title
Pool Admin Bypasses `maxAdminSpreadFeeE6` Cap via Uncapped `setPoolBinAdditionalFees` â€” (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`setPoolBinAdditionalFees` imposes no upper-bound check on `addFeeBuyE6` / `addFeeSellE6`, while the parallel `setPoolAdminFees` path is explicitly capped at `maxAdminSpreadFeeE6`. A pool admin can set per-bin additional fees up to `uint16` max (65 535, â‰ˆ 6.55 % in E6 units) on every bin, directly inflating the effective fee applied in swap math and causing traders to overpay with no factory-level guard.

---

### Finding Description

`setPoolAdminFees` enforces the factory cap: [1](#0-0) 

```solidity
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
```

`setPoolBinAdditionalFees` has no equivalent check â€” it passes the caller-supplied values straight through: [2](#0-1) 

`setBinAdditionalFees` on the pool only validates the bin index: [3](#0-2) 

In every swap path the per-bin fee is added directly to `baseFeeX64` before the bin math executes: [4](#0-3) 

```solidity
params.baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6)
```

The same uncapped addition appears in all four swap variants (specified-in/out Ã— buy-token0/buy-token1). [5](#0-4) [6](#0-5) [7](#0-6) 

The factory hard cap is `HARD_MAX_SPREAD_FEE_E6 = 200_000` (20 %): [8](#0-7) 

`uint16` max is 65 535 â‰ˆ 6.55 %. A pool admin can set this on every bin, adding up to 6.55 % on top of the oracle-derived `baseFeeX64` and the global `spreadFeeE6`, with no factory-level rejection.

---

### Impact Explanation

Every swap through an affected bin pays `baseFeeX64 + addFeeBuyE6 * ONE_X64 / 1e6` as the effective spread. With `addFeeBuyE6 = 65535` the trader receives fewer output tokens than the oracle spread alone would dictate â€” a direct, per-swap loss of principal. The pool admin controls this parameter with no cap, no timelock, and no protocol override path for per-bin fees.

---

### Likelihood Explanation

The pool admin is a semi-trusted role that is explicitly expected to be constrained by factory caps. The bypass requires only a single call to `setPoolBinAdditionalFees` with `addFeeBuyE6 = 65535`. No special conditions, no timelock, no co-signer. Any pool whose admin is compromised, malicious, or simply misconfigured can exploit this immediately.

---

### Recommendation

Add a cap check in `setPoolBinAdditionalFees` (and propagate it through `setBinAdditionalFees`) analogous to the check in `setPoolAdminFees`:

```solidity
function setPoolBinAdditionalFees(
    address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6
) external override nonReentrant onlyPoolAdmin(pool) {
    if (addFeeBuyE6  > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

The same guard should be applied to the initial bin fees supplied at pool creation time via `_unpackAndValidateBinStates`.

---

### Proof of Concept
