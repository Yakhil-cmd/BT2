### Title
Unprivileged Caller Can Block Reward Distribution via `update_rewards(disable_rewards: true)` - (`src/staking/staking.cairo`)

---

### Summary

The `update_rewards` function in the Staking contract writes to the global `last_reward_block` storage variable **before** checking the `disable_rewards` flag. Because the function has no access control beyond the basic pause/non-zero-caller check, any unprivileged caller can invoke it with `disable_rewards: true` to consume the block's reward slot without distributing any rewards. A subsequent legitimate call in the same block will revert with `REWARDS_ALREADY_UPDATED`, causing all stakers to lose rewards for that block.

---

### Finding Description

`update_rewards` is publicly callable (embedded in the ABI via `#[abi(embed_v0)]`) and its only gate is `general_prerequisites()`, which checks only that the contract is unpaused and the caller is non-zero. [1](#0-0) 

Inside the function, `last_reward_block` is written **before** the early-return guard on `disable_rewards`: [2](#0-1) 

`general_prerequisites` enforces only pause state and non-zero caller — no role check: [3](#0-2) 

The invariant that is violated: **`last_reward_block` should only advance when rewards are actually distributed**. Because the write happens unconditionally before the `disable_rewards` branch, an attacker can advance `last_reward_block` to the current block without triggering any reward calculation or transfer.

---

### Impact Explanation

`last_reward_block` is a **single global variable** shared across all stakers: [4](#0-3) 

When an attacker calls `update_rewards(any_active_staker, disable_rewards: true)` in block N, `last_reward_block` becomes N. Every subsequent call in block N — regardless of which staker is targeted — will fail the assertion:

```
assert!(current_block_number > self.last_reward_block.read(), ...)
```

This means **all stakers** lose their block rewards for block N. Repeated every block, this permanently freezes unclaimed yield across the entire protocol. This matches the allowed impact: **Griefing with no profit motive but damage to users or protocol** (Medium), escalating toward **Permanent freezing of unclaimed yield** (High) if sustained.

---

### Likelihood Explanation

The attack requires no special privilege — only a non-zero address and enough gas to submit one transaction per block. The attacker gains nothing financially, but the cost to execute is low (a single cheap call per block). The attack is realistic for a motivated griever or a competitor seeking to damage the protocol's reputation.

---

### Recommendation

Move the `last_reward_block.write(current_block_number)` to **after** the `disable_rewards` guard, so the slot is only consumed when rewards are actually distributed:

```cairo
// BEFORE (vulnerable):
self.last_reward_block.write(current_block_number);
if disable_rewards || self.is_pre_consensus() {
    return;
}

// AFTER (fixed):
if disable_rewards || self.is_pre_consensus() {
    return;
}
self.last_reward_block.write(current_block_number);
```

Alternatively, restrict `update_rewards` to a privileged role (e.g., only callable by the staker themselves or a designated operator).

---

### Proof of Concept

1. Staker Alice is active with non-zero STRK balance in the current epoch.
2. Attacker (any address) calls `update_rewards(alice_address, disable_rewards: true)` in block N.
   - `general_prerequisites()` passes (contract unpaused, caller non-zero).
   - `current_block_number (N) > last_reward_block` passes.
   - `last_reward_block` is written to N.
   - Function returns early — **no rewards distributed**.
3. Alice (or anyone) calls `update_rewards(alice_address, disable_rewards: false)` in block N.
   - `current_block_number (N) > last_reward_block (N)` → **false** → reverts with `REWARDS_ALREADY_UPDATED`.
4. Alice receives zero rewards for block N.
5. Attacker repeats step 2 every block → Alice (and all other stakers) permanently receive zero rewards.
