### Title
Swap callback overpayment silently absorbed by pool with no corresponding output — (`metric-core/contracts/MetricOmmPool.sol`)

### Summary

The post-callback balance check in `MetricOmmPool.swap` uses a one-sided `>` comparison. It reverts only when the pool receives **less** than the computed `amountDelta`, but silently accepts any **excess** tokens sent by the callback. Those extra tokens are permanently absorbed into the pool with no corresponding output token returned to the user.

### Finding Description

In `MetricOmmPool.swap`, after `_executeSwap` computes the exact `amount0Delta`/`amount1Delta` and updates all bin accounting, the pool transfers output tokens to `recipient`, then calls `metricOmmSwapCallback` to collect the input leg. The post-callback guard is:

```solidity
// zeroForOne branch
if (amount0Delta > 0 && balance0Before + uint256(amount0Delta) > balance0()) {
    revert IncorrectDelta();
}
```

```solidity
// !zeroForOne branch
if (amount1Delta > 0 && balance1Before + uint256(amount1Delta) > balance1()) {
    revert IncorrectDelta();
}
``` [1](#0-0) 

The condition `balance0Before + uint256(amount0Delta) > balance0()` is `false` (no revert) whenever `balance0() >= balance0Before + uint256(amount0Delta)`. This means any callback that transfers **more** than `amount0Delta` (or `amount1Delta`) passes the guard without error.

Because `_executeSwap` already finalized `binTotals` before the callback is invoked, the extra tokens are not credited to any bin and are not returned to the caller. They become permanently stranded in the pool's ERC-20 balance, unaccounted for by the bin ledger. [2](#0-1) 

### Impact Explanation

A user whose callback implementation (third-party router, wrapper contract, or buggy integration) sends even one wei more than the required input amount will:

1. Lose the excess tokens permanently — they are absorbed into the pool's raw ERC-20 balance.
2. Receive no additional output tokens, because the output transfer (`transferToken0`/`transferToken1`) already executed before the callback and is fixed to the pre-computed delta.
3. Have no on-chain recourse; the pool emits a `Swap` event with the correct deltas, masking the discrepancy.

The stranded tokens inflate `balance0()`/`balance1()` relative to `binTotals`, which can also distort the bin-boundary guard logic that uses `balance0()`/`balance1()` to cap available output in exact-output swaps. [3](#0-2) 

### Likelihood Explanation

Any `IMetricOmmSwapCallback` implementation that does not compute the exact transfer amount — for example, a router that rounds up, a wrapper that adds a dust buffer, or a third-party integration that re-reads its own balance and transfers the full available amount — will trigger this silently. The `MetricOmmSimpleRouter` itself is correct, but the pool's interface is public and callable by arbitrary contracts. [4](#0-3) 

### Recommendation

Change both post-callback guards from a one-sided `>` to a strict equality check (`!=`), so any deviation — underpayment **or** overpayment — reverts:

```solidity
// zeroForOne branch — replace line 261
if (amount0Delta > 0 && balance0Before + uint256(amount0Delta) != balance0()) {
    revert IncorrectDelta();
}

// !zeroForOne branch — replace line 275
if (amount1Delta > 0 && balance1Before + uint256(amount1Delta) != balance1()) {
    revert IncorrectDelta();
}
``` [5](#0-4) [6](#0-5) 

### Proof of Concept

Deploy a malicious/buggy callback that sends `amount0Delta + 1e9` instead of exactly `amount0Delta`:

```solidity
function metricOmmSwapCallback(int256 amount0Delta, int256 amount1Delta, bytes calldata data) external override {
    address sender = abi.decode(data, (address));
    if (amount0Delta > 0) {
        // sends MORE than required
        IERC20(IMetricOmmPool(msg.sender).getImmutables().token0)
            .transferFrom(sender, msg.sender, uint256(amount0Delta) + 1e9);
    } else if (amount1Delta > 0) {
        IERC20(IMetricOmmPool(msg.sender).getImmutables().token1)
            .transferFrom(sender, msg.sender, uint256(amount1Delta) + 1e9);
    }
}
```

After the swap completes without revert:
- `token0.balanceOf(pool)` exceeds `balance0Before + amount0Delta` by `1e9`
- `token1.balanceOf(user)` is unchanged relative to the pre-swap value (no extra output was sent)
- The `1e9` excess is permanently locked in the pool, untracked by `binTotals` [7](#0-6)
