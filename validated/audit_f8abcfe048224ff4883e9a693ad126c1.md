### Title
Deserialization-Before-Sender-Check in `do_sign_participant` Allows Any Participant to Abort Honest Participant's Signing Session — (`src/frost/redjubjub/sign.rs`)

### Summary

In `do_sign_participant`, the `chan.recv(wait_round_1).await?` call fully deserializes the incoming message payload **before** the `from != coordinator` guard is evaluated. A malicious non-coordinator participant can inject a message at `wait_round_1` whose bytes parse as the correct header but whose payload fails `rmp_serde` deserialization as `Randomizer`. The `?` operator propagates the resulting `ProtocolError` immediately, permanently aborting the honest participant's signing session before the coordinator's legitimate `Randomizer` is ever processed.

### Finding Description

`do_sign_participant` in `src/frost/redjubjub/sign.rs` waits for the coordinator's `Randomizer` with this loop:

```rust
let randomizer = loop {
    let (from, randomizer): (_, Randomizer) = chan.recv(wait_round_1).await?;  // line 218
    if from != coordinator {   // line 219 — never reached on deser failure
        continue;
    }
    break randomizer;
};

`push_message` accepts any message whose bytes contain a valid `MessageHeader` prefix and routes it into the buffer keyed by that header:

Because the header format is deterministic and any protocol participant can observe it, a malicious participant can craft a message with the correct `wait_round_1` header but a garbage payload, deliver it via `protocol.message(malicious_participant, crafted_bytes)`, and cause `decoded?` to return `Err`. The `?` on line 218 of `sign.rs` propagates that error out of `do_sign_participant` before line 219's `from != coordinator` check is ever evaluated.

### Impact Explanation

The honest participant's signing session is permanently terminated with a `ProtocolError::DeserializationError` (or `ProtocolError::Other`). The participant cannot recover or retry within the same session. This constitutes **permanent denial of signing** for an honest party under valid protocol inputs, matching the High-severity impact category.

### Likelihood Explanation

Any participant in the signing set — not just the coordinator — can call `protocol.message(from, data)` on the victim's protocol instance. The `wait_round_1` header value is deterministic and predictable by any participant who has initialized their own signing protocol with the same parameters. No cryptographic material needs to be compromised; only a single malformed message delivery is required.

### Recommendation

Move the sender check **before** deserialization. The simplest fix is to handle the deserialization result inside the loop and `continue` on error rather than propagating it:

```rust
let randomizer = loop {
    let result: Result<(_, Randomizer), _> = chan.recv(wait_round_1).await;
    match result {
        Err(_) => continue,          // skip malformed messages from any sender
        Ok((from, r)) if from != coordinator => continue,
        Ok((_, r)) => break r,
    }
};
```

Alternatively, expose the raw `(from, MessageData)` from `recv` so the caller can check the sender before attempting deserialization.

### Proof of Concept

1. Initialize a signing session with participants `[coordinator, honest, malicious]`.
2. Before the coordinator calls `poke()` to emit the `Randomizer` broadcast, call `honest_protocol.message(malicious, crafted_bytes)` where `crafted_bytes` has the correct `wait_round_1` header prefix followed by random garbage.
3. Call `honest_protocol.poke()`.
4. Observe that `poke()` returns `Err(ProtocolError::DeserializationError(...))` or `Err(ProtocolError::Other(...))` — the session is aborted.
5. Even if the coordinator subsequently delivers the legitimate `Randomizer`, the honest participant's future has already returned an error and cannot be resumed. [4](#0-3) [5](#0-4)
