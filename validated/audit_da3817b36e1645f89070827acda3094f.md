### Title
`addLiquidity` Executes Without Pause Guard, Allowing Users to Deposit Into a Compromised Pool - (File: metric-core/contracts/MetricOmmPool.sol)

### Summary

`MetricOmmPool.addLiquidity` lacks the `whenNotPaused` modifier that gates `swap`. When a pool is paused at level 1 (admin) or level 2 (protocol) due to a security incident, any caller can still deposit tokens into the pool via `addLiquidity`, exposing newly deposited principal to whatever condition triggered the pause.

### Finding Description

`MetricOmmPool` defines a `whenNotPaused` modifier that reverts with `PoolPaused` when `pauseLevel != 0`: [1](#0-0) 

`swap` correctly applies this modifier: [2](#0-1) 

`addLiquidity` does not: [3](#0-2) 

The interface NatSpec acknowledges this asymmetry explicitly, framing it as intentional so LPs can "unwind while swaps are off": [4](#0-3) 

However, the same absence of a guard that permits `removeLiquidity` (the desired unwind path) also permits `addLiquidity` (the harmful deposit path). The two operations share no common gate, so the protocol cannot allow one without allowing the other under the current design.

`addLiquidity` in `LiquidityLib` immediately updates `binState.token0BalanceScaled` / `token1BalanceScaled` and `binTotals` before the callback, then pulls tokens from the caller: [5](#0-4) 

Tokens transferred in during a pause are immediately credited to bins and counted in `binTotals`, making them subject to whatever exploit or accounting failure caused the pause.

### Impact Explanation

A pool is paused (level 1 or 2) because a vulnerability is being exploited or a price provider is misbehaving. An unaware LP calls `addLiquidity` — the call succeeds, tokens are transferred into the pool and credited to bins. If the pool is actively being drained (the reason for the pause), the newly deposited tokens are drained alongside existing LP funds before the depositor can call `removeLiquidity`. The depositor suffers direct loss of principal with no recourse, because `swap` (the normal exit path) is blocked and `removeLiquidity` requires a separate transaction that may lose the race to the attacker.

### Likelihood Explanation

Pause events are rare but represent the highest-risk moments in a pool's lifecycle. The factory owner triggers protocol pause (`pauseLevel = 2`) from either active or admin-paused state: [6](#0-5) 

The pool admin triggers admin pause (`pauseLevel = 1`) from active state only: [7](#0-6) 

Neither pause level blocks `addLiquidity`. Any unprivileged user — including a bot or an attacker front-running the pause — can call `addLiquidity` during the window between the pause trigger and public awareness. The trigger for the vulnerability is fully unprivileged (any caller of `addLiquidity`).

### Recommendation

Apply `whenNotPaused` to `addLiquidity` so that deposits are blocked whenever swaps are blocked. `removeLiquidity` can remain ungated to preserve the LP unwind path:

```solidity
function addLiquidity(
    address owner,
    uint80 salt,
    LiquidityDelta calldata deltas,
    bytes calldata callbackData,
    bytes calldata extensionData
) external whenNotPaused nonReentrant(PoolActions.ADD_LIQUIDITY) returns (...) {
```

Alternatively, if the protocol wants to allow deposits during admin pause (level 1) but not during protocol emergency pause (level 2), a finer-grained check can be introduced:

```solidity
function _checkNotProtocolPaused() internal view {
    if (pauseLevel == 2) revert PoolPaused();
}
```

and apply it only to `addLiquidity`.

### Proof of Concept

1. Pool is active (`pauseLevel = 0`). An exploit begins draining `token0` from the pool.
2. Factory owner calls `protocolPausePool(pool)` → `pauseLevel = 2`. `swap` is now blocked.
3. Before the pause is publicly known, Alice calls `addLiquidity` with 10,000 USDC worth of `token0`. The call succeeds — no `whenNotPaused` check exists on `addLiquidity`.
4. `LiquidityLib.addLiquidity` credits Alice's tokens to `binState.token0BalanceScaled` and `binTotals.scaledToken0`.
5. The attacker's ongoing exploit drains the newly credited tokens in the same block.
6. Alice calls `removeLiquidity` in the next transaction but receives near-zero tokens because her bin balance was drained.
7. Alice loses her 10,000 USDC deposit. The pause that was meant to protect users instead created a false sense of safety while `addLiquidity` remained open.
