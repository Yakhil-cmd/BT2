### Title
Silent Precompile Failure After ERC-20 Burn Causes Permanent Fund Loss — (`etc/eth-contracts/contracts/EvmErc20.sol`)

---

### Summary

`withdrawToEthereum` (and `withdrawToNear`) in `EvmErc20.sol` burns the caller's ERC-20 tokens **before** calling the exit precompile. The assembly `call` return value is never checked. When `amount > u128::MAX`, the precompile's `parse_amount` guard returns `ERR_INVALID_AMOUNT` as a non-fatal `ExitError`, which the EVM executor maps to `PrecompileFailure::Error` — a sub-call failure that does not revert the parent context. The burn is committed, no NEP-141 withdraw promise is emitted, and the tokens are permanently destroyed.

---

### Finding Description

**Step 1 — Entry point (Solidity, unprivileged caller):**

`withdrawToEthereum` is `external` with no access control. Any caller holding sufficient balance can invoke it with an arbitrary `uint256 amount`. [1](#0-0) 

**Step 2 — Burn-before-call ordering:**

`_burn` executes first, permanently reducing the caller's ERC-20 balance. Only then is the precompile called. [2](#0-1) 

**Step 3 — Unchecked assembly `call` return value:**

The `res` local variable is assigned but never tested. No `require(res != 0)` or equivalent guard exists. [3](#0-2) 

**Step 4 — Precompile rejects `amount > u128::MAX` as a non-fatal error:**

`parse_amount` returns `Err(ExitError::Other("ERR_INVALID_AMOUNT"))` for any value exceeding `u128::MAX`. [4](#0-3) 

**Step 5 — Error classification: `PrecompileFailure::Error`, not `Fatal`:**

`process_precompile` maps every `ExitError` from `Precompile::run` to `PrecompileFailure::Error`. In SputnikVM semantics, `Error` causes the sub-call to return 0 to the caller but does **not** revert the parent execution frame. [5](#0-4) 

**Step 6 — No withdraw promise is created:**

The promise log that drives the NEP-141 `withdraw` call is only emitted on the `Ok(...)` path of `ExitToEthereum::run`. An error return skips it entirely. [6](#0-5) 

**Historical note:** CHANGES.md records that PR #681 fixed a *panic* (which would have been a `Fatal` error, reverting the whole transaction) for amounts > u128::MAX. That fix converted the panic into a graceful `ExitError::Other`, but the Solidity contract was never updated to check the sub-call result — turning a revert into a silent fund loss. [7](#0-6) 

The identical unchecked-return pattern exists in `withdrawToNear` in both `EvmErc20.sol` and `EvmErc20V2.sol`. [8](#0-7) [9](#0-8) 

---

### Impact Explanation

Any ERC-20 balance exceeding `u128::MAX` that is passed to `withdrawToEthereum` or `withdrawToNear` is burned on the EVM side with no corresponding NEP-141 unlock on the NEAR side. The tokens are permanently destroyed with no recovery path. This is **Critical — Permanent freezing/destruction of funds**.

---

### Likelihood Explanation

The `mint` function is `onlyAdmin`, so a balance > u128::MAX requires the admin to have minted such an amount (≈ 3.4 × 10^38 raw token units). While unusual, there is no on-chain guard preventing it. Once such a balance exists — whether minted directly or accumulated via transfers — any holder can trigger the loss without further admin involvement. The precondition is non-trivial but the vulnerability is fully reachable on unmodified production code.

---

### Recommendation

1. **Check the precompile return value in Solidity.** Replace the bare assembly block with a `require`:
   ```solidity
   assembly {
       let res := call(gas(), 0xb0bd02f6a392af548bdf1cfaee5dfa0eefcc8eab, 0, add(input, 32), input_size, 0, 32)
       if iszero(res) { revert(0, 0) }
   }
   ```
   Apply the same fix to `withdrawToNear` in both `EvmErc20.sol` and `EvmErc20V2.sol`.

2. **Alternatively, validate `amount` before burning.** Add an explicit upper-bound check (`require(amount <= type(uint128).max)`) before `_burn` so the transaction reverts cleanly without destroying tokens.

3. **Add an upper-bound guard in `mint`.** Prevent minting amounts that cannot be safely withdrawn.

---

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

// Precondition: admin has minted (uint128 max + 1) tokens to `attacker`
// Expected: attacker's balance is zeroed, no NEP-141 withdraw promise emitted

contract PoC {
    EvmErc20 token = EvmErc20(<deployed_address>);

    function exploit(address recipient) external {
        uint256 badAmount = uint256(type(uint128).max) + 1;
        // Admin must have minted badAmount to address(this) first
        token.withdrawToEthereum(recipient, badAmount);
        // After this call:
        //   token.balanceOf(address(this)) == 0  (burned)
        //   No NEP-141 withdraw promise was scheduled (precompile failed silently)
        assert(token.balanceOf(address(this)) == 0);
    }
}
```

The existing unit test in `native.rs` already confirms `parse_amount` rejects the value: [10](#0-9) 

A local integration test should verify that after calling `withdrawToEthereum` with `amount = u128::MAX + 1`, the ERC-20 balance is zero and no promise log with the `exitToEthereum` address appears in the transaction receipt.

fn parse_amount(input: &[u8]) -> Result<U256, ExitError> {
    let amount = U256::from_big_endian(input);

    if amount > U256::from(u128::MAX) {
        return Err(ExitError::Other(Cow::from("ERR_INVALID_AMOUNT")));
    }

    Ok(amount)
}

fn process_precompile(
    p: &dyn Precompile,
    handle: &impl PrecompileHandle,
) -> Result<PrecompileOutput, PrecompileFailure> {
    let input = handle.input();
    let gas_limit = handle.gas_limit();
    let context = handle.context();
    let is_static = handle.is_static();

    p.run(input, gas_limit.map(EthGas::new), context, is_static)
        .map_err(|exit_status| PrecompileFailure::Error { exit_status })
}
