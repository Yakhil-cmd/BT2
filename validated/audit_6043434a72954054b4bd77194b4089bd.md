### Title
`exit_to_near_precompile_callback` Ignores Partial `ft_transfer_call` Fill, Causing Permanent ERC-20 Token Loss — (`engine/src/contract_methods/connector.rs`)

---

### Summary

When a user bridges ERC-20 tokens from Aurora to NEAR using the `ExitToNear` precompile with the `Omni` message path, the engine burns the full ERC-20 amount and calls `ft_transfer_call` on the NEP-141 contract. The callback `exit_to_near_precompile_callback` only checks whether the promise succeeded (binary pass/fail) and **ignores the actual amount transferred** returned by `ft_transfer_call`. If the NEP-141 receiver returns unused tokens (a partial fill, which is valid NEP-141 behavior), the ERC-20 tokens are permanently destroyed while the user receives fewer NEP-141 tokens than expected. The difference is silently absorbed into Aurora's NEP-141 balance with no refund.

---

### Finding Description

The `ExitToNear` precompile supports an `Omni` message variant that calls `ft_transfer_call` on the NEP-141 contract instead of `ft_transfer`. In the NEP-141 standard, `ft_transfer_call` allows the receiver to return an `unused_amount` via `ft_on_transfer`, which is then refunded to the sender by `ft_resolve_transfer`. The final resolved return value of `ft_transfer_call` is the amount **actually kept** by the receiver (`amount - unused_amount`).

In `exit_erc20_token_to_near` (`engine-precompiles/src/native.rs`), the `Omni` path schedules a `ft_transfer_call` promise: [1](#0-0) 

The callback `exit_to_near_precompile_callback` in `engine/src/contract_methods/connector.rs` then runs: [2](#0-1) 

The pattern `PromiseResult::Successful(_)` discards the bytes returned by `ft_transfer_call`. Those bytes encode the amount actually transferred (i.e., `amount - unused_amount`). The callback never compares this value against the original burn amount. If the receiver returned `unused_amount > 0`, the NEP-141 tokens flow back to Aurora's account (the sender of `ft_transfer_call`), but the ERC-20 tokens were already burned for the full amount. No refund is issued to the user.

The refund branch only fires when the promise **fails** entirely: [3](#0-2) 

A partial fill (promise succeeds but returns fewer tokens than requested) is treated identically to a full fill.

---

### Impact Explanation

**Critical — Permanent loss of user funds.**

The ERC-20 tokens are burned atomically before the cross-contract call. If the NEP-141 receiver returns any unused tokens, those tokens are credited back to Aurora's NEP-141 balance (not the user's). The user permanently loses the difference between the burned ERC-20 amount and the NEP-141 tokens actually received at the destination. There is no recovery path: the ERC-20 tokens are gone, and the returned NEP-141 tokens are silently absorbed into the engine's balance.

---

### Likelihood Explanation

**Medium.** The `Omni` message path is a production feature reachable by any EVM user who calls the `ExitToNear` precompile with an `Omni`-encoded message. The trigger condition — a NEP-141 receiver that returns a non-zero `unused_amount` from `ft_on_transfer` — is standard NEP-141 behavior. Any receiver with capacity limits, allowlists, or conditional acceptance logic can produce this outcome. No admin compromise or special privilege is required; a regular EVM user initiating a bridge-out transaction is sufficient.

---

### Recommendation

In `exit_to_near_precompile_callback`, parse the bytes from `PromiseResult::Successful(bytes)` and decode the actual transferred amount. Compare it against the original burn amount stored in `ExitToNearPrecompileCallbackArgs`. If `transferred_amount < burn_amount`, treat the difference as a partial failure and issue a refund of `burn_amount - transferred_amount` ERC-20 tokens to the original caller, mirroring the existing `refund_on_error` logic.

Alternatively, use `ft_transfer` (which is atomic and does not allow partial fills) for the `Omni` path, or enforce that the receiver must accept the full amount by checking the resolved return value before considering the exit successful.

---

### Proof of Concept

1. User holds 1000 units of a bridged ERC-20 token on Aurora backed by NEP-141 `token.near`.
2. User calls the `ExitToNear` precompile with `Message::Omni(msg)`, requesting to bridge all 1000 tokens to a NEAR receiver contract.
3. The precompile burns 1000 ERC-20 tokens and schedules `token.near.ft_transfer_call(receiver, 1000, msg)`.
4. The receiver's `ft_on_transfer` returns `unused_amount = 400` (e.g., it only accepts 600 due to a capacity limit).
5. NEP-141's `ft_resolve_transfer` refunds 400 tokens to Aurora's account on `token.near`.
6. `exit_to_near_precompile_callback` fires with `PromiseResult::Successful(bytes_encoding_600)`.
7. The callback matches `Successful(_)` — the `_` discards the `600` value — and returns `None` (success).
8. Result: user lost 1000 ERC-20 tokens, received 600 NEP-141 tokens, and 400 NEP-141 tokens are permanently stuck in Aurora's `token.near` balance. No error is raised. [4](#0-3) [5](#0-4)
