### Title
Silo Whitelist Access Control Enforced by `submit`/`submit_with_args` Can Be Bypassed via `call` and `deploy_code` Entry Points - (`engine/src/contract_methods/evm_transactions.rs`)

---

### Summary

In Silo mode, Aurora Engine enforces whitelist-based access control only inside `engine::submit` (via `assert_access`). The two alternative NEAR-level entry points — `call` and `deploy_code` — perform no whitelist check at all. An unprivileged NEAR account that is not on any whitelist can bypass the Silo access restriction entirely by calling `call` instead of `submit` for EVM transactions, or `deploy_code` instead of `submit` for contract deployments.

---

### Finding Description

The Silo mode whitelist system defines four lists:

- `Admin` (NEAR accounts allowed to deploy EVM code via `submit`)
- `EvmAdmin` (EVM addresses allowed to deploy EVM code via `submit`)
- `Account` (NEAR accounts allowed to submit EVM transactions via `submit`)
- `Address` (EVM addresses allowed to submit EVM transactions via `submit`)

The enforcement gate is `assert_access`, called exclusively inside `submit_with_alt_modexp` (the shared implementation of `submit` and `submit_with_args`): [1](#0-0) 

`assert_access` routes to `is_allow_submit` for calls (`to.is_some()`) and `is_allow_deploy` for deployments (`to.is_none()`): [2](#0-1) 

However, the `call` entry point in `evm_transactions.rs` contains no whitelist check — it only calls `require_running` and then directly executes the EVM call: [3](#0-2) 

Similarly, `deploy_code` contains no whitelist check — it only calls `require_running` and then directly deploys EVM bytecode: [4](#0-3) 

Both are exposed as public NEAR extern-C entry points: [5](#0-4) 

The `submit` path, by contrast, always passes through `assert_access` at line 1052: [6](#0-5) 

---

### Impact Explanation

In Silo mode with whitelists enabled, the operator intends to restrict EVM interaction to a curated set of NEAR accounts and EVM addresses. Because `call` and `deploy_code` skip `assert_access` entirely, any NEAR account — regardless of whitelist membership — can:

1. **Call any EVM contract** in the Silo by invoking `call` with a `CallArgs` payload, bypassing the `Account`/`Address` whitelist.
2. **Deploy arbitrary EVM bytecode** by invoking `deploy_code`, bypassing the `Admin`/`EvmAdmin` whitelist.

If EVM contracts in the Silo hold user funds (e.g., ERC-20 tokens, ETH balances), an unauthorized caller can invoke those contracts to drain funds. This constitutes direct theft of user funds at rest.

**Impact: Critical** — direct theft of user funds held in EVM contracts within the Silo.

---

### Likelihood Explanation

The `call` and `deploy_code` NEAR methods are standard, documented Aurora Engine entry points. Any NEAR account can call them with no special privileges. The attacker only needs to know the target EVM contract address and the calldata to steal funds. No leaked keys, governance capture, or oracle manipulation is required. The Silo whitelist feature is specifically designed to restrict access, making this bypass directly contrary to the operator's security model.

**Likelihood: High** — trivially reachable by any NEAR account once the Silo is deployed with whitelists enabled.

---

### Recommendation

Apply the same `assert_access` (or an equivalent whitelist check) at the start of both `call` and `deploy_code` in `engine/src/contract_methods/evm_transactions.rs`.

For `call`, check `is_allow_submit` using the predecessor account ID and the EVM address derived from it. For `deploy_code`, check `is_allow_deploy`. The check should be a no-op when the Silo whitelists are disabled (which is the default), preserving backward compatibility.

---

### Proof of Concept

1. Deploy Aurora Engine in Silo mode with `set_silo_params` and enable all four whitelists via `set_whitelist_status`.
2. Fund an EVM address `victim` with ETH inside the Silo.
3. Deploy an EVM contract `Vault` (holding `victim`'s funds) via the owner's `submit`.
4. From an **unauthorized** NEAR account (not in any whitelist), call the NEAR method `call` with `CallArgs` targeting `Vault`'s withdrawal function.
5. Observe that the call succeeds and funds are transferred — the `NotAllowed` error that `submit` would have returned is never triggered.

The root cause is that `call` invokes `engine.call_with_args` directly without ever reaching `assert_access`: [7](#0-6) 

whereas `submit` always passes through `assert_access` before any EVM execution: [6](#0-5)

```rust
fn assert_access<I: IO + Copy, E: Env>(
    io: &I,
    env: &E,
    transaction: &NormalizedEthTransaction,
) -> Result<(), EngineError> {
    let allowed = if transaction.to.is_some() {
        silo::is_allow_submit(io, &env.predecessor_account_id(), &transaction.address)
    } else {
        silo::is_allow_deploy(io, &env.predecessor_account_id(), &transaction.address)
    };

    if !allowed {
        return Err(EngineError {
            kind: EngineErrorKind::NotAllowed,
            gas_used: 0,
        });
    }

    Ok(())
}
```

```rust
#[named]
pub fn call<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<SubmitResult, ContractError> {
    with_logs_hashchain(io, env, function_name!(), |mut io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        let bytes = io.read_input().to_vec();
        let args = CallArgs::deserialize(&bytes).ok_or(errors::ERR_BORSH_DESERIALIZE)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();

        let mut engine: Engine<_, E, AuroraModExp> = Engine::new_with_state(
            state,
            predecessor_address(&predecessor_account_id),
            current_account_id,
            io,
            env,
        );
        let result = engine.call_with_args(args, handler)?;
        let result_bytes = borsh::to_vec(&result).map_err(|_| errors::ERR_SERIALIZE)?;
        io.return_output(&result_bytes);
        Ok(result)
    })
}
```
