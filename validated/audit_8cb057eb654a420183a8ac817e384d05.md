### Title
`MsgTurnBridge` Handler Is a No-Op With No Permission Check — Bypasses Bridge Authorization and Permanently Disables Circuit Breaker - (File: `x/cronos/keeper/msg_server.go`)

### Summary
The `TurnBridge` message handler in `x/cronos/keeper/msg_server.go` is completely unimplemented: it returns `nil, nil` without performing any permission check or state mutation. This means (1) any unprivileged user can submit `MsgTurnBridge` and receive a success response, bypassing the bridge authorization guard entirely, and (2) the bridge circuit breaker is permanently broken — the bridge state can never be toggled regardless of who calls it.

### Finding Description
`MsgTurnBridge` is documented in ADR-009 as one of the two restricted messages that require explicit permission (`CanTurnBridge`). The permission system is correctly implemented in `HasPermission` and enforced in `UpdateTokenMapping`. However, the `TurnBridge` handler itself is a stub:

```go
// TurnBridge implements the grpc method
func (k msgServer) TurnBridge(goCtx context.Context, msg *types.MsgTurnBridge) (*types.MsgTurnBridgeResponse, error) {
    return nil, nil
}
```

Compare this to `UpdateTokenMapping`, which correctly gates on `HasPermission`:

```go
func (k msgServer) UpdateTokenMapping(...) (...) {
    if !k.HasPermission(ctx, msg.GetSigners(), CanChangeTokenMapping) {
        return nil, errors.Wrap(sdkerrors.ErrUnauthorized, "msg sender is not authorized")
    }
    ...
}
```

The `TurnBridge` handler has no such guard. Any address can broadcast a `MsgTurnBridge` transaction and receive a `code: 0` success response. Furthermore, because the handler performs no state write, the bridge enabled/disabled flag is never mutated — the circuit breaker intended for emergency use is permanently inoperative.

### Impact Explanation
Two distinct High-severity impacts apply:

1. **Bypass of bridge authorization**: The `CanTurnBridge` permission bit exists precisely to restrict who may toggle the bridge. Because the handler skips the `HasPermission` check entirely, any unprivileged user can call `MsgTurnBridge` and succeed, violating the authorization model described in ADR-009.

2. **Permanent inability to disable the bridge**: Even the legitimate admin cannot turn the bridge off. In an active exploit scenario (e.g., a bridge drain), the intended emergency stop mechanism silently does nothing, leaving the bridge permanently open and all bridged assets continuously at risk.

### Likelihood Explanation
The entry path is fully unprivileged: any account can construct and broadcast a `MsgTurnBridge` transaction via the standard Cosmos SDK tx pipeline. No special keys, governance votes, or validator access are required. The stub is reachable on every node that processes the message type.

### Recommendation
Implement `TurnBridge` with the same pattern used by `UpdateTokenMapping`:
1. Check `HasPermission(ctx, msg.GetSigners(), CanTurnBridge)` and return `ErrUnauthorized` on failure.
2. Read the current bridge-enabled state from params, toggle it, and persist via `k.SetParams`.
3. Emit an appropriate event.

### Proof of Concept
1. Any unprivileged address broadcasts `MsgTurnBridge{Enable: false}`.
2. The transaction is accepted and returns `code: 0`.
3. The bridge state is unchanged — the bridge remains enabled.
4. The admin also broadcasts `MsgTurnBridge{Enable: false}` — same result: success, no state change.
5. The bridge cannot be disabled by any actor.

```go
func (k msgServer) UpdateTokenMapping(goCtx context.Context, msg *types.MsgUpdateTokenMapping) (*types.MsgUpdateTokenMappingResponse, error) {
```

```go
func (k Keeper) HasPermission(ctx sdk.Context, accounts []sdk.AccAddress, permissionsToCheck uint64) bool {
	// case when no permission is needed
	if permissionsToCheck == 0 {
		return true
	}
	admin := k.GetParams(ctx).CronosAdmin
	for _, account := range accounts {
		if admin == account.String() {
			return true
		}
		permission := k.GetPermissions(ctx, account)
		if permission&permissionsToCheck == permissionsToCheck {
			return true
		}
	}

	return false
```
