### Title
Missing Threshold Binding in `KeygenOutput` Allows Threshold Mismatch Between Keygen and Presign/Sign Phases - (File: `src/lib.rs`, `src/ecdsa/ot_based_ecdsa/presign.rs`, `src/frost/eddsa/sign.rs`)

---

### Summary

`KeygenOutput<C>` stores only `private_share` and `public_key` — it does not record the threshold used during key generation. Every downstream protocol phase (`presign`, `sign`) accepts a caller-supplied threshold with no cross-validation against the keygen threshold. A malicious participant or misconfigured caller can supply a mismatched threshold, causing the presign/sign phase to linearize key shares with incorrect Lagrange coefficients, producing cryptographically corrupted and unusable signature outputs.

---

### Finding Description

`KeygenOutput<C>` is defined as:

```rust
pub struct KeygenOutput<C: Ciphersuite> {
    pub private_share: SigningShare<C>,
    pub public_key: VerifyingKey<C>,
}
``` [1](#0-0) 

No threshold field is present. The threshold used during DKG (which determines the degree of the secret-sharing polynomial) is silently discarded after `do_keyshare` returns.

In the OT-based ECDSA presign, `PresignArguments` carries a separate `threshold` field with only a documentation comment stating it "must match the original threshold" — there is no runtime enforcement:

```rust
pub struct PresignArguments {
    pub keygen_out: KeygenOutput,
    /// The desired threshold for the presignature, which must match the original threshold
    pub threshold: ReconstructionLowerBound,
}
``` [2](#0-1) 

The `presign` entry-point validates `args.threshold` only against the triple thresholds, never against the keygen threshold:

```rust
if args.threshold != args.triple0.1.threshold || args.threshold != args.triple1.1.threshold {
    return Err(InitializationError::BadParameters(...));
}
``` [3](#0-2) 

Inside `do_presign`, the key share is linearized using Lagrange coefficients derived from the presign participant list:

```rust
let lambda_me = participants.lagrange::<Secp256>(me)?;
let x_prime_i = lambda_me * private_share;
``` [4](#0-3) 

If `t_presign ≠ t_keygen`, the key shares (evaluations of a degree-`(t_keygen−1)` polynomial) are linearized with Lagrange coefficients computed for a different threshold, making the resulting `sigma_i` values cryptographically inconsistent with the actual secret key.

The same structural gap exists in the EdDSA (FROST) path. `sign_v1` and `sign_v2` accept a caller-supplied `threshold` independently of `keygen_output`:

```rust
pub fn sign_v1(
    participants: &[Participant],
    threshold: impl Into<ReconstructionLowerBound>,
    me: Participant,
    coordinator: Participant,
    keygen_output: KeygenOutput,
    ...
``` [5](#0-4) 

`construct_key_package` then builds the FROST `KeyPackage` with the caller-supplied threshold, not the keygen threshold:

```rust
Ok(KeyPackage::new(
    identifier,
    signing_share,
    verifying_share,
    *verifying_key,
    u16::try_from(threshold.value())...?,
))
``` [6](#0-5) 

The same pattern appears in the RedJubjub path: [7](#0-6) 

And in the generic FROST presign: [8](#0-7) 

---

### Impact Explanation

**High — Corruption of presign/sign outputs so honest parties produce unusable cryptographic outputs.**

In the OT-based ECDSA path, a threshold mismatch causes `sigma_i` to be computed with wrong Lagrange weights relative to the keygen polynomial. The sign phase then reconstructs an incorrect aggregate sigma, producing an ECDSA signature that fails standard verification. In the FROST path, the wrong threshold embedded in `KeyPackage` causes FROST's internal aggregation to use an incorrect reconstruction set, again producing an invalid signature. In both cases, honest parties complete the protocol without error but output a signature that cannot be verified — a silent, undetectable corruption.

---

### Likelihood Explanation

The threshold is a critical protocol parameter that must remain consistent across keygen → presign → sign. Because `KeygenOutput` does not carry it, every integration point is a potential mismatch site. A malicious participant who controls their own local `PresignArguments` can supply any threshold value. A misconfigured honest caller who reshares keys (changing the threshold) and then forgets to update the threshold passed to presign will silently corrupt all subsequent signatures. The comment "must match the original threshold" in `PresignArguments` is the only guard — it is not enforced. [9](#0-8) 

---

### Recommendation

Store the threshold inside `KeygenOutput<C>`:

```rust
pub struct KeygenOutput<C: Ciphersuite> {
    pub private_share: SigningShare<C>,
    pub public_key: VerifyingKey<C>,
    pub threshold: ReconstructionLowerBound,  // add this
}
```

Then, in `presign` (OT-based ECDSA), add a runtime check:

```rust
if args.threshold != args.keygen_out.threshold {
    return Err(InitializationError::BadParameters(
        "presign threshold must match keygen threshold".to_string()
    ));
}
```

Apply the same check in `sign_v1`, `sign_v2`, and the FROST presign entry-points. This mirrors the existing triple-threshold cross-check and closes the gap without requiring any protocol changes.

---

### Proof of Concept

1. Run keygen with `threshold = 3` over 5 participants → each party receives a `KeygenOutput` (share of a degree-2 polynomial).
2. Generate triples with `threshold = 2`.
3. Call `presign` with `args.threshold = 2` and the keygen output from step 1. The check at line 43 of `presign.rs` passes (triple thresholds match 2). No check against the keygen threshold (3) is performed.
4. Inside `do_presign`, `lambda_me` is computed for a 2-of-N Lagrange basis, but `private_share` is a point on a 3-of-N polynomial. `x_prime_i = lambda_me * private_share` is cryptographically inconsistent.
5. The sign phase reconstructs sigma using the wrong Lagrange weights, producing an ECDSA signature `(r, s)` that fails `verify(public_key, msg_hash, (r, s))`.
6. All honest parties complete the protocol with `Ok(...)` — the corruption is silent. [10](#0-9) [4](#0-3)

pub fn presign(
    participants: &[Participant],
    me: Participant,
    args: PresignArguments,
) -> Result<impl Protocol<Output = PresignOutput>, InitializationError> {
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }
    // Spec 1.1
    if args.threshold.value() > participants.len() {
        return Err(InitializationError::ThresholdTooLarge {
            threshold: args.threshold.value(),
            max: participants.len(),
        });
    }

    // NOTE: We omit the check that the new participant set was present for
    // the triple generation, because presumably they need to have been present
    // in order to have shares.

    // Also check that we have enough participants to reconstruct shares.
    if args.threshold != args.triple0.1.threshold || args.threshold != args.triple1.1.threshold {
        return Err(InitializationError::BadParameters(
            "New threshold must match the threshold of both triples".to_string(),
        ));
    }

    let participants =
        ParticipantList::new(participants).ok_or(InitializationError::DuplicateParticipants)?;

    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    let ctx = Comms::new();
    let fut = do_presign(ctx.shared_channel(), participants, me, args);
    Ok(make_protocol(ctx, fut))
}
