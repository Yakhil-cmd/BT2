### Title
`skip_stateful_validations` ignores `StatefulTransactionValidatorConfig::max_nonce_for_validation_skip`, allowing gateway admission of invoke transactions with invalid signatures when the skip should be disabled — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary
The gateway's `skip_stateful_validations` free function hardcodes the nonce-1 skip condition and never reads `max_nonce_for_validation_skip` from `StatefulTransactionValidatorConfig`. An operator setting `max_nonce_for_validation_skip = Nonce(Felt::ZERO)` to disable the UX skip has no effect on the gateway: a nonce=1 invoke from an undeployed account with a deploy_account in the mempool will still bypass `__validate__`, admitting the transaction without signature verification.

### Finding Description

`StatefulTransactionValidatorConfig` declares `max_nonce_for_validation_skip` as a configurable field: [1](#0-0) 

The default is `Nonce(Felt::ONE)`: [2](#0-1) 

The gateway's `StatefulTransactionValidator` holds `self.config` (which contains this field): [3](#0-2) 

But `skip_stateful_validations` is a free function that receives only `(tx, account_nonce, mempool_client)` — the config is never passed: [4](#0-3) 

Inside the function, the skip condition is hardcoded to `nonce == Nonce(Felt::ONE)`: [5](#0-4) 

`max_nonce_for_validation_skip` has **zero usages** anywhere in `crates/apollo_gateway/`. The config field is dead code in the gateway.

By contrast, the sequencer-side validator (`py_validator.rs`) correctly consults `self.max_nonce_for_validation_skip` before deciding to skip: [6](#0-5) 

### Impact Explanation

When `max_nonce_for_validation_skip = Nonce(Felt::ZERO)` is configured (operator intent: never skip), the gateway still returns `skip_validate = true` for any invoke with `tx.nonce() == 1` and `account_nonce == 0` whose sender has a deploy_account in the mempool. `run_validate_entry_point` then sets `execution_flags.validate = false`: [7](#0-6) 

The blockifier's `StatefulValidator::validate` runs without calling `__validate__`, so a wrong or malformed signature is never checked. The transaction is admitted to the mempool. This satisfies the **High** impact tier: *Mempool/gateway/RPC admission accepts invalid transactions before sequencing.*

### Likelihood Explanation

The preconditions are fully attacker-controllable from unprivileged public inputs:
- Submit a `deploy_account` transaction (nonce=0) for a fresh address — this puts the account in the mempool.
- Submit an `invoke` transaction with `nonce=1` and an arbitrary/wrong signature.

No operator or privileged access is required. The only external dependency is that `account_tx_in_pool_or_recent_block` returns `true`, which is guaranteed by the attacker's own deploy_account submission.

### Recommendation

Pass `self.config.max_nonce_for_validation_skip` into `skip_stateful_validations` (or make it a method on `StatefulTransactionValidator`) and add the upper-bound check mirroring `py_validator.rs`:

```rust
// In run_pre_validation_checks, pass the config field:
let skip_validate = skip_stateful_validations(
    executable_tx,
    account_nonce,
    mempool_client.clone(),
    self.config.max_nonce_for_validation_skip,  // add this
).await?;

// In skip_stateful_validations, add the bound check:
if tx.nonce() == Nonce(Felt::ONE)
    && account_nonce == Nonce(Felt::ZERO)
    && tx.nonce() <= max_nonce_for_validation_skip  // add this guard
{
    ...
}
```

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
```
