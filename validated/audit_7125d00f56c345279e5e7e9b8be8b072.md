### Title
Single Malicious Participant Permanently Blocks FROST Signing via Collect-All Pattern — (`File: src/frost/eddsa/sign.rs`, `src/protocol/helpers.rs`)

### Summary

The FROST signing coordinator collects signature shares from **every** participant in the signing session using `recv_from_others`, which blocks until all N participants respond. Because FROST only requires t-of-N shares to produce a valid aggregate signature, a single malicious participant who withholds their share permanently stalls the coordinator with no recovery path — a direct analog to the push-over-pull DoS in the reference report.

### Finding Description

`recv_from_others` in `src/protocol/helpers.rs` loops until it has received exactly one message from **every** other participant:

```rust
while !seen.full() {
    let (from, msg) = chan.recv(waitpoint).await?;
    if seen.put(from) {
        messages.push((from, msg));
    }
}
``` [1](#0-0) 

There is no timeout, no threshold-based early exit, and no mechanism to skip a non-responding participant. The function only returns when `seen.full()` — i.e., every participant has contributed.

This helper is called twice inside `do_sign_coordinator_v1`: once to collect round-1 commitments and once to collect round-2 signature shares:

```rust
for (from, commitment) in recv_from_others(&chan, commit_waitpoint, &participants, me).await? {
    commitments_map.insert(from.to_identifier()?, commitment);
}
// ...
for (from, signature_share) in recv_from_others(&chan, r2_wait_point, &participants, me).await? {
    signature_shares.insert(from.to_identifier()?, signature_share);
}
``` [2](#0-1) 

The same pattern appears in `do_sign_coordinator_v2`: [3](#0-2) 

The FROST aggregate step only needs `threshold` valid shares to produce a signature:

```rust
let signature = aggregate(&signing_package, &signature_shares, &public_key_package)
    .map_err(|e| ProtocolError::AssertionFailed(e.to_string()))?;
``` [4](#0-3) 

Yet the coordinator never attempts aggregation until all N participants have responded. A single participant who never sends their share causes the coordinator's async future to suspend indefinitely at `chan.recv(waitpoint).await`, and the `Protocol::poke()` loop returns `Action::Wait` forever. [5](#0-4) 

### Impact Explanation

Any participant included in the signing session can permanently deny signing for all honest parties by simply not sending their round-2 signature share (or their round-1 commitment). The coordinator has no way to proceed with the remaining t honest shares, no way to exclude the silent participant, and no timeout to abort and retry. The current protocol instance is permanently stuck.

This maps to: **High — Permanent denial of signing for honest parties under valid protocol inputs and documented trust assumptions.**

### Likelihood Explanation

The attack requires only that a malicious actor be included as one of the N participants in a signing session. No cryptographic break, no key compromise, and no privileged access is needed. The attacker simply does nothing — they withhold their message. Because the coordinator always waits for all N participants regardless of the threshold t, even a single dropout (malicious or accidental) is sufficient to permanently stall the session.

### Recommendation

Replace the collect-all pattern with a threshold-aware collection: once the coordinator has received valid contributions from any `threshold` participants, it should proceed to aggregation without waiting for the remaining participants. This mirrors the pull-over-push fix in the reference report — instead of requiring every participant to "push" their share before any progress is made, the coordinator should proceed as soon as a sufficient quorum has responded.

Concretely, `recv_from_others` should be replaced (or supplemented) with a variant that accepts a minimum count parameter and returns as soon as that count is satisfied, discarding late or missing participants.

### Proof of Concept

1. Initiate a FROST signing session with N=3 participants and threshold t=2.
2. Participants P0 (coordinator), P1, P2 are included.
3. P1 sends its round-1 commitment normally.
4. P2 withholds its round-1 commitment entirely.
5. The coordinator's `recv_from_others` call at `commit_waitpoint` blocks forever waiting for P2.
6. `Protocol::poke()` returns `Action::Wait` indefinitely.
7. The coordinator never reaches the aggregation step, even though P0+P1 alone satisfy the threshold t=2.
8. The signing session is permanently denied for all honest parties. [6](#0-5) [7](#0-6)

async fn do_sign_coordinator_v1(
    mut chan: SharedChannel,
    participants: ParticipantList,
    threshold: ReconstructionLowerBound,
    me: Participant,
    keygen_output: KeygenOutput,
    message: Vec<u8>,
    rng: &mut impl CryptoRngCore,
) -> Result<SignatureOption, ProtocolError> {
    // --- Round 1.
    // * Wait for other parties' commitments.

    let mut commitments_map: BTreeMap<frost_ed25519::Identifier, round1::SigningCommitments> =
        BTreeMap::new();

    // signing share is the private_share
    let signing_share = keygen_output.private_share;

    // Step 1.1 (and implicitely 1.2)
    let (nonces, commitments) = round1::commit(&signing_share, rng);
    let nonces = Zeroizing::new(nonces);
    commitments_map.insert(me.to_identifier()?, commitments);

    // Step 1.3
    let commit_waitpoint = chan.next_waitpoint();

    // Step 1.4
    for (from, commitment) in recv_from_others(&chan, commit_waitpoint, &participants, me).await? {
        commitments_map.insert(from.to_identifier()?, commitment);
    }

    let signing_package = frost_ed25519::SigningPackage::new(commitments_map, message.as_slice());

    let mut signature_shares: BTreeMap<frost_ed25519::Identifier, round2::SignatureShare> =
        BTreeMap::new();

    // Step 1.5
    let r2_wait_point = chan.next_waitpoint();
    chan.send_many(r2_wait_point, &signing_package)?;

    // --- Round 2
    // * Wait for each other's signature share
    // Step 2.3 (2.1 and 2.2 are implicit)
    let vk_package = keygen_output.public_key;
    let key_package = construct_key_package(threshold, me, signing_share, &vk_package)?;
    let key_package = Zeroizing::new(key_package);
    let signature_share = round2::sign(&signing_package, &nonces, &key_package)
        .map_err(|e| ProtocolError::AssertionFailed(e.to_string()))?;

    // Step 2.5 (2.4 is implicit)
    signature_shares.insert(me.to_identifier()?, signature_share);
    for (from, signature_share) in recv_from_others(&chan, r2_wait_point, &participants, me).await?
    {
        signature_shares.insert(from.to_identifier()?, signature_share);
    }

    // --- Signature aggregation.
    // * Converted collected signature shares into the signature.
    // * Signature is verified internally during `aggregate()` call.

    // Step 2.6 and 2.7
    // We supply empty map as `verifying_shares` because we have disabled "cheater-detection" feature flag.
    // Feature "cheater-detection" only points to a malicious participant, if there's such.
    // It doesn't bring any additional guarantees.
    let public_key_package = PublicKeyPackage::new(BTreeMap::new(), vk_package);
    let signature = aggregate(&signing_package, &signature_shares, &public_key_package)
        .map_err(|e| ProtocolError::AssertionFailed(e.to_string()))?;

    Ok(Some(signature))
}
