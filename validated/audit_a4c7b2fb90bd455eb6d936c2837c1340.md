### Title
Incorrect `float64` → `cadence.UFix64` Cast Silently Zeros Bridge Fees at Bootstrap - (File: fvm/blueprints/bridge.go)

### Summary
`UpdateOnboardFeeTransaction` and `UpdateBaseFeeTransaction` in `fvm/blueprints/bridge.go` accept a `fee float64` parameter and encode it as `cadence.UFix64(fee)`. Because `cadence.UFix64` is `type UFix64 uint64` with an 8-decimal fixed-point scale (1 FLOW = 1e8), a direct Go type-conversion from `float64` discards the scale entirely. The bootstrap calls these functions with `1.0` and `0.001` respectively, producing on-chain fee values of `0.00000001 FLOW` and `0.0 FLOW` instead of `1.0 FLOW` and `0.001 FLOW`. Any unprivileged user who subsequently calls the bridge pays near-zero fees.

### Finding Description
`cadence.UFix64` is an alias for `uint64` where the integer `100_000_000` (1e8) represents `1.0 FLOW`. The correct way to construct a UFix64 from a human-readable decimal is `cadence.NewUFix64("1.0")`, which returns `100_000_000`. Every other fee in the codebase uses this pattern via `mustParseUFix64`. [1](#0-0) 

However, the two bridge fee setters perform a raw Go type conversion:

```go
AddArgument(jsoncdc.MustEncode(cadence.UFix64(fee)))
```

In Go, `cadence.UFix64(1.0)` is equivalent to `uint64(1.0)` = `1`, which in UFix64 fixed-point represents `0.00000001 FLOW`. `cadence.UFix64(0.001)` truncates to `uint64(0)` = `0.0 FLOW`. [2](#0-1) [3](#0-2) 

The bootstrap calls these with `1.0` and `0.001`: [4](#0-3) 

The UFix64 scale is confirmed by the EVM balance constants: [5](#0-4) 

### Impact Explanation
After bootstrap, the on-chain bridge onboarding fee is `0.00000001 FLOW` (1e8× too small) and the base fee is `0.0 FLOW` (completely zeroed). Any unprivileged user who calls `EVM.CadenceOwnedAccount.depositTokens()` or `withdrawTokens()` — both publicly reachable Cadence entry points — pays essentially no fee. This is a bridge escrow mis-accounting issue: the fee escrow mechanism is silently disabled, causing permanent loss of protocol fee revenue for every bridge operation after network initialization. [6](#0-5) 

### Likelihood Explanation
The bootstrap procedure runs exactly once during network initialization and is not guarded by any runtime check that would catch the scale error. The incorrect fee values are committed to on-chain state. Every subsequent bridge call by any user exploits the near-zero fee. No special privileges, timing, or brute force are required — ordinary bridge usage is sufficient.

### Recommendation
Replace the direct cast with the same `cadence.NewUFix64` string-parsing pattern used everywhere else in the bootstrap:

```go
// Instead of:
cadence.UFix64(fee)  // WRONG: discards fixed-point scale

// Use:
feeVal, err := cadence.NewUFix64(fmt.Sprintf("%.8f", fee))
// or accept cadence.UFix64 directly at the call site, as done for all other fees
```

Alternatively, change the function signatures to accept `cadence.UFix64` directly (already parsed by the caller), matching the pattern used in `setupFees`. [7](#0-6) 

### Proof of Concept
```
cadence.UFix64(1.0)   → uint64(1)   → "0.00000001" FLOW  (intended: 1.0 FLOW)
cadence.UFix64(0.001) → uint64(0)   → "0.00000000" FLOW  (intended: 0.001 FLOW)

cadence.NewUFix64("1.0")   → uint64(100_000_000) → "1.00000000" FLOW  ✓
cadence.NewUFix64("0.001") → uint64(100_000)     → "0.00100000" FLOW  ✓
```

A user bridging tokens after bootstrap pays `0.0 FLOW` in base fees and `0.00000001 FLOW` in onboarding fees, regardless of the intended fee schedule, because the on-chain fee state was set with values 1e8× smaller than intended.
