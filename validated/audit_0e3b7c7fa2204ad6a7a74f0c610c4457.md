### Title
Cross-COA Signature Replay in `validateCOAOwnershipProof` Due to Missing `evmAddress` Binding in `signedData` ÔÇö (File: `fvm/evm/stdlib/contract.cdc`)

---

### Summary

`EVM.validateCOAOwnershipProof` verifies that a set of Cadence account signatures are valid over `signedData`, then checks that the COA resource at the given `path` matches the supplied `evmAddress`. It never checks that `signedData` encodes `evmAddress`. A signature produced for one COA can therefore be replayed verbatim to prove ownership of any other COA stored in the same Cadence account, bypassing the ERC-1271 `isValidSignature` gate implemented in the COA Solidity contract.

---

### Finding Description

`validateCOAOwnershipProof` performs two independent checks:

1. **Signature validity** ÔÇö the supplied signatures are verified over `signedData` using the Cadence account's keys.
2. **COA address match** ÔÇö the COA resource at `path` is borrowed and its EVM address is compared byte-by-byte against `evmAddress`.

The two checks are never correlated: nothing in the function requires that `signedData` contains `evmAddress` (or any commitment to the specific COA being claimed). The code itself documents this gap:

> *"Note: this function does not enforce that `signedData` includes `evmAddress`. In principle, a signature produced for one purpose could be replayed here against a different COA owned by the same Cadence account."*

Because `COAOwnershipProof` is a caller-supplied, RLP-encoded struct that includes `Address`, `CapabilityPath`, `KeyIndices`, and `Signatures`, an attacker who obtains a valid proof for COA\_A can freely substitute COA\_B's `CapabilityPath` and `evmAddress` while keeping the original signatures unchanged. The function will accept the modified proof because the signatures are still valid over the original `signedData`, and COA\_B genuinely exists at the new path.

The `isValidSignature` function in `coa.sol` calls the Cadence Arch precompile as:

```solidity
cadenceArch.staticcall(
    abi.encodeWithSignature(
        "verifyCOAOwnershipProof(address,bytes32,bytes)",
        address(this),   // ÔåÉ always the calling COA's EVM address
        _hash,
        _sig             // ÔåÉ attacker-controlled RLP proof
    )
);
```

`_sig` is fully attacker-controlled. The attacker replaces the `CapabilityPath` field inside `_sig` with COA\_B's path while reusing the signatures originally produced for COA\_A over `_hash`. The precompile forwards the modified proof to `validateCOAOwnershipProof`, which returns `isValid: true`.

---

### Impact Explanation

An attacker who obtains a legitimate COA ownership proof for any COA owned by a target Cadence account can forge a passing ERC-1271 `isValidSignature` response for **any other COA** owned by the same account. Concretely:

- ERC-1271-gated EVM contracts (token approvals, multisig wallets, NFT marketplaces, lending protocols) that rely on `isValidSignature` to authorise actions will accept the forged proof.
- The attacker can authorise on-chain EVM transactions on behalf of a COA they do not control, leading to direct loss of EVM-side assets held by or delegatable through that COA.

The impact class is **cross-VM asset loss / unauthorized account mutation**, matching the target scope.

---

### Likelihood Explanation

The attack requires:

1. The victim's Cadence account to hold **at least two COAs** at distinct public paths.
2. The attacker to possess a valid `COAOwnershipProof` (signatures + key indices) that the victim produced for one of those COAs over any `signedData` value.

Condition 2 is realistic: COA ownership proofs are produced and shared in off-chain authentication flows (e.g., wallet-connect style dApps, ERC-1271 signature requests). A malicious dApp can solicit a proof for COA\_A and immediately replay it against COA\_B. The `validateCOAOwnershipProof` function is `access(all)` and callable from any Cadence script or transaction, so no privileged access is required.

The developers note that "Flow wallets historically create at most one COA per account," which reduces the current attack surface, but the protocol places no on-chain restriction on creating multiple COAs, and the vulnerability is unconditional once two COAs exist.

---

### Recommendation

Bind `signedData` to the specific COA being claimed. The simplest fix is to require that `signedData` is (or commits to) the concatenation of the Cadence account address and the target `evmAddress` before verifying signatures:

```cadence
// Inside validateCOAOwnershipProof, before keyList.verify():
let expectedCommitment: [UInt8] = address.toBytes().concat(evmAddress.toStaticArray())
if signedData != expectedCommitment {
    return ValidationResult(
        isValid: false,
        problem: "EVM.validateCOAOwnershipProof(): signedData must commit to the target evmAddress."
    )
}
```

Alternatively, enforce the commitment at the encoding layer in `COAOwnershipProof` so that all callers (EVM precompile and direct Cadence callers) are protected uniformly.

### Proof of Concept

**Setup:** Alice's Cadence account `0xALICE` holds two COAs:
- COA\_A at `/public/coaA` with EVM address `0xAAAAÔÇª`
- COA\_B at `/public/coaB` with EVM address `0xBBBBÔÇª`

**Step 1 ÔÇö Obtain a legitimate proof for COA\_A.**  
A dApp asks Alice to sign `hash = keccak256("authorize action X")` and returns a `COAOwnershipProof`:
```
proof_A = RLP{
  KeyIndices:     [0],
  Address:        0xALICE,
  CapabilityPath: "coaA",
  Signatures:     [sig_over_hash]
}
```

**Step 2 ÔÇö Forge a proof for COA\_B.**  
The attacker constructs:
```
proof_B = RLP{
  KeyIndices:     [0],          // same
  Address:        0xALICE,      // same
  CapabilityPath: "coaB",       // ÔåÉ changed
  Signatures:     [sig_over_hash] // same ÔÇö still valid over `hash`
}
```

**Step 3 ÔÇö Call `isValidSignature` on COA\_B's EVM contract.**  
```solidity
// msg.sender calls COA_B.isValidSignature(hash, proof_B)
// COA_B.isValidSignature calls:
cadenceArch.staticcall(
    verifyCOAOwnershipProof(0xBBBBÔÇª, hash, proof_B)
)
```

**Step 4 ÔÇö `validateCOAOwnershipProof` accepts the forged proof.**
- `keyList.verify(signatureSet, signedData: hash, ÔÇª)` ÔåÆ `true` (signatures are valid over `hash`)
- `acc.capabilities.borrow<&EVM.CadenceOwnedAccount>(/public/coaB)` ÔåÆ succeeds
- `coaRef.address().bytes == evmAddress (0xBBBBÔÇª)` ÔåÆ `true`
- Returns `ValidationResult(isValid: true)`

The ERC-1271 gate on COA\_B passes, and the attacker can authorise any action that COA\_B's EVM contract gates behind `isValidSignature`.
