### Title
Floor Division in `delta_gas` Calculation Systematically Undercharges Users for Native Resource Consumption - (`File: basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`)

### Summary

In `compute_gas_refund`, the conversion of `native_used` back to EVM gas units uses Rust's default integer division (floor/truncation) instead of ceiling division. Because this value is used to determine how much extra gas a user must pay to cover native resource consumption, rounding down systematically undercharges every transaction where `native_used` is not an exact multiple of `native_per_gas`.

### Finding Description

The ZKsync OS double-resource accounting model tracks both EVM gas (ergs) and a native resource. After execution, the bootloader computes a `delta_gas` adjustment to ensure the user's gas payment covers native resource consumption:

```
deltaGas := (nativeUsed / nativePerGas) - gasUsed
```

The documentation in `docs/double_resource_accounting.md` describes this as the mechanism to ensure gas estimation includes enough gas to cover native resources.

The implementation in `compute_gas_refund` at line 72 is:

```rust
let delta_gas = if native_per_gas == 0 {
    0
} else {
    (native_used / native_per_gas) as i64 - (gas_used as i64)
};

if delta_gas > 0 {
    gas_used += delta_gas as u64;
}
```

`native_used / native_per_gas` uses Rust's integer division, which truncates (rounds toward zero / floor). When `native_used` is not exactly divisible by `native_per_gas`, the gas equivalent of native consumption is rounded **down**, meaning `delta_gas` is underestimated by 1. The user is charged 1 fewer gas unit than the native resource consumption warrants.

The correct direction — consistent with the principle of favoring the protocol over the user when calculating what a user must pay — is ceiling division: `native_used.div_ceil(native_per_gas)`.

Note that `native_per_gas` itself is computed with ceiling division (`gas_price.div_ceil(native_price)`) in both L2 and L1 transaction validation paths, which is the correct direction for that step. However, the inverse conversion back to gas units at refund time is not symmetrically rounded up.

### Impact Explanation

Every transaction where `native_used % native_per_gas != 0` (which is essentially every transaction) results in the user being refunded 1 extra gas unit they should not receive. The protocol collects 1 gas unit less in fees than it is owed. At a gas price of `P` wei, the per-transaction shortfall is `P` wei. This is systematic and accumulates across all transactions.

The `gas_used` value flows directly into the fee charged to the user and the payment to the operator:

```rust
let token_to_pay_operator = U256::from(context.gas_used)
    .checked_mul(gas_price_for_operator)...
```

So the undercharge directly reduces operator revenue by 1 gas unit per affected transaction.

### Likelihood Explanation

This affects every L2 transaction processed through `compute_gas_refund`. Any unprivileged user submitting a standard EIP-1559 or legacy transaction triggers this path. No special conditions are required — the rounding error occurs whenever `native_used % native_per_gas != 0`, which is the common case.

### Recommendation

Replace the floor division with ceiling division at line 72 of `refund_calculation.rs`:

```rust
// Before (floor division — undercharges by up to 1 gas unit):
(native_used / native_per_gas) as i64 - (gas_used as i64)

// After (ceiling division — correctly rounds up to favor the protocol):
native_used.div_ceil(native_per_gas) as i64 - (gas_used as i64)
```

This mirrors the rounding direction already used when computing `native_per_gas` itself (`gas_price.div_ceil(native_price)`), and is consistent with the principle that when calculating how much a user must pay, the result should round up to favor the protocol.

### Proof of Concept

**Concrete numeric example:**

- `gas_limit = 100_000`
- `gas_price = 257` wei, `native_price = 2` → `native_per_gas = ceil(257/2) = 129`
- `native_limit = 100_000 × 129 = 12_900_000`
- Transaction consumes `native_used = 12_900_001` native units (1 over a multiple of 129)
- `gas_used` from ergs = 99_999 (1 gas remaining)

Current code:
```
delta_gas = floor(12_900_001 / 129) - 99_999
          = 100_000 - 99_999 = 1
gas_used  = 99_999 + 1 = 100_000  ✓ (happens to be correct here)
```

Now with `native_used = 12_900_128` (128 over a multiple of 129):
```
delta_gas = floor(12_900_128 / 129) - 99_999
          = 99_999 - 99_999 = 0
gas_used  = 99_999  ← user pays 99,999 gas

Correct (ceiling):
delta_gas = ceil(12_900_128 / 129) - 99_999
          = 100_000 - 99_999 = 1
gas_used  = 100_000  ← user should pay 100,000 gas
```

The user receives a refund of 1 extra gas unit (worth `gas_price` wei) that the native resource consumption does not justify. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
