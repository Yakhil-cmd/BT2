### Title
`TurnBridge` Is a No-Op: Bridge Circuit Breaker Permanently Broken and Authorization Check Missing - (`x/cronos/keeper/msg_server.go`)

### Summary

`MsgTurnBridge` is the designated emergency circuit breaker for the Cronos bridge. Its handler implementation unconditionally returns `nil, nil` — it performs no permission check, writes no state, and has no effect. Any unprivileged user can call it and receive a success response, while the admin can never actually disable the bridge.

### Finding Description

The `TurnBridge` handler in `x/cronos/keeper/msg_server.go` is completely unimplemented:

```go
// TurnBridge implements the grpc method
func (k msgServer) TurnBridge(goCtx context.Context, msg *types.MsgTurnBridge) (*types.MsgTurnBridgeResponse, error) {
    return nil, nil
}
``` [1](#0-0) 

Compare this to `UpdateTokenMapping`, which correctly gates on `HasPermission`:

```go
if !k.HasPermission(ctx, msg.GetSigners(), CanChangeTokenMapping) {
    return nil, errors.Wrap(sdkerrors.ErrUnauthorized, "msg sender is not authorized")
}
``` [2](#0-1) 

The permission system explicitly defines `CanTurnBridge` as a distinct permission bit, and the query layer correctly reports it:

```go
CanTurnBridge = CanChangeTokenMapping | CanTurnBridge // bit 2
``` [3](#0-2) 

```go
CanTurnBridge: CanTurnBridge == (permissions & CanTurnBridge),
``` [4](#0-3) 

ADR-009 explicitly describes `MsgTurnBridge` as the mechanism for "disabling the bridge module in case of emergency":

> "changing the mapping to support new CRC20 auto-conversion contracts or disabling the bridge module in case of emergency" [5](#0-4) 

Two concrete defects exist simultaneously:

1. **No authorization check**: Any unprivileged address can submit `MsgTurnBridge` and receive a `nil` error (success). The `CanTurnBridge` permission bit is never consulted.
2. **No state write**: Even if the admin submits the message, no bridge-enabled flag is written to the KV store, so the bridge state can never be toggled off.

### Impact Explanation

**High — Bypass of Cronos admin/bridge authorization checks** and **permanent inability to disable the bridge**.

- Any unprivileged user can call `MsgTurnBridge` and receive a success response, bypassing the `CanTurnBridge` permission guard entirely.
- The admin/governance can never disable the bridge in an emergency. If a critical vulnerability is discovered in the IBC conversion or EVM hook logic, there is no on-chain mechanism to halt bridge flows. This is a permanent, irrecoverable loss of the circuit-breaker capability.

### Likelihood Explanation

The entry path is fully reachable: `MsgTurnBridge` is a registered, live transaction type exposed via gRPC and CLI. [6](#0-5)  Any unprivileged user can submit it at any time. The missing permission check and missing state write are both unconditional — no special conditions are required to trigger either defect.

### Recommendation

Implement `TurnBridge` analogously to `UpdateTokenMapping`:

1. Check `HasPermission(ctx, msg.GetSigners(), CanTurnBridge)` and return `ErrUnauthorized` if it fails.
2. Persist a bridge-enabled boolean to the KV store (e.g., under a new `KeyPrefixBridgeEnabled` key).
3. Gate `ConvertVouchersToEvmCoins` and `IbcTransferCoins` on that flag, returning an error when the bridge is disabled.

### Proof of Concept

1. Any unprivileged address submits `MsgTurnBridge{Sender: attacker, Enable: false}`.
2. The handler at line 85–87 returns `nil, nil` — the transaction succeeds on-chain with no error.
3. No state is written; the bridge remains fully operational.
4. The legitimate admin submits the same message — identical outcome: success response, no effect.
5. There is no code path anywhere in the keeper that reads a bridge-enabled flag, confirming the circuit breaker is permanently non-functional. [1](#0-0)
