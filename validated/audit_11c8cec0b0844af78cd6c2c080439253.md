I need to trace the full path from `block_id=Pending` through `read_pending_data` → `client_pending_data_to_execution_pending_data` → `create_block_context` → `tx_execution_output_to_fee_estimation`.

**Step 1: `read_pending_data` fallback**

**Attacker control assessment:**

The attacker controls `block_id=Pending` (a public RPC parameter). They do **not** control when the pending data is stale — that is a transient natural condition during block transitions. The attacker cannot force the stale state; they can only call the RPC during a window when it naturally occurs. This is a timing-dependent condition, not an on-demand trigger.

**Concrete wrong value:**

When stale, `l1_data_gas_price` and `l2_gas_price` in the returned `FeeEstimation` will be `NonzeroGasPrice::MIN` (value 1) instead of the actual latest block prices (which can be orders of magnitude larger). The fallback does copy `l1_gas_price` from the latest header correctly, but silently drops the other two price fields by choosing `DeprecatedPendingBlock` — a type that structurally cannot carry them.

**Is this a real vulnerability per the audit criteria?**

The design gap is real: the fallback at lines 1573–1593 has access to `latest_header.block_header_without_hash.l1_data_gas_price` and `l2_gas_price` but does not copy them, because it uses `DeprecatedPendingBlock` (which has no such fields) rather than a `Current` block. The result is that `estimateFee`/`simulateTransactions` with `block_id=Pending` returns authoritative-looking but wrong `l1_data_gas_price` and `l2_gas_price` during any stale-pending window.

However, the attacker cannot force the stale state — it is a transient condition. The impact is limited to the brief window between blocks when pending data hasn't been refreshed. Any user (not just an attacker) calling these endpoints during that window receives wrong prices. This is a correctness defect in the fallback, not an attacker-triggerable exploit.

### Title
Wrong `l1_data_gas_price` / `l2_gas_price` in `estimateFee` / `simulateTransactions` during stale-pending fallback — (`crates/apollo_rpc/src/v0_8/api/api_impl.rs`)

### Summary
When `read_pending_data` falls back to a synthetic `DeprecatedPendingBlock` (parent hash mismatch), it omits `l1_data_gas_price` and `l2_gas_price` from the latest accepted block header. Both fields default to zero, which `create_block_context` silently converts to `NonzeroGasPrice::MIN`. Every `FeeEstimation` returned by `estimateFee` and `simulateTransactions` with `block_id=Pending` during this window carries wrong prices for those two fields.

### Finding Description
`read_pending_data` constructs a `DeprecatedPendingBlock` when the cached pending data is stale. `DeprecatedPendingBlock` has no `l1_data_gas_price` or `l2_gas_price` fields; the accessor methods for the `Deprecated` variant return `GasPricePerToken::default()` (zero) by design for old block formats. The fallback copies `l1_gas_price` from the latest header but cannot copy the other two prices because the chosen type has no slots for them. Downstream, `create_block_context` converts zero to `NonzeroGasPrice::MIN` via `unwrap_or`, and `tx_execution_output_to_fee_estimation` reads those MIN values into the returned `FeeEstimation`.

### Impact Explanation
`FeeEstimation.l1_data_gas_price` and `l2_gas_price` will be `NonzeroGasPrice::MIN` (1 wei/fri) instead of the actual latest-block prices during any stale-pending window. Users relying on these estimates to set `max_price_per_unit` for `l1_data_gas` or `l2_gas` resource bounds may set bounds far too low, causing their transactions to be rejected by the sequencer. The values look authoritative (same JSON shape, no error returned).

### Likelihood Explanation
The stale window occurs naturally at every block transition. The attacker cannot force it on demand — they can only call the RPC during the naturally occurring window. This limits exploitability to a timing-dependent condition, making it a correctness defect rather than an on-demand attacker-triggered exploit.

### Recommendation
In the `read_pending_data` fallback, use `PendingBlockOrDeprecated::Current(PendingBlock { ... })` and populate `l1_data_gas_price` and `l2_gas_price` from `latest_header.block_header_without_hash.l1_data_gas_price` and `l2_gas_price`, mirroring how `l1_gas_price` is already copied.

### Proof of Concept
1. Write block N with non-zero `l1_data_gas_price` and `l2_gas_price` in the header.
2. Set `pending_data` with a `parent_block_hash` that does not match block N's hash (triggering the fallback).
3. Call `estimateFee` with `block_id=Pending`.
4. Observe that `FeeEstimation.l1_data_gas_price` and `l2_gas_price` equal `NonzeroGasPrice::MIN` (1), not the values stored in block N's header.
