Looking at the PoolEIC and Pool contract, I need to trace how `min_delegation_for_rewards` and `staking_rewards_base_value` behave for pre-V2 STRK pools after the V2→V3 upgrade, and whether rewards can be distributed before the lazy PoolEIC migration runs.

---

### Title
Uninitialized `staking_rewards_base_value` in Pre-V2 STRK Pools Causes Permanent Loss of Unclaimed Yield Before Lazy Migration - (`src/pool/pool.cairo`)

### Summary
STRK delegation pools created before V2 have `min_delegation_for_rewards` and `staking_rewards_base_value` initialized to zero. The `PoolEIC` that sets these values is applied **lazily** (via `staker_migration`), not atomically at upgrade time. If the attestation/staking reward pipeline calls `update_rewards_from_staking_contract` on an unmigrated pool before the PoolEIC runs, `compute_rewards_per_unit` silently returns zero for every reward epoch in that window. Those epochs are permanently recorded as zero in the `cumulative_rewards_trace`, and pool members can never recover the lost yield.

### Finding Description

The `PoolEIC` (`src/pool/eic.cairo`) is the contract responsible for writing `STRK_CONFIG.min_for_rewards` and `STRK_CONFIG.base_value` into pre-V2 STRK pools:

```cairo
// src/pool/eic.cairo lines 43-46
if min_delegation_for_rewards.is_zero() && staking_rewards_base_value.is_zero() {
    self.min_delegation_for_rewards.write(STRK_CONFIG.min_for_rewards);
    self.staking_rewards_base_value.write(STRK_CONFIG.base_value);
}
``` [1](#0-0) 

The `StakingEIC` stores `pool_eic_class_hash` explicitly for **lazy** pool migration — pools are not upgraded atomically at upgrade time:

```cairo
// src/staking/eic.cairo lines 54-56
// 3. Save pool EIC class hash.
assert!(pool_eic_class_hash.is_non_zero(), "{}", GenericError::ZERO_CLASS_HASH);
self.pool_eic_class_hash.write(pool_eic_class_hash);
``` [2](#0-1) 

Until the lazy `staker_migration` runs the PoolEIC on a given pool, both `min_delegation_for_rewards` and `staking_rewards_base_value` remain **zero** in that pool's storage.

The reward-recording path in `update_rewards_from_staking_contract` calls `compute_rewards_per_unit`:

```cairo
// src/pool/pool.cairo lines 967-978
fn compute_rewards_per_unit(
    self: @ContractState, staking_rewards: Amount, total_stake: Amount,
) -> Index {
    if total_stake < self.min_delegation_for_rewards.read() {
        return Zero::zero();
    }
    mul_wide_and_div(
        lhs: staking_rewards, rhs: self.staking_rewards_base_value.read(), div: total_stake,
    )
        .expect_with_err(err: InternalError::REWARDS_COMPUTATION_OVERFLOW)
}
``` [3](#0-2) 

With `staking_rewards_base_value == 0`:
- `min_delegation_for_rewards == 0` means `total_stake < 0` is always false (unsigned), so the early-return guard is bypassed.
- `mul_wide_and_div(staking_rewards, rhs: 0, div: total_stake)` evaluates to **0** without panicking.

The result is that `update_rewards_from_staking_contract` succeeds and inserts a **zero increment** into `cumulative_rewards_trace` for every reward epoch during the unmigrated window:

```cairo
// src/pool/pool.cairo lines 576-586
self.cumulative_rewards_trace.insert(
    key: self.get_current_epoch(),
    value: last + self.compute_rewards_per_unit(
        staking_rewards: rewards, total_stake: pool_balance,
    ),
);
``` [4](#0-3) 

Later, when `claim_rewards` calls `calculate_rewards`, it reads `staking_rewards_base_value` at call time (post-migration, now `10^28`), but `to_sigma - from_sigma == 0` for all pre-migration epochs because the trace was never incremented:

```cairo
// src/pool/pool.cairo lines 855, 870-873
let base_value = self.staking_rewards_base_value.read();
...
rewards += compute_rewards_rounded_down(
    amount: from_balance, interest: to_sigma - from_sigma, :base_value,
);
``` [5](#0-4) 

`compute_rewards_rounded_down(amount, interest: 0, base_value: 10^28)` returns 0. The rewards for the entire pre-migration window are **permanently unrecoverable**.

### Impact Explanation

Pool members of pre-V2 STRK pools lose all staking yield accrued between the V2→V3 upgrade and the moment their pool's lazy migration runs. The `cumulative_rewards_trace` is append-only; zero entries cannot be retroactively corrected. This constitutes **permanent freezing / theft of unclaimed yield** — a High-severity impact under the allowed scope.

### Likelihood Explanation

Every STRK pool deployed before V2 is affected. The attestation contract can trigger reward distribution at any block, and the lazy migration is only triggered by explicit staker interaction (e.g., calling `staker_migration`). Stakers who do not interact with the protocol immediately after the upgrade will silently lose yield for every reward epoch until they do. This is a realistic, medium-to-high likelihood scenario given that many stakers may not monitor upgrade events.

### Recommendation

Either:
1. **Eagerly migrate all pre-V2 STRK pools** during the upgrade (run PoolEIC for all existing pools atomically), or
2. **Guard `update_rewards_from_staking_contract`** to revert (or skip reward recording) when `staking_rewards_base_value == 0`, forcing migration before any reward epoch is recorded:
   ```cairo
   assert!(self.staking_rewards_base_value.read().is_non_zero(), "Pool not migrated");
   ```
3. Alternatively, trigger `staker_migration` automatically inside the staking contract's reward distribution path before calling `update_rewards_from_staking_contract` on any pool.

### Proof of Concept

1. Pre-upgrade: STRK pool P was created in V0/V1. `min_delegation_for_rewards == 0`, `staking_rewards_base_value == 0`.
2. V2→V3 upgrade runs `StakingEIC`. Pool P is **not** immediately migrated; `pool_eic_class_hash` is stored for lazy migration.
3. Attestation contract triggers reward distribution. Staking contract calls `P.update_rewards_from_staking_contract(rewards=R, pool_balance=B)`.
4. Inside `compute_rewards_per_unit`: `mul_wide_and_div(R, 0, B) == 0`. Cumulative trace records `last + 0`.
5. Steps 3–4 repeat for N epochs before any staker calls `staker_migration`.
6. Staker eventually calls `staker_migration`. PoolEIC runs, sets `staking_rewards_base_value = 10^28`.
7. Pool member calls `claim_rewards`. `calculate_rewards` computes `interest = 0` for all N pre-migration epochs → `rewards = 0` for that window.
8. Pool member permanently loses N epochs of yield. [6](#0-5) [7](#0-6) [8](#0-7)
