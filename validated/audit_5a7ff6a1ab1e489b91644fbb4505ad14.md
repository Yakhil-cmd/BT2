### Title
Missing `max_l2_gas_amount` Validation for Declare Transactions Allows Gateway Admission of Out-of-Bounds Resource Bounds - (File: `crates/apollo_gateway/src/stateless_transaction_validator.rs`)

### Summary

`StatelessTransactionValidator::validate_resource_bounds` explicitly skips the `max_l2_gas_amount` upper-bound check for `RpcTransaction::Declare` transactions. Every other transaction type (Invoke, DeployAccount) is rejected if `l2_gas.max_amount` exceeds the configured limit (production default: 1,210,000,000). A Declare transaction with an arbitrarily large `l2_gas.max_amount` passes stateless validation and proceeds through the gateway's stateful pre-validation path, violating the admission invariant that all transactions must satisfy `l2_gas.max_amount ≤ max_l2_gas_amount`.

### Finding Description

In `validate_resource_bounds`, the `max_l2_gas_amount` guard is structured as:

```rust
// TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
if let RpcTransaction::Declare(_) = tx {
} else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh { … });
}
``` [1](#0-0) 

The empty `if let RpcTransaction::Declare(_) = tx {}` branch is a deliberate no-op that causes the `max_l2_gas_amount` check to be silently skipped for every Declare transaction. The production configuration sets `max_l2_gas_amount = 1_210_000_000`: [2](#0-1) 

The test `valid_l2_gas_amount_on_declare` explicitly confirms this bypass is reachable: a Declare transaction with `max_amount: GasAmount(200)` passes when `max_l2_gas_amount: 100`, while the same bounds on an Invoke or DeployAccount transaction would be rejected: [3](#0-2) 

The gateway's stateful pre-validation path (`validate_state_preconditions`) also performs no `max_l2_gas_amount` check for Declare transactions: [4](#0-3) 

A Declare transaction with `l2_gas.max_amount` set to any value above the limit (e.g., `1_210_000_001`) therefore passes both stateless and stateful gateway pre-validation and is admitted to the mempool.

### Impact Explanation

**High — Mempool/gateway admission accepts invalid Declare transactions that should be rejected before sequencing.**

The `max_l2_gas_amount` limit is the gateway's primary DoS guard against transactions claiming an unreasonably large L2 gas budget. For Invoke and DeployAccount transactions this guard is enforced; for Declare it is absent. An attacker can submit Declare transactions with `l2_gas.max_amount` set to any value up to `u64::MAX` and they will clear the stateless and stateful gateway admission checks.

For values of `max_amount` that, when multiplied by `max_price_per_unit`, remain within `u128`, `verify_can_pay_committed_bounds` (called inside the blockifier during full Declare execution) computes `max_possible_fee` via saturating arithmetic: [5](#0-4) 

If the product saturates to `u128::MAX`, the balance check rejects the transaction. However, for `max_amount` values only modestly above the limit (e.g., `1_210_000_001`) with a realistic `max_price_per_unit`, the product stays within `u128`, the balance check passes if the account has sufficient funds, and the transaction is fully admitted and sequenced — bypassing the admission invariant.

### Likelihood Explanation

Any unprivileged user can submit a Declare transaction via the public RPC endpoint. The bypass requires only setting `l2_gas.max_amount` to a value above `max_l2_gas_amount` in the transaction's `resource_bounds` field. No special account state or privileged access is needed. The TODO comment in the source confirms the gap is known but unresolved.

### Recommendation

Remove the empty `if let RpcTransaction::Declare(_) = tx {}` branch and apply the same `max_l2_gas_amount` upper-bound check uniformly to all transaction types, including Declare:

```rust
if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
        gas_amount: resource_bounds.l2_gas.max_amount,
        max_gas_amount: self.config.max_l2_gas_amount,
    });
}
```

If Declare transactions legitimately require a higher L2 gas ceiling, introduce a separate `max_l2_gas_amount_declare` configuration parameter rather than removing the check entirely.

### Proof of Concept

1. Construct a valid `RpcDeclareTransactionV3` with `resource_bounds.l2_gas.max_amount = GasAmount(1_210_000_001)` (one above the production limit) and `resource_bounds.l2_gas.max_price_per_unit` set to any value ≥ `min_gas_price`.
2. Submit it to the gateway's `add_tx` endpoint.
3. Observe that `StatelessTransactionValidator::validate` returns `Ok(())` — the `MaxGasAmountTooHigh` error is never raised for Declare transactions.
4. The transaction proceeds through `extract_state_nonce_and_run_validations`, passes `validate_state_preconditions`, and is admitted to the mempool.
5. For comparison, submit an identical `RpcInvokeTransactionV3` with the same `l2_gas.max_amount`; it is rejected with `StatelessTransactionValidatorError::MaxGasAmountTooHigh`.

The existing test `valid_l2_gas_amount_on_declare` already demonstrates step 3 in isolation: [3](#0-2)
