### Title
ERC-20 and ETH Tokens Permanently Lost When `ExitToNear` Promise Fails Without `error_refund` Feature - (File: `engine-precompiles/src/native.rs`)

### Summary

The `ExitToNear` precompile burns ERC-20 tokens (or deducts ETH) from a user's EVM balance during execution and then schedules a NEAR `ft_transfer` promise. The refund mechanism that would re-mint tokens on promise failure is entirely gated behind the `error_refund` compile-time feature flag. Because `error_refund` is **not** in the `default` feature set of either `engine/Cargo.toml` or `engine-precompiles/Cargo.toml`, the production contract is compiled without it. When the `ft_transfer` promise fails (e.g., the recipient NEAR account is not registered with the NEP-141 contract), the burned tokens are permanently lost with no recovery path.

### Finding Description

In `engine-precompiles/src/native.rs`, the `ExitToNear::run()` function constructs `callback_args` as follows:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,          // always None in production
    transfer_near: transfer_near_args,
};
``` [1](#0-0) 

For the two most common exit paths — regular ERC-20 `ft_transfer` and ETH `ft_transfer` — `transfer_near_args` is also `None` (returned from `exit_erc20_token_to_near` and `exit_base_token_to_near` respectively). [2](#0-1) [3](#0-2) 

When both fields are `None`, `callback_args` equals the `Default` value, so the code takes the branch that creates a bare promise **with no callback**:

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // no callback attached
} else {
    PromiseArgs::Callback(...)
};
``` [4](#0-3) 

If the `ft_transfer` promise fails, there is no callback to invoke `engine::refund_on_error`, so the burned ERC-20 tokens (or deducted ETH) are never returned.

Even in the wNEAR-unwrap path where `transfer_near` is `Some` and a callback **is** attached, the failure branch in `exit_to_near_precompile_callback` silently does nothing because `args.refund` is `None`:

```rust
} else if let Some(args) = args.refund {
    // refund tokens -- NEVER REACHED without error_refund feature
    ...
} else {
    None   // tokens are gone
};
``` [5](#0-4) 

The `error_refund` feature is confirmed absent from both default feature sets: [6](#0-5) [7](#0-6) 

### Impact Explanation

**Critical — Permanent loss of user funds.**

Any ERC-20 or ETH exit that results in a failed `ft_transfer` NEAR promise causes the user's tokens to be permanently destroyed. The EVM-side burn is committed (it is part of the EVM state diff that is persisted), but the NEAR-side credit never materialises and there is no on-chain mechanism to recover the funds.

### Likelihood Explanation

**High.** The `ft_transfer` promise fails whenever the recipient NEAR account is not registered as a storage depositor with the NEP-141 token contract. This is a routine condition: any user who exits to a freshly created NEAR account, a contract that has not called `storage_deposit`, or a mistyped account ID will trigger the failure. The existing test `test_exit_to_near_refund` in `engine-tests/src/tests/erc20_connector.rs` explicitly demonstrates the loss:

```rust
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
// i.e. the exit amount is gone when the feature is absent
``` [8](#0-7) 

The same pattern is confirmed for ETH exits: [9](#0-8) 

### Recommendation

Enable the `error_refund` feature unconditionally in the production build by moving it into the `default` feature list of `engine-precompiles/Cargo.toml` and `engine/Cargo.toml`, or by removing the `#[cfg]` guards and always populating the `refund` field in `ExitToNearPrecompileCallbackArgs`. The `refund_call_args` helper and `engine::refund_on_error` already implement the correct recovery logic; they simply need to be reachable in the production binary. [10](#0-9) 

### Proof of Concept

1. User holds 100 ERC-20 tokens on Aurora and calls the `ExitToNear` precompile targeting a NEAR account that is not registered with the NEP-141 contract.
2. The EVM executes the burn: the user's ERC-20 balance is reduced by 100 tokens (state diff committed).
3. `ExitToNear::run()` constructs `callback_args` with `refund: None` and `transfer_near: None`; since both are `None`, a bare `PromiseArgs::Create(ft_transfer_promise)` is emitted — no callback.
4. The NEAR runtime executes `ft_transfer`; it panics because the recipient is not registered.
5. There is no callback promise to catch the failure and invoke `refund_on_error`.
6. The user's 100 ERC-20 tokens are permanently destroyed; the NEAR-side balance is unchanged.

```rust
pub fn refund_on_error<I: IO + Copy, E: Env, P: PromiseHandler>(
    io: I,
    env: &E,
    state: EngineState,
    args: &RefundCallArgs,
    handler: &mut P,
) -> EngineResult<SubmitResult> {
    let current_account_id = env.current_account_id();
    if let Some(erc20_address) = args.erc20_address {
        // ERC-20 exit; re-mint burned tokens
        let erc20_admin_address = current_address(&current_account_id);
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, erc20_admin_address, current_account_id, io, env);

        let refund_address = args.recipient_address;
        let amount = U256::from_big_endian(&args.amount);
        let input = setup_refund_on_error_input(amount, refund_address);

        engine.call(
            &erc20_admin_address,
            &erc20_address,
            Wei::zero(),
            input,
            u64::MAX,
            Vec::new(),
            Vec::new(),
            handler,
        )
    } else {
        // ETH exit; transfer ETH back from precompile address
        let exit_address = exit_to_near::ADDRESS;
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, exit_address, current_account_id, io, env);
        let refund_address = args.recipient_address;
        let amount = Wei::new(U256::from_big_endian(&args.amount));
        engine.call(
            &exit_address,
            &refund_address,
            amount,
            Vec::new(),
            u64::MAX,
            vec![
                (exit_address.raw(), Vec::new()),
                (refund_address.raw(), Vec::new()),
            ],
            Vec::new(),
            handler,
        )
    }
