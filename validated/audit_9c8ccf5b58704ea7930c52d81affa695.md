### Title
Gateway Skips `__validate__` Signature Check for Any Account Known to Mempool, Allowing Admission of Signature-Invalid Invoke Transactions — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

`skip_stateful_validations` in the gateway's stateful validation path returns `true` (skip `__validate__`) for any invoke transaction with `tx.nonce == 1` and on-chain `account_nonce == 0`, provided `account_tx_in_pool_or_recent_block` returns `true`. Because that mempool check is satisfied by **any** transaction from the account address — not specifically a deploy-account transaction — an unprivileged attacker who observes a victim's pending `deploy_account` in the mempool can submit an invoke transaction with an arbitrary, invalid signature for that address and have it admitted to the mempool without any signature verification.

### Finding Description

The sequential validation steps in `extract_state_nonce_and_run_validations` are:

1. `get_nonce_from_state` — reads on-chain nonce (= 0 for undeployed account)
2. `run_pre_validation_checks` — validates nonce range, resource bounds, mempool nonce state
3. `skip_stateful_validations` — if `tx.nonce == 1 && account_nonce == 0 && account_tx_in_pool_or_recent_block(sender)` → returns `true`
4. `run_validate_entry_point` — if `skip_validate == true`, sets `execution_flags.validate = false`, so `__validate__` (signature verification) is **never called** [1](#0-0) 

The mempool-side predicate is:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [2](#0-1) 

This returns `true` for **any** account that has ever had a transaction in the pool or a committed block — it does not verify that the pending transaction is specifically a `deploy_account`. The code comment claims this is sufficient because "it means that either it has a deploy_account transaction or transactions with future nonces that passed validations," but that reasoning is circular: future-nonce invoke transactions themselves pass through this same skip path.

In `run_validate_entry_point`, when `skip_validate = true`:

```rust
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [3](#0-2) 

`validate = false` is passed into `StatefulValidator::perform_validations`, which then returns `Ok(())` without ever calling `__validate__`:

```rust
if !tx.execution_flags.validate {
    return Ok(());
}
``` [4](#0-3) 

The transaction is then forwarded to the mempool and admitted. The signature field is never inspected.

### Impact Explanation

An invoke transaction carrying an **arbitrary, attacker-chosen signature** for a victim's undeployed account address is admitted to the mempool and forwarded to the batcher. The gateway's stateful validation — the only place where `__validate__` is called before sequencing — is bypassed entirely. The transaction will fail at block-execution time when the blockifier calls `__validate__` unconditionally, but it has already been accepted as a valid pending transaction, consuming mempool capacity and block space. This directly satisfies:

> **High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.**

### Likelihood Explanation

The attack requires only:
1. Observing the public mempool for any pending `deploy_account` transaction targeting address A (zero privilege required — the mempool is observable via RPC).
2. Constructing an invoke transaction with `sender = A`, `nonce = 1`, any calldata, and any signature bytes.
3. Submitting it to the gateway.

No special access, no key material, no on-chain funds are needed. The window is open for the entire lifetime of the `deploy_account` transaction in the mempool (potentially many blocks).

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a **`deploy_account` transaction** is pending for the sender address. The mempool should expose a dedicated `has_pending_deploy_account(address)` query, or the gateway should inspect the transaction type of the pending transaction before deciding to skip `__validate__`. Alternatively, restrict the skip to cases where the gateway itself issued the `deploy_account` transaction in the same request batch (i.e., the UX flow where both transactions arrive together).

### Proof of Concept

```
1. Victim calls POST /add_transaction with a valid deploy_account for address A.
   → Gateway admits it; mempool now has A in tx_pool.

2. Attacker calls POST /add_transaction with:
     type:   invoke_v3
     sender: A
     nonce:  1
     calldata: [arbitrary]
     signature: [0xdeadbeef, 0xdeadbeef]   ← completely invalid

3. Gateway stateful validation:
   - get_nonce_from_state(A) → 0   (A not yet deployed on-chain)
   - validate_nonce: 0 ≤ 1 ≤ 0+gap  → OK
   - validate_by_mempool: nonce/fee checks pass
   - skip_stateful_validations:
       tx.nonce == 1 ✓  account_nonce == 0 ✓
       account_tx_in_pool_or_recent_block(A) → true (deploy_account is in pool) ✓
       → returns true
   - run_validate_entry_point: validate=false → __validate__ NOT called → Ok(())

4. Gateway returns tx_hash to attacker.
   Attacker's signature-invalid invoke is now in the mempool.

5. At block execution time:
   - deploy_account executes, A is deployed.
   - invoke executes: __validate__ is called, fails (bad signature).
   - Transaction reverts; block space wasted.
``` [5](#0-4) [6](#0-5) [7](#0-6)

        let account_tx = AccountTransaction { tx: executable_tx.clone(), execution_flags };

        // Build block context.

        versioned_constants.disable_casm_hash_migration();

        let mut block_info = self.gateway_fixed_block_state_reader.get_block_info().await?;
        block_info.block_number = block_info.block_number.unchecked_next();

            self.chain_info.clone(),
            versioned_constants,

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
