### Title
STRK Rewards Permanently Frozen in Pool Contract When Pool Balance Falls Below `min_delegation_for_rewards` — (File: `src/pool/pool.cairo`)

---

### Summary

When a delegation pool's total balance is below `min_delegation_for_rewards` (1 STRK = 10^18 wei), `compute_rewards_per_unit` returns zero and the `cumulative_rewards_trace` is not updated. However, the staking contract has already transferred the calculated STRK reward tokens to the pool contract. Those tokens become permanently frozen — no pool member can claim them, and no recovery path exists.

---

### Finding Description

In `pool.cairo`, `compute_rewards_per_unit` guards against overflow with a minimum-balance threshold:

```cairo
fn compute_rewards_per_unit(
    self: @ContractState, staking_rewards: Amount, total_stake: Amount,
) -> Index {
    // Return zero if the total stake is too small, to avoid overflow below.
    if total_stake < self.min_delegation_for_rewards.read() {
        return Zero::zero();
    }
    ...
}
``` [1](#0-0) 

For STRK pools, `min_delegation_for_rewards` is set to `10^18` (1 STRK) in the constructor via `STRK_CONFIG.min_for_rewards`: [2](#0-1) 

When `compute_rewards_per_unit` returns zero, `update_rewards_from_staking_contract` inserts `last + 0` into the `cumulative_rewards_trace`, recording no reward increase for that epoch: [3](#0-2) 

The critical ordering problem is in `update_pool_rewards` in `staking.cairo`: STRK tokens are **transferred to the pool contract first**, and only then is `update_rewards_from_staking_contract` called: [4](#0-3) 

Because `cumulative_rewards_trace` shows no increment, pool members computing rewards via `calculate_rewards` receive zero for that epoch. The transferred STRK tokens are now in the pool contract's balance but are unclaimable through any existing function. The code's own comment acknowledges this:

> **Note**: Delegation rewards lost when pool balance is less than `min_delegation_for_rewards`. The staking contract continues to forward `pool_rewards` to the pool contract even in this case. [5](#0-4) 

The `calculate_staker_pools_rewards` function in `staking.cairo` only skips a pool when `pool_rewards.is_non_zero()` is false — it does **not** check whether the pool balance is below the minimum threshold before forwarding rewards: [6](#0-5) 

---

### Impact Explanation

STRK tokens representing pool rewards are permanently frozen inside the pool contract. `claim_rewards` computes rewards from the cumulative trace (which shows zero increment for the affected epoch), so it returns nothing. `exit_delegation_pool_action` only returns the staked principal (`unpool_amount`), not any stuck reward balance. There is no sweep or recovery function. This is **permanent freezing of unclaimed yield**.

---

### Likelihood Explanation

The minimum delegation amount enforced by `enter_delegation_pool` is only `amount.is_non_zero()` — a delegator can enter with 1 wei. Any pool whose total balance is below 1 STRK (10^18 wei) triggers the freeze. This is realistic for newly created pools or pools with few small delegators. The trigger is the staker's normal attestation flow (`update_rewards_from_attestation_contract`) or the consensus rewards path (`update_rewards`), both of which are called every epoch. No special attacker action is required beyond holding a small delegation.

---

### Recommendation

In `calculate_staker_pools_rewards` (`staking.cairo`), skip pools whose native balance is below the pool's `min_delegation_for_rewards` before calculating or forwarding any rewards. This prevents the staking contract from transferring tokens that the pool contract cannot distribute:

```cairo
// Before appending to pool_rewards_array, check:
let decimals = self.get_token_decimals(:token_address);
let native_pool_balance = pool_balance_curr_epoch.to_native_amount(:decimals);
// Only forward rewards if pool can distribute them
if native_pool_balance >= pool_min_delegation_for_rewards {
    pool_rewards_array.append(...);
}
```

Alternatively, in `update_rewards_from_staking_contract` (`pool.cairo`), if `compute_rewards_per_unit` returns zero, the function should revert or return the tokens rather than silently accepting them.

---

### Proof of Concept

1. Staker creates a STRK pool via `set_open_for_delegation`.
2. Delegator calls `enter_delegation_pool` with 0.5 STRK (5 × 10^17 wei) — below `min_delegation_for_rewards = 10^18`.
3. After K epochs, staker attests; `update_rewards_from_attestation_contract` is called.
4. `calculate_staker_pools_rewards` computes a non-zero `pool_rewards` (e.g., 0.001 STRK), since `pool_balance > 0` and `total_rewards > 0`.
5. `update_pool_rewards` transfers 0.001 STRK to the pool contract via `send_rewards_to_delegation_pool`.
6. `update_rewards_from_staking_contract(rewards: 0.001 STRK, pool_balance: 5×10^17)` is called.
7. `compute_rewards_per_unit` returns 0 because `5×10^17 < 10^18`.
8. `cumulative_rewards_trace` is updated with `last + 0` — no change.
9. Delegator calls `claim_rewards` → receives 0 STRK.
10. The 0.001 STRK is permanently frozen in the pool contract with no recovery path.

            if total_stake < self.min_delegation_for_rewards.read() {
