Audit Report

## Title
SNS Governance Voting Rewards Purse Permanently Lost When `total_reward_shares` Is Zero With Non-Empty Settled Proposals - (File: `rs/sns/governance/src/governance.rs`)

## Summary
In SNS Governance's `distribute_rewards`, when proposals exist in `ReadyToSettle` state but no neuron voted on any of them (`total_reward_shares == dec!(0)`), the computed `rewards_purse_e8s` is silently consumed: proposals are marked as settled and `latest_reward_event` is updated with non-empty `settled_proposals`, but zero maturity is distributed and the purse is not rolled over. In the next round, `e8s_equivalent_to_be_rolled_over()` returns 0 because `rewards_rolled_over()` checks only `settled_proposals.is_empty()`, not whether rewards were actually distributed. The entire purse — including any previously accumulated rollover — is permanently destroyed.

## Finding Description
`distribute_rewards` computes `rewards_purse_e8s` by seeding from `self.latest_reward_event().e8s_equivalent_to_be_rolled_over()` and adding the current round's interest on total token supply. [1](#0-0) 

It then tallies exercised voting power. When `total_reward_shares == dec!(0)`, it logs a warning and skips maturity distribution, leaving `distributed_e8s_equivalent = 0`. [2](#0-1) 

Regardless of whether distribution occurred, the function always concludes by writing a new `latest_reward_event` with `settled_proposals = considered_proposals` (non-empty): [3](#0-2) 

The rollover predicate in SNS `types.rs` is gated solely on `settled_proposals.is_empty()`: [4](#0-3) 

And `e8s_equivalent_to_be_rolled_over()` returns 0 when `rewards_rolled_over()` is false: [5](#0-4) 

Because `settled_proposals` is non-empty after the zero-distribution event, `e8s_equivalent_to_be_rolled_over()` returns 0 in the next round. The entire `rewards_purse_e8s` — including any previously rolled-over amounts from prior empty rounds — is permanently lost. The identical structural flaw exists in NNS Governance's `calculate_voting_rewards` when `total_voting_rights < 0.001` with non-empty `considered_proposals`. [6](#0-5) 

## Impact Explanation
SNS token holders who staked neurons lose their voting rewards for the affected round. The `rewards_purse_e8s` — proportional to `total_token_supply × reward_rate × round_duration` — is permanently destroyed with no on-chain recovery path. Any previously accumulated rollover purse from prior empty rounds is also lost in the same event. This constitutes a **permanent, irreversible loss of SNS governance reward funds** with concrete financial harm to all staked neuron holders, matching the High impact category: "Significant SNS security impact with concrete user or protocol harm."

## Likelihood Explanation
For SNS Governance: an SNS can have proposals enter `ReadyToSettle` state without any neuron casting a vote — e.g., all neurons abstain, follow a neuron that does not vote, or the SNS is newly launched with low participation. No privileged action is required; `run_periodic_tasks` fires automatically. This is a realistic operational scenario, especially for newly deployed SNS instances. The trigger is fully reachable by an unprivileged user (anyone who can submit a proposal and let the voting period expire with no participation). For NNS Governance: the `total_voting_rights < 0.001` condition is extremely unlikely in the live NNS but theoretically possible.

## Recommendation
In `distribute_rewards` (SNS) and `calculate_voting_rewards` (NNS), when `total_reward_shares == 0` (or `total_voting_rights < 0.001`) with non-empty `considered_proposals`, the rewards purse must be preserved. The safest fix is **Option 1**: do not settle the proposals in this case — skip writing the `latest_reward_event` and leave proposals in `ReadyToSettle` so the next round can attempt distribution. Alternatively, **Option 2**: modify `rewards_rolled_over()` to also return `true` when `distributed_e8s_equivalent == 0` regardless of whether `settled_proposals` is empty, so `e8s_equivalent_to_be_rolled_over()` correctly carries forward the purse. Option 1 is safer as it avoids changing rollover accounting semantics.

## Proof of Concept
1. Deploy an SNS with non-zero `voting_rewards_parameters` (reward rate > 0).
2. Create a proposal; let it reach `ReadyToSettle` state (voting period expires) with **no neuron casting a vote**.
3. Advance time past `round_duration_seconds`.
4. `run_periodic_tasks` fires → `should_distribute_rewards()` returns `true` → `distribute_rewards(supply)` is called.
5. `considered_proposals` is non-empty (the expired proposal).
6. `neuron_id_to_reward_shares` is empty (no votes cast) → `total_reward_shares == dec!(0)`.
7. The `if total_reward_shares == dec!(0)` branch logs a warning and skips maturity distribution.
8. `latest_reward_event` is written with `settled_proposals = [proposal_id]`, `distributed_e8s_equivalent = 0`, `total_available_e8s_equivalent = Some(purse)`.
9. In the next round, `rewards_rolled_over()` returns `false` (because `settled_proposals` is non-empty), so `e8s_equivalent_to_be_rolled_over()` returns `0`.
10. The entire `rewards_purse_e8s` from step 4 is permanently lost.

A deterministic unit test can be written using `NativeEnvironment` in `rs/sns/governance/src/governance.rs` test harness: set up a governance instance with one proposal in `ReadyToSettle`, no ballots cast, advance time by one round, call `distribute_rewards`, then assert that `latest_reward_event.total_available_e8s_equivalent > 0` but the next call to `e8s_equivalent_to_be_rolled_over()` returns `0` — confirming the purse is lost.
