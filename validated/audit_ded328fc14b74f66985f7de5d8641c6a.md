### Title
COA Ownership Proof Signature Replay Across Different EVM Addresses — (`File: fvm/evm/stdlib/contract.cdc`)

---

### Summary

`EVM.validateCOAOwnershipProof` does not enforce that `signedData` encodes the target `evmAddress`. Any valid `(signedData, signatures)` pair observed on-chain for one COA can be replayed against a different COA owned by the same Cadence account, causing the function to return `isValid: true` for an address the signer never authorized.

---

### Finding Description

`EVM.validateCOAOwnershipProof` is an `access(all)` Cadence function that verifies a set of Flow account key signatures over an arbitrary `signedData` blob, then checks that the account's COA resource at the given `path` matches the supplied `evmAddress`. [1](#0-0) 

The signature verification step verifies the signatures only over `signedData`: [2](#0-1) 

`evmAddress` is **never included in the signed payload**. The function itself documents this gap: [3](#0-2) 

Because the function is `access(all)`, any unprivileged Cadence script or transaction can call it with an arbitrary `evmAddress` while supplying a previously observed `signedData`+`signatures` pair. If the same Cadence account owns two COAs (stored at different `PublicPath`s), a proof produced for COA-A passes validation when submitted with `evmAddress` = COA-B.

The EVM-side precompile path (`verifyCOAOwnershipProof` in `coa.sol`) is not directly affected because the COA contract always passes `address(this)`: [4](#0-3) 

However, the Cadence-callable surface is fully attacker-controlled.

---

### Impact Explanation

**Impact: Medium**

Any protocol or dApp that calls `EVM.validateCOAOwnershipProof` directly from a Cadence script or transaction to gate privileged operations (e.g., bridge withdrawals, off-chain authentication, DAO voting) can be deceived into accepting a proof for a COA the signer never authorized. The attacker does not need any private key — only a previously observed valid proof for any COA of the target Cadence account.

The `COAOwnershipProofInContext` type used by the Go-layer validator passes all fields including `EVMAddress` to the Cadence function: [5](#0-4) 

But the Cadence function never hashes `evmAddress` into the signed message, so the address binding is absent at the cryptographic level.

---

### Likelihood Explanation

**Likelihood: Medium**

- The function is `access(all)` — reachable from any unprivileged Cadence script or transaction without any special capability.
- The code comment explicitly acknowledges the cross-address replay risk, indicating the design team is aware it is a real scenario.
- A Cadence account can own multiple COAs (the protocol does not limit this to one per account).
- Any valid proof is permanently observable on-chain once submitted in a transaction, giving an attacker an unlimited replay window. [3](#0-2) 

---

### Recommendation

Enforce inside `validateCOAOwnershipProof` that `signedData` encodes `evmAddress` before accepting the proof, rather than leaving this as a caller responsibility. Concretely, reconstruct the expected signed payload as `hash(evmAddress || callerSuppliedData)` and verify signatures against that, or reject calls where `signedData` does not contain the 20-byte `evmAddress` as a prefix/suffix. This mirrors the fix recommended in the external report: bind the signed hash to the specific target address.

---

### Proof of Concept

1. Cadence account `A` owns two COAs: `COA-X` at `/public/coaX` and `COA-Y` at `/public/coaY`.
2. Account `A` produces a valid proof for `COA-X`: signs `data = randomBytes` with their Flow key, producing `sig`.
3. This proof `(address: A, path: /public/coaX, signedData: data, keyIndices: [0], signatures: [sig], evmAddress: COA-X)` is submitted on-chain and becomes publicly observable.
4. An attacker calls `EVM.validateCOAOwnershipProof(address: A, path: /public/coaY, signedData: data, keyIndices: [0], signatures: [sig], evmAddress: COA-Y)`.
5. The signature check at line 1082–1086 passes because `sig` is valid over `data` regardless of which COA is targeted.
6. The COA address check at line 1095–1105 passes because `path: /public/coaY` resolves to `COA-Y` and `evmAddress: COA-Y` matches.
7. The function returns `ValidationResult(isValid: true)` — falsely asserting that account `A` authorized `COA-Y` with this proof. [6](#0-5)
