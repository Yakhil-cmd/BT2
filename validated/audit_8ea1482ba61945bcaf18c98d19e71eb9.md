### Title
Missing Validation of Participant Contributions in CKD Protocol Allows Malicious Participant to Corrupt Confidential Key Derivation Output - (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

The `do_ckd_coordinator` function in the CKD protocol accumulates group-element contributions (`big_y`, `big_c`) from every participant with no cryptographic verification whatsoever. A single malicious participant can substitute arbitrary curve points, causing the coordinator to assemble and return a corrupted `CKDOutput` that all honest parties accept as the legitimate protocol result.

---

### Finding Description

In `do_ckd_coordinator`, the coordinator collects each participant's `(norm_big_y, norm_big_c)` pair and blindly adds them into the running sum:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
Ok(Some(ckd_output))
``` [1](#0-0) 

No check is performed on the received values:

- **No identity-element check** — a participant may send the G1 identity point for `big_y` or `big_c`, analogous to the missing `require(updatedAt > 0)` / `latestPrice > 0` checks in the Chainlink report.
- **No proof of correct computation** — there is no ZK proof, commitment, or any other binding that forces a participant to compute `(λ_i · y_i · G, λ_i · (x_i · H(pk, app_id) + y_i · app_pk))` honestly.

This stands in direct contrast to the DKG protocol, which enforces proofs of knowledge and commitment-hash binding before accepting any participant's polynomial contribution: [2](#0-1) 

The `compute_signature_share` function that honest participants call produces the correct Lagrange-weighted pair: [3](#0-2) 

A malicious participant bypasses this entirely and sends whatever curve points they choose.

---

### Impact Explanation

The CKD protocol is designed so that the coordinator's output `(Y, C)` satisfies:

```
C − app_sk · Y  =  msk · H(pk, app_id)
```

where `msk` is the master secret key reconstructed from the threshold shares. If participant `P_m` sends `big_c_malicious = big_c_correct + Δ` for an attacker-chosen `Δ ∈ G1`, the coordinator computes `C_corrupted = C_correct + Δ`, and the application unmasks:

```
C_corrupted − app_sk · Y  =  msk · H(pk, app_id) + Δ
```

The resulting confidential key is wrong. The attacker knows `Δ`, so they know the exact deviation from the true key, but honest parties have no way to detect the corruption — the protocol produces no output that can be independently verified. This maps directly to **High: Corruption of CKD outputs so honest parties accept unusable cryptographic outputs**.

---

### Likelihood Explanation

Any single participant in the CKD protocol can trigger this. The attacker needs only to be a legitimate protocol participant (no special privilege required) and to deviate from the honest computation when sending their private message to the coordinator. The coordinator has no mechanism to distinguish a correct contribution from a malicious one.

---

### Recommendation

1. **Require a ZK proof of correct computation** alongside each participant's `(norm_big_y, norm_big_c)` — specifically a proof that `big_c` was formed as `λ_i · (x_i · H(pk, app_id) + y_i · app_pk)` with respect to the publicly known `big_y = λ_i · y_i · G` and the participant's committed public key share. The `dlogeq` proof primitive already present in the codebase (`src/crypto/proofs/dlogeq.rs`) is suitable for this. [4](#0-3) 

2. **At minimum, reject identity-element contributions** — check `big_y.is_identity()` and `big_c.is_identity()` before adding them to the running sum, mirroring the identity checks already enforced in the BLS ciphersuite serialization: [5](#0-4) 

---

### Proof of Concept

1. Honest participants `P_1 … P_{n-1}` run `compute_signature_share` correctly and send their outputs to the coordinator.
2. Malicious participant `P_m` skips `compute_signature_share` and instead sends `CKDOutput::new(G1::identity(), G1::identity())` (or any arbitrary `Δ`-shifted pair) directly to the coordinator.
3. `do_ckd_coordinator` adds the malicious values into `norm_big_y` and `norm_big_c` without error.
4. The coordinator returns `CKDOutput::new(norm_big_y, norm_big_c)` — a value that does not equal `(Y_correct, C_correct)`.
5. The application calls `ckd_output.unmask(app_sk)` and obtains `msk · H(pk, app_id) + Δ` instead of the true confidential key, silently accepting a corrupted result.

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))

/// Produce a proof for the given statement and witness, using a caller-provided nonce.
/// The nonce `k` must be sampled from a cryptographically secure RNG by the caller.
/// The challenge is derived via the Fiat-Shamir transform over the transcript.
pub fn prove_with_nonce<C: Ciphersuite>(
    transcript: &mut Transcript,
    statement: Statement<'_, C>,
    witness: Witness<C>,
    k: Scalar<C>,
) -> Result<Proof<C>, ProtocolError>
where
    Element<C>: ConstantTimeEq,
{
    if statement.generator1.ct_eq(&C::Group::identity()).into() {
        return Err(ProtocolError::IdentityElement);
    }

    transcript.message(NEAR_DLOGEQ_STATEMENT_LABEL, &statement.encode()?);

    let (big_k_0, big_k_1) = statement.phi(&k);

    // This will never raise error as k is not zero and generator1 is not the identity
    let enc = encode_two_points::<C>(&big_k_0, &big_k_1)?;

    transcript.message(NEAR_DLOGEQ_COMMITMENT_LABEL, &enc);
    let mut rng = transcript.challenge_then_build_rng(NEAR_DLOGEQ_CHALLENGE_LABEL);
    let e = frost_core::random_nonzero::<C, _>(&mut rng);

    let s = k + e * witness.x.0;
    Ok(Proof {
        e: SerializableScalar::<C>(e),
        s: SerializableScalar::<C>(s),
    })
}
