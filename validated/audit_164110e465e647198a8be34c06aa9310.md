### Title
`native_per_pubdata` Rounds Down Instead of Up, Undercharging Native Resources for Pubdata - (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

### Summary

`native_per_pubdata` is computed with floor (truncating) integer division via `wrapping_div`, while the analogous `native_per_gas` is computed with ceiling division via `div_ceil`. This inconsistency causes every ZK transaction that generates pubdata to be charged slightly fewer native resources per pubdata byte than the protocol intends, systematically undercharging users for pubdata proving costs.

### Finding Description

In `validate_and_compute_fee_for_transaction` in `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`, two related ratios are computed:

```rust
// native_per_gas: rounds UP (ceil)
u256_try_to_u64(&gas_price.div_ceil(native_price))
``` [1](#0-0) 

```rust
// native_per_pubdata: rounds DOWN (floor/truncate)
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
``` [2](#0-1) 

`native_per_pubdata` is then used in `get_resources_to_charge_for_pubdata` to compute the native cost charged for each pubdata byte consumed:

```rust
let native = current_pubdata_spent
    .checked_mul(native_per_pubdata)
    .ok_or(out_of_native_resources!())?;
``` [3](#0-2) 

This `native_per_pubdata` value flows into `create_resources_for_tx` as the per-byte native budget for pubdata, and is later used in `compute_gas_refund` to determine how much native resource is charged for pubdata after execution: [4](#0-3) 

The correct charge per pubdata byte should be `ceil(pubdata_price / native_price)` to ensure the protocol does not undercharge. Using `floor(pubdata_price / native_price)` means the user is charged at most `(native_price - 1)` fewer native units per pubdata byte than the protocol intends.

The same truncation bug exists in the Ethereum transaction flow at the same conceptual location: [2](#0-1) 

The design intent is documented in `double_resource_accounting.md`: [5](#0-4) 

The `nativePerGas` ratio is explicitly computed with `div_ceil` to be conservative for the protocol, but `nativePerPubdata` is not given the same treatment.

### Impact Explanation

Every ZK transaction that writes to storage (generating pubdata) is undercharged by up to `(native_price - 1)` native units per pubdata byte. The native resource models off-chain proving cost. Systematic undercharging means:

1. Users can publish slightly more pubdata per native unit than the protocol intends.
2. The protocol's native resource accounting for pubdata is consistently biased in the user's favor.
3. When `pubdata_price < native_price`, `native_per_pubdata` becomes 0, meaning pubdata costs zero native resources — users can generate pubdata with no native resource cost at all, limited only by the separate pubdata byte limit.

The third case (zero `native_per_pubdata`) is the most severe: any transaction where `pubdata_price < native_price` can generate pubdata without consuming any native resource budget, bypassing the native resource accounting for pubdata entirely.

### Likelihood Explanation

This is triggered by any unprivileged L2 transaction sender that writes to storage. The condition `pubdata_price < native_price` is reachable under normal operator-set block parameters. The truncation case (non-zero but rounded-down) affects every transaction with pubdata when `pubdata_price` is not an exact multiple of `native_price`.

### Recommendation

Replace `wrapping_div` with `div_ceil` for `native_per_pubdata`, consistent with how `native_per_gas` is computed:

```diff
- let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
+ let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
      .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
``` [2](#0-1) 

This mirrors the fix applied in the Mellow vault report (switching from `mulDiv` to `mulDivRoundingUp`) and is consistent with the existing `div_ceil` usage for `native_per_gas` on line 135.

### Proof of Concept

Consider block parameters: `pubdata_price = 999`, `native_price = 1000`.

- `native_per_gas = ceil(gas_price / 1000)` — correctly rounds up.
- `native_per_pubdata = floor(999 / 1000) = 0` — pubdata costs zero native resources.

A user submits a transaction that writes to 100 storage slots, generating ~3200 bytes of pubdata. With `native_per_pubdata = 0`, the call to `get_resources_to_charge_for_pubdata` returns a zero native charge:

```rust
let native = current_pubdata_spent.checked_mul(0) = 0;
``` [6](#0-5) 

The transaction's native resource budget is not consumed by pubdata at all. The `delta_gas` adjustment in `compute_gas_refund` also uses `native_used / native_per_gas`, but since `native_per_pubdata = 0` means no native was charged for pubdata, the gas adjustment does not compensate: [7](#0-6) 

The user effectively publishes pubdata for free in terms of native resources, undermining the protocol's proving-cost accounting model.

**File:** docs/double_resource_accounting.md (L37-50)
```markdown
First we define the ratio between EVM gas and native resource as:
  `nativePerGas := gasPrice/nativePrice`
Note: for call simulation we use a constant for it, as gasPrice might be set to 0.

Next we define the limit for the native resource as:
  `nativeLimit := gasLimit * nativePerGas`

Then we process the transaction, charging both Ergs for EE execution and native resource for any kind of computation (EE, bootloader or system work).

If execution doesn't run out of native resources, we first charge for pubdata from native resource.
Then we compute the difference between the implicit gas used derived from native resource consumption and the gas used by EEs from the ergs used. We call this value `deltaGas`.
  `deltaGas := (nativeUsed / nativePerGas) - gasUsed`

If `deltaGas > 0`, we add it to `gasUsed` and charge it from ergs. This ensures that gas estimation will include additional gas to cover for native resources using just base fee. We expect the base fee to be enough to cover most transactions without the need of additional gas.
