Audit Report

## Title
Unbounded Neuron Iteration in `compute_ballots_for_new_proposal` Exhausts Instruction Limit - (`rs/sns/governance/src/governance.rs`)

## Summary
The SNS Governance canister's `compute_ballots_for_new_proposal` performs a synchronous, unbounded linear scan over every neuron in the heap-resident `self.proto.neurons` BTreeMap with no instruction-limit guard. When the neuron population is large enough, any call to `make_proposal` will trap with `CanisterInstructionLimitExceeded`, permanently blocking all proposal submission and rendering the SNS canister ungovernable. Unlike the NNS Governance canister, which was retrofitted with `noop_self_call_if_over_instructions` and stable-memory neuron storage, no equivalent protection exists in the SNS path.

## Finding Description
`compute_ballots_for_new_proposal` at `rs/sns/governance/src/governance.rs:5255â€“5280` iterates unconditionally over `self.proto.neurons.iter()`:

```rust
for (k, v) in self.proto.neurons.iter() {
    if v.dissolve_delay_seconds(now_seconds) < min_dissolve_delay_for_vote {
        continue;
    }
    let voting_power = v.voting_power(...);
    total_power += voting_power as u128;
    electoral_roll.insert(k.clone(), Ballot { ... });
}
``` [1](#0-0) 

This function is called synchronously and unconditionally inside `make_proposal` before any state mutation:

```rust
let (_, electoral_roll) = self
    .compute_ballots_for_new_proposal()
    .map_err(...)?;
``` [2](#0-1) 

The only neuron-population guard in the codebase is `check_neuron_population_can_grow`, which only prevents adding neurons beyond `max_number_of_neurons`; it does not limit the iteration cost of `compute_ballots_for_new_proposal`: [3](#0-2) 

A search for `noop_self_call_if_over_instructions` in `rs/sns/` returns zero matches. The NNS Governance canister uses this guard in `rs/nns/governance/src/voting.rs` and also migrated neurons to stable memory to reduce per-neuron iteration cost. The NNS benchmark (`compute_ballots_for_new_proposal_with_stable_neurons`) records 2,450,000 instructions for 100 neurons using stable-memory neurons: [4](#0-3) 

The NNS bench explicitly projects this against `MAX_NUMBER_OF_NEURONS` with a 25B budget to verify safety: [5](#0-4) 

No equivalent benchmark or projection exists for the SNS governance path. The SNS neurons are heap-resident (not stable memory), so per-neuron iteration cost is at least as high as the NNS stable-memory baseline. Scaling linearly from the NNS benchmark (2.45M instructions / 100 neurons = ~24,500 instructions/neuron), the 5B per-message instruction limit is exhausted at approximately 200,000 neurons. The `max_number_of_neurons` parameter is configurable by governance up to `MAX_NUMBER_OF_NEURONS_CEILING` defined in `rs/sns/governance/src/types.rs` and `rs/nervous_system/common/src/lib.rs`. [6](#0-5) 

## Impact Explanation
Once the neuron population reaches the threshold where `compute_ballots_for_new_proposal` exhausts the 5B instruction budget, every `make_proposal` call traps. Since SNS upgrades, parameter changes, and all governance actions require proposals, the canister becomes permanently unupgradeable and ungovernable. This is a concrete application/platform-level DoS against SNS governance â€” a High-severity impact under the allowed ICP bounty scope ("Application/platform-level DoS... or subnet availability impact").

## Likelihood Explanation
Any principal holding SNS tokens can stake neurons via the public `manage_neuron` update call. No privileged role is required. An attacker (or organic growth) can fill the neuron population to `max_number_of_neurons` by distributing tokens across many principals and staking. Once the threshold is crossed, the trap is triggered by any legitimate user's `make_proposal` call â€” the attacker does not need to submit the proposal themselves. The attack is repeatable and permanent: there is no recovery path once all `make_proposal` calls trap.

## Recommendation
1. Add an instruction-limit guard to `compute_ballots_for_new_proposal` analogous to the NNS Governance's `noop_self_call_if_over_instructions` pattern in `rs/nns/governance/src/voting.rs`, or move ballot computation to a timer-based async task.
2. Add a canbench benchmark for `compute_ballots_for_new_proposal` in the SNS governance that projects against the actual `MAX_NUMBER_OF_NEURONS_CEILING` (as the NNS governance does with `check_projected_instructions`) and enforces the projected instruction count stays well below 5B.
3. Consider migrating SNS neuron storage to stable memory (as NNS did) to reduce per-neuron iteration cost.

## Proof of Concept
1. Deploy an SNS with `max_number_of_neurons` set to a large value (e.g., the ceiling defined in `rs/nervous_system/common/src/lib.rs`).
2. Distribute SNS tokens across many principals and stake neurons until `self.proto.neurons.len()` approaches `max_number_of_neurons`.
3. Call `manage_neuron` with `Command::MakeProposal(...)` from any neuron holder.
4. `make_proposal` calls `compute_ballots_for_new_proposal` at line 3557, which iterates all neurons synchronously at line 5255.
5. The message traps with `CanisterInstructionLimitExceeded`; no proposal can ever be submitted again.
6. A deterministic integration test using PocketIC can reproduce this by populating the neuron store to the threshold count and asserting that `make_proposal` returns a trap/error rather than succeeding.
