Audit Report

## Title
Upgrade Lock Set After Async Calls in `perform_upgrade_to_next_sns_version_legacy` Allows Concurrent Proposals to Bypass Guard - (File: `rs/sns/governance/src/governance.rs`)

## Summary

In `perform_upgrade_to_next_sns_version_legacy`, `self.proto.pending_version` is set only after all async inter-canister calls complete, leaving the upgrade-in-progress guard ineffective during the entire async execution window. A second concurrently spawned `UpgradeSnsToNextVersion` proposal can pass `check_no_upgrades_in_progress` while the first is awaiting, resulting in two simultaneous upgrade executions that corrupt SNS version tracking state and leave proposals permanently unresolved.

## Finding Description

`perform_upgrade_to_next_sns_version_legacy` calls `check_no_upgrades_in_progress` at line 2828, which gates on `self.proto.pending_version.is_none()`. However, `pending_version` is only assigned at line 2894 — after `get_or_reset_deployed_version().await`, `get_upgrade_params(...).await`, `get_wasm(...).await`, and `upgrade_non_root_canister(...).await` all complete. [1](#0-0) [2](#0-1) 

On the IC, a canister processes one message at a time, but `spawn_in_canister_env` schedules proposal futures as independent background tasks. At every `await` point, the runtime can resume another spawned future. A second `UpgradeSnsToNextVersion` proposal spawned via `start_proposal_execution` will call `check_no_upgrades_in_progress`, find `pending_version == None` (because the first future has not yet reached line 2894), and proceed with its own upgrade sequence. [3](#0-2) 

By contrast, `initiate_upgrade_if_sns_behind_target_version` correctly sets `pending_version` **before** the async call, making the guard effective: [4](#0-3) 

The inconsistency between the two code paths is the root cause. The guard is structurally sound; it is only bypassed in the legacy path due to the late lock assignment.

## Impact Explanation

Two concurrent executions both call `upgrade_non_root_canister` (or `upgrade_canister_directly`) on the same SNS canister. The second execution's `pending_version` write overwrites the first's at line 2894, so `check_upgrade_status` tracks only the second proposal's target. The first proposal's `proposal_id` is lost from `pending_version`, meaning `complete_sns_upgrade_to_next_version` never calls `set_proposal_execution_status` for it — leaving a permanently adopted-but-unresolved proposal. `deployed_version` is then updated to the second proposal's target, skipping intermediate versions and corrupting the SNS upgrade journal. Subsequent `UpgradeSnsToNextVersion` proposals will be evaluated against an incorrect `deployed_version`, potentially blocking all future SNS upgrades. This matches the **High** impact class: significant SNS governance security impact with concrete protocol harm. [5](#0-4) 

## Likelihood Explanation

Any SNS neuron holder can submit `UpgradeSnsToNextVersion` proposals. Two such proposals reaching voting threshold within the same voting period is a realistic scenario when the SNS-W upgrade path has multiple pending steps. No governance majority attack is required; normal concurrent proposal adoption is sufficient. The async window spans multiple IC rounds (stop → install_code → start through root), providing a realistic interleaving opportunity. No special privileges, leaked keys, or external compromise are needed.

## Recommendation

Move the `self.proto.pending_version = Some(...)` assignment to immediately after `check_no_upgrades_in_progress` and before the first `await` in `perform_upgrade_to_next_sns_version_legacy`, mirroring the pattern at lines 5636–5641. On any error in the subsequent async calls, clear `pending_version` in the error path (as `initiate_upgrade_if_sns_behind_target_version` does at line 5650). [6](#0-5) 

## Proof of Concept

1. SNS has `deployed_version = V1`, `pending_version = None`.
2. Two `UpgradeSnsToNextVersion` proposals P1 and P2 are adopted and both call `start_proposal_execution`, spawning two background futures via `spawn_in_canister_env`.
3. P1's future begins, passes `check_no_upgrades_in_progress` (pending_version is None), reaches `get_upgrade_params(...).await` and yields.
4. P2's future begins, passes `check_no_upgrades_in_progress` (pending_version still None), proceeds through all async calls.
5. P2 completes, sets `pending_version = {target: V2, proposal_id: P2}`.
6. P1 resumes, also completes all async calls, overwrites `pending_version = {target: V2, proposal_id: P1}`.
7. Both futures have called `upgrade_non_root_canister` — two `install_code` calls were issued on the same canister.
8. `check_upgrade_status` runs, resolves against `pending_version.proposal_id = P1`, marks P1 executed, sets `deployed_version = V2`, clears `pending_version`.
9. P2's execution status is never set; P2 remains permanently adopted-but-unresolved.
10. A deterministic integration test using PocketIC can reproduce this by adopting two proposals in the same round and asserting on `pending_version` state and proposal execution status after the upgrade completes. [7](#0-6)

    async fn perform_upgrade_to_next_sns_version_legacy(
        &mut self,
        proposal_id: u64,
    ) -> Result<bool, GovernanceError> {
        self.check_no_upgrades_in_progress(Some(proposal_id))?;

        let current_version = self.get_or_reset_deployed_version().await.map_err(|err| {
            GovernanceError::new_with_message(
                ErrorType::External,
                format!("Could not execute proposal: {err}"),
            )
        })?;

        let root_canister_id = self.proto.root_canister_id_or_panic();

        let UpgradeSnsParams {
            next_version,
            canister_type_to_upgrade,
            new_wasm_hash,
            canister_ids_to_upgrade,
        } = get_upgrade_params(&*self.env, root_canister_id, &current_version)
            .await
            .map_err(|err| {
                GovernanceError::new_with_message(
                    ErrorType::InvalidProposal,
                    format!("Could not execute proposal: {err}"),
                )
            })?;

        self.push_to_upgrade_journal(upgrade_journal_entry::UpgradeStarted::from_proposal(
            current_version.clone(),
            next_version.clone(),
            ProposalId { id: proposal_id },
        ));

        let target_wasm = get_wasm(&*self.env, new_wasm_hash.to_vec(), canister_type_to_upgrade)
            .await
            .map_err(|e| {
                GovernanceError::new_with_message(
                    ErrorType::External,
                    format!("Could not execute proposal: {e}"),
                )
            })?
            .wasm;

        let target_is_root = canister_ids_to_upgrade.contains(&root_canister_id);

        if target_is_root {
            upgrade_canister_directly(
                &*self.env,
                root_canister_id,
                target_wasm,
                Encode!().unwrap(),
            )
            .await?;
        } else {
            for target_canister_id in canister_ids_to_upgrade {
                self.upgrade_non_root_canister(
                    target_canister_id,
                    Wasm::Bytes(target_wasm.clone()),
                    Encode!().unwrap(),
                    CanisterInstallMode::Upgrade,
                )
                .await?;
            }
        }

        // A canister upgrade has been successfully kicked-off. Set the pending upgrade-in-progress
        // field so that Governance's run_periodic_tasks logic can check on the status of
        // this upgrade.
        self.proto.pending_version = Some(PendingVersion {
            target_version: Some(next_version),
            mark_failed_at_seconds: self.env.now() + 5 * 60,
            checking_upgrade_lock: 0,
            proposal_id: Some(proposal_id),
        });

        log!(
            INFO,
            "Successfully kicked off upgrade for SNS canister {:?}",
            canister_type_to_upgrade,
        );

        Ok(false)
    }
