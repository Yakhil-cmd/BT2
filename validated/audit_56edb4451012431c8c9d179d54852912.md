### Title
`MsgTurnBridge` Handler Is a No-Op, Bypassing Permission Check and Permanently Disabling the Bridge Circuit Breaker - (File: `x/cronos/keeper/msg_server.go`)

### Summary
The `TurnBridge` message handler returns `nil, nil` unconditionally, performing no permission check and no state change. Any unprivileged user can submit `MsgTurnBridge` and receive a success response, bypassing the `CanTurnBridge` authorization guard. More critically, because the function is a stub, no one — including the admin — can ever disable the bridge, permanently breaking the emergency circuit breaker.

### Finding Description
`UpdateTokenMapping` correctly enforces its permission gate before mutating state:

```go
// x/cronos/keeper/msg_server.go:69-82
func (k msgServer) UpdateTokenMapping(...) {
    if !k.HasPermission(ctx, msg.GetSigners(), CanChangeTokenMapping) {
        return nil, errors.Wrap(sdkerrors.ErrUnauthorized, "msg sender is not authorized")
    }
    if err := k.RegisterOrUpdateTokenMapping(ctx, msg); err != nil { ... }
    return &types.MsgUpdateTokenMappingResponse{}, nil
}
```

`TurnBridge`, which is supposed to follow the same pattern using the parallel `CanTurnBridge` constant, does neither:

```go
// x/cronos/keeper/msg_server.go:84-87
func (k msgServer) TurnBridge(goCtx context.Context, msg *types.MsgTurnBridge) (*types.MsgTurnBridgeResponse, error) {
    return nil, nil
}
```

The `CanTurnBridge` permission bit is defined and tested in `permissions.go` / `permissions_test.go`, confirming the intent was to gate this message behind a permission check identical to `CanChangeTokenMapping`. The proto definition at `proto/cronos/tx.proto:82-89` exposes `MsgTurnBridge` as a live, signed transaction surface. The ADR at `docs/architecture/adr-009.md` explicitly lists `MsgTurnBridge` as one of the two messages that must be restricted to permissioned addresses.

The root cause is structurally identical to the external report: an authorization gate exists (`HasPermission` / `CanTurnBridge`) but is never invoked at the call site, so the check's result — whether the caller is authorized — is never enforced.

### Impact Explanation
Two distinct impacts arise:

1. **Auth bypass (High):** Any unprivileged address can submit `MsgTurnBridge{enable: false}` or `{enable: true}` and receive a `200 OK` (nil error). The `CanTurnBridge` permission check is completely skipped, violating the permission model described in ADR-009 and the `permissions.go` constants.

2. **Permanent bridge circuit-breaker failure (High):** Because the handler performs no state write, the bridge enabled/disabled flag is never set. Authorized operators (admin, permissioned addresses) cannot disable the bridge in an emergency. Any ongoing bridge exploit — e.g., a malicious IBC packet draining CRC20/CRC21 balances via the `IBCConversionModule` middleware or the EVM hook handlers — cannot be halted through the intended `MsgTurnBridge` mechanism. This is a permanent, protocol-level inability to stop bridge/conversion flows.

### Likelihood Explanation
The entry path is fully unprivileged: `MsgTurnBridge` is a standard Cosmos SDK message reachable by any account with gas. No special keys, governance votes, or validator access are required. The transaction will be accepted by the mempool (`BlockAddressesDecorator` does not block it) and executed on-chain, returning success.

### Recommendation
Implement `TurnBridge` to mirror `UpdateTokenMapping`:

```go
func (k msgServer) TurnBridge(goCtx context.Context, msg *types.MsgTurnBridge) (*types.MsgTurnBridgeResponse, error) {
    ctx := sdk.UnwrapSDKContext(goCtx)
    if !k.HasPermission(ctx, msg.GetSigners(), CanTurnBridge) {
        return nil, errors.Wrap(sdkerrors.ErrUnauthorized, "msg sender is not authorized")
    }
    // persist bridge enabled/disabled state
    params := k.GetParams(ctx)
    params.EnableAutoDeployment = msg.Enable // or a dedicated BridgeEnabled field
    if err := k.SetParams(ctx, params); err != nil {
        return nil, err
    }
    return &types.MsgTurnBridgeResponse{}, nil
}
```

Additionally, all bridge-path handlers (EVM hooks, IBC middleware `OnRecvPacket`, `send_to_ibc` handlers) should read and enforce the bridge-enabled flag before processing.

### Proof of Concept
1. Attacker (any unprivileged address) broadcasts `MsgTurnBridge{sender: attacker, enable: false}`.
2. `TurnBridge` executes, skips all permission checks, and returns `nil, nil` — the transaction is committed on-chain as successful.
3. No bridge state is written; the bridge continues operating as if the message was never sent.
4. Simultaneously, a legitimate admin who discovers an active bridge exploit submits `MsgTurnBridge{enable: false}` — it also returns success but changes nothing, leaving the exploit path open indefinitely. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
