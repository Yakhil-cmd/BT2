### Title
Missing `msg_hash != 0` Validation in OT-Based ECDSA `sign()` Enables Signing of Attacker-Chosen Zero Hash - (File: `src/ecdsa/ot_based_ecdsa/sign.rs`)

---

### Summary

The OT-based ECDSA `sign()` function accepts `msg_hash = 0` without any validation, while the robust ECDSA `sign()` function explicitly rejects it. A malicious coordinator can pass `msg_hash = Scalar::ZERO` to all participants, causing them to produce signature shares for the zero message hash. The coordinator aggregates these shares into a valid, verifiable threshold ECDSA signature for an attacker-chosen input.

---

### Finding Description

The `sign()` function in `src/ecdsa/ot_based_ecdsa/sign.rs` accepts any `msg_hash: Scalar` value, including zero, with no guard:

```rust
pub fn sign(
    participants: &[Participant],
    coordinator: Participant,
    threshold: impl Into<ReconstructionLowerBound>,
    me: Participant,
    public_key: AffinePoint,
    presignature: RerandomizedPresignOutput,
    msg_hash: Scalar,          // ← no zero-check
) -> Result<impl Protocol<Output = SignatureOption>, InitializationError> {
``` [1](#0-0) 

In contrast, the robust ECDSA `sign()` function explicitly rejects `msg_hash == 0`:

```rust
if bool::from(msg_hash.is_zero()) {
    return Err(InitializationError::BadParameters(
        "msg_hash cannot be 0 to avoid potential split view attacks".to_string(),
    ));
}
``` [2](#0-1) 

The security documentation explicitly mandates this guard:

> **4. Do not sign with $h = 0$** (the zero message hash). This input enables a related algebraic split-view attack in the modified scheme when $N_1 > 2t + 1$. [3](#0-2) 

When `msg_hash = 0`, the OT-based ECDSA signing formula `s_i = h * k_i + R_x * sigma_i` reduces to `s_i = R_x * sigma_i`. The coordinator aggregates to `s = R_x * sigma`. The internal verification check `sig.verify(&public_key, &msg_hash)` passes because with `h=0`, the ECDSA equation `x_coord(s^{-1} * h * G + s^{-1} * R_x * X) = R_x` reduces to `x_coord(s^{-1} * R_x * X) = x_coord(k^{-1} * G) = R_x`, which holds when `s = R_x * k * x = R_x * sigma`. [4](#0-3) 

The coordinator's final verification step:

```rust
if !sig.verify(&public_key, &msg_hash) {
    return Err(ProtocolError::AssertionFailed(
        "signature failed to verify".to_string(),
    ));
}
``` [5](#0-4) 

passes for `h=0`, so the protocol completes successfully and returns a valid signature.

Additionally, the OT-based ECDSA scheme does **not** enforce `participants.len() == 2*t+1` (unlike robust ECDSA), meaning `N_1 > 2t+1` is permitted:

```rust
// ensure number of participants during the signing phase is >= threshold
if participants.len() < threshold {
    return Err(...)
}
// No exact-count check, unlike robust ECDSA
``` [6](#0-5) 

This means the condition `N_1 > 2t+1` — under which the `h=0` split-view attack is documented to be exploitable — is reachable in the OT-based scheme.

---

### Impact Explanation

**Critical — Unauthorized creation of a valid threshold signature for attacker-chosen inputs.**

A malicious coordinator passes `msg_hash = Scalar::ZERO` to all participants. Each honest participant computes `s_i = R_x * sigma_i` and sends it to the coordinator. The coordinator sums the shares and obtains a valid, verifiable ECDSA signature for `h=0`. This is a valid threshold signature produced for an attacker-chosen input (the zero hash), without the honest participants having any means to detect or reject it.

Furthermore, if the coordinator can run two parallel signing sessions against the same presignature (before it is marked consumed), one with `h=0` and one with `h≠0`, standard ECDSA nonce-reuse formulas yield the secret key trivially: `k = (s2 - s1) / h2` and `x = s1 / (k * R_x)`.

---

### Likelihood Explanation

The coordinator is a designated participant in the OT-based ECDSA signing protocol. A malicious coordinator controls the `msg_hash` value passed to each participant's `sign()` invocation. Since there is no validation of `msg_hash != 0` in the OT-based ECDSA `sign()` function, a malicious coordinator can trivially exploit this by passing `Scalar::ZERO`. No special privileges beyond the coordinator role are required.

---

### Recommendation

Add the same zero-hash guard to `src/ecdsa/ot_based_ecdsa/sign.rs` that already exists in `src/ecdsa/robust_ecdsa/sign.rs`:

```rust
if bool::from(msg_hash.is_zero()) {
    return Err(InitializationError::BadParameters(
        "msg_hash cannot be 0".to_string(),
    ));
}
```

This should be placed immediately after the participant-count checks, before the protocol is started.

---

### Proof of Concept

1. Run OT-based ECDSA keygen and presigning to obtain `(R, k_i, sigma_i)` for each participant.
2. Construct `RerandomizationArguments` with `msg_hash = [0u8; 32]` and call `RerandomizedPresignOutput::rerandomize_presign()` — this succeeds because HKDF with a zero-byte input still produces a valid non-zero `delta`.
3. Call `ot_based_ecdsa::sign::sign(..., msg_hash = Scalar::ZERO)` for each participant — no error is returned.
4. Each participant computes `s_i = 0 * k_i + R_x * sigma_i = R_x * sigma_i` and sends it to the coordinator.
5. The coordinator sums the shares: `s = R_x * sigma`.
6. The coordinator calls `sig.verify(&public_key, &Scalar::ZERO)` — this returns `true`.
7. The protocol returns `Some(Signature { big_r: R, s })` — a valid threshold ECDSA signature for the zero message hash, produced for an attacker-chosen input.

    let participants =

    // ensure my presence in the participant list

        return Err(InitializationError::MissingParticipant {
            role: "self",

        });
