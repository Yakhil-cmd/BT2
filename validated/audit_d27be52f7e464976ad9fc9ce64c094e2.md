### Title
Unchecked `old_threshold` Parameter Allows Bypassing Reshare Intersection Security Check — (`src/dkg.rs`)

---

### Summary

The `assert_reshare_keys_invariants` function in `src/dkg.rs` validates the **new** threshold against a minimum bound of 2, but applies **no minimum validation** to the caller-supplied `old_threshold` parameter. Passing `old_threshold = 0` causes the intersection check `intersection.len() < old_threshold` to evaluate as `usize < 0`, which is always `false` in Rust, unconditionally bypassing the security invariant that at least `old_threshold` old participants must be present in the new participant set before a reshare can proceed.

---

### Finding Description

In `assert_reshare_keys_invariants` the new threshold is validated through `assert_key_invariants`, which enforces `threshold >= 2`: [1](#0-0) 

However, `old_threshold` receives no analogous minimum check. It is only used in the intersection guard: [2](#0-1) 

`old_threshold` is extracted as a plain `usize`: [3](#0-2) 

`ReconstructionLowerBound` is a transparent `usize` newtype with no invariant enforcement: [4](#0-3) 

When a caller passes `old_threshold = 0`, the comparison becomes `intersection.len() < 0usize`, which is **always false** in Rust's unsigned arithmetic. The guard never fires, and the reshare is allowed to proceed regardless of how many old participants are actually present in the new set.

The public entry point `reshare()` in `src/lib.rs` forwards the caller-supplied `old_threshold` directly to `assert_reshare_keys_invariants` without any prior sanitisation: [5](#0-4) 

Once the guard is bypassed, `do_reshare` computes the intersection and derives each old participant's linearised secret contribution via Lagrange interpolation over that intersection: [6](#0-5) 

If the intersection is smaller than the actual old threshold, the Lagrange basis is wrong and the reconstructed secret is incorrect. The resulting aggregate public key will not match `old_public_key`, which is caught only later by the post-hoc check: [7](#0-6) 

By this point all polynomial commitments have already been broadcast and private share evaluations have been sent, so the protocol terminates with an error and all participants receive unusable outputs.

---

### Impact Explanation

**High — Corruption of reshare outputs so honest parties receive unusable cryptographic outputs.**

A caller who passes `old_threshold = 0` (or any value below the actual keygen threshold) bypasses the only pre-flight guard that enforces the minimum overlap between old and new participant sets. Two concrete failure modes result:

1. **Empty intersection:** `intersection.lagrange::<C>(me)` returns `ProtocolError::InvalidIndex` for every old participant, aborting the protocol immediately after commitments have been broadcast.
2. **Under-sized intersection:** Lagrange interpolation over fewer points than the actual threshold reconstructs a wrong secret. The new aggregate public key diverges from `old_public_key`; the protocol aborts at the post-hoc equality check after private shares have been distributed.

In both cases every honest participant receives a protocol error instead of valid new key shares, rendering the reshare unusable.

---

### Likelihood Explanation

**Medium.** The `old_threshold` parameter is entirely caller-controlled with no cryptographic binding to the threshold used during the original keygen. Any library consumer — including a malicious participant acting as the reshare initiator — can supply `old_threshold = 0` or `old_threshold = 1` to trigger the bypass. No special privilege beyond calling the public `reshare()` API is required.

---

### Recommendation

Add an explicit minimum validation for `old_threshold` inside `assert_reshare_keys_invariants`, mirroring the check already applied to the new threshold:

```rust
// In assert_reshare_keys_invariants, after extracting old_threshold as usize:
if old_threshold < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: old_threshold,
        min: 2,
    });
}
```

This mirrors the existing guard for the new threshold in `assert_key_invariants`: [8](#0-7) 

---

### Proof of Concept

```
1. Run keygen with participants = [P1, P2, P3], threshold = 3.
   → Each party holds a share x_i; public key = X.

2. Malicious caller invokes reshare() with:
     old_participants = [P1, P2, P3]
     old_threshold    = 0          ← attacker-controlled
     new_participants = [P4, P5, P6]  (no overlap with old set)
     new_threshold    = 2

3. assert_reshare_keys_invariants evaluates:
     intersection.len() (= 0) < old_threshold (= 0)
     → 0 < 0  →  false  →  guard does NOT fire.

4. do_reshare is entered. For each old participant Pi:
     intersection.lagrange::<C>(Pi) → ProtocolError::InvalidIndex
     (Pi is not in the empty intersection)
   Protocol aborts; all parties receive an error.

5. Alternatively, with old_threshold = 1 and one old participant P1 in new set:
     Lagrange coefficient for {P1} = 1; secret = x_1 (not the actual secret).
     New public key = x_1·G ≠ X.
     do_keyshare aborts at the old_vk != verifying_key check.
     All parties receive an error after commitments and shares have been sent.
```
