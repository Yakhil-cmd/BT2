### Title
`setPoolAdminFeeDestination()` Redirects Accrued Admin Fees to New Destination Without Prior Collection — (`File: metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary

`MetricOmmPoolFactory.setPoolAdminFeeDestination()` updates `poolAdminFeeDestination[pool]` without first calling `collectFees()`. Every other fee-configuration mutator (`setPoolAdminFees`, `setPoolProtocolFee`) flushes accrued fees at the old rates/destination before applying the change. The destination setter skips this step, so all fees that accrued under the old destination are silently redirected to the new one on the next `collectPoolFees()` call.

### Finding Description

The fee-collection model in Metric OMM separates two concerns:

1. **Fee rates** — stored in `poolFeeConfig[pool]` on the factory; used to split the pool's surplus and notional accumulators between protocol and admin.
2. **Admin fee destination** — stored in `poolAdminFeeDestination[pool]`; the address that receives the admin share on every `collectFees()` call.

Both `setPoolAdminFees()` and `setPoolProtocolFee()` follow the correct pattern: collect first at the old configuration, then update. [1](#0-0) 

`setPoolAdminFeeDestination()` does not: [2](#0-1) 

At the moment the destination is changed, the pool already holds two categories of accrued admin fees:

- **Spread surplus** — `balance0/1() * SCALE - binTotals.scaledToken0/1 - notionalFeeToken0/1Scaled`. This surplus grows with every swap and is split between protocol and admin at collection time using the stored rates.
- **Notional accumulator** — `notionalFeeToken0Scaled` / `notionalFeeToken1Scaled`, incremented on every swap. [3](#0-2) 

When `collectPoolFees()` (or any subsequent `collectFees()` call) is executed after the destination change, the pool passes the **new** `poolAdminFeeDestination` to `collectFees()`, which transfers the admin share — including all fees that accrued before the change — to the new address. [4](#0-3) 

The old destination receives nothing for the period it was the designated recipient.

### Impact Explanation

The old admin fee destination loses all admin fees (both spread and notional components) that accrued between the last `collectPoolFees()` call and the `setPoolAdminFeeDestination()` call. These tokens are not destroyed — they are transferred to the new destination — but the old destination is permanently deprived of funds it was owed. This is a direct loss of protocol/admin fee assets.

### Likelihood Explanation

The pool admin is semi-trusted and has legitimate authority to change the fee destination. The trigger is a normal admin operation (`setPoolAdminFeeDestination`) that any pool admin can call at any time. No special conditions are required beyond there being uncollected fees in the pool, which is the normal operating state between collection intervals. The pool admin and the fee destination are separate addresses (`poolAdmin[pool]` vs. `poolAdminFeeDestination[pool]`), so the admin can redirect fees away from a treasury or beneficiary that is not the admin itself.

### Recommendation

Collect fees at the current destination before updating it, mirroring the pattern used by `setPoolAdminFees()` and `setPoolProtocolFee()`:

```solidity
function setPoolAdminFeeDestination(address pool, address newAdminFeeDestination)
    external
    override
    nonReentrant
    onlyPoolAdmin(pool)
{
    if (newAdminFeeDestination == address(0)) revert InvalidAdminFeeDestination();
+   PoolFeeConfig memory c = poolFeeConfig[pool];
+   IMetricOmmPoolCollectFees(pool).collectFees(
+       c.protocolSpreadFeeE6,
+       c.adminSpreadFeeE6,
+       c.protocolNotionalFeeE8,
+       c.adminNotionalFeeE8,
+       poolAdminFeeDestination[pool]   // old destination
+   );
    poolAdminFeeDestination[pool] = newAdminFeeDestination;
    emit PoolAdminFeeDestinationUpdated(pool, newAdminFeeDestination);
}
```

### Proof of Concept

1. Pool is deployed with `adminFeeDestination = oldTreasury`.
2. Many swaps occur; `notionalFeeToken0Scaled` and the spread surplus both grow.
3. Pool admin calls `setPoolAdminFeeDestination(pool, newTreasury)` — no collection happens.
4. Anyone calls `collectPoolFees(pool)`. Inside `collectFees()`, `adminFeeDestination_` is now `newTreasury`.
5. All admin fees (spread + notional, including those accrued before step 3) are transferred to `newTreasury`.
6. `oldTreasury` receives zero tokens for the entire accrual period it was the designated recipient. [5](#0-4)
