### Title
Missing Validation That `RerandomizationArguments.participants` Matches Signing Participants Enables Split-View Secret Key Extraction — (`src/ecdsa/mod.rs`, `src/ecdsa/robust_ecdsa/sign.rs`, `src/ecdsa/ot_based_ecdsa/sign.rs`)

---

### Summary

The `RerandomizationArguments` struct includes a `participants` field that is hashed into the HKDF-derived nonce scalar `delta`. However, neither `RerandomizedPresignOutput::rerandomize_presign()` nor `sign()` validates that the `participants` used during rerandomization matches the `participants` passed to the signing phase. A malicious coordinator can exploit this inconsistency to run two signing sessions over the same presignature with different effective nonces, enabling multiplicative nonce-reuse and full secret key extraction.

---

### Finding Description

**Root cause — missing cross-phase participant binding**

`RerandomizationArguments` stores a `participants: ParticipantList` field that is serialized into the HKDF input inside `derive_randomness()`:

```rust
// src/ecdsa/mod.rs lines 157-158
for participant in self.participants.participants() {
    concatenation.extend_from_slice(&participant.bytes());
}
```

This means `delta = HKDF(entropy, pk, tweak, h, R, participants)` is participant-set-dependent. The rerandomized shares `k_i * delta^{-1}`, `sigma_i * delta^{-1}` (OT-based) or `alpha_i * delta^{-1}`, `beta_i * delta^{-1}` (robust) are therefore bound to whichever participant set was supplied to `RerandomizationArguments`.

However, `RerandomizedPresignOutput` stores **no record** of which participants were used:

```rust
// src/ecdsa/robust_ecdsa/mod.rs lines 43-52
pub struct RerandomizedPresignOutput {
    big_r: AffinePoint,
    e: Scalar,
    alpha: Scalar,
    beta: Scalar,
}
```

And `sign()` accepts a fresh `participants` slice with **no check** that it matches the participants embedded in the rerandomization:

```rust
// src/ecdsa/robust_ecdsa/sign.rs lines 33-107
pub fn sign(
    participants: &[Participant],   // ← never compared to rerandomization participants
    coordinator: Participant,
    max_malicious: impl Into<MaxMalicious>,
    me: Participant,
    public_key: AffinePoint,
    presignature: RerandomizedPresignOutput,  // ← carries no participant record
    msg_hash: Scalar,
) -> Result<impl Protocol<Output = SignatureOption>, InitializationError> {
    // ... only checks: len >= 2, no duplicates, me present, coordinator present,
    //                  len == 2*max_malicious+1, msg_hash != 0
    // NO check: presignature.rerandomization_participants == participants
```

The same gap exists in the OT-based variant at `src/ecdsa/ot_based_ecdsa/sign.rs` lines 22–75.

**Exploit flow (malicious coordinator)**

The coordinator orchestrates both rerandomization and signing. It can:

1. Run presigning with set `P = {A, B, C}`, obtaining `PresignOutput` with shared nonce `R`.
2. **Session 1**: construct `RerandomizationArguments` with `participants = {A, B, C}`, distribute to all signers → each computes `delta_1 = HKDF(..., {A,B,C})` → valid signature `(R·delta_1, s_1)`.
3. **Session 2**: construct `RerandomizationArguments` with `participants = {A, B, D}` (different set, `D` need not even exist), distribute to all signers → each computes `delta_2 = HKDF(..., {A,B,D})` → valid signature `(R·delta_2, s_2)`.

Both sessions use the **same underlying presignature** `R`. The effective nonces are `k_1 = k·delta_1^{-1}` and `k_2 = k·delta_2^{-1}`, giving the multiplicative relation `k_1 = k_2 · (delta_2/delta_1)`.

The coordinator knows `delta_1` and `delta_2` (both derived from public inputs it chose). From the two ECDSA equations:

```
s_1 = (h + x·r_1) · k_1^{-1}
s_2 = (h + x·r_2) · k_2^{-1}
```

with `k_1^{-1} = k_2^{-1} · (delta_1/delta_2)` known, the coordinator can solve for the secret key `x`. This is exactly the multiplicative nonce-reuse attack documented in `docs/ecdsa/robust_ecdsa/signing.md` lines 151–158.

---

### Impact Explanation

**Critical — secret key extraction.**

A malicious coordinator can recover the aggregate private signing key by running two signing sessions over the same presignature with attacker-chosen participant sets in `RerandomizationArguments`. The rerandomization mechanism, which is the primary defense against split-view attacks, is fully bypassed because the library never validates that the participant set used to derive `delta` matches the participant set used for Lagrange interpolation in signing. All private key shares are effectively compromised.

---

### Likelihood Explanation

The coordinator role is explicitly within the threat model (the `sign()` docstring warns about coordinator-driven split-view attacks). The coordinator controls which `RerandomizationArguments` each signer receives. No privileged access beyond the coordinator role is required. The attack requires only two signing sessions over the same presignature — a realistic scenario given that presignature reuse is also not enforced by the library. The missing check is a single absent validation that any coordinator can trivially bypass.

---

### Recommendation

1. **Store the participant commitment in `RerandomizedPresignOutput`**: record a hash (or the sorted serialization) of the participants used during rerandomization.

   ```rust
   pub struct RerandomizedPresignOutput {
       big_r: AffinePoint,
       e: Scalar,
       alpha: Scalar,
       beta: Scalar,
       participants_hash: [u8; 32],  // hash of sorted participant IDs used in rerandomization
   }
   ```

2. **Validate in `sign()`** that the signing participants produce the same hash:

   ```rust
   let signing_participants_hash = hash_participants(&participants);
   if presignature.participants_hash != signing_participants_hash {
       return Err(InitializationError::BadParameters(
           "signing participants must match those used in rerandomization".to_string()
       ));
   }
   ```

This mirrors the existing `big_r` consistency check in `rerandomize_presign()` and closes the analogous gap for the participant set.

### Proof of Concept

```
Setup: N=3 participants {A, B, C}, threshold t=1 (max_malicious=1, so 2t+1=3).
       Run presigning → PresignOutput with big_r = R.

Session 1:
  args1 = RerandomizationArguments::new(pk, tweak, h, R, {A,B,C}, entropy)
  // delta_1 = HKDF(entropy, pk, tweak, h, R, [A,B,C])
  Each of A,B,C calls rerandomize_presign(presig, &args1) → RerandomizedPresignOutput_1
  sign([A,B,C], coordinator, max_malicious=1, ..., presig_1, h) → Signature (R·delta_1, s_1)  ✓

Session 2 (same presig, different participants in args):
  args2 = RerandomizationArguments::new(pk, tweak, h, R, {A,B,D}, entropy)
  // delta_2 = HKDF(entropy, pk, tweak, h, R, [A,B,D])  ← different from delta_1
  Each of A,B,C calls rerandomize_presign(presig, &args2) → RerandomizedPresignOutput_2
  sign([A,B,C], coordinator, max_malicious=1, ..., presig_2, h) → Signature (R·delta_2, s_2)  ✓
  // sign() accepts this — no check that args2.participants == [A,B,C]

Coordinator computes delta_1, delta_2 (both from public inputs it chose).
Solves for secret key x from (R·delta_1, s_1) and (R·delta_2, s_2) using
multiplicative nonce relation k1 = k2 · (delta_2/delta_1).
```

pub fn sign(
    participants: &[Participant],
    coordinator: Participant,
    max_malicious: impl Into<MaxMalicious>,
    me: Participant,
    public_key: AffinePoint,
    presignature: RerandomizedPresignOutput,
    msg_hash: Scalar,
) -> Result<impl Protocol<Output = SignatureOption>, InitializationError> {
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }

    let participants =
        ParticipantList::new(participants).ok_or(InitializationError::DuplicateParticipants)?;

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }

    // ensure number of participants during the signing phase is >= 2 * max_malicious + 1
    let robust_ecdsa_threshold = max_malicious
        .into()
        .value()
        .checked_mul(2)
        .and_then(|v| v.checked_add(1))
        .ok_or_else(|| {
            InitializationError::BadParameters(
                "2*threshold+1 must be less than usize::MAX".to_string(),
            )
        })?;
    if robust_ecdsa_threshold > participants.len() {
        return Err(InitializationError::BadParameters(
            "2*max_malicious+1 must be less than or equals to participant count".to_string(),
        ));
    }

    // The next two conditions prevent split-view attacks
    // documented in docs/ecdsa/robust_ecdsa/signing.md
    if participants.len() != robust_ecdsa_threshold {
        return Err(InitializationError::BadParameters(
            "the number of participants during signing must be exactly 2*max_malicious+1 to avoid split view attacks".to_string(),
        ));
    }
    if bool::from(msg_hash.is_zero()) {
        return Err(InitializationError::BadParameters(
            "msg_hash cannot be 0 to avoid potential split view attacks".to_string(),
        ));
    }
