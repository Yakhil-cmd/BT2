### Title
Montgomery Form Mismatch in `scalar32_delegation::decompose` Causes Proving/Execution Divergence in ecrecover — (`crypto/src/secp256k1/scalars/scalar32_delegation.rs`)

---

### Summary

`scalar32_delegation::ScalarInner::decompose` contains a Montgomery-form consistency bug. The constants `MINUS_B1`, `MINUS_B2`, and `MINUS_LAMBDA` are stored as raw integers (via `from_be_hex_unchecked`), but `mul_in_place` performs Montgomery multiplication. This causes the intermediate values to silently leave Montgomery form mid-computation, so the final `r1` and `k2` returned by `decompose` are mathematically wrong. Because `scalar32_delegation` is the backend used in the proving path (`feature = "bigint_ops"` on riscv32), while `scalar64` is used on 64-bit execution, the two paths compute different WNAF decompositions and therefore different public key points in `ecmult`, making any valid ecrecover result unprovable.

---

### Finding Description

**Backend selection** in `crypto/src/secp256k1/scalars/mod.rs`:

```
bigint_ops  → scalar32_delegation  (proving / riscv32)
64-bit      → scalar64             (execution)
``` [1](#0-0) 

**`scalar64::decompose`** operates entirely on plain integers. `mul_in_place` is a full Barrett/schoolbook modular multiply; `MINUS_B1`, `MINUS_B2`, `MINUS_LAMBDA` are plain integers. Every step is consistent. [2](#0-1) 

**`scalar32_delegation::decompose`** uses Montgomery arithmetic. `mul_in_place` calls `u256::mul_assign_montgomery`, which computes `a·b·R⁻¹ mod p`. The constants are defined with `from_be_hex_unchecked`, which stores raw integers — **not** Montgomery form: [3](#0-2) 

Contrast with `from_be_hex` (the converting variant): [4](#0-3) 

**Step-by-step form trace through `decompose`:** [5](#0-4) 

| Step | Operation | `c1` form | Result |
|---|---|---|---|
| `to_representation()` | `c1_int · R mod p` | Montgomery | ✓ |
| `mul_in_place(&MINUS_B1)` | `mont_mul(c1_mont, B1_raw) = c1_int · B1_int mod p` | **Raw integer** | ✓ (intentional one-sided) |
| `add_in_place(&c2)` | raw + raw | **Raw integer** | ✓ |
| `r1.mul_in_place(&MINUS_LAMBDA)` | `mont_mul(r1_raw, λ_raw) = r1 · λ · R⁻¹ mod p` | **Garbled** | ✗ should be `r1 · λ mod p` |
| `r1.add_in_place(&self)` | garbled + `self_int · R mod p` | **Garbled** | ✗ |
| `r1.to_integer()` | `mont_mul(garbled, 1) = garbled · R⁻¹ mod p` | **Wrong** | ✗ |

The final `r1` returned is `(k2 · λ · R⁻² + self_int) mod p` instead of `(k2 · λ + self_int) mod p`. The difference is `k2 · λ · (R⁻² − 1) mod p`, which is non-zero for virtually all inputs.

**`ecmult` consumes the decomposed scalars directly for WNAF:** [6](#0-5) 

The `bits`/`bits_var` functions on `scalar32_delegation` are documented as expecting integer-form inputs (the outputs of `decompose`), but the values are mathematically wrong, so the WNAF windows encode a different scalar, and `ecmult` computes a different affine point.

**The `test_decompose` test acknowledges the form difference but its `bigint_ops` workaround is also broken:** [7](#0-6) 

The workaround converts `r1, r2` to Montgomery form but leaves `lambda = -Scalar::MINUS_LAMBDA` as a raw integer. The subsequent `r2 * lambda` then computes `r2_int · λ_int mod p` (raw), and `r1 + (r2*lambda)` mixes Montgomery and raw forms, so the equality check `r1 + r2*lambda == k` does not verify the correct invariant. The proptest regression file records a confirmed failure case: [8](#0-7) 

---

### Impact Explanation

Every `ecrecover` call goes through `recover_with_context` → `ecmult` → `na.decompose()`. On the 64-bit execution node, `scalar64::decompose` returns the correct GLV split. On the riscv32 prover with `bigint_ops`, `scalar32_delegation::decompose` returns a wrong split. The WNAF representations differ, `ecmult` computes a different point, and the recovered public key differs. The proof for any valid ecrecover transaction cannot be generated, breaking the proving pipeline for all transactions that use ecrecover (e.g., EOA signature verification).

---

### Likelihood Explanation

The bug is deterministic and affects every input except the degenerate case where `k2 = 0`. Any ecrecover call with a non-trivial `s` scalar triggers it. No special attacker input is required; the divergence is structural.

---

### Recommendation

Store `MINUS_B1`, `MINUS_B2`, and `MINUS_LAMBDA` in Montgomery form by pre-computing them as `const` words (analogous to how `R2` is stored), or convert them at runtime via `to_representation()` before use. Ensure all operands to `mul_in_place` are in the same form. Add a cross-backend proptest that runs both `scalar32_delegation::decompose` and `scalar64::decompose` on the same input and asserts byte-equal outputs (after normalizing to integer form).

    pub(super) fn decompose(self) -> (Self, Self) {
        // Not to efficient as we kick out of Montgomery form
        let int_form = self.to_integer();
        let mut c1 = int_form;
        c1.integer_mul_shift_384_vartime(&Self::G1);
        let mut c2 = int_form;
        c2.integer_mul_shift_384_vartime(&Self::G2);

        r1.mul_in_place(&Self::MINUS_LAMBDA);
