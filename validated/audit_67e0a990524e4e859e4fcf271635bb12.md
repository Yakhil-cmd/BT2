Audit Report

## Title
Missing Neuron Lock in SNS `disburse_neuron` Enables Re-entrant Concurrent Execution - (File: rs/sns/governance/src/governance.rs)

## Summary
The SNS governance `disburse_neuron` function performs two sequential inter-canister ledger calls across two `await` points without acquiring a per-neuron in-flight command lock. The NNS governance equivalent explicitly acquires `lock_neuron_for_command` before any ledger interaction. A principal holding a dissolved SNS neuron with `NeuronPermissionType::Disburse` can submit two concurrent `manage_neuron` (Disburse) ingress messages; both will read identical pre-update neuron state and issue duplicate ledger transfers, corrupting governance accounting and enabling double-extraction of funds when the neuron subaccount balance exceeds the cached stake.

## Finding Description

`disburse_neuron` in `rs/sns/governance/src/governance.rs` (L1119‚Äď1237) executes the following sequence with no neuron lock held:

```
1. Read neuron state (cached_neuron_stake_e8s = S, neuron_fees_e8s = F)   ‚Üź no lock
2. ledger.transfer_funds(burn max_burnable_fee).await                       ‚Üź AWAIT POINT 1
3. Update cached_neuron_stake_e8s -= F; neuron_fees_e8s -= F
4. ledger.transfer_funds(disburse_amount_e8s).await                         ‚Üź AWAIT POINT 2
5. Update cached_neuron_stake_e8s -= disburse_amount_e8s + tx_fee
```

The IC execution model allows other ingress messages to be processed between any two `await` points. A second concurrent `disburse_neuron` call on the same neuron begins executing after Call A suspends at step 2. It reads the same stale `cached_neuron_stake_e8s = S` and `neuron_fees_e8s = F` (neither has been updated yet), computes the same `max_burnable_fee` and `disburse_amount_e8s`, and issues its own pair of ledger transfers.

The function's own docstring at L1118 lists "The neuron's id is not yet in the list of neurons with ongoing operations" as a precondition, but this is never enforced in the function body ‚ÄĒ no call to `lock_neuron_for_command` appears anywhere in `disburse_neuron`.

By contrast, NNS governance (`rs/nns/governance/src/governance.rs`, L2031‚Äď2037) acquires the lock before any ledger interaction:

```rust
let _neuron_lock = self.lock_neuron_for_command(
    id.id,
    NeuronInFlightCommand { timestamp: now, command: Some(InFlightCommand::Disburse(...)) },
)?;
```

The SNS `lock_neuron_for_command` function exists and is actively used elsewhere in the same file ‚ÄĒ for example, in `finalize_disburse_maturity` at L5006 ‚ÄĒ confirming this is an omission, not an intentional design choice.

**Concrete interleaving (neuron subaccount balance B ‚Č• 2S):**

| Step | Call A | Call B | Ledger balance |
|------|--------|--------|----------------|
| 1 | reads S, F | ‚ÄĒ | B |
| 2 | burn F ‚Üí await | ‚ÄĒ | B‚ąíF |
| 3 | ‚ÄĒ | reads S, F (stale) | B‚ąíF |
| 4 | ‚ÄĒ | burn F ‚Üí await | B‚ąí2F |
| 5 | burn done; update stake=S‚ąíF, fees=0 | ‚ÄĒ | B‚ąí2F |
| 6 | disburse S‚ąíF‚ąítx ‚Üí await | ‚ÄĒ | B‚ąí2F‚ąí(S‚ąíF‚ąítx)‚ąítx = B‚ąí2S |
| 7 | ‚ÄĒ | burn done; update stake=S‚ąí2F, fees=0 | B‚ąí2S |
| 8 | ‚ÄĒ | disburse S‚ąíF‚ąítx ‚Üí await | B‚ąí2S‚ąí(S‚ąíF‚ąítx)‚ąítx = B‚ąí3S+2F (succeeds if B‚Č•3S‚ąí2F) |

When B ‚Č• 2S (achievable by direct transfer to the neuron subaccount after staking, as the SNS ledger accepts arbitrary transfers to any subaccount), both disbursements succeed and the attacker extracts 2√ó(S‚ąíF‚ąítx_fee) tokens. The governance canister's `cached_neuron_stake_e8s` saturates to 0 via two independent `saturating_sub` calls, while the actual ledger disbursement exceeds the neuron's recorded stake.

Even when B = S (no top-up) and the second disburse fails, the double-burn path (both calls issuing `burn F` before either updates state, when S ‚Č• 2F) causes 2F tokens to be burned instead of F, leaving `cached_neuron_stake_e8s = S‚ąí2F` ‚ÄĒ an incorrect value that permanently understates the neuron's stake relative to what the ledger would show after a single correct disburse.

## Impact Explanation

**High ‚ÄĒ Unauthorized access to canister-controlled funds / governance accounting corruption.**

When the neuron subaccount holds B ‚Č• 2S (achievable by the attacker topping up their own neuron subaccount, or by any external transfer), the attacker extracts 2√ó(S‚ąíF‚ąítx_fee) tokens from a neuron whose recorded stake is S. The SNS governance canister's internal accounting (`cached_neuron_stake_e8s`) is permanently corrupted: it records 0 after both calls complete, while the actual disbursement exceeded the neuron's recorded stake. Even in the no-top-up case, the double-burn path corrupts governance stake accounting for any neuron where S ‚Č• 2F, causing the SNS to permanently misreport neuron stake and voting power.

## Likelihood Explanation

Any principal holding a dissolved SNS neuron with `NeuronPermissionType::Disburse` can trigger this by submitting two `manage_neuron` (Disburse) update calls in rapid succession. No special privilege, threshold corruption, social engineering, or external dependency is required. The IC's asynchronous execution model guarantees that both messages will be inducted into the canister's input queue; the first `await` on the ledger call yields control, allowing the second message to begin execution with stale state. The top-up precondition (B ‚Č• 2S) for the double-spend path is achievable by the attacker themselves via a direct ICRC-1 transfer to the neuron subaccount address.

## Recommendation

Acquire the per-neuron in-flight command lock at the start of `disburse_neuron`, before any `await` point, mirroring the NNS governance implementation:

```rust
pub async fn disburse_neuron(
    &mut self,
    id: &NeuronId,
    caller: &PrincipalId,
    disburse: &manage_neuron::Disburse,
) -> Result<u64, GovernanceError> {
    let neuron = self.get_neuron_result(id)?;
    neuron.check_authorized(caller, NeuronPermissionType::Disburse)?;
    // ... state checks ...

    // Acquire lock before any await point
    let _neuron_lock = self.lock_neuron_for_command(
        id,
        NeuronInFlightCommand {
            timestamp: self.env.now(),
            command: Some(neuron_in_flight_command::Command::Disburse(disburse.clone())),
        },
    )?;

    // ... existing ledger calls ...
}
```

The lock must be acquired before the first `await` and held (via RAII drop) until the function returns, ensuring a second concurrent call on the same neuron is rejected rather than proceeding with stale state.

## Proof of Concept

**Deterministic integration test using PocketIC:**

1. Deploy an SNS with a governance canister and an ICRC-1 ledger.
2. Stake a neuron N for principal P with stake S = 200_000_000 e8s, fees F = 10_000_000 e8s.
3. Directly transfer S additional tokens to the neuron's subaccount address (making ledger balance B = 2S = 400_000_000 e8s).
4. Advance time past the dissolve delay; verify `neuron.state() == Dissolved`.
5. Using PocketIC's `submit_call` (non-blocking), submit two concurrent `manage_neuron { command: Disburse { amount: None, to_account: P } }` ingress messages targeting neuron N without advancing rounds between submissions.
6. Advance rounds to allow both messages to execute and ledger responses to return.
7. Assert: recipient account balance = 2√ó(S‚ąíF‚ąítx_fee) = 2√ó(190_000_000 ‚ąí tx_fee) e8s.
8. Assert: SNS governance `cached_neuron_stake_e8s` for neuron N = 0.
9. Assert: total tokens disbursed > S (the neuron's recorded stake), demonstrating the double-extraction.
