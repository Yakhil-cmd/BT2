### Title
Missing Caller Authorization in Bank Precompile `transfer` and `burn` Allows Any Contract to Drain Native Bank Tokens from Arbitrary Addresses - (File: `x/cronos/keeper/precompiles/bank.go`)

---

### Summary

The bank precompile's `transfer` and `burn` methods accept an arbitrary `sender`/`recipient` address from calldata without verifying that `contract.Caller()` is authorized to act on behalf of that address. Any deployed EVM contract can call these methods to transfer or destroy native bank tokens (`evm/0xâ€¦` denom) held by any victim address, without the victim's consent.

---

### Finding Description

In `x/cronos/keeper/precompiles/bank.go`, the `TransferMethodName` case unpacks `sender` directly from ABI-encoded calldata (`args[0]`) and passes it as the debit address to `bc.bankKeeper.SendCoins`:

```go
sender := args[0].(common.Address)          // fully attacker-controlled
...
from := sdk.AccAddress(sender.Bytes())
denom := EVMDenom(contract.Caller())        // scoped to caller's denom
...
bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt))
```

There is no check that `contract.Caller() == sender`. The precompile only scopes the denom to the caller's address (`EVMDenom(contract.Caller()) = "evm/" + callerHex`), but the address being debited is entirely caller-supplied.

The same flaw exists in the `BurnMethodName` case: `recipient` (the address whose tokens are burned) is taken from `args[0]` with no check that `contract.Caller() == recipient`:

```go
recipient := args[0].(common.Address)       // fully attacker-controlled
addr := sdk.AccAddress(recipient.Bytes())
...
bc.bankKeeper.SendCoinsFromAccountToModule(ctx, addr, types.ModuleName, ...)
bc.bankKeeper.BurnCoins(ctx, types.ModuleName, ...)
```

The intended usage pattern (shown in `integration_tests/contracts/contracts/TestBank.sol`) passes `msg.sender` as the `sender` argument, but the precompile itself does not enforce this invariant. [1](#0-0) [2](#0-1) [3](#0-2) 

---

### Impact Explanation

**Critical â€” Unauthorized transfer and burn of precompile-controlled assets.**

Any EVM contract can:

1. Call `bank.transfer(victimAddr, attackerAddr, amount)` â†’ moves `evm/<callerContract>` native bank tokens from `victimAddr` to `attackerAddr` with no victim consent.
2. Call `bank.burn(victimAddr, amount)` â†’ destroys `evm/<callerContract>` native bank tokens held by `victimAddr` with no victim consent.

`evm/0xâ€¦` tokens are real, value-bearing assets: they are the native bank layer representation of EVM tokens. A user who calls `moveToNative` (burns ERC20, receives native bank tokens) holds assets that can be stolen or destroyed by the issuing contract at any time without further user interaction. [4](#0-3) [5](#0-4) 

---

### Likelihood Explanation

The attack path requires only that a victim holds `evm/<attackerContract>` native bank tokens. This occurs whenever:

- A user calls a contract function that internally calls `bank.mint(userAddr, amount)` (e.g., `moveToNative`).
- A contract airdrop or incentive mechanism mints `evm/â€¦` tokens to users.

An attacker can engineer this condition by deploying a contract that first mints tokens to users (making the contract appear legitimate), then later drains them. No privileged access, leaked keys, or governance action is required. The attacker only needs to deploy a contract and submit EVM transactions. [6](#0-5) 

---

### Recommendation

Add a caller-authorization check in both the `transfer` and `burn` cases of `BankContract.Run`:

**For `transfer`**: require `contract.Caller() == sender` before calling `SendCoins`.

**For `burn`**: require `contract.Caller() == recipient` before calling `SendCoinsFromAccountToModule`.

```go
// Transfer case
if contract.Caller() != sender {
    return nil, errors.New("bank precompile: caller not authorized to transfer from sender")
}

// Burn case
if contract.Caller() != recipient {
    return nil, errors.New("bank precompile: caller not authorized to burn from recipient")
}
```

This mirrors the access-control pattern already used in `ModuleCRC21.sol` and `ModuleCRC20.sol`, where privileged operations require `msg.sender == module_address`. [7](#0-6) 

---

### Proof of Concept

1. Attacker deploys `MaliciousContract` at address `0xATTACKER`.
2. `MaliciousContract` exposes a public `setup(address victim)` function that calls `bank.mint(victim, 1000)` â†’ victim now holds 1000 `evm/0xATTACKER` native bank tokens.
3. Attacker calls `MaliciousContract.drain(victimAddr, attackerEOA)` which internally calls `bank.transfer(victimAddr, attackerEOA, 1000)`.
4. The precompile executes `bc.bankKeeper.SendCoins(ctx, victimAddr, attackerEOA, 1000 evm/0xATTACKER)` with no authorization check.
5. Victim loses all `evm/0xATTACKER` tokens. No victim signature or approval was required at step 3.

The same flow applies to `burn`: replace step 3 with `bank.burn(victimAddr, 1000)` to destroy the victim's tokens entirely. [8](#0-7) [9](#0-8)
