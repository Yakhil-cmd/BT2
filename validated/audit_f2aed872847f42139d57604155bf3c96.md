### Title
Signature Verification Bypass via `skip_stateful_validations` Overly Broad Mempool Presence Check — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

`skip_stateful_validations` is intended to skip `__validate__` only when a `deploy_account` transaction for the sender is pending in the mempool (UX feature). However, the guard it uses — `account_tx_in_pool_or_recent_block` — returns `true` for **any** transaction type from the sender address, not exclusively `deploy_account`. An attacker can exploit this by first placing a valid nonce-0 invoke transaction from a target address into the mempool, then submitting a second invoke transaction with nonce=1 and an **invalid/arbitrary signature**. The gateway skips `__validate__` for the nonce=1 transaction, admitting it to the mempool without signature verification.

### Finding Description

The `skip_stateful_validations` function in `crates/apollo_gateway/src/stateful_transaction_validator.rs` is the gateway's UX shortcut for the deploy-account + invoke flow: [1](#0-0) 

The condition for skipping `__validate__` is:
1. Transaction is `Invoke`
2. `tx.nonce() == Nonce(Felt::ONE)`
3. `account_nonce == Nonce(Felt::ZERO)` (account not yet deployed on-chain)
4. `account_tx_in_pool_or_recent_block(sender_address)` returns `true`

The comment in the code states:

> "It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction **or transactions with future nonces that passed validations**."

The second branch of this reasoning is the flaw. `account_tx_in_pool_or_recent_block` is implemented as: [2](#0-1) 

It checks `MempoolState::contains_account` (any committed/staged address) and `TransactionPool::contains_account` (any address in the pool): [3](#0-2) [4](#0-3) 

Neither check is restricted to `deploy_account` transactions. Any transaction type from the sender address satisfies the condition.

When `skip_validate=true` is returned, `run_validate_entry_point` sets `validate: false` in `ExecutionFlags`: [5](#0-4) 

This causes `validate_tx` to return `Ok(None)` without executing `__validate__`: [6](#0-5) 

### Impact Explanation

An attacker can submit an invoke transaction with nonce=1 and an **arbitrary/invalid signature** for any account that currently has a nonce=0 transaction in the mempool (regardless of whether that nonce=0 tx is a `deploy_account` or a regular invoke). The gateway admits the nonce=1 transaction without running `__validate__`, violating the invariant that every admitted invoke transaction must have its signature verified before sequencing.

This matches the allowed impact: **"High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."**

The admitted invalid transaction occupies a mempool slot at `(sender_address, nonce=1)`. If the legitimate owner later submits a valid nonce=1 tx, it may be rejected as a duplicate nonce or require fee escalation to displace the attacker's invalid tx. When the batcher eventually executes the invalid tx, `__validate__` fails and the tx is rejected (no fee charged), but the mempool slot was held and the legitimate tx was delayed or blocked.

### Likelihood Explanation

The precondition — an account with on-chain nonce=0 that has a pending nonce=0 invoke in the mempool — is common. Any account deployed via a factory contract (not via `deploy_account`) starts with nonce=0 and can have a nonce=0 invoke in the pool. The `max_nonce_for_validation_skip` is set to `0x1` in production config, so the attack window is exactly nonce=1 per account, but an attacker can target many accounts simultaneously. No privileged access is required; the attacker only needs to observe the mempool. [7](#0-6) 

### Recommendation

Replace the overly broad `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a `deploy_account` transaction exists for the sender address. The mempool should expose a dedicated `deploy_account_tx_in_pool(address)` query that only returns `true` when a `DeployAccount` transaction is present for that address. Alternatively, store the transaction type alongside the address in `MempoolState` and `TransactionPool` so the check can be type-filtered.

### Proof of Concept

1. Account `A` is deployed via a factory contract (not via `deploy_account`). On-chain nonce of `A` = 0.
2. Legitimate owner of `A` submits `invoke_tx_0` (nonce=0, valid signature). It passes full gateway validation including `__validate__`. `account_tx_in_pool_or_recent_block(A)` now returns `true`.
3. Attacker submits `invoke_tx_1` (nonce=1, **invalid/random signature**, sender=`A`).
4. Gateway calls `skip_stateful_validations`:
   - `tx.nonce() == 1` ✓
   - `account_nonce == 0` ✓ (on-chain nonce still 0, `invoke_tx_0` not committed)
   - `account_tx_in_pool_or_recent_block(A)` → `true` ✓ (because `invoke_tx_0` is in the pool)
   - Returns `skip_validate = true`
5. `run_validate_entry_point` sets `validate: false`; `__validate__` is **never called**.
6. `invoke_tx_1` with invalid signature is admitted to the mempool.
7. The mempool slot `(A, nonce=1)` is now occupied by the attacker's invalid tx, blocking the legitimate owner's nonce=1 transaction. [8](#0-7) [9](#0-8)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
