### Title
`DeleteAccount` with `beneficiary_id == account_id` permanently burns the account's entire balance — (`File: runtime/runtime/src/actions.rs`)

### Summary

`action_delete_account` in `runtime/runtime/src/actions.rs` does not guard against `delete_account.beneficiary_id == account_id`. When a user submits a `DeleteAccount` action naming themselves as the beneficiary, the runtime (1) enqueues a `Receipt::new_balance_refund` to the same account, (2) removes the account from state, and (3) later processes the refund receipt against a now-nonexistent named account. Because NEAR burns tokens on failed refund receipts (predecessor `"system"`) rather than re-refunding them, the account's entire liquid balance is permanently destroyed. No validation at the transaction-admission layer or the action-execution layer prevents this.

### Finding Description

`action_delete_account` captures `account_balance` and pushes a balance-refund receipt addressed to `delete_account.beneficiary_id`, then calls `remove_account` to erase the account from the trie:

```rust
// runtime/runtime/src/actions.rs  lines 349-356
let account_balance = account_ref.amount();
if account_balance > Balance::ZERO {
    result
        .new_receipts
        .push(Receipt::new_balance_refund(&delete_account.beneficiary_id, account_balance));
}
let remove_result = remove_account(state_update, account_id)?;
```

`Receipt::new_balance_refund` sets `predecessor_id = "system"`, marking the receipt as a protocol-level refund. When this receipt is later dispatched to `beneficiary_id` (which equals the now-deleted `account_id`), `check_account_existence` finds no account and calls `check_transfer_to_nonexisting_account`. For a named account (non-implicit), this returns `AccountDoesNotExist`. Because the receipt is a refund, the runtime burns the tokens rather than generating a further refund receipt, so `account_balance` is permanently lost.

Neither `validate_action` nor `validate_actions_with_mode` checks for `beneficiary_id == account_id`. The existing test `test_validate_action_valid_delete_account` explicitly passes `beneficiary_id = alice_account()` against receiver `"alice.near"` and expects `Ok(())`, confirming the absence of any guard:

```rust
// runtime/runtime/src/action_validation.rs  lines 1026-1033
fn test_validate_action_valid_delete_account() {
    validate_action(
        &test_limit_config(),
        &Action::DeleteAccount(DeleteAccountAction { beneficiary_id: alice_account() }),
        &"alice.near".parse().unwrap(),
        PROTOCOL_VERSION,
    )
    .expect("valid action");
}
```

### Impact Explanation

The account's entire liquid balance (`account_balance`, a `u128` NEAR token amount) is permanently burned. The invariant "DeleteAccount transfers the remaining balance to the beneficiary" is violated: the balance leaves the sender but never arrives at any recipient. This is an irreversible accounting loss scoped to the **accounting** and **receipt causality** target areas.

### Likelihood Explanation

Any account owner can trigger this by signing a transaction with `Action::DeleteAccount { beneficiary_id: <own account id> }`. No privileged role is required. The scenario is analogous to the Unlock protocol finding: user error is required, but (a) NEAR already validates that `beneficiary_id` is a syntactically valid account ID, signalling intent to validate inputs; (b) contracts can programmatically emit `DeleteAccount` receipts via `promise_batch_action_delete_account`, widening the surface; and (c) the resulting behavior (silent balance destruction) is unique and almost certainly unintended.

### Recommendation

Add a guard in `action_delete_account` (or equivalently in `validate_action` for `Action::DeleteAccount`) that rejects the self-referential case:

```rust
// runtime/runtime/src/actions.rs  — inside action_delete_account, before line 350
if &delete_account.beneficiary_id == account_id {
    result.result = Err(ActionErrorKind::DeleteAccountSelfBeneficiary {
        account_id: account_id.clone(),
    }.into());
    return Ok(());
}
```

Alternatively, add the check at the validation layer in `validate_action_with_mode` so it is rejected before reaching execution.

### Proof of Concept

Call path for a named account `"alice.near"` with balance `B > 0`:

1. User submits `SignedTransaction { signer_id: "alice.near", receiver_id: "alice.near", actions: [DeleteAccount { beneficiary_id: "alice.near" }] }`.
2. Transaction passes `validate_verify_and_charge_transaction` — no guard on `beneficiary_id == account_id`. [1](#0-0) 
3. Receipt is routed to `"alice.near"`. `action_delete_account` is called. [2](#0-1) 
4. `account_balance = B` is captured; a `Receipt::new_balance_refund("alice.near", B)` (predecessor `"system"`) is pushed to `result.new_receipts`. [3](#0-2) 
5. `remove_account(state_update, "alice.near")` erases the account from the trie. [4](#0-3) 
6. The refund receipt is dispatched. `check_account_existence` for `Action::Transfer` finds no account for `"alice.near"` (named, non-implicit) and returns `AccountDoesNotExist`. [5](#0-4) 
7. Because the receipt is a refund (predecessor `"system"`), the runtime burns `B` rather than generating another refund. `B` yoctoNEAR is permanently destroyed.

The exact corrupted value is `account_ref.amount()` at line 350 of `runtime/runtime/src/actions.rs` — the full liquid balance of the deleted account. [6](#0-5)
