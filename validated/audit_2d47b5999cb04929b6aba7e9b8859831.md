### Title
Missing Silo `Address` Whitelist Check in `receive_base_tokens` Allows Minting ETH to Non-Whitelisted Addresses - (File: `engine/src/engine.rs`)

### Summary
In Silo mode with the `Address` whitelist enabled, `Engine::receive_base_tokens` mints ETH to any attacker-specified recipient without checking the Silo `Address` whitelist. The parallel ERC-20 path (`receive_erc20_tokens`) correctly enforces the whitelist and redirects tokens to a fallback address when the recipient is not whitelisted. The base-token path has no such guard, allowing ETH to be deposited into non-whitelisted EVM addresses, where it becomes permanently inaccessible because those addresses are blocked from submitting transactions.

### Finding Description

`ft_on_transfer` dispatches to one of two paths depending on whether the predecessor is the ETH connector: [1](#0-0) 

The ERC-20 path (`receive_erc20_tokens`) enforces the Silo whitelist and redirects to a fallback address when the recipient is not whitelisted: [2](#0-1) 

The base-token path (`receive_base_tokens`) performs no such check. It parses the recipient directly from the attacker-controlled `msg` field and unconditionally mints ETH to that address: [3](#0-2) 

The `msg` field is a hex-encoded EVM address supplied by the caller of `ft_transfer_call` on the ETH connector: [4](#0-3) 

The Silo `Address` whitelist check that should apply here is `silo::is_allow_receive_erc20_tokens`, which delegates to `is_address_allowed`: [5](#0-4) 

### Impact Explanation

**High — Temporary (potentially permanent) freezing of funds.**

ETH minted to a non-whitelisted address in Silo mode is inaccessible: `is_allow_submit` blocks that address from submitting any EVM transaction, and there is no self-service recovery path. The funds remain frozen until the Silo operator explicitly whitelists the address. If the operator never does so, the freeze is permanent. [6](#0-5) 

### Likelihood Explanation

**High.** Any NEAR account holding NEP-141 ETH (ETH-on-NEAR) can trigger this path by calling `ft_transfer_call` on the ETH connector with `receiver_id = aurora` and `msg = <any_non_whitelisted_evm_address_hex>`. No special privileges are required. The Silo `Address` whitelist is a documented, production-deployed feature.

### Recommendation

Apply the same whitelist guard used in `receive_erc20_tokens` to `receive_base_tokens`. When the Silo fallback address is set and the recipient is not whitelisted, redirect the mint to the fallback address:

```rust
pub fn receive_base_tokens(
    &mut self,
    args: &FtOnTransferArgs,
) -> Result<Option<SubmitResult>, ContractError> {
    let message_data = FtTransferMessageData::try_from(args.msg.as_str())?;
    let amount = Wei::new_u128(args.amount.as_u128());
    let mut recipient = message_data.recipient;

    // Apply Silo whitelist check (mirrors receive_erc20_tokens)
    if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
        && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
    {
        recipient = fallback_address;
    }

    let balance = get_balance(&self.io, &recipient);
    let new_balance = balance
        .checked_add(amount)
        .ok_or(errors::ERR_BALANCE_OVERFLOW)?;
    set_balance(&mut self.io, &recipient, &new_balance);
    Ok(None)
}
```

### Proof of Concept

1. Deploy Aurora Engine in Silo mode: call `set_silo_params` with a `erc20_fallback_address` and enable the `Address` whitelist via `set_whitelist_status(WhitelistKind::Address, active: true)`.
2. Do **not** add victim address `V` to the `Address` whitelist.
3. As any NEAR account holding NEP-141 ETH, call `ft_transfer_call` on the ETH connector:
   ```json
   { "receiver_id": "aurora", "amount": "1000000000000000000", "msg": "<hex of V>" }
   ```
4. The ETH connector calls `ft_on_transfer` on Aurora → `receive_base_tokens` is invoked → ETH is minted to `V` with no whitelist check.
5. Attempt any EVM transaction from `V`: rejected by `is_allow_submit` because `V` is not in the `Address` whitelist.
6. The ETH at `V` is frozen. [7](#0-6) [2](#0-1)

```rust
impl TryFrom<&str> for FtTransferMessageData {
    type Error = errors::ParseOnTransferMessageError;

    fn try_from(message: &str) -> Result<Self, Self::Error> {
        if message.len() == 40 {
            // Parse message to determine recipient
            let recipient = {
                // Message format:
                // Recipient of the transaction - 40 characters (Address in hex)
                let mut address_bytes = [0; 20];
                hex::decode_to_slice(message, &mut address_bytes)
                    .map_err(|_| errors::ParseOnTransferMessageError::InvalidHexData)?;
                Address::from_array(address_bytes)
            };

            #[allow(deprecated)]
            return Ok(Self {
                recipient,
                fee: None,
            });
```
