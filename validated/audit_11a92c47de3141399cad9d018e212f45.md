### Title
Participant-Set Mismatch in OT-Based ECDSA Presign Allows Corrupted Presignature Acceptance - (`File: src/ecdsa/ot_based_ecdsa/presign.rs`)

---

### Summary

The `presign` function in the OT-based ECDSA scheme explicitly omits the check that the Beaver triples supplied as input were generated by the same participant set that is running the presign protocol. A malicious coordinator or misconfigured orchestrator can supply triples generated under a different participant set, causing all honest parties to compute and accept a presignature whose internal Lagrange-linearized values are cryptographically inconsistent. The resulting `PresignOutput` is unusable for signing, permanently denying the signing capability for that presignature slot.

---

### Finding Description

The `presign` function in `src/ecdsa/ot_based_ecdsa/presign.rs` accepts a `PresignArguments` struct containing two Beaver triples (`triple0`, `triple1`) and a `participants` slice. The `TriplePub` struct (defined in `src/ecdsa/ot_based_ecdsa/triples/mod.rs`) records the exact participant set that generated each triple in its `participants` field.

However, the `presign` initializer contains an explicit, acknowledged omission:

```rust
// NOTE: We omit the check that the new participant set was present for
// the triple generation, because presumably they need to have been present
// in order to have shares.
``` [1](#0-0) 

The only cross-check performed is that the threshold scalar matches:

```rust
if args.threshold != args.triple0.1.threshold || args.threshold != args.triple1.1.threshold {
    return Err(InitializationError::BadParameters(...));
}
``` [2](#0-1) 

No check is made that `args.triple0.1.participants` or `args.triple1.1.participants` match the `participants` argument. The `TriplePub` struct stores this information but it is never consulted:

```rust
pub struct TriplePub {
    pub big_a: AffinePoint,
    pub big_b: AffinePoint,
    pub big_c: AffinePoint,
    pub participants: Vec<Participant>,   // <-- never validated against presign participants
    pub threshold: ReconstructionLowerBound,
}
``` [3](#0-2) 

Inside `do_presign`, each party linearizes its triple shares using Lagrange coefficients computed over the **presign** participant set:

```rust
let lambda_me = participants.lagrange::<Secp256>(me)?;
let k_prime_i = lambda_me * k_i;
let e_i: Scalar = lambda_me * e_i;
let a_prime_i = lambda_me * a_i;
let b_prime_i = lambda_me * b_i;
let x_prime_i = lambda_me * private_share;
``` [4](#0-3) 

But the triple shares `(k_i, a_i, b_i, c_i, e_i)` were secret-shared using Lagrange coefficients over the **triple-generation** participant set. If the two sets differ, the linearized values do not reconstruct the correct secrets. The invariant `a * b = c` (and the corresponding public commitment check `E == e * G`) is violated, producing a presignature `(big_r, k, sigma)` that is cryptographically inconsistent.

---

### Impact Explanation

**Impact: High — Corruption of presign outputs so honest parties accept unusable cryptographic outputs.**

When triples generated by participant set `T` are used in a presign run by participant set `P` (where `T ≠ P`), the Lagrange linearization in `do_presign` applies coefficients from `P` to shares that were secret-shared under `P`'s basis. The reconstructed scalars `alpha = sum_j alpha_j` and `beta = sum_j beta_j` do not satisfy `alpha * G == K + A` or `beta * G == X + B` (the checks at line 162 of `presign.rs`), causing the protocol to abort with an `AssertionFailed` error — or, if the sets happen to have the same threshold and the public-commitment checks pass by coincidence of structure, producing a silently wrong presignature. In either case, the presignature slot is permanently consumed and the signing round that depends on it cannot complete, permanently denying signing for that session. [5](#0-4) 

---

### Likelihood Explanation

The library is designed to be called by an external orchestrator that manages triple pools and presign sessions. The `PresignArguments` struct is constructed entirely by the caller — there is no runtime enforcement that the triples come from the same session or participant set as the presign invocation. A malicious coordinator, a buggy orchestrator, or an attacker who can influence which triples are assigned to which presign session can trigger this condition without any privileged access beyond the ability to call `presign()` with attacker-chosen `PresignArguments`. The orchestration documentation explicitly warns that outputs must be tracked and consumed correctly, but the library itself provides no enforcement. [6](#0-5) 

---

### Recommendation

Add an explicit participant-set consistency check in the `presign` initializer, comparing the presign `participants` slice against `args.triple0.1.participants` and `args.triple1.1.participants`:

```rust
// Enforce that triples were generated by the same participant set
if args.triple0.1.participants != participants.as_slice() {
    return Err(InitializationError::BadParameters(
        "triple0 participants must match presign participants".to_string(),
    ));
}
if args.triple1.1.participants != participants.as_slice() {
    return Err(InitializationError::BadParameters(
        "triple1 participants must match presign participants".to_string(),
    ));
}
```

This removes the reliance on the "presumably" assumption noted in the comment and makes the invariant machine-enforced rather than caller-convention-enforced. [7](#0-6) 

---

### Proof of Concept

**Setup:**
- Participants for triple generation: `T = {A, B, D}`, threshold = 2
- Participants for presign: `P = {A, B, C}`, threshold = 2

**Steps:**
1. Run `generate_triple` with `T = {A, B, D}` to produce `(triple_share_A, triple_pub)` where `triple_pub.participants = [A, B, D]`.
2. Call `presign(&[A, B, C], A, PresignArguments { triple0: (triple_share_A, triple_pub), ..., threshold: 2 })`.
3. The initializer passes all checks: `participants.len() >= 2`, `threshold <= participants.len()`, `threshold == triple_pub.threshold`.
4. Inside `do_presign`, `lambda_A` is computed over `{A, B, C}` but `k_A` was secret-shared over `{A, B, D}`. The linearized `k_prime_A = lambda_A({A,B,C}) * k_A` is incorrect.
5. The reconstructed `alpha = sum_j alpha_j` does not satisfy `alpha * G == K + A`, causing `AssertionFailed` — the presignature is permanently corrupted and the signing session cannot proceed. [3](#0-2) [7](#0-6) [4](#0-3)

pub fn presign(
    participants: &[Participant],
    me: Participant,
    args: PresignArguments,
) -> Result<impl Protocol<Output = PresignOutput>, InitializationError> {
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }
    // Spec 1.1
    if args.threshold.value() > participants.len() {
        return Err(InitializationError::ThresholdTooLarge {
            threshold: args.threshold.value(),
            max: participants.len(),
        });
    }

    // NOTE: We omit the check that the new participant set was present for
    // the triple generation, because presumably they need to have been present
    // in order to have shares.

    // Also check that we have enough participants to reconstruct shares.
    if args.threshold != args.triple0.1.threshold || args.threshold != args.triple1.1.threshold {
        return Err(InitializationError::BadParameters(
            "New threshold must match the threshold of both triples".to_string(),
        ));
    }

    let participants =
        ParticipantList::new(participants).ok_or(InitializationError::DuplicateParticipants)?;

    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    let ctx = Comms::new();
    let fut = do_presign(ctx.shared_channel(), participants, me, args);
    Ok(make_protocol(ctx, fut))
}
