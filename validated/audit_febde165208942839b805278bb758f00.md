### Title
Unauthorized Burn of Victim's `evm/<contract>` Token Balance via `BankContract.Run` — (`x/cronos/keeper/precompiles/bank.go`)

### Summary
The `BurnMethodName` branch in `BankContract.Run` takes the burn-from address from ABI `args[0]` (attacker-controlled) and the denom from `contract.Caller()`, but never verifies that the caller is authorized to burn from that address. Any EVM contract can burn `evm/<itself>` tokens held by an arbitrary victim.

### Finding Description
In `bank.go` lines 113–155, `MintMethodName` and `BurnMethodName` share one unpacking block. `recipient := args[0].(common.Address)` is the address whose balance is affected. For `mint` this is the recipient (correct — minting to anyone is fine). For `burn`, this is the address whose tokens are destroyed. The denom is `evm/<contract.Caller()>`, so the attacker controls which token is burned, but `addr` (the victim) is fully attacker-controlled with no ownership check.

The burn path calls:
```go
bc.bankKeeper.SendCoinsFromAccountToModule(ctx, addr, types.ModuleName, sdk.NewCoins(amt))
bc.bankKeeper.BurnCoins(ctx, types.ModuleName, sdk.NewCoins(amt))
```
`addr` is the victim. No check of the form `contract.Caller() == recipient` exists.

### Impact Explanation
An unprivileged EVM contract can destroy `evm/<contract>` tokens held by any address without that address's consent. This is an unauthorized burn of precompile-controlled assets — a Critical-tier impact per the scope rules.

### Likelihood Explanation
Any deployed EVM contract that has previously minted `evm/<contract>` tokens to users (e.g., as part of a DeFi protocol) can immediately exploit this to destroy user balances. No privileged access, leaked keys, or external assumptions are required.

### Recommendation
Add an authorization check in the `BurnMethodName` branch requiring that the burn-from address equals the caller:

```go
} else { // BurnMethodName
    if recipient != contract.Caller() {
        return nil, errors.New("burn: caller is not authorized to burn from this address")
    }
    // ... existing burn logic
}
```

Alternatively, split the `MintMethodName` and `BurnMethodName` cases so burn always uses `contract.Caller()` as the source address, consistent with the self-authorization invariant.

### Proof of Concept

1. Deploy attacker contract `A` on Cronos EVM.
2. Call `bank.mint(victimAddress, 1000)` from `A` → victim now holds 1000 `evm/A` tokens.
3. Call `bank.burn(victimAddress, 1000)` from `A` → `SendCoinsFromAccountToModule(ctx, victimAddr, ...)` executes with no authorization check.
4. Assert victim's `evm/A` balance is now 0 — achieved without any signature or consent from the victim.
