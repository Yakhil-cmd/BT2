### Title
`skip_stateful_validations` Allows Attacker to Bypass `__validate__` Signature Check for Invoke Transactions Targeting Accounts with Pending Deploy-Account in Mempool - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function skips the blockifier `__validate__` entry-point call for any Invoke transaction with `nonce == 1` whenever `account_tx_in_pool_or_recent_block` returns `true` for the sender address. Because that predicate returns `true` for **any** transaction already in the pool for that address — not exclusively a deploy-account — an unprivileged attacker who observes a victim's pending `deploy_account` in the mempool can immediately submit a crafted Invoke with `nonce = 1` carrying an **invalid or adversarial signature**. The gateway accepts it without ever calling `__validate__`, inserting it into the mempool and blocking the victim's legitimate first Invoke.

---

### Finding Description

`skip_stateful_validations` is the UX shortcut that lets a user broadcast `deploy_account + invoke` atomically before the account exists on-chain:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs  lines 429-461
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            // ...
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                ...;
        }
    }
    Ok(false)
}
``` [1](#0-0) 

When it returns `true`, `run_validate_entry_point` sets `validate: false` in the execution flags, so the blockifier's `StatefulValidator::perform_validations` returns `Ok(())` immediately without calling `__validate__`:

```rust

    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [2](#0-1) 

```rust
// blockifier/src/blockifier/stateful_validator.rs  lines 79-81

``` [3](#0-2) 

The guard that is supposed to justify skipping is:

```rust
// mempool/src/mempool.rs  lines 697-700
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [4](#0-3) 

This predicate returns `true` for **any** transaction in the pool for that address. The code comment acknowledges this looseness ("it means that either it has a deploy_account transaction **or** transactions with future nonces that passed validations"), but the second branch is reachable by an attacker: [5](#0-4) 

**Attack path:**

1. Victim broadcasts `deploy_account` for address `A` (nonce 0). It enters the mempool; `account_tx_in_pool_or_recent_block(A)` now returns `true`.
2. Attacker observes the pending `deploy_account` and immediately submits an `Invoke` for address `A` with `nonce = 1` and an **invalid/adversarial signature** (e.g., all-zero signature, or calldata that drains the account).
3. Gateway path for the attacker's Invoke:
   - **Stateless validation** (`StatelessTransactionValidator::validate`): passes — signature length is within bounds, resource bounds are non-zero, nonce DA mode is L1.
   - **`validate_state_preconditions`**: passes — `account_nonce = 0`, `tx_nonce = 1`, within `max_allowed_nonce_gap`.
   - **`validate_by_mempool`**: passes — no duplicate hash, nonce 1 is not yet in the pool.
   - **`skip_stateful_validations`**: returns `true` because `account_tx_in_pool_or_recent_block(A) == true`.
   - **`run_validate_entry_point`**: called with `skip_validate = true` → `__validate__` is **never called**. Signature is never verified.
4. The attacker's Invoke (invalid signature) is forwarded to `mempool.add_tx`. [6](#0-5) 

5. Victim now submits their legitimate Invoke with `nonce = 1`. The mempool rejects it with `DuplicateNonce` (or demands fee escalation to replace the attacker's entry). [7](#0-6) 

The attacker's invalid Invoke occupies the `nonce = 1` slot. When the batcher eventually tries to execute it, `__validate__` fails and the transaction is dropped from the block — but it remains in the mempool until TTL expiry, continuously blocking the victim's slot.

---

### Impact Explanation

**Admitted invalid transaction / valid transaction rejected before sequencing (High).**

- An unprivileged attacker can insert a signature-less (or adversarially-signed) Invoke into the mempool for any account whose `deploy_account` is pending.
- The victim's legitimate first Invoke is blocked for the full mempool TTL, or until the victim pays a fee-escalation premium to evict the attacker's entry.
- At scale, an attacker monitoring the mempool can grief every new account deployment simultaneously with negligible cost (no fee is charged for a transaction that never lands on-chain).

---

### Likelihood Explanation

The trigger is fully unprivileged and requires only:
1. Observing a `deploy_account` transaction in the public mempool (trivially done via any node's `get_txs` or P2P gossip).
2. Submitting a well-formed Invoke with `nonce = 1` and any signature bytes before the victim does.

The race window is the entire time the `deploy_account` sits unconfirmed — typically multiple seconds to minutes. No special knowledge of the victim's private key or contract code is needed.

---

### Recommendation

**Short term:** In `skip_stateful_validations`, verify that the transaction already in the pool for the account is specifically a `deploy_account` transaction (nonce 0, `DeployAccount` type), not merely any transaction. The mempool should expose a typed query, or the gateway should call `account_tx_in_pool_or_recent_block` only after confirming the pool entry is a `DeployAccount`.

**Long term:** Audit all control-flow branches that skip cryptographic validation based on untrusted mempool state. The invariant "a pending deploy_account justifies skipping `__validate__`" must be enforced by checking the *type* of the pending transaction, not merely its existence.

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
