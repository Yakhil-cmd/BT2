### Title
Pool admin transfer does not clear `poolAdminFeeDestination`, allowing old admin to drain accrued fees after handover тАФ (`File: metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary

`acceptPoolAdmin` updates `poolAdmin[pool]` to the new admin and clears `pendingPoolAdmin[pool]`, but leaves `poolAdminFeeDestination[pool]` pointing to the old admin's chosen address. Because `collectPoolFees` is permissionless, the old admin (or any keeper) can immediately call it after the handover and sweep all accrued admin fees to the old destination before the new admin has a chance to update it.

### Finding Description

The two-step admin transfer flow in `MetricOmmPoolFactory` is:

1. Old admin calls `proposePoolAdminTransfer(pool, newAdmin)` тАФ sets `pendingPoolAdmin[pool] = newAdmin`.
2. New admin calls `acceptPoolAdmin(pool)` тАФ sets `poolAdmin[pool] = newAdmin`, deletes `pendingPoolAdmin[pool]`. [1](#0-0) 

After step 2, `poolAdminFeeDestination[pool]` is **not touched**; it still holds whatever address the old admin configured. [2](#0-1) 

`collectPoolFees` is documented as **permissionless** тАФ any address may call it. It routes the admin share of accrued fees directly to `poolAdminFeeDestination[pool]`: [3](#0-2) 

Likewise, `setPoolAdminFees` тАФ the function the new admin would naturally call to tune fees тАФ also collects accrued fees to `poolAdminFeeDestination[pool]` **before** applying the new rates: [4](#0-3) 

This means that even if the new admin calls `setPoolAdminFees` immediately after accepting, any fees that accrued before `setPoolAdminFeeDestination` is called are swept to the old admin's address.

### Impact Explanation

The new admin loses all admin fees that accrued from pool inception (or the last `collectPoolFees` call) up to the moment they successfully call `setPoolAdminFeeDestination`. The old admin (or any front-running bot) can call the permissionless `collectPoolFees` in the same block as `acceptPoolAdmin` to maximise the drained amount. This is a direct loss of protocol-fee revenue owed to the new admin, matching the "direct loss of protocol fees" impact gate.

### Likelihood Explanation

The attack window opens the moment `acceptPoolAdmin` is mined and closes when the new admin's `setPoolAdminFeeDestination` transaction is confirmed. On a busy pool the accrued admin balance can be substantial. The old admin has a clear economic incentive (they receive the drained fees) and the call is a single, cheap, permissionless transaction. No special privileges are required after the handover тАФ the old admin is already out of the `onlyPoolAdmin` gate.

### Recommendation

In `acceptPoolAdmin`, either:

- **Clear the fee destination** (set `poolAdminFeeDestination[pool] = address(0)` or a safe sentinel) so that `collectPoolFees` cannot route fees until the new admin explicitly sets a destination; or
- **Require the new admin to supply a new fee destination** as a parameter to `acceptPoolAdmin` and update it atomically in the same call, mirroring how `createPool` requires `adminFeeDestination` at creation time.

### Proof of Concept

```

2. Old admin calls proposePoolAdminTransfer(pool, newAdmin).

  /// @inheritdoc IMetricOmmPoolFactoryPoolAdmin
  function proposePoolAdminTransfer(address pool, address newAdmin) external override nonReentrant onlyPoolAdmin(pool) {
    if (newAdmin == address(0)) revert InvalidAdmin();
    if (newAdmin == poolAdmin[pool]) revert InvalidAdmin();
    pendingPoolAdmin[pool] = newAdmin;
    emit PoolAdminTransferProposed(pool, poolAdmin[pool], newAdmin);
  }

  /// @inheritdoc IMetricOmmPoolFactoryPoolAdmin
  function acceptPoolAdmin(address pool) external override nonReentrant {
    address pending = pendingPoolAdmin[pool];
    if (pending == address(0)) revert NoPendingPoolAdminTransfer();
    if (msg.sender != pending) revert NotPendingPoolAdmin(pool, msg.sender, pending);
    address previousAdmin = poolAdmin[pool];
    poolAdmin[pool] = pending;
    delete pendingPoolAdmin[pool];
    emit PoolAdminTransferred(pool, previousAdmin, pending);
  }
