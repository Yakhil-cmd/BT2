### Title
WalletConnect `chia_showNotification` Injects Arbitrary Offer Data Into Notification Panel Without Content Validation - (File: packages/gui/src/electron/utils/showNotification.ts)

### Summary
A dApp granted the `chia_showNotification` WalletConnect permission can inject any arbitrary raw offer string into the user's notification panel by supplying it as the `offer_data` parameter. The notification system accepts any non-empty string as offer data without validating its source, domain, or relationship to the dApp. The user is then presented with a "You have a new offer" notification that, when clicked, opens the offer acceptance flow for the injected offer — an offer the user did not seek out and that may be unfavorable.

### Finding Description

The `chia_showNotification` WalletConnect command is handled by `showNotification` in `packages/gui/src/electron/utils/showNotification.ts`. When `type` is `NotificationType.OFFER`, the handler accepts any non-empty string as `offer_data` and passes it verbatim into the notification payload:

```ts
if (type === NotificationType.OFFER) {
  const { offer_data: offerData } = params;
  if (typeof offerData !== 'string' || !offerData) {
    throw new Error('Notification missing offer_data');
  }
  return { ...base, type, offerData };   // raw string, no further validation
}
``` [1](#0-0) 

This notification payload is then dispatched to the renderer via `context.sendNotification(notification)`. [2](#0-1) 

In `OffersProvider.prepareOfferData`, if the string is not a URL it is returned as-is and passed directly to `getOfferSummary` and `checkOfferValidity`:

```ts
const prepareOfferData = useCallback(async (urlOrOfferData: string) => {
  if (!isValidURL(urlOrOfferData)) {
    return urlOrOfferData;   // raw offer blob, no source check
  }
  // URL path: domain must be in OfferServices allowlist
  ...
``` [3](#0-2) 

When the user clicks the resulting notification, `NotificationOffer` calls `useOffer(offerURLOrData)` with the injected string, which resolves the offer and navigates to the offer acceptance view: [4](#0-3) 

Contrast this with the URL path: when `offer_data` is a URL, `prepareOfferData` enforces that the domain must belong to a known `OfferServices` allowlist. No equivalent source restriction exists for raw offer blobs supplied via `chia_showNotification`. [5](#0-4) 

The `chia_showNotification` command definition does not set `allowConfirmationBypass: true`, so each invocation requires a per-call user confirmation dialog. However, that dialog presents the raw bech32 offer string under the label "Offer Data" — opaque to virtually all users — alongside the generic message "This app wants to show you a notification." Users routinely click through without understanding they are authorizing injection of a specific offer into their acceptance flow. [6](#0-5) 

### Impact Explanation

A dApp with `chia_showNotification` permission can push any valid on-chain offer — one crafted to be unfavorable to the user (e.g., selling an NFT far below market value, or buying a CAT at an inflated price) — directly into the user's notification panel. The notification appears attributed to the dApp ("from: [dApp name]"), lending it false legitimacy. When the user clicks it, they are taken to the offer acceptance UI with the injected offer pre-loaded. This constitutes spoofing of notification/WalletConnect state that causes a user to display and potentially approve the wrong offer — matching the High impact category.

### Likelihood Explanation

Any dApp that has obtained `chia_showNotification` permission (a low-friction grant, as it is categorized under "Show notifications") can exploit this. The per-call confirmation dialog is ineffective because the raw offer blob is meaningless to users. The attacker only needs one granted permission and one user click-through on the confirmation.

### Recommendation

- **Short term:** In `showNotification.ts`, when `type === NotificationType.OFFER`, reject `offer_data` values that are not URLs from the `OfferServices` allowlist. Raw offer blobs should not be accepted through the WalletConnect notification channel; offers should only be referenced by URL so the domain allowlist applies uniformly.
- **Long term:** Separate the "show a UI notification" capability from the "inject an offer into the acceptance flow" capability. The `chia_showNotification` command should not be able to pre-load offer data; offer acceptance should only be reachable through explicit user-initiated import flows.

### Proof of Concept

1. Register a dApp and request `chia_showNotification` permission. User grants it.
2. From the dApp, call `chia_showNotification` with:
   ```json
   {
     "type": "offer",
     "offer_data": "offer1<bech32-encoded-unfavorable-offer>"
   }
   ```
3. The confirmation dialog appears: "This app wants to show you a notification / Offer Data: offer1qpzry9x8…". User clicks "Show."
4. A notification "You have a new offer" appears in the wallet UI, attributed to the dApp.
5. User clicks the notification; `NotificationOffer` calls `useOffer(offerData)` → `OffersProvider.prepareOfferData` returns the raw blob → `getOfferSummary` + `checkOfferValidity` resolve it → offer acceptance view opens with the injected offer pre-loaded.
6. User, trusting the dApp context, accepts the offer.

The missing guard is at `showNotification.ts` line 36–44: `offer_data` is only checked to be a non-empty string. No domain, source, or format restriction is applied, unlike the URL path in `OffersProvider.prepareOfferData` which enforces the `OfferServices` allowlist. [7](#0-6) [3](#0-2)
