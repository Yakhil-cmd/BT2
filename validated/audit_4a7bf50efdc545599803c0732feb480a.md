Audit Report

## Title
`SenderInfoContent` Canister Signature Does Not Bind to Sender Principal, Enabling Cross-Sender Replay - (File: `rs/types/types/src/messages/http.rs`)

## Summary
`SenderInfoContent` wraps only the raw `info` blob in its signed content; the sender's principal is never included. An attacker who observes any accepted ingress message can copy the `{info, signer, sig}` triple verbatim into their own request under a different principal. `verify_sender_info_canister_sig` will accept the replayed attestation because the canister signature is valid over `SenderInfoContent(bytes)` regardless of who the sender is.

## Finding Description
`SenderInfoContent` is defined as a newtype over `&[u8]` and its `SignedBytesWithoutDomainSeparator` implementation appends only those raw bytes:

```rust
// rs/types/types/src/messages/http.rs:342-348
pub struct SenderInfoContent<'a>(pub &'a [u8]);

impl crate::crypto::SignedBytesWithoutDomainSeparator for SenderInfoContent<'_> {
    fn write_signed_bytes_without_domain_separator(&self, bytes: &mut Vec<u8>) {
        bytes.extend_from_slice(self.0);   // only info blob, no sender principal
    }
}
```

The full signed bytes are therefore `"\x0Eic-sender-info" || info_bytes`. The sender principal (`UserId`) is absent.

`validate_sender_info` has access to `request` (and thus `request.sender()`), but passes only `sender_info` to `verify_sender_info_canister_sig`:

```rust
// rs/validator/src/ingress_validation.rs:482-487
verify_sender_info_canister_sig(
    sender_info,
    sender_pubkey,
    ingress_signature_verifier,
    root_of_trust_provider,
)
```

Inside `verify_sender_info_canister_sig`, the signable content is constructed without the sender:

```rust
// rs/validator/src/ingress_validation.rs:530
let sender_info_content = SenderInfoContent(&sender_info.info);
```

The three checks performed (valid canister-sig public key, canister ID matches `signer`, signature valid over `SenderInfoContent(bytes)`) are all satisfied by a replayed `{info, signer, sig}` triple regardless of the request's sender principal.

## Impact Explanation
Any canister that calls `ic0.msg_caller_info_data()` and makes security-sensitive decisions based on the returned blob (e.g., Internet Identity attesting KYC status, age verification, or user-specific credentials) can be deceived. An unprivileged attacker with only their own key can present another user's attested attributes to any canister, potentially bypassing access controls or gaining unauthorized privileges. This matches the Medium bounty impact: meaningful security impact with a concrete constraint (requires a target canister that acts on `msg_caller_info_data()` without re-checking the sender principal inside the blob).

## Likelihood Explanation
Ingress messages are publicly observable on the IC. Extracting `{info, signer, sig}` from an accepted ingress message requires no privileged access, no key material, and no cryptographic break. The only constraint is finding a target canister that trusts `sender_info` for authorization — which is the explicitly intended use case for the feature (Internet Identity attesting caller attributes). The attack is immediately repeatable for any such canister.

## Recommendation
Include the sender's principal bytes in the signed content of `SenderInfoContent`:

```rust
pub struct SenderInfoContent<'a> {
    pub sender: &'a [u8],
    pub info: &'a [u8],
}

impl SignedBytesWithoutDomainSeparator for SenderInfoContent<'_> {
    fn write_signed_bytes_without_domain_separator(&self, bytes: &mut Vec<u8>) {
        bytes.extend_from_slice(self.sender);
        bytes.extend_from_slice(self.info);
    }
}
```

`validate_sender_info` must then pass `request.sender().get().as_slice()` when constructing `SenderInfoContent`, ensuring the canister signature is cryptographically bound to the specific principal that submitted the request.

## Proof of Concept
1. User A (principal `P_A`) submits an ingress call to canister C with `sender_info = {info: b"age_verified=true", signer: II, sig: σ}`, where `σ = II.sign("\x0Eic-sender-info" || b"age_verified=true")`. The message is publicly observable.
2. Attacker B (principal `P_B`) constructs a new `HttpCanisterUpdate` with `sender: P_B` and the identical `sender_info = {info: b"age_verified=true", signer: II, sig: σ}` copied verbatim.
3. B signs the new `message_id` (which commits to `P_B` and the replayed `sender_info`) with B's own key. The envelope signature is valid.
4. `validate_sender_info` → `verify_sender_info_canister_sig` accepts: `σ` is a valid canister signature over `SenderInfoContent(b"age_verified=true")`, and the canister-ID check passes.
5. Canister C executes with `ic0.msg_caller() = P_B` and `ic0.msg_caller_info_data() = b"age_verified=true"` — the attestation originally issued for user A is now accepted for attacker B.

A deterministic integration test can confirm this by: (a) generating a canister signature over `SenderInfoContent(info_bytes)` for principal `P_A`, (b) constructing a second `HttpRequest` with sender `P_B` and the same `{info, signer, sig}`, and (c) asserting that `validate_request` returns `Ok(())`.
