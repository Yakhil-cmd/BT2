### Title
`set_commission()` Does Not Settle Rewards Before Changing Commission, Allowing Retroactive Commission Increase - (File: `src/staking/staking.cairo`)

### Summary

`set_commission()` in the Staking contract changes the commission rate used during reward calculation without first settling accrued rewards. In the V2 (attestation-based) reward path, rewards for an entire epoch are calculated in a single call using the **current** commission at attestation time. A staker holding an active `CommissionCommitment` can increase their commission to `max_commission` immediately before attesting, causing the higher commission to be applied retroactively to the entire epoch's delegator rewards — stealing yield that should have accrued to delegators under the lower commission.

---

### Finding Description

`set_commission()` at `src/staking/staking.cairo:725` performs no reward settlement before writing the new commission to storage:

```cairo
fn set_commission(ref self: ContractState, commission: Commission) {
    self.general_prerequisites();
    // ... validation only ...
    let staker_pool_info = self.staker_pool_info.entry(staker_address);
    if let Option::Some(old_commission) = staker_pool_info.commission.read() {
        self.update_commission(:staker_address, :staker_pool_info, :old_commission, :commission);
    } else { ... }
}
``` [1](#0-0) 

`update_commission()` immediately writes the new value to storage with no reward checkpoint:

```cairo
// Update commission in storage.
staker_pool_info.commission.write(Option::Some(commission));
``` [2](#0-1) 

In the V2 reward path, `update_rewards_from_attestation_contract` calculates **full epoch rewards** in one shot and reads the commission at that moment:

```cairo

    .calculate_current_epoch_rewards();
// ...

``` [3](#0-2) 

Inside `_update_rewards` → `calculate_staker_pools_rewards`, the commission is read live from storage and applied to the entire epoch's pool rewards:

```cairo

    rewards_including_commission: pool_rewards_including_commission, :commission,

A `CommissionCommitment` explicitly permits a staker to **increase** commission up to `max_commission`:

```cairo

    assert!(commission <= commission_commitment.max_commission, ...);
    assert!(commission != old_commission, ...);
``` [5](#0-4) 

---

### Impact Explanation

**High — Theft of unclaimed yield from delegators.**

Timeline:
- **T1** (epoch start): commission = C1 = 10%
- **T2** (late in epoch): staker calls `set_commission(50%)` using an active commitment with `max_commission = 50%`
- **T3** (attestation): `update_rewards_from_attestation_contract` is called; the entire epoch's pool rewards are split using 50%

Delegators lose `(C2 − C1) × epoch_pool_rewards / COMMISSION_DENOMINATOR` in yield that should have accrued to them under C1 for the period T1→T2. The staker captures this as excess commission. For a large pool, this can be a material amount of STRK.

---

### Likelihood Explanation

**Medium-High.** The attack requires:
1. The staker to have previously called `set_commission_commitment` with a `max_commission` above their current commission — a normal, permissionless operation.
2. The staker to call `set_commission` with the higher value just before attesting in the same epoch — also a permissionless, self-controlled action.

No external cooperation, leaked keys, or bridge compromise is needed. The staker controls both the commission change and the attestation timing. The V2 reward path remains active until `set_consensus_rewards_first_epoch` is triggered, so this is exploitable on any deployment still in the pre-consensus phase.

---

### Recommendation

Add a reward settlement call inside `set_commission()` before writing the new commission, analogous to how `pause()` and `unpause()` settle state before making changes. Specifically, if the staker has an active pool and the system is in the pre-consensus (V2) phase, the pending epoch rewards should be finalized at the old commission rate before the new rate takes effect.

---

### Proof of Concept

1. Deploy the staking system in V2 (pre-consensus) mode.
2. Staker stakes with commission = 10% and opens a STRK pool.
3. Delegator enters the pool.
4. Staker calls `set_commission_commitment(max_commission: 5000, expiration_epoch: current_epoch + 1)`.
5. Advance to near the end of epoch E (just before the attestation window).
6. Staker calls `set_commission(5000)` — commission jumps from 10% to 50%.
7. Staker attests; `update_rewards_from_attestation_contract` is called.
8. Observe: the entire epoch E's pool rewards are split at 50% commission, not 10%.
9. Delegator's `unclaimed_rewards` is ~44% lower than it should be for the full epoch; the staker's `unclaimed_rewards_own` is correspondingly inflated.

The root cause is that `set_commission` at `src/staking/staking.cairo:725` writes the new commission directly to storage without first triggering `update_rewards_from_attestation_contract`, and `calculate_staker_pools_rewards` at line 1964 reads the live commission value, applying it to the full epoch retroactively.
