### Title
Missing Caller Access Control on `update_rewards` Allows Anyone to Front-Run Sequencer and Permanently Suppress Per-Block Rewards - (File: `src/staking/staking.cairo`)

---

### Summary

`IStakingRewardsManager::update_rewards` is specified to be callable only by the Starkware sequencer, but the implementation contains **no caller check**. Any unprivileged address can call it first in a given block with `disable_rewards: true`, which writes `last_reward_block` to the current block and returns early without distributing rewards. The sequencer's subsequent call with `disable_rewards: false` then reverts with `REWARDS_ALREADY_UPDATED`. The per-block rewards for that block are permanently lost for all stakers and pool members.

---

### Finding Description

The protocol specification at `docs/spec.md:1644-1645` explicitly states:

```
#### access control
Only starkware sequencer.
```

However, the implementation of `update_rewards` in `src/staking/staking.cairo` at lines 1449–1507 contains no such check:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();   // only checks pause state
    let current_block_number = starknet::get_block_number();
    assert!(
        current_block_number > self.last_reward_block.read(),
        "{}",
        Error::REWARDS_ALREADY_UPDATED,
    );
    ...
    // Update last block rewards — written BEFORE the disable_rewards check
    self.last_reward_block.write(current_block_number);

    if disable_rewards || self.is_pre_consensus() {
        return;   // exits without distributing any rewards
    }
    ...
    self._update_rewards(...);
}
```

The critical ordering flaw is that `last_reward_block` is committed to storage at line 1485 **before** the `disable_rewards` branch at line 1487. Once any caller writes `last_reward_block = current_block_number` with `disable_rewards: true`, the gate check at lines 1454–1458 will reject every subsequent call in the same block, including the sequencer's legitimate call with `disable_rewards: false`.

The `general_prerequisites()` function only enforces the contract-paused guard; it does not restrict the caller to the sequencer.

---

### Impact Explanation

- Per-block rewards for the targeted staker (and all pool members delegating to that staker) are **permanently lost** for every block the attacker front-runs. The `last_reward_block` slot is a single global value; once set for block `N`, no call can ever distribute rewards for block `N` again.
- If the attacker repeats this every block, stakers and delegators accrue **zero** consensus-era rewards indefinitely.
- This constitutes **permanent freezing of unclaimed yield** (High severity) and, if sustained, effective **theft of unclaimed yield** because the inflation tokens that would have been minted and credited are never allocated.

---

### Likelihood Explanation

- The function is publicly callable with no role restriction.
- The attacker needs only to submit a transaction with `disable_rewards: true` before the sequencer's transaction in the same block. On Starknet, transaction ordering within a block is controlled by the sequencer, but the function is exposed as a public entry point callable by any L2 account.
- No capital, no privileged key, and no special setup is required. The attack cost is a single transaction fee per block.
- The attacker has no profit motive but causes direct, measurable yield loss to every staker and delegator in the protocol.

---

### Recommendation

Add a caller check at the top of `update_rewards` that restricts execution to the designated sequencer address (stored in contract storage or roles), mirroring the pattern used by `update_rewards_from_attestation_contract`, which correctly enforces `CALLER_IS_NOT_ATTESTATION_CONTRACT`. For example:

```cairo
fn update_rewards(...) {
    self.general_prerequisites();
    assert!(
        get_caller_address() == self.sequencer_address.read(),
        "{}",
        Error::CALLER_IS_NOT_SEQUENCER,
    );
    ...
}
```

Alternatively, move the `last_reward_block.write(current_block_number)` call to **after** the `disable_rewards` branch so that a call with `disable_rewards: true` does not consume the block's reward slot.

---

### Proof of Concept

1. Consensus rewards are active (`!self.is_pre_consensus()`).
2. Block `N` begins. The sequencer prepares a call: `update_rewards(staker_address, disable_rewards: false)`.
3. Attacker observes the pending transaction in the mempool and submits `update_rewards(staker_address, disable_rewards: true)` with higher priority (or simply races it).
4. Attacker's transaction executes first:
   - `last_reward_block` is written to block `N` at line 1485. [1](#0-0) 
   - `disable_rewards == true` → function returns at line 1487–1489 without calling `_update_rewards`. [2](#0-1) 
5. Sequencer's transaction executes next:
   - `current_block_number > self.last_reward_block.read()` is **false** (both equal block `N`).
   - Reverts with `REWARDS_ALREADY_UPDATED` at lines 1454–1458. [3](#0-2) 
6. Block `N` rewards are permanently lost for `staker_address` and all its pool members.
7. Repeat for every block.

The spec-vs-code mismatch is confirmed: the spec mandates sequencer-only access [4](#0-3)  while the implementation exposes the function to any caller with no restriction. [5](#0-4)
