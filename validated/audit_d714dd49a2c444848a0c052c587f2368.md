### Title
Append-Only `stakers` Vector Enables Cheap Griefing to Cause Unbounded Gas Consumption in `get_stakers` — (File: src/staking/staking.cairo)

---

### Summary

The `stakers` storage vector in the Staking contract is append-only: every call to `stake()` pushes a new address, and stakers are **never removed** when they unstake. The `get_stakers` function iterates over the entire vector unconditionally. An attacker can cheaply inflate this vector by registering many stakers at `min_stake`, then unstaking to recover principal (paying only gas). The ever-growing dead entries force `get_stakers` to perform unbounded work, eventually making the function uncallable and breaking the consensus layer's ability to retrieve the active staker set.

---

### Finding Description

`src/staking/staking.cairo` declares the stakers vector as a `Vec<ContractAddress>`:

```
stakers: Vec<ContractAddress>,
/// **Note**: Stakers are not removed from this vector when they unstake.
```

Every successful `stake()` call appends to it:

```cairo
// Add staker address to the stakers vector.
self.stakers.push(staker_address);
```

`get_stakers` iterates over the **full range** of this vector on every call:

```cairo
for staker_address_ptr in self.stakers.into_iter_full_range() {
    let staker_address = staker_address_ptr.read();
    if !self.is_staker_active(:staker_address, :epoch_id) {
        continue;   // dead entries are skipped but still read
    }
    // … per-staker storage reads for staking power, public key, peer id
}
```

Because `unstake_action` deletes the staker's `staker_info` record but **never removes the address from `self.stakers`**, every exited staker permanently adds one storage-read iteration to every future `get_stakers` call.

The attack mirrors the external report's pattern exactly:

| External report (Paraspace) | This codebase (Starknet Staking) |
|---|---|
| Uniswap V3 NFTs with 0 liquidity are cheap to mint | Stakers at `min_stake` are cheap to register |
| Attacker supplies NFTs to victim's account | Attacker registers stakers under their own addresses |
| Victim's balance limit is filled | `stakers` vector is inflated with dead entries |
| `calculateUserAccountData` gas cost grows | `get_stakers` gas cost grows |
| Victim cannot add collateral | Consensus layer cannot retrieve staker set |
| Attacker recovers tokens after attack | Attacker recovers stake after `unstake_action` |

The only cost to the attacker is gas for `stake` + `unstake_intent` + `unstake_action` per address — cheap on Starknet L2.

---

### Impact Explanation

`get_stakers` is the canonical on-chain view consumed by the Starknet consensus layer to build the validator committee for each epoch. If the function exceeds the gas budget of the calling node or sequencer, the consensus layer cannot obtain the staker set, stalling block production and attestation reward distribution. This constitutes **griefing with no profit motive but damage to the protocol** and **unbounded gas consumption**, both within the allowed Medium impact scope.

---

### Likelihood Explanation

- `stake()` is permissionless; any address with `min_stake` STRK can register.
- After `unstake_action`, the full principal is returned; the attacker's net cost is only L2 gas fees.
- Starknet L2 gas is cheap, making large-scale vector inflation economically viable.
- The vector is permanent: there is no governance mechanism to prune it.
- The attack is non-reversible: once entries are in the vector they cannot be removed.

---

### Recommendation

1. **Track active count separately**: maintain a counter of currently active stakers and skip the vector iteration when the count is zero.
2. **Swap-and-pop on exit**: replace the exiting staker's slot with the last element and pop, keeping the vector compact (requires an index map).
3. **Paginated iteration**: expose `get_stakers` with offset/limit parameters so callers can bound per-call gas.
4. **Minimum stake deterrent**: raise `min_stake` to make the attack economically costly, though this does not eliminate the root cause.
