### Title
Missing u16 Upper-Bound Check on Threshold Causes Deferred Protocol Failure — (`src/frost/redjubjub/sign.rs`)

### Summary
`sign()` accepts any `threshold` that satisfies `threshold <= participants.len()`, but `construct_key_package` silently fails at protocol-execution time for any `threshold > 65535` because the FROST `KeyPackage` requires a `u16`. The error is never surfaced at initialization; the caller receives a valid `Protocol` object that immediately returns `ProtocolError::Other` when driven.

### Finding Description

`ReconstructionLowerBound` is a plain `usize` newtype with no upper-bound constraint: [1](#0-0) 

`assert_sign_inputs` — the sole gate in `sign()` — only checks that `threshold <= participants.len()`. It never checks `threshold <= u16::MAX`: [2](#0-1) 

`sign()` calls `assert_sign_inputs`, gets `Ok`, and returns a live `Protocol` object: [3](#0-2) 

When the protocol future executes, `construct_key_package` attempts the narrowing conversion: [4](#0-3) 

For any `threshold > 65535`, `u16::try_from` fails and the function returns `ProtocolError::Other("threshold cannot be converted to u16")`. Every participant in the session hits this error; no signature is ever produced. The identical defect exists in `src/frost/eddsa/sign.rs`: [5](#0-4) 

### Impact Explanation
Any caller that constructs a session with `threshold > 65535` and `participants.len() >= threshold` will receive a successfully initialized `Protocol` that permanently fails at execution. All honest participants are denied signing for the lifetime of that session. There is no recovery path — the protocol must be re-initialized with a corrected threshold.

### Likelihood Explanation
Low-to-medium. A participant set of 65 536+ is impractical for most FROST deployments, but the library places no documented upper bound on threshold, and the type system (`usize`) actively invites large values. An integrator building a large-scale system (e.g., a blockchain validator set) could hit this without any compile-time or initialization-time warning.

### Recommendation
Add a `u16` upper-bound check inside `assert_sign_inputs` (or inside `sign()` before `assert_sign_inputs` is called):

```rust
if threshold.value() > u16::MAX as usize {
    return Err(InitializationError::ThresholdTooLarge {
        threshold: threshold.value(),
        max: u16::MAX as usize,
    });
}
```

This surfaces the error at initialization time with a clear `InitializationError`, consistent with every other input-validation failure in the library.
