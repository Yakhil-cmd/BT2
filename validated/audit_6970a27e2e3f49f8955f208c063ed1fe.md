### Title
Pool Admin Bypasses `maxAdminSpreadFeeE6` Cap via Uncapped `setPoolBinAdditionalFees` — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary
`MetricOmmPoolFactory.setPoolBinAdditionalFees()` allows a pool admin to set per-bin additional fees (`addFeeBuyE6`, `addFeeSellE6`) to any `uint16` value (max 65535 = 6.5535%) with zero cap validation, while the parallel `setPoolAdminFees()` path enforces `maxAdminSpreadFeeE6`. This is a direct admin-boundary break: the pool admin is supposed to be semi-trusted only within caps, but the per-bin fee path has no cap at all.

### Finding Description

The factory's fee cap system is built around `maxAdminSpreadFeeE6` (itself bounded by `HARD_MAX_SPREAD_FEE_E6 = 200_000`). `setPoolAdminFees` enforces this cap: [1](#0-0) 

But `setPoolBinAdditionalFees` passes the values straight through with no cap check: [2](#0-1) 

`MetricOmmPool.setBinAdditionalFees()` only validates the bin index range, then stores the values unconditionally: [3](#0-2) 

In the buy-token0 swap loop, the per-bin additional fee is added directly to the base fee with no clamping: [4](#0-3) 

The same pattern applies to the sell direction via `addFeeSellE6`: [5](#0-4) 

`BinState.addFeeBuyE6` and `addFeeSellE6` are `uint16`, so the maximum settable value is 65535, representing 6.5535% in E6 units: [6](#0-5) 

The factory hard cap `HARD_MAX_SPREAD_FEE_E6 = 200_000` (20%) governs the owner-level cap system but is never consulted for per-bin fees: [7](#0-6) 

### Impact Explanation

A pool admin calls `setPoolBinAdditionalFees(pool, 0, 65535, 65535)`. Any trader whose swap routes through bin 0 pays an additional 6.5535% fee on top of the base spread and notional fees. The trader receives fewer tokens than the oracle price implies. This is direct loss of swap output. The `maxAdminSpreadFeeE6` cap — which the protocol owner may have set to a low value (e.g., 1000 = 0.1%) to protect users — is completely bypassed via this path.

### Likelihood Explanation

The pool admin role is assigned at pool creation and is a distinct, semi-trusted party. The contest scope explicitly identifies `setBinAdditionalFees` as an admin-boundary path to audit. The call requires only `onlyPoolAdmin(pool)` — no timelock, no owner approval, no delay. A malicious or compromised pool admin can execute this immediately and silently.

### Recommendation

Add a cap check in `setPoolBinAdditionalFees` before forwarding to the pool, analogous to the check in `setPoolAdminFees`:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (addFeeBuyE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

Alternatively, introduce a dedicated `maxAdminBinAdditionalFeeE6` cap variable so the owner can set a tighter bound for per-bin fees independently.
