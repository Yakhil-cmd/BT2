### Title
Missing Caller Restriction on `update_rewards` Allows Any User to Permanently Deny Per-Block Rewards to Stakers - (File: src/staking/staking.cairo)

### Summary

The `update_rewards` function in the `StakingRewardsManager` implementation is documented in the spec as callable only by the Starkware sequencer, but the implementation contains no caller access control. Any unprivileged user can call `update_rewards` with `disable_rewards: true` for any active staker, consuming the single per-block reward slot and preventing legitimate reward distribution for that block. Repeated calls across blocks permanently deny yield to targeted stakers.

### Finding Description

The `update_rewards` function in `src/staking/staking.cairo` enforces only two preconditions: the contract must be unpaused (`general_prerequisites()`) and the current block number must be strictly greater than `last_reward_block` (the `REWARDS_ALREADY_UPDATED` guard). There is no check that `get_caller_address()` equals the authorized sequencer address. [1](#0-0) 

The function accepts a caller-supplied `disable_rewards: bool` flag. When `true`, the function still writes `current_block_number` to `last_reward_block` and returns early, consuming the slot without distributing any rewards: [2](#0-1) 

Because `last_reward_block` is a single global storage variable, once any caller invokes `update_rewards` in block N, no further call can succeed in block N for any staker: [3](#0-2) 

The protocol specification explicitly states the access control requirement that is absent from the implementation: [4](#0-3) 

### Impact Explanation

An attacker who calls `update_rewards(victim_staker_address, disable_rewards: true)` in every block:

1. Marks the block as processed (`last_reward_block = current_block`)
2. Prevents the legitimate sequencer from calling `update_rewards` for that block (reverts with `REWARDS_ALREADY_UPDATED`)
3. Causes the targeted staker (and their delegation pool) to receive zero block rewards for that block

Sustained across blocks, this permanently denies all consensus-phase per-block yield to the targeted staker and their delegators. The rewards are never generated and never enter `unclaimed_rewards_own`, so they cannot be recovered. This matches the allowed impact: **Medium — Griefing with no profit motive but damage to users or protocol** (continuous, unbounded denial of yield generation).

### Likelihood Explanation

The entry point is fully public — no token approval, no privileged role, no special state is required. Any EOA or contract can call `update_rewards` with an arbitrary `staker_address` and `disable_rewards: true`. The only cost to the attacker is gas per block (~3 s on Starknet). The attack is trivially automatable with a bot.

### Recommendation

Add an explicit caller check at the top of `update_rewards`, analogous to the guard used in `update_current_epoch_block_rewards` and `claim_rewards` in the reward supplier:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
    assert!(
        get_caller_address() == self.sequencer_address.read(),
        "{}",
        Error::CALLER_IS_NOT_SEQUENCER,
    );
    ...
}
```

The authorized sequencer address should be stored in contract storage and settable only by governance, mirroring the pattern used for `staking_contract` in `RewardSupplier`: [5](#0-4) 

### Proof of Concept

```
// Block N:
// 1. Attacker (any address) calls:
staking.update_rewards(staker_address: victim, disable_rewards: true);
// -> last_reward_block is set to N, no rewards distributed.

// 2. Sequencer attempts its legitimate call in the same block:
staking.update_rewards(staker_address: victim, disable_rewards: false);
// -> PANICS with REWARDS_ALREADY_UPDATED (current_block_number == last_reward_block).

// 3. Victim staker earns zero rewards for block N.
// 4. Attacker repeats every block -> victim earns zero rewards indefinitely.
```

The `disable_rewards` flag path that returns early without distributing rewards while still consuming the block slot is the root cause: [6](#0-5)
