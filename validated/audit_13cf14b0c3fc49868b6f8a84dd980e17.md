Audit Report

## Title
Unbounded `active_assembles` / `artifact_processor_tasks` Growth via `SLOT_TABLE_NO_LIMIT` — (`rs/p2p/consensus_manager/src/receiver.rs`)

## Summary
`SLOT_TABLE_NO_LIMIT` is defined as `usize::MAX` and is passed as the `slot_limit` parameter to the `ConsensusManagerReceiver` for consensus, certifier, DKG, iDKG, and HTTPS outcalls channels in production. This renders the sole per-peer admission guard inoperative, allowing a single Byzantine subnet peer to spawn an unbounded number of persistent Tokio tasks and grow `active_assembles` without bound, exhausting heap memory and the Tokio task scheduler on honest replicas.

## Finding Description
In `rs/replica/setup_ic_network/src/lib.rs`, `SLOT_TABLE_NO_LIMIT = usize::MAX` is passed to `abortable_broadcast_channel` for five of six artifact channels (consensus, certifier, DKG, iDKG, HTTPS outcalls). In `handle_slot_update_receive` (`rs/p2p/consensus_manager/src/receiver.rs`, lines 372–390), the only admission control for new slot entries is:

```rust
Entry::Vacant(empty_slot) if peer_slot_table_len < self.slot_limit => { ... }
Entry::Vacant(_) => { /* warn and drop */ }
```

When `self.slot_limit = usize::MAX`, the condition `peer_slot_table_len < usize::MAX` is always true in practice. Every new `(slot_number, artifact_id)` pair from a peer is unconditionally admitted.

For each admitted new artifact ID not already in `active_assembles` (lines 399–420), the code:
1. Inserts a new entry into `self.slot_table[peer_id]`
2. Inserts a new `watch::Sender` into `self.active_assembles`
3. Spawns a new Tokio task into `self.artifact_processor_tasks`

These tasks are long-lived. After the assembler returns `AssembleResult::Unwanted`, the task waits at `peer_rx.wait_for(|p| p.is_empty()).await` (line 521). The peer counter only empties when the peer overwrites the slot or leaves the topology. A Byzantine peer that never overwrites its slots keeps every task alive indefinitely. Furthermore, `handle_artifact_processor_joined` (lines 300–319) restarts the task if the peer counter is still non-empty when the task finishes, preventing cleanup.

The `all_peers_deleted_artifact` select branch (lines 530–535) only fires when the peer counter empties — which the Byzantine peer controls and can prevent.

## Impact Explanation
A single Byzantine subnet peer (below the BFT fault threshold) can grow `active_assembles` (a `HashMap`) and `artifact_processor_tasks` (a `JoinSet`) to an arbitrary size, exhausting heap memory and the Tokio task scheduler on the honest replica. This causes the honest replica to crash or become unresponsive, blocking its participation in consensus and violating subnet liveness. This matches the allowed impact: **High ($2,000–$10,000) — Application/platform-level DoS, crash, consensus blocking, or subnet availability impact not based on raw volumetric DDoS.**

## Likelihood Explanation
The attacker must be a compromised subnet node, authenticated via TLS — a realistic Byzantine adversary within the IC threat model. No threshold corruption is required; a single node suffices. The attack is mechanically simple: send N `SlotUpdate` messages each with a distinct `slot_id` and a distinct artifact `id`. The `update_handler` performs only protobuf decoding before forwarding to the event loop with no rate limiting or slot count check at the transport layer. The attack is repeatable and persistent — tasks remain alive until the replica crashes or the attacker's node is removed from the topology.

## Recommendation
1. **Remove `SLOT_TABLE_NO_LIMIT`**: Replace `usize::MAX` with a protocol-appropriate bound per artifact type (e.g., consensus pipeline depth for consensus, bounded by the maximum number of in-flight artifacts a correct peer can legitimately hold).
2. **Enforce the limit at the transport/handler layer**: Reject slot updates from a peer that has already filled its quota before they reach the event loop.
3. **Add a global cap** on `active_assembles.len()` and `artifact_processor_tasks.len()` as a defense-in-depth backstop.

## Proof of Concept
The existing test infrastructure in `receiver.rs` directly supports this. The `slot_table_limit_exceeded` test (lines 846–887) already demonstrates the guard works when `slot_limit` is finite. The vulnerability is precisely that `SLOT_TABLE_NO_LIMIT = usize::MAX` disables this guard in production. A minimal state-machine test (no network required):

```rust
let (mut mgr, _) = ReceiverManagerBuilder::new()
    .with_slot_limit(usize::MAX)  // SLOT_TABLE_NO_LIMIT
    .build();
let cancellation = CancellationToken::new();
for i in 0u64..100_000 {
    mgr.handle_slot_update_receive(
        SlotUpdate {
            slot_number: SlotNumber::from(i),
            commit_id: CommitId::from(i),
            update: Update::Id(i),
        },
        NODE_1,
        ConnId::from(1),
        cancellation.clone(),
    );
}
assert_eq!(mgr.active_assembles.len(), 100_000);
assert_eq!(mgr.artifact_processor_tasks.len(), 100_000);
```
