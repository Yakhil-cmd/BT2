### Title
ETH Permanently Locked at Precompile Address on `ft_transfer` Failure When `error_refund` Feature Is Disabled — (`engine-precompiles/src/native.rs`)

---

### Summary

When the `error_refund` compile-time feature is disabled (which is the **default** configuration), a call to the `exitToNear` precompile with flag `0x00` (base ETH path) fires an `ft_transfer` promise with **no callback**. If that promise fails on the connector side, the ETH deducted from the caller's EVM balance is permanently locked at the precompile address (`exit_to_near::ADDRESS`) with no refund mechanism and no corresponding NEP-141 credit — a direct insolvency condition.

---

### Finding Description

**Feature flag default state:**

`error_refund` is an optional, non-default feature in both `engine-precompiles/Cargo.toml` and `engine/Cargo.toml`:

```
[features]
default = ["std"]
...
error_refund = []
``` [1](#0-0) [2](#0-1) 

**Callback construction in `ExitToNear::run`:**

When `error_refund` is disabled, `callback_args.refund` is hardcoded to `None`:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,
    transfer_near: transfer_near_args,
};
``` [3](#0-2) 

For the base ETH path (flag=`0x00`, no `:unwrap` or Omni message), `exit_base_token_to_near` returns `transfer_near_args = None`: [4](#0-3) 

With both fields `None`, `callback_args == ExitToNearPrecompileCallbackArgs::default()` is `true`, so the promise is emitted as a bare `PromiseArgs::Create` — **no callback is attached**:

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // <-- no callback
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs { ... })
};
``` [5](#0-4) 

**What happens on `ft_transfer` failure:**

When `error_refund` IS enabled, `exit_to_near_precompile_callback` handles the failure branch by calling `engine::refund_on_error`, which transfers ETH from `exit_to_near::ADDRESS` back to the caller: [6](#0-5) [7](#0-6) 

When `error_refund` is disabled, no callback is scheduled, so `exit_to_near_precompile_callback` is never called. The ETH remains at `exit_to_near::ADDRESS` with no recovery path.

**The test suite explicitly confirms this behavior:**

```rust
// If the refund feature is not enabled, then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let expected_balance = Wei::new_u64(INITIAL_ETH_BALANCE - ETH_EXIT_AMOUNT);
``` [8](#0-7) 

---

### Impact Explanation

The invariant broken: **ETH deducted from the EVM caller's balance must either arrive at the NEAR recipient as NEP-141 tokens, or be returned to the caller.**

When `ft_transfer` fails with `error_refund` disabled:
- Caller's EVM ETH balance is permanently reduced.
- `exit_to_near::ADDRESS` holds the ETH with no owner and no withdrawal mechanism.
- The NEAR recipient receives zero NEP-141 tokens.
- Total EVM ETH supply decreases without a corresponding burn/bridge event.

This is **Insolvency**: ETH is removed from the EVM supply with no corresponding credit anywhere.

---

### Likelihood Explanation

- `error_refund` is **not** in the default feature set, so any deployment built without explicitly enabling it is affected.
- Any unprivileged ETH holder can trigger this — no admin access required.
- `ft_transfer` can fail for legitimate reasons: connector NEP-141 balance drained, recipient account not registered with the NEP-141 contract, connector paused, etc.
- The test `test_exit_to_near_eth_refund` demonstrates the exact failure scenario (draining the Aurora connector balance) is trivially reproducible. [9](#0-8) 

---

### Recommendation

1. **Enable `error_refund` by default** in the `engine` and `engine-precompiles` crates, or remove the feature flag entirely and always attach the callback.
2. Alternatively, when `error_refund` is disabled, always attach a minimal callback that at minimum handles the failure case for the base ETH path by transferring ETH back from the precompile address to the caller.
3. The `ExitToNearPrecompileCallbackArgs` struct already supports `refund: None` with `transfer_near: None` as a no-op callback — the guard at line 470 that skips the callback when both are `None` should be removed or conditioned differently so the callback is always attached for the ETH base path.

---

### Proof of Concept

The existing test `test_exit_to_near_eth_refund` in `engine-tests/src/tests/erc20_connector.rs` is a complete proof of concept when compiled **without** the `error_refund` feature:

1. Deploy Aurora without `--features error_refund`.
2. Drain the Aurora connector NEP-141 balance via `ft_transfer` to a dummy account.
3. Call `withdrawEthToNear(bytes)` on the tester contract with `ETH_EXIT_AMOUNT` value.
4. Assert: `nep_141_balance_of(recipient) == 0` and `eth_balance_of(caller) == INITIAL_ETH_BALANCE - ETH_EXIT_AMOUNT`.

The test already asserts this exact outcome at lines 766–780, confirming ETH is permanently lost with no refund. [10](#0-9)
