Audit Report

## Title
Rate-Limit Bucket Charged to Node's Registered Operator Instead of Caller in `do_remove_node_directly_` — (`rs/registry/canister/src/mutations/node_management/do_remove_node_directly.rs`)

## Summary
In `do_remove_node_directly_`, the rate-limit reservation is keyed on `node_operator_id` (the node's registered operator), not `caller_id` (the actual ingress caller). Because `make_remove_or_replace_node_mutations` explicitly permits a different operator (operator B) to remove nodes belonging to operator A when they share the same DC and node provider, operator B can exhaust operator A's per-operator rate-limit bucket without consuming any of its own capacity, locking operator A out of all rate-limited registry operations for up to 7 days.

## Finding Description
In `do_remove_node_directly_` (lines 52–54), the rate-limit identity is taken from the node record:

```rust
let node_operator_id = get_node_operator_id_for_node(self, payload.node_id)?;
let reservation =
    self.try_reserve_capacity_for_node_operator_operation(now, node_operator_id, 1)?;
```

`caller_id` is passed only to `make_remove_or_replace_node_mutations`, which at lines 89–119 applies the DC + node-provider fallback: if `caller_id != node_operator_id`, it asserts that both operators share the same `dc_id` and the same node provider. When that assertion passes, the removal proceeds — but the rate-limit charge has already been applied to `node_operator_id` (operator A), not to `caller_id` (operator B).

`try_reserve_capacity_for_node_operator_operation` (rate_limits.rs lines 114–117) keys the reservation on `node_operator_key(node_operator_id)`, confirming the charge goes to operator A's bucket. The per-operator bucket is capped at `NODE_OPERATOR_MAX_SPIKE = 20 × 7 = 140` (rate_limits.rs lines 24–25). Once exhausted, any subsequent call keyed to operator A — including operator A's own `do_add_node` or `do_remove_node_directly` — returns `"Rate Limit Capacity exceeded. Please wait and try again later."` for up to 7 days.

The shared node-provider bucket is also depleted by each such call (rate_limits.rs lines 119–122), potentially cascading to other operators under the same provider.

## Impact Explanation
Operator B, a registered node operator in the same DC under the same node provider as operator A, can make up to 140 ordinary ingress update calls to the registry canister, each removing one of operator A's nodes. Each call passes authorization and charges operator A's rate-limit bucket. After 140 calls, operator A is locked out of all rate-limited registry node-management operations for up to 7 days. The shared node-provider bucket is simultaneously drained, potentially affecting other operators under the same provider. This constitutes a concrete, application-level DoS against a legitimate registry participant with no governance vote, admin key, or threshold corruption required. Impact class: **High — Application/platform-level DoS with concrete user and protocol harm**.

## Likelihood Explanation
The precondition — two node operators sharing the same DC and node provider — is an explicitly supported and documented redeployment scenario (lines 83–88 of `do_remove_node_directly.rs`). No special privileges beyond being a registered node operator in the same DC/provider are required. The attack is a sequence of ordinary ingress update calls; it is repeatable and deterministic.

## Recommendation
Charge the rate limit against `caller_id` (the actual caller), not `node_operator_id` (the node's registered operator). In `do_remove_node_directly_`, replace:

```rust
let node_operator_id = get_node_operator_id_for_node(self, payload.node_id)?;
let reservation =
    self.try_reserve_capacity_for_node_operator_operation(now, node_operator_id, 1)?;
```

with:

```rust
let reservation =
    self.try_reserve_capacity_for_node_operator_operation(now, caller_id, 1)?;
```

This ensures the cost of the operation is always borne by the entity initiating it, regardless of which operator owns the node being removed.

## Proof of Concept
The existing test `should_succeed_remove_node_compare_dc_and_node_provider` (lines 483–554 of `do_remove_node_directly.rs`) already demonstrates that operator B (same DC, same provider) can successfully remove operator A's node. Extending this test to exhaust the rate limit confirms the bug:

```rust
// Setup: operator1 and operator2 in same DC "dc1" under same provider (PrincipalId 3000)
// Add 140 nodes owned by operator1
// operator2 calls do_remove_node_directly_ on each of operator1's nodes
for node_id in operator1_nodes {
    registry.do_remove_node_directly_(
        RemoveNodeDirectlyPayload { node_id },
        operator2_id,
        now,
    ).unwrap();
}
// operator1's bucket is now 0; operator2's bucket is untouched
let result = registry.do_remove_node_directly_(
    RemoveNodeDirectlyPayload { node_id: some_node_of_operator1 },
    operator1_id,
    now,
);
assert_eq!(result.unwrap_err(), "Rate Limit Capacity exceeded. Please wait and try again later.");
let b_capacity = registry.get_available_node_operator_op_capacity(operator2_id, now);
assert_eq!(b_capacity, 140); // operator2 untouched
```
