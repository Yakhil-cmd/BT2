### Title
ETH Permanently Locked in `RewardSupplier` When `tick()` Called With No Pending Mint Requests - (File: `L1/starkware/solidity/stake/RewardSupplier.sol`)

### Summary
The `tick()` function in `RewardSupplier.sol` is `payable` and accepts ETH as message fees, but contains no withdrawal mechanism. When called with `msg.value > 0` while no L2 mint requests are pending (`amountToMint == 0`), the entire ETH value is permanently locked in the contract with no recovery path.

### Finding Description
`tick()` is declared `external payable` and is callable by any address without access control. Its logic only forwards ETH when `amountToMint > 0`:

```solidity
function tick() external payable {
    (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

    if (amountToMint > 0) {
        // ...
        uint256 msgFee = msg.value / 2;
        bridge().depositWithMessage{value: msgFee}(...);
        msgFee = msg.value - msgFee;
        messagingContract().sendMessageToL2{value: msgFee}(...);
    }
    // No else branch: ETH is silently retained when amountToMint == 0
}
```

When `amountToMint == 0`, the entire `if` block is skipped and `msg.value` is never forwarded or refunded. The contract has no `receive()`, `fallback()`, or withdrawal function, so the ETH is permanently irrecoverable.

A secondary case exists even when `amountToMint > 0`: if the caller sends more ETH than the bridge and messaging contract actually consume as fees, the surplus is also permanently stuck, since the contract forwards exactly `msg.value` split in two with no refund of any remainder. [1](#0-0) 

### Impact Explanation
Any ETH sent to `tick()` when no mint requests are pending is permanently frozen in the `RewardSupplier` contract. There is no admin withdrawal, no `selfdestruct`, and no sweep function anywhere in the contract or its visible inheritance chain. This constitutes **permanent freezing of funds** (High severity per the allowed impact scope). [2](#0-1) 

### Likelihood Explanation
`tick()` has no access control — any address can call it. Callers are expected to supply ETH to cover Starknet messaging fees. The window where `amountToMint == 0` is common: it occurs whenever the L2 reward supplier has not yet emitted new mint-request messages, or between epochs. A caller who sends ETH during such a window (e.g., a keeper bot that always sends a fee buffer) permanently loses that ETH. This is a realistic, recurring scenario. [3](#0-2) 

### Recommendation
1. Add a refund for unused ETH at the end of `tick()`:
   ```solidity
   if (address(this).balance > 0) {
       payable(msg.sender).transfer(address(this).balance);
   }
   ```
2. Alternatively, make `tick()` non-payable and require callers to pre-fund the contract via a separate, access-controlled deposit, or compute the exact required fee before the call and revert if `msg.value` does not match.
3. At minimum, add a privileged `withdrawETH(address recipient)` function so that accidentally locked ETH can be recovered.

### Proof of Concept
1. No L2 mint-request messages are pending (normal inter-epoch state: `messagingContract().l2ToL1Messages(msgHash) == 0`).
2. Caller invokes `tick{value: 1 ether}()`.
3. `requiredMinting()` returns `(0, 0)`.
4. The `if (amountToMint > 0)` block is skipped entirely.
5. `1 ether` is now held by `RewardSupplier` with no mechanism to retrieve it.
6. Confirm: `address(rewardSupplier).balance == 1 ether` and no function exists to withdraw it. [1](#0-0)

    function tick() external payable {
        // Check if minting is required, and how much.
        (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

        if (amountToMint > 0) {
            // Prepare the L2->L1 mintRequest message for consumption.
            uint256[] memory messagePayload = new uint256[](1);
            messagePayload[0] = TOKENS_PER_MINT_REQUEST;

            // Consume the mintRequest messages.
            for (uint256 i = 0; i < numMsgsToConsume; i++) {
                messagingContract().consumeMessageFromL2(mintRequestSource(), messagePayload);
            }

            // Reuest minting of the requested amount from the mint manager.
            mintManager().mintRequest(token(), amountToMint);

            // Deposit the minted amount onto the bridge to the credit of `mintDestination`.
            uint256 msgFee = msg.value / 2;
            bridge().depositWithMessage{value: msgFee}(
                token(),
                amountToMint,
                mintDestination(),
                new uint256[](0)
            );
            emit ConsumedL2MintRequests(numMsgsToConsume, amountToMint);

            // Send a totalSupply update to L2MintCurve.
            msgFee = msg.value - msgFee;
            messagePayload[0] = IERC20(token()).totalSupply();
            messagingContract().sendMessageToL2{value: msgFee}(
                mintingCurve(),
                UPDATE_TOTAL_SUPPLY_SELECTOR,
                messagePayload
            );
        }
    }
