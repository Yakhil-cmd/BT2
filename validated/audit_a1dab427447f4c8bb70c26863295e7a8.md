### Title
`RerandomizedPresignOutput` Derives `Clone`, Enabling Nonce Reuse and Aggregate Private Key Recovery - (File: `src/ecdsa/ot_based_ecdsa/mod.rs`, `src/ecdsa/robust_ecdsa/mod.rs`)

---

### Summary

Both OT-based and robust ECDSA implementations derive `Clone` on `RerandomizedPresignOutput`, the struct that carries secret nonce material into the signing phase. The `sign()` function takes this struct by value (move semantics) to signal single-use intent, but the `Clone` impl allows any caller — including a malicious coordinator — to duplicate the struct before passing it to `sign()`, then use the clone to sign a second message under the same nonce. Two ECDSA signatures sharing the same nonce `r` over different messages allow full recovery of the aggregate private key.

---

### Finding Description

The `sign()` function in both ECDSA variants accepts `presignature: RerandomizedPresignOutput` by value (moved), which is the library's mechanism for enforcing single-use semantics via Rust's ownership system:

```rust
// src/ecdsa/ot_based_ecdsa/sign.rs
pub fn sign(
    ...
    presignature: RerandomizedPresignOutput,   // moved — signals "consume once"
    msg_hash: Scalar,
) -> Result<impl Protocol<Output = SignatureOption>, InitializationError>
``` [1](#0-0) 

However, `RerandomizedPresignOutput` is declared with `#[derive(Clone)]` in both implementations:

```rust
// src/ecdsa/ot_based_ecdsa/mod.rs
#[derive(Debug, Clone, Serialize, Deserialize, ZeroizeOnDrop)]
pub struct RerandomizedPresignOutput { ... }
``` [2](#0-1) 

```rust
// src/ecdsa/robust_ecdsa/mod.rs
#[derive(Debug, Clone, Serialize, Deserialize, ZeroizeOnDrop)]
pub struct RerandomizedPresignOutput { ... }
``` [3](#0-2) 

A second contributing factor is that `rerandomize_presign()` takes `&PresignOutput` by shared reference (not consuming it), so the same `PresignOutput` can be rerandomized multiple times with the same tweak, producing identical `RerandomizedPresignOutput` values:

```rust
pub fn rerandomize_presign(
    presignature: &PresignOutput,   // not consumed
    args: &RerandomizationArguments,
) -> Result<Self, ProtocolError>
``` [4](#0-3) [5](#0-4) 

The library's own documentation acknowledges the danger but relies entirely on the caller to enforce it:

> "it's crucial that a presignature is never reused" [6](#0-5) [7](#0-6) 

The `Clone` derive directly contradicts and undermines the move-semantics protection the library itself establishes.

---

### Impact Explanation

**Critical — Extraction/reconstruction of aggregate secret material.**

In ECDSA, two signatures `(r, s₁)` and `(r, s₂)` produced over different message hashes `h₁ ≠ h₂` with the same nonce `k` satisfy:

```
s₁ = h₁·k + r·x
s₂ = h₂·k + r·x
⟹ k  = (s₁ − s₂) / (h₁ − h₂)
⟹ x  = (s₁ − h₁·k) / r
```

In the threshold setting, the combined `k` and `x` recovered are the **aggregate** nonce and **aggregate** private key, not just one participant's share. A malicious coordinator who obtains two valid threshold signatures over different messages using the same presignature can reconstruct the full signing key offline, without any further interaction with honest participants.

---

### Likelihood Explanation

**High.** The attack is reachable through two independent paths:

1. **Malicious coordinator**: The coordinator receives `RerandomizedPresignOutput` from the rerandomization step, calls `.clone()` before passing it to the first `sign()` invocation, then uses the clone to initiate a second signing session for a different message. Honest participants cannot detect the reuse because they only see their own share of the presignature.

2. **Careless caller**: A library integrator implementing retry logic, logging, or serialization may clone `RerandomizedPresignOutput` without understanding that doing so defeats the single-use guarantee. The `Clone` derive makes this trivially easy and gives no warning.

---

### Recommendation

Remove `Clone` from `RerandomizedPresignOutput` in both implementations. This makes nonce reuse a compile-time error for any caller who attempts it:

```diff
- #[derive(Debug, Clone, Serialize, Deserialize, ZeroizeOnDrop)]
+ #[derive(Debug, Serialize, Deserialize, ZeroizeOnDrop)]
  pub struct RerandomizedPresignOutput { ... }
```

Apply this change to both:
- `src/ecdsa/ot_based_ecdsa/mod.rs`
- `src/ecdsa/robust_ecdsa/mod.rs`

Additionally, consider consuming `PresignOutput` in `rerandomize_presign()` (take by value) to prevent the same presignature from being rerandomized multiple times with the same tweak:

```diff
- pub fn rerandomize_presign(presignature: &PresignOutput, ...) -> ...
+ pub fn rerandomize_presign(presignature: PresignOutput, ...) -> ...
```

### Proof of Concept

```rust
// Coordinator-side attack using OT-based ECDSA
let presign_output: PresignOutput = run_presign_protocol(...);

let args = RerandomizationArguments { tweak, big_r: presign_output.big_r };
let rerandomized = RerandomizedPresignOutput::rerandomize_presign(&presign_output, &args)?;

// Clone before consuming — bypasses move-semantics protection
let rerandomized_clone = rerandomized.clone();

// Sign message 1 — consumes `rerandomized`
let _sig1 = sign(&participants, coordinator, threshold, me, pubkey, rerandomized, msg_hash_1)?;

// Sign message 2 with the SAME nonce — consumes the clone
let _sig2 = sign(&participants, coordinator, threshold, me, pubkey, rerandomized_clone, msg_hash_2)?;

// Both signatures share the same `r` (same big_r → same nonce k).
// Coordinator recovers aggregate private key x offline:
//   k = (s1 - s2) / (msg_hash_1 - msg_hash_2)
//   x = (s1 - msg_hash_1 * k) / r
```
