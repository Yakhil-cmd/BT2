### Title
Gateway Proof-Facts Block-Number/State Mismatch Causes Valid Client-Side-Proving Transactions to Be Rejected - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

`run_validate_entry_point` advances the block number to `latest + 1` for the validation `BlockContext`, but the underlying state still reflects the committed state at block `latest`. `validate_proof_facts` uses `block_context.block_info.block_number` (i.e. `latest + 1`) to compute the maximum allowed proof-block number, admitting proof facts that reference block `latest + 1 − STORED_BLOCK_HASH_BUFFER`. However, that block's hash has not yet been written to the block-hash contract in the state (it is only written by `pre_process_block` when block `latest + 1` is actually executed). The hash lookup therefore returns `Felt::ZERO`, and the non-zero hash supplied in the proof facts triggers a "Block hash mismatch" error. A valid Invoke V3 transaction is rejected at the gateway even though it would succeed during actual block execution.

### Finding Description

In `run_validate_entry_point`:

```rust
let mut block_info = self.gateway_fixed_block_state_reader.get_block_info().await?;
block_info.block_number = block_info.block_number.unchecked_next();   // N → N+1
let block_context = BlockContext::new(block_info, ...);
// state still reflects committed state at block N
let state = CachedState::new(state_reader_and_contract_manager);
let mut blockifier_validator = StatefulValidator::create(state, block_context);
blockifier_validator.validate(account_tx)
``` [1](#0-0) 

For an Invoke V3 transaction carrying proof facts, `StatefulValidator::validate` calls `perform_pre_validation_stage`, which calls `validate_proof_facts`:

```rust
Self::validate_proof_block_number(
    proof_block_number,
    block_context.block_info.block_number,   // N+1
)?;
Self::validate_proof_block_hash(proof_block_hash, proof_block_number, os_constants, state)?;
``` [2](#0-1) 

`validate_proof_block_number` computes:

```rust
let max_allowed = current_block_number.0.checked_sub(STORED_BLOCK_HASH_BUFFER)...;
// max_allowed = (N+1) - 10 = N - 9
``` [3](#0-2) 

A proof referencing block `N − 9` passes this check. `validate_proof_block_hash` then reads:

```rust
let stored_block_hash = state.get_storage_at(
    block_hash_contract_address,
    StorageKey::from(proof_block_number)   // N-9
)?;
if stored_block_hash != proof_block_hash { return Err(...) }
``` [4](#0-3) 

The committed state at block `N` only contains hashes for blocks `0 … N − STORED_BLOCK_HASH_BUFFER = N − 10`. Block `N − 9` is written to state by `pre_process_block` only when block `N + 1` is actually executed:

```rust
// pre_process_block writes hash of (next_block_number - STORED_BLOCK_HASH_BUFFER)
// i.e. hash of (N+1 - 10) = hash of (N-9)
``` [5](#0-4) 

Because the hash for block `N − 9` is not yet in state, `get_storage_at` returns `Felt::ZERO`. The non-zero hash in the proof facts does not match, and the gateway returns `ValidateFailure`. The transaction is rejected even though it would pass execution in block `N + 1`.

### Impact Explanation

Any unprivileged user who submits an Invoke V3 transaction with client-side proof facts referencing block `N − STORED_BLOCK_HASH_BUFFER + 1` (the most recent block whose hash is valid for the next block but not yet stored in state) receives a spurious `ValidateFailure` rejection from the gateway. The transaction is permanently blocked from entering the mempool via the gateway path, even though it is cryptographically and semantically valid and would execute successfully.

**Impact category:** High — Mempool/gateway admission rejects a valid transaction before sequencing.

### Likelihood Explanation

The window is narrow: only proof facts referencing exactly block `N − STORED_BLOCK_HASH_BUFFER + 1` are affected. Proof facts referencing any older block pass both checks. A user who constructs a proof against the most recently eligible block (a natural choice to minimize proof staleness) will reliably hit this rejection on every submission attempt, since the gateway always advances the block number by one.

### Recommendation

**Short term:** In `run_validate_entry_point`, do not advance the block number before running `validate_proof_facts`, or pass the un-incremented block number specifically to the proof-facts check so that the allowed range matches the hashes actually present in the committed state.

**Long term:** Add an integration test that submits a client-side-proving Invoke V3 transaction with proof facts referencing block `latest − STORED_BLOCK_HASH_BUFFER + 1` through the full gateway stateful-validation path and asserts it is accepted.

### Proof of Concept

Let `N` = latest committed block number, `STORED_BLOCK_HASH_BUFFER` = 10.

1. Obtain the real block hash of block `N − 9` from the node (it is a committed block).
2. Construct a valid `SnosProofFacts` with `block_number = N − 9` and `block_hash = <real hash of N−9>`.
3. Submit an Invoke V3 transaction carrying these proof facts to the gateway.
4. `run_validate_entry_point` sets `block_context.block_info.block_number = N + 1`.
5. `validate_proof_block_number` computes `max_allowed = N + 1 − 10 = N − 9`; the check passes.
6. `validate_proof_block_hash` reads `get_storage_at(block_hash_contract, N−9)` from state at block `N`; the value is `Felt::ZERO` (not yet written).
7. `Felt::ZERO ≠ <real hash>` → error "Block hash mismatch for block N−9".
8. Gateway returns `ValidateFailure`; the transaction is rejected.
9. The same transaction submitted directly to the batcher (bypassing the gateway) would succeed in block `N + 1` because `pre_process_block` writes the hash of `N − 9` before executing transactions.

pub fn pre_process_block(
    state: &mut dyn State,
    old_block_number_and_hash: Option<BlockHashAndNumber>,
    next_block_number: BlockNumber,
    os_constants: &OsConstants,
) -> StateResult<()> {
    let should_block_hash_be_provided =
        next_block_number >= BlockNumber(constants::STORED_BLOCK_HASH_BUFFER);
    if let Some(BlockHashAndNumber { number, hash }) = old_block_number_and_hash {
        let block_hash_contract_address =
            os_constants.os_contract_addresses.block_hash_contract_address();
        let block_number_as_storage_key = StorageKey::from(number.0);
        state.set_storage_at(block_hash_contract_address, block_number_as_storage_key, hash.0)?;
    } else if should_block_hash_be_provided {
        return Err(StateError::OldBlockHashNotProvided);
    }
