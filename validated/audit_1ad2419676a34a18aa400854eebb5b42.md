### Title
Missing Caller Restriction on `update_rewards` Enables Permanent Griefing of Consensus Reward Distribution — (`src/staking/staking.cairo`)

### Summary

`update_rewards` in the Staking contract carries no caller access-control check. Any unprivileged address can invoke it with `disable_rewards: true`, which advances the global `last_reward_block` checkpoint without distributing any rewards. Because the checkpoint is global and the function enforces `current_block_number > last_reward_block`, a single such call per block permanently consumes that block's reward slot for every staker, making the legitimate `update_rewards(staker, false)` call revert with `REWARDS_ALREADY_UPDATED` for the rest of that block.

### Finding Description

`update_rewards` is the consensus-phase reward entry-point. Its implementation:

```
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();                          // only checks pause
    let current_block_number = starknet::get_block_number();
    assert!(
        current_block_number > self.last_reward_block.read(),
        "{}",
        Error::REWARDS_ALREADY_UPDATED,
    );
    ...
    // Update last block rewards.
    self.last_reward_block.write(current_block_number);   // ← written unconditionally

    if disable_rewards || self.is_pre_consensus() {
        return;                                           // ← exits before distributing
    }
    ...
``` [1](#0-0) 

There is no `assert_caller_is_X` guard anywhere in this function. `general_prerequisites` only enforces the pause flag. [2](#0-1) 

`last_reward_block` is a single scalar storage slot (not a per-staker map), so writing it for one call blocks every other staker for the same block. [3](#0-2) 

Contrast this with `update_rewards_from_attestation_contract`, the pre-consensus equivalent, which does enforce a strict caller check:

```
assert!(
    get_caller_address() == self.attestation_contract.read(),
    "{}",
    Error::CALLER_IS_NOT_ATTESTATION_CONTRACT,
);
``` [4](#0-3) 

The spec confirms this asymmetry — `update_rewards_from_attestation_contract` is "Only attestation contract", while `update_rewards` carries no equivalent restriction. [5](#0-4) 

### Impact Explanation

An attacker who calls `update_rewards(any_active_staker, disable_rewards: true)` at the start of every block:

1. Advances `last_reward_block` to the current block number.
2. Returns without distributing any STRK or BTC rewards.
3. Forces every subsequent legitimate `update_rewards` call in that block to revert with `REWARDS_ALREADY_UPDATED`.

Repeated every block, this permanently freezes all unclaimed consensus-phase yield for every staker in the system. This matches the allowed impact: **High — Permanent freezing of unclaimed yield**.

### Likelihood Explanation

- The attacker needs only a valid, active `staker_address` (all staker addresses are public on-chain).
- No tokens, no privileged role, and no special setup are required — only gas.
- On Starknet the sequencer orders transactions; an attacker submitting a transaction at the beginning of each block (or front-running the legitimate call) can reliably trigger this every block.
- The attack is cheap and fully automated.

### Recommendation

Add a caller restriction to `update_rewards` analogous to the one already present on `update_rewards_from_attestation_contract`. The function should only be callable by a designated consensus contract (or the staker themselves), not by arbitrary addresses:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
+   assert!(
+       get_caller_address() == self.consensus_contract.read(),
+       "{}",
+       Error::CALLER_IS_NOT_CONSENSUS_CONTRACT,
+   );
    self.general_prerequisites();
    ...
```

Alternatively, if the function must remain permissionless, the `last_reward_block` write should be moved to **after** the `disable_rewards` branch so that a `disable_rewards: true` call does not consume the block slot.

### Proof of Concept

1. Deploy the system normally and advance past the consensus-rewards first epoch.
2. At block N, before the legitimate staker calls `update_rewards(staker, false)`, submit:
   ```
   staking.update_rewards(staker_address: any_active_staker, disable_rewards: true)
   ```
3. Observe `last_reward_block` is now N and no rewards were distributed.
4. The legitimate staker's call to `update_rewards(staker, false)` at block N reverts with `REWARDS_ALREADY_UPDATED`.
5. Repeat every block — stakers accumulate zero consensus rewards indefinitely. [6](#0-5) [3](#0-2)
