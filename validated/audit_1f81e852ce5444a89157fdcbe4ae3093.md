### Title
Missing Zero-Address Validation in `change_reward_address` Allows Permanent Loss of Staker/Pool-Member Yield - (File: src/staking/staking.cairo, src/pool/pool.cairo)

### Summary
Both `change_reward_address` implementations in the Staking and Pool contracts validate that the new `reward_address` is not a known token address, but neither checks that it is non-zero. A staker or pool member can set their reward address to `address(0)`, causing all subsequent reward transfers to be sent to the zero address, permanently destroying accrued yield.

### Finding Description
`change_reward_address` in `staking.cairo` performs exactly one validation on the incoming address:

```cairo
// src/staking/staking.cairo  lines 517-524
fn change_reward_address(ref self: ContractState, reward_address: ContractAddress) {
    self.general_prerequisites();
    assert!(
        !self.does_token_exist(token_address: reward_address),
        "{}",
        GenericError::REWARD_ADDRESS_IS_TOKEN,
    );
    // reward_address is written directly — no zero-address guard
    staker_info.reward_address = reward_address;
``` [1](#0-0) 

The pool-contract counterpart is identical in structure:

```cairo
// src/pool/pool.cairo  lines 505-517
fn change_reward_address(ref self: ContractState, reward_address: ContractAddress) {
    assert!(
        self.token_dispatcher.contract_address.read() != reward_address,
        "{}",
        GenericError::REWARD_ADDRESS_IS_TOKEN,
    );
    // no zero-address guard
    pool_member_info.reward_address = reward_address;
``` [2](#0-1) 

Once the zero address is stored, every downstream reward-dispatch path reads it and transfers directly to it without re-checking:

```cairo
// src/staking/staking.cairo  lines 1620-1625
let reward_address = staker_info.reward_address;
...
token_dispatcher.checked_transfer(recipient: reward_address, amount: amount.into());
``` [3](#0-2) 

```cairo
// src/pool/pool.cairo  lines 339-366
let reward_address = pool_member_info.reward_address;
...
reward_token.checked_transfer(recipient: reward_address, amount: rewards.into());
``` [4](#0-3) 

The protocol already defines a `ZERO_ADDRESS` error and uses `CALLER_IS_ZERO_ADDRESS` guards elsewhere, confirming that zero-address protection is an intentional design requirement that was simply omitted from `change_reward_address`. [5](#0-4) 

### Impact Explanation
Any accrued STRK rewards that are claimed after the reward address has been set to zero are transferred to `address(0)` and are permanently unrecoverable. This constitutes **permanent freezing / destruction of unclaimed yield** for the affected staker or pool member. The loss is irreversible because the staking contract has no recovery mechanism for misdirected reward transfers.

### Likelihood Explanation
The entry point is a standard, publicly callable function available to every registered staker and pool member — no privileged role is required. The scenario is reachable by:
- A user who mistakenly passes `0` as the new reward address (fat-finger / integration bug).
- A malicious actor who deliberately burns their own yield (griefing their own position, e.g., to manipulate off-chain accounting or reputation systems).

Because the function accepts any `ContractAddress` value and the only guard is the token-address check, the path to triggering the bug requires a single transaction.

### Recommendation
Add an explicit non-zero assertion in both `change_reward_address` implementations, mirroring the existing `CALLER_IS_ZERO_ADDRESS` pattern:

```cairo
assert!(reward_address.is_non_zero(), "{}", GenericError::ZERO_ADDRESS);
```

Apply the same guard to the initial `stake` call's `reward_address` parameter for consistency.

### Proof of Concept

1. Staker calls `stake(reward_address: VALID, ...)` — staker is registered normally.
2. Staker calls `change_reward_address(reward_address: 0)` — no assertion fires; `staker_info.reward_address` is written as `0`.
3. Rewards accrue over subsequent epochs via attestation.
4. Anyone calls `claim_rewards(staker_address)` — `send_rewards_to_staker` reads `reward_address = 0` and executes `checked_transfer(recipient: 0, amount: rewards)`.
5. Tokens are transferred to the zero address and are permanently lost; `unclaimed_rewards_own` is zeroed out, so the loss cannot be recovered by any future call.

The same four-step sequence applies to a pool member using `pool.change_reward_address(0)` followed by `pool.claim_rewards(pool_member)`.
