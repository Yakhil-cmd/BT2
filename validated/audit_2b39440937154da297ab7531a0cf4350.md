Looking at the key flow in `receive_erc20_tokens` and the `set_silo_params`/`set_erc20_fallback_address` entry points, I can now synthesize the full finding.

### Title
Zero `erc20_fallback_address` Silently Mints Bridged ERC-20 Tokens to the Zero Address, Permanently Freezing User Funds - (File: `engine/src/engine.rs`)

---

### Summary

In Silo mode, the `erc20_fallback_address` is a configurable EVM address that receives ERC-20 tokens when the intended bridge recipient is not whitelisted. Neither `set_silo_params` nor `set_erc20_fallback_address` validate that this address is non-zero. If the owner sets it to `Address::zero()`, every `ft_on_transfer` call for a non-whitelisted recipient will attempt to mint bridged ERC-20 tokens to `0x0000…0000`. Depending on the deployed ERC-20 token's mint guard, this either permanently destroys the tokens (permanent freeze) or causes the bridge call to revert and return tokens to the sender (DoS of the bridging path for all non-whitelisted users).

---

### Finding Description

`SiloParamsArgs.erc20_fallback_address` is typed as a plain `Address` with no non-zero invariant: [1](#0-0) 

`set_silo_params` passes whatever address is supplied directly to storage without any zero-address check: [2](#0-1) 

`set_erc20_fallback_address` in `lib.rs` likewise accepts `Some(Address::zero())` without rejection: [3](#0-2) 

`set_erc20_fallback_address` in the silo module stores any non-`None` address verbatim: [4](#0-3) 

At bridge time, `receive_erc20_tokens` reads the stored fallback address and unconditionally replaces the recipient with it when the original recipient is not whitelisted: [5](#0-4) 

If the stored value is `Address::zero()`, the subsequent EVM `call` mints tokens to `0x0000…0000`. [6](#0-5) 

---

### Impact Explanation

**Permanent freeze of bridged ERC-20 funds (Critical):** If the deployed ERC-20 token contract does not guard against minting to the zero address, tokens are minted to `0x0000…0000` and are irrecoverable. The NEP-141 side has already transferred the tokens to the Aurora contract, so the user loses the full bridged amount permanently.

**Denial-of-service / temporary freeze (High):** If the ERC-20 token does guard against minting to zero (e.g., standard OpenZeppelin `_mint` reverts), the EVM call fails, the error is caught in `ft_on_transfer`, and the NEP-141 tokens are returned to the sender: [7](#0-6) 

In this case every bridge attempt by a non-whitelisted user is silently rejected for as long as the zero fallback address remains set, effectively freezing the bridging path for all non-whitelisted participants.

---

### Likelihood Explanation

The owner account (`require_owner_and_running`) can call `set_silo_params` or `set_erc20_fallback_address` at any time. A zero address can be supplied accidentally (e.g., using a default-constructed `SiloParamsArgs`, whose `erc20_fallback_address` field defaults to `Address::zero()` per Rust's `Default` derive) or through a misconfigured deployment script. No on-chain guard prevents this. Once set, every subsequent `ft_on_transfer` for a non-whitelisted recipient hits the broken path. [1](#0-0) 

---

### Recommendation

Add a non-zero address check in both `set_silo_params` and `set_erc20_fallback_address` before persisting the value:

```rust
// In set_erc20_fallback_address and set_silo_params:
if let Some(addr) = address {
    require!(!addr.is_zero(), "ERR_ZERO_FALLBACK_ADDRESS");
    io.write_storage(&key, addr.as_bytes());
} else {
    io.remove_storage(&key);
}
```

Apply the same guard in the `set_silo_params` path before calling `set_erc20_fallback_address`.

---

### Proof of Concept

1. Owner calls `set_silo_params` with `SiloParamsArgs { fixed_gas: X, erc20_fallback_address: Address::zero() }`.
   - `set_silo_params` stores `Address::zero()` as the fallback address with no error.
2. The Address whitelist is enabled; a user's EVM address is not whitelisted.
3. The user calls `ft_transfer_call` on a NEP-141 token contract, routing to Aurora's `ft_on_transfer`.
4. `ft_on_transfer` → `receive_erc20_tokens`: the recipient is not whitelisted, so `recipient` is overwritten with `Address::zero()`.
5. The EVM `call` attempts to mint tokens to `0x0000…0000`.
   - **If the ERC-20 allows it:** tokens are minted to the zero address and permanently lost.
   - **If the ERC-20 rejects it:** the call reverts; `ft_on_transfer` returns the full amount, and the user's bridge attempt is silently rejected indefinitely. [8](#0-7) [9](#0-8)

    pub fn receive_erc20_tokens<P: PromiseHandler>(
        &mut self,
        token: &AccountId,
        args: &FtOnTransferArgs,
        current_account_id: &AccountId,
        handler: &mut P,
    ) -> Result<Option<SubmitResult>, ContractError> {
        let amount = args.amount.as_u128();
        // Parse message to determine recipient
        let mut recipient = {
            // The message should contain the recipient EOA address.
            let message = args.msg.strip_prefix("0x").unwrap_or(&args.msg);
            // Recipient - 40 characters (Address in hex without '0x' prefix)
            if message.len() < 40 {
                return Err(ParseOnTransferMessageError::WrongMessageFormat.into());
            }
            let mut address_bytes = [0; 20];
            hex::decode_to_slice(&message[..40], &mut address_bytes)
                .map_err(|_| ParseOnTransferMessageError::WrongMessageFormat)?;
            Address::from_array(address_bytes)
        };

        if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
            && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
        {
            recipient = fallback_address;
        }

        let erc20_token = get_erc20_from_nep141(&self.io, token)?;
        let erc20_admin_address = current_address(current_account_id);
        let result = self
            .call(
                &erc20_admin_address,
                &erc20_token,
                Wei::zero(),
                setup_receive_erc20_tokens_input(&recipient, amount),
                u64::MAX,
                Vec::new(), // TODO: are there values we should put here?
                Vec::new(),
                handler,
            )
            .and_then(submit_result_or_err)?;

        sdk::log!("Mint {amount} ERC-20 tokens for: {}", recipient.encode());

        // Return SubmitResult so that it can be accessed in standalone engine.
        // This is used to help with the indexing of bridge transactions.
        Ok(Some(result))
