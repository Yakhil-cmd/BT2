### Title
Missing Lower-Bound Validation on `threshold` in FROST `presign` and `assert_sign_inputs` - (File: src/frost/mod.rs)

### Summary
`assert_key_invariants` in `src/dkg.rs` enforces both an upper bound (`threshold > participants.len()`) and a lower bound (`threshold < 2`) on the threshold parameter. The analogous validation helpers `presign` and `assert_sign_inputs` in `src/frost/mod.rs` enforce only the upper bound and are entirely missing the lower-bound check (`threshold < 2`). A caller can therefore pass `threshold = 1` (or `0`) to the EdDSA/RedJubjub signing path, which the library silently accepts and proceeds with, producing cryptographically corrupted or unusable signing outputs.

### Finding Description

`assert_key_invariants` in `src/dkg.rs` performs two threshold checks: [1](#0-0) 

Both the upper-bound check (`threshold > participants.len()`) and the lower-bound check (`threshold < 2`) are present.

In contrast, `presign` in `src/frost/mod.rs` only checks the upper bound: [2](#0-1) 

There is no corresponding `threshold < 2` guard. The same omission exists in `assert_sign_inputs`: [3](#0-2) 

Again, only the upper bound is checked; the lower bound is absent.

The `ReconstructionLowerBound` type is a plain `usize` newtype with no enforced minimum: [4](#0-3) 

Any non-negative integer, including `0` and `1`, is accepted by the type system and passes both `presign` and `assert_sign_inputs` without error.

### Impact Explanation

When `threshold = 1` (or `0`) is supplied to the FROST signing path, the protocol proceeds using Lagrange interpolation coefficients computed for a single participant. Because the key shares were generated under a threshold of at least 2 (enforced by DKG), the interpolation coefficients used during signing are inconsistent with those used during key generation. The resulting partial signatures are combined with wrong weights, producing a final signature that is cryptographically invalid and unusable. Honest parties that complete the protocol receive a corrupted signing output. This maps directly to:

> **High: Corruption of sign outputs so honest parties accept unusable cryptographic outputs.**

### Likelihood Explanation

The library is a public Rust API. Any integrator that constructs a `PresignArguments` or calls `assert_sign_inputs` with a misconfigured or attacker-supplied threshold value will silently bypass the guard. No privileged access is required; the caller only needs to invoke the public `presign` or the exported `assert_sign_inputs` function with `threshold = 1`. The DKG path would have caught this, but the signing path does not, creating an inconsistency that is easy to trigger by accident or by a malicious coordinator supplying a crafted threshold.

### Recommendation

Add the same lower-bound guard that exists in `assert_key_invariants` to both `presign` and `assert_sign_inputs` in `src/frost/mod.rs`:

```rust
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

This mirrors the existing check at `src/dkg.rs` lines 580–582 and closes the inconsistency between the DKG and signing validation paths.

### Proof of Concept

1. Run DKG with 3 participants and `threshold = 2` (accepted, enforced by `assert_key_invariants`).
2. Call `presign` from `src/frost/mod.rs` with the same 3 participants but `threshold = 1`. The function returns `Ok(...)` — no error is raised.
3. Proceed to signing using the presign output. Lagrange interpolation runs with `threshold = 1` coefficients, which are inconsistent with the key shares generated under `threshold = 2`.
4. The aggregated signature fails external verification, producing a permanently unusable signing output for honest parties despite a correctly completed protocol run.

The root cause is the absent `threshold < 2` check at `src/frost/mod.rs` lines 71–77 (`presign`) and lines 144–150 (`assert_sign_inputs`), contrasted with the complete validation at `src/dkg.rs` lines 580–582.
