### Title
`pause_contract` Does Not Check for In-Progress wNEAR Unwrap Operations, Causing Permanent Loss of User Funds - (`engine/src/contract_methods/admin.rs`)

---

### Summary

The `pause_contract` function in Aurora Engine sets `is_paused = true` without checking whether any multi-step NEAR promise chains are in flight. The wNEAR-unwrap path of the `ExitToNear` precompile is a two-receipt operation: receipt 1 calls `near_withdraw` on the wNEAR contract (burning the NEP-141 token and crediting NEAR to Aurora), and receipt 2 calls back into `exit_to_near_precompile_callback` on Aurora to forward that NEAR to the user. If the owner calls `pause_contract` between these two receipts, the callback panics at `require_running`, the NEAR stays locked in Aurora's account, and the user's ERC-20 wNEAR tokens are permanently burned with no recourse.

---

### Finding Description

**Step 1 – User initiates wNEAR unwrap via `ExitToNear` precompile.**

A user holding wNEAR ERC-20 tokens on Aurora calls the `ExitToNear` precompile with the `:unwrap` suffix. Inside `ExitToNear::run`, the precompile burns the ERC-20 tokens and constructs a two-step NEAR promise chain:

```
near_withdraw(amount)  →  exit_to_near_precompile_callback(transfer_near_args)
``` [1](#0-0) 

The `transfer_near_args` field carries the user's NEAR account ID and the amount to forward. This promise is emitted as a log and scheduled by the NEAR runtime.

**Step 2 – `near_withdraw` executes successfully.**

Receipt 1 executes: the wNEAR NEP-141 contract burns the wNEAR tokens and transfers the corresponding NEAR to Aurora's account. This state change is committed and irreversible.

**Step 3 – Owner calls `pause_contract`.**

Between the two receipts, the owner calls `pause_contract`. This function only checks `require_owner_only` and `require_running`; it has no awareness of in-flight promise chains: [2](#0-1) 

The contract state is now `is_paused = true`.

**Step 4 – `exit_to_near_precompile_callback` panics.**

Receipt 2 executes. The very first thing `exit_to_near_precompile_callback` does is call `require_running`: [3](#0-2) 

`require_running` returns `Err(ERR_PAUSED)`: [4](#0-3) 

The error propagates to `sdk_unwrap()` in `lib.rs`, which panics: [5](#0-4) 

**Step 5 – Funds are permanently lost.**

When a NEAR callback panics, the NEAR runtime reverts only the callback's own state changes. The base promise (`near_withdraw`) has already committed: the wNEAR NEP-141 tokens are burned and the NEAR is in Aurora's account. The callback never reaches the `transfer_near` branch that would forward NEAR to the user: [6](#0-5) 

The NEAR is now stranded in Aurora's account. The user's ERC-20 wNEAR tokens are burned. There is no user-accessible recovery path.

---

### Impact Explanation

**Critical – Permanent freezing of funds.**

The user's wNEAR ERC-20 tokens are burned at the start of the `ExitToNear` precompile execution (before any promise is scheduled). The NEAR that `near_withdraw` sends to Aurora is committed and cannot be automatically refunded because the callback panics before any refund logic is reached. The `refund` field in `ExitToNearPrecompileCallbackArgs` is only populated for the case where the *base* promise fails, not for the case where the callback itself panics: [7](#0-6) 

The stranded NEAR has no user-facing claim mechanism. From the user's perspective this is a permanent loss.

---

### Likelihood Explanation

**Medium.** The owner calling `pause_contract` during an emergency is an expected and legitimate operational action. NEAR processes receipts asynchronously across blocks; the window between `near_withdraw` completing and `exit_to_near_precompile_callback` executing spans at least one block. Any emergency pause issued during that block interval triggers the loss. The more users are actively unwrapping wNEAR at the time of the pause, the more funds are affected simultaneously.

---

### Recommendation

1. **Remove `require_running` from `exit_to_near_precompile_callback`.** This callback is a private self-call (`env.assert_private_call()` is already enforced) that completes a promise chain initiated before the pause. It should be allowed to complete regardless of the pause state, just as NEAR's own runtime does not roll back already-committed receipts.

2. **Alternatively**, before `pause_contract` commits the paused state, check whether any wNEAR-unwrap callbacks are pending. Because NEAR does not expose a pending-receipt count, the practical fix is option 1.

---

### Proof of Concept

**Sequence of events:**

```
Block N:
  User calls submit() → ExitToNear precompile runs
    → ERC-20 wNEAR burned from user's EVM balance
    → Promise scheduled: near_withdraw → exit_to_near_precompile_callback

Block N (same or next receipt):
  near_withdraw executes on wNEAR contract
    → wNEAR NEP-141 tokens burned
    → NEAR transferred to Aurora account  ← committed, irreversible

Block N (between receipts) or Block N+1:
  Owner calls pause_contract()
    → state.is_paused = true

Block N+1 (or N+2):
  exit_to_near_precompile_callback executes
    → require_running(&state)? → Err(ERR_PAUSED) → sdk_unwrap() panics
    → NEAR stays in Aurora account
    → User receives nothing

Result:
  User's ERC-20 wNEAR: burned (gone)
  User's NEAR:         never received
  Aurora account NEAR: increased by the unwrapped amount (stranded)
```

The entry path is fully user-controlled: any holder of wNEAR ERC-20 tokens on Aurora can trigger this scenario by initiating an unwrap at any time. The loss is realized the moment the owner issues a pause during the inter-receipt window.

#[named]
pub fn pause_contract<I: IO + Copy, E: Env>(io: I, env: &E) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        let mut state = state::get_state(&io)?;
        require_owner_only(&state, &env.predecessor_account_id())?;
        require_running(&state)?;
        state.is_paused = true;
        state::set_state(&mut io, &state)?;
        Ok(())
    })
}
