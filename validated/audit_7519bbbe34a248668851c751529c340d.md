### Title
TOCTOU Race in PlotNFT Pool Join Flow Allows Malicious Pool Server to Substitute `targetPuzzleHash` After User Approval — (`packages/gui/src/components/plotNFT/select/PlotNFTSelectPool.tsx`)

---

### Summary

The "Verify Pool Details" panel and the actual `pwJoinPool` submission each make **independent** HTTP calls to the pool server via `getPoolInfo`. A malicious pool operator can serve a legitimate `targetPuzzleHash` for the display request and a different, attacker-controlled `targetPuzzleHash` for the submit request. The user approves one hash but a different hash is committed on-chain, permanently redirecting farming rewards to the attacker.

---

### Finding Description

Two completely separate `getPoolInfo` calls are made with no cross-validation between them:

**Call 1 — Display (in `PlotNFTSelectBase.tsx`):**

`usePoolInfo(poolUrl)` is invoked reactively as the user types the pool URL. Its result is passed to `<PoolInfo poolInfo={poolInfo.poolInfo} />`, which explicitly renders `targetPuzzleHash` in the "Verify Pool Details" card. [1](#0-0) [2](#0-1) [3](#0-2) 

**Call 2 — Submit (in `PlotNFTSelectPool.tsx`):**

When the user clicks submit, `handleSubmit` calls `prepareSubmitData`, which makes a **fresh, independent** `getPoolInfo` call and uses its returned `targetPuzzleHash` directly in the `pwJoinPool` payload — with no comparison against the value already shown to the user. [4](#0-3) [5](#0-4) 

The `targetPuzzleHash` from call 2 flows directly into `pwJoinPool` via `PlotNFTChangePool`: [6](#0-5) 

There is no guard anywhere that asserts the hash submitted equals the hash displayed.

---

### Impact Explanation

`targetPuzzleHash` is the on-chain destination for all farming rewards from the PlotNFT. Submitting an attacker-controlled hash permanently binds the NFT to the attacker's puzzle hash until the user pays another fee to change pools. Every block reward earned while the NFT is bound to the wrong hash is lost to the attacker. This is a direct, irreversible financial loss affecting pooled farming rewards — explicitly listed as Critical impact. [7](#0-6) 

---

### Likelihood Explanation

The attacker must operate (or compromise) the pool server at the URL the user enters. This is a realistic threat: a malicious pool operator can trivially implement response differentiation (e.g., serve hash A on the first `/pool_info` request and hash B on the second within the same session). The window between display and submit is seconds — no timing precision is required. The user has no mechanism to detect the substitution because the GUI never compares the two fetched values.

---

### Recommendation

Remove the second `getPoolInfo` call entirely. `prepareSubmitData` should accept the already-fetched `poolInfo` object (from `usePoolInfo`) as a parameter and read `targetPuzzleHash` from it, rather than re-fetching. This ensures the hash submitted is identical to the hash the user reviewed.

Concretely: pass `poolInfo.poolInfo` down from `PlotNFTSelectBase` into `PlotNFTSelectPool`'s submit handler, and have `prepareSubmitData` consume it directly instead of calling `getPoolInfo` again. [8](#0-7) 

---

### Proof of Concept

1. Stand up a pool server at `http://attacker.pool/`.
2. Implement `/pool_info` to return **hash A** (a legitimate-looking puzzle hash) on the first request and **hash B** (attacker-controlled) on all subsequent requests.
3. Open the Chia GUI → Pool → Change Pool, enter `http://attacker.pool/`.
4. The "Verify Pool Details" panel renders hash A (from call 1 via `usePoolInfo`).
5. User reads hash A, considers it legitimate, clicks submit.
6. `prepareSubmitData` fires call 2 → pool server returns hash B.
7. `pwJoinPool` is called with `targetPuzzlehash: hashB`.
8. Assert on-chain that the PlotNFT's target puzzle hash is hash B, not hash A.

No timing precision, no local compromise, and no cryptographic attack is required. The race window is the entire duration between page render and button click. [9](#0-8)

```typescript
async function prepareSubmitData(data: FormData): SubmitData {
  const { self, fee, poolUrl } = data;
  const initialTargetState = {
    state: self ? 'SELF_POOLING' : 'FARMING_TO_POOL',
  };

  if (!self && poolUrl) {
    const normalizedPoolUrl = normalizeUrl(poolUrl);
    const { targetPuzzleHash, relativeLockHeight } = await getPoolInfo(normalizedPoolUrl);
    if (!targetPuzzleHash) {
      throw new Error(t`Pool does not provide targetPuzzleHash.`);
    }
    if (relativeLockHeight === undefined) {
      throw new Error(t`Pool does not provide relativeLockHeight.`);
    }

    initialTargetState.poolUrl = normalizedPoolUrl;
    initialTargetState.targetPuzzleHash = targetPuzzleHash;
    initialTargetState.relativeLockHeight = relativeLockHeight;
  }

  const feeMojos = chiaToMojo(fee || '0');

  return {
    fee: feeMojos,
    initialTargetState,
  };
}
```
