### Title
Unrestricted `update_rewards` Allows Any Caller to Suppress Consensus Reward Distribution via Arbitrary `staker_address` — (File: src/staking/staking.cairo)

---

### Summary

`update_rewards` in the Staking contract accepts a caller-supplied `staker_address` with no verification that the caller has any relationship to that staker. Combined with the caller-controlled `disable_rewards: true` flag, any unprivileged address can consume the single per-block reward slot without distributing rewards, permanently destroying that block's yield for all stakers.

---

### Finding Description

`update_rewards` is gated only by `general_prerequisites()`, which checks that the contract is not paused and the caller is non-zero. No check verifies that `get_caller_address() == staker_address` or that the caller is the staker's operational address. [1](#0-0) 

The function enforces a single global `last_reward_block` to allow only one reward update per block: [2](#0-1) 

When called with `disable_rewards: true`, the function writes `current_block_number` to `last_reward_block` and returns immediately, distributing nothing: [3](#0-2) 

Because `last_reward_block` is a single global storage slot, this one call exhausts the reward slot for the entire block. Any subsequent legitimate call to `update_rewards` in the same block reverts with `REWARDS_ALREADY_UPDATED`. [4](#0-3) 

The structural parallel to the external report is exact: just as `ZealousSwapPair.swap()` accepted an arbitrary `actualUser` without verifying it equaled `msg.sender`, `update_rewards` accepts an arbitrary `staker_address` without verifying the caller controls or represents that staker. In both cases the missing caller-address binding lets an unprivileged actor manipulate protocol behavior to the detriment of legitimate users.

---

### Impact Explanation

An attacker who calls `update_rewards(any_valid_staker, disable_rewards: true)` in every block permanently destroys that block's consensus rewards for all stakers. The rewards are not deferred or accumulated — they are simply never distributed. Sustained execution constitutes **permanent freezing of unclaimed yield** across the entire protocol (High severity per the allowed impact scope).

---

### Likelihood Explanation

The attack requires no capital, no privileged role, and no special knowledge beyond a valid staker address (publicly readable from the `stakers` vector in storage). [5](#0-4) 

A simple bot calling the function once per block is sufficient. The attacker only needs to land their transaction before any legitimate `update_rewards` call in the same block, which is trivially achievable via standard gas-price competition or front-running.

---

### Recommendation

Add a caller check to `update_rewards` so that only the staker themselves or their registered operational address may invoke it:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
    let caller = get_caller_address();
    let staker_info = self.internal_staker_info(:staker_address);
    assert!(
        caller == staker_address || caller == staker_info.operational_address,
        "{}",
        Error::CALLER_IS_NOT_STAKER_OR_OPERATIONAL,
    );
    // ... rest of function
}
```

Alternatively, restrict `disable_rewards: true` to a privileged role (e.g., security agent) and require the caller to be the staker or operational address for normal reward claims.

---

### Proof of Concept

1. Attacker identifies any active staker address `S` with non-zero STRK balance (readable from public storage/events via the `stakers` vec).
2. Each block, attacker submits `update_rewards(S, disable_rewards: true)` with sufficient gas to be included before any legitimate call.
3. The function writes `current_block_number` to `last_reward_block` and returns without distributing rewards. [3](#0-2) 
4. Any legitimate staker attempting `update_rewards` in the same block receives `REWARDS_ALREADY_UPDATED` and earns nothing for that block.
5. Repeated every block, this permanently prevents all consensus reward distribution across the protocol.
