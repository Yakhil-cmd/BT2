### Title
Missing Minimum-Threshold Enforcement in FROST Signing Path Breaks Invariant Enforced by DKG — (`src/frost/mod.rs`)

---

### Summary

The DKG initialization path (`assert_key_invariants`) enforces `threshold >= 2` as a hard invariant. The FROST signing path (`assert_sign_inputs`) and FROST presigning path (`presign`) in `src/frost/mod.rs` perform no such check. A malicious coordinator or unprivileged library caller can invoke FROST signing with `threshold = 1`, bypassing the minimum-threshold invariant, causing incorrect Lagrange-coefficient computation and corrupted, unverifiable signature output.

---

### Finding Description

`assert_key_invariants` in `src/dkg.rs` enforces two threshold bounds:

```rust
if threshold > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
if threshold < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
}
``` [1](#0-0) 

The FROST signing validator `assert_sign_inputs` in `src/frost/mod.rs` checks only the upper bound:

```rust
if threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
``` [2](#0-1) 

There is no corresponding `threshold < 2` guard anywhere in `assert_sign_inputs`. [3](#0-2) 

The same omission exists in the FROST `presign` entry point, which validates `threshold <= participants.len()` but never `threshold >= 2`: [4](#0-3) 

The DKG path that produced the key shares always required `threshold >= 2`. The signing path that consumes those shares does not re-enforce this invariant, creating an inconsistency directly analogous to the external report's `secondaryMinter` bypass.

---

### Impact Explanation

FROST signing uses the caller-supplied `threshold` to compute Lagrange coefficients for each participant's share. When `threshold = 1` is supplied:

- The interpolation is performed as if the underlying polynomial has degree 0 (constant), so every participant's Lagrange coefficient collapses to 1.
- The actual DKG polynomial has degree `threshold_dkg - 1 >= 1`, so the shares are not consistent with degree-0 interpolation.
- The aggregated signature scalar is computed from the wrong linear combination of shares, producing a value that fails standard ECDSA/EdDSA verification.

Honest participants complete the protocol without error (no guard fires), yet the coordinator assembles and returns an invalid, unusable signature. This maps to:

> **High: Corruption of sign outputs so honest parties accept unusable cryptographic outputs.**

---

### Likelihood Explanation

The `assert_sign_inputs` function is the public API surface that every FROST signing caller is expected to invoke. Any caller — including a malicious coordinator who controls the `threshold` argument — can pass `threshold = 1`. No special privilege, leaked key, or cryptographic break is required. The existing `participants.len() >= 2` guard still passes, so the protocol proceeds silently to produce a broken signature.

---

### Recommendation

Add the same lower-bound guard that `assert_key_invariants` applies, directly inside `assert_sign_inputs` and `presign` in `src/frost/mod.rs`:

```rust
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

This mirrors the existing check in `assert_key_invariants` and closes the inconsistency between the DKG and signing validation paths. [5](#0-4) 

---

### Proof of Concept

1. Run DKG with `participants = [P1, P2, P3]`, `threshold = 2`. Each party receives a degree-1 secret share.
2. Call `assert_sign_inputs(&[P1, P2, P3], threshold=1, me=P1, coordinator=P1)`. The call succeeds — no error is returned.
3. Proceed to FROST signing with `threshold = 1`. Lagrange coefficients are computed for degree-0 interpolation (all equal to 1), which is inconsistent with the degree-1 shares from DKG.
4. The coordinator aggregates the shares and produces a signature scalar `s` that does not satisfy the ECDSA/EdDSA verification equation for the public key derived from the DKG.
5. All honest parties accepted the protocol output; the resulting signature is invalid and unusable. [3](#0-2) [6](#0-5)

pub fn assert_sign_inputs(
    participants: &[Participant],
    threshold: impl Into<ReconstructionLowerBound>,
    me: Participant,
    coordinator: Participant,
) -> Result<ParticipantList, InitializationError> {
    let threshold = threshold.into();
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // validate threshold
    if threshold.value() > participants.len() {
        return Err(InitializationError::ThresholdTooLarge {
            threshold: threshold.value(),
            max: participants.len(),
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }
    Ok(participants)
