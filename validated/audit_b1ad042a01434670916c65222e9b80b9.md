### Title
`DelegateAction` Expiry Check Uses Strict `>` Instead of `>=`, Allowing Execution at Exact `max_block_height` — (`File: runtime/runtime/src/actions.rs`)

### Summary

`apply_delegate_action` checks expiry with `apply_state.block_height > delegate_action.max_block_height()`. The protocol specification and field documentation both state the action must be rejected when `block_height >= max_block_height`. The strict `>` operator allows a relayer to execute a `DelegateAction` at exactly `block_height == max_block_height`, one block past the user's intended deadline.

### Finding Description

`DelegateAction` (NEP-366 meta-transactions) carries a user-signed field `max_block_height`:

> "The maximal height of the block in the blockchain **below which** the given DelegateAction is valid." [1](#0-0) 

The runtime spec is explicit:

> "If the current block is **equal or greater than** `max_block_height`" → `DelegateActionExpired` [2](#0-1) 

The production enforcement in `apply_delegate_action` reads:

```rust
if apply_state.block_height > delegate_action.max_block_height() {
    result.result = Err(ActionErrorKind::DelegateActionExpired.into());
    return Ok(());
}
``` [3](#0-2) 

When `block_height == max_block_height`, the condition evaluates to `false`, so the action is **not** rejected. The existing test `test_delegate_action_max_height` only exercises `max_block_height + 1` (strictly greater) and confirms expiry there, but never tests the equal case: [4](#0-3) 

The happy-path test `test_delegate_action` explicitly passes `max_block_height` as the current block height and asserts `result.is_ok()`, cementing the off-by-one as an accepted (but incorrect) behavior: [5](#0-4) 

Both `Action::Delegate` and `Action::DelegateV2` reach the same `apply_delegate_action` function: [6](#0-5) 

### Impact Explanation

A relayer who receives a signed `DelegateAction` with `max_block_height = N` can deliberately delay submission until block `N` and still have the action executed, bypassing the user's expiry intent. The user signed the action expecting it to be invalid at block `N` (the field semantics say "valid only **below** `max_block_height`"). Any time-sensitive authorization the user encoded in the expiry — e.g., "this transfer must not execute after block N because I will have revoked the key or changed state" — is violated by exactly one block. The corrupted admission decision is: action admitted at `block_height == max_block_height` when the invariant requires rejection.

Scope: **pre-inclusion transaction validation** and **authorization** (the expiry check is the sole authorization gate on DelegateAction timing).

### Likelihood Explanation

Any relayer submitting a meta-transaction can observe the current chain height and time their submission to land at exactly `max_block_height`. No special privilege is required — the relayer is an unprivileged network participant. The user-controlled `max_block_height` field is the attacker-controlled input.

### Recommendation

Change the comparison in `apply_delegate_action` from strict `>` to `>=`:

```rust
// Before (incorrect):
if apply_state.block_height > delegate_action.max_block_height() {

// After (correct, matches spec):
if apply_state.block_height >= delegate_action.max_block_height() {
```

Add a test asserting `DelegateActionExpired` when `block_height == max_block_height`.

### Proof of Concept

1. User creates and signs a `DelegateAction` with `max_block_height = 1000`, intending the action to be invalid at block 1000 and beyond.
2. Relayer receives the signed action but waits until the chain reaches block 1000.
3. Relayer submits the action at block 1000.
4. `apply_delegate_action` evaluates `1000 > 1000` → `false` → action is **not** rejected.
5. The inner actions execute at block 1000, violating the user's expiry intent.
6. The existing unit test `test_delegate_action` at line 1289–1303 already demonstrates this: it passes `max_block_height` as `block_height` and asserts `result.is_ok()`. [3](#0-2)
