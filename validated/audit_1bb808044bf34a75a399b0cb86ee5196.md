### Title
No-Refund of Overpaid wNEAR When XCC Target Contract Rejects Attached NEAR - (`File: engine-precompiles/src/xcc.rs`, `engine/src/xcc.rs`, `etc/xcc-router/src/lib.rs`)

### Summary

When an Aurora EVM user makes a Cross-Contract Call (XCC) via the `CrossContractCall` precompile and attaches NEAR (via `attached_balance` in `PromiseCreateArgs`) to the outbound NEAR call, the wNEAR is irrevocably transferred from the user's EVM balance to the router sub-account before the NEAR promise executes. If the target NEAR contract rejects the call or the attached NEAR is not consumed, there is no callback or refund mechanism to return the unused NEAR (as wNEAR or otherwise) to the originating EVM address. The user permanently loses the overpaid or rejected NEAR value.

### Finding Description

The XCC flow works as follows:

1. The user calls the `CrossContractCall` precompile (`engine-precompiles/src/xcc.rs`) with a `CrossContractCallArgs::Eager(call)` or `Delayed(call)` containing `attached_balance` (yoctoNEAR to attach to the outbound NEAR call).

2. The precompile computes `required_near = attached_near` (or `attached_near + STORAGE_AMOUNT` for first-time deployment) and immediately executes a wNEAR `transferFrom` from the user's EVM address to the engine's implicit address: [1](#0-0) 

3. The engine then schedules a `withdraw_wnear_to_router` promise that unwraps the wNEAR into native NEAR and sends it to the router sub-account: [2](#0-1) 

4. The router's `execute` method dispatches the user's promise with the attached NEAR balance: [3](#0-2) 

5. The only refund mechanism in the entire XCC promise chain is `send_refund`, which is **only triggered** when `refund_needed` is true — i.e., only when a brand-new router account was just created (`create_needed == true`). In that case, it refunds the fixed `STORAGE_AMOUNT` (2 NEAR) back to the engine, not the user's attached NEAR: [4](#0-3) [5](#0-4) 

The `send_refund` method always sends the hardcoded `REFUND_AMOUNT` (2 NEAR = `STORAGE_AMOUNT`) back to the engine contract — not to the user, and not the user's `attached_balance`: [6](#0-5) 

There is **no callback** attached after the router's `execute` call (step 3 in `handle_precompile_promise`) to check whether the target NEAR contract accepted or rejected the attached NEAR, and no mechanism to return unused NEAR to the user's wNEAR EVM balance: [7](#0-6) 

The user's wNEAR is burned from their EVM balance at precompile execution time (synchronously, within the EVM transaction), but the NEAR delivery to the target contract is asynchronous. If the target contract panics, returns the NEAR, or simply does not use all of it, the NEAR remains stranded in the router sub-account with no path back to the user's EVM wNEAR balance.

### Impact Explanation

Any EVM user or contract that uses the XCC precompile with a non-zero `attached_balance` in their `PromiseCreateArgs` will have their wNEAR permanently deducted from their EVM balance regardless of whether the target NEAR contract accepts, rejects, or partially uses the attached NEAR. If the target contract rejects the call (panics), the NEAR is returned to the router sub-account by the NEAR runtime, but there is no callback to re-wrap it as wNEAR and credit it back to the user's EVM address. The funds are permanently stranded in the router sub-account, inaccessible to the EVM user. This constitutes a **permanent freezing of user funds** (the stranded NEAR in the router) for any XCC call where the target contract rejects or does not fully consume the attached NEAR.

### Likelihood Explanation

This is highly likely to occur in practice. Any cross-chain application that attaches NEAR to XCC calls (e.g., to pay for storage deposits, to call payable NEAR contracts, or to fund DeFi operations) and encounters a rejection or partial consumption will trigger this loss. NEAR contracts frequently reject calls with insufficient or excess attached deposits. The XCC feature is a core Aurora capability used by cross-chain dApps, making this a realistic and recurring scenario.

### Recommendation

Add a callback after the final user promise in `handle_precompile_promise` (in `engine/src/xcc.rs`) that checks the promise result. If the target NEAR contract rejected the call and returned NEAR to the router, the callback should instruct the router to re-wrap the returned NEAR as wNEAR and credit it back to the originating EVM address. Alternatively, model the refund similarly to how `send_refund` works for storage staking: attach a post-execution callback to the router that calls a new `send_near_refund(amount, evm_address)` method, which wraps any returned NEAR back into wNEAR and mints it to the user's EVM address. This mirrors the Arbitrum approach cited in the reference report: specifying a `refundRecipient` so overpaid fees are returned.

### Proof of Concept

1. User holds 10 wNEAR as ERC-20 on Aurora.
2. User calls the XCC precompile with `CrossContractCallArgs::Eager(PromiseArgs::Create(PromiseCreateArgs { target_account_id: "some_payable_contract.near", attached_balance: Yocto::new(5_000_000_000_000_000_000_000_000), ... }))`.
3. The precompile immediately executes `transferFrom` on the wNEAR ERC-20, deducting 5 wNEAR from the user's EVM balance.
4. The engine schedules `withdraw_wnear_to_router` → router `execute` → call to `some_payable_contract.near` with 5 NEAR attached.
5. `some_payable_contract.near` panics (e.g., wrong arguments, storage deposit already paid, or contract paused). The NEAR runtime returns the 5 NEAR to the router sub-account.
6. No callback exists after the `execute` promise to detect the failure and re-wrap the 5 NEAR as wNEAR for the user.
7. The user's EVM wNEAR balance is permanently reduced by 5 wNEAR. The 5 NEAR sits in the router sub-account, inaccessible from the EVM.

The root cause is the absence of a post-execution callback in `handle_precompile_promise`: [8](#0-7) 

compared to the only existing refund path which is limited to the storage-staking case: [4](#0-3)

    pub fn send_refund(&self) -> Promise {
        let parent = self.get_parent().unwrap_or_else(env_panic);

        require_caller(&parent)
            .and_then(|_| require_no_failed_promises())
            .unwrap_or_else(env_panic);

        Promise::new(parent).transfer(REFUND_AMOUNT)
    }
