### Title
Pool Admin Bypasses Fee Cap via Uncapped `setPoolBinAdditionalFees` — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary
`MetricOmmPoolFactory.setPoolBinAdditionalFees` forwards `addFeeBuyE6` / `addFeeSellE6` to the pool with no upper-bound check, while the parallel `setPoolAdminFees` path enforces `maxAdminSpreadFeeE6` / `maxAdminNotionalFeeE8` caps. A pool admin can set either value to `uint16` max (65535, ≈ 6.55% in E6 units) on any bin, causing traders who swap through that bin to pay fees far above the documented admin cap.

### Finding Description
`setPoolAdminFees` enforces caps before updating the pool:

```solidity
// MetricOmmPoolFactory.sol lines 414–415
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (newAdminNotionalFeeE8 > maxAdminNotionalFeeE8) revert AdminFeeTooHigh();
``` [1](#0-0) 

`setPoolBinAdditionalFees` has no equivalent guard:

```solidity
// MetricOmmPoolFactory.sol lines 450–457
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
  external override nonReentrant onlyPoolAdmin(pool)
{
  IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
``` [2](#0-1) 

`MetricOmmPool.setBinAdditionalFees` also has no cap — it only validates the bin index:

```solidity
// MetricOmmPool.sol lines 464–474
function setBinAdditionalFees(int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
  external onlyFactory nonReentrant(PoolActions.SET_BIN_ADDITIONAL_FEES)
{
  if (bin < LOWEST_BIN || bin > HIGHEST_BIN) revert InvalidBinIndex(bin);
  BinState storage s = _binStates[bin];
  s.addFeeBuyE6 = addFeeBuyE6;
  s.addFeeSellE6 = addFeeSellE6;
  ...
}
``` [3](#0-2) 

During every swap, the bin additional fee is added directly to the oracle base fee before being passed to `SwapMath`:

```solidity
// MetricOmmPool.sol line 999 (buyToken0 specifiedIn path)
params.baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6)
``` [4](#0-3) 

The same pattern appears in all four swap directions (lines 910, 999, 1088, 1177). [5](#0-4) [6](#0-5) [7](#0-6) 

### Impact Explanation
A pool admin sets `addFeeBuyE6 = 65535` on the active bin. Every trader buying token0 through that bin pays an additional ~6.55% fee on top of the oracle spread fee. This fee accrues into the bin's LP balance (minus the protocol share), directly reducing the trader's received output — a measurable, per-swap loss of user principal. The admin-imposed fee cap (`maxAdminSpreadFeeE6`) is bypassed entirely via this path.

### Likelihood Explanation
The pool admin is a semi-trusted role that is explicitly expected to be constrained by caps. The `setPoolBinAdditionalFees` function is callable at any time by the pool admin with no timelock. The bypass requires only a single transaction.

### Recommendation
Add a cap check in `MetricOmmPoolFactory.setPoolBinAdditionalFees` analogous to the one in `setPoolAdminFees`. Define a `maxBinAdditionalFeeE6` constant or factory-level parameter and revert if either `addFeeBuyE6` or `addFeeSellE6` exceeds it:

```solidity
if (addFeeBuyE6 > maxBinAdditionalFeeE6) revert AdminFeeTooHigh();
if (addFeeSellE6 > maxBinAdditionalFeeE6) revert AdminFeeTooHigh();
```

### Proof of Concept
1. Deploy a pool with a pool admin.
2. As pool admin, call `factory.setPoolBinAdditionalFees(pool, 0, 65535, 65535)`.
3. Perform a swap through bin 0.
4. Observe that the effective fee charged = `baseFeeX64 + 65535/1e6` in X64 units — approximately 6.55% additional fee with no factory-level rejection.
5. Confirm `setPoolAdminFees(pool, 65535, 0)` would revert with `AdminFeeTooHigh` if `maxAdminSpreadFeeE6 < 65535`, demonstrating the inconsistency.
