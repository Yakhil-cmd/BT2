### Title
Single `MAX_REF_STALENESS` Applied to Both Legs in Synthetic Ratio Mode — (`File: smart-contracts-poc/contracts/AnchoredPriceProvider.sol`)

### Summary
`AnchoredPriceProvider` supports a two-feed synthetic ratio mode (`baseFeedId / quoteFeedId`, e.g. BTC/USD ÷ ETH/USD = BTC/ETH). A single immutable `MAX_REF_STALENESS` is applied identically to both legs inside `_readLeg`. When the two feeds have different heartbeat intervals, no single threshold can be correct for both: setting it to the slower feed's heartbeat allows the faster feed to be severely stale; setting it to the faster feed's heartbeat causes unnecessary halts on the slower feed.

### Finding Description

`_getBidAndAskPrice` reads both legs through `_readLeg`:

```solidity
// AnchoredPriceProvider.sol lines 258-271
function _getBidAndAskPrice() internal returns (uint128, uint128) {
    (uint256 mid, uint256 spreadBps, , bool ok) = _readLeg(baseFeedId);
    if (!ok) return (0, type(uint128).max);

    bytes32 _quote = quoteFeedId;
    if (_quote != bytes32(0)) {
        (uint256 mid2, uint256 spreadBps2, , bool ok2) = _readLeg(_quote);
        ...
    }
    ...
}
```

Inside `_readLeg`, the staleness check uses the single contract-wide `MAX_REF_STALENESS` for whichever `feedId` is passed:

```solidity
// AnchoredPriceProvider.sol line 283
if (_isStale(refTime, block.timestamp, MAX_REF_STALENESS)) return (mid, spreadBps, refTime, false);
```

`MAX_REF_STALENESS` is a single immutable set at construction time with no per-feed override:

```solidity
// AnchoredPriceProvider.sol lines 77, 150-151
uint256 public immutable MAX_REF_STALENESS;
...
if (_maxRefStaleness > 7 days) revert MaxRefStalenessOutOfBounds();
MAX_REF_STALENESS = _maxRefStaleness;
```

There is no mechanism to assign a different staleness bound to `baseFeedId` vs `quoteFeedId`.

### Impact Explanation

In synthetic ratio mode the anchor mid is `price(baseFeedId) / price(quoteFeedId)`. Every swap's bid/ask band is derived from this anchor. If `MAX_REF_STALENESS` is set to the slower feed's heartbeat (e.g. 24 h for an AMPL-style quote feed), the faster feed (e.g. 1 h heartbeat BTC/USD base feed) can be up to 23 hours stale while still passing the staleness check. A trader who knows the real current price can buy underpriced or sell overpriced against the stale anchor, extracting value from the pool's LPs. This is a direct bad-price execution impact: a stale bid/ask quote reaches a pool swap.

### Likelihood Explanation

The `AnchoredPriceProvider` factory is permissionless — any address can deploy a provider. Synthetic ratio mode is an explicitly documented feature ("`quoteFeedId` set: synthetic ratio quoting"). Chainlink Data Streams and Pyth Lazer feeds for different assets routinely have different update frequencies. A deployer who sets `MAX_REF_STALENESS` to accommodate the slower feed's heartbeat (the natural choice to avoid unnecessary halts) silently accepts stale prices from the faster feed. No privileged action is required after deployment; the misconfiguration is baked in at construction.

### Recommendation

Add a separate immutable for the quote-leg staleness bound:

```solidity
uint256 public immutable MAX_QUOTE_REF_STALENESS;
```

Pass it as a distinct constructor parameter and apply it only when reading `quoteFeedId` inside `_readLeg` (or inline the check in `_getBidAndAskPrice`). The factory's `createAnchoredPriceProvider` call should require both values to be provided and validated independently. This mirrors the recommendation in the external report: per-feed thresholds matched to each feed's actual heartbeat.

### Proof of Concept

1. Deploy `AnchoredPriceProvider` in synthetic mode: `baseFeedId` = BTC/USD (1 h heartbeat), `quoteFeedId` = AMPL/USD (48 h heartbeat), `MAX_REF_STALENESS` = 48 h (set to avoid halting the slow feed).
2. The BTC/USD pusher stops updating for 23 hours (e.g. network congestion). The stored BTC/USD price is now 23 hours old.
3. `_readLeg(baseFeedId)` checks `(block.timestamp - refTime) > MAX_REF_STALENESS` → `23 h > 48 h` → **false** → staleness check passes.
4. The pool quotes a synthetic BTC/ETH price derived from a 23-hour-old BTC/USD price.
5. A trader observes the real BTC price has moved 10% since the stale snapshot. They swap against the pool at the stale anchor price, extracting the difference from LP funds. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

```text
    function _readLeg(bytes32 feedId)
        internal returns (uint256 mid, uint256 spreadBps, uint256 refTime, bool ok)
    {
        (mid, spreadBps, , refTime) = IPricedOracle(address(offchainOracle)).price(feedId, msg.sender);

        // Stale reference → not ok. Clamping to a stale anchor is the one false-safety case.
        if (_isStale(refTime, block.timestamp, MAX_REF_STALENESS)) return (mid, spreadBps, refTime, false);

        // Basic validity — mid positive, spreadBps not the stalled/off-hours marker (the Chainlink oracle
        // writes spreadBps = ORACLE_BPS when an RWA market is closed).
        if (mid == 0 || spreadBps >= ORACLE_BPS) return (mid, spreadBps, refTime, false);

        // Per-leg price guard.
        (uint128 guardMin, uint128 guardMax) = offchainOracle.priceGuard(feedId);
        guardMax = guardMax == 0 ? type(uint128).max : guardMax;
        if (mid < guardMin || mid > guardMax) return (mid, spreadBps, refTime, false);

        ok = true;
    }
```
