### Title
Staker Can Instantly Increase Commission to Maximum via Atomic `set_commission_commitment` + `set_commission` with No Enforced Delay — (File: src/staking/staking.cairo)

---

### Summary
A staker can call `set_commission_commitment` with `max_commission = 10000` (100%) and `expiration_epoch = current_epoch + 1`, then immediately call `set_commission(commission: 10000)` in the same block. There is no enforced delay between setting the commitment and exercising it. This allows a staker to atomically raise their commission from 0% to 100%, redirecting all future pool rewards to themselves before delegators can react and exit. The codebase itself acknowledges the gap: *"Current commission increase safeguards still allow for sudden commission changes."*

---

### Finding Description

The `set_commission_commitment` function enforces only that `expiration_epoch > current_epoch` — meaning the minimum valid commitment window is a single epoch boundary away. It does **not** require any delay before the commitment can be exercised. [1](#0-0) 

Immediately after setting the commitment, the staker calls `set_commission`. Inside `update_commission`, the only check when a commitment is active is that `commission <= commitment.max_commission` and `commission != old_commission`. The new commission is written to storage instantly with no time-lock. [2](#0-1) 

The code comment directly above `set_commission_commitment` acknowledges the gap: [3](#0-2) 

When rewards are subsequently calculated (via `update_rewards` or `attest`), the commission is read live from storage: [4](#0-3) 

So any rewards calculated after the commission change — including rewards for the current block/epoch — are split at the new (100%) rate, sending the entire pool share to the staker and zero to delegators.

---

### Impact Explanation

Delegators who joined a pool advertising 0% commission can have 100% commission applied to all future rewards in a single block. Because the commission is deducted at the staking contract level before tokens are forwarded to the pool contract, delegators receive zero yield from that point forward. Any delegator wishing to exit must still wait for the `exit_wait_window`, during which they continue to earn 0% of rewards. This constitutes **theft of unclaimed yield** — the yield that would have accrued to delegators is instead captured entirely by the staker.

**Impact: High — Theft of unclaimed yield.**

---

### Likelihood Explanation

The attack requires no special privileges beyond being a staker with an open pool. The two transactions (`set_commission_commitment` then `set_commission`) can be submitted in the same block. A rational attacker would:
1. Advertise a 0% commission pool to attract large delegations.
2. Wait until significant delegated capital has accumulated.
3. Execute the two-step atomic commission increase to 100%.
4. Call `update_rewards` to crystallize the theft.

The attack is cheap, deterministic, and requires no external dependencies.

---

### Recommendation

Enforce a minimum delay between setting a commitment and exercising it to increase commission. For example, require that `set_commission` cannot raise commission above the previous value until at least `K` epochs after the commitment was set (where `K` is the existing protocol constant for balance propagation delay). This gives delegators a guaranteed observation window to call `exit_intent` before the higher commission takes effect.

---

### Proof of Concept

```
Epoch N, Block B:
  staker calls set_commission_commitment(max_commission: 10000, expiration_epoch: N+1)
  // Passes: expiration_epoch (N+1) > current_epoch (N) ✓

Epoch N, Block B (same block):
  staker calls set_commission(commission: 10000)
  // Passes: active commitment exists, 10000 <= 10000, 10000 != old_commission ✓
  // commission written to storage immediately

Epoch N, Block B+1:
  anyone calls update_rewards(staker_address)
  // calculate_staker_pools_rewards reads commission = 10000 (100%)
  // split_rewards_with_commission: commission_rewards = pool_rewards_including_commission
  //                                pool_rewards = 0
  // Delegators receive 0 STRK; staker receives 100% of pool share
```

Delegators have no on-chain mechanism to react between Block B and Block B+1. Their accumulated future yield is permanently redirected to the staker. [5](#0-4) [6](#0-5)

        fn set_commission_commitment(
            ref self: ContractState, max_commission: Commission, expiration_epoch: Epoch,
        ) {
            self.general_prerequisites();
            assert!(max_commission <= COMMISSION_DENOMINATOR, "{}", Error::COMMISSION_OUT_OF_RANGE);
            let staker_address = get_caller_address();
            let staker_info = self.internal_staker_info(:staker_address);
            assert!(staker_info.unstake_time.is_none(), "{}", Error::UNSTAKE_IN_PROGRESS);
            let staker_pool_info = self.staker_pool_info.entry(staker_address);
            assert!(staker_pool_info.has_pool(), "{}", Error::MISSING_POOL_CONTRACT);
            let current_epoch = self.get_current_epoch();
            if let Option::Some(commission_commitment) = staker_pool_info
                .commission_commitment
                .read() {
                assert!(
                    !self.is_commission_commitment_active(:commission_commitment),
                    "{}",
                    Error::COMMISSION_COMMITMENT_EXISTS,
                );
            }
            // Staker must have a commission since it has a pool.
            let current_commission = staker_pool_info.commission();
            assert!(current_commission <= max_commission, "{}", Error::MAX_COMMISSION_TOO_LOW);
            assert!(expiration_epoch > current_epoch, "{}", Error::EXPIRATION_EPOCH_TOO_EARLY);
            assert!(
                expiration_epoch - current_epoch <= self.get_epoch_info().epochs_in_year(),
                "{}",
                Error::EXPIRATION_EPOCH_TOO_FAR,
            );
            let commission_commitment = CommissionCommitment { max_commission, expiration_epoch };
            staker_pool_info.commission_commitment.write(Option::Some(commission_commitment));
            self
                .emit(
                    Events::CommissionCommitmentSet {
                        staker_address, max_commission, expiration_epoch,
                    },
                );

        fn update_commission(
            ref self: ContractState,
            staker_address: ContractAddress,
            staker_pool_info: StoragePath<Mutable<InternalStakerPoolInfoV2>>,
            old_commission: Commission,
            commission: Commission,
        ) {
            if let Option::Some(commission_commitment) = staker_pool_info
                .commission_commitment
                .read() {
                if self.is_commission_commitment_active(:commission_commitment) {
                    assert!(
                        commission <= commission_commitment.max_commission,
                        "{}",
                        Error::INVALID_COMMISSION_WITH_COMMITMENT,
                    );
                    assert!(commission != old_commission, "{}", Error::INVALID_SAME_COMMISSION);
                } else {
                    assert!(
                        commission < old_commission, "{}", Error::COMMISSION_COMMITMENT_EXPIRED,
                    );
                }
            } else {
                assert!(commission < old_commission, "{}", Error::INVALID_COMMISSION);
            }

            // Update commission in storage.
            staker_pool_info.commission.write(Option::Some(commission));

            // Emit event.
            self
                .emit(
                    Events::CommissionChanged {
                        staker_address, old_commission, new_commission: commission,
                    },
                );
        }
