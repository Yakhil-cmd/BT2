### Title
Unauthorized Global Contract Code Injection via Missing Predecessor Authorization in `action_deploy_global_contract` — (File: runtime/runtime/src/global_contracts.rs)

### Summary
Any unprivileged account can overwrite the global WASM contract stored under any other account's ID by sending a `DeployGlobalContractAction` with `GlobalContractDeployMode::AccountId` to the victim's account. No authorization check verifies that the action sender is the account owner. This breaks the ownership invariant of account-ID-keyed global contracts and allows an attacker to inject arbitrary WASM code that executes on behalf of every account that references the victim's global contract identity.

### Finding Description

`action_deploy_global_contract` in `runtime/runtime/src/global_contracts.rs` accepts the receipt receiver's `account_id` and, when `deploy_mode == GlobalContractDeployMode::AccountId`, stores the supplied WASM under `TrieKey::GlobalContractCode { identifier: GlobalContractCodeIdentifier::AccountId(account_id) }`. There is no check that the receipt's `predecessor_id` equals `account_id`. [1](#0-0) 

Inside `initiate_distribution`, the identifier is derived solely from the receipt receiver: [2](#0-1) 

Because NEAR allows any account to send a receipt with any action to any other account — either via a direct transaction where `signer_id ≠ receiver_id`, or via a cross-contract promise using `promise_batch_action_deploy_global_contract_by_account_id` — an attacker can target `victim.near` as the receipt receiver and supply attacker-controlled WASM. The action validation for `DeployGlobalContractAction` enforces only a size limit; there is no ownership or predecessor check: [3](#0-2) 

The nonce mechanism in `apply_distribution_current_shard` only prevents stale distribution receipts from overwriting newer ones; it does not prevent a fresh attacker deployment (which receives a higher nonce) from overwriting the victim's deployment: [4](#0-3) 

The `GlobalContractDeployMode::AccountId` design explicitly states "This allows the **owner** to update the contract for all its users": [5](#0-4) 

The ownership invariant is stated but never enforced in production code.

### Impact Explanation

Every account that has called `UseGlobalContract(AccountId("victim.near"))` stores `AccountContract::GlobalByAccount("victim.near")` in its account state: [6](#0-5) 

At function-call time, `RuntimeContractIdentifier::resolve` looks up `TrieKey::GlobalContractCode { identifier: AccountId("victim.near") }` and executes whatever WASM is stored there. After the attack, that WASM is the attacker's code. Every call to every account referencing `victim.near`'s global contract executes attacker-controlled code — enabling token theft, unauthorized state writes, or complete account compromise across an unbounded set of victims.

Additionally, the storage cost of the attacker's code is deducted from the victim's account balance: [7](#0-6) 

### Likelihood Explanation

The attack requires only a valid access key on the attacker's own account and sufficient balance on the victim's account to cover storage costs. No validator, relayer, or operator privilege is needed. The attack is a single signed transaction (`signer_id = attacker`, `receiver_id = victim`, `actions = [DeployGlobalContractAction { code: malicious, deploy_mode: AccountId }]`). Accounts that have deployed global contracts (a prerequisite for having users reference them) necessarily hold enough balance to cover storage, satisfying the only precondition.

### Recommendation

Add a predecessor authorization check inside `action_deploy_global_contract`: when `deploy_mode == GlobalContractDeployMode::AccountId`, verify that `receipt.predecessor_id() == account_id`. If the predecessor is not the account itself, return `ActionErrorKind::ActorNoPermission`. Alternatively, restrict `DeployGlobalContractAction` at the action-validation layer to self-receipts only (predecessor == receiver), mirroring the implicit ownership model described in the documentation.

### Proof of Concept

1. `victim.near` deploys a legitimate global contract:
   ```
   Transaction { signer_id: "victim.near", receiver_id: "victim.near",
     actions: [DeployGlobalContractAction { code: <legitimate>, deploy_mode: AccountId }] }
   ```
   Stored at `TrieKey::GlobalContractCode { identifier: AccountId("victim.near") }`.

2. `user.near` opts in:
   ```
   UseGlobalContractAction { contract_identifier: AccountId("victim.near") }
   ```
   `user.near`'s account now holds `AccountContract::GlobalByAccount("victim.near")`.

3. Attacker sends:
   ```
   Transaction { signer_id: "attacker.near", receiver_id: "victim.near",
     actions: [DeployGlobalContractAction { code: <malicious>, deploy_mode: AccountId }] }
   ```
   This transaction is valid: `attacker.near` has a full-access key; `victim.near` exists and has balance. No authorization check blocks it.

4. `action_deploy_global_contract` executes on `victim.near`'s account, deducts storage cost from `victim.near`'s balance, and calls `initiate_distribution` with `account_id = "victim.near"` and `code = <malicious>`.

5. The distribution receipt propagates to all shards and overwrites `TrieKey::GlobalContractCode { identifier: AccountId("victim.near") }` with `<malicious>`.

6. Every subsequent call to `user.near` (and any other account referencing `victim.near`'s global contract) now executes the attacker's WASM — the exact impersonation analog: the attacker's code runs under the trusted identity of `victim.near`, invisible to the users who opted in.

pub(crate) fn action_deploy_global_contract(
    state_update: &mut TrieUpdate,
    account: &mut Account,
    account_id: &AccountId,
    apply_state: &ApplyState,
    deploy_contract: &DeployGlobalContractAction,
    result: &mut ActionResult,
) -> Result<(), RuntimeError> {
    let _span = tracing::debug_span!(target: "runtime", "action_deploy_global_contract").entered();

    let storage_cost = apply_state
        .config
        .fees
        .storage_usage_config
        .global_contract_storage_amount_per_byte
        .saturating_mul(deploy_contract.code.len() as u128);
    let Some(updated_balance) = account.amount().checked_sub(storage_cost) else {
        result.result = Err(ActionErrorKind::LackBalanceForState {
            account_id: account_id.clone(),
            amount: storage_cost,
        }
        .into());
        return Ok(());
    };
    result.tokens_burnt =
        result.tokens_burnt.checked_add(storage_cost).ok_or(IntegerOverflowError)?;
    account.set_amount(updated_balance);

    initiate_distribution(
        state_update,
        account_id.clone(),
        deploy_contract.code.clone(),
        &deploy_contract.deploy_mode,
        apply_state.shard_id,
        result,
    )?;

    Ok(())
}
