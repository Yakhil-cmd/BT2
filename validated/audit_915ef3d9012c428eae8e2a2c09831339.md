### Title
Reverted Invoke Transaction Trace Reports Zero Execute-Phase Resources in `execution_resources` - (`crates/apollo_rpc/src/v0_8/execution.rs`)

---

### Summary

When an Invoke transaction reverts, the `execution_resources` field of the RPC `InvokeTransactionTrace` is computed as `validate + 0 + fee_transfer`, silently dropping all gas consumed by the `__execute__` phase. The fee is still charged (correctly, by the blockifier), but the trace's `execution_resources` field is structurally lower than the actual resources billed, making it an authoritative-looking wrong value returned by `starknet_getTransactionTrace`, `starknet_traceBlockTransactions`, and `starknet_simulateTransactions`.

---

### Finding Description

In `crates/apollo_rpc/src/v0_8/execution.rs`, the `From<(ExecutionTransactionTrace, ThinStateDiff)> for TransactionTrace` implementation matches on `execute_invocation`: [1](#0-0) 

When the execute invocation is `Err`, `execute_execution_resources` is set to `ExecutionResources::default()` (all-zero). The total `execution_resources` is then: [2](#0-1) 

This means for any reverted Invoke with a non-`None` `fee_transfer_invocation`, the reported `execution_resources` = `validate_resources + 0 + fee_transfer_resources`.

The root cause is upstream: `crates/apollo_rpc_execution/src/objects.rs` builds the `ExecutionTransactionTrace::Invoke` from `TransactionExecutionInfo` and, when `revert_error` is `Some`, sets `execute_invocation = Err(revert_reason)` with no call info attached: [3](#0-2) 

The `ExecutionTransactionTrace` carries no `receipt.resources` field — only the individual invocation call infos. So the execute-phase resources consumed before the revert are permanently lost from the trace data structure.

Meanwhile, the blockifier correctly accounts for those resources in the receipt. When a transaction reverts, `execution_steps_consumed` (steps used during the failed execute) is passed into `get_revert_receipt`: [4](#0-3) 

The fee is charged based on this receipt. The trace's `execution_resources` field is not.

---

### Impact Explanation

The `execution_resources` field in `InvokeTransactionTrace` is described in the OpenRPC spec as "the resources consumed by the transaction, includes both computation and data." [5](#0-4) 

For a reverted transaction, the execute phase ran and consumed real resources (the fee was charged for them). The trace reports a value that is strictly lower than what was actually billed. Any client or tool that reads `execution_resources` from the trace to estimate fees for similar transactions will systematically underestimate costs. This is a **High** impact issue: the RPC returns an authoritative-looking wrong value for a well-defined, publicly observable field.

---

### Likelihood Explanation

Any unprivileged user can trigger this by submitting an Invoke transaction that reverts (e.g., calling a contract that panics). This is a routine occurrence on mainnet. The condition `execute_invocation = Err AND fee_transfer_invocation = Some` is the normal outcome of any reverted transaction where fee charging is enabled.

---

### Recommendation

The `execution_resources` field should be sourced from `TransactionExecutionInfo.receipt.resources` (which already includes reverted-execute steps via `n_reverted_steps`) rather than reconstructed by summing individual invocation call infos. The `ExecutionTransactionTrace` (or the conversion path) needs to carry the receipt resources so the RPC layer can populate `execution_resources` correctly for reverted transactions.

---

### Proof of Concept

Construct an `ExecutionTransactionTrace::Invoke` where:
- `execute_invocation = ExecutionFunctionInvocationResult::Err("revert")`
- `validate_invocation = Some(invocation_with_nonzero_resources)`
- `fee_transfer_invocation = Some(invocation_with_nonzero_resources)`

Call `TransactionTrace::from((trace, state_diff))` and inspect the resulting `InvokeTransactionTrace::execution_resources`. It will equal `validate_resources + fee_transfer_resources` with the execute-phase contribution being exactly zero, even though a real reverted transaction would have consumed non-zero execute resources that were billed. [6](#0-5)
