### Title
Declare Transaction `l2_gas.max_amount` Cap Bypassed in Stateless Validation — (`crates/apollo_gateway/src/stateless_transaction_validator.rs`)

### Summary
`StatelessTransactionValidator::validate_resource_bounds` explicitly skips the `max_l2_gas_amount` upper-bound check for `RpcTransaction::Declare` variants. No downstream layer (stateful gateway validator or blockifier pre-validation) re-applies this cap. A Declare transaction with `l2_gas.max_amount` arbitrarily exceeding the operator-configured `max_l2_gas_amount` is admitted through the gateway's stateless admission gate.

### Finding Description

In `crates/apollo_gateway/src/stateless_transaction_validator.rs`, `validate_resource_bounds` contains the following branch:

```rust
// TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
if let RpcTransaction::Declare(_) = tx {
} else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh { … });
}
``` [1](#0-0) 

The empty `if let RpcTransaction::Declare(_) = tx {}` arm is a deliberate no-op: the `max_l2_gas_amount` check is only reached for non-Declare transactions. The TODO comment in the code itself acknowledges the gap. The existing test `valid_l2_gas_amount_on_declare` explicitly asserts `Ok(())` for a Declare tx with `max_amount=200` when `max_l2_gas_amount=100`: [2](#0-1) 

**Downstream checks do not compensate:**

- The stateful gateway validator's `validate_resource_bounds` only checks `l2_gas_price` against the previous block's price threshold; it never checks `max_amount` against any cap: [3](#0-2) 

- For Declare transactions, the blockifier's `StatefulValidator::perform_validations` routes to `self.execute(tx)` (full execution), not the Invoke path that calls `perform_pre_validation_stage` directly: [4](#0-3) 

- The blockifier's `check_fee_bounds` (called during execution's pre-validation stage) only checks that `minimal_gas_amount <= resource_bounds.max_amount` (a lower-bound sufficiency check). It does not enforce any operator-configured upper-bound cap on `max_amount`: [5](#0-4) 

- `verify_can_pay_committed_bounds` checks that the account balance covers `max_amount × max_price_per_unit`. This is a balance check, not a policy cap — an attacker with sufficient balance bypasses it entirely: [6](#0-5) 

The default production `max_l2_gas_amount` is `1,210,000,000`: [7](#0-6) 

An attacker can submit a Declare transaction with `l2_gas.max_amount = u64::MAX` (or any value above the cap). As long as the account balance covers `u64::MAX × max_price_per_unit` (or a more modest overage), the transaction passes all gateway checks and enters the mempool.

### Impact Explanation

The `max_l2_gas_amount` cap is the gateway's primary admission control for L2 gas amount. Bypassing it for Declare transactions means:

1. Declare transactions with declared L2 gas amounts exceeding the block's practical capacity can be admitted to the mempool.
2. The operator's resource-bound policy is not enforced uniformly across transaction types.
3. Concrete corrupted admission value: a Declare tx with `l2_gas.max_amount > max_l2_gas_amount` returns `Ok(())` from `StatelessTransactionValidator::validate` when it should return `Err(MaxGasAmountTooHigh)`.

### Likelihood Explanation

Exploitability is straightforward: any unprivileged user can craft a V3 Declare transaction with an arbitrarily large `l2_gas.max_amount`. The only natural filter is `verify_can_pay_committed_bounds`, which is bypassed if the attacker holds sufficient STRK balance. For moderate overages (e.g., `2× max_l2_gas_amount`), the required balance is well within reach of a funded account.

### Recommendation

Remove the Declare-specific exemption and apply the same `max_l2_gas_amount` check uniformly:

```rust
if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
        gas_amount: resource_bounds.l2_gas.max_amount,
        max_gas_amount: self.config.max_l2_gas_amount,
    });
}
```

If Declare transactions legitimately require a higher L2 gas ceiling (e.g., for compilation), introduce a separate `max_l2_gas_amount_declare` config field rather than removing the check entirely.
