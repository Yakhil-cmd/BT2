### Title
`payable.transfer()` in `WSEI.withdraw()` Can Permanently Freeze Funds for Smart Contract Callers — (File: `contracts/src/WSEI.sol`)

---

### Summary

`WSEI.sol` is a protocol-level Wrapped SEI contract (compiled and embedded as on-chain artifacts in `x/evm/artifacts/wsei/`) whose `withdraw()` function uses Solidity's `payable(msg.sender).transfer(wad)`. This forwards only 2300 gas to the recipient. Any smart contract whose `receive`/`fallback` function consumes more than 2300 gas will have its withdrawal permanently revert, freezing the underlying SEI inside the WSEI contract with no recovery path.

---

### Finding Description

`WSEI.sol` is not a test contract. Its ABI and bytecode are compiled and embedded as Go artifacts: [1](#0-0) 

The `withdraw()` function at line 30 uses `.transfer()`: [2](#0-1) 

`payable(msg.sender).transfer(wad)` hard-caps the gas forwarded to the recipient at 2300. This is insufficient for:

- Smart contracts that emit events in their `receive`/`fallback`
- Smart contracts that update storage in their `receive`/`fallback`
- Any contract called through a proxy (proxy dispatch overhead alone can exceed 2300 gas)
- Multisig wallets (e.g. Gnosis Safe) used as DeFi treasury addresses

When the transfer reverts, the entire `withdraw()` call reverts. The caller's WSEI balance is unchanged (the `balanceOf` decrement at line 29 is rolled back), but the caller has no alternative withdrawal mechanism — there is no `call`-based fallback path.

---

### Impact Explanation

Any smart contract that:
1. Deposits native SEI into WSEI (via `deposit()` or direct ETH send to the `fallback`/`receive`)
2. Later calls `withdraw()`

…will have its withdrawal silently fail if its own `receive`/`fallback` exceeds 2300 gas. The SEI is effectively frozen: the WSEI balance is intact but unwithdrawable. This maps to **fund freeze** in the Sei bounty scope.

Severity: **Medium** (fund freeze; amount depends on depositor).

---

### Likelihood Explanation

- Multisig wallets, proxy-based contracts, and DeFi vaults routinely exceed 2300 gas in their receive hooks.
- No special privilege is required — any unprivileged smart contract that interacts with WSEI is affected.
- The pattern is well-known and has been exploited in production (post-Istanbul EIP-1884 raised `SLOAD` cost, breaking many 2300-gas assumptions).

---

### Recommendation

Replace `.transfer()` with a low-level `.call{value: wad}("")` guarded by a reentrancy check, following the checks-effects-interactions pattern (balance is already decremented before the external call, so reentrancy is mitigated):

```solidity
// contracts/src/WSEI.sol
function withdraw(uint wad) public {
    require(balanceOf[msg.sender] >= wad);
    balanceOf[msg.sender] -= wad;          // effect before interaction ✓
    (bool success, ) = payable(msg.sender).call{value: wad}("");
    require(success, "WSEI: ETH transfer failed");
    emit Withdrawal(msg.sender, wad);
}
```

---

### Proof of Concept

1. Deploy a contract `Vault` whose `receive()` writes to storage (costs >2300 gas):
   ```solidity
   receive() external payable { lastReceived = msg.value; } // SSTORE > 2300 gas
   ```
2. `Vault` calls `WSEI.deposit{value: 1 ether}()` — succeeds.
3. `Vault` calls `WSEI.withdraw(1 ether)` — reverts at `payable(msg.sender).transfer(wad)` because `Vault.receive()` exceeds 2300 gas.
4. `Vault`'s WSEI balance remains 1 ether; native SEI is permanently locked in WSEI. [2](#0-1) [3](#0-2)
