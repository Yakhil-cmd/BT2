### Title
Unpermissioned `update_rewards` with `disable_rewards: true` Allows Any Caller to Permanently Deny Stakers' Block Rewards - (File: `src/staking/staking.cairo`)

---

### Summary

`update_rewards` in the `Staking` contract is callable by any unprivileged address and accepts a `disable_rewards: bool` parameter. The global `last_reward_block` variable is written unconditionally before the `disable_rewards` guard, meaning an attacker can call `update_rewards(any_valid_staker, disable_rewards: true)` to consume the single per-block reward slot without distributing any rewards. Any subsequent legitimate call in the same block reverts with `REWARDS_ALREADY_UPDATED`, permanently destroying that block's rewards for all stakers.

---

### Finding Description

`update_rewards` is gated only by `general_prerequisites()`, which checks that the contract is not paused and the caller is non-zero. There is no check that the caller is the staker, the attestation contract, or any other authorized party. [1](#0-0) 

The function immediately writes the current block number to the **global** `last_reward_block` before evaluating `disable_rewards`: [2](#0-1) 

If `disable_rewards` is `true`, execution returns early with no rewards distributed, but `last_reward_block` is already committed: [3](#0-2) 

Because `last_reward_block` is a single global slot (not per-staker), the guard `current_block_number > self.last_reward_block.read()` ensures only **one** successful call per block is possible: [4](#0-3) 

An attacker who calls `update_rewards(any_valid_staker, disable_rewards: true)` first in a block therefore permanently prevents any staker from receiving rewards in that block.

The `IStakingRewardsManager` interface exposes this function with no documented caller restriction: [5](#0-4) 

---

### Impact Explanation

Block rewards that are skipped because `disable_rewards: true` was used are **permanently lost**. The `calculate_block_rewards` helper caches rewards per epoch and does not retroactively compensate for skipped blocks: [6](#0-5) 

A sustained attacker calling `update_rewards(..., disable_rewards: true)` at the start of every block would cause **permanent freezing of all consensus-era block rewards** for every staker in the protocol. This matches the allowed impact: **High — Permanent freezing of unclaimed yield**.

---

### Likelihood Explanation

The attack requires no special privilege, no capital at risk, and no leaked key. The attacker only pays gas. On Starknet, transaction ordering within a block is controlled by the sequencer; an attacker who submits a transaction early in a block (or who is the sequencer, or who bribes the sequencer) can reliably front-run the legitimate `update_rewards` call. Even without mempool visibility, the attacker can simply call `update_rewards(..., disable_rewards: true)` once per block continuously, which is cheap and sufficient to deny all block rewards indefinitely.

---

### Recommendation

1. **Restrict the caller**: Add an access-control check so that `update_rewards` can only be called by a whitelisted address (e.g., the attestation contract, a designated rewards distributor, or the staker themselves).
2. **Separate the `disable_rewards` path**: If `disable_rewards: true` is a legitimate operational mode (e.g., for migration), gate it behind a privileged role (e.g., `only_app_governor`) rather than exposing it to arbitrary callers.
3. **Write `last_reward_block` only after rewards are confirmed**: Move the `self.last_reward_block.write(current_block_number)` call to after the `disable_rewards` guard so that a no-op call does not consume the per-block slot.

---

### Proof of Concept

```
// Attacker calls at the start of every block:
staking.update_rewards(
    staker_address: any_valid_staker,  // any staker that passes is_staker_active
    disable_rewards: true,
);
// Result: last_reward_block = current_block_number, no rewards distributed.

// Legitimate staker (or consensus mechanism) calls later in the same block:
staking.update_rewards(
    staker_address: legitimate_staker,
    disable_rewards: false,
);
// Reverts: REWARDS_ALREADY_UPDATED (current_block_number == last_reward_block)
// Block rewards for legitimate_staker are permanently lost.
```

Repeating this every block causes all consensus-era block rewards to be permanently frozen with only gas cost to the attacker.
