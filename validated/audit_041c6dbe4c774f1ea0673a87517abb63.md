### Title
`estimate_minimal_gas_vector` Underestimates DA Gas for Declare V3 Transactions, Allowing `check_fee_bounds` to Accept Transactions with Insufficient `l1_data_gas` Bounds — (`crates/blockifier/src/fee/gas_usage.rs`)

---

### Summary

`estimate_minimal_gas_vector` sets `n_compiled_class_hash_updates: 0` for all Declare transactions (acknowledged by a `TODO` comment saying it should be `1`). Under KZG DA, this causes `check_fee_bounds` to compute a `l1_data_gas` minimum that is **64 blob gas units short** of the actual DA cost. A Declare V3 transaction with `AllResources` bounds whose `l1_data_gas.max_amount` falls in the gap passes `check_fee_bounds` and is admitted by the gateway, violating the invariant that admission rejects any transaction whose resource bounds cannot cover the minimal gas.

---

### Finding Description

In `estimate_minimal_gas_vector`, the `StateChangesCount` for `Transaction::Declare` is:

```rust
Transaction::Declare(_) => StateChangesCount {
    n_storage_updates: 1,
    n_class_hash_updates: 0,
    // TODO(Yoni): BLOCKIFIER-RESET: should be 1.
    n_compiled_class_hash_updates: 0,
    n_modified_contracts: 1,
},
``` [1](#0-0) 

The actual DA state changes for a V2/V3 Declare transaction include `n_compiled_class_hash_updates: 1`, as confirmed by the production test helper:

```rust
} else if version == TransactionVersion::TWO || version == TransactionVersion::THREE {
    StateChangesCount {
        n_storage_updates: 1,
        n_modified_contracts: 1,
        n_compiled_class_hash_updates: 1, // Also set compiled class hash.
        ..StateChangesCount::default()
    }
``` [2](#0-1) 

Each `n_compiled_class_hash_updates` contributes `2` field elements to the onchain data segment: [3](#0-2) 

Under KZG DA (`use_kzg_da = true`), `get_da_gas_cost` converts this to blob gas at `DATA_GAS_PER_FIELD_ELEMENT = DATA_GAS_PER_BLOB / FIELD_ELEMENTS_PER_BLOB = 131072 / 4096 = 32` blob gas units per field element: [4](#0-3) 

The underestimate is therefore `2 × 32 = 64 l1_data_gas units`.

`check_fee_bounds` calls `estimate_minimal_gas_vector` and for `AllResources` bounds compares `minimal_gas_amount_vector.l1_data_gas` against `l1_data_gas_resource_bounds.max_amount`: [5](#0-4) 

A transaction with `l1_data_gas.max_amount` set to the underestimated value (i.e., the correct minimum minus 64) passes this check.

The gateway's `run_validate_entry_point` sets `charge_fee = enforce_fee(executable_tx, only_query)` (true for non-zero bounds) and calls `blockifier_validator.validate(account_tx)`: [6](#0-5) 

`StatefulValidator::perform_validations` **fully executes** Declare transactions (to check for re-declaration), meaning `perform_pre_validation_stage` → `check_fee_bounds` is called with the wrong minimum: [7](#0-6) 

When the transaction subsequently reverts post-execution (actual DA gas > bounds), `run_revertible` returns `Ok(ValidateExecuteCallInfo::new_reverted(...))` — a revert is not an error — so `perform_validations` returns `Ok(())` and the gateway admits the transaction.

---

### Impact Explanation

**Impact: High — Mempool/gateway admission accepts an invalid transaction before sequencing.**

A Declare V3 transaction with `l1_data_gas.max_amount` set to `estimate_minimal_gas_vector.l1_data_gas` (64 blob gas units below the true minimum) passes `check_fee_bounds` and is admitted to the mempool. The concrete corrupted admission value is the `l1_data_gas` minimum bound: the check accepts `max_amount = M` when it should require `max_amount ≥ M + 64`. The transaction will revert on-chain (actual DA cost exceeds bounds), but it is still sequenced and included in a block.

---

### Likelihood Explanation

The bug is self-documented with a `TODO` comment. It is triggered by any Declare V3 transaction with `AllResources` bounds on a KZG DA block where `l1_data_gas.max_amount` is set to exactly the underestimated minimum. An attacker can compute this value deterministically from public constants and craft such a transaction with no privileges required.

---

### Recommendation

Change `n_compiled_class_hash_updates` from `0` to `1` in the `Transaction::Declare` arm of `estimate_minimal_gas_vector`:

```rust
Transaction::Declare(_) => StateChangesCount {
    n_storage_updates: 1,
    n_class_hash_updates: 0,
    n_compiled_class_hash_updates: 1, // was 0; Declare V2/V3 writes compiled_class_hash
    n_modified_contracts: 1,
},
``` [1](#0-0) 

Add a regression test that constructs a Declare V3 transaction with `use_kzg_da = true`, calls `estimate_minimal_gas_vector`, and asserts that `l1_data_gas ≥ get_da_gas_cost(&StateChangesCount{n_compiled_class_hash_updates:1,...}, true).l1_data_gas`.

---

### Proof of Concept

**Numeric derivation (no code needed):**

```
DATA_GAS_PER_FIELD_ELEMENT = 131072 / 4096 = 32

Actual DA for Declare V3 (KZG):
  StateChangesCount { n_storage_updates:1, n_modified_contracts:1, n_compiled_class_hash_updates:1 }
  onchain_data_segment_length = 1*2 + 0 + 1*2 + 1*2 = 8 field elements
  l1_data_gas = 8 * 32 = 256

Estimated DA (bug, n_compiled_class_hash_updates=0):
  onchain_data_segment_length = 1*2 + 0 + 0 + 1*2 = 6 field elements
  l1_data_gas = 6 * 32 = 192

Gap = 256 - 192 = 64 blob gas units
```

A Declare V3 transaction with `AllResources { l1_data_gas: { max_amount: 192, ... }, ... }` passes `check_fee_bounds` (192 ≥ 192) but the actual execution requires 256 blob gas units, exceeding the bound. The gateway returns `Ok(())` (revert is not an error), admitting the transaction.
