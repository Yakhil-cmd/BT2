### Title
`protocolUnpausePool` sets pause level to 1 (admin-paused) instead of 0 (active), leaving pool swap-blocked after protocol unpause — (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`MetricOmmPoolFactory.protocolUnpausePool` calls `setPause(1)` instead of `setPause(0)`. Because the pool's `whenNotPaused` guard reverts on any non-zero pause level, every pool that was active before a protocol pause remains permanently swap-blocked after the protocol owner "unpauses" it, until the pool admin separately calls `unpausePool`.

---

### Finding Description

The pause-level semantics are:

| Level | Meaning |
|---|---|
| 0 | Active |
| 1 | Paused by admin |
| 2 | Paused by protocol |

`protocolPausePool` accepts pools at level 0 **or** 1 and moves them to level 2: [1](#0-0) 

`protocolUnpausePool` is supposed to undo that, but it hard-codes the target as **1** (admin-paused) instead of **0** (active): [2](#0-1) 

The pool's `_checkNotPaused` guard rejects any non-zero level: [3](#0-2) 

`swap` is gated by `whenNotPaused`: [4](#0-3) 

So after `protocolUnpausePool` executes, `pauseLevel == 1 != 0`, and every `swap` call reverts with `PoolPaused()`. The pool is still non-functional for traders.

The parallel to the seed bug is exact: the wrong value (storage-default `1` instead of the correct target `0`) is written during a state-transition that is supposed to restore normal operation, causing the intended invariant (pool active after unpause) to be permanently violated until a second privileged actor intervenes.

---

### Impact Explanation

Core swap functionality is broken for every pool that was at level 0 when the protocol paused it. After `protocolUnpausePool` is called, traders cannot execute any swap. Liquidity providers cannot realise gains or rebalance via swaps. The pool is economically frozen until the pool admin independently discovers the residual pause and calls `unpausePool`. There is no on-chain signal that the pool is still paused after the protocol "unpause."

---

### Likelihood Explanation

High. The bug fires on every invocation of `protocolUnpausePool` for a pool that was active (level 0) at the time of the protocol pause — the common case. The protocol owner has no reason to suspect the pool remains paused after calling the unpause function.

---

### Recommendation

```diff
  function protocolUnpausePool(address pool) external override nonReentrant onlyOwner {
      (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
      if (cur != 2) revert InvalidPauseTransition(cur, 1);
-     IMetricOmmPoolFactoryActions(pool).setPause(1);
+     IMetricOmmPoolFactoryActions(pool).setPause(0);
  }
```

If the design intent is to restore the pre-protocol-pause level (0 or 1), the factory must record the previous level when `protocolPausePool` is called and restore it here. The simplest correct fix for the common case is `setPause(0)`.

---

### Proof of Concept

1. Pool deployed; `pauseLevel = 0` (active).
2. Protocol owner calls `protocolPausePool(pool)` → `pauseLevel = 2`.
3. Protocol owner calls `protocolUnpausePool(pool)` → `setPause(1)` → `pauseLevel = 1`.
4. Trader calls `swap(...)` → `_checkNotPaused()` fires → `revert PoolPaused()`.
5. Pool admin must separately call `unpausePool(pool)` (which checks `cur == 1` ✓) → `setPause(0)` → `pauseLevel = 0`.

Step 4 demonstrates that the pool is swap-blocked after the protocol "unpause," breaking core functionality for all users until step 5 is performed.
