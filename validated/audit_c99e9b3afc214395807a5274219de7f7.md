### Title
Precision Loss in `native_per_pubdata` Calculation Due to Division Before Multiplication - (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

`native_per_pubdata` is computed as `pubdata_price / native_price` (floor/truncating integer division) and then multiplied by `pubdata_bytes_used`. This is the classic division-before-multiplication ordering error: the truncation in the first step causes users to be systematically undercharged for pubdata consumption, at the expense of the operator/protocol.

---

### Finding Description

In `validation_impl.rs`, `native_per_pubdata` is derived as:

```rust
// basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs, line 142
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
``` [1](#0-0) 

This truncated `native_per_pubdata` is then multiplied by the actual pubdata bytes consumed in `gas_helpers.rs`:

```rust
// basic_bootloader/src/bootloader/transaction_flow/gas_helpers.rs, lines 430-432
let native = current_pubdata_spent
    .checked_mul(native_per_pubdata)
    .ok_or(out_of_native_resources!())?;
``` [2](#0-1) 

The same pattern appears in the off-chain API helper:

```rust
// api/src/helpers.rs, line 427
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
``` [3](#0-2) 

**The actual computation is:**
```
native_charged = pubdata_bytes * floor(pubdata_price / native_price)
```

**The correct computation should be:**
```
native_charged = floor(pubdata_bytes * pubdata_price / native_price)
```

Let `pubdata_price = q * native_price + r` where `0 ≤ r < native_price`. Then:

- **Actual:** `pubdata_bytes * q`
- **Correct:** `pubdata_bytes * q + floor(pubdata_bytes * r / native_price)`

The undercharge per transaction is `floor(pubdata_bytes * r / native_price)`, which is up to `pubdata_bytes * (native_price - 1) / native_price` — strictly less than `pubdata_bytes` native units.

---

### Impact Explanation

Every transaction that writes pubdata (storage writes, events, etc.) is undercharged for its pubdata consumption whenever `pubdata_price % native_price != 0`. The operator/protocol receives less native-resource compensation than the actual pubdata cost warrants. For a transaction writing 100 KB of pubdata with `native_price = 10` and `pubdata_price % native_price = 9`, the undercharge is up to ~90,000 native units per transaction. This accumulates across all transactions in every block.

The `native_per_pubdata` value is also used to gate whether a transaction has enough resources to cover its pubdata (`check_enough_resources_for_pubdata`), meaning the check itself is slightly too lenient — transactions that should be reverted for insufficient pubdata payment may pass. [4](#0-3) 

---

### Likelihood Explanation

This triggers on every L2 transaction that generates pubdata whenever `pubdata_price` is not an exact multiple of `native_price`. Since both values are operator-set floating-point-like U256 quantities, exact divisibility is the exception rather than the rule. Any unprivileged user submitting a transaction with storage writes, events, or other pubdata-generating operations reaches this code path. [5](#0-4) 

---

### Recommendation

Reorder the computation to multiply before dividing, preserving full precision:

```rust
// Instead of:
let native_per_pubdata = pubdata_price.wrapping_div(native_price);
// ...later...
let native = current_pubdata_spent.checked_mul(native_per_pubdata)?;

// Use:
let native = (U256::from(current_pubdata_spent)
    .checked_mul(pubdata_price)?
    .wrapping_div(native_price));
// then convert to u64
```

Apply the same fix in `api/src/helpers.rs` to keep the off-chain validation consistent with the on-chain bootloader behavior.

---

### Proof of Concept

Concrete example with `pubdata_price = 1_000_000_007` and `native_price = 1_000_000_000` (a realistic near-1:1 ratio):

- `native_per_pubdata = 1_000_000_007 / 1_000_000_000 = 1` (truncated)
- For 10,000 pubdata bytes: charged = `10_000 * 1 = 10_000` native
- Correct charge: `(10_000 * 1_000_000_007) / 1_000_000_000 = 10_000` (same here)

With `pubdata_price = 1_500_000_000` and `native_price = 1_000_000_000`:

- `native_per_pubdata = 1_500_000_000 / 1_000_000_000 = 1` (truncated, loses the 0.5)
- For 10,000 pubdata bytes: charged = `10_000 * 1 = 10_000` native
- Correct charge: `(10_000 * 1_500_000_000) / 1_000_000_000 = 15_000` native
- **Undercharge: 5,000 native units (33% loss)** [6](#0-5) [7](#0-6)

pub fn get_resources_to_charge_for_pubdata<S: EthereumLikeTypes>(
    system: &mut System<S>,
    native_per_pubdata: u64,
    base_pubdata: Option<u64>,
) -> Result<(u64, S::Resources), SystemError> {
    let current_pubdata_spent = system
        .net_pubdata_used()?
        .saturating_sub(base_pubdata.unwrap_or(0));

    Ok((current_pubdata_spent, S::Resources::from_native(native)))
