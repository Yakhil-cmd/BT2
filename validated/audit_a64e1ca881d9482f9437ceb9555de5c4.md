### Title
Gateway Stateful Validator Uses Previous Block's L2 Gas Price Instead of Next Block's Price for Resource Bounds Admission — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

`StatefulTransactionValidator::validate_resource_bounds` compares a transaction's `max_price_per_unit` against the **previous (committed) block's** L2 gas price, not the price that will actually govern the next block. Because the L2 gas price follows an EIP-1559 adjustment curve, the two values diverge every block. The gateway therefore admits transactions whose `max_price_per_unit` is below the next block's actual price (they will fail execution) and rejects transactions whose `max_price_per_unit` is above the next block's price but below the previous block's price (they would succeed). A developer TODO in the source code explicitly acknowledges the correct value is not yet used.

### Finding Description

In `validate_resource_bounds`, the gateway reads the block info from `GatewayFixedBlockSyncStateClient`, which is pinned to `latest_block_number` at validator-instantiation time:

```rust
// TODO(Arni): getnext_l2_gas_price from the block header.
let previous_block_l2_gas_price = self
    .gateway_fixed_block_state_reader
    .get_block_info()
    .await?
    .gas_prices
    .strk_gas_prices
    .l2_gas_price;
self.validate_tx_l2_gas_price_within_threshold(
    executable_tx.resource_bounds(),
    previous_block_l2_gas_price,
)?;
``` [1](#0-0) 

The threshold is then computed as `previous_block_l2_gas_price * min_gas_price_percentage / 100`. In production the percentage is **100**, so the threshold equals the previous block's price exactly. [2](#0-1) 

The L2 gas price is not static. It is recalculated after every block via `calculate_next_base_gas_price` (EIP-1559 style): a full block raises the price, an empty block lowers it. [3](#0-2) 

The result of this calculation is stored in `BlockHeaderWithoutHash::next_l2_gas_price` and is available to the gateway at validation time. [4](#0-3) 

The gateway's `run_validate_entry_point` also builds its `BlockContext` from the same stale block info (only incrementing the block number, not the gas price), so the blockifier validation inside the gateway is internally consistent with the wrong price — but diverges from the price the batcher will actually use when building the block. [5](#0-4) 

### Impact Explanation

**Admission of transactions that will fail execution (price rises):**
When the previous block was heavily used, `next_l2_gas_price > previous_l2_gas_price`. A transaction with `max_price_per_unit = previous_l2_gas_price` passes the gateway threshold check but will fail `check_fee_bounds` / `verify_can_pay_committed_bounds` inside the batcher's `perform_pre_validation_stage`, which uses the real next-block price. The gateway has admitted an invalid transaction. [6](#0-5) 

**Rejection of valid transactions (price falls):**
When the previous block was lightly used, `next_l2_gas_price < previous_l2_gas_price`. A transaction with `max_price_per_unit = next_l2_gas_price` would succeed in the batcher but is rejected by the gateway because it is below the stale threshold. A valid transaction is denied admission.

Both outcomes match the **High** impact category: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

### Likelihood Explanation

The L2 gas price changes every block. The discrepancy is therefore always present and grows with block utilization variance. No configuration or runtime guard corrects for it; the TODO comment confirms the correct value (`next_l2_gas_price`) is not yet read. [7](#0-6) 

### Recommendation

Replace the read of the previous block's `l2_gas_price` with the `next_l2_gas_price` field already stored in `BlockHeaderWithoutHash`. The `GatewayFixedBlockStateReader::get_block_info` return type already carries the full header; the fix is to surface and use `block_header.next_l2_gas_price` instead of `block_header.l2_gas_price` in `validate_resource_bounds`. [8](#0-7) 

### Proof of Concept

1. Previous block N is full → EIP-1559 computes `next_l2_gas_price` = 112 fri (up from 100 fri).
2. User submits an invoke transaction with `l2_gas.max_price_per_unit = 100 fri`.
3. Gateway calls `validate_resource_bounds`: threshold = `100 * 100% = 100`; `100 >= 100` → **admitted**.
4. Transaction enters the mempool.
5. Batcher builds block N+1 with `block_context.gas_prices.strk_gas_prices.l2_gas_price = 112 fri`.
6. `perform_pre_validation_stage` → `check_fee_bounds`: transaction's `max_price_per_unit (100) < block price (112)` → **execution fails**.
7. The gateway has admitted a transaction that cannot be included in any block at the current price level.

pub fn calculate_next_base_gas_price(
    price: GasPrice,
    gas_used: GasAmount,
    gas_target: GasAmount,
    min_gas_price: GasPrice,
) -> GasPrice {
    let versioned_constants = VersionedConstants::latest_constants();
    assert!(
        gas_target < versioned_constants.max_block_size,
        "Gas target must be lower than max block size."
    );
    assert!(gas_target.0 > 0, "Gas target must be greater than zero.");
    assert!(
        versioned_constants.gas_price_max_change_denominator > 0,
        "Denominator constant must be greater than zero."
    );

    // If the current price is below the minimum, apply a gradual adjustment and return early.
    // This allows the price to increase by at most 1/MIN_GAS_PRICE_INCREASE_DENOMINATOR per block.
    if price < min_gas_price {
        let max_increase = price.0 / MIN_GAS_PRICE_INCREASE_DENOMINATOR;
        let adjusted = price.0 + max_increase;
        // Cap at min_gas_price to avoid overshooting
        let adjusted_price = adjusted.min(min_gas_price.0);
        info!(
            "Fee Market: Price {} below minimum gas price {}, adjusted price: {} )",
            price.0, min_gas_price.0, adjusted_price
        );
        return GasPrice(adjusted_price);
    }

    // Use U256 to avoid overflow, as multiplying a u128 by a u64 remains within U256 bounds.
    let gas_delta = U256::from(gas_used.0.abs_diff(gas_target.0));
    let gas_target_u256 = U256::from(gas_target.0);
    let price_u256 = U256::from(price.0);

    // Calculate price change by multiplying first, then dividing. This avoids the precision loss
    // that occurs when dividing before multiplying.
    let denominator =
        gas_target_u256 * U256::from(versioned_constants.gas_price_max_change_denominator);
    let price_change = (price_u256 * gas_delta) / denominator;

    let adjusted_price_u256 =
        if gas_used > gas_target { price_u256 + price_change } else { price_u256 - price_change };

    // Sanity check: ensure direction of change is correct
    assert!(
        gas_used > gas_target && adjusted_price_u256 >= price_u256
            || gas_used <= gas_target && adjusted_price_u256 <= price_u256
    );

    // Price should not realistically exceed u128::MAX, bound to avoid theoretical overflow.
    let adjusted_price = u128::try_from(adjusted_price_u256).unwrap_or(u128::MAX);
    GasPrice(max(adjusted_price, min_gas_price.0))
```
