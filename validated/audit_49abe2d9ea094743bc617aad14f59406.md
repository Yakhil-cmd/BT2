### Title
Cross-COA Signature Replay via Missing EVM Address Binding in Signed Data - (File: `fvm/evm/stdlib/contract.cdc`)

---

### Summary

`EVM.validateCOAOwnershipProof` verifies signatures over caller-supplied `signedData` but does not enforce that `signedData` encodes the target `evmAddress`. A valid signature produced for one COA (Cadence-Owned Account) can be replayed against a different COA owned by the same Cadence account. The code itself acknowledges this gap in a comment at lines 1003–1009. The function is `access(all)` and is exposed both as a Cadence-callable function and as the EVM precompile `verifyCOAOwnershipProof(address,bytes32,bytes)` reachable from any EVM contract.

---

### Finding Description

**Root cause — `fvm/evm/stdlib/contract.cdc`, lines 1001–1116**

`validateCOAOwnershipProof` performs two independent checks:

1. Verifies that the supplied `signatures` are cryptographically valid over `signedData` using the account keys at `address`.
2. Verifies that the COA resource at `path` has an EVM address equal to `evmAddress`. [1](#0-0) 

Crucially, there is **no check that `signedData` encodes `evmAddress`**. The two checks are entirely independent. The code comment at line 1003 explicitly acknowledges this:

> "this function does not enforce that `signedData` includes `evmAddress`. In principle, a signature produced for one purpose could be replayed here against a different COA owned by the same Cadence account." [2](#0-1) 

The signature verification passes as long as the signatures are valid over `signedData` — regardless of which COA address is being claimed.

**Exposure surface**

The function is `access(all)` and is wired into the EVM precompile `verifyCOAOwnershipProof(address,bytes32,bytes)` via `coaOwnershipProofValidator` in `fvm/evm/handler/precompiles.go`: [3](#0-2) 

The precompile is registered in the Cadence Arch contract and is callable from any EVM contract: [4](#0-3) 

The `proofVerifier.Run` method decodes the ABI-encoded proof and calls the validator without any additional binding check: [5](#0-4) 

`DecodeABIEncodedProof` constructs the `COAOwnershipProofInContext` from caller-supplied bytes, with `EVMAddress` taken directly from the input: [6](#0-5) 

The `COAOwnershipProofInContext` struct carries `SignedData` and `EVMAddress` as independent fields with no enforced relationship: [7](#0-6) 

---

### Impact Explanation

Any EVM contract or Cadence transaction that uses `verifyCOAOwnershipProof` / `validateCOAOwnershipProof` to gate an authorization decision (e.g., a DeFi protocol granting withdrawal rights to the proven owner of a specific COA) is vulnerable to cross-COA replay. An attacker who observes a valid proof for COA_A can replay it against COA_B owned by the same Cadence account, obtaining a `true` result for COA_B without the account holder's consent. This constitutes unauthorized authentication/authorization over on-chain assets controlled by COA_B.

The COA's own ERC-1271 `isValidSignature` in `coa.sol` is safe because it always passes `address(this)`: [8](#0-7) 

However, any third-party EVM contract or Cadence script that calls the precompile or the Cadence function directly with attacker-controlled parameters is exposed.

---

### Likelihood Explanation

- **Precondition**: A Cadence account must own more than one COA (at different public paths). While uncommon today, the protocol supports it and the number of multi-COA accounts will grow as the ecosystem matures.
- **Attacker capability**: No privileged access is required. Any EVM transaction or Cadence script can call `verifyCOAOwnershipProof` / `validateCOAOwnershipProof` with arbitrary parameters.
- **Proof availability**: Proofs are produced off-chain and submitted on-chain; an attacker can observe them from transaction data.
- **Trigger**: The attacker substitutes the `path` and `evmAddress` fields to point to a different COA owned by the same account, keeping `signedData` and `signatures` identical.

---

### Recommendation

Enforce inside `validateCOAOwnershipProof` that `signedData` commits to `evmAddress` — for example, by prepending the 20-byte EVM address to the data before signature verification, or by requiring callers to include it and verifying the commitment on-chain. The current approach of delegating this responsibility to callers via a documentation comment is insufficient for a security-critical primitive exposed as `access(all)`.

---

### Proof of Concept

1. Alice's Cadence account owns two COAs: COA_A published at `/public/coa_a`, COA_B published at `/public/coa_b`.
2. An EVM DeFi protocol calls `verifyCOAOwnershipProof(COA_A_address, hash(D), proof_A)` to confirm Alice controls COA_A before granting her a withdrawal. Alice signs data `D` (which does not encode any COA address) and submits `proof_A = {Address: Alice, CapabilityPath: "coa_a", KeyIndices: [0], Signatures: [sig_D]}`.
3. An attacker observes `proof_A` and `D` from the transaction.
4. The attacker constructs a new call: `verifyCOAOwnershipProof(COA_B_address, hash(D), proof_B)` where `proof_B = {Address: Alice, CapabilityPath: "coa_b", KeyIndices: [0], Signatures: [sig_D]}`.
5. Inside `validateCOAOwnershipProof`:
   - `keyList.verify(signatureSet, signedData: D, ...)` → **passes** (sig_D is valid over D with Alice's key).
   - COA at `/public/coa_b` has address COA_B_address → **passes**.
   - Returns `ValidationResult(isValid: true)`.
6. The attacker has forged a valid ownership proof for COA_B using a signature Alice only intended for COA_A, enabling unauthorized authorization over COA_B's assets. [9](#0-8)

func (f *proofVerifier) Run(input []byte) ([]byte, error) {
	proof, err := DecodeABIEncodedProof(input)
	if err != nil {
		return nil, err
	}
	verified, err := f.proofVerifier(proof)
	if err != nil {
		return nil, err
	}

	buffer := make([]byte, EncodedBoolSize)
	return buffer, EncodeBool(verified, buffer, 0)
}

func DecodeABIEncodedProof(input []byte) (*types.COAOwnershipProofInContext, error) {
	index := 0
	caller, err := ReadAddress(input, index)
	index += FixedSizeUnitDataReadSize
	if err != nil {
		return nil, err
	}

	hash, err := ReadBytes32(input, index)
	index += Bytes32DataReadSize
	if err != nil {
		return nil, err
	}

	encodedProof, err := ReadBytes(input, index)
	if err != nil {
		return nil, err
	}

	return types.NewCOAOwnershipProofInContext(
		hash,
		types.Address(caller),
		encodedProof,
	)
}
