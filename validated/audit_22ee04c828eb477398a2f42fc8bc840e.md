### Title
Protocol Fee Truncates to Zero on Small Swaps Due to Division-Before-Multiplication Rounding — (`File: metric-core/contracts/libraries/SwapMath.sol`)

---

### Summary

In `SwapMath.sol`, the protocol (spread) fee share is computed with a plain integer division that truncates to zero whenever the LP fee amount is small relative to the `1e6` denominator. A trader can exploit this by splitting swaps into small chunks, paying LP fees but paying zero protocol fees on each chunk.

---

### Finding Description

In every swap-in-bin function, the protocol's share of the LP fee is calculated as:

```solidity
uint256 protocolFeeAmountScaled = (feeAmountScaled * spreadFeeE6) / 1e6;
``` [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) 

`spreadFeeE6` is a `uint24` field packed in slot0, representing the protocol's share of the LP fee in units of 1/1e6 (so `1e6` = 100%). [5](#0-4) 

When `feeAmountScaled * spreadFeeE6 < 1e6`, Solidity integer division truncates the result to **zero**. The LP fee (`feeAmountScaled`) is still fully credited to the bin, but the protocol receives nothing. The `binLpFeeAmount` returned to LPs is then `feeAmountScaled - 0 = feeAmountScaled`, meaning LPs silently absorb the protocol's share. [6](#0-5) [7](#0-6) 

The same pattern exists for the `notionalFeeE8` accumulator, which uses a denominator of `1e8` — a larger denominator that makes truncation to zero even easier to trigger. [8](#0-7) 

---

### Impact Explanation

The protocol permanently loses its fee revenue on any swap where `feeAmountScaled * spreadFeeE6 < 1e6`. There is no accumulation across swaps — each swap's protocol fee is computed and credited independently, so rounding losses are never recovered. The `protocolFeeAmountScaled` accumulator in `SwapState` simply never grows for these swaps. [9](#0-8) 

---

### Likelihood Explanation

The threshold for truncation is:

```
feeAmountScaled < 1e6 / spreadFeeE6
```

- If `spreadFeeE6 = 100,000` (10% protocol share): any swap where `feeAmountScaled < 10` causes zero protocol fee.
- If `spreadFeeE6 = 10,000` (1% protocol share): any swap where `feeAmountScaled < 100` causes zero protocol fee.
- If `spreadFeeE6 = 1,000` (0.1% protocol share): any swap where `feeAmountScaled < 1,000` causes zero protocol fee.

For tokens with 6 or fewer decimals (e.g., USDC), or on L2s where gas cost is negligible, a trader can split any swap into many small chunks, each below the truncation threshold, and pay zero protocol fees across the entire trade. The trigger requires no special privileges — any caller of `swap()` can do this.

---

### Recommendation

Replace the truncating division with a ceiling division for the protocol fee, or enforce a minimum non-zero protocol fee when the LP fee is non-zero:

```solidity
// Instead of:
uint256 protocolFeeAmountScaled = (feeAmountScaled * spreadFeeE6) / 1e6;

// Use ceiling division so at least 1 unit is charged when spreadFeeE6 > 0:
uint256 protocolFeeAmountScaled = Math.ceilDiv(feeAmountScaled * spreadFeeE6, 1e6);
```

Alternatively, revert or skip the swap if the resulting protocol fee would be zero but `spreadFeeE6 > 0` and `feeAmountScaled > 0`, enforcing a minimum swap size.

---

### Proof of Concept

Given:
- `spreadFeeE6 = 10_000` (1% protocol share, a realistic competitive setting)
- `feeAmountScaled = 50` (small swap on a low-decimal token)

Calculation:
```
protocolFeeAmountScaled = (50 * 10_000) / 1_000_000
                        = 500_000 / 1_000_000
                        = 0  ← truncated
```

The protocol receives 0 fees. The LP receives `feeAmountScaled - 0 = 50` instead of `50 - 1 = 49`. A trader repeating this across N small swaps that together constitute a large trade pays zero protocol fees on the entire position. [10](#0-9) [11](#0-10)
