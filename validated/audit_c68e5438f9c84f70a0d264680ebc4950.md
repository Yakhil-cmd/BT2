### Title
`SwapAllowlistExtension` Checks Caller But Not Recipient, Allowing Non-Allowlisted Addresses to Receive Swap Output Tokens - (File: metric-periphery/contracts/extensions/SwapAllowlistExtension.sol)

### Summary
`SwapAllowlistExtension.beforeSwap` validates only the `sender` (the address calling `pool.swap`) and silently discards the `recipient` parameter. An allowlisted swapper can route all swap output tokens to any non-allowlisted address, bypassing the pool's access-control gate entirely.

### Finding Description
In `SwapAllowlistExtension.beforeSwap`, the second positional argument — `recipient` — is unnamed and never read:

```solidity
// metric-periphery/contracts/extensions/SwapAllowlistExtension.sol L31-41
function beforeSwap(address sender, address, bool, int128, uint128, uint256, uint128, uint128, bytes calldata)
    external view override returns (bytes4)
{
    if (!allowAllSwappers[msg.sender] && !allowedSwapper[msg.sender][sender]) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
```

The pool's `swap` function forwards both `msg.sender` (as `sender`) and the caller-supplied `recipient` to `_beforeSwap`, then unconditionally transfers output tokens to `recipient`:

```solidity
// MetricOmmPool.sol L230-254 (condensed)
_beforeSwap(msg.sender, recipient, ...);   // extension sees recipient but ignores it
...
transferToken1(recipient, uint256(-amount1Delta));  // tokens go to recipient unchecked
```

The `IMetricOmmExtensions.beforeSwap` interface explicitly carries `recipient` as its second parameter, so the extension has the information available but discards it.

### Impact Explanation
A pool configured with `SwapAllowlistExtension` to enforce a permissioned participant set (e.g., KYC-gated or institutional-only pool) can be fully bypassed on the output side. An allowlisted swapper calls `pool.swap(recipient = non_allowlisted_address, ...)`. The extension passes because `sender` is allowlisted; the pool then transfers output tokens to the non-allowlisted `recipient`. The non-allowlisted address receives pool assets without ever appearing on the allowlist. This is the direct analog of the BakerFi M-09 pattern: caller is checked, recipient is not.

### Likelihood Explanation
Any single allowlisted address is sufficient to trigger the bypass. No special privileges, flash loans, or multi-step setup are required — a single `swap` call with an arbitrary `recipient` suffices. The bypass is trivially repeatable.

### Recommendation
Name and validate the `recipient` parameter inside `beforeSwap`:

```solidity
function beforeSwap(
    address sender,
    address recipient,   // ← was unnamed/ignored
    bool, int128, uint128, uint256, uint128, uint128, bytes calldata
) external view override returns (bytes4) {
    bool senderOk   = allowAllSwappers[msg.sender] || allowedSwapper[msg.sender][sender];
    bool recipientOk = allowAllSwappers[msg.sender] || allowedSwapper[msg.sender][recipient];
    if (!senderOk || !recipientOk) {
        revert IMetricOmmPoolActions.NotAllowedToSwap();
    }
    return IMetricOmmExtensions.beforeSwap.selector;
}
```

### Proof of Concept
1. Pool admin deploys a pool with `SwapAllowlistExtension` attached.
2. Pool admin calls `setAllowedToSwap(pool, alice, true)` — `alice` is allowlisted; `bob` is not.
3. `alice` calls `pool.swap(recipient = bob, zeroForOne, amount, ...)`.
4. `beforeSwap` fires: `msg.sender = pool`, `sender = alice` → `allowedSwapper[pool][alice] == true` → check passes.
5. Swap executes; `transferToken1(bob, outputAmount)` sends tokens to `bob`.
6. `bob` receives pool output tokens without being on the allowlist — the access-control invariant is broken. [1](#0-0) [2](#0-1) [3](#0-2)
