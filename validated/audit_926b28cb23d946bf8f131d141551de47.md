### Title
Missing Grace Period After Unpausing Causes Stakers to Permanently Lose Epoch Rewards - (File: src/attestation/attestation.cairo, src/staking/staking.cairo)

### Summary
When the staking contract is paused during a staker's attestation window and then unpaused after the window has closed, the staker permanently loses their epoch rewards. There is no grace period after unpausing to allow stakers to complete attestations they were blocked from submitting.

### Finding Description

The staking contract implements a pause mechanism: `pause()` (callable by security agent) and `unpause()` (callable by security admin). When paused, `update_rewards_from_attestation_contract` reverts with "Contract is paused".

The attestation contract's `attest()` function calls `update_rewards_from_attestation_contract` on the staking contract as its final step: [1](#0-0) 

This means that when the staking contract is paused, any call to `attest()` will revert — even though the attestation contract itself has no pause mechanism.

The attestation window is strictly block-bounded. `_assert_attest_in_window` enforces:

```
min_block = target_attestation_block + MIN_ATTESTATION_WINDOW  (11 blocks)
max_block = target_attestation_block + attestation_window
``` [2](#0-1) 

The target attestation block is deterministically computed per staker per epoch using a Poseidon hash of `(stake, epoch_id, staker_address)`: [3](#0-2) 

**Attack/Scenario Path:**

1. Staker's attestation window for epoch `N` is blocks `[X+11, X+20]`.
2. Security agent pauses the staking contract at block `X+11`.
3. Staker calls `attest()` — it reverts because `update_rewards_from_attestation_contract` is blocked.
4. Security admin unpauses at block `X+21` (after the window has closed).
5. Staker attempts to attest — `_assert_attest_in_window` now fails with `ATTEST_OUT_OF_WINDOW`.
6. Staker permanently loses epoch `N` rewards. No recovery path exists.

The `unpause()` implementation writes `is_paused = false` and emits an event, with no grace period logic: [4](#0-3) 

The pause test suite confirms `update_rewards_from_attestation_contract` is blocked when paused: [5](#0-4) 

### Impact Explanation

Stakers permanently lose epoch rewards (unclaimed yield) for any epoch where the staking contract was paused for the duration of their attestation window. Since the target attestation block is deterministic and the window is narrow (configurable, minimum 11 blocks), a pause of even moderate duration can cover the entire window for many stakers simultaneously. The lost rewards are not recoverable — there is no mechanism to retroactively credit attestation for a missed epoch.

This maps to the allowed impact: **High — Permanent freezing of unclaimed yield**.

### Likelihood Explanation

The security agent can pause the contract at any time for legitimate security reasons (e.g., a discovered vulnerability). The attestation window is narrow by design. A pause of even a few dozen blocks will overlap with the attestation windows of a significant fraction of active stakers, since each staker's window is independently distributed across the epoch. This is not a contrived scenario — any security incident requiring a pause will trigger this loss.

### Recommendation

After `unpause()`, record the block number at which the contract was unpaused. In `_assert_attest_in_window`, extend the valid window by the duration of the most recent pause:

```
effective_max_block = max_block + pause_duration_in_blocks
```

Alternatively, track a per-epoch "pause credit" that extends the attestation deadline for all stakers in epochs that overlapped with a pause period. This mirrors the Chainlink sequencer uptime feed recommendation of a grace period after resumption.

### Proof of Concept

1. Deploy the system with `attestation_window = 20`.
2. Advance to epoch 2. Staker's `target_attestation_block = T`.
3. At block `T + 11` (start of valid window), security agent calls `pause()`.
4. Staker calls `attest()` — reverts with "Contract is paused".
5. At block `T + 21`, security admin calls `unpause()`.
6. Staker calls `attest()` — reverts with `ATTEST_OUT_OF_WINDOW` because `current_block (T+21) > max_block (T+20)`.
7. Staker earns zero rewards for epoch 2 with no recourse. [6](#0-5)

            let hash = PoseidonTrait::new()
                .update(staking_attestation_info.stake().into())
                .update(staking_attestation_info.epoch_id().into())
                .update(staking_attestation_info.staker_address().into())

        fn _assert_attest_in_window(self: @ContractState, target_attestation_block: BlockNumber) {
            let attestation_window = self.attestation_window.read();
            let current_block_number = get_block_number();
            let min_block = target_attestation_block + MIN_ATTESTATION_WINDOW.into();
            let max_block = target_attestation_block + attestation_window.into();
            assert!(
                min_block <= current_block_number && current_block_number <= max_block,
                "{}",
                Error::ATTEST_OUT_OF_WINDOW,
            );
        }
