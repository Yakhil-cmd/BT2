The vulnerability claim is confirmed. Here is the analysis:

### Title
Unauthorized Token Burn from Arbitrary Victim Address in Bank Precompile — (`x/cronos/keeper/precompiles/bank.go`)

### Summary

The `BankContract.Run` burn path accepts an arbitrary `recipient` address (the address to burn **from**) without verifying that `contract.Caller() == recipient`. Any EVM contract can burn `evm/<itself>` tokens from any account that holds them, without the token holder's authorization.

### Finding Description

In `bank.go`, the `MintMethodName` and `BurnMethodName` cases share a single code path: [1](#0-0) 

- `recipient` (line 121) is the address tokens are burned **from** — it is caller-supplied and unchecked.
- `checkBlockedAddr` (line 127) only verifies the address is not a blocked **receiver** of funds; it is not an authorization check on who may initiate a burn.
- `denom` (line 130) is derived from `contract.Caller()`, meaning the denom is `evm/<attacker_contract>`.
- `SendCoinsFromAccountToModule(ctx, addr, ...)` (line 144) pulls tokens directly from the victim's bank account with no signer/authorization check. [2](#0-1) 

There is **no guard** of the form `contract.Caller() == recipient`. Compare this to `utils.go`, where the generic `exec` helper explicitly enforces `caller == signer`: [3](#0-2) 

The burn path in `bank.go` has no equivalent check.

### Impact Explanation

An attacker deploys a contract at address `A`. The denom `evm/A` is under `A`'s control. The attacker can:

1. Mint `evm/A` tokens to a victim (or the victim acquires them through a legitimate protocol that uses `evm/A` as a receipt/LP token).
2. Call `bank.burn(victimAddress, amount)` from contract `A`.
3. The precompile executes `SendCoinsFromAccountToModule(ctx, victimAddr, ...)` and `BurnCoins(...)` — the victim's balance of `evm/A` is destroyed without their consent or signature.

This is an **unauthorized burn** of tokens from an arbitrary account, matching the Critical impact category: *"Unauthorized burn … of … precompile-controlled assets."*

### Likelihood Explanation

Any unprivileged user can deploy an EVM contract and invoke this path. The only precondition is that the victim holds `evm/<attacker_contract>` tokens, which the attacker can arrange by minting to the victim first (the mint path has the same absence of recipient-consent checks). Protocols that issue `evm/<contract>` tokens as meaningful assets (staking receipts, LP shares, wrapped assets) are directly exposed.

### Recommendation

Add an authorization check in the burn branch requiring the address being burned from to equal the caller:

```go
// In the burn branch only:
if recipient != contract.Caller() {
    return nil, errors.New("burn: caller is not authorized to burn from this address")
}
```

Alternatively, restructure so that `burn` always burns from `contract.Caller()`'s own account (i.e., `addr = sdk.AccAddress(contract.Caller().Bytes())`), consistent with the principle that a caller can only burn their own tokens.

The same missing authorization check exists in the `TransferMethodName` branch (line 192), where `sender` is caller-supplied and `SendCoins(ctx, from, to, ...)` is executed without verifying `contract.Caller() == sender`. [4](#0-3) 

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

interface IBank {
    function mint(address recipient, uint256 amount) external returns (bool);
    function burn(address recipient, uint256 amount) external returns (bool);
    function balanceOf(address token, address account) external view returns (uint256);
}

contract AttackerContract {
    IBank constant bank = IBank(address(100)); // bankContractAddress = 0x64

    function setup(address victim, uint256 amount) external {
        // Step 1: mint evm/<this> tokens to victim (no victim consent needed)
        bank.mint(victim, amount);
    }

    function exploit(address victim, uint256 amount) external {
        // Step 2: burn evm/<this> tokens FROM victim without victim's signature
        bank.burn(victim, amount);
        // victim's balance of evm/<this> is now 0, no victim action required
    }
}
```

Test plan:
1. Deploy `AttackerContract` at address `A`.
2. Call `setup(victim, 1000)` — victim's `evm/A` balance becomes 1000.
3. Assert victim balance == 1000.
4. Call `exploit(victim, 1000)` — no transaction from victim.
5. Assert victim balance == 0. Burn succeeded without victim's authorization.
