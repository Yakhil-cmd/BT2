### Title
Gateway `skip_stateful_validations` Bypasses `__validate__` for Existing Accounts with Nonce=0 and Any Pending Mempool Transaction — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

`skip_stateful_validations` is intended to skip the `__validate__` entry-point call only when a `deploy_account` for a not-yet-deployed account is pending in the mempool. However, the guard condition uses `account_tx_in_pool_or_recent_block`, which returns `true` for **any** transaction type from that address — including future-nonce invoke/declare transactions. This means an already-deployed account with on-chain nonce=0 that has any future-nonce transaction in the mempool will also trigger the skip, allowing an attacker to inject a fake-signature invoke with nonce=1 that bypasses `__validate__` at the gateway and is admitted to the mempool.

---

### Finding Description

The function `skip_stateful_validations` in `crates/apollo_gateway/src/stateful_transaction_validator.rs` decides whether to skip the blockifier `__validate__` call for an incoming invoke transaction: [1](#0-0) 

The skip fires when all three conditions hold:
1. The incoming transaction is an `Invoke` with `nonce == 1`.
2. The on-chain account nonce is `0`.
3. `account_tx_in_pool_or_recent_block(sender)` returns `true`.

The third condition is implemented as: [2](#0-1) 

It returns `true` if the address appears in `state.staged`, `state.committed`, or `tx_pool`. The code comment in `skip_stateful_validations` explicitly acknowledges the second trigger path: *"transactions with future nonces that passed validations."*

The flaw is that `account_nonce == Nonce(Felt::ZERO)` does **not** exclusively identify an undeployed account. A deployed account that has never sent a transaction also has on-chain nonce=0. If such an account has a future-nonce transaction (e.g., nonce=2) in the mempool — which was admitted after passing `__validate__` — then `account_tx_in_pool_or_recent_block` returns `true`, and the skip fires for any subsequent nonce=1 invoke, regardless of its signature.

The skip flag propagates directly to `run_validate_entry_point`: [3](#0-2) 

Setting `validate: false` means the blockifier's `StatefulValidator::perform_validations` path for invoke transactions skips the `__validate__` call entirely: [4](#0-3) 

The transaction is then forwarded to the mempool via `validate_by_mempool` / `add_tx`, which only checks nonce ordering — not the signature: [5](#0-4) 

---

### Impact Explanation

An attacker who observes (via the public mempool) that account `X` has on-chain nonce=0 and any pending transaction can submit an invoke with nonce=1 carrying an arbitrary or empty signature. The gateway admits it to the mempool without running `__validate__`. When the batcher later executes the transaction, `__validate__` runs and fails, causing the transaction to be rejected. The net effects are:

- **Unauthorized transaction admission**: a transaction whose signature was never verified by the account's own `__validate__` logic is accepted into the sequencer's mempool.
- **Mempool pollution / DoS**: an attacker can continuously inject invalid nonce=1 transactions for any qualifying account, consuming mempool capacity and batcher execution slots.
- **Matching impact category**: *High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.*

---

### Likelihood Explanation

The preconditions are observable from the public mempool state:
- Accounts with on-chain nonce=0 are common (newly deployed contracts, accounts that have only received funds).
- A future-nonce transaction in the mempool is possible whenever `max_allowed_nonce_gap > 1` (the default allows a gap).
- No privileged access is required; any unprivileged caller can submit an RPC transaction.

---

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a **`DeployAccount` transaction** for the sender is pending. The skip should only fire when the account is provably undeployed (i.e., no contract code exists at the address), not merely because the on-chain nonce happens to be zero.

```rust
// Instead of:
return mempool_client
    .account_tx_in_pool_or_recent_block(tx.sender_address())
    .await ...;

// Use a dedicated check:
return mempool_client
    .deploy_account_tx_in_pool(tx.sender_address())
    .await ...;
```

Alternatively, add a state-reader check that the account has no deployed class hash before allowing the skip, ensuring the bypass is only applied to genuinely undeployed accounts.

---

### Proof of Concept

1. Deploy account `X` (on-chain nonce becomes 0 after deployment, since deploy_account itself sets nonce=1... actually wait, after deploy_account the nonce is 1. So for nonce=0 to hold after deployment, the account must have been deployed via a mechanism that doesn't increment the nonce, or the account was created with nonce=0 via state initialization).

   More precisely: account `X` exists on-chain with nonce=0 (e.g., a contract deployed via `deploy` syscall from another contract, which sets nonce=0 for the new address).

2. The legitimate owner of `X` submits an invoke with nonce=2 (within the allowed nonce gap). This passes `__validate__` at the gateway and enters the mempool. Now `tx_pool.contains_account(X)` is `true`.

3. Attacker submits `RpcInvokeTransactionV3` with `sender_address=X`, `nonce=1`, and a garbage `signature`.

4. Gateway stateless check: passes (non-zero resource bounds, valid address, etc.).

5. Gateway stateful check:
   - `validate_nonce`: `1 >= 0` and within gap → passes.
   - `validate_by_mempool`: nonce=1 not duplicate, not too old → passes.
   - `skip_stateful_validations`: `nonce==1 && account_nonce==0 && account_tx_in_pool_or_recent_block(X)==true` → returns `true`.
   - `run_validate_entry_point`: called with `skip_validate=true` → `validate=false` → `__validate__` **not called**.

6. Transaction enters the mempool. [6](#0-5) 

7. Batcher dequeues the transaction, blockifier runs `__validate__` → signature check fails → transaction rejected. The fake-signature transaction was admitted without any account-level authorization check.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
