### Title
Gateway Admits Invoke Transactions with Unverified Signatures via `skip_stateful_validations` Bypass — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator unconditionally bypasses the account's `__validate__` entry point (the sole on-chain signature-verification step) for any invoke transaction with `nonce == 1` whose sender address appears in the mempool or a recent block. An unprivileged attacker who observes a legitimate user's `deploy_account` transaction in the mempool can submit a crafted invoke with an arbitrary/invalid signature for the same address, have it admitted to the mempool without any signature check, and use fee escalation to displace the legitimate user's own invoke.

### Finding Description

`skip_stateful_validations` is called inside `run_pre_validation_checks` after the mempool nonce/fee checks but before the blockifier `__validate__` call:

```
validate_state_preconditions   ← resource bounds + nonce range only
validate_by_mempool            ← nonce ordering + fee escalation, NO signature check
skip_stateful_validations      ← may return true → skip __validate__
run_validate_entry_point       ← __validate__ skipped when skip_validate == true
```

The skip condition is:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs  lines 437-456
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
    return mempool_client
        .account_tx_in_pool_or_recent_block(tx.sender_address())
        .await
        ...
}
```

`account_tx_in_pool_or_recent_block` returns `true` whenever the address appears in `MempoolState::committed`, `MempoolState::staged`, or `TransactionPool`:

```rust
// crates/apollo_mempool/src/mempool.rs  lines 697-700
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
```

```rust
// crates/apollo_mempool/src/mempool.rs  lines 115-117
fn contains_account(&self, address: ContractAddress) -> bool {
    self.staged.contains_key(&address) || self.committed.contains_key(&address)
}
```

When `skip_validate == true`, `run_validate_entry_point` sets `execution_flags.validate = false`:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs  lines 311-312
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
```

Inside `StatefulValidator::perform_validations`, the validator returns `Ok(())` immediately without calling `__validate__`:

```rust
// crates/blockifier/src/blockifier/stateful_validator.rs  lines 78-81
tx.perform_pre_validation_stage(self.state(), &tx_context)?;
if !tx.execution_flags.validate {
    return Ok(());   // ← exits before __validate__ is called
}
```

`perform_pre_validation_stage` checks nonce, fee bounds, and proof facts — but **not** the cryptographic signature. The signature is exclusively verified by the account contract's `__validate__` entry point, which is now skipped.

The code comment justifies the skip by saying "it is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations." This reasoning is flawed: the check only confirms that *some* transaction from that address is present, not that the *incoming* invoke was signed by the account's private key. Any mempool observer can trigger the skip for any address that has a pending deploy_account.

### Impact Explanation

**High — Mempool/gateway admission accepts invalid transactions before sequencing.**

An invoke transaction carrying an arbitrary (invalid) signature is admitted to the mempool and forwarded to the batcher. The broken invariant is: *every transaction admitted by the gateway must have passed account-level signature verification*. Here that invariant is violated for the specific class of nonce-1 invokes targeting newly deploying accounts.

Secondary consequence: because the attacker's invalid invoke occupies a nonce slot, it can displace the legitimate user's own nonce-1 invoke via fee escalation (`validate_fee_escalation` runs before the skip check and accepts a higher-fee replacement). The legitimate invoke is evicted; the attacker's invalid invoke later fails at blockifier execution time; the user must resubmit.

### Likelihood Explanation

- The mempool is observable (P2P gossip, RPC `starknet_pendingTransactions`).
- The attack requires no privileged access — any user can submit transactions.
- The trigger window is narrow (nonce=1, account_nonce=0) but is exactly the window that occurs for every new account deployment, a common operation.
- The attacker must pay a higher fee to displace an existing invoke, but the cost is bounded and the attacker can choose not to escalate if only mempool pollution (not displacement) is the goal.

### Recommendation

1. **Verify the deploy_account is from the same sender.** The skip should only be granted when the mempool contains a `deploy_account` transaction whose `contract_address` matches the invoke's `sender_address`, not merely any transaction from that address.

2. **Pass the deploy_account transaction hash explicitly.** The `PyValidator` path already accepts an optional `deploy_account_tx_hash` parameter and checks it directly. The Apollo gateway path should adopt the same pattern instead of relying on the coarser `account_tx_in_pool_or_recent_block` predicate.

3. **Alternatively, restrict the skip to the exact nonce window.** The `max_nonce_for_validation_skip` config field exists in `StatefulTransactionValidatorConfig` but is not used by `skip_stateful_validations` in the Apollo gateway (only by `PyValidator`). Wiring it in would at least bound the exposure.

### Proof of Concept

```
1. Legitimate user broadcasts:
      deploy_account(address=A, nonce=0, valid_sig)
      invoke(sender=A, nonce=1, valid_sig)

2. Attacker observes deploy_account for A in the mempool.

3. Attacker submits:
      invoke(sender=A, nonce=1, INVALID_SIG, tip > legitimate_tip)

4. Gateway stateful validation for attacker's invoke:
   a. get_nonce_from_state(A) → 0  (account not yet deployed)
   b. validate_state_preconditions: nonce 1 ∈ [0, 0+200] ✓
   c. validate_by_mempool: nonce 1 ≥ mempool nonce ✓;
      validate_fee_escalation: attacker tip > legitimate tip → legitimate invoke evicted ✓
   d. skip_stateful_validations:
        tx.nonce() == 1 ✓, account_nonce == 0 ✓
        account_tx_in_pool_or_recent_block(A) == true (deploy_account present) ✓
        → returns true (skip __validate__)
   e. run_validate_entry_point: execution_flags.validate = false
        → StatefulValidator returns Ok(()) without calling __validate__

5. Attacker's invalid invoke is admitted to the mempool.
   Legitimate user's invoke has been evicted (fee escalation).

6. Batcher executes the block:
      deploy_account(A) succeeds → A is now deployed.
      invoke(A, nonce=1, INVALID_SIG) → __validate__ called → FAILS.

7. Legitimate user's invoke is gone; user must resubmit.
```

**Corrupted admission decision**: an invoke with an invalid signature receives `Ok(())` from `extract_state_nonce_and_run_validations` and is forwarded to the mempool, violating the gateway's invariant that all admitted transactions have passed account-level validation. ```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
