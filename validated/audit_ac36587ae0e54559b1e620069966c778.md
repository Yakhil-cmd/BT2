### Title
Missing Cache Invalidation in `cascade_delete_subtree_step` Produces Incorrect Ethereum State Root - (File: `basic_system/src/system_implementation/ethereum_storage_model/mpt/updates/delete_subtree.rs`)

---

### Summary

`cascade_delete_subtree_step` removes a child from a parent branch node without invalidating the branch node's cached hash key. A subsequent `recompute` call then uses the stale cached key for that branch node, producing an incorrect Ethereum Merkle Patricia Trie (MPT) state root. The ZK proof commits to this incorrect root, which is accepted by the L1 verifier.

---

### Finding Description

The vulnerability class from the external report is **incorrect interior-node state update during tree restructuring** — specifically, a node that is modified during a deletion operation is not fully updated, leaving stale data that corrupts subsequent computations.

The analog in ZKsync OS is in `cascade_delete_subtree_step`:

```rust
// delete_subtree.rs lines 34-49
} else if parent.is_branch() {
    let branch = &mut self.capacities.branch_nodes[parent.index()];
    let mut found_child_idx = 16;
    for (child_idx, child_node) in branch.child_nodes.iter().enumerate() {
        if *child_node == node_to_delete {
            found_child_idx = child_idx;
            break;
        }
    }
    assert!(found_child_idx < 16);
    branch.delete(found_child_idx)?;          // ← child removed
    // ← branch.invalidate_cache() is MISSING here
    if branch.num_occupied() == 0 {
        Ok(SubtreeDeletionControlFlow::Continue { node: parent })
    } else {
        Ok(SubtreeDeletionControlFlow::Break)
    }
```

Compare with `delete_from_branch_node`, which correctly invalidates the cache after deletion:

```rust
// delete_from_branch.rs lines 11-13
let existing_branch = &mut self.capacities.branch_nodes[branch_node.index()];
existing_branch.delete(branch_index)?;
existing_branch.invalidate_cache();   // ← correctly invalidated here
``` [1](#0-0) [2](#0-1) 

**How the stale cache propagates to an incorrect root:**

`recompute` calls `relink_if_needed` then `get_node_key`. Inside `get_branch_key`:

```rust
// mod.rs lines 508-511
let cached_key = self.capacities.branch_nodes[branch_node.index()].cached_key;
if cached_key.is_empty() == false && any_mutation == false {
    return Ok((false, cached_key));   // ← stale key returned without recomputing
}
``` [3](#0-2) 

Because `cascade_delete_subtree_step` never cleared `cached_key`, the branch node's old hash (which still encodes the deleted child's contribution) is returned as-is. `relink_if_needed` also bails out early for this node:

```rust
// reattach.rs lines 97-100
if self.get_cached_key(node).is_empty() == false {
    // bail if cached
    return Ok(None);
}
``` [4](#0-3) 

So neither `relink_if_needed` nor `get_branch_key` will recompute the hash for the affected branch node. The stale hash propagates up to the root, producing an incorrect `interned_root_node_key`. [5](#0-4) 

---

### Impact Explanation

The Ethereum MPT state root is the primary public input committed to the ZK proof. An incorrect state root means:

1. The block's committed state root does not reflect the actual post-execution state.
2. The ZK proof (generated by the proving system using the same buggy code path) proves the incorrect root, so the proof is valid for the wrong state.
3. The L1 verifier accepts the proof, anchoring an incorrect state root on-chain.
4. Subsequent blocks are built on corrupted state, enabling double-spending or denial of legitimate withdrawals.

The stale cache encodes the hash of a deleted subtree, so the committed root is larger/different than the true post-deletion root — a direct state integrity violation.

---

### Likelihood Explanation

`cascade_delete_subtree` is triggered whenever a branch node reaches `num_occupied == 0` after a leaf deletion. The code explicitly defers branch consolidation (see the comment in `delete_from_branch.rs` lines 24–28), so a branch can legitimately hold a single child. Deleting that last child triggers the cascade. Any unprivileged user can delete their own storage slots or self-destruct a contract, making this reachable in normal operation. A targeted attacker can craft a sequence of storage writes followed by deletions to reliably trigger the condition. [6](#0-5) 

---

### Recommendation

Add `branch.invalidate_cache()` immediately after `branch.delete(found_child_idx)` in `cascade_delete_subtree_step`, mirroring the pattern in `delete_from_branch_node`:

```rust
branch.delete(found_child_idx)?;
branch.invalidate_cache();   // ADD THIS LINE
``` [7](#0-6)

### Proof of Concept

**Setup**: A trie that has been fully computed (all nodes have non-empty `cached_key`). The trie contains:

```
root (branch D, cached_key = H_D)
  └─ [nibble k] → branch B (cached_key = H_B)
       ├─ [nibble i] → branch A (cached_key = H_A)
       │    └─ [nibble j] → leaf L1
       └─ [nibble m] → leaf L2
```

**Step 1**: Delete `L2` from branch B via `delete_from_branch_node(B, m)`.
- `B.delete(m)` → `B.num_occupied = 1`
- `B.invalidate_cache()` → `B.cached_key = &[]`
- `B.num_occupied != 0`, so no cascade. Consolidation deferred.

**Step 2**: Delete `L1` from branch A via `delete_from_branch_node(A, j)`.
- `A.delete(j)` → `A.num_occupied = 0`
- `A.invalidate_cache()` → `A.cached_key = &[]`
- `A.num_occupied == 0` → `cascade_delete_subtree(A)` is called.

**Step 3**: Inside `cascade_delete_subtree_step(A)`:
- `parent = A.parent_node = B`
- `B.delete(i)` → `B.num_occupied = 0`
- **`B.invalidate_cache()` is NOT called** → `B.cached_key` remains `&[]` (already empty from Step 1 — but see variant below)
- `B.num_occupied == 0` → `Continue { node: B }`

**Step 4**: `cascade_delete_subtree_step(B)`:
- `parent = B.parent_node = D`
- `D.delete(k)` → `D.num_occupied` decremented
- **`D.invalidate_cache()` is NOT called** → `D.cached_key` remains `H_D` (stale, still encodes B's old subtree)
- `D.num_occupied > 0` → `Break
