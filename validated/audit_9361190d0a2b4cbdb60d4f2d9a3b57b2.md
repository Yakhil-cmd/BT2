### Title
Attacker-Controlled `puzzleHash` in Blockchain Notification Payload Redirects Counter-Offer XCH Payment to Wrong Destination - (File: `packages/gui/src/hooks/useBlockchainNotifications.tsx`)

### Summary
The Chia GUI trusts the `ph` (puzzleHash) field embedded in an on-chain notification message payload without any validation. An unprivileged attacker can send a crafted blockchain notification to any wallet address with their own puzzle hash in the `ph` field. When the victim clicks "Counter" on the incoming offer, the GUI pre-populates the counter-offer destination with the attacker's address, causing the victim to send the XCH notification fee to the attacker instead of the legitimate offer maker.

### Finding Description

The blockchain notification system allows any on-chain participant to send a notification message to a wallet address. The message payload is a JSON object containing an offer URL (`u`) and an optional puzzle hash (`ph`) representing the sender's address for counter-offer responses.

In `useBlockchainNotifications.tsx`, the `puzzleHash` is extracted directly from the attacker-controlled message payload with no format validation, no length check, and no on-chain verification that it corresponds to the actual sender of the notification coin:

```js
const { u: url, ph: puzzleHash } = data;
// puzzleHash is stored verbatim into the notification object
```

This `puzzleHash` is stored in the `Notification` object and flows through the entire counter-offer UI path:

1. `OfferIncomingTable.tsx` `handleCounterOffer()` converts it directly to a bech32m address and passes it as the `address` state to the offer builder:
   ```js
   const address = currencyCode && puzzleHash ? toBech32m(puzzleHash, currencyCode.toLowerCase()) : '';
   navigate('/dashboard/offers/builder', { state: { isCounterOffer: true, address, offer } });
   ```

2. `NotificationOffer.tsx` `handleClick()` passes it as `address` to the offer viewer:
   ```js
   address: 'puzzleHash' in notification ? notification.puzzleHash : undefined,
   ```

3. `NotificationSendDialog.tsx` `handleSubmit()` uses it as the `target` for `sendNotification`, which sends XCH on-chain:
   ```js
   const targetPuzzleHash = fromBech32m(address);
   await sendNotification({ target: targetPuzzleHash, amount: amountMojos, message: hexMessage, fee: feeMojos }).unwrap();
   ```

The analog to the external bug is exact: just as `witch.sol`'s second `grab` overwrites `vaultOwners[vaultId]` with the wrong owner (losing the original), the Chia GUI overwrites the legitimate offer-maker's identity with the attacker's puzzle hash from the notification payload, causing the counter-offer response (and its XCH fee) to be directed to the wrong party.

### Impact Explanation

When the victim clicks "Counter" on a crafted incoming offer notification:
- The XCH notification fee (default 0.0001 XCH, user-adjustable) is sent to the **attacker's address** instead of the legitimate offer maker's address.
- The victim's counter-offer notification (which includes the victim's own puzzle hash as their receive address) is delivered to the attacker, not the legitimate offer maker — the legitimate offer never receives the counter-offer.
- The attacker learns the victim's receive puzzle hash from the counter-offer notification payload.

This matches the High impact criterion: *"unsafe trust of notification state that causes a user to send the wrong asset, identity, amount, or destination."*

### Likelihood Explanation

Any unprivileged blockchain participant can send a notification coin to any wallet address. The `send_notification` RPC is a standard wallet operation. The attacker only needs to:
1. Know the victim's wallet address (publicly observable on-chain).
2. Craft a notification with a valid offer URL (can be any existing public offer) and their own puzzle hash in the `ph` field.
3. Pay the small notification fee (≥ 0.00000010 XCH).

No leaked keys, host compromise, or cryptographic break is required.

### Recommendation

Validate the `ph` field in `useBlockchainNotifications.tsx` before trusting it as a counter-offer destination:
- Verify it is a valid 32-byte hex string (64 hex characters).
- Optionally, cross-reference it against the coin that delivered the notification (the notification coin's puzzle hash is the actual sender's address and is available from the blockchain RPC), so the GUI can warn the user if the embedded `ph` does not match the coin sender.
- Display the resolved bech32m address prominently in the counter-offer UI and warn the user that this address came from the offer sender's self-reported payload, not from a verified on-chain identity.

### Proof of Concept

1. Attacker generates their own wallet puzzle hash: `aabbcc...` (32 bytes hex).
2. Attacker calls `send_notification` RPC targeting the victim's wallet address with message payload:
   ```json
   {"v":1,"t":1,"d":{"u":"https://dexie.space/offers/some-real-offer","ph":"aabbcc..."}}
   ```
3. Victim's GUI receives the notification via `useBlockchainNotifications.tsx` and stores `puzzleHash = "aabbcc..."` in the notification object with no validation. [1](#0-0) 
4. Victim sees "You have a new offer" and clicks **Counter** in `OfferIncomingTable.tsx`. The `handleCounterOffer` function converts `"aabbcc..."` to a bech32m address and navigates to the offer builder with `address` = attacker's address. [2](#0-1) 
5. Victim fills in the counter-offer and clicks "Send Message" in `NotificationSendDialog`. The `handleSubmit` function calls `sendNotification` with `target = fromBech32m(attackerAddress)`, sending XCH to the attacker. [3](#0-2) 
6. The attacker receives the XCH notification fee and the victim's counter-offer notification (containing the victim's own puzzle hash). The legitimate offer maker receives nothing.

The `ph` field in the notification payload is defined as attacker-controlled data with no on-chain binding: [4](#0-3)

    try {
      const puzzleHash = 'puzzleHash' in notification ? notification.puzzleHash : undefined;
      const offerId =
        'offerURL' in notification
          ? notification.offerURL
          : 'offerData' in notification
            ? notification.offerData
            : undefined;
      const offerState = getOffer(offerId);

      if (!offerState || !puzzleHash || !currencyCode) {
        return;

      const address = currencyCode && puzzleHash ? toBech32m(puzzleHash, currencyCode.toLowerCase()) : '';

        return;

      navigate('/dashboard/offers/builder', {
        state: {
          referrerPath: location.pathname,
          isCounterOffer: true,
          address,
          offer,
        },
      });
