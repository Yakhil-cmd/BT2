### Title
Incorrect NFT Seller Net Proceeds Preview in TokenForNFT Offer Acceptance Flow - (File: packages/gui/src/components/offers/utils.ts)

### Summary
`calculateNFTRoyalties` in `utils.ts` hardcodes `nftSellerNetAmount = amount` (the full offered amount) with the correct net calculation commented out. For `TokenForNFT` offers (buyer offering tokens for an NFT), the seller's actual proceeds are `amount - royaltyAmount`, but both the "You will receive" summary row and the "Net Proceeds" breakdown display the full pre-royalty amount. An NFT seller viewing an imported offer sees an inflated proceeds figure and may accept an offer they would otherwise reject.

### Finding Description

In `calculateNFTRoyalties`:

```typescript
const nftSellerNetAmount: number = amount;
// : parseFloat(
//     (amount - parseFloat(royaltyAmountString) - makerFee).toFixed(12),
//   );
```

`nftSellerNetAmount` is unconditionally set to `amount` regardless of `exchangeType`. [1](#0-0) 

For `NFTForToken` (seller is maker, requests tokens), royalties are paid **on top** by the taker, so `nftSellerNetAmount = amount` is correct. For `TokenForNFT` (buyer is maker, offers tokens), the GUI's own tooltip confirms royalties are paid **from** the offered amount:

> "If the NFT has royalty payments enabled, those creator fees will be paid from the offered purchase price." [2](#0-1) 

And the "Net Proceeds" tooltip confirms the expected formula:

> "The net proceeds include the asking price, minus any associated creator fees." [3](#0-2) 

Despite these tooltips, the displayed value is `nftSellerNetAmount` = `amount` (the full amount, not net): [4](#0-3) 

Additionally, `overrideNFTSellerAmount` is derived from `nftSellerNetAmount` and injected into the "You will receive" summary row for the NFT seller:

```typescript
const overrideNFTSellerAmount =
    exchangeType === NFTOfferExchangeType.TokenForNFT
      ? assetType === OfferAsset.CHIA
        ? chiaToMojo(nftSaleInfo?.nftSellerNetAmount ?? 0)
        : catToMojo(nftSaleInfo?.nftSellerNetAmount ?? 0)
      : undefined;
``` [5](#0-4) 

This override is passed to `NFTOfferSummary` → `NFTOfferSummaryRow` → `OfferSummaryTokenRow`, where it replaces the displayed amount:

```typescript
const amount = overrideNFTSellerAmount ?? originalAmount;
``` [6](#0-5) 

Because `nftSellerNetAmount = amount` (wrong), `overrideNFTSellerAmount = chiaToMojo(amount)` equals the original mojo amount, so the "You will receive" row also shows the full amount instead of the net proceeds.

### Impact Explanation

An NFT seller viewing an imported `TokenForNFT` offer sees two incorrect figures:

1. **"You will receive"** in the purchase summary — shows `amount` (e.g., 10 XCH) instead of `amount - royaltyAmount` (e.g., 9 XCH for a 10% royalty NFT).
2. **"Net Proceeds"** in the detailed breakdown — same inflated value, with a tooltip explicitly promising the royalty-deducted figure.

The seller accepts the offer believing they will receive the full displayed amount. On-chain, they receive the correct (lower) amount. This constitutes displaying the wrong amount in the offer acceptance flow, causing a user to approve an offer based on incorrect financial information.

### Likelihood Explanation

Any NFT with a non-zero `royaltyPercentage` triggers this path. The `TokenForNFT` flow is a standard offer type reachable by any unprivileged user sharing an offer file or link. No special access is required; the victim only needs to open the offer in the GUI.

### Recommendation

In `calculateNFTRoyalties`, differentiate `nftSellerNetAmount` by `exchangeType`:

```typescript
const nftSellerNetAmount: number =
  exchangeType === NFTOfferExchangeType.TokenForNFT
    ? parseFloat((amount - parseFloat(royaltyAmountString)).toFixed(12))
    : amount;
``` [7](#0-6) 

### Proof of Concept

1. Mint or obtain an NFT with `royaltyPercentage = 1000` (10%).
2. A buyer creates a `TokenForNFT` offer: offers 10 XCH for the NFT.
3. The NFT seller opens the offer file in the GUI (`NFTOfferViewer`).
4. **Observed**: "You will receive: 10 XCH", "Net Proceeds: 10 XCH".
5. **Expected**: "You will receive: 9 XCH", "Net Proceeds: 9 XCH" (10 XCH − 10% royalty).
6. Seller accepts the offer; on-chain they receive 9 XCH, not 10 XCH as displayed.

export function calculateNFTRoyalties(
  amount: number,
  makerFee: number,
  royaltyPercentage: number,
  exchangeType: NFTOfferExchangeType,
): CalculateNFTRoyaltiesResult {
  const royaltyAmount: number = royaltyPercentage ? (royaltyPercentage / 100) * amount : 0;
  const royaltyAmountString: string = formatAmount(royaltyAmount);
  const nftSellerNetAmount: number = amount;
  // : parseFloat(
  //     (amount - parseFloat(royaltyAmountString) - makerFee).toFixed(12),
  //   );
  const totalAmount: number =
    exchangeType === NFTOfferExchangeType.NFTForToken ? amount + royaltyAmount : amount + makerFee + royaltyAmount;
  const totalAmountString: string = formatAmount(totalAmount);

  return {
    royaltyAmount,
    royaltyAmountString,
    nftSellerNetAmount,
    totalAmount,
    totalAmountString,
  };
}
```
