### Title
COA EVM-Side Sub-Threshold Deposit Permanently Locks Dust FLOW in `CadenceOwnedAccount` - (`fvm/evm/stdlib/contract.cdc`)

---

### Summary

`CadenceOwnedAccount.withdraw()` enforces a hard minimum of **1e10 attoFlow** before any FLOW can be returned to Cadence, but EVM-side transfers can credit a COA address with any arbitrary attoFlow amount — including amounts below that threshold. Any sub-threshold balance that lands in a COA cannot be withdrawn via Cadence, and if the amount is too small to cover EVM gas fees, `call()` also cannot recover it, permanently locking the dust.

---

### Finding Description

**Minimum withdrawal gate — `CadenceOwnedAccount.withdraw()`**

`withdraw()` delegates to `InternalEVM.withdraw`, which calls `AttoFlowBalanceIsValidForFlowVault`:

```go
// fvm/evm/types/balance.go:105-106
func AttoFlowBalanceIsValidForFlowVault(bal *big.Int) bool {
    return bal.Cmp(UFixToAttoConversionMultiplier) >= 0   // must be >= 1e10
}
```

Any requested withdrawal amount below `1e10` attoFlow panics with `ErrWithdrawBalanceRounding`. [1](#0-0) 

This gate is enforced in both the Go host function and the Cadence contract: [2](#0-1) [3](#0-2) 

**No minimum on EVM-side deposits to a COA address**

`CadenceOwnedAccount.deposit()` simply forwards to `EVMAddress.deposit()`:

```cadence
// fvm/evm/stdlib/contract.cdc:562-565
access(all)
fun deposit(from: @FlowToken.Vault) {
    self.address().deposit(from: <-from)
}
``` [4](#0-3) 

The Cadence path is bounded by UFix64 precision (minimum 1e10 attoFlow), but **a standard EVM transaction can send any attoFlow amount — including 1 attoFlow — directly to the COA's EVM address**. The EVM state is updated immediately; no Cadence gate exists for this path.

**All recovery paths are blocked for sub-threshold balances**

| Recovery path | Blocked when |
|---|---|
| `withdraw()` | Amount < 1e10 attoFlow → `ErrWithdrawBalanceRounding` |
| `call()` to move funds within EVM | EVM is paused (`!EVM.isPaused()` pre-condition) OR gas cost > balance |
| `deploy()` / `callWithSigAndArgs()` | Same `!EVM.isPaused()` pre-condition | [5](#0-4) [6](#0-5) 

When a COA holds only, say, 5 attoFlow:
- `withdraw(Balance(attoflow: 5))` → panics (5 < 1e10)
- `call(...)` → fails with insufficient funds for gas (5 attoFlow cannot cover any EVM gas cost)

The dust is permanently irrecoverable.

A second variant arises when a COA holds a balance that is **not a multiple of 1e10** (e.g., 1e10 + 5 attoFlow):
- `withdraw(Balance(attoflow: 1e10 + 5))` → panics (`roundedOff = true`)
- `withdraw(Balance(attoflow: 1e10))` → succeeds, leaving 5 attoFlow permanently stuck
- `withdraw(Balance(attoflow: 5))` → panics (below minimum) [7](#0-6) 

---

### Impact Explanation

Any attoFlow amount below 1e10 (up to 9,999,999,999 attoFlow ≈ 1e-8 FLOW) that reaches a COA via an EVM-side transfer is permanently unrecoverable to Cadence. The maximum per-incident loss is ~1e-8 FLOW (dust), but the condition is permanent and affects every COA that receives a non-multiple-of-1e10 EVM transfer. Aggregated across many COAs this represents a systematic, irreversible loss of on-chain assets.

---

### Likelihood Explanation

Any unprivileged EVM EOA can send an arbitrary attoFlow amount to any COA address in a single EVM transaction (via `EVM.run` or a direct EVM transfer). No special permissions are required. The condition is trivially triggerable by any EVM participant, intentionally or accidentally (e.g., EVM contracts that compute transfer amounts in attoFlow arithmetic may produce non-aligned remainders).

---

### Recommendation

**Option 1 — Enforce a minimum deposit at the EVM entry point.** In `EVMAddress.deposit()` (and the Go-level `Account.Deposit`), reject or round up any incoming balance that is not a multiple of `UFixToAttoConversionMultiplier` (1e10).

**Option 2 — Remove the minimum withdrawal floor.** Allow `withdraw()` to accept any non-zero amount and perform the UFix64 truncation, returning the truncated vault and discarding the remainder (or crediting it back). This mirrors the documented "rounding down" behaviour already described in the contract comments. [8](#0-7) 

---

### Proof of Concept

```cadence
// Step 1: Alice creates a COA (any unprivileged user)
let coa <- EVM.createCadenceOwnedAccount()
// coa.address() = 0xALICE_COA

// Step 2: An EVM EOA sends 5 attoFlow to 0xALICE_COA via EVM.run(...)
// This is a standard EVM value transfer; no Cadence gate applies.
// coa.balance().attoflow == 5

// Step 3: Alice attempts to withdraw
let bal = EVM.Balance(attoflow: 5)
let vault <- coa.withdraw(balance: bal)
// PANICS: "withdraw failed! smallest unit allowed to transfer is 1e10 attoFlow"
// (ErrWithdrawBalanceRounding from AttoFlowBalanceIsValidForFlowVault)

// Step 4: Alice attempts call() to move funds within EVM
// FAILS: COA has only 5 attoFlow, insufficient to pay any EVM gas cost.

// Result: 5 attoFlow permanently locked in coa, irrecoverable.
``` [9](#0-8) [3](#0-2)
