### Title
Unprivileged Caller Can Suppress Per-Block Reward Distribution via `update_rewards(disable_rewards: true)` - (File: src/staking/staking.cairo)

### Summary
Any non-zero address can call `update_rewards` on the staking contract with `disable_rewards: true`. Because `last_reward_block` is written to storage **before** the `disable_rewards` early-return guard, the call permanently consumes the per-block reward slot without distributing any rewards. A subsequent legitimate call in the same block reverts with `REWARDS_ALREADY_UPDATED`, causing stakers and delegators to lose all yield for that block.

### Finding Description
`StakingRewardsManagerImpl::update_rewards` is a public function gated only by `general_prerequisites` (contract not paused, caller not zero address). No role check restricts who may supply the `disable_rewards` flag. [1](#0-0) 

Inside the function, `last_reward_block` is unconditionally written to the current block number **before** the `disable_rewards` branch is evaluated: [2](#0-1) 

The guard that prevents a second call in the same block compares strictly: [3](#0-2) 

Because `last_reward_block` is a single global field shared across all stakers, one attacker call with `disable_rewards: true` for any valid staker address exhausts the reward slot for every staker in that block. [4](#0-3) 

### Impact Explanation
Every block in which the attacker front-runs the legitimate `update_rewards` call, all stakers and their delegators receive zero block rewards. Repeated across many blocks this constitutes permanent suppression of unclaimed yield for the entire protocol — matching the **High** impact category: *Permanent freezing of unclaimed yield*.

### Likelihood Explanation
Starknet sequencing is deterministic and mempool ordering is observable. An attacker needs only to submit `update_rewards(any_active_staker, disable_rewards: true)` with a higher fee than the legitimate reward-update transaction each block. The attacker spends gas but earns nothing; the damage falls entirely on stakers and delegators. The entry path requires no privileged key, no bridge compromise, and no external dependency — only a valid (non-zero) caller address.

### Recommendation
Remove the `disable_rewards` parameter from the public interface entirely, or gate it behind a privileged role (e.g., `only_app_governor` or `only_attestation_contract`). Alternatively, move the `last_reward_block.write` call to **after** the `disable_rewards` guard so that a suppressed call does not consume the block slot:

```cairo
// Correct ordering:
if disable_rewards || self.is_pre_consensus() {
    return;
}
self.last_reward_block.write(current_block_number);
// ... distribute rewards ...
```

### Proof of Concept

1. At block `N`, attacker calls:
   ```
   staking.update_rewards(any_active_staker_address, disable_rewards: true)
   ```
2. `last_reward_block` is written to `N`; no rewards are distributed.
3. The legitimate consensus/attestation reward-update call for block `N` reverts:
   ```
   assert!(current_block_number > self.last_reward_block.read(), ...)
   // N > N  →  false  →  REWARDS_ALREADY_UPDATED
   ```
4. All stakers and delegators receive zero yield for block `N`.
5. Repeating this every block permanently freezes all unclaimed yield accumulation.
