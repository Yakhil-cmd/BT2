### Title
OT-Based ECDSA `sign` Lacks Exact Participant-Count Enforcement, Enabling Split-View Secret-Key Extraction вЂ” (File: `src/ecdsa/ot_based_ecdsa/sign.rs`)

---

### Summary

The OT-based ECDSA `sign` function enforces only `participants.len() >= threshold`, not `participants.len() == threshold`. A malicious coordinator can invoke two parallel signing sessions using the **same** `RerandomizedPresignOutput` but with **different** participant subsets (both satisfying the lower-bound check). Because Lagrange coefficients differ between sessions, the coordinator can combine the two sets of signature shares to reconstruct individual secret shares and ultimately the aggregate secret key. The robust ECDSA variant explicitly closes this gap; the OT-based variant does not.

---

### Finding Description

In `src/ecdsa/ot_based_ecdsa/sign.rs`, the `sign` entry-point validates:

```rust
// ensure number of participants during the signing phase is >= threshold
if participants.len() < threshold {
    return Err(InitializationError::NotEnoughParticipantsForThreshold { вЂ¦ });
}
``` [1](#0-0) 

This is a **lower-bound** check only. There is no upper-bound check and no requirement that the signing set equals the presigning set.

By contrast, the robust ECDSA variant explicitly enforces an exact count to prevent split-view attacks:

```rust
// The next two conditions prevent split-view attacks
if participants.len() != robust_ecdsa_threshold {
    return Err(InitializationError::BadParameters(
        "the number of participants during signing must be exactly 2*max_malicious+1 вЂ¦"
    ));
}
``` [2](#0-1) 

The `RerandomizedPresignOutput` type carries only `(big_r, k, sigma)` вЂ” it does **not** embed the participant set used during presigning, and the rerandomization HKDF over `(X, Оµ, h, R, ПЃ)` does not include the participant set. [3](#0-2) 

Therefore the same `RerandomizedPresignOutput` can be legally passed to two `sign` invocations with different participant subsets. Each signer computes:

```
s_i = h В· О»_i(P) В· k_i  +  Rx В· О»_i(P) В· Пѓ_i
``` [4](#0-3) 

When the coordinator runs two sessions with subsets Pв‚Ѓ and Pв‚‚ (both в‰Ґ threshold), it collects:

- Session A: `s_i^A = h В· О»_i(Pв‚Ѓ) В· k_i + Rx В· О»_i(Pв‚Ѓ) В· Пѓ_i`
- Session B: `s_i^B = h В· О»_i(Pв‚‚) В· k_i + Rx В· О»_i(Pв‚‚) В· Пѓ_i`

Since `О»_i(Pв‚Ѓ)` and `О»_i(Pв‚‚)` are public, the coordinator has two linear equations in two unknowns `(k_i, Пѓ_i)` per participant. Solving yields the raw presignature shares, and from `Пѓ_i = О± В· x_i в€’ ОІ В· a_i + c_i` (where `О±`, `ОІ`, `a_i`, `c_i` are also recoverable from the presign transcript), the coordinator reconstructs the secret key shares `x_i` and thus the aggregate secret key `x`.

The security documentation for the robust variant explicitly names this class of attack and the required mitigation: [5](#0-4) 

No equivalent protection exists in the OT-based variant.

The OT-based README states "Each output is consumed **exactly once** (one-time use)" as a documentation requirement, but this is not enforced in code. [6](#0-5) 

---

### Impact Explanation

**Critical.** A malicious coordinator вЂ” reachable without any privileged key material, since any participant may be designated coordinator вЂ” can extract the aggregate secret signing key `x` by running two signing sessions with the same presignature and different participant subsets. This enables unauthorized creation of valid threshold signatures for arbitrary attacker-chosen messages, satisfying the Critical impact criterion: *Extraction, reconstruction, or disclosure of private signing shares or aggregate secret material.*

---

### Likelihood Explanation

**Medium.** The coordinator role requires no special privilege beyond being listed as a participant. The attack requires:
1. Controlling which participants are invited to each signing session (normal coordinator authority).
2. Reusing one `RerandomizedPresignOutput` across two sessions вЂ” the library provides no enforcement preventing this.
3. Collecting signature shares from both sessions before the protocol completes вЂ” straightforward for the coordinator since all non-coordinator shares are sent **only to the coordinator**. [7](#0-6) 

No external assumptions, leaked keys, or cryptographic breaks are required.

---

### Recommendation

Apply the same exact-count guard used in the robust ECDSA variant. In `src/ecdsa/ot_based_ecdsa/sign.rs`, after the existing lower-bound check, add:

```rust
if participants.len() != threshold {
    return Err(InitializationError::BadParameters(
        "signing participant count must equal threshold to prevent split-view attacks".to_string(),
    ));
}
```

Additionally, bind the participant set into the `RerandomizedPresignOutput` (or into the HKDF derivation of `Оґ`) so that a presignature rerandomized for one participant set cannot be reused with a different set. [8](#0-7)

### Proof of Concept

**Setup:** 3 participants `{P1, P2, P3}`, threshold = 2.

1. Run `presign` with `{P1, P2, P3}` to obtain `PresignOutput` shares `(big_r, k_i, Пѓ_i)` for each.
2. Coordinator creates `RerandomizedPresignOutput` with a fixed `(h, Оµ, ПЃ)` вЂ” identical for both sessions.
3. **Session A:** Coordinator calls `sign` with `participants = [P1, P2]`. Each signer sends their share only to the coordinator:
   - `s_1^A = h В· О»_1({P1,P2}) В· k_1 + Rx В· О»_1({P1,P2}) В· Пѓ_1`
   - `s_2^A = h В· О»_2({P1,P2}) В· k_2 + Rx В· О»_2({P1,P2}) В· Пѓ_2`
4. **Session B:** Coordinator calls `sign` with `participants = [P1, P3]` вЂ” the library accepts this because `2 >= threshold`. Each signer sends their share only to the coordinator:
   - `s_1^B = h В· О»_1({P1,P3}) В· k_1 + Rx В· О»_1({P1,P3}) В· Пѓ_1`
   - `s_3^B = h В· О»_3({P1,P3}) В· k_3 + Rx В· О»_3({P1,P3}) В· Пѓ_3`
5. For participant P1, the coordinator now has two equations in `(k_1, Пѓ_1)` with known Lagrange coefficients. Solving: `k_1 = (s_1^A В· О»_1(Pв‚‚)вЃ»В№ в€’ s_1^B В· О»_1(Pв‚ѓ)вЃ»В№) / (h В· (1 в€’ О»_ratio) + Rx В· (1 в€’ О»_ratio))` (exact algebra follows from the linear system).
6. With `k_1` and `Пѓ_1` recovered, and using `Пѓ_i = О± В· x_i в€’ ОІ В· a_i + c_i` where `О±`, `ОІ` are broadcast during presigning, the coordinator solves for `x_1` (the secret key share of P1).
7. Repeating for P2 and P3 yields all shares; Lagrange interpolation at 0 reconstructs the master secret key `x`. [1](#0-0) [4](#0-3)

    participants: &[Participant],
    coordinator: Participant,
    threshold: impl Into<ReconstructionLowerBound>,

    msg_hash: Scalar,

    let threshold = usize::from(threshold.into());
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }

    let participants =
        ParticipantList::new(participants).ok_or(InitializationError::DuplicateParticipants)?;

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }

    // ensure number of participants during the signing phase is >= threshold
    if participants.len() < threshold {
        return Err(InitializationError::NotEnoughParticipantsForThreshold {
            threshold,
            participants: participants.len(),
        });

        ctx.shared_channel(),
        participants,
        coordinator,
        me,
        public_key,
        presignature,
        msg_hash,

}

1. **Use exactly $N_1 = N_2 = 2t + 1$ participants for both presigning and signing.**

   * Do **not** allow $N_1 > 2t + 1$, and
   * Do **not** allow $N_2 < N_1$.
