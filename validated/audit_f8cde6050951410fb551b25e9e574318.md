Audit Report

## Title
Unauthenticated `refresh_buyer_tokens` Allows Forced SNS Swap Participation on Behalf of Arbitrary Principal - (`rs/sns/swap/canister/canister.rs`)

## Summary

The `refresh_buyer_tokens` canister endpoint resolves the buyer identity from the request body rather than from the authenticated caller. Any unprivileged ingress sender can supply a victim's principal in the `buyer` field, causing the swap canister to read the victim's ICP subaccount balance and register or increase the victim's participation without their consent. This locks the victim's ICP into the swap and prevents them from reclaiming accidentally-sent ICP via `error_refund_icp`.

## Finding Description

In `rs/sns/swap/canister/canister.rs` (L127–143), the `refresh_buyer_tokens` handler resolves the effective buyer from the request body:

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    PrincipalId::from_str(&arg.buyer).unwrap()  // attacker-controlled
};
```

No check is made that `arg.buyer == caller`. The resolved principal `p` is passed directly to `refresh_buyer_token_e8s` in `rs/sns/swap/src/swap.rs` (L1134–1312).

Inside `refresh_buyer_token_e8s`, the `buyer` argument is used to look up the ICP balance of `principal_to_subaccount(&buyer)` on the swap canister (L1153–1163), and if the balance meets `min_participant_icp_e8s`, the buyer's participation record is created or updated (L1285–1288):

```rust
self.buyers
    .entry(buyer.to_string())
    .or_insert_with(|| BuyerState::new(0))
    .set_amount_icp_e8s(new_balance_e8s);
```

The ticket system at L1250–1272 does **not** block this attack. The ticket check is conditional: if no ticket exists for the buyer, the code explicitly skips the check (`// If there exists no ticket for the buyer, the payment flow will simply ignore the ticket`). An attacker calling on behalf of a victim who has no open ticket bypasses this guard entirely.

The `confirmation_text` check at L1149–1150 validates the **caller's** supplied text against the SNS-configured value, which is publicly readable from swap state via `get_state`. An attacker trivially satisfies this check.

Once the victim is registered in `self.buyers`, `error_refund_icp` (L1925–1959) blocks any refund while the swap is open (only works in ABORTED or COMMITTED lifecycle), and also blocks refund post-close until `sweep_icp` has run, returning a "ICP cannot be refunded as principal X has Y ICP (e8s) in escrow" error.

## Impact Explanation

This is a **High** severity finding matching: *"Significant SNS security impact with concrete user or protocol harm."*

A victim who accidentally sent ICP to the swap subaccount and intended to reclaim it via `error_refund_icp` loses that ability permanently once an attacker calls `refresh_buyer_tokens` on their behalf — their ICP is committed to the swap and can only be recovered after finalization (as SNS tokens if committed, or as a refund after `sweep_icp` if aborted). The victim is forced into a financial commitment without consent. Additionally, an attacker can exhaust a victim's per-participant quota (`max_participant_icp_e8s`) before the victim acts, preventing them from controlling their own participation amount.

## Likelihood Explanation

The attack requires only: (1) the victim's principal ID (public on-chain), (2) the victim having ICP in their swap subaccount (observable on the public ICP ledger), and (3) the `confirmation_text` (readable from swap state via a public query). All three are trivially obtainable by any unprivileged ingress sender. The attack costs only standard IC ingress fees and is repeatable.

## Recommendation

Remove the `buyer` override and always use `caller_principal_id()`:

```rust
async fn refresh_buyer_tokens(arg: RefreshBuyerTokensRequest) -> RefreshBuyerTokensResponse {
    let p: PrincipalId = caller_principal_id();
    // ...
}
```

If third-party notification is a required use case, it should be implemented as a separate, explicitly permissioned endpoint that does not register participation on behalf of another principal.

## Proof of Concept

1. Victim `V` transfers `min_participant_icp_e8s` ICP to `swap_canister_subaccount(V)` on the ICP ledger, intending to call `refresh_buyer_tokens` themselves.
2. Attacker `A` queries `get_state` (public query) to read `confirmation_text`.
3. `A` sends an ingress update call to the swap canister as any principal:
   ```
   refresh_buyer_tokens({ buyer: V.to_text(), confirmation_text: Some("<text>") })
   ```
4. The swap canister reads the ICP balance of `principal_to_subaccount(V)`, finds it ≥ `min_participant_icp_e8s`, and registers `V` as a buyer.
5. `V`'s ICP is now committed. `V` cannot reclaim it via `error_refund_icp` (blocked by "in escrow" check at `swap.rs` L1955–1959) and must wait for swap finalization.

A deterministic integration test can be written using `PocketIc` or `StateMachine`: create a swap in `Open` state, transfer ICP to victim's subaccount, call `refresh_buyer_tokens` from a different principal with victim's principal in the `buyer` field, then assert the victim appears in `self.buyers` and that `error_refund_icp` returns the escrow precondition error.
