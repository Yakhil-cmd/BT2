### Title
Fee-on-Transfer NEP-141 Tokens Cause ERC-20 Over-Minting and Bridge Insolvency - (`engine/src/engine.rs`)

---

### Summary

Both `receive_erc20_tokens` and `receive_base_tokens` in `engine/src/engine.rs` unconditionally trust `args.amount` from the `ft_on_transfer` callback as the actual amount received by Aurora. For fee-on-transfer NEP-141 tokens, the actual amount credited to Aurora's account is `args.amount - fee`, but Aurora mints the full `args.amount` in ERC-20 tokens (or credits the full amount as ETH balance). This creates a persistent insolvency: the ERC-20 supply on Aurora exceeds the NEP-141 reserves held by the Aurora contract, and the last users to exit cannot redeem their tokens.

---

### Finding Description

The `ft_on_transfer` entrypoint in `engine/src/contract_methods/connector.rs` is called by a NEP-141 token contract after `ft_transfer_call`. It reads `args.amount` from the callback payload and passes it directly to either `receive_base_tokens` or `receive_erc20_tokens`: [1](#0-0) 

Inside `receive_erc20_tokens`, the nominal `args.amount` is extracted and used verbatim as the mint quantity: [2](#0-1) 

This value is then passed to `setup_receive_erc20_tokens_input`, which encodes it as the `amount` argument to the ERC-20 `mint` selector: [3](#0-2) 

The same pattern applies to `receive_base_tokens`, which credits `args.amount` directly to the recipient's ETH balance without verifying the actual balance change: [4](#0-3) 

For a fee-on-transfer NEP-141 token, the token contract deducts a fee during the transfer so that Aurora's actual on-chain NEP-141 balance increases by only `args.amount - fee`. Aurora never queries its own NEP-141 balance before and after the transfer; it simply trusts the callback-supplied `args.amount`. The result is that Aurora mints `args.amount` ERC-20 tokens while holding only `args.amount - fee` NEP-141 tokens in reserve.

---

### Impact Explanation

**Critical — Insolvency / Permanent fund freeze.**

Each deposit of a fee-on-transfer NEP-141 token inflates the ERC-20 supply by exactly the fee amount relative to the NEP-141 reserve. After N deposits the shortfall is `N × fee`. When users attempt to exit (bridge ERC-20 back to NEP-141), the Aurora contract calls `ft_transfer` on the NEP-141 contract. Once the cumulative shortfall exceeds the reserve, the last cohort of exit attempts will fail because Aurora cannot transfer tokens it does not hold. Those users' ERC-20 balances become permanently frozen inside Aurora with no redemption path.

---

### Likelihood Explanation

**Medium.** Any unprivileged user can deploy or interact with a fee-on-transfer NEP-141 token and call `ft_transfer_call` targeting Aurora. The `deploy_erc20_token` entrypoint is permissionless for registering new NEP-141↔ERC-20 pairs. No admin action or key compromise is required. The attacker-controlled entry path is:

1. Deploy (or use an existing) fee-on-transfer NEP-141 token.
2. Call `ft_transfer_call` on that token with Aurora as the receiver.
3. The NEP-141 contract invokes `ft_on_transfer` on Aurora with `amount = X`.
4. Aurora mints `X` ERC-20 tokens but holds only `X - fee` NEP-141 tokens.
5. Repeat to accumulate shortfall.

---

### Recommendation

Before minting, verify the actual NEP-141 balance change rather than trusting `args.amount`. Concretely, in `receive_erc20_tokens` and `receive_base_tokens`, query Aurora's NEP-141 balance (or ETH connector balance) before and after the transfer and use the observed delta as the mint amount. Alternatively, explicitly reject tokens whose `ft_on_transfer`-reported amount does not match the observed balance increase, or document and enforce that only standard (non-fee-on-transfer) NEP-141 tokens may be bridged.

---

### Proof of Concept

1. Deploy a NEP-141 token `fee_token.near` that deducts 10% on every transfer (i.e., receiver gets 90% of the specified amount).
2. Register it with Aurora: call `deploy_erc20_token` with `nep141 = "fee_token.near"`.
3. Call `ft_transfer_call` on `fee_token.near`:
   - `receiver_id = "aurora"`
   - `amount = 1000`
   - `msg = "<recipient_evm_address>"`
4. `fee_token.near` credits Aurora with `900` tokens (deducts 100 as fee), then calls `aurora::ft_on_transfer` with `amount = 1000`.
5. Aurora executes `receive_erc20_tokens`:
   - `let amount = args.amount.as_u128();` → `amount = 1000`
   - Calls `setup_receive_erc20_tokens_input(&recipient, 1000)` → mints **1000** ERC-20 tokens.
6. Aurora's NEP-141 reserve for `fee_token.near` = **900**; ERC-20 supply = **1000**. Shortfall = 100.
7. After 10 such deposits (total NEP-141 reserve = 9000, ERC-20 supply = 10000), the last 1000 ERC-20 tokens cannot be redeemed. Those holders' funds are permanently frozen. [5](#0-4) [1](#0-0)

```rust
    pub fn receive_erc20_tokens<P: PromiseHandler>(
        &mut self,
        token: &AccountId,
        args: &FtOnTransferArgs,
        current_account_id: &AccountId,
        handler: &mut P,
    ) -> Result<Option<SubmitResult>, ContractError> {
        let amount = args.amount.as_u128();
        // Parse message to determine recipient
        let mut recipient = {
            // The message should contain the recipient EOA address.
            let message = args.msg.strip_prefix("0x").unwrap_or(&args.msg);
            // Recipient - 40 characters (Address in hex without '0x' prefix)
            if message.len() < 40 {
                return Err(ParseOnTransferMessageError::WrongMessageFormat.into());
            }
            let mut address_bytes = [0; 20];
            hex::decode_to_slice(&message[..40], &mut address_bytes)
                .map_err(|_| ParseOnTransferMessageError::WrongMessageFormat)?;
            Address::from_array(address_bytes)
        };

        if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
            && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
        {
            recipient = fallback_address;
        }

        let erc20_token = get_erc20_from_nep141(&self.io, token)?;
        let erc20_admin_address = current_address(current_account_id);
        let result = self
            .call(
                &erc20_admin_address,
                &erc20_token,
                Wei::zero(),
                setup_receive_erc20_tokens_input(&recipient, amount),
                u64::MAX,
                Vec::new(), // TODO: are there values we should put here?
                Vec::new(),
                handler,
            )
            .and_then(submit_result_or_err)?;

        sdk::log!("Mint {amount} ERC-20 tokens for: {}", recipient.encode());
```

```rust
pub fn setup_receive_erc20_tokens_input(recipient: &Address, amount: u128) -> Vec<u8> {
    let selector = ERC20_MINT_SELECTOR;
    let tail = ethabi::encode(&[
        ethabi::Token::Address(recipient.raw().0.into()),
        ethabi::Token::Uint(amount.into()),
    ]);

    [selector, tail.as_slice()].concat()
}
```
