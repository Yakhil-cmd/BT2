### Title
Gateway Unconditionally Skips `__validate__` Signature Check for Invoke Transactions with Nonce=1 When Any Account Transaction Exists in Mempool — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

`skip_stateful_validations` in the gateway's stateful validator skips the `__validate__` entry-point call (the account's signature check) for any invoke transaction whose nonce is 1 and whose sender has *any* transaction in the mempool or a recent block. An attacker who observes a victim's `deploy_account` transaction in the mempool can immediately submit a forged invoke transaction (nonce=1, arbitrary/invalid signature) for the victim's address. The gateway admits it without signature verification. The forged transaction occupies the nonce=1 slot in the mempool, blocking the victim's legitimate first invoke transaction.

### Finding Description

`skip_stateful_validations` is called from `run_pre_validation_checks` and returns `true` (skip) when all three conditions hold:

1. The transaction is an `Invoke` transaction.
2. `tx.nonce() == Nonce(Felt::ONE)`.
3. `account_nonce == Nonce(Felt::ZERO)` (account not yet deployed on-chain).
4. `mempool_client.account_tx_in_pool_or_recent_block(sender_address)` returns `true`. [1](#0-0) 

When `skip_stateful_validations` returns `true`, `run_pre_validation_checks` propagates that value back to `extract_state_nonce_and_run_validations`: [2](#0-1) 

`run_validate_entry_point` then constructs `ExecutionFlags { validate: !skip_validate, … }` and calls `blockifier_validator.validate(account_tx)`: [3](#0-2) 

Inside `StatefulValidator::perform_validations`, when `execution_flags.validate == false`, the function returns `Ok(())` immediately after `perform_pre_validation_stage`, without ever calling the `__validate__` entry point: [4](#0-3) 

The `__validate__` entry point is the account contract's signature-verification function. Skipping it means the gateway never checks whether the submitter actually controls the account.

The check `account_tx_in_pool_or_recent_block` is designed to detect whether a `deploy_account` transaction is pending for the sender, but it is a coarse check: it returns `true` for *any* transaction from that address in the mempool or recent block. An attacker who sees the victim's `deploy_account` transaction in the mempool satisfies condition 4 without owning the victim's private key.

**Attack path:**

1. Victim broadcasts `deploy_account` (nonce=0) → admitted to mempool.
2. Attacker observes this via the public mempool API.
3. Attacker crafts an `Invoke` transaction: `sender_address = victim_address`, `nonce = 1`, arbitrary calldata, **invalid/forged signature**.
4. Gateway evaluates `skip_stateful_validations`: all four conditions are met → `skip_validate = true` → `__validate__` is never called → transaction is admitted to the mempool.
5. Victim's legitimate invoke (nonce=1, valid signature) arrives at the gateway. The mempool already holds a nonce=1 transaction for this address; the victim's transaction is rejected as a duplicate nonce.
6. Batcher eventually executes the attacker's forged invoke. The blockifier runs `__validate__` with `validate: true` (default execution flags), the signature check fails, and the transaction is rejected/reverted. However, the nonce slot was consumed in the mempool, and the victim must resubmit.
7. The attacker can repeat step 3 every block, creating a persistent denial-of-service against the victim's first post-deployment invoke.

### Impact Explanation

The gateway admits an invoke transaction with an invalid (forged) signature into the mempool. This directly matches the High impact category: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."* The forged transaction occupies the victim's nonce=1 slot, blocking their legitimate first invoke transaction. The attacker can repeat this indefinitely at low cost (the forged transaction is never executed, so no fees are charged to the attacker).

### Likelihood Explanation

The attack requires only:
- Monitoring the public mempool for `deploy_account` transactions (trivially observable).
- Submitting a single forged invoke transaction per block.

No privileged access, no special contract, no cryptographic capability is needed. The attack window is one block (the time between `deploy_account` submission and commitment), but it can be repeated every block. Likelihood is **Medium**.

### Recommendation

The `skip_stateful_validations` bypass was introduced for UX (allowing a user to submit `deploy_account + invoke` atomically). The fix should ensure that only the *legitimate owner* of the account can benefit from the skip. Two options:

1. **Require the invoke transaction to be submitted in the same gateway request as the deploy_account transaction** (i.e., bundle them and validate them together), rather than relying on a mempool presence check.
2. **Do not skip `__validate__` at the gateway level.** Instead, allow the mempool to accept future-nonce transactions (nonce=1 when account nonce=0) without requiring gateway-level signature validation, and let the blockifier enforce the signature check at execution time. The UX goal (accepting the invoke before the deploy is committed) can still be achieved by relaxing the nonce gap rule rather than skipping signature verification.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
