### Title
Unprotected `execute_scheduled` Allows Any Caller to Force-Execute Scheduled XCC Promises, Permanently Freezing User NEAR - (File: etc/xcc-router/src/lib.rs)

---

### Summary

The `execute_scheduled` function in the XCC router contract has no caller restriction. Any NEAR account can call it with any valid nonce, immediately removing the stored promise from state and dispatching it. A malicious actor can front-run this call at a moment when the target contract will reject it (e.g., paused, wrong state, insufficient storage). Because the promise is deleted from `scheduled_promises` before the cross-contract call resolves, a failed call leaves the user's unwrapped NEAR permanently stranded in the router with no recovery path.

---

### Finding Description

The `Router` contract in `etc/xcc-router/src/lib.rs` exposes a two-step XCC flow:

1. **`schedule`** — restricted to the parent (Aurora Engine). Stores a `PromiseArgs` under a monotonically increasing nonce and increments the nonce counter.
2. **`execute_scheduled`** — explicitly open to anyone. Removes the stored promise and dispatches it.

```rust
/// It is intentional that this function can be called by anyone (not just the parent).
/// There is no security risk to allowing this function to be open because it can only
/// act on promises that were created via `schedule`.
#[payable]
pub fn execute_scheduled(&mut self, nonce: U64) {
    let Some(promise) = self.scheduled_promises.remove(&nonce.0) else {
        env::panic_str("ERR_PROMISE_NOT_FOUND")
    };
    let promise_id = Self::promise_create(promise);
    env::promise_return(promise_id);
}
```

The developer comment asserts safety because promise *content* is parent-controlled. However, it ignores that promise *timing* is now attacker-controlled.

The XCC flow that precedes `execute_scheduled` converts the user's wNEAR into real NEAR and deposits it into the router account (via `withdraw_wnear_to_router`). That NEAR is then forwarded to the target contract as `attached_balance` when the promise executes. If the cross-contract call fails (target paused, storage not registered, wrong state, etc.), NEAR is refunded to the router — but the promise entry is already gone. The router exposes no general NEAR withdrawal function; the only egress paths are `execute_scheduled` (requires a stored promise) and `send_refund` (restricted to the parent as a callback, sends a fixed 2 NEAR). The user's wNEAR is already burned; the NEAR is permanently stranded. [1](#0-0) [2](#0-1) [3](#0-2) 

---

### Impact Explanation

**Permanent freezing of user funds — Critical.**

When a user initiates a delayed XCC with attached NEAR (e.g., a storage deposit or token transfer requiring yoctoNEAR), the engine:
1. Burns the user's wNEAR ERC-20 balance.
2. Unwraps it to real NEAR and transfers it to the router sub-account.
3. Stores the promise in `scheduled_promises`.

If an attacker calls `execute_scheduled` at a moment the target contract rejects the call, the NEAR is refunded to the router but the promise record is deleted. The user cannot re-schedule (only the parent/engine can call `schedule`), and the router has no withdrawal function for arbitrary balances. The user's funds are permanently frozen inside the router sub-account. [4](#0-3) [5](#0-4) 

---

### Likelihood Explanation

**Low-to-Medium.**

- All scheduled promises and their nonces are observable on-chain (NEAR state is public; the `schedule` call emits a log `"Promise scheduled at nonce {}"` making discovery trivial).
- Many NEAR contracts have pausable states, storage-registration requirements, or time-gated windows. An attacker who knows the target contract can reliably choose a moment the call will fail.
- The attacker pays only gas; no capital is at risk on their side.
- The attack is especially practical against DeFi or governance contracts that are routinely paused for upgrades. [6](#0-5) 

---

### Recommendation

Restrict `execute_scheduled` to the parent account (Aurora Engine) or to the EVM address that owns the router sub-account. If open execution is desired for liveness, add a fallback recovery function callable by the parent that can re-insert a failed promise or refund the stranded NEAR to the originating EVM address. At minimum, do not delete the promise from storage before confirming the cross-contract call succeeded (use a callback pattern).

---

### Proof of Concept

1. Alice (EVM user) calls the XCC precompile with `CrossContractCallArgs::Delayed(promise)` where `promise.attached_balance = 1 NEAR` targeting a pausable NEAR DeFi contract.
2. The engine calls `schedule` on Alice's router sub-account, storing the promise at nonce `N`. The engine also unwraps 1 wNEAR → 1 NEAR and deposits it into the router.
3. Bob (attacker) observes the `"Promise scheduled at nonce N"` log on-chain.
4. Bob waits until the target DeFi contract is paused (e.g., during a routine upgrade window).
5. Bob calls `router_account.execute_scheduled({"nonce": "N"})` with sufficient gas.
6. `execute_scheduled` removes the promise from `scheduled_promises` and dispatches the cross-contract call.
7. The target contract is paused; it rejects the call and refunds 1 NEAR to the router.
8. Alice's promise is gone from storage. She cannot re-schedule (only the engine can call `schedule`). Her 1 wNEAR is burned; 1 NEAR is permanently stranded in the router. [7](#0-6) [8](#0-7)

    pub fn schedule(&mut self, #[serializer(borsh)] promise: PromiseArgs) {
        self.assert_preconditions();

        let nonce = self.nonce.get().unwrap_or_default();
        self.scheduled_promises.insert(nonce, promise);
        self.nonce.set(&(nonce + 1));

        near_sdk::log!("Promise scheduled at nonce {}", nonce);
    }
