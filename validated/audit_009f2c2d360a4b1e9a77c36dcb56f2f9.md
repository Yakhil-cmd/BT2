### Title
Unbounded Loop in `calculate_rewards` Over Delegator Balance Trace Enables Permanent Freezing of Unclaimed Yield — (`src/pool/pool.cairo`)

---

### Summary

The `calculate_rewards` function in `src/pool/pool.cairo` contains an **explicitly acknowledged unbounded loop** that iterates over a delegator's `pool_member_epoch_balance` trace. Every call to `add_to_delegation_pool` or `exit_delegation_pool_intent` in a **new epoch** appends a new checkpoint to this trace. A delegator who makes many balance changes across many epochs without claiming rewards accumulates an arbitrarily large trace. When `claim_rewards` is eventually called, the loop iterates over every unclaimed entry since the last checkpoint, potentially exceeding the Starknet transaction gas limit and permanently freezing the delegator's unclaimed yield.

---

### Finding Description

**Root cause — the unbounded loop:**

In `src/pool/pool.cairo`, the internal `calculate_rewards` function iterates over the entire `pool_member_epoch_balance` trace from `entry_to_claim_from` to the current epoch:

```cairo
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
    if pool_member_checkpoint.epoch() >= until_epoch {
        break;
    }
    let to_sigma = self.find_sigma(pool_member_checkpoint, curr_epoch: until_epoch);
    rewards += compute_rewards_rounded_down(...);
    entry_to_claim_from += 1;
}
``` [1](#0-0) 

The developers themselves flag this: *"This loop is unbounded but unlikely to exceed gas limits."*

**How the trace grows:**

Each call to `set_member_balance` inserts a new checkpoint into the trace keyed by `current_epoch + K`. The `insert` function only appends a **new** entry when the key (epoch) differs from the last stored key; otherwise it updates in place. [2](#0-1) 

`set_member_balance` is called (directly or via `increase_member_balance`) by:
- `enter_delegation_pool` — on first delegation
- `add_to_delegation_pool` — on every top-up
- `exit_delegation_pool_intent` — on every exit intent
- `enter_delegation_pool_from_staking_contract` — on pool switch [3](#0-2) 

Each of these operations in a **different epoch** appends one new entry. A delegator who calls `add_to_delegation_pool` (minimum 1 wei) once per epoch, or alternates between `add_to_delegation_pool` and `exit_delegation_pool_intent` across epochs, grows the trace by one entry per epoch.

**The `entry_to_claim_from` cursor does not fully mitigate this:**

`pool_member_info.entry_to_claim_from` is updated after each successful `claim_rewards` call, so the loop only covers entries since the last claim. However, if the delegator never calls `claim_rewards` (or cannot afford to), the cursor stays at 0 and the loop covers the entire trace. [4](#0-3) 

**`claim_rewards` is the only path to collect yield:**

There is no alternative mechanism to collect rewards in smaller batches or skip entries. The entire unclaimed range must be processed in a single transaction. [5](#0-4) 

---

### Impact Explanation

If the `pool_member_epoch_balance` trace grows large enough between two consecutive `claim_rewards` calls, the transaction will exceed the Starknet per-transaction gas limit. Because there is no partial-claim mechanism, the delegator's accumulated unclaimed yield becomes **permanently unclaimable** — matching the allowed impact: **Permanent freezing of unclaimed yield**.

Additionally, each iteration of the loop calls `find_sigma`, which itself performs storage reads against the `cumulative_rewards_trace`, compounding the per-iteration cost. [6](#0-5) 

---

### Likelihood Explanation

The access control on `add_to_delegation_pool` restricts callers to the pool member or their `reward_address`:

```cairo
assert!(
    caller_address == pool_member || caller_address == pool_member_info.reward_address,
    "{}",
    Error::CALLER_CANNOT_ADD_TO_POOL,
);
``` [7](#0-6) 

This means a third-party attacker **cannot** inflate another delegator's trace. The issue is therefore self-inflicted or requires a compromised/malicious `reward_address`. However:

1. A delegator who actively manages their position (frequent top-ups or partial exits across many epochs) without regularly claiming rewards will naturally accumulate a large trace over time.
2. A malicious `reward_address` (e.g., a contract set by the pool member) can call `add_to_delegation_pool` with 1 wei per epoch to inflate the trace at minimal cost, then the pool member's `claim_rewards` becomes permanently blocked.
3. The developers' own comment acknowledges the loop is unbounded, indicating awareness of the risk.

Likelihood is **medium**: the scenario requires either long-term neglect of reward claims combined with active balance management, or a compromised reward address.

---

### Recommendation

1. **Introduce a `max_entries` parameter** to `claim_rewards` so the loop is bounded per call, allowing delegators to claim rewards in multiple transactions.
2. **Alternatively**, checkpoint rewards at each balance change so that `claim_rewards` only needs to read the latest checkpoint rather than iterating the full trace.
3. **Enforce a maximum trace length** by requiring `claim_rewards` to be called before further balance changes are permitted once the trace exceeds a threshold.

---

### Proof of Concept

```
1. Delegator Alice enters pool with 1 STRK.
2. For each of N epochs:
   a. Alice calls add_to_delegation_pool(pool_member: alice, amount: 1)
      → appends one new entry to pool_member_epoch_balance trace
   b. Epoch advances (time passes)
3. Alice never calls claim_rewards.
4. After N epochs, pool_member_epoch_balance trace has N entries.
5. Alice calls claim_rewards(pool_member: alice).
   → calculate_rewards loops N times, each iteration reading storage + calling find_sigma
   → if N is large enough, transaction exceeds gas limit
   → Alice's accumulated unclaimed yield is permanently frozen
```

The minimum cost per epoch to Alice is 1 wei + gas for `add_to_delegation_pool`. The `entry_to_claim_from` cursor provides no relief because it was never advanced (no prior `claim_rewards` call). [8](#0-7)

    fn insert(
        self: StoragePath<Mutable<PoolMemberBalanceTrace>>, key: Epoch, value: PoolMemberBalance,
    ) -> (PoolMemberBalance, PoolMemberBalance) {
        let checkpoints = self.checkpoints;

        let len = checkpoints.len();
        if len == Zero::zero() {
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
            return (Zero::zero(), value);
        }

        // Update or append new checkpoint.
        let mut last = checkpoints[len - 1].read();
        let prev = last.value;
        if last.key == key {
            last.value = value;
            checkpoints[len - 1].write(last);
        } else {
            // Checkpoint keys must be non-decreasing.
            assert!(last.key < key, "{}", TraceErrors::UNORDERED_INSERTION);
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
        }
        (prev, value)
    }
