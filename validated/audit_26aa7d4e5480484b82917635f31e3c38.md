### Title
`protocolUnpausePool` Writes Wrong Pause Level, Leaving Pool Permanently Paused After Protocol Unpause — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`protocolUnpausePool` always writes `pauseLevel = 1` (admin-paused) instead of `0` (active). When the protocol pauses an active pool (`0 → 2`) and later unpauses it (`2 → 1`), the pool remains paused. All swaps and liquidity operations continue to revert with `PoolPaused`. The pool admin must separately call `unpausePool` to restore functionality — an action the protocol cannot force and the admin may not know is required.

---

### Finding Description

`protocolPausePool` accepts pools at either level 0 (active) or level 1 (admin-paused) and unconditionally sets them to level 2: [1](#0-0) 

`protocolUnpausePool` then unconditionally writes level 1, regardless of what the pre-pause level was: [2](#0-1) 

The pool's `_checkNotPaused` guard rejects any non-zero level: [3](#0-2) 

So after `protocolUnpausePool`, `pauseLevel == 1` and every call to `swap`, `addLiquidity`, or `removeLiquidity` still reverts.

The only recovery path is `unpausePool` (pool admin), which requires `cur == 1` and sets to 0: [4](#0-3) 

The protocol has no way to call this function — it is `onlyPoolAdmin`. The factory owner cannot force the admin to act.

---

### Impact Explanation

After a `protocolPausePool` → `protocolUnpausePool` cycle on a previously active pool:

- `swap` reverts (`PoolPaused`) — traders cannot execute.
- `addLiquidity` / `removeLiquidity` revert (`PoolPaused`) — LPs cannot withdraw principal.
- `collectFees` is callable (no pause guard), but fee accrual stops because no swaps occur.

The pool is functionally bricked until the pool admin independently discovers the issue and calls `unpausePool`. If the admin is a multisig with a slow signing process, or a contract without that call path, the pool may remain unusable indefinitely. This constitutes broken core pool functionality causing loss of access to LP assets and swap settlement.

---

### Likelihood Explanation

The trigger is a routine, privileged-but-valid protocol action: the factory owner pauses an active pool (level 0) for an emergency and later unpauses it. This is the primary intended use of `protocolPausePool`. The bug fires on every such cycle where the pool was at level 0 before the pause. No attacker is needed; the protocol owner's own legitimate action causes the stuck state.

---

### Recommendation

Store the pre-pause level before writing level 2, and restore it on unpause. The simplest correct fix is to record the previous level in factory storage:

```solidity
mapping(address => uint8) public prePausePauseLevel;

function protocolPausePool(address pool) external onlyOwner nonReentrant {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 0 && cur != 1) revert InvalidPauseTransition(cur, 2);
    prePausePauseLevel[pool] = cur;          // store original
    IMetricOmmPoolFactoryActions(pool).setPause(2);
}

function protocolUnpausePool(address pool) external onlyOwner nonReentrant {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 2) revert InvalidPauseTransition(cur, 0);
    uint8 restore = prePausePauseLevel[pool]; // use original
    delete prePausePauseLevel[pool];
    IMetricOmmPoolFactoryActions(pool).setPause(restore);
}
```

---

### Proof of Concept

1. Pool is deployed and active: `pauseLevel == 0`.
2. Factory owner calls `protocolPausePool(pool)`.
   - Reads `cur = 0`, passes the `cur != 0 && cur != 1` guard.
   - Calls `pool.setPause(2)` → `pauseLevel = 2`.
3. Emergency resolved. Factory owner calls `protocolUnpausePool(pool)`.
   - Reads `cur = 2`, passes the `cur != 2` guard.
   - Calls `pool.setPause(1)` → `pauseLevel = 1`. ← **wrong value written**
4. Any user calls `pool.swap(...)`.
   - `_checkNotPaused()` evaluates `pauseLevel != 0` → `true` → reverts `PoolPaused`.
5. Pool remains paused. Protocol owner has no function to set it to 0. Only the pool admin calling `unpausePool` can recover — an action the protocol cannot compel.
