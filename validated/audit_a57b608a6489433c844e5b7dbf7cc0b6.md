### Title
Unprivileged Caller Can Permanently Suppress Per-Block Rewards via Global `last_reward_block` Gate — (`src/staking/staking.cairo::update_rewards`)

---

### Summary

`update_rewards` is documented as restricted to "Only starkware sequencer" but the implementation enforces **no caller check**. Any address can call it first in a block with `disable_rewards: true`, which atomically (1) writes `last_reward_block` to the current block and (2) skips reward distribution. Every subsequent call in that block reverts with `REWARDS_ALREADY_UPDATED`. The attacker can repeat this every block, permanently zeroing out block rewards for any targeted staker.

---

### Finding Description

`general_prerequisites` — the only guard called at the top of `update_rewards` — checks only that the contract is unpaused and the caller is non-zero: [1](#0-0) 

There is no sequencer check, no role gate, and no caller-identity assertion anywhere in `update_rewards`: [2](#0-1) 

`last_reward_block` is a **global** (not per-staker) storage variable. It is written unconditionally — before the `disable_rewards` branch — so even a no-op call with `disable_rewards: true` consumes the block's reward slot: [3](#0-2) 

After that write, any legitimate call in the same block fails:

```
assert!(current_block_number > self.last_reward_block.read(), REWARDS_ALREADY_UPDATED)
```

The spec explicitly states the intended access control: [4](#0-3) 

The interface exposes the function to any caller: [5](#0-4) 

---

### Impact Explanation

An attacker who front-runs every block with `update_rewards(victim, disable_rewards: true)` causes the victim staker to receive **zero block rewards** for every block the attacker targets. Because `last_reward_block` is global, a single call per block is sufficient to suppress rewards for **all** stakers simultaneously (only one staker address needs to be valid). Rewards for skipped blocks are never retroactively credited — they are permanently lost. This matches **High: Permanent freezing of unclaimed yield**.

---

### Likelihood Explanation

- No privilege required; any EOA or contract can call `update_rewards`.
- The attacker only needs to submit one transaction per block before the legitimate sequencer call.
- On Starknet, transaction ordering within a block is sequencer-controlled, but the sequencer itself is not the only entity that can submit transactions. A griefing attacker can submit the hostile call in every block at negligible cost.
- The attack is fully automatable and requires no special knowledge beyond a valid `staker_address`.

---

### Recommendation

Add a sequencer-only (or governance-whitelisted) caller check at the top of `update_rewards`, consistent with the spec's stated access control. For example:

```rust
assert!(
    get_caller_address() == self.sequencer_address.read(),
    "{}",
    Error::CALLER_IS_NOT_SEQUENCER,
);
```

Alternatively, restrict the `IStakingRewardsManager` interface so it is only callable from a trusted address stored at deployment time.

---

### Proof of Concept

1. Deploy two active stakers (`staker_A`, `staker_B`) past the K-epoch activation window with consensus rewards enabled.
2. In block N, attacker calls `update_rewards(staker_A, disable_rewards: true)`.
   - `last_reward_block` is set to N.
   - No rewards are distributed.
3. Legitimate sequencer attempts `update_rewards(staker_A, disable_rewards: false)` in block N → reverts `REWARDS_ALREADY_UPDATED`.
4. Repeat for every block. After M blocks, both stakers have accrued zero rewards despite being fully eligible.
5. Compare against a baseline where the attacker is absent: stakers accumulate the expected per-block reward each block.

The existing test suite already demonstrates that a `disable_rewards: true` call in a block blocks any subsequent call in the same block: [6](#0-5)
