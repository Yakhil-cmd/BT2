### Title
`PresignOutput<C>` Derives `Clone`, Enabling Nonce Reuse and Secret Key Extraction — (`File: src/frost/mod.rs`)

---

### Summary

The `PresignOutput<C>` struct in `src/frost/mod.rs` derives `Clone`, allowing the secret nonce material it contains to be duplicated and reused across multiple signing sessions. This directly contradicts the protocol's documented requirement that each presignature must be consumed exactly once. Reusing a presignature with different messages produces two FROST signatures sharing the same nonce, from which the signer's secret key share can be algebraically recovered.

---

### Finding Description

`PresignOutput<C>` is defined at line 36–41 of `src/frost/mod.rs` with `#[derive(Debug, Clone, Serialize, Deserialize, Eq, PartialEq)]`:

```rust
#[derive(Debug, Clone, Serialize, Deserialize, Eq, PartialEq)]
pub struct PresignOutput<C: Ciphersuite + Send + 'static> {
    pub nonces: SigningNonces<C>,
    pub commitments_map: BTreeMap<Identifier<C>, SigningCommitments<C>>,
}
``` [1](#0-0) 

The `nonces: SigningNonces<C>` field holds the secret per-session nonce material generated during the presigning phase. The signing functions `do_sign_coordinator_v2` and `do_sign_participant` consume `PresignOutput` by value (moved), which would normally prevent reuse. However, because `PresignOutput` derives `Clone`, any caller can trivially duplicate the struct before passing it to the signing function, retaining a live copy of the secret nonces for a second signing session. [2](#0-1) 

The `do_sign_participant` function even wraps the nonces in `Zeroizing::new(presignature.nonces)` to zero them after use, but this protection is entirely bypassed if the caller cloned the `PresignOutput` before the call — the clone retains the unzeroed nonces. [3](#0-2) 

The library's own documentation states "Each output is consumed **exactly once** (one-time use)" and the presign comment warns "it's crucial that a presignature is never reused." [4](#0-3) 

The `Clone` derive directly undermines this invariant at the type level, creating the same class of bug as M-11: a state that is documented as one-time-use but is not enforced as such, allowing repeated exploitation.

---

### Impact Explanation

FROST nonce reuse is a well-known catastrophic failure mode. If the same `SigningNonces` are used to produce two signature shares for different messages `m1` and `m2`:

- `s1 = nonce + H(R, pk, m1) * x_i`
- `s2 = nonce + H(R, pk, m2) * x_i`
- `s1 − s2 = (H(R, pk, m1) − H(R, pk, m2)) * x_i`
- `x_i = (s1 − s2) / (H(R, pk, m1) − H(R, pk, m2))`

The participant's secret signing share `x_i` is directly recovered. This matches the **Critical** allowed impact: *"Extraction, reconstruction, or disclosure of private signing shares, aggregate secret material, presign secrets, nonce material, or confidential derived secrets."*

---

### Likelihood Explanation

The `Clone` derive is present in the public API. Any application layer that:

1. **Retries a failed signing session** by cloning the presign output before the first attempt (a natural defensive programming pattern),
2. **Stores presign outputs in a cloneable container** (e.g., `Vec`, `HashMap`) and reuses them across sessions, or
3. **Is driven by a malicious coordinator** who initiates two signing sessions before the participant discards the presign output,

will trigger nonce reuse. The `Serialize`/`Deserialize` derives additionally allow the presign output — including secret nonces — to be serialized to disk or transmitted over a network, further widening the attack surface. The `commitments_map` field is legitimately public and needs serialization; the `nonces` field does not.

---

### Recommendation

Remove `Clone` from `PresignOutput<C>`. The struct should be non-`Clone` to enforce at the type level that it can only be consumed once. If serialization of the `commitments_map` is needed for network transport, split the struct:

```rust
// Public, freely cloneable/serializable
pub struct PresignCommitments<C: Ciphersuite> {
    pub commitments_map: BTreeMap<Identifier<C>, SigningCommitments<C>>,
}

// Secret, non-Clone, non-Serialize
pub struct PresignOutput<C: Ciphersuite + Send + 'static> {
    pub(crate) nonces: SigningNonces<C>,
    pub commitments: PresignCommitments<C>,
}
```

This mirrors the mitigation in M-11 (`vlt.notional = 0` after redemption): the secret state is cleared/invalidated after use and cannot be re-entered.

async fn do_presign<C: Ciphersuite + Send>(
    mut chan: SharedChannel,
    participants: ParticipantList,
    me: Participant,
    signing_share: SigningShare<C>,
    mut rng: impl CryptoRngCore,
) -> Result<PresignOutput<C>, ProtocolError> {
    // Round 1
    let mut commitments_map: BTreeMap<Identifier<C>, SigningCommitments<C>> = BTreeMap::new();

    // Creating two commitments and corresponding nonces
    let (nonces, commitments) = commit(&signing_share, &mut rng);
    commitments_map.insert(me.to_identifier()?, commitments);

    let commit_waitpoint = chan.next_waitpoint();
    // Sending the commitments to all
    chan.send_many(commit_waitpoint, &commitments)?;

    // Collecting the commitments
    for (from, commitment) in recv_from_others(&chan, commit_waitpoint, &participants, me).await? {
        commitments_map.insert(from.to_identifier()?, commitment);
    }

    Ok(PresignOutput {
        nonces,
        commitments_map,
    })
}
