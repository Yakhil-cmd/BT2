### Title
`setPoolAdminFeeDestination` Changes Fee Recipient Without Collecting Accrued Fees, Causing Permanent Misdirection of Owed Admin Fees — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`setPoolAdminFeeDestination` updates `poolAdminFeeDestination[pool]` without first flushing accrued admin fees to the old destination. Every other fee-config mutator in the factory (`setPoolAdminFees`, `setPoolProtocolFee`) calls `collectFees` before writing new state. `setPoolAdminFeeDestination` does not, so all fees that accrued under the old destination are silently redirected to the new one on the next `collectPoolFees` call.

---

### Finding Description

`setPoolAdminFees` (lines 408–435) and `setPoolProtocolFee` (lines 318–360) both follow the same safe pattern: collect at the current destination/rates first, then update state. [1](#0-0) 

`setPoolAdminFeeDestination` (lines 438–447) skips this step entirely: [2](#0-1) 

`collectPoolFees` (and every internal `collectFees` call) reads `poolAdminFeeDestination[pool]` at call time and passes it directly to the pool's `collectFees`: [3](#0-2) 

The pool's `collectFees` then transfers the entire accrued admin share to whatever `adminFeeDestination_` it receives: [4](#0-3) 

Because `collectPoolFees` is permissionless, the sequence is:

1. Fees accrue to the old destination over time.
2. Pool admin calls `setPoolAdminFeeDestination(pool, newDest)` — no flush occurs.
3. Anyone (keeper, bot, or the admin themselves) calls `collectPoolFees(pool)`.
4. All previously accrued admin fees are transferred to `newDest`.
5. The old destination receives nothing and has no on-chain recourse.

There is no mechanism to recover the misdirected fees after the fact.

---

### Impact Explanation

The old `adminFeeDestination` permanently loses all admin fees (both spread and notional components) that accrued between the last collection and the destination change. The magnitude equals the full uncollected admin fee balance at the time of the call — potentially significant for high-volume pools with infrequent collection schedules. The new destination receives fees it was not entitled to earn.

---

### Likelihood Explanation

The trigger is a routine, valid semi-trusted admin action: updating the treasury or revenue-sharing address. No exploit setup is required. The loss is automatic and irreversible the moment `collectPoolFees` is called after the destination change, which can happen immediately (permissionless) or be front-run by any observer.

---

### Recommendation

Mirror the pattern used by `setPoolAdminFees` and `setPoolProtocolFee`: flush accrued fees to the old destination before updating the pointer.

```solidity
function setPoolAdminFeeDestination(address pool, address newAdminFeeDestination)
    external
    override
    nonReentrant
    onlyPoolAdmin(pool)
{
    if (newAdminFeeDestination == address(0)) revert InvalidAdminFeeDestination();

+   // Flush accrued fees to the current destination before switching
+   PoolFeeConfig memory c = poolFeeConfig[pool];
+   IMetricOmmPoolCollectFees(pool).collectFees(
+       c.protocolSpreadFeeE6,
+       c.adminSpreadFeeE6,
+       c.protocolNotionalFeeE8,
+       c.adminNotionalFeeE8,
+       poolAdminFeeDestination[pool]   // old destination
+   );

    poolAdminFeeDestination[pool] = newAdminFeeDestination;
    emit PoolAdminFeeDestinationUpdated(pool, newAdminFeeDestination);
}
```

---

### Proof of Concept

```
State:
  poolAdminFeeDestination[pool] = oldTreasury
  Accrued admin spread surplus (token0): 1000 units
  Accrued notional fee (token1): 500 units

Step 1: admin calls setPoolAdminFeeDestination(pool, newTreasury)
  → poolAdminFeeDestination[pool] = newTreasury
  → No collectFees call; 1000 token0 + 500 token1 still sit on the pool

Step 2: keeper calls collectPoolFees(pool)
  → factory reads poolAdminFeeDestination[pool] == newTreasury
  → pool.collectFees(..., adminFeeDestination_ = newTreasury)
  → transferToken0(newTreasury, 1000)
  → transferToken1(newTreasury, 500)

Result:
  oldTreasury received: 0  (lost 1000 token0 + 500 token1)
  newTreasury received: 1000 token0 + 500 token1 (unearned)
```

The old destination has no on-chain path to recover the misdirected amounts.
