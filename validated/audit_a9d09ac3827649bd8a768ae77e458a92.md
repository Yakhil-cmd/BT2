### Title
Gateway Admission Bypasses `__validate__` Signature Check for Nonce-1 Invoke via Overly Broad `skip_stateful_validations` Guard — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function skips the blockifier `__validate__` entry-point (the account's signature-verification step) for any Invoke transaction whose nonce is exactly 1 and whose sender's on-chain nonce is 0, provided `account_tx_in_pool_or_recent_block` returns `true`. That membership check is satisfied by **any** transaction in the pool for the target address — not exclusively a `DeployAccount`. An unprivileged attacker who observes that address X has any transaction in the mempool (e.g., a legitimate `DeployAccount` or a nonce-0 Invoke from a deployed-but-fresh account) can submit a nonce-1 Invoke for X carrying an **invalid signature** and have it admitted to the mempool without ever running `__validate__`.

---

### Finding Description

**Root cause — `skip_stateful_validations`** [1](#0-0) 

The guard fires when `tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO)` and delegates the skip decision entirely to:

```rust
mempool_client.account_tx_in_pool_or_recent_block(tx.sender_address())
``` [2](#0-1) 

`account_tx_in_pool_or_recent_block` returns `true` if **any** transaction for that address is in `tx_pool` or in the committed-nonce state — it does not distinguish `DeployAccount` from `Invoke`.

**How the skip propagates to blockifier**

When `skip_validate = true`, `run_validate_entry_point` sets `execution_flags.validate = false`: [3](#0-2) 

Inside `StatefulValidator::perform_validations`, the early-return path is taken before `__validate__` is ever called: [4](#0-3) 

**Mempool `validate_tx` does not check signatures**

The mempool's own validation only checks for duplicate hashes, duplicate nonces, and nonce ordering — no signature check: [5](#0-4) 

**Attack path (step-by-step)**

1. Legitimate user submits `DeployAccount` for address X (nonce 0). It passes `__validate_deploy__` and lands in the mempool.
2. Attacker observes X in the mempool (via P2P propagation or RPC snapshot).
3. Attacker crafts an `Invoke` for X with `nonce = 1` and an **arbitrary/invalid signature**.
4. Gateway stateless validation passes (signature-length check only, not cryptographic validity). [6](#0-5) 

5. `validate_state_preconditions` passes: `account_nonce=0`, `tx_nonce=1`, within `max_allowed_nonce_gap=200`. [7](#0-6) 

6. `validate_by_mempool` passes: no duplicate hash, nonce is valid.
7. `skip_stateful_validations` returns `true` because `account_tx_in_pool_or_recent_block(X) == true`.
8. `run_validate_entry_point` runs with `validate: false`; `__validate__` is **never called**.
9. The invalid-signature Invoke is forwarded to the mempool and accepted.

The same path applies to any **deployed** account whose on-chain nonce is still 0 (class hash set, no committed transactions) and that has a nonce-0 Invoke already in the pool.

---

### Impact Explanation

An unprivileged attacker can inject Invoke transactions carrying invalid (or entirely absent) signatures into the mempool for any account address that currently has nonce 0 and at least one pending transaction. The invariant **"every Invoke admitted to the mempool has passed account-level signature verification"** is broken at the gateway admission stage. The invalid transaction will fail at `__validate__` during block execution and be rejected, but it occupies mempool capacity and block-proposal bandwidth in the interim. At scale (many fresh accounts with pending `DeployAccount`s), this enables a targeted mempool-pollution / admission-bypass attack.

Impact category: **High — Mempool/gateway admission accepts invalid transactions before sequencing.**

---

### Likelihood Explanation

- No privileged access required; any network participant can submit RPC transactions.
- `DeployAccount` transactions are publicly observable via P2P propagation and RPC.
- The attacker only needs to know the target address and submit one crafted Invoke per target.
- The window of opportunity lasts from the moment the `DeployAccount` enters the mempool until it is committed (potentially many blocks).

---

### Recommendation

Replace the overly broad `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a **`DeployAccount` transaction** is pending for the sender address. The mempool should expose a dedicated query such as `deploy_account_in_pool(address) -> bool` that inspects the transaction type, rather than returning `true` for any pooled transaction. This mirrors the intent stated in the code comment ("either it has a deploy_account transaction") and closes the bypass for deployed-but-fresh accounts and for any future transaction type that could satisfy the current check without implying account ownership.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
