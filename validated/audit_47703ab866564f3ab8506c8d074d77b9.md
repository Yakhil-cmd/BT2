### Title
`skip_stateful_validations` Bypasses `__validate__` Signature Check via Non-Deploy-Account Mempool Presence - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The gateway's `skip_stateful_validations` function is intended to skip the `__validate__` entry-point call only when a `deploy_account` transaction is pending in the mempool (UX feature for the deploy+invoke flow). However, the guard it relies on — `account_tx_in_pool_or_recent_block` — returns `true` for **any** transaction type in the pool, not exclusively `DeployAccount`. An unprivileged attacker can therefore inject an invoke transaction with nonce=1 carrying an arbitrary/forged signature for any account whose on-chain nonce is 0 and that has any pending transaction in the mempool, and the gateway will admit it without ever calling `__validate__`.

### Finding Description

`skip_stateful_validations` in `crates/apollo_gateway/src/stateful_transaction_validator.rs` fires when three conditions hold simultaneously:

1. The incoming transaction is an `Invoke`.
2. `tx.nonce() == Nonce(Felt::ONE)`.
3. `account_nonce == Nonce(Felt::ZERO)` (from the fixed-block state reader).

When all three hold, it calls `account_tx_in_pool_or_recent_block` and returns that boolean directly as the skip decision: [1](#0-0) 

`account_tx_in_pool_or_recent_block` in the mempool checks only whether the account address appears in the pool or the committed-nonce state — it does **not** filter by transaction type: [2](#0-1) 

When `skip_validate` is `true`, `run_validate_entry_point` sets `validate: false`, so the blockifier's `StatefulValidator::perform_validations` never calls `__validate__`: [3](#0-2) 

The `validate_by_mempool` call that precedes `skip_stateful_validations` passes only nonce/tip/gas-price metadata (`ValidationArgs`) — it carries no signature field and performs no cryptographic check: [4](#0-3) 

Therefore, the only place where the signature is verified — the `__validate__` entry-point — is skipped whenever the account has **any** transaction in the mempool, not just a `DeployAccount`.

### Impact Explanation

An attacker who observes a victim's pending `deploy_account` transaction (nonce=0, account on-chain nonce=0) can submit an invoke with nonce=1 carrying a forged or empty signature. The gateway admits it without signature verification. Because the mempool enforces `DuplicateNonce` for the same (address, nonce) pair, the victim's legitimate nonce=1 invoke is subsequently rejected, blocking the victim's first post-deployment transaction. The admitted invalid transaction will fail at blockifier execution time (the blockifier does call `__validate__` during execution), but the admission decision itself is wrong: an unsigned transaction for an arbitrary account is accepted into the mempool.

Matching impact: **High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

### Likelihood Explanation

The attack requires only that the target account has nonce=0 on-chain and any transaction visible in the mempool. Every new account deployment satisfies this condition during the window between `deploy_account` submission and block commitment. The attack is fully unprivileged and requires only a standard gateway API call.

### Recommendation

Replace the type-agnostic `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a `DeployAccount` transaction is pending for the address. The mempool should expose a `deploy_account_tx_in_pool(address)` query, or `skip_stateful_validations` should inspect the transaction type of the pooled entry before deciding to skip `__validate__`.

### Proof of Concept

1. Victim submits `deploy_account` (nonce=0) for address `A`. It enters the mempool; `account_tx_in_pool_or_recent_block(A)` now returns `true`.
2. Attacker submits `invoke` with `sender_address=A`, `nonce=1`, `signature=[]` (empty/forged).
3. Gateway stateful path:
   - `get_nonce_from_state(A)` → `Nonce(0)` (account not yet committed).
   - `validate_nonce`: `0 ≤ 1 ≤ 0 + max_gap` → passes.
   - `validate_by_mempool`: passes (no signature field in `ValidationArgs`).
   - `skip_stateful_validations`: `tx.nonce()==1 && account_nonce==0` → calls `account_tx_in_pool_or_recent_block(A)` → `true` → returns `true`.
   - `run_validate_entry_point`: `validate = !true = false` → `__validate__` is **never called**.
4. Attacker's forged invoke is admitted to the mempool with nonce=1.
5. Victim's legitimate invoke with nonce=1 is rejected with `MempoolError::DuplicateNonce`.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
