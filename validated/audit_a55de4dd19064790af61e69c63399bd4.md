### Title
Unbounded `stakers` Vec Growth Causes Permanent Gas Exhaustion in `get_stakers` - (File: src/staking/staking.cairo)

### Summary

The `stakers` storage Vec in `staking.cairo` is append-only: stakers are **never removed** from it even after unstaking. The `get_stakers` function iterates over the **entire** Vec on every call. An unprivileged actor can inflate this Vec by registering many staker addresses with minimum stake and then unstaking them all, permanently increasing the iteration cost of every future `get_stakers` call.

### Finding Description

**Root cause — append-only `stakers` Vec:**

The storage declaration at line 169 explicitly documents the invariant:

```
/// Vector of staker addresses.
/// **Note**: Stakers are not removed from this vector when they unstake.
stakers: Vec<ContractAddress>,
``` [1](#0-0) 

**Unbounded full-range iteration in `get_stakers`:**

`get_stakers` (part of `IStakingConsensus`) iterates over every element ever appended to `stakers`, including all long-since-unstaked addresses:

```cairo
for staker_address_ptr in self.stakers.into_iter_full_range() {
    let staker_address = staker_address_ptr.read();
    if !self.is_staker_active(:staker_address, :epoch_id) {
        continue;   // inactive entries are skipped but still read
    }
    ...
}
``` [2](#0-1) 

Each inactive entry still costs a storage read (`staker_address_ptr.read()`) plus the `is_staker_active` check. There is no upper bound on the number of entries.

**Attacker-controlled entry path:**

1. Attacker creates N addresses, each funded with `min_stake`.
2. Each address calls `stake()` — appending its address to `stakers`.
3. Each address calls `unstake_intent()` then, after `exit_wait_window`, `unstake_action()` — recovering all funds.
4. The N addresses remain in `stakers` permanently.
5. Every subsequent call to `get_stakers` must iterate over all N ghost entries.

The attack is **economically near-free**: the attacker recovers all staked funds after the exit window. Only transaction fees are lost. The damage to the protocol is permanent.

### Impact Explanation

`get_stakers` is the consensus-layer interface used to enumerate active validators for a given epoch. Inflating `stakers` with thousands of ghost entries causes every `get_stakers` invocation to consume proportionally more execution resources. On Starknet, even view/read-only calls are subject to execution-step limits. A sufficiently large Vec will cause `get_stakers` to exceed those limits, making it impossible for the consensus layer to retrieve the active staker set. This constitutes **unbounded gas consumption** and, at sufficient scale, a **permanent denial-of-service** against the consensus integration.

**Severity: Medium** — matches "Griefing with no profit motive but damage to users or protocol; Unbounded gas consumption."

### Likelihood Explanation

- Requires no privileged access; any address that can afford `min_stake` (temporarily) can participate.
- Funds are fully recoverable after `exit_wait_window`, so the net cost is only gas fees.
- The effect is permanent and cumulative: each wave of registrations adds permanently to the iteration cost.
- The attack needs to be performed only once to cause lasting harm.

### Recommendation

Remove staker addresses from `stakers` when they fully unstake (i.e., after `unstake_action`), or replace the append-only `Vec` with a structure that supports deletion (e.g., a swap-and-pop pattern or a separate active-staker set). Alternatively, cap the maximum number of registered stakers or introduce a per-staker registration fee that is not refunded on unstake.

### Proof of Concept

1. Deploy the staking contract with `min_stake = M`.
2. Create 10,000 addresses, each funded with `M` STRK.
3. For each address `addr_i`:
   - Call `staking.stake(reward_address, operational_address, M, false)` from `addr_i`.
   - `stakers` Vec length increases by 1.
4. After `exit_wait_window` passes, for each `addr_i`:
   - Call `staking.unstake_intent()` then `staking.unstake_action()`.
   - All `M` STRK is returned; `addr_i` remains in `stakers`.
5. Call `staking.get_stakers(current_epoch)`.
   - The function iterates all 10,000 entries, reading storage and calling `is_staker_active` for each.
   - Execution step count scales linearly with 10,000 ghost entries.
6. Repeat step 2–4 to further inflate `stakers`; eventually `get_stakers` exceeds Starknet's per-call execution limit and reverts on every invocation. [3](#0-2) [1](#0-0)

        fn get_stakers(
            self: @ContractState, epoch_id: Epoch,
        ) -> Span<(ContractAddress, StakingPower, Option<PublicKey>, Option<PeerId>)> {
            let curr_epoch = self.get_current_epoch();
            assert!(
                curr_epoch <= epoch_id && epoch_id < curr_epoch + K.into(),
                "{}",
                Error::INVALID_EPOCH,
            );

            let (strk_total_stake, btc_total_stake) = self
                .get_total_staking_power_at_epoch(:epoch_id);

            let mut stakers: Array<
                (ContractAddress, StakingPower, Option<PublicKey>, Option<PeerId>),
            > =
                array![];
            for staker_address_ptr in self.stakers.into_iter_full_range() {
                let staker_address = staker_address_ptr.read();
                if !self.is_staker_active(:staker_address, :epoch_id) {
                    continue;
                }

                let staking_power = self
                    .get_staker_staking_power_at_epoch(
                        :staker_address, :epoch_id, :strk_total_stake, :btc_total_stake,
                    );
                if staking_power.is_zero() {
                    continue;
                }

                let public_key = self.get_public_key_at_epoch(:staker_address, :epoch_id);
                let peer_id = self.get_peer_id_at_epoch(:staker_address, :epoch_id);
                stakers.append((staker_address, staking_power, public_key, peer_id));
            }
            stakers.span()
        }
