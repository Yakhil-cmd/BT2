### Title
Unbounded Loop in `calculate_rewards` Enables Permanent Freezing of Unclaimed Pool Yield - (File: src/pool/pool.cairo)

### Summary
The `calculate_rewards` internal function in the Pool contract contains an explicitly acknowledged unbounded loop that iterates over a pool member's entire `pool_member_epoch_balance` trace. A pool member who accumulates many balance-change epochs without claiming rewards can grow this trace without bound. When they (or anyone on their behalf) eventually calls `claim_rewards`, the loop may exceed Starknet's per-transaction gas limit, permanently bricking their ability to claim accrued yield.

### Finding Description
`calculate_rewards` (lines 837–888 of `src/pool/pool.cairo`) iterates over every entry in `pool_member_epoch_balance` that falls between `entry_to_claim_from` and the current epoch:

```cairo
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
    if pool_member_checkpoint.epoch() >= until_epoch {
        break;
    }
    let to_sigma = self.find_sigma(pool_member_checkpoint, curr_epoch: until_epoch);
    rewards += compute_rewards_rounded_down(...);
    ...
    entry_to_claim_from += 1;
}
```

The trace grows by one entry per epoch in which the pool member modifies their balance. `set_member_balance` (line 718) calls `trace.insert(key: self.get_epoch_plus_k(), ...)`, which appends a new `PoolMemberBalanceCheckpoint` whenever the key differs from the last entry. `entry_to_claim_from` is only advanced when `claim_rewards` is successfully completed (line 358). If a pool member changes their balance every epoch but never claims, the gap between `entry_to_claim_from` and `pool_member_trace_length` grows without bound.

`calculate_rewards` is called unconditionally inside `claim_rewards` (lines 349–355) and `pool_member_info_v1` (lines 532–537). Both paths are blocked if the loop exceeds the gas limit.

### Impact Explanation
Once the trace is large enough that a single `claim_rewards` transaction exceeds Starknet's gas ceiling, the pool member's accrued rewards are permanently inaccessible. There is no partial-claim mechanism and no way to split the computation across multiple transactions. This constitutes **permanent freezing of unclaimed yield** (High).

### Likelihood Explanation
The trigger is entirely self-controlled: a pool member who calls `add_to_delegation_pool` or `exit_delegation_pool_intent` once per epoch and never calls `claim_rewards` will grow the trace linearly with time. Starknet epochs are short (roughly every few hours), so a moderately active delegator who defers claiming for months can accumulate hundreds of trace entries. Each loop iteration performs a storage read (`pool_member_trace.at(...)`) and a `find_sigma` call (another storage read into `cumulative_rewards_trace`), making each iteration expensive. The threshold for gas exhaustion is reachable in practice.

### Recommendation
1. **Introduce a partial-claim mechanism**: allow `claim_rewards` to accept a `max_entries` parameter so the loop can be bounded per call, with `entry_to_claim_from` advanced incrementally.
2. **Alternatively**, enforce a maximum number of unclaimed balance-change epochs (e.g., require claiming before a new balance change is accepted if the trace exceeds a threshold).
3. **At minimum**, remove the dismissive comment "unlikely to exceed gas limits" and replace it with a documented bound or a hard cap enforced in code.

### Proof of Concept
1. Pool member `A` joins a delegation pool via `enter_delegation_pool`.
2. Each epoch, `A` calls `add_to_delegation_pool` with a small amount. Each call invokes `set_member_balance` → `trace.insert(key: get_epoch_plus_k(), ...)`, appending a new `PoolMemberBalanceCheckpoint` because the epoch key advances each time.
3. `A` never calls `claim_rewards`, so `pool_member_info.entry_to_claim_from` stays at its initial value.
4. After `N` epochs, `pool_member_trace_length - entry_to_claim_from == N`.
5. `A` (or anyone) calls `claim_rewards`. The loop in `calculate_rewards` must iterate `N` times, each performing two storage reads. For large `N`, the transaction reverts with an out-of-gas error.
6. Every subsequent `claim_rewards` call also reverts. `A`'s accumulated yield is permanently frozen.

    fn insert(
        self: StoragePath<Mutable<PoolMemberBalanceTrace>>, key: Epoch, value: PoolMemberBalance,
    ) -> (PoolMemberBalance, PoolMemberBalance) {
        let checkpoints = self.checkpoints;

        let len = checkpoints.len();
        if len == Zero::zero() {
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
            return (Zero::zero(), value);
        }

        // Update or append new checkpoint.
        let mut last = checkpoints[len - 1].read();
        let prev = last.value;
        if last.key == key {
            last.value = value;
            checkpoints[len - 1].write(last);
        } else {
            // Checkpoint keys must be non-decreasing.
            assert!(last.key < key, "{}", TraceErrors::UNORDERED_INSERTION);
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
        }
        (prev, value)
    }
