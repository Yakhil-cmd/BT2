### Title
Gateway L2 Gas Price Admission Uses Stale `l2_gas_price` Instead of `next_l2_gas_price` — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The gateway's stateful admission check validates a transaction's `max_price_per_unit` against the **previous block's `l2_gas_price`** field. However, the actual execution block uses **`next_l2_gas_price`** — the EIP-1559-adjusted price stored in the same block header. These two values diverge every block. The mismatch causes the gateway to admit transactions that will fail at execution time (when the price rose) and to reject transactions that would have been valid (when the price fell). A TODO comment in the source code explicitly acknowledges the wrong field is being read.

---

### Finding Description

`validate_resource_bounds` in `StatefulTransactionValidator` reads the L2 gas price for its threshold check as follows:

```rust
// TODO(Arni): getnext_l2_gas_price from the block header.
let previous_block_l2_gas_price = self
    .gateway_fixed_block_state_reader
    .get_block_info()
    .await?
    .gas_prices
    .strk_gas_prices
    .l2_gas_price;
``` [1](#0-0) 

`GatewayFixedBlockSyncStateClient::get_block_info_from_sync_client` populates `BlockInfo.gas_prices.strk_gas_prices.l2_gas_price` from `block_header.l2_gas_price.price_in_fri` — the price that was **used inside** the previous block:

```rust
strk_gas_prices: GasPriceVector {
    ...
    l2_gas_price: block_header.l2_gas_price.price_in_fri.try_into()?,
},
``` [2](#0-1) 

`BlockHeaderWithoutHash` stores **two distinct** L2 gas price fields:

```rust
pub l2_gas_price: GasPricePerToken,      // price used in THIS block
pub next_l2_gas_price: GasPrice,          // EIP-1559-adjusted price for the NEXT block
``` [3](#0-2) 

`next_l2_gas_price` is computed by `calculate_next_l2_gas_price_for_fin` at decision time and written into the committed block header:

```rust
next_l2_gas_price: self.l2_gas_price,   // updated via update_l2_gas_price()
``` [4](#0-3) 

The consensus context's `l2_gas_price` is updated after each finalized block via `update_l2_gas_price`, which calls `calculate_next_l2_gas_price_for_fin` — the EIP-1559 formula: [5](#0-4) 

The proposer then sends this updated price in `ProposalInit`, and the batcher builds the execution block context from it. So the **actual execution** block uses `next_l2_gas_price` from the previous block header, while the **gateway admission** check uses `l2_gas_price` from the same header.

The same wrong price is also used when `run_validate_entry_point` builds the blockifier block context for gateway-side `__validate__` execution:

```rust
let mut block_info = self.gateway_fixed_block_state_reader.get_block_info().await?;
block_info.block_number = block_info.block_number.unchecked_next();
let block_context = BlockContext::new(block_info, ...);
``` [6](#0-5) 

**Concrete divergence scenario (EIP-1559 price increase):**

When block N's gas usage exceeds `gas_target`, `calculate_next_base_gas_price` raises the price:

```rust
let adjusted_price_u256 =
    if gas_used > gas_target { price_u256 + price_change } else { price_u256 - price_change };
``` [7](#0-6) 

So `next_l2_gas_price > l2_gas_price`. A transaction with `l2_gas_price ≤ max_price_per_unit < next_l2_gas_price`:

1. **Passes** `validate_tx_l2_gas_price_within_threshold` (gateway uses `l2_gas_price`)
2. **Passes** gateway blockifier `__validate__` (same wrong price in block context)
3. **Fails** `check_fee_bounds` at actual execution in block N+1 (uses `next_l2_gas_price`) [8](#0-7) 

The reverse also holds: when the block is under-target, `next_l2_gas_price < l2_gas_price`, and the gateway rejects transactions that would have been valid at execution.

---

### Impact Explanation

**High — Mempool/gateway admission accepts invalid transactions or rejects valid transactions before sequencing.**

Transactions admitted by the gateway with `max_price_per_unit` in the range `[l2_gas_price, next_l2_gas_price)` will be sequenced into the mempool and then fail `check_fee_bounds` at execution time, wasting block capacity and producing failed-transaction receipts. Conversely, valid transactions are incorrectly rejected at the gateway when the price is falling. The discrepancy grows with block utilization variance — high-traffic periods (where EIP-1559 raises the price most aggressively) produce the largest gap between the two values.

---

### Likelihood Explanation

This fires on every block where gas usage deviates from `gas_target`. The EIP-1559 formula adjusts the price every block, so the two fields diverge continuously in production. The TODO comment at line 229 confirms the developers are aware the wrong field is being read. Any user submitting a transaction with `max_price_per_unit` set to exactly the previous block's L2 gas price (a natural choice) will be affected when the price is rising.

---

### Recommendation

In `GatewayFixedBlockSyncStateClient::get_block_info_from_sync_client`, replace the `l2_gas_price` field population with `next_l2_gas_price` from the block header:

```rust
// Before (wrong):
l2_gas_price: block_header.l2_gas_price.price_in_fri.try_into()?,

// After (correct):
l2_gas_price: block_header.next_l2_gas_price.try_into()?,
``` [2](#0-1) 

This resolves the TODO at line 229 of `stateful_transaction_validator.rs` and aligns the gateway's admission price with the price the execution block will actually enforce.

---

### Proof of Concept

1. Block N executes with high utilization (`gas_used > gas_target`). EIP-1559 computes `next_l2_gas_price = P_N * (1 + delta)` where `delta > 0`. Block N header stores `l2_gas_price = P_N`, `next_l2_gas_price = P_N + ε`.

2. User submits an invoke transaction with `l2_gas.max_price_per_unit = P_N` (matching the previous block's price, a natural wallet default).

3. Gateway `validate_resource_bounds` checks `P_N >= threshold * P_N` → passes (with `min_gas_price_percentage = 100`).

4. Gateway `run_validate_entry_point` builds block context with `l2_gas_price = P_N` → `__validate__` passes.

5. Transaction enters mempool and is included in block N+1, which uses `l2_gas_price = P_N + ε`.

6. `check_fee_bounds` in `perform_pre_validation_stage` checks `P_N >= P_N + ε` → **fails** with `MaxGasPriceTooLow`. [8](#0-7) 

The transaction was admitted by the gateway but fails at execution, producing a failed receipt and wasting block space.
