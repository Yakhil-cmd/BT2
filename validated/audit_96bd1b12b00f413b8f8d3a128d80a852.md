### Title
Missing Caller Authorization in `interop_root_reporter_event_hook` Allows Unprivileged Injection of Interop Roots — (`system_hooks/src/event_hooks/interop_root_reporter.rs`)

---

### Summary

The `interop_root_reporter_event_hook` event hook performs a privileged state-mutating operation (`add_interop_root`) based solely on an event signature match, without verifying who triggered the event emission. Any user who can cause the contract at `L2_INTEROP_ROOT_STORAGE_ADDRESS` (`0x10008`) to emit an `InteropRootAdded` event with attacker-controlled topics can inject arbitrary interop roots into the system state. This is the direct analog of the missing `onlyOwner` modifier: a privileged action is reachable without caller authorization.

---

### Finding Description

All **call hooks** in ZKsync OS explicitly verify the `caller` field before performing any privileged action:

- `l1_messenger_hook`: `if caller != L1_MESSENGER_ADDRESS` [1](#0-0) 
- `set_bytecode_on_address_hook`: `if caller != CONTRACT_DEPLOYER_ADDRESS && caller != COMPLEX_UPGRADER_ADDRESS` [2](#0-1) 
- `contract_deployer_temp_hook`: `if caller != COMPLEX_UPGRADER_ADDRESS` [3](#0-2) 
- `mint_base_token_hook`: `if caller != L2_BASE_TOKEN_ADDRESS` [4](#0-3) 

The two **event hooks**, however, receive no `caller` parameter and perform **no emitter authorization check**:

```rust
pub fn interop_root_reporter_event_hook<S: EthereumLikeTypes>(
    topics: &arrayvec::ArrayVec<...>,
    data: &[u8],
    _caller_ee: u8,          // ← caller_ee is ignored
    system: &mut System<S>,
    resources: &mut S::Resources,
) -> Result<(), SystemError>
```

The hook checks only the event signature and data format, then unconditionally calls `system.io.add_interop_root(...)` with attacker-controlled `chain_id`, `block_or_batch_number`, and `root` values extracted from the event topics/data: [5](#0-4) 

The hook is registered at `L2_INTEROP_ROOT_STORAGE_ADDRESS_LOW` (`0x10008`): [6](#0-5) 

The `System::emit_event` dispatcher passes only the emitting address and event payload to the hook — no originating caller identity is forwarded: [7](#0-6) 

The same structural defect exists in `system_context_event_hook`, which calls `update_settlement_layer_chain_id` without caller verification: [8](#0-7) 

---

### Impact Explanation

`add_interop_root` writes attacker-controlled `(chain_id, block_or_batch_number, root)` tuples into the system's interop root storage. Interop roots are used to verify cross-chain message inclusion proofs. Injecting a fake root for an arbitrary chain ID and block number allows an attacker to:

1. Fabricate a valid-looking inclusion proof for a cross-chain message that was never sent.
2. Claim funds or trigger state changes on the destination chain backed by the forged root.

This is a direct unauthorized state mutation with potential for cross-chain fund theft.

---

### Likelihood Explanation

The attack path requires the contract deployed at `0x10008` (`L2_INTEROP_ROOT_STORAGE_ADDRESS`) to expose a publicly callable function that emits `InteropRootAdded(uint256,uint256,bytes32[])`. The OS-level hook enforces **no caller restriction** — it relies entirely on the system contract for access control. If the contract at `0x10008` has any publicly reachable code path that emits this event (e.g., a function without an `onlyOwner`/`onlyBootloader` guard), the hook is exploitable by any unprivileged transaction sender. The hook's unconditional trust in the event payload is the necessary vulnerable step.

---

### Recommendation

Add an emitter-identity check inside `interop_root_reporter_event_hook` (and `system_context_event_hook`) analogous to the caller checks in all call hooks. The `caller_ee` parameter already exists in the hook signature; the hook dispatch infrastructure should be extended to also pass the emitting contract's address so the hook can assert it equals the expected system contract address before performing any state mutation.

Alternatively, the `System::emit_event` dispatcher should pass the originating caller address to event hooks, mirroring the `ExternalCallRequest::caller` field available to call hooks.

---

### Proof of Concept

**Attacker-controlled entry path:**

1. Attacker submits a transaction calling any publicly accessible function on the contract at `0x10008` that emits:
   ```
   emit InteropRootAdded(attacker_chain_id, attacker_block_number, [attacker_root])
   ```
2. The EVM LOG opcode fires; `System::emit_event` is called with `address = 0x10008`.
3. `try_intercept_event` matches the registered hook for `L2_INTEROP_ROOT_STORAGE_ADDRESS_LOW`.
4. `interop_root_reporter_event_hook` verifies only the event signature (`INTEROP_ROOT_ADDED_EVENT_SIG`) and data length — **no caller check**.
5. `system.io.add_interop_root(...)` is called with attacker-supplied `root`, `chain_id`, `block_or_batch_number`.
6. The forged interop root is now committed to system state and can be used to prove fraudulent cross-chain messages.

The root cause — the missing caller/emitter authorization check — is located at: [9](#0-8)

```rust
pub fn interop_root_reporter_event_hook<S: EthereumLikeTypes>(
    topics: &arrayvec::ArrayVec<<S::IOTypes as SystemIOTypesConfig>::EventKey, MAX_EVENT_TOPICS>,
    data: &[u8],
    _caller_ee: u8,
    system: &mut System<S>,
    resources: &mut S::Resources,
) -> Result<(), SystemError>
where
{
    // First, ensure we're capturing the InteropRootAdded event
    if topics.is_empty() || topics[0].as_u8_array() != INTEROP_ROOT_ADDED_EVENT_SIG {
        return Ok(());
    }
    // Internal error if the data supplied doesn't match the expected value
    if data.len() != 96 {
        return Err(internal_error!("Interop root reporter event hook received bad data").into());
    }

    // Parse data
    let offset: u32 = match U256::from_be_slice(&data[..32]).try_into() {
        Ok(offset) => offset,
        Err(_) => {
            return Err(
                internal_error!("Interop root reporter event hook received bad offset").into(),
            );
        }
    };
    // This event is part of the system, but we check it anyways
    if offset != 32 {
        return Err(internal_error!("Interop root reporter event hook received bad offset").into());
    }

    let len: u32 = match U256::from_be_slice(&data[32..64]).try_into() {
        Ok(offset) => offset,
        Err(_) => {
            return Err(
                internal_error!("Interop root reporter event hook received bad length").into(),
            );
        }
    };
    // It should have exactly one side
    if len != 1 {
        return Err(internal_error!("Interop root reporter event hook received bad length").into());
    }
    // Validate topics length
    if topics.len() != 3 {
        return Err(internal_error!("Interop root reporter event hook received bad topics").into());
    }

    let root = Bytes32::from_array(data[64..96].try_into().unwrap());
    let chain_id = U256::from_be_bytes(topics[1].as_u8_array());
    let block_or_batch_number = U256::from_be_bytes(topics[2].as_u8_array());
    system.io.add_interop_root(
        ExecutionEnvironmentType::NoEE,
        resources,
        InteropRoot {
            root,
            block_or_batch_number,
            chain_id,
        },
    )?;

    Ok(())
```

    pub fn emit_event(
        &mut self,
        hooks: &mut HooksStorage<S, S::Allocator>,
        ee_type: ExecutionEnvironmentType,
        resources: &mut S::Resources,
        address: &<S::IOTypes as SystemIOTypesConfig>::Address,
        topics: &ArrayVec<<S::IOTypes as SystemIOTypesConfig>::EventKey, MAX_EVENT_TOPICS>,
        data: &[u8],
    ) -> Result<(), SystemError> {
        // First, emit the event using io subsystem
        self.io
            .emit_event(ee_type, resources, address, topics, data)?;

        // If successful, intercept event hook, if any
        if let Some(address_low) = address.try_into_low() {
            let _ = hooks.try_intercept_event(
                address_low,
                topics,
                data,
                ee_type as u8,
                self,
                resources,
            )?;
        }
        Ok(())
    }

pub fn system_context_event_hook<S: EthereumLikeTypes>(
    topics: &arrayvec::ArrayVec<<S::IOTypes as SystemIOTypesConfig>::EventKey, MAX_EVENT_TOPICS>,
    data: &[u8],
    caller_ee: u8,
    system: &mut System<S>,
    resources: &mut S::Resources,
) -> Result<(), SystemError>
where
{
    if topics.is_empty() {
        return Ok(());
    }
    // For now, we only capture the SettlementLayerChainIdUpdated event
    if topics[0].as_u8_array() == SL_CHAIN_ID_UPDATED_EVENT_SIG {
        new_sl_chain_id_event_hook(topics, data, caller_ee, system, resources)
    } else {
        Ok(())
    }
}

fn new_sl_chain_id_event_hook<S: EthereumLikeTypes>(
    topics: &arrayvec::ArrayVec<<S::IOTypes as SystemIOTypesConfig>::EventKey, MAX_EVENT_TOPICS>,
    data: &[u8],
    _caller_ee: u8,
    system: &mut System<S>,
    resources: &mut S::Resources,
) -> Result<(), SystemError>
where
{
    // Internal error if the data supplied isn't empty
    if !data.is_empty() {
        return Err(
            internal_error!("New SL chain id reporter event hook received bad data").into(),
        );
    }
    // Same if there's a mismatch in expected topics
    if topics.len() != 2 {
        return Err(
            internal_error!("New SL chain id reporter event hook received bad topics").into(),
        );
    }

    let new_sl_chain_id = U256::from_be_bytes(topics[1].as_u8_array());
    system.io.update_settlement_layer_chain_id(
        ExecutionEnvironmentType::NoEE,
        resources,
        new_sl_chain_id,
    )?;

    Ok(())
}
