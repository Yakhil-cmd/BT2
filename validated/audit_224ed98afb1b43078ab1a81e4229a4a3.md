### Title
Gateway resource-bounds check uses current block's `l2_gas_price` instead of `next_l2_gas_price`, causing wrong admission decisions - (File: crates/apollo_gateway/src/stateful_transaction_validator.rs)

### Summary

`StatefulTransactionValidator::validate_resource_bounds` reads `l2_gas_price` from the **current** (latest committed) block to gate incoming transactions. Transactions submitted to the gateway will be executed in the **next** block, which uses `next_l2_gas_price` — a separate field in the block header. When the two values diverge (which happens on every block with non-target utilization), the gateway either admits transactions that will fail during execution or rejects transactions that would succeed.

### Finding Description

In `validate_resource_bounds`, the validator reads the gas price reference from the latest block's `BlockInfo`:

```rust
let previous_block_l2_gas_price = self
    .gateway_fixed_block_state_reader
    .get_block_info()
    .await?
    .gas_prices
    .strk_gas_prices
    .l2_gas_price;
``` [1](#0-0) 

The code itself acknowledges the wrong field is being used with an explicit TODO:

```rust
// TODO(Arni): getnext_l2_gas_price from the block header.
``` [2](#0-1) 

`GatewayFixedBlockSyncStateClient::get_block_info_from_sync_client` converts the block header to `BlockInfo`, mapping `block_header.l2_gas_price.price_in_fri` to `strk_gas_prices.l2_gas_price`: [3](#0-2) 

The `next_l2_gas_price` field — which is stored in `StorageBlockHeader` and represents the price that will govern the **next** block — is never included in `BlockInfo` and is therefore inaccessible to the validator: [4](#0-3) 

The threshold computation then uses the wrong baseline:

```rust
let threshold = (gas_price_threshold_multiplier
    * previous_block_l2_gas_price.get().0)
    .to_integer();
if tx_l2_gas_price.0 < threshold { return Err(...) }
``` [5](#0-4) 

The default config has `validate_resource_bounds: true` and `min_gas_price_percentage: 100`, meaning the threshold is exactly `l2_gas_price` of the current block: [6](#0-5) 

### Impact Explanation

**Case 1 — gas price rising (`next_l2_gas_price > l2_gas_price`):**
A transaction with `max_price_per_unit = l2_gas_price` passes the gateway check (threshold = `l2_gas_price`), is admitted to the mempool, but fails during execution because `max_price_per_unit < next_l2_gas_price`. The gateway **accepts an invalid transaction**.

**Case 2 — gas price falling (`next_l2_gas_price < l2_gas_price`):**
A transaction with `max_price_per_unit` between `next_l2_gas_price` and `l2_gas_price` is rejected by the gateway (below threshold), even though it would succeed in the next block. The gateway **rejects a valid transaction**.

Both outcomes match the "High" impact category: *Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.*

### Likelihood Explanation

`next_l2_gas_price` is computed from the current block's gas consumption via an EIP-1559-style adjustment. Any block that is not exactly at the target utilization produces a `next_l2_gas_price` that differs from `l2_gas_price`. On a live network with variable load this is the common case, not the exception. The condition is unprivileged — any user submitting a transaction at exactly the current block's gas price triggers it.

### Recommendation

Expose `next_l2_gas_price` through the `GatewayFixedBlockStateReader` interface (or as a dedicated method) and use it in `validate_resource_bounds` instead of `l2_gas_price`. Concretely:

1. Add `next_l2_gas_price` to the data returned by `GatewayFixedBlockSyncStateClient::get_block_info_from_sync_client` (reading `block_header.next_l2_gas_price`).
2. Update `validate_resource_bounds` to use `next_l2_gas_price` as the threshold baseline, resolving the TODO comment.

### Proof of Concept

```
Block N committed:
  l2_gas_price        = 100 fri  (price used in block N)
  next_l2_gas_price   = 120 fri  (price that will govern block N+1)

User submits invoke tx with max_price_per_unit = 100 fri.

Gateway check (validate_resource_bounds):
  threshold = 100% × 100 = 100
  100 >= 100  →  PASS  →  tx admitted to mempool

Batcher executes tx in block N+1 at price 120 fri:
  max_price_per_unit (100) < block gas price (120)  →  tx FAILS

Reverse scenario (price falling, next = 80):
  User submits tx with max_price_per_unit = 90 fri.
  Gateway check: 90 >= 100  →  FAIL  →  tx REJECTED
  But tx would have succeeded in block N+1 at price 80.
```
