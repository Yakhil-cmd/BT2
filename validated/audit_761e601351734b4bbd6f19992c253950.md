### Title
Attacker Can Zero Relayer Yield in Silo Mode by Setting `SubmitArgs.max_gas_price = Some(0)` — (`engine/src/engine.rs`)

### Summary

In silo mode, `charge_gas` computes `effective_gas_price` by capping `priority_fee_per_gas` at `min(max_gas_price, ...)`. When an attacker supplies `SubmitArgs { max_gas_price: Some(0) }` via the public `submit_with_args` entrypoint, `effective_gas_price` collapses to zero. Because `prepaid_amount = fixed_gas * 0 = 0`, the sender pays nothing, and `refund_unused_gas` short-circuits before crediting the relayer, yielding zero relayer reward despite `fixed_gas` being set.

### Finding Description

**Entrypoint:** `submit_with_args` is a public `extern "C"` function with no caller access control at the contract level. [1](#0-0) 

It Borsh-deserializes `SubmitArgs` directly from caller-supplied input: [2](#0-1) 

`SubmitArgs.max_gas_price` is `Option<u128>` — fully attacker-controlled: [3](#0-2) 

The value is forwarded verbatim into `charge_gas`: [4](#0-3) 

Inside `charge_gas`, the early-return guard that would produce a zero-cost result requires `fixed_gas.is_none()`: [5](#0-4) 

When `fixed_gas = Some(X)` (silo mode), this guard does **not** fire. Execution continues to:

```
priority_fee_per_gas = min(max_gas_price=0, tx.max_priority_fee_per_gas.min(...)) = 0
effective_gas_price  = 0 + block_base_fee_per_gas(=0 on Aurora) = 0
prepaid_amount       = fixed_gas.as_u256() * 0 = 0
``` [6](#0-5) 

In `refund_unused_gas`, the first thing checked is whether `effective_gas_price` is zero — if so, it returns immediately without crediting the relayer: [7](#0-6) 

The relayer reward computation (`fixed_gas * priority_fee_per_gas`) is never reached: [8](#0-7) 

### Impact Explanation

In silo mode the invariant is that every submitted transaction costs the sender `fixed_gas * effective_gas_price` and the relayer earns `fixed_gas * priority_fee_per_gas`. By supplying `max_gas_price = Some(0)`, an attacker executes EVM transactions at zero cost and the relayer earns nothing. This is a direct theft of unclaimed yield: the silo operator's fee collection mechanism is completely bypassed.

### Likelihood Explanation

`submit_with_args` is a public NEAR contract method. Any NEAR account (or whitelisted account in a silo with the whitelist enabled) can call it with arbitrary `SubmitArgs`. No privilege escalation, key compromise, or governance capture is required. The only prerequisite is that the silo has `fixed_gas` configured, which is the defining condition of silo mode.

### Recommendation

In `charge_gas`, add a guard that rejects or ignores `max_gas_price = Some(0)` when `fixed_gas` is set, or enforce a minimum effective gas price equal to the transaction's own `max_priority_fee_per_gas` when silo mode is active. Concretely:

```rust
// After computing effective_gas_price, before prepaid_amount:
if fixed_gas.is_some() && effective_gas_price.is_zero() {
    return Err(GasPaymentError::MaxFeePerGasLessThanBaseFee); // or a dedicated error
}
```

Alternatively, ignore `max_gas_price` entirely when `fixed_gas` is set, since the two mechanisms are semantically incompatible.

### Proof of Concept

```rust
// Silo mode: fixed_gas = Some(1_000_000)
// Attacker submits via submit_with_args with max_gas_price = Some(0)
let args = SubmitArgs {
    tx_data: rlp::encode(&signed_tx).to_vec(),
    max_gas_price: Some(0),   // <-- attacker-controlled
    gas_token_address: None,
};
// charge_gas path:
//   priority_fee_per_gas = min(0, tx.max_priority_fee_per_gas) = 0
//   effective_gas_price  = 0
//   prepaid_amount       = 1_000_000 * 0 = 0  (sender pays nothing)
// refund_unused_gas:
//   effective_gas_price.is_zero() => return Ok(())  (relayer gets nothing)
assert_eq!(relayer_balance_delta, Wei::zero()); // invariant broken
```

The existing test `test_gas_charge_for_non_empty_transaction` already demonstrates that `max_gas_price = Some(5)` correctly caps the price, confirming the `min()` path is live: [9](#0-8) 

Setting `max_gas_price = Some(0)` in the same test with `fixed_gas = Some(X)` would produce `prepaid_amount = 0` and zero relayer reward, proving the vulnerability on unmodified code.
