### Title
Staker Can Instantly Raise Commission to 100% via `set_commission_commitment` + `set_commission`, Stealing Delegator Yield - (File: `src/staking/staking.cairo`)

---

### Summary

A staker can atomically raise their commission from any value (e.g., 1%) to 100% within the same epoch by first calling `set_commission_commitment` with `max_commission = 10000` and then immediately calling `set_commission(10000)`. Because the commitment becomes active in the same epoch it is set, and because `update_rewards_from_attestation_contract` / `update_rewards` applies the **current** commission at reward-distribution time, delegators receive zero pool rewards for that epoch. The code itself acknowledges this gap with an explicit developer note.

---

### Finding Description

**Root cause — `set_commission_commitment` activates immediately:**

`is_commission_commitment_active` is defined as:

```cairo
fn is_commission_commitment_active(
    self: @ContractState, commission_commitment: CommissionCommitment,
) -> bool {
    self.get_current_epoch() < commission_commitment.expiration_epoch
}
``` [1](#0-0) 

`set_commission_commitment` only requires `expiration_epoch > current_epoch`, so setting `expiration_epoch = current_epoch + 1` makes the commitment active **immediately** in the same epoch:

```cairo
assert!(expiration_epoch > current_epoch, "{}", Error::EXPIRATION_EPOCH_TOO_EARLY);
``` [2](#0-1) 

**Root cause — `update_commission` allows increases under an active commitment:**

```cairo
if self.is_commission_commitment_active(:commission_commitment) {
    assert!(
        commission <= commission_commitment.max_commission,
        "{}",
        Error::INVALID_COMMISSION_WITH_COMMITMENT,
    );
    assert!(commission != old_commission, "{}", Error::INVALID_SAME_COMMISSION);
}
``` [3](#0-2) 

There is **no lower-bound check** — `commission` may be arbitrarily higher than `old_commission`. The developers themselves flagged this:

```cairo
/// **Note**: Current commission increase safeguards still allow for sudden commission
/// changes.
fn set_commission_commitment(...)
``` [4](#0-3) 

**Root cause — commission is read at reward-distribution time:**

`_update_rewards` (called by both `update_rewards_from_attestation_contract` and `update_rewards`) reads the commission from storage at the moment rewards are distributed and passes it to `split_rewards_with_commission`:

```cairo
pub(crate) fn split_rewards_with_commission(
    rewards_including_commission: Amount, commission: Commission,
) -> (Amount, Amount) {
    let commission_rewards = compute_commission_amount_rounded_down(
        :rewards_including_commission, :commission,
    );
    let pool_rewards = rewards_including_commission - commission_rewards;
    (commission_rewards, pool_rewards)
}
``` [5](#0-4) 

If commission is 10000 (100%) at that moment, `pool_rewards = 0` and the staker captures the entire delegated-stake reward as commission.

---

### Impact Explanation

**High — Theft of unclaimed yield.**

Delegators who have staked into the pool earn yield proportional to their delegated balance. A staker who raises commission to 100% immediately before reward distribution causes `pool_rewards = 0` for that epoch. All yield that would have gone to delegators is instead credited to the staker's `unclaimed_rewards_own`. Delegators lose one full epoch of yield with no recourse.

---

### Likelihood Explanation

**Medium-High.** The attack requires only two sequential transactions by the staker in the same epoch:
1. `set_commission_commitment(max_commission=10000, expiration_epoch=current_epoch+1)`
2. `set_commission(10000)`

No special privileges, no external dependencies, no bridge interaction. Any registered staker with a delegation pool can execute this. The only cost is the gas for two calls. The staker can revert commission back to a low value after the epoch to avoid detection and repeat the attack in future epochs (by setting a new commitment once the old one expires).

---

### Recommendation

Introduce an **activation delay** for commission increases via commitment. The commitment should only permit commission increases starting from the epoch **after** it was set, not the same epoch. Concretely, change `is_commission_commitment_active` to also require that the commitment was set in a prior epoch, or add a `start_epoch` field to `CommissionCommitment` and enforce `current_epoch >= start_epoch` before allowing increases. This mirrors the timelock fix applied to the analogous Audius `deployerCut` vulnerability.

---

### Proof of Concept

```
Epoch N:
  staker calls set_commission_commitment(max_commission=10000, expiration_epoch=N+1)
    → commitment is immediately active: get_current_epoch() (N) < expiration_epoch (N+1) ✓

  staker calls set_commission(commission=10000)
    → update_commission: commitment active, 10000 <= 10000 ✓, 10000 != old_commission ✓
    → commission stored as 10000

  attestation contract calls update_rewards_from_attestation_contract(staker_address)
    → _update_rewards reads commission = 10000
    → split_rewards_with_commission: commission_rewards = pool_rewards_including_commission * 10000 / 10000
    → pool_rewards = 0
    → staker receives 100% of delegated-stake rewards
    → delegators receive 0 yield for epoch N
``` [6](#0-5) [7](#0-6) [8](#0-7)

        fn update_commission(
            ref self: ContractState,
            staker_address: ContractAddress,
            staker_pool_info: StoragePath<Mutable<InternalStakerPoolInfoV2>>,
            old_commission: Commission,
            commission: Commission,
        ) {
            if let Option::Some(commission_commitment) = staker_pool_info
                .commission_commitment
                .read() {
                if self.is_commission_commitment_active(:commission_commitment) {
                    assert!(
                        commission <= commission_commitment.max_commission,
                        "{}",
                        Error::INVALID_COMMISSION_WITH_COMMITMENT,
                    );
                    assert!(commission != old_commission, "{}", Error::INVALID_SAME_COMMISSION);
                } else {
                    assert!(
                        commission < old_commission, "{}", Error::COMMISSION_COMMITMENT_EXPIRED,
                    );
                }
            } else {
                assert!(commission < old_commission, "{}", Error::INVALID_COMMISSION);
            }

            // Update commission in storage.
            staker_pool_info.commission.write(Option::Some(commission));

            // Emit event.
            self
                .emit(
                    Events::CommissionChanged {
                        staker_address, old_commission, new_commission: commission,
                    },
                );
        }
