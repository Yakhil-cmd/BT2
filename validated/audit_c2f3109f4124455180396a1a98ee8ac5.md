### Title
Attacker Can Bypass Signature Validation for Any Account with a Pending Deploy-Account Transaction in the Mempool — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator skips the `__validate__` entry-point check (signature verification) for an invoke transaction with `nonce = 1` whenever the sender's account has **any** transaction in the mempool, regardless of transaction type. An attacker who observes a victim's pending `deploy_account` transaction in the mempool can submit a forged invoke transaction with `nonce = 1` for the victim's address — with an arbitrary or absent signature — and have it admitted to the mempool without any signature check.

### Finding Description

In `skip_stateful_validations` (lines 429–461 of `crates/apollo_gateway/src/stateful_transaction_validator.rs`), the gateway skips the `__validate__` entry point for an invoke transaction when:

1. The transaction nonce is `1`, and
2. The on-chain account nonce is `0` (account not yet deployed), and
3. `account_tx_in_pool_or_recent_block(sender_address)` returns `true`. [1](#0-0) 

The code comment states: *"It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."*

This reasoning is flawed. `account_tx_in_pool_or_recent_block` returns `true` for **any** transaction type — including a `deploy_account` submitted by the victim. An attacker who observes the victim's `deploy_account` in the mempool satisfies condition 3 without ever possessing the victim's private key.

The result of `skip_stateful_validations` returning `true` is that `run_validate_entry_point` sets `validate: false` in the `ExecutionFlags`: [2](#0-1) 

This causes `StatefulValidator::perform_validations` to return `Ok(())` immediately after `perform_pre_validation_stage`, without ever calling `validate_tx` (which invokes `__validate__`): [3](#0-2) 

The gateway's `validate_nonce` function permits `nonce = 1` when `account_nonce = 0` as long as `1 <= max_allowed_nonce_gap`, which is the default configuration: [4](#0-3)

### Impact Explanation

An attacker can submit an invoke transaction for **any** account that has a pending `deploy_account` in the mempool, with an invalid or absent signature, and have it admitted to the mempool. This matches the external bug's invariant: a privileged path (here, the UX skip-validate shortcut) can act on behalf of an arbitrary account without that account's authorization.

When the block is built:
1. The victim's `deploy_account` executes (nonce 0 → 1).
2. The attacker's forged invoke executes (nonce 1 → 2); `__validate__` runs in the batcher with `validate = true`, fails, and the transaction reverts — but the victim's **nonce is permanently bumped to 2** and a **fee is charged** from the victim's account.
3. The victim's own legitimate invoke with `nonce = 1` is now stale and dropped from the mempool.

The attacker can thus front-run and permanently invalidate the victim's first post-deployment transaction, and drain fee balance, with no cryptographic material from the victim.

**Impact category**: High — Mempool/gateway admission accepts an invalid (unsigned) transaction before sequencing.

### Likelihood Explanation

- The mempool is observable (transactions are broadcast over P2P).
- The attacker only needs to detect a pending `deploy_account` for any address, which is a routine event.
- No special privilege or on-chain state is required.
- The attack is limited to one forged invoke per deploy-account event (nonce=1 only), but that is sufficient to disrupt the victim's first post-deployment operation.

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a **`deploy_account` transaction** is pending for the sender address. The mempool should expose a dedicated query such as `has_pending_deploy_account(address)` rather than the generic `account_tx_in_pool_or_recent_block`. This preserves the UX feature (deploy + invoke in one shot) while closing the forgery window.

```rust
// Current (vulnerable):
return mempool_client
    .account_tx_in_pool_or_recent_block(tx.sender_address())
    .await ...

// Recommended:
return mempool_client
    .has_pending_deploy_account(tx.sender_address())
    .await ...
```

### Proof of Concept

1. Victim calls `add_tx` with a valid `deploy_account` for address `A` (class_hash `C`, salt `S`). It enters the mempool; on-chain nonce of `A` is still `0`.
2. Attacker observes the mempool and sees `A` has a pending transaction.
3. Attacker calls `add_tx` with an `Invoke V3` transaction:
   - `sender_address = A`
   - `nonce = 1`
   - `calldata` = arbitrary (e.g., transfer all STRK to attacker)
   - `signature` = `[]` (empty or garbage)
   - `resource_bounds` = valid non-zero bounds
4. Gateway stateless validation passes (signature length ≤ max, resource bounds valid).
5. `extract_state_nonce_and_run_validations` fetches on-chain nonce = `0`.
6. `validate_nonce`: `0 ≤ 1 ≤ max_allowed_nonce_gap` → passes.
7. `validate_by_mempool`: mempool nonce check passes (nonce=1 ≥ resolved nonce=0).
8. `skip_stateful_validations`: `nonce==1 && account_nonce==0 && account_in_pool==true` → returns `true`.
9. `run_validate_entry_point` sets `validate = false`; `StatefulValidator::validate` returns `Ok(())` without calling `__validate__`.
10. Attacker's forged invoke is added to the mempool.
11. At block execution: `deploy_account` runs first; then the forged invoke runs with `validate = true` in the batcher, `__validate__` fails, the invoke reverts — but victim's nonce is now `2` and a fee is deducted from victim's balance. Victim's own nonce-1 invoke is permanently invalidated. [6](#0-5) [7](#0-6) [3](#0-2)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
