Audit Report

## Title
Hard `assert!` in ckBTC Minter `resubmit_transactions` Causes Deterministic Canister Trap, Permanently Blocking Withdrawal Recovery - (File: rs/bitcoin/ckbtc/minter/src/lib.rs)

## Summary
The `resubmit_transactions` function in the ckBTC minter contains a hard `assert!(tx.signed_tx.is_some())` at line 826 of `rs/bitcoin/ckbtc/minter/src/lib.rs`. If a consolidation transaction's `signed_tx` field is `None` due to any state inconsistency, the minter canister traps on every timer tick, creating a deterministic, self-repeating failure that permanently blocks ckBTC â†’ BTC withdrawal recovery. This exact scenario occurred in production in June 2025, leaving three user withdrawals stuck until an emergency NNS-governed canister upgrade was deployed.

## Finding Description
Inside `resubmit_transactions`, the code iterates over stuck transactions. For consolidation transactions, it enters a `read_state` closure and calls:

```rust
assert!(tx.signed_tx.is_some());
``` [1](#0-0) 

This is a hard `assert!` (not `debug_assert!`), active in both debug and release/production builds. It fires inside a `read_state` closure during the minter's periodic timer-driven `finalize_requests` â†’ `resubmit_transactions` flow. When `signed_tx` is `None`, the canister traps. Because the stuck transaction remains in state and the timer re-enters the same code path on every tick, the trap is deterministic and self-repeating â€” no resubmission progress is ever made. The comment `// For ConsolidatedUtxosRequest, signed_tx should always exist.` acknowledges the assumption but provides no fallback. Elsewhere in the same function, errors are handled gracefully with `log!` + `continue`, making this assert an inconsistent and dangerous outlier. [2](#0-1) 

The `signed_tx: None` pattern is reachable: grep confirms it appears in `state/eventlog.rs` and `lib.rs`, meaning there are code paths that construct `SubmittedBtcTransaction` with `signed_tx` unset.

The production upgrade proposal `rs/bitcoin/ckbtc/mainnet/minter_upgrade_2025_06_27.md` explicitly documents the incident:

> "There is a deterministic panic occurring in the minter when it tries to resubmit those transactions, which explains why those transactions are currently stuck." [3](#0-2) 

Despite PR #5713 being referenced as the fix in that upgrade, the `assert!(tx.signed_tx.is_some())` remains present in the current codebase at line 826, meaning the same class of trap can recur if the state invariant is violated again.

## Impact Explanation
This is a **High** severity finding. The impact is a significant ck-token (ckBTC) platform-level DoS with concrete, demonstrated user harm: ckBTC â†’ BTC withdrawal transactions are permanently stuck until an emergency NNS-governed canister upgrade is deployed. The minter's timer creates a deterministic, self-repeating trap â€” not a one-time crash â€” that blocks all resubmission progress for affected transactions. This matches the allowed impact: *"Significant Chain Fusion, ck-token, ledger... security impact with concrete user or protocol harm"* and *"Application/platform-level DoS, crash... or subnet availability impact."* The June 2025 incident proves real user funds (three withdrawal transactions) were locked, requiring emergency governance intervention.

## Likelihood Explanation
Likelihood is **moderate**. The `signed_tx` field is set by the minter itself, so it is normally present. However, the June 2025 production incident proves the invariant can be violated under real conditions (extremely low fee per vbyte causing transactions not to be mined, combined with state inconsistency during resubmission). The minter's state involves multiple async inter-canister calls (Bitcoin canister, ledger, threshold ECDSA), creating multiple opportunities for partial state updates. Any future bug in state management that results in `signed_tx = None` for a consolidation transaction will trigger the same deterministic trap. No special attacker privileges are required â€” a normal ckBTC withdrawal user whose transaction becomes stuck is sufficient to reach this code path.

## Recommendation
Replace the hard `assert!` with graceful error handling consistent with the rest of the function:

```rust
if tx.signed_tx.is_none() {
    log!(Priority::Error,
        "[resubmit_transactions]: consolidation tx {} has no signed_tx, skipping",
        tx.txid);
    return None; // inside the read_state closure
}
tx.signed_tx
    .clone()
    .map(|signed_tx| (s.btc_network, tx.txid, signed_tx))
```

This matches the existing pattern at lines 849â€“858 where errors are logged and execution continues rather than trapping. [4](#0-3) 

## Proof of Concept
1. A ckBTC user initiates a withdrawal (ckBTC â†’ BTC).
2. The minter creates a consolidation transaction; due to a state inconsistency (e.g., partial async update), `signed_tx` is `None`.
3. The minter's periodic timer fires `finalize_requests`, which calls `resubmit_transactions`.
4. `resubmit_transactions` iterates over stuck transactions, finds the consolidation transaction matching `old_txid`, and enters the `read_state` closure.
5. `assert!(tx.signed_tx.is_some())` fires at line 826; the canister traps.
6. On the next timer tick, the same code path is re-entered and traps again â€” a deterministic, self-repeating failure.
7. The withdrawal transaction remains permanently stuck until an NNS-governed emergency upgrade is deployed.

This exact scenario was confirmed in production in June 2025, documented in `rs/bitcoin/ckbtc/mainnet/minter_upgrade_2025_06_27.md`, requiring an emergency upgrade to unblock three stuck withdrawals. [5](#0-4)
