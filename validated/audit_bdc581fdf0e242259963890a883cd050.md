### Title
Period Mint Cap Exhausted by Single Full Tick Permanently Blocks Overflow Messages Until Next Period — (`L1/starkware/solidity/stake/RewardSupplier.sol`, `L1/starkware/solidity/stake/PeriodMintLimit.sol`)

---

### Summary

`PERIOD_MINT_CAP` is set to exactly `MAX_MESSAGES_TO_PROCESS_PER_TICK × TOKENS_PER_MINT_REQUEST` (5 × 1,300,000e18 = 6,500,000e18). A single `tick()` call that processes the maximum 5 messages fully exhausts the weekly mint cap. Any additional queued L2 messages cannot be processed until the next period (up to 1 week later), because every subsequent `tick()` call in the same period reverts with `EXCEED_PERIOD_MINTING`.

---

### Finding Description

**Relevant constants:**

- `TOKENS_PER_MINT_REQUEST = 1_300_000 × 10^18` [1](#0-0) 

- `PERIOD_MINT_CAP = 6_500_000 × 10^18`, `MINTING_PERIOD_DURATION = 1 weeks` [2](#0-1) 

The invariant `PERIOD_MINT_CAP == MAX_MESSAGES_TO_PROCESS_PER_TICK × TOKENS_PER_MINT_REQUEST` means one full tick exactly saturates the cap.

**`tick()` is permissionless:** [3](#0-2) 

**`requiredMinting()` caps at 5 messages per call:** [4](#0-3) 

**`checkAndUpdatePeriodicalQuota` enforces a hard cap with no carry-over:** [5](#0-4) 

**Execution trace with 6 queued L2 messages:**

| Step | Action | `periodMintAccounting[slot]` | Result |
|------|--------|------------------------------|--------|
| `tick()` #1 | `min(6,5)=5` msgs → mint 6,500,000e18 | 0 → 6,500,000e18 | Success |
| `tick()` #2 | `min(1,5)=1` msg → mint 1,300,000e18 | 6,500,000e18 + 1,300,000e18 > cap | **REVERT** `EXCEED_PERIOD_MINTING` |

The 6th message remains unconsumed in the Starknet messaging contract for up to 1 week.

**The L2 side can legitimately queue > 5 messages.** `request_funds` computes `num_msgs = ceil_of_division(diff, base_mint_amount)` and sends that many L2→L1 messages: [6](#0-5) 

If `diff > 5 × base_mint_amount` (e.g., after a long period of inactivity or a large staking surge), more than 5 messages are queued, and the overflow is guaranteed to be frozen.

---

### Impact Explanation

Any L2 mint request messages beyond the 5th in a given period are frozen for up to 1 week. Stakers whose rewards depend on those messages receive delayed yield. The delay is bounded by `MINTING_PERIOD_DURATION = 1 week` per overflow batch, but can compound across periods if the L2 continues to queue large batches. This constitutes **temporary freezing of unclaimed yield**.

---

### Likelihood Explanation

The L2 `request_funds` function sends as many messages as needed to cover the funding gap. Any scenario where the gap exceeds `5 × 1,300,000e18 = 6,500,000e18` tokens — such as a protocol restart after downtime, a large influx of new stakers, or accumulated unclaimed rewards — will trigger this condition without any attacker involvement. An adversary can also deliberately trigger it by calling `tick()` when exactly 6+ messages are pending, since `tick()` is permissionless.

---

### Recommendation

Either:
1. Increase `PERIOD_MINT_CAP` to a multiple of `TOKENS_PER_MINT_REQUEST` that accommodates realistic maximum queue depths (e.g., `10 × TOKENS_PER_MINT_REQUEST`), or
2. Make `checkAndUpdatePeriodicalQuota` in `tick()` aware of the cap and only consume as many messages as the remaining quota allows in the current period, deferring the rest to the next period gracefully rather than reverting.

---

### Proof of Concept

```solidity
// Setup: queue 6 L2→L1 mint request messages from the L2 RewardSupplier.
// Each message has payload [TOKENS_PER_MINT_REQUEST] = [1_300_000e18].

// Call 1: processes 5 messages, mints 6_500_000e18 (hits PERIOD_MINT_CAP exactly)
rewardSupplier.tick{value: 2 ether}();
// periodMintAccounting[slot] == 6_500_000e18

// Call 2: 1 message remains, tries to mint 1_300_000e18
// checkAndUpdatePeriodicalQuota: 6_500_000e18 + 1_300_000e18 > 6_500_000e18 → REVERT
vm.expectRevert("EXCEED_PERIOD_MINTING");
rewardSupplier.tick{value: 2 ether}();

// The 6th message is now stuck until block.timestamp / 1 weeks increments.
// Stakers are denied yield for up to 1 week.
```

    function checkAndUpdatePeriodicalQuota(address token, uint256 amount) internal {
        bytes32 periodSlot = periodAccountingSlot(token);
        uint256 mintedThisPeriodBefore = periodMintAccounting()[periodSlot];
        uint256 mintedThisPeriodAfter = mintedThisPeriodBefore + amount;
        require(mintedThisPeriodAfter <= PERIOD_MINT_CAP, "EXCEED_PERIOD_MINTING");
        periodMintAccounting()[periodSlot] = mintedThisPeriodAfter;
    }
