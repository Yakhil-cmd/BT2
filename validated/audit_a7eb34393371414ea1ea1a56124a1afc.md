### Title
`saturating_mul` Overflow in `full_native_limit` Causes `gas_used > gas_limit` Underflow in `compute_gas_refund` - (`basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`)

---

### Summary

In `compute_gas_refund`, `full_native_limit` is computed as `gas_limit.saturating_mul(native_per_gas)`. When `native_per_gas * gas_limit` exceeds `u64::MAX`, this saturates to `u64::MAX`. The actual native resources given to the transaction are capped at `MAX_NATIVE_COMPUTATIONAL` (much smaller), but `native_used` is computed against the inflated `full_native_limit ≈ u64::MAX`. This inflates `delta_gas`, pushing `gas_used` above `gas_limit`, causing underflow at `total_gas_refund = gas_limit - gas_used`. The `require_internal!` guard catches the wrapped value and returns an `InternalError`, which propagates through `before_refund` and causes `process_l2_transaction` to return early — after the fee has already been committed — leaving the user's entire prepaid fee unrefunded.

---

### Finding Description

**Root cause — `refund_calculation.rs` lines 59–64:**

```rust
let full_native_limit = if cfg!(feature = "unlimited_native") || native_per_gas == 0 {
    u64::MAX - 1
} else {
    gas_limit.saturating_mul(native_per_gas)   // ← saturates to u64::MAX on overflow
};
let native_used = full_native_limit.saturating_sub(resources.native().remaining().as_u64());
```

`full_native_limit` is intended to represent the total native budget for the transaction (`gas_limit × native_per_gas`). When this product overflows `u64`, `saturating_mul` returns `u64::MAX`. However, the actual native resources handed to the transaction in `create_resources_for_tx` are capped at `MAX_NATIVE_COMPUTATIONAL` (2³⁵ ≈ 3.4 × 10¹⁰). So `resources.native().remaining()` is at most `MAX_NATIVE_COMPUTATIONAL`, and:

```
native_used = u64::MAX − remaining_native  ≈  u64::MAX
```

This is orders of magnitude larger than the real native consumption.

**Propagation — lines 66–83:**

```rust
let delta_gas = (native_used / native_per_gas) as i64 - (gas_used as i64);
if delta_gas > 0 {
    gas_used += delta_gas as u64;
}
let total_gas_refund = gas_limit - gas_used;   // ← underflow when gas_used > gas_limit
```

With `native_used ≈ u64::MAX` and a large `native_per_gas`, `native_used / native_per_gas` can exceed `gas_limit`. For example:

| Parameter | Value |
|---|---|
| `gas_limit` | 30,000,000 |
| `native_per_gas` | 7 × 10¹¹ |
| `gas_limit × native_per_gas` | 2.1 × 10¹⁹ > `u64::MAX` → saturates |
| `full_native_limit` | `u64::MAX = 18,446,744,073,709,551,615` |
| `native_used / native_per_gas` | ≈ 26,352,491,533 |
| `gas_used` after delta | ≈ 26,352,491,533 **> 30,000,000** |
| `total_gas_refund` | 30,000,000 − 26,352,491,533 → **wraps to huge value** |

The `require_internal!` check at lines 85–89 detects `total_gas_refund > gas_limit` and returns `Err(InternalError)`.

**Error propagation — `process_l2_transaction` lines 160–167:**

```rust
F::before_refund::<Config>(
    system, &transaction, &mut tx_context,
    &execution_result, extra_info, tracer,
)?;   // ← InternalError propagates here as TxError
```

At this point, `precharge_fee` (lines 133–141) has already been committed via `system.finish_global_frame(None)?`. The fee deduction from the user's balance is permanent. The refund step never executes, so the user forfeits `gas_price × gas_limit` tokens.

---

### Impact Explanation

A user who submits a transaction with `native_per_gas × gas_limit > u64::MAX` will:
1. Have their full prepaid fee (`gas_price × gas_limit`) permanently deducted.
2. Receive no refund for unused gas.
3. Have their transaction fail with an internal error after execution.

This is a direct, irreversible loss of user funds triggered by a crafted transaction parameter. The block continues processing normally; only the affected transaction is impacted.

---

### Likelihood Explanation

The condition `native_per_gas × gas_limit > u64::MAX` requires:

```
native_per_gas > u64::MAX / gas_limit
```

`native_per_gas = ⌈gas_price / native_price⌉` where `native_price` is operator-set. If `native_price` is small (e.g., 1), a user with `gas_price > u64::MAX / gas_limit` tokens can trigger this. The user must also hold `gas_price × gas_limit` tokens to pass the balance check. While the required balance is large, it is not bounded by the protocol — no check prevents `native_per_gas × gas_limit` from overflowing. The vulnerability is deterministically triggerable by any user who can afford the prepaid fee.

---

### Recommendation

Replace `saturating_mul` with `checked_mul` for `full_native_limit`, and either reject the transaction or use the actual native limit (the one passed into `create_resources_for_tx`) rather than recomputing it:

```rust
// Option A: use checked_mul and cap at actual native limit
let full_native_limit = gas_limit
    .checked_mul(native_per_gas)
    .unwrap_or(u64::MAX - 1);

// Option B (preferred): pass the actual native limit into compute_gas_refund
// instead of recomputing it from gas_limit × native_per_gas
```

The actual native limit (before capping at `MAX_NATIVE_COMPUTATIONAL`) should be threaded through from `create_resources_for_tx` so that `native_used` reflects real consumption rather than an inflated proxy.

### Proof of Concept

**File:** `basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`

Step-by-step trigger:

1. Operator sets `native_price = 1`.
2. User submits an EIP-1559 transaction with `max_fee_per_gas = 7 × 10¹¹`, `gas_limit = 30_000_000`, and sufficient balance (`≥ 7 × 10¹¹ × 30_000_000 = 2.1 × 10¹⁹` tokens).
3. Validation passes: `native_per_gas = 7 × 10¹¹` fits in `u64`; balance check passes.
4. `create_resources_for_tx` caps native resources at `MAX_NATIVE_COMPUTATIONAL = 2³⁵`.
5. In `compute_gas_refund`:
   - `full_native_limit = 30_000_000.saturating_mul(700_000_000_000) = u64::MAX` (overflow)
   - `native_used = u64::MAX − 0 = 18,446,744,073,709,551,615`
   - `native_used / native_per_gas = 26,352,491,533`
   - `delta_gas = 26,352,491,533 − gas_used > 0`
   - `gas_used = 26,352,491,533 > gas_limit = 30,000,000`
   - `total_gas_refund = 30,000,000 − 26,352,491,533` → wraps to `18,446,744,047,357,060,083`
   - `require_internal!(total_gas_refund <= gas_limit, ...)` → **fails → `InternalError`**
6. `before_refund` propagates `InternalError` → `process_l2_transaction` returns `Err(TxError)`.
7. Fee collection (already committed) is not reversed. User loses `2.1 × 10¹⁹` tokens. [1](#0-0) [2](#0-1) [3](#0-2)
