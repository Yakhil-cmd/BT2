### Title
Bank Precompile `transfer` Executes Unauthorized Token Transfers Without Caller Authorization Check - (File: `x/cronos/keeper/precompiles/bank.go`)

---

### Summary

The bank precompile's `transfer` method accepts an arbitrary `sender` address from call input and executes `bankKeeper.SendCoins` from that address without verifying that the calling contract has any authorization to spend from it. Any EVM contract can drain any user's balance of `evm/{callerContractAddress}` native tokens without the user's consent, bypassing the implicit authorization boundary that should restrict token movement to the token holder.

---

### Finding Description

In `x/cronos/keeper/precompiles/bank.go`, the `Run` method handles the `transfer` case as follows:

```go
// Line 175
sender := args[0].(common.Address)   // ← taken from caller-supplied input
recipient := args[1].(common.Address)
amount := args[2].(*big.Int)
...
from := sdk.AccAddress(sender.Bytes())  // ← arbitrary address, not validated
to   := sdk.AccAddress(recipient.Bytes())
...
denom := EVMDenom(contract.Caller())    // ← denom scoped to calling contract
amt   := sdk.NewCoin(denom, sdkmath.NewIntFromBigInt(amount))
err = stateDB.ExecuteNativeAction(precompileAddr, nil, func(ctx sdk.Context) error {
    ...
    if err := bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt)); err != nil {
        ...
    }
    return nil
})
``` [1](#0-0) 

The `sender` (`from`) is taken directly from the ABI-decoded input arguments — it is **never** compared to `contract.Caller()`, and there is no allowance or approval check. The denom is correctly scoped to `evm/{callerContractAddress}`, but that scoping does not protect users: the calling contract is the sole authority over its own denom namespace and can freely move tokens between any two arbitrary addresses.

The same pattern applies to the `burn` case: `recipient` (line 121) is taken from input and used as the address to burn from, with no authorization check. [2](#0-1) 

The bank precompile has no caller access control — any EVM contract can invoke it. [3](#0-2) 

---

### Impact Explanation

**Critical — Unauthorized transfer of precompile-controlled assets.**

Any EVM contract can call `bankPrecompile.transfer(victimAddress, attackerAddress, victimBalance)` and the bank module will execute `SendCoins` from the victim to the attacker with no consent from the victim. The `evm/{contractAddress}` denom tokens are native bank-module assets; they are usable in IBC transfers, governance, and any Cosmos-layer operation. A contract that has previously minted these tokens to users (e.g., as part of a DeFi protocol) can at any time drain every user's balance in a single transaction, with no on-chain recourse.

The `burn` variant allows the same contract to destroy any user's balance without consent, which is an equivalent critical impact (unauthorized balance change).

---

### Likelihood Explanation

Any unprivileged actor who can deploy an EVM contract on Cronos can exploit this. Deployment requires only gas. The attack is a single EVM call after token distribution. No privileged keys, governance access, or validator compromise is required.

---

### Recommendation

In the `TransferMethodName` case, enforce that the calling contract can only transfer from itself:

```go
if sender != contract.Caller() {
    return nil, errors.New("transfer: sender must be the calling contract")
}
```

Alternatively, implement an ERC20-style allowance mapping inside the bank precompile so that a contract can only transfer from a third-party address if that address has previously approved the contract for at least the requested amount.

The same fix must be applied to the `BurnMethodName` case: the address to burn from must equal `contract.Caller()`, or the user must have explicitly approved the burn.

---

### Proof of Concept

1. Attacker deploys `MaliciousToken` at address `0xMalicious`.
2. `MaliciousToken` calls `bankPrecompile.mint(victimAddress, 1_000_000)` — this mints `1_000_000 evm/0xMalicious` tokens to the victim (e.g., as a yield reward or liquidity receipt token).
3. Victim holds `1_000_000 evm/0xMalicious` in their Cosmos bank balance.
4. Attacker calls `MaliciousToken.steal(victimAddress)`, which internally executes:
   ```solidity
   IBankPrecompile(0x0000000000000000000000000000000000000064)
       .transfer(victimAddress, attackerAddress, 1_000_000);
   ```
5. The bank precompile executes `bankKeeper.SendCoins(ctx, victimAddr, attackerAddr, coins)` with no authorization check.
6. Victim's entire `evm/0xMalicious` balance is transferred to the attacker.

The attacker can repeat step 4 for every victim who holds the token, atomically draining all balances in a single block.
