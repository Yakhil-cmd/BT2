Audit Report

## Title
Global Pending-Queue Saturation Griefing Blocks All ckETH and ckERC20 Withdrawals - (File: `rs/ethereum/cketh/minter/src/guard/mod.rs`)

## Summary
The ckETH minter enforces a single global `MAX_PENDING = 100` limit on pending withdrawal requests shared across all principals and all token types. A single unprivileged user can make 100 sequential `withdraw_eth` calls to saturate `pending_withdrawal_requests`, causing every subsequent `withdraw_eth` and `withdraw_erc20` call from any principal to fail with `TooManyPendingRequests`. Because the attacker's ckETH is eventually redeemed on Ethereum, the net cost is only gas fees, making the attack economically sustainable and repeatable.

## Finding Description
`rs/ethereum/cketh/minter/src/guard/mod.rs` defines `MAX_PENDING = 100` at line 10 and checks it globally in `Guard::new` at line 52:

```rust
if PR::pending_requests_count(s) >= MAX_PENDING {
    return Err(GuardError::TooManyPendingRequests);
}
```

`PendingWithdrawalRequests::pending_requests_count` (lines 32–34) delegates to `withdrawal_requests_len()`, which returns `self.pending_withdrawal_requests.len()` (lines 929–931 of `transactions/mod.rs`) — a raw global count with no per-principal breakdown.

The `Guard::Drop` implementation (lines 71–75 of `guard/mod.rs`) only removes the principal from `guarded_principals`; it does **not** remove the queued `WithdrawalRequest`. The `AlreadyProcessing` check (line 56) only blocks *concurrent* calls from the same principal. Once a call completes and the guard is dropped, the same principal can immediately call again, leaving another entry in `pending_withdrawal_requests`.

Both `withdraw_eth` (line 274 of `main.rs`) and `withdraw_erc20` (line 401 of `main.rs`) call `retrieve_withdraw_guard(caller)` against the same shared queue, so flooding via `withdraw_eth` also blocks all `withdraw_erc20` calls.

**Attack path:**
1. Attacker calls `withdraw_eth` 100 times sequentially with the minimum withdrawal amount (0.03 ckETH each).
2. Each call: guard acquired → ckETH burned → request appended to `pending_withdrawal_requests` → guard dropped.
3. After 100 calls, `pending_withdrawal_requests.len() == 100`.
4. Every subsequent `withdraw_eth` or `withdraw_erc20` from any principal hits the `>= MAX_PENDING` gate and is rejected.
5. Attacker re-fills the queue as the minter drains it, sustaining the DoS indefinitely.

## Impact Explanation
All ckETH and ckERC20 withdrawals are blocked for every user on the minter until the attacker's queued requests are processed. Because the attacker can continuously re-submit after each batch drains, the disruption can be sustained indefinitely. This matches the allowed High impact: **"Application/platform-level DoS… or subnet availability impact not based on raw volumetric DDoS"** and **"Significant Chain Fusion, ck-token… security impact with concrete user or protocol harm."** Users holding ckETH or ckERC20 tokens cannot convert them back to native assets, breaking the bridge's core redeemability guarantee.

## Likelihood Explanation
The attack requires ~3 ETH of capital (fully recoverable minus gas fees) and 100 sequential canister calls — both achievable by any motivated adversary. No privileged role, governance majority, or key compromise is needed. The entry path is the public `withdraw_eth` update method, reachable by any non-anonymous ingress sender. The net cost (gas fees only, roughly $500–$5,000 at typical gas prices) is low relative to the disruption caused to all minter users.

## Recommendation
1. **Add a per-principal pending-request cap** alongside the global cap. Reject a new request if the calling principal already has `N` (e.g., 5) requests in `pending_withdrawal_requests`, preventing a single principal from monopolising the queue.
2. **Separate ckETH and ckERC20 pending queues** so that flooding one token type does not block the other.
3. **Raise `MAX_PENDING`** to a value that reflects realistic throughput, reducing the feasibility of saturation.

## Proof of Concept
```rust
// Pseudocode integration test using PocketIC or local replica
for _ in 0..100 {
    minter.withdraw_eth(WithdrawalArg {
        amount: 30_000_000_000_000_000u64.into(), // minimum
        recipient: attacker_eth_address.clone(),
        from_subaccount: None,
    }).await.unwrap();
    // Wait for call to complete so AlreadyProcessing does not trigger
}

// Now pending_withdrawal_requests.len() == 100
// Any call from any principal returns TooManyPendingRequests:
let result = victim.withdraw_eth(WithdrawalArg {
    amount: 1_000_000_000_000_000_000u64.into(),
    recipient: victim_eth_address.clone(),
    from_subaccount: None,
}).await;
assert!(result.is_err()); // trapped with TooManyPendingRequests
```
A deterministic integration test can assert `withdrawal_requests_len() == 100` after the 100 attacker calls and then verify that a victim call traps with `TooManyPendingRequests`.
