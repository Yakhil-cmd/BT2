### Title
`ExitToNear` Precompile Does Not Refund User Tokens on Failed `ft_transfer` Promise When `error_refund` Feature Is Disabled in Production Build - (`engine-precompiles/src/native.rs`)

---

### Summary

The `ExitToNear` precompile in Aurora Engine burns ERC-20 tokens (or transfers ETH to the precompile address) before dispatching an asynchronous `ft_transfer` / `ft_transfer_call` promise to the NEP-141 contract. If that promise fails, the callback `exit_to_near_precompile_callback` is supposed to re-mint the burned ERC-20 tokens or return the ETH. However, the entire refund path is gated behind the compile-time feature flag `error_refund`, which is **not included in the production build**. In production, `refund` is always `None`, so the callback silently does nothing on failure, permanently destroying the user's tokens.

---

### Finding Description

**Step 1 — Production build omits `error_refund`.**

The `Makefile.toml` defines the production build features as:

```
CARGO_FEATURES_BUILD = "contract"
```

The `error_refund` feature is never listed in the production or test build targets; it only appears in the `clippy-contract-refund` linting task.

**Step 2 — The precompile sets `refund: None` in production.**

In `engine-precompiles/src/native.rs`, when constructing the callback arguments for the `exit_to_near_precompile_callback`:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,   // <-- always None in production
    transfer_near: transfer_near_args,
};
```

**Step 3 — The callback does nothing on failure when `refund` is `None`.**

In `engine/src/contract_methods/connector.rs`, the callback handler:

```rust
let maybe_result = if let Some(PromiseResult::Successful(_)) = handler.promise_result(0) {
    // success path
    ...
} else if let Some(args) = args.refund {
    // refund path — NEVER reached in production because refund is None
    let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;
    ...
} else {
    None  // <-- production always falls here on failure; no refund issued
};
```

**Step 4 — Tokens are already gone before the promise.**

For ERC-20 exits, the ERC-20 tokens are burned from the user's balance during the precompile execution, before the async `ft_transfer` promise is dispatched. For ETH exits, the ETH is transferred from the user's EVM balance to the `exit_to_near::ADDRESS` precompile address (`0xe9217bc70b7ed1f598ddd3199e80b093fa71124f`). In both cases, if the promise fails and no refund is issued, the assets are permanently lost.

**Step 5 — The test suite explicitly acknowledges this behavior.**

```rust
#[cfg(feature = "error_refund")]
let balance = FT_TRANSFER_AMOUNT.into();
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
```

This confirms the production behavior: without `error_refund`, the exit amount is permanently lost.

---

### Impact Explanation

- **ERC-20 exit failure**: ERC-20 tokens are burned from the user's EVM balance. If `ft_transfer` to the NEAR recipient fails, the tokens are never re-minted. They are permanently destroyed — **permanent freezing/theft of user funds**.
- **ETH exit failure**: ETH is moved from the user's EVM balance to the `exit_to_near::ADDRESS` precompile address. If `ft_transfer` fails, the ETH is permanently locked at that address with no recovery path — **permanent freezing of user funds**.

Both cases qualify as **Critical** impact: permanent freezing of funds.

---

### Likelihood Explanation

The `ft_transfer` promise can fail under realistic, user-triggerable conditions:

1. The recipient NEAR account ID is valid but not registered with the NEP-141 token contract (a common condition for new accounts).
2. The Aurora engine account has insufficient NEP-141 balance (e.g., after a prior accounting discrepancy).
3. The NEP-141 contract panics for any contract-specific reason.

Any unprivileged EVM user can trigger this by calling the `ExitToNear` precompile with a NEAR recipient that is not registered with the target NEP-141 contract. This is a realistic, low-effort scenario.

---

### Recommendation

Enable the `error_refund` feature in the production build by adding it to `CARGO_FEATURES_BUILD` in `Makefile.toml`:

```toml
CARGO_FEATURES_BUILD = "contract,error_refund"
```

This ensures that `refund_call_args` is populated in `ExitToNearPrecompileCallbackArgs` and that `exit_to_near_precompile_callback` correctly invokes `engine::refund_on_error` when the downstream `ft_transfer` promise fails, re-minting burned ERC-20 tokens or returning ETH to the user.

---

### Proof of Concept

**Entry path:**
1. User holds ERC-20 tokens on Aurora (bridged from a NEP-141).
2. User calls the `ExitToNear` precompile via an EVM transaction (`submit`), specifying a NEAR recipient account that is not registered with the NEP-141 contract.
3. The precompile burns the user's ERC-20 tokens and dispatches `ft_transfer` to the NEP-141 contract.
4. The `ft_transfer` call fails (unregistered recipient).
5. `exit_to_near_precompile_callback` is invoked with `args.refund == None` (production build).
6. The `else { None }` branch executes — no refund, no re-mint.
7. The user's ERC-20 tokens are permanently destroyed.

pub fn refund_on_error<I: IO + Copy, E: Env, P: PromiseHandler>(
    io: I,
    env: &E,
    state: EngineState,
    args: &RefundCallArgs,
    handler: &mut P,
) -> EngineResult<SubmitResult> {
    let current_account_id = env.current_account_id();
    if let Some(erc20_address) = args.erc20_address {
        // ERC-20 exit; re-mint burned tokens
        let erc20_admin_address = current_address(&current_account_id);
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, erc20_admin_address, current_account_id, io, env);

        let refund_address = args.recipient_address;
        let amount = U256::from_big_endian(&args.amount);
        let input = setup_refund_on_error_input(amount, refund_address);

        engine.call(
            &erc20_admin_address,
            &erc20_address,
            Wei::zero(),
            input,
            u64::MAX,
            Vec::new(),
            Vec::new(),
            handler,
        )
    } else {
        // ETH exit; transfer ETH back from precompile address
        let exit_address = exit_to_near::ADDRESS;
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, exit_address, current_account_id, io, env);
        let refund_address = args.recipient_address;
        let amount = Wei::new(U256::from_big_endian(&args.amount));
        engine.call(
            &exit_address,
            &refund_address,
            amount,
            Vec::new(),
            u64::MAX,
            vec![
                (exit_address.raw(), Vec::new()),
                (refund_address.raw(), Vec::new()),
            ],
            Vec::new(),
            handler,
        )
    }
