### Title
Missing Commission Slippage Protection in `switch_delegation_pool` Enables Front-Running by Destination Staker - (File: src/pool/pool.cairo)

### Summary
`switch_delegation_pool` in `src/pool/pool.cairo` accepts no `max_commission` parameter. A destination staker who holds an active `CommissionCommitment` can atomically front-run a pool member's switch transaction by calling `set_commission` to raise their rate up to `max_commission`, causing the pool member to land in a pool with a higher commission than they intended. This is the staking analog of the missing slippage protection in `formPOL`.

### Finding Description
`switch_delegation_pool` (pool.cairo lines 379–429) performs the following checks before routing funds to the destination pool:

- `amount.is_non_zero()`
- `pool_member_info.unpool_time.is_some()`
- `amount <= pool_member_info.unpool_amount` [1](#0-0) 

There is **no check on the commission rate of `to_pool`**. The pool member cannot express a maximum acceptable commission as part of the call.

Separately, `update_commission` in `staking.cairo` contains a branch that explicitly permits a staker to **increase** their commission whenever an active `CommissionCommitment` is in effect:

```
if self.is_commission_commitment_active(:commission_commitment) {
    assert!(commission <= commission_commitment.max_commission, ...);
    assert!(commission != old_commission, ...);   // only "different", not "lower"
}
``` [2](#0-1) 

The code itself carries a warning about this: *"Current commission increase safeguards still allow for sudden commission changes."* [3](#0-2) 

`set_commission` is callable by any staker at any time (no time-lock, no epoch delay): [4](#0-3) 

The combination means a destination staker can atomically sandwich a pool member's `switch_delegation_pool` call:

1. Observe the pending `switch_delegation_pool(to_staker=A, to_pool=poolA, amount)` transaction.
2. Submit `set_commission(max_commission)` ahead of it (sequencer ordering or same block).
3. The pool member's switch executes and they are enrolled in `poolA` at the elevated commission.

### Impact Explanation
The pool member's future staking rewards are split using the elevated commission rate for as long as they remain in the pool. The staker captures the excess commission that the pool member never agreed to. This constitutes **theft of unclaimed yield** (High severity): every reward epoch that passes while the pool member is unknowingly in a high-commission pool transfers yield from the pool member to the staker.

The magnitude is unbounded: if `max_commission = COMMISSION_DENOMINATOR` (10 000 basis points = 100 %), the staker can absorb the pool member's entire yield share. [5](#0-4) 

### Likelihood Explanation
- **Precondition**: The destination staker must have an active `CommissionCommitment` with `max_commission` strictly greater than their current commission. This is a normal, advertised protocol feature that many stakers will use to attract delegators.
- **Execution**: On Starknet the sequencer currently orders transactions; a staker who controls or monitors the sequencer, or who simply submits their `set_commission` in the same block, can reliably front-run. Even without sequencer access, a staker can watch for `SwitchDelegationPool` events from their own pool and react within the same epoch.
- **Victim awareness**: Pool members have no on-chain mechanism to atomically check commission and switch in one call, so they cannot defend themselves without protocol-level protection.

### Recommendation
Add a `max_commission: Commission` parameter to `switch_delegation_pool` and assert the destination pool's current commission does not exceed it before completing the switch:

```cairo
fn switch_delegation_pool(
    ref self: ContractState,
    to_staker: ContractAddress,
    to_pool: ContractAddress,
    amount: Amount,
    max_commission: Commission,   // <-- new
) -> Amount {
    ...
    // After resolving to_staker's commission:
    let actual_commission = staking_dispatcher.get_commission(to_staker);
    assert!(actual_commission <= max_commission, Error::COMMISSION_EXCEEDS_SLIPPAGE);
    ...
}
```

This mirrors the standard slippage-protection pattern and gives pool members a trustless guarantee about the terms they are accepting.

### Proof of Concept

**Setup**
- Staker A: `commission = 500` (5 %), `CommissionCommitment { max_commission: 10000, expiration_epoch: E+50 }` (active).
- Pool member M: currently delegating to Staker B, calls `exit_delegation_pool_intent(amount=1000)`.

**Attack**
1. M submits `pool_B.switch_delegation_pool(to_staker=A, to_pool=poolA, amount=1000)`.
2. Staker A observes the pending transaction and submits `staking.set_commission(10000)` first.
3. `update_commission` passes because `10000 <= max_commission (10000)` and `10000 != 500`. [6](#0-5) 

4. M's `switch_delegation_pool` executes: funds move to `poolA`, M is enrolled with commission = 10 000 (100 %). [7](#0-6) 

5. Every subsequent reward epoch, `split_rewards_with_commission` routes 100 % of M's pool rewards to Staker A as commission; M receives zero yield.

**Result**: M loses all future staking yield with no recourse, having had no way to specify a commission bound in `switch_delegation_pool`.

            // Prerequisites and asserts.
            self.general_prerequisites();
            assert!(commission <= COMMISSION_DENOMINATOR, "{}", Error::COMMISSION_OUT_OF_RANGE);
            let staker_address = get_caller_address();

                    );
