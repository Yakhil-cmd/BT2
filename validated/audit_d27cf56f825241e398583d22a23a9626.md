### Title
`validate_tx()` omits `add_ready_declares()`, causing stale delayed-declare state to incorrectly reject valid fee-escalation transactions — (File: crates/apollo_mempool/src/mempool.rs)

### Summary

`Mempool::validate_tx()` does not call `add_ready_declares()` before running its duplicate-nonce checks, while both `add_tx()` and `get_txs()` do. When a delayed declare's delay period has elapsed but `add_ready_declares()` has not yet been triggered, `validate_tx()` still observes the declare in `delayed_declares` and returns `DuplicateNonce` for a fee-escalation transaction at the same `(address, nonce)`. The same transaction would be accepted by `add_tx()`, which calls `add_ready_declares()` first and moves the now-ready declare into `tx_pool` before running fee-escalation logic. Because the gateway calls `validate_tx()` before `add_tx()`, a false rejection from `validate_tx()` prevents `add_tx()` from ever being reached.

### Finding Description

`add_tx()` and `get_txs()` both call `add_ready_declares()` as their first substantive action in non-FIFO mode: [1](#0-0) [2](#0-1) 

`add_ready_declares()` pops every entry from `delayed_declares` whose submission time is older than `declare_delay` and inserts it into `tx_pool` via `add_tx_inner()`: [3](#0-2) 

`validate_tx()` performs no such consolidation step: [4](#0-3) 

It calls `validate_fee_escalation()`, which calls `validate_no_delayed_declare_front_run()`: [5](#0-4) 

This check looks only at `delayed_declares`, not at `tx_pool`. If a declare's delay has elapsed but `add_ready_declares()` has not yet been called, the declare is still in `delayed_declares`. Any incoming transaction at the same `(address, nonce)` — including a legitimate fee-escalation replacement — is rejected with `DuplicateNonce`.

Had `add_ready_declares()` been called first, the declare would have been moved to `tx_pool`, `validate_no_delayed_declare_front_run()` would find nothing in `delayed_declares`, and `validate_fee_escalation()` would proceed to the normal fee-escalation path in `tx_pool`, accepting the replacement when `enable_fee_escalation = true` and the incoming tip/price satisfies the threshold.

The gateway calls `validate_tx()` (via `validate_by_mempool()`) before `add_tx()`: [6](#0-5) 

A false rejection from `validate_tx()` causes the gateway to return an error to the caller and never invoke `add_tx()`.

The gateway's `validate_by_mempool()` path: [7](#0-6) [8](#0-7) 

### Impact Explanation

A user who submitted a declare transaction (which was placed in `delayed_declares`) and later attempts to replace it with a higher-fee transaction after the delay has elapsed will receive a `DuplicateNonce` error from the gateway. The replacement is a valid transaction — `add_tx()` would accept it — but `validate_tx()` rejects it because it operates on a stale view of `delayed_declares`. This constitutes **mempool/gateway admission incorrectly rejecting a valid transaction before sequencing** (High impact per the defined scope).

### Likelihood Explanation

Requires: (1) `enable_fee_escalation = true` in `MempoolStaticConfig`; (2) a declare transaction currently sitting in `delayed_declares` whose `declare_delay` has elapsed; (3) the user submitting a fee-escalation replacement before any other `add_tx()` or `get_txs()` call triggers `add_ready_declares()`. The window is bounded by the `declare_delay` configuration and the rate of other mempool activity, making it a realistic race condition in production.

### Recommendation

Call `add_ready_declares()` at the start of `validate_tx()` (guarded by `!self.is_fifo()`, matching the pattern in `add_tx()` and `get_txs()`):

```rust
pub fn validate_tx(&mut self, args: ValidationArgs) -> MempoolResult<()> {
    if !self.is_fifo() {
        self.add_ready_declares();
    }
    let tx_reference = (&args).into();
    self.validate_incoming_tx(tx_reference, args.account_nonce)?;
    self.validate_fee_escalation(tx_reference)?;
    Ok(())
}
```

This ensures `validate_tx()` and `add_tx()` operate on the same consolidated view of `delayed_declares` vs. `tx_pool`.

### Proof of Concept

```
1. Create Mempool with declare_delay = 5s, enable_fee_escalation = true,
   fee_escalation_percentage = 0 (accept any higher fee).

2. add_tx(declare D, address=0x0, nonce=0, tip=10)
   → D enters delayed_declares (not tx_pool).

3. Advance clock by 6 seconds (delay elapsed; D is now "ready").

4. Build replacement invoke I: address=0x0, nonce=0, tip=20 (higher fee).

5. validate_tx(ValidationArgs from I)
   → validate_fee_escalation → validate_no_delayed_declare_front_run
   → delayed_declares.contains(0x0, 0) == true  (D still there, not yet moved)
   → returns Err(DuplicateNonce)   ← WRONG: should be Ok(())

6. add_tx(I)  [called directly, bypassing validate_tx]
   → add_ready_declares() moves D from delayed_declares to tx_pool
   → validate_fee_escalation: D found in tx_pool, tip 20 > 10 → replacement accepted
   → returns Ok(())                ← CORRECT

Gateway flow stops at step 5; add_tx in step 6 is never reached.
``` [4](#0-3) [1](#0-0) [3](#0-2) [5](#0-4)

    fn add_ready_declares(&mut self) {
        let now = self.clock.now();
        while let Some((submission_time, _args)) = self.delayed_declares.front() {
            if now - self.config.static_config.declare_delay < *submission_time {
                break;
            }
            let (_submission_time, args) =
                self.delayed_declares.pop_front().expect("Delay declare should exist.");
            self.add_tx_inner(args);
        }
        self.update_state_metrics();
    }
