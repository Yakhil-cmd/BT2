### Title
Missing Phase-Completion Check in `reliable_broadcast_receive_all` Enables Equivocation-Based Denial of DKG for n=3 Configurations — (File: `src/protocol/echo_broadcast.rs`)

---

### Summary

In `reliable_broadcast_receive_all`, the `MessageType::Ready` processing arm does not verify that the Send or Echo phases have completed before accepting a Ready message as valid. When combined with `echo_ready_thresholds` returning `(0, 0)` for `n <= 3`, a single Ready message from any participant immediately satisfies the delivery threshold. A malicious participant in a 3-party DKG can exploit this to send different Ready messages to different honest parties (equivocation), causing them to derive inconsistent combined session IDs and permanently failing key generation.

---

### Finding Description

The `echo_ready_thresholds` function in `src/protocol/echo_broadcast.rs` returns `(echo_t=0, ready_t=0)` for any participant count `n <= 3`: [1](#0-0) 

With `ready_t = 0`, the delivery condition `state_sid.data_ready.get(&data) > 2 * ready_t` reduces to `count > 0`, meaning a **single** Ready message from any participant triggers immediate delivery.

In the `MessageType::Ready` processing arm, there is no check that `state_sid.finish_send` or `state_sid.finish_echo` are `true` before accepting the message: [2](#0-1) 

The only guards are `seen_ready.put(from)` (deduplication per sender) and `finish_ready` (post-delivery skip). There is no guard requiring that the Send phase (`finish_send`) or Echo phase (`finish_echo`) completed for the session identified by `sid`. This is the direct analog to the Phoenix SDK's missing `is_err` check: just as the SDK processes events from failed transactions without checking the error flag, this code processes Ready messages without checking whether the prior protocol phases succeeded.

The `do_broadcast` function used throughout `do_keyshare` in `src/dkg.rs` relies entirely on `reliable_broadcast_receive_all` for its consistency guarantee: [3](#0-2) 

The first use of `do_broadcast` in `do_keyshare` is to exchange session IDs: [4](#0-3) 

The combined `session_id` derived from this broadcast is then used as a binding domain separator for all subsequent commitment hashes and proofs of knowledge: [5](#0-4) 

---

### Impact Explanation

**High — Permanent denial of DKG for honest parties in n=3 configurations.**

A malicious participant M in a 3-party DKG `{A, B, M}` can:

1. Skip the Send phase for its own session entirely.
2. Send `Ready(session_id_X)` directly to A and `Ready(session_id_Y)` (where `session_id_X ≠ session_id_Y`) directly to B, for M's session slot.
3. With `ready_t = 0`, each honest party delivers immediately upon receiving the single Ready message.
4. A computes `combined_session_id_A = H(..., session_id_X)` and B computes `combined_session_id_B = H(..., session_id_Y)`.
5. All subsequent commitment hashes are bound to these divergent session IDs.

At the commitment hash verification step, A verifies B's commitment hash using `combined_session_id_A`, but B computed it using `combined_session_id_B`: [6](#0-5) 

This returns `ProtocolError::InvalidCommitmentHash` for both honest parties. Even if this step were bypassed, `broadcast_success` would detect the mismatch: [7](#0-6) 

The DKG permanently fails for all honest parties whenever M participates. M can repeat this attack on every DKG attempt.

---

### Likelihood Explanation

**Medium-High.** The minimum supported participant count is 2 (`assert_key_invariants` enforces `participants.len() >= 2`), and 3-party configurations are fully valid inputs: [8](#0-7) 

The `echo_ready_thresholds` comment documents the n ≤ 3 assumption only as an internal code comment, not as a user-facing API constraint. Any caller using a 3-party DKG is silently exposed. The attack requires only that M send two crafted Ready messages — no cryptographic capability is needed.

---

### Recommendation

In the `MessageType::Ready` arm of `reliable_broadcast_receive_all`, add a guard that skips (or errors on) Ready messages for sessions where the Send phase has not yet completed:

```rust
MessageType::Ready(data) => {
    // Reject Ready messages that arrive before the Send phase completed.
    if !state_sid.finish_send {
        continue;
    }
    if !state_sid.seen_ready.put(from) || state_sid.finish_ready {
        continue;
    }
    // ... rest of Ready handling
}
```

Alternatively, enforce `n >= 4` at the API level (`assert_key_invariants`) if Byzantine fault tolerance is a security requirement, and document the n ≤ 3 limitation explicitly in the public API.

---

### Proof of Concept

**Setup**: 3 participants `A`, `B`, `M` (M is malicious). DKG threshold = 2.

**Attack during `do_broadcast` for session IDs (first call in `do_keyshare`)**:

1. M does **not** call `reliable_broadcast_send` for its own session slot (skips the Send phase).
2. M directly sends `MessageType::Ready(session_id_X)` to A, tagged with M's session index `sid_M`.
3. M directly sends `MessageType::Ready(session_id_Y)` to B, tagged with `sid_M`, where `session_id_X ≠ session_id_Y`.

**In A's `reliable_broadcast_receive_all`**:
- `seen_ready.put(M)` → `true` (first time from M).
- `finish_ready` → `false`.
- `data_ready` count for `session_id_X` becomes 1.
- `1 > ready_t` (= 0) → amplification fires, A sends `Ready(session_id_X)` to all.
- `1 > 2 * ready_t` (= 0) → delivery fires, A records `session_id_X` for M's slot.

**In B's `reliable_broadcast_receive_all`**: same logic, B records `session_id_Y` for M's slot.

**Result**:
- `combined_session_id_A = domain_separate_hash(session_ids_A, session_ids_B, session_id_X)`
- `combined_session_id_B = domain_separate_hash(session_ids_A, session_ids_B, session_id_Y)`
- These differ. At `verify_commitment_hash` (dkg.rs line 463), both A and B return `ProtocolError::InvalidCommitmentHash`. DKG is permanently denied for both honest parties. [9](#0-8) [10](#0-9)

fn echo_ready_thresholds(n: usize) -> (usize, usize) {
    // case where no malicious parties are assumed: when n <= 3/
    // In this case the echo and ready thresholds are both 0
    // later we compare if we have collected more votes than these thresholds
    if n <= 3 {
        return (0, 0);
    }
    // we should always have n >= 3*threshold + 1
    let broadcast_threshold = (n - 1) / 3;
    let echo_threshold = usize::midpoint(n, broadcast_threshold);
    (echo_threshold, broadcast_threshold)
}
