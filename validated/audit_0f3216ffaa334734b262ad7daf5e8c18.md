### Title
Missing Identity-Point Validation on `app_pk` in `ckd()` Allows Coordinator to Recover Confidential Derived Key — (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

The public `ckd()` entry point in `src/confidential_key_derivation/protocol.rs` accepts an `app_pk: PublicKey` parameter (a BLS12-381 G1 projective point) without checking whether it is the group identity element. When `app_pk` is the identity, the masking term `app_pk * y` inside `compute_signature_share` collapses to the identity, causing every participant's `big_c` share to equal their unmasked `big_s = msk_share * H(pk, app_id)`. The coordinator, who aggregates all shares, then directly observes `big_C = msk * H(pk, app_id)` — the confidential derived key — without needing `app_sk`.

---

### Finding Description

The CKD protocol is designed so that the MPC nodes (including the coordinator) never learn the derived key `msk * H(pk, app_id)`. The masking is achieved by each participant adding `app_pk * y_i` to their share before sending it: [1](#0-0) 

```
big_y = generator * y
big_s = H(pk, app_id) * private_share      // unmasked share
big_c = big_s + app_pk * y                 // masked share
```

The coordinator sums the Lagrange-weighted contributions:

```
big_C = Σ λᵢ · big_cᵢ = msk · H(pk, app_id) + app_pk · Σ λᵢ · yᵢ
big_Y = Σ λᵢ · big_yᵢ = generator · Σ λᵢ · yᵢ
```

The client recovers the key via `big_C − app_sk · big_Y = msk · H(pk, app_id)`.

**When `app_pk = identity`:**

```
big_c = big_s + identity * y = big_s = msk_share * H(pk, app_id)
big_C = msk * H(pk, app_id)   ← confidential key exposed in plaintext
```

The coordinator, who receives and sums all `big_c` values, directly observes the confidential derived key.

The `ckd()` function validates participants, coordinator membership, and duplicate detection, but performs **no check** that `app_pk` is not the identity: [2](#0-1) 

The internal `compute_signature_share` also performs no such check: [3](#0-2) 

This is structurally identical to the reported pattern: an internal function (`compute_signature_share`) is currently only called from paths where `app_pk` is expected to be valid, but the public API (`ckd()`) does not enforce this invariant, leaving the door open for callers — buggy or malicious — to pass the identity.

---

### Impact Explanation

**Critical — Disclosure of confidential derived secrets.**

When all participants call `ckd()` with `app_pk = identity`, the coordinator's aggregated `big_C` equals `msk * H(pk, app_id)` in plaintext. The coordinator is an MPC node that is explicitly supposed to be unable to learn this value. The entire confidentiality guarantee of the CKD protocol is defeated for that derivation.

This matches the allowed impact: *"Extraction, reconstruction, or disclosure of … confidential derived secrets."*

---

### Likelihood Explanation

**Medium.**

The `app_pk` is a caller-supplied `ElementG1` value with no enforced constraints at the API boundary. Realistic triggering paths include:

1. **Buggy application**: The default/zero value of `blstrs::G1Projective` is the identity. An uninitialized or default-constructed `app_pk` silently passes all current validation and triggers the disclosure.
2. **Malicious application layer**: An adversary controlling the application that distributes `app_pk` to all MPC nodes can supply the identity point, causing the coordinator to learn the derived key for every CKD invocation using that `app_pk`.

No cryptographic break, key compromise, or privileged access is required — only passing a single zero-valued group element to the public API.

---

### Recommendation

Add an explicit identity-point check for `app_pk` in `ckd()` before the protocol is launched:

```rust
// In ckd(), after participant validation:
if bool::from(app_pk.is_identity()) {
    return Err(InitializationError::BadParameters(
        "app_pk must not be the identity element".to_string(),
    ));
}
```

Optionally, mirror the check inside `compute_signature_share` as a defense-in-depth assertion, consistent with how the codebase already guards against zero scalars and identity elements in other protocol steps: [4](#0-3) 

---

### Proof of Concept

```rust
// All participants call ckd() with app_pk = G1 identity
let app_pk = blstrs::G1Projective::identity(); // zero point, no error raised

// Inside compute_signature_share for each participant i:
//   big_c_i = big_s_i + identity * y_i = big_s_i
//           = H(pk, app_id) * private_share_i

// Coordinator aggregates:
//   big_C = Σ λᵢ · big_s_i = msk · H(pk, app_id)
//   big_Y = Σ λᵢ · generator · y_i  (random, irrelevant)

// Coordinator's CKDOutput.big_c == msk * H(pk, app_id)
// The confidential derived key is directly visible to the coordinator.
```

The `ckd()` function currently accepts `app_pk = G1Projective::identity()` without error, as confirmed by the absence of any identity check in the validation block at lines 76–101 of `src/confidential_key_derivation/protocol.rs`. [5](#0-4)

pub fn ckd(
    participants: &[Participant],
    coordinator: Participant,
    me: Participant,
    key_pair: KeygenOutput,
    app_id: impl Into<AppId>,
    app_pk: PublicKey,
    rng: impl CryptoRngCore + Send + 'static,
) -> Result<impl Protocol<Output = CKDOutputOption>, InitializationError> {
    // not enough participants
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }

    // kick out duplicates
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }

    let comms = Comms::new();
    let chan = comms.shared_channel();

    let fut = run_ckd_protocol(
        chan,
        coordinator,
        me,
        participants,
        key_pair,
        app_id.into(),
        app_pk,
        rng,
    );
    Ok(make_protocol(comms, fut))
}

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}
