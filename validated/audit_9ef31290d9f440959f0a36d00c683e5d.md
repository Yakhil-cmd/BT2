### Title
`get_token_decimals` Returns Zero for STRK Token, Causing Near-Total Loss of Funds on Pool Exit - (File: `src/staking/staking.cairo`)

### Summary

`get_token_decimals` reads STRK token decimals from an uninitialized storage slot, returning `0` instead of `18`. When `remove_from_delegation_pool_action` uses this value to convert a `NormalizedAmount` back to native units before transferring STRK to a pool contract, the computed transfer amount is `normalized_amount / 10^18` instead of `normalized_amount`, causing pool members to receive approximately zero STRK while their funds are permanently frozen in the staking contract.

### Finding Description

`get_token_decimals` is a thin wrapper that reads from the `token_decimals` storage map: [1](#0-0) 

The only code path that writes to `token_decimals` is `add_token`, which explicitly **excludes** STRK: [2](#0-1) 

Because `token_decimals` is a `Map<ContractAddress, u8>` and STRK's entry is never written, `get_token_decimals(STRK_TOKEN_ADDRESS)` returns `0` (Cairo's default for uninitialized storage). [3](#0-2) 

`remove_from_delegation_pool_action` then uses this zero value to convert the stored `NormalizedAmount` back to a native transfer amount: [4](#0-3) 

`to_native_amount(decimals)` computes `amount / 10^(18 - decimals)`. With `decimals = 0`, this becomes `amount / 10^18`. For a pool member holding 1,000 STRK (normalized: `1000 × 10^18`), the computed `native_amount` is `1000 × 10^18 / 10^18 = 1000` — i.e., 1000 of STRK's smallest units (≈ `10^-15` STRK) — instead of `1000 × 10^18` units (= 1,000 STRK). The undelegate intent is cleared at line 1129 before the transfer, so the pool member cannot retry. [5](#0-4) 

The rest of the STRK remains permanently locked in the staking contract.

This is the direct analog of the reported issue: the original bug calls `.decimals()` on the zero address for native ETH and reverts; here, `get_token_decimals` silently returns `0` for STRK (the protocol's "native" staking token), producing a catastrophically wrong conversion factor.

### Impact Explanation

**High — Permanent freezing of funds.**

Any pool member who calls `exit_delegation_pool_action` on a STRK delegation pool loses essentially their entire delegated balance. The staking contract retains the STRK, the intent is cleared, and there is no recovery path. The pool contract receives a dust amount and forwards it to the pool member.

### Likelihood Explanation

**High.** STRK delegation pools are the primary pool type in the protocol. Every pool member who completes the exit flow (`exit_delegation_pool_intent` → `exit_delegation_pool_action`) for a STRK pool triggers this path. No special conditions or privileged access are required; any unprivileged pool member is affected.

### Recommendation

Add a special case for STRK in `get_token_decimals`, mirroring the pattern already used by `to_strk_native_amount`:

```cairo
fn get_token_decimals(self: @ContractState, token_address: ContractAddress) -> u8 {
    if token_address == STRK_TOKEN_ADDRESS {
        return 18_u8;
    }
    self.token_decimals.read(token_address)
}
```

Alternatively, initialize `token_decimals` for `STRK_TOKEN_ADDRESS` to `18` in the constructor.

### Proof of Concept

1. Staker creates a STRK delegation pool.
2. Pool member deposits 1,000 STRK via `enter_delegation_pool`.
3. Pool member calls `exit_delegation_pool_intent(amount: 1_000_000_000_000_000_000_000)` (1,000 STRK in native units).
4. Pool contract calls `remove_from_delegation_pool_intent` on the staking contract; the intent is stored with `token_address = STRK_TOKEN_ADDRESS` and `amount = 1_000 × 10^18` (normalized).
5. After the exit window, pool member calls `exit_delegation_pool_action`.
6. Pool contract calls `remove_from_delegation_pool_action(identifier)`.
7. Inside: `token_address = STRK_TOKEN_ADDRESS`, `decimals = get_token_decimals(STRK_TOKEN_ADDRESS) = 0`.
8. `native_amount = (1_000 × 10^18) / 10^(18-0) = 1_000` — 1,000 units of STRK (= `10^-15` STRK).
9. `checked_transfer` sends 1,000 units to the pool contract; pool member receives dust.
10. The remaining `≈ 1,000 × 10^18` units of STRK are permanently frozen in the staking contract.
