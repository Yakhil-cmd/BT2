### Title
WalletConnect Commands Execute Silently After No-Op Logout Due to Persistent Daemon Session — (`packages/core/src/components/Auth/AuthProvider.tsx`, `packages/gui/src/hooks/useWalletConnectCommand.tsx`)

### Summary
`AuthProvider.handleLogOut` is intentionally a no-op. The daemon is never told to log out, so `useGetLoggedInFingerprintQuery()` continues returning the previous fingerprint. Because `WalletConnectProvider` remains mounted and its `session_request` listener stays active, a previously-paired WC dApp can send commands that pass all fingerprint guards and — for bypass-enabled commands — execute without any confirmation dialog, on a wallet the user believes they have logged out of.

### Finding Description
`handleLogOut` does nothing:

```typescript
// packages/core/src/components/Auth/AuthProvider.tsx
const handleLogOut = useCallback(async () => {
  // do nothing until backend change API,
  // user is still logged in and syncing with previously selected fingerprint
}, []);
```

`LogoutButton` calls `logOut()` (the no-op) then navigates to `/`. The user sees the wallet-selection screen and believes the session is terminated. However:

1. The daemon remains logged in with fingerprint X.
2. `WalletConnectProvider` is mounted above the router and is never unmounted.
3. The `client.on('session_request', handleSessionRequest)` listener remains active.
4. When the dApp sends a `session_request`, `handleProcess` in `useWalletConnectCommand` reads `currentFingerprint` from the RTK Query cache (still X) and compares it to `pair.fingerprint` (also X) — the guard passes.
5. `window.permissionsAPI.dispatchAsPair` calls into the main process, which calls `getLoggedInFingerprint()` directly from the daemon (still X) — the main-process guard also passes.
6. For commands in `pair.bypass`, `dispatchPairRequest` skips the confirmation dialog entirely and executes the command.

### Impact Explanation
A WC dApp that the user previously paired with can silently execute spending commands (`chia_sendTransaction`, offer acceptance, etc.) on a wallet the user believed they had logged out of. For bypass-enabled commands the execution is completely silent — no dialog, no notification, no UI indication. This directly maps to **unauthorized spend/transfer of XCH, CAT, or NFT assets**.

### Likelihood Explanation
- Precondition: the attacker must be a dApp the user previously paired with (reasonable — this is the normal WC workflow).
- The user must have clicked "Log Out" (common action when switching wallets or stepping away).
- For the silent path, the user must have previously enabled bypass for a spending command (possible; the UI offers this option).
- No cryptographic material, leaked keys, or host compromise is required.

### Recommendation
`handleLogOut` must either (a) call a daemon API to actually log out the active fingerprint, or (b) at minimum set `fingerprint` state to `undefined` in `AuthProvider` and propagate that as a guard into `handleProcess` — rejecting any WC command when the UI-level fingerprint is `undefined`. Additionally, `WalletConnectProvider` should check `AuthContext.fingerprint` before processing any `session_request`.

### Proof of Concept
1. Pair a WC dApp with wallet fingerprint X; enable bypass for `chia_sendTransaction`.
2. Click "Log Out" in the GUI — observe navigation to `/` (wallet-selection screen).
3. From the dApp, send a `session_request` for `chia_sendTransaction`.
4. Assert: `dispatchAsPair` is called and the transaction is submitted — no confirmation dialog appears, and the user receives no indication that their "logged-out" wallet was accessed.
