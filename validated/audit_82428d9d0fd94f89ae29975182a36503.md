### Title
`ExitToEthereum` Precompile Has No Failure-Recovery Callback — Permanent EVM Fund Freeze on NEAR Promise Failure - (File: `engine-precompiles/src/native.rs`)

---

### Summary

The `ExitToEthereum` precompile always schedules the NEAR `withdraw` promise as a fire-and-forget `PromiseArgs::Create` with no attached callback. When the EVM call executes, the user's ETH (or ERC-20 tokens) is irrevocably deducted from their EVM balance. If the downstream NEAR `withdraw` promise fails for any reason, the EVM state and the NEAR eth-connector state permanently desync: the EVM shows the funds as gone, but the NEAR side never processed the withdrawal. There is no refund path.

---

### Finding Description

In `ExitToEthereum::run()`, after deducting ETH from the caller's EVM balance (via `context.apparent_value`) or burning ERC-20 tokens (via the ERC-20 contract calling the precompile), the code unconditionally constructs the outbound NEAR promise as:

```rust
let promise = borsh::to_vec(&PromiseArgs::Create(withdraw_promise)).unwrap();
``` [1](#0-0) 

There is no `PromiseArgs::Callback` variant here — no `exit_to_ethereum_precompile_callback` method exists anywhere in the codebase. This is structurally different from `ExitToNear`, which at least has a conditional callback path (gated by the `error_refund` feature flag) that calls `engine::refund_on_error` to re-mint burned tokens or return ETH: [2](#0-1) 

The `refund_on_error` function for the ETH case transfers ETH back from the `exit_to_near::ADDRESS` precompile address to the original sender: [3](#0-2) 

No equivalent exists for `ExitToEthereum`. The `ExitToEthereum` precompile address (`0xb0bd02f6a392af548bdf1cfaee5dfa0eefcc8eab`) is not a real contract; ETH stranded there cannot be recovered by the user. [4](#0-3) 

The eth-connector's `withdraw` method can fail for several reachable reasons: the connector is paused via `pa_pause_feature` (the codebase has a full pausability system), the connector contract panics, or the promise runs out of gas. The pause path is directly user-observable: [5](#0-4) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

When `ExitToEthereum::run()` executes inside the EVM:
- For the ETH case: the caller's EVM ETH balance is reduced by `context.apparent_value` and credited to the precompile address. If the NEAR `withdraw` promise fails, the ETH is permanently stranded at the precompile address with no recovery path.
- For the ERC-20 case: the ERC-20 contract burns tokens before calling the precompile. If the NEAR `withdraw` promise fails, the ERC-20 tokens are permanently destroyed with no re-mint.

In both cases the EVM internal accounting and the NEAR eth-connector state desync: the EVM records the funds as exited, but the NEAR side never processed the exit. The user loses their funds permanently.

---

### Likelihood Explanation

**Medium.** The NEAR `withdraw` promise can fail via:

1. **Eth-connector pause**: The eth-connector contract has a `pa_pause_feature` mechanism. If `withdraw` is paused at the time the promise executes, the call fails. Any user who submitted `ExitToEthereum` while the connector was about to be paused loses funds.
2. **Gas exhaustion**: `WITHDRAWAL_GAS` is set to `100_000_000_000_000` TGas. If the promise chain consumes more gas than allocated, the `withdraw` call fails silently.
3. **Connector contract error**: Any panic or assertion failure inside the eth-connector's `withdraw` implementation causes the promise to fail.

The attacker-controlled entry path is straightforward: any EVM user can call the `ExitToEthereum` precompile at address `0xb0bd02f6a392af548bdf1cfaee5dfa0eefcc8eab` by sending an EVM transaction with ETH value or by triggering an ERC-20 burn that calls the precompile. [6](#0-5) 

---

### Recommendation

Add a callback to `ExitToEthereum` analogous to the `exit_to_near_precompile_callback` pattern. The callback should check `handler.promise_result(0)` and, on failure, re-credit the ETH to the original sender's EVM address (for the ETH case) or re-mint the ERC-20 tokens (for the ERC-20 case), mirroring the `refund_on_error` logic already present in `engine/src/engine.rs`. The `ExitToNear` path with `error_refund` enabled demonstrates the correct pattern: [7](#0-6) [8](#0-7) 

---

### Proof of Concept

1. User holds 1 ETH in their Aurora EVM account.
2. The eth-connector's `withdraw` feature is paused by the owner (or is about to be paused).
3. User submits an EVM transaction calling `ExitToEthereum` precompile (`0xb0bd02f6a392af548bdf1cfaee5dfa0eefcc8eab`) with `value = 1 ETH`.
4. The EVM executes: 1 ETH is deducted from the user's EVM balance and credited to the precompile address. The precompile schedules `PromiseArgs::Create(withdraw_promise)` targeting the eth-connector's `withdraw` method.
5. The NEAR promise executes: the eth-connector's `withdraw` is paused → the promise fails.
6. No callback fires. The EVM state shows the user's balance as 0 ETH. The eth-connector state is unchanged. The 1 ETH is permanently stranded at the `ExitToEthereum` precompile address with no recovery mechanism.

The existing test for `ExitToNear` explicitly documents the analogous loss when `error_refund` is disabled, confirming the design gap: [9](#0-8)

    fn run(
        &self,
        input: &[u8],
        target_gas: Option<EthGas>,
        context: &Context,
        is_static: bool,
    ) -> EvmPrecompileResult {
        // ETH (Base token) transfer input format (min size 21 bytes)
        //  - flag (1 byte)
        //  - eth_recipient (20 bytes)
        // ERC-20 transfer input format: max 53 bytes
        //  - flag (1 byte)
        //  - amount (32 bytes)
        //  - eth_recipient (20 bytes)

        let required_gas = Self::required_gas(input)?;

        if let Some(target_gas) = target_gas
            && required_gas > target_gas
        {
            return Err(ExitError::OutOfGas);
        }

        // It's not allowed to call exit precompiles in static mode
        if is_static {
            return Err(ExitError::Other(Cow::from("ERR_INVALID_IN_STATIC")));
        } else if context.address != exit_to_ethereum::ADDRESS.raw() {
            return Err(ExitError::Other(Cow::from("ERR_INVALID_IN_DELEGATE")));
        }

pub fn refund_on_error<I: IO + Copy, E: Env, P: PromiseHandler>(
    io: I,
    env: &E,
    state: EngineState,
    args: &RefundCallArgs,
    handler: &mut P,
) -> EngineResult<SubmitResult> {
    let current_account_id = env.current_account_id();
    if let Some(erc20_address) = args.erc20_address {
        // ERC-20 exit; re-mint burned tokens
        let erc20_admin_address = current_address(&current_account_id);
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, erc20_admin_address, current_account_id, io, env);

        let refund_address = args.recipient_address;
        let amount = U256::from_big_endian(&args.amount);
        let input = setup_refund_on_error_input(amount, refund_address);

        engine.call(
            &erc20_admin_address,
            &erc20_address,
            Wei::zero(),
            input,
            u64::MAX,
            Vec::new(),
            Vec::new(),
            handler,
        )
    } else {
        // ETH exit; transfer ETH back from precompile address
        let exit_address = exit_to_near::ADDRESS;
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, exit_address, current_account_id, io, env);
        let refund_address = args.recipient_address;
        let amount = Wei::new(U256::from_big_endian(&args.amount));
        engine.call(
            &exit_address,
            &refund_address,
            amount,
            Vec::new(),
            u64::MAX,
            vec![
                (exit_address.raw(), Vec::new()),
                (refund_address.raw(), Vec::new()),
            ],
            Vec::new(),
            handler,
        )
    }
