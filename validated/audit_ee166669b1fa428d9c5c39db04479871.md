### Title
Coordinator Has Exclusive, Unverifiable Control Over CKD Output — (File: `src/confidential_key_derivation/protocol.rs`)

### Summary
The coordinator role in the CKD protocol holds all privileges for aggregating participant contributions and producing the final derived-key ciphertext `(Y, C)`. Non-coordinator participants send their secret contributions privately to the coordinator and receive no output and perform no verification. A single malicious coordinator can silently substitute or fabricate the aggregated output, causing the requesting application to derive a wrong or unusable confidential key, or can simply refuse to produce output, permanently denying CKD for all honest parties.

### Finding Description
In `do_ckd_participant` each non-coordinator participant computes its blinded share `(norm_big_y, norm_big_c)` and sends it **privately** to the coordinator, then immediately returns `None` with no further participation:

```rust
// src/confidential_key_derivation/protocol.rs lines 27-32
let (norm_big_y, norm_big_c) =
    compute_signature_share(participants, me, key_pair, app_id, app_pk, rng)?;
let waitpoint = chan.next_waitpoint();
chan.send_private(waitpoint, coordinator, &(norm_big_y, norm_big_c))?;
Ok(None)
```

In `do_ckd_coordinator` the coordinator aggregates all received contributions and returns the sole `Some(CKDOutput)`:

```rust
// src/confidential_key_derivation/protocol.rs lines 44-57
let (mut norm_big_y, mut norm_big_c) =
    compute_signature_share(&participants, me, key_pair, app_id, app_pk, rng)?;
let waitpoint = chan.next_waitpoint();
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
Ok(Some(ckd_output))
```

There is no broadcast, no commitment scheme, no zero-knowledge proof, and no cross-check that allows any non-coordinator participant to verify that the coordinator included all contributions faithfully. The coordinator can:

1. **Drop** one or more participants' `(λ_i Y_i, λ_i C_i)` terms, producing a `(Y, C)` that does not encode `msk · H(pk, app_id)`.
2. **Inject** arbitrary extra group elements, shifting the aggregated ciphertext to an attacker-chosen value.
3. **Return a fully fabricated** `CKDOutput` with no relation to any honest participant's share.
4. **Refuse to return any output** (abort after receiving all shares), permanently blocking the CKD session.

The same structural pattern applies to all signing protocols: in `do_sign_coordinator` (robust ECDSA, `src/ecdsa/robust_ecdsa/sign.rs` lines 127–166) and `do_sign_coordinator_v1` (FROST EdDSA, `src/frost/eddsa/sign.rs` lines 99–168), participants send signature shares exclusively to the coordinator, which is the sole producer of the final signature. Non-coordinator participants return `None` and cannot verify the coordinator's aggregation.

### Impact Explanation
A malicious coordinator can produce a corrupted `(Y', C')` such that when the TEE application unmasks it with its secret scalar `a`, it obtains a group element that is not `msk · H(pk, app_id)`. The BLS verification step (`verify_signature`) will fail, rendering the derived key unusable — a **corruption of CKD output** causing honest parties to accept an unusable cryptographic output. Alternatively, the coordinator can abort after collecting all shares, causing **permanent denial of CKD** for the requesting application. Both outcomes fall within the allowed High impact scope: *"Corruption of … CKD outputs so honest parties accept … unusable cryptographic outputs"* and *"Permanent denial of … CKD for honest parties."*

### Likelihood Explanation
The coordinator is a single participant chosen at protocol invocation time. The entire CKD session's correctness and liveness depend on that one node behaving honestly. There is no cryptographic mechanism — no commitment, no proof, no broadcast — that would allow any other participant to detect or prevent coordinator misbehavior. Any participant who is assigned the coordinator role, or any attacker who compromises that node, can silently corrupt or block every CKD request routed through it. The CKD documentation itself acknowledges: *"Liveness and correctness depend on this assumption"* (that all nodes are honest), confirming the single-point-of-failure design.

### Recommendation
**Short term:** Document explicitly that the coordinator is a fully trusted single point of failure for CKD correctness and liveness, and that coordinator compromise breaks all CKD sessions it handles.

**Long term:** Redesign the aggregation step so that the coordinator's output is publicly verifiable. Options include:
- Having each participant broadcast its `(λ_i Y_i, λ_i C_i)` to all other participants (all-to-all), so every node can independently compute and verify `(Y, C)`.
- Having the coordinator broadcast the aggregated `(Y, C)` and each participant verify it against the public contributions they received.
- Using a verifiable secret sharing or commitment scheme so participants can check that their contribution was included correctly.

### Proof of Concept
1. Honest participants P1, P2, P3 run `ckd(...)` with P1 as coordinator.
2. P2 and P3 each compute `(λ_i Y_i, λ_i C_i)` and send them privately to P1 via `chan.send_private(waitpoint, coordinator, &(norm_big_y, norm_big_c))`.
3. P1 (malicious coordinator) receives both contributions but, instead of aggregating them, calls `CKDOutput::new(ElementG1::identity(), ElementG1::identity())` — returning the identity point for both `Y` and `C`.
4. P2 and P3 return `None` and have no visibility into what P1 returned.
5. The TEE application receives `(Y=0, C=0)`, computes `sig = C − a·Y = 0`, which fails `verify_signature`, permanently denying the application its confidential key.
6. No honest participant can detect or prove that P1 deviated, because all contributions were sent privately and no participant holds a verifiable record of what the coordinator was supposed to aggregate. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

async fn do_ckd_coordinator(
    mut chan: SharedChannel,
    participants: ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<CKDOutputOption, ProtocolError> {
    let (mut norm_big_y, mut norm_big_c) =
        compute_signature_share(&participants, me, key_pair, app_id, app_pk, rng)?;

    // Receive everyone's inputs and add them together
    let waitpoint = chan.next_waitpoint();

    for (_, participant_output) in
        recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
    {
        norm_big_y += participant_output.big_y();
        norm_big_c += participant_output.big_c();
    }
    let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
    Ok(Some(ckd_output))
}

/// Depending on whether the current participant is a coordinator or not,
/// runs the ckd protocol as either a participant or a coordinator.
#[allow(clippy::too_many_arguments)]
async fn run_ckd_protocol(
    chan: SharedChannel,
    coordinator: Participant,
    me: Participant,
    participants: ParticipantList,
    key_pair: KeygenOutput,
    app_id: AppId,
    app_pk: PublicKey,
    mut rng: impl CryptoRngCore,
) -> Result<CKDOutputOption, ProtocolError> {
    if me == coordinator {
        do_ckd_coordinator(chan, participants, me, &key_pair, &app_id, app_pk, &mut rng).await
    } else {
        do_ckd_participant(
            chan,
            &participants,
            coordinator,
            me,
            &key_pair,
            &app_id,
            app_pk,
            &mut rng,
        )
    }
}

async fn do_sign_coordinator(
    mut chan: SharedChannel,
    participants: ParticipantList,
    me: Participant,
    public_key: AffinePoint,
    presignature: RerandomizedPresignOutput,
    msg_hash: Scalar,
) -> Result<SignatureOption, ProtocolError> {
    let mut s = compute_signature_share(&presignature, msg_hash, &participants, me)?.0;
    let wait_round = chan.next_waitpoint();

    for (_, s_i) in
        recv_from_others::<SerializableScalar<C>>(&chan, wait_round, &participants, me).await?
    {
        // Sum the linearized shares
        s += s_i.0;
    }

    // raise error if s is zero
    if s.is_zero().into() {
        return Err(ProtocolError::AssertionFailed(
            "signature part s cannot be zero".to_string(),
        ));
    }
    // Normalize s
    s.conditional_assign(&(-s), s.is_high());

    let sig = Signature {
        big_r: presignature.big_r,
        s,
    };

    if !sig.verify(&public_key, &msg_hash) {
        return Err(ProtocolError::AssertionFailed(
            "signature failed to verify".to_string(),
        ));
    }

    Ok(Some(sig))
}

async fn do_sign_coordinator_v1(
    mut chan: SharedChannel,
    participants: ParticipantList,
    threshold: ReconstructionLowerBound,
    me: Participant,
    keygen_output: KeygenOutput,
    message: Vec<u8>,
    rng: &mut impl CryptoRngCore,
) -> Result<SignatureOption, ProtocolError> {
    // --- Round 1.
    // * Wait for other parties' commitments.

    let mut commitments_map: BTreeMap<frost_ed25519::Identifier, round1::SigningCommitments> =
        BTreeMap::new();

    // signing share is the private_share
    let signing_share = keygen_output.private_share;

    // Step 1.1 (and implicitely 1.2)
    let (nonces, commitments) = round1::commit(&signing_share, rng);
    let nonces = Zeroizing::new(nonces);
    commitments_map.insert(me.to_identifier()?, commitments);

    // Step 1.3
    let commit_waitpoint = chan.next_waitpoint();

    // Step 1.4
    for (from, commitment) in recv_from_others(&chan, commit_waitpoint, &participants, me).await? {
        commitments_map.insert(from.to_identifier()?, commitment);
    }

    let signing_package = frost_ed25519::SigningPackage::new(commitments_map, message.as_slice());

    let mut signature_shares: BTreeMap<frost_ed25519::Identifier, round2::SignatureShare> =
        BTreeMap::new();

    // Step 1.5
    let r2_wait_point = chan.next_waitpoint();
    chan.send_many(r2_wait_point, &signing_package)?;

    // --- Round 2
    // * Wait for each other's signature share
    // Step 2.3 (2.1 and 2.2 are implicit)
    let vk_package = keygen_output.public_key;
    let key_package = construct_key_package(threshold, me, signing_share, &vk_package)?;
    let key_package = Zeroizing::new(key_package);
    let signature_share = round2::sign(&signing_package, &nonces, &key_package)
        .map_err(|e| ProtocolError::AssertionFailed(e.to_string()))?;

    // Step 2.5 (2.4 is implicit)
    signature_shares.insert(me.to_identifier()?, signature_share);
    for (from, signature_share) in recv_from_others(&chan, r2_wait_point, &participants, me).await?
    {
        signature_shares.insert(from.to_identifier()?, signature_share);
    }

    // --- Signature aggregation.
    // * Converted collected signature shares into the signature.
    // * Signature is verified internally during `aggregate()` call.

    // Step 2.6 and 2.7
    // We supply empty map as `verifying_shares` because we have disabled "cheater-detection" feature flag.
    // Feature "cheater-detection" only points to a malicious participant, if there's such.
    // It doesn't bring any additional guarantees.
    let public_key_package = PublicKeyPackage::new(BTreeMap::new(), vk_package);
    let signature = aggregate(&signing_package, &signature_shares, &public_key_package)
        .map_err(|e| ProtocolError::AssertionFailed(e.to_string()))?;

    Ok(Some(signature))
}
