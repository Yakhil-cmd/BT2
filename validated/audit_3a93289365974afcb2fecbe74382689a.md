### Title
Missing `is_static` Guard for Non-Zero Value in `CALLCODE` Instruction — (`evm_interpreter/src/instructions/host.rs`)

---

### Summary

The `call_impl` function in the EVM interpreter enforces the EIP-214 static-context value-transfer prohibition for `CALL` but omits the equivalent guard for `CALLCODE`. As a result, `CALLCODE` with a non-zero value executes without reverting inside a static call frame, violating the EVM invariant that no state changes (including ETH transfers) may occur in a static context.

---

### Finding Description

In `call_impl`, the `CallScheme::Call` branch explicitly checks whether the current frame is static before allowing a non-zero value:

```rust
CallScheme::Call => {
    let value = self.stack.pop_1()?;
    if self.is_static && *value != U256::ZERO {
        return Err(EvmError::CallNotAllowedInsideStatic.into());
    }
    *value
}
```

The `CallScheme::CallCode` branch pops the value from the stack with no analogous guard:

```rust
CallScheme::CallCode => {
    let value = self.stack.pop_1()?;
    *value   // ← no is_static check
}
```

After both branches, the value is forwarded as `nominal_token_value` in the `EVMCallRequest`. The downstream `check_depth_and_balance` helper then checks whether the caller has sufficient balance for the transfer — but only if the call is not a delegate-style call. Because `EVMCallcodeStatic` is not a pure delegate call, the balance check runs and, if the caller is funded, the ETH transfer proceeds. The static-context prohibition is never enforced for `CALLCODE`.

The `is_static` flag is correctly propagated to the child frame modifier (`EVMCallcodeStatic`) and to the `is_static` field of the new interpreter frame, so all *subsequent* state-changing opcodes inside the callee (SSTORE, LOG, etc.) are blocked. The problem is specifically that the *value transfer itself* — which is a state change — is not blocked at the `CALLCODE` dispatch site.

---

### Impact Explanation

`CALLCODE` with a non-zero value in a static context transfers ETH from the calling contract to the callee. This is a direct state mutation (balance change) that EIP-214 explicitly forbids in static frames. The consequence is an **EVM semantic mismatch**: a transaction that must revert on Ethereum mainnet (and in any compliant EVM) silently succeeds in ZKsync OS, producing a divergent post-state. This can be exploited to drain ETH from a contract that is only ever called via `STATICCALL` by a trusted caller, because the static invariant is assumed to prevent balance changes.

---

### Likelihood Explanation

`CALLCODE` is a legacy opcode still present in the EVM instruction set. Any EVM contract that uses `CALLCODE` with a non-zero value and is reachable via a `STATICCALL` path (directly or through a call chain) triggers this divergence. An attacker can craft a contract that deliberately uses `CALLCODE` with a non-zero value inside a `STATICCALL` to transfer ETH in a context where the caller assumes no state change is possible.

---

### Recommendation

Add the same static-context guard to the `CallScheme::CallCode` branch that already exists for `CallScheme::Call`:

```rust
CallScheme::CallCode => {
    let value = self.stack.pop_1()?;
    if self.is_static && *value != U256::ZERO {
        return Err(EvmError::CallNotAllowedInsideStatic.into());
    }
    *value
}
```

---

### Proof of Concept

1. Deploy contract `Victim` with 1 ETH balance and a function that executes `CALLCODE` targeting `Attacker` with `value = 1`.
2. Deploy contract `Proxy` that calls `Victim` via `STATICCALL`.
3. Call `Proxy`. On Ethereum mainnet the `STATICCALL` → `CALLCODE(value=1)` path reverts with `StateChangeDuringStaticCall`. In ZKsync OS the call succeeds and 1 ETH is transferred from `Victim` to `Attacker`, violating the static invariant.
