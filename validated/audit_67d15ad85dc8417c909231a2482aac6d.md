### Title
Gateway stateful validator enforces minimum gas price only for L2 gas, not for L1 gas or L1 data gas, admitting transactions that will always fail blockifier pre-validation — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The gateway's stateful resource-bounds check validates the `l2_gas.max_price_per_unit` against the previous block's L2 gas price threshold, but explicitly skips the same check for `l1_gas.max_price_per_unit` and `l1_data_gas.max_price_per_unit`. The blockifier's `check_fee_bounds` enforces all three prices against the actual block gas prices. Transactions with a sufficient L2 gas price but a zero or sub-threshold L1 gas or L1 data gas price therefore pass every gateway admission check, enter the mempool, and then deterministically fail blockifier pre-validation — a direct analog to the external report's pattern of validating only the "actively altered" field while ignoring the others that affect the same invariant.

### Finding Description

`StatefulTransactionValidator::validate_tx_l2_gas_price_within_threshold` is the sole price-floor check in the gateway stateful path. It reads the previous block's `strk_gas_prices.l2_gas_price` and rejects the transaction if `tx.l2_gas.max_price_per_unit` is below the configured percentage of that price. The function contains an explicit developer acknowledgement that the other two prices are not covered:

The stateless validator has the same gap: `validate_resource_bounds` checks `resource_bounds.l2_gas.max_price_per_unit` against the static `min_gas_price` floor, but never checks `l1_gas.max_price_per_unit` or `l1_data_gas.max_price_per_unit`. [2](#0-1) 

In contrast, the blockifier's `check_fee_bounds` — executed during `perform_pre_validation_stage` — iterates over **all three** resources and rejects the transaction if any `max_price_per_unit` is below the actual block gas price:

```rust

### Impact Explanation

An attacker submits `AllResources` invoke transactions with:
- `l2_gas.max_price_per_unit` ≥ gateway threshold (passes both stateless and stateful checks)
- `l1_gas.max_price_per_unit = 0` and/or `l1_data_gas.max_price_per_unit = 0`

These transactions clear every gateway admission gate (`validate_resource_bounds`, `validate_tx_l2_gas_price_within_threshold`, nonce checks, mempool `validate_tx`) and are inserted into the mempool. When the batcher pulls them for block building, `check_fee_bounds` immediately rejects them with `InsufficientResourceBounds { MaxGasPriceTooLow }`. The mempool is polluted with transactions that can never be sequenced, displacing legitimate transactions and wasting batcher execution cycles.

Impact: **High — Mempool/gateway admission accepts invalid transactions before sequencing.**

### Likelihood Explanation

The attack requires no privileged access. Any unprivileged user can craft a valid-looking `InvokeV3` transaction with a zero `l1_gas` or `l1_data_gas` price. The gateway's own TODO comment (`// TODO(Arni): Consider running this validation for all gas prices.`) confirms the gap is known but unmitigated. The default production config has `validate_resource_bounds: true` and `min_gas_price_percentage: 100`, so the L2 gas check is active, making the asymmetry between L2 and L1/L1-data checks exploitable in production. [5](#0-4) 

### Recommendation

Extend `validate_tx_l2_gas_price_within_threshold` (or replace it with a new `validate_all_gas_prices_within_threshold`) to fetch `l1_gas_price` and `l1_data_gas_price` from the previous block and apply the same percentage-threshold check to `l1_gas.max_price_per_unit` and `l1_data_gas.max_price_per_unit`. The blockifier's `check_fee_bounds` already demonstrates the correct three-resource iteration pattern to follow.

### Proof of Concept

1. Obtain the current L2 gas price from the previous block (e.g., 100 fri).
2. Craft an `InvokeV3` transaction with:
   ```
   l2_gas.max_price_per_unit = 100   // ≥ threshold, passes gateway
   l1_gas.max_price_per_unit = 0     // not checked by gateway
   l1_data_gas.max_price_per_unit = 0 // not checked by gateway
   ```
3. Submit via the gateway. Both `validate_resource_bounds` (stateless) and `validate_tx_l2_gas_price_within_threshold` (stateful) return `Ok(())`.
4. Transaction enters the mempool.
5. Batcher pulls the transaction; blockifier calls `perform_pre_validation_stage` → `check_fee_bounds`.
6. `check_fee_bounds` finds `l1_gas.max_price_per_unit (0) < actual_l1_gas_price (> 0)` and returns `InsufficientResourceBounds { MaxGasPriceTooLow { resource: L1Gas } }`.
7. Transaction is discarded. Repeat at scale to exhaust mempool capacity. [6](#0-5) [7](#0-6)

        &self,
        tx: &RpcTransaction,
    ) -> StatelessTransactionValidatorResult<()> {

            return Ok(());
        }

        let resource_bounds = *tx.resource_bounds();
