### Title
Gateway `skip_stateful_validations` Bypasses `__validate__` for Invoke Transactions on Deployed Accounts with Nonce Zero — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

`skip_stateful_validations` is intended to skip the `__validate__` entry-point only when a `deploy_account` transaction is pending in the mempool (UX feature for simultaneous deploy+invoke). However, the predicate it uses — `account_tx_in_pool_or_recent_block` — returns `true` for **any** transaction type from the account, not exclusively `deploy_account`. For a deployed account whose on-chain nonce is still `0` (deployed but never transacted), an attacker who observes a legitimate nonce-0 invoke in the mempool can immediately submit a nonce-1 invoke carrying an **invalid or empty signature** and have it admitted to the mempool without any signature check.

---

### Finding Description

`skip_stateful_validations` in `crates/apollo_gateway/src/stateful_transaction_validator.rs` fires when:

```
tx.nonce() == Nonce(Felt::ONE)  &&  account_nonce == Nonce(Felt::ZERO)
```

and `account_tx_in_pool_or_recent_block(sender)` returns `true`. [1](#0-0) 

When the condition is satisfied, `run_validate_entry_point` is called with `skip_validate = true`, which sets `execution_flags.validate = false`, so the blockifier's `StatefulValidator` never invokes `__validate__`: [2](#0-1) 

The comment in the code justifies the skip by saying the account "either has a deploy_account transaction or transactions with future nonces that passed validations." The second branch is the flaw: a nonce-0 invoke that passed `__validate__` proves the *nonce-0 transaction* was signed, not that any future transaction is signed.

`account_tx_in_pool_or_recent_block` is implemented as:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [3](#0-2) 

It returns `true` for **any** transaction type (invoke, declare, deploy_account) in the pool. There is no filter for `deploy_account` only.

The stateless validator only checks signature **length**, not cryptographic validity: [4](#0-3) 

The stateful nonce check (`validate_nonce`) passes for `account_nonce=0, tx_nonce=1`: [5](#0-4) 

The mempool's `validate_tx` only checks nonce ordering, not signature validity: [6](#0-5) 

**Concrete attack path:**

1. Account `X` is deployed on-chain with nonce `0` (class hash ≠ 0, but no executed transactions).
2. A legitimate user submits `invoke(nonce=0)` for `X`; it passes `__validate__` and enters the mempool.
3. Attacker submits `invoke(nonce=1, signature=[random_felt; max_signature_length])` for `X`.
4. Stateless check: signature length ≤ `max_signature_length` → passes.
5. Stateful nonce check: `0 ≤ 1 ≤ max_allowed_nonce_gap` → passes.
6. `validate_by_mempool`: `tx_nonce(1) ≥ account_nonce(0)` → passes.
7. `skip_stateful_validations`: `tx.nonce()==1 && account_nonce==0 && account_tx_in_pool==true` → returns `true`.
8. `run_validate_entry_point` called with `validate=false` → `__validate__` **never called**.
9. Attacker's unsigned/invalidly-signed nonce-1 invoke is admitted to the mempool.

---

### Impact Explanation

An unprivileged attacker can inject a transaction with an arbitrary (invalid) signature into the mempool for any deployed account whose on-chain nonce is `0` and that has a pending nonce-0 invoke. The gateway's admission invariant — "only accept transactions whose account contract approves via `__validate__`" — is broken. The transaction will revert during batcher execution (when `validate=true` is set), but it occupies a mempool slot, can displace legitimate transactions via fee-escalation logic, and constitutes a confirmed admission of an unauthorized transaction.

Impact: **High — Mempool/gateway admission accepts an invalid transaction (wrong signer) before sequencing.**

---

### Likelihood Explanation

Deployed accounts with nonce `0` are common (accounts that received funds but have not yet transacted). The mempool is observable by all nodes. An attacker needs only to watch for a nonce-0 invoke from such an account and race a nonce-1 invoke with a garbage signature. No privileged access is required.

---

### Recommendation

Change `skip_stateful_validations` to verify that the pending transaction for the account is specifically a `deploy_account` transaction, not just any transaction. One approach: query the mempool for the transaction type at the target address, or add a dedicated `has_pending_deploy_account(address)` API to the mempool. Alternatively, restrict the skip to the case where `tx_pool.contains_account` returns `true` **and** the lowest-nonce transaction for that address is a `DeployAccount` with nonce `0`.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
