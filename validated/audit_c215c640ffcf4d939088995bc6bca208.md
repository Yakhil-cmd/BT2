### Title
Signature Validation Bypass via Pseudo-State Check in `skip_stateful_validations` — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

`skip_stateful_validations` decides whether to skip the `__validate__` entry-point (and therefore signature verification) for an Invoke transaction by checking a pseudo-state — "does this address appear anywhere in the mempool pool or committed history?" — rather than verifying that a `deploy_account` transaction specifically exists for that address. An unprivileged attacker who observes that a freshly-deployed account (on-chain nonce = 0) already has any transaction in the pool can inject an Invoke with nonce = 1 carrying an arbitrary/invalid signature, and the gateway will accept it without ever calling `__validate__`.

### Finding Description

`skip_stateful_validations` is the gateway-side UX shortcut that allows a user to submit `deploy_account(nonce=0)` + `invoke(nonce=1)` simultaneously. The function returns `true` (skip `__validate__`) when three conditions hold:

1. The transaction is an `Invoke`.
2. `tx.nonce() == Nonce(Felt::ONE)`.
3. `account_nonce == Nonce(Felt::ZERO)` (account not yet deployed in committed state).
4. `mempool_client.account_tx_in_pool_or_recent_block(sender_address)` returns `true`. [1](#0-0) 

Condition 4 is implemented as:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [2](#0-1) 

`MempoolState::contains_account` returns `true` if the address appears in either the `staged` or `committed` maps — it does **not** verify that the transaction in the pool is a `deploy_account`: [3](#0-2) 

The code comment claims this is sufficient because "it means that either it has a deploy_account transaction or transactions with future nonces that passed validations." The second branch of that disjunction is the flaw: a freshly-deployed account (on-chain nonce = 0) can have a valid `invoke(nonce=0)` in the pool (submitted by the legitimate owner, who passed `__validate__`). Once that invoke is in the pool, `account_tx_in_pool_or_recent_block` returns `true` for that address, and any subsequent `invoke(nonce=1)` — regardless of who submits it or whether its signature is valid — will have `__validate__` skipped.

When `skip_validate = true` is returned, `run_validate_entry_point` sets `execution_flags.validate = false`: [4](#0-3) 

And `StatefulValidator::perform_validations` returns `Ok(())` immediately without calling `__validate__`: [5](#0-4) 

The `validate_by_mempool` call that precedes `skip_stateful_validations` only checks nonce ordering — it does not verify signatures: [6](#0-5) 

### Impact Explanation

An attacker can inject an `invoke(nonce=1)` transaction carrying an arbitrary or invalid signature into the mempool for any account that satisfies:
- on-chain nonce = 0 (freshly deployed), and
- any transaction with nonce = 0 already present in the pool (submitted by the legitimate owner).

The gateway accepts the transaction without signature verification. The transaction will fail during block execution (when `__validate__` is run by the blockifier with `validate = true`), but it occupies the nonce-1 slot in the mempool. The legitimate owner must perform fee escalation to displace it. This constitutes **gateway admission accepting an invalid transaction** (wrong signer/signature) before sequencing.

Matching impact: **High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

### Likelihood Explanation

The precondition (account with nonce=0 in state that has a pending nonce=0 transaction) is observable from the public mempool and arises naturally whenever any user deploys a new account and immediately submits a transaction. The attack requires no privileged access and no special resources beyond submitting a single RPC transaction.

### Recommendation

Replace the pseudo-state check with a type-specific check. `account_tx_in_pool_or_recent_block` should be replaced (or supplemented) with a query that confirms a `deploy_account` transaction specifically exists for the address, e.g., `deploy_account_tx_in_pool(sender_address)`. Alternatively, the skip logic should only activate when the mempool can positively confirm a `deploy_account` is pending, not merely that the address appears anywhere in the pool.

### Proof of Concept

```
1. Deploy account A (on-chain nonce = 0 after deployment).
2. Legitimate owner submits invoke_A(nonce=0, valid_sig) → passes __validate__ → enters pool.
   Now: account_tx_in_pool_or_recent_block(A) == true.
3. Attacker submits invoke_A(nonce=1, sig=0xdeadbeef) to the gateway.
4. Gateway stateful path:
   a. validate_nonce: 0 <= 1 <= max_gap → OK
   b. validate_by_mempool: nonce 1 >= 0 → OK (no sig check)
   c. skip_stateful_validations:
      - tx is Invoke ✓
      - tx.nonce() == 1 ✓
      - account_nonce == 0 ✓
      - account_tx_in_pool_or_recent_block(A) == true ✓
      → returns true (skip __validate__)
   d. run_validate_entry_point(skip_validate=true):
      execution_flags.validate = false → __validate__ NOT called → Ok(())
5. Attacker's transaction with invalid signature is accepted into the mempool.
6. During block execution, __validate__ is called and fails → transaction reverts,
   but the nonce-1 slot was occupied, forcing the legitimate owner to fee-escalate.
```

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
