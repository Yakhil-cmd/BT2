### Title
Malicious CKD Participant Corrupts Derivation Output by Substituting Arbitrary `public_key` in `key_pair` — (`src/confidential_key_derivation/protocol.rs`)

---

### Summary

The CKD protocol's `compute_signature_share` function uses each participant's locally-supplied `key_pair.public_key` (the master public key) to compute the hash `H(pk ‖ app_id)` that anchors the derivation. No round of the protocol broadcasts or cross-validates this value among participants. A single malicious participant can silently substitute an arbitrary `wrong_pk` in their `key_pair`, causing their contribution to the aggregated `big_c` to be computed over a different hash point. The coordinator sums all contributions and returns a `CKDOutput` that is cryptographically inconsistent with the intended confidential key, with no error or attribution.

---

### Finding Description

**Root cause — `compute_signature_share` in `src/confidential_key_derivation/protocol.rs` lines 148–181:**

    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,

**Analogy to the external report:** In the DODO bug, `params.fromToken` (from the user-controlled `message`) was never validated against the deposited `zrc20`, so the contract approved and swapped its own holdings. Here, `key_pair.public_key` (participant-controlled local input) is never validated against the shared master public key, so a malicious participant's contribution is computed over a different hash point and silently folded into the aggregate.

**Attack mechanics:**

Let `correct_pk` be the master public key all honest participants hold from DKG, and let `msk` be the master secret key. The expected confidential key is:

```
CK = msk · H(correct_pk ‖ app_id)
```

A malicious participant `m` supplies `key_pair = { private_share: x_m, public_key: wrong_pk }`. Their contribution becomes:

```
norm_big_c_m = λ_m · (x_m · H(wrong_pk ‖ app_id) + app_pk · y_m)
```

After the coordinator aggregates and the caller unmasks with `app_sk`:

```
result = big_c − app_sk · big_y
       = (msk − λ_m · x_m) · H(correct_pk ‖ app_id)
         + λ_m · x_m · H(wrong_pk ‖ app_id)
       ≠ msk · H(correct_pk ‖ app_id)
```

The coordinator returns this corrupted `CKDOutput` as a successful result. No error is raised, no participant is identified, and the caller has no way to detect the corruption without independently recomputing the expected key.
