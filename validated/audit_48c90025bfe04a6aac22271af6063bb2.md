### Title
Unchecked Return Value of Low-Level Exit Precompile Calls Causes Permanent Token Freeze - (File: `etc/eth-contracts/contracts/EvmErc20.sol`)

---

### Summary

`EvmErc20.sol` and `EvmErc20V2.sol` implement `withdrawToNear` and `withdrawToEthereum` by first burning the caller's ERC-20 tokens and then making a low-level `call` to a hardcoded exit precompile address via inline assembly. The return value (`res`) is captured but never checked. If the precompile call fails for any reason, the burn is not reverted, the ERC-20 tokens are permanently destroyed, and no corresponding NEP-141 `ft_transfer` promise is ever created — permanently freezing the bridged NEP-141 tokens inside the Aurora contract.

---

### Finding Description

In `etc/eth-contracts/contracts/EvmErc20.sol`, both exit functions follow this pattern:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // ← tokens destroyed first

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        // res is NEVER checked — no require(res != 0)
    }
}
``` [1](#0-0) 

The same pattern exists in `withdrawToEthereum`: [2](#0-1) 

And is identically reproduced in `EvmErc20V2.sol`: [3](#0-2) 

The `ExitToNear` precompile (`engine-precompiles/src/native.rs`) returns `Err(ExitError)` — causing the `call` opcode to return `0` — in multiple reachable conditions:

- `validate_input_size` fails when `recipient.length` exceeds `MAX_INPUT_SIZE - 33 = 991` bytes
- `ExitToNearParams::try_from(input)` fails on malformed input
- `get_nep141_from_erc20` returns `ERR_TARGET_TOKEN_NOT_FOUND`
- `get_eth_connector_contract_account` returns `ERR_KEY_NOT_FOUND`
- Out-of-gas during precompile execution [4](#0-3) 

When any of these conditions occur, the EVM `call` returns `0` into `res`, but since `res` is never tested, the Solidity function returns normally. The `_burn` is not reverted. The NEP-141 promise is never created.

The `error_refund` feature only handles the case where the NEAR `ft_transfer` promise fails *after* the precompile has already succeeded and emitted a promise log. It provides no protection when the precompile itself fails before any promise is created: [5](#0-4) 

---

### Impact Explanation

- The user's ERC-20 tokens are permanently burned (removed from Aurora's EVM state).
- No `ft_transfer` or `ft_transfer_call` NEAR promise is ever scheduled.
- The corresponding NEP-141 tokens remain locked in the Aurora contract with no recovery path.
- This constitutes **permanent freezing of bridged user funds**.

Impact: **Critical — Permanent freezing of funds.**

---

### Likelihood Explanation

`EvmErc20` is the standard contract deployed for every NEP-141 token bridged to Aurora. Any token holder can call `withdrawToNear` directly. The failure condition is reachable without any privileged access:

- A user supplying a `recipient` byte string longer than 991 bytes triggers `validate_input_size` failure in the precompile.
- A user supplying a recipient that is not a valid NEAR account ID causes `get_nep141_from_erc20` or downstream parsing to fail.
- Out-of-gas scenarios are always possible under congestion.

No admin compromise, governance capture, or external oracle is required. The attacker-controlled entry path is: call `withdrawToNear(longOrInvalidRecipient, amount)` on any deployed `EvmErc20` instance.

Likelihood: **Medium** (requires a specific input condition, but the function is fully open to any token holder and the failure conditions are easily triggered accidentally or deliberately).

---

### Recommendation

Add a `require` on the return value of the low-level `call` in both `withdrawToNear` and `withdrawToEthereum`, so that if the exit precompile fails the entire transaction reverts (including the `_burn`):

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

Apply the same fix to `EvmErc20V2.sol` and to `Tester.sol` / `TesterV2.sol` where the same unchecked pattern appears.

---

### Proof of Concept

1. Deploy `EvmErc20` for a registered NEP-141 token on Aurora.
2. Mint `1000` tokens to address `A`.
3. From address `A`, call `withdrawToNear(recipient, 1000)` where `recipient` is a byte string of length 992 (exceeding `MAX_INPUT_SIZE - 33 = 991`).
4. Inside the precompile, `validate_input_size` returns `Err(ExitError::Other("ERR_INVALID_INPUT"))`.
5. The `call` opcode returns `0` into `res`.
6. `res` is never checked; `withdrawToNear` returns normally with `TransactionStatus::Succeed`.
7. Address `A`'s ERC-20 balance is now `0` (burned).
8. No NEAR `ft_transfer` promise was created; the NEP-141 tokens remain locked in the Aurora contract.
9. There is no refund path: `error_refund` only triggers on a failed NEAR promise callback, not on a failed precompile call. [6](#0-5) [1](#0-0)
