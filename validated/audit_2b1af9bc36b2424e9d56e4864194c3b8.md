### Title
Missing Lower-Bound Constraint on `threshold` in FROST Signing Entry Points Allows Threshold=1 to Corrupt Sign Outputs — (`src/frost/mod.rs`)

### Summary

`assert_sign_inputs` and `presign` in `src/frost/mod.rs` validate that `threshold ≤ participants.len()` but omit the lower-bound check `threshold ≥ 2` that `assert_key_invariants` in `src/dkg.rs` enforces for key generation. A caller (malicious coordinator or misconfigured participant) can supply `threshold = 1` to the FROST signing entry points; the validation passes, the protocol proceeds with Lagrange coefficients computed for a degree-0 polynomial, and the resulting signature share combination is cryptographically inconsistent with the degree-≥1 polynomial used during DKG, producing an unusable output that honest parties accept as a completed protocol run.

### Finding Description

**Root cause — missing lower-bound check in signing validation:**

`assert_key_invariants` (`src/dkg.rs`, line 580) enforces:
```rust
if threshold < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
}
``` [1](#0-0) 

`assert_sign_inputs` (`src/frost/mod.rs`, lines 144–150) only checks the upper bound:
```rust
if threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
``` [2](#0-1) 

The same gap exists in `presign` (`src/frost/mod.rs`, lines 71–77):
```rust
if args.threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
``` [3](#0-2) 

**Exploit path:**

1. DKG runs correctly with `threshold = 2`, producing key shares on a degree-1 polynomial.
2. A malicious coordinator (or misconfigured caller) invokes `assert_sign_inputs` or `presign` with `threshold = 1` and `participants.len() = 2`. Both checks pass: `1 ≤ 2` (upper bound) and `2 ≥ 2` (participant count).
3. The FROST signing protocol proceeds. Lagrange coefficients are computed for a single-point interpolation (degree-0), which is inconsistent with the degree-1 polynomial used during DKG.
4. The coordinator combines signature shares using wrong Lagrange weights; the resulting `s` value does not satisfy the ECDSA/EdDSA verification equation for the master public key.
5. Honest participants observe a completed protocol run but receive an invalid, unusable signature.

**Cross-parameter constraint that is absent:**

The `PresignArguments` struct carries an independent `threshold` field alongside `keygen_out`, with no validation that the two are consistent:
```rust
pub struct PresignArguments<C: Ciphersuite> {
    pub keygen_out: KeygenOutput<C>,
    pub threshold: ReconstructionLowerBound,
}
``` [4](#0-3) 

There is no check that `args.threshold` matches the threshold used during DKG, nor that it satisfies `≥ 2`.

### Impact Explanation

**High — Corruption of sign outputs so honest parties accept unusable cryptographic outputs.**

When `threshold = 1` is accepted by `assert_sign_inputs` or `presign`, the FROST signing protocol runs to completion but produces a signature that fails standard verification against the master public key. Honest participants have no in-protocol signal that the output is invalid; they accept the completed run. Every signing session initiated with this misconfigured threshold produces an unusable signature, constituting a persistent denial of signing capability for the affected session.

### Likelihood Explanation

The `assert_sign_inputs` function is a public, exported validation helper that library callers are expected to invoke before running the signing protocol. Any caller — including a malicious coordinator who controls the session parameters — can supply `threshold = 1`. No privileged access is required; the attacker only needs to be a participant who initiates or configures a signing session.

### Recommendation

Add the same lower-bound guard that `assert_key_invariants` applies, in both `assert_sign_inputs` and `presign`:

```rust
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold: threshold.value(), min: 2 });
}
```

Additionally, consider storing the threshold inside `KeygenOutput` so that `PresignArguments` can validate `args.threshold == args.keygen_out.threshold` at initialization time, eliminating the cross-parameter mismatch entirely.

### Proof of Concept

```
1. Run DKG with participants = [P1, P2, P3], threshold = 2.
   → Key shares on degree-1 polynomial; DKG succeeds.

2. Coordinator calls:
   assert_sign_inputs(&[P1, P2], threshold=1, me=P1, coordinator=P1)
   → threshold.value() (1) is NOT > participants.len() (2): upper-bound check passes.
   → threshold.value() (1) is NOT < 2: lower-bound check is ABSENT.
   → Returns Ok(participants) — no error.

3. Signing protocol runs with threshold=1.
   → Lagrange coefficient for P1 alone = 1 (single-point interpolation).
   → Coordinator combines: s = s_P1 * 1 (only own share, wrong weight).
   → Resulting signature (R, s) fails verify(public_key, msg_hash).

4. Honest parties receive a completed protocol result containing an invalid signature.
``` [5](#0-4) [6](#0-5)

pub fn assert_sign_inputs(
    participants: &[Participant],
    threshold: impl Into<ReconstructionLowerBound>,
    me: Participant,
    coordinator: Participant,
) -> Result<ParticipantList, InitializationError> {
    let threshold = threshold.into();
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // validate threshold
    if threshold.value() > participants.len() {
        return Err(InitializationError::ThresholdTooLarge {
            threshold: threshold.value(),
            max: participants.len(),
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }
    Ok(participants)
}
