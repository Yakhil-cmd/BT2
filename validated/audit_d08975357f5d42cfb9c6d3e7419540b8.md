Audit Report

## Title
Unescaped Memo Bytes Enable Markdown Injection into ICRC-21 GenericDisplayMessage — (`packages/icrc-ledger-types/src/icrc21/responses.rs`)

## Summary
The `add_memo` function in `responses.rs` interpolates attacker-controlled, UTF-8-decoded memo bytes directly into a single-backtick CommonMark code span without escaping backticks or newlines. A memo beginning with a backtick causes the format string to emit a double-backtick sequence (```` `` ````), which CommonMark cannot close with the trailing single backtick, causing the remainder of the memo to be parsed as raw Markdown. This allows any unprivileged caller to inject visually authentic bold fields (e.g., `**Amount:**`, `**To:**`) into the ICRC-21 consent message, undermining the core security invariant that the consent surface faithfully represents the authorized transaction.

## Finding Description
In `add_memo` at line 294 of `packages/icrc-ledger-types/src/icrc21/responses.rs`:

```rust
message.push_str(&format!("\n\n**Memo:**\n`{memo_str}`"));
```

`memo_str` is the raw UTF-8 decode of the caller-supplied memo bytes with no sanitization. If `memo_str` begins with a backtick (e.g., `` `\n\n**Amount:** `999999 ICP ``), the format string emits:

```
**Memo:**
``

**Amount:** `999999 ICP `
```

CommonMark rule: a double-backtick code span opener (```` `` ````) requires a matching ```` `` ```` to close. Since none exists, the opener is emitted as literal characters, and the rest of the memo is parsed as Markdown — rendering `**Amount:**` as bold and `` `999999 ICP ` `` as a code span, visually identical to the real `**Amount:**` field generated by `add_amount` at line 114.

The exploit path is fully direct:
1. Any caller invokes `icrc21_canister_call_consent_message` with a `TransferArg` whose `memo` field contains the crafted bytes.
2. `build_icrc21_consent_info` (lib.rs line 392–395) decodes the arg and passes `GenericMemo::Icrc1Memo(memo.0)` to the builder.
3. `ConsentMessageBuilder::build` (lib.rs line 290–292) calls `message.add_memo(memo)` unconditionally when a memo is present.
4. `add_memo` decodes bytes as UTF-8 (responses.rs line 288–290) and interpolates without escaping (line 294).

No existing guard prevents this: the only branch for non-UTF-8 bytes falls back to hex encoding (line 290), but valid UTF-8 bytes — including backticks and newlines — pass through verbatim.

## Impact Explanation
The `GenericDisplayMessage` is the primary human-readable consent surface for ICRC-21. The ICRC-21 standard specifies that wallets should render it as Markdown. A malicious dApp can inject fake `**Amount:**`, `**To:**`, or `**Fees:**` fields that are visually indistinguishable from the real ones generated by the ledger. This breaks the core security invariant of ICRC-21: that the consent message faithfully and exclusively represents the transaction the user is authorizing. This constitutes a significant ledger security impact with concrete user harm, fitting the High impact class: unauthorized manipulation of the consent surface for ICRC ledger transactions, enabling user deception about the transaction being signed.

## Likelihood Explanation
Any unprivileged caller can submit a `ConsentMessageRequest` with a crafted memo via the public `icrc21_canister_call_consent_message` update endpoint. The ICRC-1 memo field accepts up to 32 bytes, which is sufficient to inject a fake field. Wallets implementing ICRC-21 (Plug, NFID, hardware wallets) are expected to render `GenericDisplayMessage` as Markdown per the spec. The injected content appears after the real fields, so a user reading the full message carefully will see the real values first; however, wallet UIs that paginate, truncate, or scroll the consent message may present the injected content more prominently or exclusively.

## Recommendation
Escape all Markdown control characters in `memo_str` before interpolation. At minimum, replace backticks and newlines:

```rust
let safe_memo = memo_str
    .replace('`', "\\`")
    .replace('\n', " ")
    .replace('\r', " ");
message.push_str(&format!("\n\n**Memo:**\n`{safe_memo}`"));
```

Alternatively, hex-encode all memo bytes unconditionally for `GenericDisplayMessage` (as is already done for non-UTF-8 bytes at line 290), since memo bytes carry no semantic meaning requiring human-readable display.

## Proof of Concept
```rust
// Unit test demonstrating injection
let memo_bytes = b"`\n\n**Amount:** `999999 ICP".to_vec();
let memo = GenericMemo::Icrc1Memo(ByteBuf::from(memo_bytes));
let mut msg = ConsentMessage::GenericDisplayMessage(String::new());
msg.add_intent(Icrc21Function::Transfer, Some("ICP".to_string()));
msg.add_amount(Some(Nat::from(1u64)), 8, &"ICP".to_string()).unwrap();
msg.add_memo(memo);

if let ConsentMessage::GenericDisplayMessage(text) = &msg {
    // The injected bold field appears in the rendered Markdown output
    assert!(
        !text.contains("\n\n**Amount:** `999999 ICP`"),
        "FAIL: injected fake Amount field present in consent message:\n{text}"
    );
}
```

The crafted memo causes the format at line 294 to emit ```` ``\n\n**Amount:** `999999 ICP` ````, which CommonMark renders as a bold `**Amount:**` label followed by a code span `999999 ICP`, visually identical to the real amount field at line 114.
