### Title
Silo Whitelist Access Control Bypassed via `call` and `deploy_code` Entrypoints — (File: engine/src/contract_methods/evm_transactions.rs)

### Summary

Aurora Engine's silo mode enforces a whitelist that restricts which NEAR accounts and EVM addresses may submit transactions or deploy code. This whitelist check is applied only inside `submit_with_alt_modexp`, which is reached through the `submit` entrypoint. The `call` and `deploy_code` entrypoints reach the EVM directly without invoking `assert_access`, so any unprivileged NEAR account can execute arbitrary EVM calls and deploy EVM contracts even when all four silo whitelists are enabled and the caller is absent from every list.

### Finding Description

**Root cause — missing `assert_access` call in `call` and `deploy_code`**

`submit_with_alt_modexp` (reached via `submit`) calls `assert_access` before any EVM execution:

```
engine/src/engine.rs  line 1052
assert_access(&io, env, &transaction)?;
```

`assert_access` delegates to `silo::is_allow_submit` / `silo::is_allow_deploy`:

```
engine/src/engine.rs  lines 1761-1765
let allowed = if transaction.to.is_some() {
    silo::is_allow_submit(io, &env.predecessor_account_id(), &transaction.address)
} else {
    silo::is_allow_deploy(io, &env.predecessor_account_id(), &transaction.address)
};
```

`is_allow_submit` and `is_allow_deploy` return `false` (deny) when the relevant whitelists are enabled and the caller is absent:

```
engine/src/contract_methods/silo/mod.rs  lines 136-163
pub fn is_allow_submit<I: IO + Copy>(io: &I, account: &AccountId, address: &Address) -> bool {
    is_address_allowed(io, address) && is_account_allowed(io, account)
}
fn is_account_allowed<I: IO + Copy>(io: &I, account: &AccountId) -> bool {
    let list = Whitelist::init(io, WhitelistKind::Account);
    !list.is_enabled() || list.is_exist(account)
}
```

By contrast, `call` and `deploy_code` in `evm_transactions.rs` only check `require_running` and then immediately enter the EVM:

```
engine/src/contract_methods/evm_transactions.rs  lines 21-43  (deploy_code)
engine/src/contract_methods/evm_transactions.rs  lines 46-71  (call)
```

Neither function calls `assert_access`, `is_allow_submit`, or `is_allow_deploy` at any point.

**Attacker-controlled entry path**

1. Operator enables silo mode: calls `set_whitelists_statuses` with all four kinds active (`Admin`, `EvmAdmin`, `Account`, `Address`).
2. Attacker's NEAR account and EVM address are absent from every whitelist.
3. Attacker calls `submit` → rejected with `EngineErrorKind::NotAllowed`.
4. Attacker calls `call` (or `deploy_code`) with identical intent → **succeeds**; the EVM executes the call using `predecessor_address(&predecessor_account_id)` as the EVM sender.

The `call` entrypoint is a public NEAR method; no privileged key is required.

### Impact Explanation

In a silo deployment the whitelist is the sole mechanism preventing unauthorized parties from interacting with EVM contracts that hold user funds. Bypassing it via `call` allows an attacker to:

- Invoke any EVM contract as an arbitrary EVM address derived from their NEAR account, including contracts that gate withdrawals or transfers on `msg.sender`.
- Deploy arbitrary EVM bytecode via `deploy_code`, enabling malicious contract placement in the silo's address space.

Concrete impact: **direct theft of user funds** held in EVM contracts whose access control relies on the silo whitelist being the enforcement boundary, or **temporary freezing of funds** if the unauthorized call corrupts contract state. This maps to Critical / High in the allowed scope.

### Likelihood Explanation

Any NEAR account — no keys, no deposits, no prior relationship with the silo — can call `call` or `deploy_code` on the Aurora contract. The bypass requires no special preconditions beyond knowing the contract's account ID, which is public. Likelihood is **High**.

### Recommendation

Add the same `assert_access` guard used in `submit_with_alt_modexp` to both `call` and `deploy_code` in `engine/src/contract_methods/evm_transactions.rs`. Specifically, derive the EVM address from `predecessor_account_id` (as already done for the engine sender) and pass it together with the NEAR account ID to `silo::is_allow_submit` / `silo::is_allow_deploy` before entering the EVM. If the check fails, return `EngineErrorKind::NotAllowed`.

```rust
fn assert_access<I: IO + Copy, E: Env>(
    io: &I,
    env: &E,
    transaction: &NormalizedEthTransaction,
) -> Result<(), EngineError> {
    let allowed = if transaction.to.is_some() {
        silo::is_allow_submit(io, &env.predecessor_account_id(), &transaction.address)
    } else {
        silo::is_allow_deploy(io, &env.predecessor_account_id(), &transaction.address)
    };

    if !allowed {
        return Err(EngineError {
            kind: EngineErrorKind::NotAllowed,
            gas_used: 0,
        });
    }

    Ok(())
}
```
