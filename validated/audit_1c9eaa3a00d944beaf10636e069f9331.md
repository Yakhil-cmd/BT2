### Title
Overly Strict Access Control on `claim_rewards` Permanently Freezes Unclaimed Yield When Authorized Addresses Are Inaccessible - (File: src/staking/staking.cairo, src/pool/pool.cairo)

---

### Summary

Both `claim_rewards` implementations in `staking.cairo` and `pool.cairo` gate execution behind an inline access control check that restricts callers to only the staker/pool-member address or their registered reward address. Because rewards are unconditionally transferred to the fixed, on-chain `reward_address` regardless of who triggers the call, this restriction is unnecessary and creates a single point of failure: if both authorized addresses become inaccessible, all accumulated unclaimed yield is permanently frozen with no recovery path.

---

### Finding Description

**`staking.cairo` — `claim_rewards`** [1](#0-0) 

The function reads `reward_address` from storage and asserts the caller is either `staker_address` or `reward_address`: [2](#0-1) 

Regardless of who calls it, the transfer destination is always the stored `reward_address`: [3](#0-2) 

**`pool.cairo` — `claim_rewards`**

The pool contract has an identical pattern: [4](#0-3) 

And the transfer destination is always the stored `reward_address`: [5](#0-4) 

In both cases the access control check does not protect any funds from being misdirected — the destination is a fixed, pre-registered address stored in contract state. The only effect of the check is to prevent a third party from *triggering* the delivery of rewards that would go to the correct address anyway.

---

### Impact Explanation

**Severity: High — Permanent freezing of unclaimed yield.**

If both the staker address and the registered `reward_address` become inaccessible simultaneously (e.g., a smart-contract wallet is bricked, a multisig loses quorum, or private keys are lost), all `unclaimed_rewards_own` accumulated in `staker_info` and all pool-member rewards tracked via the cumulative-rewards trace are permanently locked. There is no admin override, no governance escape hatch, and no alternative entry point that can deliver those rewards. The staker cannot call `change_reward_address` to recover because that function also requires the staker address as caller: [6](#0-5) 

---

### Likelihood Explanation

**Medium.** Smart-contract wallets (account abstraction) are the norm on Starknet. A wallet upgrade gone wrong, a multisig that loses quorum, or a delegated account that is decommissioned can render both the staker address and the reward address permanently unusable. The staker address and reward address are independent addresses, but both must be inaccessible simultaneously for the freeze to occur. Given the prevalence of AA wallets and the long-lived nature of staking positions, this scenario is realistic over a protocol lifetime.

---

### Recommendation

Remove the caller restriction from both `claim_rewards` functions. Because the transfer destination is always the pre-registered `reward_address` stored in contract state, any caller can safely trigger the delivery without any risk of fund misdirection. This mirrors the fix applied in the referenced audit (PR 315): authorization was removed so that anyone can trigger fee delivery at any time.

```cairo
// staking.cairo — remove lines 417-421
fn claim_rewards(ref self: ContractState, staker_address: ContractAddress) -> Amount {
    self.general_prerequisites();
    let mut staker_info = self.internal_staker_info(:staker_address);
    // No caller restriction needed — rewards always go to staker_info.reward_address
    let amount = staker_info.unclaimed_rewards_own;
    let token_dispatcher = strk_token_dispatcher();
    self.send_rewards_to_staker(:staker_address, ref :staker_info, :token_dispatcher);
    self.write_staker_info(:staker_address, :staker_info);
    amount
}
```

Apply the same change to `pool.cairo` `claim_rewards`.

---

### Proof of Concept

1. Staker Alice stakes STRK and sets `reward_address` to a smart-contract multisig wallet `M`.
2. Over time, attestations accumulate `unclaimed_rewards_own` in Alice's staker record.
3. Alice's staker EOA private key is lost. Multisig `M` loses quorum (e.g., 2-of-3 signers become unavailable).
4. Any third party attempts to call `staking.claim_rewards(alice_address)` to deliver Alice's rewards to `M`.
5. The call reverts at the access control assert: [2](#0-1) 
6. Alice cannot call `change_reward_address` to redirect rewards because that also requires `get_caller_address() == staker_address`: [7](#0-6) 
7. All accumulated yield is permanently frozen. The identical path applies to pool members via `pool.cairo` `claim_rewards`: [8](#0-7)
