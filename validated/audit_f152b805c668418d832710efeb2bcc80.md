### Title
`vm_resources_to_execution_resources` Returns `UnknownBuiltin` Error for Valid `range_check96`/`add_mod`/`mul_mod` Builtins, Breaking RPC Simulation and Tracing - (`crates/apollo_rpc_execution/src/objects.rs`)

---

### Summary

`vm_resources_to_execution_resources` has an incomplete `match` on `BuiltinName` variants. The `range_check96`, `add_mod`, and `mul_mod` builtins are explicitly left unhandled (commented-out TODOs), causing the function to return `Err(ExecutionError::UnknownBuiltin)` whenever a successfully-executed transaction used any of those builtins. Any unprivileged user can trigger this by invoking a deployed contract that uses these builtins and then calling `starknet_simulateTransactions` or `starknet_traceTransaction`.

---

### Finding Description

In `vm_resources_to_execution_resources`, the match arm for `BuiltinName` variants covers only 8 builtins. The three newer builtins — `range_check96`, `add_mod`, `mul_mod` — are explicitly noted as unhandled TODOs and fall through to the catch-all `_` arm: [1](#0-0) 

```rust
// TODO(DanB): what about the following?
// BuiltinName::range_check96 => todo!(),
// BuiltinName::add_mod => todo!(),
// BuiltinName::mul_mod => todo!(),
_ => {
    return Err(ExecutionError::UnknownBuiltin { builtin_name });
}
```

This function is called from `FunctionInvocation::try_from((CallInfo, GasVector))`, which is the conversion used during trace construction for both `simulate_transactions` and `starknet_traceTransaction`: [2](#0-1) 

Meanwhile, blockifier fully supports these builtins. They appear in the bouncer resource accounting (18 references), versioned constants (9 references), and CASM/native entry point execution:



This means blockifier can successfully execute a transaction that uses `range_check96` (used by `u96`-range arithmetic in Cairo), `add_mod`, or `mul_mod` (used by circuit operations). The `VmExecutionResources` returned by blockifier will contain non-zero counts for these builtins. When the RPC layer then tries to convert that `CallInfo` into a `FunctionInvocation` for the trace response, `vm_resources_to_execution_resources` fails with `UnknownBuiltin`, and the entire simulation/trace call returns an error.

---

### Impact Explanation

The invariant that "any transaction blockifier executed successfully can be traced/simulated via RPC" is broken. `starknet_simulateTransactions` and `starknet_traceTransaction` return `ExecutionError::UnknownBuiltin` instead of a valid trace for any transaction whose execution touched `range_check96`, `add_mod`, or `mul_mod` builtins. This maps to the allowed impact: **"RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value."** [3](#0-2) 

---

### Likelihood Explanation

`range_check96` is used by standard Cairo integer operations (96-bit range checks), and `add_mod`/`mul_mod` are used by Cairo's circuit/modular arithmetic builtins. Any Sierra contract compiled with these operations will trigger the bug. The trigger requires only: (1) deploying or calling an existing contract that uses these builtins, and (2) calling `starknet_simulateTransactions` or `starknet_traceTransaction` — both are standard unprivileged RPC calls.

---

### Recommendation

Add the three missing arms to the `match` in `vm_resources_to_execution_resources`. Map them to their corresponding `Builtin` enum variants (if they exist in the RPC types), or silently skip them (like `output`) if the RPC spec does not expose them:

```rust
BuiltinName::range_check96 => {
    builtin_instance_counter.insert(Builtin::RangeCheck96, count)
}
BuiltinName::add_mod => {
    builtin_instance_counter.insert(Builtin::AddMod, count)
}
BuiltinName::mul_mod => {
    builtin_instance_counter.insert(Builtin::MulMod, count)
}
```

If the RPC `Builtin` enum does not yet have these variants, they should be added, or the builtins should be silently skipped (like `output`) rather than returning a hard error.

```rust
fn vm_resources_to_execution_resources(
    vm_resources: VmExecutionResources,
    GasVector { l1_gas, l1_data_gas, l2_gas }: GasVector,
) -> ExecutionResult<ExecutionResources> {
    let mut builtin_instance_counter = HashMap::new();
    for (builtin_name, count) in vm_resources.builtin_instance_counter {
        if count == 0 {
            continue;
        }
        let count = u64_from_usize(count);
        match builtin_name {
            BuiltinName::output => continue,
            BuiltinName::pedersen => builtin_instance_counter.insert(Builtin::Pedersen, count),
            BuiltinName::range_check => builtin_instance_counter.insert(Builtin::RangeCheck, count),
            BuiltinName::ecdsa => builtin_instance_counter.insert(Builtin::Ecdsa, count),
            BuiltinName::bitwise => builtin_instance_counter.insert(Builtin::Bitwise, count),
            BuiltinName::ec_op => builtin_instance_counter.insert(Builtin::EcOp, count),
            BuiltinName::keccak => builtin_instance_counter.insert(Builtin::Keccak, count),
            BuiltinName::poseidon => builtin_instance_counter.insert(Builtin::Poseidon, count),
            BuiltinName::segment_arena => {
                builtin_instance_counter.insert(Builtin::SegmentArena, count)
            }
            // TODO(DanB): what about the following?
            // BuiltinName::range_check96 => todo!(),
            // BuiltinName::add_mod => todo!(),
            // BuiltinName::mul_mod => todo!(),
            _ => {
                return Err(ExecutionError::UnknownBuiltin { builtin_name });
            }
        };
    }
    Ok(ExecutionResources {
        steps: u64_from_usize(vm_resources.n_steps),
        builtin_instance_counter,
        memory_holes: u64_from_usize(vm_resources.n_memory_holes),
        da_gas_consumed: StarknetApiGasVector { l1_gas, l2_gas, l1_data_gas },
        gas_consumed: StarknetApiGasVector::default(),
    })
}
```
