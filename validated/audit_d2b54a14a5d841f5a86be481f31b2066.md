### Title
Incomplete `REWARD_ADDRESS_IS_TOKEN` Guard in Pool's `change_reward_address` Allows Permanent Freezing of Delegator Yield — (`src/pool/pool.cairo`)

---

### Summary

The `Pool` contract's `change_reward_address` function only validates that the new reward address is not the pool's own staked token. The `Staking` contract performs a broader check against **all** registered tokens. Because pool rewards are always paid in STRK regardless of pool type, a BTC-pool member can set their reward address to the STRK token contract address (which passes the pool's narrow guard), causing all future reward claims to transfer STRK into the STRK token contract itself — permanently locking the yield.

---

### Finding Description

**Staking contract — broad guard (all registered tokens):** [1](#0-0) 

```cairo
fn change_reward_address(ref self: ContractState, reward_address: ContractAddress) {
    self.general_prerequisites();
    assert!(
        !self.does_token_exist(token_address: reward_address),
        "{}",
        GenericError::REWARD_ADDRESS_IS_TOKEN,
    );
```

**Pool contract — narrow guard (only the pool's own token):** [2](#0-1) 

```cairo
fn change_reward_address(ref self: ContractState, reward_address: ContractAddress) {
    assert!(
        self.token_dispatcher.contract_address.read() != reward_address,
        "{}",
        GenericError::REWARD_ADDRESS_IS_TOKEN,
    );
```

`self.token_dispatcher` holds the pool's staked token (BTC for a BTC pool). The check therefore only blocks setting the reward address to the BTC token contract. It does **not** block setting it to the STRK token contract or any other registered token.

**Reward transfer in `claim_rewards` is always in STRK:** [3](#0-2) 

```cairo
let reward_token = IERC20Dispatcher { contract_address: STRK_TOKEN_ADDRESS };
reward_token.checked_transfer(recipient: reward_address, amount: rewards.into());
```

This is consistent with the staking contract's `update_pool_rewards`, which explicitly comments "Rewards are always in STRK": [4](#0-3) 

So for a BTC pool member who sets `reward_address = STRK_TOKEN_ADDRESS`:
1. `change_reward_address` passes — STRK address ≠ BTC token address.
2. `claim_rewards` calls `checked_transfer(recipient: STRK_TOKEN_ADDRESS, amount: rewards)`.
3. STRK tokens are transferred into the STRK ERC-20 contract itself, with no recovery path.

---

### Impact Explanation

**Permanent freezing of unclaimed yield (High).**

Every subsequent `claim_rewards` call for the affected pool member sends STRK into the STRK token contract address. The tokens are irretrievably locked. The pool member's `unclaimed_rewards` is zeroed out after each call, so the accounting shows the rewards as "paid" while they are actually destroyed. This matches the allowed impact: *Permanent freezing of unclaimed yield or unclaimed royalties*.

---

### Likelihood Explanation

Any delegator in a BTC delegation pool can call `change_reward_address` with `STRK_TOKEN_ADDRESS` as the argument. The call succeeds silently — no error is raised. The delegator may do this accidentally (e.g., copy-paste error, confusing the STRK token address with their wallet address) or a malicious actor could socially engineer a victim into doing so. The entry path requires no privileged role: it is reachable by any unprivileged pool member.

---

### Recommendation

Mirror the staking contract's guard in the pool contract. Replace the single-token check with a call back to the staking contract to verify the candidate address is not any registered token:

```cairo
fn change_reward_address(ref self: ContractState, reward_address: ContractAddress) {
    // Check reward_address is not any registered token (STRK or BTC variants).
    let staking_dispatcher = IStakingDispatcher {
        contract_address: self.staking_pool_dispatcher.contract_address.read()
    };
    assert!(
        !staking_dispatcher.does_token_exist(token_address: reward_address),
        "{}",
        GenericError::REWARD_ADDRESS_IS_TOKEN,
    );
    ...
}
```

Alternatively, at minimum, add an explicit check for `STRK_TOKEN_ADDRESS` alongside the existing pool-token check.

---

### Proof of Concept

1. Staker opens a BTC delegation pool via `set_open_for_delegation(btc_token_address)`.
2. Delegator calls `enter_delegation_pool` on the BTC pool.
3. Delegator calls `pool.change_reward_address(STRK_TOKEN_ADDRESS)`.
   - Check: `self.token_dispatcher.contract_address.read()` = BTC address ≠ STRK address → **passes**.
4. Staker attests; `update_rewards` sends STRK rewards to the BTC pool contract.
5. Delegator calls `pool.claim_rewards(pool_member)`.
   - `reward_token.checked_transfer(recipient: STRK_TOKEN_ADDRESS, amount: rewards)` executes.
   - STRK is transferred into the STRK token contract itself.
   - `pool_member_info.unclaimed_rewards` is zeroed — rewards are permanently lost. [5](#0-4) [3](#0-2) [4](#0-3)

        fn change_reward_address(ref self: ContractState, reward_address: ContractAddress) {
            assert!(
                self.token_dispatcher.contract_address.read() != reward_address,
                "{}",
                GenericError::REWARD_ADDRESS_IS_TOKEN,
            );
            let pool_member = get_caller_address();
            let mut pool_member_info = self.internal_pool_member_info(:pool_member);
            let old_address = pool_member_info.reward_address;

            // Update reward_address and commit to storage.
            pool_member_info.reward_address = reward_address;
            self.write_pool_member_info(:pool_member, :pool_member_info);

            // Emit event.
            self
                .emit(
                    Events::PoolMemberRewardAddressChanged {
                        pool_member, new_address: reward_address, old_address,
                    },
                );
