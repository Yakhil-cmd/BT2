### Title
Pool Admin Transfer Does Not Reset `poolAdminFeeDestination`, Allowing Previous Admin to Permanently Capture Admin Fees - (`File: metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`acceptPoolAdmin()` updates `poolAdmin[pool]` to the incoming admin but never resets `poolAdminFeeDestination[pool]`. Because `collectPoolFees()` is permissionless and always routes the admin fee share to the stale `poolAdminFeeDestination[pool]`, the outgoing admin (or any address they chose) continues to receive all admin fee revenue after the handover until the new admin explicitly calls `setPoolAdminFeeDestination()`.

---

### Finding Description

The pool admin role is transferred via a two-step flow: `proposePoolAdminTransfer()` followed by `acceptPoolAdmin()`. The `acceptPoolAdmin()` function correctly updates `poolAdmin[pool]` and clears `pendingPoolAdmin[pool]`, but it does not touch `poolAdminFeeDestination[pool]`:

```solidity
// MetricOmmPoolFactory.sol lines 518-526
function acceptPoolAdmin(address pool) external override nonReentrant {
    address pending = pendingPoolAdmin[pool];
    if (pending == address(0)) revert NoPendingPoolAdminTransfer();
    if (msg.sender != pending) revert NotPendingPoolAdmin(pool, msg.sender, pending);
    address previousAdmin = poolAdmin[pool];
    poolAdmin[pool] = pending;
    delete pendingPoolAdmin[pool];
    emit PoolAdminTransferred(pool, previousAdmin, pending);
    // ← poolAdminFeeDestination[pool] is never reset
}
``` [1](#0-0) 

`collectPoolFees()` is permissionless — any caller can trigger it — and it unconditionally reads `poolAdminFeeDestination[pool]` to route the admin share:

```solidity
// MetricOmmPoolFactory.sol lines 379-389
function collectPoolFees(address pool) external override nonReentrant {
    PoolFeeConfig memory c = poolFeeConfig[pool];
    IMetricOmmPoolCollectFees(pool).collectFees(
        c.protocolSpreadFeeE6,
        c.adminSpreadFeeE6,
        c.protocolNotionalFeeE8,
        c.adminNotionalFeeE8,
        poolAdminFeeDestination[pool]   // ← stale after admin transfer
    );
}
``` [2](#0-1) 

The only way to update the destination is `setPoolAdminFeeDestination()`, which requires `onlyPoolAdmin(pool)`: [3](#0-2) 

Nothing in the admin-transfer flow prompts or forces the new admin to call it. The `poolAdminFeeDestination` mapping is set at pool creation and is independent of `poolAdmin`: [4](#0-3) 

---

### Impact Explanation

All admin fee revenue (both spread and notional components) accrued after the admin handover continues to be transferred to the previous admin's chosen destination. The new admin receives nothing until they discover the issue and call `setPoolAdminFeeDestination()`. In the malicious variant, the outgoing admin deliberately sets `poolAdminFeeDestination[pool]` to an address they control immediately before proposing the transfer, guaranteeing they capture all future admin fees indefinitely. This is a direct loss of protocol-fee revenue owed to the new pool admin.

---

### Likelihood Explanation

Pool admin transfers are a normal operational event (key rotation, multisig migration, protocol handover). The outgoing admin is semi-trusted and has legitimate access to `setPoolAdminFeeDestination()` at any time before the transfer completes. The permissionless `collectPoolFees()` means any third party (keeper, bot, or the outgoing admin themselves) can trigger fee collection at any time after the handover, draining fees to the stale destination without the new admin's involvement.

---

### Recommendation

Reset `poolAdminFeeDestination[pool]` inside `acceptPoolAdmin()`. The safest reset is to `address(0)` (which should then be treated as "send to new admin" in `collectPoolFees`), or directly to `pending` (the incoming admin's address):

```solidity
function acceptPoolAdmin(address pool) external override nonReentrant {
    address pending = pendingPoolAdmin[pool];
    if (pending == address(0)) revert NoPendingPoolAdminTransfer();
    if (msg.sender != pending) revert NotPendingPoolAdmin(pool, msg.sender, pending);
    address previousAdmin = poolAdmin[pool];
    poolAdmin[pool] = pending;
    poolAdminFeeDestination[pool] = pending; // ← reset to new admin
    delete pendingPoolAdmin[pool];
    emit PoolAdminTransferred(pool, previousAdmin, pending);
}
```

Alternatively, collect all accrued fees at the old destination before the handover (similar to how `setPoolAdminFees()` collects before changing rates), then reset the destination.

---

### Proof of Concept

1. Pool is deployed with `admin = Alice`, `adminFeeDestination = Alice`.
2. Alice calls `setPoolAdminFeeDestination(pool, Alice)` (no-op, already set — or to a separate wallet she controls).
3. Alice calls `proposePoolAdminTransfer(pool, Bob)`.
4. Bob calls `acceptPoolAdmin(pool)`. `poolAdmin[pool]` is now `Bob`; `poolAdminFeeDestination[pool]` is still `Alice`.
5. Swaps occur; admin fees accrue on the pool.
6. Anyone calls `collectPoolFees(pool)` — admin fee share is transferred to `Alice`, not `Bob`.
7. Bob must call `setPoolAdminFeeDestination(pool, Bob)` to fix this, but nothing in the protocol flow informs or requires him to do so.
8. Until Bob acts, every `collectPoolFees` call (permissionless, callable by Alice or any keeper) drains admin fees to Alice.
