### Title
Partial Eviction Without Rollback in `try_make_space` Permanently Removes Third-Party Transactions When Replacement Is Rejected — (`crates/apollo_mempool/src/mempool.rs`)

---

### Summary

`try_make_space` permanently removes gap-account transactions from the pool even when it cannot free enough space to admit the incoming transaction. When it returns `false`, `handle_capacity_overflow` propagates `Err(MempoolFull)`, the replacement is rejected, the original transaction is untouched — but the third-party evictions are irreversible. An unprivileged attacker can exploit this to systematically drain gap-account transactions from the mempool at zero cost.

---

### Finding Description

The execution path in `add_tx_validations` is:

1. `validate_fee_escalation` returns `Some(replaced_tx_reference)` — a valid replacement exists.
2. `freed_bytes` is computed from the existing (small) transaction still in the pool.
3. `exceeds_capacity(tx_new, freed_bytes)` is `true` because `tx_new` is much larger than `tx_old`.
4. `handle_capacity_overflow` is called. [1](#0-0) 

Inside `handle_capacity_overflow`, for a fee-escalation replacement where the attacker's account has no gap and `tx.nonce() == account_nonce`, `creating_gap` evaluates to `false`, so `try_make_space(required_space)` is called: [2](#0-1) 

Inside `try_make_space`, transactions are removed from the pool **immediately and unconditionally** as they are iterated: [3](#0-2) 

If the loop exhausts all evictable accounts before accumulating `required_space` bytes, it breaks and returns `false`: [4](#0-3) 

There is **no rollback**. The already-evicted transactions are gone. `handle_capacity_overflow` then returns `Err(MempoolFull)`: [5](#0-4) 

The `?` in `add_tx_validations` propagates the error, so `remove_replaced_tx` is never reached — the original transaction remains in the pool: [6](#0-5) 

**Net state after the attack:**
- Replacement transaction: rejected.
- Original transaction (`T_old`): still in the pool (correct).
- Third-party gap-account transactions: **permanently evicted** (incorrect).
- Freed space: now available for the attacker's next submission or anyone else.

---

### Impact Explanation

Valid transactions submitted by third-party users are permanently removed from the mempool without any corresponding admission. The attacker pays nothing (their replacement is rejected, their original transaction is untouched) and can repeat the attack to drain all gap-account transactions. This directly corrupts mempool admission: valid transactions are rejected before sequencing.

Impact category: **High — Mempool admission rejects valid transactions before sequencing.**

---

### Likelihood Explanation

- Requires only: (a) an existing small transaction in the attacker's account, (b) some gap-account transactions in the mempool, (c) the ability to submit a large replacement (e.g., padded signature or large calldata within gateway limits).
- No privileged access required. The gateway's stateless validator enforces a maximum signature/calldata size, but the delta between a large-but-valid replacement and a small original can still exceed the total bytes held by gap accounts.
- The attack is repeatable and cheap: the attacker's original transaction is never removed.

---

### Recommendation

`try_make_space` must not commit evictions unless it can guarantee it will return `true`. Two correct approaches:

1. **Dry-run first**: compute how many bytes are available from gap accounts before evicting anything. Only proceed with actual removal if `available_bytes >= required_space`.
2. **Rollback on failure**: collect evicted transactions in a staging list; re-insert them if the final check fails.

Additionally, the capacity check in `handle_capacity_overflow` should be structured so that `try_make_space` is only called when there is a reasonable chance of success (e.g., pre-check total evictable bytes).

---

### Proof of Concept

```
Setup:
  - capacity_in_bytes = 1000
  - current pool usage = 900 bytes
  - gap_account B has one transaction of 50 bytes (accounts_with_gap = {B})
  - attacker account A has T_old (nonce=5, 50 bytes) in pool

Attack:
  - Attacker submits T_new (nonce=5, 200 bytes) with fee escalation ≥ threshold

Trace:
  freed_bytes = 50  (T_old.total_bytes())
  exceeds_capacity: (900 + 200) - 50 = 1050 > 1000  → true
  handle_capacity_overflow:
    creating_gap = false  (account has T_old, nonce matches account_nonce)
    required_space = 200 - 50 = 150
    try_make_space(150):
      evict B's 50-byte tx → total_space_freed = 50 < 150
      no more evictable accounts → break
      return false  (50 < 150)
    → Err(MempoolFull)

Result:
  - T_new: rejected ✓
  - T_old: still in pool ✓
  - B's 50-byte tx: permanently evicted ✗  ← BUG
```

A Rust unit test asserting `tx_pool.get_by_tx_hash(b_tx_hash).is_ok()` after the rejected `add_tx` call would fail, confirming the eviction.
