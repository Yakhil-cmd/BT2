### Title
Gas-Key Replacement Between Tx Submission and Refund Receipt Allows Attacker to Redirect Burned Refund to Account Balance — (`runtime/runtime/src/actions.rs`, `runtime/runtime/src/lib.rs`)

---

### Summary

When a gas key transaction is submitted, the gas cost is deducted from `GasKeyInfo.balance` at validation time. The unused-gas refund is delivered later via a separate refund receipt. If the attacker deletes the gas key (burning its remaining balance) and re-adds a `FullAccess` key with the same public key before the refund receipt is processed, `try_refund_gas_key_balance` returns `false` and the fallback path credits `account.amount` instead. Tokens that the protocol intended to burn (as part of the gas key balance on deletion) are instead returned to the account.

---

### Finding Description

**Step 1 — Gas cost deducted at tx validation.**

`verify_and_charge_gas_key_tx_ephemeral` (verifier.rs:447) computes `new_gas_key_balance = gas_key_info.balance - gas_cost` and commits this to state. The full `gas_cost` (prepaid gas × gas price) is deducted immediately; the unused portion (`refund_amount = gas_cost - tokens_burnt`) is returned later via a refund receipt. [1](#0-0) 

**Step 2 — Gas key deletion burns the remaining balance.**

`delete_gas_key` (access_keys.rs:112-113) adds `gas_key_info.balance` to `result.tokens_burnt` and removes the key from the trie. Deletion is allowed only when `balance ≤ GasKeyInfo::MAX_BALANCE_TO_BURN` (1 NEAR). [2](#0-1) 

**Step 3 — Refund receipt dispatched to `action_transfer_or_implicit_account_creation`.**

At lib.rs:2843-2854, `is_gas_refund` is true when `is_refund && signer_id == receiver_id`. The code calls `try_refund_gas_key_balance` using `action_receipt.signer_public_key()` read from the receipt (not from live state). If that call returns `false`, the fallback `action_transfer` credits `account.amount`. [3](#0-2) 

**Step 4 — `try_refund_gas_key_balance` returns `false` for a FullAccess key.**

`gas_key_info_mut()` matches only `GasKeyFunctionCall` and `GasKeyFullAccess` variants. A `FullAccess` key returns `None`, so the function returns `Ok(false)` at line 110. [4](#0-3) [5](#0-4) 

**Attack sequence (all unprivileged user actions):**

| Chunk | Action |
|-------|--------|
| N | Gas key tx included → action receipt created; gas_cost deducted from `GasKeyInfo.balance` |
| N | Delete-gas-key + AddKey(FullAccess, same pubkey) tx included → action receipt created |
| N+1 | Both action receipts processed: gas key deleted (balance burned), FullAccess key added, gas refund receipt generated |
| N+2 | Gas refund receipt processed: `try_refund_gas_key_balance` finds FullAccess key → `Ok(false)` → `action_transfer` credits `account.amount` |

The delete+add-key tx can be submitted in the same block as the gas key tx. Because refund receipts are always processed one chunk after the action receipt that generates them, the FullAccess key is guaranteed to be in place before the refund receipt is processed.

---

### Impact Explanation

**Exact corrupted values:**

- `GasKeyInfo.balance`: Not credited (key no longer exists as a gas key).
- `Account.amount`: Over-credited by `refund_amount` — tokens that the protocol burned (as part of the gas key balance on deletion) are instead returned to the account.

**Token conservation is broken at the gas-key layer.** In the normal flow:

```
burned = tokens_burnt(tx) + (gas_key_balance_at_deletion)
       = tokens_burnt + (X - gas_cost)
       = X - refund_amount
account_net = -X
```

In the attack:

```
burned = tokens_burnt(tx) + (X - gas_cost)   [deletion]
       = X - refund_amount                    [same]
account_net = -X + refund_amount              [WRONG: refund_amount recovered]
```

The attacker recovers `refund_amount` to `account.amount` that the protocol intended to burn. The maximum per-cycle gain is bounded by `GasKeyInfo::MAX_BALANCE_TO_BURN` = 1 NEAR (because deletion is blocked above that threshold), but the attack is repeatable. [6](#0-5) 

---

### Likelihood Explanation

The attack requires only standard user transactions (AddKey, TransferToGasKey, gas key tx, DeleteKey, AddKey). No validator or operator privilege is needed. The timing window (one chunk gap between action receipt processing and refund receipt processing) is deterministic and exploitable by any user who can submit two transactions in the same block. The constraint that the gas key balance at deletion must be ≤ 1 NEAR is easily satisfied by controlling the funded amount and gas limit.

---

### Recommendation

The refund receipt must carry a flag or the action receipt must record whether the original signer key was a gas key at tx-submission time. On refund processing, if the original key was a gas key but is no longer present (or is no longer a gas key), the refund amount should be burned rather than credited to `account.amount`. Alternatively, `try_refund_gas_key_balance` should distinguish between "key not found" (safe to fall back to account) and "key found but no longer a gas key" (should burn the refund).

---

### Proof of Concept

```rust
// 1. Setup: alice has a gas key K funded with 1 NEAR + gas_cost
// 2. Submit gas key tx (nonce_index=0) → gas_cost deducted from gas key balance
// 3. In the same block, submit:
//      DeleteKey { public_key: K }
//      AddKey { public_key: K, access_key: AccessKey::full_access() }
// 4. After two more blocks, assert:
//      get_access_key(state, alice, K).permission == FullAccess  // key replaced
//      alice.amount() == initial_amount - X + refund_amount      // over-credited
//      gas_key no longer exists (was deleted and burned)
//
// The refund_amount that should have been burned is now in alice.amount().
```

The existing test `test_gas_refund_unknown_key_falls_back_to_account` (apply.rs:4052) already demonstrates the fallback path for a missing key. The same fallback fires for a replaced key, confirming the mechanism. [7](#0-6)

#[test]
fn test_gas_refund_unknown_key_falls_back_to_account() {
    let initial_balance = Balance::from_near(1_000_000);
    let (runtime, tries, root, mut apply_state, _signers, epoch_info_provider) = setup_runtime(
        vec![alice_account()],
        initial_balance,
        Balance::ZERO,
        Gas::from_teragas(1000),
    );
    apply_state.current_protocol_version = ProtocolFeature::GasKeys.protocol_version();

    // Create a gas refund receipt with a public key that doesn't exist on the account
    let unknown_key = PublicKey::from_seed(KeyType::ED25519, "unknown_key");
    let refund_amount = Balance::from_millinear(1);
    let gas_refund = Receipt::new_gas_refund(&alice_account(), refund_amount, unknown_key);

    let shard_uid = ShardUId::single_shard();
    let apply_result = runtime
        .apply(
            tries.get_trie_for_shard(shard_uid, root),
            &None,
            &apply_state,
            &[gas_refund],
            SignedValidPeriodTransactions::empty(),
            &epoch_info_provider,
            Default::default(),
        )
        .unwrap();

    let root = commit_apply_result(&apply_result, &mut apply_state, &tries, shard_uid);
    let state = tries.new_trie_update(shard_uid, root);

    // Account balance should increase as fallback
    let alice = get_account(&state, &alice_account()).unwrap().unwrap();
    assert_eq!(alice.amount(), initial_balance.checked_add(refund_amount).unwrap());
}
