### Title
WalletConnect Commands Execute After User Logout Due to No-Op `handleLogOut` — (`packages/core/src/components/Auth/AuthProvider.tsx`)

### Summary
`AuthProvider.handleLogOut` is intentionally a no-op. When a user clicks "Log Out", the GUI navigates to the wallet selection screen but the daemon remains logged in with the previous fingerprint. Because `WalletConnectProvider` stays mounted across all routes and its `session_request` listener is never torn down, a previously-paired WalletConnect dApp can send commands that pass all fingerprint guards and reach `dispatchAsPair`, executing wallet actions (send, sign, offer, etc.) on a wallet the user believed they had logged out of.

### Finding Description
`AuthProvider.handleLogOut` contains only a comment and no implementation:

```typescript
// immutable
const handleLogOut = useCallback(async () => {
  // do nothing until backend change API,
  // user is still logged in and syncing with previously selected fingerprint
}, []);
```

`LogoutButton` calls this no-op and then calls `navigate('/')`, placing the user on the wallet selection screen. The daemon's `get_logged_in_fingerprint` endpoint continues to return the previous fingerprint.

`WalletConnectProvider` is mounted as a child of `AppProviders`, which is the root `<Route path="/">` element in `AppRouter`. It is never unmounted during in-app navigation. Its `session_request` event handler remains registered on the WC `SignClient`.

When `handleProcess` in `useWalletConnectCommand` runs, its primary guard is:

```typescript
const requestedFingerprint = fingerprint ?? currentFingerprint;
if (
  typeof requestedFingerprint !== 'number' ||
  !requestedFingerprint ||
  requestedFingerprint !== pair.fingerprint ||
  currentFingerprint !== pair.fingerprint   // daemon still returns old fingerprint
) {
  throw new WcError(...);
}
```

Because the daemon was never logged out, `currentFingerprint` still equals `pair.fingerprint`, so the guard passes. The `requiresSync` path re-fetches the fingerprint from the daemon and performs the same comparison — it also passes. `dispatchAsPair` is then called, executing the command.

### Impact Explanation
A malicious WalletConnect dApp that the user previously paired with can send `session_request` events (e.g., `chia_sendTransaction`, `chia_createOfferForIds`, `chia_signMessageById`) at any time after the user clicks "Log Out". All GUI-layer fingerprint guards pass because the daemon is still logged in. The dApp can drain funds, create or accept offers, or sign arbitrary messages on the wallet the user believed was no longer active.

### Likelihood Explanation
The attacker must have a prior WC pairing with the victim (requires prior user approval). However, once paired, the attack window is open indefinitely after any logout — the user has no way to close it short of disconnecting the WC pair or restarting the application. A malicious dApp can time the `session_request` to fire immediately after detecting the user has navigated away (e.g., by monitoring session activity or simply sending requests on a timer).

### Recommendation
`handleLogOut` must perform a real logout action. At minimum it should:
1. Call the daemon's logout/switch endpoint (or log in to a null/empty state) so `get_logged_in_fingerprint` returns `undefined`.
2. Disconnect all active WalletConnect sessions, or at minimum pause processing of `session_request` events while no fingerprint is active.

A simpler short-term guard: in `handleProcess`, reject the request if `currentFingerprint` is `undefined` or if the `AuthContext` fingerprint is `undefined` (i.e., the user is on the selection screen).

### Proof of Concept
1. Pair a WC dApp with wallet fingerprint `F`.
2. Click "Log Out" in the GUI — observe navigation to `/` (wallet selection screen).
3. From the dApp, send a `session_request` for `chia_sendTransaction` with `fingerprint: F` and a valid destination/amount.
4. Observe: `processSessionRequest` fires, `handleProcess` passes all fingerprint checks (daemon still reports `F`), `dispatchAsPair` is called, and the transaction is submitted.
5. Assert: `currentFingerprint` from `useGetLoggedInFingerprintQuery()` is still `F` (non-null) throughout.
