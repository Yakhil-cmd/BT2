### Title
Permanent ETH Freeze When `error_refund` Feature Is Disabled and `ft_transfer` Fails — (`engine-precompiles/src/native.rs`, `engine/src/contract_methods/connector.rs`)

### Summary

When the Aurora Engine is compiled without the `error_refund` feature flag, the exit-to-NEAR precompile constructs `ExitToNearPrecompileCallbackArgs` with `refund: None`. For standard ETH/ERC-20 exits (where `transfer_near` is also `None`), this causes the promise to be emitted **without any callback**. If the downstream `ft_transfer` or `ft_transfer_call` promise fails, there is no recovery path: the EVM-side ETH or ERC-20 balance has already been burned, and no refund is ever issued.

---

### Finding Description

**Step 1 — Precompile constructs callback args with `refund: None`**

In `engine-precompiles/src/native.rs`, the `ExitToNear::run()` method builds the callback args:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,
    transfer_near: transfer_near_args,
};
``` [1](#0-0) 

When `error_refund` is disabled, `refund` is hardcoded to `None`.

**Step 2 — No callback is registered for standard exits**

Immediately after, the code checks whether `callback_args` equals the default (both fields `None`):

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // ← no callback at all
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs { ... })
};
``` [2](#0-1) 

For a standard ETH or ERC-20 exit (not wNEAR unwrap), `transfer_near` is `None`. Combined with `refund: None`, `callback_args` equals `ExitToNearPrecompileCallbackArgs::default()`, so the engine emits a bare `PromiseArgs::Create` — **no callback is attached**. A failure of `ft_transfer` is silently dropped.

**Step 3 — wNEAR unwrap path: callback exists but else-branch does nothing**

For the wNEAR unwrap path, `transfer_near` is `Some(...)`, so `callback_args ≠ default()` and a callback IS registered. However, inside `exit_to_near_precompile_callback`:

```rust
} else if let Some(args) = args.refund {
    // refund path
} else {
    None   // ← silent no-op when refund is None
};
``` [3](#0-2) 

Because `args.refund` is `None` (compiled out), the else branch returns `Ok(None)` without restoring any balance.

**Step 4 — `ExitToNearPrecompileCallbackArgs` struct definition**

The struct always has `refund: Option<RefundCallArgs>` — it is never compiled out of the type itself; only the value assigned to it changes based on the feature flag: [4](#0-3) 

---

### Impact Explanation

The EVM-side ETH (or ERC-20 token) is burned **before** the NEAR-side promise is dispatched. If the NEP-141 `ft_transfer` fails (e.g., recipient has no storage deposit, NEP-141 contract panics, or any other rejection), the burned balance is never restored. The funds are permanently frozen: gone from the EVM state, never credited on the NEAR side.

Impact: **Critical — Permanent freezing of funds.**

---

### Likelihood Explanation

Likelihood is **conditional on the deployment not enabling `error_refund`**. If production Aurora always enables this feature, the likelihood is zero. However, the feature flag is a first-class build option supported by the codebase (`engine/Cargo.toml`, `engine-precompiles/Cargo.toml`), and any deployment that omits it is silently vulnerable. A user does not need to be an attacker — any ordinary exit call where the NEP-141 transfer fails (e.g., missing storage deposit on the recipient side) will permanently lose funds.

---

### Recommendation

1. **Remove the `#[cfg(not(feature = "error_refund"))] refund: None` branch** or make `error_refund` a mandatory (non-optional) feature for any production build.
2. If the feature must remain optional, **always register a callback** and have the callback detect the missing `refund` field and either revert the EVM transaction or emit an on-chain error event that can be acted upon.
3. Add a compile-time or runtime assertion that prevents the engine from accepting exit-to-NEAR calls when `error_refund` is disabled, so the risk is explicit rather than silent.

---

### Proof of Concept

```
1. Compile aurora-engine WITHOUT the `error_refund` feature.
2. Deploy the engine to a local NEAR sandbox.
3. Fund an EVM address with ETH.
4. From that EVM address, call the ExitToNear precompile (address
   0xe9217bc70b7ed1f598ddd3199e80b093fa71124f) with flag=0x0 and a
   NEAR recipient that has NO storage deposit on the eth-connector.
5. The engine emits PromiseArgs::Create(ft_transfer) with no callback.
6. The eth-connector's ft_transfer panics (no storage deposit).
7. Assert: EVM balance of sender == 0 (burned, not restored).
8. Assert: NEP-141 balance of recipient == 0 (transfer never completed).
   → Funds are permanently frozen.
```

The same path applies to ERC-20 exits (flag=0x1): the ERC-20 tokens are burned on the EVM side, `ft_transfer` is called on the NEP-141 contract, and if it fails with no callback registered, the tokens are permanently lost. [5](#0-4) [6](#0-5)
