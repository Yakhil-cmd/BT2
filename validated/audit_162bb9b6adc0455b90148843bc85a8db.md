### Title
Missing Validation of `app_pk` Allows Malicious App to Extract Confidential Derived Key - (File: `src/confidential_key_derivation/protocol.rs`)

### Summary

The `ckd()` function in `src/confidential_key_derivation/protocol.rs` accepts an `app_pk: PublicKey` (a `blstrs::G1Projective` point) from the caller without any validation. A malicious TEE application can submit the G1 identity element as `app_pk`, which collapses the ElGamal blinding layer and causes the coordinator's output `C` to directly equal `msk · H(pk, app_id)` — the confidential derived key — in plaintext.

### Finding Description

The CKD protocol is designed so that each MPC node computes:

```
C_i = S_i + y_i · A     where A = app_pk, S_i = x_i · H(pk, app_id)
```

The coordinator aggregates:

```
C = Σ λ_i · C_i = msk · H(pk, app_id) + a · Y
```

The app recovers the key as `sig = C − a · Y = msk · H(pk, app_id)`, which requires knowledge of the private scalar `a` corresponding to `A = a · G1`.

If a malicious app submits `app_pk = O` (the G1 identity element), then `y_i · O = O` for any scalar `y_i`, so:

```
C_i = S_i + y_i · O = S_i = x_i · H(pk, app_id)
C   = msk · H(pk, app_id) + a · Y = msk · H(pk, app_id) + 0 = msk · H(pk, app_id)
```

The coordinator's `CKDOutput.big_c` field is now the raw BLS signature `msk · H(pk, app_id)` — the confidential derived key — with no blinding. The attacker reads it directly from the returned `CKDOutput` without needing to know any private scalar.

The root cause is that `ckd()` validates participant membership and counts but performs **no validation of `app_pk`**:

```rust

    participants: &[Participant],
    coordinator: Participant,
    me: Participant,
    key_pair: KeygenOutput,
    app_id: impl Into<AppId>,

    rng: impl CryptoRngCore + Send + 'static,
) -> Result<impl Protocol<Output = CKDOutputOption>, InitializationError> {
    if participants.len() < 2 { ... }
    let Some(participants) = ParticipantList::new(participants) else { ... };
    if !participants.contains(me) { ... }
    if !participants.contains(coordinator) { ... }
    // No check: app_pk.is_identity(), app_pk.is_on_curve(), app_pk.is_torsion_free()
``` [1](#0-0) 

The `compute_signature_share` function then uses `app_pk` directly in the ElGamal encryption step:

```rust
// C <- S + y . A
let big_c = big_s + app_pk * y.0;
``` [2](#0-1) 

Contrast this with `verify_signature()` in `ciphersuite.rs`, which explicitly validates G1 points before use:

```rust
let element1: G1Affine = signature.into();
if (!element1.is_on_curve() | !element1.is_torsion_free() | element1.is_identity()).into() {
    return Err(frost_core::Error::InvalidSignature);
}
``` [3](#0-2) 

The same validation is absent at the `ckd()` entry point.

### Impact Explanation

A malicious TEE application submitting `app_pk = O` (identity) causes the coordinator to return `CKDOutput.big_c = msk · H(pk, app_id)` in plaintext. This is the confidential derived key `s` that the protocol is designed to keep secret from all parties except the legitimate app. The attacker reads it directly from the coordinator output without needing any private key material.

This matches the allowed Critical impact: **Extraction, reconstruction, or disclosure of confidential derived secrets**. [4](#0-3) 

### Likelihood Explanation

The `app_pk` is supplied by the TEE application as part of the CKD request. Any caller who can submit a CKD request — which is the intended use case — can supply an arbitrary G1 point, including the identity. No privileged access is required. The `blstrs::G1Projective` type natively represents the identity element, so no special encoding tricks are needed. [5](#0-4) 

### Recommendation

Add an explicit validity check for `app_pk` in `ckd()` before the protocol runs, mirroring the checks already present in `verify_signature()`:

```rust
let app_pk_affine: blstrs::G1Affine = app_pk.into();
if (!app_pk_affine.is_on_curve()
    | !app_pk_affine.is_torsion_free()
    | app_pk_affine.is_identity()).into()
{
    return Err(InitializationError::BadParameters(
        "app_pk must be a valid, non-identity G1 point".to_string(),
    ));
}
``` [6](#0-5) 

### Proof of Concept

1. Attacker (malicious TEE app) submits a CKD request with `app_pk = G1Projective::identity()`.
2. All MPC nodes call `ckd(..., app_pk = O, ...)`.
3. Each node computes `C_i = x_i · H(pk, app_id) + y_i · O = x_i · H(pk, app_id)`.
4. Coordinator aggregates `C = Σ λ_i · C_i = msk · H(pk, app_id)`.
5. Coordinator returns `CKDOutput { big_y: Y, big_c: msk · H(pk, app_id) }`.
6. Attacker reads `big_c` directly — this is the confidential derived key `s = HKDF(msk · H(pk, app_id))` input, fully exposed. [7](#0-6) [8](#0-7)

    key_pair: KeygenOutput,

    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }

    // kick out duplicates

        return Err(InitializationError::DuplicateParticipants);

        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });

        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });

    let chan = comms.shared_channel();

        chan,
        coordinator,
        me,
        participants,
        key_pair,
        app_id.into(),
        app_pk,
        rng,
    );
    Ok(make_protocol(comms, fut))
}

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))

/// BLS signature verification
/// following the standard in <https://www.ietf.org/archive/id/draft-irtf-cfrg-bls-signature-05.html#name-coreverify>
pub fn verify_signature(
    verifying_key: &VerifyingKey,
    msg: &[u8],
    signature: &Signature,
) -> Result<(), frost_core::Error<BLS12381SHA256>> {
    let element1: G1Affine = signature.into();
    if (!element1.is_on_curve() | !element1.is_torsion_free() | element1.is_identity()).into() {
        return Err(frost_core::Error::InvalidSignature);
    }
    let element2: G2Affine = verifying_key.to_element().into();
    if (!element2.is_on_curve() | !element2.is_torsion_free() | element2.is_identity()).into() {
        return Err(frost_core::Error::MalformedVerifyingKey);
    }

    // Concatenate the master public key (96 bytes) in the hash computation
    // H(pk || app_id) when H is a random oracle
    let base1 = hash_app_id_with_pk(verifying_key, msg).into();
    let base2 =
        <<BLS12381SHA256 as frost_core::Ciphersuite>::Group as frost_core::Group>::generator()
            .into();

    if blstrs::pairing(&base1, &element2).eq(&blstrs::pairing(&element1, &base2)) {
        Ok(())
    } else {
        Err(frost_core::Error::InvalidSignature)
    }
}
