### Title
Unbounded Memory Growth via Malicious Message Injection into `MessageBuffer` — (`File: src/protocol/internal.rs`)

---

### Summary

The `MessageBuffer` structure that underpins every protocol in this library accepts and stores messages for arbitrary, attacker-controlled `MessageHeader` keys with no bound on the total number of entries or total memory consumed. A malicious participant can inject messages tagged with unique, never-polled waitpoints, causing unbounded heap growth in any honest party running DKG, reshare, refresh, presign, or signing protocols. The codebase itself contains a test that explicitly proves this behavior.

---

### Finding Description

`MessageBuffer` is defined as a `HashMap<MessageHeader, SubMessageQueue>` where each `SubMessageQueue` wraps a `futures::channel::mpsc::unbounded()` channel pair. [1](#0-0) 

The `push` method inserts a new `SubMessageQueue` for every distinct `MessageHeader` it receives, with no cap: [2](#0-1) 

A `MessageHeader` is composed of a 32-byte `ChannelTag` and a `u64` `Waitpoint`: [3](#0-2) 

The public entry point `Protocol::message()` feeds directly into this path without any validation of the waitpoint value: [4](#0-3) 

`push_message` only checks that the message is long enough to contain a header and that the header parses — it performs no check on whether the waitpoint is one the protocol will ever poll: [5](#0-4) 

The codebase itself contains a test that explicitly proves this attack path and asserts the unbounded growth: [6](#0-5) 

The comment at line 543 reads: *"Attacker injects messages for waitpoints the honest code never polls."* The test asserts that all 512 injected messages persist in the buffer — they are never evicted.

---

### Impact Explanation

Every protocol in the library — `do_keyshare` (DKG/reshare/refresh), OT-based triple generation, robust ECDSA presign/sign, FROST signing, and CKD — is built on `make_protocol`, which wraps a `ProtocolExecutor` backed by a `Comms`/`MessageBuffer`. [7](#0-6) 

Any participant in any of these protocols can call `Protocol::message()` on an honest party's protocol instance with crafted messages. Each message with a unique waitpoint allocates a new `SubMessageQueue` (two heap-allocated channel endpoints plus the message payload) that is never freed during the protocol's lifetime because the honest code never polls those waitpoints and the `HashMap` has no eviction policy.

An attacker sending `N` messages with distinct waitpoints causes `O(N)` heap allocations that persist for the entire protocol session. With `u64` waitpoints, the theoretical space is 2^64 unique keys. In practice, even a few hundred thousand injected messages can exhaust available memory on a constrained node, permanently denying DKG, reshare, or signing completion for honest parties.

**Allowed impact matched**: *Medium — Griefing or resource-exhaustion by a malicious participant causing unbounded memory beyond documented behavior.*

---

### Likelihood Explanation

- The attack requires only that the adversary is a participant in the protocol (or can deliver messages to a participant's `Protocol::message()` method on behalf of any `Participant` identity).
- No cryptographic material, key compromise, or special privilege is needed — the `message()` method accepts raw bytes from any `Participant`.
- The `Participant` type is a simple `u32`-derived identifier with no authentication at the `MessageBuffer` layer. [5](#0-4) 

Any malicious co-participant in a DKG or signing session can execute this attack. The barrier to entry is low.

---

### Recommendation

1. **Cap the `MessageBuffer` size**: Enforce a maximum number of distinct `MessageHeader` keys (e.g., proportional to `n * max_waitpoints_per_protocol`). Reject or drop messages that would exceed this cap.
2. **Validate waitpoints at ingress**: In `push_message`, check that the waitpoint falls within a known valid range for the current protocol phase before inserting into the map.
3. **Evict stale entries**: After a protocol round completes, remove entries for waitpoints that will never be polled again.
4. **Bound `SubMessageQueue` depth per key**: Even for valid waitpoints, limit the number of queued messages per `(header, sender)` pair to prevent a single participant from flooding a single slot.

---

### Proof of Concept

The existing test in the repository directly demonstrates the vulnerability:

```rust
// src/protocol/internal.rs, lines 532-554
fn attacker_can_fill_message_buffer_with_unused_waitpoints() {
    let comms = Comms::new();
    let attacker = Participant::from(99_u32);
    let attack_count = 512_u64;

    for i in 0..attack_count {
        let header =
            MessageHeader::new(ChannelTag::root_shared()).with_waitpoint(1_000_000 + i);
        let mut message = header.to_bytes().to_vec();
        message.extend_from_slice(&i.to_le_bytes());

        // Attacker injects messages for waitpoints the honest code never polls.
        comms.push_message(attacker, message);
    }

    // All 512 entries persist — unbounded growth confirmed.
    assert!(messages.len() == usize::try_from(attack_count).unwrap());
}
``` [6](#0-5) 

To escalate to a full denial-of-signing scenario: a malicious participant in a DKG session calls `Protocol::message()` in a tight loop with messages whose waitpoints are offset far beyond any value the honest protocol uses (e.g., `1_000_000 + i` for `i` in `0..u64::MAX`). Each call allocates a new `SubMessageQueue` on the heap. The honest party's DKG future never polls these waitpoints, so the entries accumulate until the process is OOM-killed, permanently preventing the honest party from completing key generation or signing.
