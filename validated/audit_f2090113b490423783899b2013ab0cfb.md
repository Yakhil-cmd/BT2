### Title
`native_per_pubdata` Computed with Floor Division Instead of Ceiling, Causing Pubdata to Be Underpriced in Native Resources - (`File: basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

`native_per_pubdata` — the native resource units charged per byte of pubdata — is computed using truncating (floor) division (`wrapping_div`) while the analogous `native_per_gas` is correctly computed using ceiling division (`div_ceil`). This rounding inconsistency causes every transaction that generates pubdata to be charged fewer native resources than the actual proving cost, allowing users to underpay for pubdata at the protocol's expense.

---

### Finding Description

In `validate_and_compute_fee_for_transaction` inside `validation_impl.rs`, two related ratios are computed:

```rust
// Line 135 — native_per_gas uses CEILING division (correct)
u256_try_to_u64(&gas_price.div_ceil(native_price))

// Line 142 — native_per_pubdata uses FLOOR division (wrong)
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

`native_per_gas` uses `div_ceil` to ensure the user pays *at least* the actual cost per gas unit. `native_per_pubdata` uses `wrapping_div` (Rust integer division, which truncates toward zero — i.e., floor division), so the user pays *at most* the actual cost per pubdata byte, with the remainder silently absorbed by the protocol.

The same floor division is replicated in `api/src/helpers.rs` line 427, which mirrors bootloader behavior for off-chain validation:

```rust
// api/src/helpers.rs:427
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
```

`native_per_pubdata` is then used in two places:

1. **Intrinsic pubdata overhead** (`gas_helpers.rs` line 352): `native_per_pubdata_byte.saturating_mul(intrinsic_pubdata)` — deducted from the native limit before execution.
2. **Runtime pubdata charge** (`gas_helpers.rs` line 431): `current_pubdata_spent.checked_mul(native_per_pubdata)` — charged after execution.

Both charging sites use the floored value, so every pubdata byte is undercharged by up to `native_price - 1` native units.

**Attacker-controlled entry path:**

1. An unprivileged user submits any L2 transaction that writes to storage (generating pubdata).
2. The bootloader computes `native_per_pubdata = floor(pubdata_price / native_price)` instead of `ceil(pubdata_price / native_price)`.
3. The user's native resource budget is consumed at a rate lower than the actual proving cost per pubdata byte.
4. The user can generate more pubdata than they actually paid for in native resources.

No privileged access is required. Any normal EVM transaction with storage writes triggers this path.

---

### Impact Explanation

The rounding error is up to `native_price - 1` native units per pubdata byte. With the default test `native_price = 10`, the undercharge is up to 9 native units per pubdata byte. For a transaction generating 100,000 bytes of pubdata (a realistic upper bound), the total undercharge is up to 900,000 native units per transaction.

The protocol/operator is systematically undercompensated for pubdata proving costs. A user who crafts transactions to maximize pubdata output (e.g., many storage writes) pays less native resource than the actual cost of proving those state changes. This is a resource accounting bug with direct financial impact: the operator bears proving costs that the user did not fully pay for.

---

### Likelihood Explanation

This triggers on every L2 transaction where `pubdata_price` is not exactly divisible by `native_price`. Since both values are operator-set U256 quantities, exact divisibility is not guaranteed in practice. The bug is always active whenever `pubdata_price % native_price != 0`, which is the common case. Any user submitting a storage-writing transaction exploits this automatically, without any special crafting.

---

### Recommendation

Replace `wrapping_div` with `div_ceil` for `native_per_pubdata` in both locations, consistent with how `native_per_gas` is computed:

```diff
// basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs:142
- let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
+ let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
      .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;

// api/src/helpers.rs:427
- let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
+ let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price)).ok_or(())?;
```

### Proof of Concept

**Concrete numeric example:**

- `pubdata_price = 19` (operator-set)
- `native_price = 10` (operator-set)
- `native_per_pubdata` (current, floor): `19 / 10 = 1`
- `native_per_pubdata` (correct, ceil): `ceil(19 / 10) = 2`

For a transaction generating 50,000 pubdata bytes:
- Current charge: `50,000 × 1 = 50,000` native units
- Correct charge: `50,000 × 2 = 100,000` native units
- **Undercharge: 50,000 native units per transaction**

The user pays half the intended native resource cost for pubdata, with the protocol absorbing the proving cost difference.

**Code path:**

1. User submits EIP-1559 L2 tx with storage writes → enters `validate_and_compute_fee_for_transaction`
2. Line 142: `native_per_pubdata = pubdata_price.wrapping_div(native_price)` → floored value
3. Line 352 (`gas_helpers.rs`): intrinsic pubdata deducted at floored rate
4. Line 431 (`gas_helpers.rs`): runtime pubdata charged at floored rate
