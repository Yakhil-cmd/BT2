### Title
`update_rewards(disable_rewards=true)` Callable by Any Address, Permanently Consuming a Block's Reward Slot Without Distributing Rewards — (`src/staking/staking.cairo`)

---

### Summary

`StakingRewardsManagerImpl::update_rewards` is callable by any non-zero address. When called with `disable_rewards=true`, it writes the current block number to the global `last_reward_block` counter but distributes no rewards. Because the guard `current_block_number > self.last_reward_block.read()` is checked first and the write happens unconditionally before the `disable_rewards` branch, the block is permanently marked as "processed" and every subsequent call for that block reverts with `REWARDS_ALREADY_UPDATED`. All stakers lose their consensus-block rewards for that block with no recovery path.

---

### Finding Description

`update_rewards` is specified as "Only starkware sequencer" (docs/spec.md line 1645), but the implementation enforces only `general_prerequisites()`, which checks that the contract is unpaused and the caller is non-zero:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();                          // no role check
    let current_block_number = starknet::get_block_number();
    assert!(
        current_block_number > self.last_reward_block.read(),
        "{}",
        Error::REWARDS_ALREADY_UPDATED,
    );
    ...
    self.last_reward_block.write(current_block_number);   // written unconditionally

    if disable_rewards || self.is_pre_consensus() {
        return;                                           // rewards skipped, block consumed
    }
    ...
}
``` [1](#0-0) 

`last_reward_block` is a single global storage slot shared across all stakers: [2](#0-1) 

The spec mandates sequencer-only access: [3](#0-2) 

The accounting invariant that is broken: `last_reward_block` is the "used" counter that gates reward distribution. Writing it without distributing rewards is the direct analog of the original bug — a deletion/revocation that advances a counter without performing the corresponding accounting update.

---

### Impact Explanation

Once an attacker calls `update_rewards(any_active_staker, disable_rewards=true)` at block N:

1. `last_reward_block` is set to N globally.
2. No `unclaimed_rewards_own` is credited to any staker; no pool rewards are forwarded.
3. Every subsequent call for block N reverts with `REWARDS_ALREADY_UPDATED`.
4. The rewards that would have been earned at block N are permanently unclaimable by all stakers and pool members.

This is **permanent freezing of unclaimed yield** for every participant in the protocol for the targeted block. The attacker can repeat this every block, continuously suppressing all consensus-reward distribution at negligible cost (one transaction per block).

---

### Likelihood Explanation

- No privileged role is required; any externally-owned account suffices.
- The only precondition is that at least one active staker with non-zero balance exists (trivially true in any live deployment).
- The attack is cheap (one call per block) and fully repeatable.
- There is no on-chain mechanism to recover the lost rewards after the fact.

---

### Recommendation

Add a sequencer-only role check at the top of `update_rewards`, consistent with the specification:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
+   self.roles.only_sequencer();   // enforce spec: "Only starkware sequencer"
    self.general_prerequisites();
    ...
}
```

Alternatively, move the `self.last_reward_block.write(current_block_number)` call to after the `disable_rewards` branch so that a no-op call does not consume the block slot.

---

### Proof of Concept

1. Consensus rewards are active (`!is_pre_consensus()`).
2. Staker S is active with non-zero balance.
3. Attacker A (any EOA) calls `update_rewards(S, disable_rewards=true)` at block B.
4. `last_reward_block` is now B; no rewards are distributed.
5. The legitimate sequencer attempts `update_rewards(S, disable_rewards=false)` at block B → reverts with `REWARDS_ALREADY_UPDATED`.
6. Staker S and all pool members earn zero rewards for block B.
7. A repeats step 3 at every subsequent block, permanently suppressing all reward distribution. [4](#0-3) [5](#0-4)

            ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
        ) {
            self.general_prerequisites();
            let current_block_number = starknet::get_block_number();
            assert!(
                current_block_number > self.last_reward_block.read(),
                "{}",
                Error::REWARDS_ALREADY_UPDATED,
            );

            // Assert staker exists and active.
            // Staker is considered to exist from the moment of `stake` (when `InternalStakerInfo`
            // struct is created) until the calling to `unstake_action` (when `InternalStakerInfo`
            // struct is deleted).
            // Staker remains active until the intent period begins, i.e. K epochs after
            // `unstake_intent` is called.
            let staker_info = self.internal_staker_info(:staker_address);
            let curr_epoch = self.get_current_epoch();
            assert!(
                self.is_staker_active(:staker_address, epoch_id: curr_epoch),
                "{}",
                Error::INVALID_STAKER,
            );

            let staker_pool_info = self.staker_pool_info.entry(staker_address).as_non_mut();
            let (staker_total_strk_balance, staker_total_btc_balance) = self

                    :staker_address, :staker_pool_info, epoch_id: curr_epoch,
                );
            // Assert staker has non-zero balance.

            // becomes effective.
            assert!(staker_total_strk_balance.is_non_zero(), "{}", Error::INVALID_STAKER);
