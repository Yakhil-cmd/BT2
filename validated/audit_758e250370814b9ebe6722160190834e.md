### Title
`skip_stateful_validations` Bypasses Signature Validation via Non-`deploy_account` Mempool Presence Check - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function is designed to skip `__validate__` entry-point execution for invoke transactions with nonce 1 when the account nonce is 0, to support the UX pattern of submitting `deploy_account + invoke` simultaneously. However, the mempool presence check it relies on — `account_tx_in_pool_or_recent_block` — returns `true` for **any** transaction in the mempool for the account, not exclusively a `deploy_account` transaction. An attacker who controls a freshly deployed account (on-chain nonce = 0) can first submit a valid invoke with nonce 0, then submit a second invoke with nonce 1 carrying an invalid signature. The gateway skips signature validation for the nonce-1 invoke and admits it to the mempool, violating the invariant that all admitted transactions must have passed account-level signature verification.

---

### Finding Description

**Root cause — wrong predicate in `skip_stateful_validations`:** [1](#0-0) 

The function checks two conditions before skipping validation:
1. `tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO)` — the invoke has nonce 1 and the on-chain account nonce is 0.
2. `account_tx_in_pool_or_recent_block(tx.sender_address())` returns `true`.

The comment at line 440–443 states the assumption:

> "It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."

This assumption is **false**. A regular invoke with nonce 0 for a freshly deployed account (on-chain nonce = 0) satisfies condition 2 without any `deploy_account` being present.

**`account_tx_in_pool_or_recent_block` checks for any transaction, not `deploy_account`:** [2](#0-1) 

It returns `true` if `self.state.contains_account(account_address) || self.tx_pool.contains_account(account_address)` — no transaction-type filter.

**Consequence — `run_validate_entry_point` called with `validate = false`:** [3](#0-2) 

When `skip_validate = true`, `execution_flags.validate` is set to `false`, so the blockifier's `validate_tx` returns `Ok(None)` immediately without executing `__validate__`: [4](#0-3) 

**Contrast with the `PyValidator` implementation**, which requires an explicit `deploy_account_tx_hash` parameter and checks `deploy_account_tx_hash.is_some()` — a much stronger predicate: [5](#0-4) 

**Attack path (step by step):**

1. Attacker deploys account contract at address X. On-chain nonce for X is now 0.
2. Attacker submits a valid invoke with nonce 0 for X. It passes `validate_nonce` (0 ≤ 0 ≤ max), `validate_by_mempool`, and `run_validate_entry_point` (signature valid). It enters the mempool.
3. Attacker submits an invoke with nonce 1 for X carrying an **invalid signature** (or malicious calldata):
   - `validate_nonce` passes: `0 ≤ 1 ≤ max_allowed_nonce`. [6](#0-5) 
   - `validate_by_mempool` passes: nonce 1 is a valid future nonce for account nonce 0.
   - `skip_stateful_validations` evaluates: `tx_nonce == 1` ✓, `account_nonce == 0` ✓, `account_tx_in_pool_or_recent_block` returns `true` (nonce-0 invoke is in pool) ✓ → returns `true`. [7](#0-6) 
   - `run_validate_entry_point` is called with `validate = false` → `__validate__` is **not executed**.
4. The invalid nonce-1 invoke is admitted to the mempool without signature verification.

The gateway flow that orchestrates these steps: [8](#0-7) 

---

### Impact Explanation

**Impact: High** — Mempool/gateway admission accepts an invalid transaction (one that would fail `__validate__`) before sequencing. The broken invariant is: every transaction admitted to the mempool must have passed account-level signature validation. An attacker can poison the nonce-1 slot of any freshly deployed account, causing the batcher to receive and attempt to execute a transaction that will revert on `__validate__`, wasting sequencer resources and potentially blocking the victim's nonce-1 slot until the invalid transaction is evicted.

---

### Likelihood Explanation

**Likelihood: Medium** — The precondition (account on-chain nonce = 0) is satisfied by every freshly deployed account. The attacker only needs to submit two transactions: one valid nonce-0 invoke (to seed the mempool) and one invalid nonce-1 invoke. No privileged access is required. The window is open for as long as the nonce-0 invoke remains in the mempool or the account's nonce has not yet advanced past 0 on-chain.

---

### Recommendation

Replace the `account_tx_in_pool_or_recent_block` check with a type-specific check that confirms a `deploy_account` transaction is present in the mempool for the account, mirroring the `PyValidator` pattern:

```rust
// PyValidator (native_blockifier) correctly requires an explicit deploy_account hash:
let deploy_account_not_processed = deploy_account_tx_hash.is_some() && nonce == Nonce(Felt::ZERO);
```

In the Apollo gateway, the mempool client should expose a method such as `has_deploy_account_in_pool(address)` that filters by transaction type, or the gateway should pass the `deploy_account` transaction hash alongside the invoke when the caller submits both simultaneously, and verify it explicitly before skipping validation.

---

### Proof of Concept

```
1. Deploy account contract at address X (on-chain nonce[X] = 0).

2. Submit valid invoke_v3 for X, nonce=0, valid signature.
   → Gateway: validate_nonce(0 ≤ 0 ≤ max) ✓
   → Gateway: validate_by_mempool ✓
   → Gateway: skip_stateful_validations → false (tx_nonce=0, not 1)
   → Gateway: run_validate_entry_point(validate=true) → __validate__ passes ✓
   → Mempool: tx[X,nonce=0] inserted.

3. Submit invoke_v3 for X, nonce=1, INVALID signature (e.g., all-zero sig).
   → Gateway: validate_nonce(0 ≤ 1 ≤ max) ✓
   → Gateway: validate_by_mempool(nonce=1, account_nonce=0) ✓ (future nonce accepted)
   → Gateway: skip_stateful_validations:
       tx_nonce == 1 ✓
       account_nonce == 0 ✓
       account_tx_in_pool_or_recent_block(X) == true  ← nonce-0 invoke is in pool ✓
       → returns true
   → Gateway: run_validate_entry_point(validate=false) → __validate__ SKIPPED
   → Mempool: tx[X,nonce=1] inserted WITHOUT signature verification.

Result: Invalid (unsigned) nonce-1 invoke admitted to mempool.
         Batcher will attempt execution → __validate__ fails → revert.
         Victim's nonce-1 slot is occupied by attacker's invalid transaction.
```

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
