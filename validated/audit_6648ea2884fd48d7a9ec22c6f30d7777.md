### Title
Silo Owner Can Frontrun `ft_on_transfer` to Redirect ERC-20 Tokens via Mutable `erc20_fallback_address` - (File: `engine/src/engine.rs`)

---

### Summary

In Silo mode, the `erc20_fallback_address` is a mutable parameter that the owner can change at any time via `set_erc20_fallback_address` or `set_silo_params`. The `receive_erc20_tokens` function reads this address from storage at the moment of execution. A malicious Silo owner can frontrun a user's `ft_transfer_call` by atomically (1) changing the fallback address to their own controlled address, (2) optionally removing the recipient from the whitelist, and (3) allowing the `ft_on_transfer` callback to execute — causing the bridged ERC-20 tokens to be minted to the attacker's address rather than the intended recipient. The user's NEP-141 tokens are permanently consumed with no refund, and no error is surfaced.

---

### Finding Description

In `engine/src/engine.rs`, `receive_erc20_tokens` silently redirects the mint destination when the intended recipient is not whitelisted:

```rust
if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
    && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
{
    recipient = fallback_address;
}
```

The `erc20_fallback_address` is read from live storage at call time. It is not locked or snapshotted at the time the user initiates the bridge transfer. The owner can update it at any moment via the `set_erc20_fallback_address` entrypoint:

```rust
pub extern "C" fn set_erc20_fallback_address() {
    let mut io = Runtime;
    let state = state::get_state(&io).sdk_unwrap();
    require_owner_and_running(&state, &io.predecessor_account_id())
        .map_err(ContractError::msg)
        .sdk_unwrap();
    let args: Erc20FallbackAddressArgs = io.read_input_borsh().sdk_unwrap();
    silo::set_erc20_fallback_address(&mut io, args.address);
}
```

The owner can also manipulate the whitelist via `remove_entry_from_whitelist` to force any recipient address into the "not allowed" branch, even if it was previously whitelisted.

**Attack path:**

1. User calls `ft_transfer_call` on a NEP-141 contract, specifying their EVM address as the recipient in the `msg` field. This creates a cross-contract call that will invoke `ft_on_transfer` on the Aurora Engine.
2. The Silo owner observes the pending `ft_transfer_call` in the NEAR mempool.
3. Before the `ft_on_transfer` callback executes, the owner submits two transactions in the same block:
   - `remove_entry_from_whitelist` to remove the user's recipient address from the `Address` whitelist.
   - `set_erc20_fallback_address` to set the fallback address to an attacker-controlled EVM address.
4. The `ft_on_transfer` callback executes. `receive_erc20_tokens` reads the updated fallback address, finds the recipient is not whitelisted, and mints the full token amount to the attacker's address.
5. The owner restores the original whitelist entry and fallback address.

The user's NEP-141 tokens are fully consumed. The return value from `ft_on_transfer` is `"0"` (success), so the NEP-141 contract does not refund the sender. No error is emitted. The attack is entirely silent.

A second, simpler vector requires no whitelist manipulation: if the user is already sending to a non-whitelisted address (the normal Silo use-case for the fallback mechanism), the owner only needs to swap the `erc20_fallback_address` to their own address before the callback executes, then restore it afterward.

---

### Impact Explanation

**Critical — Direct theft of user funds.**

The user's NEP-141 tokens are permanently transferred to the Aurora Engine contract and the ERC-20 mint is directed to the attacker's address. The user receives nothing. Because `ft_on_transfer` returns `"0"` (no refund), the NEP-141 contract considers the transfer complete and does not return tokens to the sender. The stolen ERC-20 tokens are fully liquid and can be immediately exited back to NEAR via the `exitToNear` precompile.

---

### Likelihood Explanation

**Medium.** The Silo owner is a semi-trusted privileged party with legitimate access to `set_erc20_fallback_address` and whitelist management. The attack requires no key compromise — only the owner acting maliciously. NEAR's receipt ordering within a block is deterministic and predictable, making it feasible to sequence the owner's state-change transactions ahead of the `ft_on_transfer` callback receipt. The attack is completely silent (no revert, no error log visible to the user), making it the hardest class of theft to detect — directly analogous to the "affiliate root swap" case in the reference report, which was flagged as the most concerning scenario precisely because it leaves no on-chain signal.

---

### Recommendation

The `erc20_fallback_address` and the whitelist state should be treated as immutable for the duration of an in-flight `ft_transfer_call` / `ft_on_transfer` sequence. Concretely:

1. **Commit the fallback address at transfer initiation**: Record the fallback address in the `ft_on_transfer` message or derive it deterministically from immutable state, rather than reading it from mutable storage at callback time.
2. **Revert instead of silently redirecting**: If the intended recipient is not whitelisted, return the full token amount to the sender (return `args.amount` instead of `0`) rather than silently minting to the fallback address. This eliminates the silent-theft vector entirely.
3. **Separate whitelist and fallback changes from active bridge flows**: Introduce a time-lock or a two-step commit-reveal for changes to `erc20_fallback_address` and whitelist removals, so users can observe and react to parameter changes before they affect in-flight transfers.

### Proof of Concept

**Relevant code path:**

`receive_erc20_tokens` reads the fallback address from live storage at execution time: [1](#0-0) 

`set_erc20_fallback_address` allows the owner to update this address at any time with no delay: [2](#0-1) 

The underlying storage write is immediate and unconditional: [3](#0-2) 

The `SiloParamsArgs` struct confirms `erc20_fallback_address` is a plain mutable field with no access controls beyond owner-only: [4](#0-3) 

**Scenario (non-whitelisted recipient, simplest case):**

```
Block N:
  Receipt 1: ft_transfer_call(nep141, aurora, amount, msg="<victim_address>")
    → schedules ft_on_transfer callback as Receipt 2

  Receipt 2 (owner): set_erc20_fallback_address(attacker_address)
    → erc20_fallback_address = attacker_address

  Receipt 3 (ft_on_transfer callback):
    receive_erc20_tokens reads erc20_fallback_address = attacker_address
    victim_address not in whitelist → recipient = attacker_address
    ERC-20 tokens minted to attacker_address
    returns "0" → NEP-141 does NOT refund sender

  Receipt 4 (owner): set_erc20_fallback_address(original_address)
    → state restored, attack invisible
```

The user loses their NEP-141 tokens. The attacker gains the equivalent ERC-20 balance. No error is surfaced anywhere in the call chain.

```rust
/// Set ERC-20 fallback address.
pub fn set_erc20_fallback_address<I: IO>(io: &mut I, address: Option<Address>) {
    let key = erc20_fallback_address_key();

    if let Some(address) = address {
        io.write_storage(&key, address.as_bytes());
    } else {
        io.remove_storage(&key);
    }
}
```
