### Title
EIP-7702 Authorization List `is_empty` Check Reads Mutable In-Transaction State, Enabling Gas Refund Inflation — (`basic_bootloader/src/bootloader/transaction/authorization_list.rs`)

---

### Summary

In `validate_and_apply_delegation`, step 7 conditionally adds a gas refund to the refund counter when the authority account is **not empty**. The emptiness check reads the authority's nonce from the IO subsystem's live cache, which already reflects nonce increments applied by earlier entries in the same authorization list. An attacker who controls a fresh EOA can include it twice in the same EIP-7702 authorization list (with consecutive nonces 0 and 1), causing the second entry to observe a non-empty account and collect a 22 500-gas refund that the account was never entitled to.

---

### Finding Description

`validate_and_apply_delegation` implements the nine-step EIP-7702 authority-processing algorithm. Step 7 reads the current account state and grants a refund when the account is non-empty:

```rust
// 7. Add refund if authority is not empty.
let is_empty = account_properties.nonce.0 == 0
    && account_properties.has_bytecode() == false
    && account_properties.nominal_token_balance.0.is_zero();

if !is_empty {
    let ergs = Ergs(
        (evm_interpreter::gas_constants::NEWACCOUNT
            - evm_interpreter::gas_constants::PER_AUTH_BASE_COST)
            * ERGS_PER_GAS,
    );
    system
        .io
        .add_to_refund_counter(S::Resources::from_ergs(ergs))?
}
``` [1](#0-0) 

Step 9 of the same function then increments the authority's nonce in the live IO cache:

```rust
// 9. Bump nonce
resources.with_infinite_ergs(|inf_ergs| {
    system.io.increment_nonce(ExecutionEnvironmentType::NoEE, inf_ergs, &authority, 1)
})
``` [2](#0-1) 

The outer loop in `parse_authorization_list_and_apply_delegations` processes every entry sequentially with no deduplication guard:

```rust
for entry in auth_list.iter() {
    let success = validate_and_apply_delegation(
        system, resources, &chain_id, nonce, address, (y_parity, r, s), &mut hasher,
    )?;
}
``` [3](#0-2) 

Because `read_account_properties` in step 4 reads from the same mutable IO cache that step 9 writes to, the nonce bump performed by entry N is immediately visible to entry N+1. The `is_empty` predicate therefore reflects **in-transaction mutations**, not the pre-transaction account state that the EIP-7702 refund rule was designed to test.

**Attack scenario:**

| Step | Action | Authority A state | Refund counter |
|------|--------|-------------------|----------------|
| Entry 1 processed | auth_nonce=0, A.nonce=0 → `is_empty=true` → no refund; nonce bumped to 1 | nonce=1 | 0 |
| Entry 2 processed | auth_nonce=1, A.nonce=1 → `is_empty=false` → refund of 22 500 gas added; nonce bumped to 2 | nonce=2 | 22 500 gas |

The attacker can chain N entries for the same authority (nonces 0…N-1) to accumulate `(N-1) × 22 500` gas in the refund counter, subject only to the EIP-3529 cap applied later in `compute_gas_refund`:

```rust
// Following EIP-3529, refunds are capped to 1/5 of the gas used
let evm_refund = {
    let full_refund_ergs = system.io.get_refund_counter().ergs();
    let full_refund_gas = full_refund_ergs.0.div_floor(ERGS_PER_GAS);
    let max_refund = gas_used / 5;
    core::cmp::min(full_refund_gas, max_refund)
};
``` [4](#0-3) 

---

### Impact Explanation

The attacker can inflate the EVM refund counter to its maximum permitted value (`gas_used / 5`) without performing any SSTORE operations. For a transaction that spends 50 000 gas on two authorization entries, the attacker recovers 10 000 gas (20 % of total cost) that they are not entitled to. At scale (many entries, high gas limit), the attacker can systematically under-pay for every EIP-7702 transaction they submit, constituting a resource-accounting bypass that distorts fee revenue and block-limit accounting.

---

### Likelihood Explanation

The attack requires only a fresh EOA (nonce 0, zero balance, no code) and the ability to craft an EIP-7702 transaction — both trivially available to any unprivileged user. No privileged role, oracle manipulation, or external dependency is needed. The `eip-7702` feature flag must be enabled, but it is present in the codebase and exercised by existing tests.

---

### Recommendation

The `is_empty` check should be evaluated against the **pre-transaction** account state, not the live cache. Two concrete options:

1. **Snapshot the initial emptiness flag** — when an authority account is first loaded (step 4), record whether it was empty at that point and store the flag alongside the authority address for the duration of authorization-list processing.
