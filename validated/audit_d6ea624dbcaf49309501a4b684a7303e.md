### Title
Pool Admin Bypasses `maxAdminSpreadFeeE6` Cap via Uncapped Per-Bin Additional Fees — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`setPoolAdminFees` enforces `maxAdminSpreadFeeE6` on the global spread component, but the parallel `setPoolBinAdditionalFees` path sets per-bin `addFeeBuyE6` / `addFeeSellE6` with **no cap validation at all**. A pool admin can set both channels simultaneously, charging traders a total effective spread that exceeds the protocol-enforced ceiling.

---

### Finding Description

The factory enforces a hard ceiling on the admin's global spread fee:

```solidity
// MetricOmmPoolFactory.sol – setPoolAdminFees
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
``` [1](#0-0) 

The same admin also controls `setPoolBinAdditionalFees`, which forwards directly to the pool with **no cap check**:

```solidity
// MetricOmmPoolFactory.sol – setPoolBinAdditionalFees
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
``` [2](#0-1) 

The pool's `setBinAdditionalFees` also performs no cap check:

```solidity
// MetricOmmPool.sol – setBinAdditionalFees
s.addFeeBuyE6 = addFeeBuyE6;   // uint16, max 65 535 — no validation
s.addFeeSellE6 = addFeeSellE6;
``` [3](#0-2) 

During every swap the per-bin additional fee is **added on top of** the oracle-derived base fee before the trade price is computed:

```solidity
// MetricOmmPool.sol – swap inner loop (buy direction)
params.baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6)
``` [4](#0-3) 

The same pattern appears for all four swap variants (specified-in / specified-out, buy / sell). [5](#0-4) [6](#0-5) [7](#0-6) 

The same uncapped path exists at pool creation: `_unpackAndValidateBinStates` only validates `lengthE6` for distance bounds; `addFeeBuyE6` and `addFeeSellE6` packed into the bin data arrays are never range-checked. [8](#0-7) 

---

### Impact Explanation

`addFeeBuyE6` and `addFeeSellE6` are `uint16`, so the maximum uncapped additional fee is **65 535 / 1 000 000 ≈ 6.55 %** per bin. A pool admin who has already set the global admin spread to the hard cap (`HARD_MAX_SPREAD_FEE_E6 = 200 000`, i.e. 20 %) can additionally set `addFeeBuyE6 = addFeeSellE6 = 65535` on every active bin, raising the effective fee charged to traders to **≈ 26.55 %** (plus the oracle-derived base spread). The excess fee accrues inside the bin's token balance and is claimable by LPs — including the admin if they hold LP shares. Traders receive less output than the protocol's fee ceiling is supposed to guarantee, constituting a direct loss of user principal on every swap routed through the affected bin.

---

### Likelihood Explanation

The pool admin is a semi-trusted role that is explicitly bounded by caps. The bypass requires only two sequential, permissioned calls (`setPoolAdminFees` at the cap, then `setPoolBinAdditionalFees` with `uint16` max on each bin). No special market conditions, flash loans, or external dependencies are needed. Any pool admin who is also an LP, or who is acting adversarially, can execute this immediately after pool creation.

---

### Recommendation

Add a combined cap check in `setPoolBinAdditionalFees` (and symmetrically in `_unpackAndValidateBinStates` at creation time). The simplest fix is to enforce that `addFeeBuyE6` and `addFeeSellE6` each do not exceed `maxAdminSpreadFeeE6` (or a dedicated per-bin cap constant), and that the sum of the global admin spread fee and the per-bin additional fee does not exceed `HARD_MAX_SPREAD_FEE_E6`:

```solidity
// In setPoolBinAdditionalFees (factory) and _unpackAndValidateBinStates:
if (uint256(addFeeBuyE6) + poolFeeConfig[pool].adminSpreadFeeE6 > maxAdminSpreadFeeE6)
    revert AdminFeeTooHigh();
if (uint256(addFeeSellE6) + poolFeeConfig[pool].adminSpreadFeeE6 > maxAdminSpreadFeeE6)
    revert AdminFeeTooHigh();
```

Alternatively, introduce a dedicated `maxBinAdditionalFeeE6` constant and validate against it in both paths.

---

### Proof of Concept

```solidity
// 1. Pool admin sets global admin spread to the hard cap (20 %)
factory.setPoolAdminFees(pool, 200_000, 0);

// 2. Pool admin sets per-bin additional fees to uint16 max on the active bin
//    (no revert — no cap check exists)
factory.setPoolBinAdditionalFees(pool, curBinIdx, 65_535, 65_535);

// 3. A trader calls swap(); the effective fee applied is:
//    baseFeeX64  (oracle spread)
//  + 65_535 / 1e6  (≈ 6.55 % additional, uncapped)
//    → trader pays ~6.55 % more than the protocol ceiling permits
//    → excess accrues in bin balance, claimable by LPs (including admin)
```

The `setPoolBinAdditionalFees` call in step 2 succeeds because neither `MetricOmmPoolFactory.setPoolBinAdditionalFees` nor `MetricOmmPool.setBinAdditionalFees` contains any range check on `addFeeBuyE6` / `addFeeSellE6`. [2](#0-1) [9](#0-8)
