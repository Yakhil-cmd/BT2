### Title
Incorrect Multi-NFT Royalty Calculation in WalletConnect Offer Confirmation Understates Total Spend Amount - (File: `packages/gui/src/electron/commands/parseCommandDisplay.ts`)

### Summary
The `formatAmountWithRoyalties` function in `parseCommandDisplay.ts` uses a mathematically incorrect formula when computing the total royalty cost for offers involving multiple NFTs. It divides the base fungible amount by the number of NFTs before applying each NFT's royalty rate, producing a result that is systematically lower than the true total. This value is displayed to the user in the WalletConnect confirmation dialog for `chia_wallet.take_offer` and `chia_wallet.create_offer_for_ids`. A malicious dapp can craft an offer with multiple NFTs carrying different royalty percentages to cause the confirmation dialog to show a materially understated total spend, inducing the user to approve a transaction for a higher amount than they consented to.

### Finding Description

In `packages/gui/src/electron/commands/parseCommandDisplay.ts`, the `formatAmountWithRoyalties` function computes the royalty-inclusive total as follows:

```typescript
const splitAmount = amount / BigInt(royaltyPercentages.length);
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
const totalAmount = amount + royaltyAmount;
``` [1](#0-0) 

The variable `splitAmount` is `amount / royaltyPercentages.length`. Each NFT's royalty is then computed on this fractional amount rather than on the full `amount`. The correct formula is to apply each NFT's royalty rate to the full fungible amount and sum the results:

```
royaltyAmount = Σ (amount × royaltyPercentage_i / 10_000)
```

The buggy formula instead computes:

```
royaltyAmount = Σ ((amount / N) × royaltyPercentage_i / 10_000)
```

where N is the number of NFTs. This underestimates the true royalty by a factor of N when all royalty rates are equal, and produces an arbitrary underestimate when rates differ.

This computed `amountWithRoyalties` is returned as part of the `DisplayWalletDelta` object from `walletDeltaToDisplay`, which is passed directly into the WalletConnect `Confirm` dialog shown to the user before they approve the offer. [2](#0-1) 

The `parseCommandDisplay` function is called from the Electron main process at line 329 of `main.tsx` to populate the confirmation dialog for every WalletConnect command requiring user approval. [3](#0-2) 

The royalty percentages fed into this function come from `offerSummaryRoyaltyPercentages`, which reads them from the offer summary's `infos` field — data that originates from the offer blob provided by the dapp. [4](#0-3) 

### Impact Explanation

When a user accepts a WalletConnect `chia_takeOffer` request for an offer containing multiple NFTs with royalties, the confirmation dialog displays an `amountWithRoyalties` value that is lower than the actual amount the offer spend bundle will deduct from the user's wallet. The user reads the understated total, believes the cost is acceptable, and clicks "Accept." The actual spend — enforced by the offer blob and the blockchain — is higher than what was shown. This constitutes a user approving the wrong spend amount, satisfying the High impact criterion: *"causes a user to approve... the wrong... amount."*

### Likelihood Explanation

Any dapp connected to the user's wallet via WalletConnect can send a `chia_takeOffer` request. The dapp controls the offer string, which determines the number of NFTs and their embedded royalty percentages in the `infos` field. No special privilege is required beyond an established WalletConnect pairing, which is a normal user action. The discrepancy grows with the number of NFTs and the magnitude of their royalty rates, so a targeted offer can produce a large gap between displayed and actual cost.

### Recommendation

Replace the `splitAmount` division with a per-NFT computation on the full `amount`:

```typescript
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (amount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
```

Additionally, cross-validate the royalty percentages extracted from the offer's `infos` field against the on-chain NFT data (via `nftGetInfo`) before displaying them, so a crafted driver dict cannot produce a misleading confirmation screen.

### Proof of Concept

**Setup:** A malicious dapp is paired with the user's wallet via WalletConnect.

**Crafted offer:** The dapp sends `chia_takeOffer` with an offer blob containing:
- Requested: 2,000,000,000,000 mojos (2 XCH) from the user
- Offered: NFT-A (royalty 500 = 5%) and NFT-B (royalty 1,000 = 10%)

**Buggy display calculation:**
- `royaltyPercentages = [500, 1000]`
- `splitAmount = 2,000,000,000,000 / 2 = 1,000,000,000,000`
- `royaltyAmount = (1,000,000,000,000 × 500 / 10,000) + (1,000,000,000,000 × 1,000 / 10,000)`
- `= 50,000,000,000 + 100,000,000,000 = 150,000,000,000` mojos (0.15 XCH)
- **Displayed total: 2.15 XCH**

**Correct calculation:**
- `royaltyAmount = (2,000,000,000,000 × 500 / 10,000) + (2,000,000,000,000 × 1,000 / 10,000)`
- `= 100,000,000,000 + 200,000,000,000 = 300,000,000,000` mojos (0.30 XCH)
- **Actual total: 2.30 XCH**

The user sees **2.15 XCH** in the WalletConnect confirmation dialog and approves. The offer spend bundle deducts **2.30 XCH** from their wallet — 0.15 XCH more than they consented to. The gap scales with the number of NFTs and their royalty rates. [5](#0-4)

```typescript
function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],
): string | undefined {
  if (royaltyPercentages.length === 0 || line.kind === 'nft') {
    return undefined;
  }

  const splitAmount = amount / BigInt(royaltyPercentages.length);
  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;

  if (line.kind === 'xch') {
    return mojoToChiaLocaleString(totalAmount);
  }

  return mojoToCATLocaleString(totalAmount);
}
