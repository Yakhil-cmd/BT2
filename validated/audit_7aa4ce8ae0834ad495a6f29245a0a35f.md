### Title
Missing Caller Authorization in `update_rewards` Allows Any Address to Suppress Block Rewards — (`File: src/staking/staking.cairo`)

### Summary
`update_rewards` is documented as restricted to "Only starkware sequencer" but has no caller check in the implementation. Any unprivileged address can call it with `disable_rewards=true`, consuming the global `last_reward_block` slot for the current block and permanently discarding that block's consensus rewards.

### Finding Description
`StakingRewardsManagerImpl::update_rewards` calls only `self.general_prerequisites()`, which asserts the contract is unpaused and the caller is non-zero. No role check, no sequencer check, and no allowlist check is performed. The function then unconditionally writes `self.last_reward_block.write(current_block_number)` before the `disable_rewards` branch, meaning even a call that distributes zero rewards consumes the per-block slot. Because `last_reward_block` is a single global value, only one call per block can succeed. An attacker who calls first with `disable_rewards=true` for any valid active staker permanently prevents the sequencer from distributing rewards for that block.

### Impact Explanation
Each suppressed block permanently loses its consensus block reward. Repeated across many blocks, this constitutes permanent freezing of unclaimed yield for all stakers. The rewards are never minted/distributed and cannot be recovered retroactively.

**Impact: High — Permanent freezing of unclaimed yield.**

### Likelihood Explanation
The call is fully public, requires no stake, no token approval, and no privileged key. The attacker only needs to know any valid active staker address (publicly readable from `get_stakers`) and submit a transaction before the sequencer in each target block. On Starknet, transaction ordering within a block is controlled by the sequencer, but the sequencer is the same entity that is supposed to call `update_rewards` — meaning a third-party attacker racing the sequencer is the realistic threat model. The cost to the attacker is only gas per block suppressed.

### Recommendation
Add an explicit caller check enforcing that only the authorized sequencer address (or a designated role) can call `update_rewards`, consistent with the spec's stated access control. For example:

```cairo
fn update_rewards(...) {
    self.general_prerequisites();
    assert!(
        get_caller_address() == self.sequencer_address.read(),
        "{}",
        Error::CALLER_IS_NOT_SEQUENCER,
    );
    ...
}
```

### Proof of Concept
1. Deploy with two active stakers, `staker_A` and `staker_B`, both past the K-epoch activation window with consensus rewards active.
2. In block N, before the sequencer acts, call `update_rewards(staker_address: staker_A, disable_rewards: true)` from any unprivileged address.
3. Observe: `last_reward_block` is now N; no rewards credited to `staker_A`.
4. The sequencer's intended call `update_rewards(staker_address: staker_B, disable_rewards: false)` reverts with `REWARDS_ALREADY_UPDATED`.
5. Advance to block N+1 and repeat. After M blocks, both stakers have received zero consensus rewards despite being eligible, confirming permanent yield suppression.
