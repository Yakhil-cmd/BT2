### Title
Pool Admin Bypasses Spread-Fee Cap via Unchecked `addFeeBuyE6`/`addFeeSellE6` in `setPoolBinAdditionalFees` — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`setPoolAdminFees` enforces explicit caps on pool-level admin fees, but `setPoolBinAdditionalFees` applies no cap whatsoever on the per-bin `addFeeBuyE6`/`addFeeSellE6` values. A pool admin can set bin-level fees to `uint16.max` (65 535 = 6.5535% in E6 units) on every bin, exceeding the protocol's hard-coded spread-fee ceiling and charging traders more than the system is designed to permit.

---

### Finding Description

The factory enforces a hard cap of `HARD_MAX_SPREAD_FEE_E6 = 200_000` (20%) on all spread fees and validates pool-level admin fees against `maxAdminSpreadFeeE6` in `setPoolAdminFees`: [1](#0-0) 

```solidity
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (newAdminNotionalFeeE8 > maxAdminNotionalFeeE8) revert AdminFeeTooHigh();
```

However, the sibling function `setPoolBinAdditionalFees` passes the values straight through to the pool with **no cap check**: [2](#0-1) 

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

The pool itself also performs no cap validation: [3](#0-2) 

```solidity
function setBinAdditionalFees(int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external onlyFactory nonReentrant(PoolActions.SET_BIN_ADDITIONAL_FEES)
{
    if (bin < LOWEST_BIN || bin > HIGHEST_BIN) revert InvalidBinIndex(bin);
    BinState storage s = _binStates[bin];
    s.addFeeBuyE6 = addFeeBuyE6;
    s.addFeeSellE6 = addFeeSellE6;
    ...
}
```

These bin-level fees are then **added on top of** the pool-level spread fee in every swap: [4](#0-3) 

```solidity
uint256 buyFeeX64 = baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6);
uint256 sellFeeX64 = baseFeeX64 + Math.mulDiv(binState.addFeeSellE6, ONE_X64, 1e6);
```

And in the swap execution path: [5](#0-4) 

```solidity
params.baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6),
```

**Structural analog to the external bug:** In the Firedancer report, `FD_KEYGUARD_ROLE_VOTER` skips the `sz > FD_KEYGUARD_SIGN_REQ_MTU` guard that `FD_KEYGUARD_ROLE_GOSSIP` and `FD_KEYGUARD_ROLE_REPAIR` apply. Here, `setPoolBinAdditionalFees` skips the `> maxAdminSpreadFeeE6` guard that `setPoolAdminFees` applies — same missing-check pattern, different domain.

---

### Impact Explanation

- `uint16.max = 65 535` in E6 units = **6.5535%** additional fee per bin.
- The pool admin can call `setPoolBinAdditionalFees` on every bin in the pool (up to 256 bins) with `addFeeBuyE6 = addFeeSellE6 = 65535`, instantly and without a timelock.
- Combined with the pool-level spread fee (up to 20%), the effective per-trade fee reaches **~26.5%**, far above the `HARD_MAX_SPREAD_FEE_E6 = 200_000` ceiling.
- Traders receive worse execution prices than the protocol's fee cap system is designed to guarantee — a bad-price execution and admin-boundary break.
- The bin-level fees accrue to LPs (not the admin directly), so this is not admin self-enrichment, but it is an uncapped extraction from traders that the protocol explicitly intends to bound.

---

### Likelihood Explanation

- The pool admin is a **semi-trusted** role that can call `setPoolBinAdditionalFees` at any time with no timelock.
- No on-chain guard prevents setting `addFeeBuyE6 = addFeeSellE6 = type(uint16).max`.
- Any pool admin who wishes to extract maximum fees from traders can do so immediately after pool creation.

---

### Recommendation

Add a cap check in `setPoolBinAdditionalFees` analogous to the one in `setPoolAdminFees`. Define a `maxAdminBinFeeE6` constant (or reuse `maxAdminSpreadFeeE6`) and enforce it:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (addFeeBuyE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

Alternatively, enforce the cap inside `MetricOmmPool.setBinAdditionalFees` by reading the factory's cap value, keeping the invariant close to the state it protects.

---

### Proof of Concept

1. Deploy a pool via `createPool` with `adminSpreadFeeE6 = maxAdminSpreadFeeE6` (at the cap).
2. As pool admin, call `setPoolBinAdditionalFees(pool, 0, 65535, 65535)`.
3. No revert occurs. `_binStates[0].addFeeBuyE6 = 65535`.
4. A trader calls `swap(...)` routing through bin 0. The effective buy fee is `baseFeeX64 + mulDiv(65535, ONE_X64, 1e6)` — 6.5535% above the oracle mid-price spread, on top of the already-capped pool spread fee.
5. The trader pays ~26.5% total spread, exceeding the `HARD_MAX_SPREAD_FEE_E6 = 200_000` (20%) ceiling the protocol enforces everywhere else. [6](#0-5) [2](#0-1) [3](#0-2)
