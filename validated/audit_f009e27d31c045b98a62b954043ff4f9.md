Audit Report

## Title
Unbounded Correlated SQLite Subquery via `aggregate_all_subaccounts` Causes Rosetta Instance DoS — (`rs/rosetta-api/icrc1/src/common/storage/storage_operations/mod.rs`)

## Summary
An unauthenticated HTTP client can send a single `POST /account/balance` request with `metadata.aggregate_all_subaccounts=true` targeting a principal with a large number of subaccounts. This triggers an unbounded correlated SQL scan with no timeout, no row limit, and no rate limiting. Because `tokio_rusqlite` serializes all SQLite calls through a single thread, a long-running query blocks all concurrent DB operations for the entire Rosetta instance.

## Finding Description
The guard in `account_balance_with_metadata` (`services.rs` L269–278) only rejects requests where a **non-default** subaccount is explicitly provided alongside `aggregate_all_subaccounts=true`. A request with no subaccount field, or with the default `[0u8; 32]`, passes through unconditionally to the aggregation path at L281–287.

The aggregation path calls `get_aggregated_balance_for_principal_at_block_idx` (`storage_operations/mod.rs` L895–941), which executes a correlated subquery:

```sql
SELECT a1.subaccount, a1.amount
FROM account_balances a1
WHERE a1.principal = :principal
AND a1.block_idx = (
    SELECT MAX(a2.block_idx)
    FROM account_balances a2
    WHERE a2.principal = a1.principal
    AND a2.subaccount = a1.subaccount
    AND a2.block_idx <= :block_idx
)
```

The outer scan iterates every row in `account_balances` for the given principal. For each candidate row, SQLite evaluates the inner `SELECT MAX(block_idx)` correlated subquery. The schema (`schema.rs` L56–67) defines the PRIMARY KEY as `(principal, subaccount, block_idx)` and adds only a secondary index on `block_idx` alone — there is no covering index that would allow the outer query to skip non-latest rows without evaluating the inner subquery. The result set is unbounded: no `LIMIT`, no row-count cap, no query timeout, and no per-request rate limiting exist anywhere on this code path.

The `storage_client.rs` uses `tokio_rusqlite::Connection` (`storage_client.rs` L11), which dispatches all SQLite work to a single background thread. A long-running query on that thread blocks every other concurrent DB call (balance lookups, block fetches, transaction queries) for the entire Rosetta instance until the query completes.

## Impact Explanation
This is a **High** severity finding matching: *"Application/platform-level DoS … or subnet availability impact not based on raw volumetric DDoS"* and *"Significant … Rosetta … security impact with concrete user or protocol harm."* The Rosetta API is explicitly in scope. A single unauthenticated HTTP request can render the Rosetta instance unresponsive for all clients for an extended period, disrupting balance queries, block fetches, and transaction lookups.

## Likelihood Explanation
The `/account/balance` endpoint requires no authentication. Large ICRC-1 ledgers (ckBTC, ckETH) have principals — exchanges, custodians, DEX contracts — that legitimately accumulate O(10^5–10^6) subaccounts. The attacker does not need to create those subaccounts; they only need to identify such a principal (publicly observable on-chain) and issue the request. Repeated requests amplify the effect since each new request re-queues behind the still-running prior query on the serialized SQLite thread.

## Recommendation
1. Add a configurable cap in `get_aggregated_balance_for_principal_at_block_idx` (e.g., `LIMIT 10000`) to bound the number of subaccounts summed per request.
2. Rewrite the correlated subquery as a non-correlated `GROUP BY (principal, subaccount) HAVING block_idx = MAX(block_idx) WHERE block_idx <= :block_idx` form to eliminate per-row inner scans.
3. Add a SQLite `pragma busy_timeout` and a per-query progress-handler interrupt to enforce a wall-clock deadline.
4. Add rate limiting or concurrency limiting at the HTTP handler level for the `aggregate_all_subaccounts` path.
5. Add a composite index on `(principal, subaccount, block_idx DESC)` to allow the query planner to seek directly to the latest row per subaccount without a full scan.

## Proof of Concept
1. On any ICRC-1 ledger tracked by the Rosetta instance, ensure a principal `P` has ≥10^5 distinct subaccounts with balance entries in `account_balances`.
2. Send: `POST /account/balance` with body `{"network_identifier":…,"account_identifier":{"address":"<P_text>"},"metadata":{"aggregate_all_subaccounts":true}}`.
3. Concurrently send a second `POST /account/balance` for any other account.
4. Observe that the second request stalls until the first completes, confirming the single-threaded SQLite serialization bottleneck.
5. A deterministic integration test can reproduce this by inserting 10^5 `account_balances` rows for a single principal across distinct subaccounts and measuring response latency for a concurrent request.
