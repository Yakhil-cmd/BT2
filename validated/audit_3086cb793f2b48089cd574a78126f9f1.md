### Title
Double Storage-Cost Charge on Re-deployment of the Same Global Contract by CodeHash — (`runtime/runtime/src/global_contracts.rs`)

---

### Summary

`action_deploy_global_contract` unconditionally deducts `storage_cost` from `account.amount` on every invocation, with no check for whether the `TrieKey::GlobalContractCode` entry for the given code hash already exists. For `CodeHash` mode, the identifier is deterministically `hash(code)`, so re-deploying identical bytes always maps to the same trie key. The nonce mechanism that guards distribution receipts against stale overwrites does not issue any refund when a receipt is discarded as stale. The result is that `account.amount` is permanently reduced by `storage_cost` on every re-deployment of the same code, while the on-chain storage entry is written at most once.

---

### Finding Description

**Step 1 — First deployment**

`action_deploy_global_contract` computes:

```
storage_cost = global_contract_storage_amount_per_byte × code.len()
```

and unconditionally subtracts it from `account.amount`, then calls `initiate_distribution`. [1](#0-0) 

Inside `initiate_distribution`, for `CodeHash` mode the identifier is `GlobalContractIdentifier::CodeHash(hash(code))`. `increment_nonce` reads the stored nonce (0), writes 1, and embeds nonce=1 in the outgoing `GlobalContractDistributionReceipt`. [2](#0-1) [3](#0-2) 

**Step 2 — Second deployment (identical bytes)**

`action_deploy_global_contract` is called again. There is no guard that checks whether `TrieKey::GlobalContractCode { identifier }` already exists. `storage_cost` is computed identically and subtracted from `account.amount` a second time. [1](#0-0) 

`initiate_distribution` runs again for the same `CodeHash` identifier. `increment_nonce` reads stored nonce=1, writes 2, and embeds nonce=2 in a second `GlobalContractDistributionReceipt`. [4](#0-3) 

**Step 3 — Distribution receipt application**

When the first receipt (nonce=1) reaches `apply_distribution_current_shard`, `check_and_update_nonce` reads stored_nonce=2 and finds `incoming_nonce (1) < stored_nonce (2)` → returns `false` → the function returns `Ok(0)` with no code write and **no refund**. [5](#0-4) [6](#0-5) 

When the second receipt (nonce=2) arrives, `incoming_nonce (2) >= stored_nonce (2)` → fresh → code is written to `TrieKey::GlobalContractCode`. [7](#0-6) 

**Net ledger state:**
- `TrieKey::GlobalContractCode` written once (correct).
- `account.amount` reduced by `2 × storage_cost` (incorrect — one `storage_cost` unit is permanently burnt with no corresponding storage benefit).
- `result.tokens_burnt` accumulated `storage_cost` twice, so the over-charge is not refunded via the gas-refund path either. [8](#0-7) 

---

### Impact Explanation

The exact corrupted value is `account.amount`, which is `storage_cost` lower than the correct post-state after the second (and every subsequent) re-deployment of the same code hash. At the production rate of `0.0001 NEAR/byte`, a 4 MiB contract costs ≈ 419 NEAR per redundant deployment. The loss is permanent (tokens are burnt, not transferred), and the deployer receives no storage benefit. This breaks the ledger invariant that `account.amount` decreases only in proportion to actual new storage consumed or gas purchased. [9](#0-8) 

---

### Likelihood Explanation

Any unprivileged user who holds a full-access key can submit a `DeployGlobalContractAction` with `deploy_mode = CodeHash` and the same code bytes in two separate transactions. No validator or operator privilege is required. The path is reachable via a standard signed transaction. The bug is triggered on every re-deployment of the same code hash, making it repeatable.

---

### Recommendation

Before charging `storage_cost` in `action_deploy_global_contract`, check whether `TrieKey::GlobalContractCode { identifier }` already exists in `state_update`. For `CodeHash` mode, if the entry is already present, skip the storage charge (the code is immutable and the storage is already paid for). For `AccountId` mode, a similar check or a refund-of-old-cost / charge-of-new-cost delta approach is appropriate when the code size changes.
