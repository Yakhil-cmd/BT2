### Title
Proposer Key Weight Not Checked Against Threshold, Allowing Partial-Weight Key to Increment Sequence Numbers - (File: fvm/transactionVerifier.go)

### Summary
`verifyTransaction()` in `fvm/transactionVerifier.go` enforces the key-weight threshold for authorizers (checked against `payloadWeights`) and for the payer (checked against `envelopeWeights`), but applies only a weaker "signature exists" check for the proposer role when the proposer is neither an authorizer nor the payer. This is the direct analog of the reported bug: a neutral/symmetric check is used where a role-specific directional check is required.

### Finding Description
In `verifyTransaction()`, three distinct weight maps are maintained: `payloadWeights` and `envelopeWeights`. After `getAccountKeysAndAggregateWeights()` populates them, the code enforces the threshold for each role:

- **Authorizers** (lines 249–263): checked against `payloadWeights[addr] >= keyWeightThreshold`
- **Payer** (lines 267–274): checked against `envelopeWeights[tx.Payer] >= keyWeightThreshold`
- **Proposer** (lines 314–316 inside `getAccountKeysAndAggregateWeights`): only checked for *existence* of a matching signature (`foundProposalSignature = true`), with no weight threshold enforcement

When the proposer is a standalone role (not also listed as an authorizer and not the payer), its accumulated weight in either `payloadWeights` or `envelopeWeights` is never compared against `keyWeightThreshold`. A key with weight 1 (or any value below the threshold of 1000) satisfies the proposer check.

### Impact Explanation
The proposer's sequence number is incremented on every successful transaction execution. An attacker who holds a partial-weight key (e.g., weight 500 < 1000 threshold) for a victim account can craft a transaction naming that key as the proposer, sign the payload with it, and have the transaction accepted by the FVM. The victim's key sequence number is incremented, causing all legitimate pending transactions from the victim that reference the same key/sequence-number pair to be rejected with `ErrCodeInvalidProposalSeqNumberError`. This is a targeted griefing / denial-of-service against a specific account key's usability.

### Likelihood Explanation
Any unprivileged transaction sender who possesses a partial-weight private key (their own account with a deliberately split key, or a compromised low-weight key from another account) can trigger this path. No admin access, staked node, or quorum compromise is required. The attacker only needs to submit a valid Flow transaction, which is the most basic operation on the network.

### Recommendation
Add an explicit weight check for the proposer role in `verifyTransaction()`, analogous to the existing authorizer and payer checks. After `getAccountKeysAndAggregateWeights()` returns, determine which weight map the proposer signed into (payload or envelope) and verify it meets the threshold:

```go
// proposer must have sufficient weight in whichever map they signed
proposerWeights := payloadWeights
if _, signedEnvelope := envelopeWeights[tx.ProposalKey.Address]; signedEnvelope {
    proposerWeights = envelopeWeights
}
if !v.hasSufficientKeyWeight(proposerWeights, tx.ProposalKey.Address, keyWeightThreshold) {
    return errors.NewInvalidProposalSignatureError(tx.ProposalKey,
        fmt.Errorf("proposer account does not have sufficient signatures"))
}
```

### Proof of Concept

1. Create account `A` with two keys: `key0` (weight 1000, full) and `key1` (weight 500, partial).
2. Craft a transaction:
   - `ProposalKey`: `{Address: A, KeyIndex: 1, SequenceNumber: 0}`
   - `Payer`: attacker's account (full-weight key)
   - `Authorizers`: `[]` (empty, or just the payer)
3. Sign the payload with `key1` (partial weight, private key held by attacker).
4. Sign the envelope with the payer's full-weight key.
5. Submit the transaction.

**Expected (correct) behavior**: FVM rejects the transaction because `key1` weight (500) < threshold (1000).

**Actual behavior**: `verifyTransaction()` finds `foundProposalSignature = true` for `key1`, skips the weight check (no authorizer loop entry for `A`, payer check only covers the payer address), and accepts the transaction. Account `A`'s `key1` sequence number is incremented to 1. Any legitimate transaction from `A` using `key1` with `SequenceNumber: 0` is now permanently invalid.
