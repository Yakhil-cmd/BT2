### Title
Anyone Can Steal Excess Native ETH Stranded in `MetricOmmSimpleRouter` / `MetricOmmPoolLiquidityAdder` — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

---

### Summary

`PeripheryPayments.refundETH()` unconditionally transfers the entire native ETH balance of the router to `msg.sender`. Because the `pay()` helper wraps only the exact amount the pool requests and leaves any surplus `msg.value` sitting in the contract, any caller who races in after a swap that sent excess ETH can drain that surplus.

---

### Finding Description

`PeripheryPayments.pay()` handles the case where the user sends native ETH to cover a WETH-denominated swap: [1](#0-0) 

When `address(this).balance >= value`, it wraps exactly `value` wei and forwards it to the pool, leaving `balance - value` wei stranded in the contract. The user is expected to reclaim this surplus by including `refundETH()` as a second call inside a `multicall`. If they call `exactInputSingle` (or any other payable entry point) directly — or simply omit the `refundETH()` step — the surplus remains.

`refundETH()` has no access control whatsoever: [2](#0-1) 

Any address can call it at any time and receive the full ETH balance. The same surface exists on `MetricOmmPoolLiquidityAdder`, which inherits `PeripheryPayments` and exposes the same `payable` entry points (`addLiquidityExactShares`, `addLiquidityWeighted`, `multicall`). [3](#0-2) 

The `receive()` guard (`if (msg.sender != WETH) revert NotWETH()`) only blocks bare ETH transfers; it does not prevent ETH from entering via `msg.value` on any `payable` function. [4](#0-3) 

---

### Impact Explanation

A user who calls `exactInputSingle{value: X}` where `X` exceeds the actual swap cost loses `X - cost` ETH to the first back-runner who calls `refundETH()`. This is a direct loss of user principal. The same applies to `addLiquidityExactShares{value: X}` and `addLiquidityWeighted{value: X}` on the liquidity adder. The loss is bounded by the excess ETH the victim sent, but there is no floor — a user who sends 1 ETH for a 1000-wei swap loses nearly 1 ETH.

---

### Likelihood Explanation

The intended usage pattern (multicall + refundETH in one transaction) is documented only in test comments: [5](#0-4) 

Users who call the payable swap or liquidity functions directly — without wrapping them in a multicall that ends with `refundETH()` — will strand ETH. Front-end integrators that construct calls incorrectly, or users who interact with the contract directly, are realistic victims. A back-runner only needs to watch the mempool for transactions that leave a non-zero ETH balance in the router.

---

### Recommendation

Change `refundETH()` to accept a `recipient` parameter supplied by the caller, or restrict it so that it can only be invoked as part of a `multicall` (e.g., by checking a transient "multicall active" flag). The simplest safe fix is:

```solidity
function refundETH(address recipient) external payable override {
    uint256 balance = address(this).balance;
    if (balance > 0) {
        _transferETH(recipient, balance);
    }
}
```

This mirrors the pattern already used by `unwrapWETH9(uint256 amountMinimum, address recipient)` and `sweepToken(address token, uint256 amountMinimum, address recipient)`, where the caller explicitly names who receives the funds. [6](#0-5) 

---

### Proof of Concept

```solidity
// Alice swaps WETH→token1 but sends 2 ETH for a 1000-wei swap
vm.prank(alice);
router.exactInputSingle{value: 2 ether}(
    IMetricOmmSimpleRouter.ExactInputSingleParams({
        pool: address(pool),
        tokenIn: address(weth),
        tokenOut: address(token1),
        zeroForOne: true,
        amountIn: 1_000,          // only 1000 wei needed
        amountOutMinimum: 0,
        recipient: alice,
        deadline: block.timestamp + 1,
        priceLimitX64: 0,
        extensionData: ""
    })
);
// ~2 ETH - 1000 wei is now stranded in the router

// Bob (attacker) steals it
uint256 bobBefore = bob.balance;
vm.prank(bob);
router.refundETH();
assertGt(bob.balance - bobBefore, 1.9 ether); // Bob received Alice's surplus
```

The test `test_refundETH_sendsBalanceToCaller` in the existing suite already confirms that any arbitrary `swapper` address can drain the router's ETH balance: [7](#0-6)
