### Title
Attacker-Controlled `sender_address` Bypasses Signature Verification for Invoke Transactions via `skip_stateful_validations` - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

`skip_stateful_validations` in the gateway's stateful validator skips the `__validate__` entry point (which performs signature verification) for any invoke transaction with `nonce=1` when the target account's on-chain nonce is `0` and the account has any transaction in the mempool. Because `sender_address` is a user-supplied field in the RPC transaction, an attacker can craft an invoke transaction claiming any victim's address as sender, satisfy the skip condition by observing the victim's pending `deploy_account` in the mempool, and have the transaction admitted to the mempool without any signature check.

### Finding Description

The `skip_stateful_validations` function checks three conditions to decide whether to skip the `__validate__` entry point:

        .account_tx_in_pool_or_recent_block(tx.sender_address())
        .await

When this returns `true`, `run_validate_entry_point` sets `validate: !skip_validate = false`:

Inside `StatefulValidator::perform_validations`, when `validate = false`, the function returns immediately after `perform_pre_validation_stage` without ever calling `__validate__`:

```rust
if !tx.execution_flags.validate {
    return Ok(());
}
// `__validate__` call.
let (_optional_call_info, actual_cost) = self.validate(&tx, tx_context.clone())?;
``` [3](#0-2) 

The `sender_address` field is entirely user-supplied in `RpcInvokeTransactionV3`. There is no check that the submitter of the transaction is the owner of `sender_address`. The gateway's stateless validator only checks that the address is a valid contract address format, not that the signature covers it: [4](#0-3) 

The `account_tx_in_pool_or_recent_block` check uses the attacker-supplied `sender_address` to query the mempool: [5](#0-4) 

This returns `true` if the victim's address appears in the mempool pool or in recently committed block state — a condition that is publicly observable.

**Attack path:**
1. Victim broadcasts a `deploy_account` transaction (nonce=0). It enters the mempool.
2. Attacker observes the victim's address in the mempool (public information).
3. Attacker crafts an `RpcInvokeTransactionV3` with `sender_address = victim_address`, `nonce = 1`, arbitrary `calldata`, and a garbage `signature`.
4. Gateway stateless validation passes (address format is valid, resource bounds are set).
5. Stateful validation: `account_nonce = 0`, `tx.nonce() = 1` → condition met; `account_tx_in_pool_or_recent_block(victim_address) = true` → `skip_validate = true`.
6. `run_validate_entry_point` is called with `validate = false` → `__validate__` is never invoked → signature is never checked.
7. The transaction is admitted to the mempool.

### Impact Explanation

**High — Mempool/gateway admission accepts invalid transactions before sequencing.**

The gateway's invariant that every admitted transaction has passed signature verification is broken. An attacker can inject unsigned or arbitrarily-signed invoke transactions on behalf of any account that has a pending `deploy_account` in the mempool. These transactions occupy mempool slots for the victim's address, can displace or delay the victim's own legitimate nonce-1 transaction (since the mempool enforces one transaction per nonce per address), and force the batcher to process and reject them. The victim's `deploy_account + invoke` UX flow can be griefed at zero cost to the attacker.

### Likelihood Explanation

**High.** The precondition — victim has a `deploy_account` in the mempool — is publicly observable and is a common, expected state (the feature exists specifically to support the `deploy_account + invoke` UX pattern). No privileged access is required. The attacker only needs to submit a well-formed RPC transaction with the victim's address and `nonce=1`.

### Recommendation

Replace the `sender_address`-based mempool lookup with a check that ties the skip decision to the actual transaction hash of the pending `deploy_account`. Specifically, the gateway should only skip `__validate__` when it can confirm that the pending `deploy_account` transaction in the mempool was submitted by the same party (i.e., the invoke transaction's signature is valid against the account's expected public key, or the deploy_account's class hash is known and the key can be extracted). Alternatively, do not skip `__validate__` at the gateway at all — instead, skip it only inside the batcher when it is sequencing the `deploy_account` and the immediately following invoke in the same block, where the ordering guarantee is enforced.

### Proof of Concept

```
1. Victim submits:
   deploy_account_tx { sender_address: 0xVICTIM, nonce: 0, signature: <valid> }
   → mempool.account_tx_in_pool_or_recent_block(0xVICTIM) == true

2. Attacker submits:
   invoke_tx {
     sender_address: 0xVICTIM,   // user-controlled field
     nonce: 1,                    // satisfies nonce == ONE
     calldata: <drain_calldata>,
     signature: [0x0, 0x0],      // garbage — never checked
   }

3. Gateway stateful validation:
   account_nonce = get_nonce(0xVICTIM) = 0   // account not yet deployed
   tx.nonce() = 1
   → condition (nonce==1 && account_nonce==0) satisfied
   → account_tx_in_pool_or_recent_block(0xVICTIM) = true
   → skip_validate = true
   → validate flag = false
   → __validate__ NOT called
   → transaction admitted to mempool ✓

4. Batcher later executes:
   → __validate__ IS called with validate=true
   → signature check fails → transaction rejected (no state change)
   → but victim's nonce-1 slot was occupied; victim's own invoke tx
     may have been evicted or delayed

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
