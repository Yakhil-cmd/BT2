### Title
Pool Admin Can Permanently Brick `collectFees` by Setting `adminFeeDestination` to a Reverting Address, Blocking Protocol Fee Collection and Protocol Fee-Rate Updates — (`metric-core/contracts/MetricOmmPool.sol`, `metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`collectFees` in `MetricOmmPool` transfers fees to `adminFeeDestination_` **before** transferring to `FACTORY` and before zeroing the notional-fee accumulators. There is no `try/catch` around either transfer. The pool admin controls `adminFeeDestination` via `setPoolAdminFeeDestination`, which only rejects `address(0)`. If the pool admin points `adminFeeDestination` at any address whose token receipt reverts (a contract that always reverts, or a USDC/USDT-blacklisted address), every subsequent call to `collectFees` reverts. Because `setPoolProtocolFee` (owner-only) and `setPoolAdminFees` (pool-admin-only) both call `collectFees` as their first action, those entry-points also become permanently bricked. The protocol owner has no factory function that can override `poolAdminFeeDestination`; only the pool admin can call `setPoolAdminFeeDestination`.

---

### Finding Description

**Root cause — `collectFees` (MetricOmmPool.sol lines 416–430):**

```solidity
if (totalFee0ToAdmin > 0) {
    transferToken0(adminFeeDestination_, totalFee0ToAdmin);   // ← reverts here
}
if (totalFee1ToAdmin > 0) {
    transferToken1(adminFeeDestination_, totalFee1ToAdmin);
}
if (totalFee0ToProtocol > 0) {
    transferToken0(FACTORY, totalFee0ToProtocol);             // ← never reached
}
if (totalFee1ToProtocol > 0) {
    transferToken1(FACTORY, totalFee1ToProtocol);
}
notionalFeeToken0Scaled = 0;                                  // ← never zeroed
notionalFeeToken1Scaled = 0;
```

`transferToken0`/`transferToken1` use OpenZeppelin `SafeERC20.safeTransfer`, which propagates any revert from the token contract. If `adminFeeDestination_` is on a USDC/USDT blocklist, or is a contract whose fallback/`onERC20Received` reverts, the entire `collectFees` call reverts atomically.

**Attack path:**

1. Pool admin calls `setPoolAdminFees` while `adminFeeDestination` is still valid → succeeds, sets non-zero `adminSpreadFeeE6` / `adminNotionalFeeE8`.
2. Pool admin calls `setPoolAdminFeeDestination(pool, <reverting_address>)` — only check is `!= address(0)`, so it succeeds.
3. Fees accrue in the pool (spread surplus and `notionalFeeToken0/1Scaled`).
4. Any call to `collectPoolFees`, `setPoolProtocolFee`, or `setPoolAdminFees` now reverts because each calls `collectFees` first and the admin transfer fails.
5. The protocol owner has no factory path to override `poolAdminFeeDestination`; `setPoolAdminFeeDestination` is `onlyPoolAdmin`.

**Why the pool admin cannot self-recover:** `setPoolAdminFees` also calls `collectFees` first, so the pool admin cannot lower their own fee rates to zero (which would make `totalFee0/1ToAdmin == 0` and skip the failing transfer). The only recovery path is `setPoolAdminFeeDestination` — but that function does **not** call `collectFees`, so the pool admin can call it to fix the destination. However, if the pool admin is persistently malicious, they can immediately re-set the destination to a reverting address after any protocol-owner action, making the DoS indefinitely renewable with a single transaction.

---

### Impact Explanation

| Affected function | Caller | Effect |
|---|---|---|
| `collectPoolFees` | Anyone | Protocol and admin fees permanently stuck in pool |
| `setPoolProtocolFee` | Protocol owner | Cannot update protocol fee rates for the pool |
| `setPoolAdminFees` | Pool admin | Cannot update their own fee rates |

Protocol fees (`notionalFeeToken0/1Scaled` + spread surplus) accumulate in the pool contract but can never be extracted. The protocol owner loses all future fee revenue from the affected pool and cannot adjust fee policy. This satisfies the "direct loss of protocol fees" and "admin-boundary break" criteria in the impact gate.

---

### Likelihood Explanation

The pool admin is explicitly classified as semi-trusted. The attack requires two sequential, legitimate pool-admin transactions (`setPoolAdminFees` then `setPoolAdminFeeDestination`). No privileged escalation, no flash loan, and no front-running is needed. A non-malicious variant also exists: if the `adminFeeDestination` is a multisig that later gets USDC/USDT-blacklisted, the same DoS occurs without any malicious intent, and the pool admin may be unable to recover if they are the same blacklisted entity.

---

### Recommendation

1. **Wrap admin transfers in `try/catch`** inside `collectFees`: if the admin transfer fails, emit an event and skip it (leaving admin fees in the pool for a later retry), but continue to transfer protocol fees and zero the accumulators.
2. **Alternatively**, allow the protocol owner to override `poolAdminFeeDestination` in an emergency, or add a factory-owner escape hatch that calls `collectFees` with `adminFeeDestination = address(0)` (skipping the admin leg).
3. **Validate `adminFeeDestination`** at set-time with a small probe transfer or a whitelist check.

---

### Proof of Concept

```solidity
// 1. Pool admin sets non-zero admin fees (succeeds, adminFeeDestination is valid)
factory.setPoolAdminFees(pool, 1000, 0);

// 2. Pool admin points adminFeeDestination at a contract that always reverts
RevertingReceiver bad = new RevertingReceiver(); // reverts on any ERC20 transfer
factory.setPoolAdminFeeDestination(pool, address(bad));

// 3. Fees accrue via swaps ...

// 4. Protocol owner tries to update protocol fee rate → REVERTS
vm.prank(protocolOwner);
factory.setPoolProtocolFee(pool, newRate, 0); // reverts inside collectFees

// 5. Anyone tries to collect fees → REVERTS
factory.collectPoolFees(pool); // reverts inside collectFees

// Protocol fees are permanently stuck; protocol owner has no override path.

    external
    override
    nonReentrant

    if (newAdminFeeDestination == address(0)) revert InvalidAdminFeeDestination();
