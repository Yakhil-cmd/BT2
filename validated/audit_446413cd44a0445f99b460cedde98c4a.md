### Title
Invoke Transaction with Nonce=1 Bypasses `__validate__` Signature Verification via `skip_stateful_validations` — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator skips the `__validate__` entry point for any Invoke transaction with `nonce=1` when the sender account has any transaction in the mempool or a recent block. Because `__validate__` is the only place where the account's cryptographic signature is verified, an attacker can submit an Invoke transaction with an arbitrary invalid signature on behalf of any account that has a pending `deploy_account` transaction, and the gateway will admit it to the mempool without signature verification.

---

### Finding Description

**Root cause — `skip_stateful_validations`:** [1](#0-0) 

The function returns `true` (skip validation) when all three conditions hold:
1. The transaction is `ExecutableTransaction::Invoke`
2. `tx.nonce() == Nonce(Felt::ONE)` (hardcoded to 1)
3. `account_nonce == Nonce(Felt::ZERO)` (account not yet deployed on-chain)
4. `account_tx_in_pool_or_recent_block(sender)` returns `true`

**Effect on the validate entry point:**

When `skip_validate = true` is returned, `run_validate_entry_point` sets `execution_flags.validate = false`: [2](#0-1) 

Inside `StatefulValidator::perform_validations`, the Invoke branch checks this flag and returns early **before** calling `__validate__`: [3](#0-2) 

The `__validate__` entry point at line 84 is never reached. This is the account contract's signature-verification function — the only cryptographic check on the transaction signature.

**The mempool's `validate_tx` does not check signatures:**

The `ValidationArgs` struct passed to the mempool contains only `address`, `account_nonce`, `tx_hash`, `tx_nonce`, `tip`, and `max_l2_gas_price` — no signature field: [4](#0-3) 

The mempool's `validate_tx` only calls `validate_incoming_tx` (nonce ordering) and `validate_fee_escalation` (fee replacement rules): [5](#0-4) 

**Attack scenario:**

1. Account A submits a valid `deploy_account` transaction (passes full validation including `__validate_deploy__`). It enters the mempool.
2. An attacker observes (via P2P broadcast) that account A has a pending `deploy_account` in the mempool.
3. The attacker crafts an Invoke transaction: `sender_address = A`, `nonce = 1`, `signature = [0xdeadbeef, ...]` (arbitrary invalid bytes).
4. The gateway's `run_pre_validation_checks` calls `validate_state_preconditions` (nonce 1 ≥ account nonce 0 → passes) and `validate_by_mempool` (nonce/fee checks → passes). [6](#0-5) 

5. `skip_stateful_validations` returns `true` because account A is in the mempool.
6. `run_validate_entry_point` is called with `skip_validate = true` → `validate = false` → `__validate__` is never called.
7. The Invoke transaction is admitted to the mempool with an unverified, invalid signature.

The gateway's `add_tx_inner` flow confirms this path is production-reachable: [7](#0-6) 

---

### Impact Explanation

This matches the **High** impact category: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

The gateway's admission invariant — that every admitted transaction has a cryptographically valid signature — is broken. An attacker can inject Invoke transactions with invalid signatures into the mempool on behalf of any account that has a pending `deploy_account`. The transactions will fail during block execution (the batcher runs `__validate__` with `validate = true`), but the gateway has already accepted them, wasting mempool slots, batcher execution cycles, and potentially displacing legitimate transactions via fee escalation.

The broken invariant is: **all transactions in the mempool must have passed account signature verification**. The `skip_stateful_validations` path violates this for any Invoke with `nonce=1` from an account with a pending deploy.

---

### Likelihood Explanation

The attack requires only:
1. Observing that any account has a `deploy_account` transaction in the mempool — public information via P2P broadcast.
2. Crafting an Invoke with `nonce=1` and an arbitrary signature for that account address.

No privileged access, no special keys, and no on-chain state is required. Any network participant can execute this.

---

### Recommendation

The `skip_stateful_validations` logic should not skip the `__validate__` entry point entirely. The UX goal (accepting an Invoke paired with a `deploy_account` before the account exists on-chain) can be achieved by skipping only the **nonce pre-check** in `perform_pre_validation_stage`, while still running `__validate__` to verify the signature. Alternatively, the skip should be scoped to transactions submitted in the same gateway request as the `deploy_account` (not based on mempool state observable by third parties).

---

### Proof of Concept

```
1. Account A: submit deploy_account tx (valid signature) → enters mempool
   account_tx_in_pool_or_recent_block(A) now returns true

2. Attacker: submit Invoke tx with:
     sender_address = A
     nonce          = 1
     signature      = [0xdeadbeef, 0xcafebabe]  ← arbitrary invalid

3. Gateway stateful validation:
   - validate_state_preconditions: nonce 1 >= account_nonce 0 → OK
   - validate_by_mempool: nonce/fee checks → OK
   - skip_stateful_validations:
       tx.nonce() == Nonce(1) ✓
       account_nonce == Nonce(0) ✓
       account_tx_in_pool_or_recent_block(A) == true ✓
       → returns true (skip)
   - run_validate_entry_point(skip_validate=true):
       execution_flags.validate = false
       StatefulValidator::perform_validations → early return Ok(())
       __validate__ NEVER CALLED

4. Invoke tx admitted to mempool with invalid signature ← invariant broken

5. Batcher later executes tx with validate=true → __validate__ fails → tx rejected
   (no chain halt, but admission invariant violated)
```

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
