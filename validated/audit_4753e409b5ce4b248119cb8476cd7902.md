## Code Path Trace

**Step 1: `chia_spendVC` is a registered, callable dApp command.**

`DappCommands.ts` iterates all `Commands` entries and registers any with a `dapp` array. Since `'chia_wallet.vc_spend'` has `dapp: [{ command: 'chia_spendVC', ... }]`, `chia_spendVC` is inserted into the `DappCommands` map and is a valid, requestable WalletConnect command. [1](#0-0) [2](#0-1) 

**Step 2: `BlockedCommands` is NOT a security gate on execution.**

`BlockedCommands.ts` lists `'chia_wallet.vc_spend'` (the internal command ID), but this list is only imported in `SettingsAdvanced.tsx` for UI display purposes. The actual execution gate in `main.tsx` checks `DappCommands.has(command)` (using the dApp command name `chia_spendVC`), not `BlockedCommands`. `filterRequestedDappCommands` also checks against `DappCommands`, not `BlockedCommands`. [3](#0-2) [4](#0-3) [5](#0-4) 

**Step 3: `new_puzhash` is accepted as an arbitrary string with no ownership validation.**

`parseDappParams` only coerces `new_puzhash` with `String(value)`. There is no check that the value corresponds to a puzzle hash owned by the current wallet. [6](#0-5) [7](#0-6) 

**Step 4: The confirmation dialog title is misleading.**

The `dapp` entry for `chia_spendVC` overrides the parent command title with `'Add Proofs To Verifiable Credential'`. `DappCommands.ts` applies this override. So when `humanizeDappCommand` renders the dialog, the title shown to the user is **"Add Proofs To Verifiable Credential"**, not "Transfer VC" or "Confirm VC Spend". The `new_puzhash` field is shown as a raw hex string labeled "New Puzzle Hash" with no indication it is an external/attacker-controlled address. [8](#0-7) [9](#0-8) [10](#0-9) 

**Step 5: `chia_wallet.vc_spend` is NOT in `isSpendCommand`, so it is classified as "other" during permission grant.**

When a dApp requests `chia_spendVC` during pairing, `classifyDappCommands` places it in the `other` bucket (not `spending`), because `chia_wallet.vc_spend` is absent from `SPEND_COMMANDS`. This means the user sees it as a lower-risk permission category. [11](#0-10) [12](#0-11) 

**Step 6: After user approval, `vc_spend` is called with the attacker's `new_puzhash` verbatim.**

`main.tsx` calls `sendCommand(chiaCommand, destination, parsedParams)` where `parsedParams` contains the unvalidated `new_puzhash`. The VC is transferred to the attacker-controlled puzzle hash. [13](#0-12) [14](#0-13)

## Verdict

### Title
Misleading WalletConnect `chia_spendVC` Dialog Enables VC Transfer to Attacker-Controlled Address via Unvalidated `new_puzhash` — (`packages/gui/src/electron/commands/Commands.ts`)

### Summary
A WalletConnect dApp granted `chia_spendVC` can supply an arbitrary `new_puzhash` to redirect a VC spend to an attacker-controlled address. The confirmation dialog is titled "Add Proofs To Verifiable Credential" (not "Transfer VC"), shows only a raw hex puzzle hash with no ownership warning, and the command is classified as "other" (not "spending") during permission grant — all of which obscure the true impact from the user.

### Finding Description
1. `chia_spendVC` is registered in `DappCommands` and is requestable by any WalletConnect dApp.
2. `parseDappParams` accepts `new_puzhash` as an arbitrary string with no validation against wallet-owned addresses.
3. The dApp command title override (`'Add Proofs To Verifiable Credential'`) is shown in the confirmation dialog instead of the parent command's title (`'Confirm VC Spend'`), misleading the user about the operation's nature.
4. `chia_wallet.vc_spend` is absent from `SPEND_COMMANDS`, so it is classified as `other` (not `spending`) during permission grant, reducing user scrutiny.
5. After user approval, `vc_spend` is called with the attacker's `new_puzhash` verbatim, transferring the VC on-chain.

### Impact Explanation
The VC (a blockchain asset) is irreversibly transferred to an attacker-controlled address. This is an unauthorized asset transfer triggered through WalletConnect with a misleading approval dialog. Fits: **Critical — unauthorized transfer of VC asset** / **High — WalletConnect state causes user to approve transfer to wrong destination**.

### Likelihood Explanation
Requires: (1) user has paired with a malicious dApp and granted `chia_spendVC` permission, (2) user approves the misleading dialog. The misleading title ("Add Proofs") and "other" permission classification make both steps plausible for a non-expert user.

### Recommendation
- Validate that `new_puzhash`, when provided, corresponds to an address owned by the current wallet, or reject it entirely from the WalletConnect path.
- Override the dApp command title to clearly indicate a transfer is occurring when `new_puzhash` is present.
- Add `chia_wallet.vc_spend` to `SPEND_COMMANDS` so it is classified as a spending command during permission grant.
- Display `new_puzhash` with an explicit "external address" warning if it does not match a wallet-owned puzzle hash.

### Proof of Concept
1. Pair a malicious dApp and request `chia_spendVC` permission (granted as "other" category).
2. Call `chia_spendVC` with `{ vcId: "<victim_vc_id>", newPuzhash: "<attacker_puzhash>", newProofHash: "<any>", providerInnerPuzhash: "<any>" }`.
3. User sees dialog titled "Add Proofs To Verifiable Credential" with "New Puzzle Hash: `<attacker_puzhash>`".
4. User clicks "Send".
5. `vc_spend` executes with `new_puzhash=<attacker_puzhash>`, transferring the VC on-chain to the attacker.

function processDappCommands() {

        DappCommands.set(command, {
          ...rest,
          commandId,
          title,
          message,
          confirmLabel,
          params,
          destructive,
        });
