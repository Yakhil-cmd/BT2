### Title
Unbounded Loop in `calculate_rewards` Enables Permanent Freezing of Delegator Unclaimed Yield - (File: src/pool/pool.cairo)

### Summary
The `calculate_rewards` internal function in `pool.cairo` iterates over a delegator's entire `pool_member_epoch_balance` trace in an unbounded loop. The code itself acknowledges this with the comment "This loop is unbounded but unlikely to exceed gas limits." A delegator who accumulates enough balance-change trace entries without claiming rewards will eventually be unable to call `claim_rewards` at all, permanently freezing their accrued yield.

### Finding Description
In `Pool::claim_rewards`, the function calls `calculate_rewards`, which iterates over every entry in `pool_member_epoch_balance` from `entry_to_claim_from` up to the current trace length:

```
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    ...
    entry_to_claim_from += 1;
}
```

Every call to `add_to_delegation_pool` or `exit_delegation_pool_intent` in a new epoch appends a new checkpoint to the trace via `set_member_balance` → `trace.insert(key: self.get_epoch_plus_k(), value: ...)`. The `insert` function only merges entries when the epoch key matches the last entry; otherwise it pushes a new checkpoint. Because `entry_to_claim_from` is only advanced when `claim_rewards` succeeds, a delegator who makes balance changes across many epochs without claiming will accumulate an ever-growing trace. Once the trace is large enough that a single `claim_rewards` call cannot iterate through all entries within the Starknet transaction gas limit, the rewards are permanently inaccessible.

### Impact Explanation
**High — Permanent freezing of unclaimed yield.**

Once the `pool_member_epoch_balance` trace for a delegator grows beyond the gas-iteration threshold, every call to `claim_rewards` for that pool member will revert out-of-gas. The `entry_to_claim_from` cursor is only updated on a successful completion of `calculate_rewards`; there is no partial-claim or pagination mechanism. The delegator's accumulated STRK rewards are permanently locked in the pool contract with no recovery path.

### Likelihood Explanation
**Medium.** A delegator must make balance changes (via `add_to_delegation_pool` or `exit_delegation_pool_intent`) across a large number of distinct epochs without ever calling `claim_rewards`. This is an unusual but entirely permissible usage pattern — nothing in the protocol forces a delegator to claim between balance changes. On a live network with short epoch durations, a delegator who is active over many months and never claims could organically reach this state. The threshold is not known precisely but is bounded by Starknet's per-transaction gas limit divided by the per-iteration cost of the loop body (which includes a `find_sigma` call and storage reads).

### Recommendation
Introduce a pagination mechanism for `calculate_rewards` so that `claim_rewards` can be called multiple times to process the trace in bounded chunks, advancing `entry_to_claim_from` on each partial call. Alternatively, enforce a maximum number of unclaimed balance-change entries (e.g., require `claim_rewards` before a new balance change can be recorded once the trace exceeds a threshold).

2. Across N distinct epochs, the delegator alternates `add_to_delegation_pool` (small amount) and `exit_delegation_pool_intent` (partial amount), each call in a different epoch, appending a new checkpoint to `pool_member_epoch_balance` each time.
3. The delegator never calls `claim_rewards`, so `entry_to_claim_from` stays at 0 and the trace grows to length N.
