### Title
Permanent Loss of ERC-20 Tokens When NEAR-Side `ft_transfer` Fails in `ExitToNear` Precompile Without `error_refund` Feature - (`engine-precompiles/src/native.rs`, `engine/src/contract_methods/connector.rs`)

### Summary

When the `error_refund` compile-time feature is absent (it is not in the `default` feature set), ERC-20 tokens burned during a `withdrawToNear` call are permanently and irrecoverably lost if the subsequent NEAR-side `ft_transfer` promise fails. No failure callback is registered in this configuration, so there is no on-chain path to re-mint or recover the burned tokens. This is a direct structural analog to the CCTP `mintRecipient` permanent-freeze vulnerability: assets are destroyed on the source side before the destination delivery is confirmed, and the absence of a recovery mechanism makes the loss final.

### Finding Description

**Root cause — conditional callback registration in `ExitToNear::run`**

`engine-precompiles/src/native.rs` lines 449–483 build the callback arguments for the exit promise:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,          // ← always None without the feature
    transfer_near: transfer_near_args,
};
```

For a standard ERC-20 exit (flag `0x1`), `transfer_near` is also `None`. Therefore:

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // ← no callback attached
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs { … })
};
```

Without `error_refund`, `callback_args` equals `default()`, so the engine schedules only a bare `ft_transfer` promise with **no attached callback**. If that promise fails, NEAR simply discards the failure silently — there is no `exit_to_near_precompile_callback` invocation, no re-mint, and no refund.

**Burn happens before the promise**

`etc/eth-contracts/contracts/EvmErc20.sol` lines 53–62:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);   // ← irreversible burn
    …
    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, …)
    }
}
```

The ERC-20 tokens are destroyed inside the same EVM transaction that schedules the NEAR promise. There is no rollback if the promise later fails.

**Callback handler confirms the gap**

`engine/src/contract_methods/connector.rs` lines 231–241:

```rust
} else if let Some(args) = args.refund {
    // Exit call failed; need to refund tokens
    let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;
    …
} else {
    None   // ← reached when refund is None; nothing happens
};
```

Without `error_refund`, `args.refund` is always `None`, so the `else` branch is always taken on failure — the engine does nothing.

**Feature is not in the default build**

`engine/Cargo.toml` lines 43–48:

```toml
[features]
default = ["std"]
…
error_refund = ["aurora-engine-precompiles/error_refund"]
```

`error_refund` is absent from `default`. Any production build that does not explicitly opt in to this feature ships with the no-refund code path.

**Test explicitly documents the permanent-loss behavior**

`engine-tests/src/tests/erc20_connector.rs` lines 656–660:

```rust
#[cfg(feature = "error_refund")]
let balance = FT_TRANSFER_AMOUNT.into();
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
```

The test confirms that without `error_refund`, the exit amount is simply gone.

### Impact Explanation

**Severity: Critical — Permanent freezing / loss of funds.**

Any ERC-20 token withdrawal to NEAR that results in a failed `ft_transfer` promise causes the burned tokens to be unrecoverable. The EVM state records the burn as final; the NEP-141 contract never credits the recipient; and no re-mint or refund is triggered. The loss is permanent and proportional to the withdrawn amount.

### Likelihood Explanation

**Medium.** The NEAR-side `ft_transfer` fails whenever the recipient NEAR account is not registered with the NEP-141 token — a common user error. It also fails if the NEP-141 contract is paused, if the Aurora engine account has insufficient NEP-141 balance, or if the recipient account does not exist. An adversary who can observe a pending `withdrawToNear` call can front-run it by ensuring the target NEAR account is unregistered (e.g., by calling `storage_unregister` on the NEP-141 contract for that account if the NEP-141 allows it), forcing the transfer to fail and the tokens to be permanently lost. Even without adversarial action, ordinary user mistakes (wrong recipient, unregistered account) trigger the same outcome.

### Recommendation

1. Add `error_refund` to the `default` feature set in `engine/Cargo.toml`, or enforce it as a mandatory compile-time requirement for production WASM builds.
2. Alternatively, restructure the `ExitToNear` precompile so that a failure callback is always registered regardless of the feature flag, and the callback always attempts a re-mint to the original caller.
3. Ensure the deployed ERC-20 contract variant (`EvmErc20V2.sol`, which encodes the sender as the refund address) is used consistently with `error_refund` enabled, so the input format and the feature flag remain in sync.

### Proof of Concept

1. The production WASM is compiled without `error_refund` (default feature set).
2. Alice holds 10,000 units of an ERC-20 token on Aurora backed by NEP-141 `token.near`.
3. Alice calls `EvmErc20.withdrawToNear("alice.near", 10_000)`.
4. Inside the EVM transaction: `_burn(alice, 10_000)` executes — Alice's ERC-20 balance drops to zero.
5. The `ExitToNear` precompile schedules `PromiseArgs::Create(ft_transfer { receiver_id: "alice.near", amount: 10_000 })` with no callback.
6. The NEAR runtime executes the promise. `alice.near` is not registered with `token.near`, so `ft_transfer` panics and the promise fails.
7. Because no callback was attached, NEAR discards the failure. No `exit_to_near_precompile_callback` is invoked.
8. Alice's 10,000 ERC-20 tokens are burned and gone. The NEP-141 balance of the Aurora engine contract is unchanged (the transfer never completed). Alice has lost her funds with no recovery path. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
