### Title
Permanent Fund Freeze When `ExitToNear` Promise Fails Without `error_refund` Feature - (File: engine-precompiles/src/native.rs)

### Summary
When the Aurora Engine is compiled without the `error_refund` feature (the default), the `ExitToNear` precompile does not register a recovery callback for failed `ft_transfer` promises. If the downstream NEP-141 `ft_transfer` call fails after ERC-20 tokens have been burned or ETH has been moved to the precompile address, the user's funds are permanently frozen with no recovery path.

### Finding Description
The `ExitToNear` precompile (`engine-precompiles/src/native.rs`) handles two exit paths: ERC-20 token exits (which burn the ERC-20 tokens and call `ft_transfer` on the NEP-141 contract) and ETH exits (which move ETH to the precompile address and call `ft_transfer` on the eth-connector).

The refund mechanism is gated behind the `error_refund` compile-time feature flag. When this feature is absent — which is the default, as it is not listed in the `default` features of `engine-precompiles/Cargo.toml` — the `refund` field in `ExitToNearPrecompileCallbackArgs` is hardcoded to `None`: [1](#0-0) 

Because `refund` is `None` and `transfer_near` is also `None` for standard ERC-20 and ETH exits, `callback_args` equals `ExitToNearPrecompileCallbackArgs::default()`. The promise selection logic then emits a bare `PromiseArgs::Create` with **no callback attached**: [2](#0-1) 

This means if the `ft_transfer` promise fails (e.g., the recipient NEAR account is not registered with the NEP-141 contract), there is no `exit_to_near_precompile_callback` invoked, and therefore no `refund_on_error` call is ever made: [3](#0-2) 

The `error_refund` feature is defined as an opt-in with no default activation: [4](#0-3) 

The production `contract` feature does not pull in `error_refund`: [5](#0-4) 

The tests explicitly acknowledge this behavior: [6](#0-5) [7](#0-6) 

### Impact Explanation
**Impact: High — Permanent freezing of funds.**

- **ERC-20 exit path**: The ERC-20 tokens are burned in the EVM before the `ft_transfer` promise is dispatched. If `ft_transfer` fails, the burned tokens are gone from the EVM state and the NEP-141 tokens remain with the Aurora contract. The user's ERC-20 tokens are permanently destroyed with no compensation.
- **ETH exit path**: ETH is transferred from the user's EVM address to the `exit_to_near` precompile address (`0xe9217bc70b7ed1f598ddd3199e80b093fa71124f`) before the `ft_transfer` promise is dispatched. If `ft_transfer` fails, the ETH is permanently stranded at the precompile address with no recovery mechanism. [8](#0-7) 

### Likelihood Explanation
**Likelihood: Medium.**

The `ft_transfer` promise fails whenever the recipient NEAR account is not registered with the NEP-141 token contract (a common requirement for NEP-141 tokens). Any EVM user who calls the `ExitToNear` precompile with a recipient NEAR account that has not pre-registered with the target NEP-141 contract will trigger this failure. This is a realistic and reachable scenario for ordinary users unfamiliar with NEAR's storage registration requirement.

### Recommendation
Enable the `error_refund` feature in the production build by including it in the `contract` feature definition in `engine/Cargo.toml` and `engine-precompiles/Cargo.toml`. This ensures the refund callback is always registered and `refund_on_error` is invoked on promise failure, restoring burned ERC-20 tokens or returning stranded ETH to the original sender. [9](#0-8) 

### Proof of Concept
1. Compile Aurora Engine with the `contract` feature only (no `error_refund`).
2. Deploy a NEP-141 token and bridge it to an ERC-20 on Aurora.
3. As an EVM user, call the `ExitToNear` precompile (address `0xe9217bc70b7ed1f598ddd3199e80b093fa71124f`) with flag `0x01` (ERC-20 exit), specifying a recipient NEAR account that is **not registered** with the NEP-141 contract.
4. The ERC-20 tokens are burned in the EVM (irreversible).
5. The `ft_transfer` promise to the NEP-141 contract fails because the recipient is unregistered.
6. No callback is registered; no `refund_on_error` is called.
7. Observe: the user's ERC-20 balance is zero, the NEP-141 balance of the recipient is zero, and the NEP-141 tokens remain with the Aurora contract — permanently inaccessible to the user.

This is confirmed by the existing test `test_exit_to_near_refund` which explicitly documents that without `error_refund`, the exit amount is permanently lost: [10](#0-9)

pub fn refund_on_error<I: IO + Copy, E: Env, P: PromiseHandler>(
    io: I,
    env: &E,
    state: EngineState,
    args: &RefundCallArgs,
    handler: &mut P,
) -> EngineResult<SubmitResult> {
    let current_account_id = env.current_account_id();
    if let Some(erc20_address) = args.erc20_address {
        // ERC-20 exit; re-mint burned tokens
        let erc20_admin_address = current_address(&current_account_id);
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, erc20_admin_address, current_account_id, io, env);

        let refund_address = args.recipient_address;
        let amount = U256::from_big_endian(&args.amount);
        let input = setup_refund_on_error_input(amount, refund_address);

        engine.call(
            &erc20_admin_address,
            &erc20_address,
            Wei::zero(),
            input,
            u64::MAX,
            Vec::new(),
            Vec::new(),
            handler,
        )
    } else {
        // ETH exit; transfer ETH back from precompile address
        let exit_address = exit_to_near::ADDRESS;
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, exit_address, current_account_id, io, env);
        let refund_address = args.recipient_address;
        let amount = Wei::new(U256::from_big_endian(&args.amount));
        engine.call(
            &exit_address,
            &refund_address,
            amount,
            Vec::new(),
            u64::MAX,
            vec![
                (exit_address.raw(), Vec::new()),
                (refund_address.raw(), Vec::new()),
            ],
            Vec::new(),
            handler,
        )
    }
