### Title
Missing Access Control on `update_rewards` Allows Any Caller to Permanently Block Reward Distribution - (File: src/staking/staking.cairo)

### Summary
`IStakingRewardsManager::update_rewards` is specified as callable only by the Starkware sequencer, but the implementation enforces no such restriction. Any non-zero address can call it with `disable_rewards: true`, consuming the global `last_reward_block` slot for that block and permanently preventing the sequencer from distributing rewards for that block to any staker.

### Finding Description
The spec at `docs/spec.md` explicitly states the access control for `update_rewards` is "Only starkware sequencer." However, the implementation at `src/staking/staking.cairo` only calls `general_prerequisites()`, which checks:

1. Contract is not paused
2. Caller is not zero address [1](#0-0) 

There is no role-based or address-based sequencer check. The function writes to a **global** `last_reward_block` storage variable on every call: [2](#0-1) 

Because `last_reward_block` is a single global value (not per-staker), a single attacker call in block N with `disable_rewards: true` sets `last_reward_block = N` and distributes zero rewards. Any subsequent call in block N — including the sequencer's legitimate call for any staker — reverts with `REWARDS_ALREADY_UPDATED`. [3](#0-2) 

The spec confirms this is the intended sequencer-only function: [4](#0-3) 

The `general_prerequisites` function that is the only guard: [5](#0-4) 

### Impact Explanation
An unprivileged attacker calling `update_rewards(any_active_staker, disable_rewards: true)` before the sequencer in every block permanently prevents all consensus reward distribution for all stakers. Even a partial attack (front-running a subset of blocks) causes temporary freezing of unclaimed yield. No profit is required — the attacker only pays gas. This matches:

- **High**: Permanent freezing of unclaimed yield (persistent attacker)
- **Medium**: Griefing with no profit motive but damage to users or protocol (intermittent attacker)

### Likelihood Explanation
The entry path requires no privilege: any non-zero Starknet address can call `update_rewards`. The attacker must submit a transaction in the same block as the sequencer's intended call. On Starknet, where the sequencer processes transactions in arrival order, a well-timed transaction submitted just before the sequencer's own `update_rewards` invocation achieves the front-run. The cost is one transaction per block.

### Recommendation
Add an access control check enforcing that only the designated sequencer address (or a `SEQUENCER_ROLE`) can call `update_rewards`, consistent with the spec. The existing `RolesComponent` pattern used elsewhere (e.g., `only_security_agent`, `only_token_admin`) should be extended with a sequencer role. [6](#0-5) 

### Proof of Concept

1. Consensus rewards are active (`is_pre_consensus()` returns false).
2. Attacker identifies any active staker `S` with non-zero balance.
3. In block N, attacker calls `update_rewards(S, disable_rewards: true)`.
   - `last_reward_block` is written to N.
   - No rewards are distributed.
4. Sequencer attempts `update_rewards(S, disable_rewards: false)` in block N → reverts with `REWARDS_ALREADY_UPDATED`.
5. Staker S (and all other stakers, since `last_reward_block` is global) loses rewards for block N.
6. Repeating step 3 every block permanently freezes all consensus reward accrual for every staker in the protocol.
