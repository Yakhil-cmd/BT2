### Title
Incorrect Royalty Total Displayed in WalletConnect Offer Confirmation for Multi-NFT Offers — (`File: packages/gui/src/electron/commands/parseCommandDisplay.ts`)

### Summary
The `formatAmountWithRoyalties` function in `parseCommandDisplay.ts` incorrectly divides the fungible amount by the number of NFTs before computing each royalty, causing the "Total Amount with Royalties" shown in the WalletConnect confirmation dialog to be systematically understated when an offer involves multiple NFTs with royalties. A user approving a `chia_wallet.take_offer` via WalletConnect sees a lower total cost than they will actually pay.

### Finding Description
In `formatAmountWithRoyalties` (lines 363–367), the code first splits the full amount equally across all NFTs, then computes each royalty from that split amount:

```typescript
const splitAmount = amount / BigInt(royaltyPercentages.length);   // line 363
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
``` [1](#0-0) 

This produces: `royaltyTotal = (amount / N) × Σ(royaltyPercentage_i) / 10_000`

The correct Chia protocol behavior is that each NFT's royalty is computed from the **full** fungible amount, not a split share:

`royaltyTotal = amount × Σ(royaltyPercentage_i) / 10_000`

The two formulas differ by a factor of N (the number of NFTs). With N=2 NFTs having royalties of 5% and 0.1% on 0.0001 XCH (100,000,000 mojos):

- **Current (wrong):** `splitAmount = 50,000,000`; royalties = 2,500,000 + 50,000 = 2,550,000 mojos → displayed total: **0.00010255 XCH**
- **Correct:** royalties = 5,000,000 + 100,000 = 5,100,000 mojos → true total: **0.0001051 XCH**

The test suite encodes this wrong value as the expected output, confirming the bug is present and untested against the correct protocol behavior: [2](#0-1) 

The incorrect `amountWithRoyalties` value is then surfaced in the WalletConnect confirmation dialog (`Confirm.tsx`) as "Total Amount with Royalties": [3](#0-2) 

The `parseCommandDisplay` function is the sole path that generates this display for both `chia_wallet.take_offer` and `chia_wallet.create_offer_for_ids` WalletConnect commands: [4](#0-3) 

### Impact Explanation
When a user is presented with a WalletConnect approval dialog for an offer involving multiple NFTs with royalties, the displayed "Total Amount with Royalties" is understated by a factor proportional to the number of NFTs. The user approves the transaction believing the royalty cost is lower than it actually is. The Chia backend computes and deducts the correct (higher) royalty amount on-chain, so the user pays more XCH or CAT than the confirmation dialog indicated. This constitutes display of the wrong amount in the WalletConnect approval flow, causing the user to approve a spend they may not have consented to at the true cost.

### Likelihood Explanation
Any WalletConnect-connected dApp can trigger this path by sending a `chia_wallet.take_offer` command with an offer string containing multiple NFTs with royalties. No special privileges are required. The discrepancy grows with the number of NFTs and the magnitude of royalty percentages, making it exploitable by a dApp that crafts offers with many high-royalty NFTs.

### Recommendation
Remove the `splitAmount` division. Each NFT's royalty should be computed from the full `amount`:

```typescript
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (amount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
```

Update the corresponding test expectation for the multi-NFT case from `'0.00010255'` to `'0.0001051'`.

### Proof of Concept
Using the existing test fixture (two NFTs, royalties 500 and 10, amount 100,000,000 mojos):

- Current code: `splitAmount = 50,000,000`; displayed total = `0.00010255 XCH`
- Correct protocol: each NFT royalty from full amount; true total = `0.0001051 XCH`
- Understatement: ~50% of the actual royalty cost is hidden from the user

A malicious dApp offering 10 NFTs each with 10% royalties on a 1 XCH offer would show the user a royalty cost of 0.1 XCH (1/10th of correct) while the actual on-chain deduction would be 1 XCH in royalties, draining the user's wallet of 2 XCH total instead of the displayed 1.1 XCH.

export async function parseCommandDisplay(command: string, params: Record<string, unknown>) {
  if (command === 'chia_wallet.take_offer') {
    if (!params.offer || typeof params.offer !== 'string') {
      throw new Error('Offer is not valid');
    }

    const offerSummary = await getOfferSummary(params.offer);
    if (!offerSummary || !offerSummary.summary || !offerSummary.success) {
      throw new Error('Offer is not valid');
    }

    const { summary } = offerSummary;

    const walletDelta = offerSummaryToWalletDelta(summary);
    const walletInfos = await getWalletInfos();
    const assetKinds = offerSummaryAssetKinds(summary);
    const royaltyPercentages = offerSummaryRoyaltyPercentages(summary);
    const fees = parseMojos(summary.fees);

    return {
      walletDelta: await walletDeltaToDisplay(walletDelta, walletInfos, assetKinds, royaltyPercentages, fees),
    };
  }
