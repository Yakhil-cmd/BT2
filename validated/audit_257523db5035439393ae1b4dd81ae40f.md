### Title
Incomplete ibc-go v7 Migration Leaves Stale Consensus-State Subkeys That Panic the `ClientStates` gRPC Handler — (`app/upgrades.go`)

### Summary
The ibc-go v7 migration only cleaned canonical two-segment `consensusStates` keys. Three-segment variants of the form `clients/<id>/consensusStates/<rev>/<h>/clientState` survived in the IBC KV store. When the `ClientStates` gRPC handler iterates over the store it encounters these stale entries and triggers a proto-decoder panic on unmarshal, enabling an unauthenticated REST DoS via `GET /ibc/core/client/v1/client_states`.

### Finding Description
The ibc-go v7 migration was an incomplete state migration. It cleaned the canonical two-part `consensusStates` key format but left behind three-part variants:

```
clients/<id>/consensusStates/<revision>/<height>/clientState
```

These keys end with the literal segment `clientState`, which is the same suffix used by canonical client-state keys (`clients/<id>/clientState`). When the `ClientStates` gRPC handler (`rpc ClientStates` mapped to `GET /ibc/core/client/v1/client_states`) iterates over the IBC store with a `KeyClientStorePrefix` iterator, it encounters these stale entries and attempts to proto-unmarshal their values as `ClientState` objects. The stored bytes are consensus-state data, not client-state data, so the proto-decoder panics.

The root cause is confirmed in two production files:

- `app/upgrades.go` lines 55–59 (the fix comment): *"The v7 ibc-go migration only cleaned canonical 2-segment consensusStates keys; 3-segment variants … survived and cause the ClientStates gRPC handler to panic on unmarshal, enabling an unauthenticated REST DoS via GET /ibc/core/client/v1/client_states."*
- `integration_tests/ibc_v18.py` lines 35–38: *"Stale clients/<id>/consensusStates/<rev>/<h>/clientState keys … trigger a proto-decoder panic in the ClientStates gRPC handler."*

The v1.8 upgrade handler introduces `pruneStaleIBCConsensusStateSubkeys` to delete these keys, but on any node that has not yet executed the v1.8 upgrade plan the stale keys remain present and the endpoint is permanently broken.

### Impact Explanation
`GET /ibc/core/client/v1/client_states` is the standard REST/gRPC endpoint used by IBC relayers to enumerate all light clients before relaying packets. A proto-decoder panic in the handler makes the endpoint permanently unavailable (every call panics). This blocks IBC relayers from discovering or verifying client state, preventing them from submitting `RecvPacket`, `Acknowledgement`, and `Timeout` messages. The result is a permanent, unauthenticated denial-of-service against all IBC transfer flows on the chain — matching the allowed High impact: *"Permanent or long-lived inability for honest users or validators to process valid transactions, bridge/conversion flows … IBC transfers … under normal network assumptions."*

### Likelihood Explanation
The stale keys are already present in the live IBC store on any chain that ran the ibc-go v7 migration and has not yet applied the v1.8 upgrade. No special privilege is required; a single unauthenticated HTTP GET request is sufficient to trigger the panic on every call.

### Recommendation
Apply the v1.8 upgrade plan, which executes `pruneStaleIBCConsensusStateSubkeys` to delete all keys matching `clients/<id>/consensusStates/<rev>/<h>/clientState` before the endpoint is queried again. The pruning logic in `app/upgrades.go` is idempotent and safe to run multiple times.

### Proof of Concept
On a node that has completed the ibc-go v7 migration but has not yet applied the v1.8 upgrade:

```bash
curl http://<node>:<api-port>/ibc/core/client/v1/client_states
# → proto-decoder panic in the gRPC handler; endpoint returns 500 / connection reset
# Repeatable on every request; no authentication required.
```

The `pruneStaleIBCConsensusStateSubkeys` function in `app/upgrades.go` (lines 84–104) identifies the exact stale key pattern and the `check_ibc_client_states` helper in `integration_tests/ibc_v18.py` (lines 31–57) confirms the endpoint is panic-free only after the upgrade executes.
```go
func pruneStaleIBCConsensusStateSubkeys(ctx sdk.Context, store storetypes.KVStore) error {
	iterator := storetypes.KVStorePrefixIterator(store, host.KeyClientStorePrefix)
	defer sdk.LogDeferred(ctx.Logger(), func() error { return iterator.Close() })

	var staleKeys [][]byte
	for ; iterator.Valid(); iterator.Next() {
		// iterator.Key() includes the full "clients/" prefix.
		// Canonical: clients/<id>/clientState (3 parts)
		// Stale:     clients/<id>/consensusStates/<rev>/<h>/clientState (≥5 parts)
		parts := strings.Split(string(iterator.Key()), "/")
		if len(parts) >= 5 &&
			parts[2] == host.KeyConsensusStatePrefix &&
			parts[len(parts)-1] == host.KeyClientState {
			staleKeys = append(staleKeys, bytes.Clone(iterator.Key()))
		}
	}

	for _, k := range staleKeys {
		store.Delete(k)
	}
	return nil
```
