### Title
Gateway Admits Invoke Transactions With Invalid Signatures When `skip_stateful_validations` Bypasses the Validate Entry Point - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator unconditionally skips the account's `__validate__` entry point (signature check) for any invoke transaction with `nonce == 1` when the on-chain account nonce is `0` and `account_tx_in_pool_or_recent_block` returns `true`. An attacker who controls a fresh account address can submit a valid `deploy_account` to satisfy the pool check, then submit an invoke with `nonce=1` carrying an **invalid or forged signature**. The gateway admits the invoke to the mempool without ever verifying the signature.

### Finding Description

`run_pre_validation_checks` calls `skip_stateful_validations`, which returns `true` under the following conditions: [1](#0-0) 

```rust
async fn skip_stateful_validations(...) -> ... {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await ...;
        }
    }
    Ok(false)
}
```

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` is called with `validate: false`: [2](#0-1) 

```rust
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
```

This means `blockifier_validator.validate(account_tx)` is invoked with `validate: false`, so the account's `__validate__` entry point — which performs signature verification — is **never called** during gateway admission. [3](#0-2) 

The `validate_state_preconditions` and `validate_by_mempool` calls that precede this check only verify nonce range and mempool ordering — they do **not** verify the cryptographic signature. [3](#0-2) 

### Impact Explanation

**High — Mempool/gateway admission accepts invalid transactions before sequencing.**

An invoke transaction carrying a forged or invalid signature is admitted to the mempool and forwarded to the batcher. The batcher will eventually execute the transaction and the `__validate__` entry point will be called there (with `validate: true` in normal execution flags), causing a revert. However, the gateway's admission invariant — *every admitted invoke must have a cryptographically valid signature* — is broken. This enables:

- **Mempool spam / DoS**: An attacker can flood the mempool with signature-invalid invokes at the cost of only one funded `deploy_account` per account address, since the deploy_account satisfies the pool check for arbitrarily many subsequent invalid invokes (until the deploy_account is included in a block and the nonce advances past 0).
- **Wasted sequencer resources**: The batcher must execute and revert each invalid transaction, consuming block-building capacity.

### Likelihood Explanation

**Medium.** The attacker must:
1. Control a fresh account address (trivial — choose any salt/class hash).
2. Submit a valid `deploy_account` for that address (requires funding the address with enough STRK/ETH for fees).
3. Submit one or more invoke transactions with `nonce=1` and an invalid signature.

Step 2 has a cost, but a single funded address enables repeated invalid invoke submissions until the deploy_account is sequenced. The attack is fully unprivileged and requires no special access.

### Recommendation

Before deciding to skip the validate entry point, verify that the deploy_account transaction in the pool is specifically a `DeployAccount` type for the sender address, not just any transaction. Additionally, consider adding a flag or separate code path that still performs a lightweight signature format check (e.g., signature length and field element validity) even when the full `__validate__` entry point is skipped, so that structurally invalid signatures are rejected at the gateway boundary.

Concretely, in `skip_stateful_validations`:

```rust
// Current: checks if ANY tx for the account is in pool or recent block.
// Fix: also verify the queued transaction is specifically a DeployAccount,
// and/or perform a stateless signature sanity check before returning true.
```

### Proof of Concept

1. Attacker generates key pair `(K_valid, K_invalid)` and computes account address `A` from `K_valid`.
2. Attacker funds address `A` with enough STRK to cover fees.
3. Attacker submits `deploy_account` for `A` signed with `K_valid` → accepted by gateway, enters mempool.
4. Attacker submits `invoke` from `A` with `nonce=1`, `calldata=<anything>`, `signature=<garbage bytes signed with K_invalid>`.
5. Gateway evaluates:
   - `validate_state_preconditions`: nonce=1, account_nonce=0, `0 <= 1 <= 0 + max_gap` ✓
   - `validate_by_mempool`: mempool ordering check ✓
   - `skip_stateful_validations`: `nonce==1 && account_nonce==0 && account_tx_in_pool_or_recent_block(A)==true` → returns `true`
   - `run_validate_entry_point`: called with `validate: false` → `__validate__` **never executed**
6. Invalid invoke is admitted to the mempool.
7. Attacker repeats step 4 with different garbage signatures — each is admitted without signature verification. [4](#0-3) [5](#0-4)

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
```
