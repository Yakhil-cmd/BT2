### Title
Missing L1 Deployment Transaction Support Causes Silent State-Transition Failure — (`File: basic_bootloader/src/bootloader/transaction_flow/zk/process_l1_transaction.rs`)

---

### Summary

An explicit `TODO` comment in the L1 transaction execution path acknowledges that deployment transactions are not yet supported. When an L1→L2 transaction with deployment intent is submitted, the bootloader silently processes it as a regular call to the `to` address (typically `0x0`), succeeds without deploying any contract, mints the full deposited ETH to the sender, and emits a success log to L1. The depositor loses their gas fee and receives no deployed contract, while the chain records the transaction as successful — a permanent, unrevocable state-transition divergence.

---

### Finding Description

In `execute_l1_transaction_and_notify_result`, after minting the value portion of the deposit to the sender, the bootloader dispatches the transaction body via `run_single_interaction`:

```rust
// TODO: add support for deployment transactions,
// probably unify with execution logic for EOA
let (reverted, mut returndata) =
    match BasicBootloader::<S, ZkTransactionFlowOnlyEOA<S>>::run_single_interaction(
        system,
        system_functions,
        memories.reborrow(),
        calldata,
        &from,
        &to,          // ← for a deployment tx, this is 0x0
        resources_for_tx,
        &value,
        false,        // ← not a constructor frame
        tracer,
        validator,
    )
``` [1](#0-0) 

The `false` flag means no constructor frame is opened. For a deployment transaction, `to` is `0x0`. A call to address `0x0` in the EVM model is a call to an empty account, which returns success with no returndata. No contract is deployed. The bootloader then proceeds through the full post-execution accounting path — operator fee payment, refund minting, and L1 log emission — all as if the transaction succeeded normally. [2](#0-1) 

The `is_priority_op` guard that prevents upgrade transactions from failing silently does not apply here; priority-op deployment transactions are equally affected. [3](#0-2) 

The `mint_base_token` hook used for all L1 minting is itself flagged as a temporary pre-release solution: [4](#0-3) [5](#0-4)

### Impact Explanation

**Vulnerability class:** State-transition bug / valid-execution unprovability.

An L1 deployment transaction is irreversible once included in the priority queue. The L1 contracts enforce that priority-op transactions cannot be invalidated (halting the chain if they are). The bootloader therefore processes the malformed deployment as a successful call, emits a success `L1→L2TxLog`, and mints the full deposit minus the operator fee. The depositor:

- Pays the full gas fee to the operator.
- Receives the value portion of the deposit (if any) minted to their address.
- Gets **no deployed contract** at any address.
- Cannot retry: the nonce has been incremented and the L1 log records success.

Because the L1 log records success, any L1-side contract that gates further actions on the deployment succeeding (e.g., a factory that checks `extcodesize` after the L1→L2 message) will proceed incorrectly, potentially locking or misrouting funds on L1.

---

### Likelihood Explanation

Any user or protocol that submits an L1→L2 deployment transaction (e.g., deploying a contract via the L1 bridge or a cross-chain factory) triggers this path. The L1 priority queue is a standard, publicly accessible interface. No privileged access is required. The likelihood is **medium-high** for any deployment workflow that relies on L1→L2 contract creation.

---

### Recommendation

Implement the missing constructor dispatch for L1 deployment transactions, analogous to the EOA deployment path. Specifically:

1. Detect deployment intent from the transaction (e.g., `to == 0x0` or a dedicated flag in the ABI-encoded transaction).
2. Open a constructor frame (`should_make_frame = true`, `CallModifier::Constructor`) when dispatching via `run_single_interaction`.
3. Handle factory deps marking (the adjacent `// TODO: l1 transaction preparation (marking factory deps)` at line 139).
4. Add a test that submits an L1 deployment transaction and asserts `extcodesize(deployed_address) > 0`. [6](#0-5) 

Also resolve the `mint_base_token` temporary hook before production: [4](#0-3)

### Proof of Concept

1. Construct an L1→L2 ABI-encoded transaction with `to = 0x0000…0000`, `calldata = <EVM deployment bytecode>`, `value = 1 ETH`, `gasLimit = 500_000`.
2. Submit via the L1 priority queue (standard unprivileged path).
3. Observe in the forward runner (`run_block`) that:
   - `tx_result.status == true` (success).
   - `extcodesize(expected_deployed_address) == 0` — no contract deployed.
   - Sender balance increased by `1 ETH - fee` (deposit minted).
   - L1 log records the transaction as successful.
4. The depositor has paid the fee and received no contract. The L1-side factory contract, if any, reads the success log and proceeds with incorrect state.

Entry path: `run_block` → `ZKHeaderStructureTxLoop::loop_op` → `process_transaction` → `process_l1_transaction` → `execute_l1_transaction_and_notify_result` → `run_single_interaction` (call to `0x0`, not constructor). [7](#0-6) [8](#0-7)
