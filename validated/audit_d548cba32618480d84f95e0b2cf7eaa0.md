### Title
`StatelessTransactionValidator` skips `max_l2_gas_amount` upper-bound check for `Declare` transactions, admitting transactions guaranteed to fail block execution - (`File: crates/apollo_gateway/src/stateless_transaction_validator.rs`)

### Summary

`StatelessTransactionValidator::validate_resource_bounds` explicitly exempts `RpcTransaction::Declare` from the `max_l2_gas_amount` upper-bound check. A user can submit a `Declare` transaction with `l2_gas.max_amount = u64::MAX` and a valid `max_price_per_unit ≥ min_gas_price`. The transaction passes all gateway and mempool admission checks but is guaranteed to fail `verify_can_pay_committed_bounds` during block building because the implied `max_possible_fee` is astronomically large. This is the direct Sequencer analog of the external report's "missing upper bound on a parameter causes stuck/wasted resources."

### Finding Description

In `validate_resource_bounds`, the upper-bound guard on `l2_gas.max_amount` is written as:

```rust
// TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
if let RpcTransaction::Declare(_) = tx {
    // ← empty arm: no check performed
} else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh { … });
}
``` [1](#0-0) 

The production default for `max_l2_gas_amount` is `1_210_000_000` (≈1.21 B L2 gas units). [2](#0-1) 

For every other transaction type (`Invoke`, `DeployAccount`) the check is enforced and the test `test_invalid_max_l2_gas_amount` covers only `TransactionType::DeployAccount` and `TransactionType::Invoke`, explicitly excluding `Declare`. [3](#0-2) 

The test `valid_l2_gas_amount_on_declare` even **asserts that a Declare with `max_amount = 200` passes when the configured limit is `100`**, confirming the exemption is intentional but unguarded. [4](#0-3) 

**Downstream path for an oversized Declare:**

1. **Stateless validator** — `max_l2_gas_amount` check skipped for `Declare`. Passes. [1](#0-0) 
2. **Stateful validator** — `validate_resource_bounds` only checks `l2_gas.max_price_per_unit ≥ threshold`; no upper bound on `max_amount`. Passes. [5](#0-4) 
3. **Blockifier `check_fee_bounds`** — checks only that `minimal_gas_amount ≤ resource_bounds.max_amount` (a lower-bound check). Passes trivially when `max_amount = u64::MAX`. [6](#0-5) 
4. **Blockifier `verify_can_pay_committed_bounds`** — computes `max_possible_fee = max_amount × max_price_per_unit`. With `max_amount = u64::MAX = 18_446_744_073_709_551_615` and `max_price_per_unit = 8_000_000_000` (the minimum), `max_possible_fee ≈ 1.47 × 10²⁸` FRI. No account holds this balance → **transaction fails here, inside block building**, after having consumed a mempool slot and batcher execution time. [7](#0-6) 

### Impact Explanation

**High. Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

Any unprivileged user can craft a `Declare` V3 transaction with `l2_gas.max_amount = u64::MAX` and `max_price_per_unit ≥ min_gas_price`. The gateway admits it, the mempool stores it, and the batcher wastes execution resources attempting to include it in a block, only to discard it at `verify_can_pay_committed_bounds`. At scale this is a low-cost DoS vector against the mempool and block-building pipeline: the attacker pays nothing (the transaction is never included) while the sequencer expends CPU and mempool capacity.

### Likelihood Explanation

**Low-to-Medium.** The attack requires only a valid sender address and a correctly signed `Declare` transaction with an oversized `max_amount` field. No privileged access is needed. The TODO comment in the source code confirms the developers are aware the check is missing but have not yet added it.

### Recommendation

Remove the `Declare` exemption and apply the same `max_l2_gas_amount` upper-bound check uniformly to all transaction types in `validate_resource_bounds`:

```rust
if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
        gas_amount: resource_bounds.l2_gas.max_amount,
        max_gas_amount: self.config.max_l2_gas_amount,
    });
}
```

If `Declare` transactions legitimately require a higher gas ceiling (e.g., for large Sierra programs), introduce a separate `max_l2_gas_amount_declare` config field with an appropriate bound rather than removing the check entirely.

### Proof of Concept

1. Construct an `RpcDeclareTransaction::V3` with:
   - `resource_bounds.l2_gas.max_amount = GasAmount(u64::MAX)`
   - `resource_bounds.l2_gas.max_price_per_unit = GasPrice(8_000_000_000)` (≥ `min_gas_price`)
   - All other fields valid (valid Sierra class, valid sender address, valid signature, etc.)
2. Submit via the gateway `add_declare_transaction` RPC endpoint.
3. Observe: `StatelessTransactionValidator::validate` returns `Ok(())` — the `MaxGasAmountTooHigh` error is never raised for `Declare`.
4. Observe: `StatefulTransactionValidator::validate_resource_bounds` returns `Ok(())` — only the price threshold is checked.
5. Observe: the transaction is accepted into the mempool.
6. During block building, `perform_pre_validation_stage` → `verify_can_pay_committed_bounds` fails because `max_possible_fee ≈ 1.47 × 10²⁸ FRI` exceeds any realistic account balance, and the transaction is discarded — after consuming batcher resources. [8](#0-7) [9](#0-8)

    fn validate_resource_bounds(
        &self,
        tx: &RpcTransaction,
    ) -> StatelessTransactionValidatorResult<()> {
        if !self.config.validate_resource_bounds {
            return Ok(());
        }

        let resource_bounds = *tx.resource_bounds();
        // The resource bounds should be positive even without the tip.
        if ValidResourceBounds::AllResources(resource_bounds).max_possible_fee(Tip::ZERO) == Fee(0)
        {
            return Err(StatelessTransactionValidatorError::ZeroResourceBounds { resource_bounds });
        }

        if resource_bounds.l2_gas.max_price_per_unit.0 < self.config.min_gas_price {
            return Err(StatelessTransactionValidatorError::MaxGasPriceTooLow {
                gas_price: resource_bounds.l2_gas.max_price_per_unit,
                min_gas_price: self.config.min_gas_price,
            });
        }

        // TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
        if let RpcTransaction::Declare(_) = tx {
        } else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
            return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
                gas_amount: resource_bounds.l2_gas.max_amount,
                max_gas_amount: self.config.max_l2_gas_amount,
            });
        }

        Ok(())
    }
