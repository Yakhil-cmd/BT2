### Title
Missing Permission Check and Empty Implementation in `MsgTurnBridge` Handler — Unprivileged Bypass of Bridge Circuit-Breaker - (File: x/cronos/keeper/msg_server.go)

### Summary
`MsgTurnBridge` is the on-chain circuit-breaker that the Cronos admin or a permissioned account is supposed to use to halt the Gravity bridge in an emergency. The handler implementation is a stub that unconditionally returns `nil, nil` — it performs no permission check and writes no state. Any unprivileged sender can submit the message and receive a success response, while the bridge state is never actually changed, permanently disabling the emergency-stop capability.

### Finding Description
`TurnBridge` in `x/cronos/keeper/msg_server.go` is defined as:

```go
// TurnBridge implements the grpc method
func (k msgServer) TurnBridge(goCtx context.Context, msg *types.MsgTurnBridge) (*types.MsgTurnBridgeResponse, error) {
    return nil, nil
}
```

Two distinct guards are absent:

1. **Missing authorization check.** Every other privileged handler in the same file enforces a guard before acting. `UpdateTokenMapping` calls `k.HasPermission(ctx, msg.GetSigners(), CanChangeTokenMapping)`. `UpdatePermissions` and `StoreBlockList` compare `msg.From` against the stored `CronosAdmin`. `TurnBridge` does none of this. The `CanTurnBridge` permission bit exists in the permissions system (`x/cronos/keeper/permissions.go`) and is documented in ADR-009 as the required permission for this message, but it is never consulted.

2. **Missing state write.** The handler never reads or writes any bridge-enabled flag. The Gravity bridge's end-blocker and EVM hooks that gate outbound transfers are expected to consult such a flag, but because the handler is a no-op, the flag is never set. The bridge therefore cannot be halted regardless of who calls the message.

### Impact Explanation
Two independent High-severity impacts follow:

- **Authorization bypass.** Because there is no permission check, any unprivileged account can submit `MsgTurnBridge` and receive a `code 0` success response. This bypasses the `CanTurnBridge` permission guard that the Cronos permissions system is designed to enforce.

- **Permanent inability to halt the bridge.** The bridge circuit-breaker is the only on-chain mechanism for stopping Gravity bridge flows in an emergency. Because the handler writes no state, the bridge can never be turned off through the intended protocol path. This is a permanent, protocol-level inability for the admin and validators to exercise emergency bridge control under normal network assumptions.

### Likelihood Explanation
The message is publicly routable via the standard Cosmos SDK transaction path. No special access, leaked keys, or off-chain coordination is required to trigger it. The missing implementation is reachable by any account that can submit a valid `MsgTurnBridge` transaction.

### Recommendation
1. Add a permission check at the top of `TurnBridge` mirroring the pattern used in `UpdateTokenMapping`:
   ```go
   if !k.HasPermission(ctx, msg.GetSigners(), CanTurnBridge) {
       return nil, errors.Wrap(sdkerrors.ErrUnauthorized, "msg sender is not authorized")
   }
   ```
2. Implement the bridge-enabled state write: store the `msg.Enable` boolean under a dedicated key and have the Gravity bridge end-blocker and EVM hooks gate outbound transfers on that flag.

### Proof of Concept
1. Any account submits `MsgTurnBridge{Sender: <unprivileged_addr>, Enable: false}`.
2. The handler at `x/cronos/keeper/msg_server.go:85-87` returns `nil, nil` immediately — no permission check, no state change.
3. The transaction is accepted on-chain with `code 0`.
4. The Gravity bridge continues processing outbound transfers as if the message was never sent, because no bridge-enabled flag was written.
