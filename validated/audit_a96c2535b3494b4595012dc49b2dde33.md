### Title
Malicious CKD Participant Can Corrupt Confidential Key Derivation Output Without Detection — (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

The CKD coordinator blindly aggregates participant contributions `(big_y, big_c)` without any individual verification. A single malicious participant can substitute arbitrary group elements, silently corrupting the derived confidential key that the coordinator outputs and the application accepts.

---

### Finding Description

In `do_ckd_coordinator`, the coordinator collects each participant's `(norm_big_y, norm_big_c)` pair and sums them unconditionally:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
Ok(Some(ckd_output))
``` [1](#0-0) 

Each honest participant is supposed to compute:

```
Y_i  = y_i · G
C_i  = x_i · H(pk ‖ app_id) + y_i · app_pk
```

and send `(λ_i · Y_i, λ_i · C_i)` to the coordinator. [2](#0-1) 

The coordinator has **no mechanism** to verify either claim:

1. It cannot verify `Y_i = y_i · G` (no proof of knowledge of `y_i`).
2. It cannot verify `C_i = x_i · H(pk ‖ app_id) + y_i · app_pk` without knowing the participant's private share `x_i` or the application's secret key `app_sk`.

The coordinator does hold the master public key but **not** individual verification shares (`x_i · G`), so it cannot check consistency of `C_i` against the participant's committed public contribution. No commitment, NIZK, or consistency check is performed before the output is finalized.

Compare this with the robust ECDSA presign, which performs exponent-interpolation consistency checks on every received share: [3](#0-2) 

No equivalent check exists in the CKD coordinator path.

---

### Impact Explanation

A malicious participant sends `(big_y', big_c')` of their choice. The coordinator computes:

```
Σ λ_j · Y_j  +  λ_m · Y_m'   (corrupted)
Σ λ_j · C_j  +  λ_m · C_m'   (corrupted)
```

The application then unmasks with `app_sk` and obtains:

```
(Σ λ_j · C_j + λ_m · C_m') − app_sk · (Σ λ_j · Y_j + λ_m · Y_m')
= msk · H(pk ‖ app_id)  +  λ_m · (C_m' − app_sk · Y_m')
```

The additive error term `λ_m · (C_m' − app_sk · Y_m')` is fully attacker-controlled (the attacker picks `C_m'` and `Y_m'`). The coordinator emits the corrupted `CKDOutput` with `Ok(Some(ckd_output))` — no error, no detection. [4](#0-3) 

Honest parties accept this output as valid. The TEE or application receives a wrong confidential key, which it cannot distinguish from a correct one without an out-of-band oracle.

**Matched impact**: *High — Corruption of CKD outputs so honest parties accept inconsistent or unusable cryptographic outputs.*

---

### Likelihood Explanation

Any participant enrolled in the CKD session can execute this attack. No special privilege, leaked key, or cryptographic break is required — the attacker simply sends two arbitrary `ElementG1` values instead of their honest contribution. The `recv_from_others` helper accepts any well-typed message from a known participant: [5](#0-4) 

The attack is single-round, requires no coordination, and leaves no detectable trace in the protocol transcript.

---

### Recommendation

1. **Short term**: Document explicitly that the CKD protocol assumes all participants are honest and that a single malicious participant can silently corrupt the output. Callers must enforce participant honesty at the application layer before invoking `ckd()`.

2. **Long term**: Add per-participant contribution proofs. Each participant should accompany `(norm_big_y, norm_big_c)` with a Chaum–Pedersen NIZK proving:
   - Knowledge of `y_i` such that `Y_i = y_i · G`.
   - Consistency: `C_i − vk_i · H(pk ‖ app_id) = y_i · app_pk`, where `vk_i` is the participant's public verification share from DKG.

   The coordinator verifies each proof before accumulating the contribution, mirroring the exponent-interpolation checks already present in the robust ECDSA presign.

---

### Proof of Concept

```
Setup: 3 participants P1 (honest), P2 (honest), P3 (malicious), coordinator = P1.

1. P1 and P2 compute honest (norm_big_y_i, norm_big_c_i) and send to P1.
2. P3 sends (ElementG1::identity(), ElementG1::identity()) — two zero elements.
3. P1 (coordinator) sums all three contributions without error.
4. ckd_output = CKDOutput::new(
       λ1·Y1 + λ2·Y2 + 0,
       λ1·C1 + λ2·C2 + 0
   )
5. Application unmasks: result = (λ1·C1 + λ2·C2) − app_sk·(λ1·Y1 + λ2·Y2)
                                = (msk − λ3·x3) · H(pk ‖ app_id)
   — a key that differs from the correct msk · H(pk ‖ app_id) by λ3·x3·H(pk ‖ app_id).
6. No error is returned; the application silently uses the wrong key.
```
