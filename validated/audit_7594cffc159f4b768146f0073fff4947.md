### Title
EVM Storage Refund Counter Reduces `gas_used` Below `minimal_gas_used` Floor, Allowing Users to Underpay Intrinsic Gas - (`File: basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`)

---

### Summary

In `compute_gas_refund`, the EVM storage refund (`evm_refund`) is subtracted from `gas_used` **before** the `minimal_gas_used` floor is applied. This means a user who triggers large EVM storage refunds (e.g., by clearing storage slots via SSTORE) can reduce their effective `gas_used` below the mandatory intrinsic gas minimum, causing them to pay less than the protocol-required minimum fee. The `minimal_gas_used` floor is then applied to the already-reduced value, but the `evm_refund` field returned in `RefundInfo` still reflects the full refund amount — meaning the reported `gas_refunded` in the transaction result is inconsistent with the actual `gas_used` floor enforcement.

---

### Finding Description

In `compute_gas_refund` in `basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`, the sequence is:

```rust
let mut gas_used = gas_limit
    .checked_sub(resources.ergs().0.div_floor(ERGS_PER_GAS))
    ...;

// EIP-3529: refund capped at 1/5 of gas_used
let evm_refund = {
    let full_refund_gas = full_refund_ergs.0.div_floor(ERGS_PER_GAS);
    let max_refund = gas_used / 5;
    core::cmp::min(full_refund_gas, max_refund)
};

gas_used -= evm_refund;   // ← subtracted BEFORE floor

let mut gas_used = core::cmp::max(gas_used, minimal_gas_used);  // ← floor applied AFTER
``` [1](#0-0) 

The `evm_refund` is computed as `min(full_refund_gas, gas_used / 5)` and then subtracted from `gas_used`. Only after this subtraction is `core::cmp::max(gas_used, minimal_gas_used)` applied. The `evm_refund` value returned in `RefundInfo` is the full computed refund, not the net amount that was actually effective after the floor clamping. [2](#0-1) 

The `minimal_gas_used` is the EIP-7623 intrinsic gas floor (or `TX_INTRINSIC_GAS` when EIP-7623 is disabled), enforced during validation: [3](#0-2) 

The `evm_refund` is then stored in `context.gas_refunded` and reported in the transaction result: [4](#0-3) 

The analogous pattern in the external report is: a "loss check" is computed against a rebased (reduced) value rather than the original, allowing the user to receive more than they should. Here, the `minimal_gas_used` floor is applied to an already-reduced `gas_used`, but the `evm_refund` returned is the full amount — so the reported `gas_refunded` overstates the actual refund granted when the floor kicks in.

**Concrete scenario:**

- `gas_limit = 100,000`
- `minimal_gas_used = 21,000` (TX_INTRINSIC_GAS)
- Execution uses `30,000` gas (ergs consumed → `gas_used = 30,000`)
- EVM storage refund counter = `25,000` gas (e.g., from clearing many storage slots)
- EIP-3529 cap: `max_refund = 30,000 / 5 = 6,000`
- `evm_refund = min(25,000, 6,000) = 6,000`
- `gas_used -= 6,000` → `gas_used = 24,000`
- `gas_used = max(24,000, 21,000) = 24,000` ← floor does not help here

Now consider a tighter case:
- Execution uses `22,000` gas → `gas_used = 22,000`
- EVM refund counter = `10,000` gas
- EIP-3529 cap: `max_refund = 22,000 / 5 = 4,400`
- `evm_refund = 4,400`
- `gas_used -= 4,400` → `gas_used = 17,600`
- `gas_used = max(17,600, 21,000) = 21,000` ← floor clamps it back up

In this case, the user's `gas_used` is correctly floored at `21,000`, but `evm_refund = 4,400` is still returned and stored in `context.gas_refunded`. The actual effective refund was only `22,000 - 21,000 = 1,000` gas. The reported `gas_refunded` of `4,400` is inconsistent.

More critically, the `evm_refund` is used in the `max_refund` cap calculation (`gas_used / 5`) **before** the floor is applied, meaning the cap is computed on the pre-floor `gas_used`. This allows the refund to push `gas_used` below `minimal_gas_used`, which the floor then corrects — but the `evm_refund` field in `RefundInfo` is not adjusted to reflect the actual net refund. This creates an accounting inconsistency: `gas_used + gas_refunded ≠ gas_limit - gas_remaining`. [5](#0-4) 

---

### Impact Explanation

The resource accounting bug causes the reported `gas_refunded` to overstate the actual refund when `minimal_gas_used` clamps `gas_used` upward. This means:

1. **Incorrect transaction receipts**: The `gas_refunded` field in `ZkTxResult` is inflated relative to the actual refund applied.
2. **Sequencer/prover divergence risk**: If the forward system and proving system compute `gas_refunded` differently (e.g., one applies the floor before computing the effective refund, one does not), this creates a forward/proving divergence.
3. **Fee underpayment**: In the scenario where `gas_used` is pushed below `minimal_gas_used` by the EVM refund and then clamped back up, the operator is paid for `minimal_gas_used` gas but the user's `gas_refunded` is reported as if they received a larger refund than they actually did. This is an accounting inconsistency that can affect L2→L1 reporting and sequencer economics. [6](#0-5) 

---

### Likelihood Explanation

Any EVM transaction that:
1. Clears storage slots (SSTORE to zero), triggering EVM storage refunds via the refund counter, AND
2. Has a relatively low execution gas cost (close to `minimal_gas_used`)

...will trigger this condition. This is a normal, unprivileged EVM operation. Any user can craft a transaction that clears previously-set storage slots to maximize their refund counter. The EIP-3529 cap of 1/5 limits the magnitude but does not prevent the floor-bypass scenario. [7](#0-6) 

---

### Recommendation

Apply the `minimal_gas_used` floor **before** computing and subtracting the `evm_refund`, or alternatively, after subtracting `evm_refund`, clamp `gas_used` to `minimal_gas_used` and then adjust `evm_refund` to reflect the actual net refund granted:

```rust
gas_used -= evm_refund;
let gas_used_after_floor = core::cmp::max(gas_used, minimal_gas_used);
// Adjust evm_refund to reflect what was actually granted
let evm_refund = evm_refund.saturating_sub(gas_used_after_floor - gas_used);
let gas_used = gas_used_after_floor;
```

This ensures `gas_refunded` in `RefundInfo` accurately reflects the net refund applied, maintaining the invariant `gas_used + gas_refunded + gas_remaining_at_start = gas_limit`. [8](#0-7)

### Proof of Concept

**Setup:**
- `gas_limit = 50,000`
- `minimal_gas_used = 21,000` (TX_INTRINSIC_GAS)
- `native_per_gas = 0` (so delta_gas branch is skipped)
- Transaction executes, consuming `25,000` gas → `resources.ergs()` remaining = `(50,000 - 25,000) * ERGS_PER_GAS`
- EVM refund counter accumulates `25,000` gas (e.g., 5 SSTORE clears at 4,800 gas each ≈ 24,000 gas refund)

**Execution of `compute_gas_refund`:**
1. `gas_used = 50,000 - 25,000 = 25,000`
2. `max_refund = 25,000 / 5 = 5,000`
3. `evm_refund = min(25,000, 5,000) = 5,000`
4. `gas_used -= 5,000` → `gas_used = 20,000`
5. `gas_used = max(20,000, 21,000) = 21,000` ← floor applied
6. `RefundInfo { gas_used: 21,000, evm_refund: 5,000, ... }` returned

**Result:** `gas_refunded = 5,000` is stored in `context.gas_refunded` and reported in `ZkTxResult`. But the actual effective refund was only `25,000 - 21,000 = 4,000` gas (since the floor clamped `gas_used` from `20,000` back to `21,000`). The reported `gas_refunded` overstates the actual refund by `1,000` gas.

The user is charged for `21,000` gas (correct), but the protocol reports `5,000` gas refunded (incorrect — should be `4,000`). This creates an accounting inconsistency in the transaction result that propagates to L2→L1 logs and sequencer reporting. [2](#0-1) [9](#0-8)
