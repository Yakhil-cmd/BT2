### Title
Rebasing NEP-141 Token Excess Permanently Locked in Aurora Bridge — (`engine/src/engine.rs`, `etc/eth-contracts/contracts/EvmErc20.sol`)

---

### Summary

When a rebasing NEP-141 token (one whose `ft_balance_of` for a given holder increases over time, such as stNEAR or any yield-bearing NEP-141) is bridged into Aurora via `ft_transfer_call` → `ft_on_transfer`, Aurora mints a fixed number of ERC-20 mirror tokens equal to the deposited amount. As the NEP-141 balance held by the Aurora contract grows due to rebasing, the ERC-20 supply does not grow to match. The excess rebased NEP-141 tokens accumulate in Aurora's NEAR account with no mechanism to withdraw or distribute them, resulting in permanent freezing of those funds.

---

### Finding Description

**Deposit path (NEP-141 → ERC-20):**

`ft_on_transfer` in `engine/src/contract_methods/connector.rs` dispatches to `engine.receive_erc20_tokens`. That function reads `args.amount` — the amount reported by the NEP-141 contract at transfer time — and mints exactly that many ERC-20 tokens to the recipient:

```rust
// engine/src/engine.rs  receive_erc20_tokens
let amount = args.amount.as_u128();
// ...
setup_receive_erc20_tokens_input(&recipient, amount)  // mints `amount` ERC-20
```

The Aurora NEAR account now holds `amount` NEP-141 tokens. If the NEP-141 is rebasing, Aurora's NEP-141 balance silently grows beyond `amount` over time.

**Withdrawal path (ERC-20 → NEP-141):**

`withdrawToNear` in `EvmErc20.sol` burns exactly the caller-supplied `amount` of ERC-20 and encodes that same `amount` into the precompile call:

```solidity
// etc/eth-contracts/contracts/EvmErc20.sol
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);
    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    // calls ExitToNear precompile with `amount`
```

The `ExitToNear` precompile in `engine-precompiles/src/native.rs` then issues an `ft_transfer` for exactly `exit_params.amount` NEP-141 tokens:

```rust
// engine-precompiles/src/native.rs  exit_erc20_token_to_near
format!(
    r#"{{"receiver_id":"{}","amount":"{}"}}"#,
    exit_params.receiver_account_id,
    exit_params.amount.as_u128()   // ← always the burn amount, never the rebased balance
),
```

**The gap:** The total ERC-20 supply is bounded by the originally deposited amount. Users can only burn ERC-20 tokens that were minted, so they can only ever reclaim the original deposit. The rebased excess — the difference between Aurora's actual NEP-141 balance and the total ERC-20 supply — has no withdrawal path. There is no function anywhere in the engine, connector, or precompile layer to query Aurora's actual NEP-141 balance and transfer the surplus.

---

### Impact Explanation

The rebased excess NEP-141 tokens accumulate in Aurora's NEAR account and are permanently inaccessible. No user, no admin, and no contract function can recover them. This constitutes **permanent freezing of funds** (Critical).

The frozen amount grows monotonically with time for every rebasing NEP-141 that has been bridged. For a token like stNEAR (which accrues staking rewards continuously), the locked surplus compounds indefinitely.

---

### Likelihood Explanation

NEAR has live rebasing NEP-141 tokens (e.g., Meta Pool's stNEAR). The bridge is a general-purpose mechanism: `deploy_erc20_token` accepts any NEP-141 account ID, and `ft_on_transfer` accepts any NEP-141 predecessor. Any user who bridges a rebasing NEP-141 to Aurora triggers this condition. No special permissions or unusual conditions are required — the standard deposit flow is the entry path.

---

### Recommendation

1. Track the NEP-141 balance Aurora holds per token (or query it cross-contract) and allow the ERC-20 total supply to reflect the rebased balance, so users can withdraw the full accrued amount.
2. Alternatively, add an admin-callable function that queries Aurora's actual NEP-141 balance for a given token, computes the surplus over the ERC-20 total supply, and transfers that surplus to a designated beneficiary or distributes it pro-rata.
3. At minimum, document that rebasing NEP-141 tokens are not supported and add a registry check in `deploy_erc20_token` / `ft_on_transfer` to reject known rebasing tokens.

---

### Proof of Concept

1. Deploy a rebasing NEP-141 token `rebase.near` whose `ft_balance_of("aurora")` increases by 10% per epoch.
2. User calls `ft_transfer_call` on `rebase.near` with `amount = 1000`, `receiver_id = "aurora"`, `msg = <evm_address>`.
3. Aurora's `ft_on_transfer` fires; `receive_erc20_tokens` mints 1000 ERC-20 tokens to `<evm_address>`. Aurora holds 1000 NEP-141.
4. One epoch passes. `rebase.near`'s `ft_balance_of("aurora")` is now 1100. ERC-20 total supply is still 1000.
5. User calls `withdrawToNear(recipient, 1000)` on the ERC-20 contract. Burns 1000 ERC-20. Precompile calls `ft_transfer` for 1000 NEP-141. Succeeds.
6. Aurora's NEP-141 balance is now 100. ERC-20 total supply is 0. The 100 rebased tokens are permanently locked in Aurora's NEAR account with no callable function to recover them.

#[named]
pub fn ft_on_transfer<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        require_running(&state::get_state(&io)?)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();
        let mut engine: Engine<_, _> = Engine::new(
            predecessor_address(&predecessor_account_id),
            current_account_id.clone(),
            io,
            env,
        )?;

        sdk::log!("Call ft_on_transfer");

        let args: FtOnTransferArgs = read_json_args(&io)?;
        let result = if predecessor_account_id == get_connector_account_id(&io)? {
            engine.receive_base_tokens(&args)
        } else {
            engine.receive_erc20_tokens(
                &predecessor_account_id,
                &args,
                &current_account_id,
                handler,
            )
        };

        #[allow(clippy::used_underscore_binding)]
        let amount_to_return = if let Err(_err) = &result {
            sdk::log!("Error in ft_on_transfer: {_err:?}");
            // An error occurred, so we need to return the amount of tokens to the sender.
            args.amount.as_u128()
        } else {
            // Everything is ok, so return 0.
            0
        };

        let output = crate::prelude::format!("\"{amount_to_return}\"");
        io.return_output(output.as_bytes());

        // In case of an error, we just return Ok(None) to avoid a panic in the contract. It's ok
        // because in case of an error, we already returned the amount of tokens to the sender.
        Ok(result.unwrap_or(None))
    })
}
