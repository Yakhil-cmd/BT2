### Title
Unbounded Loop Over Pool Member Balance Trace in `calculate_rewards` Can Permanently Freeze Unclaimed Yield — (File: `src/pool/pool.cairo`)

### Summary
The `calculate_rewards` internal function in `pool.cairo` contains an explicitly unbounded loop over a pool member's `pool_member_epoch_balance` trace. A pool member who makes repeated balance changes across many epochs without claiming rewards will grow this trace without bound. Once the trace is long enough, every call to `claim_rewards` (and `exit_delegation_pool_action`, which also claims rewards) will exceed the Starknet transaction gas limit, permanently freezing the pool member's unclaimed yield and blocking their exit.

### Finding Description
`calculate_rewards` iterates over every entry in `pool_member_epoch_balance` from `entry_to_claim_from` up to the current trace length:

```cairo
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
    if pool_member_checkpoint.epoch() >= until_epoch {
        break;
    }
    ...
    entry_to_claim_from += 1;
}
``` [1](#0-0) 

The `entry_to_claim_from` index is stored in the pool member's checkpoint and advances only when rewards are successfully claimed. Each call to `add_to_delegation_pool` or `exit_delegation_pool_intent` in a new epoch appends a new entry to `pool_member_epoch_balance`. If a pool member makes one balance change per epoch and never claims rewards, the unclaimed window grows by one entry per epoch. The developers themselves acknowledge this risk with the comment "This loop is unbounded but unlikely to exceed gas limits."

The `claim_rewards` function in pool.cairo calls `calculate_rewards` directly. `exit_delegation_pool_action` also triggers reward calculation as part of the exit flow. Both paths are blocked if the loop is too long.

### Impact Explanation
Once the `pool_member_epoch_balance` trace grows beyond the gas-per-transaction ceiling, every invocation of `claim_rewards` and `exit_delegation_pool_action` for that pool member will revert. The pool member's accumulated unclaimed yield is permanently frozen — they can never retrieve it. Additionally, because `exit_delegation_pool_action` also traverses this path, the pool member's principal delegation may also become unrecoverable, constituting a temporary (and potentially permanent) freeze of funds.

This matches the allowed impact: **Permanent freezing of unclaimed yield** and **Temporary freezing of funds**.

### Likelihood Explanation
A pool member who actively manages their position — entering, partially exiting, re-entering across many epochs — without regularly claiming rewards will organically grow their trace. On Starknet, epochs are relatively short (on the order of hours to days). A pool member active for hundreds of epochs with frequent balance changes and infrequent reward claims can reach a trace length that makes the loop prohibitively expensive. This is a realistic scenario for long-term, active delegators. The risk is self-inflicted but irreversible once the threshold is crossed.

### Recommendation
1. **Checkpoint on every balance change**: Store the current cumulative-rewards index at the time of each balance change so that `calculate_rewards` only needs to iterate over entries since the last *claim*, not since the last *checkpoint*. This is already partially done but the window between claims is the unbounded part.
2. **Paginated claiming**: Introduce a `claim_rewards_up_to(epoch)` variant that allows partial reward collection, advancing `entry_to_claim_from` in bounded increments so a pool member can drain a large trace over multiple transactions.
3. **Cap balance-change frequency**: Enforce a minimum epoch gap between balance changes to limit trace growth rate.

### Proof of Concept
1. Deploy the staking system and create a pool member.
2. For N epochs (e.g., N = 500), in each epoch:
   - Call `add_to_delegation_pool` with a small amount (creates a new trace entry).
   - Advance to the next epoch without calling `claim_rewards`.
3. After N epochs, call `claim_rewards` for the pool member.
4. Observe that `calculate_rewards` must iterate over all N entries in `pool_member_epoch_balance` in a single transaction.
5. At sufficiently large N, the transaction exceeds the Starknet gas limit and reverts.
6. All subsequent calls to `claim_rewards` and `exit_delegation_pool_action` also revert, permanently freezing the pool member's unclaimed yield.

        fn calculate_rewards(
            self: @ContractState,
            pool_member: ContractAddress,
            from_checkpoint: PoolMemberCheckpoint,
            until_checkpoint: PoolMemberCheckpoint,
            mut entry_to_claim_from: VecIndex,
        ) -> (Amount, VecIndex) {
            let pool_member_trace = self.pool_member_epoch_balance.entry(pool_member);
            // Note: `until_epoch` is the current epoch.
            let until_epoch = until_checkpoint.epoch();

            let mut rewards = 0;

            let pool_member_trace_length = pool_member_trace.length();

            let mut from_sigma = self.find_sigma(from_checkpoint, curr_epoch: until_epoch);
            let mut from_balance = from_checkpoint.balance();

            let base_value = self.staking_rewards_base_value.read();

            // **Note**: The loop iterates over the balance changes in the pool member's balance
            // trace. This loop is unbounded but unlikely to exceed gas limits.
            while entry_to_claim_from < pool_member_trace_length {
                let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
                // If the balance change is after `until_epoch` (and therefore does not affect
                // the current reward computation), exit the loop.
                if pool_member_checkpoint.epoch() >= until_epoch {
                    break;
                }

                // Compute rewards from (inclusive) the previous balance change (or from
                // `from_checkpoint`) to (exclusive) the current entry.
                let to_sigma = self.find_sigma(pool_member_checkpoint, curr_epoch: until_epoch);
                rewards +=
                    compute_rewards_rounded_down(
                        amount: from_balance, interest: to_sigma - from_sigma, :base_value,
                    );
                from_sigma = to_sigma;
                from_balance = pool_member_checkpoint.balance();
                entry_to_claim_from += 1;
            }

            // Compute the remaining rewards from (inclusive) the last visited balance change in
            // `pool_member_trace` (or from `from_checkpoint`) to (exclusive) `until_checkpoint`.
            let to_sigma = self.find_sigma(until_checkpoint, curr_epoch: until_epoch);
            rewards +=
                compute_rewards_rounded_down(
                    amount: from_balance, interest: to_sigma - from_sigma, :base_value,
                );

            (rewards, entry_to_claim_from)
        }
