### Title
Honest Participants Permanently Stuck in DKG Final Broadcast Round When a Peer Fails Mid-Protocol — (File: src/dkg.rs)

---

### Summary

A malicious participant can selectively send an invalid proof-of-knowledge or secret share to a subset of honest participants, causing those participants to abort before reaching the final `broadcast_success` round. The remaining honest participants are then permanently stuck inside `reliable_broadcast_receive_all`, waiting for a broadcast that never arrives. The protocol contains no timeout, no early-exit signal, and no emergency abort path — a direct structural analog to the "no emergency withdrawal" pattern in the external report.

---

### Finding Description

`do_keyshare` in `src/dkg.rs` drives all three DKG variants (keygen, reshare, refresh) through several sequential rounds. In the final round it calls `broadcast_success`, which internally calls `do_broadcast` → `reliable_broadcast_receive_all`. [1](#0-0) 

`broadcast_success` always transmits `(true, session_id)` — it has no path to signal failure. Its only purpose is to confirm that every participant reached this point successfully. [2](#0-1) 

Before reaching `broadcast_success`, each participant verifies every peer's proof-of-knowledge and secret share. If participant **A** receives an invalid proof from malicious participant **M**, `verify_proof_of_knowledge` returns `ProtocolError::MaliciousParticipant(M)`: [3](#0-2) 

This error propagates out of `do_keyshare`, so **A** never calls `broadcast_success` and never sends its `MessageType::Send` frame for that waitpoint.

Meanwhile, participants **B**, **C**, **D** — who received valid data from **M** — successfully complete all verification steps and enter `broadcast_success`. They block inside `reliable_broadcast_receive_all` waiting for **A**'s `Send` message: [4](#0-3) 

The receive loop catches all channel errors with `_ => continue` and retries indefinitely. The early-termination check for insufficient echo votes only fires once a `Send` message has already been received for a given session; since **A** never sends, that check is never evaluated for **A**'s session slot. The `vote_output` `ParticipantMap` is never marked full, and the function never returns. [5](#0-4) 

There is no timeout, no out-of-band abort signal, and no emergency recovery path anywhere in the protocol stack. The `ProtocolError` variants cover detection of misbehavior but provide no mechanism for stuck honest parties to exit. [6](#0-5) 

---

### Impact Explanation

**High — Permanent denial of key generation (and by extension reshare/refresh) for honest parties.**

Once **B**, **C**, **D** enter the final broadcast waitpoint, they cannot proceed, cannot abort, and cannot restart without external intervention (e.g., an application-layer timeout that is not part of the library). The DKG session is permanently dead for those participants. Because keygen output is a prerequisite for all subsequent signing operations, this also permanently denies signing capability for the affected participant set.

---

### Likelihood Explanation

**Medium.** The attack requires one malicious participant among the DKG set — within the standard adversary model for this library. The malicious participant need only send a syntactically valid but cryptographically invalid proof or share to a single target. This is trivially achievable by any participant who controls their own message content, requires no cryptographic break, and leaves no fingerprint distinguishing it from an honest network failure.

---

### Recommendation

1. **Add a protocol-level timeout** to `reliable_broadcast_receive_all` so that waiting participants can exit after a bounded interval and surface a recoverable error to the caller.
2. **Propagate abort signals**: when a participant detects a fatal error before `broadcast_success`, it should broadcast an explicit abort/failure frame so peers can exit immediately rather than waiting indefinitely.
3. **Identify the faulty participant**: the abort frame should carry the identity of the detected malicious participant so the coordinator can exclude them and restart the session without the full participant set, analogous to an emergency-rescue path.

---

### Proof of Concept

```
Participants: A (honest), B (honest), C (honest), M (malicious)
Threshold: 2

1. All four enter do_keyshare.
2. Round 1–3 complete normally (M behaves honestly).
3. In Round 4, M sends a valid share/proof to B and C,
   but sends an invalid proof to A.
4. A calls verify_proof_of_knowledge for M → returns
   ProtocolError::MaliciousParticipant(M).
   A exits do_keyshare with an error. A never calls broadcast_success.
5. B and C pass all verifications, reach broadcast_success,
   and call do_broadcast → reliable_broadcast_receive_all.
6. B and C loop on chan.recv(wait).await waiting for A's Send frame.
   A's protocol has already terminated; no Send frame ever arrives.
7. The `_ => continue` branch re-enters the loop on every channel error.
   vote_output never becomes full. The function never returns.
8. B and C are permanently stuck. No key is generated.
   M has achieved permanent denial of key generation for all honest parties
   at the cost of sending one malformed message.
``` [7](#0-6) [8](#0-7)

fn verify_proof_of_knowledge<C: Ciphersuite>(
    session_id: &HashOutput,
    domain_separator: &mut DomainSeparator,
    threshold: ReconstructionLowerBound,
    participant: Participant,
    old_participants: Option<ParticipantList>,
    commitment: &VerifiableSecretSharingCommitment<C>,
    proof_of_knowledge: Option<&Signature<C>>,
) -> Result<(), ProtocolError> {
    let threshold = threshold.value();
    match proof_of_knowledge {
        // if participant did not send anything but he is actually an old participant
        None => {
            // if basic dkg or participant is old
            if old_participants.is_none_or(|p| p.contains(participant)) {
                return Err(ProtocolError::MaliciousParticipant(participant));
            }
            // since previous line did not abort, then we know participant is new indeed
            // check the commitment length is threshold - 1
            if commitment.coefficients().len() != threshold - 1 {
                return Err(ProtocolError::IncorrectNumberOfCommitments);
            }
            // nothing to verify
            Ok(())
        }
        // now we know the proof is not none
        Some(proof_of_knowledge) => {
            // if participant sent something but he is actually a new participant
            if old_participants.is_some_and(|p| !p.contains(participant)) {
                return Err(ProtocolError::MaliciousParticipant(participant));
            }
            // since the previous did not abort, we know the participant is old or we are dealing with a dkg
            if commitment.coefficients().len() != threshold {
                return Err(ProtocolError::IncorrectNumberOfCommitments);
            }

            // creating an identifier as required by the syntax of verify_proof_of_knowledge of frost_core
            internal_verify_proof_of_knowledge(
                session_id,
                domain_separator,
                participant,
                commitment,
                proof_of_knowledge,
            )
        }
    }

async fn broadcast_success(
    chan: &mut SharedChannel,
    participants: &ParticipantList,
    me: Participant,
    session_id: HashOutput,
) -> Result<(), ProtocolError> {
    // broadcast node me succeded
    let vote_list = do_broadcast(chan, participants, me, (true, session_id)).await?;
    // unwrap here would never fail as the broadcast protocol ends only when the map is full
    let vote_list = vote_list
        .into_vec_or_none()
        .ok_or_else(|| ProtocolError::AssertionFailed("vote_list is empty".to_string()))?;
    // go through all the list of votes and check if any is fail or some does not contain the session id

    if !vote_list.iter().all(|(_, ref sid)| sid == &session_id) {
        return Err(ProtocolError::AssertionFailed(
            "A participant
                broadcast the wrong session id. Aborting Protocol!"
                .to_string(),
        ));
    }

    if !vote_list.iter().all(|&(boolean, _)| boolean) {
        return Err(ProtocolError::AssertionFailed(
            "A participant
                seems to have failed its checks. Aborting Protocol!"
                .to_string(),
        ));
    }
    // Wait for all the tasks to complete
    Ok(())
}
