### Title
Restricted Contract Deployment Authorization Bypassed via Contract Updates - (File: `fvm/environment/contract_updater.go`)

### Summary
When restricted contract deployment is enabled on the Flow network, the authorized-accounts allowlist is enforced only for **initial** contract deployments. Contract **updates** bypass the authorization check entirely. Any account that already holds a deployed contract — regardless of whether it appears on the authorized list — can replace that contract's code with arbitrary new code.

### Finding Description
The FVM enforces a contract deployment allowlist through `ContractUpdaterImpl.SetContract`. The check reads the authorized-addresses list stored at `ContractDeploymentAuthorizedAddressesPath` on the service account and rejects the call when the signing accounts do not include an authorized address.

However, the guard is conditioned on `!exists`:

```go
// fvm/environment/contract_updater.go
func (updater *ContractUpdaterImpl) SetContract(
    location common.AddressLocation,
    code []byte,
    signingAccounts []flow.Address,
) error {
    // Initial contract deployments must be authorized by signing accounts.
    //
    // Contract updates are always allowed.
    exists, err := updater.accounts.ContractExists(location.Name, flow.Address(location.Address))
    ...
    if !exists && !updater.isAuthorizedForDeployment(signingAccounts) {
        return fmt.Errorf(
            "deploying contract failed: %w",
            errors.NewOperationAuthorizationErrorf(...))
    }
    // ← no authorization check when exists == true
    updater.draftUpdates[location] = ContractUpdate{Location: location, Code: code}
    return nil
}
``` [1](#0-0) 

When `exists == true` (i.e., the contract already exists at that address), `isAuthorizedForDeployment` is never called, and the new code is accepted unconditionally.

The authorized-accounts list is managed by `SetContractDeploymentAuthorizersTransaction`, which stores an `[Address]` array at `ContractDeploymentAuthorizedAddressesPath` on the service account. [2](#0-1) 

The restriction flag itself is toggled via `SetIsContractDeploymentRestrictedTransaction` and read back in `getIsContractDeploymentRestricted`. [3](#0-2) 

### Impact Explanation
When restricted deployment is active, the intent is that only explicitly authorized accounts may introduce new Cadence code onto the network. Because updates skip the authorization check, any account that already holds a contract — obtained during bootstrapping, during a window when restrictions were temporarily disabled, or through any other prior mechanism — can replace that contract's bytecode with arbitrary new logic. This allows unauthorized code to be introduced into the live contract set, defeating the purpose of the deployment restriction.

**Impact: Medium** — unauthorized code modification of existing on-chain contracts.

### Likelihood Explanation
The precondition is that the attacker controls an account that already has at least one contract deployed. This is realistic because:

1. During network bootstrapping many system and user contracts are deployed to accounts that are not necessarily on the authorized list.
2. If the restriction flag is toggled on after an open deployment period, every account that deployed during that period retains the ability to update.
3. The `create-and-setup-node.cdc` template and similar bootstrap scripts deploy contracts to freshly created accounts that are not added to the authorized list. [4](#0-3) 

**Likelihood: Medium** — the precondition is non-trivial but arises naturally from normal network operation.

### Recommendation
Apply `isAuthorizedForDeployment` to updates as well as initial deployments, or introduce a separate `isAuthorizedForUpdate` check that can be configured independently. The guard in `SetContract` should be:

```go
if !updater.isAuthorizedForDeployment(signingAccounts) {
    return fmt.Errorf("deploying/updating contract failed: %w", ...)
}
```

If unrestricted updates are intentional by design (analogous to the Ion Protocol team's "will not fix" stance), the comment and documentation should explicitly state that the authorized list only governs initial deployments, so operators are not misled about the scope of the restriction.

### Proof of Concept

1. Enable restricted deployment on a test network:
   ```
   SetIsContractDeploymentRestrictedTransaction(serviceAddr, true)
   ```
2. Set the authorized list to contain only the service account:
   ```
   SetContractDeploymentAuthorizersTransaction(serviceAddr, [serviceAddr])
   ```
3. Confirm that account `0xALICE` (not on the list) **cannot** deploy a new contract — the FVM returns `"deploying contracts requires authorization from specific accounts"`. [5](#0-4) 

4. Now deploy any contract to `0xALICE` while restrictions are temporarily off (or during bootstrapping).
5. Re-enable restrictions.
6. Submit a transaction signed only by `0xALICE` that calls `account.contracts.update(name: "Foo", code: <malicious_code>)`.
7. Observe that `SetContract` is called with `exists == true`, the authorization check is skipped, and the malicious code is committed — confirmed by the explicit code path in `Commit()`: [6](#0-5)

	enabled, defined := impl.getIsContractDeploymentRestricted()

		// If the contract deployment bool is not set by the state
		// fallback to the default value set by the configuration
		// after the contract deployment bool is set by the state on all

		return impl.RestrictContractDeployment
	}
	return enabled

// GetIsContractDeploymentRestricted returns if contract deployment
// restriction is defined in the state and the value of it
func (impl *contractUpdaterStubsImpl) getIsContractDeploymentRestricted() (
	restricted bool,
	defined bool,
) {
	service := impl.chain.ServiceAddress()

	runtime := impl.runtime.BorrowCadenceRuntime()
	defer impl.runtime.ReturnCadenceRuntime(runtime)

	value, err := runtime.ReadStored(
		common.Address(service),
		blueprints.IsContractDeploymentRestrictedPath)
	if err != nil {
		impl.logger.
			Debug().
			Msg("Failed to read IsContractDeploymentRestricted from the " +
				"service account. Using value from context instead.")
		return false, false
	}
	restrictedCadence, ok := value.(cadence.Bool)
	if !ok {
		impl.logger.
			Debug().
			Msg("Failed to parse IsContractDeploymentRestricted from the " +
				"service account. Using value from context instead.")
		return false, false
	}
	restricted = bool(restrictedCadence)

	return restricted, true
}
