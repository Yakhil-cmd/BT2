### Title
Rewards Permanently Frozen in Pool Contract When Pool Balance Falls Below `min_delegation_for_rewards` — (`src/pool/pool.cairo`)

---

### Summary

When a delegation pool's total balance is below `min_delegation_for_rewards`, the staking contract still transfers STRK reward tokens to the pool contract, but `compute_rewards_per_unit` silently returns zero, preventing pool members from ever claiming those rewards. The transferred STRK is permanently frozen in the pool contract with no recovery path.

---

### Finding Description

In `pool.cairo`, `compute_rewards_per_unit` guards against overflow by returning zero when `total_stake < min_delegation_for_rewards`:

```cairo
// src/pool/pool.cairo lines 967–978
fn compute_rewards_per_unit(
    self: @ContractState, staking_rewards: Amount, total_stake: Amount,
) -> Index {
    if total_stake < self.min_delegation_for_rewards.read() {
        return Zero::zero();
    }
    ...
}
```

For STRK pools, `min_delegation_for_rewards = 10^18` (1 STRK). For BTC pools with 8 decimals, it is `10^3` (1000 satoshis).

`update_rewards_from_staking_contract` calls `compute_rewards_per_unit` and appends the result to `cumulative_rewards_trace`. When the result is zero, the trace value does not increase:

```cairo
// src/pool/pool.cairo lines 569–587
fn update_rewards_from_staking_contract(
    ref self: ContractState, rewards: Amount, pool_balance: Amount,
) {
    self.assert_caller_is_staking_contract();
    let (_, last) = self.cumulative_rewards_trace.last().unwrap();
    self.cumulative_rewards_trace.insert(
        key: self.get_current_epoch(),
        value: last + self.compute_rewards_per_unit(
            staking_rewards: rewards, total_stake: pool_balance,
        ),
    );
}
```

However, in `staking.cairo`, `update_pool_rewards` **unconditionally transfers** STRK to the pool contract **before** calling `update_rewards_from_staking_contract`:

```cairo
// src/staking/staking.cairo lines 1872–1888
for (pool_contract, token_address, pool_balance, pool_rewards) in pools_rewards_data {
    self.send_rewards_to_delegation_pool(
        :staker_address,
        pool_address: pool_contract,
        amount: pool_rewards,
        token_dispatcher: strk_token_dispatcher,
    );
    ...
    pool_dispatcher.update_rewards_from_staking_contract(
        rewards: pool_rewards,
        pool_balance: pool_balance.to_native_amount(:decimals),
    );
```

Because `cumulative_rewards_trace` does not increase, `calculate_rewards` in `pool.cairo` (lines 837–888) computes zero rewards for all pool members. There is no function in the pool contract to withdraw or recover stuck STRK — `claim_rewards` returns zero, `exit_delegation_pool_action` returns only principal, and `set_staker_removed` does not transfer rewards.

The code comment at line 960–966 acknowledges the behavior but does not address the token loss:

```
/// **Note**: Delegation rewards lost when pool balance is less than
/// `min_delegation_for_rewards`. The staking contract continues to forward
/// `pool_rewards` to the pool contract even in this case.
```

---

### Impact Explanation

STRK reward tokens are transferred to the pool contract but can never be claimed by pool members. This constitutes **permanent freezing of unclaimed yield** (High). The frozen amount accumulates every time `update_rewards` or `update_rewards_from_attestation_contract` is called while the pool balance remains below the threshold. There is no admin function, sweep mechanism, or recovery path in the pool contract.

---

### Likelihood Explanation

Any pool whose total delegated balance is below 1 STRK (for STRK pools) or below `10^(decimals−5)` (for BTC pools) triggers this path. This is reachable by any unprivileged delegator who enters a pool with a small amount, or by a pool that loses delegators over time. No special access is required — the trigger is the normal `update_rewards` call path, which is public and called every block in the consensus rewards regime.

---

### Recommendation

Before transferring rewards to the pool contract, check whether `pool_balance >= min_delegation_for_rewards`. If not, skip the transfer entirely (do not send tokens that cannot be distributed), or accumulate the undistributable rewards in a recoverable reserve. The transfer and the trace update must be kept consistent: tokens should only be sent when the trace will actually record a non-zero increment.

---

### Proof of Concept

1. Pool has 0.5 STRK total delegated (`5 × 10^17 < 10^18 = min_delegation_for_rewards`).
2. Anyone calls `update_rewards(staker_address, false)` on the staking contract.
3. `calculate_staker_pools_rewards` computes `pool_rewards_including_commission = total_rewards × (5×10^17) / total_stake` — non-zero for any realistic `total_rewards`.
4. `pool_rewards` is non-zero → appended to `pool_rewards_array`.
5. `update_pool_rewards` calls `send_rewards_to_delegation_pool`: STRK tokens are transferred to the pool contract.
6. `update_rewards_from_staking_contract` is called with `pool_balance = 5×10^17 < 10^18`.
7. `compute_rewards_per_unit` returns `0`; `cumulative_rewards_trace` is updated with the same value as before.
8. Pool member calls `claim_rewards` → `calculate_rewards` returns `0` because `to_sigma − from_sigma = 0`.

        fn compute_rewards_per_unit(
            self: @ContractState, staking_rewards: Amount, total_stake: Amount,
        ) -> Index {
            // Return zero if the total stake is too small, to avoid overflow below.
            if total_stake < self.min_delegation_for_rewards.read() {
                return Zero::zero();
            }
            mul_wide_and_div(
                lhs: staking_rewards, rhs: self.staking_rewards_base_value.read(), div: total_stake,
            )
                .expect_with_err(err: InternalError::REWARDS_COMPUTATION_OVERFLOW)
        }
