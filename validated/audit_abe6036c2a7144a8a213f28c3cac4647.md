### Title
`msg.value` Reused Across `delegatecall` Multicall Allows Registration Fee Bypass — (`smart-contracts-poc/contracts/oracles/providers/OracleBase.sol`)

---

### Summary

`OracleBase` inherits OpenZeppelin's `Multicall`, whose `multicall()` dispatches each call via `delegatecall`. Because `delegatecall` preserves the original `msg.value` in every sub-call, a caller can batch N invocations of the `payable` `register()` function in a single `multicall` transaction, paying the `registrationFee` only once while registering N pools.

---

### Finding Description

`OracleBase` inherits OpenZeppelin's `Multicall`: [1](#0-0) 

OpenZeppelin's `Multicall.multicall()` dispatches each element via `Address.functionDelegateCall(address(this), data[i])`. `delegatecall` does **not** create a new call context for `msg.value`; the original `msg.value` is visible in every sub-call.

`register()` is `payable` and enforces the fee with a single `msg.value` check: [2](#0-1) 

Because every `delegatecall` frame sees the same `msg.value`, an attacker who sends exactly `registrationFee` wei and batches N `register(...)` calls through `multicall` will pass the `msg.value >= registrationFee` guard N times while only paying once.

The comment on line 199–200 explicitly states overpayment is intentionally non-refunded, but the design assumes each `register` call arrives in its own transaction with its own `msg.value`. That assumption is broken by the inherited `delegatecall`-based multicall. [3](#0-2) 

---

### Impact Explanation

The registration fee is the protocol's primary economic barrier against oracle abuse: it gates which pools may call the abuse-protected `price(feedId, pool)` path. Bypassing it allows an attacker to whitelist an arbitrary number of pools at near-zero cost, undermining the abuse-protection model and causing direct loss of protocol fee revenue.

---

### Likelihood Explanation

The attack requires no special role or privilege. Any external caller can invoke `multicall` with a crafted batch. The only prerequisite is that the caller supplies a valid `factory` address and pools recognized by that factory, which is a normal operational requirement, not a meaningful barrier.

---

### Recommendation

Override `multicall` in `OracleBase` (or its concrete subclasses) to use regular `call` instead of `delegatecall`, or remove the `Multicall` inheritance entirely and replace it with a non-`delegatecall` batch helper. Alternatively, replace the `msg.value >= registrationFee` check with a pattern that charges a fixed fee per registration by pulling ERC-20 tokens or by tracking how much ETH has already been consumed within the current transaction.

---

### Proof of Concept

```solidity
// Attacker registers 5 pools paying registrationFee only once
bytes[] memory calls = new bytes[](5);
for (uint i = 0; i < 5; i++) {
    calls[i] = abi.encodeCall(
        OracleBase.register,
        (feedId, pools[i], factory)
    );
}
// msg.value == registrationFee, but register() passes 5 times
oracle.multicall{value: registrationFee}(calls);
// All 5 pools are now registeredPool[feedId][pools[i]] == true
// Protocol received registrationFee instead of 5 * registrationFee
``` [4](#0-3)

    /// @dev    Overpayment is NOT refunded: any msg.value above registrationFee is kept and is
    ///         withdrawable by ADMIN via withdrawEth. This is intentional.
    function register(bytes32 feedId, address pool, address factory) external payable {
        require(msg.value >= registrationFee, InsufficientFee(msg.value, registrationFee));
        require(pool != address(0));
        require(approvedFactories.contains(factory), FactoryNotApproved(factory));
        require(IPoolFactory(factory).isPool(pool), NotAPool(pool));

        if (blacklisted[pool]) {
            blacklisted[pool] = false;
            emit BlacklistUpdated(pool, false);
        }

        registeredPool[feedId][pool] = true;
        emit PoolRegistered(feedId, pool, msg.sender, msg.value);
    }
```
