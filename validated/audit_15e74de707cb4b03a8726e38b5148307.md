### Title
Malicious Participant Can Permanently Abort Coordinator Signing via Type-Mismatched MessagePack Message — (`src/ecdsa/ot_based_ecdsa/sign.rs`)

### Summary
A single malicious participant in the OT-based ECDSA signing protocol can send a syntactically valid but semantically wrong MessagePack payload at `wait0`, causing `recv_from_others::<Scalar>` to fail with a deserialization error that immediately and permanently aborts the coordinator's signing protocol.

### Finding Description
In `do_sign_coordinator`, the coordinator collects signature shares from all other participants via `recv_from_others::<Scalar>`: [1](#0-0) 

`recv_from_others` loops over all participants and calls `chan.recv(waitpoint).await?`: [2](#0-1) 

The `?` operator propagates any error from `chan.recv` immediately, with no per-sender isolation. Inside `Comms::recv`, deserialization is performed with `rmp_serde`: [3](#0-2) 

A deserialization failure (e.g., a valid MessagePack tuple where a `Scalar` is expected) is converted to `ProtocolError::Other(...)` via: [4](#0-3) 

This error propagates through `recv_from_others` and `do_sign_coordinator` via `?`, permanently aborting the coordinator.

### Impact Explanation
Any single participant in the signing session can abort the coordinator's signing protocol by sending one malformed-type message. Since `recv_from_others` has no mechanism to skip bad messages from individual senders, the entire signing round fails. The coordinator cannot produce a signature, constituting a permanent denial of signing for all honest parties in that round.

### Likelihood Explanation
The attack requires only that the attacker be a registered participant in the signing session — a standard role. The attacker sends one message with valid MessagePack framing but wrong type (e.g., a two-element array instead of a 32-byte scalar). No cryptographic material is needed. The attack is trivially repeatable across every signing attempt.

### Recommendation
In `recv_from_others`, catch deserialization errors per-sender and either skip the offending participant (logging them as malicious) or return a `ProtocolError::MaliciousParticipant(from)` that identifies the bad actor without aborting the entire loop. The `MaliciousParticipant` variant already exists in `ProtocolError`: [5](#0-4) 

Alternatively, `recv_from_others` could be modified to return partial results with per-sender error attribution, allowing callers to decide whether to abort or continue with remaining honest participants.

### Proof of Concept
```
1. Set up a 3-of-3 signing session with participants [A (coordinator), B, C].
2. Participant B sends a valid MessagePack payload encoding a 2-element tuple
   (e.g., [0x01, 0x02]) instead of a 32-byte Scalar at wait0.
3. Coordinator A's recv_from_others::<Scalar> receives B's message first.
4. rmp_serde::decode::from_slice::<Scalar> fails on the tuple.
5. The error propagates via `?` at helpers.rs:20 → do_sign_coordinator:114.
6. Coordinator returns Err(ProtocolError::Other("...")) before receiving C's share.
7. No signature is produced. Honest participant C's valid share is never consumed.
8. The attack is repeatable on every signing attempt as long as B participates.
```
