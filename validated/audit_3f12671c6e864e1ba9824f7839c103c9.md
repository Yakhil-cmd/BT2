### Title
Unauthorized Transfer and Burn of Native Bank Tokens via Missing Caller Validation in Bank Precompile - (File: x/cronos/keeper/precompiles/bank.go)

### Summary

The `BankContract` precompile at address `0x0000000000000000000000000000000000000064` exposes `transfer` and `burn` methods that accept an arbitrary `sender`/`recipient` address from ABI-decoded calldata without verifying it matches `contract.Caller()`. Any unprivileged EVM contract can invoke these methods to transfer or burn native bank tokens (`evm/<caller_contract>` denom) from any victim address without their authorization.

### Finding Description

In `bank.go`, the `transfer` case unpacks `sender` directly from ABI arguments and passes it as the `from` address to `bankKeeper.SendCoins`:

```go
// x/cronos/keeper/precompiles/bank.go lines 175-192
sender := args[0].(common.Address)   // ← from calldata, never verified
recipient := args[1].(common.Address)
amount := args[2].(*big.Int)
...
from := sdk.AccAddress(sender.Bytes())
to   := sdk.AccAddress(recipient.Bytes())
denom := EVMDenom(contract.Caller())  // "evm/<caller>"
...
bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt))
```

There is no guard of the form `require sender == contract.Caller()`. Any contract can supply an arbitrary victim address as `sender`.

The `burn` case has the same flaw: the address whose tokens are burned (`recipient` in the ABI, mapped to `addr`) is taken from calldata without checking it equals the caller:

```go
// x/cronos/keeper/precompiles/bank.go lines 121-149
recipient := args[0].(common.Address)   // ← from calldata, never verified
...
addr := sdk.AccAddress(recipient.Bytes())
denom := EVMDenom(contract.Caller())
...
bc.bankKeeper.SendCoinsFromAccountToModule(ctx, addr, types.ModuleName, ...)
bc.bankKeeper.BurnCoins(ctx, types.ModuleName, ...)
```

By contrast, the `mint` case is safe because minting to an arbitrary address does not debit anyone's balance.

### Impact Explanation

An unprivileged attacker deploys a contract at address `X`. The denom namespace they control is `evm/X`. Any user who holds `evm/X` tokens (received via IBC, CRC20 conversion, or a prior `mint` call) can have those tokens stolen or destroyed without consent:

- **Unauthorized transfer**: `bank.transfer(victim, attacker, amount)` moves `amount` of `evm/X` from `victim` to `attacker` via `bankKeeper.SendCoins`.
- **Unauthorized burn**: `bank.burn(victim, amount)` destroys `amount` of `evm/X` held by `victim` via `bankKeeper.SendCoinsFromAccountToModule` + `BurnCoins`.

This constitutes a **Critical** unauthorized transfer/burn of CRC20/EVM-native assets and a **High** bypass of the bank module's authorization model (only the token holder should be able to debit their own balance).

### Likelihood Explanation

The entry path is fully unprivileged: any EOA can deploy a contract and call the bank precompile directly. No admin key, governance vote, or validator compromise is required. The precompile is registered and reachable in every block.

### Recommendation

Add a caller-equality guard in both `transfer` and `burn` before executing the state change:

```go
// transfer
if sender != contract.Caller() {
    return nil, errors.New("sender must be the caller")
}

// burn
if recipient != contract.Caller() {
    return nil, errors.New("burn target must be the caller")
}
```

Alternatively, remove the `sender`/`recipient` argument entirely and derive the debit address from `contract.Caller()`, consistent with how `denom` is already derived.

### Proof of Concept

1. Alice holds 1000 `evm/0xATTACKER` tokens (e.g., received via IBC or a prior `mint` call from the attacker's contract).
2. Attacker's contract (deployed at `0xATTACKER`) calls the bank precompile:
   ```solidity
   IBankModule bank = IBankModule(0x0000000000000000000000000000000000000064);
   bank.transfer(alice, attacker, 1000);   // steals Alice's tokens
   // OR
   bank.burn(alice, 1000);                 // destroys Alice's tokens
   ```
3. `bankKeeper.SendCoins(ctx, alice, attacker, [{denom:"evm/0xATTACKER", amount:1000}])` executes with no authorization from Alice.
4. Alice's balance is debited; the attacker's balance is credited (or the tokens are destroyed), with no signature or approval from Alice.

The root cause is identical in class to the MsgValueSimulator bug: a privileged call-forwarding/execution mechanism (`BankContract.Run`) accepts caller identity from untrusted input (ABI calldata) rather than from the authenticated EVM execution context (`contract.Caller()`), bypassing the authorization invariant that only a token holder can debit their own balance.
