### Title
Malicious Participant Can Supply Inconsistent `app_id`/`app_pk` in CKD Protocol, Corrupting the Coordinator's Output — (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary
The CKD protocol aggregates per-participant shares without verifying that every participant used the same `app_id` and `app_pk`. A single malicious participant can silently substitute different values for either parameter, causing the coordinator to assemble a `CKDOutput` that is cryptographically incoherent and unusable by the legitimate application.

---

### Finding Description
Each participant independently invokes `ckd()` with its own locally-supplied `app_id` and `app_pk`. Inside `compute_signature_share`, the share is computed as:

```
hash_point = H(pk || app_id)
big_s      = hash_point * private_share          // depends on app_id
big_c      = big_s + app_pk * y                  // depends on app_pk
``` [1](#0-0) 

The coordinator then blindly sums every received `(norm_big_y, norm_big_c)` pair:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
``` [2](#0-1) 

There is no commitment, zero-knowledge proof, or any other mechanism that binds a participant's share to a specific `app_id` or `app_pk`. The coordinator has no way to detect that a participant deviated.

This is the direct analog of the reported vulnerability: just as the `SharesCooldown.finalize()` caller could supply a different output token than the one the user originally chose, any CKD participant can supply a different `app_id` or `app_pk` than the one the coordinator intended, and the protocol will silently incorporate the inconsistent contribution.

---

### Impact Explanation
When a malicious participant uses `app_id' ≠ app_id`, their contribution encodes `H(pk || app_id')` instead of `H(pk || app_id)`. The coordinator's aggregated `big_c` becomes a linear combination of shares computed over two different hash points. The result does not equal `msk · H(pk || app_id)` masked under any key, so `CKDOutput.unmask(app_sk)` yields garbage — the legitimate application receives an unusable confidential key.

When a malicious participant uses `app_pk' ≠ app_pk`, the masking term `y_i · app_pk'` is mixed into the sum, so the coordinator's `big_c` is no longer correctly masked under `app_pk`. Again, `unmask(app_sk)` yields garbage.

Both attacks corrupt the CKD output so honest parties accept an unusable cryptographic output, matching:

> **High: Corruption of … CKD outputs so honest parties accept … unusable cryptographic outputs.** [3](#0-2) 

---

### Likelihood Explanation
Any single participant in the protocol can mount this attack unilaterally. No special privilege, leaked key, or external assumption is required — the attacker only needs to call `ckd()` with a different `app_id` or `app_pk` than the one agreed upon out-of-band. Because the protocol performs no in-band consistency check, the attack is undetectable by the coordinator and by all honest participants.

---

### Recommendation
Before participants send their shares, introduce an in-band binding of each share to the agreed `app_id` and `app_pk`. Concretely:

1. **Commit-then-reveal**: In a preliminary broadcast round, each participant broadcasts `H(app_id || app_pk || session_id)`. The coordinator verifies all commitments match before accepting any share.
2. **Zero-knowledge proof**: Each participant attaches a proof that `norm_big_c` was computed using the agreed `hash_point = H(pk || app_id)` and the agreed `app_pk`, without revealing the random blinding scalar `y`.

Either approach mirrors the fix applied in the referenced report: persist the user's chosen parameter at request time and enforce it during the permissionless (or multi-party) completion step.

---

### Proof of Concept

```
Setup:
  participants = [P1 (coordinator), P2, P3 (malicious)]
  agreed app_id  = b"target_app"
  agreed app_pk  = target_pk

P1 calls ckd(..., app_id=b"target_app", app_pk=target_pk, ...)
P2 calls ckd(..., app_id=b"target_app", app_pk=target_pk, ...)
P3 calls ckd(..., app_id=b"evil_app",   app_pk=evil_pk,   ...)  // deviation

P3's share encodes H(pk || b"evil_app") * x3 + evil_pk * y3
P1 sums all three shares without detecting the inconsistency.

Resulting big_c:
  = λ1·(msk1·H(pk||target_app) + y1·target_pk)
  + λ2·(msk2·H(pk||target_app) + y2·target_pk)
  + λ3·(msk3·H(pk||evil_app)   + y3·evil_pk)

This is NOT equal to msk·H(pk||target_app) + big_y·target_pk.

CKDOutput.unmask(target_sk) ≠ msk·H(pk||target_app)
→ the legitimate application receives an unusable confidential key.
``` [4](#0-3) [5](#0-4)
