Audit Report

## Title
Reentrancy in GTC `claim_neurons` Violates CEI Pattern, Enabling Duplicate `claim_gtc_neurons` Dispatch - (File: `rs/nns/gtc/src/lib.rs`)

## Summary
The `claim_neurons` method in the Genesis Token Canister checks `account.has_claimed` before making an inter-canister call to the Governance canister, but sets `account.has_claimed = true` only after the call returns. Because IC canisters suspend execution at every `await` and can process other queued ingress messages during that suspension, two concurrent ingress messages from the same GTC account holder can both pass the `has_claimed` guard and both dispatch `claim_gtc_neurons` to the Governance canister before either sets the flag. This can result in duplicate neuron ownership-transfer requests for the same genesis neuron IDs.

## Finding Description
In `rs/nns/gtc/src/lib.rs`, the `claim_neurons` method on `Gtc` (lines 40–70) performs:

1. **Line 62–64**: Checks `account.has_claimed` — returns early if already claimed.
2. **Line 66**: `GovernanceCanister::claim_gtc_neurons(caller, account.neuron_ids.clone()).await?` — an inter-canister call via `dfn_core::api::call` that suspends the GTC canister.
3. **Line 68**: `account.has_claimed = true` — set only after the call returns.

The public update entry point is `canister_update claim_neurons` in `rs/nns/gtc/canister/canister.rs` (lines 144–153), which calls `gtc_mut().claim_neurons(&caller(), hex_pubkey).await`. The `gtc_mut()` function returns a `&'static mut Gtc` backed by a single global `static mut GTC`.

Because the IC execution model allows the canister to process other queued messages while suspended at the `await`, the following interleaving is possible:

```
Call A: has_claimed == false → dispatches claim_gtc_neurons([N1,N2]) → suspends
Call B: has_claimed == false (unchanged) → dispatches claim_gtc_neurons([N1,N2]) → suspends
Call A: resumes → sets has_claimed = true
Call B: resumes → sets has_claimed = true (no-op, but second dispatch already sent)
```

Both calls have already sent `claim_gtc_neurons` to the Governance canister with the same neuron IDs. The `?` operator on line 66 means `has_claimed` is only set if the call succeeds; if the second Governance call fails, Call B returns an error without setting the flag — but the duplicate dispatch has already occurred.

No existing guard prevents this: `has_claimed`, `has_donated`, and `has_forwarded` are all checked before the `await`, and none are set before the suspension point.

## Impact Explanation
The GTC canister manages genesis-allocated neurons representing real ICP stake. `claim_gtc_neurons` in the Governance canister transfers neuron ownership to the caller's principal. Dispatching it twice for the same account sends two ownership-transfer requests for the same neuron IDs to the NNS Governance canister. Depending on whether the Governance canister's `claim_gtc_neurons` is idempotent, this can result in corrupted neuron ownership state or unexpected error handling for those neuron IDs in the NNS. This matches the allowed High impact: **Unauthorized access to neurons, governance assets, wallets, identities, ledgers, or canister-controlled funds** — specifically, duplicate or unauthorized neuron ownership transfers in the NNS Governance canister.

## Likelihood Explanation
The attacker must hold a valid GTC account (possess the corresponding secp256k1 private key). They submit two or more concurrent ingress messages to `claim_neurons` before any response is processed. This is trivially achievable using standard IC agent tooling (e.g., `ic-agent` in Rust or `@dfinity/agent` in JS) by firing multiple update calls in parallel. No privileged access, admin keys, or threshold corruption is required. The only constraint is owning a GTC account with `has_claimed = false`.

## Recommendation
Set `account.has_claimed = true` **before** the inter-canister call, following the Checks-Effects-Interactions pattern:

```rust
if account.has_claimed {
    return Ok(account.neuron_ids.clone());
}

// Set state BEFORE the external call
account.has_claimed = true;

// Now make the inter-canister call
if let Err(e) = GovernanceCanister::claim_gtc_neurons(caller, account.neuron_ids.clone()).await {
    // Reset on failure, or leave set and require a recovery path
    account.has_claimed = false;
    return Err(e);
}

Ok(account.neuron_ids.clone())
```

Alternatively, use a `scopeguard`-style in-flight lock (set before `await`, cleared on error) as used in the ckBTC minter's `update_balance.rs`.
