### Title
Cross-VM Asset Loss via `EVM.mustRun()` Silently Accepting `failed` EVM Execution Status — (`fvm/evm/stdlib/contract.cdc`)

---

### Summary

`EVM.mustRun()` in `fvm/evm/stdlib/contract.cdc` is explicitly designed to **not** revert the enclosing Cadence transaction when the EVM execution status is `failed`. Any Cadence cross-VM bridge contract that commits a Cadence-side asset withdrawal before calling `EVM.mustRun()` — without checking the returned `Result.status` — will permanently lose the Cadence asset if the EVM execution fails.

---

### Finding Description

`EVM.mustRun()` (lines 844–851 of `fvm/evm/stdlib/contract.cdc`) calls `EVM.run()` and then asserts only that the result status is not `unknown` — it explicitly permits `Status.failed` to pass through without reverting the Cadence transaction:

```cadence
fun mustRun(tx: [UInt8], coinbase: EVMAddress): Result {
    let runResult = self.run(tx: tx, coinbase: coinbase)
    assert(
        runResult.status == Status.failed || runResult.status == Status.successful,
        message: "EVM.mustRun(): The provided transaction is not valid for execution"
    )
    return runResult
}
```

The inline comment at lines 838–842 confirms this by design:

> *"Note that this method does not rollback if transaction is executed but an vm error is reported as the outcome of the execution (status: failed)."* [1](#0-0) 

A `failed` EVM status means the EVM transaction was included in the EVM block (nonce consumed, gas charged) but the EVM execution itself reverted. The EVM-side state change is rolled back within EVM, but the **Cadence transaction continues executing normally** — no Cadence panic, no rollback.

The `Status` enum confirms this distinction:

> *"A failed tx/call is included in a block."* [2](#0-1) 

If a Cadence cross-VM bridge contract follows this pattern:
1. Withdraw or burn a Cadence NFT/FT vault (Cadence state change committed)
2. Call `EVM.mustRun()` to mint/transfer the EVM-side equivalent
3. EVM execution fails (`status: failed`, e.g., EVM contract reverts or runs out of gas)
4. Cadence code does not check `runResult.status`

The Cadence asset is permanently destroyed with no EVM-side asset minted. The Cadence transaction does not revert.

The `BridgeAccessor` interface in the same file exposes `depositNFT`, `withdrawNFT`, `depositTokens`, and `withdrawTokens` as the canonical cross-VM bridge entry points: [3](#0-2) 

These bridge operations are reachable by any unprivileged Cadence transaction via `CadenceOwnedAccount.depositNFT()` / `withdrawNFT()`: [4](#0-3) 

The handler layer confirms that `failed` EVM results are not treated as fatal — `panicOnErrorOrInvalidOrFailedState` is only called for internal direct calls (deposit/withdraw/transfer), not for `Run()` which is the path used by `EVM.mustRun()`: [5](#0-4) 

---

### Impact Explanation

**Cross-VM asset loss**: a user's Cadence NFT or fungible token is permanently destroyed without the corresponding EVM asset being minted or transferred. This is a one-sided bridge failure — the Cadence side commits while the EVM side silently fails. The asset is unrecoverable because Cadence resource linearity means a destroyed resource cannot be recreated.

---

### Likelihood Explanation

`EVM.mustRun()` is `access(all)` and is the natural function for Cadence contracts that want to enforce "the EVM tx must be included in a block" (as opposed to `EVM.run()` which also allows `invalid`). A bridge contract author who reads the name `mustRun` may reasonably assume it guarantees success, not merely block inclusion. The EVM call can fail due to:

- **Out-of-gas**: gas limit set too low in the EVM transaction bytes
- **EVM contract revert**: the EVM-side bridge contract is paused, has a reentrancy guard triggered, or has a bug
- **Any other EVM execution error** (see `ExecutionErrCode*` range 301–400) [6](#0-5) 

An unprivileged user can trigger this by submitting a bridge transaction where the EVM-side contract is in a state that causes revert — for example, by front-running with a direct EVM call that exhausts a rate limit or triggers a guard in the EVM bridge contract, then submitting the Cadence bridge transaction whose `EVM.mustRun()` call will fail.

---

### Recommendation

1. Add a `mustRunSuccessful` variant that panics on `failed` status, making it safe for cross-VM bridge patterns:
   ```cadence
   fun mustRunSuccessful(tx: [UInt8], coinbase: EVMAddress): Result {
       let runResult = self.run(tx: tx, coinbase: coinbase)
       assert(
           runResult.status == Status.successful,
           message: "EVM.mustRunSuccessful(): EVM execution failed"
       )
       return runResult
   }
   ```
2. Rename `mustRun` to `runAndInclude` to signal it only guarantees block inclusion, not execution success.
3. All bridge contracts using `EVM.mustRun()` must explicitly check `runResult.status == Status.successful` and panic/revert if not.

---

### Proof of Concept

```cadence
// Vulnerable cross-VM bridge pattern using EVM.mustRun()
transaction {
    prepare(signer: auth(Storage) &Account) {
        // Step 1: Withdraw Cadence NFT — Cadence state change committed
        let nft <- signer.storage.load<@MyNFT>(from: /storage/myNFT)!

        // Step 2: Burn the Cadence NFT (irreversible under Cadence resource linearity)
        destroy nft

        // Step 3: Call EVM.mustRun() to mint EVM equivalent
        let evmTx: [UInt8] = buildMintTx()
        let coinbase = EVM.EVMAddress(bytes: [...])
        let result = EVM.mustRun(tx: evmTx, coinbase: coinbase)

        // BUG: result.status may be Status.failed (EVM execution reverted)
        // EVM.mustRun() does NOT revert the Cadence transaction on Status.failed.
        // The Cadence NFT is permanently lost with no EVM NFT minted.
        //
        // Correct implementation requires:
        // assert(result.status == EVM.Status.successful, message: "EVM mint failed")
    }
}
```

The root cause is at `fvm/evm/stdlib/contract.cdc` lines 844–851: the `assert` passes for `Status.failed`, so the Cadence transaction continues and commits the Cadence-side destruction with no EVM-side asset created. [7](#0-6)

```go
func panicOnErrorOrInvalidOrFailedState(res *types.Result, err error) {

	if res != nil && res.Invalid() {
		panic(fvmErrors.NewEVMError(res.ValidationError))
	}

	if res != nil && res.Failed() {
		panic(fvmErrors.NewEVMError(res.VMError))
	}

	// this should never happen
	if err == nil && res == nil {
		panic(fvmErrors.NewEVMError(types.ErrUnexpectedEmptyResult))
	}

	panicOnError(err)
}
```
