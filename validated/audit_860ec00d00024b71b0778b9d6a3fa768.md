### Title
Unprivileged Caller Can Invoke `update_rewards` With `disable_rewards: true` to Permanently Freeze Block Reward Distribution — (File: `src/staking/staking.cairo`)

---

### Summary

The `update_rewards` function in `IStakingRewardsManager` is publicly callable with no access control on the `disable_rewards` parameter. Any unprivileged caller can invoke `update_rewards(any_valid_staker, disable_rewards: true)` to advance the global `last_reward_block` to the current block without distributing any rewards. Because `last_reward_block` is a single global checkpoint, this blocks every other `update_rewards` call in the same block, denying all stakers their block rewards for that block. Repeated every block, this permanently freezes block reward accrual across the entire protocol.

---

### Finding Description

`update_rewards` is exposed as a public ABI function with no caller restriction beyond `general_prerequisites()` (pause check + zero-address check): [1](#0-0) 

Inside the function, `last_reward_block` is written to the current block **before** the `disable_rewards` branch is evaluated: [2](#0-1) 

The guard that prevents double-execution per block is: [3](#0-2) 

Because `last_reward_block` is a single global storage slot (not per-staker), once any caller advances it to block N with `disable_rewards: true`, every subsequent `update_rewards` call in block N — regardless of which staker it targets — reverts with `REWARDS_ALREADY_UPDATED`. No staker receives block rewards for that block.

The `general_prerequisites` helper confirms the only guards are pause state and zero-address: [4](#0-3) 

There is no check that the caller is the attesting staker, the consensus contract, or any other privileged address. The `disable_rewards` flag is fully attacker-controlled.

---

### Impact Explanation

`last_reward_block` is a global variable. A single call to `update_rewards(any_active_staker, disable_rewards: true)` per block is sufficient to:

1. Advance `last_reward_block` to the current block.
2. Distribute zero rewards to the named staker.
3. Cause every other `update_rewards` call in the same block to revert.

If an attacker repeats this every block, **no staker in the protocol ever accrues block rewards**. This constitutes **permanent freezing of unclaimed yield** (High impact under the allowed scope). Even sporadic execution constitutes griefing with measurable damage to stakers and the protocol (Medium impact).

---

### Likelihood Explanation

The attacker requires:
- A valid, active staker address (publicly readable from `stakers` vector or events).
- The ability to submit a transaction at the start of each block.
- Gas payment per block.

On Starknet, gas costs are low and block times are predictable. A simple bot can front-run the legitimate `update_rewards` call every block. No privileged key, leaked secret, or bridge compromise is needed. The entry path is fully permissionless.

---

### Recommendation

Restrict who may call `update_rewards` with `disable_rewards: true`. Options include:

1. **Require caller to be the named staker or an authorized consensus contract** — mirror the pattern used in `claim_rewards` and `update_rewards_from_attestation_contract`, which both enforce caller identity.
2. **Remove the `disable_rewards` parameter from the public interface** and handle the "no rewards" case internally based on on-chain state (e.g., `is_pre_consensus()`), so external callers cannot influence whether rewards are distributed.

---

### Proof of Concept

```
Block N:
  Attacker tx:  update_rewards(staker = <any active staker>, disable_rewards = true)
    → last_reward_block := N
    → rewards distributed: 0
    → returns early

  Legitimate staker tx (same block):  update_rewards(staker = <attesting staker>, disable_rewards = false)
    → assert!(N > last_reward_block)  // N > N is false → REVERTS with REWARDS_ALREADY_UPDATED

Block N+1: attacker repeats → same outcome
...
All stakers: block reward accrual = 0 indefinitely
```

The attacker pays one cheap Starknet transaction per block. No funds are stolen by the attacker; all block rewards are simply never distributed, permanently freezing yield for every staker and delegator in the protocol.
