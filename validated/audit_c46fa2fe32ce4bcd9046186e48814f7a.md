### Title
`setFeeCaps` Does Not Clamp Existing Per-Pool Admin Fees, Allowing Pool Admin to Maintain Above-Cap Fee Extraction — (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`setFeeCaps` auto-clamps the factory-level default protocol fee when a new cap is lower, but it never clamps per-pool `adminSpreadFeeE6` / `adminNotionalFeeE8` values already stored in `poolFeeConfig`. After the owner lowers `maxAdminSpreadFeeE6` or `maxAdminNotionalFeeE8`, every existing pool whose stored admin fee exceeds the new cap continues to charge and distribute that above-cap fee indefinitely. A pool admin can front-run the owner's `setFeeCaps` call to lock in the maximum old-cap fee before the reduction takes effect, and the permissionless `collectPoolFees` can then be called by anyone to extract those above-cap fees.

---

### Finding Description

`setFeeCaps` writes the four new cap values and then auto-clamps only the two factory-level defaults: [1](#0-0) 

The per-pool admin fee components stored in `poolFeeConfig[pool].adminSpreadFeeE6` and `poolFeeConfig[pool].adminNotionalFeeE8` are never touched by `setFeeCaps`. The only path that enforces the new admin cap on an existing pool is `setPoolProtocolFee`, which clamps admin fees as a side-effect when the owner explicitly calls it per pool: [2](#0-1) 

Until the owner calls `setPoolProtocolFee` for every affected pool, the pool's on-chain `spreadFeeE6` (the sum of protocol + admin components) remains above the new cap, and `collectPoolFees` — which is **permissionless** — distributes fees at those above-cap rates: [3](#0-2) 

The pool admin is semi-trusted only within caps. By front-running the owner's `setFeeCaps` call with a `setPoolAdminFees` call that sets fees at the current maximum, the pool admin can lock in an above-cap fee rate that persists until the owner manually remediates each pool. The check in `setPoolAdminFees` only prevents the admin from *raising* fees above the new cap after the fact; it does not retroactively lower already-stored above-cap values: [4](#0-3) 

The asymmetry is confirmed by the existing test `test_setPoolProtocolFee_clampsAdminSpreadWhenAboveCap`, which shows that clamping only happens when `setPoolProtocolFee` is called — not when `setFeeCaps` is called: [5](#0-4) 

---

### Impact Explanation

Every swap on an affected pool charges users a total spread fee (`spreadFeeE6 = protocolSpreadFeeE6 + adminSpreadFeeE6`) that exceeds the protocol-intended cap. The excess admin portion is transferred directly to the pool admin's fee destination via `collectPoolFees`. This constitutes a direct, ongoing extraction of user funds above the governance-set ceiling. The invariant `poolFeeConfig[pool].adminSpreadFeeE6 ≤ maxAdminSpreadFeeE6` is broken for all pools whose admin fees were set at or near the old cap.

---

### Likelihood Explanation

The owner reducing fee caps is a normal governance operation (e.g., responding to market conditions or competitive pressure). The pool admin observing a pending `setFeeCaps` transaction in the mempool and front-running with `setPoolAdminFees` at the old maximum is a straightforward MEV action requiring no special access. Even without intentional front-running, any pool whose admin fees were already at the old cap will silently violate the new cap until the owner manually remediates it pool-by-pool.

---

### Recommendation

Add clamping of per-pool admin fees inside `setFeeCaps`, mirroring the existing auto-clamp of the default protocol fee. For each registered pool, if `poolFeeConfig[pool].adminSpreadFeeE6 > newMaxAdminSpreadFeeE6`, clamp it and call `setPoolFees` on the pool with the corrected total. Alternatively, require that `setFeeCaps` can only lower caps to a value no lower than the highest currently-stored per-pool admin fee (analogous to the external report's recommended `require` guard), forcing the owner to first reduce individual pool admin fees before lowering the global cap.

---

### Proof of Concept

1. Factory caps: `maxAdminSpreadFeeE6 = 200_000` (20%).
2. Pool admin calls `setPoolAdminFees(pool, 200_000, 0)` — sets admin spread fee to 20%, within the current cap.
3. Owner signals intent to call `setFeeCaps(200_000, 30_000, ...)` to reduce the admin spread cap to 3%.
4. Pool admin (or MEV bot) front-runs with `setPoolAdminFees(pool, 200_000, 0)` again (no-op if already set, but confirms the 20% rate is locked in).
5. Owner's `setFeeCaps(200_000, 30_000, ...)` executes: `maxAdminSpreadFeeE6` becomes 30_000, but `poolFeeConfig[pool].adminSpreadFeeE6` remains 200_000.
6. Any caller invokes `collectPoolFees(pool)`: fees are split using the stored 200_000 admin rate — 20% of the spread surplus goes to the pool admin, far above the new 3% cap.
7. The pool admin cannot raise fees further (blocked by new cap), but the existing 200_000 rate persists on the pool until the owner calls `setPoolProtocolFee(pool, ...)`, which is a separate, per-pool manual step. [6](#0-5) [7](#0-6)

  function setPoolAdminFees(address pool, uint24 newAdminSpreadFeeE6, uint24 newAdminNotionalFeeE8)
    external
    override
    nonReentrant
    onlyPoolAdmin(pool)
  {
    if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    if (newAdminNotionalFeeE8 > maxAdminNotionalFeeE8) revert AdminFeeTooHigh();

    PoolFeeConfig memory c = poolFeeConfig[pool];
    IMetricOmmPoolCollectFees(pool)
      .collectFees(
        c.protocolSpreadFeeE6,
        c.adminSpreadFeeE6,
        c.protocolNotionalFeeE8,
        c.adminNotionalFeeE8,
        poolAdminFeeDestination[pool]
      );

    c.adminSpreadFeeE6 = newAdminSpreadFeeE6;
    c.adminNotionalFeeE8 = newAdminNotionalFeeE8;

    IMetricOmmPoolFactoryActions(pool)

    emit PoolAdminSpreadFeeUpdated(pool, newAdminSpreadFeeE6);
    emit PoolAdminNotionalFeeUpdated(pool, newAdminNotionalFeeE8);
