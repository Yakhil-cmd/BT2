### Title
Gateway `skip_stateful_validations` Admits Invoke Transactions with Invalid Signatures for Undeployed Accounts - (File: crates/apollo_gateway/src/stateful_transaction_validator.rs)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator skips the `__validate__` entry-point (signature verification) for any Invoke V3 transaction with `nonce == 1` sent from an address whose `account_nonce == 0` in state, provided that address has *any* transaction in the mempool or a recent block. The check does not verify that the incoming transaction originates from the legitimate account owner. An unprivileged attacker who observes a victim's `deploy_account` transaction in the mempool can submit a forged Invoke with `nonce=1` for the victim's address, carrying an arbitrary/invalid signature, and have it admitted to the mempool without signature verification.

### Finding Description

`skip_stateful_validations` is called inside `run_pre_validation_checks` after the nonce and resource-bound checks pass:

```
validate_state_preconditions → validate_by_mempool → skip_stateful_validations
``` [1](#0-0) 

`skip_stateful_validations` returns `true` (skip) when all three conditions hold:

1. The transaction is an `Invoke`.
2. `tx.nonce() == Nonce(Felt::ONE)`.
3. `account_nonce == Nonce(Felt::ZERO)`.
4. `mempool_client.account_tx_in_pool_or_recent_block(sender_address)` returns `true`. [2](#0-1) 

When `skip_validate == true`, `run_validate_entry_point` sets `execution_flags.validate = false`, so the blockifier's `StatefulValidator` never calls `validate_tx` (the `__validate__` entry point): [3](#0-2) 

The `account_tx_in_pool_or_recent_block` check only confirms that *some* transaction from the sender address is known to the mempool — it does not confirm that the *current* transaction was signed by the account owner: [4](#0-3) 

The code comment acknowledges the assumption but does not guard against a third-party submitting a transaction for the same address: [5](#0-4) 

### Impact Explanation

An attacker who monitors the mempool for a victim's `deploy_account` transaction can:

1. Observe `deploy_account` for victim address `A` (account_nonce = 0 in state, deploy_account in mempool → `account_tx_in_pool_or_recent_block(A) == true`).
2. Craft `Invoke(sender=A, nonce=1, signature=<garbage>, fee=victim_fee+1)`.
3. Submit to the gateway. The gateway passes nonce validation (`1 >= 0`), resource-bound checks, and then `skip_stateful_validations` returns `true` → `__validate__` is never called.
4. The attacker's forged Invoke is admitted to the mempool. Because its fee is higher, it replaces the victim's legitimate `nonce=1` Invoke (fee escalation).
5. At execution time: `deploy_account` runs (nonce 0→1), then the attacker's Invoke runs. `perform_pre_validation_stage` increments the nonce (1→2) before `__validate__` is called. `__validate__` then fails (invalid signature). The nonce increment is not rolled back; the victim's legitimate `nonce=1` Invoke is now stale and will be rejected by the blockifier with `InvalidNonce`. [6](#0-5) 

The broken invariant: the gateway must not skip `__validate__` for a transaction unless it can confirm the transaction originates from the legitimate account owner. The current check (`account_tx_in_pool_or_recent_block`) is a necessary but not sufficient condition.

This matches the allowed impact scope: **"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."**

### Likelihood Explanation

- Requires no privileged access; any party can submit transactions to the public gateway endpoint.
- The victim's address is deterministic from the `deploy_account` parameters (class hash, salt, constructor calldata), so it is computable by an observer before the block is committed.
- The attack window is the time between the victim's `deploy_account` entering the mempool and being included in a block — typically seconds to minutes.
- The attacker only needs to pay their own transaction fee (from their own account, not the victim's) to submit the forged Invoke; the fee escalation cost is bounded by the victim's original fee.

### Recommendation

Replace the coarse `account_tx_in_pool_or_recent_block` check with a check that specifically confirms a `deploy_account` transaction for the sender address is pending in the mempool. Expose a dedicated mempool query such as `has_pending_deploy_account(address) -> bool` and use that instead. This preserves the UX feature (deploy + invoke in one shot) while closing the window for third-party signature bypass.

Alternatively, add a cryptographic binding: require the `deploy_account` transaction hash to be embedded in the `invoke` transaction (e.g., in `account_deployment_data` or as a signed field), so the gateway can verify the two transactions were created by the same signer before skipping `__validate__`.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}

    fn handle_nonce(
        state: &mut dyn State,
        tx_info: &TransactionInfo,
        strict: bool,
    ) -> TransactionPreValidationResult<()> {
        if tx_info.is_v0() {
            return Ok(());
        }

        let address = tx_info.sender_address();
        let account_nonce = state.get_nonce_at(address)?;
        let incoming_tx_nonce = tx_info.nonce();
        let valid_nonce = if strict {
            account_nonce == incoming_tx_nonce
        } else {
            account_nonce <= incoming_tx_nonce
        };
        if valid_nonce {
            return Ok(state.increment_nonce(address)?);
        }
        Err(TransactionPreValidationError::InvalidNonce {
            address,
            account_nonce,
            incoming_tx_nonce,
        })
    }
