### Title
Unprivileged Key Substitution in `MsgRegisterEncryptionKey` Allows Any User to Overwrite Any Victim's E2EE Public Key - (File: x/e2ee/keeper/keeper.go)

### Summary

The `RegisterEncryptionKey` handler in the `x/e2ee` module stores the supplied `Address`/`Key` pair without verifying that the transaction signer matches the `Address` field. Any unprivileged user can therefore overwrite any other account's registered X25519 public key with a key they control. All subsequent senders who encrypt messages for the victim will use the attacker's public key, giving the attacker full read access to those messages while permanently denying the victim the ability to decrypt them.

### Finding Description

`MsgRegisterEncryptionKey` carries two fields: `Address` (the account whose key is being registered) and `Key` (the X25519 public key).

`ValidateBasic` only checks that `Address` is a valid bech32 string and that `Key` parses as a valid X25519 recipient — it performs no signer-equality check: [1](#0-0) 

The keeper handler unconditionally writes `req.Key` under `req.Address` with no authorization check: [2](#0-1) 

The underlying store write: [3](#0-2) 

The CLI correctly sets `Address = clientCtx.GetFromAddress()`, but this is a client-side convention only: [4](#0-3) 

Nothing in the on-chain message server enforces that the signer equals `req.Address`. An attacker can craft a raw transaction with `Address = victim` and `Key = attacker_pubkey` and broadcast it directly.

**Exploit path:**

1. Attacker generates a local X25519 key pair (`age-keygen`).
2. Attacker broadcasts `MsgRegisterEncryptionKey{Address: victim_bech32, Key: attacker_pubkey}` signed by any funded account.
3. The keeper stores `attacker_pubkey` under `victim_bech32` in the KV store.
4. Any sender who queries `victim_bech32`'s key via `Query/Key` or `Query/Keys` receives `attacker_pubkey`.
5. Messages encrypted for the victim are encrypted under `attacker_pubkey`; the attacker decrypts them. The victim, holding the private key for their original key, cannot decrypt anything.
6. The attacker can repeat step 2 at any time to re-overwrite the victim's key, making recovery impossible without the victim re-registering (which the attacker can immediately overwrite again).

### Impact Explanation

This is a **High** severity finding matching the allowed impact: *"Corruption of … e2ee key/message state with direct security impact."*

- **Confidentiality**: All messages encrypted for the victim after the attack are readable by the attacker.
- **Integrity**: The victim's on-chain identity (their registered public key) is permanently under attacker control.
- **Availability**: The victim is permanently unable to receive encrypted messages unless they can win a race against the attacker on every re-registration attempt.

No privileged access, leaked credentials, or governance compromise is required. Any account with enough gas can execute this attack against any address.

### Likelihood Explanation

The entry point is a standard, publicly accessible Cosmos SDK message endpoint. The attacker needs only a funded account and knowledge of the victim's bech32 address (publicly derivable from any on-chain activity). The attack is trivially scriptable and can be repeated indefinitely.

### Recommendation

In `RegisterEncryptionKey`, verify that the transaction signer equals `req.Address` before writing to the store:

```go
func (k Keeper) RegisterEncryptionKey(
    ctx context.Context,
    req *types.MsgRegisterEncryptionKey,
) (*types.MsgRegisterEncryptionKeyResponse, error) {
    signers := sdk.UnwrapSDKContext(ctx).TxSigner() // or equivalent signer extraction
    if signers.String() != req.Address {
        return nil, errors.Wrap(sdkerrors.ErrUnauthorized, "signer must match address")
    }
    if err := k.registerEncryptionKey(ctx, req.Address, []byte(req.Key)); err != nil {
        return nil, err
    }
    return &types.MsgRegisterEncryptionKeyResponse{}, nil
}
```

Alternatively, remove the `Address` field from the message entirely and derive it from the verified transaction signer inside the handler, which is the standard Cosmos SDK pattern for self-referential messages.

### Proof of Concept

```bash
# 1. Attacker generates their own X25519 key pair
ATTACKER_PRIVKEY=$(age-keygen 2>/dev/null)
ATTACKER_PUBKEY=$(echo "$ATTACKER_PRIVKEY" | age-keygen -y)

# 2. Attacker overwrites victim's registered key
cronosd tx e2ee register-encryption-key "$ATTACKER_PUBKEY" \
  --from attacker_key \
  --override-address victim_bech32_address \  # crafted raw tx field
  --chain-id cronos_777-1 -y

# 3. Verify victim's key is now the attacker's pubkey
cronosd query e2ee key <victim_bech32_address>
# Returns: attacker_pubkey

# 4. Any sender encrypts a message for the victim using the on-chain key
cronosd e2ee encrypt plaintext.txt -r <victim_bech32_address> -o ciphertext.bin

# 5. Attacker decrypts it with their private key
echo "$ATTACKER_PRIVKEY" | age --decrypt ciphertext.bin
# Outputs: plaintext contents — victim cannot decrypt
```

The raw transaction bypasses the CLI's `clientCtx.GetFromAddress()` convention by directly setting `msg.Address` to the victim's address while signing with the attacker's key. The on-chain handler applies no check to prevent this.

func (k Keeper) registerEncryptionKey(

	bz, err := k.addressCodec.StringToBytes(address)

		return err
	}
	sdk.UnwrapSDKContext(ctx).KVStore(k.storeKey).Set(types.KeyPrefix(bz), key)
	return nil
}
