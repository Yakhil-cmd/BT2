### Title
`CadenceOwnedAccount` Resource Destruction Does Not Clean Up EVM Account State, Permanently Locking FLOW Tokens - (File: `fvm/evm/stdlib/contract.cdc`)

### Summary
The `CadenceOwnedAccount` (COA) resource in `fvm/evm/stdlib/contract.cdc` has no `destroy` hook. When a COA resource is destroyed in Cadence, the corresponding EVM account (created via `InternalEVM.createCadenceOwnedAccount`) continues to exist with its full balance, code, and storage intact. Any FLOW tokens deposited into the COA's EVM-side account before destruction are permanently locked with no recovery path.

### Finding Description
`EVM.createCadenceOwnedAccount()` performs two coupled operations: it creates a Cadence resource and allocates a corresponding EVM account via `InternalEVM.createCadenceOwnedAccount(uuid: acc.uuid)`. [1](#0-0) 

The `CadenceOwnedAccount` resource definition spans lines 518–803 and contains `init`, `initAddress`, `deposit`, `withdraw`, `deploy`, `call`, and bridge functions — but **no `destroy` function**. [2](#0-1) 

When Cadence executes `destroy coa`, only the Cadence resource (which holds only the 20-byte `addressBytes` field) is deallocated. The EVM account at that address — including its FLOW token balance stored in the EVM state — is never touched. There is no `InternalEVM.deleteAccount` or equivalent call in the destruction path.

The `impl.go` binding confirms that `InternalEVM.createCadenceOwnedAccount` maps to `handler.DeployCOA(uuid)`, which allocates EVM state: [3](#0-2) 

There is no symmetric `handler.DeleteCOA` or `handler.DeleteAccount` call anywhere in the COA lifecycle.

The pattern is directly observable in existing test code, where a COA is created, funded, and then destroyed without withdrawing: [4](#0-3) [5](#0-4) 

In both cases the EVM account persists after `destroy`, with its balance orphaned.

### Impact Explanation
FLOW tokens deposited into a COA's EVM account via `coa.deposit(from: <-vault)` are held in the EVM state layer, not in the Cadence resource itself. Cadence's resource-safety guarantees (which prevent dropping resources with embedded assets) do not cover EVM-side balances. Once the COA resource is destroyed, the EVM account at `addressBytes` is permanently inaccessible: the COA resource is the sole authorization mechanism for `withdraw`, `call`, and bridge operations on that EVM address. The locked FLOW tokens are removed from circulation with no recovery path — a concrete cross-VM asset loss.

### Likelihood Explanation
Any unprivileged Cadence transaction sender who owns a COA resource can trigger this. The entry path requires only:
1. `let coa <- EVM.createCadenceOwnedAccount()`
2. `coa.deposit(from: <-vault)` — depositing any non-zero FLOW amount
3. `destroy coa` — without first calling `coa.withdraw(...)`

This is a realistic user-facing scenario (e.g., a contract that creates a temporary COA for a single operation and destroys it afterward, or a user who mistakenly destroys a funded COA). The `createCadenceOwnedAccount` function is `access(all)`, so it is reachable by any transaction.

### Recommendation
Add a `destroy` function to the `CadenceOwnedAccount` resource that either:
- **Panics if the EVM account balance is non-zero**, forcing callers to withdraw before destroying (analogous to how Cadence Vault resources enforce zero-balance before destruction), or
- **Automatically withdraws the remaining balance** back to a caller-supplied Cadence vault before deallocating the EVM account.

A minimal guard would be:

```cadence
// Inside CadenceOwnedAccount resource
destroy() {
    pre {
        self.balance().isZero():
            "EVM.CadenceOwnedAccount: cannot destroy a COA with a non-zero EVM balance"
    }
    // Optionally: InternalEVM.deleteAccount(address: self.addressBytes)
}
```

### Proof of Concept

```cadence
import EVM from <EVMContractAddress>
import FlowToken from <FlowTokenAddress>
import FungibleToken from <FungibleTokenAddress>

transaction {
    prepare(signer: auth(Storage) &Account) {
        // Step 1: Create a COA — allocates an EVM account
        let coa <- EVM.createCadenceOwnedAccount()

        // Step 2: Deposit FLOW tokens into the EVM account
        let vault <- signer.storage
            .borrow<auth(FungibleToken.Withdraw) &FlowToken.Vault>(from: /storage/flowTokenVault)!
            .withdraw(amount: 1.0) as! @FlowToken.Vault
        coa.deposit(from: <-vault)

        // Step 3: Destroy the COA resource WITHOUT withdrawing
        // The EVM account at coa.address() still exists with 1.0 FLOW locked.
        // No destroy hook calls InternalEVM.deleteAccount or similar.
        destroy coa

        // Result: 1.0 FLOW is permanently locked in the orphaned EVM account.
        // No Cadence resource holds addressBytes anymore; the balance is unrecoverable.
    }
}
```

The root cause is the absence of a `destroy` function in the `CadenceOwnedAccount` resource definition at `fvm/evm/stdlib/contract.cdc` lines 518–803, which mirrors the original report's pattern of a `burn()` that omits the underlying `_burn()` call. [6](#0-5)

func newInternalEVMTypeCreateCadenceOwnedAccountFunction(
	gauge common.MemoryGauge,
	handler types.ContractHandler,
) *interpreter.HostFunctionValue {
	return interpreter.NewStaticHostFunctionValue(
		gauge,
		stdlib.InternalEVMTypeCreateCadenceOwnedAccountFunctionType,
		func(invocation interpreter.Invocation) interpreter.Value {
			context := invocation.InvocationContext

			uuid, ok := invocation.Arguments[0].(interpreter.UInt64Value)
			if !ok {
				panic(errors.NewUnreachableError())
			}

			address := handler.DeployCOA(uint64(uuid))

			return EVMAddressToAddressBytesArrayValue(context, address)
		},
	)
}
