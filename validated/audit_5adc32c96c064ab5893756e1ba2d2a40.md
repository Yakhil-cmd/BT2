### Title
Unbounded Loop in `calculate_rewards` Can Permanently Freeze Delegator's Unclaimed Yield - (File: `src/pool/pool.cairo`)

---

### Summary

The `calculate_rewards` function in the `Pool` contract contains an unbounded `while` loop that iterates over every balance-change entry in a pool member's `pool_member_epoch_balance` trace since their last reward claim. A pool member who accumulates many balance changes across epochs without claiming rewards may find their `claim_rewards` call permanently failing due to gas exhaustion, permanently freezing their unclaimed yield. The code itself acknowledges this risk.

---

### Finding Description

In `src/pool/pool.cairo`, the `calculate_rewards` function contains a `while` loop at line 859 that iterates from `entry_to_claim_from` to the end of the pool member's balance trace:

```cairo
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
    if pool_member_checkpoint.epoch() >= until_epoch {
        break;
    }
    ...
    entry_to_claim_from += 1;
}
``` [1](#0-0) 

Each call to `add_to_delegation_pool` or `exit_delegation_pool_intent` invokes `set_member_balance` / `increase_member_balance`, which calls `trace.insert(key: self.get_epoch_plus_k(), ...)`. Because the key is `current_epoch + K`, each epoch in which a pool member modifies their balance appends a **new** checkpoint to `pool_member_epoch_balance`. [2](#0-1) 

The `entry_to_claim_from` index stored in `pool_member_info` is only advanced when `claim_rewards` **successfully completes**:

```cairo
pool_member_info.entry_to_claim_from = updated_entry_to_claim_from;
``` [3](#0-2) 

If a pool member accumulates N epochs of balance changes without ever claiming, the next `claim_rewards` call must iterate over all N entries in a single transaction. There is no partial-claim mechanism, no cap on iterations, and no protocol-enforced maximum gap between claims.

The `PoolMemberBalanceTrace.insert` function only deduplicates entries with the **same** key (same epoch). Entries from different epochs are always appended: [4](#0-3) 

---

### Impact Explanation

**Permanent freezing of unclaimed yield (High).**

If the number of accumulated balance-change entries since the last claim is large enough that iterating over them in a single transaction exceeds the Starknet gas/step limit, `claim_rewards` will revert every time it is called. Since `entry_to_claim_from` is only updated on success, the delegator is permanently locked out of their accumulated rewards with no recovery path. The funds themselves remain in the staking contract, but the delegator can never retrieve the yield portion.

---

### Likelihood Explanation

**Medium.** A pool member who actively participates — making at least one balance change per epoch (e.g., periodic top-ups or partial exit intents) — for a sustained period without claiming rewards will grow their trace linearly with the number of epochs. Starknet epochs are short (on the order of days), so a delegator who goes months without claiming while making regular balance changes can accumulate hundreds of entries. The code comment itself acknowledges the loop is unbounded and relies on the assumption it is "unlikely to exceed gas limits," which is not a protocol-enforced guarantee.

---

### Recommendation

1. **Partial reward claiming**: Allow `claim_rewards` to accept a `max_entries` parameter and process only up to that many trace entries per call, returning the updated `entry_to_claim_from` so the caller can resume in a subsequent transaction.
2. **Enforce a claim frequency**: Require pool members to claim rewards before their unclaimed trace exceeds a configurable maximum depth.
3. **Amortize on every balance change**: Compute and accumulate rewards incrementally at each `add_to_delegation_pool` / `exit_delegation_pool_intent` call, eliminating the need for a retrospective loop entirely.

---

### Proof of Concept

1. Delegator calls `enter_delegation_pool` in epoch `E`.
2. Each subsequent epoch, the delegator calls `add_to_delegation_pool` with a small amount. Each call appends a new checkpoint to `pool_member_epoch_balance` (since the epoch key advances each epoch).
3. After `N` epochs without calling `claim_rewards`, the trace has `N` entries and `entry_to_claim_from = 0`.
4. Delegator calls `claim_rewards`. The `calculate_rewards` loop must iterate over all `N` entries.
5. For sufficiently large `N`, the transaction exceeds the Starknet execution step limit and reverts.
6. Because `entry_to_claim_from` was never updated (the transaction reverted), every subsequent `claim_rewards` call also reverts. The delegator's yield is permanently frozen.
