### Title
Unprivileged Caller Can Block Consensus Reward Distribution via `update_rewards(disable_rewards=true)` - (File: src/staking/staking.cairo)

### Summary
The `update_rewards` function is publicly callable with no role-based access control. It accepts a user-controlled `disable_rewards` parameter. When called with `disable_rewards=true`, the function updates the global `last_reward_block` — blocking any other reward update in the same block — but skips actual reward distribution. Any unprivileged caller can exploit this ordering bug to permanently prevent all stakers from receiving consensus rewards.

### Finding Description
`update_rewards` (part of the public `IStakingRewardsManager` interface) has no access control beyond "not paused" and "caller not zero": [1](#0-0) 

The function first writes `last_reward_block` to the current block number, then checks `disable_rewards`: [2](#0-1) 

`last_reward_block` is a global variable. Once set to block N, no other call to `update_rewards` can succeed in block N: [3](#0-2) 

This is the ordering bug: the **lock** (`last_reward_block` update) is acquired **before** the **work** (reward distribution) is done. When `disable_rewards=true`, the lock is acquired but the work is permanently skipped for that block. Because `last_reward_block` is global, this blocks every staker from receiving rewards in that block — not just the one passed as `staker_address`.

The `general_prerequisites` check imposes no role restriction: [4](#0-3) 

The attacker only needs to supply any valid active staker address (publicly available from on-chain events) to pass the staker-existence and non-zero-balance assertions.

### Impact Explanation
An attacker calling `update_rewards(any_active_staker, disable_rewards=true)` once per block permanently freezes unclaimed consensus rewards for **all** stakers. This matches the allowed impact: **Permanent freezing of unclaimed yield** (High) or, at minimum, **Griefing with no profit motive but damage to users or protocol** (Medium). The staking contract's entire consensus-rewards distribution mechanism is rendered inoperable.

### Likelihood Explanation
- Callable by any unprivileged address (no role check).
- Only requires a valid active staker address, which is trivially obtained from `NewStaker` events.
- Gas cost per block on Starknet is low, making sustained attacks economically viable.
- No special knowledge, leaked keys, or privileged access required.

### Recommendation
1. Add access control to `update_rewards` so only the staker or their registered operational address may call it.
2. Alternatively, move the `last_reward_block.write(current_block_number)` to **after** the `disable_rewards` guard, so the lock is only acquired when rewards are actually distributed.
3. Or split the `disable_rewards` path into a separate, access-controlled function.

### Proof of Concept
1. Attacker identifies any active staker address from on-chain `NewStaker` events.
2. Each block, attacker calls `update_rewards(active_staker_address, disable_rewards=true)`.
3. `last_reward_block` is set to the current block number (line 1485).
4. The function returns early — no rewards are distributed (lines 1487–1489).
5. Any legitimate staker attempting `update_rewards` in the same block fails with `REWARDS_ALREADY_UPDATED` (lines 1454–1458).
6. Repeated every block, this permanently prevents consensus reward distribution for all stakers.

This is the direct analog to the external report's ordering bug: in the original finding, NFTs are transferred to the buyer before payment is received, enabling flash-loan-like exploitation; here, the reward-distribution lock (`last_reward_block`) is acquired before rewards are actually distributed, enabling an attacker to seize the lock without performing the work — permanently denying yield to all protocol participants.
