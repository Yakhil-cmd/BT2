### Title
Weight Aggregation Bypass via Duplicate Public Key Registration — (`fvm/environment/account_key_updater.go`, `fvm/transactionVerifier.go`)

---

### Summary

An account owner can register the same public key multiple times under different key indices, each with a partial weight. Because `TransactionVerifier` aggregates weights by `(address, keyIndex)` without checking whether multiple indices share the same underlying cryptographic key, a single private key can satisfy the full weight threshold by signing once per registered index.

---

### Finding Description

**Step 1 — Duplicate key registration is permitted.**

`addAccountKey` in `fvm/environment/account_key_updater.go` performs no duplicate-key check before calling `AppendAccountPublicKey`: [1](#0-0) 

`AppendAccountPublicKey` → `appendPublicKey` → `appendKeyMetadataToAccountStatusRegister` → `AppendAccountPublicKeyMetadata` → `FindDuplicateKey`. When a duplicate is detected, the system performs *storage* deduplication (`saveKey = false`) but still records the new key index with its own independent weight and revoked status: [2](#0-1) 

The duplicate key is fully registered at a new index with its own weight. This is confirmed by the existing test suite which explicitly exercises "account with 2 duplicate keys": [3](#0-2) 

**Step 2 — `GetRuntimeAccountPublicKey` returns the per-index weight with the shared public key bytes.**

For key index 1 (duplicate of index 0), the function reads weight from the key-index-specific metadata and the public key bytes from the shared stored key: [4](#0-3) 

So `GetRuntimeAccountPublicKey(A, 0)` → `{PublicKey: P, Weight: 500}` and `GetRuntimeAccountPublicKey(A, 1)` → `{PublicKey: P, Weight: 500}`.

**Step 3 — `newSignatureEntries` deduplicates only by `(address, keyIndex)`, not by underlying public key.**

Two signatures `(A, keyIndex=0, sig)` and `(A, keyIndex=1, sig)` are distinct pairs and both pass through: [5](#0-4) 

**Step 4 — `getAccountKeysAndAggregateWeights` blindly sums weights per address.**

For each signature entry it fetches the key at the given index and adds its weight to the address total, with no check that two entries share the same underlying cryptographic key: [6](#0-5) 

With two signatures referencing key 0 (weight 500) and key 1 (weight 500), the aggregated weight for address A becomes 1000, meeting the threshold.

**Step 5 — `verifySignatures` accepts both signatures.**

Both entries carry the same public key P (fetched from the shared stored key). A single private key produces a valid signature for each, so both pass cryptographic verification: [7](#0-6) 

---

### Impact Explanation

An attacker who controls an account can achieve full signing authority using a single private key registered twice at half-weight. This bypasses the intended multi-key authorization model: the threshold is designed to require cooperation of independent key holders, but a single key holder can satisfy it alone by splitting their key across two indices. Any transaction — including asset transfers, contract deployments, or capability grants — can be authorized this way.

---

### Likelihood Explanation

The attack requires only standard Cadence transaction submission (adding a key twice, then submitting a transaction with two envelope signatures). No privileged access, leaked keys, or node compromise is needed. The `FindDuplicateKey` detection is explicitly documented as intentionally less than 100%, but even when detection succeeds the duplicate key is still registered — the deduplication is purely a storage optimization, not a security rejection. The attack is locally reproducible on the emulator.

---

### Recommendation

In `getAccountKeysAndAggregateWeights`, after fetching each account key, check whether the underlying public key bytes have already contributed weight for this address in the current transaction. If the same encoded public key appears at multiple indices, count its weight only once:

```go
type seenKey struct {
    address  flow.Address
    encodedKey string // or []byte fingerprint
}
seenKeys := make(map[seenKey]struct{})

for _, signature := range signatures {
    accountKey, err := accounts.GetRuntimeAccountPublicKey(...)
    ...
    k := seenKey{signature.Address, string(accountKey.PublicKey.Encode())}
    if _, already := seenKeys[k]; already {
        return signature.newError(fmt.Errorf("duplicate public key used across multiple key indices"))
    }
    seenKeys[k] = struct{}{}
    signature.aggregateWeights[signature.Address] += accountKey.Weight
}
```

Alternatively, reject duplicate public key registration outright in `addAccountKey` rather than treating it as a storage-only concern.

### Proof of Concept

```go
// Emulator integration test sketch
privKey, _ := crypto.GeneratePrivateKey(crypto.ECDSAP256, seed)
pubKey := privKey.PublicKey()

// Register same public key twice with weight 500 each
acctKey0 := sdk.NewAccountKey().SetPublicKey(pubKey).SetWeight(500).SetHashAlgo(hash.SHA3_256)
acctKey1 := sdk.NewAccountKey().SetPublicKey(pubKey).SetWeight(500).SetHashAlgo(hash.SHA3_256)
addr, _ := adapter.CreateAccount(ctx, []*sdk.AccountKey{acctKey0, acctKey1}, nil)
// addr now has key index 0 and key index 1, both = pubKey, each weight 500

tx := sdk.NewTransaction().
    SetScript([]byte(`transaction { prepare(s: &Account) {} }`)).
    SetProposalKey(addr, 0, 0).
    SetPayer(addr).
    AddAuthorizer(addr)

// Sign envelope with key index 0
tx.SignEnvelope(addr, 0, crypto.NewInMemorySigner(privKey, hash.SHA3_256))
// Sign envelope again with key index 1 (same private key, different index)
tx.SignEnvelope(addr, 1, crypto.NewInMemorySigner(privKey, hash.SHA3_256))

adapter.SendTransaction(ctx, *tx)
result, _ := b.ExecuteNextTransaction()
// result.Error == nil  →  transaction accepted with aggregated weight 1000
assert.Nil(t, result.Error)
```
