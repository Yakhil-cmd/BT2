### Title
Unvalidated `sender_id` in `GetExecutionOutcome` Produces Silently Corrupted `outcome_root_proof` for Wrong Shard — (`chain/client/src/view_client_actor.rs`)

---

### Summary

An unprivileged RPC client can supply an arbitrary `sender_id` in a `RpcLightClientExecutionProofRequest`. The `GetExecutionOutcome` handler uses that caller-controlled value — without ever checking it against the actual transaction's sender — to select which shard's Merkle branch to return as `outcome_root_proof`. The result is a structurally valid but semantically wrong Merkle path that silently fails the standard two-step light-client proof verification.

---

### Finding Description

The handler in `chain/client/src/view_client_actor.rs` extracts `account_id` directly from the request's `sender_id` field: [1](#0-0) 

It then looks up the actual execution outcome using only the transaction hash — `account_id` plays no role here: [2](#0-1) 

Immediately after, `target_shard_id` and `target_shard_index` are derived exclusively from the caller-supplied `account_id`: [3](#0-2) 

`get_next_block_hash_with_new_chunk` is then called with the wrong `target_shard_id`, advancing to the first block that has a new chunk for the **wrong** shard: [4](#0-3) 

Finally, `outcome_root_proof` is built as the Merkle branch at `target_shard_index` — the wrong shard's position — in the block's outcome-root tree: [5](#0-4) 

There is no guard anywhere in this path that compares `account_id` to the actual sender stored in the outcome record. The `get_execution_outcome` implementation in `chain/chain/src/chain.rs` is keyed solely on the transaction hash: [6](#0-5) 

---

### Impact Explanation

The two-step light-client proof verification (documented in `docs/ChainSpec/LightClient.md`) is:

```
shard_outcome_root = compute_root(sha256(borsh(execution_outcome)), outcome_proof.proof)
block_outcome_root = compute_root(sha256(borsh(shard_outcome_root)), outcome_root_proof)
```

`outcome_proof.proof` is fetched from the real outcome record and is correct for shard 0. But `outcome_root_proof` is the Merkle branch for shard 1's position. The two are incoherent: `block_outcome_root` will not equal `block_header_lite.inner_lite.outcome_root`. Any bridge contract or light-client implementation performing this check will reject the proof, causing a valid, finalized transaction to be treated as unproven. [7](#0-6) 

---

### Likelihood Explanation

The attack requires only a public RPC call with a crafted `sender_id`. No keys, stake, or privileged access are needed. In a two-shard deployment any account whose name hashes to a different shard than the real sender suffices. The existing test harness (`test-loop-tests/src/tests/sharded_rpc.rs`) already exercises exactly this two-shard topology, confirming the scenario is reachable in production configurations. [8](#0-7) 

---

### Recommendation

After `get_execution_outcome` succeeds, derive `target_shard_id` from the **outcome record itself** (e.g., from `outcome_proof.outcome.executor_id` or by looking up the transaction's actual sender from the chain store), not from the caller-supplied `account_id`. If the caller-supplied `sender_id` does not map to the same shard as the outcome's actual shard, return a descriptive error rather than silently computing a proof for the wrong shard.

---

### Proof of Concept

1. Start a two-shard test-loop network (alice → shard 0, zoe → shard 1).
2. Submit `send_money` from `alice` to `zoe`; wait for finality.
3. Call `light_client_proof` with `id: Transaction { transaction_hash: tx_hash, sender_id: "zoe" }`.
4. Receive a response where `outcome_proof.proof` is correct for shard 0 but `outcome_root_proof` is the Merkle branch for shard 1's index.
5. Run the two-step verification:
   ```
   shard_root = compute_root(sha256(borsh(outcome)), outcome_proof.proof)   // correct shard-0 root
   block_root = compute_root(sha256(borsh(shard_root)), outcome_root_proof) // uses shard-1 branch → wrong
   assert block_root == block_header_lite.inner_lite.outcome_root            // FAILS
   ```
6. `verify_path` returns `false`; a bridge contract would reject the proof. [9](#0-8)
