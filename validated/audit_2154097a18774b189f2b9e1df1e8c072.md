### Title
Unprivileged Caller Can Permanently Freeze All Staker Rewards by Front-Running `update_rewards` with `disable_rewards: true` — (`src/staking/staking.cairo`)

### Summary

The `update_rewards` function in the Staking contract has no access control and accepts a caller-controlled `disable_rewards` boolean. Any unprivileged address can call `update_rewards(valid_staker_address, disable_rewards: true)`, which advances the global `last_reward_block` to the current block without distributing any rewards. Because the function enforces a strict one-call-per-block invariant on `last_reward_block`, the legitimate consensus-mechanism call in the same block will revert with `REWARDS_ALREADY_UPDATED`. Repeating this every block permanently freezes all stakers' unclaimed yield at zero cost beyond gas.

### Finding Description

**Root cause — no access control on `update_rewards`:** [1](#0-0) 

The function opens with only `general_prerequisites()` (a pause-state check) and no role assertion. Any externally-owned address may call it.

**The per-block gate is a global, single-slot lock:** [2](#0-1) 

`last_reward_block` is a single contract-wide storage variable. Once it equals the current block number, every subsequent call in the same block reverts.

**`disable_rewards: true` advances the lock without distributing rewards:** [3](#0-2) 

`last_reward_block` is written **before** the early-return guard. An attacker who passes `disable_rewards: true` consumes the per-block slot while distributing nothing.

**Attack path:**

1. Attacker picks any valid, active staker address (publicly readable from the `stakers` Vec or past `NewStaker` events).
2. At the start of every block, attacker submits `update_rewards(valid_staker, disable_rewards: true)`.
3. `last_reward_block` is set to the current block; no rewards are minted or credited.
4. The legitimate consensus-mechanism call (or any honest caller) in the same block hits the `REWARDS_ALREADY_UPDATED` assertion and reverts.
5. Stakers accumulate zero `unclaimed_rewards_own` indefinitely.

The attacker requires no privileged key, no stake, and no special role — only the ability to submit transactions.

### Impact Explanation

Every block in which the attacker front-runs the legitimate `update_rewards` call produces zero rewards for **all** stakers and pool members. Sustained over time this constitutes **permanent freezing of unclaimed yield** for the entire protocol, matching the High impact tier in the allowed scope.

### Likelihood Explanation

The attack is trivially reachable: the function is public, the required `staker_address` argument is publicly observable, and the gas cost on Starknet L2 is negligible. No privileged access, leaked key, or external dependency is required. A single motivated actor can sustain the attack indefinitely.

### Recommendation

1. **Add access control**: Restrict `update_rewards` to the consensus mechanism address (analogous to how `update_rewards_from_attestation_contract` is restricted to the attestation contract via `assert_caller_is_attestation_contract`).
2. **Separate the lock update from the early-return path**: Only write `last_reward_block` when rewards are actually distributed, so a `disable_rewards: true` call does not consume the per-block slot.
3. **Remove the `disable_rewards` parameter from the public interface**: If disabling is needed, derive it internally from on-chain state (e.g., whether the staker attested) rather than accepting it as caller-supplied input.

### Proof of Concept

```
// Precondition: consensus rewards are active (is_pre_consensus() == false).
// Attacker has no stake or special role.

loop every block:
    staking_contract.update_rewards(
        staker_address = any_active_staker,   // readable from stakers Vec
        disable_rewards = true
    )
    // Effect: last_reward_block = current_block, no rewards distributed.
    // Legitimate consensus call in same block → REWARDS_ALREADY_UPDATED panic.
end loop

// Result: all stakers' unclaimed_rewards_own remains 0 indefinitely.
```
