### Title
Stale-pending fallback uses `DeprecatedPendingBlock`, zeroing `l1_data_gas_price` and `l2_gas_price` in fee estimation — (`crates/apollo_rpc/src/v0_8/api/api_impl.rs`)

### Summary
When the in-memory pending block is stale (its `parent_block_hash` does not match the latest committed block hash), `read_pending_data` synthesizes a fallback `DeprecatedPendingBlock`. This type structurally omits `l1_data_gas_price` and `l2_gas_price`; its accessor methods return `GasPricePerToken::default()` (zero / `NonzeroGasPrice::MIN` after clamping). Any call to `estimate_fee` or `simulate_transactions` with `block_id=Pending` during this window receives a fee estimate computed with effectively-zero data-gas and L2-gas prices, regardless of the real prices in the latest committed header.

### Finding Description
`read_pending_data` (api_impl.rs:1572-1594) checks whether `pending_data.block.parent_block_hash() == latest_header.block_hash`. On mismatch it returns a synthetic `PendingData` wrapping a `DeprecatedPendingBlock`:

```rust
// api_impl.rs:1573-1593
Ok(PendingData {
    block: PendingBlockOrDeprecated::Deprecated(DeprecatedPendingBlock {
        parent_block_hash: latest_header.block_hash,
        eth_l1_gas_price: latest_header...l1_gas_price.price_in_wei,
        strk_l1_gas_price: latest_header...l1_gas_price.price_in_fri,
        timestamp: ...,
        sequencer_address: ...,
        starknet_version: ...,
        ..Default::default()   // ← l1_data_gas_price / l2_gas_price absent
    }),
    ...
})
```

`DeprecatedPendingBlock` has no `l1_data_gas_price` or `l2_gas_price` fields. The `PendingBlockOrDeprecated::l1_data_gas_price()` and `l2_gas_price()` methods hard-return `GasPricePerToken::default()` for the `Deprecated` variant (pending_data.rs:155-168). `client_pending_data_to_execution_pending_data` (pending.rs:18-21) passes these zero values directly into `ExecutionPendingData`, which is then used by the blockifier to compute fee estimates.

The latest committed header **does** carry real non-zero values for both prices, but the fallback path never reads them.

### Impact Explanation
Any user calling `starknet_estimateFee` or `starknet_simulateTransactions` with `block_id=Pending` during the stale window receives an `overall_fee` that omits the data-gas and L2-gas cost components. For a V3 `INVOKE` transaction with non-trivial calldata or L2 gas consumption, the returned estimate can be materially lower than the fee that will actually be charged when the transaction is sequenced. A user who sets `max_fee` / resource bounds based on this estimate may have their transaction rejected by the sequencer's `check_fee_bounds` / `verify_can_pay_committed_bounds` checks.

### Likelihood Explanation
The stale window is a normal operational condition: it occurs every time a new block is committed until the pending-block watcher pushes an updated pending block. On a busy network this window is short but non-zero and repeatable. No special privileges are required — any user can query `estimate_fee` with `block_id=Pending`.

### Recommendation
Replace the `DeprecatedPendingBlock` fallback with a `PendingBlock` (the `Current` variant) so that all price fields from the latest committed header can be copied:

```rust
block: PendingBlockOrDeprecated::Current(PendingBlock {
    parent_block_hash: latest_header.block_hash,
    l1_gas_price: latest_header...l1_gas_price,
    l1_data_gas_price: latest_header...l1_data_gas_price,
    l2_gas_price: latest_header...l2_gas_price,
    l1_da_mode: latest_header...l1_da_mode,
    timestamp: ...,
    sequencer_address: ...,
    starknet_version: ...,
    ..Default::default()
}),
```

### Proof of Concept
1. Commit a block with `l1_data_gas_price = GasPricePerToken { price_in_wei: 1000, price_in_fri: 2000 }` and `l2_gas_price = GasPricePerToken { price_in_wei: 500, price_in_fri: 1000 }`.
2. Set the shared `pending_data` to a block whose `parent_block_hash` does not match the committed block hash (simulating a stale pending block).
3. Call `exec_estimate_fee` with a V3 `INVOKE` transaction and `block_id=Pending`.
4. Observe that the returned `FeeEstimation.overall_fee` equals the value computed with `l1_data_gas_price = NonzeroGasPrice::MIN` and `l2_gas_price = NonzeroGasPrice::MIN`, not with the real prices from step 1.
5. Compute the correct fee using the real prices; assert the two values differ. [5](#0-4) [6](#0-5) [7](#0-6)
