### Title
`DelegateAction` Signature Digest Omits Chain Identity, Enabling Cross-Network Replay — (File: `core/primitives/src/action/delegate.rs`)

### Summary

`DelegateAction` (NEP-366 meta transactions) is signed by the user via `get_nep461_hash()`, which hashes only a fixed NEP discriminant and the borsh-serialized action body. Neither the discriminant nor any field in `DelegateAction` encodes a chain-specific identifier. A signed `DelegateAction` produced for mainnet is cryptographically valid on testnet (or any NEAR fork) if the same account and key exist there and the nonce is still acceptable.

### Finding Description

`DelegateAction::get_nep461_hash()` constructs the signed digest as:

```
SHA-256( MessageDiscriminant(NEP-366) || borsh(DelegateAction) )
``` [1](#0-0) 

`MessageDiscriminant` for `DelegateAction` is the constant `MIN_ON_CHAIN_DISCRIMINANT + 366 = 2^30 + 366`, identical on every NEAR network. [2](#0-1) [3](#0-2) 

The `DelegateAction` struct itself contains only `sender_id`, `receiver_id`, `actions`, `nonce`, `max_block_height`, and `public_key`: [4](#0-3) 

None of these fields are chain-specific. `max_block_height` is a bare block-height integer with no chain binding; `nonce` prevents same-chain replay but not cross-chain replay.

Contrast this with a regular `SignedTransaction`, whose hash is computed over the full `Transaction` body, which includes `block_hash` — a hash of a specific block that is unique to one chain and expires within `transaction_validity_period` blocks: [5](#0-4) [6](#0-5) 

`DelegateAction` has no equivalent anchor.

### Impact Explanation

An attacker (a malicious relayer, or anyone who intercepts a signed `DelegateAction`) can submit the same `SignedDelegateAction` on a different NEAR network (e.g., mainnet → testnet or a fork). If the sender's account exists on the target chain with the same public key and a nonce below the delegate action's nonce, the runtime will:

1. Verify the signature — it passes because the digest is chain-agnostic.
2. Validate the nonce — it passes if the target-chain access-key nonce is lower.
3. Execute the inner `actions` — token transfers, key additions/deletions, contract calls — on the target chain without the user's consent.

The corrupted values are the target-chain account balance (drained by a `Transfer` action), access-key set (modified by `AddKey`/`DeleteKey`), or any other state mutated by the inner actions. [7](#0-6) 

### Likelihood Explanation

NEAR account IDs are human-readable strings. Many users register the same account ID on both mainnet and testnet and reuse the same key pair. A relayer that receives a mainnet-signed `DelegateAction` (e.g., via a meta-transaction API) can trivially submit it on testnet. The `max_block_height` bound does not help because block heights on testnet are independent of mainnet and are typically lower, so a mainnet action with a large `max_block_height` remains valid on testnet for an extended period.

### Recommendation

Include a chain-specific binding in the signed digest. The canonical approach is to add the genesis block hash (which is unique per network) or the `chain_id` string to the `DelegateAction` struct or to the `SignableMessage` wrapper before hashing. This mirrors how `SignedTransaction` uses `block_hash` to bind a transaction to a specific chain and validity window. [8](#0-7) 

### Proof of Concept

1. Alice has account `alice.near` on mainnet and testnet, with the same ED25519 key pair registered on both. Testnet access-key nonce is 5; mainnet nonce is 50.
2. Alice signs a `DelegateAction` on mainnet: `nonce=51`, `max_block_height=10_000_000`, `actions=[Transfer{deposit: 100 NEAR}]`, `receiver_id=bob.near`.
3. The relayer (or an interceptor) takes the `SignedDelegateAction` and wraps it in a testnet transaction sent to `alice.near` on testnet.
4. The testnet runtime calls `SignedDelegateAction::verify()`:
   - Recomputes `get_nep461_hash()` — identical to the mainnet hash because no chain field exists.
   - Signature check passes.
5. `validate_delegate_action_key` checks nonce 51 > testnet ak_nonce 5 — passes.
6. `max_block_height` 10,000,000 > current testnet height — passes.
7. The transfer of 100 NEAR executes on testnet without Alice's consent. [9](#0-8)

impl SignedDelegateAction {
    pub fn verify(&self) -> bool {
        let delegate_action = &self.delegate_action;
        let hash = delegate_action.get_nep461_hash();
        let public_key = &delegate_action.public_key;

        self.signature.verify(hash.as_ref(), public_key)
    }

    pub fn sign(singer: &Signer, delegate_action: DelegateAction) -> Self {
        let signature = singer.sign(delegate_action.get_nep461_hash().as_bytes());
        Self { delegate_action, signature }
    }

    pub fn get_nep461_hash(&self) -> CryptoHash {
        let signable = SignableMessage::new(&self, SignableMessageType::DelegateAction);
        let bytes = borsh::to_vec(&signable).expect("Failed to deserialize");
        hash(&bytes)
