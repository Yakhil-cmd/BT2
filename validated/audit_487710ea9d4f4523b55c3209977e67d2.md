### Title
TOCTOU Race in Pool Join: `prepareSubmitData` Re-fetches `/pool_info` at Submit Time, Allowing Attacker-Controlled Pool Server to Substitute `targetPuzzleHash` After User Approval — (`packages/gui/src/components/plotNFT/select/PlotNFTSelectPool.tsx`)

---

### Summary

The pool-join flow makes **two independent HTTP fetches** to the pool server's `/pool_info` endpoint: one at display time (via `usePoolInfo`) and a second at submit time (inside `prepareSubmitData`). A malicious pool operator can serve a legitimate `targetPuzzleHash` for the first fetch so the user sees and approves it, then switch to an attacker-controlled puzzle hash for the second fetch. The on-chain transaction commits the second (malicious) hash, diverting all future farming rewards to the attacker.

---

### Finding Description

**Display path** — `PlotNFTSelectBase.tsx` calls `usePoolInfo(poolUrl)`: [1](#0-0) 

`usePoolInfo` calls `getPoolInfo` → `window.appAPI.fetchPoolInfo(poolUrl)` and the result is rendered via `<PoolInfo>`, which explicitly shows `targetPuzzleHash` to the user: [2](#0-1) 

**Submit path** — when the user clicks submit, `handleSubmit` calls `prepareSubmitData(data)`: [3](#0-2) 

`prepareSubmitData` issues a **fresh, independent** call to `getPoolInfo`: [4](#0-3) 

The `targetPuzzleHash` from this second fetch is placed directly into `initialTargetState` and forwarded to `pwJoinPool` (change-pool flow) or `createNewPoolWallet` (new NFT flow) with no comparison against the value already shown to the user: [5](#0-4) 

There is **no guard** that cross-checks the displayed `targetPuzzleHash` against the submitted one.

---

### Impact Explanation

`targetPuzzleHash` is the puzzle hash to which the pool sends farming rewards. If an attacker substitutes their own puzzle hash at submit time, every block reward earned by the user's plots is paid to the attacker for the entire duration the NFT remains joined to that pool (up to `relativeLockHeight` blocks to escape). This is a direct, ongoing loss of XCH farming rewards — a payout change affecting pooled farming rewards, fitting the Critical impact category.

---

### Likelihood Explanation

Any pool operator (or anyone who can MITM the HTTPS connection to the pool server) can execute this. The user enters the pool URL themselves as part of the normal join-pool workflow; no additional social engineering is required beyond advertising a pool. The timing window between the display fetch and the submit click is entirely under the attacker's control server-side — the server simply returns different JSON for the second request. No local access, no key leakage, and no cryptographic weakness is needed.

---

### Recommendation

Do not re-fetch pool info at submit time. Instead, pass the already-resolved `poolInfo` value from `usePoolInfo` (held in `PlotNFTSelectBase`) into `prepareSubmitData` (or store it in the form state), and use that cached value to populate `targetPuzzleHash` and `relativeLockHeight`. If a re-fetch is considered necessary for freshness, the result must be compared against the previously displayed values and the user must be shown a confirmation dialog if any field differs.

---

### Proof of Concept

```
1. Stand up a pool server at https://evil-pool.example.
2. On the first GET /pool_info request, return:
     { "target_puzzle_hash": "0xLEGIT...", "relative_lock_height": 100, ... }
3. On every subsequent GET /pool_info request, return:
     { "target_puzzle_hash": "0xATTACKER...", "relative_lock_height": 100, ... }
4. User opens the GUI, enters https://evil-pool.example as the pool URL.
5. GUI displays "Target Puzzle Hash: 0xLEGIT..." — user is satisfied and clicks submit.
6. prepareSubmitData fires a second fetch; server returns 0xATTACKER...
7. pwJoinPool / createNewPoolWallet is called with targetPuzzlehash = "0xATTACKER..."
8. On-chain state commits 0xATTACKER... as the reward destination.
9. All farming rewards flow to the attacker's wallet.

Unit-test equivalent:
  - Mock window.appAPI.fetchPoolInfo to return LEGIT hash on call #1, ATTACKER hash on call #2.
  - Render PlotNFTSelectPool, wait for pool info display, then trigger form submit.
  - Assert pwJoinPool was called with targetPuzzlehash === ATTACKER hash despite UI showing LEGIT hash.
```
