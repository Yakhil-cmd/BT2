### Title
Missing STRK Token Address Validation in Pool `change_reward_address` Permanently Freezes Delegator Rewards — (File: src/pool/pool.cairo)

### Summary
The `change_reward_address` function in the delegation pool contract only validates that the new reward address is not the pool's own token. For BTC pools, this omits a check against `STRK_TOKEN_ADDRESS`. Because `claim_rewards` always pays out in STRK (hardcoded), a BTC-pool member who sets `reward_address = STRK_TOKEN_ADDRESS` causes all future reward transfers to be sent to the STRK token contract itself, permanently locking those rewards with no recovery path.

### Finding Description
In `pool.cairo`, `change_reward_address` performs only a single token-address guard:

```cairo
assert!(
    self.token_dispatcher.contract_address.read() != reward_address,
    "{}",
    GenericError::REWARD_ADDRESS_IS_TOKEN,
);
``` [1](#0-0) 

For a BTC delegation pool, `token_dispatcher.contract_address` is the BTC token address. The check therefore passes when `reward_address == STRK_TOKEN_ADDRESS`.

The same incomplete guard appears in `enter_delegation_pool`:

```cairo
assert!(token_address != reward_address, "{}", GenericError::REWARD_ADDRESS_IS_TOKEN);
``` [2](#0-1) 

By contrast, the staking contract's own `change_reward_address` uses `does_token_exist`, which covers **all** registered tokens (STRK and BTC):

```cairo
assert!(
    !self.does_token_exist(token_address: reward_address),
    "{}",
    GenericError::REWARD_ADDRESS_IS_TOKEN,
);
``` [3](#0-2) 

`claim_rewards` in the pool contract hardcodes `STRK_TOKEN_ADDRESS` as the reward token regardless of the pool's own token:

```cairo
let reward_token = IERC20Dispatcher { contract_address: STRK_TOKEN_ADDRESS };
reward_token.checked_transfer(recipient: reward_address, amount: rewards.into());
``` [4](#0-3) 

When `reward_address == STRK_TOKEN_ADDRESS`, the `checked_transfer` call succeeds (the STRK ERC-20 contract can hold a balance), but those tokens are irrecoverably locked inside the token contract. There is no `recoverFunds`-equivalent anywhere in the pool or staking contracts.

### Impact Explanation
Any STRK rewards accrued by the affected BTC-pool member are permanently transferred into the STRK token contract and become unclaimable. This constitutes **permanent freezing of unclaimed yield**, a High-severity impact under the allowed scope.

### Likelihood Explanation
The entry path is reachable by any unprivileged BTC-pool member via `enter_delegation_pool` (at join time) or `change_reward_address` (at any time). The scenario requires the pool member to supply `STRK_TOKEN_ADDRESS` as their reward address — possible through a UI bug, copy-paste error, or deliberate self-griefing. BTC pools are a live feature of the multi-token staking system, so the affected code path is active. Likelihood is **Low** but non-zero.

### Recommendation
Mirror the staking contract's validation: replace the single-token check in both `enter_delegation_pool` and `change_reward_address` in `pool.cairo` with a call that rejects any registered token address (STRK and BTC) as a reward address. Alternatively, explicitly add:

```cairo
assert!(
    STRK_TOKEN_ADDRESS != reward_address,
    "{}",
    GenericError::REWARD_ADDRESS_IS_TOKEN,
);
```

alongside the existing pool-token check.

### Proof of Concept
1. A staker opens a BTC delegation pool via `set_open_for_delegation(btc_token_address)`.
2. A delegator calls `enter_delegation_pool(reward_address: STRK_TOKEN_ADDRESS, amount: X)` on the BTC pool. The guard `token_address != reward_address` passes because `btc_token_address != STRK_TOKEN_ADDRESS`.
3. After epochs pass, the staking contract distributes STRK rewards to the pool via `send_rewards_to_delegation_pool`.
4. The delegator (or anyone) calls `pool.claim_rewards(pool_member)`. The function computes `rewards > 0` and executes `STRK_token.transfer(recipient: STRK_TOKEN_ADDRESS, amount: rewards)`. The transfer succeeds; the STRK token contract's own balance increases.
5. The rewards are now permanently locked inside the STRK token contract. No function in the pool or staking contract can retrieve them. [5](#0-4) [6](#0-5)
