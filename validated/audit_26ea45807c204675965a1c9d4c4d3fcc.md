### Title
Missing Zero-Value Validation for `chain_id` in `new()` Initialization Permanently Breaks EVM Transaction Submission - (`engine/src/contract_methods/admin.rs`)

### Summary
The `new()` function in Aurora Engine accepts and stores a `chain_id` of all-zero bytes without any validation. Because no `set_chain_id` admin function exists, a deployer who accidentally passes `chain_id = [0u8; 32]` permanently misconfigures the engine so that every EIP-155 transaction is rejected, freezing EVM-based access to funds held in EVM smart contracts.

### Finding Description
`new()` in `engine/src/contract_methods/admin.rs` deserializes `NewCallArgs` and converts it directly into `EngineState` with no bounds or zero-value check on `chain_id`: [1](#0-0) 

The resulting `EngineState.chain_id` field is a `[u8; 32]` big-endian representation of a `U256`: [2](#0-1) 

All `From<NewCallArgs*>` conversions copy `chain_id` verbatim with no validation: [3](#0-2) 

During transaction submission, the engine validates the chain ID as follows: [4](#0-3) 

If `state.chain_id = [0u8; 32]`, then `U256::from_big_endian(&state.chain_id) = 0`. Every EIP-155 transaction carries a non-zero chain ID in its signature, so `U256::from(chain_id) != 0` is always true, and every such transaction is rejected with `ERR_INVALID_CHAIN_ID`. No `set_chain_id` function exists in the admin interface; the only remediation path is a full contract upgrade.

The parameter structs for all supported `new()` argument versions accept `chain_id` without any constraint: [5](#0-4) 

### Impact Explanation
All modern Ethereum tooling (MetaMask, ethers.js, Hardhat, Foundry) signs transactions with EIP-155 and a non-zero chain ID. With `chain_id = 0` stored in state, every such transaction is permanently rejected. Users who have ETH or ERC-20 tokens locked inside EVM smart contracts deployed on Aurora cannot execute any EVM transaction to withdraw or interact with those contracts. This is a **High: Temporary freezing of funds** (effectively permanent without an upgrade) for all EVM-contract-held assets. Only legacy (non-EIP-155) transactions — which modern tooling does not produce — would still be accepted.

### Likelihood Explanation
Low but non-zero. The deployer must accidentally pass `chain_id = [0u8; 32]`. Notably, the test harness in `engine-tests-connector/src/utils.rs` initializes the engine with exactly this value for convenience: [6](#0-5) 

This demonstrates that the `new()` function silently accepts an all-zero `chain_id` with no error, making the mistake easy to replicate in a production deployment. Because `chain_id` is effectively immutable post-initialization, the misconfiguration cannot be corrected without a full contract upgrade.

### Recommendation
Add an explicit guard in `new()` before calling `state::set_state`:

```rust
if state.chain_id == [0u8; 32] {
    return Err(b"ERR_INVALID_CHAIN_ID".into());
}
```

Optionally, enforce a minimum/maximum range matching known Aurora chain IDs (e.g., `1313161554`–`1313161556`) to prevent other nonsensical values.

### Proof of Concept
1. Deploy Aurora Engine calling `new()` with `chain_id = [0u8; 32]` (all zeros). The call succeeds with no error.
2. Deposit ETH into the engine via the NEAR-side `ft_transfer_call`. Funds are credited to an EVM address.
3. Deploy an EVM smart contract that holds those funds (e.g., a simple vault).
4. Attempt to call the vault using any standard Ethereum wallet or library, which signs with EIP-155 and a non-zero chain ID (e.g., `1313161554`).
5. The `submit` entrypoint evaluates `U256::from(1313161554) != U256::from_big_endian(&[0u8; 32])` → `true` → returns `ERR_INVALID_CHAIN_ID`.
6. Every EVM transaction fails. Funds locked in the EVM contract are inaccessible. No admin setter exists to correct `chain_id`; only a full contract upgrade can remediate.

pub fn new<I: IO + Copy, E: Env>(mut io: I, env: &E) -> Result<(), ContractError> {
    if state::get_state(&io).is_ok() {
        return Err(b"ERR_ALREADY_INITIALIZED".into());
    }

    let input = io.read_input().to_vec();
    let args = NewCallArgs::deserialize(&input).map_err(|_| errors::ERR_BORSH_DESERIALIZE)?;

    let initial_hashchain = args.initial_hashchain();
    let state: EngineState = args.into();

    if let Some(block_hashchain) = initial_hashchain {
        let block_height = env.block_height();
        let mut hashchain = Hashchain::new(
            state.chain_id,
            env.current_account_id(),
            block_height,
            block_hashchain,
        );

        hashchain.add_block_tx(
            block_height,
            function_name!(),
            &input,
            &[],
            &Bloom::default(),
        )?;
        crate::hashchain::save_hashchain(&mut io, &hashchain)?;
    }

    state::set_state(&mut io, &state)?;
    Ok(())
}
