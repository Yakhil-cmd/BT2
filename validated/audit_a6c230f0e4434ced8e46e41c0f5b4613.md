### Title
Missing Session ID in Triple Generation Transcript Enables ZK Proof Replay Across Sessions - (File: `src/ecdsa/ot_based_ecdsa/triples/generation.rs`)

### Summary
The `create_transcript` function used to bind ZK proofs during OT-based ECDSA Beaver triple generation does not include any per-session nonce or session identifier. Because the transcript is fully determined by the static participant list and threshold, a malicious participant can replay a `dlog` or `dlogeq` proof produced in a prior triple generation session verbatim in any subsequent session with the same participant set and threshold. Honest verifiers accept the replayed proof, allowing the attacker to contribute the same polynomial commitment — and therefore the same secret polynomial — across multiple triple generation sessions without detection.

### Finding Description

`create_transcript` constructs the Fiat-Shamir transcript that anchors all ZK proofs in the triple generation protocol:

```rust
fn create_transcript(
    participants: &ParticipantList,
    threshold: ReconstructionLowerBound,
) -> Result<Transcript, ProtocolError> {
    let mut transcript = Transcript::new(NEAR_TRIPLE_GENERATION_LABEL);
    transcript.message(b"group", NAME);
    let enc = rmp_serde::encode::to_vec(participants)...;
    transcript.message(b"participants", &enc);
    transcript.message(b"threshold", &u64::try_from(threshold.value())...);
    Ok(transcript)
}
``` [1](#0-0) 

The transcript contains only `LABEL || NAME || participants || threshold`. No session-unique nonce is ever absorbed.

The `dlog` proofs (round 2) and `dlogeq` proofs (round 3) are then generated by forking this transcript with only the participant's own identifier:

```rust
dlog::prove_with_nonce(
    &mut transcript.fork(b"dlog0", &from.bytes()),
    statement0, their_phi_proof0, ...
)
dlogeq::prove_with_nonce(
    &mut transcript.fork(b"dlogeq0", &me.bytes()),
    statement, witness, my_phi_proof_nonces[i],
)
``` [2](#0-1) [3](#0-2) 

The resulting per-participant transcript is:
`LABEL || NAME || participants || threshold || "dlog0" || P.bytes()`

This is **identical** across every triple generation session that uses the same participant set and threshold. The Fiat-Shamir challenge `e` is therefore the same for the same statement and nonce commitment `K = k·G`, making the proof fully replayable.

By contrast, the DKG protocol correctly defends against this by hashing all participants' session IDs into a `session_id` before computing any challenge: [4](#0-3) 

The `challenge` function in DKG explicitly includes `session_id` in the hash preimage: [5](#0-4) 

Triple generation has no equivalent protection.

### Impact Explanation

A malicious participant P who has completed one honest triple generation session possesses:
- A valid `dlog` proof for commitment `E_P` (proving knowledge of `a_P = dlog(E_P.eval_at_zero())`)
- A valid `dlogeq` proof for the round-3 consistency check

In any subsequent session with the same participants and threshold, P can:
1. Broadcast the same commitment `E_P` and the same proofs from the prior session.
2. Send the same polynomial evaluations to each peer (private channel).
3. All verifications pass because the transcript is identical.

The result is that P's contribution to every new triple is the same secret polynomial `f_P`. P therefore knows their own triple shares `(a_P_i, b_P_i, c_P_i)` in every session. Because the presignature nonce `k` is derived from the sum of all participants' `k`-shares, P's share of `k` is constant across all sessions in which they replay. If P participates as coordinator in two signing sessions that use triples generated with this replayed contribution, P can arrange for the same effective nonce material to appear in two distinct presignatures. Two ECDSA signatures produced under the same nonce `k` allow full recovery of the aggregate secret key via the standard nonce-reuse equation `x = (s₁·k − h₁) / r`.

This corrupts presign outputs in a way that honest parties cannot detect (the proofs verify), directly enabling secret key extraction.

### Likelihood Explanation

Any participant who has completed at least one triple generation session can mount this attack in all future sessions with the same group. No external assumptions, leaked keys, or cryptographic breaks are required. The attacker only needs to store their own proof and commitment from a prior session and retransmit them. The attack is silent — honest participants observe a well-formed, verifying proof and have no mechanism to distinguish a replayed proof from a freshly generated one.

### Recommendation

Absorb a per-session random nonce into the transcript before any proof is generated, mirroring the DKG pattern. Each participant should contribute a fresh random value at the start of the protocol; all contributions should be broadcast and hashed together into a `session_id`, which is then fed into `create_transcript`:

```rust
fn create_transcript(
    participants: &ParticipantList,
    threshold: ReconstructionLowerBound,
    session_id: &[u8],          // <-- add this
) -> Result<Transcript, ProtocolError> {
    let mut transcript = Transcript::new(NEAR_TRIPLE_GENERATION_LABEL);
    transcript.message(b"group", NAME);
    transcript.message(b"session-id", session_id);   // <-- absorb it
    ...
}
```

This ensures the Fiat-Shamir challenge is unique per session, making any proof from a prior session invalid in the current one.

### Proof of Concept

```rust
// Session 1 — honest participation
let transcript_s1 = create_transcript(&participants, threshold)?;
let proof_dlog_p = dlog::prove_with_nonce(
    &mut transcript_s1.fork(b"dlog0", &p.bytes()),
    statement_p, witness_p, nonce_p,
)?;
let commitment_p = E_P.clone();

// Session 2 — replay attack (same participants, same threshold)
// P sends `commitment_p` and `proof_dlog_p` unchanged.
// Verifier reconstructs the identical transcript:
let transcript_s2 = create_transcript(&participants, threshold)?;
// transcript_s2 == transcript_s1  (no session ID)
let ok = dlog::verify(
    &mut transcript_s2.fork(b"dlog0", &p.bytes()),
    Statement { public: &commitment_p.eval_at_zero()?.value() },
    &proof_dlog_p,
)?;
assert!(ok);  // passes — proof accepted without fresh randomness
// P's polynomial contribution in session 2 is identical to session 1.
// P knows their own triple shares and can arrange nonce reuse across
// two presigning sessions, enabling full secret key recovery.
```

fn create_transcript(

    threshold: ReconstructionLowerBound,

    let mut transcript = Transcript::new(NEAR_TRIPLE_GENERATION_LABEL);

    let enc = rmp_serde::encode::to_vec(participants).map_err(|_| ProtocolError::ErrorEncoding)?;
    transcript.message(b"participants", &enc);
    // To allow interop between platforms where usize is different
    transcript.message(
        b"threshold",
        &u64::try_from(threshold.value())
            .expect("threshold should always fit in u64")
            .to_be_bytes(),
    );
    Ok(transcript)
}

```rust
fn challenge<C: Ciphersuite>(
    domain_separator: &mut DomainSeparator,
    session_id: &HashOutput,
    id: Scalar<C>,
    vk_share: &CoefficientCommitment<C>,
    big_r: &Element<C>,
) -> Result<Challenge<C>, ProtocolError> {
    let mut preimage = vec![];
    let serialized_id = <C::Group as Group>::Field::serialize(&id);

    // Should not return Error
    // The function should not be called when the first coefficient is zero
    let serialized_vk_share = vk_share.serialize().map_err(|_| {
        ProtocolError::AssertionFailed(
            "The verification share
        could not be serialized as it is null"
                .to_string(),
        )
    })?;

    let serialized_big_r = <C::Group>::serialize(big_r).map_err(|_| {
        ProtocolError::AssertionFailed(
            "The group element R
        could not be serialized as it is the identity"
                .to_string(),
        )
    })?;

    preimage.extend_from_slice(&domain_separator.to_le_bytes());
    preimage.extend_from_slice(session_id.as_ref());
    preimage.extend_from_slice(serialized_id.as_ref());
    preimage.extend_from_slice(serialized_vk_share.as_ref());
    preimage.extend_from_slice(serialized_big_r.as_ref());

    domain_separator.increment();

    let hash = C::HDKG(&preimage[..]).ok_or(ProtocolError::DKGNotSupported)?;
    Ok(Challenge::from_scalar(hash))
}
