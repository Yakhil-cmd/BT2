### Title
Invoke Transaction Admitted to Mempool Without Signature Verification via `skip_stateful_validations` — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The gateway's `skip_stateful_validations` function bypasses the account's `__validate__` entry point for invoke transactions with nonce=1 targeting undeployed accounts that have **any** transaction in the mempool. An unprivileged attacker who observes a victim's `deploy_account` transaction in the mempool can immediately submit a fake invoke (arbitrary calldata, empty/wrong signature) for the same address. The gateway admits it without signature verification. When the batcher executes it, `__validate__` fails, the nonce is incremented, fees are charged to the victim's account, and the victim's legitimate nonce-1 invoke is permanently invalidated.

---

### Finding Description

**Root cause — `skip_stateful_validations`:** [1](#0-0) 

When an invoke transaction satisfies:
- `tx.nonce() == Nonce(Felt::ONE)`, and
- `account_nonce == Nonce(Felt::ZERO)` (account not yet deployed), and
- `account_tx_in_pool_or_recent_block(sender_address)` returns `true`

…the function returns `true` (skip validation). This causes `run_validate_entry_point` to set `execution_flags.validate = false`: [2](#0-1) 

With `validate = false`, `StatefulValidator::perform_validations` returns `Ok(())` immediately without calling `__validate__`: [3](#0-2) 

**The flawed guard — `account_tx_in_pool_or_recent_block`:**

The code comment claims this check proves a `deploy_account` exists for the account: [4](#0-3) 

But the mempool implementation checks for **any** account transaction, not specifically a `deploy_account`: [5](#0-4) 

**Attack path:**

1. Victim submits `deploy_account` for address X (nonce=0). This is publicly observable.
2. Attacker submits a fake `invoke` with `nonce=1`, `sender_address=X`, arbitrary calldata, and an empty/wrong signature.
3. Gateway stateful path: `account_nonce=0`, `tx_nonce=1`, `account_tx_in_pool_or_recent_block(X)=true` (victim's `deploy_account` is in pool) → `skip_validate=true`.
4. `run_validate_entry_point` is called with `validate=false`; `__validate__` is never invoked; the fake invoke is admitted to the mempool. [6](#0-5) 

5. Batcher executes: `deploy_account` (nonce 0→1), then fake invoke (nonce 1→2 via `handle_nonce` in `perform_pre_validation_stage`). [7](#0-6) 

6. `__validate__` is called during execution and fails (wrong signature). The transaction reverts, but the nonce has already been incremented to 2 and fees are charged to address X.
7. Victim's legitimate invoke (nonce=1) is now permanently rejected as "nonce too old."

**No existing guard prevents this.** `validate_by_mempool` only checks nonce range and duplicates, not signatures: [8](#0-7) 

---

### Impact Explanation

**High — Mempool/gateway admission accepts invalid transactions before sequencing.**

- An invoke transaction with a forged/empty signature is accepted through the gateway admission path without any cryptographic authorization check.
- The victim's account pays fees for an unauthorized transaction (fees are charged even on revert because `handle_nonce` and fee deduction occur before the revert point).
- The victim's legitimate nonce-1 invoke is permanently invalidated; the victim must resubmit with nonce=2, paying additional fees.
- The attacker can set a higher tip to ensure their fake invoke is ordered before the victim's legitimate one.

---

### Likelihood Explanation

**Medium.** All submitted transactions are publicly observable on a Starknet node. An attacker needs only to:
1. Watch for `deploy_account` transactions in the mempool (trivially observable).
2. Immediately submit a fake invoke for the same address with nonce=1 and a higher tip.

No privileged access, no special knowledge of the victim's keys, and no dependency on external systems is required.

---

### Recommendation

Replace the coarse `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a `deploy_account` transaction exists in the mempool for the account address. The mempool should expose a dedicated `deploy_account_in_pool(address)` query that inspects transaction types, not just account presence. Until then, the skip-validation UX feature should be disabled or gated behind a stricter proof of ownership (e.g., requiring the invoke to carry a signature over the deploy_account tx hash).

---

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
