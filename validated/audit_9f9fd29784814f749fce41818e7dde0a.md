### Title
Gateway Admits Invoke Transactions with Invalid Signatures via Overly Broad `account_tx_in_pool_or_recent_block` Check — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

`GatewayFixedBlockSyncStateClient::get_nonce` silently returns `Nonce::default()` (zero) for any contract that does not yet exist on-chain. That default zero is then consumed by `skip_stateful_validations`, which uses it as a signal that the account is undeployed and therefore eligible to skip `__validate__` (signature verification). The eligibility guard — `account_tx_in_pool_or_recent_block` — is supposed to confirm that a `deploy_account` transaction is pending for the address, but it actually returns `true` for **any** account that has **any** transaction in the pool or in recent committed history. An unprivileged attacker who observes a target account with a pending transaction in the mempool can therefore submit an invoke with nonce=1 and a completely invalid (or absent) signature, have it admitted to the mempool without signature verification, and cause it to be forwarded to the batcher.

---

### Finding Description

**Root cause — default nonce for non-existent contracts**

`GatewayFixedBlockSyncStateClient::get_nonce` maps `ContractNotFound` to `Nonce::default()` (i.e. `Nonce(Felt::ZERO)`):

```rust
// crates/apollo_gateway/src/gateway_fixed_block_state_reader.rs L69-77
async fn get_nonce(&self, contract_address: ContractAddress) -> StarknetResult<Nonce> {
    match self.state_sync_client.get_nonce_at(self.block_number, contract_address).await {
        Ok(nonce) => Ok(nonce),
        Err(StateSyncClientError::StateSyncError(StateSyncError::ContractNotFound(_))) => {
            Ok(Nonce::default())   // ← silent zero for non-existent accounts
        }
        Err(e) => Err(StarknetError::internal_with_logging("Failed to get nonce", e)),
    }
}
```

This zero is indistinguishable from the nonce of a legitimately deployed account that has never sent a transaction. [1](#0-0) 

**Propagation — `skip_stateful_validations` uses the zero as a skip trigger**

`extract_state_nonce_and_run_validations` calls `get_nonce_from_state`, then passes the result to `run_pre_validation_checks`, which calls `skip_stateful_validations`:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs L429-461
async fn skip_stateful_validations(...) -> ... {
    if let ExecutableTransaction::Invoke(...) = tx {
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            // comment claims this verifies a deploy_account is pending
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await ...;
        }
    }
    Ok(false)
}
```

When this returns `true`, `run_validate_entry_point` is called with `validate = false`, meaning the `__validate__` Cairo entry point — the account's signature-verification function — is **never executed** during gateway admission. [2](#0-1) [3](#0-2) 

**The broken guard — `account_tx_in_pool_or_recent_block` is not deploy-account-specific**

The comment at line 440-443 states: *"It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."* The implementation, however, checks for **any** transaction type:

```rust
// crates/apollo_mempool/src/mempool.rs L697-700
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)   // any committed history
        || self.tx_pool.contains_account(account_address)  // any pooled tx
}
```

`tx_pool.contains_account` returns `true` if the pool holds an invoke, a declare, or a deploy_account for that address. `state.contains_account` returns `true` if the address appeared in any committed block within the retention window. [4](#0-3) [5](#0-4) 

**Concrete attack path**

1. Account A is deployed on-chain with nonce = 0 (or has a pending `deploy_account` in the pool). Its legitimate owner submits an invoke with nonce = 0 carrying a valid signature; this passes `__validate__` and enters the pool.
2. `tx_pool.contains_account(A)` now returns `true`.
3. Attacker submits an invoke for A with nonce = 1 and an **arbitrary/invalid signature**.
4. Gateway stateful path: `get_nonce(A)` → 0; `validate_nonce` passes (0 ≤ 1 ≤ 200); `validate_by_mempool` passes (nonce not too old); `skip_stateful_validations` evaluates `nonce==1 && account_nonce==0 && account_tx_in_pool_or_recent_block(A)==true` → returns `true`.
5. `run_validate_entry_point` is invoked with `validate = false`; the `__validate__` entry point is **skipped entirely**.
6. The attacker's invoke with the invalid signature is forwarded to the mempool and accepted. [6](#0-5) [7](#0-6) 

The mempool's own `validate_tx` only checks for duplicate hashes and nonce ordering; it does not re-verify signatures. [8](#0-7) 

---

### Impact Explanation

The gateway's core admission invariant — *every transaction entering the mempool has had its account's `__validate__` entry point successfully executed* — is broken for invoke transactions with nonce = 1 targeting any account that has any transaction in the pool or recent committed history. An attacker can inject transactions with invalid signatures into the mempool without restriction. This matches:

> **High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.**

Secondary effects: mempool resource exhaustion (each injected transaction occupies pool capacity until the batcher rejects it at execution time), and disruption of the legitimate account's transaction ordering (the batcher will attempt to execute the invalid nonce-1 transaction after the valid nonce-0 one, fail `__validate__`, and mark it reverted, potentially delaying or displacing other transactions).

---

### Likelihood Explanation

- No privileged access is required. Any party that can submit transactions to the gateway can execute this attack.
- The precondition (target account has any pending transaction in the pool) is trivially observable from the public mempool or P2P gossip layer, and is satisfied for every active account during normal operation.
- The attack is repeatable: after the batcher reverts the invalid transaction, the attacker can immediately submit another one.

---

### Recommendation

Replace the overly broad `account_tx_in_pool_or_recent_block` check with one that specifically verifies a **`deploy_account` transaction** is pending for the address. The mempool should expose a dedicated query such as `has_pending_deploy_account(address) -> bool` that inspects the pool for a `DeployAccount` variant at nonce 0. Only when such a transaction is confirmed pending should `skip_stateful_validations` return `true`.

Additionally, consider whether the `Nonce::default()` sentinel returned for non-existent contracts should be distinguished from a genuine on-chain nonce of zero, for example by returning `Option<Nonce>` from `get_nonce` and treating `None` (contract not found) as a separate case that does not trigger the validation-skip path.

---

### Proof of Concept

```
// Setup: account A is deployed on-chain, nonce = 0.
// Step 1: legitimate owner sends valid invoke, nonce=0 → admitted to pool.
//   tx_pool.contains_account(A) == true

// Step 2: attacker constructs invoke for A, nonce=1, signature = [0xdead, 0xbeef]
//   (completely invalid)

// Step 3: gateway processes attacker's tx
//   get_nonce(A) → Nonce(0)          [account exists, nonce is 0]
//   validate_nonce: 0 <= 1 <= 200    [passes]
//   validate_by_mempool: nonce not too old [passes]
//   skip_stateful_validations:
//     nonce == 1 ✓
//     account_nonce == 0 ✓
//     account_tx_in_pool_or_recent_block(A) == true ✓  ← triggered by nonce-0 invoke, not deploy_account
//     → returns true (skip __validate__)
//   run_validate_entry_point(skip_validate=true):
//     execution_flags.validate = false
//     __validate__ is NOT called
//   → attacker's tx with invalid signature admitted to mempool ✓

// Step 4: batcher picks up nonce-0 tx (executes, nonce becomes 1),
//         then picks up attacker's nonce-1 tx → __validate__ called → FAIL → reverted.
//         But the invalid tx was in the mempool and consumed resources.
```

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
```
