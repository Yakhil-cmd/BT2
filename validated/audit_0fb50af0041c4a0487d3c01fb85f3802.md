### Title
Block-Number-Based Velocity Guard Miscalibrates Across Chains ‚Äî (`metric-periphery/contracts/extensions/PriceVelocityGuardExtension.sol`)

---

### Summary

`PriceVelocityGuardExtension` measures elapsed time in **blocks** (`block.number`) rather than **seconds** (`block.timestamp`). Because Metric OMM is deployed on Ethereum (~12 s/block), Base (~2 s/block), and HyperEVM (sub-second blocks), the effective per-second velocity budget differs by up to an order of magnitude across chains for the same `maxChangePerBlockE18` value, silently over-permitting or over-restricting price movement depending on the deployment target.

---

### Finding Description

The extension stores the last-seen block number and computes the allowed price deviation as:

```
allowedSq = maxChange¬≤ √ó (1 + blockDiff)
``` [1](#0-0) 

The parameter is typed and named `maxChangePerBlockE18` ‚Äî explicitly a per-block quantity: [2](#0-1) 

For a fixed real-world interval *T* seconds, `blockDiff` equals `T / blockTime`. On Base (2 s/block) `blockDiff` is 6√ó larger than on Ethereum (12 s/block) for the same elapsed wall-clock time. The allowed deviation therefore scales as `sqrt(1 + T/blockTime)`, which diverges significantly across chains:

| Chain | Block time | blockDiff over 12 s | Allowed multiplier vs. 1-block Ethereum |
|---|---|---|---|
| Ethereum | 12 s | 1 | 1√ó |
| Base | 2 s | 6 | ‚àö7/‚àö2 ‚âà **1.87√ó** |
| HyperEVM | ~0.5 s | 24 | ‚àö25/‚àö2 ‚âà **3.54√ó** |

A pool admin who sets `maxChangePerBlockE18` to allow, say, 0.5 % per Ethereum block (‚âà per 12 s) will inadvertently allow ~0.94 % per 12 s on Base and ~1.77 % per 12 s on HyperEVM ‚Äî with no on-chain signal that the guard has been weakened.

The converse also holds: a value calibrated for HyperEVM will be far too tight on Ethereum, causing `PriceVelocityExceeded` reverts on legitimate swaps.

---

### Impact Explanation

**Over-permissive (fast-block chains ‚Äî Base, HyperEVM):** The velocity guard is the last line of defence against rapid oracle-price manipulation that could let a swap execute at a price far from the true mid. With the guard 1.87‚Äì3.54√ó more permissive per second than the admin intended, an attacker who can push the oracle price quickly (e.g., via a high-frequency update window) can execute swaps at prices that extract value from LP bins before the guard fires. This maps to the "bad-price execution" and "LP asset loss" impact categories.

**Over-restrictive (slow-block chains / difficulty bomb):** Legitimate swaps revert with `PriceVelocityExceeded` whenever the oracle price moves at a normal rate but fewer blocks have elapsed than expected. Core swap functionality is broken for the duration of the block-time deviation.

---

### Likelihood Explanation

- The protocol explicitly targets Ethereum, Base, and HyperEVM (README), all with materially different block times.
- The parameter name `maxChangePerBlockE18` gives no hint that it must be re-calibrated per chain; a pool admin deploying the same configuration across chains will naturally produce miscalibrated guards.
- No on-chain check, cap, or warning enforces or documents a chain-specific calibration requirement.
- The error is silent and cumulative in the permissive direction: every swap that passes the weakened guard is a potential value-extraction event.

---

### Recommendation

Replace `block.number` / `lastUpdateBlock` with `block.timestamp` / `lastUpdateTimestamp` and rename the parameter to `maxChangePerSecondE18`. The allowed deviation formula becomes:

```solidity
uint256 secondsDiff = block.timestamp - s.lastUpdateTimestamp;
uint256 allowedSq = uint256(maxChange) * uint256(maxChange) * (1 + secondsDiff);
```

This decouples the guard from chain-specific block times, exactly as the Compound report recommended for interest accrual. `block.timestamp` manipulation by validators is bounded to a narrow window and, crucially, is non-cumulative.

---

### Proof of Concept

1. Deploy `PriceVelocityGuardExtension` on Base (2 s/block).
2. Pool admin calls `setMaxChangePerBlock(pool, X)` intending to allow X% per 12 s (one Ethereum block).
3. Wait 12 real seconds ‚Üí 6 Base blocks elapse ‚Üí `blockDiff = 6`.
4. `allowedSq = X¬≤ √ó 7`; effective allowed change = `X √ó ‚àö7 ‚âà 2.65X` ‚Äî **2.65√ó the intended cap**.
5. An oracle update that moves the mid-price by `2X` (which should be blocked) passes the guard and the swap executes at the manipulated price, extracting value from LP bins. [3](#0-2)
