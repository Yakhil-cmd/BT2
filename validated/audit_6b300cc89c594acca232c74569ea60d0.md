### Title
Unprivileged Caller Can Suppress All Consensus Block Rewards by Calling `update_rewards` with `disable_rewards: true` — (File: `src/staking/staking.cairo`)

---

### Summary

The `update_rewards` function in the `Staking` contract is callable by any non-zero address. It accepts a caller-controlled `disable_rewards: bool` parameter. When set to `true`, the function writes the current block number to `last_reward_block` **before** checking `disable_rewards`, then returns early without distributing any rewards. Because `last_reward_block` is already advanced, no legitimate call can distribute rewards for that block. An attacker can repeat this every block to permanently suppress all consensus-era staker yield.

---

### Finding Description

`update_rewards` is exposed as a public ABI function under `IStakingRewardsManager`. Its only gate is `general_prerequisites`, which checks the contract is not paused and the caller is non-zero: [1](#0-0) 

The function then unconditionally advances `last_reward_block` to the current block: [2](#0-1) 

Only **after** that write does it check `disable_rewards`: [3](#0-2) 

The guard that prevents double-processing is:

```cairo
assert!(
    current_block_number > self.last_reward_block.read(),
    "{}",
    Error::REWARDS_ALREADY_UPDATED,
);
``` [4](#0-3) 

Because `last_reward_block` is written before the early-return path, any call with `disable_rewards: true` permanently consumes that block's reward slot. The legitimate sequencer call that arrives later in the same block will revert with `REWARDS_ALREADY_UPDATED`.

The `general_prerequisites` helper confirms there is no role check: [5](#0-4)
