### Title
Gateway Skips Signature Verification for Invoke Transactions Targeting Accounts with Pending Deploy-Account — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validation path unconditionally bypasses the account's `__validate__` entry point (which performs ECDSA signature verification) for any Invoke transaction with `nonce == 1` sent to an account whose on-chain nonce is `0` and which has any transaction present in the mempool or a recent committed block. An unprivileged attacker can exploit this to inject Invoke transactions carrying an arbitrary or empty signature into the mempool without any cryptographic check.

---

### Finding Description

The gateway stateful path in `extract_state_nonce_and_run_validations` calls `run_pre_validation_checks`, which in turn calls `skip_stateful_validations`: [1](#0-0) 

The function returns `true` (skip validation) when all three conditions hold:

1. The transaction is an `Invoke` with `nonce == Nonce(Felt::ONE)`.
2. The on-chain account nonce is `Nonce(Felt::ZERO)` (account not yet deployed).
3. `account_tx_in_pool_or_recent_block` returns `true` for the sender address. [2](#0-1) 

When `skip_validate == true`, `run_validate_entry_point` constructs `ExecutionFlags` with `validate: false`: [3](#0-2) 

Inside the blockifier, `validate_tx` immediately returns `Ok(None)` when `execution_flags.validate` is `false`, so the account's `__validate__` entry point — which is the sole place ECDSA signature verification occurs — is never called: [4](#0-3) 

The mempool's own `validate_tx` only checks nonce ordering and fee escalation; it performs no cryptographic check: [5](#0-4) 

`account_tx_in_pool_or_recent_block` returns `true` as long as the target account appears in the pool **or** in the mempool's committed-block state, which persists after the deploy-account is finalized: [6](#0-5) 

---

### Impact Explanation

An attacker who observes any `DeployAccount` transaction for address `X` in the mempool can immediately submit an `Invoke` transaction for `X` with `nonce = 1`, arbitrary calldata, and a zeroed or random signature. The gateway will admit it to the mempool without verifying the signature. The mempool is polluted with cryptographically invalid transactions. At batcher execution time the transaction will fail `__validate__` and be dropped without fee, so the attacker bears no cost. Repeated at scale this constitutes a free, permissionless DoS against the mempool for every newly-deploying account.

Impact: **High — Mempool/gateway admission accepts invalid transactions before sequencing.**

---

### Likelihood Explanation

The trigger is fully unprivileged: any observer of the public mempool can craft the attack. The condition (`nonce == 1`, on-chain nonce `== 0`, account present in pool) is trivially satisfied for every new account deployment. No special knowledge of the victim's private key is required.

---

### Recommendation

Do not skip the `__validate__` entry point entirely. Instead, run `__validate__` against a synthetic state that includes the pending `DeployAccount` class hash, so the signature is verified even before the account is on-chain. Alternatively, require the gateway to verify the signature off-chain (e.g., via a lightweight ECDSA check against the expected public key embedded in the `DeployAccount` constructor calldata) before admitting the Invoke to the mempool when `skip_stateful_validations` would otherwise fire.

---

### Proof of Concept

1. Alice broadcasts `DeployAccount(address=X, public_key=PK, nonce=0)`. It enters the mempool; `account_tx_in_pool_or_recent_block(X)` now returns `true`.
2. Attacker submits `Invoke(sender=X, nonce=1, calldata=<anything>, signature=[0x0, 0x0])` to the gateway.
3. Gateway stateless check passes (signature length ≤ max, resource bounds valid).
4. `validate_nonce`: `account_nonce(0) ≤ tx_nonce(1) ≤ max_allowed_nonce_gap` — passes.
5. `validate_by_mempool`: nonce ordering check passes.
6. `skip_stateful_validations`: `tx.nonce()==1 && account_nonce==0 && account_tx_in_pool_or_recent_block(X)==true` → returns `true`.
7. `run_validate_entry_point` is called with `validate: false`; `blockifier_validator.validate(account_tx)` skips `__validate__` entirely.
8. Gateway returns success; the unsigned Invoke is inserted into the mempool.
9. Attacker repeats for every new `DeployAccount` observed, flooding the mempool at zero cost. [7](#0-6) [8](#0-7)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
