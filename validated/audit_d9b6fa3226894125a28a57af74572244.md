### Title
Owner Cannot Directly Remove Compromised Key Manager's Relayer Keys - (File: `engine/src/contract_methods/admin.rs`)

### Summary
`remove_relayer_key` is gated exclusively by `require_key_manager_only`, mirroring the original report's pattern exactly: the owner cannot directly revoke NEAR function-call keys added by a compromised key manager. The owner must first rotate the key manager via `set_key_manager`, then remove the keys — creating an exposure window during which a malicious relayer key can execute EVM calls with `predecessor_account_id` = `aurora`, impersonating the Aurora contract's own EVM address.

### Finding Description

`add_relayer_key` attaches a NEAR function-call key **to the Aurora contract account itself**, scoped to `"call,submit,submit_with_args"`:

```rust
// engine/src/contract_methods/admin.rs:298-344
pub fn add_relayer_key<...>(...) -> Result<(), ContractError> {
    ...
    require_key_manager_only(&state, &env.predecessor_account_id())?;  // line 306
    ...
    PromiseAction::AddFunctionCallKey {
        public_key,
        allowance,
        receiver_id: current_account_id,          // Aurora contract itself
        function_names: "call,submit,submit_with_args".into(),
    },
```

`remove_relayer_key` carries the same exclusive guard:

```rust
// engine/src/contract_methods/admin.rs:370-399
pub fn remove_relayer_key<...>(...) -> Result<(), ContractError> {
    ...
    require_key_manager_only(&state, &env.predecessor_account_id())?;  // line 379
```

`require_key_manager_only` admits **only** the key manager — the owner is not an alternative:

```rust
// engine/src/contract_methods/mod.rs:99-111
fn require_key_manager_only(state: &state::EngineState, predecessor_account_id: &AccountId)
    -> Result<(), ContractError>
{
    let key_manager = state.key_manager.as_ref().ok_or(errors::ERR_KEY_MANAGER_IS_NOT_SET)?;
    if key_manager != predecessor_account_id {
        return Err(errors::ERR_NOT_ALLOWED.into());
    }
    Ok(())
}
```

The owner's only recourse is the two-step sequence: call `set_key_manager` (owner-only) to rotate the key manager, then call `remove_relayer_key` as the new key manager. During the gap between key manager compromise and owner response, the malicious key is live on the Aurora account.

The critical consequence: because the relayer key is a function-call key **on the Aurora contract account**, any NEAR transaction signed with it has `predecessor_account_id` = `aurora`. The `call` entrypoint derives the EVM origin directly from this:

```rust
// engine/src/contract_methods/evm_transactions.rs:57-65
let predecessor_account_id = env.predecessor_account_id();
let mut engine: Engine<_, E, AuroraModExp> = Engine::new_with_state(
    state,
    predecessor_address(&predecessor_account_id),   // EVM origin = keccak("aurora")[12..]
    ...
);
```

```rust
// engine/src/contract_methods/mod.rs:113-115
fn predecessor_address(predecessor_account_id: &AccountId) -> Address {
    aurora_engine_sdk::types::near_account_to_evm_address(predecessor_account_id.as_bytes())
}
```

The Aurora contract's implicit EVM address (`near_account_to_evm_address("aurora".as_bytes())`) is the same address used as the XCC engine implicit address to hold wNEAR/ETH for sub-account funding:

```rust
// engine-precompiles/src/xcc.rs:185-190
let engine_implicit_address = aurora_engine_sdk::types::near_account_to_evm_address(
    self.engine_account_id.as_bytes(),
);
// NEAR tokens are transferred TO this address for XCC operations
```

This address accumulates real ETH/wNEAR balance. A malicious relayer key can call `call` with arbitrary `CallArgs`, executing EVM transfers **from** that address to an attacker-controlled EVM address.

### Impact Explanation

**Impact: High — Theft of funds / Temporary freezing of funds.**

A compromised key manager can add a malicious NEAR function-call key to the Aurora contract account. That key can invoke `call` with `predecessor_account_id` = `aurora`, executing arbitrary EVM calls as the Aurora contract's EVM address. Because that address holds ETH/wNEAR accumulated from XCC operations, the attacker can drain it. The owner cannot immediately revoke the key; they must first rotate the key manager, then remove the key — a two-step process that creates a real exposure window.

### Likelihood Explanation

**Likelihood: Low.**

Requires the key manager's private key to be leaked or seized — the same scenario class as the original report (LP private key compromise). The key manager is a trusted operational role, so compromise is unlikely but not impossible (key exfiltration, insider threat, infrastructure breach).

### Recommendation

Allow the owner to also call `remove_relayer_key` (and symmetrically `add_relayer_key`) by replacing the exclusive `require_key_manager_only` guard with a combined check:

```rust
// engine/src/contract_methods/admin.rs
pub fn remove_relayer_key<...>(...) -> Result<(), ContractError> {
    ...
    let predecessor = env.predecessor_account_id();
    // Allow either the key manager OR the owner to remove keys
    if require_key_manager_only(&state, &predecessor).is_err() {
        require_owner_only(&state, &predecessor)?;
    }
    ...
}
```

This mirrors the recommendation in the original report: allow the owner to act as a fallback authority without removing the key manager's normal operational ability.

### Proof of Concept

1. Key manager's private key is leaked or seized.
2. Attacker calls `add_relayer_key` (as the compromised key manager) with an attacker-controlled public key and a minimal NEAR allowance (≥ 100 yoctoNEAR). This attaches a function-call key to the Aurora contract account scoped to `"call,submit,submit_with_args"`.
3. Attacker signs a NEAR transaction with that key calling `call` on the Aurora contract. Because the key is on the Aurora account, `predecessor_account_id` = `aurora`.
4. The EVM engine sets origin = `near_account_to_evm_address("aurora".as_bytes())` — the Aurora contract's implicit EVM address.
5. The `CallArgs` payload encodes an EVM transfer from that address to the attacker's EVM address, draining its ETH/wNEAR balance.
6. The owner cannot immediately remove the malicious key: `remove_relayer_key` rejects them with `ERR_NOT_ALLOWED`. The owner must first call `set_key_manager` (rotating the key manager to themselves), then call `remove_relayer_key` — a two-block-minimum delay during which the malicious key remains active.
