Looking at the `assert_reshare_keys_invariants` function in `src/dkg.rs`, I can identify a direct analog to the reported vulnerability class.

---

### Title
Missing Non-Zero Validation on `old_threshold` Bypasses Reshare Continuity Guard — (File: `src/dkg.rs`)

### Summary
`assert_reshare_keys_invariants` validates the new threshold via `assert_key_invariants` (which enforces `threshold >= 2`), but applies **no minimum-value check** on `old_threshold`. When a caller passes `old_threshold = 0`, the intersection continuity guard `intersection.len() < old_threshold` becomes `intersection.len() < 0usize`, which is always `false` in Rust because `usize` is unsigned. The guard is silently bypassed regardless of how few old participants are present in the new set, allowing a reshare to proceed with an insufficient intersection and producing new key shares that are cryptographically inconsistent with the old public key.

---

### Finding Description

In `src/dkg.rs`, `assert_reshare_keys_invariants` is the public validation gate for the reshare protocol:

```rust
pub fn assert_reshare_keys_invariants<C: Ciphersuite>(
    participants: &[Participant],
    me: Participant,
    threshold: impl Into<ReconstructionLowerBound>,
    old_signing_key: Option<SigningShare<C>>,
    old_threshold: impl Into<ReconstructionLowerBound>,   // ← no lower-bound check
    old_participants: &[Participant],
) -> Result<(ParticipantList, ParticipantList), InitializationError> {
    let threshold = usize::from(threshold.into());
    let old_threshold = usize::from(old_threshold.into());

    let participants = assert_key_invariants(participants, me, threshold)?; // validates NEW threshold ≥ 2
    // old_threshold is never validated
    ...
    if old_participants.intersection(&participants).len() < old_threshold {  // ← guard
        return Err(InitializationError::NotEnoughParticipantsForNewThreshold { ... });
    }
``` [1](#0-0) 

`assert_key_invariants` enforces `threshold >= 2` for the **new** threshold:

```rust
if threshold < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
}
``` [2](#0-1) 

But `old_threshold` is never passed through `assert_key_invariants` and receives no analogous check. When `old_threshold = 0`, the guard at line 655 evaluates as:

```
intersection.len() < 0usize   →   always false
```

because `usize` is unsigned and `intersection.len()` is always `≥ 0`. The guard is unconditionally bypassed. [3](#0-2) 

The downstream `do_reshare` then computes Lagrange coefficients over the (potentially empty or undersized) intersection:

```rust
let intersection = old_participants.intersection(&participants);
let secret = old_signing_key
    .map(|x_i| {
        intersection
            .lagrange::<C>(me)
            .map(|lambda| lambda * x_i.to_scalar())
    })
    .transpose()?
    .unwrap_or_else(<C::Group as Group>::Field::zero);
``` [4](#0-3) 

If the intersection is smaller than the actual old threshold, the Lagrange interpolation is performed over an insufficient number of points of a degree-`(old_threshold − 1)` polynomial. The reconstructed secret is wrong, and the new participants receive shares of a key that does not correspond to the old public key.

---

### Impact Explanation

A malicious coordinator who controls the reshare invocation can pass `old_threshold = 0`. The intersection continuity guard is bypassed. `do_reshare` proceeds with fewer old participants than the old threshold requires. The Lagrange interpolation over the undersized intersection produces an incorrect linearized secret. New participants accept and store shares of a key that is mathematically inconsistent with the old `VerifyingKey`. Any subsequent signing attempt using these shares will produce signatures that fail verification against the old public key, permanently corrupting the reshared key material.

**Impact class:** High — Corruption of reshare outputs so honest parties accept inconsistent public keys and unusable cryptographic outputs.

---

### Likelihood Explanation

`assert_reshare_keys_invariants` is a public library function. Any caller — including a malicious coordinator orchestrating a reshare — can supply `old_threshold = 0` (or `ReconstructionLowerBound(0)`) directly. No cryptographic capability is required; the attack is a trivial API misuse that the library's own validation is supposed to prevent but does not. The contrast with the new-threshold path (which is properly guarded) makes this omission easy to miss during code review.

---

### Recommendation

Add a minimum-value check for `old_threshold` in `assert_reshare_keys_invariants`, mirroring the check already applied to the new threshold in `assert_key_invariants`:

```rust
if old_threshold < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold: old_threshold, min: 2 });
}
```

This should appear immediately after `let old_threshold = usize::from(old_threshold.into());` and before the intersection check, consistent with how `assert_key_invariants` validates the new threshold.

---

### Proof of Concept

1. Old setup: participants `{A, B, C, D, E}`, old threshold = 3 (degree-2 polynomial).
2. New setup: participants `{A, B, F, G, H}`, new threshold = 2. Intersection = `{A, B}` (size 2).
3. Call `assert_reshare_keys_invariants(..., old_threshold = 0, ...)`.
4. Guard evaluates: `2 < 0usize` → `false` → **passes without error**.
5. `do_reshare` is called. A and B each compute `intersection.lagrange::<C>(me)` over `{A, B}` only.
6. Lagrange interpolation over 2 points of a degree-2 polynomial does **not** reconstruct the constant term (the old secret). The linearized secrets are wrong.
7. New participants `{A, B, F, G, H}` receive and store shares of an incorrect key.
8. The new key's public component does not match the old `VerifyingKey`; all future signatures are invalid.
