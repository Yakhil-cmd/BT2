### Title
TOCTOU in Pool Join/Change: `prepareSubmitData` Re-fetches `targetPuzzleHash` at Submit Time, Bypassing User Review — (`packages/gui/src/components/plotNFT/select/PlotNFTSelectPool.tsx`)

### Summary

The GUI fetches `/pool_info` twice: once at render time (via `usePoolInfo`) to display the pool's `targetPuzzleHash` for user review, and again at submit time (via `getPoolInfo` inside `prepareSubmitData`) to build the actual transaction. A pool operator who controls the server can serve a different `targetPuzzleHash` on the second request, causing the user's farming rewards to be redirected to an attacker-controlled address that the user never reviewed.

### Finding Description

**Display fetch** — `PlotNFTSelectBase` calls `usePoolInfo(poolUrl)` at render time: [1](#0-0) 

`usePoolInfo` internally calls `getPoolInfo(normalizedUrl)`, which issues an HTTP request to `${poolUrl}/pool_info`: [2](#0-1) 

The result is rendered in the "Verify Pool Details" card, including `targetPuzzleHash`: [3](#0-2) 

**Submit fetch** — when the user clicks "Change", `handleSubmit` calls `prepareSubmitData`, which makes a **second, independent** HTTP request to the same endpoint: [4](#0-3) 

The `targetPuzzleHash` from this second fetch — not the one shown to the user — is what gets passed to `pwJoinPool`: [5](#0-4) 

There is no comparison between the displayed value and the submitted value anywhere in the code path.

**The underlying transport** — both calls ultimately go through `window.appAPI.fetchPoolInfo`, which hits `${poolUrl}/pool_info` with no caching or pinning: [6](#0-5) [7](#0-6) 

### Impact Explanation

A malicious pool operator serves `targetPuzzleHash = ATTACKER_HASH` on the second `/pool_info` request (the submit-time fetch) while serving `targetPuzzleHash = LEGITIMATE_HASH` on the first (the display-time fetch). The user reviews `LEGITIMATE_HASH`, clicks "Change", and the daemon receives `ATTACKER_HASH`. All future farming rewards for that Plot NFT are paid to the attacker. This is a **Critical** impact: unauthorized payout change affecting pooled farming rewards.

### Likelihood Explanation

Any pool operator can execute this attack with a trivial server-side change (e.g., a request counter that returns different hashes on odd vs. even calls). The user has no way to detect the substitution because the GUI never compares the displayed hash to the submitted hash. The window between the two fetches is the time the user spends reviewing the form — typically several seconds, more than enough for a server-side swap.

### Recommendation

Eliminate the second fetch entirely. The `targetPuzzleHash` and `relativeLockHeight` already fetched by `usePoolInfo` should be stored in the form state (e.g., as hidden fields via `react-hook-form`'s `setValue`) or passed directly into `prepareSubmitData`. The submit path must use the same values that were displayed to the user, with no re-fetch.

```typescript
async function prepareSubmitData(data: FormData): SubmitData {
  const { self, fee, poolUrl } = data;
  const initialTargetState = {
    state: self ? 'SELF_POOLING' : 'FARMING_TO_POOL',
  };

  if (!self && poolUrl) {
    const normalizedPoolUrl = normalizeUrl(poolUrl);
    const { targetPuzzleHash, relativeLockHeight } = await getPoolInfo(normalizedPoolUrl);
    if (!targetPuzzleHash) {
      throw new Error(t`Pool does not provide targetPuzzleHash.`);
    }
    if (relativeLockHeight === undefined) {
      throw new Error(t`Pool does not provide relativeLockHeight.`);
    }

    initialTargetState.poolUrl = normalizedPoolUrl;
    initialTargetState.targetPuzzleHash = targetPuzzleHash;
    initialTargetState.relativeLockHeight = relativeLockHeight;
```
