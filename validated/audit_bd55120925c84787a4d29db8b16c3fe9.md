### Title
Missing Caller Validation on `update_rewards` Allows Any Address to Block Reward Distribution - (File: src/staking/staking.cairo)

### Summary
The `update_rewards` function in `Staking.cairo` is documented as callable only by the Starkware sequencer, but the implementation contains no caller validation. Any unprivileged address can call it with `disable_rewards: true`, consuming the single per-block reward slot and preventing the legitimate sequencer call from succeeding. Stakers lose unclaimed yield for every block where the attacker front-runs.

### Finding Description
The specification at `docs/spec.md` lines 1644–1645 explicitly states:

> **access control**: Only starkware sequencer.

The implementation at `src/staking/staking.cairo` lines 1449–1507 enforces no such restriction. The function body checks only:

1. Contract is not paused (`general_prerequisites()`)
2. `current_block_number > last_reward_block` (not already updated this block)
3. Staker exists and is active
4. Staker has non-zero balance [1](#0-0) 

The critical state mutation happens unconditionally at line 1485, **before** the `disable_rewards` early-return guard:

```cairo
// Update last block rewards.
self.last_reward_block.write(current_block_number);

if disable_rewards || self.is_pre_consensus() {
    return;
}
``` [2](#0-1) 

`last_reward_block` is a single global storage slot (not per-staker). Once written to `current_block_number`, any subsequent call in the same block fails with `REWARDS_ALREADY_UPDATED`. [3](#0-2) 

The spec confirms this is the intended single-call-per-block design:

> **pre-condition**: Rewards did not distributed for the current block yet. [4](#0-3) 

### Impact Explanation
An attacker calls `update_rewards(any_valid_staker_address, disable_rewards: true)` on every block before the sequencer's transaction executes. Each such call:

1. Passes all validation checks (staker exists, has balance, block is new)
2. Writes `last_reward_block = current_block_number`
3. Returns immediately without distributing any rewards

The sequencer's subsequent call in the same block fails with `REWARDS_ALREADY_UPDATED`. Stakers accumulate zero `unclaimed_rewards_own` for every blocked block. Sustained over time this constitutes **permanent freezing of unclaimed yield** for all stakers.

This maps to the allowed impact: **High — Permanent freezing of unclaimed yield or unclaimed royalties**.

### Likelihood Explanation
The entry point is a public, unguarded function callable by any address with no token deposit or stake required. The attacker only needs to submit a transaction per block with a valid (existing, active) staker address as the argument. Valid staker addresses are publicly readable from on-chain events (`NewStaker`). The cost is gas per block; no capital is at risk for the attacker. [5](#0-4) 

### Recommendation
Add a caller check at the top of `update_rewards`, analogous to the existing `assert_caller_is_attestation_contract` pattern used in `update_rewards_from_attestation_contract`:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
    self.assert_caller_is_sequencer(); // <-- add this
    ...
}
```

Store the sequencer address during construction (or via a governance setter) and validate `get_caller_address() == sequencer_address`. [6](#0-5) 

### Proof of Concept

1. Deploy the system normally; advance past K epochs so a staker has effective balance and consensus rewards are active.
2. From any unprivileged EOA, call:
   ```
   IStakingRewardsManager(staking_contract).update_rewards(
       staker_address: <any_valid_staker>,
       disable_rewards: true
   )
   ```
3. Observe that `last_reward_block` is now set to the current block number and no rewards were distributed.
4. The sequencer's legitimate `update_rewards` call in the same block reverts with `REWARDS_ALREADY_UPDATED`.
5. Repeat on every block. The staker's `unclaimed_rewards_own` never increases despite being an active, attesting staker.

The test suite already demonstrates that `update_rewards` succeeds without any privileged caller — `test_update_rewards_only_staker` calls it directly with no `cheat_caller_address` wrapping, confirming the absence of access control. [7](#0-6)
