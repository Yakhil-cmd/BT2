Looking at the `skip_stateful_validations` function and the full gateway stateful validation path, I need to trace the exact conditions under which signature verification is bypassed.

### Title
Signature Bypass via Overly Broad `skip_stateful_validations` Precondition Allows Unsigned Invoke Admission to Mempool - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The gateway's `skip_stateful_validations` function skips the `__validate__` entry-point (signature verification) for any Invoke transaction with `nonce == 1` targeting an undeployed account (`account_nonce == 0`), whenever `account_tx_in_pool_or_recent_block` returns `true` for that sender address. The check does **not** verify that the transaction already in the pool is specifically a `deploy_account`. An attacker who observes a victim's `deploy_account` (nonce 0) enter the mempool can immediately front-run with a crafted Invoke (nonce 1, arbitrary invalid signature) for the victim's address. The gateway admits it without calling `__validate__`, placing an unauthenticated transaction in the mempool and causing the victim's legitimate Invoke (nonce 1) to be rejected with `DuplicateNonce`.

---

### Finding Description

**Broken invariant:** Every transaction admitted to the mempool must have passed account-contract signature verification via the `__validate__` entry point, unless the account is provably undeployed *and* a `deploy_account` for it is pending.

### Impact Explanation

**Impact: High — Mempool/gateway admission accepts an invalid (unauthenticated) transaction before sequencing.**

1. An Invoke transaction with a completely invalid signature for a victim's address is admitted to the mempool without any signature check.
2. The victim's legitimate Invoke (nonce 1) is subsequently rejected with `MempoolError::DuplicateNonce`, disrupting the intended `deploy_account + invoke` UX flow.
3. The attacker can repeat this for every new `deploy_account` they observe, systematically blocking the first post-deploy Invoke for any victim.
4. The invalid transaction consumes batcher resources when it is eventually executed and fails `__validate__` at block-execution time.

---

### Likelihood Explanation

**Likelihood: Medium.**

- The mempool is observable (transactions are propagated over P2P and visible via RPC).
- The attack window is the time between the victim's `deploy_account` entering the mempool and the victim's Invoke being submitted — typically a few seconds in a normal UX flow.
- No privileged access is required; any unprivileged network participant can submit the crafted Invoke.
- The attacker only needs to know the victim's contract address (derivable from the `deploy_account` fields) and submit an Invoke with nonce 1 and any signature bytes within the `max_signature_length` limit.

---

### Recommendation

Replace the weak presence check with a type-specific check. `account_tx_in_pool_or_recent_block` must be extended (or a new query added) to confirm that the transaction in the pool for the given address is specifically a `deploy_account` transaction. Alternatively, the gateway should query the pool for the exact transaction type at nonce 0 for the sender address before granting the skip.

A minimal fix: add a `deploy_account_in_pool(address)` query to the mempool that checks `tx_pool.get_by_address_and_nonce(address, Nonce::ZERO)` and verifies the transaction type is `DeployAccount`. Use this in `skip_stateful_validations` instead of the generic `account_tx_in_pool_or_recent_block`.

### Proof of Concept

**Setup:** Alice (victim) deploys a fresh account at address `0xALICE`. Bob (attacker) monitors the mempool.

1. Alice submits `deploy_account` (nonce 0, valid `__validate_deploy__` signature) → gateway runs full execution including `__validate_deploy__`, transaction admitted to mempool.

2. Bob submits `Invoke` with:
   - `sender_address = 0xALICE`
   - `nonce = 1`
   - `calldata = [<anything>]`
   - `signature = [0xDEAD, 0xBEEF]` (invalid)

3. Gateway stateless check: signature length 2 ≤ `max_signature_length` (4000) → passes.

4. `get_nonce_from_state(0xALICE)` → returns `Nonce(0)` (not yet deployed on-chain).

5. `validate_state_preconditions`: nonce check `0 ≤ 1 ≤ 200` → passes.

6. `validate_by_mempool`: Alice's `deploy_account` has nonce 0, Bob's Invoke has nonce 1 → no `DuplicateNonce`, no `NonceTooOld` → passes.

7. `skip_stateful_validations`: `tx.nonce() == 1 && account_nonce == 0` → true; `account_tx_in_pool_or_recent_block(0xALICE)` → `true` (Alice's `deploy_account` is in the pool) → returns `true`.

8. `run_validate_entry_point` called with `validate = false` → `__validate__` is **never called** → Bob's Invoke with invalid signature is admitted to the mempool.

9. Alice submits her legitimate Invoke (nonce 1, valid signature) → mempool returns `MempoolError::DuplicateNonce { address: 0xALICE, nonce: 1 }` → **rejected**.

10. Alice's `deploy_account + invoke` UX flow is broken for this block. The invalid Invoke will fail `__validate__` during block execution and be removed, but Alice must retry in the next block — and Bob can repeat the attack.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
