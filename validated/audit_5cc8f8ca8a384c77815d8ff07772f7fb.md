Audit Report

## Title
Third-Party Caller Can Bypass `confirmation_text` Compliance Gate on Behalf of Any Victim Who Has Pre-Funded Their Swap Subaccount - (File: `rs/sns/swap/canister/canister.rs`)

## Summary
The `refresh_buyer_tokens` update method accepts a caller-supplied `buyer` principal with no requirement that it match `msg_caller()`. When a victim has already transferred ICP to their swap subaccount, any unprivileged third party can call `refresh_buyer_tokens` with `buyer = victim_principal` and supply the correct `confirmation_text` themselves, satisfying the compliance gate on the victim's behalf. The victim is registered as a committed swap participant without ever having sent the confirmation text, directly violating the legal/compliance invariant the SNS operator intended to enforce.

## Finding Description

In `rs/sns/swap/canister/canister.rs` (L128–143), the `buyer` principal is resolved from caller-supplied input with no authorization check:

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    PrincipalId::from_str(&arg.buyer).unwrap()   // arbitrary caller-supplied value
};
```

This `p` is passed directly to `refresh_buyer_token_e8s` in `rs/sns/swap/src/swap.rs` (L1134–1163). Inside that function, `validate_confirmation_text` (L1150) checks only that the supplied text string matches the SNS-configured text — it performs no check that the caller equals the `buyer`:

```rust
pub fn validate_confirmation_text(&self, confirmation_text: Option<String>) -> Result<(), String> {
    match (self.init_or_panic().confirmation_text.as_ref(), confirmation_text) {
        (Some(expected_text), Some(text)) => {
            if &text != expected_text { Err(...) } else { Ok(()) }
        }
        ...
    }
}
```

The `confirmation_text` is a public value readable from the SNS init payload by any observer. After validation, the ICP balance is read from the victim's subaccount (L1154–1163) and the participation is recorded under the victim's key (L1285–1288):

```rust
self.buyers
    .entry(buyer.to_string())
    .or_insert_with(|| BuyerState::new(0))
    .set_amount_icp_e8s(new_balance_e8s);
```

The docstring at L1123–1126 explicitly states "the caller of this function must accept this confirmation," but the code allows the caller and the buyer to be different principals. The `RefreshBuyerTokensRequest` proto documents the `buyer` field as optional ("If not specified, the caller is used"), making this an intentional feature whose interaction with `confirmation_text` creates the bypass.

Existing guards are insufficient: the only check on `confirmation_text` is a string equality comparison against the SNS-configured text, with no binding to the identity of the caller or the buyer.

## Impact Explanation

The `confirmation_text` feature is documented as a legal/compliance gate: participants must explicitly send the matching text to be registered. Because the text is public and the `buyer` field is unconstrained, any attacker can satisfy the gate on behalf of any victim who has pre-funded their swap subaccount. The victim is registered as a committed participant without ever having sent the confirmation text, directly violating the compliance invariant. Once committed, the victim's ICP is locked until swap finalization; if the swap commits, the victim receives SNS tokens in exchange for ICP they did not consent to commit under the required terms. This constitutes a significant SNS security impact with concrete user harm, fitting the **High ($2,000–$10,000)** impact class: "Significant... SNS... security impact with concrete user or protocol harm."

## Likelihood Explanation

- `refresh_buyer_tokens` is a public, unpermissioned `#[update]` endpoint callable by any ingress sender at zero cost.
- The `confirmation_text` is a public string readable from the swap canister's state by any observer.
- The only prerequisite is that the victim has transferred ICP to their swap subaccount — a normal step in the ticket-based payment flow that occurs before `refresh_buyer_tokens` is called.
- No special privileges, key material, or victim interaction beyond the ICP transfer is required.
- The attack window exists between the victim's ICP transfer and their own call to `refresh_buyer_tokens`.

## Recommendation

Enforce that `buyer == caller` when `confirmation_text` validation is required. The simplest fix is to reject calls where `arg.buyer` differs from `caller_principal_id()` when the swap has a `confirmation_text` configured, or unconditionally:

```rust
async fn refresh_buyer_tokens(arg: RefreshBuyerTokensRequest) -> RefreshBuyerTokensResponse {
    let caller = caller_principal_id();
    let p: PrincipalId = if arg.buyer.is_empty() {
        caller
    } else {
        let supplied = PrincipalId::from_str(&arg.buyer).unwrap();
        // If a confirmation_text is required, only the buyer themselves may submit it.
        if swap().init_or_panic().confirmation_text.is_some() && supplied != caller {
            panic!("confirmation_text must be submitted by the buyer themselves");
        }
        supplied
    };
    ...
}
```

If third-party notification (bot calling on behalf of a user) must be preserved for swaps without `confirmation_text`, the guard can be scoped only to swaps where `confirmation_text` is set.

## Proof of Concept

1. Deploy an SNS swap with `confirmation_text = "I agree to the terms"`.
2. Alice transfers 10 ICP to `Account { owner: swap_canister, subaccount: principal_to_subaccount(alice) }` on the ICP ledger.
3. Alice has not yet called `refresh_buyer_tokens`.
4. Mallory (any principal) calls:
   ```
   refresh_buyer_tokens({
     buyer: alice_principal.to_text(),
     confirmation_text: Some("I agree to the terms")
   })
   ```
5. The swap canister resolves `p = alice_principal`, calls `validate_confirmation_text` with Mallory's supplied text (passes), reads Alice's subaccount balance (10 ICP), and records Alice as a committed participant.
6. Alice is now a registered participant. Her ICP is locked. She never sent the confirmation text.

A deterministic unit test can be written using the existing `SwapBuilder` + `mock_stub` pattern from `rs/sns/swap/tests/swap.rs` (see `test_swap_participation_confirmation` at L5637–5700), calling `refresh_buyer_tokens` with `buyer = alice_principal` from a different caller identity while supplying the correct `confirmation_text`, and asserting that Alice appears in `swap.buyers` despite never having called the function herself.
