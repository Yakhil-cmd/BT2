### Title
Permanent Fund Freeze via Oversized `gas_limit` in Legacy Transaction: `charge_gas` Deducts Before `try_into::<u64>()` Fails Without Refund — (`engine/src/engine.rs`)

---

### Summary

In `submit_with_alt_modexp`, `charge_gas` is called before `transaction.gas_limit` is narrowed to `u64`. If a legacy transaction carries a `gas_limit` encoded as a `U256` value exceeding `u64::MAX`, `charge_gas` successfully deducts `gas_limit * effective_gas_price` from the sender's balance using full U256 arithmetic, then the subsequent `try_into::<u64>()` returns `GasOverflow` and the function exits early — skipping `refund_unused_gas` entirely. The deducted ETH is permanently frozen with no recovery path.

---

### Finding Description

**Step 1 — RLP accepts oversized `gas_limit`.**

`TransactionLegacy.gas_limit` is typed as `U256` and decoded from RLP with no upper-bound check:

```rust
// engine-transactions/src/legacy.rs, line 121
let gas = rlp.val_at(2)?;   // decoded as U256, no u64 cap
``` [1](#0-0) 

`NormalizedEthTransaction::try_from` for the `Legacy` arm copies it verbatim:

```rust
// engine-transactions/src/lib.rs, line 110
gas_limit: tx.transaction.gas_limit,
``` [2](#0-1) 

**Step 2 — Pre-charge validation does not reject oversized `gas_limit`.**

The only gas-related guard before `charge_gas` is the intrinsic-gas check:

```rust
// engine/src/engine.rs, line 1079
if transaction.gas_limit < core::cmp::max(intrinsic_gas, floor_gas).into() {
    return Err(EngineErrorKind::IntrinsicGasNotMet.into());
}
``` [3](#0-2) 

`intrinsic_gas` is a `u64`; `.into()` widens it to `U256`. A `gas_limit` of `u64::MAX + 1` is larger than any `u64` intrinsic gas value, so this check passes.

**Step 3 — `charge_gas` deducts using U256 arithmetic.**

```rust
// engine/src/engine.rs, lines 496-500
let prepaid_amount = fixed_gas
    .map_or(transaction.gas_limit, EthGas::as_u256)  // U256 gas_limit used directly
    .checked_mul(effective_gas_price)
    .map(Wei::new)
    .ok_or(GasPaymentError::EthAmountOverflow)?;
``` [4](#0-3) 

With `gas_limit = u64::MAX + 1` and `gas_price = 1`, `prepaid_amount = u64::MAX + 1` wei. `checked_mul` succeeds (no U256 overflow). The sender's balance is reduced by `u64::MAX + 1` wei and the state is written:

```rust
set_balance(&mut self.io, sender, &new_balance);
``` [5](#0-4) 

**Step 4 — `try_into::<u64>()` fails and returns early, skipping the refund.**

```rust
// engine/src/engine.rs, lines 1107-1110
let gas_limit = transaction
    .gas_limit
    .try_into()
    .map_err(|_| EngineErrorKind::GasOverflow)?;  // early return here
``` [6](#0-5) 

The `?` operator causes `submit_with_alt_modexp` to return `Err(GasOverflow)` immediately. The `refund_unused_gas` call at lines 1148–1155 is never reached:

```rust
// engine/src/engine.rs, lines 1148-1155
refund_unused_gas(
    &mut io,
    &sender,
    gas_used,
    &prepaid_amount,
    &relayer_address,
    fixed_gas,
)
``` [7](#0-6) 

The balance reduction committed by `charge_gas` is never reversed. There is no other recovery mechanism.

---

### Impact Explanation

The sender's ETH equal to `gas_limit * gas_price` (up to ~1.8 × 10¹⁹ wei with `gas_limit = u64::MAX + 1`, `gas_price = 1`) is permanently removed from their account and not credited anywhere — it is simply destroyed from the engine's accounting perspective. This constitutes **permanent freezing of funds** (Critical).

---

### Likelihood Explanation

Any externally-owned account can craft and submit such a transaction. No special privilege is required. The attacker only needs:
- A valid ECDSA signature over a legacy transaction with `gas_limit` encoded as a 9-byte (or larger) RLP big integer (e.g., `u64::MAX + 1 = 0x10000000000000000`)
- Sufficient ETH balance to pass the `charge_gas` balance check

This is trivially constructable with any standard Ethereum transaction-signing library that accepts `U256` gas limits.

---

### Recommendation

Insert a `u64` range check on `gas_limit` **before** calling `charge_gas`, so an oversized value is rejected without touching the sender's balance:

```rust
let gas_limit: u64 = transaction
    .gas_limit
    .try_into()
    .map_err(|_| EngineErrorKind::GasOverflow.into())?;
// ... then pass gas_limit (u64) into charge_gas
```

Alternatively, restructure `charge_gas` to accept `u64` gas directly, making it impossible to charge for a gas amount that cannot later be converted.

### Proof of Concept

```
1. Craft a legacy EVM transaction:
   - gas_limit  = u64::MAX + 1  (= 0x10000000000000000, 9 bytes, valid U256 RLP)
   - gas_price  = 1 wei
   - to         = any address
   - value      = 0
   - data       = empty

2. Fund sender with balance = u64::MAX + 2 wei
   (enough to pass charge_gas's OutOfFund check)

3. Submit via the Aurora Engine `submit` entrypoint.

Expected (vulnerable) outcome:
   - charge_gas deducts u64::MAX + 1 wei from sender → sender balance = 1 wei
   - gas_limit.try_into::<u64>() fails → GasOverflow returned
   - refund_unused_gas is never called
   - sender balance remains at 1 wei permanently (u64::MAX + 1 wei frozen)

Assert:
   - Return value is Err(GasOverflow)
   - sender_balance_after == 1 wei  (not u64::MAX + 2)
   - No refund event / balance restoration occurs
```
