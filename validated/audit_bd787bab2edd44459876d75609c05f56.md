### Title
`skip_stateful_validations` Admits Unsigned Invoke Transactions via Overly Broad Pool Membership Check — (File: crates/apollo_gateway/src/stateful_transaction_validator.rs)

### Summary

The `skip_stateful_validations` function is intended to skip `__validate__` only when a matching `deploy_account` transaction is in the mempool. However, the guard it uses — `account_tx_in_pool_or_recent_block` — returns `true` for **any** transaction type in the pool, including a previously-admitted (also unvalidated) invoke. An attacker who observes a `deploy_account + invoke(nonce=1)` pair in the mempool can submit a replacement invoke with a higher tip and an invalid signature; the gateway will admit it without calling `__validate__`, silently replacing the legitimate transaction.

### Finding Description

`skip_stateful_validations` fires when:
1. The incoming transaction is `Invoke` with `nonce == 1`
2. The on-chain account nonce is `0` (account not yet deployed)
3. `account_tx_in_pool_or_recent_block(sender)` returns `true` [1](#0-0) 

The inline comment claims this is safe because the pool entry "means that either it has a deploy_account transaction or transactions with future nonces that passed validations." That claim is false: the first `invoke(nonce=1)` submitted alongside the `deploy_account` was itself admitted with `skip_validate=true`, so it never passed `__validate__`. Its presence in the pool therefore proves nothing about the existence of a `deploy_account`. [2](#0-1) 

The mempool's `account_tx_in_pool_or_recent_block` checks `tx_pool.contains_account` (any transaction type) and `state.contains_account` (any committed address). Neither distinguishes `deploy_account` from `invoke`. [3](#0-2) 

`run_pre_validation_checks` calls `validate_by_mempool` (which allows fee escalation) before calling `skip_stateful_validations`. A higher-tip replacement therefore passes the mempool's duplicate-nonce check and then triggers the skip. [4](#0-3) 

When `execution_flags.validate == false`, `perform_validations` returns `Ok(())` immediately after `perform_pre_validation_stage`, never calling `validate_tx` / `__validate__`.

**Attack path:**

1. Victim submits `deploy_account(A) + invoke(sender=A, nonce=1, tip=T, sig=valid)`.
2. Gateway admits `invoke(nonce=1)` with `skip_validate=true` (deploy_account is in pool → `account_tx_in_pool_or_recent_block` = true).
3. Attacker submits `invoke(sender=A, nonce=1, tip=T+1, calldata=malicious, sig=garbage)`.
4. `validate_by_mempool` passes (fee escalation allowed).
5. `skip_stateful_validations` returns `true` — the old `invoke(nonce=1)` is still in the pool.
6. `run_validate_entry_point(skip_validate=true)` skips `__validate__`; the malicious transaction is admitted and replaces the victim's transaction. [5](#0-4) 

### Impact Explanation

The gateway/mempool admits an invoke transaction whose signature was never verified. The victim's legitimate `invoke(nonce=1)` is evicted from the pool. When the block is built, the malicious transaction is executed: the blockifier will call `__validate__` at that point, so a standard account contract will revert it — but the victim's transaction is permanently lost from the pool and the account's first post-deploy action is denied. For account contracts whose `__validate__` does not enforce signature checks (e.g., custom or experimental accounts), the malicious calldata executes without authorization.

Matches: **High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

### Likelihood Explanation

The attack window is the time between the victim broadcasting the `deploy_account + invoke` pair and the `deploy_account` being committed. Mempool contents are observable via the RPC. The victim's account address is deterministic and computable before deployment. No privileged access is required; any unprivileged observer can mount the attack.

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a check that is specific to `deploy_account` transactions. The mempool should expose a dedicated query such as `deploy_account_tx_in_pool(address) -> bool`, or the existing function should be renamed and its semantics narrowed to only return `true` when a `deploy_account` (or a committed deployment) is present. The comment at line 441–443 of `stateful_transaction_validator.rs` should be updated to reflect the corrected invariant. [6](#0-5) 

### Proof of Concept

```

POST /gateway/add_transaction  { deploy_account(class=X, salt=S) }   # → address A, nonce 0→1

# Gateway: account_tx_in_pool_or_recent_block(A) = true (deploy_account in pool)
# → skip_validate=true, invoke admitted without __validate__

# 2. Attacker replaces the invoke with a higher-tip, invalid-signature version
POST /gateway/add_transaction  { invoke(sender=A, nonce=1, tip=11, calldata=DRAIN_FUNDS, sig=0x0) }
# validate_by_mempool: fee escalation passes (tip 11 > 10)
# skip_stateful_validations: account_tx_in_pool_or_recent_block(A) = true (old invoke still in pool)
# → skip_validate=true, malicious invoke admitted, replaces victim's invoke

# 3. Batcher includes deploy_account then malicious invoke in the same block
# blockifier calls __validate__ on A; standard accounts revert (bad sig)
# custom accounts without sig check: DRAIN_FUNDS executes
```

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
