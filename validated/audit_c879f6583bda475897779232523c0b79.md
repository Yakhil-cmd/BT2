### Title
Reliable Broadcast Accepts Single Ready Vote Without Echo Phase for n≤3, Enabling Permanent DKG Denial — (`src/protocol/echo_broadcast.rs`)

### Summary
`echo_ready_thresholds(n)` returns `(0, 0)` for `n ≤ 3`. With `ready_t = 0`, a single `Ready(bad_data)` message from a malicious participant M triggers amplification and immediately satisfies the `> 2 * ready_t` acceptance condition (2 > 0), causing honest parties to accept M's injected data as M's broadcast output without any Send or Echo phase. This violates the fundamental invariant of reliable broadcast.

### Finding Description
In `reliable_broadcast_receive_all` (`src/protocol/echo_broadcast.rs`), the Ready handler has no guard requiring that a Send message was previously received for the same session. For `n=3`, `ready_t=0`, so:

1. M sends `Ready(bad_data)` for M's own session (skipping Send/Echo).
2. Honest party H receives it: count = 1 > 0 → amplifies, simulates own Ready → count = 2 > 0 → **accepts `bad_data`**.
3. The second honest party H2 similarly accepts `bad_data` after receiving H's amplified Ready.

Both honest parties agree on `bad_data` as M's broadcast output. In DKG, this fake commitment fails `verify_commitment_hash` (the hash committed in round 1 doesn't match), causing `ProtocolError::InvalidCommitmentHash` and aborting DKG for all honest parties.

### Impact Explanation
**High: Permanent denial of DKG (and reshare/refresh) for honest parties.** Any single malicious participant in a 3-party setup can abort every DKG attempt by injecting a `Ready` message with arbitrary data, bypassing the echo phase entirely. Since M can repeat this on every retry, honest parties can never complete key generation.

### Likelihood Explanation
n=3 with threshold=2 is a common, minimal threshold configuration. The library explicitly supports it (`n >= 2` minimum). A malicious participant needs only to send one crafted message type at the right waitpoint — no cryptographic capability required.

### Recommendation
Add a guard in the Ready handler that rejects Ready messages for a session where no Send has been received (i.e., `finish_send` is false for that session). Alternatively, enforce `n >= 4` at the API level when malicious participants are possible, or raise the minimum thresholds so that `ready_t >= 1` always holds (e.g., use `ready_t = max(1, (n-1)/3)`).

### Proof of Concept
3-party test: M sends only `Ready(bad_data)` for M's session (no Send/Echo). Assert that honest parties do NOT accept `bad_data` as M's broadcast output. Under the current code, both honest parties will accept `bad_data` and subsequently abort DKG with `InvalidCommitmentHash`, demonstrating the denial-of-service. [8](#0-7) [9](#0-8)

fn echo_ready_thresholds(n: usize) -> (usize, usize) {
    // case where no malicious parties are assumed: when n <= 3/
    // In this case the echo and ready thresholds are both 0
    // later we compare if we have collected more votes than these thresholds
    if n <= 3 {
        return (0, 0);
    }
    // we should always have n >= 3*threshold + 1
    let broadcast_threshold = (n - 1) / 3;
    let echo_threshold = usize::midpoint(n, broadcast_threshold);
    (echo_threshold, broadcast_threshold)
}
