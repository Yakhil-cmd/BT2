### Title
Missing `ThresholdTooSmall` Validation in FROST Signing Allows Sub-Threshold Protocol Execution — (File: `src/frost/mod.rs`)

---

### Summary

The `assert_sign_inputs` and `presign` functions in `src/frost/mod.rs` are missing the lower-bound threshold check (`threshold < 2`) that is consistently enforced in `assert_key_invariants` in `src/dkg.rs`. A caller can pass `threshold = 1` (or `0`) to the FROST signing path without rejection, violating the invariant that a threshold must be at least 2, and causing the signing protocol to execute with incorrect Lagrange interpolation parameters, producing an unusable signature output.

---

### Finding Description

`assert_key_invariants` in `src/dkg.rs` enforces both bounds on the threshold:

```rust
if threshold > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
if threshold < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
}
``` [1](#0-0) 

`assert_reshare_keys_invariants` delegates to `assert_key_invariants` and therefore also enforces the lower bound. [2](#0-1) 

By contrast, `assert_sign_inputs` in `src/frost/mod.rs` only checks the upper bound:

```rust
// validate threshold
if threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
// ← no ThresholdTooSmall check here
``` [3](#0-2) 

The same omission exists in `presign`:

```rust
if args.threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
// ← no ThresholdTooSmall check here
``` [4](#0-3) 

`ReconstructionLowerBound` is a plain `usize` newtype with no minimum enforced at the type level, so any value including `0` or `1` can be constructed and passed by a caller. [5](#0-4) 

For comparison, `validate_triple_inputs` in the OT-based ECDSA path does enforce the lower bound correctly:

```rust
if threshold_value < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold: threshold_value, min: 2 });
}
``` [6](#0-5) 

This makes the missing check in the FROST path a clear inconsistency across the codebase.

---

### Impact Explanation

In FROST, the threshold governs Lagrange interpolation during partial-signature aggregation. If `threshold = 1` is accepted:

- The coordinator collects only one partial signature.
- The Lagrange coefficient for a single point evaluates to `1`, so the aggregated `z` equals `nonce_i + key_share_i * challenge` for that one participant.
- The correct aggregated `z` requires `nonce + secret_key * challenge`, where `secret_key = Σ λ_j * key_share_j` over all signing participants.
- Because `key_share_i ≠ secret_key` (the key was generated with threshold ≥ 2 during DKG), the resulting `(R, z)` pair fails standard EdDSA/RedJubjub verification.

Honest parties executing the protocol under these parameters receive a structurally well-formed but cryptographically invalid signature — an unusable output. This matches the **High** allowed impact: *Corruption of sign outputs so honest parties accept unusable cryptographic outputs*.

---

### Likelihood Explanation

The entry path is direct and requires no special privilege. Any caller of `assert_sign_inputs` or `presign` — including a malicious coordinator orchestrating a signing session — can supply `ReconstructionLowerBound(1)`. The `PresignArguments` struct exposes `threshold` as a plain public field, so no internal state needs to be subverted. [7](#0-6) 

---

### Recommendation

Add the lower-bound check to both `assert_sign_inputs` and `presign` in `src/frost/mod.rs`, mirroring the check in `assert_key_invariants`:

```rust
// In assert_sign_inputs and presign:
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

This should be placed immediately after the upper-bound check in both functions.

---

### Proof of Concept

1. Run DKG with `threshold = 2` and 3 participants, producing valid `KeygenOutput` shares.
2. Call `assert_sign_inputs(participants, ReconstructionLowerBound(1), me, coordinator)`.
3. Observe: the function returns `Ok(...)` — no error is raised, despite `threshold = 1` violating the protocol invariant.
4. Proceed to FROST signing with `threshold = 1`; the coordinator collects one partial signature.
5. The aggregated signature `(R, z)` fails verification against the master public key because the Lagrange interpolation is computed over a single share rather than the full threshold set.
6. Contrast: calling `assert_key_invariants(participants, me, ReconstructionLowerBound(1))` correctly returns `Err(InitializationError::ThresholdTooSmall { threshold: 1, min: 2 })`. [8](#0-7) [3](#0-2)
