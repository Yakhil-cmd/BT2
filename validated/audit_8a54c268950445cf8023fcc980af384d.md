### Title
`compute_gas_refund()` Unchecked Subtraction at Line 83 Panics When `delta_gas` Adjustment Causes `gas_used > gas_limit` — (File: `basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`)

---

### Summary

In `compute_gas_refund()`, after the native-resource `delta_gas` adjustment is applied to `gas_used`, the final line `let total_gas_refund = gas_limit - gas_used;` is an **unchecked subtraction**. If `gas_used` has been pushed above `gas_limit` by the `delta_gas` addition, this subtraction wraps/panics in debug mode and produces a wildly incorrect (wrapped) refund value in release mode, causing the user to receive a massive over-refund of native tokens.

---

### Finding Description

In `compute_gas_refund()`:

```rust
// Step 1: gas_used starts as gas consumed by EE
let mut gas_used = gas_limit
    .checked_sub(resources.ergs().0.div_floor(ERGS_PER_GAS))
    .ok_or(internal_error!("gas remaining > gas limit"))?;

// Step 2: EVM refund counter subtracted (safe, capped to gas_used/5)
gas_used -= evm_refund;

// Step 3: Floored to minimal_gas_used
let mut gas_used = core::cmp::max(gas_used, minimal_gas_used);

// Step 4: delta_gas from native resource consumption added
if delta_gas > 0 {
    gas_used += delta_gas as u64;   // <-- can push gas_used > gas_limit
}

// Step 5: UNCHECKED subtraction — panics or wraps if gas_used > gas_limit
let total_gas_refund = gas_limit - gas_used;   // <-- BUG
```

The invariant `gas_used <= gas_limit` is **not re-checked** after the `delta_gas` addition at line 78. The `delta_gas` is computed as:

```rust
let delta_gas = (native_used / native_per_gas) as i64 - (gas_used as i64);
```

`native_used` is derived from `full_native_limit.saturating_sub(resources.native().remaining())`. If the native resource was consumed heavily (e.g., by pubdata or proving-heavy computation), `native_used / native_per_gas` can exceed `gas_limit`, making `delta_gas` large enough that `gas_used + delta_gas > gas_limit`.

The only guard present is:

```rust
require_internal!(
    total_gas_refund <= gas_limit,
    "Gas refund greater than gas limit",
    system
)?;
```

But this check is **after** the subtraction, so the panic/wrap has already occurred.

The analogous pattern to the BendDAO bug: a value that should be bounded (here `gas_used`) is not zeroed/clamped before being used in a subtraction that assumes it is bounded, leading to an incorrect (over-)refund to the user.

---

### Impact Explanation

**Resource accounting bug / over-refund of native tokens to the transaction sender.**

- In **debug/proving mode** (RISC-V target): the subtraction panics, halting block processing entirely — a denial-of-service against the prover/sequencer.
- In **release mode** (forward/sequencer mode): `gas_limit - gas_used` wraps to a very large `u64`, making `total_gas_refund` enormous. This value is then used to compute the token refund to the sender:
  - For L2 transactions (`ZkTransactionFlow`): `token_to_refund = gas_price * (tx_gas_limit - gas_used)` — the sender receives a massive over-refund of native tokens, draining the protocol's balance.
  - For L1 transactions: `to_refund_recipient = prepaid_fee.checked_sub(pay_to_operator)` — the checked subtraction there would catch the overflow, but the `gas_used` value itself is corrupted, causing incorrect operator payment.

The `require_internal!` check at line 85–89 fires **after** the subtraction, so it cannot prevent the wrap.

---

### Likelihood Explanation

**Medium.** The condition requires `native_used / native_per_gas > gas_limit` after the `max(gas_used, minimal_gas_used)` floor. This is reachable when:

1. A transaction uses very little EVM gas (e.g., a simple transfer that hits `minimal_gas_used` floor), but
2. The native resource consumption is high (e.g., heavy pubdata, or a high `native_per_gas` ratio from a high gas price relative to `native_price`).

The `delta_gas` formula `(native_used / native_per_gas) - gas_used` can be positive and large when `native_per_gas` is small (low gas price) but native consumption is high. An unprivileged L2 transaction sender can craft a transaction that maximizes pubdata writes (storage writes) to inflate `native_used`, while keeping EVM gas consumption low, to trigger this condition.

---

### Recommendation

Add a clamp after the `delta_gas` addition to ensure `gas_used` never exceeds `gas_limit`:

```rust
if delta_gas > 0 {
    gas_used += delta_gas as u64;
    // Clamp to gas_limit to prevent underflow in the subtraction below
    gas_used = gas_used.min(gas_limit);
}

let total_gas_refund = gas_limit - gas_used; // now safe
```

Alternatively, use `checked_sub` with a proper error:

```rust
let total_gas_refund = gas_limit.checked_sub(gas_used)
    .ok_or(internal_error!("gas_used exceeds gas_limit after delta_gas adjustment"))?;
```

### Proof of Concept

**Setup:**
- `gas_limit = 100_000`
- `native_per_gas = 1` (low gas price, e.g., `gas_price = native_price`)
- Transaction does minimal EVM work: `gas_used_from_ergs = 21_000` (intrinsic only)
- `minimal_gas_used = 21_000`
- After `max`: `gas_used = 21_000`
- Transaction writes many storage slots, consuming `native_used = 200_000` native units
- `delta_gas = (200_000 / 1) - 21_000 = 179_000`
- `gas_used += 179_000` → `gas_used = 200_000`
- `total_gas_refund = 100_000 - 200_000` → **underflow/panic**

In release mode this wraps to `u64::MAX - 99_999`, and the refund sent to the user is `gas_price * (u64::MAX - 99_999)` — effectively draining the protocol.
