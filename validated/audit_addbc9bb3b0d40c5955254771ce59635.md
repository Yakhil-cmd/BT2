### Title
WalletConnect `chia_wallet.create_offer_for_ids` Spoofs NFT as CAT in Confirmation Dialog via Asymmetric Asset Kind Resolution - (File: `packages/gui/src/electron/commands/parseCommandDisplay.ts`)

### Summary
`createOfferAssetKinds` enforces a strict driver-dict requirement for bytes32 asset IDs on the **spending** side but silently falls back to `'cat'` on the **receiving** side when no driver is supplied. A malicious WalletConnect dapp can exploit this asymmetry to make an NFT launcher ID appear as a CAT in the user's confirmation dialog, causing the user to approve an offer with the wrong asset type.

### Finding Description
In `createOfferAssetKinds`:

```
// spending side — throws if no driver
assetKinds.spending[assetId] = assetKindForDriverDictAssetId(assetId, driverDict);
// → throws 'Asset type is not valid' when driverDict has no entry

// receiving side — silently defaults to 'cat'
assetKinds.receiving[assetId] = assetKindForDriverDictAssetId(assetId, driverDict) ?? 'cat';
``` [1](#0-0) 

When `assetKind` resolves to `'cat'`, `parseWalletDeltaItem` calls `mojoToCATLocaleString` and labels the asset as a CAT (or "Unknown CAT" if `catAssetIdToName` returns no name): [2](#0-1) 

The `driver_dict` parameter is optional in the WalletConnect schema and defaults to `{}` when omitted: [3](#0-2) 

A malicious dapp sends `chia_createOfferForIds` with an NFT launcher ID (64-char hex) on the receiving side and no `driver_dict`. Because `assetKindForDriverDictAssetId` returns `undefined` for an empty dict, the fallback `?? 'cat'` fires. The confirmation dialog then renders the NFT as `mojoToCATLocaleString(1n)` = "0.001 Unknown CAT" instead of an NFT.

The test suite explicitly documents this gap: [4](#0-3) 

### Impact Explanation
The user's confirmation dialog shows, for example, "Spend 1 XCH → Receive 0.001 Unknown CAT" while the actual signed offer encodes "Spend 1 XCH → Receive NFT `<launcher_id>`". If the user approves and a counterparty takes the offer, the user loses 1 XCH and receives an NFT they never intended to acquire. This is a **High** impact: spoofing of WalletConnect offer state causes the user to approve the wrong asset type, resulting in direct asset loss.

### Likelihood Explanation
The attacker only needs an active WalletConnect session (standard dapp pairing). No leaked keys, cryptographic breaks, or host compromise are required. The dapp simply omits `driver_dict` and places any NFT launcher ID on the receiving side of the offer map. The asymmetric fallback fires automatically.

### Recommendation
Remove the `?? 'cat'` fallback on the receiving side of `createOfferAssetKinds`. Require an explicit driver dict entry for all bytes32 asset IDs on both sides, mirroring the spending-side behavior. If backward compatibility with the legacy CAT fallback is needed, at minimum surface a prominent warning in the confirmation dialog when the asset type cannot be determined from the driver dict.

### Proof of Concept
A malicious WalletConnect dapp sends:
```json
{
  "method": "chia_createOfferForIds",
  "params": {
    "offer": {
      "1": "-1000000000000",
      "6b6b2a3b4c57c2b4596625583cbede95d081b59d18125fedb6b416a8ee46cfe5": "1"
    }
  }
}
```
No `driver_dict` is provided. `createOfferAssetKinds` resolves the NFT launcher ID on the receiving side as `'cat'` via the `?? 'cat'` fallback. `parseWalletDeltaItem` calls `mojoToCATLocaleString(1n)` and `catAssetIdToName(...)`, producing a confirmation row of "0.001 Unknown CAT". The user approves. The actual offer encodes the NFT singleton as the received asset. A counterparty takes the offer; the user loses 1 XCH and receives the NFT.
