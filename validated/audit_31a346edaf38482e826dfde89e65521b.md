### Title
Unbounded EIP-7702 Authorization List Iteration Exhausts Native Resources During Transaction Validation - (File: `basic_bootloader/src/bootloader/transaction/authorization_list.rs`)

### Summary

ZKsync OS processes EIP-7702 transactions by iterating over every entry in the authorization list and performing expensive per-entry operations (keccak256, ecrecover, account read, nonce write, delegation write). There is no hard cap on the number of authorization list entries a transaction may carry. An attacker can craft a transaction with a very large authorization list whose intrinsic native cost exceeds `MAX_NATIVE_COMPUTATIONAL`, causing the transaction to be rejected as `OutOfNativeResources` during validation — permanently preventing that transaction from being included in any block.

### Finding Description

`parse_authorization_list_and_apply_delegations` in `basic_bootloader/src/bootloader/transaction/authorization_list.rs` iterates over every entry in the authorization list:

```rust
for entry in auth_list.iter() {
    let success = validate_and_apply_delegation(
        system, resources, &chain_id, nonce, address,
        (y_parity, r, s), &mut hasher,
    )?;
}
``` [1](#0-0) 

For each entry, `validate_and_apply_delegation` charges `PER_AUTH_NATIVE_COMPUTATIONAL_OVERHEAD` (2,000 native units) upfront, then performs keccak256, ecrecover, an account read, a nonce write, and a delegation write — all charged against the transaction's native resource budget: [2](#0-1) 

The per-authorization worst-case native cost is `L2_TX_INTRINSIC_COMPUTATIONAL_NATIVE_PER_AUTHORIZATION`, which includes `ECRECOVER_NATIVE_COST` (≈522,000 native units), keccak, account read, nonce update, and delegation write: [3](#0-2) 

`ECRECOVER_NATIVE_COST` alone is `native_with_delegations!(350_000, 43_000, 0)` = 522,000 native units: [4](#0-3) 

`MAX_NATIVE_COMPUTATIONAL` is `1 << 35` = 34,359,738,368 native units: [5](#0-4) 

The intrinsic native formula pre-charges `authorization_list_num * L2_TX_INTRINSIC_COMPUTATIONAL_NATIVE_PER_AUTHORIZATION` before execution begins: [6](#0-5) 

This pre-charge is subtracted from the native limit in `create_resources_for_tx`. If the result underflows (i.e., the intrinsic native cost exceeds the native limit derived from `gas_limit * native_per_gas`), the transaction is rejected with a validation error: [7](#0-6) 

The `HomList` parser for `AuthorizationList` has no enforced upper bound on the number of entries — it accepts any count that fits within the RLP-encoded transaction body: [8](#0-7) [9](#0-8) 

The only existing guard is the transaction byte-length limit `MAX_TX_LEN_BYTES = 1 << 23` (8 MB): [10](#0-9) 

Each authorization entry is at minimum ~70 bytes (chain_id, address, nonce, y_parity, r, s in RLP). An 8 MB transaction can therefore carry up to ~114,000 authorization entries. At `L2_TX_INTRINSIC_COMPUTATIONAL_NATIVE_PER_AUTHORIZATION` ≈ 700,000+ native units per entry, even a few hundred entries can push the intrinsic native cost above `MAX_NATIVE_COMPUTATIONAL` (2^35 ≈ 34 billion), making the transaction permanently unprocessable regardless of gas limit or gas price.

The critical asymmetry: the intrinsic native formula is charged **before** execution, and the native limit is capped at `MAX_NATIVE_COMPUTATIONAL` before the intrinsic subtraction: [11](#0-10) 

This means a transaction with enough authorization entries will always fail validation — it cannot be made valid by increasing gas price or gas limit.

### Impact Explanation

**Severity: Medium — Permanent transaction unprocessability / resource exhaustion DoS.**

An attacker can craft an EIP-7702 transaction with a large authorization list (e.g., 50,000+ entries, well within the 8 MB transaction size limit) such that the intrinsic native cost exceeds `MAX_NATIVE_COMPUTATIONAL`. The transaction will be permanently rejected during validation with `OutOfNativeResources`. While this does not directly harm other users' transactions, it:

1. Wastes sequencer resources during parsing and validation of the oversized authorization list (the `HomList` with `VALIDATE=true` iterates and decodes all entries during parsing).
2. Prevents the sender from ever getting this transaction included — a self-inflicted DoS, but also a vector for griefing if the attacker can trick a user into signing such a transaction.
3. More critically: the `HomList::decode_list_from` with `VALIDATE=true` iterates all entries at parse time to count them, consuming CPU proportional to the list size before any resource check occurs. [12](#0-11) 

This unbounded parse-time iteration is the direct analog of the staking contract's unbounded reward-calculation loop.

### Likelihood Explanation

**Medium.** EIP-7702 is a newly supported transaction type. Any user or attacker can submit an EIP-7702 transaction with an arbitrarily large authorization list up to the 8 MB transaction size limit. No privileged access is required. The sequencer must parse and validate every submitted transaction, so this is reachable via the standard RPC submission path.

### Recommendation

1. **Enforce a hard cap on authorization list length** during RLP decoding or immediately after, analogous to how `MAX_TX_LEN_BYTES` caps total transaction size. A reasonable cap (e.g., 255 entries, matching Ethereum's practical limit) should be enforced in `EIP7702Tx::decode_list_body` or in `parse_authorization_list_and_apply_delegations`.

2. **Reject oversized authorization lists early** — before the `HomList` validation loop iterates all entries — to prevent unbounded parse-time CPU consumption.

3. **Verify that `L2_TX_INTRINSIC_COMPUTATIONAL_NATIVE_PER_AUTHORIZATION * MAX_AUTH_LIST_SIZE < MAX_NATIVE_COMPUTATIONAL`** holds as a compile-time assertion.

### Proof of Concept

1. Compute the maximum authorization list entries that fit in `MAX_TX_LEN_BYTES` (8 MB): each entry is ~70 bytes minimum → ~114,000 entries.
2. Compute intrinsic native cost: `L2_TX_INTRINSIC_COMPUTATIONAL_NATIVE_PER_AUTHORIZATION` ≈ 700,000+ native units (ecrecover alone is 522,000). For 50,000 entries: 50,000 × 700,000 = 35,000,000,000 > `MAX_NATIVE_COMPUTATIONAL` (34,359,738,368).
3. Craft an EIP-7702 transaction with 50,000 authorization entries (all with valid RLP structure but arbitrary/invalid signatures — they will still be parsed and counted).
4. Submit via RPC. The bootloader will:
   - Parse the `HomList` with `VALIDATE=true`, iterating all 50,000 entries at decode time.
   - Compute `intrinsic_computational_native = 50,000 × L2_TX_INTRINSIC_COMPUTATIONAL_NATIVE_PER_AUTHORIZATION > MAX_NATIVE_COMPUTATIONAL`.
   - In `create_resources_for_tx`, after capping `native_limit` at `MAX_NATIVE_COMPUTATIONAL`, subtract `intrinsic_computational_native` → underflow → `checked_sub` returns `None` → transaction rejected as `InvalidTransaction::NativeResourcesAreTooExpensive` (or equivalent validation error).
5. The transaction is permanently unprocessable regardless of gas limit or gas price. [12](#0-11) [11](#0-10) [5](#0-4)

impl<'a, T: RlpItemDecode<'a>, const VALIDATE: bool> HomList<'a, T, VALIDATE> {
    pub fn decode_list_from(r: &mut Rlp<'a>) -> Result<Self, InvalidTransaction> {
        let inner = r.list()?;
        let all = inner.remaining();

        let count = if VALIDATE {
            let mut chk = Rlp::new(all);
            let mut c = 0;
            while !chk.is_empty() {
                T::decode_from_item(&mut chk)?;
                c += 1;
            }
            Some(c)
        } else {
            None
        };

        Ok(Self {
            payload: all,
            count,
            _marker: PhantomData,
        })
    }
