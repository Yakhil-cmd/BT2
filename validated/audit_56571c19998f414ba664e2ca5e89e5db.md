### Title
Unbounded `pool_member_epoch_balance` Trace Growth Causes Permanent Freezing of Pool Member Unclaimed Yield - (File: `src/pool/pool.cairo`)

---

### Summary
A pool member (unprivileged delegator) can grow their per-member `pool_member_epoch_balance` trace without bound by calling `add_to_delegation_pool` once per epoch while never claiming rewards. The `calculate_rewards` function iterates over this trace in an unbounded loop. Once the trace is large enough, `claim_rewards` will exceed Starknet's block execution resource limit, permanently freezing the pool member's accumulated unclaimed yield inside the pool contract.

---

### Finding Description

`calculate_rewards` in `src/pool/pool.cairo` contains an explicit developer acknowledgment of the problem:

```
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    ...
    entry_to_claim_from += 1;
}
```

The loop iterates from `pool_member_info.entry_to_claim_from` up to `pool_member_trace_length`, processing every balance-change checkpoint recorded since the last `claim_rewards` call.

**How the trace grows:**

`set_member_balance` (called by both `add_to_delegation_pool` and `exit_delegation_pool_intent`) calls:

```cairo
trace.insert(key: self.get_epoch_plus_k(), value: pool_member_balance);
```

The `insert` function in `src/pool/pool_member_balance_trace/trace.cairo` appends a **new** checkpoint whenever the key (`current_epoch + K`) differs from the last stored key. Since `current_epoch` advances by 1 each epoch, every call to `add_to_delegation_pool` in a new epoch appends one new entry to the trace.

`add_to_delegation_pool` has no check for an in-progress undelegate intent and no rate-limiting mechanism. A pool member can call it once per epoch with a minimal token amount (1 unit), growing the trace by exactly 1 entry per epoch indefinitely.

**Why `entry_to_claim_from` does not protect against this:**

`entry_to_claim_from` is only advanced inside `claim_rewards` (the state-changing function). If the pool member never calls `claim_rewards`, `entry_to_claim_from` stays at its initial value while the trace length grows without bound. After N epochs of balance changes without claiming, the loop must iterate N entries.

**Both `claim_rewards` and `pool_member_info_v1` call `calculate_rewards`:**

- `claim_rewards` (state-changing, lines 349–355): iterates the trace and updates `entry_to_claim_from`.
- `pool_member_info_v1` (view, lines 532–538): iterates the trace but does **not** update `entry_to_claim_from`, so repeated view calls provide no relief.

Once the trace is large enough that the loop exceeds Starknet's per-transaction execution resource cap, `claim_rewards` reverts on every call, and the pool member's accumulated yield is permanently locked in the pool contract.

---

### Impact Explanation

**Impact: High — Permanent freezing of unclaimed yield.**

A pool member who has accumulated rewards over many epochs cannot retrieve them once their `pool_member_epoch_balance` trace grows beyond the point where `calculate_rewards` can complete within a single transaction's resource budget. The STRK rewards already transferred to the pool contract by `update_rewards_from_staking_contract` remain permanently inaccessible to that pool member.

---

### Likelihood Explanation

**Likelihood: Low.**

The attack requires the pool member to actively call `add_to_delegation_pool` once per epoch over a sustained period (hundreds to thousands of epochs) while deliberately withholding `claim_rewards` calls. Each call requires holding and spending a small token amount plus transaction fees. There is no direct profit motive; the attacker harms only themselves. However, the condition can also arise organically for a long-lived, active delegator who simply forgets to claim rewards regularly.

---

### Recommendation

1. **Enforce a maximum trace length.** In `set_member_balance`, assert that `pool_member_epoch_balance.length()` does not exceed a protocol-defined cap (e.g., 1 000 entries) before inserting a new checkpoint.
2. **Require reward claiming before balance changes.** Force `claim_rewards` (or an internal equivalent that advances `entry_to_claim_from`) before any new balance checkpoint is written, keeping the pending-iteration window bounded to at most 1–2 entries.
3. **Paginate `calculate_rewards`.** Allow partial reward computation across multiple transactions by persisting intermediate state, so no single call needs to iterate an arbitrarily large range.

---

### Proof of Concept

```
Epoch 0:  pool_member calls enter_delegation_pool(amount=MIN)
          → trace = [(epoch=K, balance=MIN)]
          entry_to_claim_from = 0

Epoch 1:  pool_member calls add_to_delegation_pool(amount=1)
          → trace = [(K, MIN), (K+1, MIN+1)]

Epoch 2:  pool_member calls add_to_delegation_pool(amount=1)
          → trace = [(K, MIN), (K+1, MIN+1), (K+2, MIN+2)]

...

Epoch N:  trace length = N+1
          entry_to_claim_from still = 0 (no claim_rewards called)

          pool_member calls claim_rewards()
          → calculate_rewards loops N+1 times
          → if N is large enough, transaction exceeds Starknet execution resource cap
          → claim_rewards reverts
          → pool member's accumulated STRK rewards are permanently frozen
```

The only precondition is that the pool member holds enough tokens to call `add_to_delegation_pool` once per epoch and pays the transaction fee. No privileged access is required.
