### Title
Global `last_reward_block` Lock Allows Unprivileged Caller to Permanently Deny Per-Block Consensus Rewards - (File: src/staking/staking.cairo)

---

### Summary

The `update_rewards` function in the `Staking` contract enforces a **global** once-per-block constraint via the `last_reward_block` storage variable. Because the function has no access control and the `disable_rewards` parameter is caller-supplied, any unprivileged address can invoke `update_rewards(any_valid_staker, disable_rewards=true)` each block to consume the per-block slot without distributing any rewards. This permanently denies all stakers their consensus-model per-block yield with no catch-up mechanism.

---

### Finding Description

**Vulnerability class**: Authorization bypass / state-transition bug (direct analog to the Redstone staleness-gate bug: a condition prevents timely updates of a cached value, here `last_reward_block`, causing stakers to permanently miss rewards).

In the consensus reward model, the `Staking` contract distributes per-block rewards to stakers through `update_rewards`. The function enforces a global once-per-block gate:

```cairo
// src/staking/staking.cairo lines 1453-1458
let current_block_number = starknet::get_block_number();
assert!(
    current_block_number > self.last_reward_block.read(),
    "{}",
    Error::REWARDS_ALREADY_UPDATED,
);
``` [1](#0-0) 

Once any call succeeds in a block, `last_reward_block` is written to that block number, and every subsequent call in the same block reverts. Critically, `last_reward_block` is a **single global scalar** — not a per-staker map:

```cairo
// src/staking/staking.cairo line 187
/// Last block number for which rewards were distributed.
last_reward_block: BlockNumber,
``` [2](#0-1) 

The `last_reward_block` write happens **before** the `disable_rewards` early-return branch:

```cairo
// src/staking/staking.cairo lines 1484-1489
// Update last block rewards.
self.last_reward_block.write(current_block_number);

if disable_rewards || self.is_pre_consensus() {
    return;
}
``` [3](#0-2) 

The only access control is `general_prerequisites()`, which checks only pause state and non-zero caller — no role restriction:

```cairo
// src/staking/staking.cairo lines 1794-1797
fn general_prerequisites(ref self: ContractState) {
    self.assert_is_unpaused();
    assert_caller_is_not_zero();
}
``` [4](#0-3) 

**Attack path**:

1. Attacker identifies any valid, active staker address with non-zero STRK balance (publicly readable from chain state).
2. In every block, attacker calls `update_rewards(valid_staker_address, disable_rewards=true)`.
3. The call passes all checks (staker exists, is active, has non-zero balance), writes `last_reward_block = current_block`, then returns early — **zero rewards distributed**.
4. The legitimate consensus-layer call for any staker in the same block hits `REWARDS_ALREADY_UPDATED` and reverts.
5. Stakers permanently lose per-block rewards for every griefed block; there is no catch-up mechanism.

The per-block reward is calculated once per epoch and cached in `block_rewards`:

```cairo
// src/staking/staking.cairo lines 1563-1570
if curr_epoch > self.last_calculated_epoch.read() {
    self.last_calculated_epoch.write(curr_epoch);
    let block_rewards = reward_supplier_dispatcher.update_current_epoch_block_rewards();
    self.block_rewards.write(block_rewards);
    block_rewards
} else {
    self.block_rewards.read()
}
``` [5](#0-4) 

Each call to `update_rewards` distributes exactly one block's worth of rewards. Missed blocks are gone permanently.

---

### Impact Explanation

**High — Permanent freezing / theft of unclaimed yield.**

Every block the attacker griefs, all stakers lose their per-block consensus reward with no recovery path. Over time this compounds: if the attacker sustains the attack for an epoch, stakers receive zero consensus rewards for that epoch. The `RewardSupplier` still mints tokens (the `unclaimed_rewards` counter in `RewardSupplier` is incremented by the staking contract only when `_update_rewards` actually runs), so the lost yield is not redistributed — it is effectively frozen in the `RewardSupplier` and never claimed.

---

### Likelihood Explanation

**Medium.** The attacker needs one transaction per block on Starknet. Starknet transaction fees are low. The attack requires no privileged access, no leaked keys, and no external dependency — only a valid staker address (publicly available). The attacker gains nothing financially (pure griefing), but the barrier to execution is minimal.

---

### Recommendation

1. **Add access control** to `update_rewards`: restrict callers to a designated consensus-layer address (e.g., a registered `consensus_operator` role), analogous to how `update_rewards_from_attestation_contract` is restricted to the attestation contract.
2. **Alternatively**, move `last_reward_block` to a per-staker map (`Map<ContractAddress, BlockNumber>`) so that one staker's update slot does not block all others.
3. **Decouple the `disable_rewards` write from `last_reward_block`**: only update `last_reward_block` when rewards are actually distributed (i.e., after the `disable_rewards` branch), so a no-op call does not consume the per-block slot.

---

### Proof of Concept

```
// Attacker script (pseudocode, one tx per block):
loop every block:
    staking_contract.update_rewards(
        staker_address = <any valid active staker>,
        disable_rewards = true
    )
    // Effect:
    //   last_reward_block = current_block  (slot consumed)
    //   no rewards distributed
    //   consensus layer call in same block → REWARDS_ALREADY_UPDATED
    //   all stakers miss rewards for this block permanently
```

The attacker address needs only to be non-zero and the contract must be unpaused — no other preconditions. The staker address used as the argument can be any staker visible in the `stakers` vector (public storage).
