### Title
Malicious Participant Can Send Arbitrary `(big_y, big_c)` to Corrupt the Coordinator's CKD Aggregation — (`src/confidential_key_derivation/protocol.rs`)

### Summary

The CKD coordinator aggregates per-participant contributions `(norm_big_y, norm_big_c)` with no proof of correct formation and no point validation. A malicious participant can send any two G1 points, causing the final `CKDOutput` to be cryptographically invalid and making `unmask(app_sk)` return a wrong value that fails `verify_signature`.

### Finding Description

In `do_ckd_coordinator`, the coordinator receives contributions from all other participants via `recv_from_others::<CKDOutput>` and unconditionally adds them: [1](#0-0) 

`recv_from_others` performs only deserialization — no structural validation: [2](#0-1) 

`CKDOutput` is a plain serde-derived struct with no invariant enforcement on its fields: [3](#0-2) 

Honest participants send a tuple `(norm_big_y, norm_big_c)`: [4](#0-3) 

A malicious participant can instead send any two `G1Projective` points. There is no zero-knowledge proof, commitment, or consistency check that `big_y = y_i * G` and `big_c = x_i * H(pk||app_id) + y_i * app_pk`. The `frost_core::Group::deserialize` for `BLS12381G1Group` does reject identity and malformed points: [5](#0-4) 

However, this path is **not used** for `CKDOutput` deserialization. `CKDOutput` uses `#[derive(Deserialize, Serialize)]` via `rmp_serde` (MessagePack), which calls `blstrs`'s own serde impl for `G1Projective` — not `frost_core::Group::deserialize`. The `blstrs` serde path accepts any valid on-curve point including the identity.

### Impact Explanation

The protocol specification requires:

> Node i sends (λ_i · Y_i, λ_i · C_i) to the coordinator [6](#0-5) 

If a malicious participant sends `(big_y = identity, big_c = X)`:
- `norm_big_y += identity` is a no-op — the malicious participant's blinding contribution is dropped
- `norm_big_c += X` — an arbitrary offset is injected into the ciphertext

The resulting `CKDOutput` has `big_c ≠ msk·H(pk,app_id) + a·Y`, so `unmask(app_sk)` returns a wrong point that fails `verify_signature`: [7](#0-6) 

The honest app cannot recover its confidential derived key. This is **High: Corruption of CKD output so honest parties receive an unusable cryptographic output**.

### Likelihood Explanation

Any registered participant in the protocol can execute this attack. The participant is already a trusted member of the participant list (per `ckd()` initialization checks), but the protocol makes no assumption that participants are honest — it should be robust against malicious-but-registered participants. The attack requires only sending crafted bytes over the existing message channel, with no cryptographic capability required.

### Recommendation

Each participant must prove their contribution is well-formed. The standard fix is to require a **Schnorr proof of knowledge** (or a sigma protocol) showing that `big_y = y_i * G` and `big_c = x_i * H(pk||app_id) + y_i * app_pk` for the same `y_i`, committed before the contribution is accepted. The coordinator should verify these proofs before aggregating. Alternatively, a commitment-then-reveal scheme can bind participants to their contributions before the coordinator aggregates.

pub async fn recv_from_others<T>(
    chan: &SharedChannel,
    waitpoint: u64,
    participants: &ParticipantList,
    me: Participant,
) -> Result<Vec<(Participant, T)>, ProtocolError>
where
    T: serde::de::DeserializeOwned,
{
    let mut seen = ParticipantCounter::new(participants);
    seen.put(me);
    let mut messages = Vec::with_capacity(participants.others(me).count());

    while !seen.full() {
        let (from, msg) = chan.recv(waitpoint).await?;
        if seen.put(from) {
            messages.push((from, msg));
        }
    }

    Ok(messages)
