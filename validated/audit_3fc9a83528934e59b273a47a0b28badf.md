### Title
Missing Minimum Threshold Validation in FROST Sign Inputs Allows Single-Participant Signature Forgery - (File: src/frost/mod.rs)

### Summary

`assert_sign_inputs` in `src/frost/mod.rs` validates that the threshold is not *too large* but never checks that it is at least 2. Every other threshold-accepting entry point in the library (`assert_key_invariants` in `src/dkg.rs`, `validate_triple_inputs` in `src/ecdsa/ot_based_ecdsa/triples/generation.rs`) enforces `threshold >= 2`. The omission in the FROST signing path means a caller can pass `threshold = 1`, causing the protocol to proceed with a single-participant Lagrange reconstruction that produces a valid threshold signature without the required co-signers.

### Finding Description

`assert_sign_inputs` is the sole validation gate for FROST EdDSA and RedJubjub signing. It performs three checks:

1. `participants.len() < 2` → error [1](#0-0) 
2. `threshold.value() > participants.len()` → error [2](#0-1) 
3. coordinator present → error [3](#0-2) 

There is **no lower-bound check** (`threshold < 2`). Compare with the identical guard present in every other threshold-accepting function:

`assert_key_invariants` (DKG / reshare path):
```rust
if threshold < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
}
``` [4](#0-3) 

`validate_triple_inputs` (OT-based triple generation):
```rust
if threshold_value < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold_value,
        min: 2,
    });
}
``` [5](#0-4) 

The same omission exists in the FROST `presign` function, which also only checks the upper bound: [6](#0-5) 

`ReconstructionLowerBound` is a plain `usize` wrapper with no enforced minimum, so `threshold = 1` (or `0`) is a valid value at the type level: [7](#0-6) 

When `threshold = 1` is accepted, FROST's Lagrange interpolation requires only one participant's share to reconstruct the group secret and produce a valid signature. The two-participant minimum enforced on the *participant list* (`participants.len() < 2`) does not compensate for this, because the threshold governs how many shares are *needed* for reconstruction, not how many parties are *present*.

### Impact Explanation

A single participant who controls one signing share can call the FROST sign entry point with `threshold = 1`. Because `assert_sign_inputs` accepts this value without error, the protocol proceeds and the single participant's Lagrange-interpolated share is sufficient to produce a fully valid threshold signature over an attacker-chosen message. This constitutes **unauthorized creation of a valid threshold signature** — the core security property of the scheme (requiring `t` cooperating parties) is silently nullified.

Impact category: **Critical** — matches "Unauthorized creation of a valid threshold signature for attacker-chosen inputs."

### Likelihood Explanation

Any unprivileged library caller who invokes the FROST signing API controls the `threshold` argument directly. No privileged access, leaked keys, or cryptographic breaks are required. The call path is a standard, documented public API. The missing check is a single missing `if` branch that exists in every analogous function in the same codebase, making accidental omission the realistic root cause and accidental misuse by integrators the realistic trigger.

### Recommendation

Add the same lower-bound guard that exists in `assert_key_invariants` and `validate_triple_inputs` to both `assert_sign_inputs` and the FROST `presign` function:

```rust
// In assert_sign_inputs and presign (src/frost/mod.rs)
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

This should be inserted immediately after the `threshold.into()` conversion, before any protocol state is created, mirroring the pattern at `src/dkg.rs:580-582`.

### Proof of Concept

```rust
// Pseudocode — uses the public FROST EdDSA signing API
let participants = generate_participants(3);   // 3 parties present
let threshold: ReconstructionLowerBound = 1.into(); // threshold = 1 — SHOULD be rejected

// assert_sign_inputs accepts threshold=1 without error:
//   participants.len() (3) >= 2  ✓
//   threshold.value() (1) <= participants.len() (3)  ✓  ← no lower-bound check
let participant_list = assert_sign_inputs(
    &participants,
    threshold,
    participants[0],   // me
    participants[0],   // coordinator
).unwrap(); // succeeds — no error returned

// Participant 0 alone now runs the FROST sign protocol.
// With threshold=1, Lagrange interpolation over a single share
// reconstructs the full secret and produces a valid signature
// without any cooperation from participants[1] or participants[2].
```

Expected: `InitializationError::ThresholdTooSmall { threshold: 1, min: 2 }` — same error returned by `assert_key_invariants` and `validate_triple_inputs` for the same input.
Actual: protocol proceeds, single-participant signature produced successfully.
