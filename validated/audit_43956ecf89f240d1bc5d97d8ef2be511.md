### Title
Zero Echo/Ready Thresholds Enable Split-View Attack in DKG Broadcast for Small Participant Sets - (File: src/protocol/echo_broadcast.rs)

### Summary

`echo_ready_thresholds` returns `(0, 0)` for any participant count `n <= 3`. Because the DKG protocol permits `n = 3` with `threshold = 2`, a malicious participant can exploit the zero thresholds to deliver inconsistent commitment data to different honest parties, causing them to derive different public keys from the same DKG session.

### Finding Description

In `src/protocol/echo_broadcast.rs`, the threshold computation is:

```rust
fn echo_ready_thresholds(n: usize) -> (usize, usize) {
    if n <= 3 {
        return (0, 0);   // <-- both thresholds collapse to zero
    }
    let broadcast_threshold = (n - 1) / 3;
    let echo_threshold = usize::midpoint(n, broadcast_threshold);
    (echo_threshold, broadcast_threshold)
}
``` [1](#0-0) 

These thresholds are consumed in `reliable_broadcast_receive_all` at three decision points:

| Check | With `ready_t = 0` |
|---|---|
| `count > echo_t` (trigger Ready) | `count > 0` — satisfied by the first self-simulated echo |
| `count > ready_t` (trigger amplification) | `count > 0` — satisfied by the first self-simulated ready |
| `count > 2 * ready_t` (trigger delivery) | `count > 0` — satisfied by the same first ready | [2](#0-1) 

Because all three gates collapse to `count > 0`, a participant's own self-simulated vote is sufficient to immediately deliver whatever data it received in the `Send` phase — with no cross-participant consistency check. A malicious participant M can therefore send `Send(data_A)` to honest party A and `Send(data_B)` to honest party B; each honest party echoes and delivers its own version without ever seeing the other's.

`do_broadcast` is called twice in `do_keyshare` — once for session IDs (Round 1) and once for `(commitment, proof_of_knowledge)` (Round 3): [3](#0-2) [4](#0-3) 

The commitment broadcast in Round 3 is the critical path. M can:
1. Send different commitment hashes to A and B in Round 1 via `recv_from_others` (point-to-point, no broadcast protection).
2. Send `(commitment_A, proof_A)` to A and `(commitment_B, proof_B)` to B in Round 3 via the zero-threshold broadcast.
3. Craft both proofs legitimately (M knows the discrete logs of both polynomials it generated).

Each honest party verifies its own commitment against its own hash and its own proof against the shared `session_id` — both checks pass. The `broadcast_success` round then passes too, since both parties broadcast `(true, session_id)` with the same session ID. [5](#0-4) 

The minimum valid DKG configuration (`n = 2`, `threshold = 2`) is enforced by `assert_key_invariants`, but no lower bound prevents `n = 3` from being used: [6](#0-5) 

### Impact Explanation

Honest parties A and B complete DKG successfully from their own perspective but hold signing shares consistent with **different public keys**. Any subsequent threshold signing attempt between A and B will produce an invalid signature or fail outright, because their shares were generated against different aggregate commitments. This matches the allowed High impact: **Corruption of DKG outputs so honest parties accept inconsistent public keys**.

### Likelihood Explanation

The minimum participant count for DKG is 2, and `n = 3` with `threshold = 2` is the smallest configuration that is supposed to tolerate one malicious party. Any deployment using exactly 3 participants is vulnerable. The attacker is a registered protocol participant (no external access required), and the attack requires only crafting two valid polynomial commitments — a trivial computation.

### Recommendation

The `echo_ready_thresholds` function should enforce a minimum absolute threshold so that delivery always requires agreement from more than one party. One approach is to combine the relative threshold with an absolute floor:

```rust
fn echo_ready_thresholds(n: usize) -> (usize, usize) {
    if n <= 3 {
        // Require at least 1 other party to agree, not zero
        return (1, 1);
    }
    let broadcast_threshold = (n - 1) / 3;
    let echo_threshold = usize::midpoint(n, broadcast_threshold);
    (echo_threshold, broadcast_threshold)
}
```

Alternatively, `assert_key_invariants` should enforce `participants.len() >= 4` so that the echo broadcast always operates with non-zero thresholds, or the documentation must explicitly state that `n <= 3` provides no Byzantine fault tolerance and callers must not use it with untrusted participants.

### Proof of Concept

With `n = 3` (participants A, B, M where M is malicious):

1. **Round 1 — session IDs**: M sends the same session ID to A and B (so the global `session_id` is identical for both). M sends **different commitment hashes** `hash_A` and `hash_B` to A and B respectively via `recv_from_others`.

2. **Round 3 — commitment broadcast**: M calls `do_broadcast` but sends `Send(commitment_A, proof_A)` to A and `Send(commitment_B, proof_B)` to B. With `echo_t = 0`, each honest party echoes its own received value immediately. With `ready_t = 0`, each honest party delivers its own received value after a single self-simulated Ready vote — before any cross-party echo is processed.

3. **Verification passes for both**: A verifies `proof_A` against `commitment_A` and `hash_A` → valid. B verifies `proof_B` against `commitment_B` and `hash_B` → valid. Both call `public_key_from_commitments` with different commitment sets and derive different public keys.

4. **Round 5 — broadcast_success**: Both A and B broadcast `(true, session_id)`. The check passes. Both parties return `KeygenOutput` with inconsistent public keys. [1](#0-0) [7](#0-6) [8](#0-7) [9](#0-8) [10](#0-9)

    // unwrap here would never fail as the broadcast protocol ends only when the map is full

        .ok_or_else(|| ProtocolError::AssertionFailed("vote_list is empty".to_string()))?;
    // go through all the list of votes and check if any is fail or some does not contain the session id

            "A participant

                .to_string(),
        ));
    }
    // Wait for all the tasks to complete
    Ok(())
