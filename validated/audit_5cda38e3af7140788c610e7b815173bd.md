### Title
Unauthenticated Invoke Transaction Admitted to Mempool via `skip_stateful_validations` Signature-Bypass — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the Apollo gateway bypasses the `__validate__` entry point (the account's on-chain signature verification) for invoke transactions with nonce=1 when the sender's address has *any* transaction in the mempool. An unprivileged attacker who observes a pending `deploy_account` transaction for address X can immediately submit an invoke transaction with nonce=1 from X carrying an invalid/arbitrary signature. Because the gateway skips `__validate__`, the invalid transaction is admitted to the mempool without any signature check.

### Finding Description

In `crates/apollo_gateway/src/stateful_transaction_validator.rs`, the function `skip_stateful_validations` implements a UX shortcut: when a user submits `deploy_account + invoke` together, the invoke's `__validate__` is skipped so the gateway does not reject it solely because the account is not yet deployed. [1](#0-0) 

The skip condition is:
1. Transaction is `Invoke`
2. `tx.nonce() == 1`
3. `account_nonce == 0` (account not yet deployed)
4. `account_tx_in_pool_or_recent_block(sender_address) == true`

Condition 4 is implemented as: [2](#0-1) 

This returns `true` if **any** transaction from the address is in the pool — it does not verify that the pooled transaction is a `deploy_account`, nor does it verify that the incoming invoke transaction is signed by the same key. When `skip_stateful_validations` returns `true`, `run_validate_entry_point` sets `execution_flags.validate = false`: [3](#0-2) 

Inside `StatefulValidator::perform_validations`, when `execution_flags.validate` is `false`, the function returns `Ok(())` immediately without calling `validate_tx` (the `__validate__` entry point): [4](#0-3) 

And in `AccountTransaction::validate_tx`, the same flag causes an early return with no signature check: [5](#0-4) 

The `validate_by_mempool` call that precedes `skip_stateful_validations` only checks mempool-level constraints (nonce gaps, duplicates, fee bounds) — it does not run `__validate__`: [6](#0-5)

### Impact Explanation

An attacker can inject an invoke transaction with an **invalid signature** into the mempool for any address that has a pending `deploy_account` transaction. This matches the allowed impact: **"High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."**

Concrete consequences:
- The attacker's invalid tx occupies the `(X, nonce=1)` slot in the mempool, potentially blocking the legitimate owner's invoke tx (the mempool enforces nonce uniqueness per address).
- The legitimate owner's `deploy_account + invoke` UX flow is disrupted; they must fee-escalate to replace the attacker's tx.
- The invalid tx will fail during batcher execution (when `__validate__` is run with `validate=true`), wasting block space and causing the legitimate owner's deploy_account to land in a block without the paired invoke.

### Likelihood Explanation

- **No privileges required.** Any external user can submit transactions to the gateway.
- **Observable trigger.** Pending `deploy_account` transactions are visible on the P2P mempool network, giving the attacker the target address and the timing signal.
- **Trivially constructable payload.** The attacker only needs to craft an invoke transaction with `sender_address=X`, `nonce=1`, and any arbitrary bytes as the signature. All other fields (resource bounds, calldata) can be minimal but valid.
- **Race condition is winnable.** The attacker submits their invoke tx immediately after observing the deploy_account tx; the legitimate owner typically does not submit the paired invoke tx until slightly later (or relies on the gateway to handle it).

### Recommendation

1. **Verify the pooled transaction type.** `account_tx_in_pool_or_recent_block` should be replaced with a check that specifically confirms a `deploy_account` transaction (not just any transaction) is pending for the address.
2. **Do not skip `__validate__` entirely.** Instead of bypassing signature verification, the gateway could defer the invoke tx until the deploy_account is committed, or run a lightweight structural signature check (non-empty, correct length) before skipping the full on-chain validation.
3. **Apply `max_nonce_for_validation_skip`.** The config field `max_nonce_for_validation_skip` is defined in `StatefulTransactionValidatorConfig` and serialized in the production config schema, but is **never read** by `skip_stateful_validations` in the new gateway path (it is only used in the legacy `PyValidator` path). The new gateway path hardcodes the check to `nonce == 1`. This dead config field should either be wired into `skip_stateful_validations` or removed. [8](#0-7) [9](#0-8) 

### Proof of Concept

```
1. Legitimate owner Alice submits deploy_account tx for address X (valid signature, nonce=0).
   → Gateway admits it; mempool now has tx_pool.contains_account(X) == true.

2. Attacker observes X in the mempool (via P2P or public gateway).

3. Attacker submits invoke tx:
     sender_address = X
     nonce          = 1
     signature      = [0xdeadbeef]   ← invalid
     resource_bounds = minimal valid values

4. Gateway stateful validation:
   a. validate_state_preconditions: nonce=1 >= account_nonce=0 ✓, resource bounds ✓
   b. validate_by_mempool: no duplicate nonce for (X,1) yet ✓
   c. skip_stateful_validations:
        tx.nonce() == 1 ✓
        account_nonce == 0 ✓
        account_tx_in_pool_or_recent_block(X) == true ✓  ← Alice's deploy_account
      → returns true (skip __validate__)
   d. run_validate_entry_point: execution_flags.validate = false → returns Ok(()) immediately

5. Attacker's invalid tx is admitted to the mempool at slot (X, nonce=1).

6. Alice now submits her own invoke tx with nonce=1 (valid signature).
   → Mempool rejects it: DuplicateNonce for (X, 1).
   → Alice must fee-escalate to displace the attacker's tx.

7. Batcher executes the block:
   → Alice's deploy_account succeeds.
   → Attacker's invoke tx fails __validate__ (invalid signature) → reverted/rejected.
   → Alice's intended invoke tx never executes in this block.
```

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
