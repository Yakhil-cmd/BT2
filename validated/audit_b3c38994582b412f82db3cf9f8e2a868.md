### Title
Pool Admin Can Set Per-Bin Additional Fees Without Any Upper-Bound Cap, Bypassing the Protocol Fee Cap System — (File: `metric-core/contracts/MetricOmmPoolFactory.sol` / `metric-core/contracts/MetricOmmPool.sol`)

### Summary

`setPoolBinAdditionalFees` in `MetricOmmPoolFactory` and `setBinAdditionalFees` in `MetricOmmPool` accept `addFeeBuyE6` / `addFeeSellE6` as raw `uint16` values with **no upper-bound cap check**. Every other fee path in the protocol is guarded by hard caps (`HARD_MAX_SPREAD_FEE_E6 = 200_000`, `HARD_MAX_NOTIONAL_FEE_E8 = 1_000_000`) and per-pool soft caps (`maxAdminSpreadFeeE6`, `maxAdminNotionalFeeE8`). The per-bin additional fees are completely exempt from this cap system, allowing the pool admin to set them to `uint16.max = 65_535` (≈ 6.55 % in E6 units) on every bin, silently exceeding the intended 20 % hard ceiling on effective swap cost.

---

### Finding Description

The factory enforces a layered fee cap system:

- `HARD_MAX_SPREAD_FEE_E6 = 200_000` (20 %) — absolute ceiling enforced in `setFeeCaps`.
- `maxAdminSpreadFeeE6` — per-pool soft cap checked in `setPoolAdminFees` and `createPool → _validatePoolParameters`.
- `maxProtocolSpreadFeeE6` — checked in `setPoolProtocolFee` and `setDefaultSpreadProtocolFeeE6`. [1](#0-0) [2](#0-1) [3](#0-2) 

The per-bin additional fees follow a completely different path. `setPoolBinAdditionalFees` in the factory passes the caller-supplied values directly to the pool with **no cap validation**: [4](#0-3) 

`setBinAdditionalFees` on the pool validates only the bin index, not the fee magnitudes: [5](#0-4) 

The same gap exists at pool creation. `_unpackAndValidateBinStates` validates `lengthE6` and cumulative distance but silently accepts any `uint16` value for `addFeeBuyE6` / `addFeeSellE6`: [6](#0-5) 

The per-bin fees are then injected directly into the swap fee multiplier used by `SwapMath`: [7](#0-6) 

The effective buy-fee passed to `buyToken0InBinSpecifiedIn` is `baseFeeX64 + mulDiv(addFeeBuyE6, ONE_X64, 1e6)`. With `addFeeBuyE6 = 65_535`, the per-bin component alone is ≈ 6.55 %, added on top of the oracle spread and the base spread fee, with no ceiling.

The same uncapped path applies to `getSellAndBuyPrices` (the public price view) and the `MetricOmmPoolDataProvider` depth ladder, both of which incorporate `addFeeBuyE6` / `addFeeSellE6` without any bound check: [8](#0-7) [9](#0-8) 

---

### Impact Explanation

The pool admin is semi-trusted **only inside caps**. By setting `addFeeBuyE6 = addFeeSellE6 = 65_535` on every bin, the pool admin inflates the effective ask price and deflates the effective bid price by ≈ 6.55 % beyond the oracle spread, on top of the already-allowed 20 % base spread. Traders executing swaps in those bins pay materially more than the protocol's intended hard ceiling permits. The excess fee accrues to LPs (not the protocol), so it is not recoverable by the protocol and represents a direct, permanent extraction from traders beyond the intended fee boundary. The same misconfiguration can be embedded at pool creation time by any pool creator, making it reachable without any post-deploy admin action.

---

### Likelihood Explanation

The trigger is a single call to `setPoolBinAdditionalFees` by the pool admin (a semi-trusted role that is explicitly in scope for cap-bypass analysis per the audit pivots). No timelock, no privileged escalation, and no special market condition is required. The same path is reachable permissionlessly at pool creation via the packed bin arrays. The missing check is a single missing `if` statement, making accidental or deliberate exploitation straightforward.

---

### Recommendation

Add an explicit upper-bound cap on `addFeeBuyE6` and `addFeeSellE6` in both the factory and the pool, consistent with the existing spread fee cap system. For example, define a constant `MAX_BIN_ADDITIONAL_FEE_E6` (e.g. `200_000`, matching `HARD_MAX_SPREAD_FEE_E6`) and enforce it in:

1. `MetricOmmPoolFactory.setPoolBinAdditionalFees` — before forwarding to the pool.
2. `MetricOmmPool.setBinAdditionalFees` — as a secondary guard.
3. `MetricOmmPoolFactory._unpackAndValidateBinStates` — for the initial bin configuration at pool creation.

```solidity
// In setPoolBinAdditionalFees / setBinAdditionalFees:
if (addFeeBuyE6 > MAX_BIN_ADDITIONAL_FEE_E6 || addFeeSellE6 > MAX_BIN_ADDITIONAL_FEE_E6)
    revert BinAdditionalFeeTooHigh();
```

```solidity
vm.prank(poolAdmin);

// Trader swaps: effective fee = baseFeeX64 + 65535/1e6 * ONE_X64
// ≈ oracle_spread + 6.55% — exceeds the 20% hard cap on base spread
// No revert, no cap enforcement
(int128 a0, int128 a1) = pool.swap(
    trader, false, int128(1_000e18), type(uint128).max, "", ""
);
// Trader paid ~6.55% more than the protocol's intended maximum
```
