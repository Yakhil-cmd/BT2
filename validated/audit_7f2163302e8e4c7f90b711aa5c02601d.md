### Title
Pool Admin Can Set Unbounded Per-Bin Additional Fees, Bypassing the Factory Fee Cap System - (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary

`setPoolBinAdditionalFees` passes `addFeeBuyE6` and `addFeeSellE6` directly to the pool with no upper-bound validation, while every other admin-controlled fee parameter is gated by `maxAdminSpreadFeeE6` / `maxAdminNotionalFeeE8`. A pool admin can set per-bin fees to `type(uint16).max` (65,535 E6 ≈ 6.55%) on every bin, charging traders fees that exceed the factory's intended cap and extracting a larger share of the resulting surplus.

### Finding Description

`MetricOmmPoolFactory.setPoolBinAdditionalFees` is the pool-admin entry point for per-bin fee overrides:

```solidity
// MetricOmmPoolFactory.sol L450-457
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
``` [1](#0-0) 

No cap check is performed on `addFeeBuyE6` or `addFeeSellE6`. Compare this with `setPoolAdminFees`, which explicitly enforces caps:

```solidity
// MetricOmmPoolFactory.sol L414-415
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (newAdminNotionalFeeE8 > maxAdminNotionalFeeE8) revert AdminFeeTooHigh();
``` [2](#0-1) 

The pool's `setBinAdditionalFees` only validates the bin index, not the fee values:

```solidity
// MetricOmmPool.sol L469-473
if (bin < LOWEST_BIN || bin > HIGHEST_BIN) revert InvalidBinIndex(bin);
BinState storage s = _binStates[bin];
s.addFeeBuyE6 = addFeeBuyE6;
s.addFeeSellE6 = addFeeSellE6;
``` [3](#0-2) 

During every swap, the per-bin additional fee is added directly to the oracle-derived base fee before being applied to the swap amount:

```solidity
// MetricOmmPool.sol L1177
params.baseFeeX64 + Math.mulDiv(binState.addFeeSellE6, ONE_X64, 1e6)
``` [4](#0-3) 

The same pattern appears for the buy direction:

```solidity
// MetricOmmPool.sol L1088
params.baseFeeX64 + Math.mulDiv(binState.addFeeSellE6, ONE_X64, 1e6)
``` [5](#0-4) 

The resulting surplus (pool balance minus LP claims) is then split between admin and protocol via `collectFees` using the `adminSpreadFeeE6 / spreadFeeE6` ratio. Setting high per-bin fees inflates the surplus, and the admin collects a proportionally larger absolute amount. [6](#0-5) 

### Impact Explanation

- `uint16.max` = 65,535 E6 ≈ **6.55% additional fee per bin**, additive on top of the oracle spread.
- A pool admin can apply this to every bin simultaneously, making the pool's effective fee far exceed the hard cap (`HARD_MAX_SPREAD_FEE_E6` = 20%).
- Traders pay more than the protocol's fee cap system intends; the admin extracts a larger share of the inflated surplus.
- LP funds are not locked (removeLiquidity is unaffected), but the swap flow is economically broken for users of that pool.

### Likelihood Explanation

The pool admin is a semi-trusted role. The factory explicitly caps all other admin-controlled fee parameters but omits the cap for per-bin fees. Any pool admin — including one that turns adversarial after deployment — can call `setPoolBinAdditionalFees` with `addFeeBuyE6 = addFeeSellE6 = type(uint16).max` in a single transaction with no timelock. [1](#0-0) 

### Recommendation

Add a cap check in `setPoolBinAdditionalFees` analogous to the check in `setPoolAdminFees`. Define a constant (e.g., `MAX_BIN_ADDITIONAL_FEE_E6`) and revert if either `addFeeBuyE6` or `addFeeSellE6` exceeds it:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (addFeeBuyE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

### Proof of Concept

1. Factory owner deploys a pool with `maxAdminSpreadFeeE6 = 200_000` (20% hard cap).
2. Pool admin calls `factory.setPoolBinAdditionalFees(pool, 0, 65535, 65535)` — no revert.
3. A trader calls `pool.swap(...)` routing through bin 0. The effective fee applied is `baseFeeX64 + mulDiv(65535, ONE_X64, 1e6)` ≈ oracle_spread + 6.55%, exceeding the 20% cap intent when combined with the global spread fee.
4. The inflated surplus is collected via `collectPoolFees`; the admin receives their `adminSpreadFeeE6 / spreadFeeE6` share of the larger-than-intended surplus. [7](#0-6) [8](#0-7)
