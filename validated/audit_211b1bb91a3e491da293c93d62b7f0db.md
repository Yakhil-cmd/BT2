### Title
Wrong Threshold Applied in `ValidateTC`: `QuorumThresholdForView` Used Instead of `TimeoutThresholdForView` - (File: `consensus/hotstuff/validator/validator.go`)

### Summary

`Validator.ValidateTC` validates a `TimeoutCertificate` (TC) by checking that the total weight of its signers meets a minimum threshold. However, it fetches the **QC supermajority threshold** (`QuorumThresholdForView`, requiring >2/3 of total weight) instead of the **TC timeout threshold** (`TimeoutThresholdForView`, requiring >1/3 of total weight). This is an incompatibility between the threshold used to *build* a TC and the threshold used to *validate* one — directly analogous to the reported bug where a quorum calculation was incompatible with the actual voting mechanism.

### Finding Description

In `consensus/hotstuff/validator/validator.go`, `ValidateTC` calls `v.committee.QuorumThresholdForView(tc.View)` to determine the minimum weight required for a valid TC:

```go
// determine whether signers reach minimally required weight threshold for consensus
threshold, err := v.committee.QuorumThresholdForView(tc.View)
...
if signers.TotalWeight() < threshold {
    return newInvalidTCError(tc, ...)
}
``` [1](#0-0) 

The QC threshold is `WeightThresholdToBuildQC(totalWeight)` = `⌊2·totalWeight/3⌋ + 1` (strictly greater than 2/3). [2](#0-1) 

However, a TC is built by `TimeoutProcessor` using the **timeout threshold** (`WeightThresholdToTimeout`), which is `⌊totalWeight/3⌋ + 1` (strictly greater than 1/3):

```go
tcTracker: accumulatedWeightTracker{
    minRequiredWeight: qcThreshold,   // ← BUG: uses QC threshold, not timeout threshold
    ...
},
``` [3](#0-2) 

Wait — looking more carefully at `NewTimeoutProcessor`, the `tcTracker` (which triggers TC construction) uses `qcThreshold` (the QC supermajority threshold), while `partialTCTracker` uses `timeoutThreshold`. This means a TC is only *built* once >2/3 weight is collected, but the protocol specification for a TC requires only >1/3. The `ValidateTC` then also applies the QC threshold, so the two are consistent with each other — but both are inconsistent with the HotStuff protocol specification for TCs.

The root cause is in `NewTimeoutProcessor`:

```go
qcThreshold, err := committee.QuorumThresholdForView(view)
...
tcTracker: accumulatedWeightTracker{
    minRequiredWeight: qcThreshold,  // should be timeoutThreshold
``` [4](#0-3) 

The `TimeoutThresholdForView` (>1/3) is fetched and assigned only to `partialTCTracker`, which merely triggers a `OnPartialTcCreated` notification — not the actual TC construction. The actual TC construction gate (`tcTracker`) uses the QC supermajority threshold (>2/3), which is the wrong threshold for a TC. [5](#0-4) 

`ValidateTC` then validates the TC against `QuorumThresholdForView` (>2/3), which matches what `TimeoutProcessor` actually requires to build a TC — so the two are internally consistent. But both deviate from the HotStuff protocol, which specifies that a TC requires only a strict minority (>1/3) of weight, not a supermajority.

### Impact Explanation

A TC is supposed to prove that enough replicas want to abandon a view — the threshold is >1/3 (one honest replica suffices to prevent a QC, so >1/3 guarantees at least one honest replica timed out). By requiring >2/3 weight to build and validate a TC, the system:

1. **Raises the liveness bar for view changes**: Honest nodes cannot trigger a view change unless >2/3 of weight agrees to time out, which is the same bar as building a QC. In a scenario where the leader is faulty and no QC can be formed, the network may be unable to advance if fewer than 2/3 of nodes time out simultaneously.
2. **Allows a Byzantine minority to block view changes**: A coalition holding just over 1/3 of weight can prevent TC formation by not sending timeouts, stalling consensus indefinitely.

This is a liveness-breaking miscalculation: the threshold used to build and validate TCs is incompatible with the protocol-specified threshold for TCs, mirroring the reported bug where the quorum calculation was incompatible with the actual voting power.

### Likelihood Explanation

This is reachable in normal protocol operation whenever a view fails to produce a QC and nodes attempt to time out. Any view where the leader is faulty or the network is partitioned such that 1/3 < weight_timing_out < 2/3 will trigger this condition. An unprivileged observer node or any network participant can observe this by submitting timeout objects and watching the view fail to advance.

### Recommendation

In `NewTimeoutProcessor`, assign `timeoutThreshold` (from `TimeoutThresholdForView`) to `tcTracker.minRequiredWeight` instead of `qcThreshold`:

```go
tcTracker: accumulatedWeightTracker{
    minRequiredWeight: timeoutThreshold, // was: qcThreshold
    ...
},
```

In `ValidateTC`, use `TimeoutThresholdForView` instead of `QuorumThresholdForView`:

```go
threshold, err := v.committee.TimeoutThresholdForView(tc.View)
``` [6](#0-5) 

### Proof of Concept

Given a committee of 10 nodes each with weight 100 (total = 1000):
- `WeightThresholdToBuildQC(1000)` = 668 (>2/3)
- `WeightThresholdToTimeout(1000)` = 334 (>1/3)

If 4 nodes (weight 400) time out:
- Protocol spec: TC should be buildable (400 > 334)
- Actual behavior: `tcTracker.minRequiredWeight` = 668, so no TC is built; `ValidateTC` would also reject any externally-supplied TC with only 400 weight

The network stalls in this view. A Byzantine leader can exploit this by simply not proposing, knowing that fewer than 668 weight of honest nodes will time out, preventing view advancement. [3](#0-2) [1](#0-0) [7](#0-6)

package committees

// WeightThresholdToBuildQC returns the weight (sum of unique, valid votes for this view)
// that is minimally required for a supermajority.
func WeightThresholdToBuildQC(totalWeight uint64) uint64 {
	// Given totalWeight, we need the smallest integer t such that 2 * totalWeight / 3 < t
	// Formally, the minimally required weight is: 2 * Floor(totalWeight/3) + max(1, totalWeight mod 3)
	floorOneThird := totalWeight / 3 // integer division, includes floor
	res := 2 * floorOneThird
	divRemainder := totalWeight % 3
	if divRemainder <= 1 {
		res = res + 1
	} else {
		res += divRemainder
	}
	return res
}

// WeightThresholdToTimeout returns the weight (sum of unique, valid timeout objects for this view)
// that is minimally required to immediately timeout and build a TO.
func WeightThresholdToTimeout(totalWeight uint64) uint64 {
	// Given totalWeight, we need the smallest integer t such that totalWeight / 3 < t
	// Formally, the minimally required weight is: Floor(totalWeight/3) + 1
	return totalWeight/3 + 1
}
```
