### Title
Pool Admin Can Set Per-Bin Additional Fees Without Any Cap, Bypassing Global Fee Caps — (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`MetricOmmPoolFactory.setPoolBinAdditionalFees` forwards `addFeeBuyE6` / `addFeeSellE6` to the pool with **no cap validation**, while every other admin fee path is gated by `maxAdminSpreadFeeE6` / `maxAdminNotionalFeeE8`. A pool admin can set per-bin additional fees up to `uint16.max = 65 535` (≈ 6.55 % in E6 units) on any bin, causing traders to pay fees that exceed the protocol's hard-coded fee-cap system.

---

### Finding Description

The factory enforces caps on the global admin fee components in `setPoolAdminFees`:

```solidity
// MetricOmmPoolFactory.sol L414-415
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (newAdminNotionalFeeE8 > maxAdminNotionalFeeE8) revert AdminFeeTooHigh();
``` [1](#0-0) 

But the per-bin path has **no equivalent guard**:

```solidity
// MetricOmmPoolFactory.sol L450-457
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
``` [2](#0-1) 

The pool-side function only validates the bin index, never the fee magnitudes:

```solidity
// MetricOmmPool.sol L464-474
function setBinAdditionalFees(int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external onlyFactory nonReentrant(PoolActions.SET_BIN_ADDITIONAL_FEES)
{
    if (bin < LOWEST_BIN || bin > HIGHEST_BIN) revert InvalidBinIndex(bin);
    BinState storage s = _binStates[bin];
    s.addFeeBuyE6 = addFeeBuyE6;
    s.addFeeSellE6 = addFeeSellE6;
    emit BinAdditionalFeesUpdated(bin, addFeeBuyE6, addFeeSellE6);
}
``` [3](#0-2) 

During every swap the bin additional fee is **added on top of** the oracle-derived base fee before the trade is priced:

```solidity
// MetricOmmPool.sol L999 (buy-token0 exact-in path)
params.baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6)
``` [4](#0-3) 

The same pattern appears for all four swap directions (lines 999, 1088, 1177). [5](#0-4) 

The `spreadFeeE6` cap (hard max 20 %) governs only the **protocol+admin split of LP fees already collected**; it does not bound the gross fee charged to the trader. Setting `addFeeBuyE6 = 65535` adds ≈ 6.55 % to every buy through that bin, of which the admin receives up to `6.55 % × spreadFeeE6 / 1e6` (≈ 1.31 % of notional at the 20 % cap) directly, with the remainder accruing to LPs — but the trader bears the full 6.55 % loss regardless.

The hard-coded constants confirm the intent to cap admin influence:

```solidity
// MetricOmmPoolFactory.sol L44-45
uint24 internal constant HARD_MAX_SPREAD_FEE_E6 = 200_000;   // 20 %
uint24 internal constant HARD_MAX_NOTIONAL_FEE_E8 = 1_000_000; // 1 %
``` [6](#0-5) 

Per-bin fees are entirely outside this cap hierarchy.

---

### Impact Explanation

A pool admin can call `setPoolBinAdditionalFees(pool, bin, 65535, 65535)` on the active bin, immediately making every subsequent swap through that bin pay ≈ 6.55 % in additional fees. This is a **direct loss of trader principal** on every swap routed through the affected bin. The loss is proportional to swap size and persists until the admin resets the fee. The admin's own extraction is bounded by `spreadFeeE6`, but the trader's loss is not.

---

### Likelihood Explanation

The pool admin is a semi-trusted role that can act at any time without a timelock on this specific function. No timelock, no governance delay, and no on-chain cap prevents the admin from setting `addFeeBuyE6 = 65535` immediately after pool creation or at any later point. Any pool whose admin key is compromised, or any admin acting adversarially within the semi-trusted boundary, can trigger this.

---

### Recommendation

Add a cap check in `setPoolBinAdditionalFees` (or inside `setBinAdditionalFees`) that mirrors the existing admin fee cap logic:

```solidity
function setPoolBinAdditionalFees(
    address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6
) external override nonReentrant onlyPoolAdmin(pool) {
    if (addFeeBuyE6  > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

Alternatively, introduce a dedicated `maxBinAdditionalFeeE6` cap that the factory owner can configure, consistent with the existing `setFeeCaps` governance surface.

---

### Proof of Concept

1. Factory owner deploys a pool with `maxAdminSpreadFeeE6 = 200_000` (20 %).
2. Pool admin calls `factory.setPoolBinAdditionalFees(pool, 0, 65535, 65535)` — no revert.
3. A trader calls `swap(...)` routing through bin 0.
4. Inside `_swapToken1ForToken0SpecifiedInput`, the effective buy fee becomes `baseFeeX64 + mulDiv(65535, ONE_X64, 1e6)` ≈ `baseFeeX64 + 6.55 %`.
5. The trader pays ≈ 6.55 % more than the oracle spread alone would require; the excess is split between LPs (≈ 80 %) and admin (≈ 20 %) according to `spreadFeeE6`, but the trader's loss is the full 6.55 % regardless.
6. The global `HARD_MAX_SPREAD_FEE_E6 = 200_000` cap is never consulted for this path.
