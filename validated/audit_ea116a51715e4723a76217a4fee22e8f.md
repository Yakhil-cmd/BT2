Audit Report

## Title
Missing Reimbursement for `AmountTooLow` and `DustOutput` Finalized Withdrawal Requests Causes Permanent Loss of User ckBTC - (`rs/bitcoin/ckbtc/minter/src/lib.rs`)

## Summary

The ckBTC minter burns a user's ckBTC tokens at withdrawal request acceptance time. If Bitcoin network fees rise before the minter batches the request into a transaction, `build_unsigned_transaction` may return `BuildTxError::AmountTooLow` or `BuildTxError::DustOutput`. Both branches finalize the request with `FinalizedStatus::AmountTooLow` and call `remove_retrieve_btc_request` without scheduling any reimbursement, permanently destroying the user's ckBTC. The analogous `BuildTxError::InvalidTransaction` branch correctly calls `reimburse_canceled_requests`, confirming the reimbursement machinery exists but is simply absent for these two branches.

## Finding Description

**Burn occurs before queuing.** In `retrieve_btc.rs`, `burn_ckbtcs` is called at line 210 before the request is enqueued at line 232. From this point forward the user's ckBTC is gone regardless of what happens to the request.

**`AmountTooLow` branch — no reimbursement.** In `lib.rs` lines 412–434, when `build_unsigned_transaction` returns `BuildTxError::AmountTooLow`, the minter iterates over the batch and calls `state::audit::remove_retrieve_btc_request(s, request, FinalizedStatus::AmountTooLow, runtime)` for each request. No call to `reimburse_canceled_requests` or `schedule_withdrawal_reimbursement` is made.

**`DustOutput` branch — same omission.** In `lib.rs` lines 436–468, the specific request that produced the dust output is also finalized with `FinalizedStatus::AmountTooLow` via `remove_retrieve_btc_request`, again with no reimbursement.

**`remove_retrieve_btc_request` does not schedule reimbursement.** In `audit.rs` lines 67–84, the function only records a `RemovedRetrieveBtcRequest` event and pushes to `finalized_requests`. There is no call to `schedule_withdrawal_reimbursement` or any equivalent.

**`FinalizedStatus::AmountTooLow` is a dead end.** In `state.rs` lines 259–267, the `FinalizedStatus` enum has only `AmountTooLow` and `Confirmed { txid }` variants. There is no associated reimbursement account or amount stored, and the status query at lines 907–916 returns `RetrieveBtcStatus::AmountTooLow` permanently with no successor state.

**`WithdrawalReimbursementReason` has no `AmountTooLow` variant.** In `reimbursement/mod.rs` lines 39–43, the enum only contains `InvalidTransaction(InvalidTransactionError)`. There is no variant for fee-related failures, so even if `reimburse_canceled_requests` were called, a new reason variant would be needed.

**Contrast with `InvalidTransaction`.** In `lib.rs` lines 400–411, `BuildTxError::InvalidTransaction` correctly constructs a `WithdrawalReimbursementReason::InvalidTransaction` reason, computes a reimbursement fee, and calls `reimburse_canceled_requests`, which enqueues a `ScheduleWithdrawalReimbursement` event. The periodic `reimburse_withdrawals` task then mints ckBTC back to the user. This path is entirely absent for `AmountTooLow` and `DustOutput`.

## Impact Explanation

This matches the allowed impact: **High — Significant Chain Fusion / ck-token security impact with concrete user or protocol harm.** Any user whose pending withdrawal request spans a Bitcoin fee spike loses their ckBTC permanently: the tokens are burned, no BTC is sent, and no ckBTC is minted back. The loss is bounded only by the user's withdrawal amount, which can be arbitrarily large (the minimum is a protocol parameter, not a hard cap). Multiple users can be affected simultaneously during a single fee spike event.

## Likelihood Explanation

Bitcoin fee spikes are a recurring, externally-driven, unpredictable event (e.g., Ordinals/Runes inscription waves). The `retrieve_btc_min_amount` is set at canister initialization/upgrade time and does not dynamically track the fee market. Any user whose request sits in `pending_retrieve_btc_requests` across a fee spike that pushes fees above their withdrawal amount will trigger this path. No attacker action is required; this is a normal operational scenario triggered by the user's own valid withdrawal call followed by an external market event.

## Recommendation

Apply the same reimbursement pattern used for `BuildTxError::InvalidTransaction` to both the `AmountTooLow` and `DustOutput` branches:

1. Add a new `WithdrawalReimbursementReason` variant (e.g., `AmountTooLow`) to `reimbursement/mod.rs`.
2. In the `AmountTooLow` branch (`lib.rs` ~line 412), replace the bare `remove_retrieve_btc_request` loop with a call to `reimburse_canceled_requests(s, batch, reason, reimbursement_fee, runtime)`.
3. In the `DustOutput` branch (`lib.rs` ~line 436), apply the same pattern to the specific request being finalized before putting the rest back.
4. Ensure `reimburse_canceled_requests` calls `schedule_withdrawal_reimbursement` so the periodic `reimburse_withdrawals` task mints ckBTC back to `request.reimbursement_account`.

## Proof of Concept

1. User calls `retrieve_btc_with_approval` with amount `X` where `X >= retrieve_btc_min_amount`. `burn_ckbtcs` executes at `retrieve_btc.rs:210`; ckBTC is burned.
2. Request enters `pending_retrieve_btc_requests` via `accept_retrieve_btc_request`.
3. Bitcoin network fees spike such that the minter's fee estimator computes `fee > X`.
4. `submit_pending_requests` is called. `build_unsigned_transaction` returns `BuildTxError::AmountTooLow`.
5. The `AmountTooLow` branch at `lib.rs:412–434` executes: `remove_retrieve_btc_request(s, request, FinalizedStatus::AmountTooLow, runtime)` is called; no reimbursement is scheduled.
6. `retrieve_btc_status_v2(block_index)` returns `RetrieveBtcStatusV2::AmountTooLow` permanently (`state.rs:913`).
7. User's ckBTC is burned; no BTC was sent; no ckBTC is minted back. Funds are permanently lost.

A deterministic integration test can reproduce this by: initializing the minter with a low `retrieve_btc_min_amount`, submitting a withdrawal, then updating the fee estimator mock to return a fee exceeding the withdrawal amount before triggering `ProcessLogic`, and asserting that `pending_withdrawal_reimbursements` remains empty while the ledger burn is unmatched by any mint.
