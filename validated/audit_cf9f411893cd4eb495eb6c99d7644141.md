### Title
`PresignOutput` Not Consumed After Rerandomization Enables Presignature Reuse and Secret Key Extraction — (`src/ecdsa/ot_based_ecdsa/mod.rs`, `src/ecdsa/robust_ecdsa/mod.rs`)

---

### Summary

Both `ot_based_ecdsa::RerandomizedPresignOutput::rerandomize_presign` and `robust_ecdsa::RerandomizedPresignOutput::rerandomize_presign` accept the presignature by shared reference (`&PresignOutput`) rather than by value. This means the `PresignOutput` is never consumed after rerandomization. Any caller can invoke `rerandomize_presign` multiple times on the same `PresignOutput` with different `RerandomizationArguments`, producing multiple `RerandomizedPresignOutput` values from the same underlying nonce material. The library's own documentation confirms that signing different messages with shares derived from the same presignature allows full secret key recovery via standard ECDSA nonce-reuse attacks.

---

### Finding Description

**Root cause — OT-based ECDSA:**

```rust
// src/ecdsa/ot_based_ecdsa/mod.rs
pub fn rerandomize_presign(
    presignature: &PresignOutput,   // ← shared reference; never consumed
    args: &RerandomizationArguments,
) -> Result<Self, ProtocolError>
``` [1](#0-0) 

**Root cause — Robust ECDSA:**

```rust
// src/ecdsa/robust_ecdsa/mod.rs
pub fn rerandomize_presign(
    presignature: &PresignOutput,   // ← shared reference; never consumed
    args: &RerandomizationArguments,
) -> Result<Self, ProtocolError>
``` [2](#0-1) 

In both schemes, `PresignOutput` derives `Clone` and is passed by reference, so after `rerandomize_presign` returns, the caller still holds the original `PresignOutput` with its secret nonce shares (`k`/`sigma` for OT-based; `c`/`e`/`alpha`/`beta` for robust) fully intact. [3](#0-2) [4](#0-3) 

The presigning documentation explicitly marks this as a critical invariant:

> "it's crucial that a presignature is never reused" [5](#0-4) [6](#0-5) 

Yet the API provides no enforcement. Because `rerandomize_presign` does not take ownership, Rust's borrow checker never prevents a second call on the same `PresignOutput`.

**Exploit flow:**

A malicious participant (or a coordinator who also holds presignature shares) proceeds as follows:

1. Completes presigning normally, obtaining `presign_out: PresignOutput`.
2. Calls `rerandomize_presign(&presign_out, &args1)` → `rerand1` (message `h1`, tweak `ε1`, entropy `ρ1`).
3. Calls `rerandomize_presign(&presign_out, &args2)` → `rerand2` (message `h2 ≠ h1`, tweak `ε2`, entropy `ρ2`).
4. Participates in two signing sessions, one with `rerand1` and one with `rerand2`.
5. Collects the two output signatures `(r1, s1)` and `(r2, s2)`.

Both signatures share the same underlying nonce `k` (only scaled by different `delta` values derived from HKDF). The library's own security analysis confirms the consequence:

> "the resulting signatures use multiplicatively related nonces and the secret key can be recovered using standard ECDSA nonce-reuse attacks." [7](#0-6) 

For the robust ECDSA variant, the documentation further notes that the novel split-view attack (due to signature-share linearization) can extract the secret key with as few as two signing sessions: [8](#0-7) 

---

### Impact Explanation

**Critical — Extraction of private signing shares / aggregate secret key.**

A participant who reuses a `PresignOutput` across two signing sessions (with different `(h, ε)` pairs) produces two ECDSA signatures whose nonces are multiplicatively related. Standard algebraic nonce-reuse recovery then yields the aggregate secret key `x`, which is equivalent to reconstructing the threshold secret. This satisfies the allowed Critical impact: *"Extraction, reconstruction, or disclosure of private signing shares, aggregate secret material, presign secrets, nonce material, or confidential derived secrets."*

---

### Likelihood Explanation

**High.** The `rerandomize_presign` function is a public API entry point. Any participant who holds a `PresignOutput` — which is the normal output of a completed presigning round — can call it multiple times without restriction. No privilege beyond participation in presigning is required. The `PresignOutput` struct derives `Clone`, making accidental or deliberate reuse trivially easy. The library documents the danger but provides no compile-time or runtime guard.

---

### Recommendation

**Short term:** Change `rerandomize_presign` to consume the `PresignOutput` by value in both schemes:

```rust
// OT-based: src/ecdsa/ot_based_ecdsa/mod.rs
pub fn rerandomize_presign(
    presignature: PresignOutput,          // ← take by value; consumed on first call
    args: &RerandomizationArguments,
) -> Result<Self, ProtocolError>

// Robust: src/ecdsa/robust_ecdsa/mod.rs
pub fn rerandomize_presign(
    presignature: PresignOutput,          // ← take by value; consumed on first call
    args: &RerandomizationArguments,
) -> Result<Self, ProtocolError>
```

Rust's ownership system will then statically prevent any second call on the same `PresignOutput`, enforcing the one-time-use invariant at compile time.

**Long term:** Add runtime tracking (e.g., a `used` flag or a session-ID registry) at the orchestration layer to detect and reject duplicate presignature use across failed or retried sessions, where the same `PresignOutput` might be reconstructed from persistent storage.

### Proof of Concept

```rust
// Both OT-based and robust ECDSA are affected identically.
// Demonstrated here for OT-based ECDSA.

let presign_out: PresignOutput = /* result of presign protocol */;

// First rerandomization — different (msg_hash, tweak, entropy)
let rerand1 = RerandomizedPresignOutput::rerandomize_presign(
    &presign_out,   // ← presign_out is NOT consumed
    &args1,         // args1 = (pk, tweak1, h1, R, entropy1)
).unwrap();

// Second rerandomization — presign_out still alive, reused here
let rerand2 = RerandomizedPresignOutput::rerandomize_presign(
    &presign_out,   // ← same nonce material, different message
    &args2,         // args2 = (pk, tweak2, h2, R, entropy2)
).unwrap();

// Both rerand1 and rerand2 are now valid inputs to sign().
// Running two signing sessions produces (r1,s1) and (r2,s2)
// with multiplicatively related nonces:
//   r1 = x_coord(delta1 * k * G),  s1 = delta1*(h1 + r1*x)/k
//   r2 = x_coord(delta2 * k * G),  s2 = delta2*(h2 + r2*x)/k
// Standard nonce-reuse algebra recovers x (the aggregate secret key).
``` [9](#0-8) [10](#0-9)

impl RerandomizedPresignOutput {
    pub fn rerandomize_presign(
        presignature: &PresignOutput,
        args: &RerandomizationArguments,
    ) -> Result<Self, ProtocolError> {
        if presignature.big_r != args.big_r {
            return Err(ProtocolError::IncompatibleRerandomizationInputs);
        }
        let delta = args.derive_randomness()?;
        if delta.is_zero().into() {
            return Err(ProtocolError::ZeroScalar);
        }

        // cannot be zero due to the previous check
        let inv_delta = delta.invert().unwrap();

        // delta . R
        let rerandomized_big_r = presignature.big_r * delta;

        //  (sigma + tweak * k) * delta^{-1}
        let rerandomized_sigma =
            (presignature.sigma + args.tweak.value() * presignature.k) * inv_delta;

        // k * delta^{-1}
        let rerandomized_k = presignature.k * inv_delta;

        Ok(Self {
            big_r: rerandomized_big_r.into(),
            k: rerandomized_k,
            sigma: rerandomized_sigma,
        })
    }
