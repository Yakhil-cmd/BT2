### Title
Missing Zero-Address Validation in `change_reward_address` Allows Permanent Loss of Unclaimed Yield - (File: `src/staking/staking.cairo`, `src/pool/pool.cairo`)

---

### Summary

Both `change_reward_address` functions in `staking.cairo` and `pool.cairo` accept `Zero::zero()` as a valid `reward_address`. When rewards are subsequently claimed or forced out via `unstake_action`, the STRK transfer is directed to the zero address, permanently burning the staker's or pool member's unclaimed yield.

---

### Finding Description

`Staking::change_reward_address` (staking.cairo) validates only that the new address is not a registered token:

```cairo
assert!(
    !self.does_token_exist(token_address: reward_address),
    "{}",
    GenericError::REWARD_ADDRESS_IS_TOKEN,
);
```

There is no `reward_address.is_non_zero()` guard. [1](#0-0) 

`Pool::change_reward_address` (pool.cairo) validates only that the address is not the pool's token contract:

```cairo
assert!(
    self.token_dispatcher.contract_address.read() != reward_address,
    "{}",
    GenericError::REWARD_ADDRESS_IS_TOKEN,
);
```

Again, no zero-address guard. [2](#0-1) 

Once `reward_address` is set to zero, every downstream reward-transfer path sends STRK to address(0):

- `send_rewards_to_staker` → `token_dispatcher.checked_transfer(recipient: reward_address, ...)` [3](#0-2) 
- `Pool::claim_rewards` → `reward_token.checked_transfer(recipient: reward_address, ...)` [4](#0-3) 

`send_rewards_to_staker` is also called unconditionally inside `unstake_action`, which is callable by **any address** after the exit window expires: [5](#0-4) 

---

### Impact Explanation

**Scenario A – ERC20 allows transfer to zero (burn):**
Rewards are silently transferred to address(0) and permanently destroyed. This constitutes **permanent loss of unclaimed yield** (High).

**Scenario B – ERC20 reverts on transfer to zero:**
`claim_rewards` and `unstake_action` both revert. Because `unstake_action` is callable by anyone, a staker who set their reward address to zero and entered the exit window cannot complete their exit until they call `change_reward_address` again — but `change_reward_address` has no restriction during the exit window, so recovery is possible. This constitutes **temporary freezing of unclaimed yield** (High).

In Scenario A the loss is irreversible once `claim_rewards` or `unstake_action` executes.

---

### Likelihood Explanation

The entry path requires only that a staker or pool member call `change_reward_address(Zero::zero())`. This is an unprivileged, publicly reachable function. Accidental zero-address submission (e.g., a front-end bug, a scripting error, or a migration script that passes an uninitialized variable) is a realistic operational risk. The `unstake_action` callable-by-anyone property means a third party can trigger the irreversible burn once the exit window has elapsed.

---

### Recommendation

Add a non-zero guard in both `change_reward_address` implementations, mirroring the pattern already used in `add_token`:

```cairo
// staking.cairo and pool.cairo change_reward_address
assert!(reward_address.is_non_zero(), "{}", GenericError::ZERO_ADDRESS);
```

Apply the same guard to the initial `stake()` call's `reward_address` parameter for consistency. [6](#0-5) 

---

### Proof of Concept

**Staker path:**

1. Staker calls `staking.change_reward_address(Zero::zero())` — succeeds, no revert.
2. Staker calls `staking.unstake_intent()`.
3. After `exit_wait_window` (≥1 week), any address calls `staking.unstake_action(staker_address)`.
4. `unstake_action` calls `send_rewards_to_staker` → `checked_transfer(recipient: 0x0, amount: unclaimed_rewards)`.
5. Unclaimed STRK rewards are sent to address(0) and permanently lost.

**Pool member path:**

1. Pool member calls `pool.change_reward_address(Zero::zero())` — succeeds.
2. Pool member calls `pool.claim_rewards(pool_member)`.
3. `claim_rewards` calls `reward_token.checked_transfer(recipient: 0x0, amount: rewards)`.
4. Accumulated STRK rewards are sent to address(0) and permanently lost.
