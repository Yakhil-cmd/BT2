### Title
Malicious Coordinator Can Substitute Arbitrary `app_pk` to Extract the Confidential Derived Key — (`src/confidential_key_derivation/protocol.rs`)

---

### Summary

The CKD protocol accepts `app_pk` (the TEE application's ElGamal public key) as a plain, unvalidated caller-supplied parameter. Because the protocol contains no round in which participants independently verify or agree on `app_pk`, a malicious coordinator can substitute their own ElGamal key pair and use it to decrypt the aggregated ciphertext, recovering the confidential derived secret `msk · H(app_id)` in full.

---

### Finding Description

In `src/confidential_key_derivation/protocol.rs`, the public entry point `ckd()` accepts `app_pk: PublicKey` (a `blstrs::G1Projective` point) as a plain parameter with no validation: [1](#0-0) 

This value flows directly into `compute_signature_share()`, where it is used as the ElGamal encryption key for each participant's blinded signature share: [2](#0-1) 

The coordinator path (`do_ckd_coordinator`) and the participant path (`do_ckd_participant`) both consume `app_pk` from the same caller-supplied argument without any cross-participant broadcast or consistency check: [3](#0-2) 

In a real deployment the coordinator is the party that receives the TEE application's `(app_id, A)` request and then calls `ckd()` on behalf of every participant, supplying `app_pk` to each. There is no protocol round in which participants independently obtain or verify `app_pk` against an external registry. The coordinator is therefore free to pass an arbitrary point `A' = a'·G` (for a scalar `a'` they control) to every participant.

---

### Impact Explanation

When a malicious coordinator substitutes `app_pk' = a'·G`:

1. Each participant `i` computes:
   `big_c_i = x_i · H(pk ‖ app_id) + y_i · a'·G`

2. The coordinator aggregates (using Lagrange coefficients `λ_i`):
   `C = Σ λ_i · big_c_i = msk · H(pk ‖ app_id) + Y · a'`

3. Because the coordinator knows `a'`, they compute:
   `C − a' · Y = msk · H(pk ‖ app_id)`

This is the confidential derived secret that the protocol is designed to keep hidden from every MPC node. The wiki explicitly states the security goal is that "no single MPC node — or even the MPC network as a whole — ever learns the resulting secret." A single malicious coordinator violates this guarantee completely.

**Impact class**: Critical — Extraction, reconstruction, or disclosure of confidential derived secrets.

---

### Likelihood Explanation

The coordinator is the natural orchestrator of the CKD protocol: it receives the TEE application's request, selects the participant set, and calls `ckd()` for each node. Substituting `app_pk` requires no cryptographic break, no leaked key, and no external dependency — only the coordinator's normal operational role. Any compromised or malicious coordinator node can execute this attack silently, with no observable deviation from the honest protocol transcript visible to participants.

---

### Recommendation

`app_pk` must be independently authenticated by each participant before it is used in `compute_signature_share`. Concretely:

1. **External registry binding**: Require each participant to look up the TEE application's registered `app_pk` from an on-chain or otherwise authenticated source (e.g., the "Developer Contract" referenced in the CKD security model) rather than accepting it as a coordinator-supplied parameter.

2. **Protocol-level broadcast and agreement**: Add a broadcast round inside the CKD protocol where the coordinator publishes `app_pk` and all participants echo-broadcast their view; abort if any participant sees a different value. This mirrors the session-ID agreement already used in the DKG (`do_broadcast` in `src/dkg.rs`). [4](#0-3) 

---

### Proof of Concept

```
Setup:
  - Honest DKG produces shares x_1, …, x_n with master public key pk = msk·G₂.
  - TEE application generates (a, A = a·G₁) and sends (app_id, A) to the coordinator.

Attack (malicious coordinator):
  1. Generate a fresh scalar a' and compute A' = a'·G₁.
  2. Call ckd(participants, coordinator=self, me=self, key_pair, app_id, app_pk=A', rng)
     for every participant, supplying A' instead of A.
  3. Each participant i computes and sends:
       norm_big_c_i = λ_i · (x_i · H(pk ‖ app_id) + y_i · A')
  4. Coordinator aggregates:
       C = Σ norm_big_c_i = msk · H(pk ‖ app_id) + Y · a'
  5. Coordinator computes:
       secret = C − a' · Y = msk · H(pk ‖ app_id)   ← confidential derived key recovered.

The legitimate TEE application's private scalar a is never needed.
``` [5](#0-4)

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
