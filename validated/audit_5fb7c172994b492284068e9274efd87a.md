### Title
Unbounded `pool_member_epoch_balance` Trace Growth Causes Permanent Freezing of Unclaimed Yield - (File: src/pool/pool.cairo)

### Summary

The `calculate_rewards` function in `Pool` iterates over every balance-change checkpoint in a pool member's `pool_member_epoch_balance` trace since the last claim. Because a new checkpoint is appended each epoch a pool member modifies their balance, and there is no cap on trace length, a pool member who repeatedly calls `add_to_delegation_pool` across many epochs without claiming will eventually make `claim_rewards` exceed the Starknet transaction gas limit, permanently freezing their unclaimed yield.

### Finding Description

**Root cause — unbounded loop in `calculate_rewards`:**

The `calculate_rewards` internal function iterates over the `pool_member_epoch_balance` trace from `entry_to_claim_from` to the current epoch:

```cairo
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
    if pool_member_checkpoint.epoch() >= until_epoch {
        break;
    }
    ...
    entry_to_claim_from += 1;
}
``` [1](#0-0) 

The code itself acknowledges the loop is unbounded.

**How the trace grows — one new checkpoint per epoch:**

`set_member_balance` (called by both `add_to_delegation_pool` and `exit_delegation_pool_intent`) inserts a checkpoint keyed by `current_epoch + K`. The trace's `insert` logic only merges into the last entry when the key is identical; otherwise it appends a new entry:

```cairo
if last.key == key {
    last.value = value;
    checkpoints[len - 1].write(last);
} else {
    checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
}
``` [2](#0-1) 

Because the key advances by one each epoch, calling `add_to_delegation_pool` once per epoch appends exactly one new checkpoint per epoch.

**`entry_to_claim_from` is only advanced on a successful `claim_rewards`:**

```cairo
pool_member_info.entry_to_claim_from = updated_entry_to_claim_from;
pool_member_info.reward_checkpoint = until_checkpoint;
self.write_pool_member_info(:pool_member, :pool_member_info);
``` [3](#0-2) 

If the transaction reverts (gas exhaustion), the pointer is never advanced, so the pool member can never successfully claim.

**`pool_member_info_v1` (a public view) also calls `calculate_rewards`:**

```cairo
fn pool_member_info_v1(self: @ContractState, pool_member: ContractAddress) -> PoolMemberInfoV1 {
    let pool_member_info = self.internal_pool_member_info(:pool_member);
    let (rewards, _) = self.calculate_rewards(
        :pool_member,
        from_checkpoint: pool_member_info.reward_checkpoint,
        until_checkpoint: self.get_current_checkpoint(:pool_member),
        entry_to_claim_from: pool_member_info.entry_to_claim_from,
    );
``` [4](#0-3) 

This means even querying the pool member's info becomes prohibitively expensive once the trace is large.

**Exploit path:**

1. Attacker (unprivileged pool member) enters the delegation pool with a minimal amount via `enter_delegation_pool`.
2. Each epoch, the attacker calls `add_to_delegation_pool` with 1 wei. Each call appends one new checkpoint to `pool_member_epoch_balance`.
3. The attacker never calls `claim_rewards`, so `entry_to_claim_from` stays at 0.
4. After N epochs, the trace has N entries. When `claim_rewards` is finally called, the loop executes N iterations, each performing storage reads (`pool_member_trace.at(...)` and `find_sigma`).
5. Once N is large enough to exceed the Starknet per-transaction gas limit, `claim_rewards` always reverts. The pool member's accumulated yield is permanently frozen.

There is no partial-claim mechanism and no way to reset `entry_to_claim_from` without a successful `claim_rewards`.

### Impact Explanation

**Permanent freezing of unclaimed yield (High).**

A pool member who has accumulated a sufficiently large `pool_member_epoch_balance` trace can never successfully execute `claim_rewards`. All rewards accrued since the last successful claim are permanently inaccessible. The pool member's principal is unaffected (they can still exit via `exit_delegation_pool_intent` / `exit_delegation_pool_action`), but all unclaimed yield is frozen forever.

### Likelihood Explanation

Starknet epochs are short (on the order of hours based on block numbers). A pool member who participates actively for months without claiming — a realistic scenario for long-term delegators — can accumulate hundreds to thousands of checkpoints. The cost to grow the trace is minimal: 1 wei per epoch plus the gas cost of `add_to_delegation_pool`. The code comment at line 857–858 explicitly acknowledges the loop is unbounded and relies on the assumption that gas limits will not be exceeded in practice, which is not guaranteed over a long protocol lifetime.

### Recommendation

1. **Enforce a maximum trace length per pool member.** Reject `add_to_delegation_pool` / `exit_delegation_pool_intent` if the trace would exceed a safe bound (e.g., 500 entries).
2. **Introduce partial / paginated reward claiming.** Allow `claim_rewards` to accept an `until_epoch` or `max_entries` parameter so a pool member can drain the trace in multiple transactions.
3. **Compact the trace on each successful `claim_rewards`.** After advancing `entry_to_claim_from`, truncate or garbage-collect processed entries so the effective iteration window stays bounded.

### Proof of Concept

```
Epoch 0:  pool_member calls enter_delegation_pool(amount=MIN_STAKE)
           → pool_member_epoch_balance trace length = 1

Epoch 1:  pool_member calls add_to_delegation_pool(amount=1)
           → trace length = 2  (key = epoch 1 + K)

Epoch 2:  pool_member calls add_to_delegation_pool(amount=1)
           → trace length = 3  (key = epoch 2 + K)

...

Epoch N:  pool_member calls add_to_delegation_pool(amount=1)
           → trace length = N+1

Epoch N+1: pool_member calls claim_rewards
           → calculate_rewards loops N+1 times, each iteration reads
             pool_member_trace.at(i) (storage read) + find_sigma (storage read)
           → at sufficiently large N, transaction exceeds gas limit → REVERT
           → entry_to_claim_from is NOT updated
           → all subsequent claim_rewards calls also revert
           → pool member's yield is permanently frozen
```

The minimum cost to reach this state is `N × (gas(add_to_delegation_pool) + 1 wei token transfer)`, which is entirely within the reach of an unprivileged pool member over a normal protocol lifetime.
