### Title
Hardcoded `L1_TX_NATIVE_PRICE = 10` Placeholder Underpays Sequencer/Prover for L1→L2 Transactions - (`basic_bootloader/src/bootloader/constants.rs`)

### Summary

`L1_TX_NATIVE_PRICE` is a hardcoded constant set to `10` with an explicit `TODO` comment acknowledging it has not been properly calibrated. This constant governs how much native resource (proving budget) an L1→L2 transaction receives per unit of gas price. If this value is too low relative to the actual proving cost per native unit, users receive a disproportionately large native resource budget for the fee they pay, causing the sequencer/prover to be underpaid for processing L1→L2 transactions.

### Finding Description

In `basic_bootloader/src/bootloader/constants.rs`, the constant used as the native price for all L1→L2 transactions is:

```rust
// Default native price for L1->L2 transactions.
// TODO (EVM-1157): find a reasonable value for it.
pub const L1_TX_NATIVE_PRICE: U256 = U256::from_limbs([10, 0, 0, 0]);
``` [1](#0-0) 

This constant is consumed in `prepare_and_check_resources` inside `process_l1_transaction.rs` to derive `native_per_gas`:

```rust
let native_price = L1_TX_NATIVE_PRICE;
let native_per_gas = ... gas_price.div_ceil(native_price) ...;
let native_prepaid_from_gas = native_per_gas.checked_mul(gas_limit);
``` [2](#0-1) 

The `native_prepaid_from_gas` value becomes the user's total native resource budget. The native resource models the off-chain cost of proving a transaction — "how many RISC-V cycles it takes to prove a given computation." [3](#0-2) 

For L2 transactions, the operator sets `native_price` dynamically via block context and can calibrate it to reflect actual proving costs: [4](#0-3) 

For L1→L2 transactions, this is **hardcoded at 10** and cannot be adjusted by the operator. The `TODO (EVM-1157)` comment is an explicit acknowledgment that this value is a placeholder.

If the operator sets `native_price = N` for L2 transactions to reflect actual proving costs, but L1→L2 transactions use `L1_TX_NATIVE_PRICE = 10`, then L1→L2 transactions receive `N/10` times more native resource per unit of gas price than L2 transactions. For example, if `N = 1000` (a plausible production value), L1→L2 transactions receive 100× more native budget than they should.

The fee actually paid to the operator is `gas_used * gas_price`: [5](#0-4) 

The fee is denominated in gas, not native units. A user paying a low gas price can still receive a large native budget if `L1_TX_NATIVE_PRICE` is too low, allowing them to submit computationally expensive transactions (high proving cost) while paying a fee that does not cover the prover's actual cost.

### Impact Explanation

The sequencer/prover is underpaid for L1→L2 transactions. An attacker can submit L1→L2 transactions that are expensive to prove (consuming many native/RISC-V cycles) while paying a fee calibrated only to the miscalibrated `L1_TX_NATIVE_PRICE = 10`. If the actual proving cost per native unit is significantly higher than what this constant implies, the prover must spend more on proving than the fee collected. This is a direct financial loss to the sequencer/prover and can make the chain uneconomical, analogous to the external report's 100× underpayment scenario.

### Likelihood Explanation

The `TODO (EVM-1157)` comment is a direct admission that the value has not been properly calibrated. Any production deployment using this constant as-is is exposed. An unprivileged attacker needs only to submit L1→L2 transactions — a standard, permissionless operation — to trigger the underpayment. No special privileges, leaked keys, or oracle manipulation are required.

### Recommendation

1. Calibrate `L1_TX_NATIVE_PRICE` against actual measured proving costs (RISC-V cycles per unit of native resource) before production deployment.
2. Consider making this value operator-configurable (passed via block context like `native_price` for L2 transactions) to allow adjustment as proving costs change, while keeping a safe minimum floor to prevent censorship.
3. Resolve `TODO (EVM-1157)`.

### Proof of Concept

1. Operator deploys ZKsync OS with `L1_TX_NATIVE_PRICE = 10` (current hardcoded value) and sets `native_price = 1000` for L2 transactions to reflect actual proving costs.
2. Attacker submits an L1→L2 transaction with `gas_price = 100`, `gas_limit = 1_000_000`.
   - `native_per_gas = ceil(100 / 10) = 10`
   - `native_prepaid_from_gas = 10 * 1_000_000 = 10_000_000` native units
   - Fee paid: `gas_used * 100` (small, e.g., 21,000 * 100 = 2,100,000 wei)
3. For an equivalent L2 transaction with `gas_price = 100` and `native_price = 1000`:
   - `native_per_gas = ceil(100 / 1000) = 1`
   - `native_prepaid_from_gas = 1 * 1_000_000 = 1_000_000` native units
4. The L1→L2 transaction receives **10× more** native budget than the equivalent L2 transaction for the same fee. If the actual proving cost is proportional to native units consumed, the prover is underpaid by up to 10× (or more, depending on the true calibrated `native_price`).
5. The attacker repeats this indefinitely, forcing the prover to spend more on proving than it collects in fees.
