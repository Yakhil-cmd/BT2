### Title
Unprivileged Caller Can Trigger `deploy_upgrade` Without Owner Authorization, Eliminating Upgrade Cancellation Window - (File: `engine/src/lib.rs`)

---

### Summary

The Aurora Engine two-stage upgrade mechanism protects `stage_upgrade` with `require_owner_only`, but the sibling function `deploy_upgrade` — which actually replaces the live contract bytecode — carries **no caller authorization check**. Any NEAR account can call `deploy_upgrade` the moment the block-height delay expires, stripping the owner of the ability to cancel a staged upgrade before it goes live.

---

### Finding Description

The engine exposes two upgrade paths:

**Path A — `upgrade`** (`engine/src/contract_methods/admin.rs:169-206`): owner-only, immediate deployment via a NEAR promise batch.

**Path B — two-stage**: `stage_upgrade` → `deploy_upgrade`.

`stage_upgrade` is correctly gated:

```rust
// engine/src/contract_methods/admin.rs:154-167
pub fn stage_upgrade<I: IO + Copy, E: Env>(io: I, env: &E) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        let delay_block_height = env.block_height() + state.upgrade_delay_blocks;
        require_owner_only(&state, &env.predecessor_account_id())?;   // ← owner check
        io.read_input_and_store(&storage::bytes_to_key(KeyPrefix::Config, CODE_KEY));
        io.write_storage(
            &storage::bytes_to_key(KeyPrefix::Config, CODE_STAGE_KEY),
            &delay_block_height.to_le_bytes(),
        );
        Ok(())
    })
}
```

`deploy_upgrade`, which performs the actual `self_deploy`, is **not** gated:

```rust
// engine/src/lib.rs:171-185
pub extern "C" fn deploy_upgrade() {
    let mut io = Runtime;
    let state = state::get_state(&io).sdk_unwrap();
    require_running(&state)          // ← only liveness check
        .map_err(ContractError::msg)
        .sdk_unwrap();
    let index = internal_get_upgrade_index();
    if io.block_height() <= index {  // ← only time-delay check
        sdk::panic_utf8(errors::ERR_NOT_ALLOWED_TOO_EARLY);
    }
    Runtime::self_deploy(&bytes_to_key(KeyPrefix::Config, CODE_KEY));  // ← replaces bytecode
    io.remove_storage(&bytes_to_key(KeyPrefix::Config, CODE_STAGE_KEY));
}
```

There is no `require_owner_only` call and no `cancel_upgrade` function anywhere in the codebase. The only cancellation mechanism available to the owner is to call `stage_upgrade` again with replacement code, but an attacker who monitors the chain can race to call `deploy_upgrade` the instant the delay expires, before the owner's cancellation transaction is included.

The `upgrade_delay_blocks` field exists precisely to create a review window. Allowing any caller to close that window on demand defeats its purpose entirely.

---

### Impact Explanation

The attacker cannot inject arbitrary code — only the owner-staged WASM is deployed. The harm arises when:

1. The owner stages an upgrade.
2. A critical bug is discovered in the staged code (e.g., a storage-corruption path, a broken withdrawal route, or a missing access check) after staging but before the delay expires.
3. The owner attempts to cancel by re-staging safe code.
4. An attacker front-runs the cancellation by calling `deploy_upgrade` immediately after the delay, deploying the known-buggy bytecode.

If the deployed buggy code contains a fund-freezing or fund-draining vulnerability, the result is **permanent freezing of user funds** or **direct theft of user funds** — both Critical-scope impacts. Even if the bug is recoverable only via another upgrade cycle, users are exposed to the vulnerable code for the entire next upgrade window.

---

### Likelihood Explanation

- The owner must have staged an upgrade — a routine, expected operation during any protocol evolution.
- The attacker needs only to watch the chain for a `stage_upgrade` transaction, compute the activation block, and submit `deploy_upgrade` at that block. No privileged access, no tokens, no special role is required.
- NEAR block times are ~1 second; a bot can reliably land a transaction at the target block height.
- The cost to the attacker is a single NEAR transaction fee (fractions of a cent).

---

### Recommendation

Add `require_owner_only` to `deploy_upgrade`, mirroring the pattern used in `stage_upgrade` and in the direct `upgrade` path:

```rust
pub extern "C" fn deploy_upgrade() {
    let mut io = Runtime;
    let state = state::get_state(&io).sdk_unwrap();
    require_running(&state).map_err(ContractError::msg).sdk_unwrap();
    require_owner_only(&state, &io.predecessor_account_id())   // ← add this
        .map_err(ContractError::msg)
        .sdk_unwrap();
    let index = internal_get_upgrade_index();
    if io.block_height() <= index {
        sdk::panic_utf8(errors::ERR_NOT_ALLOWED_TOO_EARLY);
    }
    Runtime::self_deploy(&bytes_to_key(KeyPrefix::Config, CODE_KEY));
    io.remove_storage(&bytes_to_key(KeyPrefix::Config, CODE_STAGE_KEY));
}
```

Additionally, consider adding an explicit `cancel_upgrade` function (owner-only) that removes `CODE_KEY` and `CODE_STAGE_KEY` from storage, giving the owner a clean cancellation path independent of re-staging.

---

### Proof of Concept

1. Owner calls `stage_upgrade(buggy_wasm)`. `CODE_KEY` is written; `CODE_STAGE_KEY` is set to `current_block + upgrade_delay_blocks`.
2. Owner discovers a critical bug in `buggy_wasm` and prepares a `stage_upgrade(safe_wasm)` cancellation transaction.
3. Attacker monitors the chain. At block `CODE_STAGE_KEY`, the attacker submits `deploy_upgrade()` with no arguments and no special permissions.
4. `deploy_upgrade` passes both checks (`require_running` and block-height guard) and calls `Runtime::self_deploy` with `buggy_wasm`.
5. The Aurora Engine contract is now running `buggy_wasm`. The owner's cancellation transaction, if it lands in the same or a later block, has no effect — the buggy code is already live.
6. Any vulnerability present in `buggy_wasm` (fund freeze, unauthorized withdrawal, etc.) is now exploitable by any user.

pub fn require_owner_only(
    state: &state::EngineState,
    predecessor_account_id: &AccountId,
) -> Result<(), ContractError> {
    if &state.owner_id != predecessor_account_id {
        return Err(errors::ERR_NOT_ALLOWED.into());
    }
    Ok(())
