Audit Report

## Title
SNS Governance `disburse_maturity` Missing Neuron Lock Check Allows Maturity Extraction During Locked Async Operations - (File: rs/sns/governance/src/governance.rs)

## Summary
The SNS `disburse_maturity()` function mutates neuron state (`maturity_e8s_equivalent` and `disburse_maturity_in_progress`) without first checking `in_flight_commands`, bypassing the sole concurrency guard used throughout the codebase. If a neuron becomes permanently locked (e.g., due to a canister trap after an async ledger call in `disburse_neuron`), any maturity deducted by a concurrent `disburse_maturity` call will be permanently lost because the `finalize_disburse_maturity` timer unconditionally skips locked neurons.

## Finding Description
**Missing lock in SNS `disburse_maturity`:** The function at `rs/sns/governance/src/governance.rs` L1609–1706 reads and mutates the neuron directly with no call to `lock_neuron_for_command`. It checks only authorization and percentage bounds before deducting `maturity_e8s_equivalent` and appending to `disburse_maturity_in_progress`. [1](#0-0) [2](#0-1) 

**NNS counterpart acquires lock:** The NNS `disburse_maturity()` at `rs/nns/governance/src/governance.rs` L3091–3119 explicitly constructs a `SyncCommand` in-flight command and calls `lock_neuron_for_command` before any state mutation, with a comment stating this is to prevent disbursing maturity "in the middle of another ongoing operation." [3](#0-2) 

**`lock_neuron_for_command` is the sole concurrency guard:** It checks `in_flight_commands` and returns `NeuronLocked` if an entry exists, then inserts the command as a lock. [4](#0-3) 

**`finalize_disburse_maturity` timer skips locked neurons:** At L5006–5008, if `lock_neuron_for_command` fails, the timer issues `continue`, permanently skipping that neuron's pending disbursement for the current run. [5](#0-4) 

**Lock purpose documented in proto:** The `in_flight_commands` map is explicitly documented as the mechanism preventing concurrent state mutation during async ledger calls. [6](#0-5) 

**Exploit path:**
1. Attacker (neuron controller with `DisburseMaturity` permission) waits for any async neuron operation (e.g., `disburse_neuron`) to be in-flight — the neuron is locked and the canister is awaiting a ledger response.
2. Attacker submits a `manage_neuron` ingress with `DisburseMaturity`. This is processed synchronously in the same round or a subsequent round while the await is pending.
3. `disburse_maturity` bypasses the lock check, deducts `maturity_e8s_equivalent`, and appends to `disburse_maturity_in_progress`.
4. If the async operation's callback traps (e.g., resource exhaustion, bug), the neuron remains permanently locked per the documented behavior of `in_flight_commands`.
5. `finalize_disburse_maturity` timer skips the locked neuron on every run. The deducted maturity is never transferred — it is permanently lost.

## Impact Explanation
This is a **High** severity finding. The concrete impact is permanent loss of SNS neuron maturity (a financial asset) for any neuron holder whose neuron becomes permanently locked while a concurrent `disburse_maturity` call has already deducted maturity and enqueued a disbursement. This matches the allowed impact: "Significant SNS security impact with concrete user or protocol harm" and "Unauthorized access to neurons, governance assets... where exploitation requires meaningful per-target work or other constraints." The maturity loss is irreversible without a canister upgrade with custom reconciliation code.

## Likelihood Explanation
The entry path requires only `DisburseMaturity` permission, which any neuron controller or hot-key holder possesses — no special privileges are needed. The timing window exists whenever any async neuron operation (e.g., `disburse_neuron`, `split_neuron`) is awaiting a ledger response; on a loaded canister this window spans multiple rounds. The permanent lock condition (trap after ledger call) is an exceptional but documented and known-possible failure mode on the IC. An attacker cannot force the trap but can opportunistically exploit it when it occurs naturally or can attempt to induce it.

## Recommendation
Add a `lock_neuron_for_command` call at the start of SNS `disburse_maturity`, mirroring the NNS implementation:

```rust
pub fn disburse_maturity(
    &mut self,
    id: &NeuronId,
    caller: &PrincipalId,
    disburse_maturity: &DisburseMaturity,
) -> Result<DisburseMaturityResponse, GovernanceError> {
    let now_seconds = self.env.now();
    let in_flight_command = NeuronInFlightCommand {
        timestamp: now_seconds,
        command: Some(neuron_in_flight_command::Command::SyncCommand(SyncCommand {})),
    };
    let _neuron_lock = self.lock_neuron_for_command(id, in_flight_command)?;
    // ... rest of function unchanged ...
}
```

This ensures `disburse_maturity` returns `NeuronLocked` if any async operation is in progress, consistent with all other SNS neuron mutation operations and the NNS counterpart.

## Proof of Concept
A deterministic PocketIC integration test can reproduce this:
1. Create an SNS governance canister with a neuron that has maturity.
2. Initiate `disburse_neuron` — intercept the ledger call so the canister is awaiting a response (neuron is locked).
3. Submit `disburse_maturity` for the same neuron — confirm it succeeds and deducts maturity (lock bypass confirmed).
4. Simulate a trap in the `disburse_neuron` callback (e.g., by having the mock ledger return an error that causes a panic in the callback handler, or by upgrading the canister mid-flight to simulate a trap).
5. Advance time past `MATURITY_DISBURSEMENT_DELAY_SECONDS` and run the `finalize_disburse_maturity` timer.
6. Assert the neuron remains locked, the timer skips it, and the maturity entry in `disburse_maturity_in_progress` is never cleared — confirming permanent maturity loss.

        &mut self,
        nid: &NeuronId,
        command: NeuronInFlightCommand,

        let nid = nid.to_string();
        if self.proto.in_flight_commands.contains_key(&nid) {

                ErrorType::NeuronLocked,
                "Neuron has an ongoing operation.",
