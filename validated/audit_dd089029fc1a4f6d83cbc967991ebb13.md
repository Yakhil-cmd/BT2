### Title
Single Malicious Participant Causes Permanent Hang of All Honest Parties in DKG/Reshare/Refresh via Invalid Secret Share with No Fallback - (File: src/dkg.rs)

### Summary
In `do_keyshare`, when `validate_received_share` fails for any single participant's secret share, the protocol immediately aborts via `?` with no fallback mechanism. The aborting participant never reaches `broadcast_success`, leaving all other honest participants looping indefinitely inside `reliable_broadcast_receive_all` waiting for a broadcast that never arrives. A single malicious participant can exploit this to permanently deny DKG, reshare, or refresh to all honest parties.

### Finding Description
In `do_keyshare` (Round 5, `src/dkg.rs` lines 514–528), each participant iterates over received secret shares and validates each one:

```rust
for (from, signing_share_from) in
    recv_from_others(&chan, wait_round_3, &participants, me).await?
{
    let full_commitment_from = all_full_commitments.index(from)?;
    validate_received_share::<C>(me, from, &signing_share_from, full_commitment_from)?;
    my_signing_share = my_signing_share + signing_share_from.to_scalar();
}
```

If `validate_received_share` returns `ProtocolError::InvalidSecretShare(from)`, the `?` operator immediately propagates the error and the participant exits `do_keyshare` without ever reaching:

```rust
broadcast_success(&mut chan, &participants, me, session_id).await?;
```

`broadcast_success` calls `do_broadcast` → `reliable_broadcast_receive_all`, which loops indefinitely until all N sessions have `finish_ready = true`:

```rust
if state.iter().all(|x| x.finish_ready) {
    return Ok(vote_output);
}
```

If participant A aborts before broadcasting, the session slot for A never reaches `finish_ready = true` in any other participant's state. The loop has no timeout and no early-exit path for a missing sender. All other honest participants hang permanently.

The same abort-without-notification pattern exists in Round 4 at `verify_proof_of_knowledge` (lines 452–460) and `verify_commitment_hash` (lines 463–469), but the Round 5 path is more dangerous because the malicious participant can selectively target a single victim while appearing honest to all others.

There is no fallback mechanism analogous to the external report's suggestion: no secondary path, no broadcast of failure status, no mechanism to exclude the identified malicious participant and continue with the remaining honest ones.

### Impact Explanation
**High — Permanent denial of key generation, reshare, and refresh for all honest parties.**

A single malicious participant M can:
1. Participate honestly through Rounds 1–4 (valid commitments, valid proofs, valid commitment hashes)
2. In Round 5, send a deliberately invalid secret share to exactly one honest participant A
3. Send valid shares to all other participants B, C, D…

Result: A aborts silently. B, C, D proceed to `broadcast_success` and loop forever waiting for A's broadcast. The DKG session is permanently unrecoverable. M can repeat this against any new session, indefinitely blocking key generation, reshare, or refresh.

### Likelihood Explanation
Any participant enrolled in the DKG/reshare/refresh session can execute this attack. No special privilege is required beyond being a listed participant. The attack requires only crafting one invalid scalar share for one target participant — trivially achievable. The malicious participant is identified in the error (`InvalidSecretShare(M)`) but only by the victim; no other participant learns who caused the abort or that an abort occurred.

### Recommendation
Implement a fallback mechanism analogous to the external report's suggestion:

1. **Broadcast failure before aborting**: When `validate_received_share` fails, the victim should broadcast a failure signal (e.g., `(false, session_id, culprit)`) to all other participants before returning the error, so they can abort cleanly rather than hang.

2. **Modify `broadcast_success` to accept an error flag**: The function comment already says "If err is None then broadcast success, otherwise, broadcast failure" but the implementation always broadcasts `true`. Restore the intended behavior so participants can signal failure.

3. **Graceful abort on received failure**: In `reliable_broadcast_receive_all` / `broadcast_success`, if any participant broadcasts `false`, all honest parties should abort immediately rather than waiting for a broadcast that will never arrive.

### Proof of Concept

**Setup**: N=4 participants P1, P2, P3, P4 (malicious = P4), threshold=2.

**Steps**:
1. P4 participates honestly through Rounds 1–4: sends valid `commitment_hash`, valid `(commitment, proof_of_knowledge)`.
2. In Round 5 (`wait_round_3`), P4 sends:
   - Valid `signing_share` to P2, P3
   - Invalid `signing_share` (e.g., random scalar that fails `SecretShare::verify()`) to P1
3. P1 receives P4's invalid share, calls `validate_received_share` → `ProtocolError::InvalidSecretShare(P4)` → exits `do_keyshare` via `?` at line 522.
4. P1 **never** calls `broadcast_success`.
5. P2, P3, P4 complete Round 5 validation successfully and call `broadcast_success` → `do_broadcast` → `reliable_broadcast_receive_all`.
6. Inside `reliable_broadcast_receive_all`, the session slot for P1 (`sid = participants.index(P1)`) never receives a `Ready` message with count `> 2 * ready_t`.
7. `state.iter().all(|x| x.finish_ready)` is never `true`.
8. P2, P3, P4 loop forever — **permanent hang**.

fn validate_received_share<C: Ciphersuite>(
    me: Participant,
    from: Participant,
    signing_share_from: &SigningShare<C>,
    commitment: &VerifiableSecretSharingCommitment<C>,
) -> Result<(), ProtocolError> {
    let id = me.to_identifier::<C>()?;

    // The verification is exactly the same as the regular SecretShare verification;
    // however the required components are in different places.
    // Build a temporary SecretShare so what we can call verify().
    let secret_share = SecretShare::new(id, *signing_share_from, commitment.clone());

    // Verify the share. We don't need the result.
    // Identify the culprit if an InvalidSecretShare error is returned.
    secret_share.verify().map_err(|e| {
        if let Error::InvalidSecretShare { .. } = e {
            ProtocolError::InvalidSecretShare(from)
        } else {
            ProtocolError::AssertionFailed(format!(
                "could not
            extract the verification key matching the secret
            share sent by {from:?}"
            ))
        }
    })?;
    Ok(())
}

    // unwrap here would never fail as the broadcast protocol ends only when the map is full

        ));

    Ok(())
