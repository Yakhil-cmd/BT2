### Title
Unprivileged Caller Can Permanently Suppress Block Rewards via `update_rewards(disable_rewards: true)` — (File: `src/staking/staking.cairo`)

---

### Summary

`update_rewards` in the Staking contract accepts a caller-controlled `disable_rewards: bool` parameter with no access control beyond a non-zero caller check. Any unprivileged address can call `update_rewards(active_staker, disable_rewards: true)` at every block to consume the global `last_reward_block` slot without distributing rewards, permanently preventing all consensus-phase block rewards from reaching stakers and their delegators.

---

### Finding Description

`update_rewards` is part of `IStakingRewardsManager` and is the entry point for distributing per-block consensus rewards. Its only gate is `general_prerequisites()`, which checks that the contract is unpaused and the caller is non-zero — no role, no operational-address check, no staker-ownership check. [1](#0-0) 

Inside the function, `last_reward_block` is written unconditionally before the `disable_rewards` branch is evaluated: [2](#0-1) 

When `disable_rewards` is `true`, the function returns immediately after writing `last_reward_block`, distributing nothing: [3](#0-2) 

Because `last_reward_block` is a **global** storage slot shared across all stakers, any subsequent legitimate call to `update_rewards` for the same block number is rejected: [4](#0-3) 

The attacker only needs to supply any currently-active staker address (publicly observable from `NewStaker` events) and pass the two staker-validity assertions: [5](#0-4) 

---

### Impact Explanation

**High — Permanent freezing of unclaimed yield.**

By calling `update_rewards(any_active_staker, disable_rewards: true)` once per block, an attacker can ensure that no consensus-phase block rewards are ever distributed to any staker or delegation-pool member. The rewards for each suppressed block are permanently unclaimable; there is no recovery path because `last_reward_block` cannot be rewound.

---

### Likelihood Explanation

**High.** The call requires no tokens, no privileged role, and no special state — only a non-zero caller address and a valid active staker address (trivially obtained from on-chain events). The gas cost per block is minimal. A motivated attacker (e.g., a competing validator or a griefing party) can sustain this indefinitely.

---

### Recommendation

Restrict `update_rewards` to callers that are authorized to report block participation. The most direct fix is to require the caller to be the staker's registered operational address (or the attestation/consensus contract), mirroring the access control already applied to `update_rewards_from_attestation_contract`: [6](#0-5) 

Alternatively, remove the `disable_rewards` parameter from the public interface entirely and handle the "no-reward block" case internally based on verifiable on-chain consensus data.

---

### Proof of Concept

```
// Attacker script — runs once per block in the consensus phase:
// 1. Read any active staker address S from NewStaker events.
// 2. At block B (before any legitimate update_rewards call):
staking.update_rewards(staker_address: S, disable_rewards: true);
// Effect: last_reward_block = B, no rewards distributed.
// 3. Any legitimate call staking.update_rewards(S2, false) at block B now reverts:
//    "Rewards already updated" (last_reward_block == B).
// 4. Repeat at block B+1, B+2, … to suppress all consensus rewards indefinitely.
```

The attacker pays only gas. All stakers and pool members permanently lose their per-block yield for every suppressed block.
