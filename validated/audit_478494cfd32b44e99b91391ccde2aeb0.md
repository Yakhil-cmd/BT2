### Title
`setPoolAdminFeeDestination` Skips Pre-Collection Unlike Its Sibling `setPoolAdminFees`, Allowing Accrued Admin Fees to Be Silently Redirected - (File: metric-core/contracts/MetricOmmPoolFactory.sol)

---

### Summary

`setPoolAdminFees()` enforces a `collectFees()` call at the old destination before updating fee rates. Its sibling `setPoolAdminFeeDestination()` makes an equally consequential state change — replacing the address that receives all future admin fee payouts — but performs no prior collection. A pool admin can atomically redirect all accrued-but-uncollected admin fees to an address of their choosing.

---

### Finding Description

Two factory functions govern the admin fee configuration for a pool:

**`setPoolAdminFees()`** — enforces collection first: [1](#0-0) 

It calls `collectFees()` with the **current** `poolAdminFeeDestination[pool]` before writing new rates. This ensures all surplus and notional fees accrued under the old rate split are swept to the old destination before the split changes.

**`setPoolAdminFeeDestination()`** — no collection: [2](#0-1) 

It overwrites `poolAdminFeeDestination[pool]` immediately with no prior sweep. Every subsequent `collectPoolFees()` call — which reads `poolAdminFeeDestination[pool]` at call time — will send the admin share to the **new** address, including all fees that accrued while the old address was set.

`collectFees()` on the pool computes the admin share from the live surplus and `notionalFeeToken{0,1}Scaled` at the moment of the call, using whichever `adminFeeDestination_` the factory passes in: [3](#0-2) 

`collectPoolFees()` (permissionless) always reads the current `poolAdminFeeDestination[pool]`: [4](#0-3) 

There is no snapshot, per-epoch accumulator, or checkpoint that ties accrued fees to the destination that was active when they were earned. The destination at collection time wins entirely.

---

### Impact Explanation

The pool admin can redirect all uncollected admin fees — both spread-surplus fees and notional fees tracked in `notionalFeeToken0Scaled` / `notionalFeeToken1Scaled` — from the old destination to any new address in a single transaction. If the admin fee destination is a separate entity (e.g., a revenue-sharing partner, a DAO treasury, or a co-admin), that entity loses all fees that accrued before the destination change. The loss is bounded only by how long fees have been accumulating since the last collection.

The broken invariant: *fees accrued while destination D was active should be paid to D*. `setPoolAdminFees()` upholds this invariant for rate changes; `setPoolAdminFeeDestination()` silently violates it for destination changes.

---

### Likelihood Explanation

The pool admin is a semi-trusted role. The trigger requires only a single call to `setPoolAdminFeeDestination()` — no cap bypass, no timelock, no special precondition. The admin can execute this atomically (e.g., via a multicall or private relay) before any third party can front-run with `collectPoolFees()`. The scenario is realistic whenever the admin fee destination is a party distinct from the admin key itself.

---

### Recommendation

Mirror the pattern used in `setPoolAdminFees()`: call `collectFees()` with the **current** `poolAdminFeeDestination[pool]` before overwriting it.

```solidity
function setPoolAdminFeeDestination(address pool, address newAdminFeeDestination)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (newAdminFeeDestination == address(0)) revert InvalidAdminFeeDestination();

    // Sweep accrued fees to the OLD destination before changing it
    PoolFeeConfig memory c = poolFeeConfig[pool];
    IMetricOmmPoolCollectFees(pool).collectFees(
        c.protocolSpreadFeeE6,
        c.adminSpreadFeeE6,
        c.protocolNotionalFeeE8,
        c.adminNotionalFeeE8,
        poolAdminFeeDestination[pool]   // old destination
    );

    poolAdminFeeDestination[pool] = newAdminFeeDestination;
    emit PoolAdminFeeDestinationUpdated(pool, newAdminFeeDestination);
}
```

---

### Proof of Concept

1. Pool accumulates 10 000 USDC of admin spread fees and 500 USDC of notional fees over several weeks. `poolAdminFeeDestination[pool]` = `oldTreasury`.
2. Admin calls `setPoolAdminFeeDestination(pool, attackerWallet)`. No collection occurs; `poolAdminFeeDestination[pool]` is now `attackerWallet`.
3. Admin (or anyone) calls `collectPoolFees(pool)`. The factory passes `attackerWallet` as `adminFeeDestination_` to `collectFees()`.
4. `collectFees()` computes the full surplus and notional fee balances and transfers the admin share to `attackerWallet`.
5. `oldTreasury` receives nothing despite having been the designated recipient for all of the accrued period.

The contrast with `setPoolAdminFees()` is exact: had the admin called `setPoolAdminFees()` instead, `collectFees()` would have been invoked with `oldTreasury` first, sweeping the accrued balance there before any rate or destination change took effect. [5](#0-4) [2](#0-1)
