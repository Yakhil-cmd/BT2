### Title
Gateway Admits Unsigned Invoke Transactions by Exploiting Pre-Existing Mempool State to Bypass `__validate__` - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator skips the `__validate__` entry-point call (signature verification) for any invoke transaction with `nonce=1` sent from an account whose on-chain nonce is `0`, provided `account_tx_in_pool_or_recent_block` returns `true` for that sender. Because `account_tx_in_pool_or_recent_block` returns `true` for **any** account that has **any** transaction in the pool or in a recently committed block — not specifically a `deploy_account` — an attacker can exploit a victim account's legitimately pending `deploy_account` transaction to inject an unsigned/arbitrarily-signed invoke transaction with `nonce=1` from that account into the mempool without any signature check.

---

### Finding Description

The UX feature is documented as:

> "It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations." [1](#0-0) 

The guard used is:

```rust
return mempool_client
    .account_tx_in_pool_or_recent_block(tx.sender_address())
    .await
``` [2](#0-1) 

`account_tx_in_pool_or_recent_block` is implemented as:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [3](#0-2) 

`state.contains_account` returns `true` if the address appears in either `staged` (transactions currently being batched) or `committed` (accounts seen in any recently committed block):

```rust
fn contains_account(&self, address: ContractAddress) -> bool {
    self.staged.contains_key(&address) || self.committed.contains_key(&address)
}
``` [4](#0-3) 

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` sets `execution_flags.validate = false`:

```rust
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [5](#0-4) 

Inside `StatefulValidator::perform_validations`, when `validate=false` the entire `__validate__` call is skipped and the function returns `Ok(())` immediately:

```rust
if !tx.execution_flags.validate {
    return Ok(());
}
// `__validate__` call.
let (_optional_call_info, actual_cost) = self.validate(&tx, tx_context.clone())?;
``` [6](#0-5) 

**Attack path:**

1. Victim account `A` (on-chain nonce = 0, not yet deployed) submits a `deploy_account` transaction. It enters the mempool pool; `tx_pool.contains_account(A)` becomes `true`.
2. Attacker observes the pending `deploy_account` and immediately submits an `InvokeV3` transaction with `sender_address=A`, `nonce=1`, and an **arbitrary or missing signature**.
3. Gateway stateful validation:
   - `account_nonce` read from chain state = `0` ✓
   - `tx.nonce() == Nonce(ONE)` ✓
   - `account_tx_in_pool_or_recent_block(A)` = `true` (victim's `deploy_account` is in the pool) ✓
   - `skip_stateful_validations` returns `true`
4. `run_validate_entry_point` is called with `validate=false`; `__validate__` is never invoked. The transaction passes gateway validation with no signature check.
5. `validate_by_mempool` (called before the skip check) only verifies nonce ordering (`tx_nonce=1 >= account_nonce=0`), which passes. [7](#0-6) 

6. The attacker's unsigned invoke is admitted to the mempool alongside the victim's `deploy_account`.

The reasoning in the comment is circular: it claims the check is sufficient because the account "either has a deploy_account or transactions with future nonces that passed validations." But the attacker's nonce-1 invoke is itself the transaction that exploits the check — it passes the mempool nonce check trivially (nonce 1 ≥ 0) and then uses the victim's `deploy_account` presence to skip its own signature verification.

---

### Impact Explanation

This matches **High: Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

- The gateway admits an invoke transaction with an invalid or absent signature into the mempool, violating the invariant that every admitted transaction has passed `__validate__`.
- The attacker can front-run any `deploy_account` submission and inject a nonce-1 invoke for that account. If the victim's `deploy_account` and the attacker's invoke are both sequenced in the same block, the attacker's invoke will fail at execution time (because `__validate__` runs during actual execution and will reject the bad signature), but it still occupies a mempool slot and a block slot, displacing legitimate transactions.
- Additionally, `account_tx_in_pool_or_recent_block` also returns `true` for accounts in `state.committed` (accounts seen in any recent block). An account that had a transaction committed in a recent block but whose on-chain nonce is still 0 (e.g., due to a reorg or a race between the mempool's commit history and the chain state reader) would also be exploitable.

---

### Likelihood Explanation

The attack requires only:
1. Monitoring the public mempool for pending `deploy_account` transactions (trivially observable).
2. Submitting an invoke with `nonce=1` from the same sender address before the `deploy_account` is committed.

No privileged access, no specialized knowledge, and no cryptographic capability is required. The window is the entire time the `deploy_account` sits in the mempool (potentially many blocks).

---

### Recommendation

`skip_stateful_validations` must verify that the transaction present in the mempool for the account is specifically a `deploy_account` transaction, not just any transaction. The `account_tx_in_pool_or_recent_block` API is too coarse for this purpose.

Concretely, the mempool should expose a dedicated `has_pending_deploy_account(address)` query that checks only the `tx_pool` for a `DeployAccount` transaction at `nonce=0` for that address. The skip should only fire when that stricter check passes.

Alternatively, the gateway can verify the transaction type of the pending transaction before granting the skip, or require the caller to supply the `deploy_account` transaction hash (as the `native_blockifier` path already does via `deploy_account_tx_hash`) and verify it is present in the mempool. [8](#0-7)

### Proof of Concept

```
State:
  - Account A: on-chain nonce = 0 (not deployed)

Step 1 – Victim submits deploy_account:
  POST /gateway/add_transaction
  { type: "DEPLOY_ACCOUNT", sender: A, nonce: 0, signature: <valid> }
  → Mempool: tx_pool.contains_account(A) = true

Step 2 – Attacker submits unsigned invoke:
  POST /gateway/add_transaction
  { type: "INVOKE", sender: A, nonce: 1, calldata: <arbitrary>, signature: [] }

Step 3 – Gateway stateful validation (skip_stateful_validations):
  account_nonce (from chain) = 0
  tx.nonce() = 1  → condition met
  account_tx_in_pool_or_recent_block(A) = true  → skip = true

Step 4 – run_validate_entry_point called with validate=false:
  StatefulValidator::perform_validations returns Ok(()) without calling __validate__

Step 5 – validate_by_mempool:
  Mempool::validate_tx: tx_nonce(1) >= resolve_nonce(A, 0) = 0  → Ok

Result:
  Attacker's unsigned invoke with nonce=1 is admitted to the mempool.
  Both transactions are sequenced; deploy_account succeeds, attacker's invoke
  fails at __validate__ during execution — but the invalid transaction was
  admitted and consumed a block slot.
``` [9](#0-8) [3](#0-2) [10](#0-9)

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}

    pub fn should_run_stateful_validations(
        &mut self,
        account_tx: &AccountTransaction,
        deploy_account_tx_hash: Option<TransactionHash>,
    ) -> StatefulValidatorResult<bool> {
        if account_tx.tx_type() != TransactionType::InvokeFunction {
            return Ok(true);
        }
        let tx_info = account_tx.create_tx_info();
        let nonce = self.stateful_validator.get_nonce(tx_info.sender_address())?;

        let deploy_account_not_processed =
            deploy_account_tx_hash.is_some() && nonce == Nonce(Felt::ZERO);
        let tx_nonce = tx_info.nonce();
        let is_post_deploy_nonce = Nonce(Felt::ONE) <= tx_nonce;
        let nonce_small_enough_to_qualify_for_validation_skip =
            tx_nonce <= self.max_nonce_for_validation_skip;

        let skip_validate = deploy_account_not_processed
            && is_post_deploy_nonce
            && nonce_small_enough_to_qualify_for_validation_skip;

        Ok(!skip_validate)
    }
