### Title
Pool Admin Bypasses Fee Cap via Uncapped `setPoolBinAdditionalFees` — (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary

`setPoolBinAdditionalFees` forwards `addFeeBuyE6` / `addFeeSellE6` to the pool with **no cap validation**, while the analogous base-fee setter `setPoolAdminFees` enforces `maxAdminSpreadFeeE6`. A pool admin can set per-bin additional fees to `type(uint16).max` (65 535 in E6 = 6.5535 %) on every bin, bypassing the factory's fee-cap system and extracting more from traders than the protocol intends to allow. The factory owner has no mechanism to override per-bin additional fees after the fact.

---

### Finding Description

`setPoolAdminFees` enforces the cap on the base admin spread fee:

```solidity
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
``` [1](#0-0) 

`setPoolBinAdditionalFees`, however, passes the caller-supplied values straight through to the pool with **no cap check**:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
  external override nonReentrant onlyPoolAdmin(pool)
{
  IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
``` [2](#0-1) 

The pool's `setBinAdditionalFees` also performs no cap validation — it only checks the bin index range: [3](#0-2) 

During every swap, the per-bin additional fee is added on top of the oracle-derived `baseFeeX64` in all four swap paths:

```solidity
params.baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6)
``` [4](#0-3) [5](#0-4) [6](#0-5) 

The `baseFeeX64` is derived from the oracle bid/ask spread, not from `spreadFeeE6`:

```solidity
baseFeeX64 = Math.mulDiv(askPriceX64, ONE_X64, midPriceX64, Math.Rounding.Ceil) - ONE_X64;
``` [7](#0-6) 

So the per-bin additional fee is a **separate, additive, uncapped** component on top of the oracle spread. The pool admin can set it to `type(uint16).max` = 65 535 (6.5535 % in E6) for every bin. The factory owner's `setPoolProtocolFee` clamps the base admin spread fee stored in `poolFeeConfig` but has **no path** to override `BinState.addFeeBuyE6` / `addFeeSellE6`.

The analog to the external bug is exact: the cap check is enforced only at the time `setPoolAdminFees` is called (initial or subsequent base-fee setting), but the per-bin additional fee path — which also charges traders — is never validated against any cap, either at creation time or at update time.

---

### Impact Explanation

Every swap through an affected bin charges the trader `baseFeeX64 + addFeeBuyE6/1e6` instead of the capped `baseFeeX64`. The excess fee is real token value extracted from the trader: it increases `amountInScaled` (the token the trader must pay) and is split between LPs and the protocol/admin via `spreadFeeE6`. This is a direct loss of user funds above the protocol-intended maximum, constituting an admin-boundary break where the pool admin exceeds the fee caps the factory owner set.

---

### Likelihood Explanation

The pool admin is semi-trusted and has a direct, single-call path (`setPoolBinAdditionalFees`) to set any `uint16` value with no on-chain guard. No timelock, no factory-owner co-signature, and no subsequent override mechanism exists. Any pool admin who wishes to extract above-cap fees can do so immediately after pool creation.

---

### Recommendation

Add a cap check in `setPoolBinAdditionalFees` before forwarding to the pool:

```solidity
if (addFeeBuyE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
```

Alternatively, enforce the check inside `setBinAdditionalFees` on the pool (passing the cap as a parameter from the factory), or introduce a dedicated `maxAdminBinAdditionalFeeE6` cap that the factory owner can set and that is enforced on every `setPoolBinAdditionalFees` call.

---

### Proof of Concept

1. Factory is deployed with `maxAdminSpreadFeeE6 = 5_000` (0.5 %).
2. Pool admin calls `factory.setPoolAdminFees(pool, 5_000, 0)` — succeeds, at the cap.
3. Pool admin calls `factory.setPoolBinAdditionalFees(pool, 0, 65_535, 65_535)` — **no revert**.
4. A trader swaps through bin 0. The effective buy fee is `baseFeeX64 + 65535/1e6 * ONE_X64`, i.e. the oracle spread plus an additional 6.5535 %, far above the 0.5 % admin cap.
5. The factory owner calls `setPoolProtocolFee` — this clamps `poolFeeConfig.adminSpreadFeeE6` but leaves `BinState.addFeeBuyE6 = 65_535` untouched; traders continue to pay the above-cap fee. [8](#0-7) [2](#0-1) [3](#0-2)
