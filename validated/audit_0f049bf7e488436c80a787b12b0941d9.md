### Title
Unprivileged Caller Can Permanently Freeze Consensus Block Rewards via `update_rewards(disable_rewards: true)` — (File: `src/staking/staking.cairo`)

---

### Summary

`IStakingRewardsManager::update_rewards` carries no access-control guard. Any address can call it with `disable_rewards: true`, which advances the **global** `last_reward_block` to the current block without distributing any rewards. Because the guard `current_block_number > last_reward_block` is then satisfied only in the *next* block, an attacker who repeats this call every block permanently prevents all stakers from ever receiving consensus block rewards.

---

### Finding Description

`update_rewards` is exposed as a public entrypoint in `StakingRewardsManagerImpl`:

```
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();          // only checks pause flag
    let current_block_number = starknet::get_block_number();
    assert!(
        current_block_number > self.last_reward_block.read(),
        "{}",
        Error::REWARDS_ALREADY_UPDATED,
    );
    ...
    self.last_reward_block.write(current_block_number);   // ← always written

    if disable_rewards || self.is_pre_consensus() {
        return;                                           // ← exits before distributing
    }
    // reward distribution happens here
``` [1](#0-0) 

`last_reward_block` is a **single global** storage slot, not per-staker:

```
/// Last block number for which rewards were distributed.
last_reward_block: BlockNumber,
``` [2](#0-1) 

The function accepts any `staker_address` that is currently active. Active staker addresses are publicly readable from on-chain events and view functions, so the attacker can always supply a valid one.

**Root cause**: No caller restriction on `update_rewards`. The `disable_rewards` flag was designed for the system to suppress rewards in certain conditions, but because the function is callable by anyone, an adversary can supply `disable_rewards: true` to consume the per-block "slot" without paying out rewards.

---

### Impact Explanation

**High — Permanent freezing of unclaimed yield.**

When the attacker calls `update_rewards(any_active_staker, disable_rewards: true)` in block N:

1. `last_reward_block` is set to N.
2. No rewards are distributed.
3. Every subsequent call in block N reverts with `REWARDS_ALREADY_UPDATED`.
4. The rewards that *would* have accrued in block N are permanently lost — they are never added to `unclaimed_rewards_own` or forwarded to pool contracts.

Repeated every block, this permanently zeroes out all consensus block rewards for every staker and every delegation pool in the protocol. [3](#0-2) 

---

### Likelihood Explanation

**Medium.** The attacker must submit one transaction per block. The cost is gas only; there is no capital requirement. The attack is fully permissionless and requires no privileged access, leaked key, or external dependency. Because Starknet transaction fees are low, sustaining the attack is economically feasible for a motivated griever. The attacker does not profit — this is pure damage to stakers and the protocol.

---

### Recommendation

Restrict `update_rewards` to a trusted caller (e.g., a designated sequencer/operator address, or the attestation contract), mirroring the pattern already used for `update_rewards_from_attestation_contract`:

```
fn update_rewards_from_attestation_contract(...) {
    ...
    self.assert_caller_is_attestation_contract();
    ...
}
``` [4](#0-3) 

Alternatively, remove the `disable_rewards` parameter from the public interface and expose two separate functions — one restricted to a trusted role that can suppress rewards, and one open to stakers for self-service reward collection.

---

### Proof of Concept

**Pre-condition**: Consensus rewards are active (`current_epoch >= consensus_rewards_first_epoch`).

1. Attacker observes any active `staker_address` from on-chain data.
2. At the start of every block N, attacker submits:
   ```
   staking_contract.update_rewards(staker_address, disable_rewards: true)
   ```
3. `last_reward_block` is written to N; no rewards are distributed.
4. Any legitimate call to `update_rewards(..., disable_rewards: false)` in block N reverts with `REWARDS_ALREADY_UPDATED`.
5. In block N+1 the attacker repeats step 2.
6. All stakers accumulate zero consensus block rewards indefinitely; `unclaimed_rewards_own` for every staker remains frozen at its pre-attack value. [5](#0-4) [6](#0-5)
