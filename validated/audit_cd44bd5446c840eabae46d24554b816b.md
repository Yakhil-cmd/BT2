### Title
Missing Access Control on `update_rewards` Allows Any Caller to Block Reward Distribution - (File: src/staking/staking.cairo)

### Summary
The `update_rewards` function in the Staking contract is documented as callable only by the Starkware sequencer, but the implementation enforces no such restriction. Any unprivileged address can call `update_rewards` with `disable_rewards: true`, consuming the per-block reward slot (`last_reward_block`) without distributing rewards. This permanently denies stakers and delegators their block rewards for any targeted block.

### Finding Description
The specification at `docs/spec.md` line 1645 explicitly states:

> **access control**: Only starkware sequencer.

However, the implementation of `update_rewards` in `src/staking/staking.cairo` only calls `self.general_prerequisites()`:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
    let current_block_number = starknet::get_block_number();
    assert!(
        current_block_number > self.last_reward_block.read(),
        "{}",
        Error::REWARDS_ALREADY_UPDATED,
    );
    ...
    self.last_reward_block.write(current_block_number);

    if disable_rewards || self.is_pre_consensus() {
        return;
    }
    ...
```

`general_prerequisites` only checks that the contract is unpaused and the caller is non-zero:

```cairo
fn general_prerequisites(ref self: ContractState) {
    self.assert_is_unpaused();
    assert_caller_is_not_zero();
}
```

There is no check that `get_caller_address() == sequencer_address`. The `last_reward_block` guard is a per-block mutex: once it is set to the current block number, no further call to `update_rewards` can succeed in the same block. An attacker who calls `update_rewards(any_active_staker, disable_rewards: true)` before the sequencer does will:

1. Set `last_reward_block` to the current block.
2. Return immediately without distributing any rewards.
3. Cause the sequencer's legitimate call in the same block to revert with `REWARDS_ALREADY_UPDATED`.

The staker (and all delegators in its pools) permanently lose the block rewards for that block; there is no recovery path.

### Impact Explanation
Every block in which the attacker front-runs the sequencer results in permanent loss of unclaimed yield for the targeted staker and all pool members. Repeated across many blocks, this constitutes a sustained denial of yield. This matches the allowed impact: **Griefing with no profit motive but damage to users or protocol** (Medium), and edges toward **Permanent freezing of unclaimed yield** (High) if applied continuously.

### Likelihood Explanation
Medium. The function is fully public with no access-control gate. On Starknet the sequencer controls transaction ordering, but:
- The sequencer does not necessarily call `update_rewards` in every block for every staker.
- Any block in which the sequencer has not yet called `update_rewards` is a window for the attacker.
- The attacker needs only to submit a valid transaction with any active staker address and `disable_rewards: true`.
- No capital, no privileged key, and no special role is required.

### Recommendation
Add an explicit sequencer-only guard to `update_rewards`, consistent with the specification. For example:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
    assert!(
        get_caller_address() == self.sequencer_address.read(),
        "{}",
        Error::CALLER_IS_NOT_SEQUENCER,
    );
    ...
```

Alternatively, store the sequencer address in contract storage and validate it at the top of `update_rewards`, mirroring the pattern used for `update_rewards_from_attestation_contract` (which correctly restricts to the attestation contract).

### Proof of Concept

1. Consensus rewards are active (`is_pre_consensus()` returns `false`).
2. Active staker `S` has non-zero balance at the current epoch (i.e., staked at least K epochs ago).
3. Attacker `A` (any address, no privileges) calls:
   ```
   staking.update_rewards(staker_address: S, disable_rewards: true)
   ```
4. Inside `update_rewards`:
   - `general_prerequisites()` passes (contract unpaused, caller non-zero). [1](#0-0) 
   - `current_block_number > last_reward_block` passes (first call this block). [2](#0-1) 
   - `last_reward_block` is written to `current_block_number`. [3](#0-2) 
   - `disable_rewards == true` → function returns with no rewards distributed. [4](#0-3) 
5. Sequencer calls `update_rewards(S, disable_rewards: false)` in the same block.
6. `current_block_number > last_reward_block` is now **false** → reverts with `REWARDS_ALREADY_UPDATED`. [5](#0-4) 
7. Staker `S` and all its pool delegators receive **zero rewards** for this block.

The spec mandates sequencer-only access [6](#0-5)  but the code enforces no such restriction. [7](#0-6)
