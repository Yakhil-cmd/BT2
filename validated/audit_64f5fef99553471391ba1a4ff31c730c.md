### Title
`EVM.validateCOAOwnershipProof` does not bind `signedData` to `evmAddress`, enabling cross-COA signature replay - (File: `fvm/evm/stdlib/contract.cdc`)

---

### Summary

`EVM.validateCOAOwnershipProof` verifies Cadence account key signatures over an arbitrary `signedData` blob but never enforces that `signedData` commits to the `evmAddress` argument. A signature produced to prove ownership of one COA (Cadence-Owned Account) is therefore cryptographically valid as proof of ownership of any other COA controlled by the same Cadence account. Any on-chain application or protocol that calls this function directly to gate privileged EVM actions is vulnerable to cross-COA signature replay.

---

### Finding Description

`EVM.validateCOAOwnershipProof` performs two independent checks:

1. Verify that the supplied `signatures` are valid over `signedData` using the account keys at `address`.
2. Verify that the COA capability at `path` has an EVM address equal to `evmAddress`.

The two checks are never linked: `signedData` is never required to encode `evmAddress`. The code itself acknowledges this gap:

```cadence
/// Note: this function does not enforce that `signedData` includes `evmAddress`.
/// In principle, a signature produced for one purpose could be replayed here against
/// a different COA owned by the same Cadence account.
``` [1](#0-0) 

Because `signedData` is caller-supplied and opaque to the verifier, a signature `σ = Sign(key_A, D)` that was produced to prove ownership of COA-X is equally valid when submitted with `path=/public/coaY` and `evmAddress=COA-Y`, as long as both COAs are stored under the same Cadence account `A`. The function will return `isValid: true` in both cases. [2](#0-1) 

The `COAOwnershipProof` struct that is RLP-encoded and passed as the `bytes` argument to the EVM precompile contains `KeyIndices`, `Address`, `CapabilityPath`, and `Signatures` — but no `evmAddress` field — so the proof is structurally context-free with respect to which COA it authorises. [3](#0-2) 

---

### Impact Explanation

Any Cadence script, transaction, or EVM smart contract that calls `EVM.validateCOAOwnershipProof` directly (rather than through the Cadence Arch precompile) and uses the result to gate a privileged action on a specific COA is vulnerable. An attacker who obtains a valid proof/signature blob from a victim — e.g., intercepted from a public transaction, shared off-chain, or reused from a different dApp — can substitute a different `path` and `evmAddress` pointing to another COA owned by the same Cadence account and receive a `true` result. This enables:

- Unauthorized ERC-1271 signature validation on a COA the attacker does not control (if the COA's `isValidSignature` is called with a replayed proof).
- Bypassing COA-specific access controls in any protocol that uses `validateCOAOwnershipProof` as an authorization primitive. [4](#0-3) 

---

### Likelihood Explanation

The EVM Cadence Arch precompile path (`verifyCOAOwnershipProof(address,bytes32,bytes)`) is **safe**: it always passes `address(this)` — the calling contract's own EVM address — as the first argument, so the `evmAddress` checked by `validateCOAOwnershipProof` is always the correct one. [5](#0-4) 

However, `validateCOAOwnershipProof` is `access(all)` and is directly callable from any Cadence script or transaction. The attack requires:

1. A Cadence account that owns more than one COA (at different public paths).
2. An application that calls `EVM.validateCOAOwnershipProof` directly to authorise actions on a specific COA.
3. The attacker obtaining a valid proof blob signed by the victim (e.g., from a prior on-chain transaction or off-chain exchange).

Condition 1 is explicitly called out as the primary mitigating assumption ("Flow wallets historically create at most one COA per account"), but this is a social/convention assumption, not a protocol enforcement. As multi-COA patterns emerge (e.g., per-dApp COAs), the attack surface grows. [6](#0-5) 

---

### Recommendation

Enforce that `signedData` commits to `evmAddress` inside `validateCOAOwnershipProof`. Two concrete options:

1. **Protocol-level enforcement**: Before calling `keyList.verify`, check that `signedData` contains the bytes of `evmAddress` (e.g., as a required prefix or suffix), and return `isValid: false` if it does not.
2. **Structured signed payload**: Define a canonical signed message format that includes `evmAddress`, `address`, and a nonce/sequence number, and verify that `signedData` decodes to this format before accepting the proof.

This mirrors the fix recommended in the original report: include the binding context (nonce/address) inside the signed payload so that a signature produced for one context cannot be replayed in another.

### Proof of Concept

```cadence
// Setup: Cadence account A owns two COAs
// COA-X stored at /public/coaX  (EVM address: 0xAAAA...)
// COA-Y stored at /public/coaY  (EVM address: 0xBBBB...)

// Step 1: Victim (account A) produces a legitimate proof for COA-X
// signedData = arbitrary bytes D (does NOT include 0xAAAA...)
// signatures = Sign(key_A, D)
// proof encodes: {KeyIndices:[0], Address:A, CapabilityPath:"coaX", Signatures:[sig]}

// Step 2: Attacker replays the same signedData + signatures
// but substitutes path=/public/coaY and evmAddress=0xBBBB...
import EVM from 0x...

access(all) fun main(): EVM.ValidationResult {
    return EVM.validateCOAOwnershipProof(
        address: <account_A>,
        path: /public/coaY,          // ← attacker substitutes COA-Y's path
        signedData: <D>,             // ← same bytes victim signed for COA-X
        keyIndices: [0],
        signatures: [<sig>],         // ← victim's signature, unchanged
        evmAddress: <0xBBBB...>      // ← COA-Y's EVM address
    )
    // Returns: ValidationResult(isValid: true, problem: nil)
    // Attacker has proven ownership of COA-Y using a signature for COA-X
}
```

The function returns `isValid: true` because (a) `sig` is a valid signature over `D` by account A's key, and (b) account A does have a COA at `/public/coaY` with address `0xBBBB...`. The binding between the signed data and the specific COA is never verified. [7](#0-6)
