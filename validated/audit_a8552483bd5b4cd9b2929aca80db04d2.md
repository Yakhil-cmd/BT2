### Title
ETH Permanently Locked in `RewardSupplier` When `tick()` Called With No Pending Mint Requests — (File: `L1/starkware/solidity/stake/RewardSupplier.sol`)

---

### Summary

`RewardSupplier.sol`'s `tick()` function is `payable` and requires ETH to cover StarkGate bridge and L1→L2 messaging fees. When `tick()` is called with `msg.value > 0` but there are no pending L2 mint requests (`amountToMint == 0`), the entire ETH value is silently discarded into the contract with no refund path and no recovery mechanism. The contract has no `receive()` fallback and no ETH withdrawal function.

---

### Finding Description

`tick()` is declared `external payable` at line 107:

```solidity
function tick() external payable {
    (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();
    if (amountToMint > 0) {
        ...
        uint256 msgFee = msg.value / 2;
        bridge().depositWithMessage{value: msgFee}(...);
        msgFee = msg.value - msgFee;
        messagingContract().sendMessageToL2{value: msgFee}(...);
    }
    // ← if amountToMint == 0, msg.value is never used or refunded
}
``` [1](#0-0) 

When `amountToMint == 0`, the entire `if` block is skipped. Any ETH attached to the call is absorbed by the contract with no refund. The contract defines no `receive()` or `fallback()` function and no administrative ETH-withdrawal function anywhere in its body or in `RewardSupplierStorage`. [2](#0-1) 

The `requiredMinting()` view function can be called off-chain to check whether minting is needed, but there is no on-chain guard that prevents ETH from being sent when the condition is false, and no refund if it is. [3](#0-2) 

---

### Impact Explanation

Any ETH sent to `tick()` when `amountToMint == 0` is permanently frozen inside `RewardSupplier.sol`. There is no owner-only `withdrawEth()`, no `receive()` that could forward it, and no upgrade path that would automatically recover it. This constitutes **permanent freezing of funds** for the caller.

Mapped allowed impact: **High — Permanent freezing of funds**.

---

### Likelihood Explanation

`tick()` is `external` with no access control — any address, including an unprivileged public caller, can invoke it. Realistic scenarios where ETH is sent with `amountToMint == 0`:

1. **Race condition**: A caller reads `requiredMinting()` off-chain, sees pending messages, then another transaction processes them first; by the time the caller's `tick()` executes, `amountToMint == 0`.
2. **Naive keeper**: An automated keeper sends a fixed ETH amount with every `tick()` call without pre-checking `requiredMinting()`.
3. **Deliberate griefing**: An attacker front-runs a legitimate `tick()` call to drain the pending messages, causing the legitimate caller's ETH to be locked.

Likelihood: **Medium**.

---

### Recommendation

Add an explicit refund when no minting is required:

```solidity
function tick() external payable {
    (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();
    if (amountToMint > 0) {
        // ... existing logic ...
    } else if (msg.value > 0) {
        // Refund unused ETH
        (bool ok, ) = msg.sender.call{value: msg.value}("");
        require(ok, "ETH_REFUND_FAILED");
    }
}
```

Alternatively, add a privileged `withdrawEth(address recipient)` function so that accidentally locked ETH can be recovered by an operator.

---

### Proof of Concept

1. Confirm no pending L2→L1 mint-request messages exist (or front-run a legitimate `tick()` to consume them first).
2. Call `RewardSupplier.tick{value: 1 ether}()` from any EOA.
3. `requiredMinting()` returns `(0, 0)`; the `if (amountToMint > 0)` block is skipped.
4. The function returns normally. The 1 ETH is now held by `RewardSupplier.sol`.
5. No function in the contract allows retrieval of that ETH. It is permanently frozen. [1](#0-0)

    function tick() external payable {
        // Check if minting is required, and how much.
        (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

        if (amountToMint > 0) {
            // Prepare the L2->L1 mintRequest message for consumption.
            uint256[] memory messagePayload = new uint256[](1);
            messagePayload[0] = TOKENS_PER_MINT_REQUEST;

            // Consume the mintRequest messages.
            for (uint256 i = 0; i < numMsgsToConsume; i++) {
                messagingContract().consumeMessageFromL2(mintRequestSource(), messagePayload);
            }

            // Reuest minting of the requested amount from the mint manager.
            mintManager().mintRequest(token(), amountToMint);

            // Deposit the minted amount onto the bridge to the credit of `mintDestination`.
            uint256 msgFee = msg.value / 2;
            bridge().depositWithMessage{value: msgFee}(
                token(),
                amountToMint,
                mintDestination(),
                new uint256[](0)
            );
            emit ConsumedL2MintRequests(numMsgsToConsume, amountToMint);

            // Send a totalSupply update to L2MintCurve.
            msgFee = msg.value - msgFee;
            messagePayload[0] = IERC20(token()).totalSupply();
            messagingContract().sendMessageToL2{value: msgFee}(
                mintingCurve(),
                UPDATE_TOTAL_SUPPLY_SELECTOR,
                messagePayload
            );
        }
    }
