### Title
`uint32` Truncation in `_afterTimelock` Allows Pool Admin to Bypass OracleValueStopLoss Timelock — (`File: metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

---

### Summary

`OracleValueStopLossExtension._afterTimelock` casts the sum `block.timestamp + timelock` from `uint256` to `uint32`. When `timelock` is set to a value large enough that the sum exceeds `type(uint32).max`, the result silently wraps to a past timestamp (or zero), causing `_requireElapsed` to pass immediately and the timelock to be completely bypassed.

---

### Finding Description

`_afterTimelock` computes the execute-after deadline for every pending proposal:

```solidity
function _afterTimelock(address pool_) private view returns (uint32) {
    return uint32(block.timestamp + oracleStopLossConfig[pool_].timelock);
}
``` [1](#0-0) 

`block.timestamp` is `uint256`; `timelock` is `uint32`. The addition is performed in `uint256` (no intermediate overflow), but the result is then explicitly cast to `uint32`. If `block.timestamp + timelock > type(uint32).max = 4,294,967,295`, the high bits are silently discarded.

At the current epoch (~1,753,000,000 s, July 2025), any `timelock` value greater than `4,294,967,295 − 1,753,000,000 ≈ 2,541,967,295` (≈ 80.5 years) causes overflow. Because `timelock` is `uint32`, values up to `4,294,967,295` are accepted. There is **no validation** of the `timelock` field in `initialize` or in `proposeOracleStopLossTimelock`:

```solidity
(uint32 drawdownE6, uint32 decayPerSecondE8, uint32 timelock) = abi.decode(data, (uint32, uint32, uint32));
_validateDrawdown(drawdownE6);
_validateDecay(decayPerSecondE8);
// ← timelock is never validated
``` [2](#0-1) 

The truncated `executeAfter` is then checked by:

```solidity
function _requireElapsed(uint32 executeAfter) private view {
    if (block.timestamp < executeAfter) revert OracleStopLossTimelockNotElapsed(...);
}
``` [3](#0-2) 

When `executeAfter` wraps to a past timestamp (or zero), `block.timestamp < executeAfter` is always `false`, so the guard never reverts. Every `execute*` call succeeds immediately after the corresponding `propose*` call, with zero delay.

This affects all four proposal paths:

| Proposal | Execute |
|---|---|
| `proposeOracleStopLossTimelock` | `executeOracleStopLossTimelock` |
| `proposeOracleStopLossDrawdown` | `executeOracleStopLossDrawdown` |
| `proposeOracleStopLossDecay` | `executeOracleStopLossDecay` |
| `proposeOracleStopLossHighWatermarks` | `executeOracleStopLossHighWatermarks` | [4](#0-3) 

---

### Impact Explanation

The `OracleValueStopLossExtension` timelock is the sole mechanism protecting LPs from sudden, unilateral changes to drawdown thresholds, decay rates, and high watermarks by the pool admin. Bypassing it allows the pool admin to:

1. **Immediately tighten drawdown** → stop-loss triggers on the very next swap, forcing LPs into an unusable pool state.
2. **Immediately loosen drawdown to 100%** → stop-loss is effectively disabled, removing LP protection entirely.
3. **Immediately reset high watermarks** to artificially high values → any subsequent swap triggers the stop-loss, bricking the pool.
4. **Immediately set decay to zero** → watermarks never decay, permanently locking the pool.

LPs who observe a large `timelock` value (e.g., `uint32.max`) believe they have ~136 years of advance notice before any parameter change. In reality, the pool admin can execute changes atomically in the same block as the proposal.

This is a direct admin-boundary break: the pool admin exceeds the timelock constraint that defines the boundary of their semi-trusted role.

---

### Likelihood Explanation

The pool admin sets the initial `timelock` at pool creation via `extensionInitData`. No factory-level cap or validation exists on this value. A pool admin who sets `timelock = type(uint32).max` (or any value > ~2.54 billion) at deployment silently arms the bypass for the entire pool lifetime. The attack requires only a single malicious or mistaken parameter at pool creation — a realistic scenario given the absence of any guard.

---

### Recommendation

Replace the truncating cast with a checked computation. Either:

```solidity
function _afterTimelock(address pool_) private view returns (uint32) {
    uint256 result = block.timestamp + uint256(oracleStopLossConfig[pool_].timelock);
    require(result <= type(uint32).max, TimelockOverflow());
    return uint32(result);
}
```

Or store and compare `executeAfter` as `uint256` throughout the schedule structs, eliminating the truncation entirely. Additionally, add a maximum cap on `timelock` in `initialize` and `proposeOracleStopLossTimelock` (e.g., `≤ 365 days`).

---

### Proof of Concept

```solidity
// Pool admin sets timelock = type(uint32).max at pool creation via extensionInitData.
// block.timestamp ≈ 1_753_000_000 (July 2025)
// block.timestamp + type(uint32).max = 1_753_000_000 + 4_294_967_295 = 6_047_967_295
// uint32(6_047_967_295) = 6_047_967_295 - 4_294_967_296 = 1_752_999_999  ← past timestamp

// Step 1: pool admin proposes a drawdown change
ext.proposeOracleStopLossDrawdown(pool, 0); // disable stop-loss
// _afterTimelock returns 1_752_999_999 (past) → pendingDrawdownExecuteAfter = past

// Step 2: pool admin executes immediately in the same block
ext.executeOracleStopLossDrawdown(pool);
// _requireElapsed(1_752_999_999): block.timestamp (1_753_000_000) < 1_752_999_999 → false → no revert
// drawdown is now 0, stop-loss disabled, zero delay elapsed
```
