### Title
Missing Caller Authorization in `update_rewards` Enables Permanent Freezing of Block Rewards - (File: src/staking/staking.cairo)

### Summary
The `update_rewards` function in the Staking contract carries no caller authorization check. Any unprivileged address can invoke it with `disable_rewards: true`, which (1) writes the current block number into the global `last_reward_block` slot and (2) returns immediately without distributing any rewards. Because `last_reward_block` is a single contract-wide variable, this one call per block permanently forecloses legitimate reward distribution for every staker in that block.

### Finding Description
`update_rewards` is declared under `IStakingRewardsManager` and is intended to distribute per-block consensus rewards to a staker. It accepts a caller-supplied `disable_rewards: bool` flag. The function performs no check on `get_caller_address()` before executing its state-changing logic.

The critical ordering is:

```
// Update last block rewards.
self.last_reward_block.write(current_block_number);   // ← written unconditionally

if disable_rewards || self.is_pre_consensus() {
    return;                                            // ← exits before any reward transfer
}
``` [1](#0-0) 

`last_reward_block` is a single global storage slot (not per-staker):

```
/// Last block number for which rewards were distributed.
last_reward_block: BlockNumber,
``` [2](#0-1) 

The guard at the top of the function enforces that only one call per block can succeed:

```
assert!(
    current_block_number > self.last_reward_block.read(),
    "{}",
    Error::REWARDS_ALREADY_UPDATED,
);
``` [3](#0-2) 

Because `last_reward_block` is written before the `disable_rewards` branch, an attacker who calls `update_rewards(any_valid_staker, true)` in block N:
- Consumes the block's single allowed call slot
- Distributes zero rewards
- Causes every subsequent call in block N to revert with `REWARDS_ALREADY_UPDATED`

No privileged role, no funds, and no special setup are required. The attacker only needs to supply any currently-active staker address (stakers are enumerable via the public `stakers` vector). [4](#0-3) 

### Impact Explanation
**High — Permanent freezing of unclaimed yield.** Block rewards that are not distributed in a given block are lost forever; there is no retroactive catch-up mechanism. An attacker sustaining this call once per block causes every staker to accrue zero consensus block rewards indefinitely. The `last_reward_block` write is irreversible within the block, so even if the attacker stops, the rewards for every attacked block are permanently gone.

### Likelihood Explanation
**High.** The entry point is a public, permissionless function. The only precondition is supplying a valid (non-zero-balance, active) staker address, which is trivially discoverable on-chain. The cost is one Starknet transaction per block — negligible gas. No leaked keys, no bridge compromise, and no governance action are required.

### Recommendation
Add a caller authorization check at the top of `update_rewards` (analogous to `assert_caller_is_attestation_contract` used elsewhere) so that only the designated rewards-manager role or the attestation contract can invoke it. Alternatively, remove the `disable_rewards` parameter from the public interface and expose a separate privileged function for the "skip rewards" path, keeping the public entry point always reward-distributing.

### Proof of Concept
1. Identify any active staker address `S` (readable from the public `stakers` vector).
2. In every new block N, call `update_rewards(S, true)` before any legitimate caller.
3. `last_reward_block` is set to N; the function returns without distributing rewards.
4. Any subsequent call to `update_rewards` in block N reverts with `REWARDS_ALREADY_UPDATED`.
5. All stakers lose their block-N consensus rewards permanently.
6. Repeat each block to freeze all block rewards indefinitely.
