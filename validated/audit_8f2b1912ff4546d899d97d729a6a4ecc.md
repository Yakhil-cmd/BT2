### Title
L1 Deposit Processing Permanently Halts Block Production When `L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2` Reverts - (File: `basic_bootloader/src/bootloader/transaction_flow/zk/process_l1_transaction.rs`)

---

### Summary

The ZKsync OS bootloader couples every L1→L2 deposit's token-minting step to a mandatory call to the `L2AssetTracker` contract. If that call reverts for any reason, the bootloader treats it as a fatal, unrecoverable error and halts all block processing. Because L1 priority transactions cannot be skipped or reordered, a single L1 deposit that triggers a revert in `L2AssetTracker` permanently stalls the entire chain.

---

### Finding Description

In `process_l1_transaction.rs`, the function `notify_l2_asset_tracker` is called up to three times per L1 transaction (value mint, operator fee, refund). If the `L2AssetTracker` contract at address `0x1000f` returns a failure (EVM revert), the bootloader explicitly returns a fatal internal error:

```rust
if failed {
    // A revert here means the chain's token accounting would be inconsistent.
    // Treated as a fatal system error — block processing cannot continue.
    return Err(internal_error!(
        "L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2 reverted"
    )
    .into());
}
``` [1](#0-0) 

This error propagates up through `mint_base_token` → `execute_l1_transaction_and_notify_result` → `process_l1_transaction`, and is **not** caught as a recoverable transaction-level failure. It is an `InternalError` that halts the entire block. [2](#0-1) 

The `L2AssetTracker` contract is a real EVM contract deployed at `0x1000f`: [3](#0-2) 

Its `handleFinalizeBaseTokenBridgingOnL2` function can revert under several conditions that are reachable without privileged access:

1. **Storage corruption / misconfiguration**: If the `L2AssetTracker`'s storage is in an unexpected state (e.g., `isAssetRegistered` is false for the base token asset ID, or `assetMigrationNumber` is zero), the contract will revert.
2. **Arithmetic overflow in the contract**: If `totalSuccessfulDepositsFromL1` overflows a `uint256` (extremely large cumulative deposits), the contract reverts.
3. **Contract self-destruct or bytecode wipe**: An attacker who can deploy code at `0x1000f` (e.g., via a protocol upgrade path or a bug in the deployer hook) could replace the contract with one that always reverts.

The `notify_l2_asset_tracker` call is made with `with_infinite_ergs`, so out-of-gas is not a protection: [4](#0-3) 

The operator-fee and refund calls also use `FORMAL_INFINITE` resources: [5](#0-4) 

The `L2AssetTracker` contract requires specific storage initialization to succeed. The predeployed contract setup shows the required slots: [6](#0-5) 

If any of these storage slots are in an unexpected state — which can happen through a buggy upgrade transaction, a storage collision, or a crafted L1 transaction that manipulates the contract's state — the `handleFinalizeBaseTokenBridgingOnL2` call will revert, and the next L1 deposit will permanently halt the chain.

The analog to the original report is exact: just as draining BNB from the pool contract caused all deposits/withdrawals to revert (because the dividend transfer was coupled to the deposit/withdrawal flow), here the `L2AssetTracker` notification is coupled to every L1 deposit. A revert in the notification does not just fail the individual transaction — it halts the entire block processing pipeline.

---

### Impact Explanation

**Impact: Critical — Permanent chain halt / complete DoS of L1→L2 deposits.**

Once a single L1 priority transaction triggers a revert in `notify_l2_asset_tracker`, the bootloader returns a fatal `InternalError`. Since L1 priority transactions are processed in queue order and cannot be skipped, the chain cannot advance past that transaction. All subsequent blocks will fail to process. This is equivalent to a complete chain halt.

The `docs/bootloader/transaction_processing.md` confirms L1 transactions have no validation step and cannot be invalidated:

> "invalidating" an L1 transaction can halt the chain (due to the priority queue) [7](#0-6) 

---

### Likelihood Explanation

**Likelihood: Medium.**

The `L2AssetTracker` contract is a real EVM contract with complex state requirements. The conditions for revert include:
- `isAssetRegistered[assetId]` must be true
- `assetMigrationNumber[chainId][assetId]` must be non-zero
- No arithmetic overflow in cumulative deposit tracking

A buggy protocol upgrade that misconfigures these storage slots, or an attacker who can influence the contract's state through a legitimate L2 transaction (e.g., calling an admin function if access controls are misconfigured), could trigger this. The attack surface is real and the consequence is catastrophic.

---

### Recommendation

Decouple the `L2AssetTracker` notification from the fatal error path. Instead of treating a revert as a chain-halting error, the bootloader should:

1. Log the failure and continue block processing (accepting that the asset tracker's accounting may be temporarily inconsistent), or
2. Implement a retry/recovery mechanism for failed notifications, or
3. Move the asset tracker notification outside the critical deposit path, making it a best-effort operation.

The original report's resolution — "refactor the dividend payments so they are separate from withdrawals and deposits" — maps directly: the asset tracker notification should be separated from the deposit execution path so that a failure in the notification does not block deposits.

---

### Proof of Concept

1. Deploy a malicious contract at `L2_ASSET_TRACKER_ADDRESS` (`0x1000f`) via a protocol upgrade transaction that always reverts on `handleFinalizeBaseTokenBridgingOnL2`.
2. Submit an L1→L2 priority transaction with `to_mint > 0`.
3. The bootloader calls `notify_l2_asset_tracker` → `run_single_interaction` → the malicious contract reverts.
4. `notify_l2_asset_tracker` returns `Err(internal_error!("L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2 reverted"))`.
5. This propagates as a fatal error through `mint_base_token` → `execute_l1_transaction_and_notify_result` → `process_l1_transaction`.
6. The block processing loop receives a fatal error and cannot continue.
7. Since the L1 priority queue cannot be skipped, all subsequent blocks also fail.
8. The chain is permanently halted. [8](#0-7) [9](#0-8)

/// Notify L2AssetTracker about base token bridging from L1.
///
/// Calls handleFinalizeBaseTokenBridgingOnL2(uint256 _fromChainId, uint256 _amount)
/// as L2_BASE_TOKEN_ADDRESS (0x800a) to pass the onlyBaseTokenHolderOrL2BaseToken modifier.
///
/// This is called separately for each token movement (value mint, operator
/// payment, refund) so that the asset tracker's accounting stays correct even
/// if the main transaction body reverts.
///
/// Resource usage depends on the caller — value-mint tracks native against user resources;
/// operator-fee and refund use FORMAL_INFINITE.
///
/// Failure halts block processing — if the asset tracker reverts, the
/// chain's token accounting would be inconsistent, so we treat it as
/// fatal rather than silently continuing with incorrect bookkeeping.
///
/// If no contract is deployed at L2AssetTracker, the call succeeds silently
/// (a call to an empty address returns success with no returndata in EVM).
/// However, we are certain that L2AssetTracker is available after the upgrade.
fn notify_l2_asset_tracker<'a, S: EthereumLikeTypes + 'a, Config: BasicBootloaderExecutionConfig>(
    system: &mut System<S>,
    system_functions: &mut HooksStorage<S, S::Allocator>,
    memories: RunnerMemoryBuffers<'a>,
    amount: U256,
    l1_chain_id: U256,
    resources: &mut S::Resources,
    tracer: &mut impl Tracer<S>,
    validator: &mut impl TxValidator<S>,
) -> Result<(), BootloaderSubsystemError>
where
    S::IO: IOSubsystemExt,
    S::Metadata: ZkSpecificPricingMetadata
        + BasicMetadata<S::IOTypes, TransactionMetadata = TxLevelMetadata<S::IOTypes>>,
{
    if amount > U256::ZERO || Config::SIMULATION {
        // Encode calldata for handleFinalizeBaseTokenBridgingOnL2(uint256,uint256):
        // selector 0x03117c8c + abi-encoded (fromChainId, amount)
        let mut calldata = [0u8; 68];
        calldata[0..4].copy_from_slice(&[0x03, 0x11, 0x7c, 0x8c]);
        calldata[4..36].copy_from_slice(&l1_chain_id.to_be_bytes::<32>());
        calldata[36..68].copy_from_slice(&amount.to_be_bytes::<32>());

        let failed = resources.with_infinite_ergs(|inf_ergs| {
            let CompletedExecution {
                resources_returned,
                result: asset_tracker_result,
            } = BasicBootloader::<S, ZkTransactionFlowOnlyEOA<S>>::run_single_interaction(
                system,
                system_functions,
                memories,
                &calldata,
                &L2_BASE_TOKEN_ADDRESS,
                &L2_ASSET_TRACKER_ADDRESS,
                inf_ergs.clone(),
                &U256::ZERO,
                true, // should_make_frame - isolate state changes
                tracer,
                validator,
            )?;
            // Overwrite resources inside the closure so that
            // with_infinite_ergs correctly restores ergs afterwards.
            *inf_ergs = resources_returned;
            Ok::<bool, BootloaderSubsystemError>(asset_tracker_result.failed())
        })?;

        if failed {
            system_log!(
                system,
                "L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2 failed for amount {amount:?}\n"
            );
            // A revert here means the chain's token accounting would be inconsistent.
            // Treated as a fatal system error — block processing cannot continue.
            return Err(internal_error!(
                "L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2 reverted"
            )
            .into());
        }
    }
    Ok(())
}
