Audit Report

## Title
Replicated-Query State Corruption in `estimate_withdrawal_fee` Permanently Drains `available_utxos` — (`rs/dogecoin/ckdoge/minter/src/main.rs`)

## Summary

`estimate_withdrawal_fee` is declared `#[query]` but calls `mutate_state`, passing `&mut s.available_utxos` into `estimate_retrieve_doge_fee`, which permanently removes selected UTXOs via `utxos_selection`. The inline comment claiming "any change will be discarded" is factually incorrect for replicated-query execution. When called via an ingress update message, the IC executes `canister_query_*` methods in replicated mode and commits all state mutations. An unprivileged attacker can repeatedly invoke this endpoint as an update call to silently drain the minter's UTXO pool, permanently blocking all ckDOGE withdrawals.

## Finding Description

**Root cause — `main.rs` lines 98–128:**
`estimate_withdrawal_fee` is annotated `#[query]` but immediately calls `ic_ckbtc_minter::state::mutate_state`, passing `&mut s.available_utxos` to `ic_ckdoge_minter::fees::estimate_retrieve_doge_fee`. [1](#0-0) 

**Incorrect comment — lines 102–103:**
The comment "any change will be discarded" is only true for non-replicated (direct) query calls. It is false for replicated queries invoked via ingress. [2](#0-1) 

**Call chain — `fees/mod.rs` lines 130–152:**
`estimate_retrieve_doge_fee` delegates directly to `ic_ckbtc_minter::queries::estimate_withdrawal_fee` with `available_utxos` as a mutable reference, no cloning. [3](#0-2) 

**UTXO removal — `queries.rs` lines 57–75:**
`utxos_selection` is called with `available_utxos`, which removes the selected UTXOs from the set. On success, the function returns the fee and the selected UTXOs are **never reinserted**. The only restoration path is inside `greedy` itself, which only restores UTXOs when the target cannot be met at all (i.e., when the call would return `NotEnoughFunds`). On any successful fee estimation, the UTXOs are gone. [4](#0-3) 

**`utxos_selection` / `greedy` — `lib.rs` lines 1039–1099:**
`greedy` removes UTXOs from `available_utxos` one by one. It only restores them if the loop exhausts the set before meeting the target. On success it returns the removed UTXOs to the caller; the caller (`estimate_withdrawal_fee`) discards them. [5](#0-4) [6](#0-5) 

**IC replicated-query behavior confirmed — `in_replicated_execution.rs` lines 53–65:**
`test_in_replicated_execution_for_replicated_query_returns_1` calls a `canister_query_*` method via `execute_ingress` (the update path) and asserts `REPLICATED_EXECUTION = [1,0,0,0]`. State mutations in this path are committed. [7](#0-6) 

**Missing guard — `main.rs` lines 248–252:**
`http_request` (also `#[query]`) explicitly traps when `in_replicated_execution()` is true. This guard was added precisely because query methods can be called in replicated mode. `estimate_withdrawal_fee` has no such guard. [8](#0-7) 

**Existing test does not catch this — `tests.rs` lines 609–614:**
The test checks `get_known_utxos` (user-account UTXOs) before and after, but the vulnerability affects `available_utxos` (the minter's internal UTXO pool). These are separate data structures. The test passes while the internal pool is silently drained. [9](#0-8) 

**Test utility calls via `update_call` — `minter.rs` lines 151–163:**
The test helper itself invokes `estimate_withdrawal_fee` as an update call, confirming the replicated-mode path is the one exercised in tests. The idempotency assertion at line 654–658 passes only because 2 UTXOs are deposited and a single fee estimation may consume only 1, leaving 1 for the second call — not because state is preserved. [10](#0-9) [11](#0-10) 

## Impact Explanation

Every successful replicated call to `estimate_withdrawal_fee` removes the UTXOs selected for the simulated transaction from `available_utxos` without recording any event or submitting any transaction. After enough calls, `available_utxos` is empty. All subsequent legitimate `retrieve_doge_with_approval` calls fail with `NotEnoughFunds`. Deposited DOGE is locked in the minter's Dogecoin address with no withdrawal path. The minter's event log shows no anomaly. Recovery requires a canister upgrade with special re-scanning logic. This matches the **High** impact class: significant ck-token / Chain Fusion security impact with concrete, permanent user-funds harm (withdrawal DoS and locked assets).

## Likelihood Explanation

The attack requires no privilege, no key, and no governance majority. Any principal can send an ingress update call to `estimate_withdrawal_fee` with a valid amount. The IC routes it to `canister_query_estimate_withdrawal_fee` in replicated mode. The call succeeds and commits state. The attacker repeats until `available_utxos` is empty. The cost is only cycles for ingress messages. The attack is silent: no anomalous events are logged, and the existing monitoring checks user UTXOs, not the minter's internal pool.

## Recommendation

Add the same guard used by `http_request` at the top of `estimate_withdrawal_fee`:

```rust
#[query]
fn estimate_withdrawal_fee(
    arg: EstimateFeeArg,
) -> Result<WithdrawalFee, EstimateWithdrawalFeeError> {
    if ic_cdk::api::in_replicated_execution() {
        ic_cdk::trap("update call rejected");
    }
    // ...
}
```

Alternatively, refactor to use `read_state` with a cloned or snapshot view of `available_utxos` so no mutation occurs at all, eliminating the risk regardless of execution mode.

## Proof of Concept

```
1. Minter has N UTXOs in available_utxos (deposited via update_balance).
2. Attacker sends ingress update call:
     canister_id: <ckdoge_minter>
     method: "estimate_withdrawal_fee"
     arg: record { amount = opt <valid_amount> }
     call_type: update  ← not query
3. IC executes canister_query_estimate_withdrawal_fee in replicated mode.
4. mutate_state removes UTXOs from available_utxos; state is committed.
5. Repeat until available_utxos.len() == 0.
6. Any retrieve_doge_with_approval call now returns NotEnoughFunds.
```

Deterministic state-machine test to confirm: call `estimate_withdrawal_fee` via `execute_ingress` (not `query`), then read `available_utxos` length — it will be smaller than before the call, proving the mutation is committed. The existing test at `tests.rs:654–658` already inadvertently exercises this path via `update_call` and would expose the drain if it checked `available_utxos` instead of `get_known_utxos`.

```rust
fn estimate_withdrawal_fee(

    // This is a **query** endpoint, so mutating the state is not an issue
    // (even when called in replicated mode) since any change will be discarded.
    ic_ckbtc_minter::state::mutate_state(|s| {
        let fee_estimator = DOGECOIN_CANISTER_RUNTIME.fee_estimator(s);
        let withdrawal_amount = arg.amount.unwrap_or(s.fee_based_retrieve_btc_min_amount);

        ic_ckdoge_minter::fees::estimate_retrieve_doge_fee(
            &mut s.available_utxos,
            withdrawal_amount,
            s.last_median_fee_per_vbyte
                .expect("Bitcoin current fee percentiles not retrieved yet."),
            s.max_num_inputs_in_transaction,
            &fee_estimator,
        )
        .map_err(|e| match e {
            BuildTxError::NotEnoughFunds
            | BuildTxError::InvalidTransaction(InvalidTransactionError::TooManyInputs { .. }) => {
                EstimateWithdrawalFeeError::AmountTooHigh
            }
            BuildTxError::AmountTooLow | BuildTxError::DustOutput { .. } => {
                EstimateWithdrawalFeeError::AmountTooLow {
                    min_amount: s.fee_based_retrieve_btc_min_amount,
                }
            }
        })
    })
}
```

```rust
fn utxos_selection(target: u64, available_utxos: &mut UtxoSet, output_count: usize) -> Vec<Utxo> {
    #[cfg(feature = "canbench-rs")]
    let _scope = canbench_rs::bench_scope("utxos_selection");

    let mut input_utxos = greedy(target, available_utxos);

    if input_utxos.is_empty() {
        return vec![];
    }

    if available_utxos.len() > UTXOS_COUNT_THRESHOLD {
        while input_utxos.len() < output_count + 1 {
            if let Some(min_utxo) = available_utxos.pop_first() {
                input_utxos.push(min_utxo);
            } else {
                break;
            }
        }
    }

    input_utxos
}
```

```rust
fn greedy(target: u64, available_utxos: &mut UtxoSet) -> Vec<Utxo> {
    #[cfg(feature = "canbench-rs")]
    let _scope = canbench_rs::bench_scope("greedy");

    let mut solution = vec![];
    let mut goal = target;
    while goal > 0 {
        let candidate_utxo = available_utxos
            .find_lower_bound(goal)
            .or_else(|| available_utxos.last())
            .cloned();
        match candidate_utxo {
            Some(utxo) => {
                let utxo = available_utxos.remove(&utxo).expect("BUG: missing UTXO");
                goal = goal.saturating_sub(utxo.value);
                solution.push(utxo);
            }
            None => {
                // Not enough available UTXOs to satisfy the request.
                for u in solution {
                    available_utxos.insert(u);
                }
                return vec![];
            }
        }
    }

    debug_assert!(solution.is_empty() || solution.iter().map(|u| u.value).sum::<u64>() >= target);

    solution
```
