### Title
Permanent ERC-20 Token Loss When `ft_transfer` Fails in `ExitToNear` Precompile Without `error_refund` Feature - (File: engine-precompiles/src/native.rs)

### Summary
When the `ExitToNear` precompile is invoked for an ERC-20 exit and the downstream `ft_transfer` NEAR promise fails, the burned ERC-20 tokens are permanently lost if the `error_refund` compile-time feature is not enabled. This is the direct Aurora Engine analog to the PaymentSplitter report: just as funds sent via ink!'s native `transfer` bypass the `receive` event tracking, here a failed NEAR-side `ft_transfer` has no callback path to re-mint the already-burned EVM tokens.

### Finding Description

The `ExitToNear` precompile's `run` function constructs a `callback_args` struct and decides whether to attach a callback promise based on whether that struct is non-default: [1](#0-0) 

The `refund` field is conditionally compiled:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,
    transfer_near: transfer_near_args,
};
```

For a standard ERC-20 exit (flag `0x1`, no wNEAR unwrap), `transfer_near_args` is `None` and `refund` is `None` (without `error_refund`). This makes `callback_args` equal to `ExitToNearPrecompileCallbackArgs::default()`, so the code takes the no-callback branch:

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // ← no callback attached
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs { ... })
};
``` [2](#0-1) 

The EVM transaction completes with the ERC-20 tokens already burned. The `ft_transfer` promise then executes asynchronously on NEAR. If it fails (e.g., the recipient account is not registered with the NEP-141 contract), there is no `exit_to_near_precompile_callback` to re-mint the tokens: [3](#0-2) 

The refund path in `exit_to_near_precompile_callback` is only reachable when a callback was attached, which requires `refund` to be non-`None`. Without `error_refund`, this branch is structurally unreachable for ERC-20 exits.

The test suite explicitly acknowledges this behavior: [4](#0-3) 

The same issue applies to the base ETH exit path: ETH is deducted from the caller's EVM balance and credited to the `exit_to_near` precompile address, then a `ft_transfer` is scheduled. If it fails without `error_refund`, the ETH is stranded at the precompile address with no recovery path. [5](#0-4) 

### Impact Explanation

**High — Permanent freezing of funds.**

When the `ft_transfer` promise fails:
- ERC-20 tokens are permanently burned from the EVM state.
- The corresponding NEP-141 tokens remain in Aurora's NEP-141 balance, inaccessible to the user.
- There is no recovery mechanism: the user cannot re-attempt the exit or reclaim the burned tokens.

This matches the "permanent freezing of funds" impact category.

### Likelihood Explanation

**Medium.** Any EVM user who calls the `ExitToNear` precompile targeting a NEAR account that is not registered with the relevant NEP-141 contract will trigger this path. Common causes include:
- Sending to a newly created NEAR account that has not called `storage_deposit` on the NEP-141 contract.
- Typos in the recipient account ID.
- Sending to another Aurora Engine instance (the exact scenario described in the PaymentSplitter report: one contract sending to another of the same type via the low-level transfer path).

The likelihood depends on whether `error_refund` is enabled in the production build. If it is not, every failed `ft_transfer` from an ERC-20 exit results in permanent token loss.

### Recommendation

1. Enable the `error_refund` feature in all production builds to ensure the callback path is always attached.
2. Alternatively, unconditionally attach the callback for all exit paths (not just when `error_refund` is enabled), so that a failed `ft_transfer` always triggers re-minting of burned tokens.
3. Document clearly in the ERC-20 exit interface that the recipient must be pre-registered with the NEP-141 contract, and consider adding an on-chain pre-check before burning tokens.

### Proof of Concept

1. Deploy an ERC-20 on Aurora backed by a NEP-141 token.
2. Bridge NEP-141 tokens into Aurora (user holds ERC-20 balance).
3. Call the ERC-20's `withdraw` function targeting a NEAR account that has **not** called `storage_deposit` on the NEP-141 contract.
4. The EVM transaction succeeds: ERC-20 tokens are burned.
5. The `ft_transfer` NEAR promise fails: recipient not registered.
6. Without `error_refund`, no callback fires. The ERC-20 tokens are gone and the NEP-141 tokens remain locked in Aurora's balance.
7. Confirmed by the existing test `test_exit_to_near_refund` which explicitly asserts `balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into()` (i.e., tokens are lost) when `error_refund` is not enabled. [6](#0-5)
