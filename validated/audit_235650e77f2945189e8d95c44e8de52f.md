### Title
Unchecked Exit-Precompile Return Value in `withdrawToNear`/`withdrawToEthereum` Causes Permanent ERC-20 Token Loss — (File: `etc/eth-contracts/contracts/EvmErc20.sol`)

---

### Summary

`EvmErc20.sol` and `EvmErc20V2.sol` implement `withdrawToNear` and `withdrawToEthereum` by first burning the caller's ERC-20 tokens with `_burn`, then calling the exit precompile via a low-level assembly `call`. The return value `res` is captured but **never checked**. If the precompile call fails for any reason (e.g., an invalid NEAR recipient account ID, or out-of-gas), the burn is already committed and irreversible, while no NEP-141 transfer occurs. The user's bridged tokens are permanently destroyed with no corresponding NEAR-side credit.

---

### Finding Description

In `EvmErc20.sol`:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // ← burn is committed first

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f,
                        0, add(input, 32), input_size, 0, 32)
        // res is never checked — no require(res != 0)
    }
}
``` [1](#0-0) 

The identical pattern exists in `withdrawToEthereum` (calling precompile `0xb0bd02f6a392af548bdf1cfaee5dfa0eefcc8eab`): [2](#0-1) 

And in `EvmErc20V2.sol` (the `error_refund`-enabled variant used in production when the feature flag is set): [3](#0-2) 

The engine selects which binary to embed at compile time: [4](#0-3) 

The `ExitToNear` precompile's `run` method returns `ExitError` in multiple reachable cases — invalid NEAR account ID in `parse_recipient`, amount exceeding `u128::MAX`, or out-of-gas: [5](#0-4) 

When the precompile returns an error, the EVM `call` opcode returns `0` to the caller. Because the Solidity code does not `require(res != 0)` or otherwise revert, the `_burn` state change is finalized while the NEAR-side `ft_transfer` promise is never scheduled.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

The user's ERC-20 balance is reduced to zero (burned). The corresponding NEP-141 tokens remain locked inside the Aurora contract with no mechanism to recover them. The bridge invariant (ERC-20 supply on Aurora == NEP-141 balance held by Aurora) is broken: Aurora holds more NEP-141 than the total ERC-20 supply, and the deficit is unrecoverable without an admin intervention that the protocol does not provide for this case.

---

### Likelihood Explanation

**Medium.** The `withdrawToNear` function is `external` and callable by any ERC-20 token holder. The `recipient` parameter is arbitrary user-supplied bytes. A user who passes bytes that do not parse as a valid NEAR account ID (e.g., a typo, an Ethereum address, or an empty string) will trigger the failure path. The `parse_recipient` function in the precompile enforces strict NEAR account ID validation: [6](#0-5) 

Any input that fails this check causes the precompile to return `ExitError`, the `call` returns `0`, and the burn is permanent. This is a realistic accidental trigger for ordinary users unfamiliar with NEAR account ID format constraints.

---

### Recommendation

Add a return-value check immediately after the assembly `call` in both `withdrawToNear` and `withdrawToEthereum` in both `EvmErc20.sol` and `EvmErc20V2.sol`:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f,
                    0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

This ensures that if the precompile call fails for any reason, the entire transaction reverts (including the `_burn`), preserving the user's token balance and the bridge accounting invariant.

---

### Proof of Concept

1. Deploy Aurora Engine with `EvmErc20.sol` (or `EvmErc20V2.sol`) as the ERC-20 implementation.
2. Bridge a NEP-141 token to Aurora; mint ERC-20 tokens to address `A`.
3. From address `A`, call `withdrawToNear(bytes("not_a_valid!!account"), amount)`.
4. `_burn(A, amount)` executes successfully — `A`'s ERC-20 balance drops to zero.
5. The precompile at `0xe921...` calls `parse_recipient("not_a_valid!!account")`, which fails with `ERR_INVALID_RECEIVER_ACCOUNT_ID`.
6. The assembly `call` returns `res = 0`; no revert is issued.
7. Transaction succeeds. `A`'s ERC-20 balance is zero; Aurora's NEP-141 balance is unchanged; `A` receives nothing on NEAR.
8. Funds are permanently lost. [1](#0-0) [6](#0-5) [4](#0-3)

fn parse_recipient(recipient: &[u8]) -> Result<Recipient<'_>, ExitError> {
    let recipient = str::from_utf8(recipient)
        .map_err(|_| ExitError::Other(Cow::from("ERR_INVALID_RECEIVER_ACCOUNT_ID")))?;
    let (receiver_account_id, message) = recipient.split_once(':').map_or_else(
        || (recipient, None),
        |(recipient, msg)| {
            if msg == UNWRAP_WNEAR_MSG {
                (recipient, Some(Message::UnwrapWnear))
            } else {
                (recipient, Some(Message::Omni(msg)))
            }
        },
    );

    Ok(Recipient {
        receiver_account_id: receiver_account_id
            .parse()
            .map_err(|_| ExitError::Other(Cow::from("ERR_INVALID_RECEIVER_ACCOUNT_ID")))?,
        message,
    })
