### Title
WalletConnect `take_offer` Displays Understated `amountWithRoyalties` When Offer's `infos.royalty_percentage` Exceeds On-Chain NFT Royalty — (File: `packages/gui/src/electron/commands/parseCommandDisplay.ts`)

### Summary
The WalletConnect confirmation dialog for `chia_wallet.take_offer` computes the displayed `amountWithRoyalties` using the **on-chain** royalty percentage fetched via `nftGetInfo`, while the actual royalty payment executed by the wallet backend uses the **offer's attacker-controlled `infos.royalty_percentage`**. A malicious dApp can craft an offer with an inflated `infos.royalty_percentage`, causing the confirmation dialog to show a lower total cost than what the user actually pays.

### Finding Description

In `parseCommandDisplay.ts`, the `take_offer` path calls `offerSummaryRoyaltyPercentages(summary)` to extract the initial royalty percentage from `offerSummary.infos`: [1](#0-0) 

The `infos` field is derived from the offer string provided by the dApp — it is attacker-controlled. This initial value is passed as `royaltyPercentage` into `parseWalletDeltaItem`.

Inside `parseWalletDeltaItem`, for NFT assets, the code first sets the initial (attacker-supplied) value, then **unconditionally overrides it** with the result of `nftGetInfo` if that call succeeds: [2](#0-1) 

The overridden `result.royaltyPercentage` (now the on-chain value) is then consumed by `royaltyPercentagesForSide` and `formatAmountWithRoyalties` to produce the `amountWithRoyalties` shown to the user: [3](#0-2) 

The display therefore reflects the **on-chain** royalty. However, the wallet backend, when executing `take_offer`, uses the royalty percentage embedded in the offer's driver dict (`infos`) to compute the actual royalty payment the taker must add. These two values are now decoupled: the display shows the on-chain royalty while the actual spend uses the offer's `infos` royalty.

The same structural issue applies to `create_offer_for_ids`: the attacker-controlled `driver_dict.royalty_percentage` is used to construct the actual offer, but `nftGetInfo` overrides the display value when it succeeds: [4](#0-3) 

The test suite explicitly documents the fallback behavior (when `nftGetInfo` fails, the `driver_dict` value is used for display), confirming the two sources are treated as independent: [5](#0-4) 

### Impact Explanation

**High.** A user accepting a `take_offer` WalletConnect request sees `amountWithRoyalties` computed from the legitimate on-chain royalty (e.g., 2.5%), but the wallet backend computes the actual royalty payment from the offer's `infos.royalty_percentage` (e.g., 99.99%). The user approves a transaction believing they are paying 1.025 XCH total, but the wallet executes a spend of ~2 XCH. This is a direct, unprivileged-attacker-reachable XCH loss caused by spoofed WalletConnect offer state that causes the user to approve the wrong amount.

### Likelihood Explanation

**High.** Any WalletConnect-connected dApp can issue a `take_offer` command with an arbitrary offer string. The offer's `infos.royalty_percentage` is not validated against the on-chain NFT royalty anywhere in the GUI. The only prerequisite is that the target NFT exists on-chain (so `nftGetInfo` succeeds and triggers the display override), which is the normal case for any real NFT offer.

### Recommendation

Do not use two independent sources for the royalty percentage in the same confirmation flow. Either:

1. **Use only `nftGetInfo`** for both display and as the authoritative value passed to the wallet backend when accepting the offer (reject offers whose `infos.royalty_percentage` exceeds the on-chain value), or
2. **Validate** that `offerSummary.infos.royalty_percentage` matches the `nftGetInfo` result before rendering the confirmation dialog, and block acceptance if they diverge beyond a tolerance.

The `nftGetInfo` override at line 321 should be the single source of truth, and the offer's `infos` royalty should be cross-checked against it rather than silently diverging.

### Proof of Concept

1. Connect a malicious dApp to the Chia GUI via WalletConnect.
2. The dApp crafts an offer for a real NFT (on-chain `royalty_percentage: 250`, i.e. 2.5%) but embeds `infos.royalty_percentage: 9999` (99.99%) in the offer's driver dict.
3. The dApp sends `chia_wallet.take_offer` with this crafted offer string.
4. `parseCommandDisplay` calls `offerSummaryRoyaltyPercentages` → reads `9999` from `infos`.
5. `parseWalletDeltaItem` calls `nftGetInfo` → succeeds → overrides display to `250`.
6. Confirmation dialog shows `amountWithRoyalties` based on 2.5% (e.g., 1.025 XCH for a 1 XCH offer).
7. User approves.
8. Wallet backend uses `infos.royalty_percentage: 9999` → computes ~1 XCH royalty → total spend ~2 XCH.
9. User loses ~0.975 XCH more than the confirmation dialog indicated.

function offerSummaryRoyaltyPercentages(offerSummary: OfferSummaryForDisplay): AssetRoyaltyPercentages {
  const royaltyPercentages: AssetRoyaltyPercentages = {
    spending: {},
    receiving: {},
  };

  const { infos } = offerSummary;
  if (!isPlainObject(infos)) {
    return royaltyPercentages;
  }

  for (const assetId of Object.keys(offerSummary.requested)) {
    const parsedAssetId = assetId === 'xch' ? '1' : assetId;
    royaltyPercentages.spending[parsedAssetId] = royaltyPercentageForDriverInfo(infos[assetId]);
  }

  for (const assetId of Object.keys(offerSummary.offered)) {
    const parsedAssetId = assetId === 'xch' ? '1' : assetId;
    royaltyPercentages.receiving[parsedAssetId] = royaltyPercentageForDriverInfo(infos[assetId]);
  }

  return royaltyPercentages;

function royaltyPercentagesForSide(lines: DisplayWalletDeltaItem[]): number[] {
  return lines
    .filter((line): line is Extract<DisplayWalletDeltaItem, { kind: 'nft' }> => line.kind === 'nft')
    .map((line) => line.royaltyPercentage)
    .filter(
      (royaltyPercentage): royaltyPercentage is number => royaltyPercentage !== undefined && royaltyPercentage > 0,
    );
}

function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],
): string | undefined {
  if (royaltyPercentages.length === 0 || line.kind === 'nft') {
    return undefined;
  }

  const splitAmount = amount / BigInt(royaltyPercentages.length);
  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;

  if (line.kind === 'xch') {
    return mojoToChiaLocaleString(totalAmount);
  }

  return mojoToCATLocaleString(totalAmount);
}
