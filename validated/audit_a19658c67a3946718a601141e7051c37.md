### Title
Divide-Before-Multiply Bigint Truncation in Multi-NFT Royalty Display Understates Offer Cost in WalletConnect Confirmation - (File: `packages/gui/src/electron/commands/parseCommandDisplay.ts`)

---

### Summary

`formatAmountWithRoyalties` in `parseCommandDisplay.ts` uses a divide-before-multiply pattern with bigint arithmetic when computing the total cost (including royalties) shown to the user in the WalletConnect signing confirmation dialog. When multiple NFTs with royalties are involved, the function first splits the payment amount by the number of NFTs (integer division, truncating), then multiplies each truncated share by the royalty percentage. This systematically understates the displayed `amountWithRoyalties`, causing users to approve offers believing they will pay less than the actual on-chain cost.

---

### Finding Description

In `formatAmountWithRoyalties`:

```typescript
// packages/gui/src/electron/commands/parseCommandDisplay.ts
const splitAmount = amount / BigInt(royaltyPercentages.length);   // ← division first, truncates
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
const totalAmount = amount + royaltyAmount;
``` [1](#0-0) 

The correct formula is to compute each NFT's royalty on the **full** `amount` and sum them:

```
royaltyAmount = Σ (amount × royaltyPercentage_i) / 10_000
```

The buggy formula instead computes:

```
splitAmount = amount / N          (truncated)
royaltyAmount = Σ (splitAmount × royaltyPercentage_i) / 10_000   (each also truncated)
```

This introduces two sources of bigint truncation error. The result is fed into `withRoyaltyTotals` → `walletDeltaToDisplay` → `parseCommandDisplay`, which is called in the Electron main process at line 825 to populate the `display` object shown in the WalletConnect confirmation dialog before the user approves a `take_offer` or `create_offer_for_ids` command. [2](#0-1) 

**Concrete example** (matching the existing test case):

- Payment: `100,000,000` mojos (0.0001 XCH)
- NFT1 royalty: 500 basis points (5%)
- NFT2 royalty: 10 basis points (0.1%)

| | Correct | Buggy (actual code) |
|---|---|---|
| NFT1 royalty | `(100,000,000 × 500) / 10,000 = 5,000,000` | `(50,000,000 × 500) / 10,000 = 2,500,000` |
| NFT2 royalty | `(100,000,000 × 10) / 10,000 = 100,000` | `(50,000,000 × 10) / 10,000 = 50,000` |
| Total shown | `105,100,000` mojos | `102,550,000` mojos |
| Error | — | **~2.4% understated** |

The existing test at line 320 confirms the buggy value (`'0.00010255'`) is the expected output, meaning the understatement is baked in as the accepted behavior. [3](#0-2) 

---

### Impact Explanation

The `amountWithRoyalties` field is the **only** figure shown to the user in the WalletConnect confirmation dialog representing the true total cost of accepting an NFT offer. The actual on-chain royalty enforcement is correct (computed by the wallet backend), so the user approves a transaction displaying a lower cost than they will actually pay. This is a High-severity accounting spoofing issue: a WalletConnect-connected dApp can craft a `take_offer` for multiple NFTs with royalties, and the user will approve it believing the total cost is lower than it actually is, resulting in unexpected asset loss.

---

### Likelihood Explanation

Any unprivileged dApp connected via WalletConnect can trigger this by sending a `take_offer` command for an offer containing two or more NFTs with royalties. The error magnitude grows with the number of NFTs and the royalty percentages involved. No special privileges or leaked secrets are required.

---

### Recommendation

Replace the divide-first pattern with multiply-first: compute each royalty on the full `amount`, then sum:

```typescript
function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],
): string | undefined {
  if (royaltyPercentages.length === 0 || line.kind === 'nft') {
    return undefined;
  }

  // Correct: multiply first, divide last — no premature truncation
  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (amount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;
  ...
}
```

This eliminates the `splitAmount` intermediate division entirely.

---

### Proof of Concept

Using the values from the existing test (two NFTs, royalties 500 and 10, payment 100,000,000 mojos):

**Buggy path:**
```
splitAmount = 100_000_000n / 2n = 50_000_000n
royalty[0]  = (50_000_000n * 500n) / 10_000n = 2_500_000n
royalty[1]  = (50_000_000n * 10n)  / 10_000n =    50_000n
total shown = 102_550_000n  →  "0.00010255 XCH"
```

**Correct path:**
```
royalty[0]  = (100_000_000n * 500n) / 10_000n = 5_000_000n
royalty[1]  = (100_000_000n * 10n)  / 10_000n =   100_000n
total shown = 105_100_000n  →  "0.0001051 XCH"
```

The user is shown `0.00010255 XCH` but the actual transaction costs `0.0001051 XCH` — a ~2.4% understatement that grows with the number of NFTs and royalty magnitudes. [4](#0-3) [5](#0-4)

function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],

  if (royaltyPercentages.length === 0 || line.kind === 'nft') {
    return undefined;
  }

  const splitAmount = amount / BigInt(royaltyPercentages.length);
  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;

  if (line.kind === 'xch') {
    return mojoToChiaLocaleString(totalAmount);
  }

  return mojoToCATLocaleString(totalAmount);
}

function withRoyaltyTotals(
  items: DisplayWalletDeltaItemWithKey[],
  amounts: Record<string, bigint>,
  oppositeSideLines: DisplayWalletDeltaItem[],
): DisplayWalletDeltaItem[] {
  const royaltyPercentages = royaltyPercentagesForSide(oppositeSideLines);

  return items.map(({ key, line }) => {
    if (line.kind === 'nft') {
      return line;
    }

    const amount = amounts[key];
    const amountWithRoyalties = amount ? formatAmountWithRoyalties(line, amount, royaltyPercentages) : undefined;

    return amountWithRoyalties ? { ...line, amountWithRoyalties } : line;
  });
}
