### Title
Unbounded Message Buffer Growth via Arbitrary Waitpoint Injection Enables Memory Exhaustion DoS — (`src/protocol/internal.rs`)

### Summary
The `MessageBuffer` inside `Comms` accepts and stores messages for any `MessageHeader` (channel tag + waitpoint) parsed from raw bytes, with no cap on the number of distinct entries. A malicious participant can call `Protocol::message()` with crafted byte payloads that encode novel, never-polled `MessageHeader` values, causing the internal `HashMap` to grow without bound and exhausting process memory.

### Finding Description
`Comms::push_message` parses a `MessageHeader` from the first 40 bytes of every incoming message and inserts it into a `HashMap<MessageHeader, SubMessageQueue>`:

```rust
// src/protocol/internal.rs lines 286-296
fn push_message(&self, from: Participant, message: MessageData) {
    if message.len() < MessageHeader::LEN {
        return;
    }
    let Some(header) = MessageHeader::from_bytes(&message) else {
        return;
    };
    self.incoming.push(header, from, message);
}
```

`MessageBuffer::push` then calls `or_default()` on the `HashMap` entry, creating a new `SubMessageQueue` (backed by an `unbounded` MPSC channel) for every previously-unseen header:

```rust
// src/protocol/internal.rs lines 236-239
fn push(&self, header: MessageHeader, from: Participant, message: MessageData) {
    let mut messages_lock = self.messages.lock().expect("lock should not fail");
    messages_lock.entry(header).or_default().send(from, message);
}
```

A `MessageHeader` is 40 bytes: 32 bytes of `ChannelTag` (arbitrary) + 8 bytes of `Waitpoint` (arbitrary `u64`). Every distinct 40-byte prefix that passes `from_bytes` creates a new `HashMap` entry. There is no size limit, no eviction, and no deduplication guard anywhere in the call chain.

The public entry point is `Protocol::message()`, implemented by `ProtocolExecutor`:

```rust
// src/protocol/internal.rs lines 512-514
fn message(&mut self, from: Participant, data: MessageData) {
    self.comms.push_message(from, data);
}
```

Any participant in the protocol can call this method with attacker-controlled bytes. The codebase's own test (`attacker_can_fill_message_buffer_with_unused_waitpoints`, lines 532–554) explicitly documents and asserts this behavior, confirming the buffer grows proportionally to the number of injected novel headers.

### Impact Explanation
A malicious participant sends a stream of messages whose first 40 bytes encode distinct, never-polled `MessageHeader` values. Each message allocates a new `HashMap` entry and an unbounded MPSC channel. Because honest protocol code never polls these headers, the entries are never freed. Memory grows linearly (or faster, if the attacker sends multiple messages per header) until the process is OOM-killed, permanently halting DKG, reshare, refresh, presign, or signing for all honest parties sharing that `Comms` instance.

This matches the allowed impact: **Medium — Griefing or resource-exhaustion by a malicious participant causing unbounded memory beyond documented behavior**, and potentially **High — Permanent denial of signing or key generation for honest parties**.

### Likelihood Explanation
The attack is cheap and permissionless within the protocol's trust model. Any participant who is legitimately part of a session (or any party who can deliver messages to an honest node's `Protocol::message()` endpoint) can execute it. No cryptographic material is needed — only the ability to send byte strings. The incubation is immediate: a single burst of crafted messages suffices to exhaust memory.

### Recommendation
1. **Cap the `MessageBuffer` HashMap size**: Reject or drop incoming messages when the number of distinct headers exceeds a bound proportional to `n * max_waitpoints` (where `n` is the participant count and `max_waitpoints` is the maximum number of waitpoints any protocol round uses).
2. **Validate the channel tag and waitpoint before insertion**: Only accept headers whose channel tag matches one of the known, pre-registered channels for the current session, and whose waitpoint falls within the expected range.
3. **Use bounded channels** in `SubMessageQueue` instead of `futures::channel::mpsc::unbounded()`, so that even if a novel header is created, the per-header memory is capped.

### Proof of Concept
The following is a minimal demonstration (analogous to the external report's gas-tracking test) showing unbounded growth:

```rust
// Attacker sends N messages with distinct waitpoints never used by honest code.
let comms = Comms::new();
let attacker = Participant::from(99_u32);
for i in 0..N {
    let header = MessageHeader::new(ChannelTag::root_shared()).with_waitpoint(1_000_000 + i);
    let mut message = header.to_bytes().to_vec();
    message.extend_from_slice(&i.to_le_bytes());
    comms.push_message(attacker, message); // each call allocates a new HashMap entry
}
// HashMap now has N entries; honest code never drains them.
// Memory grows as O(N) with no upper bound.
```

The codebase's own test at [1](#0-0)  confirms this behavior is reachable and asserts the buffer grows to exactly `attack_count` entries.

The unbounded `HashMap` is declared at [2](#0-1) , the uncapped insertion occurs at [3](#0-2) , the unbounded channel is created at [4](#0-3) , and the public entry point is [5](#0-4) .
