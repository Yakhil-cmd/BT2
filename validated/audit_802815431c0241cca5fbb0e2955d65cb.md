### Title
Governance `TokenMappingChangeProposal` Always Fails to Remap an Already-Mapped Non-Source Denom — (`x/cronos/keeper/keeper.go`, `x/cronos/proposal_handler.go`)

### Summary

`NewTokenMappingChangeProposalHandler` delegates to `RegisterOrUpdateTokenMapping`, which for non-source denoms (e.g., `gravity0x...`, `ibc/...`) calls `SetExternalContractForDenom`. That function unconditionally calls `ensureDenomNotMapped` and returns `ErrDenomAlreadyMapped` if the denom already has an active mapping. There is no update path — only create. A passed governance proposal to remap a non-source denom is therefore silently consumed and fails every time, leaving the denom permanently bound to the old contract until two separate proposals (delete + create) are executed.

### Finding Description

**Entrypoint:** `NewTokenMappingChangeProposalHandler` → `k.RegisterOrUpdateTokenMapping` → `SetExternalContractForDenom` → `ensureDenomNotMapped`

In `RegisterOrUpdateTokenMapping`, the non-source-denom branch (the `else` at line 384) has two sub-paths: [1](#0-0) 

- Empty contract → `DeleteExternalContractForDenom` (works correctly)
- Non-empty contract → comment reads `// update the mapping`, but calls `SetExternalContractForDenom`, which is **create-only**

`SetExternalContractForDenom` immediately calls `ensureDenomNotMapped`: [2](#0-1) 

`ensureDenomNotMapped` returns `ErrDenomAlreadyMapped` whenever `GetContractByDenom` finds an existing entry: [3](#0-2) 

The error propagates all the way back through `RegisterOrUpdateTokenMapping` → `NewTokenMappingChangeProposalHandler`: [4](#0-3) 

The governance framework marks the proposal as failed and discards it. The mapping is unchanged.

The existing test case `"Non source token, already exists, error"` in `keeper_test.go` explicitly confirms this behavior — it asserts an error when trying to remap an already-mapped non-source denom: [5](#0-4) 

### Impact Explanation

Any governance proposal of the form `TokenMappingChangeProposal{Denom="gravity0x...", Contract=B}` where the denom is already mapped to contract A will:
1. Pass governance voting (deposit spent, quorum reached)
2. Execute `NewTokenMappingChangeProposalHandler`
3. Fail at `SetExternalContractForDenom` with `ErrDenomAlreadyMapped`
4. Leave the denom mapped to the old/stale/malicious contract A

Every subsequent proposal with a non-empty contract for the same denom will fail identically. The denom is effectively frozen at contract A until a two-step workaround (delete proposal + create proposal) is executed, with an unmapped window in between during which bridge/conversion flows for that denom break.

This matches: **High — Permanent or long-lived inability for honest users or validators to process valid transactions, bridge/conversion flows, precompile calls, or IBC transfers under normal network assumptions.**

### Likelihood Explanation

This is triggered by the normal, expected governance workflow: submit a proposal to update a token mapping. No special attacker capability is required. Any denom that has ever been mapped (which is the normal production state for all active tokens) is affected. The likelihood is high because this is the standard operational path for token mapping governance.

### Recommendation

In the non-source-denom, non-empty-contract branch of `RegisterOrUpdateTokenMapping`, replace the create-only `SetExternalContractForDenom` call with an atomic delete-then-set sequence:

```go
// update the mapping: delete old entry first, then create new one
k.DeleteExternalContractForDenom(ctx, msg.Denom)
if err := k.SetExternalContractForDenom(ctx, msg.Denom, contract); err != nil {
    return err
}
```

Alternatively, implement a dedicated `UpdateExternalContractForDenom` that skips `ensureDenomNotMapped` and atomically replaces the old mapping (cleaning up the old reverse entry).

### Proof of Concept

```
1. Deploy contract A (with code) at address 0xAAAA...
2. Call RegisterOrUpdateTokenMapping{Denom="gravity0xAAAA...", Contract=0xAAAA...}
   → mapping: gravity0xAAAA... → 0xAAAA...
3. Deploy contract B (with code) at address 0xBBBB...
4. Submit TokenMappingChangeProposal{Denom="gravity0xAAAA...", Contract=0xBBBB...}
5. Governance passes (quorum + threshold met)
6. Handler executes → RegisterOrUpdateTokenMapping → SetExternalContractForDenom
   → ensureDenomNotMapped finds existing mapping → returns ErrDenomAlreadyMapped
7. Assert: proposal execution returns error
8. Assert: GetContractByDenom("gravity0xAAAA...") still returns 0xAAAA... (not 0xBBBB...)
9. Submit a second identical proposal → same failure
```

This is directly confirmed by the existing test case `"Non source token, already exists, error"` at `x/cronos/keeper/keeper_test.go:597–629`, which asserts `error: true` for exactly this scenario. [5](#0-4)

func (k Keeper) SetExternalContractForDenom(ctx sdk.Context, denom string, address common.Address) error {
	if err := k.ensureDenomNotMapped(ctx, denom); err != nil {
		return err
	}
	if err := k.ensureContractNotMapped(ctx, denom, address); err != nil {
		return err
	}

	store := ctx.KVStore(k.storeKey)
	store.Set(types.DenomToExternalContractKey(denom), address.Bytes())
	store.Set(types.ContractToDenomKey(address.Bytes()), []byte(denom))
	return nil
}

func NewTokenMappingChangeProposalHandler(k keeper.Keeper) govtypes.Handler {
	return func(ctx sdk.Context, content govtypes.Content) error {
		switch c := content.(type) {
		case *types.TokenMappingChangeProposal:
			if err := c.ValidateBasic(); err != nil {
				return err
			}

			msg := types.MsgUpdateTokenMapping{
				Denom:    c.Denom,
				Contract: c.Contract,
				Symbol:   c.Symbol,
				Decimal:  c.Decimal,
			}
			return k.RegisterOrUpdateTokenMapping(ctx, &msg)
		default:
			return errors.Wrapf(sdkerrors.ErrUnknownRequest, "unrecognized cronos proposal content type: %T", c)
		}
	}
