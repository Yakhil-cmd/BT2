### Title
`wallet_id` Hidden from WalletConnect Send Confirmation — Attacker Can Drain Non-Standard Wallets Without User Awareness - (`packages/gui/src/electron/commands/Commands.ts`)

### Summary

A malicious WalletConnect dApp can specify any `wallet_id` (e.g., a CAT wallet at ID 5) in a `chia_sendTransaction` request. The `wallet_id` parameter is explicitly declared with `hide: true` in the confirmation dialog schema, so the user only sees amount, fee, and address — never which wallet is being debited. The dApp-supplied `wallet_id` overrides the default of `1` and is forwarded verbatim to the wallet daemon after the user approves.

---

### Finding Description

**Step 1 — Schema declares `wallet_id` as hidden but allowlisted**

In `Commands.ts`, the `chia_wallet.send_transaction` command declares `wallet_id` as a valid parameter with `hide: true`, and sets a default of `wallet_id: 1` only for when the dApp omits it: [1](#0-0) 

**Step 2 — `parseDappParams` accepts dApp-supplied `wallet_id` and overrides the default**

`parseDappParams.ts` applies defaults only when the key is `undefined`: [2](#0-1) 

The test suite explicitly confirms this: sending `walletId: 2` results in `wallet_id: 2` in the parsed output, not the default of `1`: [3](#0-2) 

**Step 3 — `humanizeParams` filters out `hide: true` params from the confirmation dialog** [4](#0-3) 

The user's confirmation dialog shows only `amount`, `fee`, and `address`. `wallet_id` is silently excluded.

**Step 4 — `useWalletConnectCommand` only validates fingerprint, not `wallet_id`** [5](#0-4) 

No check is performed to ensure `wallet_id` belongs to the connected fingerprint or is restricted to wallet ID 1.

**Step 5 — `parsedParams` (including attacker-controlled `wallet_id`) is forwarded to the daemon after approval** [6](#0-5) 

---

### Impact Explanation

A malicious dApp can send `chia_sendTransaction({ amount: X, fee: Y, address: attacker_addr, wallet_id: 5 })` where wallet ID 5 is a CAT wallet. The user sees a confirmation dialog showing only the amount, fee, and destination address — with no indication that the debit is from a CAT wallet rather than the standard XCH wallet. Upon approval, the CAT tokens are transferred to the attacker's address. This constitutes an unauthorized transfer of CAT (or any non-standard wallet asset) without the user's informed consent.

The "Raw data" collapsible section in `Confirm.tsx` does render the full `parsedParams` including `wallet_id`, but this requires the user to actively expand a technical debug section and understand its meaning — it is not a security guard. [7](#0-6) 

---

### Likelihood Explanation

Any WalletConnect-paired dApp that has been granted `chia_sendTransaction` permission can exploit this immediately. No additional privileges, leaked keys, or local compromise are required. The dApp simply includes `walletId: N` in its request payload.

---

### Recommendation

Remove `wallet_id` from the dApp-controllable parameter list for `chia_sendTransaction`, or enforce that `wallet_id` is always locked to `1` (standard XCH wallet) for WalletConnect send requests. If multi-wallet support is intentional, `wallet_id` must be removed from `hide: true` and displayed prominently in the confirmation dialog (e.g., showing the wallet name/type and asset type), so the user can make an informed decision about which wallet is being debited.

---

### Proof of Concept

1. Establish a WalletConnect pairing with `chia_sendTransaction` permission.
2. Send the following WalletConnect request:
   ```json
   {
     "method": "chia_sendTransaction",
     "params": {
       "amount": "1000000000000",
       "fee": "0",
       "address": "<attacker_address>",
       "walletId": 5
     }
   }
   ```
3. Observe the confirmation dialog — it shows Amount, Fee, Address, but **no wallet identifier**.
4. User clicks "Send". The daemon receives `send_transaction` with `wallet_id=5`, debiting the CAT wallet.
5. Verify in transaction history that the CAT wallet (ID 5) was debited, not the standard XCH wallet.
