### Title
Circular `account_tx_in_pool_or_recent_block` Guard in `skip_stateful_validations` Admits Signature-Bypassed Invoke Transactions to Mempool Indefinitely — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function is designed to skip `__validate__` (account signature verification) for an invoke transaction with nonce=1 when a corresponding `deploy_account` is pending. The guard it uses — `account_tx_in_pool_or_recent_block` — returns `true` for **any** transaction from the account in the pool, not specifically a `deploy_account`. This creates a self-referential loop: the signature-bypassed invoke transaction itself satisfies the guard for future replacements. Combined with fee escalation, an attacker can keep an invoke transaction with an arbitrary wrong signature in the mempool indefinitely, with no fees ever charged because the account never exists on-chain.

---

### Finding Description

`skip_stateful_validations` in `crates/apollo_gateway/src/stateful_transaction_validator.rs` fires when three conditions hold simultaneously:

1. The transaction is an `Invoke`.
2. `tx.nonce() == Nonce(Felt::ONE)`.
3. `account_nonce == Nonce(Felt::ZERO)` (account not yet deployed on-chain). [1](#0-0) 

When all three hold, the function calls `account_tx_in_pool_or_recent_block` and returns its result as the skip decision: [2](#0-1) 

The comment asserts: *"It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."*

`account_tx_in_pool_or_recent_block` is implemented as:

```rust
self.state.contains_account(account_address)
    || self.tx_pool.contains_account(account_address)
``` [3](#0-2) 

`tx_pool.contains_account` returns `true` for **any** transaction type from that address — including an invoke transaction that itself skipped `__validate__`. There is no type-specific check for `deploy_account`.

When `skip_validate = true`, `run_validate_entry_point` sets `execution_flags.validate = false`, so the blockifier's `StatefulValidator` never calls the account's `__validate__` entry point: [4](#0-3) 

The mempool's `validate_tx` only checks for duplicate nonce and fee-escalation rules — it does not verify that a `deploy_account` exists: [5](#0-4) 

Fee escalation is enabled by default in production (`enable_fee_escalation: true`, `fee_escalation_percentage: 10`): [6](#0-5) 

The default transaction TTL is 60 seconds: [7](#0-6) 

**Attack chain:**

1. Attacker submits a valid `deploy_account` for address A → enters mempool.
2. Attacker submits `Invoke(nonce=1, sig=GARBAGE)` for A → `skip_stateful_validations` returns `true` (deploy_account is in pool) → `__validate__` skipped → enters mempool.
3. After 60 s the `deploy_account` expires and is removed by `remove_expired_txs`. [8](#0-7) 

4. The invoke tx (nonce=1, garbage sig) is still in the pool (submitted later, expires later).
5. `account_tx_in_pool_or_recent_block` still returns `true` because `tx_pool.contains_account(A)` is satisfied by the invoke tx itself.
6. Attacker submits `Invoke(nonce=1, sig=GARBAGE2, fee ≥ 110%)` → fee escalation replaces step-4 tx; `skip_stateful_validations` fires again (step-4 tx is still in pool at check time) → `__validate__` skipped → enters mempool.
7. Repeat step 6 indefinitely.

The invalid invoke transaction persists in the mempool. When the batcher eventually picks it up, execution fails because the account does not exist on-chain; no fee is charged (no balance). The attacker's nominal fee in the transaction fields is never deducted.

---

### Impact Explanation

**High — Mempool/gateway admission accepts invalid transactions before sequencing.**

The gateway's stateful validation is the only place where `__validate__` (account signature) is checked for invoke transactions in the deploy_account+invoke UX path. By exploiting the circular guard, an attacker can permanently seat an invoke transaction with an arbitrary signature in the mempool without paying any fees. This:

- Wastes batcher and blockifier resources on transactions that will always fail.
- Occupies a nonce slot for address A, blocking any legitimate nonce-1 invoke from that address (the mempool rejects it as `DuplicateNonce` unless the attacker's fee is outbid).
- Can be repeated across many fresh addresses to mount a sustained, fee-free mempool DoS.

---

### Likelihood Explanation

**Medium.** The attacker must submit one valid `deploy_account` transaction and wait one TTL period (60 s) for it to expire. After that, the self-sustaining loop requires only fee-escalation replacements, which cost nothing because the account never exists on-chain. The attack is fully unprivileged and requires no special knowledge beyond the ability to craft Starknet transactions.

---

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` guard with a type-specific check that verifies a `deploy_account` transaction (not just any transaction) is present in the pool for address A. For example, expose a `deploy_account_in_pool(address)` query from the mempool and use it in `skip_stateful_validations`. Alternatively, record the deploy_account tx hash when the UX skip is granted and verify it is still present before skipping validation on subsequent calls.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
