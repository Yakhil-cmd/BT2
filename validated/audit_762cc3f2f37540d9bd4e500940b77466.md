The exploit question describes a real vulnerability. Here is the complete analysis:

### Title
Balance Deducted Before `gas_limit` U256→u64 Conversion Fails, Causing Permanent Fund Loss — (`engine/src/engine.rs`)

### Summary

In `submit_with_alt_modexp`, `charge_gas` deducts the sender's ETH balance using the raw U256 `gas_limit` before the subsequent `transaction.gas_limit.try_into()` conversion to `u64` is attempted. If `gas_limit > u64::MAX`, the conversion fails and returns `EngineErrorKind::GasOverflow`, but the balance deduction is never reversed. The NEAR transaction commits the storage write, permanently freezing the sender's funds.

### Finding Description

The execution order in `submit_with_alt_modexp` is:

**Step 1** — Intrinsic/floor gas check (lines 1070–1081). Both `intrinsic_gas` and `floor_gas` return `Result<u64, Error>`, so `core::cmp::max(intrinsic_gas, floor_gas)` is at most `u64::MAX`. The comparison:

```rust
if transaction.gas_limit < core::cmp::max(intrinsic_gas, floor_gas).into() {
    return Err(EngineErrorKind::IntrinsicGasNotMet.into());
}
```

only enforces a **lower bound** on `gas_limit`. A value of `u64::MAX + 1` (as U256) passes this check silently. [1](#0-0) 

**Step 2** — `charge_gas` is called and deducts the balance (lines 1101–1106). Inside `charge_gas`, `transaction.gas_limit` is used directly as U256 for the multiplication:

```rust
let prepaid_amount = fixed_gas
    .map_or(transaction.gas_limit, EthGas::as_u256)
    .checked_mul(effective_gas_price)
    ...
```

`set_balance` is called, writing the reduced balance to NEAR storage. [2](#0-1) 

**Step 3** — `try_into()` fails (lines 1107–1110). After the balance is already deducted:

```rust
let gas_limit = transaction
    .gas_limit
    .try_into()
    .map_err(|_| EngineErrorKind::GasOverflow)?;
```

If `gas_limit > u64::MAX`, this returns `Err(GasOverflow)` immediately. [3](#0-2) 

**Step 4** — `refund_unused_gas` is never reached. The refund logic at lines 1148–1159 is only executed after a successful `engine.call` or `engine.deploy_code`. The early return at step 3 bypasses it entirely. [4](#0-3) 

**Root cause**: `NormalizedEthTransaction.gas_limit` is typed as `U256` and is populated directly from the RLP-decoded transaction field with no upper-bound validation before `charge_gas`. [5](#0-4) 

The `EthGas` type itself is a `u64` wrapper, but `charge_gas` never converts `transaction.gas_limit` to `EthGas` — it uses the raw `U256` field. [6](#0-5) 

### Impact Explanation

**Critical — Permanent freezing of funds.**

A sender with sufficient ETH balance (e.g., ≥ 18.4 ETH for `gas_limit = u64::MAX + 1` at 1 wei gas price) submits a crafted transaction. `charge_gas` deducts the full `gas_limit * gas_price` amount from their balance and writes it to NEAR storage. The subsequent `try_into()` failure causes the function to return an error without refunding. Because the NEAR transaction completes without a panic, all storage writes (including the balance deduction) are committed. The deducted ETH is not credited to any relayer or fee recipient — it is simply gone from the sender's account with no corresponding credit anywhere.

### Likelihood Explanation

**Medium.** Aurora Engine accepts raw transaction bytes via NEAR. The RLP encoding of the gas field is a variable-length big integer; nothing in the parsing pipeline (`EthTransactionKind::try_from`, `NormalizedEthTransaction::try_from`) rejects a value exceeding `u64::MAX`. [7](#0-6) 

Standard Ethereum mempools would reject such a transaction, but Aurora users submit directly to NEAR, bypassing mempool validation. The attacker must control a funded account and craft the RLP manually — straightforward for any developer.

### Recommendation

Add an explicit upper-bound check on `gas_limit` **before** `charge_gas` is called:

```rust
// Reject gas_limit values that cannot be represented as u64
let gas_limit: u64 = transaction
    .gas_limit
    .try_into()
    .map_err(|_| EngineErrorKind::GasOverflow)?;
```

Move this conversion to immediately after the intrinsic/floor gas check (around line 1082), before `charge_gas` is invoked. Then pass the already-validated `u64` value to the EVM executor.

### Proof of Concept

```rust
// Construct a NormalizedEthTransaction with gas_limit = u64::MAX as U256 + 1
let gas_limit_overflow = U256::from(u64::MAX) + U256::one();
let transaction = NormalizedEthTransaction {
    address: sender_address,
    chain_id: Some(chain_id),
    nonce: U256::zero(),
    gas_limit: gas_limit_overflow,
    max_priority_fee_per_gas: U256::one(),
    max_fee_per_gas: U256::one(),
    to: Some(receiver_address),
    value: Wei::zero(),
    data: vec![],
    access_list: vec![],
    authorization_list: vec![],
};

// Fund sender with enough ETH to pass charge_gas:
// prepaid = (u64::MAX + 1) * 1 wei ≈ 18.4 ETH
add_balance(&mut io, &sender_address, Wei::new(gas_limit_overflow));

let balance_before = get_balance(&io, &sender_address);

// submit_with_alt_modexp will:
// 1. Pass intrinsic_gas check (gas_limit > any u64)
// 2. charge_gas succeeds, deducts ~18.4 ETH
// 3. try_into() fails → GasOverflow returned
// 4. refund_unused_gas never called
let result = submit_with_alt_modexp(io, env, &args, state, account_id, relayer, &mut handler);

assert!(matches!(result, Err(e) if e.kind == EngineErrorKind::GasOverflow));

let balance_after = get_balance(&io, &sender_address);
// Balance is permanently reduced — funds are frozen
assert!(balance_after < balance_before);
assert_eq!(balance_after, Wei::zero());
```
