### Title
Missing Failure Callback on `ExitToEthereum` Precompile Causes Permanent Fund Freeze - (File: `engine-precompiles/src/native.rs`)

---

### Summary

The `ExitToEthereum` precompile burns EVM-side tokens (ETH or ERC-20) and schedules a NEAR `withdraw` promise to the ETH connector contract, but attaches **no callback**. If the `withdraw` call fails for any reason, the EVM-side burn is irreversible and the user's funds are permanently frozen. The `ExitToNear` precompile in the same file conditionally attaches an error-recovery callback (`exit_to_near_precompile_callback`), making the asymmetry a concrete, exploitable gap.

---

### Finding Description

In `engine-precompiles/src/native.rs`, `ExitToEthereum::run` constructs the outbound NEAR promise as a bare `PromiseArgs::Create` with no attached callback: [1](#0-0) 

```rust
let withdraw_promise = PromiseCreateArgs {
    target_account_id: nep141_address,
    method: "withdraw".to_string(),
    args: serialized_args,
    attached_balance: Yocto::new(1),
    attached_gas: costs::WITHDRAWAL_GAS,
};

let promise = borsh::to_vec(&PromiseArgs::Create(withdraw_promise)).unwrap();
```

Compare this with `ExitToNear::run`, which conditionally wraps the outbound promise in a `PromiseArgs::Callback` that targets `exit_to_near_precompile_callback` for error recovery: [2](#0-1) 

The callback in `engine/src/contract_methods/connector.rs` handles the failure branch by calling `engine::refund_on_error` to re-mint tokens to the user: [3](#0-2) 

No equivalent recovery path exists for `ExitToEthereum`.

The EVM-side debit is committed **before** the NEAR promise is dispatched:

- For ETH: `context.apparent_value` is deducted from the caller by the EVM before the precompile runs.
- For ERC-20: the ERC-20 contract burns tokens before calling the precompile (the precompile is called by the ERC-20 contract as part of its burn flow). [4](#0-3) 

Once the EVM burn is committed and the NEAR `withdraw` promise fails, there is no code path that can restore the user's balance.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Any user who calls `ExitToEthereum` (either ETH or ERC-20 path) while the ETH connector's `withdraw` method is unavailable or reverts will have their tokens permanently destroyed on the EVM side with no on-chain recovery. The ETH connector is a separate NEAR contract; its `withdraw` can fail due to:

- The connector being paused (a documented operational state).
- The connector's internal NEP-141 balance being insufficient.
- Any bug or revert in the connector's `withdraw` logic.

Because NEAR promises are fire-and-forget here, the engine has no way to detect or react to the failure after the fact.

---

### Likelihood Explanation

**Medium.** The ETH connector is a separately deployed NEAR contract with its own pause mechanism and upgrade lifecycle. A temporary pause during a security incident or upgrade window, combined with a user submitting an `ExitToEthereum` transaction in that window, is a realistic scenario. The precompile is reachable by any unprivileged EVM user or ERC-20 contract that calls the `ExitToEthereum` precompile address (`0xb0bd02f6a392af548bdf1cfaee5dfa0eefcc8eab`). [5](#0-4) 

---

### Recommendation

Mirror the `ExitToNear` pattern: wrap the `withdraw_promise` in a `PromiseArgs::Callback` that targets a new `exit_to_ethereum_precompile_callback` method on the engine. That callback should inspect the promise result and, on failure, re-mint the burned ETH or ERC-20 tokens back to the originating EVM address (analogous to `engine::refund_on_error` used in the `ExitToNear` callback). [6](#0-5) 

---

### Proof of Concept

1. User holds 1 ETH on Aurora and calls the `ExitToEthereum` precompile (flag `0x00`, 20-byte Ethereum recipient).
2. The EVM deducts 1 ETH from the user's balance before the precompile body executes.
3. `ExitToEthereum::run` schedules `PromiseArgs::Create(withdraw_promise)` targeting the ETH connector's `withdraw` method — no callback attached.
4. The ETH connector is paused (or reverts for any reason); the `withdraw` promise fails.
5. NEAR discards the failed promise result. No callback fires. No refund is issued.
6. The user's 1 ETH is permanently gone: burned on the EVM side, never released on the NEAR/Ethereum side.

The same sequence applies to the ERC-20 path (flag `0x01`): the ERC-20 contract burns tokens before calling the precompile, and the same no-callback `withdraw` promise is dispatched. [7](#0-6)

pub fn exit_to_near_precompile_callback<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        env.assert_private_call()?;

        // This function should only be called as the callback of
        // exactly one promise.
        if handler.promise_results_count() != 1 {
            return Err(errors::ERR_PROMISE_COUNT.into());
        }

        let args: ExitToNearPrecompileCallbackArgs = io.read_input_borsh()?;

        let maybe_result = if let Some(PromiseResult::Successful(_)) = handler.promise_result(0) {
            if let Some(args) = args.transfer_near {
                let action = PromiseAction::Transfer {
                    amount: Yocto::new(args.amount),
                };
                let promise = PromiseBatchAction {
                    target_account_id: args.target_account_id,
                    actions: vec![action],
                };

                // Safety: this call is safe because it comes from the exit to near precompile, not users.
                // The call is to transfer the unwrapped wNEAR tokens.
                let promise_id = handler.promise_create_batch(&promise);
                handler.promise_return(promise_id);
            }

            None
        } else if let Some(args) = args.refund {
            // Exit call failed; need to refund tokens
            let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;

            if !refund_result.status.is_ok() {
                return Err(errors::ERR_REFUND_FAILURE.into());
            }

            Some(refund_result)
        } else {
            None
        };

        Ok(maybe_result)
    })
}
