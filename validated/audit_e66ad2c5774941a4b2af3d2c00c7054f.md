### Title
XCC Router Version Not Recorded When `factory_update_address_version` Callback Fails After Successful Deployment — (File: `engine/src/xcc.rs`)

---

### Summary

When an XCC router sub-account is successfully deployed on NEAR but the subsequent `factory_update_address_version` callback fails (e.g., because the engine is paused via `pause_contract` between the two receipts, or because the 5 TGas budget is exhausted), the engine's per-address version record is never written. Every future XCC call from that EVM address then re-enters the `DeployNeeded { create_needed: true }` branch and attempts `CreateAccount` on an account that already exists, causing the entire promise chain to fail permanently. There is no admin escape hatch to manually set the version, so the freeze is irrecoverable without a contract upgrade.

---

### Finding Description

`handle_precompile_promise` in `engine/src/xcc.rs` and `fund_xcc_sub_account` both follow the same two-step pattern when a router must be created for the first time:

**Step 1 — Deploy batch** (lines 226–262 of `engine/src/xcc.rs`):
```
CreateAccount + Transfer + DeployContract + initialize
``` [1](#0-0) 

**Step 2 — Version-recording callback** attached to the deploy batch (lines 263–279):
```rust
let callback = PromiseCreateArgs {
    method: "factory_update_address_version".into(),
    attached_gas: VERSION_UPDATE_GAS,   // 5 TGas
    ...
};
Some(handler.promise_attach_callback(promise_id, &callback))
``` [2](#0-1) 

`VERSION_UPDATE_GAS` is only 5 TGas: [3](#0-2) 

The callback implementation (`factory_update_address_version`) calls `require_running` as its first substantive check: [4](#0-3) 

`require_running` reads `is_paused` from `EngineState` and returns an error if the engine is paused: [5](#0-4) 

`pause_contract` can be called by the owner at any time: [6](#0-5) 

Because NEAR receipts execute asynchronously across blocks, the deploy batch and the callback execute in separate receipts. If the engine is paused between them, `require_running` fails, the callback returns an error, and `set_code_version_of_address` is never called: [7](#0-6) 

The version for that address therefore remains absent from storage. On every subsequent XCC call, `get_code_version_of_address` returns `None`, and `AddressVersionStatus::new` maps `None` unconditionally to `DeployNeeded { create_needed: true }`: [8](#0-7) 

The engine then schedules `CreateAccount` on an account that already exists. NEAR batch actions are atomic; `CreateAccount` on an existing account fails, collapsing the entire promise chain including the user's intended XCC call. The `AddressVersionStatus::UpToDate` branch — the fast path for addresses with an up-to-date router — becomes permanently unreachable for the affected address: [9](#0-8) 

There is no privileged function that can write `code_version_of_address` directly. `set_code_version_of_address` is only reachable through the private callback: [10](#0-9) 

---

### Impact Explanation

**High — Temporary (potentially permanent) freezing of funds.**

Any EVM address whose router was deployed but whose version callback failed loses the ability to make XCC calls. If the user had wNEAR locked in the XCC flow at the time of failure, those tokens are stranded. Because there is no admin path to write the missing version record, the freeze persists until a contract upgrade is deployed that adds a recovery function. In practice this is a permanent freeze until an emergency upgrade is issued.

---

### Likelihood Explanation

**Medium.**

The trigger requires the engine to be paused between two asynchronous NEAR receipts. The owner can call `pause_contract` at any time (e.g., in response to a security incident). Because NEAR receipts for the deploy batch and the callback can span multiple blocks, a pause issued during that window — even briefly — is sufficient. The 5 TGas budget for the callback is also very tight given that `with_hashchain` performs multiple storage reads and writes on top of the core logic, making gas exhaustion a secondary trigger.

---

### Recommendation

1. **Remove `require_running` from `factory_update_address_version`**, or move it after the version write. The callback is already gated by `assert_private_call`; a paused engine should still be able to finalize its own internal bookkeeping for a deployment that already succeeded.

2. **Increase `VERSION_UPDATE_GAS`** to a value that comfortably covers `with_hashchain` overhead plus storage writes (at minimum 10–15 TGas).

3. **Add an owner-only recovery function** (e.g., `admin_set_router_version(address, version)`) so that a stuck address can be manually unblocked without a full contract upgrade.

4. **Add a guard in `AddressVersionStatus::new`** that distinguishes between "account exists on-chain but version not recorded" and "account does not exist", to avoid scheduling `CreateAccount` on an existing account.

---

### Proof of Concept

1. EVM address `0xABCD...` calls the XCC precompile for the first time. `handle_precompile_promise` detects `target_code_version = None`, enters `DeployNeeded { create_needed: true }`, and schedules:
   - Batch receipt R1: `CreateAccount + Transfer + DeployContract + initialize` targeting `abcd....aurora`
   - Callback receipt R2: `factory_update_address_version` with 5 TGas

2. R1 executes successfully in block N. The router sub-account `abcd....aurora` now exists on NEAR.

3. The owner calls `pause_contract` in block N (or N+1, before R2 executes).

4. R2 executes. `factory_update_address_version` calls `require_running`, which returns `ERR_CONTRACT_IS_PAUSED`. The callback fails. `set_code_version_of_address` is never called.

5. Owner calls `resume_contract`.

6. `0xABCD...` calls the XCC precompile again. `get_code_version_of_address` returns `None`. `AddressVersionStatus::new` returns `DeployNeeded { create_needed: true }`. The engine schedules `CreateAccount` on `abcd....aurora`, which already exists. The NEAR batch fails. The user's XCC call fails.

7. Step 6 repeats on every subsequent attempt. The address is permanently unable to use XCC.

pub fn pause_contract<I: IO + Copy, E: Env>(io: I, env: &E) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        let mut state = state::get_state(&io)?;
        require_owner_only(&state, &env.predecessor_account_id())?;
        require_running(&state)?;
        state.is_paused = true;
        state::set_state(&mut io, &state)?;
        Ok(())
    })
