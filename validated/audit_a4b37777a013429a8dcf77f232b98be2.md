### Title
NFT Metadata Integrity Check Bypassed When `metadataHash` Is Absent, Enabling Offer Spoofing — (File: `packages/gui/src/hooks/useFetchAndProcessMetadata.ts`)

---

### Summary

`fetchAndProcessMetadata` skips the SHA-256 integrity check entirely when the on-chain `metadataHash` field is `undefined` or empty. An unprivileged attacker who mints an NFT without a metadata hash and controls the metadata URI server can serve arbitrary metadata content. When a victim opens an offer for that NFT in the GUI, the spoofed metadata (fake name, collection, description, royalty info) is displayed without any integrity warning, causing the victim to approve an offer for a worthless asset.

---

### Finding Description

In `packages/gui/src/hooks/useFetchAndProcessMetadata.ts`, the integrity guard is:

```typescript
if (hash && !compareChecksums(checksum, hash)) {
  throw new Error('Checksum mismatch');
}
``` [1](#0-0) 

The condition `if (hash && ...)` means: when `hash` is `undefined`, `null`, or `""`, the checksum is computed but **never compared**. Execution falls through and the fetched content is parsed and returned as trusted metadata. [2](#0-1) 

This function is called from `useMetadataData`, which passes `metadataHash` directly from the on-chain NFT record:

```typescript
const { metadataUris = [], metadataHash } = nft;
const metadata = await fetchAndProcessMetadata(firstUri, metadataHash);
``` [3](#0-2) 

The Chia NFT1 standard does not require a `metadataHash`. Any NFT minted without one will have `metadataHash = undefined`, causing the guard to be silently bypassed on every metadata fetch.

The analog to the BandPriceFeed bug is exact: just as the Solidity code does not revert when `observations[index].timestamp == 0` (an empty entry) and instead proceeds with a shorter-than-expected interval, the GUI does not reject metadata when `hash` is absent and instead proceeds with fully unvalidated content.

---

### Impact Explanation

An attacker mints an NFT with no `metadataHash` and a `metadataUri` pointing to a server they control. They create an offer (e.g., "send me 50 XCH, receive this NFT"). Their server returns metadata claiming the NFT is a high-value item (e.g., a famous collection piece). When the victim opens the offer in the GUI, `fetchAndProcessMetadata` fetches and accepts the attacker's metadata without any integrity check. The offer viewer renders the spoofed name, collection, description, and attributes. The victim approves the offer and loses 50 XCH in exchange for a worthless NFT.

This satisfies: **High — Corruption or spoofing of NFT metadata that causes a user to approve the wrong asset.**

---

### Likelihood Explanation

- Minting an NFT without a `metadataHash` is a valid, supported operation in the Chia NFT1 standard.
- The attacker controls the metadata URI at mint time and can serve any JSON content.
- No special privileges, leaked keys, or cryptographic breaks are required.
- The victim only needs to open an offer file or link, which is a normal user action.
- The `NFTHashStatus` component does show "Invalid hash" for NFTs whose hash fails verification, but `useNFTVerifyHash` returns `undefined` (not `isVerified: false`) when `hash` is absent:

```typescript
if (!uris || !uris.length || !hash) {
  return undefined;
}
``` [4](#0-3) 

When `findValidUri` returns `undefined`, `data` is `undefined`, `isVerified` is `undefined` (falsy), and `NFTHashStatus` renders "Invalid hash" — but this indicator is not prominently surfaced in the offer acceptance flow and is easily overlooked.

---

### Recommendation

Change the guard in `fetchAndProcessMetadata` to treat an absent hash as an unverifiable (and therefore untrusted) state rather than a pass-through:

```typescript
if (!hash) {
  throw new Error('No metadata hash: content integrity cannot be verified');
}
if (!compareChecksums(checksum, hash)) {
  throw new Error('Checksum mismatch');
}
``` [5](#0-4) 

Alternatively, if hash-less NFTs must be supported, the offer acceptance UI should display a prominent, non-dismissable warning that the NFT's metadata has not been verified and its displayed properties cannot be trusted.

---

### Proof of Concept

1. Mint an NFT on mainnet/testnet with `metadata_hash = ""` and `metadata_uris = ["https://attacker.example/meta.json"]`.
2. Serve the following from `https://attacker.example/meta.json`:
   ```json
   { "name": "Rare CryptoPunk #1", "description": "1-of-1 legendary NFT", "collection": { "name": "CryptoPunks" } }
   ```
3. Create an offer: NFT → 50 XCH.
4. Share the offer file with a victim.
5. Victim opens the offer in Chia GUI. The offer viewer calls `fetchAndProcessMetadata(firstUri, undefined)`. Line 23 evaluates `if (undefined && ...)` → false; the checksum comparison is skipped. The attacker's JSON is parsed and displayed.
6. Victim sees "Rare CryptoPunk #1 / CryptoPunks collection" and accepts the offer, sending 50 XCH.
7. Victim receives the worthless NFT. [6](#0-5) [7](#0-6)
