### Title
Unprotected `new` Initialization Function Allows Any Caller to Seize Engine Ownership — (`engine/src/contract_methods/admin.rs`)

---

### Summary

The `new` function in `engine/src/contract_methods/admin.rs` performs no caller authentication. Any NEAR account can invoke it during the window between contract deployment and legitimate initialization, setting themselves as the engine `owner_id`. The owner role controls upgrade authority, full-access-key attachment, and all privileged engine operations, making this a critical auth-bypass with direct fund-theft and permanent-freeze impact.

---

### Finding Description

`admin::new` is the sole initialization entry point for the Aurora Engine state. It is exposed as the public WASM export `new()` in `engine/src/lib.rs`:

```rust
// engine/src/lib.rs  lines 76-83
#[unsafe(no_mangle)]
pub extern "C" fn new() {
    let io = Runtime;
    let env = Runtime;
    contract_methods::admin::new(io, &env)
        .map_err(ContractError::msg)
        .sdk_unwrap();
}
```

The implementation in `admin.rs` guards only against *re-initialization* — it does not restrict *who* may call it:

```rust
// engine/src/contract_methods/admin.rs  lines 55-88
pub fn new<I: IO + Copy, E: Env>(mut io: I, env: &E) -> Result<(), ContractError> {
    if state::get_state(&io).is_ok() {
        return Err(b"ERR_ALREADY_INITIALIZED".into());
    }
    // ← no predecessor_account_id / caller check here
    let input = io.read_input().to_vec();
    let args = NewCallArgs::deserialize(&input)...;
    let state: EngineState = args.into();   // owner_id comes from args
    state::set_state(&mut io, &state)?;
    Ok(())
}
```

`NewCallArgs` carries `owner_id` (and `chain_id`, `upgrade_delay_blocks`, etc.). Whoever calls `new` first supplies these values and becomes the engine owner. There is no check that the caller is the contract account itself, the NEAR deployer account, or any other trusted party.

In NEAR, contract deployment (`DeployContract` action) and initialization (`new` function call) are separate steps unless explicitly batched. If the legitimate deployer submits them as two independent transactions — or if any future deployment to a new account omits the atomic batch — an attacker observing the mempool can submit a `new` call with attacker-controlled `owner_id` before the deployer's initialization transaction is included.

The analog to the reported `CollateralsWallet` bug is exact:

| `CollateralsWallet` | Aurora Engine |
|---|---|
| `setHoldefiContract()` — no access control | `new()` — no caller check |
| Sets `holdefiContract` storage var | Sets `owner_id` in `EngineState` |
| Attacker calls `withdraw` afterward | Attacker calls `attach_full_access_key` / `upgrade` afterward |

---

### Impact Explanation

Once an attacker controls `owner_id`, every owner-gated method becomes attacker-controlled:

- **`attach_full_access_key`** (`admin.rs` lines 483–513) — adds a full-access key to the engine NEAR account, giving the attacker unrestricted on-chain control of the account and all its NEAR balance.
- **`upgrade`** (`admin.rs` lines 169–206) — deploys arbitrary WASM to the engine account, allowing replacement with a contract that drains all ERC-20 balances bridged through the connector.
- **`stage_upgrade` + `upgrade`** — same path with a configurable delay.
- **`pause_contract`** — permanently freezes all user funds.
- **`set_owner`** — transfers ownership to any address, locking out the legitimate team.

This satisfies both **Critical: Direct theft of user funds** and **Critical: Permanent freezing of funds**.

---

### Likelihood Explanation

NEAR's standard deployment pattern does not enforce atomic deploy+init. The vulnerability window exists whenever:

1. A fresh deployment is made to a new account (mainnet, testnet, or any silo instance) without batching `DeployContract` and the `new` call in a single transaction batch.
2. A future re-deployment to a new account ID follows the same non-atomic pattern.

An attacker monitoring the NEAR mempool or block explorer for a `DeployContract` action targeting the engine account can immediately submit a `new` call with attacker-controlled arguments. The NEAR block time (~1 s) gives a realistic race window. Likelihood is **medium** for any deployment that does not enforce atomic batching.

---

### Recommendation

1. **Enforce atomic initialization**: Always batch `DeployContract` and the `new` function call in a single NEAR batch transaction. This eliminates the race window entirely.
2. **Add a caller restriction** inside `new`: verify that `env.predecessor_account_id() == env.current_account_id()` (i.e., only a self-call or the deploying account is accepted), mirroring the pattern used in `factory_update_address_version` and `withdraw_wnear_to_router` which use `env.assert_private_call()`.
3. Alternatively, use NEAR SDK's `#[init]` macro with an explicit predecessor check so the guard is enforced at the framework level.

---

### Proof of Concept

1. Attacker watches the NEAR chain for a `DeployContract` action targeting the Aurora Engine account (e.g., `aurora`).
2. In the same or next block, attacker submits a call to `aurora::new` with:
   ```json
   { "owner_id": "attacker.near", "chain_id": [...], "upgrade_delay_blocks": 0 }
   ```
3. If the deployer has not yet called `new`, `state::get_state(&io).is_ok()` returns `Err`, the guard passes, and `state::set_state` writes attacker-controlled state.
4. Attacker calls `aurora::attach_full_access_key` with their own public key — succeeds because `require_owner_only` now resolves to `attacker.near`.
5. Attacker now holds a full-access key to the `aurora` account and can drain all bridged assets or deploy malicious WASM. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

#[named]
pub fn new<I: IO + Copy, E: Env>(mut io: I, env: &E) -> Result<(), ContractError> {
    if state::get_state(&io).is_ok() {
        return Err(b"ERR_ALREADY_INITIALIZED".into());
    }

    let input = io.read_input().to_vec();
    let args = NewCallArgs::deserialize(&input).map_err(|_| errors::ERR_BORSH_DESERIALIZE)?;

    let initial_hashchain = args.initial_hashchain();
    let state: EngineState = args.into();

    if let Some(block_hashchain) = initial_hashchain {
        let block_height = env.block_height();
        let mut hashchain = Hashchain::new(
            state.chain_id,
            env.current_account_id(),
            block_height,
            block_hashchain,
        );

        hashchain.add_block_tx(
            block_height,
            function_name!(),
            &input,
            &[],
            &Bloom::default(),
        )?;
        crate::hashchain::save_hashchain(&mut io, &hashchain)?;
    }

    state::set_state(&mut io, &state)?;
    Ok(())
}

pub fn upgrade<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<(), ContractError> {
    let state = state::get_state(&io)?;
    require_running(&state)?;
    require_owner_only(&state, &env.predecessor_account_id())?;

    let input = io.read_input().to_vec();
    let (code, state_migration_gas) = match UpgradeParams::try_from_slice(&input) {
        Ok(args) => (
            args.code,
            args.state_migration_gas
                .map_or(GAS_FOR_STATE_MIGRATION, NearGas::new),
        ),
        Err(_) => (input, GAS_FOR_STATE_MIGRATION), // Backward compatibility
    };

    let target_account_id = env.current_account_id();
    let batch = PromiseBatchAction {
        target_account_id,
        actions: vec![
            PromiseAction::DeployContract { code },
            PromiseAction::FunctionCall {
                name: "state_migration".to_string(),
                args: vec![],
                attached_yocto: ZERO_YOCTO,
                gas: state_migration_gas,
            },
        ],
    };
    let promise_id = handler.promise_create_batch(&batch);

    handler.promise_return(promise_id);

    Ok(())
}

pub fn attach_full_access_key<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<(), ContractError> {
    let state = state::get_state(&io)?;

    require_running(&state)?;
    require_owner_only(&state, &env.predecessor_account_id())?;

    let public_key = serde_json::from_slice::<FullAccessKeyArgs>(&io.read_input().to_vec())
        .map(|args| args.public_key)
        .map_err(|_| errors::ERR_JSON_DESERIALIZE)?;
    let current_account_id = env.current_account_id();
    let action = PromiseAction::AddFullAccessKey {
        public_key,
        nonce: 0, // not actually used - depends on block height
    };
    let promise = PromiseBatchAction {
        target_account_id: current_account_id,
        actions: vec![action],
    };
    // SAFETY: This action is dangerous because it adds a new full access key (FAK) to the Engine account.
    // However, it is safe to do so here because of the `require_owner_only` check above; only the
    // (trusted) owner account can add a new FAK.
    let promise_id = handler.promise_create_batch(&promise);

    handler.promise_return(promise_id);

    Ok(())
}

pub fn require_owner_only(
