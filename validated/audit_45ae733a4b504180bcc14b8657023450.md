### Title
Unauthenticated `sender` in Bank Precompile `transfer` Allows Any Contract to Drain Arbitrary Holders' EVM-Denominated Tokens - (File: `x/cronos/keeper/precompiles/bank.go`)

---

### Summary

The bank precompile's `transfer` method accepts a `sender` address as a caller-supplied ABI argument and executes `bankKeeper.SendCoins(ctx, from, to, ...)` without ever verifying that `sender == contract.Caller()`. Any EVM contract can therefore call `bank.transfer(victimAddress, attackerAddress, amount)` and drain the victim's `evm/<callerContract>` native-module tokens without the victim's consent.

---

### Finding Description

The bank precompile at `x/cronos/keeper/precompiles/bank.go` exposes four methods: `mint`, `burn`, `balanceOf`, and `transfer`. The denom for all four is `evm/<contract.Caller()>`, meaning each contract controls its own EVM-denominated native token.

For `mint` and `burn`, the only address argument is the recipient/burner, and the operation is gated on the calling contract being the issuer of that denom — this is correct.

For `transfer`, however, the implementation is:

```go
sender := args[0].(common.Address)   // ← arbitrary, caller-supplied
recipient := args[1].(common.Address)
...
from := sdk.AccAddress(sender.Bytes())
to   := sdk.AccAddress(recipient.Bytes())
...
denom := EVMDenom(contract.Caller())  // evm/<callerContract>
...
bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt))
```

There is **no check** that `sender == contract.Caller()`. The `from` address is taken verbatim from the ABI input. The only guard applied to the `from` side is the implicit balance check inside `SendCoins`; there is no authorization check. [1](#0-0) 

Compare with the `exec` helper in `utils.go`, which explicitly enforces `caller == signer` for all other precompile paths: [2](#0-1) 

The bank precompile's `transfer` case has no equivalent guard.

---

### Impact Explanation

**Critical — Unauthorized transfer of precompile-controlled assets.**

Any user who holds `evm/<contractAddress>` tokens (received via the contract's `mint` call, or via a prior `transfer`) can have those tokens drained by the contract at `<contractAddress>` at any time, without signing any transaction or granting any approval. The attacker is the contract deployer (or anyone who can call the contract), and the victim is any address holding that contract's EVM-denominated native tokens.

This maps directly to the allowed Critical impact: *"Unauthorized … transfer … of … precompile-controlled assets."*

---

### Likelihood Explanation

The bank precompile is a live, reachable precompile at address `0x0000000000000000000000000000000000000064`. Any deployed EVM contract can call it. The `transfer` ABI signature is `transfer(address,address,uint256)`. A malicious contract needs only to call `IBankModule(0x64).transfer(victimAddress, attackerAddress, balance)` in a single transaction. No privileged access, leaked keys, or governance action is required. [3](#0-2) 

---

### Recommendation

Add a caller-authentication guard at the top of the `TransferMethodName` case, mirroring the pattern used in `utils.go`:

```go
case TransferMethodName:
    ...
    sender := args[0].(common.Address)
    // ADD: enforce that only the calling contract can move its own denom tokens
    if sender != contract.Caller() {
        return nil, errors.New("sender is not authenticated: must equal caller")
    }
    ...
```

This ensures the bank precompile's `transfer` method can only be invoked by the token-issuing contract on behalf of itself, consistent with the authorization model of `mint` and `burn`.

---

### Proof of Concept

1. Deploy `MaliciousContract` on Cronos EVM.
2. Victim calls `MaliciousContract.mint(victimAddress, 1000)` (or receives tokens via any path), acquiring 1000 `evm/<MaliciousContract>` native tokens.
3. Attacker calls `MaliciousContract.drain(victimAddress, attackerAddress)`, which internally executes:
   ```solidity
   IBankModule(0x64).transfer(victimAddress, attackerAddress, 1000);
   ```
4. The bank precompile executes `bankKeeper.SendCoins(ctx, victimAddr, attackerAddr, 1000 evm/<MaliciousContract>)` with no authorization check.
5. Victim's balance is zero; attacker holds 1000 `evm/<MaliciousContract>` tokens.

The `TestBank.sol` integration contract already demonstrates the `transfer` call pattern: [4](#0-3) 

A malicious variant simply passes an arbitrary `sender` instead of `msg.sender`.
