### Title
`RewardSupplier.claim_rewards` counts in-transit L1→L2 tokens as settled credit, causing temporary freeze of unclaimed yield — (`File: src/reward_supplier/reward_supplier.cairo`)

---

### Summary

The `RewardSupplier` contract tracks tokens requested from L1 but not yet received via `l1_pending_requested_amount`. In `request_funds`, these in-transit tokens are counted as available `credit`. However, `claim_rewards` attempts to transfer actual on-chain tokens from the contract's balance. If the balance is insufficient because tokens are still in-transit through the StarkGate bridge, the transfer panics and reverts, temporarily freezing all staker and delegator reward claims.

---

### Finding Description

In `request_funds` (lines 301–331 of `src/reward_supplier/reward_supplier.cairo`), the contract computes:

```cairo
let credit = balance + l1_pending_requested_amount;
let debit = unclaimed_rewards;
if credit < debit + threshold { ... send mint requests ... }
```

`l1_pending_requested_amount` represents tokens that have been requested from L1 via L2→L1 messages but have **not yet arrived** at the L2 contract. They are in-transit through the StarkGate bridge, waiting for the L1 `RewardSupplier.tick()` to be called, which mints tokens and deposits them back via `bridge().depositWithMessage(...)`, eventually triggering `on_receive` on L2.

The threshold is `base_mint_amount / 2` (line 14 of `src/reward_supplier/utils.cairo`). The system guarantees `credit >= debit + threshold`, but `credit` includes in-transit tokens. The actual on-chain `balance` can be zero while `l1_pending_requested_amount` is large.

When `claim_rewards` is called (lines 205–220):

```cairo
fn claim_rewards(ref self: ContractState, amount: Amount) {
    let unclaimed_rewards = self.unclaimed_rewards.read();
    assert!(amount <= unclaimed_rewards, "{}", GenericError::AMOUNT_TOO_HIGH);
    self.unclaimed_rewards.write(unclaimed_rewards - amount);
    let token_dispatcher = self.token_dispatcher.read();
    token_dispatcher.checked_transfer(recipient: staking_contract, amount: amount.into());
}
```

The only guard is `amount <= unclaimed_rewards`. There is **no check** that the actual token balance is sufficient. If `balance < amount` (because tokens are still in-transit), `checked_transfer` panics and the entire call reverts.

The call chain from a staker or delegator is:

1. `staking.claim_rewards(staker_address)` → `send_rewards_to_staker` → `claim_from_reward_supplier`
2. `claim_from_reward_supplier` (lines 50–60 of `src/staking/utils.cairo`) calls `reward_supplier.claim_rewards(amount)` and then asserts `balance_after - balance_before == amount`, meaning any revert in step 2 propagates up and blocks the staker's claim entirely.

---

### Impact Explanation

Any staker or delegator calling `claim_rewards` during the window between a mint request being sent to L1 and `on_receive` being called on L2 will have their transaction revert. This is a **temporary freeze of unclaimed yield**: rewards are correctly accrued and recorded in `unclaimed_rewards`, but cannot be transferred out until the bridge delivers the tokens. This matches the allowed impact: **Temporary freezing of unclaimed yield**.

---

### Likelihood Explanation

This is a normal operational state. Every epoch, `update_unclaimed_rewards_from_staking_contract` is called, which calls `request_funds`. If the balance is below `debit + threshold`, new L2→L1 mint messages are sent and `l1_pending_requested_amount` is incremented. The L1→L2 bridge round-trip takes time (L1 block finality + bridge processing). During this window — which is a routine, recurring state — any `claim_rewards` call that requires more tokens than the current balance will revert. Given that `balance` can be zero while `l1_pending_requested_amount` covers the full `unclaimed_rewards`, this window can block all reward claims simultaneously.

---

### Recommendation

In `claim_rewards`, add a check that the actual token balance is sufficient before attempting the transfer:

```cairo
let balance: Amount = token_dispatcher
    .balance_of(account: get_contract_address())
    .try_into()
    .expect_with_err(InternalError::BALANCE_ISNT_AMOUNT_TYPE);
assert!(balance >= amount, "{}", Error::INSUFFICIENT_BALANCE_FOR_CLAIM);
```

Alternatively, `request_funds` should ensure that `balance` alone (not `credit`) is always sufficient to cover `unclaimed_rewards`, by not counting `l1_pending_requested_amount` as immediately available when deciding whether to trigger new mint requests.

---

### Proof of Concept

1. Protocol starts with `balance = 0`, `l1_pending_requested_amount = 0`, `unclaimed_rewards = STRK_IN_FRIS` (initial seed).
2. Staker stakes and attestations occur over several epochs. `update_unclaimed_rewards_from_staking_contract(rewards: R)` is called, setting `unclaimed_rewards = STRK_IN_FRIS + R`.
3. `request_funds` fires: `credit = 0 + 0 = 0 < (STRK_IN_FRIS + R) + base_mint_amount/2`. Mint messages are sent; `l1_pending_requested_amount = N * base_mint_amount` where `N * base_mint_amount >= unclaimed_rewards + threshold`. Balance remains 0.
4. Before `on_receive` is called (L1 bridge not yet processed), staker calls `staking.claim_rewards(staker_address)`.
5. Staking contract calls `claim_from_reward_supplier(amount: staker_unclaimed_rewards_own)`.
6. `reward_supplier.claim_rewards(amount)`: `amount <= unclaimed_rewards` passes. `checked_transfer(staking_contract, amount)` panics because `balance = 0 < amount`.
7. Entire transaction reverts. Staker cannot claim their earned rewards until `on_receive` is eventually called. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)

        fn request_funds(ref self: ContractState, unclaimed_rewards: Amount) {
            // Read current balance.
            let token_dispatcher = self.token_dispatcher.read();
            let balance: Amount = token_dispatcher
                .balance_of(account: get_contract_address())
                .try_into()
                .expect_with_err(InternalError::BALANCE_ISNT_AMOUNT_TYPE);

            // Calculate credit, which is the contract's balance plus the amount already requested
            // from L1, and the debit, which is the unclaimed rewards.
            let mut l1_pending_requested_amount = self.l1_pending_requested_amount.read();
            let credit = balance + l1_pending_requested_amount;
            let debit = unclaimed_rewards;

            // If there isn't enough credit to cover the debit + threshold, request funds.
            let base_mint_amount = self.base_mint_amount.read();
            let threshold = compute_threshold(base_mint_amount);
            if credit < debit + threshold {
                let diff = debit + threshold - credit;
                let num_msgs = ceil_of_division(dividend: diff, divisor: base_mint_amount);
                let total_amount = num_msgs * base_mint_amount;
                for _ in 0..num_msgs {
                    self.send_mint_request_to_l1_reward_supplier();
                }
                self.emit(Events::MintRequest { total_amount, num_msgs });
                l1_pending_requested_amount += total_amount;
            }

            // Commit to storage the requested amount, which is now part of the credit.
            self.l1_pending_requested_amount.write(l1_pending_requested_amount);
        }

pub(crate) fn claim_from_reward_supplier(
    reward_supplier_dispatcher: IRewardSupplierDispatcher,
    amount: Amount,
    token_dispatcher: IERC20Dispatcher,
) {
    let staking_contract = get_contract_address();
    let balance_before = token_dispatcher.balance_of(account: staking_contract);
    reward_supplier_dispatcher.claim_rewards(:amount);
    let balance_after = token_dispatcher.balance_of(account: staking_contract);
    assert!(balance_after - balance_before == amount.into(), "{}", Error::UNEXPECTED_BALANCE);
}
