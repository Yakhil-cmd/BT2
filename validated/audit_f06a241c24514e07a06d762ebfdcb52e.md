Audit Report

## Title
Unchecked Inter-Canister Call Result in `burn_and_log` Causes ICP to Become Permanently Stuck in CMC Subaccount - (File: rs/nns/cmc/src/main.rs)

## Summary
The `burn_and_log` function in the Cycles Minting Canister silently discards ledger call failures, logging the error and returning `()` with no error propagation. Because the corresponding block is already marked as successfully processed in `blocks_notified` before `burn_and_log` is called, any subsequent retry of the notification returns the cached success result immediately, making the ICP permanently unrecoverable from the CMC subaccount without a governance-authorized canister upgrade.

## Finding Description
In `rs/nns/cmc/src/main.rs`, `burn_and_log` is an `async fn` returning `()` with an intentional design decision documented in its comment:

> "Burning doesn't return errors - we don't want to reject the transaction notification because then it could be retried." [1](#0-0) 

The function makes a `call_protobuf` inter-canister call to the ICP ledger's `send_pb` method. On failure, it only logs the error: [2](#0-1) 

`burn_and_log` is called after each successful payment operation — `process_create_canister` (line 1945), `process_mint_cycles` (line 1968), and `process_top_up` (line 2001) — all of which return `Ok(...)` to their callers regardless of whether the burn succeeded. [3](#0-2) [4](#0-3) [5](#0-4) 

After these functions return `Ok(...)`, the caller inserts a terminal `NotifiedCreateCanister(Ok(...))`, `NotifiedTopUp(Ok(...))`, or `NotifiedMint(Ok(...))` status into `blocks_notified`: [6](#0-5) 

The `is_transient_error` check only removes the entry for `Err` variants where `is_retriable()` returns `true`. An `Ok(...)` result is never transient, so the entry is never removed: [7](#0-6) [8](#0-7) 

On any subsequent call with the same `block_index`, the occupied entry is matched and the cached `Ok(...)` is returned immediately, with no re-attempt of the burn: [9](#0-8) 

The ICP remains in the CMC subaccount (e.g., `Subaccount::from(&controller)` or `Subaccount::from(&canister_id)`) indefinitely. There is no heartbeat, timer, or retry queue for failed burns. Recovery requires a governance-approved canister upgrade.

## Impact Explanation
This is a **High** severity finding. If the ICP ledger rejects the `send_pb` call with a transient error (e.g., `SysTransient` during a ledger upgrade window, or a full outgoing message queue), the ICP sent by the user is permanently locked in a CMC subaccount. The user receives a successful response (canister created, topped up, or cycles minted) but their ICP is neither burned nor refundable. This constitutes a permanent, per-transaction loss of user funds with no on-chain recovery path, matching the allowed impact: *"Significant NNS/SNS or infrastructure security impact with concrete user or protocol harm"* and *"permanent loss... of in-scope ledger assets."* Additionally, ICP that should be removed from supply remains in circulation, creating a ledger conservation discrepancy.

## Likelihood Explanation
The ICP ledger is a system canister and is generally reliable, but transient inter-canister call rejections are a documented reality on the Internet Computer during ledger upgrades, subnet instability, or when the CMC's outgoing message queue is full. The `send_pb` call uses `fee: Tokens::ZERO` (a minting burn), so insufficient-balance failures are not the concern — call-level rejections (`SysTransient`, queue-full) are the realistic trigger. Any user who successfully calls `notify_create_canister`, `notify_top_up`, or `notify_mint_cycles` during such a window triggers the condition. The window is narrow but realistic, and the condition is permanent once triggered.

## Recommendation
1. Return a `Result` from `burn_and_log` and propagate the error to the caller. If the burn fails, do not insert the terminal `Notified*` status (or insert a special `BurnFailed` status), allowing the user to retry the notification which will re-attempt the burn.
2. Alternatively, store failed burn attempts (subaccount + amount) in stable state and retry them in a canister timer/heartbeat, similar to how the CMC already retries other deferred operations.
3. At minimum, emit a certified metric counter for failed burns so operators can detect accumulation of stuck ICP and initiate a governance upgrade to recover funds.

## Proof of Concept
1. User sends ICP to the CMC subaccount for `controller` with `MEMO_CREATE_CANISTER` and calls `notify_create_canister` with the resulting `block_index`.
2. CMC sets `blocks_notified[block_index] = Processing`, calls `process_create_canister`, which calls `do_create_canister` — succeeds, new canister created.
3. CMC calls `burn_and_log(Subaccount::from(&controller), amount).await`.
4. The ICP ledger rejects the `send_pb` call with `SysTransient` (e.g., ledger is mid-upgrade).
5. `burn_and_log` logs the error and returns `()`.
6. `process_create_canister` returns `Ok(new_canister_id)`.
7. CMC sets `blocks_notified[block_index] = NotifiedCreateCanister(Ok(new_canister_id))`.
8. User retries `notify_create_canister` with the same `block_index` — CMC immediately returns the cached `Ok(new_canister_id)` without re-attempting the burn.
9. ICP remains in `Subaccount::from(&controller)` of the CMC permanently.

A deterministic integration test using `StateMachine` (as used in `rs/nns/integration_tests/src/cycles_minting_canister.rs`) can reproduce this by: (a) pausing/upgrading the ledger canister mid-flow to force a `SysTransient` rejection of the `send_pb` call, (b) completing `notify_create_canister`, (c) verifying the CMC subaccount balance is non-zero after the call returns `Ok`, and (d) verifying a retry returns the cached `Ok` without draining the subaccount.

/// Attempt to burn the funds.
/// Burning doesn't return errors - we don't want to reject the transaction
/// notification because then it could be retried.
async fn burn_and_log(from_subaccount: Subaccount, amount: Tokens) {
    let msg = format!("Burning of {amount} ICPTs from subaccount {from_subaccount}");
    let minting_account_id = with_state(|state| state.minting_account_id);
    if minting_account_id.is_none() {
        print(format!("{msg} failed: minting_account_id not set"));
        return;
    }
    let minting_account_id = minting_account_id.unwrap();
    let ledger_canister_id = with_state(|state| state.ledger_canister_id);

    if amount < DEFAULT_TRANSFER_FEE {
        print(format!("{msg}: amount too small ({amount})"));
        return;
    }

    let send_args = SendArgs {
        memo: Memo::default(),
        amount,
        fee: Tokens::ZERO,
        from_subaccount: Some(from_subaccount),
        to: minting_account_id,
        created_at_time: None,
    };
    let res: CallResult<BlockIndex> = call_protobuf(ledger_canister_id, "send_pb", send_args).await;

    match res {
        Ok(block) => print(format!("{msg} done in block {block}.")),
        Err((code, err)) => {
            let code = code as i32;
            print(format!("{msg} failed with code {code}: {err:?}"))
        }
    }
}
