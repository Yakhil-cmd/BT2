### Title
Missing Callback/Refund Mechanism in `ExitToEthereum` Precompile Causes Permanent ERC-20 Token Loss on NEAR-Side Withdrawal Failure — (`etc/eth-contracts/contracts/EvmErc20V2.sol`)

---

### Summary

`withdrawToEthereum` in `EvmErc20V2.sol` burns ERC-20 tokens before invoking the `ExitToEthereum` precompile, which schedules only a bare `PromiseArgs::Create` (fire-and-forget) NEAR promise. Unlike `ExitToNear`, there is no `exit_to_ethereum_precompile_callback` and no `refund_on_error` path. If the NEAR-side `withdraw` call fails for any reason, the burned ERC-20 tokens are permanently destroyed with no Ethereum unlock and no re-mint, violating the token-supply invariant and causing insolvency. A compounding issue is that the Solidity assembly call return value is never checked, so even a precompile-level failure (e.g., precompile paused) does not revert the burn.

---

### Finding Description

**Step 1 — ERC-20 burn before async promise**

In `EvmErc20V2.sol`, `withdrawToEthereum` unconditionally burns the caller's tokens before invoking the precompile:

```solidity
// etc/eth-contracts/contracts/EvmErc20V2.sol, lines 66-77
function withdrawToEthereum(address recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // ← burn happens first, unconditionally

    bytes32 amount_b = bytes32(amount);
    bytes20 recipient_b = bytes20(recipient);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient_b);
    uint input_size = 1 + 32 + 20;

    assembly {
        let res := call(gas(), 0xb0bd02f6a392af548bdf1cfaee5dfa0eefcc8eab, 0, add(input, 32), input_size, 0, 32)
        // res is NEVER checked — precompile failure does not revert the burn
    }
}
``` [1](#0-0) 

**Step 2 — `ExitToEthereum` precompile emits only a bare `Create` promise, no callback**

The precompile constructs a `PromiseArgs::Create(withdraw_promise)` — a simple fire-and-forget call to the NEP-141 connector's `withdraw` method. There is no `PromiseArgs::Callback` wrapping it, and no `exit_to_ethereum_precompile_callback` method exists anywhere in the codebase:

```rust
// engine-precompiles/src/native.rs, lines 977-985
let withdraw_promise = PromiseCreateArgs {
    target_account_id: nep141_address,
    method: "withdraw".to_string(),
    ...
};
let promise = borsh::to_vec(&PromiseArgs::Create(withdraw_promise)).unwrap();
// ← PromiseArgs::Create only; no Callback wrapper, no error handler
``` [2](#0-1) 

**Step 3 — Contrast with `ExitToNear`, which has a full callback + refund path**

`ExitToNear` wraps its promise in `PromiseArgs::Callback` pointing to `exit_to_near_precompile_callback`, which calls `engine::refund_on_error` to re-mint burned ERC-20 tokens if the NEAR-side call fails:

```rust
// engine-precompiles/src/native.rs, lines 470-483
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs {
        base: transfer_promise,
        callback: PromiseCreateArgs {
            method: "exit_to_near_precompile_callback".to_string(),
            ...
        },
    })
};
``` [3](#0-2) 

The `exit_to_near_precompile_callback` handler explicitly re-mints tokens on failure:

```rust
// engine/src/contract_methods/connector.rs, lines 231-239
} else if let Some(args) = args.refund {
    // Exit call failed; need to refund tokens
    let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;
    ...
}
``` [4](#0-3) 

`refund_on_error` re-mints via `ERC20_MINT_SELECTOR` for ERC-20 exits:

```rust
// engine/src/engine.rs, lines 1184-1203
if let Some(erc20_address) = args.erc20_address {
    // ERC-20 exit; re-mint burned tokens
    ...
    engine.call(&erc20_admin_address, &erc20_address, Wei::zero(), input, ...)
}
``` [5](#0-4) 

**No analogous `exit_to_ethereum_precompile_callback` exists** — confirmed by exhaustive search returning zero matches.

---

### Impact Explanation

There are two concrete failure paths, both resulting in permanent ERC-20 token destruction:

**Path A — NEAR-side `withdraw` fails after precompile succeeds:**
The precompile call succeeds (EVM-level), the `PromiseArgs::Create` promise is scheduled, but the NEAR-side `withdraw` on the connector fails (connector paused, NEAR-side accounting discrepancy, or connector contract error). Since there is no callback, no re-mint occurs. The ERC-20 supply is permanently reduced without any corresponding Ethereum unlock. The user loses tokens and receives nothing on Ethereum.

**Path B — Precompile itself fails (unchecked return value):**
If the `ExitToEthereum` precompile is paused (`ERR_PAUSED`), the assembly `call` returns `0`. Since `res` is never checked, the Solidity function does not revert. `_burn` has already executed. Tokens are permanently destroyed with no promise ever scheduled.

Both paths violate the invariant: *total ERC-20 supply on Aurora + locked tokens on Ethereum = total NEP-141 supply in connector*. The result is insolvency — ERC-20 tokens are destroyed without a corresponding Ethereum unlock.

---

### Likelihood Explanation

- **Path A** requires the NEAR-side connector `withdraw` to fail. The most realistic trigger is the connector being paused (a normal operational action) while the `ExitToEthereum` precompile remains unpaused — a realistic operational window. A NEAR-side accounting discrepancy (e.g., from a prior exploit or bug) is a lower-probability but non-negligible trigger.
- **Path B** requires the `ExitToEthereum` precompile to be paused while `withdrawToEthereum` is still callable. This is a realistic operational state (admin pauses precompile for maintenance but the ERC-20 contract is not upgraded).
- No special privileges are required from the attacker — any EVM user holding ERC-20 tokens can call `withdrawToEthereum`.
- The `error_refund` feature flag that gates the `ExitToNear` refund path has no equivalent for `ExitToEthereum`, making this a structural gap rather than a configuration issue.

---

### Recommendation

1. **Add a callback to `ExitToEthereum`**: Mirror the `ExitToNear` pattern — wrap the `withdraw_promise` in `PromiseArgs::Callback` pointing to a new `exit_to_ethereum_precompile_callback` method. That callback should call `engine::refund_on_error` (with `erc20_address` set) to re-mint burned tokens if the NEAR-side `withdraw` fails.

2. **Check the assembly return value in `withdrawToEthereum`**: The Solidity contract should revert if the precompile call returns `0`:
   ```solidity
   assembly {
       let res := call(...)
       if iszero(res) { revert(0, 0) }
   }
   ```
   This prevents the burn from being committed when the precompile itself fails.

3. **Ensure operational pausing is atomic**: When the connector is paused, the `ExitToEthereum` precompile should be paused simultaneously to prevent the burn-without-withdrawal window.

---

### Proof of Concept

```
1. Deploy Aurora Engine locally with the eth-connector.
2. Mint ERC-20 tokens to address Alice (e.g., 1000 tokens).
3. Pause the eth-connector on the NEAR side (or simulate a connector `withdraw` failure).
4. Alice calls withdrawToEthereum(bob_eth_address, 1000) on EvmErc20V2.
5. Observe:
   a. Alice's ERC-20 balance = 0 (burn succeeded).
   b. The ExitToEthereum precompile schedules PromiseArgs::Create(withdraw).
   c. The NEAR-side `withdraw` call fails (connector paused).
   d. No exit_to_ethereum_precompile_callback is invoked.
   e. No re-mint occurs.
   f. Bob's Ethereum address receives nothing.
   g. Total ERC-20 supply decreased by 1000 with no corresponding Ethereum unlock.
6. Assert: Alice's balance = 0, Bob received 0 ETH/tokens, supply invariant broken.
```

For Path B (unchecked return value), additionally pause the `ExitToEthereum` precompile before step 4 and observe that the transaction still succeeds (no revert) despite the precompile returning failure, with the burn committed and no promise scheduled.
