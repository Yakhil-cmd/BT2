### Title
`skip_stateful_validations` admits invoke transactions with invalid signatures when a victim account has a pending nonce-0 transaction in the mempool - (File: crates/apollo_gateway/src/stateful_transaction_validator.rs)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator is designed to skip the `__validate__` entry-point call for an invoke transaction with `nonce == 1` when the account's on-chain nonce is `0` and the account has any transaction in the mempool or a recent block. This UX feature is intended for the deploy-account + invoke pattern (where the account does not yet exist on-chain). However, the guard condition does not distinguish between "account does not exist yet (deploy_account pending)" and "account already exists with nonce 0 (invoke nonce-0 pending)." In the second case the account exists and `__validate__` could and should be called, but it is silently skipped. An unprivileged attacker who observes a victim account with a pending nonce-0 invoke in the mempool can inject a nonce-1 invoke with an arbitrary/invalid signature that bypasses gateway signature verification and is admitted to the mempool.

### Impact Explanation

An attacker who observes that victim account `X` (on-chain nonce = 0) has a valid pending invoke(nonce=0) in the mempool can submit `invoke(nonce=1, signature=<garbage>)` for account `X`. The gateway evaluates:

- `tx.nonce() == 1` ✓  
- `account_nonce == 0` ✓  
- `account_tx_in_pool_or_recent_block(X)` → `true` (the victim's nonce-0 tx is in the pool) ✓  

`skip_validate = true` is returned, `__validate__` is never called, and the forged transaction is forwarded to the mempool. The transaction carries an invalid signature but is accepted as a valid admission decision. This matches the impact category: **High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

When the batcher later executes the transaction it will call `__validate__`, which will fail, causing the transaction to be rejected without a state change or fee charge. However, the invalid transaction has already consumed mempool capacity and batcher execution resources, and the attack can be repeated for every account that has a pending nonce-0 transaction.

### Likelihood Explanation

The mempool is a public data structure; pending transactions and their sender addresses are observable. Any account that has just been deployed (on-chain nonce = 0) and has submitted its first invoke transaction is a valid target. The attack requires no privileged access, no special funds, and no knowledge of the victim's private key. The only constraint is that the victim's nonce-0 transaction must still be pending in the mempool at the time of the attack.

### Recommendation

Inside `skip_stateful_validations`, narrow the skip condition to the case where the account genuinely does not yet exist on-chain. Concretely, after confirming `account_nonce == 0`, additionally verify that the account has no deployed class hash (i.e., `state.get_class_hash_at(sender_address) == ClassHash::default()`). Only when the account is undeployed should the `__validate__` call be skipped. If the account already exists (class hash is non-zero), the skip must not apply even if a nonce-0 transaction is pending.

Alternatively, restrict `account_tx_in_pool_or_recent_block` to return `true` only when the pending transaction for that address is specifically a `DeployAccount` transaction, which is the only case where skipping `__validate__` is semantically correct.

### Proof of Concept

1. Deploy account `X` (class hash `C`, salt `S`) so it exists on-chain with nonce = 0.
2. Submit a valid `invoke(sender=X, nonce=0, calldata=..., signature=<valid>)` through the gateway. This passes `__validate__` and lands in the mempool.
3. Construct `invoke(sender=X, nonce=1, calldata=<arbitrary>, signature=<all zeros>)`.
4. Submit the forged transaction to the gateway.
5. Gateway evaluation:
   - `validate_nonce`: `account_nonce(0) <= tx_nonce(1) <= account_nonce + max_gap` → passes.
   - `validate_by_mempool`: nonce ordering check passes.
   - `skip_stateful_validations`: `tx.nonce==1`, `account_nonce==0`, `account_tx_in_pool_or_recent_block(X)==true` → returns `true`.
   - `run_validate_entry_point` with `skip_validate=true` → `__validate__` is never called.
6. The forged transaction is admitted to the mempool with an invalid signature.
7. Observe the forged transaction in the mempool snapshot; confirm it carries the zero signature.
8. When the batcher executes it, `__validate__` fails and the transaction is rejected — but it was admitted to the mempool, confirming the invariant violation. [7](#0-6)

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
```
