### Title
WalletConnect Pair Cannot Be Revoked While WalletConnect Is Disabled, Allowing Persistent Dapp Access After Intended Revocation - (File: `packages/gui/src/components/walletConnect/WalletConnectProvider.tsx`)

### Summary

When a user disables WalletConnect, the `handleDisconnectPair` callback requires an initialized WalletConnect client to revoke a pair. Because disabling WalletConnect sets the client to `undefined`, any attempt to disconnect an existing pair silently fails. The pair record persists in `dapp-pairs.yaml`. When WalletConnect is later re-enabled, the dapp's session is still valid and it can execute spending commands — including those marked as bypass (always-allow, no confirmation) — without the user's knowledge.

### Finding Description

`useWalletConnectClient` sets `client` to `undefined` when `enabled` is `false`: [1](#0-0) 

The `handleDisconnectPair` callback in `WalletConnectProvider` gates the entire revocation path on `currentClient` being non-null: [2](#0-1) 

When `currentClient` is `undefined`, it throws `'Client not initialized'`. `processError` only silences `'No matching key'` errors and re-throws everything else: [3](#0-2) 

The module-level `disconnectPair` — which calls `window.permissionsAPI.revokePair(topic)` to remove the pair from the store — is never reached: [4](#0-3) 

`SettingsIntegration` loads pairs regardless of the `enabled` flag and presents a "Disconnect" button, but swallows the resulting error as a `console.warn`, giving the user false confidence that the pair was removed: [5](#0-4) 

The pair record therefore remains in `dapp-pairs.yaml` via `pairStore`: [6](#0-5) 

When WalletConnect is re-enabled, `cleanupPairingsAndSessions` only disconnects sessions whose pairing topic is **absent** from the pairs list. Because the pair was never removed, the dapp's session survives: [7](#0-6) 

`dispatchPairRequest` then authorizes commands for the still-registered pair, including bypassed spending commands that execute without any user confirmation dialog: [8](#0-7) 

### Impact Explanation

A dapp that was granted spending commands (e.g., `chia_sendTransaction`, `chia_takeOffer`, `chia_cancelOffer`) with bypass (always-allow) can continue to execute those commands without user confirmation after the user disables WalletConnect and attempts to disconnect the pair. The user believes the connection was severed; the pair and its bypass permissions remain fully active. This constitutes unauthorized spend/transfer of XCH, CAT, or NFT assets.

### Likelihood Explanation

Users commonly disable WalletConnect as a quick security measure when they suspect a dapp is misbehaving. The Settings > Integration page shows the "Disconnect" button and appears to succeed (no visible error, UI resets the selection). The pair persists silently. Any re-enabling of WalletConnect — even for a different dapp — restores the attacker dapp's access within the WalletConnect session TTL (~7 days).

### Recommendation

Revocation must not depend on the WalletConnect client being initialized. Split the revocation path so that `window.permissionsAPI.revokePair(topic)` is always called first (removing the pair from the store), and the WalletConnect SDK disconnect is attempted only as a best-effort cleanup:

```ts
// In handleDisconnectPair — revoke the pair record unconditionally
await window.permissionsAPI.revokePair(topic);

// Best-effort: also disconnect the live WC session if the client is available
if (currentClient) {
  await disconnectPair(currentClient, topic);
}
```

Additionally, `WalletConnectConnections` should display existing pairs even when `enabled` is `false`, so users can revoke them without needing to re-enable WalletConnect.

### Proof of Concept

1. Connect a dapp via WalletConnect; grant `chia_sendTransaction` with bypass (always allow).
2. Disable WalletConnect in Settings > Integration.
3. In Settings > Integration, select the dapp and click "Disconnect". The UI resets with no error shown.
4. Inspect `dapp-pairs.yaml` — the pair record is still present.
5. Re-enable WalletConnect.
6. From the dapp, send a `chia_sendTransaction` session request. `dispatchPairRequest` finds the pair, the command is in `bypass`, and the transaction is executed without any confirmation dialog.

```typescript
async function cleanupPairingsAndSessions(client: Client) {
  try {
    const pairs = await window.permissionsAPI.getPairs();

    // disconnect all sessions that are not in the pairs list
    const clientSessions = client.session.getAll();

    const sessionsToDisconnect = clientSessions.filter((s) => !pairs.some((p) => p.topic === s.pairingTopic));
    await Promise.all(
      sessionsToDisconnect.map((session) =>
        client.disconnect({
          topic: session.topic,
          reason: getSdkError('USER_DISCONNECTED'),
        }),
      ),
    );

  } catch (e) {
    log('Cleanup pairings error', e);

  }
}
```

```typescript
async function disconnectPair(client: Client, topic: string) {
  try {
    // revoke pair from main process
    await window.permissionsAPI.revokePair(topic);

    // get all sessions for the pairing
    const sessions = client.session.getAll();
    const relatedSessions = sessions.filter((s) => s.pairingTopic === topic);

    // disconnect all related sessions
    await Promise.all(
      relatedSessions.map((session) =>
        client.disconnect({
          topic: session.topic,
          reason: getSdkError('USER_DISCONNECTED'),
        }),
      ),
    );

    // disconnect the pairing
    await client.core.pairing.disconnect({ topic });
  } catch (e) {
    processError(e as Error);
  }
}
```
