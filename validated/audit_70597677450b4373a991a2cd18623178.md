Audit Report

## Title
ICRC-21 Consent Message Fails for u256 Token Amounts Exceeding `u64::MAX` via `nat_to_u64` - (`packages/icrc-ledger-types/src/icrc21/responses.rs`)

## Summary
The `nat_to_u64` helper in `responses.rs` converts `Nat` token amounts to `u64` for the `FieldsDisplay` consent message path. For the u256 ledger variant (used by ckETH and all ckERC20 tokens), any transfer amount exceeding `u64::MAX` (~18.4 ETH) causes `icrc21_canister_call_consent_message` to return a `GenericError` instead of a valid consent message. Additionally, `convert_tokens_to_string_representation` uses `to_f64()` which silently loses precision for amounts above 2^53 wei (~0.009 ETH), producing factually incorrect consent messages on the `GenericDisplay` path.

## Finding Description

**Root cause 1 — `FieldsDisplay` hard failure:**

`nat_to_u64` at `packages/icrc-ledger-types/src/icrc21/responses.rs` lines 329–334 calls `BigUint::to_u64()`, which returns `None` for any value exceeding `u64::MAX`, propagated as `Err(GenericError { error_code: 500 })`: [1](#0-0) 

This is called unconditionally in `add_amount`, `add_fee`, `add_allowance`, and `add_existing_allowance` when the display type is `FieldsDisplay`: [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) 

The `Value::TokenAmount` variant hard-codes `amount: u64`, which is structurally incompatible with u256 ledger amounts: [6](#0-5) 

**Root cause 2 — `GenericDisplay` silent precision loss:**

`convert_tokens_to_string_representation` uses `BigUint::to_f64()`, which (per `num_traits`) never returns `None` for a `BigUint` — it always succeeds but silently rounds to the nearest representable `f64`. The `ok_or` guard is dead code; the function always proceeds with a rounded value: [7](#0-6) 

**Exploit path:**

The public `#[update]` endpoint `icrc21_canister_call_consent_message` in the ICRC-1 ledger canister passes the raw transfer amount directly into these helpers with no pre-check: [8](#0-7) 

The u256 ledger variant (used for ckETH, ckUSDC, and all ckERC20 tokens) is compiled with `feature = "u256-tokens"` and supports amounts up to 2^256: [9](#0-8) 

Any unprivileged caller submitting a `ConsentMessageRequest` with `device_spec: Some(FieldsDisplay)` and a transfer amount of `u64::MAX + 1` or greater receives `Err(GenericError { error_code: 500, description: "Failed to convert tokens to u64" })`. No existing guard in `build_icrc21_consent_info` or `build_icrc21_consent_info_for_icrc1_and_icrc2_endpoints` checks the amount range before dispatching to `add_amount`. [10](#0-9) 

## Impact Explanation

**FieldsDisplay (High):** Hardware wallets implementing ICRC-21 (e.g., Ledger hardware wallet) exclusively use the `FieldsDisplay` path. Any ckETH or ckERC20 user holding more than ~18.4 ETH who attempts to authorize a full-balance or large transfer via a hardware wallet receives a hard error from the consent message endpoint and cannot complete the signing flow. This is a concrete, user-facing denial of the hardware wallet signing path for a production mainnet ck-token, qualifying as a significant ck-token security impact with concrete user harm under the High bounty tier.

**GenericDisplay (Medium):** For all ckETH transfers above ~0.009 ETH (2^53 wei), the displayed amount in the consent message is silently rounded, meaning the consent message shown to the user does not accurately represent the transaction being signed. This undermines the core trust guarantee of ICRC-21.

## Likelihood Explanation

ckETH is a production mainnet canister. Transfers exceeding 18.4 ETH are routine for institutional users and DeFi protocols. The endpoint is a standard public `#[update]` method reachable by any unprivileged ingress sender with no preconditions, rate limits, or authentication requirements. The failure is deterministic and 100% reproducible for any amount above `u64::MAX`.

## Recommendation

1. Change `Value::TokenAmount.amount` from `u64` to `Nat` (or at minimum `u128`) and update the Candid `.did` files accordingly. Remove `nat_to_u64` and pass the `Nat` directly.
2. Replace the `to_f64()` conversion in `convert_tokens_to_string_representation` with exact integer arithmetic using `BigUint` division and modulo to produce the integer and fractional parts separately, as is already done correctly in the ckBTC/ckDOGE minter `format_amount` helpers.
3. Add integration tests for `icrc21_canister_call_consent_message` on the u256 ledger with amounts exceeding `u64::MAX` for both `FieldsDisplay` and `GenericDisplay` paths.

    })?;

}
