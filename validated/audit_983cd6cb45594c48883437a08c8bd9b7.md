### Title
`skip_stateful_validations` Bypasses `__validate__` for Invoke nonce=1 via Non-Deploy-Account Mempool Entry — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

`skip_stateful_validations` is intended to skip blockifier `__validate__` only when a `deploy_account` transaction is pending in the mempool (UX feature: submit deploy_account + invoke together). However, the check delegates to `account_tx_in_pool_or_recent_block`, which returns `true` for **any** transaction type in the pool for that account — not exclusively `deploy_account`. An attacker who can arrange for any transaction (e.g., a valid invoke with nonce=0) to exist in the mempool for an account whose on-chain nonce is 0 can then submit a second invoke with nonce=1 carrying an **invalid or forged signature**, and the gateway will accept it into the mempool without ever calling `__validate__`.

### Finding Description

The relevant logic is in `skip_stateful_validations`: [1](#0-0) 

The condition at line 437 fires when `tx.nonce() == 1 && account_nonce == 0`. When both hold, the function calls: [2](#0-1) 

The mempool implementation of `account_tx_in_pool_or_recent_block` is: [3](#0-2) 

It checks `self.state.contains_account(account_address) || self.tx_pool.contains_account(account_address)` — **no transaction-type filter**. The test at line 1114–1118 of `fee_mempool_test.rs` confirms this returns `true` after adding a generic (non-deploy-account) invoke: [4](#0-3) 

The code comment at line 441–443 claims this is safe because the account "either has a deploy_account transaction or transactions with future nonces that passed validations." This reasoning is **incorrect**: a nonce=0 invoke that passed `__validate__` proves the account is deployed, but it says nothing about the validity of a subsequent nonce=1 invoke submitted by a different party.

When `skip_validate = true` is returned, `run_validate_entry_point` sets `validate: false` in `ExecutionFlags`: [5](#0-4) 

The blockifier's `StatefulValidator::validate` is then called with `validate: false`, meaning `__validate__` (the account's signature-checking entrypoint) is **never invoked** during gateway admission.

---

### Impact Explanation

An invalid invoke transaction with nonce=1 and an arbitrary or forged signature is admitted to the mempool without any account signature verification. This directly satisfies the **High** impact criterion: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

Concrete consequences:
- **Mempool griefing / slot squatting**: An attacker who observes a legitimate account's nonce=0 invoke in the mempool can immediately submit a nonce=1 invoke with a bad signature. The invalid transaction occupies the nonce=1 slot. The legitimate user's nonce=1 transaction is either rejected (duplicate nonce) or must fee-escalate. The invalid transaction eventually fails during execution, but the damage to ordering is done.
- **Signature bypass at admission**: The gateway's security invariant — every invoke transaction has its `__validate__` entrypoint called before admission — is violated.

---

### Likelihood Explanation

The preconditions are:
1. **Account with nonce=0 in state and deployed contract code.** This occurs for accounts deployed via the `deploy` syscall (factory pattern), which is a standard Starknet pattern. It does **not** require deploy_account.
2. **A valid nonce=0 invoke in the mempool for that account.** Either the legitimate account owner submitted one, or the attacker controls the account and submitted one themselves.
3. **Attacker submits nonce=1 invoke with invalid signature.** Trivially constructable.

All three conditions are achievable by an unprivileged external actor with no special access.

---

### Recommendation

Replace the type-agnostic `account_tx_in_pool_or_recent_block` check with an explicit check that the pending transaction for the account is specifically a `deploy_account` transaction. The mempool should expose a dedicated query such as `deploy_account_tx_in_pool_or_recent_block(address)` that filters by transaction type, or the gateway should inspect the transaction type of the pending entry before deciding to skip validation.

---

### Proof of Concept

**Logical trace (no privileged access required):**

1. Deploy account `X` via a factory contract (`deploy` syscall). On-chain state: `nonce(X) = 0`, contract code present.
2. Submit `Invoke(sender=X, nonce=0, valid_signature)` to the gateway. This passes `validate_nonce` (0 ∈ [0, 0+gap]), passes `validate_by_mempool`, `skip_stateful_validations` returns `false` (nonce ≠ 1), `__validate__` is called and succeeds. Transaction enters mempool.
3. Submit `Invoke(sender=X, nonce=1, FORGED_SIGNATURE)` to the gateway.
   - `validate_nonce`: 1 ∈ [0, 0+gap] — passes (assuming `max_allowed_nonce_gap ≥ 1`).
   - `validate_by_mempool`: nonce gap check passes.
   - `skip_stateful_validations`: nonce=1 ✓, account_nonce=0 ✓, `account_tx_in_pool_or_recent_block(X)` → `true` (step 2 tx is in pool) → returns `true`.
   - `run_validate_entry_point` called with `skip_validate=true` → `ExecutionFlags { validate: false }` → `__validate__` is **never called**.
4. The forged-signature invoke is accepted into the mempool. `account_nonce` in state is still 0. The legitimate user's nonce=1 transaction is now blocked or displaced.

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
```
