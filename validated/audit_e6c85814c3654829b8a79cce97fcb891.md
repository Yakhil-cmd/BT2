### Title
Unbounded Loop in `calculate_rewards` Enables Permanent Freezing of Pool Member Unclaimed Rewards — (File: src/pool/pool.cairo)

---

### Summary

The `calculate_rewards` internal function in `Pool.cairo` contains an explicitly acknowledged unbounded loop that iterates over every balance-change entry in a pool member's `pool_member_epoch_balance` trace since their last reward claim. A delegator who makes one balance change per epoch across many epochs without ever claiming rewards will accumulate a trace large enough to push a future `claim_rewards` call past the Starknet block gas limit, permanently locking their unclaimed yield with no recovery path short of a contract upgrade.

---

### Finding Description

**Root cause — `calculate_rewards` loop, `src/pool/pool.cairo` lines 857–877:**

```cairo
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
    if pool_member_checkpoint.epoch() >= until_epoch {
        break;
    }
    let to_sigma = self.find_sigma(pool_member_checkpoint, curr_epoch: until_epoch);
    rewards += compute_rewards_rounded_down(...);
    from_sigma = to_sigma;
    from_balance = pool_member_checkpoint.balance();
    entry_to_claim_from += 1;
}
```

The loop bound is `pool_member_trace_length − entry_to_claim_from`. `entry_to_claim_from` is only advanced when `claim_rewards` succeeds and writes back `pool_member_info.entry_to_claim_from`. [1](#0-0) 

Each call to `enter_delegation_pool`, `exit_delegation_pool_intent`, or `increase_delegate` that lands in a new epoch inserts a fresh checkpoint into `pool_member_epoch_balance` via `set_member_balance` / `trace.insert`. [2](#0-1) 

`claim_rewards` is the only caller of `calculate_rewards` that writes the updated `entry_to_claim_from` back to storage. [3](#0-2) 

There is no cap on how many trace entries may accumulate between two successive `claim_rewards` calls. The developers themselves flag this: *"This loop is unbounded but unlikely to exceed gas limits."* [4](#0-3) 

**Analog to the external report:** The external report describes a recursive function that is missing a `processing.add(name)` guard, causing infinite recursion. Here the analogous missing guard is an upper bound on loop iterations: without it, a sufficiently large trace causes the transaction to exhaust gas, mirroring the "maximum call stack exceeded" crash in the Tact compiler.

---

### Impact Explanation

When `claim_rewards` runs out of gas, the transaction reverts. The pool member's `entry_to_claim_from` and `reward_checkpoint` are never updated. On every subsequent attempt the loop must re-traverse the same (or larger) trace and will again exhaust gas. Because the current contract provides no partial-claim or trace-pruning mechanism, the pool member's accumulated unclaimed STRK rewards are permanently inaccessible — matching the allowed impact **"Permanent freezing of unclaimed yield"** (High).

---

### Likelihood Explanation

A delegator who programmatically adjusts their delegation once per epoch (e.g., a smart-contract vault that rebalances every epoch) and defers reward claims will grow the trace at one entry per epoch. Starknet's per-transaction gas limit is finite; at some epoch count N the loop cost exceeds it. The exact N depends on the current gas schedule, but the protocol imposes no epoch-count limit on delegation activity, making this reachable without any privileged access. Likelihood is **Low-Medium** (requires sustained activity without claiming, but no special permissions).

---

### Recommendation

1. **Short term:** Enforce a maximum number of trace entries processed per `claim_rewards` call (e.g., cap at 500 iterations) and store the updated `entry_to_claim_from` even on a partial run, allowing the delegator to call `claim_rewards` multiple times to drain the backlog.
2. **Long term:** Add a protocol-level invariant that bounds `pool_member_trace_length − entry_to_claim_from` (e.g., by requiring a claim before the gap exceeds a threshold), or emit a warning event when the gap grows large.

---

### Proof of Concept

1. Delegator `D` enters a pool and makes **one balance change per epoch** (e.g., `increase_delegate(1)`) for `N` epochs, never calling `claim_rewards`.
2. After `N` epochs, `pool_member_epoch_balance` for `D` contains `N` checkpoints; `entry_to_claim_from` is still `0`.
3. `D` (or `D`'s reward address) calls `claim_rewards(D)`.
4. `calculate_rewards` enters the `while` loop and iterates `N` times, each iteration calling `find_sigma` (which itself reads storage). Gas consumption is `O(N)`.
5. For sufficiently large `N`, the transaction exceeds the block gas limit and reverts.
6. Every future `claim_rewards` attempt faces the same (or larger) loop and also reverts.
7. `D`'s accumulated unclaimed STRK rewards are permanently frozen with no on-chain recovery path.
