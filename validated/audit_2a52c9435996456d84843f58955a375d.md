### Title
`PresignOutput` Derives `Clone` and `rerandomize_presign` Accepts a Reference, Providing No Enforcement of the Mandatory One-Time-Use Invariant - (`src/ecdsa/ot_based_ecdsa/mod.rs`, `src/ecdsa/robust_ecdsa/presign.rs`)

---

### Summary

The external report identifies a missing circuit-breaker: Gravity.sol had no `pause()` mechanism to halt operations once an exploit was in progress. The structural analog in this library is that `PresignOutput` — whose one-time-use property is a hard cryptographic security requirement — derives `Clone` and is accepted by `rerandomize_presign` via shared reference (`&PresignOutput`). The library therefore provides no type-level or runtime mechanism to prevent presignature reuse. A malicious coordinator or any participant who retains a copy of their `PresignOutput` can silently reuse it across multiple signing sessions, producing signatures with multiplicatively related nonces from which the aggregate secret key can be extracted.

---

### Finding Description

Both ECDSA sub-schemes document that each `PresignOutput` must be consumed **exactly once**:

> "Each output is consumed **exactly once** (one-time use)."
> — `src/ecdsa/ot_based_ecdsa/README.md`, line 12

> "Never reuse a presignature, even across failed, aborted, or partially completed signing sessions."
> — `docs/ecdsa/robust_ecdsa/signing.md`, line 176–177

Despite this, the OT-based `PresignOutput` is declared as:

```rust
// src/ecdsa/ot_based_ecdsa/mod.rs, line 40
#[derive(Debug, Clone, Serialize, Deserialize, Eq, PartialEq, ZeroizeOnDrop)]
pub struct PresignOutput {
    pub big_r: AffinePoint,
    pub k: Scalar,
    pub sigma: Scalar,
}
```

The `Clone` derive means any caller can duplicate the struct before passing it onward. The rerandomization entry point then accepts it by reference:

```rust
// src/ecdsa/ot_based_ecdsa/mod.rs, line 66
pub fn rerandomize_presign(
    presignature: &PresignOutput,
    args: &RerandomizationArguments,
) -> Result<Self, ProtocolError>
```

Because the function takes `&PresignOutput` rather than consuming `PresignOutput` by value, the caller retains the original and can call `rerandomize_presign` again with a different `RerandomizationArguments` (different message hash `h` and tweak `ε`). The same pattern applies to `RerandomizedPresignOutput`, which also derives `Clone` (line 54), and to the robust ECDSA `PresignOutput` (same structural issue in `src/ecdsa/robust_ecdsa/presign.rs`).

There is no usage counter, no consumed flag, no `Arc<AtomicBool>`, and no type-state wrapper that would prevent a second call. The library's only protection is documentation.

---

### Impact Explanation

The signing specification states explicitly:

> "if different subsets of size at least 2t+1 sign different (h, ε) values using shares derived from the same presignature, the resulting signatures use multiplicatively related nonces and the secret key can be recovered using standard ECDSA nonce-reuse attacks."
> — `docs/ecdsa/robust_ecdsa/signing.md`, lines 151–154

Two signatures `(r, s₁)` and `(r, s₂)` produced from the same nonce `k` satisfy:

```
s₁ = k⁻¹(h₁ + r·x)
s₂ = k⁻¹(h₂ + r·x)
```

Subtracting: `s₁ − s₂ = k⁻¹(h₁ − h₂)`, so `k = (h₁ − h₂)/(s₁ − s₂)`, and then `x = (s₁·k − h₁)/r`. The aggregate private key is fully recovered from two signatures.

**Impact class: Critical** — Extraction/reconstruction of the aggregate secret key material.

---

### Likelihood Explanation

The attack requires only that a single participant (or the coordinator) retains their `PresignOutput` after the first signing session and initiates a second signing session using the same presignature with a different message. Because `Clone` is a public, derived trait, this is a one-line operation (`let backup = presign_output.clone()`). No cryptographic capability is needed. A malicious coordinator who orchestrates signing sessions is the most natural attacker, but any participant who stores their own output can trigger the same outcome unilaterally. The library's pipeline explicitly allows the coordinator to choose which presignature to pair with which message, making this a realistic operational scenario.

---

### Recommendation

1. **Consume by value**: Change `rerandomize_presign` to take `presignature: PresignOutput` (owned), so the type system enforces single use at the call site.
2. **Remove `Clone` from `PresignOutput` and `RerandomizedPresignOutput`**: Deriving `Clone` on secret-nonce-bearing types directly contradicts the one-time-use requirement and should be removed. `ZeroizeOnDrop` is already present; `Clone` undermines it.
3. **Type-state wrapper**: Wrap `PresignOutput` in a newtype that tracks whether it has been consumed (e.g., via an `Option` or a `Cell<bool>`) and panics or errors on a second use, as a defense-in-depth measure for callers who bypass the by-value API.

---

### Proof of Concept

```rust
// Attacker (coordinator or participant) retains a clone before first use
let presign_out: PresignOutput = run_presign_protocol(...);
let backup = presign_out.clone(); // Clone is derived — compiles fine

// First signing session: message h1
let rerand1 = RerandomizedPresignOutput::rerandomize_presign(
    &presign_out, &args_for_h1
).unwrap();
let sig1 = run_sign_protocol(rerand1, h1, ...);

// Second signing session: same presignature, different message h2
let rerand2 = RerandomizedPresignOutput::rerandomize_presign(
    &backup, &args_for_h2  // same nonce material, different h
).unwrap();
let sig2 = run_sign_protocol(rerand2, h2, ...);

// Both signatures share the same underlying nonce R.
// Standard nonce-reuse formula recovers the private key x from (sig1, sig2, h1, h2).
```

The root cause is `Clone` on `PresignOutput` (line 40, `src/ecdsa/ot_based_ecdsa/mod.rs`) and the `&PresignOutput` signature of `rerandomize_presign` (line 66, same file), which together provide no circuit-breaker against reuse — the direct structural analog of the missing `pause()` in Gravity.sol.
