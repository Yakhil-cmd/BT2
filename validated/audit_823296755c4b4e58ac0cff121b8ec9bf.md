### Title
Silent Discard of Critical Chunk Faults Removes Verification Incentive and Leaves Malicious Execution Nodes Unslashed - (File: `engine/verification/verifier/engine.go`)

---

### Summary

When a Verification Node detects a critical chunk fault proving an Execution Node submitted a fraudulent `ExecutionResult`, the `verify()` function in `engine/verification/verifier/engine.go` silently returns `nil` — emitting no `ResultApproval` and raising no challenge. This is the direct analog to the external report: just as liquidators receive no reward for closing overwhelmingly-indebted positions and therefore skip them, Verification Nodes receive no protocol reward (no `ResultApproval` to submit) for detecting fraud, and the malicious Execution Node faces zero slashing consequence.

---

### Finding Description

In `engine/verification/verifier/engine.go`, the `verify()` function handles the result of `ChunkVerifier.Verify()`. For the most security-critical fault types — those that prove an Execution Node committed fraud — the switch statement returns `nil` with only a log line and a `// TODO raise challenge` comment:

```go
case *chmodels.CFNonMatchingFinalState:

    log.Warn()...
    return nil          // ← no ResultApproval, no challenge
case *chmodels.CFInvalidVerifiableChunk:
    // TODO raise challenge
    log.Error()...

case *chmodels.CFInvalidEventsCollection:
    // TODO raise challenge
    log.Error()...
    return nil
case *chmodels.CFSystemChunkIncludedCollection:
    log.Error()...
    return nil
case *chmodels.CFExecutionDataBlockIDMismatch:
    log.Error()...
    return nil
// ... and four more fault types, all return nil
``` [1](#0-0) 

The `ResultApproval` generation and broadcast (lines 270–314) is only reached when `err == nil` (honest chunk) or when `CFMissingRegisterTouch` falls through the switch. All other fault types exit early. [2](#0-1) 

The fault types that trigger this silent discard are defined in `model/chunks/chunkFaults.go` and `model/chunks/executionDataFaults.go`. `CFNonMatchingFinalState` is the canonical proof of fraud: the Verification Node re-executes the chunk and the resulting state commitment does not match what the Execution Node committed to. [3](#0-2) 

The `ChunkVerifier.verifyTransactionsInContext()` produces `CFNonMatchingFinalState` at the point where the computed trie root diverges from the claimed end state: [4](#0-3) 

The test suite explicitly documents that no `ResultApproval` is published for any of these fault types: [5](#0-4) 

---

### Impact Explanation

**Incentive misalignment (analog to the external report):** A Verification Node that detects fraud does the same computational work as one that verifies an honest chunk, but receives no `ResultApproval` to submit. `ResultApproval`s are the on-chain signal of a VN's work and the basis for staking rewards. A VN that consistently detects fraud on a malicious EN's chunks earns nothing for those chunks, while a VN that verifies honest chunks earns approvals for every chunk. This structurally disincentivizes VNs from acting on detected fraud.

**No slashing of malicious Execution Nodes:** The challenge mechanism — the on-chain action that would slash a fraudulent EN's stake — is entirely unimplemented (`// TODO raise challenge`). A staked Execution Node can submit `ExecutionResult`s with an incorrect `FinalStateCommitment` indefinitely. VNs detect the fraud, log a warning, and return `nil`. The EN's stake is never at risk.

**Block sealing stall:** Because no `ResultApproval` is emitted for the fraudulent chunk, the Consensus Nodes cannot accumulate the required approvals to seal the block. This stalls finality for any block whose execution result is fraudulent, compounding the protocol-level harm.

---

### Likelihood Explanation

Any single staked Execution Node can trigger this path by submitting an `ExecutionResult` with a wrong `FinalStateCommitment` for any chunk. This is the exact threat model Flow's verification layer is designed to handle. The code path from receipt broadcast → chunk assignment → chunk data pack fetch → `ChunkVerifier.Verify()` → `CFNonMatchingFinalState` → silent `return nil` is fully reachable in production. The `// TODO raise challenge` comments confirm the gap is known and unresolved.

---

### Recommendation

1. **Implement the challenge mechanism**: When a `CFNonMatchingFinalState`, `CFInvalidVerifiableChunk`, `CFInvalidEventsCollection`, or any other fraud-proving fault is detected, the Verification Node must broadcast a signed challenge (slash evidence) to Consensus Nodes identifying the offending Execution Node and the specific `ExecutionResult`.

2. **Preserve the incentive**: Even before slashing is implemented, the VN should emit a `ResultApproval` (or a distinct "fraud report") for the faulty chunk so that its work is recorded on-chain and rewarded. The current behavior where fraud detection yields zero on-chain output removes all economic incentive to report fraud.

3. **Remove the silent `return nil`**: Replace each `return nil` in the fault switch with an explicit error or a call to the (to-be-implemented) challenge submission path, so the absence of a challenge is a compile-time or test-time failure rather than a silent no-op.

---

### Proof of Concept

1. A staked Execution Node executes block `B` and produces `ExecutionResult R` but deliberately sets `Chunk[i].EndState` to an incorrect value `X` (true computed value is `Y`).
2. `R` is broadcast; Verification Nodes assigned to chunk `i` fetch the `ChunkDataPack`.
3. `ChunkVerifier.verifyTransactionsInContext()` re-executes the chunk, computes end state `Y`, compares to `X`, and returns `CFNonMatchingFinalState{expected: X, computed: Y}`. [6](#0-5) 
4. `verify()` matches `*chmodels.CFNonMatchingFinalState`, logs a warning, and executes `return nil`. [7](#0-6) 
5. `verifiableChunkHandler()` receives `nil` from `verify()` and returns without error. [8](#0-7) 
6. No `ResultApproval` is stored or broadcast. No challenge is raised. The malicious EN's stake is untouched. The block cannot be sealed. The VN receives no reward for detecting the fraud.

// CFNonMatchingFinalState is returned when the computed final state commitment
// (applying chunk register updates to the partial trie) doesn't match the one provided by the chunk
type CFNonMatchingFinalState struct {
	expected   flow.StateCommitment
	computed   flow.StateCommitment
	chunkIndex uint64
	execResID  flow.Identifier
}

var _ ChunkFaultError = (*CFNonMatchingFinalState)(nil)

func (cf CFNonMatchingFinalState) String() string {
	return fmt.Sprintf("final state commitment doesn't match, expected [%x] but computed [%x]", cf.expected, cf.computed)
}

func (cf CFNonMatchingFinalState) Error() string {
	return cf.String()
}

// ChunkIndex returns chunk index of the faulty chunk
func (cf CFNonMatchingFinalState) ChunkIndex() uint64 {
	return cf.chunkIndex
}

// ExecutionResultID returns the execution result identifier including the faulty chunk
func (cf CFNonMatchingFinalState) ExecutionResultID() flow.Identifier {
	return cf.execResID
}

// NewCFNonMatchingFinalState creates a new instance of Chunk Fault (NonMatchingFinalState)
func NewCFNonMatchingFinalState(expected flow.StateCommitment, computed flow.StateCommitment, chInx uint64, execResID flow.Identifier) *CFNonMatchingFinalState {
	return &CFNonMatchingFinalState{expected: expected,
		computed:   computed,
		chunkIndex: chInx,
		execResID:  execResID}
}
