### Title
Missing CAT Asset Identity in `chia_spendCAT` WalletConnect Confirmation Dialog - (`packages/gui/src/electron/commands/Commands.ts`)

### Summary

The `chia_spendCAT` WalletConnect confirmation dialog displays only the raw numeric `wallet_id` without resolving or displaying the associated CAT asset name or TAIL hash (assetId). A dApp that has been granted `chia_spendCAT` permission can target any CAT wallet by specifying an arbitrary `wallet_id`, and the user cannot identify which specific CAT token is being spent before approving.

### Finding Description

The `chia_wallet.cat_spend` command schema in `Commands.ts` defines the confirmation dialog params as: [1](#0-0) 

The `params` array contains `wallet_id` as a plain `number` type with no name resolution, `address`, `amount` (humanized as mojo-to-cat, showing a unit-converted number but no token identity), `fee`, and optional `memos`. There is no `assetId` field and no mechanism to resolve `wallet_id` to a CAT name or TAIL hash in the dialog.

The `humanizeDappCommand` function processes exactly these params: [2](#0-1) 

The resulting `rows` are rendered in the `Confirm` dialog: [3](#0-2) 

The `parseCommandDisplay` path (which does resolve wallet names for offer commands) has no test coverage or handling for `chia_wallet.cat_spend`, so `display?.walletDelta` is `null`/`undefined` for this command and the `WalletDeltaSection` is not rendered: [4](#0-3) 

The full confirmation flow in `main.tsx` calls both `humanizeDappCommand` and `parseCommandDisplay` but neither produces CAT identity information for `cat_spend`: [5](#0-4) 

### Impact Explanation

A dApp granted `chia_spendCAT` permission is not restricted to a specific `wallet_id` — it can pass any numeric wallet ID. The user sees:

```
Wallet Id:  5
Address:    xch1victim...
Amount:     999999.999
Fee:        0 XCH
```

There is no CAT name (e.g., "Stably USD") or TAIL hash displayed. The user cannot distinguish between their SBX, DBX, USDS, or any other CAT wallet. A malicious dApp can target the user's highest-value CAT wallet by ID, and the user has no way to detect the substitution before signing.

This fits the **High** impact category: *WalletConnect state that causes a user to approve the wrong asset*.

### Likelihood Explanation

Any dApp that has been granted `chia_spendCAT` permission can exploit this. The permission is coarse-grained (not scoped to a specific wallet_id), so the attack surface is the full set of CAT wallets the user holds. The user must actively approve the dialog, but the dialog provides insufficient information to make an informed decision.

### Recommendation

Resolve `wallet_id` to the CAT wallet's name and `assetId` (TAIL hash) at dialog-render time and display both fields. The `cat_get_asset_id` RPC (`CATWallet.getAssetId`) and `cat_get_name` RPC (`CATWallet.getName`) are already available: [6](#0-5) 

Either add a `resolve` step to the `chia_wallet.cat_spend` param schema (similar to how offer commands use `parseCommandDisplay` to resolve wallet names), or add a dedicated `parseCommandDisplay` handler for `cat_spend` that fetches and injects the CAT name and assetId as additional display rows.

### Proof of Concept

1. Connect a dApp via WalletConnect and grant it `chia_spendCAT` permission.
2. User has two CAT wallets: wallet_id=2 (low-value token) and wallet_id=5 (high-value USDS).
3. dApp sends: `chia_spendCAT({ wallet_id: 5, address: 'xch1attacker...', amount: '999999000', fee: '0' })`.
4. Confirmation dialog renders: `Wallet Id: 5 | Address: xch1attacker... | Amount: 999999 | Fee: 0 XCH`.
5. User sees no CAT name or TAIL hash — cannot distinguish wallet_id 5 from wallet_id 2.
6. User approves, believing it may be the low-value token; USDS is transferred to the attacker.

export async function humanizeDappCommand(dappCommand: string, data: Record<string, unknown>, networkPrefix?: string) {
  const dappCommandSchema = getDappCommandSchema(dappCommand);

  const rows = await humanizeParams(dappCommandSchema.params, data, networkPrefix);

  return {
    destructive: dappCommandSchema.destructive === true,
    title: dappCommandSchema.title(),
    message: dappCommandSchema.message(),
    confirmLabel: dappCommandSchema.confirmLabel(),
    rows,
  };
