### Title
Malicious Participant Can Corrupt CKD Output via Unverified Contribution — (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

In the Confidential Key Derivation (CKD) protocol, the coordinator aggregates participant contributions without any cryptographic verification that each contribution is correctly formed from the participant's actual signing share. A single malicious participant can send arbitrary group elements, corrupting the final `CKDOutput` and causing every honest caller of `unmask()` to derive an incorrect confidential key.

---

### Finding Description

In `do_ckd_coordinator` (`src/confidential_key_derivation/protocol.rs`, lines 35–58), the coordinator receives a `CKDOutput` — a pair `(big_y, big_c)` — from every other participant and unconditionally adds them together:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
``` [1](#0-0) 

The intended invariant is that each participant computes:

```
big_s  = x_i · H(pk ‖ app_id)
big_c  = big_s + y_i · app_pk
big_y  = y_i · G
```

and sends `(λ_i · big_y, λ_i · big_c)` to the coordinator. [2](#0-1) 

There is **no zero-knowledge proof, commitment, or any other check** that the received `(big_y, big_c)` pair was actually derived from the participant's legitimate signing share `x_i`. The coordinator's only "validation" is that the message was received from a known participant — analogous to the original report's `msg.sender == atlas` check that was necessary but not sufficient.

This is the direct analog of the reported vulnerability class: a guard that checks the *source* of a message (authenticated channel / known participant) but does not verify the *content* against a trusted reference (the participant's committed public verification share). Just as the `validControl` modifier read from attacker-appended calldata rather than from trusted storage, the CKD coordinator reads from attacker-supplied group elements rather than verifying them against the committed public key shares established during DKG.

The `compute_signature_share` function that honest participants run is never enforced on the receiver side: [3](#0-2) 

The `KeygenOutput` held by each participant contains a `public_key` (the aggregate verifying key) but individual verification shares (`vk_share_i = x_i · G2`) are not used anywhere in the CKD flow to cross-check the received `big_c` values. [4](#0-3) 

---

### Impact Explanation

**High — Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.**

A malicious participant sends `(big_y', big_c') = (big_y + Δ_Y, big_c + Δ_C)` for arbitrary group elements `Δ_Y, Δ_C ∈ G1`. The coordinator computes:

```
final_big_c = msk·H(pk‖app_id) + Σ(λ_i·y_i)·app_pk + Δ_C
final_big_y = Σ(λ_i·y_i)·G + Δ_Y
```

The client calls `unmask(app_sk)`:

```rust
pub fn unmask(&self, secret_scalar: Scalar) -> Signature {
    self.big_c - self.big_y * secret_scalar
}
``` [5](#0-4) 

The result is `msk·H(pk‖app_id) + Δ_C − app_sk·Δ_Y`, which is an incorrect confidential key unless `Δ_C = app_sk·Δ_Y`. The client has no way to detect this corruption because the expected output `msk·H(pk‖app_id)` is never published. Every downstream use of the derived key (e.g., decryption, authentication) silently fails or produces attacker-influenced output.

---

### Likelihood Explanation

**High.** Any single participant in the CKD session is a sufficient attacker. The entry path is direct: the malicious participant simply deviates from `compute_signature_share` and sends crafted `(big_y, big_c)` values via the private channel to the coordinator. No special privilege, leaked key, or external assumption is required — only participation in the protocol, which is the normal operating condition. [6](#0-5) 

---

### Recommendation

Each participant must accompany their `(big_y, big_c)` contribution with a zero-knowledge proof of correct formation. Concretely, the participant must prove:

1. **Correct blinding**: knowledge of `y_i` such that `big_y_i = y_i · G1` (a standard Schnorr/dlog proof).
2. **Correct share application**: knowledge of `x_i` such that `big_c_i − y_i · app_pk = x_i · H(pk ‖ app_id)` **and** `x_i · G2 = vk_share_i` (a dlog-equality proof, i.e., `dlogeq` as already implemented in `src/crypto/proofs/dlogeq.rs`). [7](#0-6) 

The coordinator must verify both proofs against the participant's public verification share (derivable from the `KeygenOutput.public_key` commitment) before accepting the contribution.

// Malicious participant implementation replacing compute_signature_share:
fn malicious_compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Instead of computing the honest share, send arbitrary group elements.
    // delta_c and delta_y are attacker-chosen; the coordinator has no way to detect this.
    let delta_y = ElementG1::generator() * blstrs::Scalar::random(rng);
    let delta_c = ElementG1::generator() * blstrs::Scalar::random(rng);

    // Lagrange weight is still applied so the coordinator doesn't abort on size checks.
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    Ok((delta_y * lambda_i, delta_c * lambda_i))
}

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}

/// Produce a proof for the given statement and witness, using a caller-provided nonce.
/// The nonce `k` must be sampled from a cryptographically secure RNG by the caller.
/// The challenge is derived via the Fiat-Shamir transform over the transcript.
pub fn prove_with_nonce<C: Ciphersuite>(
    transcript: &mut Transcript,
    statement: Statement<'_, C>,
    witness: Witness<C>,
    k: Scalar<C>,
) -> Result<Proof<C>, ProtocolError>
where
    Element<C>: ConstantTimeEq,
{
    if statement.generator1.ct_eq(&C::Group::identity()).into() {
        return Err(ProtocolError::IdentityElement);
    }

    transcript.message(NEAR_DLOGEQ_STATEMENT_LABEL, &statement.encode()?);

    let (big_k_0, big_k_1) = statement.phi(&k);

    // This will never raise error as k is not zero and generator1 is not the identity
    let enc = encode_two_points::<C>(&big_k_0, &big_k_1)?;

    transcript.message(NEAR_DLOGEQ_COMMITMENT_LABEL, &enc);
    let mut rng = transcript.challenge_then_build_rng(NEAR_DLOGEQ_CHALLENGE_LABEL);
    let e = frost_core::random_nonzero::<C, _>(&mut rng);

    let s = k + e * witness.x.0;
    Ok(Proof {
        e: SerializableScalar::<C>(e),
        s: SerializableScalar::<C>(s),
    })
}
