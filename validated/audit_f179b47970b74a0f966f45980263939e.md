### Title
`CreateAccount`-then-`Commit` silently deletes a recreated account, destroying its balance — (`fvm/evm/emulator/state/stateDB.go`, `delta.go`)

### Summary
When a contract is deployed to an address that already holds ETH balance, `DeltaView.CreateAccount` internally calls `SelfDestruct`, placing the address in both `toBeDestructed` and `created`. `StateDB.Commit` checks `HasSelfDestructed` first, deletes the account, sets `deleted = true`, and `continue`s — permanently skipping the `IsCreated` recreation branch. The account is erased from the base view and the carried-over balance is destroyed.

### Finding Description
`DeltaView.CreateAccount` (delta.go:119–161) handles an already-existing address by calling `SelfDestruct` to capture the pre-existing balance, then sets `recreated[addr]` and `created[addr]`. This leaves the address simultaneously in `toBeDestructed` and `created`.

`StateDB.Commit` (stateDB.go:472–488) iterates dirty addresses and checks `HasSelfDestructed` unconditionally first. If true, it calls `baseView.DeleteAccount`, sets `deleted = true`, and `continue`s — bypassing the `IsCreated` branch entirely. The `recreated` map is never consulted during commit; it is only used in `DeltaView.GetState` (delta.go:383) to suppress stale slot reads.

The net result: the account is deleted from persistent storage, the carried-over balance is destroyed, and the new contract code is never written.

### Impact Explanation
- **Permanent loss of funds**: any ETH balance at the target address is destroyed, not transferred anywhere.
- **Silent contract deployment failure**: the deploying transaction succeeds (no error), but the contract does not exist afterward.
- **EVM invariant violation**: the EVM specification requires that deploying to an address with existing balance carries that balance into the new contract. This invariant is broken.

This is a Critical execution-layer vulnerability: it causes unauthorized permanent mutation (deletion) of an account's state and irreversible loss of on-chain assets, reachable through a standard EVM transaction.

### Likelihood Explanation
The path is reachable via any standard EVM `CREATE2` transaction where the deployer pre-funds the target address before deployment — a common and documented pattern (e.g., counterfactual contract wallets, deterministic factory deployments). No privileged access, leaked keys, or staked-node control is required. The attacker only needs to submit two EVM transactions through the normal Access API.

### Recommendation
In `StateDB.Commit`, after calling `DeleteAccount` for a self-destructed address, check whether the address is also `IsCreated` (i.e., was recreated in the same transaction). If so, proceed to the `CreateAccount` branch rather than `continue`-ing. Concretely:

```go
if db.HasSelfDestructed(addr) {
    err = db.baseView.DeleteAccount(addr)
    ...
    deleted = true
}
if deleted && !db.IsCreated(addr) {
    continue
}
```

Alternatively, `DeltaView.CreateAccount` could remove the address from `toBeDestructed` when it sets `created`, so `HasSelfDestructed` returns false for recreated accounts at commit time.

func (d *DeltaView) CreateAccount(addr gethCommon.Address) error {
	// if is already created return
	if d.IsCreated(addr) {
		return nil
	}
	exist, err := d.Exist(addr)
	if err != nil {
		return err
	}
	if exist {
		// check if already destructed
		destructed, balance := d.HasSelfDestructed(addr)
		if !destructed {
			balance, err = d.GetBalance(addr)
			if err != nil {
				return err
			}
			err = d.SelfDestruct(addr)
			if err != nil {
				return err
			}
		}

		d.nonces[addr] = 0
		d.codes[addr] = nil
		d.codeHashes[addr] = gethTypes.EmptyCodeHash
		// carrying over the balance. (legacy behavior of the Geth stateDB)
		d.balances[addr] = balance

		}

	return nil
