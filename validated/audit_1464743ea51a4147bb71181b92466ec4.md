Audit Report

## Title
Boundary Node Health Checker Marks Nodes Unhealthy on Single Transient 503, Enabling Subnet Availability DoS via HTTP/2 Concurrency Exhaustion — (`rs/boundary_node/ic_boundary/src/check.rs`, `rs/http_endpoints/public/src/lib.rs`)

## Summary

The replica's `/api/v2/status` endpoint is protected by a `GlobalConcurrencyLimitLayer` (default: 100 slots) preceded by `.load_shed()`, which immediately returns HTTP 503 when the limit is saturated. The boundary node's `Checker::check` treats any non-200 response — including a transient 503 — as an immediate, unretried node health failure. Because `NodeActor` has no consecutive-failure threshold and `SubnetActor` immediately removes nodes from routing on the first health state change, an attacker who saturates the concurrency limit on every node in a subnet can cause the boundary node to route zero traffic to that subnet, making it effectively unreachable through the boundary node despite all replicas being fully operational.

## Finding Description

**Concurrency limit and load-shed on the status endpoint:**

`rs/http_endpoints/public/src/lib.rs` lines 591–602 wrap the status router with `.load_shed()` followed by `GlobalConcurrencyLimitLayer::new(config.max_status_concurrent_requests)`. [1](#0-0) 

The default value is 100 concurrent requests. [2](#0-1) 

The replica's HTTP/2 configuration allows up to 1000 concurrent streams per connection and holds connections open for up to 1200 seconds (`connection_read_timeout_seconds`). [3](#0-2) 

**Attacker technique — HTTP/2 slow-read stream exhaustion:**

An attacker opens one HTTP/2 connection to a replica and sends 100 GET `/api/v2/status` requests with `WINDOW_UPDATE=0`. The server cannot flush the response body and holds each stream in the concurrency counter for up to 1200 seconds. The 101st request — from the boundary node health checker — is immediately rejected with HTTP 503 by the load-shed layer.

**Boundary node checker — no retry, no 503 tolerance:**

`Checker::check` in `rs/boundary_node/ic_boundary/src/check.rs` makes a single GET to `node.health_check_url` and returns `Err(CheckError::Http(503))` for any non-200 status with no retry and no distinction between transient and permanent failures. [4](#0-3) 

`NodeActor::check` maps any `Err(_)` to `healthy = false` unconditionally. [5](#0-4) 

The `NodeActor` struct has no consecutive-failure counter field. [6](#0-5) 

**Immediate propagation to routing:**

`SubnetActor::run` triggers an immediate call to `update()` whenever a node's `healthy` field changes. [7](#0-6) 

`SubnetActor::update` filters out all nodes where `state.healthy == false` and immediately publishes the reduced node set downstream. [8](#0-7) 

## Impact Explanation

By repeating the saturation attack for every node in a subnet, the attacker causes the boundary node to publish an empty healthy-node set for that subnet, dropping all client traffic routed through the boundary node to that subnet. The subnet itself is unaffected — replicas continue to make consensus progress — but all users relying on the boundary node for access are denied service. This is a concrete, non-volumetric subnet availability impact through the boundary node, matching the High ($2,000–$10,000) impact class: "Application/platform-level DoS … or subnet availability impact not based on raw volumetric DDoS."

## Likelihood Explanation

The replica's public HTTP endpoint is accessible by design to any internet user. The attack requires no credentials, no privileged network position, and no protocol compromise. A single machine with one HTTP/2 connection per target replica is sufficient to saturate the 100-slot limit, since HTTP/2 multiplexes up to 1000 streams per connection. The 1200-second connection timeout means the attacker can hold slots open with minimal bandwidth. The absence of any retry or transient-error tolerance means the attack succeeds on the very first health check cycle that coincides with saturation. Recovery requires the attacker to release connections and wait for the next successful health check cycle.

## Recommendation

1. **In `Checker::check`** (`rs/boundary_node/ic_boundary/src/check.rs`): Distinguish transient HTTP status codes (503, 429, 502, 504) from genuine health failures. Return a new `CheckError::Transient` variant for these codes rather than `CheckError::Http`, and do not count transient errors as health failures in `NodeActor`.
2. **In `NodeActor`** (`rs/boundary_node/ic_boundary/src/check.rs`): Add a consecutive-failure counter. Only set `healthy = false` after N consecutive non-transient failures (e.g., 3), to tolerate transient load spikes.
3. **On the replica** (`rs/http_endpoints/public/src/lib.rs`): Consider a dedicated, separate concurrency pool for `/api/v2/status` that is not shared with public traffic, or exempt health-check requests from known boundary node IPs from the `GlobalConcurrencyLimitLayer`.

## Proof of Concept

```
# For each replica node in the target subnet:
# 1. Open one HTTP/2 connection to https://<node_ip>:8080
# 2. Send 100 GET /api/v2/status requests with WINDOW_UPDATE=0
#    (prevents server from flushing response; holds slot in concurrency counter
#     for up to connection_read_timeout_seconds = 1200s)
# 3. The 101st request (from the BN checker) receives HTTP 503 immediately
#    from the load_shed() layer
# 4. Checker::check returns Err(CheckError::Http(503))
# 5. NodeActor sets healthy=false on the first failure
# 6. SubnetActor immediately removes node from routing
# 7. Repeat for all N nodes in subnet
# 8. Boundary node routes 0 requests to subnet; subnet is unreachable
#    through the boundary node despite all replicas being fully operational
#
# Verifiable locally: unit test mocking http_client.execute() to return
# StatusCode::SERVICE_UNAVAILABLE and asserting NodeState.healthy == false
# after a single Checker::check call, with no retry observed.
```
