### Title
Signature Verification Bypassed for Invoke Transactions via `skip_stateful_validations` Race — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator skips the `__validate__` entry-point call (the only place where an account's signature is cryptographically verified) for any invoke transaction with `nonce == 1` when `account_tx_in_pool_or_recent_block(sender_address)` returns `true`. Because that mempool query returns `true` for **any** account that has **any** pending transaction — including a victim's own `deploy_account` — an unprivileged attacker can submit an invoke transaction for a victim's address with an arbitrary (invalid) signature and have it admitted to the mempool without signature verification.

### Finding Description

The gateway stateful validation path is:

```
extract_state_nonce_and_run_validations
  └─ run_pre_validation_checks
       ├─ validate_state_preconditions   (nonce range, gas price)
       ├─ validate_by_mempool            (duplicate/nonce-gap check only)
       └─ skip_stateful_validations      ← returns true → __validate__ skipped
  └─ run_validate_entry_point(skip_validate=true)  ← no __validate__ call
```

The critical function:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs  lines 429-461
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())  // ← only check
                .await ...;
        }
    }
    Ok(false)
}

**Attack scenario:**

1. Victim submits `deploy_account` for address `A` (nonce 0). It enters the mempool.
2. Attacker observes `A` in the mempool via `account_tx_in_pool_or_recent_block`.
3. Attacker submits `Invoke { sender_address: A, nonce: 1, signature: [garbage], calldata: [malicious] }`.
4. Gateway checks: `account_nonce = 0`, `tx_nonce = 1`, `account_tx_in_pool_or_recent_block(A) = true` → `skip_validate = true`.
5. `__validate__` is never called. The transaction is admitted to the mempool with an invalid signature.
6. Attacker can use fee escalation to replace the victim's legitimate invoke (same address, same nonce=1, higher tip).
7. Victim's `deploy_account` succeeds; attacker's invalid invoke executes and fails (signature check in the OS), consuming the victim's gas budget and evicting their legitimate invoke.

The contrast with the Python-side `PyValidator::should_run_stateful_validations` is instructive — it requires the caller to supply the actual `deploy_account_tx_hash`, binding the skip to the specific deploy transaction:

```rust
// crates/native_blockifier/src/py_validator.rs  lines 109-110
let deploy_account_not_processed =
    deploy_account_tx_hash.is_some() && nonce == Nonce(Felt::ZERO);
``` [5](#0-4) 

The Rust gateway has no equivalent binding — it only checks address presence.

### Impact Explanation

The gateway admits invoke transactions with invalid (or empty) signatures to the mempool whenever the target account has any pending transaction. This satisfies the **High** impact criterion: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."* Concretely, an attacker can evict a victim's legitimate post-deploy invoke from the mempool via fee escalation, replacing it with an invalid one that will fail on-chain, forcing the victim to resubmit and pay additional fees.

### Likelihood Explanation

The `deploy_account + invoke` UX pattern is explicitly documented and tested as a supported flow. Any user deploying a new account and sending a simultaneous invoke is exposed. The attack requires only observing the mempool (public information) and submitting two transactions, making it trivially automatable.

### Recommendation

Mirror the Python gateway's approach: require the caller to supply the `deploy_account_tx_hash` and verify it matches a transaction actually present in the mempool for the same address before skipping `__validate__`. Alternatively, restrict the skip to cases where the invoke transaction's `tx_hash` was submitted in the same gateway request as the `deploy_account` (e.g., by passing a paired hash through the gateway flow), so that the skip cannot be triggered by a third party observing the mempool.

### Proof of Concept

```
1. Victim calls gateway.add_tx(deploy_account { sender: A, nonce: 0, sig: valid })
   → A is now in mempool pool

2. Attacker calls gateway.add_tx(invoke { sender: A, nonce: 1, sig: [0x1], calldata: [drain_funds] })
   → stateless: sig length 1 ≤ max_signature_length → OK
   → stateful: account_nonce=0, tx_nonce=1, account_tx_in_pool_or_recent_block(A)=true
   → skip_stateful_validations returns true
   → run_validate_entry_point sets validate=false, returns Ok(())
   → transaction admitted to mempool WITHOUT signature verification

3. Attacker's invoke (higher tip) replaces victim's invoke via fee escalation.

4. Block is built: deploy_account(A) executes → A deployed.
   Attacker's invoke executes → __validate__ called → INVALID SIGNATURE → revert.
   Victim's legitimate invoke is gone; victim must resubmit.
``` [6](#0-5) [7](#0-6)

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
