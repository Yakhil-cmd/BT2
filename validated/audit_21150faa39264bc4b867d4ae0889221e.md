### Title
FIFO-mode `remove_rejected_txs` calls `remove_by_address` which evicts all sibling transactions for a rejected sender, permanently stranding valid higher-nonce transactions — (`crates/apollo_mempool/src/mempool.rs`)

### Summary
In Echonet/FIFO mode, when `commit_block` processes a rejected transaction, `remove_rejected_txs` calls `tx_queue.remove_by_address(address)` after removing only the rejected tx from `tx_pool`. Because `FifoTransactionQueue::remove_by_address` removes **all** entries for that address from the queue (not just the rejected one), any sibling transactions at higher nonces are evicted from the queue while remaining in the pool. No re-queuing path exists for FIFO mode, so those transactions are permanently stranded and will never be sequenced.

### Finding Description
`remove_rejected_txs` is designed for fee-priority mode, where at most one transaction per address is ever in the queue at a time. In that mode, `remove_by_address` is safe because it can only remove the one queued tx for that address. In FIFO mode, however, **all** transactions for an address are inserted into the queue immediately on arrival: [8](#0-7) 

So when a single rejected tx triggers `remove_by_address`, the entire address's queue presence is wiped: [5](#0-4) 

The `tx_pool.remove` call on line 566 removes only the rejected tx hash from the pool: [9](#0-8) 

The asymmetry — pool loses one tx, queue loses all — leaves the remaining pool entries permanently orphaned.

### Impact Explanation
Valid transactions at nonces 1, 2, … for address A are silently dropped from sequencing consideration after any single transaction for that address is rejected. They expire via TTL without ever being included in a block. This is a direct admission correctness failure: the mempool accepted these transactions, but they are never sequenced.

### Likelihood Explanation
Any block that rejects a transaction in FIFO/Echonet mode triggers this. A single attacker-submitted transaction at nonce=0 that fails execution (e.g., insufficient fee, bad signature accepted by gateway but rejected by blockifier) is sufficient to strand all subsequent transactions for that address. No privileged access is required.

### Recommendation
In `remove_rejected_txs`, replace the `remove_by_address` call with a targeted `remove_txs` call using only the specific rejected `TransactionReference`. The `tx_pool.remove` already returns the full transaction, so the reference is available. Alternatively, add a FIFO-mode branch that uses `remove_txs(&[tx_reference])` instead of `remove_by_address`.

### Proof of Concept
Build a Rust unit test in FIFO/Echonet mode:
1. Add three transactions for address `0x1` at nonces 0, 1, 2 (each with valid metadata).
2. Call `get_txs(1)` — nonce=0 is staged, nonces 1 and 2 remain in the queue.
3. Call `commit_block` with `address_to_nonce = {}` and `rejected_tx_hashes = {tx_hash_nonce_0}`.
4. Call `get_txs(10)` — assert the result is empty (nonces 1 and 2 are stranded).
5. Inspect `tx_pool` directly — assert nonces 1 and 2 are still present in the pool.

The test will confirm that after step 3, `tx_queue.queue` is empty for address `0x1` while `tx_pool` still holds the two valid transactions.
