### Title
Gateway `skip_stateful_validations` Bypasses `__validate__` Signature Check When Any Account Transaction Exists in Mempool — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function is intended to skip the `__validate__` entry point (signature verification) for an invoke transaction with nonce=1 only when a `deploy_account` is pending in the mempool — a UX feature allowing users to submit `deploy_account + invoke` simultaneously. However, the predicate `account_tx_in_pool_or_recent_block` returns `true` for **any** transaction from the account, not exclusively `deploy_account` transactions. An attacker who observes any nonce=0 transaction from a target account in the mempool can submit a nonce=1 invoke with an **invalid signature**, bypass the gateway's `__validate__` call, and have the unauthorized transaction admitted to the mempool — blocking the account owner's legitimate nonce=1 invoke.

---

### Finding Description

In `skip_stateful_validations` the gateway decides to skip blockifier validation when three conditions hold simultaneously:

1. The incoming transaction is an `Invoke` with `nonce == 1`
2. The on-chain account nonce is `0`
3. `account_tx_in_pool_or_recent_block(sender_address)` returns `true` [1](#0-0) 

The code comment reads: *"It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."*

This assumption is incorrect. `account_tx_in_pool_or_recent_block` is implemented as: [2](#0-1) 

It returns `true` whenever the account has **any** transaction in the pool or recent block — including a nonce=0 `Invoke`. There is no type-level check for `deploy_account`.

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` is called with `execution_flags.validate = false`: [3](#0-2) 

This means the account's `__validate__` entry point — the sole on-chain signature-verification step — is **never called** at the gateway for the nonce=1 invoke. The transaction is forwarded to the mempool without any signature check.

The attack path:

1. Account owner submits a valid nonce=0 `Invoke` (or `deploy_account`) for a fresh account (on-chain nonce = 0). It enters the mempool.
2. Attacker observes the account address in the mempool via `account_tx_in_pool_or_recent_block`.
3. Attacker submits a nonce=1 `Invoke` with an **invalid signature** (or arbitrary malicious calldata) targeting the same account.
4. Gateway stateful validation:
   - `validate_nonce`: passes — `0 ≤ 1 ≤ 0 + max_allowed_nonce_gap`.
   - `validate_by_mempool` (`validate_tx`): passes — no duplicate nonce=1 exists yet.
   - `skip_stateful_validations`: returns `true` — account has a tx in the pool.
   - `run_validate_entry_point` with `validate: false` — `__validate__` is **skipped**.
5. Attacker's nonce=1 invoke enters the mempool without signature verification.
6. Account owner's legitimate nonce=1 invoke is subsequently rejected with `MempoolError::DuplicateNonce`. [4](#0-3) 

When the batcher later executes the attacker's transaction, `__validate__` is called with `validate: true` and fails, so the transaction is not included in the block. However, the account owner's legitimate nonce=1 invoke has already been permanently rejected from the mempool.

---

### Impact Explanation

**Critical — Invalid/unauthorized transaction accepted through account validation logic.** The gateway admits a transaction whose signature has never been verified. The invariant *"every transaction entering the mempool must have passed the account's `__validate__` entry point"* is broken. An attacker with no knowledge of the account's private key can inject a transaction on behalf of any account that has a nonce=0 transaction pending.

**High — Valid transaction rejected before sequencing.** The account owner's legitimate nonce=1 invoke is rejected with `DuplicateNonce` because the attacker's invalid transaction occupies the slot first. The account owner cannot sequence their intended transaction until the attacker's transaction is purged by the batcher's `commit_block` cycle.

---

### Likelihood Explanation

The mempool is publicly observable. Any node can call `account_tx_in_pool_or_recent_block` or watch the P2P gossip layer to detect accounts with pending nonce=0 transactions. The attack requires only:

- Identifying a target account with on-chain nonce=0 and a pending nonce=0 transaction.
- Submitting a nonce=1 invoke before the account owner does.

This is a straightforward front-running attack with no privileged access required. The deploy_account + invoke UX pattern (the intended trigger for this feature) is a well-known and commonly used flow, making targets easy to identify.

---

### Recommendation

Replace the coarse `account_tx_in_pool_or_recent_block` check with a type-specific predicate. The mempool should expose a method such as `has_pending_deploy_account(address: ContractAddress) -> bool` that returns `true` only when the account has a `deploy_account` transaction in the pool or a recent committed block. The `skip_stateful_validations` function should call this narrower predicate:

```rust
// Proposed fix
async fn skip_stateful_validations(...) -> ... {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            return mempool_client
                .account_has_pending_deploy_account(tx.sender_address())  // type-specific check
                .await
                ...
        }
    }
    Ok(false)
}
```

This mirrors the fix recommended in the external report: enforce that the source and destination are distinct (here: enforce that the pending transaction is specifically a `deploy_account`, not just any transaction).

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
