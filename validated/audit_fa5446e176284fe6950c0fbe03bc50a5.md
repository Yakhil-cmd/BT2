### Title
Scheduled Transaction `ExecutionEffort` Silently Capped Without Returning Excess Fees to Scheduler - (`File: fvm/blueprints/scheduled_callback.go`)

### Summary

When a scheduled transaction's declared `ExecutionEffort` exceeds `flow.DefaultMaxTransactionGasLimit`, the FVM silently caps the gas limit used for execution but the scheduler already paid `Fees` based on the full declared effort. The excess fees are never returned to the scheduler, causing a direct financial loss analogous to M-8's `lossFee` being added to a pool without reimbursement.

### Finding Description

When the `FlowTransactionScheduler` contract emits a `PendingExecution` event, it carries both `ExecutionEffort` and `Fees` fields. The Go-side `callbackArgsFromEvent` function in `fvm/blueprints/scheduled_callback.go` extracts the effort and silently caps it:

```go
effort := uint64(cadenceEffort)
if effort > flow.DefaultMaxTransactionGasLimit {
    log.Warn().Uint64("effort", effort).Msg("effort is greater than max transaction gas limit, setting to max")
    effort = flow.DefaultMaxTransactionGasLimit
}
``` [1](#0-0) 

The capped `effort` is then used as the `ComputeLimit` for the execute-callback transaction:

```go
tx, err := flow.NewTransactionBodyBuilder().
    AddAuthorizer(sc.ScheduledTransactionExecutor.Address).
    SetScript(script).
    AddArgument(encID).
    SetComputeLimit(effort).   // ← capped value
    Build()
``` [2](#0-1) 

Critically, `callbackArgsFromEvent` **ignores the `Fees` field** entirely — it only returns `id` and `effort`. The `Fees` value carried in the `PendingExecution` event (representing what the scheduler already paid) is never consulted, adjusted, or returned when the effort is capped. [3](#0-2) 

The `PendingExecution` event structure confirms that `Fees` is a distinct field from `ExecutionEffort`: [4](#0-3) 

The `ScheduledTransaction` model stores both `ExecutionEffort` and `Fees` as independent fields set at scheduling time: [5](#0-4) 

For comparison, the **cancellation** path correctly tracks `FeesReturned` and `FeesDeducted`: [6](#0-5) 

But the **failure** path records no fee information at all: [7](#0-6) 

There is no analogous fee-return mechanism anywhere in the execution path for the effort-capping case.

### Impact Explanation

A scheduler who declares `ExecutionEffort > flow.DefaultMaxTransactionGasLimit` pays fees proportional to the full declared effort, but the FVM only grants `DefaultMaxTransactionGasLimit` gas. The difference in fees is silently retained by the protocol (FlowFees contract) with no refund path. This is a direct, quantifiable loss of FLOW tokens for the scheduler — the same class of loss as M-8's `lossFee` being absorbed into `commonData` without keeper reimbursement.

### Likelihood Explanation

The `log.Warn()` in `callbackArgsFromEvent` confirms the developers anticipated this case occurring in production. Any unprivileged account that can call the `FlowTransactionScheduler` contract to schedule a transaction can trigger this path by declaring a sufficiently large `ExecutionEffort`. If the Cadence contract does not enforce `ExecutionEffort <= DefaultMaxTransactionGasLimit` at scheduling time, this is directly reachable by any user.

### Recommendation

1. **Preferred:** Validate `ExecutionEffort <= flow.DefaultMaxTransactionGasLimit` inside the `FlowTransactionScheduler` Cadence contract at scheduling time, so over-declared effort is rejected before fees are collected.
2. **Alternative:** When `callbackArgsFromEvent` caps the effort, compute the proportional excess fee from the `Fees` field in the `PendingExecution` event and return it to the `TransactionHandlerOwner` before executing the callback transaction.

### Proof of Concept

1. User calls `FlowTransactionScheduler.schedule(executionEffort: 2 * DefaultMaxTransactionGasLimit, ...)` and pays `Fees` proportional to `2 * DefaultMaxTransactionGasLimit`.
2. At the scheduled block, the contract emits `PendingExecution(id: X, executionEffort: 2 * DefaultMaxTransactionGasLimit, fees: <full_amount>, ...)`.
3. `callbackArgsFromEvent` caps effort to `DefaultMaxTransactionGasLimit` and discards the `fees` field entirely.
4. `generateExecuteCallbacksTransaction` builds a transaction with `SetComputeLimit(DefaultMaxTransactionGasLimit)`.
5. The scheduler's transaction runs with half the gas it paid for; the excess fees are never returned. [3](#0-2) [8](#0-7)

func generateExecuteCallbacksTransaction(

		SetComputeLimit(effort).
		Build()

		return nil, fmt.Errorf("failed to construct execute callback transactions: %w", err)
	}

	return tx, nil
}

// callbackArgsFromEvent decodes the event payload and returns the callback ID and effort.
//
// The event for processed callback event is emitted by the process callback transaction from
// callback scheduler contract and has the following signature:
// event PendingExecution(id: UInt64, priority: UInt8, executionEffort: UInt64, fees: UFix64, callbackOwner: Address)
func callbackArgsFromEvent(event flow.Event) (uint64, uint64, error) {
	cadenceId, cadenceEffort, err := ParsePendingExecutionEvent(event)
	if err != nil {
		return 0, 0, err
	}

	effort := uint64(cadenceEffort)

	if effort > flow.DefaultMaxTransactionGasLimit {
		log.Warn().Uint64("effort", effort).Msg("effort is greater than max transaction gas limit, setting to max")
		effort = flow.DefaultMaxTransactionGasLimit
	}

	return uint64(cadenceId), uint64(effort), nil
}

func (idx *ScheduledTransactionsIndex) Cancelled(
	lctx lockctx.Proof,
	rw storage.ReaderBatchWriter,
	scheduledTxID uint64,
	feesReturned uint64,
	feesDeducted uint64,
	transactionID flow.Identifier,
) error {
	if !lctx.HoldsLock(storage.LockIndexScheduledTransactionsIndex) {
		return fmt.Errorf("missing required lock: %s", storage.LockIndexScheduledTransactionsIndex)
	}

	key := makeScheduledTxPrimaryKey(scheduledTxID)
	var tx access.ScheduledTransaction
	if err := operation.RetrieveByKey(rw.GlobalReader(), key, &tx); err != nil {
		return fmt.Errorf("could not retrieve scheduled transaction %d: %w", scheduledTxID, err)
	}
	if tx.Status != access.ScheduledTxStatusScheduled {
		return fmt.Errorf("tx %d already in terminal state %s: %w", scheduledTxID, tx.Status, storage.ErrInvalidStatusTransition)
	}

	tx.Status = access.ScheduledTxStatusCancelled
	tx.FeesReturned = feesReturned
	tx.FeesDeducted = feesDeducted
	tx.CancelledTransactionID = transactionID

	if err := operation.UpsertByKey(rw.Writer(), key, tx); err != nil {
		return fmt.Errorf("could not update scheduled transaction %d: %w", scheduledTxID, err)
	}
	return nil
