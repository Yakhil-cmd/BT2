### Title
Unprotected Bank Precompile `transfer` Method Allows Any EVM Contract to Move Native Tokens from Arbitrary Addresses Without Authorization - (File: x/cronos/keeper/precompiles/bank.go)

---

### Summary

The bank precompile's `transfer` (and `burn`) methods accept arbitrary `sender`/`recipient` address arguments with no check that the calling contract is authorized to act on behalf of those addresses. Any deployed EVM contract can call these methods to move or destroy `evm/<contract_address>`-denominated native bank tokens held by any victim, with no consent from the token holder.

---

### Finding Description

In `x/cronos/keeper/precompiles/bank.go`, the `Run` method handles the `transfer` case as follows:

```go
sender    := args[0].(common.Address)   // caller-supplied, arbitrary
recipient := args[1].(common.Address)   // caller-supplied, arbitrary
amount    := args[2].(*big.Int)
...
from := sdk.AccAddress(sender.Bytes())
to   := sdk.AccAddress(recipient.Bytes())
...
denom := EVMDenom(contract.Caller())    // "evm/0x<calling_contract>"
amt   := sdk.NewCoin(denom, sdkmath.NewIntFromBigInt(amount))
...
bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt))
``` [1](#0-0) 

There is **no assertion** that `contract.Caller() == sender`, and no allowance/approval mechanism is consulted. The precompile unconditionally calls `bankKeeper.SendCoins(from, to, ...)` with caller-controlled addresses.

The same structural gap exists in the `burn` case: the address to burn from (`recipient` argument, mapped to `addr`) is fully caller-controlled, with no ownership or approval check. [2](#0-1) 

The denomination is always `evm/<contract.Caller()>` (prefix `"evm/"`), so the attacker can only affect tokens of their own contract's denom — but that is precisely the denom that users accumulate when they interact with that contract (e.g., after the contract calls `mint(victim, amount)`). [3](#0-2) 

The bank precompile is registered at a fixed address (`0x0000…0064`) and is reachable by any EVM contract with no caller whitelist. [4](#0-3) 

---

### Impact Explanation

**Critical — Unauthorized transfer/burn of precompile-controlled assets.**

A malicious contract can:
1. Call `mint(victim, N)` → victim now holds `N` units of `evm/0x<MaliciousContract>` in the native bank module.
2. Later call `transfer(victim, attacker, N)` → the bank precompile moves those tokens from the victim to the attacker with no victim consent.

Alternatively, `burn(victim, N)` destroys the victim's holdings outright.

Because `bankKeeper.SendCoins` operates at the Cosmos bank module level, this bypasses all EVM-layer authorization (ERC-20 allowances, `transferFrom` guards, etc.) and directly mutates native bank balances. The impact class matches "Unauthorized transfer of precompile-controlled assets" in the Critical tier.

---

### Likelihood Explanation

The attacker path requires only:
1. Deploying an EVM contract (unprivileged, permissionless).
2. Inducing users to hold `evm/<contract_address>` tokens — achievable by advertising the contract as a legitimate token issuer and calling `mint` during normal interactions.
3. Calling `transfer(victim, attacker, amount)` at any later time.

No admin keys, governance votes, or validator compromise are needed. The precompile is callable from any EVM transaction.

---

### Recommendation

In the `transfer` case, enforce that the caller is only permitted to move tokens from itself:

```go
if contract.Caller() != sender {
    return nil, errors.New("transfer: caller is not the sender")
}
```

For a richer allowance model, implement an on-chain approval mapping inside the precompile (analogous to ERC-20 `approve`/`transferFrom`) before permitting third-party transfers.

Apply the same fix to the `burn` case: verify `contract.Caller() == recipient` (the address being burned from) before executing `SendCoinsFromAccountToModule`.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

interface IBankPrecompile {
    function mint(address recipient, uint256 amount) external returns (bool);
    function transfer(address sender, address recipient, uint256 amount) external returns (bool);
}

contract BankPrecompileExploit {
    IBankPrecompile constant bank = IBankPrecompile(address(0x64));

    // Step 1: attacker mints tokens to victim as part of a "legitimate" interaction
    function seedVictim(address victim, uint256 amount) external {
        bank.mint(victim, amount);
        // victim now holds `amount` of denom "evm/0x<this contract>" in the bank module
    }

    // Step 2: attacker drains victim's holdings with no victim signature
    function drain(address victim, address attacker, uint256 amount) external {
        bank.transfer(victim, attacker, amount);
        // bankKeeper.SendCoins(victim, attacker, coins) executes unconditionally
    }
}
```

1. Deploy `BankPrecompileExploit` at address `M`.
2. Call `seedVictim(alice, 1000)` — Alice now holds 1000 `evm/0xM` tokens.
3. Call `drain(alice, attacker, 1000)` — Alice's 1000 `evm/0xM` tokens are transferred to the attacker with no authorization from Alice.

The root cause — absence of a `contract.Caller() == sender` guard — is directly analogous to the DSWarp `warp()` auth bypass: a function that is supposed to be restricted to authorized callers is left fully open, allowing any unprivileged actor to manipulate critical state (token balances) with direct financial impact.
