### Title
Unchecked Precompile Return Value in `withdrawToNear` Burns ERC-20 Tokens Without Releasing NEP-141 — (`etc/eth-contracts/contracts/EvmErc20.sol`)

---

### Summary

`EvmErc20.withdrawToNear` burns the caller's ERC-20 tokens unconditionally before calling the `ExitToNear` precompile via a low-level assembly `call`. The return value of that call is captured in a local variable `res` but is **never checked**. If the precompile rejects the input (e.g., invalid recipient bytes), the burn is committed and the function returns successfully, while no NEP-141 tokens are ever released on NEAR. The user's funds are permanently destroyed.

---

### Finding Description

In `EvmErc20.sol` lines 53–63:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);                          // ← burn is unconditional

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f,
                        0, add(input, 32), input_size, 0, 32)
        // res is stored but never tested — no `if iszero(res) { revert(0,0) }`
    }
}
``` [1](#0-0) 

The same pattern exists in `withdrawToEthereum` (lines 65–76). [2](#0-1) 

**Precompile failure paths that produce a silent `res = 0`:**

The `ExitToNear` precompile (`parse_recipient`) rejects input and returns `ExitError` in at least two cases:

1. Recipient bytes are not valid UTF-8 → `ERR_INVALID_RECEIVER_ACCOUNT_ID`
2. The UTF-8 string is not a valid NEAR account ID → `ERR_INVALID_RECEIVER_ACCOUNT_ID` [3](#0-2) 

Additionally, `parse_amount` rejects amounts exceeding `u128::MAX` → `ERR_INVALID_AMOUNT`. [4](#0-3) 

**How the error propagates (and why it is silent):**

`process_precompile` maps any `ExitError` to `PrecompileFailure::Error`:

```rust
p.run(input, gas_limit.map(EthGas::new), context, is_static)
    .map_err(|exit_status| PrecompileFailure::Error { exit_status })
``` [5](#0-4) 

In standard EVM semantics, `PrecompileFailure::Error` causes the sub-call to return `0` to the caller. It does **not** revert the parent call frame. Because `EvmErc20` never tests `res`, the parent frame continues and returns successfully. The `_burn` state change is committed; no promise log is ever emitted; no NEP-141 transfer is ever scheduled.

---

### Impact Explanation

- ERC-20 token supply on Aurora decreases (tokens are burned).
- The corresponding NEP-141 tokens on NEAR are never released.
- The user receives nothing on either side.
- The bridge accounting is permanently out of sync: more NEP-141 tokens exist on NEAR than ERC-20 tokens on Aurora.
- The affected user's funds are **permanently destroyed** — matching the "Permanent freezing of funds" impact category.

---

### Likelihood Explanation

Any call to `withdrawToNear` with:
- non-UTF-8 bytes in `recipient`, or
- a string that fails NEAR account-ID validation (e.g., uppercase letters, leading/trailing dots, double dots, empty string, or a string longer than 64 characters),

will silently burn tokens. This is reachable by any token holder directly or through any contract that wraps the withdraw call. No admin access or special privilege is required.

---

### Recommendation

Add an explicit revert on precompile failure inside the assembly block:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f,
                    0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

Apply the same fix to `withdrawToEthereum`. The same pattern exists verbatim in `EvmErc20V2.sol` and should be fixed there as well. [6](#0-5)

### Proof of Concept

```solidity
// SPDX-License-Identifier: CC0-1.0
pragma solidity ^0.8.0;

interface IEvmErc20 {
    function withdrawToNear(bytes memory recipient, uint256 amount) external;
    function balanceOf(address account) external view returns (uint256);
    function totalSupply() external view returns (uint256);
}

contract PoC {
    IEvmErc20 token;

    constructor(address _token) { token = IEvmErc20(_token); }

    function exploit(uint256 amount) external {
        uint256 supplyBefore = token.totalSupply();
        uint256 balBefore    = token.balanceOf(address(this));

        // Pass a recipient that is invalid UTF-8 (0xFF byte).
        // The ExitToNear precompile will return ExitError, res == 0.
        // EvmErc20 never checks res, so the call succeeds.
        bytes memory badRecipient = hex"ff";
        token.withdrawToNear(badRecipient, amount);

        uint256 supplyAfter = token.totalSupply();
        uint256 balAfter    = token.balanceOf(address(this));

        // Tokens are burned: balAfter == balBefore - amount, supplyAfter == supplyBefore - amount
        // But no NEP-141 was released on NEAR.
        assert(balBefore - balAfter == amount);   // burn happened
        assert(supplyBefore - supplyAfter == amount); // supply decreased
        // No corresponding NEP-141 release → funds permanently lost.
    }
}
```

The same test applies to `withdrawToEthereum` by substituting the precompile address `0xb0bd02f6a392af548bdf1cfaee5dfa0eefcc8eab` and a malformed 20-byte recipient.
