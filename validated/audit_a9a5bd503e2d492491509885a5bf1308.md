### Title
Hardcoded 2300-Gas Stipend on `DepositCallGasLimit` and `DefaultGasLimitForTokenTransfer` Causes Permanent Failure of FLOW Token Deposits to EVM Smart Contracts with Non-Trivial Receive/Fallback Functions - (File: `fvm/evm/types/call.go`)

---

### Summary

`fvm/evm/types/call.go` hardcodes `DepositCallGasLimit = DefaultGasLimitForTokenTransfer = 21_000 + 2_300 = 23_300` for all deposit and transfer `DirectCall`s. The 2300 gas remaining after the 21 000-gas intrinsic cost is the exact same stipend that Solidity's deprecated `transfer()` forwards to a recipient's `receive()`/`fallback()`. Any EVM smart contract whose `receive()` or `fallback()` consumes more than 2300 gas will permanently fail to receive FLOW tokens via the Cadence `EVMAddress.deposit()` or `CadenceOwnedAccount.deposit()` path, causing the entire Cadence transaction to revert.

---

### Finding Description

In `fvm/evm/types/call.go` the following constants are defined:

```go
IntrinsicFeeForTokenTransfer    = gethParams.TxGas          // 21_000
DefaultGasLimitForTokenTransfer = IntrinsicFeeForTokenTransfer + 2_300  // 23_300
DepositCallGasLimit             = DefaultGasLimitForTokenTransfer        // 23_300
WithdrawCallGasLimit            = DefaultGasLimitForTokenTransfer        // 23_300
``` [1](#0-0) 

`NewDepositCall` uses `DepositCallGasLimit`:

```go
func NewDepositCall(...) *DirectCall {
    return &DirectCall{
        ...
        GasLimit: DepositCallGasLimit,   // 23_300
    }
}
``` [2](#0-1) 

`NewTransferCall` uses `DefaultGasLimitForTokenTransfer`:

```go
func NewTransferCall(...) *DirectCall {
    return &DirectCall{
        ...
        GasLimit: DefaultGasLimitForTokenTransfer,  // 23_300
    }
}
``` [3](#0-2) 

`Account.Deposit` in `handler.go` calls `NewDepositCall` and then calls `panicOnErrorOrInvalidOrFailedState`, which panics (reverts the Cadence transaction) if the EVM call fails:

```go
func (a *Account) Deposit(v *types.FLOWTokenVault) {
    ...
    res, err := a.fch.executeAndHandleCall(
        types.NewDepositCall(bridge, a.address, v.Balance(), bridgeAccount.Nonce()),
        v.Balance(), false,
    )
    panicOnErrorOrInvalidOrFailedState(res, err)
}
``` [4](#0-3) 

The Cadence-level `EVMAddress.deposit()` and `CadenceOwnedAccount.deposit()` both ultimately call this Go-level `Deposit`: [5](#0-4) 

The comment in `call.go` itself acknowledges the analogy: *"21_000 is the minimum for a transaction + max gas allowed for receive/fallback methods"* — this is precisely the Solidity `transfer()` stipend pattern.

The existing test suite confirms the failure mode: depositing to a contract whose `receive()` reverts (or uses more than 2300 gas) produces `gethVM.ErrExecutionReverted`:

```go
call := types.NewDepositCall(bridgeAccount, testContract, types.MakeBigIntInFlow(1), 0)
res, err := blk.DirectCall(call)
require.Equal(t, res.VMError, gethVM.ErrExecutionReverted)
``` [6](#0-5) 

---

### Impact Explanation

Any EVM smart contract whose `receive()` or `fallback()` function consumes more than 2300 gas — including proxy contracts, DeFi vaults, contracts that emit events on receipt, or contracts that update storage — **cannot receive FLOW tokens via the Cadence cross-VM deposit path**. The Cadence transaction that attempts the deposit will always revert. This is a permanent, protocol-level cross-VM asset transfer failure: the FLOW tokens cannot be bridged into those EVM addresses through the standard Cadence interface, regardless of how much gas the outer Cadence transaction provides.

---

### Likelihood Explanation

The 2300-gas stipend is extremely restrictive. A single `SSTORE` costs 20 000 gas; a `LOG1` event costs ~375 gas plus topic/data costs. Proxy contracts (which forward calls in their fallback), ERC-4337 smart wallets, DeFi protocols that track deposits, and any contract that does anything non-trivial on receipt will exceed 2300 gas. This is precisely why Ethereum deprecated `transfer()` in favour of `call()`. Any unprivileged user can deploy such a contract on Flow EVM and observe that the Cadence deposit path is permanently broken for that address.

---

### Recommendation

Replace the hardcoded `2_300` stipend with a configurable or sufficiently large gas limit for deposit and transfer direct calls, analogous to replacing Solidity's `transfer()` with `call{value: amount}("")` and checking the return value. For example, allow the caller to pass an explicit gas limit, or raise the default to a value that accommodates typical smart-contract receive logic (e.g., 100 000 gas), while still bounding it to prevent abuse.

---

### Proof of Concept

1. Deploy an EVM smart contract on Flow EVM whose `receive()` function writes to storage (costs >2300 gas):
   ```solidity
   contract HeavyReceiver {
       uint256 public count;
       receive() external payable { count += 1; }  // SSTORE: ~20_000 gas
   }
   ```
2. From a Cadence transaction, call `EVMAddress(bytes: contractAddress).deposit(from: <-vault)`.
3. The `NewDepositCall` is issued with `GasLimit = 23_300`. After the 21 000 intrinsic cost, only 2300 gas remains for the `receive()` body. The `SSTORE` exhausts gas; the EVM call reverts with `ErrExecutionReverted`.
4. `panicOnErrorOrInvalidOrFailedState` panics, reverting the entire Cadence transaction.
5. The FLOW tokens cannot be deposited to this contract via the Cadence bridge path, confirmed by the existing test at `fvm/evm/emulator/emulator_test.go:83–110`. [1](#0-0) [2](#0-1) [4](#0-3) [7](#0-6)
