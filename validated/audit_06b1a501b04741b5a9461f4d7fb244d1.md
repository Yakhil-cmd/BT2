### Title
Unbounded `GetAllBalances` Iteration in `Claim` Precompile Enables Attacker-Controlled Gas Inflation - (File: `precompiles/solo/solo.go`)

### Summary

The `Claim` function in the `solo` precompile calls `bankKeeper.GetAllBalances(ctx, sender)` to retrieve every denomination held by the sender's Cosmos address before transferring them all. Because any unprivileged account can send arbitrary token denominations to any Cosmos address, an attacker can inflate the number of denominations stored for a target sender, linearly increasing the gas cost of every subsequent `Claim` call made on behalf of that sender.

### Finding Description

In `precompiles/solo/solo.go`, the `Claim` function unconditionally fetches and transfers every denomination held by `sender`:

```go
if err := p.bankKeeper.SendCoins(ctx, sender,
    p.evmKeeper.GetSeiAddressOrDefault(ctx, caller),
    p.bankKeeper.GetAllBalances(ctx, sender)); err != nil {
``` [1](#0-0) 

`GetAllBalances` is implemented as an unbounded KV-store iteration over the account's balance prefix:

```go
func (k BaseViewKeeper) GetAllBalances(ctx sdk.Context, addr sdk.AccAddress) sdk.Coins {
    balances := sdk.NewCoins()
    k.IterateAccountBalances(ctx, addr, func(balance sdk.Coin) bool {
        balances = append(balances, balance)
        return false
    })
    return balances.Sort()
}
``` [2](#0-1) 

`IterateAccountBalances` opens a full prefix-store iterator with no upper bound: [3](#0-2) 

The Cosmos SDK gas meter charges per-byte read costs on every iterator step: [4](#0-3) 

**Attack path:**
1. Attacker creates N distinct token denominations (via TokenFactory or IBC) and sends dust amounts to the target sender's Cosmos address.
2. Each denomination adds one entry to the sender's balance KV prefix.
3. When the sender (or their designated claimer) calls `Claim`, `GetAllBalances` iterates all N entries, and `SendCoins` performs 4 KV operations per denomination (read + write for sender, read + write for recipient).
4. Gas cost grows as O(N). With enough denominations the EVM gas limit is exceeded and the call reverts.

No privileged access is required; any account can send tokens to any address in Cosmos SDK.

### Impact Explanation

The `Claim` precompile function becomes unusable for a targeted sender once the attacker has seeded enough denominations. The sender cannot reclaim their Cosmos-side native assets via this path. Impact is bounded to this specific precompile method — the sender retains access to `ClaimSpecific` (which takes an explicit asset list) and standard Cosmos `MsgSend` transactions. No funds are permanently lost or frozen. This maps to **Low** severity under the Sei bounty scope (wrong/inflated fee calculation, resource DoS on a single user function).

### Likelihood Explanation

Moderate. Creating denominations via TokenFactory costs gas per denom, and IBC transfers also cost gas, so the attacker must spend proportional resources. However, the attack is permissionless and requires no special role. The cost to the attacker scales with the number of denominations needed to push the victim's `Claim` call over the EVM gas limit.

### Recommendation

1. Cap the number of denominations processed in a single `Claim` call, or reject the call if `GetAllBalances` returns more than a configurable maximum.
2. Alternatively, require the caller to supply an explicit list of denominations (similar to `ClaimSpecific`), removing the unbounded iteration entirely from the `Claim` path.
3. Consider charging a per-denomination flat fee to the caller before processing, making the attack economically unattractive.

func (k BaseViewKeeper) IterateAccountBalances(ctx sdk.Context, addr sdk.AccAddress, cb func(sdk.Coin) bool) {
	accountStore := k.getAccountStore(ctx, addr)

	iterator := accountStore.Iterator(nil, nil)
	defer func() { _ = iterator.Close() }()

	for ; iterator.Valid(); iterator.Next() {
		var balance sdk.Coin
		k.cdc.MustUnmarshal(iterator.Value(), &balance)

		if cb(balance) {
			break
		}
	}
}
