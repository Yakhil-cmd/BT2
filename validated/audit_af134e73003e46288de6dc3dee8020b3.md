### Title
Unbounded `priceProviderTimelock` causes permanent DoS on price-provider update via arithmetic overflow — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`createPool` stores `params.priceProviderTimelock` without any upper-bound validation. The sentinel `type(uint256).max` marks a price provider as immutable, but any value just below it (e.g. `type(uint256).max - 1`) is accepted as a valid mutable timelock. When `proposePoolPriceProvider` later computes `block.timestamp + timelock`, Solidity 0.8's checked arithmetic reverts on overflow, permanently DoS-ing the only path to replace a broken oracle. LPs who deposited into a pool they believed had a mutable price provider are then trapped with a stale or malfunctioning feed.

---

### Finding Description

`_validatePoolParameters` enforces caps on fees, initial amounts, and admin addresses, but performs no check on `params.priceProviderTimelock`: [1](#0-0) 

The value is stored verbatim: [2](#0-1) [3](#0-2) 

The immutability sentinel is an exact equality check:

```solidity
bool immutablePriceProvider = params.priceProviderTimelock == type(uint256).max;
```

So `type(uint256).max - 1` passes the sentinel check (pool is treated as **mutable**) yet is stored as the timelock. Later, `proposePoolPriceProvider` executes: [4](#0-3) 

`block.timestamp + (type(uint256).max - 1)` overflows under Solidity ≥ 0.8 checked arithmetic and **always reverts**, making it impossible to ever propose a new price provider. Even without overflow, any timelock measured in decades (e.g. `100 * 365 days`) effectively freezes the oracle update path for the pool's lifetime.

---

### Impact Explanation

A pool whose price provider becomes stale, paused, or malicious cannot be rescued: `proposePoolPriceProvider` reverts on every call, so `executePoolPriceProviderUpdate` is never reachable. The pool continues to call `getBidAndAskPrice()` on the broken provider; every swap either reverts (via `PriceProviderFailed`) or executes at a bad price. LPs cannot safely remove liquidity at fair value, and swappers may receive incorrect output amounts. This maps directly to the allowed impacts: **bad-price execution** and **broken core pool functionality causing loss of funds**.

---

### Likelihood Explanation

The trigger is a single `uint256` field in `PoolParameters` with no on-chain guard. It can be set accidentally (a developer intending `type(uint256).max` for immutability but using `type(uint256).max - 1`) or deliberately by a pool creator who wants to appear to offer a mutable oracle while actually locking it. Any pool creator — an unprivileged address — can deploy such a pool. LPs have no on-chain way to distinguish a legitimately long timelock from an overflow-inducing one before depositing.

---

### Recommendation

Add an explicit upper bound on `priceProviderTimelock` inside `_validatePoolParameters`, analogous to the fee caps already enforced there:

```solidity
uint256 internal constant MAX_PRICE_PROVIDER_TIMELOCK = 30 days; // chain-appropriate

// inside _validatePoolParameters:
if (
    params.priceProviderTimelock != type(uint256).max &&   // not the immutable sentinel
    params.priceProviderTimelock > MAX_PRICE_PROVIDER_TIMELOCK
) revert PriceProviderTimelockTooLong();
```

This mirrors the fix applied to `queryDeadlineInterval` in AxiomV2Query (PR 88), which added a per-chain upper bound of one week in blocks.

---

### Proof of Concept

```solidity
// 1. Deploy pool with near-max timelock (not the immutable sentinel)
PoolParameters memory params = /* ... */;
params.priceProviderTimelock = type(uint256).max - 1;
address pool = factory.createPool(params);

// immutablePriceProvider == false  →  pool appears mutable

// 2. LPs add liquidity, believing the oracle can be replaced if needed
pool.addLiquidity(/* ... */);

// 3. Oracle feed goes stale / returns bad prices

// 4. Pool admin attempts to propose a replacement
factory.proposePoolPriceProvider(pool, newProvider);
// ↑ reverts: block.timestamp + (type(uint256).max - 1) overflows
//   under Solidity 0.8 checked arithmetic

// 5. executePoolPriceProviderUpdate is unreachable; pool is permanently
//    stuck on the broken oracle; every swap executes at a bad price
//    or reverts, trapping LP principal.
```
