Audit Report

## Title
ckERC20 Minter Mints Based on Event-Logged Amount, Not Actual Received Amount, Enabling Over-Minting with Fee-on-Transfer ERC-20 Tokens - (File: `rs/ethereum/cketh/minter/src/deposit.rs`)

## Summary
The ckERC20 deposit helper contracts (`DepositHelperWithSubaccount.sol`, `ERC20DepositHelper.sol`) emit the caller-supplied `amount` in the `ReceivedEthOrErc20`/`ReceivedErc20` event rather than the actual balance delta received by the minter address. The IC minter trusts this event value verbatim and mints the corresponding ckERC20 amount with no on-chain balance verification. For any fee-on-transfer ERC-20 token that is or becomes a supported ckERC20 token, this allows unbounded over-minting relative to the actual ERC-20 reserve, breaking the 1:1 backing invariant and eventually rendering some ckERC20 tokens unredeemable.

## Finding Description

**Root cause — Solidity helper contracts:**

`DepositHelperWithSubaccount.sol` lines 519–531: `safeTransferFrom` transfers `amount` from the caller to `minterAddress`, but for a fee-on-transfer token the minter actually receives `amount - fee`. The event is then emitted with the original `amount`, not the balance delta:

```solidity
erc20Token.safeTransferFrom(msg.sender, minterAddress, amount);
emit ReceivedEthOrErc20(erc20Address, msg.sender, amount, principal, subaccount);
```

The identical pattern exists in `ERC20DepositHelper.sol` lines 498–502.

**Log parser — `eth_logs/parser.rs` lines 149–160:**

The minter reads `value` directly from the event data bytes (the `amount` field set by the Solidity contract) with no cross-check against the minter's actual ERC-20 balance:

```rust
let value = Erc20Value::from_be_bytes(value_bytes);
Ok(ReceivedErc20Event { ..., value, ... }.into())
```

**Mint logic — `deposit.rs` lines 73–81:**

The minter calls `icrc1_transfer` with `amount: event.value()` — the event-logged value — as the mint amount:

```rust
client.transfer(TransferArg {
    ...
    amount: event.value(),  // taken verbatim from the Ethereum log
})
```

There is no step anywhere in the IC minter that queries the minter's actual ERC-20 balance before or after the deposit to verify the received amount matches the event-logged amount. A grep across `rs/ethereum/cketh/**` confirms zero references to fee-on-transfer, balance delta, or rebasing checks.

**Existing guards are insufficient:** The `add_ckerc20_token` endpoint (`main.rs` lines 562–573) only validates that the caller is the orchestrator and that the token is not a duplicate. It performs no on-chain verification that the ERC-20 token is fee-free. The `process_event` / `record_add_ckerc20_token` path in state tests confirms the same — only network and address uniqueness are checked.

## Impact Explanation

This is a **High** severity finding matching the allowed impact: *"Significant Chain Fusion, ck-token, ledger … security impact with concrete user or protocol harm."*

For any fee-on-transfer ERC-20 token active as a supported ckERC20 token, the minter mints `amount` ckERC20 tokens while holding only `amount - fee` ERC-20 tokens. The ckERC20 total supply grows beyond the minter's ERC-20 reserve by `fee` per deposit. Accumulated over many deposits, the shortfall grows unboundedly. Eventually, some ckERC20 holders cannot redeem their tokens because the minter's ERC-20 reserve is insufficient — a direct ledger conservation violation with concrete, irreversible user fund loss.

USDT (`0xdAC17F958D2ee523a2206206994597C13D831ec7`) is already listed as a supported ckERC20 token in `rs/ethereum/cketh/docs/ckerc20.adoc` lines 37–38. USDT contains a dormant fee switch controllable by Tether. If Tether enables it, every subsequent ckUSDT deposit silently over-mints.

## Likelihood Explanation

Two realistic paths exist, neither requiring IC-side privilege:

1. **Post-listing fee activation (Path 2 — higher concern):** USDT is already supported. Tether enabling the USDT transfer fee requires no IC governance action and is entirely outside IC control. Once enabled, any unprivileged Ethereum user can exploit this by calling `depositErc20` on the helper contract. The exploit is trivially repeatable and accumulates a growing shortfall with each deposit.

2. **New fee-on-transfer token added via NNS (Path 1):** A new fee-on-transfer token is proposed and accepted via NNS governance. The `add_ckerc20_token` endpoint performs no automated on-chain verification that the token is fee-free.

Path 2 is the more immediate concern: it is triggered by a third-party action (the ERC-20 token owner enabling fees) and requires no IC-side change.

## Recommendation

1. **In the Solidity helper contracts:** Measure the minter's balance before and after `safeTransferFrom` and emit the actual received amount (balance delta) in the event:

```solidity
uint256 balanceBefore = erc20Token.balanceOf(minterAddress);
erc20Token.safeTransferFrom(msg.sender, minterAddress, amount);
uint256 actualReceived = erc20Token.balanceOf(minterAddress) - balanceBefore;
emit ReceivedEthOrErc20(erc20Address, msg.sender, actualReceived, principal, subaccount);
```

2. **In the IC minter:** Add a validation step or governance gate in `add_ckerc20_token` that explicitly rejects fee-on-transfer tokens. The minter could query the ERC-20 token's balance before and after a test transfer during proposal execution.

3. **Governance process:** Require that any NNS proposal to add a new ckERC20 token includes explicit verification that the token does not implement fee-on-transfer or rebasing mechanics.

## Proof of Concept

1. USDT is already a supported ckERC20 token (confirmed in `ckerc20.adoc` line 37–38).
2. Tether enables the USDT transfer fee (e.g., 1 basis point = 0.01%).
3. Attacker calls `depositErc20(USDT_ADDRESS, 1_000_000_000, principal, subaccount)` on the helper contract (`DepositHelperWithSubaccount.sol` lines 511–532).
4. `safeTransferFrom` moves `999_900_000` USDT to the minter address (fee = 100,000 units).
5. The helper emits `ReceivedEthOrErc20(..., 1_000_000_000, ...)` with the full requested amount.
6. The IC minter scrapes the log; `parser.rs` lines 149–160 parse `value = 1_000_000_000` from the event data.
7. `deposit.rs` lines 73–81 call `icrc1_transfer` minting `1_000_000_000` ckUSDT to the attacker.
8. The attacker received 100,000 more ckUSDT than the USDT deposited.
9. Repeated deposits accumulate a growing shortfall; the last redeemers cannot withdraw their ckUSDT.

A deterministic integration test can be written using the existing `CkErc20Setup` test harness (`rs/ethereum/cketh/minter/tests/ckerc20.rs`) by deploying a mock ERC-20 token that deducts a fee in `transferFrom` and verifying that the minted ckERC20 balance exceeds the minter's ERC-20 reserve after deposit.
