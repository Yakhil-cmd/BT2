### Title
Gateway `skip_stateful_validations` Admits Unsigned Invoke Transactions for Any Address with a Pending Deploy-Account — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The gateway's UX shortcut for the deploy-account + invoke flow unconditionally skips the `__validate__` entry-point (i.e., signature verification) for any invoke transaction whose nonce is 1 and whose sender address already appears in the mempool pool or recent-block state. Because the check is purely address-based and not key-based, any observer can submit a signature-free invoke transaction on behalf of any account that has a pending deploy-account, and the gateway will accept it into the mempool without ever verifying the signature.

### Finding Description

`skip_stateful_validations` (lines 429–461 of `crates/apollo_gateway/src/stateful_transaction_validator.rs`) returns `true` — meaning "skip the `__validate__` entry point" — when all three conditions hold:

1. The transaction is an `Invoke`.
2. `tx.nonce() == Nonce(Felt::ONE)`.
3. `account_nonce == Nonce(Felt::ZERO)` **and** `account_tx_in_pool_or_recent_block(sender_address)` returns `true`. [1](#0-0) 

When the function returns `true`, `run_validate_entry_point` sets `execution_flags.validate = false`, so the blockifier's `StatefulValidator` returns immediately after `perform_pre_validation_stage` without ever calling `__validate__`: [2](#0-1) 

The condition `account_tx_in_pool_or_recent_block` is satisfied whenever the target address has **any** transaction in the pool or has appeared in a committed block: [3](#0-2) 

`tx_pool.contains_account` returns `true` as soon as a deploy-account for that address is sitting in the pool — before it is ever executed: [4](#0-3) 

The mempool's own `validate_tx` (called earlier in `validate_by_mempool`) performs only duplicate-hash and nonce-range checks; it never inspects the signature: [5](#0-4) [6](#0-5) 

The gateway's own `validate_nonce` (called before `skip_stateful_validations`) accepts nonce 1 when account nonce is 0 as long as `max_allowed_nonce_gap ≥ 1`, which is required for the UX feature to function at all: [7](#0-6) 

The full pre-validation order is: `validate_state_preconditions` → `validate_by_mempool` → `skip_stateful_validations` → `run_validate_entry_point`. None of the first three steps verify the signature; the fourth is skipped. [8](#0-7) 

### Impact Explanation

An attacker who observes that address `A` has a pending deploy-account in the mempool can submit an invoke transaction for `A` with nonce 1, arbitrary calldata, and a completely fabricated signature. The gateway accepts it into the mempool without any cryptographic check. This satisfies the **High** impact criterion: *"Mempool/gateway/RPC admission accepts invalid transactions … before sequencing."*

Secondary consequence: the mempool's fee-escalation logic allows the attacker's transaction to **displace** the legitimate user's nonce-1 invoke if the attacker bids a higher fee. The legitimate invoke is then rejected at submission time. The attacker's transaction is eventually rejected by the blockifier during execution (because the blockifier always calls `__validate__` with `validate=true`), but by then the victim's transaction slot is gone and must be resubmitted.

### Likelihood Explanation

The attack requires only:
1. Monitoring the public mempool for deploy-account transactions (trivial on a p2p network).
2. Constructing an invoke transaction with the victim's address, nonce 1, and any signature bytes.
3. Submitting it to the gateway's public JSON-RPC endpoint.

No privileged access, no special key material, and no race condition is required. The window is open for the entire lifetime of the deploy-account transaction in the mempool.

### Recommendation

The skip-validate path must not be treated as a full bypass of signature verification. Options:

- **Preferred**: Perform a lightweight ECDSA/Stark-curve signature check against the account's public key (stored in the deploy-account's constructor calldata) before granting the skip, so only the legitimate owner of the pending deploy-account can benefit from the UX shortcut.
- **Alternative**: Remove the skip entirely and require wallets to submit deploy-account and invoke in separate round-trips, or accept that the invoke will be re-validated once the deploy-account is mined.
- **Minimum**: Restrict the skip to transactions whose `tx_hash` matches a hash that was co-submitted with the deploy-account in the same gateway call, binding the two transactions cryptographically.

        &mut self,
        executable_tx: &ExecutableTransaction,
        skip_validate: bool,
    ) -> StatefulTransactionValidatorResult<()> {
        let only_query = false;
        let charge_fee = enforce_fee(executable_tx, only_query);
        let strict_nonce_check = false;
        let execution_flags =
            ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };

        let account_tx = AccountTransaction { tx: executable_tx.clone(), execution_flags };

        // Build block context.
        let mut versioned_constants = VersionedConstants::get_versioned_constants(
            self.config.versioned_constants_overrides.clone(),
        );
        // The validation of a transaction is not affected by the casm hash migration.
        versioned_constants.disable_casm_hash_migration();

        let mut block_info = self.gateway_fixed_block_state_reader.get_block_info().await?;
        block_info.block_number = block_info.block_number.unchecked_next();
        let block_context = BlockContext::new(
            block_info,
            self.chain_info.clone(),
            versioned_constants,
            BouncerConfig::max(),
        );

        // Move state into the blocking task and run CPU-heavy validation.
        let state_reader_and_contract_manager = self.take_state_reader_and_contract_manager();

        let cur_span = Span::current();
        #[allow(clippy::result_large_err)]
        tokio::task::spawn_blocking(move || {
            cur_span.in_scope(|| {
                let state = CachedState::new(state_reader_and_contract_manager);
                let mut blockifier_validator = StatefulValidator::create(state, block_context);
                blockifier_validator.validate(account_tx)
            })
        })
        .await
        .map_err(|e| StarknetError {
            code: StarknetErrorCode::UnknownErrorCode(
                "StarknetErrorCode.InternalError".to_string(),
            ),
            message: format!("Blocking task join error when running the validate entry point: {e}"),
        })?
        .map_err(|e| StarknetError {
            code: StarknetErrorCode::KnownErrorCode(KnownStarknetErrorCode::ValidateFailure),
            message: e.to_string(),
        })?;
        Ok(())
    }

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
