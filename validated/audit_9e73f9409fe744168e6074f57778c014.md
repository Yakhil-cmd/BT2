### Title
COA Ownership Proof Signature Replay Across Multiple COAs of the Same Cadence Account - (File: fvm/evm/stdlib/contract.cdc)

### Summary

`EVM.validateCOAOwnershipProof()` does not enforce that `signedData` encodes the target `evmAddress`. A signature produced by a Cadence account to prove ownership of one COA (Cadence-Owned Account) can be replayed by an unprivileged attacker to forge a valid ownership proof for a different COA belonging to the same Cadence account. This is the direct Flow analog of the ERC-7739 `verifyingContract` replay bug: both stem from a missing context-binding identifier in the signed payload.

### Finding Description

`EVM.validateCOAOwnershipProof()` is a public (`access(all)`) Cadence function that verifies a set of signatures over an arbitrary `signedData` blob, then checks that the COA stored at the given `path` matches the supplied `evmAddress`. [1](#0-0) 

The signature verification step at line 1082–1086 verifies the signatures over `signedData` using the fixed domain tag `"FLOW-V0.0-user"`: [2](#0-1) 

Critically, **nothing in the protocol requires `signedData` to encode `evmAddress`**. The code comment at lines 1003–1009 explicitly acknowledges this gap: [3](#0-2) 

The `COAOwnershipProofInContext` struct confirms that `SignedData` and `EVMAddress` are independent fields with no enforced binding: [4](#0-3) 

The EVM-side path — `COA.isValidSignature()` in `coa.sol` — is **not** vulnerable because it always passes `address(this)` as the EVM address argument to the precompile, binding the proof to the calling COA: [5](#0-4) 

However, `validateCOAOwnershipProof` is also directly callable from any Cadence script or transaction (it is `access(all)`), and in that path the caller controls all arguments including `evmAddress`, `path`, and `signedData`.

### Impact Explanation

If a Cadence account holds two COAs published at different public paths (e.g., `/public/coa1` and `/public/coa2`), a signature Alice produced over some `signedData` to prove ownership of COA-A can be replayed verbatim — with `path` changed to `/public/coa2` and `evmAddress` changed to COA-B's address — to obtain a `ValidationResult{isValid: true}` for COA-B.

Any on-chain protocol or dApp that calls `EVM.validateCOAOwnershipProof` directly from a Cadence transaction to gate privileged actions (e.g., asset withdrawals, capability grants, bridge operations) would accept this forged proof, granting the attacker unauthorized access to COA-B's assets or capabilities without possessing Alice's private key — only a previously intercepted or publicly observed signature for COA-A is required.

### Likelihood Explanation

The preconditions are:
1. A Cadence account with **two or more COAs** published at different public paths. While currently uncommon (most wallets create one COA per account), the protocol explicitly supports multiple COAs and nothing prevents this.
2. A protocol that calls `validateCOAOwnershipProof` directly from Cadence (not via the EVM precompile path) to gate on-chain actions.
3. The attacker has observed a valid signature from the victim account over any `signedData` that does not encode the target COA's address.

Condition 3 is easily satisfied: any prior authentication challenge response, ERC-1271 call, or off-chain login flow produces a reusable signature. As multi-COA usage grows (e.g., DeFi protocols assigning separate COAs per vault), likelihood increases.

### Recommendation

Enforce that `signedData` commits to the target `evmAddress` inside `validateCOAOwnershipProof`. The simplest fix is to prepend the `evmAddress` bytes to the data that is actually verified:

```cadence
// Inside validateCOAOwnershipProof, replace:
let isValid = keyList.verify(
    signatureSet: signatureSet,
    signedData: signedData,
    domainSeparationTag: "FLOW-V0.0-user"
)

// With:
let boundData = evmAddress.toVariableSized().concat(signedData)
let isValid = keyList.verify(
    signatureSet: signatureSet,
    signedData: boundData,
    domainSeparationTag: "FLOW-V0.0-user"
)
```

This mirrors the ERC-7739 fix: bind the signed payload to the specific target identity (`evmAddress`) so that a signature for COA-A cannot be replayed against COA-B.

### Proof of Concept

**Setup**: Cadence account `Alice` holds two COAs:
- COA-A at `/public/coaA` with EVM address `0xAAAA...`
- COA-B at `/public/coaB` with EVM address `0xBBBB...`

**Step 1 — Obtain a legitimate signature**: Alice signs `signedData = [0x01, 0x02, 0x03]` with her account key to prove ownership of COA-A to some dApp. The signature is publicly observable (e.g., submitted on-chain or via an API).

**Step 2 — Replay against COA-B**: An attacker submits a Cadence script:

```cadence
import EVM from 0x...

access(all)
fun main(): EVM.ValidationResult {
    return EVM.validateCOAOwnershipProof(
        address: aliceFlowAddress,
        path: /public/coaB,          // ← changed to COA-B's path
        signedData: [0x01, 0x02, 0x03], // ← same signedData as step 1
        keyIndices: [0],
        signatures: [aliceSignatureFromStep1], // ← replayed signature
        evmAddress: [0xBB, 0xBB, ...]  // ← COA-B's address
    )
}
```

**Result**: `validateCOAOwnershipProof` verifies the signature over `[0x01, 0x02, 0x03]` (valid, since Alice signed it), then checks that `/public/coaB` holds EVM address `0xBBBB...` (true). It returns `ValidationResult{isValid: true}` — a forged proof of COA-B ownership — without the attacker ever possessing Alice's private key. [6](#0-5)
