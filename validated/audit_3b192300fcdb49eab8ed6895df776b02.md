### Title
Unguarded `sweepToken`, `unwrapWETH9`, and `refundETH` allow any caller to steal stranded router balances â€” (`File: metric-periphery/contracts/base/PeripheryPayments.sol`)

---

### Summary

`PeripheryPayments.sweepToken`, `PeripheryPayments.unwrapWETH9`, and `PeripheryPayments.refundETH` are all public/external with no access control and no caller-attribution check. Any address can invoke them at any time and redirect the router's entire token, WETH, or ETH balance to an arbitrary recipient. Because the `pay` helper deliberately leaves excess native ETH on the router when `msg.value > amountIn`, a realistic stranding path exists in every ETH-input swap that is not wrapped in a `multicall` + `refundETH` pair.

---

### Finding Description

`MetricOmmSimpleRouter` inherits `PeripheryPayments` and exposes three balance-draining helpers with no restriction on who may call them or who the beneficiary is:

```solidity
// metric-periphery/contracts/base/PeripheryPayments.sol

function unwrapWETH9(uint256 amountMinimum, address recipient) public payable override { â€¦ }
function sweepToken(address token, uint256 amountMinimum, address recipient) public payable override { â€¦ }
function refundETH() external payable override {
    uint256 balance = address(this).balance;
    if (balance > 0) { _transferETH(msg.sender, balance); }
}
``` [1](#0-0) 

The concrete stranding path lives in the `pay` helper. When the token being paid is WETH and the router holds native ETH, it deposits exactly `value` wei and transfers that WETH to the pool. Any excess native ETH (`msg.value - amountIn`) is silently left on the router:

```solidity
if (nativeBalance >= value) {
    IWETH9(WETH).deposit{value: value}();
    IERC20(WETH).safeTransfer(recipient, value);
    // nativeBalance - value remains on the router
}
``` [2](#0-1) 

The protocol's own test confirms this design: a user sends `2 ether` for a swap of `1_000 wei` and must include `refundETH` in the same multicall to recover the difference. If the user calls `exactInputSingle` directly (not via multicall), or omits `refundETH` from their multicall, the excess ETH is permanently stranded on the router until an attacker drains it. [3](#0-2) 

The same pattern applies to `sweepToken` (any ERC-20 accidentally transferred to the router, or any token residue from a partial path) and `unwrapWETH9` (WETH sent to the router as an intermediate-hop recipient).

---

### Impact Explanation

Any ETH, WETH, or ERC-20 balance that reaches the router â€” through user error, an overpaid `msg.value`, or an accidental direct transfer â€” can be immediately and completely stolen by an unprivileged third party. The loss is 100% of the stranded balance with no recovery path. This is a direct loss of user principal above Sherlock thresholds whenever a user sends even a small amount of excess ETH with a swap.

---

### Likelihood Explanation

The stranding path for ETH is not hypothetical: it is the documented, tested design. Users who call `exactInputSingle` with `msg.value > amountIn` without a `refundETH` multicall step lose their excess ETH. Front-running bots can monitor the mempool for such transactions and call `refundETH` in the same block. Accidental ERC-20 transfers to the router address are also a known operational risk explicitly acknowledged in the Uniswap-lineage design this contract mirrors.

---

### Recommendation

Restrict `sweepToken`, `unwrapWETH9`, and `refundETH` so that only the address that originated the current multicall (or `msg.sender` of the outermost call) can invoke them, or record the payer in transient storage at the start of each top-level call and enforce it inside the helpers. A minimal fix is to add a `checkDeadline`-style transient-storage payer guard that is set at the entry of `exactInputSingle` / `exactInput` / `exactOutput` and checked inside the three drain helpers, rejecting any call that does not originate from the recorded payer.

---
