### Title
Gateway Unconditionally Skips Account `__validate__` (Signature Verification) for Nonce-1 Invoke Transactions When Any Pending Transaction Exists for the Sender — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator permanently bypasses the account contract's `__validate__` entry point — the sole on-chain signature-verification step — for any Invoke V3 transaction whose nonce equals 1 when the on-chain account nonce is 0 and `account_tx_in_pool_or_recent_block` returns `true`. Because the mempool is public and any observer can see pending `DeployAccount` transactions, an unprivileged attacker can submit an Invoke with an arbitrary (invalid) signature for any address that has a pending `DeployAccount`, and the gateway will admit it to the mempool without ever calling `__validate__`.

---

### Finding Description

**Step 1 — skip decision.**

`skip_stateful_validations` returns `true` (skip) when all three conditions hold:

```
tx.nonce() == Nonce(Felt::ONE)
account_nonce == Nonce(Felt::ZERO)
mempool_client.account_tx_in_pool_or_recent_block(sender) == true
``` [1](#0-0) 

The comment in the code acknowledges the check is intentionally broad: *"It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."* For an undeployed account (nonce = 0), the only transaction that can satisfy `account_tx_in_pool_or_recent_block` is a pending `DeployAccount` — which is publicly visible in the P2P mempool.

**Step 2 — validation flag propagation.**

When `skip_validate = true`, `run_validate_entry_point` sets `execution_flags.validate = false`:

```rust
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [2](#0-1) 

**Step 3 — `__validate__` is never called.**

`AccountTransaction::validate_tx` short-circuits immediately when `execution_flags.validate` is `false`:

```rust
if !self.execution_flags.validate {
    return Ok(None);
}
``` [3](#0-2) 

`__validate__` is the account contract's only opportunity to verify the ECDSA signature (or any custom authentication). Skipping it means the gateway admits the transaction to the mempool with zero cryptographic verification of the caller's authorization.

**Step 4 — the full gateway admission path.**

The gateway calls `extract_state_nonce_and_run_validations`, which calls `run_pre_validation_checks` (nonce + resource-bounds checks) and then `run_validate_entry_point` with the skip flag: [4](#0-3) 

Stateless validation (signature *size*, resource bounds, DA modes) still runs, but it does not verify the cryptographic *content* of the signature. [5](#0-4) 

---

### Impact Explanation

**Impact: High — Mempool/gateway admission accepts invalid transactions before sequencing.**

An attacker can inject an Invoke transaction carrying a completely fabricated signature into the mempool for any address that has a pending `DeployAccount`. The transaction passes every gateway check and is forwarded to the mempool. Consequences:

1. **Mempool pollution at scale.** Because `DeployAccount` transactions are broadcast over P2P, an attacker can monitor the network and immediately submit a spoofed nonce-1 Invoke for every new `DeployAccount` it observes, filling the mempool with transactions that will revert.
2. **Sequencer resource exhaustion.** The batcher will dequeue and attempt to execute each spoofed transaction. The account's `__validate__` will reject it during execution, but only after consuming block-building resources.
3. **Legitimate nonce-1 slot displacement.** The spoofed transaction occupies the nonce-1 slot for the victim's address. The legitimate user must pay a fee-escalation premium (≥ 10 %) to replace it.

The broken invariant is: *every transaction in the mempool has passed account-level authorization (signature verification)*. This invariant is unconditionally violated for the nonce-1 / account-nonce-0 case whenever a `DeployAccount` is pending.

---

### Likelihood Explanation

**Likelihood: High.**

- No special privilege is required; the attacker only needs to observe the public P2P mempool.
- `DeployAccount` transactions are broadcast in plaintext before confirmation.
- The attack is fully automated: watch for `DeployAccount`, immediately submit a spoofed nonce-1 Invoke.
- The default production config sets `max_nonce_for_validation_skip = Nonce(Felt::ONE)` and `max_allowed_nonce_gap = 200`, so the conditions are reachable in normal operation. [6](#0-5) 

---

### Recommendation

1. **Narrow the skip condition.** Instead of checking whether *any* transaction from the sender is in the pool, verify that a `DeployAccount` transaction specifically is present and that its computed contract address matches the Invoke's `sender_address`. This prevents an attacker from exploiting the skip for addresses whose pool entry is an unrelated future-nonce transaction.

2. **Perform a lightweight signature pre-check.** Even when skipping the full `__validate__` entry-point execution, validate that the signature field is structurally consistent with the account class (e.g., correct length for ECDSA) and, where possible, verify the ECDSA signature off-chain before admission.

3. **Bound the skip to a single in-flight `DeployAccount`.** Track the specific `tx_hash` of the pending `DeployAccount` and only skip `__validate__` for the Invoke if that exact `DeployAccount` is still unconfirmed. Remove the skip once the `DeployAccount` is included in a block.

---

### Proof of Concept

```
1. Attacker monitors the P2P mempool and observes:
      DeployAccount { sender: 0xALICE, nonce: 0, ... }

2. Attacker constructs:
      Invoke V3 {
          sender_address: 0xALICE,
          nonce: 1,
          signature: [0xDEAD, 0xBEEF],   // completely invalid
          calldata: [...],
          resource_bounds: { l2_gas: { max_amount: 1, max_price_per_unit: 1 }, ... },
          ...
      }

3. Attacker submits the Invoke to the gateway HTTP endpoint.

4. Gateway flow:
   a. StatelessTransactionValidator::validate() — passes (signature size ≤ limit, resource bounds > 0).
   b. TransactionConverter::convert_rpc_tx_to_internal() — computes tx_hash, no signature check.
   c. StatefulTransactionValidator::extract_state_nonce_and_run_validations():
      - get_nonce(0xALICE) → 0                          ✓
      - validate_nonce(nonce=1, account_nonce=0) → OK    ✓ (within gap)
      - validate_by_mempool() → OK
      - skip_stateful_validations():
            tx.nonce()==1 ✓, account_nonce==0 ✓,
            account_tx_in_pool_or_recent_block(0xALICE)==true ✓
            → returns true (SKIP)
      - run_validate_entry_point(skip_validate=true):
            execution_flags.validate = false
            validate_tx() → Ok(None)  // __validate__ never called
   d. mempool_client.add_tx() → transaction admitted.

5. Result: the Invoke with signature [0xDEAD, 0xBEEF] is now in the mempool.
   When the batcher executes it, __validate__ rejects it and the transaction reverts,
   but the mempool admission invariant has been violated.
```

        }
