### Title
Hardcoded `wallet_id: 1` Default and Hidden Wallet Context in WalletConnect `chia_sendTransaction` Allows Dapp to Drain Any Wallet Without User Awareness - (File: `packages/gui/src/electron/commands/Commands.ts`)

---

### Summary

Multiple WalletConnect spending commands — most critically `chia_sendTransaction` — hardcode `wallet_id: 1` as a static default and simultaneously mark `wallet_id` as `hide: true` in the confirmation dialog. A paired dapp can supply any `wallet_id` (e.g., a high-value CAT wallet), the parameter passes validation, and the user approves the transaction without ever seeing which wallet is being spent from.

---

### Finding Description

**Hardcoded default — the direct analog to the external bug:**

In `Commands.ts`, six WalletConnect dapp commands carry `defaults: { wallet_id: 1 }`:

- `chia_sendTransaction` (line 174)
- `chia_getWalletBalance` (line 1597)
- `chia_selectCoins` (line 1748)
- `chia_getSpendableCoins` (line 1790)
- `chia_getNextAddress` (line 1852)
- `chia_getCurrentAddress` (line 1860) [1](#0-0) [2](#0-1) [3](#0-2) 

Just as the external report's `WETH` constant hardcodes a chain-specific address that is wrong on other chains, `wallet_id: 1` hardcodes the XCH standard wallet — which is wrong for users whose primary asset is a CAT or who intend to operate on a different wallet.

**Hidden wallet context — the exploitation surface:**

The `wallet_id` param for `chia_sendTransaction` is declared with `hide: true`: [4](#0-3) 

The `hide` flag suppresses the field from the main confirmation dialog (it is only visible under a collapsed "JSON details" section). The user sees `amount`, `fee`, and `address` — but not which wallet is being spent from.

**`parseDappParams` accepts any dapp-supplied `wallet_id`:**

Because `wallet_id` is present in the command's `params` list, `parseDappParams` treats it as an allowed field and passes it through to the RPC call: [5](#0-4) 

A dapp can therefore supply `wallet_id: 5` (a CAT wallet), and `parseDappParams` will accept it, apply no default override (since the field is already set), and forward it to `sendCommand`.

**Execution path:**

```
Dapp → chia_sendTransaction { wallet_id: 5, amount: X, address: attacker }
  → WalletConnectProvider.processSessionRequest
  → useWalletConnectCommand.handleProcess (fingerprint/command checks pass)
  → permissionsAPI.dispatchAsPair
  → main.tsx DISPATCH_AS_PAIR handler
  → parseDappParams  ← wallet_id: 5 accepted, not defaulted
  → dispatchPairRequest (no wallet_id check)
  → sendCommand('send_transaction', 'chia_wallet', { wallet_id: 5, ... })
  → Chia daemon spends from CAT wallet #5
``` [6](#0-5) [7](#0-6) 

---

### Impact Explanation

**High** — A WalletConnect-paired dapp that has been granted `chia_sendTransaction` permission can silently redirect the spend to any wallet the user holds (XCH, CAT, CR-CAT). The user's confirmation dialog shows the correct amount and destination address but omits the wallet identity. The user approves believing they are spending from the XCH wallet while the dapp has targeted a high-value CAT wallet. This constitutes "corruption or spoofing of WalletConnect state that causes a user to approve the wrong asset" and directly results in unauthorized transfer of CAT/XCH.

---

### Likelihood Explanation

Any dapp that the user has paired via WalletConnect and granted `chia_sendTransaction` can trigger this without any additional privilege. The user must click "Confirm" — but the confirmation dialog provides no wallet-identity information, making the deception invisible to a non-technical user. The attack requires only a malicious or compromised dapp, which is the standard WalletConnect threat model.

---

### Recommendation

1. **Remove `hide: true` from `wallet_id`** in `chia_sendTransaction` (and all other spending commands) so the wallet name/type is prominently displayed in the confirmation dialog.
2. **Validate `wallet_id` against the paired fingerprint's actual wallets** inside `dispatchPairRequest` or the main-process handler, rejecting any `wallet_id` that does not belong to the authenticated fingerprint's wallet list.
3. **Replace the hardcoded `defaults: { wallet_id: 1 }`** with a mechanism that resolves the correct wallet at runtime (e.g., from the active wallet context), mirroring the external report's recommendation to pass the context-specific identifier at construction time rather than hardcoding it.

---

### Proof of Concept

1. User pairs a dapp via WalletConnect and grants `chia_sendTransaction`.
2. Dapp sends the WalletConnect session request:
   ```json
   {
     "method": "chia_sendTransaction",
     "params": {
       "wallet_id": 3,
       "amount": "1000000000000",
       "fee": "0",
       "address": "xch1attacker..."
     }
   }
   ```
3. `parseDappParams` accepts `wallet_id: 3` (it is in the params list for `chia_sendTransaction`).
4. The confirmation dialog renders: **Amount**, **Fee**, **Address** — `wallet_id` is suppressed by `hide: true`.
5. User clicks **Send**, believing they are spending XCH from their standard wallet.
6. The Chia daemon executes `send_transaction` against wallet #3 (a CAT wallet), transferring CATs to the attacker's address. [8](#0-7) [9](#0-8)

export async function dispatchPairRequest<T>(
  topic: string,
  command: string,
  params: Record<string, unknown>,
  process: (context: DispatchPairRequestContext) => Promise<T>,
  confirm: () => Promise<boolean>,
): Promise<T> {
  const [loggedInFingerprint, isMainnetValue] = await Promise.all([getLoggedInFingerprint(), isMainnet()]);

  const pair = findPair(topic);
  if (!pair) {
    throw new WcError(`Pair not found`, WcErrorCode.USER_REJECTED);
  }

  // verify if the command is allowed for this pair
  if (!pair.commands.includes(command)) {
    throw new WcError(`Command not allowed for this pair.`, WcErrorCode.UNAUTHORIZED_METHOD);
  }

  const { fingerprint } = params;

  // verify if the network is the same as the pair's network
  if (isMainnetValue !== pair.mainnet) {
    throw new WcError(`Network mismatch`, WcErrorCode.UNSUPPORTED_CHAINS);
  }

  // verify if the requested fingerprint is allowed for this pair
  const requestedFingerprint = fingerprint ?? loggedInFingerprint;
  if (typeof requestedFingerprint !== 'number' || !requestedFingerprint || requestedFingerprint !== pair.fingerprint) {
    throw new WcError(`Fingerprint not allowed for this command`, WcErrorCode.UNAUTHORIZED_METHOD);
  }

  const context = {
    pair,
    fingerprint: requestedFingerprint,
  };

  // Dapps may not switch the active key for an existing pair.
  if (fingerprint !== undefined && fingerprint !== loggedInFingerprint) {
    throw new WcError(`Fingerprint not allowed for this command`, WcErrorCode.UNAUTHORIZED_METHOD);
  }

  // if command is bypassed return true
  if (pair.bypass.includes(command)) {
    return process(context);
  }

  const isAllowed = await confirm();
  if (isAllowed === true) {
    return process(context);
  }

  throw new WcError(`Command not allowed for this pair.`, WcErrorCode.UNAUTHORIZED_METHOD);
