### Title
`EVM.mustRun()` Silently Accepts Failed EVM Transactions, Enabling Cross-VM Asset Loss — (`File: fvm/evm/stdlib/contract.cdc`)

---

### Summary

`EVM.mustRun()` in the production `EVM` Cadence contract is explicitly designed to allow EVM transactions with `Status.failed` to pass through without reverting the enclosing Cadence transaction. Any Cadence transaction that uses `EVM.mustRun()` (or `EVM.run()`) to execute an EVM token transfer — without separately asserting `Status.successful` — will silently commit Cadence-side state changes (e.g., vault withdrawals, escrow) even when the EVM transfer reverted. This is the direct Flow analog of calling a bare ERC20 `transfer()` and ignoring its `bool` return value.

---

### Finding Description

`EVM.mustRun()` is defined in `fvm/evm/stdlib/contract.cdc`:

```cadence
/// mustRun runs the transaction using EVM.run
/// It will rollback if the tx execution status is unknown or invalid.
/// Note that this method does not rollback if transaction
/// is executed but an vm error is reported as the outcome
/// of the execution (status: failed).
access(all)
fun mustRun(tx: [UInt8], coinbase: EVMAddress): Result {
    let runResult = self.run(tx: tx, coinbase: coinbase)
    assert(
        runResult.status == Status.failed || runResult.status == Status.successful,
        message: "EVM.mustRun(): The provided transaction is not valid for execution"
    )
    return runResult
}
``` [1](#0-0) 

The `assert` condition passes for **both** `Status.failed` and `Status.successful`. A `Status.failed` result means the EVM transaction was included in a block but the VM reported an error (e.g., ERC20 `transfer()` reverted, out-of-gas, contract rejected the call). The Cadence transaction is **not** rolled back in this case.

The `EVM.Status` enum makes this explicit:

```cadence
/// Returned when execution of an evm transaction/call
/// has been successful but the vm has reported an error in
/// the outcome of execution (e.g. running out of gas).
/// A failed tx/call is included in a block.
access(all) case failed
``` [2](#0-1) 

The same issue applies to the bare `EVM.run()` function, which returns a `Result` with no status enforcement at all: [3](#0-2) 

---

### Impact Explanation

A Cadence transaction that:
1. Withdraws tokens from a Cadence `FungibleToken.Vault` (irreversible within the transaction)
2. Calls `EVM.mustRun()` to mint or transfer the equivalent EVM tokens (e.g., via an ERC20 `transfer` or bridge mint)
3. Receives `Status.failed` from the EVM layer (ERC20 reverted, out-of-gas, recipient contract rejected)
4. Does **not** assert `runResult.status == EVM.Status.successful`

…will commit the Cadence vault withdrawal while the EVM credit never materialises. The tokens are permanently lost from the Cadence side with no corresponding EVM balance increase. This is a **cross-VM asset loss** with no recovery path.

The same pattern applies to `CadenceOwnedAccount.call()`, which also returns a `Result` whose `status` the caller must check: [4](#0-3) 

---

### Likelihood Explanation

- `EVM.mustRun()` and `EVM.run()` are `access(all)` functions callable by any unprivileged Cadence transaction or contract author.
- The misleading name `mustRun` implies the EVM transaction is guaranteed to succeed, making it likely that callers omit a `Status.successful` assertion.
- The `flow-evm-bridge` integration (bootstrapped via `fvm/blueprints/bridge.go`) relies on Cadence transactions that call into EVM for token minting/burning; if any such transaction uses `mustRun` without a success check, a reverted EVM call silently drains the Cadence escrow.
- The benchmark load script in `integration/benchmark/load/evm_load.go` already demonstrates the pattern of calling `EVM.run()` and **discarding the result entirely**, confirming the pattern is realistic: [5](#0-4) 

---

### Recommendation

1. **`EVM.mustRun()` should assert `Status.successful`**, not merely `Status.failed || Status.successful`. If the intent is to allow failed EVM transactions, the function should be renamed to avoid implying a success guarantee.

   ```cadence
   fun mustRun(tx: [UInt8], coinbase: EVMAddress): Result {
       let runResult = self.run(tx: tx, coinbase: coinbase)
       assert(
           runResult.status == Status.successful,
           message: "EVM.mustRun(): EVM transaction did not succeed"
       )
       return runResult
   }
   ```

2. **All Cadence transactions that perform token transfers via `EVM.run()` or `EVM.mustRun()` must assert `result.status == EVM.Status.successful`** before committing any Cadence-side state changes (vault withdrawals, escrow deposits, etc.).

3. **Document clearly** that `EVM.run()` and `EVM.mustRun()` do not revert the Cadence transaction on EVM-level failure, so contract authors understand they must check the returned `Result.status`.

---

### Proof of Concept

The following Cadence transaction demonstrates the silent-failure path. It withdraws 1.0 FLOW from the signer's vault, then calls `EVM.mustRun()` with an EVM ERC20 `transfer()` that reverts. The Cadence transaction succeeds and the vault is drained; the EVM transfer never happened.

```cadence
import EVM from 0xEVM
import FungibleToken from 0xFT
import FlowToken from 0xFlowToken

transaction(encodedERC20Transfer: [UInt8], coinbase: [UInt8; 20]) {

    prepare(signer: auth(BorrowValue) &Account) {
        // Step 1: withdraw Cadence tokens (irreversible within this tx)
        let vault = signer.storage
            .borrow<auth(FungibleToken.Withdraw) &FlowToken.Vault>(from: /storage/flowTokenVault)!
        let withdrawn <- vault.withdraw(amount: 1.0)
        // ... deposit withdrawn into escrow or burn ...
        destroy withdrawn

        // Step 2: call mustRun — EVM ERC20 transfer reverts (out-of-gas, revert, etc.)
        let result = EVM.mustRun(
            tx: encodedERC20Transfer,
            coinbase: EVM.EVMAddress(bytes: coinbase)
        )
        // result.status == EVM.Status.failed  →  assert passes, NO Cadence revert
        // Cadence vault is drained; EVM tokens never credited.
    }
}
```

The `assert` at line 846–849 of `contract.cdc` passes because `Status.failed` is explicitly included in the allowed set, so the Cadence transaction commits successfully while the EVM transfer silently failed. [6](#0-5)
