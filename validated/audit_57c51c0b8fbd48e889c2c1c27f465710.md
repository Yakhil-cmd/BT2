### Title
Gateway L2 Gas Price Admission Check Uses Stale Previous-Block Price Instead of EIP-1559 Next-Block Price — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

`StatefulTransactionValidator::validate_resource_bounds` compares a transaction's `l2_gas.max_price_per_unit` against `previous_block.l2_gas_price` — the price that was used **in** the previous block — rather than `previous_block.next_l2_gas_price`, the EIP-1559-derived price that **will be used in the next block** (the one the transaction will actually be sequenced into). When block gas usage deviates from the target, these two values diverge, causing the gateway to either admit transactions that will fail blockifier pre-validation (false admission / mempool pollution) or reject transactions that are perfectly valid for the next block (false rejection).

---

### Finding Description

`validate_resource_bounds` reads the previous block's `BlockInfo` and extracts `strk_gas_prices.l2_gas_price`:

```rust
// TODO(Arni): getnext_l2_gas_price from the block header.
let previous_block_l2_gas_price = self
    .gateway_fixed_block_state_reader
    .get_block_info()
    .await?
    .gas_prices
    .strk_gas_prices
    .l2_gas_price;
self.validate_tx_l2_gas_price_within_threshold(
    executable_tx.resource_bounds(),
    previous_block_l2_gas_price,
)?;
``` [1](#0-0) 

The developer's own TODO acknowledges the correct value is `next_l2_gas_price` from the block header. `BlockHeaderWithoutHash` stores both fields:

```rust
pub l2_gas_price: GasPricePerToken,      // price used IN this block
pub next_l2_gas_price: GasPrice,          // EIP-1559 price for the NEXT block
``` [2](#0-1) 

`GatewayFixedBlockSyncStateClient::get_block_info_from_sync_client` populates `BlockInfo.gas_prices` only from `l2_gas_price.price_in_fri`, never from `next_l2_gas_price`:

```rust
strk_gas_prices: GasPriceVector {
    ...
    l2_gas_price: block_header.l2_gas_price.price_in_fri.try_into()?,
},
``` [3](#0-2) 

`next_l2_gas_price` is computed by `calculate_next_l2_gas_price_for_fin` via EIP-1559 and stored in the block header. It diverges from `l2_gas_price` whenever block gas usage differs from the target:

```rust
let adjusted_price_u256 =
    if gas_used > gas_target { price_u256 + price_change } else { price_u256 - price_change };
``` [4](#0-3) 

The threshold check in `validate_tx_l2_gas_price_within_threshold` then compares the transaction's `max_price_per_unit` against the **wrong** reference price:

```rust
let threshold = (gas_price_threshold_multiplier
    * previous_block_l2_gas_price.get().0)
    .to_integer();
if tx_l2_gas_price.0 < threshold {
    return Err(...GAS_PRICE_TOO_LOW...);
}
``` [5](#0-4) 

**Scenario A — False Admission (next price > current price, i.e., previous block was over-target):**
`next_l2_gas_price > l2_gas_price`. The gateway threshold is computed from the lower `l2_gas_price`, so a transaction with `max_price_per_unit` in the range `[threshold_from_old_price, threshold_from_next_price)` passes the gateway check. When the batcher executes it, `check_fee_bounds` compares against the actual block's gas price (`next_l2_gas_price`) and rejects it with `MaxGasPriceTooLow`. The transaction was admitted to the mempool but will never be sequenced.

**Scenario B — False Rejection (next price < current price, i.e., previous block was under-target):**
`next_l2_gas_price < l2_gas_price`. The gateway threshold is computed from the higher `l2_gas_price`, so a transaction with `max_price_per_unit` in the range `[threshold_from_next_price, threshold_from_old_price)` is rejected by the gateway even though it would pass blockifier's `check_fee_bounds` for the next block.

---

### Impact Explanation

**False Admission (Scenario A):** The gateway accepts and forwards to the mempool transactions that will unconditionally fail blockifier pre-validation. An attacker can craft transactions with `max_price_per_unit` just above the stale threshold to flood the mempool with permanently-unsequenceable transactions, consuming mempool capacity and batcher processing cycles. This matches: *"Mempool/gateway/RPC admission accepts invalid transactions … before sequencing."*

**False Rejection (Scenario B):** Legitimate users whose wallets set `max_price_per_unit` relative to the correct next-block price are denied admission even though their transactions are economically valid. This matches: *"… or rejects valid transactions before sequencing."*

---

### Likelihood Explanation

EIP-1559 price adjustments are continuous and normal. Any block whose gas usage deviates from the target (the common case) produces a `next_l2_gas_price ≠ l2_gas_price`. The divergence grows with the magnitude of the deviation. The bug is always active; no special attacker capability is required — ordinary transaction submission during any period of non-uniform block utilization triggers both scenarios.

---

### Recommendation

Replace the use of `l2_gas_price` with `next_l2_gas_price` from the block header. The `GatewayFixedBlockStateReader` trait and its `GatewayFixedBlockSyncStateClient` implementation should expose `next_l2_gas_price` from `BlockHeaderWithoutHash`, and `validate_resource_bounds` should pass that value to `validate_tx_l2_gas_price_within_threshold`. The existing TODO comment at line 229 already identifies this fix.

---

### Proof of Concept

1. Observe a block where `l2_gas_used > gas_target` (over-target block). The sequencer computes `next_l2_gas_price = P_next > P_current = l2_gas_price`.
2. Submit an `AllResources` invoke transaction with `l2_gas.max_price_per_unit = P_current` (exactly the previous block's price, which satisfies `>= threshold * P_current` with `min_gas_price_percentage = 100`).
3. The gateway calls `validate_tx_l2_gas_price_within_threshold` with `previous_block_l2_gas_price = P_current`. The check passes: `P_current >= 1.0 * P_current`.
4. The transaction enters the mempool.
5. The batcher builds the next block using `P_next > P_current` as the block's L2 gas price.
6. `AccountTransaction::check_fee_bounds` fires `ResourceBoundsError::MaxGasPriceTooLow { resource: L2Gas, max_gas_price: P_current, actual_gas_price: P_next }`.
7. The transaction is rejected during blockifier pre-validation and never sequenced, despite having been admitted by the gateway.
