### Title
Unprivileged Caller Can Front-Run `update_rewards` with `disable_rewards: true` to Suppress Consensus Block Rewards — (File: src/staking/staking.cairo)

---

### Summary

The `update_rewards` function in the Staking contract has no access control and accepts a caller-supplied `disable_rewards: bool` parameter. Because the global `last_reward_block` is written unconditionally before the early-return guard, any unprivileged actor can call `update_rewards(any_valid_staker, disable_rewards: true)` first in a block, consuming the one-call-per-block slot without distributing any rewards. Every subsequent legitimate call for that block reverts with `REWARDS_ALREADY_UPDATED`, and all stakers lose their consensus block rewards for that block.

---

### Finding Description

`update_rewards` is a public entry point with no role check: [1](#0-0) 

The function enforces a single call per block via a global counter: [2](#0-1) 

It then **unconditionally** advances `last_reward_block` before inspecting `disable_rewards`: [3](#0-2) 

When `disable_rewards = true` the function returns immediately, distributing nothing: [4](#0-3) 

Because `last_reward_block` is a single global slot shared across all stakers, the attacker only needs to win the race once per block. Any valid, active staker address with non-zero balance satisfies the preconditions, and all such addresses are publicly enumerable from the `stakers` vector. [5](#0-4) 

---

### Impact Explanation

A single successful front-run silently drops one block's worth of consensus rewards for every staker in the protocol. Repeated across every block, the attacker permanently suppresses all consensus-phase yield accrual. No staker's principal is at risk, but unclaimed yield is frozen for as long as the attack continues. This maps to **High** impact: *Permanent freezing of unclaimed yield* (continuous attack) or at minimum **Medium** impact: *Griefing with damage to users and the protocol* (sporadic attack).

---

### Likelihood Explanation

Starknet's public mempool makes transaction ordering observable. The attacker requires only a valid staker address (publicly readable from `stakers`) and enough gas to submit one transaction per block. No privileged key, bridge access, or third-party compromise is needed. The attack is cheap relative to the yield it suppresses across all stakers.

---

### Recommendation

Restrict `update_rewards` to an authorized caller (e.g., a designated consensus role or the attestation contract). Alternatively, remove the `disable_rewards` parameter entirely and handle the pre-consensus / disabled-rewards path through a separate, access-controlled function so that the one-call-per-block slot cannot be consumed without actually distributing rewards.

---

### Proof of Concept

1. Attacker reads any active staker address `S` from the public `stakers` vector.
2. At block `N`, the attacker submits `update_rewards(S, disable_rewards: true)` with a higher gas price than the legitimate consensus caller.
3. The attacker's transaction executes first: `last_reward_block` is set to `N`; the function returns early; no rewards are distributed.
4. The legitimate `update_rewards(S, disable_rewards: false)` call reverts: `current_block_number (N) > last_reward_block (N)` is false → `REWARDS_ALREADY_UPDATED`.
5. All stakers receive zero block rewards for block `N`.
6. Repeating steps 2–5 every block permanently freezes all consensus reward accrual at negligible cost to the attacker. [6](#0-5)

        fn update_rewards(
            ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
        ) {
            self.general_prerequisites();
            let current_block_number = starknet::get_block_number();
            assert!(
                current_block_number > self.last_reward_block.read(),
                "{}",
                Error::REWARDS_ALREADY_UPDATED,
            );

            // Assert staker exists and active.
            // Staker is considered to exist from the moment of `stake` (when `InternalStakerInfo`
            // struct is created) until the calling to `unstake_action` (when `InternalStakerInfo`
            // struct is deleted).
            // Staker remains active until the intent period begins, i.e. K epochs after
            // `unstake_intent` is called.
            let staker_info = self.internal_staker_info(:staker_address);
            let curr_epoch = self.get_current_epoch();
            assert!(
                self.is_staker_active(:staker_address, epoch_id: curr_epoch),
                "{}",
                Error::INVALID_STAKER,
            );

            let staker_pool_info = self.staker_pool_info.entry(staker_address).as_non_mut();
            let (staker_total_strk_balance, staker_total_btc_balance) = self
                .get_staker_total_strk_btc_balance_at_epoch(
                    :staker_address, :staker_pool_info, epoch_id: curr_epoch,
                );
            // Assert staker has non-zero balance.
            // Staker exists with zero balance for the first K epochs after `stake`, then the stake
            // becomes effective.
            assert!(staker_total_strk_balance.is_non_zero(), "{}", Error::INVALID_STAKER);

            // Update last block rewards.
            self.last_reward_block.write(current_block_number);

            if disable_rewards || self.is_pre_consensus() {
                return;
            }

            // Get current block data and update rewards.
            let reward_supplier_dispatcher = self.reward_supplier_dispatcher.read();
            let (strk_block_rewards, btc_block_rewards) = self
                .calculate_block_rewards(:reward_supplier_dispatcher, :curr_epoch);
            self
                ._update_rewards(
                    :staker_address,
                    strk_total_rewards: strk_block_rewards,
                    btc_total_rewards: btc_block_rewards,
                    strk_total_stake: staker_total_strk_balance,
                    btc_total_stake: staker_total_btc_balance,
                    :staker_info,
                    :staker_pool_info,
                    :reward_supplier_dispatcher,
                    :curr_epoch,
                );
        }
