### Title
Missing Proof-of-Correctness for Participant Shares in CKD Protocol Allows Malicious Participant to Corrupt Derived Confidential Key - (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

The `do_ckd_coordinator` function in the Confidential Key Derivation (CKD) protocol accepts `(big_y, big_c)` share contributions from all participants and sums them without any proof of correctness or consistency check. This is the direct analog of the reported "weak access control" pattern: a security-critical operation is performed without specifying or enforcing the available protection mechanism (share verification). A single malicious participant can send arbitrary group elements, causing the coordinator to output a silently corrupted `CKDOutput` that honest parties will accept and use.

---

### Finding Description

In `do_ckd_coordinator`, the coordinator collects each participant's `(big_y_i, big_c_i)` contribution and accumulates them: [1](#0-0) 

Each participant computes their share in `compute_signature_share` as:
- `big_y_i = y_i * G` (a random blinding term)
- `big_c_i = x_i * H(pk || app_id) + y_i * app_pk` (the masked secret share contribution) [2](#0-1) 

The coordinator sums the Lagrange-weighted contributions and outputs the result directly: [3](#0-2) 

There is **no zero-knowledge proof, commitment check, or consistency verification** on the received `(big_y_i, big_c_i)` values. The coordinator has no way to distinguish a correctly computed share from an arbitrary group element sent by a malicious participant.

Contrast this with the DKG protocol (`src/dkg.rs`), which enforces `verify_proof_of_knowledge` and `validate_received_share` on every participant contribution before accepting it: [4](#0-3) 

Also contrast with the ECDSA signing coordinators, which perform a final `sig.verify(&public_key, &msg_hash)` check that would catch a corrupted sum: [5](#0-4) 

The CKD coordinator has **no equivalent final verification**. The `CKDOutput::unmask` function simply computes `big_c - app_sk * big_y` and returns it: [6](#0-5) 

There is no check that the result equals the expected `msk * H(pk || app_id)`.

---

### Impact Explanation

**High: Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.**

A malicious participant sends arbitrary `(big_y', big_c')` instead of their correctly computed share. The coordinator sums these into `big_Y` and `big_C` and outputs a `CKDOutput` that does not correspond to `msk * H(pk || app_id)`. The honest coordinator and all downstream consumers of the CKD output silently accept this corrupted result. The client calling `unmask(app_sk)` will derive a wrong confidential key with no indication of failure. Every CKD invocation involving that malicious participant is permanently corrupted.

---

### Likelihood Explanation

**High.** Any participant in the CKD protocol can trivially mount this attack by sending arbitrary `blstrs::G1Projective` elements instead of their correct share. No cryptographic capability is required. The attack requires only that the malicious party is a registered participant in the protocol, which is the standard adversary model for threshold protocols. The `recv_from_others` helper confirms the sender identity but performs no content validation: [7](#0-6) 

---

### Recommendation

Add a zero-knowledge proof of correct share formation to the CKD protocol. Each participant should prove in zero-knowledge that their submitted `(big_y_i, big_c_i)` was computed correctly from their committed private share `x_i` and a random `y_i`. Alternatively, add a consistency check at the coordinator: after unmasking, verify that the result lies on the expected curve and satisfies the protocol invariant `big_C - app_sk * big_Y == msk * H(pk || app_id)` (which requires the coordinator to know `app_sk`, consistent with the TEE threat model described in the CKD documentation). At minimum, document that the CKD protocol provides no integrity guarantee against a malicious participant and that callers must enforce participant honesty externally.

---

### Proof of Concept

1. Honest participants P1, P2, P3 run `ckd(...)` with coordinator P1.
2. Malicious participant P2 replaces the `do_ckd_participant` send with an arbitrary pair `(G, G)` (the generator point) instead of their correct `(norm_big_y, norm_big_c)`.
3. P1 (coordinator) receives P2's corrupted share and P3's honest share, sums all three contributions including its own.
4. P1 outputs `CKDOutput { big_y: big_Y_corrupted, big_c: big_C_corrupted }`.
5. The client calls `ckd_output.unmask(app_sk)` and obtains a value that is not equal to `msk * H(pk || app_id)`.
6. No error is raised anywhere in the protocol. The corrupted key is silently accepted. [8](#0-7)

        compute_signature_share(&participants, me, key_pair, app_id, app_pk, rng)?;

pub async fn recv_from_others<T>(
    chan: &SharedChannel,
    waitpoint: u64,
    participants: &ParticipantList,
    me: Participant,
) -> Result<Vec<(Participant, T)>, ProtocolError>
where
    T: serde::de::DeserializeOwned,
{
    let mut seen = ParticipantCounter::new(participants);
    seen.put(me);
    let mut messages = Vec::with_capacity(participants.others(me).count());

    while !seen.full() {
        let (from, msg) = chan.recv(waitpoint).await?;
        if seen.put(from) {
            messages.push((from, msg));
        }
    }

    Ok(messages)
