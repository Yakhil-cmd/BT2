### Title
Unauthenticated `sender` in Bank Precompile `transfer` Allows Arbitrary Token Theft - (File: `x/cronos/keeper/precompiles/bank.go`)

### Summary
The `transfer` case in `BankContract.Run()` reads the `sender` address from ABI-decoded calldata (`args[0]`) rather than from `contract.Caller()`. No check enforces that the caller is the token holder. Any EVM contract can therefore drain `evm/<callerContract>` tokens from any victim address that holds them, without the victim's authorization.

### Finding Description
In `x/cronos/keeper/precompiles/bank.go`, the `TransferMethodName` branch unpacks `sender` directly from the caller-supplied ABI arguments:

```go
// line 175-192
sender := args[0].(common.Address)   // ← attacker-controlled calldata
recipient := args[1].(common.Address)
amount := args[2].(*big.Int)
...
from := sdk.AccAddress(sender.Bytes())
to   := sdk.AccAddress(recipient.Bytes())
...
denom := EVMDenom(contract.Caller())  // evm/<callerContract>
...
bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt))
```

`contract.Caller()` is used only to derive the denom; it is never compared against `sender`. The bank module's `SendCoins` then moves `evm/<callerContract>` tokens from the victim (`from`) to any recipient, with no signature or allowance check.

The analogous bug in the external report is the same class: a function argument that is supposed to identify the acting party is silently ignored/overridden, so the actual caller has no bearing on whose assets are moved.

### Impact Explanation
**Critical — Unauthorized transfer of precompile-controlled assets.**

Any EVM contract can call `bankPrecompile.transfer(victim, attacker, fullBalance)`. The denom is `evm/<attackerContract>`, so the stolen tokens are those the attacker's contract previously minted to victims. This is a complete, permissionless drain of every holder's balance of that token, executable in a single transaction by an unprivileged account.

### Likelihood Explanation
The bank precompile is a live, registered precompile reachable by any EVM contract. No special role, key, or governance action is required. An attacker only needs to deploy a contract and call the precompile directly. The attack is deterministic and requires no race condition.

### Recommendation
Replace the calldata-supplied `sender` with the authenticated caller:

```go
// Replace:
sender := args[0].(common.Address)
// With:
sender := contract.Caller()
```

If the intent is to allow a contract to transfer tokens on behalf of a third party, an explicit allowance/approval mechanism must be added and checked before `SendCoins` is called.

### Proof of Concept

1. Attacker deploys `DrainContract` which:
   - Calls `bankPrecompile.mint(victim, 1000)` → victim now holds 1000 `evm/DrainContract` tokens.
   - Calls `bankPrecompile.transfer(victim, attacker, 1000)` → bank precompile reads `sender = victim` from calldata, skips any caller check, and executes `SendCoins(victim → attacker, 1000 evm/DrainContract)`.
2. Victim's balance is zero; attacker holds 1000 tokens. No approval from victim was ever required.
