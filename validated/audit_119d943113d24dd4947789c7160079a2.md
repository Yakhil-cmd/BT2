### Title
Missing Caller Validation in `update_rewards` Allows Anyone to Trigger Unauthorized Reward Distribution — (`File: src/staking/staking.cairo`)

---

### Summary

The `update_rewards` function in the Staking contract is documented to be callable **only by the Starknet sequencer**, but the implementation contains **no caller check**. Any unprivileged address can invoke it, bypassing the intended access control. This allows an attacker (including a staker acting in their own interest) to trigger reward distribution for blocks where the staker did not perform their duties, constituting theft of unclaimed yield from the protocol's reward supply.

---

### Finding Description

The spec explicitly states:

> **Access control**: Only starkware sequencer. [1](#0-0) 

The sequencer is the only party that knows whether a staker correctly performed their block duties. It calls `update_rewards` with `disable_rewards: false` to grant rewards, or `disable_rewards: true` to mark the block as processed without distributing rewards (i.e., the staker missed the block).

The actual implementation, however, performs no such caller check:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
    let current_block_number = starknet::get_block_number();
    assert!(
        current_block_number > self.last_reward_block.read(),
        "{}",
        Error::REWARDS_ALREADY_UPDATED,
    );
    // ← No assert!(get_caller_address() == sequencer, ...) here
``` [2](#0-1) 

The only guards are:
1. Contract is not paused (`general_prerequisites`)
2. The block number is strictly greater than `last_reward_block` (one call per block)
3. The `staker_address` argument is an active staker with non-zero balance [3](#0-2) 

When `disable_rewards = false` and consensus rewards are active, the function calls `_update_rewards`, which:
- Calls `reward_supplier_dispatcher.update_unclaimed_rewards_from_staking_contract(rewards: ...)` to register the reward obligation
- Calls `claim_from_reward_supplier` to pull tokens from the reward supplier into the staking contract
- Transfers pool rewards to pool contracts immediately
- Credits `staker_info.unclaimed_rewards_own` for the staker [4](#0-3) 

---

### Impact Explanation

**High — Theft of unclaimed yield.**

A staker (or any address acting on their behalf) can call `update_rewards(staker_address, disable_rewards: false)` for any block, regardless of whether the staker actually performed their duties. This:

1. Drains the `RewardSupplier`'s `unclaimed_rewards` balance by the block reward amount.
2. Immediately transfers pool rewards (STRK tokens) from the reward supplier to pool contracts.
3. Credits the staker's `unclaimed_rewards_own` with rewards they did not earn.

The reward supplier's token balance is finite (backed by L1 minting). Repeated unauthorized calls inflate the claimed rewards beyond what was legitimately earned, constituting direct theft of yield from the protocol.

Additionally, an attacker can call with `disable_rewards: true` to consume the per-block slot before the sequencer, permanently denying a staker their legitimate rewards for that block (permanent freezing of unclaimed yield for that block).

---

### Likelihood Explanation

**High.** The function is `#[abi(embed_v0)]` and fully public. Any address can call it. A staker has a direct financial incentive to call it themselves for every block, whether or not they attested. The only constraint is one call per block number, which does not prevent abuse — it just means the attacker must call once per block. [5](#0-4) 

---

### Recommendation

Add a caller check at the top of `update_rewards` to assert that `get_caller_address()` equals the registered sequencer address (stored in contract storage). This mirrors the pattern already used for `update_rewards_from_attestation_contract`, which correctly asserts `CALLER_IS_NOT_ATTESTATION_CONTRACT`. [2](#0-1) 

---

### Proof of Concept

1. Consensus rewards are active (after `consensus_rewards_first_epoch` has passed).
2. A staker `S` is registered and active with non-zero balance.
3. In block `N`, staker `S` does **not** attest (the sequencer would call `update_rewards(S, disable_rewards: true)`).
4. Before the sequencer acts, staker `S` (or any address) calls:
   ```
   staking.update_rewards(staker_address: S, disable_rewards: false)
   ```
5. The call succeeds — no caller check exists.
6. `last_reward_block` is set to block `N`.
7. Block rewards are calculated and distributed: the reward supplier transfers tokens to the staking contract, pool rewards are sent to pool contracts, and `S.unclaimed_rewards_own` is incremented.
8. The sequencer's subsequent call for block `N` reverts with `REWARDS_ALREADY_UPDATED`.
9. Staker `S` claims rewards via `claim_rewards(S)` and receives tokens for a block they did not earn. [6](#0-5) [4](#0-3)
