### Title
ExitToEthereum Precompile Burns ERC-20/ETH Without a Refund Callback When Downstream `withdraw` Fails - (File: `engine-precompiles/src/native.rs`)

---

### Summary

The `ExitToEthereum` precompile burns ERC-20 tokens (or deducts ETH) on Aurora before scheduling a `withdraw` promise on the eth-connector/NEP-141 contract. Unlike `ExitToNear`, which has a refund callback (`exit_to_near_precompile_callback`) to restore tokens if the downstream call fails, `ExitToEthereum` schedules only a bare `PromiseArgs::Create` with no error-handling callback. If the eth-connector's `withdraw` is paused or fails for any reason, the already-burned tokens are permanently unrecoverable.

---

### Finding Description

**Root cause — `ExitToEthereum` emits no callback promise:**

In `engine-precompiles/src/native.rs`, the `ExitToEthereum::run()` method constructs a single `PromiseArgs::Create` with no attached callback:

```rust
let withdraw_promise = PromiseCreateArgs { ... method: "withdraw" ... };
let promise = borsh::to_vec(&PromiseArgs::Create(withdraw_promise)).unwrap();
``` [1](#0-0) 

This promise is emitted as a log and later scheduled by `filter_promises_from_logs` in `engine/src/engine.rs`. If the downstream `withdraw` call reverts or is paused, NEAR simply drops the failed receipt — there is no callback to restore the burned tokens. [2](#0-1) 

**Contrast with `ExitToNear`, which has a refund callback:**

`ExitToNear::run()` wraps the transfer promise in a `PromiseArgs::Callback` that calls `exit_to_near_precompile_callback` on failure (when the `error_refund` feature is enabled):

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs {
        base: transfer_promise,
        callback: PromiseCreateArgs {
            method: "exit_to_near_precompile_callback".to_string(), ...
        },
    })
};
``` [3](#0-2) 

The callback in `engine/src/contract_methods/connector.rs` calls `engine::refund_on_error` to re-mint the burned tokens: [4](#0-3) 

No equivalent exists for `ExitToEthereum`.

**The two-step burn-then-withdraw flow:**

`EvmErc20.withdrawToEthereum()` (and `EvmErc20V2.withdrawToEthereum()`) burns tokens first, then calls the precompile:

```solidity
function withdrawToEthereum(address recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);
    // ... call precompile 0xb0bd02f6...
}
``` [5](#0-4) 

The burn is irreversible within the EVM transaction. If the subsequent NEAR promise fails, there is no on-chain path to restore the burned balance.

**The privilege asymmetry that creates the exposure window:**

The eth-connector exposes a `pa_pause_feature("engine_withdraw")` method callable by the eth-connector owner — a role entirely separate from the Aurora engine's `EngineAuthorizer` ACL that controls `pause_precompiles`. The test suite confirms this independent pausability: [6](#0-5) 

The `pause_precompiles` function on the Aurora engine side requires membership in the `EngineAuthorizer` ACL: [7](#0-6) 

These are two independent administrative domains. The eth-connector owner can pause `engine_withdraw` without the Aurora engine's ACL simultaneously pausing `ExitToEthereum`. During that window, any EVM user can call `withdrawToEthereum()`, burning tokens that will never be released.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

For ERC-20 tokens: the ERC-20 representation is burned on Aurora; the underlying NEP-141 tokens remain locked in Aurora's custody (since the `withdraw` call on the NEP-141 contract failed and no tokens were transferred out). The user loses the ERC-20 balance and cannot reclaim the NEP-141 tokens because the ERC-20 key mapping is gone.

For base ETH: the EVM ETH balance is deducted; the NEP-141 ETH remains in the eth-connector. The user loses ETH with no recourse.

---

### Likelihood Explanation

**Medium.** The eth-connector is a separately deployed contract (`aurora-eth-connector`) with its own owner and pause controls. Any legitimate emergency pause of `engine_withdraw` on the eth-connector side (e.g., a Ethereum-side bridge incident) that is not immediately mirrored by pausing `ExitToEthereum` on the Aurora engine creates the vulnerable window. The entry path requires no special privilege — any EVM token holder can call `withdrawToEthereum()` on a deployed `EvmErc20` contract.

---

### Recommendation

Add a refund callback to `ExitToEthereum` analogous to `exit_to_near_precompile_callback`. The callback should:
1. Check the promise result.
2. On failure, re-mint the burned ERC-20 tokens (or restore the ETH balance) to the original caller.

This mirrors the existing `error_refund` pattern already implemented for `ExitToNear`. [8](#0-7) 

Additionally, consider a shared administrative interface so that pausing the eth-connector's `engine_withdraw` automatically (or atomically) pauses the `ExitToEthereum` precompile on the Aurora engine side.

---

### Proof of Concept

1. The eth-connector owner calls `pa_pause_feature({"key": "engine_withdraw"})` on the eth-connector contract.
2. An EVM user calls `EvmErc20.withdrawToEthereum(recipient, 1000)` on Aurora.
3. `_burn(msg.sender, 1000)` executes — 1000 ERC-20 tokens are destroyed from the user's balance.
4. The `ExitToEthereum` precompile (`0xb0bd02f6...`) is invoked and emits a `PromiseArgs::Create` log targeting the eth-connector's `withdraw` method.
5. `filter_promises_from_logs` schedules the promise.
6. The NEAR runtime executes the `withdraw` call on the eth-connector; it reverts with `"Pausable: Method is paused"`.
7. No callback is scheduled; the failed receipt is discarded.
8. The user's 1000 ERC-20 tokens are permanently gone. No ETH is released on Ethereum. The underlying NEP-141 ETH remains locked in the eth-connector with no ERC-20 claim against it.

```rust
pub fn pause_precompiles<I: IO + Copy, E: Env>(io: I, env: &E) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        require_running(&state::get_state(&io)?)?;
        let authorizer: EngineAuthorizer = engine::get_authorizer(&io);

        if !authorizer.is_authorized(&env.predecessor_account_id()) {
            return Err(b"ERR_UNAUTHORIZED".into());
        }

        let args: PausePrecompilesCallArgs = io.read_input_borsh()?;
        let flags = PrecompileFlags::from_bits_truncate(args.paused_mask);
        let mut pauser = EnginePrecompilesPauser::from_io(io);
        pauser.pause_precompiles(flags);
        Ok(())
    })
}
