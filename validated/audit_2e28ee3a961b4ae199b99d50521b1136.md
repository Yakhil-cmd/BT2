### Title
Missing Caller Authorization in `update_rewards` Allows Anyone to Permanently Skip Block Reward Distribution - (File: `src/staking/staking.cairo`)

### Summary
`IStakingRewardsManager::update_rewards` is documented as callable only by the Starkware sequencer, but the implementation contains no caller check. Any unprivileged address can call it with `disable_rewards: true`, which writes `last_reward_block` to the current block number and then returns without distributing rewards. Because the block number is consumed, the legitimate sequencer call for that same block will revert with `REWARDS_ALREADY_UPDATED`, permanently skipping reward distribution for that block.

### Finding Description
The spec at `docs/spec.md` lines 1644–1645 states:

> **access control**: Only starkware sequencer.

The implementation at `src/staking/staking.cairo` lines 1449–1507 is:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
    let current_block_number = starknet::get_block_number();
    assert!(
        current_block_number > self.last_reward_block.read(),
        "{}",
        Error::REWARDS_ALREADY_UPDATED,
    );
    // ... staker existence checks ...
    self.last_reward_block.write(current_block_number);   // ← written unconditionally

    if disable_rewards || self.is_pre_consensus() {
        return;                                            // ← exits without distributing
    }
    // reward distribution follows
```

`general_prerequisites()` only checks the pause flag — it does **not** check `get_caller_address()`. There is no `assert_caller_is_sequencer` or equivalent guard anywhere in the function. [1](#0-0) 

The `last_reward_block` storage slot is updated at line 1485 **before** the `disable_rewards` branch at line 1487. Once written, no code path can rewind it for the same block number. [2](#0-1) 

The spec's access-control requirement is stated but never enforced: [3](#0-2) 

### Impact Explanation
**High — Permanent freezing of unclaimed yield.**

For every block in which consensus rewards are active, an attacker can call `update_rewards(any_active_staker, disable_rewards: true)` before the sequencer does. This:
1. Marks the block as processed (`last_reward_block = current_block`).
2. Returns immediately without computing or crediting any rewards.
3. Causes the sequencer's subsequent call for the same block to revert with `REWARDS_ALREADY_UPDATED`.

The rewards that should have been distributed for that block are permanently lost — they are never added to `unclaimed_rewards_own` for the staker, never forwarded to delegation pools, and never requested from the reward supplier. There is no recovery path.

### Likelihood Explanation
**High.** The entry point is fully public, requires no tokens, no stake, and no privileged key. The attacker only needs to submit a transaction before the sequencer in the same block. On Starknet, transaction ordering within a block is controlled by the sequencer, but the function is callable by any externally-owned account. A persistent attacker can target every block, causing sustained, protocol-wide reward denial for all stakers and delegators.

### Recommendation
Add an explicit caller check mirroring the pattern used for `update_rewards_from_attestation_contract`:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
    self.assert_caller_is_sequencer(); // add this guard
    // ...
```

Alternatively, restrict the interface so that `disable_rewards` can only be set by a trusted role, and allow any caller only when `disable_rewards: false`.

### Proof of Concept

1. Consensus rewards are active (`is_pre_consensus()` returns `false`).
2. Attacker (any EOA) observes the current block number `B`.
3. Attacker calls `update_rewards(victim_staker, disable_rewards: true)`.
   - `current_block_number > last_reward_block` → passes.
   - Staker existence and balance checks → pass (any active staker address works).
   - `last_reward_block.write(B)` executes.
   - `disable_rewards == true` → function returns without distributing rewards.
4. Sequencer calls `update_rewards(victim_staker, disable_rewards: false)` for block `B`.
   - `current_block_number > last_reward_block` → **fails** (`B > B` is false).
   - Reverts with `REWARDS_ALREADY_UPDATED`.
5. Block `B` rewards are permanently lost for all stakers and pool members. [4](#0-3) [2](#0-1)
