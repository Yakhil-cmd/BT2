### Title
Unvalidated Zero Entropy in `RerandomizationArguments` Defeats Presignature Rerandomization Security - (File: src/ecdsa/mod.rs)

### Summary
The `RerandomizationArguments` struct accepts a caller-supplied `entropy: [u8; 32]` field that is documented as "Fresh, Unpredictable, and Public source of entropy" but is never validated to be non-trivial (non-zero). A malicious coordinator can supply `entropy = [0u8; 32]`, making the HKDF-derived rerandomization scalar `delta` fully deterministic and computable from public inputs alone. This defeats the entire purpose of presignature rerandomization — preventing Wagner-style attacks — because the coordinator can precompute `delta` for any `(msg_hash, big_r)` pair before the signing session begins.

### Finding Description

In `src/ecdsa/mod.rs`, `RerandomizationArguments::new()` accepts `entropy` without any validation:

```rust
pub fn new(
    pk: AffinePoint,
    tweak: Tweak,
    msg_hash: [u8; 32],
    big_r: AffinePoint,
    participants: ParticipantList,
    entropy: [u8; 32],   // ← never checked for zero or non-triviality
) -> Self { ... }
```

The `derive_randomness()` method uses `entropy` as the IKM for HKDF:

```rust
let hk = Hkdf::<sha3::Sha3_256>::new(Some(&Self::SALT), &self.entropy);
```

The only check performed is that the resulting `delta` is non-zero:

```rust
while bool::from(delta.is_zero()) { ... }
```

There is no check that `self.entropy != [0u8; 32]`. All other inputs to the HKDF (`pk`, `tweak`, `msg_hash`, `big_r`, `participants`) are public. Therefore, if `entropy = [0u8; 32]`, the output `delta` is entirely determined by public values and is fully precomputable by any observer — including the coordinator — before the signing session starts.

Both `ot_based_ecdsa::RerandomizedPresignOutput::rerandomize_presign()` (`src/ecdsa/ot_based_ecdsa/mod.rs`) and `robust_ecdsa::RerandomizedPresignOutput::rerandomize_presign()` (`src/ecdsa/robust_ecdsa/mod.rs`) call `args.derive_randomness()` and only check `delta.is_zero()`, never inspecting `entropy`.

The test utility `src/test_utils/test_generators.rs` line 181 explicitly uses `let entropy = [0u8; 32]` in `make_signature`, confirming the library accepts and processes zero entropy without error.

**Attack path:**

1. Malicious coordinator constructs `RerandomizationArguments::new(pk, tweak, msg_hash, big_r, participants, [0u8; 32])`.
2. For any presignature `big_r` and any candidate `msg_hash`, the coordinator precomputes `delta = HKDF([0u8;32], pk || tweak || msg_hash || big_r || participants)`.
3. The coordinator precomputes `rerandomized_R = delta * big_r` for all candidate `(msg_hash, big_r)` pairs.
4. The coordinator selects `(msg_hash, big_r)` pairs whose rerandomized R values satisfy the Wagner generalized birthday condition.
5. Across multiple signing sessions using those selected pairs, the coordinator recovers the aggregate secret key via the Wagner attack.

### Impact Explanation

The rerandomization step exists specifically to prevent Wagner-style attacks on threshold ECDSA presignatures, as cited in the codebase's own references to [GS21]. The security guarantee requires that `delta` be unpredictable to all parties, including the coordinator. With `entropy = [0u8; 32]`, `delta` is fully determined by public inputs, so the coordinator can precompute it for any `(msg_hash, big_r)` pair. This restores the exact attack surface that rerandomization was designed to close: the coordinator can select signing sessions whose rerandomized nonces satisfy the Wagner birthday condition, ultimately recovering the private signing shares. Impact: **Critical — extraction and reconstruction of private signing shares**.

### Likelihood Explanation

The coordinator role is reachable without privileged assumptions — any participant can be designated coordinator. Supplying `entropy = [0u8; 32]` requires no special capability; it is a valid 32-byte array accepted without error. The Wagner attack requires multiple signing sessions (polynomial in the security parameter under the generalized birthday bound), but the coordinator controls session initiation and can accumulate the required sessions over time. In deployments where the coordinator also controls which messages are signed (e.g., MPC wallet infrastructure), the attack is fully within reach.

### Recommendation

Validate that `entropy` is non-zero in `RerandomizationArguments::new()` or in `derive_randomness()`:

```rust
pub fn new(..., entropy: [u8; 32]) -> Result<Self, ProtocolError> {
    if entropy == [0u8; 32] {
        return Err(ProtocolError::InvalidEntropy);
    }
    Ok(Self { ..., entropy })
}
```

Additionally, consider generating entropy internally using a `CryptoRngCore` rather than accepting it as a caller-supplied parameter, removing the possibility of misuse entirely. If external entropy must be accepted (e.g., for deterministic testing), document clearly that zero entropy is insecure and enforce the check in production paths.

### Proof of Concept

```rust
// Malicious coordinator supplies zero entropy
let entropy = [0u8; 32];  // accepted without error — confirmed by test_generators.rs:181

let rerand_args = RerandomizationArguments::new(
    pk, tweak, msg_hash_bytes, big_r, participants, entropy,
);

// delta is now fully determined by public inputs:
// delta = HKDF(salt=[...], ikm=[0u8;32], info=pk||tweak||msg_hash||big_r||participants)
// The coordinator can precompute this for any (msg_hash, big_r) pair before signing.
let delta = rerand_args.derive_randomness().unwrap(); // succeeds, delta != 0

// rerandomize_presign also succeeds — only checks delta != 0, not entropy != 0
let rerandomized = RerandomizedPresignOutput::rerandomize_presign(&presig, &rerand_args).unwrap();

// Coordinator now knows rerandomized_R = delta * big_r in advance.
// Repeating across chosen (msg_hash, big_r) pairs enables Wagner attack.
```

The `derive_randomness` function at `src/ecdsa/mod.rs:162` uses `self.entropy` as HKDF IKM with no prior check, and `rerandomize_presign` at `src/ecdsa/ot_based_ecdsa/mod.rs:70–75` and `src/ecdsa/robust_ecdsa/mod.rs:59–65` only gate on `delta.is_zero()`, never on `entropy` content.
