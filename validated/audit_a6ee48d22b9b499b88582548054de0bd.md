### Title
Missing Zero-Divisor Check in `i256_mod` Causes Panic on SMOD with Zero Divisor — (`evm_interpreter/src/i256.rs`)

### Summary
`i256_mod` in `evm_interpreter/src/i256.rs` does not guard against a zero divisor before calling `div_rem`, which unconditionally panics on a zero divisor. The EVM specification requires `SMOD(a, 0) = 0`; the ZKsync OS implementation instead panics. Any unprivileged EVM transaction can trigger this by executing `SMOD` with a zero divisor, crashing block processing.

### Finding Description
`i256_div` and `i256_mod` are the two signed-arithmetic helpers used by the EVM interpreter. `i256_div` correctly handles a zero divisor:

```rust
// evm_interpreter/src/i256.rs  line 84-89
pub fn i256_div(dividend: &mut U256, divisor_or_quotient: &mut U256) {
    let divisor_sign = i256_sign::<true>(divisor_or_quotient);
    if divisor_sign == Sign::Zero {          // ← zero-divisor guard
        *divisor_or_quotient = U256::ZERO;
        return;
    }
```

`i256_mod` does **not**. It calls `i256_sign::<true>` on the divisor but discards the result, then unconditionally calls `div_rem`:

```rust
// evm_interpreter/src/i256.rs  line 134-144
pub fn i256_mod(dividend: &mut U256, divisor_or_remainder: &mut U256) {
    let dividend_sign = i256_sign::<true>(dividend);
    if dividend_sign == Sign::Zero {         // only guards dividend == 0
        *divisor_or_remainder = U256::ZERO;
        return;
    }
    let _ = i256_sign::<true>(divisor_or_remainder); // result discarded
    let (_q, r) = dividend.div_rem(*divisor_or_remainder); // ← panics if divisor == 0

The underlying `div_rem` implementation uses a hard `assert!` (not `debug_assert!`), so it panics in both debug and release builds:

```rust

    let is_zero = divisor_or_remainder.0.is_zero_mut();
    assert!(is_zero == false);   // ← unconditional panic

### Impact Explanation
A Rust `assert!` panic in the EVM interpreter is unrecoverable in the RISC-V proving environment (no `catch_unwind`). In the forward-execution (x86) path it aborts the sequencer process. Either way, block processing halts. Because `i256_div` handles the zero case internally (the SDIV opcode handler therefore does not pre-check for zero), the SMOD opcode handler almost certainly follows the same pattern and also does not pre-check for zero before calling `i256_mod`. A single crafted transaction is sufficient to trigger the panic.

### Likelihood Explanation
`SMOD(a, 0)` is a valid, well-specified EVM operation (result = 0). Any unprivileged account can deploy or call a contract that executes `SMOD` with a zero divisor. No special privilege, governance access, or oracle manipulation is required.

### Recommendation
Add the same zero-divisor guard to `i256_mod` that already exists in `i256_div`:

```rust
pub fn i256_mod(dividend: &mut U256, divisor_or_remainder: &mut U256) {
    // Guard: SMOD(a, 0) == 0 per EVM spec
    let divisor_sign = i256_sign::<true>(divisor_or_remainder);
    if divisor_sign == Sign::Zero {
        *divisor_or_remainder = U256::ZERO;
        return;
    }
    let dividend_sign = i256_sign::<true>(dividend);
    if dividend_sign == Sign::Zero {
        *divisor_or_remainder = U256::ZERO;
        return;
    }
    let (_q, r) = dividend.div_rem(*divisor_or_remainder);
    ...
}
```

### Proof of Concept
Deploy the following minimal bytecode on ZKsync OS and call it:

```
PUSH1 0x00   // divisor = 0
PUSH1 0x01   // dividend = 1
SMOD         // calls i256_mod(1, 0) → div_rem panics
```

Per the EVM specification the result must be `0`. Under ZKsync OS the `assert!(is_zero == false)` inside `div_rem` fires, aborting block processing. No funds, keys, or privileged access are required.

pub fn i256_div(dividend: &mut U256, divisor_or_quotient: &mut U256) {
    let divisor_sign = i256_sign::<true>(divisor_or_quotient);
    if divisor_sign == Sign::Zero {
        *divisor_or_quotient = U256::ZERO;
        return;
    }

    let dividend_sign = i256_sign::<true>(dividend);
    if dividend_sign == Sign::Minus
        && *dividend.as_limbs() == MIN_NEGATIVE_VALUE_REPR
        && *divisor_or_quotient == U256::ONE
    {
        // it's signed division overflow
        *divisor_or_quotient = U256::ZERO;
        unsafe {
            divisor_or_quotient.as_limbs_mut()[3] = 0x80000000_00000000;
        }
        two_compl_mut(divisor_or_quotient);
        return;
    }

    // this is unsigned division of moduli
    let (quotient, _) = dividend.div_rem(*divisor_or_quotient);

    // set sign bit to zero

    if quotient.is_zero() {
        *divisor_or_quotient = U256::ZERO;
    } else {
        match (dividend_sign, divisor_sign) {
            (Sign::Zero, Sign::Plus)
            | (Sign::Plus, Sign::Zero)
            | (Sign::Zero, Sign::Zero)
            | (Sign::Plus, Sign::Plus)
            | (Sign::Minus, Sign::Minus) => {
                // no extra manipulation required
                Clone::clone_from(divisor_or_quotient, &quotient);
            }
            (Sign::Zero, Sign::Minus)
            | (Sign::Plus, Sign::Minus)
            | (Sign::Minus, Sign::Zero)
            | (Sign::Minus, Sign::Plus) => {
                let (res, _) = U256::ZERO.overflowing_sub(quotient);
                *divisor_or_quotient = res;
            }
        }
    }
}
