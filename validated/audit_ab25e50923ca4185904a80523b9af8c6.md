### Title
Malicious Participant Sends Malformed Message to Abort DKG/Signing via Unhandled Deserialization Failure in `recv_from_others` - (File: src/protocol/helpers.rs)

---

### Summary

`recv_from_others` in `src/protocol/helpers.rs` propagates deserialization errors from `chan.recv` directly to the caller via `?`. A malicious participant can inject a message with a valid routing header but a garbage payload at the correct waitpoint, causing `chan.recv` to return a `DeserializationError`, which immediately aborts the DKG, reshare, refresh, or signing protocol for every honest party that calls this helper. The echo-broadcast layer already handles this correctly by ignoring deserialization failures; `recv_from_others` does not.

---

### Finding Description

`recv_from_others` collects exactly one message per peer before returning:

```rust
// src/protocol/helpers.rs
while !seen.full() {
    let (from, msg) = chan.recv(waitpoint).await?;   // ← error propagated immediately
    if seen.put(from) {
        messages.push((from, msg));
    }
}
```

`chan.recv` (via `Comms::recv` in `src/protocol/internal.rs`) deserializes the raw bytes after stripping the routing header:

```rust
// src/protocol/internal.rs  (Comms::recv)
let decoded: Result<T, Box<dyn error::Error + Send + Sync>> =
    rmp_serde::decode::from_slice(message_data).map_err(std::convert::Into::into);
Ok((from, decoded?))          // ← returns Err on bad msgpack
```

`push_message` (the ingress path) accepts any message whose first `MessageHeader::LEN` bytes parse as a valid header, without validating the payload:

```rust
// src/protocol/internal.rs  (Comms::push_message)
fn push_message(&self, from: Participant, message: MessageData) {
    if message.len() < MessageHeader::LEN { return; }
    let Some(header) = MessageHeader::from_bytes(&message) else { return; };
    self.incoming.push(header, from, message);   // payload never validated
}
```

The routing header (`ChannelTag` + `Waitpoint`) is fully deterministic: the shared-channel root tag is derived from a public constant (`NEAR_CHANNEL_TAGS_DOMAIN` + `"root shared"`), and waitpoints are incremented sequentially. Any participant who knows the protocol can compute the exact header bytes for any round.

A malicious participant therefore:
1. Computes the correct `(channel_tag, waitpoint)` for the target round.
2. Appends arbitrary garbage bytes as the payload.
3. Delivers this message via the normal `Protocol::message` interface.

When `recv_from_others` dequeues this message, `chan.recv` fails to deserialize the payload and returns `Err(ProtocolError::DeserializationError(...))`. The `?` in `recv_from_others` propagates this error to the caller, aborting the entire protocol for the honest party.

`recv_from_others` is called in `do_keyshare` (the shared DKG/reshare/refresh core) at two critical points:

- Round 1 — collecting commitment hashes from all peers
- Round 5 — collecting secret signing shares from all peers

```rust
// src/dkg.rs
for (from, their_commitment_hash) in
    recv_from_others(&chan, wait_round_1, &participants, me).await?   // ← aborts here
{ ... }

for (from, signing_share_from) in
    recv_from_others(&chan, wait_round_3, &participants, me).await?   // ← or here
{ ... }
```

By contrast, `reliable_broadcast_receive_all` in `src/protocol/echo_broadcast.rs` already handles this correctly — it silently skips any message that fails to deserialize:

```rust
// src/protocol/echo_broadcast.rs
match chan.recv(wait).await {
    Ok(value) => (from, (sid, vote)) = value,
    _ => continue,    // ← malformed messages are ignored
};
```

The inconsistency between the two receive paths is the root cause.

---

### Impact Explanation

Any single malicious participant in the participant set can abort DKG, reshare, refresh, or any signing protocol that relies on `recv_from_others` for every honest party simultaneously. Because the attack can be repeated on every restart attempt, honest parties are permanently denied the ability to complete key generation or signing as long as the malicious participant remains in the set. This matches the allowed impact: **High — Permanent denial of signing, key generation, reshare, or refresh for honest parties under valid protocol inputs and documented trust assumptions**.

---

### Likelihood Explanation

The attack requires only that the adversary be a registered participant (the normal trust assumption for threshold protocols). No cryptographic break, no leaked keys, and no external dependency is needed. The routing header is fully public and deterministic. Sending a single malformed message per round is trivially cheap. Likelihood is **High**.

---

### Recommendation

Mirror the error-handling pattern already used in `reliable_broadcast_receive_all`: skip messages that fail to deserialize rather than propagating the error. For example:

```rust
pub async fn recv_from_others<T>(
    chan: &SharedChannel,
    waitpoint: u64,
    participants: &ParticipantList,
    me: Participant,
) -> Result<Vec<(Participant, T)>, ProtocolError>
where
    T: serde::de::DeserializeOwned,
{
    let mut seen = ParticipantCounter::new(participants);
    seen.put(me);
    let mut messages = Vec::with_capacity(participants.others(me).count());

    while !seen.full() {
        let Ok((from, msg)) = chan.recv::<T>(waitpoint).await else {
            continue;   // skip malformed messages; wait for a valid one
        };
        if seen.put(from) {
            messages.push((from, msg));
        }
    }

    Ok(messages)
}
```

Longer term, consider the pull pattern noted in the external report's recommendation: rather than having each participant push data into a shared receive loop, have the protocol explicitly request data from identified peers so that a failure from one peer can be attributed and isolated without aborting the entire round.

---

### Proof of Concept

**Attack setup**: 3-of-3 DKG with participants `[P0, P1, P2]`. `P2` is malicious.

1. All three participants start `do_keyshare`.
2. In Round 1, `P2` computes the shared-channel root tag (public constant) and the waitpoint for `wait_round_1` (always `0` for the first `next_waitpoint()` call after the broadcast waitpoints).
3. `P2` constructs a message: `header_bytes || 0xFF 0xFF 0xFF` (valid header, invalid msgpack).
4. `P2` delivers this message to `P0` and `P1` via `Protocol::message`.
5. When `P0` calls `recv_from_others(&chan, wait_round_1, ...)`, `chan.recv` dequeues `P2`'s malformed message, `rmp_serde::decode::from_slice` fails, and `recv_from_others` returns `Err(ProtocolError::DeserializationError(...))`.
6. `do_keyshare` propagates the error; `P0`'s DKG instance terminates with failure.
7. `P2` repeats on every restart, permanently denying key generation to `P0` and `P1`.
