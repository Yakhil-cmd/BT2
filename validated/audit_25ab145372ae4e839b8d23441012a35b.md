### Title
Hardcoded 2,300 Gas Stipend in EVM Bridge Deposit/Withdraw/Transfer Calls Breaks Smart Contract Integrations - (File: fvm/evm/types/call.go)

### Summary

Flow EVM's bridge direct-call constructors (`NewDepositCall`, `NewWithdrawCall`, `NewTransferCall`) hardcode a gas limit of `21,000 + 2,300 = 23,300` — mirroring Solidity's deprecated `transfer()` anti-pattern. Any EVM smart contract whose `receive()` or `fallback()` function consumes more than 2,300 gas will permanently fail to receive FLOW tokens via the bridge deposit path, breaking cross-VM integrations.

### Finding Description

In `fvm/evm/types/call.go`, three constants define the gas limit for all native-token bridge operations:

```go
IntrinsicFeeForTokenTransfer    = gethParams.TxGas          // 21_000
DefaultGasLimitForTokenTransfer = IntrinsicFeeForTokenTransfer + 2_300  // 23_300
DepositCallGasLimit             = DefaultGasLimitForTokenTransfer
WithdrawCallGasLimit            = DefaultGasLimitForTokenTransfer
``` [1](#0-0) 

These constants are consumed by all three bridge call constructors:

- `NewDepositCall` sets `GasLimit: DepositCallGasLimit`
- `NewWithdrawCall` sets `GasLimit: WithdrawCallGasLimit`
- `NewTransferCall` sets `GasLimit: DefaultGasLimitForTokenTransfer` [2](#0-1) 

The `Account.Deposit()` method in the handler unconditionally calls `NewDepositCall` with this fixed gas limit: [3](#0-2) 

Similarly, `Account.Withdraw()` and `Account.Transfer()` use the same fixed-gas constructors: [4](#0-3) 

The Cadence-level entry point `EVMAddress.deposit()` in `contract.cdc` calls `InternalEVM.deposit`, which routes to `Account.Deposit()` above — making this reachable from any unprivileged Cadence transaction: [5](#0-4) 

The code's own comment acknowledges the fragility:

> "Note that these gas values might need to change if we change the transaction (e.g. add access list), then it has to be updated to use Intrinsic function to calculate the minimum gas needed to run the transaction." [6](#0-5) 

### Impact Explanation

Any EVM smart contract whose `receive()` or `fallback()` function requires more than 2,300 gas (e.g., it emits events, updates storage, or calls another contract) will always revert when FLOW tokens are deposited to it via the bridge. The existing test suite already documents this failure mode: [7](#0-6) 

This permanently breaks cross-VM integrations for any EVM contract that performs non-trivial logic on token receipt — a common and legitimate pattern (vault contracts, accounting contracts, ERC-4626 wrappers, etc.). The 2,300 gas stipend is the same value Ethereum deprecated via EIP-1884 and subsequent opcode repricing; any future EVM opcode repricing on Flow EVM would similarly break even contracts whose `receive()` functions currently fit within 2,300 gas.

### Likelihood Explanation

The entry path is fully unprivileged: any Cadence transaction can call `EVMAddress.deposit(from: <-vault)` targeting any EVM address. A developer deploying an EVM smart contract with a standard `receive()` function that emits an event (costing ~375+ gas for the LOG opcode alone, on top of the 21,000 intrinsic cost) will find deposits permanently failing. This is a realistic and common contract pattern, making the likelihood high.

### Recommendation

Replace the hardcoded `+ 2_300` stipend with a configurable or dynamically computed gas limit, similar to how `NewDeployCall` and `NewContractCall` accept a caller-supplied `gasLimit` parameter. At minimum, expose a parameter so callers can specify the gas limit for deposit/withdraw/transfer operations, allowing integrations with contracts that have non-trivial receive logic.

### Proof of Concept

1. Deploy an EVM smart contract with a `receive()` function that emits an event:
   ```solidity
   contract Receiver {
       event Received(address sender, uint256 amount);
       receive() external payable {
           emit Received(msg.sender, msg.value); // ~750 gas > 2300 budget after intrinsic
       }
   }
   ```
2. From a Cadence transaction, call:
   ```cadence
   let addr = EVM.EVMAddress(bytes: receiverBytes)
   addr.deposit(from: <-vault)  // triggers NewDepositCall with GasLimit=23_300
   ```
3. The EVM execution reverts with `ErrExecutionReverted` because the `receive()` function exhausts the 2,300 gas sub-stipend, causing `panicOnErrorOrInvalidOrFailedState` to panic and the Cadence transaction to revert — permanently blocking FLOW token deposits to this contract. [8](#0-7) [9](#0-8)

// Deposit deposits the token from the given vault into the flow evm main vault
// and update the account balance with the new amount
func (a *Account) Deposit(v *types.FLOWTokenVault) {
	defer a.fch.backend.StartChildSpan(trace.FVMEVMDeposit).End()

	bridge := a.fch.addressAllocator.NativeTokenBridgeAddress()
	bridgeAccount := a.fch.AccountByAddress(bridge, false)
	// Note: its not an authorized call
	res, err := a.fch.executeAndHandleCall(
		types.NewDepositCall(
			bridge,
			a.address,
			v.Balance(),
			bridgeAccount.Nonce(),
		),
		v.Balance(),
		false,
	)
	panicOnErrorOrInvalidOrFailedState(res, err)
}

// Withdraw deducts the balance from the account and
// withdraw and return flow token from the Flex main vault.
func (a *Account) Withdraw(b types.Balance) *types.FLOWTokenVault {
	defer a.fch.backend.StartChildSpan(trace.FVMEVMWithdraw).End()

	res, err := a.executeAndHandleAuthorizedCall(
		types.NewWithdrawCall(
			a.fch.addressAllocator.NativeTokenBridgeAddress(),
			a.address,
			b,
			a.Nonce(),
		),
		b,
		true,
	)
	panicOnErrorOrInvalidOrFailedState(res, err)

	return types.NewFlowTokenVault(b)
}

			a.Nonce(),

		nil,
		false,
	)

        fun deposit(from: @FlowToken.Vault) {

            }
