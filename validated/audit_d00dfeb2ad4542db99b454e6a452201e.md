### Title
`protocolUnpausePool` Leaves Pool Permanently Swap-Blocked After Protocol Unpause - (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`protocolUnpausePool` always transitions the pool to `pauseLevel = 1` (admin-paused) instead of `pauseLevel = 0` (active). After the protocol owner calls this function to restore a pool, swaps remain blocked indefinitely because the `whenNotPaused` guard rejects any `pauseLevel != 0`. The pool admin must issue a second, independent `unpausePool` call to reach level 0. If the admin is unresponsive, the pool is permanently swap-frozen despite the protocol owner having already "unpaused" it.

---

### Finding Description

The pause state machine is documented as:

```
0 = active
1 = paused by admin
2 = paused by protocol
```

`protocolPausePool` correctly accepts level 0 or 1 as the source and writes level 2: [1](#0-0) 

`protocolUnpausePool` reads the current level, asserts it is 2, then writes **level 1** — not level 0: [2](#0-1) 

Level 1 is the admin-paused state. The `whenNotPaused` modifier on `swap` rejects every call where `pauseLevel != 0`: [3](#0-2) [4](#0-3) 

So after `protocolUnpausePool` executes, every `swap` call still reverts with `PoolPaused`. The only escape is for the pool admin to call `unpausePool`, which transitions 1 → 0: [5](#0-4) 

This is structurally identical to the external seed bug: a state variable (`pauseLevel`) is written to the wrong value by the registration/restore function, leaving the downstream consumer (`swap`) permanently unable to execute — exactly as `_pools` being empty left `distributeEx` permanently unable to distribute.

---

### Impact Explanation

Any pool that was active (level 0) before a protocol pause is left at level 1 after `protocolUnpausePool`. Swaps are completely blocked. `addLiquidity` and `removeLiquidity` are not gated by `whenNotPaused`, so LP principal is not directly frozen, but the core trading function of the pool is rendered unusable. Fee revenue stops accruing. If the pool admin is a slow multisig, a DAO, or simply unresponsive, the pool can remain swap-frozen indefinitely after the protocol owner has already taken the corrective action.

---

### Likelihood Explanation

Every invocation of `protocolUnpausePool` on a pool that was at level 0 before the protocol pause triggers this bug — it is 100% reproducible. The protocol owner is a valid, non-malicious actor exercising a documented admin function. No special setup or attacker is required; the broken transition fires on the normal operational path.

---

### Recommendation

Change `protocolUnpausePool` to write `pauseLevel = 0` instead of `pauseLevel = 1`:

```solidity
function protocolUnpausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 2) revert InvalidPauseTransition(cur, 0);   // target is 0, not 1
    IMetricOmmPoolFactoryActions(pool).setPause(0);         // restore to active
}
```

If the design intent is that a pool that was admin-paused before a protocol pause should remain admin-paused after protocol unpause, the factory must record the pre-protocol-pause level and restore it explicitly. The current code does not do this — it unconditionally writes 1 regardless of prior state.

---

### Proof of Concept

```solidity
// Pool starts active (pauseLevel == 0)
assert(poolPauseLevel(pool) == 0);

// Protocol pauses the pool
factory.protocolPausePool(pool);
assert(poolPauseLevel(pool) == 2);

// Protocol unpauses — intends to restore to active
factory.protocolUnpausePool(pool);
// BUG: pauseLevel is now 1, not 0
assert(poolPauseLevel(pool) == 1);  // still paused!

// Any swap attempt reverts with PoolPaused
vm.expectRevert(MetricOmmPool.PoolPaused.selector);
pool.swap(recipient, true, 1e18, 0, "", "");

// Pool admin must issue a second call to actually restore
factory.unpausePool(pool);          // onlyPoolAdmin
assert(poolPauseLevel(pool) == 0);  // only now active
```
