### Title
`MsgTurnBridge` Handler Is a No-Op — Bridge Circuit Breaker Permanently Broken - (File: `x/cronos/keeper/msg_server.go`)

### Summary

The `TurnBridge` gRPC message handler, which is documented and proto-defined as the mechanism to **disable or enable the Gravity Bridge**, is implemented as a complete no-op. It performs no permission check, writes no state, and always returns `(nil, nil)`. The bridge is therefore permanently enabled and cannot be disabled by any actor, including the Cronos admin.

### Finding Description

The proto service defines `TurnBridge` explicitly as the bridge on/off switch:

```
// TurnBridge defines a method to disable or enable the gravity bridge
rpc TurnBridge(MsgTurnBridge) returns (MsgTurnBridgeResponse);
``` [1](#0-0) 

The message carries an `enable bool` field: [2](#0-1) 

The permission system defines and tracks `CanTurnBridge` as a distinct permission bit, and the query layer correctly reports it: [3](#0-2) [4](#0-3) 

However, the actual message server implementation is:

```go
// TurnBridge implements the grpc method
func (k msgServer) TurnBridge(goCtx context.Context, msg *types.MsgTurnBridge) (*types.MsgTurnBridgeResponse, error) {
    return nil, nil
}
``` [5](#0-4) 

Two defects are present simultaneously:

1. **No permission check** — unlike `UpdateTokenMapping`, which gates on `HasPermission(ctx, msg.GetSigners(), CanChangeTokenMapping)`, `TurnBridge` checks nothing. Any unprivileged address can submit the message and receive a success response.
2. **No state mutation** — no `EnableGravity` / `EnableBridge` flag is written anywhere. A search across the entire codebase for any such flag (`EnableGravity`, `enable_gravity`, `EnableBridge`, `enable_bridge`, `IsBridgeEnabled`) returns zero matches, confirming the bridge has no runtime on/off state at all.

Compare with `UpdateTokenMapping`, which has the correct pattern: [6](#0-5) 

### Impact Explanation

The Gravity Bridge circuit breaker is permanently non-functional. In any emergency — a bridge exploit, a critical bug in the relayer, or a compromised validator set — the Cronos admin or any address holding `CanTurnBridge` permission **cannot disable the bridge**. The call succeeds silently and the bridge continues processing `SendToCosmos` events and batch executions unimpeded.

This is a **High** impact finding matching: *"Bypass of Cronos admin, governance authority, permission, token-mapping, bridge, block-list, or module-account authorization checks"* — the bridge disable authorization mechanism is entirely bypassed because the implementation is absent.

### Likelihood Explanation

The entry path is trivially reachable: any unprivileged user can submit a `MsgTurnBridge` transaction on-chain. The transaction is accepted, returns success, and does nothing. The admin's emergency response capability is silently broken with no on-chain indication of failure.

### Recommendation

Implement `TurnBridge` with:
1. A `CanTurnBridge` permission check (mirroring `UpdateTokenMapping`'s `HasPermission` guard).
2. A persistent boolean flag (e.g., `EnableGravity` in `Params` or a dedicated store key) that is written by `TurnBridge`.
3. A guard in the Gravity Bridge processing path (EVM hooks / IBC handlers) that reads this flag and rejects bridge operations when disabled.

### Proof of Concept

```
# Any unprivileged address can call TurnBridge with enable=false
# The tx succeeds (code 0), but the bridge is never disabled.
cronosd tx cronos turn-bridge false --from attacker

# Admin also calls it — same result: success, no effect.
cronosd tx cronos turn-bridge false --from admin

# Bridge continues processing SendToCosmos events normally.
```
