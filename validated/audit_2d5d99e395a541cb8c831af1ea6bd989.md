### Title
Pool Admin Can Atomically Swap to a Manipulated Price Provider and Drain LP Funds When `priceProviderTimelock == 0` — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

When a pool is created with `priceProviderTimelock = 0`, the pool admin can call `proposePoolPriceProvider` and `executePoolPriceProviderUpdate` in the same transaction. Because the timelock guard evaluates `block.timestamp < block.timestamp` (which is always `false`), the proposal is immediately executable. Combined with the fact that `_validatePriceProvider` only checks token-pair identity and `_getBidAndAskPriceX64` applies no price-bounds check, the admin can atomically install a malicious oracle, execute a swap at an extreme price, and drain all LP principal.

---

### Finding Description

**Step 1 — Zero timelock is a valid pool configuration.**

`createPool` stores `params.priceProviderTimelock` verbatim; only `type(uint256).max` triggers the immutable path. [1](#0-0) [2](#0-1) 

**Step 2 — Proposal sets `executeAfter = block.timestamp + 0 = block.timestamp`.**

```solidity
uint256 executeAfter = block.timestamp + timelock;   // timelock == 0
pendingPriceProviderExecuteAfter[pool] = executeAfter;
``` [3](#0-2) 

**Step 3 — Execution guard is `block.timestamp < execAfter`, which is `block.timestamp < block.timestamp` → `false`.**

The revert is never triggered, so `executePoolPriceProviderUpdate` succeeds in the very same block (or same transaction via a multicall contract). [4](#0-3) 

**Step 4 — `_validatePriceProvider` only checks token-pair identity, not price sanity.**

A malicious provider that returns `token0()` and `token1()` matching the pool but emits `bid = 1, ask = 2` (or any extreme ratio) passes validation. [5](#0-4) 

**Step 5 — `_getBidAndAskPriceX64` applies no price-bounds check.**

The pool accepts any `(bid, ask)` pair where `bid > 0` and `bid < ask`, regardless of how far it deviates from fair value. [6](#0-5) 

**Step 6 — `nonReentrant` (transient storage) does not block sequential calls in one transaction.**

`ReentrancyGuardTransient` sets and clears a transient flag per call. Two sequential external calls from an attacker contract — `proposePoolPriceProvider` then `executePoolPriceProviderUpdate` — each acquire and release the flag independently, so both succeed within a single transaction. [7](#0-6) [8](#0-7) 

---

### Impact Explanation

The pool admin can drain the entire LP reserve in a single atomic transaction. The swap path reads the price from the newly installed malicious oracle, computes bin deltas at the manipulated price, transfers tokens out to the attacker, and only then requests the callback payment — all before any LP can react. This is a direct loss of user principal (token0 and token1 held in `binTotals`). [9](#0-8) 

---

### Likelihood Explanation

- The pool admin is a semi-trusted role; the protocol explicitly scopes admin power to "inside caps and timelocks."
- A zero timelock is a valid, deployable configuration with no factory-level warning or minimum enforcement.
- LPs have no on-chain mechanism to detect or react to an instantaneous provider swap.
- The attack requires only a single malicious contract and one transaction; no special block-level access is needed.

---

### Recommendation

1. **Enforce a minimum non-zero timelock.** Add a constant (e.g., `MIN_PRICE_PROVIDER_TIMELOCK = 1 hours`) and revert in `createPool` if `params.priceProviderTimelock < MIN_PRICE_PROVIDER_TIMELOCK && params.priceProviderTimelock != type(uint256).max`.
2. **Separate propose and execute into distinct blocks.** Change the guard to `block.timestamp <= execAfter` or require `execAfter > block.timestamp` at proposal time, ensuring at least one block of separation.
3. **Add price-deviation bounds in `_getBidAndAskPriceX64`.** Compare the incoming bid/ask against a trusted reference (e.g., a TWAP or the previous provider's last price) and revert if the deviation exceeds a configured threshold.

---

### Proof of Concept

```solidity
// Attacker contract (pool admin == attacker)
contract Exploit {
    IMetricOmmPoolFactory factory;
    address pool;
    MaliciousProvider malicious; // token0()/token1() match pool; getBidAndAskPrice() returns (1, 2)

    function attack() external {
        // 1. Propose malicious provider — executeAfter = block.timestamp (timelock == 0)
        factory.proposePoolPriceProvider(pool, address(malicious));

        // 2. Execute immediately — block.timestamp < block.timestamp is false → passes
        factory.executePoolPriceProviderUpdate(pool);

        // 3. Swap at extreme price: buy all token0 for near-zero token1
        IMetricOmmPool(pool).swap(
            address(this),
            false,          // token1 → token0
            type(int128).max,
            0,
            abi.encode(address(this)),
            ""
        );
        // Pool's entire token0 reserve transferred to attacker at price 1/2 instead of fair value.
    }

    function metricOmmSwapCallback(int128 amount0, int128 amount1, bytes calldata) external {
        // Pay only the tiny token1 amount demanded at the manipulated price
        IERC20(token1).transfer(pool, uint256(int256(amount1)));
    }
}

  function _getBidAndAskPriceX64() internal returns (uint128 bidPriceX64, uint128 askPriceX64) {
