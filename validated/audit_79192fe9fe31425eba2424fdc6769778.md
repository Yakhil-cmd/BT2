### Title
Unchecked NEP-141 `ft_transfer` Promise Result in `ExitToNear` Precompile Causes Permanent Token Loss - (File: `engine-precompiles/src/native.rs`)

---

### Summary

When the `error_refund` feature is not compiled in, the `ExitToNear` precompile fires the NEP-141 `ft_transfer` promise as a bare fire-and-forget call with no callback to verify success. If the promise fails (e.g., recipient not registered with the NEP-141 contract), the ERC-20 tokens are already burned on the EVM side with no refund path, permanently destroying user funds.

---

### Finding Description

In `ExitToNear::run()`, after computing `callback_args`, the code decides whether to attach a callback:

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)          // ← no callback, result never checked
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs { ... })
};
```

`callback_args` equals `default()` (both fields `None`) whenever:
- The `error_refund` feature is **not** enabled (`refund` is forced to `None` at compile time), **AND**
- The exit is not a wNEAR unwrap (`transfer_near` is `None`).

This covers the two most common exit paths:

1. **Legacy ERC-20 exit** (`ft_transfer` to a NEAR account) — `exit_erc20_token_to_near`, `_ =>` branch, `transfer_near_args = None`.
2. **Legacy ETH/base-token exit** (`ft_transfer` to a NEAR account) — `exit_base_token_to_near`, `None =>` branch, `transfer_near_args = None`.

In both cases the ERC-20/ETH tokens are **burned first** (inside the EVM execution, via `_burn` in `EvmErc20.withdrawToNear`), and then the `ft_transfer` promise is scheduled. Because there is no callback, a failed promise is silently ignored: the NEP-141 tokens remain locked in the Aurora engine account and the ERC-20 representation is gone forever.

The project's own test suite explicitly documents this behavior:

```rust
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
```

---

### Impact Explanation

**Critical — Permanent freezing / destruction of user funds.**

When a user calls `withdrawToNear` on an ERC-20 token with a recipient NEAR account that is not registered with the underlying NEP-141 contract:

1. The ERC-20 tokens are burned in the EVM (irreversible).
2. The `ft_transfer` promise to the NEP-141 contract fails (unregistered account).
3. No callback exists to detect the failure or mint back the ERC-20 tokens.
4. The NEP-141 tokens remain in Aurora's account with no ERC-20 counterpart — permanently inaccessible to the user.

---

### Likelihood Explanation

**Medium.** The `error_refund` feature appears to be opt-in (disabled by default, as evidenced by the `#[cfg(not(feature = "error_refund"))]` guard being the "normal" path). Any user who exits to a NEAR account that has not called `storage_deposit` on the NEP-141 contract triggers the loss. This is a realistic mistake, especially for new users or automated contracts.

---

### Recommendation

Regardless of the `error_refund` feature flag, always attach a callback to the `ft_transfer` / `ft_transfer_call` promise in the `ExitToNear` precompile so that a failed transfer can be detected and the ERC-20 tokens refunded. The callback infrastructure (`exit_to_near_precompile_callback`) already exists and handles refunds correctly when `error_refund` is enabled. The fix is to ensure the callback is always attached for the legacy exit paths, not only when `error_refund` is compiled in.
