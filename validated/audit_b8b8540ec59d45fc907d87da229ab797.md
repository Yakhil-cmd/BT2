### Title
Malicious CKD Participant Can Submit Arbitrary Share Values Without Proof of Correctness, Corrupting the Derived Confidential Key — (`File: src/confidential_key_derivation/protocol.rs`)

---

### Summary

The Confidential Key Derivation (CKD) protocol accepts each participant's share contribution `(norm_big_y, norm_big_c)` with no zero-knowledge proof of correctness. A single malicious participant can send arbitrary group elements, causing the coordinator to aggregate a corrupted output. Every honest party — including the TEE that unmasks the result — will silently accept the wrong confidential key with no indication of failure.

---

### Finding Description

In `do_ckd_coordinator`, the coordinator collects each participant's output and blindly sums them: [1](#0-0) 

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
```

Each participant is supposed to compute, in `compute_signature_share`: [2](#0-1) 

- `big_y  = y_i · G`
- `big_s  = x_i · H(pk ‖ app_id)` (using their actual secret share `x_i`)
- `big_c  = big_s + y_i · app_pk`
- `norm_big_y = λ_i · big_y`, `norm_big_c = λ_i · big_c`

The coordinator then sends the aggregate `(Σ norm_big_y, Σ norm_big_c)` to the TEE, which unmasks it as:

```
confidential_key = big_c_agg - app_sk · big_y_agg
                 = msk · H(pk ‖ app_id)   (when all shares are honest)
```

There is **no proof** that:
1. The `x_i` embedded in `norm_big_c` equals the participant's actual DKG signing share.
2. The same `y_i` was used in both `norm_big_y` and `norm_big_c`.

The `dlogeq` proof module already exists in the codebase and is precisely designed to prove knowledge of a scalar that is simultaneously the discrete log of one point under the standard generator and another point under an alternate generator — exactly the relation needed here. [3](#0-2) 

That module is never invoked by the CKD protocol.

---

### Impact Explanation

A malicious participant sends `(norm_big_y', norm_big_c')` where `norm_big_c'` encodes a wrong secret share `x_i' ≠ x_i`. The coordinator aggregates without detecting the substitution. The resulting `CKDOutput` does not equal `msk · H(pk ‖ app_id)` after unmasking. Every honest party — coordinator and TEE alike — accepts this corrupted value as the legitimate derived key.

**Impact: High — Corruption of CKD outputs so honest parties accept an incorrect confidential derived key.**

---

### Likelihood Explanation

Any single participant in the CKD protocol can trigger this. No cryptographic break is required; the attacker only needs to deviate from the protocol by sending arbitrary elliptic-curve points. The coordinator has no mechanism to detect the deviation. Likelihood is **High** given that the attacker is a legitimate (but malicious) participant.

---

### Recommendation

Each participant must accompany their `(norm_big_y, norm_big_c)` with a zero-knowledge proof of correctness. Concretely, a participant should prove:

1. **DLog equality**: the scalar `x_i` used in `norm_big_c` is the same as the one committed to in the DKG verifying share (using the existing `dlogeq` proof).
2. **Consistent blinding**: the same `y_i` appears in both `norm_big_y` and the ElGamal term inside `norm_big_c` (a second `dlogeq` proof with `app_pk` as the alternate generator).

The coordinator must verify both proofs before adding a participant's contribution to the aggregate.

---

### Proof of Concept

```
Setup: 3-of-3 CKD with participants P1, P2, P3 (P3 is malicious).
       Honest master secret key: msk = λ1·x1 + λ2·x2 + λ3·x3.

1. P1 and P2 compute and send correct (norm_big_y_i, norm_big_c_i).

2. P3 computes correct norm_big_y_3 = λ3 · y3 · G  (blinding term, honest)
   but sends norm_big_c_3' = λ3 · (x3' · H(pk‖app_id) + y3 · app_pk)
   where x3' ≠ x3 is an arbitrary scalar chosen by P3.

3. Coordinator aggregates:
   big_y_agg = norm_big_y_1 + norm_big_y_2 + norm_big_y_3   (correct)
   big_c_agg = norm_big_c_1 + norm_big_c_2 + norm_big_c_3'  (corrupted)

4. TEE unmasks:
   result = big_c_agg - app_sk · big_y_agg
          = (λ1·x1 + λ2·x2 + λ3·x3') · H(pk‖app_id)
          ≠ msk · H(pk‖app_id)

5. TEE silently accepts the wrong confidential key.
   No error is raised anywhere in the protocol.
```

The entry path is `ckd()` → `run_ckd_protocol()` → `do_ckd_participant()` (malicious) → `chan.send_private(waitpoint, coordinator, &(norm_big_y, norm_big_c))` with attacker-chosen values, then `do_ckd_coordinator()` aggregates them without verification. [4](#0-3) [5](#0-4)

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
