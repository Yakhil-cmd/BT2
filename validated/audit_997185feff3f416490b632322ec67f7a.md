### Title
Unchecked `cancelOffer` Return Value Causes Silent State Corruption in Offer Conflict Resolution Dialog - (File: packages/gui/src/components/offers2/CancelOfferList.tsx)

### Summary
In `CancelOfferList.tsx`, the `cancelOffer` RPC mutation is called without checking its return value or using `.unwrap()`. The `onOfferCanceled` callback is then called unconditionally, updating the parent dialog's internal state as if the cancellation succeeded. If the RPC call fails silently, the `OfferEditorCancelConflictingOffersDialog` removes the offer from its conflict list, enables the "Proceed" button, and auto-closes — causing the user to proceed with creating a new offer while the conflicting offer is still live on-chain.

### Finding Description

In `packages/gui/src/components/offers2/CancelOfferList.tsx`, the `handleCancelOffer` function calls `cancelOffer` and then unconditionally invokes `onOfferCanceled`:

```typescript
// CancelOfferList.tsx lines 53-63
async function handleCancelOffer(tradeId: string, canCancelWithTransaction: boolean) {
  const [cancelConfirmed, cancellationOptions] = await openDialog(
    <ConfirmOfferCancellation canCancelWithTransaction={canCancelWithTransaction} />,
  );

  if (cancelConfirmed === true) {
    const secure = canCancelWithTransaction ? cancellationOptions.cancelWithTransaction : false;
    const fee = canCancelWithTransaction ? cancellationOptions.cancellationFee : 0;
    await cancelOffer({ tradeId, secure, fee });   // ← result never checked
    onOfferCanceled(tradeId, secure, fee);          // ← called unconditionally
  }
}
``` [1](#0-0) 

RTK Query mutations called without `.unwrap()` do **not** throw on failure — they return `{ error: ... }` silently. Because `cancelOffer` is not unwrapped and its result is discarded, any RPC failure (network error, offer already taken, wallet not synced, etc.) is swallowed. The `onOfferCanceled` callback fires regardless.

### Impact Explanation

When `cancelOffer` fails silently, the GUI removes the offer from the conflict list and enables the "Proceed" button. The user proceeds to create a new offer believing the conflicting offer has been canceled. The conflicting offer remains live on-chain. Both offers are now simultaneously active. If a counterparty takes the original offer, the user's locked assets are transferred. The new offer the user just created references the same coins, which are now spent — the new offer becomes invalid without the user being informed. The user has made an irreversible asset transfer decision based on a false state representation.

### Likelihood Explanation

This is reachable by any user who:
1. Has open offers with locked assets
2. Attempts to create a new offer that conflicts with existing ones (triggering `OfferEditorCancelConflictingOffersDialog`)
3. Clicks "Cancel" on a conflicting offer while the wallet RPC is temporarily unavailable, the offer was already taken by a counterparty, or any other transient failure condition

No special privileges, leaked keys, or social engineering are required. The failure condition (offer already taken by a third party at the moment the user tries to cancel it) is a realistic race condition in normal offer trading.

### Recommendation

Use `.unwrap()` on the `cancelOffer` call and gate `onOfferCanceled` on confirmed success:

```typescript
async function handleCancelOffer(tradeId: string, canCancelWithTransaction: boolean) {
  const [cancelConfirmed, cancellationOptions] = await openDialog(
    <ConfirmOfferCancellation canCancelWithTransaction={canCancelWithTransaction} />,
  );

  if (cancelConfirmed === true) {
    const secure = canCancelWithTransaction ? cancellationOptions.cancelWithTransaction : false;
    const fee = canCancelWithTransaction ? cancellationOptions.cancellationFee : 0;
    try {
      await cancelOffer({ tradeId, secure, fee }).unwrap();
      onOfferCanceled(tradeId, secure, fee); // only called on success
    } catch (e) {
      showError(e); // surface the failure to the user
    }
  }
}
```

Apply the same fix to `relistOffer` in `OfferManager.tsx`: check the result of `cancelOffer` before navigating to the offer builder.

### Proof of Concept

1. User has an open offer (Offer A) with XCH locked.
2. User attempts to create a new offer (Offer B) using the same XCH.
3. `OfferEditorCancelConflictingOffersDialog` appears, listing Offer A as a required cancellation.
4. Simultaneously, a counterparty takes Offer A on-chain.
5. User clicks "Cancel" on Offer A in the dialog.
6. `cancelOffer` RPC fails (offer already taken / coins already spent).
7. Because `.unwrap()` is absent, no exception is thrown.
8. `onOfferCanceled` fires, removing Offer A from the conflict list.
9. `areOffersAllCleared` becomes `'confirm'`; dialog auto-closes.
10. User proceeds to create Offer B, unaware that Offer A was already taken and their XCH already transferred.
11. User's XCH balance is now lower than expected; Offer B references already-spent coins and will fail when a counterparty attempts to take it — with no prior warning to the user. [5](#0-4) [6](#0-5)
