### Title
Royalty Multiplier Applied to Split Amount Instead of Full Amount in WalletConnect Offer Approval Display — (File: `packages/gui/src/electron/commands/parseCommandDisplay.ts`)

### Summary
In `formatAmountWithRoyalties`, when a WalletConnect `take_offer` or `create_offer_for_ids` command involves multiple NFTs, the royalty percentage is applied to `amount / numNFTs` (a split) rather than the full `amount`. This causes the displayed `amountWithRoyalties` shown to the user in the WalletConnect approval dialog to be systematically underestimated by a factor of `1/N` (where N is the number of NFTs), causing users to approve offers believing they will spend less than they actually will.

### Finding Description

`formatAmountWithRoyalties` in `parseCommandDisplay.ts` computes the royalty-inclusive total as follows:

```typescript
const splitAmount = amount / BigInt(royaltyPercentages.length);
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
const totalAmount = amount + royaltyAmount;
``` [1](#0-0) 

The intent is to compute `royaltyAmount = Σ (amount × Rᵢ / 10000)` for each NFT royalty `Rᵢ`. Instead, the code computes `royaltyAmount = Σ ((amount/N) × Rᵢ / 10000)`, which is exactly `1/N` of the correct value.

For comparison, the offer builder (`offerBuilderDataToOffer.ts`) correctly applies each NFT's royalty multiplier to the full spending amount, not a split:

```typescript
const royaltyMultiplier = 1 + +royaltyPercentageStr / 10_000;
const spendingXch = pendingXchOffer.spendingAmount.minus(feeInMojos);
const newSpendingXch = spendingXch.multipliedBy(royaltyMultiplier).plus(feeInMojos);
``` [2](#0-1) 

This inconsistency confirms the display calculation is wrong.

The buggy `formatAmountWithRoyalties` is called from `withRoyaltyTotals`, which is called from `walletDeltaToDisplay`, which is called for both `chia_wallet.take_offer` and `chia_wallet.create_offer_for_ids` WalletConnect commands: [3](#0-2) 

### Impact Explanation

The WalletConnect approval dialog shows users an `amountWithRoyalties` field that is the primary signal for how much XCH/CAT they will spend when accepting an NFT offer. With N NFTs in the offer, the displayed royalty total is `1/N` of the correct value. For example, with 2 NFTs each carrying a 5% royalty on 100 XCH:

- **Displayed**: 100 + (50×5% + 50×5%) = 105 XCH
- **Correct**: 100 + (100×5% + 100×5%) = 110 XCH

The user approves believing they spend 105 XCH; the actual deduction is 110 XCH. This is a **High** impact: the WalletConnect state causes a user to approve the wrong amount, directly affecting XCH/CAT balance.

### Likelihood Explanation

Any dApp connected via WalletConnect can trigger this path by sending a `take_offer` or `create_offer_for_ids` command that includes multiple NFTs with royalties. No special privileges are required. The user has no other signal to detect the discrepancy before approving.

### Recommendation

Remove the split and apply each royalty percentage to the full `amount`:

```typescript
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (amount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
```

### Proof of Concept

The existing test in `parseCommandDisplay.test.ts` encodes the buggy behavior as expected output. With `amount = 100_000_000n` mojos and two NFTs with royalties 500 (5%) and 10 (0.1%):

- **Buggy code**: `splitAmount = 50_000_000n`; royalty = `50_000_000×500/10000 + 50_000_000×10/10000 = 2_500_000 + 50_000 = 2_550_000`; total = `102_550_000` → `0.00010255` XCH
- **Correct**: royalty = `100_000_000×500/10000 + 100_000_000×10/10000 = 5_000_000 + 100_000 = 5_100_000`; total = `105_100_000` → `0.0001051` XCH

The test asserts `amountWithRoyalties: '0.00010255'`, confirming the underestimation is baked in: [4](#0-3)
