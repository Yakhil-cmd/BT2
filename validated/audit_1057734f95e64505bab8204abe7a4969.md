### Title
Unauthorized Token Drain via Unguarded `sender` in Bank Precompile `transfer` - (File: `x/cronos/keeper/precompiles/bank.go`)

### Summary
The bank precompile's `transfer` method accepts a caller-supplied `sender` address and directly invokes `bankKeeper.SendCoins(ctx, from, to, ...)` with no authorization check. Any contract can therefore drain `evm/<callerContractAddress>` bank-module tokens from any user's account without the user's consent.

### Finding Description

The `TransferMethodName` case in `BankContract.Run` unpacks three user-controlled arguments — `sender`, `recipient`, and `amount` — and immediately uses the `sender` as the `from` address in a `SendCoins` call: [1](#0-0) 

```go
sender := args[0].(common.Address)
recipient := args[1].(common.Address)
amount := args[2].(*big.Int)
...
from := sdk.AccAddress(sender.Bytes())
to := sdk.AccAddress(recipient.Bytes())
...
denom := EVMDenom(contract.Caller())          // evm/<callerContractAddress>
amt := sdk.NewCoin(denom, ...)
...
bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt))
```

The denom is correctly scoped to the calling contract (`evm/<callerAddress>`), but there is **no check** that `from == contract.Caller()` or that `from` has approved the caller to spend on its behalf. `bankKeeper.SendCoins` performs no approval check of its own — it simply moves coins between accounts. [2](#0-1) 

The `mint` method, by contrast, correctly restricts the recipient to an externally supplied address while keeping the issuing denom tied to the caller: [3](#0-2) 

No analogous restriction exists in `transfer`.

### Impact Explanation

Any contract that has previously minted `evm/<contractAddress>` tokens to users can, at any time, call `transfer(victimAddress, attackerAddress, balance)` and drain those tokens from the victim's bank account without the victim's knowledge or consent. This is an **unauthorized transfer of precompile-controlled assets** — a Critical impact under the allowed scope.

### Likelihood Explanation

The bank precompile is a production, unprivileged entry point reachable by any EVM contract. Any DeFi protocol that issues `evm/<contractAddress>` tokens to users (e.g., as liquidity receipts, reward tokens, or wrapped assets) exposes every token holder to this drain. No leaked keys, governance action, or special privilege is required — only the ability to deploy and call a contract.

### Recommendation

Add an authorization check in the `TransferMethodName` case that enforces `sender == contract.Caller()`, mirroring the pattern used by standard ERC-20 `transfer` (as opposed to `transferFrom`). If third-party-initiated transfers are intentionally supported, an on-chain allowance mapping must be introduced and checked before calling `SendCoins`.

```go
// Proposed guard
if sender != contract.Caller() {
    return nil, errors.New("transfer: sender must be the calling contract")
}
```

### Proof of Concept

1. **Setup**: Attacker deploys `MaliciousToken` at address `0xATK`. It calls the bank precompile's `mint(alice, 1000)`, crediting Alice with 1000 `evm/0xATK` tokens in the bank module. Alice may receive these legitimately (e.g., as a DeFi reward).

2. **Attack**: Attacker calls the bank precompile's `transfer` from `MaliciousToken`:
   ```solidity
   IBankModule(0x0000000000000000000000000000000000000064).transfer(
       alice,      // sender  — victim's address, user-controlled
       attacker,   // recipient
       1000        // amount
   );
   ```

3. **Result**: `bankKeeper.SendCoins(ctx, alice, attacker, [{denom:"evm/0xATK", amount:1000}])` executes with no authorization check. Alice's 1000 `evm/0xATK` tokens are transferred to the attacker. No approval from Alice was required. [1](#0-0)
