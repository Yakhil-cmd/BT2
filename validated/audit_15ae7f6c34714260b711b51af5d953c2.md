### Title
Staker Can Instantly Raise Commission to 100% via Commitment, Stealing All Delegator Yield - (File: src/staking/staking.cairo)

### Summary
A staker can exploit the `set_commission_commitment` + `set_commission` flow to atomically raise their pool commission from any low value to 100% (`COMMISSION_DENOMINATOR`), redirecting all future pool rewards to themselves and leaving delegators with zero yield. The code itself acknowledges this gap at line 745–746 with a developer note.

### Finding Description

The commission commitment mechanism is designed to let a staker signal a bounded range for future commission changes. However, there is no upper-bound cap on `max_commission` other than `COMMISSION_DENOMINATOR` (10 000 = 100%).

**Step 1 – `set_commission_commitment`** accepts any `max_commission` up to `COMMISSION_DENOMINATOR`:

```cairo
assert!(max_commission <= COMMISSION_DENOMINATOR, "{}", Error::COMMISSION_OUT_OF_RANGE);
// ...
assert!(current_commission <= max_commission, "{}", Error::MAX_COMMISSION_TOO_LOW);
``` [1](#0-0) 

So a staker with 1% commission can immediately set `max_commission = 10000` (100%).

**Step 2 – `update_commission`** (called by `set_commission`) when a commitment is active only enforces:
- `commission <= commission_commitment.max_commission`
- `commission != old_commission`

```cairo
if self.is_commission_commitment_active(:commission_commitment) {
    assert!(
        commission <= commission_commitment.max_commission,
        "{}",
        Error::INVALID_COMMISSION_WITH_COMMITMENT,
    );
    assert!(commission != old_commission, "{}", Error::INVALID_SAME_COMMISSION);
}
``` [2](#0-1) 

There is **no check that commission can only increase gradually**. The staker can jump from 1% to 100% in a single call. The code even carries an explicit developer warning:

> **Note**: Current commission increase safeguards still allow for sudden commission changes. [3](#0-2) 

**Step 3 – Reward distribution** uses the commission value at the time of attestation:

```cairo
let commission = staker_pool_info.commission();
// ...
let (commission_rewards, pool_rewards) = split_rewards_with_commission(
    rewards_including_commission: pool_rewards_including_commission, :commission,
);
``` [4](#0-3) 

`split_rewards_with_commission` computes `commission_rewards = total * commission / COMMISSION_DENOMINATOR`, so at 100% commission, `pool_rewards = 0`. [5](#0-4) 

### Impact Explanation

All pool rewards for every epoch after the commission spike are routed entirely to the staker as commission; delegators receive zero. Because rewards are only distributed at attestation time and the commission change takes effect immediately, delegators cannot exit before their accrued epoch rewards are stolen. This is **theft of unclaimed yield** — a **High** severity impact under the allowed scope.

### Likelihood Explanation

Any staker who has opened a delegation pool can execute this in two sequential transactions with no special privileges, no external dependencies, and no governance approval. The only prerequisite is that no active commitment already exists. The attack is cheap, deterministic, and leaves no recovery path for delegators whose rewards were already distributed at 100% commission.

### Recommendation

- **Short term:** In `set_commission_commitment`, cap `max_commission` to `current_commission + MAX_COMMISSION_INCREASE_DELTA` (e.g., a protocol-defined constant such as 500 bps per commitment window) so a single commitment cannot authorize a jump to 100%.
- **Long term:** Apply a time-delay or epoch-delay between when a commission increase is committed and when it takes effect, giving delegators a window to exit before the higher rate applies.

### Proof of Concept

1. Staker stakes and calls `set_open_for_delegation`, initializing commission at 100 (1%) to attract delegators.
2. Delegators join the pool.
3. Staker calls `set_commission_commitment(max_commission=10000, expiration_epoch=current_epoch+1)`.
   - Passes because `current_commission (100) <= max_commission (10000)` and `10000 <= COMMISSION_DENOMINATOR`.
4. In the same block (or next), staker calls `set_commission(commission=10000)`.
   - `update_commission` sees an active commitment, checks `10000 <= 10000` ✓ and `10000 != 100` ✓ — succeeds.
5. At the next attestation, `split_rewards_with_commission` is called with `commission=10000`:
   - `commission_rewards = pool_rewards_including_commission * 10000 / 10000 = pool_rewards_including_commission`
   - `pool_rewards = 0`
6. All delegator yield is transferred to the staker's reward address; delegators receive nothing. [6](#0-5)

pub(crate) fn split_rewards_with_commission(
    rewards_including_commission: Amount, commission: Commission,
) -> (Amount, Amount) {
    let commission_rewards = compute_commission_amount_rounded_down(
        :rewards_including_commission, :commission,
    );
    let pool_rewards = rewards_including_commission - commission_rewards;
    (commission_rewards, pool_rewards)
}
```

pub(crate) fn compute_commission_amount_rounded_down(
    rewards_including_commission: Amount, commission: Commission,
) -> Amount {
    mul_wide_and_div(
        lhs: rewards_including_commission,
        rhs: commission.into(),
        div: COMMISSION_DENOMINATOR.into(),
    )
        .expect_with_err(err: InternalError::COMMISSION_ISNT_AMOUNT_TYPE)
}
