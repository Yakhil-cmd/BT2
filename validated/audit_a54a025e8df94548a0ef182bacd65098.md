### Title
ETH Permanently Frozen in `RewardSupplier` When `tick()` Called With No Pending Mint Requests ΓÇö (File: `L1/starkware/solidity/stake/RewardSupplier.sol`)

---

### Summary

The `tick()` function in `RewardSupplier.sol` is declared `payable` and accepts ETH to cover L1ΓåÆL2 messaging fees. However, when there are no pending mint requests (`amountToMint == 0`), the function exits without consuming **or refunding** `msg.value`. Because the contract has no `receive()`, `fallback()`, or withdrawal function anywhere in its inheritance chain, any ETH sent in this code path is permanently frozen.

---

### Finding Description

`tick()` is an unrestricted `external payable` function. Its only ETH-consuming logic is gated behind `if (amountToMint > 0)`:

```solidity
// L1/starkware/solidity/stake/RewardSupplier.sol
function tick() external payable {
    (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

    if (amountToMint > 0) {
        // ...
        uint256 msgFee = msg.value / 2;
        bridge().depositWithMessage{value: msgFee}(...);   // consumes half
        msgFee = msg.value - msgFee;
        messagingContract().sendMessageToL2{value: msgFee}(...); // consumes rest
    }
    // ΓåÉ if amountToMint == 0, msg.value is silently discarded
}
``` [1](#0-0) 

When `amountToMint == 0`, the function returns after the `if` block with no refund. The contract inherits from `RewardSupplierStorage`, `Identity`, and `ProxySupportImpl`; none of these define a `receive()`, `fallback()`, or any ETH-withdrawal function. [2](#0-1) 

The grep across all `.sol` files confirms there is no `withdraw`, `receive()`, or `fallback()` function anywhere in the contract hierarchy that could recover stranded ETH.

---

### Impact Explanation

**High ΓÇö Permanent freezing of funds.**

Any ETH sent to `tick()` when `requiredMinting()` returns zero is irrecoverably locked in the contract. There is no admin escape hatch, no `selfdestruct`, and no withdrawal path. The frozen ETH is the caller's own funds (messaging-fee ETH), not protocol yield, but the impact class "Temporary freezing of funds" (and in practice permanent freezing) is within the allowed High-impact scope.

---

### Likelihood Explanation

**Medium.**

`tick()` carries no access control ΓÇö any address can call it. The condition `amountToMint == 0` is the normal state between reward epochs (when no L2 mint-request messages are pending). A caller who mistakenly (or maliciously, to grief themselves or test the contract) sends ETH during such a window loses it permanently. Off-chain keepers or bots that call `tick()` speculatively with ETH are the most realistic victims.

---

### Recommendation

Add an early revert when ETH is supplied but there is nothing to mint, or refund unused ETH after the branch:

```solidity
function tick() external payable {
    (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

    if (amountToMint > 0) {
        // ... existing logic ...
    } else {
        // Refund any ETH sent when there is nothing to process.
        if (msg.value > 0) {
            (bool ok, ) = msg.sender.call{value: msg.value}("");
            require(ok, "ETH_REFUND_FAILED");
        }
    }
}
```

Alternatively, make `tick()` non-`payable` and require callers to send ETH only via a separate, guarded entry point.

---

### Proof of Concept

1. No L2 mint-request messages are pending; `requiredMinting()` returns `(0, 0)`.
2. Any caller invokes `tick{value: 1 ether}()`.
3. `amountToMint == 0` ΓåÆ the `if` branch is skipped entirely.
4. The function returns; `1 ether` is now held by `RewardSupplier` with no recovery path.
5. Confirmed: no `receive()`, `fallback()`, or `withdraw` exists in `RewardSupplier`, `RewardSupplierStorage`, `ProxySupportImpl`, `GovernanceStub`, or `ProxySupport`. [1](#0-0) [3](#0-2)
