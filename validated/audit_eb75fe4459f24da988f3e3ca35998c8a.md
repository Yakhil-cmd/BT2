### Title
Single-Step Reward Address Change Allows Permanent Loss of Unclaimed Yield - (File: `src/staking/staking.cairo`, `src/pool/pool.cairo`)

---

### Summary

Both the staking contract and the pool contract allow a staker or pool member to change their reward address in a single, unconfirmed step. If an incorrect address is supplied (e.g., a typo, a zero address, or an address whose private key is unknown), all future reward transfers are permanently routed to the wrong address with no recovery path. The protocol itself already demonstrates awareness of the two-step confirmation pattern — `change_operational_address` requires a prior `declare_operational_address` call from the new address — but `change_reward_address` in both contracts lacks this safeguard.

---

### Finding Description

`change_reward_address` in `src/staking/staking.cairo` immediately overwrites `staker_info.reward_address` with the caller-supplied value in a single transaction: [1](#0-0) 

The same pattern exists in `src/pool/pool.cairo` for pool members: [2](#0-1) 

All subsequent reward disbursements read `staker_info.reward_address` (or `pool_member_info.reward_address`) directly and transfer to it without any further validation: [3](#0-2) [4](#0-3) 

By contrast, `change_operational_address` enforces a two-step flow: the new address must first call `declare_operational_address` to register itself as eligible, and only then can the staker call `change_operational_address` to complete the switch: [5](#0-4) 

No equivalent confirmation mechanism exists for `change_reward_address`.

---

### Impact Explanation

If a staker or pool member accidentally supplies an incorrect `reward_address` (wrong address, zero address, or address with no known key), every subsequent `claim_rewards` call — including the final reward flush during `unstake_action` — will transfer STRK tokens to the inaccessible address. The staker's principal is returned to `staker_address` on exit, but all accrued yield is permanently frozen or redirected. This matches the allowed impact: **Permanent freezing of unclaimed yield** (High) or **Theft of unclaimed yield** (High) if the wrong address is attacker-controlled.

---

### Likelihood Explanation

Any active staker or pool member can call `change_reward_address` at any time. Address entry errors (copy-paste mistakes, truncated addresses, wrong network addresses) are a realistic and well-documented class of human error. The risk is compounded by the fact that the staker may not notice the mistake until a `claim_rewards` call silently succeeds but sends funds to the wrong destination. The protocol's own use of a two-step pattern for `change_operational_address` confirms the developers recognize this risk class for address changes.

---

### Recommendation

Implement a two-step process for `change_reward_address` in both `src/staking/staking.cairo` and `src/pool/pool.cairo`:

1. **Step 1 — Propose**: The staker/pool member calls `set_pending_reward_address(new_address)`, which stores `new_address` as a pending value without activating it.
2. **Step 2 — Confirm**: The `new_address` itself calls `accept_reward_address()`, which promotes the pending value to the active `reward_address`.

This mirrors the existing `declare_operational_address` / `change_operational_address` pattern already present in the staking contract. [6](#0-5) 

---

### Proof of Concept

1. Staker `A` has accrued 1000 STRK in unclaimed rewards with `reward_address = 0xAAAA`.
2. Staker `A` calls `staking.change_reward_address(0xDEAD)` — a mistyped address for which no private key exists.
3. The call succeeds immediately; `staker_info.reward_address` is now `0xDEAD`.
4. Any caller invokes `staking.claim_rewards(staker_A)`. The contract reads `reward_address = 0xDEAD` and transfers 1000 STRK to `0xDEAD`.
5. The tokens are permanently inaccessible. Staker `A` has no mechanism to recover them.

The same scenario applies to pool members via `pool.change_reward_address` and the subsequent `pool.claim_rewards` transfer at line 366 of `src/pool/pool.cairo`. [7](#0-6) [8](#0-7)

        fn declare_operational_address(ref self: ContractState, staker_address: ContractAddress) {
            self.general_prerequisites();
            let operational_address = get_caller_address();
            assert!(
                self.operational_address_to_staker_address.read(operational_address).is_zero(),
                "{}",
                Error::OPERATIONAL_IN_USE,
            );
            assert!(
                !self.does_token_exist(token_address: operational_address),
                "{}",
                Error::OPERATIONAL_IS_TOKEN,
            );
            if self.eligible_operational_addresses.read(operational_address) == staker_address {
                return;
            }
            self.eligible_operational_addresses.write(operational_address, staker_address);
            self.emit(Events::OperationalAddressDeclared { operational_address, staker_address });
