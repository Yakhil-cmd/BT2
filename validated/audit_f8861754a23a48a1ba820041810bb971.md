### Title
WalletConnect Commands Execute Against Wallet After Logout Due to No-Op `logOut()` and Persistent `session_request` Listener — (`packages/gui/src/components/app/LogoutButton.tsx`, `packages/gui/src/components/walletConnect/WalletConnectProvider.tsx`, `packages/core/src/components/Auth/AuthProvider.tsx`)

### Summary
`LogoutButton.handleLogout` calls `logOut()` (which is a documented no-op) and then `navigate('/')`. Because `WalletConnectProvider` is mounted above the router and its `session_request` listener is never torn down, and because the backend wallet daemon remains logged in with the same fingerprint, a connected dApp can send `chia_takeOffer`, `chia_cancelOffer`, or any other permitted WalletConnect command after the user has "logged out" and have it execute successfully against the wallet.

### Finding Description

**Step 1 — `logOut()` is a no-op.**

`AuthProvider.handleLogOut` does nothing: [1](#0-0) 

The comment explicitly states the user remains logged in at the backend. The wallet daemon's `get_logged_in_fingerprint` RPC continues to return the same fingerprint.

**Step 2 — `WalletConnectProvider` stays mounted and its listener stays active.**

`WalletConnectProvider` is placed above the router outlet in `AppProviders`: [2](#0-1) 

The `session_request` listener is registered in a `useEffect` keyed on `client`: [3](#0-2) 

Since `client` does not change on logout, the listener is never torn down. `navigate('/')` only changes the rendered route child — `WalletConnectProvider` is unaffected.

**Step 3 — The fingerprint guard passes post-logout.**

`useWalletConnectCommand` reads `currentFingerprint` from the Redux query: [4](#0-3) 

The guard checks: [5](#0-4) 

Because `logOut()` is a no-op, the daemon still reports the same fingerprint. `currentFingerprint` still equals `pair.fingerprint`, so the guard does **not** throw. The `processRef` pattern in `WalletConnectProvider` ensures the latest `process` closure (with the still-valid fingerprint) is always used: [6](#0-5) 

**Step 4 — Command dispatches directly to the main process without additional approval.** [7](#0-6) 

No per-command approval dialog is shown. The command executes silently.

### Impact Explanation
A dApp that was legitimately paired before logout can send `chia_takeOffer` or `chia_cancelOffer` (or any other permitted command) after the user navigates to the SelectKey screen. The offer is accepted or cancelled against the wallet without any user interaction or awareness. This constitutes unauthorized offer acceptance/cancellation affecting XCH, CAT, and NFT — Critical impact per scope rules.

### Likelihood Explanation
Any previously paired dApp retains an active WalletConnect session. The attack window is open indefinitely after logout (until the app is closed or the user explicitly disconnects the dApp). No special timing or race condition is required — the listener is always active and the fingerprint guard always passes as long as the app process is running.

### Recommendation
1. `logOut()` must call the daemon's logout/logoff RPC (or equivalent) so `get_logged_in_fingerprint` returns `null` after logout, causing the fingerprint guard to reject all post-logout commands.
2. Independently, on logout, `WalletConnectProvider` should either disconnect all active sessions or set a flag that causes `processSessionRequest` to immediately reject all incoming `session_request` events with `USER_REJECTED` until a new wallet is selected.

### Proof of Concept
1. Pair a WalletConnect dApp with `chia_takeOffer` permission against a wallet that holds an open offer.
2. Click the logout button — the UI navigates to the SelectKey screen.
3. From the dApp, send a `chia_takeOffer` session request.
4. Observe: `processSessionRequest` fires, the fingerprint guard passes (daemon still reports the same fingerprint), and the offer is taken against the wallet — with no user prompt or error.
