### Title
Incomplete Token Address Whitelist in Pool's `change_reward_address` Enables Permanent Freezing of STRK Rewards — (File: `src/pool/pool.cairo`)

---

### Summary

`Pool.change_reward_address()` and `Pool.enter_delegation_pool()` validate the supplied `reward_address` only against the pool's own token contract, not against all registered protocol tokens. For a BTC-denominated pool, this leaves the STRK token address outside the whitelist. A pool member can therefore set their `reward_address` to the STRK token contract; subsequent reward claims transfer STRK to the token contract itself, permanently destroying the yield.

---

### Finding Description

The Staking contract applies a complete token whitelist when validating `reward_address`:

```cairo
// staking.cairo – change_reward_address
assert!(
    !self.does_token_exist(token_address: reward_address),
    "{}",
    GenericError::REWARD_ADDRESS_IS_TOKEN,
);
```

`does_token_exist` checks STRK **and** every registered BTC token:

```cairo
fn does_token_exist(self: @ContractState, token_address: ContractAddress) -> bool {
    token_address == STRK_TOKEN_ADDRESS || self.btc_tokens.read(token_address).is_some()
}
``` [1](#0-0) 

The Pool contract applies only a single-token check:

```cairo
// pool.cairo – change_reward_address
assert!(
    self.token_dispatcher.contract_address.read() != reward_address,
    "{}",
    GenericError::REWARD_ADDRESS_IS_TOKEN,
);
``` [2](#0-1) 

The same incomplete check appears in `enter_delegation_pool`:

```cairo
let token_address = token_dispatcher.contract_address;
assert!(token_address != pool_member, "{}", Error::POOL_MEMBER_IS_TOKEN);
assert!(token_address != reward_address, "{}", GenericError::REWARD_ADDRESS_IS_TOKEN);
``` [3](#0-2) 

For a BTC pool, `token_dispatcher.contract_address` is the BTC token address. `STRK_TOKEN_ADDRESS` is therefore **not** in the whitelist, so `change_reward_address(STRK_TOKEN_ADDRESS)` passes validation without error.

When `claim_rewards` is later called, rewards are always paid in STRK regardless of pool token:

```cairo
let reward_token = IERC20Dispatcher { contract_address: STRK_TOKEN_ADDRESS };
reward_token.checked_transfer(recipient: reward_address, amount: rewards.into());
``` [4](#0-3) 

If `reward_address == STRK_TOKEN_ADDRESS`, the transfer sends STRK to the STRK token contract itself, where it is irrecoverable.

---

### Impact Explanation

**High — Permanent freezing of unclaimed yield.**

A pool member in a BTC pool who sets `reward_address = STRK_TOKEN_ADDRESS` will have all future STRK reward claims permanently transferred into the STRK token contract. The STRK token contract has no recovery mechanism for tokens sent to its own address. The yield is frozen forever.

---

### Likelihood Explanation

**Low–Medium.** The scenario requires a pool member to supply the STRK token contract address as their reward address. This can occur through a UI bug, copy-paste error, or deliberate self-harm. The protocol explicitly intends to block this class of mistake (the `REWARD_ADDRESS_IS_TOKEN` guard exists in both contracts), but the pool-side guard is incomplete for multi-token deployments. Once BTC pools are live, the gap is reachable by any unprivileged delegator.

---

### Recommendation

Replace the single-address comparison in `Pool.change_reward_address` and `Pool.enter_delegation_pool` with a call back to the staking contract's full token registry, or cache the complete token set in the pool at construction/upgrade time. At minimum, hard-code the STRK token address as an additional forbidden value:

```cairo
// pool.cairo – change_reward_address
let pool_token = self.token_dispatcher.contract_address.read();
assert!(pool_token != reward_address, "{}", GenericError::REWARD_ADDRESS_IS_TOKEN);
assert!(
    STRK_TOKEN_ADDRESS != reward_address,
    "{}",
    GenericError::REWARD_ADDRESS_IS_TOKEN,
);
```

The same dual check should be applied inside `enter_delegation_pool`.

---

### Proof of Concept

1. A staker calls `set_open_for_delegation(btc_token_address)` on the Staking contract, deploying a BTC pool.
2. A delegator calls `Pool.enter_delegation_pool(reward_address: STRK_TOKEN_ADDRESS, amount: X)` on the BTC pool. The assertion `btc_token_address != STRK_TOKEN_ADDRESS` passes; the pool member record is created with `reward_address = STRK_TOKEN_ADDRESS`. [5](#0-4) 
3. Alternatively, after entering with a valid address, the delegator calls `Pool.change_reward_address(STRK_TOKEN_ADDRESS)`. The check `btc_token_address != STRK_TOKEN_ADDRESS` passes; storage is updated. [6](#0-5) 
4. Rewards accrue over epochs. The delegator (or anyone authorized) calls `Pool.claim_rewards(pool_member)`.
5. The pool executes `reward_token.checked_transfer(recipient: STRK_TOKEN_ADDRESS, amount: rewards)`, sending STRK into the STRK token contract. [4](#0-3) 
6. The STRK is permanently locked; the delegator's unclaimed yield is frozen with no recovery path.
