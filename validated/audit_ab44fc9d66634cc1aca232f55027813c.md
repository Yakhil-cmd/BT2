### Title
Malicious Pool Server Can Redirect All Pooling Rewards via TOCTOU on `targetPuzzleHash` — (`packages/gui/src/components/plotNFT/select/PlotNFTSelectPool.tsx`)

---

### Summary

When a user joins or changes a pool, the GUI makes **two independent HTTP fetches** to the pool server: one to display pool details for user review, and a second at submit time to obtain the `targetPuzzleHash` used in the on-chain transaction. A malicious pool operator can serve a legitimate `targetPuzzleHash` during the display phase and switch to an attacker-controlled puzzle hash between display and submit, causing the user's PlotNFT to be permanently linked to the attacker's reward-collection address. All future pooling block rewards flow to the attacker.

---

### Finding Description

The pool-join flow has two separate, uncoordinated fetches to the pool server:

**Fetch 1 — Display (check):** `usePoolInfo` in `PlotNFTSelectBase` reactively fetches pool info as the user types the URL and renders it in the "Verify Pool Details" card. [1](#0-0) [2](#0-1) 

**Fetch 2 — Submit (use):** `prepareSubmitData` calls `getPoolInfo` again at the moment the user clicks "Create" or "Change", and the `targetPuzzleHash` from *this second response* is what gets embedded in the `pw_join_pool` / `create_new_wallet` transaction. [3](#0-2) 

`getPoolInfo` is a thin wrapper that calls `window.appAPI.fetchPoolInfo(poolUrl)`, which in the main process appends `/pool_info` and issues a fresh HTTP GET with no caching or consistency guarantee: [4](#0-3) [5](#0-4) 

The only validation on the returned `targetPuzzleHash` is a non-empty check: [6](#0-5) 

There is no cross-check that the `targetPuzzleHash` used in the transaction matches what was displayed to the user. The same TOCTOU path exists in `PlotNFTChangePool`: [7](#0-6) 

---

### Impact Explanation

`targetPuzzleHash` is the on-chain puzzle hash that receives block rewards from the farmer's PlotNFT singleton. Once `pw_join_pool` is confirmed on-chain, the PlotNFT is cryptographically bound to that puzzle hash for the duration of the `relativeLockHeight` waiting period. If the attacker controls the puzzle hash, they collect all pooling block rewards during that period. The farmer cannot recover rewards already paid to the attacker's address. This is a **Critical** impact: unauthorized payout change affecting pooled farming rewards (XCH).

---

### Likelihood Explanation

The attacker must operate a pool server the victim connects to. This is a reachable, unprivileged role — anyone can run a pool endpoint. The attack window is the round-trip time between the display fetch and the submit fetch (typically milliseconds to seconds), which is easily controlled server-side by delaying the display response and immediately switching the `targetPuzzleHash` in the submit response. No leaked keys, host compromise, or cryptographic break is required.

---

### Recommendation

Capture the `targetPuzzleHash` (and `relativeLockHeight`) from the **display fetch** and pass it directly into `prepareSubmitData` rather than re-fetching. Concretely:

1. Store the pool info object returned by `usePoolInfo` in form state or a ref when the user has reviewed it.
2. In `prepareSubmitData`, use the stored value instead of calling `getPoolInfo` again.
3. Optionally, perform a final consistency re-fetch and abort with an error if `targetPuzzleHash` differs from what was displayed.

This eliminates the TOCTOU window entirely.

---

### Proof of Concept

1. Attacker hosts a pool server at `https://evil-pool.example`.
2. For the first request to `/pool_info` (triggered by the user typing the URL), the server returns a legitimate-looking response with `target_puzzle_hash: "0xLEGIT..."`.
3. The GUI displays this hash in the "Verify Pool Details" card. The user reviews it and clicks "Create" / "Change".
4. `prepareSubmitData` fires a second GET to `https://evil-pool.example/pool_info`.
5. The server now returns `target_puzzle_hash: "0xATTACKER..."` (attacker's own puzzle hash).
6. `pwJoinPool` / `createNewPoolWallet` is called with `targetPuzzlehash: "0xATTACKER..."`.
7. The on-chain transaction confirms. The farmer's PlotNFT is now linked to the attacker's puzzle hash. All block rewards flow to the attacker for the lock-height period.

```typescript
async function prepareSubmitData(data: FormData): SubmitData {
  const { self, fee, poolUrl } = data;
  const initialTargetState = {
    state: self ? 'SELF_POOLING' : 'FARMING_TO_POOL',
  };

  if (!self && poolUrl) {
    const normalizedPoolUrl = normalizeUrl(poolUrl);
    const { targetPuzzleHash, relativeLockHeight } = await getPoolInfo(normalizedPoolUrl);
    if (!targetPuzzleHash) {
      throw new Error(t`Pool does not provide targetPuzzleHash.`);
    }
    if (relativeLockHeight === undefined) {
      throw new Error(t`Pool does not provide relativeLockHeight.`);
    }

    initialTargetState.poolUrl = normalizedPoolUrl;
    initialTargetState.targetPuzzleHash = targetPuzzleHash;
    initialTargetState.relativeLockHeight = relativeLockHeight;
  }

  const feeMojos = chiaToMojo(fee || '0');

  return {
    fee: feeMojos,
    initialTargetState,
  };
}
```
