### Title
Attacker Can Grief New-Account Invoke Admission by Submitting a Signature-Less Invoke That Bypasses `__validate__` via `skip_stateful_validations` - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The gateway's UX feature `skip_stateful_validations` intentionally skips the `__validate__` entry-point call for an invoke transaction with nonce=1 when the sender's account is not yet deployed but a deploy-account transaction is already in the mempool. Because `__validate__` is the only place where the account contract verifies the transaction signature, an attacker who observes a victim's pending `deploy_account` transaction can immediately submit a competing invoke transaction (nonce=1) for the victim's address with an arbitrary/wrong signature. The gateway accepts it without any signature check. The victim's legitimate invoke transaction is then rejected with `DuplicateNonce`, permanently blocking it until the attacker's transaction is executed and reverts.

---

### Finding Description

**Step 1 – The skip path.**

`extract_state_nonce_and_run_validations` calls `run_pre_validation_checks`, which calls `skip_stateful_validations`:

```
validate_state_preconditions  (nonce range check only)
validate_by_mempool           (duplicate-hash / nonce-too-old only)
skip_stateful_validations     ← returns true when conditions met
``` [1](#0-0) 

The condition that triggers the skip:

```rust
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
    return mempool_client
        .account_tx_in_pool_or_recent_block(tx.sender_address())
        .await
        ...
}
``` [2](#0-1) 

**Step 2 – `__validate__` is not called.**

When `skip_validate = true`, `run_validate_entry_point` sets `execution_flags.validate = false`:

```rust
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [3](#0-2) 

Inside `StatefulValidator::perform_validations`, the `__validate__` call is guarded by this flag:

```rust
if !tx.execution_flags.validate {
    return Ok(());
}
// `__validate__` call.
let (_optional_call_info, actual_cost) = self.validate(&tx, tx_context.clone())?;
``` [4](#0-3) 

Because `__validate__` is the account contract's only signature-verification hook, skipping it means the transaction's signature is **never checked** at gateway admission time.

**Step 3 – `account_tx_in_pool_or_recent_block` checks any account transaction, not specifically a deploy-account.**

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [5](#0-4) 

Any transaction from the address (including the attacker's own invoke) satisfies this check.

**Step 4 – Nonce check allows nonce=1 when account_nonce=0.**

`validate_nonce` for non-deploy-account transactions accepts any nonce in `[account_nonce, account_nonce + max_allowed_nonce_gap]`. With `account_nonce=0` and `max_allowed_nonce_gap ≥ 1` (required for the UX feature to function), nonce=1 passes: [6](#0-5) 

**Step 5 – Mempool nonce check blocks the victim's legitimate transaction.**

`validate_by_mempool` → `validate_tx` → `validate_fee_escalation` rejects a second transaction at the same `(address, nonce)` with `DuplicateNonce` (when fee escalation is disabled):

```rust
if self.tx_pool.get_by_address_and_nonce(address, nonce).is_some() {
    return Err(MempoolError::DuplicateNonce { address, nonce });
}
``` [7](#0-6) 

---

### Impact Explanation

**High – Mempool/gateway admission accepts an invalid transaction (wrong signature) and rejects a valid transaction before sequencing.**

An attacker can permanently grief any user who sends a `deploy_account` + `invoke` pair by:
1. Accepting an invoke transaction with an arbitrary/wrong signature into the mempool (no signature check).
2. Causing the victim's legitimate invoke transaction to be rejected with `DuplicateNonce`.
3. Repeating the attack every time the victim resubmits, since the attacker's transaction reverts at execution time (nonce is not consumed on revert), and the attacker can immediately re-submit.

The victim's account also pays fees for the attacker's reverted transaction (fee is charged from the sender's account even on `__validate__` failure).

---

### Likelihood Explanation

**Medium.** The attacker only needs to:
- Monitor the public mempool for `deploy_account` transactions (trivially observable).
- Submit a competing invoke transaction (nonce=1) with an empty/wrong signature before the victim submits their invoke.

No privileged access, no special funds, and no cryptographic capability are required. The attack is repeatable at negligible cost.

---

### Recommendation

1. **Verify the signature independently of `__validate__`.** Before skipping the entry-point call, perform a lightweight signature-format or pre-image check that does not require the account contract to exist.
2. **Alternatively, verify that the account-in-pool transaction is specifically a `deploy_account` transaction**, not just any transaction. The current `account_tx_in_pool_or_recent_block` check is too broad.
3. **Or, accept the skip only when the invoke transaction's sender address matches the contract address computed from the pending deploy-account transaction**, ensuring the attacker cannot inject a competing invoke for an arbitrary victim address.

---

### Proof of Concept

```
1. Alice submits deploy_account tx (nonce=0, valid signature) → accepted by gateway.

2. Bob (attacker) observes Alice's deploy_account in the mempool.

3. Bob submits invoke tx targeting Alice's address:
     sender_address = Alice's address
     nonce          = 1
     signature      = [0x0, 0x0]   ← arbitrary wrong signature

4. Gateway stateful validation for Bob's invoke:
     validate_nonce:              account_nonce=0, tx_nonce=1, gap=1 → OK
     validate_by_mempool:         no duplicate hash → OK
     skip_stateful_validations:   nonce==1 && account_nonce==0
                                  account_tx_in_pool_or_recent_block → true
                                  → skip_validate = true
     run_validate_entry_point:    execution_flags.validate = false
                                  __validate__ NOT called
   → Bob's invoke tx accepted into mempool.

5. Alice submits her legitimate invoke tx (nonce=1, correct signature).
     validate_by_mempool → validate_tx → DuplicateNonce
   → Alice's tx REJECTED.

6. Batcher executes Bob's invoke tx:
     __validate__ runs → wrong signature → tx reverts.
     Alice's account is charged the validation fee.

7. Alice resubmits; Bob repeats from step 3 indefinitely.
``` [8](#0-7) [9](#0-8)

/// Perform transaction validation by the mempool.
async fn validate_by_mempool(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<()> {
    let validation_args = ValidationArgs::new(tx, account_nonce);
    mempool_client
        .validate_tx(validation_args)
        .await
        .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
}

/// Check if validation of an invoke transaction should be skipped due to deploy_account not being
/// processed yet. This feature is used to improve UX for users sending deploy_account + invoke at
/// once.
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
