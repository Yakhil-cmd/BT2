### Title
Silo Whitelist Restriction Applies Only to `submit`, Leaving `call` and `deploy_code` Entrypoints Unrestricted — (File: `engine/src/contract_methods/evm_transactions.rs`)

---

### Summary

In Silo mode, Aurora Engine enforces an `Account`+`Address` whitelist on the `submit` entrypoint (raw Ethereum transactions) but applies **no equivalent whitelist check** to the `call` and `deploy_code` entrypoints (NEAR-native EVM execution paths). Any unprivileged NEAR account can bypass the Silo whitelist entirely by routing EVM execution through `call` or `deploy_code`, defeating the intended access-control invariant and enabling unauthorized interaction with EVM contracts that hold user funds.

---

### Finding Description

Silo mode exposes four whitelist-checking helpers: [1](#0-0) 

`is_allow_submit` (line 136) requires **both** the `Account` and `Address` whitelists to pass. `is_allow_deploy` (line 131) requires **both** the `Admin` and `EvmAdmin` whitelists. These are the intended guards for EVM execution and contract deployment respectively.

The `submit` entrypoint delegates to `engine::submit`, which is the code path where the whitelist check is enforced (confirmed by the integration test `test_submit_access_right` which shows `EngineErrorKind::NotAllowed` is returned when the caller is not whitelisted): [2](#0-1) 

However, the `call` entrypoint only calls `require_running` and then directly invokes `engine.call_with_args` — **no whitelist check is present**: [3](#0-2) 

Likewise, `deploy_code` only calls `require_running` before invoking `engine.deploy_code_with_input` — **no `is_allow_deploy` check is present**: [4](#0-3) 

The whitelist infrastructure is defined and functional: [5](#0-4) 

But it is only wired into the `submit` path, leaving `call` and `deploy_code` as unrestricted bypass routes — a direct structural analog to M-04's partial `onlyApprovedContracts` restriction.

---

### Impact Explanation

**Impact: High — Theft of user funds / permanent freezing of funds.**

A Silo operator enables whitelists specifically to restrict which NEAR accounts and EVM addresses may execute EVM transactions or deploy contracts. Any EVM contract deployed inside the Silo that relies on the Silo whitelist as its sole access-control layer (i.e., it does not implement its own `msg.sender` checks) is vulnerable: a non-whitelisted NEAR account can call `call` with crafted calldata to invoke privileged functions on that contract, drain balances, or lock funds.

Additionally, `deploy_code` being unrestricted means non-whitelisted accounts can deploy arbitrary EVM bytecode, including contracts that interact with existing Silo contracts holding user funds. [6](#0-5) 

---

### Likelihood Explanation

**Likelihood: Medium.**

- The `call` and `deploy_code` entrypoints are public NEAR contract methods callable by any NEAR account.
- No special privilege, leaked key, or social engineering is required.
- The attacker only needs to know the Aurora contract account ID and the ABI of the target EVM contract.
- Silo mode is an opt-in feature; the vulnerability is only exploitable when a Silo operator has enabled whitelists, which is the primary use-case for Silo mode.

---

### Recommendation

Add whitelist enforcement to `call` and `deploy_code` that mirrors the check already present in the `submit` path:

- In `call`: call `is_allow_submit(io, &predecessor_account_id, &predecessor_address)` and return `ERR_NOT_ALLOWED` if it fails, but only when Silo mode is active (i.e., `get_silo_params` returns `Some`).
- In `deploy_code`: call `is_allow_deploy(io, &predecessor_account_id, &predecessor_address)` and return `ERR_NOT_ALLOWED` if it fails, but only when Silo mode is active. [7](#0-6) 

---

### Proof of Concept

1. Silo operator calls `set_silo_params` to enable Silo mode and calls `set_whitelists_statuses` to enable all four whitelists (`Admin`, `EvmAdmin`, `Account`, `Address`).
2. Operator adds only `alice.near` / Alice's EVM address to the whitelists. `bob.near` is **not** whitelisted.
3. Alice deploys an EVM vault contract (via `submit`) that holds ERC-20 tokens and has no internal `msg.sender` guard, relying on the Silo whitelist to restrict callers.
4. Bob (non-whitelisted) calls `submit` → receives `EngineErrorKind::NotAllowed`. ✓ Restriction works.
5. Bob calls `call` with calldata encoding `vault.withdraw(bob_address, amount)` → **succeeds** because `call` does not invoke `is_allow_submit`.
6. Bob's EVM address receives the tokens; he calls `ExitToNear` precompile (also via `call`) to bridge them to `bob.near`.
7. Funds are stolen from Alice's vault with no whitelist enforcement triggered. [3](#0-2) [6](#0-5)

        io.return_output(&result_bytes);

        Ok(result)
    })

    /// Check if the whitelist is enabled.
    pub fn is_enabled(&self) -> bool {
        // White list is disabled by default. So return `false` if the key doesn't exist.
        let key = self.key(STATUS);
        self.io
            .read_storage(&key)
            .is_some_and(|value| value.to_vec() == [1])
    }

    fn key(&self, value: &[u8]) -> Vec<u8> {
        let mut bytes = Vec::with_capacity(1 + value.len());

        bytes.push(u8::from(self.kind));
        bytes.extend_from_slice(value);
        bytes_to_key(KeyPrefix::Whitelist, &bytes)
    }

    /// Add a new element to the whitelist.
    pub fn add<A: AsBytes + ?Sized>(&mut self, element: &A) {
        let key = self.key(element.as_bytes());
        self.io.write_storage(&key, &[]);
    }

    /// Remove a new element from the whitelist.
    pub fn remove<A: AsBytes + ?Sized>(&mut self, element: &A) {
        let key = self.key(element.as_bytes());
        self.io.remove_storage(&key);
    }

    /// Check if the element is present in the whitelist.
    pub fn is_exist<A: AsBytes + ?Sized>(&self, element: &A) -> bool {
        let key = self.key(element.as_bytes());
        self.io.storage_has_key(&key)
    }
