### Title
Single Malicious Participant Can Permanently Abort Signing, DKG, and CKD by Sending a Malformed Message via `recv_from_others` — (File: `src/protocol/helpers.rs`)

---

### Summary

`recv_from_others` unconditionally propagates deserialization errors from any individual participant. A single malicious participant who sends a message with a syntactically valid routing header but an invalid MessagePack payload can cause the function to return an error immediately, aborting the entire protocol for all honest parties. This is the direct analog of the H-03 push-strategy pattern: just as `claimAuction()` iterates over all bidders and can be aborted by one malicious bidder's callback, `recv_from_others` iterates over all participants and can be aborted by one malicious participant's malformed message.

---

### Finding Description

`recv_from_others` in `src/protocol/helpers.rs` is the central collection primitive used in virtually every protocol round:

```rust
// src/protocol/helpers.rs lines 19-24
while !seen.full() {
    let (from, msg) = chan.recv(waitpoint).await?;   // <-- error propagated immediately
    if seen.put(from) {
        messages.push((from, msg));
    }
}
```

`chan.recv` in `src/protocol/internal.rs` deserializes the payload with `rmp_serde`:

```rust
// src/protocol/internal.rs lines 338-340
let decoded: Result<T, Box<dyn error::Error + Send + Sync>> =
    rmp_serde::decode::from_slice(message_data).map_err(std::convert::Into::into);
Ok((from, decoded?))   // <-- ProtocolError::DeserializationError propagated
```

If any participant sends a message whose header passes the `push_message` filter (valid length and parseable `MessageHeader`) but whose payload fails MessagePack deserialization, `chan.recv` returns `Err(ProtocolError::DeserializationError(...))`. The `?` in `recv_from_others` propagates this error immediately, before `seen.put(from)` is ever called, aborting the entire protocol.

**The echo broadcast implementation explicitly avoids this exact problem.** `reliable_broadcast_receive_all` in `src/protocol/echo_broadcast.rs` ignores errors from `recv`:

```rust
// src/protocol/echo_broadcast.rs lines 176-182
// The recv should be failure-free
// This translates to ignoring the received message when deemed wrong
match chan.recv(wait).await {
    Ok(value) => (from, (sid, vote)) = value,
    _ => continue,   // <-- errors are silently skipped
};
```

`recv_from_others` has no equivalent guard. The inconsistency means the echo-broadcast rounds of DKG are resilient to malformed messages, but every other round — including all signing rounds — is not.

### Impact Explanation

A single malicious participant who is a valid member of the participant set can permanently deny signing, key generation, reshare, refresh, and CKD for all honest parties. The attacker can repeat the attack for every protocol invocation, making the denial indefinite. There is no fallback path: the coordinator's protocol instance returns a `ProtocolError`, no output is produced, and honest parties receive nothing.

This matches the allowed impact: **High — Permanent denial of signing, key generation, reshare, refresh, or CKD for honest parties under valid protocol inputs and documented trust assumptions.**

The README documents only the case where a message is *never received* (indefinite wait, caller handles with timeout). It does not document — and does not intend — the case where a *malformed message causes the protocol to fail with an error*. That is the undocumented, unmitigated path exploited here.

---

### Likelihood Explanation

The attack requires no special privileges beyond being a valid participant. The malicious participant knows the protocol structure (they participate in it), so they can construct a message with a valid `MessageHeader` (correct channel tag and waitpoint) and a garbage payload. The `push_message` filter only validates the header, not the payload, so the malformed message is accepted into the buffer and routed to the correct `recv` call. The attacker can time the message to arrive at any point during the collection loop. The attack is trivially repeatable across every signing session.

---

### Recommendation

Mirror the error-handling pattern already used in `reliable_broadcast_receive_all`: skip messages that fail deserialization and continue waiting, rather than propagating the error.

```rust
// Proposed fix for src/protocol/helpers.rs
while !seen.full() {
    let result: Result<(Participant, T), _> = chan.recv(waitpoint).await;
    let Ok((from, msg)) = result else {
        continue; // skip malformed messages, keep waiting
    };
    if seen.put(from) {
        messages.push((from, msg));
    }
}
```

This preserves the documented "wait indefinitely" semantics for non-responsive participants while removing the ability of a single participant to abort the protocol via a crafted message.

### Proof of Concept

1. A set of N participants runs a signing protocol (e.g., OT-based ECDSA `sign`). One participant, P_m, is malicious.
2. P_m observes the protocol and determines the current `waitpoint` for the signature-share collection round (waitpoint 0 in `do_sign_coordinator`).
3. P_m constructs a raw message: a valid `MessageHeader` encoding the correct channel tag and waitpoint, followed by arbitrary bytes that are not valid MessagePack for `Scalar`.
4. P_m delivers this message to the coordinator via the application's `protocol.message(p_m, malformed_bytes)` call.
5. The coordinator's `recv_from_others::<Scalar>` at `src/ecdsa/ot_based_ecdsa/sign.rs:114` calls `chan.recv(wait0)`, which calls `rmp_serde::decode::from_slice` on the garbage payload.
6. Deserialization fails; `chan.recv` returns `Err(ProtocolError::DeserializationError(...))`.
7. The `?` in `recv_from_others` at `src/protocol/helpers.rs:20` propagates the error.
8. `do_sign_coordinator` returns `Err`, the coordinator's protocol instance terminates with an error, and no signature is produced.
9. P_m repeats this for every subsequent signing attempt, permanently denying signing for all honest parties.
