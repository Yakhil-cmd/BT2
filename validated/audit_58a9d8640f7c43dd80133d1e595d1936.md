### Title
Duplicate Blob Hashes Not Rejected in `parse_blobs_list`, Leading to Incorrect Blob Gas Accounting — (File: `basic_bootloader/src/bootloader/transaction/blobs.rs`)

---

### Summary

`parse_blobs_list` accepts EIP-4844 transactions containing duplicate blob versioned hashes without any deduplication check. Because `blob_gas_used` is derived directly from `blobs.len()` (the raw, undeduped count), a user who crafts a transaction with repeated blob hashes causes the system to compute an inflated blob gas fee and to expose the same hash at multiple `BLOBHASH` indices — a state that Ethereum consensus would reject outright.

---

### Finding Description

`parse_blobs_list` in `basic_bootloader/src/bootloader/transaction/blobs.rs` iterates over the raw `BlobHashesList` from the transaction and unconditionally pushes every entry into the result `ArrayVec`:

```rust
for blob_hash in blobs_list.iter() {
    let blob_hash = blob_hash?;
    if blob_hash[0] != VERSIONED_HASH_VERSION_KZG {
        return Err(...);
    }
    let blob_hash = Bytes32::from_array(*blob_hash);
    result.push(blob_hash);   // ← no duplicate check
}
``` [1](#0-0) 

The returned `ArrayVec` is stored verbatim in `TxLevelMetadata.blobs` / `EthereumTransactionMetadata.blobs`: [2](#0-1) [3](#0-2) 

Both the ZK and Ethereum validation paths then derive `blob_gas_used` from `blobs.len()` (the inflated count):

```rust
// ZK path
let num_blobs = system.metadata.num_blobs();   // == blobs.len()
let blob_gas_used = num_blobs as u64 * GAS_PER_BLOB;
``` [4](#0-3) 

```rust
// Ethereum path
let blob_gas_used = (blobs.len() as u64) * GAS_PER_BLOB;
``` [5](#0-4) 

`num_blobs()` is implemented as `self.blobs.len()`: [6](#0-5) 

The `BLOBHASH` opcode is served by `get_blob_hash(idx)`, which indexes directly into this array: [7](#0-6) 

---

### Impact Explanation

1. **Incorrect blob gas accounting (resource accounting bug):** `blob_gas_used` is proportional to the raw blob count including duplicates. A transaction with `N` copies of the same hash is charged `N × GAS_PER_BLOB` blob gas instead of `1 × GAS_PER_BLOB`. The fee charged to the sender is correspondingly inflated.

2. **EVM semantic mismatch:** EIP-4844 mandates that all versioned hashes in a transaction are unique. ZKsync OS accepts and executes transactions that Ethereum consensus would reject, creating a divergence between the ZKsync OS state transition and the expected Ethereum-equivalent behavior.

3. **`BLOBHASH` opcode returns the same hash at multiple indices:** Any contract that relies on blob hashes being unique identifiers (e.g., using them as keys or expecting `BLOBHASH(i) != BLOBHASH(j)` for `i != j`) will observe incorrect behavior when a transaction with duplicate hashes is processed.

---

### Likelihood Explanation

Any unprivileged user can craft a valid-looking EIP-4844 transaction (type `0x03`) with a `blob_versioned_hashes` list containing repeated entries. The only constraint is that each entry must start with `VERSIONED_HASH_VERSION_KZG` (`0x01`). No special access, governance role, or oracle manipulation is required. The attack path is a single crafted transaction submitted to the sequencer.

---

### Recommendation

Add a deduplication check inside `parse_blobs_list` before pushing each hash:

```rust
if result.contains(&blob_hash) {
    return Err(TxError::Validation(InvalidTransaction::DuplicateBlobHash));
}
result.push(blob_hash);
```

Alternatively, collect into a `BTreeSet` first and verify `set.len() == raw_count` before constructing the `ArrayVec`. A corresponding `InvalidTransaction::DuplicateBlobHash` variant should be added to the error enum.

---

### Proof of Concept

1. Craft an EIP-4844 transaction with `blob_versioned_hashes = [H, H]` where `H` is any 32-byte value with `H[0] == 0x01`.
2. Submit the transaction to ZKsync OS.
3. `parse_blobs_list` returns an `ArrayVec` of length 2 containing `[H, H]`.
4. `blob_gas_used = 2 * GAS_PER_BLOB`; the sender is charged twice the expected blob gas fee.
5. `BLOBHASH(0)` and `BLOBHASH(1)` both return `H`, violating the EIP-4844 uniqueness invariant.
6. The transaction is accepted and executed; on Ethereum mainnet the same transaction would be rejected at the consensus layer. [8](#0-7) [9](#0-8) [10](#0-9)

```rust
pub fn parse_blobs_list<const MAX_BLOBS_IN_TX: usize>(
    blobs_list: BlobHashesList<'_>,
) -> Result<arrayvec::ArrayVec<Bytes32, MAX_BLOBS_IN_TX>, TxError> {
    let mut result = arrayvec::ArrayVec::<_, MAX_BLOBS_IN_TX>::new();
    if blobs_list.count > MAX_BLOBS_IN_TX {
        return Err(TxError::Validation(InvalidTransaction::BlobListTooLong));
    }

    for blob_hash in blobs_list.iter() {
        let blob_hash = blob_hash?;

        if blob_hash[0] != VERSIONED_HASH_VERSION_KZG {
            return Err(TxError::Validation(
                InvalidTransaction::BlobElementIsNotSupported,
            ));
        }

        // NOTE: we do NOT check that this blob hash is meaningful - we are not worried about block validity
        // from consensus perspective. And KZG blob precompile requires explicit preimage anyway
        let blob_hash = Bytes32::from_array(*blob_hash);
        result.push(blob_hash);
    }

    if result.is_empty() {
        // transactions that allow blobs should have at least one
        return Err(TxError::Validation(InvalidTransaction::EmptyBlobList));
    }

    Ok(result)
