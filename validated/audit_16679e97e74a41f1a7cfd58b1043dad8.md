Audit Report

## Title
Hot Keys Unconditionally Inherited by Child Neuron When Spawned to a Different Controller - (File: rs/nns/governance/src/governance.rs)

## Summary
When `spawn_neuron` is called with a `new_controller` different from the parent neuron's controller, the child neuron is created with the parent's full `hot_keys` list copied verbatim. The new controller receives a neuron they believe they fully own, but the spawner's hot key holders retain the ability to vote, change followees, and join the Neurons' Fund on that neuron without the new controller's knowledge or consent.

## Finding Description
In `rs/nns/governance/src/governance.rs`, `child_controller` is resolved at L2651–2655: if `spawn.new_controller` is `Some(p)`, the child's controller is set to `p`, which may be a completely different principal from the parent's controller. Immediately after, at L2704–2718, the child neuron is built with `.with_hot_keys(parent_neuron.hot_keys.clone())` — unconditionally, with no branch that clears or filters the hot key list when `child_controller != parent_neuron.controller()`. [1](#0-0) [2](#0-1) 

Authorization for non-privileged neuron operations is checked via `is_hotkey_or_controller` in `rs/nns/governance/src/neuron/types.rs` at L253–256, which returns `true` for any principal in `hot_keys`. This means every hot key inherited from the parent neuron immediately grants its holder the ability to call `RegisterVote`, `Follow`, `JoinCommunityFund`, `LeaveCommunityFund`, and `ChangeAutoStakeMaturity` on the child neuron. [3](#0-2) 

There is no existing guard, conditional, or post-spawn cleanup that removes inherited hot keys when the child's controller differs from the parent's controller.

## Impact Explanation
This constitutes unauthorized access to a neuron's governance functions. A principal (User A) who holds a hot key on the parent neuron retains the ability to influence the child neuron's governance participation — voting on NNS proposals, altering followees, and committing the new controller's maturity to the Neurons' Fund — without the new controller's (User B's) knowledge or consent. The `JoinCommunityFund` action is the most consequential: it commits User B's future maturity to SNS swap participation, which is financially significant and not automatically reversed. This maps to the **Medium** impact tier: meaningful security impact on governance assets with a concrete, demonstrable exploit path, but constrained by the requirement that User B does not proactively audit and remove inherited hot keys.

## Likelihood Explanation
`spawn_neuron` with a distinct `new_controller` is the standard on-chain mechanism for transferring maturity-derived ICP to another party. A spawner who has added hot keys for operational convenience (e.g., a mobile signing key or a backup key) will silently transfer those authorizations to the recipient's neuron. The recipient has no in-protocol notification of inherited hot keys. Deliberate exploitation is low-to-medium likelihood; unintentional authorization leakage is medium-to-high likelihood given common hot key usage patterns.

## Recommendation
In `spawn_neuron`, when `child_controller` differs from `parent_neuron.controller()`, the child neuron should be constructed with an empty hot key list instead of `parent_neuron.hot_keys.clone()`. If hot key inheritance is desired in same-controller spawns, the current behavior can be preserved for that case only. Explicit opt-in inheritance via a new `Spawn` message field would be the safest design.

## Proof of Concept
1. User A (controller of neuron N) calls `manage_neuron` → `AddHotKey { hot_key_to_add: H1 }` on neuron N.
2. User A calls `manage_neuron` → `Spawn { new_controller: Some(UserB), nonce: None, percentage_to_spawn: Some(100) }`.
3. Governance executes `spawn_neuron`: `child_controller = UserB`, but `.with_hot_keys(parent_neuron.hot_keys.clone())` sets `child.hot_keys = [H1]`.
4. User A (signing as H1) calls `manage_neuron` on child neuron C → `RegisterVote { proposal_id: X, vote: Yes }`. Governance accepts because `is_hotkey_or_controller(H1)` returns `true`.
5. User A (signing as H1) calls `manage_neuron` on C → `JoinCommunityFund {}`. Governance accepts for the same reason, committing User B's maturity to the Neurons' Fund without User B's consent.
6. A local PocketIC or integration test can reproduce this deterministically by: creating a neuron, adding a hot key, spawning to a different principal, then asserting that the hot key principal can successfully submit a `RegisterVote` or `JoinCommunityFund` command on the child neuron.
