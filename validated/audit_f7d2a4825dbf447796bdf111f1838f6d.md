### Title
ERC-20 Tokens Permanently Burned Without Recovery When `ft_transfer` Promise Fails in `ExitToNear` Precompile ‚Äî (`engine-precompiles/src/native.rs`, `etc/eth-contracts/contracts/EvmErc20.sol`)

### Summary

When a user calls `withdrawToNear` on `EvmErc20`, ERC-20 tokens are burned atomically within the EVM transaction before the asynchronous NEAR `ft_transfer` promise executes. If that promise fails and the `error_refund` compile-time feature is not enabled, the burned ERC-20 tokens are permanently destroyed with no recovery path, while the corresponding NEP-141 tokens remain stranded in the Aurora contract. This is the direct analog of the reported "surplus burn" pattern: an irreversible debit on one side of a two-step cross-chain operation is not matched by a compensating credit when the second step fails.

### Finding Description

**Step 1 ‚Äî Burn before promise (EVM side):**

`EvmErc20.withdrawToNear` burns the caller's ERC-20 tokens first, then calls the `ExitToNear` precompile:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // irreversible within EVM tx
    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;
    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    }
}
```

The burn and promise scheduling are atomic in the EVM transaction. The NEAR `ft_transfer` executes asynchronously in a later receipt. [1](#0-0) 

**Step 2 ‚Äî Callback args carry `refund: None` without the feature flag:**

In `ExitToNear::run`, the callback args are constructed with `refund: None` when `error_refund` is not compiled in:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,
    transfer_near: transfer_near_args,
};
``` [2](#0-1) 

**Step 3 ‚Äî Callback silently drops the failure:**

In `exit_to_near_precompile_callback`, when the `ft_transfer` promise fails, the code checks `args.refund`:

```rust
} else if let Some(args) = args.refund {
    // Exit call failed; need to refund tokens
    let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;
    ...
} else {
    None   // ‚Üê always taken when error_refund is off; no re-mint, no recovery
};
``` [3](#0-2) 

Because `refund` is `None`, the `else` branch is always taken on failure. `refund_on_error` ‚Äî which would re-mint the ERC-20 tokens ‚Äî is never called. [4](#0-3) 

**Step 4 ‚Äî The accounting gap:**

After the failed promise:
- The user's ERC-20 tokens are permanently burned (EVM supply reduced).
- The NEP-141 tokens that Aurora held on behalf of the user remain in the Aurora contract with no claimant ‚Äî the exact "orphaned asset" pattern from the reference report.
- There is no on-chain mechanism for the user to recover either asset without admin intervention.

### Impact Explanation

**Critical ‚Äî Permanent freezing of funds.**

Any user whose `ft_transfer` promise fails loses their ERC-20 tokens permanently. The corresponding NEP-141 balance stays locked in the Aurora contract. Neither the user nor any other party can reclaim the ERC-20 value through normal protocol interactions. This matches the reference report's "orphaned base asset" impact: the burn happens outside the normal redemption path, leaving the backing asset with no claimant.

### Likelihood Explanation

The `ft_transfer` promise can fail for user-reachable reasons without any admin involvement:

- The NEAR recipient account is not registered with the NEP-141 contract (a common requirement for NEP-141 tokens). Any user can trigger this by specifying an unregistered recipient.
- The existing integration test explicitly confirms this behavior and accepts it as the non-`error_refund` outcome:

```rust
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
``` [5](#0-4) 

The `error_refund` feature is not the default compilation path (the `#[cfg(not(feature = "error_refund"))]` branches are present and tested), meaning any production build without this flag is vulnerable.

### Recommendation

1. **Enable `error_refund` unconditionally in production** or promote it from a compile-time feature to always-on behavior. The `refund_call_args` and `refund_on_error` machinery already exists and is correct.
2. **For `EvmErc20` (legacy):** The legacy contract does not embed the sender address in the precompile input, so even with `error_refund` enabled the refund address would be parsed from the wrong bytes. Migrate users to `EvmErc20V2`, which correctly encodes the sender as the refund address.
3. **Alternatively**, restructure the exit flow to escrow (transfer to a holding address) rather than burn ERC-20 tokens before the NEAR promise resolves, burning only on confirmed success ‚Äî eliminating the asymmetry entirely. [6](#0-5) 

### Proof of Concept

The existing test `test_exit_to_near_refund` in `engine-tests/src/tests/erc20_connector.rs` is the proof of concept. It demonstrates that when `ft_transfer` fails (recipient `"unregistered.near"` is not registered with the NEP-141 contract):

- Without `error_refund`: `erc20_balance == FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT` ‚Äî the burned tokens are gone.
- With `error_refund`: `erc20_balance == FT_TRANSFER_AMOUNT` ‚Äî tokens are refunded.

The test confirms the permanent loss path is reachable by any unprivileged user who specifies an unregistered recipient, which is a normal user mistake with catastrophic and unrecoverable consequences under the default (non-`error_refund`) build. [7](#0-6)
