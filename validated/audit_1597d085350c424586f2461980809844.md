### Title
`calculateNFTRoyalties` Hardcodes `nftSellerNetAmount` to Full Asking Price, Permanently Suppressing Royalty Deduction Display and Negative-Amount Guard - (File: packages/gui/src/components/offers/utils.ts)

### Summary

`calculateNFTRoyalties` in `packages/gui/src/components/offers/utils.ts` assigns `nftSellerNetAmount = amount` unconditionally — the correct deduction (`amount - royaltyAmount - makerFee`) is commented out. This causes the GUI to display the full asking price as "Net Proceeds" / "You will receive" instead of the post-royalty net, and permanently disables the negative-amount safety guard that is supposed to block offer creation when royalties exceed the asking price.

### Finding Description

In `calculateNFTRoyalties`:

```typescript
// packages/gui/src/components/offers/utils.ts  lines 306-329
const royaltyAmount: number = royaltyPercentage ? (royaltyPercentage / 100) * amount : 0;
const royaltyAmountString: string = formatAmount(royaltyAmount);
const nftSellerNetAmount: number = amount;          // ← hardcoded; correct calc is commented out
// : parseFloat(
//     (amount - parseFloat(royaltyAmountString) - makerFee).toFixed(12),
//   );
```

`nftSellerNetAmount` is always equal to `amount` (the full asking price), regardless of royalty percentage. [1](#0-0) 

This value propagates to two consumer sites:

**1. NFTOfferEditor (offer creation)**

```typescript
// NFTOfferEditor.tsx  lines 182-199
const result = useMemo(() => {
  ...
  return { ...calculateNFTRoyalties(...), royaltyPercentage };
}, [...]);
const { nftSellerNetAmount, ... } = result ?? {};
const showNegativeAmountWarning = (nftSellerNetAmount ?? 0) < 0;  // always false
```

The "You will receive" / "They will receive" label renders `nftSellerNetAmount` — always the full price. The `showNegativeAmountWarning` guard that should block offer creation when royalties exceed the asking price can never fire. [2](#0-1) [3](#0-2) [4](#0-3) 

**2. NFTOfferViewer (offer acceptance)**

```typescript
// NFTOfferViewer.tsx  lines 397-402
const overrideNFTSellerAmount =
  exchangeType === NFTOfferExchangeType.TokenForNFT
    ? assetType === OfferAsset.CHIA
      ? chiaToMojo(nftSaleInfo?.nftSellerNetAmount ?? 0)   // full price, not net
      : catToMojo(nftSaleInfo?.nftSellerNetAmount ?? 0)
    : undefined;
```

`overrideNFTSellerAmount` — derived from the wrong `nftSellerNetAmount` — is passed to `NFTOfferSummary` to override the displayed seller amount. The "Net Proceeds" label also renders `nftSaleInfo?.nftSellerNetAmount` directly. [5](#0-4) [6](#0-5) 

### Impact Explanation

An NFT seller who sets an asking price of 10 XCH on an NFT with a 20% royalty will see "You will receive: 10 XCH" and "Net Proceeds: 10 XCH" in both the offer editor and the offer viewer. The actual on-chain proceeds are 8 XCH (10 − 2 royalty). The seller is systematically shown an inflated net amount and may create or accept offers at prices they would not otherwise agree to. Additionally, the negative-amount guard — which should prevent offer creation when royalties exceed the asking price — is permanently disabled because `nftSellerNetAmount` is always positive.

This matches the High impact criterion: **display of the wrong amount that causes a user to approve or sign the wrong offer terms**.

### Likelihood Explanation

Any user who creates or views an NFT offer where the NFT has a non-zero `royaltyPercentage` triggers this path. NFT royalties are a standard feature of the Chia NFT1 standard. The bug is unconditional — there is no configuration or oracle that could correct it at runtime. The commented-out correct calculation confirms this was a deliberate (if temporary) change that was never reverted.

### Recommendation

Restore the correct `nftSellerNetAmount` calculation:

```typescript
const nftSellerNetAmount: number = parseFloat(
  (amount - parseFloat(royaltyAmountString) - (exchangeType === NFTOfferExchangeType.NFTForToken ? 0 : makerFee)).toFixed(12)
);
```

Verify that `showNegativeAmountWarning` correctly fires when royalties exceed the asking price, and add a test covering the `nftSellerNetAmount` field in `calculateNFTRoyalties`.

### Proof of Concept

1. Open the Chia GUI and navigate to the NFT offer editor.
2. Select an NFT that has a non-zero royalty percentage (e.g., 20%).
3. Enter an asking price of 1 XCH.
4. Observe: "You will receive: 1 XCH" is displayed. Correct value should be 0.8 XCH.
5. Enter an asking price of 0.001 XCH with a 100% royalty NFT.
6. Observe: no negative-amount warning appears, and offer creation is not blocked. The seller would receive 0 XCH net but the GUI shows 0.001 XCH.

The root cause is confirmed at: [7](#0-6)

export function calculateNFTRoyalties(
  amount: number,
  makerFee: number,
  royaltyPercentage: number,
  exchangeType: NFTOfferExchangeType,
): CalculateNFTRoyaltiesResult {
  const royaltyAmount: number = royaltyPercentage ? (royaltyPercentage / 100) * amount : 0;
  const royaltyAmountString: string = formatAmount(royaltyAmount);
  const nftSellerNetAmount: number = amount;
  // : parseFloat(
  //     (amount - parseFloat(royaltyAmountString) - makerFee).toFixed(12),
  //   );
  const totalAmount: number =
    exchangeType === NFTOfferExchangeType.NFTForToken ? amount + royaltyAmount : amount + makerFee + royaltyAmount;
  const totalAmountString: string = formatAmount(totalAmount);

  return {
    royaltyAmount,
    royaltyAmountString,
    nftSellerNetAmount,
    totalAmount,
    totalAmountString,
  };
}
