Audit Report

## Title
SNS Swap `refresh_buyer_token_e8s` Griefing Locks Victim ICP and Blocks New Participation - (File: rs/sns/swap/src/swap.rs)

## Summary
An existing SNS swap participant can top up their participation by a small amount to reduce `available_direct_participation_e8s` below `min_participant_icp_e8s`. Any subsequent new participant whose ICP is already in the swap subaccount will have their `refresh_buyer_token_e8s` call hard-rejected, locking their funds until the swap closes. The same root cause blocks `new_sale_ticket` via `compute_participation_increment`.

## Finding Description
In `refresh_buyer_token_e8s`, the available capacity is fetched at line 1177 and used to cap the accepted increment: [1](#0-0) [2](#0-1) 

After capping, the per-participant maximum is applied, and then a hard rejection fires if the resulting balance is below the minimum: [3](#0-2) 

For a **new participant** (`old_amount_icp_e8s = 0`), `new_balance_e8s = actual_increment_e8s = min(remaining_capacity, requested)`. If an attacker has already reduced `remaining_capacity` below `min_participant_icp_e8s`, then `new_balance_e8s < min_participant_icp_e8s` unconditionally, and the call returns an error.

For an **existing participant** (`old_amount_icp_e8s >= min_participant_icp_e8s`), `new_balance_e8s >= old_amount >= min_participant_icp_e8s`, so the check always passes regardless of remaining capacity. This asymmetry is the root cause.

The code itself acknowledges the ICP-locking consequence: [4](#0-3) 

The identical issue exists in `compute_participation_increment` called from `new_sale_ticket`: [5](#0-4) [6](#0-5) 

When `user_participation = 0` and `max_available_increment < min_user_participation`, the condition at line 3240 returns `Err((0, 0))`, blocking ticket creation for new participants while existing ones (with `user_participation > 0`) are unaffected.

## Impact Explanation
This matches the **High** bounty impact: "Significant SNS security impact with concrete user or protocol harm." Victims who have already transferred ICP to the swap subaccount cannot reclaim it until the swap closes (committed or aborted) via `error_refund_icp`. The attacker can then absorb the residual capacity themselves, monopolizing the remaining allocation and reducing the number of unique token holders — directly undermining the decentralization goal of the SNS swap. The harm is concrete and financial (temporary ICP lockup), not merely theoretical.

## Likelihood Explanation
Moderate. The attacker must already be a registered participant (low barrier — anyone can join early with the minimum amount) and must have headroom below `max_participant_icp_e8s` to top up. No privileged role, admin key, or governance majority is required. Swap state is publicly queryable, so the attacker can observe exact remaining capacity before acting. The IC's deterministic per-subnet message ordering allows reliable sequencing of the attacker's top-up before a victim's `refresh_buyer_token_e8s` call.

## Recommendation
Waive the minimum enforcement when the remaining capacity is itself already below `min_participant_icp_e8s`. Replace the check at lines 1241–1246 with:

```rust
// Only enforce the minimum when there is enough remaining capacity
// to satisfy it. If remaining capacity is already below the minimum,
// accept the partial increment so the victim is not locked out.
if new_balance_e8s < params.min_participant_icp_e8s
    && max_increment_e8s >= params.min_participant_icp_e8s
{
    return Err(format!(
        "Rejecting participation of effective amount {}; minimum required to participate: {}",
        new_balance_e8s, params.min_participant_icp_e8s
    ));
}
```

Apply the analogous fix to `compute_participation_increment` at line 3240: only block a new participant when the shortfall is not caused by the global cap already being below the minimum.

## Proof of Concept
**Parameters:** `max_direct_participation = 100 ICP`, `min_participant_icp_e8s = 2 ICP`, `max_participant_icp_e8s = 40 ICP`.

1. Bob participates with 38 ICP. Remaining = 62 ICP.
2. Other participants fill to 98 ICP. Remaining = 2 ICP.
3. Alice transfers 5 ICP to the swap canister's subaccount for her principal.
4. Bob tops up by 1 ICP → total = 99 ICP, remaining = 1 ICP. Bob's call succeeds because `38 + 1 = 39 >= min_participant_icp_e8s`.
5. Alice calls `refresh_buyer_token_e8s`:
   - `e8s = 5 ICP` passes the raw-balance check at line 1202 (`5 >= 2`).
   - `max_increment_e8s = 1 ICP`.
   - `actual_increment_e8s = min(1, 5) = 1 ICP`.
   - `new_balance_e8s = 0 + 1 = 1 ICP`.
   - Hard error at line 1241: `1 < 2` → rejected.
6. Alice's 5 ICP is locked in the swap subaccount until the swap closes.
7. Bob tops up by 1 more ICP, consuming the last remaining capacity.

A deterministic unit test in `rs/sns/swap/tests/swap.rs` can reproduce this by setting up the described state, calling `refresh_buyer_token_e8s` for Alice, and asserting the `Err` return while confirming Alice's subaccount balance remains non-zero.
