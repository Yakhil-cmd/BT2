### Title
Malicious CKD Participant Sends Unverified Share to Corrupt Aggregated Output, Permanently Denying Key Derivation - (File: src/confidential_key_derivation/protocol.rs)

### Summary

The CKD coordinator in `do_ckd_coordinator` aggregates each participant's `(Y_i, C_i)` contribution by simple addition with no proof of correctness. A single malicious participant can send an arbitrary, crafted `CKDOutput` that shifts the aggregated ciphertext `(Y, C)` by an attacker-controlled delta. Because the coordinator performs no per-share verification and the protocol provides no mechanism to identify the corrupt contributor, the app's final BLS verification will fail and the derived key is permanently undeliverable for that request.

### Finding Description

**Root cause — no proof of correctness on received CKD shares**

`do_ckd_coordinator` collects each participant's `CKDOutput` and unconditionally adds the two curve points together:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
``` [1](#0-0) 

There is no zero-knowledge proof, consistency check, or any other mechanism that forces a participant to prove that the submitted `(λ_i · Y_i, λ_i · C_i)` was computed from their actual secret share `x_i` and a freshly sampled `y_i`. The coordinator simply trusts every received value.

**Protocol specification**

According to the CKD specification, each node `i` must compute:

- `y_i ←$ Z_q`, `Y_i = y_i · G₁`
- `S_i = x_i · H(pk, app_id)`
- `C_i = S_i + y_i · A`

and send `(λ_i · Y_i, λ_i · C_i)` to the coordinator. [2](#0-1) 

The coordinator then forms `Y = Σ λ_i · Y_i` and `C = Σ λ_i · C_i`, and the app recovers `sig = C + (−a) · Y = msk · H(pk, app_id)`, verifying it as a BLS signature. [3](#0-2) 

**Exploit path**

1. Malicious participant `P_m` sends `(λ_m · Y_m + δ_Y, λ_m · C_m + δ_C)` with arbitrary non-zero `(δ_Y, δ_C)`.
2. The coordinator adds these values without any check.
3. The aggregated output becomes `Y' = Y + δ_Y`, `C' = C + δ_C`.
4. The app computes `sig' = C' + (−a) · Y' = msk · H(pk, app_id) + δ_C − a · δ_Y`.
5. Since `a` (the app's private key) is unknown to `P_m`, the attacker cannot satisfy `δ_C = a · δ_Y`, so `sig'` is not a valid BLS signature for `H(pk, app_id)`.
6. The app's verification fails; the derived key is not delivered.
7. Because no per-share proof exists, the coordinator cannot identify `P_m` and cannot exclude them from a retry. Every subsequent CKD request involving `P_m` can be corrupted in the same way.

**Analogy to the reference report**

The external report shows that `CoreSaltyFeed` uses an unverified spot price that any participant can shift, causing the price aggregator to produce an incorrect average. Here, the CKD coordinator uses unverified participant curve-point contributions that any participant can shift, causing the aggregated ciphertext to be incorrect. In both cases the root cause is the same: a critical aggregation step accepts attacker-controlled inputs without any proof of correctness, and the downstream consumer (liquidation engine / app key derivation) is harmed.

### Impact Explanation

**High — Corruption of CKD outputs / Permanent denial of CKD for honest parties.**

A single malicious MPC participant can, at zero cryptographic cost, corrupt every CKD request that includes them. The app receives an `(Y, C)` pair whose BLS verification always fails, so the derived application key is never produced. Because the coordinator has no per-share proofs, it cannot identify or exclude the malicious node; the denial persists for every retry that includes the same participant set.

This matches the allowed impact: *"High: Corruption of DKG, reshare, refresh, presign, sign, or CKD outputs so honest parties accept inconsistent public keys, participant sets, transcripts, or unusable cryptographic outputs"* and *"High: Permanent denial of … CKD for honest parties under valid protocol inputs and documented trust assumptions."*

### Likelihood Explanation

Any participant in the CKD protocol can execute this attack. No privileged access, leaked keys, or cryptographic break is required — the attacker only needs to send two arbitrary curve points instead of the correctly computed ones. The attack is deterministic, repeatable, and undetectable by the coordinator with the current protocol design.

### Recommendation

Add a zero-knowledge proof of correct share computation to each participant's CKD message. Specifically, each participant should prove in zero knowledge that their submitted `(λ_i · Y_i, λ_i · C_i)` satisfies the relation `C_i = x_i · H(pk, app_id) + y_i · A` and `Y_i = y_i · G₁` for the same `y_i`, without revealing `x_i` or `y_i`. A standard Sigma protocol (e.g., a Chaum–Pedersen proof of discrete-log equality) over the BLS12-381 curve suffices. The coordinator must verify all proofs before aggregating, and reject (and identify) any participant whose proof fails.

### Proof of Concept

```
Setup: n MPC participants run CKD for app_id with app public key A.
       Participant P_m is malicious.

Step 1: P_m computes the correct (λ_m·Y_m, λ_m·C_m) but instead sends
        (λ_m·Y_m + G₁, λ_m·C_m + G₁)   // δ_Y = δ_C = G₁ (the generator)

Step 2: Coordinator executes do_ckd_coordinator():
        norm_big_y += participant_output.big_y()  // adds λ_m·Y_m + G₁
        norm_big_c += participant_output.big_c()  // adds λ_m·C_m + G₁
        // No proof check; no error raised.

Step 3: Coordinator returns CKDOutput { Y' = Y + G₁, C' = C + G₁ }.

Step 4: App computes sig' = C' + (−a)·Y'
                          = msk·H(pk,app_id) + G₁ − a·G₁
                          = msk·H(pk,app_id) + (1−a)·G₁

Step 5: App verifies: e(sig', G₂) =? e(H(pk,app_id), pk)
        This fails because sig' ≠ msk·H(pk,app_id).

Step 6: App cannot derive its key. P_m repeats for every retry.
        Coordinator has no proof to identify P_m.
```
