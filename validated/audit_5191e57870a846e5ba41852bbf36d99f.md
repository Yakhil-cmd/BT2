### Title
`setPoolAdminFeeDestination` Does Not Flush Accrued Fees to Old Destination Before Rotation — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary

When the pool admin changes the admin fee destination via `setPoolAdminFeeDestination`, all fees that accrued under the old destination are silently redirected to the new destination on the next `collectPoolFees` call. The old destination receives nothing for the period it was active.

### Finding Description

`setPoolAdminFees` and `setPoolProtocolFee` both call `collectFees` with the current `poolAdminFeeDestination` **before** updating any configuration, ensuring the old destination receives all fees earned up to that point: [1](#0-0) 

`setPoolAdminFeeDestination` performs no such flush. It updates `poolAdminFeeDestination[pool]` atomically without first draining the pool's accrued spread surplus and `notionalFeeToken{0,1}Scaled` balances to the old address: [2](#0-1) 

After the update, the next call to `collectPoolFees` (permissionless) or any fee-triggering admin action passes `poolAdminFeeDestination[pool]` — now the **new** address — to `collectFees`: [3](#0-2) 

Inside `collectFees`, the entire spread surplus and all accumulated notional fees are transferred to `adminFeeDestination_` in one shot: [4](#0-3) 

There is no per-epoch or per-address accounting; the pool holds one undivided fee balance. All fees earned before the destination change are therefore delivered to the new destination.

### Impact Explanation

The old `adminFeeDestination` — which may be a separate treasury, revenue-sharing partner, or DAO contract distinct from the pool admin key — permanently loses every token of admin fees that accrued during its tenure. The new destination receives a windfall it did not earn. The magnitude equals the full spread surplus plus `notionalFeeToken0Scaled` / `notionalFeeToken1Scaled` at the time of the next collection, which can be arbitrarily large depending on trading volume and how long fees went uncollected.

### Likelihood Explanation

The trigger is a routine, permissioned pool-admin action (`setPoolAdminFeeDestination`) that the protocol explicitly supports and documents as a treasury-coordination operation. Any pool admin rotation of their fee treasury — a normal operational event — silently executes this loss. No exploit setup is required beyond calling the function.

### Recommendation

Mirror the pattern used in `setPoolAdminFees` and `setPoolProtocolFee`: call `collectFees` with the **current** `poolAdminFeeDestination[pool]` before overwriting it.

```solidity
function setPoolAdminFeeDestination(address pool, address newAdminFeeDestination)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (newAdminFeeDestination == address(0)) revert InvalidAdminFeeDestination();

    // Flush accrued fees to the OLD destination before rotating.
    PoolFeeConfig memory c = poolFeeConfig[pool];
    IMetricOmmPoolCollectFees(pool).collectFees(
        c.protocolSpreadFeeE6,
        c.adminSpreadFeeE6,
        c.protocolNotionalFeeE8,
        c.adminNotionalFeeE8,
        poolAdminFeeDestination[pool]   // old destination
    );

    poolAdminFeeDestination[pool] = newAdminFeeDestination;
    emit PoolAdminFeeDestinationUpdated(pool, newAdminFeeDestination);
}
```

### Proof of Concept

1. Pool is deployed with `adminFeeDestination = oldTreasury`. Trading occurs; spread surplus and notional fees accumulate in the pool.
2. Pool admin calls `setPoolAdminFeeDestination(pool, newTreasury)`. No `collectFees` is triggered; `poolAdminFeeDestination[pool]` is overwritten.
3. Anyone calls `collectPoolFees(pool)`. The factory reads `poolAdminFeeDestination[pool]` → `newTreasury` and passes it to `collectFees`.
4. `collectFees` transfers the entire accumulated admin share to `newTreasury`.
5. `oldTreasury` receives zero tokens despite having been the designated recipient throughout the entire fee-accrual period.
