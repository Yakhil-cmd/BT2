### Title
Unbounded Loop Over Pool Member Balance Trace in `calculate_rewards` Enables Permanent Freezing of Unclaimed Yield - (File: src/pool/pool.cairo)

### Summary

The `calculate_rewards` function in the Pool contract iterates over a pool member's entire balance trace without any bound. The trace grows by one entry per epoch in which the member makes a balance change (via `add_to_delegation_pool` or `exit_delegation_pool_intent`). A delegator who accumulates enough trace entries without claiming rewards will find their `claim_rewards` call permanently reverts due to gas exhaustion, permanently freezing their unclaimed yield.

### Finding Description

In `src/pool/pool.cairo`, the internal `calculate_rewards` function contains an explicitly acknowledged unbounded loop:

```cairo
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
    if pool_member_checkpoint.epoch() >= until_epoch {
        break;
    }
    let to_sigma = self.find_sigma(pool_member_checkpoint, curr_epoch: until_epoch);
    rewards += compute_rewards_rounded_down(...);
    entry_to_claim_from += 1;
}
``` [1](#0-0) 

The loop iterates over `pool_member_epoch_balance`, a `Vec`-backed trace stored in contract storage. Each iteration performs a storage read (`pool_member_trace.at(entry_to_claim_from)`) and a `find_sigma` call. [2](#0-1) 

The trace grows via `set_member_balance`, which calls `trace.insert(key: self.get_epoch_plus_k(), value: ...)`. The `insert` implementation appends a **new** checkpoint whenever the key (epoch) differs from the last entry's key — i.e., one new entry per epoch in which a balance change occurs. [3](#0-2) [4](#0-3) 

Both `add_to_delegation_pool` and `exit_delegation_pool_intent` trigger a balance change and thus a new trace entry when called in a new epoch: [5](#0-4) [6](#0-5) 

The `entry_to_claim_from` cursor is only advanced and saved when `claim_rewards` succeeds. If a delegator never calls `claim_rewards`, the cursor stays at 0 and the loop must traverse the entire trace on the next claim attempt. [7](#0-6) 

### Impact Explanation

A delegator who makes one balance change per epoch (e.g., calling `add_to_delegation_pool` with 1 wei each epoch) without claiming rewards accumulates one trace entry per epoch. After enough epochs, the gas cost of iterating the full trace in `calculate_rewards` exceeds Starknet's per-transaction gas limit. At that point, `claim_rewards` permanently reverts, and the delegator's accumulated unclaimed yield is frozen forever with no recovery path — the `entry_to_claim_from` cursor can never advance past the gas limit.

This matches the **High** impact category: **Permanent freezing of unclaimed yield**.

### Likelihood Explanation

- Any delegator (unprivileged) can trigger this by calling `add_to_delegation_pool` or `exit_delegation_pool_intent` once per epoch with a minimal amount (1 wei satisfies the `amount.is_non_zero()` check).
- The attack is self-inflicted but realistic for long-term delegators who do not claim rewards frequently.
- The developers themselves acknowledge the risk in a code comment: *"This loop is unbounded but unlikely to exceed gas limits"* — confirming awareness but no mitigation.
- The cost to reach the threshold is low: only token gas fees per epoch, with negligible token amounts. [8](#0-7) 

### Recommendation

1. **Enforce a maximum trace length per pool member** — reject `add_to_delegation_pool` and `exit_delegation_pool_intent` if the trace already has more than `MAX_TRACE_ENTRIES` unclaimed entries.
2. **Alternatively, require `claim_rewards` before balance changes** if the unclaimed trace depth exceeds a threshold, forcing the cursor to advance before new entries are appended.
3. **Or implement paginated reward claiming** — allow `claim_rewards` to process a bounded number of entries per call and save the cursor, so the full trace can be consumed over multiple transactions.

### Proof of Concept

1. Delegator calls `enter_delegation_pool` with a valid amount.
2. Each epoch, delegator calls `add_to_delegation_pool(pool_member, 1)` — this appends one new entry to `pool_member_epoch_balance` via `set_member_balance` → `trace.insert(key: current_epoch + K, ...)`.
3. Delegator never calls `claim_rewards`, so `entry_to_claim_from` remains at 0.
4. After N epochs (where N is large enough to exceed the gas limit per iteration), delegator calls `claim_rewards`.
5. `calculate_rewards` enters the loop with `entry_to_claim_from = 0` and `pool_member_trace_length = N`, performing N storage reads and N `find_sigma` calls.
6. The transaction runs out of gas and reverts.
7. The delegator's accumulated rewards are permanently inaccessible — no alternative path exists to advance `entry_to_claim_from` or drain the trace. [9](#0-8) [10](#0-9)

        fn claim_rewards(ref self: ContractState, pool_member: ContractAddress) -> Amount {
            // Asserts.
            let mut pool_member_info = self.internal_pool_member_info(:pool_member);
            let caller_address = get_caller_address();
            let reward_address = pool_member_info.reward_address;
            assert!(
                caller_address == pool_member || caller_address == reward_address,
                "{}",
                Error::POOL_CLAIM_REWARDS_FROM_UNAUTHORIZED_ADDRESS,
            );

            let until_checkpoint = self.get_current_checkpoint(:pool_member);

            // Calculate rewards and update entry_to_claim_from.
            let (mut rewards, updated_entry_to_claim_from) = self
                .calculate_rewards(
                    :pool_member,
                    from_checkpoint: pool_member_info.reward_checkpoint,
                    :until_checkpoint,
                    entry_to_claim_from: pool_member_info.entry_to_claim_from,
                );
            rewards += pool_member_info._unclaimed_rewards_from_v0;
            pool_member_info._unclaimed_rewards_from_v0 = Zero::zero();
            pool_member_info.entry_to_claim_from = updated_entry_to_claim_from;
            pool_member_info.reward_checkpoint = until_checkpoint;

            // Write the updated pool member info to storage.
            self.write_pool_member_info(:pool_member, :pool_member_info);

            // Transfer rewards to the pool member.
            let reward_token = IERC20Dispatcher { contract_address: STRK_TOKEN_ADDRESS };
            reward_token.checked_transfer(recipient: reward_address, amount: rewards.into());

            // Emit event.
            self
                .emit(
                    Events::PoolMemberRewardClaimed {
                        pool_member, reward_address, amount: rewards,
                    },
                );

            rewards
        }

    fn insert(
        self: StoragePath<Mutable<PoolMemberBalanceTrace>>, key: Epoch, value: PoolMemberBalance,
    ) -> (PoolMemberBalance, PoolMemberBalance) {
        let checkpoints = self.checkpoints;

        let len = checkpoints.len();
        if len == Zero::zero() {
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
            return (Zero::zero(), value);
        }

        // Update or append new checkpoint.
        let mut last = checkpoints[len - 1].read();
        let prev = last.value;
        if last.key == key {
            last.value = value;
            checkpoints[len - 1].write(last);
        } else {
            // Checkpoint keys must be non-decreasing.
            assert!(last.key < key, "{}", TraceErrors::UNORDERED_INSERTION);
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
        }
        (prev, value)
    }
