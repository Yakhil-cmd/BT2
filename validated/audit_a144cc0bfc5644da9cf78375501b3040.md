### Title
Unbounded Loop Over Pool Member Balance Trace in `calculate_rewards` Enables Permanent Freezing of Unclaimed Yield - (File: `src/pool/pool.cairo`)

### Summary

The `calculate_rewards` function in `src/pool/pool.cairo` iterates over the entire `pool_member_epoch_balance` trace without any upper bound on the number of iterations. A pool member can grow this trace by making balance changes across different epochs without claiming rewards. If the trace grows large enough, every call to `claim_rewards` will exceed the Starknet gas limit, permanently freezing the pool member's unclaimed yield with no recovery path.

### Finding Description

**Root cause:** `calculate_rewards` (lines 837–888) contains an explicitly unbounded loop:

```cairo
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
    if pool_member_checkpoint.epoch() >= until_epoch {
        break;
    }
    let to_sigma = self.find_sigma(pool_member_checkpoint, curr_epoch: until_epoch);
    ...
    entry_to_claim_from += 1;
}
``` [1](#0-0) 

The loop bound is `pool_member_trace_length`, which is the length of the `pool_member_epoch_balance` storage `Vec`. This trace grows by one entry each time `set_member_balance` is called with a new epoch key:

```cairo
fn set_member_balance(ref self: ContractState, pool_member: ContractAddress, amount: Amount) {
    let trace = self.pool_member_epoch_balance.entry(pool_member);
    ...
    trace.insert(key: self.get_epoch_plus_k(), value: pool_member_balance);
}
``` [2](#0-1) 

The `insert` function only deduplicates entries with the **same epoch key**; if the epoch has advanced, a new checkpoint is appended:

```cairo
if last.key == key {
    last.value = value;
    checkpoints[len - 1].write(last);
} else {
    assert!(last.key < key, "{}", TraceErrors::UNORDERED_INSERTION);
    checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
}
``` [3](#0-2) 

`set_member_balance` is called from three public entry points, all accessible by the pool member themselves:

- `enter_delegation_pool` → `set_member_balance` [4](#0-3) 
- `add_to_delegation_pool` → `increase_member_balance` → `set_member_balance` [5](#0-4) 
- `exit_delegation_pool_intent` → `set_member_balance` [6](#0-5) 

**Exploit path:**

1. Pool member enters the delegation pool in epoch E. Trace: `[(E+K, amount)]`.
2. In epoch E+1, pool member calls `add_to_delegation_pool` with a minimal amount. Trace: `[(E+K, amount), (E+1+K, amount+1)]`.
3. In epoch E+2, pool member calls `exit_delegation_pool_intent` for the minimal amount. Trace grows to 3 entries.
4. Repeat steps 2–3 each epoch, never calling `claim_rewards`.
5. After N epochs, the trace has N entries. When `claim_rewards` is finally called, the loop iterates over all N entries (each requiring a `find_sigma` storage read), eventually exceeding the block gas limit.

**`claim_rewards` calls `calculate_rewards` directly:**

**`pool_member_info_v1` (a read-only view callable by anyone) also calls `calculate_rewards`:**

```cairo

Once the trace is large enough, both `claim_rewards` and `pool_member_info_v1` will always revert with an out-of-gas error. There is no administrative function to prune the trace or reset `entry_to_claim_from`. The pool member's accumulated rewards are permanently inaccessible.

### Impact Explanation

**Permanent freezing of unclaimed yield (High).** Once the `pool_member_epoch_balance` trace grows beyond the gas-iteration threshold, the pool member's `claim_rewards` transaction will always revert. The rewards already accrued and held in the pool contract can never be transferred to the reward address. There is no recovery mechanism: no admin function prunes the trace, and the pool member cannot split the claim into smaller batches because `entry_to_claim_from` is only advanced by a successful `claim_rewards` call.

Additionally, `pool_member_info_v1` becomes permanently broken for that pool member, breaking any on-chain integrations that query it.

### Likelihood Explanation

**Low-to-medium.** The trace grows at most one entry per epoch. The number of epochs required to exceed the gas limit depends on the cost of each loop iteration (primarily the `find_sigma` storage read) and the Starknet block gas limit. With short epoch durations (e.g., hours) and a pool member who actively changes their balance each epoch without claiming, the threshold could be reached within months to years. The scenario is self-inflicted but irreversible once triggered, and the code itself acknowledges the risk with the comment "This loop is unbounded but unlikely to exceed gas limits."

### Recommendation

1. **Enforce a maximum trace length per pool member.** Reject `add_to_delegation_pool` and `exit_delegation_pool_intent` if the trace length exceeds a safe constant (e.g., 1000 entries).
2. **Alternatively, require `claim_rewards` before balance changes when the trace exceeds a threshold**, forcing `entry_to_claim_from` to advance and keeping the effective loop length bounded.
3. **Remove the "unlikely to exceed gas limits" assumption** from the code comment and replace it with an enforced invariant.

### Proof of Concept

```
Epoch 1:  pool_member calls enter_delegation_pool(amount=MIN_STAKE)
          → trace length = 1

Epoch 2:  pool_member calls add_to_delegation_pool(amount=1)
          → trace length = 2

Epoch 3:  pool_member calls exit_delegation_pool_intent(amount=1)
          → trace length = 3

... (repeat add/exit each epoch, never call claim_rewards) ...

Epoch N:  trace length = N

Epoch N+1: pool_member calls claim_rewards()
           → calculate_rewards loops N times, each calling find_sigma (storage read)
           → transaction reverts with out-of-gas
           → pool member's accumulated rewards are permanently frozen
```

The minimum cost per epoch is one `add_to_delegation_pool` call with `amount=1` (the minimum non-zero amount), requiring only a trivial token balance. The attack is fully reachable by an unprivileged pool member with no special access.
