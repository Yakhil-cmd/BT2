### Title
Unbounded Loop Over Growing Balance Trace in `calculate_rewards` Can Permanently Freeze Pool Member Unclaimed Yield - (File: src/pool/pool.cairo)

### Summary
The `calculate_rewards` function in `Pool` iterates over every entry in a pool member's `pool_member_epoch_balance` trace from `entry_to_claim_from` to the current length. Because the trace grows by one entry per epoch whenever the member modifies their balance, a pool member who interacts once per epoch for a sufficient number of epochs will accumulate a trace large enough to cause `claim_rewards` to exceed the Starknet transaction gas limit, permanently freezing their unclaimed yield.

### Finding Description
`calculate_rewards` in `src/pool/pool.cairo` contains an explicit `while` loop that iterates over the pool member's balance-change trace:

```cairo
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
    if pool_member_checkpoint.epoch() >= until_epoch {
        break;
    }
    ...
    entry_to_claim_from += 1;
}
``` [1](#0-0) 

The trace is stored as an append-only `Vec<PoolMemberBalanceCheckpoint>` in `pool_member_epoch_balance`. It grows by one entry per epoch whenever `set_member_balance` or `increase_member_balance` is called: [2](#0-1) 

These helpers are invoked from every balance-modifying public entry point: `enter_delegation_pool`, `add_to_delegation_pool`, `exit_delegation_pool_intent`, and `enter_delegation_pool_from_staking_contract`. [3](#0-2) [4](#0-3) [5](#0-4) 

The `insert` function in `PoolMemberBalanceTrace` appends a new checkpoint whenever the epoch key differs from the last entry, meaning one new entry is added per epoch of activity: [6](#0-5) 

`claim_rewards` calls `calculate_rewards` starting from the stored `entry_to_claim_from` index. Because the Vec is never pruned and `entry_to_claim_from` only advances on a successful (non-reverting) transaction, a trace that is too large to process in one transaction will cause every future `claim_rewards` call to revert, permanently locking the member's yield. [7](#0-6) 

The developers themselves acknowledge the risk in the inline comment: *"This loop is unbounded but unlikely to exceed gas limits."* [8](#0-7) 

### Impact Explanation
A pool member whose balance trace grows beyond the per-transaction gas limit will have their `claim_rewards` call revert on every attempt. Because `entry_to_claim_from` is only written on a successful transaction, no partial progress is saved. The accumulated STRK rewards become permanently unclaimable — matching the **High** impact category: *Permanent freezing of unclaimed yield*.

### Likelihood Explanation
Starknet epochs are short (on the order of hours). A pool member who calls `add_to_delegation_pool` or `exit_delegation_pool_intent` once per epoch accumulates one trace entry per epoch. Over months to years of normal protocol use, the trace grows into the thousands of entries. No special privilege is required; any ordinary delegator triggers this path through routine interactions. The protocol is designed for long-term participation, making this a realistic outcome for active, long-lived pool members.

### Recommendation
Replace the unbounded loop with a paginated reward-claiming mechanism that accepts a `max_entries` parameter and saves intermediate progress to storage, or prune already-processed entries from the trace after each successful claim. The `entry_to_claim_from` index already provides a natural cursor; the fix is to commit it after processing a bounded batch and allow multiple transactions to drain the full range.

### Proof of Concept

1. Alice enters a delegation pool at epoch 0 via `enter_delegation_pool`.
2. Each subsequent epoch, Alice calls `add_to_delegation_pool(amount: 1)`. Each call appends one new `PoolMemberBalanceCheckpoint` to `pool_member_epoch_balance[Alice]`.
3. After N epochs without claiming, `pool_member_trace_length == N`.
4. Alice (or anyone) calls `claim_rewards(pool_member: Alice)`.
5. `calculate_rewards` enters the `while` loop and must execute N iterations, each performing a storage read (`pool_member_trace.at(entry_to_claim_from)`) and a `find_sigma` call (itself a storage lookup into `cumulative_rewards_trace`).
6. For sufficiently large N (reachable within a year of normal use), the transaction exceeds the Starknet gas limit and reverts.
7. Because `entry_to_claim_from` is only written on success, every subsequent `claim_rewards` attempt also reverts.
8. Alice's accumulated STRK rewards are permanently frozen with no recovery path.

    fn insert(
        self: StoragePath<Mutable<PoolMemberBalanceTrace>>, key: Epoch, value: PoolMemberBalance,
    ) -> (PoolMemberBalance, PoolMemberBalance) {
        let checkpoints = self.checkpoints;

        let len = checkpoints.len();
        if len == Zero::zero() {
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
            return (Zero::zero(), value);
        }

        // Update or append new checkpoint.
        let mut last = checkpoints[len - 1].read();
        let prev = last.value;
        if last.key == key {
            last.value = value;
            checkpoints[len - 1].write(last);
        } else {
            // Checkpoint keys must be non-decreasing.
            assert!(last.key < key, "{}", TraceErrors::UNORDERED_INSERTION);
            checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
        }
        (prev, value)
    }
