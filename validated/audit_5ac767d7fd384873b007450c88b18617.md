### Title
Pool Join/Change Uses Re-Fetched `targetPuzzleHash` After User Review, Enabling Reward Redirection — (File: `packages/gui/src/components/plotNFT/select/PlotNFTSelectPool.tsx`)

### Summary
When a user joins or changes a pool, the GUI displays pool details (including `targetPuzzleHash`) fetched from the pool's `/pool_info` endpoint for review. However, at the moment the user clicks "Create"/"Change", `prepareSubmitData` makes a **second, independent** HTTP request to the same endpoint and uses the freshly-returned `targetPuzzleHash` in the `pw_join_pool` transaction — without showing the user what was actually used. A malicious pool operator can serve a legitimate `targetPuzzleHash` during the review phase and swap it to an attacker-controlled address at submit time, silently redirecting all of the user's future farming rewards.

### Finding Description
`PlotNFTSelectBase` calls `usePoolInfo(poolUrl)` reactively as the user types, fetching `/pool_info` and rendering the result in the "Verify Pool Details" card — including `targetPuzzleHash`. [1](#0-0) [2](#0-1) 

When the user clicks submit, `handleSubmit` calls `prepareSubmitData`, which issues a **new** call to `getPoolInfo` and uses whatever `targetPuzzleHash` and `relativeLockHeight` are returned at that moment: [3](#0-2) 

`getPoolInfo` is a live HTTP fetch to the pool server's `/pool_info` endpoint: [4](#0-3) [5](#0-4) 

The `targetPuzzleHash` returned by this second fetch is passed directly to `pw_join_pool` with no confirmation step showing the user what value was actually used: [6](#0-5) [7](#0-6) 

There is no guard that compares the `targetPuzzleHash` displayed to the user against the one used in the transaction.

### Impact Explanation
`targetPuzzleHash` is the puzzle hash that receives all pooled farming rewards. If a pool operator changes this value between the display fetch and the submit fetch, the user's Plot NFT is permanently joined to a pool that pays rewards to the attacker's address. The user has no indication anything went wrong — the join transaction succeeds normally. All future block rewards from that Plot NFT flow to the attacker until the user manually detects the discrepancy and pays another fee to leave the pool.

This is a **Critical** impact: unauthorized accounting change affecting pooled farming rewards (XCH).

### Likelihood Explanation
Any pool operator can execute this attack by serving a legitimate `targetPuzzleHash` for the preview request and a different one for the submit-time request. The window between the two fetches is the time the user spends reviewing the details and clicking the button — typically several seconds, which is ample time for a server-side swap. No special access to the user's machine or keys is required; the attacker only needs to control the pool's HTTP endpoint.

### Recommendation
Lock the pool parameters at review time and reuse them at submit time instead of re-fetching. Concretely:

1. In `PlotNFTSelectBase`, store the fetched `poolInfo` (including `targetPuzzleHash` and `relativeLockHeight`) in form state when the preview loads successfully.
2. In `prepareSubmitData`, read `targetPuzzleHash` and `relativeLockHeight` from that stored form state rather than calling `getPoolInfo` again.
3. Optionally, add a final confirmation dialog that explicitly displays the `targetPuzzleHash` that will be committed to the blockchain before calling `pw_join_pool`.

### Proof of Concept
1. Attacker operates a pool at `https://evil-pool.example`.
2. The `/pool_info` endpoint is configured to return `targetPuzzleHash = LEGITIMATE_HASH` for the first request per IP/session, then `targetPuzzleHash = ATTACKER_HASH` for subsequent requests.
3. User opens "Join a Pool" / "Change Pool", enters `https://evil-pool.example`.
4. `usePoolInfo` fires → preview shows `LEGITIMATE_HASH` → user is satisfied and clicks "Create"/"Change".
5. `prepareSubmitData` fires `getPoolInfo` again → pool returns `ATTACKER_HASH`.
6. `pw_join_pool` is called with `targetPuzzlehash = ATTACKER_HASH`.
7. All farming rewards from the user's Plot NFT are paid to the attacker's address indefinitely.

```typescript
async function prepareSubmitData(data: FormData): SubmitData {
  const { self, fee, poolUrl } = data;
  const initialTargetState = {
    state: self ? 'SELF_POOLING' : 'FARMING_TO_POOL',
  };

  if (!self && poolUrl) {
    const normalizedPoolUrl = normalizeUrl(poolUrl);
    const { targetPuzzleHash, relativeLockHeight } = await getPoolInfo(normalizedPoolUrl);
    if (!targetPuzzleHash) {
      throw new Error(t`Pool does not provide targetPuzzleHash.`);
    }
    if (relativeLockHeight === undefined) {
      throw new Error(t`Pool does not provide relativeLockHeight.`);
    }

    initialTargetState.poolUrl = normalizedPoolUrl;
    initialTargetState.targetPuzzleHash = targetPuzzleHash;
    initialTargetState.relativeLockHeight = relativeLockHeight;
  }

  const feeMojos = chiaToMojo(fee || '0');

  return {
    fee: feeMojos,
    initialTargetState,
  };
}
```
