### Title
Blob Fee Deducted from Sender but Never Paid to Operator in ZK STF — (`basic_bootloader/src/bootloader/transaction_flow/zk/mod.rs`)

---

### Summary

In `ZkTransactionFlow::refund_and_commit_fee`, blob fees (`fee_for_blob_gas`) are included in `fee_to_prepay` and fully deducted from the transaction sender at precharge time, but `refund_and_commit_fee` never pays the blob fee portion to the operator (coinbase). The blob fee tokens are effectively destroyed: the sender is charged, no one receives the funds, and the operator is systematically underpaid for L1 data-availability costs.

---

### Finding Description

**Step 1 — Blob fee is computed and added to `fee_to_prepay` in validation:**

In `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs` (lines ~466–491):

```rust
let num_blobs = system.metadata.num_blobs();
// NOTE: it's a special resource - not transaction gas. Will be used to charge fee only
let blob_gas_used = num_blobs as u64 * GAS_PER_BLOB;
let fee_for_blob_gas = if blob_gas_used > 0 {
    system.get_blob_base_fee_per_gas().checked_mul(U256::from(blob_gas_used))
    ...
};
let fee_to_prepay = gas_fee_amount.checked_add(fee_for_blob_gas)...;
```

`fee_to_prepay` = `gas_price × gas_limit + blob_fee`. This is stored in `TxContextForPreAndPostProcessing::fee_to_prepay`.

**Step 2 — Full `fee_to_prepay` (including blob fee) is deducted from sender in `precharge_fee`:**

In `basic_bootloader/src/bootloader/transaction_flow/zk/mod.rs` (lines ~233–283):

```rust
let fee = context.fee_to_prepay;
system.io.update_account_nominal_token_balance(
    ..., &from, &fee, true /* deduct */, ...
)
```

The sender loses `gas_price × gas_limit + blob_fee`.

**Step 3 — `refund_and_commit_fee` only accounts for execution gas; blob fee is silently dropped:**

In `basic_bootloader/src/bootloader/transaction_flow/zk/mod.rs` (lines ~444–546):

```rust
// Refund: only unused execution gas
let token_to_refund = context.gas_price * U256::from(context.tx_gas_limit - context.gas_used);

// Operator payment: only execution gas used
let token_to_pay_operator = U256::from(context.gas_used).checked_mul(gas_price_for_operator)?;
```

Neither `token_to_refund` nor `token_to_pay_operator` includes the blob fee. The blob fee is neither returned to the sender nor forwarded to the coinbase.

**Token accounting summary (without `burn_base_fee`):**

| Party | Change |
|---|---|
| Sender | `−(gas_price × gas_used + blob_fee)` |
| Operator | `+(gas_price × gas_used)` |
| Net | `−blob_fee` (destroyed) |

The `blob_fee` tokens are permanently destroyed on every ZK transaction executed in a block that uses blobs.

---

### Impact Explanation

The operator/coinbase receives less than the total fees paid by users. In ZKsync, the operator pays for L1 data availability by posting blobs to Ethereum. The blob fee charged to users is the mechanism intended to compensate the operator for this cost. Because `refund_and_commit_fee` never forwards the blob fee to the coinbase, the operator is systematically underpaid by exactly `blob_base_fee_per_gas × num_blobs × GAS_PER_BLOB` per transaction in every blob-carrying block. The tokens are destroyed rather than redistributed, constituting a direct resource accounting loss.

---

### Likelihood Explanation

ZKsync uses EIP-4844 blobs for data availability in production. Any block that posts blobs (`num_blobs > 0`) causes every ZK L2 transaction in that block to trigger this path. No special attacker action is required — any unprivileged user submitting a standard ZK transaction in a blob-carrying block is sufficient. This is a normal, high-frequency execution path.

---

### Recommendation

In `ZkTransactionFlow::refund_and_commit_fee` (`basic_bootloader/src/bootloader/transaction_flow/zk/mod.rs`), add the blob fee to the operator payment:

```rust
// Retrieve blob_fee from context (blob_gas_used × blob_base_fee_per_gas, stored at validation time)
let blob_fee = context.blob_fee; // add this field to TxContextForPreAndPostProcessing
