### Title
Accept Button Not Disabled When Offer Contains Unknown Requested CATs — (`packages/gui/src/components/offers2/OfferBuilderViewer.tsx`)

---

### Summary

`OfferBuilderViewer` computes `disableAccept` without including `missingRequestedCATs`. When an offer's "requested" side (tokens the taker will **receive**) contains CAT asset IDs not present in the taker's wallet, a warning is displayed but the Accept button remains fully clickable. A taker can therefore accept an offer that sends their XCH or NFTs in exchange for an attacker-controlled, worthless unknown token.

---

### Finding Description

`offerToOfferBuilderData` maps the raw offer summary with an **inverted** perspective for the viewer:

- `offerSummary.requested` (what the maker wants from the taker) → `computedOfferBuilderData.offered` (what the taker **sends**)
- `offerSummary.offered` (what the maker gives to the taker) → `computedOfferBuilderData.requested` (what the taker **receives**)

This is confirmed by the comment in the function: [1](#0-0) 

And the mapping itself — `offerSummary.requested` CATs are pushed into `offeredTokens`: [2](#0-1) 

While `offerSummary.offered` CATs are pushed into `requestedTokens`: [3](#0-2) 

Back in `OfferBuilderViewer`, both unknown-CAT sets are computed: [4](#0-3) 

But `disableAccept` only gates on `missingOfferedCATs` (unknown CATs the taker must **send**): [5](#0-4) 

`missingRequestedCATs` (unknown CATs the taker will **receive**) is absent from this expression. The Accept button is wired directly to `disableAccept`: [6](#0-5) 

A warning is rendered when `missingRequestedCATs` is true: [7](#0-6) 

But this warning is purely informational — it does not disable the button, does not open a blocking confirmation dialog, and does not require any explicit user acknowledgment before `handleAcceptOffer` can be invoked.

---

### Impact Explanation

An attacker publishes an offer file or shares an offer string (via QR, file, WalletConnect, or direct link) structured as:

- **Maker offers**: attacker-minted unknown CAT (zero real value, asset ID not in any wallet)
- **Maker requests**: taker's XCH or NFT

When the taker opens this offer in `OfferBuilderViewer`:
1. `requestedUnknownCATs` is non-empty → `missingRequestedCATs = true`
2. A soft warning appears, but `disableAccept` remains `false`
3. The taker clicks **Accept Offer** → `handleAcceptOffer` → `handleSubmit` → `acceptOffer` RPC call executes
4. The taker's XCH or NFT is transferred to the maker in exchange for a worthless unknown token

This is a **High** impact: bypass of offer confirmation with direct asset loss (XCH, CAT, NFT).

---

### Likelihood Explanation

- Offer files are a standard, unprivileged attacker entry point — no wallet access or key material required.
- The warning text ("Be sure to verify that the asset IDs…") is easy to overlook, especially for non-technical users.
- The asymmetry with `missingOfferedCATs` (which correctly blocks acceptance) makes this inconsistency non-obvious to users who may assume both directions are equally protected.

---

### Recommendation

Add `missingRequestedCATs` to the `disableAccept` expression, or — preferably — replace the soft warning with a blocking confirmation dialog that requires explicit user acknowledgment before the Accept button becomes active:

```ts
// Current (line 194):
const disableAccept = missingOfferedCATs || showInvalid || isExpired;

// Fixed:
const disableAccept = missingOfferedCATs || missingRequestedCATs || showInvalid || isExpired;
```

A stronger mitigation would be a modal confirmation (similar to the DataLayer offer path's `ConfirmDialog`) that forces the user to explicitly acknowledge the unknown-token risk before proceeding.

---

### Proof of Concept

Render `OfferBuilderViewer` with a mocked `offerSummary` where `offered` contains an unknown CAT asset ID (not present in the mocked `wallets` list) and `requested` contains XCH. Assert that:

1. `missingRequestedCATs` evaluates to `true`
2. The **Accept Offer** `ButtonLoading` has `disabled={false}` (i.e., it is clickable)
3. No blocking dialog is shown before `handleSubmit` fires

This directly demonstrates that a taker can complete acceptance of an offer that gives away XCH in exchange for an unrecognized token, with only a dismissible warning and no hard gate.
