### Title
Gateway `skip_stateful_validations` bypasses `__validate__` for invoke nonce=1 based on any pending pool transaction, not only `deploy_account` — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

`skip_stateful_validations` skips the `__validate__` entry-point (the account's signature check) for an invoke transaction with nonce=1 whenever `account_tx_in_pool_or_recent_block` returns `true`. That helper returns `true` for **any** transaction type in the pool from the target address, not only `deploy_account`. An unprivileged attacker can therefore submit an invoke transaction with a completely arbitrary signature for any account that already has a pending nonce=0 transaction in the mempool, and the gateway will admit it without running `__validate__`.

### Finding Description

`skip_stateful_validations` (lines 429–461) contains the following guard:

```rust
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
    return mempool_client
        .account_tx_in_pool_or_recent_block(tx.sender_address())
        ...
}
``` [1](#0-0) 

When this returns `true`, `skip_validate` is set to `true` and `run_validate_entry_point` is called with `ExecutionFlags { validate: false, … }`, so the `__validate__` entry point is never executed. [2](#0-1) 

The code comment justifying the check reads:

> "It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."

`account_tx_in_pool_or_recent_block` is implemented as:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [3](#0-2) 

`tx_pool.contains_account` returns `true` if the address has **any** transaction in the pool, regardless of type: [4](#0-3) 

The invariant assumed by the comment is therefore broken: a deployed account with nonce=0 in state that has a regular invoke(nonce=0) in the pool satisfies `account_tx_in_pool_or_recent_block` just as well as one with a `deploy_account` in the pool.

The nonce check in `validate_nonce` allows nonce=1 when `account_nonce=0` because `max_allowed_nonce_gap` defaults to 200 in production: [5](#0-4) [6](#0-5) 

`validate_by_mempool` only checks nonce ordering and fee escalation — it does not verify signatures: [7](#0-6) 

### Impact Explanation

The gateway admits an invoke transaction with an arbitrary/invalid signature to the mempool without executing `__validate__`. This is a direct violation of the admission invariant: **every invoke transaction must have its signature verified by the account contract before being accepted**. The corrupted value is the admission decision itself — the gateway returns success for a transaction that should have been rejected with `ValidateFailure`.

At execution time the batcher will run `__validate__` and the transaction will revert, but the invalid transaction was already admitted to the mempool, consuming mempool capacity and sequencer execution resources. An attacker can target any account that has a pending nonce=0 transaction in the mempool (a common state for newly deployed accounts), flooding the mempool with invalid-signature nonce=1 transactions for those accounts.

### Likelihood Explanation

Any account that is deployed and has submitted its first transaction (nonce=0) is a valid target. This is a routine state for any active account. The attacker needs no privileged access — only the target account's address and the knowledge that it has a pending nonce=0 transaction in the mempool (observable via public RPC). The attack requires submitting a single transaction per target.

### Recommendation

In `skip_stateful_validations`, replace the generic `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a `deploy_account` transaction exists in the mempool for the given address. The mempool should expose a dedicated `has_deploy_account_in_pool(address)` query, or the transaction type should be stored alongside the pool entry so the gateway can distinguish `deploy_account` from other transaction types when deciding whether to skip `__validate__`.

### Proof of Concept

**Precondition:** Account X is deployed (has contract code) with `account_nonce = 0` in state.

1. Legitimate user submits `invoke(sender=X, nonce=0, valid_signature)` → gateway runs `__validate__`, signature passes, tx admitted to mempool. Now `tx_pool.contains_account(X) == true`.

2. Attacker submits `invoke(sender=X, nonce=1, signature=[0xdeadbeef, 0xdeadbeef])`:
   - `validate_nonce`: `0 <= 1 <= 0+200` → **passes**
   - `validate_by_mempool` → checks nonce ordering only → **passes**
   - `skip_stateful_validations`: `tx.nonce()==1 && account_nonce==0 && account_tx_in_pool_or_recent_block(X)==true` → returns `true`
   - `run_validate_entry_point` called with `validate=false` → `__validate__` **skipped**
   - Gateway returns success; invalid-signature tx is admitted to mempool.

3. Batcher later executes the invalid tx: `__validate__` runs, signature check fails, tx reverts. But the invalid tx was admitted to the mempool and consumed sequencer resources. [8](#0-7) [3](#0-2) [9](#0-8)

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
