### Title
Transaction-Level `da_gas` Propagated Unchanged to All Inner Calls in Simulation Trace — (`crates/apollo_rpc_execution/src/objects.rs`)

### Summary

`FunctionInvocation::try_from` receives the transaction-level `receipt.da_gas` and passes it unchanged, by value, to every inner call recursively. Every node in the call tree — outer call and all inner calls at all depths — ends up with `da_gas_consumed` equal to the full transaction DA gas total. Any unprivileged user submitting an invoke transaction with inner calls triggers this, and `starknet_simulateTransactions` / `starknet_traceTransaction` return authoritative-looking wrong values.

### Finding Description

In `InvokeTransactionTrace::try_from` (and the equivalent `DeployAccountTransactionTrace` and `L1HandlerTransactionTrace` converters), the transaction-level `receipt.da_gas` is passed as the `gas_vector` argument to `FunctionInvocation::try_from`: [1](#0-0) 

Inside `FunctionInvocation::try_from`, that same `gas_vector` is forwarded unchanged to every element of `inner_calls`: [2](#0-1) 

The recursion is unbounded in depth — each level passes the same `gas_vector` down. At the leaf of the recursion, `vm_resources_to_execution_resources` writes that value directly into `da_gas_consumed`: [3](#0-2) 

DA gas is a transaction-level quantity computed from aggregate state changes (storage writes, class hash updates, nonce updates, etc.) across the whole transaction. It is not tracked per `CallInfo` — `CallInfo.resources` only carries `vm_resources`. There is no per-call DA gas field available to substitute. The result is that every `FunctionInvocation` node in the returned trace, regardless of depth, carries the same `da_gas_consumed` value equal to the transaction total.

### Impact Explanation

`starknet_simulateTransactions` and `starknet_traceTransaction` return a call tree where every inner `FunctionInvocation.execution_resources.da_gas_consumed` is inflated to the transaction total. Tooling, wallets, and developers that consume these fields to attribute per-call DA costs, build fee-estimation heuristics, or audit contract behavior receive authoritative-looking wrong data. The concrete corrupted value is: inner call `da_gas_consumed` = transaction `receipt.da_gas` instead of the correct per-call amount (which should be zero or a fraction of the total).

This falls squarely within: **High — RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value.**

### Likelihood Explanation

Any invoke transaction with at least one inner call triggers this. No special privileges are required. The condition is met by the vast majority of real-world transactions (ERC-20 transfers, DEX swaps, multicalls, etc.).

### Recommendation

DA gas is not tracked per call in `CallInfo`. The correct fix is to **not propagate the transaction-level `da_gas` into inner calls at all**. Inner calls should receive `GasVector::default()` (all zeros) for `da_gas_consumed`, since DA gas is a transaction-level aggregate. Only the top-level `FunctionInvocation` (the direct child of the trace root) should carry the transaction `receipt.da_gas`, and even that attribution is approximate. Alternatively, if per-call DA gas attribution is desired in the future, it must be computed and stored in `CallInfo` during blockifier execution, not retrofitted from the transaction receipt.

### Proof of Concept

Build a Rust unit test in `crates/apollo_rpc_execution/src/objects.rs`:

```rust
#[test]
fn test_da_gas_not_propagated_to_inner_calls() {
    // Construct a CallInfo with one inner call.
    // Outer call: da_gas will come from receipt (nonzero).
    // Inner call: should have zero da_gas_consumed.
    let inner_call_info = CallInfo {
        call: CallEntryPoint {
            class_hash: Some(ClassHash::default()),
            ..Default::default()
        },
        ..Default::default()
    };
    let outer_call_info = CallInfo {
        call: CallEntryPoint {
            class_hash: Some(ClassHash::default()),
            ..Default::default()
        },
        inner_calls: vec![inner_call_info],
        ..Default::default()
    };

    let tx_da_gas = GasVector {
        l1_gas: GasAmount(100),
        l1_data_gas: GasAmount(50),
        l2_gas: GasAmount(0),
    };

    let invocation = FunctionInvocation::try_from((outer_call_info, tx_da_gas)).unwrap();

    // The inner call should NOT have the transaction-level da_gas.
    // This assertion FAILS, demonstrating the bug:
    assert_eq!(
        invocation.calls[0].execution_resources.da_gas_consumed,
        GasVector::default(),
        "Inner call da_gas_consumed should be zero, not the transaction total"
    );
    // Actual value equals tx_da_gas (100, 50, 0) — the transaction total.
}
```

Running this test will show the inner call's `da_gas_consumed` equals `{l1_gas: 100, l1_data_gas: 50, l2_gas: 0}` instead of `{0, 0, 0}`, confirming the propagation bug.
