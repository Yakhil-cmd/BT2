### Title
Missing Zero-Address Validation in `change_reward_address` Enables Permanent Loss of Unclaimed Yield — (File: `src/pool/pool.cairo`, `src/staking/staking.cairo`)

---

### Summary
Both `change_reward_address` implementations validate that the new reward address is not a token address, but omit a check that it is non-zero. A staker or pool member can set their reward address to the zero address, causing all subsequent reward claims to transfer STRK tokens to address `0`, permanently destroying their accumulated yield.

---

### Finding Description

**Staking contract** — `change_reward_address` in `staking.cairo`: [1](#0-0) 

The only guard on the incoming `reward_address` parameter is `REWARD_ADDRESS_IS_TOKEN`. There is no `reward_address.is_non_zero()` assertion. The protocol already enforces a zero-address check on the *caller* via `general_prerequisites()` (which asserts `CALLER_IS_ZERO_ADDRESS`), but this protection does not extend to the *parameter* being stored.

**Pool contract** — `change_reward_address` in `pool.cairo`: [2](#0-1) 

The check here is even narrower: it only compares against `self.token_dispatcher.contract_address` (the pool's staked token, e.g., BTC). It does **not** check against `STRK_TOKEN_ADDRESS`. For BTC pools, a pool member can set their reward address to the STRK token contract address; since pool rewards are always paid in STRK, those rewards would be locked inside the STRK contract with no recovery path.

When `claim_rewards` is later invoked, the stored (zero) address is used directly as the transfer recipient: [3](#0-2) [4](#0-3) 

ERC-20 `transfer` to address `0` does not revert, so the yield is silently burned.

---

### Impact Explanation

**Permanent freezing of unclaimed yield (High).** Any staker or pool member who sets their reward address to zero will have all future reward claims permanently destroyed. The yield is irrecoverable once transferred. For BTC pool members, the same outcome occurs if the reward address is set to the STRK token contract address.

---

### Likelihood Explanation

Low-to-medium. The action requires an explicit call by the staker or pool member. However:
- Scripted or programmatic integrations may pass an uninitialized (zero) `felt252`/`ContractAddress`.
- The protocol already validates zero addresses in other contexts (e.g., `CALLER_IS_ZERO_ADDRESS` in `general_prerequisites`), making this omission an inconsistency that users may not anticipate.
- No on-chain warning or revert prevents the mistake once made.

---

### Recommendation

Add a non-zero assertion in both `change_reward_address` implementations, and in the initial `stake`/`enter_delegation_pool` entry points:

```cairo
assert!(reward_address.is_non_zero(), "{}", Error::REWARD_ADDRESS_IS_ZERO);
```

For `pool.cairo`, also check against `STRK_TOKEN_ADDRESS` (the reward token), not only the pool's staked token:

```cairo
assert!(reward_address != STRK_TOKEN_ADDRESS, "{}", GenericError::REWARD_ADDRESS_IS_TOKEN);
```

---

### Proof of Concept

1. Pool member calls `enter_delegation_pool(reward_address: valid_address, amount: X)` — succeeds.
2. Pool member calls `change_reward_address(reward_address: 0)` — **succeeds** (no zero-address check).
3. Staking contract distributes rewards to the pool via `update_rewards_from_attestation_contract`.
4. Pool member calls `claim_rewards(pool_member)` — the pool member is the caller, so the access check at line 341 passes.
5. `reward_token.checked_transfer(recipient: 0, amount: rewards)` executes, burning the yield permanently. [5](#0-4)
