### Title
Invoke Transaction with Invalid Signature Bypasses Gateway `__validate__` via `skip_stateful_validations` — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator skips the account's `__validate__` entry point for any invoke transaction with `nonce=1` when `account_nonce=0` and `account_tx_in_pool_or_recent_block` returns `true`. The guard is supposed to confirm a `deploy_account` is pending, but it accepts **any** transaction from that address in the pool — including an attacker-injected invoke with `nonce=0`. This lets an attacker submit an invoke with `nonce=1` carrying an **invalid or forged signature** for a victim's undeployed account, have it admitted to the mempool without signature verification, and — if the attacker's fee is higher — **replace the victim's legitimate invoke** via fee escalation. The batcher then executes the invalid invoke, `__validate__` fails, and the victim's freshly-deployed account is charged fees for an unauthorized transaction.

---

### Finding Description

**Invariant being violated:** Every transaction admitted to the mempool must carry a signature that passes the account's `__validate__` entry point, or must be provably unable to run `__validate__` (i.e., the account does not yet exist on-chain).

**Root cause — `skip_stateful_validations`:** [1](#0-0) 

The function returns `true` (skip validation) when:
1. The transaction is an `Invoke`.
2. `tx.nonce() == Nonce(Felt::ONE)`.
3. `account_nonce == Nonce(Felt::ZERO)` (account not yet on-chain).
4. `mempool_client.account_tx_in_pool_or_recent_block(sender)` returns `true`.

Condition 4 is implemented as: [2](#0-1) 

which delegates to: [3](#0-2) 

This returns `true` if the address appears in `staged` or `committed` nonce maps **or** has any transaction in `tx_pool`. It does **not** verify that the existing transaction is specifically a `deploy_account`.

**How the skip propagates to execution flags:** [4](#0-3) 

When `skip_validate=true`, `validate` is set to `false`. The blockifier's `perform_validations` then short-circuits before calling `__validate__`: [5](#0-4) 

So the signature is **never checked** at gateway admission time.

**Why `validate_by_mempool` does not save the invariant:** [6](#0-5) 

The mempool's `validate_tx` only checks for duplicate transaction hash and nonce-too-old: [7](#0-6) 

It does **not** check for duplicate `(address, nonce)` pairs, so an attacker's `invoke(nonce=1)` passes even when the victim's `invoke(nonce=1)` is already in the pool.

**Attack path:**

1. Victim submits `deploy_account` for address `A` (valid signature, fee `F`).
2. Attacker observes `deploy_account` in the mempool; `account_tx_in_pool_or_recent_block(A)` now returns `true`.
3. Attacker submits `invoke(nonce=1, fee=F+1)` for address `A` with an **invalid or arbitrary signature**.
4. Gateway calls `skip_stateful_validations`:
   - `tx.nonce() == 1` ✓
   - `account_nonce == 0` ✓
   - `account_tx_in_pool_or_recent_block(A) == true` ✓
5. `run_validate_entry_point` is called with `validate=false`; `__validate__` is **skipped**.
6. The attacker's invalid invoke is admitted to the mempool.
7. Because the attacker's fee is higher, fee-escalation logic replaces the victim's legitimate invoke.
8. Batcher executes `deploy_account` → account `A` is deployed.
9. Batcher executes the attacker's `invoke(nonce=1)` with `validate=true`; `__validate__` runs, signature fails, transaction reverts.
10. Account `A` is charged fees for the unauthorized transaction; the victim's original invoke never executes.

---

### Impact Explanation

**Matches:** *High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.*

- An invoke transaction carrying an invalid signature is admitted to the mempool without any signature check.
- Via fee escalation, the attacker can **displace the victim's legitimate invoke** with the invalid one.
- The victim's freshly-deployed account is charged fees for a transaction it never authorized.
- The attack is repeatable for every `deploy_account` transaction visible in the mempool.

---

### Likelihood Explanation

- `deploy_account` transactions are publicly visible in the mempool.
- No special privilege is required; any external party can submit a transaction.
- The attack window is the time between `deploy_account` entering the mempool and being committed (typically several blocks).
- The only cost to the attacker is the gas fee for the invalid invoke, which is charged to the **victim's** account, not the attacker's.

---

### Recommendation

Replace the coarse `account_tx_in_pool_or_recent_block` check with a check that specifically confirms a `deploy_account` transaction is pending for the sender address. The mempool should expose a dedicated query such as `has_pending_deploy_account(address) -> bool` that inspects transaction types, not just address presence. Alternatively, the gateway can inspect the `InternalRpcTransaction` type of the pending transaction before granting the skip.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
