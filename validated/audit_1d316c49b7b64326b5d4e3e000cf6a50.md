### Title
Missing Chain-ID and Context Binding in `EVM.validateCOAOwnershipProof` Signature Verification — (`fvm/evm/stdlib/contract.cdc`)

---

### Summary

`EVM.validateCOAOwnershipProof` verifies Cadence account key signatures over a caller-supplied `signedData` blob using the hardcoded domain separation tag `"FLOW-V0.0-user"`. This tag contains no chain ID, no contract address, and no nonce. The function also does not enforce that `signedData` encodes the `evmAddress` being proved. As a result, any valid signature produced under `"FLOW-V0.0-user"` for any purpose on any Flow network is a valid COA ownership proof, enabling cross-chain replay and cross-context signature reuse — the exact vulnerability class found in NFT Port's Factory.sol.

---

### Finding Description

`EVM.validateCOAOwnershipProof` is a public function in the EVM system contract that proves a Cadence account owns a specific COA (Cadence Owned Account). It is the backend for the `verifyCOAOwnershipProof` Cadence Arch precompile, which is in turn called by the COA's ERC-1271 `isValidSignature` implementation.

The signature verification at the core of the function is:

```cadence
let isValid = keyList.verify(
    signatureSet: signatureSet,
    signedData: signedData,
    domainSeparationTag: "FLOW-V0.0-user"
)
``` [1](#0-0) 

Three properties of this call create the vulnerability:

**1. No chain ID in the domain tag.** `"FLOW-V0.0-user"` is identical on Flow Mainnet, Testnet, Sandboxnet, Previewnet, and every other Flow network. The tag is defined as a plain string with no network-specific component. [2](#0-1) 

**2. No contract address or function context in the tag.** The tag `"FLOW-V0.0-user"` is the generic user-facing signing tag used across all Cadence applications. Any signature produced by a wallet for any Cadence dApp that uses this tag is structurally identical to a COA ownership proof signature.

**3. `signedData` is fully caller-controlled and not validated.** The function accepts arbitrary bytes as `signedData` and does not enforce that those bytes encode the `evmAddress` being proved. The code itself acknowledges this:

```cadence
/// Note: this function does not enforce that `signedData` includes `evmAddress`.
/// In principle, a signature produced for one purpose could be replayed here against
/// a different COA owned by the same Cadence account.
``` [3](#0-2) 

The call chain from EVM into this function is:

1. EVM contract calls `cadenceArch.staticcall("verifyCOAOwnershipProof(address,bytes32,bytes)", ...)` — e.g., from the COA's ERC-1271 `isValidSignature`: [4](#0-3) 

2. The Cadence Arch precompile's `proofVerifier.Run()` decodes the call and invokes `coaOwnershipProofValidator`: [5](#0-4) 

3. Which calls `EVM.validateCOAOwnershipProof` in the system contract: [6](#0-5) 

---

### Impact Explanation

**Cross-chain signature replay:** Flow Mainnet and Testnet share the same address space for many users (Cadence addresses are deterministic from the same seed). A user who signs any `signedData` blob on Testnet under `"FLOW-V0.0-user"` produces a signature that is cryptographically valid on Mainnet for the same Cadence account. An attacker who observes that Testnet signature can submit it to Mainnet's `validateCOAOwnershipProof` with the same `signedData`, and if the user has a COA on Mainnet, the proof will return `isValid: true`.

**Cross-context signature reuse / phishing:** Any Cadence wallet or dApp that asks a user to sign arbitrary bytes under `"FLOW-V0.0-user"` (the standard user tag) produces a signature that is valid as a COA ownership proof. An attacker can set up a Cadence dApp that asks a victim to sign a specific 32-byte hash — one that happens to be the `_hash` argument an EVM contract will pass to `isValidSignature` — and then replay that signature through `verifyCOAOwnershipProof` to pass ERC-1271 checks on behalf of the victim's COA.

**Concrete downstream effect:** A successful false COA ownership proof causes the COA's ERC-1271 `isValidSignature` to return `ValidERC1271Signature (0x1626ba7e)` for a hash the COA owner never intended to authorize. EVM contracts that gate asset transfers, approvals, or governance votes on ERC-1271 checks will accept the forged proof, resulting in unauthorized on-chain actions under the victim's COA identity.

---

### Likelihood Explanation

The attack is reachable by any unprivileged EVM transaction sender or Cadence transaction author. No staked node access, admin keys, or privileged capability is required. The attacker only needs:

- A valid signature from the target Cadence account under `"FLOW-V0.0-user"` over any bytes (obtainable from Testnet activity, from any other Cadence dApp, or via phishing).
- Knowledge of the target account's COA path (public capability, observable on-chain).
- The ability to submit an EVM transaction or Cadence script that calls `verifyCOAOwnershipProof` or `EVM.validateCOAOwnershipProof` directly.

The cross-chain replay vector is particularly realistic because Flow Mainnet and Testnet share address derivation, and many developers test on Testnet before deploying to Mainnet using the same keys.

---

### Recommendation

1. **Include chain ID in the domain separation tag.** Replace the hardcoded `"FLOW-V0.0-user"` with a tag that encodes the network chain ID, e.g., `"FLOW-V0.0-user-" + chainID`. This prevents cross-chain replay.

2. **Enforce `evmAddress` in `signedData`.** The function should verify that `signedData` encodes the `evmAddress` argument (e.g., as a prefix or structured field), so a signature produced for one COA cannot be replayed against another, and a signature produced for a different application cannot be repurposed here.

3. **Add a nonce or context binding.** Include a nonce or a function-specific context string in the signed payload to prevent replay across different invocations and applications.

A minimal fix for the domain tag:

```cadence
// Before:
domainSeparationTag: "FLOW-V0.0-user"

// After (pseudocode):
domainSeparationTag: "FLOW-V0.0-COAOwnershipProof-" + chainID
```

And enforce that `signedData` is structured as `evmAddress || nonce || applicationData` rather than accepting arbitrary bytes.

### Proof of Concept

**Cross-chain replay attack:**

1. On Flow Testnet, victim account `0xABCD` owns a COA at `/public/coa`. The victim signs `signedData = 0xDEADBEEF...` under `"FLOW-V0.0-user"` for a legitimate Testnet dApp. The attacker observes the signature `sig`.

2. On Flow Mainnet, `0xABCD` also exists (same address, same key) and has a COA at `/public/coa`.

3. Attacker submits to Mainnet:
```cadence
import EVM from <mainnet-address>

access(all) fun main(): EVM.ValidationResult {
    return EVM.validateCOAOwnershipProof(
        address: 0xABCD,
        path: /public/coa,
        signedData: [0xDE, 0xAD, 0xBE, 0xEF, ...],  // same bytes as Testnet
        keyIndices: [0],
        signatures: [[<sig from Testnet>]],
        evmAddress: <victim's Mainnet COA address>
    )
}
```

4. `keyList.verify(signedData, domainSeparationTag: "FLOW-V0.0-user")` succeeds because the tag and message are identical on both chains. The function returns `isValid: true`.

5. Any EVM contract on Mainnet that calls `cadenceArch.staticcall("verifyCOAOwnershipProof(...)", 0xABCD_COA, 0xDEADBEEF..., sig)` will receive `true`, granting the attacker ERC-1271 authorization under the victim's COA identity. [7](#0-6) [8](#0-7) [9](#0-8)

```text
    fun validateCOAOwnershipProof(
        address: Address,
        path: PublicPath,
        signedData: [UInt8],
        keyIndices: [UInt64],
        signatures: [[UInt8]],
        evmAddress: [UInt8; 20]
    ): ValidationResult {
        // make signature set first
        // check number of signatures matches number of key indices
        if keyIndices.length != signatures.length {
            return ValidationResult(
                isValid: false,
                problem: "EVM.validateCOAOwnershipProof(): Key indices array length doesn't match the signatures array length!"
            )
        }

        // fetch account
        let acc = getAccount(address)

        var signatureSet: [Crypto.KeyListSignature] = []
        let keyList = Crypto.KeyList()
        var keyListLength = 0
        let seenAccountKeyIndices: {Int: Int} = {}
        for signatureIndex, signature in signatures {
            // index of the key on the account
            let accountKeyIndex = Int(keyIndices[signatureIndex]!)
            // index of the key in the key list
            var keyListIndex = 0

            if !seenAccountKeyIndices.containsKey(accountKeyIndex) {
                // fetch account key with accountKeyIndex
                if let key = acc.keys.get(keyIndex: accountKeyIndex) {
                    if key.isRevoked {
                        return ValidationResult(
                            isValid: false,
                            problem: "EVM.validateCOAOwnershipProof(): Cannot validate COA ownership for Cadence account \(address). The account key at index \(accountKeyIndex) is revoked."
                        )
                    }

                    keyList.add(
                      key.publicKey,
                      hashAlgorithm: key.hashAlgorithm,
                      // normalization factor. We need to divide by 1000 because the
                      // `Crypto.KeyList.verify()` function expects the weight to be
                      // in the range [0, 1]. 1000 is the key weight threshold.
                      weight: key.weight / 1000.0,
                   )

                   keyListIndex = keyListLength
                   keyListLength = keyListLength + 1
                   seenAccountKeyIndices[accountKeyIndex] = keyListIndex
                } else {
                    return ValidationResult(
                        isValid: false,
                        problem: "EVM.validateCOAOwnershipProof(): Cannot validate COA ownership for Cadence account \(address). The key index \(accountKeyIndex) is invalid."
                    )
                }
            } else {
               // if we have already seen this accountKeyIndex, use the keyListIndex
               // that was previously assigned to it
               // `Crypto.KeyList.verify()` knows how to handle duplicate keys
               keyListIndex = seenAccountKeyIndices[accountKeyIndex]!
            }

            signatureSet.append(Crypto.KeyListSignature(
               keyIndex: keyListIndex,
               signature: signature
            ))
        }

        let isValid = keyList.verify(
            signatureSet: signatureSet,
            signedData: signedData,
            domainSeparationTag: "FLOW-V0.0-user"
        )
