### Title
Missing Lower-Bound Enforcement on `threshold` in FROST Signing Validation — (File: `src/frost/mod.rs`)

---

### Summary

`assert_sign_inputs` in `src/frost/mod.rs` validates that `threshold ≤ participants.len()` but never checks that `threshold ≥ 2`. The DKG entry point `assert_key_invariants` in `src/dkg.rs` explicitly enforces this lower bound, but the invariant is not re-enforced in the FROST signing path. A malicious coordinator or caller can supply `threshold = 0` or `threshold = 1` to the signing function, bypassing the declared minimum, causing Lagrange interpolation to execute with an invalid threshold and producing a corrupted, unusable signing output.

---

### Finding Description

`assert_key_invariants` in `src/dkg.rs` enforces both bounds on `threshold`:

```rust
if threshold > participants.len() { ... }  // upper bound
if threshold < 2 { ... }                   // lower bound (min: 2)
``` [1](#0-0) 

`assert_sign_inputs` in `src/frost/mod.rs`, which is the validation gate for FROST EdDSA and RedJubjub signing, only checks the upper bound:

```rust
// validate threshold
if threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
``` [2](#0-1) 

There is no corresponding `threshold < 2` guard. A caller supplying `threshold = 0` or `threshold = 1` passes this check silently (since `0 ≤ N` and `1 ≤ N` for any valid participant set). The same gap exists in the FROST `presign` function, which also only checks the upper bound:

```rust
if args.threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
``` [3](#0-2) 

Neither function enforces the minimum threshold of 2 that the protocol requires and that DKG guarantees.

---

### Impact Explanation

FROST signing uses `threshold` to drive Lagrange interpolation over participant shares. With `threshold = 1`, the Lagrange coefficient for a single participant collapses to 1 (the interpolation treats that participant as the sole point), so the reconstructed signing scalar is just that participant's raw share — not the actual aggregate secret. The resulting signature fails verification against the master public key produced by DKG (which used `threshold ≥ 2`). With `threshold = 0`, the interpolation set is empty, which causes a panic or produces a zero/garbage scalar.

In both cases, honest participants who complete the protocol receive a corrupted, unusable signature output. This maps to:

**High: Corruption of sign outputs so honest parties accept inconsistent or unusable cryptographic outputs.**

---

### Likelihood Explanation

The `threshold` parameter is supplied by the caller at signing time — it is not read from the `KeygenOutput` struct, which does not carry the threshold forward from DKG. Any party that constructs the signing call (including a malicious coordinator) can freely set `threshold` to any value. The missing check is a single missing `if threshold < 2` guard, identical in structure to the missing `MAX_SUPPLY` check in the Mintify report. No cryptographic break or key compromise is required; only a malformed API call is needed.

---

### Recommendation

Add the same lower-bound guard that `assert_key_invariants` already applies, in both `assert_sign_inputs` and `presign` in `src/frost/mod.rs`:

```rust
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

This mirrors the existing enforcement in `assert_key_invariants`: [4](#0-3) 

---

### Proof of Concept

1. Run DKG with 3 participants and `threshold = 2` (passes `assert_key_invariants`).
2. Call `assert_sign_inputs` with the same 3 participants but `threshold = 1`.
3. Observe: no `InitializationError` is returned — the call succeeds.
4. Proceed to FROST signing with `threshold = 1`; the coordinator collects only 1 share, computes Lagrange with a single-point set (coefficient = 1), and reconstructs a scalar equal to one participant's raw share.
5. The produced signature fails verification against the master public key, delivering a corrupted, unusable output to all honest participants.

The root cause is the absent lower-bound check at: [2](#0-1) 

compared to the enforced lower bound at: [4](#0-3)
