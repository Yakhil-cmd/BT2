Audit Report

## Title
Hot Keys Not Reset When Spawning a Neuron with a New Controller - (File: `rs/nns/governance/src/governance.rs`)

## Summary
The `spawn_neuron` function in the NNS Governance canister unconditionally copies the parent neuron's `hot_keys` to the child neuron, even when the child is assigned a different controller via `spawn.new_controller`. This allows the parent's hot key holders to retain voting authority over the child neuron without the new controller's knowledge or consent.

## Finding Description
In `spawn_neuron` (L2613), the `child_controller` is resolved from `spawn.new_controller` and can be any self-authenticating principal distinct from the parent's controller:

```rust
// L2651-2655
let child_controller = if let Some(child_controller) = &spawn.new_controller {
    *child_controller
} else {
    parent_neuron.controller()
};
```

The child neuron is then constructed at L2704-2718 with `child_controller` as its controller, but the parent's hot keys are cloned verbatim:

```rust
// L2714
.with_hot_keys(parent_neuron.hot_keys.clone())
```

There is no conditional check to clear hot keys when `child_controller != parent_neuron.controller()`. The `is_authorized_to_vote` check at L5594-5595 delegates to `is_hotkey_or_controller` (L254-255 in `neuron/types.rs`), which returns `true` for any principal in `hot_keys`. This means any hot key registered on the parent neuron can immediately cast votes using the child neuron's voting power, which now belongs to the new controller.

The same pattern exists in `split_neuron` at L2241-2249, but there the child controller is always `*caller`, so the hot key inheritance is not a cross-principal authorization issue.

## Impact Explanation
This is an unauthorized access to governance assets: a third party (hot key holder on the parent) retains the ability to vote on NNS proposals using voting power that belongs to the new controller, who never authorized them. The new controller's governance influence can be silently hijacked. This matches the **High** bounty impact: "Unauthorized access to neurons, governance assets... where exploitation requires meaningful per-target work or other constraints." The constraint here is that the attacker must own a neuron with sufficient maturity to spawn, which is a low bar.

## Likelihood Explanation
The `spawn_neuron` endpoint is callable by any neuron controller via `manage_neuron` ingress — no privileged role is required. The attacker needs only: (1) a neuron with maturity, (2) a hot key registered on it (self-controlled), and (3) a target `new_controller` to spawn to. This is directly applicable to neuron marketplaces or maturity-transfer arrangements. The new controller receives no on-chain notification of inherited hot keys and the spawned neuron starts in `Spawning` state, reducing the likelihood of immediate discovery.

## Recommendation
In `spawn_neuron`, replace the unconditional hot key copy with a conditional that clears hot keys when the controller changes:

```rust
.with_hot_keys(if child_controller == parent_neuron.controller() {
    parent_neuron.hot_keys.clone()
} else {
    vec![]
})
```

## Proof of Concept
1. Alice controls neuron N with maturity. Alice adds principal `BOB` as a hot key on N.
2. Alice calls `manage_neuron` → `Spawn { new_controller: Some(CAROL), percentage_to_spawn: 100 }`.
3. Governance executes `spawn_neuron`: `child_controller = CAROL`, child neuron C is created with `hot_keys = [BOB]` (L2704-2718).
4. BOB calls `manage_neuron` → `RegisterVote { neuron_id: C, ... }`.
5. Governance checks `is_authorized_to_vote` at L5594-5595 → `is_hotkey_or_controller` at L254-255 → `hot_keys.contains(BOB)` → `true`.
6. BOB successfully casts votes using CAROL's neuron's voting power. CAROL never authorized BOB.

A deterministic integration test can be written using PocketIC: create a neuron, add a hot key, spawn to a new controller, then assert that the hot key principal can call `RegisterVote` on the child neuron and that the call succeeds.
