### Title
Malicious Participant Can Corrupt CKD Output Without Detection - (File: src/confidential_key_derivation/protocol.rs)

### Summary
The `do_ckd_coordinator` function aggregates participant shares into the final CKD output without any cryptographic verification of correctness. A single malicious participant can send arbitrary `(big_y, big_c)` group elements, silently corrupting the derived confidential key returned to the caller.

### Finding Description
In `do_ckd_coordinator` (lines 44–57 of `src/confidential_key_derivation/protocol.rs`), the coordinator receives a `CKDOutput` from every other participant and unconditionally adds the values together:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
```

Each participant is supposed to compute their share via `compute_signature_share` (lines 148–182):

```
big_y  = y · G
big_s  = x_i · H(pk ‖ app_id)
big_c  = big_s + y · app_pk
norm_big_y = λ_i · big_y
norm_big_c = λ_i · big_c
```

However, participants transmit `(norm_big_y, norm_big_c)` as raw group elements with **no zero-knowledge proof, no commitment, and no consistency check** that the values were computed using the agreed-upon `app_id` and `app_pk`. The coordinator has no mechanism to distinguish a correctly formed share from an arbitrary pair of group elements.

This is structurally analogous to the reported `closeTradeMarketCallback` pattern: an aggregation step that accepts externally supplied values and produces a final output without validating that those values satisfy the protocol's invariants.

### Impact Explanation
Let `a` be the application secret key (`app_pk = a · G`). The correct unmasked output is:

```
S = C − a · Y = msk · H(pk ‖ app_id)
```

A malicious participant substitutes `(norm_big_y_m + Δ_Y, norm_big_c_m + Δ_C)` for any group elements `Δ_Y`, `Δ_C` of their choice. The coordinator then computes:

```
Y' = Y + Δ_Y
C' = C + Δ_C
```

The caller unmasks:

```
S' = C' − a · Y' = msk · H(pk ‖ app_id) + (Δ_C − a · Δ_Y)
```

Choosing `Δ_C ≠ a · Δ_Y` (trivially achievable since `a` is unknown to the attacker — any non-zero `Δ_C` with `Δ_Y = 0` suffices) yields `S' ≠ S`. The derived confidential key is silently wrong and unusable. This matches the allowed **High** impact: *"Corruption of CKD outputs so honest parties accept unusable cryptographic outputs."*

### Likelihood Explanation
Any single participant in the CKD session can mount this attack. No special privilege beyond holding a valid key share is required. The attack is a single-message substitution with no observable side-effect at the coordinator — the protocol completes normally and returns a `CKDOutput` that appears structurally valid. Likelihood is **Medium**: it requires a malicious insider participant, which is a documented threat in threshold protocols, but not an external attacker.

### Recommendation
Add a zero-knowledge proof of correct share formation to each participant's message. Concretely, each participant should prove in zero knowledge that:

1. `big_y = y · G` for some scalar `y`, and
2. `big_c = x_i · H(pk ‖ app_id) + y · app_pk`

using a standard Chaum–Pedersen or sigma-protocol proof over G1. The coordinator must verify all proofs before aggregating. Alternatively, a commitment-then-reveal round can be added so that the coordinator can abort if any commitment fails to open to a valid share.

### Proof of Concept

1. All `n` participants are initialized with the same `app_id`, `app_pk`, and their respective key shares.
2. Participant `m` (malicious) runs `compute_signature_share` honestly to obtain `(norm_big_y_m, norm_big_c_m)`, then replaces the message sent to the coordinator with `(norm_big_y_m, norm_big_c_m + Δ)` for any non-identity `Δ ∈ G1`.
3. The coordinator receives all messages, adds them without verification (lines 50–55), and returns `CKDOutput { big_y: Y, big_c: C + Δ }`.
4. The caller invokes `ckd_output.unmask(app_sk)` (line 54 of `mod.rs`): `result = (C + Δ) − a · Y = msk · H(pk ‖ app_id) + Δ`.
5. Since `Δ ≠ 0`, the result differs from the expected `msk · H(pk ‖ app_id)` by an attacker-controlled offset, rendering the derived key incorrect for any downstream use.
