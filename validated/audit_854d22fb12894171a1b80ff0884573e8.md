Looking at the factory's pause transition logic in `MetricOmmPoolFactory.sol`, I found a direct analog to the external bug: a wrong value is hardcoded as the target state in `protocolUnpausePool`, causing the pool to remain paused after the protocol owner explicitly unpauses it.

---

### Title
`protocolUnpausePool` Assigns Wrong Pause Level (1 Instead of 0), Leaving Pool Permanently Paused After Protocol Unpause — (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary
`MetricOmmPoolFactory.protocolUnpausePool()` calls `setPause(1)` instead of `setPause(0)`. After the protocol owner unpauses a protocol-paused pool, the pool is left in admin-paused state (level 1), not active state (level 0). Swaps remain blocked indefinitely until the pool admin separately calls `unpausePool`.

### Finding Description

The pool uses three pause levels:
- `0` = active (swaps allowed)
- `1` = paused by admin (swaps blocked)
- `2` = paused by protocol (swaps blocked)

`_checkNotPaused()` reverts for **any** non-zero level:

```solidity
// MetricOmmPool.sol
function _checkNotPaused() internal view {
    if (pauseLevel != 0) revert PoolPaused();
}
```

`protocolPausePool` correctly transitions from 0 or 1 → 2:

```solidity
// MetricOmmPoolFactory.sol
function protocolPausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 0 && cur != 1) revert InvalidPauseTransition(cur, 2);
    IMetricOmmPoolFactoryActions(pool).setPause(2);
}
```

But `protocolUnpausePool` sets the wrong target value — `1` (admin-paused) instead of `0` (active):

```solidity
// MetricOmmPoolFactory.sol
function protocolUnpausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 2) revert InvalidPauseTransition(cur, 1);
    IMetricOmmPoolFactoryActions(pool).setPause(1);  // ← BUG: should be setPause(0)
}
```

This is the direct analog to the external bug: just as `token` was assigned the value of `user` (wrong variable from the decoded tuple), here the target pause state is assigned `1` (admin-paused) instead of `0` (active) — the wrong value for the intended operation.

### Impact Explanation

After `protocolUnpausePool` is called, `pauseLevel` is `1`, not `0`. Because `_checkNotPaused()` reverts for any non-zero level, **all swaps remain blocked** even though the protocol owner has explicitly decided to unpause the pool. The pool's core swap functionality is broken. If the pool admin is unavailable or unresponsive, the pool is permanently stuck in a paused state despite the protocol's intent to restore it.

### Likelihood Explanation

This triggers every time `protocolUnpausePool` is called on a pool that was active (level 0) before being protocol-paused. The protocol owner is performing the correct, expected action; the bug is in the function itself. No special conditions are required beyond the normal protocol pause/unpause lifecycle.

### Recommendation

Change `setPause(1)` to `setPause(0)` in `protocolUnpausePool`:

```solidity
function protocolUnpausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 2) revert InvalidPauseTransition(cur, 0);
    IMetricOmmPoolFactoryActions(pool).setPause(0);  // ← correct: restore to active
}
```

### Proof of Concept

1. Pool is active: `pauseLevel = 0`
2. Protocol owner calls `protocolPausePool(pool)` → `pauseLevel = 2` ✓
3. Protocol owner calls `protocolUnpausePool(pool)` → `pauseLevel = 1` ✗ (should be `0`)
4. Any user calls `swap(...)` → reverts with `PoolPaused()` because `pauseLevel != 0`
5. Pool remains paused and unusable for swaps until the pool admin separately calls `unpausePool(pool)` to transition `1 → 0`
6. If the pool admin is absent or unresponsive, the pool is permanently paused despite the protocol's explicit unpause action [1](#0-0) [2](#0-1) [3](#0-2)
