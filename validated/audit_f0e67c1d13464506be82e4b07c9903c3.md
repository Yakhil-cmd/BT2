### Title
Attacker-Controlled Stray CAT Name Imported Without Confirmation via Visibility Toggle - (`packages/wallets/src/hooks/useWalletsList.ts`)

---

### Summary

An attacker who sends an unsolicited CAT to a victim can set an arbitrary `name` (e.g., `"Chia (XCH)"`) in the transaction memo. The Chia wallet daemon surfaces this name verbatim through `get_stray_cats`. When the victim opens the "Manage token list" panel and toggles the stray cat visible, `handleShow` calls `addCATToken({ name: strayCat.name, assetId: strayCat.assetId })` with no confirmation dialog, permanently registering a wallet whose display name is fully attacker-controlled.

---

### Finding Description

**Data flow:**

1. `useGetStrayCatsQuery` polls `get_stray_cats` every 10 s. The daemon returns `{ assetId, name, firstSeenHeight, senderPuzzleHash }` where `name` originates from the transaction memo — fully attacker-controlled on-chain data. [1](#0-0) [2](#0-1) 

2. `useWalletsList` builds a `ListItem` for each non-imported stray cat, propagating `strayCat.name` directly into the list item's `name` field via `getCATName`. [3](#0-2) [4](#0-3) 

3. `WalletsManageTokens` renders a `WalletTokenCard` per list item. The card shows the attacker-controlled name in an editable text field and a `Switch` toggle. [5](#0-4) 

4. When the user flips the switch ON for a stray cat (which has no `walletId`), `handleVisibleChange` calls `onShow(assetId)` — no confirmation dialog, no intermediate step. [6](#0-5) 

5. `handleShow` in `useWalletsList` finds the stray cat by `assetId` and calls `addCATToken({ name: strayCat.name, assetId: strayCat.assetId })` unconditionally. [7](#0-6) 

6. `addCATToken` calls `createWalletForExisting` (creates a persistent CAT wallet in the daemon) and then `setName` with the attacker-supplied string, permanently registering the wallet under that name. [8](#0-7) 

**No guard exists** between the toggle action and wallet creation: no confirmation modal, no name sanitization, no cross-check against the known CAT list, and no warning that the name is unverified.

The `localCatName` override only fires if the user had previously stored a local name for that `assetId` — impossible for a fresh unsolicited CAT. [9](#0-8) 

---

### Impact Explanation

The attacker permanently registers a wallet in the victim's Chia wallet daemon under an arbitrary display name (e.g., `"Chia (XCH)"`, `"USDC"`, `"Stably USD"`). The victim's wallet list then shows a misleadingly-named entry. Because the name is the primary human-readable identifier, the victim may:

- Believe they received a valuable or trusted token when they received a worthless CAT.
- Make financial decisions (e.g., not selling, treating the balance as XCH) based on the spoofed identity.
- Be confused when attempting to send from the wallet, since the wallet type is CAT but the name implies XCH.

This is "Corruption, spoofing, or unsafe trust of RPC/event/metadata state that causes a user to import or display the wrong asset identity" — a confirmed High impact under the scope rules.

---

### Likelihood Explanation

- Sending a CAT with an arbitrary memo/name requires no special privilege — any Chia user can do it.
- The victim only needs to open the "Manage token list" panel (a normal workflow) and toggle the stray cat visible.
- The stray cat appears automatically after the transaction confirms; no phishing or social engineering is required beyond the initial send.
- The name is visible in the text field before toggling, but there is no explicit warning that it is unverified on-chain data, and many users will not cross-check the `assetId` hex string.

---

### Recommendation

1. **Add a confirmation dialog** before `addCATToken` is called for stray cats, displaying both the raw `assetId` and the unverified `name` with a clear warning that the name is set by the sender and may be misleading.
2. **Default the name to the `assetId`** (or a generic placeholder like `"Unknown CAT"`) for stray cats, requiring the user to explicitly set a name they trust.
3. **Sanitize / length-limit** the `name` field from `get_stray_cats` before it is stored or displayed.

---

### Proof of Concept

```
1. Attacker mints or holds any CAT.
2. Attacker sends 1 mojo of that CAT to the victim's address with memo = "Chia (XCH)".
3. The Chia wallet daemon records this as a stray cat; get_stray_cats returns
   { assetId: '<attacker_cat_id>', name: 'Chia (XCH)', ... }.
4. Victim opens Dashboard → Wallets → "Manage token list".
5. The stray cat appears with name "Chia (XCH)" and a toggle switch (hidden by default).
6. Victim toggles the switch ON.
7. handleVisibleChange → onShow(assetId) → handleShow → addCATToken({
     name: 'Chia (XCH)', assetId: '<attacker_cat_id>'
   }) fires with no confirmation.
8. The Chia wallet daemon now has a persistent CAT wallet named "Chia (XCH)"
   tracking the attacker's worthless CAT assetId.
9. Assert: wallet list contains a wallet named "Chia (XCH)" whose assetId
   is NOT the XCH asset — imported without any user confirmation of the name.
```
