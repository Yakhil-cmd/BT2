### Title
`floor_gas` Computed but Not Applied in Refund Accounting, Allowing Underpayment of EIP-7623 Floor Fee - (File: `engine/src/engine.rs`)

### Summary
The EIP-7623 `floor_gas` value is computed and used only for a gas-limit validation check, but is never applied in the actual gas refund calculation. As a result, any sender whose transaction has `floor_gas > gas_used` receives a larger refund than EIP-7623 mandates, underpaying for their transaction and depriving the relayer of its rightful priority-fee reward.

### Finding Description
In `engine/src/engine.rs`, the `submit_transaction` execution path computes both `intrinsic_gas` and `floor_gas` (EIP-7623):

```rust
let intrinsic_gas = transaction.intrinsic_gas(CONFIG)...;
let floor_gas     = transaction.floor_gas(CONFIG)...;

if transaction.gas_limit < core::cmp::max(intrinsic_gas, floor_gas).into() {
    return Err(EngineErrorKind::IntrinsicGasNotMet.into());
}
```

`floor_gas` is used **only** to gate the transaction (reject it if `gas_limit < floor_gas`). After that gate, `floor_gas` is dropped entirely. The EVM is invoked with the full `gas_limit`, and the refund is computed in `refund_unused_gas` using only the raw `gas_used` returned by the EVM:

```rust
let gas_to_wei = |price: U256| {
    fixed_gas
        .map_or_else(|| gas_used.into(), EthGas::as_u256)  // ← always gas_used, never floor_gas
        .checked_mul(price)
        ...
};
let spent_amount  = gas_to_wei(gas_result.effective_gas_price)?;
let reward_amount = gas_to_wei(gas_result.priority_fee_per_gas)?;
let refund        = gas_result.prepaid_amount.checked_sub(spent_amount)?;
```

Per EIP-7623, when `floor_gas > gas_used`, the minimum gas that must be charged is `floor_gas`, not `gas_used`. The correct `spent_amount` should be `max(gas_used, floor_gas) * effective_gas_price`. Because this maximum is never taken, the sender is refunded `(floor_gas − gas_used) * effective_gas_price` more than they should be, and the relayer receives `(floor_gas − gas_used) * priority_fee_per_gas` less than they are owed.

The engine is configured with the Prague EVM config globally:

```rust
const CONFIG: &Config = &Config::prague();
```

Prague enables `has_floor_gas = true`, so `floor_gas` is non-zero for any transaction carrying calldata.

### Impact Explanation
- **Relayer yield theft (High):** The relayer's reward is `gas_used * priority_fee_per_gas` instead of the correct `floor_gas * priority_fee_per_gas`. The difference `(floor_gas − gas_used) * priority_fee_per_gas` is silently returned to the sender.
- **Base-fee under-burn:** The base-fee portion `(floor_gas − gas_used) * base_fee_per_gas` is also returned to the sender instead of being burned, inflating the ETH supply on Aurora relative to what EIP-7623 requires.

### Likelihood Explanation
Any unprivileged EVM user can trigger this by submitting a transaction with large calldata (e.g., many non-zero bytes) to a contract that ignores the calldata, keeping `gas_used` near the intrinsic minimum while `floor_gas` is much larger. For example, 1 000 non-zero calldata bytes yields `floor_gas = 1000 × 4 × 10 + 21 000 = 61 000` while `gas_used` may be only ~21 000, giving a free refund of `40 000 × effective_gas_price`. This requires no special privilege and is trivially repeatable.

### Recommendation
In `refund_unused_gas` (or at the call site in `submit_transaction`), replace the raw `gas_used` with `max(gas_used, floor_gas)` when computing `spent_amount` and `reward_amount`. The `floor_gas` value should be threaded through from the validation site to the refund site so that the correct minimum is enforced.

### Proof of Concept
1. Deploy Aurora Engine with Prague config (current default).
2. Submit a transaction via `submit` / `submit_with_args` with 1 000 non-zero calldata bytes to a contract that immediately returns (e.g., a no-op).
   - `floor_gas` = 1 000 × 4 × 10 + 21 000 = 61 000
   - `gas_used` ≈ 21 000
3. Observe that `refund_unused_gas` computes `spent_amount = 21 000 × price` instead of `61 000 × price`.
4. Sender receives `40 000 × effective_gas_price` extra refund; relayer receives `40 000 × priority_fee_per_gas` less reward than EIP-7623 mandates.

    #[allow(clippy::naive_bytecount)]
    pub fn floor_gas(&self, config: &aurora_evm::Config) -> Result<u64, Error> {
        if config.has_floor_gas {
            let num_zero_bytes = u64::try_from(self.data.iter().filter(|b| **b == 0).count())
                .map_err(|_e| Error::IntegerConversion)?;
            let data_len = u64::try_from(self.data.len()).map_err(|_e| Error::IntegerConversion)?;
            let num_non_zero_bytes = data_len
                .checked_sub(num_zero_bytes)
                .ok_or(Error::GasOverflow)?;

            let base_gas = config.gas_transaction_call;
            let tokens_in_calldata = num_non_zero_bytes
                .checked_mul(4)
                .and_then(|gas| gas.checked_add(num_zero_bytes))
                .ok_or(Error::GasOverflow)?;

            tokens_in_calldata
                .checked_mul(config.total_cost_floor_per_token)
                .and_then(|gas| gas.checked_add(base_gas))
                .ok_or(Error::GasOverflow)
        } else {
            Ok(0)
        }
    }
