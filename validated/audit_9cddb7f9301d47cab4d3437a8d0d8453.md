Audit Report

## Title
Cross-Subaccount Allowance Leakage in `get_allowances` Due to Incomplete Boundary Check — (File: rs/ledger_suite/icrc1/ledger/src/lib.rs)

## Summary
The `get_allowances` function terminates BTreeMap iteration using an owner-only check (`account_spender.account.owner != from.owner`), but the map is ordered by full `Account` (owner + subaccount). After exhausting entries for the queried subaccount, iteration continues into other subaccounts of the same owner, and their allowances are appended to the response. This violates per-account isolation in ICRC-103 and constitutes an information-disclosure vulnerability on the ICRC ledger.

## Finding Description
In `get_allowances` (line 1174), the loop iterates over `ALLOWANCES_MEMORY` — a `StableBTreeMap<AccountSpender, StorableAllowance>` — starting from the queried `from` account:

```rust
allowances.range(start_account_spender.clone()..)
```

`AccountSpender::cmp` (lines 372–378) orders by `(account, spender)`, and `Account::cmp` (lines 54–60 of `account.rs`) orders by `(owner, effective_subaccount())`. The full BTreeMap key order is therefore `(owner, subaccount, spender_owner, spender_subaccount)`.

The loop's boundary check at line 1204 is:
```rust
if account_spender.account.owner != from.owner { break; }
```
This only tests the owner field. When the iterator advances past all entries for `{owner:X, subaccount:SA}` to `{owner:X, subaccount:SA+1}`, the condition evaluates to `false` (same owner X), so the loop does not break. Entries belonging to `{owner:X, subaccount:SA+1}` are pushed into `result` with `from_account` set to `{X, SA+1}`, not the queried `{X, SA}`. The existing size cap at line 1201 and the expiry check at lines 1207–1210 do not prevent this cross-subaccount leakage.

## Impact Explanation
Any caller of `icrc103_get_allowances` querying `from_account = {owner:X, subaccount:SA}` receives allowances belonging to other subaccounts of owner X that sort after SA in the BTreeMap. The leaked data includes allowance amounts, spender identities, and expiry times for unqueried subaccounts. This is unauthorized disclosure of ledger allowance data on the in-scope ICRC ledger, constituting a significant ledger security impact with concrete user harm. This matches: **High ($2,000–$10,000) — Significant ledger security impact with concrete user or protocol harm.**

## Likelihood Explanation
The exploit requires no privilege. It is a deterministic, locally reproducible query call. Any principal who knows (or guesses) that owner X has approved from multiple subaccounts can trigger it. The precondition — owner X having approved from two different subaccounts — is a normal, expected usage pattern under ICRC-2. No victim mistake, social engineering, or special access is required.

## Recommendation
Replace the owner-only boundary check at line 1204 with a full-account equality check:

```rust
// Change from:
if account_spender.account.owner != from.owner {
// To:
if account_spender.account != from {
```

`Account::eq` (lines 40–43 of `account.rs`) already uses `effective_subaccount()` for comparison, so `None` and `Some([0u8;32])` are treated as equal, which is the correct semantic for ICRC account identity.

## Proof of Concept
```
1. icrc2_approve: caller={X, [0;32]}, spender=S1, amount=100
2. icrc2_approve: caller={X, [1;32]}, spender=S2, amount=200
3. icrc103_get_allowances(from_account={X, [0;32]}, prev_spender=None, take=Some(500))
```
Expected: `[{from_account:{X,[0;32]}, to_spender:S1, allowance:100}]`

Actual (with bug): also includes `{from_account:{X,[1;32]}, to_spender:S2, allowance:200}` because after iterating past all entries for `{X,[0;32]}`, the iterator reaches `{X,[1;32]}` and the check `account_spender.account.owner != from.owner` is `false` (both are X), so the entry is appended to `result`. This can be reproduced as a deterministic PocketIC integration test or unit test over a mock `StableBTreeMap`.
