### Title
Unguarded `Debug` Derive on `KeygenOutput` and `PresignOutput` Exposes Secret Signing Shares and Nonce Material in Plaintext - (File: `src/lib.rs`, `src/ecdsa/ot_based_ecdsa/mod.rs`, `src/ecdsa/robust_ecdsa/mod.rs`)

---

### Summary

`KeygenOutput<C>`, `PresignOutput` (OT-based), `RerandomizedPresignOutput` (OT-based), `PresignOutput` (Robust ECDSA), and `RerandomizedPresignOutput` (Robust ECDSA) all unconditionally derive `Debug`. These structs contain secret scalar fields — the private signing share and presign nonce shares — that are printed verbatim when any caller formats the struct with `{:?}`. This is the direct library analog of the clipboard-exposure class: sensitive cryptographic material is disclosed through an unintended output channel (the `Debug` trait) with no redaction, no warning, and no opt-in.

---

### Finding Description

**`KeygenOutput<C>` — private signing share exposed via `Debug`** [1](#0-0) 

`KeygenOutput<C>` is the primary output of `keygen()`, `reshare()`, and `refresh()`. It holds `private_share: SigningShare<C>`, the participant's secret scalar share of the distributed key. The struct unconditionally derives `Debug`, so any call to `println!("{:?}", keygen_output)`, `tracing::debug!("{:?}", ...)`, `format!("{:?}", ...)`, or any logging framework that auto-formats values will emit the raw secret scalar in plaintext.

**`PresignOutput` (OT-based) — secret nonce shares `k` and `sigma` exposed via `Debug`** [2](#0-1) 

`k` is the participant's share of the signing nonce; `sigma` is the participant's share of the masked secret. Both are secret scalars. `Debug` is derived unconditionally.

**`RerandomizedPresignOutput` (OT-based) — rerandomized secret nonce shares exposed via `Debug`** [3](#0-2) 

**`PresignOutput` (Robust ECDSA) — secret nonce shares `c`, `e`, `alpha`, `beta` exposed via `Debug`** [4](#0-3) 

`c`, `e`, `alpha`, and `beta` are all secret scalar shares of the presignature nonces. `Debug` is derived unconditionally.

**`RerandomizedPresignOutput` (Robust ECDSA) — rerandomized secret nonce shares exposed via `Debug`** [5](#0-4) 

The serialization test confirms that `private_share` is emitted as a full hex scalar when the struct is formatted: [6](#0-5)

### Impact Explanation

**`KeygenOutput` — Critical.** The `private_share` field is the participant's secret signing share. Disclosure of this value to any observer (log aggregator, crash reporter, monitoring system, developer console) directly satisfies the allowed critical impact: *"Extraction, reconstruction, or disclosure of private signing shares."* With `t` such shares (where `t` is the reconstruction threshold), an attacker can reconstruct the full private key and produce arbitrary valid signatures without any protocol participation.

**`PresignOutput` (both variants) — Critical.** The nonce shares `k`/`sigma` (OT-based) and `c`/`e`/`alpha`/`beta` (Robust) are presign secrets. Disclosure of nonce material satisfies: *"Extraction, reconstruction, or disclosure of presign secrets, nonce material."* In ECDSA, nonce reuse or nonce leakage is sufficient to extract the private key.

---

### Likelihood Explanation

Rust's `Debug` derive is routinely triggered by:
- `println!("{:?}", output)` during development or error handling
- Logging frameworks (`tracing`, `log`) that format values with `{:?}`
- `unwrap()`/`expect()` on `Result<KeygenOutput<C>, _>` — the error path formats the `Ok` variant in some contexts
- Test assertion macros (`assert_eq!`) that print both sides on failure

The `public` visibility of `private_share` and the `pub` fields of `PresignOutput` mean any downstream crate can trigger this path. The library exports these types as part of its public API, so any integrator is exposed. The `ZeroizeOnDrop` on `KeygenOutput` shows the authors are aware of secret-handling concerns, making the unguarded `Debug` derive an inconsistency rather than an intentional design choice.

---

### Recommendation

1. **Remove `Debug` from the derive list** for `KeygenOutput<C>`, `PresignOutput` (both variants), and `RerandomizedPresignOutput` (both variants), or implement a custom `Debug` that redacts secret scalar fields:

```rust
impl<C: Ciphersuite> fmt::Debug for KeygenOutput<C> {
    fn fmt(&self, f: &mut fmt::Formatter<'_>) -> fmt::Result {
        f.debug_struct("KeygenOutput")
            .field("private_share", &"[REDACTED]")
            .field("public_key", &self.public_key)
            .finish()
    }
}
```

2. Apply the same custom `Debug` to all `PresignOutput` and `RerandomizedPresignOutput` structs, redacting all secret scalar fields (`k`, `sigma`, `c`, `e`, `alpha`, `beta`).

3. Consider using the [`secrecy`](https://crates.io/crates/secrecy) crate to wrap secret scalars, which prevents accidental `Debug` and `Display` exposure at the type level.

### Proof of Concept

```rust
use threshold_signatures::{keygen, KeygenOutput, participants::Participant};
// ... setup participants, run protocol ...
let keygen_output: KeygenOutput<Secp256K1Sha256> = /* result of keygen protocol */;

// The following line prints the raw private signing share scalar in hex:
println!("{:?}", keygen_output);
// Output: KeygenOutput { private_share: SigningShare(0x<SECRET_SCALAR_HEX>), public_key: ... }

// Any logging framework call has the same effect:
tracing::debug!(output = ?keygen_output, "keygen complete");
```

The serialization test at `src/ecdsa/mod.rs:262–269` already demonstrates that `private_share` serializes to a full 32-byte hex scalar. The `Debug` output is equivalent in information content. An attacker with read access to logs, crash dumps, or any diagnostic output channel obtains the raw secret share without any cryptographic effort.
