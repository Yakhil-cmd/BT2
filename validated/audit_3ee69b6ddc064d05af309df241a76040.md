### Title
Malicious Participant Can Corrupt CKD Output Without Detection — (`src/confidential_key_derivation/protocol.rs`)

### Summary
The `do_ckd_coordinator` function in `src/confidential_key_derivation/protocol.rs` aggregates `CKDOutput` values received from participants without any cryptographic verification of correctness. Unlike the DKG protocol, which verifies every received share against a polynomial commitment, the CKD coordinator blindly sums participant contributions. Any single malicious participant can send arbitrary group elements, causing the coordinator to output a corrupted, unusable confidential derived key.

### Finding Description
In `do_ckd_coordinator` (lines 35–57), the coordinator collects each participant's `(norm_big_y, norm_big_c)` pair and accumulates them:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
Ok(Some(ckd_output))
``` [1](#0-0) 

There is no check that the received `big_y` or `big_c` values are correctly formed as `(λ_i · y_i · G, λ_i · (x_i · H(pk ‖ app_id) + y_i · A))`. A malicious participant can substitute arbitrary `G1` elements.

The contrast with the DKG protocol is direct. In `do_keyshare` (`src/dkg.rs`), every received secret share is verified against the sender's polynomial commitment:

```rust
validate_received_share::<C>(me, from, &signing_share_from, full_commitment_from)?;
``` [2](#0-1) 

And every commitment is verified against a proof of knowledge:

```rust
verify_proof_of_knowledge(
    &session_id,
    &mut proof_domain_separator.clone(),
    threshold,
    p,
    old_participants.clone(),
    commitment_i,
    proof_i.as_ref(),
)?;
``` [3](#0-2) 

The CKD protocol has no analogous verification step. The `do_ckd_participant` function computes the correct share and sends it, but nothing in the coordinator path enforces that the received values match any commitment or proof: [4](#0-3) 

The correct CKD output satisfies `big_c − a · big_y = msk · H(pk ‖ app_id)`. If any participant substitutes arbitrary `(big_y', big_c')`, the coordinator's aggregate no longer satisfies this equation, and the client's `unmask` call produces a wrong result. [5](#0-4) 

### Impact Explanation
A single malicious participant can make the coordinator output an incorrect `CKDOutput`. The client's call to `unmask(app_sk)` will return a value that is not `msk · H(pk ‖ app_id)`, rendering the derived confidential key permanently wrong for that invocation. This matches the allowed High impact: **Corruption of CKD outputs so honest parties accept unusable cryptographic outputs**.

### Likelihood Explanation
Every participant in the CKD protocol is a potential attacker. No elevated privilege is required — participation itself is the only precondition. The attack is a single-message substitution (send arbitrary `G1` elements instead of the honest share) and is undetectable by the coordinator with the current code.

### Recommendation
Add a non-interactive zero-knowledge proof of correct computation alongside each participant's `(norm_big_y, norm_big_c)` contribution, proving knowledge of `y_i` and `x_i` such that the submitted values are correctly formed. Alternatively, introduce a commitment round (analogous to the DKG's `commitment_hash` broadcast) so the coordinator can verify each participant's contribution before aggregating.

### Proof of Concept
1. Honest participants run `ckd(...)` normally.
2. Malicious participant `P_m` implements a custom protocol handler that, at the `waitpoint`, sends `(G1::generator(), G1::generator())` (arbitrary group elements) to the coordinator instead of the honest `(norm_big_y, norm_big_c)`.
3. The coordinator's loop at lines 50–55 adds `G1::generator()` to both `norm_big_y` and `norm_big_c`.
4. The final `CKDOutput` is `(Σ λ_i y_i G + G, Σ λ_i(x_i H + y_i A) + G)`.
5. The client computes `big_c − a · big_y = msk · H(pk ‖ app_id) + G − a · G`, which is not `msk · H(pk ‖ app_id)`.
6. The derived confidential key is permanently corrupted for this session. [6](#0-5)
