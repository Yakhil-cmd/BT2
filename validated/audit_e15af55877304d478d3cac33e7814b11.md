### Title
`skip_stateful_validations` Bypasses `__validate__` for Invoke Transactions When Any (Not Only `deploy_account`) Transaction Is in the Mempool — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The gateway's `skip_stateful_validations` function is intended to skip the `__validate__` entry-point check only when a `deploy_account` transaction is pending for a freshly-created account (UX feature: deploy + invoke in one shot). However, the predicate it uses — `account_tx_in_pool_or_recent_block` — returns `true` for **any** transaction from the sender address in the mempool, not exclusively a `deploy_account`. An attacker who already has a valid nonce-0 invoke in the pool can submit a second invoke with nonce=1 that carries an invalid signature (or that would otherwise fail the account's custom `__validate__` logic), and the gateway will admit it to the mempool without ever running `__validate__`.

---

### Finding Description

`skip_stateful_validations` is called inside `run_pre_validation_checks`:

```
validate_state_preconditions  →  validate_by_mempool  →  skip_stateful_validations
``` [1](#0-0) 

The skip condition is:

```rust
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
    return mempool_client
        .account_tx_in_pool_or_recent_block(tx.sender_address())
        .await ...;
}
``` [2](#0-1) 

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` sets `validate: false` in `ExecutionFlags`, completely suppressing the `__validate__` call:

```rust
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [3](#0-2) 

The mempool's `account_tx_in_pool_or_recent_block` returns `true` if the account has **any** transaction in the pool or any committed block:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [4](#0-3) 

The code comment acknowledges the imprecision: *"it means that either it has a deploy_account transaction **or** transactions with future nonces that passed validations."* The second branch is the exploitable one.

**Attack scenario:**

1. Account `A` is deployed via the `deploy` syscall (on-chain nonce = 0, contract exists).
2. Attacker submits invoke `T1` with nonce=0 for `A` — valid signature, passes `__validate__` normally, gets added to the mempool.
3. Attacker submits invoke `T2` with nonce=1 for `A` — **invalid signature** or carrying a payload that the account's custom `__validate__` would reject.
4. During `T2`'s gateway validation: `account_nonce == 0`, `tx.nonce() == 1`, `account_tx_in_pool_or_recent_block` returns `true` (because `T1` is in the pool) → `skip_stateful_validations` returns `true` → `__validate__` is **skipped**.
5. `T2` is admitted to the mempool without signature verification.

The broken invariant: the gateway must run `__validate__` for every invoke transaction from an already-deployed account. The skip is only safe when the account does not yet exist on-chain (i.e., the pending transaction is specifically a `deploy_account`).

---

### Impact Explanation

**High — Mempool/gateway admission accepts invalid transactions before sequencing.**

The gateway admits an invoke transaction that carries an invalid or absent signature (or that would fail the account's custom `__validate__` logic) into the mempool. This violates the admission invariant that every accepted transaction must have passed account-level signature/authorization verification. Downstream, the batcher will re-run `__validate__` and the transaction will revert, but the gateway has already accepted and propagated an unauthorized transaction, wasting mempool capacity and potentially enabling targeted DoS against specific accounts.

---

### Likelihood Explanation

Any account deployed via the `deploy` syscall (nonce=0, contract exists) is a valid target. The attacker only needs to submit one valid nonce=0 invoke first, which is a normal operation. The second, malicious nonce=1 invoke then bypasses `__validate__` automatically. No privileged access is required.

---

### Recommendation

Replace the coarse `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a **`deploy_account` transaction** is pending for the sender address. The mempool should expose a dedicated query such as `deploy_account_tx_in_pool(address) -> bool` that only returns `true` when the pending transaction is of type `DeployAccount`. Alternatively, the gateway can inspect the transaction type of the pending pool entry before granting the skip.

The `max_nonce_for_validation_skip` field already exists in `StatefulTransactionValidatorConfig` but is not used in the gateway's `skip_stateful_validations` path (it is only used in `PyValidator::should_run_stateful_validations`). The gateway path should also respect this bound and tighten the type check. [5](#0-4) 

---

### Proof of Concept

```
1. Deploy account contract C at address A via `deploy` syscall.
   On-chain state: nonce(A) = 0, class_hash(A) = C.

2. Submit T1 = Invoke(sender=A, nonce=0, calldata=..., signature=valid_sig)
   → Gateway: account_nonce=0, tx_nonce=0 → skip_stateful_validations returns false
   → __validate__ runs, passes → T1 added to mempool.

3. Submit T2 = Invoke(sender=A, nonce=1, calldata=malicious_payload, signature=[])
   → Gateway: account_nonce=0, tx_nonce=1
   → skip_stateful_validations: nonce==1 && account_nonce==0 → calls
     account_tx_in_pool_or_recent_block(A) → true (T1 is in pool)
   → returns true → run_validate_entry_point sets validate=false
   → __validate__ SKIPPED
   → T2 admitted to mempool with empty/invalid signature.

4. Batcher picks up T2, runs __validate__ → transaction reverts.
   Gateway has admitted an unauthorized transaction.
``` [1](#0-0) [6](#0-5) [4](#0-3)

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
