### Title
ETH Permanently Locked in `RewardSupplier` When `tick()` Called With `msg.value > 0` But No Pending Mint Requests ŌĆö (File: `L1/starkware/solidity/stake/RewardSupplier.sol`)

---

### Summary

`RewardSupplier.tick()` is `payable` and forwards `msg.value` as L1ŌåÆL2 message fees only inside the `if (amountToMint > 0)` branch. When the branch is not entered ŌĆö because no L2 mint requests are pending ŌĆö any ETH sent by the caller is silently absorbed by the contract with no refund path and no withdrawal mechanism, permanently locking the funds.

---

### Finding Description

`tick()` splits `msg.value` between two L1ŌåÆL2 calls only when `amountToMint > 0`:

```solidity
// RewardSupplier.sol lines 107-143
function tick() external payable {
    (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

    if (amountToMint > 0) {
        ...
        uint256 msgFee = msg.value / 2;
        bridge().depositWithMessage{value: msgFee}(...);   // L1ŌåÆL2 fee forwarded

        msgFee = msg.value - msgFee;
        messagingContract().sendMessageToL2{value: msgFee}(...); // L1ŌåÆL2 fee forwarded
    }
    // ŌåÉ no else branch, no refund, no ETH recovery
}
``` [1](#0-0) 

When `amountToMint == 0` the function returns immediately. Any `msg.value` attached to the call is now held by the contract. Neither `RewardSupplier` nor its storage base `RewardSupplierStorage` defines a `receive()`, `fallback()`, or any ETH-withdrawal function, so the ETH is irrecoverable. [2](#0-1) 

The analog to the external report is exact: the external report's `handle()` received ETH but failed to forward it to the next cross-chain call; here `tick()` receives ETH but fails to forward or refund it when the cross-chain path is not taken.

---

### Impact Explanation

Any ETH sent to `tick()` when no mint requests are pending is permanently locked in the `RewardSupplier` contract. There is no admin withdrawal, no `receive()` fallback, and no upgrade path that recovers ETH from storage. This constitutes permanent loss of the caller's funds and qualifies as **Medium ŌĆö griefing with no profit motive but concrete financial damage to the keeper/caller**.

---

### Likelihood Explanation

The condition is reachable by any public caller:

1. **Race condition (front-run):** A keeper observes pending L2 mint requests off-chain and submits `tick()` with ETH. An adversary front-runs with their own `tick()` (paying minimal ETH for the fees), consuming all pending requests. The keeper's transaction then executes with `amountToMint == 0` and loses their ETH.
2. **Timing gap:** Between the off-chain `requiredMinting()` read and on-chain execution, another honest keeper may have already processed the requests, producing the same outcome without malicious intent.
3. **Accidental call:** Any caller who invokes `tick()` with ETH when the queue is empty loses that ETH.

`tick()` is an `external` function with no access control, so any address can trigger the front-run scenario. [3](#0-2) 

---

### Recommendation

Add an ETH refund when the `amountToMint == 0` branch is taken:

```solidity
function tick() external payable {
    (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

    if (amountToMint > 0) {
        // ... existing logic ...
    } else if (msg.value > 0) {
        // Refund unused fee to caller
        (bool ok, ) = msg.sender.call{value: msg.value}("");
        require(ok, "ETH_REFUND_FAILED");
    }
}
```

Alternatively, require `msg.value == 0` when there is nothing to process, or document that callers must first call `requiredMinting()` and only attach ETH when the return value is non-zero.

---

### Proof of Concept

1. L2 `RewardSupplier` has zero pending mint-request messages (normal state between mint cycles).
2. Keeper A calls `requiredMinting()` off-chain ŌåÆ returns `(0, 0)`.
3. Keeper B (or an adversary) calls `tick{value: 1 ether}()` ŌĆö ETH is sent but `amountToMint == 0`, so the `if` branch is skipped.
4. The 1 ETH is now held by `RewardSupplier` with no recovery path.
5. Confirm: `address(rewardSupplier).balance` increases by 1 ETH; no event is emitted; no withdrawal function exists in `RewardSupplier` or `RewardSupplierStorage`. [1](#0-0)

        // Check if minting is required, and how much.
        (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

        if (amountToMint > 0) {
            // Prepare the L2->L1 mintRequest message for consumption.
            uint256[] memory messagePayload = new uint256[](1);
            messagePayload[0] = TOKENS_PER_MINT_REQUEST;

            // Consume the mintRequest messages.
            for (uint256 i = 0; i < numMsgsToConsume; i++) {
                messagingContract().consumeMessageFromL2(mintRequestSource(), messagePayload);
            }

            // Reuest minting of the requested amount from the mint manager.
            mintManager().mintRequest(token(), amountToMint);

            // Deposit the minted amount onto the bridge to the credit of `mintDestination`.
            uint256 msgFee = msg.value / 2;
            bridge().depositWithMessage{value: msgFee}(
                token(),
                amountToMint,
                mintDestination(),
                new uint256[](0)
            );
            emit ConsumedL2MintRequests(numMsgsToConsume, amountToMint);

            // Send a totalSupply update to L2MintCurve.
            msgFee = msg.value - msgFee;
            messagePayload[0] = IERC20(token()).totalSupply();
            messagingContract().sendMessageToL2{value: msgFee}(
                mintingCurve(),
                UPDATE_TOTAL_SUPPLY_SELECTOR,
                messagePayload
