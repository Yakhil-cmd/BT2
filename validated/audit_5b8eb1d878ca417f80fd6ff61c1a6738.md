### Title
Incomplete `frost_core::Ciphersuite` Implementation for `BLS12381SHA256` with `unimplemented!()` Stubs Causes Runtime Panic in DKG/CKD Key Generation - (File: `src/confidential_key_derivation/ciphersuite.rs`)

### Summary
`BLS12381SHA256` implements `frost_core::Ciphersuite` but leaves six required methods (`H1`, `H2`, `H3`, `H4`, `H5`, `HID`) as `unimplemented!()`. This is the Rust analog to the reported Solidity bug: instead of a compile-time failure, these stubs compile successfully but panic at runtime if any of the unimplemented methods are reached during DKG or CKD key generation. The public `keygen::<BLS12381SHA256>()` API is the entry point.

### Finding Description
In `src/confidential_key_derivation/ciphersuite.rs`, the `frost_core::Ciphersuite` trait implementation for `BLS12381SHA256` provides six methods as `unimplemented!()`:

```rust
// src/confidential_key_derivation/ciphersuite.rs lines 41–87
impl frost_core::Ciphersuite for BLS12381SHA256 {
    fn H1(m: &[u8]) -> ... { unimplemented!() }
    fn H2(m: &[u8]) -> ... { unimplemented!() }
    fn H3(m: &[u8]) -> ... { unimplemented!() }
    fn H4(m: &[u8]) -> ... { unimplemented!() }
    fn H5(_m: &[u8]) -> ... { unimplemented!() }
    fn HID(m: &[u8]) -> ... { unimplemented!() }
    // Only HDKG is actually implemented
}
``` [1](#0-0) 

The comment at line 39 states: *"We are currently not using all the functionality. Therefore, I implemented only those that we use."* However, this assumption is fragile and incorrect for the full DKG execution path.

The public `keygen::<BLS12381SHA256>()` function in `src/lib.rs` is the entry point: [2](#0-1) 

It calls `do_keygen` → `do_keyshare` → `proof_of_knowledge` → `<C>::generate_nonce(rng)` (line 132 of `dkg.rs`), and also `validate_received_share` → `me.to_identifier::<C>()` (line 265 of `dkg.rs`). [3](#0-2) [4](#0-3) 

The `to_identifier` call in `validate_received_share` is particularly dangerous: in `frost_core`, `Identifier::derive` uses `HID` to map participant bytes to a field scalar. If `Participant::to_identifier` routes through `frost_core`'s `derive_identifier`, it will call `HID`, which is `unimplemented!()` for `BLS12381SHA256`, causing an unconditional panic during every DKG execution for CKD.

Additionally, `frost_core::round1::commit` (called in `do_presign` at `src/frost/mod.rs` line 101) internally calls `generate_nonce` which in some frost_core versions calls `H3` for deterministic nonce generation — also `unimplemented!()`. [5](#0-4) 

### Impact Explanation
Any caller invoking `keygen::<BLS12381SHA256>()` (required to bootstrap CKD key material) will trigger a runtime panic during the DKG execution, permanently preventing key generation for the Confidential Key Derivation protocol. All honest participants are affected. This maps to:

**High: Permanent denial of key generation for honest parties under valid protocol inputs.**

### Likelihood Explanation
The `keygen::<BLS12381SHA256>()` function is part of the public API (`pub fn keygen<C: Ciphersuite>` in `src/lib.rs`). Any integrator attempting to generate BLS12381SHA256 keys for CKD will hit this panic. The `validate_received_share` function is called unconditionally for every other participant during DKG round 5, making the panic reachable on every valid DKG execution with ≥2 participants.

### Recommendation
Either fully implement all six methods (`H1`, `H2`, `H3`, `H4`, `H5`, `HID`) for `BLS12381SHA256` following the FROST spec and the BLS12-381 ciphersuite definition, or replace `unimplemented!()` with `unreachable!()` only after formally proving (with a test) that each stub is never reachable through any code path in DKG, CKD, reshare, or refresh. The safer fix is full implementation. At minimum, add a compile-time or test-time assertion that `keygen::<BLS12381SHA256>()` completes without panic.

### Proof of Concept
1. Call `keygen::<BLS12381SHA256>(&participants, me, threshold, rng)` with valid inputs (≥2 participants, threshold ≥2).
2. Drive the returned `Protocol` to completion by calling `poke()` and routing messages between participants.
3. During round 5 of DKG, `validate_received_share` is called, which calls `me.to_identifier::<BLS12381SHA256>()`.
4. If `to_identifier` routes through `frost_core`'s `derive_identifier`, `HID` is invoked → `unimplemented!()` → **thread panics**.
5. All participants' DKG instances abort; no `KeygenOutput<BLS12381SHA256>` is ever produced; CKD is permanently unavailable. [6](#0-5) [7](#0-6)

fn proof_of_knowledge<C: Ciphersuite>(
    session_id: &HashOutput,
    domain_separator: &mut DomainSeparator,
    me: Participant,
    coefficients: &Polynomial<C>,
    coefficient_commitment: &PolynomialCommitment<C>,
    rng: &mut impl CryptoRngCore,
) -> Result<Signature<C>, ProtocolError> {
    // creates an identifier for the participant
    let id = me.scalar::<C>();
    let vk_share = coefficient_commitment.eval_at_zero()?;

    // pick a random k_i and compute R_id = g^{k_id},
    // Step 2.5
    let (k, big_r) = <C>::generate_nonce(rng);

    // Step 2.6
    // compute H(domain_separator, id, me, g^{a_0}, R_id) as a scalar
    let hash = challenge::<C>(domain_separator, session_id, id, &vk_share, &big_r)?;
    let a_0 = coefficients.eval_at_zero()?.0;
    // Step 2.7
    let mu = k + a_0 * hash.to_scalar();
    Ok(Signature::new(big_r, mu))
}

fn validate_received_share<C: Ciphersuite>(
    me: Participant,
    from: Participant,
    signing_share_from: &SigningShare<C>,
    commitment: &VerifiableSecretSharingCommitment<C>,
) -> Result<(), ProtocolError> {
    let id = me.to_identifier::<C>()?;

    // The verification is exactly the same as the regular SecretShare verification;
    // however the required components are in different places.
    // Build a temporary SecretShare so what we can call verify().
    let secret_share = SecretShare::new(id, *signing_share_from, commitment.clone());

    // Verify the share. We don't need the result.
    // Identify the culprit if an InvalidSecretShare error is returned.
    secret_share.verify().map_err(|e| {
        if let Error::InvalidSecretShare { .. } = e {
            ProtocolError::InvalidSecretShare(from)
        } else {
            ProtocolError::AssertionFailed(format!(
                "could not
            extract the verification key matching the secret
            share sent by {from:?}"
            ))
        }
    })?;
    Ok(())
}

```rust
async fn do_presign<C: Ciphersuite + Send>(
    mut chan: SharedChannel,
    participants: ParticipantList,
    me: Participant,
    signing_share: SigningShare<C>,
    mut rng: impl CryptoRngCore,
) -> Result<PresignOutput<C>, ProtocolError> {
    // Round 1
    let mut commitments_map: BTreeMap<Identifier<C>, SigningCommitments<C>> = BTreeMap::new();

    // Creating two commitments and corresponding nonces
    let (nonces, commitments) = commit(&signing_share, &mut rng);
    commitments_map.insert(me.to_identifier()?, commitments);

    let commit_waitpoint = chan.next_waitpoint();
    // Sending the commitments to all
    chan.send_many(commit_waitpoint, &commitments)?;

    // Collecting the commitments
    for (from, commitment) in recv_from_others(&chan, commit_waitpoint, &participants, me).await? {
        commitments_map.insert(from.to_identifier()?, commitment);
    }

    Ok(PresignOutput {
        nonces,
        commitments_map,
    })
}
