Audit Report

## Title
Permanent ICRC-1 Ledger Errors Cause Indefinite Deposit Retry with No User Recovery Path - (File: rs/ethereum/cketh/minter/src/deposit.rs)

## Summary
The `mint()` function in `deposit.rs` handles all ICRC-1 `TransferError` variants identically — defusing the quarantine guard and re-scheduling the mint indefinitely — with no distinction between transient errors (e.g., `TemporarilyUnavailable`) and permanent ones (e.g., `GenericError`, `BadFee`, `Duplicate`). Because the user's ETH/ERC-20 is already locked in the minter's Ethereum address at the time of scraping, a permanently failing deposit event loops in `events_to_mint` forever with no user-accessible recovery path. The same structural flaw exists in `process_reimbursement()` in `withdraw.rs`.

## Finding Description

**Root cause — `mint()` in `deposit.rs` (lines 85–101):**

Both the ledger-level error arm (`Ok(Err(err))`) and the message-delivery failure arm (`Err(err)`) execute the same three steps: log the error, increment `error_count`, and explicitly defuse the `ScopeGuard` via `ScopeGuard::into_inner(prevent_double_minting_guard)`. Defusing the guard is the critical flaw: the guard's drop closure — which emits `EventType::QuarantinedDeposit` — only fires on an *unwind* (panic). By defusing it on every error path, the code ensures quarantine is **never** triggered for any ledger-returned error.

```rust
Ok(Err(err)) => {
    log!(INFO, "Failed to mint {token_symbol}: {event:?} {err}");
    error_count += 1;
    ScopeGuard::into_inner(prevent_double_minting_guard); // guard defused → no quarantine
    continue;                                              // event stays in events_to_mint
}
```

After the loop, any non-zero `error_count` unconditionally reschedules `mint()`:

```rust
if error_count > 0 {
    ic_cdk_timers::set_timer(crate::MINT_RETRY_DELAY, async { mint().await });
}
```

The `TransferError` enum (`packages/icrc-ledger-types/src/icrc1/transfer.rs`, lines 62–71) contains multiple permanent variants: `BadFee`, `BadBurn`, `InsufficientFunds`, `TooOld`, `CreatedInFuture`, `Duplicate`, and `GenericError`. None of these are distinguished from `TemporarilyUnavailable`. Any of them causes the same indefinite retry loop.

**Why existing checks are insufficient:**

The `QuarantinedDeposit` event type exists precisely for this scenario (see `state/event.rs` lines 141–148 and `state.rs` lines 117–128), but it is only reachable via an unhandled panic in the async callback. The guard is always defused before `continue` on every error path, making the quarantine mechanism dead code for ledger errors.

**Exploit flow for ckERC20:**

1. A ckERC20 ledger canister (each token has its own, independently deployed ledger) is configured with custom logic — e.g., a supply cap, a blocked-account check, or a ledger-level invariant — that causes it to return `TransferError::GenericError` for a specific mint.
2. A user deposits the corresponding ERC-20 token via the Ethereum helper contract. The ERC-20 is transferred to the minter's Ethereum address and a `ReceivedErc20` log event is emitted.
3. The minter scrapes the log and records `EventType::AcceptedErc20Deposit`, placing the event in `state.events_to_mint`.
4. `mint()` fires, calls `client.transfer(...)`, and receives `Ok(Err(TransferError::GenericError {...}))`.
5. The `Ok(Err(err))` arm defuses the guard, increments `error_count`, and continues.
6. After the loop, `error_count > 0` triggers `ic_cdk_timers::set_timer(MINT_RETRY_DELAY, mint())`.
7. Steps 4–6 repeat indefinitely. The user's ERC-20 is locked on Ethereum; no ckERC20 is ever minted; no user-callable endpoint exists to reclaim the tokens.

The same pattern in `process_reimbursement()` (`withdraw.rs` lines 95–116) means a user who burned ckETH for a failed withdrawal can also have their reimbursement stuck indefinitely.

## Impact Explanation

This matches the **High** impact class: *"Significant Chain Fusion, ck-token, ledger … security impact with concrete user or protocol harm."* A user's deposited ERC-20 assets are irrevocably transferred to the minter's Ethereum address at deposit time. If the corresponding ckERC20 ledger permanently rejects the mint, the user receives neither ckERC20 tokens nor a refund of their original assets. There is no on-chain Ethereum reversal mechanism and no user-callable IC endpoint to cancel or reclaim a pending deposit. Resolution requires an admin-driven minter upgrade to manually quarantine the event and arrange an Ethereum refund — an out-of-band, privileged operation.

## Likelihood Explanation

An unprivileged user triggers the vulnerable path simply by depositing ERC-20 tokens. The permanent-error condition does not require governance majority for ckERC20 tokens: each ckERC20 has its own independently deployed ledger canister, and a ledger with custom supply-cap logic, account-blocking, or a regression that permanently rejects specific mint amounts is a realistic deployment scenario within the ckERC20 ecosystem. The ICRC-1 standard explicitly reserves `GenericError` as a catch-all for such conditions. No retry limit exists, so even a condition that starts transient and becomes permanent (e.g., a ledger canister that is stopped and never restarted) produces the same indefinite lock.

## Recommendation

1. **Classify errors at the call site**: Match on `TransferError` variants before deciding whether to retry. `TemporarilyUnavailable` warrants a retry; `GenericError`, `BadFee`, `BadBurn`, `InsufficientFunds`, `Duplicate`, `TooOld`, and `CreatedInFuture` are permanent and should not be retried.
2. **Quarantine on permanent errors**: For permanent `TransferError` variants, do *not* defuse the `ScopeGuard`; let it drop naturally so `EventType::QuarantinedDeposit` is emitted and the event is removed from `events_to_mint`.
3. **Bounded retries**: Introduce a per-event retry counter. After a configurable maximum, emit `QuarantinedDeposit` regardless of error type and alert operators.
4. **User recovery endpoint**: Expose a canister endpoint callable by the original depositor that, for a quarantined deposit, initiates an Ethereum refund transaction to the depositor's address.
5. **Apply the same fix to `process_reimbursement()`** in `withdraw.rs` (lines 95–116) to prevent the symmetric loss of both burned ckETH and the underlying ETH.

## Proof of Concept

**Deterministic integration test plan:**

1. Deploy a mock ckERC20 ICRC-1 ledger canister that unconditionally returns `TransferError::GenericError { error_code: 1, message: "supply cap exceeded" }` for any `icrc1_transfer` call.
2. Initialize the ckETH minter with this mock ledger registered for a test ERC-20 contract address.
3. Inject a `ReceivedErc20Event` directly into `state.events_to_mint` (bypassing Ethereum scraping) via `process_event(s, EventType::AcceptedErc20Deposit(...))`.
4. Call `mint()` (or trigger it via timer).
5. Assert: `state.events_to_mint` still contains the event after `mint()` returns (event not removed, not quarantined).
6. Assert: a new timer for `MINT_RETRY_DELAY` has been scheduled (confirming indefinite retry).
7. Assert: `state.invalid_events` does **not** contain a `QuarantinedDeposit` entry for the event source (confirming the quarantine path is never reached).
8. Repeat step 4 N times and confirm the event remains in `events_to_mint` indefinitely with no state transition.
