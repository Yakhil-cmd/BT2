### Title
Unbounded `stakers` Vec Growth Causes Unbounded Gas Consumption in `get_stakers()` - (File: src/staking/staking.cairo)

### Summary
The `stakers` storage Vec in `staking.cairo` grows monotonically as new stakers join the protocol. Stakers are **explicitly never removed** from this Vec even after fully unstaking. The `get_stakers()` function iterates over the entire Vec on every call with no upper bound, causing gas consumption to grow linearly with the total number of historical stakers. Any unprivileged address can call `stake()` to append to this Vec, and the function is publicly callable.

### Finding Description
The `stakers` storage field is declared as a `Vec<ContractAddress>` with an explicit design note:

> "**Note**: Stakers are not removed from this vector when they unstake." [1](#0-0) 

Every call to `stake()` unconditionally pushes the caller's address onto this Vec: [2](#0-1) 

The `get_stakers()` function, which is part of the public `IStakingConsensus` interface callable by any address, iterates over the **full range** of this Vec on every invocation: [3](#0-2) 

There is no cap on the number of stakers, no pagination, and no mechanism to prune exited stakers from the Vec. Each iteration also performs multiple storage reads per staker (`is_staker_active`, `get_staker_staking_power_at_epoch`, `get_public_key_at_epoch`, `get_peer_id_at_epoch`), multiplying the per-element cost.

Additionally, `get_total_staking_power_at_epoch()`, which is called inside `get_stakers()`, itself iterates over the `btc_tokens` IterableMap: [4](#0-3) 

### Impact Explanation
As the number of historical stakers grows, every call to `get_stakers()` consumes proportionally more gas/steps. On Starknet, all calls — including view calls made from other contracts — are subject to resource limits (steps, builtins, memory). The consensus layer relies on `get_stakers()` to determine the active validator set. If the Vec grows large enough, `get_stakers()` will exceed Starknet's per-transaction resource limits, permanently breaking the ability to read the validator set. This constitutes **unbounded gas consumption** and, in the worst case, a permanent freeze of consensus-critical functionality.

**Impact classification: Medium — Unbounded gas consumption / griefing with damage to protocol.**

### Likelihood Explanation
Any address meeting the `min_stake` requirement can call `stake()` and append to the Vec. An attacker willing to lock up the minimum stake amount per address can create arbitrarily many staker entries. After unstaking (recovering funds after the exit wait window), the address remains in the Vec permanently. The cost to the attacker is only the opportunity cost of locked capital during the exit wait window, not a permanent loss. This makes the attack economically feasible over time, especially as the protocol matures and legitimate staker churn accumulates naturally.

### Recommendation
1. **Pagination**: Add a paginated variant of `get_stakers()` that accepts `offset` and `limit` parameters.
2. **Compaction on unstake**: When a staker calls `unstake_action`, swap-and-pop their address from the `stakers` Vec (or maintain a separate active-staker count to skip dead entries).
3. **Cap enforcement**: Enforce a maximum number of registered stakers, or use a separate active-stakers set that is maintained in sync with stake/unstake lifecycle events.

### Proof of Concept

1. Deploy the staking contract with `min_stake = M`.
2. Create `N` addresses, each with `M` STRK tokens.
3. For each address, call `stake()` — each call appends to `self.stakers` and the address is **never removed**.
4. After the exit wait window, call `unstake_action()` for each address to recover funds. The `stakers` Vec still contains all `N` entries.
5. Call `get_stakers(epoch_id)`. The function iterates all `N` entries.
6. As `N` grows, the call's resource consumption grows linearly until it exceeds Starknet's per-call step limit, causing the call to fail permanently.

        fn get_stakers(
            self: @ContractState, epoch_id: Epoch,
        ) -> Span<(ContractAddress, StakingPower, Option<PublicKey>, Option<PeerId>)> {
            let curr_epoch = self.get_current_epoch();
            assert!(
                curr_epoch <= epoch_id && epoch_id < curr_epoch + K.into(),
                "{}",
                Error::INVALID_EPOCH,
            );

            let (strk_total_stake, btc_total_stake) = self
                .get_total_staking_power_at_epoch(:epoch_id);

            let mut stakers: Array<
                (ContractAddress, StakingPower, Option<PublicKey>, Option<PeerId>),
            > =
                array![];
            for staker_address_ptr in self.stakers.into_iter_full_range() {
                let staker_address = staker_address_ptr.read();
                if !self.is_staker_active(:staker_address, :epoch_id) {
                    continue;
                }

                let staking_power = self
                    .get_staker_staking_power_at_epoch(
                        :staker_address, :epoch_id, :strk_total_stake, :btc_total_stake,
                    );
                if staking_power.is_zero() {
                    continue;
                }

                let public_key = self.get_public_key_at_epoch(:staker_address, :epoch_id);
                let peer_id = self.get_peer_id_at_epoch(:staker_address, :epoch_id);
                stakers.append((staker_address, staking_power, public_key, peer_id));
            }
            stakers.span()
        }

                        balance_at_epoch(trace: btc_total_stake_trace, :epoch_id);
                }
            }
