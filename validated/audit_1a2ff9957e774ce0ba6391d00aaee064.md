### Title
NFT Offer "Net Proceeds" Displays Full Amount Instead of Post-Royalty Net, Causing NFT Sellers to Approve Offers Based on Inflated Amount - (File: `packages/gui/src/components/offers/utils.ts`)

---

### Summary

In `calculateNFTRoyalties`, the `nftSellerNetAmount` field is hardcoded to `amount` (the gross asking price) because the correct subtraction of royalties was commented out. When an NFT seller views and accepts a buyer-created (`TokenForNFT`) offer, both the "Net Proceeds" label and the "You will receive" override in the offer summary display the full token amount — not the post-royalty net. Because in the `TokenForNFT` flow the accepting party (the NFT seller) pays the royalties, the seller is shown an inflated figure and may accept an offer under a materially false understanding of what they will actually receive.

---

### Finding Description

In `packages/gui/src/components/offers/utils.ts`, `calculateNFTRoyalties` computes the display values for the NFT offer UI:

```typescript
// packages/gui/src/components/offers/utils.ts  lines 312–317
const royaltyAmount: number = royaltyPercentage ? (royaltyPercentage / 100) * amount : 0;
const royaltyAmountString: string = formatAmount(royaltyAmount);
const nftSellerNetAmount: number = amount;          // ← always the gross amount
// : parseFloat(
//     (amount - parseFloat(royaltyAmountString) - makerFee).toFixed(12),
//   );
```

The correct net-proceeds calculation — `amount - royaltyAmount - makerFee` — is commented out. `nftSellerNetAmount` is therefore always equal to the gross `amount`, regardless of royalty percentage. [1](#0-0) 

This broken value propagates into two consumer sites in `NFTOfferViewer.tsx`:

**Site 1 — `overrideNFTSellerAmount`** (lines 397–402): used only for the `TokenForNFT` exchange type, it converts `nftSellerNetAmount` back to mojos and passes it to `NFTOfferSummary` as `overrideNFTSellerAmount`, which overrides the "You will receive" row shown to the accepting party. [2](#0-1) 

**Site 2 — "Net Proceeds" label** (line 596): rendered explicitly in the `TokenForNFT` viewer panel under the heading "Net Proceeds", with a tooltip that reads *"The net proceeds include the asking price, minus any associated creator fees."* [3](#0-2) 

The GUI's own tooltip for `TokenForNFT` states: *"those creator fees will be paid by the party that accepts the offer"* — i.e., the NFT seller. The seller's true net is `amount − royaltyAmount`, but both display fields show `amount`. [4](#0-3) 

The same stale value appears in `NFTOfferEditor.tsx` under the "They will receive" label when a buyer is constructing a `TokenForNFT` offer, and the `showNegativeAmountWarning` guard — which was designed to fire when the net goes negative — can never trigger because `nftSellerNetAmount` is always positive. [5](#0-4) 

---

### Impact Explanation

An NFT seller importing a `TokenForNFT` offer with a non-zero royalty sees:

| Display field | Shown value | Actual on-chain value |
|---|---|---|
| "Net Proceeds" | `amount` (e.g. 10 XCH) | `amount − royalty` (e.g. 9 XCH for 10% royalty) |
| "You will receive" (summary row) | `amount` | `amount − royalty` |

The seller clicks **Accept** based on a figure that overstates their proceeds by the full royalty amount. This satisfies the High impact criterion: offer state causes a user to approve the wrong amount. The discrepancy scales linearly with both the offer size and the royalty percentage; for high-royalty NFTs (the GUI warns at ≥ 20%) the gap is substantial.

---

### Likelihood Explanation

- Any NFT with `royaltyPercentage > 0` triggers the bug — this is the common case for creator NFTs.
- The `TokenForNFT` flow (buyer creates offer, seller accepts an imported file) is a standard, documented use case.
- No user action or attacker involvement is required; the bug fires unconditionally whenever the viewer renders a `TokenForNFT` offer for an NFT with royalties.

---

### Recommendation

Restore the commented-out net-proceeds calculation in `calculateNFTRoyalties`, conditioned on exchange type:

```typescript
// For TokenForNFT: seller pays royalties, so net = amount - royalty
// For NFTForToken: buyer pays royalties, so net = amount (unchanged)
const nftSellerNetAmount: number =
  exchangeType === NFTOfferExchangeType.TokenForNFT && royaltyPercentage
    ? parseFloat((amount - parseFloat(royaltyAmountString)).toFixed(12))
    : amount;
```

Re-enable the `showNegativeAmountWarning` guard so that offers where royalties exceed the asking price are flagged before acceptance.

---

### Proof of Concept

1. Mint an NFT with `royalty_percentage = 1000` (10%).
2. As a buyer, create a `TokenForNFT` offer: offer 10 XCH for the NFT. Export the offer file.
3. As the NFT seller, import the offer file into the GUI.
4. The offer viewer displays **"Net Proceeds: 10 XCH"** and **"You will receive: 10 XCH"**.
5. The seller accepts the offer.
6. On-chain settlement: the seller receives **9 XCH**; 1 XCH is routed to the royalty address.
7. The seller approved based on a display that overstated their proceeds by 1 XCH (11%).
