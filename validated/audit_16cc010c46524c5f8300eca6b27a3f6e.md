### Title
`MsgTurnBridge` Handler Is a No-Op — Bridge Disable Mechanism Permanently Broken and Unpermissioned - (File: x/cronos/keeper/msg_server.go)

### Summary
The `TurnBridge` message handler in `x/cronos/keeper/msg_server.go` is a complete no-op: it performs no permission check and writes no state, returning `nil, nil` unconditionally. Any unprivileged address can submit `MsgTurnBridge` and receive a success response, while the bridge state is never actually changed. The intended circuit-breaker — disabling the Gravity bridge in an emergency — is permanently non-functional.

### Finding Description
`UpdateTokenMapping` correctly gates its handler behind `k.HasPermission(ctx, msg.GetSigners(), CanChangeTokenMapping)`. The `TurnBridge` handler, which is supposed to require the `CanTurnBridge` permission (value `2`, defined alongside `CanChangeTokenMapping` in `permissions.go`), instead does nothing:

```go
// TurnBridge implements the grpc method
func (k msgServer) TurnBridge(goCtx context.Context, msg *types.MsgTurnBridge) (*types.MsgTurnBridgeResponse, error) {
    return nil, nil
}
``` [1](#0-0) 

Compare with the correctly-implemented sibling:

```go
func (k msgServer) UpdateTokenMapping(...) {
    if !k.HasPermission(ctx, msg.GetSigners(), CanChangeTokenMapping) {
        return nil, errors.Wrap(sdkerrors.ErrUnauthorized, ...)
    }
    ...
}
``` [2](#0-1) 

The permission constants are defined and the `HasPermission` helper exists and works correctly: [3](#0-2) 

The ADR and CLI documentation explicitly describe `TurnBridge` as a permissioned emergency circuit-breaker for the Gravity bridge: [4](#0-3) [5](#0-4) 

The integration test `test_gravity_turn_bridge` expects that after a privileged `turn_bridge("false")` call, `send_to_evm_chain` transactions revert. With the no-op handler, this invariant is silently broken at the protocol level. [6](#0-5) 

### Impact Explanation
Two distinct impacts:

1. **Bypass of bridge authorization / admin control (High):** The admin and any address holding `CanTurnBridge` permission can never actually disable the Gravity bridge. In an emergency (e.g., an active bridge exploit draining CRC20/CRC21 assets), the intended circuit-breaker produces a success response but leaves the bridge fully operational, allowing continued unauthorized asset transfers.

2. **Unpermissioned call accepted silently:** Because there is no permission check, any unprivileged address can submit `MsgTurnBridge` and receive `code: 0`. This misrepresents the bridge's disabled/enabled state to off-chain monitors and tooling that rely on the transaction result to infer bridge status.

### Likelihood Explanation
The entry path is fully unprivileged: any address can construct and broadcast a `MsgTurnBridge` transaction. The missing logic is in the production keeper handler, not behind any feature flag. The bug is reachable on every block.

### Recommendation
Implement `TurnBridge` analogously to `UpdateTokenMapping`:

```go
func (k msgServer) TurnBridge(goCtx context.Context, msg *types.MsgTurnBridge) (*types.MsgTurnBridgeResponse, error) {
    ctx := sdk.UnwrapSDKContext(goCtx)
    if !k.HasPermission(ctx, msg.GetSigners(), CanTurnBridge) {
        return nil, errors.Wrap(sdkerrors.ErrUnauthorized, "msg sender is not authorized")
    }
    params := k.GetParams(ctx)
    params.EnableAutoDeployment = msg.Enable  // or a dedicated BridgeActive param
    if err := k.SetParams(ctx, params); err != nil {
        return nil, err
    }
    return &types.MsgTurnBridgeResponse{}, nil
}
```

The EVM hook that processes `__CronosSendToEvmChain` events must then read this flag and reject bridge-out operations when it is `false`.

### Proof of Concept

1. Any address (e.g., `signer1` with no permissions) broadcasts:
   ```
   cronosd tx cronos turn-bridge false --from signer1
   ```
2. The transaction is accepted on-chain with `code: 0` and an empty response — no error, no state change.
3. A permissioned admin subsequently broadcasts the same message expecting to disable the bridge:
   ```
   cronosd tx cronos turn-bridge false --from admin
   ```
4. Again `code: 0`, but the bridge state is unchanged. `send_to_evm_chain` calls on CRC21 contracts continue to succeed and emit `__CronosSendToEvmChain` events that the keeper processes normally, continuing to burn/transfer user tokens and queue outbound Gravity batches.
5. The admin has no on-chain mechanism to halt the bridge; the circuit-breaker is permanently non-functional.

// CmdTurnBridge returns a CLI command handler for enable or disable the bridge
func CmdTurnBridge() *cobra.Command {
	cmd := &cobra.Command{
		Use:   "turn-bridge [true/false]",
		Short: "Turn Bridge",
		Args:  cobra.ExactArgs(1),
		RunE: func(cmd *cobra.Command, args []string) error {
			clientCtx, err := client.GetClientTxContext(cmd)
			if err != nil {
				return err
			}

			enable, err := strconv.ParseBool(args[0])
			if err != nil {
				return err
			}
			msg := types.NewMsgTurnBridge(clientCtx.GetFromAddress().String(), enable)
			if err := msg.ValidateBasic(); err != nil {
				return err
			}
			return tx.GenerateOrBroadcastTxCLI(clientCtx, cmd.Flags(), msg)
		},
	}

	flags.AddTxFlagsToCmd(cmd)
	return cmd
}
