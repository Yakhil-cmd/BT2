### Title
Unbounded `MessageBuffer` Growth via Crafted Messages with Arbitrary Waitpoints Causes Memory Exhaustion in All Threshold Protocols - (File: `src/protocol/internal.rs`)

---

### Summary

The `MessageBuffer` inside `Comms` accepts and permanently stores any inbound message whose first 40 bytes parse as a valid `MessageHeader` (32-byte channel tag + 8-byte `u64` waitpoint), with no bound on the number of distinct headers stored and no eviction mechanism. A malicious participant can flood the buffer with messages addressed to waitpoints the honest protocol code never polls, causing unbounded heap growth. The codebase itself contains a test that explicitly confirms this attack succeeds.

---

### Finding Description

**Root cause — `push_message` with no waitpoint validation:**

`Comms::push_message` is the sole entry point for all inbound messages delivered via `Protocol::message`:

```rust
// src/protocol/internal.rs:286-296
fn push_message(&self, from: Participant, message: MessageData) {
    if message.len() < MessageHeader::LEN {
        return;
    }
    let Some(header) = MessageHeader::from_bytes(&message) else {
        return;
    };
    self.incoming.push(header, from, message);
}
```

The only validation performed is that the message is at least 40 bytes long and that the first 40 bytes are parseable as a `MessageHeader`. Any `u64` value is a valid waitpoint — there is no check that the waitpoint falls within the range of waitpoints the current protocol phase will ever poll. [1](#0-0) 

**Unbounded storage — `MessageBuffer` and `SubMessageQueue`:**

`MessageBuffer` is a `HashMap<MessageHeader, SubMessageQueue>` with no capacity limit. Each distinct `(channel_tag, waitpoint)` pair creates a new `HashMap` entry. Each `SubMessageQueue` wraps a `futures::channel::mpsc::unbounded()` channel — literally named "unbounded" — so even a single slot can hold an unlimited number of messages:

```rust

    messages: Arc<std::sync::Mutex<HashMap<MessageHeader, SubMessageQueue>>>,

``` [2](#0-1) 

```rust

**No eviction — messages for unpolled waitpoints are never freed:**

The `MessageBuffer::pop` function only removes a message when the protocol explicitly awaits a specific `(channel, waitpoint)` header. Messages injected for waitpoints the honest protocol never polls remain in the `HashMap` for the entire lifetime of the protocol session. There is no purging, expiry, or size-cap mechanism anywhere in the module. [4](#0-3) 

**Codebase-confirmed attack — the test asserts the attack succeeds:**

The repository contains a test that explicitly demonstrates and asserts this attack path:

```rust
// src/protocol/internal.rs:532-554
fn attacker_can_fill_message_buffer_with_unused_waitpoints() {
    let comms = Comms::new();
    let attacker = Participant::from(99_u32);
    let attack_count = 512_u64;

    for i in 0..attack_count {
        let header =
            MessageHeader::new(ChannelTag::root_shared()).with_waitpoint(1_000_000 + i);
        let mut message = header.to_bytes().to_vec();
        message.extend_from_slice(&i.to_le_bytes());
        // Attacker injects messages for waitpoints the honest code never polls.
        comms.push_message(attacker, message);
    }
    // All 512 injected messages remain in the buffer.
    assert!(messages.len() == usize::try_from(attack_count).unwrap());
}
``` [5](#0-4) 

The test comment reads: *"Attacker injects messages for waitpoints the honest code never polls."* The assertion confirms all injected messages persist. This is not a theoretical concern — the codebase explicitly acknowledges and demonstrates the attack.

**Attacker-controlled entry path:**

The `Protocol::message(from, data)` method is the public API through which the orchestrating application delivers network messages to a running protocol instance: [6](#0-5) 

```rust
// src/protocol/internal.rs:512-514
fn message(&mut self, from: Participant, data: MessageData) {
    self.comms.push_message(from, data);
}
``` [7](#0-6) 

A malicious participant in any threshold protocol session (keygen, reshare, refresh, sign, CKD) can craft messages whose first 40 bytes encode the correct shared-channel tag (`ChannelTag::root_shared()`, which is a deterministic SHA-256 hash of a public domain constant) combined with an arbitrary `u64` waitpoint value. The channel tag is not secret — it is derived from a public constant `NEAR_CHANNEL_TAGS_DOMAIN` and the fixed string `"root shared"`, so any participant can compute it. The attacker sends one such message per distinct fake waitpoint, each creating a new `HashMap` entry that is never freed. [8](#0-7) 

---

### Impact Explanation

**Impact: Medium — Griefing / resource exhaustion causing unbounded memory growth beyond documented behavior.**

A malicious participant in any active threshold protocol session (DKG, sign, reshare, refresh, CKD) can cause the honest node's heap to grow without bound by sending a continuous stream of messages with distinct fake waitpoints. Each message costs one `HashMap` entry plus one `mpsc::unbounded` channel allocation. At scale, this exhausts the host process's memory, crashing the node and permanently denying signing, key generation, reshare, refresh, or CKD for all honest parties in that session.

The README explicitly documents that the library waits indefinitely for expected messages and places timeout responsibility on the caller. However, it does not document or excuse unbounded memory growth from *unexpected* messages with fake waitpoints — this is beyond documented behavior. [9](#0-8) 

---

### Likelihood Explanation

**High likelihood.** The attack requires only that the adversary be a participant in a threshold protocol session — a role that is explicitly within scope (malicious participant). The channel tag used for the shared channel is a deterministic, publicly computable SHA-256 hash. The `waitpoint` field is an arbitrary `u64` with no validation. The attack requires sending crafted byte strings of exactly 40+ bytes, which is trivially achievable. The codebase's own test confirms the attack works with 512 messages and asserts all remain in the buffer.

---

### Recommendation

**Short term:** In `Comms::push_message`, reject messages whose parsed `waitpoint` exceeds the maximum waitpoint the current protocol phase will ever use. Alternatively, cap the total number of entries in `MessageBuffer` (e.g., `N * max_waitpoints_per_session`) and drop messages that would exceed the cap.

**Long term:** Replace `futures::channel::mpsc::unbounded()` in `SubMessageQueue` with a bounded channel. Add a maximum-size invariant to `MessageBuffer` enforced at insertion time, and evict or reject messages for unknown headers once the cap is reached.

---

### Proof of Concept

The existing test at `src/protocol/internal.rs:532–554` is a direct proof of concept. To extend it to a realistic protocol context:

1. Start an honest `keygen` or `sign` session with `N` participants, one of which is attacker-controlled.
2. The attacker computes `ChannelTag::root_shared()` (deterministic, public) and constructs messages with headers `(root_shared_tag, 1_000_000)`, `(root_shared_tag, 1_000_001)`, ..., `(root_shared_tag, 1_000_000 + K)` for arbitrarily large `K`.
3. The attacker delivers each message to the honest node via the `Protocol::message` API (i.e., by sending it over the network).
4. Each message creates a new `HashMap<MessageHeader, SubMessageQueue>` entry that is never polled and never freed.
5. As `K` grows, the honest node's heap grows proportionally until OOM, crashing the process and permanently denying the threshold operation. [5](#0-4) [10](#0-9)
