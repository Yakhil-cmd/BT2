### Title
Malicious CKD Participant Can Corrupt CKD Output by Sending Unvalidated Arbitrary Contribution Values — (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

The `do_ckd_coordinator` function in the CKD protocol blindly aggregates `(norm_big_y, norm_big_c)` values received from each participant without any proof of correct computation. A malicious participant can send arbitrary group elements in place of their honest contribution, corrupting the final `CKDOutput` delivered to the TEE. This is directly analogous to the external report's root cause: an externally supplied value is used in a critical computation without being validated against the locally derivable correct value.

---

### Finding Description

In `do_ckd_coordinator` the coordinator collects each participant's share and accumulates it unconditionally:

```rust
// src/confidential_key_derivation/protocol.rs  lines 50-55
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
```

Each honest participant is supposed to compute and send:

```
norm_big_y  =  λ_i · y_i · G
norm_big_c  =  λ_i · (x_i · H(pk ‖ app_id) + y_i · app_pk)
```

as implemented in `compute_signature_share` (lines 148–181). However, the coordinator performs **no verification** that the received pair satisfies this relation. There is no commitment, no NIZK proof, and no consistency check against the participant's public key share. Any participant can substitute arbitrary `ElementG1` values.

The contrast with the OT-based ECDSA presign is instructive: that protocol validates its analogous aggregated values with explicit equality checks against public commitments (e.g., `big_e != (ProjectivePoint::GENERATOR * e).to_affine()` at line 127 of `src/ecdsa/ot_based_ecdsa/presign.rs`, and the `alpha*G =?= K + A` check at line 162). The CKD coordinator has no equivalent guard.

---

### Impact Explanation

The intended decryption identity is:

```
C − app_sk · Y  =  Σ λ_i · x_i · H(pk ‖ app_id)  =  msk · H(pk ‖ app_id)
```

If participant `i` replaces its honest `norm_big_c` with an arbitrary value `norm_big_c'`, the coordinator computes a corrupted `C'`. The TEE then decrypts to a value `msk' · H(pk ‖ app_id)` where `msk' ≠ msk`. The TEE has no independent means to detect this: it does not know `msk` and cannot verify the output. Honest parties therefore accept and act on an incorrect confidential derived key.

This matches the allowed impact: **High — Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.**

---

### Likelihood Explanation

The attack requires only that the adversary be a registered participant in the CKD session. No leaked keys, no cryptographic breaks, and no coordinator compromise are needed. A single malicious participant out of `n` is sufficient to corrupt the output for every honest party, including the TEE. The entry path is the standard `ckd()` public API followed by sending a crafted message at the single protocol waitpoint.

---

### Recommendation

Add a zero-knowledge proof of correct computation alongside each participant's `(norm_big_y, norm_big_c)` contribution. Concretely, each participant should prove in zero knowledge that:

1. `norm_big_y` is of the form `λ_i · y_i · G` for some scalar `y_i`, and  
2. `norm_big_c` is of the form `λ_i · (x_i · H(pk ‖ app_id) + y_i · app_pk)` using the same `y_i` and the participant's committed public key share `λ_i · x_i · G2`.

A standard Sigma protocol (Chaum–Pedersen or a linear combination proof) over the BLS12-381 G1 group suffices. The coordinator must verify all proofs before aggregating.

---

### Proof of Concept

**Setup**: 3-participant CKD session; participant `P2` is malicious.

**Honest flow** (participants `P1`, `P3`): each calls `compute_signature_share` and sends the result to the coordinator.

**Malicious action** (`P2`): instead of calling `compute_signature_share`, `P2` sends:

```rust
// Malicious participant sends zeroed-out contribution
let fake_output = CKDOutput::new(ElementG1::identity(), ElementG1::identity());
chan.send_private(waitpoint, coordinator, &fake_output)?;
```

**Coordinator aggregation** (lines 50–55): the coordinator sums all contributions including `P2`'s zeroed pair. The resulting `(Y, C)` is missing `P2`'s Lagrange-weighted term.

**TEE decryption**: `C − app_sk · Y` yields `(msk − λ_2 · x_2) · H(pk ‖ app_id)`, a key that is neither the intended `msk · H(pk ‖ app_id)` nor any key the TEE can detect as wrong. fn compute_signature_share(
```rust
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
```
