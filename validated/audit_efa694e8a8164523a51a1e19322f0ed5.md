### Title
Unbounded Loop in `calculate_rewards` Over Pool Member Balance Trace Can Permanently Freeze Unclaimed Yield — (`File: src/pool/pool.cairo`)

---

### Summary

The `calculate_rewards` internal function in `pool.cairo` contains an unbounded loop that iterates over every entry in a pool member's `pool_member_epoch_balance` trace since their last claim. Because there is no cap on how many balance-change entries a delegator can accumulate, a delegator who makes sufficiently many balance changes across epochs without claiming will eventually cause `claim_rewards` to exceed the Starknet transaction gas limit, permanently freezing their unclaimed yield. The code itself acknowledges this risk with the comment: *"This loop is unbounded but unlikely to exceed gas limits."*

---

### Finding Description

In `src/pool/pool.cairo`, the internal function `calculate_rewards` iterates over the pool member's balance trace from `entry_to_claim_from` up to `pool_member_trace_length`:

```
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    ...
    entry_to_claim_from += 1;
}
``` [1](#0-0) 

Each call to `add_to_delegation_pool` or `exit_delegation_pool_intent` (partial) inserts a new entry into the `pool_member_epoch_balance` trace keyed by epoch. There is no maximum cap on the number of entries. [2](#0-1) 

The `entry_to_claim_from` cursor advances only when `claim_rewards` is successfully executed. If a delegator accumulates N unclaimed balance-change entries across N distinct epochs, the next `claim_rewards` call must iterate all N entries in a single transaction. Once N is large enough to exceed the Starknet gas limit, the transaction reverts and the rewards are permanently unclaimable.

A secondary unbounded loop exists in `get_stakers` in `staking.cairo`, which iterates over `self.stakers.into_iter_full_range()` — a vector that grows with every new staker registration (`self.stakers.push(staker_address)` at line 348) and is **never pruned** even after `unstake_action`. However, `get_stakers` is a view function primarily called off-chain by the consensus layer, so its gas impact is less direct. [3](#0-2) [4](#0-3) 

---

### Impact Explanation

**High — Permanent freezing of unclaimed yield.**

Once the `pool_member_epoch_balance` trace for a delegator grows beyond the gas-iteration threshold, every subsequent call to `claim_rewards` for that delegator will revert. The accumulated STRK rewards remain locked in the pool contract with no recovery path, since there is no administrative function to reset or paginate the trace.

---

### Likelihood Explanation

**Medium.** A delegator who calls `add_to_delegation_pool` once per epoch (the minimum interval for a new trace entry) and never claims will accumulate one entry per epoch. Starknet epochs are short (measured in blocks), so over a long operational period a legitimate power user who frequently adjusts their delegation position can reach the gas limit organically. A deliberate attacker who controls a delegator address can accelerate this by making one small balance change per epoch across many epochs. The minimum stake requirement for each change is low (any non-zero amount above `min_for_rewards`), making the attack affordable.

---

### Recommendation

1. **Paginate `claim_rewards`**: Accept a `max_entries` parameter so the loop processes at most N trace entries per call, advancing `entry_to_claim_from` incrementally across multiple transactions.
2. **Cap balance-change frequency**: Enforce a minimum interval (e.g., one change per epoch) and reject redundant same-epoch insertions to bound trace growth.
3. **Perform gas simulations**: As recommended in the analogous Compound audit finding, identify the maximum trace length that fits within the Starknet transaction gas limit and enforce it as a hard cap on `pool_member_epoch_balance` length.

---

### Proof of Concept

1. Delegator `D` enters pool `P` with `enter_delegation_pool`.
2. For each of `N` consecutive epochs, `D` calls `add_to_delegation_pool` with a minimal amount. Each call inserts a new entry into `pool_member_epoch_balance` for `D`.
3. `D` never calls `claim_rewards` during this period, so `entry_to_claim_from` remains at its initial value.
4. After `N` epochs, `D` (or anyone) calls `claim_rewards`. The `calculate_rewards` loop must iterate all `N` entries.
5. When `N` exceeds the gas-per-iteration threshold for Starknet, the transaction reverts with an out-of-gas error.
6. All subsequent `claim_rewards` calls for `D` also revert. `D`'s accumulated STRK yield is permanently frozen in pool contract `P`. [5](#0-4)

```text
        fn calculate_rewards(
            self: @ContractState,
            pool_member: ContractAddress,
            from_checkpoint: PoolMemberCheckpoint,
            until_checkpoint: PoolMemberCheckpoint,
            mut entry_to_claim_from: VecIndex,
        ) -> (Amount, VecIndex) {
            let pool_member_trace = self.pool_member_epoch_balance.entry(pool_member);
            // Note: `until_epoch` is the current epoch.
            let until_epoch = until_checkpoint.epoch();

            let mut rewards = 0;

            let pool_member_trace_length = pool_member_trace.length();

            let mut from_sigma = self.find_sigma(from_checkpoint, curr_epoch: until_epoch);
            let mut from_balance = from_checkpoint.balance();

            let base_value = self.staking_rewards_base_value.read();

            // **Note**: The loop iterates over the balance changes in the pool member's balance
            // trace. This loop is unbounded but unlikely to exceed gas limits.
            while entry_to_claim_from < pool_member_trace_length {
                let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
                // If the balance change is after `until_epoch` (and therefore does not affect
                // the current reward computation), exit the loop.
                if pool_member_checkpoint.epoch() >= until_epoch {
                    break;
                }

                // Compute rewards from (inclusive) the previous balance change (or from
                // `from_checkpoint`) to (exclusive) the current entry.
                let to_sigma = self.find_sigma(pool_member_checkpoint, curr_epoch: until_epoch);
                rewards +=
                    compute_rewards_rounded_down(
                        amount: from_balance, interest: to_sigma - from_sigma, :base_value,
                    );
                from_sigma = to_sigma;
                from_balance = pool_member_checkpoint.balance();
                entry_to_claim_from += 1;
            }

            // Compute the remaining rewards from (inclusive) the last visited balance change in
            // `pool_member_trace` (or from `from_checkpoint`) to (exclusive) `until_checkpoint`.
            let to_sigma = self.find_sigma(until_checkpoint, curr_epoch: until_epoch);
            rewards +=
                compute_rewards_rounded_down(
                    amount: from_balance, interest: to_sigma - from_sigma, :base_value,
                );

            (rewards, entry_to_claim_from)
        }
