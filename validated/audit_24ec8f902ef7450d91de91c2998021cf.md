Let me look at the mempool's `validate_tx` and the `ValidationArgs` to understand if there's any signature check there, and also look at the `validate_nonce` function in the gateway.

### Title
`skip_stateful_validations` Bypasses Signature Verification Without Confirming Deploy-Account Presence — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

`skip_stateful_validations` is designed to skip the `__validate__` entry-point call (signature verification) for an invoke transaction with nonce=1 when the account's deploy-account has not yet been processed. The guard it uses — `account_tx_in_pool_or_recent_block` — returns `true` for **any** transaction from the account in the pool or a recent committed block, not specifically a deploy-account transaction. An attacker who observes that a deployed account (on-chain nonce = 0) has a nonce-0 invoke sitting in the mempool can submit a nonce-1 invoke with an **invalid signature** that passes all gateway checks and is admitted to the mempool without signature verification.

---

### Finding Description

The UX feature is documented in the code:

> "Check if validation of an invoke transaction should be skipped due to deploy_account not being processed yet."

The trigger condition is:

```
tx.nonce() == Nonce(Felt::ONE)  &&  account_nonce == Nonce(Felt::ZERO)
```

When both are true, the gateway calls:

```rust
mempool_client.account_tx_in_pool_or_recent_block(tx.sender_address()).await
```

and uses the boolean result directly as `skip_validate`. [1](#0-0) 

`account_tx_in_pool_or_recent_block` is implemented as:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [2](#0-1) 

It returns `true` whenever the account has **any** transaction in the pool — an invoke, a declare, or a deploy-account. There is no type filter.

When `skip_validate` is `true`, `run_validate_entry_point` sets `execution_flags.validate = false`:

```rust
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [3](#0-2) 

`StatefulValidator::perform_validations` then short-circuits before calling `__validate__`:

```rust
tx.perform_pre_validation_stage(self.state(), &tx_context)?;
if !tx.execution_flags.validate {
    return Ok(());
}
// `__validate__` call.
``` [4](#0-3) 

The mempool's `validate_tx` — called earlier in `run_pre_validation_checks` — only checks duplicate tx-hash, nonce ordering, and fee-escalation rules; it carries no signature data and performs no signature check. [5](#0-4) [6](#0-5) 

The gateway's `validate_nonce` for a non-declare, non-deploy-account transaction only checks that `account_nonce <= tx_nonce <= account_nonce + max_allowed_nonce_gap`, so nonce=1 against account_nonce=0 passes cleanly. [7](#0-6) 

**Broken invariant (direct analog to the external bug):**
The external bug unlocked tokens whenever delegation ended, without checking `_totalDelegated >= _purchased`. Here, the gateway skips signature validation whenever `account_tx_in_pool_or_recent_block` returns `true`, without checking that the matching pool entry is specifically a deploy-account transaction. The required condition (`account has a pending deploy-account`) is replaced by a weaker one (`account has any pending transaction`).

---

### Impact Explanation

An attacker can submit an invoke transaction with a **forged or invalid signature** from any deployed account that currently has a nonce-0 invoke in the mempool. The transaction bypasses the `__validate__` entry-point check at the gateway and is admitted to the mempool. It will fail during blockifier execution (the blockifier always validates), but it:

- Pollutes the mempool with unauthenticated transactions.
- Consumes sequencer resources (mempool slots, batcher execution cycles, fee-estimation overhead).
- Can be repeated at scale to degrade mempool quality for targeted accounts.

**Impact category:** High — Mempool/gateway admission accepts invalid transactions before sequencing.

---

### Likelihood Explanation

- The attacker needs only to observe that a target account (on-chain nonce = 0) has a nonce-0 invoke in the public mempool — no privileged access required.
- The attack is stateless and repeatable: a new invalid nonce-1 invoke can be submitted every time the previous one is evicted or the nonce-0 invoke is replaced.
- No special contract or cryptographic capability is needed; any well-formed invoke transaction body with a garbage signature suffices.

---

### Recommendation

Replace the type-agnostic `account_tx_in_pool_or_recent_block` check with a check that specifically confirms a **deploy-account** transaction for the sender address is present in the mempool. Expose a dedicated mempool query such as `deploy_account_in_pool(address) -> bool` that inspects the transaction type, and use that result as the skip condition. Alternatively, add a type assertion inside `skip_stateful_validations` before returning `true`.

---

### Proof of Concept

```
1. Deploy account A on-chain (class hash set, on-chain nonce = 0).

2. Account A submits a legitimate invoke with nonce=0.
   → Gateway: validate_nonce passes (0 == 0), validate_by_mempool passes,
     skip_stateful_validations: nonce != 1 → false, __validate__ runs, signature OK.
   → Admitted to mempool. tx_pool.contains_account(A) == true.

3. Attacker crafts invoke from A with nonce=1 and signature = [0xdeadbeef, ...].

4. Gateway stateful path:
   a. validate_nonce: account_nonce=0, tx_nonce=1, 0 <= 1 <= 0+gap → passes.
   b. validate_by_mempool: checks hash uniqueness + nonce ordering → passes
      (nonce=1 is a new slot, no duplicate).
   c. skip_stateful_validations:
        tx.nonce() == 1  ✓
        account_nonce == 0  ✓
        account_tx_in_pool_or_recent_block(A) → true (nonce-0 invoke is in pool)  ✓
      → skip_validate = true
   d. run_validate_entry_point: execution_flags.validate = false
      → StatefulValidator returns Ok(()) without calling __validate__.

5. Invalid nonce-1 invoke is forwarded to the mempool and admitted.
   Signature is never verified at the gateway layer.
```

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
