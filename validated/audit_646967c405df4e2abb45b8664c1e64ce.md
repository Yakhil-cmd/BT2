### Title
`max_l2_gas_amount` Cap Defined but Not Enforced for Declare Transactions at Gateway Admission - (File: `crates/apollo_gateway/src/stateless_transaction_validator.rs`)

### Summary
`StatelessTransactionValidator::validate_resource_bounds` defines and enforces a `max_l2_gas_amount` cap for Invoke and DeployAccount transactions, but explicitly skips this check for Declare transactions. An attacker can submit a Declare transaction with an arbitrarily large `l2_gas.max_amount` (up to `u64::MAX`) that passes all gateway stateless checks and is admitted to the mempool, violating the cap invariant the config declares.

### Finding Description

`StatelessTransactionValidatorConfig` declares `max_l2_gas_amount` (production default: `1_210_000_000`) as the maximum allowed L2 gas amount for transactions: [1](#0-0) 

The enforcement in `validate_resource_bounds` reads:

```rust
// TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
if let RpcTransaction::Declare(_) = tx {
} else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh { ... });
}
``` [2](#0-1) 

The `if let RpcTransaction::Declare(_) = tx { }` branch is an empty arm that intentionally falls through, meaning the cap check is entirely skipped for Declare transactions. The TODO comment confirms this is a known gap. The test `valid_l2_gas_amount_on_declare` explicitly asserts that a Declare transaction with `l2_gas.max_amount = 200` passes when `max_l2_gas_amount = 100`: [3](#0-2) 

The `min_gas_price` check (line 71) still applies to `l2_gas.max_price_per_unit` for all transaction types including Declare, so the attacker must set `l2_gas.max_price_per_unit >= min_gas_price` (8 GWei in production). With `l2_gas.max_amount = u64::MAX` and `l2_gas.max_price_per_unit = 8_000_000_000`, `max_possible_fee()` saturates to `u128::MAX`: [4](#0-3) 

### Impact Explanation

A Declare transaction with `l2_gas.max_amount` exceeding `max_l2_gas_amount` passes `StatelessTransactionValidator::validate` and is forwarded to the stateful path and mempool. The stateful validator's `validate_resource_bounds` only checks `l2_gas.max_price_per_unit` against the previous block price — it has no `max_amount` check at all: [5](#0-4) 

The transaction is therefore admitted to the mempool. At blockifier pre-validation, `verify_can_pay_committed_bounds` will reject it if the account cannot cover `u128::MAX` fee, but by then gateway and mempool resources have been consumed. This matches the **High** impact: *"Mempool/gateway/RPC admission accepts invalid transactions before sequencing."*

### Likelihood Explanation

Any unprivileged user can craft a valid Declare transaction (with a real Sierra class) and set `l2_gas.max_amount` to any value above `max_l2_gas_amount`. No special account state is required to reach the gateway. The bypass is unconditional for all Declare transactions.

### Recommendation

Remove the empty `if let RpcTransaction::Declare(_) = tx { }` arm and apply the `max_l2_gas_amount` check uniformly to all transaction types, resolving the TODO:

```rust
if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
        gas_amount: resource_bounds.l2_gas.max_amount,
        max_gas_amount: self.config.max_l2_gas_amount,
    });
}
``` [2](#0-1) 

### Proof of Concept

1. Construct a valid `RpcDeclareTransactionV3` with any Sierra class.
2. Set `resource_bounds.l2_gas.max_amount = GasAmount(u64::MAX)`.
3. Set `resource_bounds.l2_gas.max_price_per_unit = GasPrice(8_000_000_000)` (meets `min_gas_price`).
4. Set `resource_bounds.l1_gas` and `resource_bounds.l1_data_gas` to any non-zero amount to pass the zero-bounds check.
5. Submit to the gateway. `StatelessTransactionValidator::validate` returns `Ok(())` — the `max_l2_gas_amount` branch is skipped.
6. The transaction is forwarded to the stateful validator and admitted to the mempool, despite `l2_gas.max_amount` being `18_446_744_073_709_551_615` — over 15,000× the configured cap of `1_210_000_000`. [6](#0-5)

        match self {

            }
