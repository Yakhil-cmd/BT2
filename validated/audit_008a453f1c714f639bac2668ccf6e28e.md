### Title
Unbounded `timelock` in `OracleValueStopLossExtension` causes `uint32` truncation in `_afterTimelock`, silently bypassing LP-protection delay — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

---

### Summary

`OracleValueStopLossExtension._afterTimelock` computes `uint32(block.timestamp + timelock)` where `timelock` is a `uint32` with no upper-bound validation. When `block.timestamp + timelock` exceeds `type(uint32).max`, the result silently wraps to a past timestamp. Every subsequent `_requireElapsed` check passes immediately, letting the pool admin bypass the timelock that is the sole LP-protection mechanism for drawdown, decay, and high-watermark changes.

---

### Finding Description

`_afterTimelock` is the single function that computes the `executeAfter` deadline for every timelocked parameter change in the extension:

```solidity
// OracleValueStopLossExtension.sol L297-299
function _afterTimelock(address pool_) private view returns (uint32) {
    return uint32(block.timestamp + oracleStopLossConfig[pool_].timelock);
}
``` [1](#0-0) 

`block.timestamp` is `uint256`; `timelock` is `uint32`. The addition is performed in `uint256` space and then **truncated** to `uint32`. Solidity 0.8+ overflow protection does not apply to explicit narrowing casts — the truncation is silent.

`type(uint32).max` is 4,294,967,295. At the time of writing (July 2026), `block.timestamp ≈ 1,753,000,000`. Therefore any `timelock` value above approximately **2,541,967,295 seconds (~80.6 years)** causes the sum to exceed `type(uint32).max`, and the cast wraps the result to a timestamp that is already in the past.

`_requireElapsed` then compares `block.timestamp` (a `uint256`) against the wrapped `uint32` value, which is implicitly promoted to `uint256`:

```solidity
// OracleValueStopLossExtension.sol L301-303
function _requireElapsed(uint32 executeAfter) private view {
    if (block.timestamp < executeAfter) revert OracleStopLossTimelockNotElapsed(executeAfter, block.timestamp);
}
``` [2](#0-1) 

Because `executeAfter` is a past timestamp, `block.timestamp < executeAfter` is `false`, and the guard never reverts. The pending change is immediately executable.

The `initialize` function validates `drawdownE6` and `decayPerSecondE8` but performs **no validation on `timelock`**:

```solidity
// OracleValueStopLossExtension.sol L56-58
(uint32 drawdownE6, uint32 decayPerSecondE8, uint32 timelock) = abi.decode(data, (uint32, uint32, uint32));
_validateDrawdown(drawdownE6);
_validateDecay(decayPerSecondE8);
// timelock is stored without any upper-bound check
``` [3](#0-2) 

`proposeOracleStopLossTimelock` also has no upper-bound check on `newTimelock`, so the overflow can be introduced or preserved through the update path as well. [4](#0-3) 

Every propose-then-execute flow for drawdown, decay, and high watermarks calls `_afterTimelock` to compute `executeAfter`: [5](#0-4) [6](#0-5) [7](#0-6) 

All four parameter families are therefore affected by the same overflow.

---

### Impact Explanation

The `OracleValueStopLossExtension` is explicitly designed so that drawdown and decay changes are **timelocked so LPs can react** (NatSpec line 16). The timelock is the only mechanism that bounds the pool admin's power over LP funds in this extension.

With the bypass active, the pool admin can:

1. **Disable the stop-loss immediately** by setting `drawdownE6 = 1e6` (100% drawdown floor = stop-loss never triggers) or `decayPerSecondE8 = 1e8` (watermarks decay to zero in one second, so the ratchet never accumulates a meaningful high-water mark).
2. **Raise high watermarks immediately** to values that will never be breached, again neutralising the guard.

Once the stop-loss is neutralised, swaps that drain LP value in a single direction are no longer blocked, exposing LP principal to direct loss.

---

### Likelihood Explanation

The trigger requires the pool admin to supply a `timelock` value above ~2.54 billion seconds at pool creation (or via a subsequent `proposeOracleStopLossTimelock` call). This is a valid `uint32` value and passes all existing validation. A pool admin who presents a pool with an apparently large timelock (e.g., `type(uint32).max`) to attract LP deposits, then immediately disables the stop-loss, is the realistic attack path. The pool admin is semi-trusted only within the bounds of the timelock; this bug removes that bound.

---

### Recommendation

Add an upper-bound check on `timelock` in both `initialize` and `proposeOracleStopLossTimelock`. A reasonable maximum is 30 days (2,592,000 seconds), well below the overflow threshold:

```solidity
uint32 private constant MAX_TIMELOCK = 30 days; // 2_592_000 — safely below uint32 overflow

function _validateTimelock(uint32 timelock) private pure {
    if (timelock > MAX_TIMELOCK) revert OracleStopLossTimelockTooLarge(timelock);
}
```

Apply `_validateTimelock` in `initialize` alongside the existing `_validateDrawdown` / `_validateDecay` calls, and at the top of `proposeOracleStopLossTimelock` before storing `newTimelock`.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.35;

// Scenario: pool admin sets timelock = type(uint32).max at initialization.
// block.timestamp ≈ 1_753_000_000 (July 2026)
// type(uint32).max = 4_294_967_295
// sum = 1_753_000_000 + 4_294_967_295 = 6_047_967_295
// uint32(6_047_967_295) = 6_047_967_295 - 4_294_967_296 = 1_752_999_999
// 1_752_999_999 < block.timestamp (1_753_000_000) → _requireElapsed passes immediately

// 1. Pool creator calls createPool with extensionInitData encoding
//    (drawdownE6=500_000, decayPerSecondE8=58, timelock=type(uint32).max)
//    → initialize stores timelock = type(uint32).max, no revert.

// 2. Admin calls proposeOracleStopLossDrawdown(pool, 1_000_000)
//    → _afterTimelock returns uint32(block.timestamp + type(uint32).max)
//      = a timestamp already in the past.

// 3. Admin immediately calls executeOracleStopLossDrawdown(pool)
//    → _requireElapsed(past_timestamp) passes (block.timestamp >= past_timestamp).
//    → drawdownE6 set to 1_000_000 (100%) — stop-loss disabled.

// 4. Swaps that drain LP value in one direction are no longer blocked.
```
