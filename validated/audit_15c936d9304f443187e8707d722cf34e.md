### Title
Simulation Underestimates Gas for L1 Transactions When `eip1559_basefee == 0` — (`basic_bootloader/src/bootloader/transaction_flow/zk/process_l1_transaction.rs`)

---

### Summary

When `eip1559_basefee == 0`, the simulation path for L1→L2 transactions computes `native_per_gas = 0`, triggering unlimited-native mode and suppressing the `delta_gas` adjustment. Actual execution with `gas_price == 0` uses the hardcoded constant `FREE_L1_TX_NATIVE_PER_GAS = 10000`, which applies the delta_gas correction. The simulation therefore systematically underestimates `gas_used`, mirroring the original report's pattern of an estimation function diverging from actual execution under a zero-state condition.

---

### Finding Description

**Root cause — `prepare_and_check_resources` in `process_l1_transaction.rs`:**

When `Config::SIMULATION && gas_price.is_zero()`, `native_per_gas` is derived from the block base fee:

```rust
// process_l1_transaction.rs lines 455-467
let native_per_gas = if is_priority_op {
    if gas_price.is_zero() {
        if Config::SIMULATION {
            u256_try_to_u64(&system.get_eip1559_basefee().div_ceil(native_price))
                .unwrap_or_else(|| { ... u64::MAX })
        } else {
            FREE_L1_TX_NATIVE_PER_GAS   // = 10_000
        }
    } else { ... }
```

When `basefee == 0`, `0u256.div_ceil(native_price) == 0`, so `native_per_gas = 0`. This is then passed to `create_resources_for_tx` as `free_native = (native_per_gas == 0) = true`, giving the transaction an unlimited native budget (`u64::MAX - 1`).

**Divergence in `compute_gas_refund` in `refund_calculation.rs`:**

```rust
// refund_calculation.rs lines 59-73
let full_native_limit = if cfg!(feature = "unlimited_native") || native_per_gas == 0 {
    u64::MAX - 1
} else {
    gas_limit.saturating_mul(native_per_gas)
};
let native_used = full_native_limit.saturating_sub(resources.native().remaining().as_u64());

#[cfg(not(feature = "unlimited_native"))]
{
    let delta_gas = if native_per_gas == 0 {
        0   // ← no delta_gas adjustment in simulation when basefee == 0
    } else {
        (native_used / native_per_gas) as i64 - (gas_used as i64)
    };
    if delta_gas > 0 {
        gas_used += delta_gas as u64;
    }
}
```

**Simulation path (basefee == 0, gas_price == 0):**
- `native_per_gas = 0` → unlimited native budget
- `delta_gas = 0` → `gas_used` is never increased for native consumption
- Returned `gas_used` = EVM ergs consumed only

**Execution path (gas_price == 0):**
- `native_per_gas = FREE_L1_TX_NATIVE_PER_GAS = 10_000`
- `delta_gas = native_used / 10_000 − gas_used_from_evm` (can be positive)
- Returned `gas_used` includes native-resource overhead

The two paths produce different `gas_used` values for the same transaction when `basefee == 0`.

---

### Impact Explanation

A user or frontend that calls simulation to estimate the gas limit for an L1→L2 transaction when `basefee == 0` receives an underestimated `gas_used`. If the user sets `gas_limit` equal to the simulated value and submits the transaction to L1:

1. The L1 transaction is accepted and enters the priority queue.
2. On L2, actual execution uses `FREE_L1_TX_NATIVE_PER_GAS = 10_000`, applies `delta_gas`, and computes a higher `gas_used`.
3. If `gas_limit < actual_gas_used`, the transaction body reverts on L2.
4. Because L1 transactions cannot be invalidated, the deposited funds (minus the fee) are lost or partially refunded incorrectly.

Additionally, simulation may succeed where execution would fail outright due to native resource exhaustion: simulation gives unlimited native, while execution caps native at `10_000 × gas_limit`.

---

### Likelihood Explanation

- `basefee == 0` is a valid, explicitly tested configuration (see `test_gas_price_zero_fee_zero` with `eip1559_basefee: U256::ZERO`).
- L1 transactions with `gas_price == 0` are explicitly supported and tested (`test_l1_tx_zero_gas_price_is_free`).
- Any frontend or wallet that calls `simulate_block` to estimate gas for an L1 transaction during a period when `basefee == 0` will receive a wrong estimate.
- The attacker-controlled entry path is simply submitting an L1 transaction with `gas_price == 0` after observing `basefee == 0` on-chain; no privileged access is required.

---

### Recommendation

When `basefee == 0` in simulation mode, fall back to `FREE_L1_TX_NATIVE_PER_GAS` instead of computing `native_per_gas = 0`, matching the execution path:

```rust
// In prepare_and_check_resources, simulation branch:
if Config::SIMULATION {
    let basefee_native = u256_try_to_u64(
        &system.get_eip1559_basefee().div_ceil(native_price)
    ).unwrap_or(u64::MAX);
    // Match execution: never use 0 (unlimited) in simulation
    if basefee_native == 0 { FREE_L1_TX_NATIVE_PER_GAS } else { basefee_native }
}
```

This ensures simulation never underestimates gas relative to execution, consistent with the invariant documented in the codebase: *"simulation of a transaction must never underestimate gas/pubdata compared to actual execution."*

### Proof of Concept

**Divergence path:**

1. Deploy a ZKsync OS chain with `eip1559_basefee = 0`.
2. Construct an L1 transaction with `gas_price = 0`, `gas_limit = G`, and calldata that consumes significant native resources (e.g., many SSTORE operations).
3. Call `simulate_block` → receives `gas_used_sim = X` (no delta_gas applied).
4. Submit the L1 transaction to L1 with `gas_limit = X`.
5. On L2, `process_l1_transaction` uses `native_per_gas = FREE_L1_TX_NATIVE_PER_GAS = 10_000`, computes `delta_gas = native_used / 10_000 − X > 0`, and sets `gas_used_exec = X + delta_gas > X`.
6. Since `gas_limit = X < gas_used_exec`, the transaction body reverts; the user's deposit is consumed.
