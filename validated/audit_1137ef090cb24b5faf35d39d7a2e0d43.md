### Title
Unprivileged Caller Can Permanently Suppress Per-Block Reward Distribution via User-Controlled `disable_rewards` Parameter - (File: src/staking/staking.cairo)

---

### Summary

`update_rewards` in `src/staking/staking.cairo` accepts a caller-controlled `disable_rewards: bool` parameter and has **no access control**. Any unprivileged address can call it with `disable_rewards: true`, which advances `last_reward_block` to the current block while skipping all reward distribution. Because the contract enforces a strict one-call-per-block invariant (`REWARDS_ALREADY_UPDATED`), the legitimate sequencer is permanently locked out of distributing rewards for that block. Repeating this every block permanently freezes all staker and delegator yield.

---

### Finding Description

The spec explicitly states the access control for `update_rewards` is **"Only starkware sequencer"**: [1](#0-0) 

However, the implementation only calls `general_prerequisites()`, which checks for pause state and a non-zero caller — no role check: [2](#0-1) 

```cairo
fn general_prerequisites(ref self: ContractState) {
    self.assert_is_unpaused();
    assert_caller_is_not_zero();
}
``` [3](#0-2) 

The critical logic inside `update_rewards` is:

```cairo
// Update last block rewards.
self.last_reward_block.write(current_block_number);

if disable_rewards || self.is_pre_consensus() {
    return;
}
``` [4](#0-3) 

`last_reward_block` is written **before** the `disable_rewards` guard. Any caller who passes `disable_rewards: true` consumes the block's reward slot without distributing any rewards. The subsequent legitimate sequencer call for the same block hits:

```cairo
assert!(
    current_block_number > self.last_reward_block.read(),
    "{}",
    Error::REWARDS_ALREADY_UPDATED,
);
``` [5](#0-4) 

The interface definition confirms the parameter is fully user-supplied with no validation: [6](#0-5) 

---

### Impact Explanation

When `disable_rewards: true` is passed by an attacker on every block, no staker ever accumulates `unclaimed_rewards_own` and no pool ever receives rewards via `update_rewards_from_staking_contract`. This constitutes **permanent freezing of unclaimed yield** for all stakers and delegators — a High-severity impact under the allowed scope.

---

### Likelihood Explanation

The entry point is fully public (any non-zero address). No tokens, stake, or special role is required. The attacker only needs to front-run or race the sequencer once per block. On Starknet, where block production is deterministic and observable, this is trivially automatable. The cost is only gas.

---

### Recommendation

Add a sequencer-only access control guard at the top of `update_rewards`, consistent with the spec. For example, store the authorized sequencer address in storage and assert `get_caller_address() == self.sequencer_address.read()`, or reuse the existing roles component with an appropriate role (e.g., `only_app_governor` or a dedicated sequencer role).

---

### Proof of Concept

1. Consensus rewards are active (`is_pre_consensus()` returns `false`).
2. Staker S has staked and K epochs have elapsed (balance is effective).
3. Attacker A (any address) calls `update_rewards(staker_address: S, disable_rewards: true)` at block N.
   - `last_reward_block` is written to N.
   - The `if disable_rewards` branch fires; no rewards are computed or distributed.
4. The legitimate sequencer calls `update_rewards(staker_address: S, disable_rewards: false)` at block N.
   - `assert!(current_block_number > self.last_reward_block.read())` fails with `REWARDS_ALREADY_UPDATED`.
5. Staker S and all delegators receive zero rewards for block N.
6. Repeating step 3 every block permanently freezes all yield.

The flow tests confirm that calling `update_rewards` with `disable_rewards: true` when consensus is active produces zero rewards, and a second call in the same block panics with `REWARDS_ALREADY_UPDATED`: [7](#0-6)
