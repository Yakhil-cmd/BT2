### Title
Stale `target_shard_index` from pre-resharding layout used to index post-resharding `outcome_roots`, producing an unverifiable `outcome_root_proof` — (`chain/client/src/view_client_actor.rs`)

---

### Summary

`ViewClientActor::handle(GetExecutionOutcome)` computes `target_shard_index` once from the shard layout of the **outcome block's epoch**, then calls `get_next_block_hash_with_new_chunk`, which may cross a resharding boundary and return a **child** shard in a **new** layout. The code then collects `outcome_roots` from the post-resharding block and indexes into them with the **stale** `target_shard_index`. Because shard indices are not preserved across resharding in V2/V3 layouts, the returned `outcome_root_proof` is a `MerklePath` that does not verify against the block's `outcome_root`.

---

### Finding Description

The handler at `chain/client/src/view_client_actor.rs:1148–1182` executes this sequence:

**Step 1 — index computed from old layout:**
```rust
let epoch_id = *self.chain.get_block(&outcome_proof.block_hash)?.header().epoch_id();
let shard_layout = self.epoch_manager.get_shard_layout(&epoch_id)...;
let target_shard_id   = account_id_to_shard_id(..., &account_id, &epoch_id)...;
let target_shard_index = shard_layout.get_shard_index(target_shard_id)...;
``` [1](#0-0) 

`target_shard_index` is the parent shard's position in the **old** layout.

**Step 2 — `get_next_block_hash_with_new_chunk` may cross a resharding boundary:**

When the next epoch has a different shard layout, the function replaces `shard_ids` with the children of the original shard and returns `(h, child_shard_id)` — a child shard in the **new** layout. [2](#0-1) 

**Step 3 — `outcome_roots` collected from the post-resharding block, but indexed with the stale index:**
```rust
if let Some((h, target_shard_id)) = res {   // target_shard_id is now a child in new layout
    let outcome_roots = self.chain.get_block(&h)?.chunks().iter()
        .map(|header| *header.prev_outcome_root())
        .collect::<Vec<_>>();
    // target_shard_index is NEVER recomputed for the new layout
    outcome_root_proof: merklize(&outcome_roots).1[target_shard_index].clone(),
``` [3](#0-2) 

In V2/V3 shard layouts, `ShardId` and `ShardIndex` are decoupled. After a split, the child shards receive new IDs (`max+1`, `max+2`) and their indices in the new layout differ from the parent's index in the old layout. The code never calls `new_layout.get_shard_index(child_shard_id)` to recompute the index for the new layout.

The existing comment in `account_id_to_shard_id` itself warns that using `receiver_id` to determine the target shard is incorrect: [4](#0-3)

### Impact Explanation

The exact corrupted value is `GetExecutionOutcomeResponse.outcome_root_proof`: a `MerklePath` whose Merkle path does not verify against the block's `outcome_root` for the receipt's actual shard. Concretely:

```
verify_path(block_outcome_root, outcome_root_proof, chunk_outcome_root) == false
```

Any light client or bridge relying on `EXPERIMENTAL_light_client_proof` to verify a receipt executed in a shard that was subsequently split will receive an invalid proof and be unable to confirm the execution outcome. The proof cannot be used to assert a false statement (it fails verification), but it breaks the correctness invariant of the RPC and renders legitimate cross-chain proofs unusable for affected receipts.

---

### Likelihood Explanation

The trigger requires a resharding event between the outcome block and the next block that includes a new chunk for the affected shard. With dynamic resharding enabled (`ProtocolFeature::DynamicResharding`), this is a normal protocol operation. Any unprivileged user whose receipt lands in the shard being split can trigger the wrong proof by calling the public `EXPERIMENTAL_light_client_proof` RPC endpoint. No validator or operator privileges are required.

---

### Recommendation

After `get_next_block_hash_with_new_chunk` returns `(h, child_shard_id)`, recompute `target_shard_index` using the shard layout of block `h`'s epoch:

```rust
if let Some((h, target_shard_id)) = res {
    outcome_proof.block_hash = h;
    let h_epoch_id = *self.chain.get_block(&h)?.header().epoch_id();
    let h_shard_layout = self.epoch_manager.get_shard_layout(&h_epoch_id)...?;
    let target_shard_index = h_shard_layout.get_shard_index(target_shard_id)...?;
    let outcome_roots = ...;
    ...
    outcome_root_proof: merklize(&outcome_roots).1[target_shard_index].clone(),
}
```

This ensures the index used to select the Merkle path is always consistent with the layout of the block whose `outcome_roots` are being merkleized.

### Proof of Concept

1. Configure a test-loop environment with two epochs and a resharding event (e.g., using `derive_new_epoch_config_from_boundary`).
2. Submit a receipt whose `receiver_id` maps to the parent shard in the old layout.
3. Let the receipt execute in epoch E (old layout). Record `receipt_id` and `receiver_id`.
4. Advance past the epoch boundary so resharding takes effect in epoch E+1.
5. Call `view_client.handle(GetExecutionOutcome { id: TransactionOrReceiptId::Receipt { receipt_id, receiver_id } })`.
6. Compute `chunk_outcome_root = compute_root_from_path_and_item(&outcome_proof.proof, &outcome_proof.to_hashes())`.
7. Assert `verify_path(block_outcome_root, &execution_outcome.outcome_root_proof, &chunk_outcome_root)` — this assertion **fails**, demonstrating the corrupted proof.

The existing test `test_light_client_execution_outcome_proof` in `test-loop-tests/src/tests/light_client.rs` already performs this verification but does not exercise a resharding boundary; extending it with a resharding event between outcome and confirmation block would reproduce the failure.
