### Title
Silent Discard of `Result` from EIP-7002/7251 System Functions Corrupts `requests_hash` Block Header Field — (`basic_bootloader/src/bootloader/block_flow/ethereum/post_tx_op_sequencing.rs`)

---

### Summary

In `post_tx_op_sequencing.rs`, the return values of `eip7002_system_part` and `eip7251_system_part` are explicitly discarded with `let _ =`. Both functions return `Result<bool, SystemError>` and share a SHA-256 hasher (`requests_hasher`) that accumulates data for the EIP-7685 `requests_hash` block header field. If either function fails **after** it has already written its EIP-7685 type-prefix byte into the shared hasher, the error is silently swallowed and execution continues. The final `requests_hash` is then computed from a corrupted, partially-written hasher state, producing an invalid block header.

---

### Finding Description

In `post_tx_op_sequencing.rs` the post-block hook runs three EIP-specific processors against a single shared SHA-256 hasher:

```rust
// Environment may have no such contracts predeployed for tests or sequencing purposes
let _ = eip6110_events_parser(&system, &mut requests_hasher);
let _ = eip7002_system_part(&mut system, &mut requests_hasher);
let _ = eip7251_system_part(&mut system, &mut requests_hasher);

let requests_hash = Bytes32::from_array(requests_hasher.finalize().into());
``` [1](#0-0) 

Both `eip7002_system_part` and `eip7251_system_part` follow this internal sequence:

1. Verify the predeploy contract exists → return `Err` if not (hasher **not** yet touched — safe).
2. Read queue head/tail from storage → return `Err` on failure (hasher **not** yet touched — safe).
3. If queue is empty, call `update_excess_…_and_reset_count(system)?` and return `Ok(false)` (hasher **not** touched — safe).
4. **Write the EIP-7685 type-prefix byte into the shared hasher** (`requests_hasher.update([WITHDRAWAL_REQUEST_EIP_7685_TYPE])`).
5. Loop over queue entries, reading three storage slots per entry with `?` — return `Err` on any failure (**hasher already modified**).
6. Write updated queue-head pointer back to storage with `?` — return `Err` on failure (**hasher already modified**).
7. Call `update_excess_…_and_reset_count(system)?` — return `Err` on failure (**hasher already modified**). [2](#0-1) [3](#0-2) 

Any `Err` returned from steps 5–7 is silently discarded by `let _ =`. The hasher now contains the type-prefix byte (and possibly partial queue-entry data) for a request type that was never fully serialised. The subsequent call to `requests_hasher.finalize()` produces a hash over this corrupted state, which is then stored in the block header.

The identical pattern exists in `eip7251_system_part`: [4](#0-3) [5](#0-4) 

The corrupted `requests_hash` is then sealed into the block header: [6](#0-5) 

---

### Impact Explanation

The `requests_hash` field is a consensus-critical component of the Ethereum block header (EIP-7685). An incorrect value means the sealed block header does not match what the Ethereum consensus layer will independently compute from the same set of deposit/withdrawal/consolidation requests. This produces an **invalid block header**, which:

- Causes the block to be rejected by Ethereum consensus clients verifying the ZKsync OS state transition.
- Creates a divergence between the sequencer's committed state and the provable state, breaking the validity proof.
- In the worst case, if the prover also silently swallows the same error (it runs the same code path), both sequencer and prover agree on the wrong `requests_hash`, making the divergence undetectable internally but externally invalid — a **state-transition correctness bug**.

---

### Likelihood Explanation

The comment acknowledges the `let _ =` is intentional for environments where the predeploy contracts are absent (failure at step 1, before the hasher is touched — harmless). However, the same broad discard applies to all other failure modes. A failure at steps 5–7 is reachable when:

- The withdrawal/consolidation queue is non-empty (an unprivileged user can enqueue requests by interacting with the predeploy contracts).
- Any storage oracle read returns an error during proof generation (e.g., a missing Merkle witness for a queue-entry slot).

Because an unprivileged user can ensure the queue is non-empty, and oracle/storage errors are possible in the proving path, the preconditions for the corrupted-hasher scenario are reachable without privileged access.

---

### Recommendation

Replace the broad `let _ =` discard with targeted error handling that only ignores the "contract not deployed" case and propagates all other errors:

```rust
match eip7002_system_part(&mut system, &mut requests_hasher) {
    Ok(_) => {}
    Err(SystemError::LeafDefect(_)) => {
        // Contract not predeployed — expected in some environments
    }
    Err(e) => return Err(e.into()),
}
```

Apply the same pattern to `eip7251_system_part`. For `eip6110_events_parser` the risk is lower (it always returns `Ok` or panics), but the same discipline should be applied for consistency.

---

### Proof of Concept

1. Deploy the EIP-7002 withdrawal predeploy contract at `WITHDRAWAL_REQUEST_PREDEPLOY_ADDRESS` with `nonce == 1` and non-zero bytecode.
2. Submit at least one withdrawal request so that `queue_tail_index > queue_head_index`.
3. Arrange for a storage oracle read of one of the three queue-entry slots (slot_0/slot_1/slot_2 in the loop at lines 188–211) to return a `SystemError` (e.g., by providing a malformed Merkle witness in the proving path).
4. `eip7002_system_part` writes `WITHDRAWAL_REQUEST_EIP_7685_TYPE` (0x01) into `requests_hasher` at line 177, then hits the `?`-propagated error and returns `Err`.
5. `let _ = eip7002_system_part(...)` discards the error.
6. `eip7251_system_part` is called; if it succeeds it appends `CONSOLIDATION_REQUEST_EIP_7685_TYPE` (0x02) after the orphaned 0x01 byte.
7. `requests_hasher.finalize()` produces `SHA256(0x01 || 0x02 || <consolidation data>)` instead of the correct `SHA256(0x02 || <consolidation data>)`.
8. The sealed block header contains the wrong `requests_hash`, making the block invalid to any Ethereum consensus client. [1](#0-0) [2](#0-1)
