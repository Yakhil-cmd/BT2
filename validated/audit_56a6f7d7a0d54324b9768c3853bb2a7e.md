### Title
Gateway `skip_stateful_validations` ignores `max_nonce_for_validation_skip` config, hardcoding nonce=1 and rejecting valid transactions - (File: crates/apollo_gateway/src/stateful_transaction_validator.rs)

### Summary

`StatefulTransactionValidatorConfig` exposes a `max_nonce_for_validation_skip` field documented as "Maximum nonce for which the validation is skipped." This field is intended to control the deploy-account + invoke UX feature, allowing users to submit multiple invoke transactions before their `deploy_account` is processed. However, the gateway's `skip_stateful_validations` function hardcodes the nonce check to `Nonce(Felt::ONE)` and never reads the configured value. Invoke transactions with nonce > 1 are always subjected to the `__validate__` entry point, which fails when the account does not yet exist, causing valid transactions to be rejected by the gateway.

### Finding Description

`skip_stateful_validations` in `crates/apollo_gateway/src/stateful_transaction_validator.rs` hardcodes the nonce check:

```rust
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
``` [1](#0-0) 

The caller `run_pre_validation_checks` has access to `self.config` but passes no config value to `skip_stateful_validations`:

```rust
let skip_validate =
    skip_stateful_validations(executable_tx, account_nonce, mempool_client.clone()).await?;
``` [2](#0-1) 

`StatefulTransactionValidatorConfig` defines and serializes the field:

```rust
pub max_nonce_for_validation_skip: Nonce,
``` [3](#0-2) 

The production config schema exposes it as a public, operator-settable parameter:

```json
"gateway_config.static_config.stateful_tx_validator_config.max_nonce_for_validation_skip": {
    "description": "Maximum nonce for which the validation is skipped.",
    "privacy": "Public",
    "value": "0x1"
}
``` [4](#0-3) 

By contrast, the native-blockifier `PyValidator::should_run_stateful_validations` correctly reads `self.max_nonce_for_validation_skip`:

```rust
let nonce_small_enough_to_qualify_for_validation_skip =
    tx_nonce <= self.max_nonce_for_validation_skip;
let skip_validate = deploy_account_not_processed
    && is_post_deploy_nonce
    && nonce_small_enough_to_qualify_for_validation_skip;
``` [5](#0-4) 

The gateway's `skip_stateful_validations` is a free function that receives no config parameter, so the field stored in `StatefulTransactionValidatorConfig` is structurally unreachable from it. The existing test explicitly confirms nonce=2 is never skipped, regardless of config:

```rust
#[case::should_not_skip_validation_nonce_over_one(
    executable_invoke_tx(invoke_tx_args!(nonce: nonce!(2))),
    nonce!(0),
    true,
    true  // should_validate = true (never skipped)
)]
``` [6](#0-5) 

### Impact Explanation

When an operator sets `max_nonce_for_validation_skip > 1`, the gateway still only skips `__validate__` for invoke transactions with nonce exactly equal to 1. Invoke transactions with nonces 2 through `max_nonce_for_validation_skip` are subjected to `run_validate_entry_point` with `validate = true`:

```rust
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [7](#0-6) 

Because the account does not yet exist on-chain (deploy_account is still pending in the mempool), the `__validate__` entry point call fails and the transaction is rejected with `ValidateFailure`. These are valid transactions that should be admitted per the operator's configuration. This matches: **High — Mempool/gateway/RPC admission rejects valid transactions before sequencing.**

### Likelihood Explanation

The default config value is `Nonce(Felt::ONE)`, which coincidentally matches the hardcoded value, so the bug is silent in default deployments. The field is a public, documented, operator-settable parameter in the production config schema. Any operator who raises `max_nonce_for_validation_skip` to improve UX for batch deploy-account + invoke flows will silently get no effect, and users submitting invoke transactions with nonce > 1 alongside a deploy_account will have those transactions rejected.

### Recommendation

Pass `self.config.max_nonce_for_validation_skip` from `run_pre_validation_checks` into `skip_stateful_validations` and replace the hardcoded `Nonce(Felt::ONE)` equality check with a range check `tx.nonce() <= max_nonce_for_validation_skip`, mirroring the logic already implemented in `PyValidator::should_run_stateful_validations`.

### Proof of Concept

1. Operator sets `max_nonce_for_validation_skip = 5` in gateway config.
2. User submits `deploy_account` (nonce=0) and `invoke` (nonce=2) simultaneously.
3. Gateway processes `deploy_account`: account nonce=0, nonce=0 → passes all checks, enters mempool.
4. Gateway processes `invoke` (nonce=2):
   - `validate_nonce`: nonce=2 ∈ [0, 200] → passes.
   - `validate_by_mempool`: deploy_account is in mempool → passes.
   - `skip_stateful_validations`: `tx.nonce() == Nonce(Felt::ONE)` → **false** (nonce is 2, hardcoded check fails).
   - `run_validate_entry_point` called with `validate = true`.
   - `__validate__` entry point executed against on-chain state where account does not yet exist → **fails**.
   - Transaction rejected with `ValidateFailure`.
5. The invoke(nonce=2) is rejected despite `max_nonce_for_validation_skip = 5` being set. The configured behavior is unreachable because `skip_stateful_validations` never reads the config field.
