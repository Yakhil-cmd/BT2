### Title
`EVM.mustRun()` Does Not Revert on Failed EVM Execution, Enabling Silent Cross-VM Asset Loss - (File: `fvm/evm/stdlib/contract.cdc`)

---

### Summary

`EVM.mustRun()` in `fvm/evm/stdlib/contract.cdc` asserts only that the EVM transaction was executed (i.e., not `unknown`/`invalid`), but explicitly does **not** revert when the EVM transaction fails (`Status.failed`). This is the direct Flow analog of the unchecked ERC20 `transfer()` return value: any Cadence transaction that calls `mustRun` to perform an EVM token transfer and does not separately inspect the returned `Result.status` will silently proceed even when the underlying EVM operation reverts, enabling cross-VM asset loss.

---

### Finding Description

In `fvm/evm/stdlib/contract.cdc` at lines 838–851, `EVM.mustRun()` is defined as:

```cadence
/// mustRun runs the transaction using EVM.run
/// It will rollback if the tx execution status is unknown or invalid.
/// Note that this method does not rollback if transaction
/// is executed but an vm error is reported as the outcome
/// of the execution (status: failed).
access(all)
fun mustRun(tx: [UInt8], coinbase: EVMAddress): Result {
    let runResult = self.run(tx: tx, coinbase: coinbase)
    assert(
        runResult.status == Status.failed || runResult.status == Status.successful,
        message: "EVM.mustRun(): The provided transaction is not valid for execution"
    )
    return runResult
}
```

The `assert` condition is `Status.failed || Status.successful`. This means:

- The function **does not revert** when `runResult.status == Status.failed` (EVM execution reverted, ran out of gas, etc.).
- It only reverts on `Status.unknown` or `Status.invalid`.

The `Status` enum is defined at lines 341–365 of the same file:

```cadence
access(all) case failed    // executed but VM reported an error (e.g. revert, out of gas)
access(all) case successful // executed with no error
```

`Status.failed` means the EVM transaction was included in a block but its execution outcome was an error — exactly the case where an ERC20 `transfer()` would return `false`. The function name `mustRun` implies to callers that the transaction "must" succeed, but it only guarantees the transaction was "run" (included in a block). Any Cadence transaction that calls `EVM.mustRun()` to perform an ERC20 token transfer and does not separately check `result.status == EVM.Status.successful` will silently proceed even when the EVM transfer reverts. ### Impact Explanation

A Cadence transaction author (unprivileged) calls `EVM.mustRun()` with an RLP-encoded ERC20 `transfer()` transaction. The EVM transfer reverts (e.g., the ERC20 contract has insufficient balance, a custom revert, or runs out of gas). `mustRun` returns a `Result` with `status == Status.failed` **without panicking**. The Cadence transaction continues executing and commits all Cadence-side state changes as if the EVM transfer succeeded. This produces a cross-VM accounting discrepancy: the Cadence side records a completed transfer while no tokens moved on the EVM side. [3](#0-2) 

---

### Likelihood Explanation

`EVM.mustRun()` is a public, permissionless function callable by any Cadence transaction. EVM transactions fail for routine reasons (out of gas, ERC20 revert, nonce issues). The misleading name `mustRun` actively encourages callers to omit a result-status check. Any Cadence contract or transaction that wraps an ERC20 transfer via `mustRun` without asserting `Status.successful` is silently vulnerable. The entry path requires no special privileges: any user can submit a Cadence transaction. [4](#0-3) 

---

### Recommendation

Change the `assert` in `mustRun` to require `Status.successful`, making it a true "must succeed" guard analogous to `safeTransfer`:

```cadence
fun mustRun(tx: [UInt8], coinbase: EVMAddress): Result {
    let runResult = self.run(tx: tx, coinbase: coinbase)
    assert(
        runResult.status == Status.successful,
        message: "EVM.mustRun(): EVM transaction did not succeed"
    )
    return runResult
}
```

If the current semantics (allow `failed`, reject `invalid`/`unknown`) are intentional, rename the function (e.g., `runOrRevertOnInvalid`) and add prominent documentation warning callers to check `result.status` before proceeding with any asset-accounting logic. [1](#0-0)

### Proof of Concept

```cadence
import EVM from <EVMContractAddress>

transaction(erc20TransferTx: [UInt8], coinbaseBytes: [UInt8; 20]) {
    prepare(account: &Account) {
        let coinbase = EVM.EVMAddress(bytes: coinbaseBytes)

        // mustRun does NOT revert if the EVM transfer reverts (Status.failed).
        // Caller assumes success; Cadence state is committed with no EVM transfer.
        let result = EVM.mustRun(tx: erc20TransferTx, coinbase: coinbase)

        // result.status may be EVM.Status.failed here — no panic, no revert.
        // Any subsequent Cadence accounting (e.g., marking a withdrawal as complete)
        // proceeds as if the EVM transfer succeeded.
    }
}
```

The EVM transfer encoded in `erc20TransferTx` reverts (e.g., ERC20 `transfer()` returns `false` and reverts, or gas is exhausted). `mustRun` returns `Result { status: failed, ... }` without aborting the Cadence transaction. The outer Cadence transaction commits, recording a completed transfer that never occurred on the EVM side. [5](#0-4) [6](#0-5)
