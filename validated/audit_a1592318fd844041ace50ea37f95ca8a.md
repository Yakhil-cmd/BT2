### Title
Attacker Bypasses `__validate__` for Invoke Transactions via `skip_stateful_validations` UX Feature, Causing Unauthorized Fee Charge on Pre-Funded Victim Accounts — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function is designed to improve UX by allowing an Invoke with nonce=1 to skip blockifier `__validate__` when the account's deploy_account is pending in the mempool. However, the check `account_tx_in_pool_or_recent_block` does not verify that the pending transaction is specifically a deploy_account — it returns `true` for any account with any transaction in the pool. An unprivileged attacker who observes a victim's deploy_account in the mempool can submit an Invoke with the victim's address, nonce=1, and an invalid signature. The gateway admits it without running `__validate__`. When the sequencer later executes the block, `__validate__` runs and fails, and the victim's pre-funded account is charged the validation fee.

---

### Finding Description

**Step 1 — `skip_stateful_validations` triggers on attacker-controlled input** [1](#0-0) 

The function returns `true` when:
- tx is `ExecutableTransaction::Invoke`
- `tx.nonce() == Nonce(Felt::ONE)`
- `account_nonce == Nonce(Felt::ZERO)`
- `account_tx_in_pool_or_recent_block(sender_address)` returns `true`

The comment says "it means that either it has a deploy_account transaction or transactions with future nonces that passed validations." But the check is not type-filtered — it returns `true` for any account with any transaction in the pool. [2](#0-1) 

`account_tx_in_pool_or_recent_block` checks `self.state.contains_account(account_address) || self.tx_pool.contains_account(account_address)` — no deploy_account type filter.

**Step 2 — `run_validate_entry_point` sets `validate=false`** [3](#0-2) 

When `skip_validate=true`, `execution_flags.validate` is set to `false`.

**Step 3 — `perform_validations` returns `Ok(())` without running `__validate__`** [4](#0-3) 

For Invoke transactions, after `perform_pre_validation_stage`, the code checks `if !tx.execution_flags.validate { return Ok(()); }` — the signature is never verified at the gateway.

**Step 4 — `perform_pre_validation_stage` passes for a pre-funded account** [5](#0-4) 

- `handle_nonce` with `strict_nonce_check=false`: checks `account_nonce(0) <= tx_nonce(1)` → passes, increments nonce.
- `verify_can_pay_committed_bounds`: checks balance >= max_possible_fee. **If the victim pre-funded their account address (the standard Starknet deployment pattern), this passes.**

**Step 5 — Transaction admitted to mempool, victim charged at execution**

The attacker's Invoke is admitted. When the sequencer executes the block: deploy_account runs (nonce 0→1), then the attacker's Invoke runs with `validate=true` (execution flags are set fresh by the sequencer, not carried from gateway). `__validate__` runs, fails on the invalid signature, and the victim's account is charged the validation fee.

---

### Impact Explanation

- **Admission impact (High):** An Invoke transaction with an invalid/attacker-controlled signature is admitted to the mempool without cryptographic signature verification, violating the gateway's security invariant.
- **Economic impact (Critical):** The victim's pre-funded account is charged a validation fee for a transaction they never authorized. The attacker pays nothing (their transaction is never executed from their own account).
- **Nonce-slot griefing:** The attacker's transaction occupies the victim's nonce=1 slot. The victim's legitimate nonce=1 transaction must either be rejected or pay a fee-escalation premium to replace it.

---

### Likelihood Explanation

- Pre-funding the account address before submitting deploy_account is the **standard, documented Starknet account deployment flow**. The vast majority of new account deployments involve a pre-funded address.
- The victim's deploy_account is publicly observable in the mempool.
- The attacker needs only the victim's address (derivable from the deploy_account transaction) and the ability to submit a transaction — both are unprivileged.
- The attack window is the time between the victim's deploy_account entering the mempool and being sequenced.

---

### Recommendation

In `skip_stateful_validations`, replace the generic `account_tx_in_pool_or_recent_block` check with a type-specific check that verifies a **deploy_account** transaction (not just any transaction) is pending for the sender address. The mempool should expose a dedicated `deploy_account_in_pool(address)` query, or the gateway should pass the deploy_account transaction hash (as the legacy `native_blockifier` path does via `deploy_account_tx_hash: Option<TransactionHash>`) and verify it is present. [6](#0-5) 

The legacy `PyValidator::should_run_stateful_validations` correctly requires an explicit `deploy_account_tx_hash` parameter supplied by the caller — the new gateway path lost this type-safety by replacing it with the untyped `account_tx_in_pool_or_recent_block` check.

---

### Proof of Concept

```
Preconditions:
  - victim_address = 0xABCD (pre-funded with STRK, nonce=0 in state)
  - victim submits DeployAccount(sender=victim_address, nonce=0, valid_sig)
    → mempool admits it; account_tx_in_pool_or_recent_block(victim_address) = true

Attack:
  - attacker submits Invoke(sender=victim_address, nonce=1, sig=[0xDEAD,0xBEEF], resource_bounds=reasonable)

Gateway stateful path:
  1. get_nonce_from_state(victim_address) → Nonce(0)
  2. validate_nonce: 0 <= 1 <= 0+gap → Ok
  3. validate_by_mempool: nonce=1 is future nonce → Ok
  4. skip_stateful_validations:
       tx is Invoke ✓, nonce==1 ✓, account_nonce==0 ✓
       account_tx_in_pool_or_recent_block(victim_address) → true ✓
       returns true
  5. run_validate_entry_point(skip_validate=true):
       execution_flags.validate = false
       perform_pre_validation_stage:
         handle_nonce: 0 <= 1 → Ok (nonce incremented in temp state)
         verify_can_pay_committed_bounds: balance > 0 (pre-funded) → Ok
       if !validate { return Ok(()) }  ← __validate__ never called
  6. Transaction admitted to mempool

Sequencer execution:
  Block N:
    tx1: DeployAccount(victim, nonce=0) → executes, account deployed, nonce=1
    tx2: Invoke(victim, nonce=1, sig=garbage) → __validate__ runs → FAILS
         → victim charged validation fee
         → victim's nonce=1 slot consumed by attacker's tx
```

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)

    pub fn perform_pre_validation_stage<S: State + StateReader>(
        &self,
        state: &mut S,
        tx_context: &TransactionContext,
    ) -> TransactionPreValidationResult<()> {
        let tx_info = &tx_context.tx_info;
        Self::handle_nonce(state, tx_info, self.execution_flags.strict_nonce_check)?;

        if self.execution_flags.charge_fee {
            self.check_fee_bounds(tx_context)?;

            verify_can_pay_committed_bounds(state, tx_context).map_err(Box::new)?;
        }

        self.validate_proof_facts(&tx_context.block_context, state)?;

        Ok(())
    }

    pub fn should_run_stateful_validations(
        &mut self,
        account_tx: &AccountTransaction,
        deploy_account_tx_hash: Option<TransactionHash>,
    ) -> StatefulValidatorResult<bool> {
        if account_tx.tx_type() != TransactionType::InvokeFunction {
            return Ok(true);
        }
        let tx_info = account_tx.create_tx_info();
        let nonce = self.stateful_validator.get_nonce(tx_info.sender_address())?;

        let deploy_account_not_processed =
            deploy_account_tx_hash.is_some() && nonce == Nonce(Felt::ZERO);
        let tx_nonce = tx_info.nonce();
        let is_post_deploy_nonce = Nonce(Felt::ONE) <= tx_nonce;
        let nonce_small_enough_to_qualify_for_validation_skip =
            tx_nonce <= self.max_nonce_for_validation_skip;

        let skip_validate = deploy_account_not_processed
            && is_post_deploy_nonce
            && nonce_small_enough_to_qualify_for_validation_skip;

        Ok(!skip_validate)
    }
