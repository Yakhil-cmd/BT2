### Title
Zero-value `priceProviderTimelock` allows pool admin to rotate oracle with no LP reaction window — (`File: metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`MetricOmmPoolFactory.createPool` accepts `priceProviderTimelock = 0` without any minimum-value guard. When the timelock is zero, `proposePoolPriceProvider` and `executePoolPriceProviderUpdate` can be called in the same block (or same transaction via a contract), giving LPs zero time to exit before a manipulated oracle is live.

---

### Finding Description

The M01 audit fix introduced a two-step oracle-rotation flow with a per-pool `priceProviderTimelock` to protect LPs from sudden oracle changes. However, `_validatePoolParameters` enforces no minimum on this value:

```solidity
// metric-core/contracts/MetricOmmPoolFactory.sol  _validatePoolParameters (lines 548-563)
// No check on params.priceProviderTimelock — any value other than type(uint256).max is accepted.
``` [1](#0-0) 

When `priceProviderTimelock[pool] == 0`, `proposePoolPriceProvider` computes:

```solidity
uint256 executeAfter = block.timestamp + timelock;  // = block.timestamp + 0
``` [2](#0-1) 

And `executePoolPriceProviderUpdate` checks:

```solidity
if (block.timestamp < execAfter) revert ...;  // block.timestamp < block.timestamp → false → passes
``` [3](#0-2) 

Both calls can be batched in a single transaction through a contract, atomically replacing the oracle with no observable pending state for LPs.

The timelock is stored once at creation and is never updatable afterward:

```solidity
priceProviderTimelock[pool] = params.priceProviderTimelock;
``` [4](#0-3) 

---

### Impact Explanation

The oracle (`priceProvider`) is the sole source of bid/ask prices for every swap. If the pool admin atomically rotates to a manipulated oracle (one that returns an extreme or inverted price), the very next swap executes at that bad price. LPs bear the loss because the pool's bin balances are drained at the wrong rate. This is a direct bad-price execution impact on LP principal.

The M01 fix explicitly states the timelock exists so LPs can observe the pending change and exit before it takes effect. A zero-second timelock makes that protection illusory. [5](#0-4) 

---

### Likelihood Explanation

**Low.** Requires: (1) a pool deployed with `priceProviderTimelock = 0` (not the default in tests, which use `type(uint256).max`), and (2) a pool admin who is malicious or whose key is compromised. LPs who inspect `PoolCreated` event data can detect the zero timelock before depositing. However, the factory emits no warning and enforces no minimum, so unsophisticated LPs may not notice. [6](#0-5) 

---

### Recommendation

Add a minimum timelock constant (e.g., `MIN_PRICE_PROVIDER_TIMELOCK = 1 days`) and enforce it in `_validatePoolParameters`:

```solidity
if (params.priceProviderTimelock != type(uint256).max
    && params.priceProviderTimelock < MIN_PRICE_PROVIDER_TIMELOCK) {
    revert PriceProviderTimelockTooShort();
}
```

This preserves the immutable-mode path (`type(uint256).max`) while guaranteeing that any mutable-oracle pool gives LPs at least one day to observe and react to a pending rotation.

---

### Proof of Concept

```solidity
// Attacker contract — executes atomically in one transaction
contract AtomicOracleSwap {
    function attack(
        MetricOmmPoolFactory factory,
        address pool,
        address maliciousOracle
    ) external {
        // Step 1: propose (executeAfter = block.timestamp + 0 = block.timestamp)
        factory.proposePoolPriceProvider(pool, maliciousOracle);

        // Step 2: execute immediately — block.timestamp < block.timestamp is false
        factory.executePoolPriceProviderUpdate(pool);

        // Step 3: swap against the now-live manipulated oracle, draining LP bins
        // IMetricOmmPool(pool).swap(...);
    }
}
```

The pool admin calls `attack()` in a single transaction. No LP has any opportunity to front-run the rotation and exit, because the oracle is already live before the transaction is mined. [7](#0-6)

  function proposePoolPriceProvider(address pool, address newPriceProvider)
    external
    override
    nonReentrant
    onlyPoolAdmin(pool)
  {
    PoolImmutables memory p = IMetricOmmPool(pool).getImmutables();
    uint256 timelock = priceProviderTimelock[pool];
    if (p.immutablePriceProvider != address(0)) revert PriceProviderImmutable();
    _validatePriceProvider(p.token0, p.token1, newPriceProvider);

    address mutableProvider = PoolStateLibrary._slot3(pool);
    address current = mutableProvider != address(0) ? mutableProvider : p.immutablePriceProvider;
    uint256 executeAfter = block.timestamp + timelock;
    pendingPriceProvider[pool] = newPriceProvider;
    pendingPriceProviderExecuteAfter[pool] = executeAfter;
    emit PoolPriceProviderChangeProposed(pool, current, newPriceProvider, executeAfter);
  }

  /// @inheritdoc IMetricOmmPoolFactoryPoolAdmin
  function executePoolPriceProviderUpdate(address pool) external override nonReentrant onlyPoolAdmin(pool) {
    address pending = pendingPriceProvider[pool];
    if (pending == address(0)) revert NoPriceProviderChangeProposed();
    uint256 execAfter = pendingPriceProviderExecuteAfter[pool];
    // forge-lint: disable-next-line(block-timestamp) -- timelock enforcement legitimately relies on `block.timestamp`.
    if (block.timestamp < execAfter) revert PriceProviderTimelockNotElapsed(execAfter, block.timestamp);
    PoolImmutables memory p = IMetricOmmPool(pool).getImmutables();
    if (p.immutablePriceProvider != address(0)) revert PriceProviderImmutable();
    _validatePriceProvider(p.token0, p.token1, pending);
    IMetricOmmPoolFactoryActions(pool).setPriceProvider(pending);
    delete pendingPriceProvider[pool];
    delete pendingPriceProviderExecuteAfter[pool];
    emit PoolPriceProviderUpdated(pool, pending);
  }

## M01 - Admin can arbitrarily change the pool priceProvider

### Finding summary

`setPriceProvider()` allowed admin to replace oracle at runtime, and `priceProvider` was returned by `getImmutables()`.

### Resolution

Fixed.

Changes:

- Added constructor config `priceProviderTimelock`.
- Added admin flow:
  - `proposePriceProvider(address newPriceProvider)`
  - `executePriceProviderUpdate()`
- Added events:
  - `PriceProviderChangeProposed(currentPriceProvider, newPriceProvider, executeAfter)`
  - `PriceProviderUpdated(newPriceProvider)`

Behavior now:

- `priceProviderTimelock == type(uint256).max` => immutable mode (admin updates disabled).
- `priceProviderTimelock < type(uint256).max` => mutable mode with delayed execution.
- Price provider used by swaps/reports resolves through timelock mode; active provider is read via `MetricOmmPoolStateView.priceProvider(pool)`.

### Verification

- Added tests:
  - `test_priceProviderTimelock_immutableMode_revertsPropose`
  - `test_priceProviderTimelock_proposeAndExecute`
