### Title
Invoke Transactions with Invalid Signatures Bypass Gateway `__validate__` via Overly Broad `skip_stateful_validations` Check — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function is designed to skip the `__validate__` entry-point call (signature verification) for an invoke with nonce 1 when the account does not yet exist on-chain, as a UX convenience for the simultaneous deploy_account + invoke flow. The guard it uses — `account_tx_in_pool_or_recent_block` — returns `true` for **any** transaction in the mempool for that address, not exclusively for a pending deploy_account. An account that already exists on-chain with nonce 0 and has any transaction (e.g., a valid invoke with nonce 0) in the mempool satisfies the guard. An unprivileged attacker can therefore submit an invoke with nonce 1 and a completely fabricated signature for such an account; the gateway skips `__validate__`, and the transaction is admitted into the mempool without signature verification.

---

### Finding Description

`skip_stateful_validations` fires when three conditions are simultaneously true:

1. The incoming transaction is an `Invoke`.
2. The transaction nonce is exactly `1`.
3. The on-chain account nonce is `0`.

When all three hold, the function calls `account_tx_in_pool_or_recent_block` and, if it returns `true`, returns `true` (skip validation). [1](#0-0) 

The comment inside the function states:

> "It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."

The second branch of that disjunction is the flaw. An account that **already exists on-chain** with nonce 0 (deployed but never transacted) can have a valid invoke with nonce 0 in the mempool — that invoke passed `__validate__` because the account exists. `account_tx_in_pool_or_recent_block` checks: [2](#0-1) 

which returns `true` as soon as `tx_pool.contains_account(address)` is true — regardless of transaction type. There is no deploy_account-specific check anywhere in this path.

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` sets `execution_flags.validate = false`: [3](#0-2) 

Inside `StatefulValidator::perform_validations`, an invoke with `validate = false` returns `Ok(())` immediately after `perform_pre_validation_stage`, never reaching the `__validate__` call: [4](#0-3) 

`perform_pre_validation_stage` runs `handle_nonce`, `check_fee_bounds`, `verify_can_pay_committed_bounds`, and `validate_proof_facts` — none of which verify the transaction signature: [5](#0-4) 

The stateless validator also does not verify signature validity — it only checks signature **size**: [6](#0-5) 

Therefore, a transaction with a completely fabricated signature passes every gateway check and is admitted into the mempool.

---

### Impact Explanation

**High — Mempool/gateway admission accepts invalid transactions before sequencing.**

Any unprivileged actor who knows the address of an account that (a) exists on-chain with nonce 0 and (b) currently has any transaction in the mempool can inject an invoke with nonce 1 carrying an arbitrary, invalid signature. The transaction enters the mempool without signature verification. At execution time `__validate__` will be called and will fail, but by then the invalid transaction has already consumed mempool capacity, can displace a legitimate nonce-1 transaction via fee-escalation replacement, and pollutes the pending state visible to RPC callers.

---

### Likelihood Explanation

The precondition — an account with on-chain nonce 0 that has a pending transaction in the mempool — is the normal state for every newly deployed account sending its very first transaction. The window is open for the entire duration that the nonce-0 transaction sits in the mempool (potentially many blocks). No privileged access is required; the attacker only needs to know the target address and submit a well-formed but signature-invalid invoke.

---

### Recommendation

Replace the generic `account_tx_in_pool_or_recent_block` check with one that specifically verifies a **deploy_account** transaction is pending for the address. The mempool should expose a dedicated query such as `has_pending_deploy_account(address) -> bool` that inspects the transaction type stored in the pool, rather than returning `true` for any transaction type. Alternatively, the gateway can check the on-chain class hash: if `get_class_hash_at(sender_address) != ClassHash::default()`, the account already exists and `__validate__` must never be skipped.

---

### Proof of Concept

1. Account `A` is deployed on-chain (class hash ≠ 0) with nonce 0 — it has never sent a transaction.
2. The legitimate owner of `A` submits `invoke_A_nonce0` (valid signature, nonce 0). It passes `__validate__` and enters the mempool's `tx_pool`.
3. An attacker constructs `invoke_A_nonce1` for address `A`, nonce 1, with a completely fabricated signature (e.g., `[0x1, 0x2]`).
4. The attacker submits `invoke_A_nonce1` to the gateway.
5. Gateway stateless check: signature size ≤ limit → passes. [6](#0-5) 
6. `extract_state_nonce_and_run_validations`: on-chain nonce of `A` = 0. `validate_nonce`: nonce 1 is within `[0, max_allowed_nonce_gap]` → passes. [7](#0-6) 
7. `validate_by_mempool`: mempool nonce check passes (nonce 1 ≥ account nonce 0).
8. `skip_stateful_validations`: tx nonce == 1, account nonce == 0, `account_tx_in_pool_or_recent_block(A)` → `tx_pool.contains_account(A)` → `true` (because `invoke_A_nonce0` is in the pool) → returns `true`. [8](#0-7) 
9. `run_validate_entry_point` sets `validate = false`; `__validate__` is never called. The fabricated-signature transaction is accepted and forwarded to the mempool. [3](#0-2) 
10. `invoke_A_nonce1` now occupies a slot in the mempool. If the attacker sets a higher tip than the legitimate nonce-1 transaction, it displaces it via fee escalation. At execution time `__validate__` is called, the invalid signature is rejected, and the transaction fails — but the legitimate nonce-1 transaction has already been evicted.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}

    pub fn perform_pre_validation_stage<S: State + StateReader>(
        &self,
        state: &mut S,
        tx_context: &TransactionContext,
    ) -> TransactionPreValidationResult<()> {
        let tx_info = &tx_context.tx_info;
        Self::handle_nonce(state, tx_info, self.execution_flags.strict_nonce_check)?;

        if self.execution_flags.charge_fee {
            self.check_fee_bounds(tx_context)?;

            verify_can_pay_committed_bounds(state, tx_context).map_err(Box::new)?;
        }

        self.validate_proof_facts(&tx_context.block_context, state)?;

        Ok(())
    }
