### Title
Unauthenticated Invoke Admission via `skip_stateful_validations` Race — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The gateway's UX shortcut for the deploy-account + invoke flow skips `__validate__` (signature verification) for any invoke with `nonce == 1` sent to an account whose on-chain nonce is still `0`, provided `account_tx_in_pool_or_recent_block` returns `true`. Because that check is not restricted to deploy-account transactions and is not tied to the submitter's identity, any unprivileged observer can front-run the legitimate user and inject a fake invoke — with an arbitrary or empty signature — for a victim account the moment that account's deploy-account lands in the mempool.

### Finding Description

`skip_stateful_validations` in `crates/apollo_gateway/src/stateful_transaction_validator.rs` returns `true` (skip the `__validate__` entry-point call) when all three conditions hold:

1. The incoming transaction is an `Invoke`.
2. The transaction nonce equals `Nonce(Felt::ONE)`.
3. The on-chain account nonce equals `Nonce(Felt::ZERO)`.
4. `mempool_client.account_tx_in_pool_or_recent_block(sender_address)` returns `true`. [1](#0-0) 

When `skip_validate` is `true`, `run_validate_entry_point` sets `execution_flags.validate = false`, so `StatefulValidator::perform_validations` returns immediately after `perform_pre_validation_stage` without ever calling the account's `__validate__` entry point: [2](#0-1) [3](#0-2) 

The `account_tx_in_pool_or_recent_block` predicate checks only whether the account has *any* transaction in the pool or was seen in a recent block — it does not verify that the pooled transaction is a deploy-account, nor that the submitter of the invoke is the same party who submitted the deploy-account: [4](#0-3) 

**Attack path:**

1. Victim broadcasts a valid `DeployAccountV3` for address `A`. It passes full gateway validation (constructor + `__validate_deploy__` run) and enters the mempool. `account_tx_in_pool_or_recent_block(A)` now returns `true`.
2. Attacker observes the mempool, constructs an `InvokeV3` for address `A` with `nonce = 1` and an arbitrary/empty signature, and submits it to the gateway.
3. Gateway stateless checks pass (valid address, resource bounds, etc.).
4. Gateway stateful path: `account_nonce = 0`, `tx.nonce() = 1`, `account_tx_in_pool_or_recent_block(A) = true` → `skip_validate = true` → `__validate__` is **never called** → the fake invoke is admitted to the mempool.
5. If the attacker submits before the victim's own invoke, `validate_by_mempool` sees no duplicate and the fake invoke occupies the `(A, nonce=1)` slot. [5](#0-4) 

### Impact Explanation

**Mempool admission of an invalid (unauthenticated) transaction — High.**

The fake invoke occupies the `(A, nonce=1)` slot in the mempool. The victim's legitimate invoke with `nonce=1` is then rejected by `validate_by_mempool` with `DuplicateNonce`, blocking the victim's first post-deployment call.

When the fake invoke is later pulled from the mempool and executed by the batcher, `AccountTransaction::new_for_sequencing` sets `validate = true`: [6](#0-5) 

`perform_pre_validation_stage` increments the nonce on the transactional state before `__validate__` is called: [7](#0-6) 

If the account has been pre-funded (a common pattern), `verify_can_pay_committed_bounds` passes, `__validate__` then fails (invalid signature), the transaction is **reverted** (nonce stays incremented, fee is charged), and the victim's nonce is now `2`. The victim's legitimate invoke (nonce=1) is permanently stale.

### Likelihood Explanation

The attack requires only:
- Monitoring the public mempool for deploy-account transactions (trivial).
- Submitting a single fake invoke before the victim submits their own invoke (a narrow but realistic race window, especially since the deploy-account + invoke UX pattern is explicitly documented and tested).

No privileged access, special keys, or prior on-chain state is required.

### Recommendation

Replace the generic `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a **deploy-account** transaction for the same address is pending. For example, expose a `has_pending_deploy_account(address)` query on the mempool, or store the transaction type alongside the account record so `skip_stateful_validations` can confirm the pooled transaction is actually a `DeployAccount`. [8](#0-7) 

### Proof of Concept

```
1. Victim calls gateway: DeployAccountV3 { sender_address: A, nonce: 0, signature: <valid> }
   → passes __validate_deploy__, enters mempool.
   → account_tx_in_pool_or_recent_block(A) == true

2. Attacker calls gateway: InvokeV3 { sender_address: A, nonce: 1, signature: [] }
   → stateless checks pass (valid address, non-zero resource bounds)
   → stateful: account_nonce=0, tx.nonce=1, account_tx_in_pool_or_recent_block(A)=true
   → skip_stateful_validations returns true
   → __validate__ is NOT called
   → fake invoke admitted to mempool at slot (A, nonce=1)

3. Victim calls gateway: InvokeV3 { sender_address: A, nonce: 1, signature: <valid> }
   → validate_by_mempool → DuplicateNonce error → REJECTED

4. Batcher executes block:
   - DeployAccount(A, nonce=0) executes → A deployed, nonce becomes 1
   - FakeInvoke(A, nonce=1) executes:
       perform_pre_validation_stage: nonce 1→2 incremented, fee check passes (if funded)
       run_or_revert → __validate__ called → FAILS (invalid signature)
       → transaction REVERTED, nonce stays at 2, fee charged

5. Victim's legitimate invoke (nonce=1) is now permanently invalid (account nonce is 2).
```

/// Check if validation of an invoke transaction should be skipped due to deploy_account not being
/// processed yet. This feature is used to improve UX for users sending deploy_account + invoke at
/// once.
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)

    pub fn perform_pre_validation_stage<S: State + StateReader>(
        &self,
        state: &mut S,
        tx_context: &TransactionContext,
    ) -> TransactionPreValidationResult<()> {
        let tx_info = &tx_context.tx_info;
        Self::handle_nonce(state, tx_info, self.execution_flags.strict_nonce_check)?;

        if self.execution_flags.charge_fee {
            self.check_fee_bounds(tx_context)?;

            verify_can_pay_committed_bounds(state, tx_context).map_err(Box::new)?;
        }

        self.validate_proof_facts(&tx_context.block_context, state)?;

        Ok(())
    }
