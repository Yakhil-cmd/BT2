### Title
Fee-on-Transfer NEP-141 Token Causes ERC-20 Mirror Over-Minting and Insolvency — (File: `engine/src/engine.rs`)

---

### Summary

`Engine::receive_erc20_tokens` unconditionally trusts the `amount` field supplied by the calling NEP-141 contract in `ft_on_transfer` and mints exactly that many ERC-20 mirror tokens. A NEP-141 token that silently deducts a transfer fee (i.e., delivers `amount − fee` to Aurora's custody while reporting `amount` in the callback) causes Aurora to mint more ERC-20 tokens than it holds NEP-141 backing for, producing a permanently insolvent ERC-20 mirror.

---

### Finding Description

When a NEP-141 token is bridged into Aurora via `ft_transfer_call`, the NEAR runtime invokes Aurora's `ft_on_transfer` entrypoint. The handler dispatches to `Engine::receive_erc20_tokens`:

```rust
// engine/src/engine.rs  line 803
let amount = args.amount.as_u128();
// ...
setup_receive_erc20_tokens_input(&recipient, amount)  // mints `amount` ERC-20 tokens
``` [1](#0-0) 

`args.amount` is the value the NEP-141 contract chose to pass in the callback — Aurora never independently verifies how many NEP-141 tokens it actually received. The ERC-20 mint is performed unconditionally for the caller-supplied `amount`.

The `ft_on_transfer` dispatcher in `contract_methods/connector.rs` also does not perform any balance reconciliation before or after the call:

```rust
// engine/src/contract_methods/connector.rs  line 80-90
let args: FtOnTransferArgs = read_json_args(&io)?;
let result = if predecessor_account_id == get_connector_account_id(&io)? {
    engine.receive_base_tokens(&args)
} else {
    engine.receive_erc20_tokens(&predecessor_account_id, &args, ...)
};
``` [2](#0-1) 

A fee-on-transfer NEP-141 token can implement `ft_transfer_call` to transfer `amount − fee` to Aurora while passing `amount` in the `ft_on_transfer` callback. Because NEAR cross-contract calls are asynchronous, Aurora cannot synchronously query its own NEP-141 balance to detect the discrepancy.

The ERC-20 mirror contract (`EvmErc20`) mints tokens via an admin-only `mint` call driven by `setup_receive_erc20_tokens_input`, so the over-minted supply is permanently recorded on-chain:

```solidity
// etc/eth-contracts/contracts/EvmErc20.sol  line 49-51
function mint(address account, uint256 amount) public onlyAdmin {
    _mint(account, amount);
}
``` [3](#0-2) 

When ERC-20 holders later call `withdrawToNear` (which burns ERC-20 tokens and calls `ft_transfer` on the NEP-141 contract), Aurora's NEP-141 balance is insufficient to cover all outstanding ERC-20 tokens. The last withdrawers receive nothing or the `ft_transfer` call fails, permanently freezing their funds.

---

### Impact Explanation

**Insolvency / Permanent fund freeze (Critical/High).**

The ERC-20 mirror's total supply exceeds Aurora's actual NEP-141 custody. Every unit of fee silently extracted by the NEP-141 token represents an ERC-20 token that can never be redeemed. Users who hold those ERC-20 tokens lose their funds permanently. The deficit compounds with every bridging transaction.

---

### Likelihood Explanation

`deploy_erc20_token` is a permissionless entrypoint — any NEAR account can register a NEP-141 token and create an ERC-20 mirror on Aurora. An attacker deploys a fee-on-transfer NEP-141 token, registers it, bridges tokens to Aurora (receiving over-minted ERC-20 tokens), sells those ERC-20 tokens to other users, and exits. The victims discover the shortfall only when they attempt to withdraw. No privileged access is required.

---

### Recommendation

1. **Measure actual received amount**: After `ft_on_transfer` is called, schedule a follow-up cross-contract call to query Aurora's NEP-141 balance and use the balance delta (not `args.amount`) as the mint amount.
2. **Alternatively, return the full amount on mismatch**: If the balance delta is less than `args.amount`, return `args.amount` (refund everything) rather than minting a partial or inflated amount.
3. **Document the trust assumption**: At minimum, add explicit documentation that only NEP-141 tokens whose `ft_transfer_call` delivers exactly `amount` tokens to the receiver are safe to bridge, and consider a registry or allowlist enforced at `deploy_erc20_token` time.

---

### Proof of Concept

1. Deploy a malicious NEP-141 token `FeeToken` whose `ft_transfer_call` transfers `amount * 99 / 100` to the receiver but passes `amount` in the `ft_on_transfer` callback.
2. Call `deploy_erc20_token` on Aurora with `FeeToken`'s account ID — this is permissionless.
3. Call `FeeToken::ft_transfer_call(receiver_id: aurora.near, amount: 1_000_000, msg: <evm_address>)`.
4. Aurora's `ft_on_transfer` receives `args.amount = 1_000_000` and mints `1_000_000` ERC-20 tokens, but Aurora's NEP-141 balance only increased by `990_000`.
5. The attacker holds `1_000_000` ERC-20 tokens backed by only `990_000` NEP-141 tokens.
6. The attacker calls `withdrawToNear` for `1_000_000` — the `ft_transfer` call on `FeeToken` fails because Aurora only holds `990_000`, permanently freezing the `10_000`-token shortfall for any subsequent user.

The root cause is at `engine/src/engine.rs` line 803 where `args.amount` is used as the mint quantity without any verification against the actual NEP-141 balance change. [4](#0-3)

    pub fn receive_erc20_tokens<P: PromiseHandler>(
        &mut self,
        token: &AccountId,
        args: &FtOnTransferArgs,
        current_account_id: &AccountId,
        handler: &mut P,
    ) -> Result<Option<SubmitResult>, ContractError> {
        let amount = args.amount.as_u128();
        // Parse message to determine recipient
        let mut recipient = {
            // The message should contain the recipient EOA address.
            let message = args.msg.strip_prefix("0x").unwrap_or(&args.msg);
            // Recipient - 40 characters (Address in hex without '0x' prefix)
            if message.len() < 40 {
                return Err(ParseOnTransferMessageError::WrongMessageFormat.into());
            }
            let mut address_bytes = [0; 20];
            hex::decode_to_slice(&message[..40], &mut address_bytes)
                .map_err(|_| ParseOnTransferMessageError::WrongMessageFormat)?;
            Address::from_array(address_bytes)
        };

        if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
            && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
        {
            recipient = fallback_address;
        }

        let erc20_token = get_erc20_from_nep141(&self.io, token)?;
        let erc20_admin_address = current_address(current_account_id);
        let result = self
            .call(
                &erc20_admin_address,
                &erc20_token,
                Wei::zero(),
                setup_receive_erc20_tokens_input(&recipient, amount),
                u64::MAX,
                Vec::new(), // TODO: are there values we should put here?
                Vec::new(),
                handler,
            )
            .and_then(submit_result_or_err)?;

        sdk::log!("Mint {amount} ERC-20 tokens for: {}", recipient.encode());

        // Return SubmitResult so that it can be accessed in standalone engine.
        // This is used to help with the indexing of bridge transactions.
        Ok(Some(result))
    }
