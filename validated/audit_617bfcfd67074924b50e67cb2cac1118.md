### Title
Missing Lower-Bound Validation on `threshold` in FROST Signing and Presigning Allows Malicious Coordinator to Corrupt Signing Outputs - (File: src/frost/mod.rs)

---

### Summary

`assert_sign_inputs` and `presign` in `src/frost/mod.rs` validate that `threshold` is not too large but never enforce a minimum of 2. The `ReconstructionLowerBound` type itself imposes no floor. A malicious coordinator can supply `threshold = 1` (or `0`) to honest participants, who will accept it and proceed, producing cryptographically invalid/unusable signature outputs. The same gap exists in `src/ecdsa/ot_based_ecdsa/sign.rs`.

---

### Finding Description

`ReconstructionLowerBound` is a plain newtype over `usize` with no enforced minimum: [1](#0-0) 

`assert_key_invariants` (DKG) and `validate_triple_inputs` (triple generation) both explicitly reject `threshold < 2`: [2](#0-1) [3](#0-2) 

However, `assert_sign_inputs` — the shared validation gate for EdDSA and RedJubjub signing — only checks the upper bound: [4](#0-3) 

There is no corresponding `threshold < 2` guard. The same omission exists in `presign` for FROST: [5](#0-4) 

And in OT-based ECDSA `sign`: [6](#0-5) 

Because DKG correctly rejects `threshold < 2`, any live key share was produced under a polynomial of degree ≥ 1 (threshold ≥ 2). When a signing session is then initiated with `threshold = 1`, the Lagrange interpolation inside the signing round uses only one share and the wrong basis, producing a scalar that does not correspond to the actual secret. The resulting signature fails standard verification.

---

### Impact Explanation

Honest participants call `assert_sign_inputs` to validate the coordinator-supplied parameters before committing their nonces and signature shares. Because the lower-bound check is absent, they accept `threshold = 1` and complete the protocol, outputting a signature that is cryptographically invalid against the public key established during DKG. Every honest participant produces and accepts an unusable signing transcript. This maps to:

> **High: Corruption of sign outputs so honest parties accept unusable cryptographic outputs.**

---

### Likelihood Explanation

The coordinator role is a designated participant, not a fully trusted party — the protocol is designed to tolerate a malicious coordinator in the sense that honest participants validate inputs independently. The attack requires only that the coordinator supply a crafted `threshold` value in the signing invocation. No key material, side channels, or external assumptions are needed. Any participant who is also the coordinator (a common deployment pattern) can trivially trigger this against co-signers.

---

### Recommendation

Add the same lower-bound guard that exists in `assert_key_invariants` and `validate_triple_inputs` to both `assert_sign_inputs` and `presign` in `src/frost/mod.rs`, and to `sign` in `src/ecdsa/ot_based_ecdsa/sign.rs`:

```rust
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

Alternatively, enforce the minimum inside `ReconstructionLowerBound::new` so the invariant is guaranteed at the type level across all call sites.

---

### Proof of Concept

1. Run DKG for 3 participants with `threshold = 2` (accepted; produces valid shares).
2. Initiate a FROST EdDSA signing session. The coordinator calls `assert_sign_inputs` with the same 3 participants but passes `threshold = 1`.
3. The check `threshold.value() > participants.len()` evaluates `1 > 3 = false`; no error is returned.
4. Honest participants proceed, compute Lagrange coefficients for a degree-0 polynomial (single-point interpolation), and emit signature shares that reconstruct the wrong scalar.
5. The coordinator assembles the final signature; it fails `ed25519` verification against the public key from step 1.
6. Contrast: calling `keygen` with `threshold = 1` is immediately rejected at line 580–581 of `src/dkg.rs`, confirming the inconsistency is specific to the signing path. [7](#0-6) [8](#0-7)

pub fn assert_sign_inputs(
    participants: &[Participant],
    threshold: impl Into<ReconstructionLowerBound>,
    me: Participant,
    coordinator: Participant,
) -> Result<ParticipantList, InitializationError> {
    let threshold = threshold.into();
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // validate threshold
    if threshold.value() > participants.len() {
        return Err(InitializationError::ThresholdTooLarge {
            threshold: threshold.value(),
            max: participants.len(),
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }
    Ok(participants)
}
