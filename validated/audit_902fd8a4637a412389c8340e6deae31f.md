### Title
`protocolUnpausePool` Sets Pause Level to 1 Instead of 0, Leaving Pool Permanently Paused After Protocol Unpause - (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`MetricOmmPoolFactory.protocolUnpausePool` calls `setPause(1)` instead of `setPause(0)`, so a pool that was protocol-paused (level 2) is moved only to admin-paused (level 1) after the owner "unpauses" it. The pool remains paused and all swaps continue to revert.

---

### Finding Description

The pool's `pauseLevel` has three states:

| Value | Meaning |
|---|---|
| `0` | Active |
| `1` | Paused by admin |
| `2` | Paused by protocol |

`_checkNotPaused` rejects any non-zero level:

```solidity
// MetricOmmPool.sol line 643-645
function _checkNotPaused() internal view {
    if (pauseLevel != 0) revert PoolPaused();
}
```

The factory's `protocolUnpausePool` is supposed to restore a protocol-paused pool to active, but it calls `setPause(1)` — the admin-paused level — instead of `setPause(0)`:

```solidity
// MetricOmmPoolFactory.sol lines 399-403
function protocolUnpausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 2) revert InvalidPauseTransition(cur, 1);
    IMetricOmmPoolFactoryActions(pool).setPause(1);   // ← should be setPause(0)
}
```

After this call the pool is at level 1. Because `_checkNotPaused` rejects any value `!= 0`, every subsequent `swap` call still reverts with `PoolPaused`. The pool admin must then call `unpausePool` (which transitions 1 → 0) to actually restore the pool. If the admin is unresponsive, the pool is permanently stuck at level 1.

Compare with the correctly implemented admin-side counterpart:

```solidity
// MetricOmmPoolFactory.sol lines 467-471
function unpausePool(address pool) external override nonReentrant onlyPoolAdmin(pool) {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 1) revert InvalidPauseTransition(cur, 0);
    IMetricOmmPoolFactoryActions(pool).setPause(0);   // ← correct: sets to 0
}
```

The admin path correctly targets level 0; the protocol path does not.

---

### Impact Explanation

After `protocolUnpausePool` is called:

- `pauseLevel == 1` → `_checkNotPaused` still reverts → **all swaps remain blocked**.
- LPs can still withdraw (withdraw path does not call `whenNotPaused`), so principal is not directly lost, but the pool's core swap functionality is completely broken.
- If the pool admin is absent or malicious, the protocol owner has no further lever to restore the pool; `protocolUnpausePool` can only be called once (it requires `cur == 2`), and after it runs the pool is at level 1 — outside the protocol owner's reach.

This is a broken core pool functionality impact: swap flows are permanently unusable after a protocol unpause unless the pool admin cooperates.

---

### Likelihood Explanation

The trigger is a normal, expected owner action: pausing a pool for an emergency and then unpausing it. No special attacker setup is required. Any pool that has ever been protocol-paused and then "unpaused" via `protocolUnpausePool` is affected. The call succeeds (no revert), so the owner receives no on-chain signal that the pool is still paused.

---

### Recommendation

Change `setPause(1)` to `setPause(0)` in `protocolUnpausePool`:

```solidity
function protocolUnpausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 2) revert InvalidPauseTransition(cur, 0);   // target is 0
    IMetricOmmPoolFactoryActions(pool).setPause(0);         // ← fix
}
```

---

### Proof of Concept

1. Owner calls `protocolPausePool(pool)` — pool moves from level 0 → 2.
2. Owner calls `protocolUnpausePool(pool)` — pool moves from level 2 → **1** (not 0).
3. Any user calls `pool.swap(...)` — `_checkNotPaused` sees `pauseLevel == 1 != 0` → reverts with `PoolPaused`.
4. Pool admin must call `unpausePool(pool)` to reach level 0. Without admin cooperation the pool is permanently paused.
