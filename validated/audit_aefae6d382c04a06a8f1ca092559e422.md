### Title
Presignature Material Has No Single-Use Enforcement — Reuse via `rerandomize_presign` Enables Private Key Extraction - (File: `src/ecdsa/ot_based_ecdsa/mod.rs`, `src/ecdsa/robust_ecdsa/mod.rs`)

---

### Summary

Both the OT-based ECDSA and Robust ECDSA `PresignOutput` types derive `Clone + Serialize + Deserialize` and expose a `rerandomize_presign` function that accepts the presignature **by shared reference** (`&PresignOutput`). This means the same presignature can be rerandomized and consumed in multiple independent signing sessions without any library-level enforcement of single-use semantics. A malicious coordinator can exploit this to obtain two valid signatures sharing a related nonce, enabling full private key extraction via standard ECDSA nonce-reuse algebra.

---

### Finding Description

**Root cause — `rerandomize_presign` takes `PresignOutput` by reference:**

For OT-based ECDSA: [1](#0-0) 

For Robust ECDSA: [2](#0-1) 

Both functions accept `presignature: &PresignOutput` — a shared reference — so the original `PresignOutput` is **never consumed**. A caller (or a malicious coordinator directing participants) can call `rerandomize_presign` on the same `PresignOutput` any number of times, each time with different `RerandomizationArguments` (different `h`, `ε`, `ρ`), producing distinct `RerandomizedPresignOutput` values that are each passed by value into separate signing sessions.

**`PresignOutput` is `Clone + Serialize + Deserialize`:**

OT-based ECDSA: [3](#0-2) 

Robust ECDSA: [4](#0-3) 

These derives mean the presignature can be cloned in memory or serialized to persistent storage and deserialized later, making reuse trivially achievable across process restarts or crash-recovery scenarios.

**The sign function consuming `RerandomizedPresignOutput` by value provides no protection:** [5](#0-4) 

Consuming the *rerandomized* output by value only prevents reuse of a single rerandomized copy. It does not prevent the caller from calling `rerandomize_presign` again on the still-live `PresignOutput` to produce a second rerandomized copy.

**The library's own security documentation confirms the attack:** [6](#0-5) 

The documentation explicitly states that using the same presignature for two signing sessions with different `(h, ε)` values yields multiplicatively related nonces and allows secret key recovery. It mandates "never reuse a presignature" but the API does not enforce this. [7](#0-6) 

---

### Impact Explanation

In ECDSA, if two signatures `(R, s₁)` and `(R', s₂)` are produced from the same presignature with different messages `h₁`, `h₂` and tweaks `ε₁`, `ε₂`, the rerandomized nonces satisfy `k₁ = k · δ₁⁻¹` and `k₂ = k · δ₂⁻¹` where `δᵢ = HKDF(Y, εᵢ, hᵢ, R, ρᵢ)`. The ratio `k₁/k₂ = δ₂/δ₁` is fully known to the attacker (all inputs to HKDF are public). With two such signatures the attacker can solve for the secret key `x` using standard linear algebra over the scalar field. This constitutes **full extraction of the aggregate private signing key** — a Critical impact.

---

### Likelihood Explanation

The attack is reachable by a **malicious coordinator**, which is an explicitly documented adversary in this library. The coordinator controls which presignature is used for each signing session. By directing participants to use the same presignature for two signing requests (e.g., by aborting the first session before broadcasting the result, then initiating a second session with a different message), the coordinator obtains two signatures from the same nonce material. Participants have no library-enforced mechanism to detect or refuse presignature reuse. Additionally, the `Serialize + Deserialize` derives make crash-recovery scenarios (where a participant deserializes a stored presignature after an aborted session) a realistic accidental reuse vector.

---

### Recommendation

1. **Change `rerandomize_presign` to consume `PresignOutput` by value** (`presignature: PresignOutput`) for both OT-based and Robust ECDSA. This makes the Rust type system enforce single-use at the API boundary.
2. **Remove `Clone` from `PresignOutput`** (both variants) to prevent callers from trivially bypassing the by-value consumption.
3. **Consider removing `Serialize + Deserialize` from `PresignOutput`**, or document explicitly that deserialization of a previously used presignature is a critical security violation. If persistence for crash recovery is required, provide a separate, audited mechanism that marks presignatures as consumed before persisting them.
4. For the Robust ECDSA variant, the existing `N₁ = N₂ = 2t+1` enforcement in `sign()` prevents split-view attacks across different subsets but does **not** prevent the same subset from signing twice with the same presignature — this gap must be closed at the `rerandomize_presign` API level.

### Proof of Concept

```rust
// Attacker-controlled coordinator scenario (Robust ECDSA):

// Step 1: Participants run presign, each obtains PresignOutput
let presign_out: PresignOutput = /* result of presign protocol */;

// Step 2: Coordinator initiates signing session 1 (message m1, tweak e1)
let args1 = RerandomizationArguments { big_r: presign_out.big_r, tweak: e1, ... };
let rerand1 = RerandomizedPresignOutput::rerandomize_presign(&presign_out, &args1)?;
// rerand1 is passed to sign() and consumed — but presign_out is still alive

// Step 3: Coordinator aborts session 1 before result is broadcast

// Step 4: Coordinator initiates signing session 2 (message m2, tweak e2)
let args2 = RerandomizationArguments { big_r: presign_out.big_r, tweak: e2, ... };
let rerand2 = RerandomizedPresignOutput::rerandomize_presign(&presign_out, &args2)?;
// presign_out is reused — no compile error, no runtime error

// Step 5: Coordinator now holds two signatures (R1,s1) and (R2,s2) derived from
// the same underlying nonce k. Since δ1 and δ2 are both known (HKDF inputs are public),
// the ratio k1/k2 = δ2/δ1 is known, and the private key x is recoverable via:
//   s1 = k1*(h1 + R1x*x)  =>  x = (s1/k1 - h1) / R1x
// where k1 = k * δ1^{-1} and k is recoverable from k1 and k2.
```
