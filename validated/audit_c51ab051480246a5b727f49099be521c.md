### Title
Unbounded `stakers` Vec Growth Causes Permanent DoS on `get_stakers` - (`src/staking/staking.cairo`)

---

### Summary

The `stakers` storage `Vec` is append-only: every call to `stake()` pushes the staker address, but `remove_staker()` (called from `unstake_action()`) never removes it. `get_stakers()` iterates the **entire** Vec on every call. An unprivileged attacker can cycle N addresses through `stake → unstake_intent → unstake_action`, permanently growing the Vec to size N and eventually making `get_stakers` exceed the Starknet gas limit.

---

### Finding Description

**Root cause — push without pop:**

`stake()` unconditionally appends to `self.stakers`: [1](#0-0) 

`remove_staker()`, called inside `unstake_action()`, clears `staker_info`, zeroes balances, and clears the operational-address mapping — but **never touches `self.stakers`**: [2](#0-1) 

**Full-range iteration in `get_stakers`:**

`get_stakers` iterates `self.stakers.into_iter_full_range()`, visiting every slot ever written, including dead stakers: [3](#0-2) 

Dead stakers are skipped via `is_staker_active`, which returns `false` for `VInternalStakerInfo::None`: [4](#0-3) 

The `continue` avoids adding them to the output array, but the storage read and branch still consume gas for every dead entry.

---

### Impact Explanation

`get_stakers` is the consensus endpoint used to determine the validator set for each epoch. As the dead-staker Vec grows, each call's gas cost grows linearly with the total number of addresses that have ever staked. Once the Vec is large enough, the call will exceed Starknet's per-transaction gas ceiling and revert on every invocation — a **permanent denial of service** for the consensus layer.

---

### Likelihood Explanation

The attack requires no privilege. Each attacker address needs only `min_stake` STRK, which is returned after `unstake_action`. The only real cost is gas for the transactions and waiting through the exit window (default 1 week). An attacker can run many addresses in parallel, recovering capital from earlier batches while new batches are in the exit window. The attack is therefore economically feasible at scale.

---

### Recommendation

Remove the staker's address from `self.stakers` when `remove_staker` is called, or replace the `Vec` with a data structure that supports O(1) deletion (e.g., a swap-and-pop pattern using a `Map<index, address>` plus a length counter). Alternatively, maintain a separate active-staker count and cap `get_stakers` iteration to only active entries.

---

### Proof of Concept

```
for i in 0..N:
    addr_i = fresh_address()
    addr_i.approve(staking, min_stake)
    staking.stake(addr_i, min_stake, ...)   // stakers.push(addr_i) → Vec grows
    staking.unstake_intent()                // marks exit, does NOT pop Vec
    advance_time(exit_wait_window + 1)
    staking.unstake_action(addr_i)          // remove_staker: no Vec pop

// After N iterations, stakers.len() == N (all dead entries)
// get_stakers now iterates N slots, each requiring a storage read
// For large N, gas(get_stakers) > block_gas_limit → permanent revert
```

Invariant violated: `gas(get_stakers) <= C` for a fixed constant `C` — the actual cost is `O(N)` in the total number of historical stakers, not the number of active ones.
