### Title
`MsgTurnBridge` Handler is a No-Op with No Permission Check, Permanently Disabling the Bridge Circuit Breaker — (File: `x/cronos/keeper/msg_server.go`)

### Summary
The `TurnBridge` message handler is implemented as a complete no-op (`return nil, nil`). It performs no authorization check and makes no state changes. Any unprivileged user can submit `MsgTurnBridge` and receive a success response, bypassing the `CanTurnBridge` permission guard entirely. The bridge circuit breaker is permanently non-functional.

### Finding Description
The `MsgTurnBridge` message is fully defined in the proto surface, exposed via CLI, and documented in ADR-009 as the emergency circuit breaker for the Gravity bridge. The permission system explicitly reserves `CanTurnBridge` as a restricted operation requiring either the `CronosAdmin` address or a delegated permissioned address.

However, the production handler in `x/cronos/keeper/msg_server.go` is:

```go
// TurnBridge implements the grpc method
func (k msgServer) TurnBridge(goCtx context.Context, msg *types.MsgTurnBridge) (*types.MsgTurnBridgeResponse, error) {
    return nil, nil
}
``` [1](#0-0) 

Compare this to `UpdateTokenMapping`, which correctly enforces the permission check before any state mutation:

```go
if !k.HasPermission(ctx, msg.GetSigners(), CanChangeTokenMapping) {
    return nil, errors.Wrap(sdkerrors.ErrUnauthorized, "msg sender is not authorized")
}
``` [2](#0-1) 

The `TurnBridge` handler has no equivalent guard. The proto definition and CLI both expose the message as a live, signable transaction: [3](#0-2) 

ADR-009 explicitly states that `MsgTurnBridge` is one of the two messages that must require permission: [4](#0-3) 

### Impact Explanation
**High — Bypass of Cronos bridge authorization check.**

1. **Permission bypass**: Any unprivileged address can broadcast `MsgTurnBridge` with any `enable` value. The transaction is accepted by the chain (code 0, no error), bypassing the `CanTurnBridge` authorization check that the permission system is designed to enforce.

2. **Permanently non-functional circuit breaker**: Because the handler writes no state, the bridge state is never toggled. The Gravity bridge cannot be disabled via this message under any circumstances. In the event of a bridge exploit or oracle manipulation, the intended emergency stop mechanism silently does nothing, allowing continued unauthorized bridge activity and potential asset loss.

### Likelihood Explanation
The vulnerability is unconditional and present in every block. Any unprivileged user who submits a `MsgTurnBridge` transaction will receive a success response. The bridge circuit breaker is permanently broken regardless of network conditions.

### Recommendation
Implement `TurnBridge` to:
1. Check `HasPermission(ctx, msg.GetSigners(), CanTurnBridge)` and return `ErrUnauthorized` on failure, mirroring the pattern in `UpdateTokenMapping`.
2. Persist the bridge-enabled flag to the keeper's KV store.
3. Enforce the flag in the Gravity bridge's `EndBlocker` / batch-creation path so that bridge operations are actually halted when `enable=false`.

### Proof of Concept
1. Any unprivileged address signs and broadcasts `MsgTurnBridge{Sender: <attacker>, Enable: false}`.
2. The chain processes the transaction; `TurnBridge` returns `nil, nil` — the transaction succeeds with code 0.
3. No bridge-state key is written to the KV store.
4. The Gravity bridge continues processing batches and send-to-Ethereum requests as if the message was never sent.
5. Conversely, a legitimate admin calling `turn_bridge false` also receives code 0 but achieves no effect — the circuit breaker is silently broken for all callers. [1](#0-0) [5](#0-4) [6](#0-5)
