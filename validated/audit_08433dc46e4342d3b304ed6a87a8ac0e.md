### Title
`priceProviderTimelock = 0` Bypasses Oracle Timelock Entirely, Enabling Atomic Oracle Swap — (`File: metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary

`_validatePoolParameters` does not enforce a minimum value on `params.priceProviderTimelock` for mutable-oracle pools. When `priceProviderTimelock[pool] == 0`, the pool admin can call `proposePoolPriceProvider` and `executePoolPriceProviderUpdate` in the same block, atomically replacing the oracle with no delay. This completely nullifies the timelock protection that was introduced specifically to fix audit finding M01 ("Admin can arbitrarily change the pool priceProvider").

### Finding Description

`createPool` stores `params.priceProviderTimelock` verbatim into `priceProviderTimelock[pool]` with no lower-bound check: [1](#0-0) [2](#0-1) 

`_validatePoolParameters` validates tokens, admin, fees, and initial amounts, but contains no check that `priceProviderTimelock` is non-zero when it is not `type(uint256).max`: [3](#0-2) 

In `proposePoolPriceProvider`, `executeAfter` is computed as:

```solidity
uint256 executeAfter = block.timestamp + timelock; // = block.timestamp when timelock == 0
``` [4](#0-3) 

In `executePoolPriceProviderUpdate`, the guard is:

```solidity
if (block.timestamp < execAfter) revert PriceProviderTimelockNotElapsed(...);
``` [5](#0-4) 

When `timelock == 0`, `execAfter == block.timestamp`, so `block.timestamp < block.timestamp` is `false` — the revert never fires. Both calls succeed in the same transaction.

The prior audit finding M01 was explicitly fixed by introducing this timelock mechanism to prevent the admin from arbitrarily replacing the oracle at runtime: [6](#0-5) 

Allowing `priceProviderTimelock = 0` restores the original vulnerability.

### Impact Explanation

A pool admin operating a pool with `priceProviderTimelock = 0` can atomically:
1. Call `proposePoolPriceProvider(pool, maliciousOracle)` — sets `pendingPriceProviderExecuteAfter = block.timestamp`.
2. Call `executePoolPriceProviderUpdate(pool)` in the same transaction — guard passes, malicious oracle is live.
3. Execute swaps against the malicious oracle returning extreme bid/ask prices, draining LP funds.

This is a **bad-price execution** impact: an unbounded or adversarially-controlled bid/ask quote reaches pool swap math, causing traders to receive more than permitted or LPs to lose principal. It also constitutes an **admin-boundary break**: the pool admin is supposed to be constrained by timelocks for oracle changes, but the constraint is absent.

### Likelihood Explanation

Any pool created with `priceProviderTimelock` set to any value other than `type(uint256).max` and not explicitly set to a meaningful delay (including `0`) is affected. The factory imposes no minimum, and the documentation only provides a guideline ("pick a finite delay"), not an enforcement. A pool admin who is also the pool creator can deploy such a pool deliberately. LPs depositing into a mutable-oracle pool with `priceProviderTimelock = 0` have no on-chain protection.

### Recommendation

In `_validatePoolParameters`, add a check that rejects `priceProviderTimelock == 0` for mutable-oracle pools:

```solidity
if (params.priceProviderTimelock != type(uint256).max && params.priceProviderTimelock == 0) {
    revert InvalidPriceProviderTimelock();
}
```

Alternatively, enforce a protocol-level minimum (e.g., 1 day) for mutable-oracle pools, consistent with the security model described in the documentation.

### Proof of Concept

```solidity
// Pool created with priceProviderTimelock = 0 (mutable mode, no delay)
PoolParameters memory params = ...;
params.priceProviderTimelock = 0; // NOT type(uint256).max
address pool = factory.createPool(params);

// Admin atomically swaps oracle in one transaction
vm.startPrank(admin);
factory.proposePoolPriceProvider(pool, address(maliciousOracle));
// pendingPriceProviderExecuteAfter[pool] == block.timestamp
// block.timestamp < block.timestamp == false → no revert
factory.executePoolPriceProviderUpdate(pool);
// maliciousOracle is now live — returns extreme prices
vm.stopPrank();

// Swap executes at adversarial price, draining LP funds
pool.swap(...);
```

## M01 - Admin can arbitrarily change the pool priceProvider

### Finding summary

`setPriceProvider()` allowed admin to replace oracle at runtime, and `priceProvider` was returned by `getImmutables()`.

### Resolution

Fixed.

Changes:

- Added constructor config `priceProviderTimelock`.
- Added admin flow:
  - `proposePriceProvider(address newPriceProvider)`
  - `executePriceProviderUpdate()`
- Added events:
  - `PriceProviderChangeProposed(currentPriceProvider, newPriceProvider, executeAfter)`
  - `PriceProviderUpdated(newPriceProvider)`

Behavior now:

- `priceProviderTimelock == type(uint256).max` => immutable mode (admin updates disabled).
- `priceProviderTimelock < type(uint256).max` => mutable mode with delayed execution.
- Price provider used by swaps/reports resolves through timelock mode; active provider is read via `MetricOmmPoolStateView.priceProvider(pool)`.
```
