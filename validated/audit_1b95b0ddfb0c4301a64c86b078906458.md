### Title
`programInvalidator.ShouldInvalidateEntries()` ignores `invalidated` map, causing stale program cache on Access nodes after contract updates — (`module/state_synchronization/indexer/util.go`)

### Summary
`programInvalidator.ShouldInvalidateEntries()` returns only `inv.invalidateAll`, ignoring the populated `inv.invalidated` map. When a contract is updated in a block that contains no service-account-authorized transaction (`invalidateAll=false`), `TableTransaction.AddInvalidator` skips registration of the invalidator entirely. The program cache is never purged for the updated contract, and subsequent script executions on the Access node use the stale pre-update Cadence program.

### Finding Description
In `module/state_synchronization/indexer/util.go`, `programInvalidator.ShouldInvalidateEntries()` (line 114–116) returns only `inv.invalidateAll`. The companion method `ShouldInvalidateEntry()` (line 118–121) correctly checks `inv.invalidateAll || ok` against the `inv.invalidated` map, but it is never reached because `AddInvalidator` in `fvm/storage/derived/table.go` (line 531) gates on `ShouldInvalidateEntries()` first and returns early when it is `false`. The `invalidated` map populated by `findContractUpdates()` is therefore completely unused in the non-service-account case.

### Impact Explanation
An Access node's program cache retains the pre-update version of any contract updated in a block without a service-account transaction. Scripts executed via the Access API against that node use the stale Cadence program. If the contract update was intended to remove a public function, revoke a capability, or restrict access to private fields, the old code path remains callable through the stale cache, exposing data or capabilities the contract owner intended to restrict.

### Likelihood Explanation
Most ordinary contract updates (user-deployed contracts) do not include a service-account-authorized transaction in the same block, so `invalidateAll` is `false` for the vast majority of real-world updates. The stale cache persists until the next block that happens to contain a service-account transaction. This is a deterministic, always-reproducible bug on unmodified Access node code — not a race condition or probabilistic event.

### Recommendation
Fix `ShouldInvalidateEntries()` in `module/state_synchronization/indexer/util.go` to also return `true` when the `invalidated` map is non-empty:

```go
func (inv *programInvalidator) ShouldInvalidateEntries() bool {
    return inv.invalidateAll || len(inv.invalidated) > 0
}
```

This mirrors the pattern already used correctly in the test helper `testInvalidator.ShouldInvalidateEntries()` in `fvm/storage/derived/table_invalidator_test.go` (line 17–20), which returns `invalidator.invalidateAll || invalidator.invalidateName != ""`. [5](#0-4) 

### Proof of Concept
1. Deploy contract `C` with a public function `getSecret()` on a localnet Access node that has `derivedChainData` enabled.
2. Submit a block that updates `C` to remove `getSecret()`, with no service-account transaction in the block.
3. `updateProgramCache` is called: `invalidateAll=false`, `invalidated={C:{}}`.
4. `tx.AddInvalidator(...)` calls `ShouldInvalidateEntries()` → returns `false` → returns early.
5. `tx.Commit()` runs; no invalidation loop executes; `C`'s cache entry remains.
6. Execute a script via the Access API that calls `C.getSecret()` — it succeeds using the stale cached program, even though the on-chain contract no longer exposes that function.
