### Title
Silo Mode `fixed_gas` Can Be Increased by Owner Immediately and Without Notice, Charging Users More ETH Than Expected - (File: engine/src/engine.rs)

---

### Summary

In Aurora Engine's Silo mode, the owner can call `set_fixed_gas()` or `set_silo_params()` to change the `fixed_gas` parameter at any time with no timelock and no notice period. This parameter completely replaces the actual gas consumed when computing the ETH fee charged to a user. Because `SubmitArgs` exposes a `max_gas_price` cap but no `max_fixed_gas` cap, users have no on-chain mechanism to bound the fee they will actually pay. An owner who raises `fixed_gas` — even without malicious intent — causes every subsequent transaction to be charged `fixed_gas × gas_price` instead of `actual_gas_used × gas_price`, draining more ETH from users than they signed for.

---

### Finding Description

**Admin-controlled parameter, no upper bound, no timelock**

`set_fixed_gas()` and `set_silo_params()` are owner-only entry points that write the new value directly to storage with no delay:

```rust
// engine/src/lib.rs  lines 784-793
pub extern "C" fn set_fixed_gas() {
    let mut io = Runtime;
    let state = state::get_state(&io).sdk_unwrap();
    require_owner_and_running(&state, &io.predecessor_account_id())
        .map_err(ContractError::msg).sdk_unwrap();
    let args: FixedGasArgs = io.read_input_borsh().sdk_unwrap();
    silo::set_fixed_gas(&mut io, args.fixed_gas);   // immediate, unbounded
}
```

`FixedGasArgs.fixed_gas` is `Option<EthGas>` where `EthGas` wraps a bare `u64` — there is no protocol-level ceiling.

**`fixed_gas` is read at execution time and replaces `gas_limit` in the fee formula**

Inside `submit_with_alt_modexp()` the current storage value is fetched at the moment the transaction is processed:

```rust
// engine/src/engine.rs  line 1049
let fixed_gas = silo::get_fixed_gas(&io);
```

The only guard is that `fixed_gas` must not exceed the transaction's own `gas_limit`; if it does the transaction is rejected outright:

```rust
// engine/src/engine.rs  lines 1066-1068
if fixed_gas.is_some_and(|gas| gas.as_u256() > transaction.gas_limit) {
    return Err(EngineErrorKind::FixedGasOverflow.into());
}
```

Otherwise `fixed_gas` is forwarded to `charge_gas()`, where it **completely replaces** `transaction.gas_limit` in the prepaid-amount calculation:

```rust
// engine/src/engine.rs  lines 496-500
let prepaid_amount = fixed_gas
    .map_or(transaction.gas_limit, EthGas::as_u256)
    .checked_mul(effective_gas_price)
    .map(Wei::new)
    .ok_or(GasPaymentError::EthAmountOverflow)?;
```

**`refund_unused_gas()` also uses `fixed_gas`, so no refund is issued**

```rust
// engine/src/engine.rs  lines 1276-1283
let gas_to_wei = |price: U256| {
    fixed_gas
        .map_or_else(|| gas_used.into(), EthGas::as_u256)
        .checked_mul(price)
        .map(Wei::new)
        .ok_or(GasPaymentError::EthAmountOverflow)
};
let spent_amount = gas_to_wei(gas_result.effective_gas_price)?;
```

Because both `prepaid_amount` and `spent_amount` equal `fixed_gas × gas_price`, the refund is always zero when `fixed_gas` is active. The user pays the full `fixed_gas × gas_price` regardless of how little gas the EVM actually consumed.

**No user-side cap exists for `fixed_gas`**

`SubmitArgs` carries a `max_gas_price` field that lets a user bound the gas price:

```rust
// engine/src/engine.rs  line 1100
let max_gas_price = args.max_gas_price.map(Into::into);
```

There is no analogous `max_fixed_gas` field. A user who signs a transaction with `gas_limit = 1_000_000` and `gas_price = 1 Gwei` expecting to pay ~21,000 Gwei (a simple transfer) will pay up to `999_999 × 1 Gwei` if the owner raises `fixed_gas` to `999_999` before the transaction lands.

---

### Impact Explanation

Every Silo user who submits a transaction after an unannounced `fixed_gas` increase pays `fixed_gas × gas_price` instead of `actual_gas_used × gas_price`. The difference is a direct, irreversible loss of ETH from the user's Aurora balance. Because `fixed_gas` can be set to any value up to the transaction's `gas_limit` (which users routinely set high to avoid out-of-gas failures), the overcharge can be orders of magnitude larger than the true cost. The excess is distributed to the relayer as a priority-fee reward, constituting a transfer of user funds to a third party without the user's informed consent.

**Impact: High — theft of user funds (excess fees taken from user balances).**

---

### Likelihood Explanation

Silo operators legitimately adjust `fixed_gas` to reflect changes in NEAR gas costs or to recoup infrastructure expenses. No malice is required: a routine upward adjustment made while user transactions are in flight (e.g., sitting in a relayer queue) silently increases the fee those transactions pay. Because NEAR gas prices are not fixed and Silo operators have operational incentives to tune `fixed_gas`, this scenario is realistic and requires no attacker beyond the normal admin role acting in good faith.

---

### Recommendation

1. **Add a `max_fixed_gas` field to `SubmitArgs`** (analogous to the existing `max_gas_price`). Before calling `charge_gas()`, reject the transaction if `fixed_gas > args.max_fixed_gas`, giving users a hard on-chain bound on the fee they will pay.
2. **Alternatively, implement a timelock for `fixed_gas` changes** (similar to `upgrade_delay_blocks`) so that users have advance notice before a new value takes effect and can choose not to transact during the transition window.

---

### Proof of Concept

1. Silo owner deploys Aurora Engine in Silo mode with `fixed_gas = 21_000`.
2. User signs a transaction: `gas_limit = 1_000_000`, `gas_price = 1 Gwei`. User expects to pay ≈ 21,000 Gwei for a simple ETH transfer.
3. Owner calls `set_fixed_gas(fixed_gas = Some(999_999))` — takes effect immediately, no delay.
4. Relayer submits the user's signed transaction. At line 1066 the check passes (`999_999 < 1_000_000`).
5. `charge_gas()` deducts `999_999 × 1 Gwei = 999_999 Gwei` from the user's balance.
6. `refund_unused_gas()` computes `spent_amount = 999_999 × 1 Gwei`; refund = 0.
7. User loses `999_999 − 21_000 = 978_999 Gwei` more than they agreed to when signing — with no recourse.

    pub fn charge_gas(
        &mut self,
        sender: &Address,
        transaction: &NormalizedEthTransaction,
        max_gas_price: Option<U256>,
        fixed_gas: Option<EthGas>,
    ) -> Result<GasPaymentResult, GasPaymentError> {
        let block_base_fee_per_gas = self.block_base_fee_per_gas();
        if transaction.max_fee_per_gas.is_zero()
            && fixed_gas.is_none()
            && block_base_fee_per_gas.is_zero()
        {
            return Ok(GasPaymentResult::default());
        }

        if transaction.max_fee_per_gas < block_base_fee_per_gas {
            return Err(GasPaymentError::MaxFeePerGasLessThanBaseFee);
        }

        let priority_fee_per_gas = transaction
            .max_priority_fee_per_gas
            .min(transaction.max_fee_per_gas - block_base_fee_per_gas);
        let priority_fee_per_gas = max_gas_price.map_or(priority_fee_per_gas, |price| {
            price.min(priority_fee_per_gas)
        });
        let effective_gas_price = priority_fee_per_gas + block_base_fee_per_gas;
        // First, we try to use `fixed_gas`. At this point we already know that the `fixed_gas` is
        // less than the `gas_limit`. It allows avoiding refunding unused gas to the sender later.
        let prepaid_amount = fixed_gas
            .map_or(transaction.gas_limit, EthGas::as_u256)
            .checked_mul(effective_gas_price)
            .map(Wei::new)
            .ok_or(GasPaymentError::EthAmountOverflow)?;

        let new_balance = get_balance(&self.io, sender)
            .checked_sub(prepaid_amount)
            .ok_or(GasPaymentError::OutOfFund)?;

        set_balance(&mut self.io, sender, &new_balance);

        self.gas_price = effective_gas_price;

        Ok(GasPaymentResult {
            prepaid_amount,
            effective_gas_price,
            priority_fee_per_gas,
        })

pub fn refund_unused_gas<I: IO>(
    io: &mut I,
    sender: &Address,
    gas_used: u64,
    gas_result: &GasPaymentResult,
    relayer: &Address,
    fixed_gas: Option<EthGas>,
) -> Result<(), GasPaymentError> {
    if gas_result.effective_gas_price.is_zero() {
        return Ok(());
    }

    let (refund, relayer_reward) = {
        let gas_to_wei = |price: U256| {
            fixed_gas
                .map_or_else(|| gas_used.into(), EthGas::as_u256)
                .checked_mul(price)
                .map(Wei::new)
                .ok_or(GasPaymentError::EthAmountOverflow)
        };

        let spent_amount = gas_to_wei(gas_result.effective_gas_price)?;
        let reward_amount = gas_to_wei(gas_result.priority_fee_per_gas)?;

        let refund = gas_result
            .prepaid_amount
            .checked_sub(spent_amount)
            .ok_or(GasPaymentError::EthAmountOverflow)?;

        (refund, reward_amount)
    };

    if !refund.is_zero() {
        add_balance(io, sender, refund)?;
    }

    if !relayer_reward.is_zero() {
        add_balance(io, relayer, relayer_reward)?;
    }

    Ok(())
