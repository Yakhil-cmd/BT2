### Title
Missing Lower-Bound Threshold Validation in FROST Signing Allows Sub-Threshold Protocol Execution - (File: src/frost/mod.rs)

### Summary

`assert_sign_inputs` in `src/frost/mod.rs` validates that `threshold <= participants.len()` but omits the symmetric lower-bound check (`threshold >= 2`) that is present in every other entry-point in the library. A caller can pass `threshold = 1` (or `0`) to `sign_v1`, `sign_v2` (EdDSA), or `sign` (RedJubjub), causing the signing protocol to proceed with a threshold that violates the documented security invariant, producing an unusable or cryptographically corrupt signature output.

### Finding Description

`assert_key_invariants` in `src/dkg.rs` enforces both bounds on the threshold:

```
if threshold > participants.len() { ... ThresholdTooLarge ... }
if threshold < 2                  { ... ThresholdTooSmall ... }
``` [1](#0-0) 

`validate_triple_inputs` in the OT-based triple generation likewise enforces both bounds: [2](#0-1) 

`assert_sign_inputs` in `src/frost/mod.rs`, however, only checks the upper bound:

```rust
if threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
// ← no check for threshold < 2
``` [3](#0-2) 

`ReconstructionLowerBound` is a plain `usize` newtype with no constructor-level invariant, so any value including `0` and `1` is representable and accepted: [4](#0-3) 

The three public signing entry-points that delegate to `assert_sign_inputs` without adding their own lower-bound check are:

- `sign_v1` / `sign_v2` in `src/frost/eddsa/sign.rs` [5](#0-4) 
- `sign` in `src/frost/redjubjub/sign.rs` [6](#0-5) 

The `presign` function in `src/frost/mod.rs` has the same omission for the EdDSA/RedJubjub presign path: [7](#0-6) 

### Impact Explanation

When `threshold = 1` is accepted, the FROST `KeyPackage` is constructed with `min_signers = 1`. The FROST `aggregate` function then requires only one signature share. The coordinator can include only its own nonce commitment in the `SigningPackage`, collect only its own signature share, and call `aggregate`. Because the Lagrange coefficient for a single-element signing set is `1` rather than the correct multi-party coefficient, the resulting aggregate `z` equals `nonce_i + signing_share_i * challenge` rather than `nonce + secret_key * challenge`. The output is a structurally well-formed but cryptographically invalid signature. Honest participants have consumed their nonces and believe the signing session completed, but the output is unusable.

This maps to: **High — Corruption of sign outputs so honest parties accept unusable cryptographic outputs.**

### Likelihood Explanation

The coordinator role is a documented trust boundary in the FROST signing API. A misconfigured or malicious coordinator can trivially pass `threshold = 1` as a `usize` literal; `ReconstructionLowerBound::from(1usize)` compiles and passes all existing guards. No cryptographic capability or key material is required. The call path is fully reachable from the public API.

### Recommendation

Add the same lower-bound guard that `assert_key_invariants` and `validate_triple_inputs` already enforce:

```rust
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

Apply this to both `assert_sign_inputs` and the `presign` function in `src/frost/mod.rs`. [8](#0-7) 

### Proof of Concept

```rust
// Keys generated with threshold = 3, participants = [P1, P2, P3]
// Malicious coordinator calls sign_v1 with threshold = 1:
let result = sign_v1(
    &[p1, p2, p3],
    1usize,          // ← threshold = 1, passes assert_sign_inputs unchecked
    p1,              // me = coordinator
    p1,              // coordinator
    keygen_output,
    message,
    rng,
);
// result is Ok(...) — protocol proceeds
// Coordinator only needs its own signature share (min_signers = 1)
// Aggregate produces z = nonce_1 + signing_share_1 * c ≠ nonce + secret_key * c
// Output: invalid, unusable signature; honest parties' nonces consumed
```

The root cause is the absent `threshold < 2` guard in `assert_sign_inputs` at `src/frost/mod.rs:144–150`, directly analogous to the reported pattern of a state invariant being modified without validating that the controlling input is non-trivial. [3](#0-2)

```rust
/// Verifies that the sign inputs are valid
pub fn assert_sign_inputs(
    participants: &[Participant],
    threshold: impl Into<ReconstructionLowerBound>,
    me: Participant,
    coordinator: Participant,
) -> Result<ParticipantList, InitializationError> {
    let threshold = threshold.into();
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // validate threshold
    if threshold.value() > participants.len() {
        return Err(InitializationError::ThresholdTooLarge {
            threshold: threshold.value(),
            max: participants.len(),
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }
    Ok(participants)
}
```
