### Title
Unprivileged Caller Can Inject Fake Interop Roots via Unguarded Event Hook - (`system_hooks/src/event_hooks/interop_root_reporter.rs`)

### Summary

The `interop_root_reporter_event_hook` fires for **any** contract that emits an `InteropRootAdded` event with the correct ABI format. There is no check on the emitting contract's address. An unprivileged user can deploy a malicious contract, emit a crafted `InteropRootAdded` event, and inject arbitrary fake interop roots into the system's `InteropRootStorage`, corrupting the cross-chain rolling hash committed to in the block output.

### Finding Description

The event hook in `system_hooks/src/event_hooks/interop_root_reporter.rs` is registered to fire whenever any EVM contract emits an event whose first topic matches `INTEROP_ROOT_ADDED_EVENT_SIG` (`0x6b451b84...`). [1](#0-0) 

The function signature accepts `topics`, `data`, `_caller_ee`, `system`, and `resources` — but **no `caller` / emitting-address parameter**: [1](#0-0) 

The only validation performed is:

1. First topic matches the event signature.
2. `data.len() == 96`.
3. ABI offset word equals `32`.
4. Length word equals `1`.
5. `topics.len() == 3`. [2](#0-1) 

If all five checks pass, the hook unconditionally calls `system.io.add_interop_root(...)` with attacker-controlled `root`, `chain_id`, and `block_or_batch_number`: [3](#0-2) 

The intended emitter is the `L2_INTEROP_ROOT_STORAGE_ADDRESS` contract, which is only callable via a signed service transaction (`0x7d`) with a whitelisted `(destination, selector)` pair: [4](#0-3) 

However, the event hook itself enforces **no such restriction**. Any EVM contract deployed by any user can emit the matching event.

The injected roots are stored in `InteropRootStorage` and are iterated over when computing the `interop_roots_rolling_hash` committed to in the block output: [5](#0-4) [6](#0-5) 

### Impact Explanation

An attacker can corrupt the interop roots rolling hash that is committed to in the block output. This hash is the settlement layer's source of truth for cross-chain state. Injecting fake roots with attacker-chosen `chain_id`, `block_or_batch_number`, and `root` values allows:

- Forging cross-chain state proofs for chains the attacker controls.
- Corrupting the rolling hash so that legitimate cross-chain messages from honest chains fail verification.
- Permanently poisoning the interop root set for a block, since `InteropRootStorage` has no deduplication or source-address guard.

The `InteropRoot` struct explicitly documents that `root` cannot be zero for valid roots, but imposes no other constraint on who may supply it: [7](#0-6) 

### Likelihood Explanation

The attack requires only a standard EVM transaction: deploy a contract that emits `InteropRootAdded(chainId, blockNumber, [fakeRoot])` with the correct ABI encoding, then call it. No privileged key, governance vote, or operator access is needed. The event hook fires during normal EVM execution for any emitting address.

### Recommendation

Add an emitting-address guard to `interop_root_reporter_event_hook`. The hook should accept the emitting contract's address as a parameter and reject any event not originating from `L2_INTEROP_ROOT_STORAGE_ADDRESS`:

```rust
pub fn interop_root_reporter_event_hook<S: EthereumLikeTypes>(
    emitter: &B160,          // <-- add this
    topics: &arrayvec::ArrayVec<...>,
    data: &[u8],
    ...
) -> Result<(), SystemError> {
    if emitter != &L2_INTEROP_ROOT_STORAGE_ADDRESS {
        return Ok(());
    }
    // ... existing checks
}
```

This mirrors the pattern already used by every call hook in the codebase (e.g., `l1_messenger_hook` checks `caller != L1_MESSENGER_ADDRESS` and returns an empty-account response): [8](#0-7) 

### Proof of Concept

1. Attacker deploys a contract with the following logic (pseudocode):

```solidity
// Emits InteropRootAdded(chainId, blockNumber, [fakeRoot])
// topic[0] = 0x6b451b8422636e45b93bf7f594fa2c1769d039766c4254a6e7f9c0ee1715cdb0
// topic[1] = attacker_chain_id
// topic[2] = attacker_block_number
// data     = abi.encode(32, 1, fakeRoot)  // offset=32, len=1, root=fakeRoot
function injectRoot(uint256 chainId, uint256 blockNum, bytes32 fakeRoot) external {
    emit InteropRootAdded(chainId, blockNum, [fakeRoot]);
}
```

2. Attacker submits a standard EVM transaction calling `injectRoot(victimChainId, victimBlockNum, fakeRoot)`.

3. During execution, `emit_event` fires the `interop_root_reporter_event_hook`. All five format checks pass. `system.io.add_interop_root(...)` is called with attacker-supplied values.

4. At block finalization, `calculate_interop_roots_rolling_hash` iterates over the now-poisoned `InteropRootStorage` and commits the corrupted hash to the block output. [9](#0-8) [5](#0-4)

```rust
pub fn interop_root_reporter_event_hook<S: EthereumLikeTypes>(
    topics: &arrayvec::ArrayVec<<S::IOTypes as SystemIOTypesConfig>::EventKey, MAX_EVENT_TOPICS>,
    data: &[u8],
    _caller_ee: u8,
    system: &mut System<S>,
    resources: &mut S::Resources,
) -> Result<(), SystemError>
where
{
    // First, ensure we're capturing the InteropRootAdded event
    if topics.is_empty() || topics[0].as_u8_array() != INTEROP_ROOT_ADDED_EVENT_SIG {
        return Ok(());
    }
    // Internal error if the data supplied doesn't match the expected value
    if data.len() != 96 {
        return Err(internal_error!("Interop root reporter event hook received bad data").into());
    }

    // Parse data
    let offset: u32 = match U256::from_be_slice(&data[..32]).try_into() {
        Ok(offset) => offset,
        Err(_) => {
            return Err(
                internal_error!("Interop root reporter event hook received bad offset").into(),
            );
        }
    };
    // This event is part of the system, but we check it anyways
    if offset != 32 {
        return Err(internal_error!("Interop root reporter event hook received bad offset").into());
    }

    let len: u32 = match U256::from_be_slice(&data[32..64]).try_into() {
        Ok(offset) => offset,
        Err(_) => {
            return Err(
                internal_error!("Interop root reporter event hook received bad length").into(),
            );
        }
    };
    // It should have exactly one side
    if len != 1 {
        return Err(internal_error!("Interop root reporter event hook received bad length").into());
    }
    // Validate topics length
    if topics.len() != 3 {
        return Err(internal_error!("Interop root reporter event hook received bad topics").into());
    }
```

```rust
pub fn calculate_interop_roots_rolling_hash<'a>(
    old_rolling_hash: Bytes32,
    roots: impl Iterator<Item = &'a InteropRoot>,
    hasher: &mut crypto::sha3::Keccak256,
) -> Bytes32 {
    let mut data = [0u8; 96];

    let mut rolling_hash = old_rolling_hash;
    for root in roots {
        data[0..32].copy_from_slice(&rolling_hash.as_u8_ref());
        data[32..64].copy_from_slice(&root.chain_id.to_be_bytes::<{ U256::BYTES }>());
        data[64..96].copy_from_slice(&root.block_or_batch_number.to_be_bytes::<{ U256::BYTES }>());
        hasher.update(data);

        // Note: now we have only one side
        hasher.update(root.root.as_u8_ref());

        rolling_hash = hasher.finalize_reset().into()
    }

    rolling_hash
}
```
