### Title
Malicious Participant Sends Unchecked CKD Contribution to Corrupt Confidential Key Derivation Output — (`File: src/confidential_key_derivation/protocol.rs`)

### Summary

In the Confidential Key Derivation (CKD) protocol, the coordinator collects `(big_y, big_c)` contribution pairs from each participant and sums them directly with no zero-knowledge proof or commitment-binding check. A single malicious participant can send an arbitrarily crafted pair, causing the coordinator to compute and accept a corrupted `CKDOutput`. Because the coordinator is the sole consumer of the aggregated result, every honest party that relies on the derived key receives an unusable or attacker-chosen output.

### Finding Description

**Root cause — `src/confidential_key_derivation/protocol.rs`, lines 50–57:**

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
```

Each participant is supposed to compute:

```
big_y  = y_i · G
big_c  = x_i · H(pk ‖ app_id) + y_i · app_pk
```

and then Lagrange-normalize both before sending. The coordinator simply adds every received pair to a running sum. There is **no proof** that the received `big_c` was formed with the participant's actual secret share `x_i`, and **no proof** that `big_y` and `big_c` are consistent with each other (i.e., that the same `y_i` was used in both).

**Attack path:**

1. A malicious participant `P_m` participates in a legitimate CKD session.
2. Instead of computing the correct `(norm_big_y, norm_big_c)`, `P_m` sends an arbitrary pair — e.g., `(G, G)`, `(0, 0)`, or any point of their choosing — to the coordinator via `chan.send_private`.
3. The coordinator adds the malicious values to the honest sum without any check.
4. The resulting `CKDOutput` is `(Y_honest + Y_malicious, C_honest + C_malicious)`.
5. When the coordinator calls `ckd_output.unmask(app_sk)`, the result is `C_honest + C_malicious − app_sk · (Y_honest + Y_malicious)`, which is not equal to `msk · H(pk ‖ app_id)`.

The honest parties accept this corrupted output as the legitimate confidential derived key.

**Why existing checks do not prevent this:**

The initialization checks in `ckd()` (lines 76–101) only validate participant-list membership and deduplication. The `recv_from_others` helper (used at line 51) only enforces that exactly one message per participant is received; it performs no cryptographic validation of the message content. There is no Feldman-style commitment, no Pedersen commitment, and no sigma proof binding `big_c` to the participant's actual key share.

### Impact Explanation

A single malicious participant can force the coordinator to output an incorrect `CKDOutput`. Every downstream consumer that calls `unmask` on this output receives a wrong confidential key — one that is neither `msk · H(pk ‖ app_id)` nor any predictable value. This permanently breaks the CKD session for all honest parties: the derived key is unusable, and the session cannot be distinguished from a successful one without an independent oracle for the correct key.

This matches the allowed impact: **High — Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.**

### Likelihood Explanation

Any participant in a CKD session is a reachable, unprivileged attacker entry point. No special keys, no network access beyond normal protocol participation, and no cryptographic breaks are required. The malicious participant only needs to deviate from the protocol in a single message. The attack succeeds in one protocol execution, analogous to the single-transaction drain in the external report.

### Recommendation

Require each participant to accompany their `(big_y, big_c)` contribution with a zero-knowledge proof of correct formation — specifically, a proof of knowledge of `y_i` such that `big_y = y_i · G` and `big_c − x_i · H(pk ‖ app_id) = y_i · app_pk`. Because `x_i · H(pk ‖ app_id)` can be computed from the public key share, this is a standard Chaum–Pedersen discrete-log equality proof. The coordinator must verify this proof before adding the contribution to the running sum.

### Proof of Concept

```
Participants: [P1 (honest), P2 (honest), P3 (malicious)]
Coordinator: P1

Normal execution:
  P2 → P1: (lambda2·big_y2, lambda2·big_c2)   [correct]
  P3 → P1: (lambda3·big_y3, lambda3·big_c3)   [correct]
  P1 computes own share, sums all → CKDOutput = msk·H(pk‖app_id) after unmask ✓

Malicious execution (P3 deviates):
  P2 → P1: (lambda2·big_y2, lambda2·big_c2)   [correct]
  P3 → P1: (G, G)                              [arbitrary, no proof required]
  P1 sums all → CKDOutput.big_y  = Y_honest + G
                CKDOutput.big_c  = C_honest + G
  unmask(app_sk) = C_honest + G − app_sk·(Y_honest + G)
                 = msk·H(pk‖app_id) + G·(1 − app_sk)   ≠ msk·H(pk‖app_id) ✗
```

pub fn ckd(
    participants: &[Participant],
    coordinator: Participant,
    me: Participant,
    key_pair: KeygenOutput,
    app_id: impl Into<AppId>,
    app_pk: PublicKey,
    rng: impl CryptoRngCore + Send + 'static,
) -> Result<impl Protocol<Output = CKDOutputOption>, InitializationError> {
    // not enough participants
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }

    // kick out duplicates
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
