### Title
Missing Zero-Address Validation in `change_reward_address` Causes Permanent Loss of Unclaimed Yield — (`src/staking/staking.cairo`, `src/pool/pool.cairo`)

---

### Summary

Both `Staking::change_reward_address` and `Pool::change_reward_address` accept a zero (`0x0`) address as the new reward address without any validation. When rewards are subsequently claimed, they are transferred to address `0x0` and permanently lost. This is the direct analog of the DebtRegistry null-beneficiary bug: a valid participant can store a null address in a field that the rest of the protocol assumes is always a live, non-zero address.

---

### Finding Description

`Staking::change_reward_address` (staking.cairo) performs only one validation on the incoming address — it must not be a registered token address:

```cairo
assert!(
    !self.does_token_exist(token_address: reward_address),
    "{}",
    GenericError::REWARD_ADDRESS_IS_TOKEN,
);
``` [1](#0-0) 

There is no check that `reward_address.is_non_zero()`. The address is then written directly to storage:

```cairo
staker_info.reward_address = reward_address;
self.write_staker_info(:staker_address, :staker_info);
``` [2](#0-1) 

The same pattern exists in `Pool::change_reward_address` — the only guard is a token-address check, and zero is accepted silently:

```cairo
assert!(
    self.token_dispatcher.contract_address.read() != reward_address,
    "{}",
    GenericError::REWARD_ADDRESS_IS_TOKEN,
);
// ...
pool_member_info.reward_address = reward_address;
self.write_pool_member_info(:pool_member, :pool_member_info);
``` [3](#0-2) 

When `Pool::claim_rewards` is later called, the stored `reward_address` is read back and used as the transfer recipient with no further validation:

```cairo
let reward_address = pool_member_info.reward_address;
// ...
reward_token.checked_transfer(recipient: reward_address, amount: rewards.into());
``` [4](#0-3) 

If `reward_address` is `0x0`, the transfer targets the zero address. Depending on the ERC-20 implementation, this either silently burns the tokens or reverts — in either case the yield is permanently unrecoverable by the pool member.

The same transfer-to-stored-address pattern exists in the staking contract's reward-sending path (`send_rewards_to_staker`), which is invoked during `unstake_action`. [5](#0-4) 

---

### Impact Explanation

A staker or pool member who sets their reward address to `0x0` — whether by mistake or through a front-end bug — will have all future reward transfers directed to the zero address. The tokens are permanently unrecoverable. This matches the allowed High impact: **permanent freezing of unclaimed yield**.

---

### Likelihood Explanation

The entry path requires only that the staker or pool member call `change_reward_address(0)`. No privileged role is needed. The call is open to any registered staker (staking contract) or any registered pool member (pool contract). Accidental zero-address submission is a realistic user error (e.g., a front-end that passes an uninitialized variable, or a script that passes `0` as a default). Likelihood is **Low-to-Medium** given the realistic accident scenario.

---

### Recommendation

Add an explicit non-zero check at the top of both `change_reward_address` implementations, mirroring the fix applied to DebtRegistry's `modifyBeneficiary`:

```cairo
// In Staking::change_reward_address and Pool::change_reward_address
assert!(reward_address.is_non_zero(), "Reward address cannot be zero");
```

The same guard should be applied wherever a reward address is first written — including the initial `stake` registration path — to ensure the invariant holds from the moment of entry.

---

### Proof of Concept

1. A pool member is registered with a valid reward address.
2. The pool member calls `Pool::change_reward_address(reward_address: 0)`.
   - The token-address check passes (0 is not a token address).
   - No zero-address check exists; the call succeeds.
   - `pool_member_info.reward_address` is now `0x0` in storage.
3. Rewards accrue over subsequent epochs.
4. The pool member (or anyone) calls `Pool::claim_rewards(pool_member)`.
   - `reward_address` is read as `0x0`.
   - `checked_transfer(recipient: 0x0, amount: rewards)` is executed.
   - Rewards are sent to the zero address and permanently lost.
5. The pool member has no way to recover the transferred rewards. [6](#0-5)
