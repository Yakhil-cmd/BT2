Audit Report

## Title
Byzantine Peer Can Exhaust Bidi Stream Budget and Permanently Freeze Inbound Stream Acceptance — (`rs/p2p/quic_transport/src/request_handler.rs`)

## Summary

A Byzantine subnet peer with an established QUIC connection can open all 1,000 permitted bidirectional streams and hold each one open indefinitely by never sending a FIN frame. Because `read_request` calls `recv_stream.read_to_end(MAX_MESSAGE_SIZE_BYTES)` with no timeout, all 1,000 `inflight_requests` tasks block permanently. The QUIC layer stops advertising new stream credit, `accept_bi()` never returns a new stream, and the victim replica's inbound bidi P2P stream acceptance on that connection is permanently frozen with no automatic connection close or recovery triggered.

## Finding Description

**Stream limit constant and configuration**

`MAX_CONCURRENT_BIDI_STREAMS` is set to 1,000 and applied to the server-side transport config: [1](#0-0) [2](#0-1) 

**Unbounded blocking read**

Every accepted bidi stream spawns a task that calls `read_to_end` with no deadline: [3](#0-2) 

Quinn's `read_to_end` blocks until the peer sends a FIN or a connection-level error occurs. If the Byzantine peer opens a stream, writes 1 byte, and never calls `finish()`, this future never resolves. The `RecvStream` is never dropped, so the QUIC stream slot is never returned to the credit window.

**Event loop has no escape hatch for stream exhaustion**

The `start_stream_acceptor` loop selects on `inflight_requests.join_next()` and `conn_handle.conn().accept_bi()`. Once 1,000 tasks are blocked in `read_to_end`, the QUIC layer stops advertising stream credit; `accept_bi()` never yields a new stream. The loop remains alive (it can still process uni streams, datagrams, and metrics ticks) but is permanently stalled on new inbound bidi streams: [4](#0-3) 

The developers themselves acknowledge this risk in a comment but leave it unmitigated: [5](#0-4) 

**No connection-level recovery**

`IDLE_TIMEOUT` (5 s) only fires when the connection is completely idle. With `KEEP_ALIVE_INTERVAL` at 1 s, a Byzantine peer sending periodic keep-alive packets keeps the connection alive indefinitely: [6](#0-5) 

Because `start_stream_acceptor` never exits (the loop is still alive processing other stream types), the `active_connections` task in the connection manager does not complete, so no reconnection is attempted: [7](#0-6) 

## Impact Explanation

The victim replica permanently loses the ability to accept any new inbound bidi QUIC streams from the Byzantine peer on that connection. All push-based P2P artifact delivery from that peer is blocked. Because the connection is not closed, the connection manager does not attempt reconnection, so the blockage persists until a topology change or external event forces a connection teardown. This constitutes an application/platform-level subnet availability impact on the targeted replica's P2P layer, not based on raw volumetric DDoS — matching the **High ($2,000–$10,000)** impact class.

## Likelihood Explanation

The attacker must be an authenticated subnet peer (TLS-verified). A single Byzantine node below the consensus fault threshold can execute this with minimal resources: open 1,000 bidi streams, write 1 byte each, never send FIN, send QUIC keep-alives. No volumetric traffic is required. The attack is deterministic, locally reproducible, and self-sustaining once the stream budget is exhausted.

## Recommendation

Wrap `read_to_end` in a per-stream deadline:

```rust
tokio::time::timeout(READ_TIMEOUT, recv_stream.read_to_end(MAX_MESSAGE_SIZE_BYTES)).await
```

If the timeout fires, the task returns an error, the `RecvStream` is dropped, the stream slot is reclaimed, and the QUIC layer re-advertises credit. Alternatively, enforce an application-level cap on `inflight_requests.len()` and stop calling `accept_bi()` when the cap is reached, then close the connection if the cap is held for too long — as the developer comment at lines 52–56 already suggests as a better approach.

## Proof of Concept

1. Establish an authenticated QUIC connection to the victim's `start_stream_acceptor`.
2. Open 1,000 bidi streams from the Byzantine peer; on each stream write 1 byte but never call `finish()`.
3. Assert that `accept_bi()` on the victim side never returns a new stream (stream credit exhausted).
4. Assert that the connection is **not** closed after `IDLE_TIMEOUT * 2` (keep-alives prevent it).
5. Assert that a legitimate 1,001st stream open attempt from the Byzantine peer is blocked by the QUIC layer (`STREAMS_BLOCKED` frame sent, no `accept_bi` delivery on victim).

Expected result without fix: victim's inbound bidi stream acceptance is permanently frozen; connection stays open; `inflight_requests` holds 1,000 stalled tasks indefinitely; `start_stream_acceptor` never exits; connection manager never reconnects.
