### Title
Weight Aggregation Counts Duplicate Key Material Multiple Times, Allowing Sub-Threshold Key to Authorize Transactions — (`fvm/transactionVerifier.go`)

### Summary

The transaction verifier deduplicates signatures by `(address, keyIndex)` pair, not by underlying cryptographic key material. Since `addAccountKey` imposes no uniqueness constraint on public key bytes, an attacker can register the same public key at N distinct indices (each at weight=500), sign the transaction N times using the same private key but different key indices, and cause `aggregateWeights` to accumulate `weight × N` — reaching the 1000-threshold with a single sub-threshold key.

### Finding Description

**Step 1 — No duplicate-key-material guard in `addAccountKey`.**

`addAccountKey` in `fvm/environment/account_key_updater.go` assigns the next available index and calls `AppendAccountPublicKey` with no check that the public key bytes are unique across existing keys. [1](#0-0) 

**Step 2 — Deduplication in `newSignatureEntries` is keyed on `(address, keyIndex)`, not on key material.**

The `uniqueKey` struct used for deduplication contains only `address` and `index`. Two signatures referencing the same cryptographic key at different indices pass through without error. [2](#0-1) 

**Step 3 — `getAccountKeysAndAggregateWeights` blindly accumulates weight per key index.**

For every signature that survives deduplication, the function looks up the key by `(address, keyIndex)` and adds its weight to `aggregateWeights[address]`. There is no check that the same public key bytes have already contributed weight. [3](#0-2) 

**Step 4 — `hasSufficientKeyWeight` only checks the accumulated total.**

The threshold check is a simple integer comparison against the accumulated sum, so a total inflated by duplicate key material passes. [4](#0-3) 

### Impact Explanation

An account holder with a single key at weight=500 (below the 1000 threshold) can self-authorize arbitrary transactions by registering the same key twice and providing two signatures. This completely bypasses the multi-key authorization invariant: a single cryptographic identity accumulates threshold weight, allowing unauthorized transaction execution under any account that applies this technique.

### Likelihood Explanation

The attack requires only standard Cadence account key management (`Account.keys.add`), which is available to any account owner. No privileged access, staked node control, or external dependency is needed. It is reproducible on an unmodified Flow emulator.

### Recommendation

In `getAccountKeysAndAggregateWeights`, track the set of public key bytes (or a canonical fingerprint) already counted for each address. If the same key material appears at a second index, skip its weight contribution (or return an error). Alternatively, enforce uniqueness of public key bytes in `AppendAccountPublicKey` / `addAccountKey` at write time. [5](#0-4) [6](#0-5) 

### Proof of Concept

```
1. Create account A with key K (ECDSA-P256, SHA3-256, weight=500) → stored at index 0.
2. In a setup transaction, call Account.keys.add(K, weight=500) → stored at index 1.
3. Construct a transaction with A as both payer and authorizer.
4. Sign the envelope with (A, keyIndex=0, sig_K) and (A, keyIndex=1, sig_K).
   - Both signatures are cryptographically valid (same private key).
5. Submit to verifyTransaction:
   - newSignatureEntries: sees {A,0} and {A,1} — no duplicate error.
   - getAccountKeysAndAggregateWeights: envelopeWeights[A] = 500 + 500 = 1000.
   - hasSufficientKeyWeight: 1000 >= 1000 → passes.
   - verifySignatures: both signatures verify against key K → passes.
6. Transaction executes successfully despite A holding only one sub-threshold key.
``` [7](#0-6)

	publicKey *runtime.PublicKey,

	error,

	if err != nil {
		return nil, fmt.Errorf("adding account key failed: %w", err)

	}

	keyIndex, err := updater.accounts.GetAccountPublicKeyCount(address)
	if err != nil {
		return nil, fmt.Errorf("adding account key failed: %w", err)
	}

	accountPublicKey, err := NewAccountPublicKey(
		publicKey,
		hashAlgo,
		keyIndex,
		weight)
	if err != nil {
		return nil, fmt.Errorf("adding account key failed: %w", err)
	}

	err = updater.accounts.AppendAccountPublicKey(address, *accountPublicKey)
	if err != nil {
		return nil, fmt.Errorf("adding account key failed: %w", err)
	}

func (v *TransactionVerifier) verifyTransaction(
	tracer tracing.TracerSpan,
	proc *TransactionProcedure,
	txnState storage.TransactionPreparer,
	keyWeightThreshold int,
) error {
	span := tracer.StartChildSpan(trace.FVMVerifyTransaction)
	if span.Tracer != nil {
		span.SetAttributes(
			attribute.String("transaction.ID", proc.ID.String()),
		)
	}
	defer span.End()

	tx := proc.Transaction
	if tx.Payer == flow.EmptyAddress {
		return errors.NewInvalidAddressErrorf(tx.Payer, "payer address is invalid")
	}

	// return the signature entries (both payload and envelope) and empty weight maps
	// that will be used to aggregate weights.
	// the account keys are deduplicated during this call.
	signatures, payloadWeights, envelopeWeights, err := newSignatureEntries(tx)
	if err != nil {
		return err
	}

	accounts := environment.NewAccounts(txnState)

	if keyWeightThreshold < 0 {
		return nil
	}

	// at this point, account keys are guaranteed to be unique across all signatures
	err = v.getAccountKeysAndAggregateWeights(txnState, accounts, signatures, tx.ProposalKey)
	if err != nil {
		return errors.NewInvalidProposalSignatureError(tx.ProposalKey, err)
	}

	// all authorizers must have sufficient weights
	for _, addr := range tx.Authorizers {
		// Skip this authorizer if it is also the payer. In the case where an account is
		// both a PAYER as well as an AUTHORIZER or PROPOSER, that account is required
		// to sign only the envelope.
		if addr == tx.Payer {
			continue
		}
		// hasSufficientKeyWeight
		if !v.hasSufficientKeyWeight(payloadWeights, addr, keyWeightThreshold) {
			return errors.NewAccountAuthorizationErrorf(
				addr,
				"authorizer account does not have sufficient signatures (%d < %d)",
				payloadWeights[addr],
				keyWeightThreshold)
		}
	}

	// payer must have sufficient weights
	if !v.hasSufficientKeyWeight(envelopeWeights, tx.Payer, keyWeightThreshold) {
		// TODO change this to payer error (needed for fees)
		return errors.NewAccountAuthorizationErrorf(
			tx.Payer,
			"payer account does not have sufficient signatures (%d < %d)",
			envelopeWeights[tx.Payer],
			keyWeightThreshold)
	}

	// Verify all cryptographic signatures against account public keys (concurrently)
	// and fail if at least one signature is invalid.
	// (at this point, signatures have been deduplicated and weights have been checked,
	// we wouldn't verify the signatures if any of those checks failed)
	err = v.verifySignatures(signatures)
	if err != nil {
		return errors.NewInvalidProposalSignatureError(tx.ProposalKey, err)
	}

	return nil
}

func (v *TransactionVerifier) getAccountKeysAndAggregateWeights(
	_ storage.TransactionPreparer,
	accounts environment.Accounts,
	signatures []*signatureContinuation,
	proposalKey flow.ProposalKey,
) error {
	foundProposalSignature := false
	for _, signature := range signatures {
		accountKey, err := accounts.GetRuntimeAccountPublicKey(
			signature.Address,
			signature.KeyIndex)
		if err != nil {
			return signature.newError(err)
		}

		if accountKey.Revoked {
			return signature.newError(
				fmt.Errorf("account key has been revoked"))
		}

		signature.accountKey = accountKey
		// aggregateWeight
		signature.aggregateWeights[signature.Address] += accountKey.Weight

		if !foundProposalSignature && signature.matches(proposalKey) {
			foundProposalSignature = true
		}
	}

	if !foundProposalSignature {
		return fmt.Errorf(
			"either the payload or the envelope should provide proposal " +
				"signatures")
	}

	return nil
}
