### Title
Unprivileged Caller Can Permanently Freeze Staker Yield via Unguarded `disable_rewards` Flag in `update_rewards` — (File: `src/staking/staking.cairo`)

---

### Summary

The `update_rewards` function in `staking.cairo` accepts a caller-controlled `disable_rewards: bool` parameter with no access control. Any unprivileged address can invoke `update_rewards(victim_staker, disable_rewards: true)`, which advances `last_reward_block` to the current block **before** any rewards are distributed, then returns early. Because the per-block guard prevents a second call in the same block, the staker's yield for that block is permanently lost. Repeated every block, this completely freezes a staker's unclaimed yield.

---

### Finding Description

`update_rewards` in `StakingRewardsManagerImpl` (lines 1449–1507) follows this sequence:

1. **Guard check** — asserts `current_block_number > last_reward_block` [1](#0-0) 

2. **Advance the guard** — writes `last_reward_block = current_block_number` **before** any reward is calculated or transferred [2](#0-1) 

3. **Early return** — if `disable_rewards || self.is_pre_consensus()`, the function returns immediately with no reward distribution [3](#0-2) 

4. **Reward path** — only reached when `disable_rewards == false` and consensus rewards are active [4](#0-3) 

There is no role check, caller check, or any other guard on the `disable_rewards` parameter. `general_prerequisites()` only verifies the contract is unpaused and the caller is non-zero. [5](#0-4) 

**Attack path:**

1. Attacker monitors the chain for any block where `current_block_number > last_reward_block`.
2. Attacker calls `update_rewards(victim_staker_address, disable_rewards: true)`.
3. `last_reward_block` is set to `current_block_number`.
4. Function returns early — zero rewards distributed.
5. Any legitimate call to `update_rewards` in the same block now reverts with `REWARDS_ALREADY_UPDATED`.
6. The staker's yield for that block is permanently lost.

The attacker can repeat this every block, completely preventing any staker from ever accumulating rewards.

---

### Impact Explanation

This directly matches the **"Permanent freezing of unclaimed yield"** impact category. Rewards that should have been credited to `staker_info.unclaimed_rewards_own` and forwarded to pool contracts are never calculated or transferred. Because `last_reward_block` is already advanced, no recovery path exists for the skipped block — the yield is gone forever.

The analog to the Alchemix bug is exact in structure: in Alchemix, `_distribute()` consumes the distribution slot (resets `claimable[gauge]`) before `minter.updatePeriod()` sends the rewards, so the rewards arrive but are stuck. Here, `update_rewards` consumes the block slot (`last_reward_block`) before rewards are calculated, so the rewards are never generated and the slot cannot be reused.

---

### Likelihood Explanation

- **No privilege required** — any EOA or contract can call `update_rewards`.
- **No capital required** — only gas cost.
- **Fully deterministic** — the attacker simply needs to call the function before the legitimate caller in any given block.
- **Scalable** — a single attacker can grief every staker every block by iterating over staker addresses.

---

### Recommendation

Remove `disable_rewards` from the public interface entirely, or restrict it to a trusted caller (e.g., only the staker themselves or the attestation contract). The simplest fix is to delete the parameter and always distribute rewards when the function is called in consensus mode:

```cairo
// Before:
fn update_rewards(ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool)

// After:
fn update_rewards(ref self: ContractState, staker_address: ContractAddress)
```

If `disable_rewards` is needed for internal protocol logic, gate it with a role check (e.g., `only_security_agent`) or restrict the caller to `staker_address` itself.

---

### Proof of Concept

```
// Pseudocode — any unprivileged address `attacker` can execute this each block:

// Block N:
// last_reward_block = N-1 (staker has not yet claimed this block)

attacker.call(
    staking_contract.update_rewards(
        staker_address = alice,   // any valid active staker
        disable_rewards = true
    )
)
// Result:
//   last_reward_block = N  ← slot consumed
//   alice.unclaimed_rewards_own unchanged  ← no rewards credited
//   pool rewards = 0  ← no transfer to pool

// Any subsequent call in block N:
alice.call(staking_contract.update_rewards(alice, false))
// → PANICS: REWARDS_ALREADY_UPDATED

// Alice permanently loses block N's yield.
// Attacker repeats every block → Alice never accumulates any rewards.
```
