### Title
FROST `PresignOutput` Derives `Clone`, Enabling Nonce Reuse and Private Key Extraction — (File: src/frost/mod.rs)

---

### Summary

The `PresignOutput<C>` struct in `src/frost/mod.rs` derives `Clone` and `Serialize`/`Deserialize`, making it trivially possible for callers to reuse the same FROST signing nonces across multiple signing sessions. FROST nonce reuse is a well-documented catastrophic vulnerability that allows algebraic extraction of private key shares from any two signature shares produced with the same nonces.

---

### Finding Description

`PresignOutput<C>` is defined at lines 36–41 of `src/frost/mod.rs` as:

```rust
#[derive(Debug, Clone, Serialize, Deserialize, Eq, PartialEq)]
pub struct PresignOutput<C: Ciphersuite + Send + 'static> {
    pub nonces: SigningNonces<C>,
    pub commitments_map: BTreeMap<Identifier<C>, SigningCommitments<C>>,
}
``` [1](#0-0) 

The struct contains `nonces: SigningNonces<C>`, which are the secret per-session hiding and binding nonces used in FROST signing. The FROST protocol's security is entirely predicated on each nonce pair being used **exactly once**. Deriving `Clone` and `Serialize`/`Deserialize` on this struct removes any type-level enforcement of that invariant, allowing callers to:

1. **Clone** the `PresignOutput` before passing it to the signing function, retaining a live copy for reuse.
2. **Serialize** the `PresignOutput` to persistent storage and **deserialize** it later for a second signing session.

The `presign` function returns the `PresignOutput` to the caller with no mechanism to mark it as consumed: [2](#0-1) 

The library documentation warns about presignature reuse but does not enforce it at the type level. The analogous warning exists in the robust ECDSA presign module as well:

> "it's crucial that a presignature is never reused" [3](#0-2) 

**Attack path (malicious coordinator):**

1. Coordinator runs the presign protocol → each participant obtains a `PresignOutput` containing their secret nonces `(d_i, e_i)`.
2. Coordinator asks participants to sign message `M1`. Each participant calls the signing function, consuming their `PresignOutput` by value — but because `Clone` is derived, they may have retained a copy (e.g., for retry logic, or the coordinator serialized and re-delivered it).
3. Coordinator asks participants to sign a different message `M2` using the **same** presign output (the clone or deserialized copy).
4. The coordinator now holds two sets of signature shares `(s_i1, s_i2)` produced with identical nonces `(d_i, e_i)` for different messages. Using standard FROST nonce-reuse equations, the coordinator solves for each participant's secret key share `x_i`.

---

### Impact Explanation

**Critical.** FROST nonce reuse directly enables extraction of private signing key shares. Given two signature shares from the same participant using the same nonces for different messages, an attacker can algebraically recover that participant's secret key share. Collecting threshold-many such shares allows full private key reconstruction. This matches the allowed impact: *"Extraction, reconstruction, or disclosure of private signing shares, aggregate secret material, presign secrets, nonce material, or confidential derived secrets."*

---

### Likelihood Explanation

**Medium.** The `Clone` and `Serialize`/`Deserialize` derives on `PresignOutput` make nonce reuse trivially possible for any caller. A malicious coordinator can orchestrate reuse by running parallel or sequential signing sessions against the same presign output. Legitimate callers implementing retry logic (e.g., on network failure) may also inadvertently reuse presign outputs, since the library provides no guard against it. The attack requires the coordinator role, which is a documented trust boundary in the protocol.

---

### Recommendation

1. **Remove `Clone` from `PresignOutput<C>`** to enforce single-use at the type level. Since the signing function already takes `PresignOutput` by value (moving it), removing `Clone` would make it impossible in safe Rust to pass the same nonces to two signing sessions.
2. **Remove or restrict `Serialize`/`Deserialize`** on `PresignOutput`, or document clearly that deserialized outputs must never be used more than once. Consider a newtype wrapper that zeroizes nonces on drop.
3. Ensure the same fix is applied to the OT-based ECDSA `PresignOutput` if it similarly derives `Clone`.

---

### Proof of Concept

```rust
// Participant holds their PresignOutput after the presign protocol
let presign_output: PresignOutput<C> = run_presign_protocol(...);

// Clone retained before first use (e.g., for retry logic)
let presign_clone = presign_output.clone(); // <-- enabled by Clone derive

// First signing session: M1
let _sig_share_1 = sign_v2(presign_output, signing_share, M1, ...);

// Second signing session: M2 — SAME NONCES reused
let _sig_share_2 = sign_v2(presign_clone, signing_share, M2, ...);

// Malicious coordinator collects sig_share_1 and sig_share_2.
// Since nonces (d_i, e_i) are identical in both, the coordinator
// solves the FROST nonce-reuse equations to extract x_i (private key share).
```
