### Title
Unauthorized Transfer of Native Bank Tokens via Missing Caller Authorization in `BankContract.transfer()` — (`x/cronos/keeper/precompiles/bank.go`)

---

### Summary

The `transfer` method of the `BankContract` precompile accepts an arbitrary `sender` address from call arguments and passes it directly to `bankKeeper.SendCoins` without verifying that `contract.Caller()` is authorized to spend on behalf of that `sender`. Any EVM contract or EOA can invoke this precompile to drain native bank tokens (denom `evm/<callerAddress>`) from any holder without the holder's consent.

---

### Finding Description

In `x/cronos/keeper/precompiles/bank.go`, the `Run` function handles the `transfer` method as follows:

```go
case TransferMethodName:
    sender    := args[0].(common.Address)   // ← arbitrary, from call input
    recipient := args[1].(common.Address)
    amount    := args[2].(*big.Int)
    from := sdk.AccAddress(sender.Bytes())
    to   := sdk.AccAddress(recipient.Bytes())
    denom := EVMDenom(contract.Caller())    // "evm/0x<callerAddress>"
    amt  := sdk.NewCoin(denom, ...)
    ...
    bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt))
``` [1](#0-0) 

`sender` is taken verbatim from the ABI-decoded call arguments (`args[0]`), not from `contract.Caller()`. There is no allowance check, no ownership check, and no other guard verifying that the calling contract is permitted to move tokens on behalf of `sender`. `bankKeeper.SendCoins` is then called with that arbitrary `from` address.

The token denom is `EVMDenom(contract.Caller())` — i.e., `"evm/0x<callerAddress>"` — so the tokens moved are the native bank-module tokens whose denom is namespaced to the calling contract. [2](#0-1) 

The `mint` method (same switch block) also accepts an arbitrary `recipient` and mints `evm/<callerAddress>` tokens to it with no access control, meaning any caller can first mint tokens into a victim's account and then immediately drain them via `transfer`. [3](#0-2) 

---

### Impact Explanation

**High — Unauthorized transfer of native bank / CRC20-denominated assets.**

Any EVM contract (or EOA sending a transaction directly to `bankContractAddress = 0x...64`) can:

1. Call `mint(victimAddress, amount)` → victim receives `evm/0x<caller>` tokens.
2. Call `transfer(victimAddress, attackerAddress, amount)` → those tokens are moved from victim to attacker without victim's consent.

If `evm/<callerAddress>` tokens carry real economic value — e.g., they are issued by a DeFi protocol as LP receipts, yield tokens, or collateral claims — any holder of those tokens can be drained at will by the issuing contract (or any contract that shares the same caller address). The `bankKeeper.SendCoins` call succeeds as long as the victim's balance is sufficient; no approval or signature from the victim is required.

This matches the **"Unauthorized transfer of CRC20/ERC20 assets"** criterion in the allowed High impact scope.

---

### Likelihood Explanation

The bank precompile is a live, publicly callable precompile at a fixed address. Any EVM transaction can reach it. No privileged role, leaked key, or governance action is required. The only precondition is that the victim holds native bank tokens with denom `evm/<callerAddress>`, which is a realistic condition for any protocol that uses the bank precompile to issue tokens to users.

---

### Recommendation

Replace the caller-supplied `sender` with `contract.Caller()` in the `transfer` case, mirroring how `mint`/`burn` already derive the denom from `contract.Caller()`:

```go
// Before (vulnerable):
sender := args[0].(common.Address)
from   := sdk.AccAddress(sender.Bytes())

// After (fixed):
from := sdk.AccAddress(contract.Caller().Bytes())
// recipient and amount remain from args
```

If the design intent is to allow a contract to move tokens on behalf of a third-party address, an explicit on-chain allowance mapping (analogous to ERC-20 `approve`/`allowance`) must be introduced and checked before calling `SendCoins`.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

interface IBankPrecompile {
    function mint(address recipient, uint256 amount) external returns (bool);
    function transfer(address sender, address recipient, uint256 amount) external returns (bool);
}

contract BankDrainer {
    IBankPrecompile constant bank = IBankPrecompile(address(100)); // bankContractAddress

    // Step 1: seed victim with evm/0x<this> tokens
    function seedVictim(address victim, uint256 amount) external {
        bank.mint(victim, amount);
    }

    // Step 2: drain victim's evm/0x<this> tokens — no approval needed
    function drain(address victim, address attacker, uint256 amount) external {
        bank.transfer(victim, attacker, amount);
        // bankKeeper.SendCoins(victim → attacker, "evm/0x<BankDrainer>", amount)
        // succeeds with no authorization from victim
    }
}
``` [4](#0-3) 

The `transfer` call at line 192 uses `from = sdk.AccAddress(sender.Bytes())` where `sender` is attacker-controlled, bypassing any ownership or allowance check on the victim's balance. [5](#0-4)
