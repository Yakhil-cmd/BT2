### Title
Missing Caller Authorization on `update_rewards` Allows Any User to Suppress Consensus Block Rewards - (File: src/staking/staking.cairo)

### Summary
The `update_rewards` function in `StakingRewardsManagerImpl` is specified to be callable only by the Starkware sequencer, but no caller check is implemented. Any unprivileged address can invoke it with `disable_rewards: true`, consuming the per-block reward slot and permanently preventing consensus rewards from being distributed to stakers for that block.

### Finding Description
The protocol specification at `docs/spec.md` line 1645 explicitly states:

> **access control**: Only starkware sequencer.

However, the implementation of `update_rewards` in `src/staking/staking.cairo` contains no such check:

```rust
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
    let current_block_number = starknet::get_block_number();
    assert!(
        current_block_number > self.last_reward_block.read(),
        "{}",
        Error::REWARDS_ALREADY_UPDATED,
    );
    // ...
    self.last_reward_block.write(current_block_number);

    if disable_rewards || self.is_pre_consensus() {
        return;
    }
    // distribute rewards...
``` [1](#0-0) 

The function writes `current_block_number` to `last_reward_block` unconditionally before the `disable_rewards` branch. This single global storage slot gates all reward distribution for the block. Any caller who sets `disable_rewards: true` consumes the slot without distributing rewards, and the `REWARDS_ALREADY_UPDATED` assertion then blocks any subsequent call in the same block. [2](#0-1) 

The spec's access control requirement is documented but never enforced in code: [3](#0-2) 

### Impact Explanation
**High — Permanent freezing of unclaimed yield.**

An attacker who calls `update_rewards(any_staker, disable_rewards: true)` every block:
1. Sets `last_reward_block` to the current block without distributing any rewards.
2. Causes every subsequent legitimate call in that block to revert with `REWARDS_ALREADY_UPDATED`.
3. Stakers accumulate zero consensus rewards for every block the attacker griefs.

Because `last_reward_block` is a single global value shared across all stakers, a single call per block is sufficient to suppress rewards for the entire protocol. Sustained over time, this permanently freezes all unclaimed consensus-phase yield. [4](#0-3) 

### Likelihood Explanation
**Medium.** The attack requires one transaction per block with no financial return to the attacker. It is economically irrational under normal conditions but is trivially executable by any motivated adversary (e.g., a competing validator, a protocol saboteur). The entry path is fully permissionless — no privileged role, no leaked key, no bridge compromise is required. The only cost is gas per block.

### Recommendation
Add an explicit caller check matching the specification. The intended caller is the Starkware sequencer, which in Starknet protocol transactions appears as a designated non-zero address. The check should mirror the pattern used elsewhere in the codebase:

```rust
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
    assert!(
        get_caller_address() == self.sequencer_address.read(),
        "{}",
        Error::CALLER_IS_NOT_SEQUENCER,
    );
    // ...
```

Alternatively, if the function is intentionally open to any caller, the `disable_rewards` parameter must be removed and rewards must always be distributed when consensus is active, so no caller can suppress them.

### Proof of Concept
1. Consensus rewards are active (`consensus_rewards_first_epoch` has passed).
2. Attacker (any EOA) calls `update_rewards(victim_staker, disable_rewards: true)` at the start of every block.
3. `last_reward_block` is set to the current block number; no rewards are distributed.
4. The legitimate sequencer's call in the same block reverts with `REWARDS_ALREADY_UPDATED`.
5. After N blocks, `victim_staker.unclaimed_rewards_own` remains zero despite being an active staker with non-zero balance.
6. The attack is confirmed by the test infrastructure: `test_update_rewards_only_staker` demonstrates the function succeeds when called from an arbitrary non-privileged address with no caller restriction. [5](#0-4) [6](#0-5)
