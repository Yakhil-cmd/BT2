### Title
Unrestricted `update_rewards` with `disable_rewards=true` Permanently Blocks Consensus Reward Distribution - (File: `src/staking/staking.cairo`)

---

### Summary

The `update_rewards` function is publicly callable by any address. It unconditionally writes the global `last_reward_block` state variable **before** checking the `disable_rewards` flag. An unprivileged attacker can call `update_rewards(any_active_staker, true)` on every block to permanently prevent all stakers from receiving consensus rewards, because the one-call-per-block guard is consumed without distributing any rewards.

---

### Finding Description

In `src/staking/staking.cairo`, the `update_rewards` function has no role-based access control. `general_prerequisites` only checks that the contract is unpaused and the caller is non-zero: [1](#0-0) 

The function enforces a global one-call-per-block invariant via the `last_reward_block` storage variable: [2](#0-1) 

Critically, `last_reward_block` is written **before** the `disable_rewards` guard is evaluated: [3](#0-2) 

Because `last_reward_block` is a single global variable shared across all stakers, any caller who invokes `update_rewards(any_valid_staker, true)` for block N permanently exhausts the reward slot for that block. No subsequent call — including a legitimate one with `disable_rewards=false` — can succeed for block N, since it will revert with `REWARDS_ALREADY_UPDATED`.

This is the direct analog of the sUSDz accounting bug: a state variable (`last_reward_block`) is updated to reflect that a block has been "processed," but the corresponding value (block rewards) is never actually distributed. The stored state diverges from the true accounting state, and the divergence is irrecoverable.

The `update_rewards` entry point is declared in `IStakingRewardsManager` with no caller restriction: [4](#0-3) 

The `last_reward_block` storage field is defined as a single global slot: [5](#0-4) 

---

### Impact Explanation

An attacker calls `update_rewards(any_active_staker, disable_rewards=true)` on every block in the post-consensus phase. Each call marks the block as processed but distributes zero rewards. All stakers permanently lose their consensus block rewards. The yield is never accumulated and cannot be recovered. This constitutes **permanent freezing of unclaimed yield** for all stakers and the protocol (High impact).

---

### Likelihood Explanation

The attack is trivially executable by any unprivileged address. The attacker only needs any active staker's address, which is publicly observable on-chain via `NewStaker` events. The attack requires one transaction per block. Given that consensus rewards accumulate continuously, the economic incentive to block them (e.g., by a competing validator or a malicious actor) is realistic.

---

### Recommendation

Either:
1. Restrict `update_rewards` to a trusted caller (e.g., a designated role or the block producer), or
2. Move `self.last_reward_block.write(current_block_number)` to **after** the `disable_rewards` check, so a call with `disable_rewards=true` does not consume the block's reward slot:

```cairo
if disable_rewards || self.is_pre_consensus() {
    return;
}
// Only consume the block slot when rewards are actually distributed.
self.last_reward_block.write(current_block_number);
```
