### Title
Unbounded Loop in `calculate_rewards` Allows Permanent Freezing of Pool Member's Unclaimed Yield - (File: src/pool/pool.cairo)

### Summary

The `calculate_rewards` function in `src/pool/pool.cairo` contains an explicitly acknowledged unbounded loop that iterates over a pool member's `pool_member_epoch_balance` trace. A pool member can grow this trace to an arbitrary size by calling `add_to_delegation_pool` across many different epochs without ever calling `claim_rewards`. Once the trace is large enough, the `claim_rewards` call will exceed the block gas limit and permanently revert, freezing the pool member's unclaimed yield forever.

### Finding Description

The `calculate_rewards` internal function, called by the public `claim_rewards` entry point, iterates over every unprocessed entry in the pool member's `pool_member_epoch_balance` trace:

```cairo
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
    if pool_member_checkpoint.epoch() >= until_epoch {
        break;
    }
    let to_sigma = self.find_sigma(pool_member_checkpoint, curr_epoch: until_epoch);
    // ...
    entry_to_claim_from += 1;
}
```

The developers themselves flag this as unbounded. [1](#0-0) 

The trace grows via `set_member_balance`, which is called by `add_to_delegation_pool`, `enter_delegation_pool`, `exit_delegation_pool_intent`, and `enter_delegation_pool_from_staking_contract`. Each call inserts a new checkpoint keyed by `current_epoch + K`. [2](#0-1) 

Crucially, the `PoolMemberBalanceTrace` only deduplicates insertions with the **same key** (same epoch). Insertions in different epochs each append a new entry, growing the trace length by one per epoch. [3](#0-2) 

The `entry_to_claim_from` cursor stored in `pool_member_info` is only advanced when `claim_rewards` succeeds. If the pool member never calls `claim_rewards`, the cursor stays at 0 and every future call must iterate the entire accumulated trace. [4](#0-3) 

Each loop iteration calls `find_sigma`, which performs multiple storage reads against `cumulative_rewards_trace`. There is no mechanism to process only a partial range of the trace — the entire unprocessed window is consumed atomically in a single transaction. [5](#0-4) 

### Impact Explanation

Once the trace is large enough that a single `claim_rewards` call exceeds the Starknet block gas limit, the pool member's unclaimed yield is **permanently frozen**. There is no partial-claim mechanism, no way to shrink the trace, and no alternative path to recover the rewards. The `exit_delegation_pool_action` function does not call `claim_rewards` and only returns the principal, so even a full exit leaves the accumulated rewards inaccessible. [6](#0-5) 

This matches the allowed impact: **Permanent freezing of unclaimed yield**.

### Likelihood Explanation

The attack path is reachable by any unprivileged pool member (delegator). The minimum amount for `add_to_delegation_pool` is 1 token unit (only `amount.is_non_zero()` is checked), so the cost per epoch is negligible. [7](#0-6) 

A pool member who delegates and then repeatedly tops up their position across many epochs — a realistic long-term staking pattern — will naturally accumulate trace entries. If they do not claim rewards frequently, the trace grows without bound. The scenario is realistic for passive delegators who stake and forget.

Additionally, `add_to_delegation_pool` can be called by the pool member's **reward address** as well as the pool member themselves, meaning a reward address set to a contract could automate trace growth. [8](#0-7) 

### Recommendation

1. **Introduce a `max_entries` parameter** to `claim_rewards` so the loop processes at most N entries per call, allowing incremental reward collection across multiple transactions.
2. **Alternatively**, restructure the reward accounting so that `claim_rewards` does not need to replay the full balance history — for example, by eagerly computing and storing accrued rewards at each balance-change checkpoint, so `claim_rewards` only needs to read the latest stored value.
3. **At minimum**, document a hard cap on how many balance-change entries a pool member should accumulate before claiming, and enforce it by reverting `add_to_delegation_pool` if the trace length exceeds a safe threshold.

### Proof of Concept

1. Pool member calls `enter_delegation_pool` with a minimum amount.
2. For N epochs (e.g., N = 10,000), the pool member calls `add_to_delegation_pool` with 1 token unit per epoch, without ever calling `claim_rewards`. Each call inserts a new entry into `pool_member_epoch_balance` via `set_member_balance`.
3. After N epochs, the pool member calls `claim_rewards`. The `calculate_rewards` loop must iterate N times, each time calling `find_sigma` (multiple storage reads). The transaction runs out of gas and reverts.
4. Because `entry_to_claim_from` was never updated (the transaction always reverts), the pool member can never claim their accumulated rewards. The yield is permanently frozen. [9](#0-8) [10](#0-9)

```text
        fn calculate_rewards(
            self: @ContractState,
            pool_member: ContractAddress,
            from_checkpoint: PoolMemberCheckpoint,
            until_checkpoint: PoolMemberCheckpoint,
            mut entry_to_claim_from: VecIndex,
        ) -> (Amount, VecIndex) {
            let pool_member_trace = self.pool_member_epoch_balance.entry(pool_member);
            // Note: `until_epoch` is the current epoch.
            let until_epoch = until_checkpoint.epoch();

            let mut rewards = 0;

            let pool_member_trace_length = pool_member_trace.length();

            let mut from_sigma = self.find_sigma(from_checkpoint, curr_epoch: until_epoch);
            let mut from_balance = from_checkpoint.balance();

            let base_value = self.staking_rewards_base_value.read();

            // **Note**: The loop iterates over the balance changes in the pool member's balance
            // trace. This loop is unbounded but unlikely to exceed gas limits.
            while entry_to_claim_from < pool_member_trace_length {
                let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
                // If the balance change is after `until_epoch` (and therefore does not affect
                // the current reward computation), exit the loop.
                if pool_member_checkpoint.epoch() >= until_epoch {
                    break;
                }

                // Compute rewards from (inclusive) the previous balance change (or from
                // `from_checkpoint`) to (exclusive) the current entry.
                let to_sigma = self.find_sigma(pool_member_checkpoint, curr_epoch: until_epoch);
                rewards +=
                    compute_rewards_rounded_down(
                        amount: from_balance, interest: to_sigma - from_sigma, :base_value,
                    );
                from_sigma = to_sigma;
                from_balance = pool_member_checkpoint.balance();
                entry_to_claim_from += 1;
            }

            // Compute the remaining rewards from (inclusive) the last visited balance change in
            // `pool_member_trace` (or from `from_checkpoint`) to (exclusive) `until_checkpoint`.
            let to_sigma = self.find_sigma(until_checkpoint, curr_epoch: until_epoch);
            rewards +=
                compute_rewards_rounded_down(
                    amount: from_balance, interest: to_sigma - from_sigma, :base_value,
                );

            (rewards, entry_to_claim_from)
        }
```
