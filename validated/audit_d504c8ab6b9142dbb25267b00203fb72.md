Audit Report

## Title
Permissionless Third-Party Participation Registration Bypasses Confirmation Text in SNS Swap `refresh_buyer_tokens` - (File: rs/sns/swap/canister/canister.rs)

## Summary
The `refresh_buyer_tokens` endpoint in the SNS swap canister accepts an arbitrary `buyer` principal from any unprivileged caller with no enforcement that the caller matches the buyer. Because the `confirmation_text` is a publicly readable initialization parameter, any third party can register a victim's ICP participation — including supplying the confirmation text on the victim's behalf — without the victim ever having consented. This renders the confirmation text mechanism ineffective and enables forced participation of a victim's ICP funds.

## Finding Description
In `rs/sns/swap/canister/canister.rs` at lines 130–134, the buyer identity is resolved as:

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    PrincipalId::from_str(&arg.buyer).unwrap()
};
```

There is no check that `caller_principal_id() == p` when `arg.buyer` is non-empty. Any ingress sender can supply an arbitrary buyer principal.

The `validate_confirmation_text` function in `rs/sns/swap/src/swap.rs` at lines 363–384 performs only a string equality check against the SNS-configured text stored in `Init`:

```rust
(Some(expected_text), Some(text)) => {
    if &text != expected_text {
        Err("The value of `confirmation_text` does not match...")
    } else {
        Ok(())
    }
}
```

It does not verify that the confirmation originated from the buyer. The `confirmation_text` is stored in the `Init` struct (proto field 15, `rs/sns/swap/proto/ic_sns_swap/pb/v1/swap.proto` line 325) and is publicly readable via `get_sale_parameters`. An attacker reads the text, supplies it verbatim, and the check passes.

The full exploit path in `refresh_buyer_token_e8s` (`rs/sns/swap/src/swap.rs` lines 1134–1291):
1. Lifecycle and ICP target are validated (no caller check).
2. `validate_confirmation_text` passes because the attacker supplies the correct public string.
3. The ledger balance for the victim's subaccount is read — the victim's ICP is already there.
4. The victim's participation is registered in `self.buyers` and their ICP is committed to the swap.

No existing guard prevents this. The proto definition (`rs/sns/swap/proto/ic_sns_swap/pb/v1/swap.proto` lines 843–851) explicitly documents that the `buyer` field defaults to the caller only when empty, with no restriction on third-party use.

## Impact Explanation
This is a **High** severity finding matching the allowed impact: *"Significant SNS security impact with concrete user or protocol harm."*

Two concrete harms:

1. **Confirmation text mechanism rendered ineffective.** The field exists to ensure participants explicitly acknowledge legal or regulatory terms before their ICP is committed. Because the text is public and any caller can supply it on behalf of any buyer, the mechanism provides zero protection. A victim's ICP can be locked into a swap without the victim ever having acknowledged the terms.

2. **Forced participation / unauthorized ICP locking.** If a user transferred ICP to the swap subaccount but changed their mind before calling `refresh_buyer_tokens`, a third party can force their participation. The victim's ICP is then locked until the swap concludes (committed or aborted), at which point recovery requires calling `error_refund_icp`. This is a concrete, time-bounded loss of fund control.

## Likelihood Explanation
**Medium.** The ICP ledger is public; any observer can detect when a principal transfers ICP to a swap subaccount. The confirmation text is readable via `get_sale_parameters`. The attacker requires no privileged access, no key material, and no threshold corruption — only the ability to submit an ingress message. On the IC there is no gas auction, so a low-latency attacker can reliably race a victim's message. The attack is repeatable for any victim who has transferred ICP but not yet called `refresh_buyer_tokens`.

## Recommendation
1. **Enforce caller == buyer**: In `refresh_buyer_tokens` (`rs/sns/swap/canister/canister.rs`), reject calls where `arg.buyer` is non-empty and does not match `caller_principal_id()`. If intentional third-party registration is needed (e.g., by NNS governance), scope it to an explicit allowlist.
2. **Redesign confirmation text**: If the confirmation text has legal significance, it must be cryptographically tied to the buyer's identity. The IC ingress message already carries the caller's identity via the canister's `caller()` API; the confirmation should only be accepted when `caller == buyer`.

## Proof of Concept
1. Alice transfers 10 ICP to her swap subaccount (`principal_to_subaccount(alice)` on the swap canister).
2. Attacker queries `get_sale_parameters` and reads `confirmation_text = "I agree to the terms"`.
3. Attacker submits an ingress call to `refresh_buyer_tokens` with `buyer = alice_principal_text` and `confirmation_text = Some("I agree to the terms")`.
4. The swap canister executes `refresh_buyer_token_e8s(alice, Some("I agree to the terms"), ...)`:
   - `validate_lifecycle_is_open()` passes.
   - `validate_confirmation_text(Some("I agree to the terms"))` passes (string matches).
   - Ledger balance for Alice's subaccount returns 10 ICP.
   - Alice's entry is created in `self.buyers` with `amount_icp_e8s = 10 * E8`.
5. Alice's 10 ICP is now committed to the swap without Alice having called anything or confirmed any terms.
6. A deterministic integration test or PocketIC test can reproduce this by: initializing a swap with `confirmation_text`, funding Alice's subaccount on a mock ledger, calling `refresh_buyer_token_e8s` with a different `PrincipalId` as the caller but Alice's principal as `buyer`, and asserting that `swap.buyers` contains Alice's entry.
