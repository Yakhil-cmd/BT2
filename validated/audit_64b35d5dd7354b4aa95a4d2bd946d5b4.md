### Title
FROST `PresignOutput` Derives `Clone` Enabling Nonce Reuse and Private Key Share Extraction — (`src/frost/mod.rs`)

### Summary
`PresignOutput<C>` in `src/frost/mod.rs` derives `Clone` and `Serialize/Deserialize`, allowing its secret `SigningNonces<C>` field to be copied and reused across multiple signing sessions with no library-level single-use enforcement. This is the direct analog of the original nonce-tracking state deletion bug: just as `offboard()` deleting `packedPayoutNonces[]` allowed replay by resetting nonce state, `Clone` on `PresignOutput` allows nonce state to be "reset" by copying. A malicious coordinator can orchestrate two signing sessions with the same nonce material and extract participants' private key shares.

### Finding Description

In `src/frost/mod.rs` at lines 36–41, `PresignOutput<C>` is defined as:

```rust
#[derive(Debug, Clone, Serialize, Deserialize, Eq, PartialEq)]
pub struct PresignOutput<C: Ciphersuite + Send + 'static> {
    pub nonces: SigningNonces<C>,
    pub commitments_map: BTreeMap<Identifier<C>, SigningCommitments<C>>,
}
``` [1](#0-0) 

The `nonces: SigningNonces<C>` field contains the secret FROST nonce pair `(hiding_nonce, binding_nonce)` that must never be reused. FROST's security proof requires each nonce pair be used for exactly one signing session; reuse with a different message allows the private key share to be recovered algebraically.

The `Clone` derivation means any caller can clone a `PresignOutput` before passing it to `sign_v2`, preserving the nonce material for reuse. The `Serialize/Deserialize` derivation means nonce material can be persisted to disk and reloaded, enabling reuse across process restarts. The `nonces` field is `pub`, giving callers direct access to the raw nonce scalars.

**Attack path via a malicious coordinator:**

1. All participants run `presign()` producing a `PresignOutput` per participant containing their secret nonces and the shared `commitments_map`. [2](#0-1) 

2. The coordinator initiates signing session 1 for message `m1` via `sign_v2`. Each participant passes their `PresignOutput` (or a clone of it) and signs with their nonces. [3](#0-2) 

3. The coordinator claims session 1 failed (e.g., by not responding or returning an error) and initiates session 2 for message `m2`.

4. Participants, having cloned their `PresignOutput` (which the library permits via `Clone`), reuse the same nonces for session 2.

5. In `do_sign_participant_v2`, each participant computes `round2::sign(&signing_package, &presignature.nonces, &key_package)` with the same `presignature.nonces` for both sessions. [4](#0-3) 

6. The coordinator now holds two signature shares `(s1, s2)` from the same participant with the same nonces but different messages. From the FROST signing equation `s = nonce + key_share * challenge`, the coordinator solves: `key_share = (s1 - s2) / (challenge(m1, commitments) - challenge(m2, commitments))`.

The same structural issue exists in the ECDSA presignature structs, which also derive `Clone` over secret nonce material: [5](#0-4) [6](#0-5) 

Note also that `PresignOutput<C>` in `src/frost/mod.rs` lacks `ZeroizeOnDrop` entirely, unlike the ECDSA presignature structs which include it — meaning nonce material is not even cleared from memory after use.

### Impact Explanation
**Critical.** A malicious coordinator who orchestrates two signing sessions with the same presignature nonces can extract each participant's private key share from the two resulting signature shares. With ≥ threshold key shares, the aggregate private key can be reconstructed. This directly enables unauthorized creation of valid threshold signatures for attacker-chosen inputs and full disclosure of private signing shares — matching the allowed Critical impact scope.

### Likelihood Explanation
**Medium.** The attack requires a malicious coordinator who can orchestrate multiple signing sessions. The coordinator is a designated role in the protocol and is explicitly in scope as a potential adversary. The `Clone` derivation makes accidental nonce reuse likely in production code, especially when handling session failures or retries — a common operational scenario. The `Serialize/Deserialize` derivation further enables reuse across process restarts without any additional attacker capability.

### Recommendation
1. Remove `Clone` from `PresignOutput<C>` in `src/frost/mod.rs`. The struct should be consumed (moved) when passed to `sign_v2`, and the library should not provide a mechanism to copy it.
2. Add `ZeroizeOnDrop` to `PresignOutput<C>` to ensure nonce material is cleared from memory after use (currently missing, unlike the ECDSA presignature structs).
3. Make the `nonces` field private and provide no public accessor, so callers cannot extract the raw nonce scalars.
4. Apply the same `Clone` removal to `PresignOutput` in `src/ecdsa/ot_based_ecdsa/mod.rs` and `src/ecdsa/robust_ecdsa/mod.rs`.

### Proof of Concept
```rust
// Participant P1 runs presign and clones the output — library permits this
let presign_output: PresignOutput<_> = run_presign(...);
let presign_clone = presign_output.clone(); // Clone is derived; nonces are copied

// Session 1: sign message m1 — presign_output is moved (consumed)
let s1 = sign_v2(participants, threshold, me, coordinator,
                 keygen_output.clone(), presign_output, m1.to_vec());

// Coordinator claims session 1 failed; participant reuses cloned presignature
// Session 2: sign message m2 — same nonces used again
let s2 = sign_v2(participants, threshold, me, coordinator,
                 keygen_output.clone(), presign_clone, m2.to_vec());

// Coordinator now holds (s1_share, s2_share) from P1 with identical nonces.
// FROST signing: s = nonce + key_share * H(commitments, group_commitment, msg)
// key_share = (s1_share - s2_share) / (challenge(m1) - challenge(m2))
// With threshold key shares recovered this way, the aggregate secret key is reconstructed.
```

```rust
async fn do_presign<C: Ciphersuite + Send>(
    mut chan: SharedChannel,
    participants: ParticipantList,
    me: Participant,
    signing_share: SigningShare<C>,
    mut rng: impl CryptoRngCore,
) -> Result<PresignOutput<C>, ProtocolError> {
    // Round 1
    let mut commitments_map: BTreeMap<Identifier<C>, SigningCommitments<C>> = BTreeMap::new();

    // Creating two commitments and corresponding nonces
    let (nonces, commitments) = commit(&signing_share, &mut rng);
    commitments_map.insert(me.to_identifier()?, commitments);

    let commit_waitpoint = chan.next_waitpoint();
    // Sending the commitments to all
    chan.send_many(commit_waitpoint, &commitments)?;

    // Collecting the commitments
    for (from, commitment) in recv_from_others(&chan, commit_waitpoint, &participants, me).await? {
        commitments_map.insert(from.to_identifier()?, commitment);
    }

    Ok(PresignOutput {
        nonces,
        commitments_map,
    })
}
