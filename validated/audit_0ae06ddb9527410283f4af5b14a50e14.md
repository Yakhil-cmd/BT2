### Title
Missing Lower-Bound Threshold Validation in FROST Presign and Sign Inputs - (File: `src/frost/mod.rs`)

### Summary
The `frost::presign` function and the `assert_sign_inputs` helper in `src/frost/mod.rs` validate that the caller-supplied `threshold` does not exceed the participant count, but they never enforce the minimum threshold of 2. The DKG entry point (`assert_key_invariants` in `src/dkg.rs`) correctly enforces both bounds, but the FROST presign and sign paths omit the lower-bound check entirely. A malicious caller can supply `threshold = 1`, causing the signing protocol to accept a single participant's contribution as sufficient to produce a valid threshold signature.

### Finding Description

`assert_key_invariants` in `src/dkg.rs` enforces both bounds on the threshold:

```rust
if threshold > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
if threshold < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
}
``` [1](#0-0) 

By contrast, `frost::presign` in `src/frost/mod.rs` only checks the upper bound:

```rust
if args.threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
``` [2](#0-1) 

There is no corresponding `ThresholdTooSmall` guard. The same omission exists in `assert_sign_inputs`, the public validation helper for the FROST signing phase:

```rust
if threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
``` [3](#0-2) 

Neither function rejects `threshold = 0` or `threshold = 1`. The `ReconstructionLowerBound` type itself imposes no minimum:

```rust
pub struct ReconstructionLowerBound(usize);
impl ReconstructionLowerBound {
    pub fn value(self) -> usize { self.0 }
}
``` [4](#0-3) 

The OT-based ECDSA presign path avoids this problem indirectly because it cross-checks the caller-supplied threshold against the threshold embedded in the Beaver triples, which were themselves validated by `validate_triple_inputs` (which does enforce `threshold_value < 2`). [5](#0-4)  The FROST path has no equivalent cross-check.

### Impact Explanation

In FROST, the threshold controls how many participants must contribute signing shares before a valid aggregate signature can be produced. With `threshold = 1`, a single participant's nonce commitment and signing share are sufficient to complete the protocol. A malicious caller (e.g., a malicious coordinator or integrator) who controls the `threshold` argument can set it to 1, reducing the effective security of the scheme to a single-party signature while honest participants believe a multi-party threshold is being enforced. This constitutes unauthorized creation of a valid threshold signature for attacker-chosen inputs — **Critical** impact under the allowed scope.

### Likelihood Explanation

The entry path is direct and requires no privileged access. `frost::presign` and `assert_sign_inputs` are public library functions. Any library caller — including a malicious coordinator or a malicious participant acting as the session initiator — can supply `threshold = 1` as a plain integer argument. No cryptographic material needs to be leaked or compromised. The inconsistency with `assert_key_invariants` means the protection exists in the codebase but is simply not wired into the FROST signing path.

### Recommendation

Add the same lower-bound check that `assert_key_invariants` already performs to both `frost::presign` and `assert_sign_inputs`:

```rust
if args.threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: args.threshold.value(),
        min: 2,
    });
}
```

This mirrors the existing guard in `src/dkg.rs` at lines 580–582 and closes the inconsistency.

### Proof of Concept

1. Call `frost::presign` with any valid participant list (≥2 participants) and `threshold = 1`. The function returns `Ok(...)` without error because only the upper-bound check fires. [6](#0-5) 
2. Call `assert_sign_inputs` with the same participant list and `threshold = 1`. Again, `Ok(participants)` is returned. [7](#0-6) 
3. Proceed with the FROST signing protocol using the returned participant list and `threshold = 1`. A single participant's share is now sufficient to produce a valid aggregate signature, bypassing the intended multi-party threshold requirement.

pub fn assert_sign_inputs(
    participants: &[Participant],
    threshold: impl Into<ReconstructionLowerBound>,
    me: Participant,
    coordinator: Participant,
) -> Result<ParticipantList, InitializationError> {
    let threshold = threshold.into();
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // validate threshold
    if threshold.value() > participants.len() {
        return Err(InitializationError::ThresholdTooLarge {
            threshold: threshold.value(),
            max: participants.len(),
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }
    Ok(participants)
}
