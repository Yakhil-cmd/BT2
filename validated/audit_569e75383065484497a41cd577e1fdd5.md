### Title
Missing Zero-Address Check in `receive_base_tokens` Causes Permanent ETH Freeze - (File: engine/src/engine.rs)

---

### Summary

The `receive_base_tokens` function in `engine/src/engine.rs` does not validate that the recipient address parsed from the bridge message is non-zero. Any user bridging ETH from NEAR to Aurora via `ft_transfer_call` can specify `address(0)` as the recipient, causing the ETH to be permanently minted to `address(0)` in Aurora's internal EVM state with no recovery path. This is a direct analog to the reported zero-address transfer vulnerability: missing recipient validation in a token-crediting path leads to permanent fund freeze.

---

### Finding Description

When ETH is bridged from NEAR to Aurora, the NEAR eth-connector calls `ft_on_transfer` on the Aurora engine. The engine dispatches to `receive_base_tokens` when the predecessor is the registered connector account:

```rust
// engine/src/contract_methods/connector.rs:81-82
let result = if predecessor_account_id == get_connector_account_id(&io)? {
    engine.receive_base_tokens(&args)
```

Inside `receive_base_tokens`, the recipient is parsed from `args.msg` via `FtTransferMessageData::try_from`, then used directly without any zero-address check:

```rust
// engine/src/engine.rs:777-785
let message_data = FtTransferMessageData::try_from(args.msg.as_str())?;
let amount = Wei::new_u128(args.amount.as_u128());
let receipient = message_data.recipient;
let balance = get_balance(&self.io, &receipient);
let new_balance = balance
    .checked_add(amount)
    .ok_or(errors::ERR_BALANCE_OVERFLOW)?;
set_balance(&mut self.io, &receipient, &new_balance);
```

The `FtTransferMessageData::try_from` parser accepts any valid 40-character hex string as the recipient, including `"0000000000000000000000000000000000000000"`:

```rust
// engine-types/src/parameters/connector.rs:41-56
if message.len() == 40 {
    let mut address_bytes = [0; 20];
    hex::decode_to_slice(message, &mut address_bytes)
        .map_err(|_| errors::ParseOnTransferMessageError::InvalidHexData)?;
    Address::from_array(address_bytes)  // no zero-address check
```

There is a critical inconsistency: the ERC-20 bridging path (`receive_erc20_tokens`) calls `_mint` on the deployed `EvmErc20` contract, which inherits from OpenZeppelin's ERC-20 and explicitly reverts on `address(0)`. The base ETH path has no equivalent guard.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

ETH credited to `address(0)` in Aurora's internal EVM state (via `set_balance`) is irrecoverable:
- No private key controls `address(0)`.
- Aurora has no admin sweep or recovery mechanism for balances at arbitrary addresses.
- The NEP-141 ETH tokens on NEAR are consumed by the connector upon a successful `ft_on_transfer` return (the function returns `"0"` on success, meaning no refund), so the underlying NEAR-side tokens are also gone.
- The total ETH supply tracked inside Aurora becomes inflated relative to what is actually accessible, causing accounting drift.

---

### Likelihood Explanation

**Medium.** The `msg` field in `ft_transfer_call` is fully user-controlled. A user can accidentally supply an uninitialized or zeroed address (common in scripting errors, copy-paste mistakes, or programmatic integrations that fail to populate the recipient field). A malicious actor can also deliberately trigger this to grief another user's funds or inflate Aurora's internal supply. No special privilege is required — any NEAR account holding ETH connector NEP-141 tokens can trigger this.

---

### Recommendation

Add an explicit zero-address guard in `receive_base_tokens` before crediting the balance:

```rust
// engine/src/engine.rs — inside receive_base_tokens
let receipient = message_data.recipient;
if receipient == Address::zero() {
    return Err(errors::ERR_ZERO_ADDRESS_RECIPIENT.into());
}
```

Alternatively, add the check inside `FtTransferMessageData::try_from` so that all callers benefit from the validation uniformly.

---

### Proof of Concept

1. User holds NEP-141 ETH connector tokens on NEAR.
2. User calls `ft_transfer_call` on the ETH connector contract:
   - `receiver_id`: Aurora engine account (e.g., `aurora`)
   - `amount`: `1000000000000000000` (1 ETH worth)
   - `msg`: `"0000000000000000000000000000000000000000"` (zero address in hex)
3. The connector calls `ft_on_transfer` on Aurora with `predecessor = connector_account_id`.
4. Aurora dispatches to `receive_base_tokens`.
5. `FtTransferMessageData::try_from("0000000000000000000000000000000000000000")` succeeds, returning `recipient = Address::zero()`.
6. `set_balance` credits 1 ETH to `address(0)` in Aurora's internal state.
7. `ft_on_transfer` returns `"0"` (no refund), so the connector finalizes the transfer — the user's NEP-141 tokens are consumed.
8. The 1 ETH is permanently frozen at `address(0)` in Aurora's EVM state with no recovery path.
