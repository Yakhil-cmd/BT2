### Title
Zero-Timelock Allows Pool Admin to Atomically Disable Stop-Loss Protection in the Same Block — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

---

### Summary

`OracleValueStopLossExtension` enforces a timelock between proposing and executing parameter changes (drawdown, decay, watermarks) so that LPs can monitor and exit before protection is weakened. However, when `timelock == 0`, the `_afterTimelock` / `_requireElapsed` pair resolves to `executeAfter = block.timestamp` and the elapsed check `block.timestamp < block.timestamp` is immediately `false`. A pool admin operating with a zero timelock can propose and execute any parameter change — including setting `drawdownE6 = 0` to fully disable the stop-loss — in a single atomic transaction, giving LPs no reaction window.

---

### Finding Description

`_afterTimelock` computes the execution timestamp as:

```solidity
// OracleValueStopLossExtension.sol L297-299
function _afterTimelock(address pool_) private view returns (uint32) {
    return uint32(block.timestamp + oracleStopLossConfig[pool_].timelock);
}
```

`_requireElapsed` enforces the delay with a strict less-than:

```solidity
// OracleValueStopLossExtension.sol L301-303
function _requireElapsed(uint32 executeAfter) private view {
    if (block.timestamp < executeAfter) revert OracleStopLossTimelockNotElapsed(executeAfter, block.timestamp);
}
```

When `timelock == 0`:
- `executeAfter = block.timestamp + 0 = block.timestamp`
- Guard: `block.timestamp < block.timestamp` → `false` → **no revert**

Every propose/execute pair (drawdown, decay, watermarks) calls `_afterTimelock` then `_requireElapsed`. All three paths are affected:

```solidity
// L103-119 (drawdown path — same pattern for decay L130-146 and watermarks L157-176)
function proposeOracleStopLossDrawdown(address pool_, uint256 newMaxDrawdownE6) external onlyPoolAdmin(pool_) {
    ...
    uint32 executeAfter = _afterTimelock(pool_);   // = block.timestamp when timelock==0
    sched.pendingDrawdownExecuteAfter = executeAfter;
}

function executeOracleStopLossDrawdown(address pool_) external onlyPoolAdmin(pool_) {
    ...
    _requireElapsed(sched.pendingDrawdownExecuteAfter);  // passes immediately
    oracleStopLossConfig[pool_].drawdownE6 = drawdown;
}
```

The timelock value is accepted without a minimum-value check at initialization:

```solidity
// L56-62
(uint32 drawdownE6, uint32 decayPerSecondE8, uint32 timelock) = abi.decode(data, (uint32, uint32, uint32));
_validateDrawdown(drawdownE6);
_validateDecay(decayPerSecondE8);
// ← no _validateTimelock; timelock == 0 is silently accepted
oracleStopLossConfig[pool] = PoolStopLossConfig({
    drawdownE6: drawdownE6, decayPerSecondE8: decayPerSecondE8, timelock: timelock, initialized: true
});
```

Additionally, `proposeOracleStopLossTimelock` accepts `newTimelock = 0` with no minimum guard, so a pool that starts with a non-zero timelock can have it reduced to zero after the current delay elapses, and from that point on all subsequent changes are atomic.

---

### Impact Explanation

The NatDoc comment on the contract explicitly states the purpose of the timelock:

> *"Drawdown and decay changes are timelocked so LPs can react; monitor at least as often as the timelock or trust the pool admin."*

When the timelock is zero, a malicious pool admin can in a single block:
1. Call `proposeOracleStopLossDrawdown(pool, 0)` — sets `drawdownE6 = 0` pending, `executeAfter = block.timestamp`
2. Call `executeOracleStopLossDrawdown(pool)` — immediately applies `drawdownE6 = 0`

With `drawdownE6 == 0`, `_afterSwapOracleStopLoss` returns early (`if (drawdown == 0) return;`), disabling all stop-loss checks. Swaps that would previously have been blocked by the watermark breach can now drain LP value without restriction. LPs have zero blocks to observe the pending change and withdraw.

**Impact class:** Admin-boundary break — the timelock is the explicit cap on pool-admin power over LP protection; a zero timelock eliminates it entirely, enabling direct loss of LP principal.

---

### Likelihood Explanation

- **Initialization path:** Any pool created with `timelockSeconds = 0` in the extension init data is immediately vulnerable. No on-chain enforcement prevents this.
- **Reduction path:** A pool admin who initially set a non-zero timelock can reduce it to zero by waiting one timelock period, then atomically disabling protection thereafter.
- **Trigger:** Only the pool admin can call these functions (`onlyPoolAdmin`), so the actor is semi-trusted. However, the entire purpose of the timelock is to bound the admin's power; a zero timelock removes that bound entirely.

---

### Recommendation

1. **Enforce a minimum timelock at initialization and on updates.** Add a `MIN_TIMELOCK` constant (e.g., `1 days`) and validate in both `initialize` and `proposeOracleStopLossTimelock`:

```solidity
uint32 private constant MIN_TIMELOCK = 1 days;

function _validateTimelock(uint256 t) private pure {
    if (t < MIN_TIMELOCK) revert OracleStopLossTimelockTooShort(t);
}
```

2. **Use strict greater-than in `_requireElapsed`** (`block.timestamp <= executeAfter` reverts) to prevent same-block execution even if `timelock == 1`.

---

### Proof of Concept

```solidity
// Pool initialized with timelock = 0:
// abi.encode(uint32 drawdownE6=500_000, uint32 decayPerSecondE8=58, uint32 timelock=0)

// In a single transaction (same block.timestamp):
vm.startPrank(poolAdmin);

// Step 1: propose drawdown = 0 (disable stop-loss)
// _afterTimelock returns block.timestamp + 0 = block.timestamp
extension.proposeOracleStopLossDrawdown(pool, 0);

// Step 2: execute immediately — _requireElapsed: block.timestamp < block.timestamp == false → no revert
extension.executeOracleStopLossDrawdown(pool);

vm.stopPrank();

// Result: drawdownE6 == 0, stop-loss disabled, afterSwap returns early for all bins
// LPs had 0 blocks to react; swaps that drain value are now unblocked
(uint32 drawdown,,,) = extension.oracleStopLossConfig(pool);
assertEq(drawdown, 0); // stop-loss fully disabled
```
