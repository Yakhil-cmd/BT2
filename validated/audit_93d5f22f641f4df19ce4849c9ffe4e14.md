### Title
Missing Caller Access Control on `update_rewards` Allows Anyone to Freeze Staker Yield - (File: src/staking/staking.cairo)

### Summary
The `update_rewards` function in the `StakingRewardsManagerImpl` implementation has no check on who the caller is. Any unprivileged address can invoke it with `disable_rewards: true` every block, consuming the global `last_reward_block` slot and permanently preventing legitimate reward distribution for all stakers.

### Finding Description
`update_rewards` is the function through which the attestation contract triggers per-block reward accounting for stakers. It is exposed as a public ABI entry point via `IStakingRewardsManager` with no `assert!(get_caller_address() == self.attestation_contract.read(), ...)` guard. [1](#0-0) 

The function's only rate-limiting mechanism is a single global storage slot `last_reward_block`: [2](#0-1) 

After passing that check, the function unconditionally writes the current block number to `last_reward_block`: [3](#0-2) 

Then, if `disable_rewards` is `true`, it returns immediately without distributing any rewards: [4](#0-3) 

The attestation contract calls into the staking contract to trigger reward updates after a successful attestation: [5](#0-4) 

Because `last_reward_block` is a single global slot shared across all callers, any address that races to call `update_rewards(any_active_staker, disable_rewards: true)` before the attestation contract does so in a given block will:
1. Consume the block's reward slot.
2. Distribute zero rewards.
3. Cause every subsequent call in that block to revert with `REWARDS_ALREADY_UPDATED`.

### Impact Explanation
An attacker submitting one cheap transaction per block with `disable_rewards: true` permanently freezes unclaimed yield for every staker in the protocol. No staker can ever accumulate rewards as long as the attacker continues the griefing. This matches the allowed impact: **High — Permanent freezing of unclaimed yield**.

### Likelihood Explanation
The entry point is fully public with no role or address restriction. Any Starknet account can call it. The only cost is one L2 transaction per block (~seconds on Starknet), making sustained griefing economically trivial. A competitor validator, a malicious actor, or even an automated bot could execute this indefinitely.

### Recommendation
Add an explicit caller check at the top of `update_rewards`, restricting it to the registered attestation contract address:

```cairo
assert!(
    get_caller_address() == self.attestation_contract.read(),
    "{}",
    Error::CALLER_IS_NOT_ATTESTATION_CONTRACT,
);
```

This mirrors the pattern already used in `RewardSupplier` for its staking-contract-only functions: [6](#0-5) 

### Proof of Concept

1. Attacker observes any active staker address `S` on-chain (trivially available from `stake` events).
2. Each block, attacker calls:
   ```
   IStakingRewardsManager(staking_contract).update_rewards(S, disable_rewards: true)
   ```
3. `last_reward_block` is set to the current block number; no rewards are minted or credited.
4. When the attestation contract later calls `update_rewards_from_attestation_contract` (which routes through the same `update_rewards` path), it reverts with `REWARDS_ALREADY_UPDATED`.
5. All stakers accumulate zero unclaimed rewards indefinitely.

The staking contract stores the attestation contract address but never uses it to gate `update_rewards`: [7](#0-6)
