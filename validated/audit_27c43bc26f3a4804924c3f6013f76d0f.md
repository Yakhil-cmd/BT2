### Title
Missing Zero-Address Validation on `reward_address` Enables Permanent Freezing of Unclaimed Yield - (File: `src/staking/staking.cairo`, `src/pool/pool.cairo`)

---

### Summary
The `stake()` function in `staking.cairo` and `enter_delegation_pool()` in `pool.cairo` accept a caller-supplied `reward_address` without validating it against the zero address. If a staker or pool member supplies `reward_address = 0`, all accrued STRK rewards will be routed to address(0) on every `claim_rewards()` call, permanently destroying them. The same gap exists in both `change_reward_address()` functions, allowing the zero address to be set post-registration.

---

### Finding Description

**Root cause — `stake()` in `staking.cairo`:** [1](#0-0) 

The only address-level guard on `reward_address` is the token-address check:

```cairo
assert!(
    !self.does_token_exist(token_address: reward_address),
    "{}",
    GenericError::REWARD_ADDRESS_IS_TOKEN,
);
```

There is no `assert!(reward_address.is_non_zero(), ...)` guard. The zero address passes this check because it is not a registered token.

**Root cause — `change_reward_address()` in `staking.cairo`:** [2](#0-1) 

Same pattern: only the token-address check, no zero-address check.

**Root cause — `enter_delegation_pool()` in `pool.cairo`:** [3](#0-2) 

**Root cause — `change_reward_address()` in `pool.cairo`:** [4](#0-3) 

**Reward transfer sink — `send_rewards_to_staker()` in `staking.cairo`:** [5](#0-4) 

When `reward_address` is zero, `checked_transfer(recipient: 0, ...)` is called unconditionally. Depending on the STRK ERC20 implementation:
- If the token **reverts** on zero-address transfers: `claim_rewards` panics on every call, permanently freezing the staker's unclaimed yield until `change_reward_address` is called (but rewards accrued in the interim are still inaccessible).
- If the token **allows** zero-address transfers (burn): rewards are silently destroyed on every `claim_rewards` call.

The same applies to pool member rewards in `pool.cairo`: [6](#0-5) 

**Why `claim_rewards` does not self-protect:** [7](#0-6) 

The authorization check `caller_address == staker_address || caller_address == reward_address` is satisfied by the staker calling with their own address, so `claim_rewards` proceeds even when `reward_address` is zero.

**Contrast with the existing zero-caller guard** (which only protects the *caller*, not the *parameter*): [8](#0-7) 

---

### Impact Explanation

**High — Permanent freezing of unclaimed yield.**

A staker or pool member who registers with `reward_address = 0` (or later calls `change_reward_address(0)`) will have all future reward claims either silently burned or cause every `claim_rewards` invocation to revert. In the revert scenario the yield is frozen until the address is corrected, but any rewards already transferred to address(0) are irrecoverable. In the burn scenario every claim permanently destroys the yield. Both outcomes satisfy the "permanent freezing of unclaimed yield" impact criterion.

---

### Likelihood Explanation

**Low-Medium.** A staker or delegator must supply the zero address themselves. This is realistic for:
- Programmatic integrations or scripts that pass an uninitialized variable.
- Users who misunderstand the parameter and leave it as the default zero value.
- Adversarial self-harm to grief a protocol metric (total rewards claimed).

No privileged role, leaked key, or external dependency is required. The entry path is fully permissionless via `stake()` or `enter_delegation_pool()`.

---

### Recommendation

Add an explicit zero-address assertion on `reward_address` in all four entry points:

```cairo
assert!(reward_address.is_non_zero(), "Reward address cannot be zero");
```

Apply this in:
- `stake()` — `src/staking/staking.cairo`
- `change_reward_address()` — `src/staking/staking.cairo`
- `enter_delegation_pool()` — `src/pool/pool.cairo`
- `change_reward_address()` — `src/pool/pool.cairo`

---

### Proof of Concept

1. Attacker/user calls `stake(reward_address: 0, operational_address: X, amount: MIN_STAKE)`.
   - The `does_token_exist(0)` check passes (zero is not a registered token).
   - Staker record is written with `reward_address = 0`.
2. Epochs advance; the staker earns STRK rewards via attestation.
3. Staker calls `claim_rewards(staker_address)`.
   - `caller_address == staker_address` → authorization passes.
   - `send_rewards_to_staker` calls `checked_transfer(recipient: 0, amount: rewards)`.
   - Rewards are either burned (if ERC20 permits) or the call reverts, freezing the yield.
4. The same sequence applies to a pool member calling `enter_delegation_pool(reward_address: 0, amount: X)` followed by `claim_rewards(pool_member)`.
