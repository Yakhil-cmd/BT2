### Title
Excess NEAR Deposit Not Refunded to Caller in `fund_xcc_sub_account` — (File: engine/src/xcc.rs)

---

### Summary

The `fund_xcc_sub_account` function accepts an arbitrary NEAR deposit from any caller but unconditionally forwards the **entire** attached amount to the XCC router sub-account. When a user sends more than the minimum required (`STORAGE_AMOUNT` = 2 NEAR), the excess is not refunded to the caller (`predecessor_account_id`). The surplus NEAR is permanently locked inside the router sub-account, which the user cannot directly withdraw from.

---

### Finding Description

In `engine/src/xcc.rs`, `fund_xcc_sub_account` captures the full attached deposit:

```rust
let fund_amount = Yocto::new(env.attached_deposit());
``` [1](#0-0) 

When the router account needs to be created for the first time (`create_needed = true`), the code enforces a minimum of `STORAGE_AMOUNT` but then transfers the **entire** `fund_amount` — not just the minimum — to the new sub-account:

```rust
if fund_amount < STORAGE_AMOUNT {
    return Err(FundXccError::InsufficientBalance);
}
promise_actions.push(PromiseAction::Transfer {
    amount: fund_amount,   // ← entire deposit, not just STORAGE_AMOUNT
});
``` [2](#0-1) 

Similarly, in the upgrade path (`create_needed = false`), the full `fund_amount` is attached to the `deploy_upgrade` call:

```rust
promise_actions.push(PromiseAction::FunctionCall {
    name: "deploy_upgrade".into(),
    args: ...,
    attached_yocto: fund_amount,   // ← entire deposit
    gas: UPGRADE_GAS + INITIALIZE_GAS,
});
``` [3](#0-2) 

And in the already-up-to-date path, the full amount is also transferred:

```rust
promise_actions.push(PromiseAction::Transfer {
    amount: fund_amount,
});
``` [4](#0-3) 

There is **no code path** that computes `excess = fund_amount - STORAGE_AMOUNT` and returns it to `env.predecessor_account_id()`. The router sub-account's only outbound transfer mechanism is `send_refund`, which sends a fixed `REFUND_AMOUNT` (2 NEAR) back to the **engine** (the parent), not to the original caller:

```rust
pub fn send_refund(&self) -> Promise {
    let parent = self.get_parent().unwrap_or_else(env_panic);
    ...
    Promise::new(parent).transfer(REFUND_AMOUNT)
}
``` [5](#0-4) 

The router exposes no `withdraw` function. The only way NEAR leaves the router is via `execute`/`execute_scheduled` (for XCC calls) or `send_refund` (fixed amount to the engine). A user who overpays cannot recover the excess.

The function is publicly callable by any unprivileged user when `wnear_account_id` is `None`:

```rust
if args.wnear_account_id.is_some() {
    require_owner_only(&state, &env.predecessor_account_id())?;
}
xcc::fund_xcc_sub_account(&io, handler, env, args)?;
``` [6](#0-5) 

The NEAR-level entry point is the exported `fund_xcc_sub_account` function: [7](#0-6) 

---

### Impact Explanation

Any NEAR sent above `STORAGE_AMOUNT` (2 NEAR) is permanently locked inside the router sub-account. The user cannot directly withdraw it; they can only consume it indirectly through future XCC calls. If the user never makes XCC calls, the funds are permanently inaccessible. This constitutes **High — temporary (potentially permanent) freezing of user funds**.

---

### Likelihood Explanation

The function is publicly callable by any NEAR account without special privileges (when `wnear_account_id` is `None`). A user who sends, for example, 10 NEAR to create their XCC router (minimum is 2 NEAR) will silently lose 8 NEAR to the router sub-account with no on-chain indication that a refund was expected. Likelihood is **Medium** — the scenario is realistic for any user who does not know the exact minimum deposit.

---

### Recommendation

After computing the required amount, refund the excess to `env.predecessor_account_id()`. For the `create_needed = true` branch, transfer only `STORAGE_AMOUNT` to the router and return `fund_amount - STORAGE_AMOUNT` to the caller via a `PromiseAction::Transfer` on the predecessor account. Apply the same pattern to the upgrade branch.

---

### Proof of Concept

1. Any NEAR account calls `fund_xcc_sub_account` with `wnear_account_id = None` and attaches 10 NEAR (`fund_amount = 10 NEAR`).
2. The router for the caller's EVM address does not yet exist (`create_needed = true`).
3. `STORAGE_AMOUNT` = 2 NEAR; the check `fund_amount < STORAGE_AMOUNT` passes.
4. `PromiseAction::Transfer { amount: fund_amount }` transfers all 10 NEAR to the router sub-account.
5. No refund promise is created for the caller.
6. The caller's 8 NEAR excess is locked in the router sub-account.
7. The router has no `withdraw` function; `send_refund` only returns a fixed 2 NEAR to the engine, not to the caller.
8. The 8 NEAR is inaccessible to the caller unless they make future XCC calls that consume it.
