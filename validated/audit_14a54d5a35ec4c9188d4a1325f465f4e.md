### Title
Asymmetric `delta_gas` Adjustment in `compute_gas_refund` Causes Users to Systematically Overpay Gas Fees — (`basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`)

---

### Summary

In `compute_gas_refund`, the `delta_gas` reconciliation between native resource consumption and EVM gas consumption is applied **only when positive**. When `delta_gas < 0` (EVM gas consumed exceeds native-implied gas), the negative difference is silently dropped. Users are charged the full EVM gas amount rather than the lower native-resource-implied amount, and the excess is never refunded. The code itself flags this with a `// TODO: return delta_gas to gas_used?` comment.

---

### Finding Description

ZKsync OS implements a double resource accounting model: every transaction is charged both **Ergs** (EVM gas × 256) and **native resource** (RISC-V proving cycles). After execution, `compute_gas_refund` reconciles the two dimensions via `delta_gas`:

```
delta_gas := (native_used / native_per_gas) - gas_used
```

The documentation (`docs/double_resource_accounting.md`) states:

> "If `deltaGas > 0`, we add it to `gasUsed` and charge it from ergs."

The code implements exactly this — but **only this**:

```rust
// basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs, lines 69-80
let delta_gas = if native_per_gas == 0 {
    0
} else {
    (native_used / native_per_gas) as i64 - (gas_used as i64)
};

if delta_gas > 0 {
    // In this case, the native resource consumption is more than the
    // gas consumption accounted for. Consume extra gas.
    gas_used += delta_gas as u64;
}
// TODO: return delta_gas to gas_used?
```

When `delta_gas < 0` — meaning EVM gas consumption is **higher** than native resource consumption warrants — no adjustment is made. `gas_used` remains at the inflated EVM gas value. The resulting `gas_used` is then used directly to:

1. Compute the user's refund: `refund = gas_price × (gas_limit − gas_used)` — **smaller than it should be**
2. Compute the operator's fee: `fee = gas_price × gas_used` — **larger than it should be**

For ZK (L2) transactions this flows through `refund_and_commit_fee` in `basic_bootloader/src/bootloader/transaction_flow/zk/mod.rs` (lines 452–546). For L1→L2 transactions it flows through `process_l1_transaction` in `basic_bootloader/src/bootloader/transaction_flow/zk/process_l1_transaction.rs` (lines 277–279).

The documentation describes only the `delta_gas > 0` branch and is silent on the `delta_gas < 0` case, and the in-code TODO confirms the negative branch was noticed but never resolved.

---

### Impact Explanation

Every transaction where EVM gas consumption exceeds native-implied gas consumption results in the user paying more than the actual resource cost. The excess `gas_price × |delta_gas|` tokens are transferred to the operator instead of being refunded to the sender. This is a direct, repeatable loss of user funds on every such transaction, with no mechanism for recovery.

**Impact: Medium** — funds are lost per-transaction but bounded by `gas_price × gas_limit`; no single transaction can drain arbitrary funds.

---

### Likelihood Explanation

The condition `delta_gas < 0` arises whenever EVM opcode execution consumes more gas than the native (RISC-V proving) resource consumption implies. This is a normal operating condition: EVM opcodes like `SLOAD` on warm slots, arithmetic, and memory operations carry fixed EVM gas costs that may exceed their proving cost at certain `nativePrice` / `gasPrice` ratios. Any standard EVM transaction submitted with a gas price close to the base fee (low `native_per_gas`) is likely to trigger this path.

**Likelihood: Medium** — occurs routinely under normal network conditions whenever `gasPrice / nativePrice` is low relative to EVM opcode costs.

---

### Recommendation

Apply the negative `delta_gas` symmetrically. When `delta_gas < 0`, reduce `gas_used` by `|delta_gas|` (subject to the `minimal_gas_used` floor already enforced earlier in the function):

```rust
if delta_gas > 0 {
    gas_used += delta_gas as u64;
} else if delta_gas < 0 {
    // Native consumption implies less gas than EVM gas used.
    // Refund the difference, but never go below the intrinsic minimum.
    let reduction = (-delta_gas) as u64;
    gas_used = gas_used.saturating_sub(reduction).max(minimal_gas_used);
}
```

This mirrors the documented intent: `gas_used` should reflect the maximum of EVM gas and native-implied gas only when native is the binding constraint, not unconditionally.

---

### Proof of Concept

**Setup**: Submit an EIP-1559 L2 transaction with:
- `gasLimit = 100_000`
- `gasPrice = baseFee` (no priority fee, so `native_per_gas = baseFee / nativePrice`)
- A workload that consumes ~50,000 EVM gas but only ~20,000 native-implied gas (e.g., warm-slot `SLOAD` loops)

**Execution trace**:
1. `gas_used` (from ergs) = 50,000
2. `native_used / native_per_gas` = 20,000
3. `delta_gas` = 20,000 − 50,000 = **−30,000**
4. The `if delta_gas > 0` branch is **not taken**; `gas_used` stays at 50,000
5. User refund = `gasPrice × (100,000 − 50,000)` = `gasPrice × 50,000`
6. **Expected** refund = `gasPrice × (100,000 − 20,000)` = `gasPrice × 80,000`
7. User loses `gasPrice × 30,000` tokens; operator receives them instead
