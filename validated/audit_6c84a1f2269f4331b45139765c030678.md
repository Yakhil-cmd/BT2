### Title
Stale Pre-Resharding `target_shard_index` Corrupts `outcome_root_proof` for Shards Ordered After the Split Shard — (`chain/client/src/view_client_actor.rs`)

### Summary

The `GetExecutionOutcome` handler computes `target_shard_index` from the **pre-resharding** shard layout of the outcome block, but then uses that index to select a Merkle path from `outcome_roots` collected from a **post-resharding** block (which has a different number of chunks and different shard-to-index mapping). For any shard whose index shifts upward after the split (i.e., any shard ordered after the split shard), the returned `outcome_root_proof` is the Merkle path for the wrong shard's `prev_outcome_root`, causing light-client proof verification to fail.

### Finding Description

The handler at lines 1148–1182 of `chain/client/src/view_client_actor.rs` executes the following sequence:

**Step 1 — epoch and index from the outcome block (pre-resharding):** [1](#0-0) 

`epoch_id` is the epoch of the outcome block (pre-resharding). `shard_layout` and `target_shard_index` are therefore derived from the **pre-resharding** layout.

**Step 2 — `get_next_block_hash_with_new_chunk` crosses the epoch boundary:** [2](#0-1) 

When the function crosses into the next epoch and detects a shard layout change, it replaces `shard_ids` with the children of the split shard and updates `shard_layout` to the post-resharding layout. It returns `(h, child_shard_id)` where `h` is a **post-resharding** block.

**Step 3 — `outcome_roots` collected from the post-resharding block:** [3](#0-2) 

`outcome_roots` has `N+1` entries (post-resharding shard count). The Merkle path is selected with `target_shard_index` — which was computed from the **pre-resharding** layout at step 1 and is **never updated**.

**Concrete index mismatch example:**

| Layout | Shards | Indices |
|--------|--------|---------|
| Pre-resharding (2 shards) | A=0, B=1 | A→0, B→1 |
| Post-resharding (3 shards, B splits into B1, B2) | A=0, B1=1, B2=2 | A→0, B1→1, B2→2 |

If `receiver_id` maps to shard A (pre-index 0, post-index 0): **correct** — index unchanged.  
If `receiver_id` maps to shard B (pre-index 1, post-index 1 for B1): **correct** — left child inherits parent's index.

Now consider a 3-shard layout where the middle shard splits:

| Layout | Shards | Indices |
|--------|--------|---------|
| Pre-resharding (3 shards) | A=0, B=1, C=2 | A→0, B→1, C→2 |
| Post-resharding (4 shards, B splits) | A=0, B1=1, B2=2, C=3 | A→0, B1→1, B2→2, C→3 |

If `receiver_id` maps to shard C (pre-index **2**, post-index **3**):
- `target_shard_index` = 2 (from pre-resharding layout, never updated)
- `outcome_roots` = `[root_A, root_B1, root_B2, root_C]` (4 entries)
- `merklize(&outcome_roots).1[2]` = Merkle path for `root_B2` (index 2)
- **Correct path would be** `merklize(&outcome_roots).1[3]` = Merkle path for `root_C`

The returned `outcome_root_proof` is the path for B2's outcome root, not C's. A light client computing:
```
block_outcome_root = compute_root(sha256(borsh(C_shard_outcome_root)), B2_path)
```
will obtain a value that does not match `block_header_lite.inner_lite.outcome_root`, causing verification to fail. [4](#0-3) 

### Impact Explanation

Any call to `light_client_proof` (or `GetExecutionOutcome`) for a receipt whose:
1. outcome block is the **last block of the pre-resharding epoch**, and
2. `receiver_id` maps to a shard whose index shifts upward after the split (any shard ordered after the split shard)

…will receive a structurally invalid `outcome_root_proof`. The proof fails `verify_path` against `block_header_lite.inner_lite.outcome_root`. Bridges, cross-chain protocols, and any application relying on NEAR light-client proof semantics cannot confirm execution of these receipts.

The corrupted value is precisely: `merklize(&outcome_roots).1[target_shard_index]` where `target_shard_index` is an index into the pre-resharding layout but `outcome_roots` is indexed by the post-resharding layout.

### Likelihood Explanation

Resharding epochs are infrequent, but the window is deterministic and predictable. Any cross-shard receipt executed in the last block of the pre-resharding epoch where the receiver's shard is ordered after the split shard triggers the bug. No special privileges are required — any user submitting a cross-shard receipt can land in this window.

### Recommendation

After `get_next_block_hash_with_new_chunk` returns `(h, new_target_shard_id)`, recompute `target_shard_index` using the shard layout of block `h` (the post-resharding layout) and the returned `new_target_shard_id`:

```rust
if let Some((h, target_shard_id)) = res {
    // Recompute index using the layout of the block that contains the proof
    let h_epoch_id = *self.chain.get_block(&h)?.header().epoch_id();
    let h_shard_layout = self.epoch_manager.get_shard_layout(&h_epoch_id).into_chain_error()?;
    let target_shard_index = h_shard_layout
        .get_shard_index(target_shard_id)
        .map_err(Into::into)
        .into_chain_error()?;
    // ... rest unchanged
```

### Proof of Concept

1. Configure a test-loop with a 3-shard layout (shards A, B, C) and a resharding that splits B.
2. Submit a cross-shard receipt whose `receiver_id` maps to shard C (the shard after the split shard).
3. Arrange for the receipt to execute in the last block of the pre-resharding epoch.
4. Call `GetExecutionOutcome` with `TransactionOrReceiptId::Receipt { receipt_id, receiver_id: C_account }`.
5. Assert that `verify_path(block_header_lite.inner_lite.outcome_root, outcome_root_proof, chunk_outcome_root)` returns **false** (proof is for B2's position, not C's).
6. Fix: recompute `target_shard_index` from the post-resharding layout of block `h` and assert the same call returns **true**.
