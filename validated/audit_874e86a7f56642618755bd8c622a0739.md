### Title
Untrusted On-Chain `puzzleHash` Pre-fills Counter Offer Notification Destination Without Validation ‚Äî (`packages/gui/src/components/offers2/OfferIncomingTable.tsx`, `packages/gui/src/hooks/useBlockchainNotifications.tsx`)

---

### Summary

An attacker can broadcast an on-chain offer notification whose `ph` (puzzle hash) field is set to the attacker's own puzzle hash. The GUI parses this field with no validation and uses it ‚Äî after `toBech32m` conversion ‚Äî as the pre-filled, **user-uneditable** notification-reply destination in the counter offer flow. If the victim subsequently creates a counter offer and clicks "Send Notification," the small XCH notification coin is sent to the attacker's address rather than the legitimate offer creator's address.

---

### Finding Description

**Step 1 ‚Äî Attacker crafts on-chain notification**

The notification message payload is a JSON blob embedded in a blockchain coin:
```json
{"v":1,"t":1,"d":{"u":"https://attacker.example/offer.bin","ph":"<attacker_hex_puzzle_hash>"}}
```
No signature, ownership proof, or wallet-binding check is required to broadcast this.

**Step 2 ‚Äî GUI parses `ph` without validation**

In `useBlockchainNotifications.tsx`, the `ph` field is extracted directly from the decoded on-chain message and stored verbatim in the notification object: [1](#0-0) 

There is no check that the puzzle hash belongs to the sender, matches any known wallet, or is even a valid 32-byte hex string.

**Step 3 ‚Äî `handleCounterOffer` converts it to a bech32m address**

When the victim clicks "Counter" in `OfferIncomingTable`, `handleCounterOffer` reads `notification.puzzleHash` and converts it unconditionally: [2](#0-1) 

The resulting `address` (attacker-controlled bech32m) is placed into React Router state and the user is navigated to the offer builder.

**Step 4 ‚Äî Address flows into `CreateOfferBuilder` and `onOfferCreated`**

The router state `address` is passed directly to `CreateOfferBuilder`: [3](#0-2) 

After the offer is created, `address` is forwarded to `onOfferCreated`: [4](#0-3) 

**Step 5 ‚Äî `OfferShareDialog` opens with attacker's address as notification destination**

`handleOfferCreated` in `OfferManager` receives the address and passes it to `OfferShareDialog`: [5](#0-4) 

**Step 6 ‚Äî Notification destination field is disabled; user cannot correct it**

In `NotificationSendDialog`, the address field is rendered as `disabled`, so the victim cannot change the pre-filled attacker address: [6](#0-5) 

If the victim clicks "Send Message," `sendNotification` is called with `target = fromBech32m(address)` ‚Äî the attacker's puzzle hash ‚Äî and the notification XCH coin is sent to the attacker. [7](#0-6) 

**Note on the `OfferIncomingTable` path vs. `NotificationOffer` path**

In the `OfferIncomingTable` path, `handleCounterOffer` converts the raw puzzle hash to bech32m before storing it in router state. `handleOfferCreated` then calls `toBech32m` on the already-encoded bech32m string (double-encoding), which would likely corrupt or invalidate the address at that stage.

The cleaner exploit path is through `NotificationOffer.tsx`, which passes the raw puzzle hash as `address` through to `handleOfferCreated`, where it is correctly encoded once: [8](#0-7) 

Both paths share the same root cause: the `puzzleHash` from on-chain data is trusted without any validation.

---

### Impact Explanation

The attacker controls the notification reply destination. The victim's "Send Notification" XCH coin (default 0.0001 XCH, hardcoded) is sent to the attacker's address instead of the legitimate offer creator's address. The address field in the dialog is disabled ‚Äî the victim cannot correct it. This constitutes corruption of notification state causing the user to send XCH to the wrong destination, fitting the defined High impact category.

---

### Likelihood Explanation

Broadcasting a crafted on-chain notification requires only a small XCH amount (the notification coin itself) and no special privileges. Any unprivileged actor can do this. The victim only needs to click "Counter" on the notification and then "Send Message" in the share dialog ‚Äî a natural workflow for a user responding to an offer.

---

### Recommendation

1. **Validate `puzzleHash` format** in `useBlockchainNotifications` before storing it: verify it is a 64-character lowercase hex string.
2. **Do not trust the `ph` field as the counter offer reply destination** without displaying it prominently and allowing the user to edit or confirm it.
3. **Enable the address field** in `NotificationSendDialog` so the user can correct a pre-filled address, or add an explicit confirmation step showing the destination address.
4. Fix the double-encoding bug in `handleOfferCreated` ‚Äî it receives a bech32m address but calls `toBech32m` on it again as if it were a raw puzzle hash.

---

### Proof of Concept

1. Attacker broadcasts a notification coin to the victim's wallet address with message:
   `{"v":1,"t":1,"d":{"u":"https://dexie.space/offers/some-valid-offer","ph":"<attacker_32byte_hex_puzzle_hash>"}}`
2. Victim opens the Chia GUI; the notification appears in the Activity dropdown.
3. Victim clicks the notification ‚Üí "Counter Offer" button.
4. GUI navigates to `/dashboard/offers/builder` with `state.address = toBech32m(attacker_puzzle_hash, "xch")`.
5. Victim fills in the counter offer and clicks "Create Counter Offer."
6. `OfferShareDialog` opens; victim clicks "Send Notification."
7. `sendNotification` is called with `target = attacker_puzzle_hash`; the XCH notification coin is delivered to the attacker.
