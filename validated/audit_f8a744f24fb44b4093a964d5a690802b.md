### Title
Commented-Out Royalty Subtraction in `calculateNFTRoyalties` Causes "Net Proceeds" to Display Full Pre-Royalty Amount, Misleading NFT Sellers into Accepting Offers - (`packages/gui/src/components/offers/utils.ts`)

---

### Summary

`calculateNFTRoyalties` hardcodes `nftSellerNetAmount = amount` with the correct royalty-and-fee subtraction commented out. Every UI surface that renders this value — "Net Proceeds" in `NFTOfferViewer` and "You will receive" in `NFTOfferEditor` — therefore displays the full offered amount regardless of royalty percentage, while the on-chain settlement will deduct the royalty from what the seller actually receives.

---

### Finding Description

In `packages/gui/src/components/offers/utils.ts`, `calculateNFTRoyalties` computes `royaltyAmount` correctly but then assigns:

```typescript
const nftSellerNetAmount: number = amount;
// : parseFloat(
//     (amount - parseFloat(royaltyAmountString) - makerFee).toFixed(12),
//   );
``` [1](#0-0) 

The subtraction `amount - royaltyAmountString - makerFee` is entirely commented out, so `nftSellerNetAmount` always equals the raw offered `amount`.

This value propagates to two display surfaces:

**1. NFTOfferViewer — "Net Proceeds" label (offer acceptance flow)**

When `exchangeType === NFTOfferExchangeType.TokenForNFT`, the viewer renders a prominent "Net Proceeds" heading with a tooltip that explicitly promises *"The net proceeds include the asking price, minus any associated creator fees"*, but then renders the uncorrected `nftSaleInfo?.nftSellerNetAmount`: [2](#0-1) 

**2. NFTOfferEditor — "You will receive" label (offer creation flow)**

When the seller is creating an NFT-for-token offer (`tab === NFTOfferExchangeType.NFTForToken`), the editor shows "You will receive" using the same broken value: [3](#0-2) 

The royalty amount itself is displayed correctly in a separate "Creator Fee" row, but the "Net Proceeds" / "You will receive" figure is never reduced by it.

---

### Impact Explanation

An NFT seller — either creating their own offer or reviewing an incoming offer — sees "Net Proceeds: 100 XCH" (or equivalent CAT amount) when the on-chain royalty program will actually transfer 10 XCH to the creator and deliver only 90 XCH to the seller. The seller makes an accept/create decision based on a materially inflated figure. The discrepancy scales linearly with royalty percentage and offer size; at a 20 % royalty on a 1000 XCH offer the seller is shown 1000 XCH but receives 800 XCH. This constitutes direct asset loss caused by the GUI displaying the wrong amount at the offer-approval decision point.

---

### Likelihood Explanation

Any NFT with a nonzero `royaltyPercentage` (the majority of creator-minted NFTs on Chia mainnet) triggers the bug on every offer. No special attacker capability is required; a buyer simply creates a standard offer for such an NFT. The seller's only defense is to manually subtract the separately-displayed "Creator Fee" row from the displayed "Net Proceeds" — a step the UI explicitly claims to have already performed via its own tooltip text.

---

### Recommendation

Restore the commented-out subtraction in `calculateNFTRoyalties`:

```typescript
const nftSellerNetAmount: number = parseFloat(
  (amount - parseFloat(royaltyAmountString) - makerFee).toFixed(12),
);
``` [1](#0-0) 

Also add a unit test asserting `calculateNFTRoyalties(100, 0, 10, NFTOfferExchangeType.NFTForToken).nftSellerNetAmount === 90`.
