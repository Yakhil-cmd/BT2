### Title
Attacker-Controlled `puzzleHash` in On-Chain Notification Redirects Counter-Offer to Attacker — (`packages/gui/src/components/notification/NotificationOffer.tsx`)

### Summary

An unprivileged on-chain attacker who crafts a blockchain notification with `type=1` (OFFER) and sets the `ph` (puzzle hash) field to their own address can cause a victim who clicks the notification and then clicks "Counter Offer" to send their counter-offer notification — including the counter-offer URL — to the attacker's address instead of the legitimate offer creator's address. The attacker can then accept the counter-offer, taking the victim's offered assets.

---

### Finding Description

**Step 1 — Attacker-controlled entry point: on-chain notification payload**

The notification message format is defined in `utils.ts`. The `ph` field is parsed directly from the on-chain message with no ownership verification: [1](#0-0) 

`parseNotificationOfferData` extracts `ph` from the JSON payload and returns it as `puzzleHash` with no validation that it belongs to the offer creator.

**Step 2 — `canCounterOffer` derived solely from `puzzleHash` being non-empty**

In `NotificationOffer.tsx`, `canCounterOffer` is set to `true` whenever `notification.puzzleHash` is non-empty: [2](#0-1) 

No check is made that the puzzle hash belongs to the actual offer creator or is otherwise trustworthy.

**Step 3 — `handleClick` passes attacker's puzzle hash as `address` to router state** [3](#0-2) 

`address: notification.puzzleHash` is the attacker-controlled value. It is passed verbatim into the router state for `/dashboard/offers/view`.

**Step 4 — `OfferBuilderViewer` renders "Counter Offer" button and passes `address` through**

`OfferBuilderViewer` receives `canCounterOffer=true` and `address=attackerPuzzleHash` as props: [4](#0-3) 

When the victim clicks "Counter Offer": [5](#0-4) 

`address` (the attacker's puzzle hash) is forwarded to the offer builder with no verification.

**Step 5 — `NotificationSendDialog` uses `destination` as the notification target**

`NotificationSendDialog` accepts `destination` and uses it directly as the default address for the notification send: [6](#0-5) 

On submit, `fromBech32m(address)` is used as `target` for `sendNotification`: [7](#0-6) 

The notification sent to `target` contains the victim's counter-offer URL (`offerURL`) and the victim's own puzzle hash (for the attacker to send a counter-counter-offer). The attacker receives the counter-offer URL and can accept it, taking the victim's offered XCH/CAT assets.

---

### Impact Explanation

- The victim's counter-offer notification (containing the counter-offer URL) is sent to the attacker's address instead of the legitimate offer creator's address.
- The attacker receives the counter-offer URL and can accept the counter-offer, taking the victim's offered XCH/CAT.
- A small notification fee (0.0001 XCH by default) is also spent to the attacker's address.
- The victim has no indication that the destination is wrong — the address field in `NotificationSendDialog` is `disabled` (read-only), so the victim cannot correct it. [8](#0-7) 

---

### Likelihood Explanation

- The attacker entry point is on-chain: any node can broadcast a notification message to any puzzle hash. No special privileges are required.
- The victim must click the notification and then click "Counter Offer" — two deliberate user actions, but both are the natural response to receiving an offer notification.
- The address field in the send dialog is disabled/read-only, so the victim cannot detect or correct the spoofed destination.

---

### Recommendation

1. **Verify `puzzleHash` against the offer's maker address**: Before setting `canCounterOffer=true`, validate that `notification.puzzleHash` matches the maker's puzzle hash extracted from the offer summary itself (which is cryptographically bound to the offer).
2. **Do not trust `ph` from the notification payload as the counter-offer destination** without cross-referencing it against the offer's on-chain maker address.
3. **Display the destination address prominently** (not in a disabled field) so the user can verify it before sending.

---

### Proof of Concept

1. Attacker crafts an on-chain notification message with payload:
   ```json
   {"v":1,"t":1,"d":{"u":"<valid_offer_url>","ph":"<attacker_puzzle_hash_bech32m>"}}
   ```
   and broadcasts it to the victim's puzzle hash via `send_notification` RPC.
2. Victim sees the notification in the GUI, clicks it.
3. `NotificationOffer.handleClick` fires: `canCounterOffer=true`, `address=attackerPuzzleHash` → navigates to `/dashboard/offers/view`.
4. `OfferBuilderViewer` renders the "Counter Offer" button (because `canCounterOffer=true`).
5. Victim clicks "Counter Offer" → `handleCounterOffer` navigates to `/dashboard/offers/builder` with `address=attackerPuzzleHash`.
6. Offer builder opens `NotificationSendDialog` with `destination=attackerPuzzleHash`.
7. The address field is disabled — victim cannot see or change it.
8. Victim clicks "Send Message" → `sendNotification` is called with `target=attackerPuzzleHash`, message containing the victim's counter-offer URL.
9. Attacker receives the notification, extracts the counter-offer URL, and accepts the counter-offer — taking the victim's offered XCH/CAT.

export function parseNotificationOfferData(payload: string): NotificationOfferData | null {
  const parsed = parseNotificationPayload(payload);
  if (!parsed) {
    return null;
  }

  if (parsed.type !== NotificationTypeId.OFFER) {
    return null;
  }

  return {
    u: parsed.data?.u ?? '',
    ph: parsed.data?.ph,
  };
}
