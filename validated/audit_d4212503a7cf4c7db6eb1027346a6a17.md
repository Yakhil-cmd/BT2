### Title
Pool Admin Bypasses `maxAdminSpreadFeeE6` Cap via Uncapped `setPoolBinAdditionalFees` — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary

`MetricOmmPoolFactory.setPoolBinAdditionalFees` passes `addFeeBuyE6` / `addFeeSellE6` directly to the pool with no upper-bound check, while the parallel path `setPoolAdminFees` is explicitly capped at `maxAdminSpreadFeeE6`. A pool admin can set per-bin additional fees to `uint16` max (65 535 = 6.5535%) on any bin, causing traders who swap through that bin to pay fees far above the factory-enforced cap.

### Finding Description

`setPoolAdminFees` enforces the factory cap:

```solidity
// MetricOmmPoolFactory.sol lines 414-415
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (newAdminNotionalFeeE8 > maxAdminNotionalFeeE8) revert AdminFeeTooHigh();
``` [1](#0-0) 

`setPoolBinAdditionalFees` has **no such check**:

```solidity
// MetricOmmPoolFactory.sol lines 450-457
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
``` [2](#0-1) 

`setBinAdditionalFees` on the pool also has no cap — it stores the values verbatim:

```solidity
// MetricOmmPool.sol lines 464-474
function setBinAdditionalFees(int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external onlyFactory nonReentrant(PoolActions.SET_BIN_ADDITIONAL_FEES)
{
    if (bin < LOWEST_BIN || bin > HIGHEST_BIN) revert InvalidBinIndex(bin);
    BinState storage s = _binStates[bin];
    s.addFeeBuyE6 = addFeeBuyE6;
    s.addFeeSellE6 = addFeeSellE6;
    emit BinAdditionalFeesUpdated(bin, addFeeBuyE6, addFeeSellE6);
}
``` [3](#0-2) 

These stored values are then added directly to the effective fee in every swap path:

```solidity
// MetricOmmPool.sol line 999 (buyToken0 specifiedIn)
params.baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6),
``` [4](#0-3) 

```solidity
// MetricOmmPool.sol line 1088 (buyToken1 specifiedOut) and line 1177 (buyToken1 specifiedIn)
params.baseFeeX64 + Math.mulDiv(binState.addFeeSellE6, ONE_X64, 1e6),
``` [5](#0-4) 

The hard cap `HARD_MAX_SPREAD_FEE_E6 = 200_000` (20%) applies only to `setFeeCaps` / `setPoolAdminFees`. The `uint16` type used for per-bin fees allows a maximum of 65 535 (6.5535%), which is entirely unconstrained by any factory-level guard. [6](#0-5) 

### Impact Explanation

A pool admin who calls `setPoolBinAdditionalFees(pool, 0, 65535, 65535)` causes every swap routed through bin 0 to pay an additional 6.5535% fee on top of the base spread fee. This is a direct loss of trader principal. The fee is collected into the pool's LP balances (not to the admin directly), but the trader receives materially less output than the oracle-derived price and the factory-enforced cap would permit. This constitutes an admin-boundary break: the pool admin exceeds the cap that the factory is supposed to enforce.

### Likelihood Explanation

The pool admin is a semi-trusted role, explicitly bounded by caps in the factory design. The path is a single permissioned call with no timelock. Any pool admin — whether malicious from the start or later compromised — can execute this immediately after pool creation, affecting all traders who swap through the targeted bin.

### Recommendation

Add a cap check in `setPoolBinAdditionalFees` analogous to the one in `setPoolAdminFees`. For example:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (addFeeBuyE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

Alternatively, introduce a dedicated `maxAdminBinAdditionalFeeE6` cap settable by the factory owner.

### Proof of Concept

1. Deploy factory with `maxAdminSpreadFeeE6 = 10_000` (1%).
2. Create a pool; pool admin is `attacker`.
3. `attacker` calls `factory.setPoolBinAdditionalFees(pool, 0, 65535, 65535)` — succeeds with no revert.
4. Read `PoolStateLibrary._binState(pool, 0)` → `addFeeBuyE6 == 65535`, `addFeeSellE6 == 65535`.
5. A trader swaps token1 → token0 through bin 0; the effective fee applied is `baseFeeX64 + mulDiv(65535, ONE_X64, 1e6)`, i.e. 6.5535% additional on top of the base spread, far exceeding the 1% `maxAdminSpreadFeeE6` cap.

The existing test `test_setPoolBinAdditionalFees_updatesStorage_emitsEvent` already confirms the storage write succeeds for arbitrary values — it just never tests a value above `maxAdminSpreadFeeE6`. [7](#0-6)
