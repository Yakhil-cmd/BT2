### Title
Unhandled Partial `ft_on_transfer` Refund in Omni `withdrawToNear` Path Permanently Burns ERC-20 Tokens - (File: `engine/src/contract_methods/connector.rs`, `engine-precompiles/src/native.rs`)

---

### Summary

When `withdrawToNear` is called with an Omni-style recipient (containing a `:` separator with a non-`unwrap` suffix), the engine issues a `ft_transfer_call` to the NEP-141 contract. The NEP-141 standard explicitly permits `ft_on_transfer` to return a non-zero refund amount (partial acceptance). When this happens, the NEP-141 contract credits the refunded tokens back to the Aurora engine account. However, `exit_to_near_precompile_callback` treats any `PromiseResult::Successful` as a complete success and performs no re-mint. The user's ERC-20 tokens are permanently burned while the refunded NEP-141 tokens accumulate in the engine account.

---

### Finding Description

**Step 1 — Entry point.**
`withdrawToNear` in `EvmErc20.sol` unconditionally burns the caller's ERC-20 balance, then forwards the raw `recipient` bytes to the exit precompile at `0xe9217bc7...`. [1](#0-0) 

**Step 2 — Omni path selection.**
`parse_recipient` splits on the first `:`. Any suffix that is not `"unwrap"` produces `Message::Omni(msg)`. [2](#0-1) 

**Step 3 — `ft_transfer_call` is scheduled.**
`exit_erc20_token_to_near` selects `"ft_transfer_call"` as the method and sets `transfer_near_args = None` for the Omni branch. [3](#0-2) 

**Step 4 — Callback args carry no partial-refund handler.**
`callback_args.transfer_near` is `None` (Omni path). `callback_args.refund` is only populated under the `error_refund` feature flag, and even then it is only consumed on the *failure* branch of the callback. [4](#0-3) 

**Step 5 — The callback discards the `ft_transfer_call` return value.**
`exit_to_near_precompile_callback` matches `PromiseResult::Successful(_)` with a wildcard. The NEP-141 standard encodes the *used* amount in the return value of `ft_transfer_call`; any difference from the original amount means tokens were refunded to the sender (the engine account). The callback ignores this value entirely and returns `None`. [5](#0-4) 

**Step 6 — The failure-path re-mint is never reached.**
`refund_on_error` (which would re-mint ERC-20 tokens) is only called when `handler.promise_result(0)` is *not* `Successful`. A partial refund still produces a `Successful` result, so this branch is never entered. [6](#0-5) 

**Invariant broken:** ERC-20 tokens burned = `N`. NEP-141 tokens delivered to receiver = `N - R`. NEP-141 tokens credited back to engine account = `R`. ERC-20 tokens re-minted = `0`. The user permanently loses `R` tokens.

---

### Impact Explanation

The refunded NEP-141 tokens (`R`) accumulate in the Aurora engine account with no corresponding ERC-20 re-mint. The original ERC-20 holder suffers a permanent, unrecoverable loss equal to the refunded amount. These tokens constitute unclaimed yield sitting in the engine account — matching the **High: Theft of unclaimed yield** scope.

---

### Likelihood Explanation

The Omni path is a documented, production-supported feature (test vectors exist in `test_parse_exit_to_near_params`). Any NEP-141 receiver that legitimately returns a partial refund (e.g., a DEX with slippage, a lending protocol with capacity limits, or a deliberately crafted receiver) triggers the bug. No privileged access is required; any ERC-20 holder can call `withdrawToNear` with an Omni-style recipient. [7](#0-6) 

---

### Recommendation

In `exit_to_near_precompile_callback`, parse the `PromiseResult::Successful(bytes)` payload for the `ft_transfer_call` case. The NEP-141 standard returns the used amount as a JSON-encoded `U128` string. Compute `refunded = original_amount - used_amount`. If `refunded > 0`, call `refund_on_error` (or an equivalent re-mint path) to restore the original holder's ERC-20 balance for the refunded portion. The `RefundCallArgs` struct already carries `recipient_address` and `erc20_address`, so the re-mint infrastructure exists — it just needs to be invoked on the success path when the return value indicates a partial transfer. [8](#0-7) 

---

### Proof of Concept

1. Deploy a NEP-141 contract whose `ft_on_transfer` always returns `amount / 2` (half refund).
2. Register this NEP-141 contract with the Aurora engine (so it has a mirrored ERC-20).
3. Mint `1000` ERC-20 tokens to address `Alice`.
4. Alice calls `withdrawToNear(bytes("nep141.test:some_omni_msg"), 1000)`.
   - `_burn(Alice, 1000)` executes; Alice's ERC-20 balance → 0.
   - Exit precompile schedules `ft_transfer_call` on `nep141.test` with `amount=1000, msg="some_omni_msg"`.
5. `nep141.test`'s `ft_on_transfer` returns `"500"` (refund 500).
   - NEP-141 `ft_resolve_transfer` credits 500 back to the Aurora engine account.
   - `ft_transfer_call` resolves as `Successful("500")` (used amount).
6. `exit_to_near_precompile_callback` fires:
   - `handler.promise_result(0)` → `Some(Successful("500"))` — matches the success branch.
   - `args.transfer_near` is `None` — nothing happens.
   - Returns `None`.
7. **Assert**: Alice's ERC-20 balance = 0 (lost 1000). NEP-141 balance of receiver = 500. NEP-141 balance of Aurora engine = 500. ERC-20 re-minted = 0. Alice is short 500 ERC-20 tokens with no recovery path.

    io: I,
    env: &E,
    state: EngineState,
    args: &RefundCallArgs,
    handler: &mut P,

    let current_account_id = env.current_account_id();
    if let Some(erc20_address) = args.erc20_address {
