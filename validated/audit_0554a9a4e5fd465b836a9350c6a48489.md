Audit Report

## Title
Missing Response-Type/Callback-ID Cross-Check in `validate_self_validating_payload_impl` and `push_response` Allows Byzantine Node to Inject Mismatched Bitcoin Adapter Responses — (`rs/bitcoin/consensus/src/payload_builder.rs`, `rs/replicated_state/src/bitcoin.rs`)

## Summary

`validate_self_validating_payload_impl` performs no validation of whether a `BitcoinAdapterResponse`'s response variant matches the request type stored in state for the given `callback_id`. Separately, `push_response` for `SendTransactionResponse` enqueues `EmptyBlob` into `consensus_queue` for any `callback_id` without verifying it belongs to a `bitcoin_send_transaction_internal_contexts` entry. A single Byzantine subnet node can propose a block with a `SendTransactionResponse` carrying a `callback_id` that belongs to a live `BitcoinGetSuccessors` context; all honest nodes accept the block, and the execution environment delivers `EmptyBlob` to the bitcoin canister's `GetSuccessors` callback, permanently consuming the context and stalling UTXO synchronization.

## Finding Description

**Root cause 1 — `validate_self_validating_payload_impl` (L234–251, `rs/bitcoin/consensus/src/payload_builder.rs`):**

```rust
fn validate_self_validating_payload_impl(
    &self,
    payload: &SelfValidatingPayload,
    validation_context: &ValidationContext,
) -> Result<NumBytes, SelfValidatingPayloadValidationError> {
    let since = Instant::now();
    if *payload == SelfValidatingPayload::default() {
        return Ok(0.into());
    }
    self.metrics.observe_validate_duration(VALIDATION_STATUS_VALID, since);
    let size = NumBytes::new(payload.count_bytes() as u64);
    Ok(size)
}
```

The function checks only emptiness and byte count. It never loads state, never checks whether each `callback_id` exists in any context map, and never checks whether the response variant (`GetSuccessorsResponse` vs. `SendTransactionResponse`) matches the request type stored for that `callback_id`. [1](#0-0) 

**Root cause 2 — `validate_payload` (L357–397, `rs/bitcoin/consensus/src/payload_builder.rs`):**

`validate_payload` decodes the payload, calls `validate_self_validating_payload_impl`, and checks the byte limit. No callback_id or response-type validation is added at this layer either. [2](#0-1) 

**Root cause 3 — `push_response` for `SendTransactionResponse` (L64–76, `rs/replicated_state/src/bitcoin.rs`):**

```rust
BitcoinAdapterResponseWrapper::SendTransactionResponse(_) => {
    let callback_id = CallbackId::from(response.callback_id);
    let payload = Payload::Data(EmptyBlob.encode());
    state.consensus_queue.push(ConsensusResponse::new(callback_id, payload));
    Ok(())
}
```

No lookup in `bitcoin_send_transaction_internal_contexts` is performed. Any `callback_id` is accepted and `EmptyBlob` is unconditionally enqueued. [3](#0-2) 

**Contrast with `GetSuccessorsResponse` (L28–39, `rs/replicated_state/src/bitcoin.rs`):**

```rust
let context = state
    .metadata
    .subnet_call_context_manager
    .bitcoin_get_successors_contexts
    .get_mut(&callback_id)
    .ok_or_else(|| StateError::BitcoinNonMatchingResponse {
        callback_id: callback_id.get(),
    })?;
```

This branch returns `Err(StateError::BitcoinNonMatchingResponse)` if the `callback_id` is absent. The `SendTransactionResponse` branch has no equivalent guard. [4](#0-3) 

**`retrieve_context` (L269–349, `rs/replicated_state/src/metadata_state/subnet_call_context_manager.rs`):**

When the execution environment processes the `consensus_queue` entry, `retrieve_context` is called with the `callback_id`. It removes the entry from `bitcoin_get_successors_contexts` (L324–336), permanently consuming the context regardless of whether the canister subsequently traps. [5](#0-4) 

**Exploit flow:**

1. State contains a live `BitcoinGetSuccessors` context at `callback_id = X`.
2. Byzantine proposer crafts a `SelfValidatingPayload` containing `BitcoinAdapterResponse { response: SendTransactionResponse {}, callback_id: X }`.
3. All honest validators call `validate_payload` → `validate_self_validating_payload_impl`; both pass (only size is checked).
4. Block is finalized; `push_response` is called with the crafted response.
5. `push_response` enqueues `ConsensusResponse { callback_id: X, payload: EmptyBlob }` into `consensus_queue`.
6. Execution environment processes the entry; `retrieve_context(X)` removes the `BitcoinGetSuccessors` context from `bitcoin_get_successors_contexts`.
7. `EmptyBlob` is delivered to the bitcoin canister's `GetSuccessors` callback; the canister traps on decode.
8. The `BitcoinGetSuccessors` context is permanently gone; `bitcoin_requests_iter` will never re-emit `callback_id = X`; no honest proposer can deliver the real response.

Note: `GetSuccessorsReject` (L77–88) and `SendTransactionReject` (L90–102) branches also lack callback_id validation, enabling symmetric mismatches. [6](#0-5) 

## Impact Explanation

The concrete impact is permanent loss of a `BitcoinGetSuccessors` request context and stalling of UTXO chain-tip synchronization for the bitcoin canister. Because `bitcoin_requests_iter` iterates only over live contexts, the lost slot is never re-queued. This directly disrupts Chain Fusion / ckBTC UTXO indexing, which is an in-scope Chain Fusion security impact with concrete protocol harm. This maps to **High ($2,000–$10,000): Significant Chain Fusion, ck-token, ledger, or infrastructure security impact with concrete user or protocol harm**, and also to **High: Application/platform-level DoS or certified-state disruption not based on raw volumetric DDoS**. The claim of false ckBTC minting is not supported by this path alone and is correctly disclaimed by the submitter.

## Likelihood Explanation

Requires a single Byzantine subnet node below the consensus fault threshold, which is explicitly in-scope per the ICP bounty attacker model. The attack is deterministic: one crafted block proposal suffices. No special privileges, no victim interaction, no external dependencies. Repeatability is high; the attacker can target any live `GetSuccessors` context visible in the certified state.

## Recommendation

1. In `validate_self_validating_payload_impl`, load the certified state at `validation_context.certified_height` and for each `BitcoinAdapterResponse` in the payload, verify that `callback_id` exists in the correct context map (`bitcoin_get_successors_contexts` for `GetSuccessorsResponse`/`GetSuccessorsReject`; `bitcoin_send_transaction_internal_contexts` for `SendTransactionResponse`/`SendTransactionReject`). Return `InvalidArtifact` on any mismatch or unknown `callback_id`.

2. In `push_response` for `SendTransactionResponse` (and `SendTransactionReject`), add the same guard present in the `GetSuccessorsResponse` branch: look up `callback_id` in `bitcoin_send_transaction_internal_contexts` and return `StateError::BitcoinNonMatchingResponse` if absent.

3. Apply the symmetric fix to `GetSuccessorsReject`: verify `callback_id` exists in `bitcoin_get_successors_contexts` before enqueuing.

## Proof of Concept

```rust
// Setup: insert a BitcoinGetSuccessors context at callback_id=0
state.metadata.subnet_call_context_manager.push_context(
    SubnetCallContext::BitcoinGetSuccessors(BitcoinGetSuccessorsContext { ... }),
);

// Byzantine push_response: SendTransactionResponse with callback_id=0
let result = push_response(&mut state, BitcoinAdapterResponse {
    response: BitcoinAdapterResponseWrapper::SendTransactionResponse(
        SendTransactionResponse {}
    ),
    callback_id: 0,
});

// Passes with Ok(()) — no error returned
assert!(result.is_ok());

// consensus_queue now contains EmptyBlob for a GetSuccessors callback_id
assert_eq!(state.consensus_queue[0].payload, Payload::Data(EmptyBlob.encode()));

// GetSuccessors context is still present (not removed by push_response),
// but will be permanently removed by retrieve_context during execution processing
assert!(state.metadata.subnet_call_context_manager
    .bitcoin_get_successors_contexts.contains_key(&CallbackId::from(0)));

// Validate path: validate_self_validating_payload_impl also passes for this payload
let svp = SelfValidatingPayload::new(vec![BitcoinAdapterResponse {
    response: BitcoinAdapterResponseWrapper::SendTransactionResponse(SendTransactionResponse {}),
    callback_id: 0,
}]);
assert!(builder.validate_self_validating_payload_impl(&svp, &ctx).is_ok());
```
