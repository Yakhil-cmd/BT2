### Title
If both `pool_member` and `reward_address` are smart contracts unable to call `claim_rewards`, pool delegation rewards are permanently frozen — (`src/pool/pool.cairo`)

### Summary

`Pool::claim_rewards` restricts callers to either `pool_member` or `reward_address`. When both are smart contracts that lack the interface to invoke `claim_rewards`, accumulated STRK rewards are permanently unclaimable. The same pattern exists in `Staking::claim_rewards` for staker rewards.

### Finding Description

In `pool.cairo`, `claim_rewards` enforces a two-address access control:

```cairo
fn claim_rewards(ref self: ContractState, pool_member: ContractAddress) -> Amount {
    let caller_address = get_caller_address();
    let reward_address = pool_member_info.reward_address;
    assert!(
        caller_address == pool_member || caller_address == reward_address,
        "{}",
        Error::POOL_CLAIM_REWARDS_FROM_UNAUTHORIZED_ADDRESS,
    );
    ...
    reward_token.checked_transfer(recipient: reward_address, amount: rewards.into());
``` [1](#0-0) 

Rewards are always transferred to `reward_address`, not to the caller. [2](#0-1) 

The identical pattern exists in the staking contract:

```cairo
assert!(
    caller_address == staker_address || caller_address == reward_address,
    "{}",
    Error::CLAIM_REWARDS_FROM_UNAUTHORIZED_ADDRESS,
);
``` [3](#0-2) 

**Root cause**: No third-party or permissionless path exists to trigger reward delivery to `reward_address`. If `pool_member` is a smart contract (e.g., a DAO treasury, multisig, or protocol contract) that does not implement a call to `Pool::claim_rewards`, and `reward_address` is also a smart contract without that interface, the rewards accumulate in the pool contract's accounting but can never be transferred out.

The `change_reward_address` function in the pool contract is callable only by `pool_member` itself. If `pool_member` cannot call any pool function, the reward address cannot be rotated to an EOA as a recovery path. [1](#0-0) 

### Impact Explanation

Accumulated STRK delegation rewards are permanently frozen. The `reward_address` balance never increases, and there is no admin override or permissionless sweep path. This matches the allowed impact: **High — Permanent freezing of unclaimed yield**.

### Likelihood Explanation

Smart contract participation in staking is a standard and expected use case (DAOs, liquid staking protocols, multisigs). It is realistic for a protocol to:
1. Enter the delegation pool from a contract address (`pool_member` = contract).
2. Set a separate treasury or distribution contract as `reward_address`.
3. Neither contract implements a call to `Pool::claim_rewards(pool_member)`.

This requires two conditions (both addresses are contracts without the interface), which is less likely than the single-condition original report, but remains a realistic deployment scenario for institutional or protocol-level delegators.

### Recommendation

Add a permissionless `claim_rewards` path that transfers rewards directly to `reward_address` without requiring the caller to be `pool_member` or `reward_address`. Since rewards are always sent to `reward_address` regardless of who triggers the call, there is no security benefit to restricting the caller — any address should be allowed to trigger the transfer on behalf of `reward_address`:

```cairo
fn claim_rewards(ref self: ContractState, pool_member: ContractAddress) -> Amount {
    // Remove the caller restriction entirely, or allow any caller.
    // Rewards are always sent to reward_address, so no privilege is granted to the caller.
    ...
    reward_token.checked_transfer(recipient: reward_address, amount: rewards.into());
```

The same fix applies to `Staking::claim_rewards`. [4](#0-3) 

### Proof of Concept

1. Deploy a smart contract `ContractA` that has no method to call `Pool::claim_rewards`.
2. Deploy a smart contract `ContractB` (reward distributor) that also has no method to call `Pool::claim_rewards`.
3. From `ContractA`, call `Pool::enter_delegation_pool(reward_address: ContractB, amount: X)`.
4. Advance epochs so rewards accrue.
5. Attempt to call `Pool::claim_rewards(pool_member: ContractA)` from any EOA — reverts with `POOL_CLAIM_REWARDS_FROM_UNAUTHORIZED_ADDRESS`.
6. Attempt from `ContractA` — not possible (no interface).
7. Attempt from `ContractB` — not possible (no interface).
8. Rewards are permanently frozen. [5](#0-4)

        fn claim_rewards(ref self: ContractState, pool_member: ContractAddress) -> Amount {
            // Asserts.
            let mut pool_member_info = self.internal_pool_member_info(:pool_member);
            let caller_address = get_caller_address();
            let reward_address = pool_member_info.reward_address;
            assert!(
                caller_address == pool_member || caller_address == reward_address,
                "{}",
                Error::POOL_CLAIM_REWARDS_FROM_UNAUTHORIZED_ADDRESS,
            );

            let until_checkpoint = self.get_current_checkpoint(:pool_member);

            // Calculate rewards and update entry_to_claim_from.
            let (mut rewards, updated_entry_to_claim_from) = self
                .calculate_rewards(
                    :pool_member,
                    from_checkpoint: pool_member_info.reward_checkpoint,
                    :until_checkpoint,
                    entry_to_claim_from: pool_member_info.entry_to_claim_from,
                );
            rewards += pool_member_info._unclaimed_rewards_from_v0;
            pool_member_info._unclaimed_rewards_from_v0 = Zero::zero();
            pool_member_info.entry_to_claim_from = updated_entry_to_claim_from;
            pool_member_info.reward_checkpoint = until_checkpoint;

            // Write the updated pool member info to storage.
            self.write_pool_member_info(:pool_member, :pool_member_info);

            // Transfer rewards to the pool member.
            let reward_token = IERC20Dispatcher { contract_address: STRK_TOKEN_ADDRESS };
            reward_token.checked_transfer(recipient: reward_address, amount: rewards.into());

            // Emit event.
            self
                .emit(
                    Events::PoolMemberRewardClaimed {
                        pool_member, reward_address, amount: rewards,
                    },
                );

            rewards
        }
