### Title
Missing Zero-Address Validation for `reward_address` Causes Permanent Freezing of Unclaimed Yield - (File: `src/staking/staking.cairo`, `src/pool/pool.cairo`)

### Summary
The `stake()` and `change_reward_address()` functions in `src/staking/staking.cairo`, and `change_reward_address()` in `src/pool/pool.cairo`, validate that `reward_address` is not a registered token address, but perform no check that it is non-zero. This is the direct analog of the reported incomplete regex validation: both cases share the same root cause class — a validation guard that covers one invalid input class (token address) while silently accepting another (zero address). Accepting the zero address as a reward destination causes all accumulated unclaimed yield to be permanently burned when rewards are disbursed.

### Finding Description
In `src/staking/staking.cairo`, `stake()` validates:

```cairo
assert!(
    !self.does_token_exist(token_address: reward_address),
    "{}",
    GenericError::REWARD_ADDRESS_IS_TOKEN,
);
```

and `change_reward_address()` applies the identical single-condition guard:

```cairo
assert!(
    !self.does_token_exist(token_address: reward_address),
    "{}",
    GenericError::REWARD_ADDRESS_IS_TOKEN,
);
```

In `src/pool/pool.cairo`, `change_reward_address()` checks only:

```cairo
assert!(
    self.token_dispatcher.contract_address.read() != reward_address,
    "{}",
    GenericError::REWARD_ADDRESS_IS_TOKEN,
);
```

None of these three entry points assert `reward_address.is_non_zero()`. The codebase already uses this pattern elsewhere — `add_token()` asserts `token_address.is_non_zero()` and `assert_caller_is_not_zero()` exists in `src/staking/utils.cairo` — confirming the omission is inconsistent, not intentional.

When rewards are later disbursed, `send_rewards_to_staker()` unconditionally transfers to whatever address is stored:

```cairo
token_dispatcher.checked_transfer(recipient: reward_address, amount: amount.into());
```

On Starknet, address `0` is not a deployed account; tokens transferred there are permanently inaccessible.

### Impact Explanation
A staker who registers or later sets `reward_address = 0` — whether through a front-end bug, a scripting error, or a misunderstanding — will have every future reward disbursement sent to the zero address. This applies both to rewards claimed via `claim_rewards()` and to the final reward flush inside `unstake_action()`. The yield is permanently frozen with no recovery path. This matches the allowed High impact: **permanent freezing of unclaimed yield**.

### Likelihood Explanation
The likelihood is low-to-medium. The zero address is the default/uninitialized value in many tooling environments and is a common accidental input. Front-end integrations that fail to populate the field, scripts that pass a default `felt252` of `0`, or users who misread documentation could all trigger this path. Because the consequence is irreversible and the validation gap is present at both initial staking and at every subsequent reward-address update, the exposure window is permanent for any affected staker.

### Recommendation
Add an explicit non-zero assertion for `reward_address` in all three entry points:

- `stake()` in `src/staking/staking.cairo` — add `assert!(reward_address.is_non_zero(), "{}", GenericError::ZERO_ADDRESS);` alongside the existing token-address check.
- `change_reward_address()` in `src/staking/staking.cairo` — same addition.
- `change_reward_address()` in `src/pool/pool.cairo` — same addition.

This mirrors the pattern already applied to `token_address` in `add_token()`.

### Proof of Concept

1. Staker calls `stake(reward_address: 0, operational_address: <valid>, amount: <min_stake>)`.
   - The only guard is `!self.does_token_exist(token_address: 0)`. Address `0` is not a registered token, so the assertion passes.
   - Staker record is written with `reward_address = 0`. [1](#0-0) 

2. Over subsequent epochs the staker attests and accumulates `unclaimed_rewards_own > 0`.

3. Anyone calls `claim_rewards(staker_address)`. The authorization check `caller_address == staker_address || caller_address == reward_address` passes when the caller is the staker. [2](#0-1) 

4. `send_rewards_to_staker()` executes `checked_transfer(recipient: 0, amount: rewards)`. The transfer succeeds on Starknet (no protocol-level zero-address rejection on ERC-20 transfers), and the tokens are permanently lost. [3](#0-2) 

5. The same outcome occurs via `change_reward_address(reward_address: 0)` on an existing staker [4](#0-3) , and via `change_reward_address(reward_address: 0)` on a pool member [5](#0-4) , in both cases silently accepting the zero address and routing all future reward transfers to it.
