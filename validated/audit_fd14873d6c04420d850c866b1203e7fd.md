### Title
Pre-EIP-155 Legacy Transactions Bypass Chain ID Validation, Enabling Cross-Chain Replay — (`File: engine/src/engine.rs`)

---

### Summary

The Aurora Engine accepts pre-EIP-155 legacy transactions (those with `v = 27` or `v = 28`) without any chain ID check. Because the chain ID validation in `submit_with_alt_modexp` is guarded by `if let Some(chain_id)`, and legacy transactions without EIP-155 produce `chain_id = None`, the guard is silently skipped. Any transaction signed on Ethereum mainnet (or any other EVM chain) without EIP-155 protection is valid on Aurora and can be replayed by an unprivileged attacker.

---

### Finding Description

In `engine/src/engine.rs`, the `submit_with_alt_modexp` function parses the submitted bytes into a `NormalizedEthTransaction` and then validates the chain ID:

```rust
// engine/src/engine.rs  lines 1054-1059
if let Some(chain_id) = transaction.chain_id
    && U256::from(chain_id) != U256::from_big_endian(&state.chain_id)
{
    return Err(EngineErrorKind::InvalidChainId.into());
}
``` [1](#0-0) 

The guard fires **only when `chain_id` is `Some`**. For legacy transactions the `chain_id` field is populated from `LegacyEthSignedTransaction::chain_id()`:

```rust
// engine-transactions/src/legacy.rs  lines 88-93
pub const fn chain_id(&self) -> Option<u64> {
    match self.v {
        0..=34 => None,
        _ => Some((self.v - 35) / 2),
    }
}
``` [2](#0-1) 

When `v = 27` or `v = 28` (the pre-EIP-155 encoding), `chain_id()` returns `None`. This value flows directly into `NormalizedEthTransaction`:

```rust
// engine-transactions/src/lib.rs  lines 106-118
Legacy(tx) => Self {
    address: tx.sender()?,
    chain_id: tx.chain_id(),   // None for v=27/28
    ...
}
``` [3](#0-2) 

Because `chain_id` is `None`, the `if let Some(chain_id)` branch in `submit_with_alt_modexp` is never entered, and the transaction is accepted and executed regardless of which chain it was originally signed for.

---

### Impact Explanation

An attacker who observes a pre-EIP-155 Ethereum transaction (e.g., from a public mempool or historical chain data) can submit the identical raw bytes to Aurora's `submit` entry point. If the signer's Aurora address holds ETH or ERC-20 tokens, the replayed transaction executes with full effect: ETH transfers drain the victim's balance, contract calls execute arbitrary logic, and token approvals/transfers are processed — all without the victim's knowledge or consent. This constitutes **direct theft of user funds**.

---

### Likelihood Explanation

Pre-EIP-155 transactions are still broadcast on Ethereum mainnet by legacy wallets and certain tooling. They are publicly visible in the mempool and in historical block data. No special privilege is required: any unprivileged actor can copy the raw transaction bytes and call Aurora's `submit` method. The attacker only needs to know that the victim's address holds funds on Aurora.

---

### Recommendation

Reject legacy transactions that do not carry a chain ID (i.e., `v = 27` or `v = 28`). Change the chain ID check to treat `None` as a validation failure rather than a pass:

```rust
// Require chain_id to be present and match
let chain_id = transaction.chain_id
    .ok_or(EngineErrorKind::InvalidChainId)?;
if U256::from(chain_id) != U256::from_big_endian(&state.chain_id) {
    return Err(EngineErrorKind::InvalidChainId.into());
}
```

This ensures every transaction submitted to Aurora must be explicitly bound to Aurora's chain ID, eliminating the replay surface.

---

### Proof of Concept

1. On Ethereum mainnet, broadcast a pre-EIP-155 ETH transfer from address `A` to address `B` (signed with `v = 27`). The raw RLP-encoded bytes are publicly visible.
2. Address `A` also holds ETH on Aurora.
3. Submit the identical raw bytes to Aurora's `submit` NEAR method (the `engine/src/lib.rs` → `submit_with_alt_modexp` path).
4. `EthTransactionKind::try_from` parses it as `EthTransactionKind::Legacy`.
5. `NormalizedEthTransaction::try_from` sets `chain_id: None` because `v = 27`.
6. The `if let Some(chain_id)` guard at `engine/src/engine.rs:1055` is skipped.
7. The transaction executes on Aurora, transferring ETH from `A`'s Aurora balance to `B` — a replay of the Ethereum transaction on a chain the signer never intended. [4](#0-3) [5](#0-4) [6](#0-5)

    /// Returns sender of given signed transaction by doing ecrecover on the signature.
    pub fn sender(&self) -> Result<Address, Error> {
        let mut rlp_stream = RlpStream::new();
        // See details of CHAIN_ID computation here - https://github.com/ethereum/EIPs/blob/master/EIPS/eip-155.md#specification
        let (chain_id, rec_id) = match self.v {
            0..=26 | 29..=34 => return Err(Error::InvalidV),
            27..=28 => (
                None,
                u8::try_from(self.v - 27).map_err(|_e| Error::InvalidV)?,
            ),
            _ => (
                Some((self.v - 35) / 2),
                u8::try_from((self.v - 35) % 2).map_err(|_e| Error::InvalidV)?,
            ),
        };
        self.transaction
            .rlp_append_unsigned(&mut rlp_stream, chain_id);

            .map_err(|_| Error::EcRecover)
