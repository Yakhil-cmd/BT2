### Title
Unbound `old_participants` Input Enables Malicious Coordinator to Permanently Deny Resharing — (`src/dkg.rs`)

### Summary

`do_reshare` accepts `old_participants` as a caller-supplied local input that is never broadcast, never included in the session-ID transcript, and never cross-verified between parties. A malicious coordinator can supply a different `old_participants` list to different honest parties. This causes each party to compute a different Lagrange coefficient for its secret, producing inconsistent polynomial constant terms. The resulting aggregate public key will not match `old_public_key`, triggering `AssertionFailed` at the public-key consistency check — or, in a second failure mode, causing an earlier `MaliciousParticipant` abort — permanently denying resharing to all honest parties.

---

### Finding Description

`do_reshare` computes each party's Lagrange-weighted secret contribution locally using the caller-supplied `old_participants`: [1](#0-0) 

The intersection and Lagrange coefficient are computed purely from the local `old_participants` argument. This value is never committed to the session ID: [2](#0-1) 

The session ID is derived only from the random `my_session_id` values broadcast by each party — `old_participants` is not mixed in. It is also never broadcast or echoed to other parties for cross-verification.

Later, `do_keyshare` uses the same local `old_participants` to classify each peer as "old" or "new" when verifying proofs of knowledge: [3](#0-2) 

Inside `verify_proof_of_knowledge`, if a party's local `old_participants` does not contain participant `p` but `p` sends a proof of knowledge (because it is genuinely old), the verifier immediately aborts: [4](#0-3) 

Even if that early abort is avoided, the aggregate public key check at Round 4 will fail because the Lagrange coefficients are inconsistent across parties: [5](#0-4) 

---

### Impact Explanation

The protocol invariant requires every old participant to compute `f_i(0) = λ_i(I) · x_i` using the **same** intersection set `I`. If party B uses `I = {A,B}` while parties A and C use `I = {A,B,C}`, then:

```
sum_j C_j(0) = λ_A({A,B,C})·x_A·G  +  λ_B({A,B})·x_B·G  +  λ_C({A,B,C})·x_C·G
             ≠ old_public_key
```

All parties compute the same `verifying_key` (from the same broadcast commitments), but it will not equal `old_public_key`, so every party returns `AssertionFailed("new public key does not match old public key")`. Resharing is permanently denied for all honest parties.

---

### Likelihood Explanation

The attack requires only that the coordinator supply a different `old_participants` slice to at least one party's `do_reshare` call. No cryptographic assumption needs to be broken. The library exposes `do_reshare` as a `pub async fn` whose `old_participants` parameter is entirely under the coordinator's control, with no in-protocol mechanism to detect divergence. [6](#0-5) 

---

### Recommendation

Bind `old_participants` (and `old_public_key`) to the session transcript before any cryptographic material is derived. Concretely, include a serialization of `old_participants` in the session-ID hash (alongside the random `sid_i` values), so that parties with different `old_participants` will compute different session IDs and the reliable-broadcast layer will detect the inconsistency. Alternatively, add an explicit broadcast-and-verify round for `old_participants` at the start of `do_reshare`, analogous to how `session_ids` are broadcast in Round 1.

---

### Proof of Concept

Setup: old key generated by `{A, B, C}` with threshold 2.

1. Coordinator starts a reshare to new participants `{A, B, C}`.
2. Coordinator calls `do_reshare` for party A with `old_participants = [A, B, C]`.
3. Coordinator calls `do_reshare` for party B with `old_participants = [A, B]` (C omitted).
4. Coordinator calls `do_reshare` for party C with `old_participants = [A, B, C]`.

Party B computes `λ_B({A,B}) · x_B` instead of `λ_B({A,B,C}) · x_B`. The aggregate commitment sum differs from `old_public_key`. All parties return `ProtocolError::AssertionFailed("new public key does not match old public key")` at line 492.

Additionally, when party B processes C's broadcast commitment (which includes a proof of knowledge because C is genuinely old), `verify_proof_of_knowledge` at line 200 sees that C is not in B's `old_participants = [A,B]` and returns `ProtocolError::MaliciousParticipant(C)` — causing B to abort even earlier.
