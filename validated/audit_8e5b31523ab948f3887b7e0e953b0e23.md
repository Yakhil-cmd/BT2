### Title
Mandatory External EVM Call in `notify_l2_asset_tracker` Treats Any Revert as Fatal, Enabling Block-Processing DoS via L1→L2 Deposits - (File: `basic_bootloader/src/bootloader/transaction_flow/zk/process_l1_transaction.rs`)

---

### Summary

The `notify_l2_asset_tracker` function in the ZKsync OS bootloader makes a mandatory call to the `L2_ASSET_TRACKER_ADDRESS` EVM contract during every L1→L2 transaction that carries a non-zero deposit. If that call reverts for **any** reason, the bootloader immediately returns a fatal `internal_error!` that halts all block processing. This is the direct analog of the Augur C07 vulnerability class: a critical protocol path makes a push-style call to an external contract, and any revert in that contract permanently blocks the critical path.

---

### Finding Description

In `process_l1_transaction`, the bootloader calls `mint_base_token` up to three times per L1→L2 transaction (value mint, operator fee, refund). Each call to `mint_base_token` first calls `notify_l2_asset_tracker`, which executes `handleFinalizeBaseTokenBridgingOnL2(uint256,uint256)` on `L2_ASSET_TRACKER_ADDRESS` (a real EVM contract) via `run_single_interaction`.

The critical code path:

```rust
// notify_l2_asset_tracker — lines 870–912
if amount > U256::ZERO || Config::SIMULATION {
    // ...build calldata...
    let failed = resources.with_infinite_ergs(|inf_ergs| {
        let CompletedExecution { result: asset_tracker_result, .. } =
            BasicBootloader::<S, ZkTransactionFlowOnlyEOA<S>>::run_single_interaction(
                system, system_functions, memories,
                &calldata,
                &L2_BASE_TOKEN_ADDRESS,
                &L2_ASSET_TRACKER_ADDRESS,   // ← external EVM contract
                inf_ergs.clone(),
                &U256::ZERO,
                true,
                tracer, validator,
            )?;
        *inf_ergs = resources_returned;
        Ok::<bool, BootloaderSubsystemError>(asset_tracker_result.failed())
    })?;

    if failed {
        // ← ANY revert → fatal error → block processing halts
        return Err(internal_error!(
            "L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2 reverted"
        ).into());
    }
}
```

The `amount` passed to the asset tracker is derived directly from attacker-controlled L1 transaction fields (`to_mint`, `gas_limit * gas_price`, refund). The `l1_chain_id` is read from `L2_ASSET_TRACKER_ADDRESS` storage slot 154 — if the contract is uninitialized or in an unexpected state, this returns `0`, which may itself trigger a revert inside the asset tracker.

There is **no fallback, no graceful degradation, and no recovery path**. A single revert from `L2_ASSET_TRACKER_ADDRESS` propagates as a fatal bootloader error, aborting the entire block.

---

### Impact Explanation

- **Block-level DoS**: Any L1→L2 transaction with `total_deposited > 0` that causes `L2_ASSET_TRACKER_ADDRESS` to revert will abort block processing entirely. All subsequent transactions in the block are dropped.
- **Permanent freeze of L1→L2 deposits**: If the L2AssetTracker enters a state where it consistently reverts for a given input range (e.g., specific `amount` values, overflow in internal accounting, or an uninitialized `L1_CHAIN_ID` slot returning `0`), no L1→L2 deposit can be processed until the contract is upgraded — but the upgrade itself requires a block to be processed.
- **Funds at risk**: Deposited funds committed on L1 cannot be credited on L2, and the refund path is also blocked (it also calls `notify_l2_asset_tracker`).

---

### Likelihood Explanation

- The L2AssetTracker is an upgradeable EVM proxy contract. Any upgrade that introduces a revert path (even accidentally) immediately triggers this DoS.
- The `l1_chain_id` is read from storage slot 154 of the asset tracker. If the contract is not yet initialized (slot = 0), `fromChainId = 0` is passed. If the asset tracker's `handleFinalizeBaseTokenBridgingOnL2` reverts on `fromChainId == 0`, every L1→L2 deposit before initialization halts block processing.
- An attacker can submit L1→L2 transactions with crafted `to_mint` values to probe for revert conditions in the asset tracker's arithmetic (e.g., overflow in `chainBalance` mapping updates).
- The call is made with `FORMAL_INFINITE` ergs for the operator-fee and refund legs, so gas exhaustion cannot be used as a mitigation.

---

### Recommendation

1. **Do not treat asset tracker reverts as fatal for block processing.** Degrade gracefully: log the failure, skip the asset tracker notification, and continue block processing. Accounting inconsistency is recoverable via a subsequent upgrade; a halted chain is not.
2. **Implement a pull-style pattern**: record the amounts that should be notified to the asset tracker and let a separate, non-fatal reconciliation step process them.
3. **Guard against uninitialized `L1_CHAIN_ID`**: if `read_l1_chain_id` returns `0`, skip the asset tracker call rather than passing `fromChainId = 0` to a contract that may not handle it.
4. **Add a circuit-breaker**: if `L2_ASSET_TRACKER_ADDRESS` has no deployed code, the call already succeeds silently (as noted in the comment). Extend this logic to treat any revert as a non-fatal warning rather than a fatal error.

---

### Proof of Concept

1. Deploy a ZKsync OS chain with `L2_ASSET_TRACKER_ADDRESS` either uninitialized (slot 154 = 0) or upgraded to a version that reverts when `amount` exceeds a threshold.
2. Submit an L1→L2 transaction with `to_mint > 0` (any non-zero deposit):
   ```
   L1TxBuilder::new()
       .from(attacker)
       .to(any_address)
       .gas_price(1000)
       .gas_limit(100_000)
       .value(U256::from(1))
       .to_mint(U256::from(100_001_000))  // triggers asset tracker revert
       .build()
   ```
3. The bootloader calls `notify_l2_asset_tracker` → `run_single_interaction` → `L2_ASSET_TRACKER_ADDRESS.handleFinalizeBaseTokenBridgingOnL2(0, amount)`.
4. The asset tracker reverts.
5. `notify_l2_asset_tracker` returns `Err(internal_error!("L2AssetTracker.handleFinalizeBaseTokenBridgingOnL2 reverted"))`.
6. `process_l1_transaction` propagates the fatal error.
7. Block processing halts; all transactions in the block are dropped; all deposited funds are frozen on L2 until a governance upgrade can fix the asset tracker — but that upgrade itself requires a block to be processed.

fn read_l1_chain_id<S: EthereumLikeTypes>(system: &mut System<S>) -> U256
where
    S::IO: IOSubsystemExt,
{
    // L2AssetTracker storage layout (verified via `forge inspect`):
    //   slots 0-100:   Initializable + OwnableUpgradeable + Ownable2StepUpgradeable
    //   slots 101-150: Ownable2Step __gap
    //   slot 151:      mapping chainBalance
    //   slot 152:      mapping assetMigrationNumber
    //   slot 153:      mapping isAssetRegistered
    //   slot 154:      uint256 L1_CHAIN_ID
    let l1_chain_id_slot = Bytes32::from_u256_be(&U256::from(154));
    let mut inf_resources = S::Resources::FORMAL_INFINITE;
    let chain_id = system
        .io
        .storage_read::<false>(
            ExecutionEnvironmentType::NoEE,
            &mut inf_resources,
            &L2_ASSET_TRACKER_ADDRESS,
            &l1_chain_id_slot,
        )
        .expect("must read L2AssetTracker L1_CHAIN_ID");
    U256::from_be_bytes(chain_id.as_u8_array())
}
