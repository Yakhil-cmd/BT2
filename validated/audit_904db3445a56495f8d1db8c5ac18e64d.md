### Title
Gateway Signature Bypass for Invoke Transactions via Overly Broad `skip_stateful_validations` Check — (File: crates/apollo_gateway/src/stateful_transaction_validator.rs)

### Summary

The `skip_stateful_validations` function in the gateway's stateful transaction validator is designed to skip `__validate__` (signature verification) for an invoke with nonce 1 when a `DeployAccount` is pending for the same account (UX feature). However, the guard it uses — `account_tx_in_pool_or_recent_block` — returns `true` for **any** transaction from that address in the mempool, not specifically a `DeployAccount`. An unprivileged attacker can exploit this by submitting an invoke with nonce 1 carrying an arbitrary/invalid signature for any address whose `DeployAccount` is already in the mempool, bypassing gateway-level signature verification entirely.

### Finding Description

`skip_stateful_validations` in `crates/apollo_gateway/src/stateful_transaction_validator.rs` (lines 429–461) fires when:

```
tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO)
```

and then calls:

```rust
mempool_client.account_tx_in_pool_or_recent_block(tx.sender_address()).await
``` [1](#0-0) 

`account_tx_in_pool_or_recent_block` is implemented as:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [2](#0-1) 

`MempoolState::contains_account` returns `true` if the address appears in the `staged` or `committed` maps — populated by any transaction type, not exclusively `DeployAccount`: [3](#0-2) 

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` is called with `skip_validate = true`, which sets `execution_flags.validate = false` and causes `__validate__` to be skipped entirely: [4](#0-3) 

The code comment itself acknowledges the ambiguity: *"it is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction **or** transactions with future nonces that passed validations."* The second branch of that disjunction is the exploitable gap.

**Attack path:**

1. Victim submits a `DeployAccount` for address X (nonce 0). The gateway admits it; `tx_pool.contains_account(X)` becomes `true`.
2. Attacker submits an `Invoke` with nonce 1 from address X carrying an **invalid/arbitrary signature**.
3. Gateway stateful validation:
   - `validate_nonce`: nonce 1 ≥ account_nonce 0 ✓
   - `validate_by_mempool`: no duplicate nonce ✓
   - `skip_stateful_validations`: `tx.nonce()==1 && account_nonce==0` → calls `account_tx_in_pool_or_recent_block(X)` → `true` (victim's `DeployAccount` is in pool) → returns `true`
4. `run_validate_entry_point` is called with `skip_validate = true` → `__validate__` is **not called**.
5. The attacker's invalid invoke is admitted to the mempool without any signature check.

The attacker can now occupy the nonce-1 slot for address X. When the victim subsequently submits their legitimate nonce-1 invoke, the mempool rejects it as a duplicate nonce. The victim must use fee escalation to displace the attacker's transaction, and the attacker can repeat the attack for free (failed `__validate__` at execution time charges no fees).

### Impact Explanation

The gateway admits an invalid (unsigned) Starknet invoke transaction to the mempool, breaking the invariant that every admitted invoke must pass `__validate__`. Concretely:

- **Mempool pollution**: The attacker can flood the mempool with signature-less invokes targeting any address with a pending `DeployAccount`.
- **DoS of legitimate nonce-1 invokes**: The attacker occupies the nonce-1 slot for the victim's not-yet-deployed account. The victim's legitimate invoke is rejected as a duplicate nonce and must be resubmitted with a higher fee.
- **Zero-cost attack**: Because `__validate__` fails at batcher execution time, no fees are charged to the attacker, making the attack economically free and repeatable.

This matches: **High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
