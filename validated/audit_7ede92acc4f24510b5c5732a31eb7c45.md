### Title
Indefinitely Valid COA Ownership Proof Signatures Enable Cross-Context Replay — (`fvm/evm/stdlib/contract.cdc`)

### Summary

`EVM.validateCOAOwnershipProof` and its EVM-side precompile `verifyCOAOwnershipProof` accept signatures over arbitrary `signedData` with no protocol-enforced deadline, nonce, or expiry. Once a Cadence account holder signs a hash for one ERC-1271 context, that signature remains cryptographically valid indefinitely and can be replayed by any attacker in a different ERC-1271 context that presents the same `signedData`. The code itself acknowledges the replay concern in a comment but leaves enforcement entirely to callers.

### Finding Description

`EVM.validateCOAOwnershipProof` in `fvm/evm/stdlib/contract.cdc` verifies that a set of Cadence account key signatures covers `signedData` and that the referenced COA matches `evmAddress`. The function signature is:

```cadence
fun validateCOAOwnershipProof(
    address: Address,
    path: PublicPath,
    signedData: [UInt8],
    keyIndices: [UInt64],
    signatures: [[UInt8]],
    evmAddress: [UInt8; 20]
): ValidationResult
``` [1](#0-0) 

The function verifies the signatures cryptographically and checks the COA address, but imposes **no deadline, no nonce, and no expiry** on `signedData`. The code comment at line 1003 explicitly acknowledges the consequence:

> "In principle, a signature produced for one purpose could be replayed here against a different COA owned by the same Cadence account." [2](#0-1) 

This function is exposed to any EVM contract via the Cadence Arch precompile `verifyCOAOwnershipProof(address,bytes32,bytes)`: [3](#0-2) 

The COA contract's ERC-1271 implementation calls this precompile directly, passing the caller-supplied `_hash` as `signedData`:

```solidity
function isValidSignature(bytes32 _hash, bytes memory _sig) external view virtual returns (bytes4) {
    (bool ok, bytes memory data) = cadenceArch.staticcall(
        abi.encodeWithSignature("verifyCOAOwnershipProof(address,bytes32,bytes)", address(this), _hash, _sig)
    );
    ...
}
``` [4](#0-3) 

**Exploit path:**

1. Victim signs `hash_A = keccak256(abi.encode(action_A))` to authorize action A in EVM contract X. The resulting COA ownership proof (Cadence address, capability path, key indices, signatures over `hash_A`) is submitted on-chain and becomes publicly observable.
2. EVM contract Y uses `isValidSignature` for a different purpose but happens to present the same `hash_A` (e.g., a different action encoded to the same bytes, or a deliberately crafted call).
3. Attacker calls `contractY.someFunction()` which internally calls `contractY.isValidSignature(hash_A, capturedProof)`.
4. The precompile calls `validateCOAOwnershipProof` with the captured proof. The function verifies the signatures over `hash_A` (valid — the key is not revoked and the signature is cryptographically correct), checks the COA address (valid — the proof references the victim's COA), and returns `isValid: true`.
5. Contract Y treats this as a valid authorization from the victim, executing an action the victim never intended.

There is no protocol-level mechanism to invalidate a specific `signedData` signature short of revoking the entire Cadence account key, which invalidates all future transactions from that key.

### Impact Explanation

An attacker who captures a COA ownership proof signature (which is public once submitted on-chain) can replay it indefinitely in any EVM contract that uses `isValidSignature` (ERC-1271) with a matching `signedData` hash. This enables unauthorized authorization of on-chain actions — including token transfers, NFT approvals, or any other ERC-1271-gated operation — without the victim's consent and without the victim having any protocol-level recourse to invalidate the specific signature.

### Likelihood Explanation

The attack is realistic in a DeFi ecosystem where multiple EVM contracts deployed on Flow EVM use ERC-1271 for authorization. The attacker needs only to:
- Observe a victim's COA ownership proof on-chain (no privileged access required),
- Find or construct a contract that calls `isValidSignature` with the same `signedData` hash for a different purpose.

The code comment at line 1003–1009 explicitly acknowledges the replay concern, confirming the developers are aware the protocol does not prevent it. The mitigation is delegated to application-layer callers, but the precompile and Cadence function impose no enforcement.

### Recommendation

Add a protocol-enforced deadline or nonce field to the signed payload within `validateCOAOwnershipProof`. Specifically:

1. Require that `signedData` encodes a structured message including at minimum: the `evmAddress`, a nonce (or block-height-based expiry), and a domain separator. Reject proofs where the encoded expiry has passed.
2. Alternatively, maintain an on-chain nonce per COA that is incremented on each successful proof verification, so a captured proof cannot be replayed.
3. At minimum, enforce that `signedData` encodes `evmAddress` at the protocol level (not just as a caller recommendation), closing the cross-COA replay vector acknowledged in the existing comment.
