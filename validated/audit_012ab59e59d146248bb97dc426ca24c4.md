### Title
Silent NEAR Promise Failure in `ExitToNear` Precompile Burns Tokens Without Refund When `error_refund` Feature Is Disabled - (`engine-precompiles/src/native.rs`)

---

### Summary

The `ExitToNear` precompile (and separately, `ExitToEthereum`) dispatches a fire-and-forget NEAR promise to transfer tokens. When the `error_refund` feature is not compiled in (it is **not** a default feature), no callback is attached to detect promise failure. If the outbound NEAR `ft_transfer` call fails — for example because the recipient account is not registered with the NEP-141 contract — the EVM-side token burn is already committed and the user's funds are permanently destroyed with no refund.

---

### Finding Description

In `ExitToNear::run()`, after computing `callback_args`, the code decides whether to attach a callback:

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)          // fire-and-forget
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs { ... })  // has callback
};
``` [1](#0-0) 

`callback_args` equals `default()` (i.e., `{ refund: None, transfer_near: None }`) whenever:

- The `error_refund` feature is **not** compiled in (so `refund` is always `None`), **and**
- The exit is not a wNEAR unwrap (so `transfer_near` is `None`). [2](#0-1) 

This covers the two most common exit paths: ETH base-token exit (flag `0x00`) and ERC-20 token exit (flag `0x01`, non-wNEAR). In both cases the EVM-side burn is committed atomically with the EVM transaction, but the NEAR promise to `ft_transfer` is asynchronous and unmonitored.

The `error_refund` feature is **not** in the `default` feature set of either crate: [3](#0-2) [4](#0-3) 

The `ExitToEthereum` precompile has the same structural problem but with **no feature-flag mitigation at all** — it unconditionally creates a `PromiseArgs::Create` with no callback: [5](#0-4) 

---

### Impact Explanation

When the outbound `ft_transfer` NEAR promise fails (e.g., recipient not registered with the NEP-141 contract), the ERC-20 tokens or ETH that were burned on the EVM side are **permanently destroyed**. There is no callback to detect the failure and re-mint or refund the tokens. The existing test suite explicitly acknowledges this outcome:

```rust
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
``` [6](#0-5) [7](#0-6) 

Impact: **Critical — permanent freezing/destruction of user funds.**

---

### Likelihood Explanation

The failure condition is directly user-controllable. NEAR NEP-141 contracts require recipients to call `storage_deposit` before they can receive tokens. Any user who calls `ExitToNear` with a recipient account that has not registered with the target NEP-141 contract will trigger a failed `ft_transfer` promise. The user loses their ERC-20 tokens (already burned) and receives nothing on NEAR. No admin action or external oracle is required — the attacker only needs to supply an unregistered `receiver_id`.

The entry path is:
1. Unprivileged EVM user calls the `ExitToNear` precompile address (`0xe9217bc7...`) with flag `0x01` (ERC-20 exit) and a recipient account not registered with the NEP-141 contract.
2. The EVM executes the precompile: ERC-20 tokens are burned from the caller's balance.
3. A fire-and-forget NEAR promise to `ft_transfer` is scheduled.
4. The promise fails on NEAR (recipient not registered).
5. No callback exists; the failure is never observed by Aurora Engine.
6. Tokens are permanently lost. [8](#0-7) 

---

### Recommendation

1. **Enable `error_refund` as a default feature**, or unconditionally attach the `exit_to_near_precompile_callback` callback for all exit paths, not only when `error_refund` is compiled in.
2. **Add a callback to `ExitToEthereum`** analogous to the one in `ExitToNear`, so that a failed `withdraw` promise can trigger a re-mint of the burned tokens.
3. Validate that the recipient account is registered with the NEP-141 contract before burning EVM-side tokens, or make the burn conditional on promise success (which requires a two-step flow with a callback).

---

### Proof of Concept

The existing test `test_exit_to_near_refund` (compiled without `error_refund`) already demonstrates the loss:

```rust
exit_to_near(
    &ft_owner,
    "unregistered.near",   // not registered → ft_transfer fails
    FT_EXIT_AMOUNT,
    &erc20,
    &aurora,
).await.unwrap();

// ERC-20 balance is reduced by FT_EXIT_AMOUNT with no refund:
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
assert_eq!(erc20_balance(&erc20, ft_owner_address, &aurora).await, balance);
``` [9](#0-8) 

The tokens are burned on the EVM side and the NEP-141 balance of `aurora` is unchanged — the funds are gone.
