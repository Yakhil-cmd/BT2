### Title
Missing Token Activity Check Allows Delegation Into Disabled-Token Pools — (`src/staking/staking.cairo`, `src/pool/pool.cairo`)

---

### Summary

The `set_open_for_delegation` function checks only that a token *exists* (`does_token_exist`), not that it is *active* (`is_active_token`). Consequently, a staker can open a delegation pool for a disabled BTC token. Once the pool exists, `enter_delegation_pool` and `add_to_delegation_pool` in the pool contract also perform no token-activity check, so delegators can deposit funds into a pool whose token is disabled. Disabled tokens contribute zero staking power, so those delegators earn no rewards while their funds are locked for the full exit-wait window.

---

### Finding Description

The system supports multiple tokens (STRK and BTC variants). Tokens can be disabled via `disable_token` (callable by the security agent). After disabling, the token's staking power is excluded from `get_current_total_staking_power`, so any stake in a disabled-token pool earns zero rewards.

**Gap 1 — `set_open_for_delegation` (staking.cairo)**

```
fn set_open_for_delegation(ref self: ContractState, token_address: ContractAddress) -> ContractAddress {
    self.general_prerequisites();
    ...
    assert!(self.does_token_exist(:token_address), "{}", Error::TOKEN_NOT_EXISTS);  // ← only existence
    // ← NO is_active_token check
    ...
}
```

`does_token_exist` returns `true` for any registered token regardless of its active/disabled status. There is no call to `is_active_token`. The existing test `test_set_open_for_delegation_with_disabled_btc_token` explicitly confirms this succeeds without error.

**Gap 2 — `enter_delegation_pool` / `add_to_delegation_pool` (pool.cairo)**

```
fn enter_delegation_pool(ref self: ContractState, reward_address: ContractAddress, amount: Amount) {
    self.assert_staker_is_active();   // ← only checks staker removal, not token status
    // ← NO token-activity check
    ...
    self.transfer_to_staking_contract(:amount, :token_dispatcher, :staker_address);
}
```

`assert_staker_is_active` only checks the `staker_removed` flag. Neither `enter_delegation_pool` nor `add_to_delegation_pool` queries the staking contract to verify the pool's token is currently active.

**Gap 3 — `add_stake_from_pool` (staking.cairo)**

```
fn add_stake_from_pool(ref self: ContractState, staker_address: ContractAddress, amount: Amount) {
    self.general_prerequisites();
    let staker_info = self.internal_staker_info(:staker_address);
    assert!(staker_info.unstake_time.is_none(), "{}", Error::UNSTAKE_IN_PROGRESS);
    let pool_contract = get_caller_address();
    let token_address = self.staker_pool_info.entry(staker_address)
        .get_pool_token(:pool_contract).expect_with_err(Error::CALLER_IS_NOT_POOL_CONTRACT);
    // ← NO is_active_token check
    ...
    self.add_to_total_stake(:token_address, amount: normalized_amount);
}
```

Funds are accepted and added to the total stake for the disabled token with no activity guard.

By contrast, `get_total_stake_for_token` *does* enforce the active check:

```
assert!(self.is_active_token(:token_address, epoch_id: curr_epoch), "{}", Error::TOKEN_NOT_ACTIVE);
```

This inconsistency is the direct analog of the external report's pattern: validity is enforced in some read paths but not in the write/entry paths.

---

### Impact Explanation

A delegator who enters a pool backed by a disabled token:

1. Transfers real funds to the staking contract (irreversible until the exit window expires).
2. Earns **zero rewards** for the entire staking period because the disabled token contributes zero staking power to `get_current_total_staking_power`.
3. Must wait the full `DEFAULT_EXIT_WAIT_WINDOW` (one week) before recovering funds.

This constitutes **temporary freezing of funds** and **theft of unclaimed yield** — both within the allowed impact scope.

A malicious staker can deliberately open a pool for a disabled token and advertise it to attract delegators, causing them to lock funds with no yield. Alternatively, a token disabled after a pool is created allows new delegators to keep entering the now-worthless pool.

---

### Likelihood Explanation

- The security agent can disable a token at any time. After disabling, the K-epoch delay means the token is still shown as active for up to K epochs, during which new delegators may enter.
- A malicious staker can open a pool for a token that was added but never enabled (disabled by default at `add_token`), then attract delegators.
- No privileged access is required from the attacker's side: any registered staker can call `set_open_for_delegation`, and any address can call `enter_delegation_pool`.
- The existing test `test_set_open_for_delegation_with_disabled_btc_token` demonstrates the path is reachable and produces no error.

---

### Recommendation

Add an `is_active_token` check in `set_open_for_delegation`:

```cairo
assert!(
    self.is_active_token(:token_address, epoch_id: self.get_current_epoch()),
    "{}",
    Error::TOKEN_NOT_ACTIVE,
);
```

Additionally, add the same check in `add_stake_from_pool` (or propagate it through `enter_delegation_pool` / `add_to_delegation_pool` by querying the staking contract for token status before transferring funds).

---

### Proof of Concept

1. Security agent calls `disable_token(btc_token)`. Token becomes inactive after K epochs.
2. Before K epochs pass (token still technically active per `is_btc_active`), or using a token that was `add_token`-ed but never `enable_token`-ed:
   - Staker calls `set_open_for_delegation(token_address: btc_token)` → succeeds (no active check). [1](#0-0) 
3. Delegator calls `enter_delegation_pool(reward_address, amount)` on the new pool → succeeds. [2](#0-1) 
4. `add_stake_from_pool` is called internally — no token-activity check, funds transferred. [3](#0-2) 
5. After K epochs the token is fully disabled. `get_current_total_staking_power` returns zero for BTC, so the delegator earns zero rewards. [4](#0-3) 
6. Delegator must wait `DEFAULT_EXIT_WAIT_WINDOW` to recover funds with no yield earned.

The test `test_set_open_for_delegation_with_disabled_btc_token` already confirms step 2 succeeds without any panic. [5](#0-4)

        fn add_stake_from_pool(
            ref self: ContractState, staker_address: ContractAddress, amount: Amount,
        ) {
            // Prerequisites and asserts.
            self.general_prerequisites();
            let staker_info = self.internal_staker_info(:staker_address);
            assert!(staker_info.unstake_time.is_none(), "{}", Error::UNSTAKE_IN_PROGRESS);
            let pool_contract = get_caller_address();
            let token_address = self
                .staker_pool_info
                .entry(staker_address)
                .get_pool_token(:pool_contract)
                .expect_with_err(Error::CALLER_IS_NOT_POOL_CONTRACT);
            let decimals = self.get_token_decimals(:token_address);
            let normalized_amount = NormalizedAmountTrait::from_native_amount(:amount, :decimals);

            // Update the staker's staked amount, and add to total_stake.
            let old_delegated_stake = self.get_delegated_balance(:staker_address, :pool_contract);
            let new_delegated_stake = old_delegated_stake + normalized_amount;
            self
                .insert_staker_delegated_balance(
                    :staker_address, :pool_contract, delegated_balance: new_delegated_stake,
                );
            self.add_to_total_stake(:token_address, amount: normalized_amount);

            // Transfer funds from the pool contract to the staking contract.
            // Sufficient approval is a pre-condition.
            let token_dispatcher = IERC20Dispatcher { contract_address: token_address };
            token_dispatcher
                .checked_transfer_from(
                    sender: pool_contract, recipient: get_contract_address(), amount: amount.into(),
                );

            // Emit event.
            self
                .emit(
                    Events::StakeDelegatedBalanceChanged {
                        staker_address,
                        token_address,
                        old_delegated_stake: old_delegated_stake.to_native_amount(:decimals),
                        new_delegated_stake: new_delegated_stake.to_native_amount(:decimals),
                    },
                );
        }

fn test_set_open_for_delegation_with_disabled_btc_token() {
    let mut cfg: StakingInitConfig = Default::default();

    let staker_address = cfg.test_info.staker_address;
    let commission = cfg.staker_info._deprecated_get_pool_info()._deprecated_commission;
    let disabled_btc_token_address = deploy_mock_erc20_decimals_contract(
        initial_supply: cfg.test_info.initial_supply,
        owner_address: cfg.test_info.owner_address,
        name: BTC_TOKEN_NAME_2(),
        decimals: constants::TEST_BTC_DECIMALS,

    assert!(staker_info.pool_info.is_none());
    let staker_pool_info = staking_dispatcher.staker_pool_info(:staker_address);
    assert!(staker_pool_info.pools.len() == 1);
    let expected_pool_info = PoolInfo {
        pool_contract, token_address: disabled_btc_token_address, amount: Zero::zero(),
    };
    assert!(*staker_pool_info.pools[0] == expected_pool_info);
}
