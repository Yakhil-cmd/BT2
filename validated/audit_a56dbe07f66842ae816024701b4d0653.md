### Title
`skip_stateful_validations` Admits Unsigned Invoke Transactions to Mempool by Checking Account Presence Instead of Deploy-Account Type — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The gateway's UX feature that allows a `deploy_account + invoke` bundle to be submitted atomically contains a missing authorization check that is the direct Sequencer analog of the ERC4626 `withdraw` bug. The `skip_stateful_validations` function skips the `__validate__` entry point (the account's signature check) for any invoke transaction with nonce=1 when the sender address is already known to the mempool — regardless of whether the caller is the account owner. Any third party who observes a victim's `deploy_account` in the mempool can front-run it with an unsigned invoke transaction for the victim's address, bypassing signature validation entirely and occupying the victim's nonce=1 slot.

---

### Finding Description

`skip_stateful_validations` is called inside `run_pre_validation_checks` to decide whether to set `execution_flags.validate = false` before running the blockifier's `StatefulValidator`: [1](#0-0) 

When `skip_validate` is `true`, `run_validate_entry_point` sets `validate: !skip_validate` — i.e., `false`: [2](#0-1) 

Inside `StatefulValidator::perform_validations`, an invoke transaction with `validate=false` returns `Ok(())` immediately after `perform_pre_validation_stage`, without ever calling `__validate__`: [3](#0-2) 

The condition that triggers the skip is: [4](#0-3) 

The guard used to justify skipping is `account_tx_in_pool_or_recent_block`, which checks only whether the address appears anywhere in the mempool pool or state — it does **not** verify that the existing transaction is a `deploy_account`: [5](#0-4) 

The inline comment claims this is sufficient because the account "either has a deploy_account transaction or transactions with future nonces that passed validations." The second branch is circular: the only nonce-1 invoke that can reach the mempool for an undeployed account is one that already bypassed validation via this same path. No other nonce (2, 3, …) can pass `__validate__` on a non-existent account. Therefore the only legitimate reason `contains_account` returns `true` for an account with on-chain nonce=0 is that a `deploy_account` is in the pool — but the check does not enforce this type constraint, so an attacker can exploit the window.

**Concrete attack path:**

1. Victim broadcasts `deploy_account` (nonce=0). It enters the mempool and `tx_pool.contains_account(victim_addr)` becomes `true`.
2. Attacker observes the victim's address in the public mempool.
3. Attacker crafts an `InvokeV3` with `sender_address = victim_addr`, `nonce = 1`, arbitrary `calldata`, and a garbage `signature`.
4. Gateway stateless checks pass (signature length is not zero, resource bounds are valid, etc.).
5. `extract_state_nonce_and_run_validations` fetches on-chain nonce = 0.
6. `run_pre_validation_checks` → `validate_by_mempool` passes (no duplicate hash, nonce within gap).
7. `skip_stateful_validations`: `tx.nonce() == 1 && account_nonce == 0 && account_tx_in_pool_or_recent_block(victim_addr) == true` → returns `true`.
8. `run_validate_entry_point` sets `validate = false`; `StatefulValidator::perform_validations` returns `Ok(())` without calling `__validate__`. The garbage signature is never checked.
9. The attacker's unsigned invoke is forwarded to the mempool and occupies the victim's nonce=1 slot.
10. The victim's legitimate nonce=1 invoke is now rejected as a duplicate nonce or must win a fee-escalation race against the attacker's transaction. [6](#0-5) 

---

### Impact Explanation

**High — Mempool/gateway admission accepts invalid transactions before sequencing.**

An attacker can inject an unsigned invoke transaction for any account that has a pending `deploy_account` in the mempool. The invalid transaction occupies the nonce=1 slot, forcing the legitimate owner to either:

- Lose the race entirely (if the attacker's tip is higher), leaving the account stuck at nonce=1 until the attacker's transaction is eventually rejected by the batcher and cleaned up via `commit_block`; or
- Pay an arbitrarily higher tip to displace the attacker's transaction via fee escalation.

Because the attacker's transaction fails `__validate__` during batcher execution (no fee is charged for failed validation), the attacker can repeat this attack at zero cost across every block, sustaining a permanent DoS against any newly deploying account.

---

### Likelihood Explanation

The mempool is public. Any observer can enumerate pending `deploy_account` transactions and their sender addresses. The attack requires no privileged access, no special knowledge beyond the victim's address, and no on-chain funds (the attacker's transaction is never included in a block). The attack window is the time between the victim's `deploy_account` entering the mempool and the victim's nonce=1 invoke being admitted — a race that the attacker can win by monitoring the mempool and submitting immediately.

---

### Recommendation

Replace the type-agnostic `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a `deploy_account` transaction is present in the pool for the sender address. The mempool should expose a dedicated query such as `deploy_account_tx_in_pool(address) -> bool` that inspects the transaction type stored in `tx_pool`. Only if a `DeployAccount` variant is found for the address should the signature skip be granted.

Alternatively, restrict the skip to cases where the gateway itself is the source of the bundle (i.e., the `deploy_account` and the `invoke` arrive in the same RPC call), rather than relying on a mempool state check that any third party can satisfy.

---

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
