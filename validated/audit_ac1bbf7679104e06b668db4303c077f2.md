### Title
Permanent Fund Freeze in `ExitToNear` Precompile When `error_refund` Feature Is Disabled — (`engine-precompiles/src/native.rs`)

---

### Summary

When the Aurora Engine is compiled without the `error_refund` feature flag (the default, as it is absent from the `[features] default` list in `engine/Cargo.toml`), the `ExitToNear` precompile provides **no refund path** if the downstream NEAR `ft_transfer` or `ft_transfer_call` promise fails. ETH deducted from the user's EVM balance is transferred to the precompile address and permanently frozen there; ERC-20 tokens are burned and permanently lost. There is no recovery mechanism.

---

### Finding Description

The `ExitToNear` precompile (`engine-precompiles/src/native.rs`) handles two token exit flows:

**ETH (base token) exit:** The user's EVM ETH balance is debited and credited to the `exit_to_near` precompile address. A NEAR `ft_transfer` promise is then created to send the corresponding NEP-141 ETH to the recipient NEAR account.

**ERC-20 exit:** The ERC-20 tokens are burned from the user's EVM balance. A NEAR `ft_transfer` promise is created to send the corresponding NEP-141 tokens to the recipient.

In both cases, a `callback_args` struct is built:

```rust
// engine-precompiles/src/native.rs, lines 449–455
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,          // ← always None without the feature
    transfer_near: transfer_near_args,
};
```

When `error_refund` is not compiled in, `refund` is unconditionally `None`. The callback promise is only attached when `callback_args` is non-default:

```rust
// engine-precompiles/src/native.rs, lines 470–483
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // ← no callback at all
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs { ... })
};
```

For a plain ETH or ERC-20 exit (no wNEAR unwrap, no OMNI message), `transfer_near` is also `None`, so `callback_args` equals `default()`. The result: **no callback is ever scheduled**, and if the `ft_transfer` promise fails, the engine has no way to detect the failure and re-credit the user.

The `error_refund` feature is not part of the default feature set:

```toml
# engine/Cargo.toml, lines 42–48
[features]
default = ["std"]
...
error_refund = ["aurora-engine-precompiles/error_refund"]
```

The codebase itself acknowledges this behavior in tests:

```rust
// engine-tests/src/tests/erc20_connector.rs, lines 658–660
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
```

And for ETH:

```rust
// engine-tests/src/tests/erc20_connector.rs, lines 773–775
// If the refund feature is not enabled, then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let expected_balance = Wei::new_u64(INITIAL_ETH_BALANCE - ETH_EXIT_AMOUNT);
```

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

- **ETH exits:** The ETH is moved from the user's EVM account to the `exit_to_near` precompile address before the NEAR promise is dispatched. If the promise fails, the ETH sits at the precompile address with no withdrawal function and no refund callback. It is permanently inaccessible.
- **ERC-20 exits:** The tokens are burned from the user's EVM balance before the NEAR promise is dispatched. If the promise fails, the tokens are gone with no re-mint path.

In both cases the loss is permanent and proportional to the amount the user attempted to bridge.

---

### Likelihood Explanation

**Medium.** The NEAR `ft_transfer` promise can fail for several realistic, user-triggerable reasons:

1. The recipient NEAR account is not registered with the NEP-141 token contract (the most common case — any user can supply an unregistered account ID).
2. Aurora's NEP-141 ETH balance is insufficient due to accounting drift.
3. The recipient NEAR account does not exist.

Scenario 1 is entirely within the control of an ordinary user who simply provides an unregistered account ID as the exit destination. No privileged access is required.

---

### Recommendation

Enable the `error_refund` feature in the production build, or — if the feature cannot be enabled — implement an unconditional refund callback that fires regardless of the feature flag. The callback should:

- For ETH exits: transfer the ETH back from the `exit_to_near` precompile address to the original sender (as `refund_on_error` already does in `engine/src/engine.rs` lines 1204–1224 when `error_refund` is enabled).
- For ERC-20 exits: re-mint the burned tokens to the original sender (as `setup_refund_on_error_input` does in `engine/src/engine.rs` lines 1165–1174).

The refund callback must always be attached to the `ft_transfer` promise so that promise failure is observable and recoverable.

---

### Proof of Concept

1. Deploy Aurora Engine **without** the `error_refund` feature (the default build).
2. Fund an EVM address `A` with ETH on Aurora.
3. From address `A`, call the `ExitToNear` precompile with flag `0x00` (ETH exit) and a recipient NEAR account ID that is **not registered** with the NEP-141 ETH contract (e.g., a freshly created account that has never interacted with Aurora).
4. The precompile debits `A`'s EVM balance and schedules `ft_transfer` on the NEP-141 contract.
5. The `ft_transfer` fails because the recipient is unregistered.
6. No callback fires. Address `A`'s EVM balance is permanently reduced. The ETH is now held at the `exit_to_near` precompile address with no recovery path.
7. Confirm: `eth_balance_of(A)` is reduced by the exit amount; the precompile address holds the ETH; no function exists to reclaim it.
