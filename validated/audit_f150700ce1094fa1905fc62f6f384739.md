### Title
Malicious Coordinator Can Corrupt CKD Output by Providing Inconsistent `app_pk` to Participants - (File: `src/confidential_key_derivation/protocol.rs`)

### Summary
The CKD protocol accepts `app_pk: PublicKey` (the client's ElGamal encryption key) as a caller-supplied parameter with no cross-participant consistency check. A malicious coordinator can distribute different `app_pk` values to different participants, causing the aggregated `CKDOutput` to be permanently unusable by the legitimate client.

### Finding Description
In `src/confidential_key_derivation/protocol.rs`, the public entry point `ckd()` accepts `app_pk: PublicKey` as a plain function argument:

```rust
pub fn ckd(
    participants: &[Participant],
    coordinator: Participant,
    me: Participant,
    key_pair: KeygenOutput,
    app_id: impl Into<AppId>,
    app_pk: PublicKey,          // ← caller-supplied, never cross-verified
    rng: impl CryptoRngCore + Send + 'static,
) -> Result<impl Protocol<Output = CKDOutputOption>, InitializationError>
```

This value flows directly into `compute_signature_share()`, where it is used as the ElGamal blinding key:

```rust
// C <- S + y . A
let big_c = big_s + app_pk * y.0;          // line 174
```

The protocol has exactly one communication round: each participant sends `(norm_big_y, norm_big_c)` privately to the coordinator, and the coordinator sums them:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
```

There is **no broadcast step, no commitment, and no transcript binding** that would force all participants to use the same `app_pk`. In a real deployment the coordinator is the entity that distributes `app_pk` to each participant over the network. A malicious coordinator can send `A₁` to participant 1 and `A₂` to participant 2.

The aggregated output then becomes:

```
big_C = Σ λᵢ · (xᵢ · H(pk ‖ app_id) + yᵢ · Aᵢ)
      = msk · H(pk ‖ app_id) + Σ λᵢ · yᵢ · Aᵢ
```

The client holds secret scalar `a` with `A = a·G` and attempts to unmask via `CKDOutput::unmask(a)`:

```rust
pub fn unmask(&self, secret_scalar: Scalar) -> Signature {
    self.big_c - self.big_y * secret_scalar   // expects C − a·Y = msk·H(app_id)
}
```

Because `Aᵢ ≠ a·G` for at least one participant, the result is not `msk · H(pk ‖ app_id)`. The confidential key is permanently unrecoverable for this request.

The same root cause applies to `app_id`: if the coordinator supplies different `app_id` values to different participants, the hash points `H(pk ‖ app_idᵢ)` diverge and the aggregated `big_c` is equally corrupted.

### Impact Explanation
**High.** The CKD output is cryptographically corrupted: honest participants complete the protocol without error, the coordinator receives a well-formed `CKDOutput`, yet the client can never unmask it to obtain `msk · H(pk ‖ app_id)`. This constitutes corruption of a CKD output causing honest parties to accept an unusable cryptographic result, matching the allowed High impact: *"Corruption of … CKD outputs so honest parties accept … unusable cryptographic outputs."*

### Likelihood Explanation
**Low–Medium.** The coordinator is a trusted role in the documented threat model, analogous to the Keeper in the Infrared report. However, unlike the Infrared case there is no offchain computation required: the coordinator need only pass a different `app_pk` byte string to each participant's `ckd()` invocation. No cryptographic capability is needed. Any deployment where the coordinator is a single process that fans out calls to remote participants is directly exposed.

### Recommendation
Add a reliable-broadcast (echo-broadcast) round at the start of the CKD protocol in which every participant commits to and then reveals the `app_pk` (and `app_id`) they received. Any mismatch aborts the protocol before secret material is used. Alternatively, include `app_pk` and `app_id` in a session-ID hash (analogous to the DKG's `session_id` derived from `do_broadcast` in `src/dkg.rs`) so that inconsistent inputs produce divergent session IDs and the protocol self-aborts.

### Proof of Concept
1. Coordinator controls the orchestration layer and calls `ckd()` on behalf of each participant.
2. For participant P₁ the coordinator passes `app_pk = A` (the legitimate client key `a·G`).
3. For participant P₂ the coordinator passes `app_pk = B` (an arbitrary point, e.g. the generator `G`).
4. P₁ computes `big_c₁ = λ₁·(x₁·H(pk‖app_id) + y₁·A)` and sends it to the coordinator.
5. P₂ computes `big_c₂ = λ₂·(x₂·H(pk‖app_id) + y₂·B)` and sends it to the coordinator.
6. Coordinator aggregates: `big_C = big_c₁ + big_c₂ + coordinator_share`.
7. Client calls `ckd_output.unmask(a)` → `big_C − a·big_Y ≠ msk·H(pk‖app_id)`.
8. The confidential key is permanently lost; no error is surfaced to any honest participant. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}
