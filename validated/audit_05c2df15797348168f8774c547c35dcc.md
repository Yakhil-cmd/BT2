### Title
Threshold Parameter Split from `KeygenOutput` Enables Signing Denial via Unchecked Threshold Mismatch - (File: `src/ecdsa/ot_based_ecdsa/mod.rs`, `src/frost/eddsa/sign.rs`, `src/frost/redjubjub/sign.rs`)

---

### Summary

The `threshold` used during DKG is never stored in `KeygenOutput`. Every downstream signing and presigning API accepts `threshold` as a separate, caller-supplied parameter with no validation that it matches the threshold used during keygen. A malicious coordinator or misconfigured caller can supply a mismatched threshold, causing the signing protocol to proceed with cryptographically inconsistent parameters and fail at aggregation — permanently denying signing for honest parties.

---

### Finding Description

`KeygenOutput<C>` stores only `private_share` and `public_key`:

```rust
pub struct KeygenOutput<C: Ciphersuite> {
    pub private_share: SigningShare<C>,
    pub public_key: VerifyingKey<C>,
}
```

The threshold used during keygen is discarded after `do_keyshare` completes. [1](#0-0) 

Every signing and presigning function then accepts `threshold` as a fresh, independent parameter:

- `sign_v1` and `sign_v2` (EdDSA): [2](#0-1) 
- `sign` (RedJubjub): [3](#0-2) 
- OT-based ECDSA `PresignArguments`: [4](#0-3) 

The only validation performed in `assert_sign_inputs` is that `threshold <= participants.len()` — there is no check against the keygen threshold:

```rust
if threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
``` [5](#0-4) 

The library itself acknowledges the requirement in a comment on `PresignArguments.threshold` — *"The desired threshold for the presignature, which must match the original threshold"* — but provides no enforcement: [6](#0-5) 

The caller-supplied `threshold` is then used directly to construct the `KeyPackage` passed to FROST's `round2::sign`:

```rust
fn construct_key_package(threshold: ReconstructionLowerBound, ...) -> Result<KeyPackage, ...> {
    Ok(KeyPackage::new(identifier, signing_share, verifying_share, *verifying_key,
        u16::try_from(threshold.value())...))
}
``` [7](#0-6) [8](#0-7) 

The `min_signers` field in `KeyPackage` controls how many commitments FROST requires in the signing package before computing signature shares. If this value is lower than the actual keygen threshold, FROST will accept a signing package with fewer participants than the polynomial degree requires, compute Lagrange coefficients over an insufficient participant set, and produce signature shares that cannot be aggregated into a valid signature.

---

### Impact Explanation

**High: Permanent denial of signing for honest parties.**

When a malicious coordinator supplies `threshold_sign < threshold_keygen` to `sign_v1`/`sign_v2`/`sign`:

1. The `assert_sign_inputs` check passes (threshold_sign ≤ participants.len()).
2. Each participant builds a `KeyPackage` with `min_signers = threshold_sign`.
3. FROST's `round2::sign` accepts a signing package with only `threshold_sign` commitments.
4. Lagrange coefficients are computed over `threshold_sign` participants — insufficient to reconstruct a degree-`(threshold_keygen - 1)` polynomial.
5. All signature shares are cryptographically wrong; `aggregate()` fails with a `ProtocolError`.
6. Signing is permanently denied for that session.

When `threshold_sign > participants.len()` (but still passes the `<=` check with a crafted participant list), or when different participants are given different threshold values, the signing package's `min_signers` check fails for some participants, producing inconsistent protocol outcomes across the honest party set.

---

### Likelihood Explanation

The coordinator role is explicitly modeled in the library and is a documented trust boundary. The `sign_v1`/`sign_v2` APIs require the caller to supply both `keygen_output` and `threshold` as independent arguments. Any coordinator that reshares keys (changing the threshold) and then passes the old threshold to signing — or any misconfigured integration — triggers this silently. The OT-based ECDSA `PresignArguments` struct makes the split explicit and even documents the requirement without enforcing it, making accidental misconfiguration highly likely in practice.

---

### Recommendation

1. **Store the threshold in `KeygenOutput`**: Add a `threshold: ReconstructionLowerBound` field to `KeygenOutput<C>` so the keygen threshold travels with the key material.

2. **Validate at signing entry points**: In `assert_sign_inputs` (and the equivalent for OT-based ECDSA presign), assert that the caller-supplied threshold equals `keygen_output.threshold`.

3. **Remove the redundant parameter**: Once `threshold` is stored in `KeygenOutput`, remove it as a separate parameter from `sign_v1`, `sign_v2`, `sign` (redjubjub), and `PresignArguments` to eliminate the split entirely — analogous to moving `tournamentBPS`/`otherBPS` into the Claiming contract in the reference report.

---

### Proof of Concept

```rust
// Keygen with threshold=5 among 7 participants
let keygen_outputs = run_keygen(&participants, 5, &mut rng);

// Malicious coordinator supplies threshold=2 to sign_v1
// assert_sign_inputs only checks: 2 <= 7 → passes
let result = sign_v1(
    &participants[..2],   // only 2 participants
    2usize,               // wrong threshold (keygen used 5)
    me,
    coordinator,
    keygen_outputs[me].clone(),
    message,
    rng,
);
// Protocol proceeds, round2::sign executes with min_signers=2
// Lagrange interpolation over 2 points of a degree-4 polynomial → wrong shares
// aggregate() returns ProtocolError → signing permanently denied
```
