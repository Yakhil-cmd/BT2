Audit Report

## Title
Unbounded DKG Unvalidated Pool Growth via Byzantine Subnet Peer — (`rs/artifact_pool/src/dkg_pool.rs`)

## Summary
`DkgPoolImpl::insert` performs an unconditional insertion into an unbounded `BTreeMap` with no capacity check. The DKG P2P channel is configured with `SLOT_TABLE_NO_LIMIT = usize::MAX`, and `DkgBouncer` accepts any message whose `id.height` equals the current DKG interval start height without checking dealer membership or signature validity. A single Byzantine subnet node can flood the unvalidated pool with syntactically valid, height-correct DKG messages carrying distinct dealing payloads, each producing a unique `DkgMessageId`, exhausting replica heap memory before the validator can drain the pool.

## Finding Description

**Root cause — unbounded pool insert:**
`DkgPoolImpl::insert` delegates directly to `PoolSection::insert`, which calls `BTreeMap::insert` with no size guard. [1](#0-0) 
`PoolSection` itself is a plain `BTreeMap` wrapper with no capacity field. [2](#0-1) 

**No per-peer slot limit for DKG:**
`SLOT_TABLE_NO_LIMIT = usize::MAX` is the constant passed to `abortable_broadcast_channel` for the DKG channel. [3](#0-2) [4](#0-3) 
The slot-table guard at `receiver.rs:373` (`peer_slot_table_len < self.slot_limit`) is therefore never triggered for DKG peers. [5](#0-4) 

**`DkgBouncer` checks only height:**
`new_bouncer` returns `BouncerValue::Wants` for any `id` whose `height == start_height`. It does not inspect dealer identity, signature, or dealing content. [6](#0-5) 

**`DkgMessageId` is a full-message crypto hash:**
Every distinct `NiDkgDealing` payload produces a unique `DkgMessageId`, so N distinct payloads occupy N distinct pool entries. [7](#0-6) 

**Validator drains only one message per `(dealer, NiDkgId)` group per cycle:**
`on_state_change` groups unvalidated messages by `(signer, NiDkgId)` and calls `validate_dealings_for_dealer` with the full group, but that function processes only `messages.first()` and returns a single mutation. [8](#0-7) [9](#0-8) 
If the Byzantine node sends faster than the validator removes entries, the pool grows without bound.

**Purge does not cover the current interval:**
`purge` removes only entries with `id.height < height`; messages at the current `start_height` are never purged until the next summary block advances the interval. [10](#0-9) 

**Exploit flow:**
1. Byzantine node (registered subnet dealer) opens N distinct slot numbers toward honest peers (no slot limit for DKG).
2. Each slot advertises a unique `DkgMessageId` with `height == current_start_height` → `DkgBouncer` returns `Wants` for all N.
3. Honest replica fetches and calls `DkgPoolImpl::insert` for all N artifacts → N entries in the unvalidated `BTreeMap`.
4. `on_state_change` removes at most one entry per `(dealer, NiDkgId)` group per cycle.
5. Byzantine node continuously sends new slots faster than the validator drains them → pool grows unboundedly → OOM kill.

## Impact Explanation
A single Byzantine subnet node (one of the f tolerated faults) can exhaust the heap of every honest replica it is connected to. `NiDkgDealing` messages carry large cryptographic blobs; 10^4–10^5 unique messages at the current DKG interval height can grow the unvalidated pool to gigabytes before the next purge cycle. The replica process is killed by the OS OOM killer, halting its participation in consensus. If enough replicas are simultaneously affected, the subnet stalls. This matches the allowed impact: **High — Application/platform-level DoS, consensus blocking, or subnet availability impact not based on raw volumetric DDoS** ($2,000–$10,000).

## Likelihood Explanation
The attacker must be a registered subnet node with a valid TLS certificate — not a fully external party. However, a single compromised node is within the IC Byzantine fault model (f out of 3f+1). No special preconditions beyond being a P2P peer at the current DKG interval are required. The `DkgBouncer` height check is trivially satisfied by using the correct `start_block_height`. The slot table imposes no limit. The attack is mechanically straightforward and repeatable across every DKG interval.

## Recommendation
1. **Add a per-peer slot limit for DKG** analogous to `SLOT_TABLE_LIMIT_INGRESS = 50_000`. Replace `SLOT_TABLE_NO_LIMIT` with a registry-derived constant (e.g., `num_dealers × num_dkg_configs × small_multiplier`) in `rs/replica/setup_ic_network/src/lib.rs`.
2. **Add a capacity check in `DkgPoolImpl::insert`** that rejects insertions beyond a computed bound (e.g., `num_subnet_nodes × num_active_dkg_configs × MAX_DEALINGS_PER_DEALER`).
3. **Extend `DkgBouncer` to check dealer membership** against the current DKG summary before returning `BouncerValue::Wants`, so messages from non-dealers or messages referencing unknown `NiDkgId`s are dropped at the P2P layer before pool insertion.
4. **Drain all messages per group per cycle** in `validate_dealings_for_dealer` rather than only `messages.first()`, or add a per-group cap to bound the backlog.

## Proof of Concept
A deterministic integration test using `DkgPoolImpl` directly:

```rust
#[test]
fn test_dkg_pool_unbounded_growth() {
    let mut pool = DkgPoolImpl::new(MetricsRegistry::new(), no_op_logger(), Height::from(100));
    let bouncer_factory = DkgBouncer::new(&MetricsRegistry::new());
    let bouncer = bouncer_factory.new_bouncer(&pool);

    // Simulate Byzantine node: 100_000 unique messages at current start height
    for i in 0u64..100_000 {
        // Vary the dealing bytes to produce a unique DkgMessageId per message
        let msg = make_message_with_unique_payload(Height::from(100), node_test_id(1), i);
        let id = DkgMessageId::from(&msg);

        // Bouncer accepts all (height == start_height)
        assert_eq!(bouncer(&id), BouncerValue::Wants);

        // No capacity check — insert succeeds unconditionally
        pool.insert(UnvalidatedArtifact { message: msg, peer_id: node_test_id(1), timestamp: UNIX_EPOCH });
    }

    // Pool holds 100_000 entries; validator removes at most 1 per on_state_change cycle
    assert_eq!(pool.get_unvalidated().count(), 100_000);
}
```

The test requires a `make_message_with_unique_payload` helper that varies the `NiDkgDealing` internal bytes by index `i` to produce distinct `crypto_hash` values. No mocking of crypto is needed for the pool-growth assertion — the insert path is entirely in-memory.

impl<K: Ord, V: HasLabel> PoolSection<K, V> {
    pub(crate) fn new(metrics_registry: MetricsRegistry, pool: &str, pool_type: &str) -> Self {
        Self {
            messages: Default::default(),
            metrics: PoolMetrics::new(metrics_registry, pool, pool_type),
        }
    }

    pub(crate) fn insert(&mut self, key: K, value: V) -> Option<V> {
        self.metrics
            .observe_insert(MESSAGE_SIZE_BYTES, value.label());
        let replaced = self.messages.insert(key, value);
        if let Some(replaced) = &replaced {
            self.metrics
                .observe_duplicate(MESSAGE_SIZE_BYTES, replaced.label());
        }
        replaced
    }

impl<Pool: DkgPool> BouncerFactory<DkgMessageId, Pool> for DkgBouncer {
    fn new_bouncer(&self, dkg_pool: &Pool) -> Bouncer<DkgMessageId> {
        let _timer = self.metrics.update_duration.start_timer();

        let start_height = dkg_pool.get_current_start_height();
        Box::new(move |id| {
            use std::cmp::Ordering;
            match id.height.cmp(&start_height) {
                Ordering::Equal => BouncerValue::Wants,
                Ordering::Greater => BouncerValue::MaybeWantsLater,
                Ordering::Less => BouncerValue::Unwanted,
            }
        })
    }

    fn refresh_period(&self) -> std::time::Duration {
        std::time::Duration::from_secs(3)
    }
}
