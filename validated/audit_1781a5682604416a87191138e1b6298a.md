### Title
Malicious CKD Participant Can Use Arbitrary `app_id` to Corrupt Key Derivation Output — (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

In the Confidential Key Derivation (CKD) protocol, each participant locally computes a signature share using a caller-supplied `app_id` parameter and sends the resulting elliptic-curve points `(λ_i·Y_i, λ_i·C_i)` to the coordinator. The coordinator aggregates these shares by simple addition with no verification that every participant used the same `app_id`. A single malicious participant can silently substitute an arbitrary `app_id'` when computing their share, causing the coordinator to assemble a cryptographically invalid CKD output that the requesting application cannot unmask to the correct key.

---

### Finding Description

**Vulnerability class:** Transcript confusion / participant-set parameter mismatch.

**Root cause — `do_ckd_coordinator` in `src/confidential_key_derivation/protocol.rs` lines 35–57:**

```rust
async fn do_ckd_coordinator(
    mut chan: SharedChannel,
    participants: ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,          // coordinator's own app_id
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<CKDOutputOption, ProtocolError> {
    let (mut norm_big_y, mut norm_big_c) =
        compute_signature_share(&participants, me, key_pair, app_id, app_pk, rng)?;

    let waitpoint = chan.next_waitpoint();

    for (_, participant_output) in
        recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
    {
        norm_big_y += participant_output.big_y();   // ← blindly added
        norm_big_c += participant_output.big_c();   // ← blindly added
    }
    let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
    Ok(Some(ckd_output))
}
```

The coordinator receives raw `CKDOutput { big_y, big_c }` values from every other participant and adds them together without any proof that each participant computed their share over the agreed-upon `app_id`.

**What each participant sends — `compute_signature_share` lines 148–181:**

```rust
fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,          // ← each participant supplies their own copy
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    let y = Zeroizing::new(ScalarWrapper(Scalar::random(rng)));
    let big_y = ElementG1::generator() * y.0;
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);  // H(pk, app_id)
    let big_s = hash_point * private_share.to_scalar();                   // x_i · H(pk, app_id)
    let big_c = big_s + app_pk * y.0;                                     // S_i + y_i · A
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    Ok((big_y * lambda_i, big_c * lambda_i))
}
```

There is **no zero-knowledge proof** or any other binding that ties the transmitted `(λ_i·Y_i, λ_i·C_i)` to the correct `app_id`. The `CKDOutput` struct carries only the two curve points:

```rust
pub struct CKDOutput {
    big_y: ElementG1,
    big_c: ElementG1,
}
```

**Exploit path:**

1. A malicious MPC node (participant `m`) receives a legitimate CKD request for `app_id = "victim_app"`.
2. Instead of computing `S_m = x_m · H(pk, "victim_app")`, it computes `S_m = x_m · H(pk, "attacker_app")` (or any arbitrary point).
3. It sends the resulting `(λ_m·Y_m, λ_m·C_m)` to the coordinator.
4. The coordinator adds all shares together. The aggregated `C` is now:

   ```
   C = Σ_{i≠m} λ_i·(x_i·H(pk,"victim_app") + y_i·A)
     + λ_m·(x_m·H(pk,"attacker_app") + y_m·A)
   ```

   This is **not** equal to `msk · H(pk, "victim_app") + a·Y` for any valid `a`.

5. The coordinator returns this corrupted `CKDOutput` to the application.
6. When the app calls `ckd_output.unmask(app_sk)` (`C − a·Y`), the result is not `msk · H(pk, app_id)` and the derived key is cryptographically invalid/unusable.

---

### Impact Explanation

**Severity: High — Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.**

A single malicious participant (below the Byzantine threshold for liveness, but not for this integrity property) can silently corrupt the CKD output for any targeted `app_id`. The application receives a point that does not correspond to any valid BLS signature over `H(pk, app_id)`, so:

- `verify_signature(verifying_key, app_id, &corrupted_sig)` will fail (pairing check in `ciphersuite.rs` lines 239–243 will not hold).
- The app cannot derive its deterministic secret `s = HKDF(sig)`.
- The attack is undetectable by the coordinator or the application without an out-of-band integrity check.
- The attack is repeatable: the malicious node can corrupt every CKD request it participates in.

---

### Likelihood Explanation

**Realistic.** The attacker is a single compromised or malicious MPC node — an explicitly modeled threat in the CKD security assumptions (`docs/confidential_key_derivation/confidential-key-derivation.md` lines 113–120 state that individual nodes must not be able to leak `s` even if malicious, implying malicious nodes are in scope). No privileged access beyond being a protocol participant is required. The attack requires only that the malicious node substitute a different `app_id` locally before calling `compute_signature_share` — a trivial one-line code change in a compromised node.

---

### Recommendation

Each participant must prove to the coordinator that their `(λ_i·Y_i, λ_i·C_i)` was computed using the correct `app_id`. Two concrete options:

1. **Zero-knowledge proof of correct evaluation:** Each participant sends a Chaum-Pedersen-style DLEQ proof showing that `C_i − y_i·A = x_i · H(pk, app_id)` for the public `H(pk, app_id)` that all parties can compute independently. The coordinator verifies each proof before aggregating.

2. **Coordinator-side recomputation of `H(pk, app_id)`:** Since `H(pk, app_id)` is a public, deterministic value, the coordinator can verify the pairing equation `e(C_i − y_i·A, G2) == e(H(pk, app_id), VK_i)` for each participant's share before aggregating, where `VK_i = x_i · G2` is the participant's public verification share (already available from the DKG output).

### Proof of Concept

```rust
// Honest participant computes:
//   hash_point = H(pk, "victim_app")
//   big_s      = x_i * hash_point
//   big_c      = big_s + y_i * app_pk

// Malicious participant instead computes:
//   hash_point_evil = H(pk, "attacker_app")   // different app_id
//   big_s_evil      = x_i * hash_point_evil
//   big_c_evil      = big_s_evil + y_i * app_pk

// Sends (lambda_i * Y_i, lambda_i * big_c_evil) to coordinator.
// Coordinator aggregates without verification:
//   norm_big_c += participant_output.big_c();  // adds corrupted value

// Result: coordinator output C ≠ msk * H(pk, "victim_app") + a*Y
// Application's unmask(app_sk) = C - a*Y ≠ msk * H(pk, "victim_app")
// verify_signature check fails → app cannot derive its key.
```

The attack requires no cryptographic break, no privileged access, and no coordination with other nodes. A single malicious participant is sufficient.
