### Title
Unbounded `stakers` Vec Growth Causes Unbounded Gas Consumption in `get_stakers` - (File: src/staking/staking.cairo)

### Summary
The `stakers` storage Vec in the Staking contract grows permanently with every new staker and is never pruned, even after a staker fully exits via `unstake_action`. The `get_stakers` function iterates over the entire Vec on every call. An unprivileged attacker can bloat this Vec by cycling many addresses through `stake` → `unstake_intent` → `unstake_action`, making `get_stakers` progressively more expensive until it approaches or exceeds execution limits.

### Finding Description
In `src/staking/staking.cairo`, the `stake` function unconditionally appends the caller's address to the `stakers` Vec:

```cairo
// Add staker address to the stakers vector.
self.stakers.push(staker_address);
```

The storage declaration includes an explicit note that this Vec is never cleaned up:

```cairo
/// Vector of staker addresses.
/// **Note**: Stakers are not removed from this vector when they unstake.
stakers: Vec<ContractAddress>,
```

The `get_stakers` function in `StakingConsensusImpl` iterates over the **full range** of this Vec on every invocation:

```cairo
for staker_address_ptr in self.stakers.into_iter_full_range() {
    let staker_address = staker_address_ptr.read();
    if !self.is_staker_active(:staker_address, :epoch_id) {
        continue;
    }
    // ... per-staker storage reads for staking power, public key, peer id
}
```

Each iteration performs multiple storage reads (`is_staker_active`, `get_staker_staking_power_at_epoch`, `get_public_key_at_epoch`, `get_peer_id_at_epoch`). Inactive/exited stakers are skipped via `continue` but still consume a storage read per iteration. There is no cap on the Vec length and no mechanism to remove entries.

The `assert_staker_address_not_reused` guard prevents address reuse, so each bloat entry requires a distinct address, but the attacker recovers their staked tokens after `unstake_action`, meaning the only sustained cost is gas per cycle.

### Impact Explanation
`get_stakers` is the consensus-layer entry point used to determine the active validator set for a given epoch. As the `stakers` Vec grows, the gas cost of every `get_stakers` call grows linearly. A sufficiently large Vec will cause `get_stakers` to exceed Starknet execution limits, making it impossible to retrieve the validator set. This disrupts the consensus mechanism and constitutes **unbounded gas consumption** and **griefing with damage to the protocol**.

**Impact: Medium** — Unbounded gas consumption / Griefing with no profit motive but damage to users or protocol.

### Likelihood Explanation
The attack requires:
1. Many distinct addresses (easily generated).
2. Each address must hold `min_stake` STRK tokens temporarily (recovered after `unstake_action`).
3. Gas cost per cycle (stake + unstake_intent + unstake_action = 3 transactions).

The minimum stake is a configurable parameter (`min_stake`). At any realistic minimum stake, a well-funded attacker can cycle thousands of addresses. The funds are fully recoverable, so the only sustained cost is gas. The attack is permissionless and requires no privileged access.

**Likelihood: Medium.**

### Recommendation
1. **Lazy deletion / active-set index**: Maintain a separate mapping from staker address to its index in `stakers`, and swap-remove on `unstake_action` to keep the Vec compact.
2. **Separate active-staker set**: Track only currently-active stakers in a dedicated iterable structure; move exited stakers out of it on `unstake_action`.
3. **Pagination**: Add a paginated variant of `get_stakers` that accepts an offset and limit, so callers can bound per-call gas regardless of Vec size.

### Proof of Concept

**Attacker steps (repeated N times with distinct addresses):**

1. Fund address `A_i` with `min_stake` STRK.
2. Call `staking.stake(reward_address, operational_address, min_stake)` from `A_i`.
   - `self.stakers.push(A_i)` executes — Vec length increases by 1.
3. Call `staking.unstake_intent()` from `A_i`.
4. Wait `exit_wait_window` seconds.
5. Call `staking.unstake_action(A_i)` — funds returned, but `A_i` remains in `stakers` Vec forever.

**After N cycles:**

- `stakers.len() == N` (all inactive entries).
- Every call to `get_stakers(epoch_id)` iterates all N entries, performing at minimum N storage reads for `is_staker_active`.
- At large N, `get_stakers` exceeds execution gas limits and reverts, making the consensus validator-set query permanently unavailable.
