Audit Report

## Title
Unbounded Concurrent `encode_manifest` in `state_sync_chunk_handler` Enables OOM/CPU Exhaustion by a Byzantine Peer — (`rs/p2p/state_sync_manager/src/routes/chunk.rs`)

## Summary
Every incoming `StateSyncChunkRequest` targeting a manifest chunk ID is dispatched unconditionally into a `tokio::task::spawn_blocking` task with no per-peer concurrency limit. Each such task calls `encode_manifest` from scratch, allocating ≥100 MiB for large checkpoints, with no caching between calls. A Byzantine subnet peer can open up to 1,000 concurrent QUIC streams (the configured limit) and flood the serving replica with manifest chunk requests, driving peak heap allocation to ≥50 GiB and causing OOM or sustained CPU saturation that prevents the replica from participating in consensus.

## Finding Description

**Unbounded `spawn_blocking` in `state_sync_chunk_handler`**

`rs/p2p/state_sync_manager/src/routes/chunk.rs:51-70` dispatches every chunk request into `tokio::task::spawn_blocking` with no semaphore, no per-peer counter, and no `TOO_MANY_REQUESTS` guard:

```rust
let jh = tokio::task::spawn_blocking(
    move || match state.state_sync.chunk(&artifact_id, chunk_id) {
        Some(data) => { ... zstd::bulk::compress(...) ... }
        None => Err(StatusCode::NO_CONTENT),
    },
);
```

The `TOO_MANY_REQUESTS` status code is handled on the client side (`chunk.rs:129`) but is never emitted by the server.

**Full manifest re-encoding on every manifest chunk request, no caching**

`rs/state_manager/src/state_sync/types.rs:498-515` — the `ManifestChunk` arm of `StateSyncMessage::get_chunk` calls `encode_manifest(&self.manifest)` unconditionally on every invocation. There is no `OnceCell`, `Mutex<Option<Vec<u8>>>`, or any other caching mechanism; the full serialization is repeated for every request:

```rust
StateSyncChunk::ManifestChunk(index) => {
    let index = index as usize;
    if index < self.meta_manifest.sub_manifest_hashes.len() {
        let encoded_manifest = encode_manifest(&self.manifest); // full re-encode every time
        ...
    }
}
```

**`DEFAULT_CHUNK_SIZE` is 1 MiB; manifests exceed 100 MiB at scale**

`rs/state_manager/src/state_sync/types.rs:95`:
```rust
pub const DEFAULT_CHUNK_SIZE: u32 = 1 << 20; // 1 MiB.
```

The test suite explicitly asserts that large-state manifests exceed 100 × 1 MiB = 100 MiB (`rs/state_manager/src/manifest/tests/compatibility.rs:592-594`):
```rust
assert!(
    encode_manifest(&manifest_v2).len() > 100 * DEFAULT_CHUNK_SIZE as usize,
    "The encoded manifest is supposed to be larger than 100 MiB."
);
```

**QUIC stream limit is the only guard — and the code acknowledges it is insufficient**

`rs/p2p/quic_transport/src/connection_manager.rs:74`:
```rust
const MAX_CONCURRENT_BIDI_STREAMS: VarInt = VarInt::from_u32(1_000);
```

`rs/p2p/quic_transport/src/request_handler.rs:52-56` contains the developer acknowledgment:
```
// The extreme result of a slow handler is that the stream limit will be reach, hence
// having buffered up to the stream limit number of messages/requests.
// A better approach will be to use a router implemented as a tower service and accept
// streams iff the router is ready.
```

No application-layer throttling was ever added. The `JoinSet` at `request_handler.rs:50` spawns a new task for every accepted stream without bound.

**Exploit flow**

1. Byzantine peer B authenticates via mutual TLS to victim replica A (valid subnet peer, within the IC `f`-Byzantine fault model).
2. B opens a QUIC connection and sends 1,000 concurrent `StateSyncChunkRequest { chunk_id: MANIFEST_CHUNK_ID_OFFSET + 0 }` targeting A's current checkpoint height.
3. A's `state_sync_chunk_handler` spawns 1,000 `spawn_blocking` tasks; Tokio's blocking thread pool runs up to 512 simultaneously.
4. Each task calls `encode_manifest` → ≥100 MiB heap allocation → zstd compression buffer on top.
5. Peak live heap: ≥50 GiB → OOM kill or severe memory pressure.
6. A cannot rate-limit B at the application layer; B can repeat indefinitely.

## Impact Explanation

This is a **High** severity finding matching: *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."*

An OOM-killed or CPU-saturated replica cannot participate in consensus, sign blocks, or serve certified state. On a subnet with a small committee, losing even one replica to this attack meaningfully degrades fault tolerance and can stall finalization if combined with other Byzantine nodes.

## Likelihood Explanation

The attacker must be a valid subnet peer authenticated via mutual TLS — a Byzantine node already admitted to the subnet. This is explicitly within the IC fault model (up to `f` Byzantine nodes tolerated). No special knowledge is required beyond the advertised state height/hash, which is broadcast every ~5 seconds. The chunk ID `MANIFEST_CHUNK_ID_OFFSET + 0` is always valid whenever the replica holds a checkpoint (normal operating condition). The attack is repeatable and requires no victim interaction.

## Recommendation

1. **Cache the encoded manifest** inside `StateSyncMessage` using `once_cell::sync::OnceCell<Vec<u8>>` (or `std::sync::OnceLock`) so `encode_manifest` is called at most once per state object, not once per chunk request.
2. **Add a per-peer concurrency semaphore** in `StateSyncChunkHandler` (e.g., `Arc<Semaphore>` keyed by peer `NodeId`) limiting concurrent `spawn_blocking` tasks per peer to a small value (4–8).
3. **Return `StatusCode::TOO_MANY_REQUESTS`** when the semaphore cannot be acquired; the client already handles this gracefully via `DownloadChunkError::Overloaded` (`chunk.rs:129`).

## Proof of Concept

**Deterministic integration test plan:**

```
1. Construct a StateSyncMessage with a manifest large enough that
   encode_manifest() > 100 * DEFAULT_CHUNK_SIZE (use dummy_file_table_and_chunk_table
   from the existing test suite).
2. Spawn 512 threads each calling message.get_chunk(ChunkId::new(MANIFEST_CHUNK_ID_OFFSET))
   in a loop.
3. Measure peak RSS; assert it exceeds 50 GiB or observe OOM.
4. Alternatively: in a local replica integration test, connect a second node as a
   Byzantine peer, send 1,000 concurrent StateSyncChunkRequests via the QUIC transport,
   and observe the serving replica's memory usage spike and/or crash.
```

The existing `deterministic_manifest_hash` test in `rs/state_manager/src/manifest/tests/compatibility.rs` already proves the >100 MiB allocation path is reachable with `dummy_file_table_and_chunk_table`.
