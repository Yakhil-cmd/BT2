### Title
Proposer-Authorizer Who Signs Envelope Passes Access Validation But Fails FVM Execution — (`fvm/transactionVerifier.go`)

---

### Summary

There is a concrete authorization-check inconsistency between the access node's pre-inclusion validator (`checkAccounts`) and the FVM's execution-time verifier (`verifyTransaction`). When an account is simultaneously the **proposer** and an **authorizer** (but not the payer), the access node explicitly permits that account to satisfy its obligation by signing the **envelope**. The FVM, however, only counts **payload** signatures toward authorizer weight. The result is that a validly-constructed, access-node-accepted transaction is silently rejected at execution time with an `AccountAuthorizationError`, while the payer is still charged fees.

---

### Finding Description

**Step 1 — Access node allows proposer to sign either payload or envelope.**

`checkAccounts` in `access/validator/validator.go` line 448 accepts the proposer's signature on either message:

```go
if !observedEnvelopeSig[tx.ProposalKey.Address] && !observedPayloadSig[tx.ProposalKey.Address] {
    return MissingSignatureError{...}
}
``` [1](#0-0) 

**Step 2 — Access node skips the authorizer check for proposer-authorizers.**

Because the proposer is "guaranteed to have a signature", `checkAccounts` skips the per-authorizer check for any account that is also the proposer:

```go
for _, authorizer := range tx.Authorizers {
    if authorizer == tx.Payer || authorizer == tx.ProposalKey.Address {
        continue   // ← proposer-authorizer silently passes
    }
    ...
}
``` [2](#0-1) 

**Step 3 — FVM only skips the authorizer weight check for payer-authorizers, not proposer-authorizers.**

`verifyTransaction` in `fvm/transactionVerifier.go` skips the weight check only when `addr == tx.Payer`. A proposer-authorizer (who is not the payer) is **not** skipped:

```go
for _, addr := range tx.Authorizers {
    if addr == tx.Payer {   // ← only payer is exempted
        continue
    }
    if !v.hasSufficientKeyWeight(payloadWeights, addr, keyWeightThreshold) {
        return errors.NewAccountAuthorizationErrorf(...)
    }
}
``` [3](#0-2) 

**Step 4 — Payload weight is zero for an envelope-only signer.**

`newSignatureEntries` partitions signatures into two separate weight maps: `payloadWeights` (from `tx.PayloadSignatures`) and `envelopeWeights` (from `tx.EnvelopeSignatures`). A proposer-authorizer who signed only the envelope contributes nothing to `payloadWeights`:

```go
payloadWeights  := make(map[flow.Address]int, len(tx.PayloadSignatures))
envelopeWeights := make(map[flow.Address]int, len(tx.EnvelopeSignatures))
``` [4](#0-3) 

The authorizer weight check at line 257 therefore sees `payloadWeights[addr] == 0` and returns `AccountAuthorizationError`, even though the account did provide a valid cryptographic signature on the envelope. [5](#0-4) 

---

### Impact Explanation

A user who is simultaneously the **proposer** and an **authorizer** (a common pattern when a third-party fee-payer service is used) and who signs the envelope — which the access node's `checkAccounts` explicitly permits — will:

1. Have their transaction accepted by the access node and included in a block.
2. Have their transaction fail at FVM execution time with `ErrCodeAccountAuthorizationError`.
3. Cause the payer to be charged transaction fees for a transaction that the protocol's own pre-validation said was valid.

This directly mirrors the original report: a legitimately authorized party (proposer-authorizer) is blocked from executing a transaction by an authorization check (`verifyTransaction`) that is narrower than the system's own stated rules (`checkAccounts`).

---

### Likelihood Explanation

The scenario requires:
- An account acting as both proposer and authorizer (but not payer) — common in fee-delegation patterns.
- That account signing the envelope rather than the payload — permitted by `checkAccounts` (line 448) but unusual in practice.

The access node's explicit allowance of envelope-signing for proposers (line 448) means this is a supported code path, not a theoretical edge case. Any SDK or wallet that follows the access node's stated rules and places the proposer-authorizer's signature on the envelope will trigger this failure deterministically.

---

### Recommendation

In `verifyTransaction`, extend the existing payer-authorizer exemption to also cover proposer-authorizers who are not the payer:

```go
for _, addr := range tx.Authorizers {
    if addr == tx.Payer {
        continue
    }
    // NEW: proposer-authorizer signs the payload OR envelope; check envelope weight too
    if addr == tx.ProposalKey.Address {
        if v.hasSufficientKeyWeight(payloadWeights, addr, keyWeightThreshold) ||
           v.hasSufficientKeyWeight(envelopeWeights, addr, keyWeightThreshold) {
            continue
        }
    }
    if !v.hasSufficientKeyWeight(payloadWeights, addr, keyWeightThreshold) {
        return errors.NewAccountAuthorizationErrorf(...)
    }
}
```

Alternatively, tighten `checkAccounts` to require proposers who are not the payer to sign the payload, aligning the pre-validation rule with the FVM's enforcement.

---

### Proof of Concept

Construct a `TransactionBody` where:

| Role | Account |
|---|---|
| Proposer | Account A |
| Payer | Account B |
| Authorizer | Account A |

Sign: Account A signs the **envelope** (valid per `checkAccounts` line 448). Account B signs the **envelope**.

**Access node (`checkAccounts`):**
- Line 448: proposer (A) has an envelope sig → passes.
- Line 453: authorizer A == proposer → skipped.
- Result: **accepted**, transaction enters a block.

**FVM (`verifyTransaction`):**
- Line 253: `addr (A) != tx.Payer (B)` → not skipped.
- Line 257: `payloadWeights[A] == 0` (A signed the envelope, not the payload).
- Result: `AccountAuthorizationError` — **transaction fails at execution**.

The payer (B) is charged fees for a transaction the access node certified as valid. [6](#0-5) [7](#0-6)
