### Title
Missing Lower-Bound Threshold Validation in FROST Signing and Presigning Functions - (File: src/frost/mod.rs)

### Summary

`assert_sign_inputs` and `presign` in `src/frost/mod.rs` omit the `threshold < 2` lower-bound check that every other analogous validation function in the codebase enforces. A caller supplying `threshold = 1` (or `0`) bypasses the guard, causing the FROST signing protocol to run Lagrange interpolation with an incorrect degree, producing a corrupted or unusable signature output.

### Finding Description

Every other parameter-validation function in the library enforces a minimum threshold of 2:

- `assert_key_invariants` (DKG): [1](#0-0) 
- `validate_triple_inputs` (OT-based triple generation): [2](#0-1) 

Both of these return `InitializationError::ThresholdTooSmall { threshold, min: 2 }` when `threshold < 2`.

`assert_sign_inputs`, which is the shared validation entry point for FROST EdDSA and RedJubjub signing, only checks the upper bound: [3](#0-2) 

The lower-bound check (`threshold < 2`) is entirely absent. The same omission exists in the FROST `presign` function: [4](#0-3) 

`assert_sign_inputs` is called by both the EdDSA and RedJubjub signing paths: [5](#0-4) 

### Impact Explanation

FROST signing uses Lagrange interpolation over the participant identifiers to combine signature shares. The `threshold` value controls the degree of the interpolating polynomial. When `threshold = 1` is supplied for a key that was generated with `threshold = 2`, the Lagrange coefficients are computed over a degree-0 polynomial instead of degree-1, yielding a combined signature scalar that does not correspond to the correct secret. The resulting signature fails verification. When `threshold = 0`, the interpolation has no basis points, which can cause a panic or produce an identity-element output.

In both cases honest participants complete the protocol and believe they have produced a valid signature, but the output is cryptographically unusable. This matches:

> **High: Corruption of sign outputs so honest parties accept inconsistent transcripts or unusable cryptographic outputs.**

### Likelihood Explanation

The `threshold` parameter is caller-supplied at every invocation of `presign` and the signing entry points. A malicious coordinator, a misconfigured integration, or any unprivileged library caller can pass `threshold = 1` without triggering any error. No special privilege or leaked key material is required. The missing check is a straightforward off-by-one omission relative to the pattern used everywhere else in the codebase.

### Recommendation

Add the same lower-bound guard that `assert_key_invariants` and `validate_triple_inputs` already use, in both `assert_sign_inputs` and `presign` inside `src/frost/mod.rs`:

```rust
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

This should be inserted immediately after the upper-bound check in each function, mirroring the pattern at: [6](#0-5) 

### Proof of Concept

1. Generate a key with `threshold = 2`, `n = 3` participants using the normal DKG path (which correctly rejects `threshold < 2`).
2. Call `presign` or the EdDSA `sign` entry point with the same participants but `threshold = 1`. Validation passes because only the upper-bound check fires.
3. All participants complete the protocol and receive a `Signature` value.
4. Verify the signature against the public key produced in step 1 — verification fails, confirming the output is corrupted.

The root cause is the absent guard in `assert_sign_inputs` at [7](#0-6)  and in `presign` at [8](#0-7) , contrasted with the enforced minimum in `assert_key_invariants` at [9](#0-8) .
