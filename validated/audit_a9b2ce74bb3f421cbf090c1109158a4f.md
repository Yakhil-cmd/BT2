### Title
Permanent Freezing of NEAR Funds via Failed Transfer to Non-Existent Account in wNEAR Unwrap Path — (`engine-precompiles/src/native.rs`, `engine/src/contract_methods/connector.rs`)

---

### Summary

When a wNEAR ERC-20 holder exits with the `:unwrap` suffix targeting a syntactically valid but non-existent NEAR named account, the two-step promise chain (`near_withdraw` → `exit_to_near_precompile_callback`) has an unhandled failure mode: `near_withdraw` succeeds (NEAR lands in the engine's account), but the subsequent `PromiseAction::Transfer` to the non-existent account fails silently with no callback and no recovery path. The wNEAR ERC-20 tokens are already burned, and the unwrapped NEAR is permanently locked in the engine's account.

---

### Finding Description

**Step 1 — Input parsing (no existence check)**

In `exit_erc20_token_to_near`, when `Message::UnwrapWnear` is detected for the wNEAR ERC-20 address, the `receiver_account_id` is taken directly from user-supplied input and stored in `TransferNearArgs`: [1](#0-0) 

The `receiver_account_id` is validated only for format via `AccountId::validate` (length, allowed characters), which accepts any syntactically valid string such as `definitely-does-not-exist.near`. There is no on-chain existence check. [2](#0-1) 

**Step 2 — Promise chain construction**

The precompile builds a `PromiseWithCallbackArgs` where:
- **Base**: `near_withdraw` on the wNEAR NEP-141 contract (burns wNEAR, credits NEAR to the engine's account)
- **Callback**: `exit_to_near_precompile_callback` on the engine itself [3](#0-2) 

**Step 3 — Callback fires after successful `near_withdraw`**

In `exit_to_near_precompile_callback`, when `near_withdraw` succeeds, a `PromiseBatchAction::Transfer` is created targeting `args.target_account_id`. Critically, **no callback is attached to this second promise**: [4](#0-3) 

**Step 4 — The `refund` branch does not cover this failure**

The `else if let Some(args) = args.refund` branch only executes when `near_withdraw` itself fails (promise result is not `Successful`). It has no bearing on the outcome of the subsequent transfer promise: [5](#0-4) 

**Step 5 — NEAR Protocol behavior**

In NEAR Protocol, a `Transfer` action targeting a non-existent **named account** (e.g., `nonexistent.near`) causes the receipt to fail. Only transfers to non-existent **implicit accounts** (64-char lowercase hex) succeed by auto-creating the account. Since `AccountId::validate` accepts any syntactically valid string, a user can supply a non-existent named account that passes all engine-side checks.

**Step 6 — No recovery mechanism**

There is no admin function, no rescue path, and no re-mint logic for NEAR that becomes stranded in the engine's account due to a failed transfer in this path. The `refund_on_error` function only handles ERC-20 re-minting or ETH refunds when the *first* promise fails: [6](#0-5) 

---

### Impact Explanation

- The user's wNEAR ERC-20 tokens are burned before the promise chain executes.
- `near_withdraw` succeeds: the NEAR is now in the engine's account.
- The `Transfer` to the non-existent named account fails silently.
- The NEAR remains in the engine's account with no recovery path.
- **Impact: Critical — Permanent freezing of user funds.**

---

### Likelihood Explanation

Any wNEAR ERC-20 holder can trigger this with no special privileges. The attack surface includes:
- Accidental user error (typo in account name)
- Deliberate self-harm (user intentionally destroys their own funds)

The only precondition is holding wNEAR ERC-20 tokens and submitting an EVM transaction with a syntactically valid but non-existent NEAR named account as the recipient. This is a fully unprivileged, publicly reachable path.

---

### Recommendation

Attach a failure-handling callback to the `PromiseBatchAction::Transfer` promise created inside `exit_to_near_precompile_callback`. If the transfer fails, the callback should re-mint the burned wNEAR ERC-20 tokens to the original sender (mirroring the existing `refund_on_error` logic). The `ExitToNearPrecompileCallbackArgs` struct already carries a `refund` field that could be threaded through to this second-level callback.

Concretely:
- Instead of `handler.promise_create_batch` + `handler.promise_return`, use `handler.promise_attach_callback` on the transfer promise to invoke a new `exit_to_near_transfer_callback` method.
- That callback checks the transfer result; on failure, calls `refund_on_error` to re-mint the wNEAR ERC-20 tokens.
- Ensure `refund_address` (the `error_refund` field) is always populated for the wNEAR unwrap path, regardless of whether the `error_refund` feature flag is set.

---

### Proof of Concept

```
1. Bridge wNEAR to Aurora; hold N wNEAR ERC-20 tokens at address `attacker`.
2. Construct EVM calldata for the wNEAR ERC-20 `withdrawToNear` (or equivalent burn+exit):
     flag=0x01 | amount=N | recipient="definitely-does-not-exist-99999.near:unwrap"
3. Submit the EVM transaction. The engine:
     a. Burns N wNEAR ERC-20 from `attacker`.
     b. Schedules: near_withdraw(amount=N) → exit_to_near_precompile_callback.
4. NEAR runtime executes near_withdraw: succeeds, N yoctoNEAR credited to engine account.
5. exit_to_near_precompile_callback fires (promise_result(0) = Successful):
     Creates PromiseBatchAction::Transfer { target: "definitely-does-not-exist-99999.near", amount: N }.
     No callback attached.
6. NEAR runtime executes the Transfer: FAILS (account does not exist).
7. Assert: attacker's wNEAR ERC-20 balance = 0.
8. Assert: N yoctoNEAR remains in engine's NEAR account with no recovery path.
```

            // Instead the implementation is based on the previous character checks.

            // We can safely assume that last char was a separator.
            let mut last_char_is_separator = true;

            for c in account_id.bytes() {
                let current_char_is_separator = match c {
                    b'a'..=b'z' | b'0'..=b'9' => false,
                    b'-' | b'_' | b'.' => true,
                    _ => {
                        return Err(ParseAccountError::Invalid);
                    }
                };
                if current_char_is_separator && last_char_is_separator {
                    return Err(ParseAccountError::Invalid);
                }
                last_char_is_separator = current_char_is_separator;

                .then_some(())
                .ok_or(ParseAccountError::Invalid)

pub fn refund_on_error<I: IO + Copy, E: Env, P: PromiseHandler>(
    io: I,
    env: &E,
    state: EngineState,
    args: &RefundCallArgs,
    handler: &mut P,
) -> EngineResult<SubmitResult> {
    let current_account_id = env.current_account_id();
    if let Some(erc20_address) = args.erc20_address {
        // ERC-20 exit; re-mint burned tokens
        let erc20_admin_address = current_address(&current_account_id);
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, erc20_admin_address, current_account_id, io, env);

        let refund_address = args.recipient_address;
        let amount = U256::from_big_endian(&args.amount);
        let input = setup_refund_on_error_input(amount, refund_address);

        engine.call(
            &erc20_admin_address,
            &erc20_address,
            Wei::zero(),
            input,
            u64::MAX,
            Vec::new(),
            Vec::new(),
            handler,
        )
    } else {
        // ETH exit; transfer ETH back from precompile address
        let exit_address = exit_to_near::ADDRESS;
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, exit_address, current_account_id, io, env);
        let refund_address = args.recipient_address;
        let amount = Wei::new(U256::from_big_endian(&args.amount));
        engine.call(
            &exit_address,
            &refund_address,
            amount,
            Vec::new(),
            u64::MAX,
            vec![
                (exit_address.raw(), Vec::new()),
                (refund_address.raw(), Vec::new()),
            ],
            Vec::new(),
            handler,
        )
    }
