### Title
Gateway `skip_stateful_validations` Bypasses `__validate__` for Invoke Transactions with Nonce=1 via Overly Broad Mempool Presence Check - (File: crates/apollo_gateway/src/stateful_transaction_validator.rs)

---

### Summary

The `skip_stateful_validations` function in the gateway stateful path skips the `__validate__` entry point (signature verification) for invoke transactions with `nonce=1` when `account_tx_in_pool_or_recent_block` returns `true`. That helper returns `true` for **any** transaction from the sender address in the mempool or a recent block — not exclusively a `deploy_account`. An attacker who controls an existing account (on-chain nonce=0) can submit a valid `nonce=0` invoke to satisfy the mempool-presence check, then submit a `nonce=1` invoke with an **invalid signature** that bypasses gateway admission validation and is accepted into the mempool.

---

### Finding Description

In `skip_stateful_validations` the gateway skips `__validate__` when three conditions hold simultaneously:

```
tx.nonce() == Nonce(Felt::ONE)
account_nonce == Nonce(Felt::ZERO)
account_tx_in_pool_or_recent_block(tx.sender_address()) == true
``` [1](#0-0) 

The design intent is to support the UX pattern of submitting `deploy_account` (nonce=0) and `invoke` (nonce=1) together: because the account contract does not yet exist on-chain, `__validate__` cannot be called, so the gateway defers to the mempool-presence check as a proxy for "a deploy_account is pending."

The proxy is broken. `account_tx_in_pool_or_recent_block` is implemented as:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [2](#0-1) 

`state.contains_account` returns `true` for any address present in `staged` or `committed` nonces — populated by **any** transaction type, not only `deploy_account`: [3](#0-2) 

`tx_pool.contains_account` similarly returns `true` for any transaction in the pool from that address: [4](#0-3) 

Consequently, an **existing** account (contract deployed, on-chain nonce=0) that has a valid `nonce=0` invoke already in the mempool satisfies condition 3. A subsequent `nonce=1` invoke from the same address — even with a completely invalid signature — will have `skip_validate=true` returned, causing `run_validate_entry_point` to set `execution_flags.validate = false` and skip the `__validate__` call entirely: [5](#0-4) 

The invalid transaction is then forwarded to the mempool and admitted. The code comment itself acknowledges the assumption that is violated:

> *"it means that either it has a deploy_account transaction or transactions with future nonces that passed validations"*

A `nonce=0` invoke from an existing account is neither a `deploy_account` nor a future-nonce transaction — it is a present-nonce transaction from an already-deployed account — yet it satisfies the check.

---

### Impact Explanation

An unprivileged attacker who controls any Starknet account with on-chain nonce=0 can:

1. Submit a valid `nonce=0` invoke to seed the mempool.
2. Submit arbitrarily many `nonce=1` invokes with invalid signatures; each bypasses `__validate__` at the gateway and is admitted to the mempool.

This matches the **High** impact category: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

The invalid transactions will revert during batcher execution (the batcher always sets `validate=true`), but they consume mempool capacity, batcher CPU, and block space before being discarded. With fee escalation disabled, the attacker's invalid `nonce=1` transaction also occupies the nonce slot, preventing the legitimate owner's `nonce=1` transaction from being accepted (it would receive `DuplicateNonce`).

---

### Likelihood Explanation

The preconditions are minimal: the attacker needs one deployed account with on-chain nonce=0 and the ability to submit two gateway transactions. No privileged access, no special contract, no network position is required. The attack is limited to `nonce=1` transactions (the check is hardcoded to `Nonce(Felt::ONE)`), but that is sufficient for the DoS and nonce-slot-griefing primitives described above.

---

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a **`deploy_account`** transaction is pending for the sender address. The mempool should expose a dedicated query such as `has_pending_deploy_account(address) -> bool` that inspects transaction types in the pool. Alternatively, the gateway can inspect the `InternalRpcTransactionWithoutTxHash` variant of the pending transaction to confirm it is `DeployAccount` before granting the skip.

---

### Proof of Concept

```
1. Deploy account X on-chain (on-chain nonce becomes 0, contract exists).

2. Submit invoke_A:
     sender = X, nonce = 0, signature = <valid>
   → validate_nonce: 0 <= 0 <= max_allowed  ✓
   → validate_by_mempool: nonce 0 >= account_nonce 0  ✓
   → skip_stateful_validations: nonce != 1  → false (no skip)
   → run_validate_entry_point: __validate__ called, passes  ✓
   → invoke_A admitted to mempool.

3. Submit invoke_B:
     sender = X, nonce = 1, signature = <INVALID / all-zeros>
   → validate_nonce: 0 <= 1 <= max_allowed  ✓
   → validate_by_mempool: nonce 1 >= account_nonce 0  ✓
   → skip_stateful_validations:
       tx.nonce() == 1  ✓
       account_nonce == 0  ✓
       account_tx_in_pool_or_recent_block(X) == true  ✓  (invoke_A is in pool)
       → returns true  (skip validation)
   → run_validate_entry_point: SKIPPED (validate = false)
   → invoke_B admitted to mempool with invalid signature.

4. Batcher retrieves invoke_B, sets validate=true, calls __validate__:
   → signature verification fails → transaction reverts.
   → Mempool slot for (X, nonce=1) was occupied; legitimate nonce=1 tx
     from X is rejected with DuplicateNonce until invoke_B is evicted.
```

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
