### Title
Malicious Participant Can Panic Honest Parties via Under-Length Vector in `do_generation_many` - (File: src/ecdsa/ot_based_ecdsa/triples/generation.rs)

### Summary
In `do_generation_many`, every round that receives a `Vec` from a remote participant indexes into it with `for i in 0..N` without first validating that the received vector has exactly `N` elements. A malicious participant can send a vector shorter than `N`, triggering an index-out-of-bounds panic in every honest party's async task, permanently aborting triple generation and therefore OT-based ECDSA signing.

### Finding Description
`do_generation_many<const N: usize>` is the core async function backing the public API `generate_triple_many`. Throughout the function, messages received over the network are deserialized into `Vec<_>` and then immediately indexed with `for i in 0..N { … received_vec[i] … }` — with no length check.

The first occurrence is at the commitment-collection round:

```rust
// line 180-183
let (from, commitments): (_, Vec<_>) = chan.recv(wait0).await?;
for i in 0..N {
    all_commitments_vec[i].put(from, commitments[i]); // panics if commitments.len() < N
}
```

The same unchecked pattern repeats for every subsequent round's received vectors:

| 407–408 | `a_j_i_v`, `b_j_i_v` | `a_j_i_v[i]`, `b_j_i_v[i]` |
| 474–475 | `big_c_j_v`, `their_phi_proofs` | `big_c_j_v[i]`, `their_phi_proofs[i]` |

In every case the loop bound is the compile-time constant `N`, but the length of the deserialized vector is fully attacker-controlled. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5) 

### Impact Explanation
`generate_triple_many` is a public API. [7](#0-6) 

A Rust index-out-of-bounds is a panic, not a recoverable `Err`. Unless the caller wraps the protocol executor in `catch_unwind`, the entire async task crashes. Because every honest party receives the same malformed message from the malicious participant, all honest parties panic simultaneously, permanently aborting the triple generation session. Since OT-based ECDSA signing requires a pre-generated triple, this constitutes **permanent denial of signing for all honest parties** — matching the High impact tier.

### Likelihood Explanation
Any single participant in the protocol can trigger this. The attacker only needs to serialize and send a `Vec` with fewer than `N` elements in the very first broadcast round (wait0). No cryptographic material, no prior state, and no coordination with other parties is required. Likelihood is **High**.

### Recommendation
Before the `for i in 0..N` loop, validate that every received vector has exactly `N` elements and return a `ProtocolError` (not a panic) if not:

```rust
let (from, commitments): (_, Vec<_>) = chan.recv(wait0).await?;
if commitments.len() != N {
    return Err(ProtocolError::AssertionFailed(format!(
        "expected {N} commitments from {from:?}, got {}", commitments.len()
    )));
}
for i in 0..N {
    all_commitments_vec[i].put(from, commitments[i]);
}
```

Apply the same guard to every other round that receives a `Vec` and indexes it with `0..N`.

### Proof of Concept
1. Honest parties call `generate_triple_many::<4>(participants, me, threshold, rng)`.
2. The malicious participant serializes a `Vec<Commitment>` of length 1 (instead of 4) and sends it at `wait0`.
3. Each honest party deserializes it into `commitments` (length 1) and enters `for i in 0..4`.
4. At `i = 1`, `commitments[1]` panics with `index out of bounds: the len is 1 but the index is 1`.
5. The async task unwinds; triple generation is permanently aborted for all honest parties; OT-based ECDSA signing is unavailable.
