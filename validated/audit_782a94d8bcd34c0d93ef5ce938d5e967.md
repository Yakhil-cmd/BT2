### Title
Unchecked Precompile Return Value in `withdrawToEthereum` Allows Silent Token Burn Without Ethereum Unlock — (`etc/eth-contracts/contracts/EvmErc20V2.sol`)

### Summary

`withdrawToEthereum` in `EvmErc20V2` burns ERC-20 tokens before calling the `ExitToEthereum` precompile, but never checks the return value of the assembly `call`. If the precompile fails for any reason, the burn is permanent and no NEAR-side withdraw promise is created, destroying user funds with no corresponding Ethereum-side unlock.

### Finding Description

In `withdrawToEthereum`, the sequence is:

1. `_burn(_msgSender(), amount)` — permanently reduces ERC-20 supply and user balance.
2. An inline assembly `call` is made to the `ExitToEthereum` precompile at `0xb0bd02f6...`.
3. The return value `res` is captured but **never checked**. [1](#0-0) 

The `ExitToEthereum` precompile (`engine-precompiles/src/native.rs`) can return `Err(ExitError::...)` in at least three production paths:

- `get_nep141_from_erc20` returns `ERR_TARGET_TOKEN_NOT_FOUND` if the ERC-20 address has no NEP-141 mapping in storage. [2](#0-1) 

- `get_eth_connector_contract_account` returns `ERR_KEY_NOT_FOUND` if the `EthConnectorAccount` storage key is absent. [3](#0-2) 

- `get_withdraw_serialize_type` returns `ERR_DESERIALIZE` on a corrupt storage value. [4](#0-3) 

The withdraw promise is only emitted inside the `Ok(PrecompileOutput { ... })` path. An `Err(...)` return produces no promise log at all. [5](#0-4) 

In standard EVM semantics (and Aurora's SputnikVM-based EVM), a failed sub-`call` to a precompile returns `0` to the caller but does **not** revert the calling frame. The caller is responsible for checking `res`. Since `withdrawToEthereum` never checks `res`, the outer call succeeds — the burn is committed, but no unlock promise is scheduled.

The identical pattern exists in `EvmErc20.sol`. [6](#0-5) 

### Impact Explanation

Every token burned via `withdrawToEthereum` under a failing precompile condition is permanently destroyed on the EVM side with no corresponding `withdraw` call to the NEP-141 contract and no ETH unlock on Ethereum. The ERC-20 total supply decreases, the user's balance is zeroed, and the bridged ETH remains locked in the Ethereum bridge contract forever. This is a direct, irreversible insolvency event: the ERC-20 supply and the Ethereum-side collateral become permanently mismatched.

### Likelihood Explanation

The precondition — precompile failure — is realistic in at least one concrete scenario: an ERC-20 contract deployed (or whose NEP-141 mapping entry was deleted/corrupted) without a corresponding entry in the `Erc20Nep141Map` storage. Any user holding tokens in such a contract who calls `withdrawToEthereum` will silently lose their funds. The user has no way to detect the failure at the Solidity level because `res` is discarded. The condition does not require admin compromise; it requires only that the storage mapping is absent, which can occur through misconfiguration or a partial deployment.

### Recommendation

Check the return value of the precompile `call` and revert if it is zero:

```solidity
assembly {
    let res := call(gas(), 0xb0bd02f6a392af548bdf1cfaee5dfa0eefcc8eab, 0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

Apply the same fix to `withdrawToNear` in both `EvmErc20.sol` and `EvmErc20V2.sol`, which have the identical unchecked-return pattern. [7](#0-6) 

### Proof of Concept

1. Deploy `EvmErc20V2` with a valid admin.
2. Call `mint(user, 1000)` from admin — user holds 1000 tokens.
3. Do **not** register the ERC-20 address in the `Erc20Nep141Map` storage (or use a mock IO that returns `None` for that key), so `get_nep141_from_erc20` will return `ERR_TARGET_TOKEN_NOT_FOUND`.
4. Call `withdrawToEthereum(recipient, 1000)` from `user`.
5. Observe: `balanceOf(user) == 0`, `totalSupply` decreased by 1000, transaction succeeded (no revert), and no promise log was produced by the precompile.
6. The 1000 tokens are permanently destroyed; no ETH is unlocked on Ethereum. [8](#0-7)

fn get_nep141_from_erc20<I: IO>(erc20_token: &[u8], io: &I) -> Result<AccountId, ExitError> {
    AccountId::try_from(
        io.read_storage(bytes_to_key(KeyPrefix::Erc20Nep141Map, erc20_token).as_slice())
            .map(|s| s.to_vec())

    .map_err(|_| ExitError::Other(Cow::Borrowed("ERR_INVALID_NEP141_ACCOUNT")))
}
