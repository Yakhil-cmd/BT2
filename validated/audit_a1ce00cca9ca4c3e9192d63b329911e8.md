The key facts from the code:

1. `HARD_MAX_SPREAD_FEE_E6 = 200_000` is the hard ceiling; `maxAdminSpreadFeeE6` is initialized to `200_000` and can only be lowered by the factory owner.
2. `setPoolBinAdditionalFees` has **no cap check** against `maxAdminSpreadFeeE6`.
3. `setPoolAdminFees` **does** check `if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh()`.
4. `addFeeBuyE6`/`addFeeSellE6` are `uint16` (max 65535); `maxAdminSpreadFeeE6` is `uint24` (range 0–200,000).

**Critical flaw in the PoC:** The question's specific scenario (`maxAdminSpreadFeeE6=200_000`, `addFeeBuyE6=65535`) does **not** demonstrate a bypass — 65535 < 200,000, so the invariant is preserved in that exact case. The assertion `65535 > maxAdminSpreadFeeE6` is false when the cap is at its default.

However, the **underlying missing check is a real vulnerability** when the factory owner lowers `maxAdminSpreadFeeE6` below 65535 (e.g., to 10,000 = 1%). In that configuration, a pool admin can call `setPoolBinAdditionalFees(pool, bin, 65535, 65535)` and store 6.5535% per-bin fees, bypassing the 1% cap — because the factory never validates the values before forwarding them to the pool.

### Title
Pool Admin Bypasses `maxAdminSpreadFeeE6` Cap via Unchecked `setPoolBinAdditionalFees` — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary
`MetricOmmPoolFactory.setPoolBinAdditionalFees` forwards `addFeeBuyE6`/`addFeeSellE6` to the pool with no validation against `maxAdminSpreadFeeE6`, while the parallel `setPoolAdminFees` path correctly enforces the cap. A pool admin can set per-bin fees exceeding the factory-configured maximum, causing traders to pay uncapped fees on swaps through that bin.

### Finding Description
`setPoolAdminFees` enforces the cap:

```solidity
// MetricOmmPoolFactory.sol:414-415
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
``` [1](#0-0) 

`setPoolBinAdditionalFees` has no such check — it passes values directly through:

```solidity
// MetricOmmPoolFactory.sol:450-457
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
``` [2](#0-1) 

`MetricOmmPool.setBinAdditionalFees` also has no cap check — it stores the raw values:

```solidity
// MetricOmmPool.sol:464-474
s.addFeeBuyE6 = addFeeBuyE6;
s.addFeeSellE6 = addFeeSellE6;
``` [3](#0-2) 

These stored values are applied unclamped in every swap path:

```solidity
// MetricOmmPool.sol:910
params.baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6)
``` [4](#0-3) 

### Impact Explanation
When `maxAdminSpreadFeeE6` is set below 65535 (e.g., 10,000 = 1%), a pool admin can set `addFeeBuyE6 = 65535` (6.5535%), causing every trader swapping through that bin to pay 6.5× the factory-permitted maximum. The excess fee accrues to the pool (LP benefit) but is extracted from traders beyond what the protocol's fee cap permits — direct loss of trader principal above the factory-enforced ceiling.

### Likelihood Explanation
The factory owner can lower `maxAdminSpreadFeeE6` at any time (it is a mutable `uint24` initialized to 200,000 but reducible to any value including 0). [5](#0-4) 
Once lowered, any pool admin can immediately exploit the unchecked path. The pool admin role is semi-trusted and explicitly scoped as a boundary that must be enforced by caps.

### Recommendation
Add a cap check in `setPoolBinAdditionalFees` before forwarding to the pool:

```solidity
if (addFeeBuyE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
```

### Proof of Concept
**Corrected PoC** (the question's numbers are wrong — 65535 < 200,000 does not bypass the default cap):

1. Deploy factory; factory owner calls `setFeeCaps` to lower `maxAdminSpreadFeeE6` to `10_000` (1%).
2. Pool admin calls `factory.setPoolBinAdditionalFees(pool, 0, 65535, 65535)`.
3. No revert occurs — the call succeeds.
4. Assert `binState.addFeeBuyE6 == 65535 > maxAdminSpreadFeeE6 (10_000)` — invariant violated.
5. A trader swapping through bin 0 pays 6.5535% additional fee instead of the capped 1%, losing the difference.
