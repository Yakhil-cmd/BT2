### Title
Unchecked Precompile Return Value in `withdrawToNear` Burns ERC-20 Tokens Without Transfer - (`etc/eth-contracts/contracts/EvmErc20.sol`)

### Summary

`EvmErc20.sol` and `EvmErc20V2.sol` burn a user's ERC-20 tokens **before** calling the `ExitToNear` precompile, and never check the return value of that call. If the precompile reverts for any reason (invalid recipient account ID, missing NEP-141 mapping, etc.), the ERC-20 tokens are permanently destroyed with no corresponding NEP-141 transfer to NEAR, resulting in irreversible fund loss for the user.

---

### Finding Description

In `EvmErc20.sol`, the `withdrawToNear` function executes `_burn` in the parent EVM context, then issues a low-level `call` to the `ExitToNear` precompile at `0xe9217bc70b7ed1f598ddd3199e80b093fa71124f`. The assembly block captures the return value in `res` but never inspects it:

```solidity
// etc/eth-contracts/contracts/EvmErc20.sol  lines 53-63
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // ← tokens destroyed here, in parent context

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f,
                        0, add(input, 32), input_size, 0, 32)
        // res is NEVER checked
    }
}
```

The same pattern appears in `withdrawToEthereum` (lines 65-76) and in both functions of `EvmErc20V2.sol` (lines 53-63, 66-77).

The `ExitToNear` precompile (`engine-precompiles/src/native.rs`) returns `Err(ExitError)` — causing the sub-call to return `0` — in multiple reachable conditions:

| Condition | Code location |
|---|---|
| Recipient account ID fails NEAR format validation | `native.rs:374-377` |
| ERC-20 has no NEP-141 mapping (`ERR_TARGET_TOKEN_NOT_FOUND`) | `native.rs:302-309` |
| Input byte length outside `[MIN_INPUT_SIZE, MAX_INPUT_SIZE]` | `native.rs:295-300` |
| Non-zero ETH value attached to an ERC-20 exit | `native.rs:576-580` |

In standard EVM semantics, a failed sub-`call` does **not** revert the parent frame. Because `_burn` executes in the parent frame before the sub-call, the burn is committed regardless of whether the precompile succeeds. The function then returns normally, leaving the user with zero ERC-20 tokens and zero NEP-141 tokens received.

---

### Impact Explanation

**Critical — Permanent freezing / direct theft of user funds.**

The user's ERC-20 tokens are irreversibly destroyed. The corresponding NEP-141 tokens remain locked inside the Aurora contract with no mechanism to recover them to the user. The bridge's NEP-141 reserve becomes permanently larger than the outstanding ERC-20 supply for that token, but the individual user suffers a total loss of the withdrawn amount.

---

### Likelihood Explanation

**Medium.**

The most realistic trigger is a user supplying a recipient string that is syntactically valid UTF-8 but fails NEAR `AccountId` parsing (e.g., uppercase letters, leading/trailing dots, or an account ID exceeding 64 characters). This is a common user mistake when interacting with cross-chain bridges. No privileged access or external oracle failure is required — any token holder can trigger the loss on their own funds by providing a malformed recipient.

---

### Recommendation

Check the return value of the precompile call and revert if it fails, so that `_burn` is also reverted:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f,
                    0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

Apply the same fix to `withdrawToEthereum` and to both functions in `EvmErc20V2.sol`.

---

### Proof of Concept

1. Alice holds 1000 units of a bridged ERC-20 token on Aurora.
2. Alice calls `withdrawToNear("INVALID..ACCOUNT", 1000)` — the recipient string contains consecutive dots, which is rejected by NEAR's `AccountId` parser.
3. `_burn(Alice, 1000)` executes in the parent EVM frame; Alice's balance drops to 0.
4. The `call` to the `ExitToNear` precompile returns `0` because `parse_recipient` returns `Err(ExitError::Other("ERR_INVALID_RECEIVER_ACCOUNT_ID"))` at `native.rs:374-377`.
5. `res` is never inspected; the Solidity function returns normally.
