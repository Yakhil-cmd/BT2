### Title
COA Full Withdrawal Permanently Fails for Sub-UFix64 Precision Balances Due to attoFlow→UFix64 Unit Conversion - (File: fvm/evm/impl/impl.go)

### Summary
`newInternalEVMTypeWithdrawFunction` in `fvm/evm/impl/impl.go` uses `AttoFlowBalanceIsValidForFlowVault` which only checks `bal >= 1e10` (minimum threshold), but does **not** check that the balance is a multiple of `1e10`. EVM gas fees naturally produce balances with sub-UFix64 precision. When a COA owner attempts a full withdrawal, the truncation in the withdraw function silently discards the sub-UFix64 remainder, leaving it permanently locked in the EVM account with no recovery path.

### Finding Description
The `newInternalEVMTypeWithdrawFunction` (lines 744–827 of `fvm/evm/impl/impl.go`) implements the `InternalEVM.withdraw` host function called by `CadenceOwnedAccount.withdraw(balance:)` in `fvm/evm/stdlib/contract.cdc`.

The guard at line 779 is:

```go
if !types.AttoFlowBalanceIsValidForFlowVault(amountValue.BigInt) {
    panic(types.ErrWithdrawBalanceRounding)
}
```

`AttoFlowBalanceIsValidForFlowVault` is defined as:

```go
func AttoFlowBalanceIsValidForFlowVault(bal *big.Int) bool {
    return bal.Cmp(UFixToAttoConversionMultiplier) >= 0  // only checks bal >= 1e10
}
```

It does **not** check `bal % 1e10 == 0`. The actual truncation then occurs at lines 784–785:

```go
value := new(big.Int).Div(amountValue.BigInt, types.UFixToAttoConversionMultiplier)
amount := types.NewBalanceFromUFix64(cadence.UFix64(value.Uint64()))
```

`amount` is always a multiple of `1e10`. The difference `amountValue.BigInt - amount` (the sub-UFix64 remainder) stays in the EVM account. If that remainder is `< 1e10`, any subsequent attempt to withdraw it will hit the `AttoFlowBalanceIsValidForFlowVault` guard and revert with `ErrWithdrawBalanceRounding`, permanently locking those funds.

EVM gas fees are `gasUsed × gasPrice` in attoFlow. Since `gasPrice` can be any value (e.g., `1` attoFlow, as used in tests at `fvm/evm/emulator/emulator_test.go` line 459), gas fees are routinely **not** multiples of `1e10`. A COA that executes any EVM transaction with a non-zero gas price will accumulate sub-UFix64 precision in its balance.

**Concrete example:**
1. COA deposits `1e10` attoFlow (= 0.00000001 FLOW, the minimum).
2. COA executes an EVM transaction: `gasUsed = 21000`, `gasPrice = 1` attoFlow → fee = `21000` attoFlow.
3. Remaining balance = `1e10 − 21000 = 9999979000` attoFlow.
4. `9999979000 < 1e10` → `AttoFlowBalanceIsValidForFlowVault` returns `false`.
5. Any call to `coa.withdraw(balance: EVM.Balance(attoflow: 9999979000))` reverts with `ErrWithdrawBalanceRounding`.
6. The `9999979000` attoFlow is permanently irrecoverable.

For larger balances the same pattern applies: the truncated withdrawal succeeds, but the remainder `< 1e10` is left behind and can never be withdrawn.

### Impact Explanation
Any COA owner who executes EVM transactions with non-zero gas prices (the normal case) will have sub-UFix64 precision in their balance. Attempting a full withdrawal either:
- **Reverts entirely** if the full balance is `< 1e10` attoFlow (direct analog to the external report's "full repayment fails"), or
- **Silently leaves up to `9999999999` attoFlow permanently stuck** after a partial withdrawal.

The stuck amount (up to `< 0.00000001 FLOW` per COA) is irrecoverable — there is no "max withdrawal" path and no mechanism to sweep sub-UFix64 remainders. This is a systematic, permanent loss of user funds for every COA that interacts with EVM.

### Likelihood Explanation
Any COA that calls `EVM.run(tx:coinbase:)` or `EVM.batchRun(txs:coinbase:)` with a non-zero gas price will produce a sub-UFix64 balance. This is the standard usage pattern for COAs interacting with EVM contracts. The condition is triggered automatically and requires no special attacker action — any unprivileged user who owns a COA and executes EVM transactions is affected.

### Recommendation
Add a "max withdrawal" sentinel (e.g., `UInt.max` as the `attoflow` value) to indicate a full withdrawal. When this sentinel is detected, compute the withdrawable amount as `floor(balance / 1e10) * 1e10` on-chain and withdraw that, rather than requiring the caller to supply an exact amount. This mirrors the fix recommended in the external report (using `repayAmount == uint.max` to indicate full repayment). Alternatively, expose a `maxWithdrawable()` view function on `CadenceOwnedAccount` that returns `floor(balance / 1e10) * 1e10` so callers can query the correct amount before withdrawing.

### Proof of Concept
```cadence
import EVM from <EVMAddress>
import FlowToken from <FlowTokenAddress>

transaction() {
    prepare(account: auth(BorrowValue) &Account) {
        // Step 1: Mint and deposit exactly 1e10 attoFlow (minimum withdrawable)
        let admin = account.storage.borrow<&FlowToken.Administrator>(
            from: /storage/flowTokenAdmin)!
        let minter <- admin.createNewMinter(allowedAmount: 0.00000001)
        let vault <- minter.mintTokens(amount: 0.00000001)
        destroy minter

        let coa <- EVM.createCadenceOwnedAccount()
        coa.deposit(from: <-vault)
        // coa.balance().attoflow == 10000000000 (= 1e10)

        // Step 2: Execute an EVM tx with gasPrice=1, gasUsed=21000
        // → fee = 21000 attoFlow (not a multiple of 1e10)
        // → remaining balance = 9999979000 attoFlow (< 1e10)
        // (omitted for brevity; any EVM tx with gasPrice=1 achieves this)

        // Step 3: Attempt full withdrawal — REVERTS
        let bal = EVM.Balance(attoflow: 9999979000)
        // panics: "withdraw failed! smallest unit allowed to transfer is 1e10 attoFlow"
        let stuck <- coa.withdraw(balance: bal)

        destroy coa
        destroy stuck
    }
}
```

The `9999979000` attoFlow is permanently locked with no recovery path.
