### Title
Temporary oracle price spike permanently locks high watermarks in `OracleValueStopLossExtension`, causing indefinite swap DoS — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

---

### Summary

`OracleValueStopLossExtension` uses a ratcheting high-watermark that only increases (never decreases without decay). When `decayPerSecondE8 == 0`, a temporary oracle price spike causes `hwm1` (or `hwm0`) to ratchet up permanently. Once the price reverts to normal, the per-share metric falls below the drawdown floor relative to the elevated watermark, triggering `OracleStopLossTriggered` on every subsequent swap in the affected direction — indefinitely. Any unprivileged user can trigger this by executing a swap during a natural price spike.

---

### Finding Description

The ratchet logic in `_applyWatermark` only moves the watermark upward:

```solidity
function _applyWatermark(uint256 metric, uint256 hwm, uint256 floorMultiplier)
    private pure returns (uint256 newHwm, bool breached)
{
    if (metric >= hwm) return (metric, false);   // ratchet up
    breached = metric < (hwm * floorMultiplier) / E6;
    return (hwm, breached);                       // hwm frozen if metric < hwm
}
``` [1](#0-0) 

The decay helper returns the watermark unchanged when `decayPerSecondE8 == 0`:

```solidity
function _decayed(uint256 hwm, uint256 ratePerSecondE8, uint256 dt) private pure returns (uint256) {
    if (ratePerSecondE8 == 0 || dt == 0 || hwm == 0) return hwm;
    ...
}
``` [2](#0-1) 

Zero decay is an explicitly valid configuration — `_validateDecay` only rejects values `> E8`: [3](#0-2) 

The `metricToken1` formula is:

```
metricToken1 = (t0 * mid / 2^64) * SCALE / shares + t1 * SCALE / shares
``` [4](#0-3) 

When the oracle mid price spikes high, `metricToken1` increases (token0 is worth more in token1 terms). A swap executed during the spike causes `hwm1` to ratchet up to the elevated value. When the price reverts to normal, `metricToken1` returns to its baseline, but `hwm1` remains at the spike level. The breach check then fires:

```
breached = metricT1 < (hwm1_elevated * floorMultiplier) / E6
```

With `drawdownE6 = 50_000` (5%) and a 2× spike: `hwm1 = 2M`, floor = `1.9M`, baseline `metricT1 = M` → `M < 1.9M` → breach → every subsequent `zeroForOne == false` swap reverts. [5](#0-4) 

The watermarks are updated at the end of `_checkAndUpdateWatermarks` only when no revert occurs, so the elevated `hwm1` persists across all future calls: [6](#0-5) 

The existing test suite explicitly confirms this behavior — `test_setDecayZero_freezesRecovery` shows that after setting decay to 0, the pool remains blocked even after 30 days: [7](#0-6) 

And `test_vShapeMove_blocksEachSideThenDecays` confirms that even with decay enabled, a V-shape price move blocks one swap direction until the watermark decays sufficiently (2 days at rate 58): [8](#0-7) 

The only recovery path is `proposeOracleStopLossHighWatermarks` + `executeOracleStopLossHighWatermarks`, both gated behind the pool's timelock: [9](#0-8) 

---

### Impact Explanation

Swaps in the affected direction revert with `OracleStopLossTriggered` indefinitely. With `decayPerSecondE8 == 0`, the block is permanent until the pool admin manually resets the watermarks — which itself requires waiting out the pool's timelock. This constitutes an unusable swap flow for a core pool action, matching the "Broken core pool functionality" impact gate. No LP principal is directly drained, but the pool is rendered non-functional for one (or both) swap directions.

---

### Likelihood Explanation

The trigger is unprivileged: any swap executed while the oracle mid price is temporarily elevated ratchets up `hwm1`. Oracle price spikes are routine market events (volatility, brief arbitrage windows, oracle update latency). No malicious intent is required — a normal trader swapping during a spike is sufficient. Pools initialized with `decayPerSecondE8 == 0` are permanently vulnerable; pools with slow decay (e.g., `decayPerSecondE8 = 1`) may be blocked for months.

---

### Recommendation

1. **Enforce a minimum non-zero `decayPerSecondE8`** at initialization and when updating via `executeOracleStopLossDecay`. A zero decay rate means the watermark can never self-heal from a temporary price spike.
2. **Alternatively**, use a time-weighted or rolling-window watermark rather than a pure ratchet, so that a transient spike does not permanently elevate the reference level.
3. **At minimum**, document prominently that `decayPerSecondE8 == 0` makes the pool permanently vulnerable to watermark lock-in from any price spike, and that recovery requires a timelocked admin action.

---

### Proof of Concept

```
Setup:
  - Pool with OracleValueStopLossExtension
  - drawdownE6 = 50_000 (5%), decayPerSecondE8 = 0, timelock = 7 days

Step 1: Normal swap at mid = 1.0
  → metricT1 = M; hwm1 ratchets to M

Step 2: Oracle mid spikes to 2.0 (temporary)
  → metricT1 = 2M (token0 worth more in token1 terms)

Step 3: Any user executes a zeroForOne=false swap at mid=2.0
  → _applyWatermark(2M, M, ...) → 2M >= M → hwm1 ratchets to 2M
  → swap succeeds

Step 4: Oracle mid returns to 1.0
  → metricT1 = M

Step 5: Any user attempts a zeroForOne=false swap at mid=1.0
  → _applyWatermark(M, 2M, 950_000)
  → M < 2M * 950_000 / 1_000_000 = 1.9M → breached = true
  → breach1 && !zeroForOne → revert OracleStopLossTriggered

Result: All zeroForOne=false swaps revert permanently.
Recovery: pool admin must wait 7-day timelock to reset watermarks.
```

  function _decayed(uint256 hwm, uint256 ratePerSecondE8, uint256 dt) private pure returns (uint256) {
    if (ratePerSecondE8 == 0 || dt == 0 || hwm == 0) return hwm;
    uint256 factor = ratePerSecondE8 * dt;
    if (factor >= E8) return 0;
    return hwm - (hwm * factor) / E8;
  }
