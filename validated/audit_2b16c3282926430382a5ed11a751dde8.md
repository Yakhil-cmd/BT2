### Title
`native_per_pubdata` Floor Division Precision Loss Allows Free Pubdata — (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

In the ZK transaction validation path, `native_per_pubdata` is computed using floor (`wrapping_div`) division. When `pubdata_price < native_price`, the result truncates to zero, making every pubdata byte free in native resources. This is the direct analog of the NFTLoanFacilitator precision-loss bug: a small numerator divided by a larger denominator floors to zero, silently disabling a cost-enforcement constraint.

---

### Finding Description

In `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`, the per-pubdata-byte native cost is computed as:

```rust
// line 142
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
``` [1](#0-0) 

`wrapping_div` is integer floor division. When `pubdata_price < native_price` (e.g., `pubdata_price = 1`, `native_price = 2`), the result is `0`.

Contrast this with `native_per_gas`, computed on the very next lines using **ceiling** division:

```rust
// line 135
u256_try_to_u64(&gas_price.div_ceil(native_price)).ok_or(...)
``` [2](#0-1) 

The asymmetry is the root cause: `native_per_gas` uses `div_ceil` (correct), but `native_per_pubdata` uses `wrapping_div` (floor, incorrect).

When `native_per_pubdata == 0`, the downstream pubdata cost functions compute:

```rust
// gas_helpers.rs line 430-432
let native = current_pubdata_spent
    .checked_mul(native_per_pubdata)   // 0 * anything = 0
    .ok_or(out_of_native_resources!())?;
``` [3](#0-2) 

This means `to_charge_for_pubdata` is always zero native resources. The `check_enough_resources_for_pubdata` call in `execute_or_deploy_inner` will always return `has_enough = true`:

```rust
// zk/mod.rs line 880-885
let (has_enough, to_charge_for_pubdata, pubdata_used) = check_enough_resources_for_pubdata(
    system,
    context.native_per_pubdata,   // == 0
    &resources_for_check,
    Some(context.validation_pubdata),
)?;
``` [4](#0-3) 

The pubdata-cost revert branch (`!has_enough`) is never triggered, and `compute_gas_refund` receives a zero-cost pubdata charge, so no gas is consumed for pubdata either.

---

### Impact Explanation

A transaction that generates large state diffs (pubdata) pays **zero** native resources for that pubdata whenever `pubdata_price < native_price`. The pubdata cost enforcement — the primary mechanism preventing pubdata spam on ZKsync — is silently bypassed. An attacker can write large amounts of state (e.g., via SSTORE loops) at no pubdata cost, inflating L1 data costs for the protocol without paying for them. This is a **resource accounting bug** with direct financial impact on the protocol.

---

### Likelihood Explanation

The `pubdata_price` and `native_price` are oracle-supplied block-level parameters (`BlockMetadataFromOracle`):

```rust
pub eip1559_basefee: U256,
pub pubdata_price: U256,
pub native_price: U256,
``` [5](#0-4) 

These values are set by the sequencer/operator per block. Any block configuration where `pubdata_price < native_price` — which is a plausible pricing regime (e.g., cheap pubdata relative to proving cost) — triggers the bug. An attacker only needs to submit a normal ZK transaction during such a block; no special privileges are required.

---

### Recommendation

Replace `wrapping_div` with `div_ceil` for `native_per_pubdata`, consistent with how `native_per_gas` is computed:

```rust
// Before (floor division — broken):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;

// After (ceiling division — correct):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

This ensures that even when `pubdata_price < native_price`, at least 1 native unit is charged per pubdata byte, preventing the constraint from being silently zeroed out.

---

### Proof of Concept

1. Operator sets block parameters: `pubdata_price = 1`, `native_price = 2`.
2. Attacker submits a ZK transaction with a contract that executes many `SSTORE` operations, generating, say, 10,000 bytes of pubdata.
3. In `validate_and_compute_fee_for_transaction`:
   - `native_per_pubdata = 1.wrapping_div(2) = 0`
4. During execution, `check_enough_resources_for_pubdata` computes `native = 10000 * 0 = 0`, so `has_enough = true` unconditionally.
5. In `before_refund`, `compute_gas_refund` receives `to_charge_for_pubdata = 0 native`, so no gas is consumed for pubdata.
6. The attacker's transaction succeeds, having written 10,000 bytes of pubdata at zero cost.
7. The protocol bears the full L1 data cost.
