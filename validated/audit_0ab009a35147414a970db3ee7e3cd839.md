### Title
Silo ERC-20 Whitelist Bypass via EVM-Layer Transfer After Whitelisted Deposit — (`engine/src/contract_methods/silo/mod.rs`, `engine/src/engine.rs`)

### Summary
The silo whitelist is enforced only at two entry points: `ft_on_transfer` (via `receive_erc20_tokens`) and `submit`/`deploy_code` (via `is_allow_submit`/`is_allow_deploy`). Neither the ERC-20 contract itself nor the EVM execution layer enforces the whitelist on the **recipient** of an EVM-internal ERC-20 `transfer` call. A whitelisted address that legitimately receives ERC-20 tokens can immediately transfer them to any non-whitelisted address via a standard `submit` transaction, breaking silo isolation.

### Finding Description

**Step 1 — Entry via `ft_on_transfer`:**

`receive_erc20_tokens` checks `is_allow_receive_erc20_tokens` on the parsed recipient address. If the recipient is whitelisted, tokens are minted directly to it: [1](#0-0) 

`is_allow_receive_erc20_tokens` only checks the `Address` whitelist: [2](#0-1) 

So whitelisted address `W` legitimately receives ERC-20 tokens. This is the only whitelist gate on the receive path.

**Step 2 — EVM-layer transfer via `submit`:**

The attacker (controlling `W`) submits a signed EVM transaction calling `erc20.transfer(non_whitelisted_address, amount)`. The `submit` function calls `engine::submit`, which enforces `is_allow_submit`: [3](#0-2) 

This checks the **sender** (`W`, whitelisted) and the NEAR **account** (also whitelisted). It does **not** check the ERC-20 transfer recipient. The check passes.

**Step 3 — No whitelist check on EVM-internal recipient:**

The ERC-20 contract is a standard EVM contract. Its `transfer` function updates balances in EVM storage with no reference to the silo whitelist. There is no hook, precompile, or engine-level guard that intercepts EVM-internal ERC-20 transfers and validates the recipient against the `Address` whitelist.

After execution, `non_whitelisted_address` holds ERC-20 tokens that should never have left the silo's whitelisted participant set.

### Impact Explanation
The silo isolation invariant — that only whitelisted addresses accumulate ERC-20 tokens — is broken at the EVM layer. Non-whitelisted addresses can receive and hold ERC-20 tokens, including any yield/rewards distributed to ERC-20 holders, without triggering the fallback redirect. This constitutes theft of unclaimed yield: yield that should flow only to whitelisted silo participants can be diverted to arbitrary addresses.

### Likelihood Explanation
Any whitelisted silo participant can execute this. It requires no special privileges beyond being whitelisted, no admin compromise, and no external dependencies. The path is a standard two-step sequence: deposit via `ft_on_transfer`, then `submit` an ERC-20 `transfer` call. Likelihood is high given the low barrier.

### Recommendation
Enforce the silo address whitelist on the **recipient** of EVM-internal ERC-20 transfers. Options include:
- Adding a precompile hook or EVM callback that validates the recipient of ERC-20 `transfer`/`transferFrom` calls against the `Address` whitelist.
- Overriding the ERC-20 token contract deployed by the engine to include a `_beforeTokenTransfer` hook that calls back into the engine's whitelist check.
- Documenting explicitly that silo isolation only applies at the NEAR→EVM bridge boundary, if that is the intended design — but in that case the invariant claim must be corrected.

### Proof of Concept
1. Deploy engine in silo mode with `Address` whitelist enabled and a fallback address set.
2. Whitelist address `W` in the `Address` whitelist.
3. Call `ft_on_transfer` with `msg = hex(W)` and `amount = X` → `W` receives `X` ERC-20 tokens (fallback not triggered).
4. From the NEAR account controlling `W`, call `submit` with a signed EVM transaction: `erc20.transfer(non_whitelisted_address, X)`.
5. `is_allow_submit` passes (W is whitelisted sender).
6. Assert `erc20.balanceOf(non_whitelisted_address) == X` → silo isolation is broken. [4](#0-3) [5](#0-4)

    pub fn receive_erc20_tokens<P: PromiseHandler>(
        &mut self,
        token: &AccountId,
        args: &FtOnTransferArgs,
        current_account_id: &AccountId,
        handler: &mut P,
    ) -> Result<Option<SubmitResult>, ContractError> {
        let amount = args.amount.as_u128();
        // Parse message to determine recipient
        let mut recipient = {
            // The message should contain the recipient EOA address.
            let message = args.msg.strip_prefix("0x").unwrap_or(&args.msg);
            // Recipient - 40 characters (Address in hex without '0x' prefix)
            if message.len() < 40 {
                return Err(ParseOnTransferMessageError::WrongMessageFormat.into());
            }
            let mut address_bytes = [0; 20];
            hex::decode_to_slice(&message[..40], &mut address_bytes)
                .map_err(|_| ParseOnTransferMessageError::WrongMessageFormat)?;
            Address::from_array(address_bytes)
        };

        if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
            && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
        {
            recipient = fallback_address;
        }

        let erc20_token = get_erc20_from_nep141(&self.io, token)?;
        let erc20_admin_address = current_address(current_account_id);
        let result = self
            .call(
                &erc20_admin_address,
                &erc20_token,
                Wei::zero(),
                setup_receive_erc20_tokens_input(&recipient, amount),
                u64::MAX,
                Vec::new(), // TODO: are there values we should put here?
                Vec::new(),
                handler,
            )
            .and_then(submit_result_or_err)?;

        sdk::log!("Mint {amount} ERC-20 tokens for: {}", recipient.encode());

        // Return SubmitResult so that it can be accessed in standalone engine.
        // This is used to help with the indexing of bridge transactions.
        Ok(Some(result))
    }
