### Title
Multi-NFT Royalty Display Truncation Understates Actual Spend in Offer Confirmation Dialog - (File: packages/gui/src/electron/commands/parseCommandDisplay.ts)

### Summary

`formatAmountWithRoyalties` in `parseCommandDisplay.ts` uses a flawed BigInt split-then-multiply algorithm to compute the "Total Amount with Royalties" shown in the WalletConnect and GUI offer-confirmation dialog. When an offer involves multiple NFTs with royalties, the function divides the fungible amount equally among NFTs before computing each royalty, producing a displayed total that is a factor of N lower than the actual on-chain royalty cost. A user who approves the offer based on the displayed figure will spend significantly more XCH or CAT than shown.

### Finding Description

`formatAmountWithRoyalties` is called from `withRoyaltyTotals` → `walletDeltaToDisplay` → `parseCommandDisplay`, which is invoked at two points in `main.tsx`: once for every WalletConnect `take_offer` request (line 329) and once for every direct GUI `take_offer` command (line 825). The result populates `display.walletDelta`, which is rendered in `Confirm.tsx` as the "Total Amount with Royalties" line the user sees before clicking Accept.

The flawed calculation:

```typescript
// packages/gui/src/electron/commands/parseCommandDisplay.ts  lines 363-367
const splitAmount = amount / BigInt(royaltyPercentages.length);   // ① integer-divides amount by NFT count
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,  // ② royalty on split, not full amount
  0n,
);
```

On-chain, each NFT's royalty puzzle receives a percentage of the **full** fungible amount. The GUI instead divides the amount by the number of NFTs first, then multiplies by each royalty percentage. For N NFTs with identical royalty R, the displayed royalty is `R/N × N = R`, but the actual royalty is `R × N`. The display is understated by a factor of N.

**Concrete example — 2 NFTs, each 5% royalty, 1 XCH offered:**

| | Displayed (GUI) | Actual (on-chain) |
|---|---|---|
| splitAmount | 0.5 XCH | — |
| royalty per NFT | 0.025 XCH | 0.05 XCH |
| total royalty | **0.05 XCH** | **0.10 XCH** |
| shown to user | **1.05 XCH** | **1.10 XCH** |

The user approves believing they pay 1.05 XCH; the blockchain deducts 1.10 XCH.

The confirmation dialog renders this at:

```typescript
// packages/gui/src/electron/dialogs/Confirm/Confirm.tsx  lines 109-113
{line.amountWithRoyalties && (
  <div className="text-xs text-chia-text-secondary">
    {i18n._(/* i18n */ { id: 'Total Amount with Royalties' })}: {line.amountWithRoyalties} ...
  </div>
)}
```

### Impact Explanation

A user who relies on the "Total Amount with Royalties" figure to decide whether to accept an offer will approve a transaction that deducts more XCH or CAT than displayed. For N NFTs each carrying a royalty percentage R, the underpayment shown is `(N-1) × R × amount / 10000`. At 5% royalty and 2 NFTs on a 1 XCH offer this is 0.05 XCH (~$30–50). At higher royalties or more NFTs the gap grows proportionally. This is a direct, quantifiable balance/accounting impact on XCH and CAT assets caused by spoofed offer state in the confirmation dialog.

### Likelihood Explanation

Any offer containing two or more NFTs with royalties triggers the bug. A malicious dApp connected via WalletConnect can craft such an offer and invoke `chia_wallet.take_offer`; the confirmation dialog will show the understated total. The user has no other in-dialog signal of the true cost. No privileged access is required.

### Recommendation

Replace the split-then-multiply approach with the correct per-NFT-on-full-amount calculation:

```typescript
// Correct: each NFT's royalty is computed on the full amount
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (amount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
const totalAmount = amount + royaltyAmount;
```

Remove the `splitAmount` variable entirely. Also add a test case asserting that two NFTs with 5% royalties on 1 XCH display 1.10 XCH, not 1.05 XCH.

### Proof of Concept

Using the existing test harness in `parseCommandDisplay.test.ts`, modify the "multiple NFT creator royalties" test to use two NFTs each with `royalty_percentage: 500` (5%) and `xch: '1000000000000'` (1 XCH). The current code returns `amountWithRoyalties: '1.05'`; the correct value is `'1.1'`. The discrepancy is 0.05 XCH per additional NFT with 5% royalty. function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],
): string | undefined {
  if (royaltyPercentages.length === 0 || line.kind === 'nft') {
    return undefined;
  }

  const splitAmount = amount / BigInt(royaltyPercentages.length);
  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;

  if (line.kind === 'xch') {
    return mojoToChiaLocaleString(totalAmount);
  }

  return mojoToCATLocaleString(totalAmount);
}
