### Title
EVM Refund Silently Discarded When `gas_used - evm_refund < minimal_gas_used`, Causing User Overpayment — (`basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`)

---

### Summary

In `compute_gas_refund`, the EVM storage refund (`evm_refund`) is correctly computed and subtracted from `gas_used`, but a subsequent `max(gas_used, minimal_gas_used)` floor can silently override that subtraction. When `gas_used - evm_refund < minimal_gas_used`, the floor restores `gas_used` to `minimal_gas_used`, and the user is charged as if the refund never happened. This is a direct resource-accounting analog to the external report: a deduction that should reduce what the user pays is computed but not applied.

---

### Finding Description

`compute_gas_refund` in `refund_calculation.rs` executes the following sequence:

```
gas_used  = gas_limit - remaining_ergs / ERGS_PER_GAS   // actual gas consumed
evm_refund = min(refund_counter / ERGS_PER_GAS, gas_used / 5)  // EIP-3529 cap
gas_used -= evm_refund                                   // line 48 – refund applied
gas_used  = max(gas_used, minimal_gas_used)              // line 56 – floor may undo refund
``` [1](#0-0) 

When `gas_used - evm_refund < minimal_gas_used`, the `max` on line 56 sets `gas_used` back to `minimal_gas_used`, effectively discarding the entire EVM refund. The user is then charged `minimal_gas_used × gas_price` instead of the correct `(gas_used - evm_refund) × gas_price`.

This function is called from both the ZK L2 transaction flow (`before_refund` in `zk/mod.rs`) and the Ethereum STF flow (`before_refund` in `ethereum/mod.rs`), so both paths are affected. [2](#0-1) [3](#0-2) 

The downstream fee settlement uses the corrupted `gas_used` directly:

- **ZK flow** — `token_to_refund = gas_price × (tx_gas_limit - gas_used)` and `token_to_pay_operator = gas_used × gas_price_for_operator`. Both use the inflated `gas_used`. [4](#0-3) 

- **Ethereum flow** — `refund = tx_gas_price × (tx_gas_limit - gas_used)` and `fee = priority_fee_per_gas × gas_used`. Same inflation. [5](#0-4) 

---

### Impact Explanation

A user who performs a warm `SSTORE` that clears a storage slot (EVM refund = 4 800 gas, warm access cost = 100 gas) in a transaction whose total gas usage falls in the range `[minimal_gas_used, 5/4 × minimal_gas_used)` will have their entire EVM refund silently discarded. For a standard EOA transaction (`minimal_gas_used = 21 000`):

| Value | Amount |
|---|---|
| `gas_used` | 21 100 (21 000 intrinsic + 100 warm SSTORE) |
| `evm_refund` | `min(4 800, 21 100 / 5)` = 4 220 |
| `gas_used − evm_refund` | 16 880 |
| After floor | 21 000 (floor wins) |
| User overpays | **4 120 gas × gas_price** |

The overpayment flows to the operator (or is burned under `burn_base_fee`). The user receives no compensation. The maximum overpayment per transaction is bounded by `evm_refund ≤ gas_used / 5 < minimal_gas_used / 4` (≈ 5 250 gas for a standard tx), but it is systematic and repeatable.

---

### Likelihood Explanation

The trigger condition is realistic and requires no privileged access:

1. Any unprivileged EOA can send a transaction that performs a warm `SSTORE` to zero out a previously written slot.
2. The transaction's total gas usage must fall in `[21 000, 26 250)` — achievable with a single warm SSTORE and no other significant computation.
3. No oracle manipulation, governance control, or special role is needed.

The condition is reachable on every block and is triggered by ordinary user behaviour (e.g., clearing an allowance or resetting a flag in a contract the user already interacted with in the same block).

---

### Recommendation

Apply the `minimal_gas_used` floor **before** computing and subtracting the EVM refund, so the refund is always honoured:

```rust
// 1. Apply validation floor first
let gas_used = core::cmp::max(
    gas_limit.checked_sub(resources.ergs().0.div_floor(ERGS_PER_GAS))
             .ok_or(internal_error!("gas remaining > gas limit"))?,
    minimal_gas_used,
);
resources.exhaust_ergs();

// 2. Compute refund against the floored value (preserves EIP-3529 cap semantics)
let evm_refund = {
    let full_refund_gas = system.io.get_refund_counter().ergs().0.div_floor(ERGS_PER_GAS);
    core::cmp::min(full_refund_gas, gas_used / 5)
};

// 3. Apply refund — floor already satisfied, so this cannot go below it
let gas_used = gas_used - evm_refund;
```

This ensures the EVM refund is always applied and the validation minimum is always respected.

### Proof of Concept

**Setup:** ZK L2 transaction, `gas_limit = 30 000`, `gas_price = 1 gwei`, `minimal_gas_used = 21 000`.

**Execution:**
- Transaction calls a contract that performs one warm `SSTORE` to zero out a slot (100 gas, adds 4 800 to refund counter).
- Total ergs consumed → `gas_used = 21 100`.

**`compute_gas_refund` trace (current code):**
```
gas_used  = 30000 - (30000 - 21100) = 21100
evm_refund = min(4800, 21100/5) = min(4800, 4220) = 4220
gas_used -= 4220  →  16880
gas_used  = max(16880, 21000) = 21000   ← floor silently discards refund
```

**Fee settlement:**
- User refunded: `(30 000 − 21 000) × 1 gwei = 9 000 gwei`
- Correct refund should be: `(30 000 − 16 880) × 1 gwei = 13 120 gwei`
- **User loses: 4 120 gwei per transaction** [1](#0-0)
