### Title
Unverified Participant Contributions in CKD Protocol Allow Malicious Participant to Corrupt Confidential Key Derivation Output — (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

The CKD coordinator blindly aggregates `(norm_big_y, norm_big_c)` values received from participants with no proof-of-correctness check. Any single malicious participant can substitute arbitrary group elements, causing the coordinator to assemble a corrupted `CKDOutput` that does not correspond to `msk · H(pk, app_id)`. The honest application then unmasks a wrong value, silently accepting a corrupted confidential key.

---

### Finding Description

`do_ckd_coordinator` in `src/confidential_key_derivation/protocol.rs` (lines 35–58) collects each participant's `(norm_big_y, norm_big_c)` pair and accumulates them with plain addition:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
``` [1](#0-0) 

The expected contribution from participant `i` is:

```
norm_big_y_i = λ_i · y_i · G
norm_big_c_i = λ_i · (x_i · H(pk, app_id) + y_i · app_pk)
```

computed in `compute_signature_share` (lines 148–182): [2](#0-1) 

No zero-knowledge proof, commitment binding, or consistency check is attached to the sent pair. The coordinator has no mechanism to enforce that the received elements were computed from the participant's actual signing share `x_i` and the agreed `app_pk`. A malicious participant can transmit any two `G1` points instead.

The analog to the external report is direct: just as `liquidatePortion` accepts an oracle-derived transfer amount with no caller-enforced upper bound, `do_ckd_coordinator` accepts participant-derived group elements with no coordinator-enforced correctness bound. In both cases the aggregating party cannot assert its assumptions about the inputs it receives.

---

### Impact Explanation

**High — Corruption of CKD output so honest parties accept an unusable or attacker-chosen cryptographic output.**

After aggregation the coordinator returns:

```rust
let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
Ok(Some(ckd_output))
``` [3](#0-2) 

The application then calls `ckd_output.unmask(app_sk)` expecting `msk · H(pk, app_id)`: [4](#0-3) 

If even one participant substituted arbitrary `(big_y', big_c')`, the unmasked value is `big_c' − app_sk · big_y'`, which is an attacker-chosen point unrelated to the master secret. The honest coordinator and application have no way to detect this; they accept the corrupted output as valid.

---

### Likelihood Explanation

Any single participant in the CKD session is a sufficient attacker. The participant need only deviate from the honest `compute_signature_share` computation and send two arbitrary `G1` elements. No cryptographic capability beyond participation in the protocol is required. In a decentralized MPC network — the stated deployment target — at least one participant being adversarial is a realistic and documented threat model.

---

### Recommendation

**Short term:** Require each participant to attach a Sigma-protocol (Chaum–Pedersen or equivalent) proving that `norm_big_c_i` was formed as `λ_i · (x_i · H(pk, app_id) + y_i · app_pk)` for the same `y_i` used in `norm_big_y_i = λ_i · y_i · G`, and that `x_i` is consistent with the participant's public verification share from the DKG output. The coordinator must verify all proofs before aggregating.

**Long term:** Adopt the same pattern used in the DKG (`verify_proof_of_knowledge`, `validate_received_share`) where every participant-provided value is cryptographically bound to previously committed public material before it is incorporated into the protocol state.

---

### Proof of Concept

1. Honest participants run DKG; coordinator holds `public_key` (the BLS12-381 verifying key).
2. All parties invoke `ckd(participants, coordinator, me, key_pair, app_id, app_pk, rng)`.
3. Malicious participant `p_evil` intercepts the internal send in `do_ckd_participant` and replaces the computed `(norm_big_y, norm_big_c)` with two arbitrary `G1` points `(P, Q)`.
4. `do_ckd_coordinator` receives `(P, Q)` from `p_evil` and adds them to the running sum with no check (lines 50–55 of `protocol.rs`).
5. The final `CKDOutput` contains `big_y = Σ(honest) + P` and `big_c = Σ(honest) + Q`.
6. The application calls `unmask(app_sk)` and obtains `big_c − app_sk · big_y`, a value fully controlled by the attacker's choice of `P` and `Q`, not equal to `msk · H(pk, app_id)`.
7. The honest coordinator and application accept this corrupted key with no error.

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}
