### Title
Gateway `skip_stateful_validations` Hardcodes `nonce == 1` Instead of Using `max_nonce_for_validation_skip`, Rejecting Valid Post-Deploy-Account Invoke Transactions - (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The gateway's `skip_stateful_validations` function uses the overly narrow condition `tx.nonce() == Nonce(Felt::ONE)` to decide whether to bypass blockifier validation for invoke transactions sent alongside an unprocessed `deploy_account`. This hardcoded equality check ignores the `max_nonce_for_validation_skip` configuration field that exists in `StatefulTransactionValidatorConfig` for exactly this purpose. As a result, any invoke transaction with nonce ≥ 2 (but within the allowed nonce gap) that arrives while the account has not yet been deployed is incorrectly subjected to blockifier `__validate__` execution, which fails because the account contract does not yet exist on-chain. The gateway rejects these transactions with a `ValidateFailure` error even though they are structurally valid and should be admitted to the mempool.

---

### Finding Description

The gateway's stateful path for a new account follows this sequence in `extract_state_nonce_and_run_validations`:

1. `get_nonce_from_state` → returns `account_nonce = 0` (account not yet deployed)
2. `run_pre_validation_checks` → calls `validate_nonce`, then `validate_by_mempool`, then `skip_stateful_validations`
3. `run_validate_entry_point(skip_validate)` → runs blockifier `__validate__` unless `skip_validate == true`

`validate_nonce` permits invoke transactions with nonces in the range `[account_nonce, account_nonce + max_allowed_nonce_gap]` (default gap = 200). So with `account_nonce = 0`, nonces 0–200 all pass the nonce check. [1](#0-0) 

`skip_stateful_validations` is then called to determine whether to bypass blockifier validation: [2](#0-1) 

The critical condition at line 437 is:

```rust
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
```

This hardcodes the skip to **only nonce == 1**. For any invoke with nonce ≥ 2 (e.g., nonce 2, 3, …), `skip_stateful_validations` returns `false`, causing `run_validate_entry_point` to run with `validate: true`. The blockifier then calls `__validate__` on an account that does not yet exist, producing a `ValidateFailure` error and rejecting the transaction.

`StatefulTransactionValidatorConfig` already declares a `max_nonce_for_validation_skip` field (default `Nonce(Felt::ONE)`) that is serialized into the node config schema: [3](#0-2) [4](#0-3) 

This field is **never read** by `skip_stateful_validations`, which is a free function with no access to `self.config`. The field is dead configuration.

The correct logic is implemented in the legacy `PyValidator::should_run_stateful_validations`, which uses `Nonce(Felt::ONE) <= tx_nonce && tx_nonce <= self.max_nonce_for_validation_skip`: [5](#0-4) 

The gateway's `skip_stateful_validations` is the narrower, broken version. The test suite even encodes the broken behavior as expected: [6](#0-5) 

The test case `should_not_skip_validation_nonce_over_one` asserts that nonce 2 with `account_nonce = 0` should **not** skip validation — but this is the bug, not the intended invariant.

---

### Impact Explanation

**Impact: High** — Mempool/gateway admission rejects valid invoke transactions before sequencing.

A user who submits `deploy_account` (nonce 0) + `invoke` (nonce 1) + `invoke` (nonce 2) simultaneously — a standard wallet UX pattern — will have the second invoke rejected at the gateway with a `ValidateFailure` error. The first invoke (nonce 1) is correctly admitted via the skip path. All subsequent invokes (nonce 2, 3, …, up to `max_nonce_for_validation_skip`) are incorrectly rejected. The transactions are structurally valid, pass the nonce range check, and would succeed once the deploy_account is executed.

---

### Likelihood Explanation

**Likelihood: High** — Any wallet or SDK that batches `deploy_account` with more than one invoke transaction triggers this path. The `max_allowed_nonce_gap` default of 200 means users can legitimately submit many future-nonce invokes alongside a deploy. The `max_nonce_for_validation_skip` config field being present but unused suggests this was intended to be wired up but was not.

---

### Recommendation

Pass `self.config.max_nonce_for_validation_skip` into `skip_stateful_validations` and replace the equality check with a range check, mirroring `PyValidator::should_run_stateful_validations`:

```rust
// In run_pre_validation_checks, pass the config value:
let skip_validate = skip_stateful_validations(
    executable_tx,
    account_nonce,
    self.config.max_nonce_for_validation_skip,
    mempool_client.clone(),
).await?;

// In skip_stateful_validations:
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    max_nonce_for_validation_skip: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        let tx_nonce = tx.nonce();
        if account_nonce == Nonce(Felt::ZERO)
            && Nonce(Felt::ONE) <= tx_nonce
            && tx_nonce <= max_nonce_for_validation_skip
        {
            // existing mempool check ...
        }
    }
    Ok(false)
}
```

### Proof of Concept

**Setup:** `account_nonce = 0` (account not yet deployed), `max_allowed_nonce_gap = 200`, `max_nonce_for_validation_skip = 1` (default).

**Step 1:** Submit `deploy_account` (nonce 0) → accepted, enters mempool.

**Step 2:** Submit `invoke` (nonce 1, same sender):
- `validate_nonce`: passes (0 ≤ 1 ≤ 200)
- `validate_by_mempool`: passes (deploy_account is in mempool)
- `skip_stateful_validations`: `tx.nonce() == Nonce(Felt::ONE)` → **true** → skip blockifier validation
- Result: **accepted**

**Step 3:** Submit `invoke` (nonce 2, same sender):
- `validate_nonce`: passes (0 ≤ 2 ≤ 200)
- `validate_by_mempool`: passes
- `skip_stateful_validations`: `tx.nonce() == Nonce(Felt::ONE)` → **false** (nonce is 2, not 1)
- `run_validate_entry_point` runs with `validate: true`
- Blockifier calls `__validate__` on a non-existent account → `ValidateFailure`
- Result: **rejected** ← incorrect, this is a valid transaction

The invoke with nonce 2 is a valid transaction that should be admitted. It passes the nonce range check and would execute successfully once the deploy_account is processed. The rejection is caused solely by the hardcoded `== Nonce(Felt::ONE)` condition in `skip_stateful_validations` instead of the correct `>= Nonce(Felt::ONE) && <= max_nonce_for_validation_skip` range check. async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
