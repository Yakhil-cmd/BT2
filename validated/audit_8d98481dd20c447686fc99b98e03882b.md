### Title
Replacing ETH Connector Account Without Balance Check Permanently Freezes All Bridged ETH — (`engine/src/contract_methods/connector.rs`)

---

### Summary

`set_eth_connector_contract_account` overwrites the stored connector account ID with a new one without verifying whether the old connector still holds NEP-141 ETH tokens representing user deposits. After replacement, all bridge operations route to the new connector, permanently stranding every user's bridged ETH in the old connector contract.

---

### Finding Description

`set_eth_connector_contract_account` in `engine/src/contract_methods/connector.rs` performs only two actions: it reads the new account ID from input and unconditionally overwrites the stored connector pointer. [1](#0-0) 

The two writes are: [2](#0-1) 

There is no call to the old connector to check its `ft_total_supply` or any other balance indicator before the pointer is overwritten.

Every bridge-facing method in the engine resolves the connector account at call time by reading `EthConnectorStorageId::EthConnectorAccount` through `get_connector_account_id`, then schedules a cross-contract promise to that account: [3](#0-2) 

This single `return_promise` helper is used by `withdraw`, `ft_transfer`, `ft_transfer_call`, `storage_deposit`, `storage_unregister`, and `storage_withdraw`. [4](#0-3) 

The `ExitToNear` precompile reads the same key to route ETH exits: [5](#0-4) 

The `ft_on_transfer` callback also uses the stored account ID to distinguish base-token (ETH) deposits from ERC-20 deposits: [6](#0-5) 

The engine's EVM state tracks user ETH balances internally (via `KeyPrefix::Balance`). The connector contract holds the actual NEP-141 ETH tokens that back those balances. When a user withdraws, the engine calls `engine_withdraw` on the connector, which burns NEP-141 tokens and releases ETH. After the connector pointer is replaced:

- The old connector still holds all NEP-141 ETH tokens.
- The engine still records all users' EVM ETH balances.
- Every `withdraw` call routes to the **new** connector, which holds zero NEP-141 tokens; the call fails.
- The old connector's NEP-141 tokens are permanently unreachable because the engine no longer routes to it.

Both the EVM ETH balances in the engine and the NEP-141 tokens in the old connector are permanently frozen.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

All ETH that was bridged into Aurora before the connector replacement becomes permanently inaccessible. Users cannot withdraw their EVM ETH balances (the new connector has no tokens to release), and the old connector's NEP-141 token supply is stranded with no engine-level path to recover it. The total frozen amount equals the entire ETH TVL of the Aurora bridge at the time of the replacement.

---

### Likelihood Explanation

The owner may legitimately need to replace the connector contract (e.g., to deploy a bug fix or upgrade). The function is designed for exactly this purpose and is callable by the owner at any time. Without a balance guard, a routine upgrade that is executed while users have funds deposited silently freezes all of those funds. The risk is not hypothetical: the test suite itself calls `set_eth_connector_contract_account` as a standard initialization step. [7](#0-6) 

---

### Recommendation

Before overwriting the connector account ID, query the old connector's `ft_total_supply`. If it is non-zero, reject the call with an explicit error. Only allow the replacement when the old connector holds no outstanding NEP-141 ETH tokens (i.e., all users have already withdrawn). Alternatively, provide a migration path that atomically transfers the old connector's token supply to the new connector before updating the pointer.

---

### Proof of Concept

1. Users deposit ETH into Aurora; the old connector (`old_connector.near`) accumulates NEP-141 ETH tokens; users' EVM ETH balances are recorded in the engine.
2. The owner calls `set_eth_connector_contract_account` with `new_connector.near` (a freshly deployed connector with zero NEP-141 supply).
3. `set_connector_account_id` overwrites `EthConnectorStorageId::EthConnectorAccount` with `new_connector.near`.
4. A user calls `withdraw` → the engine schedules `engine_withdraw` on `new_connector.near` → the new connector has no NEP-141 tokens → the call fails or returns zero.
5. The user's EVM ETH balance remains in the engine with no way to redeem it.
6. `old_connector.near` still holds all NEP-141 ETH tokens, but the engine never routes to it again.
7. All bridged ETH is permanently frozen.

pub fn set_eth_connector_contract_account<I: IO + Copy, E: Env>(
    io: I,
    env: &E,

    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        let is_private = env.assert_private_call();

        if is_private.is_err() {
            require_owner_only(&state, &env.predecessor_account_id())?;
        }

        let args: SetEthConnectorContractAccountArgs = io.read_input_borsh()?;

        set_connector_account_id(io, &args.account);
        set_connector_withdraw_serialization_type(io, &args.withdraw_serialize_type);

        Ok(())
    })
}

fn get_eth_connector_contract_account<I: IO>(io: &I) -> Result<AccountId, ExitError> {
    io.read_storage(&construct_contract_key(
        EthConnectorStorageId::EthConnectorAccount,
    ))
    .ok_or(ExitError::Other(Cow::Borrowed("ERR_KEY_NOT_FOUND")))
