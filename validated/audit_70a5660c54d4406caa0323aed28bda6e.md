### Title
Integer Division Truncation Zeroes `native_per_pubdata`, Making Pubdata Free When `pubdata_price < native_price` - (File: `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

The same integer-division-ordering bug class as the FiinuCoin report exists in ZKsync OS's pubdata native-resource charging. The expression `native_per_pubdata = pubdata_price / native_price` (floor integer division) truncates to **0** whenever `pubdata_price < native_price`. Every downstream pubdata charge then multiplies by zero, making pubdata completely free in native resources for any transaction submitted while this price relationship holds. An unprivileged sender can spam storage writes to exhaust the block's pubdata budget without paying the intended native cost.

---

### Finding Description

The native cost per pubdata byte is computed in two places with the same floor-division expression:

**`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`, line 142:**
```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
``` [1](#0-0) 

**`api/src/helpers.rs`, line 427:**
```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
``` [2](#0-1) 

This `native_per_pubdata` value is then used as the rate in every pubdata charge:

1. **Intrinsic pubdata overhead** (in `create_resources_for_tx`):
   ```rust
   let intrinsic_pubdata_overhead = native_per_pubdata_byte.saturating_mul(intrinsic_pubdata);
   ``` [3](#0-2) 

2. **Execution pubdata charge** (in `get_resources_to_charge_for_pubdata`):
   ```rust
   let native = current_pubdata_spent
       .checked_mul(native_per_pubdata)
       .ok_or(out_of_native_resources!())?;
   ``` [4](#0-3) 

The mathematical identity of the bug:

| Intended formula | Actual code |
|---|---|
| `pubdata_bytes × pubdata_price / native_price` | `pubdata_bytes × (pubdata_price / native_price)` |

When `pubdata_price < native_price`, the inner division `pubdata_price / native_price` truncates to **0**, so every pubdata charge evaluates to `pubdata_bytes × 0 = 0`. This is the exact structural analog of the FiinuCoin `_wei.div(raisedWei.div(targetRaiseWei))` bug: the inner division prematurely collapses to zero, silencing the outer calculation.

The `native_per_gas` computation correctly uses `div_ceil` to avoid this:
```rust
let native_per_gas = u256_try_to_u64(&gas_price.div_ceil(native_price))...
``` [5](#0-4) 

But `native_per_pubdata` uses `wrapping_div` (floor), creating an asymmetry.

---

### Impact Explanation

When `pubdata_price < native_price` (a valid operational state — the test default sets `pubdata_price = 0` and `native_price = 10`):

- `native_per_pubdata = 0`
- All pubdata native charges become zero
- A transaction can perform unlimited storage writes (each write produces pubdata) without consuming any native resources for those writes
- The block's pubdata budget (`pubdata_limit`) is consumed without the sender paying the intended native cost
- This allows a single attacker transaction to fill the block's pubdata quota, starving other transactions of pubdata capacity

The double-resource accounting model is documented to charge native resources for pubdata: [6](#0-5) 

That invariant is silently violated whenever `pubdata_price < native_price`.

---

### Likelihood Explanation

The condition `pubdata_price < native_price` is a normal operational state, not an edge case:

- The test harness default is `pubdata_price = 0`, `native_price = 10` [7](#0-6) 
- `pubdata_price` reflects L1 data-availability costs; `native_price` reflects RISC-V proving costs. These are independently set by the operator and there is no enforced ordering between them.
- Any block where `pubdata_price` is set below `native_price` activates the bug for every transaction in that block.
- No special privileges are required — any unprivileged L2 transaction sender can exploit this by submitting a pubdata-heavy transaction (e.g., writing to many storage slots).

---

### Recommendation

Replace the floor-division intermediate with a formula that preserves precision by multiplying before dividing. Two options:

**Option A — use `div_ceil` (mirrors the `native_per_gas` fix):**
```rust
// validation_impl.rs line 142
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```
This rounds up to 1 when `0 < pubdata_price < native_price`, ensuring pubdata is never free when `pubdata_price > 0`.

**Option B — defer the division to the charge site:**
Instead of storing a pre-divided rate, compute `pubdata_bytes * pubdata_price / native_price` directly at each charge site, preserving full precision.

The same fix must be applied in `api/src/helpers.rs` line 427 to keep the off-chain validation helper consistent with the bootloader. [2](#0-1) 

---

### Proof of Concept

**Setup:**
- `native_price = 1000`
- `pubdata_price = 999` (just below `native_price`)
- Transaction writes to 500 storage slots → ~500 × 64 = 32,000 pubdata bytes

**Actual behavior (buggy):**
```
native_per_pubdata = 999 / 1000 = 0   (floor division)
pubdata_charge     = 32000 × 0   = 0  native units
```
The attacker pays **zero** native resources for 32,000 bytes of pubdata.

**Expected behavior (correct):**
```
pubdata_charge = 32000 × 999 / 1000 = 31,968  native units
```

**Exploit path:**
1. Observe that the operator has set `pubdata_price < native_price` (e.g., `pubdata_price = 999`, `native_price = 1000`).
2. Submit a transaction that writes to many storage slots (e.g., a loop writing to 500 distinct keys).
3. The transaction's pubdata native charge is 0 due to the truncation.
4. The block's `pubdata_limit` is consumed without the sender paying native resources for it.
5. Repeat across multiple transactions to exhaust the block's pubdata budget, preventing other users' pubdata-producing transactions from being included.
