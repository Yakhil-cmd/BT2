### Title
Missing Minimum Threshold Lower-Bound Check in FROST Signing Path Allows Corrupted Sign Outputs - (File: `src/frost/mod.rs`)

### Summary
The `assert_sign_inputs` and `presign` functions in `src/frost/mod.rs` enforce an upper-bound check on the threshold (`threshold > participants.len()`) but omit the lower-bound check (`threshold < 2`). This is inconsistent with every other protocol entry point in the codebase, which all enforce a minimum threshold of 2. A malicious coordinator or misconfigured caller can pass `threshold = 0` or `threshold = 1` into the FROST signing path without triggering any validation error, causing the signing protocol to proceed with an invalid threshold and produce unusable or corrupted signature outputs.

### Finding Description

In `src/dkg.rs`, `assert_key_invariants` explicitly enforces the minimum threshold:

```rust
if threshold < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
}
``` [1](#0-0) 

Likewise, `validate_triple_inputs` in the OT-based ECDSA triple generation path enforces the same lower bound:

```rust
if threshold_value < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold_value,
        min: 2,
    });
}
``` [2](#0-1) 

However, in `src/frost/mod.rs`, the `presign` function only checks the upper bound:

```rust
// validate threshold
if args.threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
``` [3](#0-2) 

And `assert_sign_inputs` — the shared validation gate for FROST EdDSA and RedJubjub signing — has the same gap:

```rust
// validate threshold
if threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
``` [4](#0-3) 

There is no corresponding `ThresholdTooSmall` guard anywhere in the FROST signing path. The `ReconstructionLowerBound` and `MaxMalicious` wrapper types themselves impose no minimum:

```rust
pub struct ReconstructionLowerBound(usize);
impl ReconstructionLowerBound {
    pub fn value(self) -> usize { self.0 }
}
``` [5](#0-4) 

### Impact Explanation

With `threshold = 1` accepted silently by `assert_sign_inputs`, the FROST signing protocol proceeds with only one participant required. Because the key was generated with `threshold >= 2` (enforced by DKG), the Lagrange coefficients computed during signing are inconsistent with the key's polynomial degree. The aggregated partial signatures will not reconstruct the correct group element, producing an invalid, unusable signature. Honest participants complete the protocol and receive a corrupted output they cannot use for any downstream purpose.

With `threshold = 0`, Lagrange interpolation over an empty signer set is undefined and may cause a panic or silent arithmetic failure inside the FROST aggregation layer.

**Impact class:** High — Corruption of sign outputs so honest parties accept unusable cryptographic outputs.

### Likelihood Explanation

The FROST signing entry points (`presign`, `assert_sign_inputs`) are public API surfaces callable by any library user or coordinator. A malicious coordinator controlling the threshold argument — or a misconfigured integration that passes a wrong value — can trigger this path without any special privilege. The inconsistency with the DKG path means developers familiar with the DKG validation pattern would not expect this gap to exist.

### Recommendation

Add the same lower-bound guard present in `assert_key_invariants` and `validate_triple_inputs` to both `presign` and `assert_sign_inputs` in `src/frost/mod.rs`:

```rust
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

This should be inserted immediately after the `threshold.into()` conversion, before any other threshold comparison, mirroring the pattern in `src/dkg.rs` line 580. [6](#0-5) 

### Proof of Concept

```rust
// Calling assert_sign_inputs with threshold = 1 and 2 valid participants
// passes all validation and returns Ok — no error is raised.
let participants = vec![participant_a, participant_b];
let result = assert_sign_inputs(
    &participants,
    ReconstructionLowerBound::from(1usize), // invalid: below minimum of 2
    participant_a,
    participant_b,
);
assert!(result.is_ok()); // passes — missing lower-bound check

// The same call with threshold = 0 also passes validation.
let result2 = assert_sign_inputs(
    &participants,
    ReconstructionLowerBound::from(0usize),
    participant_a,
    participant_b,
);
assert!(result2.is_ok()); // passes — undefined behavior downstream
```

The signing protocol then proceeds with the invalid threshold, computing Lagrange coefficients over a signer set inconsistent with the key's polynomial degree, and produces a signature that fails verification. [7](#0-6)

/// Verifies that the sign inputs are valid
pub fn assert_sign_inputs(
    participants: &[Participant],
    threshold: impl Into<ReconstructionLowerBound>,
    me: Participant,
    coordinator: Participant,
) -> Result<ParticipantList, InitializationError> {
    let threshold = threshold.into();
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // validate threshold
    if threshold.value() > participants.len() {
        return Err(InitializationError::ThresholdTooLarge {
            threshold: threshold.value(),
            max: participants.len(),
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }
    Ok(participants)
}
```
