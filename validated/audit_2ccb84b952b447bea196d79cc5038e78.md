### Title
Missing Zero-Address Validation in `change_reward_address` Enables Permanent Loss of Unclaimed Yield - (File: `src/staking/staking.cairo`, `src/pool/pool.cairo`)

---

### Summary
Both `Staking::change_reward_address` and `Pool::change_reward_address` accept a `reward_address` parameter but only validate that it is not a token address. There is no zero-address check. Any staker or pool member can set their reward address to `ContractAddress::zero()`, causing all subsequent reward claims to transfer tokens to the zero address, permanently destroying unclaimed yield.

---

### Finding Description

In `staking.cairo`, `change_reward_address` performs only one validation — that the new address is not a registered token — before writing it to storage: [1](#0-0) 

The only guard is `REWARD_ADDRESS_IS_TOKEN`. There is no `assert!(reward_address.is_non_zero(), ...)` check. The zero address passes this guard because it is not a registered token.

The same pattern exists in `pool.cairo`: [2](#0-1) 

Again, only the token-address check is present; zero address is accepted.

When `claim_rewards` is later called on the staking contract, it reads `staker_info.reward_address` and transfers the accumulated rewards there: [3](#0-2) 

If `reward_address` is zero, the ERC-20 transfer goes to address(0), permanently destroying the tokens. The same applies in `unstake_action`, which calls `send_rewards_to_staker` before returning the principal: [4](#0-3) 

The existing test suite confirms no zero-address test exists for `change_reward_address` — only `STAKER_NOT_EXISTS` and `REWARD_ADDRESS_IS_TOKEN` are tested: [5](#0-4) 

---

### Impact Explanation
**High — Permanent freezing of unclaimed yield.**

A staker who (accidentally or via griefing) sets `reward_address = 0` will have all future `claim_rewards` and `unstake_action` reward payouts sent to address(0). Those tokens are irrecoverable. The same applies to pool members via `Pool::change_reward_address`. The principal stake is unaffected, but all accrued yield is permanently destroyed.

---

### Likelihood Explanation
**Medium.** The function is callable by any active staker or pool member with no privileged role required. Accidental misuse (e.g., passing an uninitialized variable) is realistic. Intentional griefing of one's own rewards is also possible. The call path is short and requires no special setup beyond being a registered staker or pool member.

---

### Recommendation

Add a zero-address assertion in both `change_reward_address` implementations:

```cairo
// staking.cairo
fn change_reward_address(ref self: ContractState, reward_address: ContractAddress) {
    self.general_prerequisites();
+   assert!(reward_address.is_non_zero(), "{}", GenericError::ZERO_ADDRESS);
    assert!(
        !self.does_token_exist(token_address: reward_address),
        "{}",
        GenericError::REWARD_ADDRESS_IS_TOKEN,
    );
    ...
}

// pool.cairo
fn change_reward_address(ref self: ContractState, reward_address: ContractAddress) {
+   assert!(reward_address.is_non_zero(), "{}", GenericError::ZERO_ADDRESS);
    assert!(
        self.token_dispatcher.contract_address.read() != reward_address,
        "{}",
        GenericError::REWARD_ADDRESS_IS_TOKEN,
    );
    ...
}
```

The same zero-address check should also be applied to the `reward_address` parameter in `stake()` and `enter_delegation_pool()` for defense-in-depth.

---

### Proof of Concept

**Staking contract path:**
1. Staker is registered and has accrued rewards.
2. Staker calls `staking.change_reward_address(reward_address: ContractAddress::zero())`.
3. No assertion fires; `staker_info.reward_address` is written as `0`.
4. Attestation occurs; `update_rewards_from_attestation_contract` credits `unclaimed_rewards_own`.
5. Anyone calls `staking.claim_rewards(staker_address)`.
6. The contract transfers the reward amount to address(0) — tokens are permanently lost.

**Pool contract path:**
1. Pool member is registered and has accrued rewards.
2. Pool member calls `pool.change_reward_address(reward_address: ContractAddress::zero())`.
3. No assertion fires; `pool_member_info.reward_address` is written as `0`.
4. Pool member calls `pool.claim_rewards(pool_member)`.
5. Rewards are transferred to address(0) — permanently lost.
