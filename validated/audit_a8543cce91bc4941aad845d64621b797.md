### Title
Off-by-One in Threshold Upper Bound Check Allows `threshold == N`, Enabling Permanent Denial of Signing — (File: `src/dkg.rs`)

---

### Summary

`assert_key_invariants` in `src/dkg.rs` uses a non-strict inequality (`>`) to guard the upper bound of the reconstruction threshold, allowing callers to pass `threshold == N` (where N is the total participant count). The protocol specification explicitly requires `threshold < N` (strictly less than N). When `threshold == N` is accepted, the resulting key material requires all N participants to be present for every signing session; any single participant who becomes unavailable or refuses to cooperate causes a permanent, unrecoverable denial of signing for all honest parties.

---

### Finding Description

`assert_key_invariants` is the shared pre-flight validation function called by `keygen`, `reshare`, and `refresh` before any protocol work begins.

```rust
// src/dkg.rs  lines 572-578
// Step 1.1
// validate threshold
if threshold > participants.len() {
    return Err(InitializationError::ThresholdTooLarge {
        threshold,
        max: participants.len(),
    });
}
```

The condition `threshold > participants.len()` only rejects when `threshold > N`; it silently accepts `threshold == N`.

The protocol specification in `docs/dkg.md` is unambiguous:

> **1.1** Each $P_i$ asserts that $1 < \mathsf{threshold} < N$.

The strict upper bound `threshold < N` is a documented invariant. The code enforces the lower bound correctly (`threshold < 2` → error) but uses a non-strict comparison for the upper bound, creating an off-by-one gap at the boundary `threshold == N`.

When `threshold == N` is accepted:

1. `do_keyshare` generates a polynomial of degree `N − 1` (line 373–376 in `src/dkg.rs`).
2. Reconstruction of the master secret requires all N evaluations.
3. Every subsequent signing call (`ecdsa::ot_based_ecdsa::sign`, `ecdsa::robust_ecdsa::sign`, `frost::sign`) must gather responses from all N participants to produce a valid signature.
4. A single participant who drops out, crashes, or deliberately withholds their share makes signing permanently impossible without re-running the full DKG.

The same non-strict check is replicated in `assert_sign_inputs` in `src/frost/mod.rs` (line 145), compounding the exposure for FROST signing sessions.

---

### Impact Explanation

**High — Permanent denial of signing for honest parties.**

Once a key is generated with `threshold == N`, no subset smaller than N can sign. Any single participant — whether unavailable due to failure or acting maliciously — can unilaterally and permanently block all signing operations for the entire group. The only recovery path is a full DKG re-run, which itself requires all N participants to cooperate. The denial is therefore self-reinforcing: the same participant who blocked signing can also block the recovery reshare.

---

### Likelihood Explanation

**Medium.**

The library exposes `threshold` as a plain caller-supplied integer with no documentation warning against `threshold == N` at the API boundary. A library integrator who interprets the parameter as "minimum signers" and sets it equal to the full participant count — a natural mistake — silently produces a permanently fragile key. A malicious coordinator or a participant who learns the threshold was set to N can then exploit the situation by simply going offline at signing time.

---

### Recommendation

Change the upper-bound guard in `assert_key_invariants` from a non-strict to a strict comparison, matching the specification:

```rust
// src/dkg.rs  ~line 573
// Before (buggy):
if threshold > participants.len() {

// After (correct):
if threshold >= participants.len() {
    return Err(InitializationError::ThresholdTooLarge {
        threshold,
        max: participants.len() - 1,   // document the true maximum
    });
}
```

Apply the same fix to `assert_sign_inputs` in `src/frost/mod.rs` (line 145).

---

### Proof of Concept

```
Setup:
  participants = [P1, P2, P3]   (N = 3)
  threshold    = 3              // allowed by current code; spec forbids it

Step 1: keygen(participants, me=P1, threshold=3, rng) → Ok(protocol)
        Protocol completes; all three parties hold valid shares.

Step 2: P3 goes offline (hardware failure, network partition, or deliberate refusal).

Step 3: P1 and P2 attempt sign(participants=[P1,P2,P3], threshold=3, ...)
        → Protocol blocks waiting for P3's share.
        → P3 never responds.
        → Signing never completes.

Step 4: P1 and P2 attempt reshare to recover:
        reshare(old_participants=[P1,P2,P3], old_threshold=3, ...)
        → assert_reshare_keys_invariants checks intersection.len() >= old_threshold
        → intersection = {P1, P2} → len = 2 < 3 = old_threshold
        → Err(NotEnoughParticipantsForNewThreshold)
        → Recovery is also permanently blocked.

Result: Honest parties P1 and P2 can never sign or recover without P3.
