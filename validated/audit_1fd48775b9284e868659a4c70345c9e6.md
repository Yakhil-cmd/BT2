### Title
Malicious Participant Can Permanently Deny Triple Generation for Targeted Honest Party via Unverified Private Share Injection — (`src/ecdsa/ot_based_ecdsa/triples/generation.rs`)

### Summary

In `do_generation_many`, private polynomial evaluations received at `wait3` are blindly accumulated into a running sum with no per-sender verification. The aggregate consistency check at lines 424–430 can only detect that *something* went wrong, not *who* caused it. A single malicious participant can send a random scalar instead of the correct evaluation, causing the targeted honest party to abort with `ProtocolError::AssertionFailed("received bad private share")` every time, with no ability to identify or exclude the attacker.

### Finding Description

**Step 1 — Polynomial commitments are verified (wait2)**

At lines 331–396, each party's polynomial commitments `big_E_j`, `big_f_j`, `big_l_j` are received and individually verified: the Pedersen commitment is opened, and two dlog proofs are checked per sender. After this loop, `big_e_v[i] = Σ_j big_E_j` is a sum of authenticated commitments. [1](#0-0) 

**Step 2 — Private shares are sent privately (wait3)**

At lines 283–296, each party sends `e.eval_at_participant(p)` privately to each peer `p` via `chan.send_private`. These are the scalar evaluations of each party's secret polynomial at the recipient's index. [2](#0-1) 

**Step 3 — Private shares are accumulated with no per-sender check and sender identity is discarded**

At lines 400–412, the received shares are summed into `a_i_v` and `b_i_v`. The `from` identifier is explicitly discarded (`_`). There is no check that `G * a_j_i == big_E_j.eval_at_participant(me)` for each individual sender `j`. [3](#0-2) 

**Step 4 — The aggregate check fires and aborts the honest party**

At lines 424–430, the check is:

```
big_e.eval_at_participant(me) == G * a_i
```

where `big_e = Σ_j big_E_j` and `a_i = Σ_j a_j_i`. This is equivalent to checking `Σ_j(big_E_j.eval_at_participant(me)) == Σ_j(G * a_j_i)`. If any single `a_j_i` is wrong, the sum is poisoned and the check fails. The honest party returns `ProtocolError::AssertionFailed("received bad private share")` with no information about which party caused the failure. [4](#0-3) 

**Contrast with DKG**, which correctly performs per-sender share verification and names the culprit: [5](#0-4) 

### Impact Explanation

A malicious participant M can:
1. Participate honestly through `wait2` (sending valid polynomial commitments that pass all verification).
2. At `wait3`, send a random scalar to a targeted honest party H instead of `e_M.eval_at_participant(H)`.
3. H's aggregate check at line 424 fails; H aborts permanently.
4. M can repeat this every protocol invocation, targeting H indefinitely.
5. H has no way to identify M as the culprit and cannot exclude M from future runs without external coordination.

This is **permanent denial of triple generation** for the targeted honest party, which directly blocks all downstream signing operations that depend on triples.

### Likelihood Explanation

The attack requires only that a participant deviate from the protocol at a single private message step. No cryptographic assumption needs to be broken. The malicious party passes all public verification (commitments, dlog proofs) and only corrupts the private scalar sent to the target. The attack is trivially repeatable and requires no special capability beyond being a registered participant.

### Recommendation

Before accumulating received shares, verify each one individually against the sender's already-authenticated polynomial commitment:

```rust
for (from, (a_j_i_v, b_j_i_v)) in recv_from_others::<(...)>(&chan, wait3, &participants, me).await? {
    for i in 0..N {
        let big_e_j = /* retrieve big_E_j for `from` */;
        if big_e_j.eval_at_participant(me)?.value() != ProjectivePoint::GENERATOR * a_j_i_v[i].0 {
            return Err(ProtocolError::AssertionFailed(
                format!("bad private share from {from:?}")
            ));
        }
        a_i_v[i] += &a_j_i_v[i].0;
        b_i_v[i] += &b_j_i_v[i].0;
    }
}
```

This mirrors the pattern already used in `src/dkg.rs` (`validate_received_share`), which correctly identifies the culprit and allows the honest parties to exclude the malicious one and retry.

### Proof of Concept

In a 3-party test, replace one party's `a_i_j_v` for a specific target with a random scalar before sending at `wait3`. The target party will return `ProtocolError::AssertionFailed("received bad private share")` while the other two parties complete successfully. The malicious party's polynomial commitments pass all verification at `wait2`, so the attack is undetectable until the aggregate check fires — at which point the culprit's identity is already lost. fn validate_received_share<C: Ciphersuite>(
    me: Participant,
    from: Participant,
    signing_share_from: &SigningShare<C>,
    commitment: &VerifiableSecretSharingCommitment<C>,
) -> Result<(), ProtocolError> {
    let id = me.to_identifier::<C>()?;

    // The verification is exactly the same as the regular SecretShare verification;
    // however the required components are in different places.
    // Build a temporary SecretShare so what we can call verify().
    let secret_share = SecretShare::new(id, *signing_share_from, commitment.clone());

    // Verify the share. We don't need the result.
    // Identify the culprit if an InvalidSecretShare error is returned.
    secret_share.verify().map_err(|e| {
        if let Error::InvalidSecretShare { .. } = e {
            ProtocolError::InvalidSecretShare(from)
        } else {
            ProtocolError::AssertionFailed(format!(
                "could not
            extract the verification key matching the secret
            share sent by {from:?}"
            ))
        }
    })?;
    Ok(())
