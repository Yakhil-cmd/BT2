### Title
Omitted Participant-Set Binding in OT-Based ECDSA Presigning Allows Cross-Session Triple Misuse - (File: `src/ecdsa/ot_based_ecdsa/presign.rs`)

---

### Summary

The `presign()` function in the OT-based ECDSA module explicitly omits the check that the presigning participant set matches the participant set recorded inside the `TriplePub` struct at triple-generation time. `TriplePub` stores `participants` precisely to enable this binding, but the binding is never enforced. A malicious coordinator can supply triples generated for one participant set while running presigning with a different participant set, causing honest parties to compute Lagrange-linearized shares over the wrong domain and producing an unusable (aborted) presign output.

---

### Finding Description

`TriplePub` records the exact participant set and threshold used during triple generation: [1](#0-0) 

This value is populated at the end of `do_generation_many` and stored in every returned `TriplePub`: [2](#0-1) 

In `presign()`, the threshold is cross-checked against both triples: [3](#0-2) 

However, the participant set stored in `TriplePub.participants` is **never compared** against the `participants` argument passed to `presign()`. The code explicitly acknowledges this omission with a comment: [4](#0-3) 

Inside `do_presign`, Lagrange coefficients are computed from the presigning `participants` list, not from the triple's participant set: [5](#0-4) 

If the presigning participant set differs from the triple's participant set, the Lagrange linearization of `k_i`, `e_i`, `a_i`, `b_i`, and `x_i` is computed over the wrong domain. The subsequent consistency checks (`big_e == e*G` and `alpha*G == K + A`) will fail, causing all honest parties to abort: [6](#0-5) 

---

### Impact Explanation

This is a **participant-set mismatch** bug. The `TriplePub.participants` field is the analog of the `txType` registration in the external report: it is stored during initialization (triple generation) but never enforced during execution (presigning). A malicious coordinator can supply triples generated for participant set P1 while running presigning with participant set P2. Every honest party will compute Lagrange coefficients for P2 but apply them to shares that are valid only under P1. The aggregate reconstruction fails the public-commitment checks, and all honest parties abort. The presign output is permanently corrupted/unusable for that session.

**Impact category:** High — Corruption of presign outputs so honest parties accept unusable cryptographic outputs (aborted presignature), matching the allowed scope: *"Corruption of DKG, reshare, refresh, presign, sign, or CKD outputs so honest parties accept inconsistent public keys, participant sets, transcripts, or unusable cryptographic outputs."*

---

### Likelihood Explanation

The `presign()` function is a public API. Any application-level coordinator that assembles `PresignArguments` controls both the `participants` slice and the `TriplePub` values inside `triple0` and `triple1`. A malicious coordinator (or a compromised orchestration layer) can trivially supply mismatched values. No cryptographic capability is required — only the ability to call `presign()` with caller-controlled arguments, which is the normal usage pattern of the library.

---

### Recommendation

Add an explicit check in `presign()` that the presigning `participants` set equals `args.triple0.1.participants` and `args.triple1.1.participants` before proceeding. The `TriplePub` struct already carries this information for exactly this purpose. The check should be added immediately after the threshold check at line 43, replacing the current comment with an enforced invariant:

```rust
// Enforce that the presigning participant set matches the triple's participant set.
let triple0_participants = ParticipantList::new(&args.triple0.1.participants)
    .ok_or(InitializationError::DuplicateParticipants)?;
let triple1_participants = ParticipantList::new(&args.triple1.1.participants)
    .ok_or(InitializationError::DuplicateParticipants)?;
if participants != triple0_participants || participants != triple1_participants {
    return Err(InitializationError::BadParameters(
        "Presigning participant set must match the participant set used during triple generation".to_string(),
    ));
}
```

---

### Proof of Concept

1. Run triple generation with participant set P1 = {A, B, C}, threshold 2. Each party receives a `(TripleShare, TriplePub)` where `TriplePub.participants == [A, B, C]`.
2. Initiate presigning with participant set P2 = {A, B, D} (same threshold, different third member), passing the triples from step 1.
3. `presign()` accepts the call — the threshold check at line 43 passes because `args.threshold == args.triple0.1.threshold`. The participant-set check is omitted.
4. Inside `do_presign`, each party computes `lambda_me = participants.lagrange(me)` using P2's Lagrange basis, but their triple shares `k_i`, `e_i`, `a_i`, `b_i` are valid only under P1's Lagrange basis.
5. The aggregate `e = Σ lambda_j(P2) * e_j` does not equal the true secret `kd`, so `big_e ≠ e*G`. All honest parties hit the assertion at line 127 and abort with `ProtocolError::AssertionFailed`, permanently destroying the presigning session. [7](#0-6) [1](#0-0) [8](#0-7)
