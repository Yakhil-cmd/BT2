### Title
Unrestricted `disable_rewards` Parameter in `update_rewards` Permanently Consumes Per-Block Reward Slot Without Distributing Rewards — (File: src/staking/staking.cairo)

---

### Summary

The `update_rewards` function in the staking contract is publicly callable by any address. It enforces a global, single-use-per-block slot via `last_reward_block`. Critically, `last_reward_block` is written to storage **before** the `disable_rewards` guard is evaluated. Any caller can pass `disable_rewards = true` to consume the block's reward slot without distributing any rewards, permanently denying all stakers their consensus-phase block rewards for that block.

---

### Finding Description

`update_rewards` is the sole entry point for consensus-phase per-block reward distribution: [1](#0-0) 

The function enforces a global one-call-per-block invariant: [2](#0-1) 

After validating the staker, it unconditionally commits the current block number to `last_reward_block`: [3](#0-2) 

Only **after** that write does it check `disable_rewards`: [4](#0-3) 

Because `last_reward_block` is a **global** storage slot (not per-staker), one successful call per block exhausts the slot for every staker: [5](#0-4) 

There is no access control on `update_rewards`; `general_prerequisites` only checks the contract is unpaused and the caller is non-zero: [6](#0-5) 

The interface confirms the function is fully public: [7](#0-6) 

---

### Impact Explanation

In the consensus phase (after `consensus_rewards_first_epoch` is reached), `update_rewards` is the only mechanism through which per-block STRK and BTC rewards are distributed to stakers. Because `last_reward_block` is global and is written before the `disable_rewards` branch, a single call with `disable_rewards = true` per block permanently voids that block's reward distribution for **all** stakers. The rewards are never minted or accrued; they are irreversibly lost. This constitutes **permanent freezing of unclaimed yield** — a High-severity impact under the allowed scope.

---

### Likelihood Explanation

The attack requires no special role, no leaked key, and no privileged access. Any EOA or contract can call `update_rewards(any_active_staker, true)`. The attacker must front-run the legitimate staker's call in each block, which is straightforward on Starknet where transaction ordering is observable in the mempool. The attacker spends gas but gains nothing; this is a pure griefing vector. Likelihood is **Medium** (sustained cost, but trivially executable by any motivated actor).

---

### Recommendation

Move the `last_reward_block` write to **after** the `disable_rewards` / `is_pre_consensus` guard, so that a no-op call does not consume the block's reward slot:

```cairo
// Only mark the block as processed when rewards are actually distributed.
if disable_rewards || self.is_pre_consensus() {
    return;
}
self.last_reward_block.write(current_block_number); // moved here
// ... distribute rewards ...
```

Alternatively, restrict who may pass `disable_rewards = true` (e.g., only the staker or their operational address).

---

### Proof of Concept

1. Consensus rewards are active (`current_epoch >= consensus_rewards_first_epoch`).
2. Attacker calls `update_rewards(any_valid_active_staker, disable_rewards: true)` at the start of block N.
3. The check `current_block_number > last_reward_block` passes (line 1455–1458).
4. `last_reward_block` is written to `N` (line 1485).
5. The function returns early at line 1487 — no rewards distributed, no `_update_rewards` called.
6. Every subsequent call to `update_rewards` in block N reverts with `REWARDS_ALREADY_UPDATED`.
7. Block N's per-block rewards are permanently lost for all stakers.
8. Repeating this every block permanently halts all consensus reward distribution. [8](#0-7)
