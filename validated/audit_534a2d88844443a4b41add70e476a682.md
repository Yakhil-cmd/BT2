### Title
Gateway Skips `__validate__` Signature Check for Any Invoke with Nonce=1 When Target Account Has Any Mempool Entry — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

`skip_stateful_validations` in the gateway's stateful validator unconditionally skips the `__validate__` entry-point call — and therefore all account-contract signature verification — for any invoke transaction whose nonce is 1 and whose sender address appears in the mempool or a recent committed block. An unprivileged attacker who observes a pending `deploy_account` transaction for address A can immediately submit a second invoke transaction for A with nonce=1 and a completely arbitrary signature; the gateway admits it to the mempool without ever calling `__validate__`.

### Finding Description

`skip_stateful_validations` is the UX helper that lets a user broadcast `deploy_account` + `invoke(nonce=1)` atomically before the account is on-chain. Its guard is:

```rust

    return mempool_client
        .account_tx_in_pool_or_recent_block(tx.sender_address())
        .await
        ...;   // returns true → skip_validate = true
}

Inside `AccountTransaction::validate_tx`, the very first thing done when `validate == false` is an early return with no signature check:

```rust

The mempool's own `validate_tx` path (`validate_incoming_tx` → `validate_fee_escalation`) checks only for duplicate hashes and nonce ordering; it never inspects the signature:

```rust
// crates/apollo_mempool/src/mempool.rs  lines 402-408
pub fn validate_tx(&mut self, args: ValidationArgs) -> MempoolResult<()> {
    let tx_reference = (&args).into();
    self.validate_incoming_tx(tx_reference, args.account_nonce)?;
    self.validate_fee_escalation(tx_reference)?;
    Ok(())
}
``` [4](#0-3) 

The `account_tx_in_pool_or_recent_block` predicate is satisfied by **any** transaction from the target address — including the legitimate owner's `deploy_account` tx that is already in the pool:

```rust
// crates/apollo_mempool/src/mempool.rs  lines 697-700
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [5](#0-4) 

There is no check that the incoming invoke transaction was signed by the same key that signed the `deploy_account` transaction already in the pool.

### Impact Explanation

An attacker who observes a pending `deploy_account` for address A (visible via P2P gossip or the public mempool) can submit an `InvokeV3` transaction for A with `nonce=1` and a garbage signature. The gateway's stateful validator will:

1. Confirm `account_nonce == 0` (account not yet on-chain).
2. Confirm `account_tx_in_pool_or_recent_block(A) == true` (the legitimate `deploy_account` is in the pool).
3. Set `skip_validate = true`, bypassing `__validate__`.
4. Admit the transaction to the mempool.

The admitted transaction carries an invalid signature and arbitrary calldata. This matches the **High** impact criterion: *"Mempool/gateway/RPC admission accepts invalid transactions … before sequencing."* [6](#0-5) 

Secondary consequences:
- The attacker's transaction competes with the legitimate user's `invoke(nonce=1)` for the same nonce slot; with a higher tip it can displace the legitimate transaction via fee escalation.
- The batcher wastes execution resources attempting to run `__validate__` on the invalid transaction (which will panic/revert at execution time).
- Repeated submissions constitute a low-cost DoS against any account in the deploy-account-pending state.

### Likelihood Explanation

`deploy_account` transactions are broadcast over the P2P network and are visible to any node. The attack requires no privileged access: any observer can extract the sender address, craft an invoke with nonce=1, and submit it to the gateway. The condition window lasts until the `deploy_account` is committed to a block.

### Recommendation

Replace the coarse `account_tx_in_pool_or_recent_block` check with a check that specifically confirms a **`deploy_account`** transaction (not just any transaction) is pending for the sender address. For example, expose a `has_pending_deploy_account(address)` query from the mempool, or store the deploy-account tx hash alongside the account state and verify the type before skipping validation.

Alternatively, require that the invoke transaction's signature be verifiable against the public key embedded in the pending `deploy_account` constructor calldata, so that only the legitimate account owner can benefit from the skip.

### Proof of Concept

1. Legitimate user broadcasts `deploy_account` for address A (class_hash C, salt S, public_key P). This tx enters the mempool; `account_tx_in_pool_or_recent_block(A)` now returns `true`.
2. Attacker observes A in the mempool.
3. Attacker submits `InvokeV3 { sender_address: A, nonce: 1, calldata: [<arbitrary>], signature: [0x1, 0x2] }` to the gateway.
4. Gateway stateless validator passes (valid address, resource bounds, etc.).
5. `extract_state_nonce_and_run_validations` fetches `account_nonce = 0` from chain state.
6. `run_pre_validation_checks` → `validate_nonce` passes (0 ≤ 1 ≤ max_allowed_nonce_gap).
7. `skip_stateful_validations` returns `true` (nonce==1, account_nonce==0, account in pool).
8. `run_validate_entry_point` sets `validate: false`; `validate_tx` returns `Ok(None)` — **no `__validate__` call, no signature check**.
9. Transaction is forwarded to the mempool and admitted.
10. Attacker's transaction now occupies nonce=1 for address A in the mempool, potentially displacing the legitimate user's invoke via fee escalation. [7](#0-6) [8](#0-7)

        versioned_constants.disable_casm_hash_migration();

        block_info.block_number = block_info.block_number.unchecked_next();

            block_info,
            self.chain_info.clone(),
            versioned_constants,
            BouncerConfig::max(),

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
