### Title
Unprivileged Caller Can Permanently Suppress Consensus Block Rewards via Unguarded `disable_rewards` Parameter — (File: `src/staking/staking.cairo`)

---

### Summary

The `IStakingRewardsManager::update_rewards` function accepts a caller-supplied `disable_rewards: bool` parameter with no access control. Because the contract enforces a strict one-call-per-block invariant via `last_reward_block`, any unprivileged address can call `update_rewards(valid_staker, disable_rewards: true)` at every block to consume the per-block slot without distributing rewards, permanently suppressing all consensus block rewards for all stakers and delegators.

---

### Finding Description

`update_rewards` is the entry point for distributing consensus block rewards in the V3 staking system. Its only access guard is `general_prerequisites()`, which checks only that the contract is not paused and the caller is non-zero: [1](#0-0) 

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
    let current_block_number = starknet::get_block_number();
    assert!(
        current_block_number > self.last_reward_block.read(),
        "{}",
        Error::REWARDS_ALREADY_UPDATED,
    );
```

After the block-uniqueness check passes, `last_reward_block` is unconditionally written to the current block number: [2](#0-1) 

```cairo
    // Update last block rewards.
    self.last_reward_block.write(current_block_number);

    if disable_rewards || self.is_pre_consensus() {
        return;
    }
```

When `disable_rewards` is `true`, the function returns immediately after writing `last_reward_block`, skipping all reward calculation and distribution. Because `last_reward_block` is now set to the current block, **no subsequent call can distribute rewards for that block** — the `REWARDS_ALREADY_UPDATED` assertion will revert any further attempt.

`general_prerequisites` imposes no role restriction: [3](#0-2) 

```cairo
fn general_prerequisites(ref self: ContractState) {
    self.assert_is_unpaused();
    assert_caller_is_not_zero();
}
```

The attacker only needs to supply any currently-active staker address with non-zero balance (public on-chain information) to pass the staker validity checks: [4](#0-3) 

This is directly analogous to the BribeVault M-10 finding: a parameter (`disable_rewards`) is blindly accepted from any caller without being validated against the actual protocol state (whether rewards are legitimately owed), allowing an unprivileged actor to corrupt the reward accounting for every block.

---

### Impact Explanation

**High — Permanent freezing of unclaimed yield.**

An attacker who calls `update_rewards(valid_staker, disable_rewards: true)` once per block permanently destroys the block rewards for that block for every staker and delegator in the system. Automating this across all blocks effectively zeroes out all consensus reward income indefinitely. The rewards are not deferred — they are computed per-block and the per-block slot is irrecoverably consumed.

---

### Likelihood Explanation

**High.** The attack requires:
- No special role or privilege.
- No capital at risk.
- Only a valid staker address (trivially obtained from on-chain events).
- One transaction per block.

The attacker has no profit motive but can inflict maximum damage on all stakers and delegators at negligible cost.

---

### Recommendation

Restrict who may pass `disable_rewards: true`. Either:

1. **Remove the parameter entirely** and derive the disable condition purely from on-chain state (e.g., `is_pre_consensus()`), or
2. **Gate the `disable_rewards: true` path behind a trusted role** (e.g., `only_security_agent` or `only_attestation_contract`), so that unprivileged callers can only invoke the function with `disable_rewards: false`.

---

### Proof of Concept

```
For each new block N:
  1. Attacker calls: staking.update_rewards(valid_staker_address, disable_rewards=true)
  2. Contract checks: current_block_number (N) > last_reward_block → passes
  3. Contract writes: last_reward_block = N
  4. Contract hits: `if disable_rewards { return; }` → exits with no rewards distributed
  5. Any legitimate call to update_rewards at block N now reverts with REWARDS_ALREADY_UPDATED
  6. Block N rewards are permanently lost for all stakers and delegators
  7. Repeat at block N+1, N+2, ...
```

The attacker needs only a funded address and knowledge of any active staker address (e.g., read from `NewStaker` events). No governance access, no leaked keys, no bridge compromise required.
