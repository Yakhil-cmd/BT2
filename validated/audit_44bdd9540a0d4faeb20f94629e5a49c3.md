### Title
Gateway `skip_stateful_validations` Admits Invoke Transactions with Invalid Signatures by Checking Any Mempool Presence Instead of Deploy-Account Presence — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the gateway skips the `__validate__` entry point (account signature verification) for any Invoke transaction with `nonce=1` whenever the sender address has **any** transaction in the mempool, not specifically a `deploy_account` transaction. An attacker who observes a victim's pending `nonce=0` Invoke in the mempool can submit a forged `nonce=1` Invoke with the victim's address and an invalid signature. The gateway admits it without running `__validate__`. When the batcher executes the block, `__validate__` fails, the transaction is reverted, but the pre-validation stage has already incremented the victim's nonce and charged fees from the victim's balance.

---

### Finding Description

`skip_stateful_validations` is designed to support the UX pattern of sending `deploy_account + invoke` simultaneously: because the account does not yet exist on-chain, `__validate__` would fail, so the gateway skips it when a `deploy_account` is pending. The guard condition is:

```
tx is Invoke
AND tx.nonce() == Nonce(Felt::ONE)
AND account_nonce == Nonce(Felt::ZERO)
AND account_tx_in_pool_or_recent_block(sender_address) == true
``` [1](#0-0) 

The fourth condition delegates to `Mempool::account_tx_in_pool_or_recent_block`, which returns `true` if the address has **any** transaction in the pool or a recent block:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [2](#0-1) 

This is too broad. An existing account (contract deployed, nonce=0 on-chain) that has a valid `nonce=0` Invoke pending in the mempool satisfies the condition. The code comment claims "it means that either it has a deploy_account transaction or transactions with future nonces that passed validations," but this is incorrect: a `nonce=0` Invoke is not a `deploy_account`, and it does not imply anything about the validity of a `nonce=1` Invoke.

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` sets `execution_flags.validate = false`:

```rust
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [3](#0-2) 

The blockifier's `StatefulValidator::perform_validations` then returns `Ok(())` immediately for Invoke without calling `validate_tx`:

```rust
ApiTransaction::Invoke(_) => {
    ...
    if !tx.execution_flags.validate {
        return Ok(());
    }
    // `__validate__` call.
    let (_optional_call_info, actual_cost) = self.validate(&tx, tx_context.clone())?;
``` [4](#0-3) 

The forged Invoke is admitted to the mempool with no signature check. When the batcher later executes the block, `perform_pre_validation_stage` runs unconditionally (incrementing the nonce and checking balance), then `__validate__` runs with `validate=true` and fails. The transaction is reverted, but the nonce increment and fee deduction from the victim's account are committed. [5](#0-4) 

---

### Impact Explanation

**High — Mempool/gateway admission accepts an invalid transaction (invalid signature) before sequencing.**

A forged Invoke transaction whose `__validate__` would fail is admitted to the mempool and included in a block as a reverted transaction. The victim's account nonce is incremented and a fee is deducted from the victim's balance for a transaction the victim never signed. This satisfies:

- *High*: "Mempool/gateway/RPC admission accepts invalid transactions … before sequencing."
- *Critical* (economic): "Incorrect fee … balance … with economic impact" — fees are charged from the victim's account without their authorization.

---

### Likelihood Explanation

**Medium.** The mempool is observable. An attacker monitors for any address that has a pending `nonce=0` Invoke (account exists on-chain, nonce=0). The attack window is the time between the victim's `nonce=0` Invoke being submitted and being included in a block. During that window the attacker submits a forged `nonce=1` Invoke. No privileged access is required; the attacker only needs to submit a standard RPC transaction.

---

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a check that is specific to `deploy_account` transactions. The gateway should only skip `__validate__` when the pending transaction for the account is a `deploy_account` (i.e., the account does not yet exist on-chain). One approach:

1. Add a `deploy_account_tx_in_pool(address)` method to the mempool that returns `true` only when a `deploy_account` transaction is queued for that address.
2. Replace the call in `skip_stateful_validations` with this narrower check.

Alternatively, check whether the account contract exists on-chain: if the account exists (class hash is non-zero at the address), `__validate__` must always run regardless of mempool state. [6](#0-5) 

---

### Proof of Concept

**Setup:** Alice has a deployed account at address `X`, on-chain nonce = 0.

1. **Alice** submits a valid Invoke: `sender=X, nonce=0, valid_signature`. It passes stateless and stateful validation (including `__validate__`) and enters the mempool.

2. **Bob (attacker)** submits a forged Invoke: `sender=X, nonce=1, invalid_signature, non-zero resource bounds`.

3. **Stateless validation** (`StatelessTransactionValidator::validate`): passes — signature length is within bounds, resource bounds are non-zero. [7](#0-6) 

4. **Stateful validation** (`extract_state_nonce_and_run_validations`):
   - `get_nonce_from_state(X)` → returns `Nonce(0)`.
   - `validate_nonce`: `nonce=1 ∈ [0, 200]` → passes.
   - `validate_by_mempool`: no duplicate hash → passes.
   - `skip_stateful_validations`: `nonce=1`, `account_nonce=0`, `account_tx_in_pool_or_recent_block(X)=true` (Alice's nonce=0 Invoke is in the pool) → **returns `true`**.
   - `run_validate_entry_point`: `validate=false` → `__validate__` **skipped**.
   - Bob's forged Invoke is **admitted to the mempool**.

5. **Batcher executes the block:**
   - Alice's `nonce=0` Invoke executes successfully; Alice's on-chain nonce becomes 1.
   - Bob's `nonce=1` Invoke: `perform_pre_validation_stage` runs — nonce incremented to 2, fee bounds checked, balance verified. Then `__validate__` runs with `validate=true` and **fails** (invalid signature). Transaction is reverted. **Fee is charged from Alice's account.** [8](#0-7) [5](#0-4)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}

    pub fn perform_pre_validation_stage<S: State + StateReader>(
        &self,
        state: &mut S,
        tx_context: &TransactionContext,
    ) -> TransactionPreValidationResult<()> {
        let tx_info = &tx_context.tx_info;
        Self::handle_nonce(state, tx_info, self.execution_flags.strict_nonce_check)?;

        if self.execution_flags.charge_fee {
            self.check_fee_bounds(tx_context)?;

            verify_can_pay_committed_bounds(state, tx_context).map_err(Box::new)?;
        }

        self.validate_proof_facts(&tx_context.block_context, state)?;

        Ok(())
    }
