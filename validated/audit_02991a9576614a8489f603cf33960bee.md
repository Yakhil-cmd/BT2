### Title
Unbounded Loop Over Pool Member Balance Trace Can Permanently Freeze Delegator's Unclaimed Yield — (File: `src/pool/pool.cairo`)

---

### Summary

The `calculate_rewards_from_trace` internal function in `src/pool/pool.cairo` contains an unbounded `while` loop that iterates over every entry in a delegator's `pool_member_epoch_balance` trace since their last claim. Because there is no cap on how many balance-change entries a delegator can accumulate, a delegator who makes a large number of `add_to_delegation_pool` or `exit_delegation_pool_intent` calls without claiming rewards can grow this trace until the gas cost of `claim_rewards` permanently exceeds Starknet's execution-resource limits, irreversibly freezing their unclaimed yield.

---

### Finding Description

`calculate_rewards_from_trace` (pool.cairo, lines 839–888) accepts a `mut entry_to_claim_from: VecIndex` and iterates forward through `pool_member_epoch_balance` until it reaches the current epoch:

```cairo
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
    if pool_member_checkpoint.epoch() >= until_epoch {
        break;
    }
    let to_sigma = self.find_sigma(pool_member_checkpoint, curr_epoch: until_epoch);
    rewards += compute_rewards_rounded_down(...);
    entry_to_claim_from += 1;
}
``` [1](#0-0) 

The developers themselves flag this as unbounded. Every call to `add_to_delegation_pool` or `exit_delegation_pool_intent` appends a new checkpoint to `pool_member_epoch_balance`: [2](#0-1) 

There is no maximum on how many such entries can accumulate between two consecutive `claim_rewards` calls. The `entry_to_claim_from` cursor is only advanced and persisted on a *successful* claim; if the transaction reverts due to resource exhaustion, the cursor is not updated and the next attempt must re-process the same oversized range. There is no chunked-claim mechanism.

Inside each loop iteration, `find_sigma` is also called, which itself reads from `cumulative_rewards_trace`: [3](#0-2) 

This compounds the per-iteration cost, making the effective complexity super-linear in the number of accumulated trace entries.

---

### Impact Explanation

**High — Permanent freezing of unclaimed yield.**

Once the `pool_member_epoch_balance` trace for a delegator grows beyond the point where a single transaction can iterate it within Starknet's execution-resource budget, every subsequent `claim_rewards` call for that delegator will revert. Because the cursor is only saved on success, there is no way to recover the accumulated rewards; they are permanently inaccessible to the delegator.

---

### Likelihood Explanation

**Low.**

A delegator must accumulate a very large number of balance-change entries without ever claiming. This can happen organically (a long-lived, active delegator who never claims) or deliberately (a delegator who repeatedly calls `add_to_delegation_pool` with minimal amounts across many epochs). Each call costs gas, so deliberate exploitation is expensive for the attacker. However, the developers' own comment acknowledges the loop is unbounded, confirming the structural risk exists.

---

### Recommendation

1. **Enforce a maximum number of unclaimed balance-change entries.** Reject `add_to_delegation_pool` / `exit_delegation_pool_intent` if the delegator's trace length minus `entry_to_claim_from` exceeds a safe constant (e.g., 100 entries).
2. **Implement partial/chunked reward claiming.** Allow `claim_rewards` to accept an optional `max_entries` parameter so a delegator can drain the trace over multiple transactions.
3. **Require periodic claiming.** Revert balance-change operations if the delegator has more than N unclaimed trace entries, forcing them to claim before accumulating further.

---

### Proof of Concept

1. Delegator calls `add_to_delegation_pool` (or alternates `add_to_delegation_pool` / `exit_delegation_pool_intent`) a large number of times across successive epochs, never calling `claim_rewards`.
2. Each call appends a new `PoolMemberCheckpoint` to `pool_member_epoch_balance`.
3. After accumulating enough entries, the delegator calls `claim_rewards`.
4. `calculate_rewards_from_trace` iterates over all accumulated entries; each iteration also invokes `find_sigma` against `cumulative_rewards_trace`.
5. The transaction exhausts Starknet's execution-resource limit and reverts.
6. Because `entry_to_claim_from` is only persisted on success, every future `claim_rewards` attempt faces the same oversized loop and also reverts.
7. The delegator's unclaimed yield is permanently frozen with no recovery path. [4](#0-3)
