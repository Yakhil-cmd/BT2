### Title
Cross-COA Signature Replay via Missing `evmAddress` Binding in `validateCOAOwnershipProof` - (File: `fvm/evm/stdlib/contract.cdc`)

---

### Summary

`EVM.validateCOAOwnershipProof` does not enforce that `signedData` encodes the target `evmAddress`. Because the signature is verified only over the caller-supplied `signedData` bytes — with no protocol-level binding to the specific COA being authenticated — a valid signature produced for one COA can be replayed against a different COA owned by the same Cadence account. This is a direct analog of the reported cross-organization signature replay: in both cases, the signed digest omits the target contract/address context, allowing the same signature to satisfy ownership proofs for multiple distinct contexts.

---

### Finding Description

`EVM.validateCOAOwnershipProof` in `fvm/evm/stdlib/contract.cdc` performs two independent checks:

1. It verifies that the provided `signatures` are valid over `signedData` using keys from the Cadence account at `address`.
2. It checks that the COA resource at the provided `path` has an EVM address matching `evmAddress`.

The code itself acknowledges the gap:

> *"Note: this function does not enforce that `signedData` includes `evmAddress`. In principle, a signature produced for one purpose could be replayed here against a different COA owned by the same Cadence account."* [1](#0-0) 

The signature verification step at line 1082–1086 verifies only over `signedData` with no COA-address binding: [2](#0-1) 

Because `signedData` is fully attacker-controlled (passed in by the caller), and the `path` pointing to the COA resource is also attacker-controlled, an attacker can construct a proof that passes both checks for a different COA than the one the signature was originally intended for.

The EVM-side entry point is the Cadence Arch precompile `verifyCOAOwnershipProof(address,bytes32,bytes)`, which decodes the caller-supplied `address` (COA EVM address), `bytes32` (signed data), and `bytes` (encoded proof) and passes them directly to `validateCOAOwnershipProof`: [3](#0-2) [4](#0-3) 

The COA contract's ERC-1271 implementation calls this precompile with `address(this)` as the COA address: [5](#0-4) 

---

### Impact Explanation

If a Cadence account owns two COAs (COA_A at `/public/coaA`, COA_B at `/public/coaB`), and a signature is produced over `signedData` (e.g., a challenge nonce) for COA_A without binding the COA address into the signed bytes, an attacker can replay that exact signature to pass `validateCOAOwnershipProof` for COA_B by:

- Supplying the same `signedData` and `signatures`
- Setting `path = /public/coaB` and `evmAddress = COA_B_address`

Both checks pass: the signature is valid over `signedData` (unchanged), and the COA at `coaB` does have address `COA_B_address`. The function returns `isValid: true` for COA_B.

Through the ERC-1271 path, this means `COA_B.isValidSignature(_hash, proof_for_COA_B)` returns `ValidERC1271Signature` (0x1626ba7e) even though the account owner only signed `_hash` in the context of COA_A. Any EVM protocol relying on ERC-1271 for authentication (NFT marketplaces, DeFi, governance) would accept COA_B as having authorized the action. [6](#0-5) 

---

### Likelihood Explanation

The precondition is a Cadence account with two or more COAs stored at different capability paths. While Flow wallets have historically created at most one COA per account, the protocol imposes no such constraint. Any Cadence transaction can create and store multiple COAs. The `validateCOAOwnershipProof` function is `access(all)` and callable from any Cadence script or transaction. The EVM precompile is callable from any EVM contract or transaction. No privileged access is required. An attacker who obtains a valid `(signedData, signatures)` pair for COA_A — e.g., by observing an on-chain authentication flow — can immediately replay it against COA_B with no additional capability. [7](#0-6) 

---

### Recommendation

Enforce at the protocol level that `signedData` encodes `evmAddress`. Inside `validateCOAOwnershipProof`, before calling `keyList.verify`, prepend or hash the `evmAddress` bytes into the data that is actually verified:

```cadence
// Bind the evmAddress into the signed payload before verification
let boundData = signedData.concat(evmAddress.toConstantSized<[UInt8; 20]>()!)
let isValid = keyList.verify(
    signatureSet: signatureSet,
    signedData: boundData,
    domainSeparationTag: "FLOW-V0.0-user"
)
```

This mirrors the fix applied in the referenced Story Protocol remediation (commit `ca26a84`), which included the contract address in the signature digest to prevent cross-context replay. [2](#0-1) 

---

### Proof of Concept

**Setup**: Cadence account `Alice` owns two COAs:
- COA_A stored at `/storage/coaA`, published at `/public/coaA`
- COA_B stored at `/storage/coaB`, published at `/public/coaB`

**Step 1**: An EVM protocol asks Alice to prove ownership of COA_A by signing a challenge `_hash`. Alice signs `_hash` with her Cadence key and produces `proof_A = {Address: Alice, CapabilityPath: "coaA", KeyIndices: [0], Signatures: [sig]}`.

**Step 2**: The attacker constructs `proof_B = {Address: Alice, CapabilityPath: "coaB", KeyIndices: [0], Signatures: [sig]}` — identical except for the capability path.

**Step 3**: The attacker calls `COA_B.isValidSignature(_hash, abi.encode(proof_B))` from an EVM transaction.

**Step 4**: Inside `isValidSignature`, `verifyCOAOwnershipProof(COA_B_address, _hash, proof_B)` is called on the Cadence Arch precompile.

**Step 5**: `validateCOAOwnershipProof` verifies `sig` over `_hash` using Alice's key (valid — same data, same key). Then it borrows the COA at `/public/coaB` and confirms its address equals `COA_B_address` (valid — that is COA_B's actual address).

**Step 6**: Returns `ValidERC1271Signature` (0x1626ba7e). COA_B is falsely authenticated as having signed `_hash`, enabling the attacker to impersonate COA_B in any EVM protocol using ERC-1271. [8](#0-7) [4](#0-3) [9](#0-8)
