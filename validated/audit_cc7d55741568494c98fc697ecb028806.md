### Title
`protocolUnpausePool` Always Transitions to Admin-Paused State (Level 1) Instead of Restoring Pre-Pause State, Leaving Swaps Permanently Blocked After Protocol Unpause â€” (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`MetricOmmPoolFactory.protocolUnpausePool` hardcodes the target pause level to `1` (admin-paused) regardless of the pool's state before the protocol pause. When a pool was at level `0` (active) before the protocol paused it, after protocol unpause the pool is at level `1` (admin-paused) instead of level `0` (active). Swaps remain blocked until the pool admin separately calls `unpausePool`. If the admin is unresponsive or the admin key is lost, the pool is permanently stuck in admin-paused state with swaps disabled.

---

### Finding Description

The pool pause state machine has three levels:

| Level | Meaning |
|---|---|
| `0` | Active (no pause) |
| `1` | Paused by admin |
| `2` | Paused by protocol |

`protocolPausePool` accepts pools at level `0` **or** `1` and transitions them to level `2`: [1](#0-0) 

`protocolUnpausePool` always hardcodes the target as `1`, regardless of whether the pool was at `0` or `1` before the protocol pause: [2](#0-1) 

The pool's `_checkNotPaused()` guard blocks swaps for **any** non-zero pause level: [3](#0-2) 

The `swap` function is the only function gated by `whenNotPaused`: [4](#0-3) 

After `protocolUnpausePool` runs, the pool is at level `1`. Swaps still revert with `PoolPaused`. The only path from `1 â†’ 0` is through the pool admin's `unpausePool`: [5](#0-4) 

The contract stores no record of the pre-protocol-pause state, so it cannot restore it. The protocol and the pool admin are separate roles â€” the protocol owner cannot directly call `unpausePool`.

---

### Impact Explanation

After `protocolUnpausePool` is called on a pool that was previously active (level `0`), swaps remain blocked. This is broken core pool functionality: the AMM's primary operation (swapping) is unusable. LPs cannot trade, arbitrageurs cannot rebalance, and users cannot exit positions via swap. The pool is stuck until the pool admin separately calls `unpausePool`. If the admin is unresponsive, the pool is permanently swap-disabled despite the protocol having "unpaused" it.

---

### Likelihood Explanation

The most common protocol-pause scenario is pausing an active pool (level `0`). The `protocolPausePool` function explicitly allows pausing from level `0` or `1`, meaning the protocol routinely pauses active pools. Every such pause followed by a protocol unpause triggers this bug. No special configuration or attacker action is required â€” it is triggered by normal protocol operations.

---

### Recommendation

Store the pre-protocol-pause level before transitioning to `2`, and restore it on `protocolUnpausePool`. Alternatively, add a direct `2 â†’ 0` transition path for the protocol owner:

```solidity
function protocolUnpausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 2) revert InvalidPauseTransition(cur, 0);
    // Restore to fully active, not admin-paused
    IMetricOmmPoolFactoryActions(pool).setPause(0);
}
```

Or store the previous level in factory storage before the protocol pause and restore it on unpause.

---

### Proof of Concept

1. Pool is at level `0` (active). Swaps work normally.
2. Protocol owner calls `protocolPausePool(pool)`. Pool transitions `0 â†’ 2`. Swaps revert.
3. Emergency resolved. Protocol owner calls `protocolUnpausePool(pool)`. Pool transitions `2 â†’ 1`.
4. Pool is now at level `1` (admin-paused). `_checkNotPaused()` still reverts with `PoolPaused`.
5. Swaps remain blocked. The pool admin must separately call `unpausePool(pool)` (`1 â†’ 0`) to restore swap functionality.
6. If the pool admin is a multisig with lost signers, or an unresponsive third party, the pool is permanently swap-disabled despite the protocol having "unpaused" it.
