### Title
Missing Minimum Threshold Lower-Bound Check in FROST Sign and Presign Validation — (`File: src/frost/mod.rs`)

### Summary
`assert_sign_inputs` and `presign` in `src/frost/mod.rs` validate that the caller-supplied `threshold` does not exceed the participant count (upper bound), but never enforce the minimum lower bound (`threshold >= 2`). A malicious coordinator or library caller can pass `threshold = 1` (or `0`) to `sign_v1`, `sign_v2`, or `presign`, bypassing the multi-party security requirement and allowing a single participant to produce a valid FROST threshold signature.

### Finding Description

`assert_sign_inputs` (called by both `sign_v1` and `sign_v2` in `src/frost/eddsa/sign.rs`) performs the following threshold validation:

```rust
// src/frost/mod.rs  lines 144-150
if threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge {
        threshold: threshold.value(),
        max: participants.len(),
    });
}
```

There is **no corresponding lower-bound check** (`threshold.value() < 2`). The same omission exists in `presign` (lines 71–77).

Compare this to the analogous validation in `validate_triple_inputs` (OT-based ECDSA triple generation) and `assert_key_invariants` (DKG), which both enforce the minimum:

```rust
// src/ecdsa/ot_based_ecdsa/triples/generation.rs  lines 699-703
if threshold_value < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold_value,
        min: 2,
    });
}
```

```rust
// src/dkg.rs  lines 580-581
if threshold < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
}
```

The FROST signing path (`sign_v1` → `assert_sign_inputs` → `fut_wrapper_v1`) accepts the caller-supplied threshold and uses it to construct a `KeyPackage` with that threshold value, which FROST then uses to determine how many shares are required for Lagrange interpolation and aggregation. With `threshold = 1`, only one participant's share is needed to produce a valid aggregate signature.

### Impact Explanation

A caller passing `threshold = 1` to `sign_v1` or `sign_v2` causes FROST to require only a single signing share for aggregation. Because each participant's share is a valid share of the secret key, a single participant (e.g., the coordinator itself) can unilaterally produce a cryptographically valid signature over an attacker-chosen message, without the participation of any other honest party. This directly violates the threshold security guarantee.

**Impact: Critical — Unauthorized creation of a valid threshold signature for attacker-chosen inputs.**

### Likelihood Explanation

The `threshold` parameter is fully caller-controlled at the public API boundary. No privileged access is required. Any integrator or malicious coordinator invoking `sign_v1` or `sign_v2` with `threshold = 1` triggers the issue. The DKG and triple-generation paths already enforce the minimum, making this an inconsistency that is easy to miss and easy to exploit.

### Recommendation

Add the same lower-bound guard present in `validate_triple_inputs` and `assert_key_invariants` to `assert_sign_inputs` and `presign` in `src/frost/mod.rs`:

```rust
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

This check should be inserted before (or alongside) the existing upper-bound check in both functions.

### Proof of Concept

1. Run a standard FROST key generation with `threshold = 3`, `participants = [P1, P2, P3]`.
2. Call `sign_v1(&participants, /*threshold=*/1usize, me, coordinator, keygen_output, message, rng)`.
3. `assert_sign_inputs` passes: `1 <= 3` (upper bound satisfied), no lower-bound check fires.
4. The coordinator constructs a `KeyPackage` with `threshold = 1`.
5. The coordinator alone (without waiting for P2 or P3) aggregates a single signing share and produces a valid Ed25519 signature.
6. The signature verifies successfully against the group public key, despite only one of three required parties participating.

pub fn sign_v1(
    participants: &[Participant],
    threshold: impl Into<ReconstructionLowerBound>,
    me: Participant,
    coordinator: Participant,
    keygen_output: KeygenOutput,
    message: Vec<u8>,
    rng: impl CryptoRngCore + Send + 'static,
) -> Result<impl Protocol<Output = SignatureOption>, InitializationError> {
    let threshold = threshold.into();
    let participants = assert_sign_inputs(participants, threshold, me, coordinator)?;

    let comms = Comms::new();
    let chan = comms.shared_channel();
    let fut = fut_wrapper_v1(
        chan,
        participants,
        threshold,
        me,
        coordinator,
        keygen_output,
        message,
        rng,
    );
    Ok(make_protocol(comms, fut))
}
