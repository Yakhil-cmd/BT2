### Title
Silent CRO Bridge Accounting Failure for Sub-Minimum Amounts Strands User Funds in Cosmos Bank Layer - (File: x/cronos/keeper/evmhandlers/send_cro_to_ibc.go)

### Summary
`SendCroToIbcHandler.Handle()` unconditionally moves CRO from the bridge contract's Cosmos address to the sender's Cosmos address **before** calling `IbcTransferCoins`. When the amount is below `10^10` wei (the minimum representable in 8-decimal IBC format), `IbcTransferCoins` silently skips the IBC transfer and returns `nil`. The EVM transaction succeeds with `status=1`, but the bridge never completes. The user's CRO is permanently moved out of the EVM layer into the Cosmos bank layer without being bridged to Crypto.org Chain.

### Finding Description

The `SendCroToIbcHandler` in `x/cronos/keeper/evmhandlers/send_cro_to_ibc.go` handles the `__CronosSendCroToIbc` event emitted by authorized bridge contracts (e.g., `CroBridge`). The handler unconditionally executes two steps:

**Step 1** — fund transfer from contract to sender (always executes): [1](#0-0) 

**Step 2** — IBC transfer initiation (silently skips for small amounts): [2](#0-1) 

Inside `IbcTransferCoins`, when the denom is the EVM denom (CRO), the function computes `amountToBurn = amount - (amount % 10^10)`. If `amountToBurn == 0` (i.e., `amount < 10^10 wei = 10 gwei`), the function silently skips the IBC transfer with `continue` and returns `nil`: [3](#0-2) 

The `TenPowTen` constant is `10^10`: [4](#0-3) 

The result is a state inconsistency:
- Step 1 always executes: CRO moves from the contract's Cosmos address → sender's Cosmos bank address
- Step 2 silently no-ops: no IBC packet is ever sent
- `PostTxProcessing` returns `nil` (no error), so the EVM transaction commits with `status=1` [5](#0-4) 

The `CroBridge` contract emits the event with `msg.value` directly, so any user sending less than 10 gwei triggers this path: [6](#0-5) 

### Impact Explanation

Any unprivileged user calling an authorized `CroBridge` contract with `value < 10^10 wei` will:
1. Have their EVM balance debited (CRO paid to the contract)
2. Receive a successful EVM receipt (`status=1`), falsely indicating the bridge succeeded
3. Find their CRO silently moved to their Cosmos bank address — not bridged, not in EVM
4. Have no on-chain indication that the bridge failed

This is a **bridge accounting failure**: CRO is irreversibly moved from the EVM layer to the Cosmos bank layer without completing the bridge. The user's funds are stranded in the Cosmos bank layer, inaccessible from their EVM wallet and not delivered to the destination chain. Recovery requires Cosmos-layer tooling that most EVM users do not have.

This matches the allowed High impact: *"Permanent or long-lived inability for honest users to process valid bridge/conversion flows under normal network assumptions."*

### Likelihood Explanation

The entry path is fully unprivileged. Any user can call an authorized `CroBridge` contract (authorization is a governance parameter, not per-user). The `CroBridge` contract passes `msg.value` directly as the amount with no minimum check. Sending 1 wei is a valid EVM transaction. No special access, leaked keys, or admin compromise is required.

### Recommendation

In `SendCroToIbcHandler.Handle()`, validate that the amount meets the minimum IBC-transferable threshold (`>= 10^10 wei`) **before** executing the fund transfer from the contract to the sender. If the amount is below the minimum, either:
- Return an error (causing the EVM transaction to revert, preserving the user's EVM balance), or
- Skip the entire handler (both the fund transfer and the IBC call) so no state change occurs.

The fund transfer at line 96 must never execute unless the IBC transfer at line 100 will also execute. [7](#0-6) 

### Proof of Concept

1. Deploy a `CroBridge` contract and authorize it via governance (`CroBridgeContractAddresses` param).
2. Call `CroBridge.send_cro_to_crypto_org(recipient)` with `value = 1 wei` (any amount `< 10^10 wei`).
3. Observe: EVM transaction receipt `status = 1` (success).
4. Check sender's Cosmos bank balance: increased by 1 `aphoton` (CRO moved from EVM to Cosmos bank).
5. Check IBC packet queue: no packet was sent.
6. Check sender's EVM balance: decreased by 1 wei + gas.
7. The 1 wei of CRO is now stranded in the sender's Cosmos bank address, not bridged to Crypto.org Chain, and not accessible from the EVM wallet.

func (h LogProcessEvmHook) PostTxProcessing(ctx sdk.Context, _ *core.Message, receipt *ethtypes.Receipt) error {
	addLogToReceiptFunc := newFuncAddLogToReceipt(receipt)
	for _, log := range receipt.Logs {
		if len(log.Topics) == 0 {
			continue
		}
		handler, ok := h.handlers[log.Topics[0]]
		if !ok {
			continue
		}
		err := handler.Handle(ctx, log.Address, log.Topics, log.Data, addLogToReceiptFunc)
		if err != nil {
			return err
		}
	}
	return nil
}
