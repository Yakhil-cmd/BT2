### Title
Single Malicious Participant Permanently Blocks DKG via Echo Broadcast Early-Abort - (File: `src/protocol/echo_broadcast.rs`)

### Summary
The `reliable_broadcast_receive_all` function contains an early-abort optimization that terminates the entire broadcast — and therefore the entire DKG — when it determines that no data value can accumulate enough echo votes to cross the threshold. A single malicious participant can engineer exactly this split-vote condition by equivocating as sender, permanently denying DKG, reshare, and refresh to all honest parties, even when the participant count satisfies the documented BFT tolerance `MaxFaulty = (N-1)/3`.

### Finding Description

`reliable_broadcast_receive_all` in `src/protocol/echo_broadcast.rs` runs `n` concurrent broadcast sessions (one per participant). Inside the echo-handling branch it performs an early-abort check: [1](#0-0) 

```rust
else if !state_sid.finish_amplification {
    let received_echo_cnt = state_sid.data_echo.get_sum_counters();
    let non_received_echo_cnt = n - received_echo_cnt;
    let mut is_enough = false;
    for (_, cnt) in state_sid.data_echo.iter() {
        if cnt + non_received_echo_cnt > echo_t {
            is_enough = true;
            break;
        }
    }
    if !is_enough {
        return Err(ProtocolError::AssertionFailed(format!(
            "The original sender in session {sid:?} is malicious! ..."
        )));
    }
}
```

The echo threshold is computed as: [2](#0-1) 

For `n = 4`: `broadcast_threshold = (4-1)/3 = 1`, `echo_t = midpoint(4,1) = 2`. To proceed to the ready phase a party needs strictly more than 2 echo votes for the same value, i.e. ≥ 3.

**Attack with n = 4, 1 malicious party (within documented MaxFaulty = 1):**

1. Malicious party `P_m` acts as sender in its own broadcast session (Round 1 session-ID broadcast or Round 3 commitment broadcast in DKG).
2. `P_m` sends `Send(data_A)` to one honest party and `Send(data_B)` to the other two honest parties.
3. Honest parties echo what they received: 1 × `Echo(data_A)` + 2 × `Echo(data_B)`.
4. `P_m` sends `Echo(data_A)` to all honest parties.
5. Every honest party now has: `cnt(data_A) = 2`, `cnt(data_B) = 2`, `non_received = 0`.
6. Neither `2 + 0 > 2` nor `2 + 0 > 2` → `is_enough = false` → **all honest parties abort**.

Because `reliable_broadcast_receive_all` returns an `Err` immediately, the error propagates through `do_broadcast` → `do_keyshare` → `do_keygen`/`do_reshare`, permanently failing DKG for every honest participant. [3](#0-2) [4](#0-3) [5](#0-4) 

The documentation explicitly states the BFT tolerance: [6](#0-5) 

> "we set MaxFaulty = (N-1)/3 as an invariable parameter"

For `n = 4`, `MaxFaulty = 1`, so the protocol is documented to tolerate exactly this scenario. The implementation violates its own stated guarantee.

The existing test confirms the abort is reachable and is treated as expected behavior, but it contradicts the documented BFT bound: [7](#0-6) 

### Impact Explanation
A single malicious participant — one that is legitimately part of the participant set — can permanently abort DKG, reshare, and refresh for all honest parties by equivocating in its own broadcast session. No honest party can complete key generation or key resharing. This maps directly to the allowed High impact: **Permanent denial of key generation, reshare, or refresh for honest parties under valid protocol inputs and documented trust assumptions.**

### Likelihood Explanation
Any participant in the DKG who wishes to prevent key generation can execute this attack with no special privileges. The attacker only needs to send two different `Send` messages to different subsets of honest parties (a standard equivocation attack), then send a single `Echo` message. The attack requires no cryptographic breaks, no leaked keys, and no external dependencies. It is executable on the first round of DKG.

### Recommendation
Remove or redesign the early-abort optimization. The standard Bracha reliable broadcast protocol with `n ≥ 3f+1` guarantees liveness without an early-abort shortcut. If early detection of malicious senders is desired, the abort should be scoped to the individual session of the detected malicious sender (skipping/ignoring that session's output) rather than terminating the entire `reliable_broadcast_receive_all` call and all concurrent sessions with it.

### Proof of Concept

**Setup:** `n = 4` participants `[P0, P1, P2, P_m]` where `P_m` is malicious. `MaxFaulty = (4-1)/3 = 1` (within documented tolerance). `echo_t = midpoint(4,1) = 2`.

**Round 1 of DKG — session-ID broadcast, `P_m`'s session (`sid = 3`):**

| Step | Action |
|------|--------|
| 1 | `P_m` sends `Send(sid_A)` to `P0`; sends `Send(sid_B)` to `P1`, `P2` |
| 2 | `P0` echoes `Echo(sid_A)` to all; `P1`, `P2` echo `Echo(sid_B)` to all |
| 3 | `P_m` sends `Echo(sid_A)` to `P0`, `P1`, `P2` |
| 4 | Each honest party accumulates: `cnt(sid_A)=2`, `cnt(sid_B)=2`, `non_received=0` |
| 5 | Check: `2+0=2`, not `> echo_t=2` for either value → `is_enough=false` |
| 6 | All honest parties return `Err(AssertionFailed("...malicious..."))` |
| 7 | `do_broadcast` → `do_keyshare` → `keygen` all propagate the error |
| 8 | DKG permanently fails for `P0`, `P1`, `P2` | [8](#0-7) [2](#0-1)

```rust
fn echo_ready_thresholds(n: usize) -> (usize, usize) {
    // case where no malicious parties are assumed: when n <= 3/
    // In this case the echo and ready thresholds are both 0
    // later we compare if we have collected more votes than these thresholds
    if n <= 3 {
        return (0, 0);
    }
    // we should always have n >= 3*threshold + 1
    let broadcast_threshold = (n - 1) / 3;
    let echo_threshold = usize::midpoint(n, broadcast_threshold);
    (echo_threshold, broadcast_threshold)
}
```
