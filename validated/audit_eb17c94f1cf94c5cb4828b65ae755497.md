### Title
Silent Error Discard in `convertAddress` Returns Zero EVM Address for Malformed Bech32 Receiver ÔÇö (File: x/cronos/events/decoders.go)

### Summary
`convertAddress` in `x/cronos/events/decoders.go` silently discards the error returns from `sdk.ValAddressFromBech32` and `sdk.AccAddressFromBech32`. When either call fails, `addr` stays `nil`, and `common.BytesToAddress(nil)` silently produces the all-zero EVM address `0x0000ÔÇª0000`. The function returns this zero address with a `nil` error, so callers believe the conversion succeeded. This corrupted address is then embedded in the EVM log data emitted by the relayer precompile for IBC packet events.

### Finding Description
In `convertAddress` the two bech32 branches both blank-discard the error:

```go
case strings.HasPrefix(addrString, cfg.GetBech32ValidatorAddrPrefix()):
    addr, _ = sdk.ValAddressFromBech32(addrString)   // error silently dropped
case strings.HasPrefix(addrString, cfg.GetBech32AccountAddrPrefix()):
    addr, _ = sdk.AccAddressFromBech32(addrString)   // error silently dropped
``` [1](#0-0) 

When either SDK call fails (invalid checksum, truncated payload, wrong sub-prefix), `addr` is `nil`. The next line:

```go
to := common.BytesToAddress(addr)
return &to, nil
``` [2](#0-1) 

returns the zero address with no error. The caller `ConvertPacketData` checks `err` and proceeds normally:

```go
receiver, err := convertAddress(tokenPacketData.Receiver)
if err != nil {
    return nil, err
}
``` [3](#0-2) 

The zero address is then placed into the `IRelayerModulePacketData.Receiver` field that is ABI-encoded and emitted as an EVM log by the relayer precompile. [4](#0-3) 

### Impact Explanation
The relayer precompile emits EVM logs containing the decoded receiver address. Any smart contract that subscribes to these logs and uses the `Receiver` field to credit tokens, gate withdrawals, or make authorization decisions will receive `address(0)` instead of the real recipient. This constitutes **corruption of EVM receipt/log processing with direct security impact**: tokens credited by such a contract go to the zero address (permanent loss), and any zero-address guard logic is bypassed.

### Likelihood Explanation
An unprivileged attacker on any IBC-connected chain can craft a `MsgTransfer` whose `receiver` field starts with the Cronos account prefix (`crc`) or validator prefix (`crcvaloper`) but carries an invalid bech32 checksum. The IBC core module performs only a non-empty-string check on the receiver field before relaying; bech32 validity is left to the application. The packet therefore reaches Cronos, the relayer precompile fires `ConvertPacketData`, and the bug is triggered deterministically.

### Recommendation
Propagate the errors instead of discarding them:

```go
case strings.HasPrefix(addrString, cfg.GetBech32ValidatorAddrPrefix()):
    addr, err = sdk.ValAddressFromBech32(addrString)
    if err != nil {
        return nil, fmt.Errorf("invalid validator bech32 address %q: %w", addrString, err)
    }
case strings.HasPrefix(addrString, cfg.GetBech32AccountAddrPrefix()):
    addr, err = sdk.AccAddressFromBech32(addrString)
    if err != nil {
        return nil, fmt.Errorf("invalid account bech32 address %q: %w", addrString, err)
    }
``` [5](#0-4)

func convertAddress(addrString string) (*common.Address, error) {
	cfg := sdk.GetConfig()
	var addr []byte
	// try hex, then bech32
	switch {
	case common.IsHexAddress(addrString):
		addr = common.HexToAddress(addrString).Bytes()
	case strings.HasPrefix(addrString, cfg.GetBech32ValidatorAddrPrefix()):
		addr, _ = sdk.ValAddressFromBech32(addrString)
	case strings.HasPrefix(addrString, cfg.GetBech32AccountAddrPrefix()):
		addr, _ = sdk.AccAddressFromBech32(addrString)
	default:
		return nil, fmt.Errorf("expected a valid hex or bech32 address (acc prefix %s), got '%s'", cfg.GetBech32AccountAddrPrefix(), addrString)
	}
	to := common.BytesToAddress(addr)
	return &to, nil
