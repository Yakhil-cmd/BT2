### Title
Single Malicious Participant Can Permanently Abort DKG/Reshare/Refresh for All Honest Parties via False Success Broadcast - (File: src/dkg.rs)

---

### Summary

The `broadcast_success` function in `src/dkg.rs` unconditionally aborts the DKG (and by extension reshare and refresh) for every honest participant if any single participant broadcasts `false` in the final success round. Because the function hardcodes `true` as its own vote but then requires **all** received votes to also be `true`, a single malicious participant can deviate by broadcasting `false`, causing every honest party to return a `ProtocolError` and discard their freshly computed key material. This violates the documented n/3 BFT tolerance and constitutes a permanent denial of key generation, resharing, and refresh.

---

### Finding Description

`broadcast_success` is the final step of `do_keyshare`, which is the shared core of `keygen`, `reshare`, and `refresh`:

```
// src/dkg.rs line 531
broadcast_success(&mut chan, &participants, me, session_id).await?;
```

The function always broadcasts `(true, session_id)` via the reliable echo broadcast, then collects the full vote map and enforces two hard requirements:

```rust
// src/dkg.rs lines 321-335
if !vote_list.iter().all(|(_, ref sid)| sid == &session_id) {
    return Err(ProtocolError::AssertionFailed(
        "A participant broadcast the wrong session id. Aborting Protocol!"
    ));
}
if !vote_list.iter().all(|&(boolean, _)| boolean) {
    return Err(ProtocolError::AssertionFailed(
        "A participant seems to have failed its checks. Aborting Protocol!"
    ));
}
```

Because the echo broadcast guarantees **agreement** — all honest parties deliver the same message from each sender — a malicious participant who broadcasts `(false, session_id)` causes every honest party to observe `false` for that participant's slot. The `all(|&(boolean, _)| boolean)` predicate then fails for every honest party simultaneously, and they all propagate the error up through `do_keyshare`, discarding their computed `KeygenOutput`.

The malicious participant needs only to deviate in this single final round; all prior cryptographic checks (proof of knowledge, commitment hash, share validation) have already passed for the honest parties, meaning the key material was valid and complete before the abort.

---

### Impact Explanation

This maps directly to the allowed impact: **High — Permanent denial of DKG, reshare, and refresh for honest parties under valid protocol inputs and documented trust assumptions.**

The README explicitly states the protocol tolerates up to n/3 malicious parties:

> "our DKG/Resharing protocol can only tolerate n/3 malicious parties where n is the total number of parties"

However, `broadcast_success` requires **unanimous** success votes. A single malicious participant — even one that is within the n/3 tolerance — can abort the entire DKG for all honest parties. The abort is permanent in the sense that the honest parties must restart the entire protocol from scratch, and the malicious participant can repeat the attack on every attempt indefinitely.

All three protocol entry points are affected: `keygen`, `reshare`, and `refresh`, since all three call `do_keyshare`, which calls `broadcast_success` as its final step.

---

### Likelihood Explanation

**High.** Any participant in the DKG session is a potential attacker. The attack requires only that the malicious participant send `(false, session_id)` instead of `(true, session_id)` in the final broadcast round. No cryptographic capability, no leaked key material, and no privileged access is required. The attacker participates honestly through all prior rounds (to avoid being detected early) and deviates only in the last round, after all honest parties have already computed their key shares. The attack is trivially repeatable on every retry.

---

### Recommendation

Remove the unanimity requirement from `broadcast_success`. The honest parties have already independently verified all cryptographic material (proof of knowledge, commitment hash, secret share) before reaching this round. A participant broadcasting `false` does not invalidate the honest party's own verified output. The final broadcast round should either:

1. **Be removed entirely** — honest parties output their key material once all cryptographic checks in rounds 1–5 pass, without waiting for a final success vote from others.
2. **Tolerate dissenting votes** — log or identify the participant that broadcast `false` (as a `MaliciousParticipant` error) but do not abort the honest party's own output if its local checks all passed.

---

### Proof of Concept

A malicious participant `P_m` joins a DKG session with `n >= 4` honest participants. `P_m` participates honestly through all rounds (broadcasting valid session IDs, commitments, proofs of knowledge, and secret shares). In the final round (Step 5.4), instead of broadcasting `(true, session_id)`, `P_m` broadcasts `(false, session_id)`.

The echo broadcast delivers `(false, session_id)` to all honest parties. Each honest party reaches:

```rust

if !vote_list.iter().all(|&(boolean, _)| boolean) {
    return Err(ProtocolError::AssertionFailed(
        "A participant seems to have failed its checks. Aborting Protocol!"
    ));
}
```

The predicate fails because `P_m`'s slot contains `false`. Every honest party returns `Err(ProtocolError::AssertionFailed(...))` and discards its `KeygenOutput`.
