### Title
Asymmetric Whitelist Checks Allow ERC-20 Tokens to Be Minted to an Address Whose NEAR Account Cannot Submit Transactions, Permanently Freezing Funds — (`engine/src/contract_methods/silo/mod.rs`)

---

### Summary

`is_allow_receive_erc20_tokens` only checks the **Address whitelist**, while `is_allow_submit` checks **both** the Address whitelist and the Account whitelist. When these two whitelists are out of sync — a victim EVM address is present in the Address whitelist but its controlling NEAR account is absent from the Account whitelist — bridged ERC-20 tokens are minted to that EVM address with no possible exit path.

---

### Finding Description

The two gating functions diverge in what they check:

**`is_allow_receive_erc20_tokens`** — only checks the Address whitelist:

```rust
// engine/src/contract_methods/silo/mod.rs, lines 141-143
pub fn is_allow_receive_erc20_tokens<I: IO + Copy>(io: &I, address: &Address) -> bool {
    is_address_allowed(io, address)
}
``` [1](#0-0) 

**`is_allow_submit`** — checks **both** the Address whitelist and the Account whitelist:

```rust
// engine/src/contract_methods/silo/mod.rs, lines 136-138
pub fn is_allow_submit<I: IO + Copy>(io: &I, account: &AccountId, address: &Address) -> bool {
    is_address_allowed(io, address) && is_account_allowed(io, account)
}
``` [2](#0-1) 

The two underlying helpers are independent:

```rust
fn is_address_allowed<I: IO + Copy>(io: &I, address: &Address) -> bool {
    let list = Whitelist::init(io, WhitelistKind::Address);
    !list.is_enabled() || list.is_exist(address)
}

fn is_account_allowed<I: IO + Copy>(io: &I, account: &AccountId) -> bool {
    let list = Whitelist::init(io, WhitelistKind::Account);
    !list.is_enabled() || list.is_exist(account)
}
``` [3](#0-2) 

Because the two whitelists are managed independently (separate `add_entry_to_whitelist` calls, separate enable/disable calls), the system provides no enforcement that every EVM address in the Address whitelist has a corresponding NEAR account in the Account whitelist. [4](#0-3) 

**Concrete reachable state:**

| Whitelist | State |
|---|---|
| Address whitelist | Enabled; victim EVM address present |
| Account whitelist | Enabled; victim NEAR account **absent** |

Under this state:
- `is_allow_receive_erc20_tokens(victim_address)` → `true` (Address whitelist passes)
- `is_allow_submit(victim_near_account, victim_address)` → `false` (Account whitelist fails)

The `ft_on_transfer` / `receive_erc20_tokens` path mints ERC-20 tokens to the victim EVM address because `is_allow_receive_erc20_tokens` returns `true` and no fallback redirect occurs. The fallback redirect only fires when `is_allow_receive_erc20_tokens` returns `false`. [1](#0-0) 

After minting, every subsequent EVM transaction from the victim's NEAR account is blocked by `is_allow_submit` returning `false` because `is_account_allowed` fails. There is no alternative exit path: the tokens are permanently locked in the EVM address. [2](#0-1) 

---

### Impact Explanation

Bridged ERC-20 tokens are permanently frozen inside the silo EVM with no exit path. The victim cannot transfer, approve, or otherwise move the tokens because every EVM transaction requires `is_allow_submit` to pass, which requires the Account whitelist to include the caller's NEAR account. This satisfies **Critical — Permanent freezing of funds**.

---

### Likelihood Explanation

The two whitelists are independently managed with no code-level consistency enforcement. An operator who adds an EVM address to the Address whitelist (to allow token receipt) without adding the corresponding NEAR account to the Account whitelist — a natural operational mistake given the separate management APIs — creates this state. Additionally, a third party can bridge tokens to any whitelisted EVM address, triggering the freeze without the victim's participation. No admin compromise is required; only a reachable, supported configuration state.

---

### Recommendation

`is_allow_receive_erc20_tokens` must enforce the same Account whitelist check as `is_allow_submit`. The corrected function should be:

```rust
pub fn is_allow_receive_erc20_tokens<I: IO + Copy>(
    io: &I,
    account: &AccountId,
    address: &Address,
) -> bool {
    is_address_allowed(io, address) && is_account_allowed(io, account)
}
```

This ensures that tokens are only minted to an EVM address when the controlling NEAR account can also submit EVM transactions to spend them. If the combined check fails, the fallback redirect should fire as intended.

---

### Proof of Concept

Using `StoragePointer` (as the question suggests):

```rust
// Pseudocode integration test
let storage = RefCell::new(Storage::default());
let io = StoragePointer(&storage);

// Enable Address whitelist, add victim EVM address
let mut addr_list = Whitelist::init(&io, WhitelistKind::Address);
addr_list.enable();
addr_list.add(&victim_address);

// Enable Account whitelist, do NOT add victim NEAR account
let mut acct_list = Whitelist::init(&io, WhitelistKind::Account);
acct_list.enable();
// victim_near_account intentionally omitted

// Assert: receive check passes (tokens will be minted)
assert!(is_allow_receive_erc20_tokens(&io, &victim_address)); // true

// Assert: submit check fails (tokens cannot be spent)
assert!(!is_allow_submit(&io, &victim_near_account, &victim_address)); // false

// Tokens are now permanently frozen.
``` [5](#0-4)
