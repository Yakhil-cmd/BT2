### Title
Gateway Stateful Validator Uses Current Block's L2 Gas Price Instead of `next_l2_gas_price` for Admission Threshold, Admitting Transactions That Will Fail Blockifier Pre-Validation — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

`StatefulTransactionValidator::validate_resource_bounds` reads `block_info.gas_prices.strk_gas_prices.l2_gas_price` — the **current** (latest committed) block's L2 gas price — as the reference for its admission threshold. Transactions are executed in the **next** block, whose L2 gas price is `next_l2_gas_price` stored in the same block header. When EIP-1559 adjustments push `next_l2_gas_price` above `l2_gas_price` (full-block scenario), transactions whose `max_price_per_unit` falls between the two values pass the gateway threshold but are rejected by the blockifier's `check_fee_bounds` pre-validation. The code itself carries an unresolved `TODO` acknowledging the wrong field is being used.

The blockifier's `check_fee_bounds` (called from `perform_pre_validation_stage`) enforces that `max_price_per_unit >= actual_gas_price` where `actual_gas_price` is the **next block's** L2 gas price:

```rust

``` [5](#0-4) 

The threshold computed by the gateway is:

```
threshold = (min_gas_price_percentage / 100) * l2_gas_price   [WRONG]
```

It should be:

```
threshold = (min_gas_price_percentage / 100) * next_l2_gas_price   [CORRECT]
``` [6](#0-5) 

### Impact Explanation

**High — Mempool/gateway admission accepts invalid transactions before sequencing.**

Any user can craft a V3 `Invoke` transaction with `AllResources` bounds whose `l2_gas.max_price_per_unit` satisfies the gateway threshold (`>= min_gas_price_percentage% × l2_gas_price`) but falls below `next_l2_gas_price`. The gateway admits the transaction to the mempool; the batcher later attempts execution; the blockifier rejects it at `check_fee_bounds` pre-validation. The transaction occupies mempool slots, consumes batcher scheduling resources, and is silently dropped — the user receives no fee refund because pre-validation failure does not charge fees, but the transaction is never sequenced. In the reverse direction (falling `next_l2_gas_price`), the threshold is too high and valid transactions are incorrectly rejected at the gateway.

### Likelihood Explanation

The EIP-1559 fee market adjusts `next_l2_gas_price` every block. During sustained high load the price rises by up to `1/8` per block. The discrepancy between `l2_gas_price` and `next_l2_gas_price` is therefore a routine, not exceptional, condition. Any user submitting a transaction priced at the current block's price (a natural choice) during a rising-price period triggers the issue. The `validate_resource_bounds` flag must be enabled in the `StatefulTransactionValidatorConfig`, which is the intended production configuration.

### Recommendation

1. Extend `GatewayFixedBlockStateReader::get_block_info` (or add a new method) to expose `next_l2_gas_price` from `BlockHeaderWithoutHash`.
2. In `validate_resource_bounds`, replace `previous_block_l2_gas_price` with the `next_l2_gas_price` value so the threshold reflects the price at which the transaction will actually be executed.
3. Remove the `TODO(Arni)` comment once the fix is applied.

### Proof of Concept

```
State:
  latest committed block:
    l2_gas_price         = 100 fri   (used IN that block)
    next_l2_gas_price    = 112 fri   (12.5% increase; block was full)
  StatefulTransactionValidatorConfig:
    validate_resource_bounds   = true
    min_gas_price_percentage   = 100   (threshold = 100% of reference price)

Step 1 — User submits Invoke V3 with:
    l2_gas.max_price_per_unit = 105 fri

Step 2 — Gateway threshold check:
    threshold = 100% × 100 = 100
    105 >= 100  →  ADMITTED to mempool   ✓ (wrong)

Step 3 — Batcher builds next block:
    block l2_gas_price = next_l2_gas_price = 112 fri

Step 4 — Blockifier check_fee_bounds:
    105 < 112  →  MaxGasPriceTooLow  →  REJECTED at pre-validation

Result: transaction admitted by gateway, rejected by blockifier, never sequenced.
```

The same logic applies in reverse: if `next_l2_gas_price = 90` (falling price), the gateway threshold is `100`, so a transaction with `max_price_per_unit = 95` is rejected by the gateway even though the blockifier would accept it (`95 >= 90`).

    // TODO(Arni): Consider running this validation for all gas prices.
    fn validate_tx_l2_gas_price_within_threshold(
        &self,
        tx_resource_bounds: ValidResourceBounds,
        previous_block_l2_gas_price: NonzeroGasPrice,
    ) -> StatefulTransactionValidatorResult<()> {
        match tx_resource_bounds {
            ValidResourceBounds::AllResources(tx_resource_bounds) => {
                let tx_l2_gas_price = tx_resource_bounds.l2_gas.max_price_per_unit;
                let gas_price_threshold_multiplier =
                    Ratio::new(self.config.min_gas_price_percentage.into(), 100_u128);
                let threshold = (gas_price_threshold_multiplier
                    * previous_block_l2_gas_price.get().0)
                    .to_integer();
                if tx_l2_gas_price.0 < threshold {
                    return Err(StarknetError {
                        // We didn't have this kind of an error.
                        code: StarknetErrorCode::UnknownErrorCode(
                            "StarknetErrorCode.GAS_PRICE_TOO_LOW".to_string(),
                        ),
                        message: format!(
                            "Transaction L2 gas price {tx_l2_gas_price} is below the required \
                             threshold {threshold}.",
                        ),
                    });
                }
            }
            ValidResourceBounds::L1Gas(_) => {
                // No validation required for legacy transactions.
            }
        }
        Ok(())
    }
