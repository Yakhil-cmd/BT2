### Title
`native_per_pubdata` Floor Division Causes Systematic Undercharging of Pubdata Costs - (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

`native_per_pubdata` is computed with floor (truncating) division while `native_per_gas` uses ceiling division. This asymmetry causes the native resource cost of every pubdata byte to be systematically underestimated, which propagates through the `deltaGas` mechanism and results in the operator being undercompensated for pubdata publication costs.

---

### Finding Description

In `validation_impl.rs` the two key ratios are computed back-to-back:

```rust
// line 135 — ceiling division
u256_try_to_u64(&gas_price.div_ceil(native_price))

// line 142 — floor (truncating) division
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

`native_per_gas` rounds **up** (conservative for the protocol), while `native_per_pubdata` rounds **down** (favorable for the user). The same floor division appears in `api/src/helpers.rs` line 427.

`native_per_pubdata` feeds two downstream calculations:

1. **Intrinsic pubdata overhead** in `create_resources_for_tx` (`gas_helpers.rs` line 352):
   ```rust
   let intrinsic_pubdata_overhead = native_per_pubdata_byte.saturating_mul(intrinsic_pubdata);
   ```

2. **Runtime pubdata charge** in `get_resources_to_charge_for_pubdata` (`gas_helpers.rs` line 430):
   ```rust
   let native = current_pubdata_spent.checked_mul(native_per_pubdata)?;
   ```

Because `native_per_pubdata` is truncated, the native resources consumed for pubdata are underreported. This shrinks `native_used`, which in turn shrinks `deltaGas` in `refund_calculation.rs` line 72:

```rust
let delta_gas = if native_per_gas == 0 {
    0
} else {
    (native_used / native_per_gas) as i64 - (gas_used as i64)
};
if delta_gas > 0 {
    gas_used += delta_gas as u64;
}
```

A smaller `deltaGas` means fewer gas units are added to `gas_used`, so the user is charged less ETH and the operator receives less fee.

The worst case occurs when `pubdata_price` is just below an integer multiple of `native_price` (e.g., `pubdata_price = 2·native_price − 1`). In that scenario `native_per_pubdata` is underestimated by nearly 1 unit per pubdata byte — roughly a **50 % undercharge** on the pubdata portion of the native cost.

---

### Impact Explanation

The operator is undercompensated for the cost of publishing transaction data to L1. The undercharge per transaction is bounded by:

```
undercharge_gas ≈ pubdata_bytes × frac(pubdata_price / native_price) / native_per_gas
```

For `pubdata_bytes = 1 000`, `pubdata_price = 1 999 999`, `native_price = 1 000 000`, `native_per_gas = 2`:

- `native_per_pubdata` = 1 (true ≈ 2)
- native undercharge = 1 000 units
- `deltaGas` undercharge ≈ 500 gas
- At `gas_price = 2 000 000 wei`: ETH undercharge ≈ 1 Gwei per transaction

The amount is small per transaction but is **systematic** — it affects every ZK transaction that generates pubdata, and the magnitude scales with pubdata volume and the fractional part of `pubdata_price / native_price`.

---

### Likelihood Explanation

Any unprivileged L2 transaction sender triggers this path. The `pubdata_price` and `native_price` are operator-set block-level parameters visible on-chain. A sender can observe when `pubdata_price % native_price` is large (close to `native_price`) and submit pubdata-heavy transactions at that moment to maximise the undercharge. No special privileges are required.

---

### Recommendation

Use `div_ceil` for `native_per_pubdata` to match the rounding convention already applied to `native_per_gas`:

```rust
// validation_impl.rs line 142
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

Apply the same fix to `api/src/helpers.rs` line 427 so the off-chain validation helper stays consistent with the bootloader.

---

### Proof of Concept

**Setup**
- `native_price = 1_000_000`
- `pubdata_price = 1_999_999` (just below `2 × native_price`)
- `gas_price = 2_000_000` → `native_per_gas = ceil(2_000_000 / 1_000_000) = 2`
- Transaction generates **1 000 bytes** of pubdata

**Actual (buggy) path**
- `native_per_pubdata = floor(1_999_999 / 1_000_000) = 1`
- Native charged for pubdata = `1 000 × 1 = 1 000`
- `native_used` = 1 000 (pubdata portion only)
- `deltaGas = floor(1_000 / 2) = 500`

**Correct path**
- `native_per_pubdata = ceil(1_999_999 / 1_000_000) = 2`
- Native charged for pubdata = `1 000 × 2 = 2 000`
- `native_used` = 2 000
- `deltaGas = floor(2_000 / 2) = 1 000`

**Difference**: user pays **500 fewer gas units** (≈ 1 Gwei at the example gas price). The operator is undercompensated by this amount on every such transaction.
