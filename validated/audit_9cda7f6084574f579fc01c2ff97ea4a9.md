Looking at the `broadcast_success` function and the `do_broadcast` (echo-broadcast) protocol in detail to assess whether a malicious party can cause honest parties to abort.

### Title
Single Malicious Participant Causes Permanent DKG Abort via Wrong `session_id` in `broadcast_success` — (`src/dkg.rs`)

---

### Summary

The final round of DKG (`broadcast_success`) requires **every** participant's broadcast `session_id` to match the locally computed one. A single malicious participant who broadcasts a wrong `session_id` causes all honest parties to abort with `ProtocolError::AssertionFailed`, permanently denying key generation — even though all prior cryptographic checks passed.

---

### Finding Description

In `do_keyshare`, after all cryptographic rounds complete, `broadcast_success` is called: [1](#0-0) 

Inside `broadcast_success`, `do_broadcast` (the reliable echo-broadcast protocol) is used to collect each participant's `(bool, session_id)` tuple: [2](#0-1) 

After collection, the check at line 321 requires **all** entries to match the local `session_id`: [3](#0-2) 

The reliable echo-broadcast protocol (`do_broadcast`) guarantees **consistency** — all honest parties receive the same value from each sender. This means if a malicious party broadcasts `(true, wrong_session_id)`, every honest party consistently receives `wrong_session_id` from that party. The `.all()` check then fails for every honest party simultaneously, and all abort.

The echo-broadcast protocol itself is parameterized to tolerate up to `f = (n-1)/3` malicious parties: [4](#0-3) 

But `broadcast_success` applies **zero fault tolerance** at the application layer — it requires all `n` parties to agree, not just `n - f`.

---

### Impact Explanation

**High — Permanent denial of key generation for honest parties.**

A single malicious participant (1 out of n, well within the f-fault tolerance the protocol claims to support) can abort DKG for all honest parties at the very last step, after all expensive cryptographic work has been completed. The honest parties have no way to identify the culprit and retry; they all receive the same `AssertionFailed` error with no actionable information.

---

### Likelihood Explanation

**High.** The attack requires no cryptographic capability — the malicious party simply sends a random or zeroed `session_id` in the final broadcast round. Any single participant in the DKG can execute this trivially. There is no defense in the current code.

---

### Recommendation

Replace the unanimous `.all()` check with a threshold-based check. Honest parties should succeed if at least `n - f` participants broadcast the correct `session_id`. Specifically:

- Count how many entries in `vote_list` have `sid == &session_id`.
- Abort only if that count falls below `n - broadcast_threshold` (where `broadcast_threshold = (n-1)/3`).
- Optionally, log or expose the identities of parties that sent a wrong `session_id` so callers can identify and exclude them.

---

### Proof of Concept

With `n = 4` parties (threshold `f = 1`):

1. Parties P1, P2, P3 complete all DKG rounds honestly; all compute the same `session_id = H(...)`.
2. P4 (malicious) participates honestly through all prior rounds but in `broadcast_success` broadcasts `(true, 0x000...0)` instead of the correct `session_id`.
3. The echo-broadcast protocol delivers `(true, 0x000...0)` consistently to P1, P2, P3.
4. Each of P1, P2, P3 evaluates `vote_list.iter().all(|(_, ref sid)| sid == &session_id)` — this is `false` because P4's entry has the wrong `session_id`.
5. All three honest parties return `Err(ProtocolError::AssertionFailed("A participant broadcast the wrong session id. Aborting Protocol!"))`. [3](#0-2) 

Key generation is permanently denied for the honest majority despite all cryptographic invariants being satisfied.
