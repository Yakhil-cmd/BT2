### Title
Missing Share Validation in CKD Coordinator Allows Malicious Participant to Corrupt Derived Key Output — (File: `src/confidential_key_derivation/protocol.rs`)

### Summary
The `do_ckd_coordinator` function receives `CKDOutput` shares (`big_y`, `big_c`) from each participant and accumulates them without any cryptographic validation. A single malicious participant can inject arbitrary group elements, causing the coordinator to produce a silently corrupted confidential derived key that honest parties will accept as legitimate.

### Finding Description
In `do_ckd_coordinator`, the coordinator collects each participant's share and blindly adds them:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
Ok(Some(ckd_output))
``` [1](#0-0) 

Each honest participant computes their share as:
- `big_y_i = λ_i · y_i · G`
- `big_c_i = λ_i · (x_i · H(pk ∥ app_id) + y_i · app_pk)`

The coordinator is supposed to accumulate these so that the final `big_c = msk · H(pk ∥ app_id) + big_y · app_pk`. However, there is **no NIZK proof, DLEQ proof, or any consistency check** that the received `(big_y_i, big_c_i)` pair satisfies this relationship with respect to the participant's known public key share.

Compare this to the robust ECDSA signing coordinator, which validates the final assembled output before returning it:

```rust
if !sig.verify(&public_key, &msg_hash) {
    return Err(ProtocolError::AssertionFailed(
        "signature failed to verify".to_string(),
    ));
}
``` [2](#0-1) 

No equivalent final-output verification exists in `do_ckd_coordinator`. [3](#0-2) 

The `compute_signature_share` function that each participant runs locally is also never verified by the coordinator against any public commitment: [4](#0-3) 

### Impact Explanation
A malicious participant sends `big_y' = big_y_i + Δ_Y` and `big_c' = big_c_i + Δ_C` for arbitrary non-zero group elements `Δ_Y`, `Δ_C`. The coordinator accumulates these without complaint and returns a `CKDOutput` whose `big_c` is offset by `Δ_C` and `big_y` is offset by `Δ_Y`. When the application calls `ckd_output.unmask(app_sk)`, it computes `big_c - app_sk · big_y`, which equals the correct key plus `Δ_C - app_sk · Δ_Y` — a value the malicious participant cannot zero out without knowing `app_sk`, so the result is an unpredictable, wrong confidential key. The honest coordinator and all honest participants accept this corrupted output as the protocol result.

This matches: **High — Corruption of CKD outputs so honest parties accept inconsistent or unusable cryptographic outputs.**

### Likelihood Explanation
Any single participant in the CKD session can trigger this. The attack requires only that the participant deviate from the protocol by sending malformed group elements — no special cryptographic capability, no key material, and no coordination with other parties is needed. In a decentralized MPC network where participants are untrusted, this is a realistic threat.

### Recommendation
Each participant should accompany their `(big_y_i, big_c_i)` share with a DLEQ (discrete-log equality) proof demonstrating that:

`big_c_i - x_i · H(pk ∥ app_id) = y_i · app_pk` and `big_y_i = y_i · G`

i.e., the same scalar `y_i` is used in both `big_y_i` and `big_c_i`, and `big_c_i` is correctly formed with respect to the participant's committed public key share. The coordinator must verify this proof before accumulating each share, analogous to how the triple generation protocol verifies `dlogeq` proofs before accumulating `big_c_j` values: [5](#0-4) 

### Proof of Concept
1. Honest participants P1, P2, P3 run `ckd(...)` with coordinator P1.
2. Malicious P2 intercepts the protocol at the `waitpoint` send step in `do_ckd_participant` and instead of sending its honest `(norm_big_y, norm_big_c)`, sends `(norm_big_y + Δ_Y, norm_big_c + Δ_C)` for arbitrary non-identity `Δ_Y, Δ_C ∈ G1`.
3. P1's coordinator loop at lines 50–55 adds these values without any check.
4. The returned `CKDOutput` has `big_y` and `big_c` each shifted by the injected deltas.
5. The application calls `unmask(app_sk)` and receives `msk · H(pk ∥ app_id) + Δ_C - app_sk · Δ_Y` — a wrong key — with no error raised anywhere in the protocol. [6](#0-5)

    mut chan: SharedChannel,
    participants: ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,

    // Receive everyone's inputs and add them together

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}
