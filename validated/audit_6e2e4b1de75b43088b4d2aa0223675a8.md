### Title
Missing Reward Settlement Before Commission Increase Allows Retroactive Commission Extraction From Delegators - (File: `src/staking/staking.cairo`)

---

### Summary

When a staker with an active commission commitment increases their commission via `set_commission`, the new higher commission is applied retroactively to the entire current epoch's rewards (V2) or current block's rewards (V3), including rewards that accrued under the old lower commission rate. This allows a staker to steal unclaimed yield from delegators.

---

### Finding Description

The `set_commission` function updates the commission rate in storage without first settling pending rewards for the current epoch. When `update_rewards_from_attestation_contract` (V2) or `update_rewards` (V3) is later called, it reads the **current** commission from storage and applies it to all pending rewards — including those that accrued before the commission change.

The internal `update_commission` function simply writes the new commission to storage with no reward settlement step: [1](#0-0) 

The reward computation function `calculate_staker_pools_rewards` then reads the current commission at the time of the call: [2](#0-1) 

and applies it to the entire epoch's (V2) or block's (V3) pool rewards: [3](#0-2) 

There is no historical tracking of commission changes. The commission at the moment of reward computation is applied to all pending rewards regardless of when they accrued.

The critical enabler is the commission commitment mechanism. Without a commitment, commission can only be **decreased**: [4](#0-3) 

But with an active commitment, commission can be **increased** up to `max_commission`: [5](#0-4) 

The developers themselves noted this gap: [6](#0-5) 

---

### Impact Explanation

**High — Theft of unclaimed yield from delegators.**

In V2 (attestation-based rewards), rewards are computed once per epoch when the staker attests. If a staker increases commission mid-epoch, the higher commission is applied to the **entire** epoch's pool rewards. Delegators receive less than they are owed for the portion of the epoch that elapsed under the old commission rate.

Concrete example:
- Pool earns 100 STRK in epoch N
- Commission starts at 10%: delegators entitled to 90 STRK
- Staker increases commission to 50% just before attesting
- `update_rewards_from_attestation_contract` applies 50% commission to all 100 STRK
- Delegators receive 50 STRK instead of 90 STRK — **40 STRK stolen** [7](#0-6) 

---

### Likelihood Explanation

**Medium.** The attack requires a staker to have previously called `set_commission_commitment` with a `max_commission` above their current commission. Any staker can do this at any time. The commitment is public, but delegators cannot protect themselves from retroactive application within the same epoch. The attack is most profitable in epochs with high delegation and large reward pools. [8](#0-7) 

---

### Recommendation

Before updating the commission in `update_commission`, settle pending rewards for the current epoch using the old commission rate. This mirrors the fix applied in the Blend protocol: load and store the state (here: compute and credit pending rewards) before changing the rate parameter.

Specifically, `set_commission` should trigger reward settlement for all active pools of the staker using the old commission before writing the new commission to storage.

---

### Proof of Concept

1. Staker calls `set_commission_commitment(max_commission: 5000, expiration_epoch: current + 10)` — commits to allowing commission up to 50%.
2. Staker sets initial commission to 10% (`set_commission(1000)`).
3. Delegators enter the pool, epoch N begins. Rewards accrue at 10% commission.
4. Near the end of epoch N (just before attesting), staker calls `set_commission(5000)` — increases to 50%.
5. Staker's operational address calls `attest(...)` on the attestation contract.
6. Attestation contract calls `update_rewards_from_attestation_contract(staker_address)`.
7. `calculate_staker_pools_rewards` reads `commission = 5000` (50%) and applies it to the entire epoch N pool rewards.
8. Delegators receive 50% of pool rewards instead of 90% — the 40% difference is credited to the staker as commission. [9](#0-8) [10](#0-9)

            // Prerequisites and asserts.
            self.general_prerequisites();

                        :staker_address, :staker_pool_info, :old_commission, :commission,
                    );

                staker_pool_info.commission.write(Option::Some(commission));
                self.emit(Events::CommissionInitialized { staker_address, commission });
            }
        }

        fn set_commission_commitment(
            ref self: ContractState, max_commission: Commission, expiration_epoch: Epoch,
        ) {
            self.general_prerequisites();
            assert!(max_commission <= COMMISSION_DENOMINATOR, "{}", Error::COMMISSION_OUT_OF_RANGE);
            let staker_address = get_caller_address();
            let staker_info = self.internal_staker_info(:staker_address);
            assert!(staker_info.unstake_time.is_none(), "{}", Error::UNSTAKE_IN_PROGRESS);
            let staker_pool_info = self.staker_pool_info.entry(staker_address);
            assert!(staker_pool_info.has_pool(), "{}", Error::MISSING_POOL_CONTRACT);
            let current_epoch = self.get_current_epoch();
            if let Option::Some(commission_commitment) = staker_pool_info
                .commission_commitment
                .read() {
                assert!(
                    !self.is_commission_commitment_active(:commission_commitment),
                    "{}",
                    Error::COMMISSION_COMMITMENT_EXISTS,
                );
            }
            // Staker must have a commission since it has a pool.
            let current_commission = staker_pool_info.commission();
            assert!(current_commission <= max_commission, "{}", Error::MAX_COMMISSION_TOO_LOW);
            assert!(expiration_epoch > current_epoch, "{}", Error::EXPIRATION_EPOCH_TOO_EARLY);
            assert!(
                expiration_epoch - current_epoch <= self.get_epoch_info().epochs_in_year(),
                "{}",
                Error::EXPIRATION_EPOCH_TOO_FAR,
            );
            let commission_commitment = CommissionCommitment { max_commission, expiration_epoch };
            staker_pool_info.commission_commitment.write(Option::Some(commission_commitment));
            self
                .emit(
                    Events::CommissionCommitmentSet {
                        staker_address, max_commission, expiration_epoch,
                    },
                );

        fn update_commission(
            ref self: ContractState,
            staker_address: ContractAddress,
            staker_pool_info: StoragePath<Mutable<InternalStakerPoolInfoV2>>,
            old_commission: Commission,
            commission: Commission,
        ) {
            if let Option::Some(commission_commitment) = staker_pool_info
                .commission_commitment
                .read() {
                if self.is_commission_commitment_active(:commission_commitment) {
                    assert!(
                        commission <= commission_commitment.max_commission,
                        "{}",
                        Error::INVALID_COMMISSION_WITH_COMMITMENT,
                    );
                    assert!(commission != old_commission, "{}", Error::INVALID_SAME_COMMISSION);
                } else {
                    assert!(
                        commission < old_commission, "{}", Error::COMMISSION_COMMITMENT_EXPIRED,
                    );
                }
            } else {
                assert!(commission < old_commission, "{}", Error::INVALID_COMMISSION);
            }

            // Update commission in storage.
            staker_pool_info.commission.write(Option::Some(commission));

            // Emit event.
            self
                .emit(
                    Events::CommissionChanged {
                        staker_address, old_commission, new_commission: commission,
                    },
                );
        }
