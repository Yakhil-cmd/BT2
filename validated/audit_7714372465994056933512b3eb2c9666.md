### Title
Triple Generation Transcript Lacks Session Binding, Enabling Proof Replay Across Sessions — (File: `src/ecdsa/ot_based_ecdsa/triples/generation.rs`)

### Summary
The `create_transcript` function in the OT-based ECDSA triple generation protocol constructs a Merlin transcript that is fully deterministic given only the participant list and threshold — no per-session nonce or session identifier is included. The dlog proofs that each participant generates to prove knowledge of their polynomial constant term `l0` are bound to this transcript. Because the transcript is identical across any two sessions sharing the same participants and threshold, a malicious participant can record their commitment `hat_big_c_i` and proof `phi_proof` from one session and replay them verbatim in a future session, contributing the same secret randomness without detection. This corrupts the independence of the resulting Beaver triples and violates the foundational security assumption of the Cait-Sith presigning protocol.

### Finding Description

**Root cause — missing session ID in `create_transcript`:**

The transcript factory for triple generation is:

```rust
fn create_transcript(
    participants: &ParticipantList,
    threshold: ReconstructionLowerBound,
) -> Result<Transcript, ProtocolError> {
    let mut transcript = Transcript::new(NEAR_TRIPLE_GENERATION_LABEL);
    transcript.message(b"group", NAME);
    let enc = rmp_serde::encode::to_vec(participants)...;
    transcript.message(b"participants", &enc);
    transcript.message(b"threshold", &u64::try_from(threshold.value())...);
    Ok(transcript)
}
``` [1](#0-0) 

The transcript contains only `(label, group_name, participants, threshold)`. There is no random session nonce. Every triple generation session run by the same participant set at the same threshold produces an **identical** transcript.

**How the dlog proof is bound to this transcript:**

Each participant proves knowledge of `l0` (the constant term of their polynomial `l`) by calling:

```rust
let my_l0_phi_proof = dlog::prove_with_nonce(
    &mut transcript.fork(b"dlog2", &me.bytes()),
    statement,   // hat_big_c_i = g^l0
    witness,     // l0
    my_l0_phi_proof_nonces[i],
)?;
``` [2](#0-1) 

The Fiat-Shamir challenge `e` is derived from `H(transcript || statement || big_k)`. Because the transcript is session-independent, the challenge `e` is fully determined by `(participants, threshold, me, hat_big_c_i, big_k)`. The proof `(e, s)` with `s = k + e·l0` is therefore **replayable** in any future session with the same parameters.

**Verification accepts the replayed proof:**

The verifier recomputes `big_k' = g^s − X^e` and re-derives the challenge from the same transcript state. Since the transcript, statement, and recomputed `big_k'` are identical to the original session, the challenge matches and the proof is accepted:

```rust
pub fn verify<C: Ciphersuite>(
    transcript: &mut Transcript,
    statement: Statement<'_, C>,
    proof: &Proof<C>,
) -> Result<bool, ProtocolError> {
    transcript.message(NEAR_DLOG_STATEMENT_LABEL, &statement.encode()?);
    let big_k = C::Group::generator() * proof.s.0 - *statement.public * proof.e.0;
    ...
    let e = frost_core::random_nonzero::<C, TranscriptRng>(&mut rng);
    Ok(e == proof.e.0)
}
``` [3](#0-2) 

**Contrast with DKG — which correctly uses a session ID:**

The DKG protocol generates a fresh 256-bit random `session_id` per run, broadcasts it, and folds it into every proof-of-knowledge challenge:

```rust
let mut my_session_id = [0u8; 32];
rng.fill_bytes(&mut my_session_id);
let session_ids = do_broadcast(&mut chan, &participants, me, my_session_id).await?;
let session_id = domain_separate_hash(&mut domain_separator, &session_ids)?;
``` [4](#0-3) 

```rust
preimage.extend_from_slice(session_id.as_ref());
``` [5](#0-4) 

Triple generation has no equivalent mechanism.

### Impact Explanation

A malicious participant who replays their `(hat_big_c_i, phi_proof)` from session 1 into session 2 contributes the **same secret scalar `l0`** to both sessions. The Beaver triple's `c` component is the sum of all participants' `l0` values; fixing one participant's contribution across sessions makes `c` values correlated across triple batches.

The Cait-Sith presigning protocol's security proof requires that triples are independently and uniformly random. Correlated triples break this assumption. In the presigning phase, the triple's `a` component feeds directly into the nonce share `k_i`; correlated `c` values across sessions can produce correlated nonce material. If a malicious coalition controls up to `t` participants and fixes all their `l0` values, the honest participant's `l0` can be inferred from the public commitment `hat_big_c_i` and the aggregated public value, leaking private nonce material. Correlated nonces across ECDSA signing sessions enable standard nonce-reuse key extraction.

This maps to: **High — Corruption of presign outputs so honest parties accept inconsistent or unusable cryptographic outputs**, with a path toward **Critical — extraction of nonce material** when the attacker controls the maximum allowed malicious participants.

### Likelihood Explanation

The attack requires only that a malicious participant is present in two or more triple generation sessions with the same participant set and threshold. In any production deployment, the same signer nodes generate many triple batches over time (triple generation is the offline phase run repeatedly to build up a supply of presignatures). The attacker needs no privileged access beyond being a legitimate protocol participant. The replay is silent — honest participants have no mechanism to detect that a commitment was used in a prior session.

### Recommendation

Add a per-session random nonce to the triple generation transcript, mirroring the DKG pattern:

1. At the start of triple generation, each participant generates a 32-byte random nonce and broadcasts it via `do_broadcast`.
2. All participants compute `session_id = H(all_nonces)`.
3. `create_transcript` absorbs `session_id` as an additional transcript message before any proofs are generated.

This ensures the Fiat-Shamir challenge is unique per session, making proof replay cryptographically infeasible.

### Proof of Concept

1. Participant `P1` (malicious) runs triple generation with `{P1, P2, P3}`, threshold `T`. It records `(hat_big_c_i, phi_proof_i)` for each of its `N` parallel triples.
2. The same group runs triple generation again (same participants, same threshold — normal operation).
3. `P1` sends the **identical** `(hat_big_c_i, phi_proof_i)` values from step 1.
4. `P2` and `P3` call `dlog::verify` with the transcript produced by `create_transcript(participants, threshold)` — identical to step 1's transcript — and the same statement `hat_big_c_i`. The recomputed challenge matches `proof.e`, so verification returns `Ok(true)`.
5. The protocol completes normally. The resulting triples from session 2 contain the same `l0` contribution from `P1` as session 1. Across `k` sessions, `P1`'s `l0` is fixed, making the `c` components of all `k` triple batches linearly dependent on a single secret value controlled by the attacker.

pub fn verify<C: Ciphersuite>(
    transcript: &mut Transcript,
    statement: Statement<'_, C>,
    proof: &Proof<C>,
) -> Result<bool, ProtocolError> {
    transcript.message(NEAR_DLOG_STATEMENT_LABEL, &statement.encode()?);

    let big_k = C::Group::generator() * proof.s.0 - *statement.public * proof.e.0;

    // Create a serialization of big_k
    // Raises error if the big_k turned out to be the identity element
    let ser = C::Group::serialize(&big_k).map_err(|_| ProtocolError::IdentityElement)?;

    transcript.message(NEAR_DLOG_COMMITMENT_LABEL, ser.as_ref());
    let mut rng = transcript.challenge_then_build_rng(NEAR_DLOG_CHALLENGE_LABEL);
    let e = frost_core::random_nonzero::<C, TranscriptRng>(&mut rng);

    Ok(e == proof.e.0)
}
