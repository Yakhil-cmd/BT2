### Title
Cross-COA Signature Replay in `EVM.validateCOAOwnershipProof` Allows Proving Ownership of a Different EVM Identity — (`File: fvm/evm/stdlib/contract.cdc`)

---

### Summary

`EVM.validateCOAOwnershipProof` does not enforce that `signedData` encodes the target `evmAddress`. A signature produced to prove ownership of one Cadence-Owned Account (`COA_1`) can be replayed verbatim to prove ownership of a different COA (`COA_2`) belonging to the same Cadence account. This is the direct Flow analog of the MetaMask account-switch mismatch: a credential bound to one EVM identity is silently accepted as proof of a different EVM identity.

---

### Finding Description

`EVM.validateCOAOwnershipProof` is an `access(all)` Cadence function callable by any unprivileged script or transaction. Its validation logic performs two independent checks:

1. **Signature check** — verifies that the supplied signatures over `signedData` satisfy the key-weight threshold for the Cadence account at `address`.
2. **COA address check** — verifies that the COA resource stored at `path` has an EVM address matching the caller-supplied `evmAddress`.

The two checks are never cross-linked: `signedData` is never required to commit to `evmAddress`. The code itself documents this gap:

> *"this function does not enforce that `signedData` includes `evmAddress`. In principle, a signature produced for one purpose could be replayed here against a different COA owned by the same Cadence account."*

Because a Cadence account can hold multiple COA resources at different storage paths (there is no protocol-level restriction on `EVM.createCadenceOwnedAccount()` being called more than once per account), an attacker who possesses a valid proof for `COA_1` can construct a new proof struct pointing `path` to `COA_2` and `evmAddress` to `COA_2`'s bytes, while reusing the identical `signedData` and signatures. Both independent checks pass, and the function returns `isValid = true` for `COA_2`. [1](#0-0) [2](#0-1) 

The function is exposed as a public Cadence entry point and is also invoked by the Cadence Arch precompile (`verifyCOAOwnershipProof`) from within EVM execution: [3](#0-2) [4](#0-3) 

The COA Solidity contract's `isValidSignature` (ERC-1271) calls the precompile with `address(this)` hardcoded as the EVM address, which is safe for that specific path. However, the Cadence function itself is `access(all)` and is directly callable with attacker-controlled `path` and `evmAddress` arguments, making the replay reachable without any privileged access. [5](#0-4) 

---

### Impact Explanation

Any off-chain or on-chain authentication flow that calls `EVM.validateCOAOwnershipProof` directly to verify that a user controls a specific EVM address is vulnerable. An attacker who obtains a valid COA ownership proof for `COA_1` (e.g., intercepted from a public transaction, or produced by the account owner for one protocol) can replay it to obtain a `isValid = true` result for `COA_2` — a completely different EVM address — without producing any new signature. This constitutes **transaction authorization bypass**: the wrong EVM identity is authenticated, enabling the attacker to impersonate `COA_2`'s owner in any system that trusts the proof result (e.g., KYC whitelists, liquidity pool access controls, ERC-1271-gated operations on EVM contracts that call `validateCOAOwnershipProof` via Cadence scripts rather than through `isValidSignature`).

---

### Likelihood Explanation

- `EVM.validateCOAOwnershipProof` is `access(all)` — reachable by any unprivileged Cadence script or transaction sender with no staking or special role required.
- A Cadence account can hold multiple COA resources at different storage paths; `EVM.createCadenceOwnedAccount()` has no per-account limit.
- The attacker only needs to obtain one valid proof for any COA of the target Cadence account. Such proofs are submitted in public transactions and are fully observable on-chain.
- No cryptographic break is required; the replay is a pure parameter substitution (`path` → `/public/coa2`, `evmAddress` → COA_2 bytes). [6](#0-5) [7](#0-6) 

---

### Recommendation

Enforce that `signedData` commits to `evmAddress` inside `validateCOAOwnershipProof`. Concretely, before verifying signatures, assert that the first 20 bytes of `signedData` equal `evmAddress` (or use a structured encoding such as `evmAddress || applicationData`). This binds each proof to exactly one EVM identity and eliminates cross-COA replay. The existing NatSpec comment already identifies this as the correct mitigation; it should be enforced at the protocol level rather than left as a caller responsibility.

---

### Proof of Concept

**Setup:** Cadence account `A` owns two COAs:
- `COA_1` stored at `/storage/coa`, published at `/public/coa`
- `COA_2` stored at `/storage/coa2`, published at `/public/coa2`

**Step 1 — Obtain a valid proof for `COA_1`:**

A legitimate protocol asks account `A` to sign challenge `C` and submit:
```
proof_1 = {
  address:        A,
  path:           /public/coa,
  signedData:     C,
  keyIndices:     [0],
  signatures:     [sig_A_over_C],
  evmAddress:     COA_1.address
}
```
`validateCOAOwnershipProof(proof_1)` → `isValid: true` ✓

**Step 2 — Replay against `COA_2` (no new signature needed):**

Attacker constructs:
```
proof_2 = {
  address:        A,          // same Cadence account
  path:           /public/coa2,  // ← changed to COA_2's path
  signedData:     C,          // same signed data
  keyIndices:     [0],        // same key indices
  signatures:     [sig_A_over_C],  // same signatures
  evmAddress:     COA_2.address   // ← changed to COA_2's EVM address
}
```

Inside `validateCOAOwnershipProof`:
- Signature check: `keyList.verify(signatureSet, signedData: C, ...)` → **passes** (same account keys, same data)
- COA address check: `coaRef.address().bytes == COA_2.address` → **passes** (path points to COA_2, evmAddress is COA_2)

Result: `isValid: true` for `COA_2` — using a signature that was never produced for `COA_2`. [2](#0-1) [8](#0-7)
