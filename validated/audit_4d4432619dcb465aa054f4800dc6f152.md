### Title
`tick()` Requires ETH at Call Time With No Pre-Funding Mechanism, Blocking Reward Minting Pipeline - (File: `L1/starkware/solidity/stake/RewardSupplier.sol`)

---

### Summary

The L1 `RewardSupplier.tick()` function requires the caller to supply ETH (`msg.value`) at call time to pay for two L1→L2 messaging operations. The contract has no `receive()` or `fallback()` function to accumulate ETH in advance, and the fee is split 50/50 between the two calls regardless of actual fee requirements. If no actor is willing to pay the ETH fees, the entire reward minting pipeline is permanently blocked and stakers cannot claim their earned yield.

---

### Finding Description

`RewardSupplier.sol` is the L1 contract responsible for consuming L2→L1 mint-request messages, minting STRK tokens, and bridging them back to L2 via StarkGate. This is triggered by calling `tick()`: [1](#0-0) 

```solidity
function tick() external payable {
    (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

    if (amountToMint > 0) {
        // consume L2→L1 messages
        for (uint256 i = 0; i < numMsgsToConsume; i++) {
            messagingContract().consumeMessageFromL2(mintRequestSource(), messagePayload);
        }
        mintManager().mintRequest(token(), amountToMint);

        uint256 msgFee = msg.value / 2;                          // ← hardcoded 50/50 split
        bridge().depositWithMessage{value: msgFee}(...);         // ← requires ETH fee

        msgFee = msg.value - msgFee;
        messagingContract().sendMessageToL2{value: msgFee}(...); // ← requires ETH fee
    }
}
```

Two structural problems exist:

1. **No pre-funding mechanism.** The contract declares no `receive()` or `fallback()` function. ETH cannot be deposited in advance; it must be supplied by whoever calls `tick()` at that exact moment. Any actor wishing to trigger minting must pay the L1→L2 messaging fees out of pocket.

2. **Hardcoded 50/50 fee split.** The fee is divided as `msg.value / 2` for `depositWithMessage` and `msg.value - msg.value/2` for `sendMessageToL2`. The actual fee requirements of the StarkGate bridge and the Starknet messaging contract are independent and may differ. A caller who provides exactly the right total ETH may still have one of the two calls revert because the split does not match the actual fee ratio. [2](#0-1) 

The L2 side of the pipeline (`update_unclaimed_rewards_from_staking_contract` → `request_funds` → `send_mint_request_to_l1_reward_supplier`) sends L2→L1 messages requesting funds, but those messages are only consumed and acted upon when `tick()` is called on L1 with sufficient ETH. [3](#0-2) 

When the L2 RewardSupplier's STRK balance is insufficient, `claim_rewards` reverts with `AMOUNT_TOO_HIGH`, blocking all staker and pool reward claims. [4](#0-3) 

---

### Impact Explanation

**High — Permanent/temporary freezing of unclaimed yield.**

If `tick()` is never called with sufficient ETH:
- Mint-request messages accumulate on L1 but are never consumed.
- The L2 `RewardSupplier` never receives STRK tokens.
- Every staker and pool member call to `claim_rewards` reverts.
- All earned but unclaimed yield is frozen for the entire protocol.

The 50/50 fee split compounds this: even a well-intentioned caller who provides enough total ETH may trigger a revert if the bridge fee and messaging fee are not equal, making it non-trivial to determine the correct `msg.value`.

---

### Likelihood Explanation

**Medium.** `tick()` is callable by any address, so there is no access-control barrier. However:
- There is no on-chain incentive or enforcement mechanism to ensure someone calls it with ETH.
- The correct `msg.value` is not documented or enforced; callers must guess or compute the fee split externally.
- As Ethereum gas and L1→L2 fee costs fluctuate, the required ETH changes, making automation fragile.
- A griefing actor could repeatedly call `tick()` with `msg.value = 0` (causing reverts) to waste gas of legitimate callers, though this does not itself freeze funds.

---

### Recommendation

1. **Add a `receive()` function** to allow the contract to accumulate ETH for fees, so `tick()` can draw from an internal balance rather than requiring the caller to supply ETH each time.
2. **Remove the hardcoded 50/50 split.** Accept separate `bridgeFee` and `messagingFee` parameters, or compute fees dynamically from the respective contracts.
3. **Alternatively**, implement a keeper/incentive mechanism that reimburses callers for the ETH spent on fees, ensuring liveness of the minting pipeline.

---

### Proof of Concept

1. L2 stakers accumulate rewards; `update_unclaimed_rewards_from_staking_contract` is called, which triggers `request_funds`, which calls `send_mint_request_to_l1_reward_supplier`, emitting L2→L1 messages. [5](#0-4) 

2. On L1, `requiredMinting()` returns `amountToMint > 0` because pending messages exist. [6](#0-5) 

3. `tick()` is called with `msg.value = 0` (or with an amount where `msg.value / 2` is less than the bridge's required fee). The call to `bridge().depositWithMessage{value: 0}` or `messagingContract().sendMessageToL2{value: 0}` reverts. The entire transaction reverts; no tokens are minted or bridged.

4. This repeats indefinitely. The L2 `RewardSupplier` balance remains zero.

5. Any staker calling `claim_rewards` on L2 hits:
   ```
   assert!(amount <= unclaimed_rewards, "{}", GenericError::AMOUNT_TOO_HIGH);
   ```
   and reverts, permanently blocking yield withdrawal. [7](#0-6)

    function tick() external payable {
        // Check if minting is required, and how much.
        (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

        if (amountToMint > 0) {
            // Prepare the L2->L1 mintRequest message for consumption.
            uint256[] memory messagePayload = new uint256[](1);
            messagePayload[0] = TOKENS_PER_MINT_REQUEST;

            // Consume the mintRequest messages.
            for (uint256 i = 0; i < numMsgsToConsume; i++) {
                messagingContract().consumeMessageFromL2(mintRequestSource(), messagePayload);
            }

            // Reuest minting of the requested amount from the mint manager.
            mintManager().mintRequest(token(), amountToMint);

            // Deposit the minted amount onto the bridge to the credit of `mintDestination`.
            uint256 msgFee = msg.value / 2;
            bridge().depositWithMessage{value: msgFee}(
                token(),
                amountToMint,
                mintDestination(),
                new uint256[](0)
            );
            emit ConsumedL2MintRequests(numMsgsToConsume, amountToMint);

            // Send a totalSupply update to L2MintCurve.
            msgFee = msg.value - msgFee;
            messagePayload[0] = IERC20(token()).totalSupply();
            messagingContract().sendMessageToL2{value: msgFee}(
                mintingCurve(),
                UPDATE_TOTAL_SUPPLY_SELECTOR,
                messagePayload
            );
        }
    }

            // Asserts.
            let staking_contract = self.staking_contract.read();
            assert!(
                get_caller_address() == staking_contract,
                "{}",
                GenericError::CALLER_IS_NOT_STAKING_CONTRACT,
            );
            let unclaimed_rewards = self.unclaimed_rewards.read();
            assert!(amount <= unclaimed_rewards, "{}", GenericError::AMOUNT_TOO_HIGH);

            // Update unclaimed_rewards and transfer the requested rewards to the staking contract.
            self.unclaimed_rewards.write(unclaimed_rewards - amount);
            let token_dispatcher = self.token_dispatcher.read();
            token_dispatcher.checked_transfer(recipient: staking_contract, amount: amount.into());
        }

        /// Requests funds from L1 to account for new rewards, if the contract's balance is too low.
        fn request_funds(ref self: ContractState, unclaimed_rewards: Amount) {
            // Read current balance.
            let token_dispatcher = self.token_dispatcher.read();
            let balance: Amount = token_dispatcher
                .balance_of(account: get_contract_address())
                .try_into()
                .expect_with_err(InternalError::BALANCE_ISNT_AMOUNT_TYPE);

            // Calculate credit, which is the contract's balance plus the amount already requested
            // from L1, and the debit, which is the unclaimed rewards.
            let mut l1_pending_requested_amount = self.l1_pending_requested_amount.read();
            let credit = balance + l1_pending_requested_amount;
            let debit = unclaimed_rewards;

            // If there isn't enough credit to cover the debit + threshold, request funds.
            let base_mint_amount = self.base_mint_amount.read();
            let threshold = compute_threshold(base_mint_amount);
            if credit < debit + threshold {
                let diff = debit + threshold - credit;
                let num_msgs = ceil_of_division(dividend: diff, divisor: base_mint_amount);
                let total_amount = num_msgs * base_mint_amount;
                for _ in 0..num_msgs {
                    self.send_mint_request_to_l1_reward_supplier();
                }
                self.emit(Events::MintRequest { total_amount, num_msgs });
                l1_pending_requested_amount += total_amount;
            }

            // Commit to storage the requested amount, which is now part of the credit.
            self.l1_pending_requested_amount.write(l1_pending_requested_amount);
        }

        fn send_mint_request_to_l1_reward_supplier(self: @ContractState) {
            let payload: Span<felt252> = array![self.base_mint_amount.read().into()].span();
            let to_address = self.l1_reward_supplier.read();
            send_message_to_l1_syscall(:to_address, :payload).unwrap_syscall();
        }
