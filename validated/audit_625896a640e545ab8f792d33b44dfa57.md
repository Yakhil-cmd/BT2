### Title
Pool Admin Bypasses `maxAdminSpreadFeeE6` Cap via Uncapped Per-Bin Additional Fees — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary
`MetricOmmPoolFactory.setPoolAdminFees` enforces `maxAdminSpreadFeeE6` on the pool-level admin spread fee, but `setPoolBinAdditionalFees` applies no cap at all on `addFeeBuyE6`/`addFeeSellE6`. A pool admin can call `setPoolBinAdditionalFees` for every bin and set each bin's additional fee to `type(uint16).max` (65 535 E6 ≈ 6.55%), stacking uncapped per-bin fees on top of the already-at-cap pool-level spread fee. The aggregate effective fee paid by traders exceeds the cap the factory is supposed to enforce.

### Finding Description
`MetricOmmPoolFactory` maintains four fee caps enforced by `setFeeCaps` and checked in `setPoolAdminFees`:

```solidity
// MetricOmmPoolFactory.sol:414-415
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (newAdminNotionalFeeE8 > maxAdminNotionalFeeE8) revert AdminFeeTooHigh();
``` [1](#0-0) 

The companion function `setPoolBinAdditionalFees` passes the caller-supplied values straight through to the pool with **no cap check**:

```solidity
// MetricOmmPoolFactory.sol:450-457
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
``` [2](#0-1) 

The pool's `setBinAdditionalFees` also performs no cap check — it writes the values directly to storage: [3](#0-2) 

During every swap, the bin additional fee is **added on top of** the pool-level spread fee:

```solidity
// MetricOmmPool.sol (swap path, buy side)
uint256 buyFeeX64 = baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6);
``` [4](#0-3) 

The same pattern appears for the sell side: [5](#0-4) 

The hard cap constants are:

```solidity
uint24 internal constant HARD_MAX_SPREAD_FEE_E6 = 200_000;   // 20 %
uint24 internal constant HARD_MAX_NOTIONAL_FEE_E8 = 1_000_000;
``` [6](#0-5) 

`uint16` max is 65 535, which in E6 units equals ≈ 6.55%. A pool admin who has already set `adminSpreadFeeE6 = maxAdminSpreadFeeE6` (20%) can additionally call `setPoolBinAdditionalFees(pool, bin, 65535, 65535)` for every bin in the pool. The effective fee a trader pays in any such bin becomes `spreadFeeE6 + addFeeBuyE6`, i.e. up to ≈ 26.55% — 6.55 percentage points above the cap the factory is supposed to enforce.

The analog to the external report's bug class is exact: `setPoolAdminFees` checks the pool-level fee per-call against the cap, but `setPoolBinAdditionalFees` applies no check, so the admin splits the excess fee increase across per-bin calls that individually look innocuous but in aggregate push the effective fee above the intended ceiling.

### Impact Explanation
Every swap that crosses a bin with `addFeeBuyE6 = 65535` pays ≈ 6.55% more than the factory cap permits. On a pool with $1 M daily volume, a malicious pool admin can extract an extra ≈ $65 500/day in fees beyond the capped amount. This is a direct, continuous loss of trader principal routed to the admin fee destination. The loss is proportional to volume and persists until the factory owner lowers `maxAdminSpreadFeeE6` (which does not retroactively clear bin fees) or the pool admin is replaced.

### Likelihood Explanation
The pool admin is semi-trusted and explicitly permitted to call `setPoolBinAdditionalFees`. No timelock, no governance delay, and no cap guard stands between the admin and setting `addFeeBuyE6 = type(uint16).max` on every bin in a single block. The trigger requires only the pool admin's private key — no external conditions.

### Recommendation
Add a cap check in `setPoolBinAdditionalFees` mirroring the check in `setPoolAdminFees`. The simplest correct fix:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (addFeeBuyE6  > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

Alternatively, introduce a dedicated `maxAdminBinSpreadFeeE6` cap that the factory owner can set independently, and enforce it here.

### Proof of Concept

```solidity
// Assume pool is deployed with adminSpreadFeeE6 = maxAdminSpreadFeeE6 (200_000 = 20%)
// Pool has bins from LOWEST_BIN to HIGHEST_BIN (e.g. -1 to 0)

factory.setPoolBinAdditionalFees(pool, -1, 65535, 65535);

// Effective buy fee = spreadFeeE6 (20%) + addFeeBuyE6 (6.55%) = 26.55%

// The excess fee accrues to the admin fee destination on every collectFees call.
