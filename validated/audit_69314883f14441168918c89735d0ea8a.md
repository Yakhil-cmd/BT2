### Title
Cross-Exit Promise Chaining Causes wNEAR Unwrap Loss When ETH Exit Fails — (`engine/src/engine.rs`)

### Summary

`filter_promises_from_logs` maintains a single shared `previous_promise` state across all exit precompile logs in one EVM transaction. When a contract emits two `ExitToNear` logs — first an ETH base-token exit (`PromiseArgs::Create`) and then a wNEAR ERC-20 exit (`PromiseArgs::Callback`) — the wNEAR `near_withdraw` call is incorrectly attached as a NEAR callback of the ETH exit promise. If the ETH exit fails on the NEAR side, `near_withdraw` never executes, and the already-burned wNEAR ERC-20 tokens are permanently lost.

---

### Finding Description

**Root cause — `filter_promises_from_logs`:** [1](#0-0) 

The function initialises `previous_promise = None` and then, for every exit-precompile log, chains the new promise onto whatever `previous_promise` currently holds. For a `PromiseArgs::Callback` variant (lines 1666–1678), `promise.base` is attached as a callback of `previous_promise`, and `promise.callback` is attached as a callback of that. After processing, `previous_promise` is updated to the last scheduled ID.

**How the two exit types are produced:**

ETH base-token exit (`exit_base_token_to_near`) always returns `transfer_near_args = None`: [2](#0-1) 

Because `callback_args == ExitToNearPrecompileCallbackArgs::default()`, the ETH exit emits `PromiseArgs::Create(ft_transfer)`: [3](#0-2) 

wNEAR ERC-20 exit (`exit_erc20_token_to_near`) returns `transfer_near_args = Some(...)`: [4](#0-3) 

Because `callback_args != default()`, the wNEAR exit emits `PromiseArgs::Callback { base: near_withdraw, callback: exit_to_near_precompile_callback }`: [5](#0-4) 

**Resulting NEAR promise graph (wrong):**

```
ft_transfer (ETH exit)
    └─ near_withdraw          ← callback, only fires if ft_transfer succeeds
           └─ exit_to_near_precompile_callback  ← only fires if near_withdraw succeeds
```

**Expected independent graphs:**

```
ft_transfer (ETH exit)        [independent]
near_withdraw                 [independent]
    └─ exit_to_near_precompile_callback
```

The `exit_to_near_precompile_callback` is the function that actually transfers the unwrapped NEAR to the recipient: [6](#0-5) 

---

### Impact Explanation

When the ETH `ft_transfer` fails on the NEAR side (e.g., recipient account does not exist, ETH connector rejects the transfer), the NEAR runtime never schedules `near_withdraw`. The wNEAR ERC-20 tokens have already been burned from the EVM state (committed before promises are dispatched), so the user loses the full wNEAR principal with no recourse. The NEAR equivalent remains locked in Aurora's account permanently.

Impact: **High — Theft of unclaimed yield / permanent loss of the unwrapped NEAR.**

---

### Likelihood Explanation

- Any Solidity contract that performs two sequential `ExitToNear` calls in one transaction (e.g., a DeFi router that exits ETH and wNEAR in a single atomic step) is affected.
- The ETH `ft_transfer` can fail for ordinary reasons: the recipient NEAR account does not exist, the ETH connector is paused, or the connector has insufficient registered balance.
- No special privilege or admin access is required; any EVM contract can trigger this path.

---

### Recommendation

Each exit precompile log should be scheduled **independently** rather than chained onto the previous promise. The `previous_promise` chaining was designed for XCC (cross-contract call) sequential ordering, but it must not apply across distinct exit precompile invocations. One fix is to reset `previous_promise` to `None` after each complete exit promise group is scheduled, or to track per-exit promise IDs separately from the XCC chain.
