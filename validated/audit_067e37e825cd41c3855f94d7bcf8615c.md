### Title
NFT Wallet Type Missing in WalletConnect `create_offer_for_ids` Display — (`packages/gui/src/electron/commands/parseCommandDisplay.ts`)

### Summary

`assetKindForWalletId` in `parseCommandDisplay.ts` only distinguishes `STANDARD_WALLET` (XCH) from everything else, returning the generic `'wallet'` kind for all other wallet types including `WalletType.NFT`. When a WalletConnect dApp sends `chia_wallet.create_offer_for_ids` with an NFT wallet ID on the spending side, the approval dialog renders the NFT as a generic wallet line with a CAT-formatted mojo amount instead of as an NFT, causing signing-context confusion.

### Finding Description

`assetKindForWalletId` is called inside `createOfferAssetKinds` for every numeric (wallet-ID) key on the spending side of a `create_offer_for_ids` request:

```typescript
// parseCommandDisplay.ts lines 83-91
function assetKindForWalletId(walletId: string, walletInfos: Record<string, WalletInfo>): AssetDisplayKind {
  const walletInfo = walletInfos[walletId];
  if (walletId === '1' || walletInfo?.type === WalletType.STANDARD_WALLET) {
    return 'chia';
  }
  return 'wallet';   // ← NFT, CAT, RCAT, CRCAT all collapse here
}
``` [1](#0-0) 

The returned `'wallet'` kind is then handled in `parseWalletDeltaItem`:

```typescript
// lines 293-299
if (assetKind === 'wallet' || isNumericKey(key)) {
  return {
    kind: 'wallet',
    walletId: key,
    walletName: walletInfos[key]?.name,
    amount: mojoToCATLocaleString(value),   // ← CAT decimal formatting applied to NFT quantity
  };
}
``` [2](#0-1) 

`WalletType.NFT` (value `10`) is defined in the electron-side constants but is never checked in this path: [3](#0-2) 

`getWalletInfos` does populate the `type` field for every wallet including NFT wallets, so the information is available but unused: [4](#0-3) 

### Impact Explanation

A malicious WalletConnect dApp can:

1. Query `get_wallets` to enumerate the user's NFT wallet IDs.
2. Send `chia_wallet.create_offer_for_ids` with the NFT wallet ID mapped to `-1` (offering the NFT) and a trivially small XCH amount as the requested side.
3. The WalletConnect approval dialog renders the NFT line as `kind: 'wallet'` with `amount: mojoToCATLocaleString(1n)` — displaying `"0.001"` — instead of the NFT's bech32m ID and a clear NFT indicator.
4. The user, seeing what appears to be a minor token swap, approves.
5. The dApp immediately takes the offer, transferring the NFT for near-zero value.

This satisfies the **High** impact criterion: *WalletConnect state spoofing that causes a user to approve the wrong asset, identity, or status.*

### Likelihood Explanation

Any WalletConnect-connected dApp can trigger this without any special privilege. The `chia_wallet.create_offer_for_ids` command is explicitly listed as a dApp-allowed command: [5](#0-4) 

The user's NFT wallet IDs are discoverable via `get_wallets`. No leaked keys, host compromise, or cryptographic break is required.

### Recommendation

Add an `NFT` branch in `assetKindForWalletId` (and analogously for `CRCAT`/`RCAT` if CAT-specific display is desired):

```typescript
function assetKindForWalletId(walletId: string, walletInfos: Record<string, WalletInfo>): AssetDisplayKind {
  const walletInfo = walletInfos[walletId];
  if (walletId === '1' || walletInfo?.type === WalletType.STANDARD_WALLET) {
    return 'chia';
  }
  if (walletInfo?.type === WalletType.NFT) {
    return 'nft';
  }
  return 'wallet';
}
```

When `assetKind === 'nft'`, `parseWalletDeltaItem` already has the correct NFT rendering path (lines 302–329) that calls `nftGetInfo` and displays the bech32m NFT ID with royalty info. [6](#0-5) 

### Proof of Concept

Using the existing test harness in `parseCommandDisplay.test.ts`, the following scenario demonstrates the mislabeling:

```typescript
mockGetWalletInfos.mockResolvedValue({
  10: { name: 'My NFT Wallet', type: WalletType.NFT },
});

const result = await parseCommandDisplay('chia_wallet.create_offer_for_ids', {
  offer: { 10: '-1' },   // spending 1 NFT
});

// Actual (broken): kind === 'wallet', amount === '0.001'
// Expected:        kind === 'nft',    nftId === 'nft1...'
```

The spending line resolves to `{ kind: 'wallet', walletName: 'My NFT Wallet', amount: '0.001' }` — indistinguishable from a small CAT transfer — rather than an NFT entry, because `assetKindForWalletId` returns `'wallet'` for `WalletType.NFT`. [7](#0-6)
