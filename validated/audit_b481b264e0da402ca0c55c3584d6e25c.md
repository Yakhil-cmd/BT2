### Title
`skip_stateful_validations` Admits Invoke Transactions with Invalid Signatures for Addresses with a Pending Deploy-Account - (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator skips the blockifier's `validate` entry-point call (i.e., signature verification) for any invoke transaction with `nonce = 1` when `account_tx_in_pool_or_recent_block` returns `true`. The check does not verify that the transaction in the pool is specifically a `deploy_account`; it returns `true` for **any** transaction from that address. An unprivileged attacker who observes a victim's pending `deploy_account` in the mempool can submit an invoke with `nonce = 1` from the victim's address carrying an invalid (or malicious) signature, and the gateway will admit it to the mempool without ever checking the signature.

### Finding Description

`skip_stateful_validations` is the Sequencer analog of the Splitter's `IRoyaltyVault(msg.sender).getSplitter() == address(this)` check: both rely on a condition that an attacker can satisfy through a third-party action (victim's own `deploy_account` in the pool), thereby bypassing the real authorization gate (signature validation / `incrementWindow` guard).

**Root cause — `skip_stateful_validations`:**

    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,

    }
    Ok(false)

### Impact Explanation

**High — Mempool/gateway admission accepts invalid transactions before sequencing.**

An attacker can inject an invoke with an arbitrary (invalid or malicious) signature for any victim address that has a pending `deploy_account`. The invalid transaction enters the mempool and occupies the victim's `nonce = 1` slot. When the block is produced, the `deploy_account` executes first (nonce → 1), then the attacker's invoke is executed; the `__validate__` entry point is called with `validate: true` and fails. Depending on the account's balance, the victim pays fees for the failed validation and their own legitimate `nonce = 1` invoke is displaced (requiring fee escalation or resubmission at `nonce = 2`).

### Likelihood Explanation

**Medium.** The attacker only needs to watch the public mempool for `deploy_account` transactions and race to submit a `nonce = 1` invoke before the victim does. No privileged access, special keys, or on-chain state is required. The attack window is the time between the victim's `deploy_account` entering the mempool and the victim submitting their first `nonce = 1` invoke.

### Recommendation

Replace the generic `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a `deploy_account` transaction is pending for the address. Alternatively, keep the `validate` entry point enabled but use `strict_nonce_check = false` so the nonce mismatch does not cause a hard failure — this preserves the UX intent while still verifying the signature.

### Proof of Concept

1. Victim submits `deploy_account` for address `X` (nonce = 0). It enters the mempool pool.
2. Attacker submits `Invoke V3` with `sender_address = X`, `nonce = 1`, and an invalid/arbitrary signature.
3. Gateway stateful validation path:
   - `validate_nonce`: `0 ≤ 1 ≤ 200` → **passes**
   - `validate_by_mempool` → mempool `validate_tx` checks duplicate hash and nonce gap → **passes**
   - `skip_stateful_validations`: `nonce == 1`, `account_nonce == 0`, `account_tx_in_pool_or_recent_block(X) == true` (victim's `deploy_account` is in pool) → returns **`true`**
   - `run_validate_entry_point` called with `validate = false` → blockifier `__validate__` **never called** → signature **never checked**
4. Invalid invoke is admitted to the mempool.
5. Block execution: `deploy_account` executes (nonce → 1); attacker's invoke executes with `validate = true`; `__validate__` fails; transaction reverts; victim's account is charged fees; victim's own `nonce = 1` invoke is now displaced. [7](#0-6) [1](#0-0) [3](#0-2)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
