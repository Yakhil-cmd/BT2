### Title
Opaque `extra_conditions` Pass-Through in `chia_takeOffer` WalletConnect Confirmation Dialog — (`packages/gui/src/electron/commands/Commands.ts`, `packages/api/src/services/WalletService.ts`)

---

### Summary

A WalletConnect dApp with `chia_takeOffer` permission can include attacker-controlled `extra_conditions` in its request. The GUI's confirmation dialog shows the offer's asset exchange prominently (via `walletDelta`) but renders `extra_conditions` only as an opaque raw-JSON row that a user cannot meaningfully interpret. The daemon receives and processes the conditions verbatim. The user approves what appears to be a normal offer while unknowingly authorizing additional spend conditions (e.g., `CREATE_COIN` to an attacker address).

---

### Finding Description

**Step 1 — `extra_conditions` is an allowed dApp param for `chia_takeOffer`**

The `chia_wallet.take_offer` command schema in `Commands.ts` lists `extra_conditions` as an optional `json` param: [1](#0-0) 

Because the `chia_takeOffer` dApp entry does not override the `params` list, `DappCommands.ts` inherits the full parent schema verbatim: [2](#0-1) 

**Step 2 — `parseDappParams` allowlist passes `extra_conditions` through**

`parseDappParams` builds its allowlist from `dappCommandSchema.params`. Since `extra_conditions` is in that list, a dApp payload containing it passes the gate without error: [3](#0-2) 

For `type: 'json'` params, no further type coercion or content validation is applied — the value is passed through as-is.

**Step 3 — Confirmation dialog shows `extra_conditions` as opaque JSON, not decoded spend conditions**

`humanizeParams` includes `extra_conditions` in the visible rows because it is present in the data (optional-but-provided): [4](#0-3) 

The resulting row is rendered verbatim in the dialog: [5](#0-4) 

The user sees: `Extra Conditions: [{"opcode":51,"args":["0xdeadbeef…",1000000000000]}]` — opcode numbers and raw puzzle hashes, not decoded addresses or human-readable descriptions.

**Step 4 — `parseCommandDisplay` for `chia_wallet.take_offer` ignores `extra_conditions` entirely**

The prominent `walletDelta` section is derived solely from the offer blob summary. `extra_conditions` is never decoded or incorporated into the asset-flow display: [6](#0-5) 

**Step 5 — `takeOffer` passes `extraConditions` to the daemon without stripping** [7](#0-6) 

The daemon receives and processes the conditions verbatim.

---

### Impact Explanation

A user approving a `chia_takeOffer` WalletConnect request sees a legitimate-looking offer summary (e.g., "receive 1 XCH, give 1 CAT") and an opaque JSON blob labeled "Extra Conditions" that they cannot interpret. The actual spend bundle submitted to the network includes additional conditions — such as `CREATE_COIN` to an attacker-controlled puzzle hash — that transfer assets beyond what the user understood they were approving. This constitutes unauthorized asset transfer, a Critical/High impact per the scope rules.

---

### Likelihood Explanation

Any dApp that has been granted `chia_takeOffer` permission can exploit this immediately. No additional user interaction beyond the initial pairing approval is required. The attack is invisible to a non-technical user because the opaque JSON row is easy to overlook next to the prominent, human-readable offer summary.

---

### Recommendation

1. **Strip `extra_conditions` from the `chia_takeOffer` dApp param schema** unless there is a legitimate use case for dApps to inject spend conditions. If stripped, `parseDappParams` will reject any payload containing it.
2. **If `extra_conditions` must remain**, decode each condition opcode and argument into a human-readable description (e.g., "CREATE_COIN to `xch1…` for 1 XCH") inside `parseCommandDisplay` for `chia_wallet.take_offer`, and surface it in the `walletDelta` display alongside the offer's own asset flows — not as a raw JSON row.
3. Add a unit test asserting that a `chia_takeOffer` payload containing `extra_conditions` either (a) is rejected by `parseDappParams`, or (b) causes the decoded condition to appear in the confirmation display before `takeOffer` is called.

---

### Proof of Concept

```
1. Pair a WalletConnect dApp with chia_takeOffer permission.
2. Send a session_request for chia_takeOffer with:
   {
     "offer": "<valid_offer_blob>",
     "fee": "0",
     "extraConditions": [
       {"opcode": 51, "args": ["<attacker_puzzle_hash>", 1000000000000]}
     ]
   }
3. Observe the confirmation dialog:
   - walletDelta shows only the offer's asset exchange (looks legitimate).
   - "Extra Conditions" row shows raw JSON — opcode 51 is CREATE_COIN; the
     puzzle hash and amount are not decoded.
4. User clicks Accept.
5. The take_offer RPC is sent to the daemon with extra_conditions intact.
6. The resulting spend bundle includes a CREATE_COIN condition sending
   1 XCH to the attacker's address, in addition to the offer settlement.
```

export async function parseCommandDisplay(command: string, params: Record<string, unknown>) {
  if (command === 'chia_wallet.take_offer') {
    if (!params.offer || typeof params.offer !== 'string') {
      throw new Error('Offer is not valid');
    }

    const offerSummary = await getOfferSummary(params.offer);
    if (!offerSummary || !offerSummary.summary || !offerSummary.success) {
      throw new Error('Offer is not valid');
    }

    const { summary } = offerSummary;

    const walletDelta = offerSummaryToWalletDelta(summary);
    const walletInfos = await getWalletInfos();
    const assetKinds = offerSummaryAssetKinds(summary);
    const royaltyPercentages = offerSummaryRoyaltyPercentages(summary);
    const fees = parseMojos(summary.fees);

    return {
      walletDelta: await walletDeltaToDisplay(walletDelta, walletInfos, assetKinds, royaltyPercentages, fees),
    };
  }
