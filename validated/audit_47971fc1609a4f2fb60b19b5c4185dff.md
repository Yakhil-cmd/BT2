### Title
Pool Admin Can Set Uncapped Per-Bin Additional Fees via `setPoolBinAdditionalFees`, Bypassing `maxAdminSpreadFeeE6` Cap â€” (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

The factory enforces `maxAdminSpreadFeeE6` and `maxAdminNotionalFeeE8` caps on the pool admin's base fee via `setPoolAdminFees`. However, `setPoolBinAdditionalFees` passes `addFeeBuyE6` / `addFeeSellE6` directly to the pool with **no cap validation**, allowing the pool admin to impose per-bin fees up to `uint16.max = 65 535` (â‰ˆ 6.55 % in E6 units) on top of the already-capped base spread fee, bypassing the factory's admin-fee boundary.

---

### Finding Description

`setPoolAdminFees` correctly enforces the factory cap:

```solidity
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (newAdminNotionalFeeE8 > maxAdminNotionalFeeE8) revert AdminFeeTooHigh();
``` [1](#0-0) 

But `setPoolBinAdditionalFees` performs **no analogous validation** before forwarding to the pool:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
``` [2](#0-1) 

The pool's `setBinAdditionalFees` likewise applies no cap:

```solidity
s.addFeeBuyE6 = addFeeBuyE6;
s.addFeeSellE6 = addFeeSellE6;
``` [3](#0-2) 

During every swap the bin additional fee is **added on top of** the oracle-derived base fee before the swap step executes:

```solidity
params.baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6),
``` [4](#0-3) 

Because `addFeeBuyE6` / `addFeeSellE6` are `uint16`, the pool admin can set them to `65 535` (â‰ˆ 6.55 %) for any bin. Combined with a base `spreadFeeE6` already at the hard cap (`HARD_MAX_SPREAD_FEE_E6 = 200 000` = 20 %), the effective per-bin fee reaches â‰ˆ 26.55 %, well above the hard limit the factory is supposed to enforce. [5](#0-4) 

The external bug-report analog is **"missing validation of bid parameters"**: just as the router accepted auction bids without range-checking expiry or amount, the factory here accepts bin fee parameters without range-checking them against the admin cap.

---

### Impact Explanation

A pool admin (semi-trusted, expected to operate only within factory-enforced caps) can silently impose fees far above the `maxAdminSpreadFeeE6` boundary on any individual bin. Traders routing through that bin receive fewer output tokens than the oracle price and the declared fee caps would permit â€” a direct, per-swap loss of user principal. Because the fee is applied inside `_executeSwap` before the callback balance check, the trader cannot detect or reject it at the protocol level.

---

### Likelihood Explanation

Any pool admin â€” including one who accepted the role after a `proposePoolAdminTransfer` / `acceptPoolAdmin` sequence â€” can call `setPoolBinAdditionalFees` at any time with no timelock. The call requires no special privilege beyond being the current `poolAdmin[pool]`. A malicious or compromised admin can target the bin the current price sits in (`curBinIdx`) to maximise immediate impact.

---

### Recommendation

Add a cap check in `setPoolBinAdditionalFees` before forwarding to the pool, mirroring the pattern used in `setPoolAdminFees`:

```solidity
function setPoolBinAdditionalFees(
    address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6
) external override nonReentrant onlyPoolAdmin(pool) {
    // NEW: enforce the same admin spread cap
    if (addFeeBuyE6  > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

Alternatively, introduce a dedicated `maxAdminBinAdditionalFeeE6` cap that the factory owner can configure independently.

---

### Proof of Concept

1. Factory is deployed with `maxAdminSpreadFeeE6 = 10 000` (1 %).
2. Pool is created; `spreadFeeE6` is set to 1 % (admin portion at cap).
3. Pool admin calls:
   ```solidity
   factory.setPoolBinAdditionalFees(pool, 0, 65535, 65535);
   ```
   No revert â€” `setPoolBinAdditionalFees` performs no cap check.
4. A trader calls `pool.swap(...)` with a route through bin 0.
5. Inside `_swapToken0ForToken1SpecifiedInput`, the effective fee applied is:
   `baseFeeX64 + mulDiv(65535, ONE_X64, 1e6)` â‰ˆ base + 6.55 %.
6. The trader receives â‰ˆ 5.55 % fewer tokens than the declared 1 % admin cap would allow, with the excess silently retained in the pool as LP/admin surplus â€” a direct loss of trader principal above the protocol-enforced boundary.
