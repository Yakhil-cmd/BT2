### Title
WalletConnect `take_offer` Approval Dialog Understates Total Royalty Cost via Division-by-N Error in `formatAmountWithRoyalties` - (File: packages/gui/src/electron/commands/parseCommandDisplay.ts)

### Summary

`formatAmountWithRoyalties` in `parseCommandDisplay.ts` incorrectly divides the fungible amount by the number of NFTs before computing each NFT's royalty, producing a "Total Amount with Royalties" figure that is systematically understated in the WalletConnect approval dialog. A user who accepts a multi-NFT offer via WalletConnect will spend significantly more XCH/CAT than the approval dialog shows.

### Finding Description

When a WalletConnect dApp sends a `chia_wallet.take_offer` command containing multiple NFTs with royalties, `formatAmountWithRoyalties` is called to compute the displayed total cost including royalties:

```typescript
// packages/gui/src/electron/commands/parseCommandDisplay.ts
const splitAmount = amount / BigInt(royaltyPercentages.length);   // ← divides by N
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
const totalAmount = amount + royaltyAmount;
``` [1](#0-0) 

The `splitAmount = amount / N` step is wrong. Each NFT's royalty is independently owed on the **full** fungible amount, not on `amount / N`. The correct formula is:

```
royaltyAmount = Σ (amount × royaltyPercentage_i / 10_000)
```

The buggy formula computes:

```
royaltyAmount = Σ ((amount / N) × royaltyPercentage_i / 10_000)
             = (1/N) × Σ (amount × royaltyPercentage_i / 10_000)
```

This understates the true royalty cost by a factor of N (the number of NFTs), plus additional truncation from BigInt integer division at each step.

The resulting `amountWithRoyalties` string is then rendered directly in the WalletConnect confirmation dialog as "Total Amount with Royalties": [2](#0-1) 

The existing test confirms the buggy behavior — for 2 NFTs with 5% and 0.1% royalties on 0.0001 XCH, the dialog shows `0.00010255` XCH, but the correct total is `0.0001051` XCH: [3](#0-2) 

### Impact Explanation

A user approves a WalletConnect `take_offer` request believing they will spend the displayed "Total Amount with Royalties." The actual on-chain transaction charges the correct (higher) royalty amounts computed by the backend. The user's wallet is debited more XCH or CAT than the approval dialog showed. With N NFTs each carrying a royalty percentage R, the displayed total is understated by a factor of N relative to the true royalty cost. For example, 3 NFTs with 10%, 5%, and 3% royalties would show a 6% combined royalty when the true cost is 18%.

### Likelihood Explanation

Any unprivileged WalletConnect dApp can craft a `take_offer` payload containing multiple NFTs with royalties. The victim only needs to have a WalletConnect session open and approve the offer. No special access, leaked keys, or social engineering beyond the normal WalletConnect approval flow is required. The discrepancy grows with the number of NFTs and the magnitude of royalty percentages, making it straightforward to construct offers where the understated amount is material.

### Recommendation

Remove the `splitAmount` division. Each NFT's royalty must be computed on the full `amount`:

```typescript
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (amount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
``` [4](#0-3) 

Update the corresponding test to assert the correct value (`0.0001051` XCH, not `0.00010255` XCH) to prevent regression.

### Proof of Concept

**Setup:** Attacker controls a WalletConnect dApp. Victim has an active WalletConnect session.

1. Attacker creates an offer: 2 NFTs (NFT-A with 5% royalty, NFT-B with 0.1% royalty) in exchange for 0.0001 XCH (100,000,000 mojos).
2. Attacker sends `chia_wallet.take_offer` via WalletConnect.
3. `parseCommandDisplay` calls `formatAmountWithRoyalties(line, 100_000_000n, [500, 10])`.
4. `splitAmount = 100_000_000n / 2n = 50_000_000n`.
5. `royaltyAmount = (50_000_000 × 500 / 10_000) + (50_000_000 × 10 / 10_000) = 2_500_000 + 50_000 = 2_550_000`.
6. Dialog displays: **"Total Amount with Royalties: 0.00010255 XCH"**.
7. Correct total: `(100_000_000 × 500 / 10_000) + (100_000_000 × 10 / 10_000) = 5_000_000 + 100_000 = 5_100_000` → **0.0001051 XCH**.
8. Victim approves seeing 0.00010255 XCH; wallet actually spends 0.0001051 XCH — ~2.4% more than shown. With higher royalties or more NFTs the gap scales proportionally. [1](#0-0) [5](#0-4)

function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],
): string | undefined {
  if (royaltyPercentages.length === 0 || line.kind === 'nft') {
    return undefined;
  }

  const splitAmount = amount / BigInt(royaltyPercentages.length);
  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;

  if (line.kind === 'xch') {
    return mojoToChiaLocaleString(totalAmount);
  }

  return mojoToCATLocaleString(totalAmount);
}
```
