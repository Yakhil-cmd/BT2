### Title
Unvalidated `app_pk` Parameter in CKD Protocol Allows Coordinator to Extract Confidential Derived Key - (File: src/confidential_key_derivation/protocol.rs)

---

### Summary

The `ckd` function and its internal `compute_signature_share` helper accept an `app_pk` (`PublicKey = blstrs::G1Projective`) parameter with no validation that it is a non-identity group element. A malicious coordinator who controls the `app_pk` value supplied to all participants can set it to the G1 identity point, causing each participant's masked ciphertext share to collapse to the unmasked secret share. The coordinator then directly reconstructs `msk · H(pk, app_id)` — the confidential derived key — without possessing the application's secret key `app_sk`.

---

### Finding Description

In `src/confidential_key_derivation/protocol.rs`, `compute_signature_share` computes each participant's contribution as:

```
big_s  = hash_point * private_share   // = x_i · H(pk, app_id)
big_c  = big_s + app_pk * y           // masking term
norm_big_c = big_c * lambda_i
``` [1](#0-0) 

The masking term `app_pk * y` is the only protection that prevents the coordinator from reading the secret contribution `big_s`. If `app_pk` is the G1 identity element (`G1Projective::identity()`), then `app_pk * y = identity`, so `big_c = big_s` and `norm_big_c = lambda_i · big_s_i`.

The coordinator aggregates all received shares:

```
sum(norm_big_c_i) = sum(lambda_i · big_s_i)
                  = sum(lambda_i · x_i) · H(pk, app_id)
                  = msk · H(pk, app_id)
``` [2](#0-1) 

This is the confidential derived key, which is supposed to be recoverable only by the application holding `app_sk` via `CKDOutput::unmask`. [3](#0-2) 

Neither the public entry point `ckd()` nor `compute_signature_share` validates that `app_pk` is a non-identity element: [4](#0-3) 

The only validations performed are participant-list checks (count, duplicates, membership). No check on `app_pk` exists.

---

### Impact Explanation

**Critical — Extraction/disclosure of confidential derived secrets.**

When `app_pk = G1::identity()`, the coordinator receives each participant's unmasked `lambda_i · x_i · H(pk, app_id)` share directly. Summing these yields `msk · H(pk, app_id)` — the full confidential derived key — without needing `app_sk`. The entire confidentiality guarantee of the CKD protocol is broken for any `app_id` the attacker targets.

This matches the allowed impact: *"Critical: Extraction, reconstruction, or disclosure of … confidential derived secrets."*

---

### Likelihood Explanation

In a real MPC deployment the coordinator is the entity that orchestrates the protocol and distributes session parameters (including `app_pk`) to participants. A malicious coordinator can trivially supply `G1Projective::identity()` as `app_pk` to all participants. Because the library performs no identity-point check, every honest participant silently computes and sends the unmasked share. The attack requires no cryptographic break, no leaked keys, and no external dependency — only control of the `app_pk` argument, which is a normal coordinator privilege.

---

### Recommendation

Add an explicit non-identity check for `app_pk` in `ckd()` before the protocol begins:

```rust
if app_pk == ElementG1::identity() {
    return Err(InitializationError::BadParameters(
        "app_pk must not be the identity element".to_string(),
    ));
}
```

Additionally, consider having participants broadcast a commitment to `app_pk` at the start of the protocol so that all parties can verify they are using the same value, preventing a coordinator from supplying different `app_pk` values to different participants.

---

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}
