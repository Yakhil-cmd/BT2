### Title
Malicious Participant Can Abort Triple Generation by Crafting a dlogeq Proof with Identity Reconstructed Commitment — (`src/crypto/proofs/dlogeq.rs`, `src/ecdsa/ot_based_ecdsa/triples/generation.rs`)

### Summary
`dlogeq::verify` propagates `Err(ProtocolError::IdentityElement)` when the reconstructed commitment `big_k1` is the group identity, instead of returning `Ok(false)`. A malicious participant can craft a proof `(e, s)` and a statement value `public1` (= `big_c_j`) such that `big_k1 = generator1*s - public1*e = identity`, triggering this error and permanently aborting triple generation for all honest parties.

### Finding Description
In `dlogeq::verify` (`src/crypto/proofs/dlogeq.rs`, lines 153–157), the verifier reconstructs:

```
big_k0 = G*s - public0*e
big_k1 = generator1*s - public1*e
```

These are passed to `encode_two_points`, which calls `C::Group::serialize` on each. Serialization of the identity element fails, returning `Err(ProtocolError::IdentityElement)`. This error propagates through `verify` via `?` and then through `do_generation_many` via `?` at line 487, aborting the entire protocol.

The only identity guard in `verify` checks whether `generator1` (the alternate generator) is identity — it does not guard against the *reconstructed* `big_k0` or `big_k1` being identity. [3](#0-2) 

In `do_generation_many`, `public1` is taken directly from the attacker-controlled `big_c_j` field received at `wait4`, with no prior commitment binding it: [6](#0-5) 

`generator1` is the aggregated `big_f.eval_at_zero()?.value()`, computed from all participants' `big_f_i` values received at `wait2`. The attacker receives these values before they must send at `wait4`, so they know `generator1` when crafting their malicious `big_c_j` and proof.

### Impact Explanation
Triple generation (`generate_triple` / `generate_triple_many`) is permanently aborted for all honest parties. Since OT-based ECDSA signing depends on pre-generated triples, this constitutes permanent denial of signing capability. This matches: **High: Permanent denial of signing for honest parties under valid protocol inputs and documented trust assumptions.**

### Likelihood Explanation
Any single malicious participant in the triple generation protocol can execute this attack. It requires no cryptographic breaks — only arithmetic over the group. The attacker needs to observe `wait2` messages (public broadcast) to learn `generator1`, then send a crafted `big_c_j` and proof at `wait4`. This is a single-round, low-effort attack.

### Recommendation
In `dlogeq::verify`, before calling `encode_two_points`, check whether `big_k0` or `big_k1` is the group identity and return `Ok(false)` rather than propagating `Err`:

```rust
let big_k0 = phi0 - *statement.public0 * proof.e.0;
let big_k1 = phi1 - *statement.public1 * proof.e.0;

if big_k0.ct_eq(&C::Group::identity()).into()
    || big_k1.ct_eq(&C::Group::identity()).into()
{
    return Ok(false);
}

let enc = encode_two_points::<C>(&big_k0, &big_k1)?;
```

This treats identity-producing proofs as invalid (verification failure) rather than exceptional (protocol abort).
