### Title
Permanent Freezing of STRK Rewards When Pool Balance Falls Below `min_delegation_for_rewards` Threshold — (File: src/pool/pool.cairo)

---

### Summary

When the total pool balance is below `min_delegation_for_rewards` (1 STRK = 10^18 fri for STRK pools), the pool's `compute_rewards_per_unit` function returns zero, causing STRK rewards already transferred into the pool contract by the staking contract to be permanently unclaimable by any pool member.

---

### Finding Description

In `src/pool/pool.cairo`, the internal `compute_rewards_per_unit` function (lines 967–978) explicitly returns zero when `total_stake < self.min_delegation_for_rewards.read()`:

```cairo
fn compute_rewards_per_unit(
    self: @ContractState, staking_rewards: Amount, total_stake: Amount,
) -> Index {
    if total_stake < self.min_delegation_for_rewards.read() {
        return Zero::zero();
    }
    mul_wide_and_div(
        lhs: staking_rewards, rhs: self.staking_rewards_base_value.read(), div: total_stake,
    )
        .expect_with_err(err: InternalError::REWARDS_COMPUTATION_OVERFLOW)
}
``` [1](#0-0) 

For STRK pools, `min_delegation_for_rewards = 10^18` (1 STRK), set at construction:

```cairo
pub(crate) const STRK_CONFIG: TokenRewardsConfig = TokenRewardsConfig {
    decimals: 18, min_for_rewards: 10_u128.pow(18), base_value: 10_u128.pow(28),
};
``` [2](#0-1) 

This zero return is used inside `update_rewards_from_staking_contract` (lines 569–587), which is called by the staking contract after it has **already transferred** STRK rewards to the pool:

```cairo
fn update_rewards_from_staking_contract(
    ref self: ContractState, rewards: Amount, pool_balance: Amount,
) {
    self.assert_caller_is_staking_contract();
    let (_, last) = self.cumulative_rewards_trace.last().unwrap();
    self
        .cumulative_rewards_trace
        .insert(
            key: self.get_current_epoch(),
            value: last
                + self
                    .compute_rewards_per_unit(
                        staking_rewards: rewards, total_stake: pool_balance,
                    ),
        );
}
``` [3](#0-2) 

The staking contract's `update_pool_rewards` (lines 1865–1891) transfers STRK to the pool via `send_rewards_to_delegation_pool` and then calls `update_rewards_from_staking_contract`. The transfer happens unconditionally whenever `pool_rewards > 0`: [4](#0-3) 

The staking contract's `calculate_staker_pools_rewards` only gates on `pool_rewards.is_non_zero()` — it has no awareness of the pool's `min_delegation_for_rewards` threshold: [5](#0-4) 

When `pool_balance < 10^18` but `pool_rewards > 0` (which is arithmetically possible — e.g., pool_balance = 5×10^17, total_stake = 10^21, total_rewards = 10^18 → pool_rewards = 5×10^14 ≠ 0), the sequence is:

1. Staking contract computes `pool_rewards > 0`.
2. Staking contract transfers STRK to pool contract.
3. Staking contract calls `update_rewards_from_staking_contract(rewards, pool_balance)`.
4. Pool contract calls `compute_rewards_per_unit` → returns **0** because `pool_balance < 10^18`.
5. Cumulative sigma is unchanged.
6. All reward calculations in `calculate_rewards` use `compute_rewards_rounded_down(amount, interest: to_sigma - from_sigma, base_value)` — with `sigma_diff = 0`, every member gets **0**. [6](#0-5) 

The STRK tokens sit in the pool contract's ERC-20 balance with no mechanism to recover them. The code itself acknowledges this in the docstring:

> **Note**: Delegation rewards lost when pool balance is less than `min_delegation_for_rewards`. The staking contract continues to forward `pool_rewards` to the pool contract even in this case. [7](#0-6) 

---

### Impact Explanation

STRK reward tokens transferred to the pool contract when `pool_balance < min_delegation_for_rewards` are permanently unclaimable. `claim_rewards` computes rewards from sigma deltas (which are zero), and `exit_delegation_pool_action` only returns the staked principal. There is no admin sweep or recovery function. This constitutes **permanent freezing of unclaimed yield**, a High-severity impact under the allowed scope.

---

### Likelihood Explanation

The pool contract enforces no minimum delegation beyond `amount.is_non_zero()`: [8](#0-7) 

Any unprivileged pool member can enter with 1 fri (the smallest unit). Once the total pool balance is below 1 STRK, every subsequent reward epoch permanently locks STRK in the pool. The scenario is reachable by any delegator and requires no privileged access.

---

### Recommendation

Align the two contracts on the threshold. Either:

1. **Enforce a minimum delegation** of at least `min_delegation_for_rewards` in `enter_delegation_pool` and `add_to_delegation_pool`, so the pool balance can never fall below the threshold while active.
2. **Skip the STRK transfer** in the staking contract when `pool_balance < min_delegation_for_rewards` (requires the staking contract to query or be aware of the pool's threshold).
3. **Accumulate unclaimable rewards** in a recoverable admin-controlled reserve rather than silently discarding the sigma increment while still holding the tokens.

---

### Proof of Concept

1. Pool member calls `enter_delegation_pool` with `amount = 5×10^17` (0.5 STRK). Pool balance = 5×10^17 < `min_delegation_for_rewards` = 10^18.
2. Staker attests; staking contract computes `pool_rewards = total_rewards × 5×10^17 / total_stake`. With `total_rewards = 10^18` and `total_stake = 10^21`, `pool_rewards = 5×10^14` (non-zero).
3. Staking contract transfers 5×10^14 fri STRK to pool contract and calls `update_rewards_from_staking_contract(5×10^14, 5×10^17)`.
4. Pool contract: `compute_rewards_per_unit(5×10^14, 5×10^17)` → `5×10^17 < 10^18` → returns **0**. Sigma unchanged.
5. Pool member calls `claim_rewards` → `calculate_rewards` computes `sigma_diff = 0` → rewards = 0.
6. 5×10^14 fri STRK is permanently locked in the pool contract. Repeats every reward epoch.
