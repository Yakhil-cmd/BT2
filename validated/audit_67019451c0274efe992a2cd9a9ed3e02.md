### Title
`StateDB.Commit` Skips Account Recreation After Self-Destruct, Causing Permanent Asset Loss - (`File: fvm/evm/emulator/state/stateDB.go`)

### Summary

`StateDB.Commit` unconditionally skips the `IsCreated` check for any address that is flagged as self-destructed, even when that same address was subsequently recreated in the same transaction. The `recreated` flag tracked in `DeltaView` is never consulted during commit, so the recreated account is permanently deleted from the base view. Any balance carried over to the recreated account is silently burned.

### Finding Description

`DeltaView.CreateAccount` handles the case where `CREATE2` deploys to an address that already exists in the parent state. It internally calls `SelfDestruct` on the existing account, sets the `recreated` flag, and then marks the address as `created`: [1](#0-0) 

After this call, for address `addr`:
- `toBeDestructed[addr]` is set → `HasSelfDestructed(addr)` returns `true`
- `recreated[addr]` is set
- `created[addr]` is set → `IsCreated(addr)` returns `true`

In `StateDB.Commit`, the loop over dirty addresses checks `HasSelfDestructed` first. When it is true, the account is deleted from `baseView` and `deleted = true` is set. The subsequent `if deleted { continue }` guard then **unconditionally skips** the `IsCreated` branch: [2](#0-1) 

The `recreated` map is only consulted inside `DeltaView.GetState` to prevent reading stale storage slots from the parent: [3](#0-2) 

It is never checked in `Commit`. As a result, when both `HasSelfDestructed` and `IsCreated` are true for the same address (the recreated-account case), `Commit` deletes the account and never re-creates it in `baseView`.

### Impact Explanation

Any EVM transaction that causes `CREATE2` to deploy to an address that already holds a balance (e.g., pre-funded via a plain ETH transfer) triggers the `recreated` path in `DeltaView.CreateAccount`. After `Commit`, the account does not exist in `baseView`. The balance that was carried over to the recreated account is permanently lost — it is neither credited to any account nor returned to the sender. Code and storage set on the recreated contract are also silently discarded. This constitutes a cross-VM asset loss: FLOW tokens bridged into the EVM and held at the affected address are burned with no recovery path.

### Likelihood Explanation

The trigger is a standard EVM pattern: pre-fund a deterministic `CREATE2` address, then deploy a contract to it. This is used in production protocols (e.g., counterfactual deployment, metamorphic contracts). Any unprivileged EVM transaction author can construct this sequence. No privileged access, leaked keys, or compromised nodes are required.

### Recommendation

In `StateDB.Commit`, after deleting a self-destructed account, check whether the address is also marked as recreated (`IsCreated(addr)` is true). If so, proceed to create the account in `baseView` instead of skipping it:

```go
if db.HasSelfDestructed(addr) {
    err = db.baseView.DeleteAccount(addr)
    // ...
    deleted = true
}
// If the account was recreated after self-destruct, still create it
if deleted && db.IsCreated(addr) {
    bal := db.GetBalance(addr)
    nonce := db.GetNonce(addr)
    code := db.GetCode(addr)
    codeHash := db.GetCodeHash(addr)
    err = db.baseView.CreateAccount(addr, bal, nonce, code, codeHash)
    // ...
    continue
}
if deleted {
    continue
}
```

### Proof of Concept

1. In a prior transaction, commit an account at address `X` with a non-zero balance (e.g., via a plain ETH transfer). `baseView` now has `X` with balance `B`.
2. In a new transaction, call `db.CreateAccount(X)`.
   - `DeltaView.CreateAccount` sees `Exist(X)` = true, calls `SelfDestruct(X)` internally, sets `recreated[X]`, `created[X]`, carries balance `B` into `balances[X]`.
   - At this point: `HasSelfDestructed(X)` = true, `IsCreated(X)` = true.
3. Call `db.Commit(true)`.
   - Loop reaches `X`: `HasSelfDestructed(X)` = true → `baseView.DeleteAccount(X)` → `deleted = true` → `continue`.
   - `IsCreated(X)` check is never reached.
4. Construct a new `StateDB` from the same ledger. `db.Exist(X)` = false, `db.GetBalance(X)` = 0. Balance `B` is permanently lost. [4](#0-3) [5](#0-4) [6](#0-5)

func (d *DeltaView) GetState(sk types.SlotAddress) (gethCommon.Hash, error) {
	val, found := d.slots[sk]

		return val, nil
