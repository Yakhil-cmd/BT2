## Title
Byzantine Dealer Can Force Hours-Long BSGS Decryption via Out-of-Range Chunks Passing `verify_chunking` â€” (`rs/crypto/internal/crypto_lib/threshold_sig/bls12_381/src/ni_dkg/fs_ni_dkg/forward_secure.rs`)

## Summary

`verify_chunking` enforces only a statistical bound on the proof response `z_s` (a linear combination of chunks), not a direct per-chunk range bound. A single Byzantine dealer below the fault threshold can craft a dealing whose plaintext chunks exceed `CHUNK_MAX` while still producing a passing `verify_chunking` proof. When honest nodes call `dec_chunks` during `load_transcript`, the `HonestDealerDlogLookupTable` fails to resolve the out-of-range chunk, and the code unconditionally falls through to `CheatingDealerDlogSolver`, which the codebase explicitly documents as potentially taking hours per chunk. This blocks every subnet node from completing the DKG round and loading the threshold key.

## Finding Description

**Root cause â€” `verify_chunking` has no per-chunk range check.**

In `nizk_chunking.rs`, the only range enforcement is:

```rust
let ss = n * m * (CHUNK_SIZE - 1) * CHALLENGE_MASK;
let zz = 2 * NUM_ZK_REPETITIONS * ss;
let zz_big = Scalar::from_usize(zz);

for z_sk in nizk.z_s.iter() {
    if z_sk >= &zz_big {
        return Err(ZkProofChunkingError::InvalidProof);
    }
}
``` [1](#0-0) 

This checks that the proof response `z_s[k] = Î£(e_ijk Â· s_ij) + Ïƒ_k` is below `zz_big`. It is a bound on a *linear combination* of chunks, not on individual `s_ij` values. A cheating dealer choosing chunks slightly above `CHUNK_MAX` (e.g., `CHUNK_SIZE + 1 = 65537`) increases the sum only marginally; the rejection-sampling loop in `prove_chunking` still terminates with non-negligible probability, producing a valid `z_s < zz_big`. The three algebraic checks in `verify_chunking` (Thread 1, 2, 3) verify group-element equations that hold regardless of whether chunks are in `[0, CHUNK_MAX]`.

**`verify_dealing` pipeline passes the dealing.**

`verify_zk_proofs` in `encryption.rs` calls `verify_ciphertext_integrity`, `verify_chunking`, and `verify_sharing` in sequence. [2](#0-1) 

None of these checks enforce per-chunk range bounds. The dealing is accepted into the transcript.

**`dec_chunks` falls through to `CheatingDealerDlogSolver`.**

`HonestDealerDlogLookupTable` is built only over `[CHUNK_MIN..=CHUNK_MAX]`: [3](#0-2) 

When any chunk is out of that range, `solve_several` returns `None` for that slot. `dec_chunks` then unconditionally constructs `CheatingDealerDlogSolver::new(n, m)` and calls `solve` on every unresolved chunk: [4](#0-3) 

The comment at line 830 reads: *"It may take hours to brute force a cheater's discrete log."*

**`CheatingDealerDlogSolver` is explicitly designed for an enormous search space.**

The BSGS table is up to 2 GiB, the search range is `2 * zz - 1` (which grows with `n` and `m`), and the outer loop iterates over `scale_range = 1 << CHALLENGE_BITS` values of `delta`: [5](#0-4) 

The `#[ignore]` test `print_time_for_cheating_dlog_solver_to_run` exists precisely to measure this cost: [6](#0-5)

## Impact Explanation

Every honest node in the subnet calls `dec_chunks` on every dealing included in the transcript during `load_transcript`. A single dealing with one out-of-range chunk forces all nodes to run `CheatingDealerDlogSolver` for that chunk. With production subnet sizes (28â€“40 nodes), the BSGS range and `scale_range` outer loop make this a multi-hour computation per node. During this time, `load_transcript` blocks, preventing the subnet from completing the DKG round, loading the threshold key, and resuming threshold signing. This is a concrete **subnet availability impact / consensus blocking** matching the High bounty impact: *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."*

## Likelihood Explanation

A single Byzantine dealer (one node, below any fault threshold) suffices. The attacker only needs to choose one chunk value slightly above `CHUNK_MAX` (e.g., `CHUNK_SIZE + 1 = 65537`), run the standard `prove_chunking` rejection-sampling loop (which still terminates quickly for small overflows), and submit the dealing through the normal DKG protocol flow. No privileged access, no key material, and no majority collusion is required. The attack is repeatable across every DKG round.

## Recommendation

Add an explicit per-chunk range check inside `verify_chunking` (or in `verify_dealing`/`verify_zk_proofs` before accepting a dealing into the transcript pool) that rejects any dealing whose ciphertext chunks imply plaintext values outside `[CHUNK_MIN, CHUNK_MAX]`. The most robust fix is to enforce the range at the `ChunkingWitness` construction boundary so that out-of-range witnesses are rejected before a proof is even attempted. `CheatingDealerDlogSolver` should remain as a correctness fallback but must never be reachable from a dealing that passed `verify_dealing`.
