### Title
`skip_stateful_validations` Skips `__validate__` Signature Check Based on Overly Broad Mempool Presence Test — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

`skip_stateful_validations` is intended to skip the `__validate__` entry-point call (the account's signature verification) only when a `deploy_account` transaction for the sender is pending in the mempool. However, the guard it uses — `account_tx_in_pool_or_recent_block` — returns `true` for **any** transaction in the pool or any recently committed transaction, not exclusively a `deploy_account`. An unprivileged attacker can therefore submit an unsigned (or arbitrarily signed) invoke transaction with `nonce=1` for any victim account whose on-chain nonce is `0` and that has **any** transaction in the mempool, bypassing the `__validate__` entry point at the gateway admission stage and inserting the invalid transaction into the mempool.

### Finding Description

The stateful validation path in `extract_state_nonce_and_run_validations` is:

1. Read `account_nonce` from state.
2. Call `run_pre_validation_checks` → returns `skip_validate: bool`.
3. Call `run_validate_entry_point(executable_tx, skip_validate)`. [1](#0-0) 

Inside `run_pre_validation_checks`, the nonce and resource-bounds checks run first, then `skip_stateful_validations` is called: [2](#0-1) 

`skip_stateful_validations` returns `true` (skip the `__validate__` call) when:
- tx is `Invoke`
- `tx.nonce() == Nonce(Felt::ONE)`
- `account_nonce == Nonce(Felt::ZERO)`
- `account_tx_in_pool_or_recent_block(sender_address)` returns `true` [3](#0-2) 

When `skip_validate = true`, `run_validate_entry_point` sets `validate: false` in `ExecutionFlags`, so `StatefulValidator::perform_validations` returns `Ok(())` immediately after `perform_pre_validation_stage` without ever calling the account's `__validate__` entry point: [4](#0-3) [5](#0-4) 

The critical flaw is in `account_tx_in_pool_or_recent_block`:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [6](#0-5) 

`state.contains_account` returns `true` if the address appears in `staged` or `committed` nonces — i.e., for **any** account that has ever had a transaction staged or committed, not only accounts with a pending `deploy_account`: [7](#0-6) 

`tx_pool.contains_account` returns `true` if the address has **any** transaction in the pool: [8](#0-7) 

The code comment acknowledges the broader scope but treats it as safe:

> "It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction **or transactions with future nonces that passed validations**." [9](#0-8) 

The second branch ("future nonces that passed validations") is the problem. A future-nonce invoke tx (e.g., `nonce=2`) submitted by the legitimate account owner with a valid signature causes `account_tx_in_pool_or_recent_block` to return `true`. An attacker can then submit a `nonce=1` invoke for the same account with no valid signature, and the gateway will skip `__validate__` and admit it.

The `validate_by_mempool` call that precedes `skip_stateful_validations` only checks for duplicate tx hash and `NonceTooOld`; it does not verify the signature: [10](#0-9) [11](#0-10) 

### Impact Explanation

An attacker can insert an unsigned (or garbage-signed) invoke transaction with `nonce=1` into the mempool for any victim account satisfying:
- on-chain `account_nonce == 0`
- any transaction already in the mempool or recently committed for that address

The invalid transaction occupies the `nonce=1` slot. When the victim subsequently submits their legitimate `nonce=1` invoke, the mempool rejects it with `DuplicateNonce` (or forces fee escalation). The attacker's transaction will eventually fail during batcher execution when `__validate__` is called with `validate=true`, but by then the victim's UX flow is disrupted and the mempool has accepted an unauthorized transaction.

This matches: **High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

### Likelihood Explanation

The condition is easy to trigger: any account with `nonce=0` in state that has submitted a `deploy_account` (the standard new-account flow) satisfies the bypass condition. The attacker only needs to know the victim's address (public information) and submit an invoke with `nonce=1` and a dummy signature of valid length (to pass stateless size checks). No privileged access is required.

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a `deploy_account` transaction is pending for the sender address. The mempool should expose a dedicated query such as `deploy_account_tx_in_pool(sender_address)` that only returns `true` when the pending transaction is of type `DeployAccount`. This mirrors the `PyValidator.should_run_stateful_validations` intent, which explicitly checks `deploy_account_tx_hash.is_some()`: [12](#0-11) 

### Proof of Concept

1. Victim submits `deploy_account` tx for address `V` (nonce=0). It enters the mempool. `account_tx_in_pool_or_recent_block(V)` now returns `true`.
2. Attacker constructs an `InvokeV3` transaction: `sender_address=V`, `nonce=1`, arbitrary `calldata`, dummy signature (e.g., `[0x1, 0x2]` to pass stateless length check).
3. Attacker POSTs to the gateway `/add_transaction`.
4. Gateway stateless check passes (signature length is non-zero).
5. `extract_state_nonce_and_run_validations`: `account_nonce=0`, nonce check passes (`0 ≤ 1 ≤ 200`), `validate_by_mempool` passes (no duplicate nonce=1 yet), `skip_stateful_validations` returns `true` (V is in pool, nonce==1, account_nonce==0).
6. `run_validate_entry_point` is called with `skip_validate=true` → `validate=false` → `__validate__` is never called.
7. Attacker's unsigned tx is forwarded to the mempool with `nonce=1`.
8. Victim submits their legitimate `nonce=1` invoke → mempool returns `DuplicateNonce` error. [13](#0-12)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
