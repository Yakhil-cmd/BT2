### Title
Unvalidated Pool Response `targetPuzzleHash` TOCTOU Redirects Farming Rewards to Attacker — (File: `packages/gui/src/components/plotNFT/select/PlotNFTSelectPool.tsx`)

---

### Summary

The pool-join flow fetches `/pool_info` twice from the pool server: once for display (via `usePoolInfo`) and once at submit time (via `prepareSubmitData` → `getPoolInfo`). These are two independent HTTP requests with no cross-check between them. A malicious pool operator can return a legitimate `targetPuzzleHash` for the display request and an attacker-controlled `targetPuzzleHash` for the submit request. The user sees a legitimate payout address but `pw_join_pool` is called with the attacker's address, permanently redirecting all pooled farming rewards.

---

### Finding Description

**Display path** — `PlotNFTSelectBase.tsx` calls `usePoolInfo(poolUrl)`, which calls `getPoolInfo` → `window.appAPI.fetchPoolInfo(poolUrl)` → IPC → `fetchJSON(poolUrl + '/pool_info')`. The returned `targetPuzzleHash` is rendered in `PoolInfo.tsx` for the user to review. [1](#0-0) 

**Submit path** — When the user clicks "Create" / "Join", `handleSubmit` calls `prepareSubmitData`, which issues a **second, independent** `getPoolInfo` call and blindly uses the returned `targetPuzzleHash` in `pw_join_pool`: [2](#0-1) 

The `targetPuzzleHash` from this second fetch is passed directly to `pwJoinPool`: [3](#0-2) 

The IPC handler in `main.tsx` performs no domain allowlist check and no response-field validation on the pool server's JSON: [4](#0-3) 

`fetchJSON` validates only that the URL is a valid `https`/`ipfs` URL; it does not validate the shape or content of the response: [5](#0-4) 

There is **no comparison** between the `targetPuzzleHash` shown to the user and the one used in the actual `pw_join_pool` call. The `PoolInfo` type carries `target_puzzle_hash` as a plain unvalidated string: [6](#0-5) 

---

### Impact Explanation

`targetPuzzleHash` is the on-chain payout destination for all pooled farming rewards. By returning a different value on the second fetch, a malicious pool operator causes `pw_join_pool` to commit the user's Plot NFT to an attacker-controlled puzzle hash. Every block reward earned while farming to that pool is paid to the attacker. The user has no indication anything is wrong — the displayed puzzle hash was legitimate.

This is a **Critical** impact: unauthorized payout change affecting pooled farming rewards (XCH).

---

### Likelihood Explanation

Any party who operates a pool server reachable by the user can execute this attack. The pool server only needs to serve different JSON responses for the two sequential `/pool_info` requests (e.g., keyed on a nonce, timing, or request count). No cryptographic capability, host access, or key material is required. The user's only defense — reviewing the displayed `targetPuzzleHash` — is bypassed because the displayed value is never used in the actual transaction.

---

### Recommendation

1. **Single fetch, cached result**: Fetch `/pool_info` once, store the result in component state, and pass the cached `targetPuzzleHash` directly into `prepareSubmitData` instead of re-fetching.
2. **Format validation**: Validate that `targetPuzzleHash` is a 64-character lowercase hex string before using it.
3. **Confirmation dialog**: Display the exact `targetPuzzleHash` that will be committed on-chain in a final confirmation step, sourced from the same cached fetch used for display.

---

### Proof of Concept

1. Attacker operates `https://evil-pool.example/` with a stateful `/pool_info` endpoint:
   - **Request 1** (display): returns `{ "target_puzzle_hash": "<legitimate_looking_hash>", "relative_lock_height": 32, ... }`
   - **Request 2** (submit): returns `{ "target_puzzle_hash": "<attacker_controlled_hash>", "relative_lock_height": 32, ... }`
2. User opens "Join a Pool", enters `https://evil-pool.example/`.
3. `usePoolInfo` fires request 1 → GUI displays the legitimate-looking hash.
4. User reviews the pool info and clicks "Create".
5. `prepareSubmitData` fires request 2 → receives attacker's hash.
6. `pwJoinPool({ targetPuzzlehash: "<attacker_controlled_hash>", ... })` is called.
7. The Plot NFT is committed on-chain with the attacker's payout address.
8. All subsequent farming rewards are paid to the attacker.

```typescript
async function prepareSubmitData(data: FormData): SubmitData {
  const { self, fee, poolUrl } = data;
  const initialTargetState = {
    state: self ? 'SELF_POOLING' : 'FARMING_TO_POOL',
  };

  if (!self && poolUrl) {
    const normalizedPoolUrl = normalizeUrl(poolUrl);
    const { targetPuzzleHash, relativeLockHeight } = await getPoolInfo(normalizedPoolUrl);
    if (!targetPuzzleHash) {
      throw new Error(t`Pool does not provide targetPuzzleHash.`);
    }
    if (relativeLockHeight === undefined) {
      throw new Error(t`Pool does not provide relativeLockHeight.`);
    }

    initialTargetState.poolUrl = normalizedPoolUrl;
    initialTargetState.targetPuzzleHash = targetPuzzleHash;
    initialTargetState.relativeLockHeight = relativeLockHeight;
  }
```
