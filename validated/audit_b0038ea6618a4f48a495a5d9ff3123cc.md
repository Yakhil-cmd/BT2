### Title
`EvmErc20.sol` (V1) `withdrawToNear` Sends Incompatible Input Format to `ExitToNear` Precompile When `error_refund` Feature Is Active, Causing Permanent Token Burn Without NEAR Transfer — (`etc/eth-contracts/contracts/EvmErc20.sol`, `engine-precompiles/src/native.rs`)

---

### Summary

The Aurora Engine ships two versions of its bridged ERC-20 contract (`EvmErc20.sol` V1 and `EvmErc20V2.sol`) that encode different byte layouts when calling the `ExitToNear` precompile. The precompile's own parsing logic is also version-gated by the compile-time `error_refund` feature flag. When a V1 contract calls `withdrawToNear` on an engine compiled with `error_refund`, the precompile misparses the input, returns `ERR_INVALID_AMOUNT`, and the sub-call silently fails. Because neither `EvmErc20.sol` nor `EvmErc20V2.sol` checks the assembly `call` return value, the preceding `_burn` is committed while the NEAR-side `ft_transfer` is never scheduled — permanently destroying the user's tokens.

---

### Finding Description

**Two incompatible input layouts exist side-by-side.**

`EvmErc20.sol` (V1) `withdrawToNear` encodes:

```
[0x01][amount (32 bytes)][recipient (variable)]
``` [1](#0-0) 

`EvmErc20V2.sol` encodes:

```
[0x01][sender (20 bytes)][amount (32 bytes)][recipient (variable)]
``` [2](#0-1) 

**The precompile's parser is feature-gated.**

Without `error_refund`, `parse_input` strips only the 1-byte flag:

```rust
fn parse_input(input: &[u8]) -> Result<&[u8], ExitError> {
    validate_input_size(input, MIN_INPUT_SIZE, MAX_INPUT_SIZE)?;
    Ok(&input[1..])
}
``` [3](#0-2) 

With `error_refund`, it additionally consumes bytes 1–21 as a `refund_address`:

```rust
fn parse_input(input: &[u8]) -> Result<(Address, &[u8]), ExitError> {
    validate_input_size(input, MIN_INPUT_SIZE, MAX_INPUT_SIZE)?;
    let mut buffer = [0; 20];
    buffer.copy_from_slice(&input[1..21]);
    let refund_address = Address::from_array(buffer);
    Ok((refund_address, &input[21..]))
}
``` [4](#0-3) 

For the ERC-20 flag `0x1`, the precompile then reads the next 32 bytes as the amount:

```rust
0x1 => {
    let amount = parse_amount(&input[..32])?;
    let Recipient { receiver_account_id, message } = parse_recipient(&input[32..])?;
``` [5](#0-4) 

**The mismatch when V1 meets an `error_refund`-compiled engine:**

| Byte range (after flag) | V1 intent | Precompile reads (error_refund) |
|---|---|---|
| 1–21 | first 20 bytes of `amount` | `refund_address` |
| 21–53 | last 12 bytes of `amount` + first 20 bytes of recipient | `amount` |
| 53+ | rest of recipient | `recipient` |

The "amount" the precompile sees is `amount[20..32] ++ recipient[0..20]` — a 256-bit value whose low 160 bits are ASCII text. `parse_amount` rejects it:

```rust
fn parse_amount(input: &[u8]) -> Result<U256, ExitError> {
    let amount = U256::from_big_endian(input);
    if amount > U256::from(u128::MAX) {
        return Err(ExitError::Other(Cow::from("ERR_INVALID_AMOUNT")));
    }
    Ok(amount)
}
``` [6](#0-5) 

**The unchecked assembly call seals the loss.**

In both contract versions the precompile call result is stored in `res` but never inspected:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
}
``` [7](#0-6) 

Because `_burn` executes unconditionally before the assembly block, the ERC-20 supply is reduced and the user's balance is zeroed, but no NEAR-side `ft_transfer` promise is ever created.

---

### Impact Explanation

**Critical — Permanent freezing / destruction of funds.**

Any token holder who calls `withdrawToNear` on a V1 `EvmErc20` contract deployed on an engine binary compiled with `error_refund` will have their tokens burned with no corresponding NEAR credit. The tokens cannot be recovered: the ERC-20 balance is gone, no NEAR-side receipt is emitted, and there is no refund path.

---

### Likelihood Explanation

The scenario is reachable whenever the engine is upgraded to a build that enables `error_refund` while V1 contracts remain live on-chain (a normal upgrade path, since old token contracts are not automatically migrated). Any unprivileged token holder who calls `withdrawToNear` on a V1 contract after such an upgrade triggers the loss. No special permissions or front-running are required.

---

### Recommendation

1. **Check the precompile call return value** in both `EvmErc20.sol` and `EvmErc20V2.sol` and `revert` if `res == 0`, so that `_burn` is rolled back on precompile failure.
2. **Migrate or upgrade** all live V1 `EvmErc20` contracts to V2 before enabling `error_refund` in a production build, or add a version-discriminator byte to the precompile input so a single binary can handle both layouts.
3. **Add a guard** in `validate_input_size` or the flag-dispatch logic that rejects inputs whose length is inconsistent with the expected layout for the active feature set.

---

### Proof of Concept

1. Deploy a V1 `EvmErc20` contract (bytecode from `EvmErc20.sol`) on an Aurora Engine compiled **with** the `error_refund` feature.
2. Mint tokens to `alice`.
3. `alice` calls `withdrawToNear("alice.near", 1000)`.
4. Inside `withdrawToNear`: `_burn(alice, 1000)` succeeds; alice's balance drops to 0.
5. The assembly `call` to `0xe9217bc70b7ed1f598ddd3199e80b093fa71124f` is made with input `[0x01][amount(32)]["alice.near"]`.
6. The `error_refund`-enabled precompile reads bytes 1–21 as `refund_address` (first 20 bytes of `amount`), then bytes 21–53 as `amount` (last 12 bytes of `amount` + `"alice.nea"`), producing a value >> `u128::MAX`.
7. `parse_amount` returns `ERR_INVALID_AMOUNT`; the precompile call returns 0.
8. The Solidity contract does not check `res`; execution completes normally.
9. Alice's 1000 tokens are permanently destroyed; no NEAR-side transfer occurs.
