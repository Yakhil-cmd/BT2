### Title
Missing Zero-Address Validation in `change_reward_address` Allows Permanent Freezing of Unclaimed Yield - (File: src/staking/staking.cairo, src/pool/pool.cairo)

---

### Summary
Both `staking.cairo` and `pool.cairo` implement `change_reward_address` without checking whether the supplied `reward_address` is the zero address (`0x0`). A staker or pool member can set their reward address to `0x0`, causing all future reward claims to permanently transfer tokens to the zero address, irreversibly destroying accumulated yield. The same gap exists in `stake()` and `enter_delegation_pool()`.

---

### Finding Description

**`staking.cairo::change_reward_address`** (lines 517–540) performs only a `REWARD_ADDRESS_IS_TOKEN` check and no zero-address check:

```cairo
fn change_reward_address(ref self: ContractState, reward_address: ContractAddress) {
    self.general_prerequisites();
    assert!(
        !self.does_token_exist(token_address: reward_address),
        "{}",
        GenericError::REWARD_ADDRESS_IS_TOKEN,
    );
    // ← No assert!(reward_address.is_non_zero(), ...)
    let staker_address = get_caller_address();
    let mut staker_info = self.internal_staker_info(:staker_address);
    staker_info.reward_address = reward_address;   // zero accepted
    self.write_staker_info(:staker_address, :staker_info);
``` [1](#0-0) 

**`pool.cairo::change_reward_address`** (lines 505–526) has the same gap:

```cairo
fn change_reward_address(ref self: ContractState, reward_address: ContractAddress) {
    assert!(
        self.token_dispatcher.contract_address.read() != reward_address,
        "{}",
        GenericError::REWARD_ADDRESS_IS_TOKEN,
    );
    // ← No assert!(reward_address.is_non_zero(), ...)
    let pool_member = get_caller_address();
    let mut pool_member_info = self.internal_pool_member_info(:pool_member);
    pool_member_info.reward_address = reward_address;   // zero accepted
    self.write_pool_member_info(:pool_member, :pool_member_info);
``` [2](#0-1) 

The same missing check exists at entry points `stake()` and `enter_delegation_pool()`: [3](#0-2) [4](#0-3) 

The codebase already defines `GenericError::ZERO_ADDRESS` and uses it elsewhere, confirming the pattern is known but not applied here: [5](#0-4) 

When `claim_rewards` is later called, the staking contract sends accumulated STRK to whatever address is stored as `reward_address`. If that address is zero, the tokens are transferred to `0x0` and permanently lost: [6](#0-5) 

---

### Impact Explanation
A staker or pool member who sets `reward_address = 0x0` (accidentally or via a malicious front-end) will have every future reward claim silently transfer STRK to the zero address. Because the zero address cannot sign transactions on Starknet, those tokens are unrecoverable. This constitutes **permanent freezing of unclaimed yield**, matching the allowed High impact category.

---

### Likelihood Explanation
The call path is fully reachable by any unprivileged staker or pool member with no special preconditions beyond being registered. The risk is elevated by:
- Malicious or buggy front-ends that pass `0x0` as the reward address.
- Copy-paste errors in scripts or integrations.
- The absence of any on-chain guard means the error is silent and irreversible.

---

### Recommendation
Add a non-zero assertion before accepting any `reward_address` parameter in all four affected functions:

```cairo
assert!(reward_address.is_non_zero(), "{}", GenericError::ZERO_ADDRESS);
```

Apply this to:
- `staking.cairo::stake` (before writing `staker_info`)
- `staking.cairo::change_reward_address`
- `pool.cairo::enter_delegation_pool`
- `pool.cairo::change_reward_address`

---

### Proof of Concept

1. A staker calls `stake(reward_address: 0x0, operational_address: <valid>, amount: <min_stake>)`. All checks pass (zero is not a registered token, zero is not in the operational mapping).
2. Alternatively, an existing staker calls `change_reward_address(reward_address: 0x0)`. The only guard (`REWARD_ADDRESS_IS_TOKEN`) passes because zero is not a token address.
3. The staker earns rewards over subsequent epochs via attestation.
4. Any caller invokes `claim_rewards(staker_address)`. The contract executes `checked_transfer(recipient: 0x0, amount: rewards)`, permanently destroying the accumulated STRK.
5. The staker's `unclaimed_rewards_own` is reset to zero; the tokens are gone with no recovery path.
