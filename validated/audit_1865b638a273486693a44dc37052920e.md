### Title
`EVM.mustRun()` Does Not Revert on EVM `status == failed`, Silently Proceeding Past Failed Token Transfers - (File: `fvm/evm/stdlib/contract.cdc`)

---

### Summary

`EVM.mustRun()` in the EVM system contract explicitly allows EVM transactions with `status == failed` to return without reverting the enclosing Cadence transaction. Any Cadence transaction that uses `EVM.mustRun()` (or `EVM.run()` / `coa.call()`) to perform an EVM token transfer and then proceeds with Cadence-side asset operations — without checking `res.status` — will silently continue past a failed EVM transfer, enabling cross-VM asset loss.

---

### Finding Description

`EVM.mustRun()` is defined in `fvm/evm/stdlib/contract.cdc` as:

```cadence
/// mustRun runs the transaction using EVM.run
/// It will rollback if the tx execution status is unknown or invalid.
/// Note that this method does not rollback if transaction
/// is executed but an vm error is reported as the outcome
/// of the execution (status: failed).
access(all)
fun mustRun(tx: [UInt8], coinbase: EVMAddress): Result {
    let runResult = self.run(tx: tx, coinbase: coinbase)
    assert(
        runResult.status == Status.failed || runResult.status == Status.successful,
        message: "EVM.mustRun(): The provided transaction is not valid for execution"
    )
    return runResult
}
``` [1](#0-0) 

The `assert` only rejects `Status.invalid` and `Status.unknown`. It explicitly passes through `Status.failed` — the status returned when an EVM transaction executes but the VM reports an error (e.g., out of gas, Solidity `revert`, failed ERC20 `transfer`). [2](#0-1) 

The underlying `EVM.run()` function similarly returns a `Result` struct without reverting on failure: [3](#0-2) 

At the Go handler layer, `ContractHandler.Run()` calls `panicOnError(err)` — which only panics on Go-level errors — but does **not** call `panicOnErrorOrInvalidOrFailedState`, meaning a `StatusFailed` result is returned to the Cadence layer without panicking: [4](#0-3) 

Compare this to the internal `Deposit`, `Withdraw`, and `Transfer` operations, which correctly use `panicOnErrorOrInvalidOrFailedState` and would panic on a failed EVM result: [5](#0-4) 

The `panicOnErrorOrInvalidOrFailedState` helper explicitly panics on both `Invalid()` and `Failed()` states: [6](#0-5) 

---

### Impact Explanation

An unprivileged Cadence transaction author can submit a transaction that:

1. Calls `EVM.mustRun()` (or `coa.call()`) with an EVM transaction encoding an ERC20 `transfer` or native FLOW transfer.
2. The EVM transaction fails (e.g., the ERC20 contract reverts, or gas runs out).
3. `EVM.mustRun()` returns `Result { status: failed }` **without reverting the Cadence transaction**.
4. The Cadence transaction proceeds to release Cadence-side escrowed assets (e.g., unlocking a Cadence vault) under the false assumption that the EVM transfer succeeded.

This results in **cross-VM asset loss**: Cadence-side assets are released while the EVM-side transfer never occurred. This is the direct analog to ERC20 `transfer` returning `false` instead of reverting — the outer context silently continues past a failed transfer.

The `COA.call()` and `COA.deploy()` functions have the same property — they return `Result` without reverting on `status == failed`: [7](#0-6) [8](#0-7) 

---

### Likelihood Explanation

The entry path is reachable by any unprivileged Cadence transaction sender. The `EVM.mustRun()` and `EVM.run()` functions are `access(all)` — no special entitlement or privileged account is required. Any Cadence transaction can call them. The misleading name `mustRun` implies the operation must succeed, increasing the likelihood that callers omit the result status check. The `EVM.Status` enum has four values (`unknown`, `invalid`, `failed`, `successful`), and only checking for `successful` is the correct pattern — but `mustRun` normalizes the pattern of not checking for `failed`.

---

### Recommendation

`EVM.mustRun()` should revert the Cadence transaction when the EVM result has `status == failed`, not just when it is `invalid` or `unknown`. The assert condition should be:

```cadence
assert(
    runResult.status == Status.successful,
    message: "EVM.mustRun(): EVM transaction execution failed"
)
```

Alternatively, rename the function to make clear it does not guarantee success on EVM execution failure, and add explicit documentation warnings at every call site that uses `EVM.run()` or `coa.call()` for asset-critical operations. [9](#0-8)

### Proof of Concept

```cadence
import EVM from <EVMContractAddress>
import FlowToken from <FlowTokenAddress>

// Attacker-controlled Cadence transaction
transaction(erc20TransferTx: [UInt8], coinbaseBytes: [UInt8; 20]) {
    prepare(account: auth(Storage) &Account) {
        let coinbase = EVM.EVMAddress(bytes: coinbaseBytes)

        // EVM tx encodes an ERC20.transfer() that will revert (e.g., insufficient allowance)
        let res = EVM.mustRun(tx: erc20TransferTx, coinbase: coinbase)

        // res.status == EVM.Status.failed — but mustRun did NOT revert
        // Cadence transaction continues here as if the EVM transfer succeeded

        // Attacker now releases Cadence-side escrowed vault, causing asset loss
        let vault <- account.storage.load<@FlowToken.Vault>(from: /storage/escrowedVault)!
        // deposit vault to attacker's receiver ...
    }
}
```

The EVM transaction status is `failed` (confirmed by the `EVM.Status` enum definition and the `panicOnError`-only guard in `ContractHandler.Run()`), yet `mustRun` returns without reverting, allowing the Cadence transaction to proceed and drain the escrowed vault. [10](#0-9) [11](#0-10)

func panicOnErrorOrInvalidOrFailedState(res *types.Result, err error) {

	if res != nil && res.Invalid() {
		panic(fvmErrors.NewEVMError(res.ValidationError))
	}

	if res != nil && res.Failed() {
		panic(fvmErrors.NewEVMError(res.VMError))
	}

	// this should never happen
	if err == nil && res == nil {
		panic(fvmErrors.NewEVMError(types.ErrUnexpectedEmptyResult))
	}

	panicOnError(err)
}
