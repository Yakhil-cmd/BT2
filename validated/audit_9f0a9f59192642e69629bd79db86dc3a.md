### Title
Declare Authorization Bypass via `delayed_declares` Queue: `block_declare`/`authorized_declarer_accounts` Not Re-Checked on Promotion - (File: `crates/apollo_mempool/src/mempool.rs`)

### Summary

The gateway enforces declare authorization (`block_declare` flag and `authorized_declarer_accounts` allowlist) only at admission time. Declare transactions are then held in the mempool's `delayed_declares` queue for a configurable delay period. When the delay elapses, `add_ready_declares()` promotes the declare to the active pool via `add_tx_inner()` without re-checking either authorization control. If the operator changes the config during the delay window — setting `block_declare = true` or removing an account from `authorized_declarer_accounts` — the already-queued declare bypasses the updated restriction and is sequenced.

### Finding Description

The gateway's `add_tx_inner()` calls `check_declare_permissions()` as the first gate for declare transactions: [1](#0-0) 

`check_declare_permissions()` enforces two controls: [2](#0-1) 

If both checks pass, the declare is forwarded to the mempool. The mempool's `add_tx()` detects it is a declare and, in non-FIFO mode, pushes it to the `delayed_declares` queue instead of the active pool: [3](#0-2) 

After the `declare_delay` elapses, `add_ready_declares()` is triggered (on the next `add_tx()` or `get_txs()` call) and promotes every matured entry directly via `add_tx_inner()`: [4](#0-3) 

`add_tx_inner()` performs no authorization check — it only inserts the transaction into `tx_pool` and conditionally enqueues it: [5](#0-4) 

The mempool has no reference to `GatewayStaticConfig` and no knowledge of `block_declare` or `authorized_declarer_accounts`: [6](#0-5) 

There is no mechanism to flush or invalidate `delayed_declares` entries when the gateway config changes.

### Impact Explanation

A declare transaction from an account that was authorized at submission time but subsequently removed from `authorized_declarer_accounts` — or submitted before `block_declare` was set to `true` — will be promoted from `delayed_declares` to the active pool and returned by `get_txs()` to the batcher. The batcher will execute it, causing a new contract class to be declared on-chain that the operator explicitly intended to block. This matches the **High** impact category: "Mempool/gateway/RPC admission accepts invalid transactions … before sequencing."

### Likelihood Explanation

The `declare_delay` is configurable and defaults to 1 second in production config: [7](#0-6) 

The `authorized_declarer_accounts` allowlist is the more realistic trigger: an operator revokes an account's declare permission (e.g., after a key compromise or policy change) expecting immediate effect, but any declare already in `delayed_declares` will still be sequenced. The `block_declare` flag is described as a temporary emergency block: [8](#0-7) 

An operator enabling this flag to halt all declares during an incident would find that in-flight declares in the delay queue are unaffected.

### Recommendation

`add_ready_declares()` should re-validate each matured declare against the current authorization state before calling `add_tx_inner()`. Since the mempool does not hold a reference to `GatewayStaticConfig`, two options exist:

1. Pass a predicate or authorization callback into the mempool at construction time that `add_ready_declares()` can invoke per entry, dropping entries that no longer pass.
2. Expose a `flush_delayed_declares()` method on the mempool that the gateway calls whenever `block_declare` or `authorized_declarer_accounts` changes, draining the queue of now-unauthorized entries.

### Proof of Concept

1. Operator configures `authorized_declarer_accounts = [0xALICE]` and `declare_delay = 60s`.
2. Alice submits a declare transaction. Gateway calls `check_declare_permissions()` → passes. Declare is forwarded to mempool and stored in `delayed_declares` with `submission_time = T`.
3. At `T + 1s`, operator removes Alice from `authorized_declarer_accounts` (config hot-reload or restart). Any new declare from Alice is now rejected at the gateway.
4. At `T + 60s`, the next `add_tx()` or `get_txs()` call triggers `add_ready_declares()`. Alice's declare is popped from `delayed_declares` and passed to `add_tx_inner()` with no authorization check.
5. The declare enters `tx_pool`, is enqueued, returned by `get_txs()`, and executed by the batcher — declaring a class on-chain despite Alice being de-authorized. [4](#0-3) [2](#0-1)

    fn add_tx_inner(&mut self, args: AddTransactionArgs) {
        let AddTransactionArgs { tx, account_state } = args;
        info!("Adding transaction to mempool.");
        trace!("{tx:#?}");

        let tx_reference = TransactionReference::new(&tx);

        // Pre-count this tx as stuck; update_accounts_with_gap will correct the count if this tx
        // resolves the gap.
        if self.accounts_with_gap.contains(&account_state.address) {
            self.n_stuck_txs += 1;
        }

        self.tx_pool
            .insert(tx)
            .expect("Duplicate transactions should cause an error during the validation stage.");

        let AccountState { address, nonce: incoming_account_nonce } = account_state;
        let account_nonce = self.state.resolve_nonce(address, incoming_account_nonce);

        if self.is_fifo() {
            // FIFO mode: add all transactions to the queue immediately, regardless of nonce.
            // Keep all transactions from the same address in the queue.
            self.insert_to_tx_queue(tx_reference);
        } else if tx_reference.nonce == account_nonce {
            // Fee mode: only add transactions with matching account nonce.
            // Remove queued transactions the account might have. This includes old nonce
            // transactions that have become obsolete; those with an equal nonce should
            // already have been removed via fee escalation (`remove_replaced_tx`).
            self.tx_queue.remove_by_address(address);
            self.insert_to_tx_queue(tx_reference);
        }
    }

    fn add_ready_declares(&mut self) {
        let now = self.clock.now();
        while let Some((submission_time, _args)) = self.delayed_declares.front() {
            if now - self.config.static_config.declare_delay < *submission_time {
                break;
            }
            let (_submission_time, args) =
                self.delayed_declares.pop_front().expect("Delay declare should exist.");
            self.add_tx_inner(args);
        }
        self.update_state_metrics();
    }
