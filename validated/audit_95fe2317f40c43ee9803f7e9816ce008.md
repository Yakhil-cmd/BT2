### Title
`skip_stateful_validations` Admits Nonce=1 Invoke With Invalid Signature When Any Transaction Exists in Mempool for a Deployed Account - (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

`skip_stateful_validations` is designed to skip `__validate__` for the deploy_account + invoke UX case (account not yet deployed). However, it triggers on **any** transaction in the mempool for the account — not just a deploy_account. An attacker with a deployed account at state nonce=0 can first submit a valid nonce=0 invoke (which enters the mempool), then submit a nonce=1 invoke with an **invalid signature**. All gateway checks pass, `skip_stateful_validations` returns `true`, `execution_flags.validate` is set to `false`, and `__validate__` is never called. The invalid-signature transaction is admitted to the mempool.

---

### Finding Description

The full stateful validation path in `extract_state_nonce_and_run_validations` is:

```
get_nonce_from_state → run_pre_validation_checks → run_validate_entry_point
```

`run_pre_validation_checks` calls three sub-checks in order: [1](#0-0) 

1. `validate_state_preconditions` — checks resource bounds and nonce range
2. `validate_by_mempool` — checks for duplicate/stale nonces in the mempool
3. `skip_stateful_validations` — decides whether to skip `__validate__`

**`validate_nonce` (inside step 1):** For an Invoke with `account_nonce=0` and `tx_nonce=1`, the check is `0 ≤ 1 ≤ (0 + max_allowed_nonce_gap)`. The production default is `max_allowed_nonce_gap = 200`, so this **passes**. [2](#0-1) [3](#0-2) 

**`validate_by_mempool` (step 2):** Calls `mempool.validate_tx` with `account_nonce=0, tx_nonce=1`. The mempool's `MempoolState::validate_incoming_tx` resolves the nonce as `staged.get(A).or(committed.get(A)).unwrap_or(0) = 0`. Since `tx_nonce(1) >= 0`, no `NonceTooOld` error. `validate_fee_escalation` finds no existing tx at nonce=1. This **passes**. [4](#0-3) 

**`skip_stateful_validations` (step 3):** The three conditions are:
- `tx` is `ExecutableTransaction::Invoke` ✓
- `tx.nonce() == Nonce(Felt::ONE)` ✓ (nonce=1)
- `account_nonce == Nonce(Felt::ZERO)` ✓ (state nonce=0)

Then it calls `account_tx_in_pool_or_recent_block(A)`: [5](#0-4) 

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [6](#0-5) 

Because the valid nonce=0 invoke is already in `tx_pool`, `tx_pool.contains_account(A)` returns `true`. The function returns `true` (skip_validate = true).

**`run_validate_entry_point` with `skip_validate=true`:** Sets `execution_flags.validate = !skip_validate = false`. The blockifier's `validate_tx` immediately returns `Ok(None)` without calling `__validate__`: [7](#0-6) [8](#0-7) [9](#0-8) 

The invalid-signature transaction is admitted to the mempool without signature verification.

**Root cause — flawed comment and logic:** The code comment says:
> "It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."

This reasoning is wrong. Having a nonce=0 invoke in the pool for an already-deployed account does **not** imply a deploy_account transaction exists. The check `account_tx_in_pool_or_recent_block` is not filtered to deploy_account transactions. [10](#0-9) 

---

### Impact Explanation

An unprivileged attacker can admit a nonce=1 invoke with an **invalid signature** to the mempool, bypassing the gateway's `__validate__` entry point call. The concrete corrupted admission value is: a transaction with an invalid signature is accepted by the gateway and enters the mempool. When the sequencer later executes the transaction, `__validate__` will be called (with `execution_flags.validate=true` by default during execution), the invalid signature will cause a revert, and the fee will be charged. The invalid transaction does not execute successfully, but it is admitted to the mempool — satisfying the **High** impact criterion: "Mempool/gateway/RPC admission accepts invalid transactions before sequencing."

---

### Likelihood Explanation

Any user with a deployed account at state nonce=0 (e.g., a freshly deployed account that has not yet sent any transactions) can trigger this. The attacker needs only:
1. One valid nonce=0 invoke in the mempool (trivially achievable)
2. Submit a nonce=1 invoke with an arbitrary/invalid signature

No privileged access is required. The attack is deterministic and requires no race condition.

---

### Recommendation

`skip_stateful_validations` must verify that the transaction in the mempool for the account is specifically a **deploy_account** transaction, not just any transaction. The `account_tx_in_pool_or_recent_block` API should be replaced with a more specific check, e.g., `deploy_account_tx_in_pool_or_recent_block`, that only returns `true` if the account has a deploy_account transaction in the pool or a recent block. Alternatively, the condition should additionally verify that the account does **not** exist in state (i.e., `get_class_hash_at(address) == ClassHash::default()`), which would confirm the account is genuinely not yet deployed and the skip is warranted.

---

### Proof of Concept

**Concrete scenario:**
1. Account `A` is deployed (exists in state), state nonce = 0.
2. Attacker submits `invoke_v3(sender=A, nonce=0, signature=VALID)` → passes all checks → enters mempool. Now `tx_pool.contains_account(A) = true`.
3. Attacker submits `invoke_v3(sender=A, nonce=1, signature=GARBAGE)`.

**Trace through gateway for step 3:**
- `account_nonce = 0` (from state)
- `validate_nonce`: `0 ≤ 1 ≤ 200` → passes
- `validate_by_mempool`: `resolve_nonce(A, 0) = 0`, `1 >= 0` → passes
- `skip_stateful_validations`: nonce=1 ✓, account_nonce=0 ✓, `tx_pool.contains_account(A)=true` ✓ → returns `true`
- `run_validate_entry_point(skip_validate=true)`: `execution_flags.validate=false` → `__validate__` not called
- Transaction admitted to mempool with invalid signature.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
