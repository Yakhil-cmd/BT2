### Title
Unvalidated `disable_rewards` Parameter in Public `update_rewards` Enables Global Reward-Distribution Griefing — (File: `src/staking/staking.cairo`)

---

### Summary

The `update_rewards` function is publicly callable by any non-zero address. Its `disable_rewards` boolean parameter is entirely user-controlled and carries no access restriction. The global `last_reward_block` checkpoint is written **before** the `disable_rewards` branch is evaluated, so any caller can advance the checkpoint without distributing a single token of rewards. Because the checkpoint is global (not per-staker), one attacker transaction per block is sufficient to deny block rewards to every staker in the protocol.

---

### Finding Description

`update_rewards` is exposed through `IStakingRewardsManager` with no caller restriction beyond the generic `general_prerequisites()` guard (not paused, caller ≠ zero):

```

The storage field `last_reward_block` is a single protocol-wide value: [2](#0-1) 

Because the write at line 1485 occurs before the `disable_rewards` branch at line 1487, any caller who passes `disable_rewards: true` consumes the per-block "slot" for the entire protocol without paying out rewards.

The only prerequisite the attacker must satisfy is supplying a `staker_address` that is currently active with a non-zero STRK balance. All such addresses are publicly observable from on-chain `NewStaker` events. [3](#0-2) 

---

### Impact Explanation

**Medium — Griefing with no profit motive but damage to users or protocol.**

By calling `update_rewards(any_valid_staker, disable_rewards: true)` once per block, an attacker:

1. Advances `last_reward_block` to the current block.
2. Distributes zero rewards.
3. Causes every subsequent legitimate call to `update_rewards` for that block to revert with `REWARDS_ALREADY_UPDATED`.

All stakers are denied block rewards for every block the attacker griefs. The attacker gains nothing; the protocol and all stakers lose accrued yield proportional to the duration of the attack.

---

### Likelihood Explanation

**Medium.** The attack requires one Starknet transaction per block (~3-second block time). Starknet transaction fees are low, making sustained griefing economically feasible for a motivated adversary. No privileged access, leaked key, or external dependency is required — only a valid staker address (trivially obtained from public events) and a non-zero caller address.

---

### Recommendation

1. **Restrict `disable_rewards: true` to the staker themselves** (or an authorized address), so that only the staker whose rewards are being skipped can suppress distribution.
2. Alternatively, **only advance `last_reward_block` when rewards are actually distributed** (move the write below the `disable_rewards` branch), so that a no-op call does not consume the block slot.
3. Consider making `disable_rewards` an internal-only flag rather than a public parameter.

---

### Proof of Concept

```
// Attacker (any non-zero EOA or contract) executes once per block:
staking_contract.update_rewards(
    staker_address: <any_active_staker>,   // obtained from NewStaker events
    disable_rewards: true,
);
// Effect:
//   last_reward_block = current_block   (global checkpoint consumed)
//   rewards distributed = 0
//
// Any legitimate call in the same block:
staking_contract.update_rewards(valid_staker, false);
// → panics: REWARDS_ALREADY_UPDATED
//
// Repeat every block → all stakers receive zero block rewards indefinitely.

        fn update_rewards(
            ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
        ) {
            self.general_prerequisites();
            let current_block_number = starknet::get_block_number();
            assert!(
                current_block_number > self.last_reward_block.read(),
                "{}",
                Error::REWARDS_ALREADY_UPDATED,
            );

            // Assert staker exists and active.
            // Staker is considered to exist from the moment of `stake` (when `InternalStakerInfo`
            // struct is created) until the calling to `unstake_action` (when `InternalStakerInfo`
            // struct is deleted).
            // Staker remains active until the intent period begins, i.e. K epochs after
            // `unstake_intent` is called.
            let staker_info = self.internal_staker_info(:staker_address);
            let curr_epoch = self.get_current_epoch();
            assert!(
                self.is_staker_active(:staker_address, epoch_id: curr_epoch),
                "{}",
                Error::INVALID_STAKER,
            );

            let staker_pool_info = self.staker_pool_info.entry(staker_address).as_non_mut();
            let (staker_total_strk_balance, staker_total_btc_balance) = self
                .get_staker_total_strk_btc_balance_at_epoch(
                    :staker_address, :staker_pool_info, epoch_id: curr_epoch,
                );
            // Assert staker has non-zero balance.
            // Staker exists with zero balance for the first K epochs after `stake`, then the stake
            // becomes effective.
            assert!(staker_total_strk_balance.is_non_zero(), "{}", Error::INVALID_STAKER);

            // Update last block rewards.
            self.last_reward_block.write(current_block_number);

            if disable_rewards || self.is_pre_consensus() {
                return;
            }

            // Get current block data and update rewards.
            let reward_supplier_dispatcher = self.reward_supplier_dispatcher.read();
            let (strk_block_rewards, btc_block_rewards) = self
                .calculate_block_rewards(:reward_supplier_dispatcher, :curr_epoch);
            self
                ._update_rewards(
                    :staker_address,
                    strk_total_rewards: strk_block_rewards,
                    btc_total_rewards: btc_block_rewards,
                    strk_total_stake: staker_total_strk_balance,
                    btc_total_stake: staker_total_btc_balance,
                    :staker_info,
                    :staker_pool_info,
                    :reward_supplier_dispatcher,
                    :curr_epoch,
                );
        }
