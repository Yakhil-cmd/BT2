### Title
`protocolUnpausePool` Sets Wrong Pause Level, Leaving Pool Permanently Swap-Blocked After Protocol Unpause - (File: metric-core/contracts/MetricOmmPoolFactory.sol)

### Summary

`protocolUnpausePool` calls `setPause(1)` instead of `setPause(0)`, so after a protocol pause/unpause cycle the pool is left in admin-paused state (level 1) rather than active state (level 0). Because the `swap` function is gated by `whenNotPaused` which rejects any `pauseLevel != 0`, swaps remain permanently blocked until the pool admin separately calls `unpausePool`.

### Finding Description

The three pause levels are documented in `MetricOmmPool.sol`:

```
/// @dev 0 = active, 1 = paused by admin, 2 = paused by protocol.
```

`protocolPausePool` correctly transitions from 0 or 1 → 2. However, `protocolUnpausePool` transitions 2 → 1 (admin-paused), not 2 → 0 (active):

```solidity
// MetricOmmPoolFactory.sol L399-L403
function protocolUnpausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 2) revert InvalidPauseTransition(cur, 1);
    IMetricOmmPoolFactoryActions(pool).setPause(1);   // ← should be setPause(0)
}
``` [1](#0-0) 

The `whenNotPaused` modifier in `MetricOmmPool.sol` rejects any non-zero pause level:

```solidity
function _checkNotPaused() internal view {
    if (pauseLevel != 0) revert PoolPaused();
}
``` [2](#0-1) 

`swap` carries `whenNotPaused`:

```solidity
function swap(...) external whenNotPaused nonReentrant(PoolActions.SWAP) ...
``` [3](#0-2) 

So after the protocol calls `protocolUnpausePool`, the pool is at level 1 and every `swap` call reverts with `PoolPaused`. The pool admin must then call `unpausePool` (which transitions 1 → 0) to restore swap functionality. [4](#0-3) 

For contrast, `unpausePool` (admin path) correctly targets level 0:

```solidity
function unpausePool(address pool) external override nonReentrant onlyPoolAdmin(pool) {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 1) revert InvalidPauseTransition(cur, 0);
    IMetricOmmPoolFactoryActions(pool).setPause(0);   // ← correct
}
``` [4](#0-3) 

### Impact Explanation

After a protocol pause/unpause cycle on a pool that was previously active (level 0), the pool is left at level 1. All `swap` calls revert. LPs cannot trade, arbitrageurs cannot rebalance, and the pool is economically dead until the admin takes a separate action. If the pool admin is unavailable, unaware, or the admin key is lost, the pool is permanently swap-blocked. This constitutes broken core pool functionality making the swap flow unusable.

### Likelihood Explanation

The trigger is the factory owner (protocol) calling `protocolUnpausePool` — a normal operational action expected during incident response. It fires every single time the protocol pauses and then unpauses any pool. No special conditions, no attacker required; the bug manifests deterministically on every protocol unpause.

### Recommendation

Change `setPause(1)` to `setPause(0)` in `protocolUnpausePool`:

```solidity
function protocolUnpausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
-   if (cur != 2) revert InvalidPauseTransition(cur, 1);
-   IMetricOmmPoolFactoryActions(pool).setPause(1);
+   if (cur != 2) revert InvalidPauseTransition(cur, 0);
+   IMetricOmmPoolFactoryActions(pool).setPause(0);
}
``` [1](#0-0) 

### Proof of Concept

1. Pool is deployed and active (`pauseLevel == 0`).
2. Protocol owner calls `protocolPausePool(pool)` → pool transitions to level 2.
3. Protocol owner calls `protocolUnpausePool(pool)` → pool transitions to level **1** (not 0).
4. Any user calls `pool.swap(...)` → `_checkNotPaused()` fires → `PoolPaused` revert.
5. Pool admin must call `unpausePool(pool)` to reach level 0 and restore swaps.

If the pool admin is unavailable, swaps are permanently blocked despite the protocol having "unpaused" the pool.
