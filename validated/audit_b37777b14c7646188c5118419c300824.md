### Title
Caller Nonce Permanently Incremented on CREATE Collision Before Rollback Frame Exists — (`evm_interpreter/src/ee_trait_impl.rs`)

### Summary

In `before_executing_frame`, the caller's nonce is incremented **before** the collision check for `Constructor` (CREATE/CREATE2) calls. When the collision check subsequently fails, the function returns `Ok(false)` (a silent non-revert failure), but the nonce increment is never rolled back. Because `before_executing_frame` is called before `start_global_frame()` in the bootloader runner, no rollback handle exists for this increment. Standard EVM semantics require that a failed CREATE due to collision reverts all state changes, including the caller's nonce increment.

---

### Finding Description

In `evm_interpreter/src/ee_trait_impl.rs`, `before_executing_frame` executes the following sequence for `Constructor` calls at depth > 0:

1. **Depth/balance check** — if it fails, return `Ok(false)` (no nonce change).
2. **Increment caller's nonce** via `system.io.increment_nonce(... &caller, 1u64)`.
3. **Collision check** — if `deployee_code_len != 0 || deployee_nonce != 0`, burn all gas and return `Ok(false)`. [1](#0-0) 

The nonce increment at step 2 is a real write to the account cache. When step 3 detects a collision and returns `Ok(false)`, the nonce write is **not** undone.

In `basic_bootloader/src/bootloader/runner.rs`, `before_executing_frame` is called **before** `start_global_frame()`:

```
// Pre-checks and operations that should not be rolled back if call fails
SupportedEEVMState::before_executing_frame(...)  // nonce incremented here
...
// Create snapshot for rollbacks
let rollback_handle = self.system.start_global_frame()?;  // too late
``` [2](#0-1) 

When `before_executing_frame` returns `Ok(false)`, the runner returns immediately — `start_global_frame` is never called, so no rollback handle is ever created for the nonce write. The caller's nonce is permanently incremented within the current outer frame.

The EVM Yellow Paper (EIP-161 + EIP-684) specifies that if CREATE fails due to address collision, **all** state changes are reverted, including the caller's nonce increment. ZKsync OS violates this invariant.

---

### Impact Explanation

**EVM semantic mismatch / state-transition bug:**

1. A contract factory executing `CREATE` against a colliding address has its nonce permanently incremented even though the deployment failed.
2. All subsequent `CREATE` calls from that factory compute deployment addresses using the wrong (inflated) nonce, causing them to deploy to different addresses than expected.
3. Any protocol relying on deterministic `CREATE` address prediction (e.g., counterfactual deployments, factory patterns) is broken silently — the outer transaction succeeds, the factory's nonce is wrong, and no error is surfaced.
4. This is a forward/proving-consistent divergence from Ethereum EVM semantics, meaning ZKsync OS blocks that include such transactions would produce state roots that differ from what a correct EVM implementation would produce.

---

### Likelihood Explanation

A collision is reachable by any unprivileged caller:

- Deploy a contract at address `A`.
- From another contract factory, execute `CREATE` with a nonce that computes to address `A`.
- The collision check fires, the CREATE fails, but the factory's nonce is incremented.

This is a realistic scenario for contract factories, proxy deployers, or any protocol that pre-computes deployment addresses. No privileged access, oracle manipulation, or governance action is required.

---

### Recommendation

Move the collision check **before** the caller nonce increment inside `before_executing_frame`, or move the caller nonce increment **inside** the rollback frame (i.e., into `start_executing_frame` after the collision check has already passed). The deployee nonce increment (EIP-161 nonce = 1) is already correctly placed inside `start_executing_frame` where it is subject to rollback. [3](#0-2) 

Concretely, the fix is to reorder the checks in `before_executing_frame`:

```rust
// CORRECT ORDER:
// 1. Check depth and balance
// 2. Check collision (deployee_code_len != 0 || deployee_nonce != 0)
// 3. Only then increment caller's nonce
```

---

### Proof of Concept

1. Deploy contract `Factory` at address `F` (nonce = 1 after deployment).
2. Deploy contract `Victim` at address `V = keccak(F, 1)[12:]` (the address CREATE would compute for `F`'s next deployment).
3. Call `Factory.deploy()` which executes `CREATE` — this computes address `V`, finds it occupied, fires the collision path in `before_executing_frame`, and returns `Ok(false)`.
4. Observe: `Factory`'s nonce is now 2 (incremented), even though the CREATE failed.
5. Call `Factory.deploy()` again — it now deploys to `keccak(F, 2)[12:]` instead of `keccak(F, 1)[12:]`.
6. Any off-chain system that pre-computed the factory's next deployment address using nonce 1 now points to the wrong address.

The outer transaction in steps 3 and 5 succeeds with no error indication, matching the "silent failure" pattern from the reference report. [4](#0-3) [5](#0-4) [2](#0-1)
