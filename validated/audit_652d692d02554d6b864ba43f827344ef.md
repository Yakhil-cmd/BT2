### Title
Missing Access Control on `update_rewards` Allows Any Caller to Permanently Deny Block Rewards - (`src/staking/staking.cairo`)

---

### Summary

The `update_rewards` function in the `Staking` contract is specified to be callable only by the Starkware sequencer, but the implementation contains no on-chain caller check. Any unprivileged address can call `update_rewards(staker_address, disable_rewards: true)` once per block to consume the global `last_reward_block` slot without distributing any rewards. Because `last_reward_block` is a single global storage variable, this permanently prevents the legitimate sequencer from distributing block rewards for that block to any staker.

---

### Finding Description

The spec at `docs/spec.md:1644â€“1645` states:

> **access control**: Only starkware sequencer.

However, the implementation at `src/staking/staking.cairo:1449â€“1507` contains no caller assertion:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();                          // only checks pause flag
    let current_block_number = starknet::get_block_number();
    assert!(
        current_block_number > self.last_reward_block.read(),
        "{}",
        Error::REWARDS_ALREADY_UPDATED,
    );
    // ...
    self.last_reward_block.write(current_block_number);   // global slot consumed

    if disable_rewards || self.is_pre_consensus() {
        return;                                            // exits with no rewards distributed
    }
    // ... distribute rewards
}
```

`last_reward_block` is declared as a single global `BlockNumber` field in storage (`src/staking/staking.cairo:187`), not a per-staker map. The guard `current_block_number > self.last_reward_block.read()` means only one call per block can succeed. An attacker who calls first with `disable_rewards: true` writes the current block number into `last_reward_block` and returns immediately. Any subsequent call in the same block â€” including the legitimate sequencer call â€” reverts with `REWARDS_ALREADY_UPDATED`.

The attack is repeatable every block at negligible cost (only Starknet gas fees).

---

### Impact Explanation

**High â€” Permanent freezing / theft of unclaimed yield.**

Block rewards that are not distributed in a given block are permanently lost; there is no catch-up mechanism. The `test_update_rewards_miss_blocks` test (`src/staking/tests/test.cairo:3530â€“3561`) confirms that missed blocks do accumulate a multiplier, but only when the sequencer itself skips blocks â€” not when an attacker pre-empts the slot. Once the attacker writes `last_reward_block = N`, the sequencer's call for block `N` is rejected, and those rewards are never credited to any staker or pool member. Sustained griefing across many blocks causes permanent, irrecoverable loss of yield for all stakers and their delegators.

---

### Likelihood Explanation

**High.** The function is publicly callable with no role check. On Starknet, transaction fees are low, making it economically viable to submit one griefing transaction per block indefinitely. The attacker needs only to know any valid, active staker address (readable from public events or `get_stakers`). No stake, no delegation, and no privileged role is required.

---

### Recommendation

Add a sequencer-role check at the top of `update_rewards`, analogous to how other privileged functions use `self.roles.only_*()` guards. For example:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.roles.only_rewards_manager(); // or equivalent sequencer role
    self.general_prerequisites();
    // ...
}
```

Alternatively, restrict the function to be callable only by the attestation contract or a dedicated sequencer address registered at construction time.

---

### Proof of Concept

1. Consensus rewards are active (`is_pre_consensus()` returns false).
2. Attacker (any address, zero stake) calls:
   ```
   update_rewards(staker_address: <any_valid_staker>, disable_rewards: true)
   ```
   at the start of block `N`.
3. `last_reward_block` is written to `N`; no rewards are distributed.
4. The legitimate sequencer calls `update_rewards(<staker>, disable_rewards: false)` in the same block `N`.
5. The assertion `current_block_number (N) > last_reward_block (N)` fails â†’ `REWARDS_ALREADY_UPDATED` panic.
6. All stakers lose their block-`N` rewards permanently.
7. Repeat every block.

The `update_rewards_disable_rewards_consensus_rewards_flow_test` flow test at `src/flow_test/test.cairo:2806â€“2910` already demonstrates that calling `update_rewards` twice in the same block always reverts with `REWARDS_ALREADY_UPDATED`, confirming the one-slot-per-block invariant that the attacker exploits.
