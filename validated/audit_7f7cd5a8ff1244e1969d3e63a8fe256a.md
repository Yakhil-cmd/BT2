### Title
WalletConnect `chia_cancelOffer` Approval Dialog Omits Offer Asset Details, Enabling Blind On-Chain Offer Cancellation — (`packages/gui/src/electron/commands/Commands.ts`, `packages/gui/src/electron/commands/parseCommandDisplay.ts`)

### Summary

A paired WalletConnect dApp can invoke `chia_cancelOffer` with an attacker-chosen `trade_id`. The approval dialog shown to the user displays only the raw opaque hex `trade_id` and fee — no offer assets, amounts, or counterparty details are resolved or shown. The user cannot make an informed decision, and approving triggers an on-chain cancellation (`secure=true`) of the targeted offer.

### Finding Description

The `chia_cancelOffer` dApp command is defined in `Commands.ts` with a dApp-facing `params` list that shows only `trade_id` (raw string) and `fee`, hiding `secure` (which is hardcoded to `true` via `defaults`): [1](#0-0) 

The approval dialog content is enriched by `parseCommandDisplay`, which resolves offer data into human-readable asset/amount deltas for `chia_wallet.take_offer` and `chia_wallet.create_offer_for_ids`. However, `chia_wallet.cancel_offer` is **not handled** — the function falls through to `return undefined`: [2](#0-1) 

In `main.tsx`, `display` is set from `parseCommandDisplay(commandId, parsedParams)` and passed to the `Confirm` dialog: [3](#0-2) 

In `Confirm.tsx`, `walletDelta` is only rendered when `display` is non-null. For `cancel_offer`, `display` is `undefined`, so only the raw `rows` are shown: [4](#0-3) 

The result: the user sees only `Trade Id: <64-char hex>` and `Fee: 0 XCH` — no indication of which assets or amounts are locked in the offer being cancelled.

### Impact Explanation

A malicious paired dApp can:
1. Call `chia_getAllOffers` (which has `allowConfirmationBypass: true`, so it can be silently enumerated after initial bypass grant) to list all open offers and their `trade_id`s.
2. Select a high-value offer's `trade_id`.
3. Send `chia_cancelOffer({ trade_id: victimTradeId, fee: 0 })`.
4. The user sees only an opaque hex string in the approval dialog and may approve, believing it is a routine or expected cancellation.
5. The offer is cancelled on-chain with `secure=true`.

The locked coins are returned to the user's wallet (no direct theft), but the open offer — which may represent a favorable trade the user was waiting to complete — is irreversibly destroyed. This fits the High scope: a user is caused to approve cancellation of the wrong offer because the dialog spoofs/omits the offer's asset identity and amounts.

### Likelihood Explanation

Any dApp that has been paired with the wallet can trigger this. The `chia_getAllOffers` bypass makes silent enumeration feasible after initial pairing. The attack is concrete, local-testable, and requires no leaked keys or host compromise — only a paired (potentially malicious) dApp.

### Recommendation

Add a `chia_wallet.cancel_offer` branch to `parseCommandDisplay` that:
1. Calls `get_offer` (or equivalent) using the `trade_id` to fetch the offer record.
2. Resolves the offer's `summary` into a `walletDelta` (offered/requested assets and amounts) using the existing `offerSummaryToWalletDelta` / `walletDeltaToDisplay` pipeline.
3. Returns the populated `display` object so `Confirm.tsx` renders the full asset breakdown before the user can confirm.

This mirrors the existing treatment of `take_offer` and `create_offer_for_ids`.

### Proof of Concept

1. Pair a test dApp with a wallet that has an open XCH↔CAT offer.
2. From the dApp, call `chia_getAllOffers` to retrieve the `trade_id` of the open offer.
3. Call `chia_cancelOffer({ trade_id: <retrieved_id>, fee: 0 })`.
4. Observe the approval dialog: it shows only `Trade Id: <hex>` and `Fee: 0 XCH` — no asset names or amounts.
5. Approve. Confirm via `get_offer` that the offer status is now `CANCELLED` and an on-chain spend was submitted.

export async function parseCommandDisplay(command: string, params: Record<string, unknown>) {
  if (command === 'chia_wallet.take_offer') {
    if (!params.offer || typeof params.offer !== 'string') {
      throw new Error('Offer is not valid');
    }

    const offerSummary = await getOfferSummary(params.offer);
    if (!offerSummary || !offerSummary.summary || !offerSummary.success) {
      throw new Error('Offer is not valid');
    }

    const { summary } = offerSummary;

    const walletDelta = offerSummaryToWalletDelta(summary);
    const walletInfos = await getWalletInfos();
    const assetKinds = offerSummaryAssetKinds(summary);
    const royaltyPercentages = offerSummaryRoyaltyPercentages(summary);
    const fees = parseMojos(summary.fees);

    return {
      walletDelta: await walletDeltaToDisplay(walletDelta, walletInfos, assetKinds, royaltyPercentages, fees),
    };
  }

  if (command === 'chia_wallet.create_offer_for_ids') {
    if (!params.offer || !isPlainObject(params.offer)) {
      throw new Error('Offer is not valid');
    }

    if (params.driver_dict !== undefined && !isPlainObject(params.driver_dict)) {
      throw new Error('Driver Dict is not valid');
    }

    const walletDelta = createOfferToWalletDelta(params.offer);
    const walletInfos = await getWalletInfos();
    const driverDict = params.driver_dict ?? {};
    const assetKinds = createOfferAssetKinds(walletDelta, walletInfos, driverDict);
    const royaltyPercentages = createOfferRoyaltyPercentages(walletDelta, driverDict);

    return {
      walletDelta: await walletDeltaToDisplay(walletDelta, walletInfos, assetKinds, royaltyPercentages, undefined),
    };
  }

  return undefined;
}
