### Title
Missing Threshold Enforcement in `ckd()` Allows CKD Output Corruption via Sub-Threshold Participant Set — (File: `src/confidential_key_derivation/protocol.rs`)

### Summary
The `ckd()` function does not validate that the number of participants meets the cryptographic threshold established during DKG. Unlike `keygen()`, which enforces `participants.len() >= threshold` via `assert_key_invariants()`, `ckd()` only checks `participants.len() >= 2`. A malicious library caller or coordinator can invoke `ckd()` with fewer participants than the DKG threshold, causing `compute_signature_share()` to compute Lagrange coefficients over the wrong participant set, producing a corrupted CKD output `(Y, C)` that the app cannot use.

### Finding Description

**Root cause — missing threshold lower-bound check in `ckd()`:**

`keygen()` (via `assert_key_invariants()`) enforces both bounds on the threshold:

```rust
// src/dkg.rs
if threshold > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
if threshold < 2 {
    return Err(InitializationError::ThresholdTooSmall { ... });
}
```

`ckd()` enforces only a minimum participant count of 2 and membership checks, with no check against the DKG threshold:

```rust
// src/confidential_key_derivation/protocol.rs
if participants.len() < 2 {
    return Err(InitializationError::NotEnoughParticipants { ... });
}
// No check: participants.len() >= threshold
```

The `KeygenOutput` struct (passed to `ckd()`) does not store the threshold, so `ckd()` has no way to enforce it without a caller-supplied threshold parameter.

**Exploit path:**

Inside `compute_signature_share()`, each participant computes:

```rust
let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
let norm_big_y = big_y * lambda_i;
let norm_big_c = big_c * lambda_i;
```

The Lagrange coefficient `λ_i` is computed over the caller-supplied `participants` set. If that set has `k < threshold` members, the coefficients are computed over the wrong domain. The coordinator's aggregation:

```
C = Σ λ_i(k-set) · C_i  ≠  msk · H(pk, app_id) + a · Y
```

because Lagrange interpolation over `k < threshold` points does not reconstruct the secret polynomial's constant term. The resulting `(Y, C)` is cryptographically incorrect.

**Concrete attack scenario (n=3, threshold=2):**

1. DKG runs with 3 participants and threshold=2; each party holds a valid key share.
2. An app requests a CKD key.
3. A malicious library caller (or coordinator) invokes `ckd()` with only 1 participant in the `participants` list.
4. The honest participant computes `λ_1 = 1` (Lagrange coefficient over a 1-element set) and sends `(Y_1, C_1)` to the coordinator.
5. The coordinator aggregates: `C = C_1 = x_1 · H(pk, app_id) + y_1 · A`.
6. The app computes `sig = C + (−a)·Y = x_1 · H(pk, app_id)`.
7. BLS verification fails: `e(x_1·H, G2) ≠ e(H, msk·G2)` because `x_1 ≠ msk`.
8. The app cannot derive its confidential key; CKD is denied.

### Impact Explanation

The CKD output `(Y, C)` is cryptographically corrupted. Honest parties (the requesting app) receive an unusable cryptographic output — the BLS signature verification fails and the derived key cannot be computed. This maps directly to the allowed High impact: **Corruption of CKD outputs so honest parties accept inconsistent or unusable cryptographic outputs**.

Additionally, a malicious coordinator can selectively corrupt CKD for any app by controlling which participants are invited, causing targeted denial of CKD for specific `app_id` values.

### Likelihood Explanation

The `participants` slice is fully caller-controlled. Any library integrator or malicious coordinator who controls protocol orchestration can pass a sub-threshold participant list to `ckd()`. No privileged access, leaked keys, or cryptographic breaks are required. The library provides no defense against this at the API boundary.

### Recommendation

Add an explicit `threshold` parameter to `ckd()` and validate it before proceeding:

```rust
pub fn ckd(
    participants: &[Participant],
    coordinator: Participant,
    me: Participant,
    threshold: impl Into<ReconstructionLowerBound>,  // add this
    key_pair: KeygenOutput,
    app_id: impl Into<AppId>,
    app_pk: PublicKey,
    rng: impl CryptoRngCore + Send + 'static,
) -> Result<impl Protocol<Output = CKDOutputOption>, InitializationError> {
    let threshold = usize::from(threshold.into());
    if participants.len() < threshold {
        return Err(InitializationError::NotEnoughParticipantsForThreshold {
            threshold,
            participants: participants.len(),
        });
    }
    if threshold < 2 {
        return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
    }
    // ... existing checks ...
}
```

Alternatively, store the threshold inside `KeygenOutput` so `ckd()` can enforce it without requiring the caller to re-supply it.

### Proof of Concept

**Entry point:** `src/confidential_key_derivation/protocol.rs`, `ckd()` (line 66).

**Missing guard:** No check equivalent to `participants.len() >= threshold` exists anywhere in `ckd()`.

**Trigger:** Call `ckd(&[single_participant], coordinator, me, key_pair, app_id, app_pk, rng)` where the DKG was run with threshold=2 and 3 participants. The call succeeds initialization. `compute_signature_share()` at line 148 computes `lambda_i` over a 1-element set (returning scalar 1), producing `norm_big_c = x_1 · H(pk, app_id) + y_1 · A`. The coordinator aggregates this single contribution and returns it as the CKD output. The app's BLS verification fails, confirming the output is corrupted.
