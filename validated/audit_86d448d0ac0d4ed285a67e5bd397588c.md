### Title
Gateway L2 Gas Price Threshold Uses Wrong Block Field (`l2_gas_price` Instead of `next_l2_gas_price`) — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

`StatefulTransactionValidator::validate_resource_bounds` reads the L2 gas price threshold from `block_info.gas_prices.strk_gas_prices.l2_gas_price` (the price that was used *inside* the previous block). The correct reference price is `block_header_without_hash.next_l2_gas_price` (the EIP-1559-derived price that will be enforced in the *next* block). The developer acknowledged this in a TODO comment on the same line. Because the two values diverge whenever gas usage deviates from the target, the gateway either rejects valid transactions or admits transactions that the blockifier will later reject.

---

### Finding Description

`GatewayFixedBlockSyncStateClient::get_block_info_from_sync_client` constructs a `BlockInfo` from the stored block header:

```rust
strk_gas_prices: GasPriceVector {
    l1_gas_price: block_header.l1_gas_price.price_in_fri.try_into()?,
    l1_data_gas_price: block_header.l1_data_gas_price.price_in_fri.try_into()?,
    l2_gas_price: block_header.l2_gas_price.price_in_fri.try_into()?,  // ← price_in_fri of the *current* block
},
``` [1](#0-0) 

The `BlockHeaderWithoutHash` struct carries two distinct L2 gas price fields:

| Field | Meaning |
|---|---|
| `l2_gas_price` | The price that was applied to transactions *in this block* |
| `next_l2_gas_price` | The EIP-1559-computed price that will be applied to transactions *in the next block* | [2](#0-1) 

`validate_resource_bounds` then reads the wrong field and the developer left an explicit TODO acknowledging it:

```rust
// TODO(Arni): getnext_l2_gas_price from the block header.
let previous_block_l2_gas_price = self
    .gateway_fixed_block_state_reader
    .get_block_info()
    .await?
    .gas_prices
    .strk_gas_prices
    .l2_gas_price;   // ← l2_gas_price of the previous block, NOT next_l2_gas_price
``` [3](#0-2) 

This value is passed to `validate_tx_l2_gas_price_within_threshold`, which computes:

```
threshold = (min_gas_price_percentage / 100) * previous_block_l2_gas_price
```

and rejects the transaction if `tx.max_price_per_unit < threshold`. [4](#0-3) 

The blockifier's `check_fee_bounds`, however, compares the transaction's `max_price_per_unit` against the gas price of the block being built — which is `next_l2_gas_price` from the previous block header, not `l2_gas_price`. [5](#0-4) 

The `next_l2_gas_price` is computed by `calculate_next_l2_gas_price_for_fin` using the EIP-1559 formula and stored in the block header: [6](#0-5) [7](#0-6) 

---

### Impact Explanation

Two failure modes arise whenever `next_l2_gas_price ≠ l2_gas_price`:

**Case 1 — Gas price falling** (`next_l2_gas_price < l2_gas_price`):  
The gateway threshold is computed from the higher `l2_gas_price`. A transaction whose `max_price_per_unit` satisfies `next_l2_gas_price * pct ≤ tx_price < l2_gas_price * pct` is **rejected by the gateway** even though the blockifier would accept it. Valid user transactions are denied admission.

**Case 2 — Gas price rising** (`next_l2_gas_price > l2_gas_price`):  
The gateway threshold is computed from the lower `l2_gas_price`. A transaction whose `max_price_per_unit` satisfies `l2_gas_price * pct ≤ tx_price < next_l2_gas_price * pct` is **admitted by the gateway** but will be rejected by the blockifier's `check_fee_bounds` during execution. Invalid transactions enter the mempool.

Both outcomes match the **High** impact: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

---

### Likelihood Explanation

The EIP-1559 fee market (`calculate_next_base_gas_price`) adjusts `next_l2_gas_price` every block based on gas consumption relative to the target. Any block that is not exactly at the gas target produces a `next_l2_gas_price` that differs from `l2_gas_price`. This is the normal operating condition of the network, so the discrepancy is triggered on virtually every block. The magnitude grows with sustained high or low utilization.

---

### Recommendation

`GatewayFixedBlockStateReader::get_block_info` (or a new dedicated method) should expose `next_l2_gas_price` from the block header. `validate_resource_bounds` should then use that value as the reference price:

```rust
// In GatewayFixedBlockSyncStateClient::get_block_info_from_sync_client:
// Return next_l2_gas_price alongside BlockInfo, or add a separate accessor.

// In validate_resource_bounds:
let next_block_l2_gas_price = self
    .gateway_fixed_block_state_reader
    .get_next_l2_gas_price()   // new method reading block_header.next_l2_gas_price
    .await?;
self.validate_tx_l2_gas_price_within_threshold(
    executable_tx.resource_bounds(),
    next_block_l2_gas_price,
)?;
```

This mirrors the fix in the external report: query the correct source (`next_l2_gas_price` / AToken) rather than the wrong one (`l2_gas_price` / Pool).

---

### Proof of Concept

1. Observe that `BlockHeaderWithoutHash` stores both `l2_gas_price` (current block price) and `next_l2_gas_price` (next block price). [2](#0-1) 

2. `GatewayFixedBlockSyncStateClient::get_block_info_from_sync_client` maps `block_header.l2_gas_price.price_in_fri` into `strk_gas_prices.l2_gas_price` but **never reads** `block_header.next_l2_gas_price`. [8](#0-7) 

3. `validate_resource_bounds` reads `strk_gas_prices.l2_gas_price` and the developer's own TODO confirms the correct field is `next_l2_gas_price`. [3](#0-2) 

4. The consensus context writes `next_l2_gas_price: self.l2_gas_price` into the block header, where `self.l2_gas_price` is the EIP-1559-updated price for the next block — confirming it is semantically distinct from the block's own `l2_gas_price`. [9](#0-8) 

5. A concrete scenario: suppose block N has `l2_gas_price = 100` and `next_l2_gas_price = 90` (falling due to low utilization). With `min_gas_price_percentage = 100`, the gateway threshold is `100`, so a transaction with `max_price_per_unit = 95` is rejected. But the blockifier would use `90` as the block price and accept the transaction — a valid transaction is incorrectly denied.
