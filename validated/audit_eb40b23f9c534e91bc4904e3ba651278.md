### Title
`setPoolAdminFeeDestination` Silently Redirects All Uncollected Admin Fees to the New Destination — (`File: metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`setPoolAdminFeeDestination` updates `poolAdminFeeDestination[pool]` without first flushing accrued fees to the old destination. Every other fee-parameter setter in the factory (`setPoolAdminFees`, `setPoolProtocolFee`) calls `collectFees` before mutating any routing state. `setPoolAdminFeeDestination` omits this step, so all admin fees that accrued under the previous destination are silently redirected to the new one on the next `collectPoolFees` call.

---

### Finding Description

`setPoolAdminFees` and `setPoolProtocolFee` both follow the same invariant: collect all outstanding fees at the current rates and destination **before** changing any fee-routing state.

`setPoolAdminFees` (lines 417–425):
```solidity
PoolFeeConfig memory c = poolFeeConfig[pool];
IMetricOmmPoolCollectFees(pool).collectFees(
    c.protocolSpreadFeeE6, c.adminSpreadFeeE6,
    c.protocolNotionalFeeE8, c.adminNotionalFeeE8,
    poolAdminFeeDestination[pool]   // ← old destination, correct
);
// then updates poolFeeConfig
```

`setPoolAdminFeeDestination` (lines 438–447):
```solidity
function setPoolAdminFeeDestination(address pool, address newAdminFeeDestination)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (newAdminFeeDestination == address(0)) revert InvalidAdminFeeDestination();
    poolAdminFeeDestination[pool] = newAdminFeeDestination;   // ← no prior collectFees
    emit PoolAdminFeeDestinationUpdated(pool, newAdminFeeDestination);
}
```

`collectPoolFees` (lines 379–389) reads `poolAdminFeeDestination[pool]` at call time:
```solidity
IMetricOmmPoolCollectFees(pool).collectFees(
    c.protocolSpreadFeeE6, c.adminSpreadFeeE6,
    c.protocolNotionalFeeE8, c.adminNotionalFeeE8,
    poolAdminFeeDestination[pool]   // ← now the NEW destination
);
```

Any admin fees that accrued between the last collection and the `setPoolAdminFeeDestination` call are therefore delivered to the new destination, not the old one.

---

### Impact Explanation

All uncollected admin fees owed to the previous `poolAdminFeeDestination` are permanently redirected to the new address. The old destination receives nothing for the period it was the legitimate recipient. This is a direct, quantifiable loss of protocol-level fee revenue for the original fee destination, proportional to the volume traded since the last `collectPoolFees` call.

---

### Likelihood Explanation

The pool admin is the only caller required. No special market conditions are needed. The loss occurs automatically on every `setPoolAdminFeeDestination` call where fees have accrued since the last collection — a routine operational scenario (treasury rotation, multisig migration, etc.). A malicious or careless admin can trigger this at any time.

---

### Recommendation

Mirror the pattern used by `setPoolAdminFees` and `setPoolProtocolFee`: flush accrued fees to the **current** destination before updating `poolAdminFeeDestination`:

```solidity
function setPoolAdminFeeDestination(address pool, address newAdminFeeDestination)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (newAdminFeeDestination == address(0)) revert InvalidAdminFeeDestination();
    // Flush fees to the current destination before rotating
    PoolFeeConfig memory c = poolFeeConfig[pool];
    IMetricOmmPoolCollectFees(pool).collectFees(
        c.protocolSpreadFeeE6, c.adminSpreadFeeE6,
        c.protocolNotionalFeeE8, c.adminNotionalFeeE8,
        poolAdminFeeDestination[pool]
    );
    poolAdminFeeDestination[pool] = newAdminFeeDestination;
    emit PoolAdminFeeDestinationUpdated(pool, newAdminFeeDestination);
}
```

---

### Proof of Concept

1. Pool is created; `poolAdminFeeDestination[pool] = oldDest`.
2. Swaps execute; admin fees accrue inside the pool contract.
3. Pool admin calls `setPoolAdminFeeDestination(pool, newDest)`. No `collectFees` is triggered; `poolAdminFeeDestination[pool]` is now `newDest`.
4. Anyone calls `collectPoolFees(pool)`. The factory reads `poolAdminFeeDestination[pool]` → `newDest` and transfers all accrued admin fees (including those from step 2) to `newDest`.
5. `oldDest` receives zero; all fees accrued during its tenure are lost to it.

The root cause is in `MetricOmmPoolFactory.sol` at lines 438–447. [1](#0-0) 

The missing `collectFees` call should match the pattern established at lines 417–425 in `setPoolAdminFees`. [2](#0-1) 

`collectPoolFees` at lines 379–389 confirms that `poolAdminFeeDestination[pool]` is read at call time, not at accrual time, making the redirect unconditional. [3](#0-2)
