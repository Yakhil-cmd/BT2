### Title
`protocolUnpausePool()` Sets Pause Level to 1 (Admin-Paused) Instead of 0 (Active), Leaving Pool Permanently Paused - (`metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary

`MetricOmmPoolFactory.protocolUnpausePool()` is intended to lift a protocol-level pause (level 2) and restore the pool to active (level 0). Instead, it calls `setPause(1)`, transitioning the pool into the admin-paused state (level 1) rather than the active state. The pool remains paused after the protocol owner invokes the unpause function, and the protocol owner has no further path to directly restore activity.

### Finding Description

The pool's `pauseLevel` has three defined states:

- `0` = active (swaps allowed)
- `1` = paused by admin
- `2` = paused by protocol [1](#0-0) 

`protocolUnpausePool` correctly guards that the current level is 2, but then calls `setPause(1)` instead of `setPause(0)`: [2](#0-1) 

Compare with `unpausePool` (pool-admin path), which correctly calls `setPause(0)`: [3](#0-2) 

After `protocolUnpausePool` executes, the pool is at level 1. The protocol owner cannot call `protocolUnpausePool` again because the guard `if (cur != 2)` now reverts (cur is 1, not 2). [4](#0-3) 

The only recovery path is for the pool admin to call `unpausePool`, which checks `cur != 1` and calls `setPause(0)`. This means the protocol owner's emergency unpause silently hands control back to the pool admin rather than restoring the pool to active. [3](#0-2) 

### Impact Explanation

Every swap is gated by `whenNotPaused`, which reverts whenever `pauseLevel != 0`: [5](#0-4) 

After `protocolUnpausePool` is called, the pool remains paused at level 1. Swaps are blocked, LP withdrawals that depend on swap routing are blocked, and the protocol owner has no direct mechanism to restore the pool to active. The pool is effectively stuck in a paused state until the pool admin independently calls `unpausePool`. This breaks core pool functionality (swap and liquidity flows) and can cause loss of LP opportunity and inability to exit positions.

### Likelihood Explanation

The trigger is a valid, privileged-but-expected action: the protocol owner calling `protocolUnpausePool` after a protocol-level pause. This is a routine operational action with no special preconditions beyond the pool being at level 2. Any protocol-paused pool that the owner attempts to unpause will be silently left in admin-paused state.

### Recommendation

Change `protocolUnpausePool` to call `setPause(0)` instead of `setPause(1)`:

```solidity
function protocolUnpausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 2) revert InvalidPauseTransition(cur, 0);
-   IMetricOmmPoolFactoryActions(pool).setPause(1);
+   IMetricOmmPoolFactoryActions(pool).setPause(0);
}
```

### Proof of Concept

1. Protocol owner calls `protocolPausePool(pool)` → pool enters level 2.
2. Protocol owner calls `protocolUnpausePool(pool)` → guard passes (`cur == 2`), `setPause(1)` is called.
3. Pool is now at level 1 (admin-paused). Swaps revert with `PoolPaused`.
4. Protocol owner calls `protocolUnpausePool(pool)` again → reverts with `InvalidPauseTransition(1, 1)` because `cur != 2`.
5. Pool remains paused; only the pool admin can recover by calling `unpausePool`. [6](#0-5)
