### Title
Malicious participant aborts honest verifier's triple generation by sending a `PolynomialCommitment` with identity as constant coefficient, causing `dlog::verify` to return `Err(PointSerialization)` - (`src/ecdsa/ot_based_ecdsa/triples/generation.rs`)

### Summary
A malicious participant can permanently abort any honest party's triple generation session by sending a `PolynomialCommitmentsMessageMany` whose `big_e_v[i]` (or `big_f_v[i]`) has the group identity as its constant coefficient. The verifier path calls `dlog::verify`, which calls `Statement::encode()`, which calls `Group::serialize` on the identity point. This fails with `Err(ProtocolError::PointSerialization)`, which propagates via `?` and terminates the honest party's session.

### Finding Description

**Root cause 1 — `PolynomialCommitment::new()` accepts identity as constant coefficient.**

`PolynomialCommitment::new()` uses `rposition` to strip only *trailing* identity elements from the back of the coefficient vector. A polynomial like `[identity, G, G, …, G]` (identity at index 0, non-identity at all higher indices) passes construction and deserialization without error. [1](#0-0) 

**Root cause 2 — `dlog::Statement::encode()` fails on identity with `Err(PointSerialization)`.**

`dlog::Statement::encode()` calls `Group::serialize(self.public)`. On secp256k1, serializing the identity point fails. The error arm returns `Err(ProtocolError::PointSerialization)` — not a soft `Ok(false)`. [2](#0-1) 

**Root cause 3 — `dlog::verify` has no identity guard, unlike `dlogeq::verify`.**

`dlogeq::verify` explicitly checks `generator1` for identity before proceeding. `dlog::verify` has no equivalent guard — it calls `statement.encode()?` immediately. [3](#0-2) [4](#0-3) 

**Root cause 4 — The error propagates fatally through `do_generation_many`.**

The call site uses `?` on `dlog::verify(...)`, so `Err(PointSerialization)` aborts the entire async task rather than being caught as a soft protocol failure. [5](#0-4) 

**Root cause 5 — The commitment check does not prevent this.**

The round-1 commitment check at lines 353–364 only verifies that the revealed polynomial matches the hash commitment sent earlier. A malicious prover who committed to `[identity, G, G, …, G]` in round 1 will pass this check when revealing the same polynomial in round 3. [6](#0-5) 

The degree check also passes: `[identity, G, G, …, G]` with `threshold` coefficients has degree `threshold - 1`, exactly what is required. [7](#0-6) 

The code even contains a comment acknowledging the identity-serialization problem for `big_l` (line 345: *"degree is threshold - 2 because the constant element identity is not serializable"*), but no analogous protection exists for `big_e` or `big_f`.

### Impact Explanation
Any single malicious participant can permanently abort every honest party's triple generation session. Since triples are a prerequisite for ECDSA presigning and signing, this constitutes a permanent denial of the signing protocol for all honest parties under normal trust assumptions (one malicious participant among `n`).

### Likelihood Explanation
The attack requires only that the malicious participant be a registered member of the protocol session. No cryptographic assumption needs to be broken. The malicious polynomial is trivially constructable, the commitment is binding so the prover must commit to it in round 1 (which they can do), and the reveal in round 3 passes all existing checks. The attack is deterministic and repeatable.

### Recommendation
Add an explicit identity check in `dlog::verify` (and `dlog::prove_with_nonce`) before calling `statement.encode()`, mirroring the guard already present in `dlogeq::verify`:

```rust
// In dlog::verify and dlog::prove_with_nonce
if statement.public.ct_eq(&C::Group::identity()).into() {
    return Err(ProtocolError::IdentityElement);
}
```

Alternatively, add the check inside `dlog::Statement::encode()` itself so it is enforced in both prover and verifier paths.
