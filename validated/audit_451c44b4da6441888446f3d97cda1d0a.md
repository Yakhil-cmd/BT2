### Title
`native_per_pubdata` Truncates to Zero via Floor Division, Enabling Free Pubdata Publication — (File: `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

In the ZKsync OS ZK transaction validation path, `native_per_pubdata` is computed using floor division (`wrapping_div`). When `pubdata_price < native_price`, the result truncates to zero. This silently disables all native-resource charging for pubdata, allowing any transaction to publish unlimited pubdata at zero native cost — even when the operator has set a non-zero `pubdata_price`.

---

### Finding Description

In `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs` at line 142, `native_per_pubdata` is derived as:

```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
``` [1](#0-0) 

`wrapping_div` is integer floor division. When `pubdata_price < native_price` (e.g., `pubdata_price = 50`, `native_price = 100`), the result is `0`.

By contrast, `native_per_gas` is computed with `div_ceil` to prevent this exact truncation:

```rust
u256_try_to_u64(&gas_price.div_ceil(native_price)).ok_or(TxError::Validation(
    InvalidTransaction::NativeResourcesAreTooExpensive,
))?
``` [2](#0-1) 

The same floor-division bug appears in `api/src/helpers.rs`:

```rust
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
``` [3](#0-2) 

When `native_per_pubdata = 0`, two downstream charging sites are silently bypassed:

**1. Intrinsic pubdata overhead in `create_resources_for_tx`:**
```rust
let intrinsic_pubdata_overhead = native_per_pubdata.saturating_mul(intrinsic_pubdata);
// = 0 * intrinsic_pubdata = 0  →  no native deducted for intrinsic pubdata
``` [4](#0-3) 

**2. Per-execution pubdata charging in `get_resources_to_charge_for_pubdata`:**
```rust
let native = current_pubdata_spent
    .checked_mul(native_per_pubdata)
    .ok_or(out_of_native_resources!())?;
// = current_pubdata_spent * 0 = 0  →  no native charged for any pubdata written
``` [5](#0-4) 

There is no zero-guard for `native_per_pubdata`. The `free_native` flag that triggers unlimited-native mode is tied only to `native_per_gas == 0`, not `native_per_pubdata == 0`:

```rust
let native_limit = if cfg!(feature = "unlimited_native") || free_native {
    u64::MAX - 1
} else {
    native_prepaid_from_gas
};
``` [6](#0-5) 

The `pubdata_price` and `native_price` are block-level operator-set values read from the oracle:

```rust
let pubdata_price = system.get_pubdata_price();
let native_price = system.get_native_price();
``` [7](#0-6) 

The condition `pubdata_price < native_price` is a valid and observable operational state. The existing test suite demonstrates this directly — `test_check_pubdata_encoding_version` uses `pubdata_price = 2`, `native_price = 100`: [8](#0-7) 

In that configuration, `native_per_pubdata = floor(2/100) = 0`, so all pubdata is free.

---

### Impact Explanation

When `pubdata_price < native_price`, any transaction can write unlimited storage slots, emit unlimited logs, and deploy large contracts — all generating pubdata — without paying any native resource cost for it. The sequencer and prover bear the full L1 data-availability cost. This is a resource accounting bug: the operator's intent to charge for pubdata (expressed by setting `pubdata_price > 0`) is silently nullified by integer truncation. An attacker can spam pubdata-heavy transactions at the cost of only EVM gas, with zero native resource expenditure for the pubdata component.

---

### Likelihood Explanation

The condition `pubdata_price < native_price` is reachable whenever L1 calldata/blob costs are low relative to proving costs — a normal market condition. The block metadata is publicly observable (any node can read `pubdata_price` and `native_price` from the oracle before submitting a transaction). An attacker monitoring these values can time pubdata-spam attacks to windows where the condition holds. The existing test suite already exercises this configuration without flagging it as anomalous, confirming it is a routine operational state.

---

### Recommendation

Replace `wrapping_div` with `div_ceil` for `native_per_pubdata`, mirroring the treatment of `native_per_gas`:

```rust
// Before (floor division — truncates to 0 when pubdata_price < native_price):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;

// After (ceiling division — minimum of 1 when pubdata_price > 0):
let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

Apply the same fix in `api/src/helpers.rs` line 427. Additionally, add an explicit guard: if `pubdata_price > 0` and `native_per_pubdata == 0` after division, return a validation error or clamp to 1, to make the operator's intent enforceable.

### Proof of Concept

**Configuration:** `native_price = 100`, `pubdata_price = 50` (operator sets pubdata cheaper than one native unit — a valid state).

**Computation:**
```
native_per_pubdata = floor(50 / 100) = 0
```

**Effect in `get_resources_to_charge_for_pubdata`:**
```
native_charged = pubdata_bytes_written * 0 = 0
```

**Attack:**
1. Attacker observes block metadata: `pubdata_price (50) < native_price (100)`.
2. Attacker submits a transaction calling a contract that writes to 100 new storage slots (each slot write produces ~65 bytes of pubdata → ~6500 bytes total).
3. The transaction pays EVM gas for the `SSTORE` opcodes normally.
4. Post-execution pubdata charging: `6500 * 0 = 0` native resources charged.
5. The sequencer publishes 6500 bytes to L1 DA at its own expense.
6. Attacker repeats across many transactions, bloating L1 DA at zero native cost.

With `div_ceil`, the same configuration would yield `native_per_pubdata = ceil(50/100) = 1`, charging `6500 * 1 = 6500` native units — correctly enforcing the operator's pricing intent. [1](#0-0) [3](#0-2) [5](#0-4)
