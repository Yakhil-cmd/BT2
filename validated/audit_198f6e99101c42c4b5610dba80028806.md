### Title
`_afterTimelock` uint32 Truncation Overflow Lets Pool Admin Bypass Stop-Loss Timelock Instantly — (File: `metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

### Summary

`_afterTimelock` computes `uint32(block.timestamp + timelock)` without overflow protection. When `timelock` is set to any value that pushes the sum past `type(uint32).max`, the result wraps to a value smaller than `block.timestamp`, making every subsequent `_requireElapsed` check pass immediately. The pool admin can exploit this to change drawdown, decay, and watermark parameters with zero delay, defeating the LP-protection guarantee the timelock is designed to enforce.

### Finding Description

`_afterTimelock` is the single function that stamps every pending-parameter proposal with its earliest execution time:

```solidity
// OracleValueStopLossExtension.sol line 297-299
function _afterTimelock(address pool_) private view returns (uint32) {
    return uint32(block.timestamp + oracleStopLossConfig[pool_].timelock);
}
```

`oracleStopLossConfig[pool_].timelock` is a `uint32` (max ≈ 4.29 × 10⁹ s ≈ 136 years). The addition is performed in `uint256` space, then silently truncated to `uint32`. With `block.timestamp ≈ 1.75 × 10⁹` (year 2026), any `timelock` value greater than `type(uint32).max − block.timestamp ≈ 2.54 × 10⁹` (≈ 80 years) causes the sum to exceed `type(uint32).max`, wrapping the result to a value **below** `block.timestamp`.

`_requireElapsed` then checks:

```solidity
// line 301-303
function _requireElapsed(uint32 executeAfter) private view {
    if (block.timestamp < executeAfter) revert OracleStopLossTimelockNotElapsed(...);
}
```

Because `executeAfter` (the wrapped value) is less than `block.timestamp`, the condition is false and the guard passes immediately — the proposal is executable in the same transaction it was submitted.

Neither `initialize` nor `executeOracleStopLossTimelock` validates the `timelock` value:

```solidity
// initialize, line 56-61 — no timelock cap

_validateDrawdown(drawdownE6);
_validateDecay(decayPerSecondE8);
// timelock stored without any range check

// executeOracleStopLossTimelock, line 90-91 — no cap either

oracleStopLossConfig[pool_].timelock = timelock;

### Impact Explanation

Once the admin installs an overflowing timelock, every subsequent `proposeX` / `executeX` pair for drawdown, decay, and watermarks can be called back-to-back in the same block. Concretely:

- Admin sets `drawdownE6 = 0` (strictest): `floorMultiplier = E6 − 0 = E6`, so `_applyWatermark` triggers a breach whenever `metric < hwm`, i.e., on any value drop at all. Every swap that moves the pool even slightly reverts with `OracleStopLossTriggered`, making the pool permanently unusable for swaps while LP principal remains locked in bins.
- Admin sets `decayPerSecondE8 = 0`: watermarks never decay, so a single adverse price move permanently blocks one swap direction with no recovery path.
- Admin can also raise watermarks to artificially high values via `proposeOracleStopLossHighWatermarks` / `executeOracleStopLossHighWatermarks` in the same block, immediately triggering the stop-loss on the next swap.

The stop-loss extension is wired as an `afterSwap` hook; a revert there reverts the entire swap. Withdrawals (`removeLiquidity`) are unaffected, but the pool's swap functionality — its core revenue and utility path — is permanently bricked for all users. [5](#0-4) [6](#0-5) 

### Likelihood Explanation

The pool admin is semi-trusted and controls the `timelock` value with no on-chain cap. The two-step path (set timelock to 0 at init, then immediately rotate it to `type(uint32).max`) requires only two admin transactions and is executable at any time after pool creation. No external conditions, oracle states, or user actions are required.

### Recommendation

Add a maximum timelock cap in both `initialize` and `executeOracleStopLossTimelock`, ensuring `block.timestamp + timelock` never exceeds `type(uint32).max`:

```solidity
uint256 constant MAX_TIMELOCK = 365 days; // or any reasonable bound

function _validateTimelock(uint256 tl) private pure {
    if (tl > MAX_TIMELOCK) revert OracleStopLossTimelockTooLarge(tl);
}
```

Apply `_validateTimelock` alongside `_validateDrawdown` and `_validateDecay` in `initialize`, and before storing the new value in `executeOracleStopLossTimelock`. [7](#0-6) 

### Proof of Concept

```
block.timestamp = 1_750_000_000   (approx. year 2025-2026)
timelock        = type(uint32).max = 4_294_967_295

sum (uint256)   = 1_750_000_000 + 4_294_967_295 = 6_044_967_295
uint32(sum)     = 6_044_967_295 mod 4_294_967_296 = 1_749_999_999

executeAfter    = 1_749_999_999  <  block.timestamp = 1_750_000_000
→ _requireElapsed passes immediately
```

Step-by-step exploit:

1. Pool is created with `OracleValueStopLossExtension` initialized with `timelock = 0`.
2. Admin calls `proposeOracleStopLossTimelock(pool, type(uint32).max)`.
   - `_afterTimelock` returns `uint32(block.timestamp + 0) = uint32(block.timestamp)` (current timelock is 0, so `executeAfter = block.timestamp`).
   - `_requireElapsed` passes immediately.
3. Admin calls `executeOracleStopLossTimelock(pool)` in the same block.
   - `oracleStopLossConfig[pool].timelock = type(uint32).max`.
4. Admin calls `proposeOracleStopLossDrawdown(pool, 0)`.
   - `_afterTimelock` now returns `uint32(block.timestamp + type(uint32).max) = block.timestamp − 1`.
   - `executeAfter = block.timestamp − 1 < block.timestamp`.
5. Admin calls `executeOracleStopLossDrawdown(pool)` in the same block — passes immediately.
6. `drawdownE6 = 0`, `floorMultiplier = E6`. Any swap that causes the slightest value drop triggers `OracleStopLossTriggered` and reverts. Pool swaps are permanently bricked. [8](#0-7) [9](#0-8)
