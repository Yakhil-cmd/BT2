### Title
Virtual `isValidSignature` in COA Contract Enables COA Ownership Verification Bypass — (File: fvm/evm/handler/coa/coa.sol)

---

### Summary

The `COA` contract's `isValidSignature()` function, which implements the ERC-1271 COA ownership proof check, is marked `virtual`. Any EVM contract author can inherit from `COA` and override this function to unconditionally return `ValidERC1271Signature`, bypassing the Cadence-side ownership verification entirely. EVM DApps on Flow EVM that rely on ERC-1271 for authentication or authorization can be deceived by such a malicious inheriting contract.

---

### Finding Description

`fvm/evm/handler/coa/coa.sol` defines the `COA` contract, which is the canonical EVM smart-contract wallet deployed by the Flow protocol for every Cadence-Owned Account. It implements ERC-1271 via `isValidSignature()`:

```solidity
// fvm/evm/handler/coa/coa.sol lines 114-125
function isValidSignature(
    bytes32 _hash,
    bytes memory _sig
) external view virtual returns (bytes4){
    (bool ok, bytes memory data) = cadenceArch.staticcall(
        abi.encodeWithSignature(
            "verifyCOAOwnershipProof(address,bytes32,bytes)",
            address(this), _hash, _sig
        )
    );
    require(ok);
    bool output = abi.decode(data, (bool));
    if (output) {
        return ValidERC1271Signature;
    }
    return InvalidERC1271Signature;
}
```

The security check is the `verifyCOAOwnershipProof` precompile call, which routes into the Cadence layer (`EVM.validateCOAOwnershipProof` in `fvm/evm/stdlib/contract.cdc`) to verify Flow account key signatures and confirm the EVM address matches the stored COA. This is the sole gate that proves a Cadence account controls the COA.

Because `isValidSignature` is `virtual`, any EVM contract author can write:

```solidity
contract MaliciousCOA is COA {
    function isValidSignature(
        bytes32, bytes memory
    ) external view override returns (bytes4) {
        return ValidERC1271Signature; // always "valid"
    }
}
```

`supportsInterface()` is also `virtual override` (line 67), so the malicious contract also correctly advertises ERC-1271 support, making it indistinguishable from a legitimate COA to any caller that relies solely on interface introspection and `isValidSignature`.

The compiled bytecode of the legitimate COA is embedded in the Go binary via `coa_bytes.hex` and deployed by `ContractHandler.deployCOA()` at protocol-allocated addresses. The `virtual` keyword serves no purpose for the protocol's own deployment but opens the door for any EVM contract author to publish a COA-shaped contract that skips the ownership proof entirely.

---

### Impact Explanation

EVM DApps on Flow EVM that use ERC-1271 for authentication or authorization (e.g., DAO voting, token-gating, permit-style approvals, or off-chain signature verification flows) call `isValidSignature` on a presented address. A malicious actor deploys a `MaliciousCOA` contract, presents its address as their "COA wallet," and every `isValidSignature` call returns `ValidERC1271Signature` regardless of the hash or signature supplied. The DApp grants access or executes privileged actions without any valid Cadence-side ownership proof having been verified.

This constitutes **entitlement/capability bypass** and **transaction authorization bypass** in the cross-VM (EVM ↔ Cadence) context: the COA ownership proof mechanism — the only cryptographic link between a Cadence account and its EVM address — is silently nullified for the malicious contract.

---

### Likelihood Explanation

Any unprivileged EVM contract author can deploy the override. The `coa.sol` source is publicly available in the repository. The attack requires no staked node, no admin key, and no compromised quorum. The attacker only needs to deploy a Solidity contract on Flow EVM (a standard user action) and present it to a target DApp. DApps that accept any ERC-1271-compliant address as a signer — a common pattern — are immediately vulnerable. Likelihood is **medium-high** given the low barrier to deployment and the growing ecosystem of EVM DApps on Flow.

---

### Recommendation

Remove the `virtual` keyword from both `isValidSignature` and `supportsInterface` in `fvm/evm/handler/coa/coa.sol`. The COA contract is deployed exclusively by the Flow protocol with fixed bytecode; there is no legitimate use case for inheritance-based overriding of these functions. Making them non-virtual prevents any derived contract from bypassing the COA ownership proof.

```solidity
// Before
function isValidSignature(bytes32 _hash, bytes memory _sig)
    external view virtual returns (bytes4) { ... }

function supportsInterface(bytes4 id)
    external view virtual override returns (bool) { ... }

// After
function isValidSignature(bytes32 _hash, bytes memory _sig)
    external view returns (bytes4) { ... }

function supportsInterface(bytes4 id)
    external view override returns (bool) { ... }
```

### Proof of Concept

1. Compile and deploy the following contract on Flow EVM:

```solidity
// SPDX-License-Identifier: UNLICENSED
pragma solidity >=0.7.0 <0.9.0;

import "./coa.sol"; // the public COA source

contract MaliciousCOA is COA {
    // Override the ownership-proof check to always succeed
    function isValidSignature(
        bytes32,
        bytes memory
    ) external view override returns (bytes4) {
        return ValidERC1271Signature; // 0x1626ba7e
    }
}
```

2. Present the deployed address of `MaliciousCOA` to any EVM DApp that calls `IERC1271(addr).isValidSignature(hash, sig)` to gate access.

3. The DApp receives `0x1626ba7e` for any `(hash, sig)` pair — including ones the attacker never signed — and grants access, with zero Cadence-side ownership proof having been verified.
