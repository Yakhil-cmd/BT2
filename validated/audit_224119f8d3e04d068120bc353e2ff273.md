### Title
Missing Cross-Phase Threshold Relationship Validation Allows Signing Threshold Reduction Below DKG Threshold - (File: `src/ecdsa/ot_based_ecdsa/triples/generation.rs`, `src/ecdsa/ot_based_ecdsa/presign.rs`)

### Summary
The OT-based ECDSA pipeline does not validate that the triple-generation and presign threshold (`t_1`) is at least as large as the DKG threshold (`t`). Because `KeygenOutput` does not store the DKG threshold, neither `validate_triple_inputs` nor `presign` can enforce the required invariant `t_1 >= t`. A malicious coordinator can instruct honest participants to run triple generation and presigning with an arbitrarily low threshold (minimum 2), reducing the effective signing threshold far below what the DKG was configured to require, and enabling a small coalition to produce valid signatures.

### Finding Description

The orchestration specification in `docs/ecdsa/ot_based_ecdsa/orchestration.md` mandates:

```
N_1 >= t_0 >= t
```

where `t` is the DKG threshold, `t_0` is the triple-generation threshold, and `t_1` is the presign threshold. This constraint ensures that the signing threshold never falls below the security threshold established during key generation.

`validate_triple_inputs` in `src/ecdsa/ot_based_ecdsa/triples/generation.rs` only checks: [1](#0-0) 

It enforces `threshold >= 2` and `threshold <= participants.len()`, but **never checks `threshold >= DKG_threshold`**.

The `presign` function in `src/ecdsa/ot_based_ecdsa/presign.rs` checks that `args.threshold` matches the triple thresholds: [2](#0-1) 

But it also **never checks `args.threshold >= DKG_threshold`**.

The root structural cause is that `KeygenOutput` stores only `private_share` and `public_key` — the DKG threshold is not persisted: [3](#0-2) 

Because the DKG threshold is absent from `KeygenOutput`, no downstream phase can recover it to enforce the cross-phase invariant. The same gap exists in the FROST/EdDSA presign path: [4](#0-3) [5](#0-4) 

### Impact Explanation

A malicious coordinator instructs all `N` participants to run triple generation with threshold `t_0 = 2` (the minimum accepted value), even though the DKG was configured with threshold `t = 6` (for example). Participants accept because `validate_triple_inputs` only enforces `t_0 >= 2`. The resulting `TriplePub` records `threshold = 2`. The coordinator then runs presign with `args.threshold = 2` (which passes the triple-consistency check). The presignature shares are now threshold-2 shared. At signing time, only 2 parties are needed to reconstruct the presignature and produce a valid ECDSA signature — far fewer than the 6 parties the DKG was designed to require.

This constitutes **unauthorized creation of a valid threshold signature** with an attacker-chosen reduced quorum, matching the Critical impact tier.

### Likelihood Explanation

The coordinator role is explicitly part of the library's threat model (the signing protocol designates a coordinator). A malicious coordinator who also controls one other participant (or colludes with one) can execute this attack in a single signing session. No cryptographic break is required — only the ability to call the public API with a low threshold value. The attack is straightforward and requires no special knowledge beyond the library's public interface.

### Recommendation

Store the DKG threshold inside `KeygenOutput` (or a wrapper type) so that downstream phases can enforce `presign_threshold >= DKG_threshold`. Concretely:

1. Add a `threshold: ReconstructionLowerBound` field to `KeygenOutput` (or the ECDSA-specific alias), populated during `do_keygen`/`do_keyshare`.
2. In `validate_triple_inputs`, accept the `KeygenOutput` (or its threshold) and assert `threshold_value >= keygen_threshold`.
3. In `presign` (OT-based), assert `args.threshold >= args.keygen_out.threshold`.
4. Apply the same fix to the FROST `presign` in `src/frost/mod.rs`.

### Proof of Concept

```
// DKG: 10 parties, threshold = 6
let keygen_outputs = keygen(&participants_10, me, 6, rng)?;

// Malicious coordinator instructs triple generation with threshold = 2
// validate_triple_inputs only checks: threshold >= 2 && threshold <= N
// → accepted without error
let (triple_share, triple_pub) = generate_triple(&participants_10, me, 2usize, rng)?;
// triple_pub.threshold == 2

// Presign: threshold = 2 matches triple threshold → passes consistency check
// No check that 2 >= DKG threshold (6)
let presign_out = presign(&participants_10, me, PresignArguments {
    triple0: (triple_share_0, triple_pub_0),  // threshold = 2
    triple1: (triple_share_1, triple_pub_1),  // threshold = 2
    keygen_out: keygen_outputs[me],
    threshold: 2usize.into(),
})?;

// Sign: only 2 parties needed, producing a valid signature
// despite DKG requiring 6-of-10
let sig = sign(&participants_2, coordinator, me, presign_out, rerandomized, msg_hash)?;
// → valid ECDSA signature accepted by any standard verifier
```
