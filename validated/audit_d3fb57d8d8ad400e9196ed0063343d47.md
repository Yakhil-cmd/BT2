### Title
Blacklisted Admin Fee Destination Blocks Protocol Fee Collection and Bricks `setPoolProtocolFee` / `setPoolAdminFees` — (File: metric-core/contracts/MetricOmmPool.sol)

---

### Summary

`collectFees` transfers to `adminFeeDestination_` before transferring to `FACTORY`. When the admin fee destination is a USDC/USDT-blacklisted address, the first `safeTransfer` reverts and the entire function reverts atomically — permanently blocking protocol fee collection and making `setPoolProtocolFee` and `setPoolAdminFees` uncallable for that pool until the admin voluntarily rotates the destination.

---

### Finding Description

`collectFees` in `MetricOmmPool.sol` executes four sequential token transfers in a single atomic call:

```
transferToken0(adminFeeDestination_, totalFee0ToAdmin);   // line 417
transferToken1(adminFeeDestination_, totalFee1ToAdmin);   // line 420
transferToken0(FACTORY, totalFee0ToProtocol);             // line 423
transferToken1(FACTORY, totalFee1ToProtocol);             // line 426
``` [1](#0-0) 

If `adminFeeDestination_` is blacklisted by USDC or USDT, `safeTransfer` on line 417 or 420 reverts. Because all four transfers share one atomic transaction, the protocol's own transfers to `FACTORY` (lines 423–426) never execute. The `notionalFeeToken0Scaled` / `notionalFeeToken1Scaled` reset on lines 429–430 also never executes, so the notional fee counters remain non-zero and continue accumulating. [2](#0-1) 

Every caller of `collectFees` is blocked:

**`collectPoolFees`** (permissionless, `MetricOmmPoolFactory.sol` lines 379–389) — any keeper or bot calling this reverts. [3](#0-2) 

**`setPoolProtocolFee`** (owner-only, lines 318–360) — the protocol owner's first action is to call `collectFees` at lines 328–335. If that reverts, the owner cannot update protocol fees for this pool at all. [4](#0-3) 

**`setPoolAdminFees`** (admin-only, lines 408–435) — the admin's first action is to call `collectFees` at lines 418–425. If that reverts, the admin cannot update their own fee rates. [5](#0-4) 

The only escape hatch is `setPoolAdminFeeDestination`, which does **not** call `collectFees` and can rotate the destination to a non-blacklisted address. [6](#0-5) 

However, `setPoolAdminFeeDestination` is gated to `onlyPoolAdmin`. If the admin is unresponsive, a compromised multisig, or deliberately non-cooperative, the protocol owner has **no independent path** to collect protocol fees or update protocol fee rates for that pool.

---

### Impact Explanation

- **Protocol fees stuck**: All spread and notional fees accrued in the pool cannot be transferred to `FACTORY`. The amounts grow with every swap but remain locked.
- **Protocol owner authority blocked**: `setPoolProtocolFee` is the owner's only mechanism to adjust per-pool protocol fees. It is unconditionally gated behind a `collectFees` call. A blacklisted admin destination makes this function permanently uncallable without admin cooperation — an admin-boundary break where the admin's state can neutralize the protocol owner's authority.
- **Admin fee update blocked**: `setPoolAdminFees` is similarly gated, so the admin cannot reduce their own fees either.
- **Severity**: Medium — protocol fees are not permanently destroyed (they remain in the pool and can be recovered once the destination is rotated), but the protocol owner loses the ability to exercise fee governance on the affected pool for an indefinite period.

---

### Likelihood Explanation

- USDC and USDT maintain on-chain blacklists and have historically blacklisted addresses associated with hacks, sanctions, or regulatory orders.
- The `adminFeeDestination` is set at pool creation and can be any address (e.g., a multisig, a DAO treasury, a smart contract). Any of these can be blacklisted post-deployment.
- The condition is non-malicious and externally triggered (USDC/USDT issuer action), making it a realistic operational risk rather than a theoretical attack.
- USDC/USDT are explicitly in scope per the allowed impact gate.

---

### Recommendation

Decouple the admin and protocol fee transfers so that a failure in one does not block the other:

1. **Transfer protocol fees first** (lines 422–427 before lines 416–421). This ensures the protocol always receives its share even if the admin destination is broken.
2. **Wrap the admin transfer in a try/catch or low-level call** and emit an event on failure, allowing the admin share to remain claimable later.
3. **Allow `setPoolProtocolFee` to skip `collectFees`** (or use a separate `collectProtocolFeesOnly` path) so the protocol owner can update fee rates independently of the admin destination state.

---

### Proof of Concept

1. Pool is deployed with USDC as `token0` and `adminFeeDestination = 0xADMIN`.
2. Swaps occur; `notionalFeeToken0Scaled` and spread surplus accumulate.
3. USDC blacklists `0xADMIN` (e.g., regulatory action).
4. Anyone calls `collectPoolFees(pool)` → `collectFees` → `safeTransfer(0xADMIN, ...)` → USDC reverts → entire call reverts. Protocol fees remain in pool.
5. Protocol owner calls `setPoolProtocolFee(pool, newFee, newFee)` → same `collectFees` call at line 328 → same revert. Protocol owner cannot change fees.
6. Admin calls `setPoolAdminFees(pool, newFee, newFee)` → same `collectFees` call at line 418 → same revert. Admin cannot change fees.
7. Only resolution: admin calls `setPoolAdminFeeDestination(pool, nonBlacklistedAddr)`, then retry. If admin is unresponsive, protocol owner has no recourse.
