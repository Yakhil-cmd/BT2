### Title
Malicious Coordinator Can Consume One-Time Presignatures Without Producing a Signature, Permanently Denying Signing for Honest Parties — (File: `src/ecdsa/ot_based_ecdsa/sign.rs`, `src/ecdsa/robust_ecdsa/sign.rs`, `src/confidential_key_derivation/protocol.rs`)

---

### Summary

In both OT-based and Robust ECDSA signing protocols, non-coordinator participants compute their signature share `s_i`, send it **privately and exclusively** to the coordinator, and immediately return `Ok(None)` with no mechanism to verify the coordinator produced or distributed the final signature. Because presignatures are strictly one-time-use, a malicious coordinator can receive all `s_i` shares — permanently consuming the presignatures — and then abort without producing a valid signature. This is directly analogous to the reported admin-burn pattern: a privileged role unilaterally destroys cryptographic state belonging to other participants without their consent and without producing the promised output, breaking the invariant that a consumed presignature corresponds to a produced signature.

---

### Finding Description

**Root cause — OT-based ECDSA (`src/ecdsa/ot_based_ecdsa/sign.rs`):**

In `do_sign_participant`, each non-coordinator participant computes its linearized signature share and sends it privately to the coordinator, then returns `Ok(None)` with no further protocol steps:

```rust
fn do_sign_participant(...) -> Result<SignatureOption, ProtocolError> {
    let s_i = compute_signature_share(participants, me, presignature, msg_hash)?;
    let wait0 = chan.next_waitpoint();
    chan.send_private(wait0, coordinator, &s_i)?;   // share sent, presignature consumed
    Ok(None)                                         // participant is done — no verification
}

**Root cause — CKD (`src/confidential_key_derivation/protocol.rs`):**

The same asymmetric pattern applies to Confidential Key Derivation: each non-coordinator participant sends its `(norm_big_y, norm_big_c)` contribution privately to the coordinator and returns `Ok(None)`: [3](#0-2) 

**Why presignature consumption is irreversible:**

The orchestration documentation explicitly states that presignature outputs are one-time-use and must be destroyed after use:

> "It's **critical** that the output is then destroyed, so that no other group of parties attempts to re-use that output for another phase." [4](#0-3) 

Once a participant sends `s_i` to the coordinator, the participant must treat the presignature as consumed — reusing it would violate the security requirement against nonce reuse. There is no rollback mechanism.

**Exploit flow:**

1. A malicious participant is designated as coordinator for a signing session (no privileged access required — any participant can be coordinator).
2. The malicious coordinator initiates the signing protocol with honest participants.
3. Each honest participant computes `s_i`, sends it privately to the coordinator, and returns `Ok(None)`. Their presignature is now consumed.
4. The malicious coordinator receives all `s_i` shares but aborts — it does not produce or distribute the final signature.
5. Honest participants have no signature and no presignature. They must re-run the expensive offline presigning phase (and for OT-based ECDSA, the triple generation phase) to sign again.
6. The malicious coordinator can repeat this attack on every new batch of presignatures, indefinitely blocking signing.

**Why existing checks do not prevent this:**

The only coordinator-side check is signature validity verification (`sig.verify`), which runs *after* all shares are received. There is no mechanism for non-coordinator participants to:
- Verify the coordinator produced the signature.
- Recover their presignature if the coordinator aborts.
- Detect that the coordinator withheld the output. [5](#0-4) 

The coordinator is explicitly placed in the threat model by the documentation (split-view attack warnings), and the RESEARCHER playbook lists "malicious coordinator" as a valid attacker profile. [6](#0-5) 

---

### Impact Explanation

**High — Permanent denial of signing for honest parties under valid protocol inputs.**

A malicious coordinator can permanently exhaust all pre-generated presignatures without producing any valid signatures. For OT-based ECDSA, this also wastes the upstream Beaver triple generation phase. For CKD, the same attack denies confidential key derivation outputs. Honest parties are left with no signing capability until they complete a new (expensive) offline phase, which the malicious coordinator can immediately attack again. The invariant "a consumed presignature corresponds to a produced signature" is broken — directly analogous to the reported admin-burn breaking the invariant "total supply equals recorded supply."

---

### Likelihood Explanation

**Medium.** The coordinator role requires no privileged access — it is assigned per signing session and any participant in the signing set can be designated coordinator. A participant who is malicious and is ever assigned the coordinator role can execute this attack. In deployments where the coordinator rotates or is chosen by an external scheduler, the attack surface is broad.

---

### Recommendation

1. **Broadcast the final signature to all participants.** After the coordinator computes the signature, it should broadcast it to all signing participants (not just return it locally). Participants should verify the broadcast signature before considering their presignature consumed.

2. **Deferred presignature invalidation.** Participants should not treat their presignature as consumed until they receive and verify the coordinator's broadcast signature. This requires a two-phase commit: participants send `s_i`, coordinator broadcasts the result, participants verify and then mark the presignature as used.

3. **Alternatively, use a threshold broadcast for the signing phase** so that no single coordinator can unilaterally abort after collecting all shares.

---

### Proof of Concept

```
Setup:
  - 5 participants, threshold t=2 (robust ECDSA: N = 2t+1 = 5)
  - Run DKG → keygen outputs
  - Run presigning → presign outputs for all 5 participants

Attack:
  1. Designate participant P1 as coordinator (malicious).
  2. Initiate signing session: all participants call sign(..., coordinator=P1, ...).
  3. P2..P5 each execute do_sign_participant:
       - compute s_i
       - chan.send_private(wait_round, P1, &s_me)  ← presignature consumed
       - return Ok(None)
  4. P1 (coordinator) receives all s_i values via recv_from_others.
  5. P1 aborts (e.g., returns Err or simply drops the result without broadcasting).

Result:
  - P2..P5 have returned Ok(None) — they have no signature.
  - P2..P5's presignatures are consumed (must not be reused).
  - No valid signature was produced.
  - P2..P5 must re-run the full offline presigning phase to sign again.
  - P1 can repeat this attack on every new presigning batch.
``` [7](#0-6) [8](#0-7)

async fn do_sign_coordinator(
    mut chan: SharedChannel,
    participants: ParticipantList,
    me: Participant,
    public_key: AffinePoint,
    presignature: RerandomizedPresignOutput,
    msg_hash: Scalar,
) -> Result<SignatureOption, ProtocolError> {
    let mut s = compute_signature_share(&presignature, msg_hash, &participants, me)?.0;
    let wait_round = chan.next_waitpoint();

    for (_, s_i) in
        recv_from_others::<SerializableScalar<C>>(&chan, wait_round, &participants, me).await?
    {
        // Sum the linearized shares
        s += s_i.0;
    }

    // raise error if s is zero
    if s.is_zero().into() {
        return Err(ProtocolError::AssertionFailed(
            "signature part s cannot be zero".to_string(),
        ));

    // Normalize s
    s.conditional_assign(&(-s), s.is_high());

    let sig = Signature {
        big_r: presignature.big_r,
        s,
    };

    if !sig.verify(&public_key, &msg_hash) {
        return Err(ProtocolError::AssertionFailed(
            "signature failed to verify".to_string(),
        ));
