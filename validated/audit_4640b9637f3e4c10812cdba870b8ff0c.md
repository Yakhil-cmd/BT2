### Title
Missing Security-Critical Constraint Documentation on `presign` Public API Enables Split-View Secret Key Extraction - (File: src/ecdsa/robust_ecdsa/presign.rs)

---

### Summary

The `presign` function in `src/ecdsa/robust_ecdsa/presign.rs` is a public API entry point whose docstring omits the critical security constraint that the presigning participant count `N1` must equal exactly `2t+1` (`2 * max_malicious + 1`). This constraint is documented only in an external markdown file and partially referenced in the `sign` function's docstring — not at the `presign` call site. Because the `presign` function does not document (and, based on available evidence, does not enforce) the `N1 = 2t+1` bound, a caller who reads only the `presign` API will unknowingly generate presignatures with `N1 > 2t+1`. A malicious coordinator can then exploit this to run a split-view attack across two signing sessions, recovering the aggregate secret key.

---

### Finding Description

The robust ECDSA implementation (`src/ecdsa/robust_ecdsa/`) is a two-phase protocol: `presign` (offline) followed by `sign` (online). The security of the scheme depends critically on the presigning participant count `N1` being exactly `2t+1`.

The `presign` function's docstring reads:

> "This is the first phase of performing a signature, in which we perform all the work we can do without yet knowing the message to be signed. This work does depend on the private key though, and it's crucial that a presignature is never reused." [1](#0-0) 

The docstring mentions only the presignature-reuse constraint. It says **nothing** about the `N1 = 2t+1` bound.

The `sign` function's docstring does briefly mention this constraint:

> "To reduce risk in this implementation, require `N1 = N2 = 2 * max_malicious + 1`, ensure all participants agree on `(msg_hash, tweak, participants)` when creating `RerandomizedPresignOutput`, never reuse a presignature, and do not sign with `msg_hash == 0`." [2](#0-1) 

However, this is a recommendation placed on the `sign` caller — it is not present at the `presign` call site where the dangerous presignature is actually created. The `sign` function body enforces `N2 = 2t+1`: [3](#0-2) 

But there is no corresponding enforcement or documentation of `N1 = 2t+1` in `presign`. The external specification explicitly warns:

> "Do **not** allow `N1 > 2t+1`... Allowing larger presigning sets or smaller signing sets enables split-view and presignature-reuse attacks when a coordinator can run parallel or partially overlapping signing sessions." [4](#0-3) 

And the attack consequence is documented as:

> "A novel split-view attack exists that can extract the secret key using as few as `2t+2` presigning participants, with as few as two signing sessions." [5](#0-4) 

---

### Impact Explanation

**Critical — Extraction of the aggregate secret key.**

If `presign` is called with `N1 = 2t+2` (or more) participants — which the `presign` API neither documents against nor enforces — the resulting presignature is structurally unsafe. A malicious coordinator can then invoke `sign` twice with two different subsets of `N2 = 2t+1` participants (which `sign` accepts, since `N2 = 2t+1` satisfies its check), presenting different `(msg_hash, tweak)` pairs to each subset. Because the signature-share linearization step in this modified scheme ties the nonce to the participant set, the two resulting signatures have multiplicatively related nonces. Standard ECDSA nonce-reuse analysis then recovers the full secret key. [6](#0-5) 

---

### Likelihood Explanation

**High.** The `presign` function is a top-level public API function. Any integrator who reads only the `presign` docstring — the natural starting point — receives no warning about the `N1 = 2t+1` constraint. The constraint is buried in an external markdown file and partially mentioned in the `sign` docstring, which a developer may read only after already having designed their presign invocation. The malicious coordinator role is explicitly within the threat model of this library. [7](#0-6) 

---

### Recommendation

1. **Add an explicit security warning to the `presign` function docstring** stating that `N1` (the number of presigning participants) must equal exactly `2 * max_malicious + 1`, and that violating this enables split-view attacks leading to full secret key recovery.

2. **Enforce `N1 = 2t+1` inside `presign`** by adding the same participant-count check that `sign` applies for `N2`, rejecting calls where `participants.len() != 2 * max_malicious + 1`. [8](#0-7)

### Proof of Concept

1. Integrator reads `presign` docstring: learns only "never reuse a presignature." Calls `presign` with `N1 = 2t+2` participants (e.g., `t=1`, so `N1=4`). No error is returned.

2. Malicious coordinator receives the presignature output. Calls `sign` twice:
   - Session A: participants `{P1, P2, P3}` (`N2 = 3 = 2t+1`), message hash `h1`.
   - Session B: participants `{P1, P2, P4}` (`N2 = 3 = 2t+1`), message hash `h2 ≠ h1`.
   - Both calls pass the `sign` validation (N2 = 2t+1, msg_hash ≠ 0).

3. The two resulting ECDSA signatures `(R, s1)` and `(R, s2)` share the same nonce point `R` (derived from the same presignature) but have different `s` values computed with different linearization coefficients. Standard nonce-reuse algebra recovers the secret key `x` from `s1 - s2 = (h1 - h2) * k^{-1}` (modulo the linearization structure documented in the spec). [9](#0-8) [1](#0-0) [10](#0-9)

# Security considerations

Before implementing or using the robust ECDSA scheme implemented here,
be aware that it is vulnerable to **split-view attacks** in the robust setting when the
signing parameters are not globally consistent. If different subsets of size at least
$2t + 1$ sign different $(h, \epsilon)$ values using shares derived from the same
presignature, the resulting signatures use multiplicatively related nonces and the
secret key can be recovered using standard ECDSA nonce-reuse attacks.

Moreover, due to protocol modifications relative to [[DJNPO20](https://eprint.iacr.org/2020/501)] (notably signature-share
linearization), **a novel split-view attack exists that can extract the secret key using as
few as $2t + 2$ presigning participants**, with as few as two signing sessions.

To reduce the risk of accidental misuse, enforce the following constraints:

1. **Use exactly $N_1 = N_2 = 2t + 1$ participants for both presigning and signing.**
   Do **not** allow any deviation from this value. In particular:

   * Do **not** allow $N_1 > 2t + 1$, and
   * Do **not** allow $N_2 < N_1$.

   Allowing larger presigning sets or smaller signing sets enables split-view and
   presignature-reuse attacks when a coordinator can run parallel or partially overlapping
   signing sessions.

2. **Ensure all participants agree on $(h, \epsilon)$ and the signing set.**
   The coordinator must not be able to present different message hashes, tweaks, or
   participant lists to different signers.

3. **Never reuse a presignature**, even across failed, aborted, or partially completed
   signing sessions.

4. **Do not sign with $h = 0$** (the zero message hash).
   This input enables a related algebraic split-view attack in the modified scheme when
   $N_1 > 2t + 1$.
