### Title
CKD Output Corruption via Unverified Participant Set — Malicious Participant Supplies Wrong Lagrange Coefficients - (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

The `ckd()` entry point accepts a caller-supplied `participants` list that is never verified against the participant set used during DKG. Because Lagrange coefficients are derived solely from this unverified list, a malicious participant can pass a different participant set, inject a wrong Lagrange-weighted share into the coordinator's aggregation, and corrupt the final CKD derived-key output that honest parties accept.

---

### Finding Description

`KeygenOutput` for CKD stores only `public_key` and `private_share`; it does not record the participant set or threshold from the DKG session that produced those shares. [1](#0-0) 

When `ckd()` is called, the `participants` slice is validated only for duplicates and self-membership; it is never cross-checked against any DKG-time participant set. [2](#0-1) 

Inside `compute_signature_share`, the Lagrange coefficient `λ_i` is computed exclusively from the caller-supplied `participants` list: [3](#0-2) 

The coordinator aggregates every participant's `(norm_big_y, norm_big_c)` contribution with no proof-of-knowledge and no consistency check: [4](#0-3) 

Because `KeygenOutput` carries no record of the DKG participant set, neither the library nor the coordinator can detect that a participant used a different set. [5](#0-4) 

---

### Impact Explanation

The correct CKD output requires `Σ λᵢ · xᵢ = x` (the master secret). If malicious participant A substitutes Lagrange coefficients from a different participant set `{A, B}` instead of the true set `{A, B, C}`, the coordinator computes:

```
norm_big_c_total = λ'_A·(x_A·H + y_A·Apk) + λ_B·(x_B·H + y_B·Apk) + λ_C·(x_C·H + y_C·Apk)
```

After unmasking, the derived key becomes `(λ'_A·x_A + λ_B·x_B + λ_C·x_C)·H ≠ x·H`. Honest parties accept this incorrect derived key as the legitimate CKD output. This matches the allowed impact: **High — Corruption of CKD outputs so honest parties accept inconsistent or unusable cryptographic outputs.**

---

### Likelihood Explanation

Any single participant in the CKD session can trigger this unilaterally. No coordination with other attackers is required. The attacker only needs to instantiate `ckd()` with a participant list that differs from the DKG list — a trivial API-level action available to every library caller. No cryptographic assumption needs to be broken.

---

### Recommendation

1. **Bind participant set to `KeygenOutput`**: Store the sorted participant list (or a hash thereof) inside `KeygenOutput` at DKG time and verify it matches the `participants` argument inside `ckd()`.
2. **Add per-participant proofs**: Require each participant to supply a zero-knowledge proof that their `(norm_big_y, norm_big_c)` was computed using the correct private share and the agreed Lagrange coefficient, so the coordinator can reject malformed contributions before aggregation.

---

### Proof of Concept

```
Setup:
  DKG with participants {A=1, B=2, C=3}, threshold=2.
  Each participant holds private share xᵢ; master key x = Σ λᵢ·xᵢ (over {A,B,C}).

Attack:
  CKD session is initiated with participants {A, B, C}.
  Honest B and C call ckd(&[A,B,C], coordinator, ...).
  Malicious A calls ckd(&[A,B], coordinator, ...).   // ← different participant set

  A's λ'_A = scalar_B / (scalar_B - scalar_A)       // 2-party Lagrange, not 3-party
  A sends norm_big_c_A = λ'_A · (x_A·H(pk,app_id) + y_A·Apk)

  Coordinator sums:
    total = norm_big_c_A + norm_big_c_B + norm_big_c_C
          = (λ'_A·x_A + λ_B·x_B + λ_C·x_C)·H + (...)·Apk

  After unmasking: derived_key = (λ'_A·x_A + λ_B·x_B + λ_C·x_C)·H ≠ x·H

  Coordinator outputs and honest parties accept a corrupted derived key.
  No error is raised; the library returns Ok(Some(ckd_output)).
```

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}
