### Title
Permanent Fund Freeze When `ft_transfer` Promise Fails in `ExitToNear` Precompile Without `error_refund` Feature - (File: `engine-precompiles/src/native.rs`)

---

### Summary

The `ExitToNear` precompile burns ERC-20 tokens (or deducts ETH) on the Aurora EVM side before scheduling a NEAR `ft_transfer` promise. When the `error_refund` Cargo feature is not compiled in, no callback is attached to that promise. If the `ft_transfer` call fails on the NEAR side, the burned tokens are permanently unrecoverable — an exact structural analog to the Scroll stuck-ETH bug.

---

### Finding Description

The `ExitToNear` precompile's `run` method constructs a `callback_args` struct and then decides whether to schedule a bare promise or a promise-with-callback: [1](#0-0) 

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,          // ← always None when feature is absent
    transfer_near: transfer_near_args,
};
``` [2](#0-1) 

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // ← no callback, no refund
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs {
        base: transfer_promise,
        callback: PromiseCreateArgs {
            method: "exit_to_near_precompile_callback".to_string(),
            ...
        },
    })
};
```

For the common ERC-20 and ETH exit paths (`ft_transfer`, not wNEAR unwrap), `transfer_near_args` is `None`. Without `error_refund`, both fields of `callback_args` are `None`, so `callback_args == default()` is `true`, and **no callback is ever scheduled**. [3](#0-2) 

The `error_refund` feature is **not** listed in the `default` feature set of either crate: [4](#0-3) [5](#0-4) 

The callback that would perform the refund — `exit_to_near_precompile_callback` — only executes `engine::refund_on_error` when `args.refund` is `Some(...)`: [6](#0-5) 

Without the feature, `args.refund` is always `None`, so even if the callback were somehow reached, no refund would occur.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

When a user calls the `ExitToNear` precompile (directly or via an ERC-20 contract's burn/withdraw function):

1. ERC-20 tokens are **burned** on the Aurora EVM (state is committed).
2. A NEAR `ft_transfer` promise is dispatched to the NEP-141 contract.
3. If that promise fails for any reason (recipient account not registered with the NEP-141 token, contract paused, insufficient storage deposit, etc.), there is **no callback** to re-mint the burned tokens.
4. The user's tokens are permanently destroyed with no recovery path.

The test suite explicitly documents this outcome: [7](#0-6) 

```rust
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
```

---

### Likelihood Explanation

**High.** The trigger conditions are realistic and user-controlled:

- A user specifies a NEAR recipient account that has never called `storage_deposit` on the target NEP-141 contract (a common mistake for new users).
- A user specifies a NEAR account that does not exist.
- The NEP-141 contract is paused at the time the promise executes.
- Any other transient or permanent failure of the `ft_transfer` call.

The `error_refund` feature is absent from the default build, meaning any deployment compiled without explicitly enabling it is vulnerable. The attacker-controlled entry path is the standard ERC-20 `withdraw` (or direct ETH send to the precompile address), which is a fully unprivileged, user-facing interface.

---

### Recommendation

1. **Add `error_refund` to the `default` feature set** in both `engine-precompiles/Cargo.toml` and `engine/Cargo.toml`, or unconditionally include the refund callback logic without a feature gate.
2. Ensure the `exit_to_near_precompile_callback` always receives a populated `refund` field so that any `ft_transfer` failure triggers `engine::refund_on_error` and re-mints the burned tokens to the original sender.
3. Apply the same pattern to `ExitToEthereum`, which currently schedules a bare `withdraw` promise with **no callback at all** — a structurally identical but unconditional fund-freeze risk. [8](#0-7) 

---

### Proof of Concept

**Setup:** Deploy Aurora without the `error_refund` feature (the default). Deploy a NEP-141 token and its corresponding ERC-20 mirror on Aurora. Fund a user with ERC-20 tokens.

**Steps:**

1. User calls the ERC-20 contract's `withdraw` function specifying a NEAR recipient account that is **not registered** with the NEP-141 contract (no `storage_deposit` made).
2. The ERC-20 contract burns the user's tokens and calls the `ExitToNear` precompile.
3. The precompile schedules `PromiseArgs::Create(ft_transfer_promise)` — no callback.
4. The NEAR runtime executes `ft_transfer` on the NEP-141 contract; it panics with `"The account X is not registered"`.
5. Because no callback was attached, the panic is silently absorbed by the NEAR runtime.
6. The user's ERC-20 balance is zero. The NEP-141 balance of the recipient is zero. The tokens are gone.

**Confirmed by test:** [9](#0-8) 

The test `test_exit_to_near_refund` explicitly passes `"unregistered.near"` as the recipient, causing `ft_transfer` to fail, and asserts that without `error_refund` the ERC-20 balance is reduced by `FT_EXIT_AMOUNT` with no refund — confirming permanent token loss.
