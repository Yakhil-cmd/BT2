### Title
`skip_stateful_validations` Bypasses `__validate__` Signature Check for Invoke Transactions When Any Account Tx Exists in Mempool — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The gateway's `skip_stateful_validations` function uses `account_tx_in_pool_or_recent_block` as a proxy for "this account has a pending `deploy_account` transaction." The proxy is incorrect: it returns `true` for **any** account that has any transaction in the pool, including accounts deployed via a factory contract (`deploy` syscall) that carry a regular invoke tx at nonce 0. An attacker who observes such an account in the mempool can submit a nonce-1 invoke transaction with an **invalid signature** and have it admitted to the mempool without the `__validate__` entry point ever being called.

---

### Finding Description

**Root cause — `skip_stateful_validations`** [1](#0-0) 

The function skips the blockifier `__validate__` call when all three conditions hold:

1. The incoming transaction is an `Invoke` with `tx.nonce() == 1`.
2. The on-chain account nonce is `0` (account not yet seen on-chain from the gateway's perspective).
3. `account_tx_in_pool_or_recent_block(sender_address)` returns `true`.

The comment claims condition 3 is sufficient to prove a `deploy_account` tx exists:

> "It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."

**The claim is false.** `account_tx_in_pool_or_recent_block` is implemented as: [2](#0-1) 

It returns `true` whenever `tx_pool.contains_account(address)` is true — i.e., whenever **any** transaction from that address is in the pool, regardless of type or nonce. An account deployed via the `deploy` syscall (factory pattern) has on-chain nonce `0` and can have a valid nonce-0 invoke tx in the pool. That is enough to satisfy condition 3.

**Validation path that is bypassed**

When `skip_validate = true` is returned, `run_validate_entry_point` sets `execution_flags.validate = false`: [3](#0-2) 

Inside `StatefulValidator::perform_validations`, the `__validate__` call is gated on that flag: [4](#0-3) 

So the account's `__validate__` entry point — the only place where the transaction signature is verified — is never executed.

**`validate_by_mempool` does not check signatures**

The mempool's `validate_tx` only checks nonce ordering and fee escalation: [5](#0-4) 

`ValidationArgs` carries no signature field: [6](#0-5) 

**Full call chain** [7](#0-6) 

`validate_state_preconditions` checks resource bounds and nonce range (not signature); `validate_by_mempool` checks nonce ordering (not signature); `skip_stateful_validations` then returns `true` and the signature check is skipped entirely.

---

### Impact Explanation

**Impact: High — Mempool/gateway admission accepts an invalid (unauthorized) transaction before sequencing.**

An attacker can inject a nonce-1 invoke transaction bearing an arbitrary (invalid) signature on behalf of any victim account that currently has a nonce-0 invoke tx pending in the mempool. The gateway admits the transaction without verifying the signature. The corrupted value is the admission decision: a transaction that should be rejected as `ValidateFailure` is instead accepted and forwarded to the mempool.

Downstream consequences:
- The invalid tx occupies a mempool slot for the victim's nonce-1 position, potentially blocking or delaying the victim's own legitimate nonce-1 tx (the attacker's tx would only be displaced if the victim pays a higher fee via fee escalation).
- The sequencer wastes execution resources when the batcher later calls `__validate__` and the tx reverts.
- Repeated at scale this constitutes a targeted DoS against specific accounts.

---

### Likelihood Explanation

**Likelihood: Medium.**

Prerequisites:
1. The victim account must have been deployed via a factory contract (`deploy` syscall), giving it on-chain nonce `0`.
2. The victim must have a pending nonce-0 invoke tx in the mempool at the time of the attack.

Both conditions are observable from the public mempool. The attacker does not need the victim's private key. The attack requires only knowledge of the victim's address and the ability to submit a crafted transaction to the gateway.

---

### Recommendation

Replace the `account_tx_in_pool_or_recent_block` proxy with a check that specifically confirms a `deploy_account` transaction exists for the sender address. The mempool should expose a dedicated query such as `has_deploy_account_tx_in_pool(address) -> bool` that inspects the transaction type, rather than returning `true` for any pooled transaction. Until that is available, the gateway should additionally verify that the account does **not** already exist on-chain (i.e., `get_class_hash_at(sender_address) == ClassHash::default()`) before skipping `__validate__`. An account with nonce `0` that already has a deployed class hash is a factory-deployed account, not an undeployed one, and must not have its validation skipped.

---

### Proof of Concept

1. **Setup**: Deploy account `A` via a factory contract. `A` has on-chain nonce `0` and a deployed class hash.

2. **Seed the mempool**: Submit a valid nonce-0 invoke tx from `A` (with a correct signature). It passes all gateway checks including `__validate__` and enters the mempool. `account_tx_in_pool_or_recent_block(A)` now returns `true`.

3. **Attack**: Construct a nonce-1 invoke tx for `A` with an **invalid/random signature** and arbitrary calldata. Submit it to the gateway.

4. **Gateway processing**:
   - `validate_state_preconditions`: nonce `1` satisfies `0 <= 1 <= 0 + max_allowed_nonce_gap` (passes). [8](#0-7) 
   - `validate_by_mempool`: mempool `validate_tx` checks nonce ordering only — nonce `1 >= resolve_nonce(A, 0) = 0` (passes). [9](#0-8) 
   - `skip_stateful_validations`: `tx.nonce() == 1`, `account_nonce == 0`, `account_tx_in_pool_or_recent_block(A) == true` → returns `true`. [10](#0-9) 
   - `run_validate_entry_point(skip_validate=true)`: `execution_flags.validate = false`, `__validate__` is never called. [11](#0-10) 

5. **Result**: The invalid-signature nonce-1 tx is admitted to the mempool. The gateway returns success. The broken invariant: a transaction with an unauthorized signature was accepted through account validation.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
