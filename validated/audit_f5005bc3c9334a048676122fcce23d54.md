### Title
`factory_set_wnear_address` Silently Invalidates All Existing User wNEAR Approvals, Permanently Bricking XCC for Users Requiring NEAR Payment - (`engine/src/contract_methods/xcc.rs`)

---

### Summary

When the owner calls `factory_set_wnear_address` to update the wNEAR ERC-20 contract address, the XCC precompile immediately begins calling `transferFrom` on the new wNEAR contract. However, all existing users had approved the XCC precompile address on the **old** wNEAR contract. Since no approvals exist on the new contract, every XCC call that requires NEAR payment will revert, bricking the entire XCC feature for all affected users.

---

### Finding Description

The XCC precompile (`engine-precompiles/src/xcc.rs`) collects wNEAR from the calling user by issuing an EVM sub-call to `transferFrom(sender, engine_implicit_address, required_near)` against whichever address is currently stored as the wNEAR ERC-20 contract:

```rust
// engine-precompiles/src/xcc.rs  L193-200
let wnear_address = state::get_wnear_address(&self.io);
let context = aurora_evm::Context {
    address: wnear_address.raw(),
    caller: cross_contract_call::ADDRESS.raw(),
    apparent_value: U256::zero(),
};
let (exit_reason, return_value) =
    handle.call(wnear_address.raw(), None, tx_data, None, false, &context);
```

This `transferFrom` succeeds only if the user has previously called `old_wnear.approve(XCC_PRECOMPILE_ADDRESS, amount)`. The approval is stored inside the **old** wNEAR ERC-20 contract's storage.

The owner-callable `factory_set_wnear_address` replaces the stored address with a new one:

```rust
// engine/src/contract_methods/xcc.rs  L102-115
pub fn factory_set_wnear_address<I: IO + Copy, E: Env>(io: I, env: &E) -> Result<(), ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        ...
        require_owner_only(&state, &env.predecessor_account_id())?;
        let address = io.read_input_arr20()?;
        xcc::set_wnear_address(&mut io, &Address::from_array(address));
        Ok(())
    })
}
```

`set_wnear_address` only writes the new address to storage — no approval migration, no event to prompt users to re-approve, no grace period:

```rust
// engine/src/xcc.rs  L369-373
pub fn set_wnear_address<I: IO>(io: &mut I, address: &Address) {
    let key = storage::bytes_to_key(KeyPrefix::CrossContractCall, WNEAR_KEY);
    io.write_storage(&key, address.as_bytes());
}
```

After the call, the XCC precompile reads the new address and attempts `transferFrom` on a contract where no user has any allowance for `cross_contract_call::ADDRESS`. The sub-call reverts, and the entire XCC transaction reverts with it.

The `transferFrom` path is taken whenever `required_near != ZERO_YOCTO`:

```rust

```

This means:
- **Every new user** (no deployed router) is always charged `STORAGE_AMOUNT` and is therefore always affected.
- **Every existing user** who attaches any NEAR to their XCC call is affected.

---

### Impact Explanation

After `factory_set_wnear_address` is called with a new address, the XCC cross-contract-call feature is bricked for all users who need to pay NEAR. New users cannot deploy a router contract at all (storage staking always requires NEAR). Existing users cannot attach NEAR to any XCC call. Because the approval is an EVM-level allowance stored inside the old wNEAR ERC-20 contract, it cannot be migrated by the engine — each user must individually re-approve on the new contract before XCC works again. Until they do, their wNEAR tokens intended for XCC are frozen in place and the XCC feature is entirely unavailable to them.

**Impact: High — Temporary freezing of funds.**

---

### Likelihood Explanation

`factory_set_wnear_address` is a documented, intentional admin function whose stated purpose is to update the wNEAR ERC-20 address used by XCC. It would naturally be called whenever the wNEAR contract is redeployed, migrated, or replaced. There is no in-code warning, no migration step, and no documentation noting that all users must re-approve after the change. The likelihood of this being called in a live deployment without awareness of the approval-invalidation side-effect is high.

---

### Recommendation

`factory_set_wnear_address` should perform a migration step before replacing the address. Concretely:

1. **Emit a prominent log/event** at the EVM level notifying all users that they must re-approve `cross_contract_call::ADDRESS` on the new wNEAR contract before using XCC.
2. **Optionally enforce a time-lock** between announcing the new address and activating it, giving users time to re-approve.
3. **Document** in the function's NatSpec/comments that calling this function invalidates all existing user approvals and that users must re-approve on the new contract.

---

### Proof of Concept

1. User calls `old_wnear.approve(cross_contract_call::ADDRESS, large_amount)` — approval stored in `old_wnear` contract storage.
2. Owner calls `factory_set_wnear_address(new_wnear_address)` — engine now stores `new_wnear_address`.
3. User submits an XCC transaction (e.g., first-ever XCC call, so `required_near = STORAGE_AMOUNT > 0`).
4. XCC precompile reads `wnear_address = new_wnear_address` from storage.
5. Precompile issues EVM sub-call: `new_wnear.transferFrom(user, engine_implicit, STORAGE_AMOUNT)`.
6. `new_wnear` has zero allowance for `cross_contract_call::ADDRESS` from `user` → ERC-20 reverts.
7. XCC precompile propagates the revert → user's transaction reverts.
8. User cannot use XCC at all until they call `new_wnear.approve(cross_contract_call::ADDRESS, amount)`.
