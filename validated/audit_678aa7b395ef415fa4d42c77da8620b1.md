### Title
`protocolUnpausePool` Sets Wrong Pause Level, Leaving Pool Permanently Swap-Blocked — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`protocolUnpausePool` calls `setPause(1)` instead of `setPause(0)`. After the protocol owner "unpauses" a pool, the pool remains in admin-paused state (`pauseLevel = 1`), and all swaps continue to revert. The pool admin must then separately call `unpausePool` to reach the active state. This is a direct analog to the LayerEdge boundary bug: the wrong target state is written (the old intermediate level rather than the correct new level), leaving the invariant broken.

---

### Finding Description

The pause level encoding is:

```
0 = active
1 = paused by admin
2 = paused by protocol
```

`protocolPausePool` correctly accepts pools at level 0 **or** 1 and moves them to 2:

```solidity
// MetricOmmPoolFactory.sol L392-396
function protocolPausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 0 && cur != 1) revert InvalidPauseTransition(cur, 2);
    IMetricOmmPoolFactoryActions(pool).setPause(2);
}
```

`protocolUnpausePool` then moves the pool from 2 → **1** instead of 2 → **0**:

```solidity
// MetricOmmPoolFactory.sol L399-403
function protocolUnpausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 2) revert InvalidPauseTransition(cur, 1);
    IMetricOmmPoolFactoryActions(pool).setPause(1);   // ← should be setPause(0)
}
```

`_checkNotPaused` in the pool reverts on **any** non-zero level:

```solidity
// MetricOmmPool.sol L643-645
function _checkNotPaused() internal view {
    if (pauseLevel != 0) revert PoolPaused();
}
```

`swap` is gated by `whenNotPaused`:

```solidity
// MetricOmmPool.sol L224
) external whenNotPaused nonReentrant(PoolActions.SWAP) returns (int128, int128) {
```

So after `protocolUnpausePool`, `pauseLevel == 1`, `_checkNotPaused` still reverts, and every swap call fails.

---

### Impact Explanation

After the protocol owner calls `protocolUnpausePool` on a pool that was active before the protocol pause, the pool is left in admin-paused state. All swaps revert with `PoolPaused`. LPs earn zero spread fees during the extended unintended pause. The pool admin must discover the problem and separately call `unpausePool` (which transitions 1 → 0) to restore functionality. Until that happens, the swap flow is completely broken and LP fee revenue is lost.

---

### Likelihood Explanation

The trigger is a normal, expected operational sequence: protocol pauses a pool for an emergency, then unpauses it. No attacker is required. Any pool that was at level 0 (active) before the protocol pause will be left at level 1 after the protocol unpause. The protocol owner has no on-chain signal that the pool is still paused; the `PoolPriceProviderUpdated` / `PauseLevelUpdated` events would show level 1, but the owner may not notice the distinction.

---

### Recommendation

Change `setPause(1)` to `setPause(0)` in `protocolUnpausePool`:

```diff
function protocolUnpausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
-   if (cur != 2) revert InvalidPauseTransition(cur, 1);
-   IMetricOmmPoolFactoryActions(pool).setPause(1);
+   if (cur != 2) revert InvalidPauseTransition(cur, 0);
+   IMetricOmmPoolFactoryActions(pool).setPause(0);
}
```

If the design intent is to restore the pre-protocol-pause level (0 or 1), the factory must store the previous level before calling `protocolPausePool` and restore it in `protocolUnpausePool`.

---

### Proof of Concept

1. Pool is deployed; `pauseLevel = 0` (active). Swaps succeed.
2. Protocol owner calls `protocolPausePool(pool)`. Pool transitions 0 → 2. Swaps revert.
3. Protocol owner calls `protocolUnpausePool(pool)`. Pool transitions 2 → **1**. Swaps **still revert** with `PoolPaused`.
4. Pool admin must call `unpausePool(pool)` (1 → 0) to restore swap functionality.
