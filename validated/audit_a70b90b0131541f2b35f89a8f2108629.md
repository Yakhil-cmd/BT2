### Title
Gateway Admits Transactions Using Stale Previous-Block Gas Prices That Fail at Batcher Execution When Prices Rise — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The gateway's stateful resource-bounds validation and its embedded blockifier pre-validation both use the **previous block's** gas prices. The batcher builds the actual block with the **current block's** gas prices. When any gas price rises between blocks, transactions that cleared every gateway check are admitted to the mempool but fail at batcher execution with `MaxGasPriceTooLow`. An observer who can predict an imminent gas-price increase (e.g., by watching L1 gas price feeds before the sequencer's oracle updates) can flood the mempool with transactions that are guaranteed to fail, wasting sequencer resources and displacing legitimate transactions.

---

### Finding Description

**Step 1 — Gateway threshold check uses previous-block L2 gas price.**

`validate_resource_bounds` reads the gas price from `gateway_fixed_block_state_reader.get_block_info()`, which returns the **last finalized block's** info. The developer TODO at line 229 explicitly acknowledges this is wrong:

```rust
// TODO(Arni): getnext_l2_gas_price from the block header.
let previous_block_l2_gas_price = self
    .gateway_fixed_block_state_reader
    .get_block_info()
    .await?
    .gas_prices
    .strk_gas_prices
    .l2_gas_price;
self.validate_tx_l2_gas_price_within_threshold(
    executable_tx.resource_bounds(),
    previous_block_l2_gas_price,
)?;
``` [1](#0-0) 

The threshold is `min_gas_price_percentage% × previous_block_l2_gas_price`. A transaction with `max_price_per_unit = P` passes if `P ≥ threshold(prev_price)`.

**Step 2 — The blockifier validation inside the gateway also uses the same stale prices.**

`run_validate_entry_point` calls `get_block_info()` again (same source), increments only the block number, and passes the resulting `block_info` — with the **previous block's gas prices** — into `BlockContext`:

```rust
let mut block_info = self.gateway_fixed_block_state_reader.get_block_info().await?;
block_info.block_number = block_info.block_number.unchecked_next();
let block_context = BlockContext::new(block_info, ...);
``` [2](#0-1) 

Inside `StatefulValidator::perform_validations`, `perform_pre_validation_stage` is called, which invokes `check_fee_bounds`. That function checks `resource_bounds.max_price_per_unit >= block_info.gas_prices.*_gas_price(fee_type)` for every resource dimension: [3](#0-2) 

Because `block_info` carries the **previous** block's prices, a transaction with `max_price_per_unit = P` passes as long as `P ≥ prev_price` — not `P ≥ next_price`.

**Step 3 — The batcher uses the current block's gas prices.**

When the batcher builds the next block it constructs a fresh `BlockContext` with the **current** block's gas prices. `AccountTransaction::new_for_sequencing` sets `validate: true` and `strict_nonce_check: true`, and `perform_pre_validation_stage` is called again with the new context. If the current L2 gas price `Q > P`, the check `P ≥ Q` fails and the transaction is rejected with `TransactionFeeError::InsufficientResourceBounds { MaxGasPriceTooLow }`. [4](#0-3) 

**The broken invariant:** Every transaction admitted to the mempool by the gateway should pass blockifier pre-validation at the gas prices that will be in effect when the block is built. Because the gateway validates against `prev_price` but the batcher validates against `current_price`, any rise in gas price between blocks breaks this invariant.

---

### Impact Explanation

**Impact: High — Mempool/gateway admission accepts transactions that will be rejected before sequencing.**

Concretely:

- Transactions with `prev_price ≤ max_price_per_unit < current_price` pass all gateway checks and enter the mempool, but are dropped or reverted at batcher execution.
- The sequencer expends CPU on gateway blockifier validation, mempool insertion, and batcher pre-validation for transactions that can never be included.
- Legitimate transactions with correct prices may be displaced from the mempool by the flood of stale-price transactions.
- The `validate_resource_bounds` check only covers L2 gas price for `AllResources` transactions; for legacy `L1Gas` transactions there is **no** gateway-level gas-price threshold check at all, so the same stale-price gap exists for L1 gas price movements too (caught only by the blockifier inside the gateway, still using the previous block's price). [5](#0-4) 

---

### Likelihood Explanation

L2 gas prices on Starknet are derived from L1 gas prices. L1 gas prices are publicly observable in real time (e.g., via Ethereum mempool or gas-price APIs) and change every block. An attacker can:

1. Watch the L1 gas price oracle or the sequencer's pending block header to detect an imminent price increase.
2. Craft V3 (`AllResources`) invoke transactions with `l2_gas.max_price_per_unit` set just above `threshold(prev_price)` but below the expected `next_price`.
3. Submit them in bulk; all pass gateway validation and enter the mempool.
4. When the batcher builds the block at `next_price`, every submitted transaction fails `check_fee_bounds` and is discarded.

This is directly analogous to the DYAD H-03 oracle-frontrunning pattern: the attacker exploits the lag between the price used for admission and the price used for execution.

---

### Recommendation

Replace the `get_block_info()` call in both `validate_resource_bounds` and `run_validate_entry_point` with the **next block's** projected gas prices (as the TODO already notes). The batcher already computes the next block's gas prices before building; expose that value to the gateway's stateful validator so both the threshold check and the blockifier pre-validation use the same price the batcher will use. Until that is available, the threshold multiplier (`min_gas_price_percentage`) should be set conservatively high enough to absorb typical inter-block price swings, and the gateway should apply the threshold check to **all** resource dimensions (L1 gas, L1 data gas, L2 gas), not only L2 gas.

---

### Proof of Concept

```
Setup:
  previous_block.l2_gas_price  = 100
  min_gas_price_percentage     = 50   → threshold = 50
  next_block.l2_gas_price      = 110  (L1 gas price rose)

Attacker submits 1000 V3 invoke transactions each with:
  l2_gas.max_price_per_unit = 60   (60 ≥ 50 → passes gateway threshold)
  l1_gas.max_price_per_unit = prev_l1_price
  l1_data_gas.max_price_per_unit = prev_l1_data_price

Gateway path (all using previous block prices = 100):
  validate_resource_bounds:  60 ≥ 50  → OK
  check_fee_bounds (blockifier, prev prices):
    60 ≥ 100?  NO — wait, this would also fail.
```

Wait — let me correct the scenario. The `check_fee_bounds` inside the gateway also uses `prev_price = 100`. So `max_price_per_unit = 60 < 100` would fail there too. The exploitable window is narrower: the attacker must set `max_price_per_unit` between `prev_price` and `next_price`:

```
Setup:
  previous_block.l2_gas_price  = 100
  min_gas_price_percentage     = 50   → threshold = 50
  next_block.l2_gas_price      = 110

Attacker submits transactions with:
  l2_gas.max_price_per_unit = 105   (105 ≥ 50 → passes threshold)
                                    (105 ≥ 100 → passes blockifier check in gateway)
                                    (105 < 110 → FAILS blockifier check in batcher)

Gateway result: ADMITTED to mempool
Batcher result: REJECTED — MaxGasPriceTooLow { max=105, actual=110 }

1000 such transactions enter the mempool, consume gateway CPU and mempool
slots, and are all discarded at batcher time, displacing legitimate traffic.
```

The attacker's cost is only the gas to submit the transactions to the RPC endpoint; since they never execute, no on-chain fee is paid. The sequencer bears the full validation cost. [6](#0-5) [2](#0-1) [7](#0-6) [4](#0-3)

    // TODO(Arni): Consider running this validation for all gas prices.
    fn validate_tx_l2_gas_price_within_threshold(
        &self,
        tx_resource_bounds: ValidResourceBounds,
        previous_block_l2_gas_price: NonzeroGasPrice,
    ) -> StatefulTransactionValidatorResult<()> {
        match tx_resource_bounds {
            ValidResourceBounds::AllResources(tx_resource_bounds) => {
                let tx_l2_gas_price = tx_resource_bounds.l2_gas.max_price_per_unit;
                let gas_price_threshold_multiplier =
                    Ratio::new(self.config.min_gas_price_percentage.into(), 100_u128);
                let threshold = (gas_price_threshold_multiplier
                    * previous_block_l2_gas_price.get().0)
                    .to_integer();
                if tx_l2_gas_price.0 < threshold {
                    return Err(StarknetError {
                        // We didn't have this kind of an error.
                        code: StarknetErrorCode::UnknownErrorCode(
                            "StarknetErrorCode.GAS_PRICE_TOO_LOW".to_string(),
                        ),
                        message: format!(
                            "Transaction L2 gas price {tx_l2_gas_price} is below the required \
                             threshold {threshold}.",
                        ),
                    });
                }
            }
            ValidResourceBounds::L1Gas(_) => {
                // No validation required for legacy transactions.
            }
        }
        Ok(())
    }

    pub fn perform_validations(&mut self, tx: AccountTransaction) -> StatefulValidatorResult<()> {
        // Deploy account transaction should be fully executed, since the constructor must run
        // before `__validate_deploy__`. The execution already includes all necessary validations,
        // so they are skipped here.
        // Declare transaction should also be fully executed - otherwise, if we only go through
        // the validate phase, we would miss the check that the class was not declared before.
        match tx.tx {
            ApiTransaction::DeployAccount(_) | ApiTransaction::Declare(_) => self.execute(tx),
            ApiTransaction::Invoke(_) => {
                let tx_context = Arc::new(self.tx_executor.block_context.to_tx_context(&tx));
                tx.perform_pre_validation_stage(self.state(), &tx_context)?;
                if !tx.execution_flags.validate {
                    return Ok(());
                }

                // `__validate__` call.
                let (_optional_call_info, actual_cost) = self.validate(&tx, tx_context.clone())?;

                // Post validations.
                PostValidationReport::verify(
                    &tx_context,
                    &actual_cost,
                    tx.execution_flags.charge_fee,
                )?;

                Ok(())
            }
        }
    }
