### Title
`PresignOutput` Lacks Consumed-State Enforcement, Enabling Presignature Reuse and Secret Key Extraction — (`File: src/ecdsa/ot_based_ecdsa/mod.rs`, `src/ecdsa/robust_ecdsa/mod.rs`)

---

### Summary

Both the OT-based and Robust ECDSA schemes expose `rerandomize_presign` as a function that accepts `&PresignOutput` (a shared reference). Because `PresignOutput` is never consumed or flagged as used, a malicious coordinator can call `rerandomize_presign` on the same presignature multiple times with different `RerandomizationArguments`, producing multiple valid `RerandomizedPresignOutput` values. Running the signing protocol with each of these causes honest participants to unknowingly sign different messages under the same underlying nonce, enabling full secret key extraction via standard ECDSA nonce-reuse algebra.

---

> "**Never reuse a presignature**, even across failed, aborted, or partially completed signing sessions." [6](#0-5)

---

### Impact Explanation

**Critical — Extraction of the aggregate secret key.**

ECDSA nonce reuse is a well-known total-break: given two signatures `(r, s₁)` and `(r, s₂)` over messages `h₁` and `h₂` produced with the same nonce `k`, the secret key is recovered as:

```
x = (s₁·h₂ − s₂·h₁) / (r·(s₂ − s₁))   (mod q)
```

When the same `PresignOutput` is rerandomized twice with different `RerandomizationArguments` (different `msg_hash` or `tweak`), the resulting `RerandomizedPresignOutput` values share the same underlying nonce `k` (scaled by the same `delta⁻¹`). The two signing sessions therefore produce two signatures with a multiplicatively related nonce, which is sufficient for the standard nonce-reuse key-recovery attack documented in the library's own security notes:

> "the resulting signatures use multiplicatively related nonces and the secret key can be recovered using standard ECDSA nonce-reuse attacks" [8](#0-7) 

---

### Likelihood Explanation

**High.** The malicious coordinator is an explicitly in-scope attacker role. The coordinator is the single party that:

1. Holds or receives each participant's `PresignOutput` share.
2. Calls `rerandomize_presign` locally before distributing `RerandomizedPresignOutput` to participants.
3. Initiates signing sessions.

Because `rerandomize_presign` takes `&PresignOutput`, the coordinator can call it twice — once for message `h₁` and once for message `h₂` — before running two signing sessions. Honest participants have no visibility into whether the presignature was already used; they simply receive a `RerandomizedPresignOutput` and contribute their share. No cryptographic primitive needs to be broken; the attack is purely a protocol-level replay enabled by the missing consumed-state check.

---

### Recommendation

Enforce single-use at the type level by consuming `PresignOutput` in `rerandomize_presign`:

```rust
pub fn rerandomize_presign(
    presignature: PresignOutput,   // ← take by value, consuming it
    args: &RerandomizationArguments,
) -> Result<Self, ProtocolError> { … }
```

Additionally:
- Remove the `Clone` derive from `PresignOutput` in both schemes so callers cannot trivially copy it before passing it in.
- Consider removing `Serialize`/`Deserialize` from `PresignOutput`, or at minimum documenting that deserialization re-creates a "fresh" (unconsumed) presignature and must only be done once per stored value. [1](#0-0) [3](#0-2)

### Proof of Concept

```
Setup:
  - N = 3 participants, t = 1 (MaxMalicious), coordinator = P1 (malicious)
  - Honest parties P2, P3 complete presigning, each producing their PresignOutput share.
  - P1 (coordinator) collects all shares and assembles the full PresignOutput.

Attack:
  1. P1 calls rerandomize_presign(&presign_output, &args_for_msg_h1)
       → produces RerandomizedPresignOutput_1 (nonce k·δ⁻¹, message h1)
  2. P1 calls rerandomize_presign(&presign_output, &args_for_msg_h2)   ← same presign_output
       → produces RerandomizedPresignOutput_2 (nonce k·δ⁻¹, message h2)
     (Both share the same underlying k because delta is the same for the same presignature R.)
  3. P1 runs signing session 1 with RerandomizedPresignOutput_1 and h1.
     P2, P3 contribute shares → coordinator assembles sig1 = (r, s1).
  4. P1 runs signing session 2 with RerandomizedPresignOutput_2 and h2.
     P2, P3 contribute shares → coordinator assembles sig2 = (r, s2).
     (Same r because big_r = δ·R is the same for both rerandomizations of the same R.)
  5. P1 recovers the master secret key:
       x = (s1·h2 − s2·h1) / (r·(s2 − s1))  mod q
```

No privileged access beyond the coordinator role is required. The `rerandomize_presign` function is called twice on the same `&PresignOutput` with no error, no state change, and no detection by honest participants. [9](#0-8) [10](#0-9)

    pub fn rerandomize_presign(
        presignature: &PresignOutput,
        args: &RerandomizationArguments,
    ) -> Result<Self, ProtocolError> {
        if presignature.big_r != args.big_r {
            return Err(ProtocolError::IncompatibleRerandomizationInputs);
        }
        let delta = args.derive_randomness()?;
        if delta.is_zero().into() {
            return Err(ProtocolError::ZeroScalar);
        }

        // cannot be zero due to the previous check
        let inv_delta = delta.invert().unwrap();

        // delta . R
        let rerandomized_big_r = presignature.big_r * delta;

        //  (sigma + tweak * k) * delta^{-1}
        let rerandomized_sigma =
            (presignature.sigma + args.tweak.value() * presignature.k) * inv_delta;

        // k * delta^{-1}
        let rerandomized_k = presignature.k * inv_delta;

        Ok(Self {
            big_r: rerandomized_big_r.into(),
            k: rerandomized_k,
            sigma: rerandomized_sigma,
        })
    }
