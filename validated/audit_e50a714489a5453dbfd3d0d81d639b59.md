### Title
Missing Caller Authorization in Bank Precompile `transfer` Allows Any Contract to Drain Arbitrary Holders' Native Token Balances — (`File: x/cronos/keeper/precompiles/bank.go`)

---

### Summary

The bank precompile's `transfer` method accepts the `sender` address as a caller-supplied argument and uses it directly as the `from` address in `bankKeeper.SendCoins`, without ever verifying that `contract.Caller()` equals the supplied `sender`. Any EVM contract can therefore move `evm/<caller_contract_address>`-denominated native tokens out of any holder's account to any recipient, without the holder's consent.

---

### Finding Description

In `x/cronos/keeper/precompiles/bank.go`, the `Run` function dispatches on method name. For `TransferMethodName`:

```go
// lines 175-192
sender    := args[0].(common.Address)   // ← taken from attacker-controlled calldata
recipient := args[1].(common.Address)
amount    := args[2].(*big.Int)
...
from  := sdk.AccAddress(sender.Bytes())
to    := sdk.AccAddress(recipient.Bytes())
...
denom := EVMDenom(contract.Caller())    // "evm/<calling_contract_address>"
...
bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt))
``` [1](#0-0) 

The `denom` is correctly scoped to `contract.Caller()`, so the calling contract can only move tokens of its own denomination. However, the `from` address is taken verbatim from the ABI-decoded input — there is no check of the form `contract.Caller() == sender`. The precompile never validates that the entity initiating the call has authority over the `sender` account.

Compare this with the `mint`/`burn` branch, where `contract.Caller()` is used as the denom authority and the `recipient`/`addr` argument is the target of the operation — a design that is consistent with the contract being the issuer. The `transfer` branch breaks that model by also accepting an arbitrary `from` address. [2](#0-1) [3](#0-2) 

---

### Impact Explanation

**Critical — Unauthorized transfer of precompile-controlled assets.**

Attack path:

1. Attacker deploys contract `A` at address `0xATTACKER`.
2. Contract `A` calls `bankPrecompile.mint(victim, N)` → victim now holds `N` units of `evm/0xATTACKER`.
3. Contract `A` calls `bankPrecompile.transfer(victim, attacker_eoa, N)` → `SendCoins` moves `N evm/0xATTACKER` from `victim` to `attacker_eoa` with no authorization from `victim`.

The victim's native-layer balance is drained without any EVM-level approval (`approve`/`transferFrom`) or Cosmos-level authorization. The attacker does not need any privileged role; a standard unprivileged EVM transaction is sufficient.

The same flaw applies to `burn`: `args[0]` is the address whose tokens are burned, and there is no check that `contract.Caller()` owns or has been approved over that address. [4](#0-3) 

---

### Likelihood Explanation

Any contract that calls the bank precompile can exploit this immediately. The precompile is registered at a fixed address (`0x0000...0064`) and is callable by any EVM transaction. No special network position, validator key, or governance permission is required. The only precondition is that the victim holds a balance of the attacker's token denomination, which the attacker can arrange by minting to the victim first. [5](#0-4) 

---

### Recommendation

Add a caller-identity check at the top of the `TransferMethodName` case before executing `SendCoins`:

```go
if contract.Caller() != sender {
    return nil, errors.New("transfer: caller is not the sender")
}
```

This mirrors the standard ERC-20 security model where only the token holder (or an approved spender) can initiate a transfer. Alternatively, redesign the `transfer` method to only allow the calling contract to move tokens from its own contract address, removing the `sender` argument entirely and using `contract.Caller()` as `from`. [6](#0-5)

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

interface IBankPrecompile {
    function mint(address recipient, uint256 amount) external payable returns (bool);
    function transfer(address sender, address recipient, uint256 amount) external payable returns (bool);
}

contract BankDrainer {
    IBankPrecompile constant BANK = IBankPrecompile(address(0x0000000000000000000000000000000000000064));

    // Step 1: mint tokens to victim so they hold evm/<this_contract> balance
    function seedVictim(address victim, uint256 amount) external {
        BANK.mint(victim, amount);
    }

    // Step 2: drain victim's evm/<this_contract> balance to attacker
    // No approval from victim required
    function drain(address victim, address attacker, uint256 amount) external {
        BANK.transfer(victim, attacker, amount);
    }
}
```

Deploy `BankDrainer`, call `seedVictim(victim, 1000)`, then call `drain(victim, attacker, 1000)`. The victim's `evm/<BankDrainer_address>` native balance is transferred to the attacker without any consent. The `bankKeeper.SendCoins` call at line 192 succeeds because the precompile never validates that `contract.Caller()` is authorized to spend from `victim`. [7](#0-6)
