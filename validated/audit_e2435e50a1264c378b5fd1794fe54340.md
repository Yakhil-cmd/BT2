### Title
Unprivileged Caller Can Permanently Freeze Consensus Rewards via `update_rewards` with `disable_rewards=true` - (File: src/staking/staking.cairo)

### Summary
The `update_rewards` function in the `Staking` contract is publicly callable with no access control. It accepts a `disable_rewards: bool` parameter that, when `true`, advances the global `last_reward_block` state variable to the current block without distributing any rewards. Because `last_reward_block` gates all reward distribution (only one call per block is allowed), any unprivileged caller can permanently suppress consensus rewards for all stakers by calling `update_rewards(any_active_staker, true)` once per block.

### Finding Description

`update_rewards` is exposed as part of `IStakingRewardsManager` with no caller restriction beyond `general_prerequisites()`, which only checks that the contract is not paused and the caller is non-zero. [1](#0-0) 

The function first writes `last_reward_block` unconditionally, then branches on `disable_rewards`: [2](#0-1) 

When `disable_rewards = true`, the function returns immediately after writing `last_reward_block`, skipping the entire reward calculation and distribution path: [3](#0-2) 

The `last_reward_block` variable is a single global slot. The guard at the top of the function enforces that only one call per block can succeed: [4](#0-3) 

Once an attacker's call with `disable_rewards = true` succeeds in a given block, every subsequent legitimate call in that same block reverts with `REWARDS_ALREADY_UPDATED`. The attacker only needs to supply any currently-active staker address (public information, emitted in `NewStaker` events and stored in the `stakers` vector). [5](#0-4) 

### Impact Explanation

In the consensus rewards phase (`!is_pre_consensus()`), `update_rewards` is the sole mechanism by which per-block STRK and BTC rewards are calculated and credited to stakers and their delegation pools. If an attacker calls `update_rewards(active_staker, true)` in every block, no staker ever receives consensus rewards. The `unclaimed_rewards_own` field in each staker's info is never incremented, and pool `cumulative_rewards_trace` entries are never appended. Stakers and delegators cannot claim rewards that were never accrued. This constitutes **permanent freezing of unclaimed yield** for the entire protocol. [6](#0-5) 

### Likelihood Explanation

The attacker requires only:
1. A non-zero Starknet account (trivially available).
2. The address of any one active staker (publicly observable from `NewStaker` events or the `stakers` storage vector).
3. The ability to submit one transaction per block — no privileged role, no leaked key, no bridge compromise.

On Starknet the sequencer orders transactions within a block. An attacker who submits a `update_rewards(..., true)` transaction at the start of each block will consistently land before the legitimate node operator's call, especially since the attacker can use a higher fee or simply race. The cost is gas per block; the damage is protocol-wide reward suppression.

### Recommendation

Restrict `update_rewards` to a trusted caller — either the staker themselves, their operational address, or a designated rewards-manager role. Alternatively, remove the `disable_rewards` parameter entirely and handle the pre-consensus / zero-balance cases with explicit early-return guards that do **not** advance `last_reward_block`, so that a legitimate caller can still succeed in the same block. [7](#0-6) 

### Proof of Concept

```
// Attacker script — run once per block, indefinitely
// 1. Read any active staker address from the NewStaker event log.
let victim_staker: ContractAddress = <any active staker>;

// 2. Call update_rewards with disable_rewards = true.
//    No special role required; general_prerequisites() only checks
//    is_paused == false and caller != 0.
staking_contract.update_rewards(victim_staker, disable_rewards: true);

// Effect:
//   - last_reward_block is now set to the current block number.
//   - No rewards are calculated or distributed.
//   - Any subsequent update_rewards call in this block reverts with
//     REWARDS_ALREADY_UPDATED.
//   - Stakers' unclaimed_rewards_own is never incremented.
//   - Pool cumulative_rewards_trace receives no new entries.
//   - claim_rewards() and pool claim_rewards() return 0 forever.
``` [2](#0-1)
