### Title
Signature Re-Use Across COA Addresses in `validateCOAOwnershipProof` - (File: `fvm/evm/stdlib/contract.cdc`)

### Summary

`EVM.validateCOAOwnershipProof` in the Flow EVM system contract does not enforce that the `signedData` includes the `evmAddress` being proven. A valid signature produced by a Cadence account for one purpose (or for one COA address) can be replayed against a different COA address owned by the same Cadence account, or against any off-chain system that calls `validateCOAOwnershipProof` directly with attacker-controlled `signedData` and `evmAddress`. The code itself acknowledges this gap in a comment.

### Finding Description

`EVM.validateCOAOwnershipProof` accepts six caller-controlled arguments: `address`, `path`, `signedData`, `keyIndices`, `signatures`, and `evmAddress`. The function verifies that the provided signatures are valid over `signedData` using the account's keys, and then checks that the COA resource at `path` has an EVM address matching `evmAddress`. However, it never checks that `signedData` encodes or commits to `evmAddress`.

The code comment at line 1003–1009 explicitly acknowledges this:

> "Note: this function does not enforce that `signedData` includes `evmAddress`. In principle, a signature produced for one purpose could be replayed here against a different COA owned by the same Cadence account."

The function is `access(all)` — callable by any Cadence script or transaction — and all six arguments are fully attacker-controlled. An attacker who observes a valid `(signedData, signatures)` pair produced by a victim's Cadence account for any purpose (e.g., a prior ownership proof for COA-A) can replay those same bytes with a different `evmAddress` (COA-B, also owned by the same Cadence account) and a different `path` pointing to COA-B's capability. The function will return `isValid: true` for COA-B even though the victim never signed anything related to COA-B. [1](#0-0) [2](#0-1) 

### Impact Explanation

Any off-chain or on-chain system that calls `EVM.validateCOAOwnershipProof` directly (not through the EVM precompile) and relies on the result to authorize an action tied to a specific COA address can be deceived. An attacker can present a valid proof for COA-A and have it accepted as a valid proof for COA-B, as long as both COAs are owned by the same Cadence account. This constitutes a **transaction authorization bypass**: the attacker proves ownership of an EVM address they did not sign for, enabling unauthorized access to any system that trusts the proof result.

The EVM-side precompile `verifyCOAOwnershipProof` is not directly vulnerable because it always passes `address(this)` (the calling contract's own address) as the `evmAddress` argument, so the caller cannot substitute a different EVM address there. However, the Cadence-level `validateCOAOwnershipProof` function is `access(all)` and callable directly from any transaction or script with fully attacker-controlled arguments. [3](#0-2) [4](#0-3) 

### Likelihood Explanation

The function is `access(all)` and reachable by any unprivileged transaction sender or script caller on the network. The attacker only needs to observe a prior valid `(signedData, signatures)` pair from the victim's account — which may be publicly visible on-chain or in transaction history — and then replay it with a different `evmAddress` and `path`. No privileged access, leaked keys, or social engineering is required. The precondition (victim must own more than one COA, or the attacker must find a context where the same signature is accepted for a different purpose) limits the immediate practical scope, but the root cause is a structural absence of binding between the signed data and the specific EVM address being proven. [2](#0-1) 

### Recommendation

Enforce that `signedData` commits to `evmAddress`. The simplest fix is to require that `signedData` is the concatenation (or a hash) of `evmAddress` and any application-specific data, and verify this inside `validateCOAOwnershipProof` before accepting the proof. Alternatively, prepend `evmAddress` to the data that is verified:

```cadence
// Require signedData to begin with evmAddress bytes
let evmAddressBytes = evmAddress.toConstantSized<[UInt8; 20]>()!
for i, b in evmAddressBytes {
    if signedData[i] != b {
        return ValidationResult(isValid: false, problem: "signedData must begin with evmAddress")
    }
}
```

This mirrors the fix described in the external report: include the binding identity (here, `evmAddress`) in the signed message so that a signature cannot be replayed for a different target address. [1](#0-0) 

### Proof of Concept

1. Victim account `0xALICE` owns two COAs: COA-A at `/public/coaA` and COA-B at `/public/coaB`.
2. Alice previously produced a valid proof for COA-A: `signedData = <some_bytes>`, `signatures = <alice_sigs_over_some_bytes>`.
3. Attacker calls `EVM.validateCOAOwnershipProof` directly from a Cadence script:
   ```cadence
   EVM.validateCOAOwnershipProof(
       address: 0xALICE,
       path: /public/coaB,          // ← attacker substitutes COA-B's path
       signedData: <some_bytes>,    // ← replayed from Alice's prior proof for COA-A
       keyIndices: <alice_key_indices>,
       signatures: <alice_sigs_over_some_bytes>,
       evmAddress: <COA-B address>  // ← attacker substitutes COA-B's EVM address
   )
   ```
4. The function verifies the signatures over `signedData` (valid, because Alice signed those bytes), then checks that COA-B's EVM address matches `evmAddress` (also valid, because the attacker supplied COA-B's real address). It returns `isValid: true`.
5. Any off-chain or on-chain system trusting this result now believes Alice authorized COA-B, even though she only ever signed data for COA-A. [5](#0-4) [6](#0-5)
