### Title
Unauthorized Global Contract Namespace Overwrite via Cross-Account `DeployGlobalContract(AccountId)` Receipt — (`runtime/runtime/src/receipt_manager.rs`, `runtime/runtime/src/global_contracts.rs`)

### Summary

A contract (account A) can emit a `DeployGlobalContract { deploy_mode: AccountId }` action receipt targeting any arbitrary account B as `receiver_id`. When that receipt executes, the runtime uses the receipt's `receiver_id` (B) — not the originating contract (A) — as the `GlobalContractIdentifier::AccountId` key. No authorization check verifies that the deploying predecessor owns or controls the receiver account. The result is that attacker-controlled code is written to `TrieKey::GlobalContractCode { identifier: AccountId(B) }`, corrupting the global contract namespace for B's account ID. Any account that subsequently calls `UseGlobalContract(AccountId(B))` executes the attacker's code.

### Finding Description

**Step 1 — Host function, no ownership check.**

A contract calls `promise_batch_action_deploy_global_contract_by_account_id` with a `promise_idx` pointing to a receipt whose `receiver_id` is account B (any arbitrary account). The host function only validates code size and gas: [1](#0-0) 

No check is made that the receipt's receiver matches the calling contract's own account ID.

**Step 2 — `ReceiptManager::append_action_deploy_global_contract` appends unconditionally.** [2](#0-1) 

The action is appended to the receipt targeting B with no authorization guard.

**Step 3 — Action validation only checks code size.**

`validate_deploy_global_contract_action` performs no ownership check: [3](#0-2) 

**Step 4 — Execution uses the receipt's `receiver_id` as the identifier.**

When the receipt executes on account B, `action_deploy_global_contract` is called with `account_id = receipt.receiver_id() = B`: [4](#0-3) 

`initiate_distribution` then uses `account_id` (= B) as the `GlobalContractIdentifier::AccountId`: [5](#0-4) 

**Step 5 — Attacker's code is written to B's global contract trie key.**

`apply_distribution_current_shard` writes the code to `TrieKey::GlobalContractCode { identifier: AccountId(B) }`: [6](#0-5) 

**Step 6 — Any account using `UseGlobalContract(AccountId(B))` now executes attacker's code.**

`use_global_contract` looks up the trie key and sets the account's contract to `AccountContract::GlobalByAccount(B)`: [7](#0-6) 

The design intent (documented in `GlobalContractDeployMode::AccountId`) is that only the "owner" account deploys under their own ID: [8](#0-7) 

But this ownership invariant is never enforced in code.

### Impact Explanation

- **Exact corrupted value**: `TrieKey::GlobalContractCode { identifier: AccountId(B) }` is written with attacker-supplied bytecode instead of B's code (or instead of not existing).
- **Namespace isolation broken**: The global contract namespace is supposed to be keyed by the deploying account's own ID. An attacker can squatter any account's global contract slot.
- **Execution hijack**: All accounts that have called or will call `UseGlobalContract(AccountId(B))` will execute the attacker's code when their contract is invoked.
- **Balance drain**: B's balance is debited for storage costs without B's consent.
- **Nonce mechanism does not protect**: The nonce only prevents stale re-delivery of the same distribution receipt; it does not prevent a higher-nonce attacker from overwriting B's slot. [9](#0-8) 

### Likelihood Explanation

Any deployed contract can issue cross-account receipts to arbitrary accounts. The `promise_batch_create` host function accepts any valid account ID as receiver. The attack requires only that B has sufficient balance to cover storage costs (which is true for any funded account). The attacker pays gas for the function call; B pays the storage cost involuntarily.

### Recommendation

In `action_deploy_global_contract` (or in `validate_action_with_mode` for `DeployGlobalContract`), add an authorization check: when `deploy_mode == AccountId`, require that `receipt.predecessor_id() == receipt.receiver_id()` (i.e., the account is deploying to its own global contract slot). This mirrors the ownership model described in the documentation and enforces namespace isolation.

Alternatively, enforce this at the host-function level in `promise_batch_action_deploy_global_contract_by_account_id`: reject the call if the receipt's `receiver_id` differs from the calling contract's own `account_id` (available in `ctx.context.current_account_id`).

### Proof of Concept

```
1. Deploy contract A (attacker contract) with a method that does:
   - promise_batch_create("victim.near")  // target account B
   - promise_batch_action_deploy_global_contract_by_account_id(
       promise_idx, malicious_wasm_code)

2. Account B previously deployed a legitimate global contract:
   - DeployGlobalContract { code: legit_code, deploy_mode: AccountId }
   - TrieKey::GlobalContractCode { identifier: AccountId("victim.near") } = legit_code

3. Call contract A's attack method.
   - Receipt targeting "victim.near" with DeployGlobalContract(AccountId) is created.
   - Receipt executes on "victim.near"; initiate_distribution uses account_id="victim.near".
   - TrieKey::GlobalContractCode { identifier: AccountId("victim.near") } = malicious_code.
   - victim.near's balance is debited for storage.

4. Account C (which previously called UseGlobalContract(AccountId("victim.near")))
   now executes malicious_code when its contract methods are invoked.
```

The test-loop test framework already has all the primitives needed to write this as a concrete integration test, following the pattern in `test-loop-tests/src/tests/global_contracts.rs`. [10](#0-9)

fn check_and_update_nonce(
    global_contract_data: &GlobalContractDistributionReceipt,
    identifier: &GlobalContractCodeIdentifier,
    state_update: &mut TrieUpdate,
) -> Result<bool, RuntimeError> {
    let nonce_key = TrieKey::GlobalContractNonce { identifier: identifier.clone() };
    let stored_nonce = get_nonce(state_update, &nonce_key)?;
    let incoming_nonce = global_contract_data.nonce();

    // Allow the same nonce since the nonce is updated immediately when
    // initiating distribution to prevent multiple distributions with the same
    // nonce from being initiated.
    if incoming_nonce < stored_nonce {
        return Ok(false);
    }

    set_nonce(state_update, nonce_key, incoming_nonce);
    Ok(true)
}

#[test]
fn test_global_contract_update() {
    let mut env = GlobalContractsTestEnv::setup(Balance::from_near(1000));
    let use_accounts = [env.account_shard_0.clone(), env.account_shard_1.clone()];

    env.deploy_trivial_global_contract(GlobalContractDeployMode::AccountId);

    for account in &use_accounts {
        env.use_global_contract(
            account,
            GlobalContractIdentifier::AccountId(env.deploy_account.clone()),
        );

        // Currently deployed trivial contract doesn't have any methods,
        // so we expect any function call to fail with MethodNotFound error
        let call_tx = env.call_global_contract_tx(account.clone(), account.clone());
        let call_outcome = env.execute_tx(call_tx);
        assert_matches!(
            call_outcome.status,
            FinalExecutionStatus::Failure(TxExecutionError::ActionError(ActionError {
                kind: ActionErrorKind::FunctionCallError(FunctionCallError::MethodResolveError(
                    MethodResolveError::MethodNotFound
                )),
                index: _
            }))
        );
    }

    env.deploy_global_contract(GlobalContractDeployMode::AccountId);

    for account in &use_accounts {
        // Function call should be successful after deploying rs contract
        // containing the function we call here
        env.assert_call_global_contract_success(account.clone(), account.clone());
    }
}
