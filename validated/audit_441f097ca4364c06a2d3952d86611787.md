### Title
Unbounded `MessageBuffer` HashMap Enables Memory Exhaustion via Crafted Message Headers — (`File: src/protocol/internal.rs`)

---

### Summary
The `MessageBuffer` in `src/protocol/internal.rs` uses an unbounded `HashMap<MessageHeader, SubMessageQueue>` to buffer incoming messages. Any message sender can craft messages with arbitrary `MessageHeader` values (unique channel tags or waitpoint values), causing the HashMap to grow without bound. Messages buffered for headers the protocol never polls are never evicted, leading to unbounded memory growth. The codebase itself contains a test that explicitly demonstrates this attack path.

---

### Finding Description

`MessageBuffer` stores incoming messages keyed by `MessageHeader`, which encodes a 32-byte `ChannelTag` and an 8-byte `Waitpoint` (u64): [1](#0-0) 

The `push` method unconditionally inserts a new `SubMessageQueue` for any previously-unseen header via `entry(header).or_default()`, with no cap on the number of entries: [2](#0-1) 

Each `SubMessageQueue` allocates an `UnboundedSender`/`UnboundedReceiver` pair from `futures::channel::mpsc::unbounded()`: [3](#0-2) 

The public entry point `Protocol::message` feeds directly into `Comms::push_message`, which parses the header from raw bytes and inserts it into the buffer with no validation against expected waitpoints or participant identity: [4](#0-3) [5](#0-4) 

Messages buffered for headers the honest protocol logic never polls are never removed. There is no eviction, no size cap, and no timeout. The `Waitpoint` field alone is a `u64`, giving an attacker 2^64 unique keys per channel tag.

The developers have already documented this exact attack in a test titled `attacker_can_fill_message_buffer_with_unused_waitpoints`, which asserts that 512 unique HashMap entries are created by injecting messages for waitpoints the honest code never reads: [6](#0-5) 

---

### Impact Explanation

A malicious participant in any threshold protocol (DKG, presign, sign, reshare, refresh, CKD) can call `Protocol::message()` with messages whose headers encode arbitrary, never-polled waitpoints. Each unique header permanently allocates a new `SubMessageQueue` (unbounded MPSC channel) in the HashMap. With no eviction, the buffer grows proportionally to the number of injected messages. Sustained flooding causes OOM on the honest node, permanently denying its participation in signing, key generation, or any other protocol operation. This matches the **Medium: Griefing or resource-exhaustion** impact category.

---

### Likelihood Explanation

Any participant in a threshold protocol session can deliver messages to another participant's `Protocol::message()` interface — this is the normal, documented message-delivery API. No special privilege beyond protocol participation is required. The attack requires only crafting messages with a valid 40-byte header prefix (32-byte channel tag + 8-byte waitpoint) pointing to a waitpoint the honest code will never consume. The `Waitpoint` field is a plain `u64` with no authentication or range restriction. The developers' own test confirms the attack is trivially reproducible.

---

### Recommendation

1. **Bound the HashMap**: Enforce a maximum number of entries in `MessageBuffer`. Reject or drop messages that would exceed the cap.
2. **Validate headers on ingress**: In `Comms::push_message`, reject messages whose `MessageHeader` does not correspond to a waitpoint the current protocol state expects. This requires the executor to register expected waitpoints before polling.
3. **Evict stale entries**: After a protocol round completes, remove HashMap entries for waitpoints that are no longer relevant.
4. **Per-sender rate limiting**: Track the number of buffered messages per `Participant` and drop excess messages from any single sender.

---

### Proof of Concept

```rust
// Attacker is a participant in any threshold protocol session.
// They call Protocol::message() with messages for unused waitpoints.

let comms = Comms::new();
let attacker = Participant::from(99_u32);

for i in 0u64..1_000_000 {
    // Craft a message header with a unique, never-polled waitpoint.
    let header = MessageHeader::new(ChannelTag::root_shared()).with_waitpoint(1_000_000 + i);
    let mut message = header.to_bytes().to_vec();
    // Append minimal payload (any bytes).
    message.extend_from_slice(&i.to_le_bytes());

    // Deliver via the public Protocol::message() API.
    // Each call inserts a new SubMessageQueue (unbounded MPSC channel) into the HashMap.
    comms.push_message(attacker, message);
}
// HashMap now contains 1,000,000 entries, none ever consumed.
// Memory grows without bound; no cleanup occurs.
```

This is structurally identical to the developers' own test at `src/protocol/internal.rs:532–554`, scaled to an unbounded loop.

struct MessageBuffer {
    messages: Arc<std::sync::Mutex<HashMap<MessageHeader, SubMessageQueue>>>,
}

impl MessageBuffer {
    fn new() -> Self {
        Self {
            messages: Arc::new(std::sync::Mutex::new(HashMap::new())),
        }
    }

    /// Push a message into this buffer.
    ///
    /// We also need the header for the message, and the participant who sent it.
    fn push(&self, header: MessageHeader, from: Participant, message: MessageData) {
        let mut messages_lock = self.messages.lock().expect("lock should not fail");
        messages_lock.entry(header).or_default().send(from, message);
    }
