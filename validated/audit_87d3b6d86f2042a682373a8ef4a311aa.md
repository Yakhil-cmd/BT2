### Title
ETH Permanently Stuck in `RewardSupplier` When `tick()` Called With Value But No Minting Required — (File: `L1/starkware/solidity/stake/RewardSupplier.sol`)

---

### Summary

`RewardSupplier.tick()` is `payable` and uses `msg.value` to pay L1→L2 messaging fees, but only forwards that value inside the `if (amountToMint > 0)` branch. When called with ETH while no minting is required, the entire `msg.value` is silently retained by the contract with no recovery path.

---

### Finding Description

`tick()` is declared `payable` at line 107 of `RewardSupplier.sol`. Inside, it splits `msg.value` between two inner calls only when `amountToMint > 0`:

```solidity
function tick() external payable {
    (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

    if (amountToMint > 0) {
        // ...
        uint256 msgFee = msg.value / 2;
        bridge().depositWithMessage{value: msgFee}(...);   // forwards half
        msgFee = msg.value - msgFee;
        messagingContract().sendMessageToL2{value: msgFee}(...); // forwards rest
    }
    // ← if amountToMint == 0, msg.value is never forwarded or refunded
}
``` [1](#0-0) 

When `requiredMinting()` returns `(0, 0)` — the normal state between minting cycles when no L2 mint-request messages are pending — the entire `if` block is skipped and the function returns. Any ETH sent by the caller is absorbed by the contract. The `RewardSupplier` contract contains no `receive()` fallback, no ETH withdrawal function, and no refund logic, making the stuck ETH unrecoverable without a privileged contract upgrade. [2](#0-1) 

---

### Impact Explanation

ETH sent to `tick()` when no minting is required is permanently frozen inside `RewardSupplier`. The contract has no mechanism to sweep or return ETH. Recovery requires a proxy upgrade, which is a privileged governance action. This maps to **permanent freezing of funds** for any caller who sends ETH during a no-mint tick.

---

### Likelihood Explanation

`tick()` has no access control — any address can call it. The no-mint path (`amountToMint == 0`) is the **common case**: it occurs whenever the L2 reward supplier has not yet emitted a mint-request message, or between minting cycles. A caller who estimates that minting will be needed (e.g., based on a stale off-chain view) and sends ETH, but whose transaction lands after another caller already consumed the pending messages, will lose their ETH. This is a realistic race condition on a public, permissionless function.

---

### Recommendation

Add a refund or guard for the no-mint path:

```solidity
function tick() external payable {
    (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();
    if (amountToMint == 0) {
        require(msg.value == 0, "NO_MINT_NO_FEE_NEEDED");
        return;
    }
    // ... existing forwarding logic
}
```

Alternatively, refund `msg.value` to `msg.sender` when the early-return path is taken.

---

### Proof of Concept

1. No L2 mint-request messages are pending (normal inter-epoch state).
2. An off-chain keeper calls `tick{value: 0.01 ether}()` to cover anticipated messaging fees.
3. `requiredMinting()` returns `(0, 0)` because `messagingContract().l2ToL1Messages(msgHash) == 0`.
4. The `if (amountToMint > 0)` block is skipped entirely.
5. `tick()` returns. `msg.value` (0.01 ETH) is now held by `RewardSupplier` with no withdrawal path.
6. The ETH is permanently frozen; recovery requires a governance-controlled proxy upgrade. [1](#0-0)

    function tick() external payable {
        // Check if minting is required, and how much.
        (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

        if (amountToMint > 0) {
            // Prepare the L2->L1 mintRequest message for consumption.
            uint256[] memory messagePayload = new uint256[](1);
            messagePayload[0] = TOKENS_PER_MINT_REQUEST;

            // Consume the mintRequest messages.
            for (uint256 i = 0; i < numMsgsToConsume; i++) {
                messagingContract().consumeMessageFromL2(mintRequestSource(), messagePayload);
            }

            // Reuest minting of the requested amount from the mint manager.
            mintManager().mintRequest(token(), amountToMint);

            // Deposit the minted amount onto the bridge to the credit of `mintDestination`.
            uint256 msgFee = msg.value / 2;
            bridge().depositWithMessage{value: msgFee}(
                token(),
                amountToMint,
                mintDestination(),
                new uint256[](0)
            );
            emit ConsumedL2MintRequests(numMsgsToConsume, amountToMint);

            // Send a totalSupply update to L2MintCurve.
            msgFee = msg.value - msgFee;
            messagePayload[0] = IERC20(token()).totalSupply();
            messagingContract().sendMessageToL2{value: msgFee}(
                mintingCurve(),
                UPDATE_TOTAL_SUPPLY_SELECTOR,
                messagePayload
            );
        }
    }
