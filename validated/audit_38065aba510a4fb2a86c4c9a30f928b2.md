### Title
Burned ERC-20/ETH Tokens Are Permanently Lost When `exit_to_near` NEAR-Side Transfer Fails in Production Build - (File: `engine-precompiles/src/native.rs`)

---

### Summary

The production Aurora Engine binary is built without the `error_refund` feature flag. When a user calls the `exit_to_near` precompile to bridge ERC-20 tokens (or ETH) from Aurora to NEAR, the tokens are burned on the EVM side first, then a NEAR `ft_transfer` promise is dispatched. If that NEAR-side transfer fails for any reason, the `refund` field in the callback args is hardcoded to `None` in production, so no re-mint or ETH refund is ever issued. The burned tokens are permanently unrecoverable.

---

### Finding Description

The `exit_to_near` precompile constructs a `ExitToNearPrecompileCallbackArgs` struct that carries an optional `refund` field. This field is populated only when the `error_refund` Cargo feature is enabled: [1](#0-0) 

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,   // ← always None in production
    transfer_near: transfer_near_args,
};
```

The production build target is defined in `Makefile.toml`: [2](#0-1) 

```toml
CARGO_FEATURES_BUILD = "contract"
```

The `error_refund` feature is **not** included in `CARGO_FEATURES_BUILD`. It is only referenced as a separate optional clippy target (`clippy-contract-refund`) and is never part of the standard `build` or `build-docker` tasks. [3](#0-2) 

The callback handler `exit_to_near_precompile_callback` in the engine contract checks the promise result and the refund field: [4](#0-3) 

```rust
let maybe_result = if let Some(PromiseResult::Successful(_)) = handler.promise_result(0) {
    // success path
    None
} else if let Some(args) = args.refund {
    // Exit call failed; need to refund tokens
    let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;
    ...
} else {
    None   // ← production always reaches here on failure: no refund, no error
};
```

When `ft_transfer` fails and `args.refund` is `None`, the callback silently returns `Ok(None)`. The ERC-20 tokens that were burned (or the ETH that was transferred to the precompile address) are never restored.

The refund logic that *would* re-mint burned ERC-20 tokens or transfer ETH back exists in `engine/src/engine.rs`: [5](#0-4) 

But it is unreachable in production because `args.refund` is always `None`.

The test suite explicitly documents this behavior: [6](#0-5) 

```rust
#[cfg(feature = "error_refund")]
let balance = FT_TRANSFER_AMOUNT.into();
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
```

And for ETH: [7](#0-6) 

```rust
#[cfg(feature = "error_refund")]
let expected_balance = Wei::new_u64(INITIAL_ETH_BALANCE);
// If the refund feature is not enabled, then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let expected_balance = Wei::new_u64(INITIAL_ETH_BALANCE - ETH_EXIT_AMOUNT);
```

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

Any ERC-20 tokens burned during a failed `exit_to_near` call are permanently destroyed with no recovery path. The same applies to ETH sent to the `exit_to_near` precompile address. There is no admin escape hatch, no retry mechanism, and no way for the user to reclaim their assets after the NEAR-side promise fails.

---

### Likelihood Explanation

**High.** NEAR NEP-141 `ft_transfer` calls fail in several common, user-triggerable scenarios:

1. The recipient NEAR account is not registered with the NEP-141 token contract (storage deposit not paid). This is the most common mistake — a user specifying a NEAR account that has never interacted with that token.
2. The recipient NEAR account does not exist.
3. The NEP-141 contract is paused (Aurora's own connector supports pausing via `pause_ft_transfers`).
4. The attached 1 yoctoNEAR is insufficient (edge cases in some NEP-141 implementations).

Any unprivileged EVM user who calls `exit_to_near` with a misconfigured recipient triggers this path. No special access is required.

---

### Recommendation

Enable the `error_refund` feature in the production build by adding it to `CARGO_FEATURES_BUILD`:

```toml
CARGO_FEATURES_BUILD = "contract,error_refund"
```

This ensures the `refund` field is populated in `ExitToNearPrecompileCallbackArgs` and the `exit_to_near_precompile_callback` can re-mint burned ERC-20 tokens or return ETH to the sender when the NEAR-side transfer fails. [8](#0-7) 

---

### Proof of Concept

1. User holds 100 units of a bridged ERC-20 token on Aurora (backed by a NEP-141 on NEAR).
2. User calls the `exit_to_near` precompile specifying a NEAR recipient account that is not registered with the NEP-141 contract.
3. The ERC-20 contract burns 100 tokens and the precompile emits a `ft_transfer` promise to the NEP-141 contract.
4. The NEP-141 `ft_transfer` call fails because the recipient is unregistered.
5. `exit_to_near_precompile_callback` is invoked. `handler.promise_result(0)` is not `Successful`. `args.refund` is `None` (production build). The callback returns `Ok(None)`.
6. The 100 ERC-20 tokens are permanently gone. The user has no recourse. [1](#0-0) [9](#0-8) [2](#0-1)

pub fn refund_on_error<I: IO + Copy, E: Env, P: PromiseHandler>(
    io: I,
    env: &E,
    state: EngineState,
    args: &RefundCallArgs,
    handler: &mut P,
) -> EngineResult<SubmitResult> {
    let current_account_id = env.current_account_id();
    if let Some(erc20_address) = args.erc20_address {
        // ERC-20 exit; re-mint burned tokens
        let erc20_admin_address = current_address(&current_account_id);
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, erc20_admin_address, current_account_id, io, env);

        let refund_address = args.recipient_address;
        let amount = U256::from_big_endian(&args.amount);
        let input = setup_refund_on_error_input(amount, refund_address);

        engine.call(
            &erc20_admin_address,
            &erc20_address,
            Wei::zero(),
            input,
            u64::MAX,
            Vec::new(),
            Vec::new(),
            handler,
        )
    } else {
        // ETH exit; transfer ETH back from precompile address
        let exit_address = exit_to_near::ADDRESS;
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, exit_address, current_account_id, io, env);
        let refund_address = args.recipient_address;
        let amount = Wei::new(U256::from_big_endian(&args.amount));
        engine.call(
            &exit_address,
            &refund_address,
            amount,
            Vec::new(),
            u64::MAX,
            vec![
                (exit_address.raw(), Vec::new()),
                (refund_address.raw(), Vec::new()),
            ],
            Vec::new(),
            handler,
        )
    }
