### Title
Gateway `skip_stateful_validations` Bypasses `__validate__` for Invoke Transactions via Non-Deploy-Account Mempool Entry - (File: crates/apollo_gateway/src/stateful_transaction_validator.rs)

### Summary
The `skip_stateful_validations` function uses `account_tx_in_pool_or_recent_block` to decide whether to skip the `__validate__` entry point for an invoke transaction with nonce=1. This helper returns `true` for **any** transaction from the account in the mempool, not exclusively a `deploy_account` transaction. An attacker who controls a freshly-deployed account (committed nonce=0) can submit one valid invoke (nonce=0) to seed the mempool, then submit a second invoke (nonce=1) with an invalid signature or forbidden calldata; the gateway accepts the second transaction without ever calling `__validate__`.

### Finding Description

`skip_stateful_validations` is intended to support the deploy-account + invoke UX: when a user sends a `deploy_account` and an `invoke` (nonce=1) simultaneously, the account does not yet exist in committed state, so `__validate__` would fail. The gateway skips it when it can confirm a `deploy_account` is pending.

The guard used is:

`account_tx_in_pool_or_recent_block` is implemented as:

```rust
// crates/apollo_mempool/src/mempool.rs  lines 697-700
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [2](#0-1) 

`tx_pool.contains_account` returns `true` for **any** transaction type (invoke, declare, deploy_account) from that address. The code comment claims this is sufficient because "it means that either it has a deploy_account transaction or transactions with future nonces that passed validations," but this reasoning is circular: the very skip mechanism being guarded is what allows future-nonce transactions to bypass validation.

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` sets `execution_flags.validate = false`:

```rust
// lines 311-312

    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [3](#0-2) 

Inside `StatefulValidator::perform_validations`, when `validate=false` the function returns `Ok(())` immediately after `perform_pre_validation_stage`, never calling `__validate__`:

```rust
// crates/blockifier/src/blockifier/stateful_validator.rs  lines 78-81
tx.perform_pre_validation_stage(self.state(), &tx_context)?;
if !tx.execution_flags.validate {
    return Ok(());
}
``` [4](#0-3) 

`perform_pre_validation_stage` checks nonce, fee bounds, and proof facts, but **not the signature**. The signature is only verified inside `__validate__`. [5](#0-4) 

### Impact Explanation

The gateway accepts an invoke transaction with nonce=1 whose signature is invalid or whose calldata would be rejected by the account's custom `__validate__` logic. This violates the admission invariant: **High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

Concrete consequences:
- An account with a custom `__validate__` (multisig, calldata whitelist, time-lock) can have its second transaction bypass that logic entirely at the gateway layer.
- The invalid transaction occupies a mempool slot and will be forwarded to the batcher, which will run `__validate__` and revert it, charging the fee. The attacker loses the fee but the gateway's security boundary is broken.

### Likelihood Explanation

The preconditions are common:
1. A deployed account whose committed nonce is still 0 (any freshly deployed account before its first committed transaction).
2. The ability to submit one valid invoke transaction with nonce=0 (requires the account's private key or a valid signature for nonce=0 only).

No privileged access is required. Any user who deploys an account and submits one valid transaction can immediately exploit this for the nonce=1 slot.

### Recommendation

Replace the `account_tx_in_pool_or_recent_block` heuristic with a check that is specific to `deploy_account` transactions. The mempool should expose a method such as `has_pending_deploy_account(address) -> bool` that inspects only `DeployAccount`-typed entries. Alternatively, the gateway can inspect the transaction type of the pending entry before deciding to skip `__validate__`.

### Proof of Concept

```
1. Deploy account A on-chain. Committed nonce = 0.

2. POST /add_transaction:
     invoke { sender=A, nonce=0, calldata=X, signature=valid_sig_for_nonce_0 }
   → passes StatelessValidator, nonce check (0 ≤ 0 ≤ 200), validate_by_mempool,
     skip_stateful_validations returns false (nonce≠1), __validate__ runs → OK.
   → invoke(nonce=0) is now in the mempool.

3. POST /add_transaction:
     invoke { sender=A, nonce=1, calldata=FORBIDDEN, signature=0xdeadbeef }
   → passes StatelessValidator.
   → validate_state_preconditions: account_nonce=0, tx_nonce=1, 0≤1≤200 → OK.
   → validate_by_mempool: no duplicate hash, nonce=1 not in pool → OK.
   → skip_stateful_validations:
       tx.nonce()==1 ✓  account_nonce==0 ✓
       account_tx_in_pool_or_recent_block(A) → true (invoke(nonce=0) is in pool)
       returns true  ← skip __validate__
   → run_validate_entry_point(skip_validate=true):
       execution_flags.validate = false
       perform_pre_validation_stage passes (fee/nonce OK)
       returns Ok() without calling __validate__
   → Gateway returns 200 OK with a transaction hash.

Result: invoke(nonce=1, sig=0xdeadbeef) is accepted by the gateway and enters the
mempool despite having an invalid signature. The account's __validate__ was never
called at admission time.
```

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}

    // Performs static checks before executing validation entry point.
    // Note that nonce is incremented during these checks.
    pub fn perform_pre_validation_stage<S: State + StateReader>(
        &self,
        state: &mut S,
        tx_context: &TransactionContext,
    ) -> TransactionPreValidationResult<()> {
        let tx_info = &tx_context.tx_info;
        Self::handle_nonce(state, tx_info, self.execution_flags.strict_nonce_check)?;

        if self.execution_flags.charge_fee {
            self.check_fee_bounds(tx_context)?;

            verify_can_pay_committed_bounds(state, tx_context).map_err(Box::new)?;
        }

        self.validate_proof_facts(&tx_context.block_context, state)?;

        Ok(())
    }
