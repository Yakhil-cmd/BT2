Audit Report

## Title
Missing Re-Check of `migrations_disabled` Flag After Async Validation in `migrate_canister` — (File: rs/migration_canister/src/migration_canister.rs)

## Summary

The `migrate_canister` update function checks `migrations_disabled()` at entry (line 63) but does not re-check it after the long-running async `validate_request` call returns. Because `validate_request` makes nine sequential xnet calls across multiple IC rounds, a `PauseCanisterMigrations` NNS proposal executed during that window will not stop the in-flight request: it proceeds to `insert_request` and enters the timer-driven processing pipeline, bypassing the emergency stop.

## Finding Description

In `rs/migration_canister/src/migration_canister.rs` lines 62–93, `migrate_canister` performs an initial guard:

```rust
if migrations_disabled() {
    return Err(Some(ValidationError::MigrationsDisabled(Reserved)));
}
```

It then calls `validate_request(...)` which suspends across nine xnet calls (two `get_subnet_for_canister`, two `get_subnet`, two `get_canister_info`, two `canister_status`, one `assert_no_snapshots`) in `rs/migration_canister/src/validation.rs` lines 70–185. After `validate_request` returns `Ok`, the code re-checks `rate_limited()` (line 85) but never re-checks `migrations_disabled()` before calling `insert_request` (line 89).

The `disable_api` endpoint in `rs/migration_canister/src/privileged.rs` lines 34–38 sets the `DISABLED` stable cell via `set_disabled_flag(true)`. This is callable by controllers and the NNS governance canister. The NNS maps `PauseCanisterMigrations` directly to `disable_api` (confirmed in `rs/nns/governance/src/proposals/execute_nns_function.rs`). The `migrations_disabled()` function in `rs/migration_canister/src/canister_state.rs` lines 56–58 reads this cell.

The asymmetry is clear: `rate_limited()` is re-checked post-await (line 85), but `migrations_disabled()` is not, despite both being mutable state that can change during the async suspension window.

## Impact Explanation

When governance executes a `PauseCanisterMigrations` proposal as an emergency stop, any `migrate_canister` call already suspended inside `validate_request` will complete and call `insert_request`, queuing the migration for timer-driven processing (`process_accepted`, `process_controllers_changed`, etc.). The emergency stop is silently bypassed for all in-flight validations (up to `MAX_ONGOING_VALIDATIONS` concurrent requests). This constitutes a meaningful infrastructure security impact: the protocol's ability to halt canister migrations in response to a discovered bug, compromised subnet, or incorrect routing-table update is undermined. This fits the **Medium** impact class — the attack requires strict timing conditions (governance proposal execution must overlap with the validation window) but produces concrete protocol harm.

## Likelihood Explanation

`validate_request` spans at least nine sequential xnet calls, each requiring at least one IC round (~1–2 seconds), giving a total window of roughly 10–20 seconds per in-flight validation. Any canister developer who controls both the migrated and replaced canisters can call `migrate_canister` without special privilege. A `PauseCanisterMigrations` proposal that passes and executes during this window — a realistic scenario given that NNS proposals can be executed at any time after they pass — will silently fail to stop the in-flight request. The condition is narrow but not contrived, particularly in an emergency scenario where governance is actively trying to halt migrations.

## Recommendation

Add a `migrations_disabled()` re-check immediately after `validate_request` returns `Ok`, mirroring the existing `rate_limited()` re-check at line 85:

```rust
Ok((request, _guards)) => {
    if migrations_disabled() {
        return Err(Some(ValidationError::MigrationsDisabled(Reserved)));
    }
    if rate_limited() {
        return Err(Some(ValidationError::RateLimited(Reserved)));
    }
    println!("Accepted request {}", request);
    insert_request(RequestState::Accepted { request });
}
```

This closes the TOCTOU window and mirrors the pattern already applied to `rate_limited()`.

## Proof of Concept

1. Deploy the migration canister with the API enabled.
2. As a controller of both `migrated_canister` and `replaced_canister`, call `migrate_canister`. The initial `migrations_disabled()` check at line 63 passes.
3. `validate_request` begins, suspending across 9+ xnet calls.
4. During this window, call `disable_api` (as a controller or via a governance proposal). `DISABLED` is set to `true`.
5. `validate_request` completes successfully.
6. Execution reaches line 85: `rate_limited()` is checked (passes). `migrations_disabled()` is **not** checked.
7. `insert_request(RequestState::Accepted { request })` is called at line 89.
8. Verify via `migration_status` that the request is now `InProgress`, confirming the emergency stop was bypassed.

A deterministic integration test using PocketIC can reproduce this by interleaving a `disable_api` call between two of the xnet call suspension points inside `validate_request`.
