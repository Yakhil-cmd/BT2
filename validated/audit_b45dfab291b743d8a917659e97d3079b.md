### Title
Division Before Multiplication in `native_per_pubdata` Computation Causes Systematic Undercharging of Native Resources for Pubdata - (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

In ZKsync OS, the per-pubdata-byte native resource rate (`native_per_pubdata`) is computed by integer-dividing `pubdata_price` by `native_price` before multiplying by the actual pubdata bytes consumed. This is the same division-before-multiplication ordering flaw as the external report. When `pubdata_price < native_price`, `native_per_pubdata` truncates to zero and the transaction pays **zero** native resources for all pubdata it produces. Even when `pubdata_price ≥ native_price`, the truncation systematically understates the native cost by up to `(native_price − 1) × pubdata_bytes_used` native units per transaction.

---

### Finding Description

In `validation_impl.rs`, the L2 transaction validation path computes:

```rust
// Line 142-143
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
``` [1](#0-0) 

This intermediate `u64` value is then used in `get_resources_to_charge_for_pubdata` to compute the actual native cost:

```rust
// gas_helpers.rs lines 430-432
let native = current_pubdata_spent
    .checked_mul(native_per_pubdata)
    .ok_or(out_of_native_resources!())?;
``` [2](#0-1) 

The order of operations is:
1. `native_per_pubdata = pubdata_price / native_price` **(DIV first — truncating)**
2. `native_cost = pubdata_bytes_used × native_per_pubdata` **(MUL second)**

The correct order to avoid precision loss is:
```
native_cost = pubdata_bytes_used × pubdata_price / native_price
```

The same pattern is independently replicated in the API helper:

```rust
// api/src/helpers.rs line 427
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price)).ok_or(())?;
``` [3](#0-2) 

A secondary instance of division-before-multiplication also appears in the `delta_gas` adjustment in `compute_gas_refund`:

```rust
// refund_calculation.rs line 72
(native_used / native_per_gas) as i64 - (gas_used as i64)
``` [4](#0-3) 

Here `native_used / native_per_gas` truncates the gas-equivalent of native consumption, causing `delta_gas` to be understated and the sender to be undercharged for native-heavy operations.

---

### Impact Explanation

**Case 1 — `pubdata_price < native_price` (zero truncation):**
`native_per_pubdata = 0`. Every call to `get_resources_to_charge_for_pubdata` returns zero native cost regardless of pubdata bytes consumed. The transaction's native budget is never charged for pubdata. A transaction can write to as many storage slots as the pubdata limit allows without any native resource penalty.

**Case 2 — `pubdata_price ≥ native_price` (systematic undercharge):**
The truncation error is up to `(native_price − 1)` native units per pubdata byte. For a transaction using `N` pubdata bytes, the undercharge is up to `(native_price − 1) × N`. With `native_price = 1000` and `pubdata_price = 1999`, a transaction using 10,000 bytes of pubdata is charged `10,000 × 1 = 10,000` native instead of the correct `10,000 × 1999 / 1000 = 19,990` — a ~50% undercharge.

The native resource is the mechanism that limits how much proving work a transaction can impose on the network. Systematic undercharging of pubdata-heavy transactions allows them to consume more proving resources than they pay for, potentially enabling block-level resource exhaustion or denial-of-service against the prover.

---

### Likelihood Explanation

Any unprivileged L2 transaction sender can trigger this by submitting a transaction that writes to storage (SSTORE opcodes), which generates pubdata. The operator sets `pubdata_price` and `native_price` via block metadata. The truncation error is present for any combination of these values where `pubdata_price` is not an exact multiple of `native_price` — which is the common case in practice. The zero-truncation case (`pubdata_price < native_price`) is reachable whenever the operator sets these prices in that relationship, which is not prevented by any on-chain constraint. [5](#0-4) 

---

### Recommendation

Defer the division to after the multiplication. Replace the two-step computation with a single expression that multiplies first:

```rust
// In get_resources_to_charge_for_pubdata (gas_helpers.rs)
// Instead of: native = pubdata_bytes * (pubdata_price / native_price)
// Use:        native = pubdata_bytes * pubdata_price / native_price
let native = current_pubdata_spent
    .checked_mul(native_per_pubdata_numerator)   // pubdata_price
    .and_then(|v| v.checked_div(native_price))
    .ok_or(out_of_native_resources!())?;
```

This requires threading `pubdata_price` and `native_price` separately rather than pre-computing the truncated ratio. The same fix applies to the API helper in `api/src/helpers.rs` and the `delta_gas` calculation in `refund_calculation.rs`.

---

### Proof of Concept

**Setup:** Operator sets `native_price = 1000`, `pubdata_price = 999`.

**Step 1:** Transaction validation computes:
```
native_per_pubdata = 999 / 1000 = 0  (integer truncation)
``` [6](#0-5) 

**Step 2:** Transaction executes 100 SSTORE operations, producing ~3,200 bytes of pubdata.

**Step 3:** `get_resources_to_charge_for_pubdata` is called:
```
native_cost = 3200 × 0 = 0
``` [2](#0-1) 

**Step 4:** `check_enough_resources_for_pubdata` returns `enough = true` (zero native required). The transaction succeeds and pays zero native for all pubdata.

**Correct behavior:** `native_cost = 3200 × 999 / 1000 = 3196` native units should be charged and deducted from the transaction's native budget.
