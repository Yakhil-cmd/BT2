### Title
Duplicate Account Key Weight Double-Counting Bypasses Signature Weight Threshold — (`fvm/transactionVerifier.go`, `fvm/environment/accounts_status.go`)

---

### Summary

An account owner can add the same ECDSA public key material twice under two different key indices (each with weight 600), then sign a single transaction with both key indices using one private key. The transaction verifier aggregates 600+600=1200 into `payloadWeights[A]`, satisfying the 1000-weight threshold, while only one private key was used. The deduplication system correctly identifies the duplicate for storage purposes but does **not** prevent the weight from being counted twice during transaction verification.

---

### Finding Description

**Step 1 — Adding duplicate keys is permitted and recorded with independent weights.**

When `AppendAccountPublicKeyMetadata` is called for key1 with the same encoded key bytes as key0, `FindDuplicateKey` detects the match via digest + byte comparison and returns `isDuplicateKey = true`. [1](#0-0) 

The branch taken is `AppendDuplicateKeyMetadata`, which stores weight=600 for key1 in the weight-and-revoked-status section of key metadata, and records a mapping `key1 → storedKeyIndex 0`. Critically, `saveKey = false` is returned — no new stored key bytes are written — but the **weight for key1 is still independently stored**. [2](#0-1) [3](#0-2) 

**Step 2 — `GetRuntimeAccountPublicKey` for key1 returns weight=600 and the same public key bytes as key0.**

When the verifier fetches key1, `AccountPublicKeyMetadata` returns weight=600 (from key metadata) and storedKeyIndex=0 (from the deduplication mapping). `getStoredPublicKey` then fetches the stored key at index 0 — the same ECDSA key material as key0. [4](#0-3) 

**Step 3 — `newSignatureEntries` deduplicates only by `(address, keyIndex)` pair, not by underlying key material.**

Key index 0 and key index 1 are distinct pairs, so both signatures pass the deduplication check and are added to the continuation list. [5](#0-4) 

**Step 4 — `getAccountKeysAndAggregateWeights` adds each key's weight independently.**

For key0: `aggregateWeights[A] += 600` → 600. For key1: `aggregateWeights[A] += 600` → 1200. No check is made for whether two keys share the same underlying stored key. [6](#0-5) 

**Step 5 — `verifySignatures` succeeds for both key indices using one private key.**

`verify()` calls `crypto.VerifySignatureFromTransaction` against `entry.accountKey.PublicKey`. Since key1's `accountKey.PublicKey` is identical to key0's (same storedKeyIndex), a signature produced by the single private key is valid for both key indices. [7](#0-6) 

---

### Impact Explanation

An account owner with a single private key can authorize any transaction on their own account that would normally require two independent signers (e.g., a multisig policy requiring 1000 weight from two separate keys of 600 each). This violates the invariant that weight threshold policies enforce independent key control. The attacker fully controls account A and can execute arbitrary authorized operations — contract deployment, resource transfers, key management — that the account's weight policy was designed to require multiple parties to approve.

---

### Likelihood Explanation

The attack requires only:
1. Ownership of account A (standard, no privilege needed).
2. Two sequential `auth(AddKey)` transactions adding the same key material with weight 600 each — both are valid operations the Cadence runtime permits.
3. Constructing a transaction with two payload signatures referencing key indices 0 and 1, both signed with the same private key.

No admin access, no leaked keys, no staked node control, and no brute force is required. The path is fully reachable through the standard transaction submission API.

---

### Recommendation

In `getAccountKeysAndAggregateWeights`, after fetching each key's `storedKeyIndex` (available via `GetRuntimeAccountPublicKey` or a parallel lookup), track which `storedKeyIndex` values have already contributed weight for a given address. If a `storedKeyIndex` has already been counted, skip adding its weight again. This enforces the invariant that a single underlying key material can only contribute weight once per transaction, regardless of how many account key indices map to it.

Alternatively, `newSignatureEntries` could be extended to deduplicate not only by `(address, keyIndex)` but also by `(address, storedKeyIndex)`, rejecting transactions that present two signatures backed by the same stored key.

---

### Proof of Concept

```
1. Create account A with key0: ECDSA_P256, weight=600, hashAlgo=SHA3_256.
2. Submit tx: auth(AddKey) on account A, adding the same ECDSA_P256 public key bytes, weight=600.
   → Account A now has key0 (weight=600) and key1 (weight=600), both backed by storedKeyIndex=0.
3. Construct a transaction with:
   - Authorizer: account A
   - PayloadSignatures: [{address: A, keyIndex: 0, sig: Sign(payload, privKey)},
                         {address: A, keyIndex: 1, sig: Sign(payload, privKey)}]
   - EnvelopeSignature: [{address: A, keyIndex: 0, sig: Sign(envelope, privKey)}]
4. Submit transaction.
5. Observe: payloadWeights[A] = 1200 >= 1000 threshold → transaction executes.
   Only one private key was used; the two-independent-key policy is bypassed.
```

// AppendDuplicateKeyMetadata appends duplicate key metadata.
func (m *KeyMetadataAppender) AppendDuplicateKeyMetadata(
	keyIndex uint32,
	duplicateStoredKeyIndex uint32,
	revoked bool,
	weight uint16,
) (err error) {

	// Append revoked status and weight
	m.weightAndRevokedStatusBytes, err = appendWeightAndRevokedStatus(m.weightAndRevokedStatusBytes, revoked, weight)
	if err != nil {
		return err
	}

	if !m.deduplicated {
		// Set deduplication flag
		m.deduplicated = true

		// Save mapping start key index.
		m.startIndexForMapping = keyIndex
	}

	// Append duplicate stored key index to mapping
	m.mappingBytes, err = appendStoredKeyIndexToMappings(m.mappingBytes, duplicateStoredKeyIndex)
	if err != nil {
		return err
	}

	return nil
