### Title
Staker Can Instantly Increase Commission via `set_commission_commitment` + `set_commission`, Stealing Delegator Yield — (File: `src/staking/staking.cairo`)

---

### Summary

A staker can set a `CommissionCommitment` with a high `max_commission`, then immediately call `set_commission` to raise their commission to that maximum — with no time delay before the new rate takes effect on reward calculations. Delegators who joined based on a low commission rate have their unclaimed yield stolen during the exit wait window, with no ability to react before the change is applied.

---

### Finding Description

`set_commission` in `staking.cairo` normally enforces that commission can only be **decreased**. However, when a `CommissionCommitment` is active, the guard changes: the staker may set commission to any value `<= max_commission` and `!= old_commission`, which explicitly permits **increases**. [1](#0-0) 

The commission is stored immediately with no epoch delay: [2](#0-1) 

At reward calculation time, the commission is read directly from live storage — no historical snapshot is used: [3](#0-2) 

The `set_commission_commitment` function only requires `max_commission >= current_commission`, so a staker with commission 0 can set `max_commission = 10000` (100%): [4](#0-3) 

The commitment becomes active immediately and can be as short as one epoch: [5](#0-4) 

The developers themselves acknowledge this gap in a code comment: [6](#0-5) 

**Attack path:**

1. Staker stakes with commission = 0 and opens a delegation pool. Delegators join, trusting the 0% commission.
2. Staker calls `set_commission_commitment(max_commission: 10000, expiration_epoch: current_epoch + 1)`. This is visible on-chain but delegators may not monitor it.
3. Staker calls `set_commission(10000)`. Commission is now 100%, effective immediately.
4. The next reward calculation (`update_rewards_from_attestation_contract` or `update_rewards`) applies `commission = 10000`, so `split_rewards_with_commission` routes 100% of pool rewards to the staker as commission and 0 to the pool.
5. Delegators call `exit_delegation_pool_intent` but must wait through the `exit_wait_window` (default 1 week, up to 12 weeks).
6. During the entire exit wait window, delegators earn zero yield while the staker collects all pool rewards. [7](#0-6) 

---

### Impact Explanation

**High — Theft of unclaimed yield.**

All delegators in the pool lose 100% of their yield for the duration of the exit wait window. The staker directly profits by receiving commission on the full pool balance. The delegators' principal is locked and cannot be recovered until the exit window expires. With a default exit wait window of one week and a large pool, this represents a substantial theft of yield.

---

### Likelihood Explanation

**Medium.** The staker must first call `set_commission_commitment` with a high `max_commission`, which is an on-chain event. However:
- Delegators are not required to monitor commitment changes.
- No UI or protocol mechanism forces delegators to re-confirm their delegation after a commitment is set.
- On Starknet, the sequencer can order the staker's `set_commission` call ahead of any delegator reaction in the same block.
- The attack is profitable and requires no external resources beyond the staker's own account.

---

### Recommendation

1. **Epoch-delay commission changes**: Apply commission increases only starting from `current_epoch + K`, mirroring how balance changes are delayed. This gives delegators a window to observe and exit before the new rate takes effect.
2. **Emit a prominent event on commitment creation**: Ensure indexers and frontends surface `CommissionCommitmentSet` events so delegators are alerted before the staker can act on the commitment.
3. **Limit `max_commission` increase magnitude**: Restrict how much `max_commission` can exceed the current commission in a single commitment (e.g., no more than a fixed delta per epoch).

---

### Proof of Concept

```
1. Staker stakes with commission = 0, opens STRK pool.
2. Delegators enter pool; pool balance = D.
3. Staker calls: set_commission_commitment(max_commission=10000, expiration_epoch=E+1)
   → CommissionCommitmentSet event emitted; no delay enforced.
4. Staker calls: set_commission(10000)
   → update_commission() passes: 10000 <= 10000 (max_commission) && 10000 != 0 (old)
   → staker_pool_info.commission written = 10000 immediately.
5. Attestation / update_rewards fires:
   → calculate_staker_pools_rewards reads commission = 10000
   → split_rewards_with_commission(pool_rewards_including_commission, 10000)
     → commission_rewards = pool_rewards_including_commission * 10000 / 10000 = full amount
     → pool_rewards = 0
   → Delegators receive 0 STRK.
6. Delegators call exit_delegation_pool_intent; must wait exit_wait_window (≥ 1 week).
7. During wait window, every reward cycle repeats step 5 → staker steals all pool yield.
```

pub(crate) fn split_rewards_with_commission(
    rewards_including_commission: Amount, commission: Commission,
) -> (Amount, Amount) {
    let commission_rewards = compute_commission_amount_rounded_down(
        :rewards_including_commission, :commission,
    );
    let pool_rewards = rewards_including_commission - commission_rewards;
    (commission_rewards, pool_rewards)
}
