### Title
Hardcoded Placeholder `Fee(1)` Used as `paid_fee_on_l1` for All Consensus-Path L1 Handler Transactions - (File: crates/apollo_transaction_converter/src/transaction_converter.rs)

### Summary

In `TransactionConverter::convert_consensus_l1_handler_to_internal_l1_handler`, the `paid_fee_on_l1` field of every L1 handler transaction processed through the consensus path is unconditionally set to the hardcoded placeholder value `Fee(1)` instead of the actual fee paid on L1. A TODO comment in the code explicitly acknowledges this is not the real value. This is the direct sequencer analog of the external report's pattern: a field that must carry a real, caller-supplied value is instead silently assigned a wrong placeholder, with a reminder to fix it before production.

### Finding Description

`crates/apollo_transaction_converter/src/transaction_converter.rs` lines 473–483:

```rust
fn convert_consensus_l1_handler_to_internal_l1_handler(
    &self,
    tx: transaction::L1HandlerTransaction,
) -> TransactionConverterResult<executable_transaction::L1HandlerTransaction> {
    Ok(executable_transaction::L1HandlerTransaction::create(
        tx,
        &self.chain_id,
        // TODO(Gilad): Change this once we put real value in paid_fee_on_l1.
        Fee(1),
    )?)
}
``` [1](#0-0) 

This function is called from the consensus conversion path:

```rust
ConsensusTransaction::L1Handler(tx) => {

    Ok((InternalConsensusTransaction::L1Handler(internal_tx), None))

``` [2](#0-1) 

The resulting `executable_transaction::L1HandlerTransaction` carries `paid_fee_on_l1 = Fee(1)` into the blockifier's execution pipeline. The blockifier uses `paid_fee_on_l1` in `verify_can_pay_committed_bounds` and in fee-transfer logic to determine whether the L1 handler can pay its committed resource bounds and to compute the actual fee charged. [3](#0-2) 

### Impact Explanation

Every L1 handler transaction sequenced through the consensus path enters blockifier execution with `paid_fee_on_l1 = 1 wei` regardless of the actual amount locked on L1. This produces two concrete wrong outcomes:

1. **Wrong receipt / fee accounting**: The `TransactionReceipt` records a fee of 1 wei for every L1 handler, corrupting on-chain fee records and any downstream accounting that reads receipts.
2. **Wrong fee-sufficiency decision**: `verify_can_pay_committed_bounds` compares `paid_fee_on_l1` against the committed resource bounds. With `Fee(1)`, a transaction whose actual L1 payment is sufficient may be rejected (1 wei < required bound), or a transaction whose actual L1 payment is insufficient may be accepted (if the bound happens to be ≤ 1 wei), breaking the fee invariant in either direction.

This matches the allowed impact: *"Critical. Incorrect fee, gas, bouncer, resource accounting, refund, balance, or L1 gas price effect with economic impact."*

### Likelihood Explanation

The code path is unconditional and active in production. Any L1 handler transaction that reaches the sequencer via the consensus flow (i.e., every L1→L2 message processed after consensus) triggers this bug. No special attacker capability is required; the wrong value is applied automatically by the converter.

### Recommendation

Pass the actual `paid_fee_on_l1` value through the `ConsensusTransaction::L1Handler` variant so it is available when `convert_consensus_l1_handler_to_internal_l1_handler` is called. Remove the `Fee(1)` placeholder and the associated TODO comment once the real value is wired through.

### Proof of Concept

1. Submit an L1→L2 message on L1 with a non-trivial fee (e.g., 1 ETH worth of STRK).
2. The sequencer picks up the corresponding `ConsensusTransaction::L1Handler` event.
3. `convert_consensus_l1_handler_to_internal_l1_handler` is called; `paid_fee_on_l1` is set to `Fee(1)`.
4. The blockifier executes the transaction with `paid_fee_on_l1 = 1 wei`.
5. Observe the emitted `TransactionReceipt`: the recorded fee is 1 wei, not the actual L1 payment.
6. If the committed resource bounds exceed 1 wei, `verify_can_pay_committed_bounds` rejects the transaction despite the L1 payment being sufficient — a valid L1 message is silently dropped.
