### Title
Duplicate `data_id` in `ActionReceipt.input_data_ids` permanently stalls postponed receipt via `PendingDataCount` overcounting — (`runtime/runtime/src/lib.rs`)

### Summary
`process_action_receipt` iterates `input_data_ids` without deduplication. For each unseen `data_id` it increments `pending_data_count` and writes `PostponedReceiptId`. A duplicate `data_id` passes the `has_received_data` guard twice (the guard checks `ReceivedData`, not `PostponedReceiptId`), inflating `PendingDataCount` by 2 while only one `PostponedReceiptId` entry exists. The single arriving `DataReceipt` decrements the counter to 1, leaving the postponed receipt permanently stuck.

### Finding Description
In `process_action_receipt` [1](#0-0) , the loop over `action_receipt.input_data_ids()` calls `has_received_data` to decide whether to increment `pending_data_count`. `has_received_data` reads `TrieKey::ReceivedData` [2](#0-1) , which is never written inside this loop — only `TrieKey::PostponedReceiptId` is written. Therefore a duplicate `data_id` passes the guard on both iterations, incrementing `pending_data_count` twice and writing the same `PostponedReceiptId` value twice (second write is a no-op overwrite). [3](#0-2) 

When the `DataReceipt` for that `data_id` arrives, the runtime finds exactly one `PostponedReceiptId` entry, removes it, reads `PendingDataCount = 2`, and since `2 != 1` decrements to 1 and saves it. [4](#0-3)  No further `DataReceipt` for that `data_id` will ever arrive (it was already delivered), so `PendingDataCount` never reaches 0 and `apply_action_receipt` is never called.

`validate_action_receipt` imposes only a length limit on `input_data_ids` and validates actions; it performs no uniqueness check on the `data_id` values. [5](#0-4) 

### Impact Explanation
The exact corrupted value is `TrieKey::PendingDataCount` for the affected receipt, permanently stuck at 1. The postponed `ActionReceipt` and its `PostponedReceipt` trie entry remain in state forever. Any actions inside that receipt — token transfers, state mutations, cross-contract calls — never execute. Any `NEAR` attached as deposit to those actions is permanently locked in the postponed receipt's storage. This breaks the ledger invariant: *every `ActionReceipt` whose `input_data_ids` are all satisfied executes exactly once*.

### Likelihood Explanation
An unprivileged user deploys a contract that calls `promise_and([p, p])` with the same promise index, which is a valid host function invocation. The resulting `ActionReceipt` carries `input_data_ids: [X, X]`. No special privilege, validator access, or network position is required. The attack is deterministic and reproducible on any shard.

### Recommendation
In `process_action_receipt`, deduplicate `input_data_ids` before the loop (e.g., collect into a `HashSet<CryptoHash>` and iterate the set), or add an early-return error if any duplicate is detected. Additionally, add a uniqueness check in `validate_action_receipt` analogous to the existing length check, so malformed receipts are rejected at the boundary before they reach the runtime loop.

### Proof of Concept
A runtime `apply` test:
1. Create an `ActionReceipt` with `input_data_ids: [X, X]` where `X` is an arbitrary `CryptoHash`.
2. Create a `DataReceipt` with `data_id: X`.
3. Call `runtime.apply(...)` with both receipts in the same chunk (action receipt first, then data receipt).
4. Assert that `apply_action_receipt` is called (i.e., the receipt executes). Without the fix, `PendingDataCount` is left at 1 and the assertion fails — the receipt is permanently postponed.

    receiver_id: &AccountId,

        receipt.actions(),
        receiver,
        current_protocol_version,
        mode,
