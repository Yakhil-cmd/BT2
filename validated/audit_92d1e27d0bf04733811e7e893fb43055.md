### Title
Unbounded `pool_member_epoch_balance` Trace Growth Causes Permanent Freezing of Pool Member Rewards - (File: `src/pool/pool.cairo`)

### Summary
The `calculate_rewards` function in `pool.cairo` iterates over a pool member's `pool_member_epoch_balance` trace without any upper bound. The trace grows by one checkpoint per epoch whenever the pool member's balance changes. A pool member's `reward_address` — an unprivileged role — can call `add_to_delegation_pool` once per epoch to grow this trace indefinitely. After enough epochs, `claim_rewards` will exhaust the Starknet steps/gas limit, permanently freezing the pool member's unclaimed yield.

### Finding Description

The `calculate_rewards` internal function in `pool.cairo` contains a loop that the developers themselves annotate as unbounded:

```cairo
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
    if pool_member_checkpoint.epoch() >= until_epoch {
        break;
    }
    let to_sigma = self.find_sigma(pool_member_checkpoint, curr_epoch: until_epoch);
    rewards += compute_rewards_rounded_down(...);
    ...
    entry_to_claim_from += 1;
}
``` [1](#0-0) 

The trace is stored in `pool_member_epoch_balance: Map<ContractAddress, PoolMemberBalanceTrace>`. [2](#0-1) 

The trace insert logic in `set_member_balance` appends a new checkpoint at `current_epoch + K` whenever the epoch key differs from the last stored key: [3](#0-2) 

The `add_to_delegation_pool` function is callable by either the pool member **or their `reward_address`**:

```cairo
assert!(
    caller_address == pool_member || caller_address == pool_member_info.reward_address,
    "{}",
    Error::CALLER_CANNOT_ADD_TO_POOL,
);
``` [4](#0-3) 

Each call to `add_to_delegation_pool` in a new epoch appends one checkpoint to the trace via `increase_member_balance` → `set_member_balance`. [5](#0-4) 

**Attack path:**
1. A pool member sets (or initially has) a `reward_address` that is a separate, potentially malicious entity.
2. The malicious `reward_address` calls `add_to_delegation_pool` with a minimal non-zero amount (e.g., 1 token unit) once per epoch.
3. Each call in a distinct epoch appends a new checkpoint to the victim's `pool_member_epoch_balance` trace.
4. After `N` epochs of this, the trace has `N` entries.
5. When the pool member calls `claim_rewards`, `calculate_rewards` iterates over all `N` entries (each requiring a storage read), exhausting the Starknet steps budget.
6. `claim_rewards` reverts permanently; the pool member's accumulated unclaimed rewards are frozen.

`claim_rewards` calls `calculate_rewards` with no partial-claim mechanism: [6](#0-5) 

The `entry_to_claim_from` cursor advances only on a successful `claim_rewards` call. If the call always reverts, the cursor never advances and the freeze is permanent. [7](#0-6) 

### Impact Explanation

**Permanent freezing of unclaimed yield (High).**

Once the trace is large enough to exceed the Starknet steps limit in a single transaction, `claim_rewards` will always revert. There is no partial-claim path and no administrative recovery function. The pool member's accrued STRK rewards are permanently inaccessible.

`pool_member_info_v1` (a view function) also calls `calculate_rewards`, so the pool member cannot even query their pending rewards once the trace is too large. [8](#0-7) 

### Likelihood Explanation

**Low-to-Medium.** The `reward_address` is an unprivileged role explicitly listed in the allowed attacker set. The economic cost to the attacker is minimal: one token unit per epoch (the tokens are added to the pool member's balance, not burned). Over a long staking horizon (hundreds of epochs), the trace can grow to a size that causes out-of-gas. The developers themselves flagged this loop as "unbounded" in a code comment, acknowledging the risk.

### Recommendation

1. **Add a partial-claim mechanism**: Allow `claim_rewards` to process at most `MAX_ENTRIES` checkpoints per call, advancing `entry_to_claim_from` incrementally.
2. **Cap trace growth**: Enforce a maximum number of checkpoints per pool member (analogous to the five-range cap applied in the referenced fix).
3. **Restrict `add_to_delegation_pool` callers**: Consider whether the `reward_address` truly needs the ability to add stake on behalf of the pool member, or restrict it to the pool member only.

### Proof of Concept

```
Epoch 1:  reward_address calls add_to_delegation_pool(pool_member, amount=1)
          → trace: [(epoch=1+K, balance=initial+1)]

Epoch 2:  reward_address calls add_to_delegation_pool(pool_member, amount=1)
          → trace: [(epoch=1+K, ...), (epoch=2+K, balance=initial+2)]

...

Epoch N:  trace has N entries.

Pool member calls claim_rewards(pool_member):
  → calculate_rewards iterates N times, each reading storage
  → Starknet steps budget exhausted at iteration M < N
  → Transaction reverts
  → Pool member's unclaimed rewards permanently frozen
``` [9](#0-8)
