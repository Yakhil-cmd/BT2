### Title
Unbounded `stakers` Vec causes DoS on `get_stakers` via permanent staker accumulation — (File: src/staking/staking.cairo)

### Summary
Every call to `stake()` appends the caller's address to a `Vec<ContractAddress>` named `stakers`. Stakers are **never removed** from this vector, even after `unstake_action`. The `get_stakers` function iterates over the entire vector on every call. An unprivileged attacker can bloat this vector by repeatedly staking the minimum amount from freshly generated addresses, eventually causing `get_stakers` to exhaust execution resources and become permanently unusable by the consensus layer.

---

### Finding Description

**Unbounded append, no removal:** [1](#0-0) 

The storage comment explicitly states stakers are never removed:
```
/// Vector of staker addresses.
/// **Note**: Stakers are not removed from this vector when they unstake.
stakers: Vec<ContractAddress>,
```

**Every `stake()` call pushes to the vector:** [2](#0-1) 

```cairo
// Add staker address to the stakers vector.
self.stakers.push(staker_address);
```

**`get_stakers` iterates the full vector unconditionally:** [3](#0-2) 

```cairo
for staker_address_ptr in self.stakers.into_iter_full_range() {
    let staker_address = staker_address_ptr.read();
    if !self.is_staker_active(:staker_address, :epoch_id) {
        continue;
    }
    ...
}
```

Even though inactive stakers are skipped with `continue`, every slot in the vector — including all historical, already-unstaked addresses — is still read from storage on each call. Storage reads are the dominant cost in Starknet execution.

---

### Impact Explanation

`get_stakers` is the canonical entry point used by the consensus layer to retrieve the active staker set for a given epoch. If this function exceeds Starknet's per-call execution resource limits (steps / builtins), it will revert or be rejected by the sequencer. Because the vector only grows and is never pruned, once the threshold is crossed the function is **permanently** broken — no governance action can shrink the vector without a contract upgrade.

Consequences:
- The consensus layer loses the ability to read the staker committee.
- Stakers with non-zero staking power cannot be included in consensus, effectively freezing their participation and any associated rewards.
- The protocol suffers irreversible operational damage proportional to the attacker's investment.

This maps to the allowed impact: **Medium — Unbounded gas consumption; Griefing with no profit motive but damage to users or protocol.**

---

### Likelihood Explanation

- Any address that meets `min_stake` can add itself to `stakers` permanently.
- After calling `unstake_intent` + `unstake_action`, the attacker recovers their principal (minus opportunity cost), while the dead address remains in the vector forever.
- The cost per slot is therefore only the opportunity cost of locked capital during the `exit_wait_window` (one week by default), not a permanent loss.
- The attack is cumulative: each wave of addresses permanently increases the iteration cost. Organic protocol growth (legitimate stakers who later exit) also contributes to the same degradation without any attacker involvement. [4](#0-3) 

---

### Recommendation

1. **Lazy deletion / active-set index**: Maintain a separate counter or bitmap of active stakers and skip dead slots without reading them, or use a swap-and-pop pattern to remove unstaked addresses from the vector.
2. **Paginated `get_stakers`**: Accept `offset` and `limit` parameters so callers can iterate in bounded chunks.
3. **Separate active-staker set**: Keep a second, pruned data structure (e.g., an `IterableMap` that supports deletion) for consensus queries, distinct from the append-only historical log.

---

### Proof of Concept

```
1. Attacker generates addresses A_1 … A_N, each pre-funded with min_stake STRK.
2. For each A_i:
   a. Call stake(reward_address, operational_address, min_stake) from A_i.
      → self.stakers.push(A_i)  [line 348]
   b. After exit_wait_window, call unstake_intent() then unstake_action().
      → A_i is removed from staker_info but NOT from self.stakers [line 168-169]
3. self.stakers now contains N permanently dead entries plus all legitimate stakers.
4. Any call to get_stakers(epoch_id) must iterate all N + M entries [line 918],
   reading each from storage.
5. Once N is large enough, get_stakers exceeds the Starknet per-call step limit
   and reverts on every invocation, permanently denying the consensus layer
   access to the staker committee.
``` [5](#0-4)

        fn get_stakers(
            self: @ContractState, epoch_id: Epoch,
        ) -> Span<(ContractAddress, StakingPower, Option<PublicKey>, Option<PeerId>)> {
            let curr_epoch = self.get_current_epoch();
            assert!(
                curr_epoch <= epoch_id && epoch_id < curr_epoch + K.into(),
                "{}",
                Error::INVALID_EPOCH,
            );

            let (strk_total_stake, btc_total_stake) = self
                .get_total_staking_power_at_epoch(:epoch_id);

            let mut stakers: Array<
                (ContractAddress, StakingPower, Option<PublicKey>, Option<PeerId>),
            > =
                array![];
            for staker_address_ptr in self.stakers.into_iter_full_range() {
                let staker_address = staker_address_ptr.read();
                if !self.is_staker_active(:staker_address, :epoch_id) {
                    continue;
                }

                let staking_power = self
                    .get_staker_staking_power_at_epoch(
                        :staker_address, :epoch_id, :strk_total_stake, :btc_total_stake,
                    );
                if staking_power.is_zero() {
                    continue;
                }

                let public_key = self.get_public_key_at_epoch(:staker_address, :epoch_id);
                let peer_id = self.get_peer_id_at_epoch(:staker_address, :epoch_id);
                stakers.append((staker_address, staking_power, public_key, peer_id));
            }
            stakers.span()
        }
