### Title
Unprivileged Caller Can Permanently Suppress All Consensus Block Rewards via `update_rewards(disable_rewards: true)` - (File: `src/staking/staking.cairo`)

---

### Summary

The `IStakingRewardsManager::update_rewards` function accepts a caller-controlled `disable_rewards` boolean with no access control beyond a basic non-zero/unpaused check. Any unprivileged address can call it with `disable_rewards: true` every block, consuming the single global `last_reward_block` slot while skipping reward distribution, permanently denying all stakers their consensus block rewards.

---

### Finding Description

`update_rewards` is exposed as a public function under `IStakingRewardsManager` with only `general_prerequisites()` as a guard:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();   // only checks: not paused, caller != zero
``` [1](#0-0) 

`general_prerequisites()` enforces no role restriction:

```cairo
fn general_prerequisites(ref self: ContractState) {
    self.assert_is_unpaused();
    assert_caller_is_not_zero();
}
``` [2](#0-1) 

The function then writes the current block to the **global** `last_reward_block` unconditionally, before checking `disable_rewards`:

```cairo
self.last_reward_block.write(current_block_number);

if disable_rewards || self.is_pre_consensus() {
    return;
}
``` [3](#0-2) 

The guard that prevents double-execution per block is:

```cairo
assert!(
    current_block_number > self.last_reward_block.read(),
    "{}",
    Error::REWARDS_ALREADY_UPDATED,
);
``` [4](#0-3) 

Because `last_reward_block` is a single global slot (not per-staker), once any caller writes to it in a given block, no other call can succeed in that block. An attacker who calls `update_rewards(any_staker, disable_rewards: true)` every block:

1. Writes `last_reward_block = current_block`
2. Returns immediately without distributing rewards
3. Blocks every legitimate `update_rewards` call for that block with `REWARDS_ALREADY_UPDATED`

The `last_reward_block` storage field:

```cairo
/// Last block number for which rewards were distributed.
last_reward_block: BlockNumber,
``` [5](#0-4) 

The analog to the XSS report is direct: just as an ERC20 symbol (user-controlled string) was passed unsanitized into SVG construction and executed as JavaScript, here a user-controlled boolean (`disable_rewards: true`) is passed unsanitized into a reward-distribution function with no authorization check, and its effect propagates globally to harm all other protocol participants.

---

### Impact Explanation

**High — Permanent freezing of unclaimed yield.**

An attacker can suppress all consensus block rewards for every staker in the protocol indefinitely. The `update_rewards` path is the sole mechanism for distributing per-block consensus rewards:

```cairo
let (strk_block_rewards, btc_block_rewards) = self
    .calculate_block_rewards(:reward_supplier_dispatcher, :curr_epoch);
self
    ._update_rewards(
        :staker_address,
        strk_total_rewards: strk_block_rewards,
        ...
    );
``` [6](#0-5) 

If the attacker front-runs every block with `disable_rewards: true`, no staker ever accumulates `unclaimed_rewards_own` from consensus rewards, and no pool ever receives `update_rewards_from_staking_contract`. All yield is permanently frozen.

---

### Likelihood Explanation

**High.** The attack requires:
- No funds
- No privileged role
- No complex setup

The only cost is one transaction per block. A motivated adversary (competing validator, protocol saboteur) can sustain this indefinitely. The function is part of the public ABI and callable by any EOA or contract.

---

### Recommendation

Add an explicit caller restriction to `update_rewards`. The function should only be callable by a trusted role (e.g., the attestation contract, a designated rewards manager role, or the staker/operational address themselves). At minimum, the `disable_rewards: true` path should require a privileged role:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
    if disable_rewards {
        self.roles.only_rewards_manager(); // add a dedicated role
    }
    ...
}
```

Alternatively, remove `disable_rewards` from the public interface entirely and handle the disabling logic through a separate privileged function.

---

### Proof of Concept

1. Attacker deploys a simple contract (or uses an EOA script) that calls `staking.update_rewards(any_active_staker, disable_rewards: true)` once per block.
2. Each call passes the `current_block_number > last_reward_block` guard (since a new block has started).
3. `last_reward_block` is written to `current_block_number`.
4. Execution returns at `if disable_rewards { return; }` — no rewards distributed.
5. Any legitimate validator or staker calling `update_rewards` in the same block receives `REWARDS_ALREADY_UPDATED` and reverts.
6. All stakers' `unclaimed_rewards_own` and all pool `cumulative_rewards_trace` entries remain frozen indefinitely.

The attacker pays only gas; every staker in the protocol loses all consensus block rewards for as long as the attack continues.
