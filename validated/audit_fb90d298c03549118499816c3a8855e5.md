### Title
Permanent ERC-20 Token Loss When `ft_transfer` Promise Fails Without `error_refund` Feature — (`File: engine-precompiles/src/native.rs`)

### Summary

When the `error_refund` compile-time feature is disabled, the `ExitToNear` precompile schedules the NEP-141 `ft_transfer` promise with **no callback** for the regular ERC-20 exit path. If that promise fails (e.g., recipient account not registered with the NEP-141 token), the ERC-20 tokens are already burned in the EVM with no on-chain recovery path. Tokens are permanently lost.

### Finding Description

The `ExitToNear` precompile's `run` method constructs `ExitToNearPrecompileCallbackArgs` with `refund` hardcoded to `None` when the `error_refund` feature is absent: [1](#0-0) 

For the regular ERC-20 exit path (non-wNEAR, non-Omni), `transfer_near_args` is also `None`: [2](#0-1) 

This makes `callback_args` equal to `ExitToNearPrecompileCallbackArgs::default()`, so the branch at line 470 fires and the promise is created **without any callback**: [3](#0-2) 

If the `ft_transfer` promise fails, there is no `exit_to_near_precompile_callback` receipt scheduled. The callback that would otherwise re-mint burned tokens is: [4](#0-3) 

With `refund: None`, the `else if let Some(args) = args.refund` branch is never taken, and the `else { None }` path silently succeeds — no refund, no re-mint.

For the wNEAR unwrap sub-path, a callback **is** scheduled (because `transfer_near` is `Some`), but if `near_withdraw` fails, the same `else { None }` path is taken — wNEAR ERC-20 tokens are burned and no NEAR is received.

### Impact Explanation

**Permanent freezing / loss of user funds.** The ERC-20 tokens are burned inside the EVM at the time the precompile runs (the ERC-20 contract calls the precompile from its burn function). The NEP-141 `ft_transfer` then executes asynchronously. If it fails, the burned ERC-20 supply is gone and the NEP-141 tokens remain locked in the Aurora contract with no mechanism to return them to the user. The test suite explicitly acknowledges this outcome: [5](#0-4) 

### Likelihood Explanation

Any EVM user holding a bridged ERC-20 token can trigger this path by calling the `ExitToNear` precompile with a recipient NEAR account that is not registered with the NEP-141 token (storage not deposited). This is a realistic user mistake. The `ft_transfer` call will fail on the NEP-141 side, and without `error_refund`, no refund receipt is ever created. The entry path is fully unprivileged.

### Recommendation

1. Ensure `error_refund` is always enabled in production builds, or make it a non-optional default feature in `engine-precompiles/Cargo.toml`.
2. For the regular ERC-20 exit path, always schedule a callback even when `error_refund` is disabled, so that a failed `ft_transfer` can at minimum be detected and logged, or a re-mint attempted.
3. For the wNEAR unwrap path, populate `refund` unconditionally so that a failed `near_withdraw` triggers a re-mint of the burned wNEAR ERC-20 tokens.

### Proof of Concept

1. Deploy Aurora Engine **without** the `error_refund` feature.
2. Bridge a NEP-141 token to Aurora (mint ERC-20 representation).
3. From an EVM address, call the `ExitToNear` precompile (flag `0x01`) targeting a NEAR account that has **not** called `storage_deposit` on the NEP-141 contract.
4. The ERC-20 `burn` fires → precompile runs → `ft_transfer` promise is scheduled with no callback.
5. `ft_transfer` fails on the NEP-141 side (unregistered recipient).
6. No callback receipt is created; no re-mint occurs.
7. Observe: ERC-20 balance reduced by exit amount, NEP-141 balance of recipient is 0, NEP-141 balance of Aurora contract unchanged — tokens are permanently lost.

This is confirmed by the existing test `test_exit_to_near_refund` which asserts `balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT)` when `error_refund` is not enabled, proving the burned tokens are unrecoverable. [6](#0-5) [7](#0-6) [8](#0-7)
