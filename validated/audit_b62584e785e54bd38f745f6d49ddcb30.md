### Title
Hardcoded 2,300-Gas Stipend in `DefaultGasLimitForTokenTransfer` Breaks COA Deposits/Transfers to Smart Contract Addresses - (File: `fvm/evm/types/call.go`)

### Summary

`NewDepositCall`, `NewWithdrawCall`, and `NewTransferCall` all use a hardcoded gas limit of `21,000 + 2,300 = 23,300`, where the 2,300 is the exact same gas stipend as Solidity's deprecated `transfer()`. Any EVM smart contract whose `receive()` or `fallback()` function requires more than 2,300 gas will permanently fail to receive FLOW tokens via COA operations, breaking cross-VM integrations.

### Finding Description

In `fvm/evm/types/call.go`, the gas limit for all native token bridge direct calls is defined as:

```go
IntrinsicFeeForTokenTransfer = gethParams.TxGas          // 21_000
DefaultGasLimitForTokenTransfer = IntrinsicFeeForTokenTransfer + 2_300  // 23_300

DepositCallGasLimit  = DefaultGasLimitForTokenTransfer
WithdrawCallGasLimit = DefaultGasLimitForTokenTransfer
``` [1](#0-0) 

This constant is then baked into every `NewDepositCall`, `NewWithdrawCall`, and `NewTransferCall`: [2](#0-1) 

These constructors are called directly from the COA handler's `Deposit()`, `Withdraw()`, and `Transfer()` methods: [3](#0-2) 

After the EVM deducts the 21,000 intrinsic gas, only 2,300 gas remains for the recipient contract's `receive()` or `fallback()` execution. This is identical to Solidity's `transfer()` stipend — a value widely known to be insufficient for contracts that emit events, update storage, or delegate to a proxy. The existing test suite even confirms this failure mode explicitly: a deposit to a contract with an empty body (no `receive`) already reverts with `gethVM.ErrExecutionReverted`. [4](#0-3) 

### Impact Explanation

Any EVM smart contract address whose `receive()` or `fallback()` consumes more than 2,300 gas — including Gnosis Safe multisigs, ERC-4337 smart wallets, proxy contracts, or any contract that emits an event on receipt — **cannot receive FLOW tokens via COA `Deposit()` or `Transfer()`**. The call reverts at the EVM level, `panicOnErrorOrInvalidOrFailedState` panics, and the entire Cadence transaction is rolled back. The user's funds are not lost (the panic reverts the vault withdrawal), but the transfer is permanently impossible to complete through the COA interface, breaking cross-VM integrations.

### Likelihood Explanation

This is not a future risk — it is a present, deterministic failure. Any user who attempts to deposit FLOW into a smart contract wallet (e.g., a Gnosis Safe deployed on Flow EVM) via their COA will have the transaction revert every time. The 2,300 gas limit is already insufficient for the most common smart contract wallet patterns in production use on EVM chains.

### Recommendation

Remove the hardcoded `+ 2_300` stipend from `DefaultGasLimitForTokenTransfer`. For `DepositCallGasLimit` and `WithdrawCallGasLimit`, either:

1. Use a higher configurable default (e.g., `300_000`) that accommodates typical smart contract `receive()` implementations, or
2. Allow the caller to pass a `gasLimit` parameter to `Deposit()` and `Transfer()` in the COA handler, similar to how `call()` already accepts a caller-supplied `gasLimit`.

The comment in the code itself acknowledges the fragility:

> "Note that these gas values might need to change if we change the transaction" [5](#0-4) 

### Proof of Concept

1. Deploy a Solidity contract on Flow EVM with a `receive()` that emits one event (costs ~1,500 gas for the LOG opcode + topic hashing, leaving no margin for other operations under 2,300 total).
2. From a Cadence transaction, call `coa.deposit(from: <-vault)` targeting that contract's EVM address.
3. The `NewDepositCall` is constructed with `GasLimit = 23_300`.
4. The EVM subtracts 21,000 intrinsic gas, leaving 2,300 for the `receive()`.
5. If `receive()` needs more than 2,300 gas, execution reverts with `ErrExecutionReverted`.
6. `panicOnErrorOrInvalidOrFailedState` panics, rolling back the Cadence transaction.
7. The deposit is impossible regardless of how much FLOW the user holds. [6](#0-5) [7](#0-6)

// Deposit deposits the token from the given vault into the flow evm main vault
// and update the account balance with the new amount
func (a *Account) Deposit(v *types.FLOWTokenVault) {
	defer a.fch.backend.StartChildSpan(trace.FVMEVMDeposit).End()

	bridge := a.fch.addressAllocator.NativeTokenBridgeAddress()
	bridgeAccount := a.fch.AccountByAddress(bridge, false)
	// Note: its not an authorized call
	res, err := a.fch.executeAndHandleCall(
		types.NewDepositCall(
			bridge,
			a.address,
			v.Balance(),
			bridgeAccount.Nonce(),
		),
		v.Balance(),
		false,
	)
	panicOnErrorOrInvalidOrFailedState(res, err)
}

// Withdraw deducts the balance from the account and
// withdraw and return flow token from the Flex main vault.
func (a *Account) Withdraw(b types.Balance) *types.FLOWTokenVault {
	defer a.fch.backend.StartChildSpan(trace.FVMEVMWithdraw).End()

	res, err := a.executeAndHandleAuthorizedCall(
		types.NewWithdrawCall(
			a.fch.addressAllocator.NativeTokenBridgeAddress(),
			a.address,
			b,
			a.Nonce(),
		),
		b,
		true,
	)
	panicOnErrorOrInvalidOrFailedState(res, err)

	return types.NewFlowTokenVault(b)
}

// Transfer transfers tokens between accounts
func (a *Account) Transfer(to types.Address, balance types.Balance) {
	res, err := a.executeAndHandleAuthorizedCall(
		types.NewTransferCall(
			a.address,
			to,
			balance,
			a.Nonce(),
		),
		nil,
		false,
	)
	panicOnErrorOrInvalidOrFailedState(res, err)
}
