Audit Report

## Title
Missing aggregate memory-size check in `validate_ingress_payload` allows Byzantine block proposer to force ~2 GB ingress payloads past consensus — (`rs/ingress_manager/src/ingress_selector.rs`)

## Summary

`validate_ingress_payload` enforces per-message size and message count but never checks the total in-memory size of the payload against `max_ingress_bytes_per_block`. When `hashes_in_blocks_enabled=true`, the wire size returned by the validator equals only the sum of message IDs (~32 bytes each), so a payload carrying 1000 × 2 MiB messages has a wire size of ~32 KB (passes the wire limit) but a memory size of ~2 GB (never checked). A single Byzantine block proposer below the BFT fault threshold can craft such a payload, have it accepted by all honest replicas, and force the entire subnet to deserialize and process 2 GB of ingress data.

## Finding Description

`validate_ingress_payload` (lines 323–423) calls `self.payload_size_estimates(payload)` at line 420 and returns only `size_estimates.wire` at line 422, with no comparison of `size_estimates.memory` against any limit.

`payload_size_estimates` (lines 644–658) computes:
- `memory_bytes = total_messages_size_estimate() + total_ids_size_estimate()`
- `wire_bytes = total_ids_size_estimate()` when `hashes_in_blocks_enabled=true`

The builder `get_ingress_payload` (lines 289–294) correctly enforces both: `size_estimates.wire <= wire_byte_limit && size_estimates.memory <= memory_byte_limit`. The validator enforces neither — it only returns the wire estimate for the caller to compare against the wire limit. The memory dimension is silently discarded.

With `MAX_INGRESS_MESSAGES_PER_BLOCK = 1000` and `MAX_INGRESS_BYTES_PER_MESSAGE_APP_SUBNET = 2 MiB`, a crafted payload passes all per-message and count checks while carrying up to 2 GB of message data. The validator accepts it (`Ok(size_estimates.wire)`), consensus finalizes the block, and the execution environment on every honest replica must process the full 2 GB payload.

## Impact Explanation

Every honest replica calls `validate_ingress_payload` before accepting a proposed block. A finalized block carrying 2 GB of ingress data forces all replicas to deserialize and induct the full message bodies, causing memory exhaustion and execution stalls across the entire subnet. This constitutes a concrete subnet-level availability impact — consensus blocking and execution halt — matching the **High ($2,000–$10,000)** bounty impact: *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."*

## Likelihood Explanation

Requires a single Byzantine block proposer — one malicious or compromised replica node scheduled to propose a block. This is within the IC's BFT model (below the 1/3 threshold). No governance majority, no key compromise, no external dependency. The attacker only needs subnet membership and to wait for their proposal slot. The attack is repeatable every time the attacker is scheduled to propose. The exploit is conditional on `hashes_in_blocks_enabled=true` being active on the target subnet.

## Recommendation

Add a memory-size check in `validate_ingress_payload` immediately after computing `size_estimates`, mirroring the check already present in `get_ingress_payload`:

```rust
let size_estimates = self.payload_size_estimates(payload);
let memory_limit = NumBytes::new(settings.max_ingress_bytes_per_block as u64);
if size_estimates.memory > memory_limit {
    return Err(ValidationError::InvalidArtifact(
        InvalidIngressPayloadReason::PayloadTooBig {
            size: size_estimates.memory,
            limit: memory_limit,
        },
    ));
}
Ok(size_estimates.wire)
```

`max_ingress_bytes_per_block` is already fetched from the registry via `get_ingress_message_settings` at line 335–337, so no additional registry call is needed. This mirrors the logic in `memory_byte_limit` (lines 619–642) for the `hashes_in_blocks_enabled=true` branch.

## Proof of Concept

```rust
// Byzantine block proposer constructs payload directly (bypassing get_ingress_payload):
let messages: Vec<SignedIngress> = (0..1000)
    .map(|i| SignedIngressBuilder::new()
        .canister_id(canister_test_id(0))
        .nonce(i)
        .method_payload(vec![0u8; 2 * 1024 * 1024]) // 2 MiB each
        .expiry_time(valid_expiry)
        .build())
    .collect();

let payload = IngressPayload::from(messages); // ~2 GB total

// With hashes_in_blocks_enabled=true:
// wire_bytes  ≈ 1000 × 32 = 32 KB  → passes wire limit
// memory_bytes ≈ 1000 × 2 MiB = 2 GB → NOT checked, passes validation

let result = ingress_manager.validate_ingress_payload(
    &payload, &HashSet::new(), &validation_context
);
assert!(result.is_ok()); // passes — all replicas accept the block
```

A deterministic integration test using `IngressManagerBuilder` with `hashes_in_blocks_enabled=true` and the above payload construction reproduces the issue without mainnet interaction.
