### Title
Unbounded Iteration Over All Block Events Without Native Resource Accounting in `eip6110_events_parser` — (File: `basic_bootloader/src/bootloader/block_flow/ethereum/eip_6110_deposit_events_parser/mod.rs`)

---

### Summary

`eip6110_events_parser` iterates over every event emitted in the entire block to locate EIP-6110 deposit events. The loop is unbounded — it scans all events regardless of how many are from the deposit contract — and no native computational resources are charged for the iteration work. An unprivileged attacker can fill a block with cheap `LOG` events from non-deposit addresses, forcing the prover to perform O(N) scanning work that is not reflected in the block's resource budget, creating a proving-cost amplification path and a potential valid-execution unprovability condition.

---

### Finding Description

`eip6110_events_parser` is called at block finalization in both the sequencing path and the proving path. Its implementation unconditionally iterates over the full block-level events iterator:

```rust
for event in system.io.events_iterator() {
    if event.address != &DEPOSIT_CONTRACT_ADDRESS {
        continue;          // ← scans every non-deposit event, no charge
    }
    ...
}
``` [1](#0-0) 

`events_iterator()` returns every event accumulated across all transactions in the block via `EventsStorage::events_ref_iter`, which is a plain O(N) walk of the internal `HistoryList`: [2](#0-1) 

The function is invoked at block finalization in the proving path with `.expect(...)`, meaning any failure aborts proof generation: [3](#0-2) 

And in the sequencing path: [4](#0-3) 

The entire loop runs with system-level resources. No ergs and no native computational units are charged for scanning non-deposit events. The only cost the attacker pays is the per-transaction EVM gas for `LOG` opcodes, which is charged inside the transaction frame — completely decoupled from the post-loop system scan.

By contrast, the EIP-7002 and EIP-7251 queue processors correctly cap their iteration at `MAX_WITHDRAWAL_REQUESTS_PER_BLOCK = 16` and `MAX_CONSOLIDATION_REQUESTS_PER_BLOCK = 2` respectively: [5](#0-4) [6](#0-5) 

No analogous cap exists for the event scan in `eip6110_events_parser`.

---

### Impact Explanation

**Vulnerability class:** Resource accounting bug / valid-execution unprovability.

In the ZKsync OS proving environment the RISC-V binary executes inside Airbender with a finite cycle budget. The native resource (`Resources::Native`) tracks computational work that must be paid for. Because `eip6110_events_parser` charges zero native resources for its O(N) scan, an attacker can construct a block whose post-loop finalization consumes a disproportionate number of RISC-V cycles relative to what was paid for in gas.

Concretely, with a 30 M gas block a single `LOG0` costs 375 gas, allowing roughly 80 000 events. Every one of those events is visited by the loop even though none of them are deposit events. The prover must execute the address-comparison branch for each, consuming cycles that are not budgeted. If the cycle count exceeds the prover's limit the block is valid on the sequencer but unprovable — a forward/proving divergence that stalls the rollup.

---

### Likelihood Explanation

- Any unprivileged user can emit `LOG` events from any contract.
- `LOG0` is the cheapest opcode for this purpose (375 gas, no topics).
- A single attacker transaction can emit thousands of events; a full block of such transactions maximizes the scan cost.
- No special role, key, or governance access is required.
- The attack is repeatable every block.

---

### Recommendation

1. **Add a hard cap on deposit events processed per block.** EIP-6110 specifies a maximum of 8192 deposit requests per block; enforce this limit inside the loop and break early once reached.
2. **Charge native resources per event scanned.** Introduce a per-event native cost inside the loop body (analogous to `EVENT_STORAGE_BASE_NATIVE_COST` already used in the IO subsystem) so that the prover's work is always covered by the block's resource budget.
3. **Maintain a separate deposit-event index.** Rather than scanning all events at finalization, record deposit events into a dedicated side-channel during transaction execution (where gas is already charged), and iterate only that list at finalization — mirroring the queue-based approach used by EIP-7002 and EIP-7251.

---

### Proof of Concept

1. Deploy a contract `Spammer` whose fallback emits `LOG0` with 32 bytes of data in a tight loop (e.g., 200 iterations per call).
2. Submit enough transactions calling `Spammer` to fill the block gas limit (~30 M gas), producing ≈ 80 000 events total, none from `DEPOSIT_CONTRACT_ADDRESS`.
3. The sequencer executes the block successfully; all gas is charged normally.
4. At block finalization `eip6110_events_parser` is called. It iterates all 80 000 events, performing an address comparison on each, with zero native resource charge.
5. In the proving path the RISC-V binary must execute the same 80 000-iteration loop. If the resulting cycle count exceeds the prover's budget, `post_op` cannot complete and the block is valid but unprovable — a forward/proving divergence that halts the chain. [7](#0-6)

pub fn eip6110_events_parser<S: EthereumLikeTypes>(
    system: &System<S>,
    requests_hasher: &mut impl crypto::sha256::Digest,
) -> Result<bool, SystemError>
where
    S::IO: IOSubsystemExt + IOTeardown<S::IOTypes>,
{
    // we can not easily get the number from one scan, so we will accumulate into hasher directly

    let mut event_encountered = false;
    let mut logger = system.get_logger();
    for event in system.io.events_iterator() {
        if event.address != &DEPOSIT_CONTRACT_ADDRESS {
            continue;
        }
        if event.topics.len() > 0 && event.topics[0] == DEPOSIT_EVENT_SIGNATURE_HASH {
            if event_encountered == false {
                event_encountered = true;
                requests_hasher.update(&[DEPOSIT_REQUEST_EIP_7685_TYPE]);
            }
            let Ok(_) = validate_and_write_event_data(event, requests_hasher, &mut logger) else {
                panic!("invalid deposit event structure");
            };
        }
    }

    Ok(event_encountered)
}
