### Title
Gateway `validate_resource_bounds` checks against stale `l2_gas_price` instead of `next_l2_gas_price`, causing incorrect transaction admission - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The gateway's stateful validator computes its minimum-price threshold from the **current** block's `l2_gas_price`, but the next block will execute at `next_l2_gas_price` (the EIP-1559-adjusted price stored in the same block header). When the fee market price is falling, valid transactions are incorrectly rejected; when it is rising, transactions that will fail at blockifier execution are incorrectly admitted. A developer-acknowledged TODO in the source confirms the wrong field is being read.

### Finding Description

`StatefulTransactionValidator::validate_resource_bounds` reads the reference price as follows:

```rust
// TODO(Arni): getnext_l2_gas_price from the block header.
let previous_block_l2_gas_price = self
    .gateway_fixed_block_state_reader
    .get_block_info()
    .await?
    .gas_prices
    .strk_gas_prices
    .l2_gas_price;          // ← l2_gas_price of block N
``` [1](#0-0) 

`GatewayFixedBlockSyncStateClient::get_block_info_from_sync_client` populates `BlockInfo` from `block_header.l2_gas_price` and never exposes `block_header.next_l2_gas_price`:

```rust
strk_gas_prices: GasPriceVector {
    ...
    l2_gas_price: block_header.l2_gas_price.price_in_fri.try_into()?,
    // next_l2_gas_price is silently dropped
},
``` [2](#0-1) 

The consensus orchestrator, however, builds the next block's `ProposalInit` with `l2_gas_price_fri: self.l2_gas_price`, where `self.l2_gas_price` is the value computed by `calculate_next_l2_gas_price` — i.e., `next_l2_gas_price` of block N:

```rust
next_l2_gas_price: self.l2_gas_price,   // written into block N's header
``` [3](#0-2) 

```rust
fn update_l2_gas_price(&mut self, height: BlockNumber, l2_gas_used: GasAmount) {
    self.l2_gas_price = self.calculate_next_l2_gas_price(height, l2_gas_used);
    ...
}
``` [4](#0-3) 

The batcher then passes this value as the block's execution price to the mempool and blockifier:

```rust
mempool_client
    .update_gas_price(
        propose_block_input.block_info.gas_prices.strk_gas_prices.l2_gas_price.get(),
    )
``` [5](#0-4) 

The blockifier's pre-validation then enforces `tx.max_price_per_unit >= actual_block_l2_gas_price` (= `next_l2_gas_price` of block N):

```rust
if resource_bounds.max_price_per_unit < actual_gas_price.get() {
    insufficiencies_resource.push(ResourceBoundsError::MaxGasPriceTooLow { ... });
}
``` [6](#0-5) 

The EIP-1559 formula in `calculate_next_base_gas_price` can both raise and lower the price each block:

```rust
let adjusted_price_u256 =
    if gas_used > gas_target { price_u256 + price_change } else { price_u256 - price_change };
``` [7](#0-6) 

So the gateway threshold (`min_gas_price_percentage × l2_gas_price_N`) and the blockifier threshold (`next_l2_gas_price_N`) diverge every block.

### Impact Explanation

**Incorrect rejection (price falling):** When `next_l2_gas_price < l2_gas_price`, the gateway threshold is higher than the actual execution price. A transaction with `max_price_per_unit` in the range `[next_l2_gas_price, l2_gas_price)` satisfies the blockifier's requirement but is rejected by the gateway. Valid, fee-paying transactions are silently dropped before sequencing.

**Incorrect admission (price rising):** When `next_l2_gas_price > l2_gas_price`, the gateway threshold is lower than the actual execution price. A transaction with `max_price_per_unit` in the range `[threshold × l2_gas_price, next_l2_gas_price)` passes gateway admission but fails blockifier pre-validation, wasting mempool slots and batcher execution cycles.

Both cases match the allowed impact: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

### Likelihood Explanation

The EIP-1559 price adjusts every block based on gas usage. Any block with gas usage below the target lowers `next_l2_gas_price` below `l2_gas_price`, immediately triggering the incorrect-rejection path for any transaction priced between the two values. No special attacker capability is required; any ordinary user submitting a transaction at the correct next-block price is affected.

### Recommendation

1. Extend `GatewayFixedBlockSyncStateClient::get_block_info_from_sync_client` (or add a dedicated accessor) to surface `block_header_without_hash.next_l2_gas_price`.
2. In `validate_resource_bounds`, replace the read of `strk_gas_prices.l2_gas_price` with the `next_l2_gas_price` value, resolving the existing TODO comment.

### Proof of Concept

| Step | Value |
|------|-------|
| Block N `l2_gas_price` | 100 fri |
| Block N `next_l2_gas_price` (EIP-1559, low usage) | 90 fri |
| `min_gas_price_percentage` | 100 % |
| Transaction `max_price_per_unit` | 91 fri |
| Gateway threshold (`100% × 100`) | 100 fri |
| Gateway decision | **REJECT** (91 < 100) |
| Blockifier threshold (next block price = 90) | 90 fri |
| Blockifier decision (if reached) | **ACCEPT** (91 ≥ 90) |

A transaction that would execute successfully in block N+1 is permanently rejected at the gateway because the gateway compares against the stale `l2_gas_price` of block N rather than the `next_l2_gas_price` that block N+1 will actually enforce.
