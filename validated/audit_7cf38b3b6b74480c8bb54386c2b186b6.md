### Title
Zero `max_fee_per_gas` Bypasses All Relayer Fee Collection Due to Hardcoded Zero `block_base_fee_per_gas` - (File: `engine/src/engine.rs`)

### Summary
Because `block_base_fee_per_gas()` is hardcoded to return `U256::zero()`, any user who submits an EVM transaction with `max_fee_per_gas = 0` (or `gas_price = 0` in legacy transactions) causes `charge_gas` to immediately return a zero-fee result. The relayer who paid NEAR gas to process the transaction receives no ETH compensation, and the sender pays nothing for EVM execution.

### Finding Description
In `engine/src/engine.rs`, the `Backend` trait implementation for `Engine` hardcodes `block_base_fee_per_gas` to always return zero:

```rust
fn block_base_fee_per_gas(&self) -> U256 {
    U256::zero()
}
``` [1](#0-0) 

The `charge_gas` function reads this value and applies the following early-exit condition:

```rust
if transaction.max_fee_per_gas.is_zero()
    && fixed_gas.is_none()
    && block_base_fee_per_gas.is_zero()
{
    return Ok(GasPaymentResult::default());
}
``` [2](#0-1) 

Because `block_base_fee_per_gas` is **always** zero, the condition collapses to: if the user sets `max_fee_per_gas = 0` and no `fixed_gas` override is active, the function returns `GasPaymentResult::default()` — zero prepaid amount, zero effective gas price, zero priority fee. No ETH is deducted from the sender.

Downstream, `refund_unused_gas` also short-circuits on zero effective gas price, so the relayer receives no reward:

```rust
if gas_result.effective_gas_price.is_zero() {
    return Ok(());
}
``` [3](#0-2) 

The `NormalizedEthTransaction` struct carries `max_fee_per_gas` as a plain `U256` field with no enforced minimum: [4](#0-3) 

For legacy transactions, `gas_price` maps directly to `max_fee_per_gas`, and `TransactionLegacy` accepts any `U256` value including zero: [5](#0-4) 

### Impact Explanation
Any unprivileged EVM user can submit valid transactions with `gas_price = 0` / `max_fee_per_gas = 0`. The sender pays zero ETH for EVM execution. The relayer, who paid real NEAR gas to submit the transaction on-chain, receives zero ETH compensation. This constitutes theft of unclaimed yield (relayer fees) at scale: a high-volume actor can drain relayer profitability entirely, making relaying economically unviable and degrading the liveness of the Aurora network.

**Impact: High — Theft of unclaimed yield (relayer fees).**

### Likelihood Explanation
The condition is trivially reachable: any standard Ethereum wallet or script can set `gas_price = 0`. No special privileges, no contract deployment, no governance interaction is required. The `block_base_fee_per_gas` hardcoded zero is a permanent, unconditional property of the current codebase. Likelihood is **High**.

### Recommendation
- **Short term:** In `charge_gas`, enforce a minimum `effective_gas_price` floor (e.g., reject transactions where `max_fee_per_gas` is below a configurable minimum stored in `EngineState`). Alternatively, treat a zero `max_fee_per_gas` as invalid when `fixed_gas` is also absent.
- **Long term:** Replace the hardcoded `block_base_fee_per_gas() → U256::zero()` with a real, governance-configurable base fee stored in engine state. This mirrors EIP-1559 semantics and ensures the protocol always collects a minimum fee independent of user-supplied gas price values.

### Proof of Concept
1. Construct a `TransactionLegacy` with `gas_price: U256::zero()` and any valid `to`, `value`, and `data`.
2. Sign and submit it via the Aurora `submit` endpoint.
3. Observe that `charge_gas` returns `GasPaymentResult { prepaid_amount: Wei::zero(), effective_gas_price: U256::zero(), priority_fee_per_gas: U256::zero() }`.
4. Observe that `refund_unused_gas` returns `Ok(())` immediately without crediting the relayer.
5. The sender's ETH balance is unchanged; the relayer's ETH balance is unchanged; the transaction executes successfully.

This is directly confirmed by the existing unit test `test_gas_charge_for_empty_transaction_is_zero` in `engine/src/engine.rs` (lines 2478–2513), which asserts that a transaction with `max_fee_per_gas = U256::MAX` but `max_priority_fee_per_gas = U256::default()` and a zero `block_base_fee_per_gas` produces a zero prepaid amount — demonstrating the engine's own test suite validates this zero-fee path as reachable.
