### Title
Truncation in `delta_gas` Computation Causes Systematic Undercharge of Gas Fees — (`basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`)

### Summary

In `compute_gas_refund`, the conversion from native resource consumption back to EVM gas units uses integer division (floor truncation). This causes every transaction whose `native_used` is not an exact multiple of `native_per_gas` to be undercharged by exactly 1 gas unit. The operator is systematically underpaid by up to 1 gas unit per transaction, and the user receives a slightly larger refund than the protocol intends. This is the direct ZKsync OS analog of Velodrome Issue 6.4.8 (rounding error causing slightly reduced resource accounting).

### Finding Description

`compute_gas_refund` in `basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs` implements the double-resource reconciliation step described in `docs/double_resource_accounting.md`. After execution, it computes `delta_gas` to bridge the gap between EVM gas consumed and the gas implied by native resource consumption:

```rust
let delta_gas = if native_per_gas == 0 {
    0
} else {
    (native_used / native_per_gas) as i64 - (gas_used as i64)
};

if delta_gas > 0 {
    gas_used += delta_gas as u64;
}
// TODO: return delta_gas to gas_used?
``` [1](#0-0) 

The expression `native_used / native_per_gas` is Rust integer division, which truncates toward zero (floor). The documentation specifies the intended formula as:

```
deltaGas := (nativeUsed / nativePerGas) - gasUsed
``` [2](#0-1) 

When `delta_gas > 0`, the final `gas_used` is set to `floor(native_used / native_per_gas)`. The correct value to fully compensate the operator for native resource consumption is `ceil(native_used / native_per_gas)`. The difference is exactly 1 gas unit whenever `native_used % native_per_gas != 0`.

The `TODO` comment on line 80 acknowledges the symmetric case (`delta_gas < 0`) is also unhandled, but the undercharge direction (user benefits) is the security-relevant path here.

This function is called from both the ZK L2 transaction path and the L1 transaction path: [3](#0-2) [4](#0-3) 

The `native_per_gas` ratio is derived from `gas_price / native_price` (ceiling division at setup time): [5](#0-4) 

### Impact Explanation

Every transaction where `native_used % native_per_gas != 0` is undercharged by exactly 1 gas unit. The operator (`coinbase`) receives `gas_used * gas_price` where `gas_used` is 1 unit lower than it should be. The user's refund is correspondingly 1 gas unit larger than the protocol intends. The financial loss per transaction is bounded by `1 * gas_price` (e.g., at 1 gwei gas price, ≤ 1 gwei per transaction). The impact is negligible per transaction but is systematic and affects every transaction with a non-zero remainder in the native-to-gas conversion.

### Likelihood Explanation

The condition `native_used % native_per_gas != 0` holds for the vast majority of real transactions. `native_per_gas = ceil(gas_price / native_price)` is typically a small integer (e.g., 1–100), and `native_used` is an arbitrary accumulation of per-opcode native charges. The probability that `native_used` is exactly divisible by `native_per_gas` is approximately `1 / native_per_gas`, so the rounding error affects nearly all transactions. No special attacker action is required; it is triggered by any normal transaction submission.

### Recommendation

Replace the floor division with ceiling division when computing the gas equivalent of native consumption:

```rust
// Before (truncates):
(native_used / native_per_gas) as i64 - (gas_used as i64)

// After (ceiling, ensures operator is fully compensated):
(native_used.div_ceil(native_per_gas)) as i64 - (gas_used as i64)
```

Additionally, the `TODO` comment on line 80 should be resolved: if `delta_gas < 0`, the current behavior silently overcharges the user (they pay more EVM gas than their native consumption warrants). Decide whether to implement the refund or document the intentional asymmetry.

### Proof of Concept

Concrete numeric example:

- `gas_limit = 100_000`, `gas_price = 1000`, `native_price = 3`
- `native_per_gas = ceil(1000 / 3) = 334`
- Transaction executes, consuming `gas_used = 50_000` EVM gas and `native_used = 16_700_001` native units
- `native_used / native_per_gas = 16_700_001 / 334 = 50_000` (exact, no rounding here)

Now a case that triggers the bug:

- `native_used = 16_700_100`, `native_per_gas = 334`
- `floor(16_700_100 / 334) = 50_000` (since `16_700_100 = 334 * 50_000 + 100`)
- `ceil(16_700_100 / 334) = 50_001`
- `delta_gas = 50_000 - 50_000 = 0` → no extra gas charged
- Correct behavior: `delta_gas = 50_001 - 50_000 = 1` → 1 extra gas unit should be charged
- Result: user underpays by `1 * 1000 = 1000 wei`; operator is underpaid by the same amount

The undercharge is bounded to 1 gas unit per transaction but is systematic across all transactions where `native_used % native_per_gas != 0`. [6](#0-5)
