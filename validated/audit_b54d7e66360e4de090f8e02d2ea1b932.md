### Title
Unbounded `stakers` Vec Iteration in `get_stakers` Enables Griefing via Monotonically Growing Staker Registry - (File: src/staking/staking.cairo)

---

### Summary

The `stakers` storage `Vec<ContractAddress>` in the Staking contract grows monotonically: every `stake()` call appends to it, but stakers are **never removed** even after a full `unstake_intent` → `unstake_action` cycle. The `get_stakers` function iterates over the **entire** Vec on every invocation. An unprivileged actor can repeatedly stake with the minimum amount across many addresses and unstake to recover tokens, inflating the Vec without bound and eventually making `get_stakers` prohibitively expensive or impossible to execute.

---

### Finding Description

**Storage declaration — Vec grows forever:** [1](#0-0) 

The comment at line 168 explicitly states: *"Stakers are not removed from this vector when they unstake."*

**`stake()` appends unconditionally:** [2](#0-1) 

**`get_stakers` iterates the full Vec with no bound:** [3](#0-2) 

The loop calls `self.stakers.into_iter_full_range()`, visiting every address ever registered — including all historical stakers who have long since unstaked. There is no cap on the Vec's length, no pagination, and no on-chain enforcement of a maximum staker count.

A secondary self-acknowledged unbounded loop exists in `calculate_rewards` inside `pool.cairo`: [4](#0-3) 

The code comment at line 857–858 reads: *"This loop is unbounded but unlikely to exceed gas limits."* While `entry_to_claim_from` limits re-processing, the trace grows by one entry per epoch per pool member and is never pruned.

---

### Impact Explanation

`get_stakers` is the entry point of `IStakingConsensus` used by the consensus layer to retrieve the active validator set for a given epoch. [5](#0-4) 

If the `stakers` Vec grows large enough, this function will exceed Starknet's execution-step limit whether called on-chain (by another contract) or off-chain (by the sequencer/consensus client). The result is that the consensus layer can no longer obtain the validator set, constituting **unbounded gas consumption** and **griefing with damage to the protocol** — both within the allowed Medium impact scope.

---

### Likelihood Explanation

Any unprivileged staker can execute the inflation loop:

1. Call `stake()` with `min_stake` → address appended to `stakers`.
2. Call `unstake_intent()` → wait `exit_wait_window`.
3. Call `unstake_action()` → tokens returned; address **remains** in `stakers`.
4. Repeat from a fresh address (tokens are fully recovered minus gas).

The only cost per cycle is gas. With Starknet's low fees, an attacker can inflate the Vec to tens of thousands of entries at modest cost. No privileged role, leaked key, or external dependency is required.

---

### Recommendation

1. **Remove stakers on exit**: When `unstake_action` deletes a staker's info, also remove or tombstone the entry in `stakers` (e.g., swap-and-pop, or a separate active-staker set).
2. **Enforce a maximum staker count**: Add an on-chain cap and reject `stake()` calls that would exceed it.
3. **Paginate `get_stakers`**: Accept `offset` / `limit` parameters so callers can retrieve the list in bounded chunks.
4. **Address the `calculate_rewards` loop**: Document and enforce a maximum number of unclaimed epochs, or add a pruning mechanism for old balance-trace entries.

---

### Proof of Concept

```
// Attacker controls addresses A_1 … A_N, each funded with min_stake STRK.
for i in 1..N:
    A_i.stake(reward_address, operational_address, min_stake)
    // stakers Vec length += 1

// After exit_wait_window:
for i in 1..N:
    A_i.unstake_intent()
    // wait exit_wait_window
    A_i.unstake_action(A_i)
    // tokens returned; stakers Vec unchanged (length still N)

// Repeat with N fresh addresses using the recovered tokens.
// After K rounds: stakers Vec length = K*N.

// Eventually:
staking_contract.get_stakers(epoch_id)
// → iterates K*N entries → exceeds execution-step limit → reverts / times out
``` [6](#0-5) 

The `unstake_action` function transfers tokens back to the staker but never removes the address from `stakers`, confirming the permanent inflation of the Vec.

        fn get_stakers(
            self: @ContractState, epoch_id: Epoch,
        ) -> Span<(ContractAddress, StakingPower, Option<PublicKey>, Option<PeerId>)> {
            let curr_epoch = self.get_current_epoch();
            assert!(
                curr_epoch <= epoch_id && epoch_id < curr_epoch + K.into(),
                "{}",
                Error::INVALID_EPOCH,
            );

            let (strk_total_stake, btc_total_stake) = self
                .get_total_staking_power_at_epoch(:epoch_id);

            let mut stakers: Array<
                (ContractAddress, StakingPower, Option<PublicKey>, Option<PeerId>),
            > =
                array![];
            for staker_address_ptr in self.stakers.into_iter_full_range() {
                let staker_address = staker_address_ptr.read();
                if !self.is_staker_active(:staker_address, :epoch_id) {
                    continue;
                }

                let staking_power = self
                    .get_staker_staking_power_at_epoch(
                        :staker_address, :epoch_id, :strk_total_stake, :btc_total_stake,
                    );
                if staking_power.is_zero() {
                    continue;
                }

                let public_key = self.get_public_key_at_epoch(:staker_address, :epoch_id);
                let peer_id = self.get_peer_id_at_epoch(:staker_address, :epoch_id);
                stakers.append((staker_address, staking_power, public_key, peer_id));
            }
            stakers.span()
        }
