### Title
`CadenceOwnedAccount` Resource Can Be Destroyed While Holding EVM Balance, Permanently Locking FLOW Tokens - (File: fvm/evm/stdlib/contract.cdc)

### Summary

The `CadenceOwnedAccount` (COA) resource in `fvm/evm/stdlib/contract.cdc` can be destroyed while its underlying EVM address still holds a non-zero FLOW token balance. Because `withdraw` is gated behind the `Owner | Withdraw` entitlement and only the COA resource itself can authorize withdrawals from its EVM address, destroying the COA without first draining its balance permanently locks those tokens in the EVM state with no recovery path.

### Finding Description

The `CadenceOwnedAccount` resource wraps a deterministically-derived EVM address and is the sole authorized agent for withdrawing FLOW from that address. The `withdraw` function enforces this:

```cadence
access(Owner | Withdraw)
fun withdraw(balance: Balance): @FlowToken.Vault { ... }
``` [1](#0-0) 

On the Go side, `Account.Withdraw` calls `executeAndHandleAuthorizedCall`, which enforces `isAuthorized == true` — a flag only set when the call originates from the COA resource itself:

```go
func (a *Account) executeAndHandleAuthorizedCall(...) (*types.Result, error) {
    if !a.isAuthorized {
        return nil, types.ErrUnauthorizedMethodCall
    }
    ...
}
``` [2](#0-1) 

The `CadenceOwnedAccount` resource definition contains no Cadence `destroy` destructor and no `pre`-condition that checks `self.balance().attoflow == 0` before allowing destruction: [3](#0-2) 

This means a Cadence transaction can legally execute:

```cadence
let coa <- EVM.createCadenceOwnedAccount()
coa.deposit(from: <-vault)          // deposits FLOW into EVM
// ... partial withdraw or no withdraw ...
destroy coa                          // COA gone; EVM balance permanently locked
```

The EVM address still exists in the EVM state with its balance, but no entity can ever authorize a withdrawal from it again.

A secondary scenario arises from the minimum-withdrawable-amount constraint (`1e10 attoFlow`). If a COA accumulates a sub-threshold remainder (e.g., from gas refunds or rounding), the user literally **cannot** withdraw it, yet can still destroy the COA, permanently locking that remainder: [4](#0-3) [5](#0-4) 

### Impact Explanation

FLOW tokens deposited into a COA's EVM address become permanently inaccessible when the COA resource is destroyed without a full balance withdrawal. The EVM total-supply accounting (`bp.TotalSupply`) is not reduced on destruction — only on a successful `Withdraw` call — so the locked tokens create a permanent discrepancy between the EVM total supply and the actually-accessible supply: [6](#0-5) 

This is a **cross-VM asset loss**: FLOW tokens move from Cadence into EVM via `deposit`, but can never return to Cadence if the COA is destroyed while holding a balance.

### Likelihood Explanation

The scenario is reachable by any unprivileged transaction sender:

1. **User mistake**: A user creates a COA, deposits FLOW, performs some EVM operations, and destroys the COA without realizing a non-zero balance (or a sub-threshold dust remainder) remains. Test code in the repository already demonstrates this exact pattern — `destroy cadenceOwnedAccount` after a partial withdraw — without any runtime guard preventing it. [7](#0-6) 

2. **Dust lock**: Because the minimum withdrawable unit is `1e10 attoFlow`, any COA that accumulates a remainder below this threshold (e.g., from EVM gas refunds credited in attoFlow) cannot be fully drained before destruction.

3. **Malicious contract**: A third-party Cadence contract that accepts a COA capability or resource transfer could destroy it without withdrawing the balance, causing the original depositor to lose funds.

### Recommendation

Add a `pre`-condition to the Cadence `CadenceOwnedAccount` resource that prevents destruction when the EVM balance is non-zero:

```cadence
// In CadenceOwnedAccount resource, add a destroy function:
destroy() {
    pre {
        self.balance().attoflow == 0:
            "EVM.CadenceOwnedAccount: cannot destroy a COA with a non-zero EVM balance"
    }
}
```

Alternatively, emit a `ResourceDestroyed` event and automatically transfer any remaining balance to a designated recovery address (e.g., the Flow service account or a protocol-controlled vault) before destruction.

### Proof of Concept

```cadence
import EVM from <EVMContractAddress>
import FlowToken from <FlowTokenAddress>

transaction() {
    prepare(account: auth(BorrowValue) &Account) {
        let admin = account.storage.borrow<&FlowToken.Administrator>(
            from: /storage/flowTokenAdmin
        )!
        let minter <- admin.createNewMinter(allowedAmount: 1.0)
        let vault <- minter.mintTokens(amount: 1.0)
        destroy minter

        // Create COA and deposit 1.0 FLOW into EVM
        let coa <- EVM.createCadenceOwnedAccount()
        coa.deposit(from: <-vault)

        // Verify balance is non-zero
        let bal = coa.balance()
        assert(bal.attoflow > 0, message: "balance should be non-zero")

        // Destroy COA without withdrawing — 1.0 FLOW is now permanently locked
        destroy coa

        // The EVM address still exists with 1.0 FLOW, but no one can ever
        // authorize a withdrawal from it. The tokens are permanently lost.
    }
}
```

The `CadenceOwnedAccount` resource has no destructor guard, so this transaction succeeds and the deposited FLOW is permanently inaccessible. [8](#0-7)

                    "EVM.CadenceOwnedAccount.initAddress(): Cannot initialize the address bytes if it has already been set!"

        /// Withdraws the balance from the cadence owned account's balance.
        /// Note that amounts smaller than 1e10 attoFlow can't be withdrawn,
        /// given that Flow Token Vaults use UFix64 to store balances.
        /// In other words, the smallest withdrawable amount is 1e10 attoFlow.
        /// Amounts smaller than 1e10 attoFlow, will cause the function to panic
        /// with: "withdraw failed! smallest unit allowed to transfer is 1e10 attoFlow".
        /// If the given balance conversion to UFix64 results in rounding loss,
        /// the withdrawal amount will be truncated to the maximum precision for UFix64.
        ///
        /// @param balance: The EVM balance to withdraw
        ///
        /// @return A FlowToken Vault with the requested balance
        access(Owner | Withdraw)
        fun withdraw(balance: Balance): @FlowToken.Vault {
            pre {
                !EVM.isPaused(): "EVM operations are temporarily paused"
            }

            if balance.isZero() {
                return <-FlowToken.createEmptyVault(vaultType: Type<@FlowToken.Vault>())
            }
            let vault <- InternalEVM.withdraw(
                from: self.addressBytes,
                amount: balance.attoflow
            ) as! @FlowToken.Vault
            emit FLOWTokensWithdrawn(
                address: self.address().toString(),
                amount: balance.inFLOW(),
                withdrawnUUID: vault.uuid,
                balanceAfterInAttoFlow: self.balance().attoflow
            )
            return <-vault
        }
