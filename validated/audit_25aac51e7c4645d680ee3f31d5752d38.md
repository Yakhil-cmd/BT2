### Title
ETH Permanently Locked in `RewardSupplier` When `tick()` Called With No Pending Mint Requests - (File: `L1/starkware/solidity/stake/RewardSupplier.sol`)

---

### Summary

The `tick()` function in `RewardSupplier.sol` is declared `payable` and accepts ETH as message fees for L1→L2 bridge and messaging calls. However, the ETH forwarding logic is gated inside an `if (amountToMint > 0)` branch. When `tick()` is called with ETH but no pending mint requests exist (`amountToMint == 0`), the entire `msg.value` is permanently locked in the contract with no recovery path.

---

### Finding Description

`tick()` is the public entry point for the L1 reward minting pipeline:

```solidity
function tick() external payable {
    (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

    if (amountToMint > 0) {
        // ...
        uint256 msgFee = msg.value / 2;
        bridge().depositWithMessage{value: msgFee}(
            token(), amountToMint, mintDestination(), new uint256[](0)
        );
        // ...
        msgFee = msg.value - msgFee;
        messagingContract().sendMessageToL2{value: msgFee}(
            mintingCurve(), UPDATE_TOTAL_SUPPLY_SELECTOR, messagePayload
        );
    }
    // ← no else branch, no refund, no revert
}
``` [1](#0-0) 

The function is `payable` and is designed to be called with ETH (to cover StarkGate bridge fees and `sendMessageToL2` fees). The ETH is only consumed inside the `if (amountToMint > 0)` block. When the condition is false, `msg.value` is silently retained by the contract. There is no `receive()`, `fallback()`, or withdrawal function in `RewardSupplier.sol`, and the contract inherits only `RewardSupplierStorage`, `Identity`, and `ProxySupportImpl` — none of which provide an ETH recovery mechanism. [2](#0-1) 

The analog to the original report is direct: the original bug had a function that **could** forward value but the caller hardcoded `0`; here the function **accepts** ETH but silently discards it when the branch condition is false — both are partial implementations of value-forwarding logic.

---

### Impact Explanation

Any ETH sent to `tick()` when `amountToMint == 0` is permanently locked in the `RewardSupplier` contract. This maps to **"Temporary freezing of funds"** (Medium/High) in the allowed impact scope. The locked ETH belongs to the caller (the entity paying message fees), not to stakers, but the loss is irreversible and the contract provides no recovery path.

---

### Likelihood Explanation

**Medium.** Two realistic paths exist:

1. **Accidental call**: A keeper or bot calls `tick()` with ETH without first checking `requiredMinting()`. This is plausible in automated pipelines.

2. **Front-running griefing**: An attacker observes a pending `tick()` call with ETH in the Ethereum mempool. The attacker front-runs with their own `tick()` call (also with ETH) that consumes all pending L2→L1 mint-request messages. The victim's `tick()` then executes with `amountToMint == 0`, and their ETH is permanently locked. The attacker spends ETH but causes the victim to lose theirs — a griefing attack with no profit motive.

---

### Recommendation

Add a guard at the top of `tick()` that either:
- Reverts if `msg.value > 0` and `amountToMint == 0`, or
- Refunds `msg.value` to `msg.sender` when the `if` branch is not taken:

```solidity
function tick() external payable {
    (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

    if (amountToMint > 0) {
        // ... existing logic ...
    } else {
        // Refund any ETH sent when there is nothing to mint.
        if (msg.value > 0) {
            (bool ok, ) = msg.sender.call{value: msg.value}("");
            require(ok, "ETH_REFUND_FAILED");
        }
    }
}
```

---

### Proof of Concept

1. L2 `RewardSupplier` has no pending mint requests (all messages already consumed, or none sent yet).
2. A keeper calls `tick{value: 0.01 ether}()` on L1 `RewardSupplier`.
3. `requiredMinting()` returns `(0, 0)` because `messagingContract().l2ToL1Messages(msgHash) == 0`.
4. The `if (amountToMint > 0)` branch is skipped.
5. `tick()` returns normally. The `0.01 ether` is now held by `RewardSupplier` with no withdrawal function.
6. The ETH is permanently locked. [1](#0-0)

    function tick() external payable {
        // Check if minting is required, and how much.
        (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

        if (amountToMint > 0) {
            // Prepare the L2->L1 mintRequest message for consumption.
            uint256[] memory messagePayload = new uint256[](1);
            messagePayload[0] = TOKENS_PER_MINT_REQUEST;

            // Consume the mintRequest messages.
            for (uint256 i = 0; i < numMsgsToConsume; i++) {
                messagingContract().consumeMessageFromL2(mintRequestSource(), messagePayload);
            }

            // Reuest minting of the requested amount from the mint manager.
            mintManager().mintRequest(token(), amountToMint);

            // Deposit the minted amount onto the bridge to the credit of `mintDestination`.
            uint256 msgFee = msg.value / 2;
            bridge().depositWithMessage{value: msgFee}(
                token(),
                amountToMint,
                mintDestination(),
                new uint256[](0)
            );
            emit ConsumedL2MintRequests(numMsgsToConsume, amountToMint);

            // Send a totalSupply update to L2MintCurve.
            msgFee = msg.value - msgFee;
            messagePayload[0] = IERC20(token()).totalSupply();
            messagingContract().sendMessageToL2{value: msgFee}(
                mintingCurve(),
                UPDATE_TOTAL_SUPPLY_SELECTOR,
                messagePayload
            );
        }
    }
