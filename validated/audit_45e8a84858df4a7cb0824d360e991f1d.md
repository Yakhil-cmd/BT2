I need to verify the key technical claims: whether `Cycles` subtraction is saturating, how the rate limiter check behaves with zero, and whether `deposit_cycles` still sends the full amount after `ensure_balance` returns.

Audit Report

## Title
Rate-Limiter Bypass and `total_cycles_minted` Understatement via Donated Cycles in `ensure_balance` — (File: rs/nns/cmc/src/main.rs)

## Summary
The `ensure_balance` function in the Cycles Minting Canister computes `cycles_to_mint` as the saturating difference between the requested cycles and the CMC's live `canister_balance128()`. Because `Cycles` subtraction is saturating, any cycles donated to the CMC by an unprivileged caller reduce `cycles_to_mint` to zero. This silently bypasses the rate limiter (charged zero), understates `total_cycles_minted` (incremented by zero), and mints zero new cycles — while the CMC still forwards the full requested cycles from its existing balance and burns the corresponding ICP.

## Finding Description
`ensure_balance` at `rs/nns/cmc/src/main.rs` lines 2306–2325 reads the CMC's total live balance:

```rust
let current_balance = Cycles::from(ic_cdk::api::canister_balance128());
let cycles_to_mint = cycles - current_balance;  // saturating_sub → 0 when balance ≥ cycles
```

`Cycles::sub` is implemented as `saturating_sub` (`rs/types/cycles/src/cycles.rs` lines 133–139), so when `current_balance >= cycles`, `cycles_to_mint` is zero. The rate limiter at `rs/nns/cmc/src/limiter.rs` lines 43–54 checks `count + cycles_to_mint > limit`; with `cycles_to_mint = 0` this check always passes and adds nothing to the running count. `state.total_cycles_minted += 0` and `ic0_mint_cycles128(0)` are called. After `ensure_balance` returns, `deposit_cycles` at line 2126 still calls `call_with_payment128(..., u128::from(cycles))` — the full original `cycles` amount — forwarding it to the target canister from the CMC's existing (donated) balance. `burn_and_log` then burns the ICP. The same path is triggered for `notify_top_up` (line 1999), `do_mint_cycles` (line 2151), and canister-creation flows.

Any canister on any subnet can call `management_canister.deposit_cycles` targeting the CMC's canister ID, inflating `canister_balance128()` without any privileged access. Once the CMC's balance exceeds the per-conversion `cycles` amount, every subsequent ICP-to-cycles conversion in that period charges the rate limiter zero and increments `total_cycles_minted` by zero, regardless of how much ICP is burned.

## Impact Explanation
This is a **High** severity finding. The rate limiter is the sole on-chain guard preventing burst ICP-to-cycles conversions in the CMC, a core NNS canister. Bypassing it allows an unbounded amount of ICP to be burned in a single rate-limit period without the limiter firing, breaking the protocol's intended conversion-rate cap. Additionally, `total_cycles_minted` — a public query used by governance and monitoring tooling to audit the ICP-to-cycles conservation invariant — silently diverges from the true amount of cycles distributed. This constitutes a significant NNS/CMC security impact with concrete protocol harm, matching the High bounty tier for significant NNS infrastructure security impact.

## Likelihood Explanation
The precondition is trivially achievable: any canister on any subnet can call `deposit_cycles` on the management canister targeting the CMC's well-known canister ID, attaching cycles as payment. Cycles can be sourced from chain-fusion bridges (ckETH, ckBTC) or any canister that has accumulated cycles, without having previously been subject to the CMC rate limiter. No privileged access, governance majority, threshold-crypto compromise, or victim mistake is required. The attacker's cost is the donated cycles, which are recouped when their own ICP-to-cycles conversions succeed without the limiter firing. The attack is repeatable as long as the CMC's balance remains above the per-conversion threshold.

## Recommendation
Track the CMC's "minted-from-ICP" balance separately from its total live balance. Introduce a `cycles_held_from_minting: Cycles` field in `State` that is incremented by `cycles_to_mint` after each successful `ic0_mint_cycles128` call and decremented when cycles are forwarded out. Use this field — not `canister_balance128()` — to compute `cycles_to_mint` in `ensure_balance`:

```rust
let cycles_to_mint = cycles.saturating_sub(state.cycles_held_from_minting);
```

This ensures that donated cycles do not reduce the amount the rate limiter is charged, and that `total_cycles_minted` accurately reflects all cycles distributed from ICP conversions.

## Proof of Concept
1. Deploy an attacker canister on any subnet.
2. Attacker canister calls `management_canister.deposit_cycles({ canister_id: CMC_CANISTER_ID })` with `LARGE_CYCLES` attached — donating them to the CMC.
3. CMC's `canister_balance128()` is now `LARGE_CYCLES + prior_balance`.
4. Call `notify_top_up` for `N` ICP (≡ `X` cycles, where `X ≤ LARGE_CYCLES`).
5. `ensure_balance(X)` runs: `cycles_to_mint = X - (LARGE_CYCLES + prior_balance) = 0` (saturating).
6. `check_and_add_cycles(state, now, 0)` — rate limiter not charged; `state.total_cycles_minted += 0`.
7. `deposit_cycles` forwards `X` cycles to the target canister from the CMC's existing balance.
8. `burn_and_log` burns `N` ICP.
9. Repeat steps 4–8 for any number of conversions until `LARGE_CYCLES` is exhausted; the rate limiter never fires regardless of total ICP burned.
