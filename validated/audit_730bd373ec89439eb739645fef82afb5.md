### Title
Pool `claim_rewards` Bypasses Staking Contract Pause — (`File: src/pool/pool.cairo`)

---

### Summary

The `Pool` contract has no pause mechanism of its own. When the `Staking` contract is paused, pool members can still call `Pool::claim_rewards` to claim accumulated yield, because that function does not invoke any state-changing function on the `Staking` contract and therefore never hits the `general_prerequisites()` pause guard.

---

### Finding Description

The `Staking` contract enforces a pause via `general_prerequisites()`, which is called at the top of every state-changing entry point. [1](#0-0) 

The staker-side `claim_rewards` is correctly guarded: [2](#0-1) 

The pause test suite confirms this expectation: [3](#0-2) 

However, `Pool::claim_rewards` in `src/pool/pool.cairo` performs its entire execution without touching any state-changing function on the `Staking` contract:

1. It reads `pool_member_info` from pool-local storage.
2. It calls `get_current_checkpoint`, which internally calls `staking_dispatcher.get_current_epoch()` — a **view** (read-only) function that is never blocked by the pause.
3. It calculates rewards from the pool-local `cumulative_rewards_trace`.
4. It writes updated `pool_member_info` back to pool-local storage.
5. It transfers STRK tokens directly from the pool contract's balance to the reward address. [4](#0-3) 

None of steps 1–5 calls a guarded function on the `Staking` contract. The pool's internal `get_current_epoch` helper only dispatches a read call: [5](#0-4) 

The `Pool` contract has no `is_paused` storage field, no pause role, and no check against the staking contract's pause state anywhere in its implementation. [6](#0-5) 

---

### Impact Explanation

When the `Staking` contract is paused — typically because an accounting bug, reward miscalculation, or active exploit has been detected — the intent is to freeze all fund movements and state changes. The staker-side `claim_rewards` is correctly frozen. But pool members can still drain their accumulated (potentially mis-calculated) STRK rewards through `Pool::claim_rewards` without restriction.

If the pause was triggered precisely because rewards were over-minted or incorrectly distributed (e.g., a bug in `update_rewards_from_attestation_contract` or `cumulative_rewards_trace` accounting), pool members can extract those inflated rewards before the protocol team can remediate the issue. This constitutes **theft of unclaimed yield** — a **High** severity impact under the allowed scope.

---

### Likelihood Explanation

Any pool member (an unprivileged delegator) can call `Pool::claim_rewards` at any time. No special role or privileged access is required. The attacker only needs to be a registered pool member with accumulated rewards. The entry path is fully reachable by an ordinary delegator.

---

### Recommendation

Add a check in `Pool::claim_rewards` (and, for consistency, `Pool::change_reward_address`) that queries the staking contract's pause state and reverts if it is paused:

```cairo
fn claim_rewards(ref self: ContractState, pool_member: ContractAddress) -> Amount {
    // Add pause guard
    let staking_dispatcher = IStakingDispatcher {
        contract_address: self.staking_pool_dispatcher.contract_address.read(),
    };
    assert!(!staking_dispatcher.is_paused(), "Contract is paused");
    // ... rest of function
}
```

Alternatively, introduce a dedicated pause flag in the `Pool` contract that is set/cleared by the `Staking` contract when it pauses/unpauses.

---

### Proof of Concept

1. Deploy the system normally and have a pool member accumulate rewards over several epochs.
2. Trigger a pause on the `Staking` contract via the security agent role.
3. Confirm that `staking.claim_rewards(staker_address)` reverts with `"Contract is paused"`.
4. Call `pool.claim_rewards(pool_member)` directly on the `Pool` contract.
5. Observe that the call succeeds and STRK tokens are transferred to the pool member's reward address, bypassing the pause. [4](#0-3) [3](#0-2)

        fn claim_rewards(ref self: ContractState, pool_member: ContractAddress) -> Amount {
            // Asserts.
            let mut pool_member_info = self.internal_pool_member_info(:pool_member);
            let caller_address = get_caller_address();
            let reward_address = pool_member_info.reward_address;
            assert!(
                caller_address == pool_member || caller_address == reward_address,
                "{}",
                Error::POOL_CLAIM_REWARDS_FROM_UNAUTHORIZED_ADDRESS,
            );

            let until_checkpoint = self.get_current_checkpoint(:pool_member);

            // Calculate rewards and update entry_to_claim_from.
            let (mut rewards, updated_entry_to_claim_from) = self
                .calculate_rewards(
                    :pool_member,
                    from_checkpoint: pool_member_info.reward_checkpoint,
                    :until_checkpoint,
                    entry_to_claim_from: pool_member_info.entry_to_claim_from,
                );
            rewards += pool_member_info._unclaimed_rewards_from_v0;
            pool_member_info._unclaimed_rewards_from_v0 = Zero::zero();
            pool_member_info.entry_to_claim_from = updated_entry_to_claim_from;
            pool_member_info.reward_checkpoint = until_checkpoint;

            // Write the updated pool member info to storage.
            self.write_pool_member_info(:pool_member, :pool_member_info);

            // Transfer rewards to the pool member.
            let reward_token = IERC20Dispatcher { contract_address: STRK_TOKEN_ADDRESS };
            reward_token.checked_transfer(recipient: reward_address, amount: rewards.into());

            // Emit event.
            self
                .emit(
                    Events::PoolMemberRewardClaimed {
                        pool_member, reward_address, amount: rewards,
                    },
                );

            rewards
        }
