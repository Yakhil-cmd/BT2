### Title
WalletConnect Commands Execute Against Wallet After GUI Logout Due to No-Op `handleLogOut` — (`packages/core/src/components/Auth/AuthProvider.tsx`)

### Summary

`handleLogOut` is intentionally a no-op. The Chia daemon remains logged in after the user clicks "Log Out" in the GUI. Because both the renderer-side fingerprint guard (`useWalletConnectCommand`) and the main-process guard (`dispatchPairRequest`) derive the active fingerprint by querying the daemon at request time, all WalletConnect authorization checks continue to pass. An established WalletConnect dApp can send `session_request` events after logout and have them processed — silently for bypass-listed commands, or via a confirmation dialog for non-bypassed ones — against a wallet the user believes they have logged out of.

---

### Finding Description

**1. `handleLogOut` is a no-op** [1](#0-0) 

`LogoutButton` calls `await logOut()` (no-op) then `navigate('/')`. The daemon's `get_logged_in_fingerprint` still returns the active fingerprint. [2](#0-1) 

**2. WalletConnect sessions are not torn down on logout**

`WalletConnectProvider` registers a `session_request` listener on the WalletConnect client. Nothing in the logout path calls `disconnectPair` or removes this listener. [3](#0-2) 

**3. Renderer-side fingerprint guard reads stale/live daemon state**

`useWalletConnectCommand` reads `currentFingerprint` from `useGetLoggedInFingerprintQuery()`, which polls the daemon. Since the daemon is still logged in, `currentFingerprint` still equals `pair.fingerprint`, so the guard at lines 38–46 passes. [4](#0-3) 

**4. Main-process guard also queries the daemon live**

`dispatchPairRequest` calls `getLoggedInFingerprint()` which issues `get_logged_in_fingerprint` to `chia_wallet`. Since the daemon is still logged in, `loggedInFingerprint === pair.fingerprint`, and the check at lines 41–44 passes. [5](#0-4) [6](#0-5) 

**5. Bypass-listed commands execute with zero user interaction**

If the dApp has any command in `pair.bypass`, `dispatchPairRequest` skips the confirmation dialog entirely and calls `process(context)` directly. [7](#0-6) 

The `DISPATCH_AS_PAIR` IPC handler in `main.tsx` wires this all together: [8](#0-7)

### Impact Explanation

A WalletConnect dApp that the user previously approved and that has at least one bypass-listed command (e.g., `chia_getSyncStatus`, or any command the user checked "always allow" for) can send `session_request` events after the user clicks "Log Out" and have them processed silently, with no confirmation dialog and no indication to the user. For non-bypassed commands, a confirmation dialog still appears — but it appears on a wallet the user believes they have logged out of, violating the logout invariant and creating a signing-context confusion scenario. The direct impact is unauthorized WalletConnect command execution (including signing and transaction commands) against a wallet the user believed was no longer active.

---

### Likelihood Explanation

- Requires a previously-approved WalletConnect pair (prior user consent), which is a realistic precondition for any WalletConnect dApp interaction.
- The user only needs to click "Log Out" — a normal, expected action.
- The dApp only needs to send a `session_request` after logout; no special timing or race condition is required since the daemon stays logged in indefinitely until the app is closed.
- Bypass permissions are user-granted but common for dApps that request "always allow" for read or low-risk commands.

---

### Recommendation

On logout, either:
1. Disconnect all active WalletConnect pairs/sessions (call `disconnectPair` for each pair in `pairs`), or
2. Set a renderer-side "logged out" flag that `processSessionRequest` and `handleProcess` check before processing any request, rejecting with `USER_REJECTED` if the user is not logged in at the GUI level.

The no-op `handleLogOut` comment acknowledges a missing backend API, but the WalletConnect session teardown can be done entirely in the renderer/main process without a new daemon API.

---

### Proof of Concept

1. Establish a WalletConnect pair with a dApp; grant `chia_sendTransaction` with "always allow" (bypass).
2. Click "Log Out" in the GUI — the app navigates to `/` (wallet selection screen).
3. From the dApp, immediately send a `chia_sendTransaction` `session_request`.
4. Observe: the request is processed by `handleProcess` → `dispatchAsPair` → `dispatchPairRequest` → `process(context)` without any confirmation dialog, returning a result rather than a rejection.
5. Expected (secure) behavior: the request should be rejected with `USER_REJECTED` because the user has logged out.

    pairTopic: string,

    const { fingerprint } = params;

    // verify if pair exists

    if (!pair) {
      throw new WcError(`Pair not found`, WcErrorCode.INTERNAL_ERROR);

    if (ctx.mainnet !== pair.mainnet) {
      throw new WcError(`Network mismatch`, WcErrorCode.UNSUPPORTED_CHAINS);
    }

    // verify if pair allows the requested command
    if (!pair.commands.includes(command)) {
      throw new WcError(`Command not allowed for this pair`, WcErrorCode.UNAUTHORIZED_METHOD);
    }

    // verify if pair allows the requested fingerprint

      !requestedFingerprint ||
      requestedFingerprint !== pair.fingerprint ||
      currentFingerprint !== pair.fingerprint

      throw new WcError(`Fingerprint not allowed for this command`, WcErrorCode.UNAUTHORIZED_METHOD);

ipcMainHandle(
  PermissionsAPI.DISPATCH_AS_PAIR,
  async (payload: {
    topic: string;
    command: string;
    params: string; // serialized params because of bigints
  }) => {
    const { topic, command, params } = payload;

    try {
      if (!mainWindow) {
        throw new WcError('mainWindow is empty', WcErrorCode.INTERNAL_ERROR);
      }

      const dappCommandSchema = DappCommands.get(command);
      if (!dappCommandSchema) {
        throw new WcError(`Unknown wc command: ${command}`, WcErrorCode.METHOD_NOT_FOUND);
      }

      const { commandId } = dappCommandSchema;
      const parsedParams = parseDappParams(command, params);

      // verify all permissions and execute command after user confirmation
      const result = await dispatchPairRequest(
        topic,
        command,
        parsedParams,
        // process the command
        async (context) => {
          const { destination, command: chiaCommand } = parseCommandId(commandId);

          const response = dappCommandSchema.handler
            ? await dappCommandSchema.handler(parsedParams, {
                ...context,
                sendNotification: sendRendererNotification,
                canBypassCommand: (requestedCommand) =>
                  DappCommands.get(requestedCommand)?.allowConfirmationBypass === true,
              })
            : await sendCommand(chiaCommand, destination, parsedParams);

          const transformedResponse = dappCommandSchema.transform ? dappCommandSchema.transform(response) : response;

          // dapp is sending back camelCase response
          const camelCaseResponse = toCamelCase(transformedResponse as Record<string, unknown>, {
            deep: !dappCommandSchema.preserveNestedDataKeys,
          });

          return dappCommandSchema.handler ? camelCaseResponse : { data: camelCaseResponse };
        },
        // show the confirm dialog to the user
        async () => {
          // humanize all data from command
          const { title, message, confirmLabel, destructive, rows } = await humanizeDappCommand(
            command,
            parsedParams,
            networkPrefix,
          );

          const pair = findPair(topic);
          if (!pair) {
            throw new WcError(`Pair not found`, WcErrorCode.USER_REJECTED);
          }

          if (!mainWindow) {
            throw new WcError('mainWindow is empty', WcErrorCode.INTERNAL_ERROR);
          }

          const display = await parseCommandDisplay(commandId, parsedParams);

          const confirmResult = await openReactDialog<ConfirmDialogResult, ConfirmProps>(
            mainWindow,
            Confirm,
            {
              networkPrefix,
              command: commandId,
              data: parsedParams,
              title,
              message,
              confirmLabel,
              destructive,
              rows,
              pair,
              display,
              showBypassToggle: dappCommandSchema.allowConfirmationBypass === true,
            },
            {
              title,
              width: 640,
              height: 600,
            },
          );

          if (confirmResult && confirmResult.isAllowed === true) {
            if (confirmResult.rememberBypass && dappCommandSchema.allowConfirmationBypass === true) {
              addBypassCommand(topic, command);
            }

            return true;
          }

          throw new WcError('Operation cancelled by user', WcErrorCode.USER_REJECTED);
        },
      );

      return JSONbig.stringify(result);
