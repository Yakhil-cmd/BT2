### Title
Missing Zero-Address Validation for `reward_address` Enables Permanent Freezing of Unclaimed Yield — (File: `src/staking/staking.cairo`, `src/pool/pool.cairo`)

---

### Summary

Both the `Staking` and `Pool` contracts accept a user-specified `reward_address` parameter — the destination to which all accrued STRK rewards are routed — without validating it against the zero address. This is the direct structural analog of the Across V2 `repaymentChainId` issue: a user-controlled routing/destination parameter is accepted without validation against a set of acceptable values. If the zero address is supplied, the protocol silently routes all future reward claims to an uncontrolled, unrecoverable address, permanently freezing the user's unclaimed yield.

---

### Finding Description

**In `src/staking/staking.cairo`:**

`Staking::stake` (line 288) accepts `reward_address` with a single guard:

```cairo
assert!(
    !self.does_token_exist(token_address: reward_address),
    "{}",
    GenericError::REWARD_ADDRESS_IS_TOKEN,
);
``` [1](#0-0) 

There is no `assert!(reward_address.is_non_zero(), ...)` check. The zero address is not a registered token, so this guard passes silently.

`Staking::change_reward_address` (line 517) applies the identical single guard and no zero-address check:

```cairo
assert!(
    !self.does_token_exist(token_address: reward_address),
    "{}",
    GenericError::REWARD_ADDRESS_IS_TOKEN,
);
``` [2](#0-1) 

When `claim_rewards` is later called, `send_rewards_to_staker` reads the stored `reward_address` and unconditionally transfers to it:

```cairo
token_dispatcher.checked_transfer(recipient: reward_address, amount: amount.into());
``` [3](#0-2) 

**In `src/pool/pool.cairo`:**

`Pool::enter_delegation_pool` (line 182) checks only that `reward_address` is not the pool token:

```cairo
assert!(token_address != reward_address, "{}", GenericError::REWARD_ADDRESS_IS_TOKEN);
``` [4](#0-3) 

`Pool::change_reward_address` (line 505) applies the same single guard:

```cairo
assert!(
    self.token_dispatcher.contract_address.read() != reward_address,
    "{}",
    GenericError::REWARD_ADDRESS_IS_TOKEN,
);
``` [5](#0-4) 

When `Pool::claim_rewards` executes, it reads the stored `reward_address` and transfers directly to it:

```cairo
let reward_token = IERC20Dispatcher { contract_address: STRK_TOKEN_ADDRESS };
reward_token.checked_transfer(recipient: reward_address, amount: rewards.into());
``` [6](#0-5) 

A grep across the entire repository for any zero-address guard on `reward_address` returns no matches, confirming the absence of this check in all four entry points (`stake`, `change_reward_address` in staking, `enter_delegation_pool`, `change_reward_address` in pool).

---

### Impact Explanation

A staker or pool member who registers or updates their `reward_address` to the zero address will have every subsequent reward claim silently routed to `0x0`. The STRK ERC20 `transfer` to the zero address does not revert on Starknet; the tokens are sent and permanently unrecoverable. This constitutes **permanent freezing of unclaimed yield**, which is within the allowed impact scope.

The `unstake_action` path is also affected: `send_rewards_to_staker` is called inside `unstake_action` before the staker record is erased, so any accumulated rewards are also lost at that point. [7](#0-6) 

---

### Likelihood Explanation

Low-to-medium. The zero address is the natural default/uninitialized value in many languages and toolchains. A front-end bug, a scripting error, or a developer passing an uninitialized variable as `reward_address` will silently succeed. Because the protocol emits a `StakerRewardAddressChanged` / `PoolMemberRewardAddressChanged` event but performs no on-chain validation, the mistake is irreversible once rewards are claimed. The same risk applies at initial registration (`stake` / `enter_delegation_pool`), where a user may not realize the address they supplied was zero until rewards are already lost.

---

### Recommendation

Add a non-zero assertion in all four entry points that accept or update `reward_address`:

```cairo
assert!(reward_address.is_non_zero(), "{}", GenericError::ZERO_ADDRESS);
```

This should be added to:
- `Staking::stake` — before the token-address check
- `Staking::change_reward_address` — before the token-address check
- `Pool::enter_delegation_pool` — before the token-address check
- `Pool::change_reward_address` — before the token-address check

---

### Proof of Concept

**Staker path:**

1. Staker calls `Staking::stake(reward_address: 0, operational_address: X, amount: MIN_STAKE)`. No revert — zero address passes the `does_token_exist` guard.
2. Staker accumulates rewards over multiple epochs via `update_rewards` / attestation.
3. Anyone calls `Staking::claim_rewards(staker_address)`.
4. `send_rewards_to_staker` reads `reward_address = 0` and executes `checked_transfer(recipient: 0, amount: rewards)`.
5. STRK rewards are transferred to the zero address and permanently lost.

**Pool member path:**

1. Pool member calls `Pool::enter_delegation_pool(reward_address: 0, amount: X)`. No revert.
2. Pool member accumulates rewards.
3. Anyone calls `Pool::claim_rewards(pool_member)`.
4. `reward_token.checked_transfer(recipient: 0, amount: rewards)` executes.
5. STRK rewards are permanently lost.

**Post-registration path (change_reward_address):**

1. Staker or pool member calls `change_reward_address(reward_address: 0)`. No revert.
2. All future reward claims are routed to zero address.
