### Title
`protocolUnpausePool` Hardcodes Wrong Pause Level, Leaving Pool Permanently Admin-Paused After Protocol Unpause — (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary

`protocolUnpausePool` calls `setPause(1)` instead of `setPause(0)`, so a pool that was active before a protocol pause is left in admin-paused state after the protocol owner unpauses it. Core pool functionality (swap, addLiquidity, removeLiquidity) remains blocked until the pool admin separately calls `unpausePool`.

### Finding Description

The pool has three pause levels:

| Level | Meaning |
|---|---|
| 0 | Active |
| 1 | Paused by admin |
| 2 | Paused by protocol |

`protocolPausePool` accepts either level 0 or level 1 as the pre-condition and always transitions to level 2: [1](#0-0) 

`protocolUnpausePool` then transitions from level 2 to level **1** — not level 0: [2](#0-1) 

The pool's `whenNotPaused` modifier blocks all swaps whenever `pauseLevel != 0`: [3](#0-2) 

So after the sequence `protocolPausePool` → `protocolUnpausePool`, a pool that was fully active (level 0) ends up at level 1 — still paused — with no automatic path back to level 0. The admin must separately call `unpausePool`, which requires `cur == 1` and sets level 0: [4](#0-3) 

This is the direct analog of the Gearbox bug: just as `creditManager.executeOrder()` received `msg.sender` instead of the intended `_account`, `setPause` here receives `1` instead of the intended `0` — a wrong literal passed to the critical state-change call.

### Impact Explanation

Every swap, `addLiquidity`, and `removeLiquidity` call reverts with `PoolPaused` until the pool admin manually intervenes. If the pool admin is unavailable, unresponsive, or the admin key is lost, the pool is permanently bricked. LP funds are locked in the pool contract; traders cannot execute. This satisfies **Broken core pool functionality causing unusable withdraw/swap/liquidity flows**.

### Likelihood Explanation

The trigger is the normal, intended use of `protocolUnpausePool` by the factory owner — no adversarial input is required. Any emergency pause/unpause cycle on an active pool (level 0) reproduces the issue deterministically. The factory owner is semi-trusted and this function is part of the expected operational surface.

### Recommendation

Change `setPause(1)` to `setPause(0)` in `protocolUnpausePool`:

```solidity
function protocolUnpausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 2) revert InvalidPauseTransition(cur, 0); // target is 0
    IMetricOmmPoolFactoryActions(pool).setPause(0);       // restore to active
}
```

If the intent is to preserve a prior admin-pause, the factory must record the pre-protocol-pause level in storage and restore it on unpause.

### Proof of Concept

1. Pool is deployed; `pauseLevel == 0` (active).
2. Factory owner calls `protocolPausePool(pool)` → `pauseLevel` becomes `2`.
3. Factory owner calls `protocolUnpausePool(pool)` → `setPause(1)` is executed → `pauseLevel` becomes `1`.
4. Any user calls `pool.swap(...)` → `_checkNotPaused()` fires → `revert PoolPaused()`.
5. Pool admin must call `unpausePool(pool)` to reach level 0; if admin is unavailable, the pool is permanently paused. [5](#0-4) [6](#0-5)
