### Title
Permanent NEAR Freeze When wNEAR Unwrap Transfer Fails in `exit_to_near_precompile_callback` — (File: `engine/src/contract_methods/connector.rs`)

---

### Summary

When a user unwraps wNEAR ERC-20 tokens via the `ExitToNear` precompile using the `:unwrap` suffix, the engine calls `near_withdraw` on the wNEAR NEP-141 contract (which burns wNEAR and deposits NEAR into the Aurora engine account), then schedules a bare NEAR `Transfer` batch promise to the user-supplied target account. No failure callback is attached to that transfer promise. If the transfer fails — for example because the target named NEAR account does not exist — the NEAR is permanently stranded in the Aurora engine account with no on-chain recovery path for the user.

---

### Finding Description

The wNEAR unwrap path is set up in `exit_erc20_token_to_near`:

```rust
Some(Message::UnwrapWnear) if erc20_address == get_wnear_address(io).raw() => {
    (
        nep141_account_id,
        format!(r#"{{"amount":"{}"}}"#, exit_params.amount.as_u128()),
        "near_withdraw",
        Some(TransferNearArgs {
            target_account_id: exit_params.receiver_account_id.clone(), // user-supplied
            amount: exit_params.amount.as_u128(),
        }),
        ...
    )
}
``` [1](#0-0) 

This produces a two-step promise chain: `near_withdraw` (base) → `exit_to_near_precompile_callback` (callback). Inside the callback, when `near_withdraw` succeeds, a second `Transfer` batch promise is created and returned — but **no callback is attached to it**:

```rust
let maybe_result = if let Some(PromiseResult::Successful(_)) = handler.promise_result(0) {
    if let Some(args) = args.transfer_near {
        let action = PromiseAction::Transfer { amount: Yocto::new(args.amount) };
        let promise = PromiseBatchAction {
            target_account_id: args.target_account_id,
            actions: vec![action],
        };
        let promise_id = handler.promise_create_batch(&promise);
        handler.promise_return(promise_id);   // ← no failure callback
    }
    None
} else if let Some(args) = args.refund {
    // only reached when near_withdraw itself failed
    engine::refund_on_error(io, env, state, &args, handler)?;
    ...
}
``` [2](#0-1) 

The `refund` branch is only reachable when `near_withdraw` fails. There is no analogous branch for the case where `near_withdraw` succeeds but the subsequent NEAR transfer fails.

The developers are explicitly aware of the fund-locking risk in the ERC-20 exit path (they guard against ETH being attached to an ERC-20 exit call for exactly this reason):

> "In case of withdrawing ERC-20 tokens, the `apparent_value` should be zero. In opposite way the funds will be locked in the address of the precompile without any possibility to withdraw them in the future." [3](#0-2) 

Yet the symmetric risk in the wNEAR unwrap path — NEAR locked in the engine account after a failed transfer — is unguarded.

The Aurora engine exposes no public method to withdraw arbitrary NEAR from its account balance to a specific user. The `withdraw` function handles ETH (NEP-141 tokens), not native NEAR. [4](#0-3) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

The full sequence is irreversible:

1. `EvmErc20.withdrawToNear` burns the user's wNEAR ERC-20 balance in the EVM (irreversible).
2. `near_withdraw` burns the corresponding wNEAR NEP-141 balance held by the Aurora engine account and credits the engine account with native NEAR (irreversible).
3. The `Transfer` promise to the user-supplied target account fails.
4. The NEAR remains in the Aurora engine account with no on-chain path to return it to the user.

The user loses the full unwrapped NEAR amount permanently.

---

### Likelihood Explanation

**Medium.**

The entry point is `EvmErc20.withdrawToNear(bytes recipient, uint256 amount)` on the wNEAR ERC-20 contract, callable by any EVM user. [5](#0-4) 

The `recipient` bytes are passed directly through to the NEAR promise as the target account ID. In NEAR Protocol, a `Transfer` action to a non-existent **named** account (e.g. `"alice.near"`) fails — unlike Ethereum where any address can receive ETH. A user who makes a typo in the target account name, or who specifies an account that has not yet been created, will trigger this failure path. No validation of the target account's existence is performed before the wNEAR is burned.

---

### Recommendation

Attach a failure callback to the NEAR `Transfer` promise created inside `exit_to_near_precompile_callback`. If the transfer fails, the callback should re-credit the user's wNEAR ERC-20 balance (analogous to the existing `refund_on_error` path used when `near_withdraw` itself fails). Concretely:

- Extend `ExitToNearPrecompileCallbackArgs` with a `transfer_near_refund` field (similar to the existing `refund` field).
- In `exit_to_near_precompile_callback`, when `transfer_near` is set, use `PromiseArgs::Callback` instead of a bare batch promise, attaching a second callback that calls `refund_on_error` if the transfer fails. [6](#0-5) [7](#0-6) 

---

### Proof of Concept

**Attacker-controlled entry path:**

1. User holds wNEAR ERC-20 tokens on Aurora.
2. User calls `EvmErc20.withdrawToNear("nonexistent.near:unwrap", amount)` — `"nonexistent.near"` is a named NEAR account that does not exist.
3. The EVM burns `amount` wNEAR ERC-20 tokens from the user's balance.
4. The `ExitToNear` precompile emits a promise log: `near_withdraw(amount)` on the wNEAR NEP-141 contract, with `exit_to_near_precompile_callback` as the callback.
5. `near_withdraw` succeeds: wNEAR NEP-141 tokens are burned, `amount` yoctoNEAR is deposited into the Aurora engine account.
6. `exit_to_near_precompile_callback` is invoked; `promise_result(0)` is `Successful`; a bare `Transfer { amount }` batch promise targeting `"nonexistent.near"` is created and returned.
7. The `Transfer` action fails because `"nonexistent.near"` does not exist in the NEAR state.
8. No callback fires. The `amount` yoctoNEAR remains in the Aurora engine account.
9. The user's wNEAR ERC-20 and wNEAR NEP-141 balances are both zero. There is no on-chain function to recover the NEAR. [8](#0-7) [9](#0-8)

pub fn refund_on_error<I: IO + Copy, E: Env, P: PromiseHandler>(
    io: I,
    env: &E,
    state: EngineState,
    args: &RefundCallArgs,
    handler: &mut P,
) -> EngineResult<SubmitResult> {
    let current_account_id = env.current_account_id();
    if let Some(erc20_address) = args.erc20_address {
        // ERC-20 exit; re-mint burned tokens
        let erc20_admin_address = current_address(&current_account_id);
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, erc20_admin_address, current_account_id, io, env);

        let refund_address = args.recipient_address;
        let amount = U256::from_big_endian(&args.amount);
        let input = setup_refund_on_error_input(amount, refund_address);

        engine.call(
            &erc20_admin_address,
            &erc20_address,
            Wei::zero(),
            input,
            u64::MAX,
            Vec::new(),
            Vec::new(),
            handler,
        )
    } else {
        // ETH exit; transfer ETH back from precompile address
        let exit_address = exit_to_near::ADDRESS;
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, exit_address, current_account_id, io, env);
        let refund_address = args.recipient_address;
        let amount = Wei::new(U256::from_big_endian(&args.amount));
        engine.call(
            &exit_address,
            &refund_address,
            amount,
            Vec::new(),
            u64::MAX,
            vec![
                (exit_address.raw(), Vec::new()),
                (refund_address.raw(), Vec::new()),
            ],
            Vec::new(),
            handler,
        )
    }
