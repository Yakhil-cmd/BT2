Audit Report

## Title
`ensure_balance()` Uses Saturating-Subtracted `cycles_to_mint` for Rate-Limiter and Accounting Instead of Full Distribution Amount - (File: rs/nns/cmc/src/main.rs)

## Summary
`ensure_balance()` computes `cycles_to_mint = cycles - current_balance` using saturating subtraction, then passes this derived value — not the full `cycles` amount being distributed — to both the rate limiter and `total_cycles_minted`. Any pre-existing CMC balance (including cycles deposited by an unprivileged attacker canister via the management canister's `deposit_cycles` endpoint) causes `cycles_to_mint` to saturate to zero, silently bypassing the hourly minting cap and leaving `total_cycles_minted` unincremented while the full `cycles` amount is still paid out to the user.

## Finding Description
In `rs/nns/cmc/src/main.rs` lines 2306–2325, `ensure_balance()` reads the live CMC balance and computes:

```rust
let current_balance = Cycles::from(ic_cdk::api::canister_balance128());
let cycles_to_mint = cycles - current_balance;  // saturating sub
```

`Cycles::sub` is explicitly saturating (`rs/types/cycles/src/cycles.rs` lines 133–139), so when `current_balance >= cycles`, `cycles_to_mint == Cycles(0)`.

Both the rate limiter and accounting counter are then gated on this zero value:

```rust
limiter_to_use.check_and_add_cycles(state, now, cycles_to_mint)?;  // sees 0
state.total_cycles_minted += cycles_to_mint;                        // += 0
```

In `rs/nns/cmc/src/limiter.rs` lines 34–56, `check_and_add_cycles` enforces `count + cycles_to_mint > limit`. With `cycles_to_mint == 0`, this check always passes and records nothing. The `CyclesMintingLimiterSelector` wrapper at lines 402–419 passes `cycles_to_mint` directly to the underlying limiter without modification.

After `ensure_balance` returns, `deposit_cycles` (lines 2110–2138) calls `call_with_payment128(..., u128::from(cycles))` — the **full** original `cycles` amount — paying out from the pre-loaded CMC balance. The user receives the full cycles; the rate limiter and counter are untouched.

The exploit path is permissionless: any canister can call `ic00::deposit_cycles(CMC_CANISTER_ID, amount)` to inflate the CMC's balance before a `notify_top_up` or `notify_create_canister` call triggers `process_top_up` → `deposit_cycles(..., mint_cycles=true, BaseLimit)` → `ensure_balance`.

## Impact Explanation
This is a **High** severity finding. The hourly minting cap (`base_cycles_limit`, default 150 × 10¹⁵ cycles) is a governance-controlled NNS parameter intended to bound the rate of ICP-to-cycles conversion and protect token economics. Bypassing it allows sustained conversion of ICP to cycles at rates exceeding what NNS governance has authorized, constituting a significant NNS/CMC infrastructure security impact with concrete protocol harm. Additionally, `state.total_cycles_minted` — exposed via the `total_cycles_minted` query and the `cmc_cycles_minted_total` Prometheus metric used for governance and economic analysis (lines 2327–2330, 2469–2473) — is permanently undercounted, corrupting the canonical on-chain record of cycles ever distributed.

## Likelihood Explanation
The attack requires no privileged role. Any canister on any subnet can call `deposit_cycles` on the management canister targeting the CMC's canister ID. The primary constraint is cost: pre-loading 150 × 10¹⁵ cycles costs approximately $195,000 USD equivalent, and those cycles are permanently consumed to fulfill user requests. This makes casual exploitation unlikely but does not prevent a motivated actor (e.g., a large ICP holder seeking to convert beyond the hourly cap, or an entity seeking to corrupt governance metrics). The `total_cycles_minted` undercount is reachable even from small residual CMC balances arising from normal operation, without any deliberate attacker.

## Recommendation
Decouple the rate-limiter check and `total_cycles_minted` accounting from `cycles_to_mint`. The limiter should be checked against the full `cycles` amount being distributed, since that is the economically meaningful quantity regardless of whether it comes from fresh minting or pre-existing balance:

```rust
fn ensure_balance(
    cycles: Cycles,
    limiter_to_use: CyclesMintingLimiterSelector,
) -> Result<(), String> {
    let now = now_system_time();
    let current_balance = Cycles::from(ic_cdk::api::canister_balance128());
    let cycles_to_mint = cycles - current_balance; // saturating; may be 0

    with_state_mut(|state| {
        // Rate-limit on the full distribution amount, not just freshly minted portion
        limiter_to_use.check_and_add_cycles(state, now, cycles)?;
        // Only count truly new cycles in the accounting metric
        state.total_cycles_minted += cycles_to_mint;
        Ok::<_, String>(())
    })?;

    let _minted_cycles = ic0_mint_cycles128(cycles_to_mint);
    assert!(ic_cdk::api::canister_balance128() >= cycles.get());
    Ok(())
}
```

If the intent is that `total_cycles_minted` tracks only ICP-sourced minting (not pre-existing balance), the current `+= cycles_to_mint` for accounting may be acceptable, but the rate limiter **must** be checked against `cycles` to prevent bypass.

## Proof of Concept
1. Deploy an attacker canister. Call `ic00::deposit_cycles(CMC_CANISTER_ID, 150_000_000_000_000_000u128)` from it. This is a standard management canister call requiring no special privileges.
2. Observe `ic_cdk::api::canister_balance128()` on the CMC is now ≥ 150 × 10¹⁵.
3. Send 1,500 ICP to the CMC's subaccount for a target canister and call `notify_top_up`. At the current exchange rate this converts to ~150 × 10¹⁵ cycles, which would normally hit the hourly cap.
4. Inside `ensure_balance`: `cycles_to_mint = cycles - current_balance = Cycles(0)`. The limiter's `check_and_add_cycles(now, Cycles(0), limit)` passes and records 0. `total_cycles_minted += 0`. `ic0_mint_cycles128(0)` mints nothing. `call_with_payment128(..., u128::from(cycles))` sends 150 × 10¹⁵ cycles to the target canister from the pre-loaded balance.
5. Verify: `total_cycles_minted` query returns the same value as before; `base_limiter.get_count()` returns 0; the target canister received 150 × 10¹⁵ cycles.
6. A deterministic PocketIC integration test can reproduce this by: (a) pre-crediting the CMC canister with cycles via the test harness, (b) triggering `notify_top_up`, and (c) asserting that `total_cycles_minted` and the limiter count remain at zero while the target canister balance increased by the full amount.
