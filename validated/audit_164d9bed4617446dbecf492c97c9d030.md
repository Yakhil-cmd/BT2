### Title
Declare V2 and V3 Transactions Unsupported in `estimateFee` and `simulateTransactions` RPC Methods — (File: `crates/apollo_rpc/src/v0_8/api/mod.rs`)

### Summary
The `starknet_estimateFee` and `starknet_simulateTransactions` RPC endpoints unconditionally return an internal server error for any `BroadcastedDeclareTransaction::V2` or `V3` input. Because Declare V2/V3 are the only transaction types used to deploy Cairo 1 contracts on Starknet, this makes fee estimation and simulation completely unavailable for the dominant contract-deployment path, returning an authoritative-looking error to every caller.

### Finding Description
In `crates/apollo_rpc/src/v0_8/api/mod.rs`, the `TryFrom<BroadcastedDeclareTransaction> for ExecutableTransactionInput` conversion — which is the sole conversion path used by both `estimate_fee` and `simulate_transactions` — hard-codes failures for V2 and V3:

```rust
BroadcastedDeclareTransaction::V2(_) => {
    // TODO(yair): We need a way to get the casm of a declare V2 transaction.
    Err(internal_server_error("Declare V2 is not supported yet in execution."))
}
BroadcastedDeclareTransaction::V3(_) => {
    // TODO(yair): We need a way to get the casm of a declare V3 transaction.
    Err(internal_server_error("Declare V3 is not supported yet in execution."))
}
``` [1](#0-0) 

Both `estimate_fee` and `simulate_transactions` call this conversion unconditionally:

```rust
let executable_txns =
    transactions.into_iter().map(|tx| tx.try_into()).collect::<Result<_, _>>()?;
``` [2](#0-1) [3](#0-2) 

The gateway path (`convert_rpc_tx_to_internal`) correctly handles Declare V3 by calling `class_manager_client.add_class(tx.contract_class)` to obtain the CASM and class hash. The RPC simulation path has no equivalent mechanism and was never wired up. [4](#0-3) 

### Impact Explanation
Any unprivileged caller submitting a `BroadcastedDeclareTransaction::V2` or `V3` to `starknet_estimateFee` or `starknet_simulateTransactions` receives an `internal_server_error`. This is the **only** way to deploy Cairo 1 contracts on Starknet. The result is:

- Fee estimation is entirely unavailable for Cairo 1 contract declarations — wallets, SDKs, and tooling that call `estimateFee` before broadcasting a declare transaction will always fail.
- Simulation is entirely unavailable for Declare V2/V3 — developers cannot inspect execution traces or validate declare transactions before submission.
- The error is returned as an HTTP 500 with an opaque message, which looks authoritative and gives no indication that the transaction type itself is the cause.

This matches the **High** impact scope: *"RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value."*

### Likelihood Explanation
Any developer or wallet attempting to estimate fees for a Cairo 1 contract declaration will trigger this unconditionally. No special privileges, timing, or state are required. The trigger is a single well-formed JSON-RPC call.

### Recommendation
Wire the RPC simulation path to the class manager the same way the gateway does:

1. In the `TryFrom<BroadcastedDeclareTransaction> for ExecutableTransactionInput` implementation (or in a new async helper), call `class_manager_client.add_class(contract_class)` to compile the Sierra class and obtain the CASM and class hash.
2. Construct `ExecutableTransactionInput::DeclareV2` / `DeclareV3` with the resulting `compiled_class`, `sierra_program_length`, `abi_length`, and `sierra_version`, mirroring the existing `DeclareV1` path.
3. Remove the two `Err(internal_server_error(...))` arms and the associated TODO comments.
