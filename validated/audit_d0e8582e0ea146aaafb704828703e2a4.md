### Title
Missing `relativeLockHeight` Range Validation Allows Attacker-Controlled Pool to Inject Unsafe Lock Heights — (`packages/gui/src/components/plotNFT/select/PlotNFTSelectPool.tsx`)

### Summary

The `prepareSubmitData` function in `PlotNFTSelectPool.tsx` fetches `relativeLockHeight` from an attacker-controlled pool server and passes it verbatim to `pwJoinPool` with no range validation. The only guard is a presence check (`=== undefined`), which passes for any numeric value including `0` or `4294967295`.

### Finding Description

In `prepareSubmitData`, `getPoolInfo` is called with the user-supplied pool URL: [1](#0-0) 

The only validation performed is:

```ts
if (relativeLockHeight === undefined) {
  throw new Error(t`Pool does not provide relativeLockHeight.`);
}
```

Values like `0`, `1`, `31`, or `4294967295` pass this check and are stored directly into `initialTargetState`: [2](#0-1) 

`getPoolInfo` itself performs no validation — it fetches from the pool URL via `window.appAPI.fetchPoolInfo` and returns a camelCase-converted raw response: [3](#0-2) 

`PlotNFTChangePool.handleSubmit` then passes `relativeLockHeight` directly to `pwJoinPool` with no further checks: [4](#0-3) 

### Impact Explanation

An attacker who operates a pool server can advertise `relative_lock_height: 0` (enabling instant pool exit and replay) or `relative_lock_height: 4294967295` (effectively permanent lock-in, ~136 years at 18.75 s/block). The GUI will sign and submit the `pw_join_pool` transaction with that value without warning the user.

**Critical caveat:** The actual on-chain enforceability depends on whether the Chia wallet daemon (`chia-blockchain` Python code, out of scope here) independently validates `relativeLockHeight`. If the daemon enforces the protocol minimum of 32 blocks, the RPC call would be rejected and the impact degrades to a failed action. This is not verifiable from the GUI code alone.

### Likelihood Explanation

The attacker must operate a pool server that the user voluntarily connects to. This is a realistic scenario (malicious pool operator, DNS hijack of a legitimate pool URL). The GUI provides no warning or range check to protect the user.

### Recommendation

In `prepareSubmitData`, add a range guard before using `relativeLockHeight`:

```ts
const MIN_LOCK_HEIGHT = 32;
const MAX_LOCK_HEIGHT = 4294967295; // u32 max; consider a tighter bound
if (relativeLockHeight < MIN_LOCK_HEIGHT || relativeLockHeight > MAX_LOCK_HEIGHT || !Number.isInteger(relativeLockHeight)) {
  throw new Error(t`Pool relativeLockHeight ${relativeLockHeight} is outside the safe range.`);
}
```

### Proof of Concept

1. Stand up a pool server at `https://evil.pool/pool` returning:
   ```json
   { "relative_lock_height": 0, "target_puzzle_hash": "<valid_hash>", ... }
   ```
2. In the Chia GUI, navigate to Pool → Change Pool, enter `https://evil.pool/pool`, and submit.
3. Observe that `prepareSubmitData` accepts `relativeLockHeight = 0` (only `undefined` is rejected) and calls `pwJoinPool` with `relativeLockHeight: 0`.
4. Repeat with `4294967295` to test the lock-in scenario.
5. Assert that neither value triggers a GUI-level error or warning. [5](#0-4)

```typescript
async function prepareSubmitData(data: FormData): SubmitData {
  const { self, fee, poolUrl } = data;
  const initialTargetState = {
    state: self ? 'SELF_POOLING' : 'FARMING_TO_POOL',
  };

  if (!self && poolUrl) {
    const normalizedPoolUrl = normalizeUrl(poolUrl);
    const { targetPuzzleHash, relativeLockHeight } = await getPoolInfo(normalizedPoolUrl);
    if (!targetPuzzleHash) {
      throw new Error(t`Pool does not provide targetPuzzleHash.`);
    }
    if (relativeLockHeight === undefined) {
      throw new Error(t`Pool does not provide relativeLockHeight.`);
    }

    initialTargetState.poolUrl = normalizedPoolUrl;
    initialTargetState.targetPuzzleHash = targetPuzzleHash;
    initialTargetState.relativeLockHeight = relativeLockHeight;
  }

  const feeMojos = chiaToMojo(fee || '0');

  return {
    fee: feeMojos,
    initialTargetState,
  };
}
```
