### Title
`EVM.mustRun()` Does Not Revert on `Status.failed`, Enabling Cross-VM Bridge Escrow Mis-Accounting — (File: `fvm/evm/stdlib/contract.cdc`)

---

### Summary

`EVM.mustRun()` is a public Cadence function in the Flow EVM contract that is explicitly designed to **not** revert the enclosing Cadence transaction when the inner EVM transaction returns `Status.failed`. Any Cadence contract author who uses `mustRun` in a cross-VM bridge pattern (burn/lock on EVM side, then release/mint on Cadence side) without explicitly checking `res.status == Status.successful` will produce a vulnerable bridge where a failed EVM-side operation silently allows the Cadence-side asset release to proceed — yielding assets on both sides simultaneously.

---

### Finding Description

`EVM.mustRun()` is defined in `fvm/evm/stdlib/contract.cdc`:

```cadence
/// mustRun runs the transaction using EVM.run
/// It will rollback if the tx execution status is unknown or invalid.
/// Note that this method does not rollback if transaction
/// is executed but an vm error is reported as the outcome
/// of the execution (status: failed).
access(all)
fun mustRun(tx: [UInt8], coinbase: EVMAddress): Result {
    let runResult = self.run(tx: tx, coinbase: coinbase)
    assert(
        runResult.status == Status.failed || runResult.status == Status.successful,
        message: "EVM.mustRun(): The provided transaction is not valid for execution"
    )
    return runResult
}
``` [1](#0-0) 

The `assert` only rejects `Status.unknown` and `Status.invalid`. It explicitly passes through `Status.failed` — meaning the enclosing Cadence transaction continues executing normally even when the EVM-side operation reverted.

The three EVM result statuses are:

- `Status.invalid` — transaction rejected before execution (nonce mismatch, gas cap exceeded, etc.)
- `Status.failed` — transaction was executed but the EVM VM reported an error (out of gas, revert, etc.); EVM state changes are rolled back but the Cadence transaction is **not** rolled back
- `Status.successful` — transaction executed without error [2](#0-1) [3](#0-2) 

The underlying `EVM.run()` function also returns the result without reverting on failure: [4](#0-3) 

The emulator confirms that a `failed` EVM transaction commits a state change (nonce increment, gas fee deduction) to the EVM state while reverting the user's intended operation: [5](#0-4) 

By contrast, the internal Go-level handler `RunOrPanic` used by the protocol itself calls `panicOnErrorOrInvalidOrFailedState`, which **does** panic on `failed` status — protecting the protocol's own internal bridge calls: [6](#0-5) [7](#0-6) 

The public Cadence API (`mustRun`) does not provide this protection.

---

### Impact Explanation

A Cadence contract author building a cross-VM bridge using `EVM.mustRun()` in the following pattern:

```cadence
// Step 1: burn tokens on EVM side
let res = EVM.mustRun(tx: burnTx, coinbase: coinbase)
// Step 2: release tokens on Cadence side (proceeds even if res.status == failed)
vault.deposit(from: <- escrow.withdraw(amount: amount))
```

will produce a bridge where:

1. The EVM burn transaction fails (e.g., out of gas, contract revert, insufficient EVM balance)
2. `mustRun` returns `Status.failed` without reverting the Cadence transaction
3. The Cadence-side asset release executes unconditionally
4. The user retains tokens on the EVM side (burn did not happen) **and** receives tokens on the Cadence side

This is a direct analog to the Redis relayer bug: the state-tracking step (EVM burn) fails silently, the subsequent release step proceeds unaware, and the user receives assets on both sides. The impact is **cross-VM asset double-spending / bridge escrow mis-accounting**.

---

### Likelihood Explanation

**Medium.** The function name `mustRun` strongly implies "this must succeed or revert," which is the standard expectation for a function with that naming convention. A contract author building a bridge who uses `mustRun` without reading the documentation comment will naturally assume it reverts on failure. The misleading name makes this mistake realistic. The `EVM.run()` / `EVM.mustRun()` API is the primary public interface for cross-VM operations available to any unprivileged Cadence transaction sender or contract author.

---

### Recommendation

1. **Rename `mustRun` to `runOrSkip` or `runAllowingFailure`** to accurately reflect that it does not revert on `Status.failed`.
2. **Add a new `mustRunSuccessful` function** that asserts `runResult.status == Status.successful` and reverts otherwise — providing a safe default for bridge authors.
3. **Add a post-condition** or explicit documentation warning at the call site that callers must check `res.status` before proceeding with Cadence-side asset operations.

---

### Proof of Concept

A vulnerable Cadence bridge transaction:

```cadence
import EVM from <EVMAddress>
import FungibleToken from <FTAddress>

transaction(burnTxBytes: [UInt8], coinbaseBytes: [UInt8; 20], amount: UFix64) {
    prepare(account: auth(Storage) &Account) {
        let coinbase = EVM.EVMAddress(bytes: coinbaseBytes)

        // Attacker crafts burnTxBytes to fail (e.g., insufficient gas, revert)
        // mustRun does NOT revert the Cadence tx on Status.failed
        let res = EVM.mustRun(tx: burnTxBytes, coinbase: coinbase)
        // res.status == EVM.Status.failed — EVM tokens NOT burned

        // Bridge contract proceeds without checking res.status
        // Cadence-side escrow releases tokens unconditionally
        let escrow = account.storage.borrow<&EscrowVault>(from: /storage/escrow)!
        let vault <- escrow.withdraw(amount: amount)
        account.storage.borrow<&{FungibleToken.Receiver}>(from: /storage/receiver)!
            .deposit(from: <-vault)
        // Result: attacker has tokens on EVM (not burned) AND on Cadence (released)
    }
}
```

The root cause is confirmed at: [8](#0-7) 

The `assert` condition `runResult.status == Status.failed || runResult.status == Status.successful` passes for `failed`, so execution continues into the Cadence-side release logic with no revert.

// RunTransaction runs an evm transaction
func (bl *BlockView) RunTransaction(
	tx *gethTypes.Transaction,
) (result *types.Result, err error) {
	// create a new procedure
	proc, err := bl.newProcedure()
	if err != nil {
		return nil, err
	}

	// constructs a core.message from the tx
	msg, err := gethCore.TransactionToMessage(
		tx,
		GetSigner(bl.config),
		proc.config.BlockContext.BaseFee)
	if err != nil {
		// this is not a fatal error (e.g. due to bad signature)
		// not a valid transaction
		return types.NewInvalidResult(tx.Type(), tx.Hash(), err), nil
	}

	// call tracer
	if proc.evm.Config.Tracer != nil && proc.evm.Config.Tracer.OnTxStart != nil {
		proc.evm.Config.Tracer.OnTxStart(proc.evm.GetVMContext(), tx, msg.From)
	}

	// run msg
	res, err := proc.run(msg, tx.Hash(), tx.Type())
	if err != nil {
		return nil, err
	}

	// all commit errors (StateDB errors) has to be returned
	res.StateChangeCommitment, err = proc.commit(true)
	if err != nil {
		return nil, err
	}

	// call tracer on tx end
	if proc.evm.Config.Tracer != nil &&
		proc.evm.Config.Tracer.OnTxEnd != nil &&
		res != nil {
		proc.evm.Config.Tracer.OnTxEnd(res.Receipt(), res.ValidationError)
	}

	// call OnLog tracer hook, upon successful tx result
	if proc.evm.Config.Tracer != nil &&
		proc.evm.Config.Tracer.OnLog != nil &&
		res != nil {
		for _, log := range res.Logs {
			proc.evm.Config.Tracer.OnLog(log)
		}
	}

	return res, nil

func panicOnErrorOrInvalidOrFailedState(res *types.Result, err error) {

	if res != nil && res.Invalid() {
		panic(fvmErrors.NewEVMError(res.ValidationError))
	}

	if res != nil && res.Failed() {
		panic(fvmErrors.NewEVMError(res.VMError))
	}

	// this should never happen
	if err == nil && res == nil {
		panic(fvmErrors.NewEVMError(types.ErrUnexpectedEmptyResult))
	}

	panicOnError(err)
}
