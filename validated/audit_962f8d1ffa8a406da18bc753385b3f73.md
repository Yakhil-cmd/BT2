### Title
Gateway `skip_stateful_validations` Allows Fee-Escalating Invoke to Bypass `__validate__` Signature Check for Undeployed Accounts - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` UX feature, which skips the `__validate__` entry-point execution for an invoke transaction with nonce=1 when a deploy-account is pending in the mempool, can be abused by an attacker to admit an invoke transaction with an **arbitrary/invalid signature** into the mempool via fee escalation. The "permission" granted by the presence of the victim's deploy-account transaction in the pool is consumed twice: once for the legitimate invoke, and once for the attacker's fee-escalating replacement.

### Finding Description

`skip_stateful_validations` returns `true` (skip `__validate__`) when all three conditions hold:

1. The incoming transaction is an `Invoke` with `nonce == 1`
2. The on-chain account nonce is `0` (account not yet deployed)
3. `account_tx_in_pool_or_recent_block(sender_address)` returns `true` [1](#0-0) 

When `skip_validate = true`, `run_validate_entry_point` is called with `execution_flags.validate = false`, meaning the account's `__validate__` Cairo entry point is **never executed** during gateway admission: [2](#0-1) 

The attack proceeds as follows:

1. **Alice** pre-funds her counterfactual address and submits `deploy_account` (nonce=0) + `invoke` (nonce=1) together — the standard UX flow.
2. Both transactions enter the mempool. `account_tx_in_pool_or_recent_block` returns `true` because Alice's `deploy_account` is in `tx_pool`.
3. **Attacker** submits a malicious `invoke` with `nonce=1`, a **higher tip/gas price** (fee escalation), and an **arbitrary signature**.
4. `validate_by_mempool` is called first and accepts the attacker's tx as a valid fee escalation replacement for Alice's invoke: [3](#0-2) 

5. `skip_stateful_validations` is then evaluated: nonce=1, account_nonce=0, `account_tx_in_pool_or_recent_block=true` (Alice's `deploy_account` is still in the pool) → returns `true`.
6. `run_validate_entry_point` runs with `validate=false`. `perform_pre_validation_stage` executes the nonce check and fee-balance check (passes because the address is pre-funded), but **skips `__validate__`**: [4](#0-3) 

7. The attacker's tx — carrying an invalid signature — is admitted to the mempool. Alice's legitimate invoke is evicted.

The comment in the code acknowledges the reasoning but does not account for the fee-escalation replacement path: [5](#0-4) 

The "permission" (the deploy-account tx in the pool) is used once to skip validation for Alice's invoke, and then the **same permission** is exploited again to skip validation for the attacker's fee-escalating replacement — a direct analog to the double-voting pattern.

### Impact Explanation

**High — Mempool/gateway admission accepts an invalid transaction before sequencing.**

The gateway admits a transaction with an invalid/arbitrary signature into the mempool. When the batcher later executes the block, the `deploy_account` is processed first (deploying Alice's account), then the attacker's invoke is executed with `__validate__` called on the now-deployed contract. The signature check fails, the tx is reverted, and the fee is charged from Alice's pre-funded balance. Alice's legitimate invoke is permanently evicted from the mempool and must be resubmitted with a new nonce.

Concrete consequences:
- Alice's invoke tx is displaced (griefing / denial of service)
- Alice's pre-funded balance is partially drained by the fee for the attacker's reverted tx
- Alice must resubmit with nonce=2, paying additional fees

### Likelihood Explanation

Any unprivileged actor who observes the mempool (or the gateway submission) can identify accounts in the deploy-account + invoke UX flow. The attacker only needs to know the target address and submit a fee-escalating invoke with nonce=1. No private key material is required. The pre-funding requirement is a necessary precondition of the victim's own setup, not an attacker constraint.

### Recommendation

In `skip_stateful_validations`, restrict the skip to the **first** invoke submitted for an undeployed account, not to any fee-escalating replacement. One approach: before returning `true`, verify that no invoke with nonce=1 already exists in the mempool for this address. Alternatively, require that the incoming tx hash matches the one already in the pool (i.e., only allow re-validation skips for the exact same transaction, not replacements). A stricter fix is to require that the deploy-account tx hash is explicitly provided and verified to be in the pool before skipping validation.

### Proof of Concept

```
1. Alice pre-funds address A with STRK tokens.
2. Alice submits:
   - deploy_account tx (nonce=0, valid signature)
   - invoke tx (nonce=1, tip=10, valid signature, calldata=X)
3. Both are admitted to the mempool.
4. Attacker submits:
   - invoke tx (sender=A, nonce=1, tip=20, signature=[0xDEAD, 0xBEEF], calldata=Y)
5. Gateway stateful validator:
   - validate_by_mempool: accepts as fee escalation (tip 20 > 10), evicts Alice's invoke
   - skip_stateful_validations: nonce==1, account_nonce==0, account_tx_in_pool==true → returns true
   - run_validate_entry_point: validate=false → __validate__ NOT called
   - perform_pre_validation_stage: balance check passes (A is pre-funded)
6. Attacker's tx is in the mempool. Alice's invoke is gone.
7. Batcher executes block:
   - deploy_account succeeds (Alice's account deployed at A)
   - attacker's invoke: __validate__ called → signature [0xDEAD, 0xBEEF] fails → tx reverted, fee charged from A
8. Alice's invoke is lost; her balance is reduced by the attacker's tx fee.
```

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
