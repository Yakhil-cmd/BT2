### Title
P2P Transaction Sync Silently Drops All Events, Causing RPC to Return Empty Events for Synced Blocks - (File: crates/apollo_protobuf/src/converters/receipt.rs)

### Summary

The P2P transaction sync protocol never serializes transaction events into the protobuf wire format. Every `TryFrom<protobuf::receipt::*>` deserialization implementation hardcodes `events = vec![]`. As a result, any node that syncs blocks via P2P stores empty event lists for every transaction, and subsequent `starknet_getEvents` and `starknet_getTransactionReceipt` RPC calls return authoritative-looking but completely wrong (empty) event data.

### Finding Description

The vulnerability spans four components in a single unbroken chain:

**Step 1 — Serialization omits events.**
`From<TransactionOutput> for protobuf::Receipt` dispatches to per-type `From` implementations (e.g. `From<InvokeTransactionOutput> for protobuf::receipt::Invoke`). Each one calls `create_proto_receipt_common_from_txn_output_fields`, which only encodes `actual_fee`, `messages_sent`, `execution_resources`, and `revert_reason`. The `events` field of every `*TransactionOutput` struct is silently discarded and never placed in the protobuf message. [1](#0-0) [2](#0-1) 

**Step 2 — Deserialization hardcodes empty events.**
Every `TryFrom<protobuf::receipt::*>` implementation (Invoke, Declare, Deploy, DeployAccount, L1Handler) explicitly sets `let events = vec![]` before constructing the output struct. The code comments acknowledge this: `// The output will have an empty events vec`. [3](#0-2) [4](#0-3) [5](#0-4) 

**Step 3 — P2P client writes event-stripped outputs to storage.**
`TransactionStreamFactory::parse_data_for_block` receives `FullTransaction` objects (deserialized via `TryFrom<protobuf::TransactionWithReceipt> for FullTransaction`, which calls `TransactionOutput::try_from(receipt)`) and pushes the event-stripped `transaction_output` into `block_body.transaction_outputs`. The developer TODO comment explicitly flags the missing events protocol: `// TODO(Eitan): Add events protocol to BlockBody or split their write to storage`. [6](#0-5) [7](#0-6) 

The assembled `BlockBody` is then committed to storage: [8](#0-7) 

**Step 4 — Storage writes zero events.**
`append_body` → `write_transactions` → `write_events` iterates `tx_output.events()`. Because the events vec is empty, nothing is inserted into the `events_table`. The `transaction_output` file (used by `starknet_getTransactionReceipt`) is also written with empty events. [9](#0-8) [10](#0-9) 

**Step 5 — RPC returns wrong data.**
`starknet_getEvents` reads from the events table (empty for all P2P-synced blocks) and returns no events. `starknet_getTransactionReceipt` reads the stored `transaction_output` (which has `events = vec![]`) and returns empty events in the receipt. [11](#0-10) 

### Impact Explanation

Any non-sequencer node (or any node that syncs historical blocks via P2P rather than re-executing them) will store and serve empty events for every transaction in every synced block. `starknet_getEvents` returns `[]` for all queries over synced blocks. `starknet_getTransactionReceipt` returns receipts with `events: []` regardless of what the transaction actually emitted. This is a persistent, silent data corruption: the node presents authoritative-looking wrong values with no error or warning.

This matches the allowed impact: **High — RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value.**

### Likelihood Explanation

The condition is triggered by normal P2P sync operation, which is the standard path for every non-sequencer full node. No special transaction crafting or privileged access is required. The bug is structural: the protobuf schema for `Receipt` has no events field, so every synced block is affected unconditionally.

### Recommendation

Add an events field to the protobuf `Receipt` (or a separate `Events` message in the sync protocol). Update `create_proto_receipt_common_from_txn_output_fields` to serialize events, and update all `TryFrom<protobuf::receipt::*>` implementations to deserialize them instead of hardcoding `vec![]`. The existing TODO comment at `TransactionStreamFactory` (`// TODO(Eitan): Add events protocol to BlockBody or split their write to storage`) already identifies the correct fix location.

### Proof of Concept

1. Start a P2P sync client node that syncs from a sequencer node.
2. Wait for the client to sync one or more blocks containing transactions with events (e.g. any ERC-20 transfer).
3. Call `starknet_getEvents` with `from_block`/`to_block` covering the synced range.
4. Observe: the response is `{"events": []}` despite the transactions having emitted events.
5. Call `starknet_getTransactionReceipt` for any synced transaction hash.
6. Observe: the receipt contains `"events": []` regardless of actual execution output.

The root cause is confirmed at:
- Serialization: `From<InvokeTransactionOutput> for protobuf::receipt::Invoke` — events field absent [12](#0-11) 
- Deserialization: `TryFrom<protobuf::receipt::Invoke> for InvokeTransactionOutput` — `let events = vec![]` [4](#0-3) 
- Storage: `write_events` iterates an empty vec, writes nothing [10](#0-9)

fn create_proto_receipt_common_from_txn_output_fields(
    actual_fee: Fee,
    messages_sent: Vec<MessageToL1>,
    execution_resources: ExecutionResources,
    execution_status: TransactionExecutionStatus,
) -> protobuf::receipt::Common {
    let actual_fee = Felt::from(actual_fee).into();
    let messages_sent = messages_sent.into_iter().map(protobuf::MessageToL1::from).collect();
    let execution_resources = execution_resources.into();
    let revert_reason =
        if let TransactionExecutionStatus::Reverted(reverted_status) = execution_status {
            Some(reverted_status.revert_reason)
        } else {
            None
        };
    protobuf::receipt::Common {
        actual_fee: Some(actual_fee),
        price_unit: 0,
        messages_sent,
        execution_resources: Some(execution_resources),
        revert_reason,
    }
}

fn write_events<'env>(
    tx_output: &TransactionOutput,
    txn: &DbTransaction<'env, RW>,
    events_table: &'env EventsTable<'env>,
    transaction_index: TransactionIndex,
) -> StorageResult<()> {
    let mut contract_addresses_set = HashSet::new();

    for event in tx_output.events().iter() {
        contract_addresses_set.insert(event.from_address);
    }

    for contract_address in contract_addresses_set {
        let key = (contract_address, transaction_index);
        // Here, we use the function assumption; the append will fail if an older transaction_index
        // is a table.
        events_table.append_greater_sub_key(txn, &key, &NoValue)?;
    }
    Ok(())
}
