### Title
Native Resource Cannot Be Limited Per Subcall — `FatalRuntimeError` Bypasses EVM Error Handling - (`basic_bootloader/src/bootloader/runner.rs`, `evm_interpreter/src/lib.rs`)

---

### Summary

ZKsync OS passes native resource (the proving-cost resource) **fully and unconditionally** from caller to callee with no per-subcall cap. When a callee exhausts native resource — for example, during `BytecodePreprocessingData::create_artifacts` for a `DEFAULT_CODE_VERSION_BYTE` contract — a `FatalRuntimeError` propagates up through the Rust call stack and reverts the **entire transaction**. This bypasses any EVM-level error handling (`try-catch` in Solidity) in the caller, which is the direct analog of the gas-bomb pattern in the reference report.

---

### Finding Description

**Root cause 1 — Native resource given fully to callee with no cap:**

In `basic_bootloader/src/bootloader/runner.rs`, when preparing a subcall frame, all native resource is transferred from the caller to the callee:

```rust
// Give native resource to the callee.
resources_in_caller_frame.give_native_to(&mut callee_resources);
``` [1](#0-0) 

This is explicitly documented as a design property:

> "The native resources are passed fully from frame to frame, a call cannot set a limit on how much of it the callee can spend." [2](#0-1) 

Unlike EVM gas (where the 63/64 rule lets callers retain a reserve), there is no mechanism for a caller to cap native resource passed to a callee.

---

**Root cause 2 — Bytecode preprocessing charges native on every call for `DEFAULT_CODE_VERSION_BYTE` contracts:**

`BytecodePreprocessingData::create_artifacts` charges `BYTECODE_PREPROCESSING_BYTE_NATIVE_COST (= 6) × bytecode_len` native units. For a max-size contract (24,576 bytes), this is **147,456 native units per call**. If native runs out, it returns `FatalRuntimeError`:

```rust
let native_cost = R::Native::from_computational(
    BYTECODE_PREPROCESSING_BYTE_NATIVE_COST.saturating_mul(deployed_code.len() as u64),
);
resources
    .charge(&R::from_native(native_cost))
    .map_err(|e| -> SystemError {
        ...
        e @ SystemError::LeafRuntime(RuntimeError::FatalRuntimeError(_)) => e,
    })?;
``` [3](#0-2) 

This is called on **every invocation** of a `DEFAULT_CODE_VERSION_BYTE` contract (no caching):

```rust
DEFAULT_CODE_VERSION_BYTE => {
    assert_eq!(artifacts_len, 0);
    let bytecode_preprocessing = BytecodePreprocessingData::create_artifacts(
        system.get_allocator(),
        bytecode,
        &mut available_resources,
    )?;
``` [4](#0-3) 

By contrast, `ARTIFACTS_CACHING_CODE_VERSION_BYTE` contracts use `parse_bytecode`, which does **not** charge native, so only `DEFAULT_CODE_VERSION_BYTE` contracts exhibit this behavior.

---

**Root cause 3 — `FatalRuntimeError` propagates past the EVM interpreter, bypassing CALL return-value handling:**

When `start_executing_frame` returns a `FatalRuntimeError`, it is wrapped and propagated up through `call_execute_callee_frame`:

```rust
let mut preemption = new_vm
    .start_executing_frame(...)
    .map_err(wrap_error!())?;
``` [5](#0-4) 

This error never reaches the EVM interpreter's CALL return-value processing path. The interpreter never sets the CALL success/failure flag on the stack. Therefore, any Solidity `try-catch` wrapping the CALL never fires — the entire transaction reverts instead.

Compare with the normal EVM error path, which does reach the interpreter and sets the return value:

```rust
if let ExitCode::EvmError(evm_error) = exit_code {
    if evm_error != EvmError::Revert {
        self.gas.consume_all_gas();
        return_values.returndata = &[];
    }
    ...
    return Ok(ExecutionEnvironmentPreemptionPoint::End(CompletedExecution {
        resources_returned: self.gas.take_resources(),
        result: CallResult::Failed { return_values },
    }));
``` [6](#0-5) 

`FatalRuntimeError` never reaches this path.

---

### Impact Explanation

A Solidity contract using `try-catch` to defensively call an untrusted or unknown contract **cannot protect itself** from native resource exhaustion in the callee. Instead of the `catch` block firing (as EVM semantics require), the entire transaction reverts. This is an **EVM semantic mismatch**:

- In standard EVM: subcall OOG → subcall fails → caller's `catch` fires → transaction continues.
- In ZKsync OS: subcall native OOG → `FatalRuntimeError` → entire transaction reverts → caller's `catch` never fires.

Concrete consequences:
1. A contract iterating over a list of addresses and calling each with `try-catch` (e.g., a router, a multi-token redemption, a batch operation) will have the **entire transaction reverted** if any callee exhausts native resource, even though the caller expected graceful failure handling.
2. An attacker can deploy a contract with `DEFAULT_CODE_VERSION_BYTE` and maximum bytecode size (24,576 bytes) to act as a native-resource sink. Any victim contract that calls it with `try-catch` will have its transaction reverted.
3. Wallet contracts or smart-account contracts that call arbitrary user-specified targets with `try-catch` are vulnerable to griefing.

---

### Likelihood Explanation

- **Attacker-controlled entry path**: Any unprivileged user can deploy a contract and cause another contract to call it.
- **Trigger condition**: The victim transaction must have a native resource budget low enough that the callee's preprocessing exhausts it. The native budget is `gasLimit × (gasPrice / nativePrice)`. At low gas prices (or high `nativePrice`), the native budget can be tight.
- **No privileged access required**: Fully reachable by an unprivileged transaction sender.
- **Likelihood**: Medium — requires specific conditions (low native budget, `DEFAULT_CODE_VERSION_BYTE` callee), but the conditions are realistic and attacker-controllable.

---

### Recommendation

1. **Convert native OOG in subcalls to a subcall failure**: When `FatalRuntimeError` due to native exhaustion occurs inside a subcall frame, treat it as a failed CALL (return 0 on the stack) rather than propagating it as a transaction-level fatal error. This would restore EVM semantics.
2. **Allow per-subcall native caps**: Analogous to the EVM 63/64 gas rule, allow callers to retain a portion of native resource when making subcalls, so native exhaustion in a callee cannot consume the caller's entire budget.
3. **Charge native for bytecode preprocessing upfront at deployment**: Ensure `DEFAULT_CODE_VERSION_BYTE` contracts are upgraded to `ARTIFACTS_CACHING_CODE_VERSION_BYTE` so that `create_artifacts` is not called on every invocation.

---

### Proof of Concept

```solidity
// Attacker deploys this with DEFAULT_CODE_VERSION_BYTE and 24,576 bytes of bytecode
contract NativeSink {
    fallback() external { /* 24,576 bytes of bytecode */ }
}

// Victim contract — expects try-catch to protect it
contract Victim {
    NativeSink sink = NativeSink(/* attacker address */);

    function batchProcess(address[] calldata targets) external {
        for (uint i = 0; i < targets.length; i++) {
            try ITarget(targets[i]).process() {
                // success
            } catch {
                // expected to catch failures gracefully
                // NEVER REACHED if targets[i] == NativeSink and native budget is tight
            }
        }
    }
}
```

Steps:
1. Attacker deploys `NativeSink` with `DEFAULT_CODE_VERSION_BYTE` and 24,576 bytes of bytecode.
2. Victim calls `batchProcess` with `NativeSink` as one of the targets, at a gas price where `nativePerGas` is low.
3. When `NativeSink` is called, `create_artifacts` charges 147,456 native units.
4. If native budget is exhausted, `FatalRuntimeError` propagates up, reverting the entire transaction.
5. The `catch` block in `batchProcess` never fires.

The gas cost difference between calling a `DEFAULT_CODE_VERSION_BYTE` contract with large bytecode vs. a small one mirrors the `cdai` vs. `ceth` gas difference in the reference report.
