### Title
WalletConnect `chia_revokeVC` Confirmation Dialog Exposes Raw `vc_parent_id` Without VC Identity Resolution, Enabling Arbitrary VC Revocation — (`packages/gui/src/electron/commands/Commands.ts`, `packages/gui/src/electron/main.tsx`)

---

### Summary

A dApp granted `chia_revokeVC` permission can supply any `vc_parent_id` from the wallet in its request. The WalletConnect confirmation dialog displays only the raw hex parent-coin-ID string labeled "Parent Coin Id" — it never resolves that ID to a VC name, launcher ID, or any human-readable identity. The user cannot determine which VC is being targeted, so they can be tricked into approving revocation of a VC they did not intend to revoke.

---

### Finding Description

**Step 1 — Schema definition**

The `chia_wallet.vc_revoke` command schema in `Commands.ts` declares two params: `vc_parent_id` (required, type `string`) and `fee` (required, type `bigint`). There is no validation that `vc_parent_id` corresponds to any specific VC the user selected. [1](#0-0) 

**Step 2 — `parseDappParams` accepts any string value**

`parseDappParams` validates that only declared params are present and that required params are provided. For `vc_parent_id` (type `string`), it simply calls `String(value)` — no check that the value corresponds to a VC owned by the wallet. [2](#0-1) 

**Step 3 — Confirmation dialog shows only the raw hex string**

In `main.tsx`, `humanizeDappCommand` is called to build the confirmation dialog rows. It calls `humanizeParams` against the schema, which renders `vc_parent_id` as the label "Parent Coin Id" with the raw hex value as the displayed value. No VC name, launcher ID, or human-readable identity is shown. [3](#0-2) 

**Step 4 — `parseCommandDisplay` provides no enrichment for `vc_revoke`**

`parseCommandDisplay` only enriches the display for `chia_wallet.take_offer` and `chia_wallet.create_offer_for_ids`. For `chia_wallet.vc_revoke` it returns `undefined`, so no additional VC identity context is injected into the dialog. [4](#0-3) 

**Step 5 — Confirmed `parsedParams` forwarded verbatim to daemon**

After user confirmation, `sendCommand('vc_revoke', 'chia_wallet', parsedParams)` is called with the dApp-supplied `vc_parent_id` unchanged. The daemon revokes whichever VC has that parent coin. [5](#0-4) 

**Contrast with the native UI flow**

In the native `VCCard.tsx` flow, `vcParentId` is always taken from `vcRecord.vc?.coin.parentCoinInfo` — the VC the user explicitly navigated to — and the dialog shows the VC's human-readable `vcTitle`. The WalletConnect path has no equivalent binding. [6](#0-5) 

---

### Impact Explanation

A malicious dApp that has been granted `chia_revokeVC` can send `chia_revokeVC` with the `vc_parent_id` of any VC in the wallet — not the one the user expects. The confirmation dialog shows only a raw 32-byte hex string ("Parent Coin Id: `0xBBBB…`"). A typical user cannot map that to a VC name or launcher ID without external tooling. If the user clicks "Revoke", the daemon executes an on-chain `vc_revoke` spend for the wrong VC, permanently destroying it. This is an irreversible on-chain action.

Impact category: **High** — WalletConnect state causes a user to approve revocation of the wrong asset/identity.

---

### Likelihood Explanation

Preconditions are realistic:
1. The user has granted the dApp `chia_revokeVC` (a destructive permission, but one users grant to VC-management dApps).
2. The wallet holds more than one VC.
3. The user does not manually verify the raw hex parent-coin-ID in the dialog (virtually no user does).

The dApp needs no additional privileges beyond what was already granted during pairing.

---

### Recommendation

In `parseCommandDisplay` (or a new enrichment step), resolve `vc_parent_id` to the corresponding VC record by calling `vc_get_list` and matching `coin.parentCoinInfo`. Display the VC's launcher ID and human-readable title in the confirmation dialog alongside (or instead of) the raw hex string. If the `vc_parent_id` does not match any VC in the wallet, reject the request before showing the dialog. [4](#0-3) 

---

### Proof of Concept

1. Connect a dApp via WalletConnect and grant it `chia_revokeVC`.
2. Ensure the wallet has two VCs: VC-A (launcher `0xAAAA…`, parent coin `0xPPPP…`) and VC-B (launcher `0xBBBB…`, parent coin `0xQQQQ…`).
3. The dApp sends:
   ```json
   { "method": "chia_revokeVC", "params": { "vcParentId": "0xQQQQ…", "fee": "0" } }
   ```
   where `0xQQQQ…` is VC-B's parent coin ID, not VC-A's.
4. The confirmation dialog appears showing:
   - Title: "Confirm Revoke VC"
   - "Parent Coin Id: 0xQQQQ…"
   - No VC name, no launcher ID
5. User clicks "Revoke" believing they are revoking VC-A.
6. The daemon executes `vc_revoke` for VC-B. VC-B is permanently revoked on-chain. [7](#0-6)

export async function parseCommandDisplay(command: string, params: Record<string, unknown>) {
  if (command === 'chia_wallet.take_offer') {
    if (!params.offer || typeof params.offer !== 'string') {
      throw new Error('Offer is not valid');
    }

    const offerSummary = await getOfferSummary(params.offer);
    if (!offerSummary || !offerSummary.summary || !offerSummary.success) {
      throw new Error('Offer is not valid');
    }

    const { summary } = offerSummary;

    const walletDelta = offerSummaryToWalletDelta(summary);
    const walletInfos = await getWalletInfos();
    const assetKinds = offerSummaryAssetKinds(summary);
    const royaltyPercentages = offerSummaryRoyaltyPercentages(summary);
    const fees = parseMojos(summary.fees);

    return {
      walletDelta: await walletDeltaToDisplay(walletDelta, walletInfos, assetKinds, royaltyPercentages, fees),
    };
  }

  if (command === 'chia_wallet.create_offer_for_ids') {
    if (!params.offer || !isPlainObject(params.offer)) {
      throw new Error('Offer is not valid');
    }

    if (params.driver_dict !== undefined && !isPlainObject(params.driver_dict)) {
      throw new Error('Driver Dict is not valid');
    }

    const walletDelta = createOfferToWalletDelta(params.offer);
    const walletInfos = await getWalletInfos();
    const driverDict = params.driver_dict ?? {};
    const assetKinds = createOfferAssetKinds(walletDelta, walletInfos, driverDict);
    const royaltyPercentages = createOfferRoyaltyPercentages(walletDelta, driverDict);

    return {
      walletDelta: await walletDeltaToDisplay(walletDelta, walletInfos, assetKinds, royaltyPercentages, undefined),
    };
  }

  return undefined;
}
