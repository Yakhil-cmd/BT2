### Title
WalletConnect `take_offer` Confirmation Displays Attacker-Controlled Royalty When NFT Lookup Fails - (File: `packages/gui/src/electron/commands/parseCommandDisplay.ts`)

### Summary
When a WalletConnect dapp sends `chia_wallet.take_offer`, the confirmation dialog's `amountWithRoyalties` field is seeded from the offer blob's attacker-controlled `infos` field. The code attempts to override this with the authoritative on-chain royalty via `nftGetInfo`, but silently falls back to the attacker-supplied value if that call fails. A user can therefore be shown a lower total cost than they will actually pay, causing them to approve a transaction for the wrong amount.

### Finding Description

In `parseCommandDisplay` for `chia_wallet.take_offer`, the royalty percentage used to compute the displayed `amountWithRoyalties` is initialized from `offerSummaryRoyaltyPercentages(summary)`: [1](#0-0) 

This function reads `royalty_percentage` directly out of the offer summary's `infos` field, which is derived from the offer blob supplied by the dapp (attacker-controlled): [2](#0-1) 

Inside `parseWalletDeltaItem`, the code then calls `nftGetInfo` to override the attacker-supplied value with the authoritative on-chain royalty. However, if `nftGetInfo` throws or returns `success: false`, the entire catch block is silently swallowed and `result.royaltyPercentage` retains the attacker-controlled seed value: [3](#0-2) 

`withRoyaltyTotals` then computes `amountWithRoyalties` from whatever `royaltyPercentage` is present on the NFT line items: [4](#0-3) 

If the attacker omits or zeroes `royalty_percentage` in the offer's `infos`, `royaltyPercentagesForSide` returns an empty array (it filters out values `<= 0`), so `amountWithRoyalties` is `undefined` — the dialog shows only the base price with no royalty line at all. [5](#0-4) 

The full `take_offer` dispatch path in `parseCommandDisplay`: [6](#0-5) 

The `chia_wallet.take_offer` command is registered as a WalletConnect-dispatchable command, meaning any connected dapp can trigger this confirmation flow: [7](#0-6) 

### Impact Explanation

The `amountWithRoyalties` field is the primary signal shown to the user in the WalletConnect confirmation dialog indicating how much XCH or CAT they will actually spend (base price + royalty). If this value is suppressed or understated because the attacker zeroed the royalty in the offer's `infos` and `nftGetInfo` failed, the user approves a transaction believing they are spending, say, 1 XCH, while the wallet backend enforces the actual on-chain royalty and debits 1.5 XCH. This is a direct, concrete accounting discrepancy between what the user approved and what is spent — matching the "wrong amount" criterion under the High impact scope.

### Likelihood Explanation

`nftGetInfo` calls `nft_get_info` on the local wallet daemon: [8](#0-7) 

This call can fail or return `success: false` when the wallet is not yet synced, when the NFT is not yet indexed (e.g., a freshly minted NFT), or when there is a transient RPC error. A dapp that times its offer delivery to coincide with wallet startup or a resync window can reliably trigger the fallback path. The attacker needs only a connected WalletConnect session and a crafted offer blob — no leaked keys or host access required.

### Recommendation

When `nftGetInfo` fails for an NFT asset in a `take_offer` command, the confirmation dialog should not silently fall back to the offer-embedded royalty. Instead:

1. Display an explicit warning that royalty information could not be verified and the total cost is unknown.
2. Or refuse to render `amountWithRoyalties` entirely and block approval until the royalty can be confirmed from the chain.

The catch block comment ("metadata is best effort") is appropriate for cosmetic fields like `previewUrl`, but `royaltyPercentage` directly affects the financial total shown to the user and must not be treated as best-effort.

### Proof of Concept

1. A WalletConnect dapp connects to the user's wallet.
2. The dapp calls `chia_takeOffer` with an offer blob for a real NFT that carries a 50% on-chain royalty (`royalty_percentage = 5000`).
3. The offer blob's driver dict sets `royalty_percentage: 0` (or omits the `also.also.transfer_program` path entirely).
4. The user's wallet is mid-sync, so `nftGetInfo` returns `{ success: false }`.
5. `parseWalletDeltaItem` returns `royaltyPercentage: undefined`; `royaltyPercentagesForSide` returns `[]`; `amountWithRoyalties` is `undefined`.
6. The confirmation dialog shows: **Spending: 1 XCH** (no royalty line).
7. The user approves.
8. The wallet backend enforces the actual 50% royalty; the user's balance is debited **1.5 XCH**.

function royaltyPercentageForDriverInfo(driverInfo: unknown): number | undefined {
  if (!isPlainObject(driverInfo)) {
    return undefined;
  }

  const { also } = driverInfo;
  if (!isPlainObject(also)) {
    return undefined;
  }

  const ownershipLayer = also.also;
  if (!isPlainObject(ownershipLayer)) {
    return undefined;
  }

  const transferProgram = ownershipLayer.transfer_program;
  if (!isPlainObject(transferProgram)) {
    return undefined;
  }

  return parseRoyaltyPercentage(transferProgram.royalty_percentage);
}

function offerSummaryRoyaltyPercentages(offerSummary: OfferSummaryForDisplay): AssetRoyaltyPercentages {
  const royaltyPercentages: AssetRoyaltyPercentages = {
    spending: {},
    receiving: {},
  };

  const { infos } = offerSummary;
  if (!isPlainObject(infos)) {
    return royaltyPercentages;
  }

  for (const assetId of Object.keys(offerSummary.requested)) {
    const parsedAssetId = assetId === 'xch' ? '1' : assetId;
    royaltyPercentages.spending[parsedAssetId] = royaltyPercentageForDriverInfo(infos[assetId]);
  }

  for (const assetId of Object.keys(offerSummary.offered)) {
    const parsedAssetId = assetId === 'xch' ? '1' : assetId;
    royaltyPercentages.receiving[parsedAssetId] = royaltyPercentageForDriverInfo(infos[assetId]);
  }

  return royaltyPercentages;

function royaltyPercentagesForSide(lines: DisplayWalletDeltaItem[]): number[] {
  return lines
    .filter((line): line is Extract<DisplayWalletDeltaItem, { kind: 'nft' }> => line.kind === 'nft')
    .map((line) => line.royaltyPercentage)
    .filter(
      (royaltyPercentage): royaltyPercentage is number => royaltyPercentage !== undefined && royaltyPercentage > 0,
    );
}

function withRoyaltyTotals(
  items: DisplayWalletDeltaItemWithKey[],
  amounts: Record<string, bigint>,
  oppositeSideLines: DisplayWalletDeltaItem[],
): DisplayWalletDeltaItem[] {
  const royaltyPercentages = royaltyPercentagesForSide(oppositeSideLines);

  return items.map(({ key, line }) => {
    if (line.kind === 'nft') {
      return line;
    }

    const amount = amounts[key];
    const amountWithRoyalties = amount ? formatAmountWithRoyalties(line, amount, royaltyPercentages) : undefined;

    return amountWithRoyalties ? { ...line, amountWithRoyalties } : line;
  });
}

export async function parseCommandDisplay(command: string, params: Record<string, unknown>) {
  if (command === 'chia_wallet.take_offer') {
    if (!params.offer || typeof params.offer !== 'string') {
      throw new Error('Offer is not valid');
    }

    const offerSummary = await getOfferSummary(params.offer);
    if (!offerSummary || !offerSummary.summary || !offerSummary.success) {
      throw new Error('Offer is not valid');
    }

    const { summary } = offerSummary;

    const walletDelta = offerSummaryToWalletDelta(summary);
    const walletInfos = await getWalletInfos();
    const assetKinds = offerSummaryAssetKinds(summary);
    const royaltyPercentages = offerSummaryRoyaltyPercentages(summary);
    const fees = parseMojos(summary.fees);

    return {
      walletDelta: await walletDeltaToDisplay(walletDelta, walletInfos, assetKinds, royaltyPercentages, fees),
    };
  }
