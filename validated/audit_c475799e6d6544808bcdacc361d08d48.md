### Title
Missing Participant-Set Consistency Check Between Triple Generation and Presign Phases — (`src/ecdsa/ot_based_ecdsa/presign.rs`)

---

### Summary

The OT-based ECDSA `presign()` function explicitly omits any check that the participant set invoking presign matches the participant set that was used during Beaver triple generation. A participant who was absent from triple generation — or who uses triple shares produced under a different participant set — can participate in presigning without triggering any validation error, causing the presign output to be cryptographically inconsistent and unusable.

---

### Finding Description

In `src/ecdsa/ot_based_ecdsa/presign.rs`, the `presign()` entry-point validates threshold consistency between the current session and the embedded triple metadata, but contains an explicit admission that participant-set consistency is not enforced:

```rust
// NOTE: We omit the check that the new participant set was present for
// the triple generation, because presumably they need to have been present
// in order to have shares.
``` [1](#0-0) 

The only cross-phase check that is performed is a threshold equality guard:

```rust
if args.threshold != args.triple0.1.threshold || args.threshold != args.triple1.1.threshold {
    return Err(InitializationError::BadParameters(
        "New threshold must match the threshold of both triples".to_string(),
    ));
}
``` [2](#0-1) 

No check is made that the `ParticipantList` passed to `presign()` is identical to — or a valid subset of — the participant list under which `triple0` and `triple1` were generated. The `PresignArguments` struct is caller-supplied, so any participant can invoke `presign()` with:

- Triple shares produced in a prior session under a **different participant set** (e.g., a set that included parties no longer present, or excluded parties now present), or
- Triple shares from a session in which the invoking participant was **not present**.

Because Beaver triple shares encode Lagrange-interpolation coefficients relative to the generating participant set, using them under a different participant set causes the masked multiplication values to be computed with wrong coefficients. The resulting presign output — nonces, masked key shares, and the combined `R` point — will be cryptographically inconsistent across honest parties.

The analog to the external report is direct: just as a late lottery joiner could call `updateAfterLoss()` and accumulate `winChance` increments for rounds they never participated in, a participant absent from triple generation can call `presign()` and have their fabricated or mismatched triple shares accepted and folded into the protocol state, corrupting the output for all honest parties.

---

### Impact Explanation

**High — Corruption of presign outputs so honest parties accept unusable cryptographic outputs.**

When the presign participant set differs from the triple-generation participant set, the Lagrange interpolation inside `do_presign` operates on shares that were secret-shared under a different evaluation domain. The resulting presign transcript will be internally inconsistent: each honest party will compute a different effective nonce contribution, and the aggregate `R = k·G` will not correspond to any consistent scalar `k`. Any subsequent signing attempt using this presign output will produce an invalid ECDSA signature, permanently denying signing for the affected session.

---

### Likelihood Explanation

The `PresignArguments` struct — including both triples and the threshold — is constructed entirely by the caller before invoking `presign()`. There is no binding between a triple's provenance (which participant set generated it) and the participant list passed to `presign()`. A malicious coordinator or a participant who controls their own node can trivially supply triple shares from a prior session with a different participant set. The comment in the source code confirms the omission is intentional, meaning no future internal check will catch this at runtime.

---

### Recommendation

Record the generating participant set inside the triple metadata (alongside the existing `threshold` field) and enforce equality — or a documented subset relationship — between that stored set and the `participants` argument at the start of `presign()`. The threshold check already demonstrates the pattern; the participant-set check should follow the same structure:

```rust
if args.participants_at_generation != current_participants {
    return Err(InitializationError::BadParameters(
        "Participant set must match the set used during triple generation".to_string(),
    ));
}
```

If supporting subset-presign is a design goal, document the exact subset rules and enforce them cryptographically (e.g., by re-randomizing Lagrange coefficients relative to the generating set before use).

---

### Proof of Concept

1. Run triple generation with participant set `P_gen = {A, B, C, D}` at threshold `t`. Each party receives shares `(a_i, b_i, c_i)` with Lagrange weights relative to `P_gen`.

2. Party `D` is excluded from the signing session. A new presign is initiated with `P_sign = {A, B, C}` at the same threshold `t`.

3. `A`, `B`, `C` each call `presign(participants = [A,B,C], me = self, args = PresignArguments { threshold: t, triple0: (share_from_P_gen, meta_with_t), triple1: ... })`.

4. The threshold check `args.threshold == triple0.1.threshold` passes (both are `t`). No participant-set check exists.

5. Inside `do_presign`, each party computes its masked contribution using Lagrange coefficients for `P_sign = {A,B,C}`, but the underlying shares `a_i` were secret-shared for `P_gen = {A,B,C,D}`. The reconstructed masked values are wrong.

6. The presign output is cryptographically inconsistent. All subsequent signing attempts using this presign output produce invalid signatures, permanently denying signing for this session. [3](#0-2)

    me: Participant,

        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });

    // Spec 1.1

            threshold: args.threshold.value(),
            max: participants.len(),
        });

    // the triple generation, because presumably they need to have been present
    // in order to have shares.

    // Also check that we have enough participants to reconstruct shares.
    if args.threshold != args.triple0.1.threshold || args.threshold != args.triple1.1.threshold {
        return Err(InitializationError::BadParameters(
            "New threshold must match the threshold of both triples".to_string(),

            role: "self",
            participant: me,
        });
    }
