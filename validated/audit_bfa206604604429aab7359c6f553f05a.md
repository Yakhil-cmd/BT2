### Title
Missing `msg_hash == 0` Validation in OT-Based ECDSA Signing Enables Secret Key Extraction - (File: `src/ecdsa/ot_based_ecdsa/sign.rs`)

---

### Summary

The OT-based ECDSA `sign` function accepts a zero `msg_hash` without rejection. The robust ECDSA `sign` function explicitly rejects `msg_hash == 0` to prevent algebraic split-view attacks. This asymmetric validation gap allows a malicious coordinator to trigger a signing session with `h = 0`, and by reusing the same presignature in a second session with `h ≠ 0`, extract the aggregate secret key from the two resulting signature values.

---

### Finding Description

The robust ECDSA `sign` function enforces a zero-hash guard:

```rust
// src/ecdsa/robust_ecdsa/sign.rs:91-95
if bool::from(msg_hash.is_zero()) {
    return Err(InitializationError::BadParameters(
        "msg_hash cannot be 0 to avoid potential split view attacks".to_string(),
    ));
}
``` [1](#0-0) 

The OT-based ECDSA `sign` function performs no equivalent check. Its entire input validation block is:

```rust
// src/ecdsa/ot_based_ecdsa/sign.rs:32-63
if participants.len() < 2 { ... }
// participants dedup, self-presence, coordinator-presence, threshold checks
if participants.len() < threshold { ... }
// msg_hash is never validated
``` [2](#0-1) 

The signature share computation is:

```rust
// src/ecdsa/ot_based_ecdsa/sign.rs:155-158
let r = x_coordinate(&presignature.big_r);
Ok(msg_hash * k_i + r * sigma_i)
``` [3](#0-2) 

When `msg_hash = 0`, each participant's share collapses to `R_x · σ_i`, stripping the `h · k_i` term entirely. The coordinator aggregates `s_A = R_x · σ` (with `σ = k · x`), which is a direct linear function of the secret key `x`.

---

### Impact Explanation

The OT-based ECDSA signing formula (from the spec) is:

```
s = h · k + R_x · σ    where σ = k · x
``` [4](#0-3) 

This is equivalent to the standard ECDSA formula `s = k⁻¹ · (h + R_x · x)` (the library uses the inverse nonce convention). With `h = 0`:

- **Session A** (`h = 0`): `s_A = R_x · x / k_std`
- **Session B** (`h ≠ 0`, same presignature): `s_B = (h + R_x · x) / k_std`

From these two values:
- `k_std = h / (s_B − s_A)`
- `x = s_A · k_std / R_x`

The aggregate secret key `x` is fully recovered. This maps directly to the allowed Critical impact: **Extraction, reconstruction, or disclosure of private signing shares or aggregate secret material**.

---

### Likelihood Explanation

The attack requires a malicious coordinator who:
1. Initiates two signing sessions using the same presignature shares held by honest participants.
2. Presents `h = 0` in session A (accepted without error by the OT-based `sign` function).
3. Presents any `h ≠ 0` in session B.

The library does not enforce single-use of presignatures at the protocol level — it is a caller responsibility stated only in documentation. A malicious coordinator can trivially replay the same signing request to participants with different message hashes. Participants have no mechanism to detect presignature reuse. The coordinator role is a documented threat actor in this library's trust model. [5](#0-4) 

---

### Recommendation

Add the same zero-hash guard to `src/ecdsa/ot_based_ecdsa/sign.rs` that already exists in the robust ECDSA variant:

```rust
if bool::from(msg_hash.is_zero()) {
    return Err(InitializationError::BadParameters(
        "msg_hash cannot be 0".to_string(),
    ));
}
```

This should be inserted in the `sign` function immediately after the threshold check, mirroring the pattern at `src/ecdsa/robust_ecdsa/sign.rs:91-95`. [6](#0-5) 

---

### Proof of Concept

1. Run OT-based ECDSA keygen and presigning for `N` participants to obtain `(R, k_i, σ_i)` shares.
2. Invoke `ot_based_ecdsa::sign::sign(...)` with `msg_hash = Scalar::ZERO`. Observe: no `InitializationError` is returned; the protocol proceeds.
3. Coordinator collects all `s_i = R_x · σ_i` shares and sums them: `s_A = R_x · σ`.
4. Invoke `sign(...)` again with the **same presignature shares** and any `msg_hash = h ≠ 0`. Coordinator collects `s_B = (h + R_x · σ) / k_std · k_std = h/k_std + s_A`.
5. Compute `k_std = h / (s_B − s_A)` and `x = s_A · k_std / R_x`.
6. Verify `x · G` equals the public key output from keygen.

The robust ECDSA `sign` function would reject step 2 at line 91 with `"msg_hash cannot be 0"`. The OT-based ECDSA `sign` function has no equivalent guard. [1](#0-0) [2](#0-1)

pub fn sign(
    participants: &[Participant],
    coordinator: Participant,
    threshold: impl Into<ReconstructionLowerBound>,
    me: Participant,
    public_key: AffinePoint,
    presignature: RerandomizedPresignOutput,
    msg_hash: Scalar,
) -> Result<impl Protocol<Output = SignatureOption>, InitializationError> {
    let threshold = usize::from(threshold.into());
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }

    let participants =
        ParticipantList::new(participants).ok_or(InitializationError::DuplicateParticipants)?;

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }

    // ensure number of participants during the signing phase is >= threshold
    if participants.len() < threshold {
        return Err(InitializationError::NotEnoughParticipantsForThreshold {
            threshold,
            participants: participants.len(),
        });
    }

    let ctx = Comms::new();
    let fut = fut_wrapper(
        ctx.shared_channel(),
        participants,
        coordinator,
        me,
        public_key,
        presignature,
        msg_hash,
    );
    Ok(make_protocol(ctx, fut))
}
