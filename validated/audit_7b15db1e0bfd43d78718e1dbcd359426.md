### Title
Gateway Admits Unsigned Invoke Transactions via `skip_stateful_validations` UX Bypass — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` UX feature, designed to let users submit a `deploy_account` + `invoke` pair simultaneously, unconditionally skips the `__validate__` entry-point call (and therefore all signature verification) for any invoke transaction with `nonce == 1` sent from an address whose on-chain nonce is `0` and that has any transaction in the mempool. An unprivileged attacker who observes a victim's pending `deploy_account` can immediately submit an invoke with an arbitrary (invalid) signature from the victim's address, and the gateway will admit it to the mempool without ever verifying the signature.

### Finding Description

The gateway's stateful validation pipeline in `extract_state_nonce_and_run_validations` calls three sub-steps in sequence:

```
validate_state_preconditions  →  validate_by_mempool  →  skip_stateful_validations
                                                                  ↓
                                                       run_validate_entry_point(skip=true)
```

`skip_stateful_validations` returns `true` (skip) when all three conditions hold:

1. The transaction is an `Invoke`.
2. `tx.nonce() == Nonce(Felt::ONE)`.
3. `account_nonce == Nonce(Felt::ZERO)` (account not yet deployed on-chain).
4. `mempool_client.account_tx_in_pool_or_recent_block(sender_address)` returns `true`. [1](#0-0) 

When `skip_validate == true`, `run_validate_entry_point` sets `ExecutionFlags { validate: false, … }` and calls `blockifier_validator.validate(account_tx)`: [2](#0-1) 

Inside `StatefulValidator::perform_validations`, an invoke transaction with `validate == false` returns `Ok(())` immediately after `perform_pre_validation_stage`, **without ever calling `validate_tx` / the `__validate__` entry point**: [3](#0-2) 

`validate_by_mempool` (called before the skip check) only passes `ValidationArgs` containing `address`, `account_nonce`, `tx_hash`, `tx_nonce`, `tip`, and `max_l2_gas_price` — no signature field: [4](#0-3) 

The mempool's own `validate_tx` checks only for duplicate hashes, nonce ordering, and fee escalation — never the signature: [5](#0-4) 

`account_tx_in_pool_or_recent_block` returns `true` if **any** transaction from the address is in the pool — not specifically a `deploy_account`: [6](#0-5) 

**Attack path:**

1. Victim broadcasts `deploy_account` for address `X` (nonce 0). The mempool now contains a transaction from `X`, so `account_tx_in_pool_or_recent_block(X) == true`.
2. Attacker submits `Invoke { sender_address: X, nonce: 1, calldata: <anything>, signature: [] }`.
3. Gateway checks: `tx.nonce==1`, `account_nonce==0`, `account_tx_in_pool_or_recent_block(X)==true` → `skip_validate=true`.
4. `__validate__` is never called; the empty/forged signature is never checked.
5. The transaction passes `validate_by_mempool` (nonce 1 ≥ account nonce 0, no duplicate hash).
6. The transaction is admitted to the mempool, occupying the `(X, nonce=1)` slot.

### Impact Explanation

The gateway admits an invoke transaction with an invalid (or empty) signature to the mempool. This violates the admission invariant that every accepted transaction must have been authenticated by the account's `__validate__` entry point. Concretely:

- The attacker occupies the `nonce=1` slot for any address that has a pending `deploy_account`.
- The victim's legitimate invoke (also nonce=1) is either rejected as `DuplicateNonce` or must win a fee-escalation race against the attacker's invalid transaction.
- When the batcher eventually executes the attacker's transaction, it will revert (the blockifier does call `__validate__` during execution), but the victim's deploy-account + invoke UX flow is disrupted.
- The attacker can repeat this for every new `deploy_account` observed in the mempool, causing systematic griefing of all new account deployments.

This matches: **High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

### Likelihood Explanation

The `deploy_account` transaction is publicly visible in the mempool. Any observer can immediately submit a competing invoke with nonce=1 and an empty signature. No privileged access, no cryptographic secret, and no special knowledge of the victim's account parameters are required beyond reading the mempool. The window is open from the moment the `deploy_account` enters the mempool until it is committed on-chain.

### Recommendation

The `skip_stateful_validations` path must not skip signature verification entirely. Two complementary fixes:

1. **Restrict the skip to nonce=1 only when the mempool entry is specifically a `deploy_account`**, not any transaction. Add a `deploy_account_in_pool(address)` query to the mempool that checks the transaction type.

2. **Even when skipping the `__validate__` entry point** (because the account contract does not exist yet), perform a static signature check against the expected account address using the class hash and constructor data from the pending `deploy_account`. This is the approach used by other Starknet implementations.

Alternatively, require the user to submit the `deploy_account` first and wait for it to be included before accepting the invoke — removing the UX shortcut entirely in favour of correctness.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
