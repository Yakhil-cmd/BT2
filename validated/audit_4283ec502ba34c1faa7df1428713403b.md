### Title
Missing Silo Fallback Guard in `receive_base_tokens` Allows ETH to Be Permanently Frozen in Non-Whitelisted Contracts — (File: `engine/src/engine.rs`)

---

### Summary

In Silo mode, `receive_erc20_tokens` redirects bridged ERC-20 tokens to a configured fallback address when the intended recipient is not on the address whitelist. The analogous function `receive_base_tokens`, which handles bridged ETH (base tokens), contains no such guard. ETH credited to a non-whitelisted EVM contract that has no withdrawal capability is permanently frozen.

---

### Finding Description

`receive_erc20_tokens` contains an explicit silo fallback check:

```rust
if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
    && !silo::is_allow_receive_erc20_tokens(&self.io, &recipient)
{
    recipient = fallback_address;
}
``` [1](#0-0) 

`receive_base_tokens`, which runs on the same `ft_on_transfer` code path when the sender is the ETH connector, performs no equivalent check:

```rust
pub fn receive_base_tokens(
    &mut self,
    args: &FtOnTransferArgs,
) -> Result<Option<SubmitResult>, ContractError> {
    let message_data = FtTransferMessageData::try_from(args.msg.as_str())?;
    let amount = Wei::new_u128(args.amount.as_u128());
    let receipient = message_data.recipient;
    ...
    set_balance(&mut self.io, &receipient, &new_balance);
    Ok(None)
}
``` [2](#0-1) 

The silo `erc20_fallback_address` is documented as the address used "for withdrawing ERC-20 base tokens in case a recipient of the tokens is not in the silo white list": [3](#0-2) 

The whitelist check `is_allow_receive_erc20_tokens` simply delegates to the `Address` whitelist: [4](#0-3) 

The silo tests confirm that ERC-20 tokens sent to a non-whitelisted address are correctly redirected to the fallback address: [5](#0-4) 

No equivalent test or guard exists for base token (ETH) deposits to non-whitelisted addresses.

---

### Impact Explanation

When Silo mode is active and the `Address` whitelist is enabled, a NEAR user can call `ft_transfer_call` on the ETH connector specifying any EVM address as the recipient. `receive_base_tokens` credits the ETH balance to that address unconditionally. If the recipient is an EVM contract that has no `withdrawToNear`, `withdrawToEthereum`, or any other ETH-moving function, the ETH balance is permanently inaccessible — a permanent fund freeze. The silo fallback mechanism, which exists precisely to prevent this for ERC-20 tokens, is entirely absent for ETH.

**Impact class**: Critical — Permanent freezing of funds.

---

### Likelihood Explanation

Silo mode is a supported, documented production feature of Aurora Engine. Any NEAR account can call `ft_transfer_call` on the ETH connector and supply an arbitrary EVM address in the message field. A user who mistakenly (or deliberately) targets a non-whitelisted contract address with no ETH withdrawal logic will permanently lose their ETH. No privileged access is required beyond being a normal NEAR token holder.

---

### Recommendation

Apply the same silo fallback guard to `receive_base_tokens` that is already applied in `receive_erc20_tokens`:

```rust
pub fn receive_base_tokens(
    &mut self,
    args: &FtOnTransferArgs,
) -> Result<Option<SubmitResult>, ContractError> {
    let message_data = FtTransferMessageData::try_from(args.msg.as_str())?;
    let amount = Wei::new_u128(args.amount.as_u128());
    let mut receipient = message_data.recipient;

    // Mirror the ERC-20 silo fallback guard for base tokens
    if let Some(fallback_address) = silo::get_erc20_fallback_address(&self.io)
        && !silo::is_allow_receive_erc20_tokens(&self.io, &receipient)
    {
        receipient = fallback_address;
    }

    let balance = get_balance(&self.io, &receipient);
    let new_balance = balance
        .checked_add(amount)
        .ok_or(errors::ERR_BALANCE_OVERFLOW)?;
    set_balance(&mut self.io, &receipient, &new_balance);
    Ok(None)
}
```

---

### Proof of Concept

1. Deploy Aurora Engine in Silo mode: call `set_silo_params` with a valid `erc20_fallback_address` and enable the `Address` whitelist via `set_whitelist_status`.
2. Deploy a minimal EVM contract `Sink` that has no `receive()`, no `fallback()`, and no function that moves ETH or calls `withdrawToNear`.
3. Do **not** add `Sink`'s address to the whitelist.
4. From any NEAR account, call `ft_transfer_call` on the ETH connector, targeting the Aurora Engine contract, with `msg` set to the hex-encoded address of `Sink`.
5. `ft_on_transfer` → `receive_base_tokens` is invoked. No silo check is performed. ETH balance is credited to `Sink`.
6. Observe: `Sink` holds ETH balance. No EVM transaction can move it out. The ETH is permanently frozen.
7. Contrast: repeat with an ERC-20 token via `receive_erc20_tokens` — the tokens are correctly redirected to the fallback address instead. [2](#0-1) [1](#0-0)
