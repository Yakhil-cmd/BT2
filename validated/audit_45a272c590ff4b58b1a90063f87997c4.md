### Title
Missing Access Control on `update_rewards` Allows Any Caller to Permanently Deny Block Rewards - (File: src/staking/staking.cairo)

### Summary

The `update_rewards` function in the `StakingRewardsManagerImpl` is specified to be callable only by the Starkware sequencer, but the implementation contains **no caller check whatsoever**. Because the function writes to a single global `last_reward_block` storage variable before distributing rewards, any unprivileged address can call it with `disable_rewards: true` to permanently consume a block's reward slot without distributing any rewards, causing irreversible loss of yield for all stakers and delegators.

### Finding Description

The protocol specification for `update_rewards` explicitly states:

> **Access control:** Only starkware sequencer.

However, the implementation at `src/staking/staking.cairo` lines 1449–1507 contains no such check. The function only calls `self.general_prerequisites()` (a pause check) and validates the staker, but never verifies the caller's identity:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
    let current_block_number = starknet::get_block_number();
    assert!(
        current_block_number > self.last_reward_block.read(),
        "{}",
        Error::REWARDS_ALREADY_UPDATED,
    );
    // ... staker validation ...

    // Update last block rewards.
    self.last_reward_block.write(current_block_number);  // <-- global slot consumed HERE

    if disable_rewards || self.is_pre_consensus() {
        return;  // <-- returns with NO rewards distributed
    }
    // ... reward distribution ...
}
```

There is no `CALLER_IS_NOT_SEQUENCER` error, no `only_sequencer` guard, and no `sequencer_address` check anywhere in the codebase.

The critical design detail is that `last_reward_block` is a **single global storage variable** (not per-staker). The protocol allows exactly **one** `update_rewards` call per block number. Once `last_reward_block` is written to the current block, any subsequent call in the same block reverts with `REWARDS_ALREADY_UPDATED`. This means:

- Calling `update_rewards(any_valid_staker, disable_rewards: true)` consumes the block's reward slot globally.
- No rewards are distributed for that block to any staker or delegator.
- The block is gone; the rewards are permanently lost.

### Impact Explanation

**High — Permanent freezing of unclaimed yield.**

An attacker who calls `update_rewards(valid_staker, disable_rewards: true)` once per block permanently destroys the block rewards for every staker and every delegator in the system for that block. Because the attack is cheap (only gas cost) and repeatable every block, a sustained attacker can deny 100% of consensus-era block rewards indefinitely. Delegators and stakers accumulate zero `unclaimed_rewards_own` and zero pool balances, constituting permanent freezing of all unclaimed yield.

### Likelihood Explanation

**High.** The entry point is fully public — no stake, no role, no special key is required. The attacker only needs to submit a transaction before the sequencer's own `update_rewards` call each block. On Starknet, where the sequencer ordering is observable, this is straightforward. The attack requires no capital and has no revert risk as long as a valid `staker_address` is supplied.

### Recommendation

Add a sequencer-only guard at the top of `update_rewards`. In Starknet Cairo, the sequencer address is available via `starknet::get_execution_info().unbox().block_info.unbox().sequencer_address`. The check should be:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
    let sequencer = starknet::get_execution_info().unbox().block_info.unbox().sequencer_address;
    assert!(get_caller_address() == sequencer, "{}", Error::CALLER_IS_NOT_SEQUENCER);
    // ... rest of function
}
```

A corresponding `CALLER_IS_NOT_SEQUENCER` error variant should be added to `src/staking/errors.cairo`.

### Proof of Concept

1. Consensus rewards are active (post `consensus_rewards_first_epoch + K`).
2. At block N, attacker submits: `update_rewards(staker: any_valid_staker, disable_rewards: true)`.
3. The function passes all checks, writes `last_reward_block = N`, then returns early — no rewards distributed.
4. The sequencer's own `update_rewards` call for block N now reverts with `REWARDS_ALREADY_UPDATED`.
5. All stakers' `unclaimed_rewards_own` and all pool balances remain unchanged for block N.
6. Attacker repeats every block. All consensus-era block rewards are permanently destroyed.
