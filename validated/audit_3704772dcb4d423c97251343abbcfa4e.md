### Title
Duplicate Authorizer Addresses Passed to Cadence Runtime Without Uniqueness Check — (`fvm/environment/transaction_info.go`)

### Summary

`NewTransactionInfo` in `fvm/environment/transaction_info.go` iterates over `tx.Authorizers` and appends each entry directly to `runtimeAddresses` without checking for duplicates. Because neither the Access-node validator (`access/validator/validator.go`) nor the FVM's `TransactionVerifier` (`fvm/transactionVerifier.go`) reject duplicate authorizer addresses, an unprivileged transaction sender can craft a transaction whose `Authorizers` field contains the same address twice. The FVM's weight-check logic uses a map and therefore only requires **one** cryptographic signature, yet `GetSigningAccounts()` returns the raw duplicated list to the Cadence runtime, which binds both `prepare` parameters to the same account. This creates an authorization inconsistency: the protocol enforces one signature while the Cadence runtime observes two signing accounts.

---

### Finding Description

**Root cause — `NewTransactionInfo` appends without deduplication:** [1](#0-0) 

```go
for _, auth := range params.TxBody.Authorizers {
    runtimeAddresses = append(
        runtimeAddresses,
        common.Address(auth))
    if auth == serviceAccount {
        isServiceAccountAuthorizer = true
    }
}
```

The resulting `runtimeAddresses` slice (which may contain duplicates) is returned verbatim by `GetSigningAccounts()`: [2](#0-1) 

**No duplicate-authorizer check in the Access-node validator:**

`checkAddresses` only validates that each address is a valid chain address; it does not detect repeated entries: [3](#0-2) 

`checkAccounts` detects duplicate *signatures* (same address + key-index pair) but never checks whether the same address appears more than once in `tx.Authorizers`: [4](#0-3) 

**No duplicate-authorizer check in the FVM verifier:**

`verifyTransaction` iterates over `tx.Authorizers` and checks `payloadWeights[addr]` for each entry. Because `payloadWeights` is a map, a duplicated address passes the weight check on both iterations with only one signature: [5](#0-4) 

**`AddAuthorizer` appends without uniqueness enforcement:** [6](#0-5) 

The validation pipeline registered in `initValidationSteps` contains no step that rejects duplicate authorizer addresses: [7](#0-6) 

---

### Impact Explanation

A transaction with `Authorizers = [addr, addr]` passes all validation and execution checks while requiring only a single cryptographic signature from `addr`. The Cadence runtime receives `[addr, addr]` from `GetSigningAccounts()` and binds both `prepare` parameters to the same account object. Any Cadence transaction or contract interaction that is designed to require two *distinct* authorizing accounts (e.g., a two-party asset swap, a multi-sig escrow release, or a co-signer approval flow) can be satisfied by a single account duplicating itself in the authorizer list. This constitutes an **entitlement/authorization bypass**: the protocol's signature-weight enforcement is circumvented for the second authorizer slot, allowing one party to impersonate two.

---

### Likelihood Explanation

The attack path is fully reachable by any unprivileged user who can submit a transaction to the Access API. No special node role, leaked key, or admin privilege is required. The attacker only needs to set `Authorizers = [theirAddress, theirAddress]` in a transaction body targeting a contract that does not itself assert `signerA.address != signerB.address`. Standard multi-party Cadence transaction templates that rely on the protocol to enforce distinct signers are vulnerable.

---

### Recommendation

Add a uniqueness check for `tx.Authorizers` in the Access-node validator (analogous to the existing duplicate-signature check) and in `NewTransactionInfo` before building `runtimeAddresses`:

**In `access/validator/validator.go` (`checkAddresses` or a new step):**
```go
seen := make(map[flow.Address]struct{}, len(tx.Authorizers))
for _, addr := range tx.Authorizers {
    if _, exists := seen[addr]; exists {
        return DuplicateAuthorizerError{Address: addr}
    }
    seen[addr] = struct{}{}
}
```

**In `fvm/environment/transaction_info.go` (`NewTransactionInfo`):**
```go
seen := make(map[flow.Address]struct{}, len(params.TxBody.Authorizers))
for _, auth := range params.TxBody.Authorizers {
    if _, exists := seen[auth]; exists {
        continue // or return an error
    }
    seen[auth] = struct{}{}
    runtimeAddresses = append(runtimeAddresses, common.Address(auth))
    ...
}
```

```go
func NewTransactionInfo(
	params TransactionInfoParams,
	tracer tracing.TracerSpan,
	serviceAccount flow.Address,
) TransactionInfo {

	isServiceAccountAuthorizer := false
	runtimeAddresses := make(
		[]common.Address,
		0,
		len(params.TxBody.Authorizers))

	for _, auth := range params.TxBody.Authorizers {
		runtimeAddresses = append(
			runtimeAddresses,
			common.Address(auth))
		if auth == serviceAccount {
			isServiceAccountAuthorizer = true
		}
	}

	return &transactionInfo{
		params:                     params,
		tracer:                     tracer,
		runtimeAuthorizers:         runtimeAddresses,
		isServiceAccountAuthorizer: isServiceAccountAuthorizer,
	}
}
```

```go
func (v *TransactionValidator) checkAccounts(tx *flow.TransactionBody) error {
	// check for duplicate account key
	type uniqueKey struct {
		address flow.Address
		index   uint32
	}
	observedSigs := make(map[uniqueKey]bool)
	for _, sig := range append(tx.PayloadSignatures, tx.EnvelopeSignatures...) {
		if observedSigs[uniqueKey{sig.Address, sig.KeyIndex}] {
			return DuplicatedSignatureError{Address: sig.Address, KeyIndex: sig.KeyIndex}
		}
		observedSigs[uniqueKey{sig.Address, sig.KeyIndex}] = true
	}
	// check for minimum account signatures
	observedEnvelopeSig := make(map[flow.Address]bool)
	observedPayloadSig := make(map[flow.Address]bool)
	for _, sig := range tx.EnvelopeSignatures {
		observedEnvelopeSig[sig.Address] = true
	}
	for _, sig := range tx.PayloadSignatures {
		observedPayloadSig[sig.Address] = true
	}

	if !observedEnvelopeSig[tx.Payer] {
		return MissingSignatureError{Address: tx.Payer, Message: "payer envelope signature is missing"}
	}

	if !observedEnvelopeSig[tx.ProposalKey.Address] && !observedPayloadSig[tx.ProposalKey.Address] {
		return MissingSignatureError{Address: tx.ProposalKey.Address, Message: "proposer signature on either payload or envelope is missing"}
	}

	for _, authorizer := range tx.Authorizers {
		if authorizer == tx.Payer || authorizer == tx.ProposalKey.Address {
			// at this point, payer and proposer are guaranteed to have signatures
			continue
		}
		if !observedEnvelopeSig[authorizer] && !observedPayloadSig[authorizer] {
			return MissingSignatureError{Address: authorizer, Message: "authorizer signature on either payload or envelope is missing"}
		}
	}

	// check for unrelated account signatures
	relatedAccounts := make(map[flow.Address]struct{})
	relatedAccounts[tx.Payer] = struct{}{}
	relatedAccounts[tx.ProposalKey.Address] = struct{}{}
	for _, authorizer := range tx.Authorizers {
		relatedAccounts[authorizer] = struct{}{}
	}
	for _, sig := range append(tx.PayloadSignatures, tx.EnvelopeSignatures...) {
		if _, ok := relatedAccounts[sig.Address]; !ok {
			return UnrelatedAccountSignatureError{Address: sig.Address}
		}
	}
	return nil
```
