### Title
Gateway Converter Unconditionally Rejects V1 CASM Hash Declares That Blockifier Accepts When `block_casm_hash_v1_declares=false` — (`crates/apollo_transaction_converter/src/transaction_converter.rs`)

### Summary
`convert_rpc_tx_to_internal` always compares `tx.compiled_class_hash` against `executable_class_hash_v2` (the Blake2/V2 hash returned by `add_class`). When a user submits a DeclareV3 transaction with `compiled_class_hash = casm.hash(&HashVersion::V1)`, the converter rejects it unconditionally. However, when `block_casm_hash_v1_declares=false`, the blockifier's `run_execute` skips `check_compile_class_hash_v2_declaration` entirely and would accept the same transaction. The gateway is strictly more restrictive than the blockifier, violating the admission invariant.

### Finding Description
In `convert_rpc_tx_to_internal` (lines 348–360 of `transaction_converter.rs`), after calling `class_manager_client.add_class()`, the converter receives `ClassHashes { executable_class_hash_v2 }` — always the Blake2 hash. It then checks:

```rust
if tx.compiled_class_hash != executable_class_hash_v2 {
    return Err(ValidateCompiledClassHashError::CompiledClassHashMismatch { ... });
}
```

This check is unconditional — it does not consult `block_casm_hash_v1_declares` or any versioned constant. A user who submits `compiled_class_hash = casm.hash(&HashVersion::V1)` (Poseidon) will always fail here, because `V1_hash ≠ V2_hash` for any non-trivial CASM.

In contrast, the blockifier's `DeclareTransaction::run_execute` only calls `check_compile_class_hash_v2_declaration()` when `block_casm_hash_v1_declares && version >= V3`. When `block_casm_hash_v1_declares=false` (all versions ≤ 0.14.0), the blockifier proceeds directly to `try_declare` with the user-supplied hash, accepting it.

The TODO comment at lines 351–352 of `transaction_converter.rs` explicitly acknowledges this open question: `"Ensure that we do not want to allow declare with compiled class hash v1."` This confirms the check was added without resolving the invariant alignment.

### Impact Explanation
A user on a network running versioned constants with `block_casm_hash_v1_declares=false` (Starknet ≤ 0.14.0) who submits a DeclareV3 transaction with a V1 (Poseidon) compiled_class_hash will be rejected at the gateway converter stage with `InvalidCompiledClassHash`, even though the blockifier would accept and execute the transaction. The transaction is permanently blocked from sequencing despite being blockifier-valid. This is a concrete admission invariant violation: the gateway rejects a valid declare V3 transaction before sequencing.

### Likelihood Explanation
The scenario requires a user to deliberately compute and submit a V1 hash. Standard tooling always produces V2 hashes, so this does not affect typical users. However, any user or tool that computes the Poseidon hash of the CASM (the original algorithm) and submits it on a pre-0.14.1 network will be incorrectly rejected. The mismatch is structural and deterministic — it will always fire for any V1 hash on any network version where `block_casm_hash_v1_declares=false`.

### Recommendation
The converter check at lines 353–359 of `transaction_converter.rs` should be made consistent with the blockifier's `block_casm_hash_v1_declares` flag. Either:
1. When `block_casm_hash_v1_declares=false`, also accept `tx.compiled_class_hash == casm.hash(&HashVersion::V1)` in the converter; or
2. Resolve the TODO and document that V1 hashes are intentionally rejected at the gateway regardless of the blockifier flag, and update the invariant documentation accordingly.

The `class_manager_client.add_class()` should also be extended to return both `executable_class_hash_v1` and `executable_class_hash_v2` so the converter can perform a version-aware comparison.

    pub fn check_compile_class_hash_v2_declaration(&self) -> Result<(), StarknetApiError> {
        let compiled_class = &self.class_info.contract_class;
        let compiled_class_hash_v2 = match &compiled_class {
            ContractClass::V0(_) => return Ok(()),
            ContractClass::V1((casm, _)) => casm.hash(&HashVersion::V2),
        };
        let compiled_class_hash = self.compiled_class_hash();
        if compiled_class_hash_v2 != compiled_class_hash {
            let err_var = CasmHashMismatch {
                hash: self.class_hash(),
                actual: compiled_class_hash,
                expected: compiled_class_hash_v2,
            };
            return Err(StarknetApiError::DeclareTransactionCasmHashMissMatch(Box::new(err_var)));
        }
        Ok(())
    }
