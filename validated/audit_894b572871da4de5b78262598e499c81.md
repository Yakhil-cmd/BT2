### Title
Unvalidated Participant Contributions in CKD Coordinator Enable Undetected Corruption of Confidential Key Derivation Output - (File: src/confidential_key_derivation/protocol.rs)

### Summary
The CKD coordinator in `do_ckd_coordinator` receives `(big_y, big_c)` elliptic-curve point pairs from every participant and accumulates them with no proof-of-correctness check. Any single malicious participant can substitute arbitrary curve points, silently biasing the aggregated output so that the coordinator returns a wrong confidential derived key. Because no validation occurs, honest parties have no way to detect the corruption.

### Finding Description
**Root cause – `do_ckd_coordinator`, lines 50–55 of `src/confidential_key_derivation/protocol.rs`:**

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
Ok(Some(ckd_output))
```

Each honest participant `i` is supposed to send:
- `norm_big_y_i = λ_i · y_i · G`
- `norm_big_c_i = λ_i · (x_i · H(pk ‖ app_id) + y_i · app_pk)`

The coordinator blindly adds every received pair. There is no check that:
1. The received points are not the identity element.
2. The received `big_c` is consistent with the participant's public key share (i.e., no zero-knowledge proof that `big_c` was formed using the participant's actual secret share `x_i`).

**Exploit path:**
A malicious participant `m` sends `big_y = 0_G1` (identity) and `big_c = 0_G1` (identity) instead of their legitimate contribution. The coordinator computes:

```
big_y_total = Σ_{j ≠ m} λ_j · y_j · G
big_c_total = Σ_{j ≠ m} λ_j · (x_j · H(pk ‖ app_id) + y_j · app_pk)
```

The unmask operation then yields:

```
unmask(app_sk) = big_c_total − app_sk · big_y_total
               = Σ_{j ≠ m} λ_j · x_j · H(pk ‖ app_id)
```

This is **not** `msk · H(pk ‖ app_id)` (the correct confidential key), because the malicious participant's Lagrange-weighted share `λ_m · x_m · H(pk ‖ app_id)` is absent. The coordinator returns this corrupted value as `Some(ckd_output)` with no error, and every honest caller accepts it as the legitimate derived secret.

Alternatively, the attacker can send any non-zero crafted points to steer the output toward a value of their choosing (within the constraint that they cannot know the other participants' randomness `y_j`), enabling targeted corruption rather than mere zeroing.

### Impact Explanation
**High.** This directly matches the allowed impact: *"Corruption of CKD outputs so honest parties accept inconsistent … unusable cryptographic outputs."* The coordinator produces and returns a structurally valid `CKDOutput` whose embedded confidential key is wrong. Any downstream consumer (TEE application, key-derivation pipeline) silently operates on an incorrect secret. Because the protocol emits no error and no mismatch signal, the corruption is permanent for that session and undetectable without an out-of-band oracle.

### Likelihood Explanation
**Medium.** The attacker needs only to be a legitimate participant in the CKD session — no privileged role, no leaked key material, and no break of any cryptographic primitive is required. The malicious participant simply deviates from the protocol by sending identity points or other crafted values. This is a reachable, low-barrier deviation for any participant who wishes to disrupt or bias the derived key.

### Recommendation
Require each participant to accompany their `(big_y, big_c)` contribution with a zero-knowledge proof of correct formation — specifically, a proof that `big_c` was computed as `λ_i · (x_i · H(pk ‖ app_id) + y_i · app_pk)` for the participant's committed public key share `λ_i · x_i · G`. A Chaum–Pedersen / dlog-equality proof (the same `dlogeq` primitive already used in the OT-based triple generation at `src/ecdsa/ot_based_ecdsa/triples/generation.rs`) is sufficient. Additionally, add a guard in `do_ckd_coordinator` rejecting any contribution whose `big_y` or `big_c` is the identity element:

```rust
for (from, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
+   if participant_output.big_y().is_identity().into()
+       || participant_output.big_c().is_identity().into()
+   {
+       return Err(ProtocolError::IdentityElement);
+   }
+   // verify ZK proof of correct formation here
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
```

### Proof of Concept
1. Instantiate a CKD session with participants `{P1, P2, P3}`, coordinator `P1`, and a known `app_id` / `app_pk`.
2. `P2` (malicious) overrides its `compute_signature_share` result and sends `(0_G1, 0_G1)` to the coordinator instead of its legitimate `(norm_big_y, norm_big_c)`.
3. `P3` sends its legitimate contribution.
4. The coordinator accumulates: `big_y_total = λ_3 · y_3 · G`, `big_c_total = λ_3 · (x_3 · H(pk ‖ app_id) + y_3 · app_pk)`.
5. `unmask(app_sk)` returns `λ_3 · x_3 · H(pk ‖ app_id)`, which differs from the correct `msk · H(pk ‖ app_id) = (λ_1·x_1 + λ_2·x_2 + λ_3·x_3) · H(pk ‖ app_id)`.
6. The coordinator returns `Some(ckd_output)` with no error; honest parties accept the wrong key. [1](#0-0) [2](#0-1)

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
