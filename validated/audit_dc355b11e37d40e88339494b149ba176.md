### Title
`max_nonce_for_validation_skip` Config Field Defined but Never Enforced in `skip_stateful_validations` — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

`StatefulTransactionValidatorConfig` declares `max_nonce_for_validation_skip: Nonce` as the operator-controlled threshold governing when the gateway may skip the account's `__validate__` entry point for an invoke transaction. The field is serialized, present in every production config file, and documented as "Maximum nonce for which the validation is skipped." However, the function that actually decides whether to skip validation, `skip_stateful_validations`, is a free function that never receives or reads this field; it hardcodes `Nonce(Felt::ONE)` unconditionally. The config field is therefore inert: any operator change to it has zero effect on runtime behavior.

### Finding Description

`StatefulTransactionValidatorConfig` at line 283 declares:

```rust
pub max_nonce_for_validation_skip: Nonce,
```

with a default of `Nonce(Felt::ONE)` at line 295, and it is serialized in the config dump at lines 313–317. [1](#0-0) 

The gateway's `run_pre_validation_checks` calls the free function `skip_stateful_validations` without passing the config value: [2](#0-1) 

`skip_stateful_validations` hardcodes `Nonce(Felt::ONE)` directly: [3](#0-2) 

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` sets `validate: !skip_validate = false`, so the blockifier's `StatefulValidator` skips the `__validate__` entry point entirely for that transaction: [4](#0-3) [5](#0-4) 

### Impact Explanation

The `__validate__` entry point is the account contract's signature-verification gate. Skipping it at the gateway means an invoke transaction with an invalid or absent signature can pass gateway admission and enter the mempool.

The skip is intentional for the deploy-account + invoke UX flow (nonce=1, account not yet on-chain). The config field `max_nonce_for_validation_skip` exists precisely so operators can tighten or disable this bypass. Because the field is never read, an operator who sets `max_nonce_for_validation_skip = 0x0` to disable the skip entirely — or to any value other than `0x1` — gets no change in behavior. The gateway continues to skip `__validate__` for every nonce-1 invoke from an undeployed account regardless of the configured value.

Concretely: an attacker who has any deploy-account transaction in the mempool for address `A` (satisfying the `account_tx_in_pool_or_recent_block` check) can submit an invoke transaction for `A` with nonce=1 and a completely invalid signature. The gateway will skip `__validate__`, accept the transaction, and forward it to the mempool — matching the **High** impact criterion "Mempool/gateway/RPC admission accepts invalid transactions before sequencing."

### Likelihood Explanation

The default value of `max_nonce_for_validation_skip` (`0x1`) happens to match the hardcoded value, so the bug is invisible in default deployments. It becomes exploitable the moment an operator changes the field — a reasonable action given the field is documented and present in every production config. The production config files already show the field set to `"0x1"`, confirming operators are expected to tune it. [6](#0-5) 

### Recommendation

Pass `self.config.max_nonce_for_validation_skip` into `skip_stateful_validations` (or make it a method on `StatefulTransactionValidator`) and replace the hardcoded `Nonce(Felt::ONE)` with the config value:

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    max_nonce_for_validation_skip: Nonce,   // ← add parameter
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        if tx.nonce() <= max_nonce_for_validation_skip   // ← use config
            && account_nonce == Nonce(Felt::ZERO)
        {
            // existing mempool check …
        }
    }
    Ok(false)
}
```

Update `run_pre_validation_checks` to pass `self.config.max_nonce_for_validation_skip`.

### Proof of Concept

1. Deploy a gateway with `max_nonce_for_validation_skip = 0x0` (operator intends to disable the skip).
2. Submit a `DeployAccount` transaction for address `A` — it enters the mempool.
3. Submit an `Invoke` transaction for `A` with `nonce = 1` and a garbage signature.
4. `skip_stateful_validations` evaluates `tx.nonce() == Nonce(Felt::ONE)` → `true`; `account_nonce == Nonce(Felt::ZERO)` → `true`; `account_tx_in_pool_or_recent_block(A)` → `true` (step 2).
5. Returns `true` → `validate` flag is `false` → `__validate__` is skipped.
6. The transaction is forwarded to the mempool with no signature check, despite the operator having set `max_nonce_for_validation_skip = 0x0` to prevent exactly this. [7](#0-6) [8](#0-7)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
