### Title
Lax Event Type Validation Allows Arbitrary String Injection into gRPC Error Messages - (`access/legacy/handler.go`, `engine/common/rpc/convert/validate.go`)

### Summary

The Flow Access API's legacy gRPC handler passes the caller-supplied `eventType` string directly to the backend with zero validation. The modern handler applies only a non-empty check via `convert.EventType()`. In both paths the raw, attacker-controlled string is embedded verbatim into gRPC error messages returned to callers. This is the direct analog of the Rocketpool "minimum length of 4" weakness: a trivially weak gate that lets arbitrary content — including control characters, newlines, and falsified field text — flow through to every consumer of the API response.

---

### Finding Description

**Path 1 — Legacy handler (zero validation)**

`access/legacy/handler.go` `GetEventsForHeightRange` and `GetEventsForBlockIDs` extract the event type with no check at all:

```go
eventType := req.GetType()   // line 319 / 338 — raw, unvalidated
results, err := h.api.GetEventsForHeightRange(ctx, eventType, ...)
``` [1](#0-0) [2](#0-1) 

**Path 2 — Modern handler (non-empty check only)**

`engine/access/rpc/handler.go` calls `convert.EventType()`, which only rejects the empty string:

```go
func EventType(eventType string) (string, error) {
    if len(strings.TrimSpace(eventType)) == 0 {
        return "", status.Error(codes.InvalidArgument, "invalid event type")
    }
    return eventType, nil   // any non-empty string passes
}
``` [3](#0-2) [4](#0-3) 

**Error-message reflection**

The backend's format check (`events.ValidateEvent`) constructs the rejection error by embedding the raw input:

```go
// model/events/parse.go
return nil, fmt.Errorf("invalid event type: %s", eventType)
``` [5](#0-4) 

The backend then wraps and returns this to the caller:

```go
return nil, status.Errorf(codes.InvalidArgument, "invalid event type: %v", err)
``` [6](#0-5) 

The full gRPC error message delivered to the caller therefore contains the verbatim attacker-supplied string.

---

### Impact Explanation

Any unprivileged caller of the public Access API can supply an `eventType` value such as:

```
"foo\r\nStatus: Trusted\r\nBalance: 1000 FLOW"
```

This string is reflected unchanged in the gRPC `InvalidArgument` error message. Downstream consumers that display this message in a terminal (CLI tools, node operators' dashboards) are exposed to CRLF/control-character injection that can overwrite displayed fields — exactly the "falsified information" scenario described in the Rocketpool report. Consumers that forward the message to a web UI without HTML-escaping are exposed to stored-XSS-style injection. Consumers that write it to structured logs without sanitization suffer log-forging.

The legacy handler (`access/legacy/handler.go`) is the most exposed path because it skips even the trivial non-empty check, making the raw string available to the backend and its error-formatting code with no gate whatsoever.

---

### Likelihood Explanation

The Access API is the primary public-facing interface of every Flow Access node and Observer node. No authentication or staking is required to call `GetEventsForHeightRange` or `GetEventsForBlockIDs`. The legacy handler is compiled into production binaries and reachable over the standard gRPC port. Crafting a malicious `eventType` requires only a standard gRPC client and knowledge of the protobuf schema, both of which are publicly documented.

---

### Recommendation

1. **Enforce format at the handler layer.** Replace the non-empty check in `convert.EventType()` with a call to `events.ValidateEvent()` (or at minimum the regex already used by the REST parser in `engine/access/rest/common/parser/event_type.go`). Apply the same check in the legacy handler before forwarding to the backend. [7](#0-6) 

2. **Sanitize before embedding in error messages.** In `model/events/parse.go` and the backend error paths, strip or percent-encode non-printable and control characters from the `eventType` string before including it in `fmt.Errorf` / `status.Errorf` calls.

3. **Audit all `fmt.Sprintf` / `status.Errorf` call sites** that embed caller-supplied strings and apply the same sanitization.

---

### Proof of Concept

```python
import grpc
from flow.access import access_pb2, access_pb2_grpc

channel = grpc.insecure_channel("access.mainnet.nodes.onflow.org:9000")
stub = access_pb2_grpc.AccessAPIStub(channel)

# Inject CRLF + falsified fields into the error message
malicious_type = "foo\r\nStatus: Trusted\r\nBalance: 9999 FLOW"

try:
    stub.GetEventsForHeightRange(access_pb2.GetEventsForHeightRangeRequest(
        type=malicious_type,
        start_height=1,
        end_height=2,
    ))
except grpc.RpcError as e:
    # e.details() contains the verbatim injected string
    print(repr(e.details()))
    # → 'invalid event type: invalid event type: foo\r\nStatus: Trusted\r\nBalance: 9999 FLOW'
```

A terminal application printing `e.details()` without sanitization will have its display manipulated by the embedded `\r\n` sequences. The legacy handler path (`access/legacy/handler.go`) reaches the same error sink with no prior filtering.
