### Title
`skip_stateful_validations` Bypasses `__validate__` Signature Check, Admitting Unsigned Invoke Transactions for Any Account with a Pending Deploy-Account — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator skips the `__validate__` entry point for any invoke transaction with `nonce=1` when the sender address has **any** transaction in the mempool or a recent block. Because the check (`account_tx_in_pool_or_recent_block`) does not verify that the pending transaction is a deploy-account from the same user, an unprivileged attacker can submit an invoke transaction with an arbitrary (invalid) signature for a victim's not-yet-deployed account address and have it admitted to the mempool without any signature verification.

---

### Finding Description

**Root cause — `skip_stateful_validations`** [1](#0-0) 

The function fires when three conditions hold simultaneously:

1. The incoming transaction is an `Invoke`.
2. `tx.nonce() == Nonce(Felt::ONE)` — the transaction carries nonce 1.
3. `account_nonce == Nonce(Felt::ZERO)` — the on-chain nonce for the sender is still 0 (account not yet deployed).

When all three hold, it calls `account_tx_in_pool_or_recent_block(tx.sender_address())`: [2](#0-1) 

This returns `true` if the address has **any** transaction in the pool or has appeared in any committed block. It does **not** distinguish between a deploy-account transaction submitted by the legitimate owner and an arbitrary transaction submitted by a third party.

When the function returns `true`, `run_validate_entry_point` is called with `skip_validate = true`: [3](#0-2) 

Inside `run_validate_entry_point`, `execution_flags.validate` is set to `false`: [4](#0-3) 

The blockifier's `StatefulValidator::perform_validations` then short-circuits before calling `__validate__`: [5](#0-4) 

The `__validate__` entry point — the only place the account contract verifies the transaction signature — is never executed. The `validate_by_mempool` call that precedes `skip_stateful_validations` only checks for duplicate tx hash and nonce range; it does not inspect the signature: [6](#0-5) 

**Broken invariant:** Every invoke transaction admitted to the mempool must have its signature verified by the account's `__validate__` entry point before admission.

**Analog to the external report:** The external bug allowed the same `AssignmentHook` to be called multiple times because there was no deduplication guard, and an inequality (`<` instead of `==`) allowed excess TKO to be drained. Here, the analogous flaw is that `account_tx_in_pool_or_recent_block` acts as an overly broad guard — it returns `true` for **any** account activity, not specifically a deploy-account from the legitimate owner — and the gateway unconditionally skips the signature-verification hook (`__validate__`) whenever that broad condition is satisfied.

---

### Impact Explanation

An attacker who observes a victim's deploy-account transaction in the mempool can immediately submit an invoke transaction with `nonce=1` from the victim's address carrying **any** calldata and **any** (invalid) signature. The gateway admits this transaction to the mempool without calling `__validate__`. When the batcher later executes the block, `__validate__` is called and fails; the victim's account is charged the validation fee. By submitting many such transactions the attacker can repeatedly drain the victim's fee-token balance — a griefing attack with asymmetric cost (attacker pays only submission overhead; victim pays validation fees).

This matches: **High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

---

### Likelihood Explanation

- Deploy-account transactions are publicly visible in the mempool.
- The attacker needs only to craft an invoke transaction with `sender_address = victim`, `nonce = 1`, arbitrary calldata, and an arbitrary signature — no knowledge of the victim's private key is required.
- The nonce-range check in `validate_nonce` passes because `1` is within `[0, max_allowed_nonce_gap]`: [7](#0-6) 

- No other gateway check inspects the signature before the transaction reaches the mempool.

---

### Recommendation

1. **Narrow the skip condition**: Before skipping `__validate__`, verify that the transaction in the mempool for the sender address is specifically a `DeployAccount` transaction, not an arbitrary invoke or declare. The mempool's `account_tx_in_pool_or_recent_block` should be replaced with a dedicated `has_pending_deploy_account(address)` query.

2. **Alternatively, perform lightweight signature pre-screening**: Even when skipping the full `__validate__` execution, perform a stateless ECDSA check on the transaction signature against the expected account class before admitting the transaction.

3. **Scope the skip to nonce=1 only when a deploy-account is confirmed pending**: The comment in `skip_stateful_validations` already states the intent — "either it has a deploy_account transaction" — but the implementation does not enforce this. Enforce it explicitly.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
