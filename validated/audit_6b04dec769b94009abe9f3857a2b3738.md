### Title
Unvalidated Participant Contributions in CKD Coordinator Allow Malicious Participant to Corrupt Derived Confidential Key — (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

In the Confidential Key Derivation (CKD) protocol, the coordinator aggregates group-element contributions (`big_y`, `big_c`) received from each participant without any proof of correctness. A single malicious participant can send arbitrary `G1` elements, causing the coordinator to produce a corrupted `CKDOutput` that honest parties accept as valid, permanently destroying the usability of the derived confidential key for that invocation.

---

### Finding Description

In `do_ckd_coordinator` (lines 50–55 of `src/confidential_key_derivation/protocol.rs`), the coordinator collects each participant's `CKDOutput` and unconditionally adds the two group elements into the running aggregate:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
``` [1](#0-0) 

Each honest participant is supposed to send:
- `norm_big_y_i = λ_i · y_i · G`
- `norm_big_c_i = λ_i · (x_i · H(pk, app_id) + y_i · app_pk)`

where `y_i` is a fresh random scalar, `x_i` is the participant's private signing share, and `λ_i` is the Lagrange coefficient. The coordinator sums these to obtain `(big_Y, big_C)`, and the application later recovers the confidential key as `big_C − app_sk · big_Y = msk · H(pk, app_id)`. [2](#0-1) 

There is **no zero-knowledge proof** attached to the participant's message, and no check that `big_c` and `big_y` are consistent with the participant's public key share or the hash point. The `CKDOutput` struct is a plain pair of `ElementG1` values with no cryptographic binding:

```rust
pub struct CKDOutput {
    big_y: ElementG1,
    big_c: ElementG1,
}
``` [3](#0-2) 

The codebase already contains a `dlogeq` proof module (`src/crypto/proofs/dlogeq.rs`) capable of proving exactly this kind of consistency, but it is not applied here. [4](#0-3) 

---

### Impact Explanation

A malicious participant sends `(big_y' = G1::identity(), big_c' = G1::identity())` or any arbitrary pair of `G1` points. The coordinator adds these into the aggregate without error. The resulting `CKDOutput` is structurally valid (it deserializes and passes all type checks) but cryptographically wrong. The application's `unmask` call:

```rust
pub fn unmask(&self, secret_scalar: Scalar) -> Signature {
    self.big_c - self.big_y * secret_scalar
}
``` [5](#0-4) 

produces a value that is not `msk · H(pk, app_id)`. The derived confidential key is permanently wrong for that protocol run. There is no mechanism for honest parties to detect the corruption after the fact, because the coordinator outputs a single aggregated `CKDOutput` with no per-participant accountability.

**Impact:** High — Corruption of CKD output so honest parties accept an unusable or attacker-biased cryptographic output.

---

### Likelihood Explanation

The attacker is any single participant in the CKD protocol — a role explicitly within the threat model of a threshold scheme. No special privilege beyond participation is required. The attack requires only that the malicious participant deviate from the protocol by sending arbitrary group elements, which is trivially achievable by any library caller who controls their own `Protocol` message handler. The protocol has no round where participants can detect or attribute the bad contribution.

**Likelihood:** Medium — requires one malicious participant; no cryptographic assumption is broken.

---

### Recommendation

Require each participant to attach a **proof of discrete-log equality** (dlogeq) demonstrating that `big_c` and `big_y` are correctly formed with respect to the same secret scalar `y_i` and the participant's public key share. Concretely, the participant should prove:

- `big_y = y_i · G` (dlog of `big_y` base `G`)
- `big_c − x_i · H(pk, app_id) = y_i · app_pk` (same `y_i` used in both)

The existing `dlogeq::prove` / `dlogeq::verify` infrastructure in `src/crypto/proofs/dlogeq.rs` is directly applicable. The coordinator should verify each participant's proof before adding their contribution to the aggregate, and abort with `ProtocolError::MaliciousParticipant` on failure. [4](#0-3) 

---

### Proof of Concept

1. Honest participants `p_1, p_2` and malicious participant `p_3` run `ckd(...)`.
2. `p_3` intercepts the `do_ckd_participant` path and, instead of computing the correct `(norm_big_y, norm_big_c)`, sends `(ElementG1::identity(), ElementG1::identity())` to the coordinator via `chan.send_private(waitpoint, coordinator, &(identity, identity))`.
3. The coordinator's loop at lines 50–55 adds the identity elements without error.
4. The coordinator outputs `CKDOutput { big_y: big_Y - λ_3·y_3·G, big_c: big_C - λ_3·(x_3·H+y_3·app_pk) }`.
5. The application calls `ckd_output.unmask(app_sk)` and obtains a value ≠ `msk · H(pk, app_id)`.
6. The derived confidential key is permanently wrong; no honest party can detect which participant caused the corruption. [6](#0-5)

async fn do_ckd_coordinator(
    mut chan: SharedChannel,
    participants: ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,

    let (mut norm_big_y, mut norm_big_c) =
        compute_signature_share(&participants, me, key_pair, app_id, app_pk, rng)?;

    // Receive everyone's inputs and add them together
    let waitpoint = chan.next_waitpoint();

    for (_, participant_output) in
        recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?

        norm_big_c += participant_output.big_c();
    }
    let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
```

/// Verify that a proof attesting to the validity of some statement.
///
/// We use a transcript in order to verify the Fiat-Shamir transformation.
pub fn verify<C: Ciphersuite>(
    transcript: &mut Transcript,
    statement: Statement<'_, C>,
    proof: &Proof<C>,
) -> Result<bool, ProtocolError>
where
    Element<C>: ConstantTimeEq,
{
    if statement.generator1.ct_eq(&C::Group::identity()).into() {
        return Err(ProtocolError::IdentityElement);
    }

    transcript.message(NEAR_DLOGEQ_STATEMENT_LABEL, &statement.encode()?);

    let (phi0, phi1) = statement.phi(&proof.s.0);
    let big_k0 = phi0 - *statement.public0 * proof.e.0;
    let big_k1 = phi1 - *statement.public1 * proof.e.0;

    let enc = encode_two_points::<C>(&big_k0, &big_k1)?;

    transcript.message(NEAR_DLOGEQ_COMMITMENT_LABEL, &enc);
    let mut rng = transcript.challenge_then_build_rng(NEAR_DLOGEQ_CHALLENGE_LABEL);
    let e = frost_core::random_nonzero::<C, _>(&mut rng);

    Ok(e == proof.e.0)
}
