### Title
COA Ownership Proof Signature Replay Across Multiple COAs Owned by the Same Cadence Account - (File: `fvm/evm/stdlib/contract.cdc`)

### Summary

`EVM.validateCOAOwnershipProof` does not enforce that `signedData` includes the target `evmAddress`. A signature produced for COA-A can be replayed verbatim against COA-B when both are owned by the same Cadence account, because the function only checks that the signatures are valid over `signedData` and that the COA at the given `path` matches `evmAddress` — it never checks that `signedData` itself commits to `evmAddress`. The vulnerability is reachable by any unprivileged EVM caller via the ERC-1271 `isValidSignature` entry point on the COA contract.

### Finding Description

`EVM.validateCOAOwnershipProof` performs two independent checks:

1. Cryptographic: the provided `signatures` are valid over `signedData` for the Cadence account at `address`.
2. Structural: the COA resource at `path` has an EVM address equal to `evmAddress`. [1](#0-0) 

The function never verifies that `signedData` encodes `evmAddress`. The code itself documents this gap: [2](#0-1) 

The ERC-1271 entry point in `coa.sol` calls the Cadence Arch precompile as:

```solidity
cadenceArch.staticcall(
    abi.encodeWithSignature(
        "verifyCOAOwnershipProof(address,bytes32,bytes)",
        address(this),   // ← always the calling COA's EVM address
        _hash,           // ← becomes SignedData
        _sig             // ← encoded COAOwnershipProof (Address, CapabilityPath, KeyIndices, Signatures)
    )
);
``` [3](#0-2) 

`DecodeABIEncodedProof` extracts `caller` (= `address(this)`) as `EVMAddress` and `hash` as `SignedData`: [4](#0-3) 

This is then forwarded to `validateCOAOwnershipProof` via `coaOwnershipProofValidator`: [5](#0-4) 

**Attack path:**

1. Alice owns a Cadence account with two COAs: COA-A at `/public/coaA` and COA-B at `/public/coaB`.
2. An EVM dApp asks Alice to sign hash `H` to authorize an action on COA-A. Alice signs `H` without embedding COA-A's EVM address in `H`.
3. The attacker (who observed Alice's signature) constructs a `COAOwnershipProof` with `CapabilityPath = "coaB"` and Alice's same `KeyIndices` and `Signatures`.
4. The attacker calls `COA-B.isValidSignature(H, encodedProof)`.
5. The EVM sets `EVMAddress = address(COA-B)` and `SignedData = H`.
6. `validateCOAOwnershipProof(Alice, /public/coaB, H, keyIndices, sigs, COA-B_addr)` passes both checks: signatures over `H` are valid for Alice ✓, COA at `/public/coaB` has EVM address = COA-B ✓.
7. `isValidSignature` returns `0x1626ba7e` (valid) — Alice's signature for COA-A is accepted as authorization for COA-B.

The `COAOwnershipProof` struct that the attacker controls includes `CapabilityPath`, which is the only field that determines which COA is checked: [6](#0-5) 

### Impact Explanation

Any EVM contract that uses ERC-1271 (`isValidSignature`) to gate privileged actions — NFT marketplace approvals, DeFi permit-style authorizations, governance votes — can be tricked into accepting a signature intended for COA-A as valid authorization for COA-B. Because `validateCOAOwnershipProof` is also `access(all)` in Cadence, the same replay is possible in any Cadence-based off-chain authentication flow that calls the function directly. The result is unauthorized EVM-side authorization: an attacker can impersonate Alice's COA-B using a signature Alice only ever produced for COA-A, potentially draining assets or executing privileged EVM calls on COA-B's behalf.

### Likelihood Explanation

The preconditions are: (a) the victim Cadence account holds more than one COA, and (b) the victim has signed arbitrary data for one COA without binding the EVM address into the signed payload. Condition (b) is the default behavior — nothing in the protocol or wallet UX forces `signedData` to include `evmAddress`. Condition (a) is currently uncommon ("Flow wallets historically create at most one COA per account," per the inline comment), but the protocol imposes no limit on COA count per account, and multi-COA patterns are expected to grow as the EVM ecosystem matures. The attacker needs only to observe a prior valid signature (e.g., from a public EVM transaction log) and submit a crafted proof — no privileged access is required.

### Recommendation

Enforce inside `validateCOAOwnershipProof` that `signedData` contains the 20-byte `evmAddress` as a prefix or suffix before accepting the proof as valid. Alternatively, hash `evmAddress` into the domain separation tag used by `keyList.verify`. This mirrors the standard EIP-712 mitigation of binding the signed message to the specific contract address and context, preventing cross-COA replay without breaking the existing signature format for single-COA accounts (since their only COA address would simply need to be included in the data they sign).

### Proof of Concept

```
// Attacker-controlled Cadence script (no privileges required)
import EVM from <EVM_CONTRACT_ADDRESS>

access(all)
fun main(
    aliceCadenceAddress: Address,
    signedData: [UInt8],          // hash H that Alice signed for COA-A
    keyIndices: [UInt64],         // same key indices Alice used
    signatures: [[UInt8]],        // same signatures Alice produced
    coaBEvmAddress: [UInt8; 20]   // COA-B's EVM address
): EVM.ValidationResult {
    // path points to COA-B, not COA-A
    return EVM.validateCOAOwnershipProof(
        address: aliceCadenceAddress,
        path: /public/coaB,        // ← attacker substitutes COA-B's path
        signedData: signedData,    // ← unchanged from COA-A proof
        keyIndices: keyIndices,    // ← unchanged
        signatures: signatures,   // ← unchanged
        evmAddress: coaBEvmAddress // ← COA-B's address
    )
    // Returns isValid: true — Alice's COA-A signature accepted for COA-B
}
```

The same replay is triggered automatically via ERC-1271 when the attacker calls `COA-B.isValidSignature(H, encodedProofWithCoaBPath)` from any EVM contract, with `address(this)` supplying COA-B's EVM address and the attacker-controlled `_sig` supplying the `CapabilityPath` pointing to COA-B.

func DecodeABIEncodedProof(input []byte) (*types.COAOwnershipProofInContext, error) {
	index := 0
	caller, err := ReadAddress(input, index)
	index += FixedSizeUnitDataReadSize
	if err != nil {
		return nil, err
	}

	hash, err := ReadBytes32(input, index)
	index += Bytes32DataReadSize
	if err != nil {
		return nil, err
	}

	encodedProof, err := ReadBytes(input, index)
	if err != nil {
		return nil, err
	}

	return types.NewCOAOwnershipProofInContext(
		hash,
		types.Address(caller),
		encodedProof,
	)
}
