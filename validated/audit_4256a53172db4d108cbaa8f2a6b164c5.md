### Title
Unchecked `u64 → i64` Cast in `compute_gas_refund` Causes Incorrect `delta_gas` Calculation and Excess Gas Refund - (File: `basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`)

---

### Summary

In `compute_gas_refund`, the `delta_gas` calculation at line 72 performs two unchecked `u64 → i64` casts. When `native_used / native_per_gas` exceeds `i64::MAX`, the cast wraps to a large negative value, causing `delta_gas` to be incorrectly negative. The `if delta_gas > 0` guard then suppresses the required upward adjustment to `gas_used`, resulting in an excess gas refund paid to the transaction sender.

---

### Finding Description

In `compute_gas_refund`:

```rust
let delta_gas = if native_per_gas == 0 {
    0
} else {
    (native_used / native_per_gas) as i64 - (gas_used as i64)
};

if delta_gas > 0 {
    gas_used += delta_gas as u64;
}
``` [1](#0-0) 

Both `native_used` and `gas_used` are `u64`. The expression `(native_used / native_per_gas) as i64` is an unchecked narrowing cast. `i64::MAX = 9_223_372_036_854_775_807`, while `u64::MAX = 18_446_744_073_709_551_615`. Any value of `native_used / native_per_gas` above `i64::MAX` wraps to a large negative `i64` in two's-complement, making `delta_gas` negative regardless of the true arithmetic relationship.

The `full_native_limit` is computed as `gas_limit.saturating_mul(native_per_gas)`, so `native_used` can be as large as `gas_limit * native_per_gas`. [2](#0-1) 

Therefore `native_used / native_per_gas` can be as large as `gas_limit`. For L2 transactions, the only overflow guard on `gas_limit` is that `gas_limit * ERGS_PER_GAS` must not overflow `u64`. With `ERGS_PER_GAS = 256`, the maximum valid `gas_limit` is `u64::MAX / 256 ≈ 7.2 × 10^16`, which is approximately **8× larger** than `i64::MAX ≈ 9.2 × 10^15`. A transaction sender can therefore supply a `gas_limit` in the range `(i64::MAX, u64::MAX/256]` and trigger the wrap. [3](#0-2) 

When the wrap occurs, `delta_gas` is negative, the `if delta_gas > 0` branch is skipped, `gas_used` is not increased, and `total_gas_refund = gas_limit - gas_used` is larger than it should be. The excess refund is transferred back to the sender.

---

### Impact Explanation

The sender receives a larger ETH refund than their actual native resource consumption warrants. The operator is paid less than the true cost of processing the transaction. This is a direct, attacker-controlled fee underpayment: by choosing `gas_limit > i64::MAX` and crafting a transaction that consumes significant native resources, a sender can systematically extract excess refunds from the protocol on every transaction.

---

### Likelihood Explanation

Any unprivileged L2 or L1 transaction sender can set `gas_limit` to a value above `i64::MAX` (e.g., `10_000_000_000_000_000_000`). This value passes the `gas_limit * ERGS_PER_GAS <= u64::MAX` check. No special privilege, leaked key, or oracle manipulation is required. The path through `compute_gas_refund` is exercised on every transaction that reaches the refund phase. [4](#0-3) 

---

### Recommendation

Replace the unchecked casts with checked or saturating arithmetic. The correct fix mirrors the suggestion from the referenced audit: use `i64::try_from` (or equivalent) and handle the overflow case explicitly. For example:

```rust
let native_gas_equiv = native_used / native_per_gas;
let delta_gas: i64 = match (i64::try_from(native_gas_equiv), i64::try_from(gas_used)) {
    (Ok(n), Ok(g)) => n.saturating_sub(g),
    _ => {
        // native_gas_equiv or gas_used exceeds i64::MAX;
        // saturate: treat delta as the full difference clamped to i64::MAX
        i64::MAX
    }
};
```

Alternatively, keep the computation in `u64` and only convert after the comparison:

```rust
let native_gas_equiv = native_used / native_per_gas;
if native_gas_equiv > gas_used {
    gas_used = native_gas_equiv; // or gas_used += native_gas_equiv - gas_used
}
```

### Proof of Concept

1. Craft a transaction with `gas_limit = i64::MAX + 1 = 9_223_372_036_854_775_808` (valid: `9_223_372_036_854_775_808 * 256 < u64::MAX`).
2. Set `native_per_gas` to a small value (e.g., 1, achievable with a low `gas_price / native_price` ratio).
3. Execute a transaction that consumes nearly all native resources (`native_used ≈ gas_limit * native_per_gas`).
4. In `compute_gas_refund`:
   - `native_used / native_per_gas ≈ gas_limit = 9_223_372_036_854_775_808`
   - `(9_223_372_036_854_775_808u64) as i64 = -9_223_372_036_854_775_808` (wraps)
   - `delta_gas = -9_223_372_036_854_775_808 - (gas_used as i64)` → large negative
   - `if delta_gas > 0` → **false**, `gas_used` is not adjusted
5. `total_gas_refund = gas_limit - gas_used` is inflated; sender receives excess ETH refund. [5](#0-4)
