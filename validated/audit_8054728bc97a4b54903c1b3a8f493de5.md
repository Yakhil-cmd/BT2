### Title
Unprivileged Caller Can Permanently Deny Any Staker Their Block Rewards via `update_rewards(disable_rewards: true)` — (File: `src/staking/staking.cairo`)

---

### Summary

The public `update_rewards` function in the Staking contract accepts a caller-controlled `disable_rewards: bool` parameter. When called with `disable_rewards: true`, the function writes the current block number to `last_reward_block` — consuming the one-call-per-block slot — but then returns early without distributing any rewards. Because the slot is consumed before the early-return guard, any attacker can front-run the legitimate reward update for any staker, permanently destroying that staker's block rewards for that block.

---

### Finding Description

`update_rewards` is a public, permissionless function (gated only by the contract-not-paused and caller-not-zero checks in `general_prerequisites`):

```cairo
// src/staking/staking.cairo  lines 1448–1507
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();                          // only: not paused, caller != 0
    let current_block_number = starknet::get_block_number();
    assert!(
        current_block_number > self.last_reward_block.read(),
        "{}",
        Error::REWARDS_ALREADY_UPDATED,
    );
    ...
    // ← last_reward_block is written HERE, before the disable_rewards guard
    self.last_reward_block.write(current_block_number);

    if disable_rewards || self.is_pre_consensus() {
        return;                                            // ← exits with NO rewards distributed
    }
    // reward distribution only reached when disable_rewards == false
    ...
}
``` [1](#0-0) 

The ordering is the root cause:

1. `last_reward_block` is updated to `current_block_number` unconditionally.
2. Only *after* that write does the code check `disable_rewards`.
3. If `disable_rewards` is `true`, the function returns without distributing rewards.
4. The one-call-per-block invariant (`current_block_number > last_reward_block`) is now permanently satisfied for block N — no second call can succeed.

Because `disable_rewards` is a plain `bool` argument with no access-control check, any address can supply `true`.

---

### Impact Explanation

**High — Permanent theft / freezing of unclaimed yield.**

For every block in the consensus-rewards phase (V3), a staker is entitled to block rewards proportional to their stake. An attacker who calls `update_rewards(victim_staker, disable_rewards: true)` at block N:

- Permanently consumes the reward slot for block N.
- Ensures the staker receives zero rewards for that block.
- Cannot be undone: `last_reward_block` is a monotonically-advancing value; there is no recovery path for a missed block.

The attacker can repeat this every block, completely zeroing out a targeted staker's (and their delegators') consensus rewards indefinitely. This maps directly to **High: Theft of unclaimed yield / Permanent freezing of unclaimed yield**.

---

### Likelihood Explanation

**High.**

- The function is publicly callable with no role restriction.
- The only prerequisite is that the contract is not paused and the caller is non-zero.
- The attack requires a single transaction per block, costing only gas.
- On Starknet, transaction ordering within a block is deterministic and observable, making front-running straightforward.
- No capital, no leaked key, no privileged access is required.

---

### Recommendation

Move `last_reward_block.write(current_block_number)` to *after* the `disable_rewards` guard, so that a call with `disable_rewards: true` does not consume the per-block slot:

```cairo
// Update last block rewards only when rewards are actually being processed.
if disable_rewards || self.is_pre_consensus() {
    return;
}
self.last_reward_block.write(current_block_number);   // ← moved here
// ... reward distribution ...
```

Alternatively, restrict who may pass `disable_rewards: true` (e.g., only the staker themselves or a trusted operator role), or remove the parameter entirely and handle the disable logic through a separate privileged function.

---

### Proof of Concept

**Setup:** Staker A is active in the consensus-rewards phase with a non-zero STRK balance. Block N is the current block.

**Attack steps:**

1. Attacker (any address) calls:
   ```
   update_rewards(staker_A_address, disable_rewards: true)
   ```
   at block N.

2. Inside `update_rewards`:
   - `current_block_number (N) > last_reward_block` → assertion passes.
   - `last_reward_block` is written to N.
   - `disable_rewards == true` → function returns early; **no rewards distributed**.

3. The legitimate reward-update call (from the sequencer or any honest party) for block N now fails:
   ```
   assert!(current_block_number > self.last_reward_block.read())
   // N > N  →  false  →  REWARDS_ALREADY_UPDATED panic
   ```

4. Staker A permanently loses their block rewards for block N. The attacker repeats this every block to continuously deny rewards. [2](#0-1)
