### Title
Scheduled Transactions Executed in Unordered Sequence Due to Missing EventIndex Sort - (File: `fvm/blueprints/scheduled_callback.go`)

---

### Summary

The primary `ExecuteCallbacksTransactions` function used by the Execution Node to build the on-chain execution sequence of scheduled transactions does **not** sort `PendingExecution` events by `EventIndex` before constructing the transaction list. A sibling implementation for the Access Node (V0) correctly sorts by `EventIndex`. The unsorted path even contains a `// todo` comment acknowledging the gap. As a result, the execution order of scheduled transactions is determined by the arbitrary order in which the FVM returns events, not by their canonical emission order, directly mirroring the LIFO/wrong-order settlement bug in the reference report.

---

### Finding Description

`fvm/blueprints/scheduled_callback.go:ExecuteCallbacksTransactions` iterates over the raw `processEvents` slice without any ordering guarantee:

```go
for _, event := range processEvents {
    // todo make sure to check event index to ensure order is indeed correct
    // event.EventIndex
    if !IsPendingExecutionEvent(env, event) {
        continue
    }
    ...
    txs = append(txs, tx)
}
``` [1](#0-0) 

The `// todo` comment explicitly flags that `event.EventIndex` is not being used to enforce order.

In contrast, the V0 system-collection builder used by the Access Node **does** sort before iterating:

```go
slices.SortFunc(processEvents, func(a, b flow.Event) int {
    return int(a.EventIndex - b.EventIndex)
})
``` [2](#0-1) 

The Execution Node calls the **unsorted** `blueprints.ExecuteCallbacksTransactions` at the point where it constructs the actual scheduled-transaction execution queue for a block: [3](#0-2) 

The resulting `callbackTxs` slice is then fed directly into `queueSystemTransactions`, which enqueues them in the order they appear in the slice: [4](#0-3) 

---

### Impact Explanation

The `FlowTransactionScheduler.process()` Cadence transaction emits one `PendingExecution` event per ready scheduled transaction. The `EventIndex` field on each event encodes the canonical emission order within the transaction. If the FVM returns these events in any order other than ascending `EventIndex` (which is not guaranteed by the current code path), the Execution Node will execute the scheduled transactions in that non-canonical order.

For a user who has scheduled multiple interdependent transactions (e.g., transaction B depends on state written by transaction A), executing B before A causes B to fail or produce incorrect on-chain state. This is a direct analog of the reference report's scenario where a liquidation request is processed before the share-purchase request that funds it.

Because the system collection is part of every sealed block, any divergence in execution order between nodes (e.g., if one node receives events in a different order) also risks **execution result disagreement**, which can stall sealing.

---

### Likelihood Explanation

Any unprivileged user can schedule transactions via the `FlowTransactionScheduler` contract. The bug is triggered automatically every block in which two or more scheduled transactions become ready simultaneously. The `// todo` comment confirms the developers are aware the ordering is not enforced. The V0 access-node path already has the fix, making it straightforward to confirm the execution-node path is missing it.

---

### Recommendation

Apply the same `slices.SortFunc` by `EventIndex` that `system_collection_v0.go` already uses, inside `blueprints.ExecuteCallbacksTransactions` before the iteration loop:

```go
slices.SortFunc(processEvents, func(a, b flow.Event) int {
    return int(a.EventIndex - b.EventIndex)
})
``` [1](#0-0) 

Remove the `// todo` comment once the fix is applied. Add a unit test asserting that when multiple `PendingExecution` events are supplied out of `EventIndex` order, the returned transaction list is still in ascending `EventIndex` order.

---

### Proof of Concept

1. A user schedules two transactions, A (lower `EventIndex`) and B (higher `EventIndex`), where B depends on state written by A.
2. Both become ready in the same block; `FlowTransactionScheduler.process()` emits `PendingExecution(A)` at `EventIndex=0` and `PendingExecution(B)` at `EventIndex=1`.
3. The FVM returns the event list in reverse order: `[B, A]`.
4. `blueprints.ExecuteCallbacksTransactions` (no sort) produces `[txB, txA]`.
5. `queueSystemTransactions` enqueues `txB` first, then `txA`.
6. `txB` executes before `txA`; because the state written by A is absent, `txB` fails or produces incorrect results — exactly the wrong-order settlement described in the reference report. [5](#0-4) [6](#0-5) [7](#0-6)

func (b *builderV0) ExecuteCallbacksTransactions(chain flow.Chain, processEvents flow.EventsList) ([]*flow.TransactionBody, error) {
	txs := make([]*flow.TransactionBody, 0, len(processEvents))
	sc := systemcontracts.SystemContractsForChain(chain.ChainID())
	env := sc.AsTemplateEnv()
	script := b.prepareExecuteScheduledTransactionTemplate(env)

	slices.SortFunc(processEvents, func(a, b flow.Event) int {
		return int(a.EventIndex - b.EventIndex)
	})

	for _, event := range processEvents {
		// skip any fee events or other events that are not pending execution events
		if !blueprints.IsPendingExecutionEvent(env, event) {
			continue
		}

		id, effort, err := b.callbackArgsFromEvent(event)
		if err != nil {
			return nil, fmt.Errorf("failed to get callback args from event: %w", err)
		}

		tx, err := flow.NewTransactionBodyBuilder().
			AddAuthorizer(sc.FlowServiceAccount.Address).
			SetScript(script).
			AddArgument(id).
			SetComputeLimit(effort).
			Build()
		if err != nil {
			return nil, fmt.Errorf("failed to construct execute callback transactions: %w", err)
		}
		txs = append(txs, tx)
	}

	return txs, nil
}

func (e *blockComputer) executeProcessCallback(
	systemCtx fvm.Context,
	systemCollectionInfo collectionInfo,
	database *transactionCoordinator,
	blockSpan otelTrace.Span,
	txnIndex uint32,
	systemLogger zerolog.Logger,
) ([]*flow.TransactionBody, uint32, error) {
	callbackLogger := systemLogger.With().
		Bool("system_transaction", true).
		Logger()

	request := newTransactionRequest(
		systemCollectionInfo,
		systemCtx,
		callbackLogger,
		txnIndex,
		e.processCallbackTxn,
		ComputerTransactionTypeSystem,
		false)

	txnIndex++

	txn, err := e.executeTransactionInternal(blockSpan, database, request, 1)
	if err != nil {
		snapshotTime := logical.Time(0)
		if txn != nil {
			snapshotTime = txn.SnapshotTime()
		}

		return nil, 0, fmt.Errorf(
			"failed to execute system process transaction %v (%d@%d) for block %s at height %v: %w",
			request.txnIdStr,
			request.txnIndex,
			snapshotTime,
			request.blockIdStr,
			request.ctx.BlockHeader.Height,
			err)
	}

	if txn.Output().Err != nil {
		// if the process transaction fails we log the critical error but don't return an error
		// so that block execution continues and only the scheduled transactions halt
		callbackLogger.Error().
			Err(txn.Output().Err).
			Bool("critical_error", true).
			Uint64("height", request.ctx.BlockHeader.Height).
			Msg("system process transaction output error")

		return nil, txnIndex, nil
	}

	callbackTxs, err := blueprints.ExecuteCallbacksTransactions(e.vmCtx.Chain, txn.Output().Events)
	if err != nil {
		return nil, 0, err
	}

	if len(callbackTxs) > 0 {
		// calculate total computation limits for execute scheduled transaction transactions
		var totalExecuteComputationLimits uint64
		for _, tx := range callbackTxs {
			totalExecuteComputationLimits += tx.GasLimit
		}

		// report metrics for scheduled transactions executed
		e.metrics.ExecutionScheduledTransactionsExecuted(
			len(callbackTxs),
			txn.Output().ComputationUsed,
			totalExecuteComputationLimits,
		)
	}

	return callbackTxs, txnIndex, nil
