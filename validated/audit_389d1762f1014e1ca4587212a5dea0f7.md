### Title
Permanent Fund Freeze in wNEAR Unwrap Flow When NEAR Transfer to Recipient Fails - (File: engine/src/contract_methods/connector.rs)

### Summary
When a user unwraps wNEAR ERC-20 tokens via the `ExitToNear` precompile (`:unwrap` suffix), the ERC-20 tokens are burned in the EVM and `near_withdraw` is called on the wNEAR NEP-141 contract. If `near_withdraw` succeeds but the subsequent NEAR transfer to the user-specified recipient fails (e.g., recipient account does not exist on NEAR), the NEAR is permanently locked in Aurora's engine account with no recovery path. This is structurally identical to the H-04 pattern: funds are consumed/recorded in one layer but the actual asset transfer in a second layer can silently fail.

### Finding Description

The wNEAR unwrap flow is a two-step async promise chain:

**Step 1 — EVM execution + promise creation** (`engine-precompiles/src/native.rs`, `exit_erc20_token_to_near`):

When the ERC-20 `withdrawToNear` function burns the user's wNEAR ERC-20 tokens and calls the `ExitToNear` precompile with the `:unwrap` suffix, the precompile constructs a `PromiseArgs::Callback` with:
- **base**: `near_withdraw` on the wNEAR NEP-141 account (unwraps wNEAR → NEAR into Aurora's account)
- **callback**: `exit_to_near_precompile_callback` on the Aurora engine itself [1](#0-0) 

The ERC-20 tokens are burned in the EVM before any promise executes. The `transfer_near_args` carry the user-supplied `target_account_id` and `amount`. [2](#0-1) 

**Step 2 — Callback execution** (`engine/src/contract_methods/connector.rs`, `exit_to_near_precompile_callback`):

If `near_withdraw` succeeds, the callback creates a new NEAR `Transfer` batch promise to the recipient and returns it with `promise_return`. There is **no further callback** attached to this transfer promise to handle its failure: [3](#0-2) 

The `else if let Some(args) = args.refund` branch only executes when the **base** promise (`near_withdraw`) fails — it does not cover the case where `near_withdraw` succeeds but the subsequent NEAR transfer fails: [4](#0-3) 

**Root cause**: After `near_withdraw` succeeds, the NEAR is held in Aurora's NEAR account balance. The `Transfer` action to the user-specified `target_account_id` is a separate promise with no failure callback. If that transfer fails (recipient account does not exist, or was deleted), the NEAR remains in Aurora's account permanently. There is no admin method or EVM-accessible path to recover it. [5](#0-4) 

### Impact Explanation

**High — Permanent freezing of funds.**

The user's wNEAR ERC-20 tokens are burned in the EVM (irreversible). The wNEAR NEP-141 tokens are unwrapped to NEAR and deposited into Aurora's NEAR account (irreversible after `near_withdraw` succeeds). If the NEAR transfer to the recipient fails, the NEAR is permanently stranded in Aurora's NEAR account balance with no EVM-accessible or admin-accessible recovery mechanism. The user loses the full unwrapped amount.

### Likelihood Explanation

**Medium.**

The user-controlled `target_account_id` is parsed directly from the ERC-20 call input. A non-existent NEAR account ID (valid format, no on-chain account) will cause the `Transfer` action to fail. This can occur via:
- User typo in the recipient account name
- Race condition: the recipient account is deleted between EVM execution and NEAR promise execution (NEAR accounts can be deleted)
- Deliberate self-griefing (user burns their own tokens)

Any EVM user who calls `withdrawToNear` with the `:unwrap` suffix is exposed to this path.

### Recommendation

Attach a failure callback to the NEAR `Transfer` promise created inside `exit_to_near_precompile_callback`. If the transfer fails, the callback should re-mint the equivalent amount of wNEAR ERC-20 tokens to the original sender (analogous to the existing `refund_on_error` path used when `near_withdraw` itself fails). Alternatively, before creating the transfer promise, validate that the recipient account exists on NEAR (via a view call), though this adds latency and complexity.

### Proof of Concept

1. User holds 100 wNEAR ERC-20 tokens on Aurora.
2. User calls `withdrawToNear("nonexistent_account.near:unwrap", 100)` on the wNEAR ERC-20 contract.
3. EVM execution: 100 wNEAR ERC-20 tokens burned from user's EVM balance. `ExitToNear` precompile schedules a `PromiseArgs::Callback`:
   - base: `near_withdraw({"amount":"100"})` on wNEAR NEP-141
   - callback: `exit_to_near_precompile_callback` with `transfer_near = Some(TransferNearArgs { target_account_id: "nonexistent_account.near", amount: 100 })`
4. NEAR runtime executes `near_withdraw` → succeeds, 100 yoctoNEAR transferred to Aurora's account.
5. NEAR runtime executes `exit_to_near_precompile_callback` → promise result is `Successful`, so the `Transfer` batch promise to `nonexistent_account.near` is created and returned.
6. NEAR runtime executes the `Transfer` action → **fails** because `nonexistent_account.near` does not exist.
7. Result: 100 wNEAR ERC-20 tokens are permanently burned, 100 yoctoNEAR is permanently locked in Aurora's NEAR account. No refund, no recovery path. [6](#0-5) [7](#0-6)

#[named]
pub fn exit_to_near_precompile_callback<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        env.assert_private_call()?;

        // This function should only be called as the callback of
        // exactly one promise.
        if handler.promise_results_count() != 1 {
            return Err(errors::ERR_PROMISE_COUNT.into());
        }

        let args: ExitToNearPrecompileCallbackArgs = io.read_input_borsh()?;

        let maybe_result = if let Some(PromiseResult::Successful(_)) = handler.promise_result(0) {
            if let Some(args) = args.transfer_near {
                let action = PromiseAction::Transfer {
                    amount: Yocto::new(args.amount),
                };
                let promise = PromiseBatchAction {
                    target_account_id: args.target_account_id,
                    actions: vec![action],
                };

                // Safety: this call is safe because it comes from the exit to near precompile, not users.
                // The call is to transfer the unwrapped wNEAR tokens.
                let promise_id = handler.promise_create_batch(&promise);
                handler.promise_return(promise_id);
            }

            None
        } else if let Some(args) = args.refund {
            // Exit call failed; need to refund tokens
            let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;

            if !refund_result.status.is_ok() {
                return Err(errors::ERR_REFUND_FAILURE.into());
            }

            Some(refund_result)
        } else {
            None
        };

        Ok(maybe_result)
    })
}
