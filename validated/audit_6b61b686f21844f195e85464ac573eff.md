### Title
WalletConnect Pair Revocation Does Not Cancel In-Flight Confirmation Dialogs, Allowing Execution of Commands After Disconnect - (File: packages/gui/src/electron/utils/dispatchPairRequest.ts, packages/gui/src/electron/main.tsx)

### Summary
A TOCTOU (Time-of-Check-Time-of-Use) race condition in the WalletConnect command dispatch pipeline allows a dapp command (e.g., `chia_sendTransaction`, `chia_takeOffer`, `chia_nft_transfer_nft`) to execute against a wallet even after the user has revoked the pair. The pair authorization is checked before the confirmation dialog is shown, but is never re-validated after the user clicks "Allow." A user who disconnects a pair while a confirmation dialog is open — believing this cancels the pending request — can still inadvertently authorize the command by clicking "Allow" in the still-active dialog.

### Finding Description

The `dispatchPairRequest` function in the main process is the central authorization gate for all WalletConnect commands. Its flow is:

1. **CHECK**: `findPair(topic)` is called to verify the pair exists.
2. **CHECK**: The `confirm()` callback re-checks `findPair(topic)` before opening the dialog.
3. **ASYNC GAP**: `openReactDialog(...)` awaits user input — this can take seconds to minutes.
4. **NO RE-CHECK**: After `confirmResult.isAllowed === true`, `process(context)` is called immediately with no re-validation of pair status. [1](#0-0) [2](#0-1) 

In the `confirm` callback registered in `main.tsx`, the pair is checked a second time before the dialog opens, but there is no check after the user clicks "Allow": [3](#0-2) [4](#0-3) 

When the user disconnects a pair, `revokePair` is called via IPC, which synchronously removes the pair from the in-memory `pairStore`: [5](#0-4) 

However, the `DISPATCH_AS_PAIR` handler is already suspended at `await openReactDialog(...)`. After the dialog resolves, `process(context)` executes the wallet command using the stale `context` captured before revocation, with no re-check: [6](#0-5) 

The `disconnectPair` function in the renderer revokes the pair and disconnects WC sessions, but cannot cancel the already-open Electron dialog window or the suspended IPC handler: [7](#0-6) 

### Impact Explanation

After the user revokes a pair, the invariant "a disconnected pair cannot execute commands against the wallet" is violated for any command whose confirmation dialog was already open. Spend commands — `chia_wallet.send_transaction`, `chia_wallet.take_offer`, `chia_wallet.nft_transfer_nft`, `chia_wallet.cat_spend`, `chia_wallet.did_transfer_did`, `chia_wallet.spend_clawback_coins` — can all be dispatched through this path: [8](#0-7) 

A user who disconnects a pair while a confirmation dialog is open may believe the pending request is cancelled. If they then click "Allow" (e.g., thinking the dialog is now inert, or because they intended to approve before disconnecting), the transaction executes against their wallet despite the pair being revoked. This constitutes a bypass of WalletConnect approval authority with direct asset-loss impact.

### Likelihood Explanation

The scenario requires: (1) a dapp sends a spend command, (2) the user opens the disconnect flow in a separate UI location while the dialog is open, (3) the user clicks "Allow" in the dialog. Steps 1–2 are realistic — a user suspicious of a dapp may attempt to disconnect it while reviewing the dialog. Step 3 is the confused action enabled by the missing invariant. The window is bounded by the dialog's lifetime (no timeout is enforced by `openReactDialog`). [9](#0-8) 

### Recommendation

After `confirmResult.isAllowed === true` is received inside the `confirm` callback in `main.tsx`, re-validate the pair with `findPair(topic)` before returning `true`. If the pair is no longer present, throw a `WcError` with `USER_REJECTED`. This closes the TOCTOU window:

```typescript
if (confirmResult && confirmResult.isAllowed === true) {
  // Re-validate pair after async dialog — pair may have been revoked while dialog was open
  const pairAfterConfirm = findPair(topic);
  if (!pairAfterConfirm) {
    throw new WcError(`Pair was revoked during confirmation`, WcErrorCode.USER_REJECTED);
  }
  if (confirmResult.rememberBypass && dappCommandSchema.allowConfirmationBypass === true) {
    addBypassCommand(topic, command);
  }
  return true;
}
``` [10](#0-9) 

### Proof of Concept

1. Malicious dapp sends `chia_sendTransaction` (or any spend command) via WalletConnect to the paired wallet.
2. The confirmation dialog opens in a separate Electron `BrowserWindow`.
3. User navigates to Settings → Integration (or WalletConnect Connections) and clicks **Disconnect** on the dapp's pair.
4. `revokePair` IPC fires → `removePair(topic)` removes the pair from `pairStore`. The pair is now gone.
5. The confirmation dialog is still open (it is a separate `BrowserWindow` with no awareness of the revocation).
6. User clicks **Allow** in the dialog, believing the pair is disconnected and the action is now harmless.
7. `confirmResult.isAllowed === true` → `process(context)` fires → `sendCommand('send_transaction', 'chia_wallet', parsedParams)` executes → funds are transferred.
8. The transaction completes against a wallet whose authorizing pair was revoked before execution. [11](#0-10) [12](#0-11)

async function disconnectPair(client: Client, topic: string) {
  try {
    // revoke pair from main process
    await window.permissionsAPI.revokePair(topic);

    // get all sessions for the pairing
    const sessions = client.session.getAll();
    const relatedSessions = sessions.filter((s) => s.pairingTopic === topic);

    // disconnect all related sessions
    await Promise.all(
      relatedSessions.map((session) =>
        client.disconnect({
          topic: session.topic,
          reason: getSdkError('USER_DISCONNECTED'),
        }),
      ),
    );

    // disconnect the pairing
    await client.core.pairing.disconnect({ topic });
  } catch (e) {
    processError(e as Error);
  }
}
