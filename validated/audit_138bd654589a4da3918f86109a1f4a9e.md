### Title
Missing Participant Share Verification in CKD Protocol Allows Malicious Participant to Corrupt Confidential Key Output - (File: src/confidential_key_derivation/protocol.rs)

### Summary
The CKD coordinator aggregates participant shares without any cryptographic verification. A single malicious participant can send arbitrary group elements in place of their correct share, silently corrupting the confidential key derivation output. No detection mechanism exists.

### Finding Description
In `do_ckd_coordinator`, the coordinator receives `(norm_big_y, norm_big_c)` from every participant via `recv_from_others` and unconditionally adds them to the running sum: [1](#0-0) 

There is no proof, commitment, or consistency check on the received values. The correct computation each honest participant performs is: [2](#0-1) 

A malicious participant is free to deviate and send any pair of `ElementG1` values. The coordinator has no way to distinguish a correct share from a fabricated one.

Contrast this with the verification discipline applied elsewhere in the same codebase:
- DKG verifies proofs of knowledge and commitment hashes before accepting any share. [3](#0-2) 
- Robust ECDSA presign performs exponent-interpolation consistency checks on every received value. [4](#0-3) 
- Both signing protocols verify the final assembled signature before returning it. [5](#0-4) 

The CKD protocol has no analogous enforcement step.

### Impact Explanation
The CKD output `(Y, C)` is an ElGamal ciphertext:

```
Y = Σ λᵢ · yᵢ · G
C = msk · H(pk ‖ app_id) + Y · app_sk
```

A malicious participant `pₘ` substitutes `(ΔY, ΔC)` for their correct share. The coordinator computes:

```
Y' = Y_honest + ΔY
C' = C_honest + ΔC
```

Decryption by the app yields:

```
C' − app_sk · Y' = msk · H(pk ‖ app_id) + ΔC − app_sk · ΔY
```

This is not the intended confidential key. The app silently receives a corrupted, unusable derived secret. This matches **High: Corruption of CKD outputs so honest parties accept unusable cryptographic outputs**.

### Likelihood Explanation
The CKD protocol is an n-of-n aggregation: every participant's contribution is summed with equal weight via Lagrange coefficients. [6](#0-5) 

Any single participant in the session can execute this attack. No special knowledge is required — the attacker does not need `app_sk` or any other participant's private material. The attack is a one-round substitution of arbitrary group elements and is completely undetectable by the coordinator or any honest participant.

### Recommendation
Add cryptographic enforcement of share correctness before aggregation. Concrete options:

1. **Pairing-based verification**: Store per-participant public key shares `Xᵢ = xᵢ · G₂` in the DKG output. For each received `big_s`, verify `e(big_s, G₂) = e(H(pk ‖ app_id), Xᵢ)` before accepting the share.
2. **NIZK proof (Chaum-Pedersen)**: Each participant attaches a zero-knowledge proof that `big_s` is a scalar multiple of `H(pk ‖ app_id)` consistent with their public key share.
3. **Commit-then-reveal**: Bind participants to their `(norm_big_y, norm_big_c)` via a hash commitment in a prior round, analogous to the commitment-hash round in DKG. [7](#0-6) 

### Proof of Concept
A malicious participant `pₘ` replaces their call to `compute_signature_share` with a function that returns `(ElementG1::generator(), ElementG1::generator())`. The coordinator at lines 50–55 adds these values unconditionally. The resulting `CKDOutput` decrypts to `msk · H(pk ‖ app_id) + G − app_sk · G` — a value that is neither the intended confidential key nor detectable as wrong by any party in the protocol. [8](#0-7)

    mut chan: SharedChannel,
    participants: ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<CKDOutputOption, ProtocolError> {
    let (mut norm_big_y, mut norm_big_c) =
        compute_signature_share(&participants, me, key_pair, app_id, app_pk, rng)?;

    // Receive everyone's inputs and add them together
    let waitpoint = chan.next_waitpoint();

    for (_, participant_output) in
        recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
    {
        norm_big_y += participant_output.big_y();
        norm_big_c += participant_output.big_c();
    }
    let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
    Ok(Some(ckd_output))

    rng: impl CryptoRngCore + Send + 'static,

    // not enough participants

        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }

    // kick out duplicates

        return Err(InitializationError::DuplicateParticipants);

    // ensure my presence in the participant list

        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });

    // ensure the coordinator is a participant

        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
