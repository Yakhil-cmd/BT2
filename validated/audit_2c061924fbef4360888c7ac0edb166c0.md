Audit Report

## Title
`sender_info` Canister Signature Has No Expiration — Issued Attestations Are Irrevocable - (File: `rs/validator/src/ingress_validation.rs`)

## Summary
The `sender_info` feature allows a canister to attach a signed attestation blob to ingress/query requests. The signed content (`SenderInfoContent`) contains only the raw `info` bytes with no expiration timestamp, and `verify_sender_info_canister_sig` performs no time-bound check. Once a canister issues a `sender_info` signature, the recipient can reuse it indefinitely across new requests by simply refreshing `ingress_expiry`, with no mechanism for targeted revocation.

## Finding Description
`SignedSenderInfo` is defined with three fields — `info`, `signer`, `sig` — and no expiration field ( [1](#0-0) ). The signable content `SenderInfoContent` wraps only the raw `info` bytes; the `write_signed_bytes_without_domain_separator` implementation appends nothing beyond those bytes ( [2](#0-1) ). The validator `verify_sender_info_canister_sig` performs exactly three checks: (1) `sender_pubkey` is a valid canister signature public key, (2) the canister ID in the pubkey matches `sender_info.signer`, and (3) the signature over `info` is cryptographically valid — there is no `current_time` parameter and no expiry comparison ( [3](#0-2) ). The contrast with `validate_sender_delegation_expiry`, which explicitly checks `delegation.expiration() < current_time`, makes the omission structural, not incidental ( [4](#0-3) ). Because `sender_info.sig` is a standalone signature over `"ic-sender-info" || info_bytes` with no binding to a request ID or time window, the credential is fully portable and replayable. The outer `ingress_expiry` limits individual message submission to ~5 minutes but does not constrain reuse of the `sender_info` credential itself across an unlimited number of freshly constructed messages.

## Impact Explanation
Any canister acting as an attribute or identity provider (e.g., Internet Identity attesting role membership or KYC status) cannot revoke a `sender_info` credential it has already issued. A user whose access has been revoked can continue presenting the old `sender_info.sig` in new requests; the receiving canister reads the attestation via `msg_caller_info_data` and has no way to distinguish a live credential from a revoked one. The only available remediation for the signing canister is to rotate its canister signature seed, invalidating every credential it has ever issued. This constitutes unauthorized access to canister-controlled resources via a stale credential — matching the **High** impact tier: unauthorized access to identities or canister-controlled assets where exploitation requires no special privileges beyond possession of a legitimately issued credential.

## Likelihood Explanation
No special privileges are required. Any user who legitimately received a `sender_info` signature and subsequently had their access revoked can exploit this immediately by reusing the credential bytes in new requests with a fresh `ingress_expiry`. The `sender_info` mechanism is explicitly designed for identity/attribute providers such as Internet Identity, making deployments that issue time-limited or revocable credentials (session tokens, role grants, KYC status) directly affected. The attack requires no key compromise, no social engineering, and no network-level access — only possession of previously issued bytes.

## Recommendation
Add an `expiry_ns: u64` field (nanoseconds since Unix epoch) to `SenderInfoContent` so the signed payload becomes `"ic-sender-info" || expiry_ns || info_bytes`. Expose this field in `RawSignedSenderInfo` and `SignedSenderInfo`. In `verify_sender_info_canister_sig`, add a `current_time: Time` parameter and check `expiry_ns >= current_time.as_nanos_since_unix_epoch()` before accepting the signature, mirroring the pattern in `validate_sender_delegation_expiry` ( [4](#0-3) ).

## Proof of Concept
1. Canister C issues `sig = canister_sign("ic-sender-info" || b"role=admin")` for user U.
2. User U submits a valid request with this `sender_info`; the receiving canister grants admin access.
3. Canister C revokes U's admin role in its internal state but cannot invalidate the already-issued `sig`.
4. User U constructs a new `HttpCanisterUpdate` with a fresh `ingress_expiry` (within the 5-minute TTL), reuses the identical `sender_info` struct, and signs the new message with their own key.
5. `validate_sender_info` at line 482 calls `verify_sender_info_canister_sig` ( [5](#0-4) ), which passes — the canister signature remains cryptographically valid. The receiving canister reads `msg_caller_info_data` and sees `role=admin`, granting access despite the revocation.
6. A deterministic unit test can confirm this by: (a) creating a `SignedSenderInfo` with a fixed `info` blob and valid canister signature, (b) calling `verify_sender_info_canister_sig` at two different `current_time` values far apart, and (c) asserting both calls return `Ok(())` — demonstrating the absence of any time-bound rejection.

fn verify_sender_info_canister_sig<R: RootOfTrustProvider>(
    sender_info: &SignedSenderInfo,
    sender_pubkey_bytes: &[u8],
    validator: &dyn IngressSigVerifier,
    root_of_trust_provider: &R,
) -> Result<(), RequestValidationError>
where
    R::Error: std::error::Error,
{
    // Parse the envelope-level sender_pubkey DER to extract the raw
    // public key bytes and verify it's a valid canister signature public key.
    let pk_bytes = public_key_bytes_from_der(sender_pubkey_bytes).map_err(|e| {
        InvalidSenderInfo(format!(
            "sender_pubkey is not a valid canister signature public key: {e}"
        ))
    })?;

    // Extract the canister ID from the parsed public key and verify
    // it matches the declared signer.
    let parsed_pk = ic_crypto_iccsa::types::PublicKey::try_from(&pk_bytes)
        .map_err(|e| InvalidSenderInfo(format!("invalid canister sig public key: {e:?}")))?;
    let pubkey_canister_id = parsed_pk.signing_canister_id();
    if pubkey_canister_id != sender_info.signer {
        return Err(InvalidSenderInfo(format!(
            "signer {} does not match canister ID {} in sender_pubkey",
            sender_info.signer, pubkey_canister_id
        )));
    }

    // Construct the UserPublicKey for verification.
    let public_key = UserPublicKey {
        key: pk_bytes.0,
        algorithm_id: AlgorithmId::IcCanisterSignature,
    };

    // Construct the signable content (domain = "ic-sender-info")
    let sender_info_content = SenderInfoContent(&sender_info.info);
    let canister_sig = CanisterSigOf::from(CanisterSig(sender_info.sig.clone()));

    verify_canister_sig_with_fallback!(
        validator,
        &canister_sig,
        &sender_info_content,
        &public_key,
        root_of_trust_provider,
        |e| InvalidSenderInfo(format!("signature verification failed: {e}")),
        |e: <R as RootOfTrustProvider>::Error| InvalidSenderInfo(format!(
            "failed to get root of trust: {e}"
        ))
    );
    Ok(())
}

fn validate_sender_delegation_expiry(
    sender_delegation: &Option<Vec<SignedDelegation>>,
    current_time: Time,
) -> Result<(), RequestValidationError> {
    if let Some(delegations) = &sender_delegation {
        for delegation in delegations.iter() {
            let expiry = delegation.delegation().expiration();
            if delegation.delegation().expiration() < current_time {
                return Err(InvalidDelegationExpiry(format!(
                    "Specified sender delegation has expired:\n\
                     Provided expiry:    {expiry}\n\
                     Local replica time: {current_time}",
                )));
            }
        }
    }
    Ok(())
}
