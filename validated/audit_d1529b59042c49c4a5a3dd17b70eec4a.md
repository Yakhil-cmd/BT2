### Title
Attacker Can Bypass Signature Validation and Grief Victim's Post-Deploy Invoke via `skip_stateful_validations` — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the gateway's stateful transaction validator skips the `__validate__` entry point (on-chain signature verification) for invoke transactions with nonce=1 when `account_tx_in_pool_or_recent_block` returns `true` for the sender address. Because that check returns `true` for **any** transaction from the address in the pool — not exclusively a `deploy_account` — an attacker who observes a victim's pending `deploy_account` in the mempool can submit an invoke with nonce=1 for the victim's address carrying an **invalid signature**, bypass the `__validate__` call entirely, and get the invalid transaction admitted to the mempool. The victim's legitimate post-deploy invoke is then rejected as `DuplicateNonce`.

---

### Finding Description

**Relevant code path** (`extract_state_nonce_and_run_validations`):

```
get_nonce_from_state(address)          // on-chain nonce = 0 for undeployed account
  → run_pre_validation_checks(...)
      → validate_state_preconditions(...)   // nonce range check passes: 0 ≤ 1 ≤ max_gap
      → validate_by_mempool(...)            // mempool: no duplicate nonce yet → passes
      → skip_stateful_validations(...)      // returns true  ← root cause
  → run_validate_entry_point(..., skip_validate=true)
      → ExecutionFlags { validate: false }  // __validate__ NOT called
      → perform_pre_validation_stage(...)   // only nonce + fee-bound checks
      → returns Ok(())                      // invalid signature admitted
```

**`skip_stateful_validations` logic** ( [1](#0-0) ):

```rust
async fn skip_stateful_validations(...) -> ... {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())  // ← any tx, not deploy_account
                .await ...;
        }
    }
    Ok(false)
}
```

The comment states: *"It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."* This reasoning is **incorrect as a security invariant**: the attacker can cause `account_tx_in_pool_or_recent_block` to return `true` for the victim's address simply by observing the victim's `deploy_account` already in the pool — without the attacker having submitted anything for that address.

**`account_tx_in_pool_or_recent_block` implementation** ( [2](#0-1) ):

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
```

This returns `true` for **any** address that has ever had a transaction in the pool or a committed block — it does not distinguish `deploy_account` from `invoke`.

**`run_validate_entry_point` with `skip_validate=true`** ( [3](#0-2) ):

When `skip_validate` is `true`, `ExecutionFlags { validate: false }` is set. Inside `StatefulValidator::perform_validations`, the `__validate__` call is guarded by `if !tx.execution_flags.validate { return Ok(()); }` — so the account's signature is **never verified** at the gateway. [4](#0-3) 

**`validate_by_mempool` does not check signatures** ( [5](#0-4) ): `ValidationArgs` contains only `address`, `account_nonce`, `tx_hash`, `tx_nonce`, `tip`, `max_l2_gas_price` — no signature field. The mempool's `validate_tx` only checks nonce ordering and fee escalation.

**Fee-bound check**: `perform_pre_validation_stage` does call `verify_can_pay_committed_bounds` when `charge_fee` is `true` (determined by `enforce_fee`, which returns `true` for non-zero resource bounds). [6](#0-5)  This requires the victim's address to hold a small STRK balance. The attacker can satisfy this by pre-funding the deterministic address with a minimal amount of STRK before the attack, since the address is computable from the `deploy_account` parameters visible in the mempool.

---

### Impact Explanation

**Impact: High — Mempool/gateway admission accepts invalid transactions before sequencing.**

An invoke transaction carrying an **invalid (attacker-crafted) signature** for an account the attacker does not control is admitted to the mempool without any signature check. The victim's legitimate post-deploy invoke with the same nonce is then rejected as `DuplicateNonce` by `validate_by_mempool`. The attacker can repeat this indefinitely (the invalid invoke will be dropped during block execution since `__validate__` runs there, but the nonce slot is re-opened only after the block is committed, giving the attacker another window). This constitutes a targeted, repeatable denial-of-service against any user performing the `deploy_account + invoke` UX flow.

---

### Likelihood Explanation

**Likelihood: Medium.**

- The attacker must monitor the public mempool for `deploy_account` transactions (trivial with a full node).
- The attacker must pre-fund the victim's deterministic address with a small STRK amount (low cost, computable from the `deploy_account` parameters).
- The attacker must submit the invalid invoke before the victim's invoke reaches the mempool (a race condition, but the attacker has the advantage of acting immediately upon seeing the `deploy_account`).
- No special privileges are required; any unprivileged network participant can execute this.

---

### Recommendation

Replace the coarse `account_tx_in_pool_or_recent_block` check with a dedicated `has_pending_deploy_account(address) -> bool` query that returns `true` **only** when a `deploy_account` transaction for that address is currently in the pool (not yet committed). This eliminates the false-positive that allows the attacker to trigger the skip using the victim's own `deploy_account`.

Alternatively, require the caller to supply the `deploy_account` transaction hash (as the `native_blockifier` path does via `deploy_account_tx_hash: Option<TransactionHash>` in `PyValidator::should_run_stateful_validations`) and verify it is present in the pool before skipping validation. [7](#0-6) 

---

### Proof of Concept

**Setup**: Victim intends to deploy account at address `X` (deterministic from `class_hash`, `salt`, `constructor_calldata`) and immediately invoke it.

1. **Victim** submits `deploy_account` for address `X` → admitted to mempool. `account_tx_in_pool_or_recent_block(X)` now returns `true`.

2. **Attacker** computes address `X` from the `deploy_account` parameters visible in the mempool. Sends a small STRK transfer to `X` to satisfy `verify_can_pay_committed_bounds`.

3. **Attacker** submits `invoke_v3` with:
   - `sender_address = X`
   - `nonce = 1`
   - `signature = [0x0, 0x0]` (invalid)
   - `resource_bounds` = minimum non-zero values

4. **Gateway processing** of attacker's invoke:
   - `validate_nonce`: `0 ≤ 1 ≤ max_gap` → passes. [8](#0-7) 
   - `validate_by_mempool`: no existing nonce=1 tx for `X` → passes.
   - `skip_stateful_validations`: `nonce==1 && account_nonce==0 && account_tx_in_pool_or_recent_block(X)==true` → returns `true`. [9](#0-8) 
   - `run_validate_entry_point(skip_validate=true)`: `ExecutionFlags { validate: false }` → `__validate__` skipped. [10](#0-9) 
   - Attacker's **invalid invoke admitted** to mempool.

5. **Victim** submits their legitimate `invoke` with `nonce=1` for address `X`:
   - `validate_by_mempool`: nonce=1 already occupied by attacker's tx → `DuplicateNonce` error → **victim's invoke rejected**.

6. Attacker's invalid invoke is eventually dropped during block execution (when `__validate__` runs on-chain and rejects the bad signature), but the attacker can immediately repeat step 3, continuously blocking the victim's invoke at zero marginal cost beyond the pre-funding.

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
```

```rust
    pub fn should_run_stateful_validations(
        &mut self,
        account_tx: &AccountTransaction,
        deploy_account_tx_hash: Option<TransactionHash>,
    ) -> StatefulValidatorResult<bool> {
        if account_tx.tx_type() != TransactionType::InvokeFunction {
            return Ok(true);
        }
        let tx_info = account_tx.create_tx_info();
        let nonce = self.stateful_validator.get_nonce(tx_info.sender_address())?;

        let deploy_account_not_processed =
            deploy_account_tx_hash.is_some() && nonce == Nonce(Felt::ZERO);
        let tx_nonce = tx_info.nonce();
        let is_post_deploy_nonce = Nonce(Felt::ONE) <= tx_nonce;
        let nonce_small_enough_to_qualify_for_validation_skip =
            tx_nonce <= self.max_nonce_for_validation_skip;

        let skip_validate = deploy_account_not_processed
            && is_post_deploy_nonce
            && nonce_small_enough_to_qualify_for_validation_skip;

        Ok(!skip_validate)
    }
```
