### Title
Commission Frontrunning via `set_commission_commitment` Allows Staker to Steal Delegator Yield — (`File: src/pool/pool.cairo`, `src/staking/staking.cairo`)

---

### Summary

`enter_delegation_pool` and `add_to_delegation_pool` accept no `max_commission` guard. A malicious staker can use `set_commission_commitment` to pre-authorize a large commission increase, then frontrun a delegator's delegation transaction with `set_commission`, forcing the delegator into a pool with a far higher commission than they observed — with no recourse for at least one `exit_wait_window` (≥ 1 week).

---

### Finding Description

`set_commission_commitment` allows a staker to set a `max_commission` that is **higher** than the current commission:

```cairo
// src/staking/staking.cairo:769-770
let current_commission = staker_pool_info.commission();
assert!(current_commission <= max_commission, "{}", Error::MAX_COMMISSION_TOO_LOW);
``` [1](#0-0) 

While an active commitment is in force, `update_commission` permits setting commission to **any value up to `max_commission`**, including values strictly greater than the current commission:

```cairo
// src/staking/staking.cairo:1583-1589
if self.is_commission_commitment_active(:commission_commitment) {
    assert!(
        commission <= commission_commitment.max_commission, ...
    );
    assert!(commission != old_commission, ...);
}
``` [2](#0-1) 

The code itself acknowledges this gap:

```cairo
/// **Note**: Current commission increase safeguards still allow for sudden commission
/// changes.
``` [3](#0-2) 

Meanwhile, `enter_delegation_pool` and `add_to_delegation_pool` accept no `max_commission` parameter:

```cairo
fn enter_delegation_pool(
    ref self: ContractState, reward_address: ContractAddress, amount: Amount,
) { ... }
``` [4](#0-3) 

```cairo
fn add_to_delegation_pool(
    ref self: ContractState, pool_member: ContractAddress, amount: Amount,
) -> Amount { ... }
``` [5](#0-4) 

The commission used at reward time is always the **current live value** read from storage:

```cairo
let commission = staker_pool_info.commission();
...
let (commission_rewards, pool_rewards) = split_rewards_with_commission(
    rewards_including_commission: pool_rewards_including_commission, :commission,
);
``` [6](#0-5) 

---

### Impact Explanation

**High — Theft of unclaimed yield.**

A delegator who delegates expecting commission `X` can be forced to earn rewards at commission `max_commission` (up to 10 000 = 100%). The staker captures the difference as commission rewards. The delegator cannot exit immediately: `exit_delegation_pool_intent` starts a wait window of at least `DEFAULT_EXIT_WAIT_WINDOW` (1 week), during which all rewards accrue at the inflated commission rate. [7](#0-6) 

---

### Likelihood Explanation

**Medium.** The attack requires the staker to:
1. Call `set_commission_commitment` with a high `max_commission` (permissioned only to the staker themselves — no privileged role needed).
2. Observe a pending `enter_delegation_pool` mempool transaction and frontrun it with `set_commission`.

On Starknet, transaction ordering is controlled by the sequencer, but a staker can submit `set_commission` in the same block or immediately before a delegator's transaction. The setup cost is one extra transaction (`set_commission_commitment`), and the profit scales with the delegated amount and the commission delta.

---

### Recommendation

Add a `max_commission` parameter to `enter_delegation_pool`, `add_to_delegation_pool`, and `switch_delegation_pool`. Before completing the delegation, assert that the pool's current commission does not exceed the caller-supplied bound:

```cairo
fn enter_delegation_pool(
    ref self: ContractState,
    reward_address: ContractAddress,
    amount: Amount,
    max_commission: Commission,   // <-- new
) {
    ...
    let current_commission = self.get_commission_from_staking_contract();
    assert!(current_commission <= max_commission, "Commission exceeds max");
    ...
}
```

This mirrors the `maxCash` fix applied in the referenced DutchAuction report and gives delegators a slippage guard against commission frontrunning.

---

### Proof of Concept

1. Staker stakes with commission `100` (1%) and opens a pool.
2. Staker calls `set_commission_commitment(max_commission: 9000, expiration_epoch: current_epoch + 1)`.
   - Valid: `100 <= 9000`. [1](#0-0) 
3. Delegator observes commission `100`, approves tokens, and submits `enter_delegation_pool(reward_address, amount)`.
4. Staker frontruns with `set_commission(9000)`.
   - Valid during active commitment: `9000 <= 9000` and `9000 != 100`. [2](#0-1) 
5. Delegator's `enter_delegation_pool` executes. No commission check exists in the function. [8](#0-7) 
6. All rewards for the delegator are now split with 90% going to the staker as commission. The delegator must wait ≥ 1 week to exit. [7](#0-6)
