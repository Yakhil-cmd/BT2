### Title
Missing Caller Validation in `update_rewards` Allows Any Address to Permanently Suppress Consensus Block Rewards - (File: src/staking/staking.cairo)

---

### Summary
The `update_rewards` function in the staking contract is documented to be callable only by the Starkware sequencer, but the implementation enforces no such restriction. Any unprivileged address can call it with `disable_rewards: true`, consuming the single per-block reward slot and permanently denying all stakers their consensus block rewards for that block.

---

### Finding Description
`update_rewards` is the consensus-phase reward distribution entry point. The spec at `docs/spec.md:1644–1645` explicitly states its access control is **"Only starkware sequencer."** However, the implementation at `src/staking/staking.cairo:1449–1507` only calls `general_prerequisites()`, which checks that the contract is not paused and the caller is not the zero address — no sequencer identity check is performed. [1](#0-0) [2](#0-1) 

The function enforces a **global** `last_reward_block` guard: only one `update_rewards` call is permitted per block across the entire contract. [3](#0-2) 

Critically, `last_reward_block` is written **before** the `disable_rewards` branch is evaluated: [4](#0-3) 

When an attacker calls `update_rewards(valid_staker_address, disable_rewards: true)`:
1. `last_reward_block` is updated to the current block number.
2. The function returns early — no rewards are calculated or distributed.
3. Any subsequent legitimate call in the same block reverts with `REWARDS_ALREADY_UPDATED`.

The `last_reward_block` field is a single scalar in storage (not per-staker), so one attacker call poisons the entire block for all stakers. [5](#0-4) 

---

### Impact Explanation
**High — Permanent freezing of unclaimed yield.**

By calling `update_rewards(any_active_staker, true)` at the start of every block, an attacker permanently prevents all stakers and their delegators from accruing any consensus-phase block rewards. The rewards for each skipped block are never calculated and never enter `unclaimed_rewards_own` or the pool contracts — they are irreversibly lost. Delegators in pool contracts are equally affected because pool rewards flow through `_update_rewards` → `update_pool_rewards` → `update_rewards_from_staking_contract`, none of which execute when `disable_rewards: true` short-circuits the function. [6](#0-5) 

---

### Likelihood Explanation
**High.** The function is publicly callable with no privileged role requirement. The attacker only needs to know any one active staker address (all are public on-chain via `NewStaker` events) and pay gas per block. The attack is cheap, deterministic, and requires no special setup. The `REWARDS_ALREADY_UPDATED` guard makes it impossible for the legitimate sequencer to recover within the same block once the attacker fires first.

---

### Recommendation
Add a caller check inside `update_rewards` that restricts invocation to the authorized sequencer address (or a dedicated role), matching the access control stated in the specification. For example, store the sequencer address in contract storage and assert `get_caller_address() == self.sequencer_address.read()` before any other logic. Alternatively, move the `last_reward_block.write` to after the `disable_rewards` guard so that a no-op call does not consume the block slot. [4](#0-3) 

---

### Proof of Concept
1. Consensus rewards are active (`!is_pre_consensus()`).
2. Attacker identifies any active staker address `S` (e.g., from a `NewStaker` event).
3. At the start of block `N`, attacker calls `update_rewards(S, disable_rewards: true)`.
4. `last_reward_block` is set to `N`; function returns without distributing rewards.
5. The legitimate sequencer calls `update_rewards(S, disable_rewards: false)` in the same block → reverts with `REWARDS_ALREADY_UPDATED`.
6. All stakers and delegators receive zero rewards for block `N`.
7. Attacker repeats step 3 every block → all consensus rewards are permanently frozen.

The test suite itself demonstrates that calling with `disable_rewards: true` followed by `disable_rewards: false` in the same block always fails: [7](#0-6) [8](#0-7)

        fn update_rewards(
            ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
        ) {
            self.general_prerequisites();
            let current_block_number = starknet::get_block_number();
            assert!(
                current_block_number > self.last_reward_block.read(),
                "{}",
                Error::REWARDS_ALREADY_UPDATED,
            );

            // Assert staker exists and active.
            // Staker is considered to exist from the moment of `stake` (when `InternalStakerInfo`
            // struct is created) until the calling to `unstake_action` (when `InternalStakerInfo`
            // struct is deleted).
            // Staker remains active until the intent period begins, i.e. K epochs after
            // `unstake_intent` is called.
            let staker_info = self.internal_staker_info(:staker_address);
            let curr_epoch = self.get_current_epoch();
            assert!(
                self.is_staker_active(:staker_address, epoch_id: curr_epoch),
                "{}",
                Error::INVALID_STAKER,
            );

            let staker_pool_info = self.staker_pool_info.entry(staker_address).as_non_mut();
            let (staker_total_strk_balance, staker_total_btc_balance) = self
                .get_staker_total_strk_btc_balance_at_epoch(
                    :staker_address, :staker_pool_info, epoch_id: curr_epoch,
                );
            // Assert staker has non-zero balance.
            // Staker exists with zero balance for the first K epochs after `stake`, then the stake
            // becomes effective.
            assert!(staker_total_strk_balance.is_non_zero(), "{}", Error::INVALID_STAKER);

            // Update last block rewards.
            self.last_reward_block.write(current_block_number);

            if disable_rewards || self.is_pre_consensus() {
                return;
            }
