### Title
Pool Admin Can Set Uncapped Per-Bin Additional Fees via `setPoolBinAdditionalFees`, Bypassing Global Fee Caps — (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`MetricOmmPoolFactory.setPoolBinAdditionalFees` forwards `addFeeBuyE6` and `addFeeSellE6` to the pool with **no cap validation**, while every other admin fee setter is bounded by `maxAdminSpreadFeeE6` / `maxAdminNotionalFeeE8`. A pool admin can set per-bin additional fees to `type(uint16).max = 65 535` (≈ 6.55 % in E6 scale) on any bin, charging traders fees that exceed the protocol-enforced cap and increasing the admin's own fee extraction beyond the intended ceiling.

---

### Finding Description

The factory enforces fee caps on the admin's spread and notional components through `setPoolAdminFees`:

```solidity
// MetricOmmPoolFactory.sol:414-415
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (newAdminNotionalFeeE8 > maxAdminNotionalFeeE8) revert AdminFeeTooHigh();
```

`maxAdminSpreadFeeE6` is itself bounded by the hard constant `HARD_MAX_SPREAD_FEE_E6 = 200_000` (20 %).

However, `setPoolBinAdditionalFees` passes the caller-supplied values straight through with **no analogous check**:

```solidity
// MetricOmmPoolFactory.sol:450-457
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

The pool itself also performs no cap check:

```solidity
// MetricOmmPool.sol:464-474
function setBinAdditionalFees(int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external onlyFactory nonReentrant(PoolActions.SET_BIN_ADDITIONAL_FEES)
{
    if (bin < LOWEST_BIN || bin > HIGHEST_BIN) revert InvalidBinIndex(bin);
    BinState storage s = _binStates[bin];
    s.addFeeBuyE6 = addFeeBuyE6;
    s.addFeeSellE6 = addFeeSellE6;
    emit BinAdditionalFeesUpdated(bin, addFeeBuyE6, addFeeSellE6);
}
```

During a swap, `addFeeBuyE6` is added directly to the oracle-derived base fee to form the total bin fee charged to the trader:

```solidity
// MetricOmmPool.sol:540-541
uint256 buyFeeX64  = baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6);
uint256 sellFeeX64 = baseFeeX64 + Math.mulDiv(binState.addFeeSellE6, ONE_X64, 1e6);
```

The LP fee collected per bin is `currBinBuyFeeX64 × grossInput`. The admin's share of that LP fee is `adminSpreadFeeE6 / spreadSumE6` of `spreadFeeE6 / 1e6 × lpFee`. Because `addFeeBuyE6` enlarges the LP fee base, the admin's absolute extraction grows proportionally — even though `adminSpreadFeeE6` itself is at its cap.

---

### Impact Explanation

**Direct trader loss.** A pool admin can call `setPoolBinAdditionalFees(pool, bin, 65535, 65535)` to impose an additional ≈ 6.55 % fee on every swap through that bin, on top of the oracle spread and the capped admin spread fee. Traders receive fewer output tokens than the fee-cap system was designed to guarantee.

**Admin-boundary break.** The admin's total fee extraction from a bin becomes:

```
adminSpreadFeeE6/1e6 × (oracleSpread + addFeeBuyE6/1e6) × tradeAmount
```

At maximum values (adminSpreadFeeE6 = 200 000, addFeeBuyE6 = 65 535) this adds ≈ 1.31 % of trade volume to the admin's take, beyond what `maxAdminSpreadFeeE6` was intended to permit. The protocol's fee-cap invariant — that the admin cannot extract more than the owner-set ceiling — is broken without any privileged owner action.

**No timelock.** Unlike price-provider changes, `setPoolBinAdditionalFees` takes effect immediately, so there is no window for LPs or traders to exit before the elevated fee applies.

---

### Likelihood Explanation

The pool admin is a semi-trusted role that is explicitly constrained by caps. The bypass requires only a single admin transaction with no special preconditions. Any pool whose admin is compromised, acts maliciously, or makes an unchecked configuration error can trigger this immediately on any active bin.

---

### Recommendation

Add a cap check in `setPoolBinAdditionalFees` consistent with the existing admin fee ceiling. For example:

```solidity
function setPoolBinAdditionalFees(
    address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6
) external override nonReentrant onlyPoolAdmin(pool) {
    // addFeeBuyE6 and addFeeSellE6 are E6-scaled; compare against the same cap used for adminSpreadFeeE6
    if (uint24(addFeeBuyE6) > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    if (uint24(addFeeSellE6) > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

Alternatively, introduce a dedicated `maxAdminBinAdditionalFeeE6` cap that the owner can configure, mirroring the pattern used for `maxAdminSpreadFeeE6`.

---

### Proof of Concept

1. Pool is created with `maxAdminSpreadFeeE6 = 200_000` (20 %) and `adminSpreadFeeE6 = 200_000`.
2. Pool admin calls:
   ```solidity
   factory.setPoolBinAdditionalFees(pool, 0, 65535, 65535);
   ```
   No revert occurs; the call succeeds.
3. A trader executes a swap through bin 0. The effective buy fee is:
   ```
   baseFeeX64 + mulDiv(65535, ONE_X64, 1e6)
   ```
   — approximately 6.55 % additional fee on top of the oracle spread, charged to the trader with no cap enforcement.
4. The admin's fee extraction from that swap is `adminSpreadFeeE6/1e6 × totalLpFee`, where `totalLpFee` now includes the uncapped 6.55 % component, yielding extraction above the intended 20 %-of-oracle-spread ceiling. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4)
