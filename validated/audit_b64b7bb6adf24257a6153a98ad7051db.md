### Title
Gateway Stateful Validator Checks L2 Gas Price Against Current Block's `l2_gas_price` Instead of `next_l2_gas_price` — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The stateful gateway validator's `validate_resource_bounds` function computes its admission threshold from the **current block's** `l2_gas_price` field. Transactions are destined for the **next** block, whose L2 gas price is `next_l2_gas_price` (the EIP-1559-adjusted value stored separately in the block header). The two values diverge every block. A developer TODO comment in the code explicitly acknowledges the wrong field is being read. The result is that the gateway either admits transactions that will fail execution (when the price is rising) or rejects transactions that would succeed (when the price is falling).

---

### Finding Description

In `crates/apollo_gateway/src/stateful_transaction_validator.rs`, `validate_resource_bounds` reads the reference gas price as:

```rust
// TODO(Arni): getnext_l2_gas_price from the block header.
let previous_block_l2_gas_price = self
    .gateway_fixed_block_state_reader
    .get_block_info()
    .await?
    .gas_prices
    .strk_gas_prices
    .l2_gas_price;
``` [1](#0-0) 

This value comes from `GatewayFixedBlockSyncStateClient::get_block_info_from_sync_client`, which maps `block_header.l2_gas_price.price_in_fri` into the returned `BlockInfo`:

```rust
strk_gas_prices: GasPriceVector {
    ...
    l2_gas_price: block_header.l2_gas_price.price_in_fri.try_into()?,
},
``` [2](#0-1) 

`block_header.l2_gas_price` is the price **used inside block N**. The price that will govern block N+1 (the block the incoming transaction will actually be included in) is `block_header.next_l2_gas_price`, a separate field computed by the EIP-1559 mechanism:

```rust
pub next_l2_gas_price: GasPrice,
``` [3](#0-2) 

`next_l2_gas_price` is never surfaced through the `GatewayFixedBlockStateReader::get_block_info()` interface — the `BlockInfo` struct returned by that trait does not carry it. The validator therefore always computes its threshold from the stale field.

The threshold comparison is:

```rust
let threshold = (gas_price_threshold_multiplier
    * previous_block_l2_gas_price.get().0)
    .to_integer();
if tx_l2_gas_price.0 < threshold {
    return Err(...);
}
``` [4](#0-3) 

Because `l2_gas_price` ≠ `next_l2_gas_price`, the threshold is wrong in both directions.

---

### Impact Explanation

**Rising-price scenario** (`next_l2_gas_price > l2_gas_price`):  
The threshold is too low. A transaction whose `max_price_per_unit` satisfies `threshold_current ≤ max_price < threshold_next` passes the gateway check and enters the mempool, but the blockifier's `check_fee_bounds` (which uses the actual block context gas price) will reject it during execution. The transaction is admitted as valid but is not executable — matching **"Mempool/gateway/RPC admission accepts invalid transactions before sequencing."**

**Falling-price scenario** (`next_l2_gas_price < l2_gas_price`):  
The threshold is too high. A transaction whose `max_price_per_unit` satisfies `threshold_next ≤ max_price < threshold_current` would succeed in the next block but is rejected at the gateway — matching **"Mempool/gateway/RPC admission rejects valid transactions before sequencing."**

---

### Likelihood Explanation

The L2 gas price changes every block via the EIP-1559 `calculate_next_base_gas_price` function. During any period of non-zero block utilization the two values diverge. The gap is bounded by `gas_price_max_change_denominator` per block, but it is non-zero on every block where gas usage differs from the target. Any user submitting a transaction with `max_price_per_unit` in the gap between the two prices triggers the wrong admission decision without any special privilege.

---

### Recommendation

1. Extend `GatewayFixedBlockStateReader::get_block_info` (or add a new method) to also return `next_l2_gas_price` from `BlockHeaderWithoutHash`.
2. In `validate_resource_bounds`, replace the read of `gas_prices.strk_gas_prices.l2_gas_price` with the `next_l2_gas_price` value from the block header.
3. Add tests covering both the rising-price and falling-price cases to confirm the threshold is computed from `next_l2_gas_price`.

---

### Proof of Concept

1. Observe the current `next_l2_gas_price` for block N (e.g., 110 FRI) and `l2_gas_price` for block N (e.g., 100 FRI). With `min_gas_price_percentage = 100`, the threshold is 100 FRI.
2. Submit an invoke transaction with `l2_gas.max_price_per_unit = 105 FRI`.
3. The gateway admits the transaction (105 ≥ 100).
4. The batcher includes it in block N+1, which uses `next_l2_gas_price = 110 FRI`.
5. Blockifier's `check_fee_bounds` rejects it: `max_price_per_unit (105) < actual_gas_price (110)`.
6. The transaction was admitted by the gateway but cannot be executed — the admission invariant is violated. [5](#0-4) [6](#0-5) [7](#0-6)
