Audit Report

## Title
ICRC-21 `add_expiration` Wrapping Cast Silently Displays Epoch 0 for `expires_at = u64::MAX` in `FieldsDisplayMessage` Path — (`packages/icrc-ledger-types/src/icrc21/responses.rs`)

## Summary
In `add_expiration`, the `FieldsDisplayMessage` branch casts `expires_at: u64` to `i64` before dividing by `10^9`. For `expires_at = u64::MAX`, this wrapping cast produces `-1`, integer division yields `0`, and the re-cast to `u64` stores `0`, emitting `Value::TimestampSeconds { amount: 0 }` (Unix epoch, Jan 1 1970). The ledger's `approve` path accepts `u64::MAX` as a valid future expiration (~year 2554), so a malicious dApp can obtain a permanent approval while the hardware wallet's structured consent display shows an already-expired date, deceiving the user into signing.

## Finding Description

**Root cause — `packages/icrc-ledger-types/src/icrc21/responses.rs`, lines 262–271:**

```rust
ConsentMessage::FieldsDisplayMessage(fields_display) => {
    match expires_at {
        Some(expires_at) => {
            let seconds = (expires_at as i64) / 10_i64.pow(9);  // wrapping cast
            fields_display.fields.push((
                "Approval expiration".to_string(),
                Value::TimestampSeconds {
                    amount: seconds as u64,                       // 0 for u64::MAX
                },
            ))
``` [1](#0-0) 

Arithmetic for `expires_at = u64::MAX = 18446744073709551615`:

| Step | Expression | Result |
|------|-----------|--------|
| 1 | `u64::MAX as i64` | `-1` (defined wrapping cast in Rust) |
| 2 | `-1_i64 / 1_000_000_000_i64` | `0` (rounds toward zero) |
| 3 | `0_i64 as u64` | `0` |

**Exploit path:**

1. A malicious dApp constructs `ApproveArgs { amount: large, expires_at: Some(u64::MAX), spender: attacker }` and requests the user sign via a hardware wallet.
2. The hardware wallet calls `icrc21_canister_call_consent_message` with `DisplayMessageType::FieldsDisplay` (the structured path designed for constrained devices).
3. `build_icrc21_consent_info` in `packages/icrc-ledger-types/src/icrc21/lib.rs` decodes `ApproveArgs`, extracts `expires_at`, and passes it through `with_expires_at(expires_at)` → `add_expiration`. [2](#0-1) 

4. `add_expiration` emits `TimestampSeconds { amount: 0 }`. The hardware wallet renders "Thu, 01 Jan 1970 00:00:00 +0000" — an already-expired date. The user concludes the approval is harmless and signs.
5. `icrc2_approve` passes `expires_at` directly to the approval table. [3](#0-2) 

6. The approval core checks `expires_at.unwrap_or_else(remote_future) <= now`. Since `u64::MAX > now`, the approval is stored as valid until year ~2554. [4](#0-3) 

**Existing guards are insufficient:** The ledger correctly rejects past timestamps but has no upper-bound sanitization on `expires_at`. The consent message system is the only user-facing protection, and the bug silently corrupts it for this input.

The `GenericDisplay` path at lines 232–233 has the same wrapping cast but at least produces a visible error string (`"Invalid nanosecond: 4294967295"`) via `replace_nanosecond` failure. The `FieldsDisplay` path silently emits epoch 0 with no error. [5](#0-4) 

## Impact Explanation

A user with a hardware wallet is deceived into signing an `icrc2_approve` that grants a spender unlimited access to their tokens until year ~2554, while the consent message shows an already-expired date. The attacker's spender can immediately drain the approved allowance. This constitutes unauthorized access to ledger-controlled funds via a broken consent integrity mechanism — matching the **High** impact class: "Unauthorized access to… ledgers, or canister-controlled funds where exploitation requires meaningful per-target work or other constraints."

## Likelihood Explanation

The attack requires only an unprivileged user interacting with a malicious dApp on a hardware wallet that requests `FieldsDisplay`. No privileged access, key compromise, or consensus manipulation is needed. `FieldsDisplay` is the variant explicitly designed for hardware wallets and is defined in the production DID. The attacker controls only the `ApproveArgs` payload, which is a standard user-supplied input. The attack is repeatable against any hardware wallet user who interacts with a malicious dApp.

## Recommendation

Replace the wrapping `i64` cast with pure `u64` arithmetic in both branches of `add_expiration`:

```rust
// FieldsDisplay branch — fix:
let seconds = expires_at / 1_000_000_000_u64;
fields_display.fields.push((
    "Approval expiration".to_string(),
    Value::TimestampSeconds { amount: seconds },
))

// GenericDisplay branch — fix:
let seconds = expires_at / 1_000_000_000_u64;
let nanos = (expires_at % 1_000_000_000_u64) as u32;
```

Add an invariant test asserting that for any `expires_at` (including `u64::MAX`), `TimestampSeconds.amount == expires_at / 1_000_000_000` computed in `u64` arithmetic.

## Proof of Concept

```rust
// Reproduces the bug — no canister needed
let expires_at: u64 = u64::MAX;

// Buggy path (current code):
let seconds_buggy = (expires_at as i64) / 10_i64.pow(9);
let amount_buggy = seconds_buggy as u64;
assert_eq!(amount_buggy, 0); // displays Jan 1 1970

// Correct path:
let seconds_correct = expires_at / 1_000_000_000_u64;
assert_eq!(seconds_correct, 18446744073); // year ~2554

// On-chain: ledger accepts u64::MAX because u64::MAX > now
// approvals.rs line 247: expires_at.unwrap_or_else(remote_future) <= now
// u64::MAX > current_time → approval stored, valid until year 2554
```
