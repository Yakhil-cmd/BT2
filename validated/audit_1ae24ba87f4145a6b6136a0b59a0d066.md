### Title
Unbacked ERC-20 Token Minting via Direct `ft_on_transfer` Call with Attacker-Controlled Amount — (File: `engine/src/contract_methods/connector.rs`)

### Summary

Aurora Engine's `ft_on_transfer` entry point trusts the caller-supplied `amount` field without verifying that the corresponding NEP-141 tokens were actually transferred. Because `deploy_erc20_token` is also unrestricted, any unprivileged NEAR account can register a NEP-141 token and then call `ft_on_transfer` directly — bypassing the `ft_transfer_call` deduction step — to mint an arbitrary quantity of ERC-20 mirror tokens on Aurora.

### Finding Description

The NEP-141 standard's `ft_transfer_call` flow is:

1. Caller invokes `ft_transfer_call` on the NEP-141 contract.
2. NEP-141 contract deducts tokens from the caller's balance.
3. NEP-141 contract calls `ft_on_transfer` on the receiver (Aurora Engine).
4. Aurora mints the corresponding ERC-20 tokens.
5. NEP-141 contract calls `ft_resolve_transfer` to finalise.

Aurora Engine's `ft_on_transfer` implementation performs only one access check: it verifies that `predecessor_account_id` (the NEAR caller) has a registered NEP-141 → ERC-20 mapping.

```rust
// engine/src/contract_methods/connector.rs  lines 80-90

        &predecessor_account_id,
        &args,

There is no assertion that the call arrived as a callback of a `ft_transfer_call` promise, and there is no cross-check of `args.amount` against any on-chain token balance. The `amount` field is entirely attacker-controlled.

`deploy_erc20_token` — the function that creates the NEP-141 → ERC-20 mapping — contains no `require_owner_only` guard:

### Impact Explanation

An attacker can mint an unbounded quantity of ERC-20 mirror tokens on Aurora without holding the corresponding NEP-141 tokens on NEAR. If the attacker has seeded a DEX liquidity pool for that token (using a small amount of real tokens), they can dump the unbacked ERC-20 tokens into the pool and drain the paired asset (e.g., ETH or USDC). This constitutes **direct theft of user funds** held in Aurora DEX liquidity pools.

**Impact**: Critical — direct theft of user funds.

### Likelihood Explanation

The attack requires only:
1. Creating a NEAR account and deploying a minimal NEP-141 contract (permissionless on NEAR).
2. Calling `deploy_erc20_token` on Aurora (no access restriction observed).
3. Calling `ft_on_transfer` directly from the attacker-controlled NEP-141 account with an inflated `amount`.

No privileged access, leaked keys, or governance capture is required. Any unprivileged EVM user or NEAR account holder can execute this.

**Likelihood**: High.

### Recommendation

1. **Verify the call is a NEP-141 callback**: Check `promise_results_count() == 1` and that the promise result is successful before processing any mint. This ensures `ft_on_transfer` was invoked as part of a genuine `ft_transfer_call` and not called directly.
2. **Restrict `deploy_erc20_token`**: Add `require_owner_only` (or a whitelist check) so that only the Aurora operator can register new NEP-141 → ERC-20 mappings, limiting the attack surface to tokens the operator has vetted.
3. **Bound `args.amount`**: Cross-check the minted amount against the actual NEP-141 balance change if a reliable on-chain oracle is available.

### Proof of Concept

```
1. Attacker deploys a NEP-141 contract at `evil_token.near`.
2. Attacker calls aurora.deploy_erc20_token("evil_token.near")
   → Aurora creates ERC-20 mirror at address 0xABCD...
3. Attacker seeds a USDC/evil_token pool on an Aurora DEX with 1 real token.
4. Attacker calls aurora.ft_on_transfer from evil_token.near with:
     { sender_id: "evil_token.near", amount: "1000000000000", msg: "<attacker_evm_addr>" }
   → Aurora mints 1,000,000,000,000 ERC-20 evil_token to attacker's EVM address.
5. Attacker swaps all minted tokens into USDC via the DEX pool.
   → Pool drained; attacker profits; USDC LPs lose funds.
```

The root cause is identical in structure to the Chainlink report: the system accepts an attacker-supplied parameter (`amount` / callback address) without binding it to the identity of the legitimate originator, allowing the attacker to substitute their own value and have it honoured by the protocol.
