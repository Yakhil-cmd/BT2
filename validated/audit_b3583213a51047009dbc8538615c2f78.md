### Title
Missing Origin and rpIdHash Validation in WebAuthn Transaction Signature Allows Cross-Origin Credential Abuse - (File: model/flow/webauthn.go)

---

### Summary

The Flow FVM's WebAuthn transaction signature validation (`validateWebAuthNExtensionData`) parses but never validates the `origin` field from `clientDataJSON`, and only performs a trivial negative check on `rpIdHash` (rejecting only the exact `TransactionDomainTag` value). Any WebAuthn assertion produced by any origin or relying party is accepted as a valid Flow transaction signature, as long as the challenge and flags are structurally correct. This is the direct analog of the iOS URI scheme hijacking vulnerability: just as any iOS app can register the same URI scheme and intercept messages, any web origin can produce a WebAuthn assertion that the FVM accepts as a valid transaction authorization.

---

### Finding Description

`validateWebAuthNExtensionData` in `model/flow/webauthn.go` implements the FVM-side verification of WebAuthn-signed Flow transactions (FLIP 264 / `WebAuthnScheme`). After decoding the RLP-encoded extension data, it performs the following checks:

1. `clientData.Type` must equal `"webauthn.get"` [1](#0-0) 
2. The challenge must equal `SHA256(TransactionDomainTag || payload)` [2](#0-1) 
3. `authenticatorData` must be at least 37 bytes [3](#0-2) 
4. `rpIdHash` must **not** equal `TransactionDomainTag[:]` [4](#0-3) 
5. Authenticator flags (UP, BE/BS, AT/ED) are validated [5](#0-4) 

**What is never checked:**
- The `origin` field in `CollectedClientData` is parsed into the struct but is never compared against any expected value. [6](#0-5) 
- The `rpIdHash` is only checked to be *not equal* to `TransactionDomainTag`. Any other 32-byte value — including `SHA256("evil.com")` — is accepted. [7](#0-6) 

The test suite explicitly encodes this as intentional behavior: the test case `"empty origin (valid)"` passes `origin: ""` and asserts `extensionOk: true`, confirming the FVM accepts assertions from any origin. [8](#0-7) 

The `CollectedClientData` struct has an `Origin` field that is populated during JSON unmarshalling but is never read after that point. [9](#0-8) 

---

### Impact Explanation

In standard WebAuthn (W3C spec), the `origin` and `rpIdHash` are the binding between a credential and a specific application. The authenticator device enforces that a credential created for `flow-wallet.com` can only produce assertions for `flow-wallet.com` (via rpId). The *server* (here, the FVM) is required to verify that the `origin` in `clientDataJSON` matches the expected origin and that `rpIdHash` equals `SHA256(expected_rpId)`.

Because the FVM skips both checks, the following attack is possible:

1. A user registers a WebAuthn key on their Flow account via a legitimate wallet at `flow-wallet.com` (rpId = `flow-wallet.com`).
2. A malicious site `evil.com` also registers a WebAuthn credential for the same user (different key, rpId = `evil.com`). If the user is tricked into adding this key to their Flow account (e.g., via a social-engineering step or a compromised wallet UI), the malicious key is now a valid signer.
3. `evil.com` presents the user with a WebAuthn authentication prompt. The challenge is crafted to be `SHA256(TransactionDomainTag || malicious_tx_payload)`.
4. The user's authenticator signs the challenge, producing `authenticatorData` with `rpIdHash = SHA256("evil.com")` and `clientDataJSON` with `origin = "https://evil.com"`.
5. The attacker submits this as a Flow transaction signature. The FVM validates: type ✓, challenge ✓, rpIdHash ≠ TransactionDomainTag ✓, flags ✓, **origin not checked** ✓.
6. The transaction executes with the user's authorization, despite the user never knowingly signing a Flow transaction.

This is **transaction authorization bypass** — the FVM accepts a WebAuthn assertion produced by an arbitrary, attacker-controlled origin as a valid transaction signature. The impact is unauthorized mutation of on-chain state (asset transfers, contract deployments, key additions) on behalf of the victim account.

---

### Likelihood Explanation

The attack requires:
- The victim to have a WebAuthn key registered on their Flow account (prerequisite for any WebAuthn-scheme transaction).
- The attacker to have a key registered on the victim's Flow account (either the same key via phishing, or a separate key added through a separate attack step).
- The attacker to trick the victim into performing a WebAuthn gesture with a crafted challenge.

The first condition is the normal operating state for WebAuthn users. The second condition is the primary barrier, but it is realistic in phishing scenarios where a malicious dApp presents itself as a legitimate wallet and tricks the user into "linking" their account. The third condition is straightforward once the second is met. The missing origin validation removes the cryptographic guarantee that was supposed to prevent exactly this class of cross-origin credential abuse.

---

### Recommendation

1. **Short term:** Validate the `origin` field in `clientDataJSON` against a configurable allowlist of expected origins (e.g., registered Flow wallet domains). At minimum, reject empty origins. Document that the current behavior intentionally accepts any origin if that is a deliberate design choice, with a clear security rationale.

2. **Long term:** Validate `rpIdHash` against `SHA256(expected_rpId)` for each registered key, binding each WebAuthn public key on a Flow account to a specific relying party identifier at key registration time. This mirrors the W3C WebAuthn spec's relying party verification step and is the direct analog of Apple's Universal Links recommendation in the original report — binding the credential to a domain that is solely owned by the legitimate application.

---

### Proof of Concept

The following demonstrates that `validateWebAuthNExtensionData` accepts an assertion with an empty origin and an arbitrary `rpIdHash`:

```go
// Attacker constructs extension data with origin="" and rpIdHash=SHA256("evil.com")
clientDataJSON := map[string]string{
    "type":      flow.WebAuthnTypeGet,
    "challenge": base64.RawURLEncoding.EncodeToString(SHA256(TransactionDomainTag + payload)),
    "origin":    "",  // or "https://evil.com" — neither is rejected
}
rpIdHash := SHA256([]byte("evil.com"))  // any value except TransactionDomainTag is accepted
authenticatorData := concat(rpIdHash, []byte{0x01}, sigCounter)
```

The test case at `model/flow/transaction_test.go` lines 510–518 already encodes this as passing behavior (`extensionOk: true`, `signatureOk: true` for `origin: ""`). [8](#0-7) 

The root cause is in `validateWebAuthNExtensionData`: after extracting `rpIdHash` and parsing `clientData`, neither the `origin` nor the `rpIdHash` is compared against any expected value. [10](#0-9)
