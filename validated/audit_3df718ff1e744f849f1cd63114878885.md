### Title
Inconsistent `withdrawToNear` Precompile Calldata Encoding Between `EvmErc20` and `EvmErc20V2` Causes Silent Token Burn Without NEAR Transfer — (File: `etc/eth-contracts/contracts/EvmErc20.sol` / `etc/eth-contracts/contracts/EvmErc20V2.sol`)

---

### Summary

`EvmErc20` and `EvmErc20V2` both implement the `IExit` interface and expose the same `withdrawToNear(bytes memory recipient, uint256 amount)` function signature, but they encode the raw calldata sent to the ExitToNear precompile (`0xe9217bc70b7ed1f598ddd3199e80b093fa71124f`) differently. The precompile can only correctly parse one encoding. Because the assembly `call` return value is never checked in either contract, a mismatch causes the ERC-20 tokens to be burned while the corresponding NEAR transfer silently never occurs — permanently destroying user funds.

---

### Finding Description

`EvmErc20.withdrawToNear` encodes the precompile input as:

```
[0x01][32-byte amount][recipient bytes]
``` [1](#0-0) 

`EvmErc20V2.withdrawToNear` encodes the precompile input as:

```
[0x01][20-byte sender address][32-byte amount][recipient bytes]
``` [2](#0-1) 

Both contracts call the identical precompile address `0xe9217bc70b7ed1f598ddd3199e80b093fa71124f` with these divergent byte layouts. The ExitToNear precompile has a single fixed parsing logic; it can only correctly interpret one of these two formats. Whichever contract sends the wrong layout will cause the precompile to misparse the amount and recipient fields, resulting in a failed or corrupted withdrawal.

Critically, in both contracts the assembly `call` result is captured in a local variable `res` but is **never checked or acted upon**:

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
}
``` [3](#0-2) [4](#0-3) 

The `_burn` call that precedes the assembly block executes unconditionally and commits before the precompile is invoked. If the precompile call fails (due to malformed input), execution continues without revert, leaving the user's ERC-20 balance permanently reduced with no NEAR tokens released.

The `IExit` interface defines a single shared signature:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external;
``` [5](#0-4) 

Both contracts satisfy this interface at the Solidity ABI level, masking the internal encoding divergence from callers and integrators. By contrast, `withdrawToEthereum` uses identical encoding in both contracts, making the `withdrawToNear` inconsistency a targeted, non-obvious defect. [6](#0-5) [7](#0-6) 

---

### Impact Explanation

**Critical — Permanent freezing / destruction of user funds.**

Any user who calls `withdrawToNear` on whichever contract version sends the malformed precompile payload will have their ERC-20 tokens burned with no corresponding NEP-141 release on NEAR. The tokens are irrecoverably destroyed: the ERC-20 supply is reduced, the precompile silently fails, and no NEAR-side transfer is scheduled. There is no revert, no refund, and no on-chain signal of failure visible to the caller.

---

### Likelihood Explanation

**High.** Both `EvmErc20` and `EvmErc20V2` are production contracts deployed by the Aurora Engine factory for every bridged NEP-141 token. Any ordinary token holder who calls `withdrawToNear` on the incorrectly-encoding version triggers the loss. No special privileges, no front-running, and no external dependency is required — a single standard ERC-20 withdrawal call is sufficient.

---

### Recommendation

1. Align the precompile calldata encoding in `withdrawToNear` across both `EvmErc20` and `EvmErc20V2` to match the exact format the ExitToNear precompile parses.
2. Check the return value of the assembly `call` and revert if it returns `0`, so that a failed precompile invocation never silently follows a successful `_burn`.
3. Extract the encoding logic into a shared internal library or base contract (analogous to the Fairswap recommendation of a shared interface file) so future contract versions cannot diverge silently.

---

### Proof of Concept

1. Deploy `EvmErc20V2` for a NEP-141 token via the Aurora factory.
2. Bridge NEP-141 tokens to Aurora; user holds `N` ERC-20 units.
3. User calls `EvmErc20V2.withdrawToNear(recipientBytes, N)`.
4. `_burn(sender, N)` executes — user's ERC-20 balance drops to 0.
5. The assembly block calls the ExitToNear precompile with layout `[0x01][20-byte sender][32-byte amount][recipient]`.
6. If the precompile expects the V1 layout `[0x01][32-byte amount][recipient]`, it misparsed the first 20 bytes of `amount` as a sender address and the remaining bytes as the amount — the call fails or produces a garbage withdrawal.
7. `res` is never checked; execution returns normally.
8. User's `N` ERC-20 tokens are gone; their NEAR account receives nothing. [8](#0-7) [9](#0-8)
