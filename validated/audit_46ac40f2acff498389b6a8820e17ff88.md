### Title
Missing Zero-Address Validation for `reward_address` Causes Temporary Freezing of Staker Funds and Unclaimed Yield — (`File: src/staking/staking.cairo`, `src/pool/pool.cairo`)

---

### Summary

The `stake()` and `change_reward_address()` functions in the `Staking` contract, and the `enter_delegation_pool()` and `change_reward_address()` functions in the `Pool` contract, accept a `reward_address` parameter without validating it is non-zero. If a staker or pool member sets `reward_address` to the zero address, all subsequent reward-claiming and unstaking flows revert because the ERC20 `transfer` to the zero address panics. This temporarily freezes both unclaimed yield and staked principal until the staker corrects the address.

---

### Finding Description

In `Staking::stake()`, the only validation applied to `reward_address` is that it is not a registered token address:

```cairo
assert!(
    !self.does_token_exist(token_address: reward_address),
    "{}",
    GenericError::REWARD_ADDRESS_IS_TOKEN,
);
```

There is no `assert!(reward_address.is_non_zero(), ...)` guard. The same omission exists in `Staking::change_reward_address()`, `Pool::enter_delegation_pool()`, and `Pool::change_reward_address()`.

The stored `reward_address` is later used as the recipient in `send_rewards_to_staker()`:

```cairo
token_dispatcher.checked_transfer(recipient: reward_address, amount: amount.into());
```

OpenZeppelin Cairo ERC20's `_transfer` unconditionally asserts `!recipient.is_zero()` regardless of amount. Therefore, if `reward_address` is zero and `unclaimed_rewards_own > 0`, this call panics.

`send_rewards_to_staker()` is invoked unconditionally inside both `claim_rewards()` and `unstake_action()`:

```cairo
// unstake_action — called before returning principal
self.send_rewards_to_staker(:staker_address, ref :staker_info, :token_dispatcher);
```

A revert here blocks the entire `unstake_action()` execution, preventing the staker from recovering their principal.

The same pattern exists in `Pool::claim_rewards()`:

```cairo
let reward_token = IERC20Dispatcher { contract_address: STRK_TOKEN_ADDRESS };
reward_token.checked_transfer(recipient: reward_address, amount: rewards.into());
```

---

### Impact Explanation

- **`Staking::claim_rewards()`** reverts → unclaimed STRK yield is frozen.
- **`Staking::unstake_action()`** reverts → staked principal is frozen (the staker cannot withdraw even after the exit window expires).
- **`Pool::claim_rewards()`** reverts → pool member's unclaimed STRK yield is frozen.

The freeze persists until the affected party calls `change_reward_address()` with a valid non-zero address. Because `change_reward_address()` has no `unstake_time.is_none()` guard, the staker can self-rescue even mid-exit. The freeze is therefore **temporary**, matching the "Temporary freezing of funds" impact category.

---

### Likelihood Explanation

Any unprivileged staker or pool member can trigger this by passing `reward_address = ContractAddress::zero()` to `stake()` or `enter_delegation_pool()`, or by calling `change_reward_address(zero)` at any time. No privileged role is required. Accidental misconfiguration (e.g., an uninitialized address variable in a deployment script) is a realistic path. Likelihood is **low-medium**.

---

### Recommendation

Add a non-zero assertion for `reward_address` in all four entry points:

- `Staking::stake()` — after the existing token-address check
- `Staking::change_reward_address()` — after the existing token-address check
- `Pool::enter_delegation_pool()` — after the existing token-address check
- `Pool::change_reward_address()` — after the existing token-address check

```cairo
assert!(reward_address.is_non_zero(), "{}", GenericError::ZERO_ADDRESS);
```

The error constant `GenericError::ZERO_ADDRESS` already exists in the codebase (used in `add_token()`).

---

### Proof of Concept

1. Deploy the full system (`general_contract_system_deployment`).
2. Call `Staking::stake(reward_address: Zero::zero(), operational_address: <valid>, amount: <min_stake>)`.
3. Advance K epochs and trigger attestation so `unclaimed_rewards_own > 0`.
4. Call `Staking::unstake_intent()`.
5. Advance time past `exit_wait_window`.
6. Call `Staking::unstake_action(staker_address)` — **reverts** with `ERC20: transfer to 0` because `send_rewards_to_staker` attempts `checked_transfer(recipient: Zero::zero(), amount: rewards)`.
7. The staker's principal remains locked in the contract.
8. Staker calls `Staking::change_reward_address(<valid_address>)` — succeeds (no `unstake_time` guard).
9. `Staking::unstake_action(staker_address)` now succeeds and returns principal.
