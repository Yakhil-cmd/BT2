### Title
Unauthorized Transfer of Native Token Balances via Bank Precompile `transfer` Method — (`x/cronos/keeper/precompiles/bank.go`)

---

### Summary

The bank precompile's `transfer` handler accepts an arbitrary `sender` address as a call argument and passes it directly to `bankKeeper.SendCoins` without verifying that the calling contract is authorized to debit that address. Any unprivileged EVM contract can call `bankPrecompile.transfer(victimAddress, attackerAddress, amount)` and drain native tokens from the victim's Cosmos bank balance, provided the victim holds tokens of the denom `evm/<callerContractAddress>`.

---

### Finding Description

In `x/cronos/keeper/precompiles/bank.go`, the `TransferMethodName` case of `BankContract.Run()` unpacks `sender` from the ABI-encoded call arguments (`args[0]`) and uses it as the `from` address in a `bankKeeper.SendCoins` call:

```go
sender := args[0].(common.Address)
recipient := args[1].(common.Address)
...
from := sdk.AccAddress(sender.Bytes())
to   := sdk.AccAddress(recipient.Bytes())
...
denom := EVMDenom(contract.Caller())          // "evm/<callerAddress>"
amt   := sdk.NewCoin(denom, ...)
...
bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt))
``` [1](#0-0) 

There is **no check** that `contract.Caller() == sender`. The denom is scoped to the calling contract (`evm/<callerAddress>`), but the `from` address is fully attacker-controlled. Any EVM contract deployed at address `0xA` can therefore call:

```
address(0x64).call(abi.encodeWithSignature(
    "transfer(address,address,uint256)",
    victimAddress, attackerAddress, amount
));
```

and `bankKeeper.SendCoins` will move `evm/0xA` tokens from `victimAddress` to `attackerAddress` with no authorization from the victim.

The same root cause exists in the `burn` case: `recipient` (the address whose tokens are burned) is taken from `args[0]` with no check against `contract.Caller()`, allowing a contract to destroy any holder's native balance of its own denom. [2](#0-1) 

---

### Impact Explanation

**Critical — Unauthorized transfer/burn of precompile-controlled native assets.**

A victim who holds `evm/<attackerContract>` tokens in their Cosmos bank balance (obtained by converting CRC21 tokens to native via `ConvertCoinFromCRC21ToNative`) can have those tokens silently transferred to the attacker or burned, with no transaction signed by the victim. This is a direct, unauthorized balance change for precompile-controlled assets. [3](#0-2) 

---

### Likelihood Explanation

The bank precompile is deployed at the fixed address `0x0000000000000000000000000000000000000064` and is callable by any EVM contract. [4](#0-3) 

The attack requires only that a victim holds native tokens of the attacker's CRC21 contract denom — a realistic condition whenever users convert CRC21 tokens to native tokens for IBC bridging or other Cosmos-layer use. The attacker needs no special privilege, no leaked key, and no governance action.

---

### Recommendation

In the `TransferMethodName` case, verify that the calling contract is authorized to debit `sender`. The minimal fix is to require `contract.Caller()` to equal `sender`:

```go
if contract.Caller() != sender {
    return nil, errors.New("caller is not authorized to transfer from sender")
}
```

Apply the same guard to the `BurnMethodName` case (verify `contract.Caller() == recipient`). This mirrors the `require(msg.sender == module_address)` pattern already enforced in the CRC20/CRC21 contracts themselves. [5](#0-4) 

---

### Proof of Concept

1. Attacker deploys a malicious CRC21-compatible contract at address `0xA`.
2. Victim acquires `evm/0xA` native tokens (e.g., by calling `ConvertCoinFromCRC21ToNative` after receiving CRC21 tokens from `0xA`).
3. Attacker's contract calls the bank precompile at `0x64`:
   ```solidity
   IBankModule(0x64).transfer(victimAddress, attackerAddress, victimBalance);
   ```
4. `BankContract.Run` unpacks `sender = victimAddress`, computes `denom = "evm/0xA"`, and calls `bankKeeper.SendCoins(ctx, victimAddress, attackerAddress, coins)` — no signature or approval from the victim is required.
5. The victim's `evm/0xA` bank balance is zero; the attacker holds the full amount. [6](#0-5)
