### Title
No Safety Check for Identical Triples in `presign` - (`src/ecdsa/ot_based_ecdsa/presign.rs`)

### Summary

The OT-based ECDSA `presign` function accepts two Beaver triples (`triple0`, `triple1`) but performs no check that they are distinct. A caller — or a malicious participant who controls their own `PresignArguments` — can supply the same triple for both slots. All internal protocol assertions still pass, but the resulting presignature is cryptographically corrupted, producing signatures that fail standard ECDSA verification and permanently denying signing for honest parties.

### Finding Description

`presign()` in `src/ecdsa/ot_based_ecdsa/presign.rs` validates that both triples share the same threshold as the presign threshold, and that the participant list has no duplicates, but it never verifies that `triple0` and `triple1` are distinct triples: [1](#0-0) 

The only cross-triple check is a threshold equality test:

```rust
if args.threshold != args.triple0.1.threshold || args.threshold != args.triple1.1.threshold {
    return Err(...);
}
```

There is no check such as `triple0.1.big_a != triple1.1.big_a` (or any equivalent identity check on the public triple commitments). [2](#0-1) 

The `PresignArguments` struct exposes both triple slots as plain public fields with no invariant enforcement: [3](#0-2) 

When `triple0 == triple1`, the internal `do_presign` logic extracts:
- `k_i = args.triple0.0.a` and `a_i = args.triple1.0.a` → **`k_i = a_i`**
- `e_i = args.triple0.0.c` and `c_i = args.triple1.0.c` → **`e_i = c_i`** [4](#0-3) 

With `k = a` across all parties, the reconstructed values satisfy:
- `alpha = k + a = 2k`, `beta = x + b`
- The check `alpha·G == K + A` becomes `2k·G == 2k·G` ✓ — **passes silently**
- The check `beta·G == X + B` ✓ — **passes silently**
- The check `E == e·G` becomes `C == c·G` ✓ — **passes silently** [5](#0-4) [6](#0-5) 

The presign completes without error, but `sigma_i = alpha·x_i − beta·a_i + c_i` evaluates to `k·x_i` instead of the correct `r·k·x_i`. The resulting signature `s = k·H(m) + sigma = k·(H(m) + x)` fails standard ECDSA verification (which expects `s = k·(H(m) + r·x)`).

### Impact Explanation

**High — Corruption of presign outputs so honest parties accept unusable cryptographic outputs.**

All internal protocol assertions pass, so every honest participant concludes the presign succeeded. The `PresignOutput` they hold is structurally valid but cryptographically wrong. Every subsequent signing attempt using this presignature produces an invalid ECDSA signature that fails on-chain or off-chain verification. Signing is permanently denied for that presignature slot, and the presignature cannot be reused (by design), so the signing session is unrecoverably lost.

### Likelihood Explanation

**Medium.** The `presign` public API accepts two independent `(TripleShare, TriplePub)` arguments with no type-level or runtime distinction between them. A mistaken integrator can trivially pass the same triple twice (e.g., by reusing a cached triple object). A malicious participant who controls their own `PresignArguments` can deliberately supply identical triples to corrupt their sigma share, causing the coordinator to assemble an invalid aggregate signature and permanently blocking that signing session for all honest parties.

### Recommendation

Add an explicit identity check on the public triple commitments before entering the protocol:

```rust
// In presign(), after threshold checks:
if args.triple0.1.big_a == args.triple1.1.big_a
    && args.triple0.1.big_b == args.triple1.1.big_b
    && args.triple0.1.big_c == args.triple1.1.big_c
{
    return Err(InitializationError::BadParameters(
        "triple0 and triple1 must be distinct triples".to_string(),
    ));
}
```

Alternatively, assign each generated triple a unique session identifier (e.g., a hash of its transcript) stored in `TriplePub`, and require the two identifiers to differ at presign initialization.

pub fn presign(
    participants: &[Participant],
    me: Participant,
    args: PresignArguments,
) -> Result<impl Protocol<Output = PresignOutput>, InitializationError> {
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }
    // Spec 1.1
    if args.threshold.value() > participants.len() {
        return Err(InitializationError::ThresholdTooLarge {
            threshold: args.threshold.value(),
            max: participants.len(),
        });
    }

    // NOTE: We omit the check that the new participant set was present for
    // the triple generation, because presumably they need to have been present
    // in order to have shares.

    // Also check that we have enough participants to reconstruct shares.
    if args.threshold != args.triple0.1.threshold || args.threshold != args.triple1.1.threshold {
        return Err(InitializationError::BadParameters(
            "New threshold must match the threshold of both triples".to_string(),
        ));
    }

    let participants =
        ParticipantList::new(participants).ok_or(InitializationError::DuplicateParticipants)?;

    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    let ctx = Comms::new();
    let fut = do_presign(ctx.shared_channel(), participants, me, args);
    Ok(make_protocol(ctx, fut))
}
