### Title
Gateway `skip_stateful_validations` Bypasses `__validate__` for Invoke Transactions from Non-Deploy-Account Accounts - (File: crates/apollo_gateway/src/stateful_transaction_validator.rs)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator is intended to skip `__validate__` only when a `deploy_account` transaction is pending in the mempool (UX feature for simultaneous deploy+invoke). However, the actual check is `account_tx_in_pool_or_recent_block`, which returns `true` for **any** account with any pending transaction — not specifically a `deploy_account`. This allows an attacker to submit an invoke transaction with nonce=1 and an invalid/arbitrary signature from any account that has nonce=0 in state and any pending transaction in the mempool, bypassing the `__validate__` entry point entirely at the gateway admission stage.

### Finding Description

In `extract_state_nonce_and_run_validations`, the gateway calls `run_pre_validation_checks`, which calls `skip_stateful_validations`:

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            // ...
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                ...
        }
    }
    Ok(false)
}

When this returns `true`, `run_validate_entry_point` sets `execution_flags.validate = false`, which causes `StatefulValidator::perform_validations` to return `Ok(())` without ever calling `__validate__`: [2](#0-1) [3](#0-2) 

The comment in the code states: *"It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."* This reasoning is incorrect. `account_tx_in_pool_or_recent_block` checks: [4](#0-3) 

This returns `true` for **any** account with any transaction in the pool — including accounts deployed via `deploy_syscall` from another contract (which have `class_hash != 0` and `nonce = 0` in state) that have a legitimate invoke with nonce=0 in the mempool.

**Attack scenario:**
1. Account A is deployed via `deploy_syscall` (not via `deploy_account` tx): `class_hash != 0`, `nonce = 0` in state.
2. Account A submits a legitimate invoke with nonce=0 → passes `__validate__` → admitted to mempool.
3. Attacker submits an invoke with nonce=1 from Account A with an **invalid/arbitrary signature**.
4. `validate_nonce` passes: `0 <= 1 <= max_allowed_nonce_gap`. [5](#0-4) 
5. `skip_stateful_validations` returns `true` (account has a tx in pool, account_nonce=0, tx_nonce=1).
6. `__validate__` is **skipped entirely**.
7. The invalid invoke is admitted to the mempool without signature verification.

### Impact Explanation

The gateway admits an unauthorized invoke transaction — one with an invalid signature that would fail `__validate__` — into the mempool. This matches the "High" impact: **Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

Concrete consequences:
- **Mempool pollution**: Invalid transactions occupy mempool slots.
- **Batcher resource waste**: The batcher will execute and reject these transactions (the batcher always runs `__validate__` with `execution_flags.validate = true`).
- **Transaction replacement via fee escalation**: If the attacker submits the invalid nonce=1 invoke with a higher tip than the legitimate one, it can replace the legitimate transaction in the mempool. The legitimate transaction is then lost when the batcher rejects the invalid one.

### Likelihood Explanation

The condition is reachable by any unprivileged attacker who can observe the public mempool. Accounts deployed via `deploy_syscall` (e.g., counterfactual deployments, factory patterns) with nonce=0 and a pending transaction are a realistic and common pattern. The attacker only needs to observe a pending nonce=0 invoke from such an account and submit a competing nonce=1 transaction.

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a `deploy_account` transaction is pending for the sender address. The mempool should expose a dedicated `has_pending_deploy_account(address)` query, or the gateway should track the deploy_account tx hash and verify it is present in the mempool before skipping `__validate__`.

### Proof of Concept

1. Deploy Account A via `deploy_syscall` from another contract. Account A now has `class_hash != 0`, `nonce = 0` in state.
2. Submit a legitimate `RpcInvokeTransaction::V3` from Account A with `nonce = 0` and a valid signature. It passes `__validate__` and enters the mempool.
3. Construct a second `RpcInvokeTransaction::V3` from Account A with `nonce = 1`, arbitrary `calldata`, and a **random/invalid signature** (e.g., `[Felt::ONE, Felt::TWO]`).
4. Submit this transaction to the gateway's `POST /add_transaction` endpoint.
5. Observe: `validate_nonce` passes (nonce 1 is within `[0, max_allowed_nonce_gap]`); `skip_stateful_validations` returns `true` because `account_tx_in_pool_or_recent_block` returns `true`; `run_validate_entry_point` sets `validate = false` and returns `Ok(())` without calling `__validate__`.
6. The transaction is forwarded to the mempool and admitted. The invalid invoke now occupies a mempool slot for Account A at nonce=1 without any signature verification having occurred. [6](#0-5) [7](#0-6)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
