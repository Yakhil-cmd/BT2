### Title
Unbounded Native Resource Limit for Ethereum Transactions Enables Prover Resource Exhaustion - (File: `basic_bootloader/src/bootloader/transaction_flow/ethereum/validation_impl.rs`)

### Summary

In the Ethereum-style transaction validation path, the per-transaction native resource limit is set to `u64::MAX` rather than being derived from the transaction's gas price and gas limit. This is the direct analog of the reported gRPC `i32::MAX` message-length vulnerability: an effective ceiling of "infinity" allows a single attacker-submitted transaction to consume the entire block's native resource budget (up to `MAX_NATIVE_COMPUTATIONAL = 2^35` RISC-V cycles) without paying proportionally, forcing the prover to execute an unbounded amount of work.

### Finding Description

`create_resources_for_tx` in the Ethereum validation path constructs the per-transaction resource envelope:

```rust
// basic_bootloader/src/bootloader/transaction_flow/ethereum/validation_impl.rs
let native_limit =
    <<S as zk_ee::system::SystemTypes>::Resources as Resources>::Native::from_computational(
        u64::MAX,
    );
let main_resources = S::Resources::from_ergs_and_native(Ergs(ergs), native_limit);
``` [1](#0-0) 

The ZK-path counterpart (`basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`) correctly derives the native limit from `gasPrice / nativePrice * gasLimit`, so users pay proportionally for proving work. The Ethereum path skips this derivation entirely. [2](#0-1) 

The block-level guard `MAX_NATIVE_COMPUTATIONAL = 1 << 35` exists:

```rust
pub const MAX_NATIVE_COMPUTATIONAL: u64 = 1 << 35;
``` [3](#0-2) 

and is checked in `check_for_block_limits`: [4](#0-3) 

However, this check fires only **after** the transaction has been fully executed by the prover. The prover must execute every RISC-V cycle before discovering the limit was exceeded. Because the per-transaction native limit is `u64::MAX`, nothing stops a single transaction from consuming all `2^35` cycles in one shot.

Native resources are not a 1:1 proxy for EVM gas. The double-accounting model explicitly acknowledges that native resource consumption can exceed EVM gas consumption (`deltaGas` reconciliation), meaning operations such as cold storage reads with Merkle proof verification, precompile delegation, and keccak/blake2s hashing consume native resources beyond what EVM gas charges. An attacker can craft a transaction that maximises this gap. [5](#0-4) 

### Impact Explanation

- **Prover DoS**: A single Ethereum-style transaction can force the prover to execute up to `2^35` RISC-V cycles (the full block native budget) without the sender paying for that proving work. The block-level check invalidates the transaction after the fact, but the prover has already spent the cycles.
- **Sequencer stall**: Because the forward runner also executes the same code path, the sequencer's forward execution is similarly forced to process the full native budget for a single transaction before rejecting it.
- **No proportional payment**: The ZK path charges native resources proportional to `gasPrice * gasLimit / nativePrice`. The Ethereum path charges nothing for native resources beyond EVM gas, creating a free-rider attack surface.

### Likelihood Explanation

- Any unprivileged transaction sender can submit an Ethereum-style transaction (type 0, 1, 2, or 4) with a high gas limit and low gas price.
- Operations that are cheap in EVM gas but expensive in native resources (cold storage reads, precompile calls, keccak/blake2s hashing) are well-documented and easily targeted.
- No special access, leaked keys, or governance majority is required.

### Recommendation

**Short term**: Derive the per-transaction native resource limit in the Ethereum path using the same formula as the ZK path:
```
nativeLimit = gasLimit * (gasPrice / nativePrice)
```
This mirrors the fix recommended in the original report: replace the effectively-infinite limit with a value proportional to what the user has paid.

**Long term**: Unify the resource-creation logic between the Ethereum and ZK validation paths so that any future change to native resource accounting is applied consistently. Add an assertion or compile-time check that the per-transaction native limit never exceeds `MAX_NATIVE_COMPUTATIONAL`.

### Proof of Concept

1. Craft an EIP-1559 (type 2) transaction with `gasLimit = MAX_BLOCK_GAS_LIMIT` and `maxFeePerGas = baseFee` (minimum accepted).
2. Fill the transaction body with a tight loop of cold `SLOAD` opcodes targeting distinct storage slots (each cold read triggers a Merkle proof verification in native resources, costing far more native cycles than the 2100 EVM gas charged).
3. Submit the transaction. The Ethereum validation path sets `native_limit = u64::MAX`; the transaction executes until EVM gas is exhausted.
4. Because each cold `SLOAD` consumes `COLD_NEW_STORAGE_READ_NATIVE_COST` native cycles beyond its EVM gas cost, the total native consumption can reach `MAX_NATIVE_COMPUTATIONAL = 2^35` before EVM gas runs out, forcing the prover to execute the full block native budget for a single transaction.
5. The block-level check in `check_for_block_limits` invalidates the transaction only after all cycles have been spent. [6](#0-5) [7](#0-6)

fn create_resources_for_tx<S: EthereumLikeTypes>(
    gas_limit: u64,
    is_deployment: bool,
    calldata_len: u64,
    calldata_tokens: u64,
) -> Result<ResourcesForEthereumTx<S>, TxError> {
    let mut intrinsic_overhead = TX_INTRINSIC_GAS;
    if is_deployment {
        if calldata_len > MAX_INITCODE_SIZE as u64 {
            return Err(TxError::Validation(CreateInitCodeSizeLimit));
        }
        intrinsic_overhead = intrinsic_overhead.saturating_add(DEPLOYMENT_TX_EXTRA_INTRINSIC_GAS);
        let initcode_gas_cost =
            evm_interpreter::gas_constants::INITCODE_WORD_COST * calldata_len.div_ceil(32);
        intrinsic_overhead = intrinsic_overhead.saturating_add(initcode_gas_cost);
    }
    intrinsic_overhead =
        intrinsic_overhead.saturating_add(calldata_tokens.saturating_mul(CALLDATA_TOKEN_GAS_COST));

    if intrinsic_overhead > gas_limit {
        Err(TxError::Validation(
            InvalidTransaction::OutOfGasDuringValidation,
        ))
    } else {
        let gas_limit_for_tx = gas_limit - intrinsic_overhead;
        let ergs = gas_limit_for_tx.saturating_mul(ERGS_PER_GAS); // we checked at the very start that gas_limit * ERGS_PER_GAS doesn't overflow
        let native_limit =
            <<S as zk_ee::system::SystemTypes>::Resources as Resources>::Native::from_computational(
                u64::MAX,
            );
        let main_resources = S::Resources::from_ergs_and_native(Ergs(ergs), native_limit);

        Ok(ResourcesForEthereumTx { main_resources })
    }
