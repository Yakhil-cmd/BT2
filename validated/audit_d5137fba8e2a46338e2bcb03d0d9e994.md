Audit Report

## Title
ckBTC Permanently Burned Without Reimbursement on `AmountTooLow`/`DustOutput` at Batch Submission — (`rs/bitcoin/ckbtc/minter/src/lib.rs`)

## Summary
The ckBTC minter burns a user's ckBTC at `retrieve_btc` call time before any Bitcoin transaction is built. If Bitcoin fees spike before the minter's timer processes the batch, `build_unsigned_transaction` returns `BuildTxError::AmountTooLow` or `BuildTxError::DustOutput`, and the request is finalized via `remove_retrieve_btc_request(..., FinalizedStatus::AmountTooLow, ...)` with no reimbursement issued. The user's ckBTC is permanently destroyed and zero BTC is sent. The `InvalidTransaction` error path, by contrast, calls `reimburse_canceled_requests`, proving the reimbursement infrastructure exists but is simply not invoked for these two error variants.

## Finding Description
**Root cause — asymmetric error handling in `submit_pending_requests`:**

In `rs/bitcoin/ckbtc/minter/src/updates/retrieve_btc.rs` lines 209–210, `burn_ckbtcs` destroys the full withdrawal amount from the caller's account before the request enters the pending queue. A `reimbursement_account` is stored in the `RetrieveBtcRequest` struct (lines 218–221), confirming the design anticipated reimbursement paths.

In `rs/bitcoin/ckbtc/minter/src/lib.rs`, the timer-driven `submit_pending_requests` re-estimates fees at batch-build time. Three `BuildTxError` variants are handled:

- `InvalidTransaction` (lines 400–410): calls `reimburse_canceled_requests`, which schedules a mint back to `reimbursement_account`.
- `AmountTooLow` (lines 412–434): calls only `remove_retrieve_btc_request(..., FinalizedStatus::AmountTooLow, ...)` — no reimbursement.
- `DustOutput` (lines 436–467): same — calls only `remove_retrieve_btc_request(..., FinalizedStatus::AmountTooLow, ...)` for the offending request — no reimbursement.

`remove_retrieve_btc_request` in `rs/bitcoin/ckbtc/minter/src/state/audit.rs` lines 67–84 only records a `RemovedRetrieveBtcRequest` event and pushes to `finalized_requests`. It contains no reimbursement scheduling. The `reimburse_withdrawal` function (audit.rs lines 240–265), which schedules the actual mint-back, is never reached for these two error paths.

**Why existing guards are insufficient:** The `fee_based_retrieve_btc_min_amount` guard at call time uses the fee estimate at that moment. There is no lock or reservation preventing the fee estimate from changing between request acceptance and batch processing. The window is unbounded — it spans however long until the minter's next timer tick and batch attempt.

## Impact Explanation
This is a permanent, irreversible loss of user ckBTC funds with no corresponding BTC output — a ledger conservation violation. The ckBTC ledger records a burn; the Bitcoin network records nothing. The `reimbursement_account` stored in the request is never consulted. This matches the allowed High impact: **"Significant Chain Fusion, ck-token, ledger security impact with concrete user or protocol harm"** — specifically, permanent loss of in-scope chain-key token assets triggered without any privileged access.

## Likelihood Explanation
Bitcoin mempool fees are historically volatile (10–100× spikes during Ordinals waves, halving periods, etc.). No attacker action is required — ordinary network congestion is sufficient. Any user who submits a `retrieve_btc` request near the minimum threshold during a low-fee period is at risk if fees spike before the next minter timer tick. The condition is repeatable and requires no special privileges.

## Recommendation
Apply the same reimbursement logic used for `InvalidTransaction` to both `AmountTooLow` and `DustOutput` branches. Replace the bare `remove_retrieve_btc_request` calls with `reimburse_canceled_requests`, passing a new `WithdrawalReimbursementReason` variant (e.g., `AmountTooLow`). The `reimbursement_account` field is already present in every `RetrieveBtcRequest`, so no structural change to the request type is needed. A small reimbursement fee (analogous to the one charged for `InvalidTransaction`) may be deducted to cover minter overhead.

## Proof of Concept
1. Bitcoin mainnet fees are low (e.g., 5 sat/vbyte); `fee_based_retrieve_btc_min_amount` computes to 100,000 sat.
2. User calls `retrieve_btc` with `amount = 110,000 sat`. The guard passes; `burn_ckbtcs` destroys 110,000 ckBTC. The request enters the pending queue with `reimbursement_account = caller`.
3. Before the minter's next timer tick, a fee spike pushes the median fee to 800 sat/vbyte.
4. `submit_pending_requests` calls `build_unsigned_transaction`. The total fee for a 1-input/2-output transaction at 800 sat/vbyte exceeds 110,000 sat → `BuildTxError::AmountTooLow`.
5. The `AmountTooLow` branch executes `remove_retrieve_btc_request(..., FinalizedStatus::AmountTooLow, ...)`. No call to `reimburse_canceled_requests`.
6. The user queries `retrieve_btc_status(block_index)` and receives `FinalizedStatus::AmountTooLow`. Their 110,000 ckBTC is permanently destroyed; they received 0 BTC; no reimbursement is ever issued.

A deterministic integration test can reproduce this by: (a) initializing the minter with a low fee rate, (b) submitting a `retrieve_btc` request just above the minimum, (c) updating the fee estimate to a high value before the timer fires, (d) triggering `submit_pending_requests`, and (e) asserting that `retrieve_btc_status` returns `AmountTooLow` while the user's ckBTC balance has not been restored.
