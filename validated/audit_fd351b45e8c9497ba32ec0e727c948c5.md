### Title
Gateway `skip_stateful_validations` Admits Arbitrarily-Signed Invoke Transactions via Overly Broad Mempool Presence Check — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

`skip_stateful_validations` skips the `__validate__` entry-point call (signature verification) for any invoke transaction whose nonce is 1 and whose sender's on-chain nonce is 0, whenever `account_tx_in_pool_or_recent_block` returns `true`. That helper returns `true` for **any** transaction from the account in the pool — not only a `deploy_account`. An attacker who knows that a target account has *any* pending transaction in the mempool can submit a nonce-1 invoke with an arbitrary signature, bypass gateway signature verification, and have the transaction admitted to the mempool. Combined with fee escalation, the attacker can silently evict the account owner's legitimate nonce-1 transaction.

---

### Finding Description

`skip_stateful_validations` is the gateway's UX shortcut that lets a user broadcast a `deploy_account` (nonce 0) and an `invoke` (nonce 1) simultaneously, before the account is deployed on-chain. Because the account contract does not yet exist, calling `__validate__` on the invoke would fail, so the gateway skips it when it can confirm a `deploy_account` is pending.

The confirmation check is:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs  lines 437-456
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
    let account_address = tx.sender_address();
    // We verify that a deploy_account transaction exists for this account. It is sufficient
    // to check if the account exists in the mempool since it means that either it has a
    // deploy_account transaction or transactions with future nonces that passed validations.
    return mempool_client
        .account_tx_in_pool_or_recent_block(tx.sender_address())
        .await
        ...
}
``` [1](#0-0) 

`account_tx_in_pool_or_recent_block` is implemented as:

```rust
// crates/apollo_mempool/src/mempool.rs  lines 697-700
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [2](#0-1) 

`tx_pool.contains_account` returns `true` for **any** transaction type (invoke, declare, deploy_account) from that address. The comment's reasoning — "either it has a deploy_account transaction or transactions with future nonces that passed validations" — is incorrect: a future-nonce invoke that was itself admitted under the same skip-validate shortcut (or any other pooled transaction) satisfies the check without any `deploy_account` being present.

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` sets `validate: false` in the execution flags and the blockifier's `StatefulValidator` never calls `__validate__`:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs  lines 311-312
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [3](#0-2) 

The `ExecutionFlags` default and `new_for_sequencing` both set `validate: true`, so the **batcher** will call `__validate__` when it later executes the transaction. A transaction with a wrong signature will therefore fail at execution time. However, the gateway has already admitted it to the mempool, and the mempool's fee-escalation path will have evicted the account owner's legitimate nonce-1 transaction if the attacker supplied a higher fee.

---

### Impact Explanation

**High — Mempool/gateway admission accepts an invalid (unsigned) transaction before sequencing.**

1. The gateway admits a nonce-1 invoke with an arbitrary signature without calling `__validate__`.
2. If the account owner's legitimate nonce-1 transaction is already in the pool, the attacker's higher-fee transaction evicts it via fee escalation (`remove_replaced_tx`).
3. The batcher later executes the attacker's transaction, `__validate__` fails, the transaction is rejected, and the nonce is **not** incremented (the transactional state is aborted). However, the account owner's legitimate transaction is permanently gone from the pool and must be resubmitted.
4. The attack can be repeated on every new nonce-1 submission, creating a sustained DoS against any account in the deploy-account + invoke UX flow.

---

### Likelihood Explanation

Medium-high. The preconditions are:

- The target account's on-chain nonce is 0 (i.e., it is in the process of being deployed — the most common new-account flow).
- The account has **any** transaction in the mempool (the deploy_account itself satisfies this).
- The attacker submits a nonce-1 invoke with a fee above the victim's nonce-1 transaction (or simply submits first if no nonce-1 transaction exists yet).

All three conditions are routinely satisfied during the standard `deploy_account` + `invoke` UX flow that the feature was designed to support.

---

### Recommendation

Replace the overly broad `account_tx_in_pool_or_recent_block` check with a check that specifically verifies the presence of a **`deploy_account`** transaction for the sender address in the pool. The mempool should expose a dedicated query such as `has_deploy_account_in_pool(address) -> bool`, or the gateway should inspect the transaction type of the pooled entry before deciding to skip `__validate__`.

---

### Proof of Concept

1. Account A does not yet exist on-chain (`account_nonce = 0`).
2. Account owner broadcasts `deploy_account` (nonce 0) and `invoke` (nonce 1, valid signature, fee = F).
3. Both pass gateway stateless validation. The `deploy_account` is added to the pool; `account_tx_in_pool_or_recent_block(A)` now returns `true`.
4. The owner's `invoke` (nonce 1) also passes because `skip_stateful_validations` returns `true` — `__validate__` is skipped.
5. **Attacker** submits `invoke` (nonce 1, **wrong/arbitrary signature**, fee = F + ε).
6. Gateway stateless checks pass (signature length is within bounds, resource bounds are valid).
7. `validate_by_mempool` passes: nonce 1 ≥ account nonce 0, fee escalation threshold met.
8. `skip_stateful_validations` returns `true` again — `account_tx_in_pool_or_recent_block(A)` is still `true`.
9. `__validate__` is **not called**. The attacker's transaction is forwarded to `mempool.add_tx`.
10. Fee escalation evicts the owner's nonce-1 transaction (`remove_replaced_tx`).
11. Batcher executes `deploy_account` (nonce 0) — account A is deployed.
12. Batcher executes attacker's `invoke` (nonce 1) with `validate: true` — `__validate__` fails, transaction is rejected, nonce unchanged.
13. The owner's legitimate nonce-1 transaction is permanently lost from the pool. [4](#0-3) [5](#0-4) [6](#0-5) [7](#0-6)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
