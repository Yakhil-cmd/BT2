### Title
`protocolUnpausePool` Leaves Pool Permanently Stuck in Admin-Paused State Instead of Restoring Active State ŌĆö (`metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary

`protocolUnpausePool` calls `setPause(1)` (admin-paused) instead of `setPause(0)` (active). After the protocol owner "unpauses" a protocol-paused pool, the pool remains paused and requires a second, independent action from the pool admin to restore swap functionality. If the pool admin is unresponsive, a dead address, or a bricked multisig, swaps are permanently broken.

### Finding Description

The pause state machine has three levels:

| Level | Meaning |
|---|---|
| 0 | Active |
| 1 | Admin-paused |
| 2 | Protocol-paused |

`protocolPausePool` correctly transitions from 0 or 1 ŌåÆ 2. `protocolUnpausePool` is supposed to reverse that, but instead of calling `setPause(0)` it calls `setPause(1)`:

```solidity
// MetricOmmPoolFactory.sol L399-403
function protocolUnpausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 2) revert InvalidPauseTransition(cur, 1);
    IMetricOmmPoolFactoryActions(pool).setPause(1);   // ŌåÉ sets to admin-paused, NOT active
}
``` [1](#0-0) 

The admin-side `unpausePool` enforces `cur == 1` as a precondition before it can set to 0:

```solidity
// MetricOmmPoolFactory.sol L467-471
function unpausePool(address pool) external override nonReentrant onlyPoolAdmin(pool) {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 1) revert InvalidPauseTransition(cur, 0);
    IMetricOmmPoolFactoryActions(pool).setPause(0);
}
``` [2](#0-1) 

So the only path from protocol-paused (2) back to active (0) is a mandatory two-step sequence: protocol owner calls `protocolUnpausePool` (ŌåÆ 1), then pool admin calls `unpausePool` (ŌåÆ 0). The protocol owner has no unilateral path to restore the pool.

The `swap` function enforces `whenNotPaused`, which reverts on any non-zero pause level:

```solidity
// MetricOmmPool.sol L643-645
function _checkNotPaused() internal view {
    if (pauseLevel != 0) revert PoolPaused();
}
``` [3](#0-2) 

### Impact Explanation

After `protocolUnpausePool` executes, `pauseLevel == 1`. All calls to `swap` revert with `PoolPaused`. If the pool admin is a dead address, a compromised multisig, or simply unresponsive, the pool is permanently stuck: swaps are broken, LPs cannot earn fees, and traders cannot use the pool. The protocol owner has no further lever ŌĆö calling `protocolUnpausePool` again reverts because `cur == 1 != 2`. The only remaining option is `protocolPausePool` (which re-pauses to level 2), creating an infinite loop that never reaches level 0.

### Likelihood Explanation

The trigger is a valid, privileged-but-expected action: the protocol owner pausing and then attempting to unpause a pool. Pool admins being EOAs that lose keys, multisigs that lose quorum, or contracts that lack an `unpausePool` call path are realistic scenarios. The protocol owner has no way to detect this failure mode before it occurs.

### Recommendation

Change `protocolUnpausePool` to call `setPause(0)` directly, restoring the pool to active state:

```solidity
function protocolUnpausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 2) revert InvalidPauseTransition(cur, 0);
    IMetricOmmPoolFactoryActions(pool).setPause(0);  // restore to active
}
```

If the two-step handoff is intentional design, it must be explicitly documented and the function should be renamed (e.g., `protocolHandBackToAdmin`) to avoid the misleading implication that calling it restores swap functionality.

### Proof of Concept

1. Pool is active (`pauseLevel == 0`).
2. Protocol owner calls `protocolPausePool(pool)` ŌåÆ `pauseLevel == 2`.
3. Protocol owner calls `protocolUnpausePool(pool)` ŌåÆ `setPause(1)` ŌåÆ `pauseLevel == 1`.
4. Any call to `pool.swap(...)` reverts with `PoolPaused`.
5. Pool admin address is a dead EOA. `unpausePool` is uncallable.
6. Protocol owner calls `protocolUnpausePool(pool)` again ŌåÆ reverts: `cur == 1 != 2`.
7. Pool is permanently stuck. All swap functionality is broken. [4](#0-3) [5](#0-4)
