### Title
Missing Threshold Validation in `ckd` Allows Coordinator to Corrupt CKD Output with Insufficient Participants - (File: src/confidential_key_derivation/protocol.rs)

### Summary
The `ckd` function in `src/confidential_key_derivation/protocol.rs` does not accept a `threshold` parameter and therefore cannot validate that the supplied `participants` list contains at least `threshold` members. A malicious coordinator (or a buggy caller) can invoke `ckd` with fewer participants than the threshold established during keygen, causing the Lagrange interpolation to reconstruct a different scalar than the master secret key, producing a cryptographically incorrect CKD output that honest parties accept as valid.

### Finding Description

The `ckd` public entry point validates only four conditions before proceeding: [1](#0-0) 

It checks `participants.len() >= 2`, absence of duplicates, presence of `me`, and presence of `coordinator`. It does **not** check `participants.len() >= threshold`.

Inside `compute_signature_share`, each participant computes a Lagrange coefficient relative to the caller-supplied `participants` list: [2](#0-1) 

The correctness of the CKD output depends entirely on the Lagrange coefficients summing to reconstruct the master secret. This is only guaranteed when `|participants| >= threshold`. If `|participants| < threshold`, the coefficients reconstruct a different (wrong) scalar, and the coordinator aggregates a corrupted derived key: [3](#0-2) 

The `KeygenOutput` struct carries no threshold field, so `ckd` has no internal means to recover the threshold: [4](#0-3) 

Compare with every other protocol entry point in the library. `assert_key_invariants` (keygen/reshare) enforces `threshold >= 2` and `threshold <= N`: [5](#0-4) 

`assert_reshare_keys_invariants` enforces `|intersection| >= old_threshold`: [6](#0-5) 

`assert_sign_inputs` (FROST signing) enforces `threshold <= participants.len()`: [7](#0-6) 

`ckd` is the only protocol entry point that accepts a participant list with no lower-bound check against the threshold.

### Impact Explanation

When `|participants| < threshold`, the Lagrange interpolation over the supplied subset does not reconstruct the master secret key `msk`. The coordinator outputs `CKDOutput { big_y, big_c }` where `big_c` encodes `H(pk, app_id) * wrong_scalar + app_pk * y` instead of `H(pk, app_id) * msk + app_pk * y`. The TEE that calls `ckd_output.unmask(app_sk)` recovers an incorrect confidential key. This is a corruption of CKD output: honest parties (the coordinator and any downstream TEE consumer) accept an unusable or attacker-influenced derived secret. This maps to the **High** allowed impact: *Corruption of CKD outputs so honest parties accept unusable cryptographic outputs*.

### Likelihood Explanation

The `ckd` function is a public API. Any caller who controls the `participants` slice — including the coordinator role — can pass a list shorter than the threshold. No cryptographic capability is required; it is a straightforward API misuse. Because `KeygenOutput` does not embed the threshold, even an honest but careless integrator can accidentally pass a subset smaller than threshold (e.g., only online participants at the time of the call), silently receiving a wrong derived key with no error returned.

### Recommendation

Add a `threshold: impl Into<ReconstructionLowerBound>` parameter to `ckd` and insert the following guard after the existing participant-count check:

```rust
if participants.len() < threshold.value() {
    return Err(InitializationError::NotEnoughParticipantsForNewThreshold {
        threshold: threshold.value(),
        participants: participants.len(),
    });
}
```

This mirrors the pattern already used in `assert_reshare_keys_invariants` and `assert_key_invariants`. [8](#0-7) 

### Proof of Concept

1. Run `keygen` with 5 participants and `threshold = 3`. All participants receive valid `KeygenOutput` values whose private shares are a degree-2 Shamir sharing of `msk`.
2. Call `ckd` with only 2 of those participants (e.g., participants `[P1, P2]`). The call passes all current validations.
3. Each participant computes `lambda_i` relative to the 2-element list `{P1, P2}`. These Lagrange coefficients reconstruct a degree-1 polynomial evaluated at 0, which is a different scalar than `msk`.
4. The coordinator aggregates the two contributions and outputs `CKDOutput` encoding `H(pk, app_id) * wrong_scalar + app_pk * y`.
5. The TEE calls `unmask(app_sk)` and recovers `H(pk, app_id) * wrong_scalar`, which differs from the expected `H(pk, app_id) * msk`. The derived confidential key is silently corrupted with no error surfaced to any participant.

pub fn assert_key_invariants(
    participants: &[Participant],
    me: Participant,
    threshold: impl Into<ReconstructionLowerBound>,
) -> Result<ParticipantList, InitializationError> {
    let threshold = usize::from(threshold.into());
    // need enough participants
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }

    // Step 1.1
    // validate threshold
    if threshold > participants.len() {
        return Err(InitializationError::ThresholdTooLarge {
            threshold,
            max: participants.len(),
        });
    }
    // Step 1.1
    if threshold < 2 {
        return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
    }

    // ensure uniqueness of participants in the participant list
    let participants =
        ParticipantList::new(participants).ok_or(InitializationError::DuplicateParticipants)?;

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }
    Ok(participants)
}
