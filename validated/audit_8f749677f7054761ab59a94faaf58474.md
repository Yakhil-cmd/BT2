### Title
Zero-reward `claim_rewards` updates state checkpoints and emits misleading events - (File: src/pool/pool.cairo)

### Summary
The `claim_rewards` function in `pool.cairo` performs no zero-reward guard before updating the pool member's `reward_checkpoint` and `entry_to_claim_from`, writing to storage, executing an ERC20 transfer of zero, and emitting a `PoolMemberRewardClaimed` event. Any pool member or their reward address can call this repeatedly with zero accrued rewards, consuming unbounded gas and polluting the event log with zero-amount reward events — a direct analog to the zero-`borrowFresh`/`repayBorrowFresh` pattern in the reference report.

### Finding Description
In `src/pool/pool.cairo`, `claim_rewards` (lines 335–377) executes the following unconditionally, even when `rewards == 0`:

1. **State update** — `pool_member_info.reward_checkpoint = until_checkpoint` and `pool_member_info.entry_to_claim_from = updated_entry_to_claim_from` are written to storage regardless of whether any rewards were earned.
2. **Storage write** — `self.write_pool_member_info(:pool_member, :pool_member_info)` is called unconditionally.
3. **Zero ERC20 transfer** — `reward_token.checked_transfer(recipient: reward_address, amount: rewards.into())` is called with `amount = 0`.
4. **Misleading event** — `Events::PoolMemberRewardClaimed { pool_member, reward_address, amount: rewards }` is emitted with `amount = 0`.

The `reward_checkpoint` field is the direct analog of `borrowIndex` in the reference report: it is an accounting cursor that advances even when no value changes hands.

The existing test `test_claim_rewards_no_rewards` (pool tests, line 582) confirms the function succeeds and returns `0` when no rewards exist, proving the path is reachable and does not revert.

The same pattern exists in `src/staking/staking.cairo` `claim_rewards` (lines 411–431) via `send_rewards_to_staker` (lines 1614–1629): when `unclaimed_rewards_own == 0`, `claim_from_reward_supplier` is called with `amount = 0`, `checked_transfer` is called with `amount = 0`, and `StakerRewardClaimed` is emitted with `amount = 0`.

### Impact Explanation
**Medium — Griefing / Unbounded gas consumption.**

A pool member or their reward address can call `claim_rewards` in a tight loop (or via an automated contract) with zero accrued rewards. Each call:
- Performs two storage writes (pool member info fields).
- Executes an external ERC20 `transfer(0)` call.
- Emits a `PoolMemberRewardClaimed(amount=0)` event.

This wastes gas for the caller and for any off-chain indexer processing the event stream, and pollutes reward accounting logs with zero-value entries that are indistinguishable from legitimate claims. There is no direct theft of funds, but the protocol's event integrity and gas efficiency are damaged.

### Likelihood Explanation
**Medium.** The access control (`caller == pool_member || caller == reward_address`) limits the call surface to the pool member and their designated reward address. However:
- The reward address is an arbitrary address set by the pool member; it could be a contract that automatically re-invokes `claim_rewards`.
- A pool member who sets their reward address to a shared or automated contract (e.g., a DAO treasury) exposes themselves to repeated zero-reward calls from any participant of that contract.
- No special privilege, leaked key, or external dependency is required — only a valid pool membership.

### Recommendation
Add an early-return guard after computing `rewards` in both `pool.cairo` and `staking.cairo`:

**`src/pool/pool.cairo` — `claim_rewards`:**
```cairo
// After: rewards += pool_member_info._unclaimed_rewards_from_v0;
if rewards.is_zero() {
    return rewards;
}
```

**`src/staking/staking.cairo` — `send_rewards_to_staker`:**
```cairo
// Before: claim_from_reward_supplier(...)
if amount.is_zero() {
    return;
}
```

This mirrors the recommendation in the reference report: exit early after checks when the operative amount is zero, avoiding unnecessary state writes, external calls, and event emissions.

### Proof of Concept
1. Deploy the system and stake with a pool enabled.
2. A pool member calls `enter_delegation_pool` — no rewards have accrued yet.
3. The pool member (or their reward address) calls `claim_rewards(pool_member)` immediately.
4. `calculate_rewards` returns `0`; `_unclaimed_rewards_from_v0 == 0`; total `rewards == 0`.
5. Despite `rewards == 0`, the function:
   - Writes updated `reward_checkpoint` and `entry_to_claim_from` to storage.
   - Calls `STRK_TOKEN.transfer(reward_address, 0)`.
   - Emits `PoolMemberRewardClaimed { pool_member, reward_address, amount: 0 }`.
6. Steps 3–5 can be repeated indefinitely, each time consuming gas and emitting a zero-amount event.

The existing test at `src/pool/tests/test.cairo:582` (`test_claim_rewards_no_rewards`) already demonstrates this path succeeds without reverting, confirming the vulnerability is reachable in production.
