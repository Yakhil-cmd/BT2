### Title
Missing Lower-Bound Threshold Validation in FROST Signing Allows Corrupted Sign Outputs — (File: `src/frost/mod.rs`)

---

### Summary

The `assert_sign_inputs` and `presign` functions in `src/frost/mod.rs` validate that the `threshold` does not exceed the participant count, but omit the lower-bound check (`threshold < 2`) that is explicitly enforced in `assert_key_invariants` in `src/dkg.rs`. A caller passing `threshold = 0` or `threshold = 1` to the FROST signing or presigning entry points receives no error and proceeds into the protocol, producing cryptographically incorrect outputs.

---

### Finding Description

`assert_key_invariants` in `src/dkg.rs` enforces both bounds on the threshold: [1](#0-0) 

```rust
if threshold > participants.len() { ... ThresholdTooLarge ... }
if threshold < 2 { ... ThresholdTooSmall ... }
```

`assert_sign_inputs` in `src/frost/mod.rs` only enforces the upper bound: [2](#0-1) 

```rust
if threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
```

The lower-bound check (`threshold < 2`) is entirely absent. The same omission exists in `presign`: [3](#0-2) 

```rust
if args.threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
```

`ReconstructionLowerBound` is a plain newtype with no internal validation: [4](#0-3) 

Any `usize` value, including `0` and `1`, is accepted without error.

---

### Impact Explanation

In FROST, the threshold governs Lagrange interpolation during signing. If `threshold = 1` is supplied, each participant's Lagrange coefficient is computed as if only one share is needed to reconstruct the secret, yielding a coefficient of `1` for the sole participant. Because the key was generated with `threshold ≥ 2` (enforced by DKG), the interpolation basis is inconsistent: the resulting partial signatures aggregate to a value that does not correspond to the actual secret key, producing an invalid, unusable final signature. If `threshold = 0`, the subtraction `threshold - 1` used to derive the polynomial degree in related code paths can underflow or produce a zero-degree polynomial, causing protocol failure.

This matches: **High — Corruption of sign outputs so honest parties accept unusable cryptographic outputs.**

---

### Likelihood Explanation

The `threshold` parameter is supplied directly by each library caller at signing time. A misconfigured integration, a malicious participant who controls their own call site, or a coordinator that advertises a manipulated threshold to other participants can trivially pass `threshold = 1` without triggering any error. The inconsistency between the DKG guard and the signing guard makes accidental misconfiguration likely in real deployments.

---

### Recommendation

Add the same lower-bound guard to `assert_sign_inputs` and `presign` that already exists in `assert_key_invariants`:

```rust
// In assert_sign_inputs and presign:
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

This mirrors the existing pattern in `src/dkg.rs` at lines 580–582 and closes the gap between the DKG and signing validation paths.

---

### Proof of Concept

1. Run DKG with `participants = [P0, P1, P2]`, `threshold = 2` → produces valid `KeygenOutput` for each participant.
2. Call `presign` or the FROST `sign` entry point with the same participants but `threshold = 1`.
3. Observe: no `InitializationError` is returned; the protocol proceeds.
4. The Lagrange coefficients are computed for a 1-of-N scheme instead of 2-of-3; the aggregated signature fails verification against the public key produced in step 1.
5. Contrast with calling `keygen` with `threshold = 1`: `assert_key_invariants` immediately returns `ThresholdTooSmall { threshold: 1, min: 2 }`. [5](#0-4) [6](#0-5)
