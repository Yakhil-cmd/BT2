### Title
Missing Zero-Address Recipient Check in `ExitToEthereum` Precompile Allows Permanent Fund Loss - (File: engine-precompiles/src/native.rs)

### Summary
The `ExitToEthereum` precompile (`ExitToEthereum::run`) accepts a user-supplied Ethereum recipient address without validating that it is non-zero. An unprivileged EVM user can supply `address(0)` as the recipient, causing tokens to be burned on Aurora while the cross-chain withdrawal promise is directed to the uncontrolled zero address on Ethereum, permanently destroying the funds.

### Finding Description
In `engine-precompiles/src/native.rs`, the `ExitToEthereum::run` function processes two withdrawal paths. In the ETH base-token path (flag `0x0`), the recipient address is parsed directly from the 20 bytes of user-supplied calldata:

```rust
let recipient_address: Address = input
    .try_into()
    .map_err(|_| ExitError::Other(Cow::from("ERR_INVALID_RECIPIENT_ADDRESS")))?;
```

No check is performed to ensure `recipient_address != Address::zero()`. The code proceeds to serialize the zero address into the withdrawal arguments and creates a `PromiseCreateArgs` targeting the eth connector's `withdraw` method with `address(0)` as the Ethereum recipient.

The same absence of a zero-address check applies to the ERC-20 path (flag `0x1`), where `recipient_address` is parsed from `input[32..52]` without validation.

The `EvmErc20.withdrawToEthereum(address recipient, uint256 amount)` function in `etc/eth-contracts/contracts/EvmErc20.sol` and `EvmErc20V2.sol` burns the caller's tokens unconditionally before invoking the precompile, so by the time the zero-address recipient is used, the tokens are already destroyed on the Aurora side.

### Impact Explanation
When a user calls `withdrawToEthereum(address(0), amount)` on a deployed `EvmErc20`/`EvmErc20V2` token, or calls the precompile directly with a zero recipient:

1. The ERC-20 tokens are burned from the caller's balance (or ETH is deducted for the base-token path).
2. The precompile succeeds without reversion and emits a withdrawal promise targeting `address(0)` on Ethereum.
3. The eth connector processes the withdrawal to `address(0)`, an address for which no private key exists.
4. The funds are permanently inaccessible — a permanent freeze of user funds.

This matches the **Critical: Permanent freezing of funds** impact category.

### Likelihood Explanation
The entry path is fully reachable by any unprivileged EVM user. The `EvmErc20.withdrawToEthereum` function is a public interface with no access control. A user can accidentally pass `address(0)` (e.g., from an uninitialized variable in a calling contract, a UI bug, or a scripting error). There is no on-chain guard to prevent this. Likelihood is **Medium** — not a common mistake, but the absence of a guard makes it a realistic accident or griefing vector against oneself.

### Recommendation
Add a zero-address sanity check in `ExitToEthereum::run` for both the `0x0` and `0x1` flag branches, immediately after parsing `recipient_address`:

```rust
if recipient_address == Address::zero() {
    return Err(ExitError::Other(Cow::from("ERR_INVALID_RECIPIENT_ADDRESS")));
}
```

Similarly, add the same guard in `EvmErc20.withdrawToEthereum` and `EvmErc20V2.withdrawToEthereum`:

```solidity
require(recipient != address(0), "ERR_ZERO_RECIPIENT");
```

### Proof of Concept

**Flag `0x0` (ETH base token) path:**
1. Deploy any contract on Aurora that calls the `ExitToEthereum` precompile at `0xb0bd02f6a392af548bdf1cfaee5dfa0eefcc8eab` with calldata `0x00` + 20 zero bytes, attaching ETH value.
2. The precompile parses `recipient_address = Address::zero()`, no error is returned.
3. A `withdraw` promise is created targeting `address(0)` on Ethereum.
4. The caller's ETH is permanently lost.

**Flag `0x1` (ERC-20) path:**
1. Call `EvmErc20.withdrawToEthereum(address(0), 1000)`.
2. `_burn(msg.sender, 1000)` executes — tokens are destroyed.
3. The precompile is invoked with `recipient = address(0)` encoded in the input.
4. `recipient_address` is parsed as `Address::zero()` with no validation error.
5. A `withdraw` promise is created targeting `address(0)` on Ethereum.
6. 1000 tokens are permanently lost.
