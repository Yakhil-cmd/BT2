### Title
Attacker Bypasses Invoke Signature Validation by Exploiting Victim's Deploy-Account in Mempool via `skip_stateful_validations` - (File: crates/apollo_gateway/src/stateful_transaction_validator.rs)

### Summary

The `skip_stateful_validations` function skips the `__validate__` entry-point call (signature check) for any invoke transaction with `nonce == 1` when the account's on-chain nonce is `0` and `account_tx_in_pool_or_recent_block` returns `true`. Because that helper returns `true` for **any** transaction in the pool for the account—not only a deploy-account—an unprivileged attacker can observe a victim's deploy-account in the mempool and immediately submit an invoke with `nonce = 1` and an **invalid signature** for the same address. The gateway admits the invalid invoke without ever calling `__validate__`.

### Finding Description

**Root cause — `skip_stateful_validations`** [1](#0-0) 

The function returns `true` (skip validation) when:
1. The transaction is an `Invoke`.
2. `tx.nonce() == Nonce(Felt::ONE)`.
3. `account_nonce == Nonce(Felt::ZERO)` (on-chain).
4. `mempool_client.account_tx_in_pool_or_recent_block(sender)` returns `true`.

When it returns `true`, `run_validate_entry_point` is called with `validate: false`: [2](#0-1) 

This means `validate_tx` in the blockifier immediately returns `Ok(None)` without executing `__validate__`: [3](#0-2) 

**The over-broad mempool check**

`account_tx_in_pool_or_recent_block` returns `true` if the account has **any** transaction in the pool: [4](#0-3) 

The code comment says *"either it has a deploy_account transaction or transactions with future nonces that passed validations"*, but the implementation does not distinguish between a deploy-account and any other transaction type. A victim's deploy-account in the pool satisfies condition 4 for **any** attacker-submitted invoke targeting the same address.

**Attack path**

1. Victim submits `DeployAccount` for address `X` (passes full gateway validation including `__validate_deploy__`).
2. Attacker observes the deploy-account in the public mempool.
3. Attacker submits `Invoke { sender_address: X, nonce: 1, signature: <garbage> }`.
4. Gateway stateless checks pass (valid resource bounds, size, DA modes).
5. `validate_nonce` passes: `account_nonce (0) ≤ tx_nonce (1) ≤ max_allowed_nonce_gap`. [5](#0-4) 

6. `validate_by_mempool` passes (only checks duplicates and nonce gaps, not signatures).
7. `skip_stateful_validations` returns `true` because the victim's deploy-account is in the pool.
8. `run_validate_entry_point` is called with `validate: false` → no `__validate__` call.
9. The invalid invoke is forwarded to the mempool and admitted. [6](#0-5) 

### Impact Explanation

An unprivileged attacker can inject an invoke transaction carrying an arbitrary (invalid) signature for any account whose deploy-account is currently pending in the mempool. This satisfies the **High** impact criterion:

> *Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.*

Concrete consequences:
- **Mempool pollution**: the attacker can flood the mempool with invalid invokes for every observed deploy-account.
- **Transaction displacement**: if the mempool supports fee-escalation replacement (same nonce, higher tip), the attacker's invalid invoke can evict the victim's legitimate nonce-1 invoke, forcing the victim to resubmit.
- **Sequencer resource waste**: the batcher will execute the invalid invoke, run `__validate__`, fail, roll back state, and charge the (likely unfunded) account—consuming block space.

### Likelihood Explanation

- Deploy-account transactions are publicly visible in the mempool.
- The attacker only needs to craft an invoke with `nonce = 1` and any bytes as the signature.
- No privileged access, no special knowledge of the victim's private key, and no on-chain funds are required.
- The race window is the time between the deploy-account entering the mempool and the batcher including it in a block—typically multiple seconds to minutes.

### Recommendation

Replace the coarse `account_tx_in_pool_or_recent_block` check with a targeted query that confirms a **deploy-account** transaction specifically exists for the address. Expose a dedicated mempool method such as `deploy_account_in_pool(address) -> bool` and use it in `skip_stateful_validations`. This preserves the UX feature (deploy + invoke in one shot) while closing the window for third-party signature bypass.

```rust
// Proposed fix sketch
async fn skip_stateful_validations(...) -> ... {
    if let ExecutableTransaction::Invoke(...) = tx {
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            // Only skip when a deploy-account (not any tx) is pending for this account.
            return mempool_client
                .deploy_account_in_pool(tx.sender_address())   // new targeted query
                .await
                ...
        }
    }
    Ok(false)
}
```

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
