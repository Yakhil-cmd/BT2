### Title
`execute_scheduled` Silently Absorbs Attached NEAR Deposit, Causing Permanent Fund Freeze — (`etc/xcc-router/src/lib.rs`)

---

### Summary

The `execute_scheduled` function in the XCC Router contract is marked `#[payable]` but never reads or forwards the attached NEAR deposit. Any NEAR sent by a caller is silently absorbed into the router sub-account's balance with no recovery path available to the sender.

---

### Finding Description

`execute_scheduled` at `etc/xcc-router/src/lib.rs:150-156` is annotated `#[payable]`, which in NEAR signals to callers that the function accepts NEAR deposits. However, the function body never calls `env::attached_deposit()` and never transfers the deposit anywhere: [1](#0-0) 

```rust
#[payable]
pub fn execute_scheduled(&mut self, nonce: U64) {
    let Some(promise) = self.scheduled_promises.remove(&nonce.0) else {
        env::panic_str("ERR_PROMISE_NOT_FOUND")
    };
    let promise_id = Self::promise_create(promise);
    env::promise_return(promise_id);
}
```

The scheduled promise already has its `attached_balance` encoded at scheduling time via `schedule`. Any NEAR attached at execution time is not forwarded to the promise and is not returned to the caller — it is silently absorbed into the router contract's balance. [2](#0-1) 

The router contract provides no general withdrawal mechanism. The only balance-reducing function is `send_refund`: [3](#0-2) 

`send_refund` is:
1. Restricted to the parent (Aurora engine) caller only via `require_caller(&parent)`.
2. Hardcoded to send exactly `REFUND_AMOUNT` = 2 NEAR (the storage staking amount), not the full balance. [4](#0-3) 

Therefore, any NEAR attached to `execute_scheduled` is permanently locked in the router sub-account with no recovery path for the sender.

---

### Impact Explanation

Any user who calls `execute_scheduled` with attached NEAR loses those funds permanently. The router contract has no mechanism to return arbitrary NEAR to callers. This constitutes a **permanent freezing of user funds** (Critical impact).

---

### Likelihood Explanation

The function is explicitly documented as callable by anyone: [5](#0-4) 

The `#[payable]` annotation signals to callers that NEAR is accepted. A user executing a scheduled promise that has a non-zero `attached_balance` may reasonably attach NEAR to cover the promise's cost, not realizing the deposit is not forwarded. This is a realistic mistake enabled by the misleading `#[payable]` annotation on a publicly callable function. The XCC router is a production contract deployed as a sub-account for every EVM address that uses cross-contract calls.

---

### Recommendation

Remove the `#[payable]` attribute from `execute_scheduled`. Without it, any NEAR attached to the call will cause a NEAR runtime panic, preventing accidental fund loss. If NEAR attachment is intentionally supported (e.g., to top up the router's balance for the promise), add explicit handling to either forward the deposit to the promise or refund it to `env::predecessor_account_id()`.

---

### Proof of Concept

1. A user observes a scheduled promise with `attached_balance > 0` and calls `execute_scheduled(nonce)` with attached NEAR (e.g., 1 NEAR) to cover the promise's cost.
2. The function removes the scheduled promise from storage and executes it using the pre-encoded `attached_balance` from storage — the user's 1 NEAR is not forwarded.
3. The 1 NEAR attached by the user is absorbed into the router contract's balance.
4. The user has no way to recover the 1 NEAR: `send_refund` is parent-only and sends a fixed 2 NEAR amount; no other withdrawal function exists in the router.
5. The 1 NEAR is permanently frozen in the router sub-account. [1](#0-0) [3](#0-2)

        self.assert_preconditions();

        let nonce = self.nonce.get().unwrap_or_default();
        self.scheduled_promises.insert(nonce, promise);
