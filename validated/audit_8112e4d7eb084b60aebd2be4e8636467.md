### Title
Staker Can Instantly Raise Commission to 100% via Commitment Mechanism, Stealing All Delegator Yield - (`src/staking/staking.cairo`)

---

### Summary

A staker can use `set_commission_commitment` followed immediately by `set_commission` to raise their commission from any value to 100% within the same epoch. Because `calculate_staker_pools_rewards` reads the commission from live storage at reward-calculation time — with no epoch-based snapshotting — the inflated commission applies to the current epoch's rewards, routing all delegator yield to the staker as commission. Delegators have no recourse: they cannot exit before rewards are distributed, and the two-step attack requires no governance timelock.

---

### Finding Description

The commission mechanism has two paths:

**Normal path** (`set_commission` without commitment): commission can only decrease. [1](#0-0) 

**Commitment path** (`set_commission_commitment` + `set_commission`): a staker first sets a `CommissionCommitment` with an arbitrary `max_commission` (up to 10000 = 100%), then immediately calls `set_commission` to raise commission to that ceiling. [2](#0-1) 

The only guard on `set_commission_commitment` is that `max_commission >= current_commission` and `expiration_epoch > current_epoch`. There is no minimum delay before the commitment can be used: [3](#0-2) 

The code itself acknowledges the gap: [4](#0-3) 

When rewards are calculated — either via `update_rewards_from_attestation_contract` (V2) or `update_rewards` (V3) — `calculate_staker_pools_rewards` reads the commission directly from current storage with no historical snapshot: [5](#0-4) 

That live commission value is then passed to `split_rewards_with_commission`, which determines how much of the pool's gross rewards goes to delegators vs. the staker: [6](#0-5) 

At commission = 10000 (100%), `split_rewards_with_commission` returns `(pool_rewards_including_commission, 0)` — the staker captures everything, delegators receive zero. [7](#0-6) 

---

### Impact Explanation

**High — Theft of unclaimed yield.**

Delegators who joined a pool expecting a low commission (e.g., 5%) can have 100% of their accrued epoch/block rewards redirected to the staker as commission. The staker's `unclaimed_rewards_own` is credited with `staker_own_rewards + commission_rewards`, while the pool contract receives zero: [8](#0-7) 

Delegators cannot exit before the attack completes: `exit_intent` only starts the exit window; funds are not returned until after `exit_wait_window` elapses. The commission change and reward distribution can occur within the same epoch.

---

### Likelihood Explanation

**Medium-High.** The attack requires only two consecutive transactions by the staker (no external dependencies, no governance vote, no timelock):

1. `set_commission_commitment(max_commission: 10000, expiration_epoch: current_epoch + 1)`
2. `set_commission(commission: 10000)`

Any staker who has opened a delegation pool can execute this. The motivation is clear: steal one epoch's worth of delegator yield, then optionally lower commission again to retain delegators. The attack is repeatable every epoch as long as no active commitment exists.

---

### Recommendation

1. **Epoch-snapshot the commission**: Record the commission value at the start of each epoch (alongside the balance trace). `calculate_staker_pools_rewards` should use the commission that was in effect at `curr_epoch`, not the current live value.

2. **Enforce a minimum delay between commitment creation and use**: Require that `set_commission` with an active commitment cannot be called in the same epoch the commitment was set.

3. **Alternatively, cap the maximum commission increase per epoch** (e.g., no more than N basis points per epoch), preventing a single-block jump from 5% to 100%.

---

### Proof of Concept

```
Epoch E, Block B:
  staker.set_commission_commitment(max_commission=10000, expiration_epoch=E+1)
  // Commitment is immediately active (expiration_epoch > current_epoch = E)

Epoch E, Block B+1:
  staker.set_commission(commission=10000)
  // update_commission: commitment is active, 10000 <= max_commission=10000, 10000 != old_commission
  // Commission stored = 10000 (100%)

Epoch E, Block B+2 (attestation window):
  attestation_contract.attest(block_hash)
  // → staking.update_rewards_from_attestation_contract(staker_address)
  // → _update_rewards → calculate_staker_pools_rewards
  //   commission = staker_pool_info.commission()  // reads 10000
  //   split_rewards_with_commission(pool_gross_rewards, 10000)
  //   → commission_rewards = pool_gross_rewards, pool_rewards = 0
  // staker.unclaimed_rewards_own += staker_own_rewards + pool_gross_rewards
  // pool contract receives 0 STRK

Delegators call claim_rewards → receive 0 for epoch E.
```

The staker has captured 100% of the delegated pool's epoch rewards. After the attack, the staker can call `set_commission(commission=500)` (now a decrease, always allowed) to restore the original rate and retain delegators for future epochs.

pub(crate) fn split_rewards_with_commission(
    rewards_including_commission: Amount, commission: Commission,
) -> (Amount, Amount) {
    let commission_rewards = compute_commission_amount_rounded_down(
        :rewards_including_commission, :commission,
    );
    let pool_rewards = rewards_including_commission - commission_rewards;
    (commission_rewards, pool_rewards)
}
