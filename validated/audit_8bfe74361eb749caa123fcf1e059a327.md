### Title
Incorrect Multi-NFT Royalty Amount Calculation in WalletConnect Offer Confirmation Understates User's Spending — (File: `packages/gui/src/electron/commands/parseCommandDisplay.ts`)

---

### Summary

The `formatAmountWithRoyalties` function in `parseCommandDisplay.ts` incorrectly divides the total fungible amount by the number of NFTs before applying each NFT's royalty percentage. When a WalletConnect dApp requests the user to take an offer containing multiple NFTs with royalties, the confirmation dialog's "You Spend" section displays a lower total than the user will actually pay, causing them to approve a transaction for the wrong amount.

---

### Finding Description

`formatAmountWithRoyalties` is called during `chia_wallet.take_offer` and `chia_wallet.create_offer_for_ids` WalletConnect command processing to compute the `amountWithRoyalties` field shown in the confirmation dialog. [1](#0-0) 

The critical flaw is on line 363:

```typescript
const splitAmount = amount / BigInt(royaltyPercentages.length);   // ← divides by N
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
```

This computes: `totalRoyalty = Σᵢ( ⌊amount / N⌋ × Rᵢ / 10000 )`

The correct formula — each NFT's royalty applied to the full purchase price — is: `totalRoyalty = Σᵢ( amount × Rᵢ / 10000 )`

The code's result is systematically lower due to both the division-by-N factor and BigInt truncation. The `amountWithRoyalties` value is then merged into the `DisplayWalletDeltaItem` and rendered in the WalletConnect confirmation dialog's "You Spend" section: [2](#0-1) [3](#0-2) 

The correct per-NFT royalty calculation is used in the offer builder UI via the daemon's `calculate_royalties` RPC, which sums each NFT's royalty against the full fungible amount: [4](#0-3) 

The WalletConnect confirmation path uses a different, incorrect client-side approximation instead.

---

### Impact Explanation

A user approving a WalletConnect `chia_wallet.take_offer` request for a multi-NFT offer with royalties sees an understated total in the confirmation dialog. They approve believing they will spend X XCH, but the actual on-chain transaction deducts more. This is a direct instance of "WalletConnect state that causes a user to approve… the wrong… amount" — a High-severity impact under the defined scope.

---

### Likelihood Explanation

Any WalletConnect-connected dApp can send a `chia_wallet.take_offer` request. A malicious dApp can craft an offer with multiple NFTs each carrying high royalty percentages to maximize the gap between the displayed and actual spending amounts. The user has no other indicator of the correct total in the confirmation dialog; the `amountWithRoyalties` field is the only royalty-inclusive figure shown. [5](#0-4) 

---

### Recommendation

Replace the split-amount calculation with the correct formula that applies each NFT's royalty to the full fungible amount:

```typescript
// Before (incorrect):
const splitAmount = amount / BigInt(royaltyPercentages.length);
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);

// After (correct):
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (amount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
```

---

### Proof of Concept

Two NFTs with `royalty_percentage` 500 (5 %) and 10 (0.1 %), purchase price = 1,000,000,000,000 mojos (1 XCH):

**Current code:**
- `splitAmount` = 500,000,000,000
- `royaltyAmount` = (500,000,000,000 × 500)/10,000 + (500,000,000,000 × 10)/10,000 = 25,500,000,000
- **Displayed total** = 1,025,500,000,000 mojos = **1.0255 XCH**

**Correct calculation (royalties on full amount):**
- `royaltyAmount` = (1,000,000,000,000 × 500)/10,000 + (1,000,000,000,000 × 10)/10,000 = 51,000,000,000
- **Actual total** = 1,051,000,000,000 mojos = **1.051 XCH**

The user is shown **1.0255 XCH** in the WalletConnect approval dialog but the transaction deducts **1.051 XCH** — a discrepancy of 0.0255 XCH per approval. The gap scales linearly with the number of NFTs and their royalty percentages. [6](#0-5)

function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],
): string | undefined {
  if (royaltyPercentages.length === 0 || line.kind === 'nft') {
    return undefined;
  }

  const splitAmount = amount / BigInt(royaltyPercentages.length);
  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;

  if (line.kind === 'xch') {
    return mojoToChiaLocaleString(totalAmount);
  }

  return mojoToCATLocaleString(totalAmount);
}
