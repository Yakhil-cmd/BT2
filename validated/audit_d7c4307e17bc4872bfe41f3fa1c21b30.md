### Title
Unrestricted `update_rewards` Allows Any Caller to Permanently Freeze Staker Yield - (File: src/staking/staking.cairo)

### Summary
The `update_rewards` function in the `Staking` contract specifies "Only starkware sequencer" as its access control in the protocol specification, but the implementation contains no caller check. Any unprivileged address can call it with `disable_rewards: true`, consuming the single per-block reward slot and permanently preventing legitimate reward distribution for every block.

### Finding Description
The protocol specification at `docs/spec.md` line 1645 states:

```
#### access control
Only starkware sequencer.
```

However, the implementation at `src/staking/staking.cairo` lines 1449–1488 contains no caller validation whatsoever:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
    let current_block_number = starknet::get_block_number();
    assert!(
        current_block_number > self.last_reward_block.read(),
        "{}",
        Error::REWARDS_ALREADY_UPDATED,
    );
    ...
    // Update last block rewards.
    self.last_reward_block.write(current_block_number);   // <-- always written

    if disable_rewards || self.is_pre_consensus() {
        return;   // <-- exits before distributing any rewards
    }
```

The storage field `last_reward_block` is a **single global** value (not per-staker):

```cairo
/// Last block number for which rewards were distributed.
last_reward_block: BlockNumber,
```

The guard `current_block_number > self.last_reward_block.read()` enforces at most one call per block. Once `last_reward_block` is written to the current block number, every subsequent call in that block reverts with `REWARDS_ALREADY_UPDATED`. Because `last_reward_block` is written unconditionally before the `disable_rewards` branch, an attacker who calls `update_rewards(valid_staker, disable_rewards: true)` first in any block:

1. Passes all precondition checks (staker exists, is active, has non-zero balance).
2. Writes `last_reward_block = current_block`.
3. Returns immediately without distributing rewards.
4. Causes every subsequent legitimate sequencer call in that block to revert. [1](#0-0) [2](#0-1) [3](#0-2) 

### Impact Explanation
**High — Permanent freezing of unclaimed yield.**

By submitting one transaction per block calling `update_rewards(any_active_staker, true)`, an attacker can ensure that `last_reward_block` is always equal to the current block before the sequencer's legitimate call arrives. This prevents `_update_rewards` from ever executing, meaning:

- `staker_info.unclaimed_rewards_own` is never incremented for any staker.
- Pool rewards are never forwarded to delegation pool contracts.
- `update_unclaimed_rewards_from_staking_contract` on the `RewardSupplier` is never called.

All stakers and pool members are permanently denied yield with no recovery path, because the skipped blocks are gone forever. [4](#0-3) [5](#0-4) 

### Likelihood Explanation
**High.** The entry point is a public, permissionless function. The attacker only needs to know one active staker address (trivially obtained from on-chain events such as `NewStaker`). The cost is one transaction per block. No privileged access, no leaked keys, and no external dependencies are required. The attack is sustainable indefinitely. [6](#0-5) 

### Recommendation
Add a caller check at the top of `update_rewards` that restricts execution to the designated sequencer address (stored in contract storage, analogous to how `attestation_contract` is stored). For example:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    assert!(
        get_caller_address() == self.sequencer_address.read(),
        "ONLY_SEQUENCER",
    );
    self.general_prerequisites();
    ...
}
```

Alternatively, if the sequencer address is not known at deploy time, restrict to the attestation contract or introduce a dedicated `SEQUENCER_ROLE` in the roles component. [7](#0-6) 

### Proof of Concept

1. Deploy the system normally. Let `STAKER` be any address that has called `stake()` and whose balance is effective (i.e., at least `K` epochs have passed).
2. Each block, before the sequencer's transaction is included, submit:
   ```
   staking_contract.update_rewards(staker_address=STAKER, disable_rewards=true)
   ```
3. Observe that `last_reward_block` is set to the current block number.
4. The sequencer's call to `update_rewards(staker_address=X, disable_rewards=false)` reverts with `REWARDS_ALREADY_UPDATED`.
5. After any number of blocks, call `staking_contract.claim_rewards(staker_address=STAKER)` and observe that `unclaimed_rewards_own` remains zero — no yield has accrued. [8](#0-7) [9](#0-8)
