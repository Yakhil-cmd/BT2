### Title
Gateway `skip_stateful_validations` Admits Unsigned Invoke Transactions via Overly Broad Mempool Existence Check — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator is intended to skip `__validate__` (signature verification) for a nonce-1 invoke transaction only when a `deploy_account` transaction is pending for the same undeployed account. However, the implementation checks whether **any** transaction from the sender exists in the mempool or a recent block — not specifically a `deploy_account`. An attacker who observes a victim's pending `deploy_account` can front-run the victim's first invoke by submitting a malicious nonce-1 invoke (with no valid signature) that passes gateway admission without signature verification, blocking the victim's legitimate invoke from entering the mempool.

### Finding Description

In `extract_state_nonce_and_run_validations`, the gateway fetches the on-chain nonce, runs pre-validation checks, and then calls `run_validate_entry_point` with a `skip_validate` flag: [1](#0-0) 

`run_pre_validation_checks` calls `skip_stateful_validations`, which returns `true` (skip `__validate__`) when the incoming invoke has `nonce == 1`, the on-chain `account_nonce == 0`, and `account_tx_in_pool_or_recent_block` returns `true`: [2](#0-1) 

When `skip_validate` is `true`, `run_validate_entry_point` sets `validate: false` in the `ExecutionFlags`, so the blockifier's `validate_tx` — which calls the account's `__validate__` entry point — is never invoked: [3](#0-2) 

The `validate_tx` implementation confirms: when `execution_flags.validate == false`, it returns immediately without calling `__validate__`: [4](#0-3) 

The critical flaw is that `account_tx_in_pool_or_recent_block` checks for **any** transaction from the sender address, not specifically a `deploy_account`. The comment acknowledges this but incorrectly reasons that any pooled transaction implies a valid deployment path: [5](#0-4) 

### Impact Explanation

**Attack scenario (front-running):**

1. Victim V submits a `deploy_account` transaction → accepted into mempool.
2. Attacker A observes this and immediately submits an invoke with `nonce=1` from address V, with arbitrary calldata and no valid signature.
3. Gateway evaluates: `tx.nonce() == 1` ✓, `account_nonce == 0` ✓, `account_tx_in_pool_or_recent_block(V)` → `true` (V's `deploy_account` is in pool) ✓ → `skip_validate = true` → `__validate__` is **not called** → malicious invoke is admitted to the mempool.
4. Victim V now submits their legitimate invoke with `nonce=1` → rejected by the mempool as a duplicate nonce (attacker's malicious invoke already occupies that slot).
5. During batcher execution, the malicious invoke reverts (the batcher runs `__validate__` with `validate: true`, which fails because the attacker has no valid signature). But the victim's legitimate invoke was never admitted.

**Result:** An invalid, unsigned transaction is admitted to the mempool (bypassing the signature-verification invariant), and the victim's valid transaction is rejected. This matches the "wrong payer/source" pattern from the external report: the authorization source used (any pooled transaction) is wrong — it should be specifically a `deploy_account`. [6](#0-5) 

### Likelihood Explanation

Any attacker who monitors the public mempool for `deploy_account` transactions can execute this attack. No privileged access is required. The attacker only needs to submit a transaction with `nonce=1` from the victim's address before the victim does. This is a straightforward front-running attack feasible on any public sequencer node.

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a `deploy_account` transaction exists for the sender address in the mempool. The mempool client interface should expose a method such as `deploy_account_in_pool(address)` that returns `true` only when a `deploy_account` transaction (not any transaction) is pending for that address.

### Proof of Concept

```
1. Victim V: submit deploy_account for address V → mempool accepts it.
2. Attacker A: submit InvokeV3 { sender_address: V, nonce: 1, calldata: [malicious], signature: [] }
   → Gateway: account_nonce=0, tx.nonce=1, account_tx_in_pool_or_recent_block(V)=true
   → skip_stateful_validations returns true
   → run_validate_entry_point sets validate=false → __validate__ NOT called
   → Malicious invoke admitted to mempool.
3. Victim V: submit InvokeV3 { sender_address: V, nonce: 1, calldata: [legitimate], signature: [valid] }
   → Mempool rejects: DuplicateNonce { address: V, nonce: 1 }
4. Batcher executes: deploy_account (V deployed), then malicious invoke
   → __validate__ called with validate=true → fails (no valid signature) → reverts.
5. Victim's legitimate invoke was never sequenced.
``` [2](#0-1) [7](#0-6)

        &mut self,
        executable_tx: &ExecutableTransaction,
        skip_validate: bool,
    ) -> StatefulTransactionValidatorResult<()> {
        let only_query = false;
        let charge_fee = enforce_fee(executable_tx, only_query);
        let strict_nonce_check = false;
        let execution_flags =
            ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };

        let account_tx = AccountTransaction { tx: executable_tx.clone(), execution_flags };

        // Build block context.
        let mut versioned_constants = VersionedConstants::get_versioned_constants(
            self.config.versioned_constants_overrides.clone(),
        );
        // The validation of a transaction is not affected by the casm hash migration.
        versioned_constants.disable_casm_hash_migration();

        let mut block_info = self.gateway_fixed_block_state_reader.get_block_info().await?;
        block_info.block_number = block_info.block_number.unchecked_next();
        let block_context = BlockContext::new(
            block_info,
            self.chain_info.clone(),
            versioned_constants,
            BouncerConfig::max(),
        );

        // Move state into the blocking task and run CPU-heavy validation.
        let state_reader_and_contract_manager = self.take_state_reader_and_contract_manager();

        let cur_span = Span::current();
        #[allow(clippy::result_large_err)]
        tokio::task::spawn_blocking(move || {
            cur_span.in_scope(|| {
                let state = CachedState::new(state_reader_and_contract_manager);
                let mut blockifier_validator = StatefulValidator::create(state, block_context);
                blockifier_validator.validate(account_tx)
            })
        })
        .await
        .map_err(|e| StarknetError {
            code: StarknetErrorCode::UnknownErrorCode(
                "StarknetErrorCode.InternalError".to_string(),

            message: format!("Blocking task join error when running the validate entry point: {e}"),
        })?
        .map_err(|e| StarknetError {
            code: StarknetErrorCode::KnownErrorCode(KnownStarknetErrorCode::ValidateFailure),
            message: e.to_string(),
        })?;
        Ok(())
    }

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
