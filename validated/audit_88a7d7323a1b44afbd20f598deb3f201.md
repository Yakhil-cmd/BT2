Audit Report

## Title
`SenderInfoContent` Signature Missing Target Canister ID Enables Cross-Canister Replay - (File: `rs/validator/src/ingress_validation.rs`)

## Summary
The `sender_info` ingress feature signs only `domain("ic-sender-info") || info_bytes` with no target canister ID bound into the payload. Because the signed bytes are canister-agnostic, a valid `{info, signer, sig}` triple obtained for canister A can be replayed verbatim in a request to any canister B. Canister B will observe the same attested `info` blob as if the signing canister had issued it specifically for B, bypassing any authorization logic that assumes the attestation is canister-scoped.

## Finding Description
`SenderInfoContent` is defined as a bare wrapper over the raw `info` bytes:

```rust
// rs/types/types/src/messages/http.rs L344-347
impl crate::crypto::SignedBytesWithoutDomainSeparator for SenderInfoContent<'_> {
    fn write_signed_bytes_without_domain_separator(&self, bytes: &mut Vec<u8>) {
        bytes.extend_from_slice(self.0);  // only the raw info bytes
    }
}
```

Its `SignatureDomain` prepends only the fixed string `"ic-sender-info"`:

```rust
// rs/types/types/src/crypto/sign.rs L161-164
impl<'a> SignatureDomain for SenderInfoContent<'a> {
    fn domain(&self) -> Vec<u8> {
        domain_with_prepended_length("ic-sender-info")
    }
}
```

So the bytes actually signed are `\x0Eic-sender-info || info_bytes` — no canister ID, no nonce, no expiry.

`verify_sender_info_canister_sig` (L494–545) performs exactly three checks: (1) `sender_pubkey` is a valid canister-signature public key, (2) the canister ID in `sender_pubkey` matches the declared `signer` field, and (3) the canister signature over `SenderInfoContent(&sender_info.info)` is cryptographically valid. There is no check that the `info` blob contains or is bound to the target canister ID.

`validate_sender_info` (L459–488) calls `verify_sender_info_canister_sig` without passing the request's canister ID. Structurally, `validate_sender_info` accepts `HttpRequest<C>` where `C: HttpRequestContent`, but `HttpRequestContent` does not extend `HasCanisterId`, so the target canister ID is not even accessible at the point of `sender_info` validation.

The validated `info` blob is then forwarded verbatim to the target canister via `ic0_msg_caller_info_data_copy` / `ic0_msg_caller_info_signer_copy`.

**Exploit flow:**
1. Attacker (or legitimate user) triggers a normal flow that causes the signing canister (e.g., Internet Identity) to produce `{info: b"user-is-premium", signer: II_id, sig: S}` for canister A.
2. Attacker constructs a new ingress call to canister B, embedding the identical `{info, signer, sig}` triple.
3. The boundary node runs `verify_sender_info_canister_sig`: pubkey valid ✓, signer matches ✓, signature over `SenderInfoContent(b"user-is-premium")` valid ✓ (same bytes, same signature), no canister ID check ✓ — passes.
4. Canister B receives `ic0_msg_caller_info_data_copy` → `b"user-is-premium"` and grants premium access, even though II never issued this attestation for B.

## Impact Explanation
This matches **High ($2,000–$10,000): Unauthorized access to identities, canister-controlled funds, or platform resources where exploitation requires meaningful per-target work or other constraints.** Any canister that uses `sender_info` for authorization (the explicitly intended use case for Internet Identity integration) is structurally vulnerable. The attacker needs only one legitimately obtained signature triple; replaying it to additional canisters is trivial and repeatable with zero additional cost.

## Likelihood Explanation
No privileged access, no key compromise, and no threshold corruption is required. The attacker-controlled entry path is a standard ingress HTTP call with a crafted envelope. The only precondition is obtaining one valid `sender_info` signature from the signing canister for any canister — a normal user flow. The `sender_info` feature is explicitly designed for Internet Identity, making identity-gating canisters the primary and realistic target population.

## Recommendation
Bind the signed payload to the target canister ID. Change `SenderInfoContent` to include the destination canister ID in the signed bytes:

```
\x0Eic-sender-info || canister_id_bytes || info_bytes
```

Alternatively, require that `verify_sender_info_canister_sig` extract the target canister ID from the `info` blob and verify it against the request's `canister_id`. Either approach prevents a signature issued for canister A from being accepted by canister B. The `validate_sender_info` function must be updated to accept and pass the target canister ID, which requires `C` to also implement `HasCanisterId` (or passing the canister ID separately).

## Proof of Concept
**Minimal deterministic integration test plan:**

1. Instantiate a local replica (PocketIC or state-machine-tests).
2. Deploy a mock "signing canister" that produces a canister signature over `SenderInfoContent(b"grant-access")` using its canister signature key.
3. Deploy canister A and canister B, both of which call `ic0_msg_caller_info_data_copy` and return the raw bytes.
4. Submit an ingress call to canister A with `sender_info = {info: b"grant-access", signer: signing_canister_id, sig: <sig>}`. Confirm it is accepted and canister A returns `b"grant-access"`.
5. Construct an identical ingress call to canister B, reusing the same `{info, signer, sig}` triple verbatim (only the outer envelope's `canister_id` field changes).
6. Assert the call to canister B is also accepted and returns `b"grant-access"` — demonstrating that the signature issued for canister A is accepted by canister B without modification.
