### Title
Gateway Admits Invoke Transactions with Unverified Signatures via Overly Broad `skip_stateful_validations` Check - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the gateway's stateful transaction validator bypasses the `__validate__` entry point (account signature verification) for any invoke transaction with nonce 1 when `account_tx_in_pool_or_recent_block` returns `true`. The check is too broad: it returns `true` for **any** transaction from the account in the mempool, not specifically a `deploy_account` transaction. An attacker can exploit this to submit invoke transactions with invalid/fake signatures for any address that has a pending `deploy_account` in the mempool, causing those transactions to be admitted to the mempool without signature verification.

---

### Finding Description

**Root cause — `skip_stateful_validations` uses an overly broad membership check**

In `crates/apollo_gateway/src/stateful_transaction_validator.rs`, the function `skip_stateful_validations` (lines 429–461) is designed to improve UX by allowing a user to submit a `deploy_account` + `invoke` pair simultaneously. When an invoke transaction has nonce 1 and the account's on-chain nonce is 0, the function calls `account_tx_in_pool_or_recent_block` to decide whether to skip `__validate__`: [1](#0-0) 

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` sets `validate = !skip_validate = false`: [2](#0-1) 

Inside `StatefulValidator::perform_validations`, when `validate = false`, the function returns `Ok(())` immediately after `perform_pre_validation_stage` — the `__validate__` entry point (which contains the account's signature check) is **never called**: [3](#0-2) 

**The broken invariant — `account_tx_in_pool_or_recent_block` is not deploy-account-specific**

The mempool's `account_tx_in_pool_or_recent_block` returns `true` if the account has **any** transaction in the pool, not specifically a `deploy_account`: [4](#0-3) 

The code comment in `skip_stateful_validations` claims: *"it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."* The second clause is circular: if a prior invoke with nonce 1 was itself admitted via this same skip (because a `deploy_account` was in the pool at that time), it is now in the pool and will cause all subsequent nonce-1 invokes to also skip validation — even after the original `deploy_account` has been evicted.

**Attack path**

1. Alice submits a `deploy_account` for address A (nonce 0) → enters the mempool.
2. Bob observes Alice's `deploy_account` in the mempool (public mempool state).
3. Bob submits an `invoke` with nonce 1 for address A, carrying a fake/invalid signature, with a tip higher than Alice's pending invoke.
4. Gateway stateful validation:
   - `account_nonce = 0`, `tx_nonce = 1` → nonce check passes.
   - `account_tx_in_pool_or_recent_block(A) = true` (Alice's `deploy_account` is in the pool).
   - `skip_stateful_validations` returns `true` → `validate = false` → `__validate__` is **not called**.
5. Bob's fake invoke is accepted into the mempool. Via fee escalation it replaces Alice's legitimate invoke.
6. Batcher executes: `deploy_account` (succeeds, account deployed) → Bob's fake invoke (`__validate__` is called with `validate = true` during execution, fails, transaction rejected).
7. Alice's legitimate invoke has been evicted from the mempool; Alice must resubmit.

The mempool's own `validate_tx` does not check signatures either: [5](#0-4) 

---

### Impact Explanation

**Primary impact (High):** The gateway/mempool admission path accepts invalid transactions — specifically, invoke transactions with forged or invalid signatures — before sequencing. This directly matches the allowed impact: *"High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

Concretely: any attacker who observes a `deploy_account` in the mempool can submit a competing invoke with nonce 1 carrying an invalid signature, pay a higher tip to win fee escalation, and cause the legitimate user's invoke to be dropped from the mempool. The attacker's transaction will fail during batcher execution (because `__validate__` is enforced there), but the victim's transaction is already gone.

**Escalated impact (Critical, conditional):** For account contracts whose `__validate__` is permissive (e.g., always returns success, or is absent), Bob's fake invoke would execute successfully with arbitrary calldata on Alice's account. This would allow unauthorized state changes, fund transfers, or storage writes — matching the Critical impact tier: *"Invalid or unauthorized Starknet transaction accepted through account validation, signature … logic."*

---

### Likelihood Explanation

- The mempool is observable; a `deploy_account` in the pending pool is visible to any node.
- The attack window is the time between the `deploy_account` entering the mempool and being included in a block (potentially many seconds to minutes).
- The only cost to the attacker is paying a higher tip than the victim's invoke. If the attacker's transaction fails during execution, the attacker is charged a fee, but the victim's transaction is still lost.
- No special privileges, keys, or access are required.

---

### Recommendation

1. **Narrow the check**: `skip_stateful_validations` should verify that the account has specifically a `deploy_account` transaction in the mempool, not just any transaction. Expose a dedicated `has_pending_deploy_account(address) -> bool` query from the mempool that filters by transaction type.

2. **Alternatively**: Pass the expected `deploy_account` transaction hash (as `PyValidator::should_run_stateful_validations` does via `deploy_account_tx_hash`) and verify it matches a `deploy_account` in the pool for the correct address. [6](#0-5) 

3. **Defense in depth**: The mempool could tag transactions admitted with `skip_validate = true` and re-validate them when the `deploy_account` is committed, evicting any that fail.

### Proof of Concept

```
State: Alice's deploy_account for address A (nonce 0) is in the mempool.
       Alice's invoke for address A (nonce 1, tip = 10) is in the mempool.

Step 1: Bob calls gateway add_transaction with:
        - type: Invoke
        - sender_address: A
        - nonce: 1
        - signature: [0xdeadbeef]  ← invalid/fake
        - tip: 11  ← higher than Alice's

Step 2: Gateway stateless validation passes (signature size is within limit).

Step 3: Gateway stateful validation:
        account_nonce = get_nonce(A) = 0
        validate_nonce: 0 <= 1 <= 0 + max_gap → passes
        validate_by_mempool: no duplicate hash, tip > Alice's → fee escalation accepted
        skip_stateful_validations:
            tx.nonce() == 1 ✓
            account_nonce == 0 ✓
            account_tx_in_pool_or_recent_block(A) == true ✓ (Alice's deploy_account)
            → returns true (skip)
        run_validate_entry_point(skip_validate=true):
            validate = false → __validate__ NOT called → Ok(())

Step 4: Bob's fake invoke enters the mempool, replacing Alice's invoke.

Step 5: Batcher executes deploy_account → A is deployed.
        Batcher executes Bob's invoke → __validate__ called → FAILS (invalid sig).
        Bob's invoke is rejected. Alice's invoke is gone.

Result: Alice's transaction is lost. Alice must resubmit.
        For accounts with permissive __validate__: Bob's invoke executes successfully.
```

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}

    pub fn should_run_stateful_validations(
        &mut self,
        account_tx: &AccountTransaction,
        deploy_account_tx_hash: Option<TransactionHash>,
    ) -> StatefulValidatorResult<bool> {
        if account_tx.tx_type() != TransactionType::InvokeFunction {
            return Ok(true);
        }
        let tx_info = account_tx.create_tx_info();
        let nonce = self.stateful_validator.get_nonce(tx_info.sender_address())?;

        let deploy_account_not_processed =
            deploy_account_tx_hash.is_some() && nonce == Nonce(Felt::ZERO);
        let tx_nonce = tx_info.nonce();
        let is_post_deploy_nonce = Nonce(Felt::ONE) <= tx_nonce;
        let nonce_small_enough_to_qualify_for_validation_skip =
            tx_nonce <= self.max_nonce_for_validation_skip;

        let skip_validate = deploy_account_not_processed
            && is_post_deploy_nonce
            && nonce_small_enough_to_qualify_for_validation_skip;

        Ok(!skip_validate)
    }
