Audit Report

## Title
SNS Treasury 50% Deposit Cap Bypassed via Concurrent `ExecuteExtensionOperation` Proposal Race - (File: `rs/sns/governance/src/extensions.rs`)

## Summary
`validate_deposit_operation_impl` enforces a 50% treasury cap by fetching the live balance at validation time, but because proposal execution is spawned as a background task via `spawn_in_canister_env`, two concurrently executing `ExecuteExtensionOperation` proposals both read the same pre-transfer balance snapshot, both independently pass the 50% check, and both proceed to transfer funds — together exceeding the intended cap by up to ~100% of the SNS treasury.

## Finding Description
`validate_deposit_operation_impl` (extensions.rs L276–321) fetches the live SNS and ICP treasury balances over async calls and checks each requested amount against 50% of the current balance:

```rust
if sns_requested > sns_balance.checked_div(2).unwrap() { return Err(...) }
if icp_requested > icp_balance.checked_div(2).unwrap() { return Err(...) }
```

This validation is called inside `perform_execute_extension_operation` (governance.rs L2570–2574), which is itself launched as a background future via `spawn_in_canister_env` (governance.rs L2132–2133):

```rust
let governance: &'static mut Governance = unsafe { std::mem::transmute(self) };
spawn_in_canister_env(governance.perform_action(proposal_id, action));
```

IC canisters interleave execution at every `await` point. When two proposals are spawned concurrently, both reach `validate_deposit_operation_impl` before either has completed its transfer. Both see the same pre-transfer balance and both pass the 50% check. Execution then proceeds to `execute_treasury_manager_deposit` (extensions.rs L1545–1610), which calls `approve_treasury_manager` (the actual fund transfer/approval) only after several additional `await` points — including `treasury_manager_deposit_context().await` — giving the second proposal ample opportunity to complete its own validation before the first transfer settles.

There is no canister-level lock, no "pending deposit" counter, and no re-validation at the point of actual transfer. The 50% check is structurally correct in isolation but is not atomic with respect to concurrent proposal execution.

## Impact Explanation
The 50% cap is a protocol-level safety constraint on the SNS framework, specifically designed to prevent any single governance action from depositing the majority of the treasury into an external extension canister. Bypassing it allows up to ~100% of both the SNS token treasury and the ICP treasury to be transferred to an extension canister in a single coordinated governance action. If the extension canister is malicious, buggy, or compromised, the entire SNS treasury is at risk of permanent loss. This matches the allowed High impact: **"Significant SNS security impact with concrete user or protocol harm"** and **"Unauthorized access to governance assets where exploitation requires meaningful per-target work or other constraints."**

## Likelihood Explanation
Any SNS neuron holder (or coalition) with sufficient voting power to pass two proposals can exploit this. Early-stage SNS deployments commonly have a single whale neuron with majority voting power. The attacker submits two deposit proposals each requesting just under 50% of the current treasury balance, ensures both are adopted before either executes (trivially achieved by submitting them in the same voting period), and the race condition does the rest. No privileged system access, no phishing, and no external dependency is required beyond normal SNS neuron ownership.

## Recommendation
1. **Re-validate at transfer time**: Inside `execute_treasury_manager_deposit`, re-fetch the treasury balance immediately before calling `approve_treasury_manager` and re-apply the 50% check at that point, so the check reflects any concurrent transfers already in flight.
2. **Pending-deposit counter**: Introduce a canister-level `pending_sns_deposit_e8s` / `pending_icp_deposit_e8s` counter that is incremented (synchronously, before any `await`) when a deposit proposal begins execution and decremented on completion or failure. The 50% check should include pending amounts: `requested + pending <= balance / 2`.
3. **Concurrency lock**: Limit concurrent `ExecuteExtensionOperation` deposit proposals to one at a time using a canister-level lock (analogous to `update_balance_accounts` in the ckBTC minter, `rs/bitcoin/ckbtc/minter/src/state.rs` L444–446), rejecting any second deposit attempt while one is already in flight.

## Proof of Concept
1. Deploy an SNS with a treasury of 100 SNS tokens and 200 ICP.
2. Submit Proposal A: `ExecuteExtensionOperation` with `operation_name = "deposit"`, `treasury_allocation_sns_e8s = 49e8`, `treasury_allocation_icp_e8s = 99e8` (each < 50%).
3. Submit Proposal B: identical parameters.
4. Ensure both proposals reach `Adopted` status before either begins executing (submit them in the same voting period or use immediate-execution test setup).
5. Both proposals are spawned via `spawn_in_canister_env`; in a PocketIC or multi-round integration test, advance rounds so both futures reach `validate_deposit_operation_impl` before either `approve_treasury_manager` call completes.
6. Observe: both proposals pass the 50% check against the same pre-transfer balance (100 SNS / 200 ICP), both call `approve_treasury_manager`, and the treasury is drained of 98 SNS + 198 ICP — 98% and 99% respectively, far exceeding the 50% cap.

A deterministic integration test using PocketIC can reproduce this by controlling the interleaving of async steps across the two proposal futures.
