### Title
Unbounded `pool_member_epoch_balance` Trace Loop in `calculate_rewards()` Enables Permanent Freezing of Unclaimed Yield — (File: `src/pool/pool.cairo`)

---

### Summary

The `calculate_rewards` function in `src/pool/pool.cairo` iterates over every entry in a pool member's `pool_member_epoch_balance` trace since the last claimed checkpoint. Each balance-changing operation (`add_to_delegation_pool`, `exit_delegation_pool_intent`, `enter_delegation_pool`) in a distinct epoch appends a new, permanent entry to this trace. A pool member who repeatedly cycles their stake across many epochs without claiming rewards will accumulate an unbounded number of trace entries. Once the trace is large enough, the `claim_rewards` transaction will always revert due to Starknet gas exhaustion, permanently freezing the pool member's unclaimed yield with no recovery path.

---

### Finding Description

**Root cause — unbounded loop in `calculate_rewards`:** [1](#0-0) 

The developers themselves annotate this loop as unbounded:

```cairo
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
    if pool_member_checkpoint.epoch() >= until_epoch { break; }
    let to_sigma = self.find_sigma(pool_member_checkpoint, curr_epoch: until_epoch);
    ...
    entry_to_claim_from += 1;
}
```

Each iteration performs at least two storage reads: one `pool_member_trace.at(...)` and one `find_sigma(...)` call into `cumulative_rewards_trace`. [2](#0-1) 

**How trace entries accumulate:**

The `insert` function in `trace.cairo` only deduplicates entries sharing the same epoch key; a new epoch always appends a new checkpoint: [3](#0-2) 

The key used is `current_epoch + K`: [4](#0-3) 

Every call to `add_to_delegation_pool`, `exit_delegation_pool_intent`, or `enter_delegation_pool` in a new epoch appends one entry:

- `enter_delegation_pool` → `set_member_balance` [5](#0-4) 
- `add_to_delegation_pool` → `increase_member_balance` [6](#0-5) 
- `exit_delegation_pool_intent` → `set_member_balance` [7](#0-6) 

**Why `entry_to_claim_from` does not protect against this:**

`entry_to_claim_from` advances only when `claim_rewards` succeeds. If the pool member never calls `claim_rewards` (or if the call begins reverting before it can advance the pointer), the loop always starts from the original checkpoint and must traverse the entire accumulated trace. [8](#0-7) 

**Exploit cycle (one iteration per epoch pair):**

1. Pool member calls `add_to_delegation_pool` in epoch E → appends entry for epoch `E+K`.
2. Pool member calls `exit_delegation_pool_intent` in epoch `E+1` → appends entry for epoch `E+1+K`.
3. Pool member waits for exit window, calls `exit_delegation_pool_action` (no trace entry added).
4. Pool member calls `add_to_delegation_pool` again in epoch `E+W+2` → appends another entry.
5. Repeat without ever calling `claim_rewards`.

Each full cycle adds two entries. The trace grows monotonically and is never pruned.

---

### Impact Explanation

**Impact: Permanent freezing of unclaimed yield.**

Once the `pool_member_epoch_balance` trace for a given pool member is large enough that a single `claim_rewards` call exhausts Starknet's per-transaction gas limit, the call will always revert. Because `entry_to_claim_from` is only advanced inside a successful `claim_rewards` execution, there is no partial-claim mechanism and no way to recover the frozen rewards. The pool member's entire accumulated unclaimed yield is permanently locked in the contract with no administrative escape hatch.

This matches the allowed impact: *"Permanent freezing of unclaimed yield or unclaimed royalties."*

---

### Likelihood Explanation

**Likelihood: Low-Medium.**

- The trace grows at most one entry per epoch (same-epoch operations share the same key and overwrite rather than append).
- The attacker must be the pool member themselves — no external party can grow another user's trace, since `add_to_delegation_pool` is gated to `pool_member` or `reward_address` only. [9](#0-8) 
- A long-lived, active delegator who never claims rewards is the realistic victim. The protocol has no mechanism to force periodic claims.
- The developers' own comment acknowledges the risk ("unlikely to exceed gas limits"), implying no hard cap exists.

---

### Recommendation

1. **Enforce a maximum unclaimed-entry depth**: Before allowing a new balance change, assert that `pool_member_trace_length - entry_to_claim_from` is below a safe threshold (e.g., 500), reverting with a descriptive error if exceeded. This forces users to claim before accumulating too many entries.
2. **Implement partial/paginated claiming**: Allow `claim_rewards` to accept an optional `max_entries` parameter, advancing `entry_to_claim_from` by at most that many steps per call, so rewards can always be claimed in batches regardless of trace length.
3. **Compact the trace on claim**: After a successful `claim_rewards`, truncate or zero out all trace entries before `entry_to_claim_from` to prevent unbounded storage growth.

### Proof of Concept

```
Preconditions:
  - Staker is active with a delegation pool open.
  - Pool member has sufficient STRK to cycle stake.
  - Exit wait window = W epochs.

Steps:
  For i in 1..N:
    1. pool_member calls add_to_delegation_pool(amount=min_stake) in epoch 2i
       → pool_member_epoch_balance trace length = 2i-1
    2. pool_member calls exit_delegation_pool_intent(amount=min_stake) in epoch 2i+1
       → pool_member_epoch_balance trace length = 2i
    3. Advance W epochs.
    4. pool_member calls exit_delegation_pool_action()
       (trace length unchanged, funds returned)
    // Do NOT call claim_rewards at any point.

After N cycles:
  - pool_member_epoch_balance trace length ≈ 2N
  - entry_to_claim_from = 0 (never advanced)

Attack completion:
  5. pool_member calls claim_rewards(pool_member)
     → calculate_rewards loops 2N times, each iteration doing 2+ storage reads
     → transaction reverts: OUT_OF_GAS
     → entry_to_claim_from remains 0
     → all accumulated rewards are permanently frozen
``` [10](#0-9)

        fn calculate_rewards(
            self: @ContractState,
            pool_member: ContractAddress,
            from_checkpoint: PoolMemberCheckpoint,
            until_checkpoint: PoolMemberCheckpoint,
            mut entry_to_claim_from: VecIndex,
        ) -> (Amount, VecIndex) {
            let pool_member_trace = self.pool_member_epoch_balance.entry(pool_member);
            // Note: `until_epoch` is the current epoch.
            let until_epoch = until_checkpoint.epoch();

            let mut rewards = 0;

            let pool_member_trace_length = pool_member_trace.length();

            let mut from_sigma = self.find_sigma(from_checkpoint, curr_epoch: until_epoch);
            let mut from_balance = from_checkpoint.balance();

            let base_value = self.staking_rewards_base_value.read();

            // **Note**: The loop iterates over the balance changes in the pool member's balance
            // trace. This loop is unbounded but unlikely to exceed gas limits.
            while entry_to_claim_from < pool_member_trace_length {
                let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
                // If the balance change is after `until_epoch` (and therefore does not affect
                // the current reward computation), exit the loop.
                if pool_member_checkpoint.epoch() >= until_epoch {
                    break;
                }

                // Compute rewards from (inclusive) the previous balance change (or from
                // `from_checkpoint`) to (exclusive) the current entry.
                let to_sigma = self.find_sigma(pool_member_checkpoint, curr_epoch: until_epoch);
                rewards +=
                    compute_rewards_rounded_down(
                        amount: from_balance, interest: to_sigma - from_sigma, :base_value,
                    );
                from_sigma = to_sigma;
                from_balance = pool_member_checkpoint.balance();
                entry_to_claim_from += 1;
            }

            // Compute the remaining rewards from (inclusive) the last visited balance change in
            // `pool_member_trace` (or from `from_checkpoint`) to (exclusive) `until_checkpoint`.
            let to_sigma = self.find_sigma(until_checkpoint, curr_epoch: until_epoch);
            rewards +=
                compute_rewards_rounded_down(
                    amount: from_balance, interest: to_sigma - from_sigma, :base_value,
                );

            (rewards, entry_to_claim_from)
        }
