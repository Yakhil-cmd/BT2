### Title
Gateway Stateful Validator Uses Stale Previous-Block L2 Gas Price as Admission Threshold Instead of `next_l2_gas_price` from Block Header — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

`StatefulTransactionValidator::validate_resource_bounds` reads the **current (previous) block's** `l2_gas_price` as the admission threshold for V3 `AllResources` transactions. The actual blockifier execution of those transactions uses the **next block's** L2 gas price, which is computed by the consensus orchestrator via `calculate_next_l2_gas_price_for_fin` and stored in `BlockHeaderWithoutHash::next_l2_gas_price`. The code itself carries a developer TODO acknowledging the wrong field is being read. The mismatch causes the gateway to admit transactions that will fail blockifier pre-validation (when the price rises) or to reject transactions that would succeed (when the price falls).

---

### Finding Description

**Root cause — wrong field read in `validate_resource_bounds`:** [1](#0-0) 

The function reads `gas_prices.strk_gas_prices.l2_gas_price` — the price that was used **inside the already-committed block** — and uses it as the threshold against which the incoming transaction's `max_price_per_unit` is compared: [2](#0-1) 

The developer TODO on line 229 explicitly states the correct value is `next_l2_gas_price` from the block header.

**What the correct value is:**

`next_l2_gas_price` is an EIP-1559-style forward-looking price computed at block-finalization time by `calculate_next_l2_gas_price_for_fin`, stored in `BlockHeaderWithoutHash::next_l2_gas_price`, and propagated in `ProposalFin` / `L2GasInfo`: [3](#0-2) [4](#0-3) [5](#0-4) 

The consensus orchestrator updates `self.l2_gas_price` to this computed value after each block decision, and the next block is built with that updated price. The gateway, however, reads the **old** price from the committed block's `BlockInfo`, not the `next_l2_gas_price` field.

**Where the mismatch causes a broken invariant:**

During blockifier pre-validation (`perform_pre_validation_stage`), `check_fee_bounds` is called when `charge_fee` is true. It compares the transaction's `max_price_per_unit` against the **actual block's** gas prices: [6](#0-5) 

If the next block's L2 gas price is higher than the previous block's L2 gas price, a transaction whose `max_price_per_unit` equals the previous block's price passes the gateway check but fails `check_fee_bounds` at execution time.

---

### Impact Explanation

**Scenario A — price rising (gateway admits invalid transactions):**

- Previous block L2 gas price: P
- Next block L2 gas price: P′ > P (e.g., gas consumption exceeded target)
- User submits V3 transaction with `max_price_per_unit = P`
- Gateway check: `P ≥ P × 100% = P` → **ADMITTED**
- Blockifier execution: `P < P′` → `MaxGasPriceTooLow` → **FAILS**

The transaction enters the mempool and wastes sequencer resources. Any unprivileged user can trigger this by submitting a transaction priced at exactly the previous block's L2 gas price.

**Scenario B — price falling (gateway rejects valid transactions):**

- Previous block L2 gas price: P
- Next block L2 gas price: P′ < P
- User submits V3 transaction with `max_price_per_unit = P′`
- Gateway check: `P′ < P × 100%` → **REJECTED** with `GAS_PRICE_TOO_LOW`
- Actual execution would succeed since `P′ ≥ P′`

Valid transactions are denied admission.

Both outcomes match: **High — Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing.**

---

### Likelihood Explanation

The L2 gas price changes every block via the fee-market formula. The maximum increase per block is bounded by `MIN_GAS_PRICE_INCREASE_DENOMINATOR = 333` (~0.3%) when below the configured minimum: [7](#0-6) 

However, the minimum price itself can jump discontinuously when `min_l2_gas_price_per_height` is updated via dynamic config, causing a larger gap between `l2_gas_price` and `next_l2_gas_price`. The default `min_gas_price_percentage = 100` means the threshold is set to exactly the previous block's price, so even a 1-fri increase in the next block's price is sufficient to trigger Scenario A. This condition occurs at every block where gas consumption exceeds the target. [8](#0-7) 

---

### Recommendation

Replace the read of `gas_prices.strk_gas_prices.l2_gas_price` with a read of `next_l2_gas_price` from the latest committed block header, as the TODO comment already prescribes:

```rust
// In validate_resource_bounds:
let next_block_l2_gas_price = self
    .gateway_fixed_block_state_reader
    .get_block_header()   // expose next_l2_gas_price from StorageBlockHeader
    .await?
    .next_l2_gas_price;
self.validate_tx_l2_gas_price_within_threshold(
    executable_tx.resource_bounds(),
    next_block_l2_gas_price,
)?;
```

`StorageBlockHeader` already stores `next_l2_gas_price`: [9](#0-8) 

---

### Proof of Concept

1. Observe the current committed block's L2 gas price P (e.g., 100 fri) and the `next_l2_gas_price` P′ (e.g., 101 fri, because gas consumption exceeded the target).
2. Submit a V3 `AllResources` invoke transaction with `l2_gas.max_price_per_unit = 100`.
3. Gateway `validate_resource_bounds` computes threshold = `100 × 100% = 100`; `100 ≥ 100` → transaction is **admitted**.
4. When the batcher pulls the transaction and the blockifier runs `check_fee_bounds` against the next block's gas prices (101 fri), it finds `100 < 101` and raises `MaxGasPriceTooLow`.
5. The transaction reverts or is dropped, having consumed mempool and batcher resources without producing useful work.

The attacker needs no special privileges — only the ability to submit a standard V3 transaction to the gateway.

    fn validate_tx_l2_gas_price_within_threshold(
        &self,
        tx_resource_bounds: ValidResourceBounds,
        previous_block_l2_gas_price: NonzeroGasPrice,
    ) -> StatefulTransactionValidatorResult<()> {
        match tx_resource_bounds {
            ValidResourceBounds::AllResources(tx_resource_bounds) => {
                let tx_l2_gas_price = tx_resource_bounds.l2_gas.max_price_per_unit;
                let gas_price_threshold_multiplier =
                    Ratio::new(self.config.min_gas_price_percentage.into(), 100_u128);
                let threshold = (gas_price_threshold_multiplier
                    * previous_block_l2_gas_price.get().0)
                    .to_integer();
                if tx_l2_gas_price.0 < threshold {
                    return Err(StarknetError {
                        // We didn't have this kind of an error.
                        code: StarknetErrorCode::UnknownErrorCode(
                            "StarknetErrorCode.GAS_PRICE_TOO_LOW".to_string(),
                        ),
                        message: format!(
                            "Transaction L2 gas price {tx_l2_gas_price} is below the required \
                             threshold {threshold}.",
                        ),
                    });
                }
            }
            ValidResourceBounds::L1Gas(_) => {
                // No validation required for legacy transactions.
            }
        }
        Ok(())
    }

    l2_gas_used: GasAmount,

        Some(fa) => GasPrice(max(config_min.0, fa.0)),
        None => config_min,
    };
    calculate_next_base_gas_price(current_l2_gas_price, l2_gas_used, gas_target, effective_min)
}
