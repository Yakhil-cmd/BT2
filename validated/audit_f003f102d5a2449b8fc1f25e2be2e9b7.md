### Title
Gateway Skips `__validate__` Signature Check for Invoke Transactions with Nonce=1 Based on Overly Broad Mempool Presence Check — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator is designed to skip the `__validate__` entry point call (which performs account-level signature verification) for invoke transactions with nonce=1 when a deploy_account transaction is pending. However, the predicate used — `account_tx_in_pool_or_recent_block` — returns `true` for any account with **any** transaction in the mempool or any recent committed block, not exclusively for accounts with a pending deploy_account transaction. An unprivileged attacker can exploit this to submit an invoke transaction with an invalid signature for any account with on-chain nonce=0 that has any pending transaction, and the gateway will admit it to the mempool without ever calling `__validate__`.

### Finding Description

In `run_pre_validation_checks`, after nonce and resource-bound checks pass, `skip_stateful_validations` is called:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs:399-410
async fn run_pre_validation_checks(...) -> StatefulTransactionValidatorResult<bool> {
    self.validate_state_preconditions(executable_tx, account_nonce).await?;
    validate_by_mempool(executable_tx, account_nonce, mempool_client.clone()).await?;
    let skip_validate =
        skip_stateful_validations(executable_tx, account_nonce, mempool_client.clone()).await?;
    Ok(skip_validate)
}
```

`skip_stateful_validations` returns `true` (skip) when:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs:434-457
if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
    if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
        return mempool_client
            .account_tx_in_pool_or_recent_block(tx.sender_address())
            .await
            ...
    }
}
```

The comment states: *"It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction **or transactions with future nonces that passed validations**."*

The second clause is the flaw. `account_tx_in_pool_or_recent_block` is implemented as:

```rust
// crates/apollo_mempool/src/mempool.rs:697-700
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
```

And `MempoolState::contains_account`:

```rust
// crates/apollo_mempool/src/mempool.rs:115-117
fn contains_account(&self, address: ContractAddress) -> bool {
    self.staged.contains_key(&address) || self.committed.contains_key(&address)
}
```

This returns `true` for **any** account that has any transaction in the pool, is staged for a block, or appears in the committed nonce history — regardless of whether that transaction is a deploy_account. The function does not distinguish transaction types.

When `skip_validate = true` is returned, `run_validate_entry_point` sets `execution_flags.validate = false`:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs:311-312
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
```

And `StatefulValidator::perform_validations` then short-circuits before calling `__validate__`:

```rust
// crates/blockifier/src/blockifier/stateful_validator.rs:79-81
if !tx.execution_flags.validate {
    return Ok(());
}
```

The transaction is admitted to the mempool with no signature verification.

### Impact Explanation

An attacker who observes that account A (on-chain nonce=0) has any pending transaction in the mempool — for example, a deploy_account submitted by the legitimate user — can craft an invoke transaction with nonce=1 carrying an **arbitrary or invalid signature**. The gateway's stateful validator will skip the `__validate__` entry point call and admit the transaction to the mempool. This directly satisfies the impact criterion: **"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."**

The invalid transaction will eventually fail during block execution when the blockifier calls `__validate__` with `validate=true`, but it has already been admitted past the gateway admission gate, consuming mempool capacity and batcher resources. Additionally, if the attacker races the legitimate user's nonce=1 invoke, the attacker's invalid transaction occupies the (address, nonce=1) slot in the pool, potentially delaying or displacing the legitimate transaction.

### Likelihood Explanation

The trigger conditions are:
1. Target account has on-chain nonce=0 (not yet deployed, or just deployed before first tx commits).
2. The account has **any** transaction in the mempool — the most common case being the deploy_account transaction submitted by the legitimate user as part of the standard deploy_account + invoke UX flow.

Both conditions are observable from the public mempool. No privileged access, no private key for the target account, and no special network position is required. The attacker only needs to monitor the mempool for deploy_account transactions and submit a crafted invoke with nonce=1 before the deploy_account is committed.

### Recommendation

Replace the overly broad `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a **deploy_account** transaction exists for the sender address in the mempool. The mempool should expose a dedicated `has_pending_deploy_account(address)` query, or the gateway should inspect the transaction type of the pending transaction before deciding to skip validation. The current comment acknowledges the broader semantics but does not justify why a future-nonce invoke that passed validation is sufficient grounds to skip signature verification for a different nonce=1 invoke with potentially different calldata and signer.

### Proof of Concept

1. Account A has on-chain nonce=0 (not yet deployed).
2. Legitimate user submits `deploy_account` tx for A (nonce=0) → admitted to mempool; `account_tx_in_pool_or_recent_block(A)` now returns `true`.
3. Attacker constructs an `Invoke V3` transaction for account A with `nonce=1` and a garbage/invalid signature (e.g., `signature = [0x1337]`).
4. Attacker submits the transaction to the gateway.
5. Gateway stateless validator passes (signature length ≤ max, resource bounds non-zero, etc.).
6. `extract_state_nonce_and_run_validations` reads `account_nonce=0` from state.
7. `validate_state_preconditions`: nonce check passes (0 ≤ 1 ≤ 200).
8. `validate_by_mempool`: nonce-too-old check passes.
9. `skip_stateful_validations`: `tx.nonce()==1 && account_nonce==0 && account_tx_in_pool_or_recent_block(A)==true` → returns `true`.
10. `run_validate_entry_point` is called with `skip_validate=true` → `execution_flags.validate=false` → `__validate__` is **never called**.
11. Transaction is admitted to the mempool with an invalid signature.
12. When the batcher later executes the transaction, `__validate__` is called with `validate=true`, fails, and the transaction is rejected — but it has already passed gateway admission.
