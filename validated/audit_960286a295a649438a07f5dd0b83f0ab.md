### Title
Unchecked Precompile Call Return Value in Bridge Withdrawal Burns Tokens Without Completing Transfer - (File: `etc/eth-contracts/contracts/EvmErc20.sol`, `etc/eth-contracts/contracts/EvmErc20V2.sol`)

---

### Summary

Both `EvmErc20.sol` and `EvmErc20V2.sol` burn a user's ERC-20 tokens before calling the `ExitToNear` or `ExitToEthereum` precompile via inline assembly. The return value of the low-level `call()` is captured in a local variable `res` but is **never checked**. If the precompile call fails for any reachable reason, the burn is irreversible and the bridge transfer is never initiated, permanently destroying the user's funds.

---

### Finding Description

In `withdrawToNear()` and `withdrawToEthereum()` in both `EvmErc20.sol` and `EvmErc20V2.sol`, the execution order is:

1. `_burn(msg.sender, amount)` — irreversibly destroys the caller's tokens.
2. An inline assembly `call()` to the precompile address — initiates the bridge transfer.
3. The `res` variable from the assembly block is **never read or checked**.

`EvmErc20.sol`, `withdrawToNear`:
```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);                          // tokens destroyed here

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        // res is never checked — silent failure
    }
}
```

The same pattern appears in `withdrawToEthereum()` in both contracts, and in `EvmErc20V2.sol` for both withdrawal functions. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) 

Per EVM semantics, a low-level `call()` to a precompile that returns an error sets `res = 0` but does **not** revert the calling frame. The `_burn()` that already executed is not rolled back.

The `ExitToNear` precompile (`engine-precompiles/src/native.rs`) returns `Err(ExitError)` — which the EVM translates to `res = 0` — in several reachable conditions:

- `get_nep141_from_erc20` fails: the ERC-20 address has no registered NEP-141 mapping in storage.
- `ExitToNearParams::try_from(input)` fails: the `recipient` bytes encode an invalid NEAR account ID (too long, invalid characters, etc.).
- `exit_erc20_token_to_near` returns an error (e.g., `ERR_ETH_ATTACHED_FOR_ERC20_EXIT`, `ERR_INVALID_MESSAGE`). [5](#0-4) [6](#0-5) [7](#0-6) 

---

### Impact Explanation

When the precompile call fails silently:

- The user's ERC-20 tokens are permanently burned on the Aurora EVM side.
- The corresponding NEP-141 tokens that were locked in the Aurora contract when the ERC-20 was originally minted are never released.
- The user receives nothing on the NEAR or Ethereum side.

This constitutes **permanent freezing of funds** (the locked NEP-141 tokens can never be recovered) and effective **destruction of user funds** (the ERC-20 balance is gone with no recourse). Impact is Critical.

---

### Likelihood Explanation

The `recipient` parameter in `withdrawToNear()` is a raw `bytes` value supplied entirely by the caller. A user who passes a recipient string that fails NEAR account ID validation (e.g., exceeds the maximum length, contains uppercase letters, or contains invalid characters) will trigger a precompile error. This is a realistic user mistake with no on-chain guard preventing it. The `withdrawToEthereum()` path is less likely to fail given the fixed 20-byte address format, but the same structural defect exists. Likelihood is Medium-High.

---

### Recommendation

Check the return value of the assembly `call()` and revert if it is zero. This mirrors the fix recommended in the external report (checking return values of external calls):

```solidity
assembly {
    let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

Apply this fix to all four assembly blocks across `EvmErc20.sol` and `EvmErc20V2.sol`. Alternatively, restructure the functions to call the precompile first and only burn on success, though this is not possible with the current burn-then-bridge design without a refund mechanism.

---

### Proof of Concept

1. Deploy `EvmErc20` (or `EvmErc20V2`) on Aurora with a valid NEP-141 backing.
2. Mint tokens to `attacker`.
3. Call `withdrawToNear(invalidRecipient, amount)` where `invalidRecipient` is a byte string that fails NEAR account ID validation (e.g., `"INVALID ACCOUNT!!"`).
4. The `_burn()` executes successfully, reducing `attacker`'s balance to zero.
5. The precompile call returns `res = 0` (failure) because `ExitToNearParams::try_from` or `get_nep141_from_erc20` rejects the input.
6. The assembly block exits without reverting.
7. The transaction succeeds on-chain. The user's ERC-20 tokens are gone. No NEP-141 tokens are received. The underlying NEP-141 tokens remain locked in the Aurora contract permanently. [1](#0-0) [8](#0-7)

fn get_nep141_from_erc20<I: IO>(erc20_token: &[u8], io: &I) -> Result<AccountId, ExitError> {
    AccountId::try_from(
        io.read_storage(bytes_to_key(KeyPrefix::Erc20Nep141Map, erc20_token).as_slice())
            .map(|s| s.to_vec())
            .ok_or(ExitError::Other(Cow::Borrowed(ERR_TARGET_TOKEN_NOT_FOUND)))?,
    )
    .map_err(|_| ExitError::Other(Cow::Borrowed("ERR_INVALID_NEP141_ACCOUNT")))
}
