### Title
Invoke transactions with invalid signatures bypass gateway `__validate__` via `skip_stateful_validations` — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function unconditionally suppresses the account's `__validate__` entry-point call for any invoke with `nonce == 1` targeting an undeployed address that has *any* pending transaction in the mempool. An unprivileged attacker who observes a victim's `deploy_account` in the mempool can inject an invoke carrying an arbitrary/invalid signature for the victim's address, which the gateway admits without any signature verification.

---

### Finding Description

In `skip_stateful_validations` [1](#0-0) , when three conditions hold simultaneously:

1. The incoming transaction is `Invoke` with `nonce == Felt::ONE`
2. The on-chain account nonce is `Nonce(Felt::ZERO)` (account not yet deployed)
3. `account_tx_in_pool_or_recent_block` returns `true` for the sender address

…the function returns `true`. This causes `run_validate_entry_point` to be invoked with `execution_flags.validate = false` [2](#0-1) , completely suppressing the call to the account's `__validate__` entry point during gateway stateful validation.

The `account_tx_in_pool_or_recent_block` check only verifies that *some* transaction exists for the address in the mempool or a recent block — it does not verify that the pending transaction is specifically a `deploy_account`, nor does it verify any relationship between the deploy_account's class/constructor and the invoke's signature. [3](#0-2) 

The gateway's `add_tx_inner` flow is:

1. Stateless validation (signature length, calldata size, DA modes) — does not check signature validity [4](#0-3) 
2. `convert_rpc_tx_to_internal_and_executable_txs` — computes tx hash, no signature check [5](#0-4) 
3. `extract_state_nonce_and_run_validations` → `run_pre_validation_checks` → `validate_nonce` passes (nonce=1 is within `[account_nonce, account_nonce + max_gap]` when `account_nonce=0`) → `validate_by_mempool` passes (mempool checks nonce gaps and fee bounds, not signature validity) → `skip_stateful_validations` returns `true` → `__validate__` skipped [6](#0-5) 
4. Invalid invoke is forwarded to the mempool via `mempool_client.add_tx` [7](#0-6) 

The `validate: false` flag is a gateway-only execution flag. During actual block execution by the batcher/blockifier, `AccountTransaction` is constructed with independent `ExecutionFlags` where `validate: true`, so `__validate__` is called at execution time — but by then the transaction is already in the mempool and will be included in a block.

---

### Impact Explanation

Once the invalid invoke is in the mempool, it is included in a block after the `deploy_account` executes. During blockifier execution `__validate__` is called with `validate: true`; the invalid signature causes it to fail. Under Starknet protocol semantics a failed `__validate__` still charges a fee (bounded by the validation gas limit) to prevent DoS. The attacker can therefore drain the victim's newly-deployed account by submitting many such invalid invokes — each consuming a fee from the victim — without controlling the account or spending any funds themselves.

Matching impact: **High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing**, with secondary economic impact (fee drain) on the victim.

---

### Likelihood Explanation

The mempool is observable via P2P broadcast and RPC. Any `deploy_account` transaction is visible before it is included in a block. The attack requires no privileged access, no special knowledge beyond the victim's address (derivable from the deploy_account transaction itself), and no on-chain funds from the attacker. The exploitation window is the time between the `deploy_account` entering the mempool and being mined.

---

### Recommendation

Replace the unconditional skip of `__validate__` with a more targeted guard:

- Verify that the pending mempool transaction for the address is specifically a `deploy_account` (not just any transaction), and
- Perform at minimum a lightweight signature format/range check (e.g., ECDSA point validity) even when skipping full entry-point execution, or
- Require the invoke's signature to be verifiable against the public key committed in the deploy_account's constructor calldata before skipping `__validate__`.

This preserves the UX benefit (deploy_account + invoke submitted together) while closing the window for arbitrary-signature injection.

---

### Proof of Concept

```
1. Victim submits:
     DeployAccount(address=A, class=C, nonce=0, sig=<valid>)
   → Mempool now contains a deploy_account for address A.

2. Attacker observes address A in the mempool (via RPC/P2P).

3. Attacker submits:
     Invoke(sender_address=A, nonce=1, calldata=[...], sig=[0xdead, 0xbeef])

4. Gateway processing:
   - Stateless: sig length=2 ≤ max_signature_length          → PASS
   - validate_nonce: account_nonce=0, incoming=1,
       0 ≤ 1 ≤ 0+max_gap                                     → PASS
   - validate_by_mempool: nonce gap OK, fee OK                → PASS
   - skip_stateful_validations:
       nonce==1 ∧ account_nonce==0
       ∧ account_tx_in_pool_or_recent_block(A)==true          → returns true
   - run_validate_entry_point(skip_validate=true):
       execution_flags.validate = false
       → __validate__ NOT called                              → PASS

5. Invalid invoke admitted to mempool.

6. Block N:   deploy_account executes → A deployed, funded with F tokens.
   Block N+k: invalid invoke executes → __validate__ fails
              → fee charged to A (victim pays, attacker pays nothing).

7. Attacker repeats step 3 with nonce=1 replacements (or nonce=2,3,…
   using the same skip path for subsequent nonces already in mempool)
   until A is drained.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
