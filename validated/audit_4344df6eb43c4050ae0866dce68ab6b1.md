### Title
Malicious Participant Can Silently Corrupt CKD Output via Unverified Contributions — (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

In the Confidential Key Derivation (CKD) protocol, the coordinator aggregates each participant's `(big_y, big_c)` contribution by simple addition, with no cryptographic proof that the values were honestly computed. A single malicious participant can send arbitrary group elements, silently shifting the final `(Y, C)` ElGamal ciphertext to an attacker-chosen value. The coordinator accepts the corrupted output as valid, causing honest parties to derive an incorrect confidential key. This maps directly to the external report's root cause: an unverified externally-supplied value is used in a critical aggregation without accounting for the dynamic state that would expose the manipulation.

---

### Finding Description

`do_ckd_coordinator` collects each participant's share of the CKD output and sums them unconditionally:

```rust

{

    norm_big_c += participant_output.big_c();
}
let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
``` [1](#0-0) 

The honest computation each participant is *supposed* to perform is:

```
norm_big_y_i = λ_i · y_i · G
norm_big_c_i = λ_i · (x_i · H(pk ‖ app_id) + y_i · A)
```

as implemented in `compute_signature_share`: [2](#0-1) 

There is **no zero-knowledge proof, no commitment-then-reveal, and no consistency check** that enforces this relationship. The coordinator has no way to distinguish a correctly-formed `(norm_big_y_i, norm_big_c_i)` from an arbitrary pair of group elements, because:

- `y_i` is a fresh random nonce known only to participant `i`; the coordinator cannot recompute `λ_i · y_i · G` independently.
- The coordinator does not hold `app_sk`, so it cannot verify `norm_big_c_i - norm_big_y_i · app_sk = λ_i · x_i · H(pk ‖ app_id)`.

**Analog to the external report:** The inflation attack works because `health()` does not account for the dynamic fee that would make the reverse trade expensive — the check is evaluated against a state that does not reflect the true cost. Here, the coordinator's aggregation does not account for the *cryptographic constraint* that each contribution must satisfy — the check (summation) is evaluated against values that may not reflect the true secret-share relationship.

---

### Impact Explanation

The final CKD output `(Y, C)` is an ElGamal encryption of `msk · H(pk ‖ app_id)` under `app_pk`. When a malicious participant `i` sends `(big_y_i', big_c_i')` instead of the correct `(norm_big_y_i, norm_big_c_i)`, the coordinator computes:

```
Y'  = Y_honest + Δ_y          where Δ_y = big_y_i' − norm_big_y_i
C'  = C_honest + Δ_c          where Δ_c = big_c_i' − norm_big_c_i
```

The app then unmasks via `C' − app_sk · Y'`, obtaining:

```
msk · H(pk ‖ app_id) + (Δ_c − app_sk · Δ_y)
```

which is not the correct confidential key unless `Δ_c = app_sk · Δ_y` (a condition the attacker cannot satisfy without knowing `app_sk`). The coordinator accepts this corrupted `CKDOutput` as valid — there is no post-aggregation verification step. Honest parties therefore derive an **incorrect, unusable confidential key**, matching the allowed High impact: *"Corruption of CKD outputs so honest parties accept inconsistent or unusable cryptographic outputs."*

---

### Likelihood Explanation

The attack requires only that one participant in the CKD session behaves maliciously. No special privilege, no leaked key, and no external oracle is needed. The malicious participant simply deviates from `compute_signature_share` and sends an arbitrary `CKDOutput` to the coordinator via the private channel. The coordinator has no mechanism to detect or attribute the deviation. The attack is therefore reachable by any unprivileged malicious participant.

---

### Recommendation

Each participant should accompany their `(norm_big_y_i, norm_big_c_i)` with a non-interactive zero-knowledge proof of correct formation. A standard Chaum-Pedersen / sigma-protocol proof suffices: prove knowledge of `(x_i, y_i)` such that:

```
norm_big_y_i = λ_i · y_i · G
norm_big_c_i = λ_i · x_i · H(pk ‖ app_id) + λ_i · y_i · A
```

using the participant's public verification share (from the DKG output) as the public commitment to `x_i`. The coordinator must verify all proofs before aggregating. Alternatively, a commit-then-reveal round can bind each participant to their `y_i` before the coordinator sees any contribution.

---

### Proof of Concept

1. Run a 3-of-3 CKD session. Participants P1, P2 are honest; P3 is malicious.
2. P1 and P2 call `compute_signature_share` correctly and send their outputs privately to the coordinator.
3. P3 sends `(ElementG1::identity(), ElementG1::identity())` — the additive identity — instead of its correct contribution.
4. The coordinator executes the loop at lines 50–55 and accumulates:
   - `norm_big_y = λ_1·Y_1 + λ_2·Y_2 + 0`
   - `norm_big_c = λ_1·C_1 + λ_2·C_2 + 0`
5. `CKDOutput::new(norm_big_y, norm_big_c)` is returned with no error.
6. The app calls `ckd_output.unmask(app_sk)` and obtains `(msk − λ_3·x_3) · H(pk ‖ app_id)` — a value that differs from the correct confidential key by exactly `λ_3 · x_3 · H(pk ‖ app_id)`.
7. The honest parties have accepted an incorrect, unusable CKD output with no indication of failure. [3](#0-2) [4](#0-3)

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}
