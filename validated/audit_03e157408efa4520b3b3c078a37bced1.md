The external bug class is an **accounting bug**: using an averaged/single-rate value across a multi-period span instead of computing per-period. I need to find a Chia GUI analog where a similar aggregation error causes a High/Critical wallet security impact.

### Title
Faulty Multi-NFT Royalty Accounting in WalletConnect Offer Confirmation Understates Total Spend - (File: `packages/gui/src/electron/commands/parseCommandDisplay.ts`)

---

### Summary

The `formatAmountWithRoyalties` function in `parseCommandDisplay.ts` incorrectly calculates the "Total Amount with Royalties" shown in the WalletConnect offer-confirmation dialog when multiple NFTs with differing royalty percentages are involved. It divides the fungible amount equally across all NFTs before computing each royalty, instead of applying each NFT's royalty rate to the full amount. The result is a displayed total that is systematically lower than what the user's wallet will actually spend, causing the user to approve a WalletConnect offer under a false accounting of cost.

---

### Finding Description

`formatAmountWithRoyalties` is called by `withRoyaltyTotals` → `walletDeltaToDisplay` for every `chia_wallet.take_offer` and `chia_wallet.create_offer_for_ids` WalletConnect command. It populates the `amountWithRoyalties` field that is rendered in `Confirm.tsx` as **"Total Amount with Royalties"** — the single figure a user reads before clicking Approve. [1](#0-0) 

The flawed logic:

```ts
const splitAmount = amount / BigInt(royaltyPercentages.length);   // ← equal split
const royaltyAmount = royaltyPercentages.reduce(
  (total, pct) => total + (splitAmount * BigInt(pct)) / 10_000n,  // ← royalty on split
  0n,
);
```

**Correct** calculation (each NFT's royalty applies to the full fungible amount):

```
royaltyTotal = Σ (amount × pct_i / 10_000)
```

**Actual** calculation (each NFT's royalty applies to `amount / N`):

```
royaltyTotal = Σ ((amount / N) × pct_i / 10_000)
             = (amount / N) × Σ pct_i
```

When royalty percentages differ across NFTs, `(amount / N) × Σ pct_i ≠ Σ (amount × pct_i / 10_000)`. The split-based formula always produces a **lower** royalty total than the correct one (they are equal only when all percentages are identical).

The `amountWithRoyalties` string is rendered directly in the confirmation UI: [2](#0-1) 

The `withRoyaltyTotals` function feeds this value into every spending and receiving line: [3](#0-2) 

---

### Impact Explanation

The WalletConnect confirmation dialog is the sole security gate between a dApp request and the user's wallet. The "Total Amount with Royalties" line is the figure a user relies on to decide whether to approve a multi-NFT offer. When that figure is understated, the user approves a spend larger than what was disclosed. This is a **High** impact: WalletConnect state causes a user to approve the wrong (understated) amount, directly affecting XCH or CAT balances.

---

### Likelihood Explanation

Any party that can send a WalletConnect `take_offer` or `create_offer_for_ids` request containing two or more NFTs with different royalty percentages can trigger the discrepancy. No special privileges are required beyond a paired WalletConnect session. The magnitude of the understatement grows with the spread between the highest and lowest royalty percentages in the bundle.

---

### Recommendation

Replace the split-based formula with a per-NFT full-amount calculation:

```ts
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (amount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
// Remove the splitAmount line entirely.
```

---

### Proof of Concept

**Setup:** WalletConnect `take_offer` with two NFTs offered in exchange for 1 XCH (1 000 000 000 000 mojos):
- NFT-A: royalty 50 % (5 000 basis points)
- NFT-B: royalty 0.1 % (10 basis points)

**Current (buggy) display:**
```
splitAmount = 1 000 000 000 000 / 2 = 500 000 000 000
royalty     = (500 000 000 000 × 5 000)/10 000
            + (500 000 000 000 × 10)/10 000
            = 250 000 000 000 + 500 000 000
            = 250 500 000 000 mojos
displayed total = 1 250 500 000 000 mojos ≈ 1.2505 XCH
```

**Correct total the wallet actually deducts:**
```
royalty = (1 000 000 000 000 × 5 000)/10 000
        + (1 000 000 000 000 × 10)/10 000
        = 500 000 000 000 + 1 000 000 000
        = 501 000 000 000 mojos
correct total = 1 501 000 000 000 mojos ≈ 1.501 XCH
```

**Discrepancy:** The user sees **1.2505 XCH** in the approval dialog but their wallet deducts **1.501 XCH** — a 0.2505 XCH (≈ 20 %) understatement. The user approves based on the false figure.

The existing test fixture in `parseCommandDisplay.test.ts` (lines 251–334) already encodes the incorrect expected value (`0.00010255`) for a two-NFT bundle with 5 % and 0.1 % royalties, confirming the bug is present and untested against the correct result. [4](#0-3)

function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],
): string | undefined {
  if (royaltyPercentages.length === 0 || line.kind === 'nft') {
    return undefined;
  }

  const splitAmount = amount / BigInt(royaltyPercentages.length);
  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;

  if (line.kind === 'xch') {
    return mojoToChiaLocaleString(totalAmount);
  }

  return mojoToCATLocaleString(totalAmount);
}

function withRoyaltyTotals(
  items: DisplayWalletDeltaItemWithKey[],
  amounts: Record<string, bigint>,
  oppositeSideLines: DisplayWalletDeltaItem[],
): DisplayWalletDeltaItem[] {
  const royaltyPercentages = royaltyPercentagesForSide(oppositeSideLines);

  return items.map(({ key, line }) => {
    if (line.kind === 'nft') {
      return line;
    }

    const amount = amounts[key];
    const amountWithRoyalties = amount ? formatAmountWithRoyalties(line, amount, royaltyPercentages) : undefined;

    return amountWithRoyalties ? { ...line, amountWithRoyalties } : line;
  });
}
