### Title
Unchecked Precompile Call Return Value After Token Burn Causes Permanent Fund Loss - (File: `etc/eth-contracts/contracts/EvmErc20.sol`, `etc/eth-contracts/contracts/EvmErc20V2.sol`)

---

### Summary

`EvmErc20.withdrawToNear()` and `EvmErc20.withdrawToEthereum()` (and their `EvmErc20V2` counterparts) burn the caller's ERC-20 tokens **before** invoking the exit precompile via a low-level assembly `call`. The return value of that `call` is stored in `res` but **never checked**. If the precompile call fails for any reason, `res = 0` is silently ignored, the burn is not reverted, and the user's tokens are permanently destroyed with no corresponding NEP-141 transfer or Ethereum withdrawal.

---

### Finding Description

In `EvmErc20.sol`:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // ← tokens destroyed here

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        // res is NEVER CHECKED — no `if iszero(res) { revert(0,0) }`
    }
}
``` [1](#0-0) 

The same pattern appears in `withdrawToEthereum()`: [2](#0-1) 

And identically in `EvmErc20V2.sol`: [3](#0-2) [4](#0-3) 

The precompile at `0xe9217bc70b7ed1f598ddd3199e80b093fa71124f` is the `ExitToNear` precompile. It can return a non-fatal `ExitError::Other(...)` in several cases, including `ERR_TARGET_TOKEN_NOT_FOUND` when `get_nep141_from_erc20` fails to find a NEP-141 mapping for the calling ERC-20 address: [5](#0-4) 

When the precompile returns a non-fatal `ExitError`, the EVM sub-call returns `res = 0` to the caller. Because `EvmErc20` never checks `res`, execution continues normally after the burn. No promise is ever scheduled, no refund is triggered, and the tokens are gone.

The `error_refund` feature only handles the case where the NEAR-side `ft_transfer` **promise** fails after the EVM transaction has already committed: [6](#0-5) 

It does not protect against the EVM-side precompile call itself returning failure — that path never schedules any promise at all, so no callback is ever invoked.

---

### Impact Explanation

**Critical — Permanent freezing/loss of user funds.**

If the precompile call fails silently:
- The ERC-20 tokens are burned and removed from the user's balance permanently.
- No NEP-141 tokens are transferred to the NEAR recipient.
- No ETH is withdrawn to Ethereum.
- The `error_refund` callback mechanism is never triggered because no promise was scheduled.
- There is no recovery path.

---

### Likelihood Explanation

**Low-Medium.** For properly registered `EvmErc20` contracts deployed by Aurora, the NEP-141 mapping is set up at deployment time and the precompile should succeed under normal conditions. However, the failure path is reachable when:

1. A custom ERC-20 contract (not deployed by Aurora's factory) calls the precompile directly — `get_nep141_from_erc20` returns `ERR_TARGET_TOKEN_NOT_FOUND`, precompile returns `ExitError::Other`, `res = 0`, burn is not reverted.
2. The NEP-141 mapping for a registered ERC-20 is removed or corrupted after deployment.
3. Any other precompile-internal error path (e.g., input parsing failure) that returns a non-fatal `ExitError`.

Any EVM user holding tokens in an `EvmErc20` contract can trigger this by calling `withdrawToNear` or `withdrawToEthereum`.

---

### Recommendation

Check the return value of the precompile `call` and revert if it fails. The burn should either happen **after** a successful precompile call, or the assembly block must revert on failure:

```diff
 assembly {
     let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
+    if iszero(res) { revert(0, 0) }
 }
```

Alternatively, restructure to call the precompile first and only burn on success.

---

### Proof of Concept

1. Deploy a custom ERC-20 contract that inherits `EvmErc20` but is **not** registered in Aurora's NEP-141 factory map (so `get_nep141_from_erc20` will return `ERR_TARGET_TOKEN_NOT_FOUND`).
2. Mint tokens to `msg.sender`.
3. Call `withdrawToNear(recipient, amount)`.
4. `_burn(msg.sender, amount)` executes — balance decremented.
5. The assembly `call` to the `ExitToNear` precompile executes; the precompile returns `ExitError::Other("ERR_TARGET_TOKEN_NOT_FOUND")` → sub-call returns `res = 0`.
6. The function returns normally without reverting.
7. Observe: ERC-20 balance is zero, no NEP-141 tokens were transferred, no refund was issued. Funds are permanently lost. [5](#0-4) [1](#0-0)
