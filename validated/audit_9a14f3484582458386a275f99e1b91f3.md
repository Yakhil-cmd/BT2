Audit Report

## Title
Wasm Trap in `canister_status` Callback Permanently Leaks `SlotLoan` via IC Heap Rollback — (`rs/nns/handlers/root/impl/canister/canister.rs`)

## Summary

The public `canister_status` update endpoint in the NNS root canister calls `.unwrap()` on the management canister result after the `SlotLoan` RAII guard has already been dropped within the same callback execution. Because the IC rolls back all heap mutations when a callback traps, the slot restoration performed by `SlotLoan::drop` is undone. An unprivileged caller can exhaust all 167 slots by repeatedly triggering this trap, permanently disabling management canister call capacity for all callers — including NNS governance — until the canister is upgraded.

## Finding Description

**Entry point — no access control:**
`canister_status` at `canister.rs:88-98` is a public `#[update]` with no caller restriction. Any principal can invoke it with an arbitrary `CanisterIdRecord`.

**Slot borrow and RAII guard:**
`LimitedOutstandingCallsManagementCanisterClient::canister_status` at `management_canister_client.rs:295-301` calls `try_borrow_slot()`, which decrements `AVAILABLE_MANAGEMENT_CANISTER_CALL_SLOT_COUNT` and returns a `SlotLoan`. This decrement is committed to the heap before the inter-canister `await`.

**Drop ordering — `_loan` drops before `.unwrap()`:**
`_loan` is a local variable scoped to `LimitedOutstandingCallsManagementCanisterClient::canister_status`. When that async function completes and returns its `Result` to the outer `.await` in `canister.rs:92-95`, `_loan` goes out of scope and `SlotLoan::drop` increments `available_slot_count` back. Only after this does execution resume in `canister.rs` at the `.unwrap()` on line 97.

**Execution sequence in Message 2 (callback) when management canister rejects:**
```
[heap at start of callback: available_slot_count = N-1]
  1. inner.canister_status returns Err(...)
  2. _loan is dropped → available_slot_count incremented to N  ← heap mutation
  3. Err propagates back to canister.rs::canister_status
  4. .map(CanisterStatusResult::from) → still Err
  5. .unwrap() panics → Wasm TRAP
  6. IC rolls back Message 2 heap → available_slot_count reverts to N-1
```

Step 6 is mandated by the IC interface specification: when a response callback traps, the canister's state is rolled back to the state at the beginning of that callback. Step 2's increment is undone. The slot is permanently leaked.

**Triggering the rejection:**
The management canister's `canister_status` endpoint requires the caller to be a controller of the target canister. NNS root is not a controller of arbitrary user canisters, so passing any user-owned canister ID causes a guaranteed rejection.

**Exhaustion check:**
`try_borrow_slot` at `management_canister_client.rs:269-276` returns `Err(SysTransient, "Unavailable...")` when `available_slot_count == 0`. Critically, this check applies to **all** callers regardless of VIP status — the VIP flag only controls `used_slot_count` (0 vs 1), not the availability gate. Once exhausted, NNS governance-initiated calls through `LimitedOutstandingCallsManagementCanisterClient` also fail.

## Impact Explanation

After 167 trapped callbacks, `AVAILABLE_MANAGEMENT_CANISTER_CALL_SLOT_COUNT` reaches 0. Every subsequent call to any method on `LimitedOutstandingCallsManagementCanisterClient` — including governance-initiated `change_canister_controllers`, `update_canister_settings`, `take_canister_snapshot`, and `load_canister_snapshot` — returns `Err(SysTransient, "Unavailable")` immediately without making any inter-canister call. The NNS root canister's management canister call capacity is permanently disabled until a canister upgrade resets the thread-local. This matches **Application/platform-level DoS with concrete NNS infrastructure harm** — High ($2,000–$10,000).

## Likelihood Explanation

- The `canister_status` endpoint is public with no access control.
- Triggering a management canister rejection requires only passing a canister ID that NNS root does not control — trivially any user-owned canister.
- Only 167 update calls are required. Each costs only standard ingress fees.
- No privileged access, key material, or subnet-majority corruption is required.
- The attack is fully deterministic and repeatable.

## Recommendation

Replace `.unwrap()` with proper error propagation so the endpoint returns a rejection to the caller instead of trapping:

```rust
// canister.rs:97 — instead of:
canister_status_response.unwrap()

// Use:
canister_status_response.map_err(|(_, msg)| ic_cdk::trap(&msg))
// or change the return type to Result<CanisterStatusResult, ...>
```

This ensures that when the management canister rejects, the error is returned as a canister rejection (not a trap), so the IC does not roll back the heap and `SlotLoan::drop`'s increment is preserved.

## Proof of Concept

1. Obtain any canister ID that NNS root does not control (e.g., a freshly created user canister on mainnet or a local replica).
2. Send 167 update calls to NNS root's `canister_status` endpoint with that canister ID.
3. Each call: management canister rejects with `Err` → `_loan` drops (slot restored) → `.unwrap()` traps → IC rolls back callback heap → slot leaked.
4. On the 168th call from any principal (including NNS governance canisters) to any management canister method, observe `Err(SysTransient, "Unavailable. Maybe, try again later?")` returned immediately.
5. Verifiable via a PocketIC integration test: configure a `LimitedOutstandingCallsManagementCanisterClient` with a mock inner client that always returns `Err`, call `canister_status` 167 times, assert `available_slot_count == 0` after each trapped callback.
