### Title
`skip_stateful_validations` Admits Invoke Transactions Without Signature Verification When Any Mempool Transaction Exists for the Account - (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The gateway's `skip_stateful_validations` function uses `account_tx_in_pool_or_recent_block` as a proxy for "a deploy_account transaction exists for this account." However, that predicate returns `true` for **any** transaction from the account in the mempool — not only deploy_account transactions. An attacker who observes that a target account (with on-chain nonce = 0) has any pending transaction in the mempool can submit an invoke transaction with nonce = 1 and an arbitrary/invalid signature for that account. The gateway skips the `__validate__` entry-point call and admits the transaction to the mempool without signature verification.

### Finding Description

**Root cause — overly broad proxy check**

`skip_stateful_validations` in `crates/apollo_gateway/src/stateful_transaction_validator.rs` (lines 429–461) fires when three conditions are simultaneously true:

1. The incoming transaction is an Invoke with `tx.nonce() == Nonce(Felt::ONE)`
2. The on-chain account nonce is `Nonce(Felt::ZERO)`
3. `mempool_client.account_tx_in_pool_or_recent_block(sender)` returns `true` [1](#0-0) 

When all three hold, the function returns `true` (skip validation), and `extract_state_nonce_and_run_validations` calls `run_validate_entry_point` with `skip_validate = true`, meaning the account's `__validate__` entry point — which performs signature verification — is **never executed** at the gateway. [2](#0-1) 

**Why the proxy is too broad**

`account_tx_in_pool_or_recent_block` returns `true` if the account has **any** transaction in the pool or appears in the mempool's committed/staged state: [3](#0-2) 

The code comment claims this is sufficient because "it means that either it has a deploy_account transaction or transactions with future nonces that passed validations." This reasoning is circular: a nonce-1 invoke transaction that itself triggered the skip would also satisfy the predicate for a subsequent attacker transaction.

More concretely, a **legitimate nonce-0 invoke** from account A (which passes full validation) placed in the mempool is enough to make `account_tx_in_pool_or_recent_block(A)` return `true`. A subsequent attacker-submitted nonce-1 invoke for account A with an invalid signature then satisfies all three conditions and is admitted without `__validate__`.

**Exploit path**

1. Account A exists on-chain with nonce = 0 (deployed, or freshly created).
2. Account A submits a legitimate invoke tx with nonce = 0; it passes full gateway validation and enters the mempool.
3. Attacker submits `invoke(sender=A, nonce=1, signature=<garbage>)`.
4. Gateway `validate_nonce`: `0 ≤ 1 ≤ 200` → passes. [4](#0-3) 
5. `validate_by_mempool` → `MempoolState::validate_incoming_tx` only checks `tx_nonce < account_nonce`; nonce 1 ≥ 0 → passes. [5](#0-4) 
6. `skip_stateful_validations`: nonce=1, account_nonce=0, `account_tx_in_pool_or_recent_block(A)` = `true` (step 2) → returns `true`.
7. `run_validate_entry_point` is called with `skip_validate = true` → `__validate__` is **not called**.
8. Attacker's transaction with invalid signature is admitted to the mempool.

The attacker can repeat step 3 with different transaction hashes (different calldata/tip) to flood the mempool with invalid transactions for account A, since `validate_incoming_tx` only rejects by `NonceTooOld` or `DuplicateTransaction` (same hash), not by nonce collision. [6](#0-5) 

### Impact Explanation

**Admission of unauthorized transactions (High):** The gateway accepts transactions with invalid signatures for any account with on-chain nonce = 0 that has a pending transaction in the mempool. This violates the invariant that every admitted transaction must pass the account's `__validate__` entry point before entering the mempool.

When the batcher later executes these transactions, `__validate__` runs (execution flags default to `validate = true`) and fails, causing the transaction to be rejected. However:

- The mempool is polluted with invalid transactions that consume pool capacity.
- The batcher wastes execution resources processing transactions that will always fail validation.
- An attacker can continuously submit invalid transactions for any active account (nonce=0 with pending txs) at zero cost (no fee is charged for gateway-rejected transactions).

The impact maps to: **High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

### Likelihood Explanation

The precondition (account with nonce=0 having any pending transaction) is common during normal operation — every newly deployed account that submits its first transaction satisfies it. The attack requires no privileged access, no special keys, and no knowledge of the victim's private key. Any external observer of the mempool can trigger it.

### Recommendation

Replace the overly broad `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a **deploy_account** transaction exists for the account. The mempool should expose a dedicated predicate such as `deploy_account_tx_in_pool(address)` that only returns `true` when a deploy_account transaction (not any transaction) is present for the given address. Alternatively, tag transactions in the pool by type and filter accordingly inside `skip_stateful_validations`.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
