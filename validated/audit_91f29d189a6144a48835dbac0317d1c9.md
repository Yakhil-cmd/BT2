### Title
Cross-COA Signature Replay in `validateCOAOwnershipProof` — (`fvm/evm/stdlib/contract.cdc`)

### Summary

`EVM.validateCOAOwnershipProof` verifies a Cadence account's ownership of a specific EVM address (COA) by checking cryptographic signatures over caller-supplied `signedData`. Because the function never enforces that `signedData` encodes the target `evmAddress`, a malicious transaction sender can replay a signature the user produced for one COA to prove ownership of a *different* COA on the same Cadence account. This is the direct Flow analog of the Solidity bug: two distinct operations share the same signed payload, so a relayer can substitute one for the other.

---

### Finding Description

`validateCOAOwnershipProof` accepts six parameters: `address`, `path`, `signedData`, `keyIndices`, `signatures`, and `evmAddress`. It performs two independent checks:

1. **Signature check** — verifies that the supplied signatures over `signedData` (domain tag `"FLOW-V0.0-user"`) meet the account's key-weight threshold.
2. **COA address check** — borrows the `CadenceOwnedAccount` resource at `path` and confirms its EVM address equals `evmAddress`.

The two checks are never linked: the function never requires that `signedData` encodes `evmAddress` or `path`.

```cadence
// fvm/evm/stdlib/contract.cdc  lines 1082-1086
let isValid = keyList.verify(
    signatureSet: signatureSet,
    signedData: signedData,          // ← no evmAddress inside
    domainSeparationTag: "FLOW-V0.0-user"
)
```

The code itself acknowledges the gap:

> *"Note: this function does not enforce that `signedData` includes `evmAddress`. In principle, a signature produced for one purpose could be replayed here against a different COA owned by the same Cadence account."* [1](#0-0) 

**Attack path (unprivileged transaction sender / Cadence script author):**

1. A Cadence account `A` owns two COAs: `COA-X` at `/public/coaX` and `COA-Y` at `/public/coaY`.
2. A user signs `signedData = <some_bytes>` intending to prove ownership of `COA-X` (e.g., to authenticate with a dApp that calls `validateCOAOwnershipProof` directly).
3. A malicious relayer submits a Cadence transaction or script that calls:
   ```cadence
   EVM.validateCOAOwnershipProof(
       address: A,
       path: /public/coaY,          // ← different path
       signedData: <same_bytes>,    // ← same signed data
       keyIndices: [...],
       signatures: [...],           // ← same signatures
       evmAddress: COA-Y.address    // ← different EVM address
   )
   ```
4. Check 1 passes: the signatures are valid over `signedData` (same key, same data, same domain tag).
5. Check 2 passes: the COA at `/public/coaY` does have `COA-Y`'s EVM address.
6. The function returns `isValid: true` — the relayer has proven ownership of `COA-Y` using a signature the user produced for `COA-X`.

The entry point is any Cadence script or transaction; `validateCOAOwnershipProof` is `access(all)` and callable by any unprivileged sender. [2](#0-1) [3](#0-2) 

---

### Impact Explanation

Any dApp or protocol that calls `EVM.validateCOAOwnershipProof` directly (rather than through the EVM-side ERC-1271 precompile) and relies on the result to authorize an action tied to a specific EVM address can be deceived. A relayer can substitute the user's intended COA for any other COA the same Cadence account owns, without the user's consent. This breaks the trustless property of COA ownership proofs: the user cannot safely delegate proof submission to a third party.

**Impact: Medium** — the user cannot specify which COA they are proving ownership of when using signatures via a relayer. A relayer can perform a different action (prove a different COA) than what the user intended.

---

### Likelihood Explanation

**Likelihood: Medium** — the preconditions are:
- The target Cadence account must hold more than one COA at different public paths (not the common case today, but not prohibited by the protocol).
- The victim must be using a relayer/transaction sender rather than submitting the proof themselves.
- The dApp must call `validateCOAOwnershipProof` directly rather than relying solely on the EVM-side `isValidSignature` / `verifyCOAOwnershipProof` precompile (which always passes `address(this)` and is therefore not affected).

Users who submit their own transactions are unaffected.

---

### Recommendation

Bind the signed payload to the specific COA being proven. The simplest fix is to require that `signedData` encodes `evmAddress` (and optionally `path`) before verifying signatures, analogous to adding a unique salt per operation:

```cadence
// Inside validateCOAOwnershipProof, before keyList.verify():
// Reconstruct the expected signed payload and compare it to signedData.
let expectedData = evmAddress.bytes.concat(address.toBytes())
assert(signedData == expectedData,
    message: "signedData must encode the target evmAddress")
```

Alternatively, document a canonical encoding that all callers must use and enforce it on-chain, so that a signature for `COA-X` cannot be accepted as a proof for `COA-Y`. [3](#0-2) 

---

### Proof of Concept

**Setup:** Cadence account `0xABCD` has two COAs:
- `COA-X` at `/public/coaX` with EVM address `0xAAAA…`
- `COA-Y` at `/public/coaY` with EVM address `0xBBBB…`

**Step 1 — User signs for COA-X:**
```
signedData = bytes("prove-coaX-ownership")
signature  = Sign(signedData, userKey, "FLOW-V0.0-user")
```

**Step 2 — Malicious relayer submits for COA-Y:**
```cadence
import EVM from 0x…
transaction {
    execute {
        let result = EVM.validateCOAOwnershipProof(
            address:    0xABCD,
            path:       /public/coaY,          // COA-Y's path
            signedData: "prove-coaX-ownership".utf8,
            keyIndices: [0],
            signatures: [<user's signature>],  // same sig
            evmAddress: [0xBB, 0xBB, …]        // COA-Y's address
        )
        assert(result.isValid)  // passes — wrong COA proven
    }
}
```

**Result:** `isValid == true` for `COA-Y`, even though the user signed only for `COA-X`. [4](#0-3) [5](#0-4) [6](#0-5)

```text
    /// validateCOAOwnershipProof validates a COA ownership proof.
    ///
    /// Note: this function does not enforce that `signedData` includes `evmAddress`.
    /// In principle, a signature produced for one purpose could be replayed here against
    /// a different COA owned by the same Cadence account. In practice this is low-risk:
    /// the EVM-side precompile (verifyCOAOwnershipProof) always passes the calling COA's
    /// address as the evmAddress argument, and Flow wallets historically create at most one
    /// COA per account. Callers building off-chain authentication flows on top of this
    /// function should ensure `signedData` encodes `evmAddress` to prevent cross-address replay.
    access(all)
    fun validateCOAOwnershipProof(
        address: Address,
        path: PublicPath,
        signedData: [UInt8],
        keyIndices: [UInt64],
        signatures: [[UInt8]],
        evmAddress: [UInt8; 20]
    ): ValidationResult {
        // make signature set first
        // check number of signatures matches number of key indices
        if keyIndices.length != signatures.length {
            return ValidationResult(
                isValid: false,
                problem: "EVM.validateCOAOwnershipProof(): Key indices array length doesn't match the signatures array length!"
            )
        }

        // fetch account
        let acc = getAccount(address)

        var signatureSet: [Crypto.KeyListSignature] = []
        let keyList = Crypto.KeyList()
        var keyListLength = 0
        let seenAccountKeyIndices: {Int: Int} = {}
        for signatureIndex, signature in signatures {
            // index of the key on the account
            let accountKeyIndex = Int(keyIndices[signatureIndex]!)
            // index of the key in the key list
            var keyListIndex = 0

            if !seenAccountKeyIndices.containsKey(accountKeyIndex) {
                // fetch account key with accountKeyIndex
                if let key = acc.keys.get(keyIndex: accountKeyIndex) {
                    if key.isRevoked {
                        return ValidationResult(
                            isValid: false,
                            problem: "EVM.validateCOAOwnershipProof(): Cannot validate COA ownership for Cadence account \(address). The account key at index \(accountKeyIndex) is revoked."
                        )
                    }

                    keyList.add(
                      key.publicKey,
                      hashAlgorithm: key.hashAlgorithm,
                      // normalization factor. We need to divide by 1000 because the
                      // `Crypto.KeyList.verify()` function expects the weight to be
                      // in the range [0, 1]. 1000 is the key weight threshold.
                      weight: key.weight / 1000.0,
                   )

                   keyListIndex = keyListLength
                   keyListLength = keyListLength + 1
                   seenAccountKeyIndices[accountKeyIndex] = keyListIndex
                } else {
                    return ValidationResult(
                        isValid: false,
                        problem: "EVM.validateCOAOwnershipProof(): Cannot validate COA ownership for Cadence account \(address). The key index \(accountKeyIndex) is invalid."
                    )
                }
            } else {
               // if we have already seen this accountKeyIndex, use the keyListIndex
               // that was previously assigned to it
               // `Crypto.KeyList.verify()` knows how to handle duplicate keys
               keyListIndex = seenAccountKeyIndices[accountKeyIndex]!
            }

            signatureSet.append(Crypto.KeyListSignature(
               keyIndex: keyListIndex,
               signature: signature
            ))
        }

        let isValid = keyList.verify(
            signatureSet: signatureSet,
            signedData: signedData,
            domainSeparationTag: "FLOW-V0.0-user"
        )

        if !isValid{
            return ValidationResult(
                isValid: false,
                problem: "EVM.validateCOAOwnershipProof(): Cannot validate COA ownership for Cadence account \(address). The given signatures are not valid or provide enough weight."
            )
        }

        if let coaRef = acc.capabilities.borrow<&EVM.CadenceOwnedAccount>(path) {
            // verify evm address matching — capture bytes once to avoid redundant borrow
            let coaAddressBytes = coaRef.address().bytes
            for index, item in coaAddressBytes {
                if item != evmAddress[index] {
                    return ValidationResult(
                        isValid: false,
                        problem: "EVM.validateCOAOwnershipProof(): Cannot validate COA ownership. The provided evm address does not match the account's COA address."
                    )
                }
            }
            return ValidationResult(
                isValid: true,
                problem: nil
            )
        }

        return ValidationResult(
            isValid: false,
            problem: "EVM.validateCOAOwnershipProof(): Cannot validate COA ownership. Could not borrow the COA resource for account \(address)."
        )
    }
```

```go
				},

				FunctionName: "validateCOAOwnershipProof",
				ArgumentTypes: []sema.Type{
					types.FlowAddressSemaType,
					types.PublicPathSemaType,

					types.AddressBytesSemaType,
				},
			},
			proof.ToCadenceValues(),
		)
		if err != nil {
			if types.IsAFatalError(err) {
				panic(err)
			}
			return false, err
```
