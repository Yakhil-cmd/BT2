### Title
`balance_of`-Based Credit Calculation Allows Donation-Driven Suppression of L1 Mint Requests, Causing Temporary Reward Freeze - (File: src/reward_supplier/reward_supplier.cairo)

### Summary

The `request_funds` internal function in `RewardSupplier` uses a live `balance_of` call on the STRK token contract to determine how much credit the contract has before deciding whether to send L1 mint requests. Because any unprivileged caller can transfer STRK tokens directly to the `RewardSupplier` contract address, an attacker can artificially inflate the observed balance, suppress all outgoing L1 mint requests, and eventually cause a window where the contract cannot fulfill `claim_rewards` calls — temporarily freezing staker and delegator unclaimed yield.

### Finding Description

In `src/reward_supplier/reward_supplier.cairo`, the internal `request_funds` function is called every time `update_unclaimed_rewards_from_staking_contract` is invoked. It reads the contract's live token balance via `balance_of` and uses it as part of a `credit` calculation:

```cairo
// src/reward_supplier/reward_supplier.cairo  lines 301-331
fn request_funds(ref self: ContractState, unclaimed_rewards: Amount) {
    let token_dispatcher = self.token_dispatcher.read();
    let balance: Amount = token_dispatcher
        .balance_of(account: get_contract_address())   // <-- externally manipulable
        .try_into()
        .expect_with_err(InternalError::BALANCE_ISNT_AMOUNT_TYPE);

    let mut l1_pending_requested_amount = self.l1_pending_requested_amount.read();
    let credit = balance + l1_pending_requested_amount;   // inflated by donation
    let debit = unclaimed_rewards;

    let base_mint_amount = self.base_mint_amount.read();
    let threshold = compute_threshold(base_mint_amount);  // base_mint_amount / 2
    if credit < debit + threshold {
        // send L1 mint requests
        ...
        l1_pending_requested_amount += total_amount;
    }
    self.l1_pending_requested_amount.write(l1_pending_requested_amount);
}

The `threshold` is simply `base_mint_amount / 2`: [2](#0-1) 

The guard condition `credit < debit + threshold` is the sole trigger for requesting new tokens from L1. If `credit` is artificially inflated, no mint requests are emitted and `l1_pending_requested_amount` is not incremented.

The `claim_rewards` function then performs a hard `checked_transfer` to the staking contract:

```cairo
// lines 205-220
fn claim_rewards(ref self: ContractState, amount: Amount) {
    let unclaimed_rewards = self.unclaimed_rewards.read();
    assert!(amount <= unclaimed_rewards, "{}", GenericError::AMOUNT_TOO_HIGH);
    self.unclaimed_rewards.write(unclaimed_rewards - amount);
    token_dispatcher.checked_transfer(recipient: staking_contract, amount: amount.into());
}
``` [3](#0-2) 

If the actual token balance of the contract is zero (because donated tokens were consumed paying out rewards and no new L1 mint was triggered), `checked_transfer` reverts, blocking all reward claims.

### Impact Explanation

**Temporary freezing of unclaimed yield (High).**

Once the attacker's donated tokens are consumed by legitimate `claim_rewards` calls, the contract's real balance reaches zero. At that point every call to `claim_rewards` reverts on `checked_transfer`, even though `unclaimed_rewards` correctly records the owed amount. The freeze persists until:

1. `request_funds` is next called (on the next `update_unclaimed_rewards_from_staking_contract` invocation),
2. the resulting L1 mint request messages are consumed by the L1 `RewardSupplier.tick()`,
3. the L1 bridge deposits the minted tokens back to L2 via `on_receive`.

This round-trip can take hours to days depending on L1 congestion and sequencer behavior, during which no staker or delegator can claim earned rewards.

### Likelihood Explanation

**Low-Medium.** The attack requires the attacker to spend their own STRK tokens (they are consumed paying legitimate rewards). However:

- No privileged access is required — any address can call `transfer` on the STRK ERC-20 to send tokens to the `RewardSupplier` contract address.
- The cost scales with `base_mint_amount` and the current `unclaimed_rewards` level, but a well-timed donation during a low-balance period can suppress requests with a relatively small amount.
- The attacker has no direct profit motive, making this a griefing vector, but the damage to stakers is real and measurable.

### Recommendation

Replace the live `balance_of` call with an internally tracked balance variable. Increment it in `on_receive` when tokens arrive from the bridge, and decrement it in `claim_rewards` when tokens are sent out. This mirrors the approach used for `unclaimed_rewards` and `l1_pending_requested_amount`, making the credit calculation immune to external token donations.

Alternatively, cap the effective balance used in `request_funds` at `unclaimed_rewards` (the maximum the contract legitimately needs), so any surplus from donations does not suppress mint requests.

### Proof of Concept

1. Protocol reaches a steady state: `balance = B`, `unclaimed_rewards = U`, `l1_pending_requested_amount = 0`, where `B` is just sufficient to cover `U`.
2. Attacker calls `STRK.transfer(reward_supplier_address, D)` for a large donation `D`.
3. Next call to `update_unclaimed_rewards_from_staking_contract` triggers `request_funds`:
   - `credit = (B + D) + 0`
   - `debit = U + new_rewards`
   - If `B + D >= debit + threshold`, the `if` branch is skipped — **no mint request is sent**.
4. Stakers continue claiming rewards; the contract's balance drains: `B + D → 0`.
5. `unclaimed_rewards` continues to grow as new rewards accrue.
6. Eventually `balance = 0`. The next `claim_rewards` call executes `checked_transfer` with zero balance → **revert**. All staker and delegator reward claims are frozen.
7. Only after `request_funds` fires again (on the next reward update), L1 processes the mint request, and the bridge callback `on_receive` delivers new tokens does the freeze lift — potentially hours or days later.

        fn request_funds(ref self: ContractState, unclaimed_rewards: Amount) {
            // Read current balance.
            let token_dispatcher = self.token_dispatcher.read();
            let balance: Amount = token_dispatcher
                .balance_of(account: get_contract_address())
                .try_into()
                .expect_with_err(InternalError::BALANCE_ISNT_AMOUNT_TYPE);

            // Calculate credit, which is the contract's balance plus the amount already requested
            // from L1, and the debit, which is the unclaimed rewards.
            let mut l1_pending_requested_amount = self.l1_pending_requested_amount.read();
            let credit = balance + l1_pending_requested_amount;
            let debit = unclaimed_rewards;

            // If there isn't enough credit to cover the debit + threshold, request funds.
            let base_mint_amount = self.base_mint_amount.read();
            let threshold = compute_threshold(base_mint_amount);
            if credit < debit + threshold {
                let diff = debit + threshold - credit;
                let num_msgs = ceil_of_division(dividend: diff, divisor: base_mint_amount);
                let total_amount = num_msgs * base_mint_amount;
                for _ in 0..num_msgs {
                    self.send_mint_request_to_l1_reward_supplier();
                }
                self.emit(Events::MintRequest { total_amount, num_msgs });
                l1_pending_requested_amount += total_amount;
            }

            // Commit to storage the requested amount, which is now part of the credit.
            self.l1_pending_requested_amount.write(l1_pending_requested_amount);
        }
