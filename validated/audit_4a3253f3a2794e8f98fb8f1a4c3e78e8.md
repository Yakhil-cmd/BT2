### Title
Accumulated Spread Fees Are Misrouted When Admin Fee Destination Is Changed Without Prior Collection — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary

`setPoolAdminFeeDestination` overwrites `poolAdminFeeDestination[pool]` without first flushing accumulated spread fees at the old destination. Every other fee-config mutator (`setPoolAdminFees`, `setPoolProtocolFee`) calls `collectFees` before updating state; `setPoolAdminFeeDestination` does not. Fees earned while the old destination was active are therefore sent to the new destination on the next collection, permanently depriving the old destination of owed funds.

### Finding Description

The factory stores the admin fee destination in `poolAdminFeeDestination[pool]` and uses it as the `adminFeeDestination_` argument every time `collectFees` is called on the pool. Spread fees are not tracked per-destination; they are computed at collection time from the live surplus:

```
surplus0Scaled = balance0() * TOKEN_0_SCALE_MULTIPLIER
               - binTotals.scaledToken0
               - notionalFeeToken0Scaled
```

All surplus accumulated since the last collection is split between protocol and admin at the moment `collectFees` executes. The admin's share goes to whatever address `poolAdminFeeDestination[pool]` holds at that instant.

`setPoolAdminFees` (line 418–425) and `setPoolProtocolFee` (line 328–335) both call `collectFees` with the current config before mutating state, ensuring fees earned under the old parameters are settled first:

```solidity
// setPoolAdminFees — correctly flushes first
IMetricOmmPoolCollectFees(pool).collectFees(
    c.protocolSpreadFeeE6, c.adminSpreadFeeE6,
    c.protocolNotionalFeeE8, c.adminNotionalFeeE8,
    poolAdminFeeDestination[pool]          // old destination
);
c.adminSpreadFeeE6 = newAdminSpreadFeeE6;
poolFeeConfig[pool] = c;
```

`setPoolAdminFeeDestination` (line 438–447) performs no such flush:

```solidity
function setPoolAdminFeeDestination(address pool, address newAdminFeeDestination)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (newAdminFeeDestination == address(0)) revert InvalidAdminFeeDestination();
    poolAdminFeeDestination[pool] = newAdminFeeDestination;   // overwrites without settling
    emit PoolAdminFeeDestinationUpdated(pool, newAdminFeeDestination);
}
```

After this call, the next invocation of `collectPoolFees` or any fee-changing function will send the entire accumulated surplus — including the portion earned while the old destination was active — to the new address.

### Impact Explanation

The old admin fee destination permanently loses all spread fees that accrued between the last collection and the destination change. The new destination receives fees it did not earn. Because spread fees accumulate continuously with every swap and are only realised at collection time, the misrouted amount can be arbitrarily large depending on trading volume and collection frequency. This is a direct loss of owed protocol/admin fee assets.

### Likelihood Explanation

The trigger is a routine, legitimate admin action: changing the fee-recipient wallet (e.g., treasury rotation, multisig migration). No malicious setup, non-standard tokens, or privileged factory-owner cooperation is required. Any pool admin can call `setPoolAdminFeeDestination` at any time, making this reachable on every pool with non-zero spread-fee activity.

### Recommendation

Mirror the pattern used by `setPoolAdminFees` and `setPoolProtocolFee`: call `collectFees` with the current config and the **old** destination before overwriting `poolAdminFeeDestination[pool]`.

```solidity
function setPoolAdminFeeDestination(address pool, address newAdminFeeDestination)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (newAdminFeeDestination == address(0)) revert InvalidAdminFeeDestination();
    // Flush accumulated fees to the current (old) destination first
    PoolFeeConfig memory c = poolFeeConfig[pool];
    IMetricOmmPoolCollectFees(pool).collectFees(
        c.protocolSpreadFeeE6, c.adminSpreadFeeE6,
        c.protocolNotionalFeeE8, c.adminNotionalFeeE8,
        poolAdminFeeDestination[pool]
    );
    poolAdminFeeDestination[pool] = newAdminFeeDestination;
    emit PoolAdminFeeDestinationUpdated(pool, newAdminFeeDestination);
}
```

### Proof of Concept

1. Pool is deployed with `adminFeeDestination = A`. Swaps occur; surplus accumulates (e.g., 1 000 token0 in spread fees owed to admin).
2. Pool admin calls `setPoolAdminFeeDestination(pool, B)`. No `collectFees` is triggered; surplus remains on the pool.
3. Anyone calls `collectPoolFees(pool)`. Inside `collectFees`, `adminFeeDestination_` is now `B`.
4. The entire 1 000 token0 admin share is transferred to `B`. Address `A` receives nothing despite having earned those fees.
