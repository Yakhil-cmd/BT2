### Title
Incorrect NFT Seller Net Proceeds Calculation Causes Inflated "You Will Receive" Display in TokenForNFT Offer Acceptance - (File: `packages/gui/src/components/offers/utils.ts`)

### Summary

`calculateNFTRoyalties` in `utils.ts` hard-codes `nftSellerNetAmount = amount` (the full offered price) instead of computing `amount - royaltyAmount`. The commented-out correct formula was deliberately disabled. This propagates through `NFTOfferViewer.tsx` to display an inflated "Net Proceeds" and "You will receive" amount to an NFT seller reviewing a `TokenForNFT` offer, causing them to accept an offer under the false belief they will receive the full offered price when they actually receive less (the price minus royalties).

### Finding Description

In `packages/gui/src/components/offers/utils.ts`, `calculateNFTRoyalties` computes:

```typescript
const nftSellerNetAmount: number = amount;
// : parseFloat(
//     (amount - parseFloat(royaltyAmountString) - makerFee).toFixed(12),
//   );
``` [1](#0-0) 

`nftSellerNetAmount` is set to the raw `amount` (the full token price offered by the buyer), with the correct subtraction of `royaltyAmountString` and `makerFee` commented out.

This value flows into `NFTOfferDetails` in `NFTOfferViewer.tsx`:

```typescript
const nftSaleInfo = useMemo(() => {
  ...
  return {
    ...calculateNFTRoyalties(amount, parseFloat(xchMakerFee), ..., exchangeType),
    royaltyPercentage,
  };
}, [...]);
``` [2](#0-1) 

Then `overrideNFTSellerAmount` is derived from the wrong `nftSellerNetAmount`:

```typescript

    exchangeType === NFTOfferExchangeType.TokenForNFT
      ? assetType === OfferAsset.CHIA
        ? chiaToMojo(nftSaleInfo?.nftSellerNetAmount ?? 0)
        : catToMojo(nftSaleInfo?.nftSellerNetAmount ?? 0)
      : undefined;
``` [3](#0-2) 

This is passed to `NFTOfferSummary` ظْ `NFTOfferSummaryRow` ظْ `OfferSummaryTokenRow`, where it replaces the displayed amount:

```typescript

``` [4](#0-3) 

Additionally, the "Net Proceeds" label directly renders the wrong value:

```typescript

``` [5](#0-4) 

The tooltip for "Net Proceeds" explicitly states it should be "the asking price, minus any associated creator fees," but the displayed value is the full asking price. [6](#0-5) 

### Impact Explanation

In a `TokenForNFT` offer, the buyer (maker) offers a token amount that is split on-chain: the seller receives `offered_amount - royalties`, and the royalty address receives `royalties`. The GUI tooltip confirms this: "those creator fees will be paid from the offered purchase price." [7](#0-6) 

**Concrete example:** Buyer creates a `TokenForNFT` offer for 100 XCH on an NFT with 10% royalty.
- On-chain: seller receives 90 XCH, royalty address receives 10 XCH.
- GUI displays: "Net Proceeds: 100 XCH" and "You will receive: 100 XCH."
- Seller accepts believing they receive 100 XCH; they actually receive 90 XCH.

The discrepancy scales with royalty percentage. NFTs with royalties up to the 20% warning threshold (where `showRoyaltyWarning` triggers) represent up to a 20% overstatement of proceeds. [8](#0-7) 

This matches the High impact category: offer state causes a user to approve the wrong amount.

### Likelihood Explanation

- Any unprivileged user can create a `TokenForNFT` offer for any NFT with a non-zero `royaltyPercentage` and share it with the NFT owner.
- The NFT owner views the offer in the GUI (imported offer path, `imported=true`), sees the inflated "Net Proceeds" and "You will receive" amounts, and accepts.
- No privileged access, leaked keys, or social engineering beyond sharing a valid offer file is required.
- The royalty percentage is visible in the UI, but the "Net Proceeds" headline figure is the primary decision signal for the seller.

### Recommendation

Restore the correct `nftSellerNetAmount` computation in `calculateNFTRoyalties`:

```typescript
const nftSellerNetAmount: number = parseFloat(
  (amount - parseFloat(royaltyAmountString) - makerFee).toFixed(12),
);
``` [1](#0-0) 

Verify that `overrideNFTSellerAmount` in `NFTOfferViewer.tsx` then correctly reflects the net amount (after royalties) in the "You will receive" summary row and the "Net Proceeds" display. [3](#0-2) 

### Proof of Concept

1. Obtain or mint an NFT with `royaltyPercentage > 0` (e.g., 1000 = 10%).
2. As an unprivileged buyer, create a `TokenForNFT` offer offering 100 XCH for that NFT using the GUI's "Buy an NFT" tab.
3. Share the resulting offer file with the NFT owner.
4. NFT owner imports the offer file into their Chia GUI.
5. The offer viewer (`NFTOfferDetails`) displays:
   - "Net Proceeds: 100 XCH" (from `nftSaleInfo.nftSellerNetAmount = amount = 100`)
   - "You will receive: 100 XCH" (from `overrideNFTSellerAmount = chiaToMojo(100)`)
   - "Creator Fee (10%): 10 XCH" (correctly computed)
6. NFT owner accepts the offer.
7. On-chain settlement: NFT owner receives 90 XCH; royalty address receives 10 XCH.
8. The seller received 10 XCH less than the GUI indicated. [9](#0-8) [10](#0-9)

export function calculateNFTRoyalties(
  amount: number,
  makerFee: number,
  royaltyPercentage: number,
  exchangeType: NFTOfferExchangeType,
): CalculateNFTRoyaltiesResult {
  const royaltyAmount: number = royaltyPercentage ? (royaltyPercentage / 100) * amount : 0;
  const royaltyAmountString: string = formatAmount(royaltyAmount);
  const nftSellerNetAmount: number = amount;
  // : parseFloat(
  //     (amount - parseFloat(royaltyAmountString) - makerFee).toFixed(12),
  //   );
  const totalAmount: number =
    exchangeType === NFTOfferExchangeType.NFTForToken ? amount + royaltyAmount : amount + makerFee + royaltyAmount;
  const totalAmountString: string = formatAmount(totalAmount);

  return {
    royaltyAmount,
    royaltyAmountString,
    nftSellerNetAmount,
    totalAmount,
    totalAmountString,
  };
}
