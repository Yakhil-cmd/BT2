### Title
BLS12-381 EIP-2537 Precompiles Charge Zero Native (Proving) Resources, Enabling Cheap Proving-Layer DOS - (File: `basic_bootloader/src/bootloader/block_flow/ethereum/hooks/eip_2537/`)

---

### Summary

All seven EIP-2537 BLS12-381 precompiles in ZKsync OS hardcode `cost_native = 0`, meaning they consume **zero native resources** (RISC-V proving cycles) regardless of the actual computational work performed. Because the double-resource-accounting model derives the proving budget from `gasLimit × nativePerGas`, an attacker can pack a transaction full of BLS12-381 calls that are cheap in EVM gas but impose an unbounded, uncompensated proving burden on the operator. This is the direct analog of the external report's "cheap repeated oracle calls exhaust the system at minimal attacker cost."

---

### Finding Description

ZKsync OS implements a **double resource accounting** model: every operation charges both EVM gas (ergs) and a *native* resource representing RISC-V proving cycles. The native limit for a transaction is `gasLimit × nativePerGas`; if native resources are exhausted the transaction reverts. The `deltaGas` mechanism then converts any excess native consumption back into gas so the operator is compensated.

All seven EIP-2537 BLS12-381 precompiles set `cost_native = 0`:

**G1/G2 Addition** (`addition.rs`): [1](#0-0) [2](#0-1) 

**G1/G2 MSM** (`msm.rs`): [3](#0-2) [4](#0-3) 

**Pairing Check** (`pairing.rs`): [5](#0-4) 

**Field-to-Curve Mappings** (`mappings.rs`): [6](#0-5) [7](#0-6) 

All other computationally expensive precompiles properly charge native resources. For example, `ECRECOVER_NATIVE_COST = 350_000`, `BN254_PAIRING_BASE_NATIVE_COST = 13_000_000`, and `POINT_EVALUATION_NATIVE_COST ≈ 49_900_000`: [8](#0-7) 

The double-resource model is designed so that if native consumption exceeds what gas covers, `deltaGas` forces the user to pay more: [9](#0-8) 

Because `cost_native = 0` for all BLS12-381 precompiles, this correction never fires for them. The prover must execute the full Miller-loop / final-exponentiation / MSM computation but receives no native-resource compensation.

---

### Impact Explanation

**Impact: High** — The proving system (RISC-V ZK prover) is the resource being exhausted. BLS12-381 pairing is among the most expensive operations in the codebase. With `cost_native = 0`, a single transaction can call the pairing precompile hundreds of times within a normal gas limit (e.g., 30 M gas ÷ 70 300 gas/pair ≈ 426 pairing calls), each imposing massive RISC-V cycle cost with zero native-resource charge. This:

1. **Delays block proving** — the prover must process far more work than the fee covers, stalling finality.
2. **Causes economic loss** — the operator pays proving infrastructure costs not covered by transaction fees.
3. **Blocks legitimate transactions** — a block slot filled with such a transaction cannot include other transactions, and the proving delay cascades to subsequent blocks.

The pairing precompile is the worst case, but G2-MSM and the field-to-curve mappings are also severe.

---

### Likelihood Explanation

**Likelihood: High** — The attack requires only a standard EVM transaction calling the BLS12-381 pairing precompile (address `0x0f`) in a loop. No privileged access, no leaked keys, no governance majority. The precompiles are unconditionally available once the `eip-2537` feature is enabled. The attacker pays only EVM gas (at the base fee), which is the minimum cost on any EVM-compatible chain.

---

### Recommendation

Assign realistic native costs to all BLS12-381 precompiles, consistent with the pattern used for BN254 and other expensive precompiles. For example:

- **Pairing**: benchmark the Miller loop + final exponentiation in RISC-V cycles and set `cost_native` proportionally (expected to be in the same order as `BN254_PAIRING_BASE_NATIVE_COST ≈ 13_000_000` per pair, likely higher for BLS12-381).
- **G1/G2 MSM**: scale native cost with the number of points and scalar bit-length.
- **G1/G2 Addition, Field Mappings**: assign a fixed native cost based on benchmarked RISC-V cycle counts.

The pattern to follow is already established in `basic_system/src/cost_constants.rs`: [10](#0-9)

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

contract BLS12PairingDOS {
    // BLS12-381 pairing precompile address (EIP-2537)
    address constant BLS12_PAIRING = address(0x0f);

    // Valid G1 point (generator) + G2 point (generator) encoding
    // 1 pair = 192 bytes (G1: 128 bytes, G2: 256 bytes... actually G1=128, G2=256 per EIP-2537)
    // For simplicity, use the identity points (all zeros) which are valid inputs
    bytes constant ONE_PAIR = new bytes(384); // G1(128) + G2(256) = 384 bytes

    function attack() external {
        // Each call costs 37700 + 32600 = 70300 gas, but 0 native resources.
        // With 30M gas limit: ~426 iterations.
        // Each pairing call triggers a full Miller loop + final exponentiation
        // in the RISC-V prover, consuming millions of cycles with zero native charge.
        for (uint256 i = 0; i < 400; i++) {
            (bool ok,) = BLS12_PAIRING.call{gas: 80000}(ONE_PAIR);
            // ok may be false for identity points but the prover still executes
        }
    }
}
```

**Attack flow:**
1. Attacker deploys `BLS12PairingDOS` on ZKsync OS with EIP-2537 enabled.
2. Attacker submits a transaction calling `attack()` with ~30M gas limit and minimum gas price.
3. The transaction executes ~400 pairing precompile calls. Each call:
   - Charges `70 300 gas` (ergs) — paid by attacker.
   - Charges `0 native` — the prover bears the full RISC-V cycle cost.
4. The block containing this transaction takes disproportionately long to prove.
5. The attacker can repeat this with multiple transactions per block, or across multiple blocks, at the cost of only base-fee gas.
