### Title
Malicious Participant Can Trigger Unattributed Protocol Abort in Triple Generation via Crafted DLOG Proof — (`src/crypto/proofs/dlog.rs`, `src/ecdsa/ot_based_ecdsa/triples/generation.rs`)

### Summary
`dlog::verify` returns `Err(ProtocolError::IdentityElement)` instead of `Ok(false)` when the reconstructed commitment `big_k = s*G - e*X` is the group identity. A malicious participant who knows their own secret `x` can trivially craft a proof `(e, s=e*x)` that triggers this path. At all three call sites in `do_generation_many`, the `?` operator propagates the error directly — bypassing the attribution branch that would name the cheater — causing an unattributed abort that honest parties cannot attribute to any specific participant.

### Finding Description

**Root cause in `dlog::verify`:** [1](#0-0) 

Line 90 computes `big_k = s*G - e*X`. Line 94 serializes it; if `big_k` is the identity, serialization fails and the function returns `Err(ProtocolError::IdentityElement)` rather than `Ok(false)`.

**Call sites in `do_generation_many` use `?` on the result:** [2](#0-1) [3](#0-2) [4](#0-3) 

The pattern `if !dlog::verify(...)?` means:
- If `dlog::verify` returns `Ok(false)` → the `if` body executes → `Err(ProtocolError::AssertionFailed("dlog proof from {from:?} failed to verify"))` — **cheater is named**.
- If `dlog::verify` returns `Err(ProtocolError::IdentityElement)` → `?` propagates it directly → **cheater is anonymous**.

### Impact Explanation

A malicious participant knows their own witness `x` (the constant term of their polynomial `e.eval_at_zero()`). They pick any scalar `e_val` and set `s = e_val * x`. The verifier computes `big_k = s*G - e_val*X = e_val*x*G - e_val*x*G = identity`. Serialization fails, `Err(ProtocolError::IdentityElement)` propagates, and `do_generation_many` aborts without identifying the source. Honest parties cannot exclude the malicious participant from retries because they do not know who caused the abort. This enables indefinite denial of triple generation, which is a prerequisite for OT-based ECDSA signing.

**Allowed impact matched:** High — Permanent denial of signing for honest parties under valid protocol inputs and documented trust assumptions.

### Likelihood Explanation

The attack requires only that the malicious party be a protocol participant (the standard adversary model for threshold protocols). No cryptographic assumptions need to be broken. The attacker uses their own legitimately-known secret. The crafted proof is a valid serializable `Proof<C>` struct and passes all message-layer checks before `dlog::verify` is called.

### Recommendation

In `dlog::verify`, treat a reconstructed identity commitment as a verification failure rather than an error:

```rust
// Instead of:
let ser = C::Group::serialize(&big_k).map_err(|_| ProtocolError::IdentityElement)?;

// Use:
let Ok(ser) = C::Group::serialize(&big_k) else {
    return Ok(false);
};
```

This ensures all invalid proofs — including identity-commitment ones — return `Ok(false)`, allowing the call sites in `do_generation_many` to reach the attribution branch and name the cheating participant.

### Proof of Concept

```rust
// Attacker knows x (their own witness scalar)
// They craft: s = e * x, so s*G - e*X = e*x*G - e*x*G = identity
let x: Scalar = /* attacker's own secret */;
let e_val = Scalar::from(42u64); // any nonzero scalar
let s_val = e_val * x;

let malicious_proof = Proof {
    e: SerializableScalar(e_val),
    s: SerializableScalar(s_val),
};

// Verifier side:
let result = dlog::verify(&mut transcript, statement, &malicious_proof);
// Returns Err(ProtocolError::IdentityElement) instead of Ok(false)
// The ? in do_generation_many propagates this, aborting without attribution
assert!(matches!(result, Err(ProtocolError::IdentityElement)));
```
