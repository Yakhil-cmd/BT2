### Title
Declare Transactions Bypass `max_l2_gas_amount` Gateway Admission Check — (`crates/apollo_gateway/src/stateless_transaction_validator.rs`)

### Summary
`StatelessTransactionValidator::validate_resource_bounds` unconditionally skips the `max_l2_gas_amount` check for all `RpcTransaction::Declare` variants. An unprivileged user can submit a Declare transaction with `l2_gas.max_amount = u64::MAX` and it will be admitted through the gateway and into the mempool, violating the invariant that all admitted transactions have `l2_gas.max_amount ≤ config.max_l2_gas_amount`.

### Finding Description

The check at lines 79–85 is structured as an empty `if`-arm followed by an `else if`:

```rust
// TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
if let RpcTransaction::Declare(_) = tx {
    // nothing — check is skipped entirely
} else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh { ... });
}
``` [1](#0-0) 

When `tx` is a `Declare`, the `if let` arm matches and the `else if` is never evaluated, regardless of the value of `l2_gas.max_amount`. The TODO comment confirms this is a known, unresolved gap.

The stateful validator's `validate_resource_bounds` only checks the L2 gas *price* against the previous block price — it does not check the gas *amount* at all, for any transaction type. [2](#0-1) 

The blockifier's `perform_validations` for Declare transactions calls `self.execute(tx)` directly, which runs execution but does not enforce the gateway-level `max_l2_gas_amount` config limit. [3](#0-2) 

The bypass is not only present in production code — it is explicitly tested and asserted as the expected behavior in `valid_l2_gas_amount_on_declare`: a Declare with `max_amount: GasAmount(200)` passes when `max_l2_gas_amount: 100`. [4](#0-3) 

The production default for `max_l2_gas_amount` is `1,210,000,000`. [5](#0-4) 

### Impact Explanation

A Declare transaction with `l2_gas.max_amount = u64::MAX` passes every check in `StatelessTransactionValidator::validate`:
- Zero-fee check: passes (amount × price > 0).
- `min_gas_price` check: passes (price is attacker-controlled and can be set above the minimum).
- `max_l2_gas_amount` check: **skipped** for Declare.

The transaction then passes the stateful validator (which only checks price, not amount) and is admitted to the mempool. The concrete corrupted admission value is: a Declare transaction with `l2_gas.max_amount = u64::MAX` is admitted when the configured limit is `1,210,000,000`. This fits the **High** impact category: "Mempool/gateway/RPC admission accepts invalid transactions before sequencing."

### Likelihood Explanation

Trivially exploitable by any unprivileged user who can submit transactions to the public gateway RPC endpoint. No special privileges, keys, or state are required. The attacker only needs to construct a valid Declare transaction (valid Sierra class, valid signature) and set `l2_gas.max_amount` to any value exceeding `max_l2_gas_amount`.

### Recommendation

Remove the empty `if let RpcTransaction::Declare(_) = tx { }` arm and apply the `max_l2_gas_amount` check uniformly to all transaction types, or restructure the check:

```rust
if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
        gas_amount: resource_bounds.l2_gas.max_amount,
        max_gas_amount: self.config.max_l2_gas_amount,
    });
}
```

If Declare transactions legitimately require a higher or separate limit, introduce a dedicated `max_l2_gas_amount_declare` config field rather than skipping the check entirely.

### Proof of Concept

```rust
#[test]
fn declare_bypasses_max_l2_gas_amount() {
    let config = StatelessTransactionValidatorConfig {
        validate_resource_bounds: true,
        max_l2_gas_amount: 100,
        min_gas_price: 0,
        ..StatelessTransactionValidatorConfig::default()
    };
    let validator = StatelessTransactionValidator { config };

    // Build a Declare tx with l2_gas.max_amount = u64::MAX
    let tx = rpc_tx_for_testing(
        TransactionType::Declare,
        RpcTransactionArgs {
            resource_bounds: AllResourceBounds {
                l2_gas: ResourceBounds {
                    max_amount: GasAmount(u64::MAX),
                    max_price_per_unit: GasPrice(1),
                },
                ..Default::default()
            },
            ..Default::default()
        },
    );

    // Asserts Ok(()) — the transaction is admitted despite max_amount >> max_l2_gas_amount
    assert_matches!(validator.validate(&tx), Ok(()));
}
```

This mirrors the existing `valid_l2_gas_amount_on_declare` test (which already asserts `Ok(())` for `max_amount: 200` with `max_l2_gas_amount: 100`), confirming the bypass is unconditional for any Declare transaction. [6](#0-5)
