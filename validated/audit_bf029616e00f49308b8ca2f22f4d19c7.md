### Title
Missing `evmAddress` Binding in `validateCOAOwnershipProof` Signed Data Enables Cross-COA Replay Attack - (File: `fvm/evm/stdlib/contract.cdc`)

---

### Summary

`EVM.validateCOAOwnershipProof` verifies Cadence account key signatures over an arbitrary `signedData` blob without enforcing that `signedData` encodes the `evmAddress` being claimed. A valid signature produced for one COA can therefore be replayed to prove ownership of any other COA held by the same Cadence account, bypassing the intended per-address binding of the proof.

---

### Finding Description

`validateCOAOwnershipProof` performs two independent checks:

1. Verifies that the supplied signatures are valid over `signedData` under domain tag `"FLOW-V0.0-user"`.
2. Verifies that the COA resource at the supplied `path` has the supplied `evmAddress`.

The two checks are never linked: the signed message (`signedData`) is never required to commit to `evmAddress` or `path`. The code itself acknowledges this gap:

```
/// Note: this function does not enforce that `signedData` includes `evmAddress`.
/// In principle, a signature produced for one purpose could be replayed here against
/// a different COA owned by the same Cadence account.
``` [1](#0-0) 

The verification logic at lines 1082–1086 calls `keyList.verify(signatureSet: signatureSet, signedData: signedData, domainSeparationTag: "FLOW-V0.0-user")` and then separately checks the EVM address at lines 1095–1105, with no step that ties the two together. [2](#0-1) [3](#0-2) 

This is the direct Flow analog of the EIP-712 finding: in the Solidity case the signed struct omits sub-type hashes and leaves dynamic fields unhashed, so the signed message does not fully commit to the data it is supposed to authorize. Here, the signed message (`signedData`) does not commit to the EVM address or capability path, so the same signature authorizes ownership of every COA the signer holds.

---

### Impact Explanation

**Impact: Medium**

An attacker who has observed a valid `COAOwnershipProof` for Alice's COA_A (e.g., from a prior on-chain transaction) can construct a new call to `validateCOAOwnershipProof` substituting COA_B's address and path while reusing the same `signedData` and signatures. Because the function is `access(all)` and callable from any Cadence transaction or script, the attacker can obtain a `ValidationResult(isValid: true)` for COA_B without Alice's knowledge or consent.

Any Cadence contract that gates asset operations on a successful `validateCOAOwnershipProof` result is therefore vulnerable to unauthorized authorization: the attacker proves ownership of a COA they do not control and triggers the guarded action. [4](#0-3) 

---

### Likelihood Explanation

**Likelihood: Medium**

- `validateCOAOwnershipProof` is a public, permissionless function callable by any unprivileged transaction sender.
- Signatures over `signedData` are observable on-chain whenever a user has previously submitted a COA ownership proof (e.g., via the EVM precompile path or a Cadence transaction).
- The precondition of multiple COAs per account is uncommon today but is explicitly supported by the protocol and will grow as the ecosystem matures.
- No privileged access, leaked keys, or social engineering is required; the attacker only needs to replay an already-published signature. [5](#0-4) 

---

### Recommendation

Require that `signedData` commits to both `evmAddress` and `path` before signature verification. A minimal fix is to define a canonical signed payload:

```cadence
let commitment = evmAddress.concat(path.toString().utf8).concat(signedData)
let isValid = keyList.verify(
    signatureSet: signatureSet,
    signedData: commitment,
    domainSeparationTag: "FLOW-V0.0-user"
)
```

Alternatively, enforce this at the precompile layer by constructing `signedData` as `keccak256(evmAddress || hash)` before passing it to `validateCOAOwnershipProof`, mirroring the EIP-712 recommendation to include all relevant context in the signed message. [6](#0-5)

### Proof of Concept

**Setup**: Alice holds two COAs — COA_A at `/public/coa_a` (EVM address `0xAAAA…`) and COA_B at `/public/coa_b` (EVM address `0xBBBB…`).

**Step 1 – Observe a legitimate proof for COA_A on-chain:**
Alice previously submitted a transaction containing:
```
proof_A = COAOwnershipProof{
    Address:        Alice,
    CapabilityPath: "coa_a",
    KeyIndices:     [0],
    Signatures:     [sig_over_D],   // sig over arbitrary data D
}
signedData = D
```

**Step 2 – Attacker replays the signature for COA_B:**
```cadence
import EVM from 0x…

transaction {
    execute {
        let result = EVM.validateCOAOwnershipProof(
            address:    Alice,
            path:       /public/coa_b,   // ← different path
            signedData: D,               // ← same data
            keyIndices: [0],             // ← same key
            signatures: [sig_over_D],   // ← same signature
            evmAddress: [0xBB, 0xBB, …] // ← COA_B's address
        )
        assert(result.isValid, message: "replay succeeded")
    }
}
```

**Result**: `validateCOAOwnershipProof` returns `isValid: true` for COA_B because:
- `sig_over_D` is a valid signature over `D` by Alice's key 0 ✓
- The COA at `/public/coa_b` has EVM address `0xBBBB…` ✓
- No check ties `D` to `0xBBBB…` ✗ [7](#0-6) [8](#0-7)
