### Title
Silent Revocation Failure Leaves Authorized Pair in `dapp-pairs.yaml` When WC Client Is Uninitialized — (`packages/gui/src/components/walletConnect/WalletConnectProvider.tsx`, `SettingsIntegration.tsx`)

### Summary

When the WalletConnect client is `null` (WC disabled mid-session), a user clicking **Disconnect** in Settings triggers a code path where `revokePair` is never called, the error is silently swallowed, and the pair record persists in `dapp-pairs.yaml` with its full original permissions. When WC is subsequently re-enabled, the pair is still authorized and can receive and execute `session_request` commands.

### Finding Description

The call chain is:

**`SettingsIntegration.tsx` → `handleDisconnectApp`** calls `disconnectPair(topic)`, which resolves to **`handleDisconnectPair`** in `WalletConnectProvider.tsx`:

```
handleDisconnectPair (WalletConnectProvider.tsx:278-295)
  try:
    if (!currentClient) throw new Error('Client not initialized')  ← throws
    await disconnectPair(currentClient, topic)                      ← never reached
  catch (e):
    processError(e as Error)                                        ← re-throws (see below)
  finally:
    updateListOfPairs()                                             ← only refreshes UI
``` [1](#0-0) 

`processError` only silences `'No matching key'` errors; `'Client not initialized'` does **not** match, so it is **re-thrown**: [2](#0-1) 

The re-thrown error propagates back to `handleDisconnectApp`, where it is caught and **swallowed** with `console.warn`. Crucially, `setTopic(null)` and `refresh()` then execute unconditionally: [3](#0-2) 

The inner `disconnectPair` function — the only place `permissionsAPI.revokePair` is called — is **never reached** when `currentClient` is null: [4](#0-3) 

The `finally` block only calls `updateListOfPairs()`, which re-reads `getPairs()` from the main process. Since `revokePair` was never called, the pair is still in `dapp-pairs.yaml` and is returned unchanged: [5](#0-4) 

The comment in `handleDisconnectApp` ("disconnectPair also calls `permissionsAPI.revokePair` in its finally block") is factually incorrect — `revokePair` is in the **try** block of the inner `disconnectPair`, not a finally block, and is unreachable when the client is null. [6](#0-5) 

### Impact Explanation

When WC is re-enabled, a new client is initialized. `cleanupPairingsAndSessions` disconnects pairings/sessions **not** in the pairs list — but since the pair was never revoked, it **is** still in the list and is not cleaned up: [7](#0-6) 

The WC SDK persists session state locally (TTL typically 7 days). On client re-initialization, existing sessions are restored. The dApp can then send `session_request` commands that are processed by `processSessionRequest`, which checks `findPair(pairTopic)` — the pair is found, its commands and fingerprint are still authorized, and the request is dispatched: [8](#0-7) 

This means a dApp the user explicitly tried to revoke retains the ability to trigger wallet operations (signing, sending, offer acceptance, etc.) scoped to the originally granted permissions.

### Likelihood Explanation

The precondition — WC disabled while a pair exists — is a natural user workflow: a user disabling WC to stop dApp access and then cleaning up connected apps in Settings. No attacker action is required to create this state; the dApp simply waits for WC to be re-enabled.

### Recommendation

In `handleDisconnectPair`, call `window.permissionsAPI.revokePair(topic)` unconditionally (e.g., before the `currentClient` guard or in the `finally` block), so that the authorization record is always removed regardless of client state. Additionally, surface a user-facing error when the disconnect fails so the user knows the operation did not complete.

### Proof of Concept

Unit-test `handleDisconnectPair` with `clientRef.current = null`:
1. Call `handleDisconnectPair(topic)`.
2. Assert `permissionsAPI.revokePair` was **never** called.
3. Assert `permissionsAPI.getPairs()` still returns the pair.
4. Set `clientRef.current` to a real mock client and call `handleDisconnectPair(topic)` again — now `revokePair` is called and the pair is gone.

This directly proves the invariant violation: a UI-confirmed disconnect does not atomically revoke the pair when the client is uninitialized.

async function cleanupPairingsAndSessions(client: Client) {
  try {
    const pairs = await window.permissionsAPI.getPairs();

    // disconnect all sessions that are not in the pairs list
    const clientSessions = client.session.getAll();

    const sessionsToDisconnect = clientSessions.filter((s) => !pairs.some((p) => p.topic === s.pairingTopic));
    await Promise.all(
      sessionsToDisconnect.map((session) =>
        client.disconnect({
          topic: session.topic,
          reason: getSdkError('USER_DISCONNECTED'),
        }),
      ),
    );

    // disconnect all pairings that are not in the pairs list
    const clientPairings = client.pairing.getAll();

    const pairingsToDisconnect = clientPairings.filter(
      (clientPair) => !pairs.some((p) => clientPair.topic === p.topic),
    );

    await Promise.all(pairingsToDisconnect.map((pairing) => client.core.pairing.disconnect({ topic: pairing.topic })));
  } catch (e) {
    log('Cleanup pairings error', e);
    processError(e as Error);
  }
}

async function disconnectPair(client: Client, topic: string) {
  try {
    // revoke pair from main process
    await window.permissionsAPI.revokePair(topic);

    // get all sessions for the pairing
    const sessions = client.session.getAll();
    const relatedSessions = sessions.filter((s) => s.pairingTopic === topic);

    // disconnect all related sessions
    await Promise.all(
      relatedSessions.map((session) =>
        client.disconnect({
          topic: session.topic,
          reason: getSdkError('USER_DISCONNECTED'),
        }),
      ),
    );

    // disconnect the pairing
    await client.core.pairing.disconnect({ topic });
  } catch (e) {
    processError(e as Error);
  }
}
```
