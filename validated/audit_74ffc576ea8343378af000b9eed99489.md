### Title
ZK Transaction Path Sets `minimal_gas_used = 0` Instead of `TX_INTRINSIC_GAS`, Allowing Gas Charge Below Intrinsic Minimum - (File: `basic_bootloader/src/bootloader/transaction_flow/zk/validation_impl.rs`)

---

### Summary

In the ZK transaction validation path, `compute_calldata_tokens` returns `minimal_gas_used = 0` when the `eip_7623` feature is disabled. The Ethereum path returns `TX_INTRINSIC_GAS = 21_000` in the same condition. Because `minimal_gas_used` is used as a floor in `compute_gas_refund` to enforce the minimum gas charged, the ZK path allows a transaction's final `gas_used` to fall below the 21,000 intrinsic minimum when EVM refunds (e.g., SSTORE-to-zero) are applied.

---

### Finding Description

In `zk/validation_impl.rs`, `compute_calldata_tokens` is defined as:

```rust
// zk/validation_impl.rs lines 519-541
#[cfg(not(feature = "eip_7623"))]
{
    (num_tokens, 0)   // minimal_gas_used = 0
}
```

The Ethereum path (`ethereum/validation_impl.rs`) returns the correct floor:

```rust
// ethereum/validation_impl.rs lines 159-162
#[cfg(not(feature = "eip_7623"))]
{
    (num_tokens, TX_INTRINSIC_GAS)   // minimal_gas_used = 21_000
}
```

The returned `minimal_gas_used` is stored in `TxContextForPreAndPostProcessing` as:

```rust
// zk/validation_impl.rs line 497
minimal_ergs_to_charge: Ergs(minimal_gas_used.saturating_mul(ERGS_PER_GAS)),
```

With `minimal_gas_used = 0`, `minimal_ergs_to_charge = Ergs(0)`.

In `refund_calculation.rs`, `compute_gas_refund` enforces the floor:

```rust
// refund_calculation.rs line 56
let mut gas_used = core::cmp::max(gas_used, minimal_gas_used);
```

When `minimal_gas_used = 0`, this floor is inoperative. EVM refunds (capped at `gas_used / 5` per EIP-3529) can reduce the final `gas_used` below `TX_INTRINSIC_GAS = 21_000`.

**Concrete example:**
- `gas_limit = 100_000`
- Transaction executes a SSTORE to zero (warm slot): costs 2,900 gas, generates 4,800 refund
- `gas_used_before_refund = 21_000 + 2_900 = 23_900`
- Refund cap = `23_900 / 5 = 4_780`
- Actual refund = `min(4_800, 4_780) = 4_780`
- `gas_used_after_refund = 23_900 − 4_780 = 19_120 < 21_000`

**ZK path**: `max(19_120, 0) = 19_120` — minimum not enforced.
**Ethereum path**: `max(19_120, 21_000) = 21_000` — minimum enforced.

The disabled EVM compatibility test confirms this is a known gap:

```yaml
# tests/evm_tester/indexes/stable-blockchain-tests.yaml lines 547-550
test_gas_limit_below_minimum.json:
  enabled: false
  comment: 'TODO: we do not enforce this limitation'
```

---

### Impact Explanation

An unprivileged user submitting a ZK transaction with SSTORE-to-zero operations can cause the final `gas_used` to fall below `TX_INTRINSIC_GAS = 21_000`. The operator/sequencer is paid for fewer gas units than the protocol minimum, resulting in systematic fee underpayment. The maximum underpayment per transaction is approximately `TX_INTRINSIC_GAS / 5 ≈ 4,200` gas units (bounded by the EIP-3529 refund cap). This is a resource accounting bug with direct financial impact on the sequencer.

---

### Likelihood Explanation

The attack requires only a standard ZK transaction that clears at least one warm storage slot (SSTORE to zero). This is a common operation (e.g., token approval reset, counter decrement to zero). No privileged access, oracle manipulation, or special tooling is required. Any unprivileged EOA can trigger this on every block.

---

### Recommendation

Change `compute_calldata_tokens` in `zk/validation_impl.rs` to return `TX_INTRINSIC_GAS` as the minimum gas used when `eip_7623` is disabled, consistent with the Ethereum path:

```rust
// zk/validation_impl.rs
#[cfg(not(feature = "eip_7623"))]
{
    (num_tokens, TX_INTRINSIC_GAS)  // was: (num_tokens, 0)
}
```

This ensures `minimal_ergs_to_charge` is always at least `Ergs(TX_INTRINSIC_GAS * ERGS_PER_GAS)`, and `compute_gas_refund` will enforce the 21,000-gas floor regardless of EVM refunds.

---

### Proof of Concept

1. Deploy a contract that stores a non-zero value in a storage slot.
2. Submit a ZK transaction that calls the contract to SSTORE that slot to zero (warm access, 2,900 gas).
3. Observe `gas_used_before_refund = 23,900`, refund = 4,780, `gas_used_final = 19,120`.
4. In the ZK path: `max(19_120, 0) = 19_120` — the operator is charged for 19,120 gas, below the 21,000 minimum.
5. In the Ethereum path: `max(19_120, 21_000) = 21_000` — the minimum is enforced.
