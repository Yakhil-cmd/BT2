The config field `max_nonce_for_validation_skip` exists in `StatefulTransactionValidatorConfig` but `skip_stateful_validations` is a free function that never receives it, hardcoding `Nonce(Felt::ONE)` instead.

---

### Title
`skip_stateful_validations` ignores `config.max_nonce_for_validation_skip`, causing valid invoke transactions with nonce > 1 to be rejected at the gateway â€” (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

`StatefulTransactionValidatorConfig` declares and serializes a `max_nonce_for_validation_skip` field intended to control the upper nonce bound for the deploy-account UX skip. The gateway's `skip_stateful_validations` free function never receives the config and hardcodes the check to `tx.nonce() == Nonce(Felt::ONE)`. Any invoke with nonce â‰Ą 2 sent alongside a pending deploy-account is forced through full blockifier validation, which fails because the account does not yet exist on-chain, causing the gateway to reject a transaction that the operator's config explicitly permits skipping.

### Finding Description

`StatefulTransactionValidatorConfig` defines:

```rust
pub max_nonce_for_validation_skip: Nonce,   // default: Nonce(Felt::ONE)
``` [1](#0-0) 

The config is serialized with the description *"Maximum nonce for which the validation is skipped"*, making it an operator-facing knob. [2](#0-1) 

However, `skip_stateful_validations` is a free function that accepts only `tx`, `account_nonce`, and `mempool_client`. It hardcodes the nonce equality check:

```rust
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
``` [3](#0-2) 

The call site in `run_pre_validation_checks` is a method on `StatefulTransactionValidator` (which holds `self.config`), but it never passes the config to the free function:

```rust
let skip_validate =
    skip_stateful_validations(executable_tx, account_nonce, mempool_client.clone()).await?;
``` [4](#0-3) 

By contrast, the legacy Python-path validator (`PyValidator::should_run_stateful_validations`) correctly reads `self.max_nonce_for_validation_skip` and allows skipping for any nonce in `[1, max_nonce_for_validation_skip]`:

```rust
let nonce_small_enough_to_qualify_for_validation_skip =
    tx_nonce <= self.max_nonce_for_validation_skip;
``` [5](#0-4) 

The Rust gateway path never implements this range check.

### Impact Explanation

When an operator sets `max_nonce_for_validation_skip` above `1` (e.g. `5`) and a user submits `deploy_account` + `invoke(nonce=1)` + `invoke(nonce=2)` simultaneously:

- `invoke(nonce=1)` correctly skips validation (hardcoded path still matches).
- `invoke(nonce=2)` does **not** skip; full blockifier validation runs against an account that does not yet exist on-chain â†’ `ValidateFailure` â†’ gateway rejects the transaction.

The transaction is valid and would succeed once the deploy-account is sequenced, but the gateway permanently rejects it. The operator's config value is silently dead code.

**Impact category:** High â€” gateway rejects valid transactions before sequencing.

### Likelihood Explanation

The default value of `max_nonce_for_validation_skip` is `Nonce(Felt::ONE)`, so the bug is latent at default settings. It activates the moment any operator raises the value above `1` â€” a configuration change that the field's own documentation invites. The deployed config files confirm the field is wired into production configuration. [6](#0-5) 

### Recommendation

Convert `skip_stateful_validations` from a free function into a method on `StatefulTransactionValidator` (or pass `max_nonce_for_validation_skip` as a parameter) and replace the hardcoded equality with a range check mirroring `PyValidator`:

```rust
if tx.nonce() >= Nonce(Felt::ONE)
    && tx.nonce() <= self.config.max_nonce_for_validation_skip
    && account_nonce == Nonce(Felt::ZERO)
```

Add a test case for `nonce=2` with `max_nonce_for_validation_skip=Nonce(Felt::from(5u64))` asserting that `skip_stateful_validations` returns `true`.

### Proof of Concept

```rust
#[tokio::test]
async fn test_skip_validate_nonce_2_with_extended_config() {
    // Config allows skipping up to nonce 5.
    let mut mock_mempool_client = MockMempoolClient::new();
    mock_mempool_client
        .expect_account_tx_in_pool_or_recent_block()
        .returning(|_| Ok(true)); // deploy_account IS in pool
    mock_mempool_client.expect_validate_tx().returning(|_| Ok(()));

    let mut mock_fixed_block = MockGatewayFixedBlockStateReader::new();
    let account_nonce = nonce!(0); // account not yet deployed
    mock_fixed_block
        .expect_get_nonce()
        .return_once(move |_| Ok(account_nonce));

    let stateful_validator: StatefulTransactionValidator<TestStateReader, _> =
        StatefulTransactionValidator {
            config: StatefulTransactionValidatorConfig {
                max_nonce_for_validation_skip: Nonce(Felt::from(5u64)), // operator raised limit
                validate_resource_bounds: false,
                ..Default::default()
            },
            chain_info: ChainInfo::create_for_testing(),
            state_reader_and_contract_manager: None,
            gateway_fixed_block_state_reader: mock_fixed_block,
        };

    let tx = executable_invoke_tx(invoke_tx_args!(nonce: nonce!(2)));
    let skip = stateful_validator
        .run_pre_validation_checks(&tx, account_nonce, Arc::new(mock_mempool_client))
        .await
        .unwrap();

    // FAILS: skip is false because skip_stateful_validations hardcodes nonce==1.
    // Expected: skip == true (config says max_nonce_for_validation_skip=5).
    assert!(skip, "Expected validation skip for nonce=2 with max_nonce_for_validation_skip=5");
}
```

The test asserts `skip == true` but the current implementation returns `false`, confirming the config field is never consulted.

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
