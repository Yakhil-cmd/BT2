### Title
Unvalidated `app_pk` Identity Element Allows Malicious Coordinator to Extract Confidential Derived Key - (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

The `ckd()` function in the Confidential Key Derivation (CKD) protocol accepts an `app_pk: PublicKey` (`ElementG1`) parameter without validating that it is a non-identity group element. If a malicious coordinator supplies `app_pk = G1::identity()` to participants, the masking term `app_pk * y` collapses to the identity, causing each participant to transmit their unmasked secret share contribution directly to the coordinator. The coordinator then aggregates these into the raw confidential derived key `msk · H(pk ∥ app_id)` — the exact secret the protocol is designed to hide from the coordinator.

---

### Finding Description

The CKD protocol is designed so that each participant computes a masked ciphertext:

```
C_i = x_i · H(pk ∥ app_id)  +  y_i · app_pk
```

where `x_i` is the participant's private share, `y_i` is a fresh random scalar, and `app_pk` is the TEE's public key. The masking term `y_i · app_pk` is what prevents the coordinator from learning the secret contribution `x_i · H(pk ∥ app_id)` directly.

In `compute_signature_share` at line 174:

```rust
let big_c = big_s + app_pk * y.0;
``` [1](#0-0) 

`app_pk` is used directly with no check that it is a valid, non-identity point. If `app_pk` is the identity element of G1, then `app_pk * y.0 = identity`, so:

```
big_c = big_s = x_i · H(pk ∥ app_id)
```

After Lagrange normalization (line 179–180):

```rust
let norm_big_y = big_y * lambda_i;
let norm_big_c = big_c * lambda_i;
``` [2](#0-1) 

each participant sends `(λ_i · y_i · G, λ_i · x_i · H(pk ∥ app_id))` to the coordinator. The coordinator aggregates in `do_ckd_coordinator` (lines 53–54):

```rust
norm_big_y += participant_output.big_y();
norm_big_c += participant_output.big_c();
``` [3](#0-2) 

yielding `total_big_c = Σ λ_i · x_i · H(pk ∥ app_id) = msk · H(pk ∥ app_id)` — the confidential derived key in the clear.

The `ckd()` entry point performs several input checks (participant deduplication, self-presence, coordinator-presence) but performs **no validation on `app_pk`**: [4](#0-3) 

This contrasts with the explicit identity-element rejection present in the G2 group deserializer:

```rust
if point.is_identity().into() {
    Err(frost_core::GroupError::InvalidIdentityElement)
}
``` [5](#0-4) 

and with the length guard on `AppId::try_new()`: [6](#0-5) 

No analogous guard exists for `app_pk`.

---

### Impact Explanation

The CKD protocol's confidentiality guarantee is that the coordinator learns only the masked value `C = msk · H(pk ∥ app_id) + a · Y`, from which it cannot recover `msk · H(pk ∥ app_id)` without the TEE's secret scalar `a`. Supplying `app_pk = identity` eliminates the masking term entirely, so the coordinator directly obtains `msk · H(pk ∥ app_id)` — the confidential derived key — without the TEE's involvement. This is a complete break of the protocol's confidentiality property and matches the allowed impact: **Critical: Extraction, reconstruction, or disclosure of confidential derived secrets**.

---

### Likelihood Explanation

In a real deployment the coordinator is the party responsible for communicating the TEE's `app_pk` to all participants (since `app_pk` is not transmitted over the protocol channel; it is a local input each participant supplies to their own `ckd()` call). A malicious coordinator can trivially substitute `G1::identity()` for the legitimate `app_pk` when instructing participants, triggering the attack with no cryptographic difficulty. The coordinator role is explicitly modeled as a potentially adversarial participant in threshold protocols.

---

### Recommendation

Validate `app_pk` in `ckd()` before constructing the protocol, rejecting the identity element:

```rust
if bool::from(app_pk.is_identity()) {
    return Err(InitializationError::BadParameters(
        "app_pk must not be the identity element".to_string(),
    ));
}
```

This mirrors the existing identity-element rejection in `BLS12381G1Group::deserialize` and `BLS12381G2Group::deserialize`. [7](#0-6) 

---

### Proof of Concept

1. Coordinator instructs all participants to call `ckd()` with `app_pk = G1Projective::identity()`.
2. Each participant's `compute_signature_share` computes `big_c = big_s + identity * y = big_s = x_i · H(pk ∥ app_id)`.
3. Each participant sends `(λ_i · y_i · G, λ_i · x_i · H(pk ∥ app_id))` to the coordinator.
4. Coordinator sums all `norm_big_c` values: `Σ λ_i · x_i · H(pk ∥ app_id) = msk · H(pk ∥ app_id)`.
5. Coordinator now holds the confidential derived key directly, without ever possessing the TEE's secret scalar `a`.

The existing test in `src/confidential_key_derivation/protocol.rs` confirms that `msk · H(pk ∥ app_id)` is the expected output of `ckd_output.unmask(app_sk)` — the coordinator obtains this value without calling `unmask` at all. [8](#0-7)

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}
