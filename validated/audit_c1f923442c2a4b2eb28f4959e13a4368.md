### Title
Declare transactions bypass `max_l2_gas_amount` admission check in `StatelessTransactionValidator` - (File: `crates/apollo_gateway/src/stateless_transaction_validator.rs`)

### Summary

The stateless gateway validator enforces a `max_l2_gas_amount` upper bound on L2 gas for Invoke and DeployAccount transactions, but explicitly skips this check for Declare transactions. This allows a Declare transaction with `l2_gas.max_amount` exceeding the configured limit to bypass gateway admission control and enter the mempool, violating the invariant that the gateway rejects transactions whose declared gas exceeds the operator-configured ceiling.

### Finding Description

In `StatelessTransactionValidator::validate_resource_bounds`, the `max_l2_gas_amount` upper-bound check is explicitly skipped for Declare transactions via an empty branch:

```rust
// TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
if let RpcTransaction::Declare(_) = tx {
} else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
        gas_amount: resource_bounds.l2_gas.max_amount,
        max_gas_amount: self.config.max_l2_gas_amount,
    });
}
```

The production default is `max_l2_gas_amount = 1_210_000_000`. A Declare transaction with `l2_gas.max_amount = 1_210_000_001` (or any value above the limit) passes stateless validation without error. The test `valid_l2_gas_amount_on_declare` explicitly confirms this behavior: a Declare with `max_amount = 200` passes when `max_l2_gas_amount = 100`.

**Downstream checks do not close the gap.** The stateful validator's `validate_resource_bounds` only checks that the L2 gas *price* exceeds a threshold relative to the previous block — it does not check the gas *amount*. The blockifier's `check_fee_bounds` in `perform_pre_validation_stage` only enforces a *lower* bound (`max_amount >= minimal_gas_amount`), not an upper bound. The only remaining check is `verify_can_pay_committed_bounds`, which verifies that the account balance covers `max_amount × max_price_per_unit`. With `max_amount = max_l2_gas_amount + 1 = 1_210_000_001` and `max_price_per_unit = min_gas_price = 8_000_000_000`, the committed fee is approximately `9.68 STRK` — a realistic balance for any funded account — so this check does not reject the transaction.

The `StatefulValidator::perform_validations` for Declare transactions calls `self.execute(tx)` (full execution), not just `perform_pre_validation_stage`. Full execution does not add any upper-bound check on `max_amount` either.

### Impact Explanation

**Impact: High — Mempool/gateway admission accepts invalid transactions before sequencing.**

A Declare transaction with `l2_gas.max_amount` exceeding `max_l2_gas_amount` is admitted by the gateway and enters the mempool. The gateway's own admission policy is violated. The `max_l2_gas_amount` limit exists to bound the gas a single transaction can claim, protecting block capacity and preventing mempool pollution. Bypassing it for Declare transactions allows an attacker to submit Declare transactions that claim more L2 gas than the operator intends to allow, potentially displacing other transactions from blocks or exhausting block gas capacity.

### Likelihood Explanation

Any unprivileged user can trigger this by submitting a well-formed Declare transaction (valid Sierra class, valid signature, valid nonce, valid price) with `l2_gas.max_amount` set to `max_l2_gas_amount + 1`. No special privileges or unusual conditions are required. The stateless validator will accept it unconditionally.

### Recommendation

Remove the empty `if let RpcTransaction::Declare(_) = tx {}` branch and apply the `max_l2_gas_amount` check uniformly to all transaction types, including Declare. The existing TODO comment acknowledges this gap; it should be resolved rather than deferred:

```rust
// Apply to all transaction types including Declare.
if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
        gas_amount: resource_bounds.l2_gas.max_amount,
        max_gas_amount: self.config.max_l2_gas_amount,
    });
}
```

The test `valid_l2_gas_amount_on_declare` would need to be updated to reflect the corrected behavior.

    pub fn perform_validations(&mut self, tx: AccountTransaction) -> StatefulValidatorResult<()> {
        // Deploy account transaction should be fully executed, since the constructor must run
        // before `__validate_deploy__`. The execution already includes all necessary validations,
        // so they are skipped here.
        // Declare transaction should also be fully executed - otherwise, if we only go through
        // the validate phase, we would miss the check that the class was not declared before.
        match tx.tx {
            ApiTransaction::DeployAccount(_) | ApiTransaction::Declare(_) => self.execute(tx),
            ApiTransaction::Invoke(_) => {
                let tx_context = Arc::new(self.tx_executor.block_context.to_tx_context(&tx));
                tx.perform_pre_validation_stage(self.state(), &tx_context)?;
                if !tx.execution_flags.validate {
                    return Ok(());
                }

                // `__validate__` call.
                let (_optional_call_info, actual_cost) = self.validate(&tx, tx_context.clone())?;

                // Post validations.
                PostValidationReport::verify(
                    &tx_context,
                    &actual_cost,
                    tx.execution_flags.charge_fee,
                )?;

                Ok(())
            }
        }
    }
