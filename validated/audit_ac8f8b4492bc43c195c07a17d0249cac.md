The `increased_enough` function in `crates/apollo_mempool/src/mempool.rs` does have a real intermediate-overflow bug. Here is the precise analysis:

---

### Title
Intermediate u128 Overflow in `increased_enough` Causes Valid Fee-Escalation Replacement to Be Rejected — (`crates/apollo_mempool/src/mempool.rs`)

---

### Summary

`Mempool::increased_enough` computes `existing_value * percentage` as the first step. When `existing_value > u128::MAX / percentage`, this `checked_mul` returns `None` and the function returns `false`, rejecting the replacement. However, the final threshold `existing_value + (existing_value * percentage / 100)` can be well within u128 range, making the rejection incorrect for a mathematically valid replacement.

---

### Finding Description

The function:

```rust
fn increased_enough(&self, existing_value: u128, incoming_value: u128) -> bool {
    let percentage = u128::from(self.config.static_config.fee_escalation_percentage);
    let Some(escalation_qualified_value) = existing_value
        .checked_mul(percentage)          // ← overflows here
        .map(|scaled| scaled.div_ceil(100))
        .and_then(|required_increase| existing_value.checked_add(required_increase))
    else {
        return false;                     // ← rejects valid replacement
    };
    incoming_value >= escalation_qualified_value
}
``` [1](#0-0) 

The overflow threshold for `checked_mul` is `existing_value > u128::MAX / percentage`. With the production default `fee_escalation_percentage = 10`: [2](#0-1) 

- Overflow occurs when `existing_value > u128::MAX / 10 ≈ 3.4 × 10^37`
- Concrete example: `existing_max_l2_gas_price = u128::MAX / 2 + 1 = 170141183460469231731687303715884105728`
  - `existing_value * 10` overflows → `checked_mul` returns `None` → returns `false`
  - But the correct threshold is `existing_value + existing_value/10 ≈ 1.87 × 10^38`, which is well within u128::MAX
  - `incoming_value = u128::MAX = 3.4 × 10^38 > 1.87 × 10^38` → replacement IS mathematically valid

The `tip` field is a `u64` (`tx.tip.0` cast to u128), so `u64::MAX * 100 < u128::MAX` — tip overflow is not reachable. The vulnerability is confined to `max_l2_gas_price`, which is a raw `u128` (`GasPrice(u128)`). [3](#0-2) 

**Gateway has no upper bound on `max_l2_gas_price`.** The stateless validator only enforces a lower bound (`max_price_per_unit >= min_gas_price`) and an upper bound on `max_amount`, not on `max_price_per_unit`: [4](#0-3) 

So a transaction with `max_l2_gas_price = u128::MAX / 2 + 1` passes all gateway checks and enters the mempool.

The existing test `test_fee_escalation_invalid_replacement_overflow_gracefully_handled` only covers cases where the replacement is also genuinely invalid (e.g., `existing = u128::MAX - 10`, where the required threshold itself overflows u128). It does **not** cover the case where the replacement is valid but rejected due to intermediate overflow: [5](#0-4) 

---

### Impact Explanation

A valid fee-escalation replacement is rejected with `MempoolError::DuplicateNonce`. The user's original transaction remains stuck in the mempool and cannot be replaced with a higher-fee version, violating the fee-escalation admission invariant. [6](#0-5) 

This falls under: **High — Mempool/gateway/RPC admission rejects valid transactions before sequencing.**

---

### Likelihood Explanation

**Low.** The trigger requires `max_l2_gas_price > u128::MAX / 10 ≈ 3.4 × 10^37`. Real-world gas prices are orders of magnitude smaller. The scenario also requires the same account owner to first submit a transaction with such an extreme price and then attempt to replace it. There is no cross-account attack vector since Starknet signature validation prevents submitting transactions for another account.

---

### Recommendation

Reorder the arithmetic to avoid the intermediate overflow. Compute the required increase as `existing_value / 100 * percentage` (integer division first), then add the ceiling correction:

```rust
fn increased_enough(&self, existing_value: u128, incoming_value: u128) -> bool {
    let percentage = u128::from(self.config.static_config.fee_escalation_percentage);
    // Compute required_increase = ceil(existing_value * percentage / 100)
    // without intermediate overflow by dividing before multiplying.
    let required_increase = existing_value / 100 * percentage
        + existing_value.rem_euclid(100) * percentage / 100
        + if (existing_value % 100) * percentage % 100 != 0 { 1 } else { 0 };
    let Some(escalation_qualified_value) = existing_value.checked_add(required_increase) else {
        return false;
    };
    incoming_value >= escalation_qualified_value
}
```

Or more simply, use 128-bit-safe division: `(existing_value * percentage).div_ceil(100)` can be replaced with `existing_value.div_ceil(100) * percentage + ...` using a u256 intermediate or the identity `ceil(a*b/c) = (a/c)*b + ceil((a%c)*b/c)`.

### Proof of Concept

```rust
#[test]
fn test_overflow_rejects_valid_replacement() {
    // existing_max_l2_gas_price = u128::MAX / 2 + 1
    // 10% of this = ~1.7e37, threshold = ~1.87e38, well within u128::MAX
    // incoming = u128::MAX > threshold → should be admitted
    let existing_tx = tx!(tx_hash: 0, tip: 0, max_l2_gas_price: u128::MAX / 2 + 1);
    let mempool = MempoolTestContentBuilder::new()
        .with_pool([existing_tx])
        .with_fee_escalation_percentage(10)
        .build_full_mempool();

    let replacement = add_tx_input!(tx_hash: 1, tip: 1, max_l2_gas_price: u128::MAX);
    // BUG: this is rejected with DuplicateNonce due to intermediate overflow
    // Expected: replacement admitted
    validate_and_add_tx_and_verify_replacement_in_pool(mempool, replacement);
}
```

The test will fail on the current code because `increased_enough(u128::MAX / 2 + 1, u128::MAX)` returns `false` due to `checked_mul` overflow, even though `u128::MAX` exceeds the correct 10%-escalated threshold.
