### Title
`max_nonce_for_validation_skip` Config Field Never Consumed by `skip_stateful_validations` — Gateway Skips `__validate__` Regardless of Operator Setting - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

`StatefulTransactionValidatorConfig` exposes a `max_nonce_for_validation_skip` field that is serialized, deserialized, and documented as "Maximum nonce for which the validation is skipped." However, the free function `skip_stateful_validations` that decides whether to skip the `__validate__` entry point hardcodes `tx.nonce() == Nonce(Felt::ONE)` and never reads the config field. An operator who sets `max_nonce_for_validation_skip = 0` to enforce strict signature validation for every transaction will find the skip still fires for every invoke with nonce 1 from an undeployed account, allowing those transactions to bypass the `__validate__` entry point.

### Finding Description

`StatefulTransactionValidatorConfig` declares and serializes `max_nonce_for_validation_skip`: [1](#0-0) 

The field is also emitted to the public config schema: [2](#0-1) 

The free function `skip_stateful_validations` is the sole decision point for whether `__validate__` is skipped. It is a free function — it receives no `config` argument — and hardcodes the nonce threshold: [3](#0-2) 

It is called from `run_pre_validation_checks`, which **does** hold `self.config` but never passes `self.config.max_nonce_for_validation_skip` to the free function: [4](#0-3) 

The result of `skip_stateful_validations` is used to set `validate: !skip_validate` in `ExecutionFlags`, which controls whether the blockifier calls the account's `__validate__` entry point: [5](#0-4) 

By contrast, the legacy `PyValidator` correctly reads `self.max_nonce_for_validation_skip` and applies it as an upper bound: [6](#0-5) 

A grep for `max_nonce_for_validation_skip` across the entire repo confirms the field appears only in `PyValidator`, the config struct, the config schema, and deployment JSON files — never in the Apollo gateway's stateful validator logic.

### Impact Explanation

When `skip_validate = true`, `ExecutionFlags.validate` is `false`, and `StatefulValidator::perform_validations` returns `Ok(())` for an invoke transaction without ever calling `tx.validate_tx(...)`: [7](#0-6) 

This means the account's `__validate__` entry point — which is responsible for signature verification — is never executed. An operator who sets `max_nonce_for_validation_skip = Nonce(Felt::ZERO)` to enforce strict validation for all transactions will find the gateway still admits invoke transactions with nonce 1 from undeployed accounts without running their `__validate__` entry point, regardless of the configured value.

### Likelihood Explanation

The default value of `max_nonce_for_validation_skip` is `Nonce(Felt::ONE)`, which coincidentally matches the hardcoded constant, so the bug is invisible under default configuration. It surfaces only when an operator explicitly sets the field to `Nonce(Felt::ZERO)` to disable the skip, or to a value greater than 1 to extend it. The field is publicly documented and present in the deployment config files, making operator misconfiguration a realistic scenario.

### Recommendation

Convert `skip_stateful_validations` from a free function into a method on `StatefulTransactionValidator` (or pass `max_nonce_for_validation_skip` as an explicit argument) and replace the hardcoded equality check with the configurable upper-bound check, mirroring the `PyValidator` logic:

```rust
// In run_pre_validation_checks, pass the config value:
let skip_validate = skip_stateful_validations(
    executable_tx,
    account_nonce,
    mempool_client.clone(),
    self.config.max_nonce_for_validation_skip,  // <-- add this
).await?;

// In skip_stateful_validations, use it as an upper bound:
if tx.nonce() >= Nonce(Felt::ONE)
    && tx.nonce() <= max_nonce_for_validation_skip  // <-- replaces == Nonce(Felt::ONE)
    && account_nonce == Nonce(Felt::ZERO)
{
    ...
}
```

### Proof of Concept

1. Operator deploys the gateway with `max_nonce_for_validation_skip = Nonce(Felt::ZERO)` in `StatefulTransactionValidatorConfig`, intending to disable the validation-skip UX feature.
2. Attacker submits a `DeployAccount` transaction for a fresh account address. This goes through `StatefulValidator::execute`, which runs `__validate_deploy__` and the constructor — the attacker signs it correctly.
3. Attacker immediately submits an `Invoke` transaction from the same address with `nonce = 1` and a **deliberately invalid signature**.
4. `skip_stateful_validations` evaluates `tx.nonce() == Nonce(Felt::ONE)` → `true`, `account_nonce == Nonce(Felt::ZERO)` → `true`, and `account_tx_in_pool_or_recent_block` → `true` (the deploy is in the mempool). It returns `true` (skip), ignoring the operator's `max_nonce_for_validation_skip = 0`.
5. `run_validate_entry_point` sets `validate = false`; the blockifier never calls `__validate__`.
6. The invoke transaction with the invalid signature is forwarded to the mempool and admitted for sequencing — contradicting the operator's explicit configuration. [8](#0-7) [1](#0-0)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
