### Title
`skip_stateful_validations` Bypasses `__validate__` for Invoke Transactions When Any Pool Transaction Exists, Not Only `deploy_account` - (File: crates/apollo_gateway/src/stateful_transaction_validator.rs)

### Summary

`skip_stateful_validations` is designed to skip the `__validate__` entry-point call for an invoke with nonce=1 when a `deploy_account` is pending but not yet processed. However, the guard condition uses `account_tx_in_pool_or_recent_block`, which returns `true` for **any** transaction from the sender — including a prior invoke with nonce=0. An attacker who controls an existing account (state nonce=0) can first submit a valid invoke with nonce=0, then submit a second invoke with nonce=1 carrying an **invalid signature**. The gateway skips `__validate__` for the second transaction and admits it to the mempool without signature verification.

### Finding Description

The two-step initialization analog to the Timed bug is:

1. **Step 1 (analogous to `Timed.duration` set in constructor):** `account_nonce = 0` is read from state. This value is ambiguous — it could mean "account not yet deployed" (the intended case for the UX skip) or "account deployed but no transactions committed yet."

2. **Step 2 (analogous to `Timed.startTime` never set):** `account_tx_in_pool_or_recent_block` is called to confirm a `deploy_account` is pending. But this function returns `true` for **any** transaction from the address, not specifically a `deploy_account`.

The check in `skip_stateful_validations`:

```rust
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
    return mempool_client
        .account_tx_in_pool_or_recent_block(tx.sender_address())
        .await
        ...
}
``` [1](#0-0) 

`account_tx_in_pool_or_recent_block` is implemented as:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [2](#0-1) 

This returns `true` if **any** transaction from the address is in the pool — including an invoke with nonce=0. The comment in the code acknowledges this ambiguity but incorrectly reasons it is safe:

> "It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."

The reasoning is flawed: a valid signature for nonce=0 does not imply a valid signature for nonce=1. Starknet account `__validate__` functions verify the signature of the **specific transaction** (including its nonce, calldata, and hash).

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` sets `validate: false`:

```rust
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [3](#0-2) 

And `StatefulValidator::perform_validations` returns `Ok(())` without calling `__validate__`:

```rust
if !tx.execution_flags.validate {
    return Ok(());
}
// `__validate__` call.
``` [4](#0-3) 

The nonce check in `validate_nonce` permits nonce=1 for an account with state nonce=0 (within the `max_allowed_nonce_gap` of 200):

```rust
let max_allowed_nonce =
    Nonce(account_nonce.0 + Felt::from(self.config.max_allowed_nonce_gap));
if !(account_nonce <= incoming_tx_nonce && incoming_tx_nonce <= max_allowed_nonce) {
``` [5](#0-4) 

### Impact Explanation

An attacker controlling an account with state nonce=0 (deployed, no committed transactions) can:

1. Submit a valid invoke with nonce=0 — passes all validations including `__validate__` — enters mempool.
2. Submit an invoke with nonce=1 carrying an **invalid/forged signature**:
   - `validate_nonce`: `0 ≤ 1 ≤ 200` → passes.
   - `skip_stateful_validations`: nonce=1, account_nonce=0, pool contains nonce=0 tx → returns `true`.
   - `run_validate_entry_point`: `validate=false` → `__validate__` not called → passes.
   - Transaction admitted to mempool without signature verification.

The invalid transaction is propagated to the mempool and will be included in a block (where it will be reverted by the batcher, which does call `__validate__`). This violates the gateway admission invariant: **all transactions in the mempool must have passed account signature verification**.

Impact: **High — Mempool/gateway admission accepts an invalid transaction (invalid signature) before sequencing.**

### Likelihood Explanation

Any account holder with state nonce=0 can trigger this. No privileged access is required. The attacker only needs to submit two transactions in sequence. The condition `account_nonce == Nonce(Felt::ZERO)` is satisfied by any freshly deployed account that has never committed a transaction.

### Recommendation

Replace the `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a `deploy_account` transaction is pending for the sender address. The mempool should expose a dedicated `has_pending_deploy_account(address)` query, or the gateway should track pending deploy-account hashes per address and only skip validation when a deploy-account is confirmed pending.

Alternatively, restrict the skip to the case where the account does **not** exist in state at all (i.e., `get_class_hash_at(sender_address) == ClassHash::default()`), which is the true indicator that the account is undeployed and a deploy_account must be pending.

### Proof of Concept

```
1. Deploy account A (state nonce becomes 0).
2. POST /add_transaction: Invoke from A, nonce=0, valid signature S0.
   → Gateway: validate_nonce passes (0≤0≤200), __validate__ called with S0 → OK.
   → Mempool: tx with nonce=0 stored.

3. POST /add_transaction: Invoke from A, nonce=1, INVALID signature S_bad.
   → Gateway: validate_nonce passes (0≤1≤200).
   → skip_stateful_validations: nonce==1 && account_nonce==0 && account_tx_in_pool_or_recent_block==true → returns true.
   → run_validate_entry_point: validate=false → __validate__ NOT called.
   → Transaction accepted into mempool with invalid signature.

4. Batcher picks up nonce=1 tx, calls __validate__ with S_bad → reverts.
   Invalid tx was admitted to mempool and included in block without gateway signature check.
``` [6](#0-5) [7](#0-6)

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
