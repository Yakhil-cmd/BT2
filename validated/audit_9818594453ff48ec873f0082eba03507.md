Audit Report

## Title
Silent ICP Burn Failure After Cycles Minting Violates ICP/Cycles Conservation Invariant — (File: rs/nns/cmc/src/main.rs)

## Summary
The `burn_and_log` function in the Cycles Minting Canister is intentionally designed to swallow all errors from the ICP ledger burn call. When the ICP ledger is transiently unreachable (e.g., during a routine upgrade), cycles are permanently minted and deposited while the corresponding ICP is never burned. The notification block is then permanently closed, preventing any retry, and the ICP remains stranded in the CMC subaccount indefinitely with no recovery path.

## Finding Description
Three async functions follow a two-phase pattern where Phase 1 (cycle minting) is irreversible and Phase 2 (ICP burn) is fire-and-forget:

- `process_create_canister` calls `do_create_canister`, then `burn_and_log` on success
- `process_mint_cycles` calls `do_mint_cycles`, then `burn_and_log` on success
- `process_top_up` calls `deposit_cycles`, then `burn_and_log` on success

The `burn_and_log` function at lines 2014–2049 is explicitly designed to never return an error. Its comment states: *"Burning doesn't return errors - we don't want to reject the transaction notification because then it could be retried."* When `call_protobuf` to `send_pb` fails, the error is only logged; the function returns `()` unconditionally.

Because `burn_and_log` always returns `()`, the callers always return `Ok(...)` after Phase 1 succeeds. The notification handlers then permanently record the block as `NotifiedTopUp(Ok(cycles))`, `NotifiedCreateCanister(Ok(...))`, or `NotifiedMint(Ok(...))`. The `is_transient_error` guard at lines 1219–1221, 1310–1312, and 1416–1418 only removes the block status on transient errors from the process functions themselves — since `burn_and_log` never propagates errors, this guard is never triggered for a burn failure. There is no heartbeat, timer, or pending-burn queue to retry failed burns.

The `SendArgs` in `burn_and_log` sets `created_at_time: None`, meaning even if a retry were attempted manually, there is no deduplication protection.

## Impact Explanation
This constitutes illegal minting of cycles: cycles are permanently created and deposited to the target without the corresponding ICP being destroyed. The ICP/cycles conservation invariant (1 XDR of ICP = 1 trillion cycles) is violated. The stranded ICP has no recovery path — the notification is closed, the user cannot re-notify, and no background process retries the burn. This maps to the High impact category: significant NNS/ledger security impact with concrete protocol harm (illegal minting and permanent ICP stranding), bounded per-incident by individual transaction amounts and a narrow trigger window.

## Likelihood Explanation
The trigger requires the ICP ledger to be transiently unreachable at the exact moment `burn_and_log` issues its `send_pb` call. This occurs during routine ICP ledger upgrades on the NNS subnet, which happen multiple times per year. During an upgrade, the ledger canister is stopped/upgraded, causing inter-canister calls to return `CanisterStopped` or `CanisterStopping` rejections for the duration. Any unprivileged user who calls `notify_top_up`, `notify_create_canister`, or `notify_mint_cycles` during this window — after their Phase 1 call completes but before the burn call returns — will trigger the condition. No special privilege is required; all three `notify_*` endpoints are open to any caller.

## Recommendation
1. **Retry via timer/heartbeat**: When `burn_and_log` fails, record a "pending burn" entry in stable state (subaccount + amount). A periodic timer retries all pending burns until they succeed. This preserves the no-retry guarantee for the user-facing notification while ensuring the burn eventually completes.
2. **Idempotent burn via `created_at_time`**: Set `created_at_time` in the `SendArgs` to a deterministic value derived from the notification block index. This makes the burn call idempotent, so a retry is safe even if the original call was delivered but the response was lost.

## Proof of Concept
1. User sends N ICP to `CMC_subaccount(canister_X)` on the ICP ledger at block index `B`.
2. User calls `notify_top_up { block_index: B, canister_id: canister_X }`.
3. CMC sets block `B` status to `Processing`.
4. CMC calls `deposit_cycles(canister_X, cycles, ...)` → management canister deposits cycles. **Cycles are now live.**
5. CMC calls `burn_and_log(subaccount_X, N ICP)` → ICP ledger is mid-upgrade → `send_pb` returns `Err(CanisterStopped, ...)`.
6. `burn_and_log` logs the failure and returns `()`.
7. `process_top_up` returns `Ok(cycles)`.
8. CMC sets block `B` status to `NotifiedTopUp(Ok(cycles))` — permanently closed.
9. **Result**: `canister_X` holds the cycles; N ICP sits unburned in `CMC_subaccount(canister_X)` forever; the ICP/cycles conservation invariant is violated with no recovery path.

A deterministic integration test using `StateMachine` can reproduce this by: installing the CMC and ledger, stopping the ledger canister to simulate an upgrade, then calling `notify_top_up` and asserting that cycles were deposited while the CMC subaccount balance remains non-zero and no burn block was recorded. /// Attempt to burn the funds.
/// Burning doesn't return errors - we don't want to reject the transaction
/// notification because then it could be retried.
async fn burn_and_log(from_subaccount: Subaccount, amount: Tokens) {
    let msg = format!("Burning of {amount} ICPTs from subaccount {from_subaccount}");
    let minting_account_id = with_state(|state| state.minting_account_id);
    if minting_account_id.is_none() {
        print(format!("{msg} failed: minting_account_id not set"));
        return;
    }
    let minting_account_id = minting_account_id.unwrap();
    let ledger_canister_id = with_state(|state| state.ledger_canister_id);

    if amount < DEFAULT_TRANSFER_FEE {
        print(format!("{msg}: amount too small ({amount})"));
        return;
    }

    let send_args = SendArgs {
        memo: Memo::default(),
        amount,
        fee: Tokens::ZERO,
        from_subaccount: Some(from_subaccount),
        to: minting_account_id,
        created_at_time: None,
    };
    let res: CallResult<BlockIndex> = call_protobuf(ledger_canister_id, "send_pb", send_args).await;

    match res {
        Ok(block) => print(format!("{msg} done in block {block}.")),
        Err((code, err)) => {
            let code = code as i32;
            print(format!("{msg} failed with code {code}: {err:?}"))
        }
    }
}
