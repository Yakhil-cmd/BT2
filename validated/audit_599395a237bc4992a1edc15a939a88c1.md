### Title
Use-After-Revocation via Duplicate Key Index: Revoking Key 0 Does Not Revoke Duplicate Key Material at Index 1 — (`fvm/environment/account_public_key_util.go`, `fvm/environment/accounts.go`, `fvm/environment/accounts_status.go`)

### Summary
Revoking account key index 0 only sets `Revoked=true` in the `apk_0` register. A duplicate of key 0 added at index 1 stores its own independent revoked-status bit in key metadata (account status register). After key 0 is revoked, key 1 retains `Revoked=false` and can still be used to authorize transactions with the same private key material, violating the invariant that revoking a key must prevent all signing with that key material.

### Finding Description

Flow's key storage uses two separate revocation stores:

1. **Key 0**: revoked flag is embedded in the `apk_0` register.
2. **Keys ≥ 1**: revoked flag is stored in key metadata bytes inside the account status register.

When a duplicate of key 0 is added at index 1, `AppendDuplicateKeyMetadata` stores `revoked=false` in key metadata and a mapping `key_index_1 → stored_key_index_0`. The stored key bytes are shared (deduplication), but the revoked status is not.

`revokeAccountPublicKey0` only writes to `apk_0`: [1](#0-0) 

`RevokeAccountPublicKey` for key index 1 writes only to key metadata in the account status register: [2](#0-1) 

`GetRuntimeAccountPublicKey` for key 1 reads `revoked` from key metadata, not from `apk_0`: [3](#0-2) 

`getStoredPublicKey` for `storedKeyIndex=0` reads from `apk_0` but returns only `{PublicKey, SignAlgo, HashAlgo}` — the revoked field is not propagated: [4](#0-3) 

Both the sequence number checker and the transaction verifier check the revoked status of the specific key index used in the transaction: [5](#0-4) [6](#0-5) 

Since key 1's revoked status is `false` in key metadata, both checks pass.

### Impact Explanation

An attacker who controls an account (has the private key) can:
1. Add a duplicate of key 0 at index 1 (requires `AddKey` entitlement, which they have via the private key).
2. The account owner (or the attacker themselves) revokes key 0.
3. The attacker continues to sign and submit transactions using the same private key material via `ProposalKey.KeyIndex=1`.

This enables **use-after-revocation**: the attacker maintains full transaction authorization capability with key material that has been explicitly revoked. In a compromise scenario where the account owner revokes key 0 to cut off an attacker, the attacker retains access indefinitely via the pre-planted duplicate at index 1.

### Likelihood Explanation

The attack requires the attacker to have already obtained the private key for key 0 and to have submitted a transaction adding the duplicate before the revocation occurs. This is a realistic post-compromise persistence technique. The `AddKey` entitlement is available to anyone who can sign a transaction with the account's key, which the attacker already can do.

### Recommendation

When revoking a key at any index, the system should also revoke all other key indices that share the same stored key index (i.e., all duplicates). Alternatively, `GetAccountPublicKeyRevokedStatus` for a duplicate key should also check the revoked status of the canonical stored key index it maps to. A simpler fix: when `GetRuntimeAccountPublicKey` resolves a duplicate key (i.e., `storedKeyIndex != keyIndex`), it should also check whether the canonical stored key (e.g., key 0 via `apk_0`) is revoked and OR that with the key metadata revoked bit.

### Proof of Concept

```cadence
// Transaction 1: Add duplicate of key 0 at index 1 (requires AddKey entitlement)
transaction(pubKeyBytes: [UInt8]) {
  prepare(signer: auth(AddKey) &Account) {
    signer.keys.add(
      publicKey: PublicKey(
        publicKey: pubKeyBytes,  // same bytes as key 0
        signatureAlgorithm: SignatureAlgorithm.ECDSA_P256
      ),
      hashAlgorithm: HashAlgorithm.SHA3_256,
      weight: 1000.0
    )
  }
}

// Transaction 2: Revoke key 0
transaction {
  prepare(signer: auth(RevokeKey) &Account) {
    signer.keys.revoke(keyIndex: 0)
  }
}

// Transaction 3: Sign with same private key but ProposalKey.KeyIndex=1
// Expected: rejected (key material revoked)
// Actual: ACCEPTED — key 1's revoked status in key metadata is still false
```

After step 2, `GetAccountPublicKeyRevokedStatus(addr, 0)` returns `true` (from `apk_0`), but `GetAccountPublicKeyRevokedStatus(addr, 1)` returns `false` (from key metadata). Transaction 3 passes both the sequence number checker and the transaction verifier, and the cryptographic signature is valid because the stored public key for index 1 maps to the same bytes as key 0.
