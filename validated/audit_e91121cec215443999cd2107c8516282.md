### Title
Missing Lower-Bound Validation on `threshold` in Signing Initialization Allows Threshold Security Guarantee Bypass - (File: src/frost/mod.rs)

### Summary
The `assert_sign_inputs` and `presign` functions in `src/frost/mod.rs`, and the `sign` function in `src/ecdsa/ot_based_ecdsa/sign.rs`, validate only the upper bound of the `threshold` parameter (`threshold <= participants.len()`) but omit the lower-bound check (`threshold >= 2`) that is correctly enforced during DKG in `src/dkg.rs`. A malicious coordinator or library caller can pass `threshold = 1` (or `threshold = 0`) to these entry points, bypassing the threshold enforcement and causing signing to proceed with fewer participants than the DKG threshold requires, producing corrupted or unusable signature outputs.

---

### Finding Description

During DKG, `assert_key_invariants` in `src/dkg.rs` enforces both bounds:

```rust
// upper bound
if threshold > participants.len() { return Err(...ThresholdTooLarge...) }
// lower bound
if threshold < 2 { return Err(...ThresholdTooSmall...) }
``` [1](#0-0) 

This guarantees that every key produced by DKG has `threshold >= 2`.

However, the signing-phase entry points apply only the upper bound and omit the lower bound entirely.

**`src/frost/mod.rs` — `assert_sign_inputs` (line 145):**
```rust
if threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
// ← no check: threshold.value() < 2
``` [2](#0-1) 

**`src/frost/mod.rs` — `presign` (line 72):**
```rust
if args.threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
// ← no check: args.threshold.value() < 2
``` [3](#0-2) 

**`src/ecdsa/ot_based_ecdsa/sign.rs` — `sign` (line 58):**
```rust
if participants.len() < threshold {
    return Err(InitializationError::NotEnoughParticipantsForThreshold { ... });
}
// ← no check: threshold < 2
``` [4](#0-3) 

`ReconstructionLowerBound` is a plain `usize` newtype with no internal invariant, so any value including `0` and `1` can be constructed and passed freely. [5](#0-4) 

---

### Impact Explanation

**High — Corruption of sign outputs so honest parties accept unusable cryptographic outputs.**

Consider a key generated with DKG threshold = 3 and 5 participants. A malicious coordinator calls the OT-based ECDSA `sign` (or FROST `assert_sign_inputs`) with:
- `participants = [P1, P2]` (only 2 participants — the minimum allowed by the `< 2` guard)
- `threshold = 1` (passes the only check: `2 < 1` → false)

The signing protocol proceeds. Inside `compute_signature_share`, Lagrange coefficients are computed over the 2-participant set:
```rust
let lambda = participants.lagrange::<Secp256K1Sha256>(me)?;
let k_i = lambda * presignature.k;
let sigma_i = lambda * presignature.sigma;
``` [6](#0-5) 

Because the presignature shares are degree-(threshold−1) = degree-2 polynomial shares (requiring 3 participants to reconstruct), interpolating over only 2 participants yields incorrect values for `k` and `sigma`. The coordinator aggregates these incorrect shares, the final signature fails the internal verification check, and honest parties receive an unusable output. The same logic applies to FROST signing where the Lagrange reconstruction over an under-sized participant set produces an invalid aggregate signature.

---

### Likelihood Explanation

**Medium.** The `threshold` parameter is caller-supplied with no type-level enforcement. Any application layer that exposes the signing entry points to a coordinator role — the documented trust model — allows the coordinator to pass `threshold = 1`. No cryptographic capability or key material is required; only the ability to invoke the public API with an out-of-range parameter value.

---

### Recommendation

Add the same lower-bound guard used in `assert_key_invariants` to every signing-phase entry point:

```rust
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold: threshold.value(), min: 2 });
}
```

Apply this to:
- `assert_sign_inputs` in `src/frost/mod.rs`
- `presign` in `src/frost/mod.rs`
- `sign` in `src/ecdsa/ot_based_ecdsa/sign.rs`

Optionally, encode the invariant directly in `ReconstructionLowerBound` by making its constructor return `Result` and rejecting values below 2, so the bound is enforced at the type level across all call sites. [5](#0-4) 

---

### Proof of Concept

1. Run DKG with 5 participants and `threshold = 3`. Each participant receives a degree-2 polynomial share of the secret key.
2. Run presign with the same 5 participants and `threshold = 3`. Each participant receives presignature shares `(k_i, sigma_i)` that are degree-2 polynomial shares.
3. A malicious coordinator calls `ecdsa::ot_based_ecdsa::sign` with:
   - `participants = [P1, P2]`
   - `threshold = 1`
4. Validation passes: `participants.len() (2) < 2` → false; `participants.len() (2) < threshold (1)` → false.
5. `compute_signature_share` computes Lagrange coefficients for the 2-participant set `{P1, P2}` and interpolates `k` and `sigma` from only 2 degree-2 shares — an under-determined reconstruction.
6. The coordinator aggregates the two incorrect shares; the resulting `(R, s)` fails `sig.verify(&public_key, &msg_hash)`. [7](#0-6) 
7. Honest parties receive `Err(ProtocolError::AssertionFailed("signature failed to verify"))` — a permanently unusable signing output for that presignature, matching the **High: Corruption of sign outputs** impact.
