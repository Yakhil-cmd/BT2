### Title
Gateway `run_validate_entry_point` uses `unchecked_next()` block number without simulating `pre_process_block`, causing false rejection of valid proof-bearing transactions - (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

`StatefulTransactionValidator::run_validate_entry_point` advances the block number by one (`unchecked_next()`) to simulate the next block, but runs `validate_proof_facts` against the **current block's state** — which has not yet had `pre_process_block` applied. This creates a one-block mismatch: the block-number check allows `proof_block_number` up to `N − STORED_BLOCK_HASH_BUFFER + 1`, but the state only contains stored hashes up to `N − STORED_BLOCK_HASH_BUFFER`. Any proof-bearing invoke transaction that references the most-recently-allowed block is rejected at the gateway with "Block hash mismatch", even though the identical transaction would succeed during actual blockifier execution of block N+1.

### Finding Description

**Step 1 — Gateway builds a synthetic block context with block number N+1 but state at block N.** [1](#0-0) 

`get_block_info()` returns the latest committed block (N). `unchecked_next()` produces N+1. The `CachedState` created on line 339 reads from the state at block N — `pre_process_block` for block N+1 has not run.

**Step 2 — `validate_proof_block_number` uses the synthetic block number N+1.** [2](#0-1) 

`max_allowed = (N+1) − STORED_BLOCK_HASH_BUFFER`. With `STORED_BLOCK_HASH_BUFFER = 10`: [3](#0-2) 

`max_allowed = N − 9`. A transaction with `proof_block_number = N − 9` passes this check.

**Step 3 — `validate_proof_block_hash` reads from state at block N.** [4](#0-3) 

`pre_process_block` for block N stored the hash of block `N − 10`. The hash of block `N − 9` is **not yet stored**. `get_storage_at` returns `Felt::ZERO`. Since the proof supplies a non-zero hash, the check fails: `stored_block_hash (ZERO) ≠ proof_block_hash` → "Block hash mismatch".

**Step 4 — During actual execution of block N+1, the same transaction succeeds.** [5](#0-4) 

`pre_process_block` runs before any transaction, writing the hash of block `(N+1) − 10 = N − 9` into state. `validate_proof_block_hash` then finds the correct hash and passes.

**The invariant broken:** The gateway is supposed to accept exactly the set of transactions that would succeed during blockifier execution. For proof-bearing invoke transactions with `proof_block_number = N − STORED_BLOCK_HASH_BUFFER + 1`, the gateway rejects what the blockifier would accept.

### Impact Explanation

Any unprivileged user submitting a client-side proving invoke transaction (`RpcInvokeTransactionV3` with non-empty `proof_facts`) that references `proof_block_number = N − STORED_BLOCK_HASH_BUFFER + 1` (the most recent block the protocol allows) receives a spurious `ValidateFailure` / "Block hash mismatch" error from the gateway. The transaction is never admitted to the mempool. This matches: **High — Mempool/gateway/RPC admission rejects valid transactions before sequencing.**

### Likelihood Explanation

The condition is deterministic and predictable. A user who generates a proof against the most recent eligible block (a natural choice to minimize proof staleness) will always hit this rejection. The window is exactly one block wide and repeats every block.

### Recommendation

Before running `validate_proof_facts`, apply the equivalent of `pre_process_block` to the gateway's cached state — i.e., write the hash of block `N − STORED_BLOCK_HASH_BUFFER + 1` into the state before validation. Alternatively, use the actual latest block number N (not N+1) as `current_block_number` for the proof-block-number check, which is consistent with the state the gateway actually holds. The two checks (`validate_proof_block_number` and `validate_proof_block_hash`) must use a consistent view of which block hashes are available.

### Proof of Concept

Let `N = 100`, `STORED_BLOCK_HASH_BUFFER = 10`.

1. User generates a SNOS proof over block 91 (`N − 9`).
2. User submits an invoke V3 transaction with `proof_facts.block_number = 91`.
3. Gateway calls `run_validate_entry_point`:
   - `block_info.block_number = 100.unchecked_next() = 101`
   - `validate_proof_block_number(91, 101)` → `max_allowed = 101 − 10 = 91` → **passes**
   - `validate_proof_block_hash(hash_of_91, 91, os_constants, state_at_block_100)`:
     - `state_at_block_100` has hashes for blocks 0–90 (block 91's hash was stored by `pre_process_block` when block 101 is processed, which hasn't happened yet)
     - `get_storage_at(block_hash_contract, 91)` → `Felt::ZERO`
     - `Felt::ZERO ≠ hash_of_91` → **"Block hash mismatch" error**
4. Gateway returns `ValidateFailure`. Transaction is rejected.
5. After block 101 is committed, the same transaction is accepted (hash of block 91 is now stored).

The transaction was valid and would have executed successfully in block 101, but the gateway incorrectly rejected it.

    fn validate_proof_block_number(
        proof_block_number: u64,
        current_block_number: BlockNumber,
    ) -> TransactionPreValidationResult<()> {
        // Proof block must be old enough to have a stored block hash.
        // Stored block hashes are guaranteed only up to: current - STORED_BLOCK_HASH_BUFFER.
        let max_allowed =
            current_block_number.0.checked_sub(STORED_BLOCK_HASH_BUFFER).ok_or_else(|| {
                TransactionPreValidationError::InvalidProofFacts(format!(
                    "The current block number {current_block_number} is below the required \
                     block-hash retention buffer: {STORED_BLOCK_HASH_BUFFER}."
                ))
            })?;

        if proof_block_number > max_allowed {
            return Err(TransactionPreValidationError::InvalidProofFacts(format!(
                "The proof block number {proof_block_number} is too recent. The maximum allowed \
                 block number is {max_allowed}."
            )));
        }

        Ok(())
    }

    fn validate_proof_block_hash(
        proof_block_hash: Felt,
        proof_block_number: u64,
        os_constants: &OsConstants,
        state: &mut dyn State,
    ) -> TransactionPreValidationResult<()> {
        if proof_block_hash == Felt::ZERO {
            return Err(TransactionPreValidationError::InvalidProofFacts(format!(
                "Proof block hash is zero for block {proof_block_number}."
            )));
        }

        // Compare the proof's block hash with the stored block hash.
        let block_hash_contract_address =
            os_constants.os_contract_addresses.block_hash_contract_address();

        let stored_block_hash = state
            .get_storage_at(block_hash_contract_address, StorageKey::from(proof_block_number))?;

        if stored_block_hash != proof_block_hash {
            return Err(TransactionPreValidationError::InvalidProofFacts(format!(
                "Block hash mismatch for block {proof_block_number}. Proof block hash: \
                 {proof_block_hash}, stored block hash: {stored_block_hash}."
            )));
        }

        Ok(())
    }

pub fn pre_process_block(
    state: &mut dyn State,
    old_block_number_and_hash: Option<BlockHashAndNumber>,
    next_block_number: BlockNumber,
    os_constants: &OsConstants,
) -> StateResult<()> {
    let should_block_hash_be_provided =
        next_block_number >= BlockNumber(constants::STORED_BLOCK_HASH_BUFFER);
    if let Some(BlockHashAndNumber { number, hash }) = old_block_number_and_hash {
        let block_hash_contract_address =
            os_constants.os_contract_addresses.block_hash_contract_address();
        let block_number_as_storage_key = StorageKey::from(number.0);
        state.set_storage_at(block_hash_contract_address, block_number_as_storage_key, hash.0)?;
    } else if should_block_hash_be_provided {
        return Err(StateError::OldBlockHashNotProvided);
    }

    Ok(())
