### Title
Unsigned Integer Underflow in `ComputationRemaining` Due to Dead-Code Guard on `uint64` Comparison - (File: `fvm/meter/computation_meter.go`)

### Summary
`ComputationRemaining` in `fvm/meter/computation_meter.go` performs an unsigned subtraction `computationLimit - computationUsed` without a valid underflow guard. The guard `if remainingComputationUsage <= 0` is dead code for `uint64` (a `uint64` is never `<= 0`), so when `computationUsed > computationLimit` the subtraction wraps around to a near-`MaxUint64` value and the function returns a huge incorrect result instead of `0`. This is the direct Go analog of the Tracer `18 - tokenDecimals` underflow: a scaling/remaining-capacity computation assumes a numeric invariant (`computationUsed ≤ computationLimit`) that is not enforced at the call site, and the guard intended to catch the violation is silently inert.

### Finding Description

`ComputationRemaining` is defined at `fvm/meter/computation_meter.go` lines 122–135:

```go
func (m *ComputationMeter) ComputationRemaining(kind common.ComputationKind) uint64 {
    w, ok := m.params.computationWeights[kind]
    if !ok || w == 0 {
        return math.MaxUint64
    }

    remainingComputationUsage := m.params.computationLimit - m.computationUsed  // ← uint64 subtraction
    if remainingComputationUsage <= 0 {   // ← DEAD CODE: uint64 is never < 0
        return 0
    }

    return remainingComputationUsage / w
}
```

`remainingComputationUsage` is `uint64`. The comparison `<= 0` is always `false` for any `uint64` value; the Go compiler does not warn about this. The guard is therefore never taken.

There are two reachable paths that make `computationUsed > computationLimit`:

**Path 1 – `MeterComputation` (lines 100–104):**
```go
m.computationUsed += w * intensity          // updated first
if m.computationUsed > m.params.computationLimit {
    return errors.NewComputationLimitExceededError(...)
}
```
The accumulator is incremented *before* the limit check. If the error is caught and execution continues in a deferred path or the EVM handler is invoked before the error propagates, `computationUsed` is already above `computationLimit`.

**Path 2 – `Merge` (lines 147–152):**
```go
func (m *ComputationMeter) Merge(child ComputationMeter) {
    m.computationUsed = m.computationUsed + child.computationUsed
    ...
}
```
`Merge` performs no limit check and returns no error. After merging a child whose usage pushes the total above the limit, `computationUsed > computationLimit` silently holds. Any subsequent call to `ComputationRemaining` will underflow.

When the underflow occurs, `remainingComputationUsage` wraps to approximately `MaxUint64 - (computationUsed - computationLimit)`, and the function returns `(MaxUint64 - delta) / w` — a value orders of magnitude larger than the actual remaining budget.

### Impact Explanation

`ComputationRemaining` is part of the `environment.Meter` interface and is called by the EVM handler (`fvm/evm/handler/handler.go`) and the EVM backend (`fvm/evm/backends/wrappedEnv.go`) to determine how much gas to allocate for EVM sub-execution. If the function returns a near-`MaxUint64` value when the computation budget is already exhausted, the EVM layer is told it has essentially unlimited gas remaining. This allows a transaction that has already consumed its full Cadence computation budget to receive an unbounded EVM gas allocation, effectively bypassing the per-transaction computation limit. The result is unauthorized execution of EVM computation beyond the protocol-enforced cap, which is a financial/accounting invariant violation and a resource-exhaustion bypass.

### Likelihood Explanation

Any unprivileged transaction sender can trigger this. The sender submits a transaction that:
1. Executes enough Cadence operations to push `computationUsed` above `computationLimit` (via `MeterComputation` or via a `Merge` of nested transaction states).
2. Then invokes EVM execution (e.g., `EVM.run`, `EVM.batchRun`, or a COA call), which calls `ComputationRemaining` to size the gas grant.

The `Merge` path is particularly reliable because it produces no error and leaves the meter in the underflowing state silently. No privileged keys, staked nodes, or social engineering are required.

### Recommendation

Replace the dead `uint64 <= 0` guard with a proper underflow check before the subtraction:

```go
func (m *ComputationMeter) ComputationRemaining(kind common.ComputationKind) uint64 {
    w, ok := m.params.computationWeights[kind]
    if !ok || w == 0 {
        return math.MaxUint64
    }

    if m.computationUsed >= m.params.computationLimit {
        return 0
    }

    remainingComputationUsage := m.params.computationLimit - m.computationUsed
    return remainingComputationUsage / w
}
```

Additionally, `Merge` should either return an error or clamp `computationUsed` to `computationLimit` when the merge would exceed the limit, so callers are not silently left with an over-limit meter.

### Proof of Concept

```go
// Demonstrate the dead-code guard and underflow in ComputationRemaining.
params := meter.DefaultComputationMeterParameters()
// Set a small limit (in internal precision units)
params.computationLimit = 10 << meter.MeterExecutionInternalPrecisionBytes
params.computationWeights = meter.ExecutionEffortWeights{
    0: 1 << meter.MeterExecutionInternalPrecisionBytes,
}
m := meter.NewComputationMeter(params)

// Merge a child whose usage exceeds the limit — no error returned.
child := meter.NewComputationMeter(params)
child.computationUsed = 20 << meter.MeterExecutionInternalPrecisionBytes // exceeds limit

m.Merge(child)
// m.computationUsed (20<<16) > m.params.computationLimit (10<<16)

remaining := m.ComputationRemaining(0)
// Expected: 0
// Actual:   (MaxUint64 - (20<<16 - 10<<16)) / (1<<16)
//         ≈ MaxUint64 / 65536  ≈ 281_474_976_710_655  (a near-MaxUint64 / w value)
fmt.Println(remaining) // prints a huge number, not 0
```

The root cause is identical in structure to the Tracer finding: a subtraction `A - B` where `B > A` is possible, the unsigned type makes the result wrap to a large positive number, and the guard intended to catch the case (`<= 0`) is inert on an unsigned type.
