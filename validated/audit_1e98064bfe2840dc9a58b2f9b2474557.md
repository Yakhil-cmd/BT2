### Title
Pool Admin Can Set Uncapped Bin Additional Fees, Bypassing the Factory Fee Cap System — (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

The factory enforces explicit caps on admin spread and notional fees via `maxAdminSpreadFeeE6` and `maxAdminNotionalFeeE8`, but `setPoolBinAdditionalFees` passes `addFeeBuyE6` / `addFeeSellE6` directly to the pool with **no cap check**. A pool admin can set per-bin additional fees to any `uint16` value (0–65 535, i.e., 0–6.5535 %) on every bin, silently charging traders far above the intended fee ceiling.

---

### Finding Description

The factory's `setPoolAdminFees` enforces caps before writing fees:

```solidity
// MetricOmmPoolFactory.sol L414-415
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (newAdminNotionalFeeE8 > maxAdminNotionalFeeE8) revert AdminFeeTooHigh();
``` [1](#0-0) 

By contrast, `setPoolBinAdditionalFees` performs **no cap check** and forwards the caller-supplied values verbatim:

```solidity
// MetricOmmPoolFactory.sol L450-457
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external
    override
    nonReentrant
    onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
``` [2](#0-1) 

The pool's `setBinAdditionalFees` only validates the bin index, not the fee magnitudes:

```solidity
// MetricOmmPool.sol L464-474
function setBinAdditionalFees(int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external
    onlyFactory
    nonReentrant(PoolActions.SET_BIN_ADDITIONAL_FEES)
{
    if (bin < LOWEST_BIN || bin > HIGHEST_BIN) revert InvalidBinIndex(bin);
    BinState storage s = _binStates[bin];
    s.addFeeBuyE6 = addFeeBuyE6;
    s.addFeeSellE6 = addFeeSellE6;
    emit BinAdditionalFeesUpdated(bin, addFeeBuyE6, addFeeSellE6);
}
``` [3](#0-2) 

These uncapped values feed directly into every swap's effective fee:

```solidity
// MetricOmmPool.sol L910 (buy path, analogous sell path at L999)
params.baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6)
``` [4](#0-3) 

`uint16` max is 65 535, representing **6.5535 %** additional fee per bin. Applied to every bin the pool admin controls, this stacks on top of the already-capped spread fee (max 20 %), producing an effective fee the cap system was designed to prevent.

---

### Impact Explanation

Traders executing swaps through any bin whose `addFeeBuyE6` / `addFeeSellE6` has been set to the `uint16` maximum pay up to **6.5535 % extra** per swap, directly reducing the tokens they receive. Because the pool admin can update all bins independently and repeatedly, the aggregate extraction is unbounded within a single pool's lifetime. This is a direct loss of user principal with no recourse once the swap settles.

---

### Likelihood Explanation

The pool admin is explicitly semi-trusted: the factory's cap system exists precisely to bound what a pool admin can extract. Any pool admin — whether malicious from the start or later compromised — can invoke `setPoolBinAdditionalFees` at any time with no timelock and no factory-level guard. The call requires only the `onlyPoolAdmin` modifier, which is a normal operational role, not a privileged protocol-owner role.

---

### Recommendation

Add a hard cap check in `setPoolBinAdditionalFees` mirroring the pattern used for spread and notional fees. Define a constant (e.g., `HARD_MAX_BIN_ADD_FEE_E6`) and a factory-owner-settable `maxAdminBinAddFeeE6`, then enforce it:

```solidity
if (addFeeBuyE6 > maxAdminBinAddFeeE6) revert AdminFeeTooHigh();
if (addFeeSellE6 > maxAdminBinAddFeeE6) revert AdminFeeTooHigh();
```

Alternatively, reuse `maxAdminSpreadFeeE6` as the ceiling for bin additional fees, since both are spread-type charges expressed in E6 units.

---

### Proof of Concept

1. Pool admin calls `setPoolBinAdditionalFees(pool, 0, 65535, 65535)`.
2. Factory forwards the values to `pool.setBinAdditionalFees(0, 65535, 65535)` with no cap check. [5](#0-4) 
3. Pool stores `addFeeBuyE6 = 65535`, `addFeeSellE6 = 65535` for bin 0. [6](#0-5) 
4. Any trader swapping through bin 0 has their effective fee computed as `baseFeeX64 + mulDiv(65535, ONE_X64, 1e6)`, adding 6.5535 % on top of the spread fee. [4](#0-3) 
5. The trader receives fewer output tokens than the factory's cap system permits; the excess accrues as LP/spread surplus claimable by the pool admin via `collectFees`. [7](#0-6)
