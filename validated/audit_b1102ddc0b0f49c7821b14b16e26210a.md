### Title
Staker Can Instantly Increase Commission to Maximum Without Advance Notice to Delegators - (`src/staking/staking.cairo`)

### Summary
A staker can call `set_commission_commitment` and `set_commission` in back-to-back transactions within the same epoch to raise their commission from 0% to 100% with zero advance notice. Delegators have no time to exit before the new rate applies to their accruing rewards, and they remain locked in the pool for the full `exit_wait_window` duration while the elevated commission drains their yield.

### Finding Description
The `set_commission_commitment` function enforces only that `expiration_epoch > current_epoch`, meaning the minimum valid commitment window is a single epoch boundary. Because the commitment becomes active immediately upon creation, a staker can call `set_commission` in the very next transaction (or even the same block) to raise their commission up to `max_commission`.

The internal `update_commission` function at `src/staking/staking.cairo` lines 1573–1608 shows the full logic:

```
if commitment is active:
    assert commission <= commitment.max_commission   // allows increase
    assert commission != old_commission
else if commitment expired:
    assert commission < old_commission               // only decrease
else (no commitment):
    assert commission < old_commission               // only decrease
```

The only path to increase commission is through an active commitment. But there is no enforced delay between setting the commitment and exercising it. The codebase itself acknowledges this gap with the developer note at line 745–746:

> **Note**: Current commission increase safeguards still allow for sudden commission changes.

The `set_commission_commitment` validation at lines 771–776 only checks:
- `expiration_epoch > current_epoch` (minimum: `current_epoch + 1`)
- `expiration_epoch - current_epoch <= epochs_in_year()`

Neither check prevents the commitment from being used immediately in the same epoch it was created.

### Impact Explanation
A malicious staker can atomically (within one epoch, potentially one block):
1. Call `set_commission_commitment(max_commission: 10000, expiration_epoch: current_epoch + 1)`.
2. Call `set_commission(commission: 10000)`.

All delegators in the pool are now subject to 100% commission. Because `exit_wait_window` is a mandatory delay between `exit_intent` and `exit_action`, delegators cannot recover their principal immediately. Every attestation reward earned during the exit window is fully captured by the staker. This constitutes **theft of unclaimed yield** from delegators — a High-severity impact under the allowed scope.

### Likelihood Explanation
Any staker who has opened a delegation pool can execute this attack. No privileged role, leaked key, or external dependency is required. The staker is an unprivileged participant who controls their own pool. The attack is cheap (two transactions), reversible only by the staker, and leaves delegators with no on-chain recourse during the exit window. Likelihood is **High** given the low cost and direct financial incentive.

### Recommendation
Enforce a minimum delay between the creation of a commission commitment and the first time it can be used to increase commission. For example, require that `expiration_epoch - current_epoch >= MIN_COMMITMENT_NOTICE_EPOCHS` where `MIN_COMMITMENT_NOTICE_EPOCHS` is at least one full epoch (or a configurable governance parameter). This mirrors the fix applied in RocketPool: changing the vote delay from 1 block to one week so that affected parties have time to react before a change takes effect.

Additionally, consider emitting a `CommissionIncreaseScheduled` event at commitment creation time and enforcing that the commission increase only takes effect starting from `expiration_epoch - 1` (i.e., the commitment signals the future rate, not an immediate one).

### Proof of Concept

```
// Epoch N, block B
staker calls: set_commission_commitment(max_commission: 10000, expiration_epoch: N + 1)
// CommissionCommitmentSet event emitted — delegators have no time to react

// Epoch N, block B+1 (same epoch, next block)
staker calls: set_commission(commission: 10000)
// update_commission: commitment is active (current_epoch N < expiration_epoch N+1)
// assert 10000 <= 10000  ✓
// assert 10000 != old_commission (e.g. 500)  ✓
// Commission is now 100%

// Delegator calls exit_intent — must wait exit_wait_window
// All rewards accrued during exit_wait_window go to staker at 100% commission
// Delegator's unclaimed yield is stolen
```
