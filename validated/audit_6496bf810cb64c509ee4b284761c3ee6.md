### Title
Signature Verification Bypass via `skip_stateful_validations` Allows Invalid-Signature Invoke Transactions into Mempool — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator unconditionally skips the `__validate__` entry point (the only place where signature validity is checked at admission) for any invoke transaction with `nonce == 1` when the sender's address appears in the mempool state. An attacker who observes a pending `deploy_account` transaction in the mempool for address A can submit an invoke transaction for A with a completely invalid signature and have it admitted to the mempool without any signature check.

---

### Finding Description

`skip_stateful_validations` returns `true` (skip validation) when all four conditions hold:

1. The transaction is an `Invoke`
2. `tx.nonce() == Nonce(Felt::ONE)`
3. `account_nonce == Nonce(Felt::ZERO)` (account not yet deployed on-chain)
4. `account_tx_in_pool_or_recent_block(sender_address)` returns `true` [1](#0-0) 

When `skip_validate = true`, `run_validate_entry_point` is called with `validate: !skip_validate = false`: [2](#0-1) 

This means `StatefulValidator::perform_validations` returns early before calling `validate_tx` (the `__validate__` entry point): [3](#0-2) 

The `account_tx_in_pool_or_recent_block` check is a point-in-time snapshot of the mempool state — it returns `true` if the address appears in `MempoolState.committed`, `MempoolState.staged`, or `tx_pool`: [4](#0-3) 

`MempoolState.contains_account` checks the `staged` and `committed` maps: [5](#0-4) 

The `committed` map is populated by `commit_block` and retained for `committed_nonce_retention_block_count` blocks. The `tx_pool.contains_account` check is true whenever any transaction for that address is in the pool.

The full gateway admission flow is: [6](#0-5) 

The `validate_by_mempool` call (which precedes `skip_stateful_validations`) only checks nonce validity — it does not verify the signature: [7](#0-6) 

The stateless validator checks only signature **length**, not validity: [8](#0-7) 

Therefore, when `skip_validate = true`, **no component in the gateway pipeline verifies the cryptographic signature** of the invoke transaction.

---

### Impact Explanation

An attacker can submit an invoke transaction with an arbitrary (invalid) signature for any address that has a pending `deploy_account` transaction in the mempool. The gateway admits this transaction without signature verification. The transaction occupies mempool space, is returned to the batcher via `get_txs`, and consumes batcher execution resources before failing at the `__validate__` entry point during block execution.

This matches the **High** impact: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

The broken invariant is: *every transaction admitted to the mempool must have passed signature verification*. The `skip_stateful_validations` mechanism breaks this invariant for invoke transactions with `nonce == 1` targeting undeployed accounts.

---

### Likelihood Explanation

The condition is easy to trigger. In a P2P network, pending `deploy_account` transactions are publicly observable. Any attacker can monitor the mempool for new `deploy_account` transactions and immediately submit a corresponding invalid-signature invoke transaction with `nonce == 1`. The window is open from the moment the `deploy_account` tx enters the mempool until the account is deployed on-chain (nonce advances to 1 in state). This is the direct analog to the ERC20 approve race: the "authorization" (signature check) is bypassed based on a stale, race-prone mempool state snapshot.

---

### Recommendation

1. **Do not skip signature verification entirely.** The `deploy_account` transaction in the mempool carries the `class_hash` of the account contract to be deployed. The gateway can use this to instantiate the account class and call `__validate__` against the not-yet-deployed account's expected logic (using the class manager), rather than skipping it.
2. **Alternatively**, require that the invoke transaction's signature be pre-verified against the class declared in the pending `deploy_account` transaction before granting the skip.
3. **At minimum**, document that the skip-validate path admits unverified signatures and add a mempool-level rate limit per address for transactions admitted via this path.

---

### Proof of Concept

1. Alice submits `deploy_account_tx` for address `A` (class_hash `C`, salt `S`). The tx enters the mempool pool.
2. Attacker observes the pending `deploy_account_tx` for address `A` via P2P gossip.
3. Attacker submits `invoke_tx(sender_address=A, nonce=1, signature=[0xdead, 0xbeef], calldata=[...], resource_bounds=valid)`.
4. Gateway stateless check: signature length ≤ `max_signature_length` ✓ (length 2 is valid).
5. Gateway stateful check:
   - `account_nonce = get_nonce_from_state(A) = 0` (account not deployed) ✓
   - `validate_nonce`: `0 ≤ 1 ≤ 0 + max_allowed_nonce_gap` ✓
   - `validate_by_mempool`: nonce not too old ✓
   - `skip_stateful_validations`: `nonce==1` ✓, `account_nonce==0` ✓, `account_tx_in_pool_or_recent_block(A)` = `true` (Alice's deploy_account is in pool) ✓ → returns `true`
   - `run_validate_entry_point(skip_validate=true)`: `validate=false` → `__validate__` NOT called ✓
6. Attacker's invoke tx is admitted to the mempool **without signature verification**.
7. Batcher calls `get_txs`, receives both Alice's `deploy_account` and attacker's `invoke`.
8. Batcher executes `deploy_account` → account A is deployed.
9. Batcher executes attacker's `invoke` → `__validate__` is called → signature `[0xdead, 0xbeef]` fails → tx reverted.
10. **Result**: Invalid-signature transaction was admitted to the mempool and consumed batcher execution resources. [9](#0-8) [10](#0-9)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
