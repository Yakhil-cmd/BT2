### Title
Unauthorized Burn and Transfer of `evm/<contract>` Native Tokens via Missing Caller Authorization in Bank Precompile — (File: x/cronos/keeper/precompiles/bank.go)

---

### Summary
The bank precompile's `burn` and `transfer` methods accept an arbitrary address as the source of the operation but perform no check that the EVM caller is authorized to act on behalf of that address. Any unprivileged contract can burn or transfer `evm/<callerContract>` native tokens from any holder without their consent.

---

### Finding Description

In `x/cronos/keeper/precompiles/bank.go`, the `BankContract.Run` method handles both `burn` and `transfer` calls.

**`burn` path** (lines 113–156):

```go
recipient := args[0].(common.Address)   // misleadingly named; this is the address burned FROM
addr := sdk.AccAddress(recipient.Bytes())
// ← NO check: addr == contract.Caller()
bc.bankKeeper.SendCoinsFromAccountToModule(ctx, addr, types.ModuleName, sdk.NewCoins(amt))
bc.bankKeeper.BurnCoins(ctx, types.ModuleName, sdk.NewCoins(amt))
``` [1](#0-0) 

**`transfer` path** (lines 167–200):

```go
sender := args[0].(common.Address)
recipient := args[1].(common.Address)
from := sdk.AccAddress(sender.Bytes())
to   := sdk.AccAddress(recipient.Bytes())
// ← NO check: from == contract.Caller()
bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt))
``` [2](#0-1) 

The denom used in every call is `EVMDenom(contract.Caller())` = `"evm/<callerContract>"`. [3](#0-2) 

Because the denom is scoped to the calling contract, any contract M can:

1. Call `bank.burn(victim, amount)` → burns `evm/M` tokens from `victim` without their consent.
2. Call `bank.transfer(victim, attacker, amount)` → moves `evm/M` tokens from `victim` to `attacker` without their consent.

The only guard present is `checkBlockedAddr`, which only rejects Cosmos module accounts — it does not verify that the caller is the token holder. [4](#0-3) 

The integration test `TestBank.sol` shows the intended pattern is `bank.burn(msg.sender, amount)` — the user burns their own tokens — but the precompile never enforces this invariant. [5](#0-4) 

---

### Impact Explanation

**Critical — Unauthorized burn and transfer of precompile-controlled assets.**

Any unprivileged contract can destroy or steal `evm/<callerContract>` native bank tokens held by any address. `SendCoinsFromAccountToModule` and `SendCoins` in the Cosmos SDK bank keeper execute unconditionally; no EVM-level approval mechanism exists to block the call. This directly satisfies the critical impact criterion: *unauthorized burn, transfer, or balance/accounting change for precompile-controlled assets*.

---

### Likelihood Explanation

**Medium.** The attacker must first get victims to hold `evm/M` tokens. This is achievable via:
- Airdropping `evm/M` tokens to victims using `bank.mint(victim, amount)`.
- Listing `evm/M` on a DEX so users acquire them through normal trading.
- Deploying a legitimate-looking DeFi contract that issues `evm/M` as a reward token.

Once victims hold the tokens (e.g., as collateral, LP shares, or rewards), the attacker can drain them in a single transaction.

---

### Recommendation

Add an authorization check in both `burn` and `transfer` to ensure the EVM caller (`contract.Caller()`) is the token holder or has been explicitly approved:

- **`burn`**: require `args[0].(common.Address) == contract.Caller()`, or implement an allowance mechanism.
- **`transfer`**: require `args[0].(common.Address) == contract.Caller()`, or implement an ERC-20-style `approve`/`transferFrom` pattern at the precompile level.

---

### Proof of Concept

**Unauthorized burn (rug-pull):**

1. Attacker deploys contract M.
2. M calls `bank.mint(victim, 1000)` — victim now holds 1000 `evm/M` tokens.
3. Victim deposits `evm/M` tokens as collateral in a DeFi protocol.
4. Attacker calls `bank.burn(victim, 1000)` — victim's collateral is destroyed without any approval from victim.
5. Victim is liquidated; attacker suffers no cost beyond gas.

**Unauthorized transfer (theft):**

1. Attacker deploys contract M and lists `evm/M` on a DEX.
2. Users buy `evm/M` tokens.
3. Attacker calls `bank.transfer(user, attacker, balance)` for each holder.
4. All `evm/M` balances are drained to the attacker's address with no user approval required.

func (bc *BankContract) checkBlockedAddr(addr sdk.AccAddress) error {
	to, err := sdk.AccAddressFromBech32(addr.String())

		return err
	}
	if bc.bankKeeper.BlockedAddr(to) {
		return errorsmod.Wrapf(errortypes.ErrUnauthorized, "%s is not allowed to receive funds", to.String())
	}
	return nil
}
