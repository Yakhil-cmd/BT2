### Title
Missing Minimum-Threshold Enforcement in FROST Signing Validation Bypasses DKG-Level Invariant — (`src/frost/mod.rs`)

---

### Summary

The DKG layer enforces a hard minimum of `threshold >= 2` on every key-generation and reshare call. The FROST signing validation function `assert_sign_inputs` omits this same lower-bound check, allowing a malicious coordinator to invoke signing with `threshold = 1` (or `threshold = 0`). This bypasses the invariant established at key-generation time, corrupts the Lagrange interpolation used to combine signature shares, and causes honest parties to produce unusable cryptographic outputs.

---

### Finding Description

`assert_key_invariants` in `src/dkg.rs` enforces two threshold bounds before any key material is created:

```rust
// src/dkg.rs lines 572-582
if threshold > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
if threshold < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
}
``` [1](#0-0) 

The public FROST signing validation function `assert_sign_inputs` in `src/frost/mod.rs` checks only the upper bound — that `threshold <= participants.len()` — and never checks the lower bound:

```rust
// src/frost/mod.rs lines 144-150
// validate threshold
if threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge {
        threshold: threshold.value(),
        max: participants.len(),
    });
}
``` [2](#0-1) 

There is no corresponding `threshold < 2` guard. A caller supplying `threshold = 1` (or `threshold = 0`) passes all validation and proceeds into the signing protocol.

The OT-based ECDSA `sign` function in `src/ecdsa/ot_based_ecdsa/sign.rs` has the same gap — it checks `participants.len() >= threshold` but never `threshold >= 2`:

```rust
// src/ecdsa/ot_based_ecdsa/sign.rs lines 57-63
if participants.len() < threshold {
    return Err(InitializationError::NotEnoughParticipantsForThreshold { ... });
}
``` [3](#0-2) 

---

### Impact Explanation

In FROST (and in the OT-based ECDSA signing path), the `threshold` value is used to compute Lagrange interpolation coefficients (`lambda_i`) that weight each participant's signature share before aggregation. When `threshold = 1`, the Lagrange basis polynomial collapses: every participant's coefficient becomes `1` regardless of the actual key-share polynomial degree used during DKG (which was `>= 2`). The combined output is therefore cryptographically inconsistent with the public key established during DKG — honest parties complete the protocol and accept a signature share that cannot aggregate into a valid signature. This matches:

> **High: Corruption of sign outputs so honest parties accept unusable cryptographic outputs.**

---

### Likelihood Explanation

The `threshold` parameter is fully caller-controlled at the signing API boundary. A malicious coordinator — a documented attacker profile in `RESEARCHER.md` — can supply any value. No out-of-band mechanism prevents `threshold = 1` from reaching `assert_sign_inputs`. The missing check is a single missing `if threshold < 2` guard, identical in form to the guard already present in `assert_key_invariants`. [4](#0-3) 

---

### Recommendation

Add the same lower-bound guard that exists in `assert_key_invariants` to `assert_sign_inputs` (and to the OT-based ECDSA `sign` entry point):

```rust
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

This should be inserted in `assert_sign_inputs` immediately after the upper-bound check, mirroring the pattern in `assert_key_invariants`. [5](#0-4) [6](#0-5) 

---

### Proof of Concept

1. Run DKG with 3 participants and `threshold = 2` (passes `assert_key_invariants`).
2. Call the FROST `sign_v1` or `sign_v2` entry point with the same 3 participants but `threshold = 1`.
3. `assert_sign_inputs` accepts the call — no `ThresholdTooSmall` error is returned.
4. The signing protocol proceeds; each participant computes their Lagrange coefficient as if only 1 share is needed for reconstruction.
5. The aggregated signature fails verification against the public key (which was generated under `threshold = 2` Lagrange weights), producing an unusable output that honest parties have accepted as a completed protocol run.

The root cause is the absent lower-bound check at `src/frost/mod.rs` lines 144–150, directly analogous to the absent debt-ceiling check in `HToken.depositUnderlying()` from the reference report — a secondary entry point that achieves the same cryptographic effect (combining shares) without the same invariant enforcement present in the primary entry point (DKG).

pub fn assert_sign_inputs(
    participants: &[Participant],
    threshold: impl Into<ReconstructionLowerBound>,
    me: Participant,
    coordinator: Participant,
) -> Result<ParticipantList, InitializationError> {
    let threshold = threshold.into();
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // validate threshold
    if threshold.value() > participants.len() {
        return Err(InitializationError::ThresholdTooLarge {
            threshold: threshold.value(),
            max: participants.len(),
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }
    Ok(participants)
}
```
