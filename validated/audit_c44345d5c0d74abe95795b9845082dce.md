### Title
COA Smart Contract Wallet Bypasses `tx.origin == msg.sender` EOA Guard in Flow EVM - (File: `fvm/evm/emulator/emulator.go`, `fvm/evm/types/call.go`)

---

### Summary

Flow EVM's Cadence-Owned Accounts (COAs) are EVM **smart contract wallets** (they have deployed bytecode at their address). When a COA executes a `DirectCall` into an EVM contract, the runtime unconditionally sets `tx.origin = msg.From` (the COA's own address). Because `msg.sender` is also the COA's address for a top-level call, the invariant `tx.origin == msg.sender` holds — even though the caller is a smart contract. Any EVM contract deployed on Flow EVM that uses `require(tx.origin == msg.sender)` as an EOA-only guard is therefore bypassable by any unprivileged user who owns a COA.

---

### Finding Description

Flow EVM supports three account types: EOAs, smart contract accounts, and COAs. COAs are explicitly described as smart contract wallets:

> "COAs are smart contract wallets that accept native token transfers and support several ERCs including ERC-165, ERC-721, ERC-777, ERC-1155, ERC-1271." [1](#0-0) 

When a COA calls an EVM contract via `DirectCall`, the `Message()` function in `call.go` constructs the geth `core.Message` with `SkipTransactionChecks: true`. The comment explicitly states this is required because a COA has non-empty code and would otherwise fail the EOA check: [2](#0-1) 

The emulator's `run()` function then sets `tx.origin` to `msg.From` — the COA's own EVM address: [3](#0-2) 

For a top-level COA `DirectCall`, `msg.sender == COA address` and `tx.origin == COA address`. The check `tx.origin == msg.sender` evaluates to `true`, even though the COA is a deployed smart contract. The standard Solidity EOA guard:

```solidity
modifier onlyEOA() {
    require(tx.origin == msg.sender, "no contracts");
    _;
}
```

is silently bypassed.

---

### Impact Explanation

Any EVM contract deployed on Flow EVM that relies on `tx.origin == msg.sender` to restrict sensitive operations to EOA callers (e.g., preventing flash-loan attacks, reentrancy guards, minting restrictions, or governance vote manipulation) will accept calls from COA smart contract wallets. The COA owner can execute arbitrary call sequences within a single Cadence transaction, enabling atomic multi-step interactions that the EOA guard was designed to prevent.

---

### Likelihood Explanation

COA creation is permissionless — any Flow account can call `EVM.createCadenceOwnedAccount()` and immediately obtain a COA. The `DirectCall` path through `ContractHandler.Call()` is the standard mechanism for COA-to-EVM interaction and is exercised in normal protocol operation. No special privileges, staked nodes, or compromised keys are required. [4](#0-3) 

---

### Recommendation

Flow EVM should expose a mechanism that allows EVM contracts to distinguish COA callers from true EOAs. Options include:

1. **Expose `tx.type` to EVM contracts**: Since COA `DirectCall`s use `DirectCallTxType = 0xff`, a precompile or opcode extension could let contracts query the transaction type and reject `0xff` callers.
2. **Document the invariant break**: Clearly document in the EVM contract developer guide that `tx.origin == msg.sender` does **not** guarantee an EOA caller on Flow EVM, and recommend using `EXTCODESIZE(msg.sender) == 0` as a supplementary check (with the caveat that COAs will still have non-zero code size, making this the correct guard).
3. **Set `tx.origin` to a sentinel value for COA calls**: Rather than setting `tx.origin = COA address`, set it to a well-known sentinel (e.g., `address(0)`) so that `tx.origin == msg.sender` is always `false` for COA calls. [5](#0-4) 

---

### Proof of Concept

**Attacker-controlled entry path:**

1. Attacker deploys a target EVM contract on Flow EVM with an `onlyEOA` modifier:
   ```solidity
   modifier onlyEOA() {
       require(tx.origin == msg.sender, "no contracts");
       _;
   }
   function privilegedAction() external onlyEOA { ... }
   ```

2. Attacker creates a COA via a Cadence transaction:
   ```cadence
   import EVM from <EVM_address>
   transaction {
       prepare(acct: auth(SaveValue, Capabilities) &Account) {
           let coa <- EVM.createCadenceOwnedAccount()
           acct.storage.save(<-coa, to: /storage/coa)
       }
   }
   ```

3. Attacker calls `privilegedAction()` through the COA:
   ```cadence
   import EVM from <EVM_address>
   transaction {
       prepare(acct: auth(Storage) &Account) {
           let coa = acct.storage.borrow<auth(EVM.Call) &EVM.CadenceOwnedAccount>(
               from: /storage/coa)!
           let callData = /* ABI-encoded privilegedAction() */
           let result = coa.call(
               to: EVM.EVMAddress(bytes: targetAddress),
               data: callData,
               gasLimit: 100_000,
               value: EVM.Balance(attoflow: 0)
           )
           assert(result.status == EVM.Status.successful)
       }
   }
   ```

4. Inside the EVM execution, `proc.evm.TxContext.Origin` is set to the COA's EVM address (line 671 of `emulator.go`), and `msg.sender` is also the COA's EVM address. The check `tx.origin == msg.sender` passes, and `privilegedAction()` executes despite the caller being a smart contract. [3](#0-2) [6](#0-5) [7](#0-6)
