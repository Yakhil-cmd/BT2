### Title
FROST `PresignOutput` Derives `Clone`, Enabling Nonce Reuse and Private Key Extraction - (File: `src/frost/mod.rs`)

---

### Summary

The `PresignOutput<C>` struct in the FROST module derives `Clone`, which undermines the library's attempt to enforce single-use nonce semantics through Rust's ownership system. A library caller can clone a `PresignOutput` before passing it to `sign_v2`, allowing the same secret nonces to be used across multiple signing sessions with different messages. In FROST/EdDSA, nonce reuse across two signatures directly enables full private signing share recovery.

---

### Finding Description

The `PresignOutput<C>` struct is defined at `src/frost/mod.rs` and explicitly derives `Clone`:

```rust
#[derive(Debug, Clone, Serialize, Deserialize, Eq, PartialEq)]
pub struct PresignOutput<C: Ciphersuite + Send + 'static> {
    pub nonces: SigningNonces<C>,
    pub commitments_map: BTreeMap<Identifier<C>, SigningCommitments<C>>,
}
```

The `nonces: SigningNonces<C>` field holds the secret, one-time-use nonce material generated during the presign phase. [1](#0-0) 

The library's `sign_v2` function correctly takes `presignature: PresignOutput` **by value** (moved), which is the idiomatic Rust mechanism to enforce single-use semantics at the type level:

```rust
pub fn sign_v2(
    ...
    presignature: PresignOutput,
    message: Vec<u8>,
) -> Result<impl Protocol<Output = SignatureOption>, InitializationError>
``` [2](#0-1) 

However, because `PresignOutput` derives `Clone`, any library caller can trivially bypass the ownership-based single-use guarantee:

```rust
let presign_output = run_presign(...);
let presign_clone = presign_output.clone(); // bypass ownership enforcement
sign_v2(..., presign_output, message1);     // first use
sign_v2(..., presign_clone, message2);      // second use — same nonces, different message
```

Inside `do_sign_participant_v2`, the nonces are consumed directly in `round2::sign`:

```rust
let signature_share = round2::sign(&signing_package, &presignature.nonces, &key_package)
``` [3](#0-2) 

And in `do_sign_coordinator_v2`, the same nonces are used:

```rust
let signature_share = round2::sign(&signing_package, &presignature.nonces, &key_package)
``` [4](#0-3) 

If the same `nonces` are used to produce two signature shares over different messages `m1` and `m2`, the standard FROST/EdDSA nonce-reuse equation system becomes solvable for the private signing share `x_i`.

The analog to the external report is exact: the Teller contract took `reclaim()` by a privileged caller but never marked the option as consumed, so it could be called repeatedly. Here, `sign_v2` takes `PresignOutput` by value to enforce single-use, but `Clone` allows the caller to bypass this, enabling repeated use of the same nonce material.

---

### Impact Explanation

Nonce reuse in FROST/EdDSA is a classical, fully practical key-recovery attack. Given two valid signature shares `(s1, s2)` produced with the same nonce `r` over messages `m1 ≠ m2`, the private signing share `x_i` can be algebraically recovered. This directly matches the allowed critical impact:

> **Critical: Extraction, reconstruction, or disclosure of private signing shares, aggregate secret material, presign secrets, nonce material, or confidential derived secrets.**

---

### Likelihood Explanation

The attack is reachable by:

1. **A malicious coordinator**: Initiates a `sign_v2` session, collects signature shares, aborts before broadcasting the final signature, then instructs participants to reuse their `PresignOutput` (possible if the application layer stores and clones it). The coordinator now holds two signature shares over different messages with the same nonces and can recover the private key.

2. **An inadvertent library caller**: Application developers may clone `PresignOutput` for logging, serialization, retry logic, or session management — all common patterns — without realizing the catastrophic cryptographic consequence.

The `pub` visibility of both `nonces` and `commitments_map` fields, combined with `Clone` and `Serialize`/`Deserialize`, makes accidental or intentional reuse straightforward. [5](#0-4) 

---

### Recommendation

Remove `Clone` from `PresignOutput`. Rust's move semantics already enforce single-use when `Clone` is absent — this is the correct mechanism for one-time-use cryptographic material.

```rust
// Remove Clone from the derive list
#[derive(Debug, Serialize, Deserialize, Eq, PartialEq)]
pub struct PresignOutput<C: Ciphersuite + Send + 'static> {
    pub nonces: SigningNonces<C>,
    pub commitments_map: BTreeMap<Identifier<C>, SigningCommitments<C>>,
}
```

Additionally, consider wrapping `nonces` in `Zeroizing<_>` to ensure the secret nonce material is cleared from memory after use, preventing extraction from memory dumps.

### Proof of Concept

```rust
// After running presign to get PresignOutput per participant:
let presign_output: PresignOutput<Ed25519Sha512> = run_presign(...);

// Attacker (or buggy application) clones before first use:
let presign_clone = presign_output.clone();

// First signing session — nonces consumed logically but clone still holds them:
sign_v2(participants, threshold, me, coordinator,
        keygen_output.clone(), presign_output, message1.to_vec());

// Second signing session — same nonces, different message:
sign_v2(participants, threshold, me, coordinator,
        keygen_output.clone(), presign_clone, message2.to_vec());

// Coordinator now holds (s1, m1) and (s2, m2) with identical nonces.
// Standard FROST nonce-reuse algebra recovers x_i (private signing share).
