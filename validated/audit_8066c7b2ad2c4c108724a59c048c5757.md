### Title
Presignature Reuse Enables Secret Key Extraction via Unlimited Rerandomization — (`src/ecdsa/robust_ecdsa/mod.rs`, `src/ecdsa/ot_based_ecdsa/mod.rs`)

---

### Summary

Both `robust_ecdsa::PresignOutput` and `ot_based_ecdsa::PresignOutput` derive `Clone` and `Serialize/Deserialize`, and `RerandomizedPresignOutput::rerandomize_presign` accepts a shared reference `&PresignOutput` rather than consuming it. This means a single presignature can be rerandomized and signed against an unlimited number of times. The library documents "each presignature is consumed **exactly once** (one-time use)" but provides no enforcement mechanism. A malicious coordinator can drive honest participants through two signing sessions using the same presignature with different `(msg_hash, tweak)` inputs, yielding two aggregate signatures from which the master secret key is algebraically recoverable.

---

### Finding Description

**Root cause — `PresignOutput` is never consumed:**

`robust_ecdsa::PresignOutput` derives `Clone`:

```rust
#[derive(Debug, Clone, Serialize, Deserialize, PartialEq, Eq, ZeroizeOnDrop)]
pub struct PresignOutput { pub big_r: AffinePoint, pub c: Scalar, pub e: Scalar, pub alpha: Scalar, pub beta: Scalar }
``` [1](#0-0) 

`rerandomize_presign` takes `presignature: &PresignOutput` — a shared reference — so the caller's `PresignOutput` is never moved or invalidated:

```rust
pub fn rerandomize_presign(
    presignature: &PresignOutput,
    args: &RerandomizationArguments,
) -> Result<Self, ProtocolError> {
``` [2](#0-1) 

The same pattern exists for the OT-based scheme: [3](#0-2) [4](#0-3) 

**Secondary amplifier — `e` is not rerandomized:**

In `rerandomize_presign` for the robust scheme, the blinding term `e` is copied verbatim into every `RerandomizedPresignOutput` produced from the same `PresignOutput`:

```rust
Ok(Self {
    big_r: rerandomized_big_r.into(),
    alpha: rerandomized_alpha,
    beta: rerandomized_beta,
    e: presignature.e,   // ← identical across all rerandomizations
})
``` [5](#0-4) 

Each participant's signature share is `s_i = h·α_i + β_i·R_x + e_i`. Because `e_i` is identical across sessions, the coordinator who receives individual `s_i` values from two sessions can subtract them to cancel `e_i`, isolating the nonce-dependent terms per participant.

**No "used" flag, no consumption, no tracking:**

The `Protocol` trait and `ProtocolExecutor` have no concept of presignature identity or usage state. Nothing in the library prevents a participant from calling `rerandomize_presign` on the same `PresignOutput` twice. [6](#0-5) 

The library's own security documentation acknowledges the requirement but leaves enforcement entirely to the caller:

> "Never reuse a presignature, even across failed, aborted, or partially completed signing sessions." [7](#0-6) 

---

### Impact Explanation

**Secret key extraction from two signing sessions.**

Let the shared presignature nonce be `k`, the master secret key be `x`, and the two rerandomization scalars be `δ₁`, `δ₂` (both derived from public inputs and therefore known to the coordinator). The two aggregate signatures satisfy:

```
s¹ = k⁻¹ · δ₁⁻¹ · (h₁ + (x + ε₁) · Rₓ¹)
s² = k⁻¹ · δ₂⁻¹ · (h₂ + (x + ε₂) · Rₓ²)
```

Setting `A = s¹·δ₁`, `B = s²·δ₂`, `a₁ = h₁ + ε₁·Rₓ¹`, `b₁ = Rₓ¹`, `a₂ = h₂ + ε₂·Rₓ²`, `b₂ = Rₓ²`:

```
x = (B·a₁ − A·a₂) / (A·b₂ − B·b₁)
```

This is a closed-form linear solve for the master secret key `x` using only public values and the two signatures. This matches the attack class documented in the codebase:

> "If different subsets of size at least 2t+1 sign different (h, ε) values using shares derived from the same presignature, the resulting signatures use multiplicatively related nonces and the secret key can be recovered using standard ECDSA nonce-reuse attacks." [8](#0-7) 

Impact: **Critical** — full extraction of the aggregate private signing key.

---

### Likelihood Explanation

A malicious coordinator is an explicitly documented attacker profile in `RESEARCHER.md`: [9](#0-8) 

The attack requires only:
1. One completed presigning session (normal protocol operation).
2. Two signing invocations with the same `big_r` but different `(msg_hash, tweak)` — both are coordinator-controlled inputs.

Honest participants have no way to detect that their `PresignOutput` has already been used; the library provides no tracking mechanism. The `sign` function accepts any `RerandomizedPresignOutput` without checking whether the underlying presignature was previously consumed. [10](#0-9) 

---

### Recommendation

1. **Consume `PresignOutput` by value in `rerandomize_presign`**: Change the signature from `presignature: &PresignOutput` to `presignature: PresignOutput`. Rust's ownership system will then statically prevent reuse.

2. **Remove `Clone` and `Serialize` from `PresignOutput`**: Deriving `Clone` and `Serialize` on secret presignature material directly enables copying and storing the value for later reuse. Removing these derives enforces single-use at the type level.

3. **Rerandomize `e` as well**: The `e` field is currently copied verbatim into every `RerandomizedPresignOutput`. It should be rerandomized with `δ` to prevent the coordinator from cancelling it across sessions.

---

### Proof of Concept

```rust
// Attacker-controlled coordinator drives two signing sessions
// from the same PresignOutput held by honest participants.

let presign_out: PresignOutput = /* result of presigning */;

// Session 1: (h1, ε1, entropy1)
let args1 = RerandomizationArguments::new(pk, tweak1, h1, presign_out.big_r, participants, entropy1);
let rerand1 = RerandomizedPresignOutput::rerandomize_presign(&presign_out, &args1).unwrap();
// participants send s_i^1 to coordinator → coordinator collects s^1

// Session 2: (h2, ε2, entropy2) — same presign_out, no error, no detection
let args2 = RerandomizationArguments::new(pk, tweak2, h2, presign_out.big_r, participants, entropy2);
let rerand2 = RerandomizedPresignOutput::rerandomize_presign(&presign_out, &args2).unwrap();
// participants send s_i^2 to coordinator → coordinator collects s^2

// Coordinator now solves for x:
// A = s1 * δ1,  B = s2 * δ2
// x = (B*(h1 + ε1*Rx1) - A*(h2 + ε2*Rx2)) / (A*Rx2 - B*Rx1)
```

The `rerandomize_presign` call on line 2 succeeds without error because `PresignOutput` is `Clone` and the function takes `&PresignOutput`. [11](#0-10) [12](#0-11)

impl RerandomizedPresignOutput {
    pub fn rerandomize_presign(
        presignature: &PresignOutput,
        args: &RerandomizationArguments,
    ) -> Result<Self, ProtocolError> {
        if presignature.big_r != args.big_r {
            return Err(ProtocolError::IncompatibleRerandomizationInputs);
        }
        let delta = args.derive_randomness()?;
        if delta.is_zero().into() {
            return Err(ProtocolError::ZeroScalar);
        }

        // cannot be zero due to the previous check
        let inv_delta = delta.invert().unwrap();

        // delta * R
        let rerandomized_big_r = presignature.big_r * delta;

        // alpha * delta^{-1}
        let rerandomized_alpha = presignature.alpha * inv_delta;

        // (beta + c*tweak) * delta^{-1}
        let rerandomized_beta =
            (presignature.beta + presignature.c * args.tweak.value()) * inv_delta;

        Ok(Self {
            big_r: rerandomized_big_r.into(),
            alpha: rerandomized_alpha,
            beta: rerandomized_beta,
            e: presignature.e,
        })
    }
