### Title
Stale Program Cache Not Invalidated After Non-Service-Account Contract Update - (`module/state_synchronization/indexer/util.go`)

### Summary

The Access Node indexer's `programInvalidator.ShouldInvalidateEntries()` returns only `inv.invalidateAll`, ignoring the populated `inv.invalidated` map. Because `TableTransaction.AddInvalidator` uses `ShouldInvalidateEntries()` as a gate to decide whether to register the invalidator at all, contract updates by non-service-account users are silently dropped from the program cache invalidation chain. Subsequent script executions on the Access Node serve stale compiled programs from before the update.

---

### Finding Description

In `module/state_synchronization/indexer/util.go`, the `programInvalidator` struct has two fields:

```go
type programInvalidator struct {
    invalidateAll bool
    invalidated   map[common.Location]struct{}
}
```

`ShouldInvalidateEntries()` returns only `inv.invalidateAll`:

```go
func (inv *programInvalidator) ShouldInvalidateEntries() bool {
    return inv.invalidateAll   // ŌåÉ ignores inv.invalidated entirely
}
```

But `ShouldInvalidateEntry()` correctly checks both:

```go
func (inv *programInvalidator) ShouldInvalidateEntry(location common.AddressLocation, ...) bool {
    _, ok := inv.invalidated[location]
    return inv.invalidateAll || ok   // ŌåÉ correctly uses inv.invalidated
}
``` [1](#0-0) 

The gate in `TableTransaction.AddInvalidator` (`fvm/storage/derived/table.go`) is:

```go
func (txn *TableTransaction[TKey, TVal]) AddInvalidator(
    invalidator TableInvalidator[TKey, TVal],
) {
    if invalidator == nil || !invalidator.ShouldInvalidateEntries() {
        return   // ŌåÉ invalidator is dropped entirely if ShouldInvalidateEntries() == false
    }
    ...
}
``` [2](#0-1) 

`updateProgramCache` in the indexer constructs the invalidator as:

```go
tx.AddInvalidator(&accessInvalidator{
    programs: &programInvalidator{
        invalidated:   updatedContracts,
        invalidateAll: hasAuthorizedTransaction(collections, c.serviceAddress),
    },
    ...
})
``` [3](#0-2) 

`hasAuthorizedTransaction` returns `true` only when the service account is an authorizer in the block's transactions: [4](#0-3) 

When a regular (non-service-account) user updates a contract, `invalidateAll` is `false` and `invalidated` is non-empty. `ShouldInvalidateEntries()` returns `false`, so `AddInvalidator` drops the invalidator without registering it. The program cache retains the pre-update compiled program indefinitely.

Compare with the FVM's own `ProgramInvalidator` (used during transaction execution), which correctly gates on `ContractUpdates.Any()`:

```go
func (invalidator ProgramInvalidator) ShouldInvalidateEntries() bool {
    return invalidator.ExecutionParametersUpdated ||
        invalidator.ContractUpdates.Any()   // ŌåÉ correct: any update triggers invalidation
}
``` [5](#0-4) 

---

### Impact Explanation

Script executions routed through the Access Node's indexer path (`NewDerivedBlockDataForScript`) will use the stale pre-update compiled program for any contract updated by a non-service-account. If a contract is updated to patch a security bug (e.g., a price oracle, access-control check, or balance computation), scripts querying that contract will continue to execute against the old, vulnerable code. dApps and users relying on script results to make financial decisions (e.g., checking collateral ratios, verifying ownership, reading balances) receive incorrect data, enabling exploitation of the stale state.

---

### Likelihood Explanation

Every contract update by a non-service-account (the vast majority of contract updates on the network) triggers this path. The condition `hasAuthorizedTransaction(collections, c.serviceAddress)` is false for all ordinary user contract deployments and updates. This is a systematic, always-reproducible failure for the common case.

---

### Recommendation

Fix `programInvalidator.ShouldInvalidateEntries()` to return `true` whenever there are specific contracts to invalidate, mirroring the FVM's own invalidator:

```go
func (inv *programInvalidator) ShouldInvalidateEntries() bool {
    return inv.invalidateAll || len(inv.invalidated) > 0
}
``` [6](#0-5) 

---

### Proof of Concept

1. Access Node is running with `derivedChainData` enabled (standard configuration).
2. User Alice deploys contract `Foo` at address `0xALICE`. The program cache is populated with the compiled `Foo`.
3. Alice submits a transaction updating `Foo` (e.g., changing a return value or access-control logic). The block is sealed and indexed.
4. `updateProgramCache` is called. `hasAuthorizedTransaction(collections, serviceAddress)` returns `false` (Alice is not the service account). `programInvalidator{invalidateAll: false, invalidated: {0xALICE.Foo: {}}}` is constructed.
5. `tx.AddInvalidator(...)` ŌåÆ `programInvalidator.ShouldInvalidateEntries()` returns `false` ŌåÆ invalidator is dropped.
6. The program cache still holds the pre-update compiled `Foo`.
7. Any script calling `import Foo from 0xALICE` executes against the old version of `Foo`, returning results inconsistent with the on-chain state.
