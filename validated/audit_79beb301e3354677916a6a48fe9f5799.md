### Title
WalletConnect Dapp Can Silently Override `wallet_id` in `chia_sendTransaction`, Causing User to Approve Spend of Wrong Asset - (File: packages/gui/src/electron/commands/Commands.ts)

### Summary
A WalletConnect dapp with `chia_sendTransaction` permission can supply an arbitrary `wallet_id` parameter that is accepted without validation against the actual wallets belonging to the current fingerprint, and is explicitly hidden from the user's confirmation dialog. The user approves a transaction seeing only the amount (always humanized as XCH) and destination address, while the actual asset spent is determined by the dapp-controlled `wallet_id`. This is a direct analog to the Across V2 `repaymentChainId` bug: an unvalidated routing identifier that determines which asset/wallet is used, accepted without checking it against a set of valid/expected values.

### Finding Description

In `Commands.ts`, the `chia_wallet.send_transaction` command schema defines `wallet_id` as a parameter with `hide: true` and a dapp-level default of `wallet_id: 1`: [1](#0-0) 

The `hide: true` flag causes `humanizeParams` to filter `wallet_id` out of the confirmation dialog entirely: [2](#0-1) 

The `parseDappParams` function validates that `wallet_id` is a valid number, but performs no check that the value corresponds to a wallet actually owned by the current fingerprint: [3](#0-2) 

The test suite confirms a dapp can override `wallet_id` from the default of `1` to any other value and it passes through: [4](#0-3) 

The `dispatchPairRequest` authorization layer only validates the fingerprint, not the `wallet_id` within that fingerprint: [5](#0-4) 

The parsed params (including the dapp-supplied `wallet_id`) are forwarded directly to the wallet daemon via `sendCommand`: [6](#0-5) 

### Impact Explanation

A dapp paired with `chia_sendTransaction` permission can set `walletId: 3` (a CAT wallet) instead of the default `walletId: 1` (XCH wallet). The confirmation dialog shows the amount humanized as XCH (`mojo-to-xch`) and the destination address, but never reveals which wallet or asset type is being spent. The user approves believing they are sending XCH; the daemon actually spends CAT tokens from wallet 3. The amount display is also misleading: CAT mojos are shown as if they were XCH mojos. This causes the user to approve the wrong asset, wrong amount, and wrong wallet — a direct asset loss scenario.

This maps to: **High — WalletConnect state causes a user to approve the wrong asset, amount, or identity.**

### Likelihood Explanation

Any dapp that has been granted `chia_sendTransaction` permission (a spending-class command requiring explicit user approval at pairing time) can exploit this on every subsequent transaction call. The bypass mechanism (`pair.bypass`) also allows this to execute without any per-transaction confirmation if the user previously granted bypass for `chia_sendTransaction`. [7](#0-6) 

### Recommendation

1. Validate `wallet_id` against the list of wallets belonging to the current fingerprint before executing the command, rejecting values that do not correspond to an existing wallet.
2. Remove `hide: true` from `wallet_id` in the `chia_sendTransaction` confirmation schema, or replace it with a human-readable wallet label (e.g., "XCH Wallet", "MyCAT Wallet") so the user can see which asset is being spent.
3. Apply the same fix to other spending commands that accept `wallet_id` with `hide: true` or a silent default, such as `chia_selectCoins` and `chia_getSpendableCoins`. [8](#0-7) 

### Proof of Concept

1. Pair a dapp with the Chia GUI wallet, granting `chia_sendTransaction` permission.
2. From the dapp, call `chia_sendTransaction` with:
   ```json
   { "amount": "1000000", "fee": "0", "address": "xch1...", "walletId": 3 }
   ```
   where wallet ID 3 is a CAT wallet.
3. The confirmation dialog displays "0.000001 XCH → xch1..." with no mention of wallet ID or asset type.
4. The user clicks "Send". The daemon executes `send_transaction` against wallet 3, spending 1 CAT token instead of XCH.
5. The user's CAT balance is reduced; they believed they sent XCH.

The `parseDappParams` test confirms `walletId: 2` is accepted and produces `wallet_id: 2` without error, and the `humanizeParams` filter confirms `wallet_id` is never rendered in the dialog due to `hide: true`. [9](#0-8)
