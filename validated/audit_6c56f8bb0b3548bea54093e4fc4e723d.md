### Title
Gateway Skips Signature Validation for Invoke Transactions When Any Transaction Exists in Mempool for Target Address - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator skips the `__validate__` entry-point call (the on-chain signature check) for any Invoke V3 transaction with `nonce=1` whenever `account_tx_in_pool_or_recent_block` returns `true` for the sender address. Because that mempool query checks for **any** transaction in the pool for that address â€” not specifically a deploy-account transaction submitted by the same signer â€” an unprivileged attacker can submit a forged Invoke with `nonce=1` for a victim address the moment the victim's `DeployAccount` transaction appears in the mempool, bypassing gateway signature verification entirely and injecting an unauthorized transaction.

### Finding Description

**Invariant broken:** Every transaction admitted to the mempool must have its account signature verified before admission. The `skip_stateful_validations` UX shortcut breaks this invariant for Invoke transactions with `nonce=1` targeting undeployed accounts.

**Root cause â€” `skip_stateful_validations`:** [1](#0-0) 

The function returns `true` (skip `__validate__`) when:
1. The incoming transaction is an `Invoke` with `nonce == 1`
2. The on-chain account nonce is `0` (account not yet deployed)
3. `account_tx_in_pool_or_recent_block(sender_address)` returns `true`

**The mempool check is not deploy-account-specific:** [2](#0-1) 

`account_tx_in_pool_or_recent_block` returns `true` if the address has **any** transaction in the pool (`tx_pool.contains_account`) or is known from a committed block (`state.contains_account`). It does not distinguish between a `DeployAccount` submitted by the legitimate owner and any other transaction type.

**When `skip_validate=true`, the blockifier validator returns early without calling `__validate__`:** [3](#0-2) [4](#0-3) 

The `execution_flags.validate = false` path exits after `perform_pre_validation_stage` (nonce/fee/proof-facts checks only) without ever calling the account's `__validate__` entry point.

### Impact Explanation

An attacker who observes a victim's `DeployAccount` transaction in the public mempool can immediately submit an `Invoke` transaction with `nonce=1` for the victim's address, carrying an arbitrary or invalid signature. The gateway accepts it without signature verification. When the batcher later executes the block:

1. The victim's `DeployAccount` is executed (nonce 0 â†’ 1).
2. The attacker's `Invoke` (nonce=1) is executed with `validate=true` and `strict_nonce_check=true` (see `AccountTransaction::new_for_sequencing`). [5](#0-4) 

3. The `__validate__` entry point is called, fails due to the invalid signature, and the transaction is reverted.
4. The fee for the failed validation is charged from the victim's account balance.

Additionally, the attacker's `nonce=1` transaction occupies the victim's nonce slot in the mempool. If the victim concurrently submitted a legitimate `nonce=1` invoke, the mempool rejects it as `DuplicateNonce`, forcing the victim to resubmit with fee escalation or wait for the attacker's transaction to be processed.

**Impact category match:** "High â€” Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."

### Likelihood Explanation

- The mempool is publicly observable; any `DeployAccount` transaction is visible to all nodes.
- The attack requires no privileged access, no special account, and no cryptographic material belonging to the victim.
- The attacker only needs to submit a well-formed `Invoke` transaction (passing stateless size/bounds checks) with a dummy signature of valid length.
- The stateless validator only checks signature **length**, not validity. [6](#0-5) 

### Recommendation

Replace the coarse `account_tx_in_pool_or_recent_block` check with a check that specifically confirms a `DeployAccount` transaction for the same address is pending. Expose a dedicated mempool query such as `deploy_account_tx_in_pool(address) -> bool` that only returns `true` when a `DeployAccount` (not any transaction type) is present for that address. Alternatively, require the user to submit the `DeployAccount` and the first `Invoke` as an atomic bundle so the gateway can verify both signatures together before admitting either.

### Proof of Concept

1. **Victim** submits `DeployAccount` for address `V` (valid signature, nonce=0). It enters the mempool.
2. **Attacker** observes `V` in the mempool via `account_tx_in_pool_or_recent_block`.
3. **Attacker** submits `Invoke` for address `V`, `nonce=1`, with a dummy signature `[0x1]` (passes stateless length check, `max_l2_gas_price` above minimum).
4. **Gateway** evaluates `skip_stateful_validations`:
   - `tx.nonce() == Nonce(Felt::ONE)` âś“
   - `account_nonce == Nonce(Felt::ZERO)` âś“ (V not yet deployed)
   - `account_tx_in_pool_or_recent_block(V)` â†’ `true` âś“ (victim's DeployAccount is in pool)
   - Returns `skip_validate = true`; `__validate__` is **not** called.
5. Attacker's `Invoke` is admitted to the mempool.
6. **Batcher** builds a block: executes victim's `DeployAccount` (succeeds), then attacker's `Invoke` (nonce=1, `validate=true`). `__validate__` is called, fails on the dummy signature, transaction reverts.
7. Fee for the failed `__validate__` execution is deducted from `V`'s balance. [7](#0-6)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
