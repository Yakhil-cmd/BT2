### Title
Fee-on-Transfer BTC Token Breaks Pool Exit Flow, Permanently Freezing Delegator Funds — (File: `src/pool/pool.cairo`, `src/staking/staking.cairo`)

---

### Summary

The protocol supports arbitrary BTC ERC20 tokens added via `add_token`. Neither the pool nor the staking contract checks the actual received balance after a transfer. When a BTC token has (or later enables) a transfer fee, the two-hop delegation flow and the exit flow both break: the pool receives `amount - fee` but is required to forward the full `amount`, causing `exit_delegation_pool_action` to permanently revert and freeze delegator principal.

---

### Finding Description

**Vulnerability class**: Accounting bug / token-freeze bug.

The protocol explicitly supports non-STRK ERC20 tokens ("BTC tokens") added by the token admin. No restriction prevents fee-on-transfer tokens from being listed. The developers are aware of the balance-check pattern — `claim_from_reward_supplier` in `src/staking/utils.cairo` already uses a before/after balance check for STRK rewards — but this pattern is absent for all BTC token transfers.

**Entry flow (enter/add delegation)**

In `src/pool/pool.cairo`, `enter_delegation_pool` and `add_to_delegation_pool` perform a two-hop transfer:

1. `transfer_from_delegator(pool_member, amount, token_dispatcher)` — pool receives `amount - fee₁`.
2. `transfer_to_staking_contract(amount, ...)` — approves `amount` and calls `add_stake_from_pool(staker_address, amount)`.

Inside `add_stake_from_pool` (`src/staking/staking.cairo` lines 1003–1046), accounting is updated for the full `amount` **before** the transfer:

```cairo
let new_delegated_stake = old_delegated_stake + normalized_amount;
self.insert_staker_delegated_balance(...);
self.add_to_total_stake(:token_address, amount: normalized_amount);
// Transfer funds from the pool contract to the staking contract.
token_dispatcher.checked_transfer_from(
    sender: pool_contract, recipient: get_contract_address(), amount: amount.into(),
);
```

The pool only holds `amount - fee₁` but the `checked_transfer_from` demands `amount`. The transaction reverts, so no accounting discrepancy occurs here — but delegation is permanently broken for that token.

**Exit flow (the fund-freeze path)**

`exit_delegation_pool_action` in `src/pool/pool.cairo` (lines 295–333):

1. Calls `staking.remove_from_delegation_pool_action`, which transfers `native_amount` to the pool:

```cairo
token_dispatcher.checked_transfer(recipient: pool_contract, amount: native_amount.into());
```

With a fee-on-transfer token, the pool receives `native_amount - fee`.

2. The pool then transfers `unpool_amount` (= `native_amount`) to the pool member:

```cairo
token_dispatcher.checked_transfer(recipient: pool_member, amount: unpool_amount.into());
```

The pool holds only `native_amount - fee` but attempts to send `native_amount`. This reverts. Because the entire transaction reverts, the staking contract's cleared intent is also rolled back, so the pool member can retry — but every retry will fail identically. **The delegator's principal is permanently inaccessible.**

The same pattern applies in `transfer_to_pools_when_unstake` (`src/staking/staking.cairo` lines 1661–1682): when a staker calls `unstake_action`, the staking contract transfers the full delegated balance to each pool. With a fee-on-transfer BTC token, the pool receives less than the recorded balance, and every subsequent member exit reverts.

**Contrast with the existing balance-check pattern**

`claim_from_reward_supplier` (`src/staking/utils.cairo` lines 50–60) already demonstrates the correct pattern:

```cairo
let balance_before = token_dispatcher.balance_of(account: staking_contract);
reward_supplier_dispatcher.claim_rewards(:amount);
let balance_after = token_dispatcher.balance_of(account: staking_contract);
assert!(balance_after - balance_before == amount.into(), "{}", Error::UNEXPECTED_BALANCE);
```

This guard exists for STRK but is absent for BTC token transfers.

---

### Impact Explanation

If a listed BTC token enables transfer fees (analogous to USDT/USDC enabling fees, as described in the reference report), every delegator in the affected pool loses access to their principal. `exit_delegation_pool_action` reverts unconditionally because the pool always receives `amount - fee` but must forward `amount`. There is no rescue path in the protocol. This constitutes **permanent freezing of delegator funds** (High impact).

---

### Likelihood Explanation

Low-to-medium. The token admin must first list a BTC token, and that token must later enable fees. Both conditions are realistic: the protocol is explicitly designed to support arbitrary ERC20 tokens with 5–18 decimals, and real-world tokens (USDT, WBTC variants) have governance mechanisms that could enable fees. No privileged action within the staking protocol is required after the token is listed; the trigger is entirely external.

---

### Recommendation

1. **Use balance-before/after checks** for all BTC token `transfer` and `transferFrom` calls, consistent with the existing `claim_from_reward_supplier` pattern. Record the actual received amount and use that for accounting.
2. **In `add_stake_from_pool`**, move the accounting update to after the transfer and base it on the actual received amount (`balance_after - balance_before`).
3. **In `exit_delegation_pool_action` / `remove_from_delegation_pool_action`**, record the actual amount received by the pool and transfer only that amount to the pool member.
4. Alternatively, add an explicit check in `add_token` that rejects fee-on-transfer tokens (e.g., by performing a test transfer and verifying the received amount).

---

### Proof of Concept

1. Token admin calls `add_token(btc_fee_token)` where `btc_fee_token` is an ERC20 with 8 decimals and a 1% transfer fee.
2. Staker calls `set_open_for_delegation(btc_fee_token)` → pool deployed.
3. Delegator approves pool for 1 BTC (100_000_000 units) and calls `enter_delegation_pool(reward_addr, 100_000_000)`.
   - Pool receives 99_000_000 (1% fee deducted).
   - `transfer_to_staking_contract(100_000_000, ...)` → `add_stake_from_pool` → `checked_transfer_from(pool, staking, 100_000_000)` → **REVERTS** (pool has only 99_000_000).
   - Delegation is impossible.
4. Alternatively, if the token had no fee at listing time and delegators already staked, and the token later enables a 1% fee:
   - Delegator calls `exit_delegation_pool_intent(100_000_000)`.
   - After the wait window, calls `exit_delegation_pool_action(pool_member)`.
   - Staking transfers 100_000_000 to pool; pool receives 99_000_000.
   - Pool calls `checked_transfer(pool_member, 100_000_000)` → **REVERTS** (pool has only 99_000_000).
