### Title
Unbounded Loop in `move_to_block` Causes Out-of-Gas on Every Transaction After Idle Period — (`engine-hashchain/src/hashchain.rs`)

---

### Summary

`Hashchain::move_to_block` iterates one NEAR block at a time, calling `keccak` on each step. This function is invoked on every Aurora transaction via `load_hashchain`. If the Aurora contract is idle for a sufficiently long period, the stored `current_block_height` falls far behind the live NEAR block height, and the next transaction triggers an unbounded loop that exhausts the 300 Tgas NEAR transaction limit, causing every subsequent user transaction to fail with out-of-gas.

---

### Finding Description

`Hashchain::move_to_block` in `engine-hashchain/src/hashchain.rs` advances the hashchain state one block at a time using a `while` loop:

```rust
while self.current_block_height < next_block_height {
    self.previous_block_hashchain = self.block_hashchain_computer.compute_block_hashchain(...);
    self.block_hashchain_computer.clear_txs();
    self.current_block_height += 1;
}
```

Each iteration calls `compute_block_hashchain`, which invokes the `keccak` NEAR host function — a non-trivial NEAR gas cost per call. [1](#0-0) 

This function is called from `load_hashchain` in `engine/src/hashchain.rs`:

```rust
fn load_hashchain<I: IO>(io: &I, block_height: u64) -> Result<Option<Hashchain>, ContractError> {
    let mut maybe_hashchain = read_current_hashchain(io)?;
    if let Some(hashchain) = maybe_hashchain.as_mut()
        && block_height > hashchain.get_current_block_height()
    {
        hashchain.move_to_block(block_height)?;
    }
    Ok(maybe_hashchain)
}
``` [2](#0-1) 

`load_hashchain` is called at the start of every `with_hashchain` and `with_logs_hashchain` wrapper, which wraps every EVM transaction submission, connector operation, and XCC call: [3](#0-2) [4](#0-3) 

The `block_height` passed to `move_to_block` is `env.block_height()` — the live NEAR block height — while `current_block_height` is the height at which the hashchain was last updated (i.e., the last block that contained an Aurora transaction). NEAR produces approximately one block per second. If Aurora is idle for hours or days, the gap grows to thousands or millions of blocks, and the loop runs that many times per transaction.

There is no cap, guard, or maximum-step limit on the loop.

---

### Impact Explanation

Once the block gap is large enough to exceed the 300 Tgas NEAR transaction limit, **every** Aurora transaction fails with out-of-gas before any EVM execution occurs. This includes:

- EVM `submit` / `call` (user funds in transit or at rest)
- ETH connector deposit/withdrawal operations
- XCC router calls

All user funds locked in the Aurora contract become inaccessible. Recovery requires the key manager to call `pause_contract` followed by `start_hashchain` with a recent `block_height` argument. `start_hashchain` constructs a fresh `Hashchain` starting at `args.block_height + 1`, so the gap to the current block is at most 1 — making that specific call safe. [5](#0-4) 

However, if `pause_contract` itself is wrapped in `with_hashchain` (which could not be fully confirmed from the available code), then the admin recovery path also fails, escalating the impact to **permanent freezing of funds (Critical)**. Under the confirmed code path alone, the impact is **High: temporary freezing of funds** until admin recovery.

---

### Likelihood Explanation

No attacker action is required. The gap grows passively whenever Aurora receives no transactions. NEAR mainnet produces ~1 block/second; a 3-day idle period creates ~260,000 blocks. At that scale, the loop cost far exceeds 300 Tgas. Any user who submits the first transaction after a long idle period triggers the condition. The hashchain is an active production feature initialized via `start_hashchain`. [6](#0-5) 

---

### Recommendation

Replace the per-block iteration in `move_to_block` with a direct jump. Since all intermediate blocks between `current_block_height` and `next_block_height` contain zero transactions (the hashchain state is empty for those blocks), their hashchain values are deterministic and do not need to be computed one-by-one. The engine should detect a gap and advance `current_block_height` directly to `next_block_height` in O(1), computing only the final hash for the last empty block (or skipping intermediate empty blocks entirely by storing the gap count). At minimum, a maximum-step guard should be enforced per call to prevent unbounded gas consumption.

---

### Proof of Concept

1. Deploy Aurora engine with hashchain enabled (`start_hashchain` called).
2. Submit no transactions for N NEAR blocks (e.g., N = 500,000, achievable after ~6 days of inactivity, or by fast-forwarding block height in a test environment).
3. Submit any EVM transaction (e.g., a simple ETH transfer).
4. Observe: `load_hashchain` calls `move_to_block(current_block_height + N)`, which loops N times calling `keccak` each iteration, exhausting the 300 Tgas limit before the EVM executes.
5. All subsequent transactions fail identically; user funds are inaccessible.

The test infrastructure already demonstrates skipping block heights in `move_to_block`: [7](#0-6) 

Scaling the skip to production-realistic values (hundreds of thousands of blocks) reproduces the out-of-gas condition.

#[named]
pub fn start_hashchain<I: IO + Copy, E: Env>(mut io: I, env: &E) -> Result<(), ContractError> {
    let mut state = state::get_state(&io)?;
    require_paused(&state)?;
    require_key_manager_only(&state, &env.predecessor_account_id())?;

    let input = io.read_input().to_vec();
    let args = StartHashchainArgs::try_from_slice(&input).map_err(|_| errors::ERR_SERIALIZE)?;
    let block_height = env.block_height();

    // Starting hashchain must be for an earlier block
    if block_height < args.block_height {
        return Err(errors::ERR_ARGS.into());
    }

    let mut hashchain = Hashchain::new(
        state.chain_id,
        env.current_account_id(),
        args.block_height + 1,
        args.block_hashchain,
    );

    if hashchain.get_current_block_height() < block_height {
        hashchain.move_to_block(block_height)?;
    }

    hashchain.add_block_tx(
        block_height,
        function_name!(),
        &input,
        &[],
        &Bloom::default(),
    )?;
    crate::hashchain::save_hashchain(&mut io, &hashchain)?;

    state.is_paused = false;
    state::set_state(&mut io, &state)?;

    Ok(())
```
