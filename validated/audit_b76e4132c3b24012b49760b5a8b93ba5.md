### Title
Token Conservation Broken When `burn_base_fee` Feature Is Enabled: Sender Pays Full `gas_price` But Operator Receives Only Priority Fee — (`basic_bootloader/src/bootloader/transaction_flow/zk/mod.rs`)

### Summary

When the `burn_base_fee` compile-time feature is enabled, the ZKsync OS bootloader deducts the full `gas_price * gas_limit` from the sender at transaction start, but at settlement only pays the operator `priority_fee_per_gas * gas_used` and refunds the sender `gas_price * (gas_limit - gas_used)`. The base fee portion of the used gas (`base_fee * gas_used`) is never transferred to anyone — it is silently destroyed. This is an undocumented, unannounced token burn that causes the sender to pay more than the sum of what the operator receives plus what the sender is refunded.

### Finding Description

The fee lifecycle for L2 transactions in ZKsync OS is:

1. **Precharge** (`precharge_fee`): Deduct `fee_to_prepay = gas_price * gas_limit` from sender.
2. **Refund** (`refund_and_commit_fee`): Return `gas_price * (gas_limit - gas_used)` to sender.
3. **Operator payment** (`refund_and_commit_fee`): Pay `gas_price_for_operator * gas_used` to coinbase.

The critical asymmetry is in step 3. When `burn_base_fee` is active:

```rust
let gas_price_for_operator = if cfg!(feature = "burn_base_fee") {
    let base_fee = system.get_eip1559_basefee();
    context.gas_price.saturating_sub(base_fee)   // only priority fee
} else {
    context.gas_price
};
```

So `gas_price_for_operator = gas_price - base_fee = priority_fee_per_gas`.

The token accounting is:
- **Deducted from sender**: `gas_price * gas_limit`
- **Returned to sender**: `gas_price * (gas_limit - gas_used)`
- **Paid to operator**: `priority_fee_per_gas * gas_used`
- **Unaccounted (burned)**: `base_fee * gas_used`

The net loss to the sender is `gas_price * gas_used`, but the net gain to the operator is only `priority_fee_per_gas * gas_used`. The difference `base_fee * gas_used` is taken from the sender's balance and never credited anywhere.

The precharge uses `context.fee_to_prepay` which is computed as `effective_gas_price * gas_limit` (the full `gas_price`, not just the priority fee), and the refund uses `context.gas_price` (also the full effective gas price). Only the operator payment uses the reduced `gas_price_for_operator`.

### Impact Explanation

Every L2 transaction executed when `burn_base_fee` is enabled results in a permanent, silent loss of `base_fee * gas_used` tokens from the sender's balance. These tokens are neither transferred to the operator nor returned to the sender — they are destroyed. This is a resource accounting bug that causes direct, quantifiable financial loss to every transaction sender proportional to the base fee and gas consumed. The total token supply on L2 decreases with every transaction, which is inconsistent with the expected behavior of a rollup where tokens are bridged 1:1 from L1.

### Likelihood Explanation

The `burn_base_fee` feature is a compile-time flag. If it is enabled in any production build, every single L2 transaction triggers this loss. Any user submitting a transaction with `base_fee > 0` and `gas_used > 0` is affected. This is reachable by any unprivileged transaction sender — no special access is required.

### Recommendation

The precharge and refund amounts must be consistent with the operator payment. Two correct approaches:

**Option A** — Charge sender only the priority fee (EIP-1559 standard behavior):
- Precharge: `priority_fee_per_gas * gas_limit`
- Refund: `priority_fee_per_gas * (gas_limit - gas_used)`
- Operator: `priority_fee_per_gas * gas_used`

**Option B** — Keep full precharge/refund but also pay base fee to a designated burn address or protocol treasury:
- Add a transfer of `base_fee * gas_used` to a designated address (e.g., address(0) or a protocol-controlled burn address) so that `precharge = refund + operator_payment + burn_payment`.

The simplest fix consistent with EIP-1559 semantics is Option A: use `gas_price_for_operator` (the priority-fee-only price) for both the precharge and the refund, not just the operator payment.

### Proof of Concept

Given:
- `base_fee = 100`
- `priority_fee_per_gas = 50`
- `gas_price = base_fee + priority_fee_per_gas = 150`
- `gas_limit = 100_000`
- `gas_used = 21_000`
- `burn_base_fee` feature enabled

**Step 1 — Precharge** (`precharge_fee`):
Sender loses: `150 * 100_000 = 15_000_000`

**Step 2 — Refund** (`refund_and_commit_fee`, line 458):
`token_to_refund = context.gas_price * (gas_limit - gas_used) = 150 * 79_000 = 11_850_000`

**Step 3 — Operator payment** (`refund_and_commit_fee`, line 514-516):
`token_to_pay_operator = gas_price_for_operator * gas_used = (150 - 100) * 21_000 = 50 * 21_000 = 1_050_000`

**Token conservation check**:
- Deducted from sender: `15_000_000`
- Returned to sender: `11_850_000`
- Paid to operator: `1_050_000`
- Sum of returns: `11_850_000 + 1_050_000 = 12_900_000`
- **Burned (unaccounted)**: `15_000_000 - 12_900_000 = 2_100_000 = base_fee * gas_used = 100 * 21_000`

The sender permanently loses `2,100,000` tokens that go nowhere.
