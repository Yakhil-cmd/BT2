### Title
`max_nonce_for_validation_skip` Config Field Ignored in Gateway — Validation Skip Cannot Be Disabled - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

`StatefulTransactionValidatorConfig` exposes a `max_nonce_for_validation_skip` field that is intended to control the nonce range for which the `__validate__` entry-point call is skipped (the deploy-account + invoke UX feature). The gateway's `skip_stateful_validations` free function never reads this field; it hardcodes the check to `Nonce(Felt::ONE)`. When an operator sets `max_nonce_for_validation_skip = 0` to disable the skip feature, the gateway still skips `__validate__` for nonce-1 invoke transactions, admitting transactions with invalid signatures to the mempool.

### Finding Description

`StatefulTransactionValidatorConfig` declares:

```rust
pub max_nonce_for_validation_skip: Nonce,  // default Nonce(Felt::ONE)
``` [1](#0-0) 

The gateway's `run_pre_validation_checks` calls the free function `skip_stateful_validations`, passing only the transaction, the account nonce, and the mempool client — **never** `self.config.max_nonce_for_validation_skip`:

```rust
let skip_validate =
    skip_stateful_validations(executable_tx, account_nonce, mempool_client.clone()).await?;
``` [2](#0-1) 

Inside `skip_stateful_validations`, the nonce threshold is hardcoded to `Nonce(Felt::ONE)`:

```rust
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
``` [3](#0-2) 

By contrast, the legacy `PyValidator` correctly reads its own `max_nonce_for_validation_skip` field and applies it as an upper bound:

```rust
let nonce_small_enough_to_qualify_for_validation_skip =
    tx_nonce <= self.max_nonce_for_validation_skip;
let skip_validate = deploy_account_not_processed
    && is_post_deploy_nonce
    && nonce_small_enough_to_qualify_for_validation_skip;
``` [4](#0-3) 

The result is that `max_nonce_for_validation_skip` in `StatefulTransactionValidatorConfig` is a dead field — it is serialised, documented, and shipped in `config_schema.json`, but never read by the gateway's validation path. [5](#0-4) 

### Impact Explanation

When an operator sets `max_nonce_for_validation_skip = 0` to disable the UX skip entirely, the gateway still skips `__validate__` for any invoke transaction whose nonce is exactly 1 and whose sender has a deploy-account transaction in the mempool or a recent block. Because `__validate__` is the account's signature-verification entry point, this allows an attacker to submit an invoke transaction with an **invalid signature** that passes gateway admission and enters the mempool. The transaction will be reverted during block execution, but it has already been admitted — matching the impact category: **High. Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

The `run_validate_entry_point` path confirms that when `skip_validate = true`, `execution_flags.validate` is set to `false` and the `__validate__` call is entirely omitted:

```rust
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [6](#0-5) 

### Likelihood Explanation

Likelihood is **Low**. The default value of `max_nonce_for_validation_skip` is `Nonce(Felt::ONE)`, which coincides with the hardcoded value, so the bug is invisible under the default configuration. It is only triggered if an operator explicitly sets the field to `0` (to disable the feature) or to a value greater than `1` (to extend it). The field is documented and present in the production config schema, making an operator change plausible but not common.

### Recommendation

Convert `skip_stateful_validations` from a free function into a method on `StatefulTransactionValidator` (or pass `max_nonce_for_validation_skip` as a parameter) and replace the hardcoded equality check with a range check mirroring the `PyValidator` logic:

```diff

+async fn skip_stateful_validations(
     tx: &ExecutableTransaction,
     account_nonce: Nonce,

-        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
+        if Nonce(Felt::ONE) <= tx.nonce()

And update the call site in `run_pre_validation_checks` to pass `self.config.max_nonce_for_validation_skip`.

### Proof of Concept

1. Deploy a gateway with `max_nonce_for_validation_skip = 0` in `StatefulTransactionValidatorConfig`.
2. Submit a `DeployAccount` transaction for account `A`; confirm it enters the mempool.
3. Craft an `Invoke` transaction from `A` with `nonce = 1` and a **deliberately invalid signature** (e.g., all-zero signature bytes).
4. Submit the invoke to the gateway.
5. Observe: `skip_stateful_validations` returns `true` (because `tx.nonce() == Nonce(Felt::ONE)` and `account_nonce == Nonce(Felt::ZERO)` and the deploy-account is in the mempool), `execution_flags.validate` is set to `false`, and the `__validate__` entry point is never called.
6. The invalid-signature invoke transaction is accepted by the gateway and forwarded to the mempool, despite `max_nonce_for_validation_skip = 0` being intended to prevent exactly this bypass.

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
