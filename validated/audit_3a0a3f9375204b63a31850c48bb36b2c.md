### Title
Missing Zero `msg_hash` Guard in OT-Based ECDSA `sign` Enables Presign Secret Share Disclosure — (File: `src/ecdsa/ot_based_ecdsa/sign.rs`)

---

### Summary
The OT-based ECDSA `sign` function accepts `msg_hash = 0` without rejection. When `msg_hash = 0`, each participant's signature share collapses to `s_i = r * sigma_i`, directly exposing their individual presign secret share `sigma_i` (a share of `k⁻¹ · x`) to the coordinator. The robust ECDSA counterpart explicitly rejects zero message hashes; the OT-based variant does not.

---

### Finding Description

In `src/ecdsa/ot_based_ecdsa/sign.rs`, the public `sign` function validates participant count, duplicates, and threshold, but never checks whether `msg_hash` is zero. [1](#0-0) 

The robust ECDSA `sign` function in `src/ecdsa/robust_ecdsa/sign.rs` explicitly rejects zero message hashes:

```rust
if bool::from(msg_hash.is_zero()) {
    return Err(InitializationError::BadParameters(
        "msg_hash cannot be 0 to avoid potential split view attacks".to_string(),
    ));
}
``` [2](#0-1) 

No equivalent guard exists in the OT-based `sign` function.

When `msg_hash = 0`, `compute_signature_share` computes:

```rust
Ok(msg_hash * k_i + r * sigma_i)
// = 0 * k_i + r * sigma_i
// = r * sigma_i
``` [3](#0-2) 

Since `r = x_coordinate(presignature.big_r)` is public, the coordinator receives `s_i = r * sigma_i` from each participant via `recv_from_others` in `do_sign_coordinator` and can trivially compute `sigma_i = s_i * r⁻¹` for every participant, recovering their individual presign secret shares. [4](#0-3) 

The security documentation explicitly warns against signing with `h = 0` for the robust ECDSA scheme, but the OT-based scheme has no such enforcement. [5](#0-4) 

---

### Impact Explanation

`sigma_i` is each participant's share of `sigma = k⁻¹ · x` — a presign secret that combines the nonce inverse with the secret key share. Revealing `sigma_i` to the coordinator constitutes disclosure of presign secrets and nonce material, matching the **Critical** impact: *"Extraction, reconstruction, or disclosure of private signing shares, aggregate secret material, presign secrets, nonce material, or confidential derived secrets."*

With all `sigma_i` values, the coordinator reconstructs `sigma = k⁻¹ · x` via Lagrange interpolation. If the coordinator can also obtain a second signing session using the same presignature (presignature reuse, which is prohibited but not cryptographically enforced), they can extract the secret key `x` as: `x = sigma · k = (k⁻¹ · x) · k`. Even without reuse, the disclosure of individual `sigma_i` values is itself a Critical violation of the security model. [6](#0-5) 

---

### Likelihood Explanation

The coordinator is a participant in the protocol and is the entity that calls `sign` and collects individual `s_i` shares from other participants privately. A malicious coordinator can trivially pass `msg_hash = Scalar::ZERO` to the `sign` function. No special capabilities are required beyond being the coordinator. The attack is a single-step operation with no cryptographic difficulty.

---

### Recommendation

Add a zero check for `msg_hash` in the OT-based ECDSA `sign` function, mirroring the robust ECDSA implementation:

```rust
if bool::from(msg_hash.is_zero()) {
    return Err(InitializationError::BadParameters(
        "msg_hash cannot be 0".to_string(),
    ));
}
```

This guard should be placed after the participant validation checks and before the protocol is started, consistent with the pattern in `src/ecdsa/robust_ecdsa/sign.rs`. [7](#0-6) 

---

### Proof of Concept

1. Malicious coordinator calls `sign(participants, coordinator, threshold, me, public_key, presignature, Scalar::ZERO)` in `src/ecdsa/ot_based_ecdsa/sign.rs`.
2. Each honest participant computes `s_i = 0 * k_i + r * sigma_i = r * sigma_i` and sends it privately to the coordinator.
3. The coordinator receives individual `s_j` values via `recv_from_others` in `do_sign_coordinator`. [8](#0-7) 
4. The coordinator computes `sigma_j = s_j * r⁻¹` for each participant `j`, recovering their presign secret shares.
5. The coordinator reconstructs `sigma = k⁻¹ · x` via Lagrange interpolation over all recovered `sigma_j` values, disclosing the aggregate presign secret.

pub fn sign(
    participants: &[Participant],
    coordinator: Participant,
    threshold: impl Into<ReconstructionLowerBound>,
    me: Participant,
    public_key: AffinePoint,
    presignature: RerandomizedPresignOutput,
    msg_hash: Scalar,
) -> Result<impl Protocol<Output = SignatureOption>, InitializationError> {
    let threshold = usize::from(threshold.into());
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }

    let participants =
        ParticipantList::new(participants).ok_or(InitializationError::DuplicateParticipants)?;

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }

    // ensure number of participants during the signing phase is >= threshold
    if participants.len() < threshold {
        return Err(InitializationError::NotEnoughParticipantsForThreshold {
            threshold,
            participants: participants.len(),
        });
    }

    let ctx = Comms::new();
    let fut = fut_wrapper(
        ctx.shared_channel(),
        participants,
        coordinator,
        me,
        public_key,
        presignature,
        msg_hash,
    );
    Ok(make_protocol(ctx, fut))
