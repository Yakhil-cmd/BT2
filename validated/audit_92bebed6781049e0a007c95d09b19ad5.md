I need to look at one more critical area — the `setPoolAdminFeeDestination` function compared to the other fee-changing functions, and verify the exact inconsistency.

### Title
Accrued admin fees are misdirected to the new destination when `setPoolAdminFeeDestination` is called without a prior fee collection — (File: metric-core/contracts/MetricOmmPoolFactory.sol)

---

### Summary

`MetricOmmPoolFactory::setPoolAdminFeeDestination` updates `poolAdminFeeDestination[pool]` without first flushing accrued fees to the old destination. Every other fee-parameter mutator (`setPoolAdminFees`, `setPoolProtocolFee`) performs a `collectFees` call before the state change. The missing flush means all fees that accrued under the old destination are silently redirected to the new destination on the next `collectPoolFees` call, causing a direct financial loss for the old recipient.

---

### Finding Description

`setPoolAdminFees` and `setPoolProtocolFee` both follow the pattern: **collect at old rates → update state**: [1](#0-0) 

`setPoolAdminFeeDestination` skips the collection step entirely: [2](#0-1) 

`collectFees` on the pool transfers the admin share to whichever address is passed as `adminFeeDestination_`: [3](#0-2) 

Because `collectPoolFees` (permissionless) reads `poolAdminFeeDestination[pool]` at call time: [4](#0-3) 

any fees that accrued while the old destination was set will be paid to the new destination the next time collection is triggered.

The pool admin and the admin fee destination are registered as independent addresses at pool creation: [5](#0-4) 

They are routinely different parties (e.g., pool admin = operations multisig; admin fee destination = a separate treasury contract). The pool admin can therefore redirect fees that were earned by the old treasury to any new address.

---

### Impact Explanation

Every swap charges spread and notional fees that accumulate on the pool. The spread surplus grows continuously; the notional accumulator (`notionalFeeToken0Scaled` / `notionalFeeToken1Scaled`) is only zeroed on `collectFees`. When the pool admin calls `setPoolAdminFeeDestination` without a prior collection, the entire accrued admin share — potentially representing many blocks of trading activity — is transferred to the new destination instead of the old one. The old destination suffers a direct, unrecoverable loss of owed fees.

---

### Likelihood Explanation

The pool admin is a semi-trusted role explicitly listed in the contest's admin-path pivot ("fee collection destinations"). The trigger is a single permissioned call with no timelock. Any routine treasury migration or destination update silently causes the loss. The permissionless `collectPoolFees` can be front-run by the pool admin: propose destination change → call `setPoolAdminFeeDestination` → let `collectPoolFees` settle to the new address.

---

### Recommendation

Add a fee collection step inside `setPoolAdminFeeDestination` before updating the mapping, mirroring the pattern used in `setPoolAdminFees`:

```solidity
function setPoolAdminFeeDestination(address pool, address newAdminFeeDestination)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (newAdminFeeDestination == address(0)) revert InvalidAdminFeeDestination();

+   PoolFeeConfig memory c = poolFeeConfig[pool];
+   IMetricOmmPoolCollectFees(pool).collectFees(
+       c.protocolSpreadFeeE6,
+       c.adminSpreadFeeE6,
+       c.protocolNotionalFeeE8,
+       c.adminNotionalFeeE8,
+       poolAdminFeeDestination[pool]   // flush to OLD destination first
+   );

    poolAdminFeeDestination[pool] = newAdminFeeDestination;
    emit PoolAdminFeeDestinationUpdated(pool, newAdminFeeDestination);
}
```

### Proof of Concept

1. Pool is deployed with `admin = Alice`, `adminFeeDestination = TreasuryA`.
2. Many swaps occur; spread surplus and notional accumulators grow on the pool.
3. Alice calls `setPoolAdminFeeDestination(pool, TreasuryB)` — no collection happens.
4. Anyone calls `collectPoolFees(pool)`.
5. `collectFees` is invoked with `adminFeeDestination_ = TreasuryB`.
6. All fees accrued under TreasuryA are transferred to TreasuryB.
7. TreasuryA receives nothing despite being the entitled recipient for the entire accrual period.

The root cause is the missing `collectFees` call in `setPoolAdminFeeDestination` at line 444–446, contrasted with the explicit flush present in `setPoolAdminFees` at lines 418–425 and in `setPoolProtocolFee` at lines 328–335. [2](#0-1) [6](#0-5) [7](#0-6)
