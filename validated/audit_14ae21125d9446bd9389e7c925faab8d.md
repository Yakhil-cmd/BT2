`schedule` only checks the caller is the parent engine; it stores any `PromiseArgs` verbatim: [4](#0-3) 

### Step 5 — `execute_scheduled` calls `recursive_promise_create`

`execute_scheduled` removes the promise from storage and immediately calls `Self::promise_create`: [5](#0-4)

The NEAR already deposited into the router sub-account (in a prior receipt) is **not** rolled back.

The router has no general withdrawal function. `send_refund` only returns the fixed 2 NEAR storage-staking amount and requires no failed promise results: [7](#0-6)

## Verdict

### Title
Unvalidated Promise Count in `CrossContractCallArgs::Delayed` Path Enables Permanent Freezing of Funds — (`engine-precompiles/src/xcc.rs`, `etc/xcc-router/src/lib.rs`)

### Summary
The `Delayed` XCC path accepts and stores arbitrarily deep `NearPromise::And` trees without checking the total promise count. wNEAR is deducted from the caller and real NEAR is deposited into the router sub-account before any structural validation occurs. A promise whose execution would exceed NEAR's per-receipt promise limit can never be executed, and no refund path exists, permanently freezing the deposited NEAR.

### Finding Description
`CrossContractCallArgs::Delayed` in `engine-precompiles/src/xcc.rs` (lines 159–172) computes `call.total_near()` but never calls `call.promise_count()`. The `Eager` branch uses `promise_count()` for gas accounting but still does not validate against NEAR's protocol limit. The router's `schedule` function (lines 136–144 of `etc/xcc-router/src/lib.rs`) stores any `PromiseArgs` without structural validation.

`recursive_promise_create` (lines 242–281 of `etc/xcc-router/src/lib.rs`) expands `NearPromise::And` trees by calling `env::promise_create` for every leaf and `env::promise_and` for every combinator node. For a two-level `And(vec![And(vec![simple; M]); M])`, the total promise count is M² + M + 1. With M = 10 this is 111, exceeding NEAR's `max_promises_per_receipt_count = 100`.

When `execute_scheduled` is called, the NEAR SDK panics on the host-function error, rolling back the storage `remove` but **not** the NEAR already deposited into the router sub-account in a prior receipt. The promise remains in storage but is permanently unexecutable. There is no refund mechanism for the deposited NEAR.

### Impact Explanation
Any wNEAR attached to the sub-promises (via `attached_balance`) is transferred from the caller to the engine's implicit address, then unwrapped and deposited into the router sub-account before `schedule` is called. If the promise can never be executed, that NEAR is permanently locked. The only recovery path is an admin-initiated router contract upgrade — there is no automatic or user-initiated refund.

### Likelihood Explanation
The attack requires only a valid EVM sender and the ability to call the XCC precompile with a crafted `Delayed` payload. No special privileges are needed. The `Delayed` path is a documented, production-supported feature. A user could trigger this accidentally (e.g. via a buggy EVM contract that constructs large `And` trees) or intentionally.

### Recommendation
1. In the `Delayed` branch of `run_with_handle` (`engine-precompiles/src/xcc.rs`), call `call.promise_count()` and reject inputs that exceed a safe threshold (e.g. 50, well below NEAR's limit of 100, to leave headroom for the engine's own promise overhead).
2. Alternatively, add the same check inside `Router::schedule` before storing the promise.
3. Consider adding a `promise_count` method that counts the actual number of NEAR host calls (not just leaf nodes) to account for `promise_and` calls as well.

    pub fn schedule(&mut self, #[serializer(borsh)] promise: PromiseArgs) {
        self.assert_preconditions();
