### Title
Missing Rescue Mechanism for NEAR Tokens Permanently Frozen in XCC Router Sub-accounts After Failed Cross-Contract Calls - (File: etc/xcc-router/src/lib.rs)

### Summary

The XCC router sub-account contract (`etc/xcc-router/src/lib.rs`) has no mechanism to recover NEAR tokens that are refunded back to it after a failed cross-contract call. When a user invokes the XCC precompile with a non-zero `attached_balance`, wNEAR is burned from the user's EVM balance and unwrapped into real NEAR inside the router sub-account. If the downstream cross-contract call fails, NEAR is refunded by the NEAR runtime to the router sub-account, but the router exposes no function to forward those tokens back to the user. The funds are permanently frozen.

---

### Finding Description

**Entry path — XCC precompile (`engine-precompiles/src/xcc.rs`)**

When a user submits an EVM transaction targeting the `cross_contract_call` precompile address with a non-zero `attached_balance`, the precompile deducts the corresponding wNEAR from the caller's EVM balance and transfers it to the engine's implicit address: [1](#0-0) 

**Engine orchestration (`engine/src/xcc.rs`)**

The engine then schedules a `withdraw_wnear_to_router` self-call that exits the wNEAR through the `ExitToNear` precompile, delivering real NEAR to the router sub-account. The user's desired promise is chained as the final callback: [2](#0-1) 

**Router execution (`etc/xcc-router/src/lib.rs`)**

The router's `execute` (eager) or `execute_scheduled` (delayed) method creates the actual NEAR cross-contract call with the `attached_balance` specified in the promise args: [3](#0-2) 

When the target contract panics or otherwise fails, the NEAR runtime refunds the attached deposit to the predecessor of the receipt that created the call — which is the router sub-account itself. The router sub-account's balance increases, but there is no function to forward those tokens back to the originating EVM address.

**The only existing refund path is `send_refund`**, which is hardcoded to transfer a fixed constant `REFUND_AMOUNT` (the storage-staking amount) back to the parent engine, and only when there are **no** failed promises: [4](#0-3) 

`require_no_failed_promises()` is enforced inside `send_refund`, so it cannot be called after a failed downstream call. Even if it could, it only transfers the fixed storage-staking constant, not the user's arbitrary attached NEAR. [5](#0-4) 

---

### Impact Explanation

Any NEAR attached by a user to an XCC call that subsequently fails is permanently frozen inside the router sub-account. The user's wNEAR ERC-20 balance was already reduced at the precompile level before the call was dispatched; there is no rollback of that EVM-side debit. The router exposes no `withdraw`, `rescue`, or arbitrary-amount refund function. Impact: **Critical — permanent freezing of user funds**.

---

### Likelihood Explanation

The cross-contract call can fail for any number of realistic reasons: the target contract panics, the target account does not exist, the target method is not found, or the target contract runs out of gas. Any EVM user who uses the XCC precompile with `attached_balance > 0` and whose downstream call fails is affected. Likelihood: **Low** (requires a failed downstream call), but the scenario is entirely realistic and unprivileged.

---

### Recommendation

Add a permissioned rescue function to the XCC router that allows the parent engine (or the EVM address that owns the router) to withdraw the full NEAR balance (minus storage staking) back to the engine, which can then re-mint the equivalent wNEAR to the user's EVM address. Alternatively, attach a failure-handling callback after the user's promise that automatically refunds the NEAR to the engine for re-crediting.

---

### Proof of Concept

1. Alice holds 10 wNEAR as an ERC-20 balance on Aurora.
2. Alice submits an EVM transaction to the `cross_contract_call` precompile with `attached_balance = 5 NEAR` targeting `some_contract.near::some_method`.
3. The precompile deducts 5 wNEAR from Alice's EVM balance and transfers it to the engine's implicit address.
4. The engine calls `withdraw_wnear_to_router` — 5 NEAR arrives in Alice's router sub-account (`<alice_address>.aurora`).
5. The engine calls `execute` on the router, which calls `some_contract.near::some_method` with 5 NEAR attached.
6. `some_contract.near` panics; the NEAR runtime refunds 5 NEAR to the router sub-account.
7. Alice's EVM wNEAR balance is still 5 (already deducted). The router sub-account holds 5 NEAR.
8. Alice has no EVM function to call, and the router exposes no rescue path. The 5 NEAR is permanently frozen. [3](#0-2) [6](#0-5)

    /// Similar security considerations here as for `execute`.
    pub fn schedule(&mut self, #[serializer(borsh)] promise: PromiseArgs) {
        self.assert_preconditions();

        let nonce = self.nonce.get().unwrap_or_default();
        self.scheduled_promises.insert(nonce, promise);
        self.nonce.set(&(nonce + 1));

        near_sdk::log!("Promise scheduled at nonce {}", nonce);
    }

    /// It is intentional that this function can be called by anyone (not just the parent).
    /// There is no security risk to allowing this function to be open because it can only
    /// act on promises that were created via `schedule`.
    #[payable]
    pub fn execute_scheduled(&mut self, nonce: U64) {
        let Some(promise) = self.scheduled_promises.remove(&nonce.0) else {
            env::panic_str("ERR_PROMISE_NOT_FOUND")
        };
        let promise_id = Self::promise_create(promise);
        env::promise_return(promise_id);
    }

    pub fn send_refund(&self) -> Promise {
        let parent = self.get_parent().unwrap_or_else(env_panic);

        require_caller(&parent)
            .and_then(|_| require_no_failed_promises())
            .unwrap_or_else(env_panic);

        Promise::new(parent).transfer(REFUND_AMOUNT)
    }

    /// Checks the following preconditions:
    ///   1. Contract is initialized
    ///   2. predecessor_account_id == self.parent
    ///   3. There are no failed promise results
    /// These preconditions must be checked on methods where are important for
    /// the security of the contract (e.g. `execute`).
    fn require_preconditions(&self) -> Result<(), Error> {
        let parent = self.get_parent()?;
        require_caller(&parent)?;
        require_no_failed_promises()?;
        Ok(())
    }

    /// Panics if any of the preconditions checked in `require_preconditions` are not met.
    fn assert_preconditions(&self) {
        self.require_preconditions().unwrap_or_else(env_panic);
    }
