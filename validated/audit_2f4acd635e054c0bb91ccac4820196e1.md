### Title
Permanent ERC-20 Token Loss via `ExitToNear` Precompile When `error_refund` Feature Is Absent — (`engine-precompiles/src/native.rs`)

### Summary

When the `error_refund` Cargo feature is not compiled in, the legacy ERC-20 exit path (flag `0x1`, no `:unwrap`/omni suffix) schedules a bare `ft_transfer` promise with **no callback**. If `ft_transfer` fails on NEAR (e.g., because the recipient account has no NEP-141 storage deposit), the burned ERC-20 tokens are permanently destroyed with no on-chain recovery path.

### Finding Description

In `ExitToNear::run`, after `exit_erc20_token_to_near` returns for the legacy `_` arm:

- `transfer_near_args` = `None`
- `method` = `"ft_transfer"`

The callback struct is assembled as:

```rust
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(not(feature = "error_refund"))]
    refund: None,          // always None without the feature
    transfer_near: transfer_near_args,  // None for legacy ft_transfer
};
``` [1](#0-0) 

Because both fields are `None`, `callback_args` equals `ExitToNearPrecompileCallbackArgs::default()`, and the branch at line 470 fires:

```rust
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // ← bare promise, no callback
} else {
    PromiseArgs::Callback(...)
};
``` [2](#0-1) 

The `error_refund` feature is **not** in the default feature set:

```toml
[features]
default = ["std"]
error_refund = ["aurora-engine-precompiles/error_refund"]
``` [3](#0-2) 

When `error_refund` is enabled, `refund_call_args` populates `callback_args.refund` with a `RefundCallArgs`, making `callback_args != default()` and causing a `PromiseArgs::Callback` to be scheduled. The callback's failure branch then calls `engine::refund_on_error`, which re-mints the burned ERC-20 tokens. [4](#0-3) 

Without `error_refund`, `refund_on_error` is never called, and the ERC-20 tokens burned during the exit are gone forever.

### Impact Explanation

The ERC-20 tokens are burned on the Aurora EVM side before the NEAR-side `ft_transfer` promise is dispatched. If `ft_transfer` fails (e.g., recipient not registered), the NEP-141 tokens remain in Aurora's account but the corresponding ERC-20 tokens no longer exist. There is no on-chain mechanism to recover them. This constitutes **permanent freezing/destruction of user funds**.

The existing test suite explicitly confirms this outcome:

```rust
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
``` [5](#0-4) 

### Likelihood Explanation

Any user holding tokens in a factory-deployed ERC-20 mirror contract can trigger this by calling the ERC-20 withdraw function with a NEAR account ID that has no storage deposit on the NEP-141 contract. No special privileges are required. The `error_refund` feature is not in the default feature set, so any deployment compiled without it is affected. The trigger condition (unregistered recipient) is easy to reach accidentally or intentionally.

### Recommendation

1. **Enable `error_refund` by default** in `engine/Cargo.toml` (add it to the `default` feature list), or make the callback unconditional for the ERC-20 exit path.
2. Alternatively, validate on the EVM side (before burning) that the recipient account is registered with the NEP-141 contract, though this is not straightforward cross-chain.
3. At minimum, document clearly that deployments without `error_refund` expose users to permanent token loss on failed exits.

### Proof of Concept

The existing test `test_exit_to_near_refund` in `engine-tests/src/tests/erc20_connector.rs` already proves the scenario:

1. Mint ERC-20 tokens to `ft_owner` (bridged from NEP-141).
2. Call `exit_to_near` with recipient `"unregistered.near"` (no NEP-141 storage deposit).
3. Without `error_refund`: `erc20_balance(ft_owner) == FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT` — the exit amount is permanently gone.
4. `nep141_balance(aurora) == FT_TRANSFER_AMOUNT` — NEP-141 tokens remain locked in Aurora's account.
5. `nep141_balance(unregistered.near) == 0` — recipient received nothing. [6](#0-5) 

The root cause is the compile-time gate at: [1](#0-0) 

combined with the no-callback branch at: [7](#0-6)
