### Title
Malicious Participant Permanently Aborts DKG/Reshare via Deserialization Error Propagation in `recv_from_others` — (File: `src/protocol/helpers.rs`)

---

### Summary
`recv_from_others` propagates deserialization errors directly to the caller via `?`. A single malicious participant can send a message with a valid routing header but a malformed payload on any shared waitpoint, causing `recv_from_others` to return a `ProtocolError` that aborts DKG, reshare, or refresh for every honest party. Because the attacker can repeat this on every protocol attempt, the denial is permanent.

---

### Finding Description

`recv_from_others` in `src/protocol/helpers.rs` collects one message per participant before proceeding:

```rust
while !seen.full() {
    let (from, msg) = chan.recv(waitpoint).await?;   // ← error propagated
    if seen.put(from) {
        messages.push((from, msg));
    }
}
``` [1](#0-0) 

`chan.recv` internally calls `Comms::recv`, which deserializes the payload and propagates failure:

```rust
let decoded: Result<T, Box<dyn error::Error + Send + Sync>> =
    rmp_serde::decode::from_slice(message_data).map_err(std::convert::Into::into);
Ok((from, decoded?))   // ← DeserializationError returned to caller
``` [2](#0-1) 

`do_keyshare` calls `recv_from_others` twice — once to collect commitment hashes (Round 1) and once to collect secret shares (Round 5):

```rust
for (from, their_commitment_hash) in
    recv_from_others(&chan, wait_round_1, &participants, me).await?
``` [3](#0-2) 

```rust
for (from, signing_share_from) in
    recv_from_others(&chan, wait_round_3, &participants, me).await?
``` [4](#0-3) 

Both call sites propagate the error with `?`, so a single bad message aborts the entire `do_keyshare` execution for the receiving honest party.

This is in direct contrast to `reliable_broadcast_receive_all`, which explicitly swallows deserialization errors and continues:

```rust
match chan.recv(wait).await {
    Ok(value) => (from, (sid, vote)) = value,
    _ => continue,   // ← error silently ignored
};
``` [5](#0-4) 

The inconsistency means the broadcast path is resilient to malformed messages, but the direct-message path is not.

---

### Impact Explanation

A malicious participant who is part of the protocol session sends a message on the correct channel tag and waitpoint (which it knows deterministically, since channel tags are derived from public participant identities) but with a payload that cannot be deserialized as the expected type `T`. `recv_from_others` returns `Err(ProtocolError::Other(...))`, which propagates through `do_keyshare` and aborts DKG, reshare, or refresh for every honest party that receives the message. Because the attacker can repeat this on every restart, honest parties are permanently denied key generation and resharing.

**Allowed impact matched**: High — Permanent denial of key generation / reshare / refresh for honest parties under valid protocol inputs.

---

### Likelihood Explanation

Any participant already admitted to the session can trigger this. No special privilege is required beyond being listed in the `ParticipantList`. The attacker needs only to craft a message whose first `MessageHeader::LEN` (40) bytes match the expected channel tag and waitpoint, with an arbitrary suffix. The channel tag for the shared channel is derived deterministically from public constants (`ChannelTag::root_shared()`), so the attacker can compute the correct header without any secret knowledge. [6](#0-5) [7](#0-6) 

---

### Recommendation

Mirror the error-handling strategy used in `reliable_broadcast_receive_all`: skip messages that fail to deserialize rather than propagating the error. Specifically, change `recv_from_others` to:

```rust
while !seen.full() {
    let Ok((from, msg)) = chan.recv::<T>(waitpoint).await else {
        continue;   // discard malformed messages, keep waiting
    };
    if seen.put(from) {
        messages.push((from, msg));
    }
}
```

This ensures a single malicious participant cannot abort the protocol for honest parties by injecting a malformed message.

---

### Proof of Concept

1. Honest participants `[P0, P1, P2]` start DKG with threshold 2.
2. Malicious participant `P1` computes the shared-channel header for `wait_round_1` (deterministic: `ChannelTag::root_shared()` + waitpoint `0`).
3. `P1` sends a raw message to `P0` whose first 40 bytes are the correct header and whose remaining bytes are not valid `msgpack` for `HashOutput`.
4. `P0`'s `recv_from_others` call at `do_keyshare` line 422 receives this message, `Comms::recv` fails to deserialize it, returns `Err(ProtocolError::Other(...))`.
5. The `?` at `recv_from_others` line 20 propagates the error; `do_keyshare` returns `Err`, aborting DKG for `P0`.
6. `P1` repeats on every restart — DKG never completes for `P0`. [8](#0-7) [9](#0-8) [10](#0-9)
