### Title
Secrets Database Silently Falls Back to Unencrypted Storage for Non-Consensus Nodes When Encryption Key File Is Absent - (File: `cmd/scaffold.go`)

### Summary

In `initSecretsDB()`, when the secrets database encryption key file is missing from disk, non-consensus nodes (Collection, Execution, Verification, Access) silently proceed to open the secrets Badger database **without any encryption**. Only Consensus nodes treat a missing key as a fatal error. This means the random beacon private key and other node secrets stored in the secrets database are written to disk in plaintext for all non-consensus node roles if the key file is absent.

### Finding Description

In `cmd/scaffold.go`, the `initSecretsDB()` function loads the AES-256 encryption key for the Badger secrets database from a flat file on disk (`secretsdb-key`). The critical logic is:

```go
encryptionKey, err := loadSecretsEncryptionKey(fnb.BootstrapDir, fnb.NodeID)
if errors.Is(err, os.ErrNotExist) {
    if fnb.NodeRole == flow.RoleConsensus.String() {
        // missing key is a fatal error for SN nodes
        return fmt.Errorf("secrets db encryption key not found: %w", err)
    }
    fnb.Logger.Warn().Msg("starting with secrets database encryption disabled")
} else if err != nil {
    return fmt.Errorf("failed to read secrets db encryption key: %w", err)
} else {
    opts = opts.WithEncryptionKey(encryptionKey)
}
```

When `os.ErrNotExist` is returned and the node is **not** a Consensus node, the code emits only a warning log and continues — opening the secrets database with `badger.DefaultOptions` that have **no encryption key set**. The secrets database then stores all sensitive data (including the random beacon private key via `InsertMyBeaconPrivateKey`) as unencrypted flat files on disk.

The encryption key itself is also stored as a raw binary flat file (`secretsdb-key`) with mode `0644` (world-readable) via `WriteText` in `cmd/util/cmd/common/utils.go`:

```go
err = os.WriteFile(path, data, 0644)
```

And the private node info JSON (`node-info.priv.json`) containing staking and networking private keys is written with the same `0644` permissions via `WriteFile` in `utils/io/write.go`:

```go
err = os.WriteFile(path, data, 0644)
```

There are two compounding issues:
1. The secrets DB encryption key file is stored unprotected (`0644`) on disk — any local user can read it.
2. If that file is missing (e.g., deleted, misconfigured, or never provisioned for non-consensus nodes), the secrets database itself opens without encryption, silently.

### Impact Explanation

The secrets database stores the node's random beacon private key (`InsertMyBeaconPrivateKey` / `UpsertMyBeaconPrivateKey`). For Collection and Execution nodes, the machine account private key (`node-machine-account-info.priv.json`) and staking/networking private keys (`node-info.priv.json`) are also written to disk in plaintext JSON with `0644` permissions.

A local attacker with read access to the node's filesystem (e.g., another process on the same host, a container escape, or a misconfigured shared volume) can:
- Read the raw `secretsdb-key` file and decrypt the Badger database, or
- If the key file is absent, read the unencrypted Badger database files directly to extract the random beacon private key.
- Read `node-info.priv.json` directly to obtain staking and networking private keys.

Compromise of a node's staking key allows impersonation of that node in the protocol. Compromise of a Collection or Execution node's machine account key allows unauthorized on-chain transactions from that account. Compromise of the random beacon private key for a consensus node undermines the threshold signature scheme used for randomness generation.

### Likelihood Explanation

The silent fallback to unencrypted storage is reachable whenever the `secretsdb-key` file is absent for a non-consensus node. This is a realistic operational scenario: the bootstrap tooling generates the key file, but it is an optional step for non-consensus nodes (the `keygen` command calls `WriteSecretsDBEncryptionKeyFiles` but the node startup does not enforce its presence for non-SN roles). Additionally, the `0644` file permissions on all private key files mean any local user on the host can read them regardless of whether the DB is encrypted.

### Recommendation

1. **Short term:** Change `initSecretsDB()` to treat a missing encryption key as a fatal error for **all** node roles, not just Consensus nodes. Remove the silent fallback.
2. **Short term:** Change file permissions for all private key files and the `secretsdb-key` file from `0644` to `0600` in both `utils/io/write.go` (`WriteFile`) and `cmd/util/cmd/common/utils.go` (`WriteText`).
3. **Long term:** Store the secrets DB encryption key in a platform secret store (e.g., environment variable injected by a secrets manager, HSM, or OS keyring) rather than as a flat file on disk.

### Proof of Concept

**Silent unencrypted DB startup for a non-consensus node:**

1. Bootstrap a Verification or Access node using `bootstrap key`. The `secretsdb-key` file is generated.
2. Delete the `secretsdb-key` file: `rm private-root-information/private-node-info_<ID>/secretsdb-key`
3. Start the node. `initSecretsDB()` calls `loadSecretsEncryptionKey`, receives `os.ErrNotExist`, checks `fnb.NodeRole == flow.RoleConsensus.String()` — which is false — and logs only a warning.
4. The Badger secrets database opens without `WithEncryptionKey`, storing all secrets in plaintext.
5. An attacker with filesystem access reads the Badger `.vlog` files directly to extract the random beacon private key bytes.

**World-readable key file:**

```
$ ls -la private-root-information/private-node-info_<ID>/secretsdb-key
-rw-r--r-- 1 nodeuser nodeuser 32 ... secretsdb-key
```

	if err != nil {
		return fmt.Errorf("could not create output dir: %w", err)
	}

	err = os.WriteFile(path, data, 0644)
	if err != nil {
		return fmt.Errorf("could not write file: %w", err)
	}

	return nil
