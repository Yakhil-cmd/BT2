### Title
FIFO Queue Silently Drops Higher-Nonce Staged Transactions When Lowest-Nonce Tx Is Rejected With No Committed Nonce — (`crates/apollo_mempool/src/fifo_transaction_queue.rs`)

---

### Summary

When `commit_block` is called with `committed_nonces` containing no entry for address A, and `rejected_tx_hashes` contains the lowest-nonce staged transaction for A, `collect_txs_to_rewind` excludes A from the rewind set entirely. The subsequent `staged_txs.clear()` then silently discards all higher-nonce staged transactions for A without re-queuing them. Those transactions remain in the `tx_pool` but are permanently absent from the FIFO queue and cannot be re-submitted (duplicate error), making them unsequenceable until TTL expiration or eviction.

---

### Finding Description

`collect_txs_to_rewind` applies the following policy for addresses with no committed nonces:

```
// Address has no committed txs in this block.
// - first nonce rejected -> do not rewind address
// - first nonce not rejected -> rewind address
let first_tx = txs.iter().min_by_key(|tx| tx.tx_reference.nonce)...;
!rejected_tx_hashes.contains(&first_tx.tx_reference.tx_hash)
``` [1](#0-0) 

When the first-nonce tx is rejected, the filter returns `false`, so address A is never inserted into `addresses_to_rewind`. Step 3 then produces an empty `txs_to_rewind` for A. [2](#0-1) 

Back in `rewind_txs`, after pushing zero transactions to the front of the queue, `staged_txs.clear()` unconditionally destroys every staged entry — including tx_A_nonce1 and tx_A_nonce2 that were never rejected. [3](#0-2) 

`remove_rejected_txs` then removes tx_A_nonce0 from the pool and calls `remove_by_address(A)` on the queue (a no-op because nonce1/2 are not in the queue). It returns `{A: nonce_0}` as the account nonce update. [4](#0-3) 

`update_accounts_with_gap({A: nonce_0})` then detects that `tx_pool.get_lowest_nonce(A)` = nonce_1 > nonce_0, marks A as a gap account, and schedules it for eventual eviction — but never re-inserts nonce1/nonce2 into the FIFO queue. [5](#0-4) 

In FIFO mode, `add_tx_inner` inserts transactions into the queue only at initial submission time. There is no path to re-queue a transaction that is already in the pool. [6](#0-5) 

Re-submission by the user fails with `DuplicateTransaction`. [7](#0-6) 

---

### Impact Explanation

Valid, admitted transactions at nonce1 and nonce2 are permanently removed from the FIFO sequencing queue. They cannot be sequenced, cannot be re-submitted, and will only leave the mempool via TTL expiration or gap-account eviction. This directly satisfies the **High** impact criterion: *Mempool/gateway/RPC admission rejects valid transactions before sequencing.*

---

### Likelihood Explanation

This triggers in normal operation whenever:
1. Multiple transactions from the same address are staged together (all returned by a single or consecutive `get_txs` call), and
2. The lowest-nonce transaction among them is rejected at block commit time (execution failure, fee underflow, etc.), and
3. The address has no other committed transactions in that block.

No privileged access is required. Any user whose nonce-0 transaction fails execution while nonce-1 and nonce-2 are already staged will silently lose those higher-nonce transactions from the queue.

---

### Recommendation

When the first staged nonce for an address is rejected and there are additional higher-nonce staged transactions, those higher-nonce transactions should not be silently discarded. Options:

1. **Re-queue higher-nonce txs as gap transactions**: After `staged_txs.clear()`, explicitly re-insert nonce1+ back into the FIFO queue (they will be stuck until nonce0 is re-submitted, but they remain sequenceable once the gap is filled).
2. **Remove higher-nonce txs from the pool**: If the design intent is to discard them, also call `tx_pool.remove` for each non-rejected, non-rewound staged tx so the pool and queue stay consistent and the user can re-submit.

The current state — pool contains them, queue does not, re-submission blocked — is the worst of all outcomes.

---

fn test_higher_nonce_txs_lost_when_first_nonce_rejected() {
    let mut mempool = /* FIFO mempool */;

    let tx0 = add_tx_input!(tx_hash: 1, address: "0x1", tx_nonce: 0, account_nonce: 0);
    let tx1 = add_tx_input!(tx_hash: 2, address: "0x1", tx_nonce: 1, account_nonce: 0);
    let tx2 = add_tx_input!(tx_hash: 3, address: "0x1", tx_nonce: 2, account_nonce: 0);

    for i in 1..=3 { mempool.update_tx_block_metadata(tx_hash!(i), tx_metadata(1000, 100)); }
    for input in [&tx0, &tx1, &tx2] { add_tx(&mut mempool, input); }

    // Stage all three transactions.
    get_txs_and_assert_expected(&mut mempool, 3, &[tx0.tx, tx1.tx.clone(), tx2.tx.clone()]);

    // tx0 is rejected; no committed nonces for "0x1".
    commit_block(&mut mempool, [], [tx_hash!(1)]);

    // FAILS: tx1 and tx2 are gone from the queue — permanently unsequenceable.
    get_txs_and_assert_expected(&mut mempool, 10, &[tx1.tx, tx2.tx]);
}
```

The assertion on the last line fails because `tx1` and `tx2` were silently dropped from `staged_txs` by `staged_txs.clear()` at [8](#0-7) 

and were never re-inserted into the queue.
