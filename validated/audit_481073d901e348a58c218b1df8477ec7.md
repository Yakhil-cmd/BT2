### Title
`protocolUnpausePool` Does Not Restore Pre-Pause State, Leaving Pool Permanently Admin-Paused After Protocol Unpause - (File: metric-core/contracts/MetricOmmPoolFactory.sol)

### Summary

`MetricOmmPoolFactory.protocolUnpausePool()` unconditionally sets the pool's pause level to `1` (admin-paused) rather than restoring the pool to its state before the protocol pause. When the protocol pauses an **active** pool (`pauseLevel == 0`) and later unpauses it, the pool is left in admin-paused state (`pauseLevel == 1`), blocking all swaps until the pool admin manually calls `unpausePool()` — an action the admin never initiated.

### Finding Description

The factory defines three pause levels:

| Level | Meaning |
|---|---|
| 0 | Active |
| 1 | Paused by admin |
| 2 | Paused by protocol |

`protocolPausePool` accepts both `cur == 0` and `cur == 1` as valid source states: [1](#0-0) 

`protocolUnpausePool` always transitions to `1`, regardless of what the pre-protocol-pause state was: [2](#0-1) 

The pre-protocol-pause state (`0` or `1`) is never stored. When the protocol pauses an **active** pool (`0 → 2`) and then unpauses it (`2 → 1`), the pool lands in admin-paused state even though the admin never called `pausePool`. The pool admin must now call `unpausePool` to restore trading — an action they never initiated and may not know is required.

The `swap` function enforces `whenNotPaused`, which reverts on any non-zero pause level: [3](#0-2) [4](#0-3) 

### Impact Explanation

All swaps are blocked (`PoolPaused` revert) for the duration between `protocolUnpausePool` and the admin's subsequent `unpausePool` call. If the admin is a multisig, a DAO, or simply unaware that the pool landed in admin-paused state (because they never paused it), this window can be extended indefinitely. Traders cannot execute swaps; the pool's core function is broken. LPs are not directly at risk of principal loss since `removeLiquidity` does not check `whenNotPaused`, but the pool is commercially inoperable.

### Likelihood Explanation

The protocol owner is expected to use `protocolPausePool` in emergency scenarios (e.g., oracle compromise, exploit response). After the emergency is resolved, `protocolUnpausePool` is the natural recovery path. The scenario where the pool was active (`0`) before the emergency — the common case — always triggers this residual admin-paused state. The admin may not monitor for this unexpected state transition.

### Recommendation

Store the pre-protocol-pause level and restore it on unpause:

```solidity
// In factory storage
mapping(address => uint8) public prePausedLevel;

function protocolPausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 0 && cur != 1) revert InvalidPauseTransition(cur, 2);
    prePausedLevel[pool] = cur;          // remember previous state
    IMetricOmmPoolFactoryActions(pool).setPause(2);
}

function protocolUnpausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 2) revert InvalidPauseTransition(cur, 1);
    uint8 restore = prePausedLevel[pool]; // restore previous state (0 or 1)
    delete prePausedLevel[pool];
    IMetricOmmPoolFactoryActions(pool).setPause(restore);
}
```

### Proof of Concept

1. Pool is deployed and active (`pauseLevel == 0`).
2. Protocol owner calls `protocolPausePool(pool)` — pool transitions `0 → 2`.
3. Emergency resolved; protocol owner calls `protocolUnpausePool(pool)` — pool transitions `2 → 1` (admin-paused).
4. Any call to `pool.swap(...)` reverts with `PoolPaused()` because `pauseLevel == 1 != 0`.
5. Pool admin (who never called `pausePool`) must discover the unexpected state and call `unpausePool(pool)` to restore trading.
6. Until step 5 completes, all swap volume is lost and the pool is commercially inoperable. [5](#0-4)
