### Title
Unconstrained `disable_rewards` Parameter in `update_rewards` Allows Any Caller to Permanently Freeze All Staker Consensus Rewards â€” (File: src/staking/staking.cairo)

---

### Summary

The `update_rewards` function in `Staking.cairo` is publicly callable with no access control on the `disable_rewards: bool` parameter. Any unprivileged caller can invoke it with `disable_rewards: true`, which advances the global `last_reward_block` counter without distributing any rewards. Because `last_reward_block` is a single global slot, one such call per block permanently consumes the reward opportunity for every staker in that block.

---

### Finding Description

`update_rewards` is defined at line 1449 of `src/staking/staking.cairo`:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();                          // only checks pause flag
    let current_block_number = starknet::get_block_number();
    assert!(
        current_block_number > self.last_reward_block.read(),
        "{}",
        Error::REWARDS_ALREADY_UPDATED,
    );
    ...
    self.last_reward_block.write(current_block_number);   // global slot consumed here

    if disable_rewards || self.is_pre_consensus() {
        return;                                           // exits without distributing rewards
    }
    ...
``` [1](#0-0) 

The only guards are:

1. `general_prerequisites()` â€” checks the pause flag only, no caller identity check. [2](#0-1) 
2. A block-number monotonicity check on the **global** `last_reward_block`. [3](#0-2) 
3. Staker existence / activity checks â€” satisfied by passing any currently active staker address. [4](#0-3) 

`last_reward_block` is declared as a single global storage slot:

```cairo
/// Last block number for which rewards were distributed.
last_reward_block: BlockNumber,
``` [5](#0-4) 

Once `last_reward_block` is written to the current block number (line 1485), every subsequent call to `update_rewards` in the same block reverts with `REWARDS_ALREADY_UPDATED`. The reward slot for that block is permanently consumed with zero rewards distributed.

The analog to the LineaRollup tree-depth bug is direct:

| LineaRollup | Starknet Staking |
|---|---|
| Operator submits unconstrained `l2MerkleTreesDepth` | Any caller submits unconstrained `disable_rewards: true` |
| Finalization succeeds, but message claiming is blocked | Block slot is consumed, but reward distribution is skipped |
| Users cannot claim cross-chain messages | Stakers cannot claim consensus rewards |

### Impact Explanation

**High â€” Permanent freezing of unclaimed yield.**

An attacker who calls `update_rewards(any_active_staker, true)` once per block:

- Advances `last_reward_block` to the current block without distributing rewards.
- Blocks every legitimate `update_rewards` call for that block (they all revert with `REWARDS_ALREADY_UPDATED`).
- Repeated every block, this permanently prevents all stakers from accumulating consensus-phase rewards.

The stakers' principal is not at risk, but all future yield is frozen for as long as the attack continues.

---

### Likelihood Explanation

**Medium.**

- No special privilege is required; any Starknet account can call `update_rewards`.
- The attacker must submit one transaction per block, incurring gas costs. On Starknet, transaction fees are low, making sustained griefing economically feasible.
- The attacker needs only one valid active staker address, which is publicly observable on-chain.
- There is no profit motive, but the damage to the protocol and all stakers is severe.

---

### Recommendation

Add an access-control check to `update_rewards` so that only a trusted caller (e.g., the sequencer, an authorized rewards manager, or the attestation contract) can invoke it. Alternatively, remove the `disable_rewards` parameter entirely and handle reward disabling through a separate privileged administrative function that does **not** advance `last_reward_block`.

---

### Proof of Concept

1. Attacker identifies any currently active staker address `S` (publicly readable from `stakers` vector or events).
2. At the start of each new block, attacker submits:
   ```
   update_rewards(staker_address: S, disable_rewards: true)
   ```
3. The call passes all checks: contract is not paused, `current_block > last_reward_block`, staker `S` is active.
4. `last_reward_block` is written to the current block number; the function returns early â€” **no rewards distributed**.
5. Any legitimate call to `update_rewards` for any staker in the same block now reverts with `REWARDS_ALREADY_UPDATED`.
6. Repeated every block, all stakers are permanently denied consensus rewards. [6](#0-5)
