Audit Report

## Title
SNS Swap `validate_participation_constraints` Underestimates Minimum SNS Tokens Per Participant When Neurons' Fund Participates, Causing Neuron Basket Creation Failure and Loss of Funds - (File: rs/sns/init/src/lib.rs)

## Summary

Check #8 in `validate_participation_constraints` computes the minimum SNS tokens a direct participant will receive using `max_direct_participation_icp_e8s` as the denominator. At finalization, `create_sns_neuron_recipes` computes the actual allocation using `current_total_participation_e8s()` — which includes Neurons' Fund (NF) contributions — as the denominator. When NF participates, the actual per-participant SNS allocation is strictly smaller than what the validation assumed, potentially falling below the per-neuron minimum stake threshold. Affected neuron recipes are marked `ClaimedSwapNeuronStatus::Invalid` and never retried, while the participant's ICP has already been irreversibly swept to SNS Governance.

## Finding Description

**Root cause — denominator mismatch between validation and finalization:**

In `validate_participation_constraints` (rs/sns/init/src/lib.rs, L1635–1638), check #8 computes:

```rust
let min_participant_sns_e8s = min_participant_icp_e8s as u128
    * initial_swap_amount_e8s as u128
    / max_direct_participation_icp_e8s as u128;   // only direct cap
``` [1](#0-0) 

This passes when `min_participant_sns_e8s >= count * (neuron_minimum_stake_e8s + transaction_fee_e8s)`. [2](#0-1) 

At finalization, `create_sns_neuron_recipes` (rs/sns/swap/src/swap.rs, L818–852) uses a different denominator:

```rust
let total_participant_icp_e8s = match NonZeroU64::try_from(
    self.current_total_participation_e8s(),   // direct + NF
) { ... };

let amount_sns_e8s = Swap::scale(
    buyer_state.amount_icp_e8s(),
    sns_being_offered_e8s,
    total_participant_icp_e8s,   // larger when NF participates
);
``` [3](#0-2) 

`Swap::scale` performs integer division, so a larger denominator yields a strictly smaller result: [4](#0-3) 

The reduced `amount_sns_e8s` is passed to `create_sns_neuron_basket_for_direct_participant`, which calls `generate_vesting_schedule` → `apportion_approximately_equally`. Neither function checks that each chunk meets `neuron_minimum_stake_e8s`: [5](#0-4) 

In `claim_swap_neurons`, each recipe is validated against `neuron_minimum_stake_e8s`. If validation fails, the recipe is marked `ClaimedSwapNeuronStatus::Invalid` and skipped — permanently: [6](#0-5) 

The participant's ICP has already been swept to SNS Governance via `sweep_icp` before `claim_swap_neurons` is called. There is no refund path once the swap is committed.

**Exploit path:**
1. SNS swap is configured with `neurons_fund_participation: true`.
2. NF participates at finalization, making `total_participant_icp_e8s > max_direct_participation_icp_e8s`.
3. A direct participant contributes exactly `min_participant_icp_e8s`.
4. Their `amount_sns_e8s` is computed with the larger denominator, falling below `count * neuron_minimum_stake_e8s`.
5. `claim_swap_neurons` marks their recipes `Invalid`; ICP is already gone.

**Existing guards are insufficient:** The validation check at L1635–1655 is the only pre-swap guard for this invariant, and it uses the wrong denominator. There is no runtime guard in `create_sns_neuron_basket_for_direct_participant` or `generate_vesting_schedule` to catch sub-minimum chunks.

## Impact Explanation

A direct swap participant contributing the minimum allowed ICP suffers complete, irreversible loss of their ICP contribution. Their ICP is transferred to SNS Governance (via `sweep_icp`), their SNS neuron recipes are rejected as `Invalid` by `claim_swap_neurons`, and no retry or refund mechanism exists. The loss per victim is bounded by `min_participant_icp_e8s`, which in real SNS launches can be hundreds of ICP. Multiple participants at the minimum contribution level can be affected simultaneously in the same swap. This matches the allowed impact: **High — Significant SNS security impact with concrete user funds loss**.

## Likelihood Explanation

All three required conditions — NF participation enabled, NF actually contributing at finalization, and a direct participant at the minimum contribution — are part of the normal, intended operation of SNS swaps. NF participation is a core feature enabled in many production SNS launches. No privileged access, key compromise, or social engineering is required. Any unprivileged user who participates at `min_participant_icp_e8s` in an NF-enabled swap is at risk. The scenario is fully deterministic and repeatable.

## Recommendation

In `validate_participation_constraints`, replace `max_direct_participation_icp_e8s` with the maximum possible total participation (direct + NF) as the worst-case denominator for check #8:

```rust
let max_total_participation_icp_e8s = max_direct_participation_icp_e8s
    .saturating_add(max_neurons_fund_participation_icp_e8s.unwrap_or(0));

let min_participant_sns_e8s = min_participant_icp_e8s as u128
    * initial_swap_amount_e8s as u128
    / max_total_participation_icp_e8s as u128;
```

Additionally, add a runtime guard in `create_sns_neuron_basket_for_direct_participant` (and the NF equivalent) that returns an `Err` if `amount_sns_token_e8s / count < neuron_minimum_stake_e8s + transaction_fee_e8s`, so finalization fails loudly rather than silently producing unclaimable recipes.

## Proof of Concept

**Parameters:**
- `min_participant_icp_e8s = 1_000_000_000` (10 ICP)
- `max_direct_participation_icp_e8s = 10_000_000_000` (100 ICP)
- `initial_swap_amount_e8s = 1_000_000_000` (10 SNS tokens)
- `neuron_basket_construction_parameters.count = 3`
- `neuron_minimum_stake_e8s = 30_000_000`
- `transaction_fee_e8s = 10_000`
- `max_neurons_fund_participation_icp_e8s = 90_000_000_000` (900 ICP)

**Validation (passes):**
```
min_participant_sns_e8s = 1e9 * 1e9 / 1e10 = 100_000_000
required = 3 * (30_000_000 + 10_000) = 90_030_000
100_000_000 >= 90_030_000  ✓
```

**At finalization (NF contributes 90 ICP, direct total = 100 ICP):**
```
total_participant_icp_e8s = 190_000_000_000
amount_sns_e8s = 1e9 * 1e9 / 19e9 = 52_631_578
per_neuron = 52_631_578 / 3 = 17_543_859  <  30_000_000  ✗
```

The participant's 10 ICP is swept to SNS Governance. All three neuron recipes have `17_543_859` e8s, below `neuron_minimum_stake_e8s`. `claim_swap_neurons` marks them `Invalid`. The participant loses 10 ICP with no recourse.

A deterministic integration test using PocketIC can reproduce this by: configuring a swap with the above parameters, simulating NF participation at finalization, having a direct participant contribute `min_participant_icp_e8s`, running `create_sns_neuron_recipes` and `claim_swap_neurons`, and asserting that the returned `ClaimedSwapNeuronStatus` is `Invalid` for all three recipes.
