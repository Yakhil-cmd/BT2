Audit Report

## Title
ckBTC Minter Permanently Destroys User ckBTC Without Reimbursement on `AmountTooLow`/`DustOutput` Batch Failure - (File: `rs/bitcoin/ckbtc/minter/src/lib.rs`)

## Summary
When `submit_pending_requests` calls `build_unsigned_transaction` and receives `BuildTxError::AmountTooLow` or `BuildTxError::DustOutput`, it finalizes the affected `RetrieveBtcRequest`(s) with `FinalizedStatus::AmountTooLow` via `remove_retrieve_btc_request` without invoking `reimburse_canceled_requests`. Because ckBTC is burned from the ICRC-1 ledger at submission time (before the request enters the pending queue), and `FinalizedStatus::AmountTooLow` is a terminal state with no downstream reimbursement path, the user's ckBTC is permanently destroyed with no BTC sent and no ckBTC returned.

## Finding Description

**Burn-before-queue ordering:** In `retrieve_btc` (`rs/bitcoin/ckbtc/minter/src/updates/retrieve_btc.rs`, lines 209–232), `burn_ckbtcs` is called and succeeds before `accept_retrieve_btc_request` places the request in the pending queue. The same ordering applies in `retrieve_btc_with_approval` (lines 314–333). The ckBTC is gone from the ledger the moment the request is accepted.

**`AmountTooLow` branch — no reimbursement:** In `submit_pending_requests` (`rs/bitcoin/ckbtc/minter/src/lib.rs`, lines 412–434), when `build_unsigned_transaction` returns `BuildTxError::AmountTooLow` (triggered at line 1307 when `fee + minter_fee > amount`), every request in the batch is passed to `state::audit::remove_retrieve_btc_request` with `FinalizedStatus::AmountTooLow`. No call to `reimburse_canceled_requests` is made.

**`DustOutput` branch — same omission:** Lines 436–453 handle `BuildTxError::DustOutput` identically for the offending request: `remove_retrieve_btc_request` with `FinalizedStatus::AmountTooLow`, no reimbursement.

**`InvalidTransaction` branch — correct behavior for contrast:** Lines 400–410 show that for `BuildTxError::InvalidTransaction`, `reimburse_canceled_requests` is called with a computed `reimbursement_fee`, scheduling a mint-back to the user's `reimbursement_account`.

**`FinalizedStatus::AmountTooLow` is terminal:** The `FinalizedStatus` enum (`rs/bitcoin/ckbtc/minter/src/state.rs`, lines 259–267) has only two variants: `AmountTooLow` and `Confirmed { txid }`. There is no variant that triggers reimbursement, and no code path transitions out of `AmountTooLow` to a reimbursement state.

**Event log replay confirms no reimbursement:** `EventType::RemovedRetrieveBtcRequest` replay (`rs/bitcoin/ckbtc/minter/src/state/eventlog.rs`, lines 405–418) unconditionally pushes `FinalizedStatus::AmountTooLow` with no `ScheduleWithdrawalReimbursement` event emitted.

**`reimbursement_account` is populated:** Both `retrieve_btc` (line 218) and `retrieve_btc_with_approval` (line 327) set `reimbursement_account: Some(Account { owner: caller, ... })` on the `RetrieveBtcRequest`. The infrastructure for reimbursement is present on the request object; the `AmountTooLow` path simply never uses it.

**Fee spike trigger:** `fee_based_retrieve_btc_min_amount` is updated dynamically each time `estimate_fee_per_vbyte` runs (lines 245–248 in `lib.rs`). A request submitted when fees were low passes the submission-time check (line 169–171 in `retrieve_btc.rs`) but can fail `build_unsigned_transaction` at batch-build time if fees have spiked, because the condition `fee + minter_fee > amount` (line 1306–1308 in `lib.rs`) is evaluated against the current fee rate, not the fee rate at submission time.

## Impact Explanation
This is a permanent, unrecoverable loss of user ckBTC: the ICRC-1 ledger burn is irreversible, no BTC is sent, and no ckBTC is minted back. This constitutes a ledger conservation violation — ckBTC supply is reduced without a corresponding BTC output — and directly harms users of the ckBTC Chain Fusion integration. This matches the allowed High impact: **"Significant Chain Fusion, ck-token, ledger … security impact with concrete user or protocol harm."** The per-user loss is bounded by the withdrawal amount (minimum ~50,000 satoshis), but multiple users in the same batch can be affected simultaneously, and the condition is repeatable across fee-spike events.

## Likelihood Explanation
The exploit path requires no privileges. Any principal can call `retrieve_btc` or `retrieve_btc_with_approval` with an amount at or above the current `fee_based_retrieve_btc_min_amount`. Bitcoin network fee spikes are a well-documented, recurring phenomenon (e.g., Ordinals/BRC-20 congestion events). The window between submission and batch processing is non-zero by design (`max_time_in_queue_nanos`). A fee spike during that window is sufficient to trigger `BuildTxError::AmountTooLow` for requests that were valid at submission. The condition is more likely when many users submit near-minimum requests that are batched together, as the per-request fee share grows with batch size.

## Recommendation
The `BuildTxError::AmountTooLow` and `BuildTxError::DustOutput` branches in `submit_pending_requests` should schedule reimbursements instead of silently finalizing requests. Concretely:
1. Add an `AmountTooLow` variant to `WithdrawalReimbursementReason` (or reuse a zero-fee path).
2. Call `reimburse_canceled_requests` (with `total_fee = 0` to return the full burned amount, bypassing the fee-deduction assertion) for affected requests, mirroring the `InvalidTransaction` branch.
3. Update the event log replay for `RemovedRetrieveBtcRequest` to emit a `ScheduleWithdrawalReimbursement` event, or introduce a new event type that carries the reimbursement information, so state reconstruction after upgrade correctly schedules the mint-back.

## Proof of Concept
1. Observe current `fee_based_retrieve_btc_min_amount` (e.g., 50,000 satoshis).
2. Call `retrieve_btc` with `amount = 50,000`. ckBTC is burned; request enters pending queue with `reimbursement_account` set.
3. Bitcoin network fees spike (e.g., during a congestion event). `estimate_fee_per_vbyte` updates `last_fee_per_vbyte`.
4. `submit_pending_requests` runs. `build_unsigned_transaction` evaluates `fee + minter_fee > 50,000` → `true` → returns `BuildTxError::AmountTooLow`.
5. `remove_retrieve_btc_request(s, request, FinalizedStatus::AmountTooLow, runtime)` is called. No `reimburse_canceled_requests` call occurs.
6. Query `retrieve_btc_status(block_index)` → `RetrieveBtcStatusV2::AmountTooLow`.
7. User's 50,000 satoshi ckBTC is permanently destroyed. No BTC was sent. No ckBTC was returned.
A deterministic integration test can reproduce this by: (a) submitting a request at a low fee rate, (b) injecting a high fee-percentile response into the mock Bitcoin canister before the next `ProcessLogic` task runs, and (c) asserting that `pending_withdrawal_reimbursements` is non-empty after the task completes (it will be empty, confirming the bug).
