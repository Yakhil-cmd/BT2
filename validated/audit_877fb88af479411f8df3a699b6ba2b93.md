### Title
COA Contract Implements ERC-1271 (`isValidSignature`) But Omits ERC-1271 from `supportsInterface`, Breaking Compatibility with ERC-165-Checking EVM Contracts - (File: `fvm/evm/handler/coa/coa.sol`)

---

### Summary

The Flow Cadence-Owned Account (COA) EVM contract implements `isValidSignature` (ERC-1271) but its `supportsInterface` function does not declare support for the ERC-1271 interface ID (`0x1626ba7e`). Any EVM contract that checks `supportsInterface(0x1626ba7e)` before calling `isValidSignature` — a standard pattern in Safe v1.4.0, OpenZeppelin's `SignatureChecker`, and many DeFi protocols — will receive `false` and refuse to accept the COA as a valid signer, even though the COA fully implements the function.

---

### Finding Description

The COA contract in `fvm/evm/handler/coa/coa.sol` is the production EVM smart contract wallet deployed for every Cadence-Owned Account. Its `supportsInterface` function (lines 67–73) returns `true` only for four interfaces:

```solidity
function supportsInterface(bytes4 id) external view virtual override returns (bool) {
    return
        id == type(ERC1155TokenReceiver).interfaceId ||
        id == type(ERC721TokenReceiver).interfaceId ||
        id == type(ERC777TokensRecipient).interfaceId ||
        id == type(IERC165).interfaceId;
}
```

However, the same contract also implements `isValidSignature` (ERC-1271) at lines 113–125:

```solidity
// ERC1271 requirement 
function isValidSignature(
    bytes32 _hash,
    bytes memory _sig
) external view virtual returns (bytes4){
    ...
}
```

The ERC-1271 interface ID is `bytes4(keccak256("isValidSignature(bytes32,bytes)"))` = `0x1626ba7e`, which is the same value stored in the `ValidERC1271Signature` constant at line 62. This value is **never returned as `true`** from `supportsInterface`.

The official EVM contract documentation in `fvm/evm/stdlib/contract.cdc` line 474 explicitly advertises COA as supporting ERC-1271:

> "COAs are smart contract wallets that accept native token transfers and support several ERCs including ERC-165, ERC-721, ERC-777, ERC-1155, **ERC-1271**."

The compiled bytecode in `coa_bytes.hex` is embedded directly into the Go binary via `//go:embed` in `coa.go` and deployed on-chain for every COA, so this is a production defect.

---

### Impact Explanation

Any EVM contract that follows the ERC-165-first pattern before invoking ERC-1271 — including Safe v1.4.0 (which added `require(Guard(guard).supportsInterface(...), "GS300")`), OpenZeppelin's `SignatureChecker.isValidSignatureNow`, and many DeFi multisig/DAO protocols — will call `supportsInterface(0x1626ba7e)` on the COA, receive `false`, and reject the COA as a valid signer. This prevents COA from being used as a smart contract wallet signer in any such protocol, blocking cross-VM asset authorization and defeating the core purpose of the COA as a smart contract wallet in the EVM ecosystem.

---

### Likelihood Explanation

The likelihood is **medium-high**. ERC-165 pre-checks before ERC-1271 calls are a well-established pattern in the EVM ecosystem. Safe v1.4.0 (the most widely deployed multisig) enforces this check. Any user who creates a COA and attempts to use it as a signer in a Safe v1.4.0 or any ERC-165-checking protocol will encounter this failure. No special privileges or attacker cooperation are required — the failure is triggered by ordinary EVM contract interaction.

---

### Recommendation

Add the ERC-1271 interface ID to the `supportsInterface` return condition in `fvm/evm/handler/coa/coa.sol`:

```solidity
function supportsInterface(bytes4 id) external view virtual override returns (bool) {
    return
        id == type(ERC1155TokenReceiver).interfaceId ||
        id == type(ERC721TokenReceiver).interfaceId ||
        id == type(ERC777TokensRecipient).interfaceId ||
        id == type(IERC165).interfaceId ||
        id == ValidERC1271Signature; // 0x1626ba7e — ERC-1271
}
```

After updating the Solidity source, recompile and update `coa_bytes.hex` and `coa_abi.json` accordingly.

---

### Proof of Concept

1. A user creates a COA on Flow EVM.
2. The user attempts to add the COA as a signer/owner in a Safe v1.4.0 multisig deployed on Flow EVM.
3. Safe v1.4.0's `setGuard` (or equivalent signer-validation path) calls `COA.supportsInterface(0x1626ba7e)`.
4. `supportsInterface` returns `false` because `0x1626ba7e` is not in the checked set (lines 67–73 of `coa.sol`).
5. Safe v1.4.0 reverts with `"GS300"`, blocking the COA from being used as a signer.
6. The COA's assets remain inaccessible via the Safe multisig workflow, even though `isValidSignature` is fully implemented and would work correctly if called. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
