### Title
Malicious CKD Participant Can Corrupt the CKD Output by Sending Arbitrary Share Values Without Verification - (`src/confidential_key_derivation/protocol.rs`)

---

### Summary

The `do_ckd_coordinator` function aggregates share contributions `(norm_big_y, norm_big_c)` from all participants and adds them together without any cryptographic verification that each participant used their actual private share. A single malicious participant can send arbitrary group elements, which the coordinator unconditionally accepts and folds into the final `CKDOutput`. The protocol then completes successfully from the coordinator's perspective, delivering a silently corrupted key to the TEE application — directly analogous to the Bridge contract paying a fee regardless of whether `_invokeMessageCall()` succeeded.

---

### Finding Description

In `do_ckd_coordinator`, after computing its own share, the coordinator collects contributions from every other participant:

```rust
// src/confidential_key_derivation/protocol.rs  lines 50-55
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
``` [1](#0-0) 

Each honest participant is supposed to compute:

```
big_y  = y · G                          (random nonce commitment)
big_s  = x_i · H(pk ‖ app_id)          (private-share contribution)
big_c  = big_s + y · A                  (ElGamal ciphertext)
norm_big_y = λ_i · big_y
norm_big_c = λ_i · big_c
```

as implemented in `compute_signature_share`: [2](#0-1) 

There is **no zero-knowledge proof, DLEQ proof, or any other check** that the participant actually used their registered private share `x_i` when forming `big_c`. The coordinator performs no verification before adding the received elements to its running totals. A malicious participant can therefore send any pair of group elements — including the identity, random points, or values crafted to bias the output toward an attacker-chosen key — and the coordinator will silently incorporate them.

The participant-side function simply sends and returns `None`: [3](#0-2) 

There is no round-trip challenge, no commitment-then-reveal, and no consistency check between `norm_big_y` and `norm_big_c`.

---

### Impact Explanation

**High — Corruption of CKD output.**

The final `CKDOutput` aggregated by the coordinator is:

```
Y_total = Σ norm_big_y_i
C_total = Σ norm_big_c_i
```

If any single participant substitutes arbitrary values, `C_total` no longer equals `msk · H(pk ‖ app_id) + Y_total · A`. When the TEE application calls `unmask(app_sk)` on the result, it obtains a value that is not the intended confidential key. The application silently accepts this wrong key with no indication of failure, matching the allowed impact: *"Corruption of CKD outputs so honest parties accept inconsistent or unusable cryptographic outputs."*

---

### Likelihood Explanation

Any single participant in the CKD session is sufficient to trigger the attack. No privileged access, leaked secrets, or external assumptions are required. The attacker only needs to be a registered participant — a realistic role in the documented threat model — and to deviate from the protocol by sending crafted group elements instead of their honest share.

---

### Recommendation

Require each participant to accompany their `(norm_big_y, norm_big_c)` with a **DLEQ (Discrete Log Equality) proof** demonstrating that the same private share `x_i` was used in both `big_s = x_i · H(pk ‖ app_id)` and the participant's registered public share `X_i = x_i · G`. The coordinator must verify every such proof before adding the contribution to the running totals. Contributions that fail verification should be rejected and the protocol aborted, identifying the malicious participant.

---

### Proof of Concept

A malicious participant replaces `compute_signature_share` output with identity elements:

```rust
// Malicious participant sends zeroed contributions
let norm_big_y = ElementG1::identity();
let norm_big_c = ElementG1::identity();
chan.send_private(waitpoint, coordinator, &(norm_big_y, norm_big_c))?;
```

The coordinator at lines 50–55 adds these without complaint. The resulting `CKDOutput` is missing the malicious participant's legitimate `λ_i · x_i · H(pk ‖ app_id)` term, so `unmask(app_sk)` returns a value that differs from `msk · H(pk ‖ app_id)`. The TEE application receives and uses this wrong key with no error.

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
```
