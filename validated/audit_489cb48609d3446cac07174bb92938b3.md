### Title
`transfer_from_cronos_module` Uses `transferFrom` Instead of `unsafe_transfer`, Permanently Blocking Source-Coin Native→CRC21 Conversion - (File: `contracts/src/ModuleCRC21.sol`)

### Summary

`ModuleCRC21.transfer_from_cronos_module` calls `transferFrom(module_address, addr, amount)` — the allowance-gated DSToken method — instead of the allowance-free `unsafe_transfer`. Because `module_address` never grants an allowance to the contract itself, every invocation reverts. The keeper calls this function as the final step of converting a source-coin native token back to its CRC21 EVM representation, making that conversion path permanently broken for all auto-deployed CRC21 contracts.

### Finding Description

`ModuleCRC21.transfer_from_cronos_module` is the contract-side function the Cronos module calls to release locked CRC21 tokens to a user when they convert a `cronos0x…` native coin back to its EVM form.

```solidity
// contracts/src/ModuleCRC21.sol  line 51-53
function transfer_from_cronos_module(address addr, uint amount) public {
    transferFrom(module_address, addr, amount);   // ← wrong: requires allowance
}
```

`DSToken.transferFrom` checks `allowance[src][msg.sender]`. Here `src = module_address` and `msg.sender = address(this)` (the contract calling itself). `module_address` never calls `approve`, so `allowance[module_address][contract] == 0` always, and the call reverts unconditionally. [1](#0-0) 

Every other module-callable function in the same contract uses the allowance-free `unsafe_transfer` or `unsafe_burn`:

```solidity
// line 46-49 — correct pattern
function transfer_by_cronos_module(address addr, uint amount) public {
    require(msg.sender == module_address);
    unsafe_transfer(addr, module_address, amount);   // no allowance needed
}
``` [2](#0-1) 

The reference test contract `TestERC21Source.sol` and `TestCRC20Proxy.sol` both implement the correct pattern (`transfer(addr, amount)` / `crc20Contract.transfer(addr, amount)`): [3](#0-2) [4](#0-3) 

The keeper calls this function in `ConvertCoinFromNativeToCRC21` when `isSource == true` (denom prefix `cronos0x…`):

```go
// x/cronos/keeper/evm.go  line 115-129
if isSource {
    err = k.bankKeeper.SendCoinsFromAccountToModule(...)   // burn step 1
    err = k.bankKeeper.BurnCoins(...)                      // burn step 2
    // unlock crc tokens — ALWAYS REVERTS
    _, err = k.CallModuleCRC21(ctx, contract, "transfer_from_cronos_module", sender, coin.Amount.BigInt())
    if err != nil {
        return err   // entire tx reverts, but conversion is still broken
    }
}
``` [5](#0-4) 

`ConvertCoinFromNativeToCRC21` is reached from `ConvertVouchersToEvmCoins` (the IBC hook path) and from `MsgConvertVouchers` (the direct user tx path): [6](#0-5) 

### Impact Explanation

Any user holding a `cronos0x…` native token (source-coin denom) who attempts to convert it back to its CRC21 EVM representation — either via `MsgConvertVouchers` or via the IBC auto-conversion hook — will always receive a failed transaction. The EVM token unlock step unconditionally reverts, making the native→CRC21 conversion path for all source-coin denoms permanently non-functional. Users whose tokens arrive as source-coin native tokens through IBC or bridge flows cannot obtain the corresponding EVM tokens, permanently locking them out of the EVM-side asset.

This matches: **High — Permanent or long-lived inability for honest users to process valid conversion flows.**

### Likelihood Explanation

The bug is triggered by any unprivileged user calling `MsgConvertVouchers` with a `cronos0x…` denom, or by any IBC transfer that auto-triggers conversion for a source-coin token. No special privileges, leaked keys, or unusual conditions are required. The revert is deterministic and 100% reproducible.

### Recommendation

Replace the allowance-gated `transferFrom` with the allowance-free `unsafe_transfer`, matching the pattern used by every other internal module function and by the reference test contracts:

```solidity
function transfer_from_cronos_module(address addr, uint amount) public {
    require(msg.sender == module_address);          // add missing auth guard
    unsafe_transfer(module_address, addr, amount);  // use unsafe_transfer
}
``` [7](#0-6) 

Also note: unlike all sibling functions, `transfer_from_cronos_module` is missing the `require(msg.sender == module_address)` guard entirely, which should be added regardless.

### Proof of Concept

1. Deploy a `ModuleCRC21` contract with `isSource = true` and denom `cronos0x<contractAddr>`.
2. Mint 100 CRC21 tokens to `module_address` (simulating locked source tokens).
3. Call `transfer_from_cronos_module(userAddr, 100)` from `module_address`.
4. Observe revert: DSToken checks `allowance[module_address][address(this)] == 0` → `ds-token-insufficient-allowance`.
5. On-chain: submit `MsgConvertVouchers` with a `cronos0x…` coin. The keeper burns the native coin, then calls `transfer_from_cronos_module` → revert → entire state rolled back → user's conversion permanently fails. [8](#0-7) [1](#0-0)
