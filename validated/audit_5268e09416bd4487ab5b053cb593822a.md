### Title
Zero-Value BTC Transfer in `transfer_to_pools_when_unstake` Permanently Freezes Staker's STRK Principal - (File: src/staking/staking.cairo)

### Summary
When a staker calls `unstake_action`, the internal `transfer_to_pools_when_unstake` function iterates over all of the staker's pools and calls `checked_transfer` with the pool's delegated balance converted to native token decimals. If a BTC pool's delegated balance is zero (because all delegators have exited intent), `to_native_amount` returns zero and `checked_transfer` is called with amount `0`. If the BTC token does not support zero-value transfers, the call reverts, permanently blocking `unstake_action` and freezing the staker's STRK principal with no recovery path.

### Finding Description

The `transfer_to_pools_when_unstake` function in `staking.cairo` iterates over every pool registered to a staker and unconditionally calls `checked_transfer` with the result of `pool_balance.to_native_amount(:decimals)`: [1](#0-0) 

The `to_native_amount` conversion performs integer division by `10^(18 - decimals)`: [2](#0-1) 

For a BTC token with 8 decimals, this divides by `10^10`. Any normalized balance below `10^10` (i.e., less than 1 satoshi in native form) rounds to zero. More critically, if all BTC delegators have previously called `exit_delegation_pool_intent` with their full amounts, the pool's delegated balance in the staking contract is already zero, making `to_native_amount` return exactly zero.

The `remove_from_delegation_pool_intent` path in the staking contract reduces the delegated balance to zero when the full amount is exited: [3](#0-2) 

`unstake_action` calls `transfer_to_pools_when_unstake` unconditionally after the staker's info is already erased from storage: [4](#0-3) 

There is no zero-amount guard before the `checked_transfer` call: [5](#0-4) 

The protocol explicitly supports BTC as a secondary token with configurable decimals (5–18): [6](#0-5) 

There is no mechanism to cancel `unstake_intent` once called, so a staker who has entered the exit window cannot escape this deadlock.

### Impact Explanation

If the BTC token does not support zero-value transfers (a known class of ERC20 tokens), `checked_transfer(pool_contract, 0)` reverts. Because `unstake_action` has no skip logic for zero-balance pools, the entire transaction reverts. The staker's STRK principal remains locked in the staking contract indefinitely with no recovery path, constituting permanent freezing of user funds.

**Allowed impact matched:** High — Permanent freezing of funds.

### Likelihood Explanation

- The protocol is designed to accept any BTC-compatible token added by the token admin; tokens that revert on zero-value transfers are a documented ERC20 variant.
- An unprivileged delegator (attacker) can be the sole member of a BTC pool, delegate any non-zero amount, then call `exit_delegation_pool_intent` for the full amount, reducing the pool's delegated balance to zero in the staking contract.
- The staker has no way to detect or prevent this before calling `unstake_action`.
- The attacker's cost is only gas; no capital is at risk.

### Recommendation

Add a zero-amount guard inside `transfer_to_pools_when_unstake` before calling `checked_transfer`:

```cairo
fn transfer_to_pools_when_unstake(...) {
    for (pool_contract, token_address) in staker_pool_info.pools {
        let pool_balance = self.get_delegated_balance(:staker_address, :pool_contract);
        let pool_dispatcher = IPoolDispatcher { contract_address: pool_contract };
        pool_dispatcher.set_staker_removed();
        self.insert_staker_delegated_balance(
            :staker_address, :pool_contract, delegated_balance: Zero::zero(),
        );
        let decimals = self.get_token_decimals(:token_address);
        let native_amount = pool_balance.to_native_amount(:decimals);
        if native_amount.is_non_zero() {  // <-- guard added
            let token_dispatcher = IERC20Dispatcher { contract_address: token_address };
            token_dispatcher.checked_transfer(
                recipient: pool_contract,
                amount: native_amount.into(),
            );
        }
    }
}
```

This mirrors the fix recommended in the original report: skip the transfer when the amount is zero, since a zero transfer carries no economic meaning and should not block the exit flow.

### Proof of Concept

1. Token admin adds a BTC token that reverts on zero-value transfers.
2. Staker calls `stake()` with STRK and enables a BTC delegation pool.
3. Attacker (sole BTC delegator) calls `enter_delegation_pool(reward_addr, 1_satoshi)` on the BTC pool.
4. Attacker calls `exit_delegation_pool_intent(1_satoshi)`, which calls `remove_from_delegation_pool_intent` on the staking contract, reducing the BTC pool's `delegated_balance` to `NormalizedAmount { amount_18_decimals: 0 }`.
5. Staker calls `unstake_intent()`. Exit window begins.
6. After the exit window, staker calls `unstake_action(staker_address)`.
7. Inside `unstake_action`, `transfer_to_pools_when_unstake` is called. For the BTC pool: `pool_balance.to_native_amount(8)` = `0 / 10^10` = `0`. `checked_transfer(btc_pool, 0)` is called on the BTC token, which reverts.
8. `unstake_action` reverts. The staker's STRK principal remains locked with no recovery path. [7](#0-6) [2](#0-1) [8](#0-7)
