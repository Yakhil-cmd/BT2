### Title
Missing Caller Authorization on `update_rewards` Allows Any Address to Steal or Freeze Block Rewards - (File: `src/staking/staking.cairo`)

### Summary

`IStakingRewardsManager.update_rewards` is specified to be callable by "Only starkware sequencer" but contains no such access control check. Any unprivileged address can call it, either to steal block rewards by front-running the sequencer with their own staker address, or to permanently freeze all yield by calling it with `disable_rewards: true` every block.

### Finding Description

The protocol spec at `docs/spec.md` line 1645 explicitly states:

> **access control**: Only starkware sequencer.

However, the implementation at `src/staking/staking.cairo` lines 1449–1507 contains no such check:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();   // only checks pause state
    let current_block_number = starknet::get_block_number();
    assert!(
        current_block_number > self.last_reward_block.read(),
        "{}",
        Error::REWARDS_ALREADY_UPDATED,
    );
    // ... no caller check ...
    self.last_reward_block.write(current_block_number);  // global, not per-staker
    if disable_rewards || self.is_pre_consensus() {
        return;
    }
    // distributes block rewards to staker_address
```

The critical design detail is that `last_reward_block` is a **global** storage variable, not per-staker. Only one call to `update_rewards` is permitted per block number. Once it is called (by anyone), the sequencer cannot call it again for the same block.

This is the direct analog to the Cooler `claimDefaulted` vulnerability: a reward-distributing function that processes caller-supplied inputs without validating the caller's authorization, allowing an attacker to hijack the reward distribution for a block.

### Impact Explanation

**Attack 1 — Permanent Freezing of Yield (High)**

An attacker calls `update_rewards(any_valid_active_staker, disable_rewards: true)` on every block. The function:
1. Passes the pause check.
2. Passes the `REWARDS_ALREADY_UPDATED` check (new block).
3. Writes `current_block_number` to `last_reward_block`.
4. Returns immediately because `disable_rewards == true`.

No rewards are ever distributed. The sequencer's subsequent call for the same block reverts with `REWARDS_ALREADY_UPDATED`. All staker and delegator yield is permanently frozen.

**Attack 2 — Theft of Block Rewards (High)**

An attacker who is a registered staker front-runs the sequencer's `update_rewards` call each block with their own `staker_address` and `disable_rewards: false`. The block rewards (computed from the minting curve) are credited to the attacker's staker account instead of whichever staker the sequencer intended to reward. Over many blocks, the attacker accumulates rewards that should have been distributed across all stakers proportionally.

### Likelihood Explanation

- The function is publicly callable with no access control.
- The attacker only needs to be a registered staker (for Attack 2) or any address (for Attack 1).
- On Starknet, transaction ordering within a block is deterministic and front-running is feasible by submitting a transaction in the same block before the sequencer's own transaction.
- No privileged access, leaked keys, or external dependencies are required.

### Recommendation

Add a caller check at the top of `update_rewards` that asserts `get_caller_address()` equals the configured sequencer address (or the attestation contract, depending on the intended caller). For example:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
    assert!(
        get_caller_address() == self.sequencer_address.read(),
        "{}",
        Error::CALLER_IS_NOT_SEQUENCER,
    );
    // ...
}
```

### Proof of Concept

1. Consensus rewards are active (post `consensus_rewards_first_epoch`).
2. Attacker (any address) monitors the mempool/block production.
3. Each block, attacker calls `update_rewards(some_valid_staker, disable_rewards: true)`.
4. `last_reward_block` is updated to the current block number.
5. The legitimate sequencer's `update_rewards` call reverts with `REWARDS_ALREADY_UPDATED`.
6. No staker or delegator ever accumulates `unclaimed_rewards_own`, and no pool ever receives STRK transfers.
7. All yield is permanently frozen.
