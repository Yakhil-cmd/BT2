### Title
`EVM.getLatestBlock()` Returns Understated `totalSupply` Due to Missing In-Progress Block Proposal Accumulation - (File: `fvm/evm/stdlib/contract.cdc`)

---

### Summary

`EVM.getLatestBlock()` reads `totalSupply` exclusively from the last **committed** EVM block (`LatestBlock` storage key). During a Flow block's execution, EVM deposits and withdrawals accumulate supply changes in the in-progress `BlockProposal` (flushed to `LatestBlockProposal`), but these are only promoted to `LatestBlock` at the very end of the Flow block via `CommitBlockProposal()` (the heartbeat system chunk). Any Cadence transaction or script that calls `EVM.getLatestBlock()` after EVM deposits have occurred within the same Flow block will receive an understated `totalSupply`, missing all supply changes from the current in-progress EVM block.

---

### Finding Description

The call chain for `EVM.getLatestBlock()` is:

1. `EVM.getLatestBlock()` in `fvm/evm/stdlib/contract.cdc` delegates to `InternalEVM.getLatestBlock()`.
2. `newInternalEVMTypeGetLatestBlockFunction` in `fvm/evm/impl/impl.go` calls `handler.LastExecutedBlock()`.
3. `ContractHandler.LastExecutedBlock()` in `fvm/evm/handler/handler.go` calls `h.backend.LatestBlock()`.
4. `BlockStore.LatestBlock()` in `fvm/environment/evm_block_store.go` reads the `LatestBlock` storage key.

The `LatestBlock` key is **only written** inside `CommitBlockProposal()`, which is called once per Flow block by the heartbeat system chunk at the very end of block execution.

During a Flow block, every EVM deposit or withdrawal updates `bp.TotalSupply` in the in-progress `BlockProposal` and stages it via `StageBlockProposal()` / `FlushBlockProposal()` into the `LatestBlockProposal` storage key — **not** into `LatestBlock`. Therefore, any call to `EVM.getLatestBlock()` made before the heartbeat commits the block will return the previous EVM block's `totalSupply`, silently omitting all supply changes accumulated so far in the current Flow block.

This is structurally identical to the `getOutstandingDebt()` bug: a public read function returns a sum that excludes a pending/accrued component (accrued interest there; staged supply delta here).

---

### Impact Explanation

`EVM.getLatestBlock().totalSupply` is the only on-chain mechanism for Cadence contracts and scripts to query the total amount of FLOW bridged into the EVM environment. Any bridge contract, DeFi protocol, or off-chain indexer that reads this value within the same Flow block as EVM deposits will observe an understated total supply. This can cause:

- Incorrect financial decisions by Cadence contracts that gate withdrawals, fee calculations, or reward distributions on the reported EVM total supply.
- Misleading data returned to users and off-chain systems querying the EVM state mid-block.

The impact matches the analog: a public accounting function returns a misleading, understated value because it does not include the pending/accrued component from the current evaluation period.

**Impact: Medium** (same class as the original finding, rated Impact 3).

---

### Likelihood Explanation

The condition is trivially reachable: any Cadence transaction that (a) executes after an EVM deposit in the same Flow block and (b) calls `EVM.getLatestBlock()` will observe the understated value. No special privileges are required. `EVM.getLatestBlock()` is `access(all)` and callable by any unprivileged transaction sender or script author.

**Likelihood: High** — the discrepancy exists for the entire duration of every Flow block that contains at least one EVM deposit or withdrawal, which is the common case on mainnet.

---

### Recommendation

`EVM.getLatestBlock()` should be updated to read from the current in-progress `BlockProposal` (which accumulates all pending supply changes) rather than from the last committed `LatestBlock`. Concretely, `ContractHandler.LastExecutedBlock()` should call `h.backend.BlockProposal()` and return its embedded `Block` field (which carries the running `TotalSupply`), or a dedicated `CurrentBlockProposal()` accessor should be exposed and wired into `newInternalEVMTypeGetLatestBlockFunction`. This mirrors the fix applied to `getOutstandingDebt()`: replacing the stale committed-state read with a function that includes the pending/accrued component.

---

### Proof of Concept

**Step-by-step:**

1. Flow block K begins execution.
2. **Cadence tx 1**: An EVM deposit of 100 FLOW is executed. Inside `executeAndHandleCall`, `bp.TotalSupply` is incremented by 100 and `StageBlockProposal(bp)` is called; at tx end, `FlushBlockProposal()` persists the updated proposal to the `LatestBlockProposal` storage key. `LatestBlock` is **not** updated.
3. **Cadence tx 2** (or a script): calls `EVM.getLatestBlock()`. The call resolves to `BlockStore.LatestBlock()`, which reads the `LatestBlock` key — the previous EVM block's data. The returned `totalSupply` is the value from before the deposit, understated by 100 FLOW.
4. At the end of Flow block K, the heartbeat system chunk calls `CommitBlockProposal()`, which writes the accumulated `BlockProposal` (including the 100 FLOW deposit) to `LatestBlock`. Only after this point does `EVM.getLatestBlock()` return the correct supply.

```go
// CommitBlockProposal commits the block proposal to the chain
func (bs *BlockStore) CommitBlockProposal(bp *types.BlockProposal) error {
	bp.PopulateRoots()

	blockBytes, err := bp.Block.ToBytes()
	if err != nil {
		return types.NewFatalError(err)
	}

	err = bs.storage.SetValue(bs.rootAddress[:], []byte(BlockStoreLatestBlockKey), blockBytes)
	if err != nil {
		return err
	}

	hash, err := bp.Block.Hash()
	if err != nil {
		return err
	}

	bhl, err := bs.getBlockHashList()
	if err != nil {
		return err
	}
	err = bhl.Push(bp.Block.Height, hash)
	if err != nil {
		return err
	}

	// Construct and store the new block proposal eagerly to maintain
	// state compatibility with the previous implementation.
	newBP, err := bs.constructBlockProposal()
	if err != nil {
		return err
	}
	err = bs.updateBlockProposal(newBP)
	if err != nil {
		return err
	}
	bs.cached = nil
	return nil
}
```

```go
// LatestBlock returns the latest executed block
func (bs *BlockStore) LatestBlock() (*types.Block, error) {
	data, err := bs.storage.GetValue(bs.rootAddress[:], []byte(BlockStoreLatestBlockKey))
	if err != nil {
		return nil, err
	}
	if len(data) == 0 {
		return types.GenesisBlock(bs.chainID), nil
	}
	return types.NewBlockFromBytes(data)
}
```
