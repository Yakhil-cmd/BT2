## Analysis

Let me trace the full execution path for this scenario.

**Step 1: P2P entry → Gateway**

`MempoolP2pRunner::start` receives the P2P transaction and unconditionally forwards it to the gateway via `gateway_client.add_tx(GatewayInput { rpc_tx, message_metadata })` — no P2P-specific filtering occurs. [1](#0-0) 

**Step 2: Gateway stateful path**

`extract_state_nonce_and_run_validations` reads `account_nonce` from state (=0), then calls `run_pre_validation_checks`, then `run_validate_entry_point(executable_tx, skip_validate)`. [2](#0-1) 

**Step 3: `run_pre_validation_checks` sequence**

Three sub-checks run in order:
1. `validate_state_preconditions` → `validate_nonce`: nonce=1 with account_nonce=0 satisfies `0 <= 1 <= max_allowed_nonce_gap`, so it **passes**.
2. `validate_by_mempool` → `mempool.validate_tx`: checks for duplicate hash and nonce conflicts only — **no signature check**.
3. `skip_stateful_validations`: the decisive call. [3](#0-2) 

**Step 4: `skip_stateful_validations` logic**

The function checks `tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO)`. Both are true in the attacker's scenario. It then calls `account_tx_in_pool_or_recent_block(sender_address)`. If that returns `true`, the function returns `true` (skip validation). [4](#0-3) 

**Step 5: `account_tx_in_pool_or_recent_block` semantics**

The mempool implementation checks `self.state.contains_account(account_address) || self.tx_pool.contains_account(account_address)`. It returns `true` for **any** transaction from the address — not specifically a deploy_account. A legitimate deploy_account submitted by the real user satisfies this. [5](#0-4) 

**Step 6: `run_validate_entry_point` with `skip_validate=true`**

`execution_flags.validate` is set to `!skip_validate = false`. The `AccountTransaction` is passed to `blockifier_validator.validate(account_tx)`. [6](#0-5) 

**Step 7: Blockifier `perform_validations` short-circuits**

For Invoke, after `perform_pre_validation_stage`, the code checks `if !tx.execution_flags.validate { return Ok(()); }`. With `validate=false`, it returns immediately — `__validate__` is never called. [7](#0-6) 

**Step 8: `validate_tx` also short-circuits**

Even if reached, `validate_tx` has its own guard: `if !self.execution_flags.validate { return Ok(None); }`. Signature verification is completely bypassed. [8](#0-7)

### Title
Invalid-Signature InvokeV3 Admitted to Mempool via `skip_stateful_validations` UX Bypass — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary
The deploy-account UX skip feature allows any unprivileged attacker who observes a legitimate `deploy_account` transaction in the mempool to submit an `InvokeV3` with nonce=1 and a garbage signature for the same sender address. The transaction passes all gateway checks and is admitted to the mempool without any signature verification.

### Finding Description
`skip_stateful_validations` returns `true` when:
- The incoming tx is `ExecutableTransaction::Invoke`
- `tx.nonce() == 1`
- `account_nonce == 0` (from state)
- `account_tx_in_pool_or_recent_block(sender_address)` returns `true`

When `true` is returned, `run_validate_entry_point` sets `execution_flags.validate = false`, which causes both `StatefulValidator::perform_validations` and `AccountTransaction::validate_tx` to return `Ok(())` without executing the `__validate__` entry point. The transaction is then forwarded to `mempool.add_tx` and admitted.

The attacker does not need the account's private key. They only need to observe a pending `deploy_account` for any address (publicly visible in the P2P gossip layer), then submit an `InvokeV3` from that address with nonce=1 and arbitrary bytes as the signature.

### Impact Explanation
The concrete corrupted value is: **an InvokeV3 with an invalid signature is admitted to the mempool**. This matches the "High" impact: *Mempool/gateway/RPC admission accepts invalid transactions before sequencing*.

Secondary effects:
- The attacker's invalid-signature invoke occupies the `(address, nonce=1)` slot. The legitimate user's invoke is rejected with `DuplicateNonce` unless they pay a higher tip for fee escalation.
- The attacker's transaction will fail `__validate__` during sequencing (the batcher runs full execution), but by then the legitimate user's UX is disrupted.
- Because the account has no funds at deploy time, the fee transfer for the failed tx also fails, so the attacker bears no cost. This enables free mempool spam targeting all pending deploy_account transactions visible on the P2P network.

### Likelihood Explanation
The P2P gossip layer makes all pending `deploy_account` transactions publicly observable. The attack requires no privileges, no key material, and no special tooling — only the ability to send a P2P message or RPC call with a crafted `InvokeV3`. The condition `nonce=1 && state_nonce=0 && account_in_pool` is reliably triggerable for any new account deployment.

### Recommendation
Replace the `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a `deploy_account` transaction (not just any transaction) is pending for the address. Alternatively, record the `deploy_account` tx hash at submission time and require the invoke to reference it, as the `native_blockifier` `PyValidator::should_run_stateful_validations` does via the `deploy_account_tx_hash` parameter. [9](#0-8) 

### Proof of Concept
1. Submit a valid `deploy_account` for address `A` (any class, any salt). Mempool now has `account_tx_in_pool_or_recent_block(A) == true`.
2. Submit an `InvokeV3` from address `A`, `nonce=1`, `signature=[0xdeadbeef, 0xdeadbeef]` (garbage), valid resource bounds.
3. Gateway path: `validate_nonce` passes (0 ≤ 1 ≤ gap), `validate_by_mempool` passes (no sig check), `skip_stateful_validations` returns `true`, `run_validate_entry_point` sets `validate=false`, blockifier returns `Ok(())`.
4. Transaction is admitted to the mempool. Assert: no `ValidateFailure` error is returned; the tx appears in `mempool_snapshot`.
5. Legitimate user submits their correctly-signed `InvokeV3` with `nonce=1` → rejected with `DuplicateNonce`.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
