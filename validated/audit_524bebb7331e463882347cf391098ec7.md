### Title
Missing Address Validation for L1→L2 Transaction `reserved[1]` (Refund Recipient) Causes Bootloader Panic — (`basic_bootloader/src/bootloader/transaction/abi_encoded/mod.rs`)

---

### Summary

The `validate_structure` function for ABI-encoded L1→L2 and upgrade transactions explicitly skips address validation for `reserved[1]` (the refund recipient field) with a `// TODO: validate address?` comment. Downstream, `u256_to_b160_checked` is called unconditionally on the raw `U256` value and uses `assert!` to enforce the address invariant. If an L1→L2 transaction arrives with upper bits set in `reserved[1]`, the assertion panics, producing a fatal bootloader error that halts forward execution or invalidates the proof.

---

### Finding Description

`validate_structure` in `AbiEncodedTransaction` validates many fields (paymaster must be zero, `reserved[2]`/`reserved[3]` must be zero, signature length, etc.) but explicitly defers address validation for `reserved[1]`:

```rust
// reserved[1] = refund recipient for l1 to l2 and upgrade txs
match tx_type {
    Self::L1_L2_TX_TYPE | Self::UPGRADE_TX_TYPE => {
        // TODO: validate address?
    }
    _ => unreachable!(),
}
``` [1](#0-0) 

Because `reserved[1]` is stored as a full 32-byte `U256` in the ABI encoding, a caller can supply any 256-bit value. The field is later consumed in `process_l1_transaction` whenever a non-zero refund is owed:

```rust
if to_refund_recipient > U256::ZERO {
    let refund_recipient = u256_to_b160_checked(transaction.reserved[1].read());
``` [2](#0-1) 

`u256_to_b160_checked` enforces the address invariant with a hard `assert!`:

```rust
pub fn u256_to_b160_checked(src: U256) -> B160 {
    assert!(src.as_limbs()[3] == 0 && src.as_limbs()[2] < (1u64 << 32));
``` [3](#0-2) 

If `reserved[1]` has any of the upper 96 bits set, the `assert!` fires. In Rust, `assert!` panics in both debug and release builds. A safer alternative already exists in the same file — `u256_try_to_b160` — which returns `Option<B160>` and allows graceful rejection: [4](#0-3) 

---

### Impact Explanation

A panic inside the bootloader is a fatal, unrecoverable error:

- **Forward/sequencer mode**: the sequencer process crashes, halting block production until restarted.
- **Proving mode (RISC-V)**: the RISC-V execution aborts, making the batch unprovable and blocking finalization.

In both cases the attacker achieves a denial-of-service against the ZKsync OS state-transition function. Because the refund amount is almost always non-zero for any L1→L2 transaction that uses less than its full gas limit, the vulnerable branch (`to_refund_recipient > U256::ZERO`) is reached in the common case.

---

### Likelihood Explanation

Any party that can submit an L1→L2 priority transaction controls the `reserved[1]` word. The ABI encoding places the refund recipient as a raw `U256` slot; nothing in ZKsync OS's own parsing pipeline rejects a value with upper bits set before `u256_to_b160_checked` is reached. The `validate_structure` function's own comment acknowledges the gap (`// TODO: validate address?`). Whether the upstream L1 Mailbox contract independently enforces a clean 20-byte address is outside ZKsync OS's control and outside the scope of this codebase; ZKsync OS must not rely on that assumption.

---

### Recommendation

1. In `validate_structure`, add an explicit check that `reserved[1]` fits in 160 bits for `L1_L2_TX_TYPE` and `UPGRADE_TX_TYPE`, rejecting the transaction with `Err(())` if not.
2. Replace the `u256_to_b160_checked` call in `process_l1_transaction` with `u256_try_to_b160`, propagating an `InvalidTransaction` error instead of panicking.

---

### Proof of Concept

1. Craft an L1→L2 transaction where `reserved[1]` = `U256::MAX` (all bits set).
2. Submit it to the L1 bridge so it enters the priority queue.
3. When ZKsync OS processes the block containing this transaction:
   - `validate_structure` passes (the TODO branch does nothing).
   - The transaction executes; unused gas produces a non-zero `to_refund_recipient`.
   - `u256_to_b160_checked(U256::MAX)` fires `assert!(0 == 0 && 0xffffffff_ffffffff < (1u64 << 32))` → `false` → **panic**.
4. The bootloader aborts; forward execution halts or the proof is invalidated.

pub fn u256_try_to_b160(src: U256) -> Option<B160> {
    if src.as_limbs()[3] != 0 || src.as_limbs()[2] >= (1u64 << 32) {
        return None;
    }
    let mut result = B160::ZERO;
    unsafe {
        result.as_limbs_mut()[0] = src.as_limbs()[0];
        result.as_limbs_mut()[1] = src.as_limbs()[1];
        result.as_limbs_mut()[2] = src.as_limbs()[2];
    }

    Some(result)
}
