### Title
Gateway `skip_stateful_validations` Bypasses `__validate__` for Any Account with a Pending Mempool Entry, Admitting Unsigned Invoke Transactions - (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator is intended to skip the `__validate__` entry-point check for the UX case where a user submits a `deploy_account` + `invoke(nonce=1)` pair simultaneously. However, the guard condition uses `account_tx_in_pool_or_recent_block`, which returns `true` for **any** transaction from that address in the mempool — not specifically a `deploy_account`. An attacker who observes a victim's pending `deploy_account` in the mempool can race-submit an `invoke(nonce=1)` with an invalid/arbitrary signature for the victim's address, and the gateway will admit it without running `__validate__`.

### Finding Description

In `extract_state_nonce_and_run_validations`, the gateway:
1. Reads `account_nonce` from state (= `0` for an undeployed account).
2. Calls `run_pre_validation_checks`, which calls `validate_by_mempool` (mempool-level duplicate/nonce check) and then `skip_stateful_validations`.
3. Calls `run_validate_entry_point` — **skipped** if `skip_validate == true`. [1](#0-0) 

`skip_stateful_validations` fires when `tx_nonce == 1 && account_nonce == 0` and `account_tx_in_pool_or_recent_block` returns `true`: [2](#0-1) 

The mempool's implementation of `account_tx_in_pool_or_recent_block` returns `true` if the address has **any** transaction in the pool or has been seen in a committed block — it does not distinguish a `deploy_account` from an `invoke`: [3](#0-2) 

The comment in `skip_stateful_validations` asserts this is safe because "it means that either it has a deploy_account transaction or transactions with future nonces that passed validations." The second branch of that disjunction is circular: future-nonce invokes for an undeployed account cannot pass `__validate__` (no contract code), so they cannot be in the pool. The only realistic path to `account_tx_in_pool_or_recent_block == true` with `account_nonce == 0` is a pending `deploy_account`. But the check does not enforce this — it accepts the presence of **any** transaction from that address.

**Attack path:**

1. Victim submits `deploy_account(nonce=0)` for address `X`. It enters the mempool.
2. Attacker observes this (mempool is observable). Attacker crafts `invoke(nonce=1, sender=X, signature=<garbage>)`.
3. Gateway stateless validator passes (signature size is valid, resource bounds non-zero).
4. `get_nonce_from_state(X)` → `0`.
5. `validate_state_preconditions`: nonce `0 ≤ 1 ≤ max_allowed_nonce_gap` → passes; resource bounds → passes.
6. `validate_by_mempool`: no duplicate nonce=1 for `X` yet → passes.
7. `skip_stateful_validations`: `tx_nonce==1`, `account_nonce==0`, `account_tx_in_pool_or_recent_block(X)==true` (victim's `deploy_account` is in pool) → returns `true`.
8. `run_validate_entry_point` is **skipped**.
9. Attacker's unsigned invoke is forwarded to `mempool_client.add_tx`. [4](#0-3) 

### Impact Explanation

The gateway admits an `invoke` transaction with an invalid/arbitrary signature to the mempool without running the account's `__validate__` entry point. This matches the **High** impact category: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

Secondary consequence: when the batcher executes the block, the blockifier **does** call `__validate__` (execution flags default to `validate=true`). The transaction fails and is reverted. However, `perform_pre_validation_stage` increments the nonce **before** `__validate__` runs, and that increment is not rolled back on revert: [5](#0-4) [6](#0-5) 

If the attacker's invalid invoke is executed before the victim's legitimate `invoke(nonce=1)`, the victim's account nonce is bumped to `2`, permanently invalidating the victim's pending transaction. This is a targeted denial-of-service against any account in the deploy-account + invoke UX flow.

### Likelihood Explanation

- The mempool is observable on a public network; pending `deploy_account` transactions are visible.
- The attacker only needs to submit a single malformed invoke before the victim submits their own invoke.
- No privileged access, no special contract, no cryptographic capability is required — only knowledge of the victim's contract address (which is deterministic from the deploy parameters and is included in the `deploy_account` transaction itself).
- The race window is the time between the victim's `deploy_account` entering the mempool and the victim's `invoke(nonce=1)` entering the mempool.

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a **`deploy_account`** transaction is pending for the address. Add a dedicated mempool query such as `has_pending_deploy_account(address) -> bool` that inspects the transaction type in the pool, and use that in `skip_stateful_validations` instead of the generic presence check. [2](#0-1) 

### Proof of Concept

```
1. Victim calls gateway: POST /add_transaction
   { type: DEPLOY_ACCOUNT, class_hash: C, salt: S, nonce: 0, signature: <valid> }
   → Admitted to mempool. account_tx_in_pool_or_recent_block(X) = true.

2. Attacker calls gateway: POST /add_transaction
   { type: INVOKE, sender_address: X, nonce: 1, calldata: [...], signature: [0xdead, 0xbeef] }

3. Gateway stateful validator:
   account_nonce = get_nonce_from_state(X) = 0
   validate_state_preconditions: 0 ≤ 1 ≤ max_gap → OK
   validate_by_mempool: no nonce=1 for X → OK
   skip_stateful_validations: nonce==1 && account_nonce==0 && pool_check==true → skip=true
   run_validate_entry_point: SKIPPED
   → Attacker's invoke admitted to mempool.

4. Batcher builds block:
   execute deploy_account(X, nonce=0) → X deployed, nonce=1
   execute attacker's invoke(X, nonce=1) → __validate__ fails (bad sig) → reverted, nonce=2

5. Victim's invoke(X, nonce=1) → rejected: InvalidNonce (account_nonce=2, tx_nonce=1)
```

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}

    fn handle_nonce(
        state: &mut dyn State,
        tx_info: &TransactionInfo,
        strict: bool,
    ) -> TransactionPreValidationResult<()> {
        if tx_info.is_v0() {
            return Ok(());
        }

        let address = tx_info.sender_address();
        let account_nonce = state.get_nonce_at(address)?;
        let incoming_tx_nonce = tx_info.nonce();
        let valid_nonce = if strict {
            account_nonce == incoming_tx_nonce
        } else {
            account_nonce <= incoming_tx_nonce
        };
        if valid_nonce {
            return Ok(state.increment_nonce(address)?);
        }
        Err(TransactionPreValidationError::InvalidNonce {
            address,
            account_nonce,
            incoming_tx_nonce,
        })
    }
