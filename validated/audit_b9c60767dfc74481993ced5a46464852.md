### Title
Missing Caller Access Control on `update_rewards` Allows Any Address to Suppress Block Rewards — (`src/staking/staking.cairo`)

---

### Summary

`update_rewards` is documented as callable only by the Starkware sequencer, but the implementation enforces no such restriction. The function's only gate is a global, single-slot `last_reward_block` variable. Any unprivileged caller can invoke `update_rewards(any_valid_staker, disable_rewards: true)`, consume the per-block reward slot, and permanently discard that block's consensus rewards for all stakers.

---

### Finding Description

The spec for `update_rewards` states:

> **access control**: Only starkware sequencer. [1](#0-0) 

The actual implementation delegates all access control to `general_prerequisites()`: [2](#0-1) 

`general_prerequisites()` only checks that the contract is unpaused and the caller is non-zero — there is no sequencer identity check: [3](#0-2) 

The reward-slot gate is a **global** single `BlockNumber` storage variable, not per-staker: [4](#0-3) 

The gate check and write: [5](#0-4) [6](#0-5) 

When `disable_rewards: true` is passed, `last_reward_block` is still written to the current block number, but the function returns before distributing any rewards: [7](#0-6) 

Because `last_reward_block` is global and written unconditionally before the `disable_rewards` branch, a single call with `disable_rewards: true` permanently exhausts the reward slot for that block number. No subsequent call — including a legitimate sequencer call — can succeed for the same block:

```
current_block_number > self.last_reward_block.read()  // false → REWARDS_ALREADY_UPDATED
```

---

### Impact Explanation

Each block has exactly one reward slot. Once an attacker calls `update_rewards(any_valid_staker, disable_rewards: true)` for block N, that block's rewards are permanently discarded. They cannot be retroactively recovered because `last_reward_block` is already set to N. Repeating this every block permanently freezes all consensus yield for all stakers and delegators.

**Impact**: High — Permanent freezing of unclaimed yield.

---

### Likelihood Explanation

- The function is publicly callable by any non-zero address with no role check.
- The attacker only needs to know any one valid, active staker address (publicly observable on-chain).
- On Starknet the sequencer controls intra-block ordering, which limits pure front-running within a single block. However, the sequencer is not required to call `update_rewards` in every block; any block the sequencer skips is immediately exploitable. Additionally, the missing access control means the invariant is broken by design regardless of ordering.
- The attack is cheap (one call per block, no capital required) and can be sustained indefinitely.

---

### Recommendation

Add a sequencer-identity check at the top of `update_rewards`, analogous to how other privileged functions gate on role membership. On Starknet this can be done by comparing `get_caller_address()` against `starknet::get_execution_info().unbox().block_info.unbox().sequencer_address`, or by storing the authorized sequencer address in contract storage and asserting equality. The check must be placed **before** the `last_reward_block` write so that unauthorized calls revert without consuming the block's reward slot.

---

### Proof of Concept

```
// Block N begins.
// Attacker (any non-zero address) calls:
staking.update_rewards(staker_address: any_active_staker, disable_rewards: true);
// → last_reward_block is now N; no rewards distributed.

// Sequencer (or anyone else) tries to distribute rewards for block N:
staking.update_rewards(staker_address: victim_staker, disable_rewards: false);
// → panics: REWARDS_ALREADY_UPDATED (N > N is false)

// Repeat every block → all consensus rewards permanently suppressed.
```

Stand up two active validators, have an attacker call `update_rewards(..., disable_rewards: true)` at the start of every block, and compare cumulative `unclaimed_rewards_own` against a baseline where the sequencer calls with `disable_rewards: false` each block. The attacker-run scenario will show zero accrual.
