### Title
Commission Commitment Expires One Epoch Early Due to Strict Boundary in `is_commission_commitment_active` - (File: `src/staking/staking.cairo`)

---

### Summary

`is_commission_commitment_active` uses a strict `<` comparison, causing the commitment to be treated as expired **at** `expiration_epoch` rather than **after** it. This lets a staker replace the expired commitment with a new one carrying a higher `max_commission` and immediately raise their commission — all within `expiration_epoch` — one epoch earlier than delegators expect.

---

### Finding Description

`is_commission_commitment_active` is defined as:

```cairo
fn is_commission_commitment_active(
    self: @ContractState, commission_commitment: CommissionCommitment,
) -> bool {
    self.get_current_epoch() < commission_commitment.expiration_epoch
}
``` [1](#0-0) 

When `current_epoch == expiration_epoch`, the expression `current_epoch < expiration_epoch` evaluates to `false`, so the commitment is classified as **expired** at that exact epoch.

`set_commission_commitment` enforces `expiration_epoch > current_epoch`:

```cairo
assert!(expiration_epoch > current_epoch, "{}", Error::EXPIRATION_EPOCH_TOO_EARLY);
``` [2](#0-1) 

And it only blocks creating a new commitment when the existing one is **still active**:

```cairo
assert!(
    !self.is_commission_commitment_active(:commission_commitment),
    "{}",
    Error::COMMISSION_COMMITMENT_EXISTS,
);
``` [3](#0-2) 

Because the old commitment is already expired at `expiration_epoch`, this guard passes, and the staker can immediately write a new commitment with an arbitrarily higher `max_commission` (up to `COMMISSION_DENOMINATOR` = 100 %).

`update_commission` then permits raising commission to the new `max_commission` because the freshly written commitment is active (`current_epoch < expiration_epoch_new`):

```cairo
if self.is_commission_commitment_active(:commission_commitment) {
    assert!(commission <= commission_commitment.max_commission, ...);
    assert!(commission != old_commission, ...);
} else {
    assert!(commission < old_commission, "{}", Error::COMMISSION_COMMITMENT_EXPIRED);
}
``` [4](#0-3) 

The full two-step sequence — `set_commission_commitment` then `set_commission` — can be executed in two transactions within the same epoch, before the staker's attestation for that epoch is submitted.

---

### Impact Explanation

**High — Theft of unclaimed yield.**

Delegators join a pool trusting that commission will not exceed `max_commission` through `expiration_epoch` (inclusive). Because the commitment is already expired at that epoch, the staker can raise commission to up to 100 % before attesting. The attestation call triggers `update_rewards_from_attestation_contract`, which distributes epoch rewards using the now-elevated commission rate. Delegators receive materially less yield for epoch `expiration_epoch` than the commitment led them to expect, with the difference captured by the staker's reward address.

---

### Likelihood Explanation

**Medium.** Any staker who has set a commission commitment can execute this in two ordinary, permissionless transactions at the start of `expiration_epoch`. No privileged access, leaked key, or external dependency is required. The only prerequisite is that the staker has an active pool with delegators — a normal operating condition. The staker has a direct financial incentive (higher commission income) to exploit the boundary.

---

### Recommendation

Change the comparison in `is_commission_commitment_active` from strict `<` to `<=` so the commitment remains active **through** `expiration_epoch`:

```cairo
fn is_commission_commitment_active(
    self: @ContractState, commission_commitment: CommissionCommitment,
) -> bool {
    self.get_current_epoch() <= commission_commitment.expiration_epoch
}
```

This ensures the commitment cannot be replaced, and commission cannot be raised, until `current_epoch > expiration_epoch`.

---

### Proof of Concept

1. Staker calls `set_commission_commitment(max_commission: X, expiration_epoch: E)` at epoch `E-2`. Delegators observe the event and join the pool expecting commission ≤ `X` through epoch `E`.

2. At the first block of epoch `E`:
   - `is_commission_commitment_active` returns `false` (`E < E` → `false`).
   - Staker calls `set_commission_commitment(max_commission: COMMISSION_DENOMINATOR, expiration_epoch: E+1)`. The guard at line 762–766 passes because the old commitment is expired.
   - Staker calls `set_commission(COMMISSION_DENOMINATOR)`. `update_commission` sees the new commitment as active (`E < E+1`) and allows the increase.

3. Staker's operational address calls `attest(block_hash)`. `update_rewards_from_attestation_contract` calculates epoch `E` rewards using commission = 100 %, sending zero yield to the pool and all rewards to the staker's own reward address.

4. Delegators' `claim_rewards` for epoch `E` returns far less than expected — or zero — constituting theft of unclaimed yield.
