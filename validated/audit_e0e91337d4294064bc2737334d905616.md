Audit Report

## Title
SNS Treasury Manager `withdraw` Structurally Prevents Slippage Protection, Enabling Sandwich Attacks on DAO Treasury Withdrawals - (`rs/sns/governance/src/extensions.rs`, `rs/sns/treasury_manager/src/lib.rs`)

## Summary
The SNS Treasury Manager's `withdraw` path contains no mechanism for specifying minimum acceptable token amounts. The governance validation layer enforces this gap structurally: `TryFrom<Option<Precise>> for ValidatedWithdrawOperationArg` actively rejects any non-empty argument with a hard error, making it impossible to pass slippage constraints at the proposal level. `execute_treasury_manager_withdraw` then calls the DEX-backed treasury manager and returns `Ok(())` regardless of how few tokens are returned. An unprivileged DEX participant can exploit the multi-day governance voting window to manipulate the pool price and drain value from the SNS DAO treasury upon withdrawal.

## Finding Description
**Root cause — no minimum-amount fields in `WithdrawRequest`:**
`rs/sns/treasury_manager/src/lib.rs` lines 302–306 define `WithdrawRequest` with only a `withdraw_accounts` field and no `min_sns_e8s`, `min_icp_e8s`, or equivalent. [1](#0-0) 

**Root cause — governance layer actively rejects all withdraw arguments:**
`TryFrom<Option<Precise>> for ValidatedWithdrawOperationArg` in `rs/sns/governance/src/extensions.rs` lines 1739–1743 returns a hard error for any non-`None` value, making it structurally impossible to pass minimum-amount constraints through a governance proposal. [2](#0-1) 

**Root cause — no post-execution minimum check:**
`execute_treasury_manager_withdraw` (lines 1612–1661) calls the treasury manager's `withdraw` endpoint, decodes the `TreasuryManagerResult`, and returns `Ok(())` after logging success. It performs no comparison of returned balances against any minimum threshold. [3](#0-2) 

**Contrast with deposit path:**
`validate_deposit_operation_impl` (lines 276–321) checks treasury balances at proposal validation time and enforces a 50%-of-balance cap. No analogous check exists for withdrawals at any stage. [4](#0-3) 

**Developer acknowledgment of the risk class — deposits only:**
`treasury_manager.did` lines 35–40 explicitly document slippage risk for deposits but make no mention of the same risk for withdrawals. [5](#0-4) 

The same deposit-only warning appears in `rs/sns/governance/src/proposal.rs` lines 1542–1545. [6](#0-5) 

**Exploit flow:**
1. An SNS DAO submits `ExecuteExtensionOperation { operation_name: "withdraw", operation_arg: { value: None } }`. Any attempt to include minimum-amount fields is rejected by `validate_withdraw_operation` → `ValidatedWithdrawOperationArg::try_from` with `"Withdraw operation does not accept arguments at this time"`.
2. The proposal enters the voting period (typically 4+ days). During this window, an attacker executes large swaps on the DEX pool, shifting the SNS/ICP price ratio against the treasury.
3. The proposal passes and `execute_treasury_manager_withdraw` calls `extension_canister_id.withdraw(WithdrawRequest { withdraw_accounts: None })` with no minimum-amount constraint.
4. The DEX returns tokens at the manipulated price. Governance logs success and returns `Ok(())` regardless of the shortfall.
5. The SNS treasury receives materially fewer tokens than it deposited, with no on-chain recourse.

## Impact Explanation
This is a **High** severity finding. An unprivileged DEX participant can cause significant, concrete financial loss to an SNS DAO treasury — specifically the `treasury_owner` balances returned by the treasury manager — by front-running or sandwiching a governance-approved withdrawal. The loss accrues to all token holders of the affected SNS. This matches the allowed impact: *"Significant SNS security impact with concrete user or protocol harm"* and *"Unauthorized access to... canister-controlled funds where exploitation requires meaningful per-target work or other constraints."* The constraint here is the capital required to move the DEX pool price, which bounds severity below Critical but does not negate the impact.

## Likelihood Explanation
The attack window is the entire governance voting period — days to weeks — during which any unprivileged, well-capitalized DEX participant can execute large swaps to shift the pool price ratio. No special privileges, governance access, or protocol-level compromise are required. The IC's asynchronous execution model means the withdrawal executes at whatever price the pool holds at execution time. The `treasury_manager.did` itself acknowledges this exact risk class for deposits, confirming developer awareness of the attack vector. The withdrawal path has no analogous protection and the governance layer structurally prevents adding it.

## Recommendation
1. Add `min_sns_e8s: Option<u64>` and `min_icp_e8s: Option<u64>` fields to `WithdrawRequest` in `rs/sns/treasury_manager/src/lib.rs` and `treasury_manager.did`.
2. Remove the blanket argument rejection in `TryFrom<Option<Precise>> for ValidatedWithdrawOperationArg` (`rs/sns/governance/src/extensions.rs` lines 1739–1743) and instead parse and validate minimum-amount fields.
3. In `execute_treasury_manager_withdraw`, decode the returned `BalanceBook` and verify that `treasury_owner` balances meet the minimums specified in the proposal before returning `Ok(())`.
4. Extend the "Known Security Risks" comment in `treasury_manager.did` to cover withdrawals, not just deposits.

## Proof of Concept
**Deterministic integration test plan (using PocketIC or StateMachine):**
1. Deploy an SNS with a treasury manager extension connected to a mock DEX pool holding SNS tokens and ICP.
2. Submit a deposit proposal and execute it; confirm tokens are in the pool.
3. Submit a withdraw proposal (`operation_arg: { value: None }`). Confirm that any attempt to pass `{ value: Some(...) }` is rejected with `"Withdraw operation does not accept arguments at this time"`.
4. Advance the state machine clock to simulate the voting period. During this window, call the mock DEX's swap function to shift the price ratio (e.g., drain SNS tokens from the pool).
5. Adopt and execute the withdraw proposal.
6. Assert that the SNS treasury received fewer SNS tokens than it deposited, and that governance returned `Ok(())` with no error — confirming the loss is silent and undetected by the protocol.

async fn validate_deposit_operation_impl(
    governance: &Governance,
    value: Option<Precise>,
) -> Result<ValidatedDepositOperationArg, String> {
    let structurally_valid = ValidatedDepositOperationArg::try_from(value)?;

    let sns_subaccount = governance.sns_treasury_subaccount();
    let icp_subaccount = governance.icp_treasury_subaccount();

    // Fail if either is asking for more than 50% of current balance.  The balance could have changed
    // since the proposal was created, and we don't assume that the proposal should work
    let sns_balance = governance
        .ledger
        .account_balance(Account {
            owner: governance.env.canister_id().get().0,
            subaccount: sns_subaccount,
        })
        .await
        .map_err(|e| format!("Failed to get SNS treasury balance: {e:?}"))?;
    let icp_balance = governance
        .nns_ledger
        .account_balance(Account {
            owner: governance.env.canister_id().get().0,
            subaccount: icp_subaccount,
        })
        .await
        .map_err(|e| format!("Failed to get ICP treasury balance: {e:?}"))?;

    let icp_requested = Tokens::from_e8s(structurally_valid.treasury_allocation_icp_e8s);
    let sns_requested = Tokens::from_e8s(structurally_valid.treasury_allocation_sns_e8s);

    // Unwrap is safe, only fails if divisor is zero, which we don't do.
    if sns_requested > sns_balance.checked_div(2).unwrap() {
        return Err(format!(
            "SNS treasury deposit request of {sns_requested} exceeds 50% of current SNS Token balance of {sns_balance}"
        ));
    }

    if icp_requested > icp_balance.checked_div(2).unwrap() {
        return Err(format!(
            "ICP treasury deposit request of {icp_requested} exceeds 50% of current ICP balance of {icp_balance}"
        ));
    }

    Ok(structurally_valid)
}

/// Execute a treasury manager withdraw operation
async fn execute_treasury_manager_withdraw(
    governance: &Governance,
    extension_canister_id: CanisterId,
    arg: ValidatedWithdrawOperationArg,
) -> Result<(), GovernanceError> {
    let arg_blob = construct_treasury_manager_withdraw_payload(arg.original).map_err(|err| {
        GovernanceError::new_with_message(
            ErrorType::PreconditionFailed,
            format!("Failed to construct treasury manager withdraw payload: {err}"),
        )
    })?;

    let balances = governance
        .env
        .call_canister(extension_canister_id, "withdraw", arg_blob)
        .await
        .map_err(|(code, err)| {
            GovernanceError::new_with_message(
                ErrorType::External,
                format!(
                    "Canister method call {extension_canister_id}.withdraw failed with code {code:?}: {err}"
                ),
            )
        })
        .and_then(|blob| {
            Decode!(&blob, sns_treasury_manager::TreasuryManagerResult).map_err(|err| {
                GovernanceError::new_with_message(
                    ErrorType::External,
                    format!(
                        "Error decoding TreasuryManager.withdraw response: {err:?}"
                    ),
                )
            })
        })?
        .map_err(|err| {
            GovernanceError::new_with_message(
                ErrorType::External,
                format!("TreasuryManager.withdraw failed: {err:?}"),
            )
        })?;

    log!(
        INFO,
        "TreasuryManager.withdraw succeeded with response: {:?}",
        balances
    );

    Ok(())
}

impl TryFrom<Option<Precise>> for ValidatedWithdrawOperationArg {
    type Error = String;

    fn try_from(value: Option<Precise>) -> Result<Self, Self::Error> {
        let original = value.unwrap_or_default();

        // For now, only allow empty arguments
        // This ensures withdraw operations don't accept parameters yet
        if original.value.is_some() {
            return Err("Withdraw operation does not accept arguments at this time".to_string());
        }

        Ok(Self { original })
    }
}
