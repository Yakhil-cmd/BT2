### Title
No Validation of Caller-Supplied `entropy` in `RerandomizationArguments` Enables Deterministic Presignature Rerandomization - (File: src/ecdsa/mod.rs)

### Summary

`RerandomizationArguments::new()` accepts a caller-supplied `entropy: [u8; 32]` field with no validation whatsoever. The library documentation explicitly requires this value to be "Fresh, Unpredictable, and Public," but nothing prevents a malicious coordinator or library caller from supplying `[0u8; 32]`. When entropy is zero (or any fixed, predictable value), the rerandomization scalar `delta` becomes fully deterministic from public parameters, nullifying the Wagner/ROS attack protection that rerandomization is designed to provide.

### Finding Description

`RerandomizationArguments` holds an `entropy` field that is used as the IKM (Input Key Material) for HKDF to derive the rerandomization scalar `delta`:

```
delta = HKDF-SHA3-256(ikm=entropy, salt=SALT, info=pk||tweak||msg_hash||R||participants)
``` [1](#0-0) 

The `new()` constructor accepts any 32-byte value without any check: [2](#0-1) 

`derive_randomness()` immediately uses `self.entropy` as HKDF IKM with no prior validation: [3](#0-2) 

When `entropy = [0u8; 32]`, `delta` is entirely determined by the public parameters `(pk, tweak, msg_hash, R, participants)`. An attacker who controls the entropy input (e.g., a malicious coordinator) can compute `delta` for any future signing session before it begins, because all other inputs are public.

This pattern is already demonstrated in production test utilities: [4](#0-3) 

The `rerandomize_presign()` functions in both ECDSA variants call `derive_randomness()` directly with no entropy check: [5](#0-4) [6](#0-5) 

### Impact Explanation

The rerandomization is explicitly adopted from [GS21] to make the Wagner/ROS attack "practically infeasible": [7](#0-6) 

The Wagner attack on threshold ECDSA allows an adversary who can initiate many signing sessions to extract the aggregate private key. The security proof requires `delta` to be unpredictable before each session. When `entropy = [0u8; 32]`, `delta` is fully determined by public parameters, so the adversary can:

1. Pre-compute `delta_i` for any desired `(msg_hash_i, R, pk, tweak, participants)` combination.
2. Select message hashes that produce algebraically favorable `delta_i` values.
3. Mount the Wagner attack across ~2^32 signing sessions (feasible for secp256k1's 256-bit field) to reconstruct the aggregate private key.

**Impact: High** — corruption of the presign/sign security guarantee (the rerandomization no longer provides its documented protection), with a realistic escalation path to **Critical** (extraction of aggregate secret material / unauthorized threshold signature creation) for a well-resourced attacker operating a signing oracle.

### Likelihood Explanation

**Medium.** The attacker must control the `entropy` value passed to `RerandomizationArguments::new()`. In practice this is the coordinator or the application layer that initiates signing. A malicious coordinator is an explicitly considered threat in this library's trust model. The attack additionally requires many signing sessions, but signing oracles that sign arbitrary messages are a common deployment pattern for this type of library.

### Recommendation

Add an explicit entropy quality check in `RerandomizationArguments::new()` or `derive_randomness()`. At minimum, reject all-zero entropy:

```rust
pub fn new(..., entropy: [u8; 32]) -> Result<Self, InitializationError> {
    if entropy == [0u8; 32] {
        return Err(InitializationError::BadParameters(
            "entropy must not be all-zeros; supply fresh randomness".to_string(),
        ));
    }
    Ok(Self { pk, tweak, msg_hash, big_r, participants, entropy })
}
```

A stronger approach is to have the library generate entropy internally from a `CryptoRngCore` rather than accepting it from the caller, removing the misuse surface entirely. If caller-supplied entropy must be retained (e.g., for deterministic testing), document the zero-entropy case as explicitly unsupported and enforce it at the API boundary.

### Proof of Concept

```rust
// Malicious coordinator supplies zero entropy
let entropy = [0u8; 32];
let rerand_args = RerandomizationArguments::new(pk, tweak, msg_hash, big_r, participants, entropy);

// delta is now fully deterministic from public parameters:
// delta = HKDF(ikm=[0u8;32], salt=SALT, info=pk||tweak||msg_hash||R||participants)
// An attacker can pre-compute this for any (msg_hash, R) pair before the session starts,
// enabling Wagner-attack message selection across signing sessions.
let delta = rerand_args.derive_randomness().unwrap();
// delta is non-zero (loop ensures this) but fully predictable — Wagner attack is now feasible.
``` [8](#0-7)

    pub fn rerandomize_presign(
        presignature: &PresignOutput,
        args: &RerandomizationArguments,
    ) -> Result<Self, ProtocolError> {
        if presignature.big_r != args.big_r {
            return Err(ProtocolError::IncompatibleRerandomizationInputs);
        }
        let delta = args.derive_randomness()?;
        if delta.is_zero().into() {
            return Err(ProtocolError::ZeroScalar);
        }

        // cannot be zero due to the previous check
        let inv_delta = delta.invert().unwrap();

        // delta . R
        let rerandomized_big_r = presignature.big_r * delta;

        //  (sigma + tweak * k) * delta^{-1}
        let rerandomized_sigma =
            (presignature.sigma + args.tweak.value() * presignature.k) * inv_delta;

        // k * delta^{-1}
        let rerandomized_k = presignature.k * inv_delta;

        Ok(Self {
            big_r: rerandomized_big_r.into(),
            k: rerandomized_k,
            sigma: rerandomized_sigma,
        })
    }
