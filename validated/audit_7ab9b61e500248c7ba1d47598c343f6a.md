### Title
Pool Admin Bypasses Hard Spread-Fee Cap via Uncapped Per-Bin Additional Fees — (`metric-core/contracts/MetricOmmPoolFactory.sol`, `metric-core/contracts/MetricOmmPool.sol`)

---

### Summary

`setPoolBinAdditionalFees` forwards `addFeeBuyE6` / `addFeeSellE6` (`uint16`, max 65 535 ≈ 6.55 % in E6 units) to the pool with no check against the hard spread-fee ceiling (`HARD_MAX_SPREAD_FEE_E6 = 200 000` = 20 %). Because the swap loop adds the per-bin additional fee on top of the oracle-derived base fee, the pool admin can push the effective spread for any bin to ≈ 26.55 %, silently exceeding the absolute cap that the protocol guarantees to traders.

---

### Finding Description

The factory enforces a two-layer fee cap hierarchy:

| Layer | Constant / variable | Max |
|---|---|---|
| Hard ceiling (immutable) | `HARD_MAX_SPREAD_FEE_E6` | 200 000 (20 %) |
| Soft cap (owner-settable) | `maxAdminSpreadFeeE6` | ≤ 20 % |
| Admin base fee | `adminSpreadFeeE6` in `poolFeeConfig` | ≤ `maxAdminSpreadFeeE6` |

`setPoolAdminFees` correctly enforces the soft cap:

```solidity
// MetricOmmPoolFactory.sol:414-415
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (newAdminNotionalFeeE8 > maxAdminNotionalFeeE8) revert AdminFeeTooHigh();
``` [1](#0-0) 

However, `setPoolBinAdditionalFees` passes the values straight through with only a bin-index range check:

```solidity
// MetricOmmPoolFactory.sol:450-457
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
  external override nonReentrant onlyPoolAdmin(pool)
{
  IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
``` [2](#0-1) 

```solidity
// MetricOmmPool.sol:464-474
function setBinAdditionalFees(int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
  external onlyFactory nonReentrant(PoolActions.SET_BIN_ADDITIONAL_FEES)
{
  if (bin < LOWEST_BIN || bin > HIGHEST_BIN) revert InvalidBinIndex(bin);
  BinState storage s = _binStates[bin];

  s.addFeeSellE6 = addFeeSellE6;

```solidity

```solidity

`addFeeBuyE6` / `addFeeSellE6` are `uint16`, so their maximum value is 65 535, which in E6 units equals ≈ 6.55 %. Combined with a base spread fee already at the 20 % hard cap, the effective per-bin spread reaches ≈ 26.55 % — 33 % above the absolute ceiling.

The hard cap itself is defined as:

```solidity
// MetricOmmPoolFactory.sol:44-45
uint24 internal constant HARD_MAX_SPREAD_FEE_E6 = 200_000;
uint24 internal constant HARD_MAX_NOTIONAL_FEE_E8 = 1_000_000;
``` [7](#0-6) 

---

### Impact Explanation

Traders swapping through any bin where `addFeeBuyE6` or `addFeeSellE6` is set above zero pay a total spread fee that exceeds the protocol's hard-coded maximum. The excess fee is retained in the pool as LP revenue, meaning the pool admin (who may also be an LP) can extract value from traders beyond what the protocol permits. This is a direct loss of trader principal and constitutes an admin-boundary break: the pool admin exceeds the hard cap without any timelock or owner approval.

---

### Likelihood Explanation

The pool admin is a semi-trusted role that is explicitly scoped to act only within caps. The call path is a single, no-timelock transaction (`setPoolBinAdditabilityFees`). Any pool admin who also holds LP positions in the affected bins has a direct financial incentive to set `addFeeBuyE6 = 65535`. The bypass requires no special conditions, no multi-step sequencing, and no external dependencies.

---

### Recommendation

Add a combined cap check inside `setPoolBinAdditionalFees` (or inside `setBinAdditionalFees` on the pool) that ensures the sum of the current base spread fee and the new per-bin additional fee does not exceed `HARD_MAX_SPREAD_FEE_E6`:

```solidity
// In MetricOmmPoolFactory.setPoolBinAdditionalFees, before forwarding:
uint24 currentSpread = poolFeeConfig[pool].adminSpreadFeeE6
                     + poolFeeConfig[pool].protocolSpreadFeeE6;
if (uint256(currentSpread) + addFeeBuyE6  > HARD_MAX_SPREAD_FEE_E6) revert AdminFeeTooHigh();
if (uint256(currentSpread) + addFeeSellE6 > HARD_MAX_SPREAD_FEE_E6) revert AdminFeeTooHigh();
```

Alternatively, cap `addFeeBuyE6` / `addFeeSellE6` independently at `HARD_MAX_SPREAD_FEE_E6` (since the base fee is already at least zero, any per-bin fee above the hard cap is always excessive).

---

### Proof of Concept

1. Factory owner sets `maxAdminSpreadFeeE6 = 200_000` (20 %, the hard ceiling).
2. Pool admin calls `setPoolAdminFees(pool, 200_000, 0)` — base spread is now at the hard cap.
3. Pool admin calls `setPoolBinAdditionalFees(pool, 0, 65535, 65535)` — no revert; `addFeeBuyE6 = addFeeSellE6 = 65535`.
4. A trader calls `swap` routing through bin 0. The effective fee applied is:

   ```
   baseFeeX64  = oracle-derived (≈ spreadFeeE6 = 200_000 / 1e6 = 20 %)
   addFeeX64   = 65535 / 1e6 ≈ 6.55 %
   totalFeeX64 ≈ 26.55 %
   ```

5. The trader pays ≈ 26.55 % spread instead of the protocol-guaranteed maximum of 20 %, with the excess accruing to LPs in that bin. No revert, no event distinguishing the cap breach.
