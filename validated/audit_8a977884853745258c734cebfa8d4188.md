### Title
Missing Identity-Point Validation for `app_pk` in CKD Allows Coordinator to Extract the Confidential Derived Secret — (`File: src/confidential_key_derivation/protocol.rs`)

---

### Summary

The `ckd()` function accepts an `app_pk` (the TEE app's ElGamal public key `A`) without validating that it is not the group identity element. When `app_pk = G1::identity()`, the ElGamal blinding term `y_i · A` collapses to zero for every node, causing the coordinator to directly receive the unblinded BLS signature `msk · H(pk, app_id)` — the confidential derived secret `s` — violating the protocol's core secrecy guarantee.

---

### Finding Description

The CKD protocol is an ElGamal-encrypted threshold BLS signature. Each node `i` computes:

```
C_i = S_i + y_i · A   where S_i = x_i · H(pk, app_id)
```

The coordinator aggregates:

```
C = Σ λ_i · C_i = msk · H(pk, app_id) + (Σ λ_i · y_i) · A
```

The app recovers `s = C − a · Y = msk · H(pk, app_id)` using its private scalar `a`.

The entire confidentiality of `s` rests on `A ≠ identity`, so that the blinding term `(Σ λ_i · y_i) · A` is non-zero and hides `msk · H(pk, app_id)` from the coordinator.

In `ckd()`, the public entry point, the only validations performed on `app_pk` are: [1](#0-0) 

There is **no check** that `app_pk` is not the identity element. The value flows directly into `compute_signature_share`: [2](#0-1) 

When `app_pk = G1Projective::identity()`, the scalar multiplication `app_pk * y.0` returns the identity for every node, so `big_c = big_s = x_i · H(pk, app_id)`. After aggregation the coordinator holds:

```
C = msk · H(pk, app_id)   // no blinding at all
```

This is exactly the confidential key `s` before HKDF, directly readable by the coordinator node.

The `verify_signature` function does guard against identity-point signatures, but it is called by the *app* after the fact — it does not protect the coordinator from learning `s` during the protocol: [3](#0-2) 

The protocol's security requirement is explicit: [4](#0-3) 

---

### Impact Explanation

When `app_pk = G1::identity()`, the coordinator — a single MPC node — directly computes `C = msk · H(pk, app_id)`, which is the confidential derived secret `s`. This violates the stated requirement that no single node can compute `s`, even if that node is malicious. The impact is **Critical: disclosure of a confidential derived secret** to a single party.

---

### Likelihood Explanation

`ckd()` is a public library function. The `app_pk` argument is a caller-supplied `blstrs::G1Projective`, and `G1Projective::identity()` is a valid, constructible value. A malicious TEE app submitting `A = 0` on-chain, or any malicious caller of the library, can trigger this with a single crafted call. No special privileges, leaked keys, or cryptographic breaks are required.

---

### Recommendation

Add a validation in `ckd()` that rejects `app_pk` if it is the identity element, analogous to the check already present in `verify_signature`:

```rust
if app_pk.is_identity().into() {
    return Err(InitializationError::BadParameters(
        "app_pk must not be the identity element".to_string(),
    ));
}
```

This should be placed immediately after the existing participant-list checks in `ckd()`: [5](#0-4)

        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }

    // kick out duplicates
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);

    // ensure my presence in the participant list
