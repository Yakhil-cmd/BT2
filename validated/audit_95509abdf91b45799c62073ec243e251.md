### Title
Malicious Participant Can Corrupt CKD Output Without Detection — (`src/confidential_key_derivation/protocol.rs`)

---

### Summary

The CKD coordinator blindly accumulates participant contributions with no proof of correctness. A single malicious participant can substitute arbitrary group elements for their legitimate share, causing the coordinator to accept and output a wrong confidential key derivation result. No verification step exists to detect this, unlike the robust ECDSA signing path which performs a final signature check.

---

### Finding Description

In `do_ckd_coordinator` the coordinator collects each participant's `(norm_big_y, norm_big_c)` pair and adds them unconditionally: [1](#0-0) 

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
```

The protocol invariant is that the accumulated output satisfies:

```
C = msk · H(pk ‖ app_id) + y_total · A
Y = y_total · G
```

so that `C − app_sk · Y = msk · H(pk ‖ app_id)` (the confidential key).

Each honest participant computes their contribution in `compute_signature_share`: [2](#0-1) 

The coordinator has no access to any participant's private share `x_i` or ephemeral scalar `y_i`, and no zero-knowledge proof or commitment is attached to the message. There is therefore no mechanism to distinguish a correct contribution from an arbitrary group element pair.

Compare this with the robust ECDSA coordinator path, which performs a final signature verification before accepting the result: [3](#0-2) 

```rust
if !sig.verify(&public_key, &msg_hash) {
    return Err(ProtocolError::AssertionFailed(
        "signature failed to verify".to_string(),
    ));
}
```

No equivalent check exists in the CKD path. The coordinator simply wraps the accumulated values and returns them: [4](#0-3) 

---

### Impact Explanation

A malicious participant sends `(Δ_Y, Δ_C)` instead of their correct contribution. The coordinator computes:

```
Y'  = Y_correct + Δ_Y
C'  = C_correct + Δ_C
C' − app_sk · Y' = msk · H(pk ‖ app_id) + Δ_C − app_sk · Δ_Y   ≠   msk · H(pk ‖ app_id)
```

The coordinator (an honest party) accepts this corrupted `CKDOutput` as valid. Any TEE application that subsequently calls `unmask(app_sk)` on this output derives a wrong secret key, rendering all data encrypted under the correct key permanently inaccessible, or allowing the attacker to steer the derived key to a value of their choosing.

This matches: **High — Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.**

---

### Likelihood Explanation

Any single participant in a CKD session can trigger this. The attacker needs only to be a registered participant and to deviate from the protocol by sending arbitrary `ElementG1` values in place of their legitimate share. No special privilege, pre-shared secret, or cryptographic break is required. The attack is undetectable by the coordinator with the information currently available to it.

---

### Recommendation

Require each participant to attach a zero-knowledge proof of correct formation alongside their `(norm_big_y, norm_big_c)` contribution. Concretely, each participant knows `(y_i, x_i)` and can prove in zero knowledge that:

- `norm_big_y = λ_i · y_i · G`
- `norm_big_c = λ_i · (x_i · H(pk ‖ app_id) + y_i · A)`

using their publicly known verification key share `X_i = x_i · G2` (available from the DKG output). The coordinator verifies each proof before accumulating the contribution, rejecting any participant whose proof fails.

---

### Proof of Concept

A malicious participant replaces their honest call to `compute_signature_share` with:

```rust
// Malicious participant sends (G1, G1) instead of correct values
let fake_y = ElementG1::generator();
let fake_c = ElementG1::generator();
chan.send_private(waitpoint, coordinator, &(fake_y, fake_c))?;
``` [5](#0-4) 

The coordinator receives this via `recv_from_others` and adds it to the running sum without any check: [6](#0-5) 

The resulting `CKDOutput` encodes `C_correct + G1` and `Y_correct + G1`. When the application calls `unmask(app_sk)`, it obtains `msk · H(pk ‖ app_id) + G1 − app_sk · G1`, which is a wrong and attacker-influenced value. The coordinator returns `Ok(Some(ckd_output))` with no error, breaking the protocol's core invariant that the output faithfully encodes the threshold BLS-derived confidential key.

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}
```
