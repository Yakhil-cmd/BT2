Audit Report

## Title
Unbounded Inbound Connection Task Accumulation Enables Resource Exhaustion DoS - (File: rs/p2p/quic_transport/src/connection_manager.rs)

## Summary
The `ConnectionManager` in the QUIC transport layer spawns an async task into an unbounded `JoinSet` for every accepted inbound QUIC connection, before the TLS handshake completes and before any application-level cap is enforced. A malicious subnet peer whose IP is firewall-whitelisted can flood the victim node with QUIC connection initiations, accumulating unbounded tasks in `inbound_connecting`, each performing TLS crypto work, exhausting CPU and memory and degrading consensus participation.

## Finding Description
`ConnectionManager` holds `inbound_connecting: JoinSet<Result<ConnectionWithPeerId, ConnectionEstablishError>>` initialized with no capacity bound at line 256. The event loop at lines 354–362 calls `self.handle_inbound_conn_attemp(incoming)` unconditionally for every `Incoming` returned by `self.endpoint.accept()`, with no check on `self.inbound_connecting.len()`. Inside `handle_inbound_conn_attemp` (lines 537–588), a new async task is spawned via `self.inbound_connecting.spawn(timeout_conn_fut)` at line 587 with no size guard. The TLS handshake (`incoming.await` at line 542–548) and certificate verification (`node_id_from_certificate_der` at line 563) execute inside the spawned task, not before it. This means each connection attempt consumes a task slot and triggers crypto work before any rejection can occur. The task lives for up to `CONNECT_TIMEOUT` (10 seconds, line 82). By contrast, the outbound side uses `JoinMap<NodeId, ...>` (line 118), which is naturally bounded by the number of unique peer IDs. The inbound side has no equivalent bound. On topology change (lines 403–411), the TLS server config is updated to restrict to subnet members, but this restriction is enforced only after the handshake completes inside the already-spawned task — it does not prevent task creation.

The nftables firewall (golden config line 33) applies a connection limit of 1000 per source IP and a rate limit of 1000 new flows/minute for TCP (`ct state new`). For UDP, the whitelisted-node rule at line 41 is a plain `accept` with no per-connection rate cap at the QUIC application level. Even within nftables limits, a malicious node can sustain hundreds of concurrent pending tasks, each holding a QUIC connection object and performing TLS handshake crypto, for the full 10-second timeout window.

## Impact Explanation
A single Byzantine subnet node (below the 1/3 fault threshold) can continuously initiate QUIC connections to a victim replica. Each initiation spawns an unbounded async task performing TLS handshake crypto. Accumulating hundreds to thousands of such tasks exhausts the Tokio runtime's CPU and heap memory on the victim node, causing it to miss consensus deadlines and become unresponsive to legitimate subnet peers. This constitutes an application/platform-level DoS and subnet availability impact not based on raw volumetric DDoS — matching the High ($2,000–$10,000) impact class.

## Likelihood Explanation
The attacker must be a current subnet member whose IP is nftables-whitelisted for UDP port 4100. This is a constrained but realistic attacker: subnets have tens of nodes, node operators are permissioned but not individually trusted, and a single Byzantine node below the 1/3 consensus fault threshold suffices. The attack requires no special privileges beyond subnet membership, no victim mistakes, and no external dependencies. It is repeatable and sustained as long as the attacker remains in the subnet topology. The nftables connection limit (1000 per IP) bounds but does not eliminate the attack surface — 1000 concurrent 10-second tasks is still sufficient to degrade a replica's Tokio runtime.

## Recommendation
1. **Cap `inbound_connecting` before spawning**: In `handle_inbound_conn_attemp`, check `self.inbound_connecting.len()` against a hard limit (e.g., `2 × subnet_size` or a fixed constant such as 200). If exceeded, call `incoming.refuse()` or drop the `Incoming` immediately without spawning a task.
2. **Per-peer rate limiting**: Track inbound connection attempts per source IP or `NodeId` and reject excessive reconnects within a short window (e.g., no more than 2 pending inbound tasks per peer at a time).
3. **Reject before spawning**: Perform a lightweight pre-check (e.g., source IP allowlist check at the QUIC layer via `Incoming::remote_address()`) before spawning the TLS handshake task, analogous to the outbound side's `can_i_dial_to` guard.

## Proof of Concept
A malicious subnet node M repeatedly calls `quinn::Endpoint::connect()` targeting victim node V's UDP port 4100, either abandoning the handshake mid-flight or completing it and immediately reconnecting. Each attempt causes V's event loop to execute:

```rust
incoming = self.endpoint.accept() => {
    // No size check on self.inbound_connecting
    self.handle_inbound_conn_attemp(incoming);  // spawns task unconditionally
}
```

A deterministic integration test can be written using the existing `PeerRestrictedTlsConfig` test harness in `rs/p2p/quic_transport/tests/common.rs`: spin up a victim `ConnectionManager`, connect a malicious peer in a loop without completing handshakes, assert that `inbound_connecting.len()` grows unboundedly and that the victim's event loop latency (measured via a separate ping task) degrades proportionally. At the nftables-permitted rate, V accumulates up to 1000 concurrent tasks in `inbound_connecting`, each holding a QUIC connection object and performing TLS crypto, exhausting the Tokio runtime and causing it to miss consensus deadlines.
