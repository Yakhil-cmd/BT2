### Title
Attacker Can Permanently Freeze Staker Rewards via `disable_rewards=true` Frontrun - (File: src/staking/staking.cairo)

### Summary

`update_rewards` in the Staking contract is publicly callable with a `disable_rewards: bool` parameter. Any unprivileged caller can invoke it with `disable_rewards=true` to advance the global `last_reward_block` checkpoint without distributing any rewards. Because the contract enforces a strict "one call per block" invariant, a subsequent legitimate call by the staker for the same block reverts with `REWARDS_ALREADY_UPDATED`. An attacker can repeat this every block to permanently deny all stakers their consensus-period rewards.

### Finding Description

`update_rewards` is exposed as a public ABI entry point with no access-control guard beyond `general_prerequisites()` (which only checks the contract is unpaused and the caller is non-zero). [1](#0-0) 

The function first asserts the current block is strictly greater than the stored `last_reward_block`: [2](#0-1) 

It then unconditionally writes the current block number to `last_reward_block` **before** checking `disable_rewards`: [3](#0-2) 

If `disable_rewards` is `true`, the function returns immediately without calling `_update_rewards`, so no rewards are ever credited: [4](#0-3) 

Because `last_reward_block` is a **single global field** (not per-staker), one attacker call per block poisons the checkpoint for every staker simultaneously: [5](#0-4) 

### Impact Explanation

Every block in the consensus period, an attacker calls `update_rewards(any_active_staker, disable_rewards=true)`. This advances `last_reward_block` to the current block without distributing rewards. Any honest staker (or keeper) who then calls `update_rewards(staker, disable_rewards=false)` in the same block receives `REWARDS_ALREADY_UPDATED` and reverts. Repeated across every block, this constitutes **permanent freezing of unclaimed yield** for all stakers and their delegators — matching the allowed High impact category.

### Likelihood Explanation

The attack requires only one cheap transaction per block from any unprivileged address. On Starknet, transaction fees are low, making sustained griefing economically viable. No privileged key, bridge compromise, or external dependency is needed. The attacker does not need to hold any stake.

### Recommendation

Restrict who may pass `disable_rewards=true`. Either:
1. Remove the `disable_rewards` parameter from the public ABI and handle the "no-reward" path internally (e.g., when the staker's balance is zero or the epoch is pre-consensus), or
2. Add an access-control check so only a trusted role (e.g., `security_agent` or the staker themselves) can supply `disable_rewards=true`.

Additionally, consider writing `last_reward_block` only after the reward distribution logic completes, so a failed or skipped distribution does not consume the block slot.

### Proof of Concept

1. Consensus period is active (`is_pre_consensus()` returns `false`).
2. Staker Alice has an active stake and is waiting for block rewards.
3. Attacker Bob calls `update_rewards(alice_address, disable_rewards=true)` at the start of every new block.
4. `last_reward_block` is updated to the current block; no rewards are distributed.
5. Alice (or any keeper) calls `update_rewards(alice_address, disable_rewards=false)` — the call reverts with `REWARDS_ALREADY_UPDATED` because `current_block_number > last_reward_block` is now `false`.
6. Bob repeats step 3 every block. Alice and all other stakers accumulate zero rewards indefinitely.
7. The attack cost is one Starknet transaction per block; the damage is total loss of consensus-period yield for the entire protocol.
