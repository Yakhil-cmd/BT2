### Title
Unrestricted `execute_scheduled` Allows Any Caller to Trigger a User's Scheduled XCC Promise at an Adversarially Chosen Time — (File: `etc/xcc-router/src/lib.rs`)

---

### Summary

The XCC router's `execute_scheduled` function imposes no caller restriction. Any NEAR account can call it at any time with any valid nonce. Because scheduled promises are committed on-chain with fixed content but no execution deadline and no cancellation path, an adversary can observe a user's pending promise (e.g., a large token swap on a NEAR DeFi protocol) and trigger it at a moment of the adversary's choosing — enabling sandwich attacks and other MEV extraction that directly drain user funds.

---

### Finding Description

When an Aurora EVM user calls the XCC precompile with `CrossContractCallArgs::Delayed(...)`, the engine calls `schedule` on the user's per-address router contract. Only the parent Aurora engine can call `schedule` (enforced by `assert_preconditions`). The promise is stored in `scheduled_promises` keyed by a monotonically incrementing nonce. [1](#0-0) 

The subsequent execution function, however, carries an explicit design comment that it is open to any caller: [2](#0-1) 

The comment asserts "there is no security risk" because the promise content was fixed at schedule time. This is incorrect for any promise whose outcome depends on the state of external contracts at execution time (e.g., DEX prices, lending rates, oracle values). The promise content is immutable, but the *result* of executing it is not — it depends entirely on when execution occurs.

There is no `cancel_scheduled` function anywhere in the codebase, and no deadline field on stored promises. Once a promise is scheduled, it persists indefinitely and can be triggered by any NEAR account. [3](#0-2) 

The `CrossContractCallArgs::Delayed` variant is the EVM-level entry point that produces this state: [4](#0-3) 

---

### Impact Explanation

**Impact: High — Theft of user funds.**

Concrete attack (sandwich):

1. User submits an EVM transaction calling the XCC precompile with `Delayed`, scheduling a large `ft_transfer` or swap call on a NEAR DEX. The promise is stored on-chain in the router's `scheduled_promises` map.
2. Adversary reads the promise content from NEAR state (it is fully public).
3. Adversary executes a front-run trade on the same DEX, moving the price adversarially.
4. Adversary calls `execute_scheduled(nonce)` on the user's router, triggering the user's swap at the manipulated price.
5. Adversary back-runs to close their position, extracting the price impact from the user.

The user receives fewer tokens than they would have at the time they scheduled the promise. The adversary captures the difference. This constitutes direct theft of user funds.

A secondary impact exists in the reverse direction: the user themselves holds a perpetual free option. Because there is no deadline and no cancellation, the user can wait indefinitely and call `execute_scheduled` only when market conditions are maximally favorable, at the expense of the counterparty in the NEAR protocol being called.

---

### Likelihood Explanation

**Likelihood: Medium.**

- The XCC `Delayed` path is a documented, intentional feature used when gas is insufficient for immediate execution.
- Scheduled promise content is fully visible on-chain; no off-chain monitoring infrastructure is required beyond a standard NEAR RPC node.
- NEAR's asynchronous execution model means there is always a window between scheduling and execution.
- The attack requires the user to schedule a price-sensitive operation, which is a realistic use case (e.g., swapping wNEAR for another token via a NEAR DEX through XCC).
- No special privileges are required; any NEAR account can call `execute_scheduled`.

---

### Recommendation

1. **Restrict `execute_scheduled` to the owner.** Add a check that `env::predecessor_account_id()` matches either the parent engine or a designated executor. The engine can expose a new method that verifies the EVM signer before forwarding the call.

2. **Add an execution deadline.** Store a `valid_until` block height alongside each scheduled promise. Reject execution after the deadline, and allow the user to cancel expired promises and reclaim any attached wNEAR.

3. **Add a cancellation path.** Expose a `cancel_scheduled(nonce)` method callable only by the parent engine (on behalf of the EVM address owner), so users are not permanently committed to a promise they can no longer execute safely.

---

### Proof of Concept

```
// Step 1: Alice (EVM address 0xALICE) submits an Aurora EVM tx calling the XCC precompile
//         with CrossContractCallArgs::Delayed(ft_transfer { amount: 100_000 wNEAR, ... })
//         Engine calls router_for_alice.schedule(promise) → stored at nonce=0

// Step 2: Bob reads router_for_alice.scheduled_promises[0] from NEAR state.
//         He sees Alice will swap 100_000 wNEAR for USDC on ref-finance.

// Step 3: Bob calls ref-finance directly, buying USDC and moving the price against Alice.

// Step 4: Bob calls:
//   router_for_alice.execute_scheduled({ "nonce": "0" })
//         → Alice's swap executes at the manipulated price.

// Step 5: Bob sells his USDC position, capturing the price impact extracted from Alice.
```

    pub fn schedule(&mut self, #[serializer(borsh)] promise: PromiseArgs) {
        self.assert_preconditions();

        let nonce = self.nonce.get().unwrap_or_default();
        self.scheduled_promises.insert(nonce, promise);
        self.nonce.set(&(nonce + 1));

        near_sdk::log!("Promise scheduled at nonce {}", nonce);
    }

            env::panic_str("ERR_PROMISE_NOT_FOUND")
