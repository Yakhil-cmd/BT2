### Title
Unvalidated `app_pk` in CKD Protocol Allows Coordinator to Extract Confidential Derived Secrets — (`File: src/confidential_key_derivation/protocol.rs`)

---

### Summary

The `ckd()` entry point in the Confidential Key Derivation (CKD) protocol accepts an arbitrary `app_pk: PublicKey` (`blstrs::G1Projective`) with no validation. A malicious coordinator who supplies the identity element as `app_pk` to all participants causes every participant to transmit their unmasked secret-share contribution directly to the coordinator, allowing the coordinator to reconstruct the confidential derived key `msk · H(pk ∥ app_id)` without ever possessing the application's secret key `app_sk`.

---

### Finding Description

The CKD protocol is designed so that participants mask their secret-share contribution with a one-time blinding term `y · app_pk` before sending it to the coordinator:

```
big_c = big_s + app_pk * y      // big_s = x_i · H(pk ∥ app_id)
big_y = y * G
```

The coordinator aggregates the Lagrange-weighted sums and forwards `(norm_big_y, norm_big_c)` to the application, which unmasks with its secret key: `norm_big_c − app_sk · norm_big_y = msk · H(pk ∥ app_id)`.

The entire confidentiality guarantee rests on `app_pk` being a valid, non-identity public key (`app_sk · G`). However, the `ckd()` initializer performs no such check: [1](#0-0) 

The only guards present are participant-count, duplicate, and membership checks. `app_pk` is accepted unconditionally and forwarded into `compute_signature_share`: [2](#0-1) 

Specifically, at line 174:

```rust
let big_c = big_s + app_pk * y.0;
```

If `app_pk` is `G1Projective::identity()` (the additive identity of the group), then `app_pk * y.0 = identity`, so:

```
big_c = big_s + identity = big_s = x_i · H(pk ∥ app_id)
```

The masking term vanishes entirely. The coordinator receives each participant's raw `big_c_i = x_i · H(pk ∥ app_id)` and aggregates:

```
norm_big_c = Σ λ_i · x_i · H(pk ∥ app_id) = msk · H(pk ∥ app_id)
```

This is the confidential derived key itself, obtained with no knowledge of `app_sk`.

`app_pk` is never transmitted between participants during the protocol — it is only used locally — so there is no cross-participant consistency check that would detect a coordinator-supplied malicious value. [3](#0-2) 

---

### Impact Explanation

A malicious coordinator who provides `app_pk = G1Projective::identity()` to all participants receives the confidential derived key `msk · H(pk ∥ app_id)` in plaintext as the aggregated `norm_big_c`, without requiring the application's secret key `app_sk`. This constitutes **extraction and disclosure of a confidential derived secret**, matching the Critical impact tier.

---

### Likelihood Explanation

The coordinator is a participant in the protocol and is the natural party that distributes session parameters (including `app_pk`) to other participants in any realistic deployment. A malicious coordinator can trivially supply `app_pk = G1Projective::identity()` — a valid Rust value of type `blstrs::G1Projective` — with no special privileges beyond their normal role. No cryptographic break or external compromise is required.

---

### Recommendation

Add an explicit identity-element check on `app_pk` inside `ckd()` before the protocol begins:

```rust
if app_pk.is_identity().into() {
    return Err(InitializationError::BadParameters(
        "app_pk must not be the identity element".to_string(),
    ));
}
```

Additionally, consider broadcasting `app_pk` as part of the protocol transcript so that all participants can verify they are operating on the same value, preventing a coordinator from supplying different `app_pk` values to different participants.

---

### Proof of Concept

1. Coordinator initiates CKD and passes `app_pk = blstrs::G1Projective::identity()` to all participants when calling `ckd()`.
2. Each participant enters `compute_signature_share`. At line 174, `big_c = big_s + identity * y = big_s = x_i · H(pk ∥ app_id)`.
3. Each participant sends `(big_y, big_c)` to the coordinator via `chan.send_private`.
4. The coordinator's `do_ckd_coordinator` aggregates: `norm_big_c = Σ λ_i · big_c_i = msk · H(pk ∥ app_id)`.
5. The coordinator now holds the confidential derived key directly — no `app_sk` needed. [4](#0-3) [5](#0-4)

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}
