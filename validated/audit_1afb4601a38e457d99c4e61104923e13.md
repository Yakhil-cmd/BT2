## Code Flow Trace

**Caller 1 — `CreateOfferBuilder.tsx`** (offer creation conflict resolution): [1](#0-0) 

`allowSecureCancelling` is explicitly set to `true` here (JSX boolean shorthand). This caller is safe.

**Caller 2 — `useAcceptOfferHook.tsx`** (offer acceptance conflict resolution): [2](#0-1) 

`allowSecureCancelling` is **not passed at all**, so it arrives as `undefined` in the dialog.

**`OfferEditorCancelConflictingOffersDialog`** passes it straight through to `CancelOfferList`: [3](#0-2)

That `false` value is then passed directly as `canCancelWithTransaction` to `handleCancelOffer`: [5](#0-4) 

Inside `handleCancelOffer`, when `canCancelWithTransaction=false`:

- The RPC call becomes `cancelOffer({ tradeId, secure: false, fee: 0 })` [6](#0-5) [7](#0-6)

## Verdict

### Title
Off-chain-only cancellation of shared offers during accept-flow conflict resolution — (`packages/gui/src/hooks/useAcceptOfferHook.tsx`)

### Summary
When a user accepts an incoming offer and has conflicting open offers, `useAcceptOfferHook` opens `OfferEditorCancelConflictingOffersDialog` without `allowSecureCancelling={true}`. This causes `CancelOfferList` to default to `allowSecureCancelling=false`, which forces all cancellations in that dialog to use `secure=false` (off-chain only), regardless of whether the conflicting offer has already been shared.

### Finding Description
`CreateOfferBuilder` correctly passes `allowSecureCancelling` (i.e., `true`) to the conflict dialog. `useAcceptOfferHook` does not. The missing prop propagates through `CancelOfferList`'s default (`= false`) and into the `cancelOffer` RPC call as `secure: false`. The `ConfirmOfferCancellation` dialog also hides the "Cancel on blockchain" UI entirely, so the user has no way to choose on-chain cancellation in this flow.

### Impact Explanation
A user who has shared a `PENDING_ACCEPT` offer and then tries to accept a different incoming offer will be shown the conflict dialog. If they cancel their shared offer through this dialog, the cancellation is off-chain only. The original offer file remains valid on-chain. Any counterparty holding the offer file can still accept it, causing an unexpected asset transfer from the user's wallet.

### Likelihood Explanation
The scenario (user has a shared open offer and simultaneously tries to accept another offer) is a normal, reachable production workflow. No special attacker capability is required — a counterparty who already received the offer file simply submits it before the local cancellation propagates.

### Recommendation
Pass `allowSecureCancelling={true}` in `useAcceptOfferHook.tsx` when opening `OfferEditorConflictAlertDialog`, matching the pattern already used in `CreateOfferBuilder.tsx`:

```tsx
<OfferEditorConflictAlertDialog
  assetsToUnlock={assetsRequiredToBeUnlocked}
  assetsBetterUnlocked={[]}
  allowSecureCancelling  // add this
/>
```

### Proof of Concept
1. Create and share an offer (XCH → CAT). It is now `PENDING_ACCEPT`.
2. In the same wallet, begin accepting a different incoming offer that uses the same XCH coins.
3. The accept flow calls `useAcceptOfferHook` → `offerBuilderDataToOffer` detects the conflict → opens `OfferEditorConflictAlertDialog` **without** `allowSecureCancelling`.
4. Click "Cancel" on the conflicting offer in the dialog. `ConfirmOfferCancellation` shows no "Cancel on blockchain" option.
5. Confirm. Observe the RPC call: `cancelOffer({ tradeId, secure: false, fee: 0 })`.
6. From a second wallet/node that holds the original offer file, submit `take_offer`. It succeeds — the offer was never invalidated on-chain.

#Vulnerability found.
