### Title
Missing Minimum Threshold Enforcement in FROST Signing Validation Allows Malicious Coordinator to Corrupt Signing Outputs - (File: src/frost/mod.rs)

---

### Summary

`assert_sign_inputs` in `src/frost/mod.rs` omits the lower-bound check (`threshold >= 2`) that is consistently enforced in every other protocol entry point in the library. A malicious coordinator can supply `threshold = 1` to `sign_v1`, `sign_v2` (EdDSA), or `sign` (RedJubjub), causing the FROST signing protocol to execute under a cryptographically invalid threshold. Honest participants commit their nonces and signing shares to a session that will produce an unusable signature, corrupting the sign output.

---

### Finding Description

Every protocol entry point in the library that accepts a `threshold` parameter enforces a minimum value of 2:

**`assert_key_invariants` — `src/dkg.rs` lines 580–582:**
```rust
if threshold < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
}
```

**`validate_triple_inputs` — `src/ecdsa/ot_based_ecdsa/triples/generation.rs` lines 699–703:**
```rust
if threshold_value < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold_value,
        min: 2,
    });
}
```

`assert_sign_inputs` in `src/frost/mod.rs` (lines 120–160) is the shared validation gate for all FROST signing paths. It checks only the upper bound:

```rust
// validate threshold
if threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge {
        threshold: threshold.value(),
        max: participants.len(),
    });
}
```

There is **no lower-bound check**. `threshold = 1` passes validation silently.

This function is the sole validation gate for:
- `sign_v1` / `sign_v2` in `src/frost/eddsa/sign.rs` (lines 47, 73)
- `sign` in `src/frost/redjubjub/sign.rs` (line 50)

The `threshold` value is then forwarded into the FROST signing protocol, where it governs Lagrange interpolation and the minimum-signers constraint embedded in the `KeyPackage`. With `threshold = 1`, the protocol constructs a degree-0 interpolation over a key that was generated with a degree-1 polynomial (threshold = 2 from DKG). The aggregated signature is cryptographically invalid against the actual public key.

---

### Impact Explanation

**High — Corruption of sign outputs so honest parties accept unusable cryptographic outputs.**

When a coordinator supplies `threshold = 1`:

1. `assert_sign_inputs` accepts the call without error.
2. Honest participants receive the signing request, generate their nonces, and broadcast signing shares — irreversibly consuming their presign material.
3. The coordinator aggregates the shares using Lagrange coefficients computed for a degree-0 polynomial, which is inconsistent with the degree-1 secret sharing established at keygen.
4. The resulting signature `(R, s)` is invalid against the actual group public key.
5. Honest participants have no mechanism to detect the mismatch before committing their shares; the protocol completes from their perspective but delivers an unusable output.

This maps directly to the allowed High impact: *"Corruption of sign outputs so honest parties accept unusable cryptographic outputs."*

---

### Likelihood Explanation

The coordinator role is a documented, explicitly trusted participant in the FROST signing protocol. The library's own documentation and code comments treat the coordinator as a privileged but potentially adversarial role (see the split-view attack mitigations in `src/ecdsa/robust_ecdsa/sign.rs`). A malicious coordinator is therefore an in-scope threat model participant. The attack requires only passing an integer `1` as the `threshold` argument — no cryptographic capability is needed. The missing check is a single-line omission relative to every other validation function in the library.

---

### Recommendation

Add the same lower-bound guard that exists in `assert_key_invariants` and `validate_triple_inputs` to `assert_sign_inputs`:

```rust
// In src/frost/mod.rs, assert_sign_inputs, after the upper-bound check:
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

Apply the same fix to `presign` in `src/frost/mod.rs` (line 72 area) and `sign` in `src/ecdsa/ot_based_ecdsa/sign.rs` (line 58 area), which also accept caller-supplied thresholds without a lower-bound check.

pub fn assert_sign_inputs(
    participants: &[Participant],
    threshold: impl Into<ReconstructionLowerBound>,
    me: Participant,
    coordinator: Participant,
) -> Result<ParticipantList, InitializationError> {
    let threshold = threshold.into();
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // validate threshold
    if threshold.value() > participants.len() {
        return Err(InitializationError::ThresholdTooLarge {
            threshold: threshold.value(),
            max: participants.len(),
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }
    Ok(participants)
}
