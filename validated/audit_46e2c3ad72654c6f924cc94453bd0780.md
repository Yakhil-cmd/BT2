### Title
Incorrect Royalty Amount Displayed in WalletConnect Offer Approval Dialog Causes User to Approve Transactions with Understated Costs - (File: `packages/gui/src/electron/commands/parseCommandDisplay.ts`)

### Summary
`formatAmountWithRoyalties` in `parseCommandDisplay.ts` incorrectly divides the fungible amount equally among all NFTs before computing each royalty, rather than applying each NFT's royalty to the full amount. This causes the "Total Amount with Royalties" shown in the WalletConnect `Confirm.tsx` approval dialog to be systematically lower than the actual on-chain cost when a `take_offer` or `create_offer_for_ids` command involves multiple NFTs with royalties. A malicious WalletConnect dapp can exploit this to cause a user to approve a transaction while seeing a materially understated spending total.

### Finding Description

**Root cause — `formatAmountWithRoyalties`:** [1](#0-0) 

```ts
const splitAmount = amount / BigInt(royaltyPercentages.length);   // ← wrong
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
```

The function divides the fungible amount by the number of NFTs (`splitAmount`), then applies each NFT's royalty percentage to that split. In Chia's protocol, every NFT's royalty is applied to the **full** fungible amount, not a fraction of it. The correct formula is:

```
royaltyAmount = Σ (amount × royaltyPercentage_i / 10_000)
```

The buggy formula computes:

```
royaltyAmount = Σ ((amount / N) × royaltyPercentage_i / 10_000)
              = (1/N) × Σ (amount × royaltyPercentage_i / 10_000)
```

This underestimates the royalty by a factor of N (number of NFTs).

**Concrete example** (confirmed by the existing test at line 251–335): [2](#0-1) 

Two NFTs with royalties 500 (5 %) and 10 (0.1 %), fungible amount = 100 000 000 mojos (0.0001 XCH):

| | Displayed (buggy) | Actual on-chain |
|---|---|---|
| NFT1 royalty | 2 500 000 | 5 000 000 |
| NFT2 royalty | 50 000 | 100 000 |
| Total royalty | 2 550 000 | 5 100 000 |
| `amountWithRoyalties` | 0.00010255 XCH | 0.0001051 XCH |

The test itself encodes the wrong value as the expected output, confirming the bug is present and untested-against-correct-behavior.

**Contrast with the correct approach** used in `OfferBuilderXCHSection.tsx`, which sums backend-computed per-NFT royalty payments directly without splitting: [3](#0-2) 

**Display surface — `Confirm.tsx`:**

The incorrect `amountWithRoyalties` string is rendered verbatim in the WalletConnect approval dialog under "You Spend → Total Amount with Royalties": [4](#0-3) 

**Entry point — `parseCommandDisplay`:** [5](#0-4) 

Both `chia_wallet.take_offer` and `chia_wallet.create_offer_for_ids` flow through `walletDeltaToDisplay` → `withRoyaltyTotals` → `formatAmountWithRoyalties`. [6](#0-5) 

### Impact Explanation

A WalletConnect-connected dapp sends a `take_offer` command containing an offer with multiple NFTs carrying royalties. The GUI's approval dialog shows a "Total Amount with Royalties" that is lower than the actual on-chain deduction. The user approves based on the understated figure; the blockchain enforces the correct (higher) royalty, draining more XCH/CAT than the user consented to. The discrepancy scales with the number of NFTs and their royalty percentages — with high-royalty NFTs (e.g., 50 % each) and several NFTs, the displayed amount can be a small fraction of the actual cost.

**Impact category:** High — WalletConnect state causes a user to approve a spend of the wrong (understated) amount of XCH/CAT.

### Likelihood Explanation

Any WalletConnect dapp that has been granted `chia_wallet.take_offer` permission can trigger this. The attacker only needs to craft an offer with ≥ 2 NFTs that carry royalties. No privileged access, leaked keys, or cryptographic break is required. The victim sees a plausible-looking confirmation dialog and has no way to detect the discrepancy without independently computing the correct royalty total.

### Recommendation

Replace the split-then-sum approach with a direct sum over the full amount:

```ts
// Correct: apply each royalty to the full amount
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (amount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
```

Remove the `splitAmount` variable entirely. Add a test case that asserts the correct multi-NFT total (e.g., 0.0001051 XCH for the two-NFT scenario above).

### Proof of Concept

1. Attacker dapp establishes a WalletConnect session with the victim and is granted `chia_wallet.take_offer`.
2. Attacker constructs an offer string containing two NFTs: NFT-A (royalty 5 000 = 50 %) and NFT-B (royalty 5 000 = 50 %), with 1 XCH requested.
3. Attacker sends `chia_wallet.take_offer` with that offer.
4. `parseCommandDisplay` calls `formatAmountWithRoyalties([5000, 5000], 1_000_000_000_000n)`:
   - `splitAmount = 500_000_000_000n`
   - `royaltyAmount = (500_000_000_000 × 5000 / 10000) × 2 = 500_000_000_000n`
   - Displayed total: **2 XCH**
5. Correct on-chain total: `1 XCH + (1 × 50 %) + (1 × 50 %) = 2 XCH` — in this symmetric case the numbers happen to match, but with asymmetric royalties (e.g., 5 000 and 10) the discrepancy is clear:
   - Displayed: `(500_000_000_000 × 5000 / 10000) + (500_000_000_000 × 10 / 10000)` = 250 500 000 000 mojos royalty → total ≈ 1.2505 XCH
   - Actual: `(1_000_000_000_000 × 5000 / 10000) + (1_000_000_000_000 × 10 / 10000)` = 501 000 000 000 mojos royalty → total ≈ 1.501 XCH
6. User sees "Total Amount with Royalties: 1.2505 XCH", approves; wallet is debited 1.501 XCH.

function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],
): string | undefined {
  if (royaltyPercentages.length === 0 || line.kind === 'nft') {
    return undefined;
  }

  const splitAmount = amount / BigInt(royaltyPercentages.length);
  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;

  if (line.kind === 'xch') {
    return mojoToChiaLocaleString(totalAmount);
  }

  return mojoToCATLocaleString(totalAmount);
}

function withRoyaltyTotals(
  items: DisplayWalletDeltaItemWithKey[],
  amounts: Record<string, bigint>,
  oppositeSideLines: DisplayWalletDeltaItem[],
): DisplayWalletDeltaItem[] {
  const royaltyPercentages = royaltyPercentagesForSide(oppositeSideLines);

  return items.map(({ key, line }) => {
    if (line.kind === 'nft') {
      return line;
    }

    const amount = amounts[key];
    const amountWithRoyalties = amount ? formatAmountWithRoyalties(line, amount, royaltyPercentages) : undefined;

    return amountWithRoyalties ? { ...line, amountWithRoyalties } : line;
  });
}
