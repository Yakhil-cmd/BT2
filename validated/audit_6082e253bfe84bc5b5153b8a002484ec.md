### Title
ERC-20 Mirror Token Supply Drain via Unchecked Precompile Return Value in `withdrawToNear`/`withdrawToEthereum` — (`etc/eth-contracts/contracts/EvmErc20.sol`)

---

### Summary

`EvmErc20.sol` unconditionally burns ERC-20 mirror tokens **before** calling the exit precompile, and the assembly `call()` return value is never checked. If the precompile call fails synchronously (returning 0), the burn is committed with no corresponding credit on NEAR or Ethereum, permanently draining the ERC-20 token supply. A second, asynchronous drain path exists when the NEAR-side `ft_transfer` promise fails and the `error_refund` feature is not compiled in — a scenario the codebase explicitly acknowledges but leaves unguarded by default.

---

### Finding Description

`EvmErc20.sol` implements two withdrawal functions:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // ← burn first, unconditionally

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        // res is NEVER checked
    }
}
``` [1](#0-0) 

The same pattern appears in `withdrawToEthereum`: [2](#0-1) 

**Drain path 1 — synchronous precompile failure.** The `exit_to_near` precompile (`0xe921...`) can return `ExitError` for several reasons: invalid input size, unknown flag, missing NEP-141 mapping (`ERR_TARGET_TOKEN_NOT_FOUND`), or invalid recipient. When it does, the EVM `call()` opcode returns `res = 0`. Because `EvmErc20.sol` never inspects `res`, execution continues, the `_burn` is committed, and no cross-chain transfer is scheduled. The tokens are gone. [3](#0-2) 

**Drain path 2 — asynchronous NEAR promise failure.** Even when the precompile succeeds in the EVM and schedules a NEAR `ft_transfer` promise, that promise can fail (recipient account has no storage deposit, NEP-141 contract is paused, gas exhaustion, etc.). The engine provides a `refund_on_error` mechanism that re-mints burned ERC-20 tokens on failure, but this is gated behind the `error_refund` feature flag: [4](#0-3) 

`error_refund` is **not** in the `default` feature set. Without it, `MIN_INPUT_SIZE` is 3 bytes and no refund address is embedded in the precompile input, so there is no callback to restore tokens when the NEAR promise fails. [5](#0-4) 

The engine test suite explicitly documents this loss:

```rust
// If the refund feature is not enabled, then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let expected_balance = Wei::new_u64(INITIAL_ETH_BALANCE - ETH_EXIT_AMOUNT);
``` [6](#0-5) 

The corrected `EvmErc20V2.sol` fixes drain path 2 by embedding the sender as the refund address (`"\x01", sender, amount_b, recipient`), but `EvmErc20.sol` remains deployed and callable. [7](#0-6) 

---

### Impact Explanation

Every successful call to `withdrawToNear` or `withdrawToEthereum` on `EvmErc20` that results in a failed precompile call or a failed NEAR promise permanently destroys ERC-20 mirror tokens without crediting the user on the other side. The ERC-20 `totalSupply` shrinks while the backing NEP-141 supply on NEAR is unchanged, creating an irreversible accounting divergence. Remaining holders cannot fully exit because the supply has been drained below the amount needed to redeem all backing tokens. This is **permanent freezing/loss of user funds** — Critical severity.

---

### Likelihood Explanation

Any unprivileged EVM user holding `EvmErc20` tokens can trigger this by calling `withdrawToNear` with a NEAR recipient account that has no storage deposit for the NEP-141 token (a common condition for new accounts). The NEAR-side `ft_transfer` will fail, and without `error_refund` compiled in, no refund occurs. This is a realistic, low-effort scenario requiring no special privileges.

---

### Recommendation

1. **Check the precompile return value** in `EvmErc20.sol`: revert if `res == 0` so the burn is rolled back on synchronous failure.
2. **Migrate to `EvmErc20V2.sol`** (which embeds the sender as refund address) and ensure the engine is compiled with `error_refund` enabled so asynchronous NEAR promise failures trigger a re-mint callback.
3. **Invert the operation order**: schedule the NEAR promise first and only burn on confirmed success, or use a two-phase commit pattern.

---

### Proof of Concept

1. Alice holds 1000 `EvmErc20` tokens (backed by NEP-141 on NEAR).
2. Alice calls `withdrawToNear("alice.near", 1000)`.
3. `_burn(alice, 1000)` executes — Alice's ERC-20 balance is now 0, `totalSupply` drops by 1000.
4. The assembly `call()` to the `exit_to_near` precompile fails (e.g., `alice.near` has no storage deposit for the NEP-141 token → `ft_transfer` promise fails on NEAR).
5. Without `error_refund`: no callback is registered, no re-mint occurs.
6. Alice has 0 ERC-20 tokens and 0 NEP-141 tokens. The 1000 tokens are permanently destroyed.
7. Other holders now cannot fully redeem their tokens because the ERC-20 supply has been drained below the backing amount. [1](#0-0) [8](#0-7)

pub fn refund_on_error<I: IO + Copy, E: Env, P: PromiseHandler>(
    io: I,
    env: &E,
    state: EngineState,
    args: &RefundCallArgs,
    handler: &mut P,
) -> EngineResult<SubmitResult> {
    let current_account_id = env.current_account_id();
    if let Some(erc20_address) = args.erc20_address {
        // ERC-20 exit; re-mint burned tokens
        let erc20_admin_address = current_address(&current_account_id);
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, erc20_admin_address, current_account_id, io, env);

        let refund_address = args.recipient_address;
        let amount = U256::from_big_endian(&args.amount);
        let input = setup_refund_on_error_input(amount, refund_address);

        engine.call(
            &erc20_admin_address,
            &erc20_address,
            Wei::zero(),
            input,
            u64::MAX,
            Vec::new(),
            Vec::new(),
            handler,
        )
    } else {
        // ETH exit; transfer ETH back from precompile address
        let exit_address = exit_to_near::ADDRESS;
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, exit_address, current_account_id, io, env);
        let refund_address = args.recipient_address;
        let amount = Wei::new(U256::from_big_endian(&args.amount));
        engine.call(
            &exit_address,
            &refund_address,
            amount,
            Vec::new(),
            u64::MAX,
            vec![
                (exit_address.raw(), Vec::new()),
                (refund_address.raw(), Vec::new()),
            ],
            Vec::new(),
            handler,
        )
    }
