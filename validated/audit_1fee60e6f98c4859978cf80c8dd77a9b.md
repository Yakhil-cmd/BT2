### Title
COA Ownership Proof Accepts Arbitrary Signed Data Without Binding to EVM Address — (`fvm/evm/stdlib/contract.cdc`, `fvm/evm/precompiles/arch.go`)

---

### Summary

`validateCOAOwnershipProof` explicitly does not enforce that `signedData` encodes `evmAddress`. An attacker who possesses any valid `FLOW-V0.0-user`-tagged signature over arbitrary data for a victim's Flow account keys can construct a well-formed proof, pass the victim's real COA address as the `evmAddress` argument, and receive `isValid: true` — without the victim ever having intended to prove ownership of that COA.

---

### Finding Description

**Full call chain:**

```

  → staticcall(archAddress, verifyCOAOwnershipProof(victimCOAAddr, arbitraryBytes32, encodedProof))

      keyList.verify(signedData=arbitraryBytes32, domainSeparationTag="FLOW-V0.0-user")
      → PASS  (attacker holds a valid sig over arbitraryBytes32)
      coaRef.address().bytes == evmAddress
      → PASS  (attacker supplies the victim's real COA address)
```

**Root cause — `contract.cdc` lines 1003–1009 and 1082–1086:**

The function explicitly documents the missing binding:

> "Note: this function does not enforce that `signedData` includes `evmAddress`. In principle, a signature produced for one purpose could be replayed here against a different COA owned by the same Cadence account." [1](#0-0) 

The signature verification only checks that the signatures are valid over `signedData`; it never checks that `signedData` commits to `evmAddress`: [2](#0-1) 

The subsequent COA address check only verifies that the `evmAddress` argument matches the on-chain COA — it does not verify that `signedData` encodes `evmAddress`: [3](#0-2) 

**Decoding path — `arch.go` lines 219–243:**

`DecodeABIEncodedProof` places the raw `bytes32` argument directly into `SignedData` with no content validation: [4](#0-3) 

**Cadence value mapping — `proof.go` lines 120–129:**

`ToCadenceValues()` passes `SignedData` (the attacker-controlled `bytes32`) verbatim as the `signedData` argument to `validateCOAOwnershipProof`: [5](#0-4)

### Impact Explanation

The COA contract's `isValidSignature` (ERC-1271) calls `verifyCOAOwnershipProof(address(this), _hash, _sig)` and returns `ValidERC1271Signature` when the precompile returns `true`: [6](#0-5) 

An attacker who has obtained any `FLOW-V0.0-user`-tagged signature over any data `D` for the victim's Flow account keys (e.g., from a prior off-chain authentication flow, a different COA ownership proof, or any other user-signed message) can:

1. Construct `encodedProof` containing the victim's Flow address, public capability path (`/public/coa`), the recycled key indices, and the recycled signatures over `D`.
2. Call `verifyCOAOwnershipProof(victimCOAAddress, D, encodedProof)` from an attacker-controlled EVM contract.
3. Receive `true`, causing any EVM contract relying on ERC-1271 or direct `verifyCOAOwnershipProof` checks to treat the attacker as the legitimate owner of the victim's COA.

This breaks the invariant that a COA ownership proof cryptographically binds the EVM address to the signed data, enabling unauthorized impersonation of a victim's COA in EVM-side authorization flows.

---

### Likelihood Explanation

The precondition — a valid `FLOW-V0.0-user` signature over any data for the victim's account — is realistic:

- Flow wallets routinely produce `FLOW-V0.0-user`-tagged signatures for off-chain authentication (e.g., wallet-connect flows, dApp sign-in).
- Any prior COA ownership proof the victim produced for a different purpose (different `signedData`) is directly replayable.
- The victim's COA EVM address and capability path (`/public/coa`) are public on-chain.
- The attacker does not need any privileged node access; the entire attack is submitted as a normal EVM transaction or Cadence script.

---

### Recommendation

Inside `validateCOAOwnershipProof`, after resolving the COA address, verify that `signedData` encodes `evmAddress` before accepting the proof. Concretely, require that `signedData` is (or contains) the ABI/RLP encoding of `evmAddress`, or define a canonical commitment scheme (e.g., `keccak256(abi.encode(evmAddress, nonce))`) and enforce it server-side. The EVM-side precompile already passes `address(this)` as `evmAddress`, so enforcing this binding on the Cadence side closes the replay window without breaking the normal COA `isValidSignature` flow.

---

### Proof of Concept

On a local Flow emulator / FLITE:

1. Create a Flow account `A` with a COA at `/public/coa`. Record the COA's EVM address `X`.
2. Have account `A` sign an arbitrary 32-byte message `D` (not encoding `X`) with tag `FLOW-V0.0-user`. This simulates any prior off-chain authentication signature.
3. RLP-encode a `COAOwnershipProof{Address: A, CapabilityPath: "coa", KeyIndices: [0], Signatures: [sig_over_D]}`.
4. From an attacker-controlled EVM contract, call:
   ```solidity
   cadenceArch.staticcall(
     abi.encodeWithSignature(
       "verifyCOAOwnershipProof(address,bytes32,bytes)",
       X,          // victim's real COA address
       D,          // arbitrary data — does NOT encode X
       encodedProof
     )
   )
   ```
5. Decode the return value. It will be `true`, demonstrating that a signature over unrelated data is accepted as a valid COA ownership proof for address `X`.

```go
func DecodeABIEncodedProof(input []byte) (*types.COAOwnershipProofInContext, error) {
	index := 0
	caller, err := ReadAddress(input, index)
	index += FixedSizeUnitDataReadSize
	if err != nil {
		return nil, err
	}

	hash, err := ReadBytes32(input, index)
	index += Bytes32DataReadSize
	if err != nil {
		return nil, err
	}

	encodedProof, err := ReadBytes(input, index)
	if err != nil {
		return nil, err
	}

	return types.NewCOAOwnershipProofInContext(
		hash,
		types.Address(caller),
		encodedProof,
	)
}
```
