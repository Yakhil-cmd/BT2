### Title
ETH Sent to `tick()` Is Permanently Locked When No Minting Is Required — (`L1/starkware/solidity/stake/RewardSupplier.sol`)

---

### Summary

`RewardSupplier.tick()` is declared `payable` and relies on `msg.value` to cover L1→L2 messaging fees. When no pending mint requests exist (`amountToMint == 0`), the function exits immediately without consuming or refunding the ETH. Because the contract has no `receive()`, `fallback()`, or ETH-recovery function, any ETH sent in that call is permanently locked.

---

### Finding Description

`tick()` is an unrestricted, publicly callable function:

```solidity
function tick() external payable {
    (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

    if (amountToMint > 0) {
        // ... uses msg.value for bridge and messaging fees ...
        uint256 msgFee = msg.value / 2;
        bridge().depositWithMessage{value: msgFee}(...);
        msgFee = msg.value - msgFee;
        messagingContract().sendMessageToL2{value: msgFee}(...);
    }
    // ← if amountToMint == 0, msg.value is silently retained
}
``` [1](#0-0) 

Two distinct loss paths exist:

**Path 1 — No pending mint requests (`amountToMint == 0`):** The entire `msg.value` is never touched. The `if` block is skipped and the function returns. The ETH stays in the contract forever.

**Path 2 — Pending requests exist but caller over-pays:** `msg.value` is split as `msgFee = msg.value / 2` for the bridge call and `msg.value - msgFee` for the messaging call. The full `msg.value` is forwarded to external contracts. If those contracts require less than `msg.value`, any change they return lands back in `RewardSupplier` with no mechanism to forward it to the original caller. [2](#0-1) 

A search across all `.sol` files confirms there is no `receive()`, `fallback()`, `withdrawETH`, or any ETH-rescue function anywhere in the contract or its storage base. [3](#0-2) 

---

### Impact Explanation

ETH sent by any caller to `tick()` when `amountToMint == 0` is permanently frozen inside the contract with no recovery path. This constitutes a **permanent freezing of user funds** (the caller's ETH). The contract accumulates ETH over time from every such call, and no privileged role can retrieve it.

Mapped allowed impact: **High — Permanent freezing of funds / Temporary freezing of funds.**

---

### Likelihood Explanation

- `tick()` has **no access control** — any EOA or contract may call it.
- Callers are incentivised to call `tick()` to trigger reward minting; they must supply ETH for the L1→L2 fee. A caller who checks `requiredMinting()` off-chain and finds `amountToMint == 0` but still sends ETH (e.g., due to a race condition where another caller processes the last message first) loses their ETH.
- Race conditions are realistic: multiple bots or keepers may monitor the same pending-message queue and race to call `tick()`. The loser's ETH is locked.
- Likelihood: **Medium** (requires ETH to be sent, but the race-condition scenario is realistic in a live keeper ecosystem).

---

### Recommendation

Refund any unused ETH at the end of `tick()`:

```solidity
function tick() external payable {
    (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

    if (amountToMint > 0) {
        // ... existing logic ...
    }

    // Refund any unused ETH to the caller.
    uint256 remaining = address(this).balance;
    if (remaining > 0) {
        (bool ok, ) = msg.sender.call{value: remaining}("");
        require(ok, "ETH_REFUND_FAILED");
    }
}
```

Alternatively, revert if `msg.value > 0` and `amountToMint == 0`, so callers are not silently drained.

---

### Proof of Concept

1. Observe that `requiredMinting()` returns `(0, 0)` (no pending L2→L1 mint-request messages).
2. Call `tick{value: 1 ether}()` from any EOA.
3. The `if (amountToMint > 0)` block is skipped.
4. `address(RewardSupplier).balance` increases by `1 ether`.
5. No function exists to recover the ETH — it is permanently locked. [1](#0-0)

    function tick() external payable {
        // Check if minting is required, and how much.
        (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

        if (amountToMint > 0) {
            // Prepare the L2->L1 mintRequest message for consumption.
            uint256[] memory messagePayload = new uint256[](1);
            messagePayload[0] = TOKENS_PER_MINT_REQUEST;

            // Consume the mintRequest messages.
            for (uint256 i = 0; i < numMsgsToConsume; i++) {
                messagingContract().consumeMessageFromL2(mintRequestSource(), messagePayload);
            }

            // Reuest minting of the requested amount from the mint manager.
            mintManager().mintRequest(token(), amountToMint);

            // Deposit the minted amount onto the bridge to the credit of `mintDestination`.
            uint256 msgFee = msg.value / 2;
            bridge().depositWithMessage{value: msgFee}(
                token(),
                amountToMint,
                mintDestination(),
                new uint256[](0)
            );
            emit ConsumedL2MintRequests(numMsgsToConsume, amountToMint);

            // Send a totalSupply update to L2MintCurve.
            msgFee = msg.value - msgFee;
            messagePayload[0] = IERC20(token()).totalSupply();
            messagingContract().sendMessageToL2{value: msgFee}(
                mintingCurve(),
                UPDATE_TOTAL_SUPPLY_SELECTOR,
                messagePayload
            );
        }
    }
