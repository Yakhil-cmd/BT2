### Title
Unconditional Zero-Value BTC Token Transfer in `transfer_to_pools_when_unstake` Can Permanently Freeze Staker Funds - (File: src/staking/staking.cairo)

---

### Summary
When a staker calls `unstake_action`, the internal helper `transfer_to_pools_when_unstake` unconditionally transfers each BTC pool's delegated balance to the corresponding pool contract — including when that balance is zero. If the BTC token's ERC20 implementation reverts on zero-value transfers, `unstake_action` permanently reverts, freezing the staker's STRK principal in the staking contract.

---

### Finding Description
In `transfer_to_pools_when_unstake` (staking.cairo lines 1661–1681), for every pool associated with the unstaking staker the function:

1. Reads `pool_balance = self.get_delegated_balance(:staker_address, :pool_contract)`.
2. Calls `token_dispatcher.checked_transfer(recipient: pool_contract, amount: pool_balance.to_native_amount(:decimals).into())` **unconditionally** — there is no `if pool_balance > 0` guard.

```cairo
// src/staking/staking.cairo ~1661-1681
fn transfer_to_pools_when_unstake(...) {
    for (pool_contract, token_address) in staker_pool_info.pools {
        let pool_balance = self.get_delegated_balance(:staker_address, :pool_contract);
        let token_dispatcher = IERC20Dispatcher { contract_address: token_address };
        let pool_dispatcher = IPoolDispatcher { contract_address: pool_contract };
        pool_dispatcher.set_staker_removed();
        self.insert_staker_delegated_balance(
            :staker_address, :pool_contract, delegated_balance: Zero::zero(),
        );
        let decimals = self.get_token_decimals(:token_address);
        token_dispatcher.checked_transfer(          // ← no zero-guard
            recipient: pool_contract,
            amount: pool_balance.to_native_amount(:decimals).into(),
        );
    }
}
```

`pool_balance` is legitimately zero whenever all delegators have already exited the pool before the staker calls `unstake_action`. Each delegator's `exit_delegation_pool_action` calls `staking_pool_dispatcher.remove_from_delegation_pool_action(...)`, which reduces the staking contract's tracked delegated balance to zero. After that point, `get_delegated_balance` returns zero for that pool.

BTC tokens are external ERC20 tokens added by governance. A non-trivial subset of real-world ERC20 implementations (including some wrapped-BTC tokens) revert on zero-value transfers. The staking contract imposes no requirement that approved BTC tokens must accept zero-value transfers, and `checked_transfer` does not add such a guard.

---

### Impact Explanation
If `transfer_to_pools_when_unstake` reverts, `unstake_action` reverts entirely. The staker's STRK principal remains locked in the staking contract with no alternative exit path. Because the staker cannot re-enter the exit window (the intent was already consumed) and cannot bypass `transfer_to_pools_when_unstake`, the freeze is permanent unless the BTC token's behavior changes — which is outside the staker's control.

**Impact: Permanent freezing of staker funds (High).**

---

### Likelihood Explanation
- A staker opens a BTC pool (a normal, documented operation).
- All delegators exit before the staker (a normal, expected sequence — delegators are free to exit at any time).
- The governance-approved BTC token reverts on zero-value transfers (a known, documented ERC20 variant; the protocol imposes no constraint against it).

No privileged access, key compromise, or bridge manipulation is required. Any staker with an empty BTC pool is affected.

---

### Recommendation
Add a zero-amount guard before each BTC token transfer in `transfer_to_pools_when_unstake`:

```cairo
let native_amount = pool_balance.to_native_amount(:decimals);
if native_amount > 0 {
    token_dispatcher.checked_transfer(
        recipient: pool_contract,
        amount: native_amount.into(),
    );
}
```

Apply the same pattern to every other unconditional `checked_transfer` call where the amount may be zero (e.g., `send_rewards_to_staker`, `send_rewards_to_delegation_pool`, and `pool.claim_rewards`), so the codebase is uniformly defensive against this ERC20 variant.

---

### Proof of Concept

1. Governance approves a BTC token whose `transfer(to, 0)` reverts.
2. Staker stakes STRK and opens a BTC pool for that token (`set_open_for_delegation`).
3. Delegator delegates BTC to the pool.
4. Delegator calls `exit_delegation_pool_intent` then `exit_delegation_pool_action` → staking contract's `delegated_balance` for that pool drops to **0**.
5. Staker calls `unstake_intent` (succeeds).
6. After the exit window, staker calls `unstake_action`.
7. `unstake_action` calls `transfer_to_pools_when_unstake`.
8. `pool_balance = 0` → `checked_transfer(pool_contract, 0)` → BTC token reverts.
9. `unstake_action` reverts. Staker's STRK is **permanently frozen**.
