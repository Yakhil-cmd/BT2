### Title
Balance Permanently Burnt When `DeleteAccount` Specifies a Non-Existent Named Beneficiary - (File: `runtime/runtime/src/actions.rs`)

### Summary
`action_delete_account` unconditionally emits a `new_balance_refund` receipt to `delete_account.beneficiary_id` without verifying that the beneficiary account exists. If the beneficiary is a syntactically valid but non-existent named account, the refund receipt's `Transfer` action fails at execution time, and the protocol's own rule — "if the execution of a refund fails, the refund amount is burnt" — permanently destroys the entire account balance.

### Finding Description

`validate_delete_action` performs only syntactic validation of `beneficiary_id`:

```rust
fn validate_delete_action(action: &DeleteAccountAction) -> Result<(), ActionsValidationError> {
    validate_action_account_id(&action.beneficiary_id)?;
    Ok(())
}
``` [1](#0-0) 

No existence check is performed. `action_delete_account` then unconditionally pushes a balance-refund receipt addressed to the user-supplied `beneficiary_id`:

```rust
let account_balance = account_ref.amount();
if account_balance > Balance::ZERO {
    result
        .new_receipts
        .push(Receipt::new_balance_refund(&delete_account.beneficiary_id, account_balance));
}
``` [2](#0-1) 

`Receipt::new_balance_refund` sets `predecessor_id = "system"`:

```rust
pub fn new_balance_refund(receiver_id: &AccountId, refund: Balance) -> Self {
    Receipt::V0(ReceiptV0 {
        predecessor_id: "system".parse().unwrap(),
        receiver_id: receiver_id.clone(),
        ...
        actions: vec![Action::Transfer(TransferAction { deposit: refund })],
    })
}
``` [3](#0-2) 

When this receipt is processed and the beneficiary does not exist, `check_transfer_to_nonexisting_account` returns `AccountDoesNotExist` for any non-implicit named account. The code itself documents this outcome:

```rust
// - Refunds don't automatically create accounts, because refunds are free and
// we don't want some type of abuse.
// - Account deletion with beneficiary creates a refund, so it'll not create a
// new account.
``` [4](#0-3) 

The protocol rule is explicit:

> If the execution of a refund fails, the refund amount is burnt. [5](#0-4) 

The failed refund receipt then generates a secondary refund to `predecessor_id = "system"`, which also has no account, causing the amount to be burnt a second time — the balance is permanently destroyed.

### Impact Explanation

The entire liquid balance of the deleted account (`account_ref.amount()`) is permanently removed from the total supply. The account is already deleted before the refund receipt is dispatched, so there is no rollback. The loss is irreversible.

### Likelihood Explanation

Any account owner can submit a `DeleteAccount` action with an arbitrary `beneficiary_id`. A typo, a recently-deleted account, or a programmatically constructed account ID that was never created are all sufficient to trigger the loss. The action is accepted at the transaction-validation layer because only syntactic validity is checked. The failure is silent at submission time — the transaction succeeds, the account is deleted, and only the subsequent refund receipt fails.

### Recommendation

Add an existence check for `beneficiary_id` inside `action_delete_account` before emitting the refund receipt. The check should mirror the logic already used for `Action::Transfer` — allow implicit-format accounts (which can be created by transfer) but reject non-existent named accounts:

```rust
// In action_delete_account, before pushing the refund receipt:
let beneficiary_exists = get_account(state_update, &delete_account.beneficiary_id)?.is_some();
let beneficiary_is_implicit = account_is_implicit(
    &delete_account.beneficiary_id,
    config.wasm_config.eth_implicit_accounts,
);
if !beneficiary_exists && !beneficiary_is_implicit {
    result.result = Err(ActionErrorKind::AccountDoesNotExist {
        account_id: delete_account.beneficiary_id.clone(),
    }.into());
    return Ok(());
}
```

Alternatively, add the same existence guard inside `validate_delete_action` at the receipt-validation layer so the action is rejected before execution.

### Proof of Concept

1. Account `alice.near` holds 100 NEAR.
2. `alice.near` submits a transaction:
   ```
   DeleteAccount { beneficiary_id: "does-not-exist.near" }
   ```
   `does-not-exist.near` is syntactically valid but has never been created.
3. Transaction is accepted (`validate_delete_action` passes — only syntax is checked). [1](#0-0) 
4. `action_delete_account` executes: `alice.near` is removed from state; a `new_balance_refund` receipt for 100 NEAR is queued to `does-not-exist.near`. [6](#0-5) 
5. The refund receipt is processed. `check_account_existence` calls `check_transfer_to_nonexisting_account`; `does-not-exist.near` is not implicit and does not exist → `AccountDoesNotExist` error. [7](#0-6) 
6. The failed refund receipt generates a secondary refund to `predecessor_id = "system"`. "system" has no account; that transfer also fails.
7. Per protocol rules, the 100 NEAR is burnt. `alice.near`'s balance is permanently lost. [5](#0-4)
