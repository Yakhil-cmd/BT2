Audit Report

## Title
Unbounded Per-Peer Slot Table Growth via `SLOT_TABLE_NO_LIMIT` Enables OOM Crash of Honest Replicas — (`rs/replica/setup_ic_network/src/lib.rs`, `rs/p2p/consensus_manager/src/receiver.rs`)

## Summary

Five of six artifact channels (consensus, certifier, dkg, idkg, https_outcalls) are wired with `SLOT_TABLE_NO_LIMIT = usize::MAX` as their per-peer slot table limit. A single Byzantine subnet peer can exploit this by advertising an unbounded stream of distinct slot IDs on any of those channels, causing `ConsensusManagerReceiver`'s `slot_table`, `active_assembles`, and `artifact_processor_tasks` to grow without bound until the replica process is OOM-killed and drops out of consensus participation.

## Finding Description

**Constant definitions** — `rs/replica/setup_ic_network/src/lib.rs` lines 74–75 define `SLOT_TABLE_LIMIT_INGRESS = 50_000` and `SLOT_TABLE_NO_LIMIT = usize::MAX`. The comment at lines 72–73 explicitly acknowledges the ingress limit exists to protect against malicious peers, yet the same protection is absent for the other five channels.

**Channel wiring** — `AbortableBroadcastChannels::new` passes `SLOT_TABLE_NO_LIMIT` to `abortable_broadcast_channel` for consensus (lines 237, 246), certifier (line 268), dkg (line 279), idkg (line 291), and https_outcalls (line 303). Only ingress (line 257) receives the bounded `SLOT_TABLE_LIMIT_INGRESS`.

**Guard failure** — In `handle_slot_update_receive` (`receiver.rs` line 373), the guard `Entry::Vacant(empty_slot) if peer_slot_table_len < self.slot_limit` is effectively unconditional when `slot_limit = usize::MAX`. Every new `Entry::Vacant` slot is inserted without restriction.

**Unbounded task spawning** — For each new slot with a previously-unseen artifact ID, lines 405–419 insert into `active_assembles` and spawn a new Tokio task via `artifact_processor_tasks.spawn_on(...)`. All three data structures (`slot_table`, `active_assembles`, `artifact_processor_tasks`) grow proportionally to the number of distinct `(slot_number, artifact_id)` pairs the attacker sends, consuming heap and Tokio task memory until the OS kills the process.

## Impact Explanation

A single Byzantine subnet node can OOM-crash any honest replica it connects to by flooding it with distinct slot IDs on the certification, DKG, IDKG, or HTTPS-outcalls channel. The replica process is killed by the OS, causing it to drop out of consensus participation until it restarts and re-syncs state. This constitutes an **application/platform-level DoS and consensus availability impact not based on raw volumetric DDoS**, matching the **High ($2,000–$10,000)** impact tier.

## Likelihood Explanation

The attacker must be a TLS-authenticated subnet member (node operator with a registry-issued certificate). A single compromised or malicious node operator below the `f`-fault threshold suffices — no threshold corruption is required. The attack is cheap to execute (sending `Update::Id(unique_id)` messages requires no valid artifact content), is repeatable across restarts, and produces no log warning before OOM because the `slot_table_limit_exceeded_total` metric and `warn!` log path at lines 381–389 are never reached when `slot_limit = usize::MAX`.

## Recommendation

Replace `SLOT_TABLE_NO_LIMIT` with per-channel bounded constants derived from protocol-level bounds on the maximum number of simultaneously active artifacts per peer for each artifact type. The ingress precedent (`SLOT_TABLE_LIMIT_INGRESS = 50_000`) demonstrates the pattern is already understood and implemented. Appropriate limits for consensus, certification, DKG, IDKG, and HTTPS-outcalls should reflect the maximum number of legitimately active artifacts per peer (e.g., number of active DKG transcripts, IDKG dealings, certification shares, HTTPS outcall responses per round).

## Proof of Concept

The existing test infrastructure in `receiver.rs` (lines 846–887) already demonstrates the bounding mechanism works correctly with a finite limit. A minimal reproduction using the same `ReceiverManagerBuilder` pattern:

```rust
// Reproduces production behavior for cert/dkg/idkg/https channels:
let (mut mgr, _) = ReceiverManagerBuilder::new()
    .with_slot_limit(usize::MAX)  // matches production for all non-ingress channels
    .build();
let cancel = CancellationToken::new();
for i in 0u64..1_000_000 {
    mgr.handle_slot_update_receive(
        SlotUpdate {
            slot_number: SlotNumber::from(i),
            commit_id: CommitId::from(i),
            update: Update::Id(i),  // distinct ID each time
        },
        NODE_1,
        ConnId::from(1),
        cancel.clone(),
    );
}
// active_assembles.len() == 1_000_000
// artifact_processor_tasks.len() == 1_000_000
```

Differential: repeat with `slot_limit = 50_000` — the loop terminates with exactly 50,000 entries and all subsequent updates are silently dropped, confirming the fix is straightforward.
