### Title
Missing `msg_hash` Binding in `RerandomizedPresignOutput` Enables Coordinator-Driven Split-View Key Extraction — (`File: src/ecdsa/robust_ecdsa/mod.rs`, `src/ecdsa/robust_ecdsa/sign.rs`)

---

### Summary

The robust ECDSA signing pipeline splits rerandomization (offline) from signing (online). A malicious coordinator can present different `(msg_hash, tweak)` values to different participants during the rerandomization step, causing each participant to compute their signature share under a different nonce scalar `delta`. Because `RerandomizedPresignOutput` does not record the `msg_hash` used to derive `delta`, and `sign()` accepts an arbitrary `msg_hash` with no consistency check, the coordinator collects shares whose nonces are multiplicatively related. Two such sessions suffice to reconstruct the aggregate secret key via standard ECDSA nonce-reuse algebra.

---

### Finding Description

**Rerandomization** is performed by each participant independently, outside the signing protocol, by calling `RerandomizedPresignOutput::rerandomize_presign()`. The randomness scalar `delta` is derived as:

```
delta = HKDF(entropy; pk || tweak || msg_hash || big_r || participants)
``` [1](#0-0) 

The resulting `RerandomizedPresignOutput` stores only the transformed shares `(big_r, alpha, beta, e)`: [2](#0-1) 

**Critically, `msg_hash` is not stored in `RerandomizedPresignOutput`.** When `sign()` is later called, it accepts `msg_hash` as a free parameter: [3](#0-2) 

The function enforces `participants.len() == 2*max_malicious+1` and `msg_hash != 0`, but performs **no check that the supplied `msg_hash` matches the one used to derive `delta` during rerandomization**: [4](#0-3) 

Each participant's signature share is then computed as:

```
s_i = alpha_rerandomized * msg_hash + beta_rerandomized * Rx + e
``` [5](#0-4) 

where `alpha_rerandomized = alpha / delta` and `delta` was derived from a potentially different `msg_hash` than the one now passed to `sign()`.

The security documentation explicitly identifies this as the attack surface: [6](#0-5) 

---

### Impact Explanation

A malicious coordinator orchestrates a single signing session with `N = 2t+1` participants. During the offline rerandomization step, the coordinator instructs participant subset `A` to rerandomize with `(h₁, ε₁)` and participant subset `B` to rerandomize with `(h₂, ε₂)`. Both subsets then call `sign()` with their respective `msg_hash` values. The coordinator collects shares:

- From subset A: `s_A = (alpha/delta_A) * h₁ + ...`
- From subset B: `s_B = (alpha/delta_B) * h₂ + ...`

Because `delta_A ≠ delta_B`, the effective nonces are multiplicatively related. With two such sessions, the coordinator can solve for the secret key `x` using standard ECDSA nonce-reuse linear algebra — as documented in the codebase itself: [7](#0-6) 

**Impact**: Critical — full extraction of the aggregate ECDSA secret key.

---

### Likelihood Explanation

The coordinator role is a documented trust assumption in this library. The `sign()` API is a public library function; any caller who controls the coordinator and can influence the rerandomization arguments of individual participants (e.g., in a networked MPC node deployment) can execute this attack. No cryptographic primitive break is required. The attack requires only two signing sessions and `2t+2` presigning participants, which is a minimal deviation from normal operation.

---

### Recommendation

1. **Bind `msg_hash` inside `RerandomizedPresignOutput`**: Store the `msg_hash` (and `tweak`) used during rerandomization as fields of `RerandomizedPresignOutput`.

2. **Verify consistency in `sign()`**: At the start of `sign()`, assert that the `msg_hash` argument equals the one stored in the `RerandomizedPresignOutput`. This makes it impossible for a coordinator to supply a mismatched hash.

3. **Bind entropy to the signing session**: Require that `entropy` is derived from a shared, publicly-agreed-upon source (e.g., a commit-reveal or VRF output visible to all participants), so the coordinator cannot supply different entropy to different participants.

---

### Proof of Concept

1. Coordinator runs presigning with `N = 2t+1` participants, obtaining `PresignOutput` for each.
2. Coordinator instructs participants `{P_1, …, P_t}` to call `rerandomize_presign()` with `RerandomizationArguments { msg_hash: h1, tweak: ε1, entropy: ρ1, … }`.
3. Coordinator instructs participants `{P_{t+1}, …, P_{2t+1}}` to call `rerandomize_presign()` with `RerandomizationArguments { msg_hash: h2, tweak: ε2, entropy: ρ2, … }`.
4. All participants call `sign()` with their respective `msg_hash` values. Because `RerandomizedPresignOutput` does not record which `msg_hash` was used, `sign()` accepts both without error.
5. Coordinator collects `s_i` shares from all `2t+1` participants. The effective nonces satisfy `k_A / k_B = delta_B / delta_A` (a known ratio), giving two linear equations in the secret key `x`.
6. Coordinator solves the system and recovers `x` in full. [8](#0-7) [9](#0-8)

impl RerandomizedPresignOutput {
    pub fn rerandomize_presign(
        presignature: &PresignOutput,
        args: &RerandomizationArguments,
    ) -> Result<Self, ProtocolError> {
        if presignature.big_r != args.big_r {
            return Err(ProtocolError::IncompatibleRerandomizationInputs);
        }
        let delta = args.derive_randomness()?;
        if delta.is_zero().into() {
            return Err(ProtocolError::ZeroScalar);
        }

        // cannot be zero due to the previous check
        let inv_delta = delta.invert().unwrap();

        // delta * R
        let rerandomized_big_r = presignature.big_r * delta;

        // alpha * delta^{-1}
        let rerandomized_alpha = presignature.alpha * inv_delta;

        // (beta + c*tweak) * delta^{-1}
        let rerandomized_beta =
            (presignature.beta + presignature.c * args.tweak.value()) * inv_delta;

        Ok(Self {
            big_r: rerandomized_big_r.into(),
            alpha: rerandomized_alpha,
            beta: rerandomized_beta,
            e: presignature.e,
        })
    }

be aware that it is vulnerable to **split-view attacks** in the robust setting when the
signing parameters are not globally consistent. If different subsets of size at least
$2t + 1$ sign different $(h, \epsilon)$ values using shares derived from the same
presignature, the resulting signatures use multiplicatively related nonces and the
secret key can be recovered using standard ECDSA nonce-reuse attacks.

Moreover, due to protocol modifications relative to [[DJNPO20](https://eprint.iacr.org/2020/501)] (notably signature-share
linearization), **a novel split-view attack exists that can extract the secret key using as
few as $2t + 2$ presigning participants**, with as few as two signing sessions.

To reduce the risk of accidental misuse, enforce the following constraints:

1. **Use exactly $N_1 = N_2 = 2t + 1$ participants for both presigning and signing.**
   Do **not** allow any deviation from this value. In particular:

   * Do **not** allow $N_1 > 2t + 1$, and
   * Do **not** allow $N_2 < N_1$.

   Allowing larger presigning sets or smaller signing sets enables split-view and
   presignature-reuse attacks when a coordinator can run parallel or partially overlapping
   signing sessions.

2. **Ensure all participants agree on $(h, \epsilon)$ and the signing set.**
   The coordinator must not be able to present different message hashes, tweaks, or
   participant lists to different signers.
