### Title
Cross-Chain Signature Replay via Unprotected Legacy Transactions - (File: `basic_bootloader/src/bootloader/transaction/rlp_encoded/transaction_types/legacy_tx.rs`)

### Summary
ZKsync OS accepts unprotected legacy transactions (v=27/28) whose signing hash is computed **without** the chain ID. A signature produced on any other EVM chain for such a transaction is equally valid on ZKsync OS, enabling cross-chain replay that can drain a victim's funds.

### Finding Description
In `LegacyPayloadParser::try_parse_and_hash_for_signature_verification`, when `legacy_signature.is_eip155()` returns `false` (i.e., `v == 27 || v == 28`), the signing hash is computed over only the six-field payload `[nonce, gasPrice, gasLimit, to, value, data]` with no chain-specific context:

```rust
let sig_hash: Bytes32 = if legacy_signature.is_eip155() == false {
    // Unprotected legacy
    let mut hasher = crypto::sha3::Keccak256::new();
    apply_list_concatenation_encoding_to_hash(inner_slice.len() as u32, &mut hasher);
    hasher.update(inner_slice);
    hasher.finalize_reset().into()
``` [1](#0-0) 

The `is_eip155()` predicate:

```rust
pub fn is_eip155(&self) -> bool {
    self.v != 27 && self.v != 28
}
``` [2](#0-1) 

returns `false` for v=27/28, so the `Legacy` variant is stored and the chain-ID-free hash is returned to the caller.

The dispatch in `RlpEncodedTxInner::parse_and_compute_signed_hash` accepts both variants without rejecting the unprotected one:

```rust
let tx = if sig_data.is_eip155() {
    Self::LegacyWithEIP155(tx, sig_data)
} else {
    Self::Legacy(tx, sig_data)
};
Ok((tx, sig_hash))
``` [3](#0-2) 

Both the ZK-path and Ethereum-STF-path validation functions then verify the ECDSA signature against this chain-ID-free hash: [4](#0-3) [5](#0-4) 

Because the hash contains no chain-specific identifier, a signature that was valid on Ethereum mainnet (or any other EVM chain) for the same `(nonce, gasPrice, gasLimit, to, value, data)` tuple is equally valid on ZKsync OS.

### Impact Explanation
An attacker who observes a historical or pending unprotected legacy transaction on another EVM chain (e.g., Ethereum mainnet) can submit the identical raw bytes to ZKsync OS. If the victim's ZKsync OS account nonce matches the replayed transaction's nonce and the account holds funds, the transaction executes successfully — transferring value or invoking arbitrary calldata — without the victim's knowledge or consent. This is a direct, unprivileged public-funds-loss path.

### Likelihood Explanation
- Unprotected legacy transactions (v=27/28) still appear in historical Ethereum data and are produced by some older tooling and hardware wallets.
- A fresh ZKsync OS account starts at nonce 0, matching any replayed transaction with nonce 0.
- The attacker needs only to scan public chain history for unprotected legacy transactions whose sender also holds funds on ZKsync OS — no privileged access, no key material, no social engineering required.
- Likelihood is **medium**: the preconditions are non-trivial but entirely realistic for users who migrated funds from Ethereum mainnet to ZKsync OS.

### Recommendation
Reject unprotected legacy transactions (v=27/28) at the parsing layer. ZKsync OS is a distinct chain with its own chain ID; there is no legitimate reason to accept signatures that carry no chain binding. The `Legacy` variant in `RlpEncodedTxInner` should be removed and `parse_and_compute_signed_hash` should return `Err(InvalidTransaction::InvalidEncoding)` when `is_eip155()` is false, mirroring the approach already taken for all EIP-2718 typed transactions which unconditionally embed and verify `chain_id`.

### Proof of Concept
1. On Ethereum mainnet, Alice signs a legacy transfer with nonce=0, v=27 (unprotected).
2. Alice bridges ETH to ZKsync OS; her ZKsync OS account nonce is 0.
3. Attacker copies the raw RLP bytes of Alice's mainnet transaction.
4. Attacker submits those bytes to ZKsync OS as a new transaction.
5. `LegacyPayloadParser` computes `sig_hash = keccak256(rlp([nonce, gasPrice, gasLimit, to, value, data]))` — identical to the mainnet hash.
6. `secp256k1_ec_recover` recovers Alice's address; `recovered_from == from` passes.
7. Nonce check passes (both are 0); the transaction executes, draining Alice's ZKsync OS balance to the attacker-controlled `to` address. [6](#0-5) [7](#0-6)
