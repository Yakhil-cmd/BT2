### Title
Missing Zero-Address Validation for `reward_address` Allows Permanent Freezing of Unclaimed Yield - (File: src/staking/staking.cairo)

---

### Summary

The `stake()` and `change_reward_address()` functions in the Staking contract accept a `reward_address` parameter without validating that it is non-zero. A staker can register or update their reward address to the zero address (`0x0`). All subsequent `claim_rewards()` calls will silently transfer STRK rewards to address zero, permanently destroying them. This is the direct analog of the appchain registration bug: a critical address field is accepted without enforcing that it is a meaningful, non-null value.

---

### Finding Description

In `src/staking/staking.cairo`, the `stake()` function (lines 288–366) accepts `reward_address: ContractAddress` and performs the following checks before writing it to storage:

- `STAKER_EXISTS` — caller not already a staker
- `OPERATIONAL_EXISTS` — operational address not already in use
- `STAKER_IS_TOKEN` / `REWARD_ADDRESS_IS_TOKEN` / `OPERATIONAL_IS_TOKEN` — addresses are not registered token contracts
- `AMOUNT_LESS_THAN_MIN_STAKE` — amount meets minimum

There is **no assertion that `reward_address` is non-zero**. [1](#0-0) 

Similarly, `change_reward_address()` (lines 517–540) only checks that the new address is not a token contract, with no zero-address guard: [2](#0-1) 

When `claim_rewards()` executes, it reads `staker_info.reward_address` and calls `checked_transfer` to that address unconditionally: [3](#0-2) 

If `reward_address` is zero, the ERC-20 transfer succeeds (address zero is a valid recipient on StarkNet), and the tokens are permanently lost.

The `general_prerequisites()` helper only asserts that the **caller** is non-zero, not any of the supplied address parameters: [4](#0-3) 

---

### Impact Explanation

**High — Permanent freezing of unclaimed yield.**

Any staker who calls `stake(reward_address: 0, ...)` or later calls `change_reward_address(0)` will have all accrued STRK rewards transferred to address zero on every `claim_rewards()` invocation. The tokens are irrecoverably burned. There is no mechanism to recover funds sent to address zero on StarkNet. The staker's principal is unaffected, but all yield is permanently destroyed.

---

### Likelihood Explanation

**Medium.** The entry path requires no privilege — any unprivileged staker can trigger this. Accidental misuse (e.g., passing an uninitialized variable, a front-end bug, or a scripting error) is realistic. Intentional self-griefing or a social-engineering attack that convinces a staker to call `change_reward_address(0)` is also plausible. The function is publicly callable by any registered staker with no additional guard.

---

### Recommendation

Add an explicit non-zero assertion for `reward_address` in both `stake()` and `change_reward_address()`, mirroring the pattern already used for the caller check:

```cairo
// In stake() and change_reward_address():
assert!(reward_address.is_non_zero(), "{}", GenericError::ZERO_ADDRESS);
```

This is consistent with the `ZERO_ADDRESS` error string already defined in the error catalogue: [5](#0-4) 

---

### Proof of Concept

1. Attacker (or victim) calls `stake(reward_address: 0x0, operational_address: <valid>, amount: <min_stake>)`.
2. No assertion fires — `0x0` is not a registered token, not already in use, and the caller is non-zero.
3. Staker accrues rewards over time via attestation.
4. Anyone calls `claim_rewards(staker_address)`.
5. `send_rewards_to_staker` executes `checked_transfer(recipient: 0x0, amount: rewards)`.
6. Tokens are transferred to address zero and permanently lost.

The same outcome is achieved post-registration via `change_reward_address(0x0)` — no checks prevent it. [6](#0-5)
