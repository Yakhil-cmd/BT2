### Title
`send_many` Used Instead of `do_broadcast` for DKG Commitment Hash, Enabling Split-View Attack - (File: `src/dkg.rs`)

### Summary

In `do_keyshare`, the Round 1 commitment hash is disseminated via `send_many` (unreliable, point-to-point broadcast with no agreement guarantee) rather than `do_broadcast` (Byzantine Reliable Broadcast). A malicious participant can exploit this to send different commitment hashes to different honest participants. Because the Round 3 commitment is sent via `do_broadcast` (which guarantees all honest parties receive the same value), the hash-vs-commitment check in Round 4 will pass for some honest parties and fail for others, producing a split-view of the participant set and causing honest parties to complete DKG with inconsistent outputs.

### Finding Description

`do_keyshare` in `src/dkg.rs` implements the DKG protocol in five rounds. In Round 1 (Step 2.9), each participant sends a hash of their polynomial commitment to all others:

```rust
// Step 2.9
let wait_round_1 = chan.next_waitpoint();
chan.send_many(wait_round_1, &commitment_hash)?;   // ← unreliable broadcast
``` [1](#0-0) 

These hashes are collected into `all_hash_commitments` via `recv_from_others`, which accepts the first message from each sender and stores it locally per-participant.

In Round 3 (Step 3.2/4.1), the actual commitment and proof of knowledge are sent via `do_broadcast` — the Byzantine Reliable Broadcast — which guarantees all honest parties receive the **same** value:

```rust
let commitments_and_proofs_map = do_broadcast(
    &mut chan, &participants, me,
    (commitment, proof_of_knowledge),
).await?;
``` [2](#0-1) 

In Round 4, each honest participant cross-checks the Round 3 commitment against the Round 1 hash it stored locally:

```rust
verify_commitment_hash(
    &session_id, p,
    &mut commit_domain_separator.clone(),
    commitment_i,
    &all_hash_commitments,   // ← populated from send_many, not do_broadcast
)?;
``` [3](#0-2) 

Because `send_many` provides no agreement guarantee — unlike `do_broadcast`, which is defined in `echo_broadcast.rs` and guarantees that if any honest party delivers a message, all honest parties deliver the **same** message — a malicious participant `M` can send `hash_A` to honest party `A` and `hash_B ≠ hash_A` to honest party `B` in Round 1. In Round 3, `do_broadcast` forces `M` to commit to a single polynomial commitment `C` for all honest parties. The Round 4 check then evaluates `H(C) == hash_A` at `A` and `H(C) == hash_B` at `B`. Since `hash_A ≠ hash_B`, at most one can pass. The result is that some honest parties accept `M`'s contribution and proceed to a completed DKG output, while others abort — a classic split-view on the participant set. [4](#0-3) 

The `send_many` primitive is explicitly documented as "peer-to-peer sending with no security guarantees": [5](#0-4) 

### Impact Explanation

**High.** Honest parties that pass the Round 4 hash check complete the DKG and derive a public key that includes `M`'s polynomial contribution. Honest parties that fail the check abort and produce no output. The two groups hold inconsistent views of the participant set: one group has a valid key share for a key that includes `M`, the other has nothing. Any subsequent signing session will fail because the participant sets are mismatched. This directly matches: *"Corruption of DKG outputs so honest parties accept inconsistent participant sets or unusable cryptographic outputs."*

### Likelihood Explanation

**Moderate.** The attacker must be a registered participant in the DKG session (a malicious participant, which is an explicitly modeled threat in threshold protocols). No external assumptions, leaked keys, or cryptographic breaks are required. The attack is straightforward to execute: the malicious participant simply sends two different 32-byte hash values to two different honest parties in Round 1. The `send_many` mechanism provides no mechanism to detect or prevent this equivocation.

### Recommendation

Replace `send_many` with `do_broadcast` for the Round 1 commitment hash exchange in `do_keyshare`. This ensures all honest participants receive the same hash from each sender, eliminating the equivocation vector. The Round 3 commitment broadcast already correctly uses `do_broadcast`; the Round 1 hash broadcast must be held to the same standard since it is the binding anchor for the Round 4 integrity check.

```rust
// Step 2.9 — use do_broadcast instead of send_many
let commitment_hashes_map = do_broadcast(
    &mut chan, &participants, me, commitment_hash,
).await?;
// populate all_hash_commitments from commitment_hashes_map
```

### Proof of Concept

1. Set up a DKG session with participants `{A, B, M}` where `M` is malicious.
2. `M` computes two distinct hashes: `hash_A = H(C_real)` and `hash_B = H(C_fake)` where `C_real` is `M`'s actual commitment.
3. In Round 1, `M` sends `hash_A` to `A` and `hash_B` to `B` (exploiting `send_many`'s lack of agreement).
4. In Round 3, `M` uses `do_broadcast` to send `C_real` to all honest parties (forced by the reliable broadcast).
5. In Round 4:
   - `A` checks `H(C_real) == hash_A` → **passes**, `A` accepts `M`'s commitment and completes DKG.
   - `B` checks `H(C_real) == hash_B` → **fails** (since `hash_B = H(C_fake) ≠ H(C_real)`), `B` aborts.
6. `A` holds a valid key share for a public key derived from `{A, B, M}`'s polynomials; `B` has no output. The DKG outputs are permanently inconsistent across honest parties. [6](#0-5) [2](#0-1) [3](#0-2)

/// This reliable broadcast function is the echo-broadcast protocol from the sender side.
/// It broadcasts some data in a vote
pub fn reliable_broadcast_send<T>(
    chan: &SharedChannel,
    wait: Waitpoint,
    participants: &ParticipantList,
    me: Participant,
    data: T,
) -> Result<MessageType<T>, ProtocolError>
where
    T: Serialize,
{
    let vote = MessageType::Send(data);
    let sid = participants.index(me)?;
    // Send vote to all participants but for myself
    chan.send_many(wait, &(&sid, &vote))?;
    // the vote is returned to be taken into consideration as received
    Ok(vote)
}
```
