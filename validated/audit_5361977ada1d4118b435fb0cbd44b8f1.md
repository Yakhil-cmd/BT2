Audit Report

## Title
Single Malicious Callee Can Exhaust Shared Instruction Budget to Permanently Block Composite Queries - (File: rs/execution_environment/src/query_handler/query_call_graph.rs)

## Summary
The composite query execution framework initializes `round_limits.instructions` to `MAX_INSTRUCTIONS_PER_COMPOSITE_QUERY_CALL` (5 B) and caps each sub-call at `max_instructions_per_query` (also 5 B). Because both limits are identical, a single malicious callee canister can consume the entire shared budget in one execution. On the next DFS iteration, `instruction_limit_reached()` returns `true` and `evaluate_query_call_graph` aborts the whole composite query with `QueryCallGraphTotalInstructionLimitExceeded`, leaving all remaining callees unexecuted.

## Finding Description
`QueryContext::new` initializes the shared budget from `max_query_call_graph_instructions`:

```rust
// query_context.rs L152-153
instructions: as_round_instructions(max_query_call_graph_instructions),
``` [1](#0-0) 

The per-sub-call limit in `execute_query` is:

```rust
// query_context.rs L425-428
let instruction_limit = self.max_instructions_per_query.min(NumInstructions::new(
    self.round_limits.instructions.get().max(0) as u64,
));
``` [2](#0-1) 

Both constants resolve to 5,000,000,000:

- `MAX_INSTRUCTIONS_PER_QUERY_MESSAGE = NumInstructions::new(5 * B)` [3](#0-2) 
- `MAX_INSTRUCTIONS_PER_COMPOSITE_QUERY_CALL: u64 = 5_000_000_000` [4](#0-3) 

The overhead deduction (`INSTRUCTION_OVERHEAD_PER_QUERY_CALL = 50_000_000`) is applied *after* execution and does not constrain how many instructions a single sub-call may consume:

```rust
// query_context.rs L857
self.round_limits.instructions -= self.instruction_overhead_per_query_call;
``` [5](#0-4) 

After the malicious sub-call returns, `round_limits.instructions ≤ 0`. On the very next DFS loop iteration, the check at `query_call_graph.rs` L73-78 fires:

```rust
if query_context.instruction_limit_reached() {
    return QueryResponse::UserError(error); // entire query fails
}
``` [6](#0-5) 

All remaining callees in the graph are never executed.

## Impact Explanation
This is a concrete application/platform-level DoS matching the High impact class. Any canister that fans out composite queries to a registry or list that an unprivileged principal can extend (DEX aggregators, open portfolio trackers, any open-registry pattern) can have its composite query endpoint permanently disabled. The attacker deploys one canister that spins a tight Wasm loop to consume all available instructions, registers it in the victim's registry, and every subsequent composite query that includes the poison canister fails with `QueryCallGraphTotalInstructionLimitExceeded`. The attack is persistent and costs nothing per query after the one-time canister deployment.

## Likelihood Explanation
The attack requires only standard unprivileged canister deployment and registration in the target's open registry — no privileged access, no social engineering, no mainnet volumetric load. The malicious canister remains deployed indefinitely. The existing test `composite_query_callgraph_max_instructions_is_enforced` explicitly confirms that exceeding the graph budget causes the entire query to fail rather than just the offending sub-call. [7](#0-6) 

## Recommendation
Decouple the per-sub-call instruction limit from the total graph budget. Cap each sub-call at `max_query_call_graph_instructions / MAX_QUERY_CALL_DEPTH` (≈ 833 M with current defaults of 5 B / 6), so no single callee can exhaust the shared pool. Alternatively, introduce a distinct configurable `max_instructions_per_composite_query_sub_call` parameter separate from `max_instructions_per_query`, and set it to a fraction of the total graph budget. [8](#0-7) 

## Proof of Concept
1. Deploy canister **M** whose query method executes a tight Wasm loop consuming all available instructions before returning.
2. Register **M** in the target canister **A**'s open registry so it appears in **A**'s composite query call graph.
3. Send a composite query to **A** that includes **M** as one of the callees.
4. **M** consumes all 5 B instructions from `round_limits.instructions` during its sub-call.
5. On the next DFS iteration in `evaluate_query_call_graph`, `instruction_limit_reached()` returns `true`; the function returns `QueryResponse::UserError(QueryCallGraphTotalInstructionLimitExceeded)`.
6. All remaining callees (**B**, **C**, …) are never executed; the user receives an error.
7. Every subsequent composite query to **A** that includes **M** fails identically at zero marginal cost to the attacker.

A deterministic integration test can reproduce this by setting `max_query_call_graph_instructions` to a small value (as the existing test does), deploying a universal canister that burns all instructions, and asserting that a composite query including it returns `ErrorCode::QueryCallGraphTotalInstructionLimitExceeded`. [9](#0-8)

#[test]
fn composite_query_callgraph_max_instructions_is_enforced() {
    const NUM_CANISTERS: u64 = 20;
    const NUM_SUCCESSFUL_QUERIES: u64 = 5; // Number of calls expected to succeed

    let mut test = ExecutionTestBuilder::new()
        .with_max_query_call_graph_instructions(NumInstructions::from(
            NUM_SUCCESSFUL_QUERIES * INSTRUCTION_OVERHEAD_PER_QUERY_CALL,
        ))
        .build();

    let mut canisters = vec![];
    for _ in 0..NUM_CANISTERS {
        canisters.push(test.universal_canister_with_cycles(CYCLES_BALANCE).unwrap());
    }

    // Generate call tree of depth 1.
    // Canister 0 will call into each canister 1..num_canisters exactly once in a sequential manner.
    // This will therefore *not* hit the call graph depth limit, but should hit a limit
    // on the maximum number of instructions in a call graph.
    fn generate_call_to(
        canisters: &[ic_types::CanisterId],
        canister_idx: usize,
    ) -> ic_universal_canister::PayloadBuilder {
        assert_lt!(canister_idx, canisters.len());

        let reply = if canister_idx <= 1 {
            wasm().stable_size().reply_int()
        } else {
            generate_call_to(canisters, canister_idx - 1)
        };

        wasm().stable_grow(10).composite_query(
            canisters[canister_idx],
            call_args()
                .other_side(wasm().reply_data(b"ignore".as_ref()))
                .on_reply(reply),
        )
    }

    // Those should succeed
    for num_calls in 1..NUM_SUCCESSFUL_QUERIES {
        let test = test.non_replicated_query(
            canisters[0],
            "composite_query",
            generate_call_to(&canisters, num_calls as usize).build(),
        );
        match &test {
            Ok(_) => {}
            Err(err) => panic!(
                "Query with {num_calls} calls failed, when it should have succeeded: {err:?}"
            ),
        }
    }
    for num_calls in NUM_SUCCESSFUL_QUERIES..NUM_CANISTERS {
        let test = test.non_replicated_query(
            canisters[0],
            "composite_query",
            generate_call_to(&canisters, num_calls as usize).build(),
        );
        match &test {
            Ok(_) => panic!("Query with {num_calls} calls should have failed!"),
            Err(err) => assert_eq!(
                err.code(),
                ErrorCode::QueryCallGraphTotalInstructionLimitExceeded
            ),
        }
    }
