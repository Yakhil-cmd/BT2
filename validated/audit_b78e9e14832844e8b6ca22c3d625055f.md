### Title
Silent Offer Cancellation Failure Causes Incorrect Offer State and Spendable Balance Corruption - (File: `packages/gui/src/components/offers2/CancelOfferList.tsx`)

### Summary
In `CancelOfferList.tsx`, the `cancelOffer` RTK Query mutation is invoked without `.unwrap()` and without any `try/catch`. In RTK Query, omitting `.unwrap()` means errors are silently absorbed — the awaited promise resolves to `{ data, error }` rather than throwing. The `onOfferCanceled` callback is then called unconditionally on the very next line, regardless of whether the cancellation actually succeeded. This causes the UI to treat the offer as cancelled (removing it from the conflict list and recalculating spendable balances) even when the underlying RPC call failed.

### Finding Description

In `handleCancelOffer` inside `CancelOfferList.tsx`:

```typescript
await cancelOffer({ tradeId, secure, fee });   // no .unwrap(), errors silently swallowed
onOfferCanceled(tradeId, secure, fee);          // called unconditionally
``` [1](#0-0) 

The `cancelOffer` mutation is defined via RTK Query's `mutation` helper: [2](#0-1) 

RTK Query mutations called without `.unwrap()` never throw — they return a settled object. Any RPC-level error (wallet not synced, daemon timeout, insufficient fee, network drop) is silently discarded.

The `onOfferCanceled` callback is wired to `onCancelOffer1` / `onCancelOffer2` in `OfferEditorCancelConflictingOffersDialog`, which update local React state to remove the offer from the conflict list and recalculate the displayed spendable balance: [3](#0-2) 

A second instance exists in `OfferManager.tsx` inside `relistOffer`, where `cancelOffer` is also called without `.unwrap()` or error handling, and the code unconditionally navigates to the offer builder: [4](#0-3) 

### Impact Explanation

**High** — State corruption of offer and spendable-balance display that causes a user to act on wrong asset/status information.

The `CancelOfferList` component is rendered inside `OfferEditorCancelConflictingOffersDialog`, which is shown specifically when a user is creating a new offer but has conflicting open offers that lock the same coins. The dialog tells the user they *must* cancel those conflicting offers to free up spendable balance before the new offer can be created.

When `cancelOffer` fails silently:
1. `onOfferCanceled` fires unconditionally, removing the offer from the conflict list and inflating the displayed spendable balance.
2. The user sees the conflict as resolved and proceeds to create the new offer.
3. The original offer is still active on-chain — the coins are still locked.
4. The new offer creation may fail unexpectedly, or both offers may remain live simultaneously, exposing the user's assets to acceptance by two different counterparties.

In the `relistOffer` path, a failed cancellation causes the user to be navigated to the offer builder to create a counter-offer while the original offer is still active, potentially resulting in two live offers over the same locked assets.

### Likelihood Explanation

The failure condition is reachable without any attacker involvement — it triggers on ordinary operational conditions: wallet not yet synced, RPC timeout, daemon restart, or a transient network error between the GUI and the local daemon. No privileged access or key material is required. The `CancelOfferList` component is used in the normal offer-creation flow whenever conflicting offers exist.

### Recommendation

Call `.unwrap()` on the mutation and wrap in `try/catch`. Only invoke `onOfferCanceled` (and any subsequent navigation) inside the `try` block, after confirmed success. Surface the error to the user via `showError`:

```typescript
try {
  await cancelOffer({ tradeId, secure, fee }).unwrap();
  onOfferCanceled(tradeId, secure, fee);
} catch (err) {
  showError(err);
}
```

Apply the same pattern to `relistOffer` in `OfferManager.tsx` — guard the `navigate` call behind a successful `.unwrap()`.

### Proof of Concept

1. Open the Chia GUI and create two offers that lock the same XCH coins.
2. Begin creating a third offer using the same coins. The `OfferEditorCancelConflictingOffersDialog` appears, listing the conflicting offers.
3. Simulate or trigger an RPC failure (e.g., pause the daemon, or cause a timeout) at the moment the user clicks "Cancel" on one of the conflicting offers.
4. `cancelOffer` resolves without throwing (no `.unwrap()`), the error is silently dropped, and `onOfferCanceled` fires.
5. The UI removes the offer from the conflict list and shows the spendable balance as freed up.
6. The user proceeds to create the new offer, believing the conflict is resolved.
7. The original conflicting offer remains active on-chain; both offers are now live over the same locked coins. [5](#0-4) [6](#0-5)
