### Title
Transaction-Level Randomness Re-Rolling via `revertibleRandom()` Abort - (File: `fvm/environment/random_generator.go`)

---

### Summary

Any unprivileged Cadence transaction sender (or EVM contract caller via the Cadence Arch precompile) can observe the output of `revertibleRandom<T>()` within a transaction and abort/panic if the output is unfavorable. Because the CSPRG is re-seeded per transaction using the transaction ID as a diversifier, resubmitting with a new transaction (new tx_id) yields a different random output. An attacker can repeat this loop — observe, abort if unfavorable, resubmit — until a favorable random is obtained, paying only gas fees per attempt.

---

### Finding Description

Flow's `revertibleRandom<T>()` Cadence built-in (and its EVM equivalent via the Cadence Arch precompile `revertibleRandom()`) derives its output from a CSPRG seeded by two inputs:

1. The block's random beacon source — the BLS threshold signature in the QC certifying the current block, fixed for all transactions in the block.
2. The transaction ID (`tx_id`) — used as a CSPRG diversifier, unique per transaction.

In `fvm/environment/random_generator.go`, `createPRG()` constructs the generator:

```go
csprg, err := prg.New(source, prg.ExecutionEnvironment, gen.salt)
```

where `gen.salt` is `params.TxId[:]` set in `fvm/environment/facade_env.go`:

```go
env.RandomGenerator = NewRandomGenerator(
    tracer,
    params.EntropyProvider,
    params.TxId[:],
)
```

The random is provided to the transaction **before it commits**. A transaction can call `revertibleRandom<UInt64>()`, inspect the value, and conditionally `panic()` or abort. Because a new transaction submission produces a new `tx_id`, the CSPRG diversifier changes, yielding a different random output from the same block beacon source.

The same attack path exists in the EVM layer. `revertibleRandomGenerator()` in `fvm/evm/handler/precompiles.go` calls `backend.ReadRandom(rand)`, which ultimately calls the same `ReadRandom` path. An EVM contract can call `revertibleRandom()` via the Cadence Arch precompile and revert the EVM transaction if the result is unfavorable, causing the outer Cadence transaction to also revert.

---

### Impact Explanation

Any on-chain protocol — Cadence or EVM on Flow — that uses `revertibleRandom()` to determine outcomes affecting user assets (lotteries, NFT trait rolls, game outcomes, prize distributions) is vulnerable to outcome gaming. An attacker can:

1. Submit a transaction calling `revertibleRandom()`.
2. Check the result against a desired threshold.
3. Abort (panic/revert) if unfavorable.
4. Resubmit with a new transaction (new tx_id → new random from same block beacon).
5. Repeat until a favorable outcome is obtained.

The cost is only the gas fee for each failed transaction. Unlike the external report's miner-based re-org attack, this requires no mining power or privileged access — any unprivileged account can execute it.

---

### Likelihood Explanation

**High.** The attack requires only the ability to submit transactions. No staked nodes, no mining power, no admin keys, and no collusion are needed. The only cost is gas fees for aborted transactions. For high-value outcomes (e.g., winning a large lottery), the expected value of repeated attempts easily exceeds the gas cost. The attack is directly reachable from any externally submitted Cadence transaction or EVM transaction via the Cadence Arch precompile.

---

### Recommendation

1. **Document and enforce the commit-reveal pattern**: Protocols requiring non-gameable randomness must use `RandomBeaconHistory.sourceOfRandomness(blockHeight)` (accessible via the EVM Cadence Arch `getRandomSource(uint64)` precompile) with a commit-reveal scheme: commit in block N, reveal using the random source of block N+k (k ≥ 1). This ensures the random is not known at commit time.

2. **Add protocol-level warnings**: The `revertibleRandom` name signals the risk, but the FVM should emit a structured warning or documentation note at the call site that any protocol using this function for asset-affecting outcomes is vulnerable to abort-based re-rolling.

3. **Consider a non-revertible random primitive**: Provide a Cadence built-in that commits the random output to state before the transaction can inspect it, preventing conditional abort.

**Attack scenario (Cadence):**
```cadence
transaction {
    execute {
        let rand = revertibleRandom<UInt64>()
        // Attacker checks outcome; aborts if unfavorable
        if rand % 100 != 0 {
            panic("re-roll")  // abort; resubmit with new tx_id for new random
        }
        // Only reaches here when rand % 100 == 0 (1% chance per attempt)
        // ... claim prize ...
    }
}
```

**Attack scenario (EVM via Cadence Arch):**
```solidity
function tryWin() external {
    (bool ok, bytes memory data) = cadenceArch.staticcall(
        abi.encodeWithSignature("revertibleRandom()")
    );
    uint64 rand = abi.decode(data, (uint64));
    require(rand % 100 == 0, "re-roll"); // revert if unfavorable
    // claim prize
}
```

The EVM transaction reverts, causing the outer Cadence transaction to revert. The attacker resubmits until `rand % 100 == 0`. [5](#0-4)

func revertibleRandomGenerator(backend backends.Backend) func() (uint64, error) {
	return func() (uint64, error) {
		rand := make([]byte, 8)
		err := backend.ReadRandom(rand)
		if err != nil {
			return 0, err
		}

		return binary.BigEndian.Uint64(rand), nil
	}
