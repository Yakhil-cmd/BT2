### Title
Signature Validation Bypass via `skip_stateful_validations` Allows Invalid-Signature Invoke Transactions into Mempool — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator skips the `__validate__` entry point (account signature verification) for invoke transactions with nonce=1 when `account_tx_in_pool_or_recent_block` returns true for the sender address. The check only verifies that *some* transaction from the address exists in the mempool — it does not verify that the submitter of the invoke is the legitimate account owner. An attacker who has a deploy_account transaction in the mempool (their own or anyone else's) can submit an invoke transaction with an arbitrary/invalid signature from that address, and it will be admitted to the mempool without any signature check.

### Finding Description

The gateway stateful validation path in `extract_state_nonce_and_run_validations` calls `run_pre_validation_checks`, which calls `skip_stateful_validations`:

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                ...
        }
    }
    Ok(false)
}

When this returns `true`, `run_validate_entry_point` is called with `validate: !skip_validate` = `false`:

Inside `StatefulValidator::perform_validations`, when `execution_flags.validate` is false, the function returns before calling `__validate__`:

The `account_tx_in_pool_or_recent_block` check is purely presence-based:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [4](#0-3) 

It returns `true` for any address that has *any* transaction in the pool — not specifically a deploy_account, and not verified to be from the same submitter. The code comment acknowledges this: "It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations." This assumption is incorrect: any party can submit an invoke from any address. [5](#0-4) 

### Impact Explanation

**Impact: High** — Mempool/gateway admission accepts invalid transactions (invalid signature) before sequencing.

An invoke transaction with a completely invalid or forged signature is admitted to the mempool without the account's `__validate__` entry point being called. The transaction will fail at execution time in the batcher, but it occupies a mempool slot and can be used to:

1. **Mempool pollution**: Flood the mempool with invalid-signature invokes at the cost of one deploy_account per slot.
2. **Front-running/blocking**: An attacker who observes a pending deploy_account for address X (submitted by a legitimate user) can immediately submit an invalid invoke with nonce=1 from X. If the attacker's invalid invoke is admitted first, the legitimate user's valid invoke with nonce=1 will be rejected as a duplicate nonce, blocking the user's first post-deploy transaction.

### Likelihood Explanation

The precondition is low-cost and unprivileged: the attacker only needs to pre-fund an account with STRK and submit a valid deploy_account transaction. The deploy_account itself passes all gateway validations normally. The subsequent invalid invoke bypasses signature verification entirely. No privileged access, leaked keys, or internal network access is required.

### Recommendation

In `skip_stateful_validations`, replace the presence-only check with a type-specific check: verify that the transaction in the mempool for the sender address is specifically a `DeployAccount` transaction, not just any transaction. Alternatively, verify that the deploy_account transaction in the pool was submitted by the same sender (i.e., the deploy_account's contract address matches the invoke's sender address and the deploy_account is the specific transaction type).

```rust
// Instead of:
mempool_client.account_tx_in_pool_or_recent_block(tx.sender_address())

// Use a new query that specifically checks for a pending deploy_account:
mempool_client.has_pending_deploy_account(tx.sender_address())
```

### Proof of Concept

1. Pre-fund address `X` with STRK on the network.
2. Submit a valid `deploy_account` transaction for address `X` to the gateway. It passes all validations and enters the mempool.
3. Submit an `invoke` transaction from address `X` with `nonce=1` and a completely invalid signature (e.g., all zeros).
4. The gateway calls `skip_stateful_validations`: `tx.nonce() == 1`, `account_nonce == 0`, `account_tx_in_pool_or_recent_block(X) == true` (deploy_account is in pool).
5. `run_validate_entry_point` is called with `validate: false`; `__validate__` is never invoked.
6. The invalid-signature invoke is admitted to the mempool.
7. Optionally, a legitimate user who also tries to submit a valid invoke with nonce=1 from `X` receives a duplicate-nonce rejection, blocking their transaction.

```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
