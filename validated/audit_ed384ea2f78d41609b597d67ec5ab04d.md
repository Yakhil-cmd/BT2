### Title
`priceProviderTimelock = 0` Collapses the Oracle Rotation Timelock to Zero, Allowing Pool Admin to Atomically Swap to a Malicious Price Provider and Drain LP Funds — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

The `priceProviderTimelock` parameter accepted by `createPool` has no lower-bound validation. When set to `0`, the two-step oracle rotation mechanism (`proposePoolPriceProvider` → `executePoolPriceProviderUpdate`) degenerates: `executeAfter` equals `block.timestamp` at proposal time, so the strict-less-than guard in `executePoolPriceProviderUpdate` passes immediately. The pool admin can propose and execute an oracle swap in a single transaction, bypassing the timelock that was explicitly introduced (audit finding M01) to give LPs time to exit before an oracle change takes effect.

---

### Finding Description

**Root cause — no minimum bound on `priceProviderTimelock`**

`_validatePoolParameters` validates every other sensitive parameter but is silent on `priceProviderTimelock`: [1](#0-0) 

The value is stored verbatim: [2](#0-1) 

The only special-cased value is `type(uint256).max` (immutable mode): [3](#0-2) 

**Degenerate `executeAfter` when timelock is 0**

In `proposePoolPriceProvider`, `executeAfter` is computed as: [4](#0-3) 

When `timelock == 0`, `executeAfter = block.timestamp`.

**Guard passes immediately**

In `executePoolPriceProviderUpdate`, the guard is a strict less-than: [5](#0-4) 

`block.timestamp < block.timestamp` is always `false`, so the revert never fires. The pool admin can call both functions back-to-back in the same block — or atomically via a contract — and the new (potentially malicious) price provider is live immediately.

**Validation of the new provider is insufficient**

`_validatePriceProvider` only checks that `token0()` and `token1()` return the correct addresses: [6](#0-5) 

A malicious provider trivially satisfies this while returning any bid/ask it chooses.

**Historical context confirms intent**

The project's own audit-resolution notes (M01) state the timelock was added precisely because "Admin can arbitrarily change the pool priceProvider" was a vulnerability. The fix is incomplete because `0` is a valid timelock value: [7](#0-6) 

---

### Impact Explanation

A pool admin who deploys with `priceProviderTimelock = 0` can, after LPs have deposited:

1. Atomically call `proposePoolPriceProvider(pool, maliciousOracle)` and `executePoolPriceProviderUpdate(pool)` in one transaction.
2. The malicious oracle returns an arbitrarily skewed bid/ask (e.g., bid ≈ 0, ask ≈ ∞ for one direction).
3. The admin (or an accomplice) calls `swap` against the pool at the manipulated price, extracting all LP-owned token balances.

LPs have zero blocks to react. This is a direct loss of user principal — the exact impact class the allowed-impact gate targets.

---

### Likelihood Explanation

- The pool admin is semi-trusted and controls `priceProviderTimelock` at creation; no factory-owner approval is required to set it to `0`.
- Any pool creator can deploy with `priceProviderTimelock = 0` without raising a factory-level revert.
- LPs inspecting `priceProviderTimelock[pool]` off-chain may not recognise `0` as "no delay" vs. a misconfiguration.
- The attack is atomic and MEV-friendly (single transaction, no front-running window for LPs).

Likelihood: **Medium** (requires pool admin to set timelock to 0 and LPs to deposit, but no other preconditions).

---

### Recommendation

Enforce a protocol-level minimum timelock in `_validatePoolParameters`. For example:

```solidity
uint256 internal constant MIN_PRICE_PROVIDER_TIMELOCK = 1 hours;

// inside _validatePoolParameters:
if (
    params.priceProviderTimelock != type(uint256).max &&
    params.priceProviderTimelock < MIN_PRICE_PROVIDER_TIMELOCK
) {
    revert PriceProviderTimelockTooShort();
}
```

This mirrors the pattern already used for other parameters (e.g., `PriceProviderL2` enforces `_maxTimeDelta != 0`): [8](#0-7)

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.35;

contract MaliciousOracle {
    address public token0;
    address public token1;
    constructor(address t0, address t1) { token0 = t0; token1 = t1; }
    // Returns a price that lets the attacker buy all token0 for near-zero token1
    function getBidAndAskPrice() external pure returns (uint128 bid, uint128 ask) {
        bid = 1;          // near-zero bid  → sell token0 cheaply
        ask = 1e18;       // huge ask       → buy token0 at 1e18× fair price
    }
}

contract AttackRouter {
    IMetricOmmPoolFactory factory;
    address pool;

    function attack(address maliciousOracle) external {
        // Step 1: propose (executeAfter = block.timestamp because timelock == 0)
        factory.proposePoolPriceProvider(pool, maliciousOracle);
        // Step 2: execute immediately — block.timestamp < block.timestamp is false
        factory.executePoolPriceProviderUpdate(pool);
        // Step 3: swap at manipulated price, drain LP token0
        IMetricOmmPool(pool).swap(
            address(this), false, type(int128).max, 0, "", ""
        );
    }
}
```

**Sequence:**
1. Pool admin deploys pool with `priceProviderTimelock = 0`.
2. LPs add liquidity.
3. Pool admin calls `AttackRouter.attack(maliciousOracle)` — single transaction.
4. `executeAfter = block.timestamp`; guard `block.timestamp < block.timestamp` is `false`; oracle rotates instantly.
5. Swap executes at the manipulated price; LP principal is drained.

## M01 - Admin can arbitrarily change the pool priceProvider

### Finding summary

`setPriceProvider()` allowed admin to replace oracle at runtime, and `priceProvider` was returned by `getImmutables()`.

### Resolution

Fixed.

Changes:

- Added constructor config `priceProviderTimelock`.
- Added admin flow:
  - `proposePriceProvider(address newPriceProvider)`
  - `executePriceProviderUpdate()`
- Added events:
  - `PriceProviderChangeProposed(currentPriceProvider, newPriceProvider, executeAfter)`
  - `PriceProviderUpdated(newPriceProvider)`

Behavior now:

- `priceProviderTimelock == type(uint256).max` => immutable mode (admin updates disabled).
- `priceProviderTimelock < type(uint256).max` => mutable mode with delayed execution.
- Price provider used by swaps/reports resolves through timelock mode; active provider is read via `MetricOmmPoolStateView.priceProvider(pool)`.
