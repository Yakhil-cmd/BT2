### Title
`SendToAccountHandler.Handle` Missing `IsSourceCoin` Branch — Permanent DoS for All `__CronosSendToAccount` Events on Source Coin Contracts - (`File: x/cronos/keeper/evmhandlers/send_to_account.go`)

---

### Summary

`SendToAccountHandler.Handle` unconditionally calls `bankKeeper.SendCoins` from the contract's bank address for every mapped denom. For source coins (`cronos0x...`), the correct flow is `MintCoins` + `SendCoinsFromModuleToAccount` (as implemented in every other analogous handler). Because source coins are minted on demand and never held in the contract's bank account, `SendCoins` always fails with insufficient funds, permanently blocking the `__CronosSendToAccount` flow for any source coin contract.

---

### Finding Description

`SendToAccountHandler.Handle` retrieves the denom for the emitting contract and immediately calls `bankKeeper.SendCoins(contractAddr, recipient, coins)`: [1](#0-0) 

There is no `types.IsSourceCoin(denom)` check. Compare this to `SendToIbcHandler.handle`, which correctly branches on source coins: [2](#0-1) 

And `ConvertCoinFromCRC21ToNative`, which does the same: [3](#0-2) 

`IsSourceCoin` returns `true` for any `cronos0x...` denom: [4](#0-3) 

For source coins, the invariant is: the contract's bank address holds **zero** native tokens. Tokens are minted into existence when moving from EVM to Cosmos, and burned when moving back. `SendCoins` from a zero-balance address always returns an insufficient-funds error, which `Handle` propagates as a non-nil error, reverting the EVM transaction.

---

### Impact Explanation

Every `__CronosSendToAccount` log emitted by a source coin contract (`cronos0x...`) will permanently fail. This is not a transient condition — the contract's bank balance for its own source denom is structurally zero at all times. No amount of retrying or user action can fix it. Honest users attempting to convert source coins to native tokens via this bridge event are permanently blocked.

Impact category: **High** — permanent inability for honest users to process valid bridge/conversion flows under normal network assumptions.

---

### Likelihood Explanation

- Any contract registered with a `cronos0x...` denom (source coin) that emits `__CronosSendToAccount` triggers this path.
- The handler is registered for all mapped contracts with no denom-type filter.
- The `ModuleCRC21` contract and `ModuleCRC20Proxy` (with `isSource=true`) are the canonical source coin contracts; any user calling a function on such a contract that emits this event hits the bug.
- No privileged access is required — any user can call the contract.

---

### Recommendation

Add the same `IsSourceCoin` branch to `SendToAccountHandler.Handle` that exists in `SendToIbcHandler.handle`:

```go
if types.IsSourceCoin(denom) {
    if err = h.bankKeeper.MintCoins(ctx, types.ModuleName, coins); err != nil {
        return err
    }
    if err = h.bankKeeper.SendCoinsFromModuleToAccount(ctx, types.ModuleName, recipient, coins); err != nil {
        return err
    }
} else {
    if err = h.bankKeeper.SendCoins(ctx, contractAddr, recipient, coins); err != nil {
        return err
    }
}
```

### Proof of Concept

1. Register a source coin contract: `denom = "cronos0x<addr>"`, mapped to contract at `<addr>`.
2. Deploy a contract at `<addr>` that emits `__CronosSendToAccount(recipient, amount)` when called.
3. Call the contract from any user account.
4. The EVM hook fires `SendToAccountHandler.Handle`.
5. `GetDenomByContract` returns `"cronos0x<addr>"`.
6. `bankKeeper.SendCoins(contractAddr, recipient, coins)` is called; the contract's bank balance for `"cronos0x<addr>"` is 0.
7. The call fails with `"spendable balance 0cronos0x<addr> is smaller than <amount>cronos0x<addr>: insufficient funds"`.
8. The transaction reverts. The flow is permanently broken for all source coin `__CronosSendToAccount` events.
