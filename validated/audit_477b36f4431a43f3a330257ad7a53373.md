### Title
Missing Vector Length Validation in `PolynomialCommitmentsMessageMany` Causes Panic and Permanent Triple Generation Abort — (`src/ecdsa/ot_based_ecdsa/triples/generation.rs`)

### Summary

A malicious participant can send a `PolynomialCommitmentsMessageMany` message in round 2 of triple generation with all vector fields set to length 0. Because `do_generation_many` indexes directly into these vectors without any prior length check, every honest party panics with an index-out-of-bounds error, permanently aborting the triple generation session. The attacker can repeat this for every new session.

### Finding Description

`PolynomialCommitmentsMessageMany` is defined with six `Vec<_>` fields: [1](#0-0) 

After receiving this message via `recv_from_others`, the code immediately indexes into all six vectors for `i in 0..N` with no prior length check: [2](#0-1) 

`recv_from_others` performs only serde deserialization — an empty `Vec` is a valid msgpack encoding and passes through without error: [3](#0-2) 

Since `N >= 1` is enforced by `assert!(N > 0)`: [4](#0-3) 

accessing `their.big_e_v[0]` on an empty `Vec` is an unconditional Rust panic (not a `Result::Err`). The commitment check at lines 353–364 is never reached — the panic fires at line 337 first. [5](#0-4) 

The grep search for `.len()` on any of these fields returns zero matches, confirming no length guard exists anywhere in the codebase.

### Impact Explanation

Every honest party independently receives the malicious message and independently panics. Triple generation is aborted for the entire participant set. Since presigning and signing depend on triples, this blocks all downstream ECDSA operations. The attacker can repeat the attack for every new session, making the denial effectively permanent.

**Impact: High — Permanent denial of triple generation (and therefore signing) for all honest parties.**

### Likelihood Explanation

Any participant in the triple generation protocol can send this message. No special privilege is required beyond being a listed participant. The attack requires only two rounds of participation: send any valid commitment in round 1, then send a `PolynomialCommitmentsMessageMany` with empty vectors in round 2. The malicious message is a valid msgpack encoding and passes deserialization.

### Recommendation

Add an explicit length check immediately after receiving `PolynomialCommitmentsMessageMany`, before any indexing:

```rust
if their.big_e_v.len() != N
    || their.big_f_v.len() != N
    || their.big_l_v.len() != N
    || their.randomizer_v.len() != N
    || their.phi_proof0_v.len() != N
    || their.phi_proof1_v.len() != N
{
    return Err(ProtocolError::AssertionFailed(format!(
        "message from {from:?} has wrong number of polynomial commitments"
    )));
}
```

This converts the panic into a graceful `ProtocolError`, allowing honest parties to identify and exclude the malicious participant rather than aborting.
