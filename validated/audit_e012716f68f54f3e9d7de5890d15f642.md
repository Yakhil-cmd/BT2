### Title
Single Compromised Oracle URL Manipulates ETH→STRK Rate, Corrupting L1 Gas Prices in FRI Committed to Every Block - (File: crates/apollo_l1_gas_price/src/exchange_rate_oracle.rs)

### Summary

`ExchangeRateOracleClient.spawn_query` iterates a list of oracle URLs and returns the **first successful response** without cross-validating the returned rate against any other source. The only validation applied is `rate != 0` and `decimals == 18`. A single compromised or malfunctioning oracle URL can therefore return an arbitrarily large or small ETH→STRK rate, which is used without any bounds check to convert L1 gas prices from WEI to FRI. Because the validator's `is_proposal_init_valid` re-derives its own reference FRI prices from the same oracle, a compromised URL that all validators share causes both proposer and validator to compute the same wrong rate, making the cross-check ineffective. The corrupted FRI prices are committed to every block, causing incorrect fee charging for all transactions in those blocks.

### Finding Description

**Root cause — `spawn_query` accepts the first successful URL response:** [1](#0-0) 

The loop tries each URL in `url_header_list` in round-robin order and immediately returns `Ok(rate)` on the first success. There is no comparison against other URLs, no median, no outlier detection.

**Validation in `resolve_query` is insufficient:** [2](#0-1) 

The only semantic checks are `rate != 0` and `decimals == 18`. Any non-zero `u128` value with the correct decimals field is accepted unconditionally, including values that are orders of magnitude away from the true market rate.

**The rate is applied without bounds to convert WEI prices to FRI:** [3](#0-2) 

`apply_fee_transformations` clamps the WEI prices to `[min_l1_gas_price_wei, max_l1_gas_price_wei]`, but the ETH→STRK rate itself is never clamped. The FRI prices are computed as `wei_price × eth_to_fri_rate / WEI_PER_ETH`, so a 100× inflated rate produces 100× inflated FRI prices even when WEI prices are within their configured bounds.

**The resulting FRI prices are committed to the block's `ProposalInit`:** [4](#0-3) 

`convert_to_sn_api_block_info` converts `ProposalInit.l1_gas_price_fri` and `l1_data_gas_price_fri` directly into the `BlockInfo.gas_prices` used by the blockifier for every transaction in the block.

**The validator's `is_proposal_init_valid` cross-check is defeated when all validators share the same oracle URL:** [5](#0-4) 

The validator calls `get_l1_prices_in_fri_and_wei` with its own oracle client. If that client is configured with the same compromised URL, it derives the same inflated rate and the same inflated FRI prices. The `within_margin` check then passes because both sides agree on the wrong value. The WEI `within_margin` check also passes because WEI prices are clamped identically on both sides.

**The `ExchangeRateOracleConfig` default ships with a single URL:** [6](#0-5) 

The default configuration contains exactly one URL. Operators who follow the default or configure a single oracle endpoint have no redundancy at all.

### Impact Explanation

**Incorrect fee/gas accounting (Critical):** Every transaction in every block produced while the oracle is compromised is charged the wrong amount of STRK for L1 gas. With a 100× inflated rate, users pay 100× more STRK for L1 gas than the actual Ethereum cost. With a deflated rate, the sequencer absorbs the L1 cost without recovering it from users. Both directions represent a direct, quantifiable economic loss.

**Blockifier pre-validation rejects valid transactions (High):** `check_fee_bounds` in `AccountTransaction::perform_pre_validation_stage` rejects any transaction whose `max_price_per_unit` for L1 gas or L1 data gas is below the block's actual gas price. [7](#0-6) 

If the block's `l1_gas_price_fri` is inflated 100×, every transaction whose `max_price_per_unit` was set to the true market rate will be rejected as `MaxGasPriceTooLow`, effectively making the network unusable for normal users during the attack window.

### Likelihood Explanation

- The `spawn_query` round-robin selects the first responding URL. If the compromised URL is first in the list (or the only URL), it is always used.
- In practice, operators configure a small number of oracle endpoints, often a single provider. The default config ships with one URL.
- The oracle URL is a private configuration field (`ParamPrivacyInput::Private`), so operators may not audit it frequently.
- The attack requires only that the attacker control or compromise one oracle HTTP endpoint — no on-chain access, no validator key, no privileged position in the sequencer.
- The `is_proposal_init_valid` guard is structurally defeated when all validators share the same compromised URL, which is the realistic scenario for a widely-used oracle service.

### Recommendation

1. **Cross-validate across all configured URLs.** When multiple URLs are configured, query all of them and compute the median (or reject if the spread exceeds a configurable threshold), rather than returning the first success. This is the direct analog of the fix recommended in the original report.

2. **Add a hard rate-bounds check in `resolve_query`.** Reject any rate that falls outside a configurable `[min_rate, max_rate]` window. Even a loose bound (e.g., 10× the previous cached rate) would prevent catastrophic manipulation.

3. **Clamp the ETH→STRK rate independently of WEI prices.** Add a `max_eth_to_fri_rate` and `min_eth_to_fri_rate` to `GasPriceParams` and apply them before the WEI→FRI conversion, so that even a fully compromised oracle cannot produce FRI prices outside a known-safe range.

4. **Alert on rate deviation.** Log a high-severity warning (and optionally halt block production) when the newly fetched rate deviates from the cached previous rate by more than a configurable percentage.

### Proof of Concept

1. Operator configures `eth_to_strk_oracle_config.url_header_list` with a single URL pointing to a service the attacker controls (or the attacker compromises an existing oracle endpoint used by most validators).

2. The attacker's oracle endpoint returns a well-formed JSON response:
   ```json
   { "price": "0x<100x_true_rate_as_hex>", "decimals": 18 }
   ```
   This passes all checks in `resolve_query`: valid JSON, `price` is a non-zero hex string, `decimals == 18`.

3. `ExchangeRateOracleClient.fetch_rate` returns the inflated rate. `get_l1_prices_in_fri_and_wei_and_conversion_rate` uses it to compute `l1_gas_price_fri = l1_gas_price_wei × (100 × true_rate) / WEI_PER_ETH`, which is 100× the correct FRI price.

4. The proposer builds a `ProposalInit` with the inflated `l1_gas_price_fri`. The validator calls `is_proposal_init_valid`, which calls `get_l1_prices_in_fri_and_wei` with its own oracle — the same compromised URL — and derives the same inflated rate. The `within_margin` check passes. The proposal is accepted.

5. `convert_to_sn_api_block_info` places the inflated `l1_gas_price_fri` into `BlockInfo.gas_prices.strk_gas_prices.l1_gas_price`.

6. For every transaction in the block, `check_fee_bounds` compares `resource_bounds.l1_gas.max_price_per_unit` against the inflated block price. Transactions whose `max_price_per_unit` was set to the true market rate are rejected with `MaxGasPriceTooLow`. Transactions that do pass are charged 100× the correct STRK amount for L1 gas.
