### Title
Receipt Size Invariant Bypassed via `promise_return` `output_data_receivers` Mutation After Validation — (`runtime/runtime/src/lib.rs`, `runtime/runtime/src/verifier.rs`)

### Summary

`validate_receipt` enforces `receipt_size ≤ max_receipt_size` only at the moment a new receipt is constructed. When a contract uses `promise_return`, the runtime subsequently appends the parent receipt's `output_data_receivers` directly to the returned receipt's action-receipt struct — after validation has already passed. No re-validation is performed. Any unprivileged user who deploys a contract can craft a receipt that passes the size check at creation but exceeds `max_receipt_size` once the `output_data_receivers` are appended, permanently violating the protocol-level receipt-size invariant.

### Finding Description

**Check 1 — size gate at creation (`verifier.rs`):**

```rust
// runtime/runtime/src/verifier.rs  lines 533–541
if mode == ValidateReceiptMode::NewReceipt {
    let receipt_size: u64 =
        borsh::object_length(receipt).unwrap().try_into()...;
    if receipt_size > limit_config.max_receipt_size {
        return Err(ReceiptValidationError::ReceiptSizeExceeded { ... });
    }
}
```

This check runs exactly once, when the receipt is first emitted by the VM logic host functions. At that point the receipt's `output_data_receivers` field is empty (`vec![]`), so the serialised size is at most `max_receipt_size` (4 194 304 bytes).

**Mutation after validation (`lib.rs`):**

```rust
// runtime/runtime/src/lib.rs  lines 1019–1037
if !action_receipt.output_data_receivers().is_empty() {
    if let Ok(ReturnData::ReceiptIndex(receipt_index)) = result.result {
        match result.new_receipts.get_mut(receipt_index as usize)...receipt_mut() {
            ReceiptEnum::Action(new_action_receipt) | ... =>
                new_action_receipt.output_data_receivers
                    .extend_from_slice(&action_receipt.output_data_receivers()),
            ...
        }
    }
}
```

When the executing receipt's result is `ReturnData::ReceiptIndex` (i.e. `promise_return`), the runtime takes the parent receipt's `output_data_receivers` and appends them directly to the already-validated child receipt. Each `DataReceiver` is `{ data_id: CryptoHash (32 B), receiver_id: AccountId (variable) }`. Adding even one receiver can push the receipt over `max_receipt_size`.

**No re-validation:** After the mutation the receipt is forwarded to the `ReceiptSink` and eventually stored in the outgoing-receipts trie. The only subsequent validation uses `ValidateReceiptMode::ExistingReceipt`, which explicitly skips the size check:

```rust
// runtime/runtime/src/verifier.rs  lines 573–585
pub enum ValidateReceiptMode {
    NewReceipt,
    /// ...
    /// 2) There is a bug which allows to create receipts that are above the
    ///    size limit. Runtime has to handle them gracefully until the receipt
    ///    size limit bug is fixed.
    ///    See https://github.com/near/nearcore/issues/12606 for details.
    ExistingReceipt,
}
```

The contradiction is structural: `validate_receipt` enforces `size ≤ max_receipt_size` on the receipt without `output_data_receivers`, while the `promise_return` path in `lib.rs` grows the same receipt beyond that limit without re-checking.

### Impact Explanation

Any unprivileged user can produce receipts that exceed `max_receipt_size` on mainnet. Concretely:

- The protocol-level receipt-size invariant is broken; receipts larger than 4 194 304 bytes are committed to the outgoing-receipts trie and forwarded cross-shard.
- The `per_receipt_storage_proof_size_limit` (4 000 000 bytes, smaller than `max_receipt_size`) can be exceeded, corrupting the storage-proof size accounting used by stateless validation.
- Chunk validators that apply `ExistingReceipt` mode silently accept the oversized receipt, so the violation propagates without any on-chain error.
- The nearcore team's own integration test (`test_max_receipt_size_promise_return`) confirms the oversized receipt reaches the chain and is processed.

### Likelihood Explanation

Likelihood is **High**. The trigger requires only:
1. Deploying a contract (permissionless).
2. Calling a method that creates a near-maximum-size receipt and then calls `promise_return` on it from within a `promise_then` callback.

No privileged access, no special network position, and no race condition is required. The contract code needed is minimal (the test contract `max_receipt_size_promise_return_method1` already demonstrates it).

### Recommendation

Re-validate the receipt size **after** appending `output_data_receivers` in `lib.rs`. Specifically, after the `extend_from_slice` call at lines 1030–1035, compute the new serialised size and return a `NewReceiptValidationError::ReceiptSizeExceeded` if it exceeds `max_receipt_size`. Alternatively, enforce the size budget on `output_data_receivers` before appending by computing the delta and rejecting the `promise_return` if it would push the receipt over the limit.

### Proof of Concept

```
1. Deploy a contract with method `exploit`:
   a. Create promise C targeting self with args of size
      (max_receipt_size - base_receipt_overhead) bytes.
      → validate_receipt passes: size == max_receipt_size exactly.
   b. Create promise B (callback) via promise_then(A, self, "noop").
      → This writes a DataReceiver into A's output_data_receivers.
   c. Inside A's execution, call promise_return(C).
      → Runtime appends A's output_data_receivers to C's receipt.
      → C's serialised size = max_receipt_size + sizeof(DataReceiver) > max_receipt_size.
      → No re-validation; receipt is forwarded.

2. Observe: assert_oversized_receipt_occurred() finds the receipt in
   the incoming-receipts proof of a subsequent block, confirming the
   invariant is violated on-chain.
```

**Exact corrupted value:** `borsh::object_length(receipt)` of the forwarded receipt exceeds `limit_config.max_receipt_size` (4 194 304 bytes) by at least the size of one serialised `DataReceiver` (~34–100 bytes depending on account-id length).
