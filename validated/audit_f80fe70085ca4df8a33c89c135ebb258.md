### Title
`HasSelfDestructed` Ignores `recreated` Flag, Causing Silent Slot Loss and Account Deletion for Recreated Contracts — (`fvm/evm/emulator/state/stateDB.go`, `fvm/evm/emulator/state/delta.go`)

### Summary

`DeltaView.HasSelfDestructed` propagates unconditionally through the parent-view chain without checking the `recreated` flag. When a contract is self-destructed in a parent `DeltaView` and then recreated via `CreateAccount` in a child `DeltaView`, `Commit()` still sees `HasSelfDestructed == true` for that address. This causes `Commit()` to (1) delete the recreated account instead of creating it, and (2) skip all storage-slot updates written after recreation — silently discarding committed EVM state.

### Finding Description

**`DeltaView.HasSelfDestructed` — the root cause**

```go
func (d *DeltaView) HasSelfDestructed(addr gethCommon.Address) (bool, *uint256.Int) {
    bal, found := d.toBeDestructed[addr]
    if found {
        return true, bal
    }
    return d.parent.HasSelfDestructed(addr)   // ← never checks d.recreated
}
``` [1](#0-0) 

When `CreateAccount(A)` is called in a child view for an address that is already self-destructed in a parent view, the child view sets `d.recreated[addr]` and `d.created[addr]`, but does **not** add `addr` to its own `toBeDestructed` map. [2](#0-1) 

Because `HasSelfDestructed` only checks `d.toBeDestructed` locally and then falls through to `d.parent.HasSelfDestructed(addr)`, the parent's self-destruct flag is always visible to the child — even after recreation.

**`Commit()` — the consequence**

`Commit()` calls `db.HasSelfDestructed(addr)`, which delegates to `db.latestView().HasSelfDestructed(addr)` (the child view). The child view has no local `toBeDestructed` entry for `addr`, but the parent does, so the call returns `true`. `Commit()` then:

1. Calls `db.baseView.DeleteAccount(addr)` and sets `deleted = true`.
2. Hits `if deleted { continue }` — the `IsCreated` branch is never reached, so the recreated account is never committed. [3](#0-2) 

3. In the slot-update loop, `db.HasSelfDestructed(sk.Address)` again returns `true`, so every slot written after recreation is silently skipped. [4](#0-3) 

**`SetState` correctly records the slot in the child view** — the slot appears in `DirtySlots()` and is collected by `Commit()` — but is then discarded by the guard above. [5](#0-4) 

**Exact call sequence that triggers the bug:**

```
SelfDestruct6780(A)          // view[0]: A added to toBeDestructed
Snapshot()                   // view[1] created as child of view[0]
CreateAccount(A)             // view[1]: recreated[A] set, created[A] set; toBeDestructed NOT set
SetState(A, key, value)      // view[1]: slots[(A,key)] = value; DirtySlots includes (A,key)
Commit()
  → HasSelfDestructed(A) via view[1] → view[0].toBeDestructed → true
  → DeleteAccount(A); skip IsCreated; skip slot update
```

Note: `db.SelfDestruct` (legacy) is blocked with an explicit error. [6](#0-5) 

The only reachable self-destruct path is `SelfDestruct6780`, which fires when `IsNewContract(addr)` is true — i.e., the contract was created in the same transaction. [7](#0-6) 

### Impact Explanation

An attacker-controlled EVM contract can execute the following within a single transaction:

1. Deploy contract A via CREATE2 (making it a "new contract").
2. A calls SELFDESTRUCT → `SelfDestruct6780(A)` flags A for deletion in the current view.
3. The outer transaction calls CREATE2 again with the same salt. The EVM takes a snapshot before the sub-call, creating a child view.
4. In the child view, `CreateAccount(A)` is called, setting `recreated[A]`.
5. The new contract writes storage via SSTORE → `SetState(A, key, value)`.
6. `Commit()` deletes A and discards all slot writes.

Result: the contract appears to write storage (SSTORE succeeds, gas is charged, no error is returned), but the state is silently discarded. The recreated account does not exist in persistent state. This corrupts EVM resource accounting and violates the invariant that storage written after account recreation must be persisted.

### Likelihood Explanation

The CREATE2 + SELFDESTRUCT + CREATE2-at-same-address pattern is a well-known EVM technique used in production contracts (e.g., factory patterns, upgradeable proxies). It is reachable through a standard EVM transaction submitted to Flow's EVM gateway — no privileged access, no operator control, and no staked node compromise is required. The bug is deterministic and locally reproducible on the Flow emulator.

### Recommendation

In `DeltaView.HasSelfDestructed`, check `d.recreated` before propagating to the parent:

```go
func (d *DeltaView) HasSelfDestructed(addr gethCommon.Address) (bool, *uint256.Int) {
    // Recreation in this view cancels any self-destruct from a parent view.
    if _, recreated := d.recreated[addr]; recreated {
        return false, nil
    }
    bal, found := d.toBeDestructed[addr]
    if found {
        return true, bal
    }
    return d.parent.HasSelfDestructed(addr)
}
```

Alternatively, `Commit()` should check `IsCreated` before `HasSelfDestructed` so that recreation takes priority over a stale parent-chain self-destruct flag.

### Proof of Concept

A differential test comparing Flow's `StateDB` against Geth's reference `stateDB` for the sequence:

```
CREATE2(A) → SELFDESTRUCT(A) → Snapshot → CREATE2(A) → SSTORE(A, k, v) → Commit
```

On Geth: `GetState(A, k)` returns `v` after commit.  
On Flow: `GetState(A, k)` returns `0` (zero hash) after commit, and `A` does not exist in persistent state.

The assertion `flowState.GetState(A, k) == gethState.GetState(A, k)` fails, confirming the divergence.
