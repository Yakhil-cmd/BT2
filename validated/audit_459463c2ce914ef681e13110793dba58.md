### Title
Invoke Transaction with Nonce=1 Bypasses `__validate__` Signature Check When Any Account Transaction Exists in Mempool — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

`skip_stateful_validations` in the gateway's stateful validator skips the `__validate__` entry-point call (the account contract's signature check) for any Invoke transaction whose nonce is 1 and whose sender's on-chain nonce is 0, provided `account_tx_in_pool_or_recent_block` returns `true`. The guard condition is too broad: it fires not only when a `deploy_account` is pending, but also when the account has **any** future-nonce transaction in the mempool. An unprivileged attacker can therefore inject an Invoke with nonce=1 and an arbitrary/invalid signature for any victim account that already has a pending transaction, bypassing gateway-level signature verification entirely.

---

### Finding Description

The UX feature is documented in the code comment:

> "It is sufficient to check if the account exists in the mempool since it means that **either** it has a deploy_account transaction **or transactions with future nonces that passed validations**." [1](#0-0) 

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` is called with `validate: !skip_validate = false`: [2](#0-1) 

This means `StatefulValidator::perform_validations` receives an `AccountTransaction` with `execution_flags.validate = false` and returns `Ok(())` immediately after `perform_pre_validation_stage`, never calling `validate_tx`: [3](#0-2) 

The `__validate__` entry point — the only place the account contract verifies the transaction signature — is therefore never executed at the gateway admission stage.

**Attack path:**

1. Victim account `A` (on-chain nonce = 0, not yet deployed) has a legitimate Invoke with nonce=2 in the mempool (it passed signature verification because `skip_stateful_validations` returns `false` for nonce ≠ 1).
2. Attacker submits an Invoke for account `A` with nonce=1 and an **arbitrary/invalid signature**.
3. Gateway stateless checks pass (no signature content check there).
4. `validate_nonce` passes: `0 ≤ 1 ≤ 0 + max_allowed_nonce_gap`. [4](#0-3) 

5. `skip_stateful_validations`: nonce=1, account_nonce=0, `account_tx_in_pool_or_recent_block(A)` = `true` (because the nonce=2 tx is already there) → returns `true`.
6. `run_validate_entry_point` is called with `validate: false` → `__validate__` is never invoked → invalid-signature transaction is admitted to the mempool.

---

### Impact Explanation

**High — Mempool/gateway admission accepts invalid (unauthorized) transactions before sequencing.**

- The gateway admits a transaction whose signature has never been verified by the account contract.
- The attacker's nonce=1 Invoke occupies a mempool slot for the victim's account. Depending on the mempool's replacement policy, it can displace the victim's legitimate nonce=1 Invoke (the one submitted as part of the deploy_account + invoke UX flow).
- When the deploy_account eventually executes and the batcher picks up the attacker's nonce=1 Invoke, the blockifier will re-run `__validate__` (because `new_for_sequencing` sets `validate: true`), causing the transaction to revert. This effectively DoSes the victim's first post-deployment transaction. [5](#0-4) 

---

### Likelihood Explanation

Any account that has submitted a future-nonce Invoke (nonce ≥ 2) while its on-chain nonce is still 0 is vulnerable. This is a publicly observable mempool state. No privileged access is required; the attacker only needs to monitor the mempool and submit a crafted transaction.

---

### Recommendation

Tighten the bypass condition in `skip_stateful_validations` to only fire when a **`deploy_account`** transaction (not an arbitrary future-nonce Invoke) is pending for the account. The mempool should expose a dedicated `has_pending_deploy_account(address)` query, and `account_tx_in_pool_or_recent_block` should be replaced with that narrower check. Alternatively, add a check inside `skip_stateful_validations` that verifies the mempool entry for the account is specifically a `DeployAccount` transaction type before returning `true`. [6](#0-5) 

---

### Proof of Concept

```
// Setup: victim account A, on-chain nonce = 0 (not deployed)

// Step 1: Victim submits a valid Invoke with nonce=2 (passes full validation, enters mempool)
submit_invoke(sender=A, nonce=2, calldata=[...], signature=valid_sig)
// → account_tx_in_pool_or_recent_block(A) now returns true

// Step 2: Attacker submits Invoke with nonce=1 and INVALID signature
submit_invoke(sender=A, nonce=1, calldata=[MALICIOUS_CALLDATA], signature=[0xDEAD])

// Gateway stateful validation:
//   account_nonce = 0  (from state)
//   incoming nonce = 1
//   skip_stateful_validations:
//     nonce == ONE && account_nonce == ZERO  → true
//     account_tx_in_pool_or_recent_block(A)  → true  (nonce=2 tx is there)
//     returns true
//   run_validate_entry_point called with validate=false
//   __validate__ is NEVER called
//   → transaction admitted to mempool with invalid signature
```

The attacker's nonce=1 Invoke with `[MALICIOUS_CALLDATA]` and an invalid signature is now in the mempool. It will fail at execution time (blockifier re-validates), but it occupies the nonce=1 slot and can displace the victim's legitimate nonce=1 transaction.

/// Check if validation of an invoke transaction should be skipped due to deploy_account not being
/// processed yet. This feature is used to improve UX for users sending deploy_account + invoke at
/// once.
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
