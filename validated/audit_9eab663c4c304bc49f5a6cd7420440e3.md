### Title
Overly Broad `account_tx_in_pool_or_recent_block` Check Allows Signature-Validation Bypass for Invoke Transactions with Nonce 1 - (File: crates/apollo_gateway/src/stateful_transaction_validator.rs)

### Summary

`skip_stateful_validations` is designed to skip the `__validate__` entry-point call for an invoke with nonce=1 when the account's on-chain nonce is 0 and a deploy-account transaction is pending in the mempool. The guard that is supposed to confirm a deploy-account exists — `account_tx_in_pool_or_recent_block` — checks for **any** transaction belonging to the account, not specifically a deploy-account. An attacker who observes a freshly-deployed account (on-chain nonce=0) submit its first valid invoke (nonce=0) can immediately submit a second invoke (nonce=1) carrying an **invalid signature**. The gateway skips `__validate__` for that second transaction, admits it to the mempool, and the account is later charged fees when the batcher executes and reverts it.

### Finding Description

`skip_stateful_validations` in `crates/apollo_gateway/src/stateful_transaction_validator.rs` fires when three conditions are simultaneously true:

1. The incoming transaction is an `Invoke`.
2. Its nonce equals `Nonce(Felt::ONE)`.
3. The on-chain account nonce equals `Nonce(Felt::ZERO)`. [1](#0-0) 

When all three hold, the function delegates to `account_tx_in_pool_or_recent_block`: [2](#0-1) 

which returns `true` if the account appears in `tx_pool` (any pending transaction) **or** in `state.committed`/`state.staged` (any recently committed or staged transaction): [3](#0-2) 

The comment in `skip_stateful_validations` asserts this is sufficient because the account "either has a deploy_account transaction or transactions with future nonces that passed validations." That reasoning is wrong for an account that was deployed in a prior block and now has on-chain nonce=0 with a valid nonce=0 invoke already sitting in the mempool. In that state all three conditions are satisfied by a **non-deploy-account** transaction, so the guard returns `true` and `run_validate_entry_point` is called with `validate: false`: [4](#0-3) 

The attacker's nonce=1 invoke with an arbitrary (invalid) signature is therefore admitted to the mempool without any signature check.

The `validate_by_mempool` call that precedes `skip_stateful_validations` only checks for duplicate hashes and stale nonces — it performs no signature verification: [5](#0-4) 

### Impact Explanation

The invalid-signature invoke is admitted to the mempool. When the batcher later executes it, `perform_pre_validation_stage` increments the nonce and checks fees, then `run_or_revert` calls `validate_tx` (the `__validate__` entry point) with the batcher's own execution flags (which include `validate: true`). `__validate__` fails, the transaction is reverted, and the victim account is charged the revert fee. The attacker controls the `resource_bounds` field in the forged transaction, so the fee charged can be set to the maximum the victim's balance allows.

This matches the allowed impact: **"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."**

### Likelihood Explanation

The preconditions are common:
- Any account that was deployed in a prior block and has not yet sent its first transaction has on-chain nonce=0.
- The moment that account broadcasts its first valid invoke (nonce=0), the attacker can race to submit a forged nonce=1 invoke before the nonce=0 invoke is committed.
- No privileged access is required; the attacker only needs to watch the public mempool/gateway.

### Recommendation

Replace the generic `account_tx_in_pool_or_recent_block` check with one that specifically verifies a **deploy-account** transaction for the sender address is present in the mempool. The mempool should expose a dedicated query such as `deploy_account_in_pool(address)` that inspects transaction types, or `skip_stateful_validations` should be restricted to the case where the account does not yet exist on-chain (i.e., `get_class_hash_at(address) == ClassHash::default()`), which is the only situation where calling `__validate__` is genuinely impossible.

### Proof of Concept

```
1. Deploy account A on-chain (nonce becomes 0 on-chain).
2. Account A submits a valid invoke I0 (nonce=0, valid signature).
   → Gateway admits I0; mempool now has A in tx_pool.
3. Attacker submits forged invoke I1 (nonce=1, garbage signature, high resource_bounds).
   → Gateway: tx.nonce()==1 ✓, account_nonce==0 ✓,
              account_tx_in_pool_or_recent_block(A)==true ✓ (due to I0)
   → skip_stateful_validations returns true → validate=false
   → __validate__ is NOT called; I1 is admitted to the mempool.
4. Batcher executes I0 (succeeds, A's nonce becomes 1).
5. Batcher executes I1 (nonce check passes: 1==1 strict).
   → __validate__ is called → fails (invalid signature).
   → Transaction reverts; A is charged fees up to resource_bounds.
``` [6](#0-5) [7](#0-6) [2](#0-1)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
