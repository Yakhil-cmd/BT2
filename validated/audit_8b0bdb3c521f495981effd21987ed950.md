### Title
Missing Caller Check in `exit_delegation_pool_action` Allows Forced Withdrawal, Destroying Pool-Switch Opportunity — (File: `src/pool/pool.cairo`)

---

### Summary

`exit_delegation_pool_action` in `pool.cairo` accepts an arbitrary `pool_member` address with no check that the caller is that pool member. Any unprivileged address can force a victim's withdrawal after the unpool window expires, irreversibly clearing the victim's `unpool_time` and `unpool_amount` before they can call `switch_delegation_pool`. The victim receives their own tokens back but permanently loses the ability to redirect their stake to a new pool without re-serving the full exit wait window.

---

### Finding Description

The intent/action two-phase exit pattern is asymmetric in its access control:

- `exit_delegation_pool_intent` (line 258) correctly binds the intent to the caller via `get_caller_address()`.
- `exit_delegation_pool_action` (lines 295–333) accepts `pool_member: ContractAddress` as a caller-supplied parameter and performs **no check** that `get_caller_address() == pool_member`. [1](#0-0) 

After the action executes, lines 322–323 unconditionally zero out `unpool_amount` and set `unpool_time = Option::None`: [2](#0-1) 

`switch_delegation_pool` (line 390) requires `pool_member_info.unpool_time.is_some()` as a hard precondition: [3](#0-2) 

Once an attacker calls `exit_delegation_pool_action(victim)`, the victim's `unpool_time` is `None` and `switch_delegation_pool` will always revert with `MISSING_UNDELEGATE_INTENT`. The victim must call `exit_delegation_pool_intent` again and wait out the full `exit_wait_window` (default one week) before they can attempt a pool switch.

The `UndelegateIntentKey` in the staking contract is `{ pool_contract, identifier }` where `identifier = pool_member.into()`: [4](#0-3) 

The intent is correctly scoped per pool member, but the action-phase gate in the pool contract does not enforce that only the owning pool member may execute it.

---

### Impact Explanation

**Medium — Griefing with no profit motive but damage to users.**

The attacker gains nothing financially; funds are transferred to `pool_member` (the victim), not the caller: [5](#0-4) 

However, the victim suffers concrete harm:
1. Their planned pool switch is permanently blocked for that intent cycle.
2. They must re-enter `exit_delegation_pool_intent` and re-serve the full exit wait window (up to 12 weeks maximum) before they can switch pools.
3. During that forced re-wait period they remain in the original pool, potentially earning lower rewards or remaining in a pool whose staker has degraded performance.

---

### Likelihood Explanation

High likelihood. The entry path requires no privilege:
- The attacker only needs to know the victim's address (publicly observable from `NewPoolMember` events) and the pool contract address.
- The attacker simply monitors the chain for `PoolMemberExitIntent` events, waits for `unpool_time` to pass, and front-runs or races the victim's `switch_delegation_pool` call with `exit_delegation_pool_action(victim)`.
- No tokens, no role, no special access required.

---

### Recommendation

Add a caller check at the top of `exit_delegation_pool_action` so only the pool member (or their designated reward address) may execute the action:

```cairo
fn exit_delegation_pool_action(
    ref self: ContractState, pool_member: ContractAddress,
) -> Amount {
    let caller = get_caller_address();
    let pool_member_info = self.internal_pool_member_info(:pool_member);
    assert!(
        caller == pool_member || caller == pool_member_info.reward_address,
        "{}",
        Error::CALLER_CANNOT_EXIT_POOL,
    );
    // ... rest of function
}
```

This mirrors the access pattern already used in `add_to_delegation_pool`: [6](#0-5) 

---

### Proof of Concept

1. **Setup**: Pool member `Victim` calls `exit_delegation_pool_intent(amount)`. The staking contract records `UndelegateIntentKey { pool_contract: P, identifier: Victim.into() }` with `unpool_time = T`. `Victim` intends to call `switch_delegation_pool` at time `T` to move stake to a better pool.

2. **Attack**: At time `T` (or any time after), `Attacker` (any address) calls `Pool.exit_delegation_pool_action(pool_member: Victim)`.

3. **Execution**: The function reads `Victim`'s `pool_member_info`, confirms `Time::now() >= T`, calls `staking_pool_dispatcher.remove_from_delegation_pool_action(identifier: Victim.into())`, sets `pool_member_info.unpool_amount = 0` and `pool_member_info.unpool_time = None`, then transfers `unpool_amount` tokens to `Victim`.

4. **Result**: `Victim` receives their tokens back. When `Victim` now calls `switch_delegation_pool(...)`, it panics at:
   ```
   assert!(pool_member_info.unpool_time.is_some(), ..., MISSING_UNDELEGATE_INTENT)
   ``` [3](#0-2) 
   `Victim` must re-call `exit_delegation_pool_intent` and wait the full exit window again before switching pools.
