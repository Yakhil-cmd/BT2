### Title
Malicious Participant Can Pre-Inject Messages for Future Waitpoints to Permanently DoS DKG Finalization - (File: src/protocol/internal.rs, src/protocol/helpers.rs, src/dkg.rs)

### Summary
The `Protocol::message()` entry point accepts and buffers incoming messages for any waitpoint at any time, with no check that the target waitpoint is currently active. A malicious participant can pre-inject a crafted message for a future protocol round before that round begins. When `recv_from_others` later reaches that waitpoint, it consumes the pre-injected crafted message and discards the attacker's legitimate message as a duplicate, causing DKG commitment-hash verification to fail and aborting the session for all honest parties. The attacker can repeat this every session, permanently denying key generation.

### Finding Description

**Root cause — unbounded, unvalidated message buffering:**

`Comms::push_message` in `src/protocol/internal.rs` (lines 286–296) parses any incoming message's `MessageHeader` (channel tag + waitpoint) and stores it in `MessageBuffer` under that header, regardless of whether the protocol has reached that waitpoint:

```rust
fn push_message(&self, from: Participant, message: MessageData) {
    if message.len() < MessageHeader::LEN { return; }
    let Some(header) = MessageHeader::from_bytes(&message) else { return; };
    self.incoming.push(header, from, message);   // stored unconditionally
}
```

`MessageBuffer::push` (lines 236–239) inserts into a `HashMap<MessageHeader, SubMessageQueue>` with no bound and no waitpoint-validity check. The library's own test (`attacker_can_fill_message_buffer_with_unused_waitpoints`, lines 532–554) explicitly confirms that 512 messages for waitpoints 1,000,000–1,000,511 are all accepted and stored.

**Exploit path through `recv_from_others` and DKG:**

`recv_from_others` in `src/protocol/helpers.rs` (lines 6–27) collects exactly one message per participant at a given waitpoint using `ParticipantCounter`:

```rust
while !seen.full() {
    let (from, msg) = chan.recv(waitpoint).await?;
    if seen.put(from) {          // first message from `from` wins
        messages.push((from, msg));
    }
}
```

`seen.put(from)` returns `false` for any subsequent message from the same participant, so the **first** message buffered for a given `(channel, waitpoint, from)` triple is the one that is accepted; all later arrivals are silently dropped.

In `do_keyshare` (`src/dkg.rs`, lines 342–538), the DKG commitment-hash round uses `wait_round_1` (the first `next_waitpoint()` call, always waitpoint 0 on the shared channel):

```rust
let wait_round_1 = chan.next_waitpoint();          // line 414
chan.send_many(wait_round_1, &commitment_hash)?;   // line 415
// ...
for (from, their_commitment_hash) in
    recv_from_others(&chan, wait_round_1, &participants, me).await?  // line 423
{
    all_hash_commitments.put(from, their_commitment_hash);
}
```

Later, in Round 4, every received commitment is verified against the stored hash (lines 462–469):

```rust
verify_commitment_hash(
    &session_id, p, &mut commit_domain_separator.clone(),
    commitment_i, &all_hash_commitments,
)?;
```

**Concrete attack sequence:**

1. Attacker (participant A) is admitted to a DKG session with honest parties B, C, D.
2. Before the session's `wait_round_1` is reached, A sends a crafted message over the network whose header encodes `(root_shared_channel_tag, waitpoint=0)` and whose payload is an arbitrary wrong hash `H_crafted`.
3. The honest participant's orchestration layer calls `Protocol::message(A, crafted_bytes)`, which stores `H_crafted` in the buffer under `(root_shared, 0)`.
4. The protocol reaches `wait_round_1`. `recv_from_others` calls `chan.recv(0)`, pops `(A, H_crafted)`, and `seen.put(A)` returns `true` → `H_crafted` is stored in `all_hash_commitments`.
5. A's legitimate commitment hash `H_legitimate` (sent at the correct protocol time) arrives. `seen.put(A)` returns `false` → silently discarded.
6. In Round 4, `verify_commitment_hash` computes `H(commitment_A)` and compares it to `H_crafted`. They differ → `ProtocolError::InvalidCommitmentHash` → DKG aborts for all honest parties.
7. A repeats this every time honest parties restart DKG.

The same attack applies to the private-share round (`wait_round_3`, lines 499–528): A pre-injects a wrong `SigningShare`, which fails `validate_received_share` and aborts DKG.

### Impact Explanation

Every DKG session (keygen, reshare, refresh) is permanently deniable by a single malicious participant. The attacker does not need to exceed the Byzantine fault threshold; one participant is sufficient. Honest parties cannot complete key generation as long as the attacker remains in the participant set. This maps to **High: Permanent denial of key generation, reshare, and refresh for honest parties under valid protocol inputs and documented trust assumptions**.

### Likelihood Explanation

Any participant admitted to a DKG session can execute this attack with no special privileges. The attacker only needs to send a single crafted network message before the target round begins. Because the protocol's network layer is asynchronous and message ordering is not enforced, early delivery is trivially achievable. The attack is repeatable with zero cost per attempt.

### Recommendation

1. **Validate waitpoints on receipt**: In `Comms::push_message`, reject messages whose waitpoint is strictly greater than the highest waitpoint allocated so far by `next_waitpoint()`. Track the current maximum allocated waitpoint in `Comms` and drop out-of-range messages.
2. **Alternatively, enforce per-round message windows**: Only accept messages for the currently active waitpoint(s) and buffer at most one message per `(waitpoint, participant)` pair, returning an error on duplicates.
3. **Bound the message buffer**: Cap the number of buffered entries per `(channel, waitpoint)` to the participant count, dropping excess messages.

pub async fn recv_from_others<T>(
    chan: &SharedChannel,
    waitpoint: u64,
    participants: &ParticipantList,
    me: Participant,
) -> Result<Vec<(Participant, T)>, ProtocolError>
where
    T: serde::de::DeserializeOwned,
{
    let mut seen = ParticipantCounter::new(participants);
    seen.put(me);
    let mut messages = Vec::with_capacity(participants.others(me).count());

    while !seen.full() {
        let (from, msg) = chan.recv(waitpoint).await?;
        if seen.put(from) {
            messages.push((from, msg));
        }
    }

    Ok(messages)
}
