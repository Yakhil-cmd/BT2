### Title
Missing Proof-of-Correct-Computation for CKD Share Contributions Allows Any Malicious Participant to Corrupt the Derived Confidential Key - (File: src/confidential_key_derivation/protocol.rs)

---

### Summary

In the Confidential Key Derivation (CKD) protocol, the coordinator blindly aggregates `(big_y, big_c)` share contributions from all participants with no cryptographic proof that each contribution was honestly computed. A single malicious participant can substitute arbitrary group elements, silently corrupting the final `CKDOutput` and causing the requester to derive a wrong, unusable confidential key.

---

### Finding Description

The CKD protocol works as follows:

1. Each participant `i` computes a share `(norm_big_y_i, norm_big_c_i)` in `compute_signature_share` where:
   - `big_y_i = y_i * G` (random blinding point)
   - `big_c_i = x_i * H(pk, app_id) + y_i * app_pk` (masked secret share contribution)
   - Both are scaled by the Lagrange coefficient `λ_i`

2. Each participant sends this pair privately to the coordinator.

3. The coordinator in `do_ckd_coordinator` aggregates all received values:

```rust
// src/confidential_key_derivation/protocol.rs lines 50-55
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
```

There is **no verification** that the received `(big_y_i, big_c_i)` from participant `i` satisfies the required relation. Specifically, there is no zero-knowledge proof that:
- `big_y_i` is a valid scalar multiple of the generator `G`
- `big_c_i` is of the form `x_i * H(pk, app_id) + y_i * app_pk` for the same `y_i` used in `big_y_i`

A malicious participant `i` can instead send `(norm_big_y_i + Δ_Y, norm_big_c_i + Δ_C)` for arbitrary group elements `Δ_Y, Δ_C ∈ G1`. The coordinator accumulates these without complaint.

The requester then calls `unmask(app_sk)`:

```rust
// src/confidential_key_derivation/mod.rs lines 54-56
pub fn unmask(&self, secret_scalar: Scalar) -> Signature {
    self.big_c - self.big_y * secret_scalar
}
```

The result becomes:

```
(msk · H(pk, app_id) + Δ_C) − app_sk · (Y_total · G + Δ_Y)
= msk · H(pk, app_id) + Δ_C − app_sk · Δ_Y
```

Unless `Δ_C = app_sk · Δ_Y` (which the attacker cannot enforce without knowing `app_sk`), the output is a uniformly random, wrong group element — indistinguishable from a valid key by the requester.

---

### Impact Explanation

**High — Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.**

The requester receives a `CKDOutput` that passes all structural checks (it is a valid `G1` point) but decrypts to a wrong value. The derived confidential key `msk · H(pk, app_id)` is silently replaced by a garbage element. Any downstream use of this key (e.g., decryption, authentication, deterministic secret derivation inside a TEE) will fail or produce attacker-influenced outputs. The honest coordinator and requester have no mechanism to detect the corruption.

---

### Likelihood Explanation

**High.** Any single participant in the CKD session is sufficient to trigger this. The attack requires only that the participant deviate from the protocol by sending two arbitrary `G1` points instead of their honest contribution. No special knowledge (e.g., `app_sk`, other shares) is required. The attack is undetectable by the coordinator because no proof accompanies the share.

---

### Recommendation

Each participant must accompany their `(norm_big_y_i, norm_big_c_i)` contribution with a non-interactive zero-knowledge proof of correct formation. Concretely, a Chaum-Pedersen / sigma proof should prove knowledge of `(x_i, y_i)` such that:

- `big_y_i = y_i · G`
- `big_c_i = x_i · H(pk, app_id) + y_i · app_pk`
- `x_i` is consistent with the participant's public commitment from DKG (i.e., `x_i · G2_generator = vk_share_i`)

The coordinator must verify all such proofs before aggregating in `do_ckd_coordinator`. Only after all proofs pass should the coordinator sum the contributions.

---

### Proof of Concept

**Setup:** 3-of-3 CKD session. Participants are `P1` (honest), `P2` (honest), `P3` (malicious). Coordinator is `P1`.

1. `P1` and `P2` compute honest shares `(Y1, C1)` and `(Y2, C2)` and send them to `P1` (coordinator).
2. `P3` computes its honest share `(Y3, C3)` but instead sends `(Y3 + Δ_Y, C3 + Δ_C)` for arbitrary `Δ_Y, Δ_C ∈ G1`.
3. The coordinator executes the loop at [1](#0-0)  and accumulates `total_Y = Y1+Y2+Y3+Δ_Y`, `total_C = C1+C2+C3+Δ_C`.
4. The coordinator returns `CKDOutput { big_y: total_Y, big_c: total_C }` at [2](#0-1) .
5. The requester calls `unmask(app_sk)` at [3](#0-2)  and obtains `msk·H(pk,app_id) + Δ_C − app_sk·Δ_Y`, which is a wrong group element.
6. No error is raised anywhere in the protocol. The requester silently accepts a corrupted confidential key.

The root cause is the absence of any proof-of-correct-computation in `compute_signature_share` [4](#0-3)  and the unconditional aggregation in `do_ckd_coordinator` [5](#0-4) .

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}
