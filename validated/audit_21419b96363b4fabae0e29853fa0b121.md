### Title
Missing Validation of Participant Contributions in CKD Protocol Allows Malicious Participant to Corrupt CKD Output - (File: src/confidential_key_derivation/protocol.rs)

### Summary
The `do_ckd_coordinator` function receives `CKDOutput` values from each participant and accumulates them without any validation or cryptographic verification. Analogous to the missing zero-amount check in `stashRewards()`, there is no guard before the accumulation step to verify that each participant's contribution is well-formed or honestly computed. A malicious participant can send arbitrary group elements, silently corrupting the final CKD output accepted by the coordinator.

### Finding Description
In `do_ckd_coordinator` (lines 50–55 of `src/confidential_key_derivation/protocol.rs`), the coordinator collects each participant's `(big_y, big_c)` pair and unconditionally adds them into the running aggregate:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
```

Each participant is expected to send:
- `big_y = λ_i · y_i · G` (a Lagrange-weighted blinding factor)
- `big_c = λ_i · (x_i · H(pk, app_id) + y_i · app_pk)` (a Lagrange-weighted encrypted share)

These values are computed locally in `compute_signature_share` (lines 148–181) and sent privately to the coordinator. There is no zero-knowledge proof, commitment binding, or any other cryptographic check that the received `(big_y, big_c)` pair was honestly derived from the participant's actual key share `x_i` and a fresh random `y_i`. The coordinator has no mechanism to distinguish a correct contribution from an arbitrary group element.

This is structurally identical to the external report's pattern: an operation (accumulation) is performed on externally supplied values without a guard check, and because the values are participant-controlled, they can be set to anything — including the group identity or a point chosen to bias the output toward an attacker-controlled value.

### Impact Explanation
A single malicious participant can replace their honest `(big_y, big_c)` with arbitrary BLS12-381 G1 elements. Because the coordinator simply sums all contributions, the final `CKDOutput` will be:

```
norm_big_y = (honest sum) + (attacker's arbitrary big_y)
norm_big_c = (honest sum) + (attacker's arbitrary big_c)
```

The resulting `CKDOutput` no longer encodes the correct confidential key `msk · H(pk, app_id)`. Any downstream consumer that calls `ckd_output.unmask(app_sk)` will recover a wrong, attacker-influenced value instead of the intended deterministic secret. This falls squarely under **High: Corruption of CKD outputs so honest parties accept unusable cryptographic outputs**.

### Likelihood Explanation
The attacker's entry point is the private message channel: any participant in the CKD session can send an arbitrary `CKDOutput` to the coordinator. No special privilege is required beyond being a listed participant. The attack requires no cryptographic break, no leaked key, and no external dependency — only the ability to deviate from the protocol when constructing the private message in `do_ckd_participant`. This is a straightforward, one-round, zero-cost manipulation.

### Recommendation
Each participant's contribution should be bound to their public key share before the coordinator accepts it. Concretely:

1. **Zero-knowledge proof of correct encryption**: Each participant should accompany `(big_y, big_c)` with a Chaum-Pedersen or Sigma proof demonstrating that `big_c - y_i · app_pk` lies on the correct coset defined by their public key share `X_i = x_i · G2` and `H(pk, app_id)`. The coordinator verifies each proof before accumulating.
2. **Minimum guard — identity check**: At minimum, reject contributions where `big_y` or `big_c` is the group identity, analogous to the recommended `if (amount > 0)` guard in the external report:

```rust
if participant_output.big_y() == ElementG1::identity()
    || participant_output.big_c() == ElementG1::identity()
{
    return Err(ProtocolError::IdentityElement);
}
norm_big_y += participant_output.big_y();
norm_big_c += participant_output.big_c();
```

The identity check is a necessary but not sufficient fix; a full ZK proof is required for security against arbitrary malicious contributions.

### Proof of Concept
A malicious participant overrides `do_ckd_participant` to send:

```rust
// Instead of the honest (norm_big_y, norm_big_c), send identity elements
let fake_output = CKDOutput::new(ElementG1::identity(), ElementG1::identity());
chan.send_private(waitpoint, coordinator, &fake_output)?;
```

The coordinator accumulates this without complaint. The resulting `ckd_output` satisfies:

```
norm_big_y = (sum of honest λ_i · y_i · G) + identity  ← unchanged
norm_big_c = (sum of honest λ_i · (x_i·H + y_i·A)) + identity  ← unchanged
```

In this degenerate case the attacker merely removes their own share from the aggregate, so `unmask(app_sk)` recovers a value that does not equal `msk · H(pk, app_id)` (it equals `(msk - λ_attacker · x_attacker) · H(pk, app_id)` plus a blinding residual). With a non-identity chosen point the attacker can bias the output to any target value in G1. [1](#0-0) [2](#0-1) [3](#0-2)

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
