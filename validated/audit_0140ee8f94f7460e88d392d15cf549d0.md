### Title
Unverified NFT Image Rendered Without Warning in Notification Thumbnails Due to `isCompact` Suppressing `NFTHashStatus` — (`packages/gui/src/components/nfts/NFTPreview.tsx`)

### Summary

`NotificationPreviewNFT` always renders `NFTPreview` with `isCompact={true}`. `NFTPreview` unconditionally suppresses the `NFTHashStatus` overlay when `isCompact` is true. Meanwhile, `useNFTVerifyHash` falls back to returning an unverified URI when hash verification fails, and `preparePreview` renders that URI's content in a `SandboxedIframe` without checking `isVerified`. The result: an attacker who controls the server behind an NFT's data URI can serve arbitrary image content that is displayed in the notification thumbnail with no visible warning.

---

### Finding Description

**Step 1 — Entry point: `NotificationPreviewNFT`** [1](#0-0) 

`isCompact` is hardcoded to `true` and `preview` is also set, so `NFTPreview` is always called in compact-preview mode.

**Step 2 — `NFTHashStatus` is gated on `!isCompact`** [2](#0-1) 

The entire `NFTHashStatus` block is wrapped in `{!isCompact && !hideStatus && ...}`. When `isCompact=true`, the overlay is never mounted, regardless of verification state.

**Step 3 — `useNFTVerifyHash` returns unverified URI as fallback**

When every URI fails hash verification, `findValidUri` stores the first failure and returns it: [3](#0-2) 

The `previewState` memo then returns this unverified state as a last resort: [4](#0-3) 

So `preview` in `NFTPreview` is `{ isVerified: false, uri: attacker_uri, error: ... }`.

**Step 4 — `preparePreview` renders the URI without checking `isVerified`** [5](#0-4) 

`preparePreview` only checks that `preview?.uri` is non-empty and that `getURI()` returns a `cache://` URI. It never inspects `preview.isVerified`. Since `getChecksum` (called inside `findValidUri`) fetches and caches the content as a side effect, `getURI` will return a `cache://` URI for the same attacker-controlled content, and the image is rendered inside `SandboxedIframe`.

**Step 5 — `isCompact` does NOT suppress image rendering** [6](#0-5) 

The compact early-return only fires when `previewFileType !== FileType.IMAGE`. For image URIs, execution falls through to the `SandboxedIframe` render path, so the attacker's image is displayed at full thumbnail size.

---

### Impact Explanation

An attacker who controls the server behind an NFT's `dataUris` (e.g., the NFT creator, or anyone who has taken over the hosting) can serve an image that does not match the on-chain `dataHash`. That image is displayed as the NFT thumbnail inside the notification panel with no "Invalid hash" chip, no error icon, and no tooltip. The user has no signal that the content is unverified. This enables spoofing of NFT identity in the notification UI — for example, displaying a fake "VERIFIED" badge image or impersonating a different NFT's artwork — which could cause a user to act on a notification believing it represents a legitimate, verified asset.

This falls under: *Corruption, spoofing, or unsafe trust of NFT metadata / notification state that causes a user to display the wrong asset identity or status.*

### Likelihood Explanation

The attacker only needs to control the HTTP server behind one of the NFT's listed URIs — a realistic capability for any NFT creator or anyone who has compromised the hosting. No wallet keys, local access, or cryptographic breaks are required. The notification flow is triggered automatically when the wallet receives an NFT-related notification.

### Recommendation

1. In `NFTPreview`, do not suppress `NFTHashStatus` solely based on `isCompact`. Either render a compact variant of the status indicator, or refuse to render unverified content in compact mode entirely.
2. In `preparePreview`, gate rendering on `preview.isVerified === true`. If `isVerified` is false, render a placeholder or icon instead of the attacker-controlled content.
3. Alternatively, `NotificationPreviewNFT` should pass `hideStatus={false}` and use a non-compact mode, or implement its own inline hash-status indicator.

### Proof of Concept

1. Mint an NFT with `dataUris: ["https://attacker.example/img.png"]` and `dataHash` set to the SHA256 of a legitimate image.
2. Configure `https://attacker.example/img.png` to serve a different image (e.g., a fake "VERIFIED" badge).
3. Trigger a wallet notification referencing that NFT ID.
4. Mount `NotificationPreviewNFT` with that `nftId`.
5. Observe: `useNFTVerifyHash` returns `{ isVerified: false, uri: "https://attacker.example/img.png" }` for `preview`; `preparePreview` fetches and caches the attacker image; `SandboxedIframe` renders it; `NFTHashStatus` is absent from the DOM because `isCompact=true`.
