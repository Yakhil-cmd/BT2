### Title
Unbounded `stakers` Vec Growth via Stake/Unstake Cycling Causes Permanent Gas Exhaustion in `get_stakers` - (File: src/staking/staking.cairo)

### Summary
An attacker can permanently grow the `stakers` storage Vec by cycling through `stake` → `unstake_intent` → `unstake_action` with many distinct addresses. Because staker addresses are never removed from the Vec, and `get_stakers` iterates over the entire Vec on every call, the function's gas cost grows without bound. This is the direct analog of the external report's "actions limits manipulation" pattern: repeated entry/exit cycles inflate a shared state counter, eventually disabling a critical protocol function for all users.

### Finding Description
In `src/staking/staking.cairo`, every call to `stake` appends the caller's address to the `stakers` Vec:

```cairo
// Add staker address to the stakers vector.
self.stakers.push(staker_address);
``` [1](#0-0) 

The storage declaration explicitly documents that entries are never removed:

```cairo
/// Vector of staker addresses.
/// **Note**: Stakers are not removed from this vector when they unstake.
stakers: Vec<ContractAddress>,
``` [2](#0-1) 

`unstake_action` deletes the staker's `staker_info` record and clears the operational address mapping, but never touches the `stakers` Vec: [3](#0-2) 

`get_stakers`, the consensus-critical view function, iterates over the **full** Vec on every invocation:

```cairo
for staker_address_ptr in self.stakers.into_iter_full_range() {
    let staker_address = staker_address_ptr.read();
    if !self.is_staker_active(:staker_address, :epoch_id) {
        continue;
    }
    ...
}
``` [4](#0-3) 

Even for inactive (already-unstaked) entries the loop still performs a storage read of the address and a call to `is_staker_active`, which itself reads `staker_info` and `staker_unstake_intent_epoch`. The per-iteration cost is non-trivial and accumulates linearly with Vec length.

The `assert_staker_address_not_reused` guard prevents reusing the same address, but does not prevent an attacker from using a fresh address each cycle: [5](#0-4) 

### Impact Explanation
`get_stakers` is the function the consensus layer calls to obtain the full validator set for a given epoch: [6](#0-5) 

As the Vec grows, the execution resources required by `get_stakers` grow linearly. Once the Vec is large enough, the function will exceed Starknet's per-call resource limits (Cairo steps / builtins), making it permanently uncallable. This freezes the protocol's ability to produce a validator set, which is a prerequisite for consensus reward distribution and attestation. This matches the **Medium** impact class: *Griefing with no profit motive but damage to users or protocol; Unbounded gas consumption*.

### Likelihood Explanation
The attacker's only recurring cost is gas. The staked principal is fully returned after `unstake_action`. With Starknet's low transaction fees, an attacker can profitably (from a griefing perspective) execute thousands of stake/unstake cycles across distinct addresses. The minimum stake requirement raises the capital cost per cycle, but since funds are returned, the net cost is gas only. The attack is permissionless and requires no privileged access.

### Recommendation
Remove staker addresses from the `stakers` Vec when `unstake_action` is called, or replace the Vec with a structure that supports O(1) deletion (e.g., a swap-and-pop pattern). Alternatively, maintain a separate active-staker count and enforce a hard cap on Vec iteration depth inside `get_stakers`, skipping already-deleted slots via a tombstone flag.

### Proof of Concept

1. Attacker generates N fresh addresses `[A_1, A_2, ..., A_N]`, each funded with `min_stake` STRK.
2. For each `A_i`:
   - Call `stake(reward_address, operational_address, min_stake)` → `stakers.push(A_i)` executes; Vec length = i.
   - Call `unstake_intent()` → sets `unstake_time`, removes from `total_stake`, but Vec is unchanged.
   - Wait `exit_wait_window` seconds.
   - Call `unstake_action(A_i)` → `staker_info` is deleted, funds returned; Vec still contains `A_i`.
3. After N cycles, `stakers.len() == N` (plus any legitimate stakers).
4. Any call to `get_stakers(epoch_id)` now iterates N entries. For each entry the loop reads the address from Vec storage and calls `is_staker_active` (two storage reads). For the N ghost entries `is_staker_active` returns `false` immediately (since `staker_info` is `None`), but the reads still consume resources.
5. At sufficiently large N, `get_stakers` exceeds Starknet's execution resource limit and reverts, permanently disabling the consensus validator-set query.
