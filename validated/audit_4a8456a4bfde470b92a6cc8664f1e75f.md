### Title
Single Malicious Participant Can Permanently Abort DKG/Reshare/Refresh via False `broadcast_success` Vote - (File: src/dkg.rs)

---

### Summary

The `broadcast_success` function in `src/dkg.rs` requires **all N participants** to broadcast `true` in the final DKG round. A single malicious participant within the documented `MaxFaulty = floor((N-1)/3)` tolerance can broadcast `false`, causing every honest party to abort the protocol permanently. This is structurally analogous to the Moloch emergency-processing bug: just as a blocked deposit transfer prevents any proposal from being processed, a single false vote here blocks the entire key-generation, reshare, or refresh operation for all honest parties.

---

### Finding Description

At the conclusion of `do_keyshare`, after all share verification is complete, the protocol calls:

```rust
// src/dkg.rs line 531
broadcast_success(&mut chan, &participants, me, session_id).await?;
```

`broadcast_success` always sends `(true, session_id)` on behalf of the honest caller, then collects the echo-broadcast output from every participant and enforces:

```rust
// src/dkg.rs lines 329-334
if !vote_list.iter().all(|&(boolean, _)| boolean) {
    return Err(ProtocolError::AssertionFailed(
        "A participant
            seems to have failed its checks. Aborting Protocol!"
            .to_string(),
    ));
}
``` [1](#0-0) 

The underlying `do_broadcast` is the Byzantine Reliable Broadcast (echo broadcast), which **guarantees agreement**: if a malicious participant sends `(false, session_id)`, every honest party will eventually deliver `false` for that participant's slot. The `.all(...)` predicate then fails for every honest party simultaneously, causing a universal abort.

The echo broadcast itself tolerates up to `MaxFaulty` Byzantine participants. However, the **application-level check** imposes a unanimity requirement (`all`) that is strictly stronger than what the BFT model guarantees. A single Byzantine participant — well within the `MaxFaulty` bound for any N ≥ 4 — can exploit this gap. [2](#0-1) 

The same `do_keyshare` function is the shared core of `do_keygen`, `do_reshare`, and the refresh path, so all three operations are affected. [3](#0-2) [4](#0-3) 

---

### Impact Explanation

**High — Permanent denial of key generation, reshare, and refresh for honest parties.**

A single malicious participant can prevent the group from ever completing DKG, resharing to a new participant set, or refreshing shares. Because the abort happens inside the echo-broadcast finalization step (after all cryptographic material has already been exchanged), the honest parties cannot simply retry with the same session; they must restart from scratch. If the malicious participant repeats the attack on every attempt, the group is permanently locked out of threshold signing.

---

### Likelihood Explanation

**High.** The attacker needs only to be one of the N registered participants — a role that is explicitly granted by the protocol operator. No cryptographic capability, leaked key, or external dependency is required. The attack requires sending a single message (`false` instead of `true`) in the final broadcast round. Any participant who wishes to grief the group (e.g., a departing member during reshare, or a participant who wants to prevent a key refresh) can trivially trigger this.

---

### Recommendation

Replace the unanimity check with a threshold-aware check. The protocol already has cryptographic evidence (verified commitments and shares) that each honest participant succeeded. The `broadcast_success` step should only abort if **more than `MaxFaulty` participants** report failure, consistent with the BFT model used everywhere else in the protocol. Alternatively, remove the `broadcast_success` round entirely: by the time it is reached, every honest party has already verified all received shares against the broadcast commitments, so the round provides no additional cryptographic guarantee and exists solely as a DoS vector.

---

### Proof of Concept

1. N=4 participants run DKG; `MaxFaulty = floor(3/3) = 1`.
2. Participant M (malicious) completes all cryptographic steps honestly (receives valid shares, passes all verifications).
3. In the `broadcast_success` round, instead of sending `(true, session_id)`, M sends `(false, session_id)` through the echo-broadcast channel.
4. The echo-broadcast agreement property ensures all three honest participants deliver `false` for M's slot.
5. Each honest participant evaluates `vote_list.iter().all(|&(boolean, _)| boolean)` → `false`, and returns `ProtocolError::AssertionFailed`.
6. DKG is aborted for all honest parties. M repeats on every retry, permanently denying key generation. [5](#0-4)

async fn broadcast_success(
    chan: &mut SharedChannel,
    participants: &ParticipantList,
    me: Participant,
    session_id: HashOutput,
) -> Result<(), ProtocolError> {
    // broadcast node me succeded
    let vote_list = do_broadcast(chan, participants, me, (true, session_id)).await?;
    // unwrap here would never fail as the broadcast protocol ends only when the map is full
    let vote_list = vote_list
        .into_vec_or_none()
        .ok_or_else(|| ProtocolError::AssertionFailed("vote_list is empty".to_string()))?;
    // go through all the list of votes and check if any is fail or some does not contain the session id

    if !vote_list.iter().all(|(_, ref sid)| sid == &session_id) {
        return Err(ProtocolError::AssertionFailed(
            "A participant
                broadcast the wrong session id. Aborting Protocol!"
                .to_string(),
        ));
    }

    if !vote_list.iter().all(|&(boolean, _)| boolean) {
        return Err(ProtocolError::AssertionFailed(
            "A participant
                seems to have failed its checks. Aborting Protocol!"
                .to_string(),
        ));
    }
    // Wait for all the tasks to complete
    Ok(())
}

pub async fn do_reshare<C: Ciphersuite>(
    chan: SharedChannel,
    participants: ParticipantList,
    me: Participant,
    threshold: impl Into<ReconstructionLowerBound>,
    old_signing_key: Option<SigningShare<C>>,
    old_public_key: VerifyingKey<C>,
    old_participants: ParticipantList,
    mut rng: impl CryptoRngCore,
) -> Result<KeygenOutput<C>, ProtocolError> {
    let threshold = threshold.into();
    let intersection = old_participants.intersection(&participants);
    // either extract the share and linearize it or set it to zero
    let secret = old_signing_key
        .map(|x_i| {
            intersection
                .lagrange::<C>(me)
                .map(|lambda| lambda * x_i.to_scalar())
        })
        .transpose()?
        .unwrap_or_else(<C::Group as Group>::Field::zero);

    let old_reshare_package = Some((old_public_key, old_participants));
    let keygen_output = do_keyshare::<C>(
        chan,
        participants,
        me,
        threshold,
        secret,
        old_reshare_package,
        &mut rng,
    )
    .await?;

    Ok(keygen_output)
