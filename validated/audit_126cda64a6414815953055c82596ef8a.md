### Title
The `N1 = N2 = 2t+1` Enforcement in Robust ECDSA Gives a False Sense of Security Against Split-View Attacks — Presignature Reuse Is Not Prevented - (File: `src/ecdsa/robust_ecdsa/sign.rs`)

---

### Summary

The Robust ECDSA signing protocol enforces `participants.len() == 2*max_malicious+1` in both `presign()` and `sign()` as a documented mitigation against split-view attacks. This check is analogous to `maxRecordsPerTransaction` in the external report: it is a real, enforced guard that gives integrators a false sense of security, because a malicious coordinator can still execute the split-view attack by reusing the same presignature across two separate signing sessions — each of which individually satisfies the `N1 = N2 = 2t+1` constraint. The code provides no mechanism to prevent presignature reuse, and the secret key can be extracted from two such sessions via standard ECDSA nonce-reuse.

---

### Finding Description

The security documentation in `docs/ecdsa/robust_ecdsa/signing.md` lists four constraints required to prevent split-view attacks:

1. Use exactly `N1 = N2 = 2t+1` participants.
2. Ensure all participants agree on `(h, ε)` and the signing set.
3. **Never reuse a presignature.**
4. Do not sign with `h = 0`.

The code enforces constraints 1 and 4 at initialization time: [1](#0-0) 

Constraint 4 (`msg_hash != 0`) is also enforced: [2](#0-1) 

However, **constraint 3 (presignature non-reuse) is entirely absent from the code**. The `sign()` function accepts a `RerandomizedPresignOutput` as a caller-supplied argument with no tracking of whether the underlying presignature has been consumed: [3](#0-2) 

The `PresignOutput` and `RerandomizedPresignOutput` structs carry no usage flag or nonce: [4](#0-3) 

The rerandomization step — which binds the presignature to a specific `(h, ε, participants, entropy)` — is performed entirely off-protocol by the caller via `RerandomizedPresignOutput::rerandomize_presign()`: [5](#0-4) 

The signing protocol itself never sees the rerandomization parameters; it only receives the already-rerandomized shares. There is no in-protocol broadcast or consistency check that all participants used the same `(h, ε)`.

---

### Impact Explanation

**Critical — Extraction of the aggregate private signing key.**

The security documentation explicitly states:

> "If different subsets of size at least 2t+1 sign different (h, ε) values using shares derived from the same presignature, the resulting signatures use multiplicatively related nonces and the secret key can be recovered using standard ECDSA nonce-reuse attacks." [6](#0-5) 

A malicious coordinator who controls the rerandomization argument distribution can:

1. Run signing session 1: distribute `RerandomizationArguments(h1, ε1, R, entropy1)` to all 2t+1 participants → each computes `delta_1 = HKDF(...)` → produces valid signature `(R·delta_1, s1)` for message `h1`.
2. Run signing session 2 with the **same** `PresignOutput` (same `big_r`): distribute `RerandomizationArguments(h2, ε2, R, entropy2)` → each computes `delta_2` → produces valid signature `(R·delta_2, s2)` for message `h2`.

Both sessions satisfy `N1 = N2 = 2t+1`. The two resulting signatures share a multiplicatively related nonce (`R·delta_1` and `R·delta_2` are both scalar multiples of the same base point `R`). Standard ECDSA nonce-reuse equations then yield the aggregate secret key `x`.

---

### Likelihood Explanation

**Medium.** The attack requires a malicious coordinator, which is a reachable role: the coordinator is simply one of the `2t+1` signing participants, chosen by the application layer. No privileged key material or external compromise is needed. The coordinator only needs to distribute different `RerandomizationArguments` to participants across two signing sessions — a straightforward API-level manipulation. The security documentation acknowledges this threat model explicitly: [7](#0-6) 

---

### Recommendation

1. **Document clearly** that the `N1 = N2 = 2t+1` check does **not** prevent presignature reuse. The current code and README imply that enforcing this constraint is sufficient to prevent split-view attacks, which is false. [8](#0-7) 

2. **Enforce single-use presignatures at the API level**: consume or zeroize the `PresignOutput` after a single call to `rerandomize_presign`, or wrap it in a type that enforces single-use (e.g., a `OnceCell` or move-only wrapper that cannot be cloned after use).

3. **Add an in-protocol consistency check**: have each participant broadcast a commitment to their rerandomization parameters `(h, ε, participants, entropy)` before sending their signature share, so that any participant can detect a coordinator presenting inconsistent parameters.

---

### Proof of Concept

```
Setup: t = 1, N = 3 participants {P1, P2, P3}, threshold = 2t+1 = 3.

Step 1: Run presigning with all 3 participants.
        Each Pi holds PresignOutput with shared big_r = R.

Step 2 (Session A): Malicious coordinator C (= P1) distributes:
        RerandomizationArguments { msg_hash: h1, tweak: ε1, big_r: R, entropy: e1 }
        to P1, P2, P3.
        Each Pi computes delta_1 = HKDF(pk, ε1, h1, R, {P1,P2,P3}, e1).
        Each Pi calls sign(participants=[P1,P2,P3], coordinator=P1, max_malicious=1,
                          presignature=rerandomize(PresignOutput, delta_1), msg_hash=h1).
        C collects s_i shares → valid signature (R·delta_1, s_A) for h1.

Step 3 (Session B): Same coordinator reuses the same PresignOutput.
        Distributes RerandomizationArguments { msg_hash: h2, tweak: ε2, big_r: R, entropy: e2 }.
        Each Pi computes delta_2 ≠ delta_1.
        Each Pi calls sign(..., presignature=rerandomize(PresignOutput, delta_2), msg_hash=h2).
        C collects s_i shares → valid signature (R·delta_2, s_B) for h2.

Step 4: C now has two ECDSA signatures:
        (r_A = x(R·delta_1), s_A) for h1
        (r_B = x(R·delta_2), s_B) for h2
        with nonces k_A = (1/k)·delta_1^{-1} and k_B = (1/k)·delta_2^{-1}.
        Since k_A / k_B = delta_2 / delta_1 (known to C), standard nonce-reuse
        linear equations recover the secret key x.
```

The `N1 = N2 = 2t+1` check passes in both sessions. No check in `sign()` or `presign()` prevents this reuse. [9](#0-8) [10](#0-9)

impl RerandomizedPresignOutput {
    pub fn rerandomize_presign(
        presignature: &PresignOutput,
        args: &RerandomizationArguments,
    ) -> Result<Self, ProtocolError> {
        if presignature.big_r != args.big_r {
            return Err(ProtocolError::IncompatibleRerandomizationInputs);
        }
        let delta = args.derive_randomness()?;
        if delta.is_zero().into() {
            return Err(ProtocolError::ZeroScalar);
        }

        // cannot be zero due to the previous check
        let inv_delta = delta.invert().unwrap();

        // delta * R
        let rerandomized_big_r = presignature.big_r * delta;

        // alpha * delta^{-1}
        let rerandomized_alpha = presignature.alpha * inv_delta;

        // (beta + c*tweak) * delta^{-1}
        let rerandomized_beta =
            (presignature.beta + presignature.c * args.tweak.value()) * inv_delta;

        Ok(Self {
            big_r: rerandomized_big_r.into(),
            alpha: rerandomized_alpha,
            beta: rerandomized_beta,
            e: presignature.e,
        })
    }
