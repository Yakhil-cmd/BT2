### Title
Pool admin bypasses oracle rotation timelock by setting `priceProviderTimelock = 0`, enabling atomic oracle swap and LP fund drain — (File: `metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`MetricOmmPoolFactory._validatePoolParameters` does not enforce a minimum value for `priceProviderTimelock` when a mutable-oracle pool is created. A pool admin who sets `priceProviderTimelock = 0` can call `proposePoolPriceProvider` and `executePoolPriceProviderUpdate` in the same block — or even the same transaction — with zero delay, bypassing the protection that was explicitly added (AuditFinding M01) to prevent arbitrary oracle replacement.

---

### Finding Description

At pool creation, `priceProviderTimelock` is stored verbatim with no lower-bound check:

```solidity
// MetricOmmPoolFactory.sol L213
priceProviderTimelock[pool] = params.priceProviderTimelock;
```

`_validatePoolParameters` only checks fees, tokens, admin, and initial amounts — `priceProviderTimelock` is never validated for a minimum:

```solidity
// MetricOmmPoolFactory.sol L548-563
function _validatePoolParameters(PoolParameters calldata params) internal view {
    // ... token, admin, fee checks ...
    // NO check: priceProviderTimelock >= MIN_TIMELOCK
}
```

When `priceProviderTimelock[pool] == 0`, the proposal step computes:

```solidity
// MetricOmmPoolFactory.sol L487
uint256 executeAfter = block.timestamp + timelock; // = block.timestamp + 0 = block.timestamp
```

The execution guard then evaluates:

```solidity
// MetricOmmPoolFactory.sol L499
if (block.timestamp < execAfter) revert PriceProviderTimelockNotElapsed(...);
// block.timestamp < block.timestamp  →  false  →  no revert
```

Because `nonReentrant` is per-call (transient storage is set and cleared within each function), a pool admin can deploy a contract that calls both `proposePoolPriceProvider` and `executePoolPriceProviderUpdate` atomically in a single transaction. The new oracle only needs to pass `_validatePriceProvider` (correct token pair), so a pre-deployed malicious oracle returning any bid/ask prices satisfies the check.

The AuditFindings.md M01 entry confirms the timelock was introduced specifically to prevent the admin from "arbitrarily changing the pool priceProvider at runtime," yet the fix never enforces a minimum timelock value, leaving the protection hollow when `priceProviderTimelock = 0`.

---

### Impact Explanation

After the atomic oracle switch, every subsequent swap uses the malicious oracle's prices. The pool admin (or a colluding address) can execute swaps at prices far from fair value, draining LP principal. This is a direct loss of user principal — the pool's token balances no longer cover LP claims — satisfying the Critical/High impact threshold under the Metric OMM allowed impact gate ("Admin-boundary break: pool admin bypasses timelocks").

The pool admin is documented as "semi-trusted only inside caps and timelocks." With `priceProviderTimelock = 0`, there is no timelock constraint, so the admin is effectively fully trusted with respect to oracle changes, violating the semi-trusted boundary the protocol relies on.

---

### Likelihood Explanation

`createPool` is permissionless. Any actor can deploy a pool with `priceProviderTimelock = 0`, attract LP deposits (e.g., by offering competitive fees and a legitimate oracle initially), and then execute the drain. The only prerequisite is a pre-deployed malicious oracle contract, which is trivial to prepare before pool creation. The attack requires no privileged access beyond being the pool admin of a pool the attacker themselves created.

---

### Recommendation

In `_validatePoolParameters`, add a minimum timelock check for mutable pools:

```solidity
uint256 internal constant MIN_PRICE_PROVIDER_TIMELOCK = 1 days;

// inside _validatePoolParameters:
if (
    params.priceProviderTimelock != type(uint256).max &&
    params.priceProviderTimelock < MIN_PRICE_PROVIDER_TIMELOCK
) {
    revert PriceProviderTimelockTooShort();
}
```

This mirrors the pattern used for fee caps (`HARD_MAX_SPREAD_FEE_E6`, `HARD_MAX_NOTIONAL_FEE_E8`) — hard limits that the factory enforces regardless of what the pool creator supplies — and ensures the pool admin always remains within the semi-trusted boundary.

---

### Proof of Concept

```solidity
// Attacker contract
contract AtomicOracleSwitch {
    MetricOmmPoolFactory factory;
    address pool;
    address maliciousOracle;

    function attack() external {
        // Step 1: propose malicious oracle (executeAfter = block.timestamp)
        factory.proposePoolPriceProvider(pool, maliciousOracle);
        // Step 2: execute immediately — block.timestamp < block.timestamp is false
        factory.executePoolPriceProviderUpdate(pool);
        // Step 3: swap against pool at manipulated prices
        IMetricOmmPool(pool).swap(...);
    }
}
```

1. Deploy `MaliciousOracle` returning `(token0, token1)` correctly but with extreme bid/ask prices.
2. Call `factory.createPool(params)` with `priceProviderTimelock = 0` and a legitimate oracle.
3. Wait for LPs to deposit.

```markdown
## M01 - Admin can arbitrarily change the pool priceProvider

### Finding summary

`setPriceProvider()` allowed admin to replace oracle at runtime, and `priceProvider` was returned by `getImmutables()`.

### Resolution

Fixed.

Changes:

- Added constructor config `priceProviderTimelock`.
- Added admin flow:
  - `proposePriceProvider(address newPriceProvider)`
  - `executePriceProviderUpdate()`
- Added events:
  - `PriceProviderChangeProposed(currentPriceProvider, newPriceProvider, executeAfter)`
  - `PriceProviderUpdated(newPriceProvider)`

Behavior now:

- `priceProviderTimelock == type(uint256).max` => immutable mode (admin updates disabled).
- `priceProviderTimelock < type(uint256).max` => mutable mode with delayed execution.
- Price provider used by swaps/reports resolves through timelock mode; active provider is read via `MetricOmmPoolStateView.priceProvider(pool)`.
```
