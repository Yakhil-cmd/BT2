### Title
Cross-COA Signature Replay in `validateCOAOwnershipProof` — (`File: fvm/evm/stdlib/contract.cdc`)

### Summary
`EVM.validateCOAOwnershipProof` verifies Cadence account key signatures over caller-supplied `signedData` but does not enforce that `signedData` encodes the target `evmAddress`. A valid COA ownership proof produced for one COA (COA\_A) owned by a Cadence account can be replayed verbatim — or with only the `CapabilityPath` field swapped — to pass ownership validation for a different COA (COA\_B) owned by the same Cadence account. The code itself acknowledges this gap in a comment but dismisses it as low-risk based on a non-enforced wallet convention.

### Finding Description
`validateCOAOwnershipProof` in `fvm/evm/stdlib/contract.cdc` performs two independent checks:

1. Verifies that the supplied `signatures` are valid over `signedData` using the Cadence account's keys.
2. Verifies that the COA resource at `path` has an EVM address matching the supplied `evmAddress`. [1](#0-0) 

These two checks are never bound together. The `signedData` is never required to include `evmAddress`. The code comment at line 1003 explicitly acknowledges this: [2](#0-1) 

Because `COAOwnershipProof` is a caller-supplied, RLP-encoded struct containing `KeyIndices`, `Address`, `CapabilityPath`, and `Signatures`, an attacker who has observed a valid proof for COA\_A can construct a new proof with:
- the same `Address` (same Cadence account),
- the same `KeyIndices` and `Signatures` (valid over the same `signedData`/hash),
- a different `CapabilityPath` pointing to COA\_B. [3](#0-2) 

The function will accept this replayed proof because the signatures are still valid over `signedData`, and the COA resource at the new path does have the expected `evmAddress`.

The EVM-side entry point is the ERC-1271 `isValidSignature` function in the COA Solidity contract, which passes `address(this)` as the `evmAddress` and the caller-supplied `_hash` as `signedData`: [4](#0-3) 

The ABI decoding in `DecodeABIEncodedProof` constructs the `COAOwnershipProofInContext` with the calling COA's address as `EVMAddress` and the hash as `SignedData`: [5](#0-4) 

The Cadence-side entry point is `EVM.validateCOAOwnershipProof`, which is `access(all)` and callable from any Cadence script or transaction: [6](#0-5) 

### Impact Explanation
An attacker who obtains a valid COA ownership proof for COA\_A (e.g., from a prior ERC-1271 interaction, an on-chain event, or a public off-chain authentication flow) can replay it against COA\_B owned by the same Cadence account. This forges a passing ERC-1271 `isValidSignature` result for COA\_B over any hash that was previously signed for COA\_A. Downstream EVM contracts that rely on ERC-1271 for authorization (token approvals, NFT transfers, multisig thresholds, permit-style flows) would incorrectly accept the forged proof as a valid signature from COA\_B's owner, enabling unauthorized asset operations on behalf of COA\_B.

### Likelihood Explanation
The preconditions are:
1. The same Cadence account owns two or more COAs stored at different public capability paths. This is not enforced against by the protocol — any Cadence transaction can call `EVM.createCadenceOwnedAccount()` multiple times.
2. A valid proof (signatures over some hash H) for one of those COAs has been exposed to the attacker (e.g., submitted on-chain, passed in a public EVM call, or shared in an off-chain authentication flow).

Both conditions are realistic in production. The comment's dismissal ("Flow wallets historically create at most one COA per account") is a social convention, not a protocol invariant. Any smart-contract wallet, bridge, or dApp that programmatically creates multiple COAs per account is immediately vulnerable.

### Recommendation
Enforce that `signedData` commits to the target `evmAddress`. The simplest fix is to require callers to include the `evmAddress` bytes inside `signedData` and verify this binding inside `validateCOAOwnershipProof` before accepting the proof. Alternatively, the function itself can prepend or hash `evmAddress` into the message that is verified, analogous to how EIP-712 domain separators bind signatures to a specific contract address.

### Proof of Concept
1. Cadence account `0xALICE` creates two COAs: COA\_A stored at `/public/coaA` and COA\_B stored at `/public/coaB`.
2. A legitimate dApp asks Alice to prove ownership of COA\_A by signing hash `H`. Alice produces proof `P_A = {Address: 0xALICE, CapabilityPath: "coaA", KeyIndices: [0], Signatures: [sig_over_H]}`.
3. Attacker constructs `P_B = {Address: 0xALICE, CapabilityPath: "coaB", KeyIndices: [0], Signatures: [sig_over_H]}` — identical except for `CapabilityPath`.
4. Attacker calls `COA_B.isValidSignature(H, encode(P_B))` from an EVM contract.
5. The precompile calls `validateCOAOwnershipProof(address=0xALICE, path=/public/coaB, signedData=H, keyIndices=[0], signatures=[sig_over_H], evmAddress=COA_B_addr)`.
6. `keyList.verify(signatureSet, signedData=H, ...)` returns `true` — `sig_over_H` is valid over `H`.
7. The COA resource at `/public/coaB` has address `COA_B_addr` — the address check passes.
8. `validateCOAOwnershipProof` returns `ValidationResult(isValid: true)`, and `isValidSignature` returns `0x1626ba7e` (valid).
9. Any EVM contract checking ERC-1271 on COA\_B now accepts the forged proof. [7](#0-6) [8](#0-7)

func DecodeABIEncodedProof(input []byte) (*types.COAOwnershipProofInContext, error) {
	index := 0
	caller, err := ReadAddress(input, index)
	index += FixedSizeUnitDataReadSize
	if err != nil {
		return nil, err
	}

	hash, err := ReadBytes32(input, index)
	index += Bytes32DataReadSize
	if err != nil {
		return nil, err
	}

	encodedProof, err := ReadBytes(input, index)
	if err != nil {
		return nil, err
	}

	return types.NewCOAOwnershipProofInContext(
		hash,
		types.Address(caller),
		encodedProof,
	)
}

					return contractAddress
				},
				LocationName: "EVM",
				FunctionName: "validateCOAOwnershipProof",
				ArgumentTypes: []sema.Type{
					types.FlowAddressSemaType,
					types.PublicPathSemaType,
					types.SignedDataSemaType,
					types.KeyIndicesSemaType,
					types.SignaturesSemaType,
					types.AddressBytesSemaType,
				},
			},
			proof.ToCadenceValues(),
		)
		if err != nil {
			if types.IsAFatalError(err) {
				panic(err)
			}
			return false, err
		}
		data, ok := value.(cadence.Struct)
		if !ok {
			return false, fmt.Errorf("invalid output data received from validateCOAOwnershipProof")

		if isValidValue == nil {
			return false, fmt.Errorf("invalid output data received from validateCOAOwnershipProof")
		}

		return bool(isValidValue.(cadence.Bool)), nil
	}
