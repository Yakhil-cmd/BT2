Audit Report

## Title
Global Nonce Cap in `withdrawal_requests_batch` Enables Forced Halt of All ckETH/ckERC20 Withdrawal Processing - (File: `rs/ethereum/cketh/minter/src/state/transactions/mod.rs`)

## Summary

`withdrawal_requests_batch` enforces a hard ceiling of `MAX_NUM_PENDING_TRANSACTION_NONCES = 1000` across `created_tx` and `sent_tx`. Once that ceiling is reached the function returns an empty batch, and the minter's timer loop stops converting any pending request into an Ethereum transaction. An attacker who can afford the capital cost of ~1,000 minimum-withdrawal-amount ckETH (recovered as ETH at destination) and the gas cost of ~1,000 Ethereum transactions can hold the entire withdrawal pipeline frozen for as long as those transactions remain unconfirmed. The claim's assertion that `pending_withdrawal_requests` is unbounded is incorrect — `retrieve_withdraw_guard` enforces `MAX_PENDING = 100` — but this partial mitigation does not prevent the attack; it only requires the attacker to submit requests in batches as the queue drains.

## Finding Description

**Root cause — the 1,000-nonce ceiling:**

`withdrawal_requests_batch` in `rs/ethereum/cketh/minter/src/state/transactions/mod.rs` (L906–923) computes:

```rust
const MAX_NUM_PENDING_TRANSACTION_NONCES: usize = 1000;
let unique_pending_transaction_nonces: BTreeSet<_> =
    self.created_tx.keys().chain(self.sent_tx.keys()).collect();
let actual_batch_size = min(
    MAX_NUM_PENDING_TRANSACTION_NONCES
        .saturating_sub(unique_pending_transaction_nonces.len()),
    requested_batch_size,
);
```

When `created_tx ∪ sent_tx` reaches 1,000 unique nonces, `saturating_sub` yields 0, `actual_batch_size = 0`, and the function returns an empty `Vec`. The caller `create_transactions_batch` in `rs/ethereum/cketh/minter/src/withdraw.rs` (L249–293) iterates over this empty result and does nothing, so no new Ethereum transactions are ever created until the nonce maps drain.

**Partial mitigation that does NOT prevent the attack:**

The claim incorrectly states the queue is unbounded. `retrieve_withdraw_guard` in `rs/ethereum/cketh/minter/src/guard/mod.rs` (L50–68) checks `pending_requests_count(s) >= MAX_PENDING` (where `MAX_PENDING = 100`) before admitting a new request, and `pending_requests_count` returns `withdrawal_requests_len()` (L32–34), i.e., the live length of `pending_withdrawal_requests`. This caps the instantaneous pending queue at 100 entries.

However, this guard only limits `pending_withdrawal_requests`. It places no cap on `created_tx` or `sent_tx`. Once a request is promoted to `created_tx` by the timer, it leaves `pending_withdrawal_requests` and the guard count drops, allowing the attacker to submit another request. With `WITHDRAWAL_REQUESTS_BATCH_SIZE = 5` (L39 of `withdraw.rs`), the attacker can refill the queue 5 slots at a time on every timer tick, steadily accumulating nonces in `created_tx`/`sent_tx` until the 1,000-nonce ceiling is hit.

**Exploit flow:**

1. Attacker submits 100 withdrawal requests (fills `pending_withdrawal_requests` to `MAX_PENDING`).
2. Timer processes 5 per tick → 5 requests move to `created_tx`; queue drops to 95.
3. Attacker submits 5 more → queue back to 100.
4. Attacker sets low `max_priority_fee_per_gas` so signed transactions sit unconfirmed in the Ethereum mempool, keeping them in `sent_tx`.
5. After ~200 timer ticks, `created_tx ∪ sent_tx` reaches 1,000 unique nonces.
6. `withdrawal_requests_batch` now returns `[]` on every tick: `min(1000 − 1000, 5) = 0`.
7. Every legitimate user's request in `pending_withdrawal_requests` is permanently stalled until the attacker's transactions finalize on-chain.

**Why existing checks are insufficient:**

- `retrieve_withdraw_guard` caps `pending_withdrawal_requests` at 100 but has no visibility into `created_tx`/`sent_tx` nonce counts.
- `withdrawal_requests_batch` has no per-principal fairness or nonce-budget accounting; a single actor can monopolise the entire 1,000-nonce budget.
- There is no mechanism to evict or reprice stuck transactions to accelerate finalization.

## Impact Explanation

Once the 1,000-nonce ceiling is reached, `create_transactions_batch` processes zero requests on every timer tick. All legitimate users whose withdrawal requests are in `pending_withdrawal_requests` remain stuck indefinitely. No new Ethereum transactions are created for any user until the attacker's 1,000 transactions finalize. This is a global, protocol-level denial of service on all ckETH and ckERC20 withdrawals — a concrete application/platform-level DoS on an in-scope Chain Fusion financial integration with significant user and protocol harm. This matches the **High ($2,000–$10,000)** impact class: "Application/platform-level DoS … or subnet availability impact not based on raw volumetric DDoS" and "Significant Chain Fusion, ck-token … security impact with concrete user or protocol harm."

## Likelihood Explanation

The attacker's ckETH principal is not lost — it is converted to ETH at the destination address — so the net cost is only Ethereum gas fees (~1,000 × gas price × 21,000 gas for ETH transfers, or ~65,000 gas for ERC-20). During low-fee periods this is economically feasible for a motivated attacker. The attack requires no privileged role, governance vote, or subnet-majority corruption. The `MAX_PENDING = 100` guard slows the attack (requiring ~200 timer ticks of patient batch submission) but does not prevent it. During Ethereum congestion the attacker can keep resubmitting with low gas to delay finalization, extending the freeze window at no additional capital cost.

## Recommendation

**Short term:** Inside `withdrawal_requests_batch`, or in the timer logic that calls it, enforce a per-principal nonce budget so that a single actor cannot monopolise the 1,000-nonce pool. Alternatively, add a check in `record_withdrawal_request` (or in the guard) that counts the caller's existing nonces already in `created_tx ∪ sent_tx` and rejects new requests once a per-principal cap is reached.

**Long term:** Decouple the global nonce cap from per-user fairness. Implement a priority or age-based promotion policy so long-queued legitimate requests are not indefinitely starved. Consider a dynamic repricing mechanism that automatically bumps gas on stuck transactions to accelerate finalization and drain the nonce maps.

## Proof of Concept

Deterministic integration test plan (safe, local replica / PocketIC):

1. Initialize the ckETH minter with a known state.
2. Create ~20 distinct principals, each holding `cketh_minimum_withdrawal_amount × 50` ckETH.
3. In a loop: each principal calls `withdraw_eth` with low `max_priority_fee_per_gas` until `pending_withdrawal_requests` hits 100; advance the timer to process 5 requests per tick; repeat until `created_tx.keys().chain(sent_tx.keys()).count() == 1000`.
4. Assert that `withdrawal_requests_batch(5)` returns an empty `Vec`.
5. Submit a withdrawal request from a fresh legitimate principal; assert it is accepted into `pending_withdrawal_requests` (queue < 100) but never promoted to `created_tx` across N subsequent timer ticks.
6. Confirm the freeze lifts only after simulating finalization of the attacker's 1,000 transactions (draining `sent_tx`).
