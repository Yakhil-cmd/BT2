### Title
Missing Consistency Validation of Participant Shares in CKD Coordinator Allows Malicious Participant to Corrupt Confidential Key Derivation Output — (`File: src/confidential_key_derivation/protocol.rs`)

---

### Summary

The `do_ckd_coordinator` function accumulates `(big_y, big_c)` group-element contributions from all participants without any consistency check against their public key shares. A single malicious participant can send arbitrary BLS12-381 G1 elements, causing the coordinator to output a corrupted `CKDOutput` that does not correspond to the correct confidential key.

---

### Finding Description

In `do_ckd_coordinator`, the coordinator receives `(big_y, big_c)` from each participant via `recv_from_others` and blindly sums them: [1](#0-0) 

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
```

There is no validation that the received `big_y` and `big_c` values are consistent with the participant's public key share. Each honest participant is supposed to compute: [2](#0-1) 

- `big_y = lambda_i * y_i * G` (a random blinding factor)
- `big_c = lambda_i * (x_i * H(pk, app_id) + y_i * app_pk)` (the masked contribution)

But the coordinator has no mechanism to verify these values. A malicious participant can send arbitrary group elements for `big_y` and `big_c`, causing the coordinator to compute an incorrect CKD output.

This is directly analogous to the external report: just as `pool.burn` returns `amount0`/`amount1` that are used without slippage checks, `recv_from_others` returns participant shares that are used without consistency checks.

By contrast, the robust ECDSA presign protocol performs explicit exponent interpolation checks to verify that received shares are consistent with a polynomial of the expected degree: [3](#0-2) 

The CKD protocol has no analogous validation. The `ckd` entry point also accepts no threshold parameter, meaning all participants must contribute and a single malicious one can corrupt the result: [4](#0-3) 

---

### Impact Explanation

A malicious participant sends `big_y = Δ_Y` and `big_c = Δ_C` (arbitrary G1 elements). The coordinator computes:

```
Y_out = correct_Y + Δ_Y
C_out = correct_C + Δ_C
```

When the application calls `unmask(app_sk)`, it computes `C_out − app_sk · Y_out`, which equals:

```
correct_confidential_key + Δ_C − app_sk · Δ_Y
```

This is an incorrect confidential key. Honest parties accept and use this corrupted `CKDOutput` with no indication of tampering. This matches the allowed impact: **High — Corruption of CKD outputs so honest parties accept unusable or incorrect cryptographic outputs.** [5](#0-4) 

---

### Likelihood Explanation

- Any single non-coordinator participant can mount this attack with no special knowledge beyond their own share.
- The coordinator has no cryptographic mechanism to detect the manipulation.
- The `ckd` protocol has no threshold parameter; all participants must contribute, so even one malicious party is sufficient.
- The attack is trivially executable: the malicious participant simply deviates from the protocol by sending arbitrary G1 elements. [6](#0-5) 

---

### Recommendation

Require each participant to accompany their `(big_y, big_c)` contribution with a non-interactive zero-knowledge proof of knowledge of `y_i` satisfying:

```
big_y = lambda_i * y_i * G
big_c = lambda_i * (x_i * H(pk, app_id) + y_i * app_pk)
```

This is a standard Sigma protocol (Chaum-Pedersen style) over G1 that allows the coordinator to verify each share without learning `y_i`. Alternatively, a commitment-reveal scheme can be used: participants commit to `y_i` in a first round, then reveal it alongside the contribution in a second round, allowing the coordinator to recompute and verify `big_y` and `big_c` directly.

---

### Proof of Concept

1. Honest participants run `compute_signature_share` and produce correct `(big_y_i, big_c_i)`.
2. Malicious participant `j` instead sends `(big_y_j = P, big_c_j = Q)` for arbitrary G1 points `P`, `Q`.
3. Coordinator sums all contributions: `Y = Σ big_y_i + P`, `C = Σ big_c_i + Q`.
4. Coordinator returns `CKDOutput::new(Y, C)` — no error is raised.
5. Application calls `ckd_output.unmask(app_sk)` and obtains `C − app_sk · Y = msk · H(pk, app_id) + Q − app_sk · P`, which is an incorrect confidential key.
6. Honest parties have no way to detect the corruption. [7](#0-6)

async fn do_ckd_coordinator(
    mut chan: SharedChannel,
    participants: ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<CKDOutputOption, ProtocolError> {
    let (mut norm_big_y, mut norm_big_c) =
        compute_signature_share(&participants, me, key_pair, app_id, app_pk, rng)?;

    // Receive everyone's inputs and add them together
    let waitpoint = chan.next_waitpoint();

    for (_, participant_output) in
        recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
    {
        norm_big_y += participant_output.big_y();
        norm_big_c += participant_output.big_c();
    }
    let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
    Ok(Some(ckd_output))
}
