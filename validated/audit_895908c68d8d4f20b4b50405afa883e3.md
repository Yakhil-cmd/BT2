### Title
Signature Bypass via Incomplete Deploy-Account Presence Check in `skip_stateful_validations` — (File: crates/apollo_gateway/src/stateful_transaction_validator.rs)

### Summary

The `skip_stateful_validations` function decides whether to skip the `__validate__` entry-point (signature verification) for a nonce=1 invoke transaction by calling `account_tx_in_pool_or_recent_block(sender_address)`. This check is an incomplete identifier — it returns `true` if **any** transaction from that sender address is known to the mempool, not specifically a `deploy_account` transaction. An unprivileged attacker who observes a legitimate pending `deploy_account` for address A in the mempool can submit a nonce=1 invoke for address A with an arbitrary invalid signature, which the gateway admits without ever calling `__validate__`.

### Finding Description

The gateway's stateful validation path in `extract_state_nonce_and_run_validations` calls `run_pre_validation_checks`, which in turn calls `skip_stateful_validations`:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            // ...
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())  // ← incomplete check
                .await
                ...
        }
    }
    Ok(false)
}
``` [1](#0-0) 

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` sets `validate: false`, so `StatefulValidator::perform_validations` never calls `__validate__` (the account's signature-checking entry point):

```rust
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [2](#0-1) 

The mempool's `account_tx_in_pool_or_recent_block` checks only whether the sender address appears in the pool or state — it does **not** check the transaction type:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [3](#0-2) 

The code comment claims this is sufficient because "it means that either it has a deploy_account transaction or transactions with future nonces that passed validations." This reasoning is correct for the **legitimate** user's own transactions, but it ignores the attacker's ability to exploit the presence of the legitimate deploy_account.

**Attack chain:**

1. Legitimate user submits `deploy_account` for address A. The gateway fully validates it (runs `__validate_deploy__` via `self.execute(tx)`) and it enters the mempool.
2. Attacker observes the pending `deploy_account` for address A (mempool is public).
3. Attacker crafts a nonce=1 invoke with `sender_address = A`, arbitrary calldata, and an **invalid/attacker-chosen signature**.
4. Gateway stateless validation passes (no signature check there).
5. `extract_state_nonce_and_run_validations`:
   - `account_nonce` = 0 (account not yet deployed on-chain).
   - `validate_state_preconditions`: nonce=1 ≥ 0 and ≤ 0+`max_allowed_nonce_gap` → passes.
   - `validate_by_mempool`: no duplicate tx_hash, no duplicate nonce=1 for address A yet → passes.
   - `skip_stateful_validations`: `tx.nonce()==1`, `account_nonce==0`, `account_tx_in_pool_or_recent_block(A)` returns **true** (the legitimate deploy_account is in the pool) → returns `true`.
   - `run_validate_entry_point` with `skip_validate=true`: `validate=false`, `__validate__` is **never called**.
6. The invalid invoke is admitted to the mempool.
7. Block execution: `deploy_account` runs first, account A is deployed.
8. Nonce=1 invoke runs: `__validate__` is now called with the real account code, fails (invalid signature), transaction **reverts**.
9. Fees are charged from address A's balance.

**Secondary impact — front-running the legitimate nonce=1 invoke:** If the attacker submits their invalid nonce=1 invoke before the legitimate user submits theirs, the mempool's `validate_fee_escalation` / `DuplicateNonce` check will reject the legitimate user's nonce=1 invoke (unless fee escalation is enabled and the legitimate user pays a higher fee). This permanently blocks the legitimate user's first post-deploy transaction until the attacker's reverted invoke is committed and the nonce advances. [4](#0-3) 

### Impact Explanation

This is a **High** impact finding matching: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

- The gateway admits a nonce=1 invoke with an invalid signature (an invalid transaction) into the mempool without calling the account's `__validate__` entry point.
- The invalid transaction is included in a block, reverts during execution, and fees are charged from the victim account.
- The attacker can also front-run the legitimate user's nonce=1 invoke, causing it to be rejected by the mempool (`DuplicateNonce`), permanently blocking the legitimate user's first post-deploy transaction.

### Likelihood Explanation

**Medium.** The attack requires:
1. Observing a pending `deploy_account` in the mempool — trivially achievable since the mempool is public.
2. Submitting a nonce=1 invoke before the legitimate user does — a simple race condition.

No privileged access, no special knowledge, and no cryptographic capability is required. Any unprivileged actor can execute this against any newly deploying account.

### Recommendation

Replace the incomplete `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a **`deploy_account` transaction** is pending for the sender address. This requires adding a new mempool API, e.g.:

```rust
// New mempool API
async fn deploy_account_in_pool(sender_address: ContractAddress) -> MempoolClientResult<bool>;
```

And updating `skip_stateful_validations` to use it:

```rust
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
    return mempool_client
        .deploy_account_in_pool(tx.sender_address())  // type-specific check
        .await
        ...
}
```

This mirrors the fix in the external report: use the full identifier (origin = protocol + hostname + port) rather than the partial one (hostname only). Here, the full identifier is `(sender_address, transaction_type = deploy_account)` rather than just `sender_address`.

/// Perform transaction validation by the mempool.
async fn validate_by_mempool(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<()> {
    let validation_args = ValidationArgs::new(tx, account_nonce);
    mempool_client
        .validate_tx(validation_args)
        .await
        .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
}

/// Check if validation of an invoke transaction should be skipped due to deploy_account not being
/// processed yet. This feature is used to improve UX for users sending deploy_account + invoke at
/// once.
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
