### Title
Missing Caller Restriction on `update_rewards` Allows Any Address to Freeze Consensus Rewards - (File: src/staking/staking.cairo)

### Summary
The `update_rewards` function in `StakingRewardsManagerImpl` is specified to be callable only by the Starkware sequencer, but the implementation contains no caller check. Any unprivileged address can call it with `disable_rewards: true` every block, permanently preventing stakers from receiving consensus rewards.

### Finding Description
The spec at `docs/spec.md:1644-1645` explicitly states:

```
#### access control
Only starkware sequencer.
```

However, the implementation in `src/staking/staking.cairo` at `StakingRewardsManagerImpl::update_rewards` performs no caller validation whatsoever:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
    let current_block_number = starknet::get_block_number();
    assert!(
        current_block_number > self.last_reward_block.read(),
        "{}",
        Error::REWARDS_ALREADY_UPDATED,
    );
    // ... no assert_caller_is_sequencer() ...
    self.last_reward_block.write(current_block_number);  // written BEFORE disable_rewards check
    if disable_rewards || self.is_pre_consensus() {
        return;
    }
    // distribute rewards ...
}
``` [1](#0-0) 

The critical detail is that `last_reward_block` is a **global** storage variable written unconditionally at line 1485, before the `disable_rewards` guard. Once written, the check `current_block_number > self.last_reward_block.read()` prevents any further call in the same block, including the legitimate sequencer's call. [2](#0-1) 

The spec confirms `last_reward_block` is global (one value for the entire contract, not per-staker), so a single attacker call per block poisons the slot for all stakers. [3](#0-2) 

### Impact Explanation
**High — Permanent freezing of unclaimed yield.**

An attacker calling `update_rewards(any_valid_staker, disable_rewards: true)` every block:
1. Sets `last_reward_block` to the current block number without distributing any rewards.
2. Causes every subsequent sequencer call in the same block to revert with `REWARDS_ALREADY_UPDATED`.
3. Stakers accumulate zero consensus rewards indefinitely.

Because the attack requires only one cheap transaction per block and is permissionless, it can be sustained indefinitely, constituting a permanent freeze of all consensus-era unclaimed yield.

### Likelihood Explanation
**High.** The entry point is a public, unguarded external function. No special role, leaked key, or privileged access is required. Any EOA or contract can call it. The attacker only needs to submit a transaction before the sequencer's `update_rewards` call each block, which is trivially achievable by monitoring the mempool or simply calling it at the start of every block.

### Recommendation
Add a sequencer-only caller check at the top of `update_rewards`, analogous to the existing `assert_caller_is_attestation_contract` pattern used in `update_rewards_from_attestation_contract`:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
    self.assert_caller_is_sequencer(); // add this
    ...
}
``` [4](#0-3) 

### Proof of Concept
1. Consensus rewards are active (`is_pre_consensus()` returns `false`).
2. Attacker monitors for a new block.
3. Attacker calls `staking.update_rewards(any_active_staker, disable_rewards: true)`.
4. `last_reward_block` is set to the current block; no rewards are distributed.
5. The legitimate sequencer calls `staking.update_rewards(target_staker, disable_rewards: false)` — it reverts with `REWARDS_ALREADY_UPDATED`.
6. Repeat every block: all stakers receive zero consensus rewards permanently. [5](#0-4) [6](#0-5)
