### Title
`protocolUnpausePool` Hardcodes Target Pause Level to 1 (Admin-Paused) Instead of 0 (Active), Leaving Pool Permanently Paused After Protocol Unpause — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`protocolUnpausePool` calls `setPause(1)` (admin-paused) instead of `setPause(0)` (active). Because `_checkNotPaused` in `MetricOmmPool` reverts on any non-zero pause level, the pool remains fully paused after the protocol owner "unpauses" it. All swap, add-liquidity, and remove-liquidity flows stay blocked until the pool admin separately calls `unpausePool`.

---

### Finding Description

The pause state machine has three levels:

| Level | Meaning |
|---|---|
| 0 | Active — all operations allowed |
| 1 | Admin-paused — all operations blocked |
| 2 | Protocol-paused — all operations blocked |

`_checkNotPaused` in `MetricOmmPool.sol` enforces this:

```solidity
function _checkNotPaused() internal view {
    if (pauseLevel != 0) revert PoolPaused();   // both 1 and 2 are paused
}
``` [1](#0-0) 

`protocolPausePool` (owner-only) accepts pools in either state 0 or state 1 as valid starting points:

```solidity
function protocolPausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 0 && cur != 1) revert InvalidPauseTransition(cur, 2);
    IMetricOmmPoolFactoryActions(pool).setPause(2);
}
``` [2](#0-1) 

`protocolUnpausePool` then always transitions to level **1** (admin-paused), never to level **0** (active):

```solidity
function protocolUnpausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 2) revert InvalidPauseTransition(cur, 1);
    IMetricOmmPoolFactoryActions(pool).setPause(1);   // ← wrong: should be 0
}
``` [3](#0-2) 

The admin-side `unpausePool` is the only path from level 1 → 0:

```solidity
function unpausePool(address pool) external override nonReentrant onlyPoolAdmin(pool) {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 1) revert InvalidPauseTransition(cur, 0);
    IMetricOmmPoolFactoryActions(pool).setPause(0);
}
``` [4](#0-3) 

**Broken invariant**: When the protocol pauses an *active* pool (`0 → 2`) and then unpauses it (`2 → 1`), the pool is left in admin-paused state. The pool admin never issued a pause, yet they are now required to call `unpausePool` to restore normal operation. Until they do, every call to `swap`, `addLiquidity`, and `removeLiquidity` reverts with `PoolPaused`.

This is structurally identical to the M-01 seed bug: a hardcoded wrong value (`1` instead of `0`) makes the intended code path (full activation) permanently unreachable through the protocol-unpause route.

---

### Impact Explanation

All three core pool operations — `swap`, `addLiquidity`, `removeLiquidity` — are gated by `whenNotPaused` / `_checkNotPaused`. [5](#0-4) 

After `protocolUnpausePool`, `pauseLevel == 1 != 0`, so every user-facing call reverts. LPs cannot withdraw principal, traders cannot swap, and fee collection is blocked. This satisfies the "unusable withdraw/swap/liquidity flows" criterion in the allowed impact gate.

---

### Likelihood Explanation

The trigger is a normal, non-malicious protocol-owner action: pause an active pool for an emergency, then unpause it. No privileged bypass, no malicious setup, and no non-standard token is required. The pool admin may not be monitoring on-chain state and may not realize they must issue a second transaction to restore the pool.

---

### Recommendation

Change `protocolUnpausePool` to target level 0 (active):

```diff
 function protocolUnpausePool(address pool) external override nonReentrant onlyOwner {
     (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
-    if (cur != 2) revert InvalidPauseTransition(cur, 1);
-    IMetricOmmPoolFactoryActions(pool).setPause(1);
+    if (cur != 2) revert InvalidPauseTransition(cur, 0);
+    IMetricOmmPoolFactoryActions(pool).setPause(0);
 }
```

If the design intent is to preserve a pre-existing admin pause (i.e., the protocol paused a pool that was already at level 1), the factory should record the pre-pause level when `protocolPausePool` is called and restore it on `protocolUnpausePool`. The current implementation does not store this, so it cannot distinguish the two cases and incorrectly defaults to level 1 in all scenarios.

---

### Proof of Concept

1. Pool is deployed and active (`pauseLevel == 0`).
2. Protocol owner calls `protocolPausePool(pool)` → `pauseLevel` transitions `0 → 2`. All operations revert.
3. Protocol owner calls `protocolUnpausePool(pool)` → `setPause(1)` is called; `pauseLevel` transitions `2 → 1`.
4. Any user calls `pool.swap(...)` → `_checkNotPaused()` evaluates `pauseLevel != 0` → `true` → reverts with `PoolPaused`.
5. Pool admin must separately call `unpausePool(pool)` to reach `pauseLevel == 0`. Until then, the pool is completely non-functional despite the protocol owner having "unpaused" it.
