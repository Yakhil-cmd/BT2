Audit Report

## Title
`AmountTooLow` branch permanently burns ckBTC without reimbursement when UTXO pool is fragmented — (`rs/bitcoin/ckbtc/minter/src/lib.rs`, `rs/bitcoin/ckbtc/minter/src/fees/mod.rs`)

## Summary

`fee_based_minimum_withdrawal_amount` computes the minimum safe withdrawal amount using a hardcoded single-input vsize of 221 vbytes. When the minter's UTXO pool is fragmented into many small UTXOs, the greedy selection algorithm picks far more inputs, making the real transaction fee exceed the withdrawal amount. The resulting `BuildTxError::AmountTooLow` path in `submit_pending_requests` finalizes the request with no reimbursement — unlike the `InvalidTransaction` path which does call `reimburse_canceled_requests` — permanently burning the user's ckBTC with no BTC sent and no refund.

## Finding Description

**Root cause — single-input vsize assumption:**
`fee_based_minimum_withdrawal_amount` uses `PER_REQUEST_VSIZE_BOUND = 221` vbytes, modeling exactly one P2WPKH input plus two outputs. [1](#0-0) 

**ckBTC burned before transaction is built:**
In `retrieve_btc`, `burn_ckbtcs` is called at line 210 before any transaction is attempted. The burn is irreversible. [2](#0-1) 

**Greedy UTXO selection accumulates many inputs:**
`greedy` iterates until the target value is met, with no input-count limit inside the function itself. With N small UTXOs each worth slightly less than the withdrawal amount, all N are selected. [3](#0-2) 

**`TooManyInputs` guard does not protect this path:**
`build_unsigned_transaction_from_inputs` checks `num_inputs > max_num_inputs_in_transaction` first (line 1251), returning `InvalidTransaction(TooManyInputs)` which **does** trigger reimbursement. However, for a moderate number of inputs (e.g., 100), this threshold is not reached, so execution falls through to the `AmountTooLow` check. [4](#0-3) 

**Real vsize used for fee, exceeds amount:**
`evaluate_transaction_fee` calls `fake_sign(tx).vsize()` — the actual vsize with all N inputs. With 100 inputs at 343 sat/vbyte, the fee (~2.37M sats) far exceeds a 200,000-sat withdrawal. [5](#0-4) [6](#0-5) 

**`AmountTooLow` branch: no reimbursement scheduled:**
The `AmountTooLow` branch calls only `remove_retrieve_btc_request` with `FinalizedStatus::AmountTooLow`. No call to `reimburse_canceled_requests`. [7](#0-6) 

**Contrast with `InvalidTransaction` branch which does reimburse:** [8](#0-7) 

**`remove_retrieve_btc_request` schedules no reimbursement:**
It only records an event and pushes to `finalized_requests`. No reimbursement task is created. [9](#0-8) 

**`FinalizedStatus::AmountTooLow` is a dead-end terminal state** with no reimbursement variant. [10](#0-9) 

## Impact Explanation

A user who submits a withdrawal at exactly `fee_based_retrieve_btc_min_amount` loses their ckBTC permanently when the minter's UTXO pool is fragmented: the ckBTC ledger burn is irreversible, no BTC transaction is sent, and the `AmountTooLow` terminal state carries no reimbursement path. This is a concrete, unrecoverable user-fund loss in ckBTC, a Chain Fusion financial integration. This matches: **High ($2,000–$10,000) — Significant ck-token/ledger security impact with concrete user or protocol harm.**

## Likelihood Explanation

Any unprivileged user can fragment the minter's UTXO pool by depositing many small UTXOs (each ≥ `deposit_btc_min_amount`). The consolidation mechanism only triggers above `UTXOS_COUNT_THRESHOLD` (typically 1,000+), so ~100 small UTXOs are sufficient to trigger the bug without activating consolidation. The attack cost (depositing ~200,100 sats + on-chain fees) is comparable to the victim's loss, making this a griefing vector. Natural UTXO fragmentation (without a deliberate attacker) can also trigger the same outcome for any user who withdraws at the minimum amount during a fragmented state.

## Recommendation

1. **Reimbursement parity (minimal fix):** The `AmountTooLow` branch in `submit_pending_requests` should call `reimburse_canceled_requests` (minus a small processing fee), matching the behavior of the `InvalidTransaction` branch. The `reimbursement_account` is already set on the request.
2. **Multi-input-aware minimum:** `fee_based_minimum_withdrawal_amount` should account for the worst-case number of inputs the greedy algorithm might select for the given amount, or use a conservative upper bound on input count.
3. **Lower consolidation threshold:** Reduce `UTXOS_COUNT_THRESHOLD` so fragmentation is addressed before it can affect minimum-amount withdrawals.

## Proof of Concept

```rust
// Construct a fragmented UTXO pool: 100 UTXOs of 2_001 sats each
let mut utxos: UtxoSet = (0..100u32)
    .map(|i| Utxo {
        outpoint: OutPoint { txid: [i as u8; 32].into(), vout: 0 },
        value: 2_001,
        height: 10,
    })
    .collect();

let fee_rate = FeeRate::from_millis_per_byte(343_000);
let fee_estimator = BitcoinFeeEstimator::new(Network::Mainnet, 100_000, 2_000);

// Minimum amount at this fee rate = 200_000 sats (single-input assumption)
let min_amount = fee_estimator.fee_based_minimum_withdrawal_amount(fee_rate);
assert_eq!(min_amount, 200_000);

// Build transaction at exactly the minimum — should succeed per the invariant
let result = build_unsigned_transaction(
    &mut utxos,
    vec![(BitcoinAddress::P2wpkhV0([1; 20]), min_amount)],
    &BitcoinAddress::P2wpkhV0([0; 20]),
    DEFAULT_MAX_NUM_INPUTS_IN_TRANSACTION,
    fee_rate,
    &fee_estimator,
);

// FAILS: returns Err(BuildTxError::AmountTooLow)
// 100 inputs × ~68 vbytes = ~6905 vbytes → fee ≈ 2.37M sats >> 200_000 sats
// In production: ckBTC already burned at this point, no reimbursement scheduled
assert!(result.is_ok(), "invariant violated: {:?}", result);
```

fn greedy(target: u64, available_utxos: &mut UtxoSet) -> Vec<Utxo> {
    #[cfg(feature = "canbench-rs")]
    let _scope = canbench_rs::bench_scope("greedy");

    let mut solution = vec![];
    let mut goal = target;
    while goal > 0 {
        let candidate_utxo = available_utxos
            .find_lower_bound(goal)
            .or_else(|| available_utxos.last())
            .cloned();
        match candidate_utxo {
            Some(utxo) => {
                let utxo = available_utxos.remove(&utxo).expect("BUG: missing UTXO");
                goal = goal.saturating_sub(utxo.value);
                solution.push(utxo);
            }
            None => {
                // Not enough available UTXOs to satisfy the request.
                for u in solution {
                    available_utxos.insert(u);
                }
                return vec![];
            }
        }
    }

    debug_assert!(solution.is_empty() || solution.iter().map(|u| u.value).sum::<u64>() >= target);

    solution

pub fn remove_retrieve_btc_request<R: CanisterRuntime>(
    state: &mut CkBtcMinterState,
    request: RetrieveBtcRequest,
    status: FinalizedStatus,
    runtime: &R,
) {
    record_event(
        EventType::RemovedRetrieveBtcRequest {
            block_index: request.block_index,
        },
        runtime,
    );

    state.push_finalized_request(FinalizedBtcRequest {
        request: request.into(),
        state: status,
    });
}
