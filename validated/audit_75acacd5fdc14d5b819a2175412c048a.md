### Title
Duplicate `input_data_ids` in ActionReceipt causes `pending_data_count` overcount, permanently blocking postponed receipt execution — (`runtime/runtime/src/lib.rs`)

### Summary
`process_action_receipt` iterates over `input_data_ids` without deduplication. Duplicate entries increment `pending_data_count` multiple times while writing to the same `PostponedReceiptId` trie key (idempotent overwrite). The DataReceipt processing path decrements the count exactly once per arriving DataReceipt. After the single DataReceipt arrives, `pending_data_count` remains permanently above zero with no remaining `PostponedReceiptId` entry to trigger further decrements, leaving the postponed receipt permanently unexecuted.

### Finding Description

In `process_action_receipt`, the loop over `action_receipt.input_data_ids()` increments `pending_data_count` for every occurrence of a missing `data_id`, including duplicates: [1](#0-0) 

The trie write `TrieKey::PostponedReceiptId { data_id: *data_id }` uses `data_id` as the key. For two identical `data_id` values, the second write silently overwrites the first with the same value — only one trie entry exists. But `pending_data_count` has been incremented twice.

When the DataReceipt for `X` arrives, the runtime:
1. Looks up `PostponedReceiptId[X]` → finds `receipt_id`
2. Removes `PostponedReceiptId[X]`
3. Reads `pending_data_count` (= 2), decrements to 1
4. Since count ≠ 1 at the check, does **not** execute the receipt [2](#0-1) 

`pending_data_count` is now 1 with no `PostponedReceiptId[X]` entry remaining. The postponed receipt and its `PendingDataCount` entry remain in trie state indefinitely.

The receipt validation in `verifier.rs` enforces `input_data_ids.len() <= max_number_input_data_dependencies` but does not check for duplicate entries, so a receipt with `[X, X]` passes validation. [3](#0-2) 

### Impact Explanation

Any attached deposit on the postponed ActionReceipt is permanently locked in trie state. The callback is never executed, so any state changes, token transfers, or promise resolutions it was supposed to perform never occur. The `PostponedReceipt` and `PendingDataCount` trie entries are never cleaned up, consuming storage permanently. This is a permanent denial-of-execution and potential fund-locking vulnerability.

### Likelihood Explanation

A malicious contract can trigger this by calling `promise_and` with the same promise index twice (e.g., `env::promise_and(&[p, p])`). This is a supported API call that any deployed contract can make. No validator or operator privileges are required. The receipt passes all existing validation checks.

### Recommendation

In `process_action_receipt`, deduplicate `input_data_ids` before iterating, or track which `data_id` values have already been counted in the current loop:

```rust
let mut seen = HashSet::new();
for data_id in action_receipt.input_data_ids() {
    if seen.insert(*data_id) && !has_received_data(state_update, account_id, *data_id)? {
        pending_data_count += 1;
        set(state_update, TrieKey::PostponedReceiptId { ... }, receipt.receipt_id());
    }
}
```

Alternatively, add a uniqueness check in `validate_receipt` in `verifier.rs` to reject ActionReceipts with duplicate `input_data_ids` at the validation boundary.

### Proof of Concept

1. Deploy a contract on account `attacker.near` with a method that:
   - Creates a promise `p = promise_create("target.near", "method", ...)`
   - Calls `promise_and(&[p, p])` to create a combined promise with duplicate dependency
   - Attaches a callback via `promise_then` on the combined promise
2. Call the method. The runtime creates an ActionReceipt for the callback with `input_data_ids = [data_id_X, data_id_X]`.
3. `process_action_receipt` sets `pending_data_count = 2` and one `PostponedReceiptId[data_id_X]` entry.
4. The DataReceipt for `data_id_X` arrives. `pending_data_count` decrements to 1. `PostponedReceiptId[data_id_X]` is removed.
5. Assert: the callback receipt is **never** executed. The `PendingDataCount` and `PostponedReceipt` trie entries remain. Any attached deposit is permanently locked. [4](#0-3)
