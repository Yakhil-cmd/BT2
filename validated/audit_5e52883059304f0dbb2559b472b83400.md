### Title
`TurnBridge` Message Handler is a Silent No-Op, Permanently Bypassing Admin Bridge Emergency Stop - (File: x/cronos/keeper/msg_server.go)

### Summary
The `TurnBridge` gRPC handler in `x/cronos/keeper/msg_server.go` returns `(nil, nil)` without executing any state change. The admin's only mechanism to disable the bridge in an emergency is silently broken: every `MsgTurnBridge` call succeeds on-chain but has zero effect, permanently bypassing the bridge control authority.

### Finding Description
`TurnBridge` at line 85–87 of `x/cronos/keeper/msg_server.go` is a stub:

```go
func (k msgServer) TurnBridge(goCtx context.Context, msg *types.MsgTurnBridge) (*types.MsgTurnBridgeResponse, error) {
    return nil, nil
}
``` [1](#0-0) 

The function accepts the message, performs no validation, writes no state, and returns success. The `MsgTurnBridge` type and its CLI command (`CmdTurnBridge`) are fully wired up and callable by any address holding the `CronosAdmin` role: [2](#0-1) 

The bridge is therefore permanently locked in whatever state it was initialized with. No on-chain action can change it.

This is the direct Cronos analog of the external report's class: a proxy/replacement implementation (`ModuleCRC20Proxy` replacing `ModuleCRC20` as the bridge entry point) was introduced, but the corresponding control-plane logic (`TurnBridge`) was never implemented — the handler is a placeholder stub, exactly as `iExecClerk` was replaced without updating the surrounding system. [3](#0-2) 

### Impact Explanation
This is a **High** impact finding under the allowed scope: *"Bypass of Cronos admin, governance authority, permission, token-mapping, bridge, block-list, or module-account authorization checks."*

The `CronosAdmin` bridge-disable authority is completely nullified. The upgrade handler itself acknowledges that an unauthenticated drain vulnerability existed in the `__CronosSendCroToIbc` hook and was closed by restricting `CroBridgeContractAddresses`: [4](#0-3) 

If a similar drain-class vulnerability is discovered in the future (e.g., in the `__CronosSendToIbc` or `__CronosSendToEvmChain` EVM hook paths processed by `LogProcessEvmHook`), the admin's only recourse — calling `TurnBridge` — silently does nothing. The bridge cannot be halted, and the drain continues indefinitely. [5](#0-4) 

### Likelihood Explanation
The `CronosAdmin` will call `MsgTurnBridge` in good faith during an incident. The transaction will be accepted by the chain (no error), the admin will believe the bridge is disabled, and bridge operations will continue unimpeded. The likelihood of this being triggered in a real attack scenario is high: bridge exploits are the most common DeFi attack class, and the upgrade history of this chain already shows one such drain was patched.

### Recommendation
Implement `TurnBridge` to write a boolean `bridge_enabled` flag into the cronos module params or a dedicated store key. Add a guard at the top of every EVM hook handler and IBC middleware that processes bridge events (e.g., `__CronosSendToIbc`, `__CronosSendToEvmChain`, `__CronosSendCroToIbc`) to reject processing when the flag is `false`. The `ConvertVouchers` and `TransferTokens` message handlers should also respect this flag. [6](#0-5) 

### Proof of Concept
1. Attacker deploys a contract that emits `__CronosSendToIbc` events to drain IBC vouchers via the `LogProcessEvmHook` path.
2. `CronosAdmin` observes the drain and submits `MsgTurnBridge{From: admin, Enable: false}`.
3. `TurnBridge` returns `nil, nil` — transaction succeeds, bridge state unchanged.
4. `LogProcessEvmHook.PostTxProcessing` continues to process every attacker-emitted log without any bridge-enabled check.
5. Drain continues; admin has no further recourse short of a governance upgrade. [5](#0-4) [1](#0-0)

func CmdTurnBridge() *cobra.Command {
	cmd := &cobra.Command{

		Short: "Turn Bridge",
		Args:  cobra.ExactArgs(1),
		RunE: func(cmd *cobra.Command, args []string) error {
			clientCtx, err := client.GetClientTxContext(cmd)
			if err != nil {
				return err
			}

			enable, err := strconv.ParseBool(args[0])
			if err != nil {
				return err
			}
			msg := types.NewMsgTurnBridge(clientCtx.GetFromAddress().String(), enable)
			if err := msg.ValidateBasic(); err != nil {
				return err
			}
			return tx.GenerateOrBroadcastTxCLI(clientCtx, cmd.Flags(), msg)
		},
	}

	flags.AddTxFlagsToCmd(cmd)
	return cmd
}

func (h LogProcessEvmHook) PostTxProcessing(ctx sdk.Context, _ *core.Message, receipt *ethtypes.Receipt) error {
	addLogToReceiptFunc := newFuncAddLogToReceipt(receipt)
	for _, log := range receipt.Logs {
		if len(log.Topics) == 0 {
			continue
		}
		handler, ok := h.handlers[log.Topics[0]]
		if !ok {
			continue
		}
		err := handler.Handle(ctx, log.Address, log.Topics, log.Data, addLogToReceiptFunc)
		if err != nil {
			return err
		}
	}
	return nil
}
