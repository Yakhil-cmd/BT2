### Title
Royalty Amount Understatement in WalletConnect Offer Confirmation Due to Integer-Division Split Error - (File: `packages/gui/src/electron/commands/parseCommandDisplay.ts`)

### Summary
The `formatAmountWithRoyalties` function, which computes the "Total Amount with Royalties" displayed in the WalletConnect `chia_wallet.take_offer` confirmation dialog, divides the trade amount equally among all NFTs using integer (truncating) division before applying each NFT's royalty percentage. This produces a displayed total that is systematically understated relative to the actual royalty obligation, causing a user to approve a `take_offer` request believing they will spend less than they actually will.

### Finding Description

In `parseCommandDisplay.ts`, `formatAmountWithRoyalties` is called for every fungible asset line in the WalletConnect confirmation dialog when the opposite side of the offer contains NFTs with royalties:

```
const splitAmount = amount / BigInt(royaltyPercentages.length);   // integer division, truncates
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
const totalAmount = amount + royaltyAmount;
``` [1](#0-0) 

Two compounding errors exist:

**Error 1 – Algorithmic (primary):** The code divides `amount` by the number of NFTs (`royaltyPercentages.length`) before computing each NFT's royalty. In Chia's protocol each NFT's royalty is applied to the full fungible trade amount, not to an equal share of it. The correct formula is `sum(floor(amount * ri / 10_000))`, but the code computes `sum(floor(floor(amount/N) * ri / 10_000))`. For N NFTs the displayed royalty is approximately N times smaller than the actual royalty obligation.

**Error 2 – Rounding (secondary, direct analog to the report):** `amount / BigInt(royaltyPercentages.length)` is BigInt integer division and truncates. The remainder `amount % N` mojos are silently dropped from every royalty calculation, so the displayed total is always ≤ the true total even if the split approach were otherwise correct.

**Concrete example** (from the existing test at line 251–335):
- `amount` = 100,000,000 mojos (0.0001 XCH), two NFTs with royalties 500 bp (5 %) and 10 bp (0.1 %)
- **Displayed** (split approach): `splitAmount` = 50,000,000; royalties = 2,500,000 + 50,000 = 2,550,000; total = 102,550,000 mojos
- **Actual** (full-amount per NFT): royalties = 5,000,000 + 100,000 = 5,100,000; total = 105,100,000 mojos
- **Discrepancy**: user sees 0.00010255 XCH but spends 0.0001051 XCH — a 2,550,000 mojo understatement [2](#0-1) 

The computed `amountWithRoyalties` string is surfaced directly in the WalletConnect confirmation dialog (`Confirm.tsx`) under the label "Total Amount with Royalties": [3](#0-2) 

### Impact Explanation

A user who receives a WalletConnect `chia_wallet.take_offer` request for multiple NFTs with royalties is shown an understated "Total Amount with Royalties" in the approval dialog. They approve believing they will spend X mojos; the actual on-chain deduction is materially larger. The discrepancy scales with the number of NFTs and their royalty percentages, and is fully attacker-controlled via the crafted offer. This constitutes displaying the wrong amount in a WalletConnect approval flow, directly matching the **High** impact tier: *"WalletConnect state that causes a user to approve … the wrong … amount."*

### Likelihood Explanation

Any unprivileged dApp connected via WalletConnect can issue a `chia_wallet.take_offer` request. The dApp controls the offer payload, including the number of NFTs and their embedded royalty percentages. No privileged access, leaked keys, or social engineering beyond a normal WalletConnect session is required. The trigger condition (multiple NFTs with royalties in a single offer) is a standard, supported Chia protocol feature. [4](#0-3) 

### Recommendation

Replace the split-then-royalty approach with the correct per-NFT full-amount calculation:

```typescript
// Correct: each NFT's royalty is applied to the full amount
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (amount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
const totalAmount = amount + royaltyAmount;
```

This eliminates both the algorithmic understatement and the integer-division truncation loss.

### Proof of Concept

1. Connect a dApp to Chia GUI via WalletConnect.
2. Issue `chia_wallet.take_offer` with an offer blob containing two NFTs, each with `royalty_percentage = 500` (5 %), and a requested XCH amount of 100,000,001 mojos.
3. Observe the confirmation dialog shows "Total Amount with Royalties: 0.00010255 XCH" (split calculation, truncated).
4. The actual on-chain deduction will be ≈ 0.0001051 XCH (each NFT's 5 % applied to the full 100,000,001 mojos).
5. The user approves at the lower displayed figure; their wallet is debited the higher actual figure. [5](#0-4)

function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],
): string | undefined {
  if (royaltyPercentages.length === 0 || line.kind === 'nft') {
    return undefined;
  }

  const splitAmount = amount / BigInt(royaltyPercentages.length);
  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;

  if (line.kind === 'xch') {
    return mojoToChiaLocaleString(totalAmount);
  }

  return mojoToCATLocaleString(totalAmount);
}
