### Title
`skip_stateful_validations` Bypasses `__validate__` for Invoke Transactions When Any Pending Transaction Exists for the Account — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator skips the `__validate__` entry-point (signature verification) for an invoke transaction with nonce=1 whenever `account_tx_in_pool_or_recent_block` returns `true` for the sender. That helper returns `true` for **any** account that has **any** transaction currently in the pool — not only a `deploy_account`. An attacker can therefore submit an invoke transaction with an arbitrary/invalid signature for any deployed account that has a pending nonce=0 transaction in the mempool, and the gateway will admit it without running `__validate__`.

---

### Finding Description

`skip_stateful_validations` is a UX feature that allows a user to submit a `deploy_account` and a first `invoke` (nonce=1) simultaneously, before the deploy has been processed. The gateway skips `__validate__` for the invoke because the account contract does not yet exist in state.

The guard condition is:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs  L437-L456
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
    return mempool_client
        .account_tx_in_pool_or_recent_block(tx.sender_address())
        .await
        ...
}
``` [1](#0-0) 

The comment asserts: *"It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."*

This reasoning is wrong. `account_tx_in_pool_or_recent_block` is implemented as:

```rust
// crates/apollo_mempool/src/mempool.rs  L697-L700
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [2](#0-1) 

It returns `true` for **any** account that has **any** transaction in the pool — including a plain invoke with nonce=0. There is no check that the pooled transaction is specifically a `deploy_account`.

When `skip_validate = true` is returned, `run_validate_entry_point` sets `validate: false` in the execution flags:

```rust
// crates/apollo_gateway/src/stateful_transaction_validator.rs  L311-L312
let execution_flags =
    ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };
``` [3](#0-2) 

Inside `StatefulValidator::perform_validations`, when `validate == false` the `__validate__` call is entirely skipped:

```rust
// crates/blockifier/src/blockifier/stateful_validator.rs  L79-L81
if !tx.execution_flags.validate {
    return Ok(());
}
``` [4](#0-3) 

The preceding `validate_by_mempool` call only checks for duplicate hashes and nonce ordering — it never inspects the signature:

```rust
// crates/apollo_mempool/src/mempool.rs  L402-L408
pub fn validate_tx(&mut self, args: ValidationArgs) -> MempoolResult<()> {
    let tx_reference = (&args).into();
    self.validate_incoming_tx(tx_reference, args.account_nonce)?;
    self.validate_fee_escalation(tx_reference)?;
    Ok(())
}
``` [5](#0-4) 

**Concrete attack path:**

1. Victim account `V` is deployed; its on-chain nonce is 0.
2. `V` submits a legitimate invoke with nonce=0 → admitted to the mempool. `tx_pool.contains_account(V)` is now `true`.
3. Attacker submits an invoke for `V` with nonce=1 and a **garbage signature**.
4. Gateway stateful validation:
   - `account_nonce == 0` ✓
   - `tx.nonce() == 1` ✓
   - `account_tx_in_pool_or_recent_block(V)` → `true` (V's nonce=0 invoke is in the pool)
   - `skip_validate = true` → `__validate__` is **not called**
5. The transaction with the invalid signature is admitted to the mempool and forwarded to the batcher.

---

### Impact Explanation

An attacker can inject invoke transactions with arbitrary/invalid signatures for any deployed account that currently has a pending nonce=0 transaction in the mempool. These transactions bypass the gateway's signature check and are admitted as valid. The batcher will attempt to execute them; they will fail at `__validate__` during sequencing (since `new_for_sequencing` sets `validate: true`), but they consume mempool capacity, batcher CPU, and block space. This matches the allowed impact: **"Mempool/gateway/RPC admission accepts invalid transactions before sequencing."**

---

### Likelihood Explanation

- Any deployed account that has ever submitted a nonce=0 transaction to the mempool is a valid target while that transaction is pending.
- The attacker needs only the victim's public address (observable on-chain or from the mempool).
- No privileged access, special keys, or prior relationship with the victim is required.
- The attack is trivially scriptable: monitor the mempool for nonce=0 transactions and immediately submit a nonce=1 invoke with a random signature for the same sender.

---

### Recommendation

Replace the coarse `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a `deploy_account` transaction is pending for the sender. The mempool should expose a dedicated query such as `has_pending_deploy_account(address)` that inspects the transaction type, not merely the account's presence. Alternatively, check the on-chain class hash: if `get_class_hash_at(sender_address)` returns a non-zero value, the account is already deployed and `__validate__` must never be skipped.

---

### Proof of Concept

```
1. Deploy account V (class_hash != 0, nonce = 0 in state).

2. Submit a valid invoke from V with nonce=0:
     sender_address = V
     nonce          = 0
     signature      = <valid ECDSA signature>
   → Gateway admits it; mempool now has V's nonce=0 tx.
   → tx_pool.contains_account(V) == true.

3. Submit a malicious invoke for V with nonce=1:
     sender_address = V
     nonce          = 1
     signature      = [0xdeadbeef]   // garbage
   → Gateway stateful check:
       account_nonce == 0  ✓
       tx.nonce()    == 1  ✓
       account_tx_in_pool_or_recent_block(V) == true  ✓
       → skip_validate = true
       → __validate__ NOT called
   → Transaction admitted to mempool with invalid signature.

4. Batcher calls get_txs(), receives both transactions.
   Executes V's nonce=0 tx (succeeds, V's nonce becomes 1).
   Executes attacker's nonce=1 tx:
     perform_pre_validation_stage: nonce check passes (1 == 1).
     __validate__ called (validate=true in new_for_sequencing).
     __validate__ FAILS (invalid signature).
     Transaction reverted / rejected.

Result: invalid-signature transaction was accepted by the gateway
        and reached the batcher, bypassing admission controls.
```
