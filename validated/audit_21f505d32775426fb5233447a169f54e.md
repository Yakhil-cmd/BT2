### Title
`Declare` Transactions Bypass `max_l2_gas_amount` Gateway Admission Check — (`File: crates/apollo_gateway/src/stateless_transaction_validator.rs`)

### Summary

`StatelessTransactionValidator::validate_resource_bounds` enforces the `max_l2_gas_amount` ceiling on `l2_gas.max_amount` for `Invoke` and `DeployAccount` transactions but explicitly skips that check for `Declare` transactions. An attacker can submit a `Declare` transaction whose `l2_gas.max_amount` exceeds the configured ceiling, bypassing the only gateway-level guard on that field. The transaction then propagates through stateful validation and into the mempool with an inflated initial Sierra gas budget, violating the admission invariant that every transaction type must respect the operator-configured L2 gas ceiling.

### Finding Description

In `validate_resource_bounds` the code reads:

```rust
// TODO(Arni): Consider adding a validation for max_l2_gas_amount for declare.
if let RpcTransaction::Declare(_) = tx {
} else if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh { … });
}
``` [1](#0-0) 

The empty `if let` arm is a deliberate no-op: when the transaction is a `Declare`, the `max_l2_gas_amount` check is skipped entirely. The TODO comment confirms this is a known gap. The test `valid_l2_gas_amount_on_declare` explicitly asserts that a `Declare` with `max_amount = 200` passes even when `max_l2_gas_amount = 100`, proving the bypass is intentional in the current code. [2](#0-1) 

The default ceiling is `1_210_000_000` gas units. [3](#0-2) 

For `Invoke` and `DeployAccount`, any value above that ceiling is rejected at the gateway before the transaction reaches the mempool. [4](#0-3) 

For `Declare`, no such rejection occurs.

The only downstream guard is `verify_can_pay_committed_bounds`, which checks that the account balance covers `max_amount × max_price_per_unit`. [5](#0-4) 

`max_possible_fee` uses saturating arithmetic, so an astronomically large `max_amount` saturates to `u128::MAX` and the balance check fails. However, an attacker who holds sufficient STRK can choose `max_amount` to be any value above the ceiling while keeping `max_amount × max_price_per_unit` within their balance. For example, with 1 000 STRK (10²¹ smallest units) and `min_gas_price = 8 000 000 000`, the attacker can set `max_amount ≈ 125 000 000 000` — roughly 100× the intended ceiling — and still pass the balance check.

Once admitted, `l2_gas.max_amount` is used directly as `initial_sierra_gas` for the transaction's execution:

```rust
TransactionInfo::Current(CurrentTransactionInfo {
    resource_bounds: ValidResourceBounds::AllResources(AllResourceBounds { l2_gas, .. }),
    ..
}) => {
    l2_gas.max_amount   // ← initial gas counter for __validate__ and execute
}
``` [6](#0-5) 

A `Declare` transaction with an inflated `max_amount` therefore executes with a gas budget far exceeding what the operator intended to allow per transaction.

### Impact Explanation

**High — Mempool/gateway admission accepts invalid transactions before sequencing.**

A `Declare` transaction with `l2_gas.max_amount` above `max_l2_gas_amount` passes stateless validation and, if the sender holds sufficient balance, passes stateful validation and enters the mempool. This violates the operator-configured admission invariant. The inflated `initial_sierra_gas` gives the transaction a larger execution budget than any `Invoke` or `DeployAccount` transaction is permitted to claim, creating an asymmetric resource-consumption capability that the gateway is specifically designed to prevent.

### Likelihood Explanation

Any user who can afford to hold STRK tokens proportional to `max_amount × min_gas_price` can trigger this. No privileged access is required. The attacker simply crafts a `Declare` RPC transaction with `l2_gas.max_amount` set above the ceiling. The bypass is unconditional — the code path is a deliberate empty branch, not a conditional guard.

### Recommendation

Remove the `Declare` exemption and apply the same `max_l2_gas_amount` ceiling to all transaction types:

```rust
// Remove the special-casing:
if resource_bounds.l2_gas.max_amount.0 > self.config.max_l2_gas_amount {
    return Err(StatelessTransactionValidatorError::MaxGasAmountTooHigh {
        gas_amount: resource_bounds.l2_gas.max_amount,
        max_gas_amount: self.config.max_l2_gas_amount,
    });
}
```

If `Declare` transactions legitimately require a higher gas ceiling (e.g., because class compilation is more expensive), introduce a separate `max_l2_gas_amount_declare` configuration parameter rather than removing the check entirely.

### Proof of Concept

1. Construct an `RpcDeclareTransaction::V3` with:
   - `resource_bounds.l2_gas.max_amount = max_l2_gas_amount + 1` (e.g., `1_210_000_001`)
   - `resource_bounds.l2_gas.max_price_per_unit = min_gas_price` (e.g., `8_000_000_000`)
   - Sender account funded with at least `1_210_000_001 × 8_000_000_000` STRK units
2. Submit to the gateway.
3. `StatelessTransactionValidator::validate` calls `validate_resource_bounds`, which hits the empty `if let RpcTransaction::Declare(_) = tx {}` branch and returns `Ok(())` without checking `max_l2_gas_amount`.
4. Stateful validation checks only the L2 gas price threshold and the balance; both pass.
5. The transaction enters the mempool with `l2_gas.max_amount = 1_210_000_001`, exceeding the ceiling that `Invoke` and `DeployAccount` transactions must respect.
6. During execution, `initial_sierra_gas` is set to `1_210_000_001`, giving the `Declare` transaction a larger execution budget than any other transaction type is permitted.

The existing test `valid_l2_gas_amount_on_declare` already demonstrates step 3 in isolation, confirming the bypass is reachable with no special setup. [2](#0-1)
