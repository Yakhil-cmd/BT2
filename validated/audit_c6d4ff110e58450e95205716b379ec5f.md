### Title
Blocklist Enforcement Bypass via Mempool Front-Running of `MsgStoreBlockList` - (`app/app.go`, `x/cronos/keeper/msg_server.go`, `app/proposal.go`)

### Summary
The Cronos encrypted blocklist is enforced through an in-memory map that is only refreshed at `EndBlock`. Because `MsgStoreBlockList` is submitted through the public mempool, a target address can observe the pending blocklist-update transaction and front-run it within the same block. Since `PrepareProposal` and `ProcessProposal` both use the stale in-memory blocklist (from the previous block's `EndBlock`), the target's transactions pass all enforcement checks and execute successfully in the same block as the blocklist update, fully bypassing the admin's intent to block them.

---

### Finding Description

The blocklist lifecycle has three distinct phases:

**Phase 1 – On-chain storage (during block execution):**
`MsgStoreBlockList` writes the encrypted blob to the KV store during `FinalizeBlock`. [1](#0-0) 

**Phase 2 – In-memory refresh (at EndBlock):**
`RefreshBlockList` is called at the very end of `EndBlocker`, reading the KV store and decrypting the blob into the in-memory `h.blocklist` map. [2](#0-1) 

**Phase 3 – Enforcement (PrepareProposal / ProcessProposal for the NEXT block):**
Both `PrepareProposal` (via `ExtTxSelector.SelectTxForProposal → ValidateTransaction`) and `ProcessProposalHandler` read only the in-memory `h.blocklist`, which was last set at the previous block's `EndBlock`. [3](#0-2) [4](#0-3) 

The consequence is a deterministic one-block enforcement gap: for the block that contains `MsgStoreBlockList`, the in-memory blocklist still reflects the state from the *previous* `EndBlock`. Any transaction from the target address that is also in the mempool will pass `PrepareProposal`'s `ValidateTransaction` check and be included in the same block.

Additionally, `BlockAddressesDecorator` — the ante-handler that gates mempool admission — only checks a **static** `blockedMap` set at node startup, not the encrypted blocklist. It therefore never rejects the target's transactions at `CheckTx`, ensuring they remain in the mempool and are available for inclusion. [5](#0-4) 

---

### Impact Explanation

This is a **High** impact finding: **Bypass of Cronos admin block-list authorization checks**.

The blocklist mechanism exists specifically to prevent a designated address from transacting on-chain (e.g., to freeze a hacker's address after an exploit). By front-running `MsgStoreBlockList`, the target address can execute arbitrary EVM transactions or Cosmos SDK messages — including asset transfers, IBC sends, or CRC20/CRC21 conversions — in the same block as the blocklist update. The admin's enforcement action is rendered ineffective for that block.

---

### Likelihood Explanation

- `MsgStoreBlockList` is a standard Cosmos SDK transaction submitted through the public mempool. Its message type is visible to all mempool observers even though the blob payload is encrypted.
- The target address does not need to decrypt the blob; observing a `MsgStoreBlockList` tx in the mempool is sufficient signal to act.
- On Cronos, EVM transactions support priority gas pricing (EIP-1559), making it straightforward for the target to outbid the admin's transaction and guarantee ordering within the same block.
- No special privileges, leaked keys, or validator compromise are required.

---

### Recommendation

1. **Private mempool submission:** The admin should submit `MsgStoreBlockList` via a private relay (analogous to Flashbots) that bypasses the public mempool, preventing the target from observing the pending transaction.

2. **Intra-block enforcement:** Apply the new blocklist immediately during `FinalizeBlock` after the `MsgStoreBlockList` message executes, rather than deferring to `EndBlock`. This would require the message handler to update the in-memory `h.blocklist` directly (or signal the proposal handler to re-read the KV store for subsequent transactions in the same block).

3. **CheckTx enforcement parity:** `BlockAddressesDecorator` should also consult the encrypted blocklist (or a cached plaintext version) so that blocked addresses are rejected at mempool admission, not just at proposal time. [6](#0-5) 

---

### Proof of Concept

**Setup:** Admin wants to block address `X` (e.g., a hacker who has drained a contract).

1. Admin constructs `MsgStoreBlockList` with a blob encrypting `{"addresses": ["X"]}` and broadcasts it to the public mempool.

2. Address `X` (or an automated bot watching the mempool) observes a pending `MsgStoreBlockList` transaction from the `CronosAdmin` address.

3. Address `X` immediately broadcasts a high-priority EVM transaction (e.g., `transfer(attacker_safe_wallet, balance)` on a CRC20 token) with a gas price exceeding the admin's transaction.

4. The block proposer runs `PrepareProposal`. The in-memory `h.blocklist` is still empty (last set at the previous `EndBlock`). Both transactions pass `ValidateTransaction` and are selected for the block. [4](#0-3) 

5. `ProcessProposalHandler` on all validators also uses the stale empty blocklist and accepts the block. [7](#0-6) 

6. During `FinalizeBlock`, address `X`'s transfer executes first (higher gas price → higher priority), moving assets to a safe wallet. Then `MsgStoreBlockList` executes, writing the new blob to the KV store.

7. At `EndBlock`, `RefreshBlockList` decrypts the blob and sets `h.blocklist = {X}`. [8](#0-7) 

8. From the next block onward, address `X` is blocked — but the asset transfer has already succeeded.

func (h *ProposalHandler) ProcessProposalHandler() sdk.ProcessProposalHandler {
	return func(ctx sdk.Context, req *abci.RequestProcessProposal) (*abci.ResponseProcessProposal, error) {
		if len(h.blocklist) == 0 {
			// fast path, accept all txs
			return &abci.ResponseProcessProposal{Status: abci.ResponseProcessProposal_ACCEPT}, nil
		}

		for _, txBz := range req.Txs {
			if err := h.ValidateTransaction(nil, txBz); err != nil {
				return &abci.ResponseProcessProposal{Status: abci.ResponseProcessProposal_REJECT}, nil
			}
		}

		return &abci.ResponseProcessProposal{Status: abci.ResponseProcessProposal_ACCEPT}, nil
	}
}
