### Title
Missing Participant-Set Membership Check in OT-Based ECDSA Presign Allows Malicious Coordinator to Deny Signing - (File: src/ecdsa/ot_based_ecdsa/presign.rs)

### Summary
The `presign()` function in the OT-based ECDSA pipeline explicitly omits validating that the presigning participant set is a subset of the triple generation participant set. The `TriplePub` struct records which participants generated the triple (historical state), but this is never checked against the current presigning participant set (current state). A malicious coordinator can exploit this by invoking presign with a participant set that diverges from the triple generation set, causing the presigning protocol to abort for all honest parties and denying signing.

### Finding Description
The `TriplePub` struct stores `participants: Vec<Participant>` — the exact set of parties who participated in triple generation. [1](#0-0) 

The `presign()` function receives `args: PresignArguments` which includes `triple0: (TripleShare, TriplePub)` and `triple1: (TripleShare, TriplePub)`. The function validates that `args.threshold == args.triple0.1.threshold` (threshold consistency), but explicitly omits checking that the current `participants` argument is a subset of `args.triple0.1.participants` or `args.triple1.1.participants`. [2](#0-1) 

This is the direct analog to the passport bug: the check only validates the current state (presigning participant set) but not the historical state (triple generation participant set). The orchestration specification explicitly requires `P1 ⊆ P0` (presigning participants must be a subset of triple generation participants): [3](#0-2) 

The `TriplePub.participants` field is populated during triple generation and stored in the output: [4](#0-3) 

But it is never read or validated inside `presign()`. The `PresignArguments` struct carries both triples (including their `TriplePub` with the `participants` field), yet the presign initialization function ignores this field entirely: [5](#0-4) 

### Impact Explanation
A malicious coordinator can invoke `presign()` with a participant set `P1` that includes parties who were never part of the triple generation set `P0`. These parties do not hold valid triple shares. When they participate in presigning, the internal consistency checks in `do_presign` will fail: [6](#0-5) [7](#0-6) 

These checks abort the presigning protocol with a `ProtocolError` for all honest parties. The coordinator can persistently repeat this attack across every presigning attempt, consuming honest parties' pre-generated triples (which are one-time-use) and forcing them to regenerate triples before retrying — effectively denying signing indefinitely. This matches the High impact: **Permanent denial of signing for honest parties under valid protocol inputs and documented trust assumptions**.

### Likelihood Explanation
The coordinator role is reachable by any participant — the library places no privilege restriction on who can act as coordinator. A malicious participant who is assigned or self-assigns the coordinator role can trivially supply a mismatched participant list to `presign()`. No leaked keys or external compromise is required. The attack is repeatable at negligible cost to the attacker while forcing honest parties to regenerate expensive OT-based triples on every attempt.

### Recommendation
Add an explicit membership check inside `presign()` that validates the presigning participant set is a subset of both `args.triple0.1.participants` and `args.triple1.1.participants`, replacing the current assumption with an enforced invariant:

```rust
// Enforce that presigning participants are a subset of triple generation participants
let triple0_set: std::collections::HashSet<_> = args.triple0.1.participants.iter().collect();
let triple1_set: std::collections::HashSet<_> = args.triple1.1.participants.iter().collect();
for p in participants.iter() {
    if !triple0_set.contains(p) || !triple1_set.contains(p) {
        return Err(InitializationError::BadParameters(
            "Presigning participant set must be a subset of the triple generation participant set".to_string(),
        ));
    }
}
```

This mirrors the pattern used in `assert_reshare_keys_invariants` which validates intersection size against the old threshold: [8](#0-7) 

### Proof of Concept
1. Honest parties A, B, C run `generate_triple` with `P0 = {A, B, C}`, threshold 2. Each receives a `TriplePub` with `participants = [A, B, C]`.
2. Malicious coordinator (party A) invokes `presign()` with `participants = [A, B, D]` where `D ∉ P0`, passing the triples generated by `P0`.
3. `presign()` accepts this call — it checks `args.threshold == args.triple0.1.threshold` (passes, both are 2) but never checks `D ∈ args.triple0.1.participants`.
4. Inside `do_presign`, D has no valid triple share. D's contribution to `e = Σ e_j` is inconsistent with the public commitment `big_e = E(0)`.
5. The check `big_e != (ProjectivePoint::GENERATOR * e).to_affine()` at line 127 fails, aborting the protocol with `ProtocolError::AssertionFailed`.
6. Honest parties B and C have consumed their one-time-use triple shares and must regenerate before retrying. The coordinator repeats this indefinitely. [9](#0-8)

pub fn presign(
    participants: &[Participant],
    me: Participant,
    args: PresignArguments,
) -> Result<impl Protocol<Output = PresignOutput>, InitializationError> {
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }
    // Spec 1.1
    if args.threshold.value() > participants.len() {
        return Err(InitializationError::ThresholdTooLarge {
            threshold: args.threshold.value(),
            max: participants.len(),
        });
    }

    // NOTE: We omit the check that the new participant set was present for
    // the triple generation, because presumably they need to have been present
    // in order to have shares.

    // Also check that we have enough participants to reconstruct shares.
    if args.threshold != args.triple0.1.threshold || args.threshold != args.triple1.1.threshold {
        return Err(InitializationError::BadParameters(
            "New threshold must match the threshold of both triples".to_string(),
        ));
    }

    let participants =
        ParticipantList::new(participants).ok_or(InitializationError::DuplicateParticipants)?;

    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    let ctx = Comms::new();
    let fut = do_presign(ctx.shared_channel(), participants, me, args);
    Ok(make_protocol(ctx, fut))
}
