### Title
Unauthenticated Invoke Admission via `skip_stateful_validations` Bypasses Signature Verification for Any Undeployed Account - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary
The `skip_stateful_validations` function skips the `__validate__` entry point for invoke transactions with `nonce=1` when the sender account has any transaction in the mempool. The check uses the attacker-controlled `sender_address` field from the transaction itself, not the actual submitter's identity. Any unprivileged attacker can therefore submit an invoke transaction claiming to be from any undeployed victim account, have it admitted to the mempool without signature verification, and displace or front-run the victim's legitimate post-deploy invoke.

### Finding Description

`skip_stateful_validations` at lines 429–461 returns `true` (skip `__validate__`) when all of:

1. The transaction is `ExecutableTransaction::Invoke`
2. `tx.nonce() == Nonce(Felt::ONE)`
3. `account_nonce == Nonce(Felt::ZERO)` (account not yet on-chain)
4. `account_tx_in_pool_or_recent_block(tx.sender_address()) == true` [1](#0-0) 

Condition 4 queries the mempool using `tx.sender_address()`, which is a field the attacker freely sets in the RPC payload. The mempool check `account_tx_in_pool_or_recent_block` only asks "does this address appear in the pool or a recent block?" — it does not verify that the submitter owns the account. [2](#0-1) 

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` is called with `skip_validate=true`, so the `__validate__` entry point (which performs the ECDSA signature check) is never invoked at the gateway level. [3](#0-2) 

The upstream `validate_by_mempool` call (which runs before the skip check) only validates nonce range and duplicate-hash; it performs no signature check. [4](#0-3) 

The stateless validator also performs no signature check. [5](#0-4) 

The result is that the gateway's `add_tx_inner` path admits the attacker's unsigned/mis-signed invoke transaction into the mempool. [6](#0-5) 

### Impact Explanation

**High — Mempool/gateway admission accepts invalid transactions before sequencing.**

An attacker can inject an invoke transaction for any undeployed victim account (one whose deploy_account is pending in the mempool) without possessing the victim's private key. The invalid transaction occupies the `(victim_address, nonce=1)` slot in the mempool. If fee escalation is enabled, the attacker can replace the victim's legitimate invoke by paying a marginally higher fee. Even without fee escalation, the attacker can race to occupy the slot first, causing the victim's own invoke to be rejected as `DuplicateNonce`. When the block executes, the attacker's transaction fails at the `__validate__` step (signature invalid), but the nonce is still incremented to 2, permanently invalidating the victim's nonce-1 transaction. The attacker can repeat this for every subsequent nonce the victim attempts to use.

### Likelihood Explanation

The attack requires only that the victim's deploy_account transaction is visible in the mempool (public information). No special privilege, key material, or on-chain state is needed. The attacker submits a well-formed invoke with `sender_address = victim`, `nonce = 1`, any calldata, and a garbage signature. The gateway's admission path has no guard against this.

### Recommendation

Restrict the validation-skip path so it can only be triggered by the account that owns the deploy_account transaction. One concrete approach: require the deploy_account and the first invoke to arrive together in a single atomic submission (e.g., a bundle endpoint), so the gateway can verify both share the same sender before skipping `__validate__`. Alternatively, perform a lightweight signature pre-check (e.g., verify the signature length and format matches the declared account class) even when skipping full `__validate__` execution, to raise the cost of the attack.

### Proof of Concept

```
1. Victim broadcasts: deploy_account(sender=V, nonce=0, sig=valid_sig_V)
   → mempool admits it; account_tx_in_pool_or_recent_block(V) = true

2. Attacker broadcasts: invoke(sender=V, nonce=1, calldata=anything, sig=0x0)
   → gateway checks:
       nonce==1 ✓, account_nonce==0 ✓, V in mempool ✓
       → skip_stateful_validations returns true
       → __validate__ NOT called
       → transaction admitted to mempool

3. (Optional, if fee_escalation enabled)
   Victim broadcasts: invoke(sender=V, nonce=1, calldata=intended, sig=valid_sig_V)
   → mempool rejects: DuplicateNonce OR attacker replaces victim's tx by paying +1% fee

4. Batcher executes block:
   - deploy_account(V, nonce=0) succeeds → V.nonce = 1
   - attacker's invoke(V, nonce=1) runs __validate__ → FAILS (bad sig) → reverts
     but nonce is incremented: V.nonce = 2

5. Victim's original invoke(nonce=1) is gone from mempool.
   Victim must resubmit with nonce=2; attacker repeats from step 2.
```
