### Title
Silo Whitelist Access Control Bypassed via `call` Method — (`engine/src/contract_methods/evm_transactions.rs`)

### Summary

The Aurora Engine's silo mode uses a whitelist system to restrict which NEAR accounts and EVM addresses may submit EVM transactions or deploy EVM code. This restriction is enforced only inside `submit` and `submit_with_args`, but the `call` and `deploy_code` entry points perform no such check. Any unprivileged NEAR account can bypass the whitelist entirely by invoking the `call` method instead of `submit`.

### Finding Description

The silo whitelist system defines four lists:

- `Admin` / `EvmAdmin` — restrict who may deploy EVM code
- `Account` / `Address` — restrict who may submit EVM transactions [1](#0-0) 

The enforcement point is `assert_access`, called inside `submit_with_alt_modexp`: [2](#0-1) 

```rust
// Check if the sender has rights to submit transactions or deploy code.
assert_access(&io, env, &transaction)?;
```

`assert_access` delegates to `silo::is_allow_submit` or `silo::is_allow_deploy` depending on whether the transaction has a `to` field: [3](#0-2) 

However, the `call` contract method — which allows any NEAR account to call an EVM contract directly — contains **no call to `assert_access`**: [4](#0-3) 

```rust
pub fn call<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I, env: &E, handler: &mut H,
) -> Result<SubmitResult, ContractError> {
    with_logs_hashchain(io, env, function_name!(), |mut io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;          // ← only checks "not paused"
        // ... NO assert_access / whitelist check ...
        let result = engine.call_with_args(args, handler)?;
        ...
    })
}
```

Similarly, `deploy_code` performs no whitelist check: [5](#0-4) 

The `call` NEAR entry point is publicly exposed: [6](#0-5) 

This is structurally identical to the H-02 pattern: the guard (`assert_access`) is only applied when the user takes one specific path (`submit`/`submit_with_args`), but a parallel path (`call`) reaches the same EVM execution without the guard.

### Impact Explanation

When a silo operator enables the `Account` and `Address` whitelists to restrict EVM access to a curated set of participants, any non-whitelisted NEAR account can still execute arbitrary EVM transactions by calling the `call` method. The EVM sender is set to the implicit address derived from the caller's NEAR account (`predecessor_address(&predecessor_account_id)`), so the attacker executes with a valid EVM identity. They can:

- Transfer ERC-20 tokens held by their EVM address to arbitrary recipients
- Interact with any EVM contract (DEX, lending protocol, bridge) deployed on the silo
- Trigger state changes that the silo operator explicitly intended to block

This constitutes a complete bypass of the silo access-control boundary, enabling unauthorized fund movement and state mutation. Impact: **High — theft of unclaimed yield / temporary freezing of funds / direct fund theft** depending on the EVM contracts deployed in the silo.

### Likelihood Explanation

The `call` method is a standard, publicly documented Aurora Engine entry point. Any NEAR account can invoke it with no special privileges. No key compromise, governance capture, or social engineering is required. An attacker only needs to know the Aurora contract account ID and craft a valid `CallArgs` payload.

### Recommendation

Add the same `assert_access` guard to `call` (and `deploy_code`) that is already present in `submit_with_alt_modexp`. Because `call` does not parse a signed EVM transaction, a synthetic `NormalizedEthTransaction` must be constructed (as is already done in the standalone storage layer for indexing purposes) or a dedicated silo check must be extracted:

```rust
pub fn call<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I, env: &E, handler: &mut H,
) -> Result<SubmitResult, ContractError> {
    with_logs_hashchain(io, env, function_name!(), |mut io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        let bytes = io.read_input().to_vec();
        let args = CallArgs::deserialize(&bytes).ok_or(errors::ERR_BORSH_DESERIALIZE)?;
        let current_account_id = env.current_account_id();
        let predecessor_account_id = env.predecessor_account_id();
        let sender = predecessor_address(&predecessor_account_id);

+       // Enforce silo whitelist for `call` just as for `submit`.
+       let allowed = silo::is_allow_submit(&io, &predecessor_account_id, &sender);
+       if !allowed {
+           return Err(EngineErrorKind::NotAllowed.into());
+       }

        let mut engine: Engine<_, E, AuroraModExp> = Engine::new_with_state(
            state, sender, current_account_id, io, env,
        );
        let result = engine.call_with_args(args, handler)?;
        ...
    })
}
```

Apply the equivalent fix to `deploy_code` using `silo::is_allow_deploy`.

### Proof of Concept

1. Deploy Aurora Engine in silo mode; enable the `Account` and `Address` whitelists with `set_whitelist_status`.
2. Do **not** add the attacker's NEAR account or EVM address to any whitelist.
3. Confirm that calling `submit` with a signed EVM transaction from the attacker's account returns `EngineErrorKind::NotAllowed` (as tested in `test_submit_access_right`).
4. Now call the `call` NEAR method from the same attacker account with a `CallArgs` payload targeting any EVM contract (e.g., an ERC-20 `transfer`).
5. Observe that the call **succeeds** — the whitelist is not consulted, and the EVM state is mutated.

The existing test suite confirms step 3 works correctly: [7](#0-6) 

but there is no corresponding test asserting that `call` is also blocked, which is where the gap lies.

fn assert_access<I: IO + Copy, E: Env>(
    io: &I,
    env: &E,
    transaction: &NormalizedEthTransaction,
) -> Result<(), EngineError> {
    let allowed = if transaction.to.is_some() {
        silo::is_allow_submit(io, &env.predecessor_account_id(), &transaction.address)
    } else {
        silo::is_allow_deploy(io, &env.predecessor_account_id(), &transaction.address)
    };

    if !allowed {
        return Err(EngineError {
            kind: EngineErrorKind::NotAllowed,
            gas_used: 0,
        });
    }

    Ok(())
}

#[test]
fn test_submit_access_right() {
    let (mut runner, signer, receiver) = initialize_transfer();
    let sender = utils::address_from_secret_key(&signer.secret_key);
    let caller: AccountId = CALLER_ACCOUNT_ID.parse().unwrap();
    let transaction = utils::transfer_with_price(
        receiver,
        TRANSFER_AMOUNT,
        INITIAL_NONCE.into(),
        ONE_GAS_PRICE.raw(),
    );

    set_silo_params(&mut runner, Some(SILO_PARAMS_ARGS));
    enable_all_whitelists(&mut runner);

    validate_address_balance_and_nonce(&runner, sender, INITIAL_BALANCE, INITIAL_NONCE.into())
        .unwrap();
    validate_address_balance_and_nonce(&runner, receiver, ZERO_BALANCE, INITIAL_NONCE.into())
        .unwrap();

    // perform transfer
    let err = runner
        .submit_transaction(&signer.secret_key, transaction.clone())
        .unwrap_err();
    assert_eq!(err.kind, EngineErrorKind::NotAllowed);

    // validate post-state
    validate_address_balance_and_nonce(&runner, sender, INITIAL_BALANCE, INITIAL_NONCE.into())
        .unwrap();
    validate_address_balance_and_nonce(&runner, receiver, ZERO_BALANCE, INITIAL_NONCE.into())
        .unwrap();

    // Add caller and signer to whitelists.
    add_account_to_whitelist(&mut runner, caller);
    add_address_to_whitelist(&mut runner, sender);

    // perform transfer
    let result = runner
        .submit_transaction(&signer.secret_key, transaction)
        .unwrap();
    assert!(matches!(result.status, TransactionStatus::Succeed(_)));

    // validate post-state
    validate_address_balance_and_nonce(
        &runner,
        sender,
        INITIAL_BALANCE - TRANSFER_AMOUNT - FIXED_GAS * ONE_GAS_PRICE,
        (INITIAL_NONCE + 1).into(),
    )
    .unwrap();
    validate_address_balance_and_nonce(&runner, receiver, TRANSFER_AMOUNT, INITIAL_NONCE.into())
        .unwrap();
}
