### Title
Truncating Division in `delta_gas` Calculation Allows Systematic Underpayment of Native Resource Fees — (`File: basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`)

---

### Summary

In `compute_gas_refund`, the conversion from native resource consumption back to EVM gas units uses integer truncation (floor division). The remainder `native_used % native_per_gas` is silently discarded every transaction, meaning a sender always pays for slightly fewer gas units than their actual native resource consumption warrants. An attacker can craft transactions that maximize this remainder, systematically underpaying for proving work.

---

### Finding Description

The ZKsync OS double-resource accounting model charges transactions for both EVM gas (ergs) and a "native" resource representing prover complexity. After execution, if native resource consumption exceeds what EVM gas alone accounts for, an additional `delta_gas` is added to `gas_used` so the sender pays for the extra proving cost:

```
deltaGas := (nativeUsed / nativePerGas) - gasUsed
```

The implementation in `compute_gas_refund` is:

```rust
let delta_gas = if native_per_gas == 0 {
    0
} else {
    (native_used / native_per_gas) as i64 - (gas_used as i64)  // line 72: truncating division
};

if delta_gas > 0 {
    gas_used += delta_gas as u64;
}
```

The division `native_used / native_per_gas` is integer truncation (floor). The remainder `native_used % native_per_gas` is never charged. Because this rounding always favors the sender, every transaction that consumes native resources leaks up to `(native_per_gas - 1)` native units for free.

`native_per_gas` is computed as `ceil(gas_price / native_price)`. With a gas price of 1000 and a native price of 3, `native_per_gas = 334`, so up to 333 native units per transaction are unaccounted for. The fee savings per transaction is up to `(native_per_gas - 1) × gas_price` wei.

A secondary, compounding rounding error exists in the `native_per_pubdata` derivation:

```rust
// validation_impl.rs line 142
let native_per_pubdata = u256_try_to_u64(&pubdata_price.wrapping_div(native_price))
    .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
```

`wrapping_div` is also floor division. Each pubdata byte is charged `floor(pubdata_price / native_price)` native units instead of the exact ratio, again systematically benefiting the sender. Note the asymmetry: `native_per_gas` is computed with `div_ceil` (protecting the protocol), while `native_per_pubdata` uses `wrapping_div` (benefiting the user).

---

### Impact Explanation

**Vulnerability class:** Resource accounting bug — systematic truncation-rounding that always benefits the transaction sender.

The operator/prover performs real work (RISC-V cycles) for every native unit consumed. When `native_used % native_per_gas` native units are consumed but not charged, the operator is underpaid for that proving work. An attacker who crafts transactions to maximize the remainder extracts free proving capacity from the network. With a large `native_per_gas` ratio (achievable by setting a high gas price relative to `native_price`), the per-transaction savings can be substantial. Across many transactions, this constitutes a material drain on operator revenue and a subsidy for block-stuffing with computationally expensive transactions.

---

### Likelihood Explanation

Any unprivileged L2 transaction sender can trigger this path. No special role, key, or governance access is required. The attacker only needs to submit standard EIP-1559 transactions with a gas price chosen to maximize `native_per_gas`. The rounding error fires on every transaction that consumes native resources, so it is not a corner case — it is the default behavior. The attack is capital-efficient: the attacker pays gas fees but recovers a fraction of the proving cost on every transaction.

---

### Recommendation

1. **Round `delta_gas` up (ceiling division):** Replace the truncating division with `div_ceil` so the sender is charged for the full native consumption:
   ```rust
   // Instead of:
   (native_used / native_per_gas) as i64 - (gas_used as i64)
   // Use:
   native_used.div_ceil(native_per_gas) as i64 - (gas_used as i64)
   ```
   This mirrors the fix described in the original report: rounding should always disfavor the party that benefits from lower charges.

2. **Use `div_ceil` for `native_per_pubdata`:** In `validation_impl.rs`, replace `wrapping_div` with `div_ceil` for consistency with `native_per_gas` and to prevent the secondary underpayment on pubdata:
   ```rust
   let native_per_pubdata = u256_try_to_u64(&pubdata_price.div_ceil(native_price))
       .ok_or(TxError::Validation(InvalidTransaction::PubdataPriceTooHigh))?;
   ```

3. **Track total native cost as a numerator/denominator pair** (analogous to the report's recommendation of maintaining total acquisition cost) to avoid accumulated rounding drift across the two division sites.

---

### Proof of Concept

**Setup:** `gas_price = 1000`, `native_price = 3` → `native_per_gas = ceil(1000/3) = 334`.

**Craft a transaction** that consumes exactly `334k + 333` native units (e.g., by tuning calldata size and EVM opcode mix to hit the target native consumption) while consuming minimal EVM gas (so `gas_used` from ergs is small, say 21,000).

**At refund time:**
```
native_used = 334 * k + 333
delta_gas   = floor((334k + 333) / 334) - 21000
            = k - 21000          ← 333 native units silently dropped
```

The sender is charged for `k` gas units of native cost instead of `k + 333/334 ≈ k + 0.997` gas units. The 333 native units (worth `333 × 1000 = 333,000 wei` in fees) are free.

Repeating across `N` transactions yields `N × 333,000 wei` in unearned fee savings, with the prover bearing the full proving cost. [1](#0-0) [2](#0-1) [3](#0-2)
