### Title
Gateway Admits Unsigned/Invalid-Signature Invoke Transactions via `skip_stateful_validations` Overly-Broad Mempool Check — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function bypasses the `__validate__` entry-point (signature verification) for any invoke transaction with nonce=1 whenever `account_tx_in_pool_or_recent_block` returns `true`. That helper returns `true` for **any** account that has **any** transaction in the pool — not specifically a `deploy_account`. An unprivileged attacker who observes a victim's pending `deploy_account` in the mempool can submit an invoke with nonce=1 carrying an arbitrary/invalid signature; the gateway admits it without signature verification. The transaction is later executed by the batcher (which does call `__validate__`), reverts, and the victim account is charged fees.

### Finding Description

**Root cause — `skip_stateful_validations`:**

```
crates/apollo_gateway/src/stateful_transaction_validator.rs  lines 429-461
```

The function returns `true` (skip validation) when all three conditions hold:

1. The incoming transaction is an `Invoke` with `nonce == 1`
2. The on-chain account nonce read from `gateway_fixed_block_state_reader` is `0`
3. `mempool_client.account_tx_in_pool_or_recent_block(sender)` returns `true` [1](#0-0) 

The code comment claims: *"It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction or transactions with future nonces that passed validations."* This is incorrect. `account_tx_in_pool_or_recent_block` returns `true` for **any** account that has any transaction in the pool — including a `deploy_account` submitted by the victim themselves — without verifying that the incoming invoke's signature belongs to the same key. [2](#0-1) 

**Effect on the gateway validation pipeline:**

When `skip_validate = true`, `run_validate_entry_point` constructs an `AccountTransaction` with `execution_flags.validate = false`: [3](#0-2) 

`StatefulValidator::perform_validations` then short-circuits immediately for invoke transactions when `validate == false`: [4](#0-3) 

The transaction is forwarded to the mempool with no signature check performed.

**Execution path does NOT inherit the skip:**

`AccountTransaction::new_for_sequencing` (used by the batcher) always sets `validate: true`: [5](#0-4) 

So `__validate__` is called at execution time, the invalid signature causes a revert, and the victim is charged fees.

**Fee-escalation amplifier:**

If the victim also submitted a valid invoke(nonce=1) alongside their `deploy_account`, the attacker can replace it by submitting an invoke with a higher `tip`. The mempool's fee-escalation logic accepts the replacement without re-running `__validate__` (the skip is triggered again), so the victim's valid transaction is evicted and replaced with the attacker's reverting one. [6](#0-5) 

**Secondary vector — stale state reader:**

`account_tx_in_pool_or_recent_block` also returns `true` for accounts recorded in the mempool's committed-block state (`self.state.contains_account`), which persists indefinitely after a block commit: [7](#0-6) 

If the gateway's `gateway_fixed_block_state_reader` lags one block behind the mempool's committed state (a normal race window in the distributed architecture), an attacker can trigger the same skip for an account whose `deploy_account` was already committed, even though no pending `deploy_account` exists in the pool.

### Impact Explanation

**Allowed impact matched:** *"High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

- The gateway admits an invoke transaction whose signature has never been verified.
- The transaction is sequenced, executed, reverts, and the victim account is charged fees.
- With the fee-escalation path, the attacker can additionally evict the victim's legitimate invoke, permanently blocking that nonce until the victim resubmits.

### Likelihood Explanation

The mempool is observable (transactions are broadcast). Any attacker watching for `deploy_account` submissions can immediately race-submit a nonce=1 invoke with an arbitrary signature. No privileged access, no special knowledge, and no cryptographic capability is required. The only constraint is submitting before the victim's `deploy_account` is committed to a block.

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a **`deploy_account` transaction** is pending in the pool for the sender address. The mempool should expose a dedicated query such as `has_pending_deploy_account(address) -> bool` that inspects only `deploy_account`-typed entries. The `skip_stateful_validations` function should call that narrower predicate instead of the current one.

### Proof of Concept

```
1. Victim submits RpcDeployAccountTransaction for address A (nonce=0).
   → Gateway admits it; mempool now has deploy_account for A.
   → account_tx_in_pool_or_recent_block(A) == true.

2. Attacker submits RpcInvokeTransaction {
       sender_address: A,
       nonce: 1,
       signature: [0x0, 0x0],   // arbitrary invalid signature
       calldata: [<drain_calldata>],
       resource_bounds: { l2_gas: { max_amount: MAX, max_price: MAX } },
       ...
   }

3. Gateway stateful validation:
   - get_nonce_from_state(A) → 0  (account not yet deployed)
   - validate_nonce: nonce=1 ≥ account_nonce=0 ✓
   - validate_by_mempool: no duplicate hash, nonce in gap ✓
   - skip_stateful_validations:
       tx.nonce()==1 ✓, account_nonce==0 ✓,
       account_tx_in_pool_or_recent_block(A)==true ✓
       → returns true (skip __validate__)
   - run_validate_entry_point(skip_validate=true):
       execution_flags.validate = false → __validate__ NOT called ✓
   → Invoke admitted to mempool.

4. Batcher executes block:
   - deploy_account(A, nonce=0) executes → A deployed, nonce→1.
   - invoke(A, nonce=1) executes:
       AccountTransaction::new_for_sequencing → validate=true
       __validate__ called → signature [0x0,0x0] fails → REVERT
       Victim A charged fees for reverted tx.

5. (Optional escalation variant)
   If victim also submitted invoke(nonce=1, tip=T), attacker submits
   invoke(nonce=1, tip=T+1, signature=invalid).
   validate_fee_escalation accepts replacement (higher tip).
   Victim's valid invoke is evicted; attacker's reverting invoke executes.
```
