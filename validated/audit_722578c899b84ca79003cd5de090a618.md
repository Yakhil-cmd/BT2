### Title
Malicious Participant Can Corrupt CKD Output by Sending Arbitrary Share Values Without Proof of Correctness — (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

The CKD coordinator in `do_ckd_coordinator` blindly aggregates `(big_y, big_c)` values received from all participants with no cryptographic proof that each share was honestly computed. Any single malicious participant can send arbitrary group elements, causing the coordinator to produce a corrupted CKD output that will never yield the correct confidential key for the app.

---

### Finding Description

In `do_ckd_participant`, each honest participant computes:

- `norm_big_y = λ_i · y_i · G`
- `norm_big_c = λ_i · (x_i · H(pk ‖ app_id) + y_i · app_pk)`

and sends `(norm_big_y, norm_big_c)` privately to the coordinator. [1](#0-0) 

In `do_ckd_coordinator`, the coordinator receives these values via `recv_from_others` and unconditionally adds them to its running aggregate: [2](#0-1) 

`recv_from_others` only validates that the **sender identity** is a known participant (via `ParticipantCounter`); it performs no validation of the **content** of the message: [3](#0-2) 

There is no zero-knowledge proof or consistency check that the received `(big_y, big_c)` pair was formed using the participant's actual key share `x_i` and a consistent blinding scalar `y_i`. A malicious participant is free to send any two arbitrary `G1` points.

This is the direct analog of the NFT `mintNFT()` issue: just as that function was `public` with no `onlyOwner` guard, the CKD coordinator's aggregation step imposes no correctness guard on each participant's contribution — any participant in the list can inject arbitrary values into the aggregate.

---

### Impact Explanation

The coordinator computes the final `CKDOutput` as:

```
big_Y = Σ norm_big_y_i
big_C = Σ norm_big_c_i
``` [4](#0-3) 

The app then unmasks the confidential key as `big_C − app_sk · big_Y`, which should equal `msk · H(pk ‖ app_id)`: [5](#0-4) 

If even one participant substitutes arbitrary `(big_y', big_c')`, the aggregate is poisoned and the unmasked value will be a random, useless group element — not the intended confidential key. Honest parties will accept this corrupted output as the protocol result, with no way to detect the manipulation.

**Allowed impact matched**: *High — Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.*

---

### Likelihood Explanation

- Any single participant in the CKD session can trigger this; no threshold of colluders is required.
- The attacker needs only to be a legitimate member of the participant list (a documented trust assumption for participants, not coordinators).
- The attack requires no leaked keys, no external compromise, and no cryptographic break — only the ability to send a crafted message over the existing protocol channel.

---

### Recommendation

Add a zero-knowledge proof of correct share formation to each participant's message. Specifically, each participant should prove in zero-knowledge that `(big_y, big_c)` satisfies:

```
big_c = x_i · H(pk ‖ app_id) + y_i · app_pk
big_y = y_i · G
```

for the same scalar `y_i`, without revealing `x_i` or `y_i`. A standard sigma protocol (e.g., a Chaum-Pedersen proof of discrete-log equality) over the BLS12-381 G1 group suffices. The coordinator must verify each proof before including the share in the aggregate. [6](#0-5) 

---

### Proof of Concept

1. Run a CKD session with participants `[P1, P2, P3]` and coordinator `P1`.
2. Participant `P2` (malicious) intercepts the `do_ckd_participant` path and instead of computing the honest `(norm_big_y, norm_big_c)`, sends `(G, G)` (the generator point) to the coordinator.
3. The coordinator's `recv_from_others` loop accepts the message because `P2` is in the participant list.
4. The coordinator computes `big_Y = norm_big_y_P1 + G + norm_big_y_P3` and `big_C = norm_big_c_P1 + G + norm_big_c_P3`.
5. The app calls `ckd_output.unmask(app_sk)` and receives a value that is not `msk · H(pk ‖ app_id)`.
6. The CKD protocol silently produces a wrong key; no error is returned to honest parties. [7](#0-6)

async fn do_ckd_coordinator(
    mut chan: SharedChannel,
    participants: ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<CKDOutputOption, ProtocolError> {
    let (mut norm_big_y, mut norm_big_c) =
        compute_signature_share(&participants, me, key_pair, app_id, app_pk, rng)?;

    // Receive everyone's inputs and add them together
    let waitpoint = chan.next_waitpoint();

    for (_, participant_output) in
        recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
    {
        norm_big_y += participant_output.big_y();
        norm_big_c += participant_output.big_c();
    }
    let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
    Ok(Some(ckd_output))

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
