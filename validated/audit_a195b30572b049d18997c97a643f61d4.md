### Title
Silent `ValidResourceBounds::L1Gas` Downgrade in `estimate_fee` for V3 Invoke with Zero L2/L1DataGas Bounds Produces Wrong `GasVectorComputationMode` and Incorrect Fee Estimate — (`crates/apollo_rpc/src/v0_8/transaction.rs`)

---

### Summary

`From<ResourceBoundsMapping> for ValidResourceBounds` silently converts any V3 invoke whose `l2_gas` and `l1_data_gas` are both zero into `ValidResourceBounds::L1Gas` instead of `ValidResourceBounds::AllResources`. This changes `get_gas_vector_computation_mode()` from `All` to `NoL2Gas`, causing `exec_estimate_fee` to compute the fee in the wrong mode and return an authoritative-looking but incorrect `overall_fee`.

---

### Finding Description

**The downgrade logic** in `crates/apollo_rpc/src/v0_8/transaction.rs`:

```rust
impl From<ResourceBoundsMapping> for ValidResourceBounds {
    fn from(value: ResourceBoundsMapping) -> Self {
        if value.l1_data_gas.is_zero() && value.l2_gas.is_zero() {
            Self::L1Gas(value.l1_gas)          // ← silent downgrade
        } else {
            Self::AllResources(AllResourceBounds { ... })
        }
    }
}
``` [1](#0-0) 

`ResourceBounds::is_zero()` returns `true` when both `max_amount == 0` and `max_price_per_unit == 0`: [2](#0-1) 

**The call path** in `api_impl.rs`:

```
estimate_fee(transactions) 
  → transactions.into_iter().map(|tx| tx.try_into())   // BroadcastedTransaction → ExecutableTransactionInput
  → exec_estimate_fee(executable_txns, ...)
``` [3](#0-2) 

The `BroadcastedTransaction::Invoke(InvokeTransaction::V3)` path converts `ResourceBoundsMapping` → `ValidResourceBounds` via the `From` impl above. The resulting `ValidResourceBounds::L1Gas` then drives `get_gas_vector_computation_mode()`:

```rust
pub fn get_gas_vector_computation_mode(&self) -> GasVectorComputationMode {
    match self {
        Self::AllResources(_) => GasVectorComputationMode::All,
        Self::L1Gas(_)        => GasVectorComputationMode::NoL2Gas,  // ← wrong for V3
    }
}
``` [4](#0-3) 

**The fee computation divergence** is concrete. `TransactionResources::to_gas_vector` and `ComputationResources::to_gas_vector` both branch on `GasVectorComputationMode`:

- `All`: computation cost → `GasVector::from_l2_gas(sierra_gas)` — reported as L2 gas, priced at `l2_gas_price`
- `NoL2Gas`: computation cost → `GasVector::from_l1_gas(sierra_gas_to_l1_gas_amount_round_up(...))` — folded into L1 gas, priced at `l1_gas_price` [5](#0-4) 

The regression test data confirms the modes produce materially different gas vectors for the same execution:

| Mode | l1_gas | l1_data_gas | l2_gas |
|------|--------|-------------|--------|
| `NoL2Gas` (kzg) | 21 544 | 2 720 | 0 |
| `All` (kzg) | 21 543 | 2 720 | 87 040 | [6](#0-5) 

The `All`-mode fee includes `87 040 * l2_gas_price` that is entirely absent from the `NoL2Gas` estimate. The `NoL2Gas` estimate only recovers `1 * l1_gas_price` extra (rounding), which is negligible compared to the L2 gas cost.

---

### Impact Explanation

A caller to `starknet_estimateFee` who submits a V3 invoke with `l2_gas = {max_amount: 0, max_price_per_unit: 0}` and `l1_data_gas = {max_amount: 0, max_price_per_unit: 0}` receives a fee estimate computed entirely in `NoL2Gas` mode. The returned `overall_fee` omits the L2 gas component (`87 040 * l2_gas_price` in the example above). A user who then submits the real transaction with non-zero L2/L1DataGas bounds will execute in `All` mode and incur a materially higher actual fee than the estimate indicated. The RPC returns no error or warning — the estimate looks authoritative.

This satisfies: **High — RPC execution, fee estimation, tracing, simulation, or pending view returns an authoritative-looking wrong value.**

---

### Likelihood Explanation

Any unprivileged caller can trigger this by omitting or zeroing `l2_gas` and `l1_data_gas` in a V3 invoke passed to `estimate_fee`. No special state, account, or privilege is required. The condition is a simple field value check with no other guard.

---

### Recommendation

The `From<ResourceBoundsMapping> for ValidResourceBounds` conversion should not silently downgrade a V3 transaction to `L1Gas` mode. For V3 transactions, always produce `AllResources`, even when `l2_gas` and `l1_data_gas` are zero:

```rust
impl From<ResourceBoundsMapping> for ValidResourceBounds {
    fn from(value: ResourceBoundsMapping) -> Self {
        Self::AllResources(AllResourceBounds {
            l1_gas: value.l1_gas,
            l1_data_gas: value.l1_data_gas,
            l2_gas: value.l2_gas,
        })
    }
}
```

The `L1Gas` variant should only be produced when deserializing pre-0.13.3 transaction formats that structurally cannot carry `l1_data_gas`/`l2_gas` fields, not as a runtime inference from zero values in a V3 payload.

            ),
        };

            panic!(
                "Computation resources to gas vector overflowed: tried to add {sierra_gas_cost:?} \

            )
        })
