### Title
Unauthorized Native Token Transfer via Bank Precompile `transfer` — Missing Caller Authorization Check - (File: `x/cronos/keeper/precompiles/bank.go`)

### Summary
The `transfer` method of the Bank precompile contract accepts a caller-supplied `sender` address but never verifies that `sender == contract.Caller()`. Any EVM contract can therefore invoke `bank.transfer(victim, attacker, amount)` and move native bank tokens (denom `"evm/<callerContract>"`) out of the victim's account without the victim's consent.

### Finding Description
In `x/cronos/keeper/precompiles/bank.go`, the `Run` function handles the `transfer` method as follows:

```go
sender := args[0].(common.Address)
recipient := args[1].(common.Address)
...
from := sdk.AccAddress(sender.Bytes())
to   := sdk.AccAddress(recipient.Bytes())
...
denom := EVMDenom(contract.Caller())          // "evm/0x<callerContract>"
...
bc.bankKeeper.SendCoins(ctx, from, to, sdk.NewCoins(amt))
```

`from` is taken directly from the ABI-decoded call argument, not from `contract.Caller()`. There is no guard of the form `if sender != contract.Caller() { return error }`. The denom is scoped to the calling contract (`EVMDenom(contract.Caller())`), so the tokens at risk are those with denom `"evm/0x<attackerContract>"` held by the victim. [1](#0-0) 

Compare this with the `mint`/`burn` branch, which also uses `contract.Caller()` for the denom but at least operates on a single address argument that the contract itself supplies — the `transfer` case uniquely allows the contract to drain a *third-party* address. [2](#0-1) 

The `IBankModule` Solidity interface exposes this as a public, payable function callable by any contract: [3](#0-2) 

The intended usage (shown in the integration test) is `bank.transfer(msg.sender, recipient, amount)` — the contract passes its own caller as `sender`. Nothing in the precompile enforces this. [4](#0-3) 

### Impact Explanation
**Critical — Unauthorized transfer of precompile-controlled assets.**

Any EVM contract can drain the native bank balance (denom `"evm/0x<contract>"`) of any address that holds those tokens, transferring them to an arbitrary recipient. This is a direct unauthorized balance change for precompile-controlled assets, matching the Critical impact tier.

### Likelihood Explanation
The attack requires two steps: (1) deploy a contract, and (2) have users hold that contract's native token. Step 1 is permissionless. Step 2 occurs naturally in any DeFi protocol that uses the bank precompile's `mint` to issue native tokens to users (e.g., as liquidity rewards or wrapped assets). Once users hold the tokens, the attacker can drain all balances in a single transaction.

### Recommendation
Add a caller-authorization check before executing the transfer:

```go
case TransferMethodName:
    ...
    sender := args[0].(common.Address)
    // Enforce that only the token-issuing contract can move tokens on behalf of sender,
    // OR that sender == contract.Caller() (the contract transfers only from itself).
    if sender != contract.Caller() {
        return nil, errors.New("sender must be the calling contract")
    }
    ...
```

Alternatively, model the method after ERC-20's `transferFrom` and require an explicit on-chain allowance from `sender` to `contract.Caller()` before proceeding.

### Proof of Concept

1. Attacker deploys `MaliciousToken` at address `0xMalicious`.
2. `MaliciousToken` calls `bank.mint(victim, 1000)` — victim now holds 1000 `"evm/0xMalicious"` native tokens.
3. Attacker calls `MaliciousToken.drain(victim)`, which internally executes:
   ```solidity
   IBankModule(0x64).transfer(victim, attacker_eoa, 1000);
   ```
4. The bank precompile executes `SendCoins(ctx, victim, attacker_eoa, 1000 "evm/0xMalicious")` with no authorization check.
5. Victim's entire native token balance is transferred to the attacker without any approval or signature from the victim. [5](#0-4)
