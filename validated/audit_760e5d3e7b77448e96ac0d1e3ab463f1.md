### Title
Malicious Participant Permanently Denies Signing/DKG by Poisoning a Single-Accept Slot in `recv_from_others` — (File: src/protocol/helpers.rs)

---

### Summary

`recv_from_others` enforces a strict "first-message-wins, slot-closed" policy per participant using `ParticipantCounter`. A single malicious participant can fill their slot with a malformed or cryptographically invalid message. Because the function propagates deserialization errors immediately via `?` with no skip-and-retry logic, the entire protocol round aborts. This permanently denies signing, DKG, reshare, refresh, and CKD for all honest parties, since the attacker can repeat the attack on every restart.

---

### Finding Description

`recv_from_others` in `src/protocol/helpers.rs` is the primary message-collection primitive used across every multi-round protocol in the library:

```
FROST signing coordinator  →  do_sign_coordinator_v1 / v2
CKD coordinator            →  do_ckd_coordinator
DKG / Reshare / Refresh    →  do_keyshare (rounds 1 and 5)
```

The function body is:

```rust
while !seen.full() {
    let (from, msg) = chan.recv(waitpoint).await?;   // ← ? propagates immediately
    if seen.put(from) {
        messages.push((from, msg));
    }
}
``` [1](#0-0) 

`ParticipantCounter::put` returns `false` for any second message from the same sender, so the slot is permanently closed after the first message is accepted. [2](#0-1) 

`chan.recv` (backed by `Comms::recv`) deserializes with `rmp_serde` and returns `ProtocolError::DeserializationError` on failure: [3](#0-2) 

**Attack path — deserialization abort:**
A malicious participant sends raw bytes that cannot be decoded as the expected type (e.g., `round1::SigningCommitments` in FROST signing). `chan.recv` returns an error; the `?` in `recv_from_others` propagates it immediately; the coordinator's `do_sign_coordinator_v1` aborts. [4](#0-3) 

**Attack path — cryptographic poisoning (no deserialization error):**
A malicious participant sends bytes that deserialize successfully but encode a random/invalid commitment. The coordinator accepts the commitment into `commitments_map`, builds a `SigningPackage` with it, broadcasts it to all participants, and then `aggregate()` fails because the malicious participant's signature share is inconsistent with the poisoned commitment: [5](#0-4) 

The same slot-poisoning pattern applies to CKD: [6](#0-5) 

And to DKG secret-share collection (round 5): [7](#0-6) 

There is no skip-bad-message, retry, or exclusion mechanism in `recv_from_others`. Once the protocol aborts, the entire session must be restarted — and the attacker can repeat the attack on every restart.

---

### Impact Explanation

**High — Permanent denial of signing, key generation, reshare, refresh, and CKD for honest parties.**

A single malicious participant within the protocol set can abort any signing or key-management session at will, indefinitely. Because the library provides no Byzantine fault tolerance outside of the echo-broadcast rounds (used only for commitment/proof-of-knowledge broadcast in DKG), even one bad actor among N participants is sufficient to halt all threshold operations permanently. [8](#0-7) 

---

### Likelihood Explanation

**Medium.** The attacker must hold a legitimate participant identity within the protocol set (authenticated channel assumption is satisfied). Any one of the N participants can execute this attack with a single malformed or randomly-crafted message at the correct waitpoint. No cryptographic material needs to be compromised. The attack is trivially repeatable on every protocol restart.

---

### Recommendation

1. **In `recv_from_others`**, catch deserialization errors per-message and skip the offending sender rather than aborting the entire round. Log or record the misbehaving participant for exclusion.
2. **For FROST signing commitment collection**, validate each received `SigningCommitments` value before inserting it into `commitments_map` (e.g., check that the commitment elements are valid curve points).
3. **Consider extending echo-broadcast coverage** to the signing commitment and signature-share collection rounds, providing the same Byzantine fault tolerance already present in DKG.
4. **Expose a participant-exclusion API** so the caller can restart the protocol without the identified malicious participant.

---

### Proof of Concept

```
Setup: N=3 participants {P1 (coordinator), P2 (honest), P3 (malicious)}, threshold=2.

1. All three call sign_v1(...) to start a FROST signing session.
2. P2 sends a valid round1::SigningCommitments to P1 at commit_waitpoint.
3. P3 sends 16 bytes of random garbage to P1 at commit_waitpoint.
4. P1's recv_from_others calls chan.recv(commit_waitpoint):
     - Receives P2's message → deserialized OK, slot P2 closed.
     - Receives P3's message → rmp_serde fails → ProtocolError::DeserializationError.
     - The `?` operator propagates the error out of recv_from_others.
5. do_sign_coordinator_v1 returns Err(ProtocolError::DeserializationError).
6. P1 (coordinator) cannot produce a signature. P2 and P3 are left waiting forever.
7. On restart, P3 repeats step 3. Signing is permanently denied.
``` [9](#0-8) [10](#0-9)

pub async fn recv_from_others<T>(
    chan: &SharedChannel,
    waitpoint: u64,
    participants: &ParticipantList,
    me: Participant,
) -> Result<Vec<(Participant, T)>, ProtocolError>
where
    T: serde::de::DeserializeOwned,
{
    let mut seen = ParticipantCounter::new(participants);
    seen.put(me);
    let mut messages = Vec::with_capacity(participants.others(me).count());

    while !seen.full() {
        let (from, msg) = chan.recv(waitpoint).await?;
        if seen.put(from) {
            messages.push((from, msg));
        }
    }

    Ok(messages)

    async fn recv<T: DeserializeOwned>(
        &self,
        header: MessageHeader,
    ) -> Result<(Participant, T), ProtocolError> {
        let (from, data) = self.incoming.pop(header).await;
        let message_data = data.get(MessageHeader::LEN..).ok_or_else(|| {
            ProtocolError::DeserializationError("Failed to deserialize message data".to_string())
        })?;
        let decoded: Result<T, Box<dyn error::Error + Send + Sync>> =

        Ok((from, decoded?))
