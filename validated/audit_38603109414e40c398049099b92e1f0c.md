### Title
ETH Sent to `tick()` Is Permanently Locked When No Minting Is Required - (File: `L1/starkware/solidity/stake/RewardSupplier.sol`)

---

### Summary

`RewardSupplier.sol`'s `tick()` function is `external payable` but contains no refund path for `msg.value` when `amountToMint == 0`. Any ETH forwarded by the caller in that branch is permanently locked in the contract, which has no ETH-withdrawal mechanism. This is the direct structural analog of the Linea bug: a cross-chain fee is consumed without being returned to the payer when the intended work does not occur.

---

### Finding Description

`tick()` is the sole entry point for triggering the L1 minting and bridging cycle:

```solidity
function tick() external payable {          // line 107
    (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

    if (amountToMint > 0) {
        // ... consume L2 messages, mint tokens ...
        uint256 msgFee = msg.value / 2;
        bridge().depositWithMessage{value: msgFee}(...);   // line 126

        msgFee = msg.value - msgFee;
        messagingContract().sendMessageToL2{value: msgFee}(...); // line 137
    }
    // ← no else branch; msg.value is silently retained when amountToMint == 0
}
``` [1](#0-0) 

There are two distinct loss paths:

**Path A – `amountToMint == 0`:** `requiredMinting()` returns zero whenever there are no pending L2→L1 mint-request messages. The `if` body is skipped entirely. `msg.value` is never forwarded and never refunded; it accumulates in the contract permanently.

**Path B – `amountToMint > 0` with excess ETH:** The entire `msg.value` is split 50/50 between the two downstream calls. If the caller supplies more ETH than the StarkNet messaging contract's minimum fee for either message, the surplus is consumed by those contracts (which also do not refund). The `RewardSupplier` itself has no mechanism to return any remainder.

The contract inherits `RewardSupplierStorage`, `Identity`, and `ProxySupportImpl`. None of these introduce a `receive()` fallback or an ETH-withdrawal function. [2](#0-1) 

The `requiredMinting()` view function confirms that zero is returned whenever `l2ToL1Messages(msgHash) == 0`, which is the steady-state condition between reward epochs. [3](#0-2) 

---

### Impact Explanation

Any ETH sent to `tick()` when no minting is pending is permanently frozen inside `RewardSupplier`. Because the contract exposes no withdrawal or sweep function, recovery requires a contract upgrade. This constitutes **permanent freezing of funds** for the caller. Under the allowed impact scope this maps to **High – Temporary/Permanent freezing of funds**.

---

### Likelihood Explanation

`tick()` carries no access control; any address may call it. Keeper bots, MEV searchers, or protocol integrators are expected to call it periodically and must attach ETH to cover the two L1→L2 message fees. Between reward epochs — the majority of wall-clock time — `requiredMinting()` returns zero, so every such call with non-zero `msg.value` triggers Path A. The condition is not edge-case; it is the normal operating state of the contract.

---

### Recommendation

Add an explicit refund of unused ETH at the end of `tick()`:

```solidity
function tick() external payable {
    (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

    if (amountToMint > 0) {
        // ... existing logic ...
    }

    // Refund any unspent ETH to the caller.
    uint256 remaining = address(this).balance;
    if (remaining > 0) {
        (bool ok, ) = msg.sender.call{value: remaining}("");
        require(ok, "ETH_REFUND_FAILED");
    }
}
```

Alternatively, revert when `amountToMint == 0` and `msg.value > 0`, or make the function non-payable and require callers to pre-fund the contract separately.

---

### Proof of Concept

1. The L2 `RewardSupplier` has sent no mint-request messages to L1 (normal between epochs). `messagingContract().l2ToL1Messages(msgHash) == 0`.
2. A keeper calls `RewardSupplier.tick{value: 0.01 ether}()`.
3. `requiredMinting()` returns `(0, 0)`.
4. The `if (amountToMint > 0)` branch is skipped.
5. The function returns. `0.01 ether` now sits in `RewardSupplier`'s balance.
6. No function in the contract allows retrieval of that ETH. It is permanently locked. [1](#0-0)

    function tick() external payable {
        // Check if minting is required, and how much.
        (uint256 amountToMint, uint256 numMsgsToConsume) = requiredMinting();

        if (amountToMint > 0) {
            // Prepare the L2->L1 mintRequest message for consumption.
            uint256[] memory messagePayload = new uint256[](1);
            messagePayload[0] = TOKENS_PER_MINT_REQUEST;

            // Consume the mintRequest messages.
            for (uint256 i = 0; i < numMsgsToConsume; i++) {
                messagingContract().consumeMessageFromL2(mintRequestSource(), messagePayload);
            }

            // Reuest minting of the requested amount from the mint manager.
            mintManager().mintRequest(token(), amountToMint);

            // Deposit the minted amount onto the bridge to the credit of `mintDestination`.
            uint256 msgFee = msg.value / 2;
            bridge().depositWithMessage{value: msgFee}(
                token(),
                amountToMint,
                mintDestination(),
                new uint256[](0)
            );
            emit ConsumedL2MintRequests(numMsgsToConsume, amountToMint);

            // Send a totalSupply update to L2MintCurve.
            msgFee = msg.value - msgFee;
            messagePayload[0] = IERC20(token()).totalSupply();
            messagingContract().sendMessageToL2{value: msgFee}(
                mintingCurve(),
                UPDATE_TOTAL_SUPPLY_SELECTOR,
                messagePayload
            );
        }
    }
