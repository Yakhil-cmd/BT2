### Title
Missing Caller Restriction on `update_rewards` Allows Any Address to Permanently Freeze Staker Consensus Rewards - (`src/staking/staking.cairo`)

### Summary

The `update_rewards` function in the Staking contract is documented as callable only by the Starkware sequencer, but the implementation contains no such access control check. Any unprivileged address can call it with `disable_rewards: true` to consume the per-block reward slot without distributing rewards, permanently preventing stakers from receiving consensus-phase block rewards.

### Finding Description

The `update_rewards` function is the consensus-phase reward distribution entry point. Its spec explicitly states:

> **access control**: Only starkware sequencer.

However, the implementation only calls `general_prerequisites()`, which checks that the contract is not paused and the caller is non-zero — no sequencer identity check exists:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();   // only checks: not paused, caller != zero
    let current_block_number = starknet::get_block_number();
    assert!(
        current_block_number > self.last_reward_block.read(),
        "{}",
        Error::REWARDS_ALREADY_UPDATED,
    );
    // ... validates staker ...
    self.last_reward_block.write(current_block_number);  // <-- consumed for this block

    if disable_rewards || self.is_pre_consensus() {
        return;   // exits without distributing any rewards
    }
    // ... actual reward distribution ...
}
```

`last_reward_block` is a **single global** storage variable. Once it is written to the current block number, no further call to `update_rewards` can succeed in the same block (the `REWARDS_ALREADY_UPDATED` guard fires). An attacker who calls `update_rewards(any_valid_staker, disable_rewards: true)` before the sequencer in any block:

1. Passes all validation (staker exists, has non-zero balance, block is new).
2. Writes `last_reward_block = current_block_number`.
3. Returns immediately without distributing rewards (because `disable_rewards == true`).
4. The legitimate sequencer call in the same block reverts with `REWARDS_ALREADY_UPDATED`.

By repeating this every block, an attacker can permanently prevent all consensus-phase reward distribution at the cost of one transaction fee per block.

### Impact Explanation

This is **permanent freezing of unclaimed yield**. Stakers and delegators in the consensus rewards phase (`!is_pre_consensus()`) accumulate zero `unclaimed_rewards_own` and zero pool rewards for every block the attacker front-runs. Since `last_reward_block` is global, a single attacker transaction per block starves the entire protocol of reward distribution. The stakers' principal is not at risk, but all earned consensus block rewards are permanently lost.

### Likelihood Explanation

The entry point is fully public — any non-zero address can call it. The attacker only needs to know any valid active staker address (trivially observable on-chain from `NewStaker` events). The cost is one Starknet transaction fee per block. The attack is economically rational for a competitor or griefing actor because the damage (all stakers lose rewards) far exceeds the attacker's cost.

### Recommendation

Add an access control check at the top of `update_rewards` that restricts the caller to the authorized sequencer address (or a designated role). For example, add a stored `sequencer_address` and assert:

```cairo
assert!(
    get_caller_address() == self.sequencer_address.read(),
    "{}",
    Error::CALLER_IS_NOT_SEQUENCER,
);
```

Alternatively, use the existing `RolesComponent` to define a `SEQUENCER_ROLE` and gate the function on it, consistent with how other privileged functions (e.g., `pause`, `set_min_stake`) are protected.

### Proof of Concept

1. Consensus rewards are active (`!is_pre_consensus()`).
2. Attacker observes a valid staker address `S` from on-chain events.
3. At the start of block `N`, attacker submits: `update_rewards(S, disable_rewards: true)`.
4. Transaction succeeds: `last_reward_block` is set to `N`, no rewards distributed.
5. Sequencer's legitimate `update_rewards(S, disable_rewards: false)` in block `N` reverts with `REWARDS_ALREADY_UPDATED`.
6. Staker `S` (and all delegators) receive zero rewards for block `N`.
7. Attacker repeats every block → stakers permanently receive zero consensus rewards.
