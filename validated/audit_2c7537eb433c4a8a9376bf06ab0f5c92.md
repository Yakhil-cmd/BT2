### Title
Missing Caller Guard on `update_rewards` Allows Any Address to Permanently Suppress Block Rewards via Global `last_reward_block` — (`src/staking/staking.cairo`)

### Summary

`update_rewards` is callable by any address. The spec mandates "Only starkware sequencer" access control, but the implementation enforces none. Because `last_reward_block` is a single contract-wide slot, the first call per block wins and locks out all subsequent calls. An attacker who front-runs every block with `disable_rewards: true` permanently zeroes out a staker's yield accumulation.

### Finding Description

The spec at `docs/spec.md` line 1644–1645 states:

> **access control**: Only starkware sequencer.

The implementation at `src/staking/staking.cairo` lines 1449–1507 only calls `self.general_prerequisites()` (a pause check) and performs no caller identity check: [1](#0-0) 

The `last_reward_block` guard is **global** — one slot for the entire contract, not per-staker: [2](#0-1) 

When `disable_rewards` is `true`, the function still writes `last_reward_block = current_block` and returns early, skipping all reward math: [3](#0-2) 

Any subsequent call in the same block — including the legitimate sequencer call — reverts with `REWARDS_ALREADY_UPDATED`. [4](#0-3) 

The spec documents this error condition as a pre-condition guard, confirming the intent that only one call per block is allowed: [5](#0-4) 

### Impact Explanation

An attacker who calls `update_rewards(any_valid_active_staker, disable_rewards: true)` as the first transaction in every block:

1. Passes the `REWARDS_ALREADY_UPDATED` check (first call in block).
2. Writes `last_reward_block = current_block`.
3. Returns early — zero rewards distributed.
4. Blocks the legitimate sequencer call for that block.

Repeated across all blocks, the targeted staker accumulates zero `unclaimed_rewards_own` indefinitely. This matches the **High** impact category: *Permanent freezing of unclaimed yield*.

### Likelihood Explanation

- No privileged access required — any EOA or contract can call `update_rewards`.
- The attacker only needs to supply any currently-active staker address with non-zero balance (publicly queryable via `get_stakers`).
- On Starknet, transaction ordering within a block is sequencer-controlled, but the attacker can submit the griefing transaction at the start of every block at negligible cost (no token transfer required).
- The attack is sustainable indefinitely.

### Recommendation

Add a sequencer-only caller check at the top of `update_rewards`, consistent with the spec:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
    assert!(
        get_caller_address() == self.sequencer_address.read(),
        "{}",
        Error::ONLY_SEQUENCER,
    );
    // ... rest of function
}
```

Alternatively, if the sequencer address is not stored on-chain, use Starknet's `get_sequencer_address()` syscall.

### Proof of Concept

```
Block N:
  tx1 (attacker): update_rewards(staker_A, disable_rewards=true)
    → last_reward_block = N, returns early, no rewards
  tx2 (sequencer): update_rewards(staker_A, disable_rewards=false)
    → REVERTS: REWARDS_ALREADY_UPDATED

Block N+1:
  tx1 (attacker): update_rewards(staker_A, disable_rewards=true)
    → last_reward_block = N+1, returns early, no rewards
  tx2 (sequencer): update_rewards(staker_A, disable_rewards=false)
    → REVERTS: REWARDS_ALREADY_UPDATED

... repeated for all blocks ...

staker_A.unclaimed_rewards_own == 0 forever
```

The test suite itself confirms that a second same-block call always reverts: [6](#0-5) 

And that `disable_rewards: true` with consensus active produces zero rewards: [7](#0-6)
