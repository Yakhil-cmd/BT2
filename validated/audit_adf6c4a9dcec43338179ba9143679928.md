### Title
`vm_resources_to_execution_resources` returns `UnknownBuiltin` for valid `range_check96`/`add_mod`/`mul_mod` builtins, breaking `starknet_simulateTransactions` and `starknet_traceTransaction` for any transaction using modular-arithmetic opcodes — (`crates/apollo_rpc_execution/src/objects.rs`)

---

### Summary

The `vm_resources_to_execution_resources` conversion function in the RPC execution layer does not handle three Cairo VM builtins — `range_check96`, `add_mod`, and `mul_mod` — that blockifier fully supports and can produce from successfully executed transactions. Any transaction whose execution trace contains a non-zero count for any of these builtins causes the function to return `Err(ExecutionError::UnknownBuiltin)`, which propagates up through `FunctionInvocation::try_from` and causes `starknet_simulateTransactions` and `starknet_traceTransaction` to return an error instead of a valid trace.

---

### Finding Description

In `vm_resources_to_execution_resources` (lines 363–401), the match over `BuiltinName` variants explicitly handles `output`, `pedersen`, `range_check`, `ecdsa`, `bitwise`, `ec_op`, `keccak`, `poseidon`, and `segment_arena`. The three builtins introduced for modular arithmetic — `range_check96`, `add_mod`, and `mul_mod` — are commented out with a `TODO(DanB)` note and fall through to the wildcard arm:

```rust
// TODO(DanB): what about the following?
// BuiltinName::range_check96 => todo!(),
// BuiltinName::add_mod => todo!(),
// BuiltinName::mul_mod => todo!(),
_ => {
    return Err(ExecutionError::UnknownBuiltin { builtin_name });
}
``` [1](#0-0) 

These three builtins are not hypothetical. Blockifier's bouncer, versioned constants, and entry-point execution code all reference them as first-class builtins that can appear in `VmExecutionResources.builtin_instance_counter` after a successful execution. [2](#0-1) 

The conversion function is called unconditionally from `FunctionInvocation::try_from((CallInfo, GasVector))` at line 352, which is the sole path for building the `FunctionInvocation` objects that populate every simulation and trace response:

```rust
execution_resources: vm_resources_to_execution_resources(
    call_info.resources.vm_resources,
    gas_vector,
)?,
``` [3](#0-2) 

Because the `?` operator propagates the error, a single inner call that uses `add_mod` or `mul_mod` (e.g., a contract performing modular multiplication) causes the entire `FunctionInvocation` tree — and therefore the entire simulation or trace response — to fail.

---

### Impact Explanation

Any unprivileged user can deploy a Sierra contract that uses `add_mod`/`mul_mod` opcodes (standard Cairo modular-arithmetic instructions). When that contract is invoked:

- Blockifier executes it successfully and records non-zero `add_mod`/`mul_mod` builtin counts in `VmExecutionResources`.
- `starknet_simulateTransactions` and `starknet_traceTransaction` both call `FunctionInvocation::try_from`, hit `UnknownBuiltin`, and return an error to the caller.
- The RPC response is an error, not a valid trace, even though the transaction executed and was accepted on-chain.

This violates the invariant that any successfully executed transaction can be simulated or traced via the RPC layer. Wallets, block explorers, and dApps relying on simulation for fee estimation or trace inspection receive a spurious error for a class of valid transactions.

Impact category: **High — RPC execution, tracing, simulation returns an authoritative-looking wrong value.**

---

### Likelihood Explanation

`add_mod` and `mul_mod` are used by any Cairo contract performing modular arithmetic (e.g., elliptic-curve operations, ZK-friendly hash functions). These are not exotic; they are standard Sierra/CASM builtins available to any contract author. The trigger requires no privileges — just deploying and invoking such a contract.

---

### Recommendation

Map the three missing builtins to their corresponding `Builtin` enum variants (or to a `continue` if they are intentionally excluded from the RPC resource summary), and remove the `TODO` comment:

```rust
BuiltinName::range_check96 => builtin_instance_counter.insert(Builtin::RangeCheck96, count),
BuiltinName::add_mod => builtin_instance_counter.insert(Builtin::AddMod, count),
BuiltinName::mul_mod => builtin_instance_counter.insert(Builtin::MulMod, count),
```

If the `Builtin` enum in the RPC types does not yet have these variants, they should be added. Alternatively, if these builtins are intentionally omitted from the public resource summary, replace the `_` arm with an explicit `continue` for each, so that future unknown builtins still trigger the error.

---

### Proof of Concept

```rust
#[test]
fn test_unknown_builtin_range_check96() {
    use cairo_vm::types::builtin_name::BuiltinName;
    use cairo_vm::vm::runners::cairo_runner::ExecutionResources;
    use std::collections::HashMap;

    let mut counters = HashMap::new();
    counters.insert(BuiltinName::range_check96, 1usize);

    let vm_resources = ExecutionResources {
        n_steps: 100,
        n_memory_holes: 0,
        builtin_instance_counter: counters,
    };

    let gas = GasVector::default();
    let result = vm_resources_to_execution_resources(vm_resources, gas);
    assert!(
        matches!(result, Err(ExecutionError::UnknownBuiltin { .. })),
        "Expected UnknownBuiltin error for range_check96"
    );
}
```

This test will pass against the current code, confirming that a transaction whose execution uses `range_check96` (or `add_mod`/`mul_mod`) causes the simulation/trace conversion to fail. [4](#0-3)

fn vm_resources_to_execution_resources(
    vm_resources: VmExecutionResources,
    GasVector { l1_gas, l1_data_gas, l2_gas }: GasVector,
) -> ExecutionResult<ExecutionResources> {
    let mut builtin_instance_counter = HashMap::new();
    for (builtin_name, count) in vm_resources.builtin_instance_counter {
        if count == 0 {
            continue;
        }
        let count = u64_from_usize(count);
        match builtin_name {
            BuiltinName::output => continue,
            BuiltinName::pedersen => builtin_instance_counter.insert(Builtin::Pedersen, count),
            BuiltinName::range_check => builtin_instance_counter.insert(Builtin::RangeCheck, count),
            BuiltinName::ecdsa => builtin_instance_counter.insert(Builtin::Ecdsa, count),
            BuiltinName::bitwise => builtin_instance_counter.insert(Builtin::Bitwise, count),
            BuiltinName::ec_op => builtin_instance_counter.insert(Builtin::EcOp, count),
            BuiltinName::keccak => builtin_instance_counter.insert(Builtin::Keccak, count),
            BuiltinName::poseidon => builtin_instance_counter.insert(Builtin::Poseidon, count),
            BuiltinName::segment_arena => {
                builtin_instance_counter.insert(Builtin::SegmentArena, count)
            }
            // TODO(DanB): what about the following?
            // BuiltinName::range_check96 => todo!(),
            // BuiltinName::add_mod => todo!(),
            // BuiltinName::mul_mod => todo!(),
            _ => {
                return Err(ExecutionError::UnknownBuiltin { builtin_name });
            }
        };
    }
    Ok(ExecutionResources {
        steps: u64_from_usize(vm_resources.n_steps),
        builtin_instance_counter,
        memory_holes: u64_from_usize(vm_resources.n_memory_holes),
        da_gas_consumed: StarknetApiGasVector { l1_gas, l2_gas, l1_data_gas },
        gas_consumed: StarknetApiGasVector::default(),
    })
}
