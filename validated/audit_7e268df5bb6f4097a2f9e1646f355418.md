### Title
Missing Symmetric Guard in `assert_reshare_keys_invariants` Allows New Participant with Non-None Signing Key to Pass Pre-flight Validation but Fail Mid-Protocol — (`src/dkg.rs`)

---

### Summary

`assert_reshare_keys_invariants` enforces only one direction of the participant-vs-signing-key invariant: it rejects an *old* participant who supplies no key, but it does not reject a *new* participant who supplies a key. The inline comment at line 662 explicitly describes the missing check, yet the code never implements it. A caller who passes this validation function then proceeds to `do_reshare`, which fails with a `ProtocolError::InvalidIndex` before sending any protocol messages, silently stalling every other honest participant that is waiting for messages from the failing node.

---

### Finding Description

In `src/dkg.rs`, `assert_reshare_keys_invariants` performs the following guard:

```rust
// if me is not in the old participant set then ensure that old_signing_key is None
if old_participants.contains(me) && old_signing_key.is_none() {
    return Err(InitializationError::BadParameters(format!(
        "party {me:?} is present in the old participant list but provided no share"
    )));
}
``` [1](#0-0) 

The comment describes a bidirectional invariant — "if `me` is NOT in the old participant set then ensure that `old_signing_key` is None" — but the code only checks the opposite direction. The symmetric guard:

```rust
if !old_participants.contains(me) && old_signing_key.is_some() {
    return Err(InitializationError::BadParameters(...));
}
```

is entirely absent. `assert_reshare_keys_invariants` therefore returns `Ok` for a new participant (not in `old_participants`) who supplies a non-`None` `old_signing_key`.

When the caller subsequently invokes `do_reshare`, the following computation runs:

```rust
let intersection = old_participants.intersection(&participants);
let secret = old_signing_key
    .map(|x_i| {
        intersection
            .lagrange::<C>(me)
            .map(|lambda| lambda * x_i.to_scalar())
    })
    .transpose()?
    .unwrap_or_else(<C::Group as Group>::Field::zero);
``` [2](#0-1) 

Because `me` is not in `old_participants`, it is not in `intersection`. `intersection.lagrange::<C>(me)` returns `Err(ProtocolError::InvalidIndex)`, which `.transpose()?` propagates immediately. `do_reshare` exits before ever calling `do_keyshare`, so no protocol messages are sent. [3](#0-2) 

The inconsistency is structurally identical to the reference report: the pre-flight validation function (`assert_reshare_keys_invariants`) accepts inputs that the actual execution path (`do_reshare`) rejects, and the two functions diverge on exactly the condition the comment says should be enforced.

---

### Impact Explanation

Every other participant in the reshare session is blocked waiting for round-1 broadcast messages from the failing node (the `do_broadcast` call inside `do_keyshare`). Because the failing participant exits before sending anything, those participants hang indefinitely (or until an external timeout). The reshare protocol cannot complete, and the key material cannot be redistributed. This constitutes **permanent denial of reshare for honest parties** under inputs that the library's own validation function declares valid. [4](#0-3) 

---

### Likelihood Explanation

The public `reshare` entry point (called as `reshare::<C>(...)` in tests) is expected to call `assert_reshare_keys_invariants` as its documented pre-flight check. A new participant who mistakenly passes their old key material — or a malicious participant who deliberately does so — will receive an `Ok` from the validation function and proceed to launch the protocol. The misleading `Ok` makes accidental misuse more likely, and deliberate abuse requires no privilege beyond being a participant in the reshare session. [5](#0-4) 

---

### Recommendation

Add the symmetric guard immediately after the existing check in `assert_reshare_keys_invariants`:

```rust
// if me is not in the old participant set then ensure that old_signing_key is None
if old_participants.contains(me) && old_signing_key.is_none() {
    return Err(InitializationError::BadParameters(format!(
        "party {me:?} is present in the old participant list but provided no share"
    )));
}
// NEW: symmetric guard — new participant must not supply a signing key
if !old_participants.contains(me) && old_signing_key.is_some() {
    return Err(InitializationError::BadParameters(format!(
        "party {me:?} is not present in the old participant list but provided a share"
    )));
}
```

This makes the validation function consistent with what `do_reshare` actually requires, matching the invariant already described in the inline comment. [6](#0-5) 

---

### Proof of Concept

1. Run a keygen with participants `[A, B]`, threshold 2.
2. Introduce a new participant `C` (not in old set).
3. Call `assert_reshare_keys_invariants` with `me = C`, `old_signing_key = Some(<any share>)`, `old_participants = [A, B]`, new participants `[A, B, C]`, threshold 2.
4. Observe: function returns `Ok(...)` — no error.
5. Call `do_reshare` (or the public `reshare`) with the same arguments.
6. Observe: `intersection.lagrange::<C>(C)` returns `Err(ProtocolError::InvalidIndex)`; `do_reshare` exits before sending any messages.
7. Participants `A` and `B` block indefinitely on `do_broadcast` waiting for `C`'s round-1 message. [2](#0-1)

pub async fn do_reshare<C: Ciphersuite>(
    chan: SharedChannel,
    participants: ParticipantList,
    me: Participant,
    threshold: impl Into<ReconstructionLowerBound>,
    old_signing_key: Option<SigningShare<C>>,
    old_public_key: VerifyingKey<C>,
    old_participants: ParticipantList,
    mut rng: impl CryptoRngCore,
) -> Result<KeygenOutput<C>, ProtocolError> {
    let threshold = threshold.into();
    let intersection = old_participants.intersection(&participants);
    // either extract the share and linearize it or set it to zero
    let secret = old_signing_key
        .map(|x_i| {
            intersection
                .lagrange::<C>(me)
                .map(|lambda| lambda * x_i.to_scalar())
        })
        .transpose()?
        .unwrap_or_else(<C::Group as Group>::Field::zero);

    let old_reshare_package = Some((old_public_key, old_participants));
    let keygen_output = do_keyshare::<C>(
        chan,
        participants,
        me,
        threshold,
        secret,
        old_reshare_package,
        &mut rng,
    )
    .await?;

    Ok(keygen_output)
}

pub fn assert_reshare_keys_invariants<C: Ciphersuite>(
    participants: &[Participant],
    me: Participant,
    threshold: impl Into<ReconstructionLowerBound>,
    old_signing_key: Option<SigningShare<C>>,
    old_threshold: impl Into<ReconstructionLowerBound>,
    old_participants: &[Participant],
) -> Result<(ParticipantList, ParticipantList), InitializationError> {
    let threshold = usize::from(threshold.into());
    let old_threshold = usize::from(old_threshold.into());

    let participants = assert_key_invariants(participants, me, threshold)?;

    let old_participants =
        ParticipantList::new(old_participants).ok_or(InitializationError::DuplicateParticipants)?;

    // Step 1.1
    if old_participants.intersection(&participants).len() < old_threshold {
        return Err(InitializationError::NotEnoughParticipantsForNewThreshold {
            threshold: old_threshold,
            participants: old_participants.intersection(&participants).len(),
        });
    }
    // Step 1.1
    // if me is not in the old participant set then ensure that old_signing_key is None
    if old_participants.contains(me) && old_signing_key.is_none() {
        return Err(InitializationError::BadParameters(format!(
            "party {me:?} is present in the old participant list but provided no share"
        )));
    }
    Ok((participants, old_participants))
}
