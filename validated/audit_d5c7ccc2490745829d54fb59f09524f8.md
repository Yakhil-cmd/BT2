### Title
Pool Admin Can Silently Overwrite Pending Price Provider Proposal, Resetting Timelock and Emitting Misleading Events — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`proposePoolPriceProvider` contains no guard against an already-pending proposal. The pool admin can call it repeatedly, silently replacing the pending oracle address and resetting `pendingPriceProviderExecuteAfter` to `block.timestamp + timelock` each time. The emitted `PoolPriceProviderChangeProposed` event always reports the **currently active** provider as `currentPriceProvider`, not the previously pending one, so a switch from pending-A to pending-B is invisible to any monitor that tracks the pending slot.

---

### Finding Description

`proposePoolPriceProvider` in `MetricOmmPoolFactory.sol` unconditionally overwrites both pending-provider storage slots:

```solidity
// MetricOmmPoolFactory.sol lines 487-490
uint256 executeAfter = block.timestamp + timelock;
pendingPriceProvider[pool] = newPriceProvider;          // no guard
pendingPriceProviderExecuteAfter[pool] = executeAfter;  // timelock reset
emit PoolPriceProviderChangeProposed(pool, current, newPriceProvider, executeAfter);
``` [1](#0-0) 

The `current` value in the event is derived from the **active** on-pool provider, not from `pendingPriceProvider[pool]`:

```solidity
// MetricOmmPoolFactory.sol lines 485-486
address mutableProvider = PoolStateLibrary._slot3(pool);
address current = mutableProvider != address(0) ? mutableProvider : p.immutablePriceProvider;
``` [2](#0-1) 

There is no `if (pendingPriceProvider[pool] != address(0)) revert ...` check anywhere in the function. [3](#0-2) 

Compare with `pausePool` and `unpausePool`, which both read the current pause level and revert on invalid transitions before acting: [4](#0-3) 

No analogous state-check exists for the price-provider proposal path.

---

### Impact Explanation

The timelock on price-provider rotation is the **sole on-chain protection** for LPs and traders against a sudden oracle swap by the pool admin (who is semi-trusted only within caps and timelocks). [5](#0-4) 

Because `proposePoolPriceProvider` can overwrite a pending proposal silently:

1. **Bait-and-switch oracle substitution**: Admin proposes a legitimate provider A. Users observe the event and do not exit. Just before `executeAfter_A`, admin re-proposes a malicious provider B. The new event shows `activeProvider → B` (not `A → B`), so monitors watching for "A being applied" see nothing alarming. The timelock resets to `now + timelock`. After the new delay, admin calls `executePoolPriceProviderUpdate`, installing provider B.

2. **Bad-price execution**: Provider B can return arbitrary bid/ask values that pass `_validatePriceProvider`'s only check (token pair match). A manipulated oracle causes swaps to execute at wrong prices, draining LP reserves or over-paying traders. [6](#0-5) 

3. **Fraudulent event log**: Multiple `PoolPriceProviderChangeProposed` events are emitted with the same `currentPriceProvider` field even though the pending slot changed, breaking any off-chain monitoring that relies on event history to reconstruct the pending state.

---

### Likelihood Explanation

- Trigger requires only the pool admin key — no external conditions, no special token behavior, no privileged factory-owner cooperation.
- The pool admin is explicitly described as semi-trusted "only inside caps and timelocks"; this path circumvents the timelock's protective intent.
- A pool with a non-zero, non-`type(uint256).max` `priceProviderTimelock` is the only prerequisite (immutable-oracle pools revert at the `PriceProviderImmutable` check). [7](#0-6) 

---

### Recommendation

Add a guard at the top of `proposePoolPriceProvider` that reverts when a proposal is already pending:

```solidity
if (pendingPriceProvider[pool] != address(0)) revert PriceProviderChangePending();
```

If intentional re-proposal is desired (e.g., to correct a mistake), require an explicit `cancelPoolPriceProviderProposal` call first — mirroring the `cancelPoolAdminTransfer` pattern already present for admin transfers — so that the cancellation is visible on-chain and the event log remains unambiguous. [8](#0-7) 

---

### Proof of Concept

```
Pool created with priceProviderTimelock = 7 days, mutable oracle mode.

T=0:   admin calls proposePoolPriceProvider(pool, providerA)
       → pendingPriceProvider[pool] = providerA
       → pendingPriceProviderExecuteAfter[pool] = T+7d
       → event: PoolPriceProviderChangeProposed(pool, activeProvider, providerA, T+7d)
       LPs see the event; providerA looks legitimate; no exits.

T=6d:  admin calls proposePoolPriceProvider(pool, providerMalicious)
       → pendingPriceProvider[pool] = providerMalicious   ← silent overwrite
       → pendingPriceProviderExecuteAfter[pool] = T+13d   ← timelock reset
       → event: PoolPriceProviderChangeProposed(pool, activeProvider, providerMalicious, T+13d)
       Monitors watching for "providerA applied" see nothing alarming.
       Monitors watching for new proposals may miss the second event.

T=13d: admin calls executePoolPriceProviderUpdate(pool)
       → pool.setPriceProvider(providerMalicious)
       → providerMalicious.getBidAndAskPrice() returns manipulated bid/ask
       → swaps execute at wrong prices; LP reserves drained.
```

`executePoolPriceProviderUpdate` re-validates the pending provider against token pair only, not against any price sanity bound, so a provider returning extreme quotes passes: [9](#0-8)

  function executePoolPriceProviderUpdate(address pool) external override nonReentrant onlyPoolAdmin(pool) {
    address pending = pendingPriceProvider[pool];
    if (pending == address(0)) revert NoPriceProviderChangeProposed();
    uint256 execAfter = pendingPriceProviderExecuteAfter[pool];
    // forge-lint: disable-next-line(block-timestamp) -- timelock enforcement legitimately relies on `block.timestamp`.
    if (block.timestamp < execAfter) revert PriceProviderTimelockNotElapsed(execAfter, block.timestamp);
    PoolImmutables memory p = IMetricOmmPool(pool).getImmutables();
    if (p.immutablePriceProvider != address(0)) revert PriceProviderImmutable();
    _validatePriceProvider(p.token0, p.token1, pending);
    IMetricOmmPoolFactoryActions(pool).setPriceProvider(pending);
    delete pendingPriceProvider[pool];
    delete pendingPriceProviderExecuteAfter[pool];
    emit PoolPriceProviderUpdated(pool, pending);
  }
```
