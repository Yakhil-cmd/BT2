### Title
Broken EVM Bridge Accessor Permanently Locks User-Bridged NFTs and Fungible Tokens — (`fvm/evm/stdlib/contract.cdc`)

---

### Summary

The `CadenceOwnedAccount.withdrawNFT` and `CadenceOwnedAccount.withdrawTokens` functions unconditionally route through `EVM.borrowBridgeAccessor()`, which panics if the bridge router resource is missing or the capability is invalid. Users who have already bridged NFTs or fungible tokens into EVM via their COA have no alternative withdrawal path if the bridge accessor contract becomes non-functional. There is no emergency exit.

---

### Finding Description

In `fvm/evm/stdlib/contract.cdc`, the `CadenceOwnedAccount` resource exposes two functions for withdrawing bridged assets from EVM back to Cadence:

**`withdrawNFT`** (lines 756–770):
```cadence
access(Owner | Bridge)
fun withdrawNFT(
    type: Type,
    id: UInt256,
    feeProvider: auth(FungibleToken.Withdraw) &{FungibleToken.Provider}
): @{NonFungibleToken.NFT} {
    pre {
        !EVM.isPaused(): "EVM operations are temporarily paused"
    }
    return <- EVM.borrowBridgeAccessor().withdrawNFT(
        caller: &self as auth(Call) &CadenceOwnedAccount,
        type: type,
        id: id,
        feeProvider: feeProvider
    )
}
```

**`withdrawTokens`** (lines 788–802):
```cadence
access(Owner | Bridge)
fun withdrawTokens(
    type: Type,
    amount: UInt256,
    feeProvider: auth(FungibleToken.Withdraw) &{FungibleToken.Provider}
): @{FungibleToken.Vault} {
    pre {
        !EVM.isPaused(): "EVM operations are temporarily paused"
    }
    return <- EVM.borrowBridgeAccessor().withdrawTokens(
        caller: &self as auth(Call) &CadenceOwnedAccount,
        type: type,
        amount: amount,
        feeProvider: feeProvider
    )
}
```

Both functions call `EVM.borrowBridgeAccessor()`, defined at lines 1184–1189:

```cadence
access(self)
view fun borrowBridgeAccessor(): auth(Bridge) &{BridgeAccessor} {
    return self.account.storage.borrow<auth(Bridge) &{BridgeRouter}>(from: /storage/evmBridgeRouter)
        ?.borrowBridgeAccessor()
        ?? panic("EVM.borrowBridgeAccessor(): Could not borrow a reference to the EVM bridge.")
}
```

If the `BridgeRouter` resource is absent from `/storage/evmBridgeRouter`, or if the stored `BridgeAccessor` capability becomes invalid (e.g., the `FlowEVMBridgeAccessor` contract is removed, upgraded with a breaking change, or its capability is revoked), `borrowBridgeAccessor()` panics unconditionally. Every call to `withdrawNFT` or `withdrawTokens` reverts, and the user's bridged assets remain permanently locked in the EVM bridge escrow.

By contrast, native FLOW token withdrawal via `CadenceOwnedAccount.withdraw` (lines 587–606) calls `InternalEVM.withdraw()` directly and does **not** route through the bridge accessor, so it is unaffected. [1](#0-0) [2](#0-1) [3](#0-2) 

---

### Impact Explanation

Any user who has bridged NFTs or fungible tokens into EVM via their `CadenceOwnedAccount` (using `depositNFT` or `depositTokens`) is exposed. If the bridge accessor becomes non-functional, those assets are permanently locked in the EVM bridge escrow with no recovery mechanism. This is a **cross-VM asset loss** scenario: the assets exist in EVM state but cannot be returned to Cadence. There is no emergency exit, no direct EVM-to-Cadence bypass, and no governance-controlled recovery path for individual users. [4](#0-3) [5](#0-4) 

---

### Likelihood Explanation

The bridge accessor is a separate, upgradeable contract (`FlowEVMBridgeAccessor`) whose capability is stored in the EVM contract's storage. Several realistic scenarios can break it:

1. A bug in the `FlowEVMBridgeAccessor` contract causes it to panic on `withdrawNFT`/`withdrawTokens` calls.
2. A governance upgrade replaces the bridge accessor with an incompatible version, invalidating the stored capability.
3. The `BridgeRouter` resource is accidentally removed from `/storage/evmBridgeRouter` during a migration.

In all cases, the `borrowBridgeAccessor()` panic fires and every user withdrawal attempt reverts. The `BridgeAccessor` interface is defined as an external resource interface that any conforming contract can implement, making it a genuine external dependency. [6](#0-5) [7](#0-6) 

---

### Recommendation

Add an emergency withdrawal path on `CadenceOwnedAccount` that bypasses the bridge accessor and allows users to retrieve their bridged assets directly from the EVM state when the bridge accessor is unavailable. This path should:

- Be callable by the COA owner without requiring a functional bridge accessor.
- Directly interact with the EVM state to transfer the asset back to Cadence (analogous to how `CadenceOwnedAccount.withdraw` calls `InternalEVM.withdraw` directly for native FLOW).
- Be gated by a condition that verifies the bridge accessor is non-functional (e.g., `borrowBridgeAccessor()` returns `nil`), so it cannot be used to bypass bridge fees under normal operation.

---

### Proof of Concept

1. User calls `COA.depositNFT(nft: <-nft, feeProvider: feeProvider)` — NFT is bridged to EVM, locked in bridge escrow.
2. The `FlowEVMBridgeAccessor` contract is upgraded with a breaking change, or its capability is revoked, making `borrowBridgeAccessor()` return `nil`.
3. User calls `COA.withdrawNFT(type: nftType, id: nftID, feeProvider: feeProvider)`.
4. Execution reaches `EVM.borrowBridgeAccessor()` at line 1186–1188, which evaluates to `nil ?? panic(...)`.
5. The transaction panics with `"EVM.borrowBridgeAccessor(): Could not borrow a reference to the EVM bridge."` and reverts.
6. The NFT remains locked in EVM bridge escrow. The user has no alternative path to recover it. [8](#0-7) [9](#0-8)

```text
    /// Interface which captures a Capability to the bridge Accessor,
    /// saving it within the BridgeRouter resource
    access(all) resource interface BridgeRouter {

        /// Returns a reference to the BridgeAccessor designated
        /// for internal bridge requests
        access(Bridge) view fun borrowBridgeAccessor(): auth(Bridge) &{BridgeAccessor}

        /// Sets the BridgeAccessor Capability in the BridgeRouter
        access(Bridge) fun setBridgeAccessor(_ accessor: Capability<auth(Bridge) &{BridgeAccessor}>) {
            pre {
                accessor.check():
                    "EVM.setBridgeAccessor(): Invalid BridgeAccessor Capability provided"
            }
            post {
                emit BridgeAccessorUpdated(
                    routerType: self.getType(),
                    routerUUID: self.uuid,
                    routerAddress: self.owner?.address ?? panic("EVM.setBridgeAccessor(): Router must be stored in an account's storage"),
                    accessorType: accessor.borrow()!.getType(),
                    accessorUUID: accessor.borrow()!.uuid,
                    accessorAddress: accessor.address
                )
            }
        }
    }
```
