### Title
Missing Vec Length Validation in `do_generation_many` Allows Malicious Participant to Panic Honest Parties — (File: src/ecdsa/ot_based_ecdsa/triples/generation.rs)

### Summary
In `do_generation_many<const N: usize>`, every received protocol message that contains a `Vec` of N-indexed elements is accessed with a bare `[i]` index without first asserting `len >= N`. A single malicious participant can send any of these messages with an under-length `Vec`, triggering an index-out-of-bounds panic in every honest party's execution, permanently aborting the current triple-generation session and blocking OT-based ECDSA signing.

### Finding Description

`do_generation_many<const N: usize>` in `src/ecdsa/ot_based_ecdsa/triples/generation.rs` collects four distinct message types from remote participants, each carrying one or more `Vec` fields that are expected to have exactly `N` elements. In every case the code iterates `for i in 0..N` and indexes directly into the received `Vec` without a prior length check:

The `PolynomialCommitmentsMessageMany` struct itself (lines 97–106) declares all fields as unbounded `Vec`, and `PolynomialCommitment` deserialization (lines 382–391) imposes no upper bound on the coefficient count, so a well-formed msgpack payload with an empty (or short) `Vec` passes deserialization cleanly and reaches the indexing code.

The degree check that does exist (lines 343–351) fires only after the `[i]` index is already evaluated, so it cannot prevent the panic.

### Impact Explanation

A single malicious participant sends any of the four message types above with one or more `Vec` fields containing fewer than `N` elements (e.g., an empty `Vec`). Every honest party that processes that message panics with an index-out-of-bounds, aborting the entire `generate_triple_many` session. Because OT-based ECDSA presigning (`presign` in `src/ecdsa/ot_based_ecdsa/presign.rs`) requires pre-generated triples, the abort permanently blocks the current signing round for all honest participants. The attacker needs only to be a legitimate participant in the triple-generation session — no privileged access is required.

**Impact: High — Permanent denial of signing for honest parties under valid protocol inputs.**

### Likelihood Explanation

Any participant admitted to a `generate_triple_many` session can trigger this with a single crafted message. The attacker does not need to break any cryptographic primitive, forge a proof, or hold a key. The malformed message passes msgpack deserialization without error. Likelihood is **Medium** (requires participation in the triple-generation protocol, which is a normal operational role).

### Recommendation

Before the `for i in 0..N` loop, validate that every received `Vec` field has length exactly `N` and return a `ProtocolError::AssertionFailed` if not:

```rust
if their.big_e_v.len() != N
    || their.big_f_v.len() != N
    || their.big_l_v.len() != N
    || their.randomizer_v.len() != N
    || their.phi_proof0_v.len() != N
    || their.phi_proof1_v.len() != N
{
    return Err(ProtocolError::AssertionFailed(
        format!("PolynomialCommitmentsMessageMany from {from:?} has wrong Vec length")
    ));
}
```

Apply the same guard to the scalar-share, `C_i`, and `hat_C_i` message types. Additionally, add a maximum-length guard on `PolynomialCommitment` deserialization (analogous to the existing `MAX_APP_ID_LEN` guard in `src/confidential_key_derivation/app_id.rs`) to prevent memory exhaustion from oversized coefficient vectors before the degree check is reached.

### Proof of Concept

1. Honest parties call `generate_triple_many::<2>(participants, me, threshold, rng)`.
2. The malicious participant sends a `PolynomialCommitmentsMessageMany` message at `wait2` where `big_e_v = []` (empty Vec), while all other fields are valid.
3. Every honest party deserializes the message successfully (empty `Vec` is valid msgpack).
4. Each honest party enters `for i in 0..2` and evaluates `their.big_e_v[0]`, panicking with:
