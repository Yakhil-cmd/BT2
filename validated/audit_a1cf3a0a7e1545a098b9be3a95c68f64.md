### Title
Missing Access Control on `update_rewards` Allows Any Caller to Permanently Block Per-Block Reward Distribution — (`src/staking/staking.cairo`)

---

### Summary

`update_rewards` is gated only by a global `last_reward_block` check, not by a caller-identity check. Any unprivileged address can call it with `disable_rewards: true` to consume the block's reward slot without distributing any rewards, permanently preventing the sequencer from distributing rewards for that block.

---

### Finding Description

The spec explicitly states that `update_rewards` is callable by **"Only starkware sequencer"**. However, the implementation enforces no such restriction. [1](#0-0) 

The only guard is `general_prerequisites()`, which checks that the contract is unpaused and the caller is non-zero: [2](#0-1) 

After passing that check, the function immediately writes the current block number to the **global** `last_reward_block` storage variable: [3](#0-2) 

The subsequent reward distribution is then skipped entirely when `disable_rewards` is `true` or `is_pre_consensus()` is true: [4](#0-3) 

Because `last_reward_block` is a single global slot (not per-staker), once any caller writes block `N` into it, the assertion `current_block_number > self.last_reward_block.read()` will fail for every subsequent call in the same block — including the legitimate sequencer call: [5](#0-4) 

The spec documents this error as `REWARDS_ALREADY_UPDATED`: [6](#0-5) 

---

### Impact Explanation

When an attacker front-runs the sequencer in block `N` by calling `update_rewards(any_valid_staker, disable_rewards: true)`:

1. `last_reward_block` is set to `N`.
2. No rewards are calculated or distributed (the function returns early).
3. The sequencer's legitimate call for block `N` reverts with `REWARDS_ALREADY_UPDATED`.
4. Block `N`'s rewards are **permanently lost** — there is no mechanism to retroactively distribute rewards for a past block.

This matches the allowed impact: **Permanent freezing of unclaimed yield**.

---

### Likelihood Explanation

- The function is public with no caller restriction.
- The attacker only needs to know any valid, active staker address (trivially discoverable from on-chain events).
- The cost is a single transaction per block. An attacker can sustain this indefinitely to deny all consensus-era rewards to all stakers.
- No special privilege, leaked key, or external dependency is required.

---

### Recommendation

Add an access-control check at the top of `update_rewards` that asserts `get_caller_address() == sequencer_address`, consistent with the spec's stated access rule. The sequencer address should be stored in contract configuration (similar to how `attestation_contract` is stored and checked in `update_rewards_from_attestation_contract`).

---

### Proof of Concept

```
// Block N:
// Attacker (any address) calls:
staking.update_rewards(staker_address: any_active_staker, disable_rewards: true);
// → last_reward_block is now N, no rewards distributed.

// Sequencer then calls (same block N):
staking.update_rewards(staker_address: staker_A, disable_rewards: false);
// → PANICS: REWARDS_ALREADY_UPDATED
// Block N rewards for staker_A are permanently lost.
```

The test suite already demonstrates this exact revert path: [7](#0-6) 

The difference from the test is that in the test the *sequencer* makes the first call; in the attack the *attacker* makes the first call with `disable_rewards: true`, consuming the slot without distributing rewards.
