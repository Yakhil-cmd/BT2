### Title
Unchecked Precompile Return Value in `withdrawToNear` Allows Silent Token Burn with No Exit Promise — (`etc/eth-contracts/contracts/EvmErc20V2.sol`)

### Summary
Any unprivileged EVM user can call `withdrawToNear` with a zero-length `recipient` argument. The ERC-20 tokens are burned first; the precompile then rejects the empty `AccountId` and returns an error; the assembly `call` result is silently ignored; and the transaction succeeds with tokens permanently destroyed and no NEP-141 transfer promise ever created.

### Finding Description
`EvmErc20V2.withdrawToNear` performs `_burn(sender, amount)` unconditionally before invoking the `ExitToNear` precompile at `0xe9217bc7...`. The assembly block captures the call result in `res` but never checks it:

```solidity
// EvmErc20V2.sol lines 53-64
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    address sender = _msgSender();
    _burn(sender, amount);                          // (1) burn is irreversible

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", sender, amount_b, recipient);
    uint input_size = 1 + 20 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc7..., 0, add(input, 32), input_size, 0, 32)
        // res is NEVER checked — no require(res != 0)
    }
}
``` [2](#0-1) 

When `recipient.length == 0`, the 53-byte input passes `validate_input_size` (min is 3 or 21 depending on feature flag). The precompile then calls `parse_recipient(&[])`:

```rust
// native.rs lines 359-378
fn parse_recipient(recipient: &[u8]) -> Result<Recipient<'_>, ExitError> {
    let recipient = str::from_utf8(recipient)   // Ok("") for empty slice
        .map_err(...)?;
    ...
    Ok(Recipient {
        receiver_account_id: receiver_account_id
            .parse()                             // "".parse::<AccountId>() → Err
            .map_err(|_| ExitError::Other(Cow::from("ERR_INVALID_RECEIVER_ACCOUNT_ID")))?,
        ...
    })
}
``` [3](#0-2) 

The precompile returns `Err`, the EVM `CALL` opcode returns 0, `res` is discarded, and the Solidity function returns normally. No NEP-141 `ft_transfer` or `ft_transfer_call` promise is ever scheduled. [5](#0-4) 

### Impact Explanation
Tokens are permanently destroyed on the EVM side with no corresponding release on the NEAR/NEP-141 side. The total supply of the bridged token becomes insolvent: ERC-20 supply decreases while the NEP-141 custodial balance is unchanged. This is a direct, irreversible loss of user funds — **Critical: Insolvency**.

### Likelihood Explanation
The attack requires no privileges. Any holder of the ERC-20 token can call `withdrawToNear(bytes(""), amount)` directly. The function has no access control, no pause gate, and no minimum-length check on `recipient`. The same structural flaw exists in `EvmErc20.sol` `withdrawToNear` as well. [6](#0-5) 

### Recommendation
Add an explicit success check after the precompile call, and/or validate `recipient.length > 0` before burning:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    require(recipient.length > 0, "ERR_EMPTY_RECIPIENT");
    address sender = _msgSender();
    _burn(sender, amount);

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", sender, amount_b, recipient);
    uint input_size = 1 + 20 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        if iszero(res) { revert(0, 0) }
    }
}
```

The same fix must be applied to `EvmErc20.sol`.

### Proof of Concept
1. Deploy `EvmErc20V2` with a registered NEP-141 mapping.
2. Mint tokens to address `A`.
3. From `A`, call `withdrawToNear(bytes(""), 1e18)`.
4. Observe: `balanceOf(A)` decreased by `1e18`; no NEAR promise log was emitted; NEP-141 custodial balance is unchanged.
5. Repeat for any non-empty but invalid recipient (e.g., `bytes("@@@")`) to confirm the same silent-burn path via `ERR_INVALID_RECEIVER_ACCOUNT_ID`. [7](#0-6)
