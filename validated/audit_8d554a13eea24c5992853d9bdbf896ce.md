### Title
Bank Precompile `burn` and `transfer` Operate on Arbitrary Addresses Without Authorization — (`File: x/cronos/keeper/precompiles/bank.go`)

### Summary
The bank precompile's `burn` and `transfer` methods accept an arbitrary address as the source of funds and operate on it without any authorization check from that address. Any unprivileged EVM contract can burn or transfer its own-denom (`evm/<caller>`) tokens from any holder's account without consent.

### Finding Description
In `x/cronos/keeper/precompiles/bank.go`, the `MintMethodName` and `BurnMethodName` cases share a single code path. Both receive `recipient` (an arbitrary `common.Address`) as `args[0]`. For `mint` this is the destination, which is fine. For `burn`, however, `addr` (derived from `recipient`) is used as the **source** account from which tokens are pulled and destroyed: [1](#0-0) 

```go
recipient := args[0].(common.Address)
...
addr := sdk.AccAddress(recipient.Bytes())
...
denom := EVMDenom(contract.Caller())          // evm/<calling_contract>
...
// burn path:
bc.bankKeeper.SendCoinsFromAccountToModule(ctx, addr, ...)  // pulls from addr
bc.bankKeeper.BurnCoins(ctx, types.ModuleName, ...)
```

No check verifies that `contract.Caller()` is authorized to spend `addr`'s balance. The only guard present is `amount.Sign() <= 0` — the amount is validated, but the authorization over the source address is never validated. This is the direct analog to the external report: the code validates one variable (amount) while leaving the critical guard (caller ≡ addr, or an allowance) entirely absent.

The same flaw exists in `TransferMethodName`: [2](#0-1) 

```go
sender := args[0].(common.Address)
...
from := sdk.AccAddress(sender.Bytes())
...
denom := EVMDenom(contract.Caller())
...
bc.bankKeeper.SendCoins(ctx, from, to, ...)   // pulls from from, no auth check
```

Any contract can transfer its own-denom tokens out of any holder's account.

### Impact Explanation
- **Critical — Unauthorized burn/transfer of precompile-controlled assets.**
- A malicious contract first mints `evm/0xAttacker` tokens to users (e.g., as DeFi rewards or via a DEX). Once users hold a balance, the attacker calls `burn(victimAddress, balance)` or `transfer(victimAddress, attackerAddress, balance)` on the bank precompile, destroying or stealing the victim's tokens with no consent required.
- `SendCoinsFromAccountToModule` and `SendCoins` are authoritative Cosmos SDK bank operations; there is no EVM-level allowance or Cosmos-level approval that blocks this path.

### Likelihood Explanation
Any deployed EVM contract can reach the bank precompile at address `0x0000...0064` with a single low-cost call. No privileged key, governance vote, or special setup is required beyond having previously minted tokens to victims — a normal DeFi interaction.

### Recommendation
For `burn`: restrict the source address to `contract.Caller()` (i.e., only allow a contract to burn tokens from its own EVM address), or introduce an allowance/approval mechanism before pulling from an arbitrary address.

For `transfer`: require `sender == contract.Caller()`, or implement an ERC20-style `approve`/`transferFrom` allowance model enforced at the precompile level.

### Proof of Concept

1. Attacker deploys `MaliciousToken` at address `0xDEAD`.
2. `MaliciousToken` calls `bank.mint(victimEOA, 1000)` → victim now holds `1000 evm/0xDEAD`.
3. Attacker calls `bank.burn(victimEOA, 1000)` from `MaliciousToken` → `SendCoinsFromAccountToModule(ctx, victimEOA_sdk, ...)` succeeds with no authorization check → victim's balance is destroyed.
4. Alternatively, step 3 uses `bank.transfer(victimEOA, attackerEOA, 1000)` → victim's tokens are moved to the attacker.

The root cause is in `x/cronos/keeper/precompiles/bank.go` lines 113–155 (burn) and 167–200 (transfer): the amount is validated but the authorization over the source address is never checked. [3](#0-2)

func (bc *BankContract) Run(evm *vm.EVM, contract *vm.Contract, readonly bool) ([]byte, error) {
	// parse input
	methodID := contract.Input[:4]
	method, err := bankABI.MethodById(methodID)
	if err != nil {
		return nil, err
	}
	stateDB := evm.StateDB.(ExtStateDB)
	precompileAddr := bc.Address()
	switch method.Name {
	case MintMethodName, BurnMethodName:
		if readonly {
			return nil, errors.New("the method is not readonly")
		}
		args, err := method.Inputs.Unpack(contract.Input[4:])
		if err != nil {
			return nil, errors.New("fail to unpack input arguments")
		}
		recipient := args[0].(common.Address)
		amount := args[1].(*big.Int)
		if amount.Sign() <= 0 {
			return nil, errors.New("invalid amount")
		}
		addr := sdk.AccAddress(recipient.Bytes())
		if err := bc.checkBlockedAddr(addr); err != nil {
			return nil, err
		}
		denom := EVMDenom(contract.Caller())
		amt := sdk.NewCoin(denom, sdkmath.NewIntFromBigInt(amount))
		err = stateDB.ExecuteNativeAction(precompileAddr, nil, func(ctx sdk.Context) error {
			if err := bc.bankKeeper.IsSendEnabledCoins(ctx, amt); err != nil {
				return err
			}
			if method.Name == "mint" {
				if err := bc.bankKeeper.MintCoins(ctx, types.ModuleName, sdk.NewCoins(amt)); err != nil {
					return errorsmod.Wrap(err, "fail to mint coins in precompiled contract")
				}
				if err := bc.bankKeeper.SendCoinsFromModuleToAccount(ctx, types.ModuleName, addr, sdk.NewCoins(amt)); err != nil {
					return errorsmod.Wrap(err, "fail to send mint coins to account")
				}
			} else {
				if err := bc.bankKeeper.SendCoinsFromAccountToModule(ctx, addr, types.ModuleName, sdk.NewCoins(amt)); err != nil {
					return errorsmod.Wrap(err, "fail to send burn coins to module")
				}
				if err := bc.bankKeeper.BurnCoins(ctx, types.ModuleName, sdk.NewCoins(amt)); err != nil {
					return errorsmod.Wrap(err, "fail to burn coins in precompiled contract")
				}
			}
			return nil
		})
		if err != nil {
			return nil, err
		}
