### Title
Extension Revert Permanently Blocks `removeLiquidity` With No Skip Mechanism — (File: `metric-core/contracts/ExtensionCalling.sol`)

---

### Summary

The `_callExtensionsInOrder` function calls all registered extensions sequentially with no error-handling or skip mechanism. If any extension reverts for any reason after pool deployment, the entire pool operation fails. Because extensions are stored as constructor immutables and cannot be changed post-deployment, a broken extension permanently blocks `removeLiquidity` for every LP in the pool, locking their principal with no recovery path.

---

### Finding Description

`_callExtensionsInOrder` in `ExtensionCalling.sol` iterates through the packed extension-order word and calls each extension via `CallExtension.callExtension`:

```solidity
function _callExtensionsInOrder(uint256 order, bytes memory data) private {
    if (order == 0) return;
    while (true) {
        uint256 extensionIndex = order & 0x7;
        if (extensionIndex == 0) break;
        address extension = _extensionAddress(extensionIndex);
        if (extension == address(0)) revert PanicEmptyExtension();
        CallExtension.callExtension(extension, data);   // ← no try/catch
        order >>= 3;
    }
}
```

`CallExtension.callExtension` unconditionally bubbles any revert from the extension back to the caller:

```solidity
function callExtension(address extension, bytes memory data) internal {
    (bool success, bytes memory result) = extension.call(data);
    if (!success) {
        if (result.length > 0) {
            assembly ("memory-safe") { revert(add(result, 32), mload(result)) }
        }
        revert ExtensionCallFailed();
    }
    if (result.length < 32) revert InvalidExtensionResponse();
    // selector check …
}
```

`removeLiquidity` in `MetricOmmPool.sol` calls extensions both before and after the core accounting:

```solidity
_beforeRemoveLiquidity(msg.sender, owner, salt, deltas, extensionData);
(amount0Removed, amount1Removed) = LiquidityLib.removeLiquidity(…);
_afterRemoveLiquidity(msg.sender, owner, salt, deltas, amount0Removed, amount1Removed, extensionData);
```

If any extension registered in `BEFORE_REMOVE_LIQUIDITY_ORDER` or `AFTER_REMOVE_LIQUIDITY_ORDER` reverts, the entire transaction reverts. Because the seven extension slots are stored as `immutable` variables set in the constructor, there is no factory or admin function that can replace or skip a broken extension after deployment. `ValidateExtensionsConfig.validateExtensionsConfig` only checks that addresses are non-zero, unique, and that order indices are in range — it does not and cannot guarantee future liveness of the extension contracts.

The EigenLayer analog had a partial workaround (`receiveAsTokens = false`). Metric OMM has none: there is no alternative code path through which an LP can reclaim their shares once an extension on the `removeLiquidity` hook is broken.

---

### Impact Explanation

All LPs in the affected pool lose access to their deposited tokens permanently. `removeLiquidity` is the sole mechanism for reclaiming principal; no admin, factory-owner, or protocol-pause function provides an alternative exit. This satisfies the "broken core pool functionality causing loss of funds or unusable liquidity flows" impact gate.

---

### Likelihood Explanation

Extensions are external contracts. Post-deployment breakage can occur through:

1. **Self-destruct**: `callExtension` would receive empty return data, triggering `InvalidExtensionResponse` on every call.
2. **Extension-internal bug**: a state transition in the extension (e.g., an allowlist or gate extension whose internal state becomes inconsistent) causes it to revert on every invocation.
3. **Extension admin action**: an extension with its own pause/admin
