### Title
`evmAddress` Not Committed to in COA Ownership Proof Signature Enables Cross-COA Replay — (`File: fvm/evm/stdlib/contract.cdc`)

---

### Summary

`EVM.validateCOAOwnershipProof` verifies Cadence account key signatures over `signedData` but does not enforce that `signedData` encodes the `evmAddress` parameter. Because `evmAddress` is passed separately and only checked against the on-chain COA resource after signature verification, a valid signature produced to prove ownership of one COA can be replayed by any unprivileged caller to prove ownership of a different COA on the same Cadence account.

---

### Finding Description

`validateCOAOwnershipProof` in `fvm/evm/stdlib/contract.cdc` accepts six parameters: `address`, `path`, `signedData`, `keyIndices`, `signatures`, and `evmAddress`. The signature verification step (lines 1082–1086) verifies the provided `signatures` only against `signedData`:

```cadence
let isValid = keyList.verify(
    signatureSet: signatureSet,
    signedData: signedData,
    domainSeparationTag: "FLOW-V0.0-user"
)
``` [1](#0-0) 

After this check passes, the function separately verifies that the caller-supplied `evmAddress` matches the COA resource stored at the caller-supplied `path`:

```cadence
if let coaRef = acc.capabilities.borrow<&EVM.CadenceOwnedAccount>(path) {
    let coaAddressBytes = coaRef.address().bytes
    for index, item in coaAddressBytes {
        if item != evmAddress[index] { ... }
    }
``` [2](#0-1) 

Neither `evmAddress` nor `path` is required to be encoded inside `signedData`. The code comment at lines 1003–1009 explicitly acknowledges this gap:

> "this function does not enforce that `signedData` includes `evmAddress`. In principle, a signature produced for one purpose could be replayed here against a different COA owned by the same Cadence account." [3](#0-2) 

The function is declared `access(all)`, making it callable by any Cadence script or transaction with arbitrary caller-controlled arguments. [4](#0-3) 

The `COAOwnershipProof` struct (which is signed) contains `KeyIndices`, `Address`, `CapabilityPath`, and `Signatures`, but `SignedData` and `EVMAddress` are kept separate in `COAOwnershipProofInContext` and are never committed to by the signature: [5](#0-4) 

---

### Impact Explanation

An attacker who obtains a valid COA ownership proof (signature set + `signedData`) that a victim produced for COA-A can replay it against COA-B on the same Cadence account by:

1. Keeping the same `address`, `keyIndices`, `signatures`, and `signedData`.
2. Substituting `path` → COA-B's public path and `evmAddress` → COA-B's EVM address.

The signature check passes (the victim's key did sign `signedData`), and the `evmAddress` check passes (COA-B's on-chain address matches the attacker-supplied `evmAddress`). The function returns `isValid: true` for COA-B even though the victim never authorized a proof for COA-B.

Any Cadence-level protocol that calls `EVM.validateCOAOwnershipProof` directly to gate access to on-chain assets or privileges associated with a specific COA is vulnerable to this cross-COA replay. The EVM precompile path (`verifyCOAOwnershipProof` via `coa.sol`'s `isValidSignature`) is not directly affected because it always passes `address(this)` as the COA address: [6](#0-5) 

However, the `access(all)` Cadence function surface is fully attacker-controlled.

---

### Likelihood Explanation

The preconditions are:
- The victim Cadence account holds **two or more COAs** at distinct public paths.
- A valid `signedData` + signature set for one of those COAs is observable by the attacker (e.g., submitted on-chain or broadcast off-chain).
- A downstream Cadence protocol calls `EVM.validateCOAOwnershipProof` directly to authorize an action tied to a specific COA.

While the comment notes that "Flow wallets historically create at most one COA per account," this is a social/historical observation, not a protocol invariant. As the Flow EVM ecosystem matures and bridge/DeFi protocols proliferate, multi-COA accounts and direct use of `validateCOAOwnershipProof` in Cadence contracts will become more common, raising the likelihood from low to medium.

---

### Recommendation

Require that `signedData` commits to both `evmAddress` and `path` (or at minimum `evmAddress`) before accepting the proof as valid. One approach is to enforce this inside `validateCOAOwnershipProof` itself:

```cadence
// Verify that signedData encodes evmAddress to prevent cross-COA replay
let evmAddressBytes: [UInt8] = evmAddress.toConstantSized()!.toVariableSized()
// Check that signedData contains evmAddress bytes as a suffix or prefix
// (exact encoding scheme to be agreed upon by the protocol)
```

A stronger fix is to define a canonical signed payload format that always includes `evmAddress` (and optionally `path`), and enforce this at the protocol level so that off-chain signers and on-chain verifiers agree on the same commitment. This mirrors the remediation recommended in the external report: include the hash of the relevant parameter in the signed message so it cannot be tampered with.

---

### Proof of Concept

```cadence
import EVM from <EVM_CONTRACT_ADDRESS>

// Attacker script: replay Alice's COA-A proof against COA-B
access(all) fun main(
    aliceAddress: Address,
    coaBPath: PublicPath,          // /public/coaB
    signedData: [UInt8],           // data Alice signed for COA-A
    keyIndices: [UInt64],          // same key indices Alice used
    signatures: [[UInt8]],         // Alice's signatures (obtained from a prior tx)
    coaBEvmAddress: [UInt8; 20]    // COA-B's EVM address
): EVM.ValidationResult {
    // This returns isValid: true even though Alice only signed for COA-A
    return EVM.validateCOAOwnershipProof(
        address: aliceAddress,
        path: coaBPath,
        signedData: signedData,
        keyIndices: keyIndices,
        signatures: signatures,
        evmAddress: coaBEvmAddress
    )
}
```

The signature verification at line 1082 passes because Alice's key did sign `signedData`. The `evmAddress` check at line 1095 passes because COA-B's on-chain address matches `coaBEvmAddress`. The function returns `isValid: true` for COA-B, constituting an unauthorized ownership proof. [1](#0-0) [7](#0-6)
