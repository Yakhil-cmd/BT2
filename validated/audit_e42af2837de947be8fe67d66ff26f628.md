### Title
`poolAdminFeeDestination` Not Reset on Pool Admin Transfer — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary

`acceptPoolAdmin` transfers the `poolAdmin[pool]` role to a new address but does not reset `poolAdminFeeDestination[pool]`. A departing admin can set the fee destination to their own address before initiating the handover; after the transfer completes, all admin-share fees collected via the permissionless `collectPoolFees` continue flowing to the old admin's address until the new admin explicitly calls `setPoolAdminFeeDestination`.

### Finding Description

`acceptPoolAdmin` in `MetricOmmPoolFactory` finalises a two-step admin handover: [1](#0-0) 

It updates `poolAdmin[pool]` and clears `pendingPoolAdmin[pool]`, but leaves `poolAdminFeeDestination[pool]` untouched.

The fee destination is the address to which the admin share of every fee collection is sent: [2](#0-1) 

`collectPoolFees` is **permissionless** — any caller can trigger it at any time: [3](#0-2) 

The admin fee destination is set by the current pool admin: [4](#0-3) 

Because `acceptPoolAdmin` does not reset `poolAdminFeeDestination[pool]`, the old admin's chosen destination persists after the role transfer.

### Impact Explanation

All admin-share fees (both spread and notional components) accrued after the admin transfer are sent to the old admin's address rather than the new admin's treasury. The new admin receives zero admin fees until they discover the stale destination and call `setPoolAdminFeeDestination`. The old admin (or any keeper) can call the permissionless `collectPoolFees` immediately after `acceptPoolAdmin` completes, atomically draining all accrued fees to the old admin's address before the new admin can react. This is a direct loss of protocol fee revenue owed to the new admin.

### Likelihood Explanation

The trigger requires the departing admin to act maliciously — setting `poolAdminFeeDestination` to their own address before proposing the transfer. The pool admin is described as semi-trusted (bounded by caps and timelocks), so this is within the contest-relevant threat model. The two-step admin transfer (`proposePoolAdminTransfer` → `acceptPoolAdmin`) is a normal operational flow, and the new admin has no on-chain signal that the fee destination was changed. The permissionless `collectPoolFees` means the old admin can extract fees without any further privileged action.

### Recommendation

Reset `poolAdminFeeDestination[pool]` to `address(0)` (or to the new admin's address) inside `acceptPoolAdmin`, and emit `PoolAdminFeeDestinationUpdated`. Because a zero destination would break `collectPoolFees`, the new admin should be required to set a valid destination before fees can be collected, or the destination should default to the new admin address:

```solidity
function acceptPoolAdmin(address pool) external override nonReentrant {
    address pending = pendingPoolAdmin[pool];
    if (pending == address(0)) revert NoPendingPoolAdminTransfer();
    if (msg.sender != pending) revert NotPendingPoolAdmin(pool, msg.sender, pending);
    address previousAdmin = poolAdmin[pool];
    poolAdmin[pool] = pending;
    delete pendingPoolAdmin[pool];
+   // Reset fee destination so old admin cannot continue receiving fees
+   poolAdminFeeDestination[pool] = pending;
+   emit PoolAdminFeeDestinationUpdated(pool, pending);
    emit PoolAdminTransferred(pool, previousAdmin, pending);
}
```

### Proof of Concept

```
1. Pool is created with admin = Alice, poolAdminFeeDestination[pool] = AliceTreasury.
2. Alice calls setPoolAdminFeeDestination(pool, AliceTreasury)  // already set, or re-confirms.
3. Alice calls proposePoolAdminTransfer(pool, Bob).
4. Bob calls acceptPoolAdmin(pool).
   → poolAdmin[pool] = Bob
   → poolAdminFeeDestination[pool] = AliceTreasury  ← NOT reset
5. Alice (or any keeper) calls collectPoolFees(pool).
   → All accrued admin-share fees are transferred to AliceTreasury.
   → Bob receives nothing.
6. Bob must discover the stale destination and call setPoolAdminFeeDestination
   before any further collection; until then every collectPoolFees call
   drains admin fees to Alice.
``` [1](#0-0) [4](#0-3) [2](#0-1)
