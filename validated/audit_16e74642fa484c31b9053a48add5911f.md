### Title
Gateway Skips Signature Validation for Invoke Transactions via `skip_stateful_validations`, Admitting Unsigned Transactions into the Mempool - (File: crates/apollo_gateway/src/stateful_transaction_validator.rs)

---

### Summary

The `skip_stateful_validations` function in the gateway skips the `__validate__` entry-point call for any invoke transaction with nonce=1 when `account_tx_in_pool_or_recent_block` returns true for the sender. An unprivileged attacker who observes a victim's pending `deploy_account` in the mempool can submit an invoke with nonce=1 for that address carrying an arbitrary (invalid) signature. The gateway accepts it without any signature check. Via fee escalation the attacker can then replace the victim's legitimate nonce=1 invoke with the invalid one, which the batcher will later reject — permanently losing the victim's transaction.

---

### Finding Description

**Root cause — `skip_stateful_validations`**

`run_pre_validation_checks` calls `skip_stateful_validations` after `validate_by_mempool`:

```
validate_state_preconditions  →  validate_by_mempool  →  skip_stateful_validations
```

`skip_stateful_validations` (lines 429-460) returns `true` — meaning "skip `__validate__`" — whenever:
- the transaction is an `Invoke`,
- `tx.nonce() == Nonce(Felt::ONE)`,
- `account_nonce == Nonce(Felt::ZERO)`, **and**
- `account_tx_in_pool_or_recent_block(sender)` returns `true`. [1](#0-0) 

`account_tx_in_pool_or_recent_block` returns `true` if the account has **any** transaction in the pool or in the mempool's staged/committed state — it does not verify that the transaction is specifically a `deploy_account`: [2](#0-1) 

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` sets `validate: !skip_validate = false` in `ExecutionFlags`: [3](#0-2) 

Inside `StatefulValidator::perform_validations`, the early-return guard fires immediately:

```rust
if !tx.execution_flags.validate {
    return Ok(());
}
``` [4](#0-3) 

`__validate__` is never called. The transaction is forwarded to the mempool with no signature check.

**`validate_by_mempool` does not fill the gap**

`validate_tx` in the mempool only checks for duplicate tx-hash and nonce-too-old; it does **not** verify the signature: [5](#0-4) [6](#0-5) 

**Fee escalation enables replacement**

`validate_fee_escalation` allows an incoming transaction to replace an existing same-nonce transaction when both `tip` and `max_l2_gas_price` are sufficiently higher: [7](#0-6) 

An attacker who sets a higher tip than the victim's legitimate invoke will atomically remove the victim's transaction and insert the invalid one.

**Batcher catches the invalid transaction — but too late**

`AccountTransaction::new_for_sequencing` always sets `validate: true`: [8](#0-7) 

So the batcher will call `__validate__`, which fails for the invalid signature, and the transaction is rejected. However, the victim's legitimate invoke has already been evicted from the mempool by fee escalation and cannot be recovered.

---

### Impact Explanation

The gateway admits an invoke transaction whose signature has never been verified. This directly satisfies the **High** impact criterion: *"Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

Concretely:
- An attacker can preemptively occupy nonce=1 for any account whose `deploy_account` is visible in the mempool, with an invalid signature and arbitrary calldata.
- If fee escalation is enabled and the attacker outbids the victim's tip, the victim's legitimate nonce=1 invoke is permanently removed from the mempool.
- The batcher rejects the attacker's invalid invoke without charging a fee (no `__validate__` success, no fee transfer), so the attack is effectively free beyond network submission cost.
- The victim must resubmit their invoke, and the attacker can repeat the attack indefinitely.

---

### Likelihood Explanation

- The mempool is publicly observable; pending `deploy_account` transactions are visible to any node.
- The attacker only needs to craft an invoke with nonce=1 for the target address, set a tip above the victim's, and submit it. No privileged access is required.
- The race window is the entire period between the victim's `deploy_account` entering the mempool and the batcher committing it — typically multiple seconds to minutes.

---

### Recommendation

Replace the coarse `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a `deploy_account` transaction for the sender is present in the mempool pool (not merely that the account appears in any mempool state). Alternatively, even when `__validate__` is skipped for execution, perform a lightweight off-chain ECDSA/Stark signature pre-check against the transaction hash before admitting the transaction to the mempool.

---

### Proof of Concept

```
1. Victim submits RpcDeployAccountTransaction (nonce=0) for address A.
   → Gateway accepts it; mempool pool now contains deploy_account for A.
   → account_tx_in_pool_or_recent_block(A) == true.

2. Attacker submits RpcInvokeTransaction (nonce=1, sender=A,
   signature=[0xdead, 0xbeef], tip=victim_tip * 2).

3. Gateway stateless validation: passes (no signature check).
   Gateway stateful validation:
     validate_state_preconditions: nonce 1 >= account_nonce 0 ✓
     validate_by_mempool: nonce not too old ✓ (no sig check)
     skip_stateful_validations: nonce==1, account_nonce==0,
       account_tx_in_pool_or_recent_block(A)==true → returns true
     run_validate_entry_point(skip_validate=true):
       ExecutionFlags { validate: false }
       StatefulValidator::perform_validations → early return Ok(())
   → Gateway accepts the invalid invoke and sends it to the mempool.

4. Mempool fee escalation: attacker's tip > victim's tip
   → victim's legitimate nonce=1 invoke is removed; attacker's invalid
     invoke is inserted.

5. Batcher: executes deploy_account (nonce=0) → success.
   Batcher: executes attacker's invoke (nonce=1):
     AccountTransaction::new_for_sequencing → validate: true
     __validate__ called → fails (invalid signature)
     Transaction rejected, no fee charged.

6. Victim's legitimate invoke is gone. Victim must resubmit.
   Attacker repeats from step 2 at zero on-chain cost.
```

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)

    fn validate_fee_escalation(
        &self,
        incoming_tx_reference: TransactionReference,
    ) -> MempoolResult<Option<TransactionReference>> {
        let TransactionReference { address, nonce, .. } = incoming_tx_reference;

        self.validate_no_delayed_declare_front_run(incoming_tx_reference)?;

        if !self.config.static_config.enable_fee_escalation {
            if self.tx_pool.get_by_address_and_nonce(address, nonce).is_some() {
                return Err(MempoolError::DuplicateNonce { address, nonce });
            };

            return Ok(None);
        }

        let Some(existing_tx_reference) = self.tx_pool.get_by_address_and_nonce(address, nonce)
        else {
            // Replacement irrelevant: no existing transaction with the same nonce for address.
            return Ok(None);
        };

        if !self.should_replace_tx(&existing_tx_reference, &incoming_tx_reference) {
            info!(
                "{existing_tx_reference} was not replaced by {incoming_tx_reference} due to \
                 insufficient fee escalation."
            );
            // TODO(Elin): consider adding a more specific error type / message.
            return Err(MempoolError::DuplicateNonce { address, nonce });
        }

        Ok(Some(existing_tx_reference))
    }
