### Title
WalletConnect dApp can silently redirect `chia_sendTransaction` to an arbitrary wallet_id hidden from the confirmation dialog — (`packages/gui/src/electron/commands/Commands.ts`, `parseDappParams.ts`)

### Summary

`chia_sendTransaction` declares `wallet_id` as an explicit, dApp-controllable parameter with `hide: true`. A connected dApp can supply any `wallet_id` value; the confirmation dialog omits it from the main rows (only showing amount, fee, address), so the user approves a transaction without knowing which wallet is being spent. The daemon then executes the send from the dApp-chosen wallet.

### Finding Description

**Step 1 — `wallet_id` is in the dapp params allowlist.**

In `Commands.ts`, the `chia_wallet.send_transaction` command declares `wallet_id` as a schema param with `hide: true`, and the `chia_sendTransaction` dapp entry inherits those params (no override) with `defaults: { wallet_id: 1 }`: [1](#0-0) 

`DappCommands.ts` builds the effective dapp schema by merging the parent params (including `wallet_id`) into the dapp entry: [2](#0-1) 

**Step 2 — `parseDappParams` accepts a dApp-supplied `wallet_id` and only applies the default when the field is absent.** [3](#0-2) 

The allowlist check at line 34–38 passes because `wallet_id` IS in `dappParamsMap`. The default `wallet_id: 1` is only applied when `nextParams['wallet_id'] === undefined` (line 44). If the dApp sends `walletId: 2`, it survives the allowlist and overrides the default. This is explicitly confirmed by the test suite: [4](#0-3) 

**Step 3 — `humanizeParams` filters out `hide: true` params from the confirmation rows.** [5](#0-4) 

`wallet_id` is filtered out. The `rows` passed to the `Confirm` dialog contain only amount, fee, and address. The comment on the `hide` field acknowledges this: *"hidden from the confirm dialog - still showed under json details"*. [6](#0-5) 

**Step 4 — The confirmation dialog renders only the `rows` prominently; raw data is in a collapsed section.** [7](#0-6) 

The `wallet_id: 2` value appears only inside the collapsed `<Collapsible title="Raw data">` section, which most users will not expand.

**Step 5 — After user approval, the full `parsedParams` (including `wallet_id: 2`) is sent to the daemon.** [8](#0-7) 

The daemon executes `send_transaction` from wallet 2, not wallet 1.

### Impact Explanation

A connected dApp can drain any XCH wallet the user holds (wallet_id ≥ 1) by specifying a non-default `wallet_id`. The user sees the correct amount, fee, and destination address in the confirmation dialog but has no visible indication that the spend originates from a different wallet than expected. This is a signing-context confusion: the user reviews and approves parameters for one wallet context while the transaction is executed against another.

### Likelihood Explanation

Any dApp that has completed the WalletConnect pairing flow (which requires user approval once) can trigger this on every subsequent `chia_sendTransaction` call. No additional privileges are required. The attack is local-testable by connecting any WalletConnect dApp and calling `chia_sendTransaction` with `walletId: 2` in the JSON-RPC params.

### Recommendation

Remove `wallet_id` from the dapp-controllable params for `chia_sendTransaction` (or mark it non-overridable), and enforce the default `wallet_id: 1` unconditionally in `parseDappParams` for this command. Alternatively, remove `hide: true` from `wallet_id` so the actual wallet being spent is always shown prominently in the confirmation dialog rows.

### Proof of Concept

1. Connect a WalletConnect dApp to a Chia wallet that has at least two standard XCH wallets (wallet_id 1 and 2).
2. From the dApp, call `chia_sendTransaction` with payload:
   ```json
   { "amount": "1000000000", "fee": "0", "address": "<attacker_address>", "walletId": 2 }
   ```
3. Observe the confirmation dialog: it shows Amount, Fee, Address — no Wallet Id row.
4. Click "Send". The daemon executes `send_transaction` with `wallet_id=2`.
5. Verify on-chain that the spend originated from wallet 2, not wallet 1.
