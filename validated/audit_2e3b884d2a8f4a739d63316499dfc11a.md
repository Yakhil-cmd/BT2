### Title
Cross-Address COA Ownership Proof Replay — Missing `evmAddress` Binding in `signedData` - (File: fvm/evm/stdlib/contract.cdc)

### Summary
`EVM.validateCOAOwnershipProof` does not enforce that `signedData` encodes the `evmAddress` being authenticated. A valid signature produced for one Cadence-Owned Account (COA_A) can be replayed against a different COA (COA_B) owned by the same Cadence account, causing `isValidSignature` (ERC-1271) on COA_B to return `true` for data the account owner never authorized for COA_B.

### Finding Description
`validateCOAOwnershipProof` performs two independent checks:

1. Verifies that the Cadence account's keys produced valid signatures over `signedData`.
2. Verifies that the COA resource at the supplied `path` has an EVM address matching `evmAddress`.

There is no check that `signedData` encodes `evmAddress`. The two checks are entirely decoupled. The code itself documents this gap:

> "Note: this function does not enforce that `signedData` includes `evmAddress`. In principle, a signature produced for one purpose could be replayed here against a different COA owned by the same Cadence account." [1](#0-0) 

Because the `path` argument is attacker-supplied (it is part of the encoded proof bytes passed to the EVM precompile), an attacker who obtains a legitimate proof for COA_A can construct a modified proof that substitutes the path to COA_B while keeping the same `signedData` and signatures. The signature check still passes (same key, same data), and the COA address check passes (COA at the new path matches COA_B's address).

The EVM-side entry point is `isValidSignature` in the COA Solidity contract, which calls the `verifyCOAOwnershipProof` precompile with `address(this)` as the `evmAddress`: [2](#0-1) 

The precompile dispatches to `validateCOAOwnershipProof` via `coaOwnershipProofValidator`: [3](#0-2) 

The full Cadence validation logic that lacks the binding: [4](#0-3) 

### Impact Explanation
Any EVM contract that relies on ERC-1271 (`isValidSignature`) to authorize actions for a specific COA can be deceived. An attacker who obtains a valid proof for COA_A (e.g., from a prior legitimate interaction) can replay it against COA_B owned by the same Cadence account. This allows unauthorized approvals, token transfers, or other EVM contract interactions to be authorized on behalf of COA_B — a cross-VM asset loss / transaction authorization bypass.

### Likelihood Explanation
Requires a Cadence account that owns at least two COAs and a prior signature over the target `signedData` for one of them. While Flow wallets currently create at most one COA per account, the protocol imposes no such limit, and the `createCadenceOwnedAccount` function is `access(all)`. [5](#0-4) 

Any off-chain authentication flow, multi-sig scheme, or EVM dApp that uses ERC-1271 against COAs is exposed once a user holds more than one COA.

### Recommendation
Enforce that `signedData` commits to `evmAddress`. Inside `validateCOAOwnershipProof`, before calling `keyList.verify`, verify that the first 20 bytes (or a well-known prefix) of `signedData` equal `evmAddress`. Alternatively, document a mandatory encoding convention and enforce it at the precompile layer before invoking the Cadence function.

### Proof of Concept

**Setup:**
- Alice's Cadence account holds two COAs: COA_A at `/public/coa_a` and COA_B at `/public/coa_b`.
- Alice legitimately signs `_hash` for COA_A, producing `proof_A = {address: alice, path: "coa_a", keyIndices: [0], signatures: [sig]}`.

**Attack:**
1. Attacker constructs `proof_B = {address: alice, path: "coa_b", keyIndices: [0], signatures: [sig]}` — identical except `path` points to COA_B.
2. Attacker calls `COA_B.isValidSignature(_hash, encode(proof_B))` from an EVM transaction.
3. The COA Solidity contract calls `verifyCOAOwnershipProof(address(COA_B), _hash, encode(proof_B))`.
4. `validateCOAOwnershipProof` is invoked with `address=alice, path=/public/coa_b, signedData=_hash, evmAddress=COA_B_address`.
5. `keyList.verify(signatureSet, signedData: _hash, ...)` → **passes** (Alice signed `_hash`).
6. COA address check: COA at `/public/coa_b` has address `COA_B_address` → **passes**.
7. Returns `ValidationResult(isValid: true)`. [6](#0-5) 

Result: `COA_B.isValidSignature` returns `ValidERC1271Signature` for data Alice only authorized for COA_A, enabling unauthorized EVM contract actions on behalf of COA_B.

        bytes memory _sig
    ) external view virtual returns (bytes4){

            return ValidERC1271Signature;
