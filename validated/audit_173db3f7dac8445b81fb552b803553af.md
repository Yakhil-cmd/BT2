### Title
WalletConnect `chia_createOfferForIds` Fee Omitted from "You Spend" Confirmation Display - (File: `packages/gui/src/electron/commands/parseCommandDisplay.ts`)

### Summary

`parseCommandDisplay` builds a rich "You Spend / You Receive" wallet-delta display for WalletConnect commands. For `chia_wallet.take_offer` it correctly extracts and forwards the fee into that display. For `chia_wallet.create_offer_for_ids` it hardcodes `undefined` as the fee, so the prominent "You Spend" card shown to the user never includes the network fee — even when a dApp supplies an arbitrarily large one.

### Finding Description

`parseCommandDisplay` handles `chia_wallet.create_offer_for_ids` by computing a `walletDelta` from `params.offer` (the wallet-IDs-and-amounts dict) and calling `walletDeltaToDisplay` with `undefined` as the fee:

```ts
// parseCommandDisplay.ts line 462-479
if (command === 'chia_wallet.create_offer_for_ids') {
  ...
  const walletDelta = createOfferToWalletDelta(params.offer);
  ...
  return {
    walletDelta: await walletDeltaToDisplay(walletDelta, walletInfos, assetKinds, royaltyPercentages, undefined),
    //                                                                                                  ^^^^^^^^^ fee always undefined
  };
}
``` [1](#0-0) 

By contrast, `chia_wallet.take_offer` correctly extracts `summary.fees` and passes it through:

```ts
// parseCommandDisplay.ts line 455-458
const fees = parseMojos(summary.fees);
return {
  walletDelta: await walletDeltaToDisplay(walletDelta, walletInfos, assetKinds, royaltyPercentages, fees),
};
``` [2](#0-1) 

`walletDeltaToDisplay` only renders the "Offer Fees" row in the confirmation card when `fee !== undefined`:

```ts
// Confirm.tsx line 221-230
{walletDelta.fee !== undefined && (
  <div className="px-5 py-2.5">
    <div ...>Offer Fees</div>
    <div ...>{walletDelta.fee} {feeUnit}</div>
  </div>
)}
``` [3](#0-2) 

The `Confirm` component renders the `walletDelta` section first (the prominent "You Spend / You Receive" card), then the generic `rows` section below it:

```tsx
// Confirm.tsx line 381-392
{walletDelta && <WalletDeltaSection walletDelta={walletDelta} networkPrefix={networkPrefix} />}
{rows.length > 0 && (
  <section ...>
    {rows.map(({ field, label, value }) => (...))}
  </section>
)}
``` [4](#0-3) 

The `fee` parameter is declared in `Commands.ts` for `create_offer_for_ids` as `isOptional: true`, so it may appear in the secondary `rows` section when supplied — but it is never surfaced in the primary "You Spend" card regardless of its value. [5](#0-4) 

### Impact Explanation

A malicious dApp connected via WalletConnect sends `chia_createOfferForIds` with a legitimate-looking `offer` dict (e.g., locking 0.001 XCH) and an arbitrarily large `fee` (e.g., 1 000 XCH). The confirmation dialog's prominent "You Spend" card shows only **0.001 XCH**. The fee — which is deducted from the user's XCH balance when the transaction executes — is absent from that card. The user approves believing the total XCH cost is 0.001 XCH; the actual cost is 1 000.001 XCH. This is a direct, concrete balance loss affecting XCH.

### Likelihood Explanation

Any dApp that has been granted the `chia_createOfferForIds` WalletConnect permission (an unprivileged, user-approved pairing) can trigger this. No additional compromise is required. The dApp simply includes a `fee` field alongside the `offer` dict. The user has no reliable way to detect the discrepancy from the primary confirmation UI.

### Recommendation

In `parseCommandDisplay`, extract `params.fee` and pass it to `walletDeltaToDisplay` for `chia_wallet.create_offer_for_ids`, mirroring the pattern already used for `chia_wallet.take_offer`:

```ts
// packages/gui/src/electron/commands/parseCommandDisplay.ts
if (command === 'chia_wallet.create_offer_for_ids') {
  ...
  const walletDelta = createOfferToWalletDelta(params.offer);
  const walletInfos = await getWalletInfos();
  const driverDict = params.driver_dict ?? {};
  const assetKinds = createOfferAssetKinds(walletDelta, walletInfos, driverDict);
  const royaltyPercentages = createOfferRoyaltyPercentages(walletDelta, driverDict);
+ const fee = params.fee !== undefined ? parseMojos(params.fee) : undefined;

  return {
-   walletDelta: await walletDeltaToDisplay(walletDelta, walletInfos, assetKinds, royaltyPercentages, undefined),
+   walletDelta: await walletDeltaToDisplay(walletDelta, walletInfos, assetKinds, royaltyPercentages, fee),
  };
}
```

### Proof of Concept

1. A dApp with an approved `chia_createOfferForIds` WalletConnect pairing sends:
   ```json
   {
     "command": "chia_wallet.create_offer_for_ids",
     "params": {
       "offer": { "1": "-1000000" },
       "fee": "1000000000000000"
     }
   }
   ```
2. `parseCommandDisplay` calls `createOfferToWalletDelta({ "1": "-1000000" })` → `spending: { "1": 1000000n }`.
3. `walletDeltaToDisplay(..., undefined)` is called — fee is `undefined`.
4. The confirmation dialog renders "You Spend: 0.000001 XCH" with no "Offer Fees" row.
5. The user approves. The backend executes the offer creation with a 1 000 XCH fee, draining the wallet.
