Audit Report

## Title
VIP Callers Blocked by Non-VIP Slot Exhaustion in `try_borrow_slot` — (`rs/nervous_system/clients/src/management_canister_client.rs`)

## Summary
`LimitedOutstandingCallsManagementCanisterClient::try_borrow_slot` unconditionally rejects all callers when `available_slot_count == 0`, including VIP callers (NNS canisters) that are designed to consume zero slots. An unprivileged attacker can exhaust all 167 shared slots via the publicly-accessible `canister_status` endpoint on NNS Root, permanently blocking governance-triggered `take_canister_snapshot` and `load_canister_snapshot` for the duration of the attack.

## Finding Description
In `try_borrow_slot` (lines 264–287 of `rs/nervous_system/clients/src/management_canister_client.rs`), `used_slot_count` is set to `0` for VIP callers, correctly indicating they should not consume a slot. However, the exhaustion guard `if *available_slot_count == 0` fires before the subtraction, returning `Err((SysTransient, ...))` for every caller — including VIP — when the pool is empty. A VIP caller with `used_slot_count = 0` would leave the counter unchanged, but it never reaches that line.

The public `canister_status` endpoint on NNS Root (lines 88–98 of `rs/nns/handlers/root/impl/canister/canister.rs`) has no caller restriction and is explicitly documented as open to anyone. Each call borrows one slot for the full round-trip duration of the inner management canister call. The shared pool is fixed at 167 (line 50 of `canister.rs`). An attacker sends 167 concurrent `canister_status` calls; NNS Root processes each up to the `await` point, suspending with all 167 slots occupied. Any subsequent call from NNS Governance (a VIP) to `take_canister_snapshot` or `load_canister_snapshot` invokes `new_management_canister_client()` with `is_caller_vip = true`, then calls `try_borrow_slot()`, which hits the `available_slot_count == 0` guard and returns `Err(SysTransient, "Unavailable. Maybe, try again later?")`. The governance-facing functions propagate this error directly to the caller.

The existing unit test (line 134 of `tests.rs`) uses `capacity=2` with VIP calls starting at `t=0` before non-VIP calls at `t=50ms`, so `available_slot_count` never reaches 0 when a VIP call arrives. The critical edge case — VIP call arriving after pool reaches exactly 0 — is not covered.

## Impact Explanation
This is a High-severity application/platform-level DoS. When all 167 slots are occupied, NNS Governance cannot execute `take_canister_snapshot` or `load_canister_snapshot` on NNS Root. These are the only on-chain disaster recovery mechanisms for restoring NNS-controlled canisters (e.g., ledger, governance) from snapshots. Blocking them during a time-sensitive recovery window (e.g., corrupted ledger state) constitutes a significant NNS governance availability impact with concrete protocol harm, matching the allowed impact: *"Application/platform-level DoS … or subnet availability impact not based on raw volumetric DDoS"* and *"Significant … NNS … security impact with concrete user or protocol harm."*

## Likelihood Explanation
No special privileges are required. The attacker deploys a canister on any subnet, sends 167 inter-canister calls to NNS Root's `canister_status`, and sustains the attack by replenishing calls as slots free up. The only cost is cycles. The IC execution model guarantees all 167 calls can be in-flight simultaneously (each suspends at the `await` point). The attack is deterministic, repeatable, and requires no social engineering, key material, or victim mistakes.

## Recommendation
Skip the exhaustion guard for VIP callers in `try_borrow_slot`, since they consume zero slots and cannot deplete the pool:

```rust
fn try_borrow_slot(&self) -> Result<SlotLoan, (i32, String)> {
    let used_slot_count = if self.is_caller_vip { 0 } else { 1 };

    self.available_slot_count
        .with_borrow_mut(|available_slot_count| {
            if !self.is_caller_vip && *available_slot_count == 0 {
                let code = RejectCode::SysTransient as i32;
                return Err((code, "Unavailable. Maybe, try again later?".to_string()));
            }
            *available_slot_count = available_slot_count.saturating_sub(used_slot_count);
            Ok(())
        })?;
    ...
}
```

Additionally, add a unit test that sets `capacity=2`, fills both slots with non-VIP calls, then asserts a VIP call succeeds while `available_slot_count == 0`.

## Proof of Concept
State-machine or PocketIC test:
1. Initialize NNS Root with a mock management canister client that stalls before responding.
2. Send 167 concurrent `canister_status` calls from an unprivileged canister to NNS Root; advance execution until all 167 are suspended at the `await` point.
3. Assert `AVAILABLE_MANAGEMENT_CANISTER_CALL_SLOT_COUNT == 0`.
4. Send a `load_canister_snapshot` call from the NNS Governance canister (VIP) to NNS Root.
5. Assert the response is `Err` with reject code `SysTransient` — confirming VIP is blocked despite `is_caller_vip = true` and `used_slot_count = 0`.
