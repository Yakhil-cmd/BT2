### Title
Missing Value Guard in `Identity::run` Allows Permanent ETH Freeze at Precompile Address 0x04 - (`engine-precompiles/src/identity.rs`)

### Summary

`Identity::run` does not call `utils::validate_no_value_attached_to_precompile`, unlike every Aurora-specific precompile that handles value. When a transaction with `value > 0` is sent to `Identity::ADDRESS` (0x04), SputnikVM credits the ETH to 0x04's balance before invoking the precompile. Because `Identity::run` returns `Ok(...)` unconditionally, the state change is committed. Address 0x04 has no EVM bytecode and no withdrawal precompile path, so the ETH is permanently frozen.

### Finding Description

`Identity::run` ignores `context.apparent_value` entirely: [1](#0-0) 

By contrast, every Aurora-specific precompile that could receive value explicitly rejects it: [2](#0-1) 

`PrepaidGas`, `RandomSeed`, `PredecessorAccount`, `CurrentAccount`, and `CrossContractCall` all call this guard at the top of their `run`/`run_with_handle` implementations: [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5) 

`Identity` has no such guard.

**Execution path:**

1. A user submits a signed EVM transaction with `to = 0x0000000000000000000000000000000000000004` and `value = N wei` via the `submit` entry point.
2. `engine::submit_with_alt_modexp` parses the transaction and calls `engine.call(&sender, &receiver=0x04, value=N, ...)`. [7](#0-6) 

3. `engine.call` invokes `executor.transact_call(origin, 0x04, N, ...)`. [8](#0-7) 

4. SputnikVM's `StackExecutor` performs the value transfer (debit sender, credit 0x04) before dispatching to the precompile set.
5. `Precompiles::execute` routes to `Identity` via `AllPrecompiles::Generic`. [9](#0-8) 

6. `Identity::run` returns `Ok(PrecompileOutput::without_logs(cost, input.to_vec()))` — no error, no revert.
7. Back in `engine.call`, `executor.into_state().deconstruct()` yields the balance changes (sender −N, 0x04 +N), and `self.apply(values, ...)` commits them to persistent storage. [10](#0-9) 

8. The `Apply::Modify` branch in `ApplyBackend::apply` writes the new balance for 0x04 via `set_balance`. [11](#0-10) 

9. Address 0x04 has no EVM code and no withdrawal precompile path. The ETH is irrecoverable.

### Impact Explanation

Any ETH sent to `Identity::ADDRESS` with `value > 0` is permanently frozen. The sender loses the funds with no recourse. This matches **Critical — Permanent freezing of funds**.

### Likelihood Explanation

The attack requires only a standard signed EVM transaction with a non-zero value field targeting 0x04. No special privileges, no admin access, no external dependencies. Any user — whether by mistake or deliberately — can trigger this. The inconsistency with every other Aurora precompile that does call the guard makes accidental triggering plausible.

### Recommendation

Add the value guard at the top of `Identity::run`, mirroring the pattern used by all other Aurora precompiles:

```rust
fn run(
    &self,
    input: &[u8],
    target_gas: Option<EthGas>,
    context: &Context,
    _is_static: bool,
) -> EvmPrecompileResult {
    utils::validate_no_value_attached_to_precompile(context.apparent_value)?;
    // ... existing logic
}
```

Audit all other `AllPrecompiles::Generic` entries (SHA256, RIPEMD160, ECRecover, ModExp, BN256 variants, etc.) for the same omission.

### Proof of Concept

```rust
#[test]
fn test_identity_freezes_eth_with_value() {
    use aurora_engine_precompiles::identity::Identity;
    use aurora_engine_transactions::legacy::TransactionLegacy;
    use aurora_engine_types::{U256, types::Wei};

    let (mut runner, mut signer, _) = initialize_transfer();
    let sender = utils::address_from_secret_key(&signer.secret_key);
    let value = Wei::new_u64(1_000);

    // Fund sender
    runner.create_address(sender, Wei::new_u64(1_000_000), 0.into());

    let tx = TransactionLegacy {
        nonce: signer.use_nonce().into(),
        gas_price: U256::zero(),
        gas_limit: u64::MAX.into(),
        to: Some(Identity::ADDRESS),   // 0x04
        value,
        data: vec![0xde, 0xad],
    };

    let result = runner.submit_transaction(&signer.secret_key, tx).unwrap();
    assert!(result.status.is_ok());  // precompile succeeds — no revert

    // ETH is now credited to 0x04
    assert_eq!(runner.get_balance(Identity::ADDRESS), value);

    // No mechanism exists to move ETH out of 0x04
    // Any subsequent call to 0x04 with value=0 just echoes input; no transfer occurs
}
```
