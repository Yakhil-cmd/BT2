### Title
Missing Participant-Set Consistency Check in OT-Based ECDSA Presign Allows Honest Parties to Accept Corrupted Presignatures - (File: `src/ecdsa/ot_based_ecdsa/presign.rs`)

---

### Summary

The `presign()` function in the OT-based ECDSA implementation explicitly omits a check that the presigning participant set matches the participant set that generated the Beaver triples. The `TriplePub` struct records the triple-generation participants, but `presign()` never validates them against the provided signing participants. A malicious coordinator can supply mismatched triples and participant lists, causing honest parties to compute and accept a cryptographically invalid presignature, permanently denying signing for that presignature.

---

### Finding Description

`TriplePub` stores the participant set that generated the triple: [1](#0-0) 

The `presign()` entry-point takes both a `participants` slice and `args: PresignArguments` (which embeds `triple0: (TripleShare, TriplePub)` and `triple1: (TripleShare, TriplePub)`). It validates threshold consistency between the two triples, but it **explicitly omits** the check that the presign participant set matches the triple-generation participant set: [2](#0-1) 

Inside `do_presign`, Lagrange coefficients are immediately computed over the **presign** participant set: [3](#0-2) 

The triple shares `(k_i, e_i, a_i, b_i, c_i)` are polynomial evaluations at the **triple-generation** participant IDs. When the presign participant set differs from the triple participant set, the Lagrange coefficients are computed for the wrong domain. The linearized values `k'_i`, `e_i`, `a'_i`, `b'_i`, `x'_i` are all wrong, producing a presignature `(R, k_i, sigma_i)` that is cryptographically invalid. Honest parties complete the protocol and accept this output without any indication of the mismatch.

The analog to the external report is direct: just as `createBid` proceeds without checking the paused state, `presign()` proceeds without checking that the triple-generation state (participant set) is consistent with the current presign invocation.

---

### Impact Explanation

**High — Corruption of presign outputs so honest parties accept unusable cryptographic outputs.**

Every subsequent signing attempt using the corrupted presignature will produce a signature that fails verification. Because presignatures are one-time-use, the corrupted presignature cannot be retried. If a malicious coordinator can repeatedly trigger this condition across all available presignatures, signing is permanently denied for the affected key material. This maps directly to the allowed impact: *"Corruption of presign outputs so honest parties accept inconsistent transcripts or unusable cryptographic outputs."*

---

### Likelihood Explanation

**Medium.** The attacker must be a malicious coordinator or a participant who controls which `(TripleShare, TriplePub)` values are delivered to the `presign()` call. In a realistic deployment, the coordinator assembles the presign session and distributes triple material; a compromised coordinator can trivially supply triples generated by a different participant set. No cryptographic break is required — only the ability to call `presign()` with mismatched arguments, which the API permits without restriction.

---

### Recommendation

Add an explicit check in `presign()` that the presign participant set is identical to the participant set recorded in both `TriplePub` values:

```rust
// After constructing `participants` (ParticipantList):
if participants.as_slice() != args.triple0.1.participants.as_slice()
    || participants.as_slice() != args.triple1.1.participants.as_slice()
{
    return Err(InitializationError::BadParameters(
        "Presign participant set must match the triple generation participant set".to_string(),
    ));
}
```

Remove or replace the comment at lines 38–40 that currently justifies the omission, since the `TriplePub.participants` field exists precisely to enable this check.

---

### Proof of Concept

```
1. Run triple generation with participant set P1 = {A, B, C}, threshold = 2.
   → Produces (TripleShare_A, TriplePub{participants: [A,B,C]}), etc.

2. Introduce a new participant D. Initiate presign() with:
     participants = [A, B, D]          ← P2 ≠ P1
     args.triple0 = (TripleShare_A, TriplePub{participants: [A,B,C]})
     args.triple1 = (TripleShare_A, TriplePub{participants: [A,B,C]})

3. presign() passes all current checks:
   - participants.len() >= 2 ✓
   - threshold <= participants.len() ✓
   - threshold matches both TriplePub.threshold ✓
   - me ∈ participants ✓
   - NO check: participants == triple.participants ✗ (missing)

4. do_presign() computes lambda_me = Lagrange([A,B,D], A)
   but TripleShare_A.k was a polynomial evaluation for domain [A,B,C].
   The linearized k'_A = lambda([A,B,D], A) * k_A is cryptographically wrong.

5. All parties complete the protocol and return a PresignOutput with
   an invalid (R, k_i, sigma_i). No error is raised.

6. Any call to sign() using this presignature produces a signature
   that fails ECDSA verification, permanently denying signing.
```
