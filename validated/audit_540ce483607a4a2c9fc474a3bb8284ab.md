### Title
`validateCOAOwnershipProof` Accepts Unbounded User-Supplied `signedData` Enabling Cross-Context Signature Replay Without Expiry - (`File: fvm/evm/stdlib/contract.cdc`)

---

### Summary

`EVM.validateCOAOwnershipProof` in `fvm/evm/stdlib/contract.cdc` verifies Cadence account key signatures over a caller-supplied `signedData: [UInt8]` blob. The function does not enforce that `signedData` commits to the target `evmAddress`, nor does it enforce any expiry or nonce. A valid COA ownership proof signature can therefore be replayed in a different context (a different COA owned by the same Cadence account, or a different ERC-1271 application) indefinitely. The code itself acknowledges the first half of this issue in a comment but dismisses it as "low-risk" based on a historical assumption that is not a protocol invariant.

---

### Finding Description

`validateCOAOwnershipProof` accepts six parameters: `address`, `path`, `signedData`, `keyIndices`, `signatures`, and `evmAddress`. The signature set is verified against the caller-supplied `signedData` using the hardcoded domain tag `"FLOW-V0.0-user"`:

```cadence
let isValid = keyList.verify(
    signatureSet: signatureSet,
    signedData: signedData,
    domainSeparationTag: "FLOW-V0.0-user"
)
```

The `evmAddress` is checked only *after* signature verification, and only to confirm that the COA resource at `path` has a matching EVM address. Because `signedData` is never required to encode `evmAddress`, a signature produced over arbitrary data `D` is valid for *any* COA owned by the same Cadence account, as long as the caller supplies the correct `path` and `evmAddress` for that COA.

The same vulnerability surfaces through the ERC-1271 entry point in `fvm/evm/handler/coa/coa.sol`:

```solidity
function isValidSignature(bytes32 _hash, bytes memory _sig) external view virtual returns (bytes4) {
    (bool ok, bytes memory data) = cadenceArch.staticcall(
        abi.encodeWithSignature("verifyCOAOwnershipProof(address,bytes32,bytes)", address(this), _hash, _sig)
    );
    ...
}
```

Here `_hash` is the `bytes32` passed by the EVM caller and becomes `signedData` in `validateCOAOwnershipProof`. It is entirely attacker-controlled. There is no nonce, no application-specific binding, and no expiry anywhere in the verification chain.

---

### Impact Explanation

**Cross-COA replay (same Cadence account, multiple COAs):** If a Cadence account owns COA_A and COA_B, a signature produced to prove ownership of COA_A (over some `signedData = D`) can be replayed by an attacker to prove ownership of COA_B. The attacker supplies `path → COA_B` and `evmAddress = COA_B.address`; the signature check passes because `D` does not commit to the COA address.

**Cross-application ERC-1271 replay:** Any EVM contract that calls `isValidSignature(hash, sig)` on a COA will accept a signature that was originally produced for a *different* ERC-1271 application, as long as the same `hash` value appears. Because `signedData` carries no application-specific context, a signature produced for Application A is cryptographically indistinguishable from one produced for Application B when both present the same `hash`.

**Indefinite validity:** There is no expiry or nonce check anywhere in the proof verification path. A captured signature remains valid forever, enabling future replay attacks.

---

### Likelihood Explanation

The attack is reachable by any unprivileged EVM transaction sender or Cadence script author:

- The `verifyCOAOwnershipProof` precompile is callable from any EVM transaction.
- `EVM.validateCOAOwnershipProof` is a public Cadence function callable from any script or transaction.
- The COA's `isValidSignature` is a standard ERC-1271 endpoint callable by any EVM contract.

The cross-COA replay requires the victim to own more than one COA, which the comment acknowledges is currently rare. However, the ERC-1271 cross-application replay requires only that two applications present the same `bytes32` hash to the same COA, which is a realistic scenario in DeFi (e.g., EIP-712 typed-data hashes that collide across protocols, or off-chain order books).

---

### Recommendation

1. **Bind `signedData` to `evmAddress`:** Enforce inside `validateCOAOwnershipProof` that `signedData` encodes `evmAddress` (e.g., require `signedData` to be `evmAddress ++ applicationContext`), or compute the signed payload internally rather than accepting it from the caller.

2. **Add a nonce or expiry field:** Introduce a per-COA nonce or a block-height expiry that must be included in `signedData` and verified on-chain, preventing indefinite replay.

3. **Harden the ERC-1271 implementation:** In `coa.sol`, prepend the COA's own EVM address and a chain-specific identifier to `_hash` before passing it to the precompile, so that a signature produced for one application cannot be replayed in another.

---

### Proof of Concept

**Step 1 – Victim signs data for Application A:**
```
signedData = keccak256("approve_transfer_1000_USDC_to_0xDEAD")
sig = sign(signedData, "FLOW-V0.0-user", victimKey)
proof = COAOwnershipProof{Address: victimFlowAddr, CapabilityPath: "coa_a", KeyIndices: [0], Signatures: [sig]}
```

**Step 2 – Attacker replays for Application B (different ERC-1271 context):**
Application B calls `isValidSignature(signedData, encodedProof)` on COA_B (also owned by victim). The call reaches `validateCOAOwnershipProof` with:
- `signedData` = same hash as Step 1
- `path` = `"coa_b"` (attacker-chosen)
- `evmAddress` = COA_B's address

**Step 3 – Verification passes:**
`keyList.verify(signatureSet, signedData, "FLOW-V0.0-user")` succeeds because the signature is over `signedData` which does not commit to `evmAddress`. The `evmAddress` check at lines 1097–1105 passes because the attacker supplied the correct COA_B address. `isValidSignature` returns `ValidERC1271Signature`.

The attacker has successfully impersonated the victim's COA_B ownership without the victim's consent, using a signature the victim produced for an entirely different purpose. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) [5](#0-4) [6](#0-5)

		value, err := backend.Invoke(
			environment.ContractFunctionSpec{
				AddressFromChain: func(_ flow.Chain) flow.Address {
					return contractAddress
				},
				LocationName: "EVM",
				FunctionName: "validateCOAOwnershipProof",
				ArgumentTypes: []sema.Type{
					types.FlowAddressSemaType,
					types.PublicPathSemaType,
					types.SignedDataSemaType,
					types.KeyIndicesSemaType,
					types.SignaturesSemaType,
					types.AddressBytesSemaType,
				},
			},
			proof.ToCadenceValues(),
		)
		if err != nil {
			if types.IsAFatalError(err) {
				panic(err)
			}
			return false, err
		}
		data, ok := value.(cadence.Struct)
		if !ok {
			return false, fmt.Errorf("invalid output data received from validateCOAOwnershipProof")

			return false, fmt.Errorf("invalid output data received from validateCOAOwnershipProof")
		}

		return bool(isValidValue.(cadence.Bool)), nil
	}
