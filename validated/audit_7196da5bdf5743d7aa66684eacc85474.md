Audit Report

## Title
ckERC20 Minter Over-Mints Tokens for Fee-on-Transfer ERC20 Deposits — (File: rs/ethereum/cketh/minter/DepositHelperWithSubaccount.sol)

## Summary

The `depositErc20` function in both Solidity helper contracts emits the `ReceivedEthOrErc20` / `ReceivedErc20` event using the caller-supplied `amount` rather than the actual tokens received by the minter address. The IC ckETH minter scrapes these events and mints exactly `event.value()` ckERC20 tokens to the depositor. For any fee-on-transfer ERC20 token, the minter receives `amount − fee` on Ethereum but mints `amount` ckERC20 on the IC, permanently breaking the 1:1 backing invariant and eventually rendering the minter insolvent.

## Finding Description

**Solidity root cause (`DepositHelperWithSubaccount.sol`, lines 511–532):**

`depositErc20` calls `erc20Token.safeTransferFrom(msg.sender, minterAddress, amount)` and then immediately emits `ReceivedEthOrErc20(..., amount, ...)` using the caller-supplied `amount`. No balance-delta measurement is performed. For a fee-on-transfer token the minter receives `amount − fee`, but the event records `amount`.

The identical pattern exists in the older helper (`ERC20DepositHelper.sol`, lines 498–503): `safeTransferFrom` followed by `emit ReceivedErc20(..., amount, ...)`.

**IC minter root cause (`deposit.rs`, lines 73–81):**

The minter calls `client.transfer(TransferArg { ..., amount: event.value(), ... })`, minting exactly the value read from the Ethereum log.

**Log parsing (`eth_logs/parser.rs`, lines 127–160):**

`value_bytes` is read directly from `entry.data` (the ABI-encoded `amount` field of the event) and stored as `Erc20Value` without any cross-check against the minter's actual ERC20 balance change.

**`value()` accessor (`eth_logs/mod.rs`, lines 205–210):**

Returns the raw `evt.value` field from the parsed event, which is the unverified caller-supplied amount.

There is no step anywhere in the deposit pipeline that compares `event.value()` against the actual ERC20 balance delta of the minter address.

## Impact Explanation

Every deposit of a fee-on-transfer ERC20 token causes the minter to mint more ckERC20 than it holds ERC20 as backing. The shortfall accumulates with each deposit. When users redeem ckERC20 for ERC20, the minter's Ethereum address will eventually lack sufficient ERC20 to honor all redemptions — the last redeemers lose funds proportional to the accumulated fee shortfall. This constitutes **illegal minting and protocol insolvency of an in-scope chain-key/ledger asset (ckERC20)**, matching the Critical impact class: *"Theft, permanent loss, illegal minting, or protocol insolvency involving exorbitant ICP/Cycles or in-scope chain-key/ledger assets."* At scale (e.g., ckUSDT with even a 1 bp fee and high deposit volume), losses easily exceed $1M.

## Likelihood Explanation

Currently supported ckERC20 tokens (USDC, USDT) have transfer fees set to zero, making the bug latent. However:
1. USDT's fee mechanism is live on-chain and can be activated by the Tether contract owner unilaterally, without any IC governance action.
2. The ckERC20 token list is expanded via NNS proposals; any newly listed token with a non-zero transfer fee would be immediately and continuously exploitable.
3. No privileged access is required — any user making a standard `depositErc20` call triggers the over-minting.

Likelihood is **Medium** for the USDT scenario (depends on Tether enabling fees) and **High** for any future fee-on-transfer token listing.

## Recommendation

**Short term:** Document that fee-on-transfer and rebasing ERC20 tokens are unsupported, and add an explicit screening step in the NNS proposal review process for new ckERC20 token listings.

**Long term:** Modify both helper contracts to measure the actual balance delta of `minterAddress` before and after `safeTransferFrom`, and emit that delta as the event amount:

```solidity
uint256 balanceBefore = erc20Token.balanceOf(minterAddress);
erc20Token.safeTransferFrom(msg.sender, minterAddress, amount);
uint256 received = erc20Token.balanceOf(minterAddress) - balanceBefore;
emit ReceivedEthOrErc20(erc20Address, msg.sender, received, principal, subaccount);
```

This ensures the IC minter mints only what was actually received, regardless of the token's transfer mechanics.

## Proof of Concept

1. USDT enables its transfer fee (e.g., 1 bp = 0.01%).
2. Alice calls `depositErc20(USDT_ADDRESS, 1_000_000e6, alice_principal, 0x00)` on the `CkDeposit` helper.
3. `safeTransferFrom` moves 1,000,000 USDT from Alice to the minter; due to the 0.01% fee, the minter receives 999,900 USDT.
4. The helper emits `ReceivedEthOrErc20(..., 1_000_000e6, ...)`.
5. The IC minter scrapes the event, reads `value = 1_000_000e6`, and mints 1,000,000 ckUSDT to Alice.
6. Alice holds 1,000,000 ckUSDT; the minter holds only 999,900 USDT — a 100 USDT shortfall per deposit.
7. After N such deposits the minter's USDT reserve is insufficient to honor all ckUSDT redemptions; the last redeemers receive nothing.

A deterministic integration test can reproduce this by deploying a mock ERC20 with a configurable transfer fee, calling `depositErc20`, and asserting that the minter's ERC20 balance equals the sum of all minted ckERC20 — the assertion will fail for any non-zero fee.
