### Title
Stale ETH Balance in `PeripheryPayments.pay()` Allows Cross-User ETH Theft — (`metric-periphery/contracts/base/PeripheryPayments.sol`)

### Summary

`PeripheryPayments.pay()` reads `address(this).balance` to determine how much native ETH to use when settling a WETH-input swap. Because the router is `payable` and has no mechanism to attribute which ETH belongs to which caller, any ETH left on the router by a prior user (e.g., excess `msg.value` from a multicall that omitted `refundETH()`) is silently consumed to fund a subsequent user's swap.

### Finding Description

The vulnerable branch in `pay()` is: [1](#0-0) 

```solidity
} else if (token == WETH) {
    uint256 nativeBalance = address(this).balance;   // ← entire contract balance, not msg.value
    if (nativeBalance >= value) {
        IWETH9(WETH).deposit{value: value}();
        IERC20(WETH).safeTransfer(recipient, value);
    } else if (nativeBalance > 0) {
        IWETH9(WETH).deposit{value: nativeBalance}();
        IERC20(WETH).safeTransfer(recipient, nativeBalance);
        IERC20(WETH).safeTransferFrom(payer, recipient, value - nativeBalance);
    } else {
        IERC20(WETH).safeTransferFrom(payer, recipient, value);
    }
}
```

`nativeBalance` is the **total** ETH held by the router, not the ETH contributed by the current caller. ETH accumulates on the router whenever a user sends `msg.value` to a payable entry point (`multicall`, `exactInputSingle`, `exactOutputSingle`, `exactInput`, `exactOutput`) and does not include a `refundETH()` call in the same multicall.

The `receive()` guard (`if (msg.sender != WETH) revert NotWETH()`) only blocks direct `transfer`/`send` calls; it does not prevent ETH from accumulating via `msg.value` on payable functions. [2](#0-1) 

The protocol's own test acknowledges that `refundETH()` must be explicitly included to recover unused ETH: [3](#0-2) 

This is a user-side obligation that is not enforced on-chain.

### Impact Explanation

**Attack scenario (cross-user ETH theft):**

1. User A calls `multicall{value: 1 ETH}([exactInputSingle(WETH→token1, amountIn: 0.5 ETH)])` — omitting `refundETH()`. After the swap, 0.5 ETH remains on the router.
2. User B calls `exactInputSingle(WETH→token1, amountIn: 0.3 ETH)` with no ETH sent (`msg.value = 0`).
3. Inside `pay()`, `nativeBalance = 0.5 ETH ≥ 0.3 ETH`, so the router deposits A's 0.5 ETH and transfers 0.3 WETH to the pool. B's swap is fully funded by A's ETH.
4. User A has permanently lost 0.3 ETH; user B received a free swap.

In the partial-cover branch (`0 < nativeBalance < value`), B's WETH allowance is only pulled for the shortfall, so A's ETH still subsidises B's swap.

**Impact:** Direct, permanent loss of user principal (ETH). No privileged access required. Any user who observes residual ETH on the router (trivially visible on-chain) can exploit it by submitting a WETH-input swap.

### Likelihood Explanation

- The `refundETH()` call is a user-side convention, not enforced by the router. Any user who sends excess ETH without including `refundETH()` in their multicall leaves ETH exploitable.
- MEV bots monitoring the mempool or router balance can front-run or back-run to capture the residual ETH.
- The `exactInputSingle` and `exactOutputSingle` functions are individually `payable`, so a user can accidentally send ETH to a non-multicall call and leave it stranded.

### Recommendation

Track only the ETH contributed by the **current call** rather than the entire contract balance. The standard fix is to snapshot `address(this).balance` before the outermost entry point and pass the caller's contribution explicitly, or to use a transient-storage slot to record `msg.value` at multicall entry and decrement it as ETH is consumed. Alternatively, enforce that any residual ETH is refunded atomically at the end of every payable entry point (revert if `address(this).balance > 0` after execution).

### Proof of Concept

```solidity
// Foundry fork test (pseudo-code)
function test_crossUserEthTheft() public {
    // Setup: userA has 1 ETH, userB has WETH approval but no ETH
    uint256 amountA = 0.5 ether;
    uint256 amountB = 0.3 ether;

    // Step 1: userA multicall with excess ETH, no refundETH()
    vm.prank(userA);
    bytes[] memory callsA = new bytes[](1);
    callsA[0] = abi.encodeWithSelector(
        router.exactInputSingle.selector,
        ExactInputSingleParams({tokenIn: WETH, amountIn: amountA, ...})
    );
    router.multicall{value: 1 ether}(callsA);
    // Router now holds 0.5 ETH belonging to userA

    assertEq(address(router).balance, 0.5 ether);

    // Step 2: userB swaps with no ETH sent; pay() consumes userA's residual
    uint256 userBWethBefore = weth.balanceOf(userB);
    vm.prank(userB);
    router.exactInputSingle(
        ExactInputSingleParams({tokenIn: WETH, amountIn: amountB, ...})
    );

    // userB paid 0 ETH and 0 WETH; userA's ETH funded the swap
    assertEq(weth.balanceOf(userB), userBWethBefore, "userB WETH unchanged");
    assertEq(address(router).balance, 0.2 ether, "only 0.3 ETH consumed from userA");
    // userA lost 0.3 ETH with no recourse
}
```

The pool receives the correct WETH amount (the invariant holds from the pool's perspective), but the **source of funds** is wrong: A's ETH, not B's WETH allowance, settled B's obligation.
