Audit Report

## Title
u64 Integer Underflow in `handle_construction_parse` via Attacker-Controlled `ingress_expiry` — (`rs/rosetta-api/icrc1/src/construction_api/utils.rs`)

## Summary

`handle_construction_parse` performs a bare `u64` subtraction on an `ingress_expiry` value extracted directly from attacker-supplied CBOR-encoded transaction bytes, with no lower-bound guard. Any unauthenticated caller submitting a crafted transaction with `ingress_expiry < 240_000_000_000` (ns) triggers the underflow. In debug/overflow-checked builds this panics and crashes the Rosetta process; in release builds it silently wraps, producing a nonsensical `ingress_start` ~584 years in the future returned in an HTTP 200 response.

## Finding Description

The bare subtraction is at: [1](#0-0) 

The subtrahend is fixed at `(MAX_INGRESS_TTL − PERMITTED_DRIFT).as_nanos() as u64` = **240,000,000,000 ns**: [2](#0-1) 

`ingress_expiry_start` is the raw minimum value from attacker-supplied envelope bytes, with no validation: [3](#0-2) 

`construction_parse` extracts `ingress_expiry_start` and passes it directly to `handle_construction_parse` without any range check: [4](#0-3) 

There is no guard anywhere between deserialization and the subtraction. The `handle_construction_parse` function signature accepts `ingress_expiry_start: Option<u64>` and the `.map(|start| start - ...)` closure performs the subtraction unconditionally on any `Some` value. [5](#0-4) 

## Impact Explanation

This maps to **High ($2,000–$10,000): Significant Rosetta/financial-integration security impact with concrete user or protocol harm.**

| Build mode | Behavior | Effect |
|---|---|---|
| Debug / overflow-checked | Rust panics on u64 underflow | Rosetta process crashes → application-level DoS |
| Release | Wraps to `u64::MAX − 240_000_000_000 + 1 + start` | `ingress_start` set ~584 years in the future; HTTP 200 returned with structurally valid but semantically corrupt metadata |

The release-mode wrap is the more dangerous outcome for a production financial API: the response is HTTP 200 with valid JSON, but the embedded `ingress_start` metadata is nonsensical. Any conforming Rosetta client using this field to gate transaction submission will silently discard the user's signed transaction with no error signal, constituting concrete user harm in a financial integration component.

## Likelihood Explanation

The `/construction/parse` endpoint is public and unauthenticated. Crafting a CBOR-encoded `UnsignedTransaction` or `SignedTransaction` with `ingress_expiry = 0` requires only knowledge of the open-source CBOR schema. No key material, privileged role, or network-level position is required. The attack is trivially repeatable and requires a single HTTP POST.

## Recommendation

Replace the bare subtraction with a checked variant and propagate an error on underflow, consistent with the function's existing `anyhow::Result` error-handling style:

```rust
ingress_start: ingress_expiry_start.map(|start| {
    let interval = (ic_limits::MAX_INGRESS_TTL - ic_limits::PERMITTED_DRIFT)
        .as_nanos() as u64;
    start.checked_sub(interval)
        .ok_or_else(|| anyhow::anyhow!(
            "ingress_expiry {} is too small to compute ingress_start \
             (must be >= {} ns)", start, interval
        ))
}).transpose()?
```

This converts the arithmetic error into a proper `anyhow::Result` propagation and surfaces it as an HTTP error to the caller rather than crashing or silently corrupting the response.

## Proof of Concept

```rust
// Craft a minimal UnsignedTransaction with ingress_expiry = 0
let envelope_content = EnvelopeContent::Call {
    canister_id: Principal::anonymous(),
    method_name: "icrc1_transfer".to_string(),
    arg: /* valid Candid-encoded TransferArg */,
    nonce: Some(vec![0u8; 8]),
    sender: Principal::anonymous(),
    ingress_expiry: 0,  // triggers underflow
};
let unsigned_tx = UnsignedTransaction { envelope_contents: vec![envelope_content] };
let tx_string = unsigned_tx.to_string();

// POST to /construction/parse with transaction_is_signed = false
// Debug build  → panic (process crash, DoS)
// Release build → HTTP 200, ingress_start = u64::MAX - 240_000_000_000 + 1
let result = construction_parse(tx_string, false, currency);
```

A unit test in `rs/rosetta-api/icrc1/src/construction_api/services.rs` calling `construction_parse` with `ingress_expiry: 0` will reproduce the panic in debug mode and the wrap in release mode, confirming both behaviors without any mainnet interaction.
