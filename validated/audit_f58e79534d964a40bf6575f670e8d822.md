Audit Report

## Title
SNS Governance `distribute_rewards` Settles Proposals Belonging to the Next Reward Round — (`File: rs/sns/governance/src/governance.rs`)

## Summary
`ready_to_be_settled_proposal_ids` in SNS governance captures `now` internally and returns all proposals whose voting period has closed as of the current wall-clock time. `distribute_rewards` calls this function without a round-boundary argument, then computes `reward_event_end_timestamp_seconds` as a value strictly ≤ `now`. Any proposal whose voting deadline falls in the gap `(reward_event_end_timestamp_seconds, now]` is swept into the current reward event even though it belongs to the next round, causing premature settlement, permanent ballot data loss, and reward dilution for current-round voters.

## Finding Description
`ready_to_be_settled_proposal_ids` captures `now = self.env.now()` and filters on `reward_status(now) == ReadyToSettle`: [1](#0-0) 

`distribute_rewards` calls this with no timestamp argument at line 5822–5823, collecting all proposals that are `ReadyToSettle` as of the current instant: [2](#0-1) 

Immediately after, `reward_event_end_timestamp_seconds` is computed by flooring `now` to the nearest completed round boundary: [3](#0-2) 

The gap `now − reward_event_end_timestamp_seconds` can be anywhere from 0 to `round_duration_seconds − 1` seconds. Any proposal whose voting deadline falls in this gap is `ReadyToSettle` as of `now` but belongs to the next round. It is included in `considered_proposals` and settled — ballots cleared, maturity credited — under a reward event whose `end_timestamp_seconds` predates the proposal's close time.

NNS governance avoids this by accepting an explicit `as_of_timestamp_seconds` parameter in its equivalent function: [4](#0-3) 

And passing the truncated round-end timestamp computed by `most_recent_fully_elapsed_reward_round_end_timestamp_seconds`: [5](#0-4) 

SNS governance has no equivalent guard. The `reward_status` function in SNS proposal logic confirms that any proposal past its voting deadline and eligible for rewards returns `ReadyToSettle` for any `now_seconds` past the deadline: [6](#0-5) 

## Impact Explanation
This is a **High** severity SNS governance impact. Every invocation of `distribute_rewards` where at least one proposal's voting deadline falls in the gap window causes: (1) permanent ballot data loss — once settled, ballots are cleared with no recourse; (2) incorrect reward accounting — the rewards purse is sized for `new_rounds_count` completed rounds but is diluted across extra proposals from the next round, reducing per-neuron maturity for legitimate current-round voters; (3) premature maturity credit — a neuron that is the dominant voter on a proposal closing just after the round boundary receives full proportional reward one full round early. This constitutes concrete, irreversible harm to SNS governance participants and matches the allowed impact: *Significant SNS security impact with concrete user or protocol harm.*

## Likelihood Explanation
The bug is deterministic and structural. `distribute_rewards` is triggered by `should_distribute_rewards` firing as soon as `seconds_since_last_reward_event > round_duration_seconds`: [7](#0-6) 

The gap window is non-empty in virtually every real reward distribution cycle because the timer fires at an arbitrary second within the new round. No special privilege beyond the ability to submit proposals (standard SNS governance participation) is required. An attacker can deterministically time a proposal's `initial_voting_period_seconds` so its deadline falls just after the round boundary, guaranteeing it is swept into the prior round's reward event.

## Recommendation
Mirror the NNS approach in SNS governance:
1. Add an `as_of_timestamp_seconds: u64` parameter to `ready_to_be_settled_proposal_ids` in `rs/sns/governance/src/governance.rs`.
2. In `distribute_rewards`, compute the round-boundary timestamp — `reward_event_end_timestamp_seconds` (already computed at line 5837) — before calling `ready_to_be_settled_proposal_ids`, and pass it as the `as_of_timestamp_seconds` argument.
3. This ensures only proposals whose voting period closed on or before the end of the most recently completed reward round are settled in the current event, exactly mirroring the NNS guard at lines 6658–6662 of `rs/nns/governance/src/governance.rs`.

## Proof of Concept
**Setup:** SNS with `round_duration_seconds = 604800`. Last reward event `end_timestamp_seconds = T₀`. Neuron A submits Proposal P with `initial_voting_period_seconds = 604801` at time `T₀ − 1`. Proposal P's deadline = `T₀ + 604800`.

**Execution:**
1. At `T₀ + 604800 + 2`, `should_distribute_rewards` returns `true` (`seconds_since_last_reward_event = 604801 > 604800`).
2. `distribute_rewards` runs: `new_rounds_count = floor(604801 / 604800) = 1`; `reward_event_end_timestamp_seconds = T₀ + 604800`.
3. `ready_to_be_settled_proposal_ids()` evaluates `reward_status(now = T₀ + 604800 + 2)`: Proposal P's deadline `T₀ + 604800 < now` → `accepts_vote = false` → `ReadyToSettle`.
4. Proposal P is included in `considered_proposals` and settled under the reward event with `end_timestamp_seconds = T₀ + 604800`, even though P closed 1 second into round 2.
5. Neuron A receives full proportional maturity for Proposal P in round 1. In round 2, Proposal P is already `Settled`; no other neuron can earn rewards for it.

A deterministic integration test using PocketIC can reproduce this by: setting `round_duration_seconds`, advancing the mock clock to `T₀ + 604800 + 2`, submitting a proposal with the crafted deadline, triggering `run_periodic_tasks`, and asserting that the resulting `RewardEvent` contains Proposal P despite its deadline falling after `reward_event_end_timestamp_seconds`.

    pub fn ready_to_be_settled_proposal_ids(
        &self,
        as_of_timestamp_seconds: u64,
    ) -> impl Iterator<Item = ProposalId> + '_ {
        self.heap_data
            .proposals
            .iter()
            .filter(move |(_, proposal)| {
                let topic = proposal.topic();
                let voting_period_seconds = self.voting_period_seconds()(topic);

                reward_status == ProposalRewardStatus::ReadyToSettle
            })
            .map(|(k, _)| ProposalId { id: *k })
    }
