### Title
`validateCOAOwnershipProof` Does Not Bind Signatures to `evmAddress`, Enabling Cross-COA Replay - (File: fvm/evm/stdlib/contract.cdc)

### Summary

`validateCOAOwnershipProof` in the EVM Cadence contract verifies signatures over `signedData` without cryptographically binding them to the specific `evmAddress`. This is the direct Flow analog of the ENS coordinate-system mismatch: an incomplete formula is used in signature validation, and the resulting gap is masked by a usage-pattern assumption (accounts typically hold only one COA). If an account holds multiple COAs, a signature produced for one COA can be replayed to prove ownership of a different COA belonging to the same account.

### Finding Description

In `fvm/evm/stdlib/contract.cdc`, `validateCOAOwnershipProof` performs two independent checks:

1. **Signature check** — verifies that the supplied signatures are valid over `signedData` using the account's keys:
   ```cadence
   let isValid = keyList.verify(
       signatureSet: signatureSet,
       signedData: signedData,
       domainSeparationTag: "FLOW-V0.0-user"
   )
   ```
2. **Address check** — after signature verification, borrows the COA resource from `path` and compares its EVM address to the caller-supplied `evmAddress`:
   ```cadence
   if let coaRef = acc.capabilities.borrow<&EVM.CadenceOwnedAccount>(path) {
       let coaAddressBytes = coaRef.address().bytes
       for index, item in coaAddressBytes {
           if item != evmAddress[index] { ... }
       }
   ```

The two checks are **independent**: the signature is never bound to `evmAddress`. The code comment explicitly acknowledges this gap:

> *"Note: this function does not enforce that `signedData` includes `evmAddress`. In principle, a signature produced for one purpose could be replayed here against a different COA owned by the same Cadence account."*

The masking mechanism is the assumption that each account holds at most one COA and that the EVM-side precompile (`verifyCOAOwnershipProof`) always passes the calling COA's own address. Both assumptions can be violated: `createCadenceOwnedAccount()` can be called multiple times, and `validateCOAOwnershipProof` is `access(all)` — callable directly from any Cadence script or transaction, bypassing the precompile's address constraint entirely.

The entry path through the EVM precompile is `fvm/evm/handler/precompiles.go` → `coaOwnershipProofValidator` → `backend.Invoke("validateCOAOwnershipProof", ...)`. An attacker can also call `validateCOAOwnershipProof` directly from a Cadence script, or deploy an EVM contract that calls the Cadence Arch precompile `verifyCOAOwnershipProof(address,bytes32,bytes)` with a replayed proof.

### Impact Explanation

If Alice holds two COAs — `COA_A` at `/public/coaA` (EVM address `0xAAA`) and `COA_B` at `/public/coaB` (EVM address `0xBBB`) — and she signs `signedData` to prove ownership of `COA_A` to some EVM contract, an attacker who observes that on-chain signature can call:

```cadence
EVM.validateCOAOwnershipProof(
    address: alice_address,
    path: /public/coaB,      // ← different COA
    signedData: signedData,  // ← same data Alice signed
    keyIndices: [alice_key_index],
    signatures: [alice_signature],
    evmAddress: 0xBBB        // ← COA_B's address
)
```

Step 1 passes: Alice's signature is cryptographically valid over `signedData`.  
Step 2 passes: the COA at `/public/coaB` has address `0xBBB`.  
Result: `isValid: true` — the attacker has proven ownership of `COA_B` using a signature Alice only intended for `COA_A`.

This allows the attacker to pass ERC-1271 (`isValidSignature`) checks on behalf of `COA_B` in any EVM contract that relies on `verifyCOAOwnershipProof`, enabling unauthorized authorization of EVM actions tied to `COA_B`.

### Likelihood Explanation

Likelihood is **low-medium**. Prerequisites are:
- The target Flow account must hold more than one COA (achievable by calling `createCadenceOwnedAccount()` multiple times).
- The attacker must observe a valid signature from the account owner over some `signedData` (realistic: on-chain signatures are public).
- The attacker must know the public path of the second COA (public paths are enumerable).

The masking assumption (one COA per account) is a social/wallet convention, not a protocol-level enforcement. As multi-COA usage grows, the attack surface widens.

### Recommendation

Enforce that `signedData` encodes `evmAddress` before signature verification, or include `evmAddress` as part of the domain-separation tag. The simplest fix is to prepend `evmAddress` to `signedData` inside `validateCOAOwnershipProof` before passing it to `keyList.verify`, so that a signature for `COA_A` cannot satisfy the check for `COA_B`.

### Proof of Concept

```
1. Alice calls createCadenceOwnedAccount() twice:
      COA_A stored at /storage/coaA, published at /public/coaA → EVM address 0xAAA
      COA_B stored at /storage/coaB, published at /public/coaB → EVM address 0xBBB

2. Alice signs signedData = 0xdeadbeef... to prove ownership of COA_A
   to some EVM contract (signature goes on-chain).

3. Attacker observes alice_signature on-chain.

4. Attacker (from a Cadence script or via an EVM contract calling the
   Cadence Arch precompile) invokes:

   EVM.validateCOAOwnershipProof(
       address:      alice_address,
       path:         /public/coaB,
       signedData:   0xdeadbeef...,
       keyIndices:   [0],
       signatures:   [alice_signature],
       evmAddress:   0xBBB
   )

5. keyList.verify() → true  (signature is valid over signedData)
6. coaRef.address().bytes == 0xBBB → true
7. Returns ValidationResult(isValid: true, problem: nil)

8. Attacker has proven ownership of COA_B using a signature Alice
   only produced for COA_A.
```

		value, err := backend.Invoke(
			environment.ContractFunctionSpec{
				AddressFromChain: func(_ flow.Chain) flow.Address {
					return contractAddress
				},
				LocationName: "EVM",
				FunctionName: "validateCOAOwnershipProof",
				ArgumentTypes: []sema.Type{
					types.FlowAddressSemaType,
					types.PublicPathSemaType,
					types.SignedDataSemaType,
					types.KeyIndicesSemaType,
					types.SignaturesSemaType,
					types.AddressBytesSemaType,
				},
			},
			proof.ToCadenceValues(),
		)
		if err != nil {
			if types.IsAFatalError(err) {
				panic(err)
			}
			return false, err
		}
		data, ok := value.(cadence.Struct)
		if !ok {
			return false, fmt.Errorf("invalid output data received from validateCOAOwnershipProof")
		}

		isValidValue := cadence.SearchFieldByName(data, ValidationResultTypeIsValidFieldName)
		if isValidValue == nil {
			return false, fmt.Errorf("invalid output data received from validateCOAOwnershipProof")
		}

		return bool(isValidValue.(cadence.Bool)), nil
	}
