### Title
Pool Admin Bypasses `maxAdminSpreadFeeE6` Cap via Uncapped `setPoolBinAdditionalFees`, Causing Traders to Pay Excess Fees — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary
`MetricOmmPoolFactory.setPoolBinAdditionalFees` forwards `addFeeBuyE6`/`addFeeSellE6` to the pool with no validation against `maxAdminSpreadFeeE6`, while the parallel path `setPoolAdminFees` does enforce that cap. A pool admin can set per-bin fees up to `uint16` max (65,535 E6 ≈ 6.55%) on any bin regardless of the factory-configured cap, causing traders to pay uncapped fees on swaps through that bin.

### Finding Description
`setPoolAdminFees` checks `newAdminSpreadFeeE6 > maxAdminSpreadFeeE6` and reverts with `AdminFeeTooHigh` if exceeded. [1](#0-0) 

`setPoolBinAdditionalFees` performs no such check and directly calls `pool.setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6)`. [2](#0-1) 

During every swap, the effective bin fee is computed as `params.baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6)`, so the uncapped `addFeeBuyE6` directly inflates the price paid by the trader. [5](#0-4) 

The same uncapped path applies to the sell direction (`addFeeSellE6`). [6](#0-5) 

### Impact Explanation
Traders swapping through a bin with `addFeeBuyE6 = 65535` pay up to ~6.55% extra above the oracle-implied price on that bin, regardless of what `maxAdminSpreadFeeE6` is configured to. This is a direct loss of trader principal. The factory's fee-cap invariant — that pool admin fees are bounded by `maxAdminSpreadFeeE6` — is broken via this alternative path.

### Likelihood Explanation
The pool admin is a semi-trusted role that is explicitly bounded by factory caps. The bypass requires only a single call to `setPoolBinAdditionalFees` with `addFeeBuyE6 = 65535`. No special conditions, timelocks, or external dependencies are needed. Any pool admin can execute this immediately.

### Recommendation
Add a cap check in `setPoolBinAdditionalFees` before forwarding to the pool:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (addFeeBuyE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    if (addFeeSellE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

### Proof of Concept
1. Deploy factory with `maxAdminSpreadFeeE6 = 1_000` (0.1%).
2. Create a pool with `adminSpreadFeeE6 = 1_000`.
3. As pool admin, call `factory.setPoolBinAdditionalFees(pool, 0, 65535, 65535)` — succeeds with no revert.
4. Execute a swap through bin 0.
5. Observe that the effective buy fee applied is `baseFeeX64 + mulDiv(65535, ONE_X64, 1e6)` ≈ 6.55% extra, far exceeding the 0.1% cap.
6. Assert trader input exceeds oracle-implied cost by ~6.55%, not the capped 0.1%.
