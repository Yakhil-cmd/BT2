### Title
Token Accounting Imbalance via Integer Truncation in `delta_gas` Calculation — (`File: basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`)

---

### Summary

In `compute_gas_refund`, the `delta_gas` adjustment that reconciles native resource consumption with EVM gas used applies only the **positive** direction (native > gas). When native consumption is **less** than gas consumption (`delta_gas < 0`), the code silently discards the negative delta, leaving `gas_used` higher than what native resource consumption justifies. This causes the sender to be charged more tokens than the system actually consumed, with the excess tokens neither refunded to the sender nor paid to the operator — they are simply destroyed from the sender's balance. Over many transactions, this creates a systematic token accounting imbalance where the sum of all account balances is less than the total supply.

---

### Finding Description

In `compute_gas_refund` in `basic_bootloader/src/bootloader/transaction_flow/refund_calculation.rs`, the `delta_gas` reconciliation logic is:

```rust
let delta_gas = if native_per_gas == 0 {
    0
} else {
    (native_used / native_per_gas) as i64 - (gas_used as i64)
};

if delta_gas > 0 {
    // In this case, the native resource consumption is more than the
    // gas consumption accounted for. Consume extra gas.
    gas_used += delta_gas as u64;
}
// TODO: return delta_gas to gas_used?
```

The `TODO` comment on line 80 explicitly acknowledges the unresolved case. When `delta_gas < 0` — meaning `native_used / native_per_gas < gas_used` — the code does **not** reduce `gas_used`. This means `gas_used` remains inflated relative to actual native consumption.

The fee flow for ZK (L2) transactions is:
1. **Precharge**: sender pays `fee_to_prepay = gas_price * gas_limit` upfront.
2. **Refund**: sender receives back `gas_price * (gas_limit - gas_used)`.
3. **Operator payment**: operator receives `gas_price * gas_used` (or `gas_price_for_operator * gas_used` under `burn_base_fee`).

When `delta_gas < 0` is silently dropped, `gas_used` is larger than it should be. The refund is smaller than it should be, and the operator payment is larger than it should be. However, the operator payment is bounded by `gas_used * gas_price_for_operator`, and the refund is `gas_price * (gas_limit - gas_used)`. The total of refund + operator payment equals `gas_price * gas_limit` only when `gas_price == gas_price_for_operator`. Under `burn_base_fee`, `gas_price_for_operator = gas_price - base_fee`, so:

- Tokens deducted from sender: `gas_price * gas_limit`
- Tokens returned to sender: `gas_price * (gas_limit - gas_used)`
- Tokens paid to operator: `(gas_price - base_fee) * gas_used`
- Tokens "burned" (base fee): `base_fee * gas_used`

The burned base fee tokens are **not credited to any account** — they are simply subtracted from the sender's balance and not added anywhere. This is the intended behavior for base fee burning. However, the `delta_gas < 0` case means `gas_used` is over-counted, so more tokens are burned than the actual computation cost warrants. The excess burned amount is `base_fee * |delta_gas|` per transaction.

Even without `burn_base_fee`, the `delta_gas < 0` case causes the operator to receive more tokens than the native resource consumption justifies, creating an imbalance between what was consumed and what was paid.

The `delta_gas < 0` case arises naturally when:
- `native_used / native_per_gas < gas_used` (integer division truncates)
- The EVM gas path consumed more ergs than the native path consumed native units
- The `minimal_gas_used` floor raises `gas_used` above what native consumption implies [1](#0-0)

### Impact Explanation

**Token accounting imbalance (resource accounting bug):** The sum of all account balances on the L2 chain will be less than the total token supply after many transactions. The discrepancy grows with each transaction where `delta_gas < 0`. Under `burn_base_fee`, the excess burned amount per transaction is `base_fee * max(0, gas_used - native_used/native_per_gas)`. This is analogous to the StWSX.sol finding: a rounding/truncation error causes the total supply to diverge from the sum of balances.

**Concrete impact:** Users are systematically overcharged. The excess tokens are neither refunded nor credited to any account — they are effectively destroyed. Over time, the L2 base token supply becomes inconsistent with the sum of all balances, which can break invariants relied upon by bridges, asset trackers, and the `L2AssetTracker` contract. [2](#0-1) 

---

### Likelihood Explanation

This condition is reachable by any unprivileged transaction sender. The `delta_gas < 0` case occurs whenever `native_used / native_per_gas < gas_used` after integer division. This is a normal outcome for transactions where:
- EVM gas consumption is high relative to native resource consumption
- The `minimal_gas_used` floor (from EIP-7623 or intrinsic gas) raises `gas_used` above the native-implied value
- `native_per_gas` is large enough that `native_used / native_per_gas` truncates significantly

The `TODO` comment in the code confirms the developers are aware this case is unhandled. The condition is triggered on every ordinary L2 transaction where native consumption is lower than EVM gas consumption, which is a common scenario. [3](#0-2) 

---

### Recommendation

When `delta_gas < 0`, reduce `gas_used` by `|delta_gas|` (subject to the `minimal_gas_used` floor), so that the sender is refunded the excess:

```rust
if delta_gas > 0 {
    gas_used += delta_gas as u64;
} else if delta_gas < 0 {
    // Native consumption implies fewer gas units were used.
    // Reduce gas_used to avoid overcharging the sender.
    let reduction = (-delta_gas) as u64;
    gas_used = gas_used.saturating_sub(reduction).max(minimal_gas_used);
}
```

This ensures that `gas_used` accurately reflects the maximum of EVM gas consumption and native-implied gas consumption, and that the sender is refunded the difference. The `minimal_gas_used` floor must still be respected to prevent undercharging for intrinsic costs. [4](#0-3)

### Proof of Concept

**Setup:**
- `gas_limit = 100_000`
- `gas_price = 1000`
- `native_price = 100` → `native_per_gas = ceil(1000/100) = 10`
- `native_limit = 100_000 * 10 = 1_000_000`
- Transaction executes, consuming `native_used = 150_000` native units and `gas_used_ergs = 20_000` EVM gas
- `minimal_gas_used = 21_000` (intrinsic gas floor)

**Computation in `compute_gas_refund`:**
1. `gas_used = 100_000 - remaining_ergs/ERGS_PER_GAS = 20_000` (from ergs)
2. After `max(gas_used, minimal_gas_used)`: `gas_used = 21_000`
3. `native_used = 1_000_000 - remaining_native = 150_000`
4. `delta_gas = (150_000 / 10) as i64 - 21_000 as i64 = 15_000 - 21_000 = -6_000`
5. Since `delta_gas < 0`, the branch is **not taken**. `gas_used` stays at `21_000`.

**Token accounting:**
- Sender was precharged: `1000 * 100_000 = 100_000_000`
- Sender refunded: `1000 * (100_000 - 21_000) = 79_000_000`
- Operator paid: `1000 * 21_000 = 21_000_000` (no `burn_base_fee`)
- Total returned: `79_000_000 + 21_000_000 = 100_000_000` ✓ (balanced without `burn_base_fee`)

**With `burn_base_fee` enabled** (`base_fee = 800`, `priority_fee = 200`):
- `gas_price_for_operator = 1000 - 800 = 200`
- Sender precharged: `1000 * 100_000 = 100_000_000`
- Sender refunded: `1000 * 79_000 = 79_000_000`
- Operator paid: `200 * 21_000 = 4_200_000`
- Total returned: `79_000_000 + 4_200_000 = 83_200_000`
- **Destroyed (burned):** `100_000_000 - 83_200_000 = 16_800_000`
- **Correct burn** (if `delta_gas` were applied): `800 * 15_000 = 12_000_000`
- **Excess burned due to bug:** `800 * 6_000 = 4_800_000` tokens permanently destroyed per transaction

This excess destruction accumulates across all transactions, causing the sum of balances to diverge from total supply. [5](#0-4) [1](#0-0)
