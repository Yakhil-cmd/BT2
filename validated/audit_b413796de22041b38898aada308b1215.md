### Title
Missing Caller Validation on `update_rewards` Allows Any Address to Permanently Block Consensus Reward Distribution - (File: `src/staking/staking.cairo`)

---

### Summary

The `update_rewards` function in the staking contract is documented to be callable only by the Starkware sequencer, but the implementation contains no caller check. Any unprivileged address can call it with `disable_rewards: true`, consuming the single per-block reward slot and permanently preventing all stakers from accumulating consensus rewards.

---

### Finding Description

The protocol specification at `docs/spec.md` lines 1644–1645 explicitly states:

> **access control:** Only starkware sequencer.

However, the implementation of `IStakingRewardsManager::update_rewards` in `src/staking/staking.cairo` (lines 1447–1488) performs no such check:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();          // only checks paused flag
    let current_block_number = starknet::get_block_number();
    assert!(
        current_block_number > self.last_reward_block.read(),
        "{}",
        Error::REWARDS_ALREADY_UPDATED,
    );
    // ... no caller identity check ...
    self.last_reward_block.write(current_block_number);  // line 1485

    if disable_rewards || self.is_pre_consensus() {
        return;                                           // line 1487-1488
    }
    // reward distribution follows
```

`last_reward_block` is a **single global storage slot** (not per-staker). Once written for block N, every subsequent call in block N reverts with `REWARDS_ALREADY_UPDATED`. A grep across the entire repository confirms there is no `only_sequencer`, `CALLER_IS_NOT_SEQUENCER`, or equivalent guard anywhere in the codebase. [1](#0-0) [2](#0-1) 

---

### Impact Explanation

An attacker who calls `update_rewards(valid_staker, disable_rewards: true)` in every block:

1. Writes `last_reward_block = current_block` (line 1485) without distributing any rewards.
2. The early return at line 1487 ensures `staker_info.unclaimed_rewards_own` is never incremented and no pool rewards are transferred.
3. Every subsequent call in that block — including the legitimate sequencer call — reverts with `REWARDS_ALREADY_UPDATED`.
4. Repeating this every block permanently prevents all stakers from accumulating consensus-phase block rewards.

This constitutes **permanent freezing of unclaimed yield** for all stakers and delegation pool members across the entire protocol. [3](#0-2) [4](#0-3) 

---

### Likelihood Explanation

- **No privilege required.** The call is open to any externally-owned address.
- **Minimal cost.** One transaction per block on Starknet is cheap.
- **Public inputs.** Any valid `staker_address` can be read from on-chain events (`NewStaker`).
- **No detection/prevention.** The contract has no rate-limiting, no allowlist, and no sequencer identity mechanism. [5](#0-4) 

---

### Recommendation

Add a caller check at the top of `update_rewards` that asserts `get_caller_address()` equals the stored sequencer address (or a role equivalent to `only_sequencer`), consistent with the access-control specification. This mirrors the pattern already used for `update_unclaimed_rewards_from_staking_contract` and `claim_rewards` in the reward supplier, which both assert `get_caller_address() == staking_contract`. [6](#0-5) 

---

### Proof of Concept

```
1. Deploy the system normally and advance past `consensus_rewards_first_epoch`.
2. Stake as a legitimate staker S (address is public via NewStaker event).
3. Advance K epochs so S has non-zero balance.
4. From any unprivileged address A (not the sequencer):
       staking.update_rewards(staker_address: S, disable_rewards: true)
   → Succeeds. last_reward_block = current_block. No rewards distributed.
5. Sequencer attempts:
       staking.update_rewards(staker_address: S, disable_rewards: false)
   → Reverts: "REWARDS_ALREADY_UPDATED".
6. Repeat step 4 every block.
   → S.unclaimed_rewards_own remains 0 indefinitely.
   → All delegation pool members also receive 0 rewards.
``` [1](#0-0) [7](#0-6)
