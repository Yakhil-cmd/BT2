### Title
Signature Validation Bypass via Overly Broad `skip_stateful_validations` Mempool Presence Check Admits Invalid Invoke Transactions — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator skips the `__validate__` entry-point call for invoke transactions with `nonce == 1` when `account_nonce == 0` and `account_tx_in_pool_or_recent_block` returns `true`. The mempool presence check is not restricted to deploy-account transactions — it returns `true` for **any** transaction from the sender. An unprivileged attacker can therefore first submit a valid deploy-account for a fresh address (causing the presence check to return `true`), then immediately submit an invoke with `nonce = 1` carrying an **invalid signature**. The invalid invoke bypasses `__validate__` entirely and is admitted to the mempool.

---

### Finding Description

**Root cause — `skip_stateful_validations` uses an overly broad presence check:** [1](#0-0) 

```rust
async fn skip_stateful_validations(...) -> ... {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                ...
        }
    }
    Ok(false)
}
```

When this function returns `true`, `run_validate_entry_point` sets `execution_flags.validate = false`: [2](#0-1) 

This causes `StatefulValidator::perform_validations` to return `Ok(())` without ever calling `tx.validate_tx` (the `__validate__` entry point): [3](#0-2) 

**The mempool presence check is not type-restricted:** [4](#0-3) 

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
```

`tx_pool.contains_account` returns `true` for **any** transaction type (invoke, declare, deploy-account) from the sender. The code comment claims this is safe because "it means that either it has a deploy_account transaction or transactions with future nonces that passed validations," but this reasoning is incorrect for the deploy-account case itself: the deploy-account transaction is in the pool *before* it has been executed, so the account contract does not yet exist and no signature has been verified against it.

**The `validate_by_mempool` call that precedes the skip check does not verify signatures:** [5](#0-4) 

It only checks nonce ordering and duplicate-hash rules, not cryptographic validity.

**The `max_nonce_for_validation_skip` config field is defined but never consulted in this code path:** [6](#0-5) 

The gateway hardcodes the threshold to `Nonce(Felt::ONE)` regardless of operator configuration.

---

### Impact Explanation

An attacker can inject an invoke transaction carrying an **arbitrary (invalid) signature** into the mempool without passing the account's `__validate__` entry point. This violates the gateway's core admission invariant. The invalid transaction occupies a mempool slot, consumes batcher execution resources when dequeued, and — if the account has been funded — results in a fee-charged revert. Repeated across many fresh addresses this constitutes a targeted mempool-spam / resource-exhaustion attack.

**Impact category matched:** *High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.*

---

### Likelihood Explanation

The attack is fully unprivileged and requires only two standard RPC calls per iteration:

1. `add_transaction` with a valid deploy-account for a fresh address (costs one deploy-account fee).
2. `add_transaction` with an invoke at `nonce = 1` carrying a zeroed or random signature.

No special knowledge of internals is required; the mempool state is observable. The cost per injected invalid transaction is bounded by the deploy-account fee, making sustained spam economically feasible.

---

### Recommendation

1. **Restrict the presence check to deploy-account transactions.** The mempool should expose a dedicated `has_deploy_account_in_pool(address)` query, or `account_tx_in_pool_or_recent_block` should be replaced with a type-aware check so that only a pending deploy-account (not an arbitrary invoke or declare) triggers the skip.

2. **Honour `max_nonce_for_validation_skip` in the gateway path.** The config field is currently dead code in `skip_stateful_validations`; wire it in so operators can tighten or disable the skip window.

3. **Consider requiring a minimum balance check before skipping validation**, so that even if the skip fires, the transaction cannot be fee-free.

---

### Proof of Concept

```
# Step 1 – attacker submits a valid deploy-account for fresh address X
POST /gateway/add_transaction
{ "type": "DEPLOY_ACCOUNT", "sender_address": X, "nonce": "0x0",
  "signature": [<valid sig>], ... }
→ accepted; mempool now has deploy-account for X
→ account_tx_in_pool_or_recent_block(X) == true

# Step 2 – attacker submits invoke with nonce=1 and INVALID signature
POST /gateway/add_transaction
{ "type": "INVOKE", "sender_address": X, "nonce": "0x1",
  "signature": ["0x0", "0x0"], ... }

Gateway path:
  validate_nonce:              0 <= 1 <= 200  → OK
  validate_by_mempool:         no dup hash, nonce in range → OK
  skip_stateful_validations:   nonce==1, account_nonce==0,
                               account_tx_in_pool_or_recent_block==true → returns true
  run_validate_entry_point:    execution_flags.validate = false
                               perform_validations returns Ok() without __validate__
  mempool.add_tx:              invalid-signature invoke admitted to mempool ✓
```

The invalid invoke now sits in the mempool. When the batcher dequeues it, `__validate__` is called during execution, the transaction reverts, and the fee is charged — but the admission invariant has already been broken.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
