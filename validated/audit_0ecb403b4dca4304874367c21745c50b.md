### Title
Pool Admin Can Set Uncapped Per-Bin Additional Fees via `setPoolBinAdditionalFees`, Bypassing the Admin Fee Cap System — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`MetricOmmPoolFactory.setPoolBinAdditionalFees` forwards `addFeeBuyE6` and `addFeeSellE6` directly to `MetricOmmPool.setBinAdditionalFees` with **no upper-bound validation**. A pool admin can set either value to `type(uint16).max` (65535, ≈ 6.55% in E6 units), while the protocol's own cap system — enforced for every other admin fee setter — is entirely absent for per-bin fees.

---

### Finding Description

`setPoolAdminFees` enforces hard caps before writing:

```solidity
// MetricOmmPoolFactory.sol:414-415
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (newAdminNotionalFeeE8 > maxAdminNotionalFeeE8) revert AdminFeeTooHigh();
``` [1](#0-0) 

`setPoolBinAdditionalFees` has no equivalent guard:

```solidity
// MetricOmmPoolFactory.sol:450-457
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
  external override nonReentrant onlyPoolAdmin(pool)
{
  IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
``` [2](#0-1) 

`setBinAdditionalFees` on the pool only validates the bin index, then writes the raw value:

```solidity
// MetricOmmPool.sol:464-474
function setBinAdditionalFees(int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
  external onlyFactory nonReentrant(PoolActions.SET_BIN_ADDITIONAL_FEES)
{
  if (bin < LOWEST_BIN || bin > HIGHEST_BIN) revert InvalidBinIndex(bin);
  BinState storage s = _binStates[bin];
  s.addFeeBuyE6 = addFeeBuyE6;
  s.addFeeSellE6 = addFeeSellE6;
  ...
}
``` [3](#0-2) 

The corrupted value flows directly into swap math for every buy-token0 swap through that bin:

```solidity
// MetricOmmPool.sol:910 (specifiedOut path), 999 (specifiedIn path)
params.baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6)
``` [4](#0-3) [5](#0-4) 

This fee is additive on top of `baseFeeX64` (derived from the oracle spread) and the notional fee, with no ceiling.

---

### Impact Explanation

With `addFeeBuyE6 = 65535` (≈ 6.55%), every trader buying token0 through that bin pays an additional ~6.55% fee on their token1 input, on top of the existing spread and notional fees. The excess fee accrues to the bin's LP balance (as `binLpFeeAmount`), effectively transferring trader principal to LPs at a rate far exceeding any documented or enforced cap. This is a direct, quantifiable loss of user principal per swap through the affected bin. [6](#0-5) 

---

### Likelihood Explanation

The pool admin is a semi-trusted role that is explicitly expected to be bounded by caps (as the contest scope states and as the spread/notional fee setters demonstrate). Any pool admin — including a compromised multisig or a malicious actor who acquires the role — can call this function immediately with no timelock. There is no on-chain mechanism preventing it.

---

### Recommendation

Add a cap check in `setPoolBinAdditionalFees` analogous to the one in `setPoolAdminFees`. Introduce a factory-level constant (e.g., `maxAdminBinFeeE6`) and revert if either `addFeeBuyE6` or `addFeeSellE6` exceeds it:

```solidity
if (addFeeBuyE6 > maxAdminBinFeeE6) revert AdminFeeTooHigh();
if (addFeeSellE6 > maxAdminBinFeeE6) revert AdminFeeTooHigh();
```

---

### Proof of Concept

1. Pool admin calls `factory.setPoolBinAdditionalFees(pool, 0, type(uint16).max, 0)`.
2. `BinState[0].addFeeBuyE6` is now 65535.
3. A trader calls `swap` buying token0 through bin 0.
4. The effective buy fee becomes `baseFeeX64 + mulDiv(65535, ONE_X64, 1e6)` ≈ base spread + 6.55%.
5. The trader's token1 input is inflated by ~6.55% beyond the oracle-implied price plus normal fees.
6. Assert `amountIn > oracle_fair_value * (1 + spreadFee + notionalFee + any_documented_cap)`. [2](#0-1) [3](#0-2)
