### Title
Missing Proof of Correct Computation for CKD Participant Shares Allows Malicious Participant to Corrupt CKD Output — (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

The Confidential Key Derivation (CKD) coordinator in `do_ckd_coordinator` aggregates `(norm_big_y, norm_big_c)` shares received from each participant via `recv_from_others` and sums them directly, with no cryptographic proof that each share was computed using the participant's actual key share. Any malicious participant in the CKD session can send arbitrary group elements in place of their honest contribution, permanently corrupting the CKD output delivered to the TEE. No privileged access is required beyond being a legitimate participant in the protocol.

---

### Finding Description

**Root cause — `src/confidential_key_derivation/protocol.rs`, lines 50–55:**

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
```

Each participant is expected to compute and send:

- `norm_big_y = λ_i · y_i · G` (a blinded randomness contribution)
- `norm_big_c = λ_i · (x_i · H(pk ‖ app_id) + y_i · app_pk)` (the masked key share)

where `x_i` is the participant's private signing share and `y_i` is a fresh random scalar. The coordinator sums all received `(norm_big_y, norm_big_c)` pairs to produce the final `CKDOutput`. The TEE then recovers the confidential key as `C − app_sk · Y = msk · H(pk ‖ app_id)`.

**What is missing:** There is no zero-knowledge proof (e.g., a proof of discrete-log equality or a Schnorr-style proof) attached to each participant's share demonstrating that `norm_big_c` was formed using the participant's committed key share `x_i` and the same randomness `y_i` used in `norm_big_y`. The `recv_from_others` helper (in `src/protocol/helpers.rs`, lines 6–26) only enforces that exactly one message arrives per participant; it performs no cryptographic validation of message content.

**Exploit path:**

1. A malicious participant `P_m` is a legitimate member of the CKD participant set (no privileged access needed).
2. Instead of computing the honest `(norm_big_y, norm_big_c)`, `P_m` sends arbitrary group elements `(Y', C')` to the coordinator.
3. The coordinator sums all contributions including `(Y', C')` without detecting the deviation.
4. The final `CKDOutput` is `(Y_honest + Y', C_honest + C')`, which is cryptographically indistinguishable from a valid output to the coordinator.
5. The TEE decrypts `(C_honest + C') − app_sk · (Y_honest + Y')`, obtaining a value that is not `msk · H(pk ‖ app_id)` — the confidential key is permanently corrupted for this session.

The protocol's authenticated-channel assumption (TLS) means the coordinator knows *who* sent the malicious share, but there is no mechanism to detect *that* the share is malicious before the output is already corrupted.

---

### Impact Explanation

**Impact category: High — Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.**

The TEE receives and stores an incorrect confidential key. Since the CKD output is the sole mechanism for delivering `msk · H(pk ‖ app_id)` to the TEE, a corrupted output means the TEE cannot perform any operation that depends on the correct confidential key. The corruption is undetectable by the coordinator or the TEE without an independent oracle for the correct key. Every CKD session in which the malicious participant is included is permanently compromised.

---

### Likelihood Explanation

Any participant who is legitimately enrolled in a CKD session can trigger this. The attacker needs only:

1. A valid participant identity in the session (no leaked keys, no admin access).
2. The ability to send an arbitrary `CKDOutput`-typed message to the coordinator over the authenticated channel.

This is a one-round attack requiring a single malformed message. The protocol provides no detection or recovery mechanism after the fact.

---

### Recommendation

Attach a zero-knowledge proof of correct computation to each participant's CKD share. Concretely, each participant should prove in zero knowledge that `norm_big_c` and `norm_big_y` were formed consistently with their committed public key share `λ_i · x_i · G2` (available from the DKG output). A proof of discrete-log equality (as already implemented in `src/crypto/proofs/dlogeq.rs`) between the discrete log of `norm_big_y` with respect to `G1` and the discrete log of `norm_big_c − λ_i · x_i · H(pk ‖ app_id)` with respect to `app_pk` would suffice. The coordinator must verify this proof before incorporating the share into the sum.

### Proof of Concept

**Setup:** 3-participant CKD session, participants `P1`, `P2` (honest), `P3` (malicious). Coordinator is `P1`.

**Honest execution:** Each participant computes `(norm_big_y_i, norm_big_c_i)` per `compute_signature_share` and sends to `P1`. `P1` sums all three and returns `CKDOutput`. TEE recovers `msk · H(pk ‖ app_id)`.

**Malicious execution:** `P3` replaces its honest share with `(G1, G1)` (the generator point). The coordinator at lines 50–55 of `src/confidential_key_derivation/protocol.rs` sums without verification:

```
norm_big_y_final = norm_big_y_P1 + norm_big_y_P2 + G1   // ≠ honest sum
norm_big_c_final = norm_big_c_P1 + norm_big_c_P2 + G1   // ≠ honest sum
```

The TEE decrypts `norm_big_c_final − app_sk · norm_big_y_final`, which equals `msk · H(pk ‖ app_id) + G1 − app_sk · G1 = msk · H(pk ‖ app_id) + (1 − app_sk) · G1`. This is not the correct confidential key. The TEE silently accepts the wrong key. No error is raised anywhere in the protocol. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)

async fn do_ckd_coordinator(
    mut chan: SharedChannel,
    participants: ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<CKDOutputOption, ProtocolError> {
    let (mut norm_big_y, mut norm_big_c) =
        compute_signature_share(&participants, me, key_pair, app_id, app_pk, rng)?;

    // Receive everyone's inputs and add them together
    let waitpoint = chan.next_waitpoint();

    for (_, participant_output) in
        recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
    {
        norm_big_y += participant_output.big_y();
        norm_big_c += participant_output.big_c();
    }
    let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
    Ok(Some(ckd_output))
}

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}

pub async fn recv_from_others<T>(
    chan: &SharedChannel,
    waitpoint: u64,
    participants: &ParticipantList,
    me: Participant,
) -> Result<Vec<(Participant, T)>, ProtocolError>
where
    T: serde::de::DeserializeOwned,
{
    let mut seen = ParticipantCounter::new(participants);
    seen.put(me);
    let mut messages = Vec::with_capacity(participants.others(me).count());

    while !seen.full() {
        let (from, msg) = chan.recv(waitpoint).await?;
        if seen.put(from) {
            messages.push((from, msg));
        }
    }

    Ok(messages)
