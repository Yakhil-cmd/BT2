### Title
Missing Validation of Participant Shares in CKD Coordinator Aggregation Allows Output Corruption — (`File: src/confidential_key_derivation/protocol.rs`)

---

### Summary

The CKD coordinator aggregates `(norm_big_y, norm_big_c)` shares received from participants with no proof of correct computation. A single malicious participant can submit arbitrary group elements, causing the coordinator to produce a corrupted `CKDOutput` that the TEE will silently accept and use to derive an incorrect confidential key.

---

### Finding Description

In `do_ckd_coordinator` (`src/confidential_key_derivation/protocol.rs`, lines 50–55), the coordinator receives each participant's `CKDOutput` and unconditionally adds the components into the running aggregate:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
``` [1](#0-0) 

Each honest participant is supposed to compute:

```
y_i  ← random scalar
big_y_i = y_i · G
big_s_i = x_i · H(pk ‖ app_id)
big_c_i = big_s_i + y_i · app_pk
norm_big_y_i = λ_i · big_y_i
norm_big_c_i = λ_i · big_c_i
``` [2](#0-1) 

The coordinator never verifies that the received `(norm_big_y, norm_big_c)` pair satisfies the required relation — specifically, that `big_c_i − big_s_i` and `big_y_i` share the same discrete-log with respect to `app_pk` and `G` respectively (a DLOGEQ / Chaum–Pedersen proof). There is no commitment, no zero-knowledge proof, and no consistency check against the participant's known public key share.

This is the direct analog of the deflationary-token issue: just as `depositedBalance += amount` is incremented before verifying the actual transferred amount, here `norm_big_y += participant_output.big_y()` is accumulated before verifying the participant actually computed the share correctly.

---

### Impact Explanation

The TEE calls `CKDOutput::unmask(app_sk)` on the coordinator's output to recover the confidential key `msk · H(pk ‖ app_id)`. If any participant injects an arbitrary `(δ_y, δ_c)` offset, the TEE computes:

```
(big_c + δ_c) − app_sk · (big_y + δ_y)
= msk · H(pk ‖ app_id) + (δ_c − app_sk · δ_y)
```

The result is an incorrect, attacker-influenced confidential key. Because the TEE has no way to detect the corruption (it only sees the final aggregated point), it will silently accept and use the wrong key. This matches the allowed impact:

> **High: Corruption of CKD outputs so honest parties accept inconsistent or unusable cryptographic outputs.**

---

### Likelihood Explanation

Any single participant in the CKD session can mount this attack. No special privilege is required beyond being a valid member of the participant list. The attacker simply sends a malformed `CKDOutput` to the coordinator in the single protocol round. The coordinator has no mechanism to detect or reject it.

---

### Recommendation

Add a zero-knowledge proof of correct share computation (a Chaum–Pedersen / DLOGEQ proof) to each participant's message. Specifically, each participant should prove:

```
DLOGEQ( big_y_i, G, big_c_i − big_s_i, app_pk )
```

i.e., that `big_y_i = y_i · G` and `big_c_i − big_s_i = y_i · app_pk` for the same `y_i`. The coordinator must verify this proof before including the share in the aggregate. The codebase already contains a `DLOGEQ` proof primitive in `src/crypto/proofs/` that can be reused for this purpose.


---

### Proof of Concept

1. Honest participants `P_1, …, P_{n-1}` run `compute_signature_share` correctly and send their `(norm_big_y_i, norm_big_c_i)` to the coordinator.
2. Malicious participant `P_n` instead sends `(δ_y, δ_c)` — two arbitrary non-zero `ElementG1` values unrelated to its key share.
3. The coordinator's loop at lines 50–55 adds `δ_y` and `δ_c` into the aggregate without any check.
4. The coordinator returns `CKDOutput::new(big_y + δ_y, big_c + δ_c)`.
5. The TEE calls `unmask(app_sk)` and derives `msk · H(pk ‖ app_id) + (δ_c − app_sk · δ_y)`, which is an incorrect confidential key.
6. All honest parties observe a successful protocol completion; the corruption is undetectable at the protocol layer.
