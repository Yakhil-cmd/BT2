### Title
Overly Broad `skip_stateful_validations` Condition Allows Signature Bypass for Nonce-1 Invoke Transactions — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function is designed to skip the `__validate__` entry point (signature verification) only when a `deploy_account` transaction is pending for an account — a UX feature allowing users to submit `deploy_account + invoke` simultaneously. However, the guard it uses, `account_tx_in_pool_or_recent_block`, returns `true` for **any** account that has **any** transaction in the mempool pool or committed state, not exclusively accounts with a pending `deploy_account` transaction. An attacker with an account at on-chain nonce=0 (e.g., deployed via a factory/`deploy` syscall) can first submit a valid nonce=0 invoke transaction to seed the mempool, then submit a nonce=1 invoke transaction with an **invalid signature** that bypasses `__validate__` entirely and is admitted to the mempool.

---

### Finding Description

**Root cause — `skip_stateful_validations` in `crates/apollo_gateway/src/stateful_transaction_validator.rs`:** [1](#0-0) 

The function skips `__validate__` when all four conditions hold:
1. The transaction is an `Invoke` tx.
2. `tx.nonce() == Nonce(Felt::ONE)` (tx nonce is exactly 1).
3. `account_nonce == Nonce(Felt::ZERO)` (on-chain account nonce is 0).
4. `mempool_client.account_tx_in_pool_or_recent_block(sender)` returns `true`.

The code comment at line 440 states:

> "It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction **or transactions with future nonces that passed validations**."

The second case is the vulnerability. The check does not distinguish between a pending `deploy_account` tx and a pending `invoke` tx.

**`account_tx_in_pool_or_recent_block` implementation — `crates/apollo_mempool/src/mempool.rs`:** [2](#0-1) 

It returns `true` if the account has **any** tx in `tx_pool` or in the committed/staged `state` — with no type discrimination between `deploy_account` and `invoke`.

**`run_validate_entry_point` propagates the skip — `crates/apollo_gateway/src/stateful_transaction_validator.rs`:** [3](#0-2) 

When `skip_validate=true`, `execution_flags.validate` is set to `false`.

**`StatefulValidator::perform_validations` short-circuits on `validate=false` — `crates/blockifier/src/blockifier/stateful_validator.rs`:** [4](#0-3) 

The function returns `Ok(())` without ever calling `__validate__`, so the account's signature is never verified.

**`AccountTransaction::validate_tx` also short-circuits — `crates/blockifier/src/transaction/account_transaction.rs`:** [5](#0-4) 

---

### Impact Explanation

An attacker can inject a nonce=1 invoke transaction with an **invalid (or entirely fabricated) signature** into the mempool and have it propagated to all peers via P2P without any signature check at the gateway. The transaction will eventually fail at batcher execution time (the batcher runs `__validate__` with `validate=true`), but the gateway has already accepted and broadcast an invalid transaction. This directly matches the **High** impact criterion: *"Mempool/gateway/RPC admission accepts invalid transactions before sequencing."*

---

### Likelihood Explanation

The precondition — an account at on-chain nonce=0 — is realistic. Accounts deployed via the `deploy` syscall from a factory contract start with nonce=0. Pre-deployed genesis accounts also start at nonce=0. The attacker only needs to submit one valid nonce=0 invoke tx first (which they can do legitimately), then submit arbitrarily many nonce=1 txs with invalid signatures (each with a distinct tx hash to avoid the duplicate-hash check).

---

### Recommendation

Replace the `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a **`deploy_account` transaction** is pending for the sender address. The mempool should expose a dedicated `has_pending_deploy_account(address)` query, or the pool should be filtered by transaction type before the skip decision is made. The current check conflates "account is known to the mempool" with "account has a pending deploy_account tx," which is the broken invariant.

---

### Proof of Concept

**Setup:** Account `A` is deployed via a factory contract. On-chain nonce of `A` = 0.

**Step 1 — Seed the mempool with a valid nonce=0 invoke tx:**
```
POST /add_transaction
{
  type: Invoke, sender: A, nonce: 0,
  signature: <valid_signature_for_nonce_0>
}
```
- Passes `validate_nonce` (0 ≤ 0 ≤ max_gap).
- Passes `validate_by_mempool` (no duplicate).
- `skip_stateful_validations`: nonce ≠ 1, returns `false` → `__validate__` IS called → passes.
- Tx admitted to mempool. `account_tx_in_pool_or_recent_block(A)` now returns `true`.

**Step 2 — Submit nonce=1 invoke tx with invalid signature:**
```
POST /add_transaction
{
  type: Invoke, sender: A, nonce: 1,
  signature: [0xdeadbeef, 0xdeadbeef]   // invalid
}
```
- `validate_nonce`: 0 ≤ 1 ≤ max_gap → **passes**.
- `validate_by_mempool`: no duplicate hash, nonce ≥ account_nonce → **passes**.
- `skip_stateful_validations`:
  - tx is Invoke ✓
  - tx.nonce() == 1 ✓
  - account_nonce == 0 ✓
  - `account_tx_in_pool_or_recent_block(A)` == `true` (due to Step 1) ✓
  - Returns `true` → `skip_validate = true`.
- `run_validate_entry_point`: sets `execution_flags.validate = false`.
- `StatefulValidator::perform_validations`: hits `if !tx.execution_flags.validate { return Ok(()); }` → **`__validate__` never called**.
- **Tx with invalid signature admitted to mempool and propagated to all peers.** [6](#0-5) [7](#0-6)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
