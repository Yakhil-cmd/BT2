### Title
Unchecked EVM Call Result Status Allows Silent Cross-VM Asset Transfer Failure - (File: fvm/evm/handler/handler.go, fvm/evm/stdlib/contract.cdc)

### Summary

`Account.Call()` in `fvm/evm/handler/handler.go` uses `panicOnError(err)` instead of `panicOnErrorOrInvalidOrFailedState(res, err)`, meaning a reverted EVM execution (e.g., a failed ERC20 `transfer`) returns a `ResultSummary` with `Status == StatusFailed` to the Cadence caller without reverting the outer Flow transaction. The Cadence-level `coa.call()` and `EVM.run()` surface this result to the transaction author, who is not required to check it. This is the direct Flow analog of ignoring the boolean return value of a Solidity `transfer()`.

### Finding Description

In `fvm/evm/handler/handler.go`, the `Account.Call()` method handles the result of an EVM contract call inconsistently compared to other EVM operations:

```go
// Account.Call — uses panicOnError only
func (a *Account) Call(...) *types.ResultSummary {
    res, err := a.executeAndHandleAuthorizedCall(...)
    panicOnError(err)          // ← does NOT check res.Failed() or res.Invalid()
    return res.ResultSummary()
}
``` [1](#0-0) 

By contrast, every other state-mutating EVM operation (`Deposit`, `Withdraw`, `Transfer`, `DeployCOA`) uses `panicOnErrorOrInvalidOrFailedState`:

```go
func (a *Account) Deposit(v *types.FLOWTokenVault) {
    res, err := a.fch.executeAndHandleCall(...)
    panicOnErrorOrInvalidOrFailedState(res, err)  // ← reverts on failed/invalid
}
``` [2](#0-1) 

`panicOnErrorOrInvalidOrFailedState` explicitly panics when `res.Failed()` is true (i.e., `res.VMError != nil`): [3](#0-2) 

Because `Call()` skips this check, a reverted EVM execution (e.g., an ERC20 `transfer` that returns `false` or `revert`s) propagates back to Cadence as a `Result` struct with `status == Status.failed`. The Cadence `coa.call()` function returns this result directly: [4](#0-3) 

The `EVM.run()` top-level function has the same property — it returns a `Result` without any status enforcement: [5](#0-4) 

`EVM.mustRun()`, which appears to be a "safe" wrapper, explicitly documents and allows `failed` status through — it only guards against `unknown`/`invalid`: [6](#0-5) 

The `EVM.Status` enum defines three non-error states: `invalid`, `failed`, and `successful`. A `failed` result means the EVM executed but the VM reported an error (e.g., `REVERT`, out-of-gas). The EVM state change is rolled back, but the Flow transaction is not: [7](#0-6) 

### Impact Explanation

A Cadence transaction or contract that calls `coa.call()` or `EVM.run()` to execute an EVM ERC20 `transfer` or `transferFrom` can silently fail. The EVM-side token transfer reverts (no tokens move), but the outer Flow transaction succeeds. Any Cadence-side accounting that assumes the transfer succeeded (e.g., updating a balance mapping, releasing escrowed Cadence tokens, minting a receipt NFT) will be incorrect. This constitutes **cross-VM asset loss**: Cadence-side assets are released or credited against an EVM transfer that never occurred.

### Likelihood Explanation

Medium. Any Cadence transaction or smart contract that uses `coa.call()` or `EVM.run()` to invoke EVM token operations without asserting `res.status == EVM.Status.successful` is affected. This is a common omission — `EVM.mustRun()` exists precisely to address this concern but only partially does so, and its inline documentation explicitly states it does not rollback on `failed` status, which is the exact failure mode for a reverted ERC20 transfer. Any unprivileged transaction sender can trigger this by submitting a Cadence transaction that omits the status check.

### Recommendation

1. **`Account.Call()`**: Change `panicOnError(err)` to `panicOnErrorOrInvalidOrFailedState(res, err)` to match the behavior of `Deposit`, `Withdraw`, and `Transfer`. This makes a reverted EVM call revert the outer Flow transaction, consistent with the rest of the protocol. [8](#0-7) 

2. **`EVM.mustRun()`**: Extend the assertion to also reject `failed` status, or introduce a separate `EVM.mustRunSuccessful()` that asserts `status == Status.successful`. The current `mustRun` assertion `status == failed || status == successful` provides no protection against reverted EVM calls. [9](#0-8) 

3. Document clearly in `EVM.run()` and `coa.call()` that callers **must** assert `res.status == EVM.Status.successful` before treating the EVM operation as having taken effect.

### Proof of Concept

```cadence
import EVM from <EVM_ADDRESS>
import FlowToken from <FLOWTOKEN_ADDRESS>

// Attacker-controlled transaction: calls an EVM ERC20 transfer that will revert,
// but does NOT check the result status.
transaction(encodedERC20Transfer: [UInt8], coinbase: [UInt8; 20]) {
    prepare(signer: auth(Storage) &Account) {
        let coa = signer.storage.borrow<auth(EVM.Call) &EVM.CadenceOwnedAccount>(
            from: /storage/evm
        ) ?? panic("no COA")

        // EVM ERC20 transfer to a blacklisted address — will revert inside EVM
        let result = EVM.run(tx: encodedERC20Transfer, coinbase: EVM.EVMAddress(bytes: coinbase))

        // result.status == EVM.Status.failed  ← EVM transfer reverted
        // BUT: no assert here — Flow transaction succeeds
        // Any Cadence logic below executes as if the transfer succeeded
        // e.g., release escrowed Cadence tokens, mint receipt NFT, etc.
    }
}
```

The `result.status` will be `EVM.Status.failed` (the EVM transfer reverted), but the Flow transaction commits successfully. Cadence-side state changes that assumed the EVM transfer succeeded are now inconsistent with EVM state — constituting cross-VM asset loss. [10](#0-9)

func panicOnErrorOrInvalidOrFailedState(res *types.Result, err error) {

	if res != nil && res.Invalid() {
		panic(fvmErrors.NewEVMError(res.ValidationError))
	}

	if res != nil && res.Failed() {
		panic(fvmErrors.NewEVMError(res.VMError))
	}

	// this should never happen
	if err == nil && res == nil {
		panic(fvmErrors.NewEVMError(types.ErrUnexpectedEmptyResult))
	}

	panicOnError(err)
}

    /// It will rollback if the tx execution status is unknown or invalid.

    fun mustRun(tx: [UInt8], coinbase: EVMAddress): Result {
