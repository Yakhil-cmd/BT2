### Title
Missing Callback/Refund Mechanism in `ExitToEthereum::run` ERC-20 Path Causes Permanent Token Loss on Failed `withdraw` Promise — (`engine-precompiles/src/native.rs`)

---

### Summary

The `ExitToEthereum::run` function (flag=`0x01`) burns ERC-20 tokens synchronously inside the EVM transaction, then schedules a bare `PromiseArgs::Create` call to the NEP-141 connector's `withdraw` method with no callback and no refund mechanism. If the `withdraw` promise fails for any reason, the burned ERC-20 tokens are permanently destroyed with no corresponding Ethereum-side credit, violating the invariant that a burned ERC-20 must always be recoverable.

---

### Finding Description

**Entry point — `EvmErc20.withdrawToEthereum`:**

In `EvmErc20.sol`, any token holder can call `withdrawToEthereum`:

```solidity
function withdrawToEthereum(address recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);   // ERC-20 tokens destroyed here, atomically
    ...
    assembly {
        let res := call(gas(), 0xb0bd02f6..., 0, ...)  // calls ExitToEthereum precompile
    }
}
``` [1](#0-0) 

The burn happens **before** the precompile call, inside the same EVM execution. The EVM call to the precompile succeeds regardless of what the NEAR-side promise will do.

**`ExitToEthereum::run` schedules only a bare `Create` promise — no callback:**

```rust
let withdraw_promise = PromiseCreateArgs {
    target_account_id: nep141_address,
    method: "withdraw".to_string(),
    ...
};
let promise = borsh::to_vec(&PromiseArgs::Create(withdraw_promise)).unwrap();
``` [2](#0-1) 

There is no `PromiseArgs::Callback` wrapping, no `exit_to_ethereum_precompile_callback` function, and no `refund_on_error` invocation anywhere in the `ExitToEthereum` path.

**Contrast with `ExitToNear` which has a full refund mechanism:**

`ExitToNear::run` builds `ExitToNearPrecompileCallbackArgs` with a `refund` field and schedules a `PromiseArgs::Callback` that invokes `exit_to_near_precompile_callback`: [3](#0-2) 

That callback calls `engine::refund_on_error`, which re-mints the burned ERC-20 tokens to the original sender: [4](#0-3) 

The `error_refund` feature flag and all `#[cfg(feature = "error_refund")]` guards in `native.rs` apply **exclusively** to the `ExitToNear` path. The `ExitToEthereum` path has no equivalent, with or without the feature flag. [5](#0-4) 

**The `withdraw` promise can fail in realistic conditions:**

- The NEP-141 connector is paused (admin action, not attacker-controlled)
- The Ethereum recipient address is not registered with the connector
- The connector has insufficient NEP-141 balance
- Any other connector-side panic or assertion failure

When any of these occur, the NEAR runtime silently discards the failed promise result. The EVM transaction has already committed the `_burn`, and there is no callback to observe the failure and re-mint.

**The `ExitToNear` ERC-20 path has an acknowledged version of this issue** (when `error_refund` is disabled), as confirmed by the test comment:

```rust
// If the refund feature is not enabled then there is no refund in the EVM
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into();
``` [6](#0-5) 

But for `ExitToEthereum`, there is **no refund path at all**, even when `error_refund` is enabled.

---

### Impact Explanation

**Insolvency.** ERC-20 tokens are permanently destroyed (total supply decreases) with no corresponding Ethereum-side credit issued. The user loses their funds with no recovery path. This breaks the fundamental bridge invariant: every ERC-20 burn for Ethereum exit must result in either a successful Ethereum withdrawal or a re-mint of the burned tokens.

---

### Likelihood Explanation

Any ERC-20 holder can trigger this by calling `withdrawToEthereum`. The failure condition (connector paused, unregistered recipient, etc.) is realistic and does not require attacker privilege — a connector pause by an admin, or simply specifying a recipient address not registered with the connector, is sufficient. The `withdraw` method on the NEP-141 connector enforces recipient registration, making the unregistered-recipient scenario trivially reachable.

---

### Recommendation

Add a callback to `ExitToEthereum::run` analogous to the one in `ExitToNear::run`. The callback should:
1. Check the promise result of the `withdraw` call.
2. On failure, call `engine::refund_on_error` with the ERC-20 address and amount to re-mint the burned tokens to the original sender.

This requires introducing an `exit_to_ethereum_precompile_callback` entry point in `engine/src/lib.rs` and `engine/src/contract_methods/connector.rs`, mirroring the existing `exit_to_near_precompile_callback` pattern. [7](#0-6) 

---

### Proof of Concept

1. Deploy Aurora locally with a bridged NEP-141/ERC-20 pair.
2. Record the ERC-20 total supply and the user's ERC-20 balance.
3. Call `withdrawToEthereum(unregistered_eth_address, amount)` — the NEP-141 connector's `withdraw` will fail because the Ethereum address is not registered.
4. After the NEAR block finalizes, assert:
   - ERC-20 total supply decreased by `amount` (burn committed).
   - No `Withdrawn` event emitted by the connector (no Ethereum credit).
   - User's ERC-20 balance decreased by `amount` with no refund.
   - The `withdraw` promise result is `Failed` with no callback having been scheduled.

The existing test `test_exit_to_near_refund` demonstrates the analogous scenario for `ExitToNear` and confirms the refund mechanism works there. No equivalent test exists for `ExitToEthereum` because no refund mechanism exists. [8](#0-7)

#[named]
pub fn exit_to_near_precompile_callback<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<Option<SubmitResult>, ContractError> {
    with_hashchain(io, env, function_name!(), |io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        env.assert_private_call()?;

        // This function should only be called as the callback of
        // exactly one promise.
        if handler.promise_results_count() != 1 {
            return Err(errors::ERR_PROMISE_COUNT.into());
        }

        let args: ExitToNearPrecompileCallbackArgs = io.read_input_borsh()?;

        let maybe_result = if let Some(PromiseResult::Successful(_)) = handler.promise_result(0) {
            if let Some(args) = args.transfer_near {
                let action = PromiseAction::Transfer {
                    amount: Yocto::new(args.amount),
                };
                let promise = PromiseBatchAction {
                    target_account_id: args.target_account_id,
                    actions: vec![action],
                };

                // Safety: this call is safe because it comes from the exit to near precompile, not users.
                // The call is to transfer the unwrapped wNEAR tokens.
                let promise_id = handler.promise_create_batch(&promise);
                handler.promise_return(promise_id);
            }

            None
        } else if let Some(args) = args.refund {
            // Exit call failed; need to refund tokens
            let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;

            if !refund_result.status.is_ok() {
                return Err(errors::ERR_REFUND_FAILURE.into());
            }

            Some(refund_result)
        } else {
            None
        };

        Ok(maybe_result)
    })
}
