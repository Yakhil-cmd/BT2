### Title
WalletConnect `take_offer` Confirmation Understates Total Royalty Cost When Multiple NFTs Are Present - (File: `packages/gui/src/electron/commands/parseCommandDisplay.ts`)

### Summary
The `formatAmountWithRoyalties` function in the WalletConnect `take_offer` confirmation pipeline divides the fungible payment amount by the number of royalty-bearing NFTs before computing each NFT's royalty, producing an `amountWithRoyalties` display value that is systematically lower than what the user actually pays on-chain. A WalletConnect dApp presenting a multi-NFT offer can exploit this to make the user approve a spend that is materially larger than the confirmed figure.

### Finding Description
`parseCommandDisplay` is the function that builds the structured `walletDelta` shown to the user in the WalletConnect `Confirm` dialog before they approve a `chia_wallet.take_offer` command. [1](#0-0) 

The royalty-inclusive total is computed by `formatAmountWithRoyalties`:

```typescript
// parseCommandDisplay.ts lines 363-368
const splitAmount = amount / BigInt(royaltyPercentages.length);   // ← divides by NFT count
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
const totalAmount = amount + royaltyAmount;
``` [2](#0-1) 

`royaltyPercentages` is the array of royalty basis-points for every NFT on the opposite side of the trade that has a non-zero royalty. [3](#0-2) 

The code assumes the fungible amount is split equally among the royalty-bearing NFTs and then applies each NFT's percentage to its share. The Chia protocol does the opposite: `nft_calculate_royalties` applies each NFT's royalty percentage to the **full** fungible amount. The display therefore understates the true royalty burden by a factor proportional to the number of NFTs.

The `amountWithRoyalties` field is surfaced directly in the "You Spend" section of the WalletConnect confirmation dialog. [4](#0-3) 

### Impact Explanation
A user accepting a WalletConnect `take_offer` for N NFTs each carrying royalty R% on a payment of A mojos sees:

| | Display (split) | On-chain (full) |
|---|---|---|
| Per-NFT royalty base | A / N | A |
| Total royalties | A × R% (once) | A × R% × N |
| Shown total | A + A×R% | A + A×R%×N |

For the concrete test case already in the codebase — 100,000,000 mojos, two NFTs at 500 bps and 10 bps — the dialog shows **0.00010255 XCH** while the wallet actually spends **0.00010510 XCH** (a 2.5% overcharge relative to the displayed figure). [5](#0-4) 

With higher royalty percentages or more NFTs the gap grows proportionally. A dApp can craft an offer with many high-royalty NFTs so that the displayed total is a small fraction of the actual spend, causing the user to approve a transaction they would otherwise reject. This matches the **High** impact criterion: "WalletConnect state that causes a user to approve … the wrong … amount."

### Likelihood Explanation
The trigger requires a WalletConnect-connected dApp to issue a `take_offer` command containing two or more NFTs that each carry a non-zero royalty. This is a normal, supported offer type. No special permissions, leaked keys, or social engineering beyond the standard WalletConnect pairing are needed. Any dApp that can present offers (e.g., an NFT marketplace) can reach this code path.

### Recommendation
Remove the division by NFT count. Each NFT's royalty should be calculated on the full fungible amount:

```typescript
// Correct: apply each royalty to the full amount
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (amount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
const totalAmount = amount + royaltyAmount;
```

This matches the semantics of `nft_calculate_royalties` used in the offer-builder flow. [6](#0-5) 

### Proof of Concept
Using the existing test fixture (two NFTs, 500 bps + 10 bps, 100,000,000 mojos):

**Current (buggy) display path:**
```
splitAmount = 100_000_000 / 2 = 50_000_000
royalty_NFT1 = 50_000_000 × 500 / 10_000 = 2_500_000
royalty_NFT2 = 50_000_000 × 10  / 10_000 =    50_000
displayed total = 102_550_000 mojos → "0.00010255 XCH"
```

**Actual on-chain spend:**
```
royalty_NFT1 = 100_000_000 × 500 / 10_000 = 5_000_000
royalty_NFT2 = 100_000_000 × 10  / 10_000 =   100_000
actual total  = 105_100_000 mojos → 0.00010510 XCH
```

The user approves 0.00010255 XCH but the wallet deducts 0.00010510 XCH — a 2.5% silent overcharge in this example, scaling linearly with NFT count and royalty rates. [7](#0-6)

```typescript
function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],
): string | undefined {
  if (royaltyPercentages.length === 0 || line.kind === 'nft') {
    return undefined;
  }

  const splitAmount = amount / BigInt(royaltyPercentages.length);
  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;

  if (line.kind === 'xch') {
    return mojoToChiaLocaleString(totalAmount);
  }

  return mojoToCATLocaleString(totalAmount);
}
```

export async function parseCommandDisplay(command: string, params: Record<string, unknown>) {
  if (command === 'chia_wallet.take_offer') {
    if (!params.offer || typeof params.offer !== 'string') {
      throw new Error('Offer is not valid');
    }

    const offerSummary = await getOfferSummary(params.offer);
    if (!offerSummary || !offerSummary.summary || !offerSummary.success) {
      throw new Error('Offer is not valid');
    }

    const { summary } = offerSummary;

    const walletDelta = offerSummaryToWalletDelta(summary);
    const walletInfos = await getWalletInfos();
    const assetKinds = offerSummaryAssetKinds(summary);
    const royaltyPercentages = offerSummaryRoyaltyPercentages(summary);
    const fees = parseMojos(summary.fees);

    return {
      walletDelta: await walletDeltaToDisplay(walletDelta, walletInfos, assetKinds, royaltyPercentages, fees),
    };
