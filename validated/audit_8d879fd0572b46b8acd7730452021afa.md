### Title
WalletConnect `take_offer` Confirmation Dialog Displays Understated Royalty Total for Multi-NFT Offers - (File: packages/gui/src/electron/commands/parseCommandDisplay.ts)

### Summary
The `formatAmountWithRoyalties` function in `parseCommandDisplay.ts`, which populates the `amountWithRoyalties` field shown in the WalletConnect approval dialog for `chia_wallet.take_offer`, uses an equal-split heuristic to distribute the fungible amount across multiple NFTs before applying each NFT's royalty percentage. The Chia wallet backend applies each NFT's royalty to the **full** fungible amount. This causes the confirmation dialog to display a systematically lower total cost than what the wallet will actually deduct, leading a user to approve a `take_offer` command under a false understanding of the amount they are spending.

---

### Finding Description

`parseCommandDisplay.ts` is the main-process function that builds the human-readable `walletDelta` shown in the WalletConnect confirmation dialog before the user clicks "Approve". For `chia_wallet.take_offer`, it calls `formatAmountWithRoyalties`:

```typescript
// packages/gui/src/electron/commands/parseCommandDisplay.ts
const splitAmount = amount / BigInt(royaltyPercentages.length);   // equal split
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
const totalAmount = amount + royaltyAmount;
``` [1](#0-0) 

When N NFTs are present, the function divides the fungible amount by N and applies each NFT's royalty to that fractional share. The correct formula â€” and the one the Chia wallet backend uses â€” applies each NFT's royalty to the **full** fungible amount:

```
correct_royalty = ÎŁ (amount Ă— royaltyPercentage_i / 10_000)
displayed_royalty = ÎŁ ((amount / N) Ă— royaltyPercentage_i / 10_000)
                  = correct_royalty / N
```

For N = 2 NFTs, the dialog shows exactly **half** the actual royalty cost. The `OfferBuilderProvider` (the non-WalletConnect path) correctly delegates royalty calculation to the backend via `useCalculateRoyaltiesForNFTsQuery`, which applies royalties to the full fungible amount. [2](#0-1) 

The inconsistency is confirmed by the existing test: a 2-NFT offer with royalties of 500 (5 %) and 10 (0.1 %) on 100,000,000 mojos produces `amountWithRoyalties: '0.00010255'` (split-based), whereas the correct total is `0.0001051` (full-amount-based). [3](#0-2) 

---

### Impact Explanation

A user connected via WalletConnect sees the `amountWithRoyalties` field in the approval dialog and uses it to decide whether to approve the `take_offer` command. Because the displayed total is understated by a factor of N (number of NFTs), the user approves believing they will spend X XCH/CAT, but the wallet actually deducts up to NĂ— more in royalties. This is a direct, concrete accounting discrepancy that causes a user to approve the wrong amount â€” matching the **High** impact criterion: *"WalletConnect state that causes a user to approve â€¦ the wrong â€¦ amount."*

---

### Likelihood Explanation

Any WalletConnect-connected dApp can issue a `chia_wallet.take_offer` command with an offer containing two or more NFTs that carry royalties. No special privileges are required. The dApp can craft the offer to maximize the gap (e.g., two NFTs each with 10 % royalty: displayed total shows 10 % extra, actual deduction is 20 % extra). The user has no other signal in the dialog to detect the discrepancy.

---

### Recommendation

Replace the equal-split heuristic with the correct formula: apply each NFT's royalty percentage to the full fungible amount.

```typescript
// Correct: apply each royalty to the full amount
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (amount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
``` [4](#0-3)

### Proof of Concept

1. A malicious dApp holds an offer containing two NFTs, each with `royalty_percentage = 1000` (10 %), requesting 1 XCH.
2. The dApp sends `chia_wallet.take_offer` via WalletConnect.
3. `formatAmountWithRoyalties` computes:
   - `splitAmount = 1_000_000_000_000 / 2 = 500_000_000_000`
   - `royaltyAmount = (500_000_000_000 Ă— 1000 / 10_000) Ă— 2 = 100_000_000_000`
   - Displayed total: **1.1 XCH**
4. Actual backend calculation:
   - `royaltyAmount = (1_000_000_000_000 Ă— 1000 / 10_000) Ă— 2 = 200_000_000_000`
   - Actual deduction: **1.2 XCH**
5. The user approves seeing 1.1 XCH; the wallet deducts 1.2 XCH â€” a 0.1 XCH (~9 %) silent overcharge per transaction. [1](#0-0) [5](#0-4)

function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],
): string | undefined {
  if (royaltyPercentages.length === 0 || line.kind === 'nft') {
    return undefined;
  }

  const splitAmount = amount / BigInt(royaltyPercentages.length);
  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;

  if (line.kind === 'xch') {
    return mojoToChiaLocaleString(totalAmount);
  }

  return mojoToCATLocaleString(totalAmount);
}
