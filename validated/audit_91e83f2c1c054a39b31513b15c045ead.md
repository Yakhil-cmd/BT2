Audit Report

## Title
Guard Exhaustion DoS on ckETH/ckERC20 Withdrawals via 100 Concurrent `withdraw_erc20` Calls — (`rs/ethereum/cketh/minter/src/guard/mod.rs`, `rs/ethereum/cketh/minter/src/main.rs`)

## Summary

`withdraw_erc20` acquires a `PendingWithdrawalRequests` guard before any async work and holds it across three sequential `await` points (an HTTP outcall for gas fee estimation and two inter-canister ledger burns). Because `MAX_CONCURRENT = 100` is the exact exhaustion threshold for the shared `pending_withdrawal_principals` set, 100 distinct principals can simultaneously hold all guard slots, causing every subsequent call to `withdraw_eth` or `withdraw_erc20` to `ic_cdk::trap`, blocking all new withdrawals for the duration of the in-flight calls.

## Finding Description

**Guard constants and shared set**

`MAX_CONCURRENT` is set to `100` and both `withdraw_eth` and `withdraw_erc20` share the same guard type (`PendingWithdrawalRequests`) backed by `state.pending_withdrawal_principals`. [1](#0-0) [2](#0-1) 

**Guard acquisition and trap on failure**

`withdraw_erc20` acquires the guard at the very top of the function, before any `await`, and calls `ic_cdk::trap` (hard abort, not a graceful `Err`) if the guard cannot be obtained: [3](#0-2) 

`withdraw_eth` does the same: [4](#0-3) 

**Guard held across three await points in `withdraw_erc20`**

After guard acquisition, `withdraw_erc20` suspends at:

1. `estimate_erc20_transaction_fee().await` → `lazy_refresh_gas_fee_estimate().await`, which makes an HTTP outcall to EVM RPC providers when the cached estimate is stale (>60 seconds old): [5](#0-4) [6](#0-5) 

2. `cketh_ledger.burn_from(...).await` — inter-canister call to the ckETH ledger: [7](#0-6) 

3. `ckerc20_ledger.burn_from(...).await` — inter-canister call to the ckERC20 ledger: [8](#0-7) 

The guard is only released when `_guard` drops at function exit (either `Ok` or `Err` path): [9](#0-8) 

**Guard limit check that triggers the trap** [10](#0-9) 

**Existing checks are insufficient**

The `AlreadyProcessing` check prevents a single principal from holding multiple guards, but does not prevent 100 distinct principals from each holding one guard simultaneously: [11](#0-10) 

## Impact Explanation

When `pending_withdrawal_principals.len() == 100`, any new call to `withdraw_eth` or `withdraw_erc20` from a principal not already in the set hits `TooManyConcurrentRequests` and is immediately trapped via `ic_cdk::trap`. No new ckETH or ckERC20 withdrawal can be initiated until at least one of the 100 in-flight calls completes. Because both withdrawal functions share the same guard set, exhausting it via `withdraw_erc20` also blocks `withdraw_eth`. This constitutes an **application-level DoS on the ckETH/ckERC20 minter's withdrawal functionality**, matching the allowed High impact: *"Application/platform-level DoS, crash, consensus blocking, certified-state disruption, or subnet availability impact not based on raw volumetric DDoS."* Severity: **High ($2,000–$10,000)**.

## Likelihood Explanation

- Requires 100 distinct non-anonymous principals (cheap on the IC).
- When the gas fee estimate is stale, the guard is held during the HTTP outcall to EVM RPC providers before any balance check — no ckETH or ckERC20 funds are required to hold the guard during that phase.
- When the estimate is cached, the attacker needs valid ckETH allowances for 100 principals to hold the guard past the first burn — an economic cost but not prohibitive for a motivated attacker targeting a high-value bridge.
- The IC scheduler interleaves messages at every `await` point, so 100 concurrent in-flight calls is the normal execution model, not a race condition.
- The DoS window per round is bounded by the combined latency of the HTTP outcall plus two sequential inter-canister calls (seconds to tens of seconds), but the attacker can sustain the DoS by continuously re-sending calls as guards are released.
- The attack is repeatable and does not require any privileged access.

## Recommendation

1. **Decouple guard scope from ledger latency**: Release the guard immediately after the withdrawal request is durably recorded in state (after both burns succeed), rather than holding it across all inter-canister calls. The guard's purpose is reentrancy prevention, not queue admission control.
2. **Raise `MAX_CONCURRENT` or make it asymmetric**: `MAX_CONCURRENT = 100` is simultaneously the concurrency limit and the exact exhaustion threshold. A value significantly higher than realistic peak concurrency (e.g., 1000) would make exhaustion impractical.
3. **Return `Err(...)` instead of `ic_cdk::trap` on `TooManyConcurrentRequests`**: Trapping is appropriate for programming errors, not for expected resource-limit conditions. Returning a typed error allows callers to retry gracefully.
4. **Separate the guard sets for `withdraw_eth` and `withdraw_erc20`**: Currently both share `pending_withdrawal_principals`, so `withdraw_erc20` exhaustion also blocks `withdraw_eth`.

## Proof of Concept

State-machine test outline (using the existing `CkErc20Setup` test harness already present in `rs/ethereum/cketh/minter/tests/ckerc20.rs`):

1. Initialize minter with ckETH + one ckERC20 token.
2. Create 100 principals, each with sufficient ckETH and ckERC20 allowances.
3. Send 100 concurrent `withdraw_erc20` ingress messages (one per principal) using `env.send_ingress`.
4. Tick the IC scheduler so each message reaches its first `await` (gas fee HTTP outcall or ckETH burn) but does NOT complete — mock ledger/RPC holds responses.
5. Assert: `read_state(|s| s.pending_withdrawal_principals.len()) == 100`.
6. Send a 101st `withdraw_erc20` from a fresh principal.
7. Assert: the 101st call traps with a message containing `"TooManyConcurrentRequests"`.
8. Also assert: a `withdraw_eth` call from the same fresh principal also traps.
9. Release the mock ledger/RPC responses; assert all 100 guards are dropped and a new call succeeds.
