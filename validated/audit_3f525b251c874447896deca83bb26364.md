### Title
Unaccounted Native Resource Cost for Per-Transaction EVM Precompile Warming Enables Block Proving DoS — (File: `basic_bootloader/src/bootloader/block_flow/ethereum/loop_op.rs`)

---

### Summary

In `generic_loop_op`, before every transaction is processed, all EVM precompile addresses are warmed using a freshly allocated `S::Resources::FORMAL_INFINITE` resource object — meaning **both ergs and native are set to `u64::MAX`** and the actual native consumed is silently discarded. This unaccounted native proving cost is never reflected in the block's native limit check. An unprivileged sender can submit many minimum-gas transactions to force the actual proving cost of a block above its native budget, producing an unprovable block and halting the chain.

---

### Finding Description

In `generic_loop_op` at lines 96–109, before each transaction is dispatched, the bootloader iterates over every address in `PRECOMPILE_ADDRESSES_LOWS` and calls `touch_account` with a **fresh `FORMAL_INFINITE` resource**:

```rust
for &address_low in PRECOMPILE_ADDRESSES_LOWS {
    let address = B160::from_limbs([address_low as u64, 0, 0]);
    let mut inf_resources = S::Resources::FORMAL_INFINITE;   // ← both ergs AND native = u64::MAX
    system
        .io
        .touch_account(ExecutionEnvironmentType::NoEE, &mut inf_resources, &address)
        .expect("must warm up precompile");
}
``` [1](#0-0) 

`FORMAL_INFINITE` is defined as `DecreasingNative(u64::MAX)` for native and `Ergs(u64::MAX)` for ergs: [2](#0-1) [3](#0-2) 

Because `inf_resources` is a **local throwaway variable**, the native units consumed by each `touch_account` call are never subtracted from the transaction's resource budget, never accumulated into `block_computational_native_used`, and never checked against `MAX_NATIVE_COMPUTATIONAL`. The block-level native limit check in `check_for_block_limits` only sees the native charged through the transaction's own resource object: [4](#0-3) 

The ZK tx loop (proving path) warms only the coinbase with `FORMAL_INFINITE` per transaction and does **not** warm precompiles per transaction: [5](#0-4) 

This asymmetry means the Ethereum-storage-model proving path has a per-transaction unaccounted native overhead proportional to the number of precompile addresses, while the block native limit is computed as if that overhead does not exist.

The intrinsic computational native formula (`calculate_l1_tx_intrinsic_computational_native_resources` / `create_resources_for_tx`) covers ecrecover, keccak, account reads, nonce writes, fee prepayment, refund, and coinbase payment — but contains no term for precompile warming: [6](#0-5) 

---

### Impact Explanation

Let:
- `N` = number of precompile addresses in `PRECOMPILE_ADDRESSES_LOWS`
- `C_warm` = native cost of one cold/warm `touch_account` IO call (Merkle proof read in proving mode)
- `T` = number of transactions in a block (bounded by `block_gas_limit / 21_000`)

The **charged** block native cost is `T × intrinsic_native_per_tx`.  
The **actual** proving cost is `T × (intrinsic_native_per_tx + N × C_warm)`.

If `N × C_warm` is non-trivial (each precompile touch requires an oracle read and proof verification in RISC-V proving mode), the actual proving cost exceeds `MAX_NATIVE_COMPUTATIONAL` while the block passes the native limit check. The resulting block is valid from the sequencer's perspective but **unprovable**, halting the chain.

---

### Likelihood Explanation

- Any unprivileged sender can submit minimum-gas (21 000 gas) transactions; no special privilege is required.
- The sequencer uses the charged native cost to decide how many transactions fit in a block. Because the precompile-warming overhead is invisible to the accounting system, a sequencer following the on-chain rules will naturally over-fill blocks.
- The attack is cheap: the attacker pays only the minimum transaction fee per transaction.
- The Ethereum storage model path is compiled for RISC-V proving (same dual-target design as the ZK path), so the unaccounted cost is real proving work.

---

### Recommendation

1. **Charge precompile warming against a per-block native budget** rather than using `FORMAL_INFINITE`. Allocate a fixed native overhead per transaction for precompile warming and deduct it from the block's native limit before accepting the transaction.
2. **Alternatively**, move precompile warming to a once-per-block operation (before the transaction loop) and account for it in the block-level native budget, mirroring the ZK tx loop design which does not warm precompiles per transaction.
3. **Add a constant** `PER_TX_PRECOMPILE_WARMING_NATIVE_OVERHEAD` to the intrinsic native formula in `calculate_tx_intrinsic_gas` / `calculate_l1_tx_intrinsic_computational_native_resources`, sized to cover `N × max_touch_account_native_cost`.

---

### Proof of Concept

1. Attacker submits `K = floor(block_gas_limit / 21_000)` transactions, each with `gas_limit = 21_000` and minimal calldata, targeting a no-op contract.
2. The sequencer fills the block: charged native = `K × intrinsic_native_per_tx` ≤ `MAX_NATIVE_COMPUTATIONAL` — block passes the native limit check.
3. In the RISC-V prover, `generic_loop_op` executes. Before each of the `K` transactions, it calls `touch_account` for every address in `PRECOMPILE_ADDRESSES_LOWS` with `FORMAL_INFINITE` resources. Each call performs real oracle reads and proof verifications consuming native cycles.
4. Actual proving native = `K × (intrinsic_native_per_tx + N × C_warm)`. For sufficiently large `K` and non-trivial `C_warm`, this exceeds `MAX_NATIVE_COMPUTATIONAL`.
5. The prover cannot complete the block; the block is unprovable; the chain stalls. [7](#0-6) [8](#0-7)
