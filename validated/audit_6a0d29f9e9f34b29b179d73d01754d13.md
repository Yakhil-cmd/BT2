### Title
Pool Admin Bypasses Factory Fee Caps via Uncapped `setPoolBinAdditionalFees()` — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

### Summary

`MetricOmmPoolFactory.setPoolAdminFees()` enforces factory-owner-set caps (`maxAdminSpreadFeeE6`, `maxAdminNotionalFeeE8`) before updating pool fees. The sibling function `setPoolBinAdditionalFees()` — which also sets pool-admin-controlled fees that directly affect swap pricing — performs **no cap validation at all**, allowing the pool admin to charge per-bin fees up to `uint16.max = 65 535` (≈ 6.55 % in E6 units) regardless of what the factory owner has configured.

### Finding Description

`setPoolAdminFees()` guards both fee dimensions against factory-owner caps:

```solidity
// MetricOmmPoolFactory.sol – setPoolAdminFees()
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (newAdminNotionalFeeE8 > maxAdminNotionalFeeE8) revert AdminFeeTooHigh();
``` [1](#0-0) 

`setPoolBinAdditionalFees()` passes the values straight through to the pool with **zero cap check**:

```solidity
// MetricOmmPoolFactory.sol – setPoolBinAdditionalFees()
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
``` [2](#0-1) 

The pool's `setBinAdditionalFees()` likewise performs no cap check — only a bin-index bounds check: [3](#0-2) 

Bin additional fees are added directly to the base spread fee in every swap:

```solidity
params.baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6)
``` [4](#0-3) 

The factory owner controls `maxAdminSpreadFeeE6` (hard-capped at `HARD_MAX_SPREAD_FEE_E6 = 200_000`, i.e. 20 %) and `maxAdminNotionalFeeE8`, but has **no corresponding cap variable for bin additional fees**. [5](#0-4) 

### Impact Explanation

The factory owner's fee-cap system is designed to bound the total fees a pool admin can impose on traders. By calling `setPoolBinAdditionalFees()` with `addFeeBuyE6 = addFeeSellE6 = 65535`, the pool admin adds ≈ 6.55 % per bin on top of whatever spread fee is already configured — even if the factory owner has set `maxAdminSpreadFeeE6 = 0` to prohibit any admin fee. Because the factory provides no mechanism for the owner to forcibly change the pool admin (only the admin can propose a transfer), the only owner recourse is `protocolPausePool()`, which freezes the pool entirely. Traders in affected bins pay materially more than the factory-owner-sanctioned maximum.

### Likelihood Explanation

Any pool admin — a semi-trusted role — can call `setPoolBinAdditionalFees()` at any time with no timelock. The call requires no special conditions beyond holding the `poolAdmin` role for the target pool.

### Recommendation

Add a factory-owner-configurable cap for bin additional fees (e.g. `maxAdminBinFeeE6`) and enforce it inside `setPoolBinAdditionalFees()`, mirroring the pattern used in `setPoolAdminFees()`:

```solidity
if (addFeeBuyE6 > maxAdminBinFeeE6) revert AdminFeeTooHigh();
if (addFeeSellE6 > maxAdminBinFeeE6) revert AdminFeeTooHigh();
```

### Proof of Concept

1. Factory owner deploys factory and sets `maxAdminSpreadFeeE6 = 0` (zero admin spread fee permitted).
2. Pool is created with `adminSpreadFeeE6 = 0`; `setPoolAdminFees` correctly enforces the cap.
3. Pool admin calls `setPoolBinAdditionalFees(pool, 0, 65535, 65535)`.
4. No revert occurs; bin 0 now carries a 6.55 % buy and sell surcharge.
5. Every swap routed through bin 0 pays 6.55 % more than the factory owner's cap of 0 % permits, with no factory-level remedy short of pausing the entire pool.
