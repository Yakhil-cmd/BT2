### Title
Staker With Delegation Pool Permanently Loses Epoch Reward When `attest` Reverts Due to Insufficient Reward Supplier Token Balance - (File: src/attestation/attestation.cairo)

### Summary

The `attest` function in `attestation.cairo` marks the attestation as done and then calls `update_rewards_from_attestation_contract` on the staking contract. Inside `_update_rewards`, `claim_from_reward_supplier` performs an actual STRK token transfer from the reward supplier to the staking contract to pay pool rewards. If the reward supplier's on-chain STRK token balance is insufficient (because L1 mint requests are pending but not yet fulfilled), this transfer reverts, causing the entire `attest` transaction to revert. Since Starknet transactions are atomic, the attestation mark is also rolled back. If the attestation window closes before the reward supplier is refunded, the staker permanently loses their epoch reward.

---

### Finding Description

The `attest` function in `src/attestation/attestation.cairo` follows this sequence:

1. Validates the attestation (block hash, window, epoch).
2. Calls `_mark_attestation_is_done` — writes `staker_last_attested_epoch[staker_address] = current_epoch`.
3. Calls `staking_dispatcher.update_rewards_from_attestation_contract(staker_address)`. [1](#0-0) 

Inside `update_rewards_from_attestation_contract` → `_update_rewards`, the following sequence executes:

1. `reward_supplier_dispatcher.update_unclaimed_rewards_from_staking_contract(rewards: staker_rewards + total_pools_rewards)` — increments the accounting variable `unclaimed_rewards` and may send an L1 mint request.
2. `claim_from_reward_supplier(reward_supplier_dispatcher, amount: total_pools_rewards, ...)` — calls `reward_supplier.claim_rewards(amount)`, which performs an actual `checked_transfer` of STRK tokens from the reward supplier contract to the staking contract. [2](#0-1) 

The `claim_from_reward_supplier` utility function in `src/staking/utils.cairo` reads the staking contract's balance before and after, calls `claim_rewards`, and asserts the balance increased by exactly `amount`: [3](#0-2) 

Inside `claim_rewards` in `src/reward_supplier/reward_supplier.cairo`, the actual token transfer is:

```
token_dispatcher.checked_transfer(recipient: staking_contract, amount: amount.into());
``` [4](#0-3) 

**The root cause**: The reward supplier's `unclaimed_rewards` accounting variable and its actual STRK token balance are decoupled. `update_unclaimed_rewards_from_staking_contract` increments `unclaimed_rewards` and may send an L1 mint request, but the actual tokens only arrive after the L1→L2 bridge delay. During this window, the reward supplier's on-chain STRK balance can be less than `total_pools_rewards`, causing `checked_transfer` to revert.

Because Starknet transactions are fully atomic, the revert propagates all the way back through `update_rewards_from_attestation_contract` → `attest`, rolling back the `_mark_attestation_is_done` write as well. The staker can retry, but only within the attestation window `[target_block + MIN_ATTESTATION_WINDOW, target_block + attestation_window]`. [5](#0-4) 

If the reward supplier's balance remains insufficient for the entire duration of the attestation window (e.e., the L1 mint has not yet been bridged), the staker permanently loses their epoch reward. This only affects stakers who have at least one delegation pool (`total_pools_rewards > 0`); stakers without pools call `claim_from_reward_supplier` with `amount = 0`, which succeeds trivially.

---

### Impact Explanation

A staker with a delegation pool who submits a valid, timely attestation permanently loses their entire epoch reward (both own rewards and pool commission) if the reward supplier's STRK balance is depleted at the time of attestation and is not replenished before the attestation window closes. This constitutes **permanent freezing / loss of unclaimed yield** for the staker and their delegators, matching the allowed High impact.

---

### Likelihood Explanation

The reward supplier proactively requests L1 mints via `request_funds` inside `update_unclaimed_rewards_from_staking_contract`, but the L1→L2 StarkGate bridge introduces a non-trivial delay (typically hours). During high-activity epochs where many stakers with pools attest in rapid succession, the reward supplier's on-chain balance can be fully depleted before the L1 mint arrives. A staker whose attestation window falls entirely within this depletion period loses their reward with no recourse. The attestation window is bounded by `attestation_window` (configurable, minimum 11 blocks), which can be shorter than the bridge delay. Likelihood is **Medium** — it requires a specific timing coincidence but is not hypothetical given the documented L1→L2 latency.

---

### Recommendation

Decouple reward accounting from the token transfer inside `attest`. Two approaches:

1. **Lazy transfer**: Do not call `claim_from_reward_supplier` inside `_update_rewards` during attestation. Instead, only update the accounting (`unclaimed_rewards_own` for the staker, and a pending pool rewards ledger). Perform the actual token transfer lazily when the pool or staker explicitly claims rewards.

2. **Graceful degradation**: Wrap the `claim_from_reward_supplier` call in a try/catch (safe dispatcher pattern). If the transfer fails due to insufficient balance, record the owed pool rewards in a pending ledger and allow the pool to claim them once the reward supplier is funded, without reverting the attestation.

Either approach ensures that a valid attestation is never rolled back due to a transient funding shortfall in the reward supplier.

---

### Proof of Concept

1. Deploy the system with a staker who has a STRK delegation pool.
2. Advance K epochs so the staker is eligible to attest.
3. Drain the reward supplier's STRK token balance to zero (this can happen naturally after many stakers attest in the same epoch and the L1 mint has not yet arrived).
4. Have the staker's operational address call `attest` with a valid block hash within the attestation window.
5. The call chain is: `attest` → `_mark_attestation_is_done` (writes epoch) → `update_rewards_from_attestation_contract` → `_update_rewards` → `claim_from_reward_supplier` → `reward_supplier.claim_rewards` → `checked_transfer` **REVERTS** (balance = 0, amount > 0).
6. The entire transaction reverts; `staker_last_attested_epoch` is rolled back to the previous value.
7. Advance blocks past `target_block + attestation_window` (close the window).
8. The staker can no longer call `attest` for this epoch (`ATTEST_OUT_OF_WINDOW`).
9. The staker's `unclaimed_rewards_own` is unchanged — the epoch reward is permanently lost. [6](#0-5) [7](#0-6) [8](#0-7)

pub(crate) fn claim_from_reward_supplier(
    reward_supplier_dispatcher: IRewardSupplierDispatcher,
    amount: Amount,
    token_dispatcher: IERC20Dispatcher,
) {
    let staking_contract = get_contract_address();
    let balance_before = token_dispatcher.balance_of(account: staking_contract);
    reward_supplier_dispatcher.claim_rewards(:amount);
    let balance_after = token_dispatcher.balance_of(account: staking_contract);
    assert!(balance_after - balance_before == amount.into(), "{}", Error::UNEXPECTED_BALANCE);
}

        fn claim_rewards(ref self: ContractState, amount: Amount) {
            // Asserts.
            let staking_contract = self.staking_contract.read();
            assert!(
                get_caller_address() == staking_contract,
                "{}",
                GenericError::CALLER_IS_NOT_STAKING_CONTRACT,
            );
            let unclaimed_rewards = self.unclaimed_rewards.read();
            assert!(amount <= unclaimed_rewards, "{}", GenericError::AMOUNT_TOO_HIGH);

            // Update unclaimed_rewards and transfer the requested rewards to the staking contract.
            self.unclaimed_rewards.write(unclaimed_rewards - amount);
            let token_dispatcher = self.token_dispatcher.read();
            token_dispatcher.checked_transfer(recipient: staking_contract, amount: amount.into());
        }
