### Title
Unprivileged Caller Can Permanently Freeze Consensus Rewards via `update_rewards(disable_rewards: true)` — (File: `src/staking/staking.cairo`)

---

### Summary

The `update_rewards` function in the Staking contract is publicly callable with no access control beyond a pause check. It accepts a `disable_rewards` boolean parameter. When called with `disable_rewards: true`, the function writes the global `last_reward_block` storage variable **before** the early-return guard, consuming the block's single reward slot without distributing any rewards. Any unprivileged caller can exploit this every block to permanently starve all stakers of consensus rewards.

---

### Finding Description

`update_rewards` is exposed as a public entry point with no role restriction — only `general_prerequisites()` (pause check) is enforced. [1](#0-0) 

The execution order is:

1. Assert `current_block_number > last_reward_block` — ensuring the function runs at most once per block. [2](#0-1) 

2. Write `last_reward_block = current_block_number` — **before** the `disable_rewards` guard. [3](#0-2) 

3. Return early if `disable_rewards || self.is_pre_consensus()`. [4](#0-3) 

Because `last_reward_block` is a **global** storage variable (not per-staker), once it is written in a given block, every subsequent call to `update_rewards` in that block reverts with `REWARDS_ALREADY_UPDATED`. An attacker calling `update_rewards(any_valid_staker, disable_rewards: true)` once per block consumes the block's reward slot without distributing anything, permanently starving all stakers of consensus rewards. [5](#0-4) 

---

### Impact Explanation

All consensus-phase block rewards are silently discarded every block. Stakers and delegators permanently lose unclaimed yield. This matches the allowed impact: **Permanent freezing of unclaimed yield**.

---

### Likelihood Explanation

- The function is publicly callable with no role check.
- Valid staker addresses are publicly readable from the `stakers` vector.
- The cost is one Starknet transaction per block — low on Starknet.
- No special knowledge, key, or privileged access is required.

---

### Recommendation

- Restrict `update_rewards` to a specific role (e.g., `only_operator`) or to the attestation contract.
- Alternatively, move `self.last_reward_block.write(current_block_number)` to **after** the `disable_rewards` guard so that a no-op call does not consume the block's reward slot.
- Or remove the `disable_rewards` parameter from the public interface and handle the disable logic internally.

---

### Proof of Concept

1. Attacker reads any valid staker address from the public `stakers` vector.
2. Each block, attacker calls `update_rewards(valid_staker, disable_rewards: true)`.
3. `last_reward_block` is updated to the current block number.
4. The function returns early — no rewards are calculated or distributed.
5. Any legitimate call to `update_rewards` in the same block reverts with `REWARDS_ALREADY_UPDATED`.
6. Repeated every block, all stakers permanently receive zero consensus rewards.
