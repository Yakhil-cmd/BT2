### Title
Pool admin bypasses `OracleValueStopLossExtension` timelock by pre-staging proposals when timelock is zero — (`File: metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

---

### Summary

`OracleValueStopLossExtension` computes `executeAfter` at **proposal time** using the then-current timelock value. When the timelock is `0`, `executeAfter` is set to `block.timestamp`, which satisfies `_requireElapsed` immediately and forever. A pool admin can pre-stage drawdown, decay, or watermark proposals while the timelock is `0`, then increase the timelock to a non-zero value, and immediately execute the pre-staged proposals — bypassing the timelock that LPs observe and rely on.

---

### Finding Description

`_afterTimelock` computes the execution timestamp at proposal time:

```solidity
function _afterTimelock(address pool_) private view returns (uint32) {
    return uint32(block.timestamp + oracleStopLossConfig[pool_].timelock);
}
``` [1](#0-0) 

When `timelock == 0`, this returns `block.timestamp`. The elapsed check is:

```solidity
function _requireElapsed(uint32 executeAfter) private view {
    if (block.timestamp < executeAfter) revert OracleStopLossTimelockNotElapsed(...);
}
``` [2](#0-1) 

`block.timestamp < block.timestamp` is always `false`, so the check passes immediately and continues to pass at any future timestamp. Every propose function uses this same helper:

- `proposeOracleStopLossDrawdown` → `pendingDrawdownExecuteAfter = _afterTimelock(pool_)`
- `proposeOracleStopLossDecay` → `pendingDecayExecuteAfter = _afterTimelock(pool_)`
- `proposeOracleStopLossHighWatermarks` → `executeAfter = _afterTimelock(pool_)`
- `proposeOracleStopLossTimelock` → `pendingTimelockExecuteAfter = _afterTimelock(pool_)` [3](#0-2) [4](#0-3) [5](#0-4) 

The timelock is mutable: the admin can change it via `proposeOracleStopLossTimelock` / `executeOracleStopLossTimelock`. The timelock defaults to `0` at initialization (no validation enforces a non-zero value):

```solidity
(uint32 drawdownE6, uint32 decayPerSecondE8, uint32 timelock) = abi.decode(data, (uint32, uint32, uint32));
oracleStopLossConfig[pool] = PoolStopLossConfig({
    drawdownE6: drawdownE6, decayPerSecondE8: decayPerSecondE8, timelock: timelock, initialized: true
});
``` [6](#0-5) 

The test suite explicitly confirms that a zero timelock allows immediate execution:

```solidity
function test_decayTimelockZeroExecutesImmediately() public {
    vm.startPrank(admin);
    extension.proposeOracleStopLossDecay(address(mockPool), 58);
    extension.executeOracleStopLossDecay(address(mockPool));
``` [7](#0-6) 

---

### Impact Explanation

The `drawdownE6` parameter is the LP stop-loss threshold. `_afterSwapOracleStopLoss` reads it on every swap and returns early if it is `0`:

```solidity
uint256 drawdown = cfg.drawdownE6;
if (drawdown == 0) return;
``` [8](#0-7) 

If the admin bypasses the timelock to set `drawdownE6 = 0` (or to a value of `1e6` = 100%), the stop-loss guard is silently disabled. Swaps that drain LP value per share below the previously enforced floor are no longer blocked. LPs who added liquidity after observing a non-zero timelock (believing they had a reaction window) suffer direct loss of principal with no on-chain warning.

The same bypass applies to `decayPerSecondE8` (accelerating watermark decay to near-zero, effectively disabling the guard) and to `highWatermarks` (resetting watermarks to arbitrarily low values).

---

### Likelihood Explanation

Pools are commonly initialized with `timelock = 0` (the default, as shown in the test `setUp` which calls `_initPool(address(mockPool), 0, 0, 0)`). [9](#0-8) 

The attack requires only that the admin:
1. Proposes a parameter change while `timelock == 0` (setting `executeAfter = block.timestamp`).
2. Increases the timelock to a non-zero value (which itself executes immediately since `timelock` is still `0` at that moment).
3. Executes the pre-staged proposal immediately, bypassing the newly visible timelock.

This is a single-block, three-transaction sequence requiring no external cooperation. The admin is semi-trusted only within caps and timelocks; bypassing the timelock is an explicit admin-boundary break.

---

### Recommendation

Revert in all `propose*` functions when the current timelock is `0`, mirroring the external report's recommendation:

```solidity
function _afterTimelock(address pool_) private view returns (uint32) {
    uint32 tl = oracleStopLossConfig[pool_].timelock;
    if (tl == 0) revert OracleStopLossTimelockNotSet(pool_);
    return uint32(block.timestamp + tl);
}
```

Alternatively, enforce a minimum non-zero timelock in `initialize` and in `executeOracleStopLossTimelock` so the timelock can never be `0` after initialization.

### Proof of Concept

```solidity
// Pool initialized with timelock = 0 (default)
extension.initialize(pool, abi.encode(uint32(900_000), uint32(0), uint32(0)));
// drawdownE6 = 900_000 (90% drawdown allowed), timelock = 0

// Step 1: Admin pre-stages a drawdown change to 0 (disables stop-loss)
// executeAfter = block.timestamp (since timelock == 0)
vm.prank(admin);
extension.proposeOracleStopLossDrawdown(pool, 0);

// Step 2: Admin proposes timelock increase to 7 days
// executeAfter = block.timestamp (since timelock is still 0)
vm.prank(admin);
extension.proposeOracleStopLossTimelock(pool, 7 days);

// Step 3: Admin executes timelock increase immediately
vm.prank(admin);
extension.executeOracleStopLossTimelock(pool);
// oracleStopLossConfig[pool].timelock == 7 days — LPs observe this and feel protected

// Step 4: Admin immediately executes the pre-staged drawdown change
// pendingDrawdownExecuteAfter == old block.timestamp <= block.timestamp → passes
vm.prank(admin);
extension.executeOracleStopLossDrawdown(pool);
// drawdownE6 == 0 → stop-loss disabled; _afterSwapOracleStopLoss returns early on every swap
// LPs have no protection despite observing a 7-day timelock
```
