### Title
Missing Access Control on `update_rewards` Allows Any Caller to Permanently Freeze All Stakers' Block Rewards - (File: src/staking/staking.cairo)

---

### Summary

The `update_rewards` function in the Staking contract accepts a caller-controlled `disable_rewards: bool` parameter with no access control beyond a basic non-zero caller check. Any unprivileged address can call `update_rewards(any_valid_staker, disable_rewards: true)` once per block to advance the global `last_reward_block` counter without distributing rewards, permanently preventing all stakers from receiving consensus block rewards for that block.

---

### Finding Description

`update_rewards` is a public function gated only by `general_prerequisites`, which checks that the contract is not paused and the caller is non-zero — no role restriction of any kind. [1](#0-0) 

The function first asserts that the current block has not already been processed: [2](#0-1) 

It then unconditionally writes the current block number to the global `last_reward_block` storage variable: [3](#0-2) 

`last_reward_block` is a single global slot shared across all stakers: [4](#0-3) 

Because `last_reward_block` is written **before** the `disable_rewards` early-return branch, any caller who passes `disable_rewards: true` will:

1. Advance `last_reward_block` to the current block.
2. Return immediately without distributing any rewards.
3. Cause every subsequent legitimate call to `update_rewards` for that block to revert with `REWARDS_ALREADY_UPDATED`.

`general_prerequisites` imposes no privileged-role check: [5](#0-4) 

The analog to the original report's vulnerability class is direct: just as a malicious strategy could freely manipulate the adapter's critical accounting storage via `delegatecall`, here an unprivileged caller can freely manipulate the critical `last_reward_block` accounting state to suppress reward distribution — the root cause in both cases is that a sensitive state-mutating path is reachable without authorization.

---

### Impact Explanation

**High — Permanent freezing of unclaimed yield.**

Block rewards that are skipped are not deferred or queued; they are permanently lost. The `last_reward_block` gate is a one-way ratchet per block: once it is advanced without distributing rewards, no mechanism exists to retroactively recover those rewards for any staker. An attacker who repeats this every block can continuously deny the entire staker set their consensus rewards at negligible cost.

---

### Likelihood Explanation

**High.** The attack requires:
- No capital or stake.
- No privileged role.
- One transaction per block calling `update_rewards(any_active_staker, disable_rewards: true)`.

The attacker only needs to submit this transaction before any legitimate `update_rewards` call in the same block. On Starknet the sequencer orders transactions, so a persistent attacker can reliably front-run the legitimate call every block.

---

### Recommendation

Add an access-control check to `update_rewards` so that only a designated privileged role (e.g., a block-proposer role, the staker themselves, or a dedicated rewards-updater role) may invoke it. At minimum, the `disable_rewards: true` path must be restricted to a trusted caller. The `last_reward_block` write should also be moved to **after** the reward distribution logic, or gated behind the same authorization check, so that a failed/skipped distribution does not permanently consume the block's reward slot.

---

### Proof of Concept

1. Consensus rewards are active (`is_pre_consensus()` returns `false`).
2. At the start of block N, attacker calls:
   ```
   staking.update_rewards(any_active_staker_address, disable_rewards: true)
   ```
3. Inside `update_rewards`:
   - `current_block_number (N) > last_reward_block` → passes.
   - `last_reward_block` is written to `N`.
   - `disable_rewards == true` → early return, zero rewards distributed.
4. The legitimate `update_rewards(staker, disable_rewards: false)` call now reverts: `current_block_number (N) <= last_reward_block (N)` → `REWARDS_ALREADY_UPDATED`.
5. All stakers permanently lose their block rewards for block N.
6. Attacker repeats at block N+1, N+2, … to continuously freeze all staker rewards. [6](#0-5)
