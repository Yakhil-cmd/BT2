### Title
Unrestricted `update_rewards` Allows Any Caller to Permanently Freeze Staker Yield via `disable_rewards: true` - (File: src/staking/staking.cairo)

### Summary

`IStakingRewardsManager::update_rewards` is specified to be callable only by the Starkware sequencer, but the implementation enforces no such restriction. Any unprivileged address can call it with `disable_rewards: true`, consuming the per-block `last_reward_block` slot without distributing rewards, permanently denying stakers and pool members their earned yield for that block.

### Finding Description

The protocol specification for `update_rewards` explicitly states:

> **access control**: Only starkware sequencer. [1](#0-0) 

However, the implementation in `StakingRewardsManagerImpl` only calls `general_prerequisites()`, which checks that the contract is unpaused and the caller is non-zero — it does **not** verify the caller is the sequencer: [2](#0-1) 

The function accepts a `disable_rewards: bool` parameter. When `true`, it writes `current_block_number` into the global `last_reward_block` storage slot and returns immediately without distributing any rewards: [3](#0-2) 

Because `last_reward_block` is a **single global variable** (not per-staker), once it is set to block `N`, every subsequent call to `update_rewards` in block `N` — regardless of which `staker_address` is passed — reverts with `REWARDS_ALREADY_UPDATED`: [4](#0-3) 

An attacker therefore needs only **one transaction per block** to prevent the legitimate sequencer from distributing rewards to **all** stakers and pool members for that block.

### Impact Explanation

- **High: Permanent freezing of unclaimed yield.** An attacker who calls `update_rewards(any_valid_staker, disable_rewards: true)` at the start of every block permanently prevents all stakers and their delegation pools from accruing consensus-era block rewards. Pool members' `cumulative_rewards_trace` is never updated, so their yield is frozen indefinitely. Because the `last_reward_block` guard is global, a single call per block is sufficient to block the entire reward pipeline. [5](#0-4) 

### Likelihood Explanation

- The function is publicly callable with no role check.
- The attacker only needs to front-run the sequencer's `update_rewards` call once per block with `disable_rewards: true`.
- No capital is required; gas cost is the only barrier.
- The attack is fully permissionless and repeatable every block.

### Recommendation

Add a caller check matching the specification. Either:

1. Introduce a dedicated `SEQUENCER_ROLE` and assert it at the top of `update_rewards`, or
2. Assert `get_caller_address() == sequencer_address` where `sequencer_address` is stored in contract storage and set by governance.

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
    self.assert_caller_is_sequencer(); // <-- add this
    ...
}
```

### Proof of Concept

```
// Block N, before the legitimate sequencer acts:
// Attacker (any address) calls:
staking.update_rewards(staker_address: any_active_staker, disable_rewards: true);
// → last_reward_block is now N, no rewards distributed.

// Sequencer then tries (same block N):
staking.update_rewards(staker_address: staker_A, disable_rewards: false);
// → PANICS: REWARDS_ALREADY_UPDATED

staking.update_rewards(staker_address: staker_B, disable_rewards: false);
// → PANICS: REWARDS_ALREADY_UPDATED
// All stakers lose rewards for block N.
// Repeat every block → permanent yield freeze.
```

The flow test at `src/flow_test/test.cairo` line 2786 already demonstrates that calling with `disable_rewards: true` produces zero rewards, confirming the mechanism: [6](#0-5)
