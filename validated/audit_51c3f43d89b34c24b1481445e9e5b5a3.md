### Title
Blocklist Front-Run: Blocked Address Bypasses `MsgStoreBlockList` in the Same Block — (`app/block_address.go`, `app/proposal.go`, `app/app.go`)

---

### Summary

The Cronos blocklist enforcement has a one-block timing gap identical in structure to the external report's freeze-bypass. `BlockAddressesDecorator.AnteHandle` only enforces the blocklist during `CheckTx` (mempool admission), never during `DeliverTx` (block execution). The `ProposalHandler` enforces the blocklist during `PrepareProposal`/`ProcessProposal`, but uses an **in-memory** map that is only refreshed at `EndBlock` — after all transactions in the block have already executed. A targeted address can therefore front-run the admin's `MsgStoreBlockList` transaction, land its own transaction in the same block, and have it execute with no blocklist check applied.

---

### Finding Description

**Step 1 — `BlockAddressesDecorator` skips the check during block execution.** [1](#0-0) 

The entire signer/destination check is wrapped in `if ctx.IsCheckTx()`. During `DeliverTx` (block execution), `ctx.IsCheckTx()` is `false`, so the decorator is a no-op. Any transaction that reaches block execution is never re-validated against the blocklist.

**Step 2 — `ProcessProposalHandler` uses a stale in-memory blocklist.** [2](#0-1) 

The handler iterates `req.Txs` and calls `ValidateTransaction` against `h.blocklist` — an in-memory map. This map is only updated by `SetBlockList`, which is called from `RefreshBlockList`.

**Step 3 — `RefreshBlockList` is called at `EndBlock`, after all transactions have executed.** [3](#0-2) 

`RefreshBlockList` reads the on-chain KV store and pushes the new map into `h.blocklist`. This happens **after** every transaction in the block has already been delivered.

**Step 4 — `StoreBlockList` writes to the KV store during transaction execution.** [4](#0-3) 

When `MsgStoreBlockList` executes, the new blob is written to `types.KeyPrefixBlockList`. The in-memory map used by `ProcessProposalHandler` is not updated until `EndBlock` completes.

**Consequence:** For block N that contains `MsgStoreBlockList`:
- `ProcessProposalHandler` validates block N's transactions against the blocklist from EndBlock of block N-1 (does not yet contain address X).
- `BlockAddressesDecorator` does not check during `DeliverTx`.
- Address X's transaction in block N executes with zero blocklist enforcement.
- Only at EndBlock of block N does `h.blocklist` gain address X, taking effect from block N+1.

---

### Impact Explanation

This is a **High** impact: **bypass of block-list authorization checks**. The blocklist is Cronos's primary mechanism to prevent sanctioned/malicious addresses from transacting. A targeted address can front-run the blocklist update to:
- Transfer all CRO or CRC20/ERC20 tokens to a fresh address before being blocked.
- Initiate an IBC transfer (`MsgTransferTokens`) or EVM bridge operation to move funds off-chain.
- Interact with any on-chain contract in the same block as the blocklist update.

Once the funds are moved, the blocklist is effectively useless against that actor.

---

### Likelihood Explanation

The attack requires only:
1. Monitoring the mempool for a `MsgStoreBlockList` transaction (trivial on a public node).
2. Submitting a competing transaction with higher gas priority before the block is finalized.

No privileged access, leaked keys, or cryptographic break is needed. The attacker is the address being blocked — an unprivileged user. The window is exactly one block, but that is sufficient to execute a full asset transfer or bridge operation.

---

### Recommendation

1. **Enforce the blocklist during `DeliverTx`** by removing the `if ctx.IsCheckTx()` guard in `BlockAddressesDecorator.AnteHandle`, or adding a separate `DeliverTx`-time check. This closes the execution-time gap regardless of proposal-layer timing.

2. **Alternatively, refresh the in-memory blocklist at `BeginBlock`** (reading the KV store written by the previous block's `MsgStoreBlockList`) so that `ProcessProposalHandler` for block N+1 already has the updated list. This does not fully close the same-block window but eliminates the one-block lag.

The most robust fix is (1): enforce the check unconditionally in the AnteHandler so that even if a blocked transaction somehow reaches `DeliverTx`, it is rejected at the state-machine level.

---

### Proof of Concept

1. Admin broadcasts `MsgStoreBlockList` adding address `X` to the blocklist (encrypted blob stored via `ctx.KVStore(k.storeKey).Set(types.KeyPrefixBlockList, msg.Blob)`).
2. Address `X` observes this transaction in the mempool.
3. `X` broadcasts `MsgTransferTokens` (or an EVM `MsgEthereumTx` transferring all CRC20 tokens) with a higher gas price.
4. Both transactions are included in block N. `ProcessProposalHandler` validates block N using `h.blocklist` from EndBlock of block N-1 — `X` is not yet in it — so the block is accepted.
5. During `DeliverTx` for `X`'s transaction, `BlockAddressesDecorator.AnteHandle` is called with `ctx.IsCheckTx() == false`, so the entire blocklist check is skipped. The transfer executes.
6. `MsgStoreBlockList` executes, writing the new blob to the KV store.
7. `EndBlocker` calls `RefreshBlockList`, decrypting the blob and populating `h.blocklist` with `X`.
8. From block N+1, `X` is blocked — but its funds have already been moved.

func (h *ProposalHandler) ProcessProposalHandler() sdk.ProcessProposalHandler {
	return func(ctx sdk.Context, req *abci.RequestProcessProposal) (*abci.ResponseProcessProposal, error) {
		if len(h.blocklist) == 0 {
			// fast path, accept all txs
			return &abci.ResponseProcessProposal{Status: abci.ResponseProcessProposal_ACCEPT}, nil
		}

		for _, txBz := range req.Txs {
			if err := h.ValidateTransaction(nil, txBz); err != nil {
				return &abci.ResponseProcessProposal{Status: abci.ResponseProcessProposal_REJECT}, nil
			}
		}

		return &abci.ResponseProcessProposal{Status: abci.ResponseProcessProposal_ACCEPT}, nil
	}
}

// EndBlocker application updates every end block
func (app *App) EndBlocker(ctx sdk.Context) (sdk.EndBlock, error) {
	rsp, err := app.ModuleManager.EndBlock(ctx)
	if err := app.RefreshBlockList(ctx); err != nil {
		app.Logger().Error("failed to update blocklist", "error", err)
	}
	return rsp, err
}

func (app *App) RefreshBlockList(ctx sdk.Context) error {
	// refresh blocklist
	return app.blockProposalHandler.SetBlockList(app.CronosKeeper.GetBlockList(ctx))
}
