### Title
Unbounded Loop in `calculate_rewards` Enables Permanent Freezing of Delegator Unclaimed Yield - (File: src/pool/pool.cairo)

### Summary
The `calculate_rewards` function in `src/pool/pool.cairo` contains an explicitly acknowledged unbounded loop that iterates over every balance-change checkpoint in a pool member's `pool_member_epoch_balance` trace since their last reward claim. Because each call to `add_to_delegation_pool` or `exit_delegation_pool_intent` in a new epoch appends a new checkpoint, a delegator who makes N balance changes across N distinct epochs without claiming rewards will force `claim_rewards` to execute N loop iterations. If N grows large enough, the transaction exceeds Starknet's gas limit, permanently freezing the delegator's unclaimed yield with no recovery path.

### Finding Description

**Root cause — the unbounded loop:**

In `calculate_rewards`, the loop has no iteration cap:

```cairo
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
    ...
    let to_sigma = self.find_sigma(pool_member_checkpoint, curr_epoch: until_epoch);
    ...
    entry_to_claim_from += 1;
}
``` [1](#0-0) 

Each iteration performs at least one storage read (`pool_member_trace.at(...)`) plus a `find_sigma` call that itself reads from `cumulative_rewards_trace_vec` in storage — making each iteration gas-heavy.

**How the trace grows:**

`set_member_balance` inserts a new checkpoint keyed at `current_epoch + K`. The underlying `insert` logic only updates in-place when the key matches the last entry; otherwise it appends a new entry:

```cairo
trace.insert(key: self.get_epoch_plus_k(), value: pool_member_balance);
``` [2](#0-1) 

```cairo
} else {
    assert!(last.key < key, "{}", TraceErrors::UNORDERED_INSERTION);
    checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
}
``` [3](#0-2) 

`set_member_balance` is called by both `add_to_delegation_pool` (via `increase_member_balance`) and `exit_delegation_pool_intent` on every invocation: [4](#0-3) [5](#0-4) 

Therefore, one balance-change call per epoch = one new trace entry per epoch.

**Why `entry_to_claim_from` does not protect against this:**

`entry_to_claim_from` is stored in `pool_member_info` and advanced only after a *successful* `claim_rewards` call:

```cairo
pool_member_info.entry_to_claim_from = updated_entry_to_claim_from;
``` [6](#0-5) 

If the delegator never calls `claim_rewards`, `entry_to_claim_from` stays at 0 while `pool_member_trace_length` grows without bound. When `claim_rewards` is eventually called, the loop must traverse every accumulated entry in one transaction. There is no partial-claim or batched-claim mechanism.

**Attacker-controlled entry path:**

A delegator (unprivileged staker) calls `add_to_delegation_pool` with a minimal token amount (e.g., 1 unit) once per epoch over many epochs, never calling `claim_rewards`. The access control only requires the caller to be the pool member or their reward address:

```cairo
assert!(
    caller_address == pool_member || caller_address == pool_member_info.reward_address,
    "{}",
    Error::CALLER_CANNOT_ADD_TO_POOL,
);
``` [7](#0-6) 

This is a fully reachable, no-privilege path.

### Impact Explanation

When the trace grows large enough that a single `claim_rewards` call exceeds Starknet's per-transaction gas limit, the transaction reverts. Because there is no partial-claim mechanism and `entry_to_claim_from` only advances on success, the delegator's accumulated unclaimed yield becomes permanently inaccessible. This matches the **High** impact category: *Permanent freezing of unclaimed yield*.

### Likelihood Explanation

A delegator who regularly adjusts their delegation (a normal protocol use-case) and defers reward claims for an extended period will naturally accumulate many trace entries. The protocol itself encourages long-term delegation. The comment in the code ("unlikely to exceed gas limits") acknowledges the risk but provides no enforcement. Given Starknet's bounded per-transaction gas, a sufficiently active delegator over a long enough horizon will trigger this condition without any adversarial intent.

### Recommendation

Add a maximum iteration cap to `calculate_rewards` and return a partial result, allowing `claim_rewards` to be called multiple times to drain the backlog. Store the advanced `entry_to_claim_from` even on partial runs so progress is preserved across calls. Alternatively, enforce a maximum number of unclaimed balance-change epochs before a new balance change is permitted, forcing users to claim before accumulating further entries.

### Proof of Concept

1. Delegator enters a pool via `enter_delegation_pool`.
2. Each epoch, delegator calls `add_to_delegation_pool` with amount = 1 (minimum non-zero). Each call in a new epoch appends one entry to `pool_member_epoch_balance` trace.
3. Delegator never calls `claim_rewards`.
4. After N epochs (N large enough to exhaust gas), delegator calls `claim_rewards`.
5. `calculate_rewards` enters the loop with `entry_to_claim_from = 0` and `pool_member_trace_length = N`.
6. Each of the N iterations reads storage (`pool_member_trace.at(i)`) and calls `find_sigma` (additional storage reads from `cumulative_rewards_trace`).
7. Transaction exceeds gas limit and reverts.
8. `entry_to_claim_from` is never updated (revert rolls back state).
9. All subsequent `claim_rewards` calls also revert — unclaimed yield is permanently frozen.
