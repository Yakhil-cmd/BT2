### Title
Pool Admin Bypasses `maxAdminSpreadFeeE6` Cap via Uncapped `setPoolBinAdditionalFees` — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

The factory enforces a hard cap on admin-settable spread fees via `maxAdminSpreadFeeE6` (hard ceiling 20 %). However, the parallel `setPoolBinAdditionalFees` path that sets per-bin additional buy/sell fees carries **no cap check at all**, allowing the pool admin to impose up to an additional 6.5535 % fee per bin on every swap, completely outside the cap accounting system.

---

### Finding Description

`setPoolAdminFees` correctly enforces the cap:

```solidity
// MetricOmmPoolFactory.sol L414-415
if (newAdminSpreadFeeE6 > maxAdminSpreadFeeE6) revert AdminFeeTooHigh();
if (newAdminNotionalFeeE8 > maxAdminNotionalFeeE8) revert AdminFeeTooHigh();
``` [1](#0-0) 

But `setPoolBinAdditionalFees` passes the values straight through with **no validation**:

```solidity
// MetricOmmPoolFactory.sol L450-457
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
``` [2](#0-1) 

The pool's `setBinAdditionalFees` also performs no cap check — only a bin-index range check:

```solidity
// MetricOmmPool.sol L464-474
function setBinAdditionalFees(int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external onlyFactory nonReentrant(PoolActions.SET_BIN_ADDITIONAL_FEES)
{
    if (bin < LOWEST_BIN || bin > HIGHEST_BIN) revert InvalidBinIndex(bin);
    BinState storage s = _binStates[bin];
    s.addFeeBuyE6 = addFeeBuyE6;
    s.addFeeSellE6 = addFeeSellE6;
    ...
}
``` [3](#0-2) 

These bin additional fees are applied directly in every swap path on top of the oracle base spread:

```solidity
// MetricOmmPool.sol L910
params.baseFeeX64 + Math.mulDiv(binState.addFeeBuyE6, ONE_X64, 1e6)
``` [4](#0-3) 

The `uint16` type allows values up to 65 535, which in E6 scale equals **6.5535 %** per bin. This is entirely additive on top of the already-capped 20 % admin spread fee and is never counted against `maxAdminSpreadFeeE6`. The `BinState` struct confirms the field is a raw `uint16` with no stored ceiling: [5](#0-4) 

---

### Impact Explanation

A pool admin can call `setPoolBinAdditionalFees` with `addFeeBuyE6 = 65535` and `addFeeSellE6 = 65535` for every configured bin. Every subsequent swap in those bins pays an extra 6.5535 % on top of the protocol-intended maximum, with the surplus accruing to the bin (LP balances). Because the admin can simultaneously be an LP, this constitutes a direct extraction of value from swappers beyond the cap the protocol owner intended to enforce. The effective total fee (oracle spread + capped 20 % admin spread + 6.5535 % bin additional) can reach ~26.5 %, violating the admin-boundary invariant and causing direct loss of user principal on every swap.

---

### Likelihood Explanation

The pool admin is a semi-trusted role that is explicitly constrained by `maxAdminSpreadFeeE6`. The bypass requires only a single transaction with no timelock. Any pool admin — whether malicious from the start or later compromised — can execute this immediately and silently, since the `BinAdditionalFeesUpdated` event is the only on-chain signal and there is no off-chain enforcement of a cap on bin fees.

---

### Recommendation

Add a cap check in `setPoolBinAdditionalFees` (factory layer) mirroring the pattern used in `setPoolAdminFees`:

```solidity
function setPoolBinAdditionalFees(address pool, int8 bin, uint16 addFeeBuyE6, uint16 addFeeSellE6)
    external override nonReentrant onlyPoolAdmin(pool)
{
    if (uint256(addFeeBuyE6) + uint256(addFeeSellE6) > maxAdminBinAdditionalFeeE6)
        revert AdminFeeTooHigh();
    IMetricOmmPoolFactoryActions(pool).setBinAdditionalFees(bin, addFeeBuyE6, addFeeSellE6);
}
```

Alternatively, enforce that `addFeeBuyE6` and `addFeeSellE6` individually cannot exceed `maxAdminSpreadFeeE6`, so the bin additional fee is always counted within the same cap envelope as the global admin spread fee.

---

### Proof of Concept

1. Protocol owner deploys factory with `maxAdminSpreadFeeE6 = 200_000` (20 %).
2. Pool admin calls `setPoolAdminFees(pool, 200_000, 0)` — sets admin spread to the maximum allowed.
3. Pool admin calls `setPoolBinAdditionalFees(pool, 0, 65535, 65535)` — no revert; bin 0 now carries an extra 6.5535 % buy and sell fee.
4. A swapper executes a swap through bin 0. The effective fee applied is `baseFeeX64 + mulDiv(65535, ONE_X64, 1e6)`, i.e., oracle spread + 6.5535 % additional, on top of the 20 % admin spread already stored in `spreadFeeE6`.
5. The swapper pays ~26.5 %+ in total fees; the surplus beyond the 20 % cap accrues to the bin balance, benefiting the admin-as-LP.
