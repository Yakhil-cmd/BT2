### Title
Gateway L2 Gas Price Threshold Uses Stale `l2_gas_price` Instead of `next_l2_gas_price` from Block Header - (File: crates/apollo_gateway/src/stateful_transaction_validator.rs)

---

### Summary

The gateway's stateful transaction validator checks a transaction's L2 gas price against the **current block's** `l2_gas_price` rather than the **next block's** `next_l2_gas_price`. Because the next block's price is what will actually govern execution, the admission threshold is systematically wrong: when the price is rising, under-priced transactions are admitted; when the price is falling, valid transactions are rejected. The codebase itself acknowledges this with a `TODO` comment at the exact line.

---

### Finding Description

In `validate_resource_bounds`, the validator reads the L2 gas price reference value as:

```rust
// TODO(Arni): getnext_l2_gas_price from the block header.
let previous_block_l2_gas_price = self
    .gateway_fixed_block_state_reader
    .get_block_info()
    .await?
    .gas_prices
    .strk_gas_prices
    .l2_gas_price;
``` [1](#0-0) 

`get_block_info()` is implemented in `GatewayFixedBlockSyncStateClient::get_block_info_from_sync_client()`, which constructs a `BlockInfo` from `BlockHeaderWithoutHash`. It copies `block_header.l2_gas_price.price_in_fri` into `strk_gas_prices.l2_gas_price`, but **silently drops** `block_header.next_l2_gas_price`:

```rust
strk_gas_prices: GasPriceVector {
    ...
    l2_gas_price: block_header.l2_gas_price.price_in_fri.try_into()?,
},
``` [2](#0-1) 

`BlockHeaderWithoutHash` carries both fields:

```rust
pub l2_gas_price: GasPricePerToken,   // price used in block N (already committed)
pub next_l2_gas_price: GasPrice,      // price that will govern block N+1 (being built)
``` [3](#0-2) 

The `GatewayFixedBlockStateReader` trait only exposes `BlockInfo`, which has no `next_l2_gas_price` field, so the correct value is structurally unreachable through the current interface:

```rust
pub trait GatewayFixedBlockStateReader: Send + Sync {
    async fn get_block_info(&self) -> StarknetResult<BlockInfo>;
    ...
}
``` [4](#0-3) 

The EIP-1559 adjustment formula can change the price by up to `price / gas_price_max_change_denominator` per block (denominator = 8 in the normal path, yielding up to ~12.5% per block at full congestion; the gradual-increase path uses denominator 333 ≈ 0.3%): [5](#0-4) 

The `next_l2_gas_price` is written into the committed block header by the orchestrator as `self.l2_gas_price` (the price computed for the next block), confirming it is a distinct, already-computed value available at admission time: [6](#0-5) 

---

### Impact Explanation

**High — Mempool/gateway admission accepts invalid transactions or rejects valid transactions before sequencing.**

Two concrete scenarios arise every block where the L2 gas price changes:

1. **Price rising (congested network):** `next_l2_gas_price > l2_gas_price`. The threshold is computed from the lower, stale value. Transactions whose `max_price_per_unit` falls in the range `[threshold(l2_gas_price), threshold(next_l2_gas_price))` pass the gateway check and enter the mempool, but their gas price is below what the next block will require. They will either be reverted or fail the batcher's fee-bound check during execution.

2. **Price falling (uncongested network):** `next_l2_gas_price < l2_gas_price`. The threshold is computed from the higher, stale value. Transactions whose `max_price_per_unit` falls in `[threshold(next_l2_gas_price), threshold(l2_gas_price))` are incorrectly rejected at the gateway even though they would be valid for the next block.

The magnitude per block is bounded by the EIP-1559 denominator, but the error is systematic and cumulative across every block where gas consumption deviates from the target.

---

### Likelihood Explanation

**High.** The L2 gas price changes on virtually every block (any deviation from the gas target moves it). The `validate_resource_bounds` path is active whenever `config.validate_resource_bounds = true`, which is the production default (the flag exists only to skip validation during bootstrap). No attacker action is required; the condition is triggered by ordinary network load.

---

### Recommendation

1. Extend `GatewayFixedBlockStateReader` to expose `next_l2_gas_price` directly (e.g., add `async fn get_next_l2_gas_price() -> StarknetResult<GasPrice>`), reading it from `BlockHeaderWithoutHash::next_l2_gas_price`.
2. In `validate_resource_bounds`, replace the call to `get_block_info().gas_prices.strk_gas_prices.l2_gas_price` with the new accessor, resolving the existing `TODO(Arni)` comment.
3. Update `GatewayFixedBlockSyncStateClient` and all mock/test implementations of the trait accordingly.

---

### Proof of Concept

**Setup:** Deploy a sequencer node with `validate_resource_bounds = true` and `min_gas_price_percentage = 100`. Drive the network to sustained high load so that `next_l2_gas_price > l2_gas_price` by the EIP-1559 formula.

**Trigger (price-rising case):**
1. Read the latest committed block header; note `l2_gas_price = P` and `next_l2_gas_price = P' > P`.
2. Submit an `InvokeTransaction` (V3, `AllResources`) with `l2_gas.max_price_per_unit = P` (exactly equal to the stale threshold, below `P'`).
3. The gateway computes `threshold = min_gas_price_percentage% * P = P` and admits the transaction (`P >= P` passes).
4. The batcher builds the next block with gas price `P'`. The transaction's committed max fee is insufficient at price `P'`; the transaction reverts or is dropped.

**Trigger (price-falling case):**
1. Drive the network to low load so `next_l2_gas_price = P' < P`.
2. Submit a transaction with `l2_gas.max_price_per_unit = P'` (valid for the next block).
3. The gateway computes `threshold = P` and rejects the transaction with `GAS_PRICE_TOO_LOW`, even though the transaction would succeed in the next block.

The root cause is confirmed by the `TODO` comment at line 229 of `crates/apollo_gateway/src/stateful_transaction_validator.rs` and the structural absence of `next_l2_gas_price` from the `BlockInfo` type returned by `GatewayFixedBlockStateReader::get_block_info()`. [7](#0-6)

```rust
pub fn calculate_next_base_gas_price(
    price: GasPrice,
    gas_used: GasAmount,
    gas_target: GasAmount,
    min_gas_price: GasPrice,
) -> GasPrice {
    let versioned_constants = VersionedConstants::latest_constants();
    assert!(
        gas_target < versioned_constants.max_block_size,
        "Gas target must be lower than max block size."
    );
    assert!(gas_target.0 > 0, "Gas target must be greater than zero.");
    assert!(
        versioned_constants.gas_price_max_change_denominator > 0,
        "Denominator constant must be greater than zero."
    );

    // If the current price is below the minimum, apply a gradual adjustment and return early.
    // This allows the price to increase by at most 1/MIN_GAS_PRICE_INCREASE_DENOMINATOR per block.
    if price < min_gas_price {
        let max_increase = price.0 / MIN_GAS_PRICE_INCREASE_DENOMINATOR;
        let adjusted = price.0 + max_increase;
        // Cap at min_gas_price to avoid overshooting
        let adjusted_price = adjusted.min(min_gas_price.0);
        info!(
            "Fee Market: Price {} below minimum gas price {}, adjusted price: {} )",
            price.0, min_gas_price.0, adjusted_price
        );
        return GasPrice(adjusted_price);
    }

    // Use U256 to avoid overflow, as multiplying a u128 by a u64 remains within U256 bounds.
    let gas_delta = U256::from(gas_used.0.abs_diff(gas_target.0));
    let gas_target_u256 = U256::from(gas_target.0);
    let price_u256 = U256::from(price.0);

    // Calculate price change by multiplying first, then dividing. This avoids the precision loss
    // that occurs when dividing before multiplying.
    let denominator =
        gas_target_u256 * U256::from(versioned_constants.gas_price_max_change_denominator);
    let price_change = (price_u256 * gas_delta) / denominator;

    let adjusted_price_u256 =
        if gas_used > gas_target { price_u256 + price_change } else { price_u256 - price_change };

    // Sanity check: ensure direction of change is correct
    assert!(
        gas_used > gas_target && adjusted_price_u256 >= price_u256
            || gas_used <= gas_target && adjusted_price_u256 <= price_u256
    );

    // Price should not realistically exceed u128::MAX, bound to avoid theoretical overflow.
    let adjusted_price = u128::try_from(adjusted_price_u256).unwrap_or(u128::MAX);
    GasPrice(max(adjusted_price, min_gas_price.0))
```
