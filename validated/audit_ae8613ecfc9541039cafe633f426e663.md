### Title
Unprotected Legacy Transaction Signing Hash Lacks Chain-ID Domain Separation, Enabling Cross-Chain Replay - (File: `basic_bootloader/src/bootloader/transaction/rlp_encoded/transaction_types/legacy_tx.rs`)

### Summary

ZKsync OS's bootloader accepts pre-EIP-155 ("unprotected") legacy transactions whose signing hash is computed over only the 6-field payload `[nonce, gasPrice, gasLimit, to, value, data]` with no chain ID folded in. A valid unprotected legacy transaction signed on Ethereum mainnet (or any other EVM chain) produces an identical signing digest on ZKsync OS, so `ecrecover` succeeds and the transaction executes without the signer's intent.

### Finding Description

In `LegacyPayloadParser::try_parse_and_hash_for_signature_verification`, when the signature's `v` field equals `27` or `28` (`is_eip155() == false`), the code takes the **unprotected branch** and hashes only the raw 6-field RLP payload:

```rust
let sig_hash: Bytes32 = if legacy_signature.is_eip155() == false {

    let mut hasher = crypto::sha3::Keccak256::new();
    apply_list_concatenation_encoding_to_hash(inner_slice.len() as u32, &mut hasher);
    hasher.update(inner_slice);
    hasher.finalize_reset().into()   // ← no chain ID
} else { … }

Because the signing hash is identical on every EVM chain for unprotected legacy transactions, a signature that is valid on Ethereum mainnet is equally valid on ZKsync OS.

### Impact Explanation

An attacker who observes a valid unprotected legacy transaction on Ethereum mainnet (or any other EVM chain) can submit the same raw bytes to ZKsync OS. If the victim's account nonce on ZKsync OS matches the nonce in the replayed transaction, the bootloader will:

1. Parse the transaction successfully.
2. Compute the same chain-ID-free signing hash.
3. Recover the correct `from` address via `ecrecover`.
4. Pass nonce validation.
5. Execute the transaction — transferring ETH or calling contracts — without the signer's authorization on ZKsync OS.

This constitutes unauthorized movement of user funds and unauthorized contract calls, directly matching the Immunefi scope for state-transition integrity failures.

### Likelihood Explanation

Pre-EIP-155 (unprotected) legacy transactions remain in active use. Many hardware wallets, older tooling, and some DeFi protocols still produce `v=27/28` signatures. An attacker can passively monitor public mempools or block explorers on Ethereum mainnet for unprotected legacy transactions, then replay them on ZKsync OS against victims whose nonce happens to match. The nonce requirement reduces but does not eliminate exploitability — accounts that are new to ZKsync OS (nonce=0) are particularly vulnerable if they have ever sent an unprotected legacy transaction with nonce=0 on any other chain.

### Recommendation

Reject unprotected legacy transactions (v=27 or v=28) at the parser level. ZKsync OS is a distinct chain and has no obligation to support the pre-EIP-155 format. Add an explicit check in `parse_and_compute_signed_hash`:

```rust
if !sig_data.is_eip155() {
    return Err(InvalidTransaction::InvalidChainId.into());
}
```

Alternatively, if unprotected legacy support is intentionally required for EVM equivalence, document the replay risk and add a network-level policy (e.g., a sequencer-side filter) that rejects unprotected legacy transactions before they reach the bootloader.

### Proof of Concept

1. On Ethereum mainnet, find any unprotected legacy transaction with `v=27` or `v=28` (e.g., from a hardware wallet or old tooling). Call its raw bytes `raw_tx`.
2. Note the sender address `A`, nonce `N`, destination `D`, and value `V`.
3. Ensure address `A` on ZKsync OS has nonce `N` and sufficient balance (e.g., a fresh account with nonce 0 that received ETH via a bridge).
4. Submit `raw_tx` verbatim to ZKsync OS via the RPC.
5. The bootloader parses it as `RlpEncodedTxInner::Legacy`, computes the same 6-field hash (no chain ID), recovers `A` via `ecrecover`, passes nonce validation, and executes the transfer of `V` ETH to `D` — a transaction the owner of `A` never intended to execute on ZKsync OS.
