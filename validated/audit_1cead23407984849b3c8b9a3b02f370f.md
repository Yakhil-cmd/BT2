### Title
Router-Mediated Swap Passes Router Address as `sender` to `SwapAllowlistExtension`, Allowing Any Trader to Bypass the Allowlist — (`metric-periphery/contracts/extensions/SwapAllowlistExtension.sol`)

---

### Summary

`SwapAllowlistExtension.beforeSwap` gates swaps by checking `allowedSwapper[pool][sender]`, where `sender` is the `msg.sender` of `MetricOmmPool.swap`. When a trader routes through `MetricOmmSimpleRouter`, the pool's `msg.sender` is the **router**, not the trader. If the router address is allowlisted for a pool, every public trader can bypass the allowlist by routing through the router.

---

### Finding Description

The call chain is:

```
trader → MetricOmmSimpleRouter.exactInputSingle(msg.sender=trader)
       → pool.swap(recipient, ...) [msg.sender=router]
       → MetricOmmPool._beforeSwap(msg.sender=router, ...)
       → ExtensionCalling._beforeSwap(sender=router, ...)
       → SwapAllowlistExtension.beforeSwap(sender=router, msg.sender=pool)
       → allowedSwapper[pool][router]  ← checks router, not trader
```

**Step 1 — Router calls pool directly, losing trader identity.**

`MetricOmmSimpleRouter.exactInputSingle` calls `pool.swap` without forwarding the original caller: [1](#0-0) 

The pool receives `msg.sender = router`.

**Step 2 — Pool passes its own `msg.sender` as `sender` to the extension.**

`MetricOmmPool.swap` passes `msg.sender` (the router) as the `sender` argument to `_beforeSwap`: [2](#0-1) 

**Step 3 — Extension checks the router address, not the trader.**

`SwapAllowlistExtension.beforeSwap` uses `msg.sender` (= pool) as the pool key and `sender` (= router) as the swapper identity: [3](#0-2) 

The check is `allowedSwapper[pool][router]`. If the router is allowlisted, the check passes for **every trader** regardless of whether they are individually allowlisted.

**The structural incompatibility:**

The `SwapAllowlistExtension` is fundamentally incompatible with the router. A pool admin who wants to support router-based swaps must allowlist the router address. But doing so opens the gate to all traders, defeating the allowlist entirely. There is no mechanism in the extension to recover the original `msg.sender` of the router call.

---

### Impact Explanation

A curated pool using `SwapAllowlistExtension` (e.g., a KYC-gated or institutional pool) can have its access control completely bypassed. Any trader — including those explicitly not in `allowedSwapper` — can execute swaps on the restricted pool by routing through `MetricOmmSimpleRouter`. The allowlist invariant ("only approved addresses may swap") is broken for all router-mediated paths. This constitutes broken core pool functionality and a direct admin-boundary break.

---

### Likelihood Explanation

The scenario requires the router to be allowlisted for the pool. This is a natural and expected admin action: a pool admin who deploys a curated pool and wants users to be able to use the standard periphery router would allowlist the router. The router is a public, permissionless contract. Once the router is allowlisted, the bypass is trivially reachable by any trader.

---

### Recommendation

The `sender` passed to `beforeSwap` must represent the economic beneficiary of the swap, not the immediate `msg.sender` of the pool. Two approaches:

1. **Pass the original caller through the router.** Add an optional `originator` field to the swap call or extension data so the router can forward `msg.sender` to the extension. The extension would then check `allowedSwapper[pool][originator]`.

2. **Check `recipient` instead of `sender`.** Since `recipient` is the address that receives output tokens, it is a closer proxy for the economic beneficiary. However, this can also be spoofed by setting `recipient` to an allowlisted address while the actual payer is not.

The cleanest fix is approach 1: the router should forward its own `msg.sender` as an authenticated originator field, and the extension should gate on that value.

### Proof of Concept

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.35;

// Setup:
// 1. Deploy pool with SwapAllowlistExtension wired to beforeSwap
// 2. allowlist ONLY the router address (not the trader)
// 3. trader (not allowlisted) calls router.exactInputSingle → succeeds
// 4. trader calls pool.swap directly → reverts with NotAllowedToSwap

function test_routerBypassesSwapAllowlist() public {
    // allowlist only the router, not the trader
    swapExtension.setAllowedToSwap(address(pool), address(router), true);
    // trader is NOT allowlisted

    uint256 outBefore = token1.balanceOf(trader);

    // trader routes through router — should revert but does not
    vm.prank(trader);
    router.exactInputSingle(IMetricOmmSimpleRouter.ExactInputSingleParams({
        pool: address(pool),
        tokenIn: address(token0),
        tokenOut: address(token1),
        zeroForOne: true,
        amountIn: 1_000,
        amountOutMinimum: 0,
        recipient: trader,
        deadline: type(uint256).max,
        priceLimitX64: 0,
        extensionData: ""
    }));

    // trader received output despite not being allowlisted
    assertGt(token1.balanceOf(trader), outBefore, "bypass succeeded");

    // direct call from trader correctly reverts
    vm.prank(trader);
    vm.expectRevert(IMetricOmmPoolActions.NotAllowedToSwap.selector);
    pool.swap(trader, true, int128(1_000), 0, "", "");
}
```

The router call succeeds because `allowedSwapper[pool][router] = true`, while the direct call correctly reverts because `allowedSwapper[pool][trader] = false`.
