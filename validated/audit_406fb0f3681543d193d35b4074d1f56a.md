### Title
Unbounded Loop Over `pool_member_epoch_balance` Trace in `calculate_rewards` Can Permanently Freeze Pool Member's Unclaimed Yield - (File: `src/pool/pool.cairo`)

---

### Summary

The `calculate_rewards` function in `src/pool/pool.cairo` iterates over every entry in a pool member's `pool_member_epoch_balance` trace without any bound. A pool member who repeatedly changes their delegated balance across many epochs (without claiming rewards) will grow this trace unboundedly. Once the trace is large enough, the `claim_rewards` call will exceed Starknet's transaction gas limit, permanently freezing the pool member's unclaimed yield with no recovery path.

---

### Finding Description

In `src/pool/pool.cairo`, the internal `calculate_rewards` function is called by `claim_rewards` to compute how much a pool member is owed. It iterates over the `pool_member_epoch_balance` trace from `entry_to_claim_from` up to the current epoch:

```cairo
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
    if pool_member_checkpoint.epoch() >= until_epoch {
        break;
    }
    // ... reward computation per entry ...
    entry_to_claim_from += 1;
}
```

The trace (`pool_member_epoch_balance`) grows by one entry each time `set_member_balance` is called with a new epoch key (i.e., a different epoch than the last stored entry). The `insert` logic in `trace.cairo` only deduplicates within the same epoch — calls in different epochs always append a new checkpoint:

```cairo
} else {
    // Checkpoint keys must be non-decreasing.
    assert!(last.key < key, "{}", TraceErrors::UNORDERED_INSERTION);
    checkpoints.push(PoolMemberBalanceCheckpoint { key, value });
}
```

`set_member_balance` is called (directly or via `increase_member_balance`) from:
- `enter_delegation_pool` — on first join
- `add_to_delegation_pool` — on every top-up
- `exit_delegation_pool_intent` — on every partial/full exit intent
- `enter_delegation_pool_from_staking_contract` — on pool switch

Each of these operations, if performed in a different epoch, appends a new entry. The `entry_to_claim_from` pointer in `InternalPoolMemberInfoV1` advances only when `claim_rewards` is successfully called. If a pool member changes their balance every epoch without ever claiming, the trace grows at a rate of one entry per epoch indefinitely.

There is no cap on the trace length, no batching mechanism in `claim_rewards`, and no alternative partial-claim path. The developers themselves acknowledge this risk with the comment: *"This loop is unbounded but unlikely to exceed gas limits."*

---

### Impact Explanation

**Impact: High — Permanent freezing of unclaimed yield.**

Once the `pool_member_epoch_balance` trace for a given pool member grows large enough that iterating it in `calculate_rewards` exceeds the Starknet transaction gas limit, every call to `claim_rewards` for that member will revert. Since there is no batching or partial-claim mechanism, the pool member's accumulated rewards become permanently inaccessible. The funds are not stolen, but they are frozen forever — matching the allowed impact of *"Permanent freezing of unclaimed yield."*

---

### Likelihood Explanation

**Likelihood: Low.**

The trace grows at most one entry per epoch. Epochs are measured in blocks (not seconds), so accumulating thousands of entries requires years of active balance changes without ever claiming. In practice, most pool members will claim periodically, resetting `entry_to_claim_from` and keeping the effective loop length small. However, the risk is non-zero for long-lived, actively-managed positions that never claim, and the protocol provides no safeguard or warning.

---

### Recommendation

1. **Add a maximum trace length cap**: Enforce a hard limit on the number of entries in `pool_member_epoch_balance` per pool member (e.g., reject `set_member_balance` if the trace exceeds a safe threshold).
2. **Introduce partial/batched claiming**: Allow `claim_rewards` to accept a `max_entries` parameter so a pool member can claim in multiple transactions, advancing `entry_to_claim_from` incrementally.
3. **Prune stale entries**: After a successful `claim_rewards`, entries before `entry_to_claim_from` are no longer needed. Consider compacting the trace to reclaim storage and bound future iteration.

---

### Proof of Concept

1. Pool member Alice joins a pool via `enter_delegation_pool` in epoch 1. Her trace has 1 entry.
2. In each subsequent epoch `e`, Alice calls `add_to_delegation_pool` with a minimal amount. Each call invokes `set_member_balance` → `trace.insert(key: e + K, ...)`. Since `e + K` differs each epoch, a new checkpoint is appended. After `N` epochs, Alice's trace has `N + 1` entries.
3. Alice never calls `claim_rewards`.
4. After enough epochs (say `N = 10,000`), Alice calls `claim_rewards`. This triggers `calculate_rewards`, which loops from `entry_to_claim_from = 0` to `pool_member_trace_length = 10,001`, performing a storage read and arithmetic per iteration. The transaction runs out of gas and reverts.
5. Every subsequent `claim_rewards` call also reverts. Alice's accumulated yield is permanently frozen.
