### Title
Unprivileged Caller Can Permanently Suppress Block Reward Distribution via `update_rewards(disable_rewards=true)` - (File: src/staking/staking.cairo)

### Summary
The `update_rewards` function in `StakingRewardsManagerImpl` is publicly callable by any non-zero address and accepts a caller-controlled `disable_rewards` boolean. When set to `true`, the function writes the current block to `last_reward_block` — consuming the one-call-per-block slot — but returns before distributing any rewards. An attacker can call this every block to permanently freeze all staker yield with no special privileges required.

### Finding Description
`update_rewards` (lines 1449–1507, `src/staking/staking.cairo`) is gated only by `general_prerequisites()`, which checks that the contract is unpaused and the caller is non-zero. No role or identity check exists.

The function accepts `disable_rewards: bool`. The critical ordering is:

```
// 1. Enforce one-call-per-block
assert!(current_block_number > self.last_reward_block.read(), ...);

// 2. Validate staker exists and is active
...

// 3. Commit last_reward_block — BEFORE the disable check
self.last_reward_block.write(current_block_number);   // line 1485

// 4. Conditionally skip reward distribution
if disable_rewards || self.is_pre_consensus() {
    return;                                            // lines 1487-1489
}
// 5. Distribute rewards (never reached when disable_rewards=true)
```

`last_reward_block` is written unconditionally at step 3. The `disable_rewards` guard at step 4 then silently discards the reward distribution
