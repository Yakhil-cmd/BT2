### Title
Unbounded loop in `calculate_rewards` can permanently freeze a pool member's unclaimed yield — (File: src/pool/pool.cairo)

---

### Summary

The `calculate_rewards` internal function in the `Pool` contract iterates over a pool member's entire balance-change history with no upper bound. A pool member who repeatedly delegates and undelegates across many epochs can grow their `pool_member_epoch_balance` trace to an arbitrarily large size. Once the trace is large enough, every call that invokes `calculate_rewards` will exhaust the block gas limit, permanently locking the pool member's accumulated, unclaimed yield.

---

### Finding Description

`calculate_rewards` in `src/pool/pool.cairo` (lines 837–888) contains a `while` loop that walks every entry in the pool member's balance trace from `entry_to_claim_from` up to `pool_member_trace_length`:

```cairo
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
    if pool_member_checkpoint.epoch() >= until_epoch {
        break;
    }
    ...
    entry_to_claim_from += 1;
}
``` [1](#0-0) 

The trace grows via `set_member_balance`, which inserts a new entry keyed to `epoch_plus_k` every time the pool member's balance changes:

```cairo
trace.insert(key: self.get_epoch_plus_k(), value: pool_member_balance);
``` [2](#0-1) 

Because each balance change in a distinct epoch appends a new entry, a pool member who repeatedly enters and exits the pool across many epochs accumulates an ever-growing trace. The codebase itself acknowledges the risk with the comment: *"This loop is unbounded but unlikely to exceed gas limits."* [3](#0-2) 

---

### Impact Explanation

Once the trace is large enough, every transaction that calls `calculate_rewards` for that pool member will consume all available gas and revert. Because reward accounting must traverse the full history before paying out, the pool member's accumulated unclaimed yield becomes permanently inaccessible — matching the allowed impact **"Permanent freezing of unclaimed yield."**

---

### Likelihood Explanation

**Low.** Growing the trace to a gas-exhausting size requires the pool member to perform a balance-changing action (delegate or undelegate) in a large number of distinct epochs. On Starknet, epochs are measured in blocks, so this requires sustained, deliberate activity over a long period. However, the entry path is fully permissionless and requires no privileged access — any pool member can execute it.

---

### Recommendation

1. **Cap the number of claimable epochs per call.** Allow `calculate_rewards` to process at most `N` balance-trace entries per invocation and store the updated `entry_to_claim_from` checkpoint so subsequent calls resume where the previous one left off.
2. **Alternatively, enforce a maximum trace length** by preventing a pool member from making more than one balance change per epoch (overwrite the pending entry rather than appending a new one when the target epoch is the same).

---

### Proof of Concept

1. Pool member `A` calls `enter_delegation_pool` in epoch 1 → trace length = 1.
2. `A` calls `exit_delegation_pool_intent` in epoch 2, waits for the exit window, calls `exit_delegation_pool_action` → trace length = 2.
3. `A` repeats steps 1–2 across epochs 3, 4, 5, … N, growing the trace to length `N`.
4. After `N` epochs (e.g., N = 10 000), `A` attempts to claim rewards. The call to `calculate_rewards` iterates over all `N` entries, exhausting the Starknet block gas limit and reverting.
5. Every subsequent claim attempt also reverts. `A`'s entire accumulated yield is permanently frozen.

The root cause is the unbounded `while` loop at `src/pool/pool.cairo:859`, fed by the append-only `set_member_balance` at `src/pool/pool.cairo:728`. [4](#0-3) [5](#0-4)
