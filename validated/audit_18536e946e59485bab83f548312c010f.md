### Title
Signature Bypass via `skip_stateful_validations`: Attacker Admits Unsigned Invoke for Any Deploying Account — (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator skips the `__validate__` entry-point call (the only on-chain signature check) for any Invoke transaction with nonce=1 whose sender address appears in `account_tx_in_pool_or_recent_block`. That mempool query returns `true` for **any** account that has **any** transaction in the pool — not specifically the account's own deploy-account. An unprivileged attacker who observes a victim's `deploy_account` in the mempool can immediately submit a crafted Invoke(nonce=1) for the victim's address with an arbitrary/wrong signature. The gateway admits it without calling `__validate__`. The batcher later executes it with `validate: true`, the signature check fails, the transaction reverts, and the victim's account is charged a revert fee.

---

### Finding Description

**Admission path** (`extract_state_nonce_and_run_validations`): [1](#0-0) 

The validator reads the on-chain nonce, runs pre-validation checks, then calls `run_validate_entry_point` with the `skip_validate` flag returned by `skip_stateful_validations`.

**The flawed skip predicate** (`skip_stateful_validations`): [2](#0-1) 

The function returns `true` (skip `__validate__`) when:
1. tx is `Invoke`
2. `tx.nonce() == Nonce(Felt::ONE)` (hardcoded)
3. `account_nonce == Nonce(Felt::ZERO)` (account not yet deployed)
4. `account_tx_in_pool_or_recent_block(sender_address)` returns `true`

**The mempool check** (`account_tx_in_pool_or_recent_block`): [3](#0-2) 

This returns `true` if the account has **any** transaction in the pool (`tx_pool.contains_account`) or is in the committed/staged state. It does **not** verify that the pooled transaction is a `deploy_account`, nor that the incoming Invoke's sender is the same party who submitted the deploy-account.

**Effect of skip on the gateway validator**: [4](#0-3) 

When `skip_validate = true`, `execution_flags.validate` is set to `false`, so `StatefulValidator::validate` returns immediately without calling the `__validate__` entry point — the only place where the account contract verifies the transaction signature.

**Batcher always re-validates with `validate: true`**: [5](#0-4) 

`AccountTransaction::new_for_sequencing` (used by the batcher) always sets `validate: true` and `strict_nonce_check: true`. The `InternalRpcTransaction` stored in the mempool carries no `execution_flags`; the batcher reconstructs them fresh. So the batcher **does** call `__validate__` for the attacker's transaction.

**Execution outcome** (`validate_tx`): [6](#0-5) 

With `validate: true`, the batcher calls the `__validate__` entry point. The attacker's wrong signature causes it to fail, the transaction reverts, and `perform_pre_validation_stage` has already incremented the nonce and confirmed the victim can pay the revert fee. [7](#0-6) 

---

### Impact Explanation

An attacker can inject a signature-invalid Invoke transaction for any account that is in the process of being deployed (deploy-account pending in mempool). The gateway admits it without any signature check. The batcher executes it, the `__validate__` entry point fails, the transaction reverts, and the victim's account is charged a revert fee proportional to the resource bounds the attacker specified (bounded by the victim's balance via `verify_can_pay_committed_bounds`). Additionally, the attacker's nonce=1 transaction occupies the slot, blocking or requiring fee-escalation for the victim's own legitimate nonce=1 invoke.

**Matching impact category:** *High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.*

---

### Likelihood Explanation

- The mempool is public; any observer can detect a pending `deploy_account` for address A.
- Submitting an Invoke(nonce=1, sender=A) with an arbitrary signature is a single RPC call.
- No privileged access, no special knowledge of the victim's private key, and no race against block finality is required — only a race against the victim's own nonce=1 submission.
- The attack is repeatable across every new account deployment visible in the mempool.

---

### Recommendation

Replace the coarse `account_tx_in_pool_or_recent_block` check with a query that specifically confirms a **deploy-account** transaction for the exact sender address is pending. Alternatively, expose a dedicated mempool query `deploy_account_in_pool(address) -> bool` and use it in `skip_stateful_validations`. A minimal patch:

```rust
// In skip_stateful_validations, replace:
return mempool_client
    .account_tx_in_pool_or_recent_block(tx.sender_address())
    .await ...;

// With:
return mempool_client
    .deploy_account_in_pool(tx.sender_address())  // new targeted query
    .await ...;
```

This ensures the skip only fires when the account's own deploy-account is genuinely pending, not when any third-party transaction happens to reference the same address.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
