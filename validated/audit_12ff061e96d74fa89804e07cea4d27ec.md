Audit Report

## Title
Silent ICP Burn Failure After Cycles Minting Breaks Conservation Invariant — (File: rs/nns/cmc/src/main.rs)

## Summary
The `burn_and_log` function in the Cycles Minting Canister unconditionally swallows all errors from the ICP ledger burn call, returning `()` regardless of outcome. Because cycles are deposited and the notification is finalized as `Ok` before (and independent of) the burn result, any transient ICP ledger failure permanently leaves ICP unburned in the CMC subaccount while the corresponding cycles remain irrevocably minted, breaking the ICP/cycles conservation invariant with no recovery path.

## Finding Description
In `rs/nns/cmc/src/main.rs`, three async process functions call `burn_and_log` after a successful cycles operation:

- `process_create_canister` (line 1945): after `do_create_canister` succeeds
- `process_mint_cycles` (line 1968): after `do_mint_cycles` succeeds
- `process_top_up` (line 2001): after `deposit_cycles` succeeds

All three functions return `Ok(...)` unconditionally after `burn_and_log` returns, because `burn_and_log` has return type `()` and its `Err` arm only logs (lines 2044–2047). The callers (`notify_top_up`, `notify_mint_cycles`, `notify_create_canister`) then store the result in `blocks_notified` as `NotifiedTopUp(Ok(...))` / `NotifiedMint(Ok(...))` / `NotifiedCreateCanister(Ok(...))` (lines 1215–1218, 1306–1309, 1412–1415). The `is_transient_error` guard (lines 1219–1221, 1310–1312, 1416–1418) only removes the entry on transient errors, but since the process functions return `Ok`, the entry is never removed. The notification is permanently finalized. Any subsequent call by the user returns the cached `Ok` result immediately (lines 1209–1210, 1299–1300, 1405–1406), making retry impossible.

Root cause: `burn_and_log` is intentionally designed to never propagate errors (comment at lines 2015–2016: "Burning doesn't return errors - we don't want to reject the transaction notification because then it could be retried"), but this design choice creates a window where cycles are minted without ICP being burned, with no compensating mechanism.

## Impact Explanation
This constitutes illegal minting of cycles: cycles are created and deposited to the target without the corresponding ICP being destroyed. The ICP/cycles conservation invariant — every cycle minted corresponds to an equal-value ICP burn — is silently violated. The stranded ICP in the CMC subaccount is effectively locked with no user-accessible or operator-accessible recovery path. This matches the Critical/High allowed impact: "Theft, permanent loss, illegal minting, or protocol insolvency involving ICP/Cycles." Severity is High given that exploitation requires a transient environmental condition (ledger unavailability), bounding the realistic per-event impact, but the invariant break is permanent and unrecoverable per affected transaction.

## Likelihood Explanation
Any unprivileged user can trigger this without special knowledge or privileges. The ICP ledger (`ryjl3-tyaaa-aaaaa-aaaba-cai`) is periodically upgraded, and during an upgrade window inter-canister calls to it return a transient error. A user who calls `notify_top_up`, `notify_mint_cycles`, or `notify_create_canister` during such a window will trigger the path. No attacker action is required beyond normal usage; the race is purely environmental. The window is narrow but recurring (every ledger upgrade), and the user need not be aware of it — any notification processed during the window is permanently affected.

## Recommendation
1. **Burn before finalizing the notification.** Reorder the sequence in each process function: attempt and confirm the ICP burn before writing `NotifiedTopUp(Ok(...))` / `NotifiedMint(Ok(...))` / `NotifiedCreateCanister(Ok(...))`. If the burn fails, leave the notification as `Processing` (or remove it) so the user can retry.
2. **Track failed burns in persistent state.** Add a `failed_burns: Vec<(Subaccount, Tokens)>` field to `State`. On burn failure in `burn_and_log`, push the entry instead of only logging, and drain it via a heartbeat/timer that retries `send_pb` until success.
3. **At minimum**, expose a metric counter for failed burns so operators can detect and manually intervene.

## Proof of Concept
1. User sends N ICP to the CMC subaccount for `canister_id` with `MEMO_TOP_UP_CANISTER`.
2. User calls `notify_top_up { block_index, canister_id }`.
3. CMC fetches the transaction, sets `blocks_notified[block_index] = Processing`, calls `process_top_up`.
4. `deposit_cycles` succeeds — N cycles are deposited to `canister_id` (irreversible).
5. `burn_and_log(sub, amount)` is awaited; the ICP ledger is mid-upgrade and returns `Err(...)`.
6. `burn_and_log` logs the error and returns `()`.
7. `process_top_up` returns `Ok(cycles)`.
8. CMC writes `blocks_notified[block_index] = NotifiedTopUp(Ok(cycles))` — permanently finalized.
9. `notify_top_up` returns `Ok(cycles)` to the user.

**Result:** N cycles are deposited. N ICP remain in the CMC subaccount, unburned. The conservation invariant is broken. The notification cannot be retried (it is already `NotifiedTopUp(Ok(...))`). A local PocketIC integration test can reproduce this deterministically by intercepting the `send_pb` call to the ledger canister and returning a reject code after `deposit_cycles` succeeds, then asserting that the CMC subaccount balance is non-zero and the notification status is `NotifiedTopUp(Ok(...))`.
