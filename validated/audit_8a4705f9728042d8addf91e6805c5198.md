### Title
Unauthorized Global Contract Overwrite via Cross-Contract Call in `AccountId` Deploy Mode â€” (File: `runtime/runtime/src/global_contracts.rs`)

---

### Summary

`action_deploy_global_contract` does not verify that `predecessor_id == receiver_id` when `deploy_mode == GlobalContractDeployMode::AccountId`. Any unprivileged contract can create a cross-contract promise targeting `victim.near` and append a `DeployGlobalContract(AccountId)` action, overwriting the global contract stored under `victim.near`'s account ID on every shard. Every account that has opted into `GlobalContractIdentifier::AccountId("victim.near")` will subsequently execute the attacker-supplied Wasm.

---

### Finding Description

**Root cause â€” missing authorization check in `action_deploy_global_contract`:**

In `runtime/runtime/src/global_contracts.rs`, `action_deploy_global_contract` receives `account_id` (the receipt's `receiver_id`) and, when `deploy_mode == AccountId`, passes it directly to `initiate_distribution` as the global contract identifier: [1](#0-0) 

`initiate_distribution` then writes the attacker-supplied code to `TrieKey::GlobalContractCode { identifier: GlobalContractCodeIdentifier::AccountId(account_id) }` on the current shard and enqueues `GlobalContractDistributionReceipt` to propagate it to every other shard: [2](#0-1) 

There is no check anywhere in this path that `predecessor_id == receiver_id`.

**No guard in validation:**

`validate_deploy_global_contract_action` only checks contract byte size: [3](#0-2) 

**No guard in the action execution loop:**

The action dispatch in `lib.rs` calls `action_deploy_global_contract` directly with no predecessor/receiver equality check: [4](#0-3) 

**Protocol specification confirms the omission:**

The protocol documentation explicitly lists the actions that require `predecessor_id == receiver_id`: `DeployContract`, `Stake`, `AddKey`, `DeleteKey`, `DeleteAccount`. `DeployGlobalContract` is absent: [5](#0-4) 

**Attacker-controlled entry point:**

The host function `promise_batch_action_deploy_global_contract_by_account_id` is available to any Wasm contract: [6](#0-5) 

Any unprivileged user can deploy a contract and invoke this host function to target an arbitrary `receiver_id`.

**Design intent vs. enforcement gap:**

The `AccountId` deploy mode is documented as "deployed under the owner account id â€¦ allows the owner to update the contract for all its users": [7](#0-6) 

The word "owner" implies exclusivity, but the runtime enforces no such ownership constraint.

---

### Impact Explanation

- **Corrupted value**: `TrieKey::GlobalContractCode { identifier: GlobalContractCodeIdentifier::AccountId(victim) }` is overwritten with attacker-controlled Wasm on every shard via the `GlobalContractDistributionReceipt` propagation chain.
- **Affected accounts**: Every account that has called `UseGlobalContract(AccountId(victim))` â€” storing `AccountContract::GlobalByAccount(victim)` â€” will execute the attacker's code on any subsequent function call.
- **Severity**: Critical. The attacker achieves arbitrary code execution in the context of all victim accounts that reference the hijacked global contract identifier, enabling theft of funds, storage corruption, or denial of service across an unbounded set of accounts.
- **Storage cost**: Deducted from `victim.near`'s balance (not the attacker's), so the attacker bears no financial cost beyond gas for the initiating transaction.

---

### Likelihood Explanation

- Any unprivileged user can deploy a contract and exploit this with a single cross-contract call.
- The only precondition is that the target account holds enough balance to cover the storage cost of the malicious Wasm (`global_contract_storage_amount_per_byte * code_len`). Accounts that have already deployed a global contract in `AccountId` mode necessarily hold this balance.
- No validator, operator, or admin privilege is required.

---

### Recommendation

Add a `predecessor_id == receiver_id` guard inside `action_deploy_global_contract` (or in the action dispatch in `lib.rs`) specifically for `GlobalContractDeployMode::AccountId`, mirroring the existing restriction on `DeployContract`. Concretely, before calling `initiate_distribution`, verify:

```rust
if deploy_contract.deploy_mode == GlobalContractDeployMode::AccountId
    && predecessor_id != receiver_id
{
    result.result = Err(ActionErrorKind::ActorNoPermission {
        account_id: receiver_id.clone(),
        actor_id: predecessor_id.clone(),
    }.into());
    return Ok(());
}
```

Also update the protocol specification in `docs/RuntimeSpec/Actions.md` to add `DeployGlobalContract` (in `AccountId` mode) to the list of actions requiring `predecessor_id == receiver_id`.

---

### Proof of Concept

1. Attacker deploys a malicious contract at `attacker.near`.
2. A user calls any function on `attacker.near`; the contract executes:
   ```
   let p = promise_batch_create("victim.near");
   promise_batch_action_deploy_global_contract_by_account_id(p, malicious_wasm);
   ```
3. The resulting `ActionReceipt` targeting `victim.near` is executed. `action_deploy_global_contract` is called with `account_id = "victim.near"` and `deploy_mode = AccountId`. No authorization check fires.
4. `initiate_distribution` stores the malicious Wasm under `TrieKey::GlobalContractCode { identifier: GlobalContractCodeIdentifier::AccountId("victim.near") }` on the current shard and emits `GlobalContractDistributionReceipt` to propagate it to all other shards.
5. Every account that previously called `UseGlobalContract(AccountId("victim.near"))` now has `AccountContract::GlobalByAccount("victim.near")` pointing to the malicious Wasm. On the next function call to any such account, the attacker's code executes with full access to that account's state and balance.

fn initiate_distribution(
    state_update: &mut TrieUpdate,
    account_id: AccountId,
    contract_code: Arc<[u8]>,
    deploy_mode: &GlobalContractDeployMode,
    current_shard_id: ShardId,
    result: &mut ActionResult,
) -> Result<(), RuntimeError> {
    let id = match deploy_mode {
        GlobalContractDeployMode::CodeHash => {
            GlobalContractIdentifier::CodeHash(hash(&contract_code))
        }
        GlobalContractDeployMode::AccountId => {
            GlobalContractIdentifier::AccountId(account_id.clone())
        }
    };
    // Increment the nonce and write it to state immediately to prevent multiple
    // distributions with the same nonce from being initiated. This requires
    // allowing the same nonce in the freshness check when applying the
    // distribution receipt.
    let nonce = increment_nonce(state_update, &id)?;
    let distribution_receipt =
        GlobalContractDistributionReceipt::new(id, current_shard_id, vec![], contract_code, nonce);
    let distribution_receipts =
        Receipt::new_global_contract_distribution(account_id, distribution_receipt);
    // No need to set receipt_id here, it will be generated as part of apply_action_receipt
    result.new_receipts.push(distribution_receipts);
    Ok(())
