### Title
Sub-minimum EVM Balance in `CadenceOwnedAccount` Permanently Inaccessible Due to UFix64 Precision Floor — (File: `fvm/evm/impl/impl.go`)

---

### Summary

When a `CadenceOwnedAccount` (COA) accumulates an EVM balance below `1e10` attoFlow — the minimum representable unit in a Cadence `UFix64` Flow Vault — the `withdraw()` function permanently rejects all withdrawal attempts. This state is reachable through normal EVM operations such as gas fee deductions. If the COA resource is subsequently destroyed, the stranded FLOW tokens are permanently lost with no recovery path.

---

### Finding Description

`CadenceOwnedAccount.withdraw()` in `fvm/evm/stdlib/contract.cdc` delegates to `InternalEVM.withdraw()`, implemented in `fvm/evm/impl/impl.go` via `newInternalEVMTypeWithdrawFunction`. Before executing the withdrawal, the implementation enforces:

```go
// fvm/evm/impl/impl.go:779-780
if !types.AttoFlowBalanceIsValidForFlowVault(amountValue.BigInt) {
    panic(types.ErrWithdrawBalanceRounding)
}
```

`AttoFlowBalanceIsValidForFlowVault` is defined as:

```go
// fvm/evm/types/balance.go:105-107
func AttoFlowBalanceIsValidForFlowVault(bal *big.Int) bool {
    return bal.Cmp(UFixToAttoConversionMultiplier) >= 0
}
```

where `UFixToAttoConversionMultiplier = 10^10 = 10,000,000,000` attoFlow. Any requested withdrawal amount below this threshold panics with `ErrWithdrawBalanceRounding`.

The same guard exists in the emulator layer:

```go
// fvm/evm/emulator/emulator.go:441-447
if !types.AttoFlowBalanceIsValidForFlowVault(call.Value) {
    return types.NewInvalidResult(
        call.Type,
        call.Hash(),
        types.ErrWithdrawBalanceRounding,
    ), nil
}
```

The `CadenceOwnedAccount.withdraw()` Cadence function only bypasses the check for a zero balance (returning an empty vault), but does not handle the case where the total balance is non-zero yet below the minimum:

```cadence
// fvm/evm/stdlib/contract.cdc:592-594
if balance.isZero() {
    return <-FlowToken.createEmptyVault(vaultType: Type<@FlowToken.Vault>())
}
```

When a COA's total EVM balance is between `1` and `9,999,999,999` attoFlow, no valid withdrawal amount exists that simultaneously satisfies:
1. Non-zero (to avoid the zero-balance early return path)
2. `>= 1e10` attoFlow (to pass `AttoFlowBalanceIsValidForFlowVault`)
3. `<= actual balance` (to avoid EVM insufficient-funds failure)

**How a COA reaches a sub-minimum balance through normal operations:**

- **EVM gas deduction**: A COA deposits exactly `1e10` attoFlow (the documented minimum), then sends an EVM transaction. Gas is deducted from the COA's EVM balance. Any non-zero gas cost leaves a remainder below `1e10` attoFlow.
- **Sub-minimum EVM-to-EVM transfer**: Any EVM address can send an amount between `1` and `9,999,999,999` attoFlow to a COA's EVM address.
- **Partial withdrawal remainder**: A user withdraws most of their balance, leaving a sub-minimum remainder (e.g., balance of `1.5e10` attoFlow, user withdraws `1e10`, leaving `0.5e10`).

---

### Impact Explanation

FLOW tokens held in a COA's EVM balance that fall below `1e10` attoFlow (`0.00000001 FLOW`) cannot be withdrawn to Cadence. The user is locked out of their own funds. If the COA Cadence resource is subsequently destroyed (a valid Cadence operation), the EVM-side balance remains in the EVM state permanently with no controlling resource — constituting irreversible cross-VM asset loss. The COA address is deterministically derived from the resource UUID and cannot be re-created, so no new COA can ever control that EVM address again.

---

### Likelihood Explanation

Any COA that executes EVM transactions (paying gas fees) or receives sub-minimum EVM transfers can reach this state through ordinary, unprivileged protocol usage. No special privileges, admin keys, or compromised nodes are required. The entry path is a standard Cadence transaction calling `coa.withdraw(balance: bal)` where `bal.attoflow < 1e10`. This is reachable by any COA owner.

---

### Recommendation

Add a `withdrawAll()` or `sweep()` function on `CadenceOwnedAccount` that withdraws the entire EVM balance, rounding down to the nearest representable `UFix64` unit and discarding the sub-minimum remainder (burning it or crediting it to a protocol-controlled address). Alternatively, modify `withdraw()` to accept a special sentinel value meaning "withdraw all available balance" and apply the floor-rounding internally rather than rejecting the call. At minimum, add a pre-condition check in any `destroy` path for `CadenceOwnedAccount` that asserts the EVM balance is zero before destruction, preventing silent permanent loss.

---

### Proof of Concept

```cadence
import EVM from <EVMAddress>
import FlowToken from <FlowTokenAddress>

transaction() {
    prepare(account: auth(BorrowValue, Storage) &Account) {
        // Step 1: Create COA and deposit exactly the minimum (1e10 attoFlow = 0.00000001 FLOW)
        let coa <- EVM.createCadenceOwnedAccount()
        let admin = account.storage.borrow<&FlowToken.Administrator>(from: /storage/flowTokenAdmin)!
        let minter <- admin.createNewMinter(allowedAmount: 0.00000001)
        let vault <- minter.mintTokens(amount: 0.00000001)
        destroy minter
        coa.deposit(from: <-vault)
        // COA EVM balance = 1e10 attoFlow exactly

        // Step 2: Send an EVM transaction from the COA (any gas-consuming call)
        // Gas is deducted from the COA's EVM balance.
        // Remaining balance = 1e10 - gas_cost < 1e10 attoFlow

        // Step 3: Attempt to withdraw the remaining sub-minimum balance
        // This will panic with ErrWithdrawBalanceRounding:
        let remainingBal = coa.balance()  // e.g., Balance(attoflow: 5_000_000_000)
        let recovered <- coa.withdraw(balance: remainingBal)
        // ^ PANICS: "withdraw failed! smallest unit allowed to transfer is 1e10 attoFlow"

        // Step 4: Attempt to withdraw 0 — returns empty vault, no funds recovered
        let zeroBal = EVM.Balance(attoflow: 0)
        let empty <- coa.withdraw(balance: zeroBal)
        // Returns empty vault, sub-minimum balance still stranded

        // Step 5: Destroy the COA — EVM balance permanently lost
        destroy coa
        destroy recovered
        destroy empty
    }
}
```

The root cause is at [1](#0-0)  where `AttoFlowBalanceIsValidForFlowVault` rejects any requested amount below `1e10` attoFlow, and at [2](#0-1)  where the threshold is defined as `UFixToAttoConversionMultiplier = 10^10`. The `CadenceOwnedAccount.withdraw()` entry point at [3](#0-2)  only bypasses the check for a zero balance, leaving sub-minimum non-zero balances with no valid withdrawal path. The same guard is enforced at the emulator layer at [4](#0-3) .
