### Title
Incorrect Multi-NFT Royalty Accounting in WalletConnect Offer Confirmation Display - (File: packages/gui/src/electron/commands/parseCommandDisplay.ts)

### Summary
The `formatAmountWithRoyalties` function in `parseCommandDisplay.ts` uses an incorrect "equal-split" scheme when computing the displayed total-with-royalties for a `take_offer` or `create_offer_for_ids` WalletConnect command that involves multiple NFTs. Instead of applying each NFT's royalty percentage to the full fungible amount, it first divides the amount equally among all NFTs and then applies each percentage to that fractional share. The result is a systematically understated `amountWithRoyalties` shown in the WalletConnect confirmation dialog, causing users to approve transactions believing they will spend less than they actually will.

### Finding Description

`formatAmountWithRoyalties` is called from `withRoyaltyTotals` to produce the `amountWithRoyalties` field that the Electron main process renders in the WalletConnect signing confirmation dialog. [1](#0-0) 

The faulty logic:

```typescript
const splitAmount = amount / BigInt(royaltyPercentages.length);   // ← divides first
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
```

For a single NFT the result is correct (splitting by 1 is a no-op). For two or more NFTs the displayed royalty is:

```
displayed_royalty = Σ( (amount / N) * pct_i / 10000 )
                 = (amount / N) * Σ(pct_i) / 10000
```

whereas the correct royalty that the blockchain will actually enforce is:

```
correct_royalty  = Σ( amount * pct_i / 10000 )
                 = amount * Σ(pct_i) / 10000
```

The displayed value is exactly `1/N` of the correct value (where N = number of NFTs). With two NFTs the user sees half the true royalty cost; with three NFTs, one-third; and so on.

**Concrete example** (matching the existing test fixture):
- Spending: 100,000,000 mojos (0.0001 XCH)
- NFT 1 royalty: 500 bp (5 %)
- NFT 2 royalty: 10 bp (0.1 %)

| | Displayed | Correct |
|---|---|---|
| NFT 1 royalty | 2,500,000 mojos | 5,000,000 mojos |
| NFT 2 royalty | 50,000 mojos | 100,000 mojos |
| Total royalties | 2,550,000 mojos | 5,100,000 mojos |
| `amountWithRoyalties` | **0.00010255 XCH** | **0.0001051 XCH** |

The existing test at line 251 of `parseCommandDisplay.test.ts` asserts the *wrong* value `'0.00010255'`, confirming the bug is baked in and untested against the correct figure. [2](#0-1) 

`parseCommandDisplay` is invoked from the Electron main process to populate the confirmation dialog shown to the user before a WalletConnect command is executed. [3](#0-2) 

### Impact Explanation

A WalletConnect-connected dApp can craft a `take_offer` payload containing multiple NFTs with high royalty percentages. The confirmation dialog will display a materially understated `amountWithRoyalties`. The user, seeing a lower-than-actual cost, approves the transaction. The blockchain then enforces the full (correct) royalties, draining more XCH or CAT from the user's wallet than the confirmation dialog indicated.

This satisfies the **High** impact criterion: *"Corruption, spoofing, or unsafe trust of … WalletConnect state that causes a user to approve … the wrong … amount."*

### Likelihood Explanation

- Any WalletConnect-paired dApp can trigger this path by issuing a `take_offer` command referencing two or more NFTs with non-zero royalties — a normal, permitted operation.
- The discrepancy grows with the number of NFTs and the magnitude of royalty percentages. With high-royalty NFTs (e.g., 10 %+) and three or more NFTs, the understated amount can be several times smaller than the actual cost.
- No special privileges, leaked keys, or social engineering are required beyond having an established WalletConnect pairing.

### Recommendation

Remove the equal-split and apply each royalty percentage to the full `amount`:

```typescript
function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],
): string | undefined {
  if (royaltyPercentages.length === 0 || line.kind === 'nft') {
    return undefined;
  }

  // Apply each NFT's royalty to the full amount, not a split share.
  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (amount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;
  ...
}
```

Update the corresponding test assertions in `parseCommandDisplay.test.ts` to reflect the corrected values.

### Proof of Concept

Using the values from the existing test (two NFTs, royalties 500 bp and 10 bp, amount 100,000,000 mojos):

**Current (buggy) output shown in WalletConnect dialog:**
```
amountWithRoyalties: "0.00010255"   // splitAmount=50M; 50M*500/10000 + 50M*10/10000 = 2.55M extra
```

**Correct output that should be shown:**
```
amountWithRoyalties: "0.0001051"    // 100M*500/10000 + 100M*10/10000 = 5.1M extra
```

A malicious dApp offering three NFTs each with a 30 % royalty (3000 bp) against a 1 XCH payment would display `1.3 XCH` total while the actual on-chain cost is `1.9 XCH` — a 46 % understatement — causing the user to approve a transaction they would otherwise reject. [4](#0-3)

function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],
): string | undefined {
  if (royaltyPercentages.length === 0 || line.kind === 'nft') {
    return undefined;
  }

  const splitAmount = amount / BigInt(royaltyPercentages.length);
  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;

  if (line.kind === 'xch') {
    return mojoToChiaLocaleString(totalAmount);
  }

  return mojoToCATLocaleString(totalAmount);
}
