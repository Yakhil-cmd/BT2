### Title
Incorrect Multi-NFT Royalty Amount Displayed in WalletConnect Signing Confirmation — (`File: packages/gui/src/electron/commands/parseCommandDisplay.ts`)

### Summary
`formatAmountWithRoyalties` in `parseCommandDisplay.ts` incorrectly divides the fungible payment amount by the number of NFTs before applying each royalty percentage. When a `take_offer` or `create_offer_for_ids` command involves multiple royalty-bearing NFTs, the displayed `amountWithRoyalties` shown in the WalletConnect signing confirmation dialog is understated by a factor of N (the number of NFTs). The user approves a transaction believing they will spend less XCH/CAT than the blockchain will actually deduct.

### Finding Description

`formatAmountWithRoyalties` computes the royalty-inclusive total as follows:

```typescript
const splitAmount = amount / BigInt(royaltyPercentages.length);   // divides by N
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
const totalAmount = amount + royaltyAmount;
``` [1](#0-0) 

The correct formula is to apply each NFT's royalty percentage to the **full** fungible amount:

```
correctRoyalty = amount × (r₁ + r₂ + … + rN) / 10000
```

The actual code computes:

```
actualRoyalty = (amount / N) × (r₁ + r₂ + … + rN) / 10000
```

This is exactly `1/N` of the correct value. The on-chain Chia royalty mechanism applies each NFT's royalty to the full purchase price, so the actual deduction from the user's wallet is `N` times larger than what the confirmation dialog shows.

`formatAmountWithRoyalties` is called by `withRoyaltyTotals`, which is called inside `walletDeltaToDisplay`, which is the function that produces the `DisplayWalletDelta` shown in the WalletConnect signing confirmation dialog for both `chia_wallet.take_offer` and `chia_wallet.create_offer_for_ids`. [2](#0-1) [3](#0-2) 

### Impact Explanation

The `amountWithRoyalties` field is the only place in the WalletConnect confirmation dialog where the user sees the true cost of accepting an offer that includes royalty-bearing NFTs. When the offer contains N ≥ 2 such NFTs, the displayed total is understated by factor N. The user clicks "Approve" believing they are spending the displayed amount; the actual blockchain transaction deducts the correct (larger) amount. This constitutes causing a user to approve the wrong amount — a direct balance/accounting impact on XCH or CAT.

**Concrete example** (matching the existing test at line 251–335):
- Payment: 100,000,000 mojos (0.0001 XCH)
- NFT 1 royalty: 5% (500 basis points)
- NFT 2 royalty: 0.1% (10 basis points)
- **Displayed** `amountWithRoyalties`: 0.00010255 XCH
- **Correct** `amountWithRoyalties`: 0.001051 XCH (≈10× larger) [4](#0-3) 

An attacker who crafts an offer with many high-royalty NFTs maximizes the gap between the displayed and actual cost.

### Likelihood Explanation

Any unprivileged party can create a valid Chia offer containing multiple royalty-bearing NFTs and share it via WalletConnect. The victim only needs to open the offer in the GUI and reach the WalletConnect approval dialog. No special access, leaked keys, or social engineering beyond presenting the offer is required. The condition (≥2 NFTs with royalties in one offer) is a normal, supported use case.

### Recommendation

Remove the `splitAmount` division. Apply each royalty percentage to the full `amount`:

```typescript
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (amount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
``` [5](#0-4) 

Update the corresponding test expectation at line 320 (`amountWithRoyalties: '0.00010255'`) to the correct value after the fix.

### Proof of Concept

Using the existing test scaffold in `parseCommandDisplay.test.ts` (lines 251–335):

- Offer: 2 NFTs offered, 100,000,000 mojos XCH requested
- NFT 1 royalty: 500 bp (5%), NFT 2 royalty: 10 bp (0.1%)
- Current code produces `amountWithRoyalties = '0.00010255'`
  - `splitAmount = 100_000_000 / 2 = 50_000_000`
  - royalty₁ = `50_000_000 × 500 / 10_000 = 2_500_000`
  - royalty₂ = `50_000_000 × 10 / 10_000 = 50_000`
  - total = `100_000_000 + 2_550_000 = 102_550_000` → `0.00010255 XCH`
- Correct value:
  - royalty₁ = `100_000_000 × 500 / 10_000 = 5_000_000`
  - royalty₂ = `100_000_000 × 10 / 10_000 = 100_000`
  - total = `100_000_000 + 5_100_000 = 105_100_000` → `0.001051 XCH`

The confirmation dialog understates the cost by ~10×. The user approves; the blockchain deducts the correct larger amount. [1](#0-0) [6](#0-5)

function formatAmountWithRoyalties(
  line: DisplayWalletDeltaItem,
  amount: bigint,
  royaltyPercentages: number[],
): string | undefined {
  if (royaltyPercentages.length === 0 || line.kind === 'nft') {
    return undefined;
  }

  const splitAmount = amount / BigInt(royaltyPercentages.length);
  const royaltyAmount = royaltyPercentages.reduce(
    (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
    0n,
  );
  const totalAmount = amount + royaltyAmount;

  if (line.kind === 'xch') {
    return mojoToChiaLocaleString(totalAmount);
  }

  return mojoToCATLocaleString(totalAmount);
}

function withRoyaltyTotals(
  items: DisplayWalletDeltaItemWithKey[],
  amounts: Record<string, bigint>,
  oppositeSideLines: DisplayWalletDeltaItem[],
): DisplayWalletDeltaItem[] {
  const royaltyPercentages = royaltyPercentagesForSide(oppositeSideLines);

  return items.map(({ key, line }) => {
    if (line.kind === 'nft') {
      return line;
    }

    const amount = amounts[key];
    const amountWithRoyalties = amount ? formatAmountWithRoyalties(line, amount, royaltyPercentages) : undefined;

    return amountWithRoyalties ? { ...line, amountWithRoyalties } : line;
  });
}
