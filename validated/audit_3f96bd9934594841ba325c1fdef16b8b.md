Looking at the `skip_stateful_validations` logic and the `account_tx_in_pool_or_recent_block` check, I need to trace the exact condition under which gateway-level signature validation is skipped and whether the guard is tight enough.

### Title
Overly Broad `skip_stateful_validations` Guard Admits Unsigned Invoke Transactions to Mempool — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

---

### Summary

`skip_stateful_validations` skips the `__validate__` entry-point (signature check) for any invoke with `nonce=1` whenever `account_tx_in_pool_or_recent_block` returns `true`. That helper returns `true` for **any** transaction type in the pool, not only `deploy_account`. An unprivileged attacker who observes a pending `nonce=0` regular-invoke for an already-deployed account can submit a forged `nonce=1` invoke with an arbitrary/empty signature, bypass gateway signature validation, and land the transaction in the mempool.

---

### Finding Description

`skip_stateful_validations` fires when three conditions hold simultaneously:

1. The incoming transaction is `ExecutableTransaction::Invoke`.
2. `tx.nonce() == Nonce(Felt::ONE)`.
3. `account_nonce == Nonce(Felt::ZERO)` (blockchain state).
4. `mempool_client.account_tx_in_pool_or_recent_block(sender)` returns `true`. [1](#0-0) 

When all four hold, `skip_validate = true` is returned, and `run_validate_entry_point` sets `validate: !skip_validate = false`, so the blockifier's `StatefulValidator::perform_validations` returns `Ok(())` without ever calling `__validate__`. [2](#0-1) 

The code comment justifies condition 4 as: *"either it has a deploy_account transaction or transactions with future nonces that passed validations."* However, `account_tx_in_pool_or_recent_block` is implemented as:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [3](#0-2) 

`tx_pool.contains_account` returns `true` for **any** transaction type at **any** nonce, and `state.contains_account` returns `true` for any address in `staged` or `committed`: [4](#0-3) 

The comment's reasoning breaks for the following case: account X is already deployed (blockchain nonce = 0), and has a valid regular `invoke` with `nonce=0` pending in the pool. That transaction proves the owner controls `nonce=0`, **not** `nonce=1`. Yet condition 4 fires, and the gateway skips signature validation for any attacker-supplied `nonce=1` invoke targeting X.

---

### Impact Explanation

The gateway admits an unsigned (or arbitrarily signed) `invoke` with `nonce=1` into the mempool without running the account's `__validate__` entry point. The batcher later executes the transaction with `validate: true` (set by `AccountTransaction::new_for_sequencing`), so `__validate__` is called at execution time and the transaction reverts. However:

* The mempool invariant "only signature-validated transactions are admitted" is broken.
* The account's `nonce=1` slot is permanently consumed by the reverted transaction; the legitimate owner's `nonce=1` transaction is rejected as a duplicate nonce (or must pay a fee-escalation premium).
* The attack is repeatable: every time the victim submits a new `nonce=N` transaction, the attacker can immediately poison `nonce=N+1` (once the victim's `nonce=N` is in the pool and the blockchain nonce is still `N`).

This matches **High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing**.

---

### Likelihood Explanation

* No privileges required — any caller of the public gateway `add_tx` endpoint can trigger this.
* The attacker only needs to observe the public mempool to learn which accounts have a pending `nonce=0` transaction.
* The attack window is the time between a `nonce=0` transaction entering the mempool and it being committed to a block (potentially many seconds to minutes).

---

### Recommendation

Replace the coarse `account_tx_in_pool_or_recent_block` check with a type-aware check that returns `true` only when the account has a **`deploy_account`** transaction pending (or staged/committed from a deploy). One approach: maintain a dedicated `pending_deploy_accounts` set in the mempool that is populated only when a `deploy_account` transaction is added and cleared when it is committed or evicted. `skip_stateful_validations` should query that set instead of the generic `account_tx_in_pool_or_recent_block`.

---

### Proof of Concept

```
1. Account X is deployed; blockchain nonce = 0.
2. Owner submits valid invoke(nonce=0) → admitted to mempool after full validation.
3. Attacker submits invoke(nonce=1, signature=[0x0]) for X to the gateway.
4. Gateway: nonce=1, account_nonce=0 → skip condition fires.
5. account_tx_in_pool_or_recent_block(X) = true  (X has nonce=0 tx in pool).
6. run_validate_entry_point called with validate=false → __validate__ NOT called.
7. Attacker's forged tx is admitted to the mempool.
8. Batcher executes nonce=0 (succeeds), then nonce=1 (validate=true → __validate__ fails → reverts).
9. Account nonce advances to 2; nonce=1 slot is permanently wasted.
10. Owner's legitimate nonce=1 tx is rejected as DuplicateNonce.
``` [5](#0-4) [6](#0-5)

        let account_tx = AccountTransaction { tx: executable_tx.clone(), execution_flags };

        // Build block context.
        let mut versioned_constants = VersionedConstants::get_versioned_constants(

        // The validation of a transaction is not affected by the casm hash migration.
        versioned_constants.disable_casm_hash_migration();

        let mut block_info = self.gateway_fixed_block_state_reader.get_block_info().await?;
        block_info.block_number = block_info.block_number.unchecked_next();

            block_info,
            self.chain_info.clone(),
            versioned_constants,
            BouncerConfig::max(),

            cur_span.in_scope(|| {

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
