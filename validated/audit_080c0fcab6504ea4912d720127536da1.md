### Title
Malicious CKD Participant Can Send Arbitrary Group Elements to Corrupt the Coordinator's Derived Key Output - (File: src/confidential_key_derivation/protocol.rs)

### Summary
The `do_ckd_coordinator` function aggregates participant contributions `(big_y, big_c)` without any cryptographic verification of their correctness. A malicious participant can send arbitrary group elements, causing the coordinator to produce a corrupted CKD output that honest parties accept as valid — directly analogous to the external report's pattern of an unvalidated attacker-controlled input being used without a whitelist or proof check.

### Finding Description

In `do_ckd_coordinator`, the coordinator receives each participant's `CKDOutput` and blindly adds the `big_y` and `big_c` fields: [1](#0-0) 

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
```

The correct values each honest participant `i` should send are computed in `compute_signature_share`: [2](#0-1) 

- `norm_big_y_i = λ_i · y_i · G`
- `norm_big_c_i = λ_i · (x_i · H(pk ‖ app_id) + y_i · app_pk)`

There is **no proof of knowledge** and **no consistency check** verifying that:
1. `big_y` is of the form `λ_i · y_i · G` for some `y_i`
2. `big_c` uses the **same** `y_i` as `big_y`
3. `big_c` encodes the participant's actual private share `x_i`

The coordinator simply sums whatever it receives.

### Impact Explanation

The final CKD output `(Y, C)` is used by the application to recover the confidential key:

```
C − sk_app · Y  =  msk · H(pk ‖ app_id)
```

If a malicious participant sends `(big_y_malicious, big_c_malicious)` instead of their correct contribution, the coordinator computes:

```
Y′ = Y_correct + Δy
C′ = C_correct + Δc
```

The application then decrypts:

```
C′ − sk_app · Y′  =  msk · H(pk ‖ app_id) + Δc − sk_app · Δy
```

Unless `Δc = sk_app · Δy` (which the attacker cannot arrange without knowing `sk_app`), the derived confidential key is **wrong and unusable**. The coordinator has no mechanism to detect the manipulation and returns the corrupted output as a successful result.

**Impact category matched:** *High — Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.*

### Likelihood Explanation

Any single participant in the CKD protocol can trigger this. The attacker needs only to deviate from the protocol by sending arbitrary `ElementG1` values instead of their correct share. No cryptographic break, no key leakage, and no external dependency is required. The attack is one-shot and undetectable by the coordinator.

### Recommendation

Each participant must accompany their `(norm_big_y, norm_big_c)` with a **Chaum-Pedersen discrete-log equality proof** demonstrating that `big_y` and `(big_c − λ_i · x_i · H(pk ‖ app_id))` share the same discrete logarithm with respect to `G` and `app_pk` respectively. Because the coordinator does not know `x_i`, the proof must be relative to the participant's public verification share `X_i = x_i · G2` (which should be included in `KeygenOutput` or a separate public-key package). Alternatively, restructure the protocol so that the coordinator can verify the ElGamal ciphertext consistency without knowing private shares.

### Proof of Concept

Given a 3-of-3 CKD session with participants `P1, P2, P3` and coordinator `P1`:

1. `P2` (malicious) computes its correct `(norm_big_y_2, norm_big_c_2)` but instead sends `(norm_big_y_2, norm_big_c_2 + Δ)` where `Δ` is any non-identity `ElementG1`.
2. `P1` (coordinator) receives and sums all contributions:
   - `Y = norm_big_y_1 + norm_big_y_2 + norm_big_y_3` (correct, since only `big_c` was tampered)
   - `C = norm_big_c_1 + (norm_big_c_2 + Δ) + norm_big_c_3 = C_correct + Δ`
3. The coordinator returns `CKDOutput { Y, C_correct + Δ }` with no error.
4. The application decrypts: `(C_correct + Δ) − sk_app · Y = msk · H(pk ‖ app_id) + Δ ≠ msk · H(pk ‖ app_id)`.
5. The derived confidential key is permanently wrong; the TEE application receives an unusable secret. [3](#0-2)

async fn do_ckd_coordinator(
    mut chan: SharedChannel,
    participants: ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<CKDOutputOption, ProtocolError> {
    let (mut norm_big_y, mut norm_big_c) =
        compute_signature_share(&participants, me, key_pair, app_id, app_pk, rng)?;

    // Receive everyone's inputs and add them together
    let waitpoint = chan.next_waitpoint();

    for (_, participant_output) in
        recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
    {
        norm_big_y += participant_output.big_y();
        norm_big_c += participant_output.big_c();
    }
    let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
    Ok(Some(ckd_output))
}

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
