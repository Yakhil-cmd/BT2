### Title
Relayer Fee Front-Running via Unbound `tx_data` in `submit`/`submit_with_args` — (File: `engine/src/contract_methods/evm_transactions.rs`)

---

### Summary

The `relayer_address` that receives EIP-1559 priority-fee (tip) rewards is derived solely from the NEAR `predecessor_account_id` of whoever submits the NEAR transaction. The raw Ethereum `tx_data` payload contains no cryptographic binding to any specific NEAR submitter. Any NEAR account that observes a pending `submit` call in the NEAR mempool can copy the identical `tx_data` bytes and re-submit them under their own NEAR account, stealing the relayer fee reward that was intended for the original submitter.

---

### Finding Description

**Step 1 — `relayer_address` is set to the NEAR caller, not to anything inside the Ethereum transaction.**

In both `submit` and `submit_with_args`, the relayer address is unconditionally derived from `predecessor_account_id()`:

```rust
// engine/src/contract_methods/evm_transactions.rs  line 84
let relayer_address = predecessor_address(&env.predecessor_account_id());
``` [1](#0-0) 

The same pattern is repeated for `submit_with_args`: [2](#0-1) 

`predecessor_address` is a simple hash of the NEAR account bytes — it carries no information from the Ethereum transaction itself: [3](#0-2) 

**Step 2 — The relayer address receives the EIP-1559 priority fee.**

`relayer_address` is forwarded into `engine::submit` → `submit_with_alt_modexp`, which calls `refund_unused_gas` after execution: [4](#0-3) 

Inside `refund_unused_gas`, the `relayer_reward` (computed from `priority_fee_per_gas`) is credited to whichever address was passed as `relayer`: [5](#0-4) 

**Step 3 — The Ethereum `tx_data` is not bound to the NEAR submitter.**

The Ethereum transaction is an RLP-encoded blob. Its ECDSA signature authenticates the Ethereum *sender* (recovered at line 1047 as `sender`), but it contains no field that names or commits to the NEAR account that will submit it. The `tx_data` bytes are identical regardless of which NEAR account calls `submit`. [6](#0-5) 

**Step 4 — No default restriction prevents arbitrary NEAR accounts from calling `submit`.**

The `assert_access` check enforces silo-mode EVM address whitelists, not NEAR-caller identity. In the standard (non-silo) deployment any NEAR account may call `submit`. Even when a `key_manager` is configured, the key-manager gate is an optional, operator-set feature and is not present by default. [7](#0-6) 

---

### Impact Explanation

A legitimate relayer who submits a high-tip Ethereum transaction loses the entire EIP-1559 priority fee to the front-runner. The Ethereum nonce is consumed by the attacker's transaction, so the original NEAR transaction also fails — the relayer pays NEAR gas and receives nothing. This is a direct theft of the fee reward that the relayer was entitled to for performing the submission service. Impact: **High — theft of unclaimed yield (relayer fee)**.

---

### Likelihood Explanation

NEAR Protocol has a public mempool; pending transactions are broadcast to validators before inclusion. No special privilege is required — any NEAR account can call `submit`. The attacker only needs to observe the mempool, copy `tx_data`, and submit from their own account. Profitability scales with `max_priority_fee_per_gas` set by the Ethereum transaction originator. Likelihood: **Medium** (requires mempool observation and racing inclusion, but no cryptographic or privileged capability).

---

### Recommendation

Bind the NEAR submitter identity to the Ethereum transaction data so that the relayer address cannot be hijacked:

1. **Preferred**: Extend `SubmitArgs` with an optional `relayer_evm_address` field that is included in the Ethereum transaction's signed payload (e.g., as an extra field or via EIP-712 typed data). The engine then verifies that the recovered Ethereum signature covers this field before crediting the reward.
2. **Alternative**: Require relayers to use a private or commit-reveal submission channel (analogous to Flashbots) so that `tx_data` is never visible in the public mempool before inclusion.
3. **Minimum mitigation**: Document that any Ethereum transaction carrying a non-zero `max_priority_fee_per_gas` must be submitted through a private NEAR RPC endpoint to avoid front-running.

---

### Proof of Concept

1. Alice (legitimate relayer) constructs an Ethereum EIP-1559 transaction with `max_priority_fee_per_gas = 1_000_000_000` (1 Gwei tip) and broadcasts the NEAR call `aurora.submit(rlp_bytes)` to the NEAR network.
2. Bob (attacker) reads `rlp_bytes` from the NEAR mempool — the bytes are identical to what Alice sent.
3. Bob submits his own NEAR transaction `aurora.submit(rlp_bytes)` from his NEAR account, racing Alice for inclusion.
4. Bob's transaction is included first. Inside `submit_with_alt_modexp`:
   - `relayer_address = predecessor_address(bob.near)` — Bob's EVM address.
   - The Ethereum nonce is consumed; the Ethereum sender's balance is debited for gas.
   - `refund_unused_gas` calls `add_balance(io, &bob_evm_address, relayer_reward)`.
5. Bob's EVM address receives the full priority-fee reward.
6. Alice's NEAR transaction is then processed; `check_nonce` fails (nonce already used); Alice receives nothing and has wasted NEAR gas. [8](#0-7) [9](#0-8)

pub fn submit<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<SubmitResult, ContractError> {
    with_logs_hashchain(io, env, function_name!(), |mut io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
        let tx_data = io.read_input().to_vec();
        let current_account_id = env.current_account_id();
        let relayer_address = predecessor_address(&env.predecessor_account_id());
        let args = SubmitArgs {
            tx_data,
            ..Default::default()
        };
        let result = engine::submit(
            io,
            env,
            &args,
            state,
            current_account_id,
            relayer_address,
            handler,
        )?;
        let result_bytes = borsh::to_vec(&result).map_err(|_| errors::ERR_SERIALIZE)?;
        io.return_output(&result_bytes);

        Ok(result)
    })
}

```rust
pub fn refund_unused_gas<I: IO>(
    io: &mut I,
    sender: &Address,
    gas_used: u64,
    gas_result: &GasPaymentResult,
    relayer: &Address,
    fixed_gas: Option<EthGas>,
) -> Result<(), GasPaymentError> {
    if gas_result.effective_gas_price.is_zero() {
        return Ok(());
    }

    let (refund, relayer_reward) = {
        let gas_to_wei = |price: U256| {
            fixed_gas
                .map_or_else(|| gas_used.into(), EthGas::as_u256)
                .checked_mul(price)
                .map(Wei::new)
                .ok_or(GasPaymentError::EthAmountOverflow)
        };

        let spent_amount = gas_to_wei(gas_result.effective_gas_price)?;
        let reward_amount = gas_to_wei(gas_result.priority_fee_per_gas)?;

        let refund = gas_result
            .prepaid_amount
            .checked_sub(spent_amount)
            .ok_or(GasPaymentError::EthAmountOverflow)?;

        (refund, reward_amount)
    };

    if !refund.is_zero() {
        add_balance(io, sender, refund)?;
    }

    if !relayer_reward.is_zero() {
        add_balance(io, relayer, relayer_reward)?;
    }

    Ok(())
}
