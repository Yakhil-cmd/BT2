### Title
Gateway L2 Gas Price Threshold Uses Current Block Price Instead of `next_l2_gas_price`, Admitting Transactions That Will Fail in Batcher - (File: crates/apollo_gateway/src/stateful_transaction_validator.rs)

### Summary

`StatefulTransactionValidator::validate_resource_bounds` reads the **current block's** `l2_gas_price` from the block header to compute the admission threshold, but the batcher executes transactions at the **next block's** L2 gas price (computed via EIP-1559). A developer TODO in the code explicitly acknowledges this is wrong. The result is that V3 transactions with `max_price_per_unit` between the current and next block's L2 gas price pass gateway admission but are rejected during batcher execution, and conversely, transactions valid for the next block may be incorrectly rejected at the gateway.

### Finding Description

In `validate_resource_bounds`, the gateway fetches the previous (latest committed) block's info and extracts `strk_gas_prices.l2_gas_price` as the reference price for the threshold check:

```rust
// TODO(Arni): getnext_l2_gas_price from the block header.
let previous_block_l2_gas_price = self
    .gateway_fixed_block_state_reader
    .get_block_info()
    .await?
    .gas_prices
    .strk_gas_prices
    .l2_gas_price;
self.validate_tx_l2_gas_price_within_threshold(
    executable_tx.resource_bounds(),
    previous_block_l2_gas_price,
)?;
``` [1](#0-0) 

The `GatewayFixedBlockSyncStateClient` populates `strk_gas_prices.l2_gas_price` from `block_header.l2_gas_price.price_in_fri` — the price **used in the current block**: [2](#0-1) 

However, `BlockHeaderWithoutHash` carries a distinct field `next_l2_gas_price: GasPrice` — the EIP-1559-adjusted price that will govern the **next** block: [3](#0-2) 

This `next_l2_gas_price` is computed by `calculate_next_l2_gas_price_for_fin` and stored in the block header by the consensus orchestrator: [4](#0-3) 

The batcher builds the next block using gas prices from `ProposalInit`, which reflects the EIP-1559-adjusted price, not the previous block's price. The gateway's `run_validate_entry_point` also uses the previous block's gas prices (it only increments the block number): [5](#0-4) 

The threshold check in `validate_tx_l2_gas_price_within_threshold` computes:

```
threshold = (min_gas_price_percentage / 100) * previous_block_l2_gas_price
``` [6](#0-5) 

When the network is under high load, EIP-1559 raises `next_l2_gas_price` above `l2_gas_price`. A transaction with `max_price_per_unit = P` where `threshold_from_current_price ≤ P < next_block_price` will:

1. Pass `validate_resource_bounds` (threshold based on current price)
2. Pass blockifier's `check_fee_bounds` in the gateway (also uses current block prices)
3. Be admitted to the mempool
4. **Fail** in the batcher with `MaxGasPriceTooLow` because the batcher uses `next_l2_gas_price`

The inverse also holds: when load drops and `next_l2_gas_price < l2_gas_price`, transactions valid for the next block may be rejected at the gateway.

### Impact Explanation

**High — Mempool/gateway admission accepts invalid transactions or rejects valid transactions before sequencing.**

Every V3 (`AllResources`) transaction submitted during a period of changing L2 gas price is affected. Admitted-but-invalid transactions waste mempool slots and batcher execution cycles. Rejected-but-valid transactions deny users service. The developer TODO comment confirms this is a known wrong reference point.

### Likelihood Explanation

**High.** The EIP-1559 mechanism is designed to adjust the L2 gas price every block based on gas usage. Any block with gas usage above or below the target will produce a `next_l2_gas_price` that differs from `l2_gas_price`. This is the normal operating condition of the network, not an edge case.

### Recommendation

Replace the read of `strk_gas_prices.l2_gas_price` with `next_l2_gas_price` from the block header. The `GatewayFixedBlockSyncStateClient` should expose `next_l2_gas_price` (already present in `BlockHeaderWithoutHash`), and `validate_resource_bounds` should use it:

```rust
// In GatewayFixedBlockStateReader, expose next_l2_gas_price.
// In validate_resource_bounds:
let next_block_l2_gas_price = self
    .gateway_fixed_block_state_reader
    .get_next_l2_gas_price()   // reads block_header.next_l2_gas_price
    .await?;
self.validate_tx_l2_gas_price_within_threshold(
    executable_tx.resource_bounds(),
    next_block_l2_gas_price,
)?;
```

This mirrors the fix in the flash loan report: use the correct reference point (post-transfer balance / next block price) rather than the stale one (pre-transfer balance / current block price).

### Proof of Concept

1. Observe the current block's `l2_gas_price` (e.g., 10 Gwei) and `next_l2_gas_price` (e.g., 11 Gwei, due to high gas usage).
2. Submit a V3 `AllResources` invoke transaction with `l2_gas.max_price_per_unit = 10.5 Gwei` (above the threshold based on current price, below the next block price).
3. The gateway's `validate_tx_l2_gas_price_within_threshold` computes `threshold = 100% * 10 Gwei = 10 Gwei`; `10.5 Gwei ≥ 10 Gwei` → **passes**.
4. The blockifier validation in `run_validate_entry_point` uses the current block's prices; `10.5 Gwei ≥ 10 Gwei` → **passes**.
5. Transaction is admitted to the mempool.
6. The batcher builds the next block at `next_l2_gas_price = 11 Gwei`; `check_fee_bounds` finds `10.5 Gwei < 11 Gwei` → **`MaxGasPriceTooLow`**, transaction fails. [7](#0-6) [8](#0-7) [9](#0-8)
