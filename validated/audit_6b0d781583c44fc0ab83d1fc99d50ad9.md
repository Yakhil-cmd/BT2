### Title
Unprivileged Caller Can Permanently Freeze Consensus Rewards for All Stakers via `update_rewards` with `disable_rewards: true` - (File: src/staking/staking.cairo)

---

### Summary

The `update_rewards` function in the Staking contract is callable by any unprivileged address and accepts a user-controlled `disable_rewards` boolean parameter. Because `last_reward_block` is written to storage **before** the `disable_rewards` guard is evaluated, any caller can consume the single per-block reward slot for every block without distributing any rewards, permanently freezing unclaimed yield for all stakers.

---

### Finding Description

`update_rewards` is exposed as a public entrypoint under `IStakingRewardsManager`. Its only access gate is `general_prerequisites`, which checks that the contract is not paused and the caller is not the zero address — no role or identity check is performed. [1](#0-0) 

The function enforces a global, single-slot-per-block invariant via `last_reward_block`: [2](#0-1) 

Critically, `last_reward_block` is committed to storage **before** the `disable_rewards` branch is evaluated: [3](#0-2) 

Because `last_reward_block` is a single global slot shared across all stakers, once it is written for block N, every subsequent call to `update_rewards` in block N reverts with `REWARDS_ALREADY_UPDATED`. An attacker who calls `update_rewards(any_active_staker, disable_rewards: true)` once per block consumes the slot without distributing any rewards, and no legitimate caller can recover it for that block. [4](#0-3) 

The only prerequisite for the attacker is supplying a valid, active staker address with non-zero STRK balance at the current epoch — information that is entirely public on-chain via `get_stakers` or emitted events. [5](#0-4) 

---

### Impact Explanation

Every block in the consensus rewards phase that is consumed by the attacker produces zero rewards for all stakers and all delegation pools. Because the attacker can repeat this every block at negligible cost, the cumulative effect is the **permanent freezing of all unclaimed consensus-phase yield** for every staker and pool member in the protocol. This matches the allowed High impact: *Permanent freezing of unclaimed yield*.

---

### Likelihood Explanation

- The entry point is fully public; no key, role, or privileged access is required.
- The attacker only needs to know one active staker address (trivially obtained from on-chain events or `get_stakers`).
- The cost per block is a single transaction with a small gas fee.
- The attack is sustainable indefinitely and cannot be mitigated by stakers themselves once started.

Likelihood: **High**.

---

### Recommendation

Move the `last_reward_block` write to **after** the `disable_rewards` guard, so that a call that distributes no rewards does not consume the per-block slot:

```cairo
if disable_rewards || self.is_pre_consensus() {
    return;
}

// Only consume the per-block slot when rewards are actually distributed.
self.last_reward_block.write(current_block_number);

// ... rest of reward calculation
```

Additionally, consider restricting `disable_rewards: true` to a privileged role (e.g., `SECURITY_AGENT`) or removing the parameter entirely if it has no legitimate public use case.

---

### Proof of Concept

```
// Attacker script (pseudocode, run once per block):
let active_staker = staking.get_stakers(current_epoch)[0].address;

// Consumes last_reward_block for this block, distributes zero rewards.
staking.update_rewards(staker_address: active_staker, disable_rewards: true);

// All subsequent legitimate calls in this block revert with REWARDS_ALREADY_UPDATED.
// staking.update_rewards(legitimate_staker, disable_rewards: false); // REVERTS
```

The attacker repeats this every block. `last_reward_block` advances each block, but no rewards are ever distributed. All staker and pool-member unclaimed yield remains permanently at zero. [6](#0-5)
