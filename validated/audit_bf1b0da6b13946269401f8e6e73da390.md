### Title
Presignature Reuse via Non-Consuming `rerandomize_presign()` API Enables Secret Key Extraction вЂ” (File: `src/ecdsa/ot_based_ecdsa/mod.rs`, `src/ecdsa/robust_ecdsa/mod.rs`)

---

### Summary

Both ECDSA implementations expose `rerandomize_presign()` accepting the `PresignOutput` as a **shared reference** (`&PresignOutput`) rather than consuming it by value. This mirrors the Multicall `msg.value` reuse pattern: just as `msg.value` persists unchanged across recursive delegate calls allowing ETH to be "spent" twice, the `PresignOutput` nonce material persists across multiple `rerandomize_presign()` invocations, allowing a malicious coordinator to derive two independent `RerandomizedPresignOutput` values from the same presignature. Running two signing sessions with different `(h, Оµ)` pairs against different participant subsets produces signatures with multiplicatively related nonces, enabling full secret key extraction via standard ECDSA nonce-reuse analysis.

Because the `PresignOutput` is borrowed rather than moved, the caller retains full ownership and can invoke `rerandomize_presign()` an arbitrary number of times with different `RerandomizationArguments` (different `(h, Оµ, ПЃ)` tuples), producing a distinct `RerandomizedPresignOutput` each time вЂ” all sharing the same underlying nonce material.

Contrast this with `sign()`, which **does** consume the rerandomized output by value, enforcing one-time use at the wrong level:

```rust
// src/ecdsa/ot_based_ecdsa/sign.rs  line 28
presignature: RerandomizedPresignOutput,   // в†ђ consumed; one-time use enforced here
``` [3](#0-2) 

The library's own security documentation acknowledges the consequence:

> "If different subsets of size at least 2t+1 sign different (h, Оµ) values using shares derived from the same presignature, the resulting signatures use multiplicatively related nonces and the secret key can be recovered using standard ECDSA nonce-reuse attacks."
> "a novel split-view attack exists that can extract the secret key using as few as 2t+2 presigning participants, with as few as two signing sessions." [4](#0-3) 

The code does **not** enforce the documented "never reuse a presignature" invariant at the type level. [5](#0-4)

### Impact Explanation

**Critical вЂ” Extraction of private signing shares / aggregate secret key material.**

A malicious coordinator who participates in a single presigning session can:

1. Obtain their `PresignOutput` share `(R, О±_i, ОІ_i, c_i, e_i)`.
2. Call `rerandomize_presign(&presign_output, &args1)` в†’ `rp1` with `(hв‚Ѓ, Оµв‚Ѓ)`.
3. Call `rerandomize_presign(&presign_output, &args2)` в†’ `rp2` with `(hв‚‚, Оµв‚‚)` вЂ” **same `presign_output`**.
4. Orchestrate two signing sessions with different participant subsets `Pв‚Ѓ` and `Pв‚‚` (each of size `2t+1`), feeding `(hв‚Ѓ, Оµв‚Ѓ)` to `Pв‚Ѓ` and `(hв‚‚, Оµв‚‚)` to `Pв‚‚`.
5. Collect signatures `(Rв‚Ѓ, sв‚Ѓ)` and `(Rв‚‚, sв‚‚)` whose nonces satisfy `kв‚Ѓ = Оґв‚ЃвЃ»В№ В· k` and `kв‚‚ = Оґв‚‚вЃ»В№ В· k` for the same underlying nonce `k`.
6. Apply standard ECDSA nonce-reuse algebra to recover the secret key `x`.

The full master private key is recoverable. All past and future signatures are compromised.

---

### Likelihood Explanation

**High.** The malicious coordinator is an explicitly documented attacker profile for this library. No privileged access beyond participation in the presigning protocol is required. The API makes reuse trivially easy вЂ” no explicit `.clone()` call is needed; the caller simply passes `&presign_output` twice. The library provides no runtime guard, counter, or flag to detect or prevent repeated rerandomization of the same `PresignOutput`.

---

### Recommendation

Change both `rerandomize_presign()` signatures to **consume** the `PresignOutput` by value:

```rust
// Before (unsafe вЂ” allows reuse)
pub fn rerandomize_presign(
    presignature: &PresignOutput,
    args: &RerandomizationArguments,
) -> Result<Self, ProtocolError>

// After (safe вЂ” enforces one-time use at the type level)
pub fn rerandomize_presign(
    presignature: PresignOutput,
    args: &RerandomizationArguments,
) -> Result<Self, ProtocolError>
```

This mirrors the existing pattern in `sign()` and makes presignature reuse a **compile-time error** rather than a runtime security assumption. Any caller wishing to reuse a presignature would be forced to call `.clone()` explicitly, making the dangerous operation visible and auditable.

### Proof of Concept
