### Title
XCC Router Sub-account Has No User-Facing NEAR Withdrawal Mechanism, Permanently Freezing User Funds - (`etc/xcc-router/src/lib.rs`)

### Summary

The XCC router sub-account (`{address_hex}.aurora`) is a NEAR contract that holds NEAR tokens on behalf of an EVM user. The router has no method allowing the EVM user to withdraw their NEAR balance. All control methods are gated to the Aurora engine (parent), and the only outgoing NEAR transfer is `send_refund`, which sends a fixed amount back to the engine — not to the user. NEAR that accumulates in the router (from storage staking costs paid by the user, or from excess NEAR returned by called contracts) is permanently frozen.

### Finding Description

When an EVM user first uses the XCC precompile, the engine creates a router sub-account and charges the user `attached_near + STORAGE_AMOUNT` (2 NEAR) in wNEAR. The flow in `handle_precompile_promise` is:

1. Engine sends `STORAGE_AMOUNT` (2 NEAR) to the router for storage staking.
2. User's wNEAR (`attached_near + STORAGE_AMOUNT`) is converted to NEAR and sent to the router via `withdraw_wnear_to_router`.
3. Router calls `send_refund`, which sends exactly `REFUND_AMOUNT` (2 NEAR) back to the engine. [1](#0-0) 

The net result: the user paid 2 NEAR (via wNEAR) for storage staking, and this 2 NEAR is permanently locked in the router. Additionally, when a user attaches NEAR to a cross-contract call and the called contract does not consume all of it, the excess NEAR is returned to the router (as the caller) and accumulates there.

The router contract exposes only these methods:

- `execute` — only callable by the parent (Aurora engine)
- `schedule` — only callable by the parent
- `execute_scheduled` — callable by anyone, but only executes previously scheduled promises
- `deploy_upgrade` — only callable by the parent
- `send_refund` — only callable by the parent, sends exactly 2 NEAR to the engine [2](#0-1) 

There is no `withdraw` method or any mechanism for the EVM user (the owner of the EVM address that maps to this router) to retrieve their NEAR balance.

The XCC precompile itself only allows users to make outgoing cross-contract calls from the router. It does not provide a "withdraw NEAR from router" operation. [3](#0-2) 

Without the `all-promise-actions` feature, even batch `Transfer` actions are unavailable: [4](#0-3) 

The `require_caller` check on all sensitive methods ensures only the engine can act on the router: [5](#0-4) 

### Impact Explanation

**Critical — Permanent freezing of funds.**

Every EVM user who uses the XCC precompile has at minimum 2 NEAR (paid via wNEAR) permanently locked in their router sub-account with no recovery path. Additionally, any excess NEAR returned by called contracts accumulates in the router and is equally unrecoverable. The user cannot delete the router account (no `delete_account` method exists), cannot call `send_refund` directly (parent-only), and cannot use the XCC precompile to transfer NEAR to their own NEAR account without a helper contract.

### Likelihood Explanation

**High.** The XCC feature is a core, user-facing capability of Aurora Engine. Every user who invokes the XCC precompile for the first time triggers router creation and pays the 2 NEAR storage cost. This is a normal, expected usage path — not an edge case. The freezing occurs unconditionally on first use. [6](#0-5) 

### Recommendation

Add a user-callable withdrawal method to the router contract that allows the EVM address owner (verified via the engine) to reclaim their NEAR balance. One approach:

1. Add a `withdraw_near` method to the router that can be triggered by the engine on behalf of the user.
2. Add a corresponding engine-level entrypoint (e.g., `xcc_withdraw_near`) that the EVM user can invoke via a NEAR transaction, which calls `withdraw_near` on their router sub-account and transfers the balance to a specified NEAR account.

Alternatively, expose a `PromiseAction::DeleteAccount` path (with `all-promise-actions`) so users can close their router and reclaim storage staking.

### Proof of Concept

1. EVM user Alice calls the XCC precompile for the first time with `attached_near = 0`.
2. The engine creates router `{alice_address_hex}.aurora`, transferring 2 NEAR for storage staking.
3. Alice's wNEAR balance is debited by 2 NEAR (converted to NEAR and sent to the router, then refunded to the engine via `send_refund`).
4. Alice's router now holds 2 NEAR for storage staking.
5. Alice wants to stop using XCC and reclaim her 2 NEAR.
6. Alice cannot call `send_refund` (parent-only). She cannot call `execute` or `schedule` (parent-only). She has no `withdraw` method available.
7. Alice's 2 NEAR is permanently frozen in `{alice_address_hex}.aurora`. [7](#0-6) [8](#0-7)

        self.assert_preconditions();

        let promise_id = Self::promise_create(promise);
        env::promise_return(promise_id);
    }

    /// Similar security considerations here as for `execute`.
    pub fn schedule(&mut self, #[serializer(borsh)] promise: PromiseArgs) {
        self.assert_preconditions();

        let nonce = self.nonce.get().unwrap_or_default();
        self.scheduled_promises.insert(nonce, promise);
        self.nonce.set(&(nonce + 1));

        near_sdk::log!("Promise scheduled at nonce {}", nonce);
    }

    /// It is intentional that this function can be called by anyone (not just the parent).
    /// There is no security risk to allowing this function to be open because it can only
    /// act on promises that were created via `schedule`.
    #[payable]
    pub fn execute_scheduled(&mut self, nonce: U64) {
        let Some(promise) = self.scheduled_promises.remove(&nonce.0) else {
            env::panic_str("ERR_PROMISE_NOT_FOUND")
        };
        let promise_id = Self::promise_create(promise);
        env::promise_return(promise_id);
    }

    /// Allows the parent contract to trigger an update to the logic of this contract
    /// (by deploying a new contract to this account);
    #[payable]
    pub fn deploy_upgrade(&mut self, #[serializer(borsh)] args: DeployUpgradeParams) {
        self.assert_preconditions();

        let promise_id = env::promise_batch_create(&env::current_account_id());
        env::promise_batch_action_deploy_contract(promise_id, &args.code);
        env::promise_batch_action_function_call(
            promise_id,
            INITIALIZE,
            &args.initialize_args,
            NearToken::default(),
            INITIALIZE_GAS,
        );
        env::promise_return(promise_id);
    }

    pub fn send_refund(&self) -> Promise {
        let parent = self.get_parent().unwrap_or_else(env_panic);

        require_caller(&parent)
            .and_then(|_| require_no_failed_promises())
            .unwrap_or_else(env_panic);

        Promise::new(parent).transfer(REFUND_AMOUNT)
    }
