### Title
Insufficient Actual Token Balance in `RewardSupplier` Causes Temporary Freezing of Rewards and Unstake Funds - (File: `src/reward_supplier/reward_supplier.cairo`)

---

### Summary

`RewardSupplier.claim_rewards()` validates only the accounting variable `unclaimed_rewards` before attempting a token transfer, but never checks the contract's actual STRK token balance. Because L1 minting is asynchronous, the actual balance can be zero (or insufficient) while `unclaimed_rewards` is large. Any call that routes through `claim_from_reward_supplier` — including `claim_rewards` and `unstake_action` — will revert until L1 tokens arrive, temporarily freezing both unclaimed yield and staked principal.

---

### Finding Description

`RewardSupplier.claim_rewards()` performs a single guard:

```cairo
let unclaimed_rewards = self.unclaimed_rewards.read();
assert!(amount <= unclaimed_rewards, "{}", GenericError::AMOUNT_TOO_HIGH);
// ...
token_dispatcher.checked_transfer(recipient: staking_contract, amount: amount.into());
``` [1](#0-0) 

The assertion only checks the *accounting* ledger (`unclaimed_rewards`), not the contract's real token balance. The actual balance is replenished asynchronously from L1 via `on_receive` (StarkGate callback). The `request_funds` helper proactively sends L1 mint messages, but it counts `l1_pending_requested_amount` as credit:

```cairo
let credit = balance + l1_pending_requested_amount;
let debit = unclaimed_rewards;
if credit < debit + threshold { /* send L1 messages */ }
``` [2](#0-1) 

So `request_funds` may decide no additional L1 messages are needed (because pending requests are already counted), while the actual on-chain balance is still zero. During this window, `checked_transfer` will revert.

The staking contract calls `RewardSupplier.claim_rewards` through `claim_from_reward_supplier`, which additionally asserts an exact balance delta:

```cairo
let balance_before = token_dispatcher.balance_of(account: staking_contract);
reward_supplier_dispatcher.claim_rewards(:amount);
let balance_after = token_dispatcher.balance_of(account: staking_contract);
assert!(balance_after - balance_before == amount.into(), "{}", Error::UNEXPECTED_BALANCE);
``` [3](#0-2) 

`claim_from_reward_supplier` is invoked in two critical paths:

1. **`send_rewards_to_staker`** — called inside both `Staking::claim_rewards` and `Staking::unstake_action`:

```cairo
claim_from_reward_supplier(:reward_supplier_dispatcher, :amount, :token_dispatcher);
token_dispatcher.checked_transfer(recipient: reward_address, amount: amount.into());
``` [4](#0-3) 

2. **`_update_rewards`** — called inside `update_rewards` and `update_rewards_from_attestation_contract` to forward pool rewards:

```cairo
claim_from_reward_supplier(
    :reward_supplier_dispatcher,
    amount: total_pools_rewards,
    token_dispatcher: strk_token_dispatcher(),
);
``` [5](#0-4) 

---

### Impact Explanation

- **Temporary freezing of unclaimed yield**: Stakers and pool members cannot call `claim_rewards` while the `RewardSupplier`'s actual balance is below `unclaimed_rewards`. The `checked_transfer` inside `RewardSupplier.claim_rewards` reverts the entire transaction.
- **Temporary freezing of funds (principal)**: `unstake_action` calls `send_rewards_to_staker` before returning the staker's principal. If `send_rewards_to_staker` reverts due to insufficient `RewardSupplier` balance, the staker cannot complete the unstake and retrieve their staked STRK — even after the exit window has elapsed.

Both effects persist until L1 minting completes and `on_receive` replenishes the actual balance.

---

### Likelihood Explanation

The L1→L2 bridge (StarkGate) introduces an inherent, non-trivial delay between when a mint request is sent and when tokens arrive. During periods of high attestation activity (many stakers attesting in the same epoch), `unclaimed_rewards` grows rapidly across multiple `update_unclaimed_rewards_from_staking_contract` calls, while the actual balance is only replenished once per L1 message cycle. Any staker or pool member who attempts to claim rewards or complete an unstake during this window will be blocked.

---

### Recommendation

In `RewardSupplier.claim_rewards`, add a check that the contract's actual token balance is sufficient before attempting the transfer:

```cairo
let balance: Amount = token_dispatcher
    .balance_of(account: get_contract_address())
    .try_into()
    .expect_with_err(InternalError::BALANCE_ISNT_AMOUNT_TYPE);
assert!(amount <= balance, "{}", Error::INSUFFICIENT_BALANCE);
```

Alternatively, decouple reward claiming from `unstake_action` so that a staker can recover their principal even when the `RewardSupplier` has a temporary liquidity shortfall (e.g., by allowing `unstake_action` to skip reward transfer if the supplier is illiquid, and letting the staker claim rewards separately once liquidity is restored).

---

### Proof of Concept

1. Many stakers attest in epoch N. Each attestation triggers `update_rewards_from_attestation_contract` → `_update_rewards` → `update_unclaimed_rewards_from_staking_contract`. `unclaimed_rewards` grows to `R`.
2. `request_funds` sends L1 mint messages. `l1_pending_requested_amount = R + threshold`. Actual balance = 0 (tokens not yet bridged).
3. Staker A calls `Staking::claim_rewards(staker_A)`.
4. `send_rewards_to_staker` → `claim_from_reward_supplier(amount = staker_A_rewards)` → `RewardSupplier.claim_rewards(amount)`.
5. `amount <= unclaimed_rewards` passes (accounting check).
6. `token_dispatcher.checked_transfer(recipient: staking_contract, amount)` **reverts** — actual balance is 0.
7. Entire transaction reverts. Staker A cannot claim rewards.
8. Staker A calls `Staking::unstake_action(staker_A)` (exit window has passed). Same revert path. Staker A cannot recover their staked STRK.
9. Condition persists until L1 bridge delivers tokens via `on_receive`. [6](#0-5) [7](#0-6) [3](#0-2)

            // Prerequisites and asserts.
            self.general_prerequisites();
            let mut staker_info = self.internal_staker_info(:staker_address);
            let unstake_time = staker_info
                .unstake_time

            assert!(Time::now() >= unstake_time, "{}", GenericError::INTENT_WINDOW_NOT_FINISHED);

            // Send rewards to staker's reward address.
            // It must be part of this function's flow because staker_info is about to be erased.
            let token_dispatcher = strk_token_dispatcher();

            // Update staker info to storage (it will be erased later).
            // This is done here to avoid re-entrancy.
            self.write_staker_info(:staker_address, :staker_info);

            let staker_amount = self.get_own_balance(:staker_address).to_strk_native_amount();
            let staker_pool_info = self.staker_pool_info.entry(staker_address);
            self.remove_staker(:staker_address, :staker_info, :staker_pool_info);

            // Return stake to staker.
            token_dispatcher

            self

                );
            // Clear staker pools.
            staker_pool_info.pools.clear();
            staker_amount
        }
