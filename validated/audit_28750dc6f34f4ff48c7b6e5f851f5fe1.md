### Title
Pre-EIP-155 Legacy Transactions Bypass Chain ID Validation, Enabling Cross-Chain Replay — (`engine/src/engine.rs`)

---

### Summary

The Aurora Engine accepts legacy Ethereum transactions signed without EIP-155 chain ID protection (v=27 or v=28). The chain ID validation in `submit_with_alt_modexp` is guarded by an `if let Some(chain_id)` check, so when a legacy transaction carries no chain ID, the guard is never entered and the transaction is processed on any Aurora deployment regardless of its configured chain ID. An attacker can take a valid pre-EIP-155 transaction broadcast on one Aurora deployment and replay it verbatim on any other Aurora deployment where the sender has a matching nonce and sufficient balance.

---

### Finding Description

In `engine-transactions/src/legacy.rs`, `LegacyEthSignedTransaction::chain_id()` returns `None` for any transaction whose `v` field is 27 or 28 (pre-EIP-155 encoding):

```rust
// engine-transactions/src/legacy.rs:88-93
pub const fn chain_id(&self) -> Option<u64> {
    match self.v {
        0..=34 => None,
        _ => Some((self.v - 35) / 2),
    }
}
``` [1](#0-0) 

When such a transaction is normalized in `engine-transactions/src/lib.rs`, the `chain_id` field of `NormalizedEthTransaction` is set to `None`:

```rust
// engine-transactions/src/lib.rs:106-108
Legacy(tx) => Self {
    address: tx.sender()?,
    chain_id: tx.chain_id(),   // None for v=27/28
    ...
``` [2](#0-1) 

In `engine/src/engine.rs`, the chain ID validation is conditional on `transaction.chain_id` being `Some`:

```rust
// engine/src/engine.rs:1054-1059
// Validate the chain ID, if provided inside the signature:
if let Some(chain_id) = transaction.chain_id
    && U256::from(chain_id) != U256::from_big_endian(&state.chain_id)
{
    return Err(EngineErrorKind::InvalidChainId.into());
}
``` [3](#0-2) 

When `transaction.chain_id` is `None`, the entire block is skipped. No other check in `submit_with_alt_modexp` binds the transaction to the engine's configured chain ID. The transaction then proceeds through nonce validation, gas charging, and EVM execution as if it were fully authorized for this chain.

The engine's own changelog confirms this path was deliberately re-opened: version 2.4.0 closed it ("Engine will no longer accept EVM transactions without a chain ID"), but version 2.6.0 re-opened it ("Original ETH transactions which do not contain a Chain ID are allowed again to allow for use of EIP-1820"). [4](#0-3) 

---

### Impact Explanation

**High — Theft of user funds / temporary freezing of funds.**

A pre-EIP-155 transaction signed for Aurora mainnet (chain ID 1313161554) is byte-for-byte valid on Aurora testnet (chain ID 1313161555) or any other Aurora Engine deployment. If the sender's account on the target chain has:
- the same nonce as when the original transaction was signed, and
- a balance sufficient to cover the value and gas,

the transaction executes in full on the target chain, transferring or consuming the sender's funds without their consent. Because Ethereum addresses are deterministic and many users hold balances on both mainnet and testnet Aurora, this is a realistic scenario for fund theft. If the replayed transaction is a contract deployment or a call that locks funds, the result can be permanent fund freeze.

---

### Likelihood Explanation

**Medium.**

Pre-EIP-155 transactions (v=27/28) are less common with modern tooling, but they remain valid and are explicitly supported by the engine for EIP-1820 compatibility. Any wallet or library that omits the chain ID (older versions of web3.js, ethers.js with `chainId: null`, raw RLP construction, or EIP-1820 factory deployment scripts) produces such transactions. Users who deploy contracts or interact with Aurora on both mainnet and testnet with the same key and similar nonce sequences are directly exposed. The attacker's only requirement is to observe the original transaction (it is public on-chain) and resubmit it to the target deployment.

---

### Recommendation

Reject all legacy transactions that do not carry a chain ID (v=27/28) at the `submit` entrypoint, unless a specific allow-list of known chain-agnostic contract addresses (e.g., the EIP-1820 registry deployer address) is maintained. Concretely, change the chain ID guard in `submit_with_alt_modexp` from an opt-in check to an opt-out check:

```rust
// Proposed: reject if chain_id is absent OR mismatched
match transaction.chain_id {
    None => return Err(EngineErrorKind::InvalidChainId.into()),
    Some(chain_id) if U256::from(chain_id) != U256::from_big_endian(&state.chain_id) => {
        return Err(EngineErrorKind::InvalidChainId.into());
    }
    _ => {}
}
```

If EIP-1820 support must be preserved, gate the `None` exception on the sender address matching the known EIP-1820 deployer (`0xa990077c3205cbDf861e17Fa532eeB069cE9fF96`) rather than allowing all pre-EIP-155 transactions unconditionally.

---

### Proof of Concept

1. Alice holds 10 ETH on Aurora mainnet and 5 ETH on Aurora testnet. Her nonce on both chains is 3.
2. Alice signs a legacy transfer transaction (v=27, no chain ID) on Aurora mainnet: `{nonce:3, to: Bob, value: 1 ETH, ...}`. The raw RLP bytes are public.
3. Attacker Charlie copies the raw bytes and calls `submit(raw_bytes)` on Aurora testnet.
4. In `submit_with_alt_modexp`, `transaction.chain_id` is `None`; the `if let Some(chain_id)` guard is not entered.
5. `check_nonce` passes (Alice's testnet nonce is also 3).
6. The EVM executes the transfer: Alice loses 1 ETH on testnet without having signed anything for testnet.

The entry point is the public `submit()` NEAR method:

```rust
// engine/src/lib.rs:274-282
#[unsafe(no_mangle)]
pub extern "C" fn submit() {
    let io = Runtime;
    let env = Runtime;
    let mut handler = Runtime;
    contract_methods::evm_transactions::submit(io, &env, &mut handler)
        .map_err(ContractError::msg)
        .sdk_unwrap();
}
``` [5](#0-4) 

No privilege is required. Any NEAR account can call `submit` with the replayed bytes.
