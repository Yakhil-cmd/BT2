### Title
COA Ownership Proof Signature Contains No Deadline or Nonce, Making It Permanently Reusable - (`fvm/evm/stdlib/contract.cdc`, `fvm/evm/handler/coa/coa.sol`)

---

### Summary

`EVM.validateCOAOwnershipProof` in `fvm/evm/stdlib/contract.cdc` verifies that a Flow account's keys signed an arbitrary `signedData` blob and that the account controls a COA resource. Neither the protocol nor the `COAOwnershipProof` struct enforces any nonce, deadline, or context binding inside `signedData`. The `COA.sol` ERC-1271 implementation (`isValidSignature`) delegates directly to this precompile, passing the caller-supplied `_hash` as `signedData`. Once a COA owner produces a valid proof for any `_hash`, that proof remains cryptographically valid forever and can be replayed by any EVM transaction sender against any EVM contract that calls `isValidSignature` on the COA.

---

### Finding Description

`COAOwnershipProof` is defined in `fvm/evm/types/proof.go` as:

```go
type COAOwnershipProof struct {
    KeyIndices     KeyIndices
    Address        FlowAddress
    CapabilityPath PublicPath
    Signatures     Signatures
}
```

It contains no nonce, no deadline, and no context identifier. [1](#0-0) 

`validateCOAOwnershipProof` in `fvm/evm/stdlib/contract.cdc` accepts an arbitrary `signedData [UInt8]` and verifies the signatures over it. The function performs three checks: (1) key validity, (2) cryptographic signature validity over `signedData`, and (3) COA address matching. There is no check that `signedData` encodes a deadline, nonce, or any time-bound context. The code itself acknowledges a related replay risk in a comment but does not fix the timeless nature of the signature: [2](#0-1) 

The `COA.sol` ERC-1271 implementation calls the `verifyCOAOwnershipProof` precompile, passing the caller-supplied `_hash` directly as `signedData`:

```solidity
function isValidSignature(bytes32 _hash, bytes memory _sig)
    external view virtual returns (bytes4) {
    (bool ok, bytes memory data) = cadenceArch.staticcall(
        abi.encodeWithSignature(
            "verifyCOAOwnershipProof(address,bytes32,bytes)",
            address(this), _hash, _sig));
    ...
}
``` [3](#0-2) 

The precompile handler in `fvm/evm/precompiles/arch.go` decodes the proof and calls `coaOwnershipProofValidator`, which in turn calls `validateCOAOwnershipProof` with no additional temporal or uniqueness constraints. [4](#0-3) 

---

### Impact Explanation

Any EVM contract on Flow EVM that uses ERC-1271 (`isValidSignature`) to authorize actions against a COA is vulnerable to signature replay. Once a COA owner signs a `_hash` for any purpose (e.g., approving a trade, authorizing a permit, signing a governance vote), the resulting `COAOwnershipProof` is valid forever. An attacker who observes the proof bytes from a past EVM transaction's calldata can replay them in a new EVM transaction to impersonate the COA owner in any contract that accepts the same `_hash`. This constitutes unauthorized authorization bypass in EVM contracts that rely on the COA's ERC-1271 interface for access control.

---

### Likelihood Explanation

The entry path is fully unprivileged: any EVM transaction sender can call `isValidSignature` on a COA contract. COA ownership proofs are transmitted as EVM calldata, which is publicly observable on-chain. ERC-1271 is a standard interface widely used by EVM DeFi protocols for smart-contract-wallet authorization. The likelihood increases as Flow EVM adoption grows and more EVM contracts integrate COA-based ERC-1271 signatures.

---

### Recommendation

The protocol should enforce that `signedData` encodes a nonce or deadline at the `validateCOAOwnershipProof` level, or the `COA.sol` `isValidSignature` implementation should require callers to include a block-height-based deadline or a per-COA nonce in the hash. A minimal fix is to track a per-COA nonce in storage and require `signedData` to encode `(hash, nonce, deadline)`, rejecting proofs where `deadline < currentBlockHeight` or where the nonce has already been consumed.

---

### Proof of Concept

1. Alice (COA owner) signs `hash(X)` to authorize action X in EVM DApp A. The resulting `COAOwnershipProof` is submitted as calldata in an EVM transaction and is publicly visible on-chain.

2. Attacker Bob extracts the `(signedData=hash(X), proof)` tuple from the transaction calldata.

3. EVM DApp B independently uses `isValidSignature(hash(X), proof)` on Alice's COA to gate a different privileged action Y (e.g., same hash coincidentally, or a cross-dApp replay).

4. Bob submits an EVM transaction calling DApp B with Alice's old proof. `COA.isValidSignature` calls `verifyCOAOwnershipProof(alice_coa, hash(X), proof)`.

5. `validateCOAOwnershipProof` verifies the signatures over `signedData = hash(X)` — they are still cryptographically valid — and returns `ValidationResult(isValid: true)`. [5](#0-4) 

6. `isValidSignature` returns `0x1626ba7e` (valid), and DApp B grants Bob the action authorized for Alice, without Alice's consent and with no expiry ever having been checked. [6](#0-5)
