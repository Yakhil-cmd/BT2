### Title
Unbounded Loop Over `pool_member_epoch_balance` Trace in `calculate_rewards` Can Permanently Freeze Pool Member Rewards - (File: `src/pool/pool.cairo`)

---

### Summary

The `calculate_rewards` function in `src/pool/pool.cairo` iterates over a pool member's entire `pool_member_epoch_balance` trace without any bound. Because a delegator can grow this trace indefinitely by repeatedly changing their balance across epochs, the loop can eventually exceed Starknet's gas limit, permanently preventing the pool member from claiming rewards.

---

### Finding Description

`calculate_rewards` (line 837) contains an explicitly acknowledged unbounded `while` loop:

```cairo
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
    let pool_member_checkpoint = pool_member_trace.at(entry_to_claim_from);
    if pool_member_checkpoint.epoch() >= until_epoch {
        break;
    }
    ...
    entry_to_claim_from += 1;
}
``` [1](#0-0) 

The loop iterates over `self.pool_member_epoch_balance.entry(pool_member)` — a `PoolMemberBalanceTrace` stored as a `Vec` of checkpoints in contract storage. Each time a pool member changes their delegated balance in a new epoch, a new checkpoint is appended to this trace. The trace is never pruned or bounded. [2](#0-1) 

The trace grows via `set_member_balance` and `increase_member_balance`, which are called from:
- `enter_delegation_pool` (new member joins)
- `add_to_delegation_pool` (delegator adds stake)
- `exit_delegation_pool_intent` (delegator signals withdrawal)
- `enter_delegation_pool_from_staking_contract` (pool switch) [3](#0-2) [4](#0-3) 

`calculate_rewards` is called from two public entry points:

1. `claim_rewards` (line 335) — the primary reward-claiming path for any pool member or their reward address.
2. `pool_member_info_v1` (line 528) — a view function that also triggers the loop. [5](#0-4) [6](#0-5) 

---

### Impact Explanation

Once the `pool_member_epoch_balance` trace for a given pool member grows large enough, every call to `claim_rewards` for that member will exceed the Starknet transaction gas limit and revert. The pool member's accrued rewards become permanently unclaimable — matching the **"Permanent freezing of unclaimed yield"** impact category (High severity).

The `entry_to_claim_from` cursor is saved in `pool_member_info` and advanced on each successful `claim_rewards` call, but if the trace grows faster than rewards are claimed (or if the member never claims), the backlog becomes unprocessable. [7](#0-6) 

---

### Likelihood Explanation

A delegator can call `add_to_delegation_pool` and `exit_delegation_pool_intent` in alternating epochs, each call appending a new checkpoint. Since epochs are time-bounded and there is no minimum interval between balance changes beyond epoch boundaries, a sufficiently active delegator (or one who simply never claims rewards over a long period) will accumulate a large trace. The code itself acknowledges the loop is unbounded (line 858), confirming the developers are aware of the risk but have not mitigated it. No privileged access is required — any unprivileged delegator can trigger this condition on their own account. [8](#0-7) 

---

### Recommendation

Replace the unbounded iteration with a paginated or checkpoint-based claim mechanism. Specifically:

1. **Paginated claiming**: Allow `claim_rewards` to accept a `max_iterations` parameter and process only that many trace entries per call, saving the cursor (`entry_to_claim_from`) between calls. The current `entry_to_claim_from` field in `pool_member_info` already supports this pattern.
2. **Trace pruning**: After rewards are claimed up to `entry_to_claim_from`, remove or compact processed entries from the trace.
3. **Enforce a cap**: Limit the number of balance-change checkpoints per member per epoch to prevent unbounded growth.

---

### Proof of Concept

1. Delegator `A` calls `enter_delegation_pool` to join the pool (epoch 1). Trace length = 1.
2. Each subsequent epoch, `A` alternates between `add_to_delegation_pool` (small amount) and `exit_delegation_pool_intent` (same amount). Each call in a new epoch appends one checkpoint. After `N` epochs, trace length = `N`.
3. `A` never calls `claim_rewards` during this period.
4. After sufficient epochs (e.g., thousands), `A` calls `claim_rewards`. The `calculate_rewards` loop must iterate over all `N` checkpoints in a single transaction.
5. When `N` exceeds the gas limit threshold, the transaction reverts. All of `A`'s accrued rewards are permanently frozen because every future `claim_rewards` call will also revert. [9](#0-8)
