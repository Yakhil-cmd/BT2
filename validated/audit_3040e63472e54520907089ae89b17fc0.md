### Title
Unauthenticated Invoke Admission via `skip_stateful_validations` Bypasses Signature Check for Any Address with a Pending Deploy-Account — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The gateway's UX-skip path (`skip_stateful_validations`) suppresses the `__validate__` entry-point call for any Invoke transaction whose nonce is 1 and whose sender address appears in the mempool — regardless of whether the caller is the legitimate account owner. Any unprivileged observer can exploit this to inject an Invoke with a garbage signature from a victim's address the moment the victim's `deploy_account` lands in the mempool, causing the victim's own Invoke to be rejected as a duplicate nonce and the victim's account nonce to be bumped on-chain.

### Finding Description

`skip_stateful_validations` in `crates/apollo_gateway/src/stateful_transaction_validator.rs` decides whether to skip the blockifier's `__validate__` call: [1](#0-0) 

The skip fires when three conditions are simultaneously true:

1. The transaction is an `Invoke`.
2. `tx.nonce() == Nonce(Felt::ONE)` and `account_nonce == Nonce(Felt::ZERO)`.
3. `mempool_client.account_tx_in_pool_or_recent_block(sender_address)` returns `true`.

The third condition is implemented as: [2](#0-1) 

`account_tx_in_pool_or_recent_block` returns `true` if **any** transaction from the address is in the pool — it does not filter by transaction type. The code comment claims this is sufficient because "it means that either it has a deploy_account transaction or transactions with future nonces that passed validations," but this reasoning does not hold against an external attacker who can observe the mempool.

When `skip_validate` is `true`, `run_validate_entry_point` sets `execution_flags.validate = false`: [3](#0-2) 

Inside `StatefulValidatorTrait::validate` → `perform_validations`, the Invoke branch returns immediately without calling `__validate__` when `execution_flags.validate` is `false`: [4](#0-3) 

The `validate_by_mempool` call that precedes `skip_stateful_validations` only checks nonce, tip, and gas price — it never inspects the signature: [5](#0-4) 

### Impact Explanation

**Concrete attack sequence:**

1. Alice broadcasts a valid `deploy_account` for address `A`. It enters the mempool; `account_tx_in_pool_or_recent_block(A)` now returns `true`.
2. Attacker submits `Invoke { sender: A, nonce: 1, signature: [garbage] }`.
3. Gateway evaluates: Invoke, nonce=1, account_nonce=0, address in pool → `skip_validate = true` → `__validate__` is **not called** → transaction is admitted to the mempool.
4. Alice's own legitimate `Invoke { sender: A, nonce: 1 }` is now rejected by the mempool as a duplicate nonce (or displaced via fee-escalation rules).
5. At execution time the batcher runs `__validate__` on the attacker's Invoke; it fails (garbage signature). The transaction reverts, but Alice's nonce is incremented to 2 and a fee may be charged from her balance.
6. Alice must resubmit with nonce 2, paying an extra fee and losing the original transaction slot.

This satisfies **"High — Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing"**: an Invoke with an invalid signature is admitted, and the victim's valid Invoke is rejected.

### Likelihood Explanation

The attack requires only:
- Monitoring the public mempool for `deploy_account` transactions (trivially observable).
- Submitting a single Invoke with an arbitrary (garbage) signature before the victim submits their own Invoke.

No privileged access, no special keys, and no on-chain funds are required. The race window is the entire time Alice's `deploy_account` sits in the mempool before she submits her Invoke.

### Recommendation

Replace the coarse `account_tx_in_pool_or_recent_block` check with a type-specific query that confirms a **`deploy_account`** transaction (not just any transaction) is pending for the address. For example, expose a `deploy_account_in_pool(address)` method on the mempool that inspects the transaction type, and use that in `skip_stateful_validations` instead of the generic presence check.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
