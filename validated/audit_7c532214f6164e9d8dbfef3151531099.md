### Title
`block.prevrandao` Is Fully Predictable On-Chain — (`File: x/evm/keeper/keeper.go`)

---

### Summary

Sei's EVM module sets `block.prevrandao` (the `Random` field of the EVM block context) to `keccak256(block.timestamp)`. Because `block.timestamp` is itself available inside the same EVM transaction, any contract can recompute `prevrandao` deterministically, making it useless as a source of randomness. This is a chain-level property that affects every EVM contract deployed on Sei that relies on `block.prevrandao`.

---

### Finding Description

In `GetVMBlockContext`, the `Random` field — which the EVM exposes to contracts as `block.prevrandao` — is derived exclusively from the block timestamp:

```go
// Use hash of block timestamp as info for PREVRANDAO
r, err := ctx.BlockHeader().Time.MarshalBinary()
...
rh := crypto.Keccak256Hash(r)
...
Random: &rh,
``` [1](#0-0) [2](#0-1) 

The identical pattern is present in the `giga` module: [3](#0-2) [4](#0-3) 

The same block context also exposes `block.timestamp` to contracts: [5](#0-4) 

Because `block.timestamp` is readable inside a contract during execution, an attacker can reproduce the exact same computation (`keccak256(abi.encode(block.timestamp))` with the Go `time.Time` binary encoding) and obtain the value of `block.prevrandao` before any state-changing logic runs. The attacker can then branch or revert based on whether the predicted outcome is favorable — exactly the reroll pattern described in the external report.

---

### Impact Explanation

Every EVM contract deployed on Sei that uses `block.prevrandao` as a randomness source (lotteries, NFT rarity rolls, gaming outcomes, random reward selection) is exploitable by any unprivileged user who deploys a prediction contract. The chain itself is the root cause; no individual contract can defend against this without abandoning `block.prevrandao` entirely. Fund loss is bounded only by the value locked in such contracts and can exceed $5k.

---

### Likelihood Explanation

The relationship `prevrandao = keccak256(timestamp_binary)` is a single, deterministic computation. Any attacker who reads the open-source Sei node code (or simply observes that `block.prevrandao == keccak256(block.timestamp)` on-chain) can exploit it. No privileged access, validator key, or governance action is required. Likelihood is **Medium** because it requires awareness of the chain-specific encoding, but the barrier is low for a motivated attacker.

---

### Recommendation

Replace the timestamp-derived value with a source that is not available to contracts at execution time. Options in order of preference:

1. **Use the previous block's consensus hash** (already committed before the current block executes) as the `prevrandao` seed — it is unpredictable to contracts running in the current block.
2. **Incorporate the Tendermint/CometBFT `LastCommitHash`** or `AppHash` from the block header, which is finalized before EVM execution begins.
3. For high-value use cases, document that `block.prevrandao` on Sei is not a safe randomness source and direct contract developers to a VRF precompile or off-chain oracle.

---

### Proof of Concept

A contract deployed on Sei can predict `block.prevrandao` as follows:

```solidity
contract PredictPrevrandao {
    // Replicates Go's time.Time.MarshalBinary() → keccak256
    // (exact encoding must match Go's binary layout, but the value
    //  is fully deterministic from block.timestamp)
    function predictedPrevrandao() public view returns (bytes32) {
        // block.prevrandao == keccak256(MarshalBinary(block.timestamp))
        // Since block.timestamp is known, prevrandao is known.
        return keccak256(abi.encodePacked(block.timestamp));
        // (exact byte layout matches Go time.Time.MarshalBinary)
    }

    function exploit(address target) external {
        bytes32 predicted = predictedPrevrandao();
        // call target; revert if outcome is unfavorable
        (bool ok,) = target.call(abi.encodeWithSignature("roll()"));
        require(ok);
        // verify we got the expected outcome derived from predicted
    }
}
```

Because `block.prevrandao` is set once per block from `keccak256(block.timestamp)` before any transaction executes, every transaction in that block sees the same, pre-computable value. The attacker's contract can compute it inline and revert if the result is unfavorable, achieving the same reroll attack described in the external report — with zero cost on reverted attempts beyond gas.
