### Title
Permanent Freezing of EVM-Side ETH/ERC-20 Funds When `error_refund` Feature Is Disabled and NEAR-Side Transfer Fails — (`engine-precompiles/src/native.rs`, `engine/src/contract_methods/connector.rs`)

---

### Summary

When the `error_refund` Cargo feature is disabled (which is the case for any build that does not explicitly opt in, since it is not in the `default` feature set), the exit-to-NEAR precompile permanently burns EVM-side ETH or ERC-20 tokens with no recovery path if the downstream NEAR-side promise fails.

---

### Finding Description

**Feature flag gating of `refund` field**

In `engine-precompiles/src/native.rs`, the `ExitToNearPrecompileCallbackArgs` struct is populated with `refund: None` unconditionally when `error_refund` is disabled: [1](#0-0) 

The `error_refund` feature is not listed under `default` in either `engine-precompiles/Cargo.toml` or `engine/Cargo.toml`: [2](#0-1) [3](#0-2) 

**Path A — Base ETH exit (no callback scheduled at all)**

For a base-ETH exit (flag `0x00`, non-wNEAR), `exit_base_token_to_near` returns `transfer_near_args = None`: [4](#0-3) 

With `error_refund` disabled, `callback_args` becomes `ExitToNearPrecompileCallbackArgs { refund: None, transfer_near: None }`, which equals `Default::default()`. The promise-scheduling branch therefore emits only a bare `PromiseArgs::Create` with no callback: [5](#0-4) 

If `ft_transfer` on the connector contract fails (e.g., recipient has no storage deposit), **no callback is ever invoked**. The ETH was already deducted from the caller's EVM balance at precompile execution time and is permanently gone.

**Path B — wNEAR unwrap (callback scheduled, but `refund` branch is dead)**

For the wNEAR unwrap path, `transfer_near_args` is `Some(...)`: [6](#0-5) 

This makes `callback_args != default()`, so a callback to `exit_to_near_precompile_callback` IS scheduled. However, inside that callback, when the `near_withdraw` promise fails, the code reaches: [7](#0-6) 

Because `args.refund` is `None` (compiled out), the `else if` branch is skipped and the function returns `Ok(None)` — silently discarding the failure. The wNEAR that was burned on the EVM side is never restored.

**Path C — ERC-20 exit (same as Path A)**

For a non-wNEAR ERC-20 exit, `transfer_near_args` is also `None`: [8](#0-7) 

Same outcome as Path A: no callback, no refund, ERC-20 tokens burned permanently.

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

The EVM deducts the caller's ETH (or burns ERC-20 tokens) at precompile execution time. If the NEAR-side NEP-141 transfer subsequently fails, there is no code path to restore the EVM balance. The tokens are destroyed on the EVM side and never credited on the NEAR side.

---

### Likelihood Explanation

- `error_refund` is **not** a default feature. Any deployment that does not explicitly pass `--features error_refund` at build time is affected.
- The `ft_transfer` call can fail for ordinary, non-adversarial reasons: the recipient account has no storage deposit on the connector contract, the connector is paused, or the recipient account does not exist. An attacker can deliberately trigger this by sending ETH to the exit precompile with a recipient that has no storage deposit.
- The attacker does not need any privileged access; the exit precompile is callable by any EVM transaction.

---

### Recommendation

1. Add `error_refund` to the `default` feature set in both `engine/Cargo.toml` and `engine-precompiles/Cargo.toml`, or remove the feature flag entirely and always populate `refund`.
2. In `exit_to_near_precompile_callback`, treat a `None` refund on a failed promise as an error (return `Err`) rather than silently returning `Ok(None)`.
3. Alternatively, always schedule a callback (even when `transfer_near` is `None`) so that a failed `ft_transfer` is always caught and can trigger a refund or at minimum an on-chain error.

---

### Proof of Concept

1. Compile the engine **without** `--features error_refund` (the default build).
2. Deploy the engine locally (NEAR sandbox).
3. From an EVM account with balance, call the exit-to-NEAR precompile (`0xe9217bc70b7ed1f598ddd3199e80b093fa71124f`) with flag `0x00` and a recipient NEAR account that has **no storage deposit** on the connector contract.
4. Observe that `ft_transfer` fails on the NEAR side (storage deposit required).
5. Assert: the EVM sender's balance is unchanged (ETH is gone), the recipient's NEP-141 balance is zero, and no `exit_to_near_precompile_callback` was ever invoked.
6. For Path B: repeat with wNEAR ERC-20 and the `:unwrap` suffix; observe the callback is invoked but returns `Ok(None)` without restoring the EVM balance.
