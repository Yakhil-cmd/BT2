### Title
Block-List Destination Check Bypassed via EVM Contract Intermediary - (`app/block_address.go`, `app/proposal.go`)

### Summary
Cronos enforces a block-list that prevents transactions from being sent to or from blocked addresses. However, the enforcement only inspects the **outer EVM transaction's `To` field** and the **tx signer**. It does not inspect EVM internal calls generated during contract execution. An unprivileged attacker can deploy an intermediary contract and route funds to a blocked destination address through it, fully bypassing the block-list destination check.

### Finding Description

The block-list enforcement is implemented in two places:

**1. `BlockAddressesDecorator.AnteHandle` (`app/block_address.go`)**

The decorator only runs during `ctx.IsCheckTx()` and checks:
- The Cosmos tx signers
- The outer EVM tx `ethTx.To()` address [1](#0-0) 

**2. `ProposalHandler.ValidateTransaction` (`app/proposal.go`)**

The proposal handler checks the same two fields — tx signers and the outer EVM `ethTx.To()` — before including a tx in a block: [2](#0-1) 

Neither check inspects EVM **internal calls** (sub-calls generated during contract execution). The EVM execution itself happens after the ante-handler and proposal phases, and there is no EVM execution hook that validates internal call destinations against the block-list.

The `send_to_ibc` / `send_to_evm_chain` EVM hooks only handle specific bridge events; they do not enforce the block-list on general EVM transfers: [3](#0-2) 

### Impact Explanation

An attacker can bypass the block-list **destination** restriction by routing funds through an intermediary contract:

1. Admin blocks address `B` (e.g., a sanctioned address).
2. Attacker deploys contract `C` (not on the block-list).
3. Attacker sends an EVM tx with `To = C` (passes the block-list check since `C` is not blocked).
4. Contract `C` internally calls `B.transfer(amount)` or uses a low-level `CALL` to send CRO/ERC20 tokens to `B`.
5. `B` receives funds despite being on the block-list.

This is a **bypass of the Cronos block-list authorization check**, which falls under the High impact category: *"Bypass of Cronos admin, governance authority, permission, token-mapping, bridge, block-list, or module-account authorization checks."*

The block-list is a compliance/sanctions mechanism. Its purpose is to prevent blocked addresses from receiving or sending assets. The destination bypass allows any blocked address to receive CRO or CRC20/CRC21 tokens through a trivially-deployed intermediary contract.

### Likelihood Explanation

- No special privileges are required. Any unprivileged user can deploy a contract on Cronos.
- The bypass requires only a single contract deployment and a standard EVM call.
- This is the exact wrapping pattern described in the external report (H-02) and has been observed in practice with veToken models and other on-chain restriction systems.
- The `BlockAddressesDecorator` only runs during `IsCheckTx()`, meaning the check is not even enforced at execution time for the outer tx, let alone for internal calls. [4](#0-3) 

### Recommendation

Enforce the block-list at EVM execution time, not only at the mempool/proposal layer. Options include:

1. **EVM execution hook**: Add a `StateDB` or EVM tracer hook that intercepts every `CALL`/`STATICCALL`/`DELEGATECALL` and checks both the caller and callee against the block-list, reverting if either is blocked.
2. **Precompile-level enforcement**: Extend the existing `checkBlockedAddr` logic in the bank precompile to also cover the Cosmos-level block-list (not just `bankKeeper.BlockedAddr`).
3. **CRC20/CRC21 transfer hook**: For CRC20/CRC21 tokens, add a transfer hook that checks the recipient against the block-list before completing any transfer. [5](#0-4) 

### Proof of Concept

```solidity
// Attacker deploys this contract (address C, not on block-list)
contract BlocklistBypass {
    // blockedAddr = address B (on the Cronos block-list)
    address public blockedAddr;
    constructor(address _blocked) { blockedAddr = _blocked; }

    // Attacker calls this with value; C is the outer tx To (not blocked)
    // C internally sends CRO to the blocked address B
    function relay() external payable {
        (bool ok,) = blockedAddr.call{value: msg.value}("");
        require(ok);
    }
}
```

**Attack flow**:
1. Admin adds `B` to the Cronos block-list via `MsgStoreBlockList`.
2. Attacker deploys `BlocklistBypass(B)` → contract `C`.
3. Attacker sends EVM tx: `To = C`, `value = X CRO`, calls `relay()`.
4. `BlockAddressesDecorator` checks `To = C` → not blocked → tx passes.
5. `ProposalHandler.ValidateTransaction` checks `To = C` → not blocked → tx included in block.
6. EVM executes `relay()`: internal `CALL` to `B` with `X CRO`.
7. `B` receives `X CRO` despite being on the block-list. [6](#0-5) [2](#0-1)

		if sigTx, ok := tx.(signing.SigVerifiableTx); ok {
			signers, err := sigTx.GetSigners()
			if err != nil {
				return ctx, err

				if _, ok := bad.blockedMap[sdk.AccAddress(signer).String()]; ok {
					return ctx, fmt.Errorf("signer is blocked: %s", sdk.AccAddress(signer).String())
				}
			}
		}
