### Title
Signature Validation Bypass via Overly Broad `skip_stateful_validations` Check Admits Unsigned Invoke Transactions to Mempool - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful transaction validator skips the account's `__validate__` entry point (the Starknet-native signature check) for any invoke transaction with `nonce=1` when `account_tx_in_pool_or_recent_block` returns `true`. Because that helper returns `true` for **any** pending transaction from the account — not exclusively a `deploy_account` — an attacker can submit an invoke with an arbitrary/invalid signature for any account that has any pending mempool entry, and the gateway will admit it without ever calling `__validate__`.

### Finding Description

The UX feature is implemented in `skip_stateful_validations`: [1](#0-0) 

The three conditions that trigger the skip are:
1. Transaction type is `Invoke`
2. `tx.nonce() == Nonce(Felt::ONE)`
3. `account_nonce == Nonce(Felt::ZERO)` (account not yet committed on-chain)

When all three hold, the function delegates to `account_tx_in_pool_or_recent_block`: [2](#0-1) 

This returns `true` if the account has **any** transaction in the pool or any committed nonce in the mempool's recent-block history — it does not filter for `deploy_account` transactions. The code comment acknowledges this:

> "It is sufficient to check if the account exists in the mempool since it means that either it has a deploy_account transaction **or transactions with future nonces that passed validations**."

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` is called with `validate: false`: [3](#0-2) 

This means `StatefulValidator::perform_validations` exits before calling `tx.validate_tx(...)` — the account's `__validate__` entry point, which is the sole on-chain signature verifier for Starknet accounts: [4](#0-3) 

**Concrete attack path:**

1. Account `A` has a pending `deploy_account` (nonce=0) in the mempool; `account_nonce` in committed state is `0`.
2. Any third party submits an `Invoke` for `A` with `nonce=1` and a completely invalid/arbitrary signature.
3. Gateway calls `run_pre_validation_checks`:
   - `validate_nonce`: `0 ≤ 1 ≤ 200` — passes.
   - `validate_by_mempool`: nonce not too old — passes.
   - `skip_stateful_validations`: `nonce==1`, `account_nonce==0`, `account_tx_in_pool_or_recent_block(A)==true` → returns `true`.
4. `run_validate_entry_point` is invoked with `validate: false`; `__validate__` is never called.
5. The invalid invoke is forwarded to the mempool and accepted.

The same path applies when `A` has a valid nonce=0 invoke (not a deploy_account) already in the pool: `account_tx_in_pool_or_recent_block` still returns `true`, triggering the skip for a nonce=1 invoke with an invalid signature.

### Impact Explanation

The gateway's admission invariant — that every accepted transaction has passed its account's `__validate__` entry point — is broken for `Invoke` transactions with `nonce=1` whenever the sender account has any pending mempool entry. Invalid-signature transactions enter the mempool and are only rejected later by the blockifier during block execution. Because `__validate__` failures in Starknet do not charge fees, an attacker can flood the mempool with signature-invalid transactions at zero cost, consuming mempool capacity and batcher CPU.

This maps to: **High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

### Likelihood Explanation

The mempool is public; any observer can enumerate accounts with pending `deploy_account` transactions. Submitting a nonce=1 invoke with a garbage signature requires no special privilege and costs nothing (the transaction is rejected fee-free at execution). The attack is therefore trivially repeatable.

### Recommendation

Replace the broad `account_tx_in_pool_or_recent_block` check with a type-specific query that confirms the pending mempool entry is a `deploy_account` transaction for the same address. Alternatively, add a dedicated mempool API `has_pending_deploy_account(address) -> bool` and use that in `skip_stateful_validations`. The skip should only fire when a genuine `deploy_account` is pending, not for arbitrary transaction types.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
