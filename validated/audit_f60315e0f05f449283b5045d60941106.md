### Title
`evmAddress` Excluded from Signed Payload in COA Ownership Proof Enables Cross-COA Signature Replay - (File: `fvm/evm/stdlib/contract.cdc`)

---

### Summary

`EVM.validateCOAOwnershipProof` verifies that a set of Cadence account key signatures are valid over `signedData`, then separately checks that the COA resource at the given `path` has the expected `evmAddress`. Because `evmAddress` is never required to be encoded inside `signedData`, a valid signature produced to prove ownership of one COA can be replayed verbatim to prove ownership of any other COA belonging to the same Cadence account. This is the direct Flow analog of the ChainportBridge bug where `airdropAmount` was excluded from the signed payload, letting the caller substitute any value.

---

### Finding Description

`EVM.validateCOAOwnershipProof` in `fvm/evm/stdlib/contract.cdc` accepts six parameters: `address`, `path`, `signedData`, `keyIndices`, `signatures`, and `evmAddress`. The cryptographic verification step at line 1082–1086 checks only that the provided signatures are valid over `signedData`:

```cadence
let isValid = keyList.verify(
    signatureSet: signatureSet,
    signedData: signedData,
    domainSeparationTag: "FLOW-V0.0-user"
)
```

The `evmAddress` parameter is checked only structurally — it is compared against the bytes of the COA resource found at `path` (lines 1097–1105) — but it is never required to appear inside `signedData`. The code itself documents this gap:

> "Note: this function does not enforce that `signedData` includes `evmAddress`. In principle, a signature produced for one purpose could be replayed here against a different COA owned by the same Cadence account."

Because `validateCOAOwnershipProof` is declared `access(all)`, any unprivileged Cadence script or transaction can call it with an arbitrary combination of `(signedData, signatures, evmAddress)`. An attacker who obtains a valid signature from a victim — produced for any purpose, over any `signedData` — can substitute a different `evmAddress` (pointing to a different COA stored at a different `path` on the same account) and the function will return `isValid: true`. ### Impact Explanation

Any off-chain or on-chain system that calls `EVM.validateCOAOwnershipProof` directly from Cadence and uses the result to gate access to EVM-side assets or permissions tied to a specific COA address can be bypassed. A concrete example:

- A DeFi protocol asks a user to sign a challenge to prove they own EVM address `0xAAA` (COA_A) before releasing escrowed tokens.
- The user's Cadence account also holds COA_B at `0xBBB`.
- An attacker replays the user's signature with `evmAddress = 0xBBB` and `path = /public/coa_b`, obtaining a valid proof for COA_B without the user's consent for that address.
- The attacker can now impersonate the user as the owner of COA_B in any system relying on `validateCOAOwnershipProof`.

The EVM precompile path (`verifyCOAOwnershipProof` called from Solidity via `cadenceArch.staticcall`) is partially mitigated because the COA contract always passes `address(this)` as the EVM address, making it caller-controlled at the EVM level. However, the Cadence-level function is fully open and the parameter is entirely attacker-controlled there. [4](#0-3) [5](#0-4) 

---

### Likelihood Explanation

- `EVM.validateCOAOwnershipProof` is `access(all)` — reachable by any unprivileged Cadence transaction or script with no special role required.
- The attacker only needs a previously-observed valid signature from the victim (e.g., one published on-chain, broadcast off-chain, or obtained from a prior proof request).
- The attack is more likely as multi-COA accounts become common (e.g., protocols that create one COA per vault type).
- No staked node, admin key, or quorum compromise is required. [6](#0-5) 

---

### Recommendation

Require that `signedData` encodes `evmAddress` as part of the signed payload before accepting the proof as valid. Concretely, inside `validateCOAOwnershipProof`, add a pre-condition that asserts the 20-byte `evmAddress` is present within `signedData` (e.g., as a suffix or a structured encoding), or document and enforce a canonical signed-data format that binds the proof to a specific EVM address. This mirrors the ChainportBridge fix of including `airdropAmount` in the signature payload. [2](#0-1)

### Proof of Concept

**Setup**: Cadence account `Alice` owns two COAs:
- COA_A stored at `/public/coa_a`, EVM address `0xAAAA...`
- COA_B stored at `/public/coa_b`, EVM address `0xBBBB...`

**Step 1** — Alice legitimately proves ownership of COA_A by signing `signedData = 0xDEADBEEF` with her account key and publishing the proof.

**Step 2** — Attacker constructs a Cadence script:

```cadence
import EVM from <EVM_CONTRACT_ADDRESS>

access(all)
fun main(): EVM.ValidationResult {
    // Reuse Alice's signedData and signatures, but swap in COA_B's address and path
    return EVM.validateCOAOwnershipProof(
        address: <Alice's Cadence address>,
        path: /public/coa_b,           // COA_B's capability path
        signedData: [0xDE, 0xAD, 0xBE, 0xEF],  // same signedData Alice signed
        keyIndices: [<Alice's key index>],
        signatures: [<Alice's signature over 0xDEADBEEF>],  // same signature
        evmAddress: [0xBB, 0xBB, ...]  // COA_B's EVM address
    )
}
```

**Step 3** — The function verifies the signature over `signedData` (valid, Alice signed it), then checks that the COA at `/public/coa_b` has address `0xBBBB...` (true). It returns `ValidationResult(isValid: true, problem: nil)`.

**Result**: The attacker has obtained a valid proof that Alice owns COA_B, using only a signature Alice produced for COA_A. No key compromise, no admin access, no staked node required. [1](#0-0) [7](#0-6) [8](#0-7)
