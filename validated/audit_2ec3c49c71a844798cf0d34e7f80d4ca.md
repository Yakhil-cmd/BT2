### Title
Burned ERC-20 Tokens Permanently Lost When Exit-to-NEAR Promise Fails Without Refund Path — (`engine-precompiles/src/native.rs`, `engine/src/contract_methods/connector.rs`)

---

### Summary

The `ExitToNear` precompile burns ERC-20 tokens (an irreversible EVM state change) and then schedules a NEAR-side `ft_transfer` or `near_withdraw` promise. When the `error_refund` compile feature is absent, no callback is attached for regular ERC-20 exits, and for the wNEAR-unwrap path the attached callback has `refund: None`. If the NEAR-side promise fails, the burned ERC-20 tokens are permanently unrecoverable — the EVM state is already committed and no re-mint is ever triggered.

---

### Finding Description

**Step 1 — ERC-20 tokens are burned (irreversible EVM state change).**
A user calls the ERC-20 contract's burn/withdraw function. The ERC-20 contract reduces the user's balance and calls the `ExitToNear` precompile. By the time the precompile runs, the EVM state change (token burn) is already in the pending apply set and will be committed when the transaction finalises.

**Step 2 — NEAR promise is scheduled with no failure handler.**
Inside `ExitToNear::run`, the precompile builds a `PromiseArgs` value and emits it as a log for the engine to dispatch:

```rust
// engine-precompiles/src/native.rs  lines 449-483
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,          // ← always None without the feature
    transfer_near: transfer_near_args,
};

let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // ← no callback at all
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs { base: transfer_promise, callback: … })
};
```

Without `error_refund`:
- **Regular ERC-20 exit** (`transfer_near = None`, `refund = None`): `callback_args == default()` → `PromiseArgs::Create` → **no callback is ever attached**. If `ft_transfer` fails on the NEP-141 side, there is no NEAR-level handler to re-mint the burned tokens.
- **wNEAR unwrap** (`transfer_near = Some(…)`, `refund = None`): a callback *is* attached (to forward the unwrapped NEAR), but when `near_withdraw` fails the callback reaches the `else { None }` branch and silently returns without issuing any refund.

**Step 3 — Callback silently discards the failure.**

```rust
// engine/src/contract_methods/connector.rs  lines 214-242
let maybe_result = if let Some(PromiseResult::Successful(_)) = handler.promise_result(0) {
    // success path – forward NEAR
    …
} else if let Some(args) = args.refund {   // ← None without error_refund
    engine::refund_on_error(…)
} else {
    None   // ← reached on failure; burned tokens are never re-minted
};
```

The burned ERC-20 tokens are gone. The user receives neither NEAR nor a refund.

---

### Impact Explanation

**Critical — Permanent freezing of user funds.**

Any user who exits ERC-20 tokens (or wNEAR) to NEAR loses their entire bridged balance if the NEP-141-side promise fails. The EVM state (token burn) is committed before the NEAR promise executes; there is no compensating write-back. The loss is permanent and proportional to the amount being bridged.

---

### Likelihood Explanation

**Medium.** The NEP-141 `ft_transfer` or `near_withdraw` call can fail for several reasons reachable in normal operation:
- The NEP-141 contract is paused or rate-limited.
- The recipient account is not registered for storage on the NEP-141 contract.
- Insufficient gas is forwarded to the promise (the gas constants are hardcoded and may be stale).
- The wNEAR contract's `near_withdraw` reverts for any internal reason.

None of these require attacker privilege; they are ordinary operational failure modes. A malicious NEP-141 operator could also deliberately cause failures to grief users.

---

### Recommendation

1. **Always enable `error_refund` in production builds** and enforce it via the `contract` feature dependency so it cannot be accidentally omitted.
2. Alternatively, make the refund path unconditional: always attach a callback that re-mints the burned ERC-20 tokens (or restores the ETH balance) when the NEAR-side promise does not return `Successful`.
3. Audit the hardcoded gas constants (`FT_TRANSFER_GAS`, `FT_TRANSFER_CALL_GAS`, `WITHDRAWAL_GAS`) to ensure they remain sufficient as NEAR gas pricing evolves.

---

### Proof of Concept

```
1. User holds 1000 units of bridged-token ERC-20 on Aurora.
2. User calls erc20.burn(1000, "alice.near") → ERC-20 contract burns 1000 tokens
   and calls ExitToNear precompile.
3. Without `error_refund`, precompile emits PromiseArgs::Create(ft_transfer to NEP-141).
   No callback is registered.
4. EVM execution ends; EVM state committed: user balance = 0.
5. NEAR runtime calls ft_transfer on the NEP-141 contract.
6. NEP-141 contract reverts (e.g., alice.near not storage-registered).
7. NEAR runtime has no callback to invoke.
8. User's 1000 ERC-20 tokens are burned; 0 NEP-141 tokens are received; no refund.
   Funds are permanently frozen.
```

The wNEAR-unwrap variant follows the same path but with a callback that reaches `else { None }` at `engine/src/contract_methods/connector.rs` line 241 instead of calling `refund_on_error`.
