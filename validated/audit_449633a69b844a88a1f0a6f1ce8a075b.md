### Title
Missing Refund Callback in `ExitToEthereum::run` Causes Permanent ETH Loss on Withdraw Promise Failure — (`engine-precompiles/src/native.rs`)

### Summary

`ExitToEthereum::run` (flag `0x0`) deducts ETH from the caller's EVM balance, transfers it to the `exit_to_ethereum::ADDRESS` precompile address, and schedules a bare `PromiseArgs::Create` withdraw call to the NEP-141 eth connector with **no callback and no refund mechanism**. If the NEAR-side `withdraw` promise fails for any reason, the ETH is permanently frozen at the precompile address with no recovery path. This is structurally asymmetric with `ExitToNear`, which has a dedicated `exit_to_near_precompile_callback` that calls `refund_on_error` on failure.

### Finding Description

In `ExitToEthereum::run`, the ETH path (flag `0x0`) constructs exactly one promise:

```rust
let withdraw_promise = PromiseCreateArgs { ... method: "withdraw" ... };
let promise = borsh::to_vec(&PromiseArgs::Create(withdraw_promise)).unwrap();
``` [1](#0-0) 

This is always `PromiseArgs::Create` — never `PromiseArgs::Callback`. There is no `exit_to_ethereum_precompile_callback` function anywhere in the codebase (confirmed by exhaustive grep). The promise log is processed by `filter_promises_from_logs` in `engine.rs`, which schedules it as a fire-and-forget NEAR promise: [2](#0-1) 

Contrast this with `ExitToNear`, which — when the `error_refund` feature is enabled — wraps the transfer promise in a `PromiseArgs::Callback` pointing to `exit_to_near_precompile_callback`: [3](#0-2) 

That callback calls `engine::refund_on_error`, which for the ETH case transfers ETH back from `exit_to_near::ADDRESS` to the original sender: [4](#0-3) 

No equivalent exists for `ExitToEthereum`. The `error_refund` feature flag and `refund_call_args` are only wired into `ExitToNear`'s code path.

When a user calls `ExitToEthereum` with ETH value `V`, the EVM executor transfers `V` from the caller's balance to `exit_to_ethereum::ADDRESS`. The precompile then schedules the `withdraw` promise. If that promise fails (gas exhaustion, connector panic, storage issues, etc.), the ETH remains at `exit_to_ethereum::ADDRESS` permanently — there is no callback to detect the failure and no `refund_on_error` call to restore the balance.

The existing test suite explicitly acknowledges the analogous gap for `ExitToNear`: [5](#0-4) 

No equivalent test exists for `ExitToEthereum` because no refund path exists to test.

### Impact Explanation

ETH deducted from the user's EVM balance is transferred to `exit_to_ethereum::ADDRESS` inside the EVM state. If the NEAR-side `withdraw` promise fails, that ETH has no owner who can spend it and no callback to return it. The funds are permanently frozen. This matches **Critical — Permanent freezing of funds**.

### Likelihood Explanation

The `withdraw` call targets the NEP-141 eth connector contract. Realistic failure modes that do not require admin compromise include:

- **Gas exhaustion**: The NEAR transaction's prepaid gas is consumed before the promise executes or the promise itself runs out of gas.
- **Connector contract panic**: Any unhandled error in the connector's `withdraw` implementation causes a promise failure.
- **Storage deposit exhaustion**: The connector account lacks sufficient NEAR storage deposit to process the withdrawal.

These are non-hypothetical NEAR runtime behaviors. Any user who calls `ExitToEthereum` during a period of high gas contention or connector instability is at risk.

### Recommendation

Mirror the `ExitToNear` refund pattern for `ExitToEthereum`:

1. Add an `ExitToEthereumPrecompileCallbackArgs` struct (analogous to `ExitToNearPrecompileCallbackArgs`) carrying the refund address and amount.
2. Change `ExitToEthereum::run` to emit `PromiseArgs::Callback` with a callback pointing to a new `exit_to_ethereum_precompile_callback` method on the engine contract.
3. Implement `exit_to_ethereum_precompile_callback` in `engine/src/contract_methods/connector.rs` to call `engine::refund_on_error` (ETH branch, using `exit_to_ethereum::ADDRESS` as the source) when the promise result is not `Successful`.
4. Gate the refund args behind the `error_refund` feature flag, consistent with the `ExitToNear` pattern.

### Proof of Concept

```
1. User calls ExitToEthereum precompile (0xb0bd02f6...) with:
   - input = [0x00] ++ eth_recipient (21 bytes)
   - msg.value = V (e.g., 1 ETH)

2. ExitToEthereum::run executes:
   - EVM deducts V from caller balance, credits exit_to_ethereum::ADDRESS
   - Schedules PromiseArgs::Create { target: eth_connector, method: "withdraw", ... }
   - No callback is attached

3. NEAR runtime executes the withdraw promise; it fails (e.g., gas exhaustion)

4. No callback fires; no refund_on_error is called

5. Assert: caller EVM balance = original - V (decremented)
   Assert: exit_to_ethereum::ADDRESS EVM balance = V (frozen)
   Assert: no refund promise was scheduled
```

This can be reproduced locally by adapting the existing `test_exit_to_near_eth_refund` test: replace the `withdrawEthToNear` call with a direct call to `exit_to_ethereum::ADDRESS` with value, force the connector `withdraw` to fail (e.g., by draining the connector balance as the test already does for ExitToNear), and assert the user balance is not restored.
