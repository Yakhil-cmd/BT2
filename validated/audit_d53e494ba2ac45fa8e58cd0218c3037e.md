### Title
Unconditional `panic!` on Malformed EIP-6110 Deposit Event Data Causes Denial of Service - (File: `basic_bootloader/src/bootloader/block_flow/ethereum/eip_6110_deposit_events_parser/mod.rs`)

### Summary

The `eip6110_events_parser` function, called during Ethereum block post-processing, unconditionally panics when it encounters a deposit event whose ABI-encoded data fails structural validation. Because events are emitted by attacker-controlled transactions, a malicious actor who can emit an event from the deposit contract address with the correct topic but malformed data will crash the entire block execution, causing a denial of service.

### Finding Description

In `eip6110_events_parser`, every event emitted from `DEPOSIT_CONTRACT_ADDRESS` whose first topic matches `DEPOSIT_EVENT_SIGNATURE_HASH` is passed to `validate_and_write_event_data`. That helper returns `Err(())` when the 576-byte event payload contains incorrect ABI offset values (checked by `validate_u16_at_most`). Instead of propagating the error up the call stack, the caller unconditionally panics:

```rust
let Ok(_) = validate_and_write_event_data(event, requests_hasher, &mut logger) else {
    panic!("invalid deposit event structure");
};
``` [1](#0-0) 

The function signature already returns `Result<bool, SystemError>`, so the error could be propagated cleanly. The `panic!` is the only reason this path terminates the process rather than returning an error.

The `validate_and_write_event_data` function returns `Err(())` in two cases:

1. `data.len() != 576` — caught before the panic path, returns `Err(())` early.
2. Any `validate_u16_at_most` check fails — these verify that the five ABI pointer words in the payload equal the expected offsets (160, 256, 320, 384, 512). A payload that is exactly 576 bytes but carries wrong pointer values passes the length guard and reaches the `validate_u16_at_most` calls, which return `Err(())`, which propagates via `?` back to the caller, triggering the `panic!`. [2](#0-1) 

### Impact Explanation

The `eip6110_events_parser` is invoked during Ethereum block post-processing (after all transactions in the block have executed). A panic at this point aborts the entire block execution. In the forward (sequencer) execution mode this crashes the sequencer process; in the proving mode it aborts proof generation. Either outcome is a complete denial of service for the affected block and all subsequent blocks until the node is restarted.

### Likelihood Explanation

The deposit contract address is `0x00000000219ab540356cbb839cbe05303d7705fa`. If this address is not pre-deployed in ZKsync OS's Ethereum mode (i.e., no bytecode is stored there at genesis), an attacker can deploy an arbitrary contract to that address and emit a `LOG1` event with topic `DEPOSIT_EVENT_SIGNATURE_HASH` and a 576-byte payload whose ABI pointer words differ from the expected values. This is a single, cheap transaction. Even if the deposit contract is pre-deployed, the `panic!` represents an unconditional crash path in production code that processes externally-supplied data, violating the principle that malformed input must never cause a panic.

### Recommendation

Replace the `panic!` with proper error propagation. The enclosing function already returns `Result<bool, SystemError>`, so the fix is straightforward:

```rust
// Before (panics on malformed data):
let Ok(_) = validate_and_write_event_data(event, requests_hasher, &mut logger) else {
    panic!("invalid deposit event structure");
};

// After (propagates error):
validate_and_write_event_data(event, requests_hasher, &mut logger)
    .map_err(|_| internal_error!("invalid deposit event structure"))?;
```

More broadly, audit all `panic!` and `.unwrap()` call sites in production (non-test) code paths that process data derived from transactions or events, and replace them with explicit `Result`-based error handling.

### Proof of Concept

1. In ZKsync OS's Ethereum mode, deploy a contract at `DEPOSIT_CONTRACT_ADDRESS` (`0x00000000219ab540356cbb839cbe05303d7705fa`) if it is not pre-deployed.
2. From that contract, emit a `LOG1` event with:
   - Topic 0 = `DEPOSIT_EVENT_SIGNATURE_HASH` (`0x649bbc62d0e31342afea4e5cd82d4049e7e1ee912fc0889aa790803be39038c5`)
   - Data = exactly 576 bytes where the first 32-byte word is **not** `0x00…00a0` (i.e., the ABI offset for the first field is wrong, e.g., all zeros).
3. Include this transaction in a block submitted to ZKsync OS.
4. During `eip6110_events_parser` post-processing, `validate_and_write_event_data` returns `Err(())` because `validate_u16_at_most(&chunk, 160)` fails.
5. The `else { panic!("invalid deposit event structure") }` branch executes, crashing the node. [3](#0-2)

    requests_hasher: &mut impl crypto::sha256::Digest,

    S::IO: IOSubsystemExt + IOTeardown<S::IOTypes>,
{
    // we can not easily get the number from one scan, so we will accumulate into hasher directly

    let mut event_encountered = false;
    let mut logger = system.get_logger();
    for event in system.io.events_iterator() {
        if event.address != &DEPOSIT_CONTRACT_ADDRESS {
            continue;
        }
        if event.topics.len() > 0 && event.topics[0] == DEPOSIT_EVENT_SIGNATURE_HASH {
            if event_encountered == false {

                panic!("invalid deposit event structure");
            };

fn validate_and_write_event_data(
    event: GenericEventContentWithTxRef<'_, MAX_EVENT_TOPICS, EthereumIOTypesConfig>,
    requests_hasher: &mut impl crypto::sha256::Digest,
    logger: &mut impl Logger,
) -> Result<(), ()> {
    let data = event.data;
    if data.len() != 576 {
        return Err(());
    }
    let mut chunks = data.as_chunks::<32>().0.iter();
    validate_u16_at_most(chunks.next().unwrap(), 160)?;
    validate_u16_at_most(chunks.next().unwrap(), 256)?;
    validate_u16_at_most(chunks.next().unwrap(), 320)?;
    validate_u16_at_most(chunks.next().unwrap(), 384)?;
    validate_u16_at_most(chunks.next().unwrap(), 512)?;
    drop(chunks);

    validate_u16_at_most(data[160..192].try_into().unwrap(), 48)?;
    validate_u16_at_most(data[256..288].try_into().unwrap(), 32)?;
    validate_u16_at_most(data[320..352].try_into().unwrap(), 8)?;
    validate_u16_at_most(data[384..416].try_into().unwrap(), 96)?;
    validate_u16_at_most(data[512..544].try_into().unwrap(), 8)?;
