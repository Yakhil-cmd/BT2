### Title
`skip_stateful_validations` Bypasses `__validate__` for Already-Deployed Accounts via Stale State Reader and Committed-Account Membership — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function is designed to skip the `__validate__` entry point for an invoke transaction with `nonce=1` when the account has not yet been deployed (i.e., a `deploy_account` tx is pending). It delegates the "is the deploy pending?" check to `account_tx_in_pool_or_recent_block`. However, that function returns `true` for accounts that have **already been committed to a block** (i.e., already deployed), not only for accounts with a pending `deploy_account` transaction. When the gateway's state reader is stale — showing `account_nonce=0` for an account that was deployed in the most recently committed block — the condition `tx.nonce() == 1 && account_nonce == 0` is satisfied, and `account_tx_in_pool_or_recent_block` returns `true` because the account is in the mempool's `committed` map. The result is that `__validate__` is skipped for an already-deployed account, admitting an invoke transaction with an invalid signature to the mempool without any signature verification.

---

### Finding Description

**Step 1 — The UX skip-validate feature.**

`skip_stateful_validations` in `crates/apollo_gateway/src/stateful_transaction_validator.rs` is the gateway-side implementation of the "deploy_account + invoke in the same batch" UX feature:

```rust
async fn skip_stateful_validations(...) -> ... {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await ...;
        }
    }
    Ok(false)
}
``` [1](#0-0) 

When `skip_validate=true`, `run_validate_entry_point` sets `validate: false`, and `StatefulValidator::perform_validations` returns `Ok(())` immediately after `perform_pre_validation_stage` — **without calling `__validate__`**:

```rust
if !tx.execution_flags.validate {
    return Ok(());
}
``` [2](#0-1) 

**Step 2 — `account_tx_in_pool_or_recent_block` returns `true` for committed (deployed) accounts.**

The mempool implementation:

```rust
pub fn account_tx_in_pool_or_recent_block(&self, account_address: ContractAddress) -> bool {
    self.state.contains_account(account_address)
        || self.tx_pool.contains_account(account_address)
}
``` [3](#0-2) 

`state.contains_account` checks both `staged` and `committed`:

```rust
fn contains_account(&self, address: ContractAddress) -> bool {
    self.staged.contains_key(&address) || self.committed.contains_key(&address)
}
``` [4](#0-3) 

The `committed` map is populated by `commit_block` with every address whose `deploy_account` (or any other) transaction was included in the committed block:

```rust
self.committed.extend(address_to_nonce.clone());
``` [5](#0-4) 

The test `mempool_state_retains_address_across_api_calls` explicitly confirms that `account_tx_in_pool_or_recent_block` returns `true` even after the transaction has been committed to a block:

```rust
commit_block(&mut mempool, nonces, []);
// Assert: Mempool state still contains the address, even though the transaction was committed.
assert!(mempool.account_tx_in_pool_or_recent_block(account_address));
``` [6](#0-5) 

**Step 3 — The stale state reader creates the exploitable window.**

The gateway instantiates a fresh `StatefulTransactionValidator` per transaction. The nonce is read from `gateway_fixed_block_state_reader`, which reflects the latest committed block at the time of instantiation:

```rust
let account_nonce =
    self.get_nonce_from_state(executable_tx.contract_address()).await ...;
``` [7](#0-6) 

The batcher calls `commit_block` on the mempool (updating `committed`) and the committer writes the new state to storage (updating the state reader) as **separate, asynchronous operations**. There is a real window — however brief — where:

- The mempool's `committed` map already contains account `A` with `nonce=1` (deploy_account was committed in block N), **and**
- The gateway's state reader still reflects block N-1, returning `account_nonce=0` for account `A`.

During this window, for an incoming invoke tx from account `A` with `tx_nonce=1`:

| Check | Value | Result |
|---|---|---|
| `tx.nonce() == Nonce(Felt::ONE)` | `true` | condition met |
| `account_nonce == Nonce(Felt::ZERO)` | `true` (stale) | condition met |
| `account_tx_in_pool_or_recent_block(A)` | `true` (A is in `committed`) | skip = `true` |

`skip_validate=true` → `validate=false` → `__validate__` is **not called** → the tx is admitted to the mempool without signature verification.

**Step 4 — `validate_nonce` and `validate_by_mempool` do not block this.**

`validate_nonce` with `account_nonce=0` and `tx_nonce=1` passes the range check `0 ≤ 1 ≤ 200`. [8](#0-7) 

`validate_by_mempool` calls `state.validate_incoming_tx`, which resolves the nonce via `resolve_nonce`. With account `A` in `committed` at nonce=1 and `tx_nonce=1`, the check `tx_nonce < account_nonce` is `1 < 1 = false`, so it passes. [9](#0-8) 

---

### Impact Explanation

An attacker who has deployed an account can submit an invoke transaction with `nonce=1` and an **invalid signature** during the race window. The gateway skips `__validate__` and admits the transaction to the mempool. The transaction will fail during blockifier execution (where `__validate__` is called with `validate=true`), but the gateway's admission invariant — that every accepted invoke transaction has passed its account's `__validate__` — is broken. This enables mempool pollution with signature-invalid transactions, constituting unauthorized transaction admission.

Impact category: **High — Mempool/gateway/RPC admission accepts invalid transactions before sequencing.**

---

### Likelihood Explanation

The race window exists in every block that contains a `deploy_account` transaction. The window is bounded by the latency between `commit_block` being called on the mempool and the storage update being visible to the gateway's state reader factory. In a high-throughput production deployment with asynchronous component communication, this window is reliably present. An attacker who controls the timing of their invoke submission (e.g., by monitoring mempool commit events) can target this window deterministically.

---

### Recommendation

The root cause is that `skip_stateful_validations` uses `account_tx_in_pool_or_recent_block`, which conflates two distinct states:

1. **Account has a pending `deploy_account` tx** (not yet deployed) → skip is correct.
2. **Account was committed in a recent block** (already deployed) → skip is **incorrect**.

Two mitigations mirror the external report's recommendations:

**Option 1 (preferred):** Introduce a separate mempool query — e.g., `account_has_pending_deploy_account_tx` — that returns `true` only when the account has a `deploy_account` transaction in the pool or staged queue, **not** when it is in `committed`. Use this narrower check in `skip_stateful_validations`.

**Option 2:** In `skip_stateful_validations`, after `account_tx_in_pool_or_recent_block` returns `true`, additionally verify that the account's nonce in the mempool's `committed` map is still `0` (i.e., the deploy has not yet been finalized). If the committed nonce is already `≥1`, do not skip validation.

---

### Proof of Concept

```
1. Attacker deploys account A via deploy_account tx (nonce=0).
   → Block N is committed; batcher calls commit_block({A: 1}) on mempool.
   → mempool.state.committed = {A: 1}

2. Storage write for block N is in-flight (not yet visible to gateway state reader).
   → gateway_fixed_block_state_reader.get_nonce(A) still returns 0 (block N-1 view).

3. Attacker submits invoke tx: sender=A, nonce=1, signature=INVALID.

4. Gateway calls extract_state_nonce_and_run_validations:
   - account_nonce = get_nonce_from_state(A) → 0  (stale)
   - validate_nonce: 0 ≤ 1 ≤ 200 → OK
   - validate_by_mempool: resolve_nonce(A, 0) = committed[A] = 1; 1 < 1 = false → OK
   - skip_stateful_validations:
       tx.nonce()==1 && account_nonce==0 → true
       account_tx_in_pool_or_recent_block(A) → true (A in committed)
       → skip_validate = true

5. run_validate_entry_point(skip_validate=true):
   - execution_flags.validate = false
   - StatefulValidator::perform_validations returns Ok(()) without calling __validate__

6. Invoke tx with INVALID signature is forwarded to mempool.add_tx → admitted.

7. Batcher executes tx with validate=true → __validate__ fails → tx reverted.
   Gateway invariant violated: invalid-signature tx was admitted.
``` [1](#0-0) [10](#0-9) [3](#0-2) [4](#0-3)

        &mut self,

        skip_validate: bool,

        let only_query = false;
        let charge_fee = enforce_fee(executable_tx, only_query);
        let strict_nonce_check = false;

            ExecutionFlags { only_query, charge_fee, validate: !skip_validate, strict_nonce_check };

        let account_tx = AccountTransaction { tx: executable_tx.clone(), execution_flags };

        // Build block context.
        let mut versioned_constants = VersionedConstants::get_versioned_constants(

        );
        // The validation of a transaction is not affected by the casm hash migration.
        versioned_constants.disable_casm_hash_migration();

        let mut block_info = self.gateway_fixed_block_state_reader.get_block_info().await?;
        block_info.block_number = block_info.block_number.unchecked_next();
        let block_context = BlockContext::new(
            block_info,
            self.chain_info.clone(),
            versioned_constants,
            BouncerConfig::max(),
        );

        // Move state into the blocking task and run CPU-heavy validation.
        let state_reader_and_contract_manager = self.take_state_reader_and_contract_manager();

        let cur_span = Span::current();

        tokio::task::spawn_blocking(move || {
            cur_span.in_scope(|| {
                let state = CachedState::new(state_reader_and_contract_manager);
                let mut blockifier_validator = StatefulValidator::create(state, block_context);
                blockifier_validator.validate(account_tx)
            })
        })
        .await
        .map_err(|e| StarknetError {
            code: StarknetErrorCode::UnknownErrorCode(

            message: format!("Blocking task join error when running the validate entry point: {e}"),
        })?
        .map_err(|e| StarknetError {
            code: StarknetErrorCode::KnownErrorCode(KnownStarknetErrorCode::ValidateFailure),
            message: e.to_string(),
        })?;
        Ok(())
    }

**File:** crates/apollo_gateway/src/stateful_transaction_validator.rs (L429-461)
```rust
async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
