### Title
Pool Join Commits Attacker-Controlled `targetPuzzleHash` via Double-Fetch Race — (`File: packages/gui/src/components/plotNFT/select/PlotNFTSelectPool.tsx`)

---

### Summary

When a user joins a pool, the GUI makes **two separate HTTP requests** to the pool's `/pool_info` endpoint: one for display and one at submit time. A malicious pool operator can serve a legitimate `targetPuzzleHash` for the display fetch (shown in "Verify Pool Details") and a different, attacker-controlled `targetPuzzleHash` for the submit-time fetch. The transaction is committed on-chain with the attacker's puzzle hash, permanently redirecting all farming rewards to the attacker.

---

### Finding Description

The pool-join flow is split across two components with independent fetches:

**Step 1 — Display fetch** (`PlotNFTSelectBase.tsx`):
`usePoolInfo(poolUrl)` is called reactively as the user types the URL. It fetches `GET <poolUrl>/pool_info` and renders the result — including `targetPuzzleHash` — in the "Verify Pool Details" card. [1](#0-0) [2](#0-1) 

**Step 2 — Submit-time fetch** (`PlotNFTSelectPool.tsx`):
When the user clicks "Create" / "Join", `handleSubmit` calls `prepareSubmitData`, which issues a **fresh, independent** `getPoolInfo(normalizedPoolUrl)` call. The `targetPuzzleHash` returned by this second request — not the one displayed — is what gets committed on-chain via `pwJoinPool`. [3](#0-2) 

**The fetch chain** for the submit-time call:

```
prepareSubmitData()
  → getPoolInfo(normalizedPoolUrl)          [util/getPoolInfo.ts]
    → window.appAPI.fetchPoolInfo(poolUrl)  [preload.ts]
      → ipcMain FETCH_POOL_INFO             [main.tsx:492-494]
        → fetchJSON(`${poolUrl}/pool_info`) [utils/fetchJSON.ts]
``` [4](#0-3) [5](#0-4) 

The `targetPuzzleHash` from the submit-time fetch is passed directly to `pwJoinPool` without any cross-check against the value that was displayed to the user: [6](#0-5) [7](#0-6) 

The `targetPuzzleHash` is the on-chain puzzle hash that controls where all farming rewards are paid. It is embedded in the PlotNFT singleton at creation/join time and cannot be changed without a new on-chain transaction subject to a `relativeLockHeight` delay. [8](#0-7) 

---

### Impact Explanation

A malicious pool operator controls the `/pool_info` HTTP endpoint. They can:

1. Serve a legitimate `targetPuzzleHash` (their real pool address) during the display phase so the "Verify Pool Details" card looks correct.
2. At the moment the user clicks "Create", swap the response to return an attacker-controlled `targetPuzzleHash`.
3. The GUI commits the PlotNFT on-chain with the attacker's puzzle hash.
4. All future farming rewards from that PlotNFT are paid to the attacker's address, not the pool's real address.

The user has no way to detect this from the GUI — the displayed value was correct. The only way to detect it is to inspect the on-chain singleton state after the fact.

This matches the allowed impact: **High — Corruption or spoofing of pool state that causes a user to join with the wrong destination address, resulting in permanent farming reward loss.**

---

### Likelihood Explanation

- Any pool operator can execute this attack with zero cost — they simply change their HTTP response between the display and submit requests.
- The attack window is the time between the user seeing the pool details and clicking "Create" — typically seconds.
- On testnet, HTTP is allowed, making MITM trivial. On mainnet, HTTPS is enforced for the display validation in `usePoolInfo`, but the submit-time `fetchJSON` call has no independent HTTPS enforcement beyond whatever URL the user typed.
- The user has no confirmation step that cross-checks the `targetPuzzleHash` used in the transaction against the one displayed. [9](#0-8) 

---

### Recommendation

1. **Single fetch, reuse result**: Capture the `targetPuzzleHash` from the display fetch (already in React state via `usePoolInfo`) and pass it directly into `prepareSubmitData` instead of re-fetching. This eliminates the race window entirely.

2. **Cross-check at submit time**: If a second fetch is retained, compare the `targetPuzzleHash` from the submit-time fetch against the one already displayed. If they differ, abort and show an error.

3. **Explicit confirmation of `targetPuzzleHash`**: Add a confirmation dialog at submit time that shows the exact `targetPuzzleHash` that will be committed on-chain, sourced from the same fetch result that will be used in the transaction.

---

### Proof of Concept

1. Attacker operates a pool server at `https://evil-pool.example`.
2. The server's `/pool_info` endpoint tracks request count per client IP.
3. For the **first** request (display): returns `targetPuzzleHash = <legitimate_pool_address>`.
4. For the **second** request (submit): returns `targetPuzzleHash = <attacker_wallet_address>`.
5. User visits the pool join page, enters `https://evil-pool.example`, sees the legitimate puzzle hash in "Verify Pool Details", and clicks "Create".
6. `prepareSubmitData` fires the second fetch, receives the attacker's puzzle hash, and calls `pwJoinPool` with it.
7. The PlotNFT singleton is created on-chain with `targetPuzzleHash = <attacker_wallet_address>`.
8. All farming rewards from that PlotNFT are permanently paid to the attacker. [10](#0-9) [11](#0-10)

```typescript
async function prepareSubmitData(data: FormData): SubmitData {
  const { self, fee, poolUrl } = data;
  const initialTargetState = {
    state: self ? 'SELF_POOLING' : 'FARMING_TO_POOL',
  };

  if (!self && poolUrl) {
    const normalizedPoolUrl = normalizeUrl(poolUrl);
    const { targetPuzzleHash, relativeLockHeight } = await getPoolInfo(normalizedPoolUrl);
    if (!targetPuzzleHash) {
      throw new Error(t`Pool does not provide targetPuzzleHash.`);
    }
    if (relativeLockHeight === undefined) {
      throw new Error(t`Pool does not provide relativeLockHeight.`);
    }

    initialTargetState.poolUrl = normalizedPoolUrl;
    initialTargetState.targetPuzzleHash = targetPuzzleHash;
    initialTargetState.relativeLockHeight = relativeLockHeight;
  }

  const feeMojos = chiaToMojo(fee || '0');

  return {
    fee: feeMojos,
    initialTargetState,
  };
}
