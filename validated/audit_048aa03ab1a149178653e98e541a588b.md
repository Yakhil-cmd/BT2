### Title
`executeOracleStopLossDecay` Changes `decayPerSecondE8` Without First Snapshotting Watermarks, Retroactively Corrupting Stop-Loss Accounting — (`metric-periphery/contracts/extensions/OracleValueStopLossExtension.sol`)

---

### Summary

`OracleValueStopLossExtension.executeOracleStopLossDecay()` writes a new `decayPerSecondE8` directly into `oracleStopLossConfig` without first applying the old rate to the stored per-bin `BinHighWatermarks`. Because decay is computed lazily on the next `afterSwap` call using `block.timestamp - hwmS.lastDecayTs` as the elapsed window, the new rate is retroactively applied to the entire period that elapsed under the old rate. This corrupts the watermark baseline that the stop-loss floor is derived from, either silently weakening LP protection (rate increase) or spuriously blocking legitimate swaps (rate decrease).

---

### Finding Description

The watermark decay is lazy: `_checkAndUpdateWatermarks` computes `dt = block.timestamp - hwmS.lastDecayTs` and calls `_decayed(hwmS.token0, decayRate, dt)` where `decayRate` is the **current** `oracleStopLossConfig[pool_].decayPerSecondE8`. [1](#0-0) 

`_decayed` applies the rate linearly over the full `dt`: [2](#0-1) 

`executeOracleStopLossDecay` writes the new rate without touching any `BinHighWatermarks` or `lastDecayTs`: [3](#0-2) 

The next `afterSwap` call therefore applies `newRate × (now − lastDecayTs)` to the stored watermark, where the elapsed window `(now − lastDecayTs)` was accumulated entirely under `oldRate`. The two rates are conflated into a single multiplication, producing a wrong effective watermark.

---

### Impact Explanation

**Rate increase (faster decay):** The watermarks are decayed more aggressively for the past window than they should have been. The effective floor `hwm_effective × (1 − drawdownE6)` is lower than the LP-agreed limit. Swaps that should trigger `OracleStopLossTriggered` pass through, allowing LP value extraction beyond the configured drawdown cap. The stop-loss guard — the sole on-chain protection for LP principal in this extension — silently fails.

**Rate decrease (slower decay):** The watermarks are decayed less than they should have been. The effective floor is higher than intended. Legitimate swaps revert with `OracleStopLossTriggered`, rendering the pool's swap path unusable until the watermarks naturally ratchet down on a future swap.

Both directions break the invariant that the stop-loss floor tracks the LP-agreed drawdown applied to the correctly-decayed watermark.

---

### Likelihood Explanation

The pool admin is semi-trusted and the decay update is timelocked, so the trigger requires a valid timelocked admin action. However:

- The timelock does not prevent the state corruption — it only delays when the rate change takes effect. Once `executeOracleStopLossDecay` is called, the corruption is immediate and irreversible for all bins whose `lastDecayTs` predates the call.
- Any legitimate decay rate adjustment (e.g., tuning from ~5%/day to ~10%/day) silently corrupts the watermark baseline for every bin that has not been touched since the last swap.
- No special attacker capability is required; the pool admin acting in good faith is sufficient to trigger the bug.

---

### Recommendation

Before writing the new `decayPerSecondE8`, apply the old decay rate to every active bin's watermarks and reset `lastDecayTs` to `block.timestamp`. Because iterating all bins on-chain is impractical, the minimal correct fix is to record the rate-change timestamp and store the old rate alongside it, then apply a two-segment decay in `_decayed` (old rate × elapsed-before-change + new rate × elapsed-after-change). Alternatively, require that `executeOracleStopLossDecay` can only be called when all bins have been touched in the current block (i.e., `lastDecayTs == block.timestamp`), or reset all watermarks to zero (forcing re-arm from the next swap's live metric).

A simpler safe fix analogous to the external report's recommendation:

```solidity
function executeOracleStopLossDecay(address pool_) external onlyPoolAdmin(pool_) {
    PoolStopLossSchedule storage sched = _initializedSchedule(pool_);
    if (sched.pendingDecayExecuteAfter == 0) revert OracleStopLossNoPendingDecay(pool_);
    _requireElapsed(sched.pendingDecayExecuteAfter);
    uint32 decay = sched.pendingDecayPerSecondE8;

    // Snapshot all active bins with the OLD rate before switching.
    // (Caller must supply the list of active bin indices, or iterate the known range.)
    // For each active bin:
    //   BinHighWatermarks storage hwm = highWatermarks[pool_][binIdx];
    //   uint256 dt = block.timestamp - hwm.lastDecayTs;
    //   uint256 oldRate = oracleStopLossConfig[pool_].decayPerSecondE8;
    //   hwm.token0 = uint104(_decayed(hwm.token0, oldRate, dt));
    //   hwm.token1 = uint104(_decayed(hwm.token1, oldRate, dt));
    //   hwm.lastDecayTs = uint32(block.timestamp);

    oracleStopLossConfig[pool_].decayPerSecondE8 = decay;
    (sched.pendingDecayPerSecondE8, sched.pendingDecayExecuteAfter) = (0, 0);
    emit OracleStopLossDecaySet(pool_, decay);
}
```

### Proof of Concept

1. Pool is configured with `drawdownE6 = 50_000` (50%) and `decayPerSecondE8 = 58` (~5%/day).
2. A swap sets `highWatermarks[pool][0].token0 = 1000`, `lastDecayTs = T0`.
3. 10 days pass with no swaps. Under the old rate, the correct decayed watermark at `T0 + 10 days` is approximately `1000 × (1 − 58 × 864000 / 1e8) ≈ 499` (just above zero).
4. At `T0 + 10 days`, the pool admin legitimately executes a decay rate increase to `decayPerSecondE8 = 116` (~10%/day).
5. `executeOracleStopLossDecay` writes `116` without touching `highWatermarks`. `lastDecayTs` remains `T0`.
6. The next swap computes `dt = 10 days`, `_decayed(1000, 116, 864000) = 1000 × (1 − 116 × 864000 / 1e8)`. Since `116 × 864000 = 100,224,000 > 1e8`, `_decayed` returns `0`.
7. The effective watermark is `0`. The floor is `0 × (1 − 0.5) = 0`. Any live metric passes the check — the stop-loss never triggers regardless of how much LP value has been extracted.
8. An LP whose value per share has dropped 40% (within the 50% drawdown limit under the old rate, but the stop-loss should still be active) receives no protection. [3](#0-2) [4](#0-3) [2](#0-1)

  function _decayed(uint256 hwm, uint256 ratePerSecondE8, uint256 dt) private pure returns (uint256) {
    if (ratePerSecondE8 == 0 || dt == 0 || hwm == 0) return hwm;
    uint256 factor = ratePerSecondE8 * dt;
    if (factor >= E8) return 0;
    return hwm - (hwm * factor) / E8;
  }
