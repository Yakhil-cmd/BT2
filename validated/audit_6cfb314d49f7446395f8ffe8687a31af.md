### Title
Partial NEP-141 Refund in Omni `ft_transfer_call` Exit Path Causes Permanent Fund Freeze — (`engine-precompiles/src/native.rs`, `engine/src/contract_methods/connector.rs`)

---

### Summary

When an EVM user exits an ERC-20 token to NEAR via the Omni path (`withdrawToNear` with a `:msg` suffix), the engine burns the full ERC-20 amount and calls `ft_transfer_call` on the corresponding NEP-141 contract. If the receiving NEAR contract's `ft_on_transfer` returns a non-zero `unused_amount` (a partial refund per the NEP-141 standard), the NEP-141 contract internally refunds that portion back to Aurora's NEP-141 balance. However, `exit_to_near_precompile_callback` treats any `PromiseResult::Successful(_)` as a complete transfer and performs no re-mint of the returned portion. The result is that the returned NEP-141 tokens are permanently stranded in Aurora's balance with no corresponding ERC-20 representation.

---

### Finding Description

**Step 1 — ERC-20 burn and `ft_transfer_call` dispatch**

In `exit_erc20_token_to_near`, the `Some(Message::Omni(msg))` branch sets `method = "ft_transfer_call"` and `transfer_near_args = None`: [1](#0-0) 

The ERC-20 contract burns the full `exit_params.amount` before calling the precompile (standard ERC-20 burn-then-exit pattern). The precompile then constructs a `ft_transfer_call` promise for the full amount.

**Step 2 — Callback construction**

The `callback_args` for the Omni ERC-20 path always has `transfer_near = None`. With the `error_refund` feature disabled (the default), `refund` is also `None`, making `callback_args` equal to `ExitToNearPrecompileCallbackArgs::default()`, so **no callback is scheduled at all**: [2](#0-1) 

With `error_refund` enabled, a callback is scheduled, but it only re-mints on **failure** (the `else if` branch), not on partial success.

**Step 3 — Callback ignores the actual transferred amount**

In `exit_to_near_precompile_callback`, the success branch discards the promise return value entirely with `_`: [3](#0-2) 

The NEP-141 standard specifies that `ft_transfer_call` resolves **successfully** even when `unused_amount > 0`; the NEP-141 contract's internal `ft_resolve_transfer` silently refunds the unused portion back to Aurora's balance. The promise result (`PromiseResult::Successful(bytes)`) contains the net transferred amount, but the callback ignores it.

**Step 4 — No re-mint path exists for the partial refund**

The `refund_on_error` / re-mint path is only reachable when the promise **fails**: [4](#0-3) 

There is no code path that reads the `unused_amount` from a successful `ft_transfer_call` result and re-mints the corresponding ERC-20 tokens.

---

### Impact Explanation

- The attacker burns X ERC-20 tokens.
- Aurora calls `ft_transfer_call` for X NEP-141 tokens.
- The receiver's `ft_on_transfer` returns `unused_amount = Y`.
- The NEP-141 contract refunds Y tokens to Aurora's NEP-141 balance.
- `ft_transfer_call` resolves as **Successful**.
- Aurora's callback does nothing (no re-mint).
- **Result**: X ERC-20 tokens are permanently destroyed; only (X−Y) NEP-141 tokens reached the receiver; Y NEP-141 tokens are permanently stranded in Aurora's NEP-141 balance with no ERC-20 counterpart and no recovery mechanism.

This violates the core accounting invariant: `burned_erc20 == net_nep141_transferred_out`. The stranded Y tokens cannot be recovered by the user through any standard EVM or NEAR interface, constituting a **permanent freeze of funds**.

---

### Likelihood Explanation

- Any unprivileged EVM user holding a bridged ERC-20 can trigger this by calling `withdrawToNear` with an Omni-format destination string pointing to any NEAR contract they control.
- The attacker deploys a NEAR contract whose `ft_on_transfer` returns a non-zero `unused_amount`.
- No admin access, key compromise, or external oracle is required.
- The path is reachable on any Aurora deployment that supports the Omni exit (i.e., any deployment where the ERC-20 mirror and NEP-141 are registered).

---

### Recommendation

In `exit_to_near_precompile_callback`, when the promise result is `Successful`, parse the returned bytes as a `U128` (the net transferred amount per the NEP-141 `ft_transfer_call` return value). If `net_transferred < original_amount`, compute `delta = original_amount - net_transferred` and call `refund_on_error` (re-mint) for `delta` to the original `refund_address`. This requires:

1. Storing the original amount and refund address in `ExitToNearPrecompileCallbackArgs` for the Omni ERC-20 path (analogous to how `error_refund` stores `RefundCallArgs`).
2. Parsing the `PromiseResult::Successful(bytes)` value as a `U128` and comparing it to the original amount.
3. Calling `engine::refund_on_error` for the difference if any.

---

### Proof of Concept

1. Deploy a NEP-141-compatible NEAR contract `mock_receiver.near` whose `ft_on_transfer` always returns `"50"` (unused) when called with amount `"100"`.
2. Register a bridged ERC-20 for this NEP-141 on Aurora.
3. Mint 100 ERC-20 tokens to an EVM address.
4. Call `withdrawToNear` with destination `"mock_receiver.near:<omni_msg>"` and amount 100.
5. Observe:
   - ERC-20 balance of the EVM address: 0 (fully burned).
   - NEP-141 balance of `mock_receiver.near`: 50 (only 50 received).
   - NEP-141 balance of `aurora`: 50 (stuck, no ERC-20 counterpart).
   - No re-mint event emitted; ERC-20 balance of the EVM address remains 0.

The 50 NEP-141 tokens in Aurora's balance are unrecoverable through any standard interface.
