### Title
`EvmErc20.sol` / `EvmErc20V2.sol` — Unchecked Precompile Return Value in `withdrawToNear` and `withdrawToEthereum` Causes Permanent Fund Freeze - (File: `etc/eth-contracts/contracts/EvmErc20.sol`, `etc/eth-contracts/contracts/EvmErc20V2.sol`)

---

### Summary

Both `EvmErc20.sol` and `EvmErc20V2.sol` implement the `IExit` interface's `withdrawToNear` and `withdrawToEthereum` functions. Each function burns the caller's ERC-20 tokens first, then calls the corresponding exit precompile via inline assembly. The return value of the `call` opcode is captured in a local variable `res` but is **never checked**. If the precompile call fails for any reason, the ERC-20 tokens are permanently destroyed with no corresponding NEAR-side transfer, freezing the user's funds.

Additionally, `EvmErc20V2.sol` encodes a `sender` address (20 bytes) between the flag byte and the amount field in the `withdrawToNear` calldata. The `ExitToNear` precompile (without the `error_refund` compile-time feature) does not expect this extra field, causing it to misparse the amount and recipient, guaranteeing a precompile failure on every `withdrawToNear` call from `EvmErc20V2`.

---

### Finding Description

**Root cause 1 — Unchecked return value (both contracts, both functions):**

In `EvmErc20.sol`:

```solidity
// Line 53-63
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);                          // tokens destroyed here
    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;
    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f,
                        0, add(input, 32), input_size, 0, 32)
        // res is NEVER checked — failure is silently swallowed
    }
}
```

The same pattern appears in `withdrawToEthereum` (lines 65–76) and in both functions of `EvmErc20V2.sol`. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3) 

**Root cause 2 — Malformed input from `EvmErc20V2.withdrawToNear` (without `error_refund` feature):**

`EvmErc20V2.sol` encodes the input as:

```
\x01 | sender (20 bytes) | amount_b (32 bytes) | recipient (variable)
``` [5](#0-4) 

The `ExitToNear` precompile's `TryFrom<&[u8]>` implementation (without `error_refund`) strips only the flag byte and then reads bytes `[0..32]` as the amount and `[32..]` as the recipient:

```rust
// native.rs line 758-763
0x1 => {
    let amount = parse_amount(&input[..32])?;
    let Recipient { receiver_account_id, message } = parse_recipient(&input[32..])?;
``` [6](#0-5) 

Without `error_refund`, `input` after stripping the flag is `sender (20 bytes) | amount_b (32 bytes) | recipient`. So:
- `input[..32]` = `sender[0..20] + amount_b[0..12]` → wrong amount
- `input[32..]` = `amount_b[12..] + recipient` → starts with 20 bytes of binary data, fails `str::from_utf8`, returns `ERR_INVALID_RECEIVER_ACCOUNT_ID` [7](#0-6) 

The precompile returns `ExitError::Other(...)`, the EVM subcall returns `0`, but `res` is never checked, so the outer transaction succeeds while the tokens are already burned.

The `error_refund` feature path does parse the 20-byte sender as a refund address and would work correctly, but without it the input is structurally incompatible: [8](#0-7) 

---

### Impact Explanation

**Permanent freezing of funds.** When `withdrawToNear` or `withdrawToEthereum` is called and the precompile fails:

1. `_burn` permanently destroys the caller's ERC-20 tokens on the Aurora side.
2. No NEP-141 `ft_transfer` promise is scheduled on the NEAR side.
3. The transaction returns success to the caller.
4. The tokens are irrecoverably lost — they no longer exist on either chain.

For `EvmErc20V2.withdrawToNear` without `error_refund`, this is not a corner case — it is the outcome of **every** call, because the input format is structurally incompatible with the precompile's parser.

---

### Likelihood Explanation

**High.** The entry path is a standard, publicly callable ERC-20 function (`withdrawToNear`, `withdrawToEthereum`) on the production bridge token contracts. Any token holder can trigger it. For `EvmErc20V2.withdrawToNear` specifically, the failure is deterministic whenever the `error_refund` compile-time feature is absent from the engine build. For `EvmErc20.sol`, failure occurs whenever the recipient is not a valid NEAR account ID or the ERC-20 is not registered in the `Erc20Nep141Map` storage (`ERR_TARGET_TOKEN_NOT_FOUND`). [9](#0-8) 

---

### Recommendation

1. **Check the return value** of the precompile `call` in both `withdrawToNear` and `withdrawToEthereum` in both contracts, and revert if it is zero:

```solidity
assembly {
    let res := call(gas(), EXIT_TO_NEAR_ADDR, 0, add(input, 32), input_size, 0, 32)
    if iszero(res) { revert(0, 0) }
}
```

2. **Align `EvmErc20V2.withdrawToNear` input encoding** with the precompile's actual expected format. If the `error_refund` feature is not guaranteed to be enabled, the `sender` field must be removed from the encoded input, or the contract must be gated to only deploy alongside a build that has `error_refund` enabled.

### Proof of Concept

**Scenario (EvmErc20.sol — invalid recipient):**

1. Alice holds 1000 units of a bridged NEP-141 token represented as `EvmErc20`.
2. Alice calls `withdrawToNear("!!!invalid!!!", 1000)`.
3. `_burn(Alice, 1000)` executes — Alice's ERC-20 balance drops to 0.
4. The precompile call is made with `recipient = b"!!!invalid!!!"`.
5. `parse_recipient` fails: `"!!!invalid!!!"` is not a valid NEAR account ID → `ExitError::Other("ERR_INVALID_RECEIVER_ACCOUNT_ID")`.
6. The EVM subcall returns `0`; `res` is ignored.
7. The outer transaction succeeds. Alice has 0 ERC-20 tokens and 0 NEAR tokens. Funds are permanently frozen.

**Scenario (EvmErc20V2.sol — always fails without `error_refund`):**

1. Bob holds 500 units of a bridged token represented as `EvmErc20V2`.
2. Bob calls `withdrawToNear("bob.near", 500)`.
3. `_burn(Bob, 500)` executes.
4. Input sent to precompile: `\x01 | Bob_addr (20 bytes) | 500_as_bytes32 (32 bytes) | "bob.near"`.
5. Precompile reads `Bob_addr[0..20] + 500_bytes[0..12]` as amount (garbage), then `500_bytes[12..] + "bob.near"` as recipient — fails UTF-8 or account-ID validation.
6. Precompile returns error; `res = 0`; not checked.
7. Transaction succeeds. Bob's 500 tokens are permanently destroyed. [1](#0-0) [3](#0-2) [10](#0-9)

fn parse_recipient(recipient: &[u8]) -> Result<Recipient<'_>, ExitError> {
    let recipient = str::from_utf8(recipient)
        .map_err(|_| ExitError::Other(Cow::from("ERR_INVALID_RECEIVER_ACCOUNT_ID")))?;
    let (receiver_account_id, message) = recipient.split_once(':').map_or_else(
        || (recipient, None),
        |(recipient, msg)| {
            if msg == UNWRAP_WNEAR_MSG {
                (recipient, Some(Message::UnwrapWnear))
            } else {
                (recipient, Some(Message::Omni(msg)))
            }
        },
    );

    Ok(Recipient {
        receiver_account_id: receiver_account_id
            .parse()
            .map_err(|_| ExitError::Other(Cow::from("ERR_INVALID_RECEIVER_ACCOUNT_ID")))?,
        message,
    })

impl<'a> TryFrom<&'a [u8]> for ExitToNearParams<'a> {
    type Error = ExitError;

    fn try_from(input: &'a [u8]) -> Result<Self, Self::Error> {
        // The first byte of the input is a flag, selecting the behavior to be triggered:
        // 0x00 -> Eth(base) token withdrawal
        // 0x01 -> ERC-20 token withdrawal
        let flag = input
            .first()
            .copied()
            .ok_or_else(|| ExitError::Other(Cow::from("ERR_MISSING_FLAG")))?;

        #[cfg(feature = "error_refund")]
        let (refund_address, input) = parse_input(input)?;
        #[cfg(not(feature = "error_refund"))]
        let input = parse_input(input)?;

        match flag {
            0x0 => {
                let Recipient {
                    receiver_account_id,
                    message,
                } = parse_recipient(input)?;

                Ok(Self::BaseToken(BaseTokenParams {
                    #[cfg(feature = "error_refund")]
                    refund_address,
                    receiver_account_id,
                    message,
                }))
            }
            0x1 => {
                let amount = parse_amount(&input[..32])?;
                let Recipient {
                    receiver_account_id,
                    message,
                } = parse_recipient(&input[32..])?;

                Ok(Self::Erc20TokenParams(Erc20TokenParams {
                    #[cfg(feature = "error_refund")]
                    refund_address,
                    receiver_account_id,
                    amount,
                    message,
                }))
            }
            _ => Err(ExitError::Other(Cow::from("ERR_INVALID_FLAG"))),
        }
    }
