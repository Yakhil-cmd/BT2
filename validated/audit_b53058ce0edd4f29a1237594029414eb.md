### Title
Unchecked `transferFrom` Boolean Return Value in XCC Precompile Allows Free Cross-Contract Calls - (`engine-precompiles/src/xcc.rs`)

### Summary

The Cross-Contract Call (XCC) precompile performs a `transferFrom` on the wNEAR ERC-20 contract to collect payment from the caller, but never inspects the ABI-encoded boolean return value of that call. If the wNEAR contract returns `false` (EVM exit succeeds but transfer did not occur), the precompile proceeds to schedule the NEAR cross-contract call without having collected any wNEAR from the user.

### Finding Description

In `run_with_handle` of `CrossContractCall`, when `required_near != ZERO_YOCTO`, the precompile issues an EVM subcall encoding a `transferFrom(sender, engine_implicit_address, required_near)` against the wNEAR ERC-20 contract:

```rust
let (exit_reason, return_value) =
    handle.call(wnear_address.raw(), None, tx_data, None, false, &context);
match exit_reason {
    // Transfer successful, nothing to do
    aurora_evm::ExitReason::Succeed(_) => (),   // ← return_value is NEVER inspected
    aurora_evm::ExitReason::Revert(r) => { ... }
    aurora_evm::ExitReason::Error(e)  => { ... }
    aurora_evm::ExitReason::Fatal(f)  => { ... }
}
``` [1](#0-0) 

The `return_value` bytes — which carry the ABI-encoded `bool` result of `transferFrom` — are silently discarded on the `Succeed` arm. A non-reverting ERC-20 that returns `false` on failure would cause `exit_reason` to be `Succeed` while `return_value` encodes `false` (`0x000...000`). The precompile treats this as a successful payment and continues to emit the promise log that schedules the NEAR cross-contract call.

The `transfer_from_args` helper correctly encodes the `transferFrom` selector and arguments: [2](#0-1) 

The test fixture even documents that `transferFrom` returns a `bool`: [3](#0-2) 

Yet the production call site never decodes or validates that bool.

### Impact Explanation

If the wNEAR ERC-20 contract returns `false` without reverting (non-compliant token behavior), an attacker can:

1. Call the XCC precompile with any valid `CrossContractCallArgs`.
2. The `transferFrom` subcall returns `false` (no wNEAR deducted from the attacker).
3. The precompile emits the promise log and the NEAR cross-contract call is scheduled.
4. The engine's own NEAR balance funds the router contract's storage staking and attached NEAR, without having collected wNEAR from the user.

This constitutes **theft of NEAR from the engine's balance** — the engine pays for NEAR gas on behalf of users who provided no wNEAR. Repeated exploitation drains the engine's NEAR balance, eventually preventing legitimate XCC calls and causing insolvency of the engine's NEAR holdings.

**Impact class:** High — Theft of unclaimed yield / temporary freezing of funds (engine NEAR balance drained).

### Likelihood Explanation

The wNEAR ERC-20 address is set by the admin via `factory_set_wnear_address`: [4](#0-3) 

The currently deployed wNEAR ERC-20 is an OpenZeppelin-based contract (`EvmErc20`) that reverts on failure rather than returning `false`, so the vulnerability is not immediately exploitable against the canonical wNEAR. However:

- The code contains a demonstrable missing check that is incorrect regardless of the current wNEAR implementation.
- If the wNEAR contract is ever upgraded or replaced with a non-reverting variant, the vulnerability becomes directly exploitable by any unprivileged EVM user.
- The root cause is entirely within Aurora Engine's own precompile code, not solely in external dependency behavior.

**Likelihood:** Low-Medium (not exploitable with current wNEAR, but the code is structurally incorrect and becomes exploitable under any non-reverting wNEAR variant).

### Recommendation

After the `Succeed` arm, decode `return_value` and revert if the boolean is `false`:

```rust
aurora_evm::ExitReason::Succeed(_) => {
    // Decode the bool return value of transferFrom
    let transfer_ok = return_value
        .get(31)
        .map(|&b| b == 1)
        .unwrap_or(false);
    if !transfer_ok {
        return Err(revert_with_message("ERR_WNEAR_TRANSFER_FAILED"));
    }
}
```

This mirrors the `safeTransferFrom` pattern recommended in the external report: check the return value and revert on failure.

### Proof of Concept

1. Deploy a non-reverting wNEAR mock ERC-20 that always returns `false` from `transferFrom` (EVM exits with `Succeed`, return data = `0x000...000`).
2. Admin calls `factory_set_wnear_address` with the mock address (or assume a future upgrade).
3. Attacker calls the XCC precompile (`0x516cded1d16af10cad47d6d49128e2eb7d27b372`) with a valid `CrossContractCallArgs::Eager` payload attaching NEAR.
4. The `transferFrom` subcall returns `false`; the precompile hits `Succeed(_) => ()` and ignores it.
5. The promise log is emitted; the engine schedules the NEAR cross-contract call and funds it from its own NEAR balance.
6. The attacker's wNEAR balance is unchanged; the engine's NEAR balance is reduced. [5](#0-4)
