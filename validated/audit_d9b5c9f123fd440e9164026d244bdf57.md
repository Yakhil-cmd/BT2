### Title
Gateway L2 Gas Price Threshold Uses Stale `l2_gas_price` Instead of `next_l2_gas_price`, Causing Wrong Admission Decisions - (File: crates/apollo_gateway/src/stateful_transaction_validator.rs)

### Summary

The gateway's stateful transaction validator checks an incoming transaction's offered L2 gas price against the **previous block's `l2_gas_price`** rather than the **previous block's `next_l2_gas_price`** (the price that will actually govern the block being built). This is the direct Sequencer analog of the Chainlink `minAnswer` bug: a price oracle value is consumed without verifying it is the correct reference point, causing the admission gate to enforce the wrong threshold.

### Finding Description

### Impact Explanation

Two concrete wrong-admission scenarios arise from the mismatch between `l2_gas_price` (gateway reference) and `next_l2_gas_price` (blockifier reference):

**Scenario A — admits transactions that will fail blockifier (price rising):**
- Previous block: `l2_gas_price = P`, `next_l2_gas_price = P′ > P`
- Gateway threshold (`min_gas_price_percentage = 100`): `P`
- A transaction with `max_price_per_unit = P` passes the gateway (`P ≥ P`) but fails the blockifier (`P < P′`)
- Result: invalid transaction admitted to mempool, wasted slot, potential DoS amplification

**Scenario B — rejects valid transactions (price falling):**
- Previous block: `l2_gas_price = P`, `next_l2_gas_price = P′ < P`
- Gateway threshold: `P`
- A transaction with `max_price_per_unit = P′` (exactly the correct next-block price) is rejected by the gateway (`P′ < P`) even though the blockifier would accept it (`P′ ≥ P′`)
- Result: valid user transaction silently dropped at admission

Both scenarios are reachable without any privilege: any user submitting a V3 (`AllResources`) transaction with an L2 gas price in the gap between `next_l2_gas_price` and `l2_gas_price` triggers the wrong decision.

### Likelihood Explanation

The EIP-1559 L2 gas price adjusts by at most `1/gas_price_max_change_denominator` (currently `1/48`) per block, so the gap between `l2_gas_price` and `next_l2_gas_price` is small under normal load. However:

- With `min_gas_price_percentage = 100`, **any** block where utilization differs from the target produces a non-zero gap, making Scenario A or B reachable every block.
- During rapid price recovery from the minimum (the `MIN_GAS_PRICE_INCREASE_DENOMINATOR = 333` path in `calculate_next_base_gas_price`), `next_l2_gas_price` can exceed `l2_gas_price` by a larger margin.
- The feature is gated by `validate_resource_bounds: bool` in `StatefulTransactionValidatorConfig`; when enabled, the wrong reference is always used. [7](#0-6) 

### Recommendation

1. Extend `GatewayFixedBlockStateReader` (and its `GatewayFixedBlockSyncStateClient` implementation) to expose `next_l2_gas_price` from `StorageBlockHeader`.
2. In `validate_resource_bounds`, replace the `l2_gas_price` read with `next_l2_gas_price` as the TODO comment already prescribes.
3. Add a test that constructs a block header where `next_l2_gas_price ≠ l2_gas_price` and verifies the gateway threshold is computed from `next_l2_gas_price`.

### Proof of Concept

```
Block N committed:
  l2_gas_price.price_in_fri     = 100 fri   (used for txs inside block N)
  next_l2_gas_price             = 102 fri   (EIP-1559 result, will govern block N+1)

User submits V3 invoke tx for block N+1:
  resource_bounds.l2_gas.max_price_per_unit = 101 fri

Gateway validate_resource_bounds (min_gas_price_percentage = 100):
  threshold = 100% * 100 = 100
  101 >= 100  →  ADMITTED to mempool  ✓

Blockifier pre-validation for block N+1 (l2_gas_price = next_l2_gas_price = 102):
  101 < 102  →  MaxGasPriceTooLow  ✗  REJECTED at execution

Outcome: gateway admits a transaction the blockifier will always reject.
```

The root cause is in `GatewayFixedBlockSyncStateClient::get_block_info_from_sync_client` at [8](#0-7)  which silently discards `next_l2_gas_price` from the block header, and in `validate_resource_bounds` at [9](#0-8)  which consumes the stale price without any guard.
