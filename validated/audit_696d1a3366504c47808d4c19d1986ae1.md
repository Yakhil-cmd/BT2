### Title
L1 Transaction `gas_per_pubdata_limit = 0` Bypasses Pubdata Fee Collection — (`basic_bootloader/src/bootloader/transaction_flow/zk/process_l1_transaction.rs`)

---

### Summary

L1→L2 priority transactions carry a user-supplied `gas_per_pubdata_limit` field. ZKsync OS reads this value without enforcing any minimum, so a sender who sets it to `0` causes `native_per_pubdata = 0` throughout the entire fee pipeline. The result is that the transaction can generate an arbitrary amount of pubdata while the post-execution pubdata check always passes, leaving the operator to absorb the L1 data-availability cost with no compensation.

---

### Finding Description

In `process_l1_transaction.rs`, the `gas_per_pubdata_limit` field is read verbatim from the transaction:

```rust
let gas_per_pubdata = transaction.gas_per_pubdata_limit.read();
``` [1](#0-0) 

It is then multiplied by `native_per_gas` to derive the per-byte native cost for pubdata:

```rust
let native_per_pubdata = (gas_per_pubdata as u64)
    .checked_mul(native_per_gas)
    .unwrap_or_else(|| { ... u64::MAX });
``` [2](#0-1) 

When `gas_per_pubdata = 0`, `native_per_pubdata = 0`. This zero propagates into two critical places:

**1. Intrinsic pubdata overhead is zeroed out in `create_resources_for_tx`:**

```rust
let intrinsic_pubdata_overhead = native_per_pubdata_byte.saturating_mul(intrinsic_pubdata);
// = 0 * intrinsic_pubdata = 0  → no native deducted for intrinsic pubdata
``` [3](#0-2) 

**2. The post-execution pubdata check always passes:**

```rust
let (enough, to_charge_for_pubdata, pubdata_used) =
    check_enough_resources_for_pubdata(system, native_per_pubdata, resources, None)?;
let is_success = !reverted && enough;
``` [4](#0-3) 

Inside `get_resources_to_charge_for_pubdata`, the native cost is `current_pubdata_spent * 0 = 0`, so `enough` is always `true` regardless of how many pubdata bytes the transaction writes:

```rust
let native = current_pubdata_spent
    .checked_mul(native_per_pubdata)
    .ok_or(out_of_native_resources!())?;
``` [5](#0-4) 

The `validate_structure` function explicitly allows `gas_per_pubdata_limit = 0` for L1 transactions (it only rejects non-zero values for non-L1 types):

```rust
match tx_type {
    Self::UPGRADE_TX_TYPE | Self::L1_L2_TX_TYPE => {}   // no minimum check
    _ => { if self.gas_per_pubdata_limit.read() != 0 { return Err(()); } }
}
``` [6](#0-5) 

No other location in ZKsync OS enforces a minimum `gas_per_pubdata_limit` for L1 transactions.

The operator fee is computed solely from gas:

```rust
let pay_to_operator = U256::from(gas_used)
    .checked_mul(U256::from(gas_price))
    .ok_or(internal_error!("gu*gp"))?;
``` [7](#0-6) 

Pubdata publication cost is never recovered when `native_per_pubdata = 0`.

---

### Impact Explanation

L1→L2 transactions are **priority transactions** that the sequencer is obligated to include; they cannot be censored. When `gas_per_pubdata_limit = 0`:

- The transaction may write to an arbitrary number of storage slots, generating large state-diff pubdata.
- The post-execution pubdata check always passes, so the transaction succeeds.
- The operator receives only `gas_used × gas_price` and receives **zero compensation** for the L1 data-availability cost of the pubdata generated.

The operator must pay to publish this pubdata to the settlement layer, resulting in a direct, uncompensated financial loss proportional to the pubdata generated. This is the exact analog of M-10: a fee parameter that should be enforced by the protocol is instead freely set to zero by an unprivileged user, causing the protocol to lose revenue it is entitled to.

---

### Likelihood Explanation

Any Ethereum account can submit an L1→L2 transaction with `gas_per_pubdata_limit = 0`. The code comment acknowledges that L1 contracts *should* enforce this invariant, but ZKsync OS does not double-check it:

> "The invariant that the user deposited more than the value needed for the transaction must be enforced on L1, but we double-check it here" [8](#0-7) 

ZKsync OS double-checks the deposit amount but **does not** double-check a minimum `gas_per_pubdata_limit`. If the L1 bridge contract ever fails to enforce a minimum (e.g., due to a misconfiguration, upgrade, or a chain that deploys its own bridge), the attack is immediately exploitable on-chain with no special privileges.

---

### Recommendation

Enforce a minimum `gas_per_pubdata_limit` for L1 transactions inside ZKsync OS, mirroring how L2 transactions derive `native_per_pubdata` from the operator-controlled block-level `pubdata_price`. A concrete fix is to reject (or saturate to a protocol-defined floor) any L1 transaction whose `gas_per_pubdata_limit` would produce `native_per_pubdata = 0`, logging a system warning as is done for other L1 resilience cases.

---

### Proof of Concept

1. Craft an L1→L2 transaction with `gas_per_pubdata_limit = 0`, a non-zero `gas_price`, and a callee that writes to 100 storage slots.
2. Submit it through the L1 bridge (or directly inject it as a priority transaction in a test environment).
3. ZKsync OS processes it: `native_per_pubdata = 0`, intrinsic pubdata overhead = 0, post-execution pubdata check passes unconditionally.
4. The transaction succeeds; the operator receives `gas_used × gas_price` but **zero** compensation for the ~3,200 bytes of pubdata (100 storage writes × 32 bytes each) it must publish to L1.
5. Repeat with increasing storage writes to amplify the uncompensated pubdata cost arbitrarily.
