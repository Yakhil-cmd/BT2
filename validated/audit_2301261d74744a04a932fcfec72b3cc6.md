Looking at the `SelfPermit.sol` contract and how it's used in multicall patterns, I can identify the analog.

### Title
`selfPermit` and `selfPermitAllowed` Revert on Front-Run Permit, Griefing Multicall Swap/Liquidity Flows — (`metric-periphery/contracts/base/SelfPermit.sol`)

### Summary

`SelfPermit.selfPermit()` and `SelfPermit.selfPermitAllowed()` call `permit()` unconditionally with no try-catch and no prior allowance check. Any observer can extract the permit signature from a pending multicall transaction, call `permit()` directly on the token to consume the nonce, and cause the victim's multicall to revert — atomically cancelling the bundled swap or liquidity operation.

### Finding Description

`SelfPermit.sol` exposes four permit helpers. Two of them are safe:

- `selfPermitIfNecessary` — skips the permit call when `allowance >= value`, so a front-run that already set the allowance causes the check to pass and the call is skipped. [1](#0-0) 
- `selfPermitAllowedIfNecessary` — same pattern, skips when `allowance == type(uint256).max`. [2](#0-1) 

Two of them are **not** safe:

- `selfPermit` — calls `IERC20Permit(token).permit(...)` unconditionally. [3](#0-2) 
- `selfPermitAllowed` — calls `IERC20PermitAllowed(token).permit(...)` unconditionally. [4](#0-3) 

The interface NatSpec for `selfPermitIfNecessary` explicitly acknowledges the front-run risk for `selfPermit`, confirming the protocol is aware of the pattern for the safe variant but left the unsafe variants unguarded: [5](#0-4) 

The production test suite demonstrates the intended multicall usage — `selfPermit` is bundled with `exactInputSingle` in a single `multicall()` call: [6](#0-5) 

### Impact Explanation

When a user submits `multicall([selfPermit(...), exactInputSingle(...)])` or `multicall([selfPermit(...), addLiquidity(...)])`:

1. An attacker reads the permit signature from the pending transaction in the mempool.
2. The attacker calls `token.permit(user, router, value, deadline, v, r, s)` directly, consuming the nonce.
3. The user's `selfPermit` call reverts because the nonce is already spent.
4. The entire `multicall` reverts atomically — the swap or liquidity operation never executes.

The user's funds are not lost, but their swap or liquidity flow is broken for that transaction. The attacker can repeat this for every retry that uses `selfPermit` instead of `selfPermitIfNecessary`, making the griefing persistent until the user changes their call pattern. This breaks the core swap/liquidity flow for users relying on the `selfPermit` path.

### Likelihood Explanation

- Requires mempool visibility (standard on public networks).
- The permit signature is fully visible in calldata.
- No capital is required by the attacker — only gas.
- The `selfPermit` + `multicall` pattern is the documented and tested usage path, so real users will hit this.

### Recommendation

Wrap the `permit()` call in `selfPermit` and `selfPermitAllowed` in a try-catch block, silently succeeding if the permit reverts (the allowance will already be set by the front-runner):

```solidity
function selfPermit(address token, uint256 value, uint256 deadline, uint8 v, bytes32 r, bytes32 s) public payable {
    try IERC20Permit(token).permit(msg.sender, address(this), value, deadline, v, r, s) {} catch {}
}

function selfPermitAllowed(address token, uint256 nonce, uint256 expiry, uint8 v, bytes32 r, bytes32 s) public payable {
    try IERC20PermitAllowed(token).permit(msg.sender, address(this), nonce, expiry, true, v, r, s) {} catch {}
}
```

Alternatively, add an allowance pre-check identical to `selfPermitIfNecessary` before calling `permit()`.

### Proof of Concept

```solidity
// Attacker front-runs victim's multicall
function attack(address token, address victim, address router, uint256 value, uint256 deadline, uint8 v, bytes32 r, bytes32 s) external {
    // Consume the nonce before victim's multicall lands
    IERC20Permit(token).permit(victim, router, value, deadline, v, r, s);
    // Victim's multicall([selfPermit(...), exactInputSingle(...)]) now reverts
    // because selfPermit calls permit() again with the same (now-spent) nonce
}
```
