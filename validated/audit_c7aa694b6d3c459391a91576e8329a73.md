### Title
`PriceVelocityGuardExtension` `lastUpdateBlock` Not Reset on Pool Unpause Inflates Allowed Velocity, Bypassing the Oracle-Manipulation Guard — (File: metric-periphery/contracts/extensions/PriceVelocityGuardExtension.sol)

---

### Summary

`PriceVelocityGuardExtension` stores `lastUpdateBlock` to enforce a per-block oracle mid-price velocity cap. Because swaps are blocked while a pool is paused, `lastUpdateBlock` is never advanced during the pause window. When the pool is unpaused, the first `beforeSwap` call computes `blockDiff = block.number − lastUpdateBlock`, which now spans the entire pause duration. The allowed-change formula `allowedSq = maxChange² × (1 + blockDiff)` grows proportionally, so a long pause silently multiplies the permitted price jump by `√(1 + pauseBlocks)`, effectively disabling the guard for the first post-pause swap.

---

### Finding Description

`PriceVelocityGuardExtension.beforeSwap` reads and writes `lastUpdateBlock` only on swap execution:

```solidity

uint128 prevMid   = s.lastMidPriceX64;
uint64  prevBlock = s.lastUpdateBlock;

s.lastMidPriceX64  = midPrice;
s.lastUpdateBlock  = uint64(block.number);   // ← only updated on swap

if (prevMid != 0) {
    uint64  maxChange = s.maxChangePerBlockE18;
    if (maxChange != 0) {
        uint256 blockDiff = block.number - prevBlock;   // ← includes pause window
        ...
        uint256 allowedSq = uint256(maxChange) * uint256(maxChange) * (1 + blockDiff);
        if (actualSq > allowedSq) revert PriceVelocityExceeded(...);
    }
}

The pool's `swap` function is gated by `whenNotPaused`, so no swap — and therefore no `lastUpdateBlock` write — can occur while `pauseLevel > 0`: [2](#0-1) 

Pause and unpause are performed by the factory without touching the extension's state: [3](#0-2) 

Neither `pausePool`, `unpausePool`, `protocolPausePool`, nor `protocolUnpausePool` calls `setLastMidPrice` on the extension. The extension has no hook that the pool calls on pause-level changes. [4](#0-3) 

---

### Impact Explanation

The velocity guard exists to protect LPs from oracle manipulation: if the oracle mid-price jumps by more than `maxChangePerBlockE18 × √(1 + blockDiff)` in a single swap, the swap is reverted. After a pause of `P` blocks the guard permits a jump of `maxChange × √(1 + P)` instead of the intended `maxChange × √2` (one-block gap). For a 10 000-block pause and `maxChange = 0.1 %/block`, the guard silently allows a `≈ 10 %` single-swap price move rather than `≈ 0.14 %`. An attacker who can influence the oracle (e.g., via a Pyth/Chainlink update or a manipulated `PriceProvider` source) can push the oracle to an extreme price immediately after the pool is unpaused and execute a swap at that price before the guard resets, draining LP value at a bad oracle quote. This satisfies the "bad-price execution" impact gate.

---

### Likelihood Explanation

The trigger requires only that the pool admin (or protocol owner) pause the pool for a non-trivial number of blocks and then unpause it — a routine operational action (security incident, upgrade, maintenance). No malicious admin intent is required; the bypass is automatic. Any third party who monitors the mempool for `unpausePool` transactions can front-run the first legitimate swap with a manipulated oracle update.

---

### Recommendation

When the pool is unpaused, reset `lastUpdateBlock` to `block.number` (and optionally `lastMidPriceX64` to the current oracle mid) so the guard treats the unpause moment as the new baseline. The cleanest fix is to add a `resetVelocityState(address pool)` function callable by the pool admin and invoke it as part of the unpause flow, or to treat `prevBlock == 0` / `prevBlock < lastPauseBlock` as a "no prior reference" condition inside `beforeSwap` and skip the velocity check for that one call.

---

### Proof of Concept

1. Deploy a pool with `PriceVelocityGuardExtension` configured: `maxChangePerBlockE18 = 1e15` (0.1 %/block).
2. At block 1 000, pool admin calls `factory.pausePool(pool)`. Last swap was at block 999; `lastUpdateBlock = 999`, `lastMidPriceX64 = P₀`.
3. At block 11 000 (10 000 blocks later), pool admin calls `factory.unpausePool(pool)`. Extension state unchanged: `lastUpdateBlock = 999`.
4. Attacker pushes oracle to `P₁ = 1.10 × P₀` (10 % above `P₀`).
5. Attacker calls `pool.swap(...)`. Inside `beforeSwap`:
   - `blockDiff = 11 000 − 999 = 10 001`
   - `allowedSq = (1e15)² × 10 002 = 1e28 × 10 002`
   - `allowed ≈ 3.16 × 1e17` (31.6 % change permitted)
   - `changeE18 = 0.10 × 1e18 = 1e17`; `actualSq = 1e34`
   - `1e34 < 1e28 × 10 002 ≈ 1.0002e32` → **guard passes**
6. Swap executes at the 10 %-manipulated oracle price; LPs receive less than fair value.

Without the pause, the same 10 % move at block 1 001 would produce `blockDiff = 2`, `allowedSq = 3e28`, `actualSq = 1e34 > 3e28` → **guard reverts**.
