### Title
Signature Validation Bypass via `skip_stateful_validations` Allows Invalid Invoke Transactions into the Mempool — (`crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The `skip_stateful_validations` function in the gateway's stateful validator skips the `__validate__` entry point (the cryptographic signature check) for any invoke transaction with nonce=1 when the sender address has *any* transaction present in the mempool. An unprivileged attacker who observes a legitimate `deploy_account` transaction in the public mempool can immediately submit a competing invoke transaction with nonce=1 carrying an invalid/forged signature for the same address. The gateway admits this transaction without ever verifying the signature, violating the invariant that all mempool-admitted transactions must have passed account-level signature validation.

---

### Finding Description

**Root cause — `skip_stateful_validations`** [1](#0-0) 

The function returns `true` (skip `__validate__`) when three conditions hold simultaneously:

1. The transaction is an `Invoke` with `nonce == 1`.
2. The on-chain account nonce is `0` (account not yet deployed).
3. `account_tx_in_pool_or_recent_block(sender_address)` returns `true`. [2](#0-1) 

The third condition is satisfied by the presence of **any** transaction from that address in the pool — including a `deploy_account` submitted by the legitimate user: [3](#0-2) 

**How `__validate__` is suppressed**

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` sets `execution_flags.validate = false`: [4](#0-3) 

Inside `StatefulValidator::perform_validations`, the invoke branch returns immediately after `perform_pre_validation_stage` without calling `__validate__`: [5](#0-4) 

`perform_pre_validation_stage` only checks nonce ordering (non-strict), fee bounds, and balance — it never verifies the cryptographic signature: [6](#0-5) 

**Stateless validation does not close the gap**

The stateless validator checks only the *length* of the signature, not its cryptographic validity: [7](#0-6) 

**Attack path**

1. Legitimate user broadcasts `deploy_account` + `invoke(nonce=1)` simultaneously (the documented UX flow).
2. Attacker observes the `deploy_account` in the public mempool for address `X`.
3. Attacker constructs `invoke(nonce=1, sender=X)` with a valid-length but cryptographically invalid signature and a fee higher than the legitimate invoke.
4. Gateway admits the attacker's transaction: stateless checks pass (length OK), nonce check passes (1 ≥ 0), `account_tx_in_pool_or_recent_block` returns `true` (deploy_account is present), `__validate__` is skipped.
5. If fee escalation is enabled, the attacker's invalid invoke replaces the legitimate one: [8](#0-7) 

6. The legitimate user's first invoke is evicted from the mempool. The attacker's invalid transaction occupies the nonce=1 slot and will revert on-chain.

---

### Impact Explanation

**Broken invariant**: Every invoke transaction admitted to the mempool must have passed account-level signature validation (`__validate__`). This invariant is violated for any invoke with nonce=1 targeting an account whose `deploy_account` is visible in the mempool.

**Concrete wrong value**: An `InternalRpcTransaction` with an invalid signature is stored in the mempool and forwarded to the batcher, where it will revert during block execution. The legitimate transaction with the same nonce is displaced.

This matches the allowed impact: *"High. Mempool/gateway/RPC admission accepts invalid transactions or rejects valid transactions before sequencing."*

---

### Likelihood Explanation

- The `deploy_account` + `invoke(nonce=1)` UX flow is explicitly supported and documented in integration tests.
- The mempool is public; any observer can detect a `deploy_account` and race to submit a competing invoke.
- No privileged access, no prior relationship with the target account, and no special tooling are required — only the ability to call the public gateway `add_tx` endpoint.
- The attack is amplified when fee escalation is enabled (`enable_fee_escalation = true`), making replacement deterministic rather than a race.

---

### Recommendation

Replace the coarse `account_tx_in_pool_or_recent_block` check with a check that specifically verifies a `deploy_account` transaction (not just any transaction) is pending for the sender address. Alternatively, expose a dedicated `has_pending_deploy_account(address)` query on the mempool so the gateway can confirm the exact transaction type before skipping `__validate__`.

---

### Proof of Concept

```
# Prerequisites: gateway running, fee escalation enabled.

# Step 1 – Legitimate user submits deploy_account + invoke(nonce=1) simultaneously.
POST /gateway/add_transaction  { deploy_account for address 0xABCD }
POST /gateway/add_transaction  { invoke nonce=1, sender=0xABCD, valid_sig }

# Step 2 – Attacker observes deploy_account in mempool, crafts competing invoke.
# Signature bytes have correct length (e.g. 2 Felts) but wrong values.
POST /gateway/add_transaction  {
  invoke nonce=1, sender=0xABCD,
  signature=[0x1111111..., 0x2222222...],   # invalid, but correct length
  tip=<higher than legitimate invoke>
}

# Expected (broken) result:
# - Gateway returns 200 OK with a tx_hash.
# - Mempool replaces the legitimate invoke with the attacker's invalid one.
# - Legitimate user's invoke is gone; attacker's invalid tx reverts on-chain.
```

The gateway call path that admits the invalid transaction: [9](#0-8) 

`extract_state_nonce_and_run_validations` → `run_pre_validation_checks` → `skip_stateful_validations` returns `true` → `run_validate_entry_point` with `validate=false` → `__validate__` never called → transaction forwarded to mempool. [10](#0-9)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
