### Title
Unprivileged `TransferToGasKey` Inflates Victim's Gas Key Balance Above `MAX_BALANCE_TO_BURN`, Permanently Blocking `DeleteKey` and `DeleteAccount` — (File: `runtime/runtime/src/access_keys.rs`)

---

### Summary

`check_actor_permissions` in `runtime/runtime/src/actions.rs` deliberately exempts `Action::TransferToGasKey` from the `actor_id == account_id` guard, allowing any unprivileged account to send a `TransferToGasKey` action targeting another account's gas key. `action_transfer_to_gas_key` adds the caller-controlled `deposit` to `gas_key_info.balance` with no upper-bound check. Once the balance exceeds `GasKeyInfo::MAX_BALANCE_TO_BURN` (1 NEAR), both `delete_gas_key` and `action_delete_account` revert with `GasKeyBalanceTooHigh`, permanently blocking the victim from deleting their gas key or their account. The victim's only recovery path — `WithdrawFromGasKey` — can be front-run by the attacker with another `TransferToGasKey`, making the block indefinitely sustainable at near-zero marginal cost once the victim's gas key balance is close to 1 NEAR.

---

### Finding Description

**Root cause — missing actor restriction on `TransferToGasKey`:**

`check_actor_permissions` enforces `actor_id == account_id` for `DeleteKey`, `AddKey`, `WithdrawFromGasKey`, and others, but explicitly places `TransferToGasKey` in the unrestricted arm alongside `Transfer` and `FunctionCall`:

```rust
Action::CreateAccount(_)
| Action::FunctionCall(_)
| Action::Transfer(_)
| Action::TransferToGasKey(_) => (),   // ← no actor check
``` [1](#0-0) 

**Unchecked balance accumulation in `action_transfer_to_gas_key`:**

The function looks up the gas key on `account_id` (the receipt receiver, not the sender) and unconditionally adds `action.deposit` to `gas_key_info.balance` with no ceiling:

```rust
gas_key_info.balance = gas_key_info.balance.checked_add(action.deposit).ok_or_else(|| { ... })?;
``` [2](#0-1) 

**Deletion gate in `delete_gas_key`:**

```rust
if gas_key_info.balance > GasKeyInfo::MAX_BALANCE_TO_BURN {
    result.result = Err(ActionErrorKind::GasKeyBalanceTooHigh { ... }.into());
    return Ok(());
}
``` [3](#0-2) 

`MAX_BALANCE_TO_BURN` is exactly 1 NEAR: [4](#0-3) 

**Same gate for account deletion** — `action_delete_account` sums all gas key balances and applies the same check: [5](#0-4) 

**`WithdrawFromGasKey` is restricted to the account owner** (`actor_id == account_id` is enforced), so the attacker cannot drain the balance themselves. The victim can call `WithdrawFromGasKey`, but the attacker can front-run that transaction with another `TransferToGasKey` to push the balance back above 1 NEAR. The `TransferToGasKey` action is also callable from contracts via the `promise_batch_action_transfer_to_gas_key` host function, enabling automated front-running. [6](#0-5) 

---

### Impact Explanation

- **`DeleteKey` is permanently blocked** for any gas key whose balance exceeds 1 NEAR.
- **`DeleteAccount` is permanently blocked** when the sum of all gas key balances on the account exceeds 1 NEAR.
- Blocking account deletion prevents the victim from reclaiming their storage-staking deposit (locked NEAR proportional to account storage usage), which is a direct, quantifiable financial loss.
- The victim's gas key remains usable for transactions, but the key and account lifecycle operations are frozen.

---

### Likelihood Explanation

- Any unprivileged account can execute this attack with no special privileges.
- The one-time cost to push a gas key from 0 to just above 1 NEAR is slightly over 1 NEAR. If the victim's gas key balance is already near 1 NEAR (e.g., 0.999 NEAR), the attacker only needs to send 1 yoctoNEAR — essentially free.
- For account deletion, the check is on the **sum** of all gas key balances. A victim with multiple gas keys each holding 0.1 NEAR (total = 1 NEAR) can be blocked by a 1 yoctoNEAR deposit to any single gas key.
- The attack can be automated via a smart contract using `promise_batch_action_transfer_to_gas_key`, enabling reliable front-running of the victim's `WithdrawFromGasKey` transactions.

---

### Recommendation

1. **Restrict `TransferToGasKey` to the account owner** by adding it to the `actor_id == account_id` arm in `check_actor_permissions`. This mirrors the restriction already applied to `WithdrawFromGasKey`.
2. Alternatively, add an upper-bound check inside `action_transfer_to_gas_key` that rejects any deposit that would push `gas_key_info.balance` above `MAX_BALANCE_TO_BURN`.

---

### Proof of Concept

```
1. Alice (victim) has gas key GK with balance = 0.999 NEAR.
2. Bob (attacker) sends tx: signer=Bob, receiver=Alice,
   actions=[TransferToGasKey { public_key: GK, deposit: 2 yoctoNEAR }].
   → GK.balance = 1.000...001 NEAR (> MAX_BALANCE_TO_BURN).
3. Alice sends tx: DeleteKey { public_key: GK }.
   → Runtime calls delete_gas_key → GK.balance > 1 NEAR → GasKeyBalanceTooHigh → revert.
4. Alice sends tx: WithdrawFromGasKey { public_key: GK, amount: 2 yoctoNEAR }.
   → Succeeds; GK.balance = 0.999 NEAR.
5. Bob front-runs step 4 (or step 3) with another TransferToGasKey { deposit: 2 yoctoNEAR }.
   → GK.balance = 1.000...001 NEAR again.
6. Alice can never delete GK or her account.
   Each round costs Bob 2 yoctoNEAR (negligible).
```

    state_update: &mut TrieUpdate,
    result: &mut ActionResult,
    account_id: &AccountId,
    action: &TransferToGasKeyAction,
) -> Result<(), RuntimeError> {
    let Some(mut access_key) = get_access_key(state_update, account_id, &action.public_key)? else {
        result.result = Err(ActionErrorKind::GasKeyDoesNotExist {
            account_id: account_id.clone(),
            public_key: Box::new(action.public_key.clone()),
        }
        .into());
        return Ok(());
    };
    let Some(gas_key_info) = access_key.gas_key_info_mut() else {
        // Key exists but is not a gas key
        result.result = Err(ActionErrorKind::GasKeyDoesNotExist {
            account_id: account_id.clone(),
            public_key: Box::new(action.public_key.clone()),
        }
        .into());
        return Ok(());
    };

    gas_key_info.balance = gas_key_info.balance.checked_add(action.deposit).ok_or_else(|| {
        RuntimeError::StorageError(StorageError::StorageInconsistentState(
            "gas key balance integer overflow".to_string(),
        ))
    })?;
    set_access_key(state_update, account_id.clone(), action.public_key.clone(), &access_key);
    Ok(())
