### Title
Single Malicious Participant Can Permanently Abort DKG for All Honest Parties via False `broadcast_success` Vote — (`File: src/dkg.rs`)

---

### Summary

The `do_keyshare` function (used by keygen, reshare, and refresh) concludes with a `broadcast_success` call that requires **unanimous** agreement from all participants. A single malicious participant included in the participant set can broadcast `(false, session_id)` in this final round, causing every honest party to abort the DKG permanently. Because there is no mechanism to identify or exclude the lying participant, the attack can be repeated across every subsequent DKG attempt, making key generation permanently unavailable for honest parties.

---

### Finding Description

`do_keyshare` in `src/dkg.rs` is the shared implementation for `keygen`, `reshare`, and `refresh`. Its final step is:

```rust
// Step 5.4 and Step 5.5
broadcast_success(&mut chan, &participants, me, session_id).await?;
```

`broadcast_success` always broadcasts `(true, session_id)` on behalf of the honest caller, then collects the broadcast outputs of all other participants and enforces unanimity:

```rust
async fn broadcast_success(
    chan: &mut SharedChannel,
    participants: &ParticipantList,
    me: Participant,
    session_id: HashOutput,
) -> Result<(), ProtocolError> {
    let vote_list = do_broadcast(chan, participants, me, (true, session_id)).await?;
    let vote_list = vote_list
        .into_vec_or_none()
        .ok_or_else(|| ProtocolError::AssertionFailed("vote_list is empty".to_string()))?;

    if !vote_list.iter().all(|(_, ref sid)| sid == &session_id) {
        return Err(ProtocolError::AssertionFailed(
            "A participant broadcast the wrong session id. Aborting Protocol!".to_string(),
        ));
    }

    if !vote_list.iter().all(|&(boolean, _)| boolean) {
        return Err(ProtocolError::AssertionFailed(
            "A participant seems to have failed its checks. Aborting Protocol!".to_string(),
        ));
    }
    Ok(())
}
```

The unanimity check at line 329 (`vote_list.iter().all(|&(boolean, _)| boolean)`) means that if **any** participant's delivered broadcast value is `false`, every honest party returns `Err` and discards all key material computed in the preceding five rounds.

A malicious participant controls their own protocol binary. Instead of calling `broadcast_success` (which always sends `true`), they can send the raw message `(false, session_id)` at the correct waitpoint. The underlying `do_broadcast` / `reliable_broadcast_receive_all` echo-broadcast protocol will faithfully deliver this `false` value to every honest party, because the echo-broadcast guarantees agreement on whatever the sender actually transmitted — it does not validate the semantic content of the payload.

After delivery, the unanimity check fires, every honest party aborts, and all key material is discarded. Because the participant set is fixed by the caller and there is no exclusion mechanism, the attacker can repeat this in every subsequent DKG attempt.

---

### Impact Explanation

**High — Permanent denial of key generation, reshare, and refresh for honest parties.**

All three public DKG entry points (`do_keygen`, `do_reshare`, and the refresh path) call `do_keyshare`, which calls `broadcast_success`. A single malicious participant in the participant list can abort all three operations indefinitely. Honest parties complete five rounds of expensive cryptographic work (polynomial generation, commitment broadcast, proof-of-knowledge verification, private share exchange, share validation) before the abort, and must restart from scratch each time. No threshold signature, presignature, or key share can ever be produced while the attacker remains in the participant set.

---

### Likelihood Explanation

**High.** The attack requires only that the adversary control one participant slot in the DKG session — a condition that is explicitly within the documented BFT threshold (`MaxFaulty = (N−1)/3`). The protocol is supposed to tolerate up to `MaxFaulty` faulty participants, yet a single faulty participant suffices to abort the entire DKG. The malicious action is trivial: send one message with the boolean flipped to `false`. No cryptographic capability, no key material, and no coordination with other parties is needed. The attack is repeatable without limit.

---

### Recommendation

1. **Remove the unanimity requirement from `broadcast_success`.** The final success vote should require only a quorum (≥ threshold, or ≥ N − MaxFaulty) of `true` votes rather than all N. Participants who vote `false` should be identified and logged, but should not be able to veto the entire protocol when honest parties hold a sufficient majority.

2. **Attribute and exclude lying participants.** Because `do_broadcast` delivers the sender's identity alongside the payload, the `false` voter is identifiable. The protocol should surface this identity so the application layer can exclude the malicious participant from future sessions.

3. **Align the success-vote semantics with the BFT model.** The echo-broadcast layer already tolerates `MaxFaulty` Byzantine participants; the application-layer check should not impose a stricter unanimity requirement that negates this tolerance.

---

### Proof of Concept

**Setup:** N = 4 participants `{P1, P2, P3, P4}`, threshold = 2, `MaxFaulty = 1`. `P4` is malicious.

1. `P4` participates honestly through all five DKG rounds: session-ID broadcast (Round 1), polynomial/commitment generation (Round 2), commitment-hash exchange (Round 2.9), commitment+proof broadcast (Round 3–4), and private share exchange + validation (Round 4–5).

2. At the `broadcast_success` waitpoint, instead of sending `(true, session_id)`, `P4` sends the raw message `(false, session_id)` to all other participants at the same waitpoint used by `do_broadcast`.

3. The echo-broadcast protocol in `reliable_broadcast_receive_all` collects echo and ready votes for `P4`'s session slot. Because `P4` sends `false` consistently to all parties, the echo-broadcast delivers `(false, session_id)` for `P4`'s slot to every honest party.

4. In `broadcast_success`, the collected `vote_list` contains `(false, session_id)` for `P4`. The check:
   ```rust
   if !vote_list.iter().all(|&(boolean, _)| boolean) { return Err(...) }
   ```
   evaluates to `Err`, and `P1`, `P2`, `P3` all abort, discarding their computed key shares.

5. `P4` repeats step 2 in every subsequent DKG attempt. Key generation is permanently unavailable. [1](#0-0) [2](#0-1) [3](#0-2) [4](#0-3)
