### Title
Missing Caller Access Control in `update_rewards` Allows Anyone to Permanently Freeze Staker Yield - (File: `src/staking/staking.cairo`)

---

### Summary

The `update_rewards` function in the Staking contract enforces a single-call-per-block invariant via a global `last_reward_block` storage variable, but performs **no check on who the caller is**. The protocol specification explicitly states access should be restricted to "Only starkware sequencer," yet the implementation omits this guard entirely. Any unprivileged address can call `update_rewards` with `disable_rewards: true` on every block, consuming the one-call-per-block slot without distributing rewards, permanently starving all stakers and delegators of yield.

---

### Finding Description

`IStakingRewardsManager::update_rewards` is the entry point through which block rewards are calculated and distributed to stakers and their delegation pools. The function signature accepts a caller-supplied `staker_address` and a `disable_rewards` flag:

```rust
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
)
```

The implementation at `src/staking/staking.cairo` lines 1449–1507 performs the following checks:

1. `general_prerequisites()` — only asserts the contract is not paused. [1](#0-0) 
2. `current_block_number > self.last_reward_block.read()` — enforces one call per block globally. [2](#0-1) 
3. Staker existence and activity checks. [3](#0-2) 

After passing these checks, `last_reward_block` is unconditionally written to the current block number **before** the `disable_rewards` branch: [4](#0-3) 

```rust
// Update last block rewards.
self.last_reward_block.write(current_block_number);

if disable_rewards || self.is_pre_consensus() {
    return;
}
```

There is **no check** that `get_caller_address()` equals the sequencer or any privileged role. The protocol specification explicitly requires: [5](#0-4) 

```
#### access control
Only starkware sequencer.
```

Because `last_reward_block` is a single global storage slot (not per-staker), a single call to `update_rewards` for any valid staker with `disable_rewards: true` consumes the entire block's reward slot for **all** stakers. [6](#0-5) 

---

### Impact Explanation

An attacker who calls `update_rewards(any_active_staker, disable_rewards: true)` on every block:

- Sets `last_reward_block` to the current block number with no reward distribution.
- Causes every subsequent legitimate sequencer call in the same block to revert with `REWARDS_ALREADY_UPDATED`.
- Prevents `_update_rewards` from ever executing, so `unclaimed_rewards_own` for stakers and pool reward balances never increase.
- Stakers and all delegators across all pools permanently receive zero yield.

This matches the allowed impact: **High — Permanent freezing of unclaimed yield**.

The pool reward path that is blocked: [7](#0-6) 

---

### Likelihood Explanation

The function is public with no role gate. Any address on Starknet can call it. The only cost is gas per block. The attacker needs only one valid `staker_address` (trivially obtained from on-chain events) and the ability to submit a transaction each block. There is no economic barrier and no slashing risk.

---

### Recommendation

Add a caller check at the top of `update_rewards` that asserts `get_caller_address()` equals the registered sequencer address (or an equivalent privileged role stored in contract state), consistent with the specification's stated access control. This mirrors the pattern already used in `reward_supplier.cairo` for `update_unclaimed_rewards_from_staking_contract` and `claim_rewards`, which both assert `get_caller_address() == self.staking_contract.read()`. [8](#0-7) 

---

### Proof of Concept

1. Deploy the system normally; wait until consensus rewards are active and a staker has a non-zero balance effective for the current epoch.
2. From any arbitrary address (attacker), call:
   ```
   IStakingRewardsManagerDispatcher { contract_address: staking_contract }
       .update_rewards(staker_address: any_active_staker, disable_rewards: true);
   ```
3. Observe that `last_reward_block` is now set to the current block and no rewards were distributed.
4. The legitimate sequencer's call to `update_rewards` in the same block reverts with `REWARDS_ALREADY_UPDATED`.
5. Repeat step 2 every block. Staker `unclaimed_rewards_own` and all pool reward balances remain at zero indefinitely, confirmed by querying `staker_info_v1` and `pool_member_info_v1`.

The existing test suite inadvertently demonstrates the absence of a caller guard: `update_rewards` is called throughout tests from arbitrary test addresses with no `cheat_caller_address` to a privileged role. [9](#0-8)
