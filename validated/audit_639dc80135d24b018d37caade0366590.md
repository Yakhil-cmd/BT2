### Title
Reward Tokens Permanently Locked in Pool Contract When Total Stake Falls Below `min_delegation_for_rewards` — (File: `src/pool/pool.cairo`)

---

### Summary

When a pool's total delegated stake is below `min_delegation_for_rewards`, the staking contract still forwards STRK reward tokens to the pool contract, but the pool records zero rewards-per-unit in its cumulative trace. Those tokens are permanently locked in the pool contract with no recovery path.

---

### Finding Description

In `pool.cairo`, `compute_rewards_per_unit` (lines 967–978) returns `Zero::zero()` when `total_stake < self.min_delegation_for_rewards.read()`:

```cairo
fn compute_rewards_per_unit(
    self: @ContractState, staking_rewards: Amount, total_stake: Amount,
) -> Index {
    // Return zero if the total stake is too small, to avoid overflow below.
    if total_stake < self.min_delegation_for_rewards.read() {
        return Zero::zero();
    }
    ...
}
``` [1](#0-0) 

The developer comment immediately above this function explicitly acknowledges the consequence:

> **Note**: Delegation rewards lost when pool balance is less than `min_delegation_for_rewards`. **The staking contract continues to forward `pool_rewards` to the pool contract even in this case.** [2](#0-1) 

The staking contract's `send_rewards_to_delegation_pool` unconditionally transfers tokens to the pool contract before calling `update_rewards_from_staking_contract`:

```cairo
fn send_rewards_to_delegation_pool(
    ref self: ContractState,
    staker_address: ContractAddress,
    pool_address: ContractAddress,
    amount: Amount,
    token_dispatcher: IERC20Dispatcher,
) {
    token_dispatcher.checked_transfer(recipient: pool_address, amount: amount.into());
    ...
}
``` [3](#0-2) 

The pool's `update_rewards_from_staking_contract` then calls `compute_rewards_per_unit`, which returns zero, so the `cumulative_rewards_trace` is incremented by zero:

```cairo
fn update_rewards_from_staking_contract(
    ref self: ContractState, rewards: Amount, pool_balance: Amount,
) {
    ...
    self.cumulative_rewards_trace.insert(
        key: self.get_current_epoch(),
        value: last + self.compute_rewards_per_unit(
            staking_rewards: rewards, total_stake: pool_balance,
        ),
    );
}
``` [4](#0-3) 

Because the cumulative trace does not increase, `calculate_rewards` computes zero for every pool member. The STRK tokens are physically present in the pool contract's balance but are unclaimable. There is no `sweep`, `recover`, or admin-rescue function in the pool contract.

The STRK minimum threshold is `10^18` (1 STRK):

```cairo
pub(crate) const STRK_CONFIG: TokenRewardsConfig = TokenRewardsConfig {
    decimals: 18, min_for_rewards: 10_u128.pow(18), base_value: 10_u128.pow(28),
};
``` [5](#0-4) 

---

### Impact Explanation

**High — Permanent freezing of unclaimed yield.**

Every epoch in which a pool's total delegated stake is below `min_delegation_for_rewards`, the STRK reward tokens forwarded by the staking contract are deposited into the pool contract's ERC-20 balance but are never credited to any pool member's claimable rewards. Because the pool exposes no recovery function, those tokens are permanently locked. Over many epochs, the locked amount accumulates proportionally to the pool's share of global staking rewards.

---

### Likelihood Explanation

**Medium.** The condition is reachable by any pool whose total delegated stake is below 1 STRK (10^18 FRI). This is realistic for:
- Newly created pools before delegators join.
- Pools whose delegators have all exited intent but not yet executed the action (their balance is removed from the active stake but the pool contract still receives rewards forwarded by the staking contract for the epoch).
- BTC pools, whose `min_for_rewards` threshold may differ and whose decimal scale makes small-balance scenarios more likely.

No privileged access is required; the condition is triggered automatically by the staking contract's reward distribution logic.

---

### Recommendation

Two complementary fixes:

1. **In the staking contract**: Before calling `send_rewards_to_delegation_pool`, check whether the pool's delegated balance is below the minimum threshold. If so, do not transfer the reward tokens to the pool (or retain them in the staking contract and credit them to the staker's own rewards, or simply skip the transfer).

2. **In the pool contract**: If the staking contract cannot be changed, add a guard in `update_rewards_from_staking_contract` that, when `compute_rewards_per_unit` would return zero, transfers the received reward tokens back to the staking contract (or to a designated recovery address) rather than silently absorbing them.

---

### Proof of Concept

1. Deploy a pool with a single delegator whose stake is `S < 10^18` STRK (e.g., `S = 1`).
2. Advance one epoch and trigger attestation so the staking contract computes pool rewards `R > 0`.
3. The staking contract calls `send_rewards_to_delegation_pool(pool, R)` → `R` STRK tokens are transferred to the pool contract.
4. The staking contract calls `pool.update_rewards_from_staking_contract(R, S)`.
5. Inside `compute_rewards_per_unit`: `S < min_delegation_for_rewards` → returns `0`.
6. `cumulative_rewards_trace` is incremented by `0`.
7. Delegator calls `claim_rewards` → `calculate_rewards` returns `0` → delegator receives nothing.
8. `token.balance_of(pool)` equals `R` (the locked tokens), and no function in the pool contract can recover them.

            self: @ContractState, staking_rewards: Amount, total_stake: Amount,

            // Return zero if the total stake is too small, to avoid overflow below.
            if total_stake < self.min_delegation_for_rewards.read() {
