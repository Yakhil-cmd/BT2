### Title
Missing Caller Restriction on `update_rewards()` Allows Any Address to Permanently Freeze Staker Yield - (File: src/staking/staking.cairo)

### Summary
`update_rewards()` in `staking.cairo` is specified as "Only starkware sequencer" but has **no caller check** in the implementation. Any unprivileged address can call it with `disable_rewards: true`, which advances `last_reward_block` to the current block without distributing rewards. The legitimate sequencer call for that same block then reverts with `REWARDS_ALREADY_UPDATED`, permanently destroying the block's yield for all stakers.

### Finding Description
The spec at `docs/spec.md:1644-1645` states:

> **access control**: Only starkware sequencer.

The implementation at `src/staking/staking.cairo:1447-1507` enforces no such restriction. The function body only checks:

1. `general_prerequisites()` — contract-pause guard
2. `current_block_number > self.last_reward_block.read()` — one call per block
3. Staker exists and is active with non-zero balance

After those checks pass, line 1485 unconditionally writes `last_reward_block`:

```cairo
self.last_reward_block.write(current_block_number);
```

Then line 1487 branches on the caller-supplied `disable_rewards` flag:

```cairo
if disable_rewards || self.is_pre_consensus() {
    return;
}
```

An attacker calls `update_rewards(any_active_staker, disable_rewards: true)` on every block. The write to `last_reward_block` occurs before the early return, so the block is permanently marked as processed. The sequencer's subsequent call for the same block reverts with `REWARDS_ALREADY_UPDATED`.

### Impact Explanation
Every block for which the attacker front-runs the sequencer is permanently lost — the protocol has no mechanism to retroactively distribute skipped block rewards. Stakers and pool members accumulate zero `unclaimed_rewards_own` and zero pool sigma increments for those blocks. This constitutes **permanent freezing of unclaimed yield** for all stakers.

### Likelihood Explanation
Starknet transaction fees are low. The attacker needs only to submit one transaction per block targeting any single active staker (the `staker_address` parameter is arbitrary — any valid active staker suffices to satisfy the checks). The attack is fully automatable, requires no capital, and can be sustained indefinitely.

### Recommendation
Add a sequencer-only guard at the top of `update_rewards()`, analogous to the guards already present on `update_current_epoch_block_rewards()` and `update_unclaimed_rewards_from_staking_contract()`:

```cairo
assert!(
    get_caller_address() == self.sequencer_address.read(),
    "{}",
    Error::CALLER_IS_NOT_SEQUENCER,
);
```

Alternatively, restrict via the existing roles component (e.g., `self.roles.only_sequencer()`), consistent with how other privileged functions are guarded throughout the contract.

### Proof of Concept
1. Deploy the system and advance past the consensus-rewards activation epoch so `is_pre_consensus()` returns `false`.
2. Stake as `staker_A` and wait K epochs for balance to become effective.
3. On every new block, before the sequencer acts, call:
   ```cairo
   IStakingRewardsManager(staking).update_rewards(staker_A, disable_rewards: true);
   ```
4. Observe that `last_reward_block` is updated to the current block (line 1485) but no rewards are credited (early return at line 1487).
5. The sequencer's call for the same block reverts with `REWARDS_ALREADY_UPDATED` (line 1454-1458).
6. After N blocks, `staker_A.unclaimed_rewards_own` remains 0 despite N blocks of entitled yield.
