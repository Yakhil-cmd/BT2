### Title
`max_nonce_for_validation_skip` Config Field Declared But Never Consumed by `skip_stateful_validations` — Gateway Validation-Skip Logic Ignores Operator Configuration - (File: `crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

`StatefulTransactionValidatorConfig` declares and serializes a field `max_nonce_for_validation_skip` that is intended to bound the nonce range for which the gateway skips `__validate__` on the deploy-account + invoke UX path. The free function `skip_stateful_validations` — the only site that implements this logic — never reads the config; it hardcodes `Nonce(Felt::ONE)`. The config field is therefore a dead parameter: any operator-set value is silently ignored, and the gateway always behaves as if the field equals `1`.

### Finding Description

`StatefulTransactionValidatorConfig` declares `max_nonce_for_validation_skip` with a default of `Nonce(Felt::ONE)`: [1](#0-0) 

The field is serialized, documented in the config schema, and present in deployment app-configs: [2](#0-1) 

`run_pre_validation_checks` calls the free function `skip_stateful_validations` without passing `self.config`: [3](#0-2) 

`skip_stateful_validations` hardcodes `Nonce(Felt::ONE)` and never references the config: [4](#0-3) 

The reference implementation in `native_blockifier/src/py_validator.rs` shows the intended design — it reads `self.max_nonce_for_validation_skip` and applies a range check `Nonce(Felt::ONE) <= tx_nonce && tx_nonce <= self.max_nonce_for_validation_skip`: [5](#0-4) 

The Apollo gateway's Rust path never replicates this. A `grep` across all `.rs` files confirms `max_nonce_for_validation_skip` appears only in `apollo_gateway_config/src/config.rs` (declaration/serialization) and `native_blockifier/src/py_validator.rs` (correct usage) — never in `apollo_gateway/src/stateful_transaction_validator.rs`.

### Impact Explanation

Two distinct failure modes arise when an operator sets `max_nonce_for_validation_skip` to a non-default value:

**Case A — operator sets the field to `0` to disable the skip feature entirely:**
`skip_stateful_validations` still returns `true` (skip `__validate__`) for any invoke with `tx.nonce() == 1` and `account_nonce == 0` that has a deploy-account in the mempool. The gateway admits the transaction without running the account's `__validate__` entry point, violating the operator's explicit security policy. This satisfies **High: Mempool/gateway admission accepts invalid transactions before sequencing**.

**Case B — operator sets the field to `N > 1` to extend the UX window:**
`skip_stateful_validations` only skips for nonce exactly `1`; invoke transactions with nonces `2 … N` from undeployed accounts are sent through `__validate__`, which fails because the account does not yet exist on-chain. Valid transactions that the operator intended to admit are rejected. This satisfies **High: Mempool/gateway admission rejects valid transactions before sequencing**.

### Likelihood Explanation

The default value (`0x1`) coincidentally matches the hardcoded literal, so the bug is invisible with a stock deployment. Likelihood rises whenever an operator consults the documented field and adjusts it — a plausible operational action given the field is public, described, and present in every deployment config file. The `native_blockifier` path consuming the field correctly further increases the chance that operators expect the Apollo gateway to honour it.

### Recommendation

Convert `skip_stateful_validations` into a method on `StatefulTransactionValidator` (or pass `max_nonce_for_validation_skip` as a parameter) and replace the hardcoded equality check with the range check used in `py_validator.rs`:

```rust
// Instead of:
if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {

// Use:
if Nonce(Felt::ONE) <= tx.nonce()
    && tx.nonce() <= self.config.max_nonce_for_validation_skip
    && account_nonce == Nonce(Felt::ZERO)
{
```

### Proof of Concept

1. Deploy a sequencer node with `max_nonce_for_validation_skip = 0x0` in `gateway_config.static_config.stateful_tx_validator_config`.
2. Submit a `DeployAccount` transaction for address `A` (nonce 0). It enters the mempool.
3. Submit an `Invoke` transaction from `A` with nonce `1` and an **invalid signature**.
4. `skip_stateful_validations` evaluates `tx.nonce() == Nonce(Felt::ONE)` → `true`; `account_nonce == Nonce(Felt::ZERO)` → `true`; mempool contains the deploy-account → returns `true`.
5. `run_validate_entry_point` is called with `skip_validate = true`, so `execution_flags.validate = false` and `__validate__` is never invoked.
6. The gateway returns success and forwards the transaction to the mempool — despite the operator having set `max_nonce_for_validation_skip = 0` to prevent exactly this bypass. [6](#0-5)

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
