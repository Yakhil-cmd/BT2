### Title
Circuit-Breaker `TurnBridge` Is a No-Op Stub ŌĆö Bridge State Is Never Written and Permission Guard Is Absent ŌĆö (`x/cronos/keeper/msg_server.go`)

### Summary
`msgServer.TurnBridge` is an unimplemented stub. It accepts any sender, performs no authorization check, writes nothing to state, and returns `(nil, nil)`. The gravity-bridge circuit-breaker is therefore permanently non-functional: no one ŌĆö including the admin or a permissioned address ŌĆö can ever disable the bridge via this message.

### Finding Description

`UpdateTokenMapping` (the sibling restricted message) correctly gates execution:

```go
// x/cronos/keeper/msg_server.go:72-75
if !k.HasPermission(ctx, msg.GetSigners(), CanChangeTokenMapping) {
    return nil, errors.Wrap(sdkerrors.ErrUnauthorized, "msg sender is not authorized")
}
``` [1](#0-0) 

`TurnBridge`, which is supposed to be equally restricted, is:

```go
// x/cronos/keeper/msg_server.go:84-87
func (k msgServer) TurnBridge(goCtx context.Context, msg *types.MsgTurnBridge) (*types.MsgTurnBridgeResponse, error) {
    return nil, nil
}
``` [2](#0-1) 

The permission system (`CanTurnBridge`) is fully defined and working in isolation: [3](#0-2) 

`HasPermission` is correctly implemented and tested: [4](#0-3) 

The ADR explicitly states `MsgTurnBridge` must be restricted and is the emergency circuit-breaker for the gravity bridge: [5](#0-4) 

`ValidateBasic` only checks that `Sender` is a valid bech32 address ŌĆö no permission check there either: [6](#0-5) 

### Impact Explanation

Two distinct impacts:

1. **Authorization bypass (High):** Any unprivileged address can submit `MsgTurnBridge` and receive a `200 OK` (`nil, nil`) response. The permission guard (`CanTurnBridge`) is entirely absent from the execution path.

2. **Permanent circuit-breaker failure (High):** Because the function writes nothing to state, the gravity bridge can never be disabled by anyone ŌĆö admin, permissioned address, or governance. In a bridge exploit scenario, the intended emergency stop mechanism is silently inoperative.

Both map to the allowed High impact: *"Bypass of Cronos admin, governance authority, permission, token-mapping, bridge, block-list, or module-account authorization checks"* and *"Permanent or long-lived inability for honest users or validators to process valid transactions, bridge/conversion flows."*

### Likelihood Explanation

The stub is reachable by any funded account via a standard Cosmos SDK transaction. No special privileges, leaked keys, or external assumptions are required. The call path is `MsgTurnBridge.ValidateBasic()` (passes for any valid bech32 sender) ŌåÆ `msgServer.TurnBridge()` (returns nil, nil).

### Recommendation

Implement `TurnBridge` following the same pattern as `UpdateTokenMapping`:

```go
func (k msgServer) TurnBridge(goCtx context.Context, msg *types.MsgTurnBridge) (*types.MsgTurnBridgeResponse, error) {
    ctx := sdk.UnwrapSDKContext(goCtx)
    if !k.HasPermission(ctx, msg.GetSigners(), CanTurnBridge) {
        return nil, errors.Wrap(sdkerrors.ErrUnauthorized, "msg sender is not authorized")
    }
    // update bridge-enabled param or dedicated KV key
    params := k.GetParams(ctx)
    params.EnableGravity = msg.Enable
    if err := k.SetParams(ctx, params); err != nil {
        return nil, err
    }
    return &types.MsgTurnBridgeResponse{}, nil
}
```

func (k Keeper) HasPermission(ctx sdk.Context, accounts []sdk.AccAddress, permissionsToCheck uint64) bool {
	// case when no permission is needed
	if permissionsToCheck == 0 {
		return true
	}
	admin := k.GetParams(ctx).CronosAdmin
	for _, account := range accounts {
		if admin == account.String() {
			return true
		}
		permission := k.GetPermissions(ctx, account)
		if permission&permissionsToCheck == permissionsToCheck {
			return true
		}
	}

	return false
```
