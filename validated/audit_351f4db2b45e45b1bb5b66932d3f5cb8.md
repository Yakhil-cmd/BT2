### Title
Malicious Participant Can Corrupt CKD Output by Sending Contribution Computed Under Arbitrary `app_pk` — (`src/confidential_key_derivation/protocol.rs`)

### Summary

The CKD coordinator aggregates participant contributions `(norm_big_y, norm_big_c)` with no cryptographic binding to the session's `app_pk`. A single malicious participant can send a contribution computed under any `app_pk'` (including one from a prior session), causing the final `CKDOutput` to be permanently unusable: `unmask(app_sk)` returns a garbage point instead of `msk · H(pk ‖ app_id)`.

---

### Finding Description

In `compute_signature_share`, each participant computes:

```
big_c = x_i · H(pk ‖ app_id) + app_pk · y_i
norm_big_c = λ_i · big_c
```

and sends `(norm_big_y, norm_big_c)` to the coordinator. [1](#0-0) 

The coordinator in `do_ckd_coordinator` receives these values and blindly sums them:

```rust
norm_big_y += participant_output.big_y();
norm_big_c += participant_output.big_c();
``` [2](#0-1) 

The underlying `recv_from_others` helper only checks that the message originates from a known participant — it performs no validation of message content: [3](#0-2) 

There is no zero-knowledge proof, commitment, or any other mechanism binding a participant's `(norm_big_y, norm_big_c)` to the session's `app_pk`. A malicious participant can freely substitute any `app_pk'` when computing their contribution.

---

### Impact Explanation

Let participant `j` (malicious) send `(norm_big_y_j, norm_big_c_j)` computed with `app_pk' ≠ app_pk`. The coordinator aggregates:

```
big_C = Σ_{i≠j} λ_i·(x_i·H(pk‖app_id) + app_pk·y_i)
      + λ_j·(x_j·H(pk‖app_id) + app_pk'·y_j)
```

When the client calls `unmask(app_sk)`:

```
big_C − app_sk · big_Y
  = msk · H(pk‖app_id) + λ_j · (app_pk' − app_pk) · y_j
```

Since `app_pk' ≠ app_pk` and `y_j` is a random non-zero scalar, the residual term is non-zero with overwhelming probability. The output is permanently corrupted — it cannot be unmasked to recover the confidential key. [4](#0-3) 

This matches the **High** impact: *Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.*

---

### Likelihood Explanation

- The attacker only needs to be a legitimate participant in the CKD session.
- No special privileges, no leaked keys, no cryptographic breaks required.
- The malicious participant does not even need a prior session to replay — they can compute `(norm_big_y, norm_big_c)` with any arbitrary `app_pk'` on the fly, since the computation is entirely local.
- A single corrupted participant is sufficient to break the output for all honest parties.

---

### Recommendation

Require each participant to provide a zero-knowledge proof of correct formation of their `(norm_big_y, norm_big_c)` contribution, proving knowledge of `y_i` such that:

- `norm_big_y = λ_i · y_i · G`
- `norm_big_c = λ_i · (x_i · H(pk ‖ app_id) + app_pk · y_i)`

This can be done with a standard Schnorr-style DLEQ proof binding `y_i` across both `G` and `app_pk`. Alternatively, a session transcript hash (committing to `app_pk`, `app_id`, participant set) can be used as a challenge in the proof to prevent cross-session replay.

---

### Proof of Concept

```rust
// Session 1: honest run with app_pk_1
// Malicious participant saves their own (norm_big_y_1, norm_big_c_1)
// computed with app_pk_1.

// Session 2: new run with app_pk_2 ≠ app_pk_1, same app_id.
// Malicious participant sends (norm_big_y_1, norm_big_c_1) instead of
// recomputing with app_pk_2.

// Coordinator aggregates without any validation:
//   norm_big_y += norm_big_y_1   // from session 1
//   norm_big_c += norm_big_c_1   // computed under app_pk_1, not app_pk_2

// Client calls unmask(app_sk_2):
//   result = big_C - app_sk_2 * big_Y
//          = msk * H(pk || app_id) + λ_j * (app_pk_1 - app_pk_2) * y_j
//          ≠ msk * H(pk || app_id)
// assert_ne!(result, expected_confidential_key);  // always fails
```

The coordinator's aggregation loop at lines 50–55 of `protocol.rs` accepts the replayed contribution without any check, and `CKDOutput::unmask` at line 54–56 of `mod.rs` produces an incorrect result. [5](#0-4) [4](#0-3)
