### Title
Immutable `FACTORY` address as protocol-fee recipient causes permanent fee lock when blacklisted by USDC/USDT — (`metric-core/contracts/MetricOmmPool.sol`)

---

### Summary

`MetricOmmPool` hard-codes the factory address as an `immutable` and transfers all protocol-share fees directly to it inside `collectFees`. If the factory contract is blacklisted by USDC or USDT, every call to `collectFees` reverts, permanently locking both protocol fees and admin fees inside the pool with no rescue path.

---

### Finding Description

`FACTORY` is declared as an immutable in `MetricOmmPool`: [1](#0-0) 

Inside `collectFees`, after computing the split, the protocol share is pushed to `FACTORY` via `SafeERC20.safeTransfer` (which reverts on failure): [2](#0-1) 

The notional-fee accumulators are zeroed **after** those transfers: [3](#0-2) 

If either `transferToken0(FACTORY, …)` or `transferToken1(FACTORY, …)` reverts (because USDC/USDT has blacklisted the factory address), the entire `collectFees` transaction reverts. The accumulators are never cleared, fees continue to accumulate, and every future call to `collectFees` will also revert.

The admin-fee leg is collected in the **same atomic call**, so admin fees are equally stuck even though `adminFeeDestination` is mutable: [4](#0-3) 

`collectFees` is called as a prerequisite inside `setPoolAdminFees` and `setPoolProtocolFee` before any fee-rate update is applied: [5](#0-4) [6](#0-5) 

So a blacklisted `FACTORY` also permanently blocks fee-rate updates for the affected pool.

The factory's only token-rescue function, `collectTokens`, operates on the factory's own balance — it cannot pull tokens held by the pool: [7](#0-6) 

There is no emergency withdrawal or alternative fee-routing path in the pool.

---

### Impact Explanation

- All accrued protocol fees (spread + notional) held in the pool are permanently unrecoverable.
- All accrued admin fees are equally stuck because they are collected in the same call.
- `setPoolAdminFees` and `setPoolProtocolFee` are permanently blocked for the affected pool, breaking fee governance.
- LP principal and swap functionality are unaffected, so the severity is **Medium** rather than Critical.

---

### Likelihood Explanation

USDC and USDT operators can blacklist any address, including smart contracts. The factory is a fixed, publicly known contract address. While blacklisting a major protocol's factory is an uncommon event, it is not impossible (regulatory action, sanctions compliance, exploit response). The scope explicitly includes USDC/USDT blacklisting behavior. Once triggered, the condition is permanent because `FACTORY` is immutable and there is no override path.

---

### Recommendation

1. Separate the protocol-fee transfer from the admin-fee transfer so that a failure on one leg does not block the other.
2. Replace the direct push to `FACTORY` with an internal accumulator (pull model): credit the protocol's share to a storage variable and let the factory owner withdraw it via `collectTokens` at any time. This removes the immutable-address transfer from the critical path entirely.
3. If a push model is retained, add a configurable `protocolFeeDestination` that the factory owner can update, analogous to how `poolAdminFeeDestination` is already mutable via `setPoolAdminFeeDestination`. [8](#0-7) 

---

### Proof of Concept

1. Deploy a pool with USDC as `token0`.
2. Perform swaps; `notionalFeeToken0Scaled` accumulates.
3. USDC operator blacklists the `MetricOmmPoolFactory` address.
4. Anyone calls `collectPoolFees(pool)` → factory calls `pool.collectFees(…)` → pool calls `transferToken0(FACTORY, amount)` → `USDC.safeTransfer(FACTORY, amount)` reverts with `ERC20: transfer to blacklisted address`.
5. The entire `collectFees` call reverts; `notionalFeeToken0Scaled` is never zeroed.
6. Pool admin calls `setPoolAdminFees(pool, …)` → same revert at the internal `collectFees` call → fee-rate update is permanently blocked.
7. Factory owner calls `collectTokens(USDC, owner, 0)` → succeeds but transfers only the factory's own USDC balance (zero), not the fees held in the pool.
8. No further path exists to recover the fees or unblock fee management.
