### Title
Unbounded Loop in `calculate_rewards` Can Permanently Freeze Delegator's Unclaimed Yield - (File: `src/pool/pool.cairo`)

---

### Summary

The `calculate_rewards` function in `src/pool/pool.cairo` contains an explicitly acknowledged unbounded `while` loop that iterates over every balance-change entry in a pool member's `pool_member_epoch_balance` trace since their last claim. Because each call to `add_to_delegation_pool` or `exit_delegation_pool_intent` in a distinct epoch appends a new entry to this trace, a delegator who makes many balance changes across many epochs without claiming rewards will accumulate an arbitrarily large trace. When `claim_rewards` is eventually called, the loop must process all accumulated entries in a single transaction with no way to break the work into smaller chunks, causing an out-of-gas error that permanently freezes the delegator's unclaimed yield.

---

### Finding Description

**Root cause — unbounded loop in `calculate_rewards`:** [1](#0-0) 

The code itself acknowledges the risk:

```cairo
// **Note**: The loop iterates over the balance changes in the pool member's balance
// trace. This loop is unbounded but unlikely to exceed gas limits.
while entry_to_claim_from < pool_member_trace_length {
```

The loop iterates from `entry_to_claim_from` (the index saved from the last successful claim) up to `pool_member_trace_length` (the total number of balance-change checkpoints). Every iteration performs a storage read (`pool_member_trace.at(entry_to_claim_from)`) and a call to `find_sigma`, both of which are non-trivial operations.

**How the trace grows — `set_member_balance`:** [2](#0-1) 

`set_member_balance` calls `trace.insert(key: self.get_epoch_plus_k(), ...)`. The underlying `insert` implementation only appends a new checkpoint when the epoch key differs from the last entry: [3](#0-2) 

Therefore, one call to `add_to_delegation_pool` per epoch appends exactly one new entry. `exit_delegation_pool_intent` also calls `set_member_balance` directly: [4](#0-3) 

**No partial-claim mechanism in `claim_rewards`:** [5](#0-4) 

`until_checkpoint` is always `get_current_checkpoint()` (the current epoch). There is no parameter to limit the range of entries processed. The updated `entry_to_claim_from` is only written back on a successful transaction; a reverted transaction leaves the state unchanged, so the delegator cannot make incremental progress.

---

### Impact Explanation

**Impact: High — Permanent freezing of unclaimed yield.**

If the `claim_rewards` transaction runs out of gas, it reverts entirely. The delegator's `entry_to_claim_from` and `reward_checkpoint` are not updated. Every subsequent `claim_rewards` attempt faces the same (or larger) trace and will also revert. The delegator's accumulated STRK rewards become permanently inaccessible with no recovery path available in the current contract.

---

### Likelihood Explanation

**Likelihood: Medium.**

A delegator who calls `add_to_delegation_pool` once per epoch without claiming rewards is a realistic long-term participant. Starknet epochs are short (on the order of hours to days). A delegator who is active for hundreds of epochs without claiming — a plausible scenario for a "set and forget" participant — will accumulate hundreds of trace entries. Each loop iteration performs multiple storage reads and arithmetic, making the gas cost per iteration non-trivial. The protocol itself acknowledges the loop is unbounded. No special privileges or external conditions are required; the delegator triggers the condition entirely through normal, permissionless protocol interactions.

---

### Recommendation

1. **Add a `max_entries` parameter to `claim_rewards`** (or an internal helper) so the loop processes at most `max_entries` checkpoints per call, writing back the updated `entry_to_claim_from` even when the full range is not exhausted. This mirrors the fix recommended for the RedeemManager bug.
2. **Alternatively**, enforce a maximum number of unclaimed balance-change entries per pool member (e.g., by requiring a claim before a new `add_to_delegation_pool` call is accepted once the trace exceeds a threshold).
3. **At minimum**, document the concrete gas limit and derive a safe upper bound on the number of balance changes a member may accumulate before claiming, and enforce it on-chain.

---

### Proof of Concept

1. Alice calls `enter_delegation_pool` with a small amount. This creates the first entry in her `pool_member_epoch_balance` trace.
2. For each of the next N epochs (e.g., N = 500), Alice calls `add_to_delegation_pool` with a minimal amount. Because each call targets a different `epoch + K`, each call appends a new checkpoint to her trace. Alice never calls `claim_rewards`.
3. After N epochs, Alice calls `claim_rewards`. The `calculate_rewards` function enters the `while` loop at line 859 with `entry_to_claim_from = 0` and `pool_member_trace_length = N`. The loop executes N iterations, each reading from storage and calling `find_sigma`. For sufficiently large N, the transaction exhausts the Starknet gas limit and reverts.
4. Every subsequent `claim_rewards` call by Alice faces the same N-entry trace (since `entry_to_claim_from` was never updated) and also reverts. Alice's accumulated yield is permanently frozen.
