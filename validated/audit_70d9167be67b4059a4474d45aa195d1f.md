### Title
EIP-7623 Floor Gas Not Enforced in Refund Calculation — Sender Underpays Gas Fee - (File: engine/src/engine.rs)

### Summary

Aurora Engine implements EIP-7623 floor gas validation in `submit_with_alt_modexp` but fails to enforce the floor gas minimum when computing the gas refund in `refund_unused_gas`. The `gas_used` value from EVM execution is used directly instead of `max(gas_used, floor_gas)`, allowing any sender to craft a data-heavy transaction with minimal EVM execution and recover the difference between the floor gas and actual EVM gas used — paying far less than EIP-7623 requires.

### Finding Description

`submit_with_alt_modexp` in `engine/src/engine.rs` correctly computes both `intrinsic_gas` and `floor_gas` and rejects transactions whose `gas_limit` is below `max(intrinsic_gas, floor_gas)`: [1](#0-0) 

The sender is then charged `gas_limit * effective_gas_price` upfront via `charge_gas`.

After EVM execution, the raw `gas_used` (from the EVM result) is passed directly to `refund_unused_gas`: [2](#0-1) 

Inside `refund_unused_gas`, the refund is computed as `prepaid_amount - gas_used * effective_gas_price`: [3](#0-2) 

`floor_gas` is never passed to `refund_unused_gas` and is never considered in the refund arithmetic. The function signature has no parameter for it: [4](#0-3) 

`floor_gas` is computed in `NormalizedEthTransaction::floor_gas` as:

```
floor_gas = gas_transaction_call + tokens_in_calldata * total_cost_floor_per_token
```

where `tokens_in_calldata = num_non_zero_bytes * 4 + num_zero_bytes`. [5](#0-4) 

Because `total_cost_floor_per_token = 10` and `gas_transaction_non_zero_data = 16`, for any transaction with non-zero calldata bytes:

```
floor_gas = 21000 + (4 * nonzero + zero) * 10
intrinsic_gas = 21000 + 16 * nonzero + 4 * zero
```

`floor_gas > intrinsic_gas` whenever there is any calldata, meaning `floor_gas` is always the binding minimum for data-heavy transactions. The engine uses `Config::prague()` which has `has_floor_gas = true`: [6](#0-5) 

### Impact Explanation

**Concrete example:**

- Sender crafts a transaction with 1 000 non-zero calldata bytes and a simple no-op target (e.g., a plain ETH transfer to a contract that does nothing).
- `floor_gas = 21 000 + 4 000 * 10 = 61 000`
- `intrinsic_gas = 21 000 + 16 000 = 37 000`
- `gas_limit = 61 000` (minimum accepted)
- EVM execution uses `gas_used = 37 000` (intrinsic only, no additional EVM work)
- `prepaid_amount = 61 000 * price`
- Refund returned to sender = `(61 000 − 37 000) * price = 24 000 * price`

The sender effectively pays only `37 000 * price` instead of the required `61 000 * price`. The `24 000 * price` that should remain consumed (burned or kept by the engine) is instead returned to the sender. The relayer also receives only `37 000 * priority_fee` instead of `61 000 * priority_fee`.

This is a direct, repeatable theft of fee revenue from the protocol and relayer on every data-heavy transaction where EVM execution gas is below the floor.

### Likelihood Explanation

Any unprivileged EVM user can trigger this by:
1. Constructing a transaction with a large calldata payload (maximising `floor_gas`).
2. Targeting a contract or address that performs minimal EVM work (keeping `gas_used` near `intrinsic_gas`).
3. Setting `gas_limit = floor_gas` (the minimum accepted value).

No special privileges, governance access, or external oracle are required. The entry path is the standard `submit` / `submit_with_alt_modexp` function, reachable by any relayer submitting a signed Ethereum transaction.

### Recommendation

In `submit_with_alt_modexp`, compute the effective gas used as `max(gas_used, floor_gas)` before calling `refund_unused_gas`:

```rust
let effective_gas_used = core::cmp::max(gas_used, floor_gas);
refund_unused_gas(
    &mut io,
    &sender,
    effective_gas_used,   // <-- was: gas_used
    &prepaid_amount,
    &relayer_address,
    fixed_gas,
)
```

This mirrors the EIP-7623 requirement that the gas consumed is `max(gas_used_by_evm, floor_gas)` when the floor gas exceeds the intrinsic gas.

### Proof of Concept

1. Deploy or use any existing contract on Aurora that accepts calldata but performs minimal computation (e.g., a contract whose fallback does nothing, or a plain ETH transfer to an EOA with calldata attached).
2. Craft a transaction with `data = [0x01u8; 1000]` (1 000 non-zero bytes), `gas_limit = 61 000`, `gas_price = 1`.
3. Submit via `submit`. Observe `gas_used ≈ 37 000` in the result.
4. Check sender balance: it decreased by `37 000` wei, not `61 000` wei.
5. The `24 000` wei difference was incorrectly refunded, violating EIP-7623's floor gas requirement.

The root cause is the absence of `floor_gas` in the `refund_unused_gas` call at: [7](#0-6)

    pub fn floor_gas(&self, config: &aurora_evm::Config) -> Result<u64, Error> {
        if config.has_floor_gas {
            let num_zero_bytes = u64::try_from(self.data.iter().filter(|b| **b == 0).count())
                .map_err(|_e| Error::IntegerConversion)?;
            let data_len = u64::try_from(self.data.len()).map_err(|_e| Error::IntegerConversion)?;
            let num_non_zero_bytes = data_len
                .checked_sub(num_zero_bytes)
                .ok_or(Error::GasOverflow)?;

            let base_gas = config.gas_transaction_call;
            let tokens_in_calldata = num_non_zero_bytes
                .checked_mul(4)
                .and_then(|gas| gas.checked_add(num_zero_bytes))
                .ok_or(Error::GasOverflow)?;

            tokens_in_calldata
                .checked_mul(config.total_cost_floor_per_token)
                .and_then(|gas| gas.checked_add(base_gas))
                .ok_or(Error::GasOverflow)
        } else {
            Ok(0)
        }
    }
