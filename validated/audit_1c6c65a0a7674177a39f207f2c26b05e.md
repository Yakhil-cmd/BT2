### Title
Missing Minimum Threshold Enforcement in FROST Signing Allows Degenerate Threshold Parameter - (File: src/frost/mod.rs)

### Summary
`assert_sign_inputs` and `presign` in `src/frost/mod.rs` do not enforce a minimum threshold of 2, unlike `assert_key_invariants` in `src/dkg.rs`. A caller can pass `threshold = 1` (or `threshold = 0`) to FROST signing/presigning even though the key was generated under a threshold ≥ 2. This causes Lagrange interpolation to use wrong coefficients, corrupting the signature output and producing an unusable cryptographic result that honest parties cannot verify.

---

### Finding Description

`assert_key_invariants` in `src/dkg.rs` explicitly enforces a minimum threshold of 2:

```rust
// src/dkg.rs lines 580-582
if threshold < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
}
``` [1](#0-0) 

However, `assert_sign_inputs` in `src/frost/mod.rs` only checks that the threshold is not *too large*; it never checks that it is not *too small*:

```rust
// src/frost/mod.rs lines 144-150
// validate threshold
if threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge {
        threshold: threshold.value(),
        max: participants.len(),
    });
}
``` [2](#0-1) 

The same gap exists in `presign`:

```rust
// src/frost/mod.rs lines 72-77
if args.threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge {
        threshold: args.threshold.into(),
        max: participants.len(),
    });
}
``` [3](#0-2) 

`ReconstructionLowerBound` is a plain `usize` wrapper with no minimum enforced at the type level, so `threshold = 1` or `threshold = 0` are both representable and accepted: [4](#0-3) 

---

### Impact Explanation

In FROST, each participant computes a signature share weighted by their Lagrange coefficient, which is derived from the signing threshold and the participant set. If the key was generated with `threshold = 2` (degree-1 polynomial) but signing is invoked with `threshold = 1`, the Lagrange coefficients are computed for a degree-0 interpolation. The resulting aggregated signature uses the wrong effective secret key and fails verification — an unusable cryptographic output. With `threshold = 0`, Lagrange interpolation may encounter a zero-denominator, causing a panic or protocol abort.

This maps to: **High — Corruption of sign outputs so honest parties accept inconsistent or unusable cryptographic outputs.** [5](#0-4) 

---

### Likelihood Explanation

The entry path is direct: any caller of `assert_sign_inputs` or `presign` (including a malicious coordinator or a misconfigured application) can supply `threshold = 1`. There is no type-level guard, no runtime guard in the FROST signing path, and no documentation warning against it. The DKG path is protected, but the signing path is not, creating an inconsistency that is easy to trigger accidentally or maliciously. [6](#0-5) 

---

### Recommendation

Add the same `threshold < 2` guard to `assert_sign_inputs` and `presign` in `src/frost/mod.rs` that already exists in `assert_key_invariants`:

```rust
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

This should be inserted immediately after the `ThresholdTooLarge` check in both functions. [7](#0-6) 

---

### Proof of Concept

1. Run DKG with 3 participants and `threshold = 2` (accepted, produces valid key shares).
2. Call `assert_sign_inputs` with the same 3 participants but `threshold = 1` — no error is returned.
3. Proceed to FROST signing with `threshold = 1`; Lagrange coefficients are computed for a single-point interpolation (coefficient = 1 for the sole participant), ignoring the actual degree-1 polynomial structure of the shares.
4. The aggregated signature `z = Σ z_i` uses the wrong weighting; signature verification against the master public key fails.
5. Honest parties receive an unusable signature with no indication of which parameter was wrong. [8](#0-7) [1](#0-0)

/// Verifies that the sign inputs are valid
pub fn assert_sign_inputs(
    participants: &[Participant],
    threshold: impl Into<ReconstructionLowerBound>,
    me: Participant,
    coordinator: Participant,
) -> Result<ParticipantList, InitializationError> {
    let threshold = threshold.into();
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // validate threshold
    if threshold.value() > participants.len() {
        return Err(InitializationError::ThresholdTooLarge {
            threshold: threshold.value(),
            max: participants.len(),
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }
    Ok(participants)
}
