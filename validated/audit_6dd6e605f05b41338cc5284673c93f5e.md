### Title
Unchecked Precompile Call Return Value in `withdrawToNear`/`withdrawToEthereum` Causes Permanent ERC-20 Token Burn Without NEP-141 Release - (File: `etc/eth-contracts/contracts/EvmErc20.sol`)

---

### Summary

`EvmErc20.sol` and `EvmErc20V2.sol` burn the caller's ERC-20 tokens before calling the exit precompile, but never check the return value of that precompile call. If the precompile call fails for any reason (e.g., invalid NEAR account ID in `recipient`), the ERC-20 tokens are permanently destroyed while the corresponding NEP-141 tokens remain locked inside the Aurora contract, causing a permanent fund freeze.

---

### Finding Description

Both `withdrawToNear` and `withdrawToEthereum` in `EvmErc20.sol` follow this pattern:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);          // ERC-20 tokens destroyed here

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    assembly {
        let res := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
        // res is NEVER checked
    }
}
``` [1](#0-0) 

The same pattern appears in `withdrawToEthereum`: [2](#0-1) 

And identically in `EvmErc20V2.sol`: [3](#0-2) 

The `_burn` call modifies the ERC-20 contract's own storage (reduces the caller's balance and total supply). In EVM semantics, when a subsequent `call` to the precompile fails, only the callee's state changes are reverted — the caller's `_burn` state change is **not** reverted. The `res` variable is assigned but never read or checked.

The `ExitToNear` precompile can return `Err(ExitError::Other(...))` in several reachable cases:

- **Invalid NEAR account ID**: `recipient` is a caller-controlled `bytes` parameter. If it does not parse as a valid NEAR account ID, `parse_recipient` inside the precompile returns an error.
- **ERC-20 not registered**: `get_nep141_from_erc20` fails if the ERC-20 address has no NEP-141 mapping.
- **Out of gas**: Insufficient gas forwarded to the precompile. [4](#0-3) 

When the precompile fails, no NEAR promise is scheduled, so no `ft_transfer` is ever sent to the NEP-141 contract. The NEP-141 tokens remain permanently locked inside the Aurora engine contract with no recovery path.

The `error_refund` feature only handles the case where the precompile call **succeeds** (promise is scheduled) but the downstream NEP-141 `ft_transfer` promise fails asynchronously. It does not handle the case where the precompile call itself fails synchronously within the EVM. [5](#0-4) 

---

### Impact Explanation

**Impact: Critical — Permanent freezing of funds.**

When the precompile call fails silently:
- The user's ERC-20 balance is permanently reduced (tokens burned).
- The NEP-141 tokens that were deposited into the Aurora contract when the ERC-20 was minted remain locked in the Aurora contract forever.
- There is no admin recovery path: the NEP-141 tokens are held by the Aurora engine contract, and no function exists to re-mint the burned ERC-20 tokens or release the NEP-141 tokens back to the user without the `error_refund` callback path (which is never triggered because no promise was scheduled).

This creates a permanent accounting inconsistency: ERC-20 total supply decreases while the NEP-141 balance held by Aurora does not, exactly analogous to the SKALE report where `locked > balance` after a burn.

---

### Likelihood Explanation

**Likelihood: Medium.**

Any ERC-20 token holder can call `withdrawToNear` or `withdrawToEthereum` directly. The `recipient` parameter is a raw `bytes` value with no on-chain validation before the burn. A user who accidentally passes:
- An empty byte string
- A string exceeding the NEAR account ID length limit
- A string with invalid characters (e.g., uppercase letters, special characters not allowed in NEAR account IDs)

...will have their tokens permanently burned. This is a realistic user error, not a contrived attack. The `withdrawToEthereum` path is similarly affected if the precompile fails for any reason.

---

### Recommendation

Check the return value of the precompile `call` and revert the entire transaction if it fails, so that the `_burn` is also reverted:

```solidity
function withdrawToNear(bytes memory recipient, uint256 amount) external override {
    _burn(_msgSender(), amount);

    bytes32 amount_b = bytes32(amount);
    bytes memory input = abi.encodePacked("\x01", amount_b, recipient);
    uint input_size = 1 + 32 + recipient.length;

    bool success;
    assembly {
        success := call(gas(), 0xe9217bc70b7ed1f598ddd3199e80b093fa71124f, 0, add(input, 32), input_size, 0, 32)
    }
    require(success, "ExitToNear precompile call failed");
}
```

Alternatively, move the `_burn` call to **after** the precompile call succeeds, so that a failed precompile call naturally prevents the burn. Apply the same fix to `withdrawToEthereum` in both `EvmErc20.sol` and `EvmErc20V2.sol`.

---

### Proof of Concept

1. User holds 100 units of a bridged NEP-141 token as ERC-20 on Aurora.
2. User calls `withdrawToNear(bytes("INVALID ACCOUNT ID WITH SPACES"), 100)`.
3. `_burn(msg.sender, 100)` executes: user's ERC-20 balance drops from 100 to 0, ERC-20 total supply decreases by 100.
4. The assembly `call` to the `ExitToNear` precompile at `0xe921...` is made. Inside the precompile, `parse_recipient` fails because `"INVALID ACCOUNT ID WITH SPACES"` is not a valid NEAR account ID. The precompile returns `Err(ExitError::Other(...))`. The EVM `call` returns `res = 0`.
5. `res` is never read. The function returns normally. No NEAR promise is scheduled.
6. The 100 NEP-141 tokens remain locked in the Aurora engine contract. The user's ERC-20 balance is 0. The user has permanently lost 100 tokens.
7. The `error_refund` callback (`exit_to_near_precompile_callback`) is never triggered because no promise was created. [1](#0-0) [6](#0-5)
