### Title
Unchecked NEAR Transfer Result in wNEAR Unwrap Callback Causes Permanent Fund Freeze - (File: `engine/src/contract_methods/connector.rs`)

---

### Summary

In `exit_to_near_precompile_callback`, after a successful `near_withdraw` (wNEAR → NEAR unwrap), a NEAR native token transfer is dispatched via `PromiseBatchAction::Transfer` without attaching any callback to verify the transfer succeeded. If the transfer fails (e.g., the target NEAR account does not exist), the NEAR tokens remain locked in Aurora's contract balance while the user's wNEAR ERC-20 tokens have already been burned — resulting in a permanent fund freeze.

---

### Finding Description

The `ExitToNear` precompile supports a wNEAR unwrap flow. When a user calls `withdrawToNear` on the wNEAR ERC-20 contract with the `:unwrap` suffix, the following sequence executes:

**Step 1 — Precompile detects wNEAR unwrap and sets `transfer_near_args`:**

In `exit_erc20_token_to_near`, when `erc20_address == get_wnear_address(io).raw()` and `exit_params.message == Some(Message::UnwrapWnear)`, a `TransferNearArgs` is constructed and returned as `transfer_near_args`: [1](#0-0) 

**Step 2 — A callback is attached because `transfer_near` is `Some`:**

Back in `ExitToNear::run`, `callback_args.transfer_near` is `Some`, so `callback_args != ExitToNearPrecompileCallbackArgs::default()`, and a `PromiseArgs::Callback` is constructed that calls `exit_to_near_precompile_callback` after `near_withdraw`: [2](#0-1) 

**Step 3 — The callback dispatches a NEAR transfer with no result check:**

In `exit_to_near_precompile_callback`, if `near_withdraw` succeeded and `args.transfer_near` is `Some`, a `PromiseBatchAction::Transfer` is created and returned — but **no callback is attached** to this transfer promise to verify it succeeded: [3](#0-2) 

The `handler.promise_create_batch(&promise)` call at line 226 dispatches the NEAR transfer, and `handler.promise_return(promise_id)` at line 227 makes it the return value of the function. There is no `promise_attach_callback` or `promise_attach_batch_callback` call to handle failure. If the NEAR transfer fails, execution silently succeeds from Aurora's perspective.

**The `TransferNearArgs` struct confirms the target account is user-controlled:** [4](#0-3) 

---

### Impact Explanation

When the NEAR transfer fails (e.g., the `target_account_id` does not exist on NEAR), the NEAR tokens remain in Aurora's contract balance. The user's wNEAR ERC-20 tokens were burned at the start of the flow (before `near_withdraw` was called). There is no recovery path: no callback re-mints the ERC-20 tokens, and no admin function exists to return the stranded NEAR to the user. This constitutes **permanent freezing of funds**.

---

### Likelihood Explanation

The trigger condition is realistic and reachable by any unprivileged EVM user:
- A user holds wNEAR ERC-20 tokens on Aurora.
- The user calls `withdrawToNear` on the wNEAR ERC-20 contract with a NEAR account ID that does not exist (e.g., a typo, a deleted account, or a not-yet-created account).
- No special privileges, admin access, or external oracle compromise is required.
- The failure is silent — the EVM transaction succeeds, giving the user no indication that their funds are frozen.

---

### Recommendation

Attach a callback to the NEAR transfer promise that checks the result. If the transfer failed, re-mint the wNEAR ERC-20 tokens to the original sender, mirroring the existing `error_refund` mechanism used for the `ft_transfer` failure path:

```rust
// After creating the transfer promise, attach a callback:
let callback_promise = PromiseCreateArgs {
    target_account_id: self_account_id.clone(),
    method: "exit_to_near_transfer_near_callback".to_string(),
    args: borsh::to_vec(&refund_args).unwrap(),
    attached_balance: Yocto::new(0),
    attached_gas: costs::EXIT_TO_NEAR_CALLBACK_GAS,
};
let callback_id = handler.promise_attach_batch_callback(promise_id, &PromiseBatchAction {
    target_account_id: self_account_id,
    actions: vec![PromiseAction::FunctionCall { ... }],
});
handler.promise_return(callback_id);
```

The callback should call `engine::refund_on_error` to re-mint the wNEAR ERC-20 tokens if the NEAR transfer failed.

---

### Proof of Concept

1. User holds 100 wNEAR ERC-20 tokens on Aurora.
2. User calls `withdrawToNear("nonexistent.near:unwrap", 100)` on the wNEAR ERC-20 contract.
3. `ExitToNear` precompile burns 100 wNEAR ERC-20 tokens from the user's EVM balance.
4. `near_withdraw({"amount": "100"})` is called on the wNEAR NEAR contract — this succeeds, converting 100 wNEAR to 100 NEAR yocto held by Aurora.
5. `exit_to_near_precompile_callback` is invoked; `near_withdraw` result is `Successful`.
6. `PromiseBatchAction::Transfer { amount: 100 }` is dispatched to `nonexistent.near`.
7. The NEAR transfer fails because `nonexistent.near` does not exist.
8. No callback handles the failure; Aurora's NEAR balance increases by 100 NEAR.
9. The user's wNEAR ERC-20 balance is 0 and there is no recovery path.
10. **100 NEAR worth of user funds are permanently frozen.** [5](#0-4) [6](#0-5)
