### Title
Unauthenticated Invoke Transaction Admitted via `skip_stateful_validations` Signature-Bypass — (`File: crates/apollo_gateway/src/stateful_transaction_validator.rs`)

### Summary

The gateway's `skip_stateful_validations` UX feature, intended to let a user submit a deploy-account and a first invoke together, contains no check that the submitter of the invoke is the same party who submitted the deploy-account. Any unprivileged attacker who observes a victim's pending `deploy_account` transaction in the mempool can submit an invoke transaction bearing the victim's `sender_address`, nonce `1`, and a completely fabricated signature. The gateway skips the `__validate__` entry-point call for that transaction and admits it to the mempool without any signature verification.

### Finding Description

**Invariant broken:** Every invoke transaction admitted to the mempool must have passed the account's `__validate__` entry point (or an equivalent cryptographic check), so that only the account owner can authorize spending from that account.

**Root cause — `skip_stateful_validations`** [1](#0-0) 

The function returns `true` (skip validation) when all three conditions hold:

1. The transaction is an `Invoke`.
2. `tx.nonce() == 1`.
3. `account_nonce == 0` (account not yet deployed on-chain).
4. `mempool_client.account_tx_in_pool_or_recent_block(tx.sender_address())` returns `true`. [2](#0-1) 

Condition 4 is satisfied whenever the victim's address has **any** transaction in the mempool pool or staged queue — including the victim's own `deploy_account` that the attacker observed via P2P gossip. [3](#0-2) 

**Effect of `skip_validate = true`**

When `skip_stateful_validations` returns `true`, `run_validate_entry_point` is called with `validate: false`: [4](#0-3) 

Inside `StatefulValidator::perform_validations`, the `validate` flag causes an early return before `__validate__` is ever called: [5](#0-4) 

The only checks that still run are `perform_pre_validation_stage` (nonce increment, fee-bound arithmetic, balance check) — none of which verify the transaction signature. [6](#0-5) 

**No earlier guard catches this.** `validate_by_mempool` only checks nonce ordering and duplicate detection; it does not verify signatures. [7](#0-6) 

### Impact Explanation

An attacker can inject an invoke transaction bearing any victim's `sender_address` into the mempool without a valid signature. When the batcher later executes the block, the blockifier runs `__validate__` with `validate: true` (via `new_for_sequencing`), the forged signature fails, and the transaction is reverted — but the victim's account is charged the validation-failure fee. Additionally, the attacker occupies the victim's nonce-1 slot in the mempool, potentially blocking or delaying the victim's own legitimate first invoke.

This matches the allowed impact: **"High. Mempool/gateway/RPC admission accepts invalid transactions … before sequencing."**

### Likelihood Explanation

- The victim's `deploy_account` transaction is observable by any network participant via P2P gossip before it is committed.
- The attacker needs only to craft an `RpcInvokeTransactionV3` with `sender_address = victim`, `nonce = 1`, arbitrary `calldata`, and a dummy `signature`. All other fields (resource bounds, DA modes) are trivially valid.
- No privileged access, no special contract deployment, and no prior relationship with the victim is required.
- The window is the entire time the victim's `deploy_account` sits in the mempool (potentially many seconds to minutes).

### Recommendation

The `skip_stateful_validations` bypass must be restricted to the **same submitter** as the pending deploy-account, or the bypass must be removed entirely and replaced with a mechanism that does not skip signature verification. Concretely:

1. **Bind the bypass to the deploy-account transaction hash.** The gateway should only skip `__validate__` for an invoke if the caller also provides the hash of the pending deploy-account transaction and that hash is verifiably linked to the same key-pair (e.g., the deploy-account is already in the mempool under the same sender and the invoke's signature covers the deploy-account hash).
2. **Alternatively**, remove the gateway-level `skip_stateful_validations` entirely and rely on the blockifier's `strict_nonce_check = false` path, which already tolerates nonce-1 invokes after a nonce-0 deploy-account without skipping `__validate__`.

async fn skip_stateful_validations(
    tx: &ExecutableTransaction,
    account_nonce: Nonce,
    mempool_client: SharedMempoolClient,
) -> StatefulTransactionValidatorResult<bool> {
    if let ExecutableTransaction::Invoke(ExecutableInvokeTransaction { tx, .. }) = tx {
        // check if the transaction nonce is 1, meaning it is post deploy_account, and the
        // account nonce is zero, meaning the account was not deployed yet.
        if tx.nonce() == Nonce(Felt::ONE) && account_nonce == Nonce(Felt::ZERO) {
            let account_address = tx.sender_address();
            debug!("Checking if deploy_account transaction exists for account {account_address}.");
            // We verify that a deploy_account transaction exists for this account. It is sufficient
            // to check if the account exists in the mempool since it means that either it has a
            // deploy_account transaction or transactions with future nonces that passed
            // validations.
            return mempool_client
                .account_tx_in_pool_or_recent_block(tx.sender_address())
                .await
                .map_err(|err| mempool_client_err_to_deprecated_gw_err(&tx.signature(), err))
                .inspect(|exists| {
                    if *exists {
                        debug!("Found deploy_account transaction for account {account_address}.");
                    } else {
                        debug!(
                            "No deploy_account transaction found for account {account_address}."
                        );
                    }
                });
        }
    }

    Ok(false)
}
