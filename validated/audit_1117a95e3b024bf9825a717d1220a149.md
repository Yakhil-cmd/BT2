Looking at the code at lines 400-411, there is no length validation on the received `(a_j_i_v, b_j_i_v)` vecs before indexing them with `[i]` in the `for i in 0..N` loop.

**Validation on `wait2` (lines 331-396):** The `PolynomialCommitmentsMessageMany` message is thoroughly validated — polynomial degrees, commitment opening, and dlog proofs are all checked. [1](#0-0) 

**No validation on `wait3` (lines 400-411):** The private share message is received as a raw `(Vec<SerializableScalar<C>>, Vec<SerializableScalar<C>>)` tuple. There is no check that `a_j_i_v.len() == N` or `b_j_i_v.len() == N` before the loop indexes into them. [2](#0-1) 

In Rust, `vec[i]` where `i >= vec.len()` is an unconditional panic — it is not a `Result` and cannot be caught by `?`. This means the honest receiver's async task (and likely the whole process, depending on the panic handler) terminates abruptly rather than returning a `ProtocolError`.

The attacker's path is:
1. Participate normally through `wait0`, `wait1`, and `wait2` (sending a valid `PolynomialCommitmentsMessageMany` that passes all checks).
2. On `wait3`, send a private message `(a_j_i_v, b_j_i_v)` where either vec has length `< N` (e.g., `N-1`).
3. The honest receiver panics at line 407 or 408.

This is a concrete, locally testable path with no cryptographic assumptions required.

### Title
Index-out-of-bounds panic via undersized private share Vec in `do_generation_many` — (`src/ecdsa/ot_based_ecdsa/triples/generation.rs`)

### Summary
A malicious participant in the triple-generation protocol can send a `wait3` private share message whose `a_j_i_v` or `b_j_i_v` vector has fewer than `N` elements, causing an unconditional Rust index-out-of-bounds panic in the honest receiver at lines 407–408, aborting triple generation.

### Finding Description
`do_generation_many` receives private polynomial evaluations from each other participant on `wait3` as a `(Vec<SerializableScalar<C>>, Vec<SerializableScalar<C>>)`. The code immediately iterates `for i in 0..N` and accesses `a_j_i_v[i]` / `b_j_i_v[i]` with no prior length check. A malicious participant can send a vec of length `N-1` (or 0), triggering a panic on the first out-of-range index. [3](#0-2) 

The `wait2` message is fully validated (commitment, degree, dlog proofs), but the `wait3` message has no structural validation at all. [4](#0-3) 

### Impact Explanation
A Rust index-out-of-bounds panic is unrecoverable at the call site — it cannot be caught with `?` or mapped to `ProtocolError`. The honest receiver's task terminates abnormally. Since the attacker can repeat this on every new triple-generation session, honest parties are permanently denied the ability to generate triples (and therefore signatures).

### Likelihood Explanation
Any single malicious participant in the protocol can trigger this. No cryptographic material needs to be compromised. The attacker only needs to send a well-formed `wait2` message (to pass all checks) and then a malformed `wait3` message. This is trivially achievable.

### Recommendation
Add an explicit length check immediately after receiving the `wait3` message, before the indexing loop:

```rust
if a_j_i_v.len() != N || b_j_i_v.len() != N {
    return Err(ProtocolError::AssertionFailed(
        "private share vector has wrong length".to_string(),
    ));
}
```

This converts the panic into a graceful `ProtocolError`, consistent with how all other malformed messages are handled.

### Proof of Concept
```rust
// Attacker sends on wait3:
let short_vec: Vec<SerializableScalar<C>> = vec![/* N-1 elements */];
chan.send_private(wait3, victim, &(short_vec.clone(), short_vec))?;
// Honest receiver panics at generation.rs:407: index out of bounds: the len is N-1 but the index is N-1
```
