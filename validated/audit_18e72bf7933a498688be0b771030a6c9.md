### Title
`validateCOAOwnershipProof` Signature Replay Across Multiple COAs of the Same Cadence Account - (File: `fvm/evm/stdlib/contract.cdc`)

---

### Summary

`EVM.validateCOAOwnershipProof` verifies that a Cadence account's signatures over `signedData` are valid, then checks that the account holds a COA at the given `path` matching `evmAddress`. Because `signedData` is never required to encode `evmAddress`, a valid proof produced for one COA can be replayed verbatim to falsely prove ownership of any other COA stored in the same Cadence account.

---

### Finding Description

`EVM.validateCOAOwnershipProof` accepts six parameters: `address`, `path`, `signedData`, `keyIndices`, `signatures`, and `evmAddress`. The cryptographic check verifies only that the signatures are valid over `signedData` using the account's keys. The `evmAddress` is checked separately — it is compared against the COA resource found at `path`, but it is **never included in the signed message**. [1](#0-0) 

The code itself acknowledges this gap: [2](#0-1) 

Because `signedData` is decoupled from `evmAddress`, any valid `(signedData, signatures)` pair produced for COA_A can be submitted again with `evmAddress = COA_B` and `path = /public/coa2` — the signature check still passes (same data, same keys), and the COA address check passes because the account legitimately owns COA_B.

The `COAOwnershipProofInContext` struct carries `SignedData` and `EVMAddress` as independent fields with no cryptographic binding between them: [3](#0-2) 

---

### Impact Explanation

Any application or protocol that calls `EVM.validateCOAOwnershipProof` directly (it is `access(all)`) to authenticate a user as the owner of a specific EVM address can be deceived. An attacker who observes a valid on-chain or API-visible proof for COA_A can replay it to impersonate the victim as the owner of COA_B — a different EVM address belonging to the same Cadence account. This breaks authentication and authorization guarantees for any off-chain or on-chain flow that relies on this proof, potentially enabling unauthorized asset access or identity impersonation.

The ERC-1271 path in `coa.sol` is protected because `isValidSignature` always passes `address(this)` as the COA address: [4](#0-3) 

However, the Cadence-level `validateCOAOwnershipProof` function is callable directly from any script or transaction with attacker-controlled arguments, and the EVM precompile `verifyCOAOwnershipProof` also accepts a caller-supplied `address` argument: [5](#0-4) 

---

### Likelihood Explanation

The precondition is that the victim Cadence account holds more than one COA. While wallets historically create one COA per account, the protocol does not enforce this limit, and multi-COA accounts are a valid and documented use case. A valid proof is observable on-chain or via the Access API whenever a user authenticates with any application. Once observed, replay requires no special privileges — any unprivileged transaction sender can submit the replayed proof.

---

### Recommendation

Require that `signedData` cryptographically commits to `evmAddress`. The simplest enforcement is to verify inside `validateCOAOwnershipProof` that `signedData` contains the 20-byte `evmAddress` as a prefix or suffix, or to document and enforce a structured encoding (e.g., `signedData = abi.encode(evmAddress, applicationData)`). Alternatively, the function can hash `evmAddress` into the message before verification, similar to how the reNFT mitigation added `orderHash` to `RentPayload`:

```cadence
// Enforce that signedData encodes evmAddress
let evmAddressBytes: [UInt8] = evmAddress.toConstantSized()!.toVariableSized()
for i, b in evmAddressBytes {
    if signedData[i] != b {
        return ValidationResult(isValid: false, problem: "signedData must begin with evmAddress")
    }
}
``` [6](#0-5) 

---

### Proof of Concept

1. Alice's Cadence account (`0xAlice`) stores two COAs:
   - COA_A at `/public/coa1` with EVM address `0xAAAA...`
   - COA_B at `/public/coa2` with EVM address `0xBBBB...`

2. A DApp asks Alice to prove ownership of COA_A. Alice signs `signedData = "login:dapp.xyz"` with her account key and submits:
   ```
   EVM.validateCOAOwnershipProof(
     address: 0xAlice, path: /public/coa1,
     signedData: "login:dapp.xyz", keyIndices: [0], signatures: [sig],
     evmAddress: 0xAAAA...
   ) // returns isValid: true
   ```

3. An attacker observes `(signedData, sig)` on-chain and calls:
   ```
   EVM.validateCOAOwnershipProof(
     address: 0xAlice, path: /public/coa2,
     signedData: "login:dapp.xyz", keyIndices: [0], signatures: [sig],
     evmAddress: 0xBBBB...
   ) // also returns isValid: true — replay succeeds
   ```

The signature check passes because `signedData` is identical; the COA address check passes because Alice legitimately owns COA_B. The attacker has falsely proven ownership of `0xBBBB...` without Alice's consent. [7](#0-6)
