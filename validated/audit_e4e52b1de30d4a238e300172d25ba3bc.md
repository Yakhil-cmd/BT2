### Title
Unauthenticated Admin HTTP Server Exposes Privileged Consensus-Bypass Command — (`admin/command_runner.go`)

---

### Summary

The Flow node admin HTTP server has **no authentication by default**. When TLS mutual-auth is not configured (the default), any network-reachable party can POST to `/admin/run_command` and invoke every registered privileged command — including `set-config` with `consensus-required-approvals-for-sealing: 0`, which causes the consensus node to seal execution results with **zero verification-node approvals**, completely bypassing the verification layer.

---

### Finding Description

`CommandRunner.runAdminServer()` starts two servers: a Unix-socket gRPC backend and an HTTP frontend that proxies to it.

```go
// admin/command_runner.go lines 249-265
httpServer := &http.Server{
    Addr:      r.httpAddress,
    Handler:   mux,
    TLSConfig: r.tlsConfig,
}
...
if r.tlsConfig == nil {
    err = httpServer.ListenAndServe()   // plain HTTP, no auth
} else {
    err = httpServer.ListenAndServeTLS("", "")
}
```

TLS is only enabled when all three flags `--admin-cert`, `--admin-key`, and `--admin-client-certs` are provided together. When they are absent (the default), the server starts with **no authentication, no token, no IP allowlist**. The `--admin-addr` flag accepts any bind address, including `0.0.0.0:9002`, which is a documented and supported configuration.

`runCommand()` dispatches to any registered handler with no caller-identity check:


The consensus node registers `consensus-required-approvals-for-sealing` as a live-updatable config:


`SetRequiredApprovalsForSealingConstruction` accepts `0` as a valid value (the only constraint is `0 ≤ value ≤ chunkAlpha`):

When `requiredApprovalsForSealConstruction == 0`, `ApprovalCollector` immediately seals every result without waiting for any verification-node signatures:


---

### Impact Explanation

**Consensus integrity failure.** An attacker who can reach the consensus node's admin HTTP port can set `consensus-required-approvals-for-sealing` to `0`. From that moment, every new execution result is sealed immediately by the consensus node without any verification-node approval. Malicious or incorrect execution results (e.g., fabricated state transitions, stolen balances) will be finalized on-chain with no cryptographic check from the verification layer. This is a complete bypass of Flow's multi-role security model.

Secondary impact on Access nodes: arbitrary Cadence script execution against the node's local state (information disclosure, potential resource exhaustion).

---

### Likelihood Explanation

The admin server is disabled by default (`--admin-addr` not set). However:

- It is explicitly documented and routinely enabled in production deployments (the README, localnet docs, and integration tests all demonstrate its use).
- The `--admin-addr` flag accepts `0.0.0.0:9002`, a supported and documented configuration.
- When TLS is omitted (the common case — all three cert flags must be provided together to enable it), the server has zero authentication.
- Any party with TCP reachability to the admin port — including a co-located malicious process, a compromised container in the same pod/network namespace, or a misconfigured firewall — can invoke all commands without credentials.

---

### Recommendation

1. **Require authentication by default.** Add a mandatory shared-secret token (e.g., a bearer token or HMAC-signed request) that must be present on every request, independent of TLS. TLS provides transport confidentiality but not caller authentication when client-cert verification is skipped.
2. **Bind to localhost by default.** Change the default `--admin-addr` to `127.0.0.1:9002` and require an explicit opt-in to bind to a broader interface.
3. **Restrict the `set-config` surface.** Commands that alter consensus-critical parameters (sealing thresholds, emergency-sealing flag) should require a separate, higher-privilege credential or be removed from the live-updatable surface entirely — analogous to the recommendation in the external report to make assimilator addition constructor-only.

```bash
  -d '{"commandName": "set-config",
```

```go
func (r *CommandRunner) runCommand(ctx context.Context, command string, data any) (any, error) {
	r.logger.Info().Str("command", command).Msg("received new command")

	req := &CommandRequest{Data: data}

	if validator := r.getValidator(command); validator != nil {
		if validationErr := validator(req); validationErr != nil {
			// for expected validation errors, return code InvalidArgument and the error text
			if IsInvalidAdminParameterError(validationErr) {
				return nil, status.Error(codes.InvalidArgument, validationErr.Error())
			}
			// for unexpected errors, return code Internal and log a warning
			r.logger.Err(validationErr).Msg("unexpected error validating admin request")
			return nil, status.Error(codes.Internal, validationErr.Error())
		}
	}

	var handleResult any
	var handleErr error

	if handler := r.getHandler(command); handler != nil {
		if handleResult, handleErr = handler(ctx, req); handleErr != nil {
			if errors.Is(handleErr, context.Canceled) {
				return nil, status.Error(codes.Canceled, "client canceled")
			} else if errors.Is(handleErr, context.DeadlineExceeded) {
				return nil, status.Error(codes.DeadlineExceeded, "request timed out")
			} else {
				r.logger.Err(handleErr).Msg("unexpected error handling admin request")
				s, _ := status.FromError(handleErr)
				return nil, s.Err()
			}
		}
	} else {
		return nil, status.Error(codes.Unimplemented, "invalid command")
	}

	return handleResult, nil
}
```
