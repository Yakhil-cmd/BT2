### Title
Blocklist Bypass via Bank Precompile `transfer()` — Missing `from` Address Check - (File: `x/cronos/keeper/precompiles/bank.go`)

### Summary
The `BankContract` precompile's `transfer(address sender, address recipient, uint256 amount)` method accepts an arbitrary `sender` address and moves native `evm/` denom tokens from it, but only calls `checkBlockedAddr` on the **recipient** (`to`). The `sender` (`from`) address is never validated against the Cronos blocklist. An unprivileged EVM contract that controls a given `evm/<contract>` denom can drain a blocklisted address's balance of that denom by calling the precompile directly with the blocklisted address as `sender`.

### Finding Description

In `x/cronos/keeper/precompiles/bank.go`, the `TransferMethodName` case of `BankContract.Run()`:

```go
sender := args[0].(common.Address)
recipient := args[1].(common.Address)
...
from := sdk.AccAddress(sender.Bytes())
to := sdk.AccAddress(recipient.Bytes())
if err := bc.checkBlockedAddr(to); err != nil {   // ← only `to` is checked
    return nil, err
}
denom := EVMDenom(contract.Caller())              // evm/<calling contract>
...
bc.bankKeeper.SendCoins(ctx, from, to, ...)       // from = arbitrary sender arg
``` [1](#0-0) 

Two guards are absent:

1. **No caller-authorization check**: `contract.Caller()` is never compared to `sender`. Any contract can pass any address as `sender` and drain its `evm/<caller>` denom balance.
2. **No blocklist check on `sender`**: `checkBlockedAddr` only calls `bc.bankKeeper.BlockedAddr(to)`, which tests Cosmos SDK module-account restrictions — not the Cronos operator blocklist. The `from` address is never tested against the blocklist. [2](#0-1) 

The Cronos blocklist enforcement in both the ante handler and the proposal handler inspects only the **tx signer** and the **EVM `to` field** (the precompile address `0x64`). It never decodes calldata to inspect the `sender` argument embedded inside the bank precompile call. [3](#0-2) [4](#0-3) 

### Impact Explanation

**High — Bypass of Cronos blocklist authorization check with direct balance impact.**

A blocklisted address's native `evm/<token>` denom tokens can be drained by any party that controls the corresponding token contract. The blocklist is intended to freeze an address's ability to move funds; this path circumvents that freeze entirely for precompile-managed native balances.

### Likelihood Explanation

Requires the attacker to control the EVM contract whose address forms the denom (`evm/<contract>`), and the victim must hold a balance in that denom. This is realistic: a token issuer who pre-distributed tokens to an address before it was blocklisted retains the ability to drain it post-blocklisting. No privileged key or admin access is needed beyond normal EVM contract deployment.

### Recommendation

In the `TransferMethodName` case, add a blocklist check on `from` analogous to the existing check on `to`. Additionally, verify that `contract.Caller()` equals `sender` (or that the caller holds an explicit allowance), so that only the token owner or an authorized spender can initiate a transfer:

```go
// Verify caller is authorized to move sender's funds
if contract.Caller() != sender {
    return nil, errors.New("caller not authorized to transfer sender's funds")
}
// Check sender is not on the Cronos blocklist
if err := bc.checkBlockedAddr(from); err != nil {
    return nil, err
}
```

### Proof of Concept

1. Attacker deploys `MaliciousToken` contract at address `0xATK`. This contract calls `bankPrecompile.transfer(victim, attacker, amount)`.
2. `MaliciousToken` previously minted `evm/0xATK` tokens to `VICTIM` (e.g., via the bank precompile's `mint`).
3. Operator blocklists `VICTIM` via `MsgStoreBlockList`.
4. Attacker (non-blocklisted) sends an EVM tx:
   - **signer**: attacker (not blocked → passes ante/proposal check)
   - **to**: `0x0000000000000000000000000000000000000064` (bank precompile, not blocked)
   - **calldata**: `transfer(VICTIM, ATTACKER, amount)`
5. `ValidateTransaction` in `app/proposal.go` passes — it only checks signer and `ethTx.To()`.
6. Bank precompile executes: `checkBlockedAddr(ATTACKER)` passes (attacker is not a module account); `sender` (`VICTIM`) is never checked against the Cronos blocklist.
7. `bankKeeper.SendCoins(VICTIM, ATTACKER, amount)` succeeds — VICTIM's `evm/0xATK` balance is drained despite being blocklisted. [5](#0-4) [6](#0-5)

func (bc *BankContract) checkBlockedAddr(addr sdk.AccAddress) error {
	to, err := sdk.AccAddressFromBech32(addr.String())
	if err != nil {
		return err
	}
	if bc.bankKeeper.BlockedAddr(to) {
		return errorsmod.Wrapf(errortypes.ErrUnauthorized, "%s is not allowed to receive funds", to.String())
	}
	return nil
}
