### Title
Missing Minimum Threshold Enforcement in FROST Signing Allows Protocol Corruption - (File: src/frost/mod.rs)

### Summary
`assert_sign_inputs` in `src/frost/mod.rs` omits the lower-bound check (`threshold >= 2`) that `assert_key_invariants` in `src/dkg.rs` enforces for key generation. A caller can pass `threshold = 1` (or `0`) to every FROST signing entry point, allowing the coordinator to aggregate a signature from a single share and bypass the threshold security property.

### Finding Description
`assert_key_invariants` enforces both bounds on the threshold:

```rust
// src/dkg.rs  lines 573-582
if threshold > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { … });
}
if threshold < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
}
``` [1](#0-0) 

`assert_sign_inputs`, the shared validation gate for all FROST signing functions, only enforces the upper bound:

```rust
// src/frost/mod.rs  lines 144-150
if threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { … });
}
// ← no lower-bound check; threshold = 1 or 0 passes silently
``` [2](#0-1) 

Every FROST signing entry point delegates to `assert_sign_inputs` and therefore inherits the missing check:

- `frost::eddsa::sign::sign_v1` — line 47 [3](#0-2) 
- `frost::eddsa::sign::sign_v2` — line 73 [4](#0-3) 
- `frost::redjubjub::sign::sign` — line 50 [5](#0-4) 

The same gap exists in the FROST `presign` function, which also only checks the upper bound: [6](#0-5) 

When `threshold = 1` is accepted, `construct_key_package` embeds it into the FROST `KeyPackage`:

```rust
// src/frost/eddsa/sign.rs  lines 360-368
Ok(KeyPackage::new(
    identifier,
    signing_share,
    verifying_share,
    *verifying_key,
    u16::try_from(threshold.value())…?,
))
``` [7](#0-6) 

FROST's `aggregate` uses the threshold stored in the `KeyPackage` to decide how many signature shares are required. With `threshold = 1`, the coordinator needs only its own share to call `aggregate`, skipping collection from all other participants.

### Impact Explanation
A malicious coordinator invoking `sign_v1`, `sign_v2`, or `redjubjub::sign` with `threshold = 1` can call `aggregate` after collecting only its own signature share. The Lagrange coefficient for a singleton signing set is `1`, but the coordinator's secret share was generated under a polynomial of degree `keygen_threshold − 1 ≥ 1`. The aggregated `s` value therefore does not reconstruct the correct group secret, producing a signature that fails verification. Honest participants who completed their round-2 work receive a `None` or an invalid `SignatureOption` as the protocol output — a corrupted, unusable signing result. This maps to **High: Corruption of signing outputs so honest parties accept inconsistent or unusable cryptographic outputs**. [8](#0-7) 

### Likelihood Explanation
The threshold parameter is caller-supplied with no type-level minimum. Any integrator or malicious coordinator who passes `threshold = 1` (e.g., by constructing `ReconstructionLowerBound::from(1usize)`) triggers the issue immediately. No cryptographic capability or key material beyond normal participation is required. [9](#0-8) 

### Recommendation
Add the same lower-bound guard that `assert_key_invariants` already uses, inside `assert_sign_inputs` and inside the FROST `presign` function:

```rust
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

Apply the same fix to `frost::presign` (line 72 area in `src/frost/mod.rs`). [6](#0-5) 

### Proof of Concept

```rust
// Attacker-controlled coordinator calls sign_v1 with threshold = 1
let result = frost::eddsa::sign::sign_v1(
    &participants,          // 3 honest participants
    1usize,                 // threshold = 1 — passes assert_sign_inputs silently
    coordinator,
    coordinator,
    keygen_output,
    message,
    rng,
)?;
// Protocol proceeds; coordinator aggregates with only its own share.
// KeyPackage threshold = 1 → frost::aggregate requires only 1 SignatureShare.
// Resulting signature is cryptographically invalid (wrong Lagrange basis),
// corrupting the signing output for all honest participants.
```

The root cause is the absent `threshold < 2` guard in `assert_sign_inputs` (`src/frost/mod.rs` lines 120–160), directly analogous to the `approveAndCall` pattern of omitting a necessary bound check and thereby granting more capability than the security model intends. [8](#0-7)

pub fn assert_sign_inputs(
    participants: &[Participant],
    threshold: impl Into<ReconstructionLowerBound>,
    me: Participant,
    coordinator: Participant,
) -> Result<ParticipantList, InitializationError> {
    let threshold = threshold.into();
    if participants.len() < 2 {
        return Err(InitializationError::NotEnoughParticipants {
            participants: participants.len(),
        });
    }
    let Some(participants) = ParticipantList::new(participants) else {
        return Err(InitializationError::DuplicateParticipants);
    };

    // ensure my presence in the participant list
    if !participants.contains(me) {
        return Err(InitializationError::MissingParticipant {
            role: "self",
            participant: me,
        });
    }

    // validate threshold
    if threshold.value() > participants.len() {
        return Err(InitializationError::ThresholdTooLarge {
            threshold: threshold.value(),
            max: participants.len(),
        });
    }

    // ensure the coordinator is a participant
    if !participants.contains(coordinator) {
        return Err(InitializationError::MissingParticipant {
            role: "coordinator",
            participant: coordinator,
        });
    }
    Ok(participants)
}
