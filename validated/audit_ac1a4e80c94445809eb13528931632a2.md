### Title
`EVM.mustRun()` Does Not Revert on EVM Execution Failure, Enabling Asset Transfer Without Payment - (File: `fvm/evm/stdlib/contract.cdc`)

### Summary

`EVM.mustRun()` is the protocol-provided "safe" wrapper around `EVM.run()`, analogous to OpenZeppelin's `SafeERC20`. However, it explicitly permits `status: failed` EVM results to pass through without reverting the enclosing Cadence transaction. Any Cadence transaction that uses `EVM.mustRun()` to execute an EVM payment (e.g., an ERC20 `transferFrom`) and then transfers a Cadence asset assumes the wrapper guarantees success — but it does not. The EVM payment can silently fail while the Cadence asset transfer proceeds, resulting in cross-VM asset loss.

### Finding Description

`EVM.mustRun()` in `fvm/evm/stdlib/contract.cdc` is defined as:

```cadence
/// mustRun runs the transaction using EVM.run
/// It will rollback if the tx execution status is unknown or invalid.
/// Note that this method does not rollback if transaction
/// is executed but an vm error is reported as the outcome
/// of the execution (status: failed).
access(all)
fun mustRun(tx: [UInt8], coinbase: EVMAddress): Result {
    let runResult = self.run(tx: tx, coinbase: coinbase)
    assert(
        runResult.status == Status.failed || runResult.status == Status.successful,
        message: "EVM.mustRun(): The provided transaction is not valid for execution"
    )
    return runResult
}
```

The `assert` condition is `status == failed || status == successful`. This means:

- `status: successful` → passes (correct)
- `status: failed` → **also passes** (incorrect for a payment guard)
- `status: invalid` → reverts (correct)
- `status: unknown` → reverts (correct)

`status: failed` is the EVM outcome when an EVM transaction executes but the VM reports an error — for example, an ERC20 `transferFrom` that reverts due to insufficient balance or allowance, or any EVM call that runs out of gas. The EVM state changes for that transaction are rolled back inside the EVM, but the Cadence transaction is **not** reverted.

The `EVM.Status` enum documents this precisely:

```cadence
/// Returned when execution of an evm transaction/call
/// has been successful but the vm has reported an error in
/// the outcome of execution (e.g. running out of gas).
/// A failed tx/call is included in a block.
access(all) case failed
```

The name `mustRun` implies the EVM transaction must succeed, mirroring the intent of `SafeERC20` in the original report. But unlike `SafeERC20`, it does not revert on `status: failed`.

### Impact Explanation

A Cadence transaction author who uses `EVM.mustRun()` to execute an EVM-side payment (e.g., an ERC20 `transferFrom` from a buyer to a seller) before releasing a Cadence NFT or fungible token will be vulnerable. If the EVM payment fails (reverts inside the EVM), `mustRun` returns without reverting the Cadence transaction. The Cadence asset transfer that follows will execute, delivering the asset to the attacker without any payment having occurred. This is a direct cross-VM asset loss: a Cadence resource (NFT or vault) is transferred to an attacker who paid nothing on the EVM side.

### Likelihood Explanation

The likelihood is realistic. `EVM.mustRun()` is `access(all)` and is the only protocol-provided wrapper that filters out `invalid` transactions. A contract author building a cross-VM marketplace or payment flow on Flow EVM would naturally reach for `mustRun` over the bare `EVM.run()`, expecting it to guarantee execution success. The misleading name and the fact that the function does filter some failure modes (invalid/unknown) makes it easy to assume it covers all failure modes. An attacker can craft an EVM transaction that passes validation (correct nonce, sufficient gas for intrinsic cost) but reverts during execution (e.g., ERC20 with zero allowance), triggering `status: failed` and bypassing the guard.

### Recommendation

The `mustRun` function should assert `status == Status.successful` rather than `status == Status.failed || status == Status.successful`. If a `failed`-but-executed result is intentionally allowed (e.g., for gas accounting purposes), the function should be renamed to make clear it does not guarantee success, and a separate `mustSucceed` wrapper should be provided that reverts on any non-successful status:

```cadence
fun mustSucceed(tx: [UInt8], coinbase: EVMAddress): Result {
    let runResult = self.run(tx: tx, coinbase: coinbase)
    assert(
        runResult.status == Status.successful,
        message: "EVM.mustSucceed(): EVM transaction did not succeed"
    )
    return runResult
}
```

### Proof of Concept

1. Attacker deploys an ERC20 token on Flow EVM with zero allowance granted to the bridge/marketplace contract.
2. Attacker submits a Cadence transaction that calls `EVM.mustRun()` with an EVM transaction encoding `erc20.transferFrom(attacker, seller, price)`.
3. The EVM transaction passes validation (correct nonce, gas ≥ intrinsic cost) but reverts during execution because allowance is zero → `status: failed`.
4. `mustRun` asserts `status == failed || status == successful` → **assertion passes**, Cadence transaction continues.
5. The Cadence transaction proceeds to transfer the NFT or vault to the attacker.
6. Attacker receives the Cadence asset; no EVM payment occurred.
