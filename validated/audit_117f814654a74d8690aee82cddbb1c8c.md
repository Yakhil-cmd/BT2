### Title
Unbounded O(n) Loop in `move_to_block` Causes Gas Exhaustion and Permanent Hashchain Freeze — (`engine-hashchain/src/hashchain.rs`)

---

### Summary

Every Aurora transaction that uses the hashchain path calls `load_hashchain`, which invokes `move_to_block` to catch up the stored hashchain state to the current NEAR block height. `move_to_block` contains an unbounded `while` loop that iterates once per block in the gap, performing a keccak hash each iteration. If the gap between the last processed Aurora transaction and the current NEAR block height grows large enough, the loop exhausts the 300 TGas NEAR gas limit. Because the state is not committed on failure, the hashchain height is never advanced, the gap only grows larger on subsequent calls, and all Aurora transactions that route through `with_hashchain` / `with_logs_hashchain` — including `submit` — are permanently blocked until an admin intervenes.

---

### Finding Description

**Root cause — unbounded loop in `move_to_block`:** [1](#0-0) 

```rust
pub fn move_to_block(&mut self, next_block_height: u64) -> Result<(), BlockchainHashchainError> {
    if next_block_height <= self.current_block_height {
        return Err(BlockchainHashchainError::BlockHeightIncorrect);
    }
    while self.current_block_height < next_block_height {   // ← iterates (gap) times, no cap
        self.previous_block_hashchain = self.block_hashchain_computer.compute_block_hashchain(
            &self.chain_id,
            self.contract_account_id.as_bytes(),
            self.current_block_height,
            self.previous_block_hashchain,
        );
        self.block_hashchain_computer.clear_txs();
        self.current_block_height += 1;
    }
    Ok(())
}
```

Each iteration calls `compute_block_hashchain`, which concatenates ~370 bytes and calls `keccak`. [2](#0-1) 

**Trigger path — called unconditionally on every hashchain-protected transaction:** [3](#0-2) 

```rust
fn load_hashchain<I: IO>(io: &I, block_height: u64) -> Result<Option<Hashchain>, ContractError> {
    let mut maybe_hashchain = read_current_hashchain(io)?;
    if let Some(hashchain) = maybe_hashchain.as_mut()
        && block_height > hashchain.get_current_block_height()
    {
        hashchain.move_to_block(block_height)?;   // ← O(gap) work here
    }
    Ok(maybe_hashchain)
}
```

`load_hashchain` is invoked at the top of both `with_hashchain` and `with_logs_hashchain` before any user logic runs: [4](#0-3) [5](#0-4) 

`with_logs_hashchain` wraps `submit` — the primary EVM transaction entry point — meaning every EVM transaction submitted by any user goes through this path.

**Self-reinforcing failure:** When the NEAR runtime kills the transaction due to gas exhaustion, all state changes are reverted. The hashchain's `current_block_height` is never updated. On the next call, the gap is even larger (NEAR has produced more blocks). Every subsequent `submit` call fails for the same reason, with an ever-growing gap.

---

### Impact Explanation

Once the gap exceeds the gas budget threshold, **all EVM transactions fail**. Users cannot transfer tokens, interact with contracts, or withdraw funds from the Aurora EVM. This constitutes a **temporary freezing of funds** (High severity) lasting until an admin pauses the contract and calls `start_hashchain` with a recent block height to reset the stored state. [6](#0-5) 

The admin recovery path itself calls `move_to_block` internally, but the admin controls `args.block_height` and can set it to `current_block_height - 1`, keeping the gap to 1 iteration.

---

### Likelihood Explanation

NEAR produces approximately one block per second. Each `move_to_block` iteration costs at minimum one keccak256 hash over ~370 bytes. NEAR's gas limit is 300 TGas. Depending on WASM execution overhead, gas exhaustion can occur after as few as **30–300 iterations** (30–300 seconds of inactivity). Aurora has experienced low-activity windows of this duration. No attacker action is required beyond submitting a normal EVM transaction after such a window — any user can trigger the freeze, and once triggered, no user can un-freeze it without admin intervention.

---

### Recommendation

Cap the number of iterations in `move_to_block`. If the gap exceeds a safe maximum (e.g., 50 blocks), return a distinct error so the engine can handle it gracefully — for example, by skipping hashchain updates for the gap period rather than aborting the transaction. Alternatively, store the hashchain block height separately and advance it lazily with a fixed-cost per-transaction increment, deferring large gaps across multiple transactions.

---

### Proof of Concept

1. Deploy Aurora Engine with hashchain enabled (`start_hashchain` called at block height B).
2. Submit no Aurora transactions for N blocks (N ≥ gas-exhaustion threshold, e.g. 200 blocks ≈ 3 minutes).
3. Submit any EVM transaction via `submit`. The call enters `with_logs_hashchain` → `load_hashchain` → `move_to_block(B + N)`.
4. The `while` loop runs N iterations of keccak; NEAR runtime aborts with "gas exceeded". State is reverted.
5. Repeat step 3 — every attempt fails. The gap is now B + N + (time elapsed since step 3), growing with each passing block.
6. All user funds in the Aurora EVM are inaccessible until an admin pauses the contract and resets the hashchain. [7](#0-6) [8](#0-7)

    env: &E,
    function_name: &str,
    f: F,

where
    I: IO + Copy,
    E: Env,
    F: for<'a> FnOnce(CachedIO<'a, I>) -> Result<T, ContractError>,
{
    let block_height = env.block_height();
    let maybe_hashchain = load_hashchain(&io, block_height)?;

    let cache = RefCell::new(IOCache::default());
    let hashchain_io = CachedIO::new(io, &cache);
    let result = f(hashchain_io)?;

    if let Some(mut hashchain) = maybe_hashchain {

        hashchain.add_block_tx(

            function_name,
            &cache_ref.input,
            &cache_ref.output,
            &Bloom::default(),
        )?;
        save_hashchain(&mut io, &hashchain)?;
    }

    Ok(result)
}

pub fn start_hashchain<I: IO + Copy, E: Env>(mut io: I, env: &E) -> Result<(), ContractError> {
    let mut state = state::get_state(&io)?;
    require_paused(&state)?;
    require_key_manager_only(&state, &env.predecessor_account_id())?;

    let input = io.read_input().to_vec();
    let args = StartHashchainArgs::try_from_slice(&input).map_err(|_| errors::ERR_SERIALIZE)?;
    let block_height = env.block_height();

    // Starting hashchain must be for an earlier block
    if block_height < args.block_height {
        return Err(errors::ERR_ARGS.into());
    }

    let mut hashchain = Hashchain::new(
        state.chain_id,
        env.current_account_id(),
        args.block_height + 1,
        args.block_hashchain,
    );

    if hashchain.get_current_block_height() < block_height {
        hashchain.move_to_block(block_height)?;
    }

    hashchain.add_block_tx(
        block_height,
        function_name!(),
        &input,
        &[],
        &Bloom::default(),
    )?;
    crate::hashchain::save_hashchain(&mut io, &hashchain)?;

    state.is_paused = false;
    state::set_state(&mut io, &state)?;

    Ok(())
}
