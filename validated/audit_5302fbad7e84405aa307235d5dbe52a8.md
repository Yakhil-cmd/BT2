### Title
Blocked Addresses Can Receive CRC20/IBC-Voucher Tokens via `__CronosSendToAccount` EVM Hook, Bypassing Blocklist Destination Enforcement — (`x/cronos/keeper/evmhandlers/send_to_account.go`)

---

### Summary

Cronos enforces its blocklist at two layers: the `BlockAddressesDecorator` ante-handler (CheckTx only) and the `ProposalHandler.ValidateTransaction` (PrepareProposal/ProcessProposal). Both layers check the EVM transaction's `To()` field against the blocklist. However, the `SendToAccountHandler` EVM log hook transfers native tokens to a `recipient` address decoded from event data — not from the transaction's `To()` field — with no blocklist check. An unprivileged, non-blocked address can therefore cause a blocked address to receive CRC20/IBC-voucher tokens by calling any registered CRC20 contract that emits `__CronosSendToAccount(blockedAddress, amount)`.

---

### Finding Description

**Blocklist enforcement scope:**

The `BlockAddressesDecorator` checks both the tx signer and the EVM `To()` destination: [1](#0-0) 

Critically, this check is gated on `ctx.IsCheckTx()` — it does not run during `FinalizeBlock`. The proposal-level `ProposalHandler.ValidateTransaction` also checks the EVM `To()` field: [2](#0-1) 

**The missing check in `SendToAccountHandler`:**

When an EVM transaction calls a registered CRC20 contract that emits `__CronosSendToAccount(recipient, amount)`, the `SendToAccountHandler` processes the log and transfers native tokens to the `recipient` decoded from the event data: [3](#0-2) 

There is no check against the blocklist before calling `bankKeeper.SendCoins`. The EVM transaction's `To()` field points to the CRC20 contract address (not the blocked address), so both the ante-handler and proposal-handler blocklist checks pass without triggering.

The only guard is that the emitting contract must be registered: [4](#0-3) 

Registered CRC20/CRC21 contracts are legitimate, user-accessible contracts. Any user can call a registered contract's `send_to_account` function (or any function that emits the event) with a blocked address as the recipient.

**Same gap exists in `SendToIbcHandler`:**

The `sender` field decoded from `__CronosSendToIbc` event data is also transferred coins without a blocklist check: [5](#0-4) 

---

### Impact Explanation

This is a **High** impact finding: **Bypass of Cronos block-list authorization checks**.

The blocklist explicitly prevents EVM transactions whose `To()` field is a blocked address. The `SendToAccountHandler` creates an indirect path that achieves the same outcome — delivering tokens to a blocked address — without triggering any of the enforced checks. A blocked address can accumulate CRC20 or IBC-voucher balances through this path, directly undermining the security guarantee the blocklist is intended to provide.

---

### Likelihood Explanation

Any non-blocked address can trigger this path by calling a registered CRC20 contract. No special privilege is required. The attacker only needs to know the blocked address and have access to a registered CRC20 contract (which is publicly visible on-chain). The attack is fully unprivileged and repeatable.

---

### Recommendation

Add a blocklist check inside `SendToAccountHandler.Handle` before executing the bank transfer. The `ProposalHandler` blocklist is not accessible from the keeper layer, but the on-chain blocklist blob is available via `cronosKeeper.GetBlockList`. Alternatively, expose a `IsBlocked(addr)` method on the keeper that decrypts and checks the current blocklist, and call it before `bankKeeper.SendCoins` in all EVM log handlers that transfer tokens to event-data-derived addresses.

```go
// In SendToAccountHandler.Handle, before bankKeeper.SendCoins:
if h.cronosKeeper.IsBlocked(ctx, recipient) {
    return fmt.Errorf("recipient %s is blocked", recipient)
}
```

Apply the same guard to the `sender` address in `SendToIbcHandler.handle` and `SendCroToIbcHandler.Handle`.

---

### Proof of Concept

1. Admin adds address `B` to the blocklist via `MsgStoreBlockList`.
2. Attacker (address `A`, not blocked) holds tokens in a registered CRC20 contract `C` that has `B` as a balance holder or can be called to emit `__CronosSendToAccount(B, amount)`.
3. `A` submits an EVM transaction: `to = C` (the CRC20 contract), calling a function that emits `__CronosSendToAccount(B, 1000)`.
4. The transaction's `To()` is `C` (not blocked) → passes `BlockAddressesDecorator` and `ProposalHandler.ValidateTransaction`.
5. Transaction is included in a block. `SendToAccountHandler.Handle` is invoked.
6. `GetDenomByContract(C)` succeeds (registered contract).
7. `bankKeeper.SendCoins(C, B, coins)` executes — `B` receives 1000 tokens.
8. `B`'s balance increases despite being on the blocklist, bypassing the blocklist's destination enforcement. [6](#0-5) [7](#0-6)

func (h SendToAccountHandler) Handle(
	ctx sdk.Context,
	contract common.Address,
	topics []common.Hash,
	data []byte,
	_ func(contractAddress common.Address, logSig common.Hash, logData []byte),
) error {
	unpacked, err := SendToAccountEvent.Inputs.Unpack(data)
	if err != nil {
		// log and ignore
		h.cronosKeeper.Logger(ctx).Error("log signature matches but failed to decode", "error", err)
		return nil
	}

	denom, found := h.cronosKeeper.GetDenomByContract(ctx, contract)
	if !found {
		return fmt.Errorf("contract %s is not connected to native token", contract)
	}

	contractAddr := sdk.AccAddress(contract.Bytes())
	recipient := sdk.AccAddress(unpacked[0].(common.Address).Bytes())
	coins := sdk.NewCoins(sdk.NewCoin(denom, sdkmath.NewIntFromBigInt(unpacked[1].(*big.Int))))
	err = h.bankKeeper.SendCoins(ctx, contractAddr, recipient, coins)
	if err != nil {
		return err
	}

	return nil
