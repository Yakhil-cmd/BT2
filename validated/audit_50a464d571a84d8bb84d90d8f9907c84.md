### Title
160-bit Truncated Keccak256 in `derive_near_deterministic_account_id` Enables Meet-in-the-Middle Account Poisoning - (`core/primitives/src/utils.rs`)

---

### Summary

`derive_near_deterministic_account_id` derives a deterministic account ID by taking only 20 bytes (160 bits) of the keccak256 hash of a fully user-controlled Borsh-encoded `DeterministicAccountStateInit`. An unprivileged attacker can brute-force a meet-in-the-middle collision between two distinct `state_init` values that map to the same account ID, then race to initialize the target account with malicious code and initial state before the legitimate user does. Because the runtime's first-initializer-wins rule makes re-initialization a no-op, any funds subsequently sent to that account ID are handled by the attacker's contract.

---

### Finding Description

**Root cause â€” truncated hash of user-controlled input** [1](#0-0) 

The function takes the full Borsh encoding of `DeterministicAccountStateInit` â€” which contains a `GlobalContractIdentifier` (either a 32-byte code hash or an arbitrary account-ID string) and an arbitrary `BTreeMap<Vec<u8>, Vec<u8>>` data map â€” hashes it with keccak256, and keeps only bytes `[12..32]` (160 bits). The resulting account ID is `"0s" + hex(keccak256(borsh(state_init))[12:32])`. [2](#0-1) 

Both the `code` field and the `data` map are entirely user-supplied. An attacker can enumerate an unbounded number of distinct `state_init` values by varying the `data` entries, producing a different 160-bit account ID for each.

**First-initializer-wins rule** [3](#0-2) 

Once `account.contract().is_none()` becomes false (i.e., the account has been initialized), every subsequent `DeterministicStateInit` action targeting the same account ID is silently skipped. There is no mechanism to re-initialize or overwrite the contract.

**Validation only checks receiver == derived ID, not uniqueness** [4](#0-3) 

The guard confirms that the transaction's `receiver_id` equals `derive_near_deterministic_account_id(&action.state_init)`. It does not and cannot prevent two distinct `state_init` values from hashing to the same account ID.

**Attack path (meet-in-the-middle)**

1. **Side A** â€” Attacker enumerates ~2^80 `state_init_A` values (varying `data`) and stores the resulting 160-bit account IDs in a Bloom filter.
2. **Side B** â€” Attacker enumerates `state_init_B` values (using a malicious global contract as `code`) until one of the derived IDs collides with an entry in the Side A set.
3. Attacker deploys the malicious global contract on-chain.
4. Attacker submits a `DeterministicStateInit` action targeting the collided account ID using `state_init_B` (malicious code + arbitrary data).
5. The account is now initialized with the attacker's contract.
6. When the legitimate user later submits `DeterministicStateInit` with `state_init_A`, the action is a no-op (account already has a contract).
7. Any NEAR tokens sent to the account ID â€” either before or after initialization â€” are now controlled by the attacker's contract logic.

### Impact Explanation

An attacker who finds a collision can:

- Initialize a deterministic account with arbitrary malicious contract code and initial state.
- Steal any NEAR tokens sent to that account ID before or after poisoning (the malicious contract can transfer them out via cross-contract calls).
- Permanently block the legitimate `DeterministicStateInit` from ever succeeding on that account ID.

Because deterministic accounts are designed for sharded-contract protocols where funds are routed to account IDs derived from expected initial state, a poisoned account breaks the protocol's security invariant: the account ID no longer uniquely identifies the initial state.

---

### Likelihood Explanation

The collision space is 160 bits. A meet-in-the-middle attack achieves a collision with probability ~86% using 2^81 hash evaluations. As cited in the external report, the Bitcoin network achieves ~6Ă—10^20 hashes/second, reaching 2^80 in ~33 minutes. A fraction of that power suffices over a longer window. The attack cost has been estimated at a few million USD â€” well below the value of funds that could be locked in sharded-contract protocols. The `data` field imposes no meaningful entropy restriction on the attacker's search space.

---

### Recommendation

1. **Use the full 256-bit keccak256 output** as the account identifier, or encode it in a format that preserves all 32 bytes (e.g., base58 or a new account-ID prefix that accommodates 32 bytes). This raises the collision cost to ~2^128, which is computationally infeasible.
2. **Alternatively**, include a protocol-controlled nonce or block hash in the hash input so the attacker cannot pre-compute collisions offline.
3. **At minimum**, document the 160-bit collision bound explicitly in the NEP-616 security model so protocol designers can account for it.

### Proof of Concept

**Collision search (pseudocode):**

```python
# Side A: legitimate state_init values
bloom = BloomFilter()
for i in range(2**80):
    state_init_A = DeterministicAccountStateInit(
        code=GlobalContractIdentifier.AccountId("legitimate.near"),
        data={b"key": i.to_bytes(32, "little")}
    )
    account_id = derive_id(state_init_A)  # keccak256(borsh(state_init_A))[12:32]
    bloom.add(account_id)

# Side B: attacker's state_init values (malicious global contract)
for j in range(2**80):
    state_init_B = DeterministicAccountStateInit(
        code=GlobalContractIdentifier.CodeHash(MALICIOUS_CONTRACT_HASH),
        data={b"atk": j.to_bytes(32, "little")}
    )
    account_id = derive_id(state_init_B)
    if account_id in bloom:
        # Collision found: derive_id(state_init_A) == derive_id(state_init_B)
        break
```

**On-chain exploitation:**

```
Tx 1: DeployGlobalContract(malicious_wasm, CodeHash)
Tx 2: DeterministicStateInit(state_init_B, deposit=0)
      â†’ receiver_id = collided_account_id
      â†’ account initialized with malicious contract
Tx 3 (victim): DeterministicStateInit(state_init_A, deposit=funds)
      â†’ account.contract().is_none() == false â†’ no-op
      â†’ deposit is accepted but handled by malicious contract
```

The malicious contract's `on_transfer` or any callable method can then forward the deposited funds to the attacker.

```rust
/// Returns a NEP-616 compliant deterministic account id.
/// This is a NEAR-implicit account ID which is fully defined by its initial state.
pub fn derive_near_deterministic_account_id(
    state_init: &DeterministicAccountStateInit,
) -> AccountId {
    use sha3::Digest;
    let data = borsh::to_vec(&state_init).expect("borsh must not fail");
    let hash = sha3::Keccak256::digest(&data);
    format!("0s{}", hex::encode(&hash[12..32])).parse().unwrap()
}
```
