### Title
`PresignOutput` Not Consumed After Rerandomization Enables Nonce Reuse and Private Key Extraction — (`File: src/ecdsa/robust_ecdsa/mod.rs`)

---

### Summary

`RerandomizedPresignOutput::rerandomize_presign` accepts the `PresignOutput` as a shared reference (`&PresignOutput`) rather than consuming it by value. This means the underlying presignature is never invalidated after use, allowing the same nonce material to be rerandomized and signed with multiple distinct messages. The security documentation explicitly states this leads to private key extraction via standard ECDSA nonce-reuse attacks.

---

### Finding Description

In `src/ecdsa/robust_ecdsa/mod.rs`, the function `rerandomize_presign` is defined as:

```rust
pub fn rerandomize_presign(
    presignature: &PresignOutput,   // shared reference — NOT consumed
    args: &RerandomizationArguments,
) -> Result<Self, ProtocolError>
``` [1](#0-0) 

Because `presignature` is taken by shared reference, the caller retains full ownership of the `PresignOutput` after the call returns. Nothing in the type system or runtime prevents calling `rerandomize_presign` again on the same `PresignOutput` with a different `RerandomizationArguments` (i.e., a different message hash `h` or tweak `ε`).

By contrast, the downstream `sign` function correctly takes `RerandomizedPresignOutput` **by value** (owned), so the rerandomized output is consumed and cannot be reused at the signing layer:

```rust
pub fn sign(
    ...
    presignature: RerandomizedPresignOutput,  // owned — consumed
    ...
)
``` [2](#0-1) 

The protection against reuse is applied one layer too late. The raw `PresignOutput` — which contains the secret nonce shares `alpha`, `beta`, `c`, `e` and the public nonce `R` — is never invalidated. [3](#0-2) 

The rerandomization step computes `R' = R^δ`, `α' = α·δ⁻¹`, `β' = (β + c·ε)·δ⁻¹` where `δ = HKDF(Y, ε, h, R, ρ)`. Two different calls with `(h₁, ε₁)` and `(h₂, ε₂)` produce nonces `R₁ = R^δ₁` and `R₂ = R^δ₂` that are **multiplicatively related** (`R₂ = R₁^(δ₂/δ₁)`), not independent. [4](#0-3) 

The project's own security documentation confirms the consequence:

> "If different subsets of size at least 2t+1 sign different (h, ε) values using shares derived from the same presignature, the resulting signatures use multiplicatively related nonces and the secret key can be recovered using standard ECDSA nonce-reuse attacks." [5](#0-4) 

---

### Impact Explanation

Two valid ECDSA signatures `(R₁, s₁)` and `(R₂, s₂)` produced from the same `PresignOutput` with different messages satisfy `R₂ = R₁^(δ₂/δ₁)`. This multiplicative nonce relationship is sufficient to recover the aggregate secret key `x` using standard lattice or algebraic nonce-reuse techniques. The impact is **Critical**: full extraction of the private signing key shared among all participants.

---

### Likelihood Explanation

The entry path requires no cryptographic break. A caller (e.g., a coordinator or any party holding a `PresignOutput`) can invoke `rerandomize_presign` twice on the same object with two different `RerandomizationArguments`. This can happen:

1. **Accidentally**: a coordinator retries a failed signing session with a new message hash but the same presignature batch.
2. **Maliciously**: a compromised coordinator deliberately presents different `(h, ε)` pairs to different signing subsets, each of which independently calls `rerandomize_presign` on their copy of the same `PresignOutput`.

The documentation warns against reuse but the API provides no enforcement. The analog to the external report is exact: just as `self.adminFees[i]` was read but not zeroed (allowing repeated withdrawal), `PresignOutput` is read but not consumed (allowing repeated rerandomization).

---

### Recommendation

Change `rerandomize_presign` to consume the `PresignOutput` by value, so the Rust ownership system statically prevents reuse:

```rust
// Before (vulnerable):
pub fn rerandomize_presign(
    presignature: &PresignOutput,
    args: &RerandomizationArguments,
) -> Result<Self, ProtocolError>

// After (fixed):
pub fn rerandomize_presign(
    presignature: PresignOutput,   // owned — consumed, cannot be reused
    args: &RerandomizationArguments,
) -> Result<Self, ProtocolError>
```

This mirrors the fix in the external report (resetting `adminFees[i] = 0` after withdrawal) but leverages Rust's ownership model to make reuse a compile-time error rather than a runtime check.

impl RerandomizedPresignOutput {
    pub fn rerandomize_presign(
        presignature: &PresignOutput,
        args: &RerandomizationArguments,
    ) -> Result<Self, ProtocolError> {
        if presignature.big_r != args.big_r {
            return Err(ProtocolError::IncompatibleRerandomizationInputs);
        }
        let delta = args.derive_randomness()?;
        if delta.is_zero().into() {
            return Err(ProtocolError::ZeroScalar);
        }

        // cannot be zero due to the previous check
        let inv_delta = delta.invert().unwrap();

        // delta * R
        let rerandomized_big_r = presignature.big_r * delta;

        // alpha * delta^{-1}
        let rerandomized_alpha = presignature.alpha * inv_delta;

        // (beta + c*tweak) * delta^{-1}
        let rerandomized_beta =
            (presignature.beta + presignature.c * args.tweak.value()) * inv_delta;

        Ok(Self {
            big_r: rerandomized_big_r.into(),
            alpha: rerandomized_alpha,
            beta: rerandomized_beta,
            e: presignature.e,
        })
    }
