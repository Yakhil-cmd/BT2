### Title
Missing Access Control on `deploy_erc20_token` Allows Anyone to Register Arbitrary NEP-141 → ERC-20 Mappings - (File: engine/src/contract_methods/connector.rs)

---

### Summary

The `deploy_erc20_token` function in `engine/src/contract_methods/connector.rs` performs no caller authentication. Any NEAR account can invoke it to deploy an ERC-20 contract and permanently register a NEP-141 token → ERC-20 address mapping in the Aurora Engine's state. This allows an attacker to front-run legitimate token registrations, causing bridged user funds to be credited to an attacker-chosen ERC-20 address, effectively freezing or misdirecting bridged assets.

---

### Finding Description

`deploy_erc20_token` is the privileged function responsible for bridging a NEP-141 token into Aurora by deploying a corresponding ERC-20 contract and writing the canonical `nep141 → erc20_address` mapping into engine storage. The only guard present is `require_running`, which checks that the contract is not paused. There is no `require_owner_only`, no role check, and no whitelist check. [1](#0-0) 

The code comment inside the `WithMetadata` branch even acknowledges the intended restriction — "this transaction could be executed by the owner of the contract only" — but no such restriction is enforced anywhere in the function body: [2](#0-1) 

Contrast this with `mirror_erc20_token`, the analogous function for silo deployments, which correctly enforces `require_owner_only` before proceeding: [3](#0-2) 

The WASM-exported entry point `deploy_erc20_token` in `engine/src/lib.rs` passes no caller context that would restrict access: [4](#0-3) 

After the access-control-free entry, `engine::deploy_erc20_token` is called, which deploys ERC-20 bytecode and calls `engine.register_token(address, nep141)` to write the permanent mapping: [5](#0-4) 

---

### Impact Explanation

**High — Permanent freezing of bridged funds / ERC-20 mirror accounting corruption.**

The `nep141 → erc20_address` mapping written by `deploy_erc20_token` is the authoritative record used by `ft_on_transfer` to credit users when they bridge a NEP-141 token into Aurora: [6](#0-5) 

If an attacker pre-registers a target NEP-141 (e.g., `usdc.near`) before the legitimate operator does, the ERC-20 address stored in state will be the one derived from the **attacker's** NEAR account EVM address and nonce — not the address the protocol would have chosen. All subsequent `ft_on_transfer` calls for that NEP-141 will mint tokens at the attacker-registered ERC-20 address. The legitimate ERC-20 for that NEP-141 can never be registered (the mapping slot is already occupied), permanently corrupting the bridge accounting for that token. Users' bridged NEP-141 assets are credited to an unexpected ERC-20 contract, making them effectively frozen or inaccessible through normal protocol flows.

---

### Likelihood Explanation

**High.** The function is a publicly callable WASM export with no authentication. Any NEAR account can call it at any time with an arbitrary NEP-141 account ID as input. The attacker only needs to submit a NEAR transaction before the legitimate operator registers a given token. For tokens not yet registered on Aurora, this is a straightforward front-run with no special privileges, capital, or timing constraints beyond observing the mempool or acting before a new token is onboarded.

---

### Recommendation

Add a `require_owner_only` (or equivalent role-based) check at the top of `deploy_erc20_token`, mirroring the pattern already used in `mirror_erc20_token`:

```rust
pub fn deploy_erc20_token<I: IO + Copy, E: Env, H: PromiseHandler>(
    io: I,
    env: &E,
    handler: &mut H,
) -> Result<PromiseOrValue<Address>, ContractError> {
    with_hashchain(io, env, function_name!(), |mut io| {
        let state = state::get_state(&io)?;
        require_running(&state)?;
+       require_owner_only(&state, &env.predecessor_account_id())?;
        // ... rest of function
    })
}
```

If broader access is needed (e.g., bridge relayers), introduce a dedicated role or allowlist, as the existing TODO comment in `mirror_erc20_token` already anticipates. [7](#0-6) 

---

### Proof of Concept

1. Attacker identifies a high-value NEP-141 token not yet registered on Aurora (e.g., `usdc.near`).
2. Attacker submits a NEAR transaction calling `deploy_erc20_token` on the Aurora Engine contract with input `DeployErc20TokenArgs::Legacy("usdc.near")`.
3. The function passes the only guard (`require_running`) and proceeds to call `engine::deploy_erc20_token("usdc.near", None, ...)`.
4. An ERC-20 contract is deployed at an address derived from the attacker's NEAR account EVM address; `register_token(attacker_erc20_address, "usdc.near")` writes the mapping to state.
5. A legitimate user later bridges `usdc.near` via `ft_transfer_call` → `ft_on_transfer`. The engine looks up `"usdc.near"` in the registry, finds the attacker-registered address, and mints ERC-20 tokens there.
6. The user's bridged USDC is credited to the wrong ERC-20 contract. The legitimate operator cannot overwrite the mapping without a separate privileged migration path. Bridged funds are frozen. [8](#0-7) [9](#0-8)

/// Used to bridge NEP-141 tokens from NEAR to Aurora. On Aurora the NEP-141 becomes an ERC-20.
pub fn deploy_erc20_token<I: IO + Copy, E: Env, P: PromiseHandler>(
    nep141: AccountId,
    metadata: Option<Erc20Metadata>,
    io: I,
    env: &E,
    handler: &mut P,
) -> Result<Address, DeployErc20Error> {
    let current_account_id = env.current_account_id();
    let input = setup_deploy_erc20_input(&current_account_id, metadata);
    let mut engine: Engine<_, _> = Engine::new(
        aurora_engine_sdk::types::near_account_to_evm_address(
            env.predecessor_account_id().as_bytes(),
        ),
        current_account_id,
        io,
        env,
    )
    .map_err(DeployErc20Error::State)?;

    let address = match engine.deploy_code_with_input(input, None, handler) {
        Ok(result) => match result.status {
            TransactionStatus::Succeed(ret) => {
                Address::new(H160(ret.as_slice().try_into().unwrap()))
            }
            other => return Err(DeployErc20Error::Failed(other)),
        },
        Err(e) => return Err(DeployErc20Error::Engine(e)),
    };

    sdk::log!("Deployed ERC-20 in Aurora at: {:#?}", address);
    engine
        .register_token(address, nep141)
        .map_err(DeployErc20Error::Register)?;

    Ok(address)
