### Title
Collection Guarantee Signature Never Verified — Forged Guarantees Accepted into Consensus Blocks - (File: `engine/consensus/ingestion/core.go`, `model/flow/collectionGuarantee.go`, `module/finalizer/collection/finalizer.go`)

---

### Summary

The `CollectionGuarantee.Signature` field is explicitly set to `nil` when a guarantee is constructed and is **never cryptographically verified** at any point in the consensus pipeline. The consensus ingestion engine (`validateGuarantors`) only checks that the claimed signer indices map to valid cluster members with sufficient stake weight — it never verifies that those members actually signed the collection. This is the direct analog of the Beacon-Kit bug: operations (collection guarantees) are accepted and included in finalized blocks without being verified against an authoritative cryptographic source (the actual cluster BLS signatures).

---

### Finding Description

**Root cause — `Signature` is always `nil` and never checked:**

In `module/finalizer/collection/finalizer.go`, when a collection is finalized and a `CollectionGuarantee` is constructed to push to consensus nodes, the `Signature` field is explicitly set to `nil`: [1](#0-0) 

The comment at line 187 reads: `// TODO: to remove because it's not easily verifiable by consensus nodes`. The `NewCollectionGuarantee` constructor explicitly documents that the `Signature` field is not validated: [2](#0-1) 

**Root cause — `validateGuarantors` never verifies the signature:**

In `engine/consensus/ingestion/core.go`, `validateGuarantors` decodes the `SignerIndices` to identify which cluster members are claimed as signers and checks that their combined weight meets the quorum threshold — but it **never calls any cryptographic signature verification** against `guarantee.Signature`: [3](#0-2) 

The comment at line 150–154 explicitly acknowledges this is not BFT: *"TODO: Eventually we should check the signatures... This implementation is NOT BFT."*

**Root cause — `guaranteeExtend` in protocol state also skips signature verification:**

When a consensus node validates a block proposal via `guaranteeExtend`, it calls `protocol.FindGuarantors` which only decodes signer indices — no signature check: [4](#0-3) [5](#0-4) 

**Attack path:**

A malicious actor (any node that can send a `CollectionGuarantee` message to a consensus node, or any staked collection node) can craft a `CollectionGuarantee` with:
- A valid `CollectionID` (any arbitrary collection hash, including one for transactions never actually collected or finalized by the cluster)
- A valid `ClusterChainID` pointing to a real cluster
- `SignerIndices` encoding a quorum of real cluster members (their indices are public)
- `Signature: nil` (or any bytes — it is never checked)

This guarantee will pass all checks in `OnGuarantee` → `validateOrigin` → `validateExpiry` → `validateGuarantors`, be added to the mempool, and subsequently be included in a consensus block proposal. The protocol state's `guaranteeExtend` will also accept it. The collection will then be executed by execution nodes, which will request the full collection from collection nodes — but since the collection was never actually finalized by the cluster, execution nodes will fail to retrieve it, or worse, if the attacker controls a collection node, they can serve an arbitrary collection body.

---

### Impact Explanation

A malicious staked collection node (or any node that can reach the consensus ingestion engine on the `PushGuarantees` channel) can inject forged `CollectionGuarantee` messages claiming a quorum of any cluster signed a collection that was never actually finalized by that cluster. These forged guarantees pass all validation and are included in finalized consensus blocks. This allows:

1. **Arbitrary transaction injection**: An attacker can claim any set of transactions was "guaranteed" by a cluster, causing execution nodes to attempt to execute them.
2. **Protocol state corruption / liveness failure**: Execution nodes will request the collection from the cluster; if it doesn't exist, execution stalls for that block.
3. **Unauthorized block content**: Collections not legitimately produced by any cluster quorum are permanently recorded in finalized blocks.

The impact matches the Beacon-Kit analog: operations (deposits / collection guarantees) are processed and committed to chain state without being verified against an authoritative on-chain source (deposit contract events / cluster BLS signatures).

---

### Likelihood Explanation

Any staked collection node can send `CollectionGuarantee` messages to consensus nodes via the `PushGuarantees` channel. The `validateOrigin` check only requires the sender to be an authorized collection node — it does not require the sender to be a member of the specific cluster that signed the guarantee. The `SignerIndices` of any cluster are deterministically derivable from public protocol state. Therefore, any single staked collection node can forge a guarantee claiming a full quorum of any cluster, with no cryptographic barrier. This is a realistic, low-effort attack for any participant with a staked collection node.

---

### Recommendation

1. **Verify the aggregated BLS signature** in `validateGuarantors` against the cluster's public keys. The cluster members' staking public keys are available via `cluster.Members()`. The aggregated signature over the `CollectionID` (or the canonical message) should be verified using the decoded guarantor public keys.
2. **Enforce non-nil `Signature`** in `NewCollectionGuarantee` once signature generation is implemented in `module/finalizer/collection/finalizer.go`.
3. **Remove the `Signature: nil` TODO** and implement real BLS aggregate signature generation in `MakeFinal` before pushing guarantees to consensus nodes.
4. Until full BFT signature verification is implemented, consider requiring the guarantee to reference a cluster block ID that is verifiable via a cluster chain follower on consensus nodes, as noted in the existing TODO comment.

---

### Proof of Concept

1. Attacker is a staked collection node (role `RoleCollection`, positive weight, non-ejected).
2. Attacker picks any valid `ReferenceBlockID` (a known recent block) and any `ClusterChainID` from the current epoch's cluster list.
3. Attacker encodes `SignerIndices` to represent a quorum (≥ 2/3 weight) of that cluster's members — all public information from protocol state.
4. Attacker sets `Signature: nil` (or any bytes).
5. Attacker sets `CollectionID` to the hash of any arbitrary set of transactions.
6. Attacker sends this `CollectionGuarantee` to a consensus node via the `PushGuarantees` channel.
7. `OnGuarantee` → `validateOrigin` passes (attacker is a valid collection node).
8. `validateExpiry` passes (reference block is recent).
9. `validateGuarantors` passes: signer indices decode to valid cluster members with sufficient weight; **no signature check is performed**.
10. Guarantee is added to the mempool and included in the next consensus block proposal.
11. `guaranteeExtend` in `ParticipantState.Extend` also passes (only checks signer indices, not signature).
12. The forged guarantee is finalized in a consensus block, permanently recording an unauthorized collection. [6](#0-5) [7](#0-6) [8](#0-7) [9](#0-8) [4](#0-3)

func (e *Core) OnGuarantee(originID flow.Identifier, guarantee *flow.CollectionGuarantee) error {

	span, _ := e.tracer.StartCollectionSpan(context.Background(), guarantee.CollectionID, trace.CONIngOnCollectionGuarantee)
	span.SetAttributes(
		attribute.String("originID", originID.String()),
	)
	defer span.End()

	log := e.log.With().
		Hex("origin_id", originID[:]).
		Hex("collection_id", guarantee.CollectionID[:]).
		Hex("signers", guarantee.SignerIndices).
		Logger()
	log.Info().Msg("collection guarantee received")

	guaranteeID := guarantee.ID()

	// skip collection guarantees that are already in our memory pool
	exists := e.pool.Has(guaranteeID)
	if exists {
		log.Debug().Msg("skipping known collection guarantee")
		return nil
	}

	// check collection guarantee's validity
	err := e.validateOrigin(originID, guarantee) // retrieve and validate the sender of the collection guarantee
	if err != nil {
		return fmt.Errorf("origin validation error: %w", err)
	}
	err = e.validateExpiry(guarantee) // ensure that collection has not expired
	if err != nil {
		return fmt.Errorf("expiry validation error: %w", err)
	}
	err = e.validateGuarantors(guarantee) // ensure the guarantors are allowed to produce this collection
	if err != nil {
		return fmt.Errorf("guarantor validation error: %w", err)
	}

	// at this point, we can add the guarantee to the memory pool
	added := e.pool.Add(guaranteeID, guarantee)
	if !added {
		log.Debug().Msg("discarding guarantee already in pool")
		return nil
	}
	log.Info().Msg("collection guarantee added to pool")

	e.mempool.MempoolEntries(metrics.ResourceGuarantee, e.pool.Size())
	return nil
}

// cluster, and ensure HotStuff finalization rules. Likely a cluster-specific
// version of the follower will be a good fit for this. For now, collection
// nodes independently decide when a collection is finalized and we only check
// that the guarantors are all from the same cluster. This implementation is NOT BFT.
func (e *Core) validateGuarantors(guarantee *flow.CollectionGuarantee) error {
	// get the clusters to assign the guarantee and check if the guarantor is part of it
	snapshot := e.state.AtBlockID(guarantee.ReferenceBlockID)
	epoch, err := snapshot.Epochs().Current()
	if err != nil {
		return fmt.Errorf("could not get current epoch: %w", err)
	}
	cluster, err := epoch.ClusterByChainID(guarantee.ClusterChainID)
	// reference block not found
	if errors.Is(err, state.ErrUnknownSnapshotReference) {
		return engine.NewUnverifiableInputError(
			"could not get clusters with chainID %v for unknown reference block (id=%x): %w", guarantee.ClusterChainID, guarantee.ReferenceBlockID, err)
	}
	// cluster not found by the chain ID
	if errors.Is(err, protocol.ErrClusterNotFound) {
		return engine.NewInvalidInputErrorf("cluster not found by chain ID %v: %w", guarantee.ClusterChainID, err)
	}
	if err != nil {
		return fmt.Errorf("internal error retrieving collector clusters for guarantee (ReferenceBlockID: %v, ClusterChainID: %v): %w",
			guarantee.ReferenceBlockID, guarantee.ClusterChainID, err)
	}

	// ensure the guarantors are from the same cluster
	clusterMembers := cluster.Members().ToSkeleton()

	// find guarantors by signer indices
	guarantors, err := signature.DecodeSignerIndicesToIdentities(clusterMembers, guarantee.SignerIndices)
	if err != nil {
		if signature.IsInvalidSignerIndicesError(err) {
			return engine.NewInvalidInputErrorf("could not decode guarantor indices: %w", err)
		}
		// unexpected error
		return fmt.Errorf("unexpected internal error decoding signer indices: %w", err)
	}

	// determine whether signers reach minimally required stake threshold
	threshold := committees.WeightThresholdToBuildQC(clusterMembers.TotalWeight()) // compute required stake threshold
	totalStake := guarantors.TotalWeight()
	if totalStake < threshold {
		return engine.NewInvalidInputErrorf("collection guarantee qc signers have insufficient stake of %d (required=%d)", totalStake, threshold)
	}

	return nil

func FindGuarantors(state State, guarantee *flow.CollectionGuarantee) ([]flow.Identifier, error) {
	snapshot := state.AtBlockID(guarantee.ReferenceBlockID)
	epoch, err := snapshot.Epochs().Current()
	if err != nil {
		return nil, fmt.Errorf("could not get current epoch: %w", err)
	}
	cluster, err := epoch.ClusterByChainID(guarantee.ClusterChainID)

	if err != nil {
		return nil, fmt.Errorf(
			"fail to retrieve collector clusters for guarantee (ReferenceBlockID: %v, ClusterChainID: %v): %w",
			guarantee.ReferenceBlockID, guarantee.ClusterChainID, err)
	}

	guarantorIDs, err := signature.DecodeSignerIndicesToIdentifiers(cluster.Members().NodeIDs(), guarantee.SignerIndices)
	if err != nil {
		return nil, fmt.Errorf("could not decode signer indices for guarantee %v: %w", guarantee.ID(), err)
	}

	return guarantorIDs, nil
