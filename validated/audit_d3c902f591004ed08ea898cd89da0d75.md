### Title
Incomplete `frost_core::Ciphersuite` Implementation in `BLS12381SHA256` Causes Panic During DKG, Permanently Denying CKD Key Generation - (File: src/confidential_key_derivation/ciphersuite.rs)

---

### Summary

`BLS12381SHA256` implements `frost_core::Ciphersuite` with six required hash methods (`H1`, `H2`, `H3`, `H4`, `H5`, `HID`) left as `unimplemented!()`. Because `BLS12381SHA256` also satisfies `crate::Ciphersuite`, it is a valid ciphersuite for the library's DKG protocol. The DKG protocol uses `frost_core::Signature` for proof-of-knowledge verification, which internally dispatches through the `Ciphersuite` hash interface. When `BLS12381SHA256` is the concrete type, any such dispatch panics unconditionally, permanently preventing DKG completion and therefore CKD initialization for all honest parties.

---

`crate::Ciphersuite` is defined as a supertrait of `frost_core::Ciphersuite`:

```rust

``` [3](#0-2) 

This makes `BLS12381SHA256` a fully valid type argument wherever `crate::Ciphersuite` is required — including the DKG protocol in `src/dkg.rs`, which is the prerequisite for producing the `KeygenOutput<BLS12381SHA256>` that `ckd()` requires as input. [4](#0-3) 

The DKG protocol imports and uses `frost_core::{Signature, SigningKey, VerifyingKey, Challenge, ...}` and calls `internal_verify_proof_of_knowledge` for every participant's Schnorr proof-of-knowledge. The `frost_core` Schnorr proof verification path dispatches through `C::H1` (or `C::H2`/`C::H3` depending on the internal challenge derivation) to compute the Fiat-Shamir challenge. With `C = BLS12381SHA256`, this dispatch hits `unimplemented!()` and panics. [5](#0-4) [6](#0-5) 

The `unimplemented!()` macro is not a graceful `Err(...)` return — it is an unconditional `panic!`, which unwinds the thread and terminates the protocol without any recoverable error path.

---

### Impact Explanation

**High — Permanent denial of key generation for honest parties.**

Because `ckd()` requires a pre-existing `KeygenOutput<BLS12381SHA256>` (produced by running DKG with `BLS12381SHA256`), and because that DKG panics during proof-of-knowledge verification, no valid `KeygenOutput` can ever be produced. This permanently blocks all honest parties from initializing CKD, regardless of their inputs or behavior. The panic propagates to the caller with no opportunity for recovery or retry.

---

### Likelihood Explanation

**High.** The DKG proof-of-knowledge verification round is not optional — it executes for every participant in every DKG run. Any attempt to run DKG with `BLS12381SHA256` as the ciphersuite will reach the `frost_core` Schnorr challenge computation and panic. No malicious input is required; the panic is triggered by the normal, honest protocol flow.

---

### Recommendation

1. **Implement all required `frost_core::Ciphersuite` hash functions** (`H1`–`H5`, `HID`) for `BLS12381SHA256` with correct domain-separated hash-to-scalar derivations consistent with the BLS12-381 ciphersuite specification.
2. **Alternatively**, introduce a separate internal type (e.g., `BLS12381SHA256CKDOnly`) that does *not* implement `crate::Ciphersuite`, preventing it from being passed to the generic DKG entry points. The CKD protocol can then use this restricted type directly without exposing an incomplete ciphersuite to the DKG layer.
3. **Add a compile-time or test-time assertion** that panics if any `unimplemented!()` stub is reachable from a DKG execution path, so regressions are caught before deployment.

---

### Proof of Concept

```
1. Library caller invokes the DKG with BLS12381SHA256 as the ciphersuite
   (required to produce KeygenOutput<BLS12381SHA256> for CKD).

2. DKG Round 2: each participant broadcasts a Schnorr proof-of-knowledge
   over their polynomial commitment.

3. DKG Round 3: verify_proof_of_knowledge() is called for each peer's proof.
   This calls internal_verify_proof_of_knowledge(), which delegates to
   frost_core's Schnorr verification.

4. frost_core's Schnorr verifier computes the Fiat-Shamir challenge via C::H1(...)
   (or C::H2/C::H3 depending on the internal derivation path).

5. BLS12381SHA256::H1 executes `unimplemented!()`, causing an unconditional panic.

6. The DKG thread unwinds. No KeygenOutput is produced.
   All subsequent calls to ckd() fail at initialization because no valid
   KeygenOutput<BLS12381SHA256> can ever exist.

Result: CKD is permanently unavailable for all honest participants.

/// performing reshare and does not exist in the set of old participants
fn verify_proof_of_knowledge<C: Ciphersuite>(
    session_id: &HashOutput,
    domain_separator: &mut DomainSeparator,
    threshold: ReconstructionLowerBound,
    participant: Participant,
    old_participants: Option<ParticipantList>,
    commitment: &VerifiableSecretSharingCommitment<C>,
    proof_of_knowledge: Option<&Signature<C>>,
) -> Result<(), ProtocolError> {
    let threshold = threshold.value();
    match proof_of_knowledge {
        // if participant did not send anything but he is actually an old participant
        None => {
            // if basic dkg or participant is old
            if old_participants.is_none_or(|p| p.contains(participant)) {
                return Err(ProtocolError::MaliciousParticipant(participant));
            }
            // since previous line did not abort, then we know participant is new indeed
            // check the commitment length is threshold - 1
            if commitment.coefficients().len() != threshold - 1 {
                return Err(ProtocolError::IncorrectNumberOfCommitments);
            }
            // nothing to verify
            Ok(())
        }
        // now we know the proof is not none
        Some(proof_of_knowledge) => {
            // if participant sent something but he is actually a new participant
            if old_participants.is_some_and(|p| !p.contains(participant)) {
                return Err(ProtocolError::MaliciousParticipant(participant));
            }
            // since the previous did not abort, we know the participant is old or we are dealing with a dkg
            if commitment.coefficients().len() != threshold {
                return Err(ProtocolError::IncorrectNumberOfCommitments);
            }

            // creating an identifier as required by the syntax of verify_proof_of_knowledge of frost_core
            internal_verify_proof_of_knowledge(
                session_id,
                domain_separator,
                participant,
                commitment,
                proof_of_knowledge,
            )
        }
    }
}
