### Title
Missing Identity Element Validation for `app_pk` Allows Coordinator to Learn Confidential Derived Key - (File: src/confidential_key_derivation/protocol.rs)

### Summary
The `ckd()` entry point and `compute_signature_share()` in the CKD protocol accept a caller-supplied `app_pk: PublicKey` (a BLS12-381 G1 projective point) without checking whether it is the group identity element. When a caller passes the identity point, the ElGamal masking term collapses to zero, causing the coordinator to directly observe the confidential derived key `msk · H(pk ‖ app_id)` — the exact secret the protocol is designed to hide from every MPC node.

### Finding Description
The CKD protocol is designed so that each participant computes:

```
big_s  = x_i · H(pk ‖ app_id)          // share of the derived key
big_y  = y · G                           // ephemeral ElGamal public key
big_c  = big_s + app_pk · y             // ElGamal ciphertext
```

The coordinator aggregates the Lagrange-normalized shares:

```
Y = Σ λ_i · big_y_i
C = Σ λ_i · big_c_i  =  msk · H(pk ‖ app_id)  +  app_pk · Y
```

The client unmasks with their secret `a` (where `app_pk = a · G`) to recover `msk · H(pk ‖ app_id)`. The coordinator sees `C` and `Y` but cannot isolate the derived key because they do not know `a`.

When `app_pk = identity` (i.e., `a = 0`):

```
big_c  = big_s + identity · y  =  big_s
C      = Σ λ_i · big_s_i       =  msk · H(pk ‖ app_id)
```

The masking term vanishes entirely. The coordinator's aggregated `C` **is** the confidential derived key, with no further computation required.

The `ckd()` function validates participant membership and coordinator presence but performs no check on `app_pk`: [1](#0-0) 

`compute_signature_share()` uses `app_pk` directly in the ElGamal ciphertext without an identity check: [2](#0-1) 

The `CKDOutput` type and `unmask` helper confirm the coordinator receives the full `(Y, C)` pair: [3](#0-2) 

### Impact Explanation
**Critical — disclosure of confidential derived secrets.**

The CKD protocol's stated security goal is that MPC nodes (including the coordinator) never learn `msk · H(pk ‖ app_id)`. Passing `app_pk = identity` causes the coordinator to directly observe this value after a single protocol execution. The derived key is the output that TEE applications are meant to receive exclusively; its disclosure to the coordinator breaks the protocol's confidentiality guarantee entirely.

### Likelihood Explanation
The `app_pk` parameter is supplied by the calling application (an unprivileged library caller). No runtime check prevents passing `ElementG1::identity()`. The identity element is the default/zero value of the type and is trivially constructable. Any caller — whether malicious or simply mistaken — can trigger this path. The coordinator is one of the MPC nodes and is present in every protocol execution, so the leakage is immediate and unconditional.

### Recommendation
Add an explicit identity-element check on `app_pk` inside `ckd()` before the protocol is launched, and mirror the check inside `compute_signature_share()` as a defense-in-depth guard:

```rust
// in ckd(), after participant validation:
if app_pk == ElementG1::identity() {
    return Err(InitializationError::BadParameters(
        "app_pk must not be the identity element".to_string(),
    ));
}
```

```rust
// in compute_signature_share(), before use:
if app_pk == ElementG1::identity() {
    return Err(ProtocolError::IdentityElement);
}
```

This mirrors the pattern already used elsewhere in the codebase for identity-element rejection: [4](#0-3)

use elliptic_curve::Group;

// Attacker (malicious client) passes the identity point as app_pk
let app_pk = G1Projective::identity();  // a = 0, so app_pk = 0·G

// Protocol runs normally; inside compute_signature_share:
//   big_c = big_s + identity * y = big_s

// Coordinator aggregates:
//   C = Σ lambda_i * big_s_i = msk * H(pk || app_id)
//
// The coordinator now holds the confidential derived key directly.
// No secret key 'a' is needed to unmask because C is already unmasked.
