### Title
Stale-Pending Fallback Silently Zeroes `l1_data_gas_price` / `l2_gas_price` in Fee Estimates — (`crates/apollo_rpc/src/v0_8/api/api_impl.rs`)

---

### Summary

When `read_pending_data` detects a `parent_block_hash` mismatch it synthesises a fallback `DeprecatedPendingBlock`. That struct has no `l1_data_gas_price` or `l2_gas_price` fields, so both prices silently become zero. `create_block_context` then clamps them to `NonzeroGasPrice::MIN` (= 1). Every `estimateFee` / `simulateTransactions` call with `block_id=Pending` made during this window returns `l1_data_gas_price = 1` and `l2_gas_price = 1` regardless of the actual latest-block prices.

---

### Finding Description

**Step 1 — `read_pending_data` fallback path** [1](#0-0) 

When `pending_data.block.parent_block_hash() != latest_header.block_hash` the function constructs a `DeprecatedPendingBlock` using `..Default::default()`. It explicitly copies `eth_l1_gas_price` / `strk_l1_gas_price` from the latest header, but `DeprecatedPendingBlock` has no `l1_data_gas_price` or `l2_gas_price` fields at all: [2](#0-1) 

**Step 2 — accessors return zero for `Deprecated` variant** [3](#0-2) 

Both `l1_data_gas_price()` and `l2_gas_price()` return `GasPricePerToken::default()` (all-zero) for the `Deprecated` variant.

**Step 3 — conversion to `ExecutionPendingData` propagates zeros** [4](#0-3) 

`client_pending_data_to_execution_pending_data` calls those accessors directly, so `ExecutionPendingData.l1_data_gas_price` and `.l2_gas_price` are zero.

**Step 4 — `create_block_context` clamps zeros to `NonzeroGasPrice::MIN`** [5](#0-4) 

`NonzeroGasPrice::new(0).unwrap_or(NonzeroGasPrice::MIN)` silently substitutes the minimum value (1) for both prices.

**Step 5 — fee estimation reads from the corrupted block context** [6](#0-5) 

`tx_execution_output_to_fee_estimation` reads `l1_data_gas_price` and `l2_gas_price` directly from the block context, so every returned `FeeEstimation` carries price = 1 instead of the real values.

---

### Impact Explanation

Any unprivileged caller who issues `starknet_estimateFee` or `starknet_simulateTransactions` with `block_id = "pending"` during the natural window between a new block being committed to storage and the pending-data cache being refreshed receives fee estimates where `l1_data_gas_price` and `l2_gas_price` are `NonzeroGasPrice::MIN` (1) rather than the actual latest-block values. On mainnet these prices are typically orders of magnitude higher. The returned `FeeEstimation` looks authoritative (no error, valid JSON-RPC response) but is materially wrong. This fits the allowed impact: **"High — RPC fee estimation returns an authoritative-looking wrong value."**

---

### Likelihood Explanation

The stale window is a normal operational condition — it occurs on every block transition, for however long the pending-sync loop takes to refresh. No special privileges are required; any public RPC caller can hit this window by polling `estimateFee` with `block_id=Pending`. The window is brief but deterministic and repeatable on every block.

---

### Recommendation

Replace the `DeprecatedPendingBlock` fallback with a `PendingBlock` (the current variant) so that `l1_data_gas_price` and `l2_gas_price` can be populated from the latest accepted block header, mirroring how `l1_gas_price` is already copied. Alternatively, change `read_pending_data` to return a `PendingBlockOrDeprecated::Current` block in the fallback branch and fill all three price fields from `latest_header.block_header_without_hash`.

---

### Proof of Concept

Logical trace (no attacker privilege required):

1. Node commits block N with `l1_data_gas_price = 1000` and `l2_gas_price = 5000`.
2. Pending-data cache still holds a block whose `parent_block_hash` points to block N-1 (natural transient state).
3. Caller sends `starknet_estimateFee([tx], block_id="pending")`.
4. `read_pending_data` detects mismatch → constructs `DeprecatedPendingBlock` → `l1_data_gas_price()` returns 0, `l2_gas_price()` returns 0.
5. `create_block_context` clamps both to `NonzeroGasPrice::MIN` = 1.
6. Returned `FeeEstimation.l1_data_gas_price = 1`, `FeeEstimation.l2_gas_price = 1` — 1000× and 5000× below actual values.
