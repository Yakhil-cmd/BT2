### Title
`protocolUnpausePool` Passes Wrong Pause Level, Leaving Pool Swap-Blocked After Protocol Unpause — (`metric-core/contracts/MetricOmmPoolFactory.sol`)

---

### Summary

`MetricOmmPoolFactory.protocolUnpausePool` calls `setPause(1)` (admin-paused) instead of `setPause(0)` (active). When the protocol unpauses a pool that was previously active (pause level 0), the pool is left in admin-paused state and all swaps remain blocked. The pool admin must separately call `unpausePool` to restore the pool — an action they never initiated and may not know is required.

---

### Finding Description

The pool tracks three pause levels:

| Level | Meaning |
|---|---|
| 0 | Active (swaps allowed) |
| 1 | Paused by admin |
| 2 | Paused by protocol |

`_checkNotPaused` reverts on **any** non-zero level: [1](#0-0) 

`protocolPausePool` accepts pools at level 0 **or** level 1 as valid starting states: [2](#0-1) 

`protocolUnpausePool` always transitions to level **1**, never to level 0: [3](#0-2) 

The correct call for a full unpause is `setPause(0)`. The wrong argument `1` is the direct analog of the seed bug's wrong function name (`endStopLoss` vs `canEndStopLoss`): both cause a critical deactivation path to silently leave the system in a broken state instead of restoring normal operation.

The factory stores no pre-pause snapshot, so it cannot distinguish whether the pool was at level 0 or level 1 before the protocol pause. The current code always writes level 1, which is only correct when the pool was already admin-paused before the protocol pause.

---

### Impact Explanation

After `protocolUnpausePool` executes on a pool that was previously active (level 0):

- `pauseLevel` is 1, so `_checkNotPaused` reverts on every `swap` call.
- `addLiquidity` and `removeLiquidity` are not gated by `whenNotPaused`, so LP operations still work, but the pool is commercially dead — no trading is possible.
- The pool admin must call `unpausePool` (level 1 → 0) to restore swaps, even though they never issued a pause. If the admin is unaware of this requirement, the pool remains swap-blocked indefinitely.

This satisfies the **broken core pool functionality causing unusable swap flows** impact gate.

---

### Likelihood Explanation

The trigger is a routine, valid protocol action: the factory owner calls `protocolUnpausePool` on any pool that was active (level 0) at the time of the protocol pause. This is the common case — most pools will be active when the protocol decides to pause them for an emergency. No attacker input is required; the bug fires on every such unpause.

---

### Recommendation

**Short term:** Change `setPause(1)` to `setPause(0)` in `protocolUnpausePool` if the intent is to fully restore the pool to active after a protocol-level pause:

```solidity
function protocolUnpausePool(address pool) external override nonReentrant onlyOwner {
    (uint8 cur,,,,,) = PoolStateLibrary._slot0(pool);
    if (cur != 2) revert InvalidPauseTransition(cur, 0);
-   IMetricOmmPoolFactoryActions(pool).setPause(1);
+   IMetricOmmPoolFactoryActions(pool).setPause(0);
}
```

**Long term:** If the two-step design (protocol unpause → admin unpause) is intentional, store the pre-protocol-pause level in factory state and restore it on unpause. Add explicit NatSpec documenting the intended transition graph and add unit tests covering the 0 → 2 → 1 path to assert swap availability after each transition.

---

### Proof of Concept

1. Pool is deployed and active (pause level 0). Swaps succeed.
2. Factory owner calls `protocolPausePool(pool)`. Pool transitions 0 → 2. Swaps revert.
3. Factory owner calls `protocolUnpausePool(pool)`. Pool transitions 2 → 1 (admin-paused). Swaps **still revert** because `pauseLevel == 1 != 0`.
4. Pool admin never called `pausePool`, yet must now call `unpausePool` (1 → 0) to restore swap functionality.

The corrupted state is `pauseLevel = 1` where `pauseLevel = 0` is the correct post-unpause value. The exact wrong value written is `1` at: [3](#0-2)
