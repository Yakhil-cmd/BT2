Audit Report

## Title
CMC `burn_and_log` Failure Silently Breaks ICP/Cycles Conservation Invariant — (File: `rs/nns/cmc/src/main.rs`)

## Summary
In `process_mint_cycles`, `process_top_up`, and `process_create_canister`, the CMC dispenses value (mints cycles, tops up a canister, or creates a canister) before burning the backing ICP. The `burn_and_log` function is explicitly typed to return `()` and swallows all errors by design. If the ICP ledger call inside `burn_and_log` fails, cycles are credited to the user while the ICP remains unburned and permanently stranded in the CMC's subaccount. The notification block is simultaneously sealed as fully successful, making the ICP irrecoverable and the cycles supply permanently unbacked by a corresponding ICP burn.

## Finding Description
`burn_and_log` at line 2017 is declared `async fn burn_and_log(...) -> ()`. The function comment at lines 2014–2016 explicitly states: *"Burning doesn't return errors — we don't want to reject the transaction notification because then it could be retried."* The ledger call at line 2040 (`call_protobuf(ledger_canister_id, "send_pb", send_args)`) can fail; the `match` at lines 2042–2047 logs the error but does not propagate it.

All three callers follow the same pattern:
- `process_create_canister` (line 1945): `burn_and_log(sub, amount).await;` followed unconditionally by `Ok(canister_id)`.
- `process_mint_cycles` (line 1968): `burn_and_log(sub, amount).await;` followed unconditionally by `Ok(NotifyMintCyclesSuccess { ... })`.
- `process_top_up` (line 2001): `burn_and_log(sub, amount).await;` followed unconditionally by `Ok(cycles)`.

The callers (`notify_mint_cycles`, `notify_create_canister`, and the top-up equivalent) then record the block index as permanently processed at lines 1305–1312 / 1411–1418:
```rust
state.blocks_notified.insert(
    block_index,
    NotificationStatus::NotifiedMint(result.clone()),
);
if is_transient_error(&result) {
    state.blocks_notified.remove(&block_index);
}
```
Because `process_mint_cycles` returns `Ok(...)` regardless of whether `burn_and_log` succeeded, `is_transient_error` evaluates to `false`, the entry is kept, and the block index is permanently sealed. No retry or refund path exists for the user.

Existing guards are insufficient: the `Processing` sentinel prevents concurrent double-notification, and `is_transient_error` only removes the entry on explicit error variants — neither addresses a silent burn failure that returns `Ok`.

## Impact Explanation
When `burn_and_log` fails, the ICP/cycles conservation invariant is broken: the cycles supply increases by the minted amount while the total ICP supply is unchanged (the N ICP sits locked in the CMC's subaccount rather than being burned). The stranded ICP cannot be recovered by the user (no retry path, no refund path) and is not automatically swept by the CMC. Repeated occurrences during ledger upgrade windows compound the imbalance. This constitutes illegal minting — cycles are issued without a corresponding reduction in ICP supply — matching the High impact category of significant ledger/NNS security impact with concrete protocol harm.

## Likelihood Explanation
The ICP ledger is an inter-canister call target that becomes transiently unavailable during canister upgrades (the ledger is briefly stopped). Ledger upgrades are proposed and executed via the NNS and are publicly observable on the NNS dashboard. An unprivileged user who monitors upgrade proposals, pre-positions ICP in the CMC's subaccount, and times a `notify_mint_cycles` (or `notify_top_up` / `notify_create_canister`) call to land during the upgrade window will trigger the bug without any special privilege. The window is narrow but recurs with every ledger upgrade. No attacker-controlled infrastructure is required.

## Recommendation
1. **Burn before dispensing (preferred).** Perform the ICP burn first; only if it succeeds, proceed to mint cycles / create canister / top up. This eliminates the window entirely and matches the correct resource-acquisition-before-resource-dispensing ordering.
2. **Propagate the burn result.** Change `burn_and_log` to return `Result<(), ...>` and propagate errors. If the burn fails, return a transient error so the notification is not sealed and the user can retry after the ledger recovers.
3. **At minimum**, if dispensing-first is kept, record the notification as a transient failure when `burn_and_log` fails so the block index is not permanently sealed and the ICP is not permanently stranded.

## Proof of Concept
1. User transfers N ICP to `AccountIdentifier(CMC_ID, Subaccount::from(&caller))` on the ICP ledger with `memo = MEMO_MINT_CYCLES`.
2. User monitors the NNS dashboard for an ICP ledger upgrade proposal and waits for the upgrade to begin (ledger briefly stopped/unavailable).
3. During the unavailability window, user calls `notify_mint_cycles { block_index, to_subaccount: None, deposit_memo: None }` on the CMC.
4. CMC executes `process_mint_cycles`: `do_mint_cycles` succeeds (cycles ledger is available) → cycles credited to user; `burn_and_log` fails (ICP ledger rejects the call) → error logged and dropped; CMC returns `Ok(NotifyMintCyclesSuccess { minted: cycles, ... })`.
5. CMC records `NotificationStatus::NotifiedMint(Ok(...))` for the block index; `is_transient_error` returns `false`; entry is kept permanently.
6. **Result:** User holds cycles worth N ICP. The N ICP sits unburned in the CMC's subaccount. The block index is permanently sealed; no refund or retry is possible. The cycles supply has increased without a corresponding ICP burn.

A deterministic integration test can reproduce this by mocking the ICP ledger to return a `TemporarilyUnavailable` reject on the `send_pb` call while allowing the cycles ledger `deposit` call to succeed, then asserting that (a) `notify_mint_cycles` returns `Ok`, (b) the block index is sealed as `NotifiedMint(Ok(...))`, and (c) the CMC subaccount balance is unchanged (ICP not burned).
