### Title
Missing Cryptographic Validation of Participant Contributions in CKD Coordinator Enables Malicious Participant to Corrupt Derived Key Output - (File: `src/confidential_key_derivation/protocol.rs`)

---

### Summary

The CKD coordinator in `do_ckd_coordinator` accumulates `(norm_big_y, norm_big_c)` contributions from all participants without any cryptographic validation. A single malicious participant can send arbitrary values, causing the coordinator to output a corrupted CKD result that all honest parties will accept as valid, analogous to `_conditionalSendValue()` accepting payment to a sanctioned address because the required check is absent.

---

### Finding Description

In `do_ckd_coordinator`, lines 50–55 of `src/confidential_key_derivation/protocol.rs`, the coordinator receives `CKDOutput` values from every participant and blindly accumulates them:

```rust
for (_, participant_output) in
    recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
{
    norm_big_y += participant_output.big_y();
    norm_big_c += participant_output.big_c();
}
```

Each honest participant is supposed to compute, in `compute_signature_share` (lines 148–182):

```
Y_i  = y_i · G
C_i  = x_i · H(pk ‖ app_id) + y_i · app_pk
norm_big_y = λ_i · Y_i
norm_big_c = λ_i · C_i
```

where `x_i` is the participant's private share and `y_i` is a fresh random scalar. There is no zero-knowledge proof or commitment binding `big_y` to `big_c`, and no check that the private share `x_i` embedded in `big_c` matches the participant's committed public share. The coordinator has no mechanism to distinguish a correctly formed contribution from an arbitrary group element pair.

The external report's pattern maps exactly: `_conditionalSendValue()` should check whether the recipient is sanctioned before transferring funds; here, `do_ckd_coordinator` should check whether each contribution is correctly formed before accumulating it. In both cases the check is absent, and the malicious party's value is accepted unconditionally.

---

### Impact Explanation

A malicious participant can send any `(big_y', big_c')` pair. The coordinator accumulates it without complaint. The final output

```
Y = Σ λ_i · Y_i    C = Σ λ_i · C_i
```

will be wrong. When the application calls `unmask(app_sk)` it computes `C − app_sk · Y`, which should equal `msk · H(pk ‖ app_id)`. With a corrupted contribution the result is

```
(Σ_{i≠j} λ_i · x_i · H(pk ‖ app_id)) + λ_j · (C_j' − app_sk · Y_j')
```

i.e., the correct master-secret-key contribution is replaced by an attacker-chosen offset. Honest parties receive and store this corrupted output as the legitimate CKD result. The derived confidential key is either unusable or controlled by the attacker.

**Allowed impact matched:** High — Corruption of CKD outputs so honest parties accept unusable cryptographic outputs.

---

### Likelihood Explanation

The CKD protocol requires **all** participants to contribute (not merely a threshold subset): `recv_from_others` waits for every other participant before the coordinator proceeds. Therefore a single malicious participant is sufficient to corrupt the output. No special privilege beyond being a listed participant is required. The attacker need only send two arbitrary `ElementG1` values instead of their honest contribution.

---

### Recommendation

Attach a zero-knowledge proof of correct ElGamal encryption to each participant's `CKDOutput`. Each participant must prove in zero knowledge that:

1. `big_y = y_i · G` for a known `y_i`, and
2. `big_c = x_i · H(pk ‖ app_id) + y_i · app_pk` where `x_i` is the private share whose public image `x_i · G2` is already committed in the DKG verifying key.

A standard Chaum–Pedersen / DLEQ proof over the BLS12-381 G1 curve suffices. The coordinator must verify each proof before calling `norm_big_y += …` and `norm_big_c += …`, and abort with `ProtocolError` on any failure, mirroring the pattern already used in triple generation (e.g., `dlogeq::verify` checks in `src/ecdsa/ot_based_ecdsa/triples/generation.rs`).

---

### Proof of Concept

1. Honest participants P1 (coordinator), P2, P3 initiate `ckd(…)`.
2. Malicious P2 overrides its protocol to send `CKDOutput { big_y: ElementG1::identity(), big_c: ElementG1::identity() }` to the coordinator instead of its correct share.
3. In `do_ckd_coordinator`, the loop at lines 50–55 executes:
   - `norm_big_y += identity` → P2's Y contribution is zeroed out.
   - `norm_big_c += identity` → P2's C contribution is zeroed out.
4. The coordinator outputs `CKDOutput` built from only P1's and P3's (Lagrange-weighted) contributions.
5. `unmask(app_sk)` returns `λ_1·x_1·H + λ_3·x_3·H` instead of `(λ_1·x_1 + λ_2·x_2 + λ_3·x_3)·H = msk·H`.
6. All honest parties store and use this incorrect confidential key, unaware of the corruption; no error is raised anywhere in the protocol. [1](#0-0) [2](#0-1) [3](#0-2)

async fn do_ckd_coordinator(
    mut chan: SharedChannel,
    participants: ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<CKDOutputOption, ProtocolError> {
    let (mut norm_big_y, mut norm_big_c) =
        compute_signature_share(&participants, me, key_pair, app_id, app_pk, rng)?;

    // Receive everyone's inputs and add them together
    let waitpoint = chan.next_waitpoint();

    for (_, participant_output) in
        recv_from_others::<CKDOutput>(&chan, waitpoint, &participants, me).await?
    {
        norm_big_y += participant_output.big_y();
        norm_big_c += participant_output.big_c();
    }
    let ckd_output = CKDOutput::new(norm_big_y, norm_big_c);
    Ok(Some(ckd_output))
}

fn compute_signature_share(
    participants: &ParticipantList,
    me: Participant,
    key_pair: &KeygenOutput,
    app_id: &AppId,
    app_pk: PublicKey,
    rng: &mut impl CryptoRngCore,
) -> Result<(ElementG1, ElementG1), ProtocolError> {
    // Ensures the value is zeroized on drop
    let private_share = Zeroizing::new(key_pair.private_share);

    // y <- ZZq* , Y <- y * G
    let y = Scalar::random(rng);

    // Ensures the value is zeroized on drop
    let y = Zeroizing::new(super::scalar_wrapper::ScalarWrapper(y));

    let big_y = ElementG1::generator() * y.0;

    // H(pk || app_id) when H is a random oracle
    let hash_point = hash_app_id_with_pk(&key_pair.public_key, app_id);

    // S <- x . H(app_id)
    let big_s = hash_point * private_share.to_scalar();

    // C <- S + y . A
    let big_c = big_s + app_pk * y.0;

    // Compute  λi := λi(0)
    let lambda_i = participants.lagrange::<BLS12381SHA256>(me)?;
    // Normalize Y and C into  (λi . Y , λi . C)
    let norm_big_y = big_y * lambda_i;
    let norm_big_c = big_c * lambda_i;
    Ok((norm_big_y, norm_big_c))
}
