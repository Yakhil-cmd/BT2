Audit Report

## Title
Caller Identity Not Verified in `refresh_buyer_tokens` Allows Forced SNS Swap Participation and Confirmation Text Bypass - (File: rs/sns/swap/canister/canister.rs)

## Summary
The `refresh_buyer_tokens` update method accepts an arbitrary `buyer` principal from the caller-supplied argument and uses it directly without verifying it matches `caller()`. Any unprivileged ingress sender can call this method with a victim's principal as `buyer` and supply the publicly-visible confirmation text, registering the victim's ICP deposit as confirmed participation without the victim ever invoking the endpoint themselves. If the swap commits, the victim's ICP is swept to the SNS treasury and they receive SNS tokens in exchange — an outcome they never explicitly consented to via the confirmation text mechanism.

## Finding Description
In `rs/sns/swap/canister/canister.rs` lines 127–143, the endpoint reads `arg.buyer` and, when non-empty, parses it as a `PrincipalId` with no check that it equals `caller()`:

```rust
let p: PrincipalId = if arg.buyer.is_empty() {
    caller_principal_id()
} else {
    PrincipalId::from_str(&arg.buyer).unwrap()  // no caller == buyer assertion
};
```

This `p` is passed directly to `swap_mut().refresh_buyer_token_e8s(p, arg.confirmation_text, ...)` in `rs/sns/swap/src/swap.rs` lines 1134–1163. That function reads the ICP balance from the subaccount derived from the supplied `buyer` principal and records the participation in `self.buyers[buyer.to_string()]`.

The only guard on the confirmation text is `validate_confirmation_text` at `rs/sns/swap/src/swap.rs` lines 363–384, which performs a plain string equality check against the SNS-init-configured text. It does not verify who is supplying the text. The confirmation text is set in the SNS initialization parameters and is publicly readable on-chain — it is not a secret.

The proto definition at `rs/sns/swap/proto/ic_sns_swap/pb/v1/swap.proto` lines 843–851 documents the `buyer` field as "If not specified, the caller is used," making the override an intentional design, but no authorization guard accompanies it. The comment on `confirmation_text` states "a **participant** should send the confirmation text," implying the participant themselves must supply it — a constraint the code does not enforce.

Exploit path:
1. Alice transfers ICP to the SNS Swap subaccount derived from her principal (normal first step of participation).
2. The SNS requires `confirmation_text = "I agree to the terms"` (public, on-chain).
3. Eve (any principal) calls `refresh_buyer_tokens({ buyer: "<Alice's principal>", confirmation_text: Some("I agree to the terms") })`.
4. The canister reads Alice's subaccount balance, passes the confirmation text validation, and writes Alice into `self.buyers` as a confirmed participant.
5. When the swap commits, Alice's ICP is swept to the SNS treasury and she receives SNS tokens — without ever having called `refresh_buyer_tokens` herself.

No existing check prevents this. The `validate_confirmation_text` function only checks string equality, not caller identity. The `validate_lifecycle_is_open` and `validate_possibility_of_direct_participation` checks are unrelated to authorization.

## Impact Explanation
This matches **High ($2,000–$10,000): Unauthorized access to ledger/canister-controlled funds with concrete user harm**. A victim who has deposited ICP into the swap subaccount (a deliberate but reversible action during the open phase) can have their participation forcibly committed by any third party. Once the swap is committed, the victim's ICP is irreversibly swept to the SNS treasury and replaced with SNS tokens — an asset exchange the victim never explicitly authorized through the confirmation text mechanism the SNS specifically configured as a consent gate. The confirmation text mechanism exists precisely to ensure informed consent; bypassing it defeats its purpose entirely.

## Likelihood Explanation
The attack requires no special privileges. The attacker needs only: (1) the victim's principal, which is observable on-chain from the ICP ledger subaccount transfer, and (2) the confirmation text, which is a public field in the SNS init parameters readable by anyone. The attack is repeatable against any principal that has ICP sitting in the swap subaccount. The only precondition — that the victim has already transferred ICP — is the normal first step of participation and is itself observable on-chain. Cost is a single ingress message.

## Recommendation
Add a caller authorization check in the canister endpoint before using the `buyer` field:

```rust
async fn refresh_buyer_tokens(arg: RefreshBuyerTokensRequest) -> RefreshBuyerTokensResponse {
    let caller = caller_principal_id();
    let p: PrincipalId = if arg.buyer.is_empty() {
        caller
    } else {
        let buyer = PrincipalId::from_str(&arg.buyer).unwrap();
        if buyer != caller {
            panic!("Caller {} is not authorized to refresh tokens for buyer {}", caller, buyer);
        }
        buyer
    };
    ...
}
```

Alternatively, remove the `buyer` override entirely and always derive the principal from `caller()`, since the only legitimate use case is a participant registering their own deposit.

## Proof of Concept
Minimal unit test plan against `rs/sns/swap/tests/swap.rs`:

1. Initialize a `Swap` in `Open` state with `confirmation_text = Some("I agree")`.
2. Mock the ICP ledger to return a non-zero balance for Alice's subaccount.
3. Call `swap.refresh_buyer_token_e8s(alice, Some("I agree"), SWAP_CANISTER_ID, &ledger)` where the caller context is Eve's principal (not Alice's).
4. Assert `Ok(...)` is returned and `swap.buyers` contains Alice's principal — demonstrating that Eve successfully registered Alice's participation by supplying the public confirmation text.
5. Confirm that without the fix, the test passes; with the caller == buyer check added, the call returns an authorization error.
