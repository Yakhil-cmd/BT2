### Title
`ExitToNear` Precompile Permanently Freezes User Funds When NEAR-Side Promise Fails Without `error_refund` Feature — (`engine-precompiles/src/native.rs`)

---

### Summary

When the `error_refund` Cargo feature is not compiled in, the `ExitToNear` precompile burns ERC-20 tokens or deducts ETH from a user's Aurora balance, then schedules a NEAR-side `ft_transfer` / `ft_transfer_call` promise with **no refund callback attached**. If that NEAR promise fails (e.g., the recipient account is not registered in the NEP-141 contract), the tokens are irrecoverable. The codebase's own integration tests explicitly document this outcome.

---

### Finding Description

`ExitToNear::run` in `engine-precompiles/src/native.rs` handles two exit paths:

**ETH (base token) exit** — ETH is deducted from the caller's Aurora balance via `context.apparent_value` when the precompile returns `Ok`. A NEAR promise is then scheduled to call `ft_transfer` on the eth-connector account.

**ERC-20 exit** — The ERC-20 contract burns tokens *before* calling the precompile. The precompile then schedules a NEAR promise to call `ft_transfer` on the corresponding NEP-141 contract.

In both paths, the refund callback is gated behind the `error_refund` feature flag:

```rust
// engine-precompiles/src/native.rs:449-454
let callback_args = ExitToNearPrecompileCallbackArgs {
    #[cfg(feature = "error_refund")]
    refund: refund_call_args(&exit_to_near_params, &exit_event),
    #[cfg(not(feature = "error_refund"))]
    refund: None,
    transfer_near: transfer_near_args,
};
``` [1](#0-0) 

When `error_refund` is absent and `transfer_near_args` is also `None` (the common case for plain `ft_transfer`), `callback_args` equals the `Default` value, so the promise is created **without any callback**:

```rust
// engine-precompiles/src/native.rs:470-483
let promise = if callback_args == ExitToNearPrecompileCallbackArgs::default() {
    PromiseArgs::Create(transfer_promise)   // ← no callback, no refund
} else {
    PromiseArgs::Callback(PromiseWithCallbackArgs { ... })
};
``` [2](#0-1) 

The `exit_to_near_precompile_callback` handler that performs the actual refund is only reachable when a `PromiseArgs::Callback` is scheduled:

```rust
// engine/src/contract_methods/connector.rs:231-239
} else if let Some(args) = args.refund {
    // Exit call failed; need to refund tokens
    let refund_result = engine::refund_on_error(io, env, state, &args, handler)?;
    ...
    Some(refund_result)
} else {
    None   // ← reached when refund is None; tokens are gone
}
``` [3](#0-2) 

The `error_refund` feature is **not** a default feature; it must be explicitly opted into:

```toml
# engine-tests/Cargo.toml:60
error_refund = ["aurora-engine/error_refund", "aurora-engine-precompiles/error_refund"]
``` [4](#0-3) 

---

### Impact Explanation

**Critical — Permanent freezing of funds.**

- **ERC-20 path**: tokens are burned inside the ERC-20 contract before the precompile is invoked. If the subsequent `ft_transfer` NEAR promise fails, the burned tokens have no mint-back path and are permanently destroyed.
- **ETH path**: ETH is deducted from the user's Aurora balance the moment the precompile returns `Ok`. If the `ft_transfer` NEAR promise fails, the ETH is permanently lost from the user's balance with no recovery.

There is no sweep function, no admin rescue path, and no on-chain state that records the failed exit for later replay.

---

### Likelihood Explanation

The trigger condition — a NEAR `ft_transfer` failing because the recipient account is not registered (has not called `storage_deposit`) in the NEP-141 contract — is a routine operational reality on NEAR. Any user who specifies a recipient that has never interacted with the NEP-141 contract will silently lose their funds. No special privilege is required; the attacker-controlled entry point is the standard ERC-20 `withdrawToNear` call or a direct call to the `ExitToNear` precompile address (`0xe9217bc70b7ed1f598ddd3199e80b093fa71124f`). [5](#0-4) 

---

### Recommendation

Enable the `error_refund` feature unconditionally in the production engine build, or restructure the code so that a refund callback is **always** attached to the NEAR promise regardless of the feature flag. The refund logic already exists in `engine::refund_on_error` and `exit_to_near_precompile_callback`; it simply needs to be reachable on every exit path. [6](#0-5) 

---

### Proof of Concept

The project's own integration tests explicitly document the fund-loss outcome when `error_refund` is absent.

**ERC-20 case** (`engine-tests/src/tests/erc20_connector.rs`):

```rust
// test_exit_to_near_refund — exits to an unregistered NEAR account
#[cfg(feature = "error_refund")]
let balance = FT_TRANSFER_AMOUNT.into();          // tokens refunded
#[cfg(not(feature = "error_refund"))]
let balance = (FT_TRANSFER_AMOUNT - FT_EXIT_AMOUNT).into(); // tokens permanently lost
``` [7](#0-6) 

**ETH case** (`engine-tests/src/tests/erc20_connector.rs`):

```rust
// test_exit_to_near_eth_refund — ft_transfer fails after ETH is deducted
#[cfg(feature = "error_refund")]
let expected_balance = Wei::new_u64(INITIAL_ETH_BALANCE);          // ETH refunded
#[cfg(not(feature = "error_refund"))]
let expected_balance = Wei::new_u64(INITIAL_ETH_BALANCE - ETH_EXIT_AMOUNT); // ETH permanently lost
``` [8](#0-7) 

Both tests confirm that without `error_refund`, the user's tokens are permanently removed from their Aurora balance with no on-chain recovery path — a direct analog to the external report's unconfigured-destination-chain fund freeze.

pub fn refund_on_error<I: IO + Copy, E: Env, P: PromiseHandler>(
    io: I,
    env: &E,
    state: EngineState,
    args: &RefundCallArgs,
    handler: &mut P,
) -> EngineResult<SubmitResult> {
    let current_account_id = env.current_account_id();
    if let Some(erc20_address) = args.erc20_address {
        // ERC-20 exit; re-mint burned tokens
        let erc20_admin_address = current_address(&current_account_id);
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, erc20_admin_address, current_account_id, io, env);

        let refund_address = args.recipient_address;
        let amount = U256::from_big_endian(&args.amount);
        let input = setup_refund_on_error_input(amount, refund_address);

        engine.call(
            &erc20_admin_address,
            &erc20_address,
            Wei::zero(),
            input,
            u64::MAX,
            Vec::new(),
            Vec::new(),
            handler,
        )
    } else {
        // ETH exit; transfer ETH back from precompile address
        let exit_address = exit_to_near::ADDRESS;
        let mut engine: Engine<_, _> =
            Engine::new_with_state(state, exit_address, current_account_id, io, env);
        let refund_address = args.recipient_address;
        let amount = Wei::new(U256::from_big_endian(&args.amount));
        engine.call(
            &exit_address,
            &refund_address,
            amount,
            Vec::new(),
            u64::MAX,
            vec![
                (exit_address.raw(), Vec::new()),
                (refund_address.raw(), Vec::new()),
            ],
            Vec::new(),
            handler,
        )
    }
