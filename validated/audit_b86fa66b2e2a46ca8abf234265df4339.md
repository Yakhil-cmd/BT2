### Title
Incomplete Token Address Validation in Pool's `change_reward_address` Enables Permanent Yield Freeze for BTC Pool Members - (File: `src/pool/pool.cairo`)

---

### Summary

The `change_reward_address` function in `pool.cairo` only validates that the new reward address is not the pool's own token contract. For BTC delegation pools, this check does **not** cover the STRK token address — yet all pool rewards are always paid in STRK. A BTC pool member can therefore set their `reward_address` to `STRK_TOKEN_ADDRESS`, causing every subsequent `claim_rewards` call to transfer STRK into the STRK token contract itself, permanently locking the yield.

---

### Finding Description

**Root cause — `src/pool/pool.cairo`, `change_reward_address`:**

```cairo
fn change_reward_address(ref self: ContractState, reward_address: ContractAddress) {
    assert!(
        self.token_dispatcher.contract_address.read() != reward_address,
        "{}",
        GenericError::REWARD_ADDRESS_IS_TOKEN,
    );
    let pool_member = get_caller_address();
    ...
    pool_member_info.reward_address = reward_address;
    self.write_pool_member_info(:pool_member, :pool_member_info);
``` [1](#0-0) 

`self.token_dispatcher` holds the pool's staked token. For a BTC pool this is the BTC token address — **not** `STRK_TOKEN_ADDRESS`. The guard therefore passes silently when `reward_address == STRK_TOKEN_ADDRESS`.

**Contrast with the staking contract**, which correctly checks every registered token:

```cairo
assert!(
    !self.does_token_exist(token_address: reward_address),
    "{}",
    GenericError::REWARD_ADDRESS_IS_TOKEN,
);
``` [2](#0-1) 

**Reward payment path — `src/pool/pool.cairo`, `claim_rewards`:**

```cairo
let reward_token = IERC20Dispatcher { contract_address: STRK_TOKEN_ADDRESS };
reward_token.checked_transfer(recipient: reward_address, amount: rewards.into());
``` [3](#0-2) 

Rewards are **always** paid in STRK regardless of the pool's staked token. Once `reward_address` is set to `STRK_TOKEN_ADDRESS`, every `claim_rewards` call silently transfers STRK into the STRK token contract, where it is irrecoverable.

---

### Impact Explanation

**High — Permanent freezing of unclaimed yield.**

Any accumulated STRK rewards transferred to the STRK token contract address are permanently locked; no recovery path exists. The pool member's entire future yield stream is also frozen because the corrupted `reward_address` persists in storage until explicitly changed.

---

### Likelihood Explanation

**Low-to-Medium.** The entry point (`change_reward_address`) is callable by any BTC pool member with no privilege requirement. The scenario is reachable by accident (copy-paste of the STRK token address as a reward destination) or by a malicious actor deliberately destroying their own yield. Because BTC pools are a supported feature of the protocol and the staking contract already enforces the correct multi-token guard, the inconsistency is a latent defect that will affect real users as BTC staking adoption grows.

---

### Recommendation

Replace the single-token check in `pool.cairo`'s `change_reward_address` with a check against all registered tokens, mirroring the staking contract's `does_token_exist` guard. At minimum, explicitly reject `STRK_TOKEN_ADDRESS` in addition to the pool's own token:

```cairo
assert!(
    self.token_dispatcher.contract_address.read() != reward_address,
    "{}", GenericError::REWARD_ADDRESS_IS_TOKEN,
);
assert!(
    reward_address != STRK_TOKEN_ADDRESS,
    "{}", GenericError::REWARD_ADDRESS_IS_TOKEN,
);
```

A cleaner fix is to call back into the staking contract's token registry (or expose a helper) so the pool inherits the same exhaustive check the staking contract already performs.

---

### Proof of Concept

1. Deploy a BTC delegation pool (staker calls `set_open_for_delegation(btc_token_address)`).
2. A delegator joins the BTC pool and accrues STRK rewards over several epochs.
3. The delegator calls `change_reward_address(STRK_TOKEN_ADDRESS)`.
   - The guard `self.token_dispatcher.contract_address.read() != reward_address` evaluates to `btc_token_address != STRK_TOKEN_ADDRESS` → **true** → assertion passes.
   - `pool_member_info.reward_address` is now set to `STRK_TOKEN_ADDRESS`.
4. The delegator (or anyone holding the pool-member or reward address role) calls `claim_rewards(pool_member)`.
   - `reward_token.checked_transfer(recipient: STRK_TOKEN_ADDRESS, amount: rewards)` executes.
   - STRK rewards are transferred into the STRK token contract and are permanently unrecoverable.
5. All future reward accruals for this pool member will also be sent to `STRK_TOKEN_ADDRESS` until the member explicitly calls `change_reward_address` again — but any rewards already transferred are gone.
