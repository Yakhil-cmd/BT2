### Title
`update_rewards` writes `last_reward_block` before `disable_rewards` check, enabling any caller to permanently freeze consensus rewards — (File: `src/staking/staking.cairo`)

---

### Summary
`update_rewards` is publicly callable by any non-zero address and accepts a `disable_rewards: bool` parameter. The function writes `last_reward_block` to storage **before** checking `disable_rewards`, meaning any caller can consume the single per-block reward slot without distributing any rewards. An attacker calling `update_rewards(any_active_staker, true)` every block permanently prevents all stakers from receiving consensus rewards.

---

### Finding Description

In `StakingRewardsManagerImpl::update_rewards` the execution order is:

1. Assert `current_block_number > last_reward_block` — only one call per block can succeed.
2. Validate the staker is active with non-zero balance.
3. **Write `last_reward_block = current_block_number`.**
4. Then check `if disable_rewards || self.is_pre_consensus() { return; }`. [1](#0-0) 

Because step 3 occurs before step 4, any caller passing `disable_rewards: true` consumes the per-block reward slot while distributing zero rewards. The `general_prerequisites()` guard only checks that the contract is unpaused and the caller is non-zero — no role restriction exists. [2](#0-1) 

This is the direct analog to the "getState changes state implicitly" class: `update_rewards` implicitly mutates the critical `last_reward_block` state variable as a side-effect even when the caller explicitly requests no reward distribution, making the state transition invisible and exploitable.

The same implicit-mutation pattern appears in `calculate_block_rewards`, which is named as a pure calculation but writes `last_calculated_epoch` and `block_rewards` to storage and calls `reward_supplier_dispatcher.update_current_epoch_block_rewards()` — a cross-contract state mutation — as a side-effect: [3](#0-2) 

---

### Impact Explanation

**High — Permanent freezing of unclaimed yield.**

An attacker who calls `update_rewards(any_active_staker, true)` in every block:
- Consumes the single per-block reward slot.
- Causes `last_reward_block` to equal the current block number.
- Prevents any legitimate `update_rewards` call from succeeding in that block (the `REWARDS_ALREADY_UPDATED` assertion fires for all subsequent callers).

All stakers' `unclaimed_rewards_own` fields stop growing, and all pool `cumulative_rewards_trace` entries stop being updated. Rewards are not merely delayed — they are permanently lost for each blocked block. [4](#0-3) 

---

### Likelihood Explanation

**High.** The function is permissionlessly callable by any non-zero address. The attacker only needs:
1. A valid active staker address (trivially discoverable on-chain via `NewStaker` events).
2. Gas to call `update_rewards` once per block.

On Starknet, block times are short and gas costs are low, making sustained griefing economically viable. No privileged key, bridge access, or external dependency is required. [5](#0-4) 

---

### Recommendation

Move the `last_reward_block` write to **after** the `disable_rewards` guard, so that consuming the per-block slot only occurs when rewards are actually distributed:

```cairo
if disable_rewards || self.is_pre_consensus() {
    return;
}
// Only consume the per-block slot when rewards are actually distributed.
self.last_reward_block.write(current_block_number);
// ... rest of reward distribution
```

Alternatively, restrict the `disable_rewards: true` path to a privileged role (e.g., `only_app_governor`), or remove the `disable_rewards` parameter entirely and handle the pre-consensus case via `is_pre_consensus()` alone.

For `calculate_block_rewards`, rename it to `update_block_rewards` or add explicit documentation that it performs storage writes, to prevent future callers from treating it as a read-only helper. [6](#0-5)

### Proof of Concept

```
Block N:
  Attacker calls: update_rewards(alice_staker, disable_rewards=true)
  → last_reward_block = N
  → returns early, no rewards distributed

Block N (same block):
  Legitimate caller calls: update_rewards(alice_staker, disable_rewards=false)
  → assert!(N > N) FAILS with REWARDS_ALREADY_UPDATED
  → alice_staker receives 0 rewards for block N

Block N+1:
  Attacker calls: update_rewards(alice_staker, disable_rewards=true)
  → last_reward_block = N+1
  → returns early, no rewards distributed

... (repeated every block) ...

Result: All stakers' unclaimed_rewards_own never increases.
        All pool cumulative_rewards_trace entries never update.
        Consensus rewards are permanently frozen for all stakers.
```
