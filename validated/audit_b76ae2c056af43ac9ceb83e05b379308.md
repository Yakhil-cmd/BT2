### Title
Incorrect Royalty Total Displayed in WalletConnect Signing Confirmation for Multi-NFT Offers - (File: `packages/gui/src/electron/commands/parseCommandDisplay.ts`)

### Summary
`formatAmountWithRoyalties` in `parseCommandDisplay.ts` incorrectly divides the fungible amount by the number of NFTs before computing each NFT's royalty contribution. This causes the `amountWithRoyalties` field shown in the WalletConnect signing confirmation dialog to be materially understated whenever a `take_offer` command involves multiple royalty-bearing NFTs. The user approves the offer believing they will spend less than they actually will.

### Finding Description

In `parseCommandDisplay.ts`, the function `formatAmountWithRoyalties` computes the total cost including royalties as follows:

```typescript
const splitAmount = amount / BigInt(royaltyPercentages.length);   // BUG: divides by NFT count
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (splitAmount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
const totalAmount = amount + royaltyAmount;
``` [1](#0-0) 

The correct formula is: each NFT's royalty = `(fullAmount × royaltyPercentage) / 10_000`. Instead, the code first splits the amount equally across all NFTs and then applies each percentage to that reduced `splitAmount`. This is the same class of accounting error as the staking contract bug: a denominator (`royaltyPercentages.length`) is incorrectly introduced into a per-unit calculation, causing the aggregate result to be wrong.

The `royaltyPercentagesForSide` helper collects all royalty percentages from the opposite side of the trade: [2](#0-1) 

These are then passed to `formatAmountWithRoyalties` via `withRoyaltyTotals`, which is called for both the `take_offer` and `create_offer_for_ids` WalletConnect commands: [3](#0-2) 

The existing test suite encodes and asserts the **wrong** value, confirming the bug is present and untested-against-correct-behavior: [4](#0-3) 

With 2 NFTs (royalties 500 = 5% and 10 = 0.1%) and 100,000,000 mojos offered:

| | Buggy (current) | Correct |
|---|---|---|
| splitAmount | 50,000,000 | — |
| royalty NFT1 | 2,500,000 | 5,000,000 |
| royalty NFT2 | 50,000 | 100,000 |
| total royalty | 2,550,000 | 5,100,000 |
| displayed total | 0.00010255 XCH | 0.0001051 XCH |

The user is shown a total ~50% lower than the actual royalty cost.

### Impact Explanation

The `amountWithRoyalties` field is the primary signal shown to the user in the WalletConnect signing confirmation dialog before they approve a `chia_wallet.take_offer` command. When the displayed total is understated, the user approves the transaction believing they will spend less XCH (or fewer CATs) than they actually will. The Chia daemon computes royalties correctly on-chain, so the actual spend exceeds what the user consented to based on the displayed figure. This fits the allowed High impact: **WalletConnect state that causes a user to approve the wrong amount**.

### Likelihood Explanation

Any WalletConnect-connected dApp that presents a multi-NFT bundle offer (two or more royalty-bearing NFTs in a single `take_offer`) triggers this path. The discrepancy grows with the number of NFTs and the magnitude of royalty percentages. No special attacker privilege is required beyond constructing or relaying a valid multi-NFT offer.

### Recommendation

Remove the `splitAmount` division. Each NFT's royalty must be applied to the full fungible amount:

```typescript
const royaltyAmount = royaltyPercentages.reduce(
  (total, royaltyPercentage) => total + (amount * BigInt(royaltyPercentage)) / 10_000n,
  0n,
);
```

Update the corresponding test assertions to reflect the corrected totals.

### Proof of Concept

Given a WalletConnect `take_offer` with:
- 2 NFTs: royalty percentages 500 (5%) and 10 (0.1%)
- Fungible side: 100,000,000 mojos (0.0001 XCH)

Current code displays `amountWithRoyalties = "0.00010255"` (confirmed by the existing test at line 320).  
Correct value is `"0.0001051"`.  
The user approves believing they spend 0.00010255 XCH; the daemon deducts 0.0001051 XCH — a ~2.5% undisclosed overspend, scaling with NFT count and royalty rates.
