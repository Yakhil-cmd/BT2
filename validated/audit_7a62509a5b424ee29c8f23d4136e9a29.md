### Title
Unprivileged Caller Can Permanently Suppress Per-Block Reward Distribution via Missing Access Control on `update_rewards` - (File: `src/staking/staking.cairo`)

---

### Summary

The `update_rewards` function is specified to be callable only by the Starkware sequencer, but the implementation contains no such access control check. Any address can call it with `disable_rewards: true`, which consumes the global `last_reward_block` slot for the current block without distributing any rewards, permanently preventing legitimate reward distribution for that block. The attack is cheap, repeatable every block, and can permanently halt all consensus reward accrual across the entire protocol.

---

### Finding Description

`update_rewards` in `src/staking/staking.cairo` is the sole mechanism for distributing per-block rewards to stakers during the consensus rewards phase. [1](#0-0) 

The function uses a **global** `last_reward_block` storage variable as a one-time-per-block gate: [2](#0-1) 

The gate check at the top of the function: [3](#0-2) 

Once `last_reward_block` equals the current block, every subsequent call to `update_rewards` in that block reverts with `REWARDS_ALREADY_UPDATED`. The slot is consumed at line 1485, **before** the `disable_rewards` guard at line 1487: [4](#0-3) 

If `disable_rewards: true` is passed, the function:
1. Writes `last_reward_block = current_block` (consuming the global slot)
2. Returns immediately without distributing any rewards

The protocol specification explicitly restricts this function to the Starkware sequencer: [5](#0-4) 

However, the implementation only calls `general_prerequisites()` (a pause check) with **no sequencer identity verification**: [6](#0-5) 

This means any unprivileged address can call `update_rewards(staker_address: any_active_staker, disable_rewards: true)` and permanently suppress that block's rewards for all stakers.

---

### Impact Explanation

**High — Permanent freezing of unclaimed yield.**

`last_reward_block` is a single global storage slot shared across all stakers. [2](#0-1) 

Once an attacker consumes it with `disable_rewards: true`, no staker can receive rewards for that block. Since the attack can be repeated every block, an attacker can permanently halt all consensus reward distribution across the entire protocol. All stakers and their delegators are denied yield they are entitled to, with no recovery path.

---

### Likelihood Explanation

**High.** The function is publicly callable with no role requirement. The only cost is gas. The attacker only needs to supply any valid active staker address (publicly readable from contract state). The attack is trivially scriptable to run every block.

---

### Recommendation

Add an access control check at the entry of `update_rewards` to enforce that only the authorized sequencer (or a designated rewards manager role) can call it, consistent with the specification at `docs/spec.md` line 1645. For example:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
    self.assert_caller_is_sequencer(); // <-- add this
    ...
}
```

Alternatively, move the `last_reward_block.write(current_block_number)` to **after** the `disable_rewards` guard so that a `disable_rewards: true` call does not consume the global slot.

---

### Proof of Concept

1. Consensus rewards are active (current epoch ≥ `consensus_rewards_first_epoch`).
2. Attacker calls `update_rewards(staker_address: <any_active_staker>, disable_rewards: true)` in block N.
3. `last_reward_block` is written to N at line 1485; function returns early at line 1487–1488 with no rewards distributed.
4. The legitimate sequencer attempts `update_rewards` in block N → reverts with `REWARDS_ALREADY_UPDATED` (line 1454–1458).
5. Block N's rewards are permanently lost for all stakers and delegators.
6. Attacker repeats every block → all consensus rewards are permanently suppressed at the cost of gas only. [4](#0-3) [3](#0-2)
