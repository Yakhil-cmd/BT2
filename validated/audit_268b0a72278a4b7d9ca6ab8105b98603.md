### Title
Global `last_reward_block` Allows Any Caller to Block Per-Block Reward Updates for All Stakers — (`src/staking/staking.cairo`)

---

### Summary

`update_rewards` in the staking contract enforces a **global** per-block guard via `last_reward_block`. Because this slot is shared across all stakers and the function is publicly callable with a `disable_rewards: true` flag, any unprivileged caller can consume the per-block reward slot without distributing any rewards, causing every staker's attestation-triggered reward update to revert with `REWARDS_ALREADY_UPDATED` in that block.

---

### Finding Description

`update_rewards` reads and writes a single global storage variable `last_reward_block`:

```cairo
fn update_rewards(
    ref self: ContractState, staker_address: ContractAddress, disable_rewards: bool,
) {
    self.general_prerequisites();
    let current_block_number = starknet::get_block_number();
    assert!(
        current_block_number > self.last_reward_block.read(),
        "{}",
        Error::REWARDS_ALREADY_UPDATED,
    );
    ...
    // Update last block rewards.
    self.last_reward_block.write(current_block_number);

    if disable_rewards || self.is_pre_consensus() {
        return;   // ← returns without distributing any rewards
    }
``` [1](#0-0) 

The guard is **not per-staker** — it is a single global slot. The function is part of the public `IStakingRewardsManager` interface and accepts an arbitrary `staker_address` and a caller-controlled `disable_rewards` flag. Passing `disable_rewards: true` causes the function to:

1. Pass the block-number guard (if first call this block).
2. Write `last_reward_block = current_block_number`.
3. Return immediately **without distributing any rewards**.

The attestation contract's `attest` flow calls `update_rewards_from_attestation_contract` on the staking contract, which shares the same `last_reward_block` storage (6 total references in `staking.cairo`, 3 of which are outside `update_rewards` itself):

```cairo
// attestation.cairo – attest()
staking_dispatcher
    .update_rewards_from_attestation_contract(
        staker_address: staking_attestation_info.staker_address(),
    );
``` [2](#0-1) 

If `last_reward_block` has already been written in the current block (by the attacker's `update_rewards` call), `update_rewards_from_attestation_contract` reverts with `REWARDS_ALREADY_UPDATED`, causing the entire `attest` transaction to revert — including the `_mark_attestation_is_done` write — so the staker's epoch attestation is not recorded and their epoch rewards are lost. [3](#0-2) [4](#0-3) 

The analog to the external report is direct: in the EVM case `msg.sender` (the intermediary contract address) is the shared key; here `last_reward_block` is the shared global slot — both cause all users to compete for a single per-block action.

---

### Impact Explanation

An attacker who calls `update_rewards(any_active_staker, disable_rewards: true)` in every block within a target staker's attestation window prevents that staker from ever successfully attesting in that epoch. The staker loses the entire epoch's block rewards. Repeated across epochs this constitutes **permanent freezing of unclaimed yield** (High).

Even without a targeted attack, two stakers whose attestation windows happen to overlap in the same block will race: only the first transaction in that block succeeds; the second staker's `attest` reverts and they lose that epoch's rewards — **temporary freezing of unclaimed yield** (High).

---

### Likelihood Explanation

**Medium.** The attestation window per staker is a bounded range of blocks (e.g., 9 blocks wide given `MIN_ATTESTATION_WINDOW = 11` and a configurable upper bound). An attacker needs to submit one cheap transaction per block in that window. The target staker's attestation window is deterministically computable from public on-chain data (stake, epoch id, staker address), so the attacker can pre-compute exactly which blocks to target. No privileged access is required. [5](#0-4) [6](#0-5) 

---

### Recommendation

1. **Restrict `update_rewards` to the attestation contract** (add an `assert_caller_is_attestation_contract` guard), eliminating the public entry point that allows `disable_rewards: true` abuse.
2. Alternatively, **track `last_reward_block` per staker** (keyed by `staker_address`) so that one staker's reward update cannot block another's.
3. If the global slot is intentional for consensus-reward accounting, separate the "block already processed" guard from the per-staker reward distribution so that a failed or skipped update for one staker does not poison the slot for all others.

---

### Proof of Concept

```
Setup:
  - Staker S has attestation window [T+11, T+20] in epoch E.
  - Attacker A is any unprivileged address.

Attack (repeated for each block B in [T+11, T+20]):
  1. Attacker A calls:
       staking.update_rewards(staker_address = S, disable_rewards = true)
     in block B (or front-runs S's attest tx in the same block).
  2. last_reward_block is written to B; no rewards are distributed.
  3. Staker S's operational address calls:
       attestation.attest(block_hash)
     in block B.
  4. attestation.attest internally calls:
       staking.update_rewards_from_attestation_contract(staker_address = S)
     which asserts current_block_number > last_reward_block → FAILS (both equal B).
  5. The entire attest tx reverts; _mark_attestation_is_done is NOT written.
  6. Staker S cannot attest in any block of their window → loses all epoch E rewards.
```
