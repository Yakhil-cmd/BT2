Audit Report

## Title
ECDSA `read_state` Envelope Uses `transaction_signature` Public Key Instead of `read_state_signature` Public Key in `construction_combine` — (`rs/rosetta-api/icp/src/request_handler/construction_combine.rs`)

## Summary
In the `SignatureType::Ecdsa` branch of `read_state_envelope` construction within `construction_combine`, `sender_pubkey` is populated from `transaction_signature.public_key.hex_bytes` (line 142) while `sender_sig` is correctly taken from `read_state_signature.hex_bytes` (line 156). The Ed25519 branch at line 121 correctly uses `read_state_signature.public_key.hex_bytes` for `sender_pubkey`. When a caller supplies two distinct ECDSA key pairs — one per signing payload — the resulting read-state envelope carries a pubkey/signature mismatch that the IC replica unconditionally rejects with HTTP 400, permanently preventing the Rosetta node from confirming that transaction's finality.

## Finding Description
`signatures_by_sig_data` is keyed by signing-payload hash, so `transaction_signature` and `read_state_signature` are independent lookups that can resolve to entries with different public keys: [1](#0-0) 

The Ed25519 `read_state_envelope` branch correctly reads `sender_pubkey` from `read_state_signature`: [2](#0-1) 

The ECDSA `read_state_envelope` branch erroneously reads `sender_pubkey` from `transaction_signature` instead: [3](#0-2) 

While `sender_sig` is correctly sourced from `read_state_signature`: [4](#0-3) 

There is no guard anywhere in `construction_combine` that enforces `transaction_signature.public_key == read_state_signature.public_key`. The IC replica's own regression test confirms that a `read_state` envelope whose `sender_pubkey` does not match the signing identity returns HTTP 400: [5](#0-4) 

**Exploit path:**
1. Attacker calls the public, unauthenticated `POST /construction/combine` with two ECDSA entries in `signatures[]`: `(tx_payload, key_A)` and `(rs_payload, key_B)`.
2. `construction_combine` builds the update envelope correctly: `sender_pubkey = DER(key_A)`, `sender_sig = sig(key_A)` → IC replica accepts; ICP transfer executes.
3. `construction_combine` builds the read-state envelope with the bug: `sender_pubkey = DER(key_A)`, `sender_sig = sig(key_B)` → IC replica returns HTTP 400.
4. The Rosetta node can never poll the status of that transaction; it remains perpetually unconfirmed from Rosetta's perspective.

## Impact Explanation
This is a **High** severity finding matching the allowed impact: *"Significant Rosetta... security impact with concrete user or protocol harm."* Any exchange or custodian using Rosetta for ICP deposit confirmation can be targeted: the attacker executes a real ICP transfer, then exploits the permanently broken read-state response to claim the deposit was never confirmed and request a refund — resulting in double-crediting. The Rosetta API is explicitly listed as an in-scope financial integration component.

## Likelihood Explanation
The endpoint is public and unauthenticated. No privilege, staked funds, or special network position is required. The attacker only needs to generate two ECDSA key pairs and craft a single `POST /construction/combine` request. The bug is invisible in the common single-key case, so it is unlikely to have been caught by standard integration tests. It is trivially repeatable.

## Recommendation
On line 142, replace `transaction_signature.public_key.hex_bytes` with `read_state_signature.public_key.hex_bytes`:

```rust
// ECDSA branch — read_state_envelope (line 142)
Secp256k1KeyPair::hex_decode_pk(
    &read_state_signature.public_key.hex_bytes,  // was: transaction_signature
)
```

Additionally, add an explicit guard after the two signature lookups that returns `ApiError::InvalidRequest` if `transaction_signature.public_key != read_state_signature.public_key`, enforcing the invariant that both envelopes must share the same sender identity. [6](#0-5) 

## Proof of Concept

// Unit test sketch
let key_a = Secp256k1KeyPair::generate();
let key_b = Secp256k1KeyPair::generate(); // distinct key

let signatures = vec![
    Signature {
        signing_payload: tx_payload,
        public_key: key_a.rosetta_pubkey(),
        hex_bytes: key_a.sign(tx_sig_data),
        signature_type: SignatureType::Ecdsa,
    },
    Signature {
        signing_payload: rs_payload,
        public_key: key_b.rosetta_pubkey(),
        hex_bytes: key_b.sign(rs_sig_data),
        signature_type: SignatureType::Ecdsa,
    },

let response = handler.construction_combine(request_with(signatures));
let signed_tx = decode(response.signed_transaction);

// Demonstrates the bug: read_state envelope carries key_A pubkey but key_B sig

assert_eq!(rs_env.sender_pubkey, Some(Blob(key_a.der_encoded()))); // BUG
