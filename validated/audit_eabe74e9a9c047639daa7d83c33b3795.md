### Title
Missing Zero-Address Validation for `reward_address` Causes Permanent Freezing of Staker Principal and Delegator Funds - (`src/staking/staking.cairo`, `src/pool/pool.cairo`)

### Summary

Neither `stake()`, `change_reward_address()` (staking), `enter_delegation_pool()`, nor `change_reward_address()` (pool) validate that the supplied `reward_address` is non-zero. Because `unstake_action()` and `exit_delegation_pool_action()` unconditionally transfer accumulated rewards to the stored `reward_address` before returning principal, a zero reward address causes those critical exit functions to revert (or silently burn rewards), permanently freezing the staker's principal and all delegators' funds.

### Finding Description

The `stake()` function in `src/staking/staking.cairo` validates that `reward_address` is not a registered token address, but performs no zero-address check:

```cairo
assert!(
    !self.does_token_exist(token_address: reward_address),
    "{}",
    GenericError::REWARD_ADDRESS_IS_TOKEN,
);
```

The same gap exists in `change_reward_address()` in the staking contract:

```cairo
fn change_reward_address(ref self: ContractState, reward_address: ContractAddress) {
    self.general_prerequisites();
    assert!(
        !self.does_token_exist(token_address: reward_address),
        "{}",
        GenericError::REWARD_ADDRESS_IS_TOKEN,
    );
    // No zero-address check
    staker_info.reward_address = reward_address;
    self.write_staker_info(:staker_address, :staker_info);
```

And in the pool's `enter_delegation_pool()` and `change_reward_address()`:

```cairo
assert!(token_address != reward_address, "{}", GenericError::REWARD_ADDRESS_IS_TOKEN);
// No zero-address check
```

When `unstake_action()` is called, it invokes `send_rewards_to_staker()` **before** returning the principal:

```cairo
self.send_rewards_to_staker(:staker_address, ref :staker_info, :token_dispatcher);
// ...
token_dispatcher.checked_transfer(recipient: staker_address, amount: staker_amount.into());
self.transfer_to_pools_when_unstake(...);
```

`send_rewards_to_staker()` unconditionally calls:

```cairo
token_dispatcher.checked_transfer(recipient: reward_address, amount: amount.into());
```

If `reward_address` is `0x0`, the ERC20 `checked_transfer` to the zero address reverts (standard ERC20 behaviour). Because `send_rewards_to_staker` is called first in `unstake_action`, the entire transaction reverts, making it impossible to ever execute `unstake_action`. The staker's principal and all delegated funds are permanently locked.

The identical pattern applies to `exit_delegation_pool_action()` in `src/pool/pool.cairo`, which transfers rewards to `reward_address` before returning the delegated principal.

### Impact Explanation

**High — Permanent freezing of staker principal and delegator funds.**

If a staker sets `reward_address = 0x0` (at `stake()` time or via `change_reward_address()`), `unstake_action()` will always revert when trying to send accumulated rewards to address zero. Because `transfer_to_pools_when_unstake()` is called after `send_rewards_to_staker()`, all delegators' funds are also frozen — they cannot be returned to the pools. The staker's own principal is equally unrecoverable. There is no admin escape hatch to override the stored `reward_address` once set.

Even if the specific ERC20 allows transfers to address zero (burning), the result is permanent loss of all unclaimed yield, which also qualifies as a high-impact finding.

### Likelihood Explanation

Any staker or delegator can trigger this by passing `reward_address: 0` to `stake()`, `enter_delegation_pool()`, or either `change_reward_address()` function. No privileged role is required. A user could do this accidentally (e.g., passing an uninitialised variable) or deliberately as a griefing attack against their own delegators. The entry path is fully reachable by an unprivileged caller.

### Recommendation

Add an explicit non-zero check for `reward_address` in all four entry points:

- `stake()` in `src/staking/staking.cairo`
- `change_reward_address()` in `src/staking/staking.cairo`
- `enter_delegation_pool()` in `src/pool/pool.cairo`
- `change_reward_address()` in `src/pool/pool.cairo`

```cairo
assert!(reward_address.is_non_zero(), "{}", GenericError::ZERO_ADDRESS);
```

The `ZERO_ADDRESS` error variant already exists in `GenericError` and is used elsewhere in the codebase.

### Proof of Concept

1. Staker calls `stake(reward_address: 0x0, operational_address: ..., amount: min_stake)` — succeeds because only `REWARD_ADDRESS_IS_TOKEN` is checked, and `0x0` is not a registered token.
2. Attestation occurs; staker accumulates non-zero `unclaimed_rewards_own`.
3. Staker calls `unstake_intent()` — succeeds.
4. After `exit_wait_window`, staker calls `unstake_action()`.
5. `unstake_action()` calls `send_rewards_to_staker()`, which calls `token_dispatcher.checked_transfer(recipient: 0x0, amount: rewards)`.
6. The ERC20 transfer to address zero reverts.
7. `unstake_action()` reverts entirely — staker's principal and all delegators' funds are permanently frozen.
