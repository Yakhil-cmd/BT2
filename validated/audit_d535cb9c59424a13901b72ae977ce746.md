### Title
Declare V2/V3 `estimate_minimal_gas_vector` Underestimates L1 Data Gas by Hardcoding `n_compiled_class_hash_updates: 0` — (`crates/blockifier/src/fee/gas_usage.rs`)

---

### Summary

`estimate_minimal_gas_vector` hardcodes `n_compiled_class_hash_updates: 0` for all `Declare` transactions. For V2/V3 Declare transactions, the actual execution produces `n_compiled_class_hash_updates: 1`. This discrepancy causes the minimal DA gas estimate to be lower than the actual DA gas consumed, violating the stated invariant that the function returns a lower bound. The bug is self-acknowledged in the codebase with a `TODO` comment.

---

### Finding Description

In `estimate_minimal_gas_vector`, the `StateChangesCount` for `Transaction::Declare(_)` is:

```rust
Transaction::Declare(_) => StateChangesCount {
    n_storage_updates: 1,
    n_class_hash_updates: 0,
    // TODO(Yoni): BLOCKIFIER-RESET: should be 1.
    n_compiled_class_hash_updates: 0,
    n_modified_contracts: 1,
},
``` [1](#0-0) 

The TODO comment explicitly acknowledges the value should be `1`. The test helper `declare_expected_state_changes_count` confirms that V2/V3 Declare transactions produce `n_compiled_class_hash_updates: 1` at execution time:

```rust
} else if version == TransactionVersion::TWO || version == TransactionVersion::THREE {
    StateChangesCount {
        n_storage_updates: 1,
        n_modified_contracts: 1,
        n_compiled_class_hash_updates: 1, // Also set compiled class hash.
        ..StateChangesCount::default()
    }
``` [2](#0-1) 

`get_onchain_data_segment_length` adds `n_compiled_class_hash_updates * 2` felts to the DA segment: [3](#0-2) 

With `n_compiled_class_hash_updates: 0` instead of `1`, the DA segment is 2 felts shorter than actual. `get_da_gas_cost` then computes blob gas as `onchain_data_segment_length * DATA_GAS_PER_FIELD_ELEMENT` (KZG DA path), so the estimate is `2 * DATA_GAS_PER_FIELD_ELEMENT` blob gas units below the actual cost. [4](#0-3) 

`check_fee_bounds` uses this estimate as the minimum acceptable resource bound for `L1DataGas`:

```rust
(
    L1DataGas,
    l1_data_gas_resource_bounds,
    minimal_gas_amount_vector.l1_data_gas,
    *l1_data_gas_price,
),
``` [5](#0-4) 

`check_fee_bounds` is called unconditionally during `perform_pre_validation_stage` when `charge_fee` is true: [6](#0-5) 

---

### Impact Explanation

**For `starknet_estimateFee` / simulation callers:** The minimal gas vector returned is too low for Declare V2/V3. Any caller using `estimate_minimal_gas_vector` directly to compute a fee estimate (e.g., gateway admission checks, client-side fee estimation) receives a wrong `l1_data_gas` value that is `2 * DATA_GAS_PER_FIELD_ELEMENT` units below the actual cost. This is a concrete wrong value returned on the RPC/estimation path.

**For `check_fee_bounds` pre-validation:** A Declare V2/V3 transaction whose `l1_data_gas` resource bound is set exactly at the (wrong) minimal estimate passes pre-validation but will have actual DA gas usage exceed the bound. Post-execution, this triggers a `MaxGasAmountExceeded` revert. The transaction is not silently accepted — it reverts — but the pre-validation gate is too lenient, allowing a class of transactions through that should be rejected earlier.

The magnitude of the underestimate is small (`2 * DATA_GAS_PER_FIELD_ELEMENT` blob gas units), but the invariant is definitively broken and the wrong value is authoritative-looking to callers.

---

### Likelihood Explanation

Every Declare V2/V3 transaction triggers this path. Any unprivileged user submitting a Declare V2/V3 transaction on a KZG DA-enabled network will receive a fee estimate that is too low. The bug is unconditional — no special state or attacker-controlled field is required beyond submitting a standard Declare V2/V3 transaction.

---

### Recommendation

Change `n_compiled_class_hash_updates` from `0` to `1` for the `Transaction::Declare(_)` arm in `estimate_minimal_gas_vector`, as the TODO comment already directs:

```rust
Transaction::Declare(_) => StateChangesCount {
    n_storage_updates: 1,
    n_class_hash_updates: 0,
    n_compiled_class_hash_updates: 1, // V2/V3 always writes compiled_class_hash.
    n_modified_contracts: 1,
},
``` [1](#0-0) 

Note: V0/V1 Declare transactions do not write a `compiled_class_hash` (they are Cairo 0 contracts). If the `Transaction::Declare` enum variant does not distinguish between versions at this point, a version check should be added, consistent with how `declare_expected_state_changes_count` handles it. [7](#0-6) 

---

### Proof of Concept

The existing test infrastructure already contains all the pieces. `declare_expected_state_changes_count` for V2/V3 returns `n_compiled_class_hash_updates: 1`, while `estimate_minimal_gas_vector` uses `0`. A test that:

1. Calls `estimate_minimal_gas_vector` on a Declare V3 `AccountTransaction` with `use_kzg_da: true`
2. Executes the same transaction
3. Reads `receipt.resources.starknet_resources.state.state_changes_for_fee.state_changes_count.n_compiled_class_hash_updates`
4. Asserts the receipt value is `1` while the estimate used `0`
5. Computes `get_da_gas_cost` with both counts and asserts the estimate's `l1_data_gas` is `2 * DATA_GAS_PER_FIELD_ELEMENT` lower than actual

...would fail on the current code and pass after the fix. The `test_declare_tx` test in `transactions_test.rs` already uses `declare_expected_state_changes_count` to build expected resources, confirming the `1` is the correct actual value. [8](#0-7)
