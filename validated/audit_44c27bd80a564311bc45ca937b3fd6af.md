### Title
Blocklisted Address Can Receive CRC20/CRC21 Tokens via Internal EVM Calls, Bypassing Destination Check - (`app/block_address.go`, `app/proposal.go`)

### Summary

Cronos enforces its blocklist only at the transaction-selection layer (mempool `AnteHandler` and `PrepareProposal`/`ProcessProposal`). The destination check is limited to the top-level `ethTx.To()` field. Internal EVM calls — contract-to-contract token transfers — are never inspected. A non-blocked user can call a non-blocked intermediary contract that internally transfers CRC20/CRC21 tokens to a blocked address, fully bypassing the blocklist's destination restriction.

### Finding Description

Cronos implements a blocklist with two enforcement points:

**1. `BlockAddressesDecorator.AnteHandle` (CheckTx only)**

```go
if ethTx.To() != nil {
    if _, ok := bad.blockedMap[sdk.AccAddress(ethTx.To().Bytes()).String()]; ok {
        return ctx, fmt.Errorf("destination address is blocked: %s", ...)
    }
}
``` [1](#0-0) 

**2. `ProposalHandler.ValidateTransaction` (PrepareProposal / ProcessProposal)**

```go
if ethTx.To() != nil {
    ...
    if _, ok := h.blocklist[encoded]; ok {
        return fmt.Errorf("destination address is blocked: %s", encoded)
    }
}
``` [2](#0-1) 

Both checks inspect only `ethTx.To()` — the outermost EVM transaction destination. Neither check is applied during actual execution (`FinalizeBlock`/`DeliverTx`), and neither inspects internal call targets or ERC20/CRC20 `transfer`/`transferFrom` recipients within EVM execution.

CRC20 and CRC21 contracts (`ModuleCRC20`, `ModuleCRC21`) implement standard `transfer` and `transferFrom` functions that can be called by any contract during EVM execution: [3](#0-2) 

A non-blocked user can deploy or use an existing intermediary contract `C` (not on the blocklist) and call `C.sendTo(blockedAddress, amount)`, where `C` internally calls `crc20Token.transfer(blockedAddress, amount)`. The blocklist check sees only `C` as the EVM tx destination — which is not blocked — and the internal transfer to the blocked address executes without any restriction.

Additionally, if a blocked address had previously granted an ERC20 `approve()` allowance to a contract before being blocked, a non-blocked user can call `transferFrom(blockedAddress, attacker, amount)` through that contract, moving assets **out of** the blocked address — effectively unfreezing it.

### Impact Explanation

**High** — Bypass of Cronos blocklist authorization checks.

The blocklist is a security control used to freeze addresses (e.g., sanctioned or compromised accounts). The bypass allows:
- Blocked addresses to receive CRC20/CRC21 tokens via internal EVM calls, undermining asset-freeze guarantees.
- Pre-existing ERC20 allowances to be exploited to drain tokens from a blocked address, circumventing the freeze entirely.

This directly matches the allowed High impact: *"Bypass of Cronos admin, governance authority, permission, token-mapping, bridge, block-list, or module-account authorization checks."*

### Likelihood Explanation

**Medium-High.** The attack requires no special privileges — any non-blocked user can execute it. The only prerequisite is holding CRC20/CRC21 tokens or having access to a contract with an allowance from the blocked address. The technique (routing through an intermediary contract) is well-known in EVM security.

### Recommendation

1. **Execution-level enforcement:** Add a blocklist check inside the EVM `StateDB` or as a post-execution hook that rejects any state transition that credits a blocked address with tokens (native or CRC20/CRC21).
2. **Revoke allowances on block:** When an address is added to the blocklist, sweep and revoke any outstanding ERC20 allowances granted by that address.
3. **Alternatively**, document explicitly that the blocklist only prevents direct transaction signing/sending and does not freeze assets against internal EVM transfers, and adjust the security model accordingly.

### Proof of Concept

1. Admin adds address `B` to the blocklist via `MsgStoreBlockList`.
2. `B` is now blocked: direct EVM txs to `B` are rejected at proposal time.
3. Non-blocked user `A` deploys intermediary contract `C`:
   ```solidity
   function relay(address crc20, address to, uint256 amount) external {
       IERC20(crc20).transfer(to, amount); // internal call, not checked
   }
   ```
4. `A` calls `C.relay(crc20Token, B, 1000)` — the EVM tx destination is `C` (not blocked), so it passes both `BlockAddressesDecorator` and `ProposalHandler.ValidateTransaction`.
5. During execution, `crc20Token.transfer(B, 1000)` executes without any blocklist check.
6. `B` receives 1000 CRC20 tokens despite being on the blocklist.

For the allowance-drain variant:
1. Before being blocked, `B` had called `crc20Token.approve(C, type(uint256).max)`.
2. After `B` is blocked, `A` calls `C.drain(crc20Token, B, A, 1000)` which internally calls `crc20Token.transferFrom(B, A, 1000)`.
3. Tokens move out of `B`'s balance, bypassing the freeze.

The root cause is that `ethTx.To()` is the only destination inspected, and no execution-level blocklist enforcement exists. [4](#0-3) [5](#0-4)
