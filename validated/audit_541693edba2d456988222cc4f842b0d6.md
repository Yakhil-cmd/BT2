### Title
Presignature Reuse Not Prevented at API Level Enables Secret Key Extraction - (File: src/ecdsa/robust_ecdsa/mod.rs, src/ecdsa/ot_based_ecdsa/mod.rs)

### Summary
Both `PresignOutput` types (OT-based and Robust ECDSA) derive `Clone` and `Serialize`/`Deserialize`, and `rerandomize_presign` accepts `&PresignOutput` by shared reference rather than consuming it. This means the library API places no enforcement barrier against presignature reuse. A malicious coordinator can orchestrate two signing sessions from the same presignature with different message hashes, enabling full secret key extraction via a nonce-reuse attack.

### Finding Description
The `PresignOutput` struct in both ECDSA variants is freely cloneable and serializable:

- `src/ecdsa/ot_based_ecdsa/mod.rs` line 40: `#[derive(Debug, Clone, Serialize, Deserialize, Eq, PartialEq, ZeroizeOnDrop)]`
- `src/ecdsa/robust_ecdsa/mod.rs` line 26: `#[derive(Debug, Clone, Serialize, Deserialize, PartialEq, Eq, ZeroizeOnDrop)]`

The `rerandomize_presign` function in both modules takes the presignature by shared reference (`&PresignOutput`), not by value:

- `src/ecdsa/ot_based_ecdsa/mod.rs` line 66–96: `pub fn rerandomize_presign(presignature: &PresignOutput, args: &RerandomizationArguments)`
- `src/ecdsa/robust_ecdsa/mod.rs` line 55–86: same signature

Because the presignature is not consumed, a caller can invoke `rerandomize_presign` on the same `PresignOutput` multiple times with different `(h, ε)` arguments, producing multiple `RerandomizedPresignOutput` values all derived from the same underlying nonce material `k`. Additionally, `RerandomizedPresignOutput` itself derives `Clone`, so even the rerandomized form can be duplicated before being passed to `sign`.

The library documentation explicitly acknowledges this danger but relies entirely on caller discipline:

> "Never reuse a presignature, even across failed, aborted, or partially completed signing sessions."

No runtime enforcement (usage counter, ownership transfer, or invalidation) exists in the production code.

### Impact Explanation
The Robust ECDSA security considerations document states:

> "If different subsets of size at least 2t+1 sign different (h, ε) values using shares derived from the same presignature, the resulting signatures use multiplicatively related nonces and the secret key can be recovered using standard ECDSA nonce-reuse attacks."

> "a novel split-view attack exists that can extract the secret key using as few as 2t+2 presigning participants, with as few as two signing sessions."

Two signing sessions using the same presignature but different `(h, ε)` yield two equations over the same underlying nonce `k`. Solving the system recovers the aggregate secret key. This is a **Critical** impact: extraction of the private signing key material.

### Likelihood Explanation
A malicious coordinator controls which `RerandomizationArguments` (including `msg_hash` and `tweak`) are presented to each participant. The coordinator can legitimately call `rerandomize_presign` twice on the same stored `PresignOutput` with different arguments, then run two signing sessions. Honest participants have no way to detect this reuse because the API provides no usage tracking. The attack requires only coordinator-level trust, which is within the documented threat model for split-view attacks.

### Recommendation
1. Change `rerandomize_presign` to consume `PresignOutput` by value (`presignature: PresignOutput`) so Rust's ownership system enforces single use at the call site.
2. Remove `Clone` and `Serialize`/`Deserialize` from `PresignOutput` (or gate them behind a feature flag with explicit warnings), preventing callers from trivially duplicating the one-time-use object.
3. Similarly, remove `Clone` from `RerandomizedPresignOutput` to prevent duplication of the rerandomized form before it is passed to `sign`.

### Proof of Concept
```rust
// Attacker is the coordinator; honest parties hold their PresignOutput shares.
// Step 1: Run presigning to obtain PresignOutput for each participant.
let presign_outputs: Vec<(Participant, PresignOutput)> = run_presign(...);

// Step 2: Rerandomize the SAME presignature for message h1 (API does not prevent this).
let rerand1 = RerandomizedPresignOutput::rerandomize_presign(
    &presign_outputs[0].1,  // shared reference — original is NOT consumed
    &args_for_h1,
).unwrap();

// Step 3: Rerandomize the SAME presignature again for a different message h2.
let rerand2 = RerandomizedPresignOutput::rerandomize_presign(
    &presign_outputs[0].1,  // same presignature, different args
    &args_for_h2,
).unwrap();

// Step 4: Run two signing sessions. Both produce valid signatures (R1,s1) and (R2,s2)
// derived from the same underlying nonce k. The coordinator now has two equations:
//   s1 = α·h1 + β·Rx1 + e
//   s2 = α·h2 + β·Rx2 + e
// Solving recovers the secret key shares.
```
