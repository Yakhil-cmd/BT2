### Title
Asymmetric Threshold Validation: Lower-Bound Check Present in DKG but Absent in FROST Signing — (File: `src/frost/mod.rs`)

### Summary

`assert_sign_inputs` in `src/frost/mod.rs` validates only the upper bound of the `threshold` parameter (`threshold > participants.len()`), while the analogous DKG validation function `assert_key_invariants` in `src/dkg.rs` validates **both** the upper bound and the lower bound (`threshold < 2`). This asymmetric validation allows a caller to pass `threshold = 1` (or `threshold = 0`) into FROST signing functions, causing the signing protocol to compute Lagrange coefficients for the wrong polynomial degree and produce invalid, unusable signatures.

### Finding Description

`assert_key_invariants` in `src/dkg.rs` enforces a symmetric two-sided bound on `threshold`:

```rust
// upper bound
if threshold > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
// lower bound
if threshold < 2 {
    return Err(InitializationError::ThresholdTooSmall { threshold, min: 2 });
}
``` [1](#0-0) 

`assert_sign_inputs` in `src/frost/mod.rs`, which gates all FROST signing entry points (`sign_v1`, `sign_v2`, and RedJubjub `sign`), only enforces the upper bound:

```rust
// validate threshold
if threshold.value() > participants.len() {
    return Err(InitializationError::ThresholdTooLarge { ... });
}
// ← no lower-bound check; threshold = 1 or threshold = 0 passes silently
``` [2](#0-1) 

Both `sign_v1` and `sign_v2` in `src/frost/eddsa/sign.rs` delegate directly to `assert_sign_inputs` without any additional threshold validation: [3](#0-2) 

The same is true for `sign` in `src/frost/redjubjub/sign.rs`: [4](#0-3) 

When `threshold = 1` reaches `construct_key_package`, it creates a `KeyPackage` with `min_signers = 1`. The FROST `round2::sign` then computes each participant's signature share using Lagrange coefficients derived for a degree-0 (1-of-N) polynomial. However, the key was generated by DKG with `threshold ≥ 2`, meaning the actual signing shares lie on a degree-1 (or higher) polynomial. The Lagrange coefficients are therefore wrong, the aggregated signature does not verify against the public key, and `aggregate` returns an error. [5](#0-4) 

### Impact Explanation

A malicious coordinator (or any caller who controls the `threshold` argument) can pass `threshold = 1` to any FROST signing entry point. Every signing session initiated with this value will produce an invalid signature that fails aggregation. Because nonces are consumed during `round1::commit`, the presignature material for that session is irrecoverably spent. A coordinator who consistently supplies `threshold = 1` causes **permanent denial of signing** for all honest participants: no valid threshold signature can ever be produced from the existing key material without a fresh signing session, and each fresh session can be sabotaged the same way.

### Likelihood Explanation

The coordinator role is explicitly part of the threat model (the documentation warns about split-view attacks by a malicious coordinator). The `threshold` parameter is caller-supplied with no out-of-band enforcement. Any party that controls the coordinator role — or any library consumer who misconfigures the call — can trigger this path. The attack requires no cryptographic capability and no leaked secrets.

### Recommendation

Add the same lower-bound check to `assert_sign_inputs` that already exists in `assert_key_invariants`:

```rust
if threshold.value() < 2 {
    return Err(InitializationError::ThresholdTooSmall {
        threshold: threshold.value(),
        min: 2,
    });
}
```

This should be inserted immediately after the upper-bound check at line 145 of `src/frost/mod.rs`. The same fix should be applied to `presign` in `src/frost/mod.rs` (line 72), which has the identical missing lower-bound check. [6](#0-5) 

### Proof of Concept

1. Run DKG with `participants = [P1, P2, P3]` and `threshold = 2` — succeeds normally.
2. Call `sign_v1(participants, threshold = 1, me = P1, coordinator = P1, keygen_output, message, rng)`.
3. `assert_sign_inputs` passes: `1 > 3` is false, no lower-bound check fires.
4. `construct_key_package(threshold = 1, ...)` creates a `KeyPackage` with `min_signers = 1`.
5. `round2::sign` computes signature shares using Lagrange coefficients for a 1-of-3 scheme (coefficient = 1 for the single signer), but the actual shares encode a 2-of-3 secret.
6. `aggregate` fails: the reconstructed signature does not verify against the public key generated by DKG.
7. The signing session is aborted; nonces are consumed; honest parties cannot recover without a new session, which can be sabotaged identically.
